# Supabase本文取得の確認と、会社アカウントからの本文取得（カスタムコネクタ）解決の検証

日時: 2026-07-24 18:54 (JST)

## 背景・目的

- かげだんがチャットで「今日のニュース」を深掘りする際、本文が毎回Fetchで取得され、note等では失敗していた。
- 「前回の大きな修正で本文を全文取得・Supabaseにも全文アップロードするようにしたはず」という記憶が正しいか、また現在のSupabaseに実際に全文が上がっているかの確認依頼。

## 完了ステップ

- [x] コード確認: `sync_supabase.py` は各記事のMD本文（frontmatter除く）を丸ごと `content` 列にアップロードする作り（`read_content`/`transform_article`）。`crawl_rss.py` は `enrich_with_fulltext`（readability）で要約フィードでも実本文を取得。git履歴に `d72f702 Embed full-text fetch in crawl_rss.py` / `1cf4726 Refetch RSS full text (2845 articles updated)` あり。→ かげだんの記憶は正しい。
- [x] 実データ確認（REST API・publishable key）: `content` 列はほぼ全件埋まっている。note 平均3,272字/最大13,128字、zenn 平均6,715字、qiita 5,726字、X 1,139字。null ほぼ無し。→ Supabaseに全文が上がっていることを実証。
- [x] 「チャットで本文が取れない」真因の特定: 参照URLが指す `articles_recent_Nd` ビューは `fix_views.sql` で設計上わざと content 列を除外（一覧を軽くするため）。`articles`/`articles_today`/`yesterday`/`2days_ago` には content あり。
- [x] データ量の実測: recent_3d に content 全込みは約2.8MB（4d=3.7MB, 7d=6.6MB, 14d=8.4MB超）→ 一覧に全文を積むのは web_fetch の取得上限で破綻。単日全カラム0.32MB、新しい順50件0.44MB、抜粋400字なら0.37MB。
- [x] id指定の1件取得URLは技術的に可能と実証（`?id=eq.<パーセントエンコード>&select=title,content` で1件・全文取得。noteでもFetch失敗なし）。ただしIDが日本語+「/」を含むためエンコード必須。
- [x] チャット環境の制約を実地検証（かげだんがモバイルチャットで実行）: 新パス（会話に出ていないURL）→ web_fetch が PERMISSIONS_ERROR で弾く。既出パス+条件後付け → 条件無視で全件返る。→ チャットは固定URLしか叩けず、動的1件取得は原理的に不可能。
- [x] 解決策の確認（かげだん到達）: 会社アカウント（Claude Desktop/モバイル）に2つ目のSupabase MCPコネクタ「Supabase howto-kb」を増設。URL=`https://mcp.supabase.com/mcp?project_ref=gryrgjnfekwptyngbmao&read_only=true&features=database`。これで MCP(execute_sql) から本文取得・検索・集計が可能に。vault記録を確認し数字を独立裏取り（articles 10,128行 / 直近3日357件 / カテゴリ内訳 claude-code157・ai-workflow148・construction38・cowork14 = 完全一致）。
- [x] 参照ガイドの所在確認: 「howto-kb 参照ガイド」の本体はリポジトリ内に存在しない（Grep空振り、docs/無し）。実体は Claude.ai プロジェクトの「指示」欄。TODO(3)「本文付きビューも1日分丸ごとは分量超過」の注記文案を提示 → かげだんが指示文の取得方法欄に追記済み。

## 備考・気づき

- チャット（Claude.ai）の web_fetch は「会話に登場した固定URLしか叩けない・条件パラメータは無視される」。動的な1件取得はできない。本文をチャットで扱うなら (a) MCP を使う か (b) 固定URLの中に最初から本文を積んでおく、の二択。
- Claude Code 環境（このセッション）では、個人アカのMCPが会社アカ接続のため権限エラー。本文確認は REST API（publishable key）で行うのが確実。
- `articles_recent_Nd` に content が無いのは仕様（一覧を軽く保つため）。本文は `articles` 系から引く。
