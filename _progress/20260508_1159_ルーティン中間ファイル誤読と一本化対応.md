## 備考・気づき

### 真因の構造

- スクリプトは「stdout に JSON を吐く」「stdin から JSON を読む」前提で書かれている（Unix流の組み合わせ）
- ところが SKILL.md がこの組み合わせ方を明示せず、ステップを単に「実行する」と並べていた
- 結果、Claudeが各日違うやり方で繋ごうとし、ある日は `_generate_mds.py`、ある日は per-source + ファイルリダイレクトを選んだ
- 今朝はファイルリダイレクトの「`>` で保存」部分が抜け、4/26時点の遺物を読む事故になった
- gitignored な作業ファイル（`_new_*.json`）は壊れていても気付きづらい

### 救出のキー: tool-results 永続ファイル

- 一定サイズ超過の Bash 出力は `<persisted-output>` として `tool-results/<id>.txt` に保存される
- jsonl 内には先頭部のサンプルしか残らないが、永続ファイルには完全な出力（stderr+stdout）が UTF-8 で保存される
- 末尾に出てくる JSON 配列を `txt.rfind('\n[\n')` から `'\n]'` で切り出せば復元可能
- これは「Claude が出力を読んだだけで保存忘れたデータ」を取り戻す重要なフォールバック手段

### _generate_mds.py 一本化の落とし穴

- 単純な置き換えではなく、`_filter_x_auto.py` の `is_useful` ロジックを `_generate_mds.py` に内包する必要があった
- そうしないと「X投稿のフィルタリング」というSKILL.md記載の挙動が失われる
- TECH_KEYWORDS / EXCLUDE_KEYWORDS / 30文字下限を移植して `is_x_useful()` として実装
- RT clippings は短いコメントの場合 is_useful で落ちるので、フィルター前の x_raw から抽出するよう分けた

### 中間ファイルの混乱

- `_new_rss.json` `_new_x.json` は gitignored だが物理的にはリポ直下に残る
- `_generate_mds.py` 一本化後はこのファイルを参照しないが、過去の遺物として残り続ける
- 次回ルーティンでも上書きされず古いままになる可能性が高い → 混乱の元
- 削除するか、`_generate_mds.py` 内で更新するかの整理が将来課題
