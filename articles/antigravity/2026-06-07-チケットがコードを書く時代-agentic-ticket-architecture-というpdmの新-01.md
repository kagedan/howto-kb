## はじめに

「エージェントを導入したのに、期待したほど速くならない」

こんな声を、最近あちこちで聞くようになりました。Cursor、Devin、Antigravity——AIコーディングエージェントが実用段階に入った2026年。多くのチームが「チケットを渡せばPRが返ってくる」という未来を期待してエージェントを導入しました。

しかし現実は、思ったより泥臭いものです。エージェントが見当違いな実装をする。意図と違う粒度でコードが生成される。何度もやり直しになり、「自分で書いた方が速かったのでは」と思う瞬間すらあります。

多くの人はこれを **エージェントの性能の問題** だと考えます。

私は違うと思っています。**問題はチケットの方にあるのです。**

---

## 1. エージェント導入チームが陥る「構造的な罠」

なぜエージェントが期待通りに動かないのでしょうか。その根本原因を整理すると、ほとんどのケースで同じパターンが見えてきます。

### 人間向けチケットの「暗黙知」問題

従来のチケットは、人間の開発者が読むことを前提に書かれています。

```
# 従来のチケット例
タイトル: ユーザープロフィール画面の改善
説明: プロフィール画面のUIを改善する。
    アバター画像のアップロード機能を追加。
    いい感じのデザインで。
受け入れ条件: デザインレビューを通ること
```

人間の開発者であれば、このチケットから「いい感じ」の意味を推測できます。過去のPRを見て、デザインシステムのトーンを理解し、チームの暗黙の品質基準に合わせてくれるでしょう。

**エージェントには、この「空気を読む」能力がありません。**

「いい感じ」は解釈不能。「デザインレビューを通ること」は検証不能。結果として、エージェントは自分なりの「いい感じ」を実装し、それが人間の期待とずれてしまうのです。

### 粒度のミスマッチ

もう一つの典型的な問題が、タスクの粒度です。

人間なら「プロフィール画面の改善」という大きなチケット1枚を渡されても、自分で作業を分解できます。「まずAPIを作って、次にコンポーネントを作って、最後にスタイルを当てよう」といった具合です。

エージェントに同じチケットを渡すと、**全部を一度にやろうとします**。結果、巨大なPRが生まれ、レビュー不能になり、どこかでバグが混入し、手戻りが発生してしまいます。

---

## 2. Agentic Ticket Architecture という考え方

ここで提案したいのが、**Agentic Ticket Architecture（エージェント指向チケット設計）** という概念です。

これは単に「チケットの書き方を工夫しよう」というテクニックの話ではありません。チケットの設計体系そのものを、エージェント実行に最適化された「アーキテクチャ」として捉え直す、という発想の転換です。

### コアテーゼ：チケットアーキテクチャ＝システムアーキテクチャ

本記事で最も伝えたいのは、この等式です。

| チケットの設計要素 | 決定されるシステム要素 |
| --- | --- |
| チケットの **粒度** | モジュール / コンポーネントの **粒度** |
| チケット間の **依存関係** | システムの **依存関係グラフ** |
| **受け入れ条件** の記述 | **自動テストケース** の設計 |
| **コンテキスト** の明示度 | エージェントの **実行精度** |
| チケットの **分割パターン** | アーキテクチャの **レイヤー構造** |

つまり、PdMがLinearにチケットを切る行為そのものが、**ソフトウェアアーキテクチャの設計行為になった**ということです。

従来、アーキテクチャ設計はテックリードの専権事項でした。しかしエージェント時代では、チケットの構造がアーキテクチャを規定します。**PdMがチケットを書く行為は、もはや「要件定義」ではなく「設計」なのです。**

## 3. 5つの Agentic Ticket Design Patterns

理論だけでは実践に落ちないので、すぐに使える5つのパターンを紹介します。

### Pattern 1: Atomic Ticket（原子チケット）

**原則：1チケット＝1ファイル変更 ＝ 1つの検証可能な振る舞い**

```
# ❌ Bad: 複合チケット
タイトル: ユーザー認証機能の実装
説明: ログイン、サインアップ、パスワードリセットを実装する

# ✅ Good: 原子チケット
タイトル: POST /api/auth/login エンドポイントの実装
説明: メールアドレスとパスワードを受け取り、JWTを返す。
コンテキスト: 認証にはnext-authを使用。DBはSupabase。
受け入れ条件:
  - 正しい認証情報で200 + JWTが返る
  - 誤ったパスワードで401が返る
  - 存在しないメールで404が返る
変更対象: app/api/auth/login/route.ts（新規作成）
```

チケットを原子レベルまで分解すると、エージェントの出力は劇的に正確になります。なぜなら、**スコープが明確だとエージェントの「判断」の余地が最小化される**からです。

### Pattern 2: Context Injection（コンテキスト注入）

**原則：エージェントが「参照すべきファイル」と「従うべきルール」を明示する**

```
コンテキスト:
  参照ファイル:
    - src/lib/auth.ts（既存の認証ユーティリティ）
    - src/types/user.ts（User型定義）
  設計ルール:
    - エラーハンドリングは src/lib/errors.ts の AppError クラスを使用
    - レスポンス形式は { data, error, status } の統一フォーマット
  禁止事項:
    - console.log によるデバッグ出力を残さない
    - any 型の使用禁止
```

人間にとっては「言わなくてもわかる」情報を、エージェントには明示的に渡す必要があります。これは手間に感じるかもしれませんが、**一度テンプレート化すれば、あとはコピペで済みます**。

### Pattern 3: Executable Acceptance Criteria（実行可能な受け入れ条件）

**原則：受け入れ条件をそのままテストコードに変換できる粒度で書く**

```
# ❌ Bad: 曖昧な受け入れ条件
受け入れ条件: ログインが正常に動作すること

# ✅ Good: 実行可能な受け入れ条件
受け入れ条件:
  1. POST /api/auth/login に { email: "test@example.com", password: "valid" } を送信
     → 200 OK, body に accessToken(string) と expiresIn(number) が含まれる
  2. POST /api/auth/login に { email: "test@example.com", password: "wrong" } を送信
     → 401 Unauthorized, body.error に "Invalid credentials" が含まれる
  3. POST /api/auth/login に { email: "", password: "" } を送信
     → 400 Bad Request, body.error にバリデーションエラーが含まれる
```

この粒度で書かれた受け入れ条件は、エージェントがそのまま **テストコードを自動生成する入力** になります。チケットがテスト仕様書を兼ねるわけです。

### Pattern 4: Dependency Declaration（依存関係宣言）

**原則：チケット間の依存関係を明示し、実行順序を定義する**

```
# チケットA: User型の定義
ID: PROJ-101
ブロッカー: なし

# チケットB: 認証APIの実装
ID: PROJ-102
ブロッカー: PROJ-101（User型が必要）

# チケットC: ログイン画面UIの実装
ID: PROJ-103
ブロッカー: PROJ-102（APIのレスポンス型が必要）
```

依存関係が明示されていれば、エージェントは正しい順序で作業を進められます。これはまさに **ビルドシステムの依存解決** と同じ発想です。チケットボード全体が、一種のDAG（有向非巡回グラフ）になります。

### Pattern 5: Prototype-First Ticket（プロト先行チケット）

**原則：まず「動く最小限」を作るチケットと「仕上げ」のチケットを分離する**

```
# Phase 1: プロトタイプ
タイトル: [PROTO] ログイン画面の仮実装
説明: 動作するログインフォームを最速で作る。
    デザインは仮。バリデーションは最低限。
    目的は「フローが動くことの確認」。
スコープ: UI + API接続のみ。エラーハンドリングは後続チケットで。

# Phase 2: 仕上げ
タイトル: [POLISH] ログイン画面のデザイン適用
ブロッカー: [PROTO] チケット完了後
説明: デザインシステムに準拠したスタイル適用。
    ローディング状態、エラー表示、アニメーションを追加。
```

プロトタイピングと仕上げを分けることで、**「まず動くものを見て判断する」** というサイクルが成立します。エージェントに一発で完璧を求めるのではなく、2段階に分けることで、各段階の精度が上がります。

## 4. 実践：Linear + AI Agent の高速プロトタイピングサイクル

これらのパターンを実際のワークフローに落とし込むとどうなるでしょうか。私が日々回しているサイクルを紹介します。

### サイクルの全体像

```
1. PdM：Linearにチケットを切る（5つのパターンを適用）
    ↓（5分）
2. エージェント：チケットを読み、実装する
    ↓（10〜30分）
3. PdM：動くものをレビューする
    ↓（5分）
4. PdM：Kill / Iterate / Ship の3択を判断する
    ↓
5. 次のサイクルへ
```

**1サイクルが30〜40分で回ります。** 従来のスプリントが1〜2週間だったことを考えると、フィードバックループの速度が桁違いです。

### Kill / Iterate / Ship フレームワーク

プロトタイプを見た後の判断を、以下の3択に絞っています。

* **Kill**：方向性が違う。チケットを閉じて別アプローチへ。
* **Iterate**：方向性は合っている。修正チケットを切って次のサイクルへ。
* **Ship**：十分な品質。仕上げチケットを切ってリリースへ。

重要なのは、**Killを恐れないこと**です。エージェントが30分で作ったプロトタイプであれば、捨てるコストはほぼゼロです。「作って捨てる」が安くなったことで、**意思決定の質が上がります**。実際に動くものを見てから判断できるからです。

## 5. PdMの役割が変わる — 「チケットアーキテクト」という新しいポジション

ここまでの議論を踏まえると、エージェント時代のPdMに求められるスキルセットが見えてきます。

### 従来のPdM

```
ユーザーの課題を発見する → 要件を定義する → 開発チームに伝える → リリースを管理する
```

### エージェント時代のPdM

```
ユーザーの課題を発見する → チケットアーキテクチャを設計する → エージェントが実装する → 動くものを見て判断する
```

変わったのは中間の2ステップです。「要件を定義して伝える」が「チケットアーキテクチャを設計する」に、「開発チームに依頼する」が「エージェントが即実装する」に変わりました。

これは **PdMの仕事が増える** という話ではありません。**PdMの仕事の性質が変わる** という話です。

曖昧な要件を書いて「あとは開発チームよろしく」が通用しなくなる代わりに、**自分が書いたチケットが直接プロダクトになる** という、圧倒的に短いフィードバックループを手に入れます。

### 「プロンプトエンジニアリング」との違い

「それってプロンプトエンジニアリングと何が違うの？」と思われるかもしれません。

プロンプトエンジニアリングは **1回のAI呼び出しの精度を上げる** 技術です。対して Agentic Ticket Architecture は **プロダクト全体の開発プロセスを設計する** 技術です。

* プロンプトエンジニアリング：「このプロンプトで良いコードが生成されるか？」
* Agentic Ticket Architecture：「このチケット構造で、良いプロダクトが生まれるか？」

スコープが違います。個人の生産性向上 vs 組織の開発プロセス変革。PdMが取り組むべきは、後者です。

---

## まとめ

エージェント時代の開発で最もレバレッジの高いスキルは、コードを書くことでもプロンプトを書くことでもありません。**チケットを設計すること**です。

今日から実践できる3つのアクション：

1. **次にチケットを切るとき、5つのパターンを1つでも適用してみる。** 特に「Atomic Ticket」と「Executable Acceptance Criteria」は即効性が高いです。
2. **「いい感じに」「適宜」「よしなに」をチケットから排除する。** これらの曖昧語は、エージェントにとって「ノイズ」でしかありません。
3. **Kill / Iterate / Ship の判断サイクルを意識する。** エージェントが作ったプロトタイプを「捨てる」ことに慣れる。

チケットの書き方を変えるだけで、エージェントの出力は変わります。そしてチケットアーキテクチャを変えれば、プロダクトの品質が変わります。

**チケットがコードを書く時代。PdMこそが、最も重要なアーキテクトです。**
