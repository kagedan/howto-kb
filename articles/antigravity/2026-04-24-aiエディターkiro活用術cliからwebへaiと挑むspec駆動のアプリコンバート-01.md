# 🚀 AIエディター「Kiro」完全活用術：CLIからWebへ。AIと挑む「Spec駆動」のアプリコンバート

## 1. はじめに：AIは「コードの書き換え」以上の仕事ができるか？

「手元で作った便利なCLIツールを、チーム全員がブラウザで使えるWebアプリにしたい」  
そう思った時、単に「コードをJavaScriptに書き換えて」とAIに頼むだけでは不十分です。

ブラウザではローカルファイルが直接読めない（セキュリティ制限）

外部コマンド（flutter pub など）が叩けない

大勢が使うためのレート制限やキャッシュはどうするのか？

本資料では、AIエディター（Kiro）の「Specモード」を使い、\*\*既存のDart製ツールをNext.jsベースのWebアプリへ移植するための「実現可能性調査」と「設計図作成」\*\*を行った実録を紹介します。

## 2. フェーズ1：AIによる「実現可能性調査（Feasibility Study）」

人間が「これ、Webアプリにできるかな？」と問いかけると、AIはプロジェクト全体をスキャンし、即座に「障壁」と「追い風」をリストアップしました。

### 🤖 Kiroの分析（要約）

* ❌ 障壁: Process.run（外部コマンド実行）がブラウザでは不可能。
* ✅ 追い風: ビジネスロジックが既にプラットフォーム非依存（純粋なDart）で書かれている。
* 💡 解決策: Flutter SDKをサーバーで動かすのではなく、ユーザーに pubspec.yaml と pubspec.lock をアップロードしてもらう「ファイル解析型」に設計を変更しましょう。

この段階で、AIは単なる「コーダー」ではなく、\*\*「システムアーキテクト」\*\*として機能しています。

## 3. フェーズ2：Specモードによる「構造化された設計」

今回のログの最大の特徴は、.kiro/specs/ フォルダを活用した「Spec（仕様書）駆動」の開発スタイルです。

行き当たりばったりでコードを書き始めるのではなく、まずAIに「仕様ドキュメント（Spec）」を書かせます。

### 要件定義: ユーザーがファイルをアップロードし、結果を画面上で閲覧できること。

### 技術スタックの選定:

* フロントエンド: Next.js + Tailwind CSS
* バックエンド/インフラ: Upstash Redis（レート制限用）

### 既存ロジックの再利用: CLI版の「カテゴリ分類ロジック」をWeb側にどう移植するか。

## 4. なぜ「Specモード」での相談が重要なのか？

今回のプロセスを通じて、AIエディターを使いこなすための3つのポイントが見えてきました。

「作れるか？」と「どう作るか？」を分離する:  
いきなり実装させず、まずAIに「今のコードのどこがWebに向かないか」を分析させることで、手戻りのない設計が可能です。

プラットフォームの制約をAIが補完する:  
「ブラウザでは外部プロセスが動かない」というエンジニアなら知っておくべき前提知識を、AIが補完し、代替案（JSON APIの利用やファイルアップロード方式）を提示してくれます。

独立性を保った移行戦略:  
「既存のロジックが独立しているので切り出しは比較的簡単です」といった、将来のメンテナンス性まで考慮したアドバイスをAIから引き出すことができます。

## 5. まとめ：AIと描く「アプリの進化」

今回のログは、1つのCLIツールが、AIとの対話を通じて「Webサービス」へと進化する第一歩を捉えたものです。

AIエディターは、ただキーボードを叩く手間を省くためのものではありません。「自分のアイデアが技術的に実現可能か」「より優れたアーキテクチャはないか」を24時間いつでも議論できる、最高に博識なパートナーなのです。

**現状の構成**  
このプロジェクトは **Dart製のCLIツール** で、Flutterプロジェクトの依存パッケージを監査し、セキュリティ・更新状況を4カテゴリに分類してレポートを生成します。Flutter GUIアプリではありません。

**Webアプリ化の障壁**

| 障壁 | 該当箇所 | 影響度 |
| --- | --- | --- |
| ファイルシステムアクセス | `pubspec_analyzer.dart` で `File()` を使用 | 高 — ブラウザはローカルファイルに直接アクセスできない |
| プロセス実行 | `outdated_analyzer.dart` で `Process.run('flutter', ...)` を使用 | 高 — ブラウザから外部コマンドは実行不可 |
| 標準入出力 | `console_reporter.dart` で `stdout.writeln()` を使用 | 中 — Web UIに置き換えが必要 |
| `exit()` | `audit.dart` で `exit(0)` / `exit(1)` を使用 | 低 — 削除するだけ |

**良いニュース**

* 依存パッケージ（`http`, `args`, `path`）は3つだけで、`http` と `path` はWeb互換
* ビジネスロジック（カテゴリ分類、アドバイザリ取得、バージョン比較）はプラットフォーム非依存で、そのまま再利用可能
* File Mode が既に存在し、Flutter SDKなしで `pubspec.yaml` + `pubspec.lock` だけで動作する設計になっている

**Webアプリ化のアプローチ**  
最も現実的な方法: File Modeをベースにした **ファイルアップロード型Webアプリ**

1. ユーザーが `pubspec.yaml` と `pubspec.lock` をブラウザにアップロード
2. Web側でファイルを解析し、pub.dev APIからアドバイザリ情報を取得
3. カテゴリ分類してブラウザ上に結果を表示

既存の `PubspecAnalyzer`、`AdvisoryAnalyzer`、`Categorizer` のロジックはほぼそのまま流用でき、変更が必要なのは主にI/O層（ファイル読み込みとレポート出力）だけです。Project Mode（`flutter pub outdated` の実行）はWeb版では対応不可ですが、File Modeだけでも十分実用的です。

結論: **Webアプリ化は十分可能です**。コアロジックの再利用性が高く、File Modeベースなら比較的少ない工数で実現できます。

Webアプリ化を進めたい場合は、specを作成して具体的な設計・実装計画を立てることもできます。進めますか？
