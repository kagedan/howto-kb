新技術のアップデートがあるたびにコードエディターの最適解に頭を悩ませている人も多いはず。今回はAntigravityが刷新されたので、情報を整理しようと思います。

Google I/O 2026の発表で、Antigravityは「ひとつのAI IDE」から「エージェント開発プラットフォームの総称」に近づきました。

なので、いま混乱しやすいポイントはここです。

> Antigravity 2.0、Antigravity IDE、Antigravity CLI、Antigravity SDKは、同じものの別名ではない。  
> 同じエージェント基盤を、別の画面・別の使い方から触るための入口です。

この記事では、Antigravityファミリーを4つの入口として整理します。

## まず結論

| 名前 | 何か | 主役 | 向いている作業 |
| --- | --- | --- | --- |
| Antigravity 2.0 | デスクトップの管制塔 | AIエージェント | 機能開発、調査、修正をまとめて依頼する |
| Antigravity IDE | コードエディタ | 人間の開発者 | 自分でコードを書きながらAIに補助してもらう |
| Antigravity CLI | ターミナルツール | CLIユーザー | ターミナルからエージェントを起動・自動化する |
| Antigravity SDK | 開発者向けライブラリ | 自社システム | 独自エージェントを組み込む |

ざっくり言うと、**2.0は「AIに仕事を振る場所」**、**IDEは「人間が書く場所」**、**CLIは「ターミナルから振る場所」**、\*\*SDKは「自分のサービスに組み込むための部品」\*\*です。

![Antigravityファミリー全体像](https://static.zenn.studio/user-upload/deployed-images/6bdd4bef11d1d2790da00902.png?sha=1e85dc05faad8ac3e56cc7b4ecd4e5354b337374)

## 関係図

ポイントは、中心にあるのが「Antigravity agent harness」だということです。GoogleはI/O 2026の発表で、Antigravityを「アイデアを本番対応アプリへ変えるための agent-first development platform」と説明しています。

その上で、2.0、CLI、SDK、Gemini APIのManaged Agents、Google Cloud向けのEnterprise Agent Platformなど、複数の入口を一気に出してきた。これが「Antigravityって結局どれ？」と感じる原因です。

## 1. Antigravity 2.0: エージェントの管制塔

Antigravity 2.0は、コードを書くためのエディタというより、**複数のAIエージェントを動かすためのデスクトップアプリ**です。

Googleの説明では、Antigravity 2.0はスタンドアロンのデスクトップアプリで、複数エージェントの並行実行、動的なサブエージェント、スケジュールタスク、Google AI Studio・Android・Firebaseとの統合を持つとされています。

ここでのユーザーの仕事は、1行ずつコードを書くことではありません。

* 「この機能を作って」
* 「このバグを再現して直して」
* 「この設計で問題がないか調査して」
* 「実装計画と差分を出して」

こうした大きめのタスクを投げ、エージェントの計画、作業ログ、成果物をレビューする。つまり、Antigravity 2.0は**実装者というより、エージェントの作業を束ねる管制塔**です。

従来のIDEに慣れている人ほど、ここで違和感が出ます。画面の主役が「ファイルツリーとエディタ」ではなく、「タスク、エージェント、Artifacts、レビュー」になるからです。

## 2. Antigravity IDE: 人間が主導するコードエディタ

Antigravity IDEは、従来の開発体験に近い入口です。

自分でファイルを開き、コードを書き、AIに補完や局所的な修正を頼む。Googleの製品ページでも、Editor viewはタブ補完、自然言語によるコードコマンド、コンテキストを理解するエージェントを提供するものとして説明されています。

こちらの主役は、あくまで人間の開発者です。

* 既存コードを読みながら少しずつ直したい
* AI補完を使いたい
* 選択範囲だけ書き換えたい
* VS Code系の見た目で作業したい

こういう場合は、Antigravity 2.0よりAntigravity IDEのほうが自然です。

混乱を避けるなら、こう覚えるとよいです。

> IDEは「自分が手を動かす場所」。  
> 2.0は「AIエージェントに仕事を任せてレビューする場所」。

## 3. Antigravity CLI: ターミナルから使う入口

Antigravity CLIは、GUIを開かずにターミナルからエージェントを動かすための入口です。コマンド名は `agy` とされています。

Google Developers Blogでは、Gemini CLIのユーザー体験をAntigravity CLIへ移行する方針が説明されています。特に重要なのは、Antigravity CLIがAntigravity 2.0と同じエージェントハーネスを共有する、という点です。

CLIが向いているのは、こういう作業です。

* ターミナルだけで開発フローを完結させたい
* シェルスクリプトやCIの中からエージェントを呼びたい
* 複数の調査や修正をバックグラウンドで走らせたい
* GUIよりログとコマンド履歴で管理したい

Googleは、個人向けのGemini CLIやGemini Code Assist IDE extensionsについて、2026年6月18日に一部リクエスト提供を停止し、Antigravity CLI / Antigravity 2.0への移行を案内しています。CLIユーザーにとっては、ここが実務上いちばん大きい変更です。

![CLIユーザー向けの移行イメージ](https://static.zenn.studio/user-upload/deployed-images/ff3e11f6e6702907ce5a7aa7.png?sha=3efe437f1ce740bcb82d1f91c784e631ae47aa70)

## 4. Antigravity SDK: 自分のシステムに組み込む入口

Antigravity SDKは、開発者やプラットフォームチーム向けです。

GoogleはSDKについて、Googleのプロダクトを支えるものと同じエージェントハーネスへプログラムからアクセスし、カスタムエージェントの挙動を定義して、任意のインフラ上でホストできるものと説明しています。

つまり、SDKは「Antigravityアプリを便利に使うための追加機能」ではありません。自社の開発基盤にエージェント機能を埋め込むための部品です。

たとえば、こういう用途が考えられます。

* 社内コーディング規約に沿ってPRを自動レビューする
* CIで失敗したテストを読み、原因調査の初動を自動化する
* 社内SDKや独自フレームワークに詳しいエージェントを作る
* セキュリティチェックや移行作業をエージェント化する

個人開発者が最初に触るものというより、**組織の開発ワークフローにAntigravityのエージェント基盤を組み込むための選択肢**です。

## なぜ今回わかりづらくなったのか

理由は、Antigravityという名前が指す範囲が広がったからです。

以前の感覚では、Antigravityと聞くと「AI IDE」を思い浮かべる人が多かったはずです。ところがI/O 2026では、GoogleはAntigravityをIDE単体ではなく、エージェントを動かすプラットフォームとして前面に出しました。

その結果、同じ発表の中に次のものが並びます。

* デスクトップで複数エージェントを管理するAntigravity 2.0
* ターミナルから使うAntigravity CLI
* 自作エージェント向けのAntigravity SDK
* Gemini APIで使うManaged Agents
* Google Cloud向けのEnterprise Agent Platform
* 既存のコード編集体験に近いAntigravity IDE

これらは競合する別製品というより、**同じエージェント基盤をどこから使うかの違い**です。

| ターミナルだけでAIエージェントを動かしたい | Antigravity CLI |
| CI/CDや社内ツールにエージェントを組み込みたい | Antigravity SDK |
| APIから管理された実行環境つきエージェントを呼びたい | Gemini APIのManaged Agents |
| Google Cloudプロジェクトと統合して組織運用したい | Gemini Enterprise Agent PlatformのAntigravity |

## ひとことで整理する

Antigravityは、もう単なる「AI付きコードエディタ」ではありません。

Google I/O 2026以降のAntigravityは、**エージェントに仕事を任せるための共通基盤と、その基盤にアクセスする複数の入口**として見るほうが理解しやすいです。

* 手で書くならIDE
* 任せて管理するなら2.0
* ターミナルで回すならCLI
* 自社基盤に組み込むならSDK

こう分けると、Antigravityファミリーの関係性はだいぶシンプルになります。
