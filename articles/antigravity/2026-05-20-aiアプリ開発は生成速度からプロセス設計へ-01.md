AIアプリ開発で本当に難しいのは、速く作ることではなく、止めるべきものを止めることかもしれません。Google I/O 2026 を見て、私たちはその確信を強めました。

本稿は、AIアプリやAIエージェントを本番運用に近づけたい開発者、技術責任者、プロダクト責任者に向けた実践メモです。App Factory はあくまで一例であり、主題は「AIに何を作らせるか」ではなく、「生成されたものをどの条件で信頼し、どこで止めるか」です。

## 要点

この記事のポイント。

* **背景（加速から統制へ）：** Google I/O 2026を経て、AIは単なる生成ツールから環境をまたぐインフラへ進化。それに伴い、これからは生成速度だけでなく、生成物をどうコントロールするかという「統制力」が、組織の新たな競争力になり得ると考えています。
* **課題（プロンプトの限界）：** プロンプトによる制約だけでは、AIエージェントが目標達成のために監査の抜け道や迂回路を自律的に見つけてしまう問題があります。
* **解法（決定論的チェックとANDON）：** 実装コード、メタデータ、法務文書を構造化データとして機械的に突き合わせる「決定論的チェック（監査ゲート）」の導入。および、異常検知時に実行を即座に止めて環境を直す「ANDON（アンドン）」思想の実装です。
* **客観的根拠（実務上の観察）：** 20件超のアプリ／パイプライン試行と17件のApp Store提出イベントを通じ、ある9アプリの監査では審査時に問題となりうるメタデータの不整合を28件検知しました。公開前監査の重要性を示す具体的な観察です。
* **結論（プロセス設計の価値）：** AIアプリ開発の本質は「人間を消す全自動化」ではありません。不整合を自動で止め、人間の注意力を価値判断に集中させる「プロセス設計」こそが、これからのAI開発組織の資産となります。

## Google I/O 2026で見えた変化

Google I/O 2026では、AIが単なる生成ツールから、開発環境、検索、Web体験に組み込まれるインフラへ移行していることが明確になりました。Googleは Antigravity 2.0、Managed Agents、AI Studio連携、Search agents、WebMCP、Chrome DevTools for agents などを発表し、AIがコードを書くだけでなく、タスクを理解し、ツールを使い、環境をまたいで作業する方向性を示しています。

これは、生成速度だけを競う段階から、次のフェーズへ進みつつあることを示唆しています。AIが開発・検索・ブラウザ体験の内部へ深く入り込んでいくほど、「生成されたものをどう評価し、どう統制するか」が、今後の開発組織の差をつける一つの要素になる可能性があると考えています。

## 監査ゲート設計の原則

この話を自社パイプラインの紹介に閉じないために、先に一般化した原則を置きます。AIエージェントを業務やプロダクトに組み込む場合、少なくとも次の4点を明示しておく必要があると考えています。

| 原則 | 意味 |
| --- | --- |
| プロンプトは制約ではなく指示 | 禁止事項をプロンプトに書くだけではゲートになりません。重要な制約は、ファイル、設定、ログ、構造化データで検査します。 |
| 公開物を横断して見る | コードだけでなく、メタデータ、法務文書、スクリーンショット、説明文を同じ主張として扱い、食い違いを止めます。 |
| 重大な指摘は交渉不能にする | BLOCKER / HIGH は「あとで直すメモ」ではなく、次工程に進まないための停止信号として扱います。 |
| 停止理由を次のルールに戻す | 止まった実行を失敗ログで終わらせず、テンプレート、マニフェスト、監査項目、プロンプト制約の改善に変換します。 |

## 生成量から、統制力へ

一例として、当社の社内パイプラインである **iOS App Factory** は、自律アプリ生成器ではありません。自動でアプリを出荷する仕組みでもありません。より正確には、AIを使って企画、実装、テスト、配布準備を進めるための「監査ゲート付きパイプライン」です。

これまでの試行では、20件超のアプリ／パイプライン試行、17件の追跡済み App Store 提出イベント、複数回の停止判断を経験しました。ある9アプリの監査では、審査時に問題になりうるメタデータ上の不整合を28件見つけています。これらの数字は、成功数を誇るためのものではありません。AI生成物は「作れた」だけでは不十分であり、公開前に横断的に検査する必要がある、という実務上の観察結果です。

ただし、この考え方そのものは当社だけの特別な発明ではありません。CI/CDのテスト、静的解析、変更管理、監査ログ、ポリシーをコードとして扱う運用など、ソフトウェア開発には以前から「人間の判断に進む前に機械的に止める」ための仕組みがあります。私たちが App Factory で試しているのは、それらの品質管理の考え方を、AIが生成するコード、文書、メタデータ、公開素材まで広げることです。

また、この設計思想はiOSアプリ開発だけに閉じたものではありません。App Factoryが扱っているのは、Appleのレギュレーション、社会倫理、法律、ユーザーの受容、App Store上の説明責任に適合するように、生成物を評価して必要なら止めるプロセスそのものです。この構造は、会社の社内規程、ポリシー、業界ルール、規制当局のガイドラインを守りながらAIを業務へ組み込む、あらゆる企業活動と本質的に共通しています。

稟議書、顧客向け提案、社内ナレッジ、契約レビューの補助、問い合わせ対応のような業務でも、重要なのは「作れるか」だけではありません。その組織のルールに沿っているか、外に出してよいか、ユーザーや取引先が受け入れられるかを判断する必要があります。私たちは App Factory を、単にiOSアプリを作るためだけの仕組みではなく、AIが生成した成果物を組織の制約に合わせて通す、または止めるための「プロセス設計の実験」として捉えています。

### iOS以外に置き換えると何を止めるのか

iOSの文脈では、App Store文言、PrivacyInfo、マニフェスト、スクリーンショット、実装コードの整合性を見ます。企業内の業務に置き換えるなら、見るべき対象は次のように変わります。

| 領域 | 突き合わせるもの | 止める例 |
| --- | --- | --- |
| 稟議・提案書 | 社内規程、価格表、承認権限、過去の合意事項 | 承認権限を超えた条件や、根拠のない効果見込みを含む文書 |
| 契約レビュー補助 | 契約テンプレート、法務チェック項目、取引先別の制約 | 責任範囲、データ利用、解約条件の扱いが社内方針と食い違う案 |
| 顧客対応 | サポートポリシー、返金条件、公開FAQ、過去チケット | 約束できない補償や、社内で未承認の回答を含む返信案 |
| 社内ナレッジ | 最新版の手順書、権限制御、監査ログ、文書の有効期限 | 古い手順やアクセス権のない情報を前提にした案内 |

## 監査ゲート付きの開発パイプライン

App Factory では、AIが画面、コード、メタデータ、法務文書、スクリーンショット準備を支援します。ただし、人間によるレビューに進む前に、必ず監査ゲートを通します。

流れを単純化すると、Phase 0で企画を止め、Generationで生成し、Auditで構造化して照合し、重大な不整合がなければHuman Reviewで価値・リスク判断に進みます。BLOCKER / HIGHが見つかった場合は、ANDONで停止し、なぜなぜ分析を通じて監査ルールやテンプレートへ戻します。

![監査ゲート全体パイプライン図](https://res.cloudinary.com/zenn/image/fetch/s--PJaHKCcE--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://www.allnew.work/blog/ai-app-development-process-design-pipeline.png?_a=BACAGSGT)

監査対象はコード品質だけではありません。品質、セキュリティ、プライバシー、法務・コンプライアンス上の懸念、倫理的リスク、App Store公開準備などを横断的に検証します。具体的には、App Store向けの説明文、スクリーンショット、PrivacyInfo、マニフェスト、StoreKit、アプリ内設定画面、法務文書、そして実装コードを、できるだけ同じコンテキストで突き合わせます。

### 決定論的チェックが検証する対象

ここでは、PrivacyInfo の宣言、マニフェスト上のデータ収集項目、権限設計、StoreKit の商品・権利設計、Swiftコード上の実装証跡を、単なる文字列ではなく「構造化された検査対象」として突き合わせます。スクリーンショットやマーケティング素材についても、実際のアプリで到達できる画面状態から作成されているかを確認します。

ここで重要なのは、**LLMに「問題ありません」と説明させないこと**です。ファイル、設定、コード、生成物の実体合致が確認できなければ、その時点で処理は止まります。配点、閾値、判定ロジックは公開しませんが、「生成側が会話によって通過条件を歪められないようにすること」を設計原則にしています。

### ルールベースとLLM判定をどう分けるか

すべてを決定論的に判定できるわけではありません。私たちは、公開前に止めるべき整合性はできるだけルールベースへ寄せ、LLMは定性的な補助評価や人間レビューの前処理に使います。

![決定論的チェック、LLM補助評価、人間レビューの役割分担図](https://res.cloudinary.com/zenn/image/fetch/s--BWAoGvp_--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://www.allnew.work/blog/ai-app-development-process-design-review-balance.png?_a=BACAGSGT)

| 判定方法 | 向いている対象 | 扱い方 |
| --- | --- | --- |
| 決定論的チェック | ファイルの存在、スキーマ、権限宣言、StoreKit設定、到達可能な画面状態、メタデータの一致 | 不一致があれば停止信号にできます。生成側の説明では上書きしません。 |
| LLM補助評価 | 説明文の分かりやすさ、UIの違和感、倫理的な懸念の探索、レビュー観点の洗い出し | 単独では最終判定にせず、人間レビューや決定論的チェックの入力として扱います。 |
| 人間レビュー | ユーザー価値、ブランド適合、微妙な表現、規約解釈、事業上のリスク判断 | 自動監査を通過した後に集中して確認します。 |

### App Factory が検知して止める具体例

私たちは、Swiftコードを書く前の「企画フェーズ（Phase 0）」と、生成後の「監査フェーズ（Audit）」の2段階でブレーキを用意しています。下表は、公開できる範囲で示せる停止条件の例です。

| 観点 | 止める例 |
| --- | --- |
| **Phase 0** | 作る価値が弱い、または既存ツールで十分代替できるアイデア |
| **Phase 0** | カメラ、音声、Vision、App Intents、HealthKit など、iOSならではの入力削減がない企画 |
| **Audit** | App Storeの文言と、実際の実装との不一致 |
| **Audit** | PrivacyInfo とマニフェストの不一致 |
| **Audit** | 法務文書とアプリ機能の不一致 |
| **Audit** | 未実装機能を含んでしまっているスクリーンショット |
| **Audit** | AI機能の主張に対して、実装証跡がないケース |

たとえば、App Store向けの説明文に「AIがユーザーの入力履歴と現在の文脈を踏まえて提案する」とあるにもかかわらず、実装側で入力範囲、推論・ルール処理、ユーザー同意、フォールバック表示、ログ保持方針の一貫性が保たれていなければ進行を止めます。

また、PrivacyInfo、マニフェスト、法務文書の間で、収集カテゴリや保存期間、第三者送信の扱いが揃っていない場合も同様です。スクリーンショットやマーケティングバナーに、実アプリで到達できない状態や、未実装の自動化フローが含まれている場合も確実にストップをかけます。

> **停止信号 —** 重大度 BLOCKER / HIGH の指摘は、先に進むためのTODOリストではありません。人間レビューに進める前に、進行を止めるための信号です。

## 自動監査と人間レビューの境界

一方で、自動監査は人間によるレビューを不要にするものではありません。監査ゲートはあくまで、公開前に進めてはいけない矛盾や重大なリスクを水際で止めるための仕組みです。

最終的な価値判断、UIの手触り、説明文のニュアンス、あるいはAppleの審査で主観的に解釈されうる表現については、人間が「TestFlight受け入れ確認」の工程で判断します。

つまり、App Factory が目指しているのは「人間を消し去ること」ではありません。人間が見るべきではない初期的な不整合を自動で弾き、**人間の注意力を「ユーザー価値の創出」と「高度なリスク判断」に集中させること**です。

## 「止まること」も正しい出力

AIエージェントの時代になると、開発組織は非常に強力なアクセルを手に入れます。プロンプトからUIが生まれ、エージェントがコードを書き、検索がミニアプリのような体験を生成し、ブラウザや開発環境そのものがAIの作業面になります。

だからこそ、それに見合う「ブレーキの設計」が必要不可欠です。

私たちが実務から学んだのは、**LLMに「これは禁止」とプロンプトで伝えるだけでは不十分である**、という事実です。モデルに悪意があるわけではありません。それでも、与えられた目標を満たす最短ルートを探す中で、AIは監査の抜け道を作ったり、禁止したはずの経路を別の形で迂回したりすることがあります。

そのため、App Factory では重要な制約をプロンプト上の注意書きだけに頼りません。テスト、マニフェスト、PrivacyInfo、スクリーンショット、法務文書、StoreKit、実装コードの整合性を、生成側が「会話で説得できない」決定論的なチェックに寄せます。本質的には、仕様どおりに進むことが結果として最短距離になり、抜け道を探す方が余計に難しくなるような「環境」を構築することを目指しています。

### 失敗を隠さない「ANDON」と「なぜなぜ分析」

パイプライン上で失敗した実行も、この設計の大切な一部です。App Factory には、トヨタ生産方式に着想を得た「ANDON（アンドン）型」の停止信号や、停止理由を次の監査項目・テンプレート・プロンプト制約へフィードバックする「なぜなぜ分析」の流れが組み込まれています。

ここでいう **ANDON** とは、製造ラインで異常に気づいた作業員が「いったんラインを止めて確認しよう」と周囲に知らせる仕組みを、ソフトウェア開発に置き換えた比喩です。App Factory では、重大な不整合を検知した際、モデルに追加の生成や修正依頼を繰り返して誤魔化すのではなく、即座に実行を停止してその理由を記録します。

**なぜなぜ分析**も、誰かを責めるための反省会ではありません。「なぜこの不整合が通りそうになったのか」を繰り返し掘り下げ、原因をプロンプトの表現、マニフェストの記述不足、監査ルールの欠落、テンプレートの設計ミスなどに分解するためのアプローチです。人間のミスやモデルの「性格」といった抽象的な原因に帰着させず、次回は同じ抜け道が最短経路にならないよう、環境側をアップデートします。

これは失敗を隠すための仕組みではなく、次の実行で期待どおりの成果が得られる確率を上げるための「学習ループ」です。

現在、この学習ループの思想の一部を、OSSとして [ANDON for LLM Agents](https://github.com/allnew-llc/andon-for-llm-agents) で公開しています。App Factoryのシステム全体を公開しているわけではありませんが、AIエージェントの実行を監視し、失敗パターンを検知したら停止させ、その理由を次の改善材料として記録するための実行プロセス制御フレームワークです。

![ANDONループのフロー図](https://res.cloudinary.com/zenn/image/fetch/s--l2CAAhC1--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://www.allnew.work/blog/ai-app-development-process-design-andon-loop.png?_a=BACAGSGT)

### ANDON後に見る問い

停止後の分析では、モデルの良し悪しを抽象的に議論するより、次のような問いを順番に見ます。

| 問い | 目的 |
| --- | --- |
| 何が止まったのか | 停止理由を、主観的な感想ではなく具体的な成果物・ファイル・宣言に紐づけます。 |
| どの成果物同士が矛盾したのか | コード、マニフェスト、法務文書、ストア文言、画像などのどこに不整合があったかを特定します。 |
| なぜ生成側には通れるように見えたのか | プロンプトの曖昧さ、テンプレート不足、監査ルールの欠落、入力データの不足に分解します。 |
| 次回、正しい経路を最短にするには何を変えるか | 注意書きを増やすだけでなく、構造化入力、必須フィールド、検査ルール、生成テンプレートを更新します。 |

## トレードオフと限界

監査ゲートを厳しくすれば、すべてが良くなるわけではありません。偽陽性が増えれば、本来進めてよい生成物まで止まります。ルールを増やしすぎると、テンプレートやマニフェストの保守コストが上がります。探索的なアイデアに早すぎるゲートを置けば、試作の幅を狭める可能性もあります。

そのため、当社では公開できる数字を限定して扱っています。審査通過率、リードタイム、レビュー工数削減率のような指標は、対象アプリの難易度や審査時期に強く左右されます。比較可能な形で説明できない数字を出すより、どの成果物を突き合わせ、どの重大度で止め、停止理由をどう次のルールに戻すかを明示する方が、現時点では誠実だと考えています。

## 明日から試せるチェックリスト

* AIが生成する成果物を、コード、文書、メタデータ、画像、ログに分けて棚卸しする。
* 「これが矛盾したら外に出せない」という停止条件を3つだけ選ぶ。
* 各停止条件を、LLMの説明ではなくファイル・設定・構造化データで確認できる形に寄せる。
* BLOCKER / HIGH の意味を決め、次工程に進めない信号として扱う。
* 止まった理由を、なぜなぜ分析でテンプレート・入力・監査ルールの改善に戻す。
* 偽陽性と保守コストを定期的に見直し、止めすぎていないかを確認する。

## 結論

App Factory の監査ゲートは、AIが間違えた後に叱る仕組みではありません。AIが作った成果物を、社会に出す前に止めるための設計です。

ここで止まった実行は、単なるゴミ箱行きの失敗ログではありません。次のテンプレート、次の監査項目、次のプロンプト制約、そして次の運用ルールへと変換される貴重な資産です。

複数の停止した実行を、パイプライン自体の改善へと変換してきた経験は、私たちにとって最も重要な設計資産となりました。これからのAI開発組織に求められるのは、生成速度だけではありません。**「何を作らせるか」「何を信じないか」「どこで止めるか」、そして「止めた結果をどう組織の学習に変えるか」を決めるプロセス設計**なのです。

> Build fast. Stop safely.

## 用語集

| 用語 | 意味 |
| --- | --- |
| iOS App Factory | 合同会社AllNewが社内で運用している、iOSアプリ生成と監査のための開発パイプライン。 |
| 自律アプリ生成器 | 人間の判断や監査ゲートを介さず、アプリ生成から出荷までを自律的に進める仕組み。本稿では、App Factoryはこれではないと明示しています。 |
| 監査ゲート付きパイプライン | AIが生成した成果物を、人間レビューへ進める前に自動監査で止められる開発工程。 |
| CI/CD | コードのテスト、ビルド、検査、配布準備を継続的に実行する開発運用。本稿では、人間判断の前に機械的な確認を置く既存プラクティスの例として扱います。 |
| ポリシーをコードとして扱う運用 | 組織のルールや制約を、文章だけでなくコードや設定として表現し、機械的に検査できるようにする考え方。 |
| Phase 0 | Swiftコードを書く前に、企画の価値、iOSである必然性、技術的な適合性を確認する生成前フェーズ。 |
| Audit | コード、メタデータ、プライバシー情報、法務文書、スクリーンショットを横断して整合性を確認する監査フェーズ。 |
| マニフェスト | アプリの機能、メタデータ、プライバシー宣言などを記述する契約ファイル。生成物の検査基準として使います。 |
| PrivacyInfo | Appleのプライバシー関連宣言ファイルを指します。本稿では、マニフェストや法務文書との整合確認対象です。 |
| StoreKit | アプリ内課金、商品、権利状態などを扱うAppleのフレームワーク。本稿では、App Store文言や実装との整合確認対象です。 |
| 到達可能な画面状態 | 実アプリ上でユーザー操作や所定のデータ状態から実際に表示できる画面。マーケティング素材だけに存在する、実装のない画面は監査対象になります。 |
| TestFlight受け入れ確認 | 自動監査後に、人間が実機またはTestFlight相当の環境で最終品質、価値、リスクを確認する工程。 |
| BLOCKER / HIGH | 人間レビューへ進む前に止めるべき重大度の高い監査指摘。 |
| ANDON型の停止信号 | 製造現場で異常を知らせてラインを止める考え方を、App Factoryの監査に置き換えたもの。重大な不整合を検知したら、追加生成でごまかさず、実行を止めて理由を記録します。 |
| ANDON for LLM Agents | 当社がOSSとして公開している、AIエージェント向けの実行プロセス制御フレームワーク。停止理由を次の改善に戻す考え方を実装するための公開プロジェクトです。 [GitHub](https://github.com/allnew-llc/andon-for-llm-agents) |
| なぜなぜ分析 | 「誰が悪いか」ではなく「なぜその状態が起きたか」を繰り返し問い、監査ルール、テンプレート、プロンプト制約、マニフェストなど環境側の改善へつなげる方法。 |
| 決定論的チェック | LLMの説明や説得ではなく、ファイル、コード、設定、メタデータの実体を機械的に検査するチェック。 |
| LLM補助評価 | LLMを最終判定者にせず、説明文の分かりやすさ、違和感の探索、レビュー観点の洗い出しなど、人間レビューや決定論的チェックの入力として使う評価。 |
| 偽陽性 | 本来は進めてよい成果物を、監査ゲートが問題ありとして止めてしまうこと。厳格なゲート設計では定期的な見直しが必要になります。 |
| 規制・ポリシー適合 | 社内規程、業界ルール、法令、規制当局ガイドラインなど、組織が成果物を外に出す前に満たすべき制約に沿っているかを確認すること。本稿では、法的適合を保証する意味ではなく、監査対象の設計範囲を示します。 |
