こんにちは、高橋です！

今回は、Google のエージェント型開発環境 **Antigravity** を使ううえで、  
成果を大きく左右する基本でありながら、非常に重要なテーマを扱います。

それは、 **「AIへの依頼は、大きく投げるほど楽になるとは限らない」** というお話です。

Antigravity は、公式ドキュメントでも「高いレベルのタスク単位で作業を進められる」「複数ステップで自律的に考え、コードやブラウザ、各種ツールを使いながら進行できる」と説明されている、とても強力な開発環境です。進行状況や成果物を Task list や Artifact として確認しながら進められるのも大きな魅力ですよね。

ただし、ここでひとつ、多くの人が陥りがちな誤解があります。   
それは、**「高レベルで頼める」ことと、「雑に大きく頼んでよい」ことはまったく違う**、ということです。

むしろ実務においては、AIが賢ければ賢いほど、次のような悲劇が起こりやすくなります。

以前の記事でも触れたように、Antigravity は「ただのコード補完AI」ではなく、ある程度自律して動く“優秀な作業者（エージェント）”です。  
だからこそ、仕事の「放り投げ方」を間違えると、プロジェクトは一瞬で迷走します。

この記事では、自律性の高いAIを使いこなすための「タスク分割の極意」をお伝えします。

AIの暴走に悩んでいる方は、ぜひ最後まで読んでみてください！

  

---

## 1. まず結論：自律型AIへの依頼は「4段階」に分けよ

Antigravity のような優秀なエージェントで精度を上げたいなら、依頼はできるだけ次の4段階に分割してください。

1. **調査 (Research)**
2. **設計 (Design)**
3. **実装 (Implementation)**
4. **検証 (Verification)**

そして、各段階ごとに以下の4つのルールをAIに明確に提示します。

### なぜ分けるだけで精度が上がるのか？

理由は非常にシンプルです。  
「1回ごとのAIの判断範囲（スコープ）が狭くなるから」です。

AIが暴走する時は、たいてい「何を優先すべきか」「どこまで進めていいか」「何を前提（暗黙の了解）にしていいか」が曖昧な状態です。  
逆に、1段階ずつ仕事を区切ってあげると、その時点でのゴールが明確になり、無駄な推測や暴走がピタッと止まります。

これは人間の新人指導とまったく同じです。 優秀だけど経験の浅い新人に、「いい感じのWebサービスを1週間で作っといて」**と丸投げするのと、**「まず競合を3件調べて、次に必要最小限の画面構成を整理して、OKが出たら問い合わせフォームだけ実装して、最後に動作確認して報告して」と指示するのでは、どちらが期待通りのものを上げてくるかは一目瞭然ですよね。

賢いAIほど、前者のような雑な依頼だと「足りない情報は自分で補って、一気に完成させなきゃ！」と善意で突っ走ります。  
だからこそ、**賢いAIほど雑な大依頼で大事故を起こす**のです。

## 2. なぜ「1回で全部やって」が危険なのか？

「せっかくの自律型AIなんだから、細かく指示するのはもったいないのでは？」と思うかもしれません。しかし、実務でそれをやると、次のような4つの問題が頻発します。

### ① 最初の前提を間違えたまま進みやすい

最初の解釈が1ミリでもズレると、その後の調査・設計・実装・検証すべてがズレます。 たとえば「社内で使うアンケート整理ツールを作って」と頼んだとき、あなたが「1画面のシンプルな表」を想像していても、AIが勝手に「ログイン必須・英語UI・多機能管理画面付き・データベース構築」と補完して進めたら、出来上がるのは「求めていない超大作」です。

### ② AIの“親切心”が裏目に出る

賢いAIは、ユーザーの指示の「空白」を埋めようとします。  
これは長所でもありますが、開発においては致命的な短所になり得ます。  
 頼んでいないフォルダをリファクタリングする、過剰に豪華なUIライブラリを勝手に導入する、ついでにデータベースの設計まで大改修してしまう……。  
AI本人は善意（ベストプラクティス）のつもりでも、人間から見れば「勝手にめちゃくちゃにされた」状態です。

### ③ 修正コスト（手戻り）が一気に大きくなる

大きな依頼を一発で進めると、ズレが発覚するのはたいてい「最後」です。 方向性、画面構成、使用技術、変数名、ゴール……すべてが間違っていた場合、それを直すのはゼロから作るより大変です。  
「ここだけ直して」と頼んでも、複雑に絡み合ったコードをAI自身が壊してしまうこともあります。時短のつもりが**手戻り地獄**に陥ります。

### ④ どこでズレたのか「原因分析」ができなくなる（最重要）

タスクを分けていないと、要件定義が悪かったのか、設計がおかしかったのか、コーディングでの解釈ミスなのか、テストが甘かったのかがブラックボックス化します。  
失敗の原因が分からないと、プロンプトの改善ができません。

## 3. なぜ「タスク分割」で精度が劇的に上がるのか？

ここがこの記事の「芯」です。タスク分割がもたらす4つの強力な効果を理解してください。

### 理由1：AIの「考える範囲」を狭められるから

AIの精度低下は、能力不足ではなく「一度に処理すべき論点が多すぎる」ことが原因です。   
要件理解、技術選定、画面設計、実装、テスト……これらを同時に考えさせると、AIの脳内メモリ（コンテキスト）が散らかります。  
「今回は調査だけ」と区切ることで、AIは「理解」というタスクに100%の能力を注ぐことができます。机の上を片付けて、今必要なものだけを置くイメージです。

### 理由2：間違いを「最小の傷」で発見できるから

大きな依頼は最後にお腹を開いてからガンが見つかりますが、小さな依頼は初期の健康診断で異常に気づけます。   
「想定ユーザーが違う」「この機能はまだ要らない」というズレに調査や設計の段階で気づければ、コードを1行も書かずに軌道修正が可能です。

### 理由3：出力形式を固定すると、品質が安定するから

「何をしてほしいか」だけでなく「どう出してほしいか（フォーマット）」を指定すると、AIの作業姿勢が整います。 たとえば「不明点を列挙して」と指示すれば、AIは自ら不足情報を探すモードに入ります。  
「変更ファイル一覧を出して」と言えば、変更範囲を最小限に抑えようと意識します。出力形式の指定は、単なる見た目の問題ではなく、AIの思考の枠組みを強制するハックなのです。

### 理由4：途中要約により「文脈（コンテキスト）」が壊れないから

長いやり取りをしていると、AIは「最初の方の指示」や「今回やらないこと」を徐々に忘れていきます。  
 これを防ぐのが**各フェーズごとの要約**です。フェーズの最後に「今回決まったこと」「今回やらないこと」をまとめさせることで、AIはそれを次の作業の「確固たる前提」として引き継ぐことができます。  
Antigravity の Task list が強力なのも、この「文脈の可視化と維持」に長けているからです。

## 4. Antigravity 攻略の4ステップ（タスク分割の基本形）

それでは、実際に4つのフェーズで何をすべきか、具体的なゴールと注意点を解説します。

### Step 1: 調査（Research）

**いきなり作らせず、まずは「理解」と「すり合わせ」だけをやらせます。**

* **やること**: 要件の整理、既存コードの把握、不明点の洗い出し、競合事例の確認。
* **ゴール**: 何を作るのかを共通認識にし、勝手な実装に入らせないこと。
* **重要ポイント**: **「絶対にまだコードを書かないでください」と強く念押しすること。** ここで実装に入らせると、推測だけで走り出して必ず事故ります。

### Step 2: 設計（Design）

**調査結果をもとに、「どう作るか」の計画を立てさせます。**

* **やること**: 画面一覧、データの流れ、MVP（Minimum Viable Product = 最小限の実用製品）の切り分け、変更ファイルの候補出し。
* **ゴール**: 実装前に構成を固め、「今回は何をやり、何をやらないか」を限定すること。
* **重要ポイント**: 「完成図」ではなく\*\*「作業計画」\*\*を出させること。過剰な設計になっていないか、人間がここでストップをかけます。

### Step 3: 実装（Implementation）

**ここで初めてコードやファイルの編集に入ります。**

* **やること**: 設計に基づいたコードの記述、ファイル作成、最低限のエラー処理。
* **ゴール**: 設計で決めた範囲「だけ」を作り、余計な変更を一切増やさないこと。
* **重要ポイント**: 「好きに直して」は禁句です。変更対象、変更理由、出力形式を厳密に指定し、人間がレビューできる状態を維持させます。

### Step 4: 検証（Verification）

**動作確認と、残課題の洗い出しを行います。**

* **やること**: 起動確認、画面表示、エラーの確認、残課題の整理。
* **ゴール**: “作ったつもり”を防ぎ、人間が最終チェックする前の粗探しをさせること。
* **重要ポイント**: 「ちゃんと動く？」と聞くのではなく、「スマホ表示で崩れないか」「CSVは読み込めるか」など、**人間側からテスト観点（テストケース）を渡すこと**が精度向上のカギです。

## 5. 実践！そのまま使えるプロンプトテンプレート集

概念は分かったけれど、どう指示すればいいか分からない方のために、コピペしてカスタマイズできるテンプレートを用意しました。

### テンプレ1：調査フェーズ

Plaintext

```
これから〇〇（作りたいものの概要）を作りたいです。
【重要】ただし、まだ実装（コーディング）は絶対にしないでください。
まずは調査と要件の整理だけを行ってください。

■ 目的：
- 〇〇を解決するため
- 〇〇を効率化するため

■ 前提条件：
- 〇〇（例：社内メンバー3名で利用）
- 〇〇（例：既存のデータベースは使わない）

■ 以下の4点について整理し、指定の形式で出力してください。
1. この依頼を達成するうえで、現時点で不明な点・不足している情報
2. 目的達成のために最低限必要そうな機能（MVP機能）
3. 最初のバージョンでは「あえてやらない・後回しでよい」機能
4. 実装前に私（ユーザー）に確認・決定してほしい事項

■ 出力形式（箇条書きで簡潔に）：
- 目的の再確認
- 想定ユーザーと利用シーン
- 不明点
- 必須機能候補 (MVP)
- 後回し機能候補
- 実装前確認事項

※繰り返しますが、コードはまだ書かないでください。
最後に、上記の内容を3行で要約してください。
```

### テンプレ2：設計フェーズ

Plaintext

```
ありがとうございます。方向性は合っています。
では次に、調査結果をもとに【設計（作業計画）】だけを行ってください。
【重要】まだ実装（コーディング）には入らないでください。

■ 設計の条件：
- 今回はMVP（最小実用版）の範囲のみを設計すること
- 〇〇の機能は必ず含めること
- 〇〇（例：ログインやDB保存）は今回は絶対に含めないこと
- 可能な限りシンプルで、拡張性よりも「まずは動く」ことを優先すること

■ 以下の形式で設計案を出力してください。
1. 今回のMVPの範囲（やることの定義）
2. 必要な画面やUI要素のリスト
3. データの流れ・処理のステップ
4. 必要なファイル構成案（新規作成・修正対象）
5. 実装の順序（ステップバイステップで）
6. 今回【明確にやらないこと】のリスト

最後に、以下の3点を簡潔にまとめてください。
- 今回決まったこと
- 今回やらないこと
- 次にやること（実装フェーズへの移行）
```

### テンプレ3：実装フェーズ

Plaintext

```
設計ありがとうございます。内容に合意します。
では、合意した設計に基づいて【実装】を開始してください。

■ 実装時の厳守ルール：
- 設計で合意したMVPの範囲「だけ」を実装すること
- 指示していない機能、ライブラリ、不要なリファクタリングは絶対に追加しないこと
- 各ファイルで「何を変更したか」を人間が把握できるようにすること

■ 出力形式：
作業が完了したら、以下の順で報告してください。
1. 新規作成・変更したファイルの一覧
2. 各ファイルの実装内容の要約（1ファイルにつき1行で）
3. ローカル環境での起動・確認手順
4. 現時点で残っている技術的な課題や妥協点

最後に、以下の3点を簡潔にまとめてください。
- 今回実装したこと
- 今回実装しなかったこと（次以降のフェーズに回したもの）
- 次にやること（検証フェーズへの移行）
```

### テンプレ4：検証フェーズ

Plaintext

```
実装お疲れ様でした。
次に【検証（テスト）】を行ってください。
【重要】このフェーズでは新機能の追加や、大規模なコード修正は絶対に行わず、確認と報告に集中してください。

■ 以下の観点でテストを実施してください：
- 画面がエラーなく正常に表示されるか
- 主要な操作（〇〇のクリック、〇〇の入力など）が機能するか
- エラー発生時にユーザーに分かりやすいメッセージが出るか
- UIの著しい崩れがないか
- 設計時の処理の流れ通りに動いているか

■ 出力形式：
検証結果を以下の形式でまとめてください。
- 確認項目 ｜ 結果（OK / 要確認 / NG） ｜ 補足・気づき

最後に、以下をまとめて報告してください。
- 修正が必須な点（バグなど）
- 人間（私）が最終確認すべきポイント
- 次に修正・開発を行うべき優先順位
```

## 6. よくある失敗と、そのリカバリー方法

最後に、この手法を実践する中でやりがちな失敗と、その解決策をお伝えします。

**失敗1：「分けたつもり」だが、実は指示が混ざっている**

**失敗2：出力形式を決めておらず、AIが長文を語り出す**

**失敗3：要約を残し忘れ、途中でAIが記憶喪失になる**

**失敗4：検証をお願いしたのに、勝手にコードを書き換え始める**

**失敗5：MVP（最小限のゴール）の線引きが甘い**

## まとめ：AIマネジメントは、人間のマネジメントと同じ

Antigravity のような高度な自律型エージェントは、魔法の杖ではありません。「非常に優秀で、手が早く、ちょっとお節介で、たまに早とちりするアシスタント」です。

彼らの能力を最大限に引き出すのは、丸投げではなく、**適切なタスクの切り出しと、明確な境界線の設定**です。  
少し手間に感じるかもしれませんが、この「4段階分割」を意識するだけで、手戻り地獄から解放され、本当の意味での「爆速開発」が体験できるはずです。
