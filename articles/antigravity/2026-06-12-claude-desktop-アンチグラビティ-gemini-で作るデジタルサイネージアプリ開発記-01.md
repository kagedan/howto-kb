# AIエージェント時代に「人間がやるべき作業」はどこに残るのか〜

## はじめに

今回は、店舗や商談スペースで使える**デジタルサイネージアプリ**(画像を20秒ごとに自動切替表示するWebアプリ)を、複数のAIツールを組み合わせて完成させるまでの記録です。

使ったツールと役割分担はこんな感じでした。

| ツール | 担当した工程 |
| --- | --- |
| **Claude Desktop** | アプリ本体のコード作成 |
| **アンチグラビティ(Antigravity)** | 手元のファイル操作・コード修正・デプロイ作業 |
| **Gemini** | Firebaseコンソール(ブラウザ操作)で詰まった部分のナビゲーション |
| **人間(私)** | Googleログインが必要な管理画面のクリック操作 |

「AIに全部任せられる時代」と言われがちですが、実際にやってみると**AIエージェントには構造的にできない作業**がはっきり存在します。この記事では、その「人間にしかできない部分」を中心に、ハマったポイントと解決策を時系列でまとめます。

---

## 1. アプリの概要

* **構成ファイル:** `admin.html`(管理用)、`signage.html`(表示用)、`firebase-config.js`(設定用)、`README.md`
* **要件:** 画像をFirebaseにアップロードし、サイネージ画面で20秒ごとに自動切替表示する
* **使用技術:** HTML/CSS/JavaScript(Vanilla)、Firebase(Firestore / Storage / Authentication / Hosting)

コード自体はClaude Desktopで作成し、その後の修正・運用はアンチグラビティに引き継ぎました。

---

## 2. なぜFirebaseの設定だけは人間がやるのか

ここが今回の記事の核心です。

アンチグラビティのようなAIエージェントは、**手元のコードを書いたりファイルを操作したりするのは非常に得意**です。しかし、**Googleアカウントでのログインが必要なFirebaseの管理画面(ブラウザ上のクリック操作)には、セキュリティ上、直接アクセスできない仕組み**になっています。

つまり、どれだけAIが優秀でも、

1. ウェブアプリの登録とAPIキーの取得
2. Firestore / Storage の有効化とルール設定
3. ローカルサーバーの起動確認

この3ステップだけは**人間が手を動かす必要がある**のです。逆に言えば、ここさえ越えればあとはAIに任せられます。

---

## 3. ステップ1:ウェブアプリの登録とAPIキーの取得

まずは、作成したコードとFirebaseをつなぐための「鍵」の取得から。

1. [Firebaseコンソール](https://console.firebase.google.com/)を開き、作成済みのプロジェクト(今回は `digital`)をクリック
2. 「</>」アイコン(ウェブ)をクリック
3. アプリのニックネーム(例: `signage-app`)を入力して「アプリを登録」
4. 表示された `const firebaseConfig = { apiKey: "...", appId: "..." }` から、`apiKey` と `appId` の値をコピー
5. 手元の `firebase-config.js` の指定箇所に貼り付けて保存

### ハマりポイント①:「</>」アイコンが見つからない

プロジェクトを作ったばかりでアプリが一つも登録されていない状態だと、「</>」アイコンは並んでいません。代わりに、プロジェクト名のすぐ下にある\*\*「＋ アプリを追加」という薄い青色のボタン\*\*から進みます。これをクリックすると、iOS / Android のアイコンと並んで「</>」(ウェブ)が出てきます。

「ドキュメント通りの画面が出てこない」というのは初心者が最初に挫折するポイントなので、AIに**スクリーンショットを見せて現在地を確認してもらう**のがとても有効でした。

---

## 4. ステップ2:Firestore と Storage の有効化

次に、データを保存する「箱」を2つ準備します。

* **Firestore Database** … 画像のメタデータ(ファイル名、URL、アップロード日時)を保存するテキストデータ用の箱
* **Storage** … 画像ファイル本体を保存する箱

### ハマりポイント②:メニューの名前がドキュメントと違う

Firebaseの画面デザインはアップデートされており、左側メニューの構成が以前と変わっています。現在は、メニュー中段の「プロダクトのカテゴリ」の下にある\*\*「Database と Storage」\*\*を展開すると、その中に「Firestore Database」と「Storage」が表示されます。

AIの学習データが古いUIを前提にしていることもあるため、ここでも「実際の画面のスクリーンショット」を共有しながら進めるのが確実です。

### ハマりポイント③:Realtime Database と Firestore を間違える(Parse error)

今回最大のつまずきポイントでした。

Firestore用のセキュリティルールを貼り付けたところ、\*\*「Parse error」\*\*で弾かれてしまったのです。原因は、開いていた画面が **「Realtime Database」** だったこと。

実はFirebaseにはデータベースが**2種類**あります。

|  | Firestore Database | Realtime Database |
| --- | --- | --- |
| 今回使うもの | ✅ こちら | ❌ 違う |
| ルールの書式 | `service cloud.firestore { ... }` | 独自のJSON形式 |

`service cloud.firestore...` で始まるルールはFirestore専用なので、Realtime Databaseの画面に貼ると書式エラーになります。左メニューから改めて\*\*「Firestore Database」\*\*を選び直し、「データベースの作成」→ ロケーションは `asia-northeast1`(東京)→「テストモードで開始」で作成しました。

作成後、「ルール」タブに以下を貼り付けて「公開」します。

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /signage_images/{doc} {
      allow read, write: if true;
    }
  }
}
```

### Storage側のルール

Storageは左メニューの「Storage」→「始める」→ テストモードで作成し、「Rules」タブに以下を設定。**ファイルサイズ5MB以下・画像形式のみ**という制限を最初から入れています。

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /signage/{file} {
      allow read: if true;
      allow write: if request.resource.size < 5 * 1024 * 1024
                   && request.resource.contentType.matches('image/.*');
    }
  }
}
```

「テキストデータを入れるFirestore」と「画像を入れるStorage」、**2つの場所にそれぞれ対応したルールを貼る**——ここが少しややこしいですが、構造を理解すると一気に見通しが良くなります。

## 5. ステップ3:ローカルサーバーで動作確認

手元のPCでアプリを動かして表示を確認します。

```
python -m http.server 8000
```

これを実行して、ブラウザで `http://localhost:8000/admin.html` を開けば管理画面が表示される……はずなのですが。

### ハマりポイント④:サーバーは起動しているのにファイルが見つからない

ターミナルには `Serving HTTP on :: port 8000` と出ていてサーバー起動自体は成功。ところがブラウザにアプリが表示されません。

原因は、コマンドプロンプトの左側が `C:\Users\User>` のままだったこと。つまり**Cドライブの中でサーバーを動かしていた**ため、`d:\デジタルサイネージ` にあるファイルを読み込めなかったのです。

解決手順は以下の通り。

1. `Ctrl + C` で今動いているサーバーを止める
2. ドライブをまたいで移動する:

**ポイント:** Windowsのコマンドプロンプトでは、`cd` のあとに `/d` を付けることで「違うドライブへの移動も一発で行う」指示になります。これを知らないと `cd d:\...` と打っても移動できず、地味に詰まります。

3. プロンプトが `d:\デジタルサイネージ>` になったことを確認して、再度 `python -m http.server 8000` を実行

これで無事に管理画面が表示されました。**「サーバーは、起動したフォルダの中身しか配信しない」**——当たり前ですが、初めてだと気づきにくい落とし穴です。

## 6. CSSの優先順位バグ:`hidden` 属性が効かない

動作確認の中で、`signage.html` に画像が表示されているのに「画像がありません」という案内文が消えない、というバグが発生しました。

* **原因:** JavaScriptで `emptyEl.hidden = true` を付けて非表示にしようとしていたが、CSSで `.empty { display: flex; }` と指定していたため、CSSの `display` 指定がHTMLの `hidden` 属性(デフォルトでは `display: none` 相当)を**上書き**してしまっていた
* **対策:** CSSに以下を追記し、`hidden` 属性がついているときは確実に非表示になるよう修正

```
.empty[hidden] {
  display: none !important;
}
```

「`hidden` 属性はCSSの `display` 指定に負ける」というのは、Vanilla JSで書いていると一度は踏む罠ですね。

## 7. 認証機能(Googleログイン)の追加

商談時など外出先からでも安全に管理画面を操作できるよう、認証機能を追加しました。

1. Firebaseコンソール > Authentication > Sign-in method で「Google」プロバイダを有効化(ここもブラウザ操作なので人間の担当)
2. `admin.html` にログインUI(HTML/CSS)を追加し、`signInWithPopup` 等でGoogleログイン処理を実装(こちらはAIエージェントの担当)
3. 未ログイン時は操作パネルを隠し、Firestoreの読み取りも行わないように変更

### セキュリティルールの本番化

ここで、テストモードのままだった「誰でも書き込めるルール」を卒業します。

* **Firestore:** 表示(read)は誰でも可、書き込み(write)は `request.auth != null`(ログイン必須)に変更
* **Storage:** 読み取りは自由、書き込みはログイン必須+5MB制限+画像のみ

「まずテストモードで動かす → 動いたら認証を入れてルールを締める」という二段階の進め方は、初心者にも分かりやすくおすすめです。

## 8. Firebase CLIのインストールとデプロイ準備

アプリをインターネット公開するため、Firebase CLI(コマンドラインツール)を導入しました。手順は3ステップ。

### ① Node.jsの確認

バージョン番号(例: `v20.x.x`)が出ればOK。「認識されていません」と出たら[Node.js公式サイト](https://nodejs.org/ja/)からLTS版をインストールし、ターミナルを開き直します。

### ② Firebase CLIのインストール

```
npm install -g firebase-tools
```

※Macで権限エラー(EACCES)が出る場合は `sudo npm install -g firebase-tools`。

完了確認は `firebase --version` で。バージョン番号が出れば成功です。

### ③ Googleアカウントでログイン

実行するとブラウザが立ち上がり、Googleのログイン画面が表示されます。Firebaseプロジェクトを作成したアカウントで「許可」をクリックすれば、ターミナルに `Success! Logged in as ...` と表示されて完了。**このログインも、AIエージェントには代行できない人間の作業**です。

## 9. Firebase Hostingで一般公開

最後に、アプリをURL化して公開します。

1. **設定ファイルの作成**(アンチグラビティが担当)
   * `firebase.json`:公開フォルダを現在のフォルダ(`.`)に指定
   * `.firebaserc`:デプロイ先プロジェクト(`digital-ffc47`)を紐付け
2. **デプロイ実行**

生成された公開URLでアプリが稼働するようになりました。  
表示モード　<https://digital-ffc47.web.app/signage>

### ハマりポイント⑤:初回デプロイで404エラー

HostingはルートURL(`/`)にアクセスした際に `index.html` を表示する仕様ですが、今回の構成は `admin.html` と `signage.html` のみで **`index.html` が存在しなかった**ため、初回デプロイ後にトップページが404になりました。

対策として、サイネージ画面と管理画面へ遷移できる入り口となる `index.html` を作成。

> **教訓:Webアプリを作るときは、必ず `index.html` を含めること。** Firebase Hostingに限らず、多くのホスティングサービスでルートURLの表示に必要になります。こちらはClaudeに学習と言うことでお願いをしておきました。

## 10. まとめ:AI時代の役割分担

今回の開発を振り返ると、役割分担はきれいに分かれていました。

* **AIが得意なこと:** コード作成、ファイル操作、設定ファイルの生成、バグ原因の特定、エラーメッセージの読解
* **人間にしかできないこと:** Googleログインを伴うコンソール操作、APIキーの取得・転記、CLIの認証、スクリーンショットでの現在地共有

特に効いたのは、**詰まったらスクリーンショットをAIに見せる**という進め方です。「UIが変わっていて手順書と違う」「開いている画面がそもそも別物(Realtime Database問題)」といったトラブルは、テキストのやり取りだけでは解決に時間がかかりますが、画像を共有すれば一発で原因が特定できました。

このフォルダ一式(コードと設定ファイル)とFirebaseコンソールへのアクセス権さえあれば、PCを買い替えたときや他の方に引き継ぐ際にも、すぐに環境を立ち上げて機能追加ができます。**「AIに任せる部分」と「人間が握っておく部分」を切り分けて記録しておく**ことが、AI協働開発のいちばんの引き継ぎ資産になるのかもしれません。

### 今回のハマりポイント一覧(再掲)

1. 「</>」アイコンがない → **「＋ アプリを追加」ボタン**から進む
2. メニュー名がドキュメントと違う → 新UIでは\*\*「Database と Storage」\*\*配下にある
3. ルールがParse error → **Realtime DatabaseではなくFirestore Database**に貼る
4. サーバー起動してもファイルが見えない → `cd /d d:\フォルダ名` で**正しいドライブに移動してから起動**
5. デプロイ後にトップが404 → **`index.html` を必ず用意する**

同じ構成でアプリを作る方の参考になれば嬉しいです。
