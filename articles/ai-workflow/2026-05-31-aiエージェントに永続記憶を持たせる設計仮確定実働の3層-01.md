AIエージェントを長く運用していると、ぶつかる壁があります。賢いモデルを使っているのに、会話をまたぐと前提が毎回リセットされ、判断の重さだけが残る。これは「記憶」を設計していないからです。

この記事では、AIエージェントに永続記憶を持たせるための実装設計を、**仮・確定・実働の3層**と**保存・昇格・想起の3段パイプライン**として整理します。複数のAIを実運用する中で固まってきた設計をベースにしています。

## なぜ「会話履歴」だけでは足りないのか

まず、混同されがちな2つを分けます。

* **会話履歴（コンテキスト）**：いま開いているスレッドの流れ。モデルがそのまま読む。ウィンドウから外れると参照されない。揮発する。
* **記憶（メモリ）**：履歴から「次も必要になる前提・決定・好み」を抽出して外部に永続化したもの。会話をまたいで積み上がる。

履歴をそのまま全部ためて毎回流し込む素朴な実装は、すぐ破綻します。トークンが膨らみ、古い前提と新しい前提が混ざり、関係ない文脈が判断に紛れ込む。必要なのは「全部覚える」ことではなく、**何を残し、何を信じて動き、いつ取り出すか**の設計です。

## 記憶の3層モデル：仮・確定・実働

記憶をフラットな倉庫にせず、確度で層を分けます。人間も「とりあえず聞いた話」と「確信した事実」を無意識に分けています。

| 層 | 役割 | 判断への使用 |
| --- | --- | --- |
| `tentative`（仮） | 入ってきたばかり。未検証。 | 使わない（参考のみ） |
| `confirmed`（確定） | 繰り返し確かめられた前提・決定。 | 判断の土台に使う |
| `working`（実働） | 今のタスクに直接効く。 | 即時に最優先で参照 |

ポイントは、新しく入った情報がいきなり判断を上書きしないこと。一度の発言で確定済みの前提がひっくり返る事故を防げます。

スキーマの最小形はこんなイメージです。

```
type MemoryTier = "tentative" | "confirmed" | "working";

interface MemoryRecord {
  id: string;
  scope: string;          // 用途・プロジェクト境界（混ぜない単位）
  type: "episodic" | "semantic" | "procedural";
  tier: MemoryTier;
  content: string;        // 1記憶=1事実が扱いやすい
  evidenceCount: number;  // 何回確かめられたか（昇格判定に使う）
  createdAt: string;
  updatedAt: string;
  embedding?: number[];   // 想起の関連検索用
}
```

`type` は記憶の性質です。**episodic**（いつ何が起きたか）、**semantic**（確立した事実・前提）、**procedural**（手順）。この3種が揃うと、エージェントは「あれと同じパターンですね」と過去を踏まえた判断ができます。

## 保存・昇格・想起の3段パイプライン

記憶を機能させる流れは3段です。

### 1. 保存（capture）

会話の終わりに、残す価値のある前提・決定・失敗を抽出して `tentative` で書き込みます。ここで全部を `confirmed` にしないのが肝心です。

```
async function capture(turn: ConversationTurn) {
  const candidates = await extractSalientFacts(turn); // LLMで要点抽出
  for (const c of candidates) {
    await db.insert({ ...c, tier: "tentative", evidenceCount: 1 });
  }
}
```

繰り返し出てきて確かだと分かったものだけを `confirmed` へ上げます。逆に、新しい事実と矛盾した確定前提は `tentative` か過去ログへ降ろします。**上げるだけでなく降ろす**のが、記憶を古い地図にしないコツです。

```
async function reconcile(memo: MemoryRecord, incoming: Fact) {
  if (semanticallyEqual(memo, incoming)) {
    memo.evidenceCount++;
    if (memo.evidenceCount >= 2) memo.tier = "confirmed"; // 昇格
  } else if (contradicts(memo, incoming)) {
    memo.tier = "tentative"; // 降格。確定の座から降ろす
  }
}
```

低頻度のバッチ（cronなど）で回すと、会話のレイテンシに影響しません。

### 3. 想起（recall）

次の会話で、**いま必要な記憶だけ**を取り出して文脈に注入します。全部を毎回入れない。ここが設計の腕の見せどころで、量より「取り出し方」で差がつきます。

```
async function recall(query: string, scope: string, tokenBudget: number) {
  const hits = await db.search({
    scope,                       // 用途の境界を越えない
    tierPriority: ["working", "confirmed"], // 仮は原則入れない
    vector: await embed(query),
    limit: 20,
  });
  return packWithinBudget(hits, tokenBudget); // トークン予算で打ち切る
}
```

想起の3原則：

1. **scope を越えない**：別用途・別プロジェクトの記憶を混ぜない。
2. **tier で絞る**：判断には `confirmed`/`working` を使い、`tentative` は基本入れない。
3. **トークン予算で打ち切る**：関連度上位から予算内で詰める。「全部入れる」をやめる。

## MCP で記憶レイヤーとして切り出す

この記憶を、モデルの中ではなく**外の独立したレイヤー**に置くと、効きます。MCP（Model Context Protocol）サーバーとして `memory.save` / `memory.recall` を公開すれば、どのエージェント・どのモデルからも同じ記憶を読み書きできます。

```
// MCPツールの例（概念）
{
  "tools": [
    { "name": "memory_save",   "description": "事実を記憶に保存（既定tier=tentative）" },
    { "name": "memory_recall", "description": "クエリとscopeで関連記憶を取得" }
  ]
}
```

外に置く利点は3つです。

* **可搬性**：モデルを乗り換えても記憶はそのまま。乗り換えが「育て直し」でなく「つなぎ替え」になる。
* **共有**：複数エージェント・チームで1つの記憶を共有でき、属人化が減る。
* **所有**：中身が見える・直せる・持ち出せる。自分の資産として扱える。

## よくある落とし穴

* **全部 confirmed にする**：検証なしで判断に使い、一度の発言で前提が壊れる。→ 仮を経由させる。
* **scope を切らない**：別文脈の記憶が紛れて判断がぶれる。→ 保存時に必ず scope を付ける。
* **recall で全件注入**：トークン爆発と精度低下。→ 関連度＋予算で打ち切る。
* **降格を実装しない**：古い前提で走り続ける。→ 矛盾検知で降ろす。

## まとめ

AIエージェントの記憶は、ためる対象ではなく**整理して取り出す対象**です。仮・確定・実働で確度を分け、保存・昇格・想起で流れを作り、scope とトークン予算で取り出しを絞る。そして記憶をモデルの外に置けば、可搬・共有・所有が手に入ります。

モデルは使い捨て、記憶は育つ資産。どちらに設計時間を積むかで、エージェントの寿命が変わります。
