## はじめに ─ 点群の宇宙で「いのち」に出会った日

昨年、大阪・関西万博の「いのち動的平衡館」を訪れました。

暗い空間に目が慣れてくると、それは突然現れました。無数の光の粒が空間に立体的に浮かび上がり、点群として描かれる「いのちの営み」が音とともに展開していきます。単なる映像体験ではありません。光の粒のひとつひとつが細胞であり、タンパク質であり、その集積が「私」という存在を作っている、そんな感覚を体感しました。

生物学者・福岡伸一さんがプロデュースした同パビリオンのテーマはシンプルです。「いのちを知る」。しかしそこで語られていたことは、生命という現象が持つ、驚くほど逆説的な本質でした。

*生命は、変わらないために変わり続けている。*

そして今、画面の向こうでは、AIエージェントが膨大なコードを次々と生成しています。タスクを与えれば自律的に動き、ファイルを読み、テストを走らせ、プルリクエストを積み上げていきます。その速度は圧倒的です。

ですが、ふと思いました。

あのパビリオンで「生命は壊しながら作り替え続けることでエントロピーにあらがっている」という話を聞いたときの感覚と、いま目の前で起きていること ─ AIが爆速でコードを積み上げながら、設計の秩序がじわじわと失われていく現象 ─ は、まったく同じ構造をしているのではないか？

コードを書くこと。設計すること。そしてときにそれを壊し、作り直すこと。これは単なる職業としての作業ではなく、宇宙の大原則に抗おうとする生命らしい営みなのかもしれない、そんな壮大なイメージを少し語らせてください。

## 1. エントロピーという宇宙の掟

高校物理で習った「熱力学第二法則」というのがあります。

孤立した系の中では、エントロピー（乱雑さ・無秩序さの度合い）は常に増大する方向にしか進まない。砂で作った城は崩れ、机の上は散らかり、ピラミッドは風化します。秩序は自然に無秩序へと向かい、その逆は起きません。これは宇宙の根本原理です。

そして、ソフトウェアの世界も例外ではありません。ソフトウェアエンジニアリングにおける「コードロット（code rot）」という概念は、まさに「エントロピーによる構造の劣化」を指しています。コードに新しい修正を加えるたびに複雑性は増し、放置すれば設計は崩れ、ドキュメントは実態とずれ、レガシーコードへと朽ちていきます。これは怠慢や失敗ではなく、開発を続けるかぎり避けられない物理的な宿命です。

読者のみなさんのチームのコードベースも、今まさにエントロピーと戦っている最中ではないでしょうか。

## 2. ソフトウェアエントロピー ─ AIが登場する前からの宿命

実はこの問題は、AIエージェントが登場するずっと前から存在していました。

ソフトウェアは使い続けるほどにエントロピーが積み上がります。機能を追加するたびにコードの層が重なり、当初の設計意図が埋もれていきます。ハードウェアやインフラが更新されるたびに、その場しのぎのパッチが継ぎ足され、一貫性のないアップデートが堆積します。

開発者が入れ替わるたびに、コードに実際の設計意図が伝わらなくなり、暗黙知が消えていきます。設計ドキュメントと実装の乖離が広がれば広がるほど、「設計書の宇宙」と「コードの宇宙」は別々に腐食していきます。

リファクタリングはそのエントロピーを排出するための定期的な担い手として機能しますが、現実の開発現場では「長期間の保守により劣化したコード…分かっているけど手が出せない」という声がよく聞かれます。

エントロピーの蓄積は部分アップデートを繰り返すほど加速し、いつしか「山を動かすより山を巧みに迂回する方が早い」という状態に至ることも珍しくありません。

これは怠慢ではなく、開放系である以上、すべてのソフトウェアが避けられない宿命です。

## 3. AIエージェントは「エントロピー生成速度の増幅装置」だった

そこにAIエージェントが登場しました。

AIエージェントは爆速で優秀です。タスクを与えれば自律的にコードを生成し、複数の処理を並行して進めます。しかし、その能力は「エントロピーの問題をなくした」のではなく、「エントロピーの生成速度を爆発的に増幅させた」という側面を持っています。

さまざまなレポートからも、AIアシスタントの普及とともに**複製コードブロックの増加と再利用率の低下**が報告されています。開発者のアンケート結果でも、「AIは見かけ上は正しいが信頼できないコードを作る」「不要・重複コードの増加によって技術的負債が増えた」と感じている人が少なくありません。  
例：[The Impact of AI-Generated Code on Technical Debt and the Need for Accelerated Codebase Appraisal](https://cerfacs.fr/coop/hpcsoftware-codemetrics-kpis)

なぜこうなるのでしょうか。LLMはトークン単位の確率生成によって「局所的に整合するコード断片」を連結していくため、全体のアーキテクチャという「グローバルな秩序」を担保する仕組みを本質的に持っていません。自然言語の曖昧な要件から構造化されたコードを得ようとする変換では、仕様側の不確定性がそのままコードの揺らぎとして増幅されやすくなります。

その結果、AI時代の開発現場には新たな三種類の負債が積み重なるようになりました。

| 負債の種類 | 内容 |
| --- | --- |
| 技術的負債 | 変更コストを前借りした設計・コードの問題（古典的なテーマ） |
| 理解負債 | AIが書いたコードを誰も説明できない状態（個人レベル） |
| 認知負債 | システム全体の設計意図をチームが説明できない状態（組織レベル） |

三大負債：[AIコーディングはなぜ後から苦しくなるのか？　技術負債に続く「理解負債」「認知負債」という新たな落とし穴](https://atmarkit.itmedia.co.jp/ait/articles/2603/12/news016.html)

「コードが自分の理解を超えて成長していく」 ─ Karpathy 氏が提唱した「Vibe Coding」の延長線上で、「Vibe Coding Cliff」と呼ばれる状態が議論されています。これはまさにエントロピーが制御を失う瞬間です。AIはエントロピーを食い止められません。それどころか、生成のスピードに比例してコードのエントロピーを爆速で積み上げる危険性を持っています。

## 4. 「わぁ動いた」に踊らされてはならない ─ 庭師なき庭の末路

AIエージェントがコードを生成するとき、それはとても気が利いています。

要件の周辺にありそうなエッジケースを先回りして処理し、「使うかもしれない」将来の拡張ポイントを用意し、念のための状態管理を追加し、呼ばれるかどうかわからないフック関数を仕込んでくれます。個々の判断は一見まともに見えます。ですが、その積み重ねが静かに恐ろしいのです。

「わぁ、できた。わぁ、動いた。」

そのとき、誰もコードを疑いません。テスト結果がグリーンになり、画面に期待通りの挙動が出れば、そこで思考が止まります。しかしその瞬間、庭にはすでに種が蒔かれているのです。

これを庭に例えてみます。

AIエージェントは、頼めば素早く花を植えてくれる優秀な植え師です。ですが、彼は「あそこにも咲いたら綺麗かも」「この種は後で役立つかも」と、頼んでいない場所にも種を蒔いていきます。個々の判断に悪意はありません。むしろ善意です。

最初の週は、庭は豊かに見えます。二週目も、まだ管理できます。ですが庭師が刈り込みをしなければ、一ヶ月後にはそこかしこで芽吹いた草が花壇を侵食し、通路が塞がれ、「どこに何を植えたか」すら分からなくなります。

庭師がいなければ、庭は庭でなくなってしまいます。

AIが生み出す典型的な「雑草コード」のパターンはいくつかあります。

* **過剰なエッジケース処理**  
  発生しえないケースに対するガード条件が多重にネストされ、本質的なロジックが深く埋もれます。読む側は「これ、本当に起きるケースなのか？」と首をかしげながら読み進めるしかありません。
* **おおげさな状態遷移**  
  シンプルなフラグで済む制御に、ステートマシンが召喚されます。多くの「状態」と「状態遷移」が定義された結果、「今どこにいるのか」を把握するだけで認知コストが跳ね上がります。
* **使うかも、の未来への配慮**  
  プラグイン機構、イベントバス、拡張ポイント用の抽象クラス。今は誰も使っていません。仕様書にも書いていません。しかし「将来のスケールに対応できるように」という親切心で置かれた構造が、設計の意図を曇らせます。

これらは個々には「ありえないほど間違っている」わけではありません。だからこそ厄介です。コードレビューで弾くには根拠が弱く、放置しているうちに「それがあること」が前提になり、次のAIが「既存の設計に従って」さらに雑草を育ててしまいます。

これは、動的平衡の文脈で言えば「分解機構が欠如した生命」の状態です。

細胞がオートファジーを止めれば、不要なタンパク質が蓄積し、細胞は自らのゴミに溺れます。生命はつくることと壊すことを同等に重視します。壊すことは後退ではなく、次のサイクルのための新陳代謝です。

AIエージェントはつくることに特化した存在です。壊す判断、省く判断、「これは要らない」と言う判断は、まだ人間の領域に深く根ざしています。

庭師の仕事とは、何を植えるかだけでなく、何を抜くかを決めることです。

具体的に、庭師として何を見るべきでしょうか。

* **このチェック処理は、実際に起きうるケースか？**  
  エラーハンドリングが何のために書かれているかを読み解き、「防御的コーディングの名の下に過剰になっていないか」を問い直します。
* **この状態遷移は、本当に必要な複雑さか？**  
  状態数が多いことは設計の豊かさではなく、問題の切り分けが不十分なサインである可能性が高いです。
* **この抽象レイヤーは、今使われているか？**  
  YAGNI の原則（You Aren't Gonna Need It） ― 「それは今必要ない」「必要になるまで作るな」 ― は、AIが登場してからむしろ重要性を増しています。AIは「将来必要になりそうな構造」を躊躇なく追加するからです。
* **コードを読む側の認知コストは増えていないか？**  
  機能が動いていることと、コードが健全であることは別の問いです。

「わぁ動いた」は出発点であって、ゴールではありません。

動いているコードは数え切れないほどあります。美しく持続可能なコードは、それよりずっと少ないです。その差を生み出すのは、AIエージェントの能力ではなく、庭師としての人間の判断です。

## 5. 生命はエントロピーに「あらがう」のではなく、エントロピーと「踊る」

ここで話を生命の世界に戻します。

1944年、物理学者エルヴィン・シュレーディンガーは著書『生命とは何か』の中で、「なぜ生き物は熱力学的平衡（死）に落ち込まないのか？」という問いを立てました。  
彼の答えは鮮やかです。「生物体は負のエントロピー（ネゲントロピー）を食べて生きている」。生命は孤立系ではなく、外界との絶え間ない物質・エネルギーの交換によって、局所的な秩序を保つ開放系なのです。

さらにその10年前、生化学者ルドルフ・シェーンハイマーは同位体元素を使った代謝実験で、生命体の構成分子が絶えず入れ替わっていることを示しました。「昨日の私」と「今日の私」は、物質レベルではすでにかなり別のモノになっています。生命は「作る」より「壊す」ことを優先し、壊すことで蓄積したエントロピーを排出し、外から新鮮な秩序を取り込んでいます。

福岡伸一さんはこれを「動的平衡」と呼びました。

> 生命は、変わらないために変わり続けている。

万博「いのち動的平衡館」の建築設計者は、「生命は変わらないために変わり続けているように、この建築は建ち続けるために動き続けている」と語り、プロセスとしての生命性を建物に埋め込んでいます。生命的なのはその「形」ではなく「流れ」であり、分解と合成の循環そのものだといえます。

プリゴジンの「散逸構造」の概念がさらにこれを深めます。竜巻や細胞のような「生きた構造」は、平衡からはるかに遠いところで、外界にエントロピーを放出しながら局所的秩序を自発的に生み出します。それは死んだ結晶のような静止した秩序ではなく、常に流れ続ける動的な秩序です。混沌（カオス）は、より大きなスケールで見ると、秩序を持続させる安定化の力として働きます。

## 6. エンジニアは「動的平衡の維持者」である

ここで、生命論とソフトウェア工学が交差します。

福岡伸一さんは動的平衡について、「生命の本質は遺伝子や細胞という要素にあるのではなく、要素と要素の関係性、それらの『あいだ』で起きる相互作用にこそある」と語っています。

これは今のAIエージェント時代のエンジニアリングにも、そのまま当てはまります。

問題はコードという部品の品質だけではありません。コードとコードの関係性、システム全体のアーキテクチャという秩序こそが問題です。AIは部品を高速に作れますが、関係性と秩序の番人にはまだなっていません。「コードを書く人」から「コードの関係性を設計・管理する人」へのシフトは、まさにこのパラダイム転換を指していると感じます。

リファクタリングとは何でしょうか。

それは、蓄積したエントロピー（複雑性・重複・腐敗）を排出するための、コードベースにおけるオートファジーだと捉えられます。

細胞がユビキチン化で古いタンパク質に「分解マーク」を付け、プロテアソームやオートファジーで除去・リサイクルするように、エンジニアは継続的なリファクタリングと設計更新によってコードの秩序を維持していきます。

これは「余分な作業」ではありません。生命にとって分解が合成と同等に重要なように、エンジニアリングにとってリファクタリングは機能実装と同等に本質的な活動です。  
AIが生成速度を上げた今こそ、「壊して更新する」サイクルを意識的に設計する必要があります。

| 生命現象 | ソフトウェア工学における対応 |
| --- | --- |
| 代謝（分解と合成のサイクル） | リファクタリング＋機能開発のサイクル |
| オートファジー（自己消化・リサイクル） | 不要コード削除、依存関係の整理 |
| ユビキチン化（分解マーク付与） | 静的解析・コードレビューによる問題箇所の可視化 |
| 新陳代謝（物質の入れ替え） | 継続的インテグレーションと設計更新のサイクル |
| 散逸構造（外界へのエントロピー放出） | ドキュメントやコラム記事の執筆による暗黙知の外部化 |
| 負のエントロピーの摂取 | 設計パターン・アーキテクチャ原則の取り込み |
| 利他（自己の分解物が他者の栄養に） | 知識共有・オープンソース・技術コラム |

## 7. 書くことも、エントロピーにあらがう行為だ

最後に、少し視点を変えてみたいと思います。

頭の中にある暗黙知、実務経験、失敗から学んだ判断 ─ これらは、放置すれば消えてしまう揮発性の高い「高エントロピーな情報」です。それを言語化してドキュメントや記事に書くことは、散乱した情報に構造を与え、低エントロピーな秩序として定着させる行為だと考えられます。

散逸構造が外界にエントロピーを放出しながら内部に秩序を作るのと同じ構造です。頭の中のカオスを言葉に「放出」することで、自分の内部に新鮮な思考空間が生まれます。書いた後に頭がスッキリする感覚は、この構造から説明できるように思います。

また、技術記事として発信することは、自分のローカルな秩序を組織・社会という大きな系の中に散逸させ、その系全体の秩序を高める利他的な行為でもあります。生命は本質的に利他的であり、自己の分解物は他者の栄養になります。同じように、私たちの失敗談や試行錯誤の記録も、誰かの学びや安心につながっていきます。

## おわりに ─ 動的平衡として流れ続けること

万博「いのち動的平衡館」のコンセプトには、こんな問いが込められています。

*生命は、なぜ、動き続けるのか。 ― それがエントロピー増大の法則に対抗する唯一の方法だったから。*

エンジニアもまた、毎日コードを書き、壊し、リファクタリングし、ドキュメントを更新し続けています。それは締め切りや評価のためだけではなく、宇宙の大原則であるエントロピー増大に抗おうとする、生き物としての本能的な営みの現れなのかもしれません。

AIエージェントが爆速でコードのエントロピーを生み出す今こそ、私たちは「動的平衡の維持者」としての役割をより明確に意識する必要があります。コードを作るだけでなく、壊し、整え、関係性を守り、秩序を更新し続けることです。

あのパビリオンの屋根が環境との絶え間ない相互作用によって浮かんでいたように、私たちのコードもまた、あやういバランスの上で、今日も動き続けています。
