テキストベースでプログラミングのように音楽を作る

## プログラミングのように音楽を作りたい

仕事でも趣味でもAIエージェントを使うようになって、プログラミングは指示出しに徹してコーディング部分はAIに任せるスタイルで8~9割くらいはいけるようになった体感があります。

一方で音楽を作るAIで言うとSunoのようなプロンプトベースに音声を作るAIは強力なものの、よりコントロール性を持って人間とAIがコライティングしようとするとプログラミングのようにはいかない点を歯痒く感じていました。音声やMIDIはテキストデータではないので、なかなかプログラミングのようにはいきません。

一方でニコニコ大百科のピコカキコ、[ABC記譜法](https://abcnotation.com/) 、[LilyPond](https://lilypond.org/) といったテキストで音声を表す記法はいくつか存在しています。これらをAIエージェントにかませればソースコードのようにテキストで音楽を書いてもらうことができます。一部だけを人間が担当したり一部だけをAIに任せたりもやりやすいです。

個人的にもメロディーや歌詞は自分で考えられるので編曲部分だけ委譲できると楽です。別にそれほどAIに音楽作って欲しいと思っていないので編曲部分だけサポートしてくれるAIが欲しいなと。

これらのフォーマットはそれぞれメリット、デメリットがあるのですが、ピコカキコは主にメロディ特化な点や、LilyPondは人間が書くと記法が少し複雑な点を考慮して**ABC記譜法ベースで作編曲AIエージェントを作る**ことにしました。ABC記譜法はJSのライブラリーの人気もあるので、最終的に簡易DAWみたいなものを作りたくなった時に融通が効きそうな点も選択の理由でした。最悪自力でパーサーもかけるくらいの文法なのもポイント。

## ABC記譜法はこんな文法

さてABC記譜法の構文は下記のような形です。ASCIIテキストで音高・リズム・調号などを書きます。

```
X:1
T: example
M:4/4
L:1/8
Q:1/4=120
K:C
V:1
"C"C2 D2 E2 F2 | "G"G4 "C"c4 |
```

ヘッダで拍子（`M:`）、基準音価（`L:`）、テンポ（`Q:`）、調（`K:`）を決めて、本体は `C` `D` `E` のような音名と長さの数字、小節線 `|` で音符を書きます。小文字は高いオクターブ、`z` は休符です。`"Am"` のようにコードを書くこともできます。`V:1` `V:2` のようにパート分けもできます。

**abc2midiのようなCLIもあり、abcファイルをmidiに変換できます**。EasyABCのようにabcファイルを直接再生させるソフトもありますが、midiをLogic ProやCubaseのようなDAWに読み込ませて音源を割り当てればよりプロ品質の音で鳴らせます。

## ABCでも、素のAIエージェントだと満足する品質にならなかった

中間表現が決まったので汎用LLMに曲を作ってとAIエージェントに依頼してできたらよかったのですがそうはいきません。

プログラミングではGitHubなどに公開リポジトリにソースコードが大量にあり、コーディングエージェントがよくある書き方を拾いやすいです。一方で**音楽は学習や参照の足場になる楽譜・進行・編成の情報が、必ずしも public に揃っていない**ことが多く、同じ感覚でエージェントに任せると頓珍漢か童謡のような曲になりがちです。コーディングエージェントを流用するとチューニングの焦点がプログラミングに向いているので作編曲とはずれているというのもあると思います。

そこで Cursor の Agent Skills として、作曲・和声・編成・MIDI 出力などの各工程を専門 skill に切り出しすことにしました。

スキル分割の参考にしたのは、人間の作編曲におけるコライティングです。1曲をメロディ担当・コード担当・編曲担当のように複数人で分担するのと同じく、工程ごとに skill を分けています。中間ファイルが受け渡し口になるので、ある工程だけ人間が適宜代替することもできます。

## 作編曲AIエージェントの全体設計

MIDI作成までの工程は以下のように設計しました。

1. abc-melody: 1番のメロディ生成
2. abc-chords-over: メロディーにコード付け
3. abc-form-expand: メロディ+コードをフル尺に
4. abc-song-architect: 各パートの演奏楽器構成、担当音域を指定
5. abc-orchestrate: 構成と音域を元に各楽器のパート譜を作成
6. abc-to-midi: abcファイルからmidiを生成

各工程の出力ファイルは以下です。

| ファイル | 中身 |
| --- | --- |
| `melody.abc` | メロディのみ（1番） |
| `melody-chords.abc` | メロディ + コード |
| `melody-chords-full.abc` | フル尺のメロディ＋コード |
| `skeleton.abc` | 編成プラン（スロット on/off と音域ヒント） |
| (`partial.abc`) | 編曲途中の累積スコア |
| `full.abc` | バンド完成譜 |
| `full.mid` | DAW向け Type 1 MIDI |

**段階ごとにファイルを分けることで、LLMの担当範囲を狭く保つ**のが設計の要点です。意図していない編集を防ぐ効果があります。

人間が行う作曲でもメロディ先行の人とコード先行の人がいると思います。両方のパターン試したのですが、自分の聴感上もAIの評価上もメロディー先行の方がいい曲になったのでコード先行は捨てて決め打ちにしました。

また、フル尺を一気に生成すると、メロディが上昇と下降を繰り返すだけの単調なものになりがちでした。そこで **1番だけを先に作り、その後フル尺へ拡張する** やり方にしています。人間がやる作曲でも比較的よくやるやり方だと思います。

## 編曲はバンドでの曲作りをエミュレート

編曲もメロディー+コードのabcをベースに⚪︎⚪︎風に編曲してと依頼するだけだと楽だったのですが、それだと満足するクオリティーにはなりませんでした。

メロディ+コードをピアノ譜にしてからポップスの歌もの風のアレンジにするスキルも試したのですが、構成を考えてから各楽器の専門化に渡す多段階フローには勝てませんでした。

弾き語りではないピアノ譜はざっくり左手: ドラム+ベース、右手: コード+メロディなので右手にコードとメロディーが合体していると分解するのは難しいのと、世の中のピアノ譜はポップスの歌ものから作成されているので、逆方向のピアノからポップスの歌ものへの変換はデータが少なく情報も拡充方向なので素のLLMだと難しいのはそうかもしれません。

最終的には**バンドが曲を作るのをエミュレートして、構成を決めた後にドラム、ベースといったリズム帯を重ねていって、バッキング、リード、装飾とのせていっています**。

## 〇〇風のスタイルはGemini Deep Researchで補強

編曲は単にバンドアレンジ固定だったら楽だったのですが、そこは柔軟にスタイルを持てるようにしています。AIへ〇〇風の曲を作ってという依頼はある程度かなえられるようにしています。

ジャンルとアーティストの二層構造で内部的にスタイルを持つようにしています。ジャンルは割合汎用的なジャンル、アーティストは自分の好きなアーティストを恣意的に選んでいます。

| 層 | 例 | 役割 |
| --- | --- | --- |
| ジャンル（23タグ） | `jpop`, `kpop`, `anime`, `citypop`, `ballad`, … | メロディ輪郭、進行ループ、セクション役割のベース |
| アーティスト overlay | `aiko`, `yoasobi`, `bts`, `beatles`, `taylorswift`, … | ベースジャンルの上に、そのアーティスト寄りの語彙・癖を重ねる |

例えば**どの歌手も音域があるわけですが、代表曲を調べるとAメロの音域はこのあたり、Bメロの音域はこのあたり、サビはこのあたりの音域というのがある程度決まっています**。そういう情報をGeminiのDeep Researchをつかってまとめています。

コード進行もJPOPなら例えば王道進行、小室進行、丸サ進行など定番は名前がついていたりしますね。

バンドの曲ならギターとベースにドラムがいて、アイドルソングならメインの楽器はシンセかもしれません。ジャズっぽい曲ならピアノにサックスにウッドベースにドラムかもしれない。そんな風に楽器構成も調べればわかります。

* リズム: ドラム/シンセドラム
* ベース: エレキベース/ウッドベース
* バッキング: ピアノ/アコギ/エレキギター/キーボード/シンセパッド
* リード: エレキギター/キーボード
* 装飾: シンセ/ブラス/ストリングス

みたいなかんじでリズム x ベース x バッキング x リード x 装飾の組み合わせを合わせ込めれるとなんとなく雰囲気似てきます。

## AIエージェントが書いた楽器の練習本を、楽器エージェントに学ばせる

ドラム・ベース・ギター・鍵盤などの パート専門性 についてはGeminiのDeep Researchも使ったのですが、**本を別のAIエージェントに作らせて学んでもらいました**。この楽器の奏法だとこういうボイシングが多くて、このジャンルだとこんな風に弾くのがパターンというのをまとめたものを練習してもらうというか覚えてもらいました。

例えばBPMがこのくらいでコード進行とメロディーがこれ、Bメロでなる楽器はこれとこれとこれと決めたときに、各楽器はこのあたりの音域で鳴らしてほしいというぼやっとした要求でもらうというのは人間のバンドでもありそうな話です。そういう作業分担をエミュレートしました。

## 比較的編曲は答えがある

こうやって工程をふむと冒頭のような曲ができます。メロディーがいまいちで編曲はまあまあというのが自分の評価です。

**編曲は努力で割と何とかなる**というか調べれば学びやすいですね。コード進行は比較的著作権性が低くネットに楽曲情報をあるというのもあると思います。メロディーは著作権に引っ掛かるので情報が少ないです。

楽曲のコンペの講座などでもメロディーをお買い上げいただくという意識が重要と教えていることもあるので、よりメロディーを考えるほうが作業の自動化では解決しづらい創作性の高い部分かもしれません。
