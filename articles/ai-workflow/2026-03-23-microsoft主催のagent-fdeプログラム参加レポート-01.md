## 1. はじめに

Microsoft社のAgent FDE（Forward Deployed Engineer）プログラムに参加し、Copilot Studioを用いて業務エージェントの試作〜検証までを行いました。

本記事では、プログラムの全体像に加え、選定した業務シナリオの概要、実際に作成したエージェントの概要、そして運用を通じて見えてきた改善点についてご紹介します。

想定読者は、Copilot Studio・生成AI活用にご興味がある方、Copilot Studioで業務エージェントのPoC/MVPを進めたい方です。

## 2. Copilot Studioとは

Copilot Studioは、業務で使う会話型エージェントを作るためのMicrosoft製ツールです。大まかには次の要素を組み合わせて設計できます。

* **会話設計**  
  ユーザーの意図に応じた分岐、入力補助、応答方針
* **業務データ／外部連携**  
  Dataverse、Teams、各種コネクタやAPIとの接続
* **生成AIの活用**  
  要約、分類、文章生成などを会話に組み込む

「問い合わせ対応」「申請支援」「ナレッジ検索」「情報収集・共有」などを、業務プロセスに沿って実装しやすいのが特徴です。

## 3. Agent FDEプログラムとは

### 3-1. 概要

Agent FDEは、価値の高い業務シナリオを選び、要件の整理からMVP構築、学びの整理までを伴走支援するプログラムです。

目的は「可能性の確認」に留まらず、実際に動く形まで落とし込み、次の展開（改善・利用拡大）に繋げることです。

今回、当社から提出した6シナリオのうち、4シナリオを本プログラムで対応することになりました。私はそのうち1シナリオを主担当として進めました。

また、他の3シナリオは当社の張が担当し、作成したエージェントの紹介記事を別途公開しています。

## 4. 選定したシナリオ概要

### 4-1. シナリオの内容：Power Platform 情報共有 Agent

今回選定したのは「**Power Platform情報共有Agent**」です。Power Platform／生成AI関連は情報量が多く、仕様・前提が頻繁に更新されます。

また、ニュース・イベント・案件ナレッジ・社内展開情報などが分散しやすい領域です。そこで、次の流れをエージェントで支援することを狙いました。

* **収集**：各種情報・ナレッジを受け取る／必要に応じてWebから取得
* **整形・要約**：共有向けの短文＋登録向けの項目分解
* **共有**：Teamsチャネルへ投稿
* **蓄積**：Dataverseに登録し、後から検索できる状態にする

### 4-2. 選定理由（解決したい課題）

* 情報が散在し「見つけた人だけが知っている」状態になりがち
* Teamsで共有しても流れて終わり、後から探せない
* 忙しいほど要約・投稿・整理が後回しになり、属人化しやすい

この手の課題は「仕組み化」するより「自動化」した方が効くと思い、エージェント化する方針にしました。

### 4-3. 期待する効果

* 共有の負荷を下げつつ、投稿頻度と情報の粒度を上げる
* Dataverseに貯めて検索・再利用の導線を作る
* 属人性を下げ、チーム全体の学習速度を上げる

## 5. エージェント作成の流れ

本取り組みでは、最初から“完璧な回答品質”を目指すのではなく、まずは最小限で業務フローが回る状態を作ることを最優先にしました。

特に情報共有系のシナリオでは、入力内容の粒度や信頼性、形式にばらつきがあるため、最初から精度を作り込むことは現実的ではありません。

そのため、「入力 → 要約 → 共有 → 登録」という一連の流れをまずは通し、実際の運用を通じて改善していく前提で設計しました。

進め方のポイントは以下です。

* **環境・権限・接続を早期に潰す**  
  Copilot StudioやDataverse、Teams連携は“作る前の準備”が詰まりやすいので、早い段階で接続検証とダミーデータ準備を実施しました。
* **作って触って直す**  
  実際に触ると、Teams投稿の見え方、要約の長さ、分類の粒度、登録項目の過不足など、仕様整理段階では気づかない点が出てきます。そこで、動くものを展開し、使いながら修正を繰り返しました。
* **データ量を増やして揺れを確認する**  
  少量データだと良く見えても、件数が増えると要約品質や分類の一貫性が揺れます。運用を想定してデータ量を増やし、改善ポイントを洗い出しました。

## 7. 実際に運用してみての感触

良かった点は、短期間でも意図したシナリオで動くMVPを作れたことです。また、ニュース投稿エージェントにより最新情報とのタッチポイントが増加し、継続的に情報に触れられる環境が整いました。

一方、課題としては **ニュース要約精度、チャット回答の一貫性、情報収集精度** が挙がりました。入力情報が揺れる領域だからこそ、参照元の明示、登録前の確認ステップ、回答精度の評価基準など、ガードレール設計が次のテーマになりそうです。

また、セミナー・案件の登録はまだ人手対応が中心で登録負荷も高いため、自動化や効率化の余地が大きいと感じています。

## 8. 学び・感想

学びは大きく2つです。

* スコープ設計が成果を決める：最初に「最小で回る業務フロー」を定義し、後で品質を上げる
* 改善サイクル前提が現実的：生成AIの出力は一度で最適化することが難しいため、出力確認→改善→再評価するというサイクルを継続的に回していくことが重要

## 9. まとめ

Agent FDEは、業務シナリオの価値を整理し、動く形まで落とし込んで次の展開に繋げるうえで有効な取り組みでした。

Copilot Studioは、会話設計・連携・生成AI活用を一つの流れで組み立てられるため、業務エージェントのPoC/MVPに向いています。

今回の成果と課題を踏まえ、運用でフィードバックを集めつつ、要約精度や回答品質の改善を進め、活用範囲を広げていきたいと思います。
