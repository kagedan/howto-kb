## はじめに

RAGを導入したのに「複雑な質問の精度が上がらない」「情報が古い」「1回の検索では答えが出ない」──そういった課題に直面したことはないでしょうか。

2026年現在、その解として注目されているのが **Agentic RAG（エージェント型RAG）** です。

従来のRAGが「1回検索して1回生成する」静的なパイプラインだったのに対し、Agentic RAGは **AIエージェントが自ら計画・検索・評価・修正を繰り返す** 動的なアーキテクチャです。

本記事では、従来RAGの限界から始まり、Agentic RAGの仕組み・設計パターン・実装の入り口までを整理します。

---

## 従来RAGの限界

まず前提として、標準RAGの課題を整理しておきます。

標準RAGのフローは単純です。

```
ユーザーの質問 → ベクトル検索 → 関連チャンク取得 → LLMで回答生成
```

このシンプルさが強みですが、2026年の本番環境では以下の3点で限界が指摘されています。

**① 検索精度の問題**  
ナイーブなベクトル検索は本番環境で40%程度の割合で的外れな文書を取得するという報告があります。クエリと文書の語彙が異なる場合（意味的ギャップ）や、チャンクが文の途中で分断されている場合に特に精度が落ちます。

**② 単一ソース・単一パスの制約**  
標準RAGは基本的に1つのデータベースに1回だけ問い合わせます。「まず業界レポートを調べ、次に競合情報を確認し、最後に自社データと照合する」といった多段階の調査が必要な質問には対応できません。

**③ 検索結果の検証ができない**  
取得した情報が正しいか、十分かを判断する仕組みがなく、古い情報や誤情報がそのまま回答に使われるリスクがあります。

---

## Agentic RAGとは

Agentic RAGは、RAGパイプラインにAIエージェントを組み込んだアーキテクチャです。

一言でいうと、**「AIが自分で考えて、何度でも調べ直して、納得いくまで答えを探し続ける仕組み」** です。

エージェントは以下のサイクルを自律的に回します。

```
計画（Plan）  → 質問を分析し、どう調べるか戦略を立てる
行動（Act）   → 検索・ツール呼び出しで情報を取得する
評価（Eval）  → 取得した情報が十分かどうか判断する
修正（Refine）→ 不足していれば別の方法で再検索する
```

このループを必要な回数だけ繰り返してから回答を生成します。

### 従来RAGとの比較

|  | 標準RAG | Agentic RAG |
| --- | --- | --- |
| 検索回数 | 1回（固定） | 必要なだけ繰り返す |
| データソース | 単一DB | 複数ソースを横断 |
| 検索の判断 | 常に検索する | 不要なら検索しない |
| 外部ツール | 基本なし | Web検索・APIも使える |
| コスト | 低（$0.001/クエリ程度） | 中〜高（$0.02〜0.10/クエリ） |
| 複雑な質問への対応 | 弱い | 強い |

## 主要な設計パターン

Agentic RAGには複数のアーキテクチャパターンがあります。代表的な3つを紹介します。

### パターン1：ReAct（Reasoning + Acting）

LLMが「考える（Reasoning）」と「行動する（Acting）」を交互に繰り返すパターンです。LangChainのAgentで最も一般的に使われます。

```
思考: この質問には最新の市場データが必要だ
行動: web_search("2026年 AI市場規模")
観察: 検索結果を取得
思考: 結果に社内データとの比較が必要だ
行動: vector_search("自社売上 2025")
観察: 関連チャンク取得
思考: 十分な情報が揃った。回答を生成する
```

### パターン2：Corrective RAG（CRAG）

取得した文書の関連度をスコアリングし、スコアが低い場合はWeb検索にフォールバックするパターンです。信頼性の低い検索結果を自動的に補正できます。

```
# CRAGの概念的な実装
def corrective_rag(query, vector_store, threshold=0.7):
    docs = vector_store.similarity_search_with_score(query)
    
    if max(score for _, score in docs) < threshold:
        # 関連度が低い → Web検索にフォールバック
        docs = web_search(query)
    
    return generate_answer(query, docs)
```

### パターン3：Self-RAG

LLMが自分の回答を自己評価し、必要なら追加検索を繰り返す設計です。「この回答は根拠があるか？」「検索が本当に必要か？」を自分で判断します。

## LangGraphによる実装パターン

Agentic RAGの実装には **LangGraph** が2026年現在の主流です。エージェントのループ・条件分岐・状態管理をグラフ構造で明示的に書けます。

```
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class AgentState(TypedDict):
    question: str
    documents: List[str]
    generation: str
    retry_count: int

def retrieve(state: AgentState):
    """ベクトルDBから関連文書を取得"""
    docs = vector_store.similarity_search(state["question"])
    return {"documents": docs}

def grade_documents(state: AgentState):
    """取得した文書の関連度を評価"""
    # LLMで関連度をスコアリング
    relevant = [d for d in state["documents"] if is_relevant(d, state["question"])]
    return {"documents": relevant}

def decide_to_generate(state: AgentState):
    """文書が十分なら生成、不十分なら再検索"""
    if len(state["documents"]) > 0 and state["retry_count"] < 3:
        return "generate"
    elif state["retry_count"] < 3:
        return "web_search"
    else:
        return "generate"  # リトライ上限に達したら生成

def generate(state: AgentState):
    """最終回答を生成"""
    answer = llm.invoke(f"Context: {state['documents']}\nQuestion: {state['question']}")
    return {"generation": answer}

# グラフを構築
workflow = StateGraph(AgentState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("grade_documents", grade_documents)
workflow.add_node("generate", generate)

workflow.set_entry_point("retrieve")
workflow.add_edge("retrieve", "grade_documents")
workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {"generate": "generate", "web_search": "retrieve"}
)
workflow.add_edge("generate", END)

app = workflow.compile()
```

## 評価指標：RAGASで品質を測る

Agentic RAGを本番運用するなら、評価フレームワーク **RAGAS** が標準です。

| 指標 | 意味 | 目標値 |
| --- | --- | --- |
| Faithfulness | 回答が取得文書に忠実か（ハルシネーション防止） | > 0.9 |
| Answer Relevancy | 回答が質問に答えているか | > 0.85 |
| Context Precision | 取得文書が実際に関連していたか | > 0.8 |
| Context Recall | 必要な文書を漏れなく取得できたか | > 0.75 |

Context Precisionが低い場合は検索・チャンキングの見直し、Faithfulnessが低い場合はプロンプト設計の見直しが必要です。

## 注意点：コストとレイテンシのトレードオフ

Agentic RAGは精度が上がる反面、コストと応答時間が増加します。

* 標準RAG：約$0.001/クエリ
* ハイブリッド検索＋リランク：約$0.005/クエリ
* Agentic RAG：約$0.02〜0.10/クエリ

月間10万クエリなら、Agentic RAGで月$2,000〜$10,000の試算になります。すべての用途にAgentic RAGを適用するのではなく、**単純なQ&Aには標準RAG、複雑な多段階タスクにはAgentic RAG** という使い分けが現実的です。

また、エージェントの自律的な動作は無限ループや誤った推論に陥るリスクもあるため、**リトライ回数の上限設定・タイムアウト・Human-in-the-loop** の設計が本番運用には不可欠です。

## まとめ

| 項目 | 内容 |
| --- | --- |
| 何が変わったか | 「1回検索→1回生成」から「自律的な計画・検索・評価ループ」へ |
| 主な設計パターン | ReAct / Corrective RAG / Self-RAG |
| 推奨実装ツール | LangGraph（本番）、LangChain Agent（プロト） |
| 評価フレームワーク | RAGAS（Faithfulness・Answer Relevancy等） |
| 向いているユースケース | 多段階調査・複数ソース横断・複合的な質問対応 |
| 注意点 | コスト増・レイテンシ増・無限ループ対策が必要 |

Agentic RAGは「RAGを賢くするための次の一手」として、2026年の実務導入が急速に進んでいます。まずはLangChainのReActエージェントから試し、精度が安定してきたらLangGraphで本番品質に移行するのが現実的なパスです。
