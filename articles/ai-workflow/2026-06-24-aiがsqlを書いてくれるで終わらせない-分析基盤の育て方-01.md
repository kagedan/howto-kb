トリビューでCTOをしている小尾（[@obiyuta](https://x.com/ob1yuta)）です。  
弊社ではAIを前提とした全社業務の再構築に取り組んでおり、[前回はその中で取り組んでいる社内MCPを載せる共通基盤（認証・認可・ログ）](https://zenn.dev/tribeau/articles/0d03b4c9fdf360)をご紹介しました。  
今回は具体的な活用事例として、非エンジニアが自分で本番データを分析できるような分析基盤と、作って終わらない改善ループをどう構築、運用しているかご紹介します。

## 「SQLが書ける」と「正しく答えられる」は違う

分析基盤そのものは特殊なものではなく、ClaudeやNotion AIなどから自然言語で問いを投げると、SQLを組み立てて、BigQueryで実行して結果を返すという、至ってシンプルな仕組みです。実際にこういったtext-to-SQLの事例は各社でも紹介されており、もう特別なことではなくなってきました。

一方で、意思決定をするにあたって必要な情報を正しく安全に提供するためには、**「AIがSQLを書いてくれる」だけの仕組みでは実用に足りないのも事実です。**  
実際に構築・運用する中でも、ツールだけではなくコンテキストやガードレール、そしてそれらを使いながら改善するループ（仕組み）までを **使いながら育てていける基盤** として整えることが、現場で本当に求められているクオリティでツールを"提供し続ける"ために重要だと強く実感しています。

## 分析基盤の全体像

現状の全体像は以下の図の通りです。細い線が日々の分析の流れ（問いを投げて、検証して、正しい数字が返るまで）。太い線が、その問いを記録して文脈そのものを育てていくループになっています。

![分析基盤の全体構成図](https://static.zenn.studio/user-upload/c89f5cd1e985-20260624.png)

構成要素として大きく4つです。

| 要素 | 役割 | 具体 |
| --- | --- | --- |
| **ツール** | AIとデータのインターフェース | データ分析MCP（自然言語 → SQL → BigQuery） |
| **ガードレール** | 誰でも安全に利用できる仕組み・ブロッカー | SQLの構造検証、コスト見積もり、認可・権限 |
| **コンテキスト** | AIに渡す文脈 | 1. メトリクス（正しい数字・指標の定義）: dbt、2. セマンティック（機能・イベントの定義）、3. インテント（その時の問い） |
| **改善のループ** | 利用ログを基にコンテキストを改善・改訂し続ける | インテントの記録 → dbt化・型の固定・モデルや定義（ドキュメント）の補強 |

「AIがSQLを書いてくれる」のはツールの部分で、**基盤全体から見ればごく一部です**。問いに正しく答えるためのコンテキスト、全員が安全に使えるガードレール、変化に対応していくための改善のループまでを1つの基盤として整備しています。

すべて実稼働していますが、仕上がりはまちまち。ツールやガードレール、コンテキストの中でもメトリクスに近い領域は、これまで整備してきたアセットに乗れたこともあり安定稼働していますが、セマンティック（機能やイベントの定義）、インテントなどを起点とした改善のループはまだ途上です。この改善のループこそ、弊社がいままさに取り組んでいる次のチャレンジでもあります。

## データで意思決定するために、本当に必要なものはなにか

「誰でも自分でデータを見て判断する」を本気でやろうとすると、**人が増えるほど、間違ったデータで意思決定される危険も増えます**。これまでは分析担当やエンジニアが「この数字はこう解釈すべき」「このテーブルにはこういう癖がある」と暗黙にチェックしていましたが、それを別の形で担保しないといけません。

**危ない操作**（勝手に`DELETE`を流されてしまった、外部にデータを送り出してしまった）や、**コストの暴走**（うっかり巨大なテーブルを全期間スキャンしてしまった）はLLMを使っていると平気で起きえますが、こういったものは比較的単純なガードレールでブロックできます。  
一方で、 **しれっと間違った数字** は非常に厄介です。SQLは構文として正しくても、JOINがずれたりフィルタが一つ抜けたりすると、エラーにならず「それっぽいけど間違った数字」を返す。利用者はそれを正しいと思って意思決定してしまう場合もあります。

たとえば「昨日のアクティブユーザーは？」くらいならわざわざAIに聞く必要もありません。業務に必要なのはもっと込み入った問いで、少なくとも以下のようなものだと思います（実際にはさらに複雑なケースが多いです）。

> 去年のQ4に広告経由で新規登録した会員のうち、初回予約まで到達した人の30日／90日の継続率を、媒体×初回カテゴリ別に出して、オーガニック経由と比較したい

「新規登録会員とは誰か」「広告経由とオーガニックの境目はどこか」「初回予約まで到達したとはどの状態か」「コホートの起点（登録月）をいつに揃えるか」「継続をどの行動で測るか」などを、これまで直接データを扱っていなかったメンバーが正しく扱えることが必要になります。問いが複雑になるほどサイレントエラーの確率は上がりますし、モデルの性能が向上しても解決できる問題ではなく、**AIに渡す文脈**が重要です。

弊社では、安全の仕組みも、文脈を整える仕組みも、「あったら便利な追加機能」ではなく、この方針を成立させるための前提条件として整備しています。ここからは、ツール・ガードレール・コンテキスト・改善ループの順に、一つずつ見ていきます。

## ツール

まず、利用者から見えるMCPのツールです。提供しているのは5つのみです。

| ツール | 役割 |
| --- | --- |
| `list_models` | どんなデータ（dbtモデル）があるかを探す |
| `describe_model` | そのモデルの列・説明・つながり（lineage）を見る |
| `get_aurora_schema` | dbtに無い生データ寄りのテーブル定義を見る（フォールバック用） |
| `dry_run_query` | SQLを実行せずに、走らせたらどれくらいスキャンするか・コストいくらかを見積もる |
| `execute_query` | 実際に実行して結果を受け取る（事前のdry\_runが必須） |

流れとしては、利用者が「○○を集計して」に対して、まず`list_models` / `describe_model`で「どのモデルのどの列を使えばいいか」を調べ、SQLを組み立てる。いきなり実行せず`dry_run_query`でコストを見積もって、問題なければ`execute_query`で実行して結果を返します。  
探索にあたってはまずdbtモデルを見るようにツールのdescriptionやAIのInstructionが定義されています。定義や数値の正確性が担保された定義済み・集計済みのモデルを正とし、生データが要るときだけ`get_aurora_schema`にフォールバックする流れになっています。

## ガードレール

SQLの検証・認可・コストといったガードレールは、サーバー側に敷いています。発想自体は各社の事例と同じですが、「**危ないものを止める**」のではなく「**安全と分かっているものだけ通す**」ことを徹底しているのが特徴です。たとえば:

* **見積もりを飛ばせない**: `dry_run_query`時に発行したトークンが無いと`execute_query`が動かないので、コスト見積もりを必ず通す
* **構文木で検証**: SQLは実行前に構文木に分解し、許可リストにある構造だけを通す。通すのは`SELECT` / `WITH`系の読み取りのみで、書き込みや外部実行・推論系（`ML.PREDICT`など）、許可外のdataset参照は弾く

```
-- 弾く例: SELECT 以外（DML / DDL）は許可リストにそもそも無い
DELETE FROM reservations WHERE created_at < '2026-01-01';

-- 弾く例: 外部実行・推論系の関数（副作用や送信先が読めない）
SELECT * FROM ML.PREDICT(MODEL `...`, TABLE `...`);
```

## コンテキスト

ここからはまだ取り組み自体も途上です。ツールとガードレール（安全）が揃っても、できるのは「SQLを安全に書いて実行する」までです。業務の問いに正しく答えるには、AIに渡す文脈が要る。さっきの3層のうち、メトリクス（正しい数字）とセマンティック（機能やイベントの定義）をここで、インテント（その時の問い）は次の改善のループで整えます。

### 1. メトリクス

BigQueryをデータウェアハウスにして、本番アプリケーションDB（AWS Aurora MySQL）、行動イベント、広告データなど、原則すべてのデータソースを集約しています。  
そしてその上に、dbtで整備されたモデル群が存在しています。生データを整える正規化層、結合・変換する統合層、用途別のマートと段階を分け、200を超えるモデルとして定義されています。テストは1,000件規模で、どのモデルがどこから来ているか（lineage）も全面的に張られています。

**このdbt基盤は、今回の取り組み以前から分析チームが時間をかけて整備を進めてきたものです。** 「初回」とはなにか、「売上」になにを含めるかといった指標の定義も、このdbtモデルに埋め込まれており、データ分析MCPは、そのアセットに乗れているからこそ、ゼロから「正しい数字」を作らずに済んでいます。  
`list_models` / `describe_model`が見ているのは、dbtがコンパイルした成果物（manifest.json）です。dbtのCIがビルドしてクラウドストレージ（GCS）に置き、MCPは実行時にそれを取ってきて読みます（5分ごとに最新を取りに行く）。MCP側にモデル定義のコピーを持たないので、dbtにモデルが増えれば、次にmanifestを読んだ時点でMCPからも見えるようになります。

### 2. セマンティック

メトリクスが整っていても、それだけで業務の問いに答えきれるわけではありません。「この『予約』はどの機能の予約か」「コードや画面で何と呼ばれているか」「どんなイベント・ログが取れているのか」といったデータと直接紐づかないプロダクトのコンテキストです。  
弊社では、どんな機能があってコードや画面で何と呼ばれているか、各機能や画面と実際のクラスやコード、計測イベントのマッピングを用意し、そこからAIが読めるドキュメントに自動生成しています。  
現在はこのカタログは分析MCPからは参照させておらず、AIクライアント上で分析MCPとこのカタログドキュメントの両方を用いてリサーチや分析を行えるようにしています。

## 改善のループ

市場、事業、プロダクトや人の問いの変化に対応し続けられる**メトリクスやセマンティックの改善・改訂のループが非常に重要です。**  
弊社がいま一番力を入れている部分であり、まだ整備途中の部分でもあります。

具体的には、`execute_query`のときに、LLMがSQLに変換する前の**元々の自然言語での問い**（**インテント**）を一緒に記録しています。  
「この人はこういう問いを立てて、それをこのSQLで確かめようとした」を実行と紐付けて残しておくことで、足りないコンテキストを補い、ズレを正していくことができます。

```
// MCPが記録するintentログ（BigQueryに蓄積）
{
  "code": "QUERY_INTENT",
  "user_email": "pdm@example.com",
  "group_email": "analyst@example.com",
  "intent": "先月の初回予約からのリピート率を媒体別に見たい",  // 元の自然言語の問い
  "job_id": "bquxjob_xxxx"  // この問いを確かめたBigQueryジョブ
}
```

あとは、これをBigQueryが自動で記録しているジョブ情報（`INFORMATION_SCHEMA.JOBS_BY_PROJECT`）と`job_id`で突き合わせるだけです。（実際に走ったSQL・スキャン量・参照テーブルをすべて併せて取得できます）

```
SELECT
  i.intent,  -- 元の自然言語の問い
  j.query AS executed_sql,  -- 実際のクエリ
  j.total_bytes_billed,
  j.total_slot_ms,
  TIMESTAMP_DIFF(j.end_time, j.creation_time, MILLISECOND) AS duration_ms,
  j.referenced_tables,
  j.state,
  j.error_result
FROM `mcp-project.mcp_observability.intent_log` i
JOIN `analytics-project.region-asia-northeast1.INFORMATION_SCHEMA.JOBS_BY_PROJECT` j
  ON j.job_id = i.job_id
WHERE i.code = 'QUERY_INTENT'
ORDER BY i.timestamp DESC
```

こうして「どの問いが、どのテーブルを、どれだけ叩いたか」を一覧にし、以下のような改善のアクションのトリガーにしています。  
現状は人が定期的にモニタリングを行っていますが、自動化の検証など自律的な仕組みを目指しています。実例としては以下のようなものです。

| ログから見える兆候 | 改善アクション |
| --- | --- |
| dbtモデル外のテーブルを叩いている | **dbtに寄せる**：整備済みモデルでなく生データを直接見にいっている場合は、dbtモデル化の候補にする |
| 同じ問いでSQLがばらつく | **テンプレ化する**：「リピート率」のような同じ問いに毎回違うSQLが当たっているなら、テンプレート化するなど毎回同じ組み立てになるようにする |
| 重いクエリが繰り返される | **集計済みのモデルを用意する**：スキャン量の大きいクエリはモデルに追加して、コストと速度を改善する |
| 詰まり・エラーが多い | **説明やlineageを補強する**：同じ問いで何度も詰まっているようならモデルの説明や列名を修正する |

## 課題・今後の取り組み

個人の分析にかかる時間は大幅に短縮されましたが、より自律的な仕組みを提供していくためには残課題もあります。

**メトリクス・セマンティックの拡充と活用**  
機能やイベントの定義の精度やメンテナンスの問題、定量情報に判断が寄ってしまう問題、プロダクト領域に偏ってしまう問題など多くの課題があります。  
こうした情報の整備と共に、MCPにはいま使っているToolsのほかにPromptsやResourcesという仕組みを活用して、一元的に提供していくことも検討しています。

**見える化から、気づいて提案する側へ**  
いまは人が問いを投げる形ですが、受動的な仕組みに留まります。KPIの異変にAIが気づいて、背景や次の打ち手まで提案することにも取り組み始めています。  
その中で事業のコンテキストや、BtoBtoCプラットフォームだからこそユーザー、クリニック両面の課題を理解した関わり方も模索中です。

## まとめ

データの抽出や分析にかかっていた時間は、分析MCPをはじめとするツールの整備で大きく短縮できました。  
ただ、**本当に効くのはツールそのものよりも、正しい数字を担保し、使いながら改善し続ける仕組みのほうです。**「AIがSQLを書いてくれる」で終わらせず、安全に渡す仕組みと、正しく答えるための文脈までをセットで基盤として整えていくことが重要です。  
今後AIのモデル性能が上がるにつれ、さらに重要性やインパクトも上がっていくと考えられますので、引き続きコツコツと整備を進めていければと思います。

あらためて実際の活用事例などもご紹介させていただきます。
