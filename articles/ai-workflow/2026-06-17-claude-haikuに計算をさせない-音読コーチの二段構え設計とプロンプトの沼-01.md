## この記事の位置づけ

音読速度計測アプリで、計測した数値を **Claude Haiku** に渡して「褒め1文 + 改善1つ」のコーチングを生成する部分を扱います。

生成AIを使った記事の多くは「認識テキストをそのままClaudeに渡して、いい感じに評価して」と丸投げします。本記事のテーマは、その逆です。**LLMに計算をさせない**。計算とルール判定はすべてコードで確定させ、Haikuには**言語化だけ**をさせる。この「二段構え」をどう設計し、実装でどんな沼にハマったかをまとめます。

扱うのは次の4点です。

* なぜ「計算はコード・言語化だけHaiku」なのか（役割分担の設計）
* 確定版プロンプトと、`messages` / `system` / `cache_control` の実装
* **プロンプトキャッシュが効かなかった**という検証の一次情報
* **「プロンプトに書いた ≠ 守られる」** を実データで突きつけられた話

---

## なぜHaikuに「計算」をさせないのか

最初に結論を言うと、**フィードバック生成にHaikuで十分**でした。むしろHaikuが得意な形に持ち込めます。鍵は役割分担です。

評価指標の計算（発話速度・淀み率・しきい値判定）は、**全部コード側で確定**させます。そのため、Haikuに渡る頃にはこういう「**整理済みの事実**」になっています（以下は開発中に平家物語サンプルを音読して得た実測を `labelMetrics` した例。`stagnationRate` はパーセント表示用の数値）:

```
{
  "pureSpeakingSpeed": 322,
  "pureSpeakingSpeedEvaluation": "やや速い",
  "stagnationRate": 0,
  "stagnationRateEvaluation": "少ない"
}
```

Haikuの仕事は、この数字とラベルを**温かく具体的な言葉に翻訳すること**だけ。これは小さなモデルが最も得意とする「言語化・要約・トーン調整」で、計算精度を要求しません。

逆に、Haikuに**させないこと**を明確に決めました。

* 精密な数値計算（速度やスコアの算出）→ コードでやる
* 多段のしきい値分岐（「322文字/分は速いか？」の判断）→ コードで「やや速い」と解釈してから渡す

この分離には、コスト以外の実利もあります。**しきい値判定をコードに寄せると、出力が安定します。** 「速いかどうか」をHaikuに毎回判断させると気分でブレますが、「やや速い」というラベルにしてから渡せば、翻訳の幅しか揺れません。

## 確定版プロンプト

システムプロンプト（静的なコーチのペルソナ）です。何度か実データで回して、最終的にこの形に落ち着きました。

```
あなたは日本語の音読を見てくれる、優しく具体的なコーチです。

# 前提（厳守）
- 渡されるJSONは、すでに計算・評価が確定した事実です。
- 数値を再計算しない。新しい指標を作らない。JSONにない事実を推測で補わない。
- JSONに書かれていることだけを根拠に話す。
- 句読点には言及しない（サンプルが短く句読点を含まない場合があるため）。

# 出力形式（厳守）
- 敬体。温かく落ち着いた口調。
- 絵文字・記号・改行・見出し・前置きは使わない。本文の2文だけを出力する。

# 出力形式（努力目標）
- 全体で日本語100字程度。

# 出力内容
1文目：最も良かった点を1つだけ選び、具体的に褒める。
2文目：改善点を1つだけ、次に試せる具体的な行動として伝える。

# 改善点の選び方（上から順に、最初に当てはまった1つだけ）
1.「stagnationRateEvaluation」が「やや多い」以上 → 間（ま）の取り方
2.「pureSpeakingSpeedEvaluation」が「やや遅い」「遅い」 → テンポを少しずつ前へ
3.「pureSpeakingSpeedEvaluation」が「やや速い」「速い」 → 意識して一呼吸おき、ゆっくり読む
4. どれにも当てはまらない → 今いちばん伸ばせる点を1つ、次の目標として前向きに伝える
```

設計の意図を3つ補足します。

* **改善点に優先順位を付け、最初に当たった1つだけ**を出させています。全項目が良好なときに無理やり粗を探させない第4分岐（フォールバック）も置きました。
* **「本文の2文だけ」と明示**しないと、前置きや箇条書きを足してきます。出力の純度はプロンプトで縛ります。
* **「100字程度」は努力目標**にしました。理由は後述しますが、LLMは日本語の文字数を正確に数えられないからです。

## API実装：`system` / `messages` / `cache_control`

Claude Messages API（`@anthropic-ai/sdk`）の `messages.create` で呼びます。

```
const result = await client.messages.create({
  model: process.env.ANTHROPIC_MODEL!, // 例: claude-haiku-4-5
  max_tokens: 256,
  system: [
    {
      type: "text",
      text: FEEDBACK_PROMPT,
      cache_control: { type: "ephemeral" }, // systemブロックの中に入れる
    },
  ],
  messages: [
    { role: "user", content: JSON.stringify(feedbackFacts) }, // 整理済みの事実だけ
  ],
});
```

構造の意味を整理します。

* **`system`** = Claudeの人格・前提ルール（会話の外側）。ここに静的なコーチのペルソナを置く
* **`messages` の `role: "user"`** = 実際の会話入力（毎回変わる評価JSON）
* **`role`** は `"user"`（こちらの入力）と `"assistant"`（Claudeの返答）の2種類。今回は1往復完結なので `user` 1件で足ります

### `max_tokens` の考え方

公式例の流用で当初 `1024` にしていましたが、今回の出力（2文）は実測で**88トークン程度**でした。`max_tokens` は「必ず生成する量」ではなく「生成してよい上限」なので、多めでも実害はほぼありません。出力が終われば自然に止まります。意図しない長文を防ぐ安全弁として、上のコードのように `256` まで絞りました。

## 沼①：プロンプトキャッシュが「効かなかった」

ここからが、検証して初めて分かった一次情報です。

静的な `system` プロンプトは毎回同じなので、**プロンプトキャッシュ**で安くできると考えました。`cache_control: { type: "ephemeral" }` を付け、損益分岐も計算し（キャッシュ書き込みは初回1.25倍、2回目以降の読み込みは約90%オフなので「2回使えば元が取れる」）、「これは使うべき」と判断していました。

**ところが、実際には効いていませんでした。**

理由は **最小トークン数の壁**です。Claude Haiku 4.5 のキャッシュ最小トークン数は **4,096トークン**。今回の `system` プロンプトは数百トークン程度で、条件に届いていませんでした。`cache_control` を書いても、実際にはキャッシュされていなかったのです。

確認はレスポンスの `usage` で取れます。

```
console.log(result.usage);
// cache_creation_input_tokens: 0
// cache_read_input_tokens: 0   ← 両方0ならキャッシュされていない
```

両方 `0` でした。損益分岐の計算に夢中で、**そもそもの最小トークン条件が盲点**だったわけです。

`cache_control` を書いても動作上の害はありません（無視されるだけ）。ですがこのアプリで「キャッシュで最適化した」と書くのは事実とズレます。**効くと思って調べたら条件に届かず効かなかった、という検証の過程**として残すのが、いちばん誠実で、読者にとっても有益だと判断しました。

## 沼②：「プロンプトに書いた ≠ 守られる」

もう1つの沼です。実データで音読すると、こんなことが起きました。

* 10秒程度の短いサンプルを読む → 認識テキストに**句読点が含まれない**
* それなのにHaikuが「**句読点で一呼吸置きましょう**」と、入力にない具体策を付け足してくる

原因は、確定版プロンプトのルールが「句読点で落ち着く」のように**句読点を前提にした表現**を持っていたことでした。対策として、

1. 前提に「句読点には言及しない」を1行追加
2. ルール3を「句読点で落ち着く」→「意識して一呼吸おき、ゆっくり読む」に変更（句読点非依存の表現へ）

と修正しました。ただし、これで**100%守られる保証はありません**。LLMは「〜しない」という禁止指示も、確率的に破ることがあります。

実は同じ実行で、もう1つブレが出ていました。入力は淀み率「やや多い」+ 速度「遅い」で、ルール上は**ルール1（間の取り方）が優先のはず**。なのに出力はルール2（テンポを前へ）寄りでした。これは優先順位の判断をプロンプトに書いてHaikuに選ばせている設計の限界が表面化したものです。

ここで効いたのが、過去の学び「**テストが通る ≠ 意図通りに動く**」と同じ構図です。**指示したかどうかではなく、守られたかを実データで検証する。** プロンプトに書いた＝守られる、ではありません。

### 安定させたいなら「コード側で確定して渡す」

出力を完全に安定させたければ、**どの改善点を出すかをコード側で決め**、`{ "改善フォーカス": "間の取り方" }` のように1フィールドで渡す方法があります。こうするとHaikuは言語化だけに専念でき、優先順位のブレが消えます。

| 方式 | 内容 | トレードオフ |
| --- | --- | --- |
| 案A（今回採用） | 優先順位をプロンプトに書いてHaikuに選ばせる | 賢く見えるが、ブレる |
| 案B | 改善フォーカスをコードで確定して1フィールドで渡す | 出力が安定するが、固い |

締切の都合で今回は案Aのまま（プロンプト修正で様子見）にしましたが、本番でブレが目立つなら案Bに倒すのが確実、という見通しを持っています。これも「賢く見せる vs 安定」という設計判断のポイントでした。

## 文字数制限はどう扱うか（Phase 1）

「100字以内」をHaiku自身にカウントさせるのは信頼できません。LLMは日本語の文字数を正確に数えられず、超過して返すことが普通にあります。Phase 1 では次のとおりです。

* プロンプトの「100字程度」は**努力目標**として残す
* 実際の上限は `max_tokens: 256` で持つ
* 超過時の切り詰めなど**コード側バリデーションは Phase 1 では未実装**（将来の選択肢）

沼②と同様、ここでも「プロンプトは仕様、厳密な保証はまだコードにない」という状態です。

## まとめ

Claude Haikuで音読コーチのフィードバックを生成する設計をまとめました。要点は3つです。

1. **計算はコード・言語化だけHaiku**の二段構え。LLMを最も得意な「翻訳」に専念させ、判定はコードで確定する
2. **プロンプトキャッシュは今回のアプリでは効かなかった**（Haikuの最小4,096トークンに届かない）。最適化は前提条件まで確認しないと「したつもり」になる
3. **プロンプトに書いた ≠ 守られる。** 句読点・優先順位のブレを実データで確認し、安定が要るなら判定をコード側に寄せる

通底するのは「**指示と検証は別**」という一点です。プロンプトは仕様であって保証ではない——その前提で、検証をコードと実データに持つのが、生成AIを実プロダクトで使うときの誠実な姿勢だと考えています。
