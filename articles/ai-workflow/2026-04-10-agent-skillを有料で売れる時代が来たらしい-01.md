# agent skill は有料で売られる

これは願望ではなく、構造の話だ。

AIが会話の道具だった時代は終わった。  
いま価値があるのは、うまい受け答えではない。  
仕事を安定して前に進める実行構造だ。

そして、その実行構造のうち、流通可能な単位が skill だ。  
だから skill は商品になる。

## そもそも skill とは何か

skill は、単なる長いプロンプトではない。  
思いつきのメモでもない。

skill は、再利用可能な実行仕様書だ。  
この種類の仕事が来たら、この手順で、この形式で、この品質基準を満たして実行しろ。  
それを agent に教えるためのプレイブックである。

ここを曖昧にすると全部ずれる。

skill は model ではない。  
tool 単体でもない。  
memory 単体でもない。  
agent 単体でもない。

skill は、仕事単位で切り出された実行知だ。  
必要な時に読み込まれ、その場の仕事の進め方を支配する。

つまり、skill の本質は文章ではない。  
再現性だ。

## harness engineering は仕事になる

なるどころか、すでに始まっている。

AI運用で本当に差がつくのは、モデルの名前ではない。  
何をどこに書くかだ。

* prompt に書くのか
* AGENTS.md に置くのか
* skill に切るのか
* 常設ルールにするのか
* 実行時だけ読む文脈にするのか

この設計が甘いと、全部崩れる。  
逆にここが強いと、同じモデルでも成果が変わる。

僕はこれをハーネスエンジニアリングと呼んでいる。  
ここでは、そう定義するという話だ。  
もちろん、guard、context、tool、memory、supervision で構成されるものをハーネスとする考えや、evaluator、planner、generator の並列運用のようなものをハーネスとみなす考えがあることも承知している。  
そのうえで、僕はこれを AI に仕事をさせるための構造設計として捉えている。

現実のタスクは、きれいに分類できない。  
skill と AGENTS.md の境界も曖昧だ。  
plan でもあり prompt でもあるものも多い。

だから必要なのは知識ではない。  
境界を切り、組み合わせる能力だ。

何を重複させ、何を分離し、何を常設し、何を呼び出し型にするか。  
この判断が、そのまま AI 運用能力になる。

ハーネスエンジニアリングは、普通に専門性になる。  
そして専門性になるなら、その構成要素にも価値がつく。  
その最小の売買単位が skill だ。

## skill が価値を持つのは当然だ

skill は売れる。  
なぜなら、仕事のやり方そのものだからだ。

もう一度言う。  
売られるのは知識そのものではない。  
知識をどう使って成果に変えるかだ。

良い skill には、次の要素がある。

つまり、1つの skill は 1つの仕事に収束している。  
これが商品にならない理由がない。

逆に、価値のない skill は広すぎる。  
何でもできる万能 skill は、だいたい何もできない。  
agent 全体の人格や、会社の仕事全部をまとめた巨大 instruction は商品ではない。  
それは設計ミスだ。

価値があるのは、ちゃんと責務が切れている skill だけだ。

## 一番大きい価値は、「知らない領域の仕事のやり方」を完成品で得られること

これが本質だ。

skill の価値は、単に精度が上がることではない。  
自分が知らない領域に、完成済みのワークフローごと入れることだ。

* 何を最初に見ればいいのか
* どこで止まればいいのか
* どの資料を参照すればいいのか
* どういう順番で進めれば破綻しないのか

この設計を、自分でゼロから作るのは重い。  
領域知識が要る。  
試行錯誤が要る。  
失敗も要る。

でも、優れた skill はそこを一気に飛ばす。  
つまり skill は、情報の販売ではない。  
参入障壁の販売だ。

自分で学べば半年かかる仕事の入り口を、数分で越えられる。  
そこに価値がないわけがない。

## では、なぜ今まで市場にならなかったのか

理由は単純だ。  
本当に価値ある skill は公開されづらいからだ。

強い skill は、そのまま競争優位になる。  
社内に閉じる。  
個人で囲い込む。  
断片だけ無料公開する。  
それで終わる。

無料で配るインセンティブは弱い。  
でも、隠しておくインセンティブは強い。

だから、価値ある実行知は市場に出てこない。  
これは才能の問題ではない。  
構造の問題だ。

ここを変えるには、精神論では足りない。  
必要なのは、公開する経済合理性だ。

* 売れるなら出す
* 評価されるなら磨く
* 収益になるなら継続する

当たり前の話だ。  
そして、この当たり前が今まで存在しなかった。

だから市場を作る必要がある。

## SkillShelf は、その市場を作る

GitHub は公開には向いている。  
X は拡散には向いている。  
でも、どちらも「実行知を商品として探し、比較し、買う場所」ではない。

その空白がある。  
そして、その空白は埋まる。

なぜなら、AIの価値の中心が、回答から運用へ移ったからだ。  
運用が価値の中心なら、運用知が市場の中心になる。  
skill はそこに入る。

これはニッチの話ではない。  
カテゴリの誕生だ。

SkillShelf は、そのカテゴリに対して作っている。  
Agent skill を、単なる配布物ではなく、値段のつく実行知として流通させるためのマーケットプレイスだ。

強い workflow を持つ人が、それを商品として公開できる。  
買う側は、自分の知らない領域の実行構造を、完成した形で導入できる。  
公開されなかった知識に、公開される理由が生まれる。

SkillShelf がやりたいのは、単なる販売ではない。  
AI時代の実行知の市場を作ることだ。

## Founding Author を募集しています

いま SkillShelf では、Founding Author を募集している。

Founding Author は、初期に参加する著者のことだ。  
ただ早い人、ではない。  
最初の市場を一緒に定義する人だ。

初期著者は重要だ。  
なぜなら、初期の棚がそのまま市場の基準になるからだ。

Founding Author には、以下を用意している。

* 恒久バッジ
* ローンチ初期の優先掲載
* Founding Authors ページへの恒久掲載

正直に言う。  
僕が欲しいのは、ただの出品者ではない。

* 自分の workflow に自信がある人
* 実行知を言語化できる人
* 単なる便利メモではなく、再利用可能な skill を持っている人
* この市場の最初の基準を作る側に立ちたい人

そういう人に来てほしい。

## 最後に

skill は売られる。  
ハーネスエンジニアリングは仕事になる。  
そして、その世界では実行知が資産になる。

もう始まっている。

あとは、誰が先に市場を定義するかだけだ。

僕は SkillShelf でそれをやる。  
そして今、その最初の著者を集めている。

自分の skill に値段がつく未来ではなく、  
値段がつくのが当然になる未来を前提に動いている。

興味があるなら、今のうちに来てほしい。  
後から入るより、最初の棚を作る側に回るほうがいい。
