## はじめに

プログラミングを本格的に学んだことのない自分が、AI（Claude と Codex）と一緒に、動画や音声をローカルで文字起こしする Web アプリ「もじおこし」を作りました。この記事はその制作の記録です。完成品の紹介というより、「なぜ作ったか」「どこで詰まって、どう抜けたか」を残すことを目的にしています。同じように AI と何か作ってみたい人の道しるべになれば嬉しいです。

もじおこしは、動画や音声をブラウザから投入すると、自宅の PC 上で Whisper が文字起こしをして、要約用のプロンプトごとワンタップでコピーできる Web アプリです。コピーした文章をそのまま ChatGPT に貼れば、講義ノートや要点整理が返ってきます。外部の API は使わず、文字起こしは全部自宅の PC の中で完結します。

このアプリは自宅の PC でだけ動く自分用の道具なので、「今すぐ試せる公開版」はありません。そのぶん、この記事では「どう作ったか」を具体的に書きます。前身にあたる「Whisper 文字起こしアプリの作り方」そのものだと思ってもらえたらと思います。

## 実録データ

* 制作期間: v1 は1日でほぼ形になりました（2026-07-05）。前身のコマンドライン版はその前の月（2026-06）に作っていて、その資産（Python 環境とモデル）を引き継いでいます
* 概算費用: ほぼ0円
  + 文字起こしはローカルの Whisper で、外部 API は使っていません。大前提として、Claude と Codex のサブスク（各 月約3,000円）を常用していますが、この開発でそれ以外の追加費用はかかっていません
* 使った AI / 道具: Claude、Codex、Whisper（small モデル・CPU）、Python、FastAPI、SQLite、素の JavaScript
* 形態: 自宅の PC でサーバーを立て、家庭内の同じネットワークにいるデスクトップ・スマホのブラウザから使うローカル Web アプリ（外部公開なし）

## なぜ作ったか

きっかけは、大学の課題動画でした。正直に言うと、自分はかなり怠惰な学生で、90分の講義動画をまじめに見て要点をまとめる、というのがどうにも億劫でした。そこで「そもそも動画を見ずに、文字起こしを AI に要約させればいい」と思ったのですが、実際にやってみると、文字起こしのテキストファイルを開いて、全文をコピーして、要約の指示を書いて、ChatGPT に貼って……という手作業が毎回発生して、これはこれで面倒でした。

自分の発信活動のテーマは「AI で、人間の面倒くささ・分からなさを少し減らす」です。まさにこの「面倒くささ」を、自分のために減らしてみることにしました。ゴールは、動画をブラウザに放り込んだら、あとは1タップで ChatGPT に渡せる状態まで持っていくことです。

念のため書いておくと、こういう使い方は本来おすすめできるものではありません。講義はきちんと見て、まじめに勉学に取り組むほうが良いに決まっています。自分はそれをサボるために作ってしまったわけですが、少なくとも「面倒だと感じたことを AI で減らす」という練習台にはなりました。真似する人は、ほどほどにしてください。

## 全体像

構成はシンプルにしました。

* サーバーは Python（FastAPI）。ビルド工程は無し
* 画面は素の JavaScript の1枚ページ（フレームワーク無し）
* 文字起こしは Whisper を**別プロセスとして起動**し、その出力を読みながら進捗を表示
* 履歴は SQLite（Python 標準ライブラリ。依存を増やさない）

「動画を投入 → 文字起こし中（進捗と認識テキストが流れる）→ 完了（全文＋要約コピーのパネル）」という3つの状態を行き来するだけの、1画面のアプリです。

## いちばん大事にした設計判断 ― 要約に API を使わない

最初に迷ったのが「要約をどうやるか」でした。素直に考えれば、文字起こしのテキストを要約用の LLM API に投げて、要約まで自動でやるのが親切です。

でもそうすると、API キーの管理が必要になり、使うほどお金がかかり、秘密情報を1つ抱えることになります。「ほぼ0円で、キーも秘密も持たない」という自分の方針と、真正面からぶつかりました。

そこで発想を変えて、**要約そのものはやらない**ことにしました。代わりに、

1. 「講義ノート」「要点箇条書き」「試験対策」の3つの要約プリセット（＝指示文）を用意する
2. 完了画面で選んだプリセットの指示文と、文字起こし全文を、まとめて**1タップでコピー**する
3. あとはユーザーが ChatGPT を開いて貼るだけ

という形にしました。要約の頭脳は ChatGPT に借りて、アプリはそこへ渡す**最後のひと手間だけ**を消す。これなら API キーも料金も秘密もゼロで、面倒くささだけがちゃんと減ります。「必要のない機能（自前の要約 API）は最初から持たない」に倒したことで、設計がぐっと軽くなりました。

## 詰まったところと、抜け方

### 1. 進捗が「十数分に1回」しか流れない

CPU での文字起こしは長くかかります。39分の講義で試したとき、認識中のテキストをリアルタイムに画面へ流したいのに、進捗が**十数分に1回**しか届きませんでした。

原因は、子プロセス（Whisper）の標準出力のバッファリングでした。Python は、出力先がターミナルではなくパイプ（＝別プログラムが受け取る）だと、約8KB たまるまで出力をため込みます。日本語講義のセグメントは1つが短いので、8KB たまるのに十数分かかっていた、というわけです。

抜け方は、子プロセスの環境変数に `PYTHONUNBUFFERED=1` を渡して、ため込みを無効化することでした。

```
# worker.py より（抜粋）
with subprocess.Popen(
    cmd,
    stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT,
    text=True,
    bufsize=1,
    # whisper(Python)はパイプ先へは8KB毎にしか吐かず逐次表示が死ぬため強制無効化
    env={**os.environ, "PYTHONUNBUFFERED": "1"},
) as proc:
    ...
```

「自分のコードは正しいのに動かない」ときに、その先の子プロセスや OS のバッファリングを疑えるようになったのは、この一件の収穫でした。実際に走っている子プロセスの環境変数を `/proc` から覗いて、ちゃんと `PYTHONUNBUFFERED` が届いているかまで確認しています。

### 2. 家の中の HTTP ページでは「コピー」が使えない

このアプリは家庭内のネットワークで、`http://` のページとして開きます。ところが最近のブラウザでは、`navigator.clipboard`（プログラムからクリップボードへ書き込む今どきの API）は「安全なコンテキスト」（＝ `https` か `localhost`）でしか使えません。スマホから LAN 経由で開いた `http://` のページでは、肝心のコピーが動かないのです。

このアプリの主役は「1タップでコピー」なので、ここが動かないのは致命的でした。抜け方は、コピー手段を三段構えにすることでした。

```
async function copyText(text) {
  try {
    // ① 安全なコンテキストなら今どきの API
    if (navigator.clipboard && window.isSecureContext) {
      await navigator.clipboard.writeText(text);
    } else {
      // ② 使えなければ、隠しテキストエリア＋古い execCommand
      const area = document.createElement("textarea");
      area.value = text;
      area.style.position = "fixed";
      area.style.left = "-9999px";
      document.body.appendChild(area);
      area.focus(); area.select();
      if (!document.execCommand("copy")) throw new Error("copy failed");
      area.remove();
    }
    showToast("コピーしました");
  } catch {
    // ③ それも失敗したら、全選択済みのテキストを見せて手動コピーの逃げ道を残す
    const modal = document.getElementById("copyFallback");
    const area = document.getElementById("fallbackText");
    area.value = text;
    modal.classList.add("open");
    setTimeout(() => { area.focus(); area.select(); }, 30);
  }
}
```

①がダメなら②、②もダメなら③、と落ちていきます。③は最後の砦で、全選択済みのテキストを画面に出すだけですが、「どうやっても必ずコピーできる手段が残る」ようにしました。便利機能ほど、動かない環境を想定して逃げ道を用意しておく、という学びでした。

### 3. 進捗バーを、正確でなくていいから出す

Whisper は「今何%終わった」を教えてくれません。そこで、`ffprobe`（動画の情報を調べる道具）で動画の総尺を取り、Whisper が逐次出してくるセグメントのタイムスタンプ（「今 12分34秒の地点を認識中」）と突き合わせて、進捗率を**推定**しました。

この方法は Whisper の出力形式に依存するので、将来出力が変わると壊れます。ただ、そこは割り切って「壊れたら進捗バーは消えて、認識テキストの逐次表示だけに戻る」という退避を用意しました。正確な進捗を頑張って保証するより、外れても致命傷にならない作りにする、という判断です。

### 4. GPU は使わず、CPU に固定した

このPCでは、同じ PC で動かしている別のプログラムが GPU を使っていて、文字起こしでも GPU を使うと取り合いになって不安定になる事情がありました。文字起こしの速度より安定を優先して、Whisper は CPU 実行に固定しました。速・打を作ったときにも似た「GPU の取り合い」に当たっていて、「必要のない環境依存は最初から避ける」を今回も採用した形です。

### 5. ChatGPT に長文をどう渡すか（実際に使って方針が変わった）

設計段階では、「文字起こしが長い（目安8,000字超）ときはコピーを何回かに分割して、順番に貼ってもらう」つもりでした。ChatGPT の貼り付けに上限があると思っていたからです。

ところが、実際に自分で使ってみると、今の ChatGPT は長文の貼り付けを添付ファイルのように受け取ってくれて、強制分割はむしろ邪魔でした。そこで方針を変え、**基本はプロンプト＋全文の一発コピー**に統一し、分割は長文のときだけ選べるオプションに格下げしました。

「使う前に完璧な仕様を決めきる」より、「まず動かして、自分で毎日使って、要らない親切を削る」ほうが良いものになる、と実感した箇所です。

## 常駐のさせ方 ― 使うときだけ起きる

自宅の PC はいろいろ動いていてメモリに余裕がないので、文字起こしアプリを24時間起動しっぱなしにはしたくありませんでした。

そこで、systemd のソケットアクティベーションを使い、**ブラウザからアクセスがあった瞬間にサーバーが起動し、一定時間（30分）使われなければ自分で終了する**形にしました。普段は何も動いておらず、使いたいときにブラウザで開けば勝手に立ち上がります。「常駐」と「省メモリ」を両立させる、地味だけれど気に入っている工夫です。

## AI の使い分け

* **Claude**: 要件を詰めるラリー（「何を作って、何を作らないか」）と、画面デザイン（モックの HTML を先に作ってもらう）を担当。
* **Codex**: サーバーとフロントの実装、レビューの往復を担当。手を動かす実装を任せる相手です。

大まかに「相談・設計は Claude、実装・整形は Codex」という分担で進めています。速・打のときと同じ組み方です。

## おわりに

自分にはアプリ開発の知識がほとんどなく、FastAPI も Whisper も、子プロセスのバッファリングも、最初は何も分かっていませんでした。それでも、「動画の文字起こしが面倒」という自分の困りごとを起点に、AI と一緒に少しずつ切り分けていったら、1日で使える道具になりました。

この記事に書いた詰まりは、どれも AI に丸投げして一発で解けたわけではなく、「どこが原因か」を一緒に切り分けながら抜けていったものです。それでも、専門知識の穴を AI が埋めてくれるおかげで、自分の作りたい道具を、自分の面倒くささに合わせて作れました。

同じように「これ面倒だな」を AI と一緒に解いてみたい人の役に立てば嬉しいです。
