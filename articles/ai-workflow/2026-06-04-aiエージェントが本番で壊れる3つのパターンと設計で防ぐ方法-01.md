**AIエージェントが本番で壊れる3つのパターンとは、ツール呼び出し失敗のハンドリング欠如・状態の肥大化によるコンテキスト上限超過・非決定的な出力への過度な依存であり、これらは設計段階でのエラー文字列化・メッセージトリミング・Pydantic構造化出力の実装で防げます。**

「ローカルでは一発で動いたのに、CloudRunに乗せた途端にループが止まらない」という状況は、この3つのパターンのどれかが抜けているケースにほぼ集約されます。本番投入前にチェックリストと照らし合わせて潰すことが、AIエージェントの安定稼働への近道です。

開発環境では動いていたのに、本番でいきなり止まった。

「ローカルでは一発で動いたのに、CloudRunに乗せた途端にループが止まらない」「エラーログを見ても原因がわからず、結局原因特定に3日かかった」というのは、AIエージェントを実装した人なら一度は経験する状況です。

なぜこうなるのか。原因は大体3つのパターンに収束します。

* ツール呼び出しが失敗したとき、エラーをハンドリングしていない
* 会話履歴や中間状態が肥大化して、コンテキスト上限を超える
* LLMの非決定的な出力に依存しすぎていて、想定外の分岐に入る

本記事ではこの3パターンを一つずつ取り上げ、「なぜ起きるのか」「設計段階で何をしておけば防げるのか」を整理します。悪い例と良い例のコードを並べるので、自分の実装と照らし合わせながら読んでください。

---

## パターン1: ツール呼び出しの失敗をハンドリングしていない

### なぜこれが本番で問題になるのか

開発環境では、API呼び出しは大体成功します。手元のネットワークは安定していて、テストで使うAPIエンドポイントはレートリミットにかかるほどリクエストを送らない。だからローカルでは動く。

本番環境は違います。同時リクエストが増え、外部APIが一時的に落ちたり、タイムアウトが発生したり、レートリミットに引っかかります。このとき、ツール呼び出し側のエラーハンドリングが不十分だと、何が起きるか。

エラーが例外として伝播してエージェント全体が止まるか、例外をキャッチしても `None` や空文字列をLLMに渡し続けてLLMが「何か返ってきたから処理を続けよう」と誤判断します。後者のケースでは、エラーが発生したことをシステムが認識できないまま処理が進み、最終的に意味不明な出力が生成されます。

### 悪い例：エラーをそのまま伝播させる

```
import requests
import json

def search_tool(query: str) -> str:
    # タイムアウトもなし、エラーハンドリングもなし
    result = requests.get(f"https://api.example.com/search?q={query}")
    return result.json()["items"][0]["content"]  # KeyError / IndexError が起きたら即死

def execute_tool(tool_call) -> dict:
    tool_name = tool_call.function.name
    args = json.loads(tool_call.function.arguments)

    # エラーをそのまま伝播させる
    result = search_tool(**args)

    return {
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": result,
    }
```

この実装の問題点は3つです。

1. `requests.get` にタイムアウトがない。外部APIが応答しなければ永遠に待ち続ける
2. `result.json()["items"][0]["content"]` は検索結果が0件だったとき `IndexError` を投げる
3. `execute_tool` が例外を受け取ると、呼び出し元のエージェントループに例外が伝播してそこで止まる

### 良い例：エラーを文字列として返し、LLMに判断させる

```
import requests
import json
from typing import Optional

def search_tool(query: str) -> str:
    """
    エラーが起きても例外を外に出さない。
    エラー内容は "TOOL_ERROR:" プレフィックス付きの文字列として返す。
    """
    try:
        response = requests.get(
            f"https://api.example.com/search?q={query}",
            timeout=10,  # タイムアウトを明示的に設定する
        )
        response.raise_for_status()  # 4xx / 5xx は HTTPError にする

        items = response.json().get("items", [])
        if not items:
            return "TOOL_ERROR: 検索結果が0件でした。別のキーワードで再試行してください。"

        return items[0]["content"]

    except requests.Timeout:
        return "TOOL_ERROR: APIがタイムアウトしました（10秒）。リトライを検討してください。"
    except requests.HTTPError as e:
        return f"TOOL_ERROR: APIがHTTPエラーを返しました（ステータス {e.response.status_code}）。"
    except requests.ConnectionError:
        return "TOOL_ERROR: ネットワーク接続エラーが発生しました。"
    except (KeyError, IndexError, ValueError) as e:
        return f"TOOL_ERROR: レスポンスの解析に失敗しました（{type(e).__name__}: {e}）。"
    except Exception as e:
        return f"TOOL_ERROR: 予期しないエラーが発生しました（{type(e).__name__}: {e}）。"

def execute_tool(tool_call) -> dict:
    tool_name = tool_call.function.name
    args = json.loads(tool_call.function.arguments)

    result = search_tool(**args)  # 例外は出てこない

    return {
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": result,  # TOOL_ERROR プレフィックスがあれば LLM が判断できる
    }
```

### リトライとフォールバックを追加する

エラーが一時的な場合（タイムアウト・レートリミット）は、自動リトライを組み込むと回復率が上がります。

```
import time
from functools import wraps

def with_retry(max_attempts: int = 3, delay_seconds: float = 1.0):
    """シンプルなリトライデコレータ。TOOL_ERROR が返ったらリトライする。"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_attempts):
                result = func(*args, **kwargs)
                if not result.startswith("TOOL_ERROR"):
                    return result
                if attempt < max_attempts - 1:
                    time.sleep(delay_seconds * (attempt + 1))  # 指数バックオフ
            return result  # 最終試行の結果（エラーのまま）をそのまま返す
        return wrapper
    return decorator

@with_retry(max_attempts=3, delay_seconds=1.0)
def search_tool(query: str) -> str:
    try:
        response = requests.get(
            f"https://api.example.com/search?q={query}",
            timeout=10,
        )
        response.raise_for_status()
        items = response.json().get("items", [])
        if not items:
            return "TOOL_ERROR: 検索結果が0件でした。"
        return items[0]["content"]
    except requests.Timeout:
        return "TOOL_ERROR: タイムアウトしました。"
    except Exception as e:
        return f"TOOL_ERROR: {type(e).__name__}: {e}"
```

### 設計原則

* **ツール関数は例外を外に出さない**。内部で全ての例外を捕捉し、文字列として返す
* **エラー文字列には "TOOL\_ERROR:" のような識別プレフィックスを付ける**。LLMが「エラーが来た」と判断できるようにする
* **systemプロンプトにエラー発生時の行動指針を書く**。「TOOL\_ERRORが返ってきた場合は別のキーワードで再試行し、3回失敗したらユーザーに報告してください」のように明示する
* **一時的な失敗にはリトライを使い、恒久的な失敗にはフォールバックを使う**。リトライは指数バックオフで

## パターン2: 状態が肥大化してコンテキスト上限を超える

### なぜ状態の肥大化が起きるのか

LangGraphのような状態管理フレームワークを使うとき、最初はシンプルな `State` 定義から始めます。しかし実装が進むにつれて「会話履歴も持ちたい」「中間出力も残したい」「デバッグ用にログも入れよう」となり、状態が肥大化していきます。

特に問題になるのが `messages` フィールドへの無制限な追記です。Annotated + add\_messages を使うと、ノードが追加するメッセージは自動的に蓄積されます。ステップが10を超えるようなエージェントでは、メッセージ総トークン数が数万を超えることも珍しくありません。

コンテキスト上限に達すると何が起きるか。APIエラーで止まる（まだわかりやすい）か、後半のメッセージが無視されて文脈が欠落した状態で処理が続く（見つけにくい）かです。

### 悪い例：状態の設計ミスと無制限な追記

```
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

# 問題1：何でも詰め込んだ状態定義
class OverloadedState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # 無制限に蓄積される
    raw_search_results: list[str]       # 毎ステップ全件保持
    all_intermediate_steps: list[dict]  # 中間ステップを全部残す
    full_document_contents: list[str]   # 取得した文書の全文を保持
    debug_logs: Annotated[list[str], lambda a, b: a + b]  # デバッグログも蓄積

def search_node(state: OverloadedState) -> dict:
    docs = vector_store.similarity_search(state["messages"][-1].content, k=10)
    return {
        # 問題2：10件の文書全文を毎ステップ状態に積み増す
        "raw_search_results": [doc.page_content for doc in docs],
        "full_document_contents": state.get("full_document_contents", []) + [d.page_content for d in docs],
    }
```

上記のコードは3つの問題を抱えています。

1. `full_document_contents` が毎ステップ増加し、ループが5回回れば文書が50件蓄積する
2. `debug_logs` も無制限に追記されるため、長い実行ではログだけで数千トークンを消費する
3. `messages` に制限がなく、エージェントが長いフローを実行するほどトークン消費が増大する

### 良い例：最小限の状態設計とトリミング実装

```
from typing import TypedDict, Annotated, Optional
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage, SystemMessage, HumanMessage, AIMessage

# 状態は「このノードに必要なもの」だけに絞る
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # 会話履歴（トリミング対象）
    current_search_results: list[str]   # 「最新の」検索結果だけ保持（上書き）
    task: str                           # 最初のタスク（変化しない）
    step_count: int                     # ステップカウンター
    is_complete: bool                   # 完了フラグ

def trim_messages(
    messages: list[BaseMessage],
    max_tokens: int = 8000,
    keep_system: bool = True,
) -> list[BaseMessage]:
    """
    メッセージリストをトークン上限に収まるようトリミングする。
    SystemMessage は常に保持し、古いメッセージから削除する。
    """
    system_messages = [m for m in messages if isinstance(m, SystemMessage)]
    other_messages = [m for m in messages if not isinstance(m, SystemMessage)]

    # 簡易トークン推定（実際は tiktoken を使うことを推奨）
    def estimate_tokens(msgs: list[BaseMessage]) -> int:
        return sum(len(m.content) // 4 for m in msgs)

    system_tokens = estimate_tokens(system_messages)
    remaining_budget = max_tokens - system_tokens

    # 古いメッセージから削除して予算に収める
    trimmed = list(other_messages)
    while trimmed and estimate_tokens(trimmed) > remaining_budget:
        trimmed.pop(0)  # 最古のメッセージを削除

    return (system_messages + trimmed) if keep_system else trimmed

def agent_node(state: AgentState) -> dict:
    # メッセージをトリミングしてからLLMに渡す
    trimmed = trim_messages(state["messages"], max_tokens=8000)

    system_prompt = f"""あなたはタスクを達成するエージェントです。
現在のタスク: {state['task']}
現在のステップ: {state['step_count']}

タスクが完了したら、必ずその旨を明示してください。"""

    messages_for_llm = [SystemMessage(content=system_prompt)] + trimmed
    response = llm.invoke(messages_for_llm)

    return {
        "messages": [response],
        "step_count": state["step_count"] + 1,
    }

def search_node(state: AgentState) -> dict:
    query = state["messages"][-1].content
    docs = vector_store.similarity_search(query, k=5)

    # 検索結果は「最新のもの」だけ保持（上書きフィールドを使う）
    return {
        "current_search_results": [doc.page_content for doc in docs],
    }
```

### 中間サマリーノードを挿入する

ステップ数が多い長いフローでは、途中で状態を圧縮するサマリーノードを挟むと効果的です。

```
def summarize_node(state: AgentState) -> dict:
    """
    直近のメッセージ履歴を要約して圧縮する。
    10ステップごとに呼び出すことでトークン消費を抑える。
    """
    messages = state["messages"]
    if len(messages) < 6:
        return {}  # 少なければ圧縮不要

    # 古いメッセージを要約して1メッセージに置き換える
    old_messages = messages[:-4]  # 直近4件は残す
    recent_messages = messages[-4:]

    summary_prompt = f"""以下の会話履歴を3〜5文で要約してください。
重要な決定・検索結果・現在の進捗を含めてください。

{chr(10).join(f'{m.type}: {m.content[:200]}' for m in old_messages)}"""

    summary = llm.invoke([HumanMessage(content=summary_prompt)])
    summary_message = SystemMessage(content=f"[これまでの経緯の要約]\n{summary.content}")

    return {"messages": [summary_message] + recent_messages}

def should_summarize(state: AgentState) -> str:
    """10ステップごとにサマリーを実行する。"""
    if state["step_count"] > 0 and state["step_count"] % 10 == 0:
        return "summarize"
    return "agent"
```

グラフの構造は次のようになります。

### 設計原則

* **状態のフィールドは「本当にノード間で共有が必要なもの」だけに絞る**。デバッグ用のログは状態に入れず、別途ロギングシステムで取る
* **更新パターンを意識する**。常に最新値だけが必要なフィールドは上書き（デフォルト）、蓄積が必要なフィールドは追記（Annotated + add\_messages）。追記フィールドはトークン爆発の起点になるため、追記を選ぶ理由を明確にする
* **messages フィールドにはトリミングを必ず実装する**。コンテキスト上限の7〜8割を目安に設定する
* **ステップ数が多くなる設計ではサマリーノードを組み込む**。10ステップを超えるフローは状態肥大化のリスクがある

## パターン3: 非決定的な出力に依存しすぎる

### LLMの出力がブレると何が起きるのか

LLMの出力は確率的です。同じプロンプトを渡しても、実行するたびに異なる出力が返ります。これ自体は問題ではありません。問題になるのは、**エージェントの制御フローをLLMの自由形式出力に依存させたとき**です。

たとえば、「次に何をすべきかを決める」分岐ロジックをLLMに任せるとします。LLMが「search」と返すか「generate」と返すかで次のノードが変わる設計です。テスト時には安定して動いていても、本番で少し違う入力が来たとき、LLMが「Search」（大文字）と返したり、「検索する」と返したりします。条件エッジがこれを想定していなければ、予期しない分岐に入ります。

これが「なぜか本番でたまに壊れる」「再現できない」の正体です。

### 悪い例：LLMの自由形式出力を制御フローに直接使う

```
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o")

def routing_node(state: AgentState) -> dict:
    """次に何をすべきかをLLMに判断させる"""
    prompt = f"""現在のタスク: {state['task']}
これまでの検索結果: {state.get('current_search_results', [])}

次のアクションを1単語で答えてください: search / generate / complete"""

    response = llm.invoke([HumanMessage(content=prompt)])
    # LLMの生出力をそのまま使う
    return {"next_action": response.content}

def route_by_action(state: AgentState) -> str:
    action = state.get("next_action", "")
    # 完全一致で分岐しているため、"Search" や "検索" が来ると壊れる
    if action == "search":
        return "search"
    elif action == "generate":
        return "generate"
    elif action == "complete":
        return "complete"
    else:
        return "search"  # デフォルトで search に落とすが、これが無限ループの原因になる
```

### 良い例：Pydanticで出力スキーマを強制する

```
from pydantic import BaseModel, Field
from typing import Literal
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o")

class RoutingDecision(BaseModel):
    """エージェントの次アクションを表す構造化出力"""
    action: Literal["search", "generate", "complete"] = Field(
        description="次に実行するアクション。search: 追加検索が必要、generate: 回答生成に進む、complete: タスク完了"
    )
    reason: str = Field(
        description="このアクションを選んだ理由（1〜2文）"
    )
    confidence: float = Field(
        ge=0.0, le=1.0,
        description="判断の確信度（0.0〜1.0）"
    )

# with_structured_output でスキーマを強制する
structured_llm = llm.with_structured_output(RoutingDecision)

def routing_node(state: AgentState) -> dict:
    prompt = f"""現在のタスク: {state['task']}
これまでの検索結果（件数）: {len(state.get('current_search_results', []))}件
ステップ数: {state['step_count']}

次のアクションを決定してください。"""

    decision: RoutingDecision = structured_llm.invoke([HumanMessage(content=prompt)])

    return {
        "next_action": decision.action,
        "routing_reason": decision.reason,
        "routing_confidence": decision.confidence,
    }

def route_by_action(state: AgentState) -> str:
    action = state.get("next_action", "search")
    # Literal型で強制されているため、search / generate / complete 以外は来ない
    return action
```

`with_structured_output` を使うと、LLMの出力が `RoutingDecision` のインスタンスとして返ります。`action` フィールドは `Literal["search", "generate", "complete"]` で制約されているため、これ以外の値が入ることはありません。バリデーションエラーが起きた場合はPydanticが例外を投げるため、想定外の値がサイレントに伝播するケースを排除できます。

### ツール選択の非決定性も抑制する

ルーティングだけでなく、ツール呼び出しの引数生成も非決定性の温床です。LLMがツールの引数を自由に生成するとき、期待する形式と微妙にズレることがあります。

```
class SearchQuery(BaseModel):
    """検索クエリの構造化定義"""
    query: str = Field(
        min_length=1,
        max_length=200,
        description="検索キーワード。日本語または英語で記述する"
    )
    max_results: int = Field(
        default=5,
        ge=1,
        le=20,
        description="取得する最大件数（1〜20）"
    )
    search_type: Literal["semantic", "keyword", "hybrid"] = Field(
        default="semantic",
        description="検索タイプ"
    )

class AnalysisResult(BaseModel):
    """LLMの分析結果の構造化定義"""
    summary: str = Field(
        description="分析結果のサマリー（200字以内）"
    )
    key_points: list[str] = Field(
        min_length=1,
        max_length=5,
        description="重要ポイントのリスト（1〜5件）"
    )
    is_sufficient: bool = Field(
        description="現在の情報でタスクを完了できるか"
    )
    missing_info: list[str] = Field(
        default_factory=list,
        description="不足している情報のリスト（ある場合）"
    )
```

このようにツールの入出力もスキーマで定義しておくと、LLMが生成した引数がバリデーションで弾かれるケースをノードの入口で検知できます。

### 設計原則

* **エージェントの制御フローに関わる出力は必ず `with_structured_output` でスキーマを強制する**。ルーティング判断・完了判断・ツール選択は特に重要
* **Pydanticの `Literal` 型で取りうる値を列挙する**。エラーメッセージが明確になり、デバッグコストが下がる
* **`reason` や `confidence` のようなフィールドを添えると、なぜその判断をしたかを後から追跡できる**。LangSmithでのトレース可視化との相性もよい
* **temperature=0 の利用を検討する**。ルーティングやツール選択のように確定的な判断が必要な場面では、temperature を下げることで再現性を高められる

## 設計で防ぐ：失敗に強いエージェントの構造

3つのパターンを踏まえると、本番で安定するエージェントには以下の構造が必要です。

各要素の役割をまとめます。

| 設計要素 | 目的 | パターンとの対応 |
| --- | --- | --- |
| ツール関数内のtry/except | エラーを文字列化してLLMに渡す | パターン1 |
| リトライとフォールバック | 一時的な失敗を自動回復する | パターン1 |
| 状態フィールドの最小化 | 不要な肥大化を防ぐ | パターン2 |
| メッセージトリミング | コンテキスト上限を超えさせない | パターン2 |
| サマリーノード | 長いフローの文脈を圧縮する | パターン2 |
| `with_structured_output` | 非決定的な出力を型で制約する | パターン3 |
| ステップ上限（`recursion_limit`） | 無限ループを防ぐ | パターン1・3 |

ステップ上限はLangGraphなら `recursion_limit` で設定します。

```
from langgraph.graph import StateGraph, END

graph = StateGraph(AgentState)
# ... ノード・エッジの設定 ...

app = graph.compile()

# 実行時に上限を指定する
result = app.invoke(
    {"task": "ユーザーのタスク", "messages": [], "step_count": 0, "is_complete": False},
    config={"recursion_limit": 25},
)
```

`compile()` の引数ではなく、`invoke()` / `stream()` の `config` に渡すことに注意してください。

## まとめ

AIエージェントが本番で壊れるパターンは3つに集約されます。

**パターン1: ツール呼び出しの失敗をハンドリングしていない**  
ツール関数内でtry/exceptを使い、エラーを文字列として返す設計にします。LLMはエラー文字列を「情報」として受け取り、代替策を検討できます。例外をそのまま伝播させると、エージェントが停止するかハルシネーションで補完するかの2択になります。

**パターン2: 状態が肥大化してコンテキスト上限を超える**  
状態フィールドは最小限に絞り、メッセージトリミングを必ず実装する。長いフローにはサマリーノードを挟んで文脈を圧縮する。状態の設計はコードを書く前にノードのInput/Outputを一覧化してから始める。

**パターン3: 非決定的な出力に依存しすぎる**  
制御フローに関わる判断は `with_structured_output` でスキーマを強制します。Pydanticの `Literal` 型で取りうる値を列挙しておけば、想定外の分岐がサイレントに発生するケースを根本から排除できます。

「開発環境では動いていたのに」という状況は、ほぼこの3つのどれかが抜けているケースです。本番投入前のチェックリストを使って、一つずつ潰してください。

## よくある質問（FAQ）

### Q. AIエージェントが本番で壊れるパターンとは何ですか？

**AIエージェントが本番で壊れる主なパターンは、ツール呼び出し失敗のハンドリング欠如・状態の肥大化によるコンテキスト超過・LLMの非決定的な出力への過依存の3つです。** 開発環境では安定していても、本番では同時リクエスト増・外部APIの一時停止・入力パターンの多様化によってこれらが顕在化します。設計段階でこの3点を意識することが、本番障害を根本から防ぐ方法です。

### Q. AIエージェントの本番障害を防ぐには何から始めればよいですか？

**まずすべてのツール関数に `try/except` を追加し、エラーを例外ではなく文字列として返す設計に変えることから始めることをお勧めします。** これだけで「ツール呼び出しが失敗してエージェント全体が止まる」という最も頻度の高い障害が防げます。次にメッセージトリミング、最後に `with_structured_output` による出力スキーマ強制という順番で、3つのパターンを一つずつ対応してください。

### Q. LLMの非決定的な出力をどうやって制御すればよいですか？

**制御フローに関わる判断（ルーティング・完了判断・ツール選択）は `with_structured_output` とPydanticの `Literal` 型で出力スキーマを強制することが最も効果的です。** 「search / generate / complete」のような選択肢を `Literal` 型で列挙しておくと、LLMがどう出力しても自動でバリデーションされ、想定外の文字列がサイレントに伝播するケースを排除できます。確定的な判断が必要な場面では `temperature=0` の利用も検討してください。

### Q. LangGraphで無限ループを防ぐにはどうすればよいですか？

**`recursion_limit` を必ず設定し、ループの終了条件を状態の `is_complete` フラグや `step_count` に基づいて決めることが基本です。** `app.invoke()` や `app.stream()` の `config` 引数に `{"recursion_limit": 25}` のように渡します。上限に達したときに「エラーを返すか」「途中結果を返すか」「Human-in-the-Loopに移行するか」を事前に決めておくことも重要です。これが決まっていないと、上限到達時の振る舞いが未定義になります。

### Q. コンテキスト上限超過を防ぐためのメッセージトリミングはどう実装しますか？

**モデルのコンテキスト上限の70〜80%を目安にトークン数を計算し、古いメッセージから削除するトリミング関数を実装してLLMに渡す前に適用します。** SystemMessageは常に保持し、他のメッセージは古い順に削除します。簡易実装では文字数を4で割った値をトークン数の近似値として使えますが、本番では `tiktoken` を使った正確な計算を推奨します。10ステップを超えるフローでは中間サマリーノードを挟むことも合わせて検討してください。
