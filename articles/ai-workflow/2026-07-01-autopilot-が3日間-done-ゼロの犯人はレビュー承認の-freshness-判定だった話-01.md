AI エージェントに開発を回させている Codens では、PRD からタスクを起こし、エージェントが実装して PR を作り、AI レビュアーが承認したら自動でマージする、というループが 24 時間回っています。人間はレビュー済みの成果物と通知を見るだけ。……のはずが、ある朝ダッシュボードを見ると **3 日間 done がゼロ** でした。

「Autopilot が動いてない」

と言いたくなるのですが、調べてみると個々のコンポーネントは全部動いていました。plan は生成されている。タスクは submit されている。エージェントは PR を作っている。AI レビュアー（Orange Codens）は PR を **APPROVE 済み**。CI も緑。なのに、マージだけが永遠に起きない。

犯人は、レビューを取り込む poll 処理に入れていた **freshness（新しさ）チェック**でした。しかもそれは、過去の別の事故を防ぐために自分たちで入れた防御です。この記事はその顛末と、「外部状態をイベントとして扱うか、スナップショットとして扱うか」という一般化できる教訓の話です。

## 前提: wait\_review という step

Codens のワークフローエンジン（Purple Codens）は、タスクごとに `implement → create_pr → wait_ci → wait_review → merge_pr` のような固定 step を進めます。AI に「次に何をするか」を決めさせるとオペレーションが詰むので、遷移はエンジン側で決め打ちです（この設計自体は以前の記事に書きました）。

`wait_review` は GitHub の PR レビューを poll して、承認が揃ったら次の step へ進める待ち合わせ地点です。ここに、こういう判定が入っていました。

```
# 簡略化しています
fresh_reviews = [
    r for r in pr_reviews
    if r.submitted_at > workflow.review_initiated_at
]
if any(r.state == "APPROVED" for r in fresh_reviews):
    advance_to_merge()
```

`review_initiated_at` より**新しい**レビューだけを見る。一見まともです。

## なぜ freshness チェックを入れたのか

もともとは、こういう事故を防ぐためでした。

1. レビュアーが APPROVE する
2. その後エージェントが追いコミットする
3. レビュアーが CHANGES\_REQUESTED を出す
4. しかし poll が**古い APPROVE を拾って**マージしてしまう

「変更要求が出たのに、過去の承認でマージが走る」のは明確にまずいので、「レビュー待ちを開始した時刻より新しいレビューだけを信じる」という防御を入れたわけです。この時点では正しい修正でした。

## 再アームが standing approval を殺す

問題は `review_initiated_at` が**いつ更新されるか**でした。

ワークフローエンジンには、Spot インスタンスの回収やプロセス再起動から復帰するためのリカバリ経路があります。中断されたワークフローを拾い直して、いまの step を**再入**する。このとき `wait_review` を再入すると、`review_initiated_at` が**現在時刻で再アーム**されていました。新規コミットの有無とは無関係に、です。

すると、こういうタイムラインが成立します。

```
10:00  レビュアーが APPROVE（submitted_at = 10:00）
10:30  Spot 回収などでワークフローがリカバリ再入
       → review_initiated_at = 10:30 に再アーム
10:31  poll: submitted_at(10:00) > review_initiated_at(10:30) は偽
       → 「新しいレビューなし」
以後    レビュアーはもう APPROVE 済みなので、二度と新しいレビューは来ない
       → 永久に待ち続ける
```

承認は GitHub 上に**立派に存在し続けている**（standing approval）のに、フィルタの基準時刻がその後ろに動いたせいで、永遠に「まだレビューが来ていない」扱いになる。デッドロックの完成です。

たちが悪いのは、これが**確率的にしか起きない**ことです。リカバリ再入が承認より前に起きれば何も問題ない。承認の後に起きたワークフローだけが静かに固まる。Spot で回している環境では再入はそこそこの頻度で起きるので、数日でじわじわ溜まり、「done がゼロ」という形でようやく表面化しました。

## 診断の道筋

「Autopilot が動いてない」という報告から真因に辿り着くまでにやったことは、要するに**端で見ずに分布で見る**ことでした。

* done 数（端の観測）だけ見ると「全部止まってる」ようにしか見えない
* step ごとの滞留数を集計すると、`wait_review` に 28 件が異常滞留していた
* 滞留中のワークフローの PR を突き合わせると、**全部承認済み・CI 緑・未マージ**

ここまで来れば「承認を認識できていない」ことは確定で、あとは poll のコードを読むだけです。「どのコンポーネントが死んでいるか」を探すのではなく、「どの step に死体が積み上がっているか」を数える方が早い。生きたプロセスが仕様どおりに間違い続けるバグは、ヘルスチェックには絶対に映りません。

## 修正: イベントではなくスナップショットとして評価する

修正の本質は 1 行で言えます。**「新着レビューがあったか」ではなく「いまの実効レビュー状態は何か」を毎回計算する**。

```
# reviewer ごとに最新のレビューだけを残す
latest_by_reviewer = {}
for r in sorted(pr_reviews, key=lambda r: r.submitted_at):
    latest_by_reviewer[r.user.login] = r

effective_states = [r.state for r in latest_by_reviewer.values()]

if "CHANGES_REQUESTED" in effective_states:
    hold()
elif "APPROVED" in effective_states:
    advance_to_merge()
```

これは実は GitHub 本家の UI がやっている意味論そのものです。PR のレビュー欄は「reviewer ごとの最新状態」を表示していて、「いつ承認されたか」は実効判定に使っていない。**外部システムの状態を判定に使うなら、その外部システム自身の意味論に合わせる**のが一番事故りにくい、という当たり前の結論に戻ってきました。

では元の事故（変更要求後に古い承認でマージ）はどう防ぐのか。reviewer ごとに最新を取れば、CHANGES\_REQUESTED を出した reviewer の状態は CHANGES\_REQUESTED のままなので、そもそも古い APPROVE は勝てません。freshness チェックは「同一 reviewer が CHANGES\_REQUESTED → 追いコミット → 再レビュー待ち」の再発火制御という本来の守備範囲だけに縮めました。

デプロイ直後、`wait_review` の滞留は 28 → 8 に減り、25 分で 15 タスクが done になりました。3 日分の承認済み PR が、判定を直した瞬間に自動で流れ切った形です。

## 第二波: 失敗は伝播させないと「静かな永久 blocked」になる

これで終わりかと思ったら、翌日また同じ症状の報告が来ました。今度の原因は違っていて、**依存チェーン**でした。

freshness バグの期間中に `wait_review` で失敗扱いになったタスクがあり、その**下流タスク**が依存関係で blocked のまま残っていた。ここで自動復旧処理が「blocked を pending に戻す」ことしかしておらず、起動条件は「依存が全部 done」のままなので、failed な依存を持つタスクは **pending に戻っては起動されない**を無限に繰り返していました。実質 no-op のリトライです。

修正は fail-fast にすること。依存が失敗したら、下流を曖昧に blocked のまま置かず**明示的に FAILED に倒して伝播**させる。失敗が見えれば人間は「失敗した祖先を retry する」という 1 アクションで復旧できます。見えない blocked は、誰も retry しようと思わないので永久に残ります。

## 教訓

一般化できる形でまとめます。AI エージェント運用に限らず、外部システムを poll する待ち合わせ全般に効く話だと思います。

1. **外部状態は「イベント」ではなく「スナップショット」として評価する。** 「基準時刻より新しい変化があったか」で判定すると、基準時刻の管理を 1 箇所でも間違えた瞬間、既に成立している事実（standing approval）を永遠に取りこぼす。毎回「いまの実効状態」を計算する方が、冪等でリカバリにも強い。
2. **タイムスタンプの再アームは「それ以前の事実を無かったことにする」操作。** リトライ・リカバリ・再入の経路で無造作に `*_initiated_at` を現在時刻にすると、その瞬間より前に成立した事実がフィルタの向こうに消える。再アームしてよいのは「事実の前提が本当に変わったとき」（新規コミットが積まれたとき等）だけ。
3. **過去の事故対策は、次の事故の第一容疑者。** freshness チェックは実在した事故への正しい対処だった。ただ守備範囲が広すぎた。防御を入れるときは「この防御が誤作動したら何が止まるか」まで書いておくと、次に止まったとき最初に疑える。
4. **「動いてない」は端ではなく step ごとの滞留分布で見る。** 各コンポーネントのヘルスチェックが全部緑でも、仕様どおりに間違い続けるプロセスは検出できない。どの待ち合わせ地点に何件積まれているかを数えるのが一番早い。
5. **失敗は伝播させる。** 依存先の失敗で進めないタスクを blocked のまま放置すると、「誰も気づかない永久待ち」になる。fail-fast で明示的に FAILED を伝播させれば、復旧は「祖先を 1 個 retry」で済む。

自動マージまで含めた Autopilot は、止まっても誰も困らない開発フローなら気づかれるのが遅れるだけで済みますが、これを本番の開発ラインにしていると「3 日間 done ゼロ」は普通に事業インパクトです。ヘルスチェックの緑を信じず、待ち合わせ地点の滞留を監視項目に入れる。これがこの事件でうちの運用に増えた監視です。
