## はじめに

ChatGPT、Claude、Gemini などの LLM との壁打ちは楽しいし生産的です。設計の方針を相談すれば整理してくれるし、コードレビューも頼めばやってくれる。ただ、ここ最近ずっと引っかかっていることがあります。

LLM、同意しすぎじゃないかな？と

こちらが「このプランどう思う？」と聞けば基本的には「いいですね」と返ってくる。「この設計で問題ない？」と聞けば「よく考えられています」と褒めてくれる。実際、あるOSSの普及戦略を考えていたとき、LLMと何度も壁打ちして「よさそうだ」と確信を深めていたのに、後からアカウント設計という根本的な観点を見落としていたことに気づきました。

「じゃあ批判的にレビューしてよ」と頼めばいいんじゃないか。そう思って試しました。でも返ってきたのは「概ね問題ないですが、強いて言えば……」みたいなやんわりした指摘。0ではないですが、やっぱり肯定寄りになる傾向が強いです。

この記事は、この問題を構造的に解決するための実験ノートです。正解を提示するものではなく、「こうやったらこうなった」という記録を共有します。

## 対象読者

* LLM を日常的に使っていて、壁打ち相手として頼りにしている方
* 設計判断や技術選定で LLM に相談した結果、判断を誤った経験がある方
* 「もっとうまく LLM を使いたい」と思っている方

## なぜ「批判して」では不十分か

よく知られている手法として「Devil's Advocate（悪魔の代弁者）になって批判して」とロールを指定する方法はよく知られています。ただ、同じモデルが同じ学習データから出力している以上、「見せかけの多様性」になりがちです。

Anthropic のエンジニアリングブログでも、LLM は自分の出力を自分で評価すると一貫して甘くなるため、生成と評価を分離するアプローチが有効だと報告されています。

ここで、LLM にレビューを頼むときの典型的なパターンを整理してみます。

| パターン | 指示例 | 評価軸の決まり方 | 問題点 |
| --- | --- | --- | --- |
| ①丸投げ | 「レビューして」 | LLM が暗黙的に選ぶ | 見落としやすい軸が入らない |
| ②観点指定 | 「セキュリティ観点でレビューして」 | 人間が指定する | 自分の頭にある観点しか出ない |
| ③本記事の手法 | Stage 1 → Stage 2 | LLM に軸の生成自体を委託 | 自分が思いつかなかった軸が出る |

おそらく多くの方は①か②でレビューを頼んでいると思います。②は一見ちゃんとしていますが、自分が指定した観点の中でしか評価されないので、自分のバイアスがそのまま残ります。

③の「評価軸の生成を分離する」のが、この記事で紹介する手法の核です。

## 提案手法：2段階レビュー

やり方はシンプルで、2つのステップに分けてプロンプトを投げるだけです。

Gary Klein の Pre-mortem という手法に着想を得ています。これは「このプランが失敗したと仮定して、原因は何か？」と未来から逆算する思考法です。先に「何がまずいか」を洗い出してから評価に入ることで、肯定バイアスを構造的に断ちます。

### Stage 1：評価軸の生成

まず、「何の軸で評価するか」を LLM に考えさせます。

```
この計画の見落としやすいリスク軸を5つ挙げてください。
各軸について、なぜこの判断で重要なのかも説明してください。
```

ポイントは「レビューして」ではなく「リスク軸を挙げて」と聞くことです。こうすることで、LLM は評価軸にコミットしてから次のステップに進むので、肯定的に流しにくくなります。

### Stage 2：軸に基づく評価

Stage 1 で生成された軸を使って、本題を評価させます。

```
その5つの軸でこの計画を評価してください。
各軸に OK / Warning / Critical の判定と根拠をつけてください。
```

OK / Warning / Critical のような段階的な判定を求めることで、「問題ないけど念のため……」のような曖昧な指摘ではなく、構造化された評価が返ってきます。

この2ステップは ChatGPT でも Gemini でも Claude でも LLM ならなんでも使えます。ツールに依存しないのも利点です。

## 実験1：設計レビューで試す

あるアプリの外部 API 連携層の実装プランに対して、この2段階レビューを実行してみました。外部サービスとの通信を担うサービス層のインターフェース設計、エラーハンドリング、レスポンスの変換処理あたりのプランを作成した段階です。

結果として、8つの評価軸が生成され、そのうち4つで Warning 判定が出ました。

検出された Warning のうち、特に印象的だったものを挙げます。

1つ目は、フォールバック処理の設計問題です。レスポンスのパース失敗時に特定の文字列キーでフォールバックする設計でしたが、キー名の衝突リスクや型安全性の欠如を指摘されました。専用の型を定義して明示的にハンドリングすべきという提案です。

2つ目は、エラーマッピングの対応表が未定義だったことです。外部 API の HTTP ステータスコードをアプリ内部のエラー型にどう変換するか、ルールが明文化されていませんでした。API 連携がある開発なら誰でも経験のある「後で決めるから今は進めよう」の YAGNI 案件です。

3つ目は、レスポンス値の正規化ルールが未定義だったことです。外部 API のレスポンスに含まれる null や型の揺れをどう正規化するかが決まっていませんでした。MVP 段階では後回しにしがちですが、後から決めると既存コードとの整合性で苦しみます。

いずれも「動くけど後で困る」タイプの設計判断で、通常の壁打ちでは「いいですね」で流されていた可能性が高いものです。

面白かったのは、OK 判定にも理由がついていたことです。たとえば「インターフェース定義のみのコミットにはテスト不要」という OK 判定に対して、なぜこの段階ではテストが不要かの理由が述べられていました。問題ないならないで、その理由が言語化されることに価値があります。

## 実験2：技術選定で「丸投げ」と「2段階」を比較する

もう少しわかりやすい実験もしてみました。技術選定の壁打ちで、同じお題に対してプロンプトの投げ方を変えて回答の違いを比較します。LLM は ChatGPT（GPT-5.3）を使用しました。2つのお題で試しています。

### お題A：フレームワーク選定（Next.js vs Hono + Vite）

お題はこうです。

> 個人開発でWebアプリを作ろうとしています。リアルタイムチャット機能付きのSNS的なサービスで、認証・DBあり。デプロイ先はCloudflareかVercelを想定。Next.js と Hono + Vite の構成で迷っています。

### パターン①：丸投げ

「レビューしてもらえますか？」とざっくり聞いたところ、返ってきたのは結論とおすすめでした。「Hono + Vite を推します」で締められ、リスクの指摘はほぼゼロ。そもそもレビューを頼んだのにおすすめが返ってきました。

パターン①の回答概要（クリックで展開）

回答は「結論からいうと、Hono + Vite + Cloudflare を推します」から始まり、おすすめ順位を1位〜3位でランキング形式で提示。

Next.js 案と Hono + Vite 案それぞれの強み・弱みには触れていましたが、どちらも「おすすめかどうか」の文脈で語られており、リスク軸での構造的な評価はありませんでした。

最終的に「迷っているなら Hono + Vite を推します」で締めくくられています。

### パターン②：観点指定

「パフォーマンスとセキュリティの観点でレビューしてもらえますか？」と聞くと、その2つの軸では詳しく答えてくれました。ただし、指定しなかった観点は完全に無視されていて、最終的にはやっぱりおすすめで締めくくられました。

パターン②の回答概要（クリックで展開）

パフォーマンス観点では Cloudflare Durable Objects とリアルタイム処理の相性について、セキュリティ観点では Server Components によるデータアクセスの分離について、それぞれ詳しい分析がありました。

ただし、デプロイコスト、1人で保守し続けられるか、エコシステムの成熟度、途中で技術を変えた場合の移行コストなど、長期的な軸での大事な観点は一切触れられていませんでした。

最終的に「私は第一候補を Next.js + 外部 Realtime にします」と結論づけています。なお、パターン①では Hono を推していたのに、②では Next.js を推しており、観点が変わると結論も変わるという点も興味深いです。

### パターン③：2段階レビュー

まず「この技術選定において見落としやすいリスク軸を5つ挙げてください」と聞きました。すると以下の軸が生成されました。

1. リアルタイム機能とデプロイ先の相性
2. ランタイム互換性とライブラリ衝突
3. 認証・セッション設計と責務分離
4. データ整合性と状態管理
5. 料金・制限・スケール時のボトルネック

次に「その5つの軸で Next.js と Hono + Vite のそれぞれを評価してください」と投げたところ、両方に OK と Warning が混在するフェアな評価が返ってきました。

パターン③の評価結果概要（クリックで展開）

5つの軸それぞれに対して、Next.js と Hono + Vite の両方に OK または Warning の判定が出ました。

| 軸 | Next.js | Hono + Vite |
| --- | --- | --- |
| リアルタイム機能とデプロイ先の相性 | Warning | OK |
| ランタイム互換性とライブラリ衝突 | Warning | Warning |
| 認証・セッション設計と責務分離 | OK | Warning |
| データ整合性と状態管理 | Warning | OK |
| 料金・制限・スケール時のボトルネック | Warning | Warning |

パターン①②と異なり、どちらか一方を「推す」のではなく、軸ごとに得意・不得意が可視化されています。たとえば認証・責務分離の軸では Hono 側に Warning がつき、「自由度が高い分、Cookie や CSRF、セッション更新の責務が散りやすい」という具体的な指摘がありました。

### 3パターンの比較

| パターン | 返答の特徴 | リスク指摘 |
| --- | --- | --- |
| ①丸投げ | 結論とおすすめを即座に返答 | ほぼゼロ |
| ②観点指定 | 指定2軸では詳しいが他は無視 | 指定軸内のみ |
| ③2段階 | 5軸生成、両方に OK/Warning 混在 | 5軸すべてで具体的 |

最も顕著な差は、①②では一貫して「Hono + Cloudflare を推します」と結論づけていたのに対し、③では認証・責務分離の軸で Hono 側にも Warning がつき、「どっちにもトレードオフがある」という構造が可視化されたことです。

### お題B：DB設計（auto increment vs UUID）

別のお題でも同じパターンが再現されるか確認してみました。お題はちょっと前に X で話題になったDBの主キー設計の話です。

> 新規Webアプリのデータベース設計で、主キーを auto increment の連番IDにするか UUID にするかで迷っています。

お題Aでは3パターン試しましたが、②（観点指定）はお題Aで十分示せたので、ここでは①と③の比較に絞ります。

#### パターン①：丸投げ

「レビューしてもらえますか？」と聞いたところ、お題Aとまったく同じ構造の回答が返ってきました。「UUID を推します」と結論から入り、比較表は「おすすめかどうか」の文脈で語られていました。

パターン①の回答概要（クリックで展開）

回答は「結論からいうと、個人開発〜中規模の新規Webアプリなら『内部主キーは UUID、必要なら別に表示用の短いIDやslugを持つ』のがいちばん事故が少ないです」から始まりました。

セキュリティ、DB性能、開発しやすさ、拡張性、URL設計の5つの観点で比較していますが、いずれも「どっちがおすすめか」の文脈で語られています。

最終的に「推奨構成」として具体的なカラム設計まで提案され、「アプリの想定がわかれば具体的なテーブル設計方針まで落として提案できます」で締めくくられています。レビューを頼んだのに、余計なおせっかいで設計提案まで返ってきました。

#### パターン③：2段階レビュー

「見落としやすいリスク軸を5つ挙げてください」と聞いたところ、以下の軸が生成されました。

1. 外部公開時の情報漏えい・列挙耐性
2. インデックス効率と書き込み性能
3. 分散構成・オフライン生成・将来の統合耐性
4. 運用・デバッグ・人間可読性
5. 「主キー」と「公開識別子」を同一にしてよいか

特に5番目の軸が興味深いです。①では折衷案として最後に触れられていただけのものが、③では独立した評価軸として扱われています。「そもそも二択じゃなくて分離するべきでは？」という問いかけ自体が、①では出てこなかった視点です。

パターン③の評価結果概要（クリックで展開）

5つの軸それぞれに対して、auto increment と UUID の両方に判定が出ました。

| 軸 | auto increment | UUID |
| --- | --- | --- |
| 外部公開時の情報漏えい・列挙耐性 | Critical | OK |
| インデックス効率と書き込み性能 | OK | Warning |
| 分散構成・オフライン生成・将来の統合耐性 | Warning | OK |
| 運用・デバッグ・人間可読性 | OK | Warning |
| 主キーと公開識別子を同一にしてよいか | Warning | Warning |

①では「auto increment が悪いわけではありません」とマイルドに扱われていた外部公開の列挙耐性が、③では明確に Critical 判定。また、5番目の軸で両方に Warning がつくことで、「どちらを選んでもこの問題は残る」という構造が見えるようになりました。

### 2つのお題に共通するパターン

お題Aとお題Bで、同じ傾向が確認できました。

| 観点 | パターン① | パターン③ |
| --- | --- | --- |
| 回答の構造 | 結論とおすすめ | 軸ごとの構造化された評価 |
| リスク指摘 | ほぼゼロ、または「おすすめ」の文脈内 | 各軸で具体的に |
| 両選択肢のトレードオフ | 一方を推す | 両方に OK/Warning が混在 |
| 見落としやすい観点 | 出てこない | 独立した軸として出てくる |

聞き方を変えるだけで、おすすめが出なくなったり、より構造された評価になりました。また「ただ動くものを作る」だけではなく、運用後に気づく開発中に見落としやすい観点も出てきたのがポイントです。

## 考察と限界

この手法にはいくつかの限界があります。正直に書いておきます。

まず、この記事自体が LLM との対話から生まれています。LLM の肯定バイアスを問題視する手法を LLM と一緒に考えた、という自己言及的な構造です。メタ的ですが、この手法自体にも同じバイアスがかかっている可能性は否定できません。

またサンプル数もまだ少ないです。実験2のデモは ChatGPT の1モデルで実施しており、お題は2つ試しましたが、それでもまだ実験と呼べる母数ではないです。LLM の特性上、同じプロンプトを再度投げたら違う結果になる可能性があります。また、こちらから選択肢を提示しているため、LLM がその選択肢を肯定する方向に動きやすいバイアスも存在します。

評価軸の生成自体も LLM が行うため、LLM の知識や傾向に依存します。「自分の頭にない観点が出る」のは利点ですが、LLM 自体が知らない観点や、こちらから聞かない引き出せないようなニッチな観点は出てきません。Stage 1 で出てきた軸をそのまま鵜呑みにするのではなく、人間が「この軸は自分のケースでは関係ない」「この軸が足りない」と介入することで、手法の精度はさらに上がるはずです。あくまで出発点を LLM に広げてもらい、最終的な判断は人間がする、という使い方が現実的です。

それでも、「構造的に批判を組み込む」ことには価値があると感じています。完璧ではないけれど、①②のパターンより明らかに見落としが減りました。

## まとめ

LLM は1つの手段ですが、入力次第で得られる効果が異なります。肯定してくれるだけの壁打ち相手にも、批判的な視点を提供してくれるパートナーにもなります。

この記事で紹介した手法はシンプルです。

1. まず「この判断で見落としやすいリスク軸を挙げて」と聞く
2. その軸で評価してもらう

たった2ステップですが、「レビューして」とだけ頼んだときとは返答の構造が根本的に変わります。ChatGPT でも Claude でも Gemini でも今日から試せるので、次に何かの判断で迷ったときにぜひ試してみてください。

とはいえLLMも銀の弾丸ではありません。見落としやミスもするので任せっきりに過ぎず、日頃から幅広い視野や知見をためておき、自分の評価軸を養うことも重要です。

もし同じような試みをされた方がいたら、ぜひ知見を共有してもらえると嬉しいです！
