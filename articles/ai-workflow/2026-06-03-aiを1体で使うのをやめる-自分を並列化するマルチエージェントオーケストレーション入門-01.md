## なぜ今「AIを1体で使う」のが頭打ちになるのか

正直に言いましょう。AIに仕事を頼むの、だいぶ慣れてきましたよね。調べもの、文章のたたき台、コードの下書き。ひとつのチャット欄に話しかければ、だいたい返ってくる。

でもね、最近ちょっと変な感覚ないですか。AIはめちゃくちゃ速いのに、自分のさばける仕事量は前ほど増えてない、っていう。

理由はシンプルで、AIを「1体」で使ってるからな気がするんです。

イメージしてほしいんですけど、これって個人商店の店主が一人で全部やってる状態なんですよね。接客して、調理して、会計して、仕込みもして。腕がいくら良くても、体はひとつ。どこかで詰まる。一方で、ちゃんと回ってる厨房は役割が分かれてます。前菜の担当、メインの担当、デザートの担当がいて、料理長（指揮者）がまとめる。同じ時間で何倍も出せる。

AIの世界でこの「役割分担した厨房」を作るのが、マルチエージェントという考え方です。先に言葉だけ揃えておきますね。前提知識ゼロでも大丈夫なように、ひとつずつ。

* エージェント: 目的を渡すと、自分で考えて手を動かすAI。チャットより一歩踏み込んで、調べたり書いたりを自走する係。
* オーケストレーター: 仕事を分解して各担当に振り、最後に結果をまとめる「料理長」役のAI（または、あなた自身）。
* ワーカー（サブエージェント）: 振られた一部分だけを担当する係。前菜担当みたいなもの。
* コンテキストウィンドウ: AIが一度に読める「作業机の広さ」。広いほど一度に見られるけど、無限ではない。
* 並列と逐次: 並列は同時に走らせること、逐次はひとつ終わってから次。厨房を分けるのは、並列で速くするためなんです。

で、ここからが本題。ただエージェントを増やせばいい、って話じゃないんですよね。むしろ増やして失敗する人がめちゃくちゃ多い。この記事では「型」と「どこで使うか」と「事故りどころ」をセットで見ていきます。

## オーケストレーター・ワーカー型という「型」

まず一番ベーシックで、一番効く型を紹介します。オーケストレーター・ワーカー型、って呼ばれてるやつです。

流れはこうです。料理長（オーケストレーター）がゴールを受け取って、仕事を3〜5個の独立した小タスクに分ける。その小タスクを、別々のワーカーに同時に渡して走らせる。最後に、返ってきた結果を料理長がひとつにまとめる。これだけ。

実はこれ、Anthropicが自社の調査システムで使っている構成とほぼ同じなんです。彼らの公開記事によると、リーダー役のエージェントが計画を立て、3〜5体のサブエージェントを並列で起動して、最後に統合する。引用（出典の確認）はさらに別のパスでやる、という作りでした。そして内部評価では、この構成が単体のAIを大きく上回ったと報告されています。

なんでそんなに効くと思います？ ポイントは「机が分かれる」ことなんです。ワーカーごとに専用のコンテキストウィンドウ（作業机）を持てるから、一体で全部抱え込むより、一人ひとりが深く調べられる。情報が単一の机に乗り切らないような幅広い調査で、特に強い。

最小の実装イメージを置いておきますね。ここでの`callLLM`は、お使いのAIクライアント（OpenAI でも Claude でも Gemini でも）に置き換えてください。難しいSDKの話は抜きで、考え方だけ掴めれば十分です。

```
// 汎用のAIクライアント。各社のSDK呼び出しに置き換えてください
declare function callLLM(system: string, user: string): Promise<string>;

type SubTask = { id: string; objective: string; scope: string };
type WorkerResult = { id: string; findings: string; sources: string[] };

// ワーカー1体。自分の担当範囲しか知らない＝コンテキストが分離されている
async function runWorker(task: SubTask): Promise<WorkerResult> {
  const system = [
    "あなたは限定スコープの調査ワーカーです。",
    `担当範囲: ${task.scope}`,
    "範囲外には手を出さない。結論と根拠URLだけをJSONで返す。",
  ].join("\n");
  const raw = await callLLM(system, task.objective);
  const { findings, sources } = JSON.parse(raw);
  return { id: task.id, findings, sources };
}

// ここが並列化の本体。逐次のfor-awaitではなく Promise.all で同時に走らせる
async function runInParallel(tasks: SubTask[]): Promise<WorkerResult[]> {
  return Promise.all(tasks.map(runWorker));
}
```

`Promise.all`の一行が、厨房を「同時に動かす」スイッチです。逐次に一個ずつ待ってたら、人を増やした意味がないですからね。

## 「並列化していい仕事／ダメな仕事」を見分ける判断軸

ここが、この記事で一番伝えたいところなんです。たぶんこれが、AI時代を生き抜く上での本当の気づきかなと。

レバレッジ（てこ）って、手の数を増やすことだと思われがちですよね。でも違うんです。本当のレバレッジは「どの仕事が並列化できるかを見分ける目」のほうにある。やみくもにエージェントを5体並べても、向いてない仕事だと、速くならないどころか壊れます。

正直に、デメリットも置きます。心配性な自分が黙ってないので。

ひとつめ、コスト。マルチエージェントは、ふつうのチャットのざっくり15倍くらいトークンを食う、という報告があります（Anthropicの公開値）。机を5つ用意して同時に考えさせるんだから、当然といえば当然。だから「高い価値のある仕事」じゃないと割に合わない。

ふたつめ、壊れやすさ。これは見落とされがちです。CognitionというチームはむしろDon't Build Multi-Agents（マルチエージェントを作るな）という主旨の記事を出していて、文脈を共有しないエージェントどうしは、それぞれ勝手に違う前提を置いてしまい、つなぎ合わせると矛盾だらけになる、と指摘しています。前菜担当とメイン担当が、別々の「今日のテーマ」を勝手に決めてたら、コースとして崩壊しますよね。それと同じ。

つまり結論はこうです。並列に分けても各自が独立して完結できる仕事は強い。途中でずっと相談し合わないと進まない仕事は、単体のほうが安定する。

この見分けを、毎回カンでやると疲れるので、関数にしてしまいましょう。点数化して「向き／不向き」を返すだけの簡単なものです。

```
type WorkShape = {
  independence: number;     // 0-1: サブ作業どうしが独立しているか
  breadth: number;          // 0-1: 探索の幅（調べる方向が多いか）
  contextOverflow: number;  // 0-1: 情報量が単一の机を超えるか
  valuePerRun: number;      // 0-1: 1回あたりの価値（コストを払う意味）
  coordinationNeed: number; // 0-1: 途中で密に相談する必要（高いほど不利）
};

function scoreParallelization(w: WorkShape): { score: number; verdict: string } {
  // 独立・幅・あふれ・価値はプラス。協調が必要なほどマイナス
  const score =
    w.independence * 0.3 +
    w.breadth * 0.25 +
    w.contextOverflow * 0.2 +
    w.valuePerRun * 0.15 -
    w.coordinationNeed * 0.3;

  if (score >= 0.4) return { score, verdict: "マルチエージェント向き" };
  if (score >= 0.2) return { score, verdict: "条件つき。小さく試す" };
  return { score, verdict: "単体エージェントで十分" };
}
```

数式の細かさはどうでもよくて、大事なのは「独立してる？ 幅がある？ 一回の価値ある？ 途中で相談いる？」を毎回自分に問う、その習慣のほうなんですよね。

## コンテキスト分離と結果の統合 ― ここで事故る

型は分かった、判断軸も持った。じゃあ実際にやると、どこで転ぶか。経験上、転ぶのはだいたい「入口」と「出口」なんです。入口＝各ワーカーへの渡し方、出口＝結果のまとめ方。

入口でやることは、ワーカーに「指示書」を渡すこと。口頭で雑に振ると、範囲がはみ出して、隣の担当と同じことを調べて無駄になったり、勝手に大きな判断をして暴走したりする。だから、触っていい範囲と、触っちゃダメな範囲を、先に書いて渡す。

```
// ワーカーに渡す「指示書」。境界を明文化するほど事故が減る
type WorkerBrief = {
  objective: string;    // 何を達成するか
  scope: string;        // 触っていい範囲
  forbidden: string[];  // 触ってはいけない範囲
  outputSchema: string; // 返してほしい形（JSON等）
};

// 複数ワーカーの結果を統合し、未検証を人間レビューへ回す
function synthesize(results: WorkerResult[]) {
  const seen = new Map<string, WorkerResult>();
  for (const r of results) {
    if (!seen.has(r.findings)) seen.set(r.findings, r); // 同じ主張は畳む
  }
  // 根拠URLが無い結論は「未検証」として人間に回す
  const needsHumanReview = results.filter((r) => r.sources.length === 0);
  return { merged: [...seen.values()], needsHumanReview };
}
```

出口の`synthesize`は、わざと単純にしてあります。重複を畳んで、根拠のない主張を人間に回すだけ。本当の「矛盾の検出」、つまり「ワーカーAとワーカーBが食い違ってる」を見抜くのは、機械的な一致比較だと無理なんですよね。意味で比べないといけない。だからそこは、もう1体のAI（統合レビュアー）に任せるのが現実的です。

このとき、ワーカーへ渡すプロンプトと、まとめるプロンプトを、ちゃんと型にしておくと安定します。3つ置いておきますね。

ワーカーを分ける「分解プロンプト」。

```
あなたは調査チームのリーダーです。次のゴールを、互いに重複しない
独立したサブタスクに3〜5個へ分解してください。

ゴール: {{ここにゴール}}

各サブタスクについて出力:
- id（短い識別子）
- objective（達成すること1行）
- scope（調べてよい範囲）
- 重複しないための境界（他と被らない理由）

制約: 密に相談し合わないと進めない場合は、分割せず
「単体で実行すべき」と返す。出力はJSON配列のみ。
```

各ワーカーに渡す「限定スコープ・プロンプト」。

```
あなたは限定スコープのワーカーです。担当範囲だけを実行します。

担当範囲: {{scope}}
やってはいけないこと: {{forbidden}}

ルール:
- 範囲外の判断・決定はしない。必要なら「要・上位判断」と書いて止める。
- 確信が持てない点は推測で埋めず「不明」と明記する。
- 出力は次の形だけ: {"findings": "...", "sources": ["url", ...]}
```

## 指揮者の仕事 ― ガードレールと撤退基準

最後に、一番大事な話。AIを何体並べても、指揮者（あなた）が手放しちゃいけない仕事がある、っていう話です。

会社で考えると分かりやすいんですよね。社長が全部の作業を自分でやる必要はない。けど、方向性と、最終的なゴーサインは社長が握る。これと同じで、オーケストレーションでも人間が握り続けるべき決議があるんです。一般化するとこんな感じ。

* ゴールと意図そのもの（何のためにやるか）
* どの結果を信じるか（採用・不採用の最終判断）
* 取り返しのつかない操作の承認（削除・公開・送信・課金など）
* どこで止めるか（ストップ条件）

この「人間が握る範囲」を決めておくのが、攻めるためのガードレールなんですよね。柵があるからこそ、安心してAIを速く走らせられる。逆に柵がないと、怖くて結局ぜんぶ自分で抱える。

で、結果をまとめる最後の一手として、統合レビュー用のプロンプトを置いておきます。3つめのプロンプトです。

```
あなたは統合レビュアーです。複数ワーカーの結果を突き合わせます。

入力: {{ワーカー結果の配列}}

出力:
1. 一致している結論（重複は1つにまとめる）
2. 矛盾している点（どのワーカーとどのワーカーが、何で食い違うか）
3. 根拠が弱い／未検証の主張（人間の判断が必要なもの）
4. 最終結論の素案（ただし決定はしない。あくまで素案）

決めるのは人間です。あなたは判断材料を整理するところまで。
```

そしてコスト。さっきの15倍を思い出すと、上限を決めずに走らせるのは結構こわい。だから、ワーカー数とトークンに天井をつけて、超えたら「増やす」んじゃなく「絞る」。1人会社の最強の武器は、絞る覚悟だったりするんですよね。

```
type Budget = { maxWorkers: number; maxTokens: number };

function withCostGuard(tasks: SubTask[], budget: Budget): SubTask[] {
  if (tasks.length <= budget.maxWorkers) return tasks;
  // 上限を超えたら、価値の高い順に絞る（増やすより絞るほうが安全）
  return tasks.slice(0, budget.maxWorkers);
}
```

最後に撤退基準。これがないと無責任なので、はっきり書きます。次のどれかに当てはまったら、迷わず単体エージェントに戻してください。

* ワーカーどうしの結果が毎回矛盾して、まとめるほうが時間かかる
* 前の結果を見てからじゃないと次が決められない（＝逐次でしか進まない）
* レイテンシ（返答の速さ）が大事で、待てない
* 仕事が小さくて、ひとつの机に余裕で乗る

## まとめ：レバレッジは「手の数」じゃなく「設計の質」

長くなったので、ぎゅっとまとめますね。

* AIを1体で使うと、自分のさばける量で頭打ちになる。役割分担した厨房＝マルチエージェントで並列化できる。
* 基本の型はオーケストレーター・ワーカー型。分解して、並列で走らせて、統合する。
* ただし万能じゃない。15倍のコストと、文脈を共有しない壊れやすさがある。独立・幅・価値が高く、協調がいらない仕事だけが向いてる。
* 入口は「指示書で範囲を絞る」、出口は「矛盾と未検証を人間に回す」。情報は少なく鋭く。
* 指揮者（人間）はゴール・採用判断・危険操作の承認・ストップ条件を握り続ける。コストに天井を、撤退に基準を。

結局のところ、AI時代のレバレッジって、手を何本持てるかじゃないんですよね。どの仕事をどう分けて、どう統合して、どこを自分が握るか。その設計の質のほうなんです。手はAIがいくらでも貸してくれる。設計するのは、あなた。

今日の30分でできる最初の一歩を置いておきます。いま抱えてる「幅広く調べる系」の仕事をひとつ選んで、3つの独立した小タスクに紙の上で割ってみる。それだけでいい。割れたなら並列向き、割ろうとして「これは前のを見ないと無理だ」となったら単体向き。その判断ができた時点で、もう指揮者の入口に立ってます。

未来の自分は、たぶんこう言うんですよね。あの日ちゃんと「分け方」を考えといてくれて、あざっす、と。
