# Fableはビジネスでも使えるのか？

Claude Fable 5 のコーディング能力については、すでに山ほどベンチマークと体験談が出ています。SWE-bench のスコアがどうだとか、巨大リポジトリの理解がどうだとか。それはそれで重要なのですが、事業会社のAI活用においては、**生成AIの利用時間の多くを占めているのはコードではなく「資料」**　です。議事録、マニュアル、説明資料、稟議の添付。ビジネスの現場でモデルの世代差が一番効いてくるのは、実はこちらだったりします。

そこで、コーディングではなくビジネス文書側の能力を検証するために、実務でそのまま発生したタスクをぶつけてみることにしました。お題は「**説明会の録画から、操作マニュアル（Word）と説明資料（PowerPoint）を作る**」。きれいにお膳立てされたベンチマークではなく、素材が雑多で、暗黙の編集判断だらけの、現場によくある難しいタスクです。

## 同じプロンプト、同じ素材、違うモデル

素材は、社内勉強会の録画（約50分）、自動書き起こし、AI生成の議事録の3点セットです。これを渡して「説明用の文書とプレゼンを作って」と頼む——という同じタスクを、Claude Opus と Claude Fable 5 のそれぞれにやらせてみました。ちなみに録画は資料はなしで操作デモのみです。

どちらも完成品は出てきます。スキーマ的に壊れたファイルが出てくるわけでもありませんし、内容が間違っているわけでもありません。Opus版も社内に配って恥ずかしくないクオリティでした。

それでも、2つを並べたときの差ははっきりしていました。そしてその差は「文章のうまさ」ではなく、**編集判断**に出ていたのです。

なお、実際の題材は社内向けの非公開機能なので、この記事では構造の差だけを抽出し、例はすべて架空の題材（「新しく導入した経費精算システムの使い方説明会」）に置き換えています。

## 差分1：進行順に並べるか、学習順に組み直すか

Opus版のスライドは、勉強会の進行をきれいになぞっていました。

> 概要 → 画面を開く → 入力する → 結果が出る → 部品の説明① → 部品の説明② → …

これは書き起こしという素材に忠実な構成です。デモの再現としては正しいですし、当日の流れを思い出すには向いています。

Fable版は順番を組み替えていました。経費精算の例でいえば、講師が雑談まじりに3回に分けて話した「締め日を過ぎた申請は差し戻される」という注意を、**冒頭に1枚の警告スライドとして集約**し、さらに表紙にも注意チップを置いていました。逆に、講師が冒頭で話した宣伝っぽい話は後ろに回しています。「読む人がどの順で理解すべきか」を基準に、素材の時系列を壊して再構成しているわけです。

書き起こしからの資料化は要約タスクだと思われがちですが、Fable版の挙動は要約というより**再編集**でした。

## 差分2：「最重要」を自分で決める

いちばん感心したのはここです。

説明会には必ず「ここを乗り越えれば後は簡単」という山場があります。経費精算の例なら「承認ルートの考え方」みたいな、急に抽象度が上がって脱落者が出るポイントです。

Opus版もこの山場を認識していて、セクションラベルで区別していました。ここまでは両者ともできています。

Fable版はさらに踏み込んで、そのスライドのタイトルを「**最重要：承認ルートの考え方**」とし、本文を3つの番号付きステップに分解し、具体例のコールアウトまで添えていました。素材のどこにも「ここが最重要」とは書かれていません。聴講者の質問が集中した箇所や講師が繰り返した説明から、**資料側が優先順位を宣言している**のです。

説明資料の良し悪しは「全部書いてあるか」ではなく「どこを太字にするか」で決まる、というのは人間の編集者の感覚ですが、それをやってきたのは予想外でした。

## 差分3：レイアウトは「内容の型」に従う

枚数と密度の差も顕著でした。

|  | Opus版 | Fable 5版 |
| --- | --- | --- |
| 枚数 | 22枚 | 15枚 |
| 支配的なレイアウト | 「左にスクショ、右に箇条書き2〜4行」が約16枚 | 同じ型の連続は最大3枚 |
| レイアウトの種類 | 実質2〜3種類 | 比較カード、設定一覧表、番号ステップ、ミニカードグリッド、フロー図、コールアウト等 |

Opus版は1枚1トピックで余白が多く、口頭発表のお供としては悪くないのですが、右カラムが半分空いているスライドが続きます。中には、箇条書きの「入力例」という項目と、その中身のボックスが画面の対角に泣き別れしているスライドもありました。

Fable版は、内容の型に応じてレイアウトを切り替えていました。

* 新旧システムの違い → 左右の**比較カード**（旧側はグレー、新側はアクセント色）
* 設定項目の説明 → **見出しセル付きの表**
* 概念の手順 → **番号付きステップ**
* 申請パターンの実例 → **箱と矢印のフロー図**

「比較は横並び」「手順は番号」「実例は図」。人間のデザイナーが無意識にやっている内容→レイアウトの対応付けを、指示なしでやっています。

## 差分4：素材に存在しないものを作る

Fable版には、書き起こしのどこにも存在しない成果物が2つ入っていました。

1つは**実例のフロー図**。講師が口頭で説明した操作の流れを、「トリガー → 処理A → 処理B → 完了通知」のような5つの箱のパイプラインとして図に起こしたものです。書き起こしには文章しかなく、図は完全にFable側の構成です。

もう1つは**設計思想の抽出**。講師が別々の文脈でポロポロ漏らした発言——「ここは自動でやらせると失敗することがある」「ここは必ずこの順で動く」——を束ねて、「**柔軟さが必要な部分はAIに、確実性が必要な部分は固定のルールに**」という1行の原則としてまとめスライドに昇格させていました。

これも要約ではありません。素材を一度バラして、教材として再鋳造しています。

## 差分5：作ったものを自分で疑う

生成プロセスも観察していたのですが、Fable 5は資料を生成した後、**全スライドを画像にレンダリングして1枚ずつ目視チェック**し、自分でバグを見つけて直していました。実際に見つけて直していたのは、たとえば：

* 図形の座標計算ミスで、接続線の幅が負の値になっていた（レンダリングすると線が変な方向に飛ぶ）
* フロー図の最後の箱がスライド右端ぎりぎりまではみ出していた
* Word側で、表の行がページまたぎで真っ二つに分断されていた
* 番号付きリストが、別のリストの続きとして「4. 5. 6.」から始まっていた

どれも、生成コードを眺めているだけでは気づきにくく、描画して初めて見えるたぐいの不具合です。「コードが書けた＝完成」ではなく「レンダリング結果が正しい＝完成」という完了条件で動いているのです。資料作成というタスクの成果物はコードではなく見た目なので、この検証ループの有無は品質に直結します。

## 念のための注意書き

公平のために書いておくと：

* これは**n=1の比較**です。プロンプトは同等ですが厳密な統制実験ではなく、モデルの出力には毎回ブレがあります。
* Opus版が「ダメ」なわけではまったくありません。構成ラベルは丁寧で、まとめスライドの完成度も高く、そのまま使える叩き台でした。差は「使える」と「ほぼ完成品」の間にあります。
* 差が出たのは文章力ではありません。1文単位で見れば両者の日本語に大差はないです。

## まとめ：資料生成は「文章生成」から「編集」のタスクになった

今回の差分を一言でまとめるなら、こうなります。

> Opusは素材を**整列**した。Fable 5は素材を**編集**した。

時系列を壊して学習順に組み直す。最重要を自分で宣言する。内容の型に合わせてレイアウトを選ぶ。素材にない図と原則を作る。完成品をレンダリングして自分で疑う。——どれも、これまで「AIに資料を作らせた後に人間がやる仕事」だったものです。

「録画と書き起こしを渡したら、配れる資料が返ってくる」が誇張でなくなったとき、私たちに残る仕事は、内容の正しさの最終確認と、「誰に何を伝えたいか」の指定だけになるのかもしれません。
