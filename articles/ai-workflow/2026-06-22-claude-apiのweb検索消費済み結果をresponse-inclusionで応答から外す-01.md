Claudeのweb検索ツールで調べ物をするエージェントを組むと、トークンの請求が思ったより速く伸びる。理由はわりと単純で、検索結果が一度きりの消費物ではないからだ。Anthropicのドキュメントによれば、会話中に取得した検索結果は入力トークンとして数えられ、しかも「同一ターン内の検索の反復でも、後続の会話ターンでも」カウントされる([Web search tool](https://platform.claude.com/docs/en/agents-and-tools/tool-use/web-search-tool))。15〜20回検索するリサーチエージェントなら、一度引いた生の結果ブロックが応答に積まれ、次のターンでまた送り返され、と何度も課金対象になる。

6月11日に出た更新は、この「積み上がり」をAPIレベルで削るための小さな、しかし実務に効くスイッチだ。リリースノートの該当行はこうある。

> The web search tool and web fetch tool now support `web_search_20260318` and `web_fetch_20260318`, adding a `response_inclusion` parameter to drop consumed result blocks from the API response for agentic workflows. No beta header is required.

ベータヘッダ不要で使える、という最後の一文が地味に大きい。すでにGAだ。

## 🧹 コンテキストから消すのと、応答から消すのは別の話

ここを混同すると効果を読み違える。Claudeのweb検索には、トークンを節約する仕組みがすでに二段ある。

一段目が**ダイナミックフィルタリング**だ。`web_search_20260209` 以降では、Claudeが検索結果に対してコードを書いて実行し、必要な部分だけを抽出してから自分のコンテキストウィンドウに載せる。生のHTMLを全部読み込んで推論する代わりに、サンドボックス内のコードで絞り込む。これは「モデルが何を読むか」を減らす仕組みで、コード実行ツール(code execution)を有効にしていることが前提になる。技術文書の検索や引用の照合のように、結果の大半がノイズになりがちなタスクで効く。

ただしダイナミックフィルタリングを使っても、生の検索結果ブロック(`server_tool_use` と `web_search_tool_result` のペア)はAPI**応答**の中には残っていた。モデルはコードの中で消費済みなのに、クライアントに返るレスポンスには丸ごと載っている。今回の `response_inclusion` はこの二段目、つまり「クライアントに何を返すか」に手を入れる。

整理するとこうなる。

| 仕組み | 削る対象 | 前提 |
| --- | --- | --- |
| ダイナミックフィルタリング | モデルのコンテキストに載る生結果 | `web_search_20260209`+ とコード実行 |
| `response_inclusion: "excluded"` | API応答に載る消費済み結果ブロック | `web_search_20260318`+ で同ターン内のコード実行が結果を消費していること |

## "excluded" が効く条件は意外と狭い

ここが設計のいちばん面白いところで、`response_inclusion` は無条件に結果を捨てるわけではない。ドキュメントの定義を引くと、このパラメータが効くのは「結果が同一ターン内の**完了したコード実行呼び出しによって消費された**場合」に限られる。

つまりダイナミックフィルタリングでClaudeがコードを書いて検索結果を読み切った、その消費済みブロックだけが対象だ。`"excluded"` を指定すると、入れ子になった `server_tool_use` と結果ブロックのペアが応答から丸ごと落ち、生の検索内容をクライアントにエコーバックする必要がないエージェントの**出力トークンコストを減らせる**。デフォルトは `"full"` のまま。

```
{
  "tools": [
    {
      "type": "web_search_20260318",
      "name": "web_search",
      "response_inclusion": "excluded"
    }
  ]
}
```

逆に言うと、Claudeが普通に検索してコンテキスト上で直接結果を推論した場合や、コード実行が完了前に一時停止(pause)した場合は、結果は常にフルで返る。後者は次のターンで送り返す必要があるからで、ここを勝手に消されたら会話が壊れる。捨てていいのは「コードがもう読み終えた、再送の必要がない結果」だけ、という線引きになっている。乱暴に全部消すのではなく、再利用の可能性がないと確定したものだけを落とす。この保守的なゲートのかけ方は、エージェント基盤を運用する側からすると安心できる挙動だ。

注意点として、削れるのは生の結果ブロックであって引用ではない。`web_search_result_location` の `cited_text` / `title` / `url` は入力にも出力にもトークンとして計上されず、常に返る。エンドユーザーに出典を表示するUIを持っているなら、結果ブロックを `"excluded"` にしても引用情報は手元に残る。ここを分けて設計しているのは妥当だと思う。

## ⏱️ 90秒の壁を、モデルに先に教える

同じ6月11日、コード実行ツールにも `code_execution_20260521` が加わった。中身のランタイムは一つ前の `code_execution_20260120` と同じで、変わったのは一点、**セルあたりの実行時間上限がツール説明文に明記された**ことだ。

各セルには90秒のウォールクロック制限があり、超えると `detection_timeout` の結果が返る。これ自体は前からある制約だが、これまではモデルがその数字を知らされていなかった。説明文に書くことで、Claudeが「この処理は重いから分割しよう」と長時間セルを事前に見積もれるようになる。こちらもベータヘッダは不要。

地味な変更に見えるが、ダイナミックフィルタリングの絞り込みコードはまさにこのセルの中で走る。検索結果が大きいほどフィルタ処理は重くなり、90秒に当たりやすい。上限を隠したままだとモデルはタイムアウトしてから学習するしかなかった。先に教えておけば、最初から分割する判断ができる。プロンプトに数字を埋め込むという素朴な手だが、エージェントの試行回数を減らす効果は実利として効く。

## 結局どう使い分けるか

組み合わせて初めて意味が出る三点だと理解している。コード実行を有効にしてダイナミックフィルタリングでモデルのコンテキストを守り、`response_inclusion: "excluded"` でクライアントに返る冗長なブロックを削り、`code_execution_20260521` で重いフィルタ処理を90秒以内に収めさせる。リサーチ系のエージェントを本番で長時間回すほど、この三つの効きが効いてくる。

コスト面を補足すると、web検索は1,000回あたり10ドル(プラス生成トークン)で、コード実行はweb検索またはweb fetchと併用する限り追加課金されない。つまりダイナミックフィルタリングのためにコードを走らせること自体は、検索とセットなら無料の範囲だ。これは「結果はコードで前処理してから読め」という設計思想を価格でも後押ししている。

割り切りも要る。`"excluded"` で生結果を応答から消すと、エージェントが実際に何を見て判断したかをログから再構成しづらくなる。デバッグやトレーサビリティを重視するなら、開発中は `"full"` のまま、本番のトークン勝負の局面で `"excluded"` に切り替える、くらいの運用が現実的だろう。利用できるモデルも `web_search_20260318` 対応のもの(Fable 5、Opus 4.8、Sonnet 4.6 など)に限られる点は確認しておきたい。

派手な新モデルの話ではない。けれど、エージェントが自分でコードを書いて情報を前処理し、いらなくなったものを文脈からも応答からも外していく。この「コードでコンテキストを掃除する」流儀が、サーバーツールのパラメータ一個ぶんずつ着実に整備されている。エージェントの運用コストを律速するのは賢さより文脈の衛生だ、という実感に、APIの設計がようやく追いついてきた感がある。
