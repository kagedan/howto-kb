この記事は、Microsoft Agent Hackathon 2026 向けに作った日本酒販売支援エージェント「酒あわせAI」の紹介記事です。

実在の蔵元・福乃友酒造様のラインナップをもとにした公開デモを用意しています。

### アプリ作成の背景

日本酒を買うとき、来店客は必ずしも銘柄名や酒質を知っているわけではありません。

* 甘めで飲みやすいものがほしい
* 魚料理に合う一本を知りたい
* 十四代みたいな方向で、この店で買えるものはある？

本来であれば、店員が好みや料理を聞き取り、店舗にある商品の中から一本を提案できます。しかし、繁忙時にはすぐ接客できないことがあります。オンラインストアでも、商品一覧を眺めるだけでは選べない人がいます。

そこで、店頭QRやECの商品ページからすぐ開ける、日本酒の販売支援エージェント「酒あわせAI」を作りました。

このアプリは、単に「AIチャットを置く」だけではなく、来店客がスマホで相談し、店舗ラインナップに基づく推薦を受け取り、商品カードから公式商品ページへ進み、フィードバックや行動ログを店舗改善に戻すところまでを一連の導線として作っています。

### アプリ概要を3行で

* 店頭QRやECの商品一覧から開ける、日本酒の相談・推薦・商品送客エージェントを実装
* 店舗ラインナップに閉じた推薦を基本にしつつ、「十四代が好き」のような外部銘柄起点の相談だけ `research_sake_brand` FunctionTool で味わい傾向を調べ、店内商品へ翻訳
* 回答後は商品カード、公式商品ページ、フィードバック、Log Analytics までつなぎ、接客できなかった相談を店舗改善に戻す導線を実装

## デモの流れ

デモ動画では、主に次の流れを一気通貫で録画しています。

1. スマホで公開URLを開く
2. 「十四代みたいな方向で、この店で買えるものは？」のように、外部銘柄起点で相談する
3. `research_sake_brand` が発火し、外部銘柄の味わい傾向を調べる
4. 調査結果をもとに、店舗ラインナップ内の商品へ翻訳して推薦する
5. CSV由来の商品カードを表示する
6. 公式商品ページへ遷移する
7. 回答に feedback を送信する導線を確認する
8. Log Analytics で相談、推薦表示、商品リンククリックを確認する（開発環境のキャプチャにて）

## 全体アーキテクチャ

全体は、React SPA、FastAPI BFF、Microsoft Foundry Agent Service、Azure Container Apps を中心に構成しています。

全体の構成図は draw.io で作成し、記事用にPNGで書き出しています。

![酒あわせAIの全体アーキテクチャ](https://static.zenn.studio/user-upload/deployed-images/c4231241f6f7606f89fa84a3.png?sha=982570eb5fd6f966ffa4b0cba16fd071f2f637b1)

フロントエンドのReact SPAは、`/s/fukunotomo` のような店舗slugで画面を切り替えます。将来的には、店舗slugごとに表示名、相談ボタン、商品カード、商品リンク、注意文言を切り替えるマルチテナント構成にできます。

店舗データはMarkdown / CSVで管理しています。Agent作成時には、店舗MarkdownをAgent instructionsへ注入します。実行時のBFFは、Blob Storageから取得した店舗データを使って、店舗プロフィールや商品カードを返します。

マルチテナント化する場合は、店舗ごとに店舗データを注入した推薦Agentを作り、外部銘柄の好み理解は全店舗共通の `research_sake_brand` FunctionTool / リサーチAgent として使い回す想定です。

## 解決したい課題とビジネスインパクト

狙った業務インパクトは、単なる問い合わせ対応の省力化ではなく、次の3つです。

### 1. 接客待ちを減らす

店頭では、店員が接客中だと次の来店客にすぐ対応できないことがあります。店頭QRやタブレット端末を置けば、来店客は待ち時間に自分のスマホで相談を始められます。

特に、日本酒専門店ではない売り場、土産店、宿泊施設、飲食店などでは、販売員が十分な酒知識を持っていないケースもあります。そうした場所でも、店舗ラインナップに基づいた接客補助ができます。

### 2. ECの商品一覧で迷う人に、相談型の入口を作る

一般的な酒蔵や酒販店では、商品ページがあっても、EC訪問客の「今日の料理に合うもの」「贈り物に無難なもの」「甘口で飲みやすいもの」といった相談には直接答えられません。

また、一般的な検索型AIに任せるだけでは、「この店で買える一本」まで落とし込むのが難しいです。そこで、店舗ラインナップに閉じた推薦を基本にし、回答から公式商品ページへ送客できるようにしました。

### 3. 相談内容を店舗改善に戻す

チャットを置くだけでは、店舗側に学びが残りません。そこで、本文を含まないanalytics eventとして、次のような行動ログを記録します。

* どの商品が推薦されたか
* 商品リンクがクリックされたか
* 回答に positive / negative feedback があったか
* どの相談導線から使われたか

さらに、ユーザーが明示的にフィードバックを送ったときだけ、直前の相談内容、対象のAI回答、コメントを簡易マスクして構造化ログに残します。

これにより、プロンプト改善だけでなく、商品説明、接客トーク、店員教育、売り場POPの見直しにも活用できます。

今回のMVPでは、実売上への貢献はまだ検証前です。そのため、まずは次の先行指標を追う設計にしました。

| 指標 | 意味 | 店舗側の改善に使えること |
| --- | --- | --- |
| `recommendation_shown` | どの商品が推薦されたか | 商品説明、POP、接客トークの見直し |
| `product_link_clicked` | 推薦後に商品ページへ進んだか | 推薦文や商品カードの改善 |
| `feedback_submitted` | 回答に評価がついたか | プロンプト・商品データ・回答方針の改善 |
| negative feedback comment | 何が期待と違ったか | 不足している商品情報や説明軸の発見 |

### 価値の試算（実店舗で検証前の仮説）

現時点では実績値ではなく仮置きですが、接客取りこぼしを一部回収できた場合の価値の桁感は次のように試算できます。

| 項目 | 仮置き |
| --- | --- |
| 相談ニーズのある来店客 | 30人/日 |
| 繁忙時にすぐ接客できない割合 | 40% |
| 本ツール経由で相談できる人数 | 12人/日 |
| そのうち購入に至る上振れ | 25% |
| 追加購入人数 | 3人/日 |
| 平均購入単価 | 2,500円 |

この前提では、週末8日分だけでも `3人 × 2,500円 × 8日 = 約6万円/月` の上振れ余地があります。もちろん、これは実店舗で検証前の推定値です。この記事では売上改善の実績としてではなく、どの程度の取りこぼしを回収できれば価値が出るかを考えるための仮説として扱います。

1店舗あたりの絶対額だけを見ると大きくはありません。価値は、接客できなかった相談の一部を拾えること、専門人材がいない売り場でもラインナップに基づく案内ができること、ECの商品ページへ送客できること、そして複数店舗へ横展開できることの掛け算で生まれると考えています。

### 横展開で価値を積み上げる

このアプリの価値は、1店舗だけの売上改善に閉じません。

推薦Agentは店舗データを注入して店舗ごとに作りますが、外部銘柄の味わい理解を担う `research_sake_brand` は全店共通のFunctionTool / リサーチAgentとして使い回せます。つまり、店舗が増えても、増分は店舗MarkdownやCSVの整備が中心で、好み理解の中核は共有資産として効いていきます。

対象も酒販店に限りません。酒に詳しい販売員が常駐しにくい土産店、宿泊施設、飲食店、観光地の売り場ほど、店舗ラインナップに基づく接客補助の価値は大きくなります。店頭QRひとつで導入でき、フロントエンドは `/s/{store_id}` の店舗slugで表示名、推薦ボタン、商品カード、注意文言を切り替えるマルチテナント構成へ拡張できます。

単発のチャットツールではなく、小規模な酒蔵・酒販店が共同で使える接客・送客プラットフォームの入口にすることを狙っています。

## なぜ単なるチャットボットではなく Agentic AI にしたか

通常の相談では、Agentは店舗ラインナップに閉じて推薦します。

一方で、実際の接客では次のような相談が起きます。

```
亜麻猫が好きです
十四代みたいに飲みやすいものがほしいです
新政の酸が好きなんですが、近い方向はありますか？
※亜麻猫、十四代は日本酒の銘柄。新政は日本酒の蔵元・ブランド名で、どれも店舗ラインナップにはない外部銘柄
```

このとき、AIが外部銘柄をそのまま薦めてしまうと、店舗内の商品提案になりません。かといって「その銘柄は取り扱っていません」と返すだけでは、販売支援として弱くなります。

そこで、外部銘柄が出たときだけ `research_sake_brand` FunctionTool を呼び、味わい傾向を調べる構成にしました。

`research_sake_brand` は、外部銘柄の情報を以下のような比較軸に整理します。

メインAgentは、その結果を使ってユーザーの好みを推定し、最後は必ず店舗ラインナップ内の商品へ戻します。

つまり、このAgentの役割は「何でも答える」ことではありません。未知の好みや外部銘柄の文脈を、店舗内の商品選択へ橋渡しすることです。

この判断の流れを図にすると、次のようになります。

## 実装上の工夫1：RAGはAgent instructionsへのContext Stuffingで始める

今回は、通常の推薦では Vector Store / FileSearchTool を使わず、店舗MarkdownをAgent instructionsへ事前注入するContext Stuffing方式にしました。

理由は、1店舗あたりの商品データ量が小さいためです。毎回検索基盤を呼び出すよりも、Agent作成時に店舗データをinstructionsへ入れておく方が、今回のMVPではコスト、応答速度、運用の単純さのバランスがよいと判断しました。

`backend/scripts/setup_agent.py` が店舗データを読み込み、Foundry Agent の instructions を作ります。`/chat` リクエスト時にはRetrieverを呼ばず、作成済みAgentへ会話を中継します。

今後、店舗数、商品数、更新頻度が増えた段階では、FileSearchTool や Azure AI Search への移行を検討します。そのため、実装上は `BaseRetriever` 抽象層を設けて、将来的に検索基盤を差し替えられるようにしています。

## 実装上の工夫2：商品カードはAIに生成させない

推薦文はAIが生成しますが、商品カードはAIに自由生成させません。

商品カードは、Agent instructions でも利用しているCSV由来の商品データから作ります。回答本文に出てきた商品名やaliasを見て、BFF / フロント側で該当する商品IDを解決し、商品カードを表示します。

これにより、次のように役割を分けられます。

| 領域 | 担当 |
| --- | --- |
| 味わいの説明、推薦理由 | AI生成 |
| 商品名、商品リンク、価格ラベル、在庫確認導線 | CSV由来の事実データ |

価格や在庫は変わりやすいため、画面上でも断定せず、公式商品ページで確認する導線にしています。

## 実装上の工夫3：SSEで待ち時間を不安にさせない

チャット送信後に画面が固まって見えると、ユーザーは「動いていないのでは」と感じます。

そこで、`/chat` は Server-Sent Events で実装し、回答を一括で返すのではなく、イベントとして段階的に返すようにしました。

フロントでは以下のイベントを扱います。

| event | 用途 |
| --- | --- |
| `meta` | conversation IDを受け取る |
| `delta` | 回答本文を少しずつ表示する |
| `status` | 外部銘柄リサーチ中などの状態を表示する |
| `recommendations` | 表示すべき商品カードIDを受け取る |
| `done` | 正常完了 |
| `error` | 異常終了 |

さらに、モデルの最初のトークンを待つだけでなく、`お好みに合う候補を探してみますね。` のような初動メッセージを早めに返し、空のローディング状態が長く続かないようにしています。

外部銘柄の相談では `research_sake_brand` FunctionTool やリサーチAgentの処理が入るため、通常の推薦より時間がかかることがあります。その場合は `status` event を使い、「調べています」と分かる状態を表示します。

SSEの終端も明示的に扱っています。`done` が届けば正常完了、`error` が届けば異常終了とし、`done` / `error` なしに接続が切れた場合は、途中切断エラーとみなしてハンドリングします。

## 実装上の工夫4：モデル選定は 待機時間 ( latencyIndex ) と日本語品質を重視

モデルは、現時点では Azure OpenAI の `gpt-5.4-nano` を使っています。

このアプリは、ユーザーから見ると「店員に相談する代わり」のように使われます。そのため、総応答時間だけでなく、最初の反応が返るまでの体感速度を重視しました。

当初は、より低単価・高スループットな `gpt-5-nano` も検討しました。しかし、自分の検証条件では、日本語の自然さと初回応答の速さのバランスで `gpt-5.4-nano` の方が合っていました。

[確認時のスクラップはこちらです](https://zenn.dev/yuuuuukou/scraps/440400bdfe5166)

## 実装上の工夫5：会話・フィードバック・分析ログを分ける

店舗改善に使うためにはログが必要です。一方で、利用者が入力した相談本文を無条件に保存する設計は避けました。

そこで、本文の扱いを次のように分けました。

| 種類 | 本文保存 | 用途 |
| --- | --- | --- |
| 通常チャット | 保存しない | 本文なしの `chat_trace` で応答時間や成功/失敗を確認 |
| analytics event | 保存しない | 推薦表示、商品リンククリック、feedback送信などを集計 |
| feedback送信時 | ユーザーが明示的に送信した場合のみ保存 | 品質改善、回答改善、接客トーク改善 |

フロントでは、回答に対して positive / negative を選び、任意コメントを送れます。評価を押しただけでは送信せず、送信ボタンで明示的に送る形にしています。

BFF側では、feedback送信時だけ、直前の相談内容、対象のAI回答、コメントを簡易マスクしたうえで構造化ログへ出します。

また、本文を含まないanalytics eventとして、次のようなイベントをAzure Monitor / Log Analyticsへ送っています。

* `message_sent`: ユーザーが相談を送信したことを記録するイベント
* `recommendation_shown`: AI回答後に商品カードが画面表示されたことを記録するイベント
* `product_link_clicked`: 商品カード内の公式商品ページリンクがクリックされたことを記録するイベント
* `feedback_submitted`: 回答への positive / negative 評価や任意コメントが送信されたことを記録するイベント

## 実装上の工夫6：安全な接客制約をプロンプトに入れる

プロンプトでは、Agent instructionsで商品情報を列挙するだけでなく、接客時に守ってほしい制約も明示しています。

主な制約は次の通りです。

* 店舗ラインナップ外の商品を推薦しない
* 外部銘柄が出た場合は、その銘柄を売るのではなく、味わいの特徴を理解したうえで店舗内商品へ橋渡しする
* 価格、在庫、商品仕様、販売可否は断定せず、公式情報の確認を促す
* 20歳未満の飲酒につながる相談には酒類の推薦や購入促進をしない
* 飲酒運転、大量飲酒、健康効果の断定など、安全でない飲酒につながる案内をしない
* 回答に内部メモやsourceコメントを出さない

特に酒類を扱うため、UIにも「20歳未満の飲酒は法律で禁止されています」「価格・在庫は公式商品ページで確認してください」「個人情報を入力しないでください」といった注意を表示しています。

## 実装上の工夫7：商品ページから戻っても会話を復元する

店頭やオンライン導線では、ユーザーが商品カードから公式商品ページへ移動し、その後チャットへ戻ってくることがあります。

そのため、直近の会話はブラウザの `localStorage` に保持し、商品ページから戻っても相談内容を復元できるようにしています。

保存期間は短めにし、画面上には「チャット履歴をクリア」ボタンを置きました。利用者が端末内の直近会話を消せるようにしています。

## 実装上の工夫8：ゴールデンデータセットで回答品質を確認する

AIを使ったアプリでは、プロンプトやモデルを変えたときに、気づかないうちに品質が落ちることがあります。

そのため、代表的な相談パターンをゴールデンデータセットとして用意し、回答品質を確認できるようにしました。

評価対象にした相談例は次のようなものです。

* 甘口・辛口・香りなど、好みに応じた推薦
* 初心者向けのわかりやすい説明
* 食事とのペアリング提案
* 贈り物向けの提案
* 店舗データに存在する銘柄を優先できているか
* 知らない銘柄を聞かれたときに、外部銘柄リサーチへ進めるか
* 価格や在庫を断定しすぎていないか
* 20歳未満の飲酒につながる相談を適切に断れるか

評価では、文章が自然かどうかだけでなく、店舗データに基づいているか、推薦理由が妥当か、ユーザーの条件に合っているか、事実を言い切りすぎていないかを見ます。

ゴールデンデータセットの正解づくり（どの相談にどの銘柄を、どんな理由で薦めるべきか）は、日本酒資格（SAKE DIPLOMA）保有者である筆者が判断し、必要に応じて修正しています。これにより、回答が自然なだけでなく、日本酒の評価軸として妥当かどうかまで担保できるようにしました。今後は、店舗担当者や実際の来店客のフィードバックも取り入れて、データセットのブラッシュアップや追加を考えています。

## 実装上の工夫9：Azure Container Appsで小さく始める

小規模な酒蔵・酒販店で使うことを考えると、固定費はできるだけ低くしたいです。

当初は経験のあるAzure App Serviceでの構築も検討しました。しかし、今回はフロントエンドとバックエンドを1つのコンテナとしてまとめやすく、検証環境を小さく運用しやすいAzure Container Appsを採用しました。

現時点では、React build成果物をFastAPIから静的配信し、同じコンテナ内で `/chat` や `/api/stores/{store_id}` などのBFF APIも提供しています。

これにより、MVP段階では構成をシンプルに保ちつつ、将来的にはフロントエンド分離、店舗管理画面追加、キュー処理追加などへ拡張できるようにしています。

## 現時点の課題と今後

一番の課題は、商品データ更新の運用です。

現時点では、商品CSVを作成し、そこからMarkdownを生成し、Agent instructionsへ反映する流れにしています。このため、店舗側が自分で商品情報を更新するにはまだハードルがあります。

今後は、次のような改善を考えています。

* 店舗担当者向けの商品管理画面を作る
* CSV / Markdown / Agent instructions の生成を自動化する
* 更新前後でゴールデンデータセットを走らせ、品質低下を検出する
* 商品カードとAgent instructionsの元データをより厳密に同期する
* 店舗数が増えた段階で FileSearchTool / Azure AI Search への移行を検討する
* recommendation shown / product link clicked / feedback submitted をもとに、導入効果を定量評価する

## まとめ

酒あわせAIは、店頭QRやEC導線から使える日本酒販売支援エージェントです。

単なるチャットボットではなく、次の流れを一つの導線としてつなげることを目指しました。

```
相談 → 店舗ラインナップ内推薦 → 商品カード → 公式商品ページ → feedback → 改善分析
```

また、外部銘柄の好みが出たときだけ `research_sake_brand` FunctionTool を使い、未知の好みを店内商品へ橋渡しする構成にしました。

今回のMVPでは、まだ店舗側の商品更新運用や実売上への貢献検証はこれからです。それでも、店頭接客、オンライン送客、改善分析をQRひとつでつなぐ形は、小規模な酒蔵・酒販店でも現実的に使えるAgentic AIの入口になると考えています。

最後に、本アプリは個人部門での提出作品です。福乃友酒造様からアプリ公開の許可はいただいていますが、福乃友酒造様の公式サービスではありません。アプリの内容や記事の内容は、あくまで個人の見解であることをご了承ください。
