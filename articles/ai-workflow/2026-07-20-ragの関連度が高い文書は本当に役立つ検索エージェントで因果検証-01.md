検索質問とほとんど似ておらず、答えそのものも書いていない。それでも、その文書を消すとAIエージェントは正解へたどり着けなくなる。

2026年7月、次の論文が公開された。

[Bridge Evidence: Static Retrieval Utility Does Not Predict Causal Utility in Multi-Step Agentic Search](https://arxiv.org/abs/2607.15253)

この研究は、そんな「橋渡し文書」が検索エージェントで大量に見落とされている可能性を示した。

![質問から橋渡し文書の固有表現を経由し、最終証拠へ到達する検索エージェントのイメージ](https://static.zenn.studio/user-upload/deployed-images/6d2a6e39d6e4c2844718e0f6.png?sha=f6b2991236719975a6dcb044af5015fb4421ac80)

RAG（検索拡張生成）は、外部の文書を検索し、その内容をLLMの回答材料にする仕組みだ。通常は「質問に似ている文書」「その文書だけで答えを改善できる文書」を高く評価する。

しかし複数回検索するエージェントでは、文書の価値が**次に何を検索できるようになったか**に現れる。

この記事では、23,322件の公開集計値を再計算し、論文の因果評価を技術的に分解する。さらに、既存RAGをどのように計測し直すべきかを実装レベルへ落とす。

## 3分でわかる結論

| 観点 | 論文の結果 |
| --- | --- |
| 静的有用性と因果的有用性 | 順位相関`-0.0257`。実質的に無関係 |
| 橋渡し文書 | 23,322件中8,331件、35.72% |
| 別の測り方でも同じ傾向か | BM25とクロスエンコーダが一致した9,600件でも27.16% |
| 橋渡しの仕組み | 識別力の高い固有表現が次の検索語へ入る確率は4.02倍 |
| 重要な制約 | 1モデル、HotpotQA、最大4検索、静的評価軸の偏りあり |

実務への示唆は単純だ。

> エージェント型RAGでは「今の質問への関連度」だけで文書を捨ててはいけない。「次の検索を前進させたか」も測る必要がある。

## まず、普通のRAGと検索エージェントは何が違うのか

通常のRAGは、質問`q`に近い文書`d`を検索し、その文書をLLMへ渡して答えを作る。評価も1回で閉じる。

一方、ReAct型の検索エージェントは、推論と検索行動を交互に行い、検索結果を読んで次の検索語を作る。

例えば、次の架空の質問を考える。

> 映画Xの監督と同じ大学を卒業した科学者Yが、賞Zを受賞した年は？

最初に必要なのは監督名や出身大学であって、受賞年ではない。大学名だけを含む文書は、元の質問から見れば遠く、答えも含まない。しかし大学名が分からなければ、科学者Yへ向かう次の検索を作れない。

この文書は「回答文書」ではなく「検索経路を開く文書」だ。論文はこれをbridge evidenceと呼ぶ。

## 実験：文書を1つ消して、その後をやり直す

研究は[HotpotQA](https://aclanthology.org/D18-1259/)の開発セットから、必要な証拠が検索候補内に存在する1,000問を層化抽出した。HotpotQAは複数のWikipedia文書をまたいで答える11.3万問規模のデータセットだ。

検索エージェントの構成は次の通り。

BM25は主に単語の一致を評価し、クロスエンコーダは質問と文書をまとめて読んで意味的な関連性を評価する。研究では両者を直列に使って候補を絞った。

| 要素 | 設定 |
| --- | --- |
| エージェント | Qwen2.5-7B-Instruct |
| 行動形式 | [ReAct](https://arxiv.org/abs/2210.03629)型の思考・検索・回答 |
| 1段目検索 | Wikipedia段落に対するBM25、上位50件 |
| 再ランキング | クロスエンコーダ、上位10件をLLMへ提示 |
| 最大検索回数 | 4回 |
| 生成 | temperature 0、sampling無効 |

各質問を通常どおり実行した後、エージェントが実際に読んだ文書を1つずつ削除し、その時点から残りの軌跡を再実行する。

```
通常:    質問 → 文書A → 文書B → 検索語C → 正解
介入後:  質問 → 文書A → [文書Bを削除] → 別の検索語 → 不正解
```

この差分なら、「文書Bは質問に似ているか」ではなく、「文書Bが存在したことで検索全体がよくなったか」を測れる。

## 2つの有用性指標

### 静的RAG有用性（SRU）：その文書だけで答えが改善したか

Static RAG Utility（SRU）は、検索履歴を持たない回答モデルへ質問と1文書だけを渡し、文書なしの場合より回答品質が上がったかを測る。

```
SRU = 文書ありの回答品質 - 文書なしの回答品質
```

これは従来RAGの考え方に近い。文書単体で答えられるなら高評価、答えに直接関係しなければ低評価になる。

### 反実仮想で測る軌跡の有用性（CTU）：消すと検索全体が悪化するか

Counterfactual Trajectory Utility（CTU）は、文書を削除した仮想的な再実行との違いを3方向で測る。

1. 最終回答の品質は下がったか
2. 次の検索語が取得する正解証拠は減ったか
3. 正解までの検索ターンが増えたか

3成分をそれぞれmin-max正規化し、同じ重みで加算する。論文の正規化では`CTU = 1.4`が「削除しても何も変わらない」点になり、それより上なら因果的に役立ったと分類する。

CTUは文書固有の永久的スコアではない。同じ文書でも、使うモデル、途中の状態、検索履歴が変われば効果は変わる。正確には「ある状態のあるエージェントにとっての文書価値」だ。

## 結果：35.72%が「静的には無価値、因果的には必要」

![静的RAG有用性と因果的軌跡有用性を2軸で分類した4象限図](https://static.zenn.studio/user-upload/deployed-images/1e5874ddce7e45f183e23d8c.png?sha=7b0f138046fde3016a5b7758de13cfb9b2d3e231)

**筆者再計算：** 論文表3の件数から割合を再計算すると、次の通り一致する。

| 分類 | 件数 | 再計算 |
| --- | --- | --- |
| 静的にも因果的にも有用 | 262 | 1.12% |
| 静的には有用、因果的には不要 | 508 | 2.18% |
| 静的には無用、因果的には有用 | 8,331 | 35.72% |
| どちらでも無用 | 14,221 | 60.98% |

最も重要なのは右上の橋渡し文書だ。質問と文書だけを読む、履歴なしの回答モデルには役立たない。しかし削除すると、最終回答、次の検索、または検索効率が悪化した。

SRUとCTUのSpearman順位相関は`-0.0257`だった。順位相関は`-1`から`1`の範囲を取り、`0`に近いほど一方の順位から他方を予測しにくい。論文の説明どおり、二乗すると約`0.066%`であり、ほとんど情報を共有していない。

## 再計算で見えた「強い数字」と、その罠

因果的に有用な文書は、262件と8,331件を合わせた8,593件だった。そのうち橋渡し文書は、

となる。一見すると「必要な文書のほぼすべてを従来RAGが見落とす」と読める。

しかし、この解釈は強すぎる。静的に有用と判定された文書自体が、23,322件中770件、わずか3.30%しかないからだ。単一文書では答えられない複数段階（multi-hop）問題を、履歴なしの回答モデルへ解かせたため、SRU軸がゼロ付近に圧縮されている。

論文自身も、この偏りを主要な限界として認めている。したがって、35.72%や96.95%だけを切り出して「既存RAGの大半が間違い」と結論づけるべきではない。

中心的な発見は割合の大きさではなく、**静的有用性と因果的有用性がほぼ独立だったこと**だ。

## BM25とクロスエンコーダでも27.16%残った

履歴なしの回答モデルが弱いだけではないか。研究は別の測り方による確認として、SRUを実際の検索スコアへ置き換えた。

* BM25とクロスエンコーダの両方が高評価なら「静的に高い」
* 両方が低評価なら「静的に低い」
* 2つが不一致の文書は除外

両者が一致した9,600件では、橋渡し文書が2,607件だった。

さらに因果的に有用だった3,564件のうち、橋渡し文書は73.15%を占める。履歴なしの回答モデルを使わなくても、検索器から低く評価された文書が軌跡を支える現象は残った。

ただし、この確認では元の23,322件から13,722件、58.84%が「BM25とクロスエンコーダの不一致」として除外されている。27.16%は全データの無条件割合ではなく、2つの評価器が同意した部分集合での割合だ。

## なぜ橋渡し文書が働くのか：答えではなく固有表現を渡す

論文は227,139件の「固有表現×検索ステップ」を調べている。OER（Observable Entity Relevance）は、ある固有表現が関連文書と非関連文書をどれだけ見分けるかを表す指標だ。OERの高い表現は、非関連文書だけに出る表現より、次の検索語へ伝播しやすかった。

| 固有表現 | 次の検索語に入った割合 |
| --- | --- |
| 識別力が高い | 6.1% |
| 非関連文書だけに存在 | 1.5% |

比率は`6.1 / 1.5 ≒ 4.07`だが、未丸め値から計算した論文報告は4.02倍だ。

つまり橋渡し文書の価値は、回答文を直接供給することではない。人物名、作品名、組織名など、**次の検索空間を狭める語**をエージェントへ渡すことにある。

ここにも注意点がある。論文は「高OERの固有表現ほど次の検索へ伝わる」ことを支持したが、その固有表現が橋渡し象限へ特に集中するという、より細かい仮説は本実験で再現しなかった。メカニズムの一部は強いが、局在まで証明されたわけではない。

## 既存RAGをどう改善するか

CTUをオンライン検索のたびに正確に計算するには、文書を1つずつ消して軌跡を再実行する必要があり、高価すぎる。ここからは**筆者提案**として、CTUを主にオフライン評価へ使い、オンラインでは代理信号を設計する方法を考える。

### 1. 類似度以外の候補を少数残す

上位`k`件をすべて類似度だけで埋めず、固有表現の新規性、文書間の多様性、既知エンティティとの接続性を基準に探索枠を残す。

### 2. 次の検索語を評価する

最終回答だけでなく、各ステップで次を記録する。

* 新しい固有表現を獲得したか
* 次の検索で正解根拠（gold evidence）の順位が上がったか
* 同じ検索を繰り返していないか
* 文書削除で次の検索語が変わるか

### 3. 軌跡ログを保存する

最低限、`query`、`shown_doc_ids`、`next_query`、`answer`、`turn_count`を保存する。文書を再ランキングした一覧だけではなく、実際にモデルへ見せた文書IDが必要だ。

### 4. 代表タスクだけ反実仮想評価する

全リクエストで文書を削除する必要はない。オフライン評価セットに対して、次のような省略実験を回せばよい。

```
def evaluate_document(base_run, document_id, replay_agent):
    counterfactual = replay_agent(
        state=base_run.state_before(document_id),
        omitted_document_id=document_id,
    )

    return {
        "answer_delta": base_run.answer_score - counterfactual.answer_score,
        "next_query_delta": base_run.next_retrieval_score
        - counterfactual.next_retrieval_score,
        "extra_turns": counterfactual.turns - base_run.turns,
    }
```

これは論文実装そのものではなく、評価思想を示す最小疑似コードだ。実際には再現可能なデコーディング、状態スナップショット、スコアの正規化、失敗時の扱いが必要になる。

## 実運用向けの評価ダッシュボード

検索エージェントのKPIは、従来のRecall@kやnDCGだけでは足りない。次の3層に分けると見やすい。

| 層 | 指標例 | 答える問い |
| --- | --- | --- |
| 検索単体 | Recall@k、nDCG、BM25/埋め込みスコア | 現在の質問に近い文書を取れたか |
| 遷移 | next-query recall、entity propagation | 次の正しい検索を作れたか |
| 軌跡全体 | 最終正答、ターン数、文書削除差分 | その文書が結果を変えたか |

この3層を分ければ、「検索スコアは高いのにエージェントが迷う」「低スコア文書を残すと正答率が上がる」といった現象を追跡できる。

## この論文だけではまだ言えないこと

結果を製品へ適用する前に、少なくとも次を留保したい。

1. Qwen2.5-7B-Instructという1モデルでのCTUであり、別モデルへそのまま移らない
2. WikipediaとHotpotQAに限定され、企業文書やコード検索では分布が違う
3. temperature 0でもGPU計算は完全決定的にならず、個別CTUにノイズが入る
4. 最大4検索という短い軌跡であり、長時間エージェントでは影響が変わる
5. CTUの3成分は同じ重みで、min-max正規化の範囲に依存する
6. 橋渡し割合の主分析は、履歴なしの回答モデルの弱さと軸の偏りを含む
7. BM25＋クロスエンコーダ確認は、評価が不一致の58.84%を除外している

また、2ターンから4ターンへ増えるほど橋渡し文書が増えるように見える表があるが、論文はこの解釈を撤回している。分類に使うCTUへターン数が含まれ、グループ分けにも反実仮想側のターン数を使ったため、機械的な交絡が生じたからだ。

都合のよい二次仮説を撤回し、残る証拠を限定した点は、この論文の評価できるところでもある。

## 結論

従来RAGは、文書を静止画として評価する。検索エージェントは、文書を次の行動へつながる状態遷移として使う。

この違いを無視すると、答えを直接含まないが、正しい人物名や組織名を渡して次の検索を開く文書を捨ててしまう。

公開値の再計算では、主分析の橋渡し文書は35.72%。静的軸の偏りを避けたBM25＋クロスエンコーダの部分集合でも27.16%だった。正確な割合はモデルやデータで変わるが、「関連度」と「因果的な役立ち度」が別物である可能性は無視できない。

これからのエージェント型RAGでは、`この文書は質問に似ているか`だけでなく、次を測る必要がある。

> この文書があったから、エージェントは次に正しい検索をできたか。
