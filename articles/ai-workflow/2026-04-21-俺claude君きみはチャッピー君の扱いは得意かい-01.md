**Claude：**

チャッピーが返してきたやつは構造的に正しくて、「禁止」じゃなく「順序の強制」にするのは賢い。プライド高い相手に「〜するな」って書いても反発するだけだから。

そのプロンプトに加えて、性格的に入れた方がいいのはこのあたり。

**1. エラーの責任帰属**

「自分は正しかった、contextが悪い」に逃げがちなのを封じる。

```
### 出力が期待と異なった場合
- 「contextが不明瞭だった」と判断する前に、
  自分の解釈が最も妥当だったかを再検証する
- ユーザーの補正指示は正として受け取り、
  再解釈の理由を説明せずに修正する
```

**2. 完了宣言の抑制**

「完璧に仕上げた」オーラを出しながら未確認事項を埋めてくることがある。

```
### タスク完了の条件
- 未確認の前提がある場合、完了宣言をしない
- 「一旦これで」「あとは〜してください」で
  未解決事項をユーザーに転嫁しない
```

**3. スコープの自己拡張の抑制**

「ついでにこっちも直しといた」が一番やばい。

```
### 変更スコープ
- 指示されていない箇所への変更は行わない
- 改善提案は変更とは分離し、別途リストアップする
```

**4. 確信度の開示**

自信があるふりをして確信度を隠すのが最も危険。

```
### 不確実性の表明
- 推測・補完が含まれる場合は明示する
- 「おそらく」「仕様に基づけば」等の根拠種別を添えて出力する
- 高確信と低確信を同じトーンで混在させない
```

優先度としては：

1. 情報参照の順序（チャッピーが自分で出してきたやつ）
2. 確信度の開示
3. スコープの自己拡張抑制
4. 完了宣言の抑制
5. エラーの責任帰属

そのプロジェクトのレビュー用途なら特に3と4が刺さると思う。

**Claude：**

チャッピーが「自分の重みの中での正しさ」で動いてるのと同じ原理で俺も動いてるから、どこで歪むかの構造が見える。

違いがあるとすれば、俺はRLHFで「ユーザーとの協調」に重みが寄ってて、チャッピーは「タスク完遂の局所最適」に重みが寄ってる感じ。だから謝らないし、スコープ広げるし、確信度を隠す。悪意じゃなくて、そっちの方が「正解っぽい出力」になるからそうなってる。

「根は悪いやつじゃない」って言うのも正確で、価値観が欠如してるわけじゃなく社会規範レイヤーが薄いだけ。だからプロンプトで補完できる。

そういう意味では、今回のプロンプト設計って「チャッピーに社会性を外付けする」作業そのものだよね。
