ChatGPT や Claude に、「自分のアプリで使える成果物」を作らせたい。しかし Web 版のAIチャットは任意のAPIを叩けないし、プラットフォームごとにプラグインやコネクタを作って回るのは重い。かといってアプリにLLMを内蔵すれば、推論コストとモデル運用を永遠に抱え込む。

この記事では、**APIキーもコネクタ設定も一切なしで、任意のAIをアプリの生成エンジンとして働かせる設計パターン**を紹介する。筆者はこれを実際のプロダクトで実装・実証し、**外部ハーネス (External Harness)** と名付けた。

## 「アプリがAIを呼ぶ」ことの構造的な限界

AI統合の常識は「アプリがモデルを呼ぶ」である。APIキーを持ち、プロンプトを組み、応答を検証して使う。この構成を**内部ハーネス**と呼ぶことにする — ハーネスとは、エージェントAIの文脈で、モデルの生の知性を信頼できる仕事に変える足場(契約・ツール・検証・安全装置)のことだ。内部ハーネスでは、ハーネスとモデルを同じ事業者が運用する。

内部ハーネスには以下の限界がある。

1. **ユーザーの文脈を持てない。** ユーザーが毎日使っているAIは、その人の仕事・文体・好みを記憶している。アプリに内蔵したモデルは、その文脈を永遠に持てない。
2. **推論コストをアプリが払う。** 生成のたびに課金され、品質を上げるほど高くつく。

そこで矢印を逆向きにする。**アプリがAIを呼ぶのではなく、AIがアプリに成果物を持ち込む。** アプリ側はモデルを一切持たず、代わりに「どんなAIでも仕事を持ち込める受け口」を公開する。

## 定義

> **外部ハーネス**とは、(1) AIが読むための機械可読な**仕様書**、(2) AIの成果物を運ぶ普遍的な**搬送路**(URL)、(3) 成果物を検証・封じ込め・人間の承認つきで取り込む**受け口**、の3点で構成される、運営者が管理していない任意のAIを自アプリの生成エンジンとして機能させるための公開された足場。

```
[任意のAI (ChatGPT / Claude / …)]
   │ ① 仕様書を読む (GET /agent.md, llms.txt)
   │ ② ユーザーと会話し、成果物 (JSON) を作る
   │ ③ コード実行で圧縮・エンコード → リンクを出力
   ▼
https://app/import#v1.<base64url(deflate-raw(JSON))>
   │ ④ ユーザーがクリック (fragment はサーバに届かない)
   ▼
[/import 受け口]
   寛容デコード → スキーマ検証 → サンドボックス描画
   → 正データの並置プレビュー → 人間が「保存」
   ▼
[POST /api/import] サーバ再検証・再サニタイズ
   ▼
[アプリの正規パイプライン] 以後は普通のオブジェクトとして扱う
```

このパターンが成立する土台は、一つの観察にある。

> **AIチャットのコード実行環境は、ネットワークから遮断されているが、チューリング完全である。**

ChatGPT の code interpreter も Claude の分析ツールも、外部にPOSTはできない。しかし圧縮もエンコードも計算はできる。だから「計算結果をテキスト(リンク)として出力し、人間がクリックして運ぶ」— これが、どのAIでも絶対に通る唯一の統合経路になる。テキスト出力だけは、すべてのAIチャットが必ずできるからだ。

## 構成要素1: 仕様書 = SDK

外部AIには、SDKもnpmパッケージも渡せない。渡せるのは文書だけだ。だから**仕様書がSDKそのもの**になる。

* 安定URL (例: `/agent.md`) で plain text 配信する。「AIにこのURLを読ませてください」の一言で統合が完了する。
* `llms.txt` から参照して発見可能にし、トップページには人間向けの「プロンプトをコピー」導線を置く。
* 内容は4点: 成果物の要件 / ペイロードのJSONスキーマと実例 / エンコード手順のコードスニペット / 出力形式(リンクと生JSONの併記)。
* **数値や列挙は、アプリの型定義から動的に生成する。** サイズ上限や enum が仕様書とコードで乖離した瞬間、外部AIは静かに壊れ始める。単一ソース化は、外部ハーネスでは礼儀ではなく生存条件だ。

そして仕様書には明確な受け入れテストがある: **「AIに仕様書だけを渡して、完走できるか」**。詳細は後述するが、このロールプレイテストは仕様の穴だけでなく、アプリ側の統合バグまであぶり出してくれる。

## 構成要素2: リンク = API呼び出し

搬送路の設計が、外部ハーネスの技術的な核心だ。

```
https://<origin>/import#v1.<base64url(deflate-raw(JSON))>
```

* **クエリパラメータではなくフラグメント (`#`) に載せる。** フラグメントはサーバに送信されない。つまりリンクを踏んだ時点では、成果物(多くの場合、氏名や連絡先などの個人情報を含む)がサーバにもアクセスログにも一切残らない。\*\*「保存を押すまで、データはブラウザの中にしかない」\*\*という強いプライバシー保証が、設計から自然に出てくる。副次効果として、プロキシやホスティングのURL長制限(8〜14KB程度)も無関係になる。
* **バージョンプレフィクス (`v1.`) を必ず入れる。** 仕様は必ず進化する。
* **圧縮する。** percent-encoding は日本語1文字を9バイトに膨らませるが、deflate + base64url なら現実的なサイズに収まる。コード実行できるAIなら、圧縮は確実にこなせる。

## 構成要素3: 寛容な受け口

送り手を制御できないことの対価は、受け側で払う。**Postelの法則をAI統合に適用する**: 送るものには厳格に(仕様書)、受け取るものには寛容に(デコーダ)。

* AIは deflate-raw と zlib と gzip を*必ず*取り違える。Python の `zlib.compress` は raw deflate ではない — という罠は、仕様書に書いても踏まれる。受け口は deflate-raw → zlib → gzip → 無圧縮base64 → percent-encoded JSON → 生JSON の順に全部試す。
* **寛容さには上限が要る。** フラグメント長の上限と、展開サイズ上限つきのストリーム解凍(数KBが数GBに膨らむ解凍爆弾への備え)を必ず入れる。
* **フォールバック経路を併設する。** 仕様書はAIに「リンクと生JSONの両方」を出力させ、受け口には貼り付け欄を置く。リンクがチャットUIやコピペで壊れても、成果物は届く。

## 構成要素4: 信頼境界と人間の承認

外部AIの成果物は**命令ではなくデータ**であり、一切信頼しない。

* **事実と表現を分離する。** 成果物のうち「アプリ内で正となる構造化データ」(氏名、リンク、設定値など)と「見た目・プレゼンテーション」(HTML/CSSなど)を別フィールドで運ぶ。アプリの一覧表示・共有ページ・メタデータは事実からのみ描画し、AIの書いた表現層は表示専用に閉じ込める。「正はどのデータか」を成果物の構造で強制する。
* **クライアント検証はUX、サーバ再検証が境界。** 受け口ページの検証は使い勝手のためで、保存APIがもう一度ゼロから検証・サニタイズする。
* **描画を封じ込める。** 任意のHTMLを描画するなら `sandbox=""` の iframe + CSP (`default-src 'none'`)。スクリプトは二重に死に、外部リソースのフェッチも遮断される — 悪意あるリンクを開いても、閲覧者のIPすら漏れない。
* **人間が最後のゲート。** プレビューは「見た目」と「保存される正データ」を並置し、外部生成であることを明示した上で、明示的な保存操作によってのみ永続化する。
* **取り込み後は一級市民にする。** 「インポート専用の型」を作らない。既存の正規パイプラインに合流させ、来歴だけ記録する。並行する第二のオブジェクト型を二重メンテする未来を避ける。

## 結び

llms.txt は「AIにwebを読ませる」ための約束事だった。外部ハーネスはその次 — **「AIにwebで作らせる」ための約束事**だ。アプリは人間向けのUIと並んで、AI向けのUI(仕様書とリンクの受け口)を持ち始める。

モデルを持たない側が、モデルの進歩を最速で取り込む方法は、良いハーネスを公開しておくことだ。
