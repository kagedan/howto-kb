## 🚀 AIエディター「Kiro」活用術：実録・RAGアプリの「中身」を別サイトへ移植＆LLM突然の死からの生還

### 1. はじめに：RAGの真価は「データの入れ替え」にあり

特定のドキュメントに基づいて回答するRAG（検索拡張生成）チャットボット。その最大の魅力は、\*「元データ（ベクトルDB）を差し替えれば、別の用途にすぐ転用できる」\*点にあります。

今回は、以前作成した「〇〇博物館」のFAQチャットボットを、架空のテーマパーク「ワンダーランド東京」用へとコンバートしました。しかし、その過程で\*「無料枠LLMの利用不可」や「ベクトルの次元不一致」\*といった数々の壁が立ちはだかりました。その解決の軌跡を記録します。

### 2. インシデント①：動いていたはずの「次元不一致」の謎

作業の冒頭、AI（Kiro）は致命的な不整合を指摘しました。

本来、次元が違えばエラーで止まるはずですが、Vercel上では「それっぽく」動いていた形跡がありました。これは\*「環境変数がローカルと本番で異なっていた」、あるいは「LLMが検索結果を無視して一般知識で回答していた」\*可能性が高いことが判明しました。  
今回の移植では、埋め込みモデルを384次元に統一し、RAGが「正しく」機能するよう根底から作り直しました。

### 3. インシデント②：LLMの「400エラー」と緊急オペ

データを入れ替え、いざテスト……という段階で、深刻なエラーが発生します。

* 💻 ターミナル（エラー発生）  
  HTTP 400: Model Mistral-7B-Instruct-v0.2 does not support chat completion

今まで使っていた無料枠のLLM（Mistral）が、Hugging FaceのAPI仕様変更により突然使えなくなっていたのです。

##### 解決策：AIエンジンの「即時差し替え」

Kiroは即座に代替モデルを探索。日本語対応に優れ、無料枠で安定して動作する 「Qwen/Qwen2.5-7B-Instruct」 への切り替えを提案し、.env.local やコードのデフォルト値を一気に修正しました。これにより、わずか数分でシステムを復旧させました。

### 4. 攻めの防御：「モデル比較ページ」の爆速実装

「またいつモデルが使えなくなるか分からない」。そんな不安を解消するため、今回新たに\*「モデル比較ページ (/llm-compare)」\*を実装しました。

* 同時推論: 同じ質問を、複数のLLMモデル（Qwen, Llama, Gemma等）に一斉に送信。
* 品質比較: どのモデルが最も自然な日本語で、RAGのコンテキストを活用できているかを目視で確認。
* 生存確認: どのAPIが生きているかを一目で判別。

これにより、特定のLLMに依存しない、\*「生存性の高いRAGアプリ」\*へと進化を遂げました。

このプロジェクトの処理フロー

```
ユーザー: 「営業時間を教えて」
         │
         ▼
┌─────────────────────────────────────────────┐
│    質問テキスト → 384次元のベクトルに変換     │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│ ② ベクトル検索 (embeddings.json)             │
│    質問ベクトルと全FAQベクトルのコサイン類似度 │
│    を計算して、似ているFAQを上位5件取得       │
│                                              │
│    → 「Q: 営業時間は？ A: 通常期は9:30〜...」 │
│    → 「Q: 再入場は？ A: 当日に限り可能...」   │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│ ③ LLM (Qwen2.5-7B-Instruct)                 │
│                                              │
│    プロンプト:                                │
│    「以下のコンテキストを参考にして答えて」    │
│    「コンテキスト: (②で取得したFAQ)」         │
│    「質問: 営業時間を教えて」                  │
│                                              │
│    → 自然な日本語で回答を生成                 │
└─────────────────────────────────────────────┘
```

つまり  
ベクトルDB（embeddings.json） → 「どのFAQが質問に関連するか」を探す検索エンジン  
LLM → 見つかったFAQを読んで、自然な文章で回答を作る「文章生成エンジン」  
LLMはベクトルDBそのものを受け取っているわけではなく、ベクトル検索で絞り込まれたFAQテキスト（上位数件）だけをプロンプトの一部として受け取っています。

LLMを変えたときに何が変わるか  
部分 LLM変更の影響  
① 埋め込み・ベクトル化 変わらない（別のモデル）  
② ベクトル検索・FAQ絞り込み 変わらない  
③ 回答の文章生成 ここだけ変わる  
同じFAQデータが渡されても、LLMによって：

日本語の自然さが違う  
要約の仕方が違う  
コンテキスト外の情報を混ぜるかどうかが違う  
そもそも動くか動かないかが違う  
だからモデル比較ページが役に立つわけです。同じ検索結果（コンテキスト）を渡して、LLMの回答品質だけを比較できます。
