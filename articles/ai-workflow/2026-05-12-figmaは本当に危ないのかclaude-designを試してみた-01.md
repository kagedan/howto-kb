## Claude Designとは

Anthropic は 2026年4月17日に Claude Design を発表しました。公式には、自然言語、スクリーンショット、ドキュメント、コードベース、デザインファイルをもとに、ビジュアル制作物を生成できるデザインプロダクトとして位置づけられています。同じ日に、Figma の株価が7%以上下落したという報道もありました。

私なりに Claude Design をひと言で表すと、こうです。

> 単なるAI画像生成ツールではなく、要件、スクリーンショット、デザインルール、コードベースを読み取って、UIのたたき台を作るツールです。

ただ、私が本当に知りたかったのは、**Claude Design は UIデザインとフロントエンド開発のどの工程を変えるのか**という点でした。

## 先に結論

4つの実務に近いシーンで試してみて、こう感じました。

* 短期的に Figma を置き換えるものではありません。
* 一番強いのは、「要件からUIの初稿へ」の工程をかなり前倒しできる点です。
* PM にとっては、要件ドキュメントを短時間でクリック可能なプロトタイプにできます。
* デザイナーにとっては、既存画面のリデザイン、デザインルールの適用、方向性の探索に向いています。
* フロントエンドエンジニアにとっては、コードベースを読ませることで、既存コンポーネント体系に近い新規ページの初稿を作れます。

## 試したこと

| 検証シーン | 入力したもの | 見たかったポイント |
| --- | --- | --- |
| 自然言語からプロトタイプ生成 | 業務要件の Prompt | PM がすばやく共有用のたたき台を作れるか |
| 既存画面のリデザイン | 旧画面のスクリーンショット + UI/UXルール | デザインルールに従えるか |
| デザイン稿がなくコードだけある場合 | ローカルのコードベース | 既存コンポーネントや見た目を再利用できるか |
| Figma のルールを読み取る | Figma UI Kit | デザインファイルからルールを抽出できるか |

私が見たかったのは、「きれいな画像が出るか」ではありません。出てきたものが、そのままチームの会話や制作プロセスに乗せられるかどうかです。

## シーン1：自然言語からクリックできるアプリプロトタイプを作る

### 検証したかったこと

最初は PM の日常に近いシーンです。要件が出た段階で、顧客や事業側と方向性を合わせるためのクリック可能なプロトタイプを先に作れるかを見ました。

### 入力と生成

今回は、日本での生活シーンを想定した「引っ越し手続き管理アプリ」を作る、という日本語の要件を入力しました。Prompt では、ホーム画面、タスク詳細、カテゴリ、進捗画面に加えて、タスクカードを押したとき、完了ボタンを押したとき、下部ナビゲーションを切り替えたときの挙動も指定しています。

![](https://static.zenn.studio/user-upload/079aea534d57-20260507.png)

*要件を入力すると、Claude Design が動くフロントエンドプロトタイプを生成し始めます。*

![](https://static.zenn.studio/user-upload/089dea026a3b-20260507.gif)  
![](https://static.zenn.studio/user-upload/25e244c5f4d3-20260507.gif)

*出力は静止画ではなく、クリックできるモバイル向けプロトタイプです。*

### 結果を見て感じたこと

このシーンでは、Claude Design の価値はかなり分かりやすかったです。これまで PM が新機能のイメージを伝えようとすると、ワイヤーフレームを描くか、長い PRD を書くことが多かったと思います。Claude Design を使うと、業務要件から 70〜80% くらいは議論に使える UI プロトタイプを直接作れます。

使いどころとしては、要件レビューの前がよさそうです。抽象的なアイデアを、まず「指摘できる形」にする。ただし、これは最終デザインではありません。視覚的な階層、コンポーネントの状態、例外ケースなどは、引き続きデザイナーが詰める必要があります。

## シーン2：UI/UXルールに沿って古い画面をリデザインする

### 検証したかったこと

2つ目は、よりデザイナーの実務に近いシーンです。古いプロジェクトでは、情報自体は残っているのに、見た目だけが時代に合わなくなっていることがあります。色が強い、線が重い、余白が少ない、といった初期のWeb画面のような状態です。

### 入力と生成

今回は、旧画面のスクリーンショットと Tailwind の設定ファイルを Claude Design に渡しました。新しい BtoB SaaS らしいトーンにリデザインしつつ、設定ファイルにある色、フォント、影のルールを厳密に使うように指示しています。

![](https://static.zenn.studio/user-upload/46db47317a5d-20260507.png)  
*旧画面には必要な情報は残っていますが、視覚的な階層やレイアウトはかなり重く見えます。*

![](https://static.zenn.studio/user-upload/9a7d0a6813a7-20260507.png)  
*旧画面のスクリーンショットと Tailwind 設定を添付し、Claude Design に視覚ルールを読ませます。*

### 結果を見て感じたこと

![](https://static.zenn.studio/user-upload/a4d6d9aeb1bf-20260507.png)

![](https://static.zenn.studio/user-upload/cf6500b3bdd6-20260507.png)

面白かったのは、Claude Design が単に画面を「きれいにする」だけではなかったことです。旧画面の情報構造を読み取ろうとしながら、カード、余白、色の使い方を変えて、画面全体を組み直していました。

既存画面のリデザインの初稿としてはかなり使えます。機械的な描き直しを減らせるからです。ただし、複雑な業務画面では、どの情報を優先するかの判断はまだ人間側に残ります。デザインシステムが整っているほど、出力は安定しやすいと感じました。

## シーン3：デザイン稿がなく、コードだけが信頼できる場合

### 検証したかったこと

3つ目は、フロントエンド開発でよくある状況です。完全な Figma ファイルがなかったり、デザイン稿がすでに古くなっていたりして、本当に信頼できる情報源がコードベースだけになっているケースです。

### 入力と生成

今回は `Attach codebase` でローカルリポジトリを Claude Design に読ませました。そのうえで、既存の UI スタイルに合わせて `app/pricing/page.tsx` の価格ページを追加し、既存の `Button`、`Card`、`Navbar`、Mantine コンポーネントを再利用するように依頼しました。

![](https://static.zenn.studio/user-upload/3d956f175f92-20260507.png)  
*新しいデザインシステムを作るのではなく、既存プロジェクトの見た目に寄せることを目的にしました。*

![](https://static.zenn.studio/user-upload/77bff913fa83-20260507.png)  
`Attach codebase`\_ でローカルリポジトリを追加し、Claude Design にコンポーネントとページの雰囲気を読ませます。\_

### 結果を見て感じたこと

![](https://static.zenn.studio/user-upload/28e82248e9e9-20260507.png)  
Claude Design にコードベースを読ませると、少なくとも既存コンポーネントの名前や使い方はある程度理解してくれます。その結果、元のプロジェクトに近い方向性の新規ページを生成できます。最終コードとしてそのまま使うものではありませんが、「空白から始める」状態を、「方向性の合っている下書きから始める」状態に変えられます。

ただし、ここには明確な注意点もあります。コードベースに顧客情報、内部API、秘密鍵、未公開の業務ロジックが含まれている場合は、アップロード前に必ず企業のデータポリシーを確認する必要があります。

## シーン4：Figma UI Kitを読み取り、ルールに沿って新規ページを作る

### 検証したかったこと

4つ目は、デザインシステム寄りのシーンです。Claude Design に Hotel Reservation UI Kit を渡し、既存ページをそのままコピーするのではなく、まず Figma ファイルから視覚ルールを抽出し、そのルールに沿って新しいホテル一覧ページを作れるかを見ました。

ここで見たかったのは、最終的な見た目そのものよりも、**Claude Design が「デザイン資産を読み取る → ルールとして整理する → 新しい画面に適用する」という流れを実行できるか**です。

### 入力と生成

![](https://static.zenn.studio/user-upload/8cb33bf89ad2-20260507.png)  
*入力に使った Figma UI Kit。*

最初に、Figma ファイルを読み取らせ、チャット上で説明させるだけではなく、キャンバス上に Design System / Style Guide として可視化してもらいました。

![](https://static.zenn.studio/user-upload/0d78bd6e9df3-20260507.png)  
*Claude Design が Figma ファイルを読み取り、可視化された UI ルールを作り始めます。*

![](https://static.zenn.studio/user-upload/50594727d511-20260507.png)  
*生成された Design System。色、タイポグラフィ、余白、コンポーネント、カードのルールが視覚的に整理されています。*

次に、抽出した UI ルールをもとに、新しいホテル一覧ページを生成するように依頼しました。このステップでは、単に元の画面をなぞるだけではなく、デザイン言語を理解して別の画面に展開できるかが見えます。

![](https://static.zenn.studio/user-upload/131717958bd5-20260507.png)  
*前のステップで抽出した UI ルールをもとに、新しいページ生成を続けます。*

### 結果を見て感じたこと

![](https://static.zenn.studio/user-upload/43d9892c8a34-20260507.png)  
*最終的に生成されたホテル一覧ページ。*

このシーンから分かったのは、Claude Design は「見たものをそのまま描き直す」だけではないということです。既存のデザイン資産から再利用可能なルールを抽出し、それを別の画面に移すことも試みています。

もちろん、これをそのまま組織の Design System の正本にできるわけではありません。コンポーネント名、token の階層、レスポンシブルール、各状態の定義などは、デザイナーとフロントエンドエンジニアが一緒に確認する必要があります。それでも、「Figma 資産から新規ページのたたき台へ」つなぐ橋渡しとしては、すでにかなり価値があります。

## 使ってみて感じたこと

今回の検証を通じて、Claude Design の価値は「どれだけきれいに描けるか」よりも、次の3点にあると感じました。

1. **抽象的な要件を、議論できる形に変える**：粗くてもクリック可能なプロトタイプは、10ページの要件書より早く問題点を見つけられることがあります。
2. **機械的な描き直しを減らす**：既存画面のリデザイン、ルールの適用、方向性の探索では、まず AI に初稿を作らせるだけでもかなり楽になります。
3. **コードベースをデザイン入力に含められる**：実装現実から離れた「きれいなだけの画面」ではなく、既存プロジェクトに近い初稿を作れる可能性があります。

現時点で気になった制約もあります。

1. 利用枠の消費はかなり速いです。連続していくつか試すと、月20ドルの Pro プランではすぐ厳しくなります。
2. 最後の20%は、やはり人間が詰める必要があります。視覚的な細部、コンポーネントの状態、例外ケース、文言、アクセシビリティなどです。
3. スクリーンショット、Figma ファイル、コードベースをアップロードする前に、企業やチームのデータ境界は必ず確認する必要があります。

## まとめ

Claude Design は Figma を置き換えるのでしょうか。短期的には、そうは思いません。

Figma は、細かい調整、共同編集、コンポーネントライブラリ管理、デザインシステム運用において、今でも非常に強いです。成熟したチームにとって、Figma は引き続きデザイン資産の中心的な作業場であり続けるはずです。

ただし、Claude Design が本当に変えようとしているのは、ここです。**UIの初稿は、必ずしもデザインツールから始まらなくなる。**

これまでは、PM が要件を書き、デザイナーがワイヤーフレームやビジュアルデザインを作り、フロントエンドエンジニアがデザイン稿を待つ、という流れが一般的でした。これからは、要件、スクリーンショット、ワイヤーフレーム、UIルールを入力すると、Claude Design が先にクリック可能な UI のたたき台を出し、そこから PM、デザイナー、フロントエンドエンジニアが一緒に詰めていく流れも増えていきそうです。

Figma が本当に警戒すべきなのは、キャンバスそのものが奪われることではありません。「0から1の初稿」を作る工程が、先に別の場所へ移っていくことだと思います。
