## なぜ今、SaaSのコストが問題になっているのか

ここ数年、SaaS費用が企業のIT予算を圧迫しはじめている。

Slackのシート、Jiraのシート、ServiceNowのシート、Zendeskのシート——使う人数に比例して膨らむ費用構造の中で、「同じことをAIにやらせたらどうなるか」という問いへの答えが、2026年になってやっと出てきた。

[Deloitteの2026年 Technology Industry Predictions](https://www.deloitte.com/us/en/insights/industry/technology/technology-media-and-telecom-predictions/2026/saas-ai-agents.html)では、**AIエージェントがITサービスリクエストの80%以上を自動解決し、ITSMライセンスコストを最大50%削減した企業事例が報告されている**。

一方、[Ardas IT の SaaS 2026 トレンドレポート](https://ardas-it.com/saas-2026-trends-from-ai-experiments-to-production-ready-platforms)によると、SaaSは現在「AIを後付けした旧来型」と「AI-Nativeな新世代」に2極分化しており、資金と市場シェアが後者に集中しはじめている。

これは単なる「AIすごいですよね」の話じゃない。  
**予算の使い方を今すぐ見直す必要がある、という話だ。**

## Deloitte調査が示した「50%削減」の実態

削減の内訳をもう少し詳しく見ておこう。

Deloitteが観測した成功企業の共通点は以下の通り：

| カテゴリ | 変更前 | 変更後 |
| --- | --- | --- |
| ITSMチケット処理 | 人間が全件対応 | AIが80%以上を自動解決 |
| ライセンスシート数 | ピーク人数分を契約 | 常時使用者+バッファのみ |
| 課金モデル | 座席数ベース | 使用量×成果ベースに移行 |
| 解決までの時間 | 平均8時間〜数日 | 即時〜2時間 |
| コスト | ベースライン | 最大50%削減 |

**重要なのは、単に「AIで自動化した」だけでは達成できない点だ。** 成功企業は「どのリクエストをAIに任せ、どこで人間が介入するか」の設計を先に固めている。

> **💡 Tips**: Deloitteが観測した成功パターンは「全部AI化」ではなく「80%自動化＋20%人間エスカレーション」。完全自動化を目指すと品質が落ちて逆効果になる。

さらにDeloitteは「**2026年中に企業の50%以上がDX予算の過半をAI自動化に投下する**」とも予測しており、先行する企業と後手に回る企業でコスト競争力の差が拡大するフェーズに入っている。

## どうやってコストが減るのか——メカニズムを図解

具体的にどんなリクエストが「定型80%」に入るかというと：

* パスワードリセット・アカウントロック解除
* ソフトウェアインストールの権限申請
* VPN・Wi-Fi接続トラブルの初期診断
* 入退社に伴うアカウント作成・削除
* 定期レポートの作成・配信
* ステータス確認・進捗問い合わせへの返答

**これらを人間が1件ずつ処理していたコストが、AIエージェントなら限りなくゼロに近づく。**  
その分、シートライセンス数を減らすか、より付加価値の高い業務に人員を振り向けられる。

## シートライセンスからの移行が加速している理由

「座席数課金」という従来のSaaSビジネスモデルが揺らいでいる根本的な理由がある。

従来のモデルは「ソフトウェアを使う人数 × 月額料金」だった。これは**ソフトウェアを使うのが人間であることを前提にした設計**だ。

AIエージェントが台頭すると、前提が崩れる。

* Slackのチャンネルに返答するのがボットになる
* Jiraのチケットを処理するのがエージェントになる
* ヘルプデスクに問い合わせるのではなく、AIが先読みして解決する

「誰が使っているか」ではなく「どれだけ価値を生んだか」に課金の軸が移ると、旧来型のシートライセンスモデルは成立しにくくなる。

実際、[Ardas ITのレポート](https://ardas-it.com/saas-2026-trends-from-ai-experiments-to-production-ready-platforms)によると、先進企業では\*\*「使用量×成果ベース」のハイブリッド課金\*\*への移行交渉を積極的に進めており、ベンダー側も対応を余儀なくされつつある。

**日本企業への影響**: 日本のSaaS契約は「年間一括払い・更新前相談不可」の慣行が残っている。更新タイミングを事前に把握し、AIエージェント導入の実績データを手にした状態で交渉テーブルに臨むことが重要。

## ステップ1: 今日やること——SaaS棚卸しとターゲット特定


棚卸したら、以下の基準で「AIエージェントでコストを削減できる候補」を絞り込む：

| チェック項目 | YesならAI化候補 |
| --- | --- |
| ✅ リクエストの種類が10パターン以内に集約できるか | ルール化しやすい |
| ✅ 対応手順がドキュメント化されているか | AIに教えやすい |
| ✅ 間違えても致命的でない業務か | 安全に試せる |
| ❌ 法的責任が伴う判断が含まれるか | 人間が必要 |
| ❌ 感情的なエスカレーション対応か | AIは苦手 |

**最初のターゲットは「よくある問い合わせTop5」を処理しているツールにしぼる。** それだけで全体の30〜40%をカバーできる場合が多い。

## ステップ2: 今週やること——最初のAIエージェントを立ち上げる

日本企業でよく使われるツール別に、最初のAIエージェント設定例を示す。

### パターンA: kintone + Claude API（中小企業向け）

webhook\_handler.js

```
// kintoneのWebhookでリクエストを受け取り、Claudeで自動分類・返答
const handleKintoneWebhook = async (event) => {
  const requestText = event.record.request_body.value;

  // Claude APIで意図を分類
  const response = await anthropic.messages.create({
    model: "claude-opus-4-6",
    max_tokens: 500,
    messages: [{
      role: "user",
      content: `以下のITリクエストを分類し、自動解決できるか判断してください。

リクエスト: ${requestText}

分類ルール:
- パスワードリセット → auto_resolve
- ソフトウェアインストール申請 → approval_needed
- 障害報告 → escalate
- その他 → human_review

JSON形式で返してください: {"type": "...", "suggested_action": "...", "draft_reply": "..."}`
    }]
  });

  const result = JSON.parse(response.content[0].text);

  // auto_resolveの場合は即時返答
  if (result.type === 'auto_resolve') {
    await kintoneAPI.updateRecord(event.record.id, {
      status: '自動解決',
      reply: result.draft_reply
    });
  }
};
```

### パターンB: Slack + Claude（スタートアップ・IT企業向け）

slack\_bot.py

```
# Slack Botで問い合わせを受け取り、Claudeで初期対応
from anthropic import Anthropic
from slack_bolt import App
import os

app = App(token=os.environ["SLACK_BOT_TOKEN"])
client = Anthropic()

@app.event("app_mention")
def handle_it_request(event, say):
    user_message = event["text"]

    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1000,
        system="""あなたはIT部門のサポートエージェントです。
社内のIT問い合わせに対して、以下の手順で対応してください:
1. 問題を理解し、解決策を提案する
2. 自力解決できる場合は手順を案内する
3. 人間の対応が必要な場合は #it-support チャンネルへの誘導を行う
返答は日本語で、簡潔に。""",
        messages=[{"role": "user", "content": user_message}]
    )

    say(response.content[0].text)
```

### パターンC: ServiceNow + Claude API（大企業向け）

ServiceNowにはVirtual Agentという機能があるが、**Claude APIをカスタム接続すると精度が大幅に上がる**。

servicenow\_agent.py

```
# ServiceNow REST APIを使ってチケット情報を取得し、Claudeで処理
import requests
from anthropic import Anthropic
import os

def process_servicenow_ticket(ticket_id: str) -> dict:
    # ServiceNowからチケット取得
    snow_response = requests.get(
        f"{os.environ['SERVICENOW_URL']}/api/now/table/incident/{ticket_id}",
        auth=(os.environ['SNOW_USER'], os.environ['SNOW_PASS']),
        headers={"Accept": "application/json"}
    )
    ticket = snow_response.json()["result"]

    # Claudeで解決策を生成
    client = Anthropic()
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=2000,
        system="社内ITサポートの専門家として、インシデントの解決策を提案してください。",
        messages=[{
            "role": "user",
            "content": f"""
インシデント情報:
- タイトル: {ticket['short_description']}
- 詳細: {ticket['description']}
- カテゴリ: {ticket['category']}
- 優先度: {ticket['priority']}

自動解決可能か判断し、可能なら解決手順を、不可能なら担当者への引き継ぎ情報を生成してください。
            """
        }]
    )

    return {
        "ticket_id": ticket_id,
        "ai_suggestion": response.content[0].text,
        "can_auto_resolve": "自動解決可能" in response.content[0].text
    }
```

**最初の1〜2週間は必ず人間がAIの判断結果をレビューする体制を取ること。** 「AIが言ったから」で重要な設定を変更する事故が起きやすいのがこの時期。

## 失敗しないための「オーケストレーション設計」

現場からの重要な教訓が一つある。

**「AIを入れたのに全然うまく動かない」という失敗の9割は、オーケストレーション設計の不備が原因。**

[Ardas ITのレポート](https://ardas-it.com/saas-2026-trends-from-ai-experiments-to-production-ready-platforms)でも「AIだけでは動かない、オーケストレーション設計が必須」という教訓が共有されている。

オーケストレーション設計とは、簡単に言うと\*\*「誰が（何が）どの判断をして、うまくいかなかったときにどこへ流すか」を事前に設計すること\*\*だ。

この設計を**先に**作らずにAIを動かすと、AIが間違った判断をしたときに誰もフォローできなくなる。

**やってはいけないこと**: 「とりあえずAIに全部任せて様子を見る」。信頼度スコアの閾値設定と人間へのエスカレーションルートを必ず事前に定義すること。
