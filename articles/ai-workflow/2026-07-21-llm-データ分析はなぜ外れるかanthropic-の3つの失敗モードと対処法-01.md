## Takeaways

* AI を使ったデータ分析の問題は「コード生成」の精度ではなく、**質問を正しいデータに結びつける「文脈」の問題**です。
* 失敗モードは3つあります——**曖昧性**（どの定義？）・**鮮度**（ドキュメントが腐る）・**検索失敗**（正解はあるのに見つからない）。
* この3モードに**4層のスタック**（データ基盤・情報源・skills・検証）で対処することで、精度の底上げができます（Anthropic で約95%）。

---

## 動くのに、間違う

LLM に分析を頼んだことのある人なら、覚えがあると思います。SQL は綺麗に書けている。実行も通る。数字も返ってきます。恐ろしいのは、間違った数字ももっともらしい見た目で返ってくることです。

コーディングでは、あんなに頼れる相棒なのに。分析になった途端、急に信用できなくなる。この落差はどこから来るのでしょうか。

この「なぜ」を、私が読んだ中で一番腑に落ちる形で分解していたのが、Anthropic のデータサイエンスチームが書いた記事です。

<https://claude.com/blog/how-anthropic-enables-self-service-data-analytics-with-claude>

Anthropic 社内では、業務分析クエリの 95% を Claude が自動化し、精度は集計で約 95% だそうです。**warehouse に agent を繋いで走らせるだけでは「精度が出ている錯覚（false sense of precision）」を生む**、と最初に釘を刺した上で、間違いを 3 つの失敗モードに分解し、効かなかった施策と未解決の穴まで公開しています。

私自身、Data Analyst として AI 活用に苦戦したことがある側です。この記事は自分の現場での経験から共感できる部分が多かったため紹介したいと考えました。以下、自分の理解を織り交ぜながら、できるだけ忠実に原文の骨格（問題提起 → 解決策 → 検証 → 限界）を保ったまま、同じように苦戦している人向けに読み直します。

## データはソフトウェアではない

まず前提の問いから。同じモデルなのに、なぜコーディングでは信頼できて、分析では信頼できないのか。原文はこれを「データはソフトウェアではない」というふうに表現しています。

|  | コーディング | 分析 |
| --- | --- | --- |
| 解空間 | 開かれている。複数の正解があり、モデルの創造性が報われる | 正解は 1 つ。正しいソースも 1 つ |
| ガードレール | 型・テスト・コンパイラが即座に間違いを教える | なし。間違った数字は正しい数字と同じように見える |
| 検証 | 決定的。通るか、落ちるか | 正しさを決定的に証明する方法がない |

コードは間違えれば落ちます。分析は間違えてもそれらしきサインはないことがほとんどです。それっぽい数字が静かに返ってくるだけです。

そして原文の核心はこの一文です。問題は「**ユーザーの質問を、データモデルの中の specific で up-to-date なデータフィールドにマッピングし、その正しい扱い方を知ること**」。それさえできれば、SQL を書く作業自体はとても簡単だと。

つまり、私たちが詰まっているのは SQL 生成力ではありません。プロンプトで SQL の書き方をいくら指導しても直らないのはそのためです。直すべき場所が違うからです。

## 間違いは 3 つの失敗モードに分解できる

不正確な回答のほぼすべてが次の 3 つの失敗モードに帰着します。

| # | 失敗モード | 何が起きるか |
| --- | --- | --- |
| 1 | **概念↔実体の曖昧性** | 「アクティブユーザー数は？」——何をもって active？ 不正ユーザーは含む？ 期間は？ データモデルには候補が何百もあり、agent は選べない |
| 2 | **データの鮮度** | スキーマも定義も変わり続けるのに、ドキュメントと agent の知識が置き去りになる。微妙に間違った答えが返り始める |
| 3 | **検索失敗** | 正解のフィールドは存在し、注釈も付いている。でも探索空間が広すぎて、agent が辿り着けない |

一番深い曖昧性の例を、原文の図から借りて描き直します。「先月のアクティブユーザー数は？」という質問の答えは 1 つのファクトテーブルにあるのに、どの dimension を基準に取るかで「active」の定義が割れます。

この 3 つが**どれも技術的には正しい**です。SQL は 3 本とも通り、3 つの違う数字が返り、どれも間違いだとは言えません。

これは私の現場の記憶とも一致します。コンサルにいた頃、同じ会社の中でも「アクティブユーザー」の定義が部門で割れていました。営業は 3 日〜1 週間の短いスパンで見たい——いま動いている客に当たるためです。マーケは 1 ヶ月の広い括りで見たい——施策の効き方を追うためです。**どちらも業務上正しい**。人間同士なら「営業さんの言うアクティブね」と会話の文脈で補正されますが、agent はこのビジネスコンテキストを明示的に教えない限り、知りようがありません。

自分の環境で AI 分析が外れたとき、「モデルが馬鹿だ」で片付ける前に、この 3 つの失敗モードを理解し、深掘ることで対策の解像度が変わります。

## 対策の全体像：どの層が、どの失敗モードを潰すか

Anthropic の解決策は 4 層のスタックとなっていて、どの層がどの失敗モードに効くかを表にすると、こうなります。

| 層 | 曖昧性 | 鮮度 | 検索失敗 | 一言で |
| --- | --- | --- | --- | --- |
| データ基盤 | ✓ | ✓ |  | canonical dataset に絞り、似た重複を廃止（「revenue」候補を 40→1） |
| 情報源（sources of truth） | ✓ |  |  | semantic layer を最優先。誰が聞いても同じ数字が返る |
| skills |  | ✓ | ✓ | 数百万フィールドの warehouse を数十ファイルに絞ってから当たる |
| 検証（validation） | ✓ | ✓ | ✓ | どの失敗モードがまだ漏れているかを検出する |

## 層1 データ基盤：データそのものを片付ける（曖昧性・鮮度）

この層が触るのは**データそのもの**です。例えば、「今月の revenue は？」ひとつでも、社内には計算元が散らばっています。

* 生取引の `raw_transactions`
* 集計済みの `finance.monthly_revenue`
* 各アナリストが過去に書いた `SUM(amount)` が何十通り

税込・税抜・返金前後で数字は食い違い、agent はどれが正解か決められません。

誤解しやすいのは「1 つにする」の意味です。**税抜と税込のように、正当に違うものは消しません。** 消すのは、誰も管理せず各自が計算した**偶然の重複**だけです。正当な差は、canonical な源を 1 つ（例：税抜）決めて、そこから**機械的に派生**させます——税込は「canonical ＋ tax」の派生であって、競合する別定義ではありません。ソースが 1 つなら、派生がいくつあっても互いに矛盾しません。こうして計算元が 1 系統に畳まれれば、どの revenue かという曖昧性も、どれが最新かという鮮度の迷いも、はじめから生まれません。

## 層2 情報源：どこから引くかを決める（曖昧性）

層1 がデータそのものを片付けるのに対し、層2 が決めるのは **agent が答えを「どこから引くか」** です。データを 1 個に畳んでも、agent にどこを見ろと教えなければ、また迷うからです。

最優先は **semantic layer**——コンパイル済みの指標定義です。「revenue を出して」に対し、関数を呼ぶように 1 つの数字が返り、社内のどのツールで聞いても同じ値になります。agent は skill の指示で、まずここを引くよう構造的に縛られています。

先ほどの「アクティブユーザー」もここで捌けます。営業は 3 日、マーケは 30 日——どちらも正当でした。semantic layer はこれを `active_3d` `active_30d` のように**名前付きの指標として両方定義**します。agent は「どちらの active か」を曖昧なまま推測せず、名指しで呼びます。層1 の revenue と同じで、用途を消すのではなく、定義を 1 箇所に集めて呼べるようにするわけです。

定義とコンパイル後の SQL を見る（擬似コード）

定義はこう置いておきます。

```
# semantic layer の定義（1 箇所で version 管理）
metric active_3d:
  measure: count_distinct(user_id)
  source:  fact_events
  filter:  event_date >= today - interval '3 days'     # 直近3日

metric active_30d:
  measure: count_distinct(user_id)
  source:  fact_events
  filter:  event_date >= today - interval '30 days'    # 違いは期間だけ
```

agent が `active_3d` を名指しすると、コンパイルされて走るのはこの SQL です。

```
SELECT COUNT(DISTINCT user_id)
FROM fact_events
WHERE event_date >= CURRENT_DATE - INTERVAL '3 days'
```

数字はここで初めて計算されます——保存された値を引くのではありません。

semantic layer で足りないときのため、参照先には信頼度の序列があります。上から、semantic layer → lineage（変換の系譜）→ query corpus（過去の SQL 資産）→ business context（社内文書・ロードマップ・組織構造）。

ここで原文が強調したのは、**信頼度の順位と重要度の順位は別物だ**という点です。business context の信頼度が最下位にもかかわらず**重要度は最上位クラス**——ビジネスを理解しない agent は「聞かれたことには答えるが、意図には答えない」からです。先ほどの営業とマーケがまさにこれです。`active_3d` と `active_30d` を両方用意しても、「営業が active と言えば 3 日のほう」という対応づけは、データモデルの中ではなく組織の中にあります。教えない限り agent には見えません。低信頼を低重要と取り違えたチームが、ここを飛ばしてしまいます。原文いわく「私たちが一番長く過小評価していた層」です。

## 層3 skills：一番効いたレバー（検索失敗・鮮度）

**skills なしでは、評価セットでの正答率は 21% を超えなかった。skills を足すと 95% 超、ドメインによっては 99% 前後**。

同じモデルで、skill によって増えたのは知能ではなく、構造です。

skill は 2 種類をペアで作ります。役割の軸は単純で、\*\*片方は「どこを見るか」、もう片方は「どう進めるか」\*\*です。

* **knowledge skill＝「どこを見るか」。** 質問を受けると「その話ならこの参照ファイル群を見ろ」と案内する役です。「まず semantic layer、無ければこのドメインの約 30 ファイル」。数百万フィールドの warehouse を、クエリを書く前に数十ファイルへ絞り込む——これが検索失敗への対処です。
* **runbook skill＝「どう進めるか」の手順書** 質問を明確化し、ソースを特定し、クエリを実行し、最後に別の agent に敵対的レビューさせる——この順番を毎回踏ませる。リテンション曲線やファネル分析といった定番の型も同梱し、毎回ゼロから考えさせません。

※ 参照ドキュメントのテンプレートが原文の中に公開されています。

skill は生きたデータモデルを説明する文書です——「revenue は税抜」「active はこのテーブル」。だからエンジニアが定義を変えたのに skill を直し忘れると、agent は古い前提のまま静かに間違えた回答を返すことになります。実際、Anthropic の検証では、放置すると **offline 精度は 1 ヶ月で 95% → 65% まで落ちました**。

直し方は、「こまめに更新しよう」ではなく、**更新し忘れを構造で不可能にする**ことです。skill のマークダウンを、**データ変換の SQL と同じリポジトリに置きます**。こうすると、`fct_revenue.sql`（revenue の定義）を変える PR を開いた時点で、それを説明する `revenue.md` が同じ変更の目の前に並びます。そのうえで CI にルールを一本足す——**レポート用モデルを触ったのに skill ファイルを 1 つも触っていない PR は、レビューで弾く**。skill のマークダウンを直さない限り、モデル変更はそもそも merge できません。

※ skill のテンプレートが原文の付録に公開されています。

## 層4 検証：測らないと、直せない（曖昧性・鮮度・検索失敗）

4 層目の validation は「どの失敗モードがまだ漏れているか」を検出する仕組みで、3 段構えになっています。

**offline eval**——質問と正解のペアを数十個持ち、システム変更のたびに通します。肝は、正解を「動かない標的」にすることです。質問に日付を決め打ちして、「2026 年 5 月の active は？→ 5,000」と Q&A ごと凍結します。これを「先月の active は？」のような相対的な問いにすると、月が変わるたびに正しい答えも動き、保存した 5,000 が正解でなくなる——eval が落ちても、「システムが後退したのか、ただ月が変わっただけか」を区別できません。日付で固定しておけば、落ちた＝システム側の後退、と一意に読めます。offline での正答率は 100% を目指しますが、これは「間違えていない証明」ではなく「明らかな穴が無いことの確認」です。

**ablation**——eval セットを固定したまま、構成要素を 1 つだけ抜き差しして前後の点数を比べます。要は、変数を 1 つに絞った A/B テストです。

たとえば「敵対的レビュー（すぐ下で触れます）は重い工程だが、元が取れているのか」。レビュー**以外は全部そのまま**にして、あり／なしで eval を 1 回ずつ回す。点数が上がれば採用、変わらなければ外します。ほかを一切動かしていないので、その差はまるごとレビューの効果に帰着できます。会議で長々と議論するより、1 回の実測で決着します。次節の「効かなかったこと」も、この手法が炙り出した成果です。

**online eval**——本番での 3 点セット。

* **敵対的レビュー**：最終回答を、別の agent が「本当にそのテーブルで合ってる？その期間で正しい？」と前提から攻撃します。精度は 6% 上がりますが、代わりにトークン 32% 増・レイテンシ 72% 増。
* **provenance footer**：すべての回答の末尾に、出所と鮮度を付けます——たとえば「Source: semantic layer ／ Freshness: 5-01 ／ Owner: finance」。答えそのものは正しくなりませんが、**受け取った人が「どれだけ信じてよいか」を判断できる**ようになります。
* **correction harvesting**：stakeholder の訂正（「それ違うテーブルだよ」）を定期 agent が拾い、参照ドキュメントへの 1 行修正 PR を自動で起こす。同じ訂正は offline eval にも足します。

直った参照ドキュメントは、次に同じ問いが来たとき agent が読む——だから同じ間違いは繰り返されません。訂正が一度きりで消えず、システムに残って次の回答を良くします。

## 彼らが「効かなかった」と言い切ったこと

| 試したこと | 結果 | 教訓 |
| --- | --- | --- |
| 指標定義を LLM に自動生成させる | 「消したかった曖昧さをそのまま encode した」定義ができ、人間が作った小さい層に eval で負けた | ドキュメントは Claude が書き、**定義は人間が持つ** |
| 過去 SQL 数千ファイルへの生アクセス | transcript で「読んでいる」ことまで確認して、精度の変化は **1 ポイント未満** | ボトルネックは access ではなく **structure** |
| ドキュメント磨き込みの重ね掛け | ある時点から 3 連続で悪化。「doc は長くなっていた、良くはなっていなかった」 | 足せば良くなる、は成り立たない |
| 敵対的レビュアーの廉価モデル化 | 精度の稼ぎを失い、大して速くもならなかった | 検証の質は削る場所ではない |

特に 2 行目は示唆が深いと感じます。過去の SQL 資産は直感的には宝の山のはずです。ところが agent に全部読ませても精度はほぼ動きませんでした。情報はそこにあり、agent は読んでいて、それでも使えませんでした。

これ以上詳しいことは割愛しますが、興味ある方は原文を参照してください。

## 未解決の穴：silent failure

答えが間違っているのに、もっともらしく見えて、誰も異議を唱えず、そのまま使われる失敗です。ここまでの検出網（eval・監視・訂正収集）は、突き詰めればすべて「誰かが気づく」ことに依存しています。誰も気づかない間違いだけは、すり抜けます。

緩和策はあります。provenance footer で出所を示す、経営向けの数字には人間のサインオフを必須にする、主要 KPI は公式ダッシュボードと毎日突き合わせる。それでも原文は正直に書いています——"we don't have a robust solution yet"。

冒頭でも書きましたが、間違った数字は正しい数字と同じ顔をしていて、しかも受け取る側は正しさそのものを検証できない——それがこの問題の出発点でした。だから\*\*最後の砦は「検証」ではなく「懐疑」\*\*です。footer の「生テーブル由来・鮮度不明」という表示は、検証できない読者にも「この数字は、上に転送する前に疑うべきだ」と判断させるためのものです。

## どこから始めるか

原文の締めはこうです。「**canonical datasets を少数、offline evals を数十個、knowledge skill を 1 つ。これでメリットの大半が取れる。この記事の残り全部は、それらを作った後に足したものだ**」と。

苦戦している側の言葉に翻訳すると、最初の一手はこの 3 つです。

1. **eval**：自分がよく受ける質問 20〜30 個を、質問と正解のペアにして固定する。正解は特定日付のスナップショットに pin する。
2. **canonical**：一番聞かれる指標の定義を 1 箇所に書き、紛らわしい重複テーブル・重複定義を潰す（少なくとも「使うな」と明記する）。
3. **skill**：担当ドメインの参照ドキュメントと、ルーターを書く。

そして「全部やるな」も原文の結論の一部です。何をどこまで作るかは状況しだいで、始める前に考えるべき 5 つの問いがあります。

1. 正しい答えは「今」どれだけ重要か——モデルの進化で無意味になるインフラを作り込んでいないか
2. ビジネスの複雑さは、これからどう変わっていくか
3. 出力の読み手はどれだけテクニカルか——間違いに自力で気づける読み手か
4. 精度向上にいくら払えるか——敵対的レビューの「+6% 精度 / +32% トークン / +72% レイテンシ」がその実例
5. アクセス制御と社内データプライバシーの許容度はどうか——agent を 1 つ作るか、スコープを切って複数作るか

## まとめ

* 精度が出ない正体は「コード生成力」でもモデルの知能でもなく、**質問を正しいデータに結びつける「文脈」の問題**でした。分析はコードと違って間違っても落ちず、誤った数字が正しい数字と同じ顔で返ります。しかも、**数字の正確性を検算できない**——だからプロンプトで SQL を教えても直らないわけです。
* その文脈の壊れ方が、3 つの**失敗モード**です——**曖昧性**（営業とマーケで「アクティブ」が違う）・**鮮度**（ドキュメントが数週間で嘘になる）・**検索失敗**（答えは在るのに辿り着けない）。まず自分の環境がどの失敗モードで詰まっているかを診断します。
* Anthropic はこの 3 モードを**4 層のスタック**（データ基盤・情報源・skills・検証）で潰し、精度を約 95% まで底上げしました。各層が担当モードを叩き、\*\*検証層だけが 3 モード全部を「検出」\*\*します（防ぐのではなく）。skill は 21%→95% を買う一方、放置すれば 1 ヶ月で 65% まで腐る——だから最小セット（**canonical 少数＋offline evals 数十＋薄い knowledge skill**）から始め、維持を仕組みにします。それでも silent failure は彼らも未解決のままです。
