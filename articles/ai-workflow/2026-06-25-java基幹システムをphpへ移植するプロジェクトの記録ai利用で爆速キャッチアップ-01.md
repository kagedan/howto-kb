## 1. はじめに

### 自分のポジション：「仕様を知らない移植担当者」

* **業務仕様・要件**：ベンダー（パッケージ提供元）が保持・管理
* **私の役割**：「旧システム（Java）と同じ動作をする PHP 版を作る」開発者

つまり、「何を作るべきか」はベンダーが定義した旧システムのコードに書いてある。私はそれを読んで、PHP で再現する。設計判断ではなく、**解読と再現**が仕事です。

これがこのプロジェクトの最大の特徴であり、最大の難しさでもありました。

### システムの概要

対象は、卸売・流通業向けの受発注・在庫・売上管理を担う **B2B基幹業務システム**です。受注から出荷・売上計上・帳票出力まで、商流のほぼ全域をカバーするオールインワンな業務パッケージで、稼働画面数は 100 を超えます。

旧システムは Java で構築されており、長い稼働実績を持ちます。今回、そのロジックをそのまま引き継いだ PHP 製の新システムを構築することになりました。言語選定の経緯は私の担当範囲外のため本稿では触れません。

### 移植方針：「直訳」を貫く

設計方針は最初から決まっていました——**「旧システムを正解として、忠実に移植（直訳）する」**。

仕様書より動いている旧システムのほうが信頼できる、という現場の現実があります。業務担当者が「前と同じように動く」と確認できることが最優先。コードをきれいにしたい衝動は封印し、旧実装を一次ソースとして扱いました。

## 2. 「仕様を知らない」まま移植する難しさ

### 2-1. コードが唯一の仕様書

業務ドメインに精通しているのはベンダー側です。私が参照できるのは：

* 旧システムの Java コード
* 断片的な設計書（必ずしも最新ではない）
* 業務担当者へのヒアリング

この状況で 100 画面超のシステムを移植するとき、**「このコードは何をしているのか」を自力で解読する**場面が繰り返し訪れます。

```
// こういうコードに出会う
if (numericConv(qty * unitPrice, 17, 0) != numericConv(billed, 17, 0)) {
    // ...何らかのチェックをしている...
}
```

`numericConv` とは何か。なぜ 17 桁なのか。ベンダーに聞けば分かるかもしれないが、毎回聞いていたらきりがない。

### 2-2. ここで AI が武器になった

このプロジェクトで最も効いた開発者の武器が、**AI（Claude）によるコード解読と新旧比較**です。

**① Java コードの意図を読み解く**

コンテキストを持たない Java コードをそのまま AI に貼り付け、「このコードは何をしているか、なぜこの実装なのか考察してほしい」と問うと、驚くほど的確な解釈が返ってきます。

```
入力: numericConv の Java実装コード

AI の回答:
「このメソッドは BigDecimal ではなく double の浮動小数点演算で丸めを行っています。
 "絶対値に eps を加算してから切り捨て" という実装は HALF_UP に近いですが、
 double の精度誤差のため x.x5 の値では HALF_UP と異なる結果になります。
 意図的に double の誤差を込みで動作させているようです。」
```

これを見て、PHP 側でも `round()` ではなく同一アルゴリズムを移植する必要があると確信できました。

**② 新旧コードの論理的等価性を検証する**

Java と PHP の実装が「同じ動作をするか」を確認するとき、目視での確認は限界があります。AI に両方のコードを渡して「このJavaとこのPHP、同じ処理をしていると言えるか、差異があれば指摘してほしい」と問うと、変数名や言語の構文差を超えてロジックの差を指摘してくれます。

```
入力: Java の otherSalesInsert() と PHP の otherSalesInsert() を並べて貼り付け

AI の回答:
「Java版では agencyCd をセットした後 null チェックがありますが、
 PHP版にはその分岐が見当たりません。
 空文字と null を別扱いしている可能性があります。」
 → 確認したところ実際に欠落していた処理でした
```

コードレビューで「旧システムの対応処理と並べて確認する」という作業を、AI が大幅に加速してくれます。

**③ 「なぜこう書いているのか分からないコード」の解釈**

システムには、「不要に見えるが実は意味がある」コードが埋まっています。コメントが残っていないことも多い。AI にそのコードのコンテキストを渡すと、「このパターンは〇〇の場合の防御コードではないか」という仮説を出してくれます。それをベースに業務担当者に的を絞った確認ができる——という使い方が特に有効でした。

## 3. 技術的な挑戦と対策

### 3-1. 浮動小数点の罠：旧実装を解読して初めて分かること

基幹システムで数値計算を扱う上で、**丸め誤差**は命取りです。在庫数量や金額が1円・1個でもズレれば、そのまま帳票の誤りになります。

旧システムでは `BigDecimal` を使っていると思い込んでいました。しかし AI の助けを借りてソースを読み込むと、**独自の浮動小数点丸めアルゴリズム**が使われていることが判明しました。

```
// Javaの旧実装（概念）
public static String numericConv(double value, int scale) {
    // 「絶対値に 0.5×10^-scale を足して切り捨て」という独自の実装
    // BigDecimal の HALF_UP/HALF_DOWN とは微妙に異なる挙動
    double factor = Math.pow(10, scale);
    double abs    = Math.abs(value);
    long truncated = (long)(abs * factor + 0.5);
    // ...
}
```

一見 HALF\_UP に見えますが、double の浮動小数点誤差が絡むと挙動が変わります。

```
// 実際に発見した差異
-1100.25 → 旧システム: -1100.2  ／ HALF_UP: -1100.3  ← ズレる
 100.25  → 旧システム:  100.2   ／ HALF_UP:  100.3   ← ズレる
 604.75  → 旧システム:  604.7   ／ HALF_UP:  604.8   ← ズレる
```

`number_format()`、`round()`、`bcscale()` のどれを使っても完全一致しません。解決策は、**旧実装のアルゴリズムをそのまま PHP に移植する**ことだけでした。

```
// PHPでの再現実装（概念）
function numericConvUpc(float $value, int $precision, int $scale): string {
    // Javaの独自floatアルゴリズムをPHPで再現
    $factor = pow(10, $scale);
    $abs    = abs($value);
    $result = floor($abs * $factor + 0.5 * pow(10, -($precision - $scale))) / $factor;
    return (string)($value < 0 ? -$result : $result);
}
```

**教訓**：「どうせ BigDecimal だろう」という先入観は危険。AI でアルゴリズムを解釈してから移植に着手する習慣が重要でした。

---

### 3-2. 型システムの違いへの向き合い方

Java はコンパイル時に型エラーを検出できます。PHP は動的型付けですが、PHP 8.x では次の組み合わせで相当な型安全性を確保できます。

| 対策 | 内容 |
| --- | --- |
| `declare(strict_types=1)` | 暗黙の型キャストを禁止 |
| 型宣言の徹底 | 引数・戻り値・プロパティに型を明記 |
| PHPStan Lv6〜8 | 静的解析でコンパイル時相当のチェック |
| IDE 支援 | PhpStorm による型推論・補完 |

Java の `null` 安全（Optional や `@NonNull` アノテーション）に慣れた身には、PHP の `null` の扱いは最初は怖く感じます。`??` 演算子や `null` チェックの書き方を統一するコーディング規約の整備が必要でした。

---

### 3-3. コード自動生成パイプライン：「md が正本」という運用ルールとの格闘

新システムには **Markdown 仕様書からコードを自動生成するパイプライン**が存在します。

```
仕様書(Markdown)
    ↓  md_to_excel.php
Excel仕様シート
    ↓  GEN.BAT (独自ジェネレータ)
PHP クラス（SQL / IPC / CLI panel）
```

このパイプラインに初めて直面したとき、最初に踏んだ罠がこれです。

```
❌ やってしまったこと: 生成後のPHPファイルを直接編集 → 再生成で消えた
✅ 正しい手順: sqlspec_md/*.md を修正 → 再生成
```

「md が正本であり、生成物を直接編集してはいけない」というルールは口頭で共有されていましたが、体験してみないと本当には分かりませんでした。

ただし慣れると、この仕組みは非常に強力です。カラム追加・WHERE 条件変更・バインド変数の順序変更がすべて Markdown の修正で完結し、PHP ファイルに手を触れる必要がない。移植作業では「SQL の仕様変更を旧システムから読み取って md に反映する」という作業が主体になります。

---

### 3-4. 「移植」と「改善」の境界線を守る

PHP で書き直す中で「ここはもっとスッキリ書けるのに」という衝動が何度も訪れます。

例えば、Java のコードに冗長に見える null チェックが連続している場面——PHP の `??` 演算子で1行にまとめられる。でも、それをやってはいけない。

理由は、**旧実装の「冗長さ」自体が仕様の一部かもしれない**からです。安易なリファクタリングは検証コストの増大に直結します。

ルールはシンプルに：**明示的に承認された差異以外は旧実装に忠実に書く**。

このルールを守るためにも AI による新旧比較が効きます。「リファクタリングしたくなった箇所」を AI に「このJavaとこのPHPは同じ動作をしているか」と確認させる習慣を持つことで、意図しない仕様変更を防げました。

## 4. 移植を通じて分かったこと

### 良かった点

**コンパイル→デプロイのサイクルがなくなった**

コードを書いてすぐブラウザで確認できる。「ちょっと試してみる」コストが下がり、移植の検証作業が格段に速くなりました。

**モダンな PHP の型サポートが思ったより使える**

「PHP は型がない言語」は過去の話です。PHP 8.1 の `enum`、8.2 の Readonly Properties など、Java エンジニアが使い慣れた概念が PHP でも自然に書けるようになっています。

### 苦労した点

**暗黙の仕様の掘り起こし**

「なぜこう書いているのか分からないコード」が旧システムに埋まっています。不要に見えるが実は特定のデータパターンへの対処だったり、コメントが当時の担当者にしか分からなかったり。AI の解釈支援はここで大きく効きましたが、最終的には業務担当者への確認が不可欠な場面も多くありました。

**「仕様はベンダー任せ」の難しさ**

何か挙動の判断に迷ったとき、「これが正しいか」を確認できる一次情報が旧システムのコードしかない。そのコードを正確に読み解く責任が、開発者である自分にすべて乗ってくる。それが移植担当者の本質的な苦しさだと感じています。

## 5. 開発者として感じたこと

このプロジェクトは設計ではなく「解読と再現」の仕事です。業務ロジックの正しさはベンダーが保証し、自分はそれを PHP で忠実に再現する。

一見地味に聞こえますが、実際には：

* **言語を超えてロジックの等価性を保証する**責任
* **暗黙知をコードから読み解く**力
* **「改善したい衝動」を抑えて直訳を貫く**自制心

という、純粋に技術的な能力が問われる仕事でした。

そして AI は、この「解読と検証」という作業において、単なる補助ツールを超えた存在になっています。Java の癖を読み解き、新旧の差分を指摘し、「なぜこうなっているのか」を仮説として出してくれる——。移植担当者にとっての AI は、コードを書くためのツールではなく、**旧システムを理解するための対話相手**として機能しています。

同じように「既存システムの忠実な移植」に取り組む誰かの参考になれば幸いです。
