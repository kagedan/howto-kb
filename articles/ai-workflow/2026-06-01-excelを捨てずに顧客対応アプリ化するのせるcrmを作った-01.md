## 一言でいうと

**今あるExcel・フォルダ・メールを、そのまま顧客対応アプリに変えるAIエージェント**です。  
CRMを新しく導入するのではなく、既存のExcel運用の「上にのせる」。だから零細企業でも今日から使えます。

## 解こうとした課題

零細企業の現場では、顧客情報がこう散らばっています。

* 顧客台帳、案件管理、納品履歴、請求が**複数のExcel**に分かれている
* 顧客とのやりとりは**Outlookメール**に埋もれている
* 関連資料（見積・納品書・提案書）は**フォルダ**に点在
* CRMを入れる費用・設定・データ移行・運用定着の余力はない

問題は「AIを使っていないこと」ではありません。**顧客対応のたびに人間が探し、照合し、書類やメールを手作業で作っていること**です。そして抜け漏れが起きる。

CRMを導入すれば解決する、という話でもありません。零細企業にとってCRM導入は重すぎる。だから「Excelをやめさせる」のではなく、**Excelの上にAIの顧客対応レイヤーをのせる**方が現実的だと考えました。

## プロダクト概要：のせるCRM

OneDrive/SharePoint上のExcel・書類と、Outlookの送受信履歴をMicrosoft Graphで横断的に読み込み、顧客ごとに自動で整理します。

* 顧客リストを自動生成（Excelの顧客名・会社名・メールアドレスで名寄せ）
* 顧客ごとに「Excel由来の案件・納品情報」「関連ファイル」「メール履歴」「対応候補」を1画面に統合
* 対応候補から、納品書PDF作成・Outlookメール下書き作成・Excelステータス更新まで支援
* すべて**人間が承認したものだけ**を実行（自動送信はしない）

## 画面構成（3カラム）

```
┌─────────────┬──────────────────────┬──────────────┐
│ 対応先リスト   │ 顧客コンテキスト          │ アクション      │
│ (左)         │ (中央)                 │ (右ドロワー)    │
│              │                       │              │
│ 検索/フィルタ  │ 概要(根拠つきAI要約)     │ 選択中の対応    │
│ 会社名        │ 対応候補                │ 確認ポイント    │
│ 担当者        │ 履歴(Excel/メール/資料)  │ 実行ステップ    │
│ ステータス     │                       │ 承認/スキップ/修正│
└─────────────┴──────────────────────┴──────────────┘
```

* 左：対応先を選ぶ
* 中央：その顧客の状況がExcel・メール・資料を横断して一目で分かる。しっくりこなければ `AI再読み込み` で選択中の相手だけ再生成
* 右：対応候補の `実行` を押すとドロワーが開き、小ステップごとに人間が承認/スキップ/修正してから `AIに実行依頼`

## アーキテクチャ

実線が今回実装した構成、点線が今後の拡張です。MVPでは **App Service + Microsoft Graph + Azure OpenAI** の3点を実装し、未サインインの審査員でも触れるようデモデータへのフォールバックを用意しました。

### 使用したMicrosoft技術

* **Azure App Service**：Webアプリ実行基盤（Node.js、`PORT`/`0.0.0.0` 対応）
* **Microsoft Graph**：OneDrive/SharePointのファイル・Excel読み取り、Outlookメール履歴取得
* **Azure OpenAI（gpt-4.1-mini）**：顧客概要・対応候補・メール下書きの生成

## Agentとしての設計

「全自動で送信するAI」ではなく、**人間の確認を構造に組み込んだAgent**にしました。

1. 顧客コンテキスト（Excel＋メール＋資料）をAzure OpenAIに渡し、概要・対応候補・確信度・理由を生成
2. 対応候補を選ぶと、必要な小ステップ（PDF作成 / メール下書き / Excel更新）に分解
3. 人間が各ステップを**承認 / スキップ / 修正**
4. 確認ポイントを全てチェックし、実行確認ポップアップを通過したものだけ実行
5. メールは**下書きまで**。送信は人間が行う

零細企業の現場で「AIが勝手に送った」が一番怖い。だから**止められる場所を増やす**設計にしています。

## デモシナリオ（動画の流れ）

1. アプリを開くと、OneDriveのExcelとOutlook履歴から対応先リストが自動生成されている
2. 顧客を選ぶと、中央にExcel由来の案件・納品、関連ファイル、メール履歴、対応候補が出る
3. 対応候補の `実行` で右ドロワーが開き、納品書PDF・メール下書き・Excel更新のステップが出る
4. 人間が承認・確認し、`AIに実行依頼`
5. 実行中インジケーターが進捗を表示し、完了後にPDFとメール下書きが生成される
6. Excelを捨てず、CRMを入れず、今ある資産を顧客対応アプリ化できた

## 実装で苦労した点

* **散らかったExcelの名寄せ**：顧客名・会社名・メールアドレスの揺れを正規化して、複数Excel＋メールを1顧客に束ねる
* **確定値はAIに任せない**：金額・明細などはAI要約ではなく構造化パーサーで別途抽出（数字の幻覚を防ぐ）
* **審査員が触れる状態**：delegated認証だと審査員のアカウントにはデモデータが無い。未サインイン時にデモデータへフォールバックする二段構えにした
* **依存パッケージゼロで動かす**：サーバーはNode標準モジュールのみ（`node_modules` なし）で構成し、導入・運用コストを最小化した

## 今後の拡張

* Power Automate / Graph による PDF・Outlook下書き・Excel更新の実書き込み
* Azure Functions をツール呼び出しとして接続
* Blob Storage / Azure AI Search による根拠提示と全文検索
* Foundry Agent Service へのツール接続

## まとめ

きれいなデータベースを前提にしないこと。現場に散らばったExcel・フォルダ・メールをそのまま出発点にすること。それが零細企業で本当に使えるAIの条件だと考えています。

**Excelを捨てなくていい。CRMを入れなくてもいい。今あるExcel・フォルダ・メールを、AIが顧客対応アプリに変える。** それが「のせるCRM」です。
