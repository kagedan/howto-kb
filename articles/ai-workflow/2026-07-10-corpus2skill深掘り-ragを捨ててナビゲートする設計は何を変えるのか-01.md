論文「Don't Retrieve, Navigate」（[arXiv:2604.14572](https://arxiv.org/abs/2604.14572)）と実装 [Corpus2Skill](https://github.com/dukesun99/Corpus2Skill) を、アーキテクチャとコスト構造の観点から深掘りします。概要レベルの解説は先に公開したサイトにまとめてあります。

<https://corpus2skill.businesshub.trueone.co.jp/>

## RAGの構造的問題は「精度」ではなく「視界」

RAG改善というと埋め込みモデルやリランカーの話になりがちですが、この論文の指摘はより根本的です。**LLMがコーパスの全体構造を観測できない**こと自体が問題だ、と。

検索ベースでは (1) 何がコーパスに存在しないかを判定できない (2) 複数トピックにまたがる証拠を計画的に集められない (3) 検索クエリの語彙がコーパスとずれると打つ手がない。エージェント型RAG（クエリ書き換え・反復検索）はこれを緩和しますが、「見えない棚から司書経由で借りる」構図は変わりません。

Corpus2Skillは**コーパスを可視の階層構造として与える**ことで、探索・撤退・横断をエージェントの推論対象にします。

## コンパイル: 埋め込みは「使い捨て」になる

パイプラインは Load → Embed → Cluster → Summarize → Label → Build の6段階。興味深いのは埋め込み（既定 Qwen3-Embedding-0.6B）の役割です。

埋め込みは**クラスタリングの足場としてコンパイル時のみ**使われ、サーブ時には登場しません。再帰的K-meansで分割し、類似クラスタを凝集マージで整理して深さのバランスを取り、各ノードにLLMが要約と短いラベルを付与します。`--p`（分岐率、既定10）と `--max-top`（トップレベル数、既定8）がツリー形状を決めるハイパーパラメータです。

成果物は `SKILL.md` / `INDEX.md` / `documents.json` のプレーンなファイル群で、`.claude/skills/` 規約に従うためAnthropic Skills APIにそのまま載ります。**「インデックス＝Markdownの木」なのでgit管理・差分レビュー・手動修正が効く**のは、ベクトルインデックスにない運用上の利点です。

## サーブ時の動き: ツリー探索としてのQA

エージェントはトップレベルスキルの説明を読み、有望なブランチへドリルダウンし、リーフで文書IDを特定して `get_document` で全文を取得します。重要なのは**バックトラックが一級の操作**であること。検索ベースでは「外れた検索結果」への対処はクエリ再発行しかありませんが、ナビゲーションでは「この枝は違った、隣の枝へ」が自然に表現でき、探索履歴自体が「どこを見て、どこに無かったか」の根拠になります。グラウンディング指標が改善するのはこの構造の帰結でしょう。

## コスト構造: 階層プロンプトはキャッシュと相性が良い

素朴に見るとナビゲーションは多段のLLM呼び出しでコスト増です（論文も「中程度のコスト増」と認めています）。しかし階層構造には決定的な性質があります。**探索の序盤で読むコンテンツ（ルート付近のスキル記述）は全クエリで共通**なのです。

`cache_control: ephemeral` を付与した実測では、WixQAベンチマークで1クエリ $0.172 → $0.089（**48%削減**）、入力トークンの約70%がキャッシュから1/10料金で供給されています。ツリーが深く共有プレフィックスが長いほど効くため、**コーパスが大きいほどキャッシュ効率が上がる**という良いスケーリング特性を持ちます。

## 汎化実験が示す適用境界

10サブセットでの検証結果は明快です。

| コーパス特性 | 勝者 |
| --- | --- |
| 単一ドメイン＋復元可能なトピック分類体系 | ナビゲーション |
| オープンドメインの事実断片プール | フラット検索 |
| 均質な表形式データ | フラット検索 |

クラスタリングが意味のある木を復元できないコーパスでは、階層は単なる遠回りになります。導入判断は「自社文書に自然な目次構造が潜在しているか」で行うべきで、この判断フローもサイトの[適用判断ページ](https://corpus2skill.businesshub.trueone.co.jp/page-08-enterprise-use.html)に図解でまとめました。
