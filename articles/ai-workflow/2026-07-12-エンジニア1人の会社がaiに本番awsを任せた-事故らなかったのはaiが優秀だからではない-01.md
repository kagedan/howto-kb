AIA株式会社の廣内です。当社はエンジニアが私1人という小さな会社で、実務の大半をAIエージェントたちと一緒に進めています。いわゆる「AIソロプレナー」に近い経営スタイルです。

先日、このような体制で、AIエージェントに本番AWSのインフラ構築を任せました。`terraform apply` も、コンテナイメージの `ECR push` も、実際に実行されています。それでも事故が起きなかったのは、AIが優秀だったからではありません。**止まるべきところで止まる仕組みを、あらかじめ用意していたからです。**

私たちは、「AIに業務を任せても事故が起きない」ための統治基盤である、SynthAgent / AgentLedgerを開発しています。

今回ご紹介するのは、公開用に用意したデモではありません。\*\*ある水曜日に、自社の本番AWSアカウントで実際に起きたことを、証跡とともにお見せします。\*\*うまくいったことだけでなく、AIが間違えた箇所や、それがどのような仕組みで止められたのかも含めてご紹介します。

## 朝、人間が決めたのは6つだけでした

その日の朝会で、経営者である私が決めたのは、6つの判断だけでした。

検証を続けるのか。どの実行方式を採用するのか。問い合わせをどの窓口に出すのか。15分ほどのやり取りを行い、その後の実務はAIエージェント側が進めました。

ただし、単に「あとは任せた」というわけではありません。**すべての実行に対して、承認単位ごとのゲートを設けています。**

AWSへの書き込みには、**レビュー済みの実行単位ごとに、人間による明示的な承認が必要です。**`terraform plan` の検算と `terraform apply` の実行についても、それぞれ別のゲートとして扱います。

この日の作業は、「AIエージェントの実行基盤が、閉域ネットワーク（VPC）内のデータベースへ安全に到達できるか」を確かめる実地検証でした。

つまり、本番アカウントにVPC・RDS・Redisを実際に構築し、動作を確認したうえで、その日のうちにすべて削除するという計画です。

## 計画は、別のAIが先に疑います

実行計画である `terraform plan` を作成したのはAIです。

最初の計画では、36個のAWSリソースを作成することになっていました。エージェント実行層の2個は後段でCLIから追加したため、当日に新規作成したリソースは合計38個です。意図しない削除の混入はありませんでした。

ここで、最初のルールが機能します。

**作成者本人は、自分の仕事をレビューできません。**

私たちは「作者≠レビュアー」という原則を、人間にもAIにも適用しています。この計画は別系統のAIであるコーディングエージェントへ渡され、独立した検算が行われました。

> 検証結果: **36 create / 0 update / 0 delete**（削除の混入なし）。開発用の公開セキュリティグループは**計画に不在**（jq検査=空配列）。DBは非公開設定。——独立レビューverdict: non-blocking

レビューは通過しましたが、同時に指摘も返ってきました。

「DBのサブネット構成が単一AZでは、作成時に失敗する可能性が高い」「監視用IAMの依存関係が計画から漏れている」という指摘です。

つまり、**AIが作成した初版の計画には、実際に2つの穴がありました。**

指摘を反映して計画を修正し、その内容を確認したうえで、初めて人間が「apply OK」を出しました。

## ガードレールが、本当に止めました

構築の途中で、ガードレールの働きを象徴する出来事がありました。

セキュリティグループとVPCエンドポイントを作成するため、AWS CLIのコマンド群をスクリプトにまとめて実行しようとしたところ、**コーディングエージェント側の実行環境に配置した分類器**が、承認境界を迂回する操作だと判断し、**処理を自動停止しました。**

停止理由の要旨は、次のとおりです。

これは正しい判断です。

私たちは、AIが `aws` コマンドを直接実行できないという制約を設けています。しかし、コマンドをスクリプトで包むことで実行できてしまえば、その制約は形骸化します。

仕組みは、その迂回を検出して処理を止めました。

最終的に、このスクリプトは\*\*人間が自分の手で1コマンドだけ実行しました。\*\*所要時間は22秒でした。

ガードレールは、実際に発火して初めて意味を持ちます。この日、そのガードレールは実際に発火しました。

SynthAgentでは、この停止原則を個々の実行環境だけに任せるのではなく、承認・実行・証跡を管理する統治境界として製品化しています。

## 見つかった欠落は、その日のうちに塞ぎます

検証の途中で、もう一つの欠落が見つかりました。

到達確認プログラムであるprobe自体は実装済みでしたが、**それをエージェント実行基盤の内部から呼び出す経路が、まだ配線されていなかったのです。**

閉域内のデータベースには、外部から直接到達できません。実行基盤内からprobeを呼び出す経路がなければ、今回の検証そのものが成立しません。

ここからの対応は、私たちが実践している統治の型そのものでした。

1. 実装は別系統のAIが担当し、「作者≠レビュアー」を維持します。
2. まず「実装が存在しなければ失敗するテスト」を作成して**redを確認**し、実装後に**green**になったことを確認します（対象17件、スイート全体では2,028件がパス）。
3. 2本のプルリクエストについて、それぞれ独立レビューを実施します。CIの全チェックが完走したことを確認し、**mergeの最終承認は人間が行います。**

欠落を発見してから、実装・テスト・レビュー・merge・再デプロイまで、すべて**同日中**に完了しました。

速さを優先して統治を省略したのではありません。**統治の型があらかじめ決まっているからこそ、速く進められたのです。**

## 権限は「使い捨て」で渡します

今回の検証では、AIエージェントがSecrets Managerに保存された認証情報を読み取る権限が必要でした。

ただし、付与したのは一般的な「Secrets読み取り権限」ではありません。

* 対象は、**2件のsecretのARNだけに限定**しました。
* 暗号鍵の復号は、**Secrets Managerを経由する場合だけ**有効になるよう、ViaService条件を設定しました。
* 検証が終了した時点で、**このポリシー自体を削除**し、権限が残存していないことを確認しました。

最小権限を単なる設計思想にとどめず、**日々の運用習慣として定着させています。**

## 結果と、後始末まで確認します

夕方までに、検証は完了しました。

> AIエージェント実行基盤（コンテナ・閉域モード）からの実測: DB照会 112ms／Redis応答 98ms／認証情報取得 130ms／監査ログ書込 確認——**4項目すべてpass**

その後、計画どおり、**主要な実行資源であるDB・キャッシュ・NAT・実行ランタイムを、その日のうちに停止・削除しました。**

削除についても、「実行したから完了」とは判断しません。**readbackによって、現実の状態が変化したことを確認します。**

実は、この後始末の中で、一度「成功に見える失敗」が起きました。

別のAWS profileを参照したまま `terraform destroy` が実行され、画面には次のように表示されました。終了コードも0でした。

```
Destroy complete! Resources: 0 destroyed
```

しかし、これを削除完了とは扱いませんでした。

対象アカウントの状態をreadbackしたところ、リソースが残っていたためです。そこでcaller identityをあらためて確認し、正しいアカウントに切り替えてリソースを回収しました。

**コマンドが成功したかどうかではなく、現実の状態が変わったかどうかを確認します。**

readbackを必須にしていたため、誤った成功報告を、本当の成功として扱わずに済みました。

最終的に、DB・キャッシュ・NATは完全に削除されました。暗号鍵は7日間の削除待機状態となり、暗号操作には使用できない状態です。Secretsについても削除予定日が設定されています。

[AWS公式ドキュメント](https://docs.aws.amazon.com/ja_jp/secretsmanager/latest/userguide/manage_delete-secret.html)によると、削除対象としてマークされたシークレットには料金が発生しません。

ネットワーク側については、非同期で行われるENIの解放を待ち、翌朝に後始末を完了しました。その後の最終readbackで、全項目がpassしていることを確認しました。

この一連の流れは、時刻付きの証跡からそのまま追跡できます。

| 時刻 | 出来事 |
| --- | --- |
| 7/8 09:47 | `terraform apply`——36リソース作成 / 変更0 / 削除0 |
| 7/8 10:21 | 人間が承認済みコマンドを1つ実行（所要22秒） |
| 7/8 11:35 | 閉域ランタイムからDB・Redis・Secrets・Logsの4項目pass |
| 7/8 12時台 | 主要実行資源のdestroy開始 |
| 7/9 09:56 | ENI解放後の最終後始末・readback全項目pass |

朝の計画から、独立レビュー、停止、修正、実装、承認、実測、破壊、readbackまで、**今回の検証の全工程が、タイムスタンプ付きの証跡として残っています。**

## これが、私たちの売り物の正体です

種明かしをすると、この日、AIエージェントが検証していたのは、**そのAIエージェント自身の実行基盤**でした。

私たちはAIエージェント統治基盤を開発しながら、その構築作業自体も、製品がお客様へ提供するものと同じ統治のもとで進めています。

承認ゲート、作者≠レビュアー、証跡、最小権限、fail-closed。これらを、製品の説明だけでなく、自社の実務でも運用しています。

AIに仕事を任せるとき、本当に不安なのは「AIが賢いかどうか」ではないと思います。

「**一度任せてしまったら、その後もずっと人間が見張り続けなければならないのではないか**」という不安ではないでしょうか。

この日の検証では、その問いに一つの答えを出すことができました。

人間が常に見張っていなくても、止まるべきところでは処理が止まりました。AIの間違いは実際に発生しましたが、独立レビューとテストによって拾うことができました。そして、何が起きたのかは、あとから証跡をすべて確認できます。

私たちが作っているのは、AIを無条件に信頼するための仕組みではありません。

AIが間違えたときには処理を止め、人間へ判断を戻し、あとから何が起きたのかを確認できるようにする。AIと人間が、現実の業務を安全に分担するための仕組みです。

**AIを信じたから、仕事を任せられたのではありません。AIを信じ切らなくても、仕事を任せられる仕組みがあったのです。**

*AIA株式会社は、AIエージェント統治基盤「SynthAgent」と、監査証跡基盤「AgentLedger」を開発しています（2026年9月、AWS Marketplaceでの提供開始を予定しています）。*
