## 課題：音声エージェントの評価ブラインドスポット

音声系のエージェントには、典型的な故障パターンがよくある。社内デモでは問題なく動くのに、本番に出すと頻繁に失敗する——特に騒がしい環境からの着信で、背景のテレビの音を発話だと誤認する。ユーザーが言い終わる前に割り込む。情報はすべて集め終えたのに、終話前に肝心の金額を復唱し忘れる。これらに共通するのは、**既存の評価手段ではほぼ事前に検出にくい**という点だ。

**また、正直にいうと、いまのエージェントテストの大半はチャット、つまり従来のテキストの一問一答に集中している**。ツールチェーンが成熟していて低コストだからだ。一方で**音声エージェントの体系的なテストは極めて少ない**。音声に入った途端、ASRの誤認識・TTSの遅延・割り込みと相槌・ノイズ・電話のコーデックといった次元が一気に立ち現れる。これらはテキストの世界には存在しない。多くのチームは「テキストで通ったから音声も大丈夫だろう」と暗に仮定し、故障はまさにその仮定の隙間で起きる。

より具体的には、音声エージェントの評価そのものが、交わらない2つの体系に分かれている。

| 評価体系 | 測るもの | **カバーできない領域** |
| --- | --- | --- |
| 会話品質系 | 割り込み・相槌・自然さ・ノイズ耐性 | タスクが本当に完了したか |
| タスク完遂系(τ-bench など) | ツール呼び出し・最終 DB 状態 | 実音声環境での挙動 |

両方が通ったときに隙間からこぼれ落ちるものこそ、本番で最も破壊的な故障——「会話は自然なのに、裏でタスクが失敗している」だ。

さらに「たまの失敗」の累積効果も過小評価されやすい。特にコールセンターは同じフローを毎日数百回こなすため、1回の成功率(pass@1)では安定性を測れない。

| pass@1 | 4 連続成功 | 8 連続成功 |
| --- | --- | --- |
| 90% | 66% | 43% |
| **80%** | **41%** | **17%** |
| 70% | 24% | 6% |

pass@1 = 80% は一見許容範囲に見えるが、4回連続成功はわずか41%。日100件なら**月に約400件の失敗呼**が積み上がる計算だ。運用視点で見るべきは `pass@1` ではなく `pass^k` である。

したがって目標は、また新しいスコア計測ツールを作ることではなく、**「音声エージェントが本番でどう失敗するか」を事前に定量化できる**評価基盤を構築する必要がある。

## 技術選定：なぜ τ-voice を採用したか

コミュニティの既存手段を調査・比較した結果、最終的にSierraがオープンソースで公開している **τ-voice**(`sierra-research/tau2-bench`)を採用した。各カテゴリのトレードオフは以下の通り。

| カテゴリ | 代表的な手段 | 本ケースに対する制約 |
| --- | --- | --- |
| 汎用 LLM 評価 | DeepEval、Arize Phoenix、promptfoo | テキスト/chat 中心で、全二重音声をカバーしない |
| 音声フレームワーク同梱テスト | LiveKit Agents、Pipecat | 自社スタックに密結合し、独立した外部サービスを接続しづらい |
| 全二重の学術ベンチマーク | Full-Duplex-Bench | 合成タスクで、実際のツール呼び出しがない |
| 商用プラットフォーム | Coval、Hamming、Vapi | クローズドソース |
| **タスク × 会話 × 音声 の 3 軸** | **τ-voice(本プロジェクトで採用)** | — |

τ-voiceの差別化点は、3つの次元を統合して評価すること——**検証可能なタスク完遂・全二重の会話ダイナミクス・現実的な音声劣化**——にあり、第1節で述べたブラインドスポットをちょうど覆う。採用理由は、単一次元で「特定の競合より優れている」からではなく、この3軸の組み合わせがコミュニティに同種のOSS実装をほとんど持たないからだ。(補足:以上は手段レベルの調査・比較であり、各フレームワークを厳密に正面から横並び評価したものではない)

エンジニアリング上の価値は3点に集約される。

1. **失敗を改善方向に自動分類できる。** 同一タスクを 3 段階の難易度で 1 回ずつ実行する:テキスト → Clean 音声 → Realistic 音声。最初に落ちた段階が根源を指す——テキスト通過・Clean 失敗は **Voice-Fragile**(モデル/プロンプト)、Clean 通過・Realistic 失敗は **Noise-Fragile**(音響処理/STT)。
2. **再利用率が極めて高い。** orchestrator、音声劣化パイプライン、evaluator、バッチ runner、6 種の realtime アダプタなど、6〜7 割をそのまま使える。
3. **アブレーションが内蔵されている。** `--speech-complexity` で音声効果・アクセント・自由なターンテイキングを個別に切り替えられ、「STT 入れ替えで全体 +3pp」を「ノイズ +7pp、アクセント横ばい、圧縮 -2pp」へと分解できる。軸別に原因を切り分けられて初めて、評価は意思決定に使える。

ただしその「便利さ」の裏には、明確にしておくべき前提がある。**τ-voice は数多くの機構を内蔵している一方で、外部の agent サービスをそのまま接続するためのインターフェースだけは提供していない現状がある。** 通路全体が「テスト対象のエージェントはフレームワーク内に生える」という前提で設計されており、外部サービスを接続するには**多くの場合、逆にこちら側で大量の準備作業を要する**。続く3つの節がまさにその作業であり、ここから一つの教訓も見える——**OSS であること・「全部入り」であることは、接続が容易であることを意味しない。接続コストこそが本当のコストであることが多い。** さらに τ-voice はネイティブでは遅延・barge-in・ターンテイキング指標を出力せず、pass^k も直接は提供しない。これらも自前で補う必要がある。

## 接続課題1：被測対象のロール不整合

最初の障壁は、フレームワークの中核的な前提にある——**providerこそがAgent本体**であり、prompt・tools・VAD を持ち、フレームワークから下ろされて実行する(OpenAI Realtime も Gemini Live もこの形だ)。だが**我々が実際に開発している agent サービスはそうではない**。独立してデプロイされ、prompt・ツール・VAD はすべて自身の DB に格納され、`agentId` で引かれる。被測対象は**本番稼働中のブラックボックス**なのだ。

この不整合のため、`configure_session()` を no-op として実装することにした。

```
async def configure_session(self, system_prompt, tools, ...):
    """No-op。agent サービスの prompt/tools/VAD はすべて自身の DB 行にあり、
    AGENT_ID で引かれる。tau2 がここで下ろしてくる内容は一律に無視する。"""
    if system_prompt or tools:
        logger.warning("provider ignores tau2 system_prompt/tools ...")
```

これでメンタルモデルが明快になる:**我々が作っているのは Agent ではなく、一本の「擬似電話回線」だ**——片側は Twilio になりすまして模擬ユーザー音声を流し込み、もう片側は agent サービスの出力を tau2 が理解できる形へ翻訳する。この回線は Twilio Media Streams プロトコルを手書き実装して接続している(raw WebSocket、`connected` → `start` ハンドシェイク → フレームごとの `media`)。**Twilio アカウントは一切不要**だ——`/media-stream` は ws アップグレード経路に認証がなく、プロトコルさえ正しければ、対向は接続元を区別できない。

## 接続課題２：ストリーミング対話の離散化

音声対話は本質的に全二重ストリーミングで、イベントは順不同に届く。一方、評価には確定的かつ再現可能なタイムラインが要る。τ-voice は**離散時間 tick**(ティック **200ms**)でこれに対処する:各 tick でユーザー音声を 1 フレーム注入し、イベントを一括受信し、`TickResult` に翻訳する。この翻訳のなかに、実践から得たいくつかの判断がある。

* **会話終了の判定。** 当初は脆いヒューリスティックだった(ws が閉じたら `###STOP###` を注入し tau2 に `AGENT_STOP` を設定させる)。しかしサービスが別の理由で切断すると誤判定し、失敗するだけのリトライを何度も誘発した。その後サービス側に canonical な `reason` を伴う `session_end` イベントを追加し、このヒューリスティックを fallback へ降格させた。
* **「Agent が話し始めた時刻」をどこに取るか。** これは遅延指標の要だ。直感的には「音声が出始めた瞬間」を取りたくなるが、ツール呼び出しを伴うフローでは先にツールを叩き、数秒後にようやく TTS することが多く、音声起点で計算すると初回応答遅延が数秒も水増しされる。そこで OpenAI の `response.created`(`agent_turn_start` として送出)を話し始めの時刻、すなわち「応答すると決めた」真の瞬間として採用している。
* **sub-tick 精度。** 速いやり取りでは `speech_stopped` と `agent_turn_started` が同一 tick に収まることが多く、整数の通し番号だけで計算すると一律に遅延 0 になる。そのためイベントにミリ秒タイムスタンプを付与し、計算では優先的にこちらを使う。

## 接続課題３：ブラックボックスサービスの可観測性改造

第 3・4 節には緊張関係がある——サービスをブラックボックスと呼びながら、その `session_end` や `agent_turn_start` イベントに依存している。その出どころは——**自社の agent サービスを、テストモードのときだけ内部状態を能動的に外へ吐き出させる**ようにしたからだ。これは接続作業のなかで最も改修量の大きい部分だった:フレームワークがインターフェースを用意しないなら、被測サービス側に対向能力を生やす。

実装段階で設計の方向転換が一つあった:当初案は Genesys AudioHook + HMAC だったが、実際には **Twilio Media Streams + bench-only 拡張イベント**へ切り替えた——サービスの transport 層は、リアルタイム transcript を送るために「bench 専用 ws イベント」の経路を既に通していたため、それを流用するほうが、AudioHook エミュレータをゼロから作るより遥かに経済的だった。サービス内に統一の `sendBenchEvent` チャネルを抽象化し、env スイッチで総制御して、重要な状態を 3 バッチで送出する。

| バッチ | 送出内容 | 解錠する能力 |
| --- | --- | --- |
| 第 1 バッチ | `tool_call` / `tool_result` | ACTION 採点 + ツール軌跡(flow 失敗分析へ) |
| 第 2 バッチ | `session_start` / `session_end`(reason 付き) | 脆い `###STOP###` 推定の置き換え |
| 第 3 バッチ | `user_speech_*` / `agent_turn_*` / `barge_in` | 遅延 / barge-in / ターンテイキング指標の解錠 |

各フィールドは特定指標の精度のために働く:`agent_turn_end.reason == "interrupted"` は真の割り込みを正確に判定し、`turnKind`(greeting / user\_response / tool\_continuation / announce)は「本当にユーザーへ応答したターン」から開始挨拶やツール継続を除外する。さもないと false-trigger はノイズだらけになる。

加えて、テスト環境向けの安全弁が 2 つある:サービスには 30 秒の「挨拶保護」タイムアウトがあり、本番では Twilio mark の返送で解除されるが、テスト回線は mark を返せないため、最初の 30 秒のユーザー音声が barge-in 扱いで捨てられてしまう。そこで `DISABLE_GREETING_PROTECTION` を導入した。また Agent はツール呼び出しの合間に無音になり、模擬ユーザー(同じく LLM)はそれを「無応答」と誤解するため、`KEEPALIVE` で μ-law の無音を埋める。これらのスイッチはすべて env gate 制御で、**本番環境では一律に無視され、本番コードにテスト分岐は一切残らない**。

## 測れること、そして測れないこと

イベントとブリッジが整うと、指標体系は 4 層に分かれる。純粋に確定的に計算できるものから、LLM 判定に依存するものまで。

| 層 | 指標 | 計算/説明 |
| --- | --- | --- |
| ① 音声品質(確定的) | 初回応答遅延 | `speech_stopped` → 次の user\_response 型 `agent_turn_started` の時間差、P50/P95 を出力、10s 超は取りこぼし扱い |
|  | 不適切な割り込み | `agent_turn_end.reason == interrupted` を計数、毎分換算 |
|  | 取りこぼし turn miss | ユーザー発話終了後 >10s 未応答の割合 |
|  | 誤発火 false trigger | Agent の話し始め前 5s 内にユーザーの `speech_stopped` 先行なし |
| ② 信頼性 | pass^k、Fragility 分類 | k=1..4;Voice-/Noise-Fragile を 100% 閾値で判定 |
| ③ フロー | フロー失敗ノード分布 | `transition_step` から各ノードの到達/通過/失敗率を算出、改善 backlog を自動生成 |
| ④ コスト | STT/TTS 推定 | サービスは ws の対向にあり LLM usage を取得できないため、静的価格 × 時間/文字数で「推定」 |

境界と負債は明確にしておく。**根本的な境界:これが評価するのは Agent と agent サービス本体であり、Twilio のような電話事業者層はカバーできない。** 手書きプロトコルで実 Twilio を迂回して音声を注入しているため、アカウント不要・再現可能という利点がある一方、**実電話回線の問題(キャリア網の揺らぎ、実コーデックの挙動、事業者自身のエッジケース)はすべて射程外**だ——「Agent が使えるか」には答えられるが、「この電話回線が信頼できるか」には答えられない。既知の負債:遅延 **P50 はフィールド精度の不具合で現状ゼロ固定**のため一貫して P95 を使用。gold `actions` は多くが空で、主力の採点は依然 LLM judge に依存。ハルシネーション検出 reviewer の基準は緩めで相対比較のみに適する。多くの構成はサンプル数 n=6 で信頼区間が広い。以上はすべて backlog に明記済み。

## 事例検証:多次元指標による失敗の原因特定

最後、VAD構成選定では約 **251 sim** を実行し、13 構成 × 4 段階ノイズ × 複数段階のノイズ抑制 × 6 タスクをカバーした。主な結論:最終的に **gpt-realtime-2 + semantic\_vad / medium / 近接**を採用(Clean で首位タイ、かつデフォルト設定で達成);**gpt-realtime-2 + server\_vad は著しく退化**(6 タスク中 1 のみ通過、同じ server\_vad は旧モデルで 4 通過)→ 強い結論として **v2 では server\_vad を無効化**;**多人数の喧騒(SNR 0dB)では全構成が失敗**。

より価値があるのは結論そのものではなく、**多人数喧騒下の失敗の原因特定**だ。pass@1 だけ見ると安直な診断に陥る:「うるさすぎて Agent が聞こえていない」、ゆえにモデルを上げ、ノイズ抑制を強める、と。だが 5 つの指標を並べると、結論は正反対になる。

| 指標 | 静音環境 | 多人数喧騒 | 変化 |
| --- | --- | --- | --- |
| 毎分の割り込み回数 | 0.24 | 4.05 | **×17** |
| 誤発火率 | 0.3% | 12.9% | **×43** |
| **取りこぼし率** | 1.7% | 1.7% | **ほぼ不変** |

**取りこぼし率がほぼ不変**ということは、Agent は「聞こえていない」のではない。真相は**過敏**だ:背景の人声を発話と誤認し、頻繁な自己割り込みと誤発火を起こしている。診断が反転すれば、対策も反転する:**モデルのアップグレードやノイズ抑制の強化ではなく、VAD の閾値を上げるべき**だ。これこそがこの基盤の核心的な価値である——通ったか否かを判定するだけでなく、失敗の原因と改善の方向まで指し示す。

## おわりに

振り返ると、この基盤が本当に解いたのは、漠然とした不安——「デモは全部通ったのに、なぜ本番に出すと落ちるのか」——を、スケジュールに乗せられ、原因まで切り分けられ、A/Bにかけられる数字へと変えたことだ。ほぼ誰もがチャットを測り、音声エージェントを真面目に測る人はまだ少ない今、この経路を通したこと自体に価値がある。

このプロセスを通じて、一つはっきりしたこともある:**フレームワークがどれだけ多くの機構を内蔵していても、自社サービスがすんなり接続できることを意味しない**。我々が手間をかける必要が必須だと考えている。また、チャットと違って、音声エージェント自体をテストする際、コストの問題も発生しやすいため、ちゃんと注意する必要があると思っている

今後も音声エージェントの自動テストまわりは継続的に掘り下げていく予定なので、どんどん実践して、新しい知見が得られたら、共有していきたいかと考えている。ぜひご期待ください〜
