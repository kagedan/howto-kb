最近、自分専用のAIエージェントとして **[Agent-Sin](https://agent.shingoirie.com/)** というものを作りました。

<https://agent.shingoirie.com/>

Agent-Sin は、ざっくり言うと「会話で自分用のスキルを呼び出して、日常や仕事の手助けをするAIエージェント」です。

Gmailを整理したり、Googleカレンダーを確認したり、GitHubの通知を見たり、Stripeの売上を集計したり、食事ログを記録したり、朝に今日の予定やニュースをまとめたりします。

一見すると、よくあるAIエージェントっぽく見えるかもしれません。

ただ、作っていく中で強く感じたのは、**AIエージェントを実用で使うには「全部をLLMに任せる」のではなく、「LLMに任せる部分」と「プログラムとして固定する部分」を分けることがかなり重要だ**ということです。

この記事では、Agent-Sinを作りながら考えた、個人用AIエージェントの設計について書いていきます。

## AIエージェントは便利だけど、そのままだと実用が難しい

LLM に自然言語で指示できるのは、とても便利です。

* 「今日の予定を見て」
* 「要返信メールある？」
* 「この投稿案どう思う？」
* 「朝のブリーフィングを送って」

こういう依頼を自然に扱えるのは、AIエージェントの大きな魅力です。

一方で、実際に日常運用しようとすると、いくつかの問題が出てきます。

* 毎回判断が少しぶれる
* 同じ作業でも実行手順が安定しない
* 外部API操作を任せると権限が怖い
* 処理が遅くなりやすい
* 毎回LLMに考えさせるとコストもかかる

特に、メール整理や予定登録、通知、削除、記録のような **副作用のある処理** は、ただ「賢いLLMに任せる」だけだと不安があります。

そこで Agent-Sin では、**会話する部分と、実際に処理する部分を分けています**。

## 会話モードと Program Skill を分ける

Agent-Sin には、**会話モード** と **Program Skill** があります。

会話モードは、ユーザーの発話を読んで、何をしたいのかを判断します。

* 「Gmail 整理して」と言われたら、Gmail 整理スキルを呼ぶ
* 「今日の予定は？」と言われたら、Google カレンダーの予定確認スキルを呼ぶ
* 「この内容メモして」と言われたら、メモ保存スキルを呼ぶ

一方で、実際の処理は Program Skill が担当します。

* Gmail API をどう叩くか
* どのラベルを付けるか
* どのメールを既読にするか
* どの形式で Telegram に通知するか
* どこにログを残すか

こういう具体的な処理は、毎回LLMに考えさせるのではなく、スキル側にプログラムとして実装します。

つまり、**LLM は「判断とルーティング」を担当し、Program Skill は「安定した実行」を担当する**、という分担です。

これは作ってみるとかなり効きます。

AIエージェントというと、つい「何でもLLMにやらせる」方向に考えがちです。でも実用上は、むしろ逆で、**安定して繰り返す処理ほどプログラム化したほうがいい**。

* LLM は曖昧な自然言語を受け止めるのが得意
* プログラムは決まった手続きを正確に実行するのが得意

この2つを混ぜすぎないことが、個人用エージェントを日常で使えるものにするポイントだと感じています。

## スキル化すると、エージェントのぶれが減る

Agent-Sin で最近僕がつくったスキルはこんな感じです。

* **Gmail 整理スキル**：未読メールを確認して、返信が必要そうなものに「要返信」ラベルを付けます。それ以外のメールは整理待ちに退避します
* **朝ブリーフィングスキル**：今日の予定、要返信メール、ToDo、ニュース、健康ログをまとめて Telegram に送ります
* **食事コーチスキル**：食事内容や体重、筋トレ履歴を記録し、カロリーやタンパク質の目安を返します
* **SNS マーケティング系スキル**：過去の投稿結果や学びを保存し、次の投稿案を考える材料にします

これらを **すべてLLMの自由実行にすると、毎回少しずつ処理が変わってしまいます**。

でもスキルにしておけば、基本の動きは固定できます。

* Gmail 整理なら Gmail 整理のルール
* 朝ブリーフィングなら朝ブリーフィングの構成
* 食事ログなら食事ログの保存形式
* SNS 分析なら SNS 分析の履歴管理

そのうえで、判断が必要な部分だけLLMを使う。

スキル1個の中身はこんな構造になっています。

スキルの中で **LLM を使いたい箇所だけ `ai_steps` として宣言しておき、`ctx.ai.run(...)` で呼ぶ**。それ以外の手続きは普通のコードです。「全体は固定、判断が要る一部だけLLM」というメリハリが付けやすい構造になっています。

この形にすると、**自然に頼めるけど、実行は安定している** という状態に近づきます。

## 記憶は人間も読める形にする

Agent-Sin では、**記憶の扱い** もかなり大事にしています。

AIエージェントに記憶を持たせるというと、ベクトルDBに全部入れるような設計を想像しがちです。

もちろん検索用のインデックスは便利ですが、それだけだと **人間が中身を把握しづらくなります**。

そこで Agent-Sin では、長期プロフィールや日次ログ、メモ、トピックごとのナレッジを **Markdown として保存** しています。

* ユーザーの長期的な好みや仕事の方針 → プロフィール
* 日々の会話や作業ログ → 日次メモ
* 特定トピックについて繰り返し出てきた話 → トピックナレッジ

こうしておくと、**AI だけでなく人間も読めます**。Obsidian で開けば、自分の活動ログや思考の蓄積としてそのまま見られます。AIに閉じたブラックボックスではなく、人間とAIが共有できる知識ベースになります。

Obsidian Vault は実体ではなく、ワークスペースの実ファイルへの **symlink ビュー** として構成しています。

個人的には、この **「人間も読める記憶」** はかなり重要だと思っています。

AIエージェントは、使えば使うほど自分の文脈を理解してほしい。でも、その記憶がどこにあって、何を覚えているのかが見えないと怖い。

Markdown で残しておけば、必要なら **自分で読めるし、直せるし、消せる**。

エージェントの記憶を、自分の外部脳として扱いやすくなります。

さらに、日次の会話ログのうち「人生に関係ありそうな情報」だけを、夜間に長期記憶へ昇格させる仕組みも持っています。

**何を昇格させたかの diff も残している** ので、後から「いつ・なぜAIがこれを覚えたか」を追えるようにしてあります。

## スキルごとに状態を持つ

もうひとつ重要なのが、**スキルごとの状態管理** です。

Agent-Sin では、スキルごとに **SQLite DB を持てる** ようにしています。

* 食事コーチなら、食事ログ・体重・筋トレ履歴
* SNS マーケティングスキルなら、投稿結果・学び・プロジェクト情報
* Gmail 整理スキルなら、過去の処理ルールや分類結果
* ToDo スキルなら、未完了タスクや期限

こういう履歴は、**全体の長期記憶に全部混ぜるよりも、スキル側に閉じ込めたほうが扱いやすい** です。

全体プロフィールには「ユーザーが何を好むか」「どんな方針を持っているか」を置く。スキル固有の細かい履歴は、そのスキルのDBに置く。

この分離によって、**記憶が散らかりにくくなります**。

AIエージェントの記憶というと何でも一箇所に入れたくなりますが、実際には **「全体の文脈」と「業務ごとの履歴」は分けたほうがいい** と感じています。

スキルごとにDBを分けるメリットはもう一つあって、**スキル単位でポータブル・削除可能** という点です。スキルを別の環境に持っていきたければ DB ごと運ぶ。要らなくなったら DB ごと消す。共有DBにしてしまうと、これがとても面倒になります。

スキル側からは、こんな1行で履歴を書き込めるようにしています。

```
await ctx.history.append({
    "time": ctx.now(),
    "meta": {"action": "label_added", "label": "要返信"},
    "content": "From: foo@bar / Subject: ...",
})
```

スキーマは `id / time / meta(JSON) / content` だけ。`meta` を JSON にしておけば、スキルごとに自由な構造化情報を入れられます。共有スキーマを決めようとすると必ず破綻するので、最初から自由にしています。

## 副作用のある操作は構造化する

Agent-Sin では、**会話モードの返答だけでは副作用が起きないようにしています**。

たとえば、ユーザーに「Gmail を整理しますね」と文章で返しただけでは、実際には何も起きません。

本当に Gmail を整理するには、Gmail 整理スキルを **構造化された skill\_call として呼び出す必要があります**。

これは地味ですが、かなり重要です。

AIエージェントで怖いのは、**「それっぽく返答しただけなのに、ユーザーには実行されたように見える」** ことです。会話モードのLLMが「やっておきました」と書けば、ユーザーは実際にやったと思います。でも実際にはAPIが叩かれていない——これが起きると、エージェント全体への信頼が一気に崩れます。

Agent-Sin では、副作用が伴うスキル（メモ保存、ToDo追加、メール操作、外部送信など）には `side_effect: true` を立てています。**このフラグが付いたスキルが呼ばれたターンでは、LLMが返した会話文のほうを捨てて、スキルの実行結果だけを履歴に残す** ようにしています。

つまり、**「やったつもり」を残さない** ことを構造的に保証しています。

* 副作用スキルが呼ばれた → スキルの実行結果（事実）だけ残す
* スキルが呼ばれなかった → LLM の会話文だけ残す
* スキルがエラーで終わった → エラーが正しく残る

LLM のプロンプトで「嘘をつかないでね」と縛るより、**そもそも narrative を物理的に殺してしまう** ほうが確実です。

## 全体としてどうつながっているか

ここまでの設計を1枚にまとめると、こうなります。

ポイントは3つ。

1. **入口は LLM、実行は Program Skill**。LLMはルーティングと判断、Skillは安定実行
2. **記憶は Markdown と SQLite に分離**。人間が読むものと、スキルが構造化で持つもの
3. **Obsidian Vault は記憶のビュー**。実体は別ディレクトリで、ユーザーが直接読み書きできる

## マルチチャネルでも同じエージェント

入口がCLI・Discord・Telegram・スマートグラスと複数ありますが、**裏側の Chat Engine・スキル・記憶はすべて共通** です。

外出中はスマホの Discord から話しかけて、家では CLI で続きを書く、出先で思いついたメモはスマートグラスから喋る——どこから入っても同じスキル・同じ記憶を共有します。

「同じエージェントが、複数の入口から呼べる」というのは、個人用エージェントの実用性をかなり上げる気がしています。

## スキル自体もAIに作らせる（Build Mode）

普段は会話モードで動いていますが、`/build` でモードを切り替えると、**スキル自体をAIに生成させる** ことができます。

生成されるのは普通のファイル（`skill.yaml` + `main.py`）なので、AIが書いたコードを後から人間が編集することもできます。

**AIエージェントの能力拡張が、AI自身によるコード生成で行われる** という再帰構造になっていて、ここはまだ実験的な領域ですが、個人用エージェントとは特に相性がいいと感じています。

## 作って分かったこと

**良かった点**

* LLM と Program の責務を分けたことで、エージェントが日常運用できる安定感まで上がった
* スキルが Git で diff を取れる「資産」として残る。半年前のスキルが今も普通に動く
* Obsidian でAIの記憶を直接読めると、不気味さが消えて信頼感が上がる
* スキル単位のSQLite は、想像以上に副作用が少なくて快適。スキル削除＝DBごとゴミ箱、で完結する

**難しかった点**

* 「これは会話で済むのか、スキルを呼ぶべきか」の判定はモデルによって精度が違う。Intent Router を別レイヤーで持っておいて良かった
* `side_effect` の概念をLLMのプロンプトだけで縛るのは限界がある。narrative を物理的に捨てる仕組みにして初めて安定した
* マルチチャネル統合は便利ですが、「どこから送ったメッセージか」をログに残しておかないと後で混乱します

**これからやりたいこと**

* スキル間のワークフロー（あるスキルの結果を別のスキルに渡す）を、LLM経由ではなく宣言的に書ける仕組み
* トピック単位の意味検索（メモだけでなく全レイヤーを横断する）
* Build Mode の生成品質をもう一段上げる

## まとめ

Agent-Sin を作りながらいちばん強く感じたのは、**AIエージェントは「LLM単体」ではなく「LLM＋プログラム」で考えたほうが圧倒的に実用的になる** ということです。

* **LLM** は曖昧な自然言語を読み取り、判断し、ルーティングする
* **Program Skill** は安定した手続きを正確に実行する
* **記憶** は人間も読める Markdown と、スキル固有の SQLite に分ける
* **副作用** は構造化された skill\_call でしか起きないようにする

「何でもLLMにやらせるAIエージェント」ではなく、**LLMをひとつの強い道具として、プログラムの中に組み込んで使う**。

個人用AIエージェントの実用性は、この責務分離の徹底で大きく変わると感じています。
