## はじめに

こんにちは！イノベーション開発チームのmiyaken85です！  
普段はITトレンドの開発をしつつ、エンジニア広報PJ活動もやらせてもらっています。

弊社のエンジニア組織では、テックブログもあることだし、アウトプットをもっと活発にしていきたいところなのですが、いざ「じゃあZenn書こう」となると、書き始めるハードルって地味に高いって人も多いのかなと思います。

なので、Zenn記事の執筆をサポートしてくれるClaude Skillを作ってみました。  
（弊社組織全体がClaudeを採用してるという背景もあります。）

今回は、そのスキルを設計する中で結構迷ったポイントがいくつかあったので、最終的にどう判断したかを共有する話をします。

## TL;DR

* チームのZennアウトプットを増やしたくて、Zenn記事執筆サポートスキルを作りました
* 設計で最重視したのは「書き手の動機・声を消さないこと」
* 結果、AIに代筆させない「介助型」のコンセプトに辿り着きました
* 主要な設計判断は4つあって、それぞれの理由を共有します

## 背景：チームメンバーのZennアウトプットを増やしたい

弊社では、エンジニアの技術発信を活性化させたいという思いがずっとあります。  
発信が増えれば、社外への認知度向上、採用、社内のナレッジ蓄積、いろいろ良いことがあるからです。

実際、1年ほど前からエンジニア広報PJを立ち上げて、Zennや[note](https://note.com/inno_tech/m/me0da0dc94db9)などでTechブログ活動をコツコツ続けてきました。少しずつ書く人も増えてきている実感はあるんですが、それでもまだまだ、「書きたい気持ちはあるけど書き始めのハードルが高い」っていうメンバーは結構いるんですよね。

書き始めの、

* 「これ書く価値あるかな」
* 「タイトルどうしよう」
* 「構成どうしよう」
* 「どこから書き出せば」

みたいな迷いを、AIで支援できないかな、というのが今回のスタートです。

## 出発点：なぜ「自動生成」ではなく「介助」なのか

スキルの方向性を考えるにあたって、選択肢としては「AIに記事を完全に書かせる」「半自動で書かせる」「執筆をサポートする（介助）」あたりがありました。

最終的に「介助」に振り切ろうと決めた理由が2つあります。

1つ目は、[Zennの利用規約](https://info.zenn.dev/2026-03-10-ai-contents-guideline)にこんな記載があったこと。

完全自動生成は明確にNGですね。

2つ目は、「AIが書いた、書き手の血が通っていない記事」って、書き手側にとっても完成までのモチベが続かないだろうし、読者側にとっても「ググったら出てくる情報の焼き直し」みたいになって価値が薄いと思うからです。

なので方向性を「AIが書く」ではなく「AIが書き手を助ける」、つまり**介助**に振り切ろうと決めました。

具体的には、スキルがやるのはこういうこと：

* 書き手の発想を引き出す（推論→提案→確認の対話）
* 構造・流れを整える
* 書き手の言葉や個性を保ったまま整形する
* 公開前の品質チェック

逆にやらないのは：

* 書き手の入力なしに本文を生成する
* 「バズりそうなネタ」を勝手に決める
* 書き手の口調を均一化する

このコンセプトが、後の全ての設計判断のベースになっています。

## 設計判断1: 「広報モード」を入れない（書き手の動機を最優先）

スキルを設計する初期段階、僕は実は「広報モード」というのを入れる気満々でした。

つまりこういうイメージ：

* **個人モード**：純粋に書き手のために動く
* **広報モード**：「うちの組織として発信すべきだが誰も書いてない領域」を提案する
* **ハイブリッド**：両方の視点を持つ

僕がエンジニア広報PJのリーダーをやってる立場でもあるので、「組織としての発信戦略」も乗せられた方が便利なんじゃないかなと思ってたんですよね。

ただ、議論を進めるうちに、これは入れない方がいい、という結論に至りました。

理由はシンプルで、「広報モードを乗せた瞬間、書き手の頭に『これバズるかな』『広報的に良い記事かな』が入ってくる」からです。

書き手がそれを気にし始めると何が起きるかというと、素直さが消えます。

書き手が一番強いネタって結局、「自分が書きたいこと」。  
「組織として発信すべきこと」じゃなくて、素直に書きたいものを書いた方が、結果的に熱量も高くなって、内容も濃くなって、最終的に良い記事になると考えています。

「広報的に書くべき内容」が頭に入った瞬間に、その素直さが死んじゃう。  
これはトレードオフですらなく、純粋なマイナスでした。

なので最終的に、スキルは「個人モード」だけにすることにしました。  
書き手の動機を100%尊重する設計です。

組織としての発信戦略は、スキルじゃなくて別の場所（PJの運営、レビュー、振り返り会など）でカバーすればいいかなと思っています。

## 設計判断2: ですます調をデフォルトに、個性はスポットで残す

文体をどうするかも、地味に悩みました。

選択肢としてはこんな感じ：

* 全文ですます調で統一（企業発信として無難）
* 全文だである調（個性が出やすいが、企業発信としては微妙）
* 書き手任せ（一貫性が出ない）

最初僕は「企業発信なんだからそりゃですます調でしょ」とサラッと決めようとしてたんですが、ここでちょっと立ち止まりました。

全文ですます調で均してしまうと、書き手の人格が消えてしまうんですよね。  
どの記事を読んでも「整っているけど、誰が書いたかは分からない」みたいな仕上がりになる。これだとAIっぽさが出ちゃって、「介助」のコンセプトと整合性が取れなくなります。

なので結論として「ですます調をデフォルト、個性はスポットで残す」方針にしました。

具体的には、以下のようなものは「書き手のスポット個性」として保護対象にしています：

* 体言止め（「あの時の心拍数。」みたいなやつ）
* 感情表現（「マジで」「やばい」「ぐぬぬ」など）
* オノマトペ（「サクッと」「ガッと」）
* 口癖・決まり文句
* 絵文字のスポット使用

これらが整形時に削除されないように、スキル内で明示しました。  
「整えるのは段落構造と接続詞だけ。個性表現は触らない」というルールです。

逆に、表記揺れ（ユーザ/ユーザーの混在など）や半角全角の混在みたいな読みにくさを生むものは、整えるようにしています。

「企業発信としての品」と「書き手の個性」の両立、口で言うほど簡単じゃないですけど、この方針が今のところしっくり来ています。

## 設計判断3: 「工夫」と「苦労」を主役級セクションに昇格

これは設計議論の中で、僕から「絶対入れた方がいい」と推した話です。

最初の設計だと、「工夫したこと」「苦労したこと」は補助的なセクションとして、本論の隙間にちょこっと書く扱いでした。

でもこれ、僕的にめちゃくちゃ違和感があって。

技術記事を読むときって、「Xとは何か」「Xの基本的な使い方」みたいな汎用情報は、正直、公式ドキュメント読めば十分なんですよね。  
読者が「ここに書いてあって良かった」と思うのは、「書いた人が実際にどう工夫したか」「どこで詰まったか」みたいな、その人にしか書けない部分だと思うんです。

これって、AIが書けない部分でもあります。  
Claudeに「〇〇について解説して」と頼んでも、「私は〇〇のとき、こう工夫しました」「ここで詰まりました」までは書けない。なぜなら、それは書き手の固有の経験だから。

つまり、工夫と苦労って、

* 読者にとって最も価値が高い情報
* AIが代替できない情報
* 書き手にしか書けない情報

の三拍子揃ったやつなんですよね。これを補助セクションに置く理由は無いな、と思いました。

なので各タイプのテンプレで、「工夫したこと」「苦労したこと」を独立した章として、他の章と同じ重みで配置するようにしました。

加えて、これを引き出すための質問例もテンプレに明記しています。  
書き手って、自分の工夫を「当たり前のこと」と思って書かないことがあるんですよね。  
スキルが「これ、書かないんですか？絶対書いた方がいいですよ」と引き出す役割を持つようにしました。

例えばこんな質問：

* 「公式手順そのままじゃなくて、自分で工夫したポイントってありました？」
* 「一番時間溶けた所どこ？」
* 「ドキュメントだけだと分からなかった所ある？」

書き手の脳内で「当たり前すぎて言語化してなかった暗黙知」を引き出すための装置として、このセクションは効くと思っています。

実際、僕がこのスキルを使ってこの記事を書いてる最中にも、  
「あれ、これ書く必要ある？」って思ったポイントが、スキルに「これ大事なんで書いた方がいい」と背中を押されて書いた箇所が何個かあります笑

## 設計判断4: Zenn CLI連携は採用しない（編集の気軽さを優先）

Zenn で記事を書くやり方は、大きく2つあります。

1. Web エディタで直接書く
2. ローカルで Markdown を書き、Zenn CLI + GitHub 連携で公開

エンジニア的には2の方がカッコ良さげに見えるんですが、これは採用しませんでした。

理由は、GitHub 連携すると Web エディタでの編集が事実上できなくなるからです。

Zenn の仕様として、GitHub 連携している場合は「GitHub 側が正」とみなされます。Web エディタで何か直しても、次のデプロイで上書きされちゃう。つまり、ブラウザでサクッと直す、というのができなくなるんですよね。

これがなぜ問題かというと、記事って最後の文の調整は、結局人間が手でちまちま直すフェーズが必ずあると思います。  
スキルで構成と本文を一通り作っても、「ここの言い回し変えたい」「この単語のニュアンスがしっくり来ない」みたいな微調整が、書き手の感性次第で発生する。

その「最後の微調整」を Web エディタで気軽にやれるようにしたかったので、CLI 連携は採用しないことにしました。

なのでスキルの最終出力は、Zenn Web エディタにコピペできる形式にしています。

```
========================================
📝 Zennにペーストする用

【UIで設定するメタデータ】
タイトル: ...
絵文字  : 🛠️
タイプ  : tech
トピックス: tag1, tag2, tag3, tag4, tag5

【本文（以下をエディタにコピー）】
─────────────────────────────
（markdown本文）
─────────────────────────────
========================================
```

書き手は、上のメタデータを Zenn UI の各フィールドに入れて、本文をコピペするだけ。

CLI を入れてもらう必要もないし、GitHub 連携の設定をしてもらう必要もありません。社内配布するときのハードルがだいぶ下がりました。

「ツールはなるべく書き手に余計な手間をかけさせない」というのも、地味にこのスキルの大事な原則の一つになっています。

## このスキルの独自の判断軸

設計の各判断を貫いている「独自の判断軸」がいくつかあります。似たような社内向けAIツールを作りたい人の参考になればと思って、整理して書いておきます。

### 軸1: 書き手の動機・熱量を最重要視する

通常、ツールの設計って「機能性」「効率」「保守性」を上位に置きがちです。このスキルは、それらより上に「書き手の動機・熱量が維持されるか」を置きました。

書き手のモチベが続かなければ記事は完成しないし、完成しないツールは機能がどれだけ豊富でも価値ゼロですよね。なので「書き手が乗れるか」を最優先にしました。

ちなみに、この軸を最優先した結果、スキル自体は少し複雑な構造になっています。記事にするトピックについて質問や提案で深ぼっていくので、このskillを使ったからって、めちゃくちゃサクサク記事が増やせるわけではありません。  
シンプルさはちょっとトレードオフしちゃったかも。

### 軸2: 書き手の固有性を引き出す

書き手にしか書けない情報を主役にする、というのを記事全体の構造に組み込みました。

設計判断3が一番表れている軸ですが、それ以外の場面でも貫いています。  
例えば「過去の自分」を対象読者にする推奨、固有の数値・期間・規模を入れる推奨、書き手特有の動機を引き出す質問など、書き手の固有性を引き出す仕掛けが各テンプレに組み込まれています。

### 軸3: 「すごくない」と思ってる経験を、価値ある記事に変える

軸2の派生なんですが、書き手って自分の工夫を「これ普通でしょ」「すごい話じゃないから書く価値ない」と判断しがちなんですよね。

でも、書き手の脳内で「当たり前すぎて言語化してなかった暗黙知」って、他のチームや会社から見たら独自の価値だったりすると思うんです。特にAIが主流になった今は尚更です。

なのでスキルは、引き出し質問で書き手の暗黙知を顕在化させる役割を持っています。「これ、書かないんですか？書いた方がいいですよ」と背中を押す装置です。

### 軸4: 「ネタがない」と感じてる人も書き始められるようにする

「書きたい気持ちはあるけど、ネタが思いつかない」って人もいます。

そういう人向けに、Backlog / Slack / GitHub / Notion などのMCP連携を通じて、書き手の実際の活動から「これ記事になりそうじゃないですか？」を提案するモードを設けました。

ここで重要なのは、AIが「ネタを作らない」こと。あくまで書き手が「やった」「言った」「詰まった」記録の中から、記事になりそうなシグナルを拾って並べる、という仕掛けです。

「これ、あなたの中に既にありますよね？」を可視化する装置として動きます。

## 設計過程で考えが変わったこと

### 「コードは5行以内」「5分で読める長さ」を一度推した

設計初期、Zenn の人気記事の傾向に乗っかって、「コードは最小5行」「5分で読める」みたいな数値ルールをスキルに入れる方向で動いてました。

でも議論の中で「字数を絞ったら、骨太な情報量が必要な記事が書けなくなるのでは？」となり、ハッとしました。

結局のところ、長さは内容で決まるべきで、数値で先に縛ると逆に弱い記事になるリスクがある。

なので最終的には、数値制限はやめて、「このコードの役割は何か？」「この章は本当に必要か？」を1回ずつ問う、というスタイルに変えました。

## 完成したスキルの構成

最終的にできあがったスキルの全体像をざっくり共有します。

### ファイル構成

```

├── SKILL.md                       # メインエントリ
├── config/
│   ├── writer.yaml                # 書き手プロフィール
│   ├── tone.yaml                  # 口調設定
│   └── sensitive-names.yaml       # 機密情報チェックリスト
├── references/
│   ├── zenn-web-editor.md         # コピペ出力フォーマット
│   ├── zenn-markdown.md           # Zenn 独自の Markdown 記法
│   ├── tone.md                    # 個性表現の扱い
│   └── ai-policy.md               # Zenn AIポリシーへの姿勢
├── templates/                     # 記事タイプ別テンプレ（5種）
│   ├── howto.md                   # 手順解説系
│   ├── troubleshoot.md            # トラブルシューティング系
│   ├── intro.md                   # 技術紹介系
│   ├── decision.md                # 設計・思想系
│   └── retrospect.md              # 体験・振り返り系
└── workflows/                     # 各 Phase のロジック
    ├── 00-idea-generation.md      # アイデア生成
    ├── 01-environment-detect.md   # 環境判定
    ├── 02-interview.md            # シード捕獲・3点確定
    ├── 03-outline.md              # アウトライン提示
    ├── 04-draft.md                # 発散執筆（雑書き＋引き出し）
    ├── 05-code-sanitize.md        # コードの安全化
    ├── 06-polish.md               # 口調レビュー
    ├── 07-publish-check.md        # 公開前チェック
    └── _state.md                  # 中断・再開
```

22ファイル、合計約3000行ほどです。

### 執筆フロー

スキルを起動すると、以下の Phase で書き手と対話を進めます：

1. **Phase 0: モード判定** — 書き手の最初のメッセージから「ネタ決まってる / ぼんやり / アイデア無し」を推測
2. **Phase 0.5: アイデア生成** — ネタが無い人向けに、MCP連携や対話で記事ネタを掘る
3. **Phase 1-2: シード捕獲と3点確定** — 対象読者・読後ゴール・タイトル仮を確定
4. **Phase 3: タイプルーティング** — 5タイプの中から該当テンプレを選ぶ
5. **Phase 4: アウトライン提示** — テンプレに基づく見出し構成を提示、合意を取る
6. **Phase 5: 発散執筆** — 章ごとに「雑書き → 工夫・苦労の引き出し → 整形」の3段
7. **Phase 6: コード・図の埋め込み** — セキュリティチェック後にコード装飾
8. **Phase 7: 口調レビュー** — 書き手の過去記事と照合して個性が消えてないか確認
9. **Phase 8: 公開前チェック** — 機密情報、表記揺れ、人物言及などを最終確認

各 Phase の完了時には「状態スナップショット」を出力して、書き手が中断・再開しやすいようにしています。

### 出力フォーマット

最終的な出力は、設計判断4で示した通り、Zenn Web エディタにそのままコピペできる形式です。CLI も GitHub 連携も不要で、ブラウザでサクッと書き、ブラウザでサクッと公開、というシンプルな体験です。

## まとめ

ということで、Zenn記事執筆をサポートするClaude Skillを作った話でした。  
この記事も、skill使って書いたものであり、これから社内で配布予定です。  
何ヶ月かやってみて、記事率がどう変わったかなども今後報告できたらと思います。

これをきっかけに、弊社のアウトプット活動がより活性化して、みんなで組織全体を盛り上げていけたらいいなと思いますし、同じように記事執筆に悩んでる方の参考になればいいなと思います。
