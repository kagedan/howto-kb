# Ralph Wiggum Loop（ラルフ・ウィガム・ループ）とは

「ラルフ・ウィガム・ループ」は、AIエージェントを自律的に何度も実行するループ制御の考え方です。

従来のAIを使ったタスク処理は「1回プロンプトを投げ、結果を受け取り、必要なら人間が修正してまたプロンプトを書く」という1ショット中心の流れでした。

![](https://static.zenn.studio/user-upload/c6edce3045cf-20260304.png)

ラルフ・ウィガム・ループはそれと違い、タスク指示をAIエージェントに投げ、その結果を評価し（テスト・成功条件など）、まだ完了していなければ同じAIに投げ直し、条件を満たすまで繰り返します。人間が「まだできてないよ」と言い続ける代わりに、評価ロジック自体をシステムに組み込むのです。

# 何が嬉しいのか

ラルフ・ループによって、人間が何度もプロンプトを書き直すことなく、AIが自律的に大きなタスクを完了させることができます。「大きな仕事をAIに丸投げできる」ことへの期待が、この手法の盛り上がりの背景にあります。

向いているのは、「いろいろな観点で考えなければいけない複雑なタスク」「単純だけど件数が多いタスク」「試行錯誤が必要なタスク」です。個人的な見解としては、**生成する側とチェックする側に分かれて品質を高めていくユースケース**と特に相性が良いと思っています。

たとえばソフトウェアの大規模リファクタリングであれば、ソースコードを修正するエージェントと、静的解析・自動テストで検証するエージェントがループします。研修教材から理解度確認テストを生成する場合は、問題を作るエージェントと「正解が正しいか・唯一か」を検証するエージェントが対になります。イベント登壇の原稿作成なら、草稿を書くエージェントと聞きやすさ・語彙レベル・矛盾を指摘するエージェントで繰り返します。提案ストーリーの作成では、論理の一貫性や説得力のチェック、反論の生成までをループの中に含めることができます。

# 背景と発端

この手法は、AIコーディングツール Claude Code を核にしたコミュニティで広まりました。オーストラリアの開発者 Geoffrey Huntley さんは「Ralphは単なる Bash ループだ」と言及しています。その本質という意味では確かにそうで、イメージとしては

```
while :; do
  cat PROMPT.md | claude-code
done
```

のように、条件が満たされるまで同じ指示を繰り返すものです。

名前の由来は、アニメ「ザ・シンプソンズ」のキャラクター、ラルフ・ウィガムです。彼は「Me fail English? That's unpossible!（僕が英語で失敗？そんなの不可能だ！）」と叫びながら挫折せずに試し続ける子で、その姿がAIが懲りずにループし続ける様子に重なります。同キャラクターの「I'm in danger（やばい状況だ）」というミームにも引っかけて、AIがリスクを取りながら突き進む様子も示唆しているということです。（筆者はシンプソンズを知らないのでアレですが）

# 実装パターン

## 大原則：引き継ぐのは「思考」ではなく「成果物の状態」

ラルフ・ループを実装するうえで、まず押さえておくべき大原則があります。

ループの各サイクルで、前のエージェントの実行結果を次のエージェントに渡すわけですが、何を渡すかが重要です。渡してはいけないのは、これまでの会話ログの全文、試行錯誤の過程、失敗した案の詳細です。これらを丸ごと渡すと、コンテキストはすぐに溢れ、次のエージェントは余計な情報に引きずられます。

渡すべきなのは「現在の世界の状態」「まだ達成できていない制約」「次のエージェントが最適な判断をするために必要な最小限の情報」です。一言でいえば、会話（Conversation）を状態（State）に変換して渡すことが本質です。

## パターン１：State Snapshot（状態スナップショット）

引き継ぎ時に「現在のタスク状態」を構造化データに圧縮する方法です。「どう考えたか」ではなく「何が分かっているか」を記述します。

```
{
  "goal": "npm run lint をエラー0で通す",
  "current_status": {
    "lint_errors_remaining": 3,
    "files_modified": [
      "src/utils/date.ts",
      "src/components/Table.vue"
    ]
  },
  "constraints": [
    "既存APIのシグネチャは変更不可",
    "Node 18対応を維持"
  ],
  "known_failures": [
    "eslint --fix だけでは解消しない",
    "date-fns の import 修正が必要"
  ],
  "next_focus": "Table.vue の unused vars"
}
```

自然言語とJSONのハイブリッド形式が最も安定します。これだけで数千トークンの会話ログを数百トークンに圧縮できます。

## パターン２：Task Ledger（タスク台帳方式）

ラルフ・ループでは「終わったタスク」と「未完了タスク」を明確に分離しておくことが重要です。

```
## Completed
- ESLint config の更新
- date.ts の型エラー修正

## In Progress
- Table.vue の unused variable 修正

## Blocked
- eslint-plugin-vue のバージョン不整合（要調査）
```

エージェントは未完了セクションだけを読み、完了済みの内容は再検討しません。人間のプロジェクト管理をそのままAIに適用する発想です。完了済みのタスクをエージェントが蒸し返して「暴走」するのを防ぐ効果もあります。

## パターン３：Decision Digest（意思決定要約）

マルチエージェント間で最もコストが高いのは「なぜその方針になったのか」を再推論することです。試行錯誤の過程を渡してしまうと、次のエージェントは同じ推論をゼロからやり直します。

対策として、引き継ぎの直前に専用のサマライザエージェントを1回だけ走らせます。

```
以下の作業ログから、今後の作業に必要な
「重要な意思決定とその理由」だけを箇条書きで抽出せよ。
試行錯誤や失敗例は含めるな。
```

出力は次のようなものになります。

```
- ESLint rule は無効化せず、コード修正で対応する方針
  - 理由：CIで strict lint が必須
- date-fns は v2 API に統一
  - 理由：v1/v2混在で型不整合が発生
```

Chain-of-Thought（思考の連鎖）を捨てて、Conclusion-of-Thought（結論だけ）を残す設計です。

## パターン４：Artifact-first 引き継ぎ

実は最も強力なのがこのパターンです。発想の転換はシンプルで、「コンテキストを渡す」のではなく「成果物を読ませる」というものです。

コードであれば `Git working tree`、調査結果であれば `DECISIONS.md`、現在の状態であれば `STATE.json` というように、引き継ぎ情報をファイルとして外部化しておきます。次のエージェントには「このリポジトリの状態を前提に作業せよ。`STATE.json` を現在状態として信頼し、`DECISIONS.md` の方針は再検討しないこと」と伝えるだけです。

LLMの強みは「読む力」にあります。気まぐれな会話ログより、整理されたファイルを正（真実の源泉）にするほうが、実際には圧倒的に安定します。

## アンチパターン

反対に避けるべきことも明確です。会話履歴を丸ごと渡すこと、前の会話を覚えている前提で指示すること、reasoning を残しすぎてコンテキストウィンドウを溢れさせること、そして状態の圧縮を人間が毎回手動でやること（自動化しないこと）——これらはいずれも、ループが長くなるほど破綻していきます。

あと、当然ですがラルフ・ループは**とにかく処理時間が掛かる**ので、そこはご注意ください。私もラルフ・ループの記事を見て興奮して「自分でも作ってみよう」と無理やりユースケースをひねり出して試作品を作りました。画面を持ったオンライン処理で実現したので、ボタンを押してリクエストしてから結果が画面表示されるまで非常に時間が掛かりました。待ち時間にユーザを心配させないようStreaming処理で中間段階の状況を画面表示していたのですが、しかしそれにしても長くて我ながらちょっと使う気になりませんでした。オンライン・リアルタイムを前提とした処理方式ではなく、バックグラウンドでのバッチ処理で使う前提で考えたほうがよいです。
