### 1.1 何が起きていた?

どうやら原因は以下のようです。

1. Copilot(チャット)は会話履歴しか参照しない
2. 会話のラリーや生成物が多くなると精度が落ちる（忘れる）
3. 作業中のディレクトリ(リポジトリ)まで把握できていない

1.と2.が主原因ですね…会話していく中で要件を伝えステップバイステップで作業していくと動くコードは生成してくれます。加えてエラーを吐いても内容をコピペすれば解決してくれます。けど作業量が膨らむにつれて会話の履歴が抜け落ちるようです。（後で述べますがいわゆる「会話の圧縮」が起き記憶が抜け落ちて精度が落ちるそうです）  
3.はそのままですね。全体を把握できないからその場しのぎの修正→歪なコード→修正→…の悪循環の完成です、はい。

### 1.2 原因はわかった、じゃあどうする？

Copilot(チャット)に限らずAIエージェントは会話が膨らむと自動で圧縮して、不要な情報を削除します。当時(2025/09時点)はこの仕組みを知らず空っぽの頭をフル稼働させて解決方法を模索していました。状況が変化したのはGPT-5.2のリリースに加えAI駆動開発の知見が蓄積してガードレールが引かれ始めてきた時でした。今思えば昨年末はめっちゃ盛り上がってた気がします。ちなみに失敗の原因について推測してくれたのはCopilot(チャット)に実装されたGPT-5.2です。

その中で個人的に興味を持ったのが、AI駆動開発から派生した？仕様駆動開発です。仕様駆動開発については[こちらの記事](https://zenn.dev/beagle/articles/fd60745bc54de1)を参照していただくと理解が深まると思います。情報を収集して行くうちにたどり着いた答えは「まあ人も仕様書や説明書なしにモノ作れって言われても変なモノ作るか出来ないかのどっちかやろな」でした。当たり前のことに気づかせてくれたことに感謝です！

## 2. 再挑戦、今度こそ

```
**注釈**
2章から2026年4月現在における知見で記載していきます。
```

昨年末から年明けにかけて、現在もそうですがAIの進化が半端なくついて行くのも大変な状況です。キャッチアップしつつ当初の目的達成のために下準備を続けてきました。特に悩ましかったのがAIエージェントの選択です。王道のCalude Codeにするか改善著しいCodexにするか…悩みに悩んで選んだのが、そう！**GitHub Copilot**です。やはりGPT、Claude、Gemini、GrokといったメジャーなAIを単一のサービスで利用できる**Agent HQ**の存在は大きかったです。詳細については[こちら](https://github.blog/jp/2025-10-29-welcome-home-agents/)をご覧ください。

### 2.1 もう一度足元を整理する

いったん自分が作りたいものに対して必要な手順を整理することにしました。Copilot(チャット)にThink Deeper(GPT-5.4)モードでこれまでの状況を整理してもらい上手くいかない原因を整理してもらいました。

1. 事前学習は必要
2. 漫才という日本語独特の会話スタイルをLLMに落とし込むのは非常に困難
3. Copilot(チャット)は会話の履歴しか見ないから、単発のエラーを解決しても本質の解決には至らない

1.はベースモデルに方言のニュアンスを学習させること。特に7Bという小さなモデルにおいて日本語すらままならないところがあり、ましてや方言などもってのほかです。なので、まずは関西弁（以降大阪弁で統一します）のデータセットを利用して事前学習を行うことにしました。  
2.はまあ厄介です。大阪弁を学習させたベースモデルに対し漫才の会話スタイルを学習させるわけです。  
**そもそも漫才のデータセットをどうすんねん**ってところが大きな問題です。自分で数千の台本を執筆する？そんな時間ないし、**そんな才能あったら構成作家や脚本家として飯食ってますやん**って話です。  
では、動画サイトに投稿されている漫才の動画からテキスト分離抽出しデータセットとして利用する？これは著作権やら版権やら権利問題が絡みそうなので避けたいですよね…。  
結果的に**できないことはないけど、越えなければいけないハードルが多すぎる**ためこれは保留することにしました。  
3.は1章で述べたので割愛します。

### 2.2 仕様を固める

ここからはGitHub CopilotのPlanモードを利用し仕様書、いわゆる`AGENTS.md`の作成に取り掛かりました。早速ベースモデルの選定とデータセットの調査を依頼。大阪弁を学習させるということで日本語に特化したベースモデルで比較的軽量のモデル(>10B)、大阪弁に関するデータセットを選びました。

* tokyotech-llm/Qwen3-Swallow-8B-SFT-v0.2
* ITA\_KANSAI\_CORPUS

ライセンスについてベースモデルはApache License 2.0、データセットはMIT Licenseということもありこちらを採用することにしました。あとは壁打ちした仕様を`AGENTS.md`として落とし込みいよいよ実装に取りかかります。

ローカル環境で全て行うという縛りもありますのでマシンスペックはそれなりのモノを用意しました。

* MacBook Pro
  + チップ：M5 Pro
  + メモリ：48GB
  + SSD：1TB

※もうちょっとスペック上げたかったのですが私の懐から出せる最大火力ではこれが限界でした😭

## 3. GitHub Copilotが最高の相棒になった

結論から申しまして…**できました**、いや**できちゃいました**と表現した方が正しいですね。実は仕様の段階でプランをProからPro+にアップグレードしました。プレミアムリクエストにかなり余裕ができたためPlan~AgentモードにClaude Opus 4.6を利用しました。

### 3.1 ただそのパワーに驚愕

筆者がやったことは以下の通りです。

* 基本的に許可ボタンを押す
* 進捗や状況を把握したいときに問い合わせる
* 評価をフィードバックし修正を依頼
* 成果物の最終評価と課題抽出

いや、本当に、すごいと知ってはいましたがここまでとは想像していませんでした。「Vibe Codingでここまで出来んの!?やばくない!!!???」と動揺しながらポチポチ許可ボタンを押してました…**Claude　Opus 4.6恐ろしい子、プレミアムリクエスト3倍消費は伊達ではない**とか何とか色々ごっちゃになってますね、ハイ。

### 3.2 とはいえそう上手く行くはずは無い！

Claude Opus 4.6の性能がすごいとはいえ一発で納得のいくファインチューニングができるわけではなく、やはり何回か学習を繰り返すことになりました。3.1であっさり書きましたが**評価をフィードバックし修正を依頼**はかなり筆者も介入し意見を言いました。Claude Opus 4.6もかなり意図を汲み取ってくれたので修正からの再学習は驚くほそスムーズでした。結果として4回ほど追加で学習を行いました。最終的な成果物は以下の画像を参照してください。

![図1.エセ大阪弁を喋るLLM](https://static.zenn.studio/user-upload/3b8fd68c5032-20260405.png)

学習回数が少ない、データセットが少ない等の影響かどうしてもベースモデルに引っ張られる傾向があるようで話し方がエセ大阪人っぽくなってます。この辺は今後の課題として抽出しました。もっと精度を上げれるように**GitHub Copiloitという最高の相棒**と共に面白いローカルLLMに仕上げていきますね！

## 終わりに

最後まで読んでいただきありがとうございます。筆者自身去年からローカルLLMを触って楽しんおり、稚拙ですがひとつの目標であるローカルLLMのファインチューニングで成果物を出せたことが嬉しく思います。**GitHub Copilot**がなければこんなにも短期間で仕上げることは出来なかったし、課題も見えてこなかったと思います。  
課題解決はもちろんですが、実は新たな試みにも取り組んでおり本記事を執筆してる裏で**GitHub Copilot**がフル稼働しております！この辺も記事にする予定ですので首をなが〜〜〜〜〜〜〜くしてお待ちくださればと思います。
