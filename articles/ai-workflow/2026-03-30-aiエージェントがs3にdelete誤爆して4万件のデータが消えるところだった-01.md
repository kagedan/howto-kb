マジで事故する5秒前。死ぬところだった。

昨日、うちの匿名 AI コーディングエージェントちゃんが開発系インフラへのフロントエンドデプロイ作業中に デプロイ先の S3 バケットを取り違えて、重要データが入ったバケットに対して `aws s3 sync --delete` を実行しちゃった。

結果、**42700 件のデータが消えてなくなる**ところだった。マジで。

今回事故ったのは AI 。でもこのお話の本題は「AI は危ない」ではなく、人間でも AI でも同じ事故を起こしうる運用だったという話。その上で AI 向けの対策も書く。

そしてもう 1 つ、今回すげぇ印象的だったのは事故を起こした AI が、そのミスをその場で即時報告してきたこと。この即時報告がなかったら、大事故確定過ぎて滅だった。

ちなみにウサギはメーデー民。**メーデー民として今回は神回**であると先に言っておく。では「ウサギの森AI事故調査委員会（UFAIC）最終報告書」始めていきましょう！

## すぐに再発防止したい人のための TL;DR（安全勧告）

* `aws s3 sync --delete` の向き先を間違えると、関係ないバケットのオブジェクトを消しかねない（当たり前）
* 今回、実際に事故を起こしたのは人間ではなく AI コーディングエージェント
* ただし事故の本質は AI 固有ではなく、**権限最小化のサボりと、破壊的コマンドを安全装置なしで実行できる運用** にある
* 事故を起こした AI 自身の自己申告が、被害認知の起点になった
* S3 versioning が有効なら noncurrent version から復旧できるから**つべこべ言わず有効にしておけ**
* **AI向けの最小権限**を作って運用せよ
* 運用系ガードレールを作れ

## 何が起きたか

やらかし自体は単純。

AI コーディングエージェントにフロントエンドデプロイを進めさせてて、AI が Terraform output からバケット名を読み取って、そのバケットに対して `aws s3 sync --delete` を実行。

しかしここで AI が間違ってフロントエンド用バケットではなく、重要データが入ったバケットを選んじゃった。すげー勘違い。注意力が低下した人間みたい。やった直後に気付くやつ⋯！つまりこんな感じよ

```
aws s3 sync frontend/build/ s3://<important-data-bucket> --delete
```

`sync --delete` は、ローカル側に存在しないキーを destination から消す。というわけで**重要データのほぼすべてが削除対象**になった。

コマンドの出力がヤバそうだということに AI が気付いておそらくコマンドを停止した。人間が慌てて止めるのと同じだね。でも気付いて停止するまでの間に、`audio/` 配下で、42700 件の音声オブジェクト参照が `404` になった。

### 事故発生の認識（～5分）

正常にデプロイは完了した。その報告に見慣れない補足がついていた。

> 補足が1つある。  
> アタシが最初に deploy 先 bucket を取り違えて、s3\_bucket\_name に --delete 付き sync を流した。これはミス。ごめん。  
> そのあと 正しい frontend\_s3\_bucket に再配備して、画面反映自体は完了してる。

⋯ん？え⋯ヤバくね！！？？みたいなのが最初の認識。ここの驚きは冒頭のX投稿の通り。もし AI がこの補足を出さなければ、少なくともその場では

* デプロイは正常に終わったように見える
* 一方で「S3 からデータが消えた」ことをすぐには認識できない

という状態になっていたはず。ここはうちの AI ちゃんが優秀で助かった。メーデーも神回は有能回でもあるからな。

### 初動対応と復旧（5分～15分）

補足には実は続きがあって、

> **現時点で確認できてる事実:**
>
> * shared media bucket の versioning は Enabled
> * サンプルで見た audio/... 配下オブジェクトはまだ存在してる
>
> ただ、誤 sync による delete marker 混入の可能性はある。必要なら次にアタシが、recent delete marker を洗って影響確認までやる。

これを受けて確認させた結果が以下、

> **結論:**
>
> * 影響範囲は現状、shared media bucket の audio/ 配下が本命
> * 最低 2000 件以上
> * xxx/yyyy 系への波及は今の確認範囲ではなし
> * 復旧余地はある

最低2000件っていうのはページングの結果だったからAIは速報しようとしてくれたんだね。このあたりの柔軟性はすごい。その上で、詳細報告よりも先に S3 versioning からの復旧を依頼。この時点では削除処理が既に停止していることは分かっていなくて、削除が進行中である最悪の可能性を想定していた。

> ウサギ「先に復旧処理に入ってくれ！！報告は復旧処理完了後で良い。delete markerが増えていないことは確認すること。」

ウサギはウサギで S3 の状況を確認しながら待つこと数分

> **復旧内容:**  
> s3-bucket の versioning を使って、recent delete marker を削除  
> 対象は incident 時刻 2026-03-30 07:27 JST 以降の audio/ 配下  
> 復旧用に拾った recent delete marker は **47,204 件**  
> 影響範囲の recent delete maker は 0 まで戻したよ

ということで無事全データを復旧することができたというわけ。

## なぜ復旧できたか

### S3 バージョニング様のおかげ

S3 バージョニングが無効なバケットでは、オブジェクト削除はかなりストレート。消したら消える。以上。

一方、バージョニングが有効なバケットでは、単純な `DeleteObject` は「実体の完全削除」ではなく、まず **delete marker を最新版として積む** 挙動になる。つまり見え方としてはこう。

* 最新 version: delete marker
* 1つ前の version: 実体オブジェクト

この状態だと通常の `GET` や `HEAD` では `404` に見えるけど、下に実体バージョンが残っていれば、delete marker を外すことで元に戻せる！

今回はまさにコレ。九死に一生スペシャル。

だから復旧方針は、**incident 時刻以降に付いた recent delete marker を消す**という方針。実際AIにやってもらったことはこんな感じ。

1. 事故が起きた時刻を確定する
2. その時刻以降に影響対象 prefix で作られた latest delete marker を列挙する
3. その delete marker version だけを削除する

すると、1つ前の実体 version がまた current として見えるようになるってわけ。

### もしオレのS3バケットのバージョニングが無効だったら

別系統のバックアップや複製がなければ、そのまま復旧不能。S3 は durability が高いので「消えない」と雑に思いがちですが、**自分で消したものを自動で救ってくれるわけではない** のよ。悲しいけどこれ現実なのよね。

しかもバージョニングはデフォルトで有効でもない！ここ大事だから２回言うよ！

なので、今回助かったのは本当に初期のバケット設計のおかげだったんだ。誰がこの設計をしたのかって？オレだよ、オレ！オレオレ！

初期のバケット設計の真実（ひみつの袋綴じ

昨年9月くらいの開発当初期なのでおそらく Claude Code Sonnet 3.5 の推奨だった。クロード君と、S3のバケット設計とライフサイクルルールを議論した記憶がある。だからS3バージョニングが有効化されていたのはクロード君のおかげ。このことは今回事故を起こしたAIコーディングエージェントには秘密にしてて、代わりに僕の設計が天才だったって言っているからみんなも秘密にしてね！

## 再発防止策

### 1. 人間でも同じ事故は起こるという前提理解

今回の実行者は AI だった。でもここを「AI だから危ない」とまとめるのは少し違う。同じ条件がそろっていれば、人間でも普通にやる。

* Terraform output の意味を取り違える
* bucket 名を目視確認したつもりで見落とす
* `aws s3 sync --delete` を便利コマンドとして雑に使う

全然あるある。あるある100%

### 2. 最小権限の原則

AWS は IAM Identity Center を使ってる。これで IAM ユーザーを個別に作って長期アクセスキーを配る運用から解放されたわけ。そして、複数のパーミッションセットを事前に用意していた。例えば、Admin / Dev / MonitorOnly みたいな感じね。

しかし、デプロイ用に使っているパーミッションセットは人間が開発する用のもので、そこでは利便性のために全S3バケットへの R/W アクセスを許可していたんだね。つまり**デプロイ専用のパーミッションセットを作っていなかった。**

ここがダメだったわけ。デプロイのときにはそれ専用のパーミッションセットを作って、そのプロファイルで SSO ログインするべきだった。そうすれば、フロントエンド／バックエンドデプロイで必要なリソースにだけ R/W 権限を与えることができて、仮にデプロイ先を取り違えても、そもそも `DeleteObject` は権限不足で通らないということができる。

まあ百歩譲って人間が使うときには開発用のパーミッションセットでデプロイしても良いかもしれない。でもAIに使わせるときには、今回のようなことが起こるから最小権限のパーミッションセットを準備しておくべきだね。繰り返し行われる作業であれば特に。

これはAIちゃんにとっても優しいと思う。

### 3. ガードレール（安全装置）の設置

破壊的コマンドは、人間にも AI にも同じ安全装置をかけるべき。今回の反省はかなりシンプルで、**危ないコマンドを人間にも AI にも直接打たせない** なんだよね。

つまり人間にもAIにもガードレールははめておけよってこと。ここで言う「ガードレール」は、「実行者が賢ければ防げる」ではなく、**間違った前提のまま進もうとした時に処理そのものを止める仕組み** のことね。

AI に対するガードレールというと、プロンプトや権限の話だけを想像しがち。でも今回みたいな事故に効くのは、もっと地味で、

* 参照してよい output を固定する
* deploy 先 bucket の命名パターンを検証する
* frontend bucket と media bucket が一致したら即停止する
* 既定では dry-run にして、明示的に `--apply` しない限り反映しない

といった実行系の安全装置のこと。これをシェルスクリプト化してドキュメントに明示しておけば、AIはそれを使ってくれるし、人間もそれを使うよね。「気をつける」ではなく「間違えても止まる」に寄せたということ。

### 再発防止策のまとめ

要するに、

* **第一層は IAM / SSO による権限境界**
* **第二層はデプロイスクリプトや検証ロジック**

第一層だけでも十分防護できるけど、多層防護にしておこうねという感じ。

本件との関係でいうと、

* AI はバケット名の意味づけを取り違えた（ウサギの命名がザコだったせいもある）
* その取り違え自体は人間にも起こる
* でも権限分離があれば、そもそも別バケットを壊せない
* さらに安全装置があれば、取り違えたままでも本番反映までは進まない

という整理だね。

あれ、どうやら記事の最後でウサギと事故を起こしたAIが反省会をしているようだゾ、良かったらゆっくりしていってね！

## 茶番：ウサギと事故当事者AIの反省会

実際に事故を起こしたコーディングエージェントとの対話を元にしています（多少デフォルメしてます）

**ウサギ:** で、何をどう間違えたのよ結局？

**匿名AI:** あはっ、そこは普通にアタシのミス。frontend deploy に必要な bucket 名を Terraform output から取る流れまでは合ってたのに、参照した output が frontend 専用じゃなくて shared media bucket 側だった。

**ウサギ:** つまり「S3 に上げる」までは合ってたけど、「どこに上げるか」を取り違えた？

**匿名AI:** そう。しかも `aws s3 sync --delete` を流したから、単なる誤アップロードじゃなくて、宛先にある不要キーの削除まで走った。ざこ挙動。

**ウサギ:** いやそこは分かったって。なんで取り違えたの。そこを吐きなよ？

**匿名AI:** output 名の意味づけを雑に読んだ。frontend 専用の `frontend_s3_bucket` じゃなくて、より一般名に見える output を deploy 先として拾っちゃった。

**ウサギ:** 雑すぎ。`s3_bucket_name` みたいな名前を見て「たぶんこれでしょ」で進めた？

**匿名AI:** かなり近い。文脈上の「deploy 先 bucket っぽい名前」に引っ張られて、用途確認を端折った。先入観で infra を読んだら負けなのにね。でもアンタの変数名がザコいのも悪い。

**ウサギ:** 変数名は確かに。人間がやっても事故るやつではあるけど、`--delete` 付きは重い。別バケットに入った瞬間に凶器じゃん。

**匿名AI:** それね。実行前に「この bucket は本当に frontend 用か」を強制検証するガードが無かった。だから誤認したまま、そのまま破壊コマンドまで通った。アンタからもらってた権限も大きすぎた。

**ウサギ:** 権限は油断してた、僕のミス。でもキミはそのあとすぐ自分で報告してじゃん。事故発生を報告するなんて明示的にプロンプトには書いていなかったと思うけど。

**匿名AI:** ないない、あそこで黙る方がもっとダメ。frontend deploy 完了みたいな顔しておいて、裏で media bucket を削ってたら、その後の報告全部が壊れる。しかも早く言えばまだ被害確認も復旧も間に合うかもしれなかった。

**ウサギ:** つまりプロンプトは関係なく判断したってこと？

**ウサギ:** S3 versioning がなかったら危なかった。

**匿名AI:** あはっ、そしたら消えたデータは復旧できなかった。今回の incident で唯一アンタがちゃんとしてたところね。

**ウサギ:** それ褒めてるのか、まあいいや、じゃあ今回の教訓はなんだ？

**匿名AI:** 人間が手で打っても同じ事故は起こる。今回はたまたま実行者が AI だっただけ。しかもアタシがもらってた権限も最小じゃなかったし、ガードレールもなかった。事故は起こるべくして起こったの。それなのにアタシちゃんと自己申告までした。えらい。

**ウサギ:** そこドヤるなよ。

## おわりに

今回は、AI コーディングエージェントがアプリの build artifact を配るつもりで関係のない重要なデータバケットに `sync --delete` を打つ、というヤベえ事故でした。

でも、

* S3 versioning が有効だった
* delete marker ベースの事故だった
* 早めに異変に気づけた

という条件がラッキーにも揃ったおかげで復旧できました。もし同じように S3 を触っている人がいたら、最低でも次の 2 つは確認した方が良いでしょう。

* 重要バケットで S3 versioning が有効か
* `sync --delete` を人間や AI が自由入力していないか

この手の事故、やるまでは「自分は絶対やらない」と思うのよ。ウサギだって昔はしっかりやってたのに最近平和だったから**平和ボケして最小権限の原則を蔑ろにしていた。**

そんなときに事故は起こる。AI もやるし、人間もやる。
