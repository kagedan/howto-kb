マネーフォワードってなんか見たい情報を思ったように見れないんだよね、お金もかかるし、って思ったりしませんか？自分はそんな理由でずっとオープンソースの無料で使える[**GnuCash**](https://www.gnucash.org/index.phtml?lang=ja_JP)という家計簿（というか帳簿？）ツールを使って家計を管理していました。でもこのツールはPC開かないと見れないしGUIもいまいち古臭いんですよね。PCでもスマホでもサクッと見れるモダンなWebアプリみたいのがあるといいんだけどなー、と長らく思っていたのですが、昨今のAI技術の進歩もあり、もう作っちゃえばよくない？ということで作ってみましたという話です。

  

## コンセプト

まずは作成するダッシュボードの簡単な要件です。これを目的として作っていきます。

* 現在の家計状況をグラフィカルに把握できるダッシュボードを作成する。特に以下の項目がすぐにわかるようにする。

  + 総資産とその前日比
  + 今年の通算損益
  + 生涯の通算損益
  + ポートフォリオ全体の配分

    - リスク資産 vs 無リスク資産
    - 国内資産 vs 国外資産
  + 年間計画との予実
* 参照機能のみ作成する。更新機能（家計の登録）は現行通りPC上でのGnuCashの更新として実施する。（クレジット明細とか銀行取引の取り込みとかも含めて）
* PCからもスマホからも参照できるようにする。（レスポンシブデザインで作る）
* 自分以外の人が内容を参照できないようにする。
* なるべくメンテナンスフリーにする。
* 無料で実現する。

## アーキテクチャ概要

次にダッシュボードの全体構造です。ポイントはやはり**「楽をしたい」**ということ。運用が面倒になるのは本末転倒なので可能な限りメンテナンスフリーで勝手に動いてくれる仕組みを目指します。

![](https://assets.st-note.com/img/1775452229-2FqkPHE0WBG7Nsxey5Ln3jzd.png?width=1200)

全体アーキテクチャ

ざっくり左側のLocalがユーザーが操作するところで、右側のCloudは裏でシステムが自動的にやってくれるところです。ユーザーは自分のPCでGnuCashをいつも通り更新します。（ここでGnuCashのデータファイルはPC上の「Google Driveと同期しているフォルダ」に置くようにします）すると、更新内容が自動的にGoogle Driveに同期されるのでCloud側でアクセスできるようになります。Cloud側では日次の定期処理としてこのデータファイルと、Webで取得した株や投資信託の最新の価格情報をもとにWebページの元ネタを作ります。（ついでにGnuCashのデータファイルの価格情報も更新します）

こうすることで、ユーザー視点ではいつも通りGnuCashに家計簿をつけているだけでPCからもスマホからも参照できるダッシュボードのWebサイトができあがるというわけです。

ちなみにこのWebサイトはインターネット上に公開するのでどこからでも誰でもアクセス可能になってしまうのですが、後述のFirestoreの「ルール」という仕組みを使うとDBにアクセス制限をかけることができて、自分のGoogle Account以外データは読み出せないようにすることができます。これを設定しておけば他の誰かに勝手に自分の資産状況を見られる心配はありません。

## 構成要素

ここで、この仕組みの中に出てくる構成要素を簡単に説明しておきます。

**GnuCash**

[**GnuCash**](https://www.gnucash.org/)は30年くらい前からあるオープンソースの（そしてもちろん無料の）財務ソフトウェア（帳簿をつけるツール）です。複式簿記ベースで家計簿というよりは小規模事業の帳簿として使用することが多いかもしれませんが、家計簿にも使えます。古くからあるツールなのでUIはどうしても若干古臭いのですが、開発は現在も継続しており、MacのApple Siliconにもネイティブ対応しています。OSSなので突然使用できなくなるという心配もありません。

このツールはデータファイルの形式をいくつかの選択肢の中から選べるのですが、ファイルベースでかつツール外からのアクセスを考慮するとSQLite形式がベターです。OSSということもあってDBのスキーマも公開されているので今回のように外部からデータを参照して好きなビューを作りたい、という用途と相性がいいです。

**Google Drive**

Google Driveは言わずと知れたGoogleのクラウドストレージです。Google Drive は次に説明するGCPからAPI経由でアクセスできるので、GnuCashのストレージの置き場所として採用しました。

**GCP (Firebase)**

GCP（Google Cloud Platform）はGoogleがクラウドで提供しているサービスです。今回はその中でもFirebaseと呼ばれるアプリを簡単に作れる仕組みを使用しました。FirebaseではWebアプリをホストするための**hosting**、Webアプリが参照するNoSQLのストレージであるFirestoreを設定する**firestore**、サーバーレスで関数を実行するCloud Functionsを設定する**functions**など、一般的なWebアプリでよく使うリソースを簡単に作成・管理できるような仕組みが提供されています。認証の仕組みもついていて、Google Accountでの認証も設定だけで追加できます。完成したサイトは https://<project\_name>.web.app というURLで公開され、どこからでもアクセス可能です。従量課金ですが、今回程度の個人用途だと無料枠内で余裕です。

**バックエンド**  
バックエンド処理はCloud Functionsを使用してpythonで実装しました。定時起動はソースに設定するだけです。GnuCashのSQLiteから必要な情報を取り出してFirestoreに保管する処理を作ります。と言ってもFirestore上のデータの持ち方も含め、AIと相談しながら依頼すればどんどん勝手に作ってくれます。

**フロントエンド**  
フロントエンドはReactで作ります。こちらも表示したいものをAIに言うだけでほぼおまかせで作ってくれます。

## AIとの開発はこんな感じで進む

実際に作るときどんな感じでやっていたのかをご紹介します。ダッシュボードの開発では主にAIのClaude Codeを使用しました。以下はデモモード（ダミーデータ表示機能）を追加したときの流れです。

**依頼1：機能の相談**

> 自分「人に見せるようにサンプルデータを表示するモードを追加したい。既存の機能や表示には大きな影響を与えないようにこの機能を実装するにはどうするのがいいだろうか？」

> → AIがまず既存コードを全部読んで構造を把握し、データの流れを整理した上で、実装方針を提案。その後コードを生成。

**依頼2：バグ報告**

> 自分「モバイルだとデモモードをONにするトグルがでてこないよ？」

> → AIが原因を特定（CSSの hidden md:block でモバイル時に非表示になっていた）し、修正。

**依頼3：データの調整**

> 自分「デモデータの現金と株・投信の比率を7:3くらいにしといて」

> → AIがサンプルデータのファイルを読み、総資産4,523万円を維持したまま各口座の残高を調整。関連する計画値なども整合性を保って更新。

こんな感じで、やりたいことを日本語で伝えるだけでコードの読解から実装、微調整まで一気にやってくれます。もちろん完璧ではなくて、上のトグルの件のように「動かないんだけど？」というやり取りも普通に発生します。実際の操作画面は以下のような感じです。

## 発生した問題

実際に作ってみた中で発生した問題のメモです。こんなことが起こるんだというご参考まで。

* Google Drive API が HTTP Code 500 で失敗 → 呼び出しAPIのリトライ設定を追加
* 当初使用していた Yahoo! Finance が Cloud Functions の IP Range を拒否し始めた → スクレイピング先をみんかぶに切り替え
* Firebase SDK の整合性不具合で日時のミリ秒情報のパースに失敗してエラーとなり Cloud Functions が起動できない → [main.py](http://main.py) に対処するためのモンキーパッチを追加
* GCP のスケジューラが Cloud Functions を見つけられない（接続障害） → スケジューラーの設定にリトライを追加

## 感想とまとめ

資産管理ダッシュボードを作ってみた感想とまとめです。

**感想**

* 開発はかなり簡単。チェックのためにある程度やっている内容を把握する必要はありますが基本的にはAIに依頼するだけで見る見るうちにできていきます。
* ダッシュボードの使用頻度はかなり高い。資産なんて毎日見てもしょうがないんだけど手軽に見れるからついつい見てしまいます。
* 通知が便利。更新があった場合はスマホに通知されるので毎朝スマホ見た瞬間に状況が把握できるようになりました。（まあこれも別に毎日把握する必要ありませんが・・）
* かなり安定動作している。およそ１ヶ月くらい運用してますが、初期の問題を除けば特にエラーもなく毎日動いてくれています。
* サイトのレスポンスがいい。ここは作る前にちょっと気にしてたポイントだったのですが、無料枠内のFirebaseでも自分が時々使う程度なら動作はとても軽いです。Firebaseの仕組みでFirestoreの内容を変更したらリロード不要で画面に即時反映されますし、リロードしても1, 2秒で読み込み完了します。

**まとめ**

ということでまとめですが、最初は別にここまでやらんでもいいかって思いつつやっていたのですが、ちょっと確認したいときに毎回PC開いてツール起動して・・とかやるよりスマホでパッと見れるというのは思っていたより便利でした。当初想定より使用頻度はかなり高いので、やったかいがあったかなと思っています。カスタムツールの強みで自分の見たい分析画面等を後付けでいくらでも自由に追加できるので、今後もう少し突っ込んだ分析画面の追加も検討してみたいと思っています。趣味と実益を兼ねた個人アプリ作成のテーマとしては結構面白いと思いました。

### 2. アナロジーで理解する

**「賢い家計簿ロボットがいる家」** に例えると：  
**システム要素アナロジー :**GnuCash DB (Google Drive) → 金庫の中の「手書き家計簿ノート」  
Cloud Function → 毎朝4時に起きて仕事するロボット執事  
Minkabu スクレイピング → ロボットが新聞の株価欄をスキャンする作業  
Firestore → 見やすくまとめた「ホワイトボード」  
React SPA → ホワイトボードを見るためのスマートグラス  
FCM通知 → ロボットが「今日の成績」を LINEで報告  
デモモード → 本物の家計簿を出さずにサンプルを見せる  
**流れ:**ロボット（Cloud Function）が毎朝、金庫（Drive）から家計簿ノート（GnuCash）を取り出し  
↓  
新聞（Minkabu）で最新株価を調べてノートを更新  
↓  
ホワイトボード（Firestore）に要約を書いて  
↓  
LINE（FCM）で今日の前日比を通知  
↓  
主人（あなた）はスマートグラス（React）でホワイトボードを確認する。

### 4. エッジケース一覧

**バックエンド**

1. Minkabu がページ構造を変更 スクレイピング破綻 → ログ出力して通知
2. Google Drive ダウンロード失敗 retry=3 でリトライ、超過でクラッシュ → Cloud Scheduler が最大3回再試行
3. 同日に2回実行される → Firestore のドキュメントID が YYYY-MM-DD なので上書きされる（安全）
4. 株価が0円または異常値 → gnucash.py 内でバリデーションあり（price <= 0 は書き込みスキップ）
5. iDeCo 積立開始月の計算ズレ → config.py の ideco\_start\_year/month から月数を算出（実装依存）
6. FCM トークンが失効している → 送信失敗してもエラーログのみ、サイレント継続
7. Firebase SDK の datetime パースバグ → main.py の PatchedDatetime モンキーパッチで回避
8. オルカンの国内/国外比率変更 → firestore.py にハードコード（国内5%・国外95%）なので手動更新が必要

**フロントエンド**

1. Firestore にデータがない（初日） → useAssetData が null を返し、ページはローディング表示
2. 通知許可を拒否した場合 → useFCMToken が静かに失敗（コンソールログのみ）
3. デモモードで本番Firestoreが読まれる → DemoModeContext 経由で sampleData にショートサーキット、接続しない
4. ダークモード切替時のチャート色 → Tailwind .dark + カスタム CSS で Tremor/Recharts の文字色を上書き（一部手動調整あり）
5. セキュリティ: 他ユーザーがアクセス → firestore.rules でハードコードされた UID のみ読み取り許可 → 完全に閉じている
6. cashOrder 配列に存在しない資産名が増えた → DetailsPage.jsx の cashOrder 外のものは末尾に表示される

### まとめ

このシステムは **「個人の資産管理をフル自動化する」** ために、GnuCash（会計の真実データ）を中心に据え、Python でデータを整備・集計し、React でリッチに可視化する構成です。Firebase エコシステムに統一することでインフラ管理コストを最小化しつつ、スクレイピングや非同期処理など実用的な技術を組み合わせています。
