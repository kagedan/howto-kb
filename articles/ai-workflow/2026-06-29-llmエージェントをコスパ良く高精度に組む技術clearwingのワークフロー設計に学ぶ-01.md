## はじめに

LLM エージェントを実務に乗せようとすると、必ず 2 つの壁にぶつかります。

* コスト：全部を賢いモデルに投げると破産する。遅い。
* 精度：LLM は平気で「それっぽい嘘」をつく（ハルシネーション）。

しかもこの 2 つはトレードオフに見えます。精度を上げようと巨大モデルで全部処理すれば高コスト、コストを削れば精度が落ちる、という直感です。

ところが、この両立を「ワークフローの組み方」でかなりのところまで解こうとしている実例があります。Lazarus AI の OSS、[Clearwing](https://github.com/Lazarus-AI/clearwing)（LLM ベースの自律型脆弱性ハンター）です。

Clearwing は Anthropic の Project Glasswing に触発されたツールで、「特権的な frontier model だけに頼らず、一般にアクセス可能な LLM でも近い成果を狙える scaffold を作る」ことを目指しています。鍵は、賢いモデルそのものではありません。安い処理と高い処理を非対称に配置し、優先度と証拠レベルで予算を集中投下するワークフロー設計にあります。

本記事は Clearwing の中から 「コスパ × 精度」を両立させているワークフローの工夫だけを抽出し、最後に 「これを参考に自前のエージェントアプリを組むならこうする」 という設計図に落とします。題材はセキュリティですが、抽出する設計パターンは汎用です（ドキュメント処理でもコードレビューでもデータ抽出でも効きます）。

## そもそも Clearwing とは

設計の話に入る前に、Clearwing が何をするツールかを押さえておきます。ひとことで言うと、リポジトリを渡すと LLM エージェントが自律的に脆弱性を探す OSS の CLI です（Lazarus AI 製、Anthropic の Project Glasswing に触発されています）。モードは 2 つあります。

* ネットワークペンテスト（`clearwing scan`）：稼働中のターゲットを調べ、Kali のツールをサンドボックスで使う
* ソースコードハント（`clearwing sourcehunt`）：ソースコードから脆弱性を探す ← 本記事の対象

ソースコードハントの流れを平たく言うと、こうです。

1. リポジトリ（URL かローカルパス）を渡す（例：`clearwing sourcehunt <repo> --depth standard`）
2. ファイルを安い処理でランク付けし、危ない順に並べる
3. ファイルごとにハンターエージェントを並列起動。各エージェントはサンドボックス内でコードを読み・grep し、必要ならコンパイルして実行し、クラッシュを証拠に脆弱性を探す
4. 別の検証エージェントが「本当に脆弱性か」を独立に検証する
5. 結果を SARIF / Markdown / JSON で出力（修正の draft PR まで作れる）

特徴的なのは、特定の API に縛られず Anthropic / OpenAI / ローカルモデルなどを差し替えられること（オンプレ運用も可能）。そして思想が「賢い単一モデルに丸投げ」ではなく、「役割分担したエージェントを安く回し、証拠で詰める」点にあります。本記事はこの仕組みのうち、コスパと精度を支える “ワークフロー設計” に注目します。

## 大方針：安い処理を手前に、高い処理を奥に

まず全体像です。Clearwing のパイプラインは 「漏斗（ファネル）」 になっています。手前は LLM すら使わない安い処理で広く浅く、奥に行くほど高価な処理で狭く深く。

ポイントは、「どこに高い処理を集中させるか」を手前の安い処理で決めていること。

なお SourceHunt は外側だけ見ると staged pipeline ですが、各ファイルに投げられた Hunter の内側では ReAct 的な tool-calling loop が回ります。つまり「repo 全体を一人の LLM に読ませる」のではなく、ファイル単位に fan-out し、それぞれが read / grep / compile / run / record などのツールを使って証拠を集める二重構造です。外側＝段階的なファネル、内側＝ツール駆動のエージェントループ、と捉えると見通しが良くなります。

以下、このファネルを支える「コスパの工夫」と「精度の工夫」に分けて見ます。

## Clearwing のアーキテクチャは何型か（A2A？ マルチエージェント？）

工夫の中身に入る前に、Clearwing の構造を一言で押さえておきます。結論は、A2A でも会話型マルチエージェントでもなく、決定論的にオーケストレーションされた orchestrator-workers 型のワークフローです。

| 用語 | ざっくり | Clearwing は？ |
| --- | --- | --- |
| A2A（Agent2Agent） | 異なるエージェント同士が対話・相互運用するプロトコル | 非対応。内部のエージェントは互いに会話しない |
| MCP（Model Context Protocol） | ツール/コンテキストを外部へ公開・接続する規格 | 対応（`clearwing mcp` で外部 IDE/エージェントへ公開）。ただし“外部公開”であって内部協調ではない |
| 会話型マルチエージェント | 複数エージェントがメッセージを往復し交渉・合意（AutoGen の group chat 等） | 違う |
| orchestrator-workers | 中央が仕事を分配し、結果を集約するワークフロー | これに該当 |

そう言える根拠はコードにあります。

* 段はコードで直列にディスパッチされる（`runner.py` の `arun()`）。LLM が全体の進行を自分で決めるわけではない。
* ステージ間でやり取りされるのは `Finding` という構造化データで、チャットの往復はない。
* 検証者は発見者の推論を意図的に見ない（`verifier.py`）。これは A2A 的な対話の真逆で、独立性のために通信を遮断している。
* ワーカー（hunter）同士は通信しない。共有するのは受動的な findings pool だけ。

ただし二層構造になっていて、内側だけは自律エージェントです。

外側＝決められたワークフロー、内側＝自律エージェント。Anthropic の「効果的なエージェント」のパターン語彙に乗せると、こう分解できます。

| Clearwing の部位 | パターン |
| --- | --- |
| ranker → tier 振り分け | Routing |
| Pool が hunter を並列起動 | Parallelization / Orchestrator-workers |
| Validator / patch oracle の評価・昇格・棄却 | Evaluator-optimizer |
| per-file hunter の tool-calling ループ | Agent（自律） |

では、なぜこの構造なのか。

おそらく最大の理由はコストです。A2A や会話型マルチエージェントは、エージェントが互いの文脈（会話履歴）を共有・保持し続ける必要があり、やり取りのたびにコンテキストが膨らみます。一方オーケストレータ型は、ステージ間で受け渡すのが `Finding` という構造化データだけなので、各エージェントは必要な分しかコンテキストを積みません。

コード全体をハントするとなると、対象は必然的に大規模になります。だからこそ、相互の文脈保持という重いコストを避けられるオーケストレータ型が、少しでも安く回すための現実的な選択になっているのではないか、と考えています。

## コスパの工夫

### 工夫1：LLM を使う前に、安い処理で絞る

Clearwing は最初に、正規表現・AST パターン・ファイルパスなどの安い信号でファイルに「関心事タグ」を付けます（`parser` / `crypto` / `auth_boundary` など）。さらに tree-sitter ベースのコールグラフで到達性を伝播し、外部入力に近いコードほど高く評価します（Semgrep や軽量な taint 解析も併用）。ここまで LLM コストはゼロ。

教訓はシンプルです。「全部をモデルに渡す」をやめる。ルール・ヒューリスティック・既存の静的解析で安く前処理し、モデルに渡す前に母集団を絞る。これだけで支出は桁で変わります。

### 工夫2：重み付けスコアで「優先度」を 1 本に畳む

集めた信号を、ランカーがひとつのスコアに統合します。

```
# priority = surface*0.5 + influence*0.2 + reachability*0.3
```

* surface（そのファイル自体の危なさ, 重み 0.5）
* reachability（攻撃者が到達できるか, 0.3）
* influence（壊れたときの波及の大きさ, 0.2）

巧いのは influence 軸。「自分は地味でも、至る所で使われる型定義・定数のようなファイル（surface=低, influence=高）」を拾い上げます。単体では無害だが壊れると広範囲に効く急所を、重み付けで救済しているわけです。コスパ設計の本質は「どこに金を使うかの優先順位付け」で、それを説明可能な 1 本の数式にしているのが要点です。

### 工夫3：予算を「傾斜配分」し、使い残しを繰り越す

スコアを tier（A/B/C） に量子化し、予算を 70 / 25 / 5 で配分します。しかも A→B→C と直列に実行し、A の使い残しは B へ、B の余りは C へ繰り越す。

「危ない順に、潤沢な予算で当たる。余ったら下位へ回す」。期待値の高いところへ資源を集中する、古典的だが効く戦略です。

### 工夫4：安い対象には「安い道具・安いモデル」を割り当てる

予算だけでなく、使えるツールとモデルも対象の価値で変えます。

* ツールゲーティング：Tier C（低優先）のエージェントは `grep` / `read` / `record` 中心で、実際にビルド・実行する重いツール（`compile` / `run`）は使わせない。
* モデルルーティング：各ステージ（ランカー / ハンター / 検証者 / exploiter）に別々の LLM クライアントを渡せる設計です。README の例でも ranker / hunter / verifier / exploiter に異なるモデルを割り当てており、どのステージにどのモデルを使うかは設定可能（例えば ranker や verifier に mini 級、hunter に強いモデル、といった配分が示されています）。

「賢いモデルとフルパワーのツールは、それに値する対象にだけ」。これがコスパの肝です。

### 工夫5：段階的エスカレーション（「脈あり」だけ深掘り）

最初から全力で当たらず、まず安いバンド（少ないステップ）で探り、手応えがあったものだけ上のバンドへ段階的に昇格させます。順序は fast → standard → deep の 3 段で、各段の通過条件が異なります。

加えて暴走防止も実装されています。予算の 90% で打ち切り、同じツール呼び出しを 3 回超で繰り返したらスキップ、`quick` 深度なら高い処理を全部飛ばして早期終了。「無駄に高い処理へ流れ込ませない」防波堤が随所にあります。

## 精度の工夫

コスパだけなら「安く雑に」でも実現できます。Clearwing が優れているのは、安く回しながら精度も担保している点。その工夫は「LLM の言い分を鵜呑みにしない」一点に集約されます。

### 工夫6：証拠ラダーで「確からしさ」を段階管理する

各 finding には 証拠レベルが付きます。`suspicion`（疑い）→ `static_corroboration`（静的に裏付け）→ `crash_reproduced`（クラッシュ再現）→ `root_cause_explained`（根本原因を説明）→ … → `patch_validated`（修正で消えることまで確認）と、証拠の強さが一列に順序づけられています。

これが効くのは、下流の高価な処理をこのレベルでゲートできるからです。「クラッシュ再現済み以上だけ実際に攻撃が成立するか試す」「根本原因まで説明できたものだけ自動修正する」。確信度の低いものに高い処理を使わない、という発想です。コスパと精度が同じ仕組みで両立する瞬間です。

### 工夫7：独立した「敵対的」検証（推論を見せず、反論を組ませる）

発見したエージェント自身に「合ってる？」と聞いても、LLM は迎合して「合ってます」と言いがちです。そこで Clearwing は別の検証エージェントを立て、2 つの仕掛けで独立性を担保します。

1. 発見者の推論チェーンを渡さない（finding と該当コードだけを見せ、思い込みの伝播を断つ）
2. 「まず最強の反論を構築してから」判定させる（verifier の prompt は pro-argument / counter-argument / tie-breaker を独立コンテキストで組ませる）

そして docs / README で説明される Validator は、判定を 4 つの軸に分解します。REAL（実在するか）、TRIGGERABLE（攻撃者が到達できるか）、IMPACTFUL（意味のある境界を越えるか）、GENERAL（現実的構成で成立するか）の 4 つです。

なぜこの 4 つなのか。LLM に脆弱性を探させると、誤りはだいたい次の 4 パターンに収まります。4 軸は、その 1 つずつを潰すためのチェックリストになっています。

| LLM が吐きがちな「嘘」 | 具体例 | 対応する軸 | 検証で問うこと |
| --- | --- | --- | --- |
| 実在しない | コードを誤読し、ありもしない欠陥を報告する（幻のバグ） | REAL | 記述通りに本当に存在するか。PoC があれば実際に再現せよ |
| 到達できない | 理論上は危ないが、呼び出し元が入力を制約／デッドコード／テスト専用で攻撃者の入力が届かない | TRIGGERABLE | 攻撃者制御の入力が、本番のデプロイでその経路に届くか |
| 影響がない・誇張 | 境界を越えない same-process の問題なのに critical を主張する（重大度の水増し） | IMPACTFUL | 権限・テナント・プロセス・カーネル等の境界を実際に越えるか |
| 特殊構成限定 | exotic なコンパイルフラグや非推奨機能でしか成立しない | GENERAL | 既定構成・一般的なデプロイパターンで成立するか |

「それっぽいが偽」を一括で弾くのではなく、どの観点で偽なのかを分けて検査するからこそ、本物だけが 4 軸すべてを通過できます。なお adversarial な verifier prompt（pro / counter / tie-breaker）と、この 4 軸 schema は、いずれも『独立した立場で反証を試みてから判定する』という同じ思想で接続されています。

### 工夫8：グラウンドトゥルースで殴る ＋ fail-closed

最終的に Clearwing が信じるのは LLM の意見ではなく、機械的に検証可能な事実です。サンドボックスで実際にコンパイル・実行し、サニタイザがクラッシュを吐けばそれは動かぬ証拠（ground truth）。さらに「最小修正を当ててクラッシュが消えるか」で根本原因の理解が正しいかを検証します（patch oracle）。

そして決定的なのが fail-closed。パッチ適用やビルドや実行が途中で失敗したら、保守的に「まだ壊れている＝直っていない」側に倒す。「うっかり OK と誤判定する」ことを構造的に防いでいます。外部の客観的オラクルを持てるなら、LLM の自己申告より必ずそちらを優先する。これが精度設計の王道です。

## なぜ「コスパ」と「精度」が同時に立つのか

ここまでを 1 枚にすると、Clearwing は 2 つのラダー（はしご）が噛み合った構造だと分かります。

* 予算ラダー：fast →（脈あり）→ standard →（強い証拠）→ deep と、高い処理へ進むほど通過条件が厳しい
* 証拠ラダー：suspicion → … → patch\_validated と、証拠が強くなるほど下流の高価な処理を解禁

この 2 つが連動するため、「確信度が低いものに高いコストはかからない」「高いコストは確信度の高いものにしか使われない」が同時に成立します。コスパのための足切りが、そのまま精度のためのゲートになっている。ここが設計の妙です。

## 自前のエージェントアプリを組むなら

ここからが本題です。Clearwing の工夫をセキュリティ以外でも使える汎用パターンとして抽出します。

| # | パターン | コスパ | 精度 |
| --- | --- | --- | --- |
| 1 | LLM の前に安い前処理で母集団を絞る | ◎ | ○ |
| 2 | 重み付けスコアで優先度を 1 本に畳む | ◎ | ○ |
| 3 | 予算の傾斜配分＋使い残し繰り越し | ◎ |  |
| 4 | 対象の価値に応じたツール／モデルの出し分け | ◎ | ○ |
| 5 | 段階的エスカレーション（脈ありだけ深掘り） | ◎ | ○ |
| 6 | 確信度（証拠）ラダーで下流をゲート | ○ | ◎ |
| 7 | 独立・敵対的検証（推論を隠し、反論を組ませる） |  | ◎ |
| 8 | 外部オラクルで殴る ＋ fail-closed |  | ◎ |

これらを組むと、汎用エージェントのリファレンス構成はこうなります。

実行時の流れをシーケンスにするとこうです（例：大量ドキュメントからの情報抽出エージェントを想定）。

### 自前で組むときの勘所

* 「外部オラクル」を何にするかが精度の上限を決める。コードなら型チェッカ・テスト・コンパイラ・実行。データ抽出ならスキーマ検証・既存マスタとの突合・合計値チェック。LLM 以外で正解を機械判定できる手段を 1 つでも持てるかが勝負どころです。Clearwing にとってはそれがサニタイザのクラッシュでした。
* 確信度を「フィールド」として持ち回す。finding/結果オブジェクトに確信度・証拠レベルを持たせ、段ごとに昇格させる設計にすると、コスパ（低信頼に金をかけない）と精度（高信頼だけ通す）が同じ仕組みで両立します。
* 検証は必ず別コンテキストで。発見と検証を同じエージェント・同じ会話でやると迎合が起きます。推論を見せずに反論させるだけで質が変わります。
* fail-closed をデフォルトに。曖昧・エラー時は「OK」ではなく「保留／人手送り」へ倒す。誤って通すコストが高いドメインほど効きます。
* 暴走ガードを最初から。予算上限・最大ステップ・同一ツール呼び出しの抑制は、PoC 段階から入れておく。

## まとめ

Clearwing から学べる最大の教訓は、「コスパ」と「精度」はトレードオフではなく、ワークフローの組み方で両立できるということです。鍵は、2 つのラダーを噛み合わせることにあります。

* 安い処理を手前に置き、重み付けで優先度を決め、価値ある対象にだけ高い処理を集中させる（コスパ）
* 確信度を段階管理し、独立・敵対的に検証し、最後は外部オラクルで殴る（精度）

そして両者は 「確信度ラダーで下流をゲートする」 という一点で繋がります。低信頼に高コストはかからず、高コストは高信頼にしか使われない。

巨大モデルを買えば精度が上がる、というのは半分正しい。しかし実務では、モデル性能より先に「どこで絞り、どこで検証し、どこで止めるか」を設計しないと、コストも誤検知も爆発します。Clearwing が面白いのは、モデル能力の不足を魔法のプロンプトで補うのではなく、安い信号・段階的予算・証拠ラダー・外部オラクルで LLM の失敗モードをワークフローで閉じ込めている点にあります。

つまり狙いは「安いモデルで最上位モデル同等の性能を保証する」ことではなく、「LLM の失敗モードを制御して、コストと誤検知を予測可能な範囲に収める」こと。自前エージェントを組むときも、この発想がそのまま指針になります。

## 付録：マルチベンダー対応と“賢さ”の差し替えはどうやっているか

Clearwing は Anthropic / OpenAI / Gemini / Ollama / 任意の OpenAI 互換エンドポイントを差し替えられます。ただし「賢いモデルかどうか」を自動で見分けているわけではありません。仕組みは 3 層に分かれます。

### 1. API の差はアダプタで正規化する（吸収しているのはここだけ）

呼び出し口（`AsyncLLMClient.achat` / `aask_text`）はプロバイダ非依存で、`_adapter_for_endpoint`（`providers/manager.py`）が「論理プロバイダ → genai アダプタ」に変換します。認証・tool-calling 形式・structured-output の差はここで吸収。挙動の癖も個別にシムしています。

* `reasoning_effort="auto"` をモデル名から自動解決し、非対応モデルでは `None` に落とす（`llm/native.py`）
* ランカーのスコアを `Literal[1..5]` の enum で渡す（Anthropic の structured-output バリデータが min/max を弾くため）

### 2. “賢さ”はタスク別のルーティング表に人手で割り当てる

モデルの能力をプローブする仕組みはありません。タスクごとの既定モデルは `DEFAULT_ROUTES` にハードコードされており、出荷時デフォルトは意図的にすべて Anthropic です（下表は私が選んだ例ではなく、素のままだとこうなる、という実際の既定。モデル名は実装時点のもの）。

| task | 既定モデル | 役割 |
| --- | --- | --- |
| ranker | claude-haiku-4-5 | ファイル分類（安い） |
| hunter | claude-opus-4-6 | 中核の脆弱性推論（強い） |
| verifier | claude-sonnet-4-6 | 独立検証（hunter と別ティア） |
| sourcehunt\_exploit | claude-opus-4-6 | 最難の推論 |

設計上の要は、hunter と verifier をあえて同じ Anthropic の別ティアにしている点です。コードのコメントいわく「独立性はティアから来るのであって、プロバイダから来るのではない」。`ANTHROPIC_API_KEY` 一つでも独立検証が成立するよう、賢さの差はベンダー間ではなくティア間で付ける、という既定になっています。

### 3. 異ベンダー混在（例：hunter=Opus / verifier=GPT）

`config.yaml` のマルチエンドポイントモードでは、タスクごとに別プロバイダを割り当てられます（コード上も「Each task can land on a different provider」と明記）。

```
providers:
  anthropic:
    api_key: ${ANTHROPIC_API_KEY}
  openai:
    api_key: ${OPENAI_API_KEY}
    base_url: https://api.openai.com/v1
routes:
  hunter: anthropic      # 発見は Opus
  verifier: openai       # 検証は GPT（別ベンダーで独立性をさらに高める）
task_models:
  hunter: claude-opus-4-6
  verifier: gpt-...       # 任意の OpenAI モデル
```

注意点。

* 混在できるのはステージ間であって、1 回の呼び出し内ではない（1 クライアント = 1 プロバイダ + 1 モデル）。
* 各ベンダーの API キーが必要（既定が全 Anthropic なのは 1 アカウントで動かすため。クロスベンダーは opt-in）。
* tool-calling が弱いモデルを hunter に置くと、コンパイル/実行を自分で回せず性能が落ちる。

そして「賢さの差」の本当の吸収先は、本文で見たワークフロー（安い前処理・証拠ラダー・独立検証・実証）です。モデルを賢くするのではなく、弱いモデルの誤りをシステムで濾し取る。だからモデルは差し替え可能で、かつ差し替えても破綻しにくいわけです。
