## 作ったものを誰にも知ってもらえないまま終わる

個人開発をしている人なら、この感覚を知っているんじゃないかと思う。

週末を削って作ったツール、深夜に試行錯誤しながら書いたOSS、副業として育てているWebアプリ。GitHubにpushして、せっかくだからとQiitaに記事を書いて、Twitterにも投稿して——最初の1週間はそれができる。でも2週間後はどうだろう。次のプロジェクトの実装が気になり始めて、PRは「気が向いたらやる」タスクに格下げされていく。

これを繰り返すたびに思っていた。問題は「やる気」じゃなくて「仕組みがない」ことだと。

## 何が本当のボトルネックだったか

一時期、複数のプロジェクトを同時並行で動かしていた。SaaS系のWebアプリ、CLIツール、Flutterで作ったモバイルアプリ。それぞれ別のターゲット層に向けて発信するべきで、媒体も違う。技術者向けにはQiitaやDev.to、英語圏にはReddit、カジュアルな宣伝はTwitter、長文はZennかブログ——と頭では分かっていても、それを毎週実行し続けるコストが重すぎた。

プロジェクトごとに「これは何の技術を使っているか」「誰が使うのか」「どんな切り口で書けばウケるか」を考えるところから始まる。そこに30分かけて、ようやく下書きを書き始める。複数プラットフォームへの投稿フォーマットは微妙に違う。Twitterは文字数制限、Qiitaはマークダウンで本文が必要、Dev.toはタグの付け方が独特。これを毎週、全プロジェクト分やるのは現実的ではなかった。

結果として、何が起きていたか。直近のプロジェクトしかPRしない。過去に作ったものはどんどん埋もれていく。せっかく作ったのに、見てもらう機会すら生まれない。

## 自動化を試みた最初の失敗

「じゃあ自動化しよう」と思い立ち、まずシェルスクリプトとGitHub Actionsを組み合わせたパイプラインを書いた。プロジェクトのREADMEを読ませてClaude CLIでSNS投稿文を生成し、APIで投稿する、という構成だ。

動いた。最初は。

問題は「プロジェクト分析」のステップだった。当初、Claude CLIをTauriのバックエンド（Rust）から起動する際に、SendKeysとPowerShellのAppActivateを組み合わせてターミナルウィンドウを自動操作していた。フォルダ信頼の承認ダイアログが出たら「y」を送信、プロンプトを貼り付けてEnterを押す——これをSleepで固定秒数待ちながら制御していた。

最初は動いていたが、マシンの負荷が高いときや、CLIの起動が少し遅れると、固定秒数のタイミングがずれる。するとプロンプトが届かず、「分析中...」の表示のままアプリが永遠に止まる。何度かデバッグして気づいた。Windowsでは非アクティブウィンドウへのキー送信が制限されており、AppActivateで一瞬フォーカスを移しても、SendKeysが届かないケースが普通に起きる。

ウィンドウ操作という「人間がやるべき操作をプログラムで模倣する」アプローチが根本的に間違いだった。

解決策はシンプルだった。`claude --print`モード（`-p`フラグ）と`--permission-mode bypassPermissions`を組み合わせることで、CLIを完全に非対話で実行できる。ウィンドウは開かず、stdoutから直接結果を受け取れる。Rustの`std::process::Command`でspawnして、stdoutをストリームとして受け取る。これだけで「止まるバグ」は完全に消えた。

```
let output = Command::new("claude")
    .args(&["-p", &prompt, "--permission-mode", "bypassPermissions"])
    .current_dir(&project_path)
    .output()
    .await?;
let result = String::from_utf8(output.stdout)?;
```

シンプルな変更で、信頼性がまったく別物になった。

## Gitをデータストアにする設計が意外と強い

もう一つ試行錯誤したのが、データの管理方法だ。

生成したコンテンツ、投稿履歴、プロジェクトのメタ情報——これらをどこに保存するか。最初はSQLiteをローカルに置くことを考えた。でもそうすると、デスクトップアプリとGitHub Actionsワークフローがデータを共有できない。

たどり着いた設計は、GitHubリポジトリ自体をデータストアとして使うことだった。`content/projects/*.json`にプロジェクト定義を置き、`content/meta/*.json`に生成コンテンツや投稿履歴を置く。デスクトップアプリはgitコマンドでpull/pushし、GitHub Actionsも同じファイルを読み書きする。

これにより「デスクトップアプリで設定したプロジェクト情報」を「GitHub Actionsが読んでコンテンツを生成・投稿する」という流れが自然に作れた。さらに、すべての変更がGitの差分として残るので、「いつ、どのプロジェクトに、何を投稿したか」の履歴が勝手に蓄積される。

ただし、データ同期で一度ひどいバグを踏んだ。アナリティクスデータをリフレッシュする際に、リモートから最新データをpullするついでにローカルの作業中変更が消えてしまう問題だ。当初のコードが`git stash drop`を呼んでいたためで、本来は`git stash → git pull --ff-only → git stash pop`の順序が正しい。この1単語の違いで、設定中のデータが無言で消える。

## GitHub Traffic APIの14日制限を回避する

アナリティクス機能を実装していて気づいた不便さがある。GitHub Traffic APIは直近14日分のデータしか返さない。週次でしか確認しない場合、気づいたら古いデータが失われている。

回避策として、`traffic-history.json`にデータをマージしながら蓄積する仕組みを作った。APIから取得したデータと既存ファイルの差分を計算し、新しい日付のデータだけを追記してコミットする。これでGit管理された長期トラフィック履歴が手に入る。

```
// 既存のhistoryと新しいデータをマージ
const merged = mergeTrafficHistory(existing, newData);
// 日付でソートして重複を除去
const deduplicated = deduplicateByDate(merged);
```

単純な処理だが、これをやらないと「3ヶ月前にバズったあの投稿の効果」が測定できなくなる。

## 型安全性を保ちながらRust-TypeScript間のIPCを設計する

Tauri v2でフロントエンド（React/TypeScript）とバックエンド（Rust）の間のIPC設計に一番時間を使った。

`invoke<T>()`のジェネリクスで戻り値の型を指定できるのは便利だが、フロントエンド側の型定義とRust側の構造体が乖離すると、実行時エラーが静かに発生する。これを防ぐために、すべてのIPCを`useApi`フックに集約した。

```
const useApi = () =>
  useMemo(() => ({
    scanProject: (path: string) =>
      invoke<ProjectScanResult>("scan_project", { path }),
    getAnalytics: (slug: string) =>
      invoke<AnalyticsData>("get_analytics", { slug }),
    // ...他のIPC
  }), []);
```

`useMemo`でメモ化しているのは意図的だ。これをしないと、コンポーネントの再レンダリングのたびに新しいオブジェクト参照が生まれ、子コンポーネントの不要な再レンダリングを引き起こす。地味だが見落としやすいReactのパターンで、一度プロファイラで見るまで気づかなかった。

## 「プロジェクトが保存されていないのに編集画面に遷移する」バグ

実装中に一番ばかばかしいと思ったバグを紹介する。

「プロジェクトを追加する画面」から「編集して追加するボタン」を押すと、編集画面に遷移する設計だった。編集画面では`get_project(slug)`を呼んでデータを取得する。

当然、その時点でプロジェクトはまだ保存されていないのでエラーになる。「追加前のプロジェクトを編集する」という操作は、そもそも「追加と編集を同時にやる」ことであって、別画面への遷移で解決できる問題ではない。

解決策はシンプルで、遷移をやめてインラインでフォームを表示するだけ。でも設計の前提が間違っていると、いくらデバッグしても直らない。

## プロジェクトの「自己紹介」ファイルを置くという発想

技術スタックの自動検出（`package.json`のdependencies逆引き、`Cargo.toml`、`pubspec.yaml`の解析）は思ったより精度が出た。NPMの14パターン、Flutterの16パターンのマッピングで主要な技術はほぼ取れる。

でも、どうしても取れない情報がある。プロジェクト名の日本語表記、タグライン、説明文。これらはコードのどこにも書いていない。

解決策として`.pr-meta.json`をプロジェクトルートに置く仕組みにした。

```
{
  "nameJa": "PRダッシュボード",
  "taglineJa": "個人開発のPRを自動化するデスクトップアプリ",
  "descriptionJa": "プロジェクト分析からSNS投稿まで一括管理",
  "targetAudience": ["個人開発者", "技術ブロガー"],
  "category": "tool"
}
```

このファイルがあれば0.5秒以下でフル情報が取れる。ファイルがなければ推論スキャン（ほぼ同速）、精度が必要なときはClaude分析（30〜120秒）という三段階の設計にした。

日本語でのPR文章生成には日本語のメタ情報が不可欠で、「コードを読んでいい感じに補完してくれるだろう」というAIへの過信は禁物だった。

## 継続できる仕組みをどう設計するか

個人開発のPRが続かない理由を突き詰めると、「毎回ゼロから考えなければならない」コストの高さに行き着く。

プロジェクトの情報を一度定義してしまえば、あとはスケジュールに従って自動的に各プラットフォーム向けのコンテンツが生成・投稿される——この状態を作ることが目標だった。Twitter用に曜日別の投稿スロットを設定し、Qiitaは月次、Dev.toは週次、といったグローバルスケジュールを一度決めれば、プロジェクトが増えても管理コストはほぼ変わらない。
