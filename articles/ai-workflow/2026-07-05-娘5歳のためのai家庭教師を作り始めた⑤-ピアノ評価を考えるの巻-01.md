## はじめに

[前回の記事](https://zenn.dev/dsflon/articles/dab979c414583b)では、UI のリニューアルを Fable 5 と Opus 4.8 に分けて進めた話を書きました。

今回はピアノ評価です。  
先に完成したものを。

![](https://static.zenn.studio/user-upload/902671cfc5c2-20260704.png)

第1回から「ピアノ: 練習動画」とだけ書いて、ずっとモックのまま置いていた機能です。ようやく手を入れました。

先に言うと、AI に評価させるところは拍子抜けするくらいすぐ書けました。

やはり動画が重い。

スマホで撮った30秒の演奏動画が、80MB程度ありました。今回はそこからの話です。

## 15分の練習を毎回送るのは無理

最初に考えていたピアノ評価は、かなりざっくりしていました。

練習をまるごと録画して、集中しているか、先生からの指摘が多いか、リズムが崩れていないかを見る。そんなイメージです。

でも、実際の練習は15分くらいあります。毎回15分の動画を撮って、アップロードして、評価を待つ。これは無理です。子どもより先に親が飽きます。

なので、仕様をかなり小さくしました。

* その日の課題曲を1曲だけ撮る。長くても1〜2分
* 課題曲の楽譜も写真で撮って、一緒に送る

評価で見たいのは、主にこのあたりです。

* 楽譜どおりに弾けているか
* 指の形や運びが崩れていないか
* 背筋が丸まりすぎていないか

楽譜を一緒に送るのは大事だと思いました。動画だけ渡しても、AI には「何が正解か」が分かりません。有名な曲なら知っているかもしれませんが、子どもの教本の短い課題曲まで知っている前提にはできない。

答え合わせの基準ごと渡す。

そのほうが、まだ評価として筋が良さそうです。

モデルは英語フレーズ評価でも使っている `gemini-3.5-flash` です（2026年7月時点）。プロンプトは他の科目と同じで、評価基準と JSON の出力形式を固定しました。

最初に書いたプロンプトは、だいたいこんな感じです。

```
【評価の観点】
1. 楽譜どおりに弾けているか（最重要）
   - 動画のピアノの音と楽譜を照らし合わせ、音の高さ・順序が正しいか
   - ミスタッチ・弾き直し・途中で止まった箇所がどのくらいあるか
2. 指の形・運び
   - 指が伸び切ってしまっていないか、アーチ型でしなやかに保てているか
3. 演奏時の姿勢
   - 背筋が伸びているか（前かがみ・猫背になっていないか）

※迷ったときは "try" ではなく "good" を選ぶ。
  モチベーションを損なわない評価を心がける
```

最後の「迷ったら good」が、あとで効いてきます。

## 実装だけ見ると簡単だった

このアプリの評価機能は、ひらがなも英語フレーズも、ファイルを base64 にして `inlineData` で Gemini に渡しています。

ピアノも最初は同じ形で書きました。動画と楽譜画像を並べて渡すだけです。

```
// 抜粋・イメージ
const response = await ai.models.generateContent({
  model: PIANO_EVALUATION_MODEL,
  contents: [
    {
      role: "user",
      parts: [
        { text: "課題曲を演奏している動画です：" },
        { inlineData: { mimeType: videoMimeType, data: videoB64 } },
        { text: "課題曲の楽譜の写真です：" },
        { inlineData: { mimeType: scoreMimeType, data: scoreB64 } },
        { text: PROMPT },
      ],
    },
  ],
});
```

lint も build も通りました。

勝ったな、と思いました。

もちろん、勝っていません。

## 30秒で80MB程度

試しにスマホで課題曲を撮ってみたら、30秒で80MB程度ありました。撮影設定にもよると思いますが、手元ではこのくらいでした。

ここで一気に現実が来ます。

| 壁 | 上限 |
| --- | --- |
| Vercel サーバーレス関数のリクエストボディ | 約4.5MB |
| Gemini のインライン入力（リクエスト全体） | 約20MB |

参考（いずれも2026年7月時点）:

80MB は、そもそも API Route に届きません。仮に Vercel の壁を越えたとしても、Gemini にインラインで渡せるのは20MBまでです。しかも base64 にすると、データはだいたい1.3倍に膨らみます。

写真や数秒の録音では気にしなくてよかった制約が、動画になった瞬間に前面へ出てきました。

## API Route を通さない

Vercel の4.5MB制限を避けるために、動画は **クライアントから Supabase Storage へ直接アップロード** する形に変えました。

API に渡すのは動画そのものではなく、Storage 上のパスだけです。サーバー側はそのパスを使って動画を取りに行き、Gemini に渡します。

これで「API Route に届かない」問題は避けられます。

ただ、Gemini 側の20MB制限は残ります。80MBをどうにか小さくしないといけません。

ここで Gemini の動画の扱いを見直しました。公式ドキュメントによると、動画は **約1フレーム/秒でサンプリング** されます。30fps のなめらかな映像を送っても、モデルが見るのは毎秒1枚くらい。

なら、映像はかなり削ってよさそうです。

逆に、音は削りすぎたくありません。ピアノ評価で見たいのは音の高さやリズムなので、ここを壊すと何を評価しているのか分からなくなります。

今回はこうしました。

* 映像は長辺720pxに縮小して、低ビットレートにする
* 音声は 96kbps を維持する

動画を倍速にして短くする案も一瞬考えました。でも、再生速度を変えると音の高さが変わります。音程を見たいのに音程を壊すのはさすがに本末転倒なので、これはやめました。

圧縮はブラウザ内でやっています。`<video>` 要素で再生しながら canvas に描き、`canvas.captureStream()` の映像と、WebAudio で取り出した音声を `MediaRecorder` に渡して録り直す作りです。

```
// 抜粋・イメージ
const canvasStream = canvas.captureStream(30);
const audioSource = audioCtx.createMediaElementSource(video);
audioSource.connect(audioDest); // スピーカーには出さない

const stream = new MediaStream([
  ...canvasStream.getVideoTracks(),
  ...audioDest.stream.getAudioTracks(),
]);

const recorder = new MediaRecorder(stream, {
  mimeType, // Safari は mp4、Chrome は webm
  videoBitsPerSecond, // 動画の長さから逆算して全体約12MBに収める
  audioBitsPerSecond: 96_000,
});
```

この方法だと、動画の実時間ぶん待つ必要があります。2分の動画なら2分かかります。

ちょっとつらいですが、今回は進捗バーを出して耐えることにしました。`ffmpeg.wasm` を使えばもっと速くできる可能性はあります。ただ、バンドルが重くなるので、今回は依存を増やさない方向です。

最終的な流れはこうなりました。

```
ブラウザで再エンコード（720px・全体約12MB目標・音声96kbps維持）
  ↓
Supabase Storage へ直接アップロード
  ↓
POST /api/evaluate/piano（動画のパス + 楽譜画像だけ送る）
  ↓
サーバーが Storage から動画を取得して Gemini へ
  ↓
評価が終わったら動画を削除
```

演奏動画は評価に使うだけの一時ファイルです。子どもの動画をストレージに溜め続けたくないので、評価が終わったら削除します。成功しても失敗しても削除が走るようにしました。

## 甘くしろと自分で書いていた

ここまで直して、ようやく評価が返ってきました。

最初に投げたのは、途中で少しつっかえた演奏です。リズムも少し崩れていました。

返ってきた評価は「たいへんよくできました」。

フィードバックの1行目はこうです。

```
よかったところ：おんぷをしっかりとよんで、さいごまでリズムよくひけました。
```

いや、つっかえていた。

ほかの行も全体的にふわっとしていて、どこを聴いてそう言ったのか分かりません。

一瞬、動画からのピアノ評価はこのモデルだと厳しいのかなと思いました。でも、第3回の英語フレーズ評価で「無音でも たいへんよくできました」が返ってきたときと同じ匂いがします。

プロンプトを読み返しました。

「多少のミスタッチがあっても流れを保って最後まで弾けていれば excellent」  
「迷ったときは good を選ぶ」

甘くしろと自分で書いていました。

モデルは、だいたい指示どおりに甘かっただけです。

ここは直しました。

rating を決める前に、観察を書かせるようにしました。`analysis` という内部用フィールドに、「つっかえた回数と場所」「テンポの安定度」「楽譜と違う音」を事実として出させます。画面には出しません。印象だけで丸めて褒めるのを減らすためです。

excellent の条件も、数えられる形にしました。「つっかえ・弾き直しが合計2回以上なら excellent にしない」「テンポが明らかに不安定なら excellent にしない」「excellent と good で迷ったら good」。

子どものやる気への配慮は、評価を甘くすることではなく、フィードバックの言葉遣いでやることにしました。

フィードバックも、抽象的な褒め言葉だけではなく、曲のどのあたりか、どちらの手か、音やリズムに何があったかを書くようにしました。

それから、思考トークンも有効にしました。コストが気になって `thinkingBudget: 0` にしていたのですが、音と楽譜を照らし合わせる処理はかなり推論寄りです。そこを削っておきながら精度に文句を言うのは、ちょっと筋が悪い。

既定はモデルまかせに戻しつつ、環境変数で絞れるようにはしています。判定のブレを抑えるために temperature も低めにしました。

第2回で思考トークンの請求に驚いた身としては、コストが跳ねる予感しかしません。ここはいったん精度側に倒します。

## できること、できなさそうなこと

プロンプトで直せる部分はあります。ただ、モデル側の限界も当然あります。

Gemini は動画を毎秒1フレームくらいで見るので、指の細かい運びまでは期待しすぎないほうがよさそうです。音符レベルで完全に聞き取れる保証もありません。音楽転写の専用モデルではないので、そこは割り切りが必要です。

今の期待値としては、「途中で止まった」「つっかえた」「テンポが大きく崩れた」あたりを拾ってくれれば十分かなと見ています。

厳格化した後の再評価は、これからです。同じ演奏が good に下がるかどうか。ここはちゃんと見たいところです。

## まとめ

ピアノ評価を、ようやくモックから本実装にしました。

AI に評価させるところより、動画を届けるための準備のほうがずっと重かったです。圧縮して、直接アップロードして、評価が終わったら消す。書くと地味ですが、ここが一番時間を使ったところでした。

動くようになったら、今度は評価が甘すぎる問題です。これは英語フレーズ評価のときと同じでした。評価基準は、実際の入力を投げて、返ってきた結果を見ながら直すしかなさそうです。

そして、いちばん引っかかっているのが、楽譜と突き合わせて「どこが違うか」を当てる部分です。存在しない小節を指摘してくる状態なので、ここは改善のよちしかない、。動画からの楽譜照合をどう安定させるかは、次の宿題です。

まだ残っている確認もあります。実機の Safari で圧縮まわりがちゃんと動くか。厳格化したプロンプトで評価がどう変わるか。動画と思考トークンありで、1回いくらになるか。

第2回の再来になりそうな気配があります。

このあたりは、娘の実際の練習で使いながら見ていきます。
