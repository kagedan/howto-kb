## この記事について

[前回](https://zenn.dev/tyler0702/articles/88697c78d30507)の記事で、AI(Claude)が出力する長文Markdownを読むためにTauri v2でリーダーアプリ「villar」を作った経緯を書きました。

今回はvillarの核心、Markdownの処理パイプラインについて書きます。カード分割、TL;DR自動生成、Mermaidの3段階フォールバック。この3つで開発時間の6割くらいを使いました。

## Markdownパイプライン：ここが本体

villarのコードで一番量が多いのはReactのUIではなく、Markdownの処理パイプラインです。

```
Markdownテキスト
  ↓ remark-parse      （Markdown → AST）
  ↓ remark-gfm        （テーブル、タスクリスト対応）
  ↓ remark-section     （★ H2でセクション分割）
  ↓ remark-tldr        （★ セクションごとにTL;DR生成）
  ↓ remark-collapse    （★ 長いリスト/コードを折りたたみ）
  ↓ remark-rehype      （AST → HTML AST）
  ↓ rehype-highlight   （コードのシンタックスハイライト）
  ↓ rehype-stringify   （HTML AST → HTML文字列）
  ↓ collapseHtml       （折りたたみのプレースホルダーを挿入）
  ↓ addCopyButtons     （コードブロックにコピーボタンを注入）
  ↓ addTableWrap       （テーブルを横スクロール可能にラップ）
  ↓ resolveImagePaths  （ローカル画像のパスを解決）
  ↓ addHeadingAnchors  （H3にアンカーリンクを追加）
  ↓ applySpeedRead     （Speed Read: 単語先頭を太字化）
  ↓
  カード表示
```

★の付いた3つが自作のremarkプラグインです。既存のnpmパッケージでは実現できなかった処理を、ASTを直接操作して実装しています。

最初はH1で分割していました。見出しの最上位だし、章の区切りはH1だろうと。自信満々でした。

実装して、テスト用のMarkdownを読み込んで、表示されたカードの枚数は……1枚。4000行が1枚のカードに収まっていました。分割とは。

Claudeが出力するMarkdownを20本くらい調べたら、H1は「# API設計書」のようにタイトルとして1回しか使われていないことがほとんどでした。H1分割だとドキュメント全体が1枚のカードになります。そりゃそうだ。

実際の章立てはH2です。「## 認証」「## レート制限」「## エラー処理」。これはAIに限った話ではなく、人間が書くMarkdownでも同じ傾向でした。GitHub上のREADME、技術ブログ、ドキュメント。H2が「章」、H3以下が「節」。Markdownの暗黙の慣習です。

「じゃあH3だ」と思って分割したら今度はカードが30枚以上。1枚が3行しかないカードを矢印キーで送り続ける。PowerPointのスライドか？ これはスクロールと同じくらいストレスでした。

H2が「一区切りで読める単位」として、ちょうどいい粒度でした。

```
// 簡略化したロジック
const headings = tree.children
  .map((node, i) => ({ node, index: i }))
  .filter(({ node }) => node.type === 'heading' && node.depth === 2);

if (headings.length > 0) {
  // H2ごとにセクションを作る
  sections = headings.map((h, i) => ({
    title: extractText(h.node),
    children: tree.children.slice(h.index, headings[i + 1]?.index)
  }));
} else {
  // H2がないドキュメントは段落5つごとに分割
  sections = chunkByParagraphs(tree.children, 5);
}
```

ちなみにH2が1つもないMarkdownも存在します。会議の議事録とか、チャットのログをそのまま`.md`にしたようなやつ。最初はこのケースを想定しておらず、H2なしのファイルを開いたら真っ白な画面が表示されました。「バグだ」と思って2時間デバッグしたら、バグではなく仕様でした。H2がないから0分割。正しく0枚のカードが表示されている。正しいけどダメです。

段落を5つずつまとめてセクションにする「レスキュー処理」を入れて解決しました。

カード分割ができた時点で、すでにだいぶ読みやすくなっていました。でも24枚のカードがあると「3枚目って何の話だっけ」問題が出てきます。各カードに要約が欲しい。

最初はClaude APIを叩くことを考えました。一番精度が出るのは間違いない。でもやめました。理由は2つ。AIが書いた文章をAIに要約させるために毎回課金するのは何かが根本的におかしい。それと、ネットワークがない環境（飛行機、カフェのWi-Fiが不安定な時）でも使いたい。

結局、ルールベース + TextRankで実装しました。

* 最初の段落から概要を取得
* リスト項目からキーポイントを抽出
* 最後の段落を結論として扱う
* TextRankで重要な文をランキング
* 50文字未満のセクションはTL;DRを生成しない

精度は完璧ではありません。コードブロックが多いセクションだと、コードの前後の説明文だけをつまみ食いするので「何の話？」になることもあります。でも8割くらいのセクションでは「ここは認証の話」「ここはDB設計の話」くらいはちゃんとわかります。自分が使う分には十分でした。

## Mermaidの処理が一番大変だった

一番苦労したのがMermaidです。期間でいうと全体の3割くらいこれに使った気がします。

Claudeは設計書にMermaidのフローチャートをよく入れてきます。誰も頼んでいないのに。確かに見やすいんですが、レンダリングする側の気持ちも考えてほしい。

```
Mermaidコードブロック
  ↓ mermaid-linear.ts で解析
  ↓
  線形グラフ？ → ステップUI（カスタムコンポーネント）
  ↓ No
  mermaid.jsでレンダリング → SVGダイアグラム
  ↓ 失敗
  コードブロックのまま表示
```

**なぜステップUIを用意したか？**

最初はすべてのMermaidをmermaid.jsでSVGレンダリングしていました。動いたんですが、mermaid.jsが1MB近くあって初期表示が明らかに遅い。

で、AIが出力するフローチャートを観察していて気づいたのが、大半は `A → B → C → D` みたいな一直線の流れだということです。分岐もループもない。これにSVGは大げさすぎる。

`mermaid-linear.ts`というパーサーを書いて、Mermaidのテキストを解析し、全ノードの次数が1以下（分岐なし）かどうかを判定するようにしました。線形だったらHTMLのステップUIで表示。非線形の場合だけmermaid.jsを動的インポートしてSVG化。これでほとんどのケースでmermaid.jsの読み込み自体が不要になりました。

ただ、Mermaidの構文パーサーを自作するのは想像以上に面倒でした。ノードIDにスペースを含むパターン、`-->`と`--->`と`-.->` の矢印バリエーション、サブグラフ。「あとこのパターンだけ対応すれば完成」を5回くらい言いました。最終的には対応しきれないケースはmermaid.jsにフォールバックする前提で割り切りました。最初からそうすべきだった。

## パフォーマンスバジェット

ある日、リファクタリング後にMarkdownを開いたら「あれ、なんか遅い？」と感じました。計測したらパイプラインが2秒かかっていた。正規表現の中で`.*`を3つネストしていたのが原因でした。正規表現は書いた本人にしかわからないし、3日経つと書いた本人にもわからない。直したら300msに戻りました。

この経験から、パフォーマンスバジェットをテストに組み込みました。

| 処理 | 上限 |
| --- | --- |
| 全パイプライン（50KB超のドキュメント） | 1000ms |
| セクション分割 | 50ms |
| TL;DR生成（セクション1つ） | 30ms |
| Mermaid線形判定 | 5ms |

Vitestのテストケースに入っているので、CIで毎回計測されます。遅くなったらテストが落ちる。「体感で遅くなった気がする」ではなく、数字で検知できるのは安心感があります。
