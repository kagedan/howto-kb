## 背景

NestJS + LLM で構築された対話型AIシステムを開発している。ある処理フローの中で、特定の条件に該当するデータはLLMに処理させず自動的にスキップする、という仕様があった。

## 発生した問題

このスキップ処理が、プロンプト上の自然言語による指示として実装されていた。

```
「処理に入る前にデータの属性を確認し、特定条件に該当する場合は
 ユーザーに説明 → デフォルト値を設定 → 次のデータに遷移すること。
 デフォルト値の設定を飛ばしてはならない」
```

これにより、いくつかの問題が発生していた。

* **判定の不安定さ**: 単純なif/elseで済む分岐をLLMに任せると、指示どおりに動かないケースがあった
* **ユーザー体験の冗長さ**: 「この項目はスキップします」という説明が毎回挟まり、テンポが悪い
* **プロンプトの肥大化**: 例外処理の指示が増えるほど、本来重要な指示への注意が散漫になる
* **型安全性の妥協**: このフィールドを許容するために、プロンプトの型定義に `[key: string]: unknown` という index signature が必要になっており、タイポや不正フィールドの混入を検出できなかった

## 原因

責務の配置ミス。「特定条件に該当するデータにはデフォルト値をセットする」は確定的なビジネスロジックであり、LLMが都度判断する必要のない処理だった。にもかかわらず、プロンプト経由でLLMに委ねていたことが根本原因。

## 解決策

方針は「システム側で対象外データの処理を完結させ、LLMには処理が必要なデータだけを渡す」こと。以下の手順で対応した。

### 1. フロー開始時の前処理を追加

該当フローに入った直後に、スキップ対象のデータにデフォルト値を自動セットする処理をサービス層に実装した。

```
// 擬似コード
const skipTargets = allItems.filter(item => shouldSkip(item));
for (const item of skipTargets) {
  await this.setDefault(item.id);
}
```

### 2. LLMに渡すデータからスキップ対象を除外

データ生成ロジックを修正し、LLMが処理すべきデータだけを渡すようにした。

```
// 変更前: 全件渡す
const data = allItems;

// 変更後: 処理対象のみ渡す
const data = allItems.filter(item => !shouldSkip(item));
```

### 3. プロンプトのクリーンアップ

プロンプト定義からスキップ処理に関するルールを削除し、対話フローの記述を簡素化した。

### 4. 型定義の厳格化

プロンプトの型定義から `[key: string]: unknown` の index signature を削除。定義されていないフィールドを追加するとコンパイルエラーになるようにした。

## 補足・学び

### LLMに任せるべきでないロジックの判断基準

以下に該当する処理は、プロンプトではなくシステム側で実装すべき。

* **確定的な分岐**: if/else で表現できる単純なロジック
* **データの前処理**: フィルタリングや正規化
* **副作用を伴うビジネスルール**: 「必ずこのAPIを呼ぶ」系の処理

LLMが得意なのは文脈に応じた自然言語生成や曖昧な判断であって、bool分岐の正確な実行ではない。

### 「LLMの入力を減らす」という設計指針

対象外のデータを事前に除外したことで、LLMが処理すべき情報量が減り、本来のタスクへの集中度が上がった。プロンプトに例外処理を足すのではなく、例外をシステム側で事前に解消してからLLMに渡す——この発想は他の場面でも応用が効く。

### 型安全性は「ついで」ではなく本質

index signature の削除は一見すると型定義のお掃除だが、実際にはタイポ検出や不正フィールドの混入防止に直結する。プロンプトをコードで管理するなら、型の厳格さはプロンプトの品質保証そのもの。
