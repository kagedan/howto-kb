# AWS Bedrockで売上分析AIエージェントを構築してみた

## はじめに

こんにちは。  
新人研修の一環として、生成AIを活用した売上分析AIエージェントの開発を行いました。

今回の取り組みでは、ユーザーが自然言語で分析内容を指示すると、その内容に応じたSQLの生成、データ取得、グラフ描画までを自動で実行する仕組みを構築しました。

本記事では、AWS Bedrockを活用した売上分析AIエージェントの構成と、実装にあたって特に考慮したプロンプト設計や構成上のポイントについて紹介します。

---

## システム構成

自然言語で売上分析の指示を入力すると、その内容に応じたグラフを自動生成するアーキテクチャとしました。

![売上分析AIエージェントのシステム構成図](https://static.zenn.studio/user-upload/ed5c88499e8e-20260319.png)

全体の流れは以下の通りです。

* Streamlitでユーザーが自然言語を入力
* API Gateway経由でLambdaを起動
* LambdaからBedrockへプロンプトを送信
* BedrockがSQLとグラフ種別を生成
* LambdaがSnowflakeからデータを取得
* Lambdaでデータ加工とグラフ描画を実施
* 生成した画像をS3に保存し、Streamlitで表示

### 使用技術

| 項目 | 技術 |
| --- | --- |
| UI | Streamlit |
| LLM | AWS Bedrock |
| 実行基盤 | AWS Lambda |
| API管理 | API Gateway |
| DWH | Snowflake |
| ストレージ | Amazon S3 |
| グラフ描画 | matplotlib |

## プロンプト設計を行う上で考えたこと

今回の実装では、ユーザーの自然言語をそのままBedrockへ渡すのではなく、  
Lambda内でプロンプトを組み立ててから送信する構成としました。

当初から、できるだけシンプルに自然言語を渡してSQLを生成させることも考えましたが、  
出力の安定性や精度を考慮し、以下の情報を明示的に付与する方針としました。

* LLMに与える役割定義
* 売上データのスキーマ情報
* 出力形式の指定（SQL＋グラフ種別）

特に、売上データのスキーマ情報をどのように扱うかは重要なポイントでした。

今回は、分析対象となるテーブルのスキーマ情報をLambda内に直接記述し、Bedrockへ渡すプロンプトに埋め込む方式を採用しました。  
この方式は、実装がシンプルで、LLMに渡す情報を明示的に制御しやすい点がメリットです。

一方で、テーブル定義が変更された場合はLambda側の修正が必要になるため、保守性の面では注意が必要です。

また、別の方法として**Snowflakeのメタデータからスキーマ情報を取得し、その内容をBedrockへ渡す**構成も考えられます。  
こちらは最新の定義を参照しやすく、テーブル追加や変更への追従性が高い一方で、実装がやや複雑になり、プロンプトが長くなりやすいという特徴があります。

今回は、対象データを限定したうえで安定してSQLを生成できることを優先し、Lambda内にスキーマを直接記述する方式を採用しました。

## 実現方式

実際の実装では、役割定義・スキーマ情報・出力形式の3つをプロンプトに含めてBedrockへ送る構成としました。

以下は、Lambda内のプロンプトの一例です。

```
prompt = f"""
# 役割定義
あなたは売上分析アシスタントです。
ユーザーの質問をもとに、Snowflakeで実行可能なSQLと適切なグラフ種別を生成してください。

# スキーマ情報
テーブル名: SALES
カラム:
- ORDER_DATE (DATE): 受注日
- PRODUCT_NAME (STRING): 商品名
- CATEGORY (STRING): 商品カテゴリ
- SALES_AMOUNT (NUMBER): 売上金額
- QUANTITY (NUMBER): 販売数量
- REGION (STRING): 地域

# 出力形式（JSON）
{{
  "sql": "生成したSQL",
  "chart_type": "line または bar または pie"
}}

# ユーザーの質問
{user_input}
"""
```

### 1. 役割定義

LLMに対して、売上分析アシスタントとして振る舞うことを明示しました。  
具体的には、ユーザーの自然言語からSQLを生成し、あわせて適切なグラフ種別も選択する役割を与えています。

```
# 役割定義
あなたは売上分析アシスタントです。
ユーザーの質問をもとに、Snowflakeで実行可能なSQLと適切なグラフ種別を生成してください。
```

### 2. スキーマ情報の付与

売上データのテーブル構造をプロンプト内に含めることで、

* 存在しないカラムを参照する誤りの抑制
* 適切な集計軸の選択
* SQL生成精度の向上

を期待しました。

```
# スキーマ情報
テーブル名: SALES
カラム:
- ORDER_DATE (DATE): 受注日
- PRODUCT_NAME (STRING): 商品名
- CATEGORY (STRING): 商品カテゴリ
- SALES_AMOUNT (NUMBER): 売上金額
- QUANTITY (NUMBER): 販売数量
- REGION (STRING): 地域
```

### 3. 出力形式の指定

Bedrockには、以下の項目をJSON形式で出力するよう指示しました。

* SQL
* 推奨グラフ型（例：line / bar / pie）

このように構造化出力とすることで、後続のLambdaでの処理を安定させています。

```
# 出力形式（JSON）
{{
  "sql": "生成したSQL",
  "chart_type": "line または bar または pie"
}}
```

## 実装時に考慮したこと

### Lambdaレイヤーの容量制限

Snowflake接続やグラフ描画を行うためには、外部ライブラリをLambdaレイヤーに含める必要があります。

今回の実装では、主に以下のライブラリを使用しました。

* matplotlib
* snowflake-connector-python

しかし、Lambdaレイヤーには容量制限があるため、必要なライブラリをそのまま追加すると制限に抵触する場面がありました。  
(2025年8月時点では、**250MB**が上限)

そのため、今回は以下の対応を行いました。

* 不要ライブラリの削除
* データ取得用とグラフ作成用でLambdaを分割

これにより、容量制限を回避しつつ、必要な機能を維持する構成としました。

## 最後に

今回は、AWS Bedrockを活用した売上分析AIエージェントの構成と、設計時に考慮したポイントについて紹介しました。

特に重要だと感じた点は、以下の2点です。

* LLMに渡すプロンプトは、役割・スキーマ・出力形式を明示することで安定しやすい
* スキーマ情報の管理方法は、精度と保守性のバランスを見て選択する必要がある

今回はLambda内にスキーマを直接記述する方式を採用しましたが、今後より実運用に近い構成を目指すのであれば、Snowflakeからスキーマ情報を動的に取得する方式も有力な選択肢になると考えています。
