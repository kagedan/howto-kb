# はじめに

## そもそも「MITRE ATLAS」とは？

本題に入る前に、「MITRE ATLAS（マイター・アトラス）」について少しだけ説明します。  
（以前の連載記事をお読みの方や、MITRE ATLASについてすでにご存知の方は飛ばしていただいて大丈夫です！）  
簡単に言うと、これは **「AIシステムに対する実際のサイバー攻撃の手口と、その防御策を体系的にまとめた世界的なナレッジベース（知識データベース）」** です。

MITRE ATLASは単なる脅威の羅列ではなく、攻撃の全体像を論理的に分析できるよう、主に以下の要素で構成されています。

* **タクティクス (Tactics - 目的):** 攻撃者が何を達成したいかという「目的」です。例えば、「Initial Access（システムへの初期侵入）」「Exfiltration（情報の持ち出し）」「Impact（システムへの破壊・妨害）」などがあります。
* **テクニック / サブテクニック (Techniques - 手法):** 目的を達成するための「具体的な攻撃手法」です。「LLM Prompt Injection（AIへの悪意ある指示の混入）」や、データを盗み出す「Exfiltration via AI Inference API（API経由でのデータ抽出）」などがこれに当たります。より細分化されたものをサブテクニックと呼びます。
* **ケーススタディ (Case Studies - 実例):** 実際に世の中で起きたインシデントや、セキュリティ研究者による実証実験の「事例集」です。攻撃者が複数のテクニックをどう組み合わせてAIを騙したかが、ステップごと解説されています。
* **緩和策 (Mitigations - 対策):** 紹介された攻撃手法を防ぐ、または被害を軽減するための「対策」です。
* **マトリクス (Matrix - 全体マップ):** これらを攻撃のライフサイクル順（偵察 → 侵入 → 実行 → 影響など）に表形式で並べた「全体マップ」です。

企業やAI開発者は、このMITRE ATLASを「AIシステムの健康診断ツール」や「設計のガイドライン」として活用できます。

# オープンソースや公開情報からのAIシステム情報収集

AI開発やAIエージェントの普及に伴い、攻撃者が公開リポジトリや構成ファイルを標的としてシステム情報を集めたり、攻撃に必要なリソースを調達したりする活動が活発化しています。

## 属するテクニック：Search Open Websites/Domains (公開ウェブサイト/ドメインの検索 - [AML.T0095](https://atlas.mitre.org/techniques/AML.T0095))

攻撃者が公開ウェブサイトやドメインを検索して、標的の情報を収集する既存の攻撃手法です。  
企業のテックブログや導入事例の記事、PRなどの公開情報を参考にし、標的が使用している技術情報を収集します。

### 追加されたサブテクニック：Code Repositories (コードリポジトリ - [AML.T0095.000](https://atlas.mitre.org/techniques/AML.T0095.000))

GitHubなどの公開コードリポジトリを検索し、標的が使用しているAIフレームワークやモデルの情報を集めるほか、誤って漏洩したAPIキーや認証情報などの機密データを収集する手法が明記されました。  
公開リポジトリでは、意図せずAPIキーなどの機密情報が含まれてしまうケースもあります。自社には関係ないと考えず、公開前の確認プロセスを徹底することが重要です。

### 対策

こうした公開ウェブサイトやドメインからの情報収集を防ぐ基本として、以下の対策が考えられます。

* **Limit Model Artifact Release (モデルアーティファクトの公開制限 - [AML.M0001](https://atlas.mitre.org/mitigations/AML.M0001))**: 本番環境で使用しているデータや技術的詳細の公開を制限します。
* **Limit Public Release of Information (情報の公開制限 - [AML.M0000](https://atlas.mitre.org/mitigations/AML.M0000))**: メタデータや公開アプリケーションから、AI技術の詳細や機密情報が推測されるような情報の公開を制限します。

## 属するテクニック：Acquire Public AI Artifacts (公開AIアーティファクトの取得 - [AML.T0002](https://atlas.mitre.org/techniques/AML.T0002))

攻撃者がクラウドストレージや公開リポジトリなどから、データセットやモデルなどのAI関連の成果物を取得する既存のテクニックです。

### 追加されたサブテクニック：AI Agent Configuration (AIエージェント構成情報 - [AML.T0002.002](https://atlas.mitre.org/techniques/AML.T0002.002))

公開されているAIエージェントの構成ファイル（システムプロンプトや、MCPなどの外部ツール連携設定）を取得し、エージェント機能の把握や認証情報の窃取に悪用する手法が新たに追加されました。

### 対策

AIアーティファクトの取得を防ぐためには、以下の対策が有効です。

* **Limit Public Release of Information (情報の公開制限 - [AML.M0000](https://atlas.mitre.org/mitigations/AML.M0000))**: 機密情報を含む構成ファイルなどの公開を制限します。
* **Limit Model Artifact Release (モデルアーティファクトの公開制限 - [AML.M0001](https://atlas.mitre.org/mitigations/AML.M0001))**: 本番環境で使用しているデータや技術的詳細の公開を制限します。
* **Verify AI Artifacts (AIアーティファクトの検証 - [AML.M0014](https://atlas.mitre.org/mitigations/AML.M0014))**: AIアーティファクトの暗号化チェックサムを検証し、改ざんを防ぎます。

## 生成AIを活用したフィッシングの高度化

生成AIの悪用により、ユーザーを騙してシステムへの初期侵入などを試みる手口がより巧妙化し、なりすましの精度が劇的に向上しています。

### 属するテクニック：Phishing (フィッシング - [AML.T0052](https://atlas.mitre.org/techniques/AML.T0052))

電子的なメッセージを通じて行われるソーシャルエンジニアリングの既存テクニックです。従来から存在する攻撃ですが、AIの活用でさらに進化しています。

#### 追加されたサブテクニック：Deepfake-Assisted Phishing (ディープフェイクを活用したフィッシング - [AML.T0052.001](https://atlas.mitre.org/techniques/AML.T0052.001))

AIで生成された音声や動画などのディープフェイクを用いて、信頼できる有名人や企業幹部などになりすまし、認証情報の開示や送金などを促す攻撃が追加されました。  
被害者が実際の業務指示と誤認し、認証情報の入力や送金承認などを実行してしまうシナリオが想定されます。特に電話や音声メッセージのように本人確認が難しいチャネルでは、被害につながりやすくなります。

#### 対策

高度化するディープフェイク・フィッシングへの対策として、今回のアップデートで以下の既存緩和策の内容も更新されています。

* **Deepfake Detection (ディープフェイク検知 - [AML.M0034](https://atlas.mitre.org/mitigations/AML.M0034))**: AIモデルや生体認証の分析（まばたきや目の動きなど）を利用し、生成されたディープフェイクコンテンツを検知してブロックします。
* **User Training (ユーザー教育 - [AML.M0018](https://atlas.mitre.org/mitigations/AML.M0018))**: ユーザーがディープフェイクやフィッシングの試みを識別できるようにトレーニングを行います。

## まとめ

MITRE ATLAS v5.6.0のアップデートでは、自律型AIエージェントの普及に伴う「構成情報の漏洩リスク」と、生成AIの進化によって極めて巧妙になった「ディープフェイク・フィッシング」という、まさに今のトレンドを反映した脅威が定義されました。  
特に **Code Repositories** のサブテクニックでは、近年のAIコーディングシステムの発展により、意図しない箇所でAPIキーなどの機密情報が外部公開されたり、学習に利用されたりするリスクも存在します。こうした便利さの裏にあるリスクを前提に、生成AIを利活用・運用することが重要です。

AIエージェントやMCPを運用する際は、以下の3点を最低限チェックすることを推奨します。

1. **公開前のシークレットチェックを必須化する:** リポジトリ公開前に、APIキーや認証情報が含まれていないかを自動スキャンし、検出時は公開を止める運用にします。
2. **エージェント構成ファイルの公開範囲を見直す:** システムプロンプト、ツール連携設定、認証情報の参照先を棚卸しし、公開不要な情報をリポジトリやドキュメントから除外します。
3. **高リスク操作に本人確認フローを入れる:** 送金、認証情報の入力、機密ファイル共有などは、電話折り返しや別チャネル確認などの二重確認を必須にします。

本記事で示した対策はMITRE ATLASの公式推奨ではなく、公開情報を踏まえた考え方の一例として紹介しています。あらかじめご了承ください。
