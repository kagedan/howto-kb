## 🚀 AIエディター「Kiro」活用術：実録・「忘れた設定」も何度でも。仕様変更の荒波をAIと越える

### 1. はじめに：エンジニアの敵は「時間の経過」と「UIの変更」

「以前設定したはずなのに、やり方を忘れてしまった」  
「ドキュメント通りにやっているはずなのに、なぜか動かない」

開発現場ではよくある光景です。特にクラウドサービスのUIは日々進化しており、数ヶ月前の知識が通用しなくなることも珍しくありません。  
本記事では、Webサイト監視ツール（WebSiteLifeChecker）のSupabase連携を再開した際に起きた、\*「サービス側の仕様変更によるサイレントな失敗」\*を、AIエディターがいかに粘り強く解決したかの実録をお届けします。

### 2. 「何度聞いても怒らない」最強のインストラクター

時間が空いて設定手順を忘れてしまった時、同僚に何度も同じことを聞くのは気が引けるものです。しかし、AIに遠慮は不要です。

#### 👤 ユーザー: 「Supabaseの設定したいので教えてください（忘れてしまいました）」

#### 🤖 Kiro: 「承知しました。プロジェクト作成からテーブル定義、APIキーの取得まで、ステップバイステップで説明しますね。」

Kiroは、以前のセッション内容を文脈として保持しつつ、現在の最新手順を丁寧に再提示してくれました。「同じことを100回聞いても、常に初回の熱量で答えてくれる」――これはAI開発における最大の心理的安全性です。

### 3. インシデント：謎の「サイレント・フォールバック」を追え

設定を終え、APIも200（成功）を返している。しかし、Supabaseのテーブルには一行もデータが入らない。この時、プロジェクトでは恐ろしい現象が起きていました。

巧妙な「親切設計」が牙を向く  
実はこのアプリには、「DB接続に失敗しても、アプリを落とさないためにインメモリで動作を継続する」というフォールバック機能が備わっていました。これが災いし、\*「エラーは出ないが、データも保存されない」\*という、原因特定が極めて困難な状態に陥ったのです。

AIが見抜いた「Supabaseの仕様変更」  
Kiroは即座にデバッグログを仕込むコードを生成・プッシュし、Vercelのサーバーログを解析。そして、ある「仕様変更」の可能性を指摘しました。

#### 🤖 Kiroの洞察

「Supabaseの新しいAPIキー形式（sb\_secret\_...）を使っていませんか？ 実はライブラリのバージョンによっては、Legacy（以前のJWT形式）のキーでないと認証に失敗するケースがあります。」

まさにこれが正解でした。ユーザーが手動で「Legacy」タブから取得した古い形式のキーに差し替えた瞬間、データがテーブルに流れ込み始めました。

### 4. 結論：AIは「変わらない親切」と「変わり続ける技術」の架け橋

今回のセッションで得られた最大の収穫は、以下の2点です。

「あ、忘れた」「あ、UI変わってる」。そんな小さな絶望に直面した時こそ、AIエディターを頼ってみてください。彼らは何度でも、あなたが正解に辿り着くまで並走してくれます。

### 🤖 AI Editor

Supabase の設定手順を説明します。

**1. Supabase プロジェクト作成**

1. <https://supabase.com> にアクセスしてログイン（GitHub アカウントでOK）
2. New Project をクリック
3. 以下を入力：
   * Project name: `website-life-checker`（任意）
   * Database Password: 強力なパスワードを設定（後で使わないが控えておく）
   * Region: Northeast Asia (Tokyo) がおすすめ
4. Create new project をクリック（作成に2分ほどかかる）

**2. テーブル作成**

プロジェクトが作成されたら：

1. 左メニューの SQL Editor をクリック
2. 以下のSQLを貼り付けて Run を押す：

```
-- 監視対象テーブル
CREATE TABLE IF NOT EXISTS monitor_targets (
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name       TEXT NOT NULL,
  url        TEXT NOT NULL,
  timeout    INT NOT NULL DEFAULT 10,
  expected_status INT NOT NULL DEFAULT 200,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- ヘルスチェックログテーブル
CREATE TABLE IF NOT EXISTS health_logs (
  id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  target_name   TEXT NOT NULL,
  is_healthy    BOOLEAN NOT NULL,
  response_time DOUBLE PRECISION NOT NULL DEFAULT 0,
  error_message TEXT,
  checked_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_health_logs_target_checked
  ON health_logs (target_name, checked_at DESC);

CREATE INDEX IF NOT EXISTS idx_health_logs_checked_at
  ON health_logs (checked_at DESC);
```

このSQLは `migration.sql` にも入っています。

**3. API キーを取得**

1. 左メニューの Settings（歯車アイコン）→ API
2. 以下の2つをコピー：
   * Project URL → `https://xxxxxxxxxx.supabase.co` の形式
   * service\_role キー（secret と書いてある方）→ 目のアイコンで表示

**4. Vercel に環境変数を設定**

1. Vercel ダッシュボード → プロジェクト → Settings → Environment Variables
2. 以下を追加：

| Name | Value |
| --- | --- |
| `SUPABASE_URL` | `https://xxxxxxxxxx.supabase.co`（手順3でコピーしたURL） |
| `SUPABASE_SERVICE_ROLE_KEY` | 手順3でコピーした service\_role キー |

3. Save を押す
4. Deployments タブ → 最新のデプロイで Redeploy をクリック（環境変数を反映するため）

**5. 動作確認**

Redeploy 完了後、ダッシュボードで「今すぐチェック」を押して、Supabase の Table Editor → `health_logs` テーブルにデータが入っていれば成功です。

何かつまずいたら教えてください。
