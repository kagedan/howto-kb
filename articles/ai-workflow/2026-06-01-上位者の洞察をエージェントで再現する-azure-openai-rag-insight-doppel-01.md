# 上位者の「本人らしさ」をAIで再現する ― Insight Doppel の設計と実装

## 1. 背景

社内外を問わず、重要な意思決定が求められる場面では上位者に判断を仰ぐ必要があります。しかし、多忙であることが多く、論点整理や相談のたびに直接確認することが難しい場面も少なくありません。

従来は、上位者への直接相談に代わる手段として、既存の生成AIを相手に壁打ちをするにとどまっていました。しかし、この方法だけでは効率的な意思決定に繋がっていないのが現状です。

この問題の背景には、以下のような課題があります。

* 上位者の判断基準が明文化・共有されておらず、属人的なものにとどまっている
* 既存の生成AIでは上位者の視点でのフィードバックが得られず、壁打ちの質に限界がある
* 上位者本人特有の評価観点や判断の癖を再現する手段がなく、相談者側で「何を聞かれるか」「何が不足しているのか」を事前に把握できない

その結果、相談者は質問や資料を十分に磨き込めないまま会議に臨むことになり、上位者の確認・修正の負担が大きくなりやすい状況が生じています。

一方で、あらかじめ上位者の意思決定ロジックを模倣したAIを相手に壁打ちができれば、相談者は上位者が重視する論点を押さえたうえで、資料の精度を高めた状態で会議に臨むことができます。これにより上位者も、品質の高い情報を基に重要な判断を下すことができるというメリットがあります。

こうした背景から、本プロジェクトでは上位者の過去資料やレビュー傾向をもとに「本人らしさ」をペルソナとして構造化し、質問の壁打ちや資料レビューを事前に行えるAI――**Insight Doppel**――を開発しました。

## 2. Insight Doppel の概要 & デモ

### Insight Doppel とは

個人のコミュニケーション履歴（Teams チャット・Outlook メール・アップロードファイル）を学習データとして活用し、「その人であれば、どのように判断し、どのように発言するか」を疑似再現する **RAG ベースの個人ペルソナAIアプリケーション**です。

### Insight Doppel という名前

**Doppel** はドイツ語で「分身」を意味します（Doppelgänger の語源）。**Insight** は「洞察」。

ただし、上位者の分身を作ることではありません。上位者が持つ**洞察の型**――何を重視し、どのように判断し、どこにリスクを見出すのか――を再現したい。Insight Doppel が目指すのは、人格の複製ではなく、**意思決定の洞察を引き継ぐ仕組み**です。

主な機能は以下のとおりです。

* 個人のデータからペルソナを自動生成（心理学・行動科学の研究指標ベース、5軸）
* 多様なファイル形式に対応（PDF・Word・PowerPoint・画像等。スキャンPDFはAI Visionで自動OCR）
* ペルソナを構造化し Cosmos DB に永続化。再抽出時にはスナップショットを保存し、プロファイルの変化を差分比較可能
* 3つの利用モード：**質問モード**（単発の意見出し）、**レビューモード**（成果物への構造化レビュー）、**意見交換モード**（マルチターン対話）
* **SKILL.md エクスポート**：Claude Agent SDK 互換の形式で他アプリからも再利用可能
* 本人承認を前提としたデータ利用（倫理ガードレール常時表示）

Insight Doppel は、上位者の判断ロジックを内在化した**エージェント**として振る舞います。単に質問に答えるだけでなく、レビューモードではドキュメントの問題点を自ら指摘し、意見交換モードでは対話を通じて論点を深掘りする。これにより相談者は**上位者の分身に事前相談しているかのような体験**を得ることができます。

### 処理フロー

① ユーザー認証・同意取得 → ② 個人データ入力・収集（ファイル / Graph API） → ③ 二系統マスキング・前処理 → ④ AIによるペルソナ生成（5軸抽出） → ⑤ ペルソナの構造化・保存 → ⑥ 壁打ち・レビュー・意見交換 → ⑦ SKILL.md エクスポート

ポイントは、単なる文章生成ではなく、**再利用可能な「人格モデル」としてペルソナを扱う**点です。

現時点で複数のペルソナが稼働しており、実際のコミュニケーション履歴から200件以上のドキュメントを取り込んだペルソナでの壁打ちが可能な状態です。

## 3. 技術構成

### 3.1 アーキテクチャ

Insight Doppel は Azure 上に構築しています。

| レイヤー | 技術 |
| --- | --- |
| LLM（生成） | Azure OpenAI `gpt-5.4-mini` |
| LLM（Embedding） | Azure OpenAI `text-embedding-3-small`（512次元） |
| バックエンド | FastAPI（Python） |
| フロントエンド | Next.js（App Router）＋ ABeam Design System |
| 検索エンジン | Azure AI Search（Basic）― ハイブリッド検索（Vector + BM25 + Semantic Ranker） |
| データストア | Azure Cosmos DB（Serverless）＋ Azure Blob Storage |
| コンテナ | Docker → Azure Container Apps（Consumption） |
| 認証・権限 | Microsoft Entra ID（OAuth 2.0 / OBO フロー）＋ Azure Key Vault（Managed Identity） |
| 監査ログ | Azure Monitor / App Insights / Log Analytics（Basic Logs、30日保持） |

PoC段階での過剰投資を避けるため、コストを意識した構成にしています。Embedding は `text-embedding-3-small` の512次元を採用し、標準（1536次元）比でコストを約1/4に抑えています。監査ログは Basic Logs（標準の約1/5価格）、Cosmos DB は Serverless（従量課金）、AI Search は Basic tier を採用し、スケーラビリティを確保しつつ初期コストを最小化しています。

### 3.2 実装上の工夫

#### ペルソナの再利用性：SKILL.md エクスポートと組織横断の展開

Insight Doppel で生成したペルソナは、アプリ内での利用にとどまらず、**Claude Agent SDK の Skill 規約に準拠した SKILL.md ファイル**としてエクスポートできます。

* エクスポートされた SKILL.md には、意思決定軸・口調・過去判断事例に加え、後述する「その人らしさ」5軸のプロファイルが含まれる
* 他のアプリケーション（社内チャットボット・コードレビュー支援・ナレッジ検索等）でペルソナをそのまま再利用可能
* 倫理ガードレール文言（「壁打ち専用」「本人未確認」）も SKILL.md の冒頭に必ず同梱

これにより、ペルソナはアプリ内に閉じた一時データではなく、**持ち運び可能な資産**になります。単発の成果物ではなく、複数の社内AIソリューションで同一のペルソナを呼び出し、「本人らしい応答」を一貫して提供できる。Insight Doppel を**単独のWebアプリの中の機能**から**組織横断のペルソナ基盤**へ昇格させることを見据えた設計です。

#### 3つの利用モード

用途に応じて3つのモードを切り替えられます。

| モード | 概要 | コンテキスト保持 |
| --- | --- | --- |
| **質問モード** | 単発の意見出し。「この方針どう思いますか？」 | なし |
| **レビューモード** | ドキュメントや成果物に対する構造化レビュー（指摘箇所・観点・推奨アクション・重要度） | レビュー対象を72時間保持 |
| **意見交換モード** | マルチターンの対話。論点を深堀りしながら壁打ち | 会話履歴を7日間保持 |

共通の不変条件（引用元・確信度・「本人未確認」ラベル・倫理ガードレール）はモードに関係なくすべての応答に必ず付与されます。

#### セキュリティ・権限設計

個人情報を扱うシステムのため、以下の仕組みを組み込んでいます。

* **OBO フロー（委任権限のみ）**：Graph API は本人スコープに限定。アプリ権限（application permission）は一切使わない
* **二系統マスキング**：個人情報系（個人名・メアド・電話番号等）と業務機密系（顧客名・案件名・プロジェクトコード等）をそれぞれ独立したルールで処理。業務機密の辞書は外部設定で案件ごとに拡張可能
* **本人同意の必須化**：ペルソナ生成には必ず本人の承認が前提。「本人が許可しない限り動かない」設計
* **横断削除（忘れられる権利）**：Blob・AI Search・Cosmos DB・App Insights を横断して1ジョブで削除可能

#### 展開性を見据えた構成

社内外への展開を見据え、環境非依存の構成としています。

* **全コンポーネントをDockerコンテナ化**：Backend（FastAPI）とFrontend（Next.js）をそれぞれコンテナとしてパッケージ化し、Azure Container Apps 上で稼働
* **インフラ構成をBicepでコード化（IaC）**：別のサブスクリプション・テナント・リージョンへの再構築は、同じテンプレートを流すだけで再現可能
* **設定値・シークレットを外部化**：環境変数と Key Vault に外出しすることで、コードは環境非依存

この構成により、社内本番テナントへの昇格、関連プロジェクトへの横展開が、アプリ本体に手を入れずに可能です。

#### AI駆動開発による実装コスト最小化

本プロジェクトは、AIを要件定義・設計・実装の全フェーズに活用した**AI駆動開発**で実施しました。要件定義書と設計書をAIと同期的に生成する**仕様駆動の仕組み**を取り入れることで、Claudeライセンス費用を除けば**約2人日**でプロトタイプを構築しました。

この開発スタイルにより、実装コストの削減だけでなく、社内のアーキテクチャレビュー用ドキュメントや本記事のような技術発信資料の作成コストも同時に低減することができました。

### 3.3 プロンプトなどの工夫

ここが Insight Doppel の核心です。背景で述べた「上位者の判断基準が属人的で、既存AIでは再現できない」という課題に対し、Insight Doppel は**心理学・行動科学の研究知見に基づく5つの軸**でペルソナを構造化するアプローチで応えています。

| 背景の課題 | Insight Doppel の解決策 |
| --- | --- |
| 判断基準が属人的で明文化されていない | 研究指標ベースの5軸で構造化・数値化し、明示的なプロファイルとして保存 |
| 既存AIでは上位者の視点を再現できない | 5軸プロファイル＋スタイルメモをシステムプロンプトに注入し、本人の判断ロジックを再現 |
| 「何を聞かれるか」を事前に把握できない | 質問類似度ベースの動的few-shotで、その質問に関連する本人の過去判断を参照して応答 |

#### 「本人らしさ」を研究指標ベースの5軸に分解

| 軸 | 理論的背景 | 概要 | 抽出形式 |
| --- | --- | --- | --- |
| **Big Five（OCEAN）** | 性格心理学 | 開放性・誠実性・外向性・協調性・神経症傾向の5因子 | 各 0.0〜1.0 のスコア |
| **Schwartz Basic Human Values** | 価値観研究 | Self-Direction・Achievement・Benevolence 等 10基本価値 | 各 0.0〜1.0 ＋ 上位3価値を明示 |
| **GDMS（一般意思決定スタイル）** | 意思決定研究 | Rational・Intuitive・Dependent・Avoidant・Spontaneous の5スタイル | 各 0.0〜1.0 ＋ 支配的スタイルを明示 |
| **Narrative Identity** | McAdams 第3層 ＋ 日本看護「その人らしさ」概念 | 重要体験・人生テーマ・口癖・関係性のアンカー | 構造化テキスト |
| **Enneagram（エニアグラム）** | Riso-Hudson 9タイプ分類 | 9つの性格タイプ＋ウィング（隣接タイプ） | 各 0.0〜1.0 ＋ 優勢タイプ＋ウィング |

5つ目のエニアグラムは、社内で「自分らしさ指標」として既に運用されている枠組みに合わせて採用しました。UIではレーダーチャートで可視化し、各タイプの説明とともに確認できます。

この5軸に加え、従来のスタイルメモ（意思決定軸・口調・頻出語彙・過去判断事例）を組み合わせることで、表面的な口調の模倣ではなく**判断のロジックや重視するポイントまで含めた「本人らしさ」を再現しています。**

#### どうやって抽出するか

上位者の過去コミュニケーション（Teams チャット・Outlook 送信メール・アップロードファイル）を取り込み、二系統マスキング（個人情報系＋業務機密系）を施した上で、Azure の `gpt-5.4-mini` の **few-shot 分類**で各軸のスコアを抽出します。

なお、口調・頻出語彙の抽出には**会話素材（メール・チャット）のみ**を使用しています。PDF等の資料は判断事例の抽出には活用しますが、口調の学習には含めていません。これにより、資料特有の硬い文体を本人の話し方として誤学習することを防いでいます。

各スコアには根拠となった発言の該当箇所が併記され、「なぜこのスコアになったか」をペルソナオーナーがチャンク単位で逆引きできます。また、抽出に使ったサンプル数が閾値（50メッセージ）未満の場合は信頼度を低位として表示し、データ不足であることを明示します。

#### プロンプト設計

推論時のシステムプロンプトには、以下の要素を注入しています。

**① Kitwood VIPS を設計原則として組み込み**

認知症ケア領域で確立された **Kitwood の Personhood / VIPS**（Values・Individualised・Perspective-taking・Social environment）を設計規範として採用しました。システムプロンプトに以下のようなガードレールを注入することで、応答が「本人視点の尊重・価値の保護」から外れないようにしています。

```
【Kitwood VIPS 規範】
- Values: 本人が大切にしている価値（Schwartz 上位3価値 + 意思決定軸）を尊重する
- Individualised: 本人のスタイル（GDMS支配型 = rational、口調 = 敬体・断定型）を
  画一化せず保持する
- Perspective-taking: 本人の視点・経験的文脈から応答する。一般論で逃げない
- Social environment: 本人が置かれた業務文脈に配慮する
```

**② 本人の一人称で応答（第三者視点の禁止）**

応答は必ず**本人の一人称（「私は〜」「自分としては〜」）で生成**するよう、システムプロンプトで強制しています。「〇〇さんならこう言うでしょう」といった第三者視点や推測形での応答は禁止しています。これにより、相談者は「上位者本人に相談している」感覚で壁打ちでき、エージェントとしてのリアリティが大きく高まります。

**③ 質問類似度ベースの動的 few-shot（固定top-3ではない）**

これが Insight Doppel のキモです。過去判断事例は固定の上位3件ではなく、**質問のたびに embedding 類似度で都度選定**します。

仕組みとしては、過去判断事例を通常の発言チャンクとは区別して同一インデックス内に格納しておき、推論時に質問の embedding に対する vector search で類似度 top-N を取得して few-shot に注入します。

```
【過去判断事例（質問類似度で動的選定）】
事例1 (類似度 0.82):
  状況: PoCスコープがブレた状態で着手判断を求められた
  決定: 1週間以内のスコープに限定して着手
  理由: 属人化解消とデモ可能性の両立

事例2 (類似度 0.71):
  状況: リスク評価の優先順位付けを求められた
  ...
```

これにより、**その質問に最も関連する本人の過去判断**を参照しながら応答を生成できます。類似度がしきい値未満の場合は直近の判断事例にフォールバックし、その旨をログに記録します。

**④ 構造化 JSON 出力の強制**

応答は必ず以下のスキーマで返却させ、バリデーションで品質を保証しています。

```
{
  "body": "応答本文",
  "citations": ["chunk-abc", "chunk-def"],
  "few_shot_case_ids": ["case-001", "case-014"],
  "confidence": 0.78,
  "unverified_label": true,
  "ethical_disclaimer": "壁打ち・事前レビュー専用。..."
}
```

引用元（citations）が0件の場合は本文を破棄し、「該当する過去発言が見つかりません」とだけ返します。**無根拠の応答生成は行わない**という設計思想です。

## 4. 今後の展望

本PoCを踏まえ、今後は以下の方向を検討しています。

* **データソースの拡大**：受信メール・Teams全チャネル履歴の取り込み（第三者同意プロセスの整備後）
* **「その人らしさ」指標の追加**：Moral Foundations（道徳基盤理論）、Risk-Time Preference（リスク・時間選好）等の追加を検討
* **ペルソナ精度の継続的な改善**：ゴールデンセット（20件の既知回答）による自動評価をCIに組み込み、スタイル一致率を継続監視
* **Skill エクスポート/取得のAPI標準化**：REST API としてペルソナの取得・エクスポートを標準化し、複数の社内アプリやエージェントが同じペルソナを呼び出せる**組織横断のペルソナ基盤**への発展を目指す
* **組織ナレッジとしての活用**：個人のペルソナを組織の意思決定支援基盤として展開。ペルソナ間比較・複数ペルソナ同時応答も次のステップとして検討

最終的には、Insight Doppel を個人の意思決定や思考を拡張するAI基盤として発展させることを目指しています。

## 5. まとめ

Insight Doppel は、上位者の「本人らしさ」を再現可能なペルソナとして構造化し、相談者が事前に壁打ちや資料レビューを行えるAIアプリケーションです。

心理学・行動科学の研究知見（Big Five・Schwartz Values・GDMS・Narrative Identity・Enneagram）に基づく5つの軸でペルソナを構造化し、Kitwood の Personhood / VIPS を設計原則として応答品質を担保する。質問のたびに類似度ベースで過去判断事例を動的選定する仕組みにより、**汎用LLMのみでは再現できない「その人ならではの判断ロジック」を引き出す仕組みです。**

このサービスが提供する価値は、**「上位者への相談コストを下げ、意思決定の質を高める」** ことです。相談者は事前に論点を磨き、上位者は質の高いインプットをもとに判断を下すことができる。**双方の時間と意思決定の質を同時に改善することが可能になります**  
本プロジェクトの出発点は、「上位者が忙しく、気軽に相談できない」という日常的な課題でした。しかし開発を進める中で、次第により本質的な問いへと発展していきました。

人の判断の「型」は、保存し、引き継げるものなのか？

退職や異動によって不在となった人の意思決定スタイルをどのように再現・活用するかという視点から、意思決定の構造化に関する設計原則の探求が始まりました。そこにある問いは一貫しています。すなわち、「**その人がいなくなった後も、その人の洞察を活かせないか**」というものです。  
もちろん、Insight Doppel は人格そのものを再現するものではありません。あくまで意思決定の傾向を構造化し、壁打ちの質を高めるためのツールです。しかしその先には、「属人化されたナレッジをいかに継承するか」という、組織にとって普遍的かつ重要なテーマが存在しています。  
このハッカソンでの取り組みは、その問いに対する最初の一歩です。
