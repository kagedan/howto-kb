## Stage 1：ルールエンジン（TypeScript）

### シグナル変化検出が核心

エージェントのメインループは `HermesAgent.analyse()` です。**シグナルが変化したときだけ Stage 2（Claude）を呼ぶ**のがコスト削減の鍵です。

```typescript
// hermes.ts（抜粋）
const engineKey  = `${symbol}:${params.engineType}:${params.engineIndex}`;
const prev       = this.lastAction.get(engineKey); // 前回のシグナル
const preliminary = generateSignal(indicators, bars);
const actionChanged = prev !== preliminary.action;

// HOLD→BUY, BUY→SELL などの変化時だけ Claude を呼ぶ
if (this.cfg.agentValidation && actionChanged && preliminary.action !== "HOLD") {
  const decision = await this.runAgentDecision(bars, indicators, preliminary, events);
  // ...
}
```

`engineKey` には `"META:structural:0"` のようにエンジンインデックスも含めています。これは後述するバグ修正で追加した仕組みです。

### シグナル生成ロジック（BaseStrategy）

`BaseStrategy.calculateSignal()` がすべてのエンジンの基盤です：

```typescript
// base.ts（抜粋）
// SELL条件スコアリング
if (rsi > rsiOverboughtStrong) {
  sellScore += 2; // 強い過熱
} else if (rsi > rsiOverboughtModerate && smaFast < smaSlow && price < smaFast) {
  sellScore += 1; // 下降トレンド中の過熱
}
if (price < ema && smaFast < smaSlow && !deathCross) {
  sellScore += 1; // 下降トレンド確認
}

// sellScore >= 2 かつ buyScore を上回ったら SELL
const isSell = sellScore >= 2 && sellScore > buyScore;
```

閾値（`rsiOverboughtStrong` など）はシンボルごとにバックテストで最適化した値が `strategy-params.json` に入っています。例えば META は `rsiOverboughtStrong: 60`、EUR_USD は `rsiOverboughtStrong: 70` と異なります。

### 7エンジンのうち特徴的なもの

**Rebound エンジン**（ボリンジャーバンド端からのリバウンド）：

```typescript
// ReboundStrategy.ts（抜粋）
protected buildSignal(ind: Indicators, bars: Bar[]): TradingSignal | null {
  const { price, rsi, bbLower, bbUpper, bbMiddle, atr } = ind;

  const atLower = price <= bbLower && rsi < rsiOversoldStrong;
  const atUpper = price >= bbUpper && rsi > rsiOverboughtStrong;

  if (atLower) {
    // TP = BB中央線（平均回帰）
    return this.buildSignalWith("BUY", price, bbMiddle - price, ...);
  }
  if (atUpper) {
    return this.buildSignalWith("SELL", price, price - bbMiddle, ...);
  }
  return null; // 基底クラスのスコアリングへ
}
```

**Pullback エンジン**（トレンド方向への押し目）：

```typescript
// PullbackStrategy.ts（抜粋）
const inUptrend   = smaFast > smaSlow;
const nearFastSMA = Math.abs(price - smaFast) / smaFast * 100 < trendingThreshold;
const rsiRecovered = rsi >= rsiEntryLevel;
const priceRising  = bars.at(-1).close > bars.at(-2).close;

if (inUptrend && nearFastSMA && rsiRecovered && priceRising) {
  action = "BUY";
}
```

## Stage 2：Claude 判断エンジン

### Claudeに渡すコンテキスト

`buildContextText()` でClaudeへの入力を組み立てます。含まれる情報：

1. **直近5本のローソク足**（OHLCV + 変化率）
2. **テクニカル指標**（SMA, RSI, ATR, BB幅）
3. **シグナル詳細**（エンジン種別・SL/TP・RR）
4. **ATRラベル**（NORMAL/HIGH/VERYHIGHで銘柄特性に応じた判断）
5. **経済指標カレンダー**（直近のCPI/FOMC等）
6. **過去トレード記録**（同シンボル・同方向・類似RSIの勝率）

### Claudeへのシステムプロンプト設計

```
EXECUTION CRITERIA (ALL must hold for EXECUTE):
  1. Signal direction is consistent with overall candle structure
  2. Candlestick patterns support (or do not contradict) the direction
  3. Memory patterns show no strong reason to doubt this setup
  4. Engine type is appropriate for current market behaviour
  5. Backtest SQN > 1.0 = good, > 2.0 = excellent

WAIT when:
  • Signal contradicts the dominant candle structure
  • Growth/momentum stocks (NVDA, TSLA, META): ATR > 5.0% of price
  • Phantom calibration shows you have been too aggressive
```

Claude には「なぜそう判断したか」を 2-3 文で書かせます：

```json
{
  "decision": "WAIT",
  "direction": null,
  "stop_loss": null,
  "take_profit": null,
  "confidence_pct": 25,
  "reasoning_brief": "Structural engine fired SELL on RSI 64 overbought,
    but the last bar is a strong bullish marubozu AND bullish engulfing
    pattern with price closing above the upper Bollinger Band on heavy
    volume (1.4M). Shorting into clear bullish momentum acceleration is
    high-risk — waiting for momentum exhaustion."
}
```

## 実装上のハマりポイントと解決策

### 問題1：同じシグナルが毎5分 Claude を呼んでいた

**原因**：同一シンボルに同じ `engineType` のエンジンが2つある場合（例：META の `structural` ×2）、`engineKey = "META:structural"` が衝突。

- エンジン1（SMA20/50）: SELL → WAIT → `lastAction["META:structural"] = "SELL"`
- エンジン2（SMA50/100）: HOLD → `lastAction["META:structural"] = "HOLD"`（上書き！）
- 次サイクル: エンジン1が `prev="HOLD"` を見て `actionChanged=true` → Claude 再呼び出し

**修正**：`engineKey` を `${symbol}:${engineType}:${engineIndex}` に変更。

```typescript
// 修正前
const engineKey = `${symbol}:${params.engineType}`;

// 修正後
const engineKey = `${symbol}:${params.engineType}:${params.engineIndex}`;
```

`engineIndex` は `getEngineConfigs()` で配列インデックスを付与します。

### 問題2：Alpaca 403 insufficient buying power

**原因**：`equity`（含み益込みの総資産）でポジション上限株数を計算していたため、既存ポジションで buying power が少ない状態でも大きな注文が生成されていた。

```typescript
// 修正前（問題あり）
const maxByEquity = Math.floor(equity / entry);  // equity は含み益込み

// 修正後
const capital = buyingPowerCap ?? equity;  // Alpaca の buying_power を使う
const maxByCapital = Math.floor(capital / entry);
```

`account-tools.ts` が `buying_power` フィールドを返していたにもかかわらず、`hermes.ts` が `equity` だけを参照していたのが根本原因でした。

## シグナル状態の永続化

エージェントが再起動したとき、`lastAction` の状態（`"META:structural:0": "SELL"` など）を `logs/last-action.json` から復元します。これがないと、再起動のたびにすべての `actionChanged=true` になり Claude を大量呼び出ししてしまいます。

```typescript
private loadLastAction(): void {
  if (!existsSync(LAST_ACTION_PATH)) return;
  const obj = JSON.parse(readFileSync(LAST_ACTION_PATH, "utf8"));
  for (const [key, act] of Object.entries(obj)) {
    this.lastAction.set(key, act as SignalAction);
  }
}
```
