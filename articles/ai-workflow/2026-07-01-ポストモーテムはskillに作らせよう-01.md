## はじめに

こんにちは、最近ClaudeのSkillを作るのにハマっているLapi（[@dragoneena12](https://github.com/dragoneena12)）です。

最近チームでポストモーテム作成用のSkillを作ってみたのですが、結構いい感じに作れてチーム内での評判も良かったので、こちらで紹介してみたいと思います。

## ポストモーテムとは

ポストモーテムとはGoogleのSite Reliability Engineeringで提唱されている、インシデントについての事後分析レポートです。

インシデントの影響や原因、再発防止策などをまとめた文書を作成し組織全体に共有することで、知識の蓄積を図り、効果的な予防措置をとることを目的とします。詳細については以下のSRE Bookを参照してください。

<https://sre.google/sre-book/postmortem-culture/>

## つくったSkill

弊社でもインシデントの後は必ずポストモーテムを作成し、知識の共有や予防措置を実行する文化があります。しかしポストモーテムの作成にはこれまでかなりの時間がかかってきました。たとえばポストモーテム内には事象を正確に把握するため詳細なタイムラインを書く必要があるのですが、事実関係をGitのコミット記録やDatadogのログ、Slackのやりとりなど各所を確認しながらまとめるのはそれだけで大変な手間がかかります。さらにポストモーテムのレビュー会議において事実関係を確認したり再発防止策を策定したりすると、会議がかなり長時間に及んでしまう問題がありました。

上記のような作業を効率化するため、ポストモーテム用のSkillを作成しました。Skillの流れとしては以下のとおりです。

インシデント対応のJIRAチケットを起点に、JIRA・GitHub・Datadogの情報を収集して自動でタイムライン・根本原因・影響範囲・検知/復旧時刻などを取りまとめてくれます。不明点がある場合はAskUserQuestionツールを使って質問し、ドラフトを提示してユーザーからフィードバックを得るように工夫しています。最後に会社全体に共有するため、Notionに投稿するまでを行ってくれます。

ポストモーテムのテンプレートとしてはGoogleが提唱するポストモーテム例を参考に、以下のようなものを使用しています。

```
# <障害の短い説明>

## 概要

<!--
このセクションだけ読めば障害の全体像がわかるように、各子セクションを 1〜2 文で簡潔に書く。
詳細は後続の章に書き、ここでは繰り返さない。
-->

### 期間

### ユーザー影響

### 検出

<!-- どう気づいたかに加えて、TTD（発生から発覚までの経過時間）を記載する -->

- TTD:

### 解決

<!-- どう収束させたかに加えて、TTR（検出から解決までの経過時間）を記載する -->

- TTR:

### 根本原因

## 影響

<!--
影響を受けた顧客・機能・期間・件数。
Datadog の集計から得た数値は、出典（クエリ・集計期間）を併記する。
-->

## 根本原因とトリガー

<!--
技術的な因果関係を「何が・なぜ」で書く。個人名を原因として書かない。
あわせて、障害を顕在化させた直接のきっかけ（デプロイ・データ・外部要因など）も書く。
根本原因（潜在していた欠陥）とトリガー（それを顕在化させたきっかけ）は区別がつくように書き分ける。
-->

## タイムライン

<!--
検知 → 対応 → 復旧の時系列を表形式（日時 / 出来事）で記載する。日時はすべて JST。
検知の経緯（アラート / 顧客問い合わせ / 開発者が偶然発見 など）と、収束までに実施した対応もこの時系列の中に書く。
出典（JIRA コメント・PR マージ時刻など）がある事実のみ記載し、空白は TODO: として残す。
人名は対応者として登場させてよい。
-->

| 日時（JST） | 出来事 |
|---|---|
| | |

## 教訓

### うまくいったこと

### うまくいかなかったこと

### 幸運だったこと

## アクションアイテム

<!--
種別は 予防 / 検知 / 緩和 のいずれか。
優先度は 高 / 中 / 低 のいずれか。
トリアージは実行するかどうかの判断（実施 / 見送り / 未定）。ドラフト作成時は「未定」とし、振り返りで決める。
チケット未作成のものは担当・チケット欄を TODO とする。
起票済みの JIRA はチケット URL へのリンクとして記載する。
-->

| 内容 | 種別 | 優先度 | トリアージ | 担当 | JIRA |
|---|---|---|---|---|---|
| | | | | | |
```

このテンプレートにもいくつか工夫した点があります。ひとつは、振り返りの観点を予防策だけに限らず、検知・復旧・影響範囲の軽減についても議論できるようにした点です。振り返りはどうしても予防策に意識が向きがちですが、TTD・TTRや影響範囲の改善も同じく重要なので、これらを自然に扱えるよう章立てを工夫しました。もうひとつは、アクションアイテムに「トリアージ」の列を設けた点です。改善案はつい数が多くなり宙に浮いてしまいがちなので、「実施 / 見送り」を必ず記録するようにしました。

## 効果とまとめ

実際にこのSkillを使ってこれまでに3回ポストモーテムを作成しましたが、手で状況を調べる必要はなく、すべてSkillだけで作成でき、かなりの時間短縮ができました。また内容もかなり正確で、レビュー会でも認識合わせにはほとんど時間がかからず、再発防止策の検討に多くの時間を使うことができました。

このようなSkillを使うことでポストモーテムにかける時間を短縮できるのみならず、ポストモーテムを作成する心理的ハードルが低くなるので、組織にポストモーテム文化を定着させる上でも効果を発揮するのではないかと感じています。

ぜひポストモーテム用のSkillを作成してみてはいかがでしょうか？この記事がそのきっかけになると幸いです。
