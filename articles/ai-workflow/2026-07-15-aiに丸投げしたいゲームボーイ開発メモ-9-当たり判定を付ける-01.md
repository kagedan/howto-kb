> この連載のいちばんの狙いは、コードを間違いなく書くことではなく、AIの説明が読めて、AIが変な作りをしたときに違和感を持てて、やりとりがかみ合うようになること——そうすると結果的に、ゲーム作りがぐっと効率よく進みます。レトロゲーム好きで開発初心者の著者が、つまずきながら要点だけ覚えていく連載です。よかったら一緒にどうぞ。

前回、受け皿を左右に動かせるようになった。上から落ちてくる玉の下に、ちゃんと構えられる。これでキャッチできる……と思ったら、玉は受け皿をすり抜けて、そのまま下へ消えていった。受け止められない。今日は、この「受け止めた」を成立させる。

## 重なっているのに、何も起きない

まず、今どうなっているかを見てみる。受け皿を玉の真下に置いて、玉が落ちてくるのを待つ。

![](https://assets.st-note.com/img/1784088931-P9kHNqyCMInaumheQ0YJO5KZ.png?width=1200)

受け皿のところまで玉が落ちてきて、受け皿と玉がぴったり重なっている。でも画面の上のほう、取った数のしるしは1つも無い

見た目は、受け止めている。玉は受け皿のところにある。でも、取った数はゼロ。何も起きていない。玉と受け皿の絵がただ同じ場所に並んでいるだけで、「受け止めた」ことには、まったくなっていない。

## 絵が重なることと、ぶつかったことは違う

ここが今日の肝だ。**絵が画面上で重なっていることと、プログラムが「ぶつかった」と分かっていることは、別もの**だ。

ゲームボーイは、受け皿の絵と玉の絵が重なっていても、それを勝手には教えてくれない。前回の入力と同じで、こっちから「この2つ、重なってる？」と確かめに行かないと、重なりは無かったことになる。Claudeに聞いてみる。

> 受け皿の真下に玉が落ちても、素通りしてしまいます。受け止めたことにするには？

> 受け皿と玉が重なったかどうかを、毎フレーム自分で確かめる必要があります。これを当たり判定と言います。いちばん簡単なのは、両方を四角とみなして、四角どうしが重なっているかを調べる方法です。

## 玉と受け皿、どっちに当たり判定を付ける？

当たり判定は、ものに「付ける」ものだ。キャラや玉に、当たり用の四角——ぶつかりを見るための四角——を持たせる。ここで素朴な疑問が出る。その四角は、玉に付けるのか、受け皿に付けるのか。

答えは、両方だ。

当たり判定は「2つのものが重なったか」を見る仕組みだから、片方だけが四角を持っていても意味がない。玉にも受け皿にも、それぞれ当たり用の四角を持たせて、その2つの四角が重なっているかを調べる。当たり判定は、どちらか一方に付くものというより、**2つのものの"あいだ"にあるもの**、と考えるとしっくりくる。「玉の四角」と「受け皿の四角」が重なった瞬間が、受け止めた瞬間だ。

ちなみに、この四角は絵とぴったり同じでなくてもいい。取りやすくしたければ受け皿の四角を少し大きめに、シビアにしたければ小さめに——当たりの大きさは自分で決められる。今日はどちらも絵と同じ8×8にしておく。

## 四角が重なっているか、だけを見る

やり方は、身構えるほどでもない。ほとんど算数だ。2つの四角が、横で重なっていて、かつ縦でも重なっていれば、重なっている。これをそのまま書くと、こうなる。

```
// 8×8の四角どうしが重なっているか
unsigned char overlap(unsigned char ax, unsigned char ay, unsigned char bx, unsigned char by) {
    return (ax < bx + 8) && (ax + 8 > bx)    // 横で重なっている
        && (ay < by + 8) && (ay + 8 > by);   // 縦でも重なっている
}
```

`ax, ay` が片方の四角の左上、`bx, by` がもう片方の左上。難しい数式は無い。「横も縦も重なっていれば、重なっている」を、そのまま書いただけだ。この四角どうしのやり方には名前があって、よくAABBと呼ばれる。今は「四角の重なりチェック」とだけ思っておけばいい。

## まず、判定を入れずに落としてみる

さっきの 画面 が、この判定を入れる前の状態だ。玉が受け皿に重なっても、誰も `overlap` を呼んでいないから、何も起きない。玉はそのまま下へ抜けていく。取った数は0のまま。

絵が重なっているのに素通りする。理由はシンプルで、重なりを見に行くコードが1行も無かったからだ。

## 重なりを見て、「取った」にする

では、毎フレーム重なりを確かめて、重なっていたら「取った」ことにする。玉の落下を書いているループの中に、判定を足す。

```
    // 受け皿(bx,136)と玉が重なったら「取った」
    if (overlap(60, d1, bx, 136)) {
        d1 = 24;                    // 取った玉は、上へ戻す
        set_bkg_tile_xy(score, 0, 2); // 画面の上に、取ったしるしを1つ置く
        score = score + 1;          // 取った数をひとつ増やす
    }
```

`overlap(60, d1, bx, 136)` で、玉（横60・縦d1）と受け皿（横bx・縦136）が重なっているかを見る。重なっていたら、玉を上へ戻して（受け止めたので消えたことにする）、画面の上に「取ったしるし」を1つ増やす。

## 取るたびに、上にしるしがたまる

これで動かすと、受け皿で玉を受けるたびに、上のしるしが増えていく。

![](https://assets.st-note.com/img/1784088967-JFIRAX4KGN3WDkrueMcU1odZ.png?width=1200)

受け皿の上に玉が落ちてきていて、画面の左上に、取ったしるしが2つ並んでいる

取れている。受け皿に重なった玉が消えて、上のしるしが2つ。しばらく受け続けると、こうなる。

![](https://assets.st-note.com/img/1784088978-kLnq0JyeCUQYom1cIfu6azRX.png?width=1200)

左上のしるしが8つに増えている。玉はまだ落ちてくる

しるしが8つ。ちゃんと数えられている。さっきまで素通りしていた玉が、今度は受け皿で止まって、取った数として積み上がっていく。「重なったか」を毎フレーム見に行くようにしただけで、ただの落下が、キャッチゲームになった。

## 四角どうしの重なりは、どのゲームでも同じ

「ものとものが当たったかを、四角の重なりで調べる」——これはゲームボーイに限らず、ほとんどのゲームで使われている考え方だ。弾が敵に当たった、キャラが床に乗った、アイテムを拾った。中身はだいたい、今日の「四角が重なっているか」だ。マリオの足が土管に乗るのも、根っこはこれと同じ。だから今日のやり方は、他のゲーム作りでもそのまま効く。

## 得意なのは四角、丸や斜めはちょっと苦手

四角の重なりチェックは、とにかく単純で速い。足し算と比べ算だけで、毎フレーム何十回やってもへっちゃら。これが得意なところ。

苦手なのは、四角じゃない形。玉は本当は丸いけど、今日は8×8の四角とみなして判定した。だから厳密には、角のあたりで「本当は当たってないのに当たった」ことがちょっとだけ起きる。丸や斜めの正確な判定はもっと面倒で、重い。

じゃあなぜ四角で済ませるのか。単純で速くて、しかもズレがせいぜい数ドットだからだ。プレイしていて気にならない程度なら、四角で近似してしまうのが定石になっている。正確さより、軽さと分かりやすさを取る場面だ。

## 「重なったら取った」と、そのまま頼む

当たり判定が分かると、AIへの頼み方も具体的になる。

ぼんやりした頼み方:

> 玉を受け止められるようにして。

「受け止める」がどういう条件かを、AIに丸投げすることになる。

はっきりした頼み方:

> 受け皿と玉を8×8の四角とみなして、重なったら「取った」ことにして。取った玉は上に戻して、取った数を数えて。

「四角で」「重なったら」「取った玉は上へ」まで言えると、こっちの思うキャッチになる。

そのまま使える質問例:

* このコード、当たり判定は四角どうしの重なりで見ていますか？
* 玉を丸として、もう少し正確に当たり判定したい。どう変えればいい？
* 受け皿の左右の端でだけ、当たり判定を少し甘く（取りやすく）できる？

## 受け取ったのは、当たり判定

今日はこれひとつ。

`overlap` の中身は、今回は追わなくていい。「四角が重なってるか、を返してくれる」くらいで通り過ぎて大丈夫。

## ゲームになった。でも、やさしすぎる

玉を受け止めて、数が増える。これはもう、ちゃんとゲームだ。うれしい。

……のだけど、遊んでみるとすぐ気づく。やさしすぎる。玉はいつも同じ速さで、のんびり落ちてくるだけ。受け皿を下に置いて立っていれば、まず取れる。緊張感がまるでない。だんだん速くして、難しくしたい。でも、速さを変えようとして「1フレームに3ドット」とかにすると、今度はカクカク飛ぶ動きになりそうな気がする。なめらかなまま、じわっと速くする方法が、たぶんどこかにあるはずだ。

**よかったら、動かしてみてください**
