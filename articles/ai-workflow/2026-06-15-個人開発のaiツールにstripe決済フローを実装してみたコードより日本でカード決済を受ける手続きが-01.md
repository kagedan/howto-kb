## 価格は感覚じゃなく原価から設計する

実際に課金するかは別として、「もし有料プランを出すなら、いくらが妥当か」を原価から設計してみた。「無制限にしたらAPIコストで赤字になるのでは？」という疑問を、数字で潰す練習。

このツールは Claude の Haiku 4.5 を使っていて、1回の生成（読書メモ→3投稿）のコストはこれくらい:

| 内訳 | コスト |
| --- | --- |
| 入力（システム+メモ 約900トークン） | $0.0009 |
| 出力（3投稿 約1000トークン） | $0.005 |
| **合計** | **約 ¥1 / 回** |

つまり**原価は1回1円**。月500円プランなら、利益が消えるのは「月500回（1日17回）」使われたときで、普通のユーザー（月5〜30回）なら利益率98%超。

唯一こわいのは**悪用の青天井**（スクリプトで叩かれて1万回＝1万円）。なので「無制限」と謳いつつ、規約で**フェアユース上限 1日100回**を引いた。これで普通の人には無制限に見えて、悪用の尻尾だけ切れる。

> 学び: 価格は「なんとなく」じゃなく**単価×想定回数**で詰めると、無制限の怖さも具体的な数字で消える。

## 課金の実装：Payment Links + 「支払いと紐づくPro解放」

最短は Stripe の **Payment Links**（ダッシュボードでリンクを作るだけ、コードほぼ不要）。ただしリンクを置くだけだと「Proを解放する」処理がない。かといって `?pro=success` で戻ってきたらPro、みたいにすると、**URLを手打ちした人が全員タダでPro**になる。詐称し放題。

なので「**本当に支払った人だけ**解放する」最小実装にした。

1. Payment Link の決済後リダイレクト先を `/api/activate?session_id={CHECKOUT_SESSION_ID}` に設定（`{CHECKOUT_SESSION_ID}` はStripeが本物のIDに置換してくれる）
2. `/api/activate` でその session\_id を **Stripe API に問い合わせ**、`payment_status === "paid"` を確認
3. OKなら **HMAC署名つきのProクッキー**を発行（サーバー秘密鍵で署名 → 改ざん不能）

```
// 支払い検証（生fetch 1回。Stripe SDKすら入れてない）
const res = await fetch(
  `https://api.stripe.com/v1/checkout/sessions/${sessionId}`,
  { headers: { Authorization: `Bearer ${STRIPE_SECRET_KEY}` } },
);
const session = await res.json();
const paid =
  session.payment_status === "paid" && session.status === "complete";
```

無料枠のカウントはクッキーで雑にやってるけど、**課金状態だけはHMACで守る**。ここを偽造されたらタダ働きなので。

## AIレビューが、地味だけど痛いバグを見つけた

マージ前に一回AIコードレビューを通したら、決済検証にこんな穴が見つかった。最初こう書いてた:

```
// ダメな例
const paid =
  session.payment_status === "paid" || session.status === "complete";
```

`||`（OR）になってる。これだと、もし**無料トライアル**や¥0セッションを設定したとき、`status` は `complete` になる一方 `payment_status` は `unpaid` のまま → **1円も払ってないのにPro解放**される。

`&&`（AND）に直して、`payment_status === "paid"` を必須にした。お金が絡むコードを本番に出す前に一回レビューを挟む価値は、こういう所に出る。

## squashマージの落とし穴でコンフリクト地獄

別セッションで入れた「アクセス解析」の変更がmainに先にマージされてて、ブランチ側にも似た変更があったせいで、`page.tsx` で何度かコンフリクトした。

しかも**squashマージ**してると、main側は1コミットに潰れてるのに、ブランチ側は元の複数コミットが残ってる。`git diff main..branch` は綺麗（最終差分は数行）なのに、GitHubの**マージは3-wayで衝突**する、という分かりにくい状態になった。

> 学び: squashマージ運用だと、マージ後のブランチはそのまま使い続けず、**main から切り直す**のが無難。横着すると後でコンフリクトを踏む。

## 本番（ここが一番しんどかった）：日本でカード決済を受けるための手続き

テストモードで決済が通るのを確認したあと、本物のお金を受け取るには Stripe アカウントの有効化が要る。ここがコードと無関係に長い。

* **本人確認 + 銀行口座登録**（KYC）
* **「セキュリティ対策措置状況申告書」の提出** ← 日本でオンラインカード決済する全事業者に義務（割賦販売法）。管理画面のアクセス制限、脆弱性対策、マルウェア対策、不正ログイン対策…のチェックリストを一問ずつ自己申告する。

個人開発でこのチェックリストは一見ビビるけど、構成がシンプルなら正直に答えられる項目が多かった:

* カード情報は**自社で保持しない**（Stripeのホスト型決済 → 「非保持」が選べる）
* DBを使ってない → **SQLインジェクションは原理的に発生しない**
* 入力は**Zodで検証**済み、XSSは**Reactが自動エスケープ**
* ログイン機能が無い → 不正ログイン対策は\*\*「該当なし」\*\*
* カードテスティング対策は**Stripeが自動**でやってくれる

唯一「定期的な脆弱性診断」だけ実体がなかったので、嘘で「はい」と答えたくなくて、**Dependabotを入れて週次で依存の脆弱性を自動検知**する運用を足した。申告と実態を一致させた。

> ここでの学び: **「作る」と「お金を受け取れる状態にする」は別物**。後者はコードじゃなくて、本人確認・規約・申告みたいな現実の手続き。ここを甘く見てると公開できない。

## 正直に残した「割り切り」

MVPとして、以下は分かったうえで残した:

* **Pro状態はブラウザ単位のクッキー**（別端末では再導線が要る）
* **決済の戻りURLを共有されると他人もProになれる**（session\_idが永続的にpaidのため）
* **解約/更新をwebhookで追ってない**ので、Proは「支払い確認から31日」で一旦切れる設計

これらを厳密に塞ぐには**ちゃんとした認証＋DB**が要る。が、いまの目的は「**他人が本当にお金を払うか**」の検証なので、そこにDBを先に作るのは過剰。次のアプリで認証を学ぶ予定なので、そのタイミングで本気の作りにする。

## まとめ

* **価格は単価×回数で決める**。原価1円/回が分かれば「無制限」の怖さは消える
* Payment Linksでも、**支払いと紐づくPro解放**（session検証 + 署名クッキー）にしないと詐称される
* **お金が絡むコードはマージ前にレビュー**。OR/ANDひとつでタダPro解放になる
* **squashマージ後のブランチは切り直す**。横着するとコンフリクト
* **「公開」より「お金を受け取る手続き」が重い**。日本のカード決済は本人確認＋セキュリティ申告がセット
* 申告に「はい」と答えるなら**実体を伴わせる**（Dependabotで脆弱性診断を自動化）

無料で試せるのはこれ。「発信のネタ出しが面倒」な人に刺さると嬉しい。
