## はじめに

株式会社ヴァル研究所で「駅すぱあと API」のインサイドセールスを担当している榎本です。情報系の大学を出てそのまま営業職に就き、気づけば9年目。エンジニアリングの素養はどこかに忘れてきました。

今回は、その「駅すぱあと API」をMCPサーバー経由でLLMに使わせてみた体験を書きます。「社員が自社サービスを紹介する記事」ではあるのですが、イチ個人ユーザーとしての視点で感じたことを、なるべく正直に書くつもりです。

## 「駅すぱあと アプリ」と「駅すぱあと API」の違い

まず整理しておきます。

|  | 「駅すぱあと アプリ」 | 「駅すぱあと API」（MCPサーバー） |
| --- | --- | --- |
| 対象 | 個人ユーザー | 法人 |
| 使い方 | スマホアプリで操作 | 自社サービスやLLMに組み込む |
| 契約 | 無料〜個人プラン | 法人契約 |

同じ「駅すぱあと」という名前がついていますが、用途がまったく異なります。アプリは個人が日常的に使うもの、APIは企業が自社サービスに交通情報の機能を組み込むためのものです。この記事で扱うのは後者、「駅すぱあと API」の方です。

## 試す前の正直な期待値

MCPサーバーを試す前、私はLLMに対して不信感を持っていました。

以前、MCPサーバーが登場する前に「LLMで乗り換え案内ができるのでは」と試してみたことがあります。しかし、結果は散々でした。存在しない経路を提示してきたり、人間が移動できる速度を考慮しているのか疑いたくなるほど所要時間がめちゃくちゃだったり、乗換回数や電車の方向が実際と異なるなど。特に土地勘のない場所や、厳密なスケジュールが必要な移動では致命的な案内内容でした。  
つまり、ハルシネーションのオンパレードだったということです。

「LLMは移動の検索には向かない」という印象が、そこで出来上がってしまいました。

今回試したのは、その印象を「駅すぱあと API MCPサーバー」が覆してくれるかどうかの確認でもあります。

## 試したこと：日常の経路検索

「Claude.ai」に「駅すぱあと API MCPサーバー」を接続し、自分が実際によく行う行動計画に近い経路検索を試しました。

### 旅行で仙台へ。お土産と駅弁を東京駅で買いたい

実は、ほぼ毎月仙台へ遊びに行っています。毎回同じルートなので出発時間は体で覚えているのですが、「お土産と駅弁を買う時間も込みで逆算したら何時に出ればいい？」を改めてMCPに聞いてみました。

**投げたプロンプト：**

```
横浜から仙台に旅行に行く。
朝8:00に東京駅を新幹線で出発したい。
東京駅でお土産と駅弁を買う時間（30分くらい）を見越して、
横浜を何時に出ればいいか教えて。
```

**返ってきた内容：**  
![](https://static.zenn.studio/user-upload/95a0077ad3d5-20260619.png)

![](https://static.zenn.studio/user-upload/b58e72df0031-20260619.png)

なんと、経路探索の途中経過から、おすすめの路線、さらにお土産を買う時間を含めたタイムライン、そして他の候補路線まで一気に出してくれました。  
「仙台の旅楽しんで！」の一言も嬉しくなりました。

念の為、出してくれた **06:59横浜発（JR東海道本線・上野東京ライン）** が本当にある路線なのか、「駅すぱあと アプリ」で確認してみることにしました。

![](https://static.zenn.studio/user-upload/cdf5a376fb94-20260619.png)

ちゃんとありました。06:59 横浜発、07:27東京着。  
8:00の新幹線に乗る前に、買い物をする時間は充分に設けられそうです。

## 使ってみた率直な感想

**よかった点**

* 複数条件を自然言語で投げられる。「買い物時間を見越して逆算して」といった条件を一度に伝えられる。
* 逆算や比較を一度の会話で処理してくれる。複数回の検索を組み合わせる必要がない。
* 条件の意図を汲んだ補足をしてくれる。「どういう商業施設がある」など。
* 「駅すぱあと API MCPサーバー」以前に感じていたハルシネーション問題が解消されていた。実際の時刻表データに基づいた正確な情報が返ってきた。

**気になった点**

* 混雑状況やホームの場所など、時刻表以外の情報はAPIの範囲外のため、そこはLLM自体の知識に依存する部分がある。
