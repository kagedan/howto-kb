# 【実務のケチり技術】gpt-4oのプロンプトを「1リクエスト2円」から「0.2円」に削減した、プロンプト圧縮とコンテキストキャッシングの極意

## はじめに
AI機能をプロダクトに組み込んで本番リリースした直後、開発チームを襲うのが**「APIコストの爆発」**です。
プロトタイプ開発時は「賢いから」という理由で `gpt-4o` や `Claude 3.5 Sonnet` に数千トークンの巨大な指示書（システムプロンプト）やドキュメントを読み込ませがちですが、これが数千ユーザーにスケールすると、一瞬で予算が溶けます。

本記事では、精度を一切落とさずにAPIコストを**10分の1（90%削減）**に抑え込んだ、実務で使える「プロンプト圧縮」と「コンテキストキャッシング」の泥臭いテクニックを解説します。

---

## 1. 2つのアプローチ：静的圧縮と動的キャッシュ

コストを削るアプローチは大きく分けて2つあります。

1. **静的アプローチ（プロンプト圧縮）**: そもそもLLMに送る文字数を減らす。
2. **動的アプローチ（コンテキストキャッシング）**: 毎回送る固定の長文プロンプトをAPI側にキャッシュさせ、2回目以降の料金を激安にする。

---

## 2. 【静的】LLMLinguaを使ったプロンプトの「お肉引き締め」
人間が書いたプロンプトには、LLMにとっては「冗長な接続詞」や「予測可能な情報」が多く含まれています。マイクロソフトが開発した `LLMLingua` を使うと、LLMの文脈理解に必要な最小限のトークンに自動圧縮できます。

### 導入
```bash
pip install llmlingua
```

### 圧縮の実装コード
```python
from llmlingua import PromptCompressor

# 1. 圧縮エンジンの初期化（小規模なモデルで高速に判定する）
compressor = PromptCompressor(model_name="microsoft/llmlingua-2-bert-base-multilingual-cased-structured-pruning", device_map="cpu")

# 元の無駄に長いプロンプト（実務の社内規定やRAGのコンテキストなどを想定）
long_prompt = """
あなたは優秀なカスタマーサポートエージェントです。以下の社内ガイドラインを厳密に遵守して、ユーザーからの質問に対して、丁寧かつ簡潔に、日本語で回答を行ってください。
なお、回答の際には絶対に嘘をついてはならず、不明な点がある場合は「わかりかねます」と正直に回答してください。また、ユーザーの気持ちに寄り添った絵文字などを適度に使用すると大変好ましいです。
【社内ガイドライン】
1. 返金対応は購入後30日以内に限ります。
2. 配送遅延に関する問い合わせは、追跡番号を必ず確認してください。
...（さらに続く数千文字のドキュメント）...
"""

# 2. プロンプトを圧縮（ターゲットの圧縮率を指定）
compressed_result = compressor.compress_prompt(
    long_prompt,
    rate=0.4, # 元の40%のサイズに絞り込む
    drop_consecutive=True
)

print(f"元の文字数: {len(long_prompt)} -> 圧縮後: {len(compressed_result['compressed_prompt'])}")
print("--- 圧縮されたプロンプト ---")
print(compressed_result['compressed_prompt'])
```
👉 **効果**: 意味を損なわずにトークン数を半分以下に削れるため、これだけで**リクエストごとのベース料金が50%~60%オフ**になります。

## 3. 【動的】OpenAI / Anthropic の「Prompt Caching」を確実に出現させる儀式
2026年現在、主要なLLM APIは**Prompt Caching（プロンプトキャッシュ）**に対応しています。固定のシステムプロンプトやRAGの参照ドキュメントが一定以上の長さ（例: OpenAIは1024トークン以上）の場合、自動的にキャッシュされ、2回目以降の入力トークン代が**半額〜8割引き**になります。

しかし、実務で「なぜかキャッシュが効かない！」と嘆く人が後を絶ちません。キャッシュを確実に効かせるための鉄則が以下です。

### ❌ キャッシュが死ぬ書き方（動的データが先頭や途中にある）
```python
# ユーザーの質問（毎回変わる）が上にあるため、それ以降の固定プロンプトのハッシュが変わり、キャッシュが全滅する
messages = [
    {"role": "user", "content": "今日のお天気は？"}, # 毎回変わる
    {"role": "system", "content": "--- ここに5000文字の固定社内マニュアル ---"} 
]
```

### ⭕ キャッシュが爆効きする書き方（固定データを先頭に固める）
```python
# 常に変わらない長文を「一番上」に配置する
messages = [
    {"role": "system", "content": "--- ここに5000文字の固定社内マニュアル ---"}, # キャッシュされる！
    {"role": "user", "content": "今日のお天気は？"} # ここだけ新規計算
]
```

### Anthropic (Claude) の場合は明示的にピン留めする
AnthropicのAPIでは、`cache_control` を使って「ここまでをキャッシュせよ」と明示的に指示する必要があります。

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-3-5-sonnet-latest",
    max_tokens=1000,
    system=[
        {
            "type": "text",
            "text": "--- ここに超巨大なシステム指示書や仕様書（数万トークン） ---",
            "cache_control": {"type": "ephemeral"} # これが超重要！
        }
    ],
    messages=[
        {"role": "user", "content": "この仕様書に基づいて、ログイン画面のバリデーションコードを書いて。"}
    ]
)
# 2回目以降の同一リクエストは、入力料金が劇的に安くなります
```

## 4. 実務での注意点：ハイドレーション（コールドスタート）問題
キャッシュは万能に見えますが、**「5分〜10分間アクセスのないキャッシュは破棄される」**という仕様があります。
深夜帯など、アクセスが極端に落ちる時間帯の「最初の1ユーザー」は、キャッシュを生成するためのフル料金と、若干のレイテンシ（遅延）を喰らいます。

**対策:** 
シビアなBtoB SaaSなどで初回レイテンシも削りたい場合は、裏側で5分に1回、Cron等で「ダミーの格安リクエスト」を固定プロンプトに対して送り、キャッシュを強制的に生存（ハイドレーション）させる泥臭いハックが有効です。

## まとめ
本番運用のコスト削減は、華やかなプロンプトエンジニアリングではなく、**「いかに文字数を削るか」「いかにAPI側のキャッシュ仕様に沿ってメッセージを並べるか」**という地味な工夫の積み重ねです。

1. `LLMLingua` で無駄なお肉を削ぎ落とす
2. 変わらない長文（マニュアル等）は必ず配列の「先頭」に置く
3. クロードなら `cache_control` を忘れない

これだけで、月末の請求書を見て青ざめることはなくなります。現場からは以上です！
