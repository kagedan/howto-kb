# AIを100体並列で動かしたら、理論に溺れて実戦で全滅した話

**3行で言うと:**

* ◯/❌の並列スキャンは「読む仕事」では500倍効率（43件のバグを5分で検出）
* だが「作る仕事」では全滅する（85%のworkerがクラッシュ）
* 解は「粘れるエージェント」だった。直線番長ではなく、カーブを曲がれる車が要る

---

先に結論を書く。

朝4時にGitHubで見つけた4行のプロンプトから始まって、1794年のフランスの通信技術と米軍のドクトリンを経由し、2,800行のCLIツールを作り、5フェーズの美しいパイプラインを設計した。

**実戦に投入したら85%のworkerがクラッシュした。**

レート制限だった。`claude -p`を26本同時に起動したら、APIが悲鳴を上げた。2回やっても同じ。理論は美しかった。実装が現実に負けた。

ホットロッドみたいなものだ。直線だけは速い。ドラッグレースなら無敵。だが公道に出た瞬間、カーブが曲がれない。◯/❌スキャンは「読んで判定する」直線番長だった。コードを書いて、テストして、失敗したら修正する — そういうカーブのある仕事は曲がれなかった。

最終的に、CLIツールを捨ててClaude CodeのAgent toolで10体並列にしたら、全バグを検出し、2,352行のコードを6ファイルに分割し、テストを通してpushまで1セッションで完了した。ホットロッドを降りて、普通の車に乗り換えたら目的地に着いた。

この記事は、その迷走と帰還の記録だ。理論の寄り道は無駄ではなかった（と思いたい）。

## 第1章: cavemanの衝撃 — 4行で95%削減

> ここから迷走が始まる。4行のプロンプトが「並列数を増やせる」と気づいた瞬間、もう止まれなかった。

### 17,000スターのOSSの正体

朝4時、GitHub Trendingで目に入った[caveman](https://github.com/JuliusBrussee/caveman)（17k★）を分析した。最初は「便利な小道具」だと思った。

cavemanの本質は驚くほど単純だ。プロンプトに4行追加するだけで、LLMの出力トークンを75%削減する。

```
Be extremely concise. Use short variable names.
Skip all explanations unless asked.
Bullet points over paragraphs.
No filler words.
```

実測してみた。同じタスク（データベーススキーマの分析）を、通常モードとcaveman適用で比較した。

```
通常:    約4,800語
caveman: 約250語
削減率:  95%
```

驚いたのは、250語の方が**親エージェントにとって把握しやすかった**ことだ。4,800語の出力は冗長で、本質を埋もれさせていた。250語は骨だけだが、骨で十分だった。

### 並列エージェントへの飛躍

ここで気づいた。子エージェントの出力を圧縮すれば、親エージェントのコンテキスト窓が空く。窓が空けば、より多くの子エージェントの報告を受け取れる。つまり**並列数が増える**。

```
通常:   1体の報告 = 4,800語 → 親の窓に20体分しか入らない
caveman: 1体の報告 = 250語  → 親の窓に400体分入る
```

トークン課金も減る。レート制限の天井も上がる。caveman の4行は、並列エージェントの世界では革命的な意味を持っていた。

## 第2章: 3つの壁 — v1の設計と限界

> v1は動いた。だが「効率をもっと上げたい」という欲が、ここから設計を暴走させる。

### 壁を発見する

cavemanの圧縮で並列数を増やせることはわかった。だが、実際に増やしていくと、3つの壁にぶつかった。

**壁1: コンテキスト窓。** 1Mトークンの窓に入る報告の数には上限がある。cavemanで1体250語に圧縮しても、400体が限界。

**壁2: RAM。** 各エージェントはプロセスとして走る。65GBマシンで30-50体、24GBなら10-15体が物理的限界。

**壁3: モデルの統合能力。** これが最も本質的な壁だった。15体の報告を渡されたClaude Opusは、矛盾を検出し、重複を排除し、統合的な結論を出せる。だが30体になると精度が落ちる。50体では破綻する。

### 指揮者の階層化 — 指数関数の底を変える

壁3への対策として、指揮者を階層化した。

```
depth=2: 1人の指揮者が15体のworkerを見る → 15体
depth=3: 1人の指揮者が15人の中間指揮者を見る → 各中間指揮者が15体 → 225体
depth=4: → 3,375体
```

15^(depth-1)でスケールする。指数関数だ。

だが、情報も指数関数で消える。各階層で圧縮率95%をかけると:

```
depth=2: 情報残存率 5%
depth=3: 情報残存率 0.25%
depth=4: 情報残存率 0.0125%
```

**エージェント数は指数で伸び、情報は指数で消える。** これが壁3の数学的な正体だった。

### v1の設計: 段階圧縮

解決策として「段階圧縮」を設計した。

```
Worker層:  ultra圧縮（95%削減）→ 量を捌く
中間指揮者: lite圧縮（60%削減）→ 推論を残す
Root:      圧縮なし → 全てを受け取る
```

さらに、内容の種類によって圧縮度を変える。

```
通常の事実報告: ultra圧縮
推論の連鎖:    lite圧縮
異常の検出:    圧縮なし（anomalyは1ビットも捨てない）
```

これがSwarm Protocol v1だった。MiroFish（53.6k★）やCrewAI、AutoGenといった既存フレームワークとの差別化点は、この段階圧縮と5軸分析（並列可能性/深さ/依存関係/量/精度）にあった。

## 第3章: ◯/❌の発明 — v2への転換

> 500倍のコスト削減。この数字に酔った。ここからworkerの能力を削り続ける旅が始まる。

### 「◯か❌で通信すれば？」

v1の段階圧縮は動いた。だが、根本的な疑問が湧いた。

「そもそも、なぜ全workerがフルレポートを書くのか？」

100人のworkerに「この判例を分類しろ」と頼む。99人は正しく分類できる。1人だけ判断に迷う。v1では、99人の正しい回答も、1人の迷った回答も、同じフォーマットで報告させていた。99人分の「問題ありません」という冗長な報告は、本質的に情報量がゼロだ。

そこで出た発想:

> 「◯か❌で通信して、❌の時だけ情報量を増やせば、全部回るんじゃない？」

### 2ラウンドプロトコル

```
Round 1: 全workerが ◯ or ❌ で回答（1行）
Round 2: ❌のworkerだけが詳細レポートを書く
```

これだけで、統合コストが劇的に変わった。

```
v1: 全員フルレポート → 指揮者が全ペアを比較 → O(n²)
    100人: 4,950回の比較 × 100tk = 495,000tk

v2: 全員◯/❌ → ❌だけ詳細 → O(k²) where k = ❌の数
    100人, 5% ❌: 5人の詳細 → 10回の比較 × 100tk = 1,000tk
```

**495,000tk → 1,000tk。** 500倍の削減。しかも情報の質は同じだ。なぜなら、◯同士は比較する必要がない。「問題なし」は「問題なし」と矛盾しないからだ。

### 質問がフォーマットを規定する

◯/❌プロトコルのもう一つの発見は、**質問の形が回答の形を決める**ということだった。

```
質問: 「異常はあるか？」   → 回答: ◯ or ❌
質問: 「何型か？」        → 回答: ◯ api or ❌
質問: 「何件ある？」      → 回答: ◯ 42 or ❌
質問: 「名前を抽出しろ」  → 回答: ◯ 田中太郎 or ❌
```

タスクの種類を事前に分類する必要がなくなった。質問を変えるだけで、workerの出力形式が自動的に変わる。v1の段階圧縮（ultra/lite/none）は**質問設計で不要になった**。

### 4型情報モデル

Round 2で❌のworkerに詳細を求める際、報告の構造も定義した。

```
fact      高圧縮可  検証可能な事実の箇条書き
relation  中圧縮可  事実間の関係（F1→F2: 理由）
reasoning 低圧縮可  推論の連鎖（各ステップが重要）
anomaly   圧縮不可  異常はそのまま届ける
```

Shannon情報理論との対応がある。予測可能性が高い情報（fact）は圧縮できる。予測不可能な情報（anomaly）は圧縮できない。この4型は、情報理論的に自然な分類だった。

## 第4章: 制約が密度を上げる

> この章から歴史の旅に出る。一見遠回りだが、これが後のアーキテクチャの核になる。実装を急ぐ人は第7章まで飛ばしてもいい。

通信技術は200年かけて帯域を広げてきた。◯/❌は**逆走している**。意図的に旗通信まで帯域を絞る。

なぜか。Shannonの通信路容量定理は物理帯域の限界を扱った。だがLLMスワームのボトルネックは物理帯域ではない。**受信者（root）の認知限界**だ。コンテキスト窓が有限である以上、送信側を絞るしかない。

LLMに「自由に答えていい」と言うと2,000トークン喋る。「◯か❌で」と言うと1トークンに凝縮される。**制約が情報密度を上げる。**

## 第5章: 腕木通信 — 中間層の知能を消す

> 1794年のフランス人が、2026年のLLMと同じ問題を解いていた。面白さに溺れた章だが、ここから最大の発見が出る。

1794年、クロード・シャップの腕木通信はパリ→ストラスブール450kmを6分で伝達した。重要なのは**中継員がメッセージの意味を知らない**こと。コードブック（符号→意味）は送信者と最終受信者だけが持つ。中継員は腕木の角度をコピーするだけだ。

ここから3つの設計原理を抽出した。

**1. Signal = ACK。** workerが◯/❌を返した時点で生存確認完了。heartbeat不要。timeout = 死亡 → 交換。

**2. コードブック方式。** 質問にコードを埋め込む:

```
「分類: A=認容 B=棄却 C=却下 D=和解」→ worker: ◯ B
→ 表記揺れゼロ → 機械集計可能
```

4型定義: BOOL(count), ENUM(countBy), VALUE(sum/avg), EXTRACT(転送のみ)。

**3. 中間層をLLMから外す（最大の発見）。**

```
v2: Root(LLM) → Conductor(LLM) → Workers(LLM)   depth=3: 90,000tk
v3: Root(LLM) → Relay(スクリプト) → Workers(LLM)  depth=3: 0tk
```

relay層は80行のTypeScript。◯を数えて❌を転送するだけ。LLMが動くのは「端」だけ。**これが腕木通信のアーキテクチャだ。**

## 第6章: 軍事ドクトリン — 異常系の設計

> 弁護士がCASの委任範囲を設計している。何の仕事だ。だが、ここから取り出した3つの原則は今も生きている。

軍隊は「大量のユニットを並列に動かし、異常時に迅速に対応する」ことを数百年かけて最適化してきた。swarmと同じ問題だ。ただし全部を取り込むと過剰になる。実際、最初に設計した8ビットバイナリヘッダは却下した。LLMはテキスト出力だし、確信度4段階は◯/❌哲学と矛盾する。

**取り込んだのは3つだけ:**

**チャネル分離（MEDEVAC原理）。** タスク結果(◯/❌)とインフラ障害(timeout/OOM)を混同しない。

```
CH0 TASK:  ◯/❌結果      → 通常フロー
CH1 INFRA: 障害           → 別系統で即時処理（worker交換）
CH2 META:  進捗           → 表示のみ、処理に影響しない
```

**Circuit Breaker。** ❌率で自動エスカレーション:

```
❌ < 10%:   何もしない
❌ 10-30%:  rootに警告（継続）
❌ > 30%:   CIRCUIT OPEN → 全停止
```

**事前委任（CAS方式）。** relay層に自律判断の範囲を事前定義:

```
自律OK: worker交換(timeout>60s)、再試行(max 2回)、スキップ(>120s)
自律NG: 質問変更、シグナル読み替え、swarm停止 ← rootの権限
```

## 第7章: 制約が品質を上げる — 実測

> ここで初めて手を動かした。そしたら43件のバグが見つかった。理論を現実に接地させた唯一の章。

### プロトコルのdog-fooding

v3のSKILL.md、swarm-worker.ts、architect SKILL.mdの3ファイルを、3体のworkerで並列に書いた。全workerに共通のvision（v3設計決定）を渡し、各自が独立にファイルを書いた。163秒で完了。逐次なら390秒かかる計算。

書き上がったswarm-worker.ts（548行）を、v3プロトコル自体でバグスキャンした。**自分のコードを自分のプロトコルで検査する。**

```
/swarm "このコードにバグ・型エラー・未処理のエッジケースはあるか？" --workers 3 --legal
```

### Round 1: 全員❌

548行を3分割し、各workerに180行ずつ渡した。

```
Worker 1 (L1-180):   ❌
Worker 2 (L181-370): ❌
Worker 3 (L371-548): ❌
```

全員が❌を返した。Round 2で詳細を収集。

### Round 2: 9件のバグ検出

4型レポートで返ってきた結果は、想像以上に精密だった。

```
B1 CRITICAL: parseSignal関数でisFail変数を宣言するが未使用。
             isOkの否定だけで判定するため、detailラウンドの
             複数行レポートが全てfail扱いになる。
             Circuit Breakerのfail rateを不当に上昇させ、
             長時間稼働で不可逆的にOPENにする。

B2 HIGH:     decideDelegatedActionでskipがreplaceより先に
             評価される。replace_workerが到達不能コード。

B3 HIGH:     Circuit Breaker OPEN時にaction: "complete"で
             応答。呼出元がescalationフィールドを検査しないと
             正常結果として処理される。

B4-B9:       daemonPost失敗の握り潰し、SIGTERM時のreport未出力、
             signals配列の無制限蓄積、runClaudeのkill後競合状態、等
```

特にB1は**本物の重大バグ**だった。Mac miniにデプロイしたばかりのswarm-worker.tsが、長時間稼働でCircuit Breakerが不可逆的にOPENになり、全タスクを拒否し続ける状態に陥るところだった。

### なぜ見つかったのか

重要なのは、「なぜ3体の並列エージェントが、1体のレビュアーよりも多くのバグを見つけたのか」だ。

**理由1: 注意力の壁。**

1体に548行を全部読ませると、全体を「ざっと」見る。L139のisFail未使用のような細部は見落とす。3体に180行ずつ渡すと、各workerが180行に全注意力を集中する。LLMの注意力は有限で、長い入力の中間部分で精度が落ちることは既知だ（Lost in the Middle現象）。並列化はこれを構造的に解消する。

**理由2: ◯/❌の強制。**

「レビューしてください」と頼むと、LLMは「概ね問題ありませんが、いくつか改善点があります...」と曖昧に答える。「バグがあるか？◯か❌で答えろ」と迫ると、❌と答えるからには根拠が必要になる。必然的にコードを精読する。**制約が精読を強制する。**

**理由3: 4型報告の構造化。**

❌と答えた後に「事実/関係/推論/異常で報告しろ」と言うと、「F1: isFail変数が宣言されるが未使用」（事実）→「F1→F2: fail誤分類がCBのfail rateを上昇」（関係）という構造化された報告が出る。「なんとなく気になる」レベルの指摘ではなくなる。

3つとも**制約を課すことで品質が上がる**パターンだ。

## 第8章: 射程 — Kill Web推論とvision共有

> DARPAまで引っ張り出した。もう完全に論文を書いている。ツールを作れ。

### 並列化できないものはあるか

v3の設計を振り返り、「何が並列化できて、何ができないのか」を整理した。

最初の分類:

```
◎ 分類（独立した判定）    → 各workerが独立に判定
◎ 検索（大量データ）      → パーティションで分割
◎ 検証（独立チェック）    → 各workerが独立に検証
◎ 抽出（文書ごと独立）    → 各workerが独立に抽出
△ 要約（全体把握が必要）
✕ 推論（連鎖的思考）      → 各ステップが前のステップに依存
✕ 創作（一貫性が必要）
✕ 設計（全体最適が必要）
```

### vision共有で△と✕を克服する

「空いたトークンでビジョンを共有すれば、統一性が必要なタスクも並列化できるのでは？」という指摘があった。

各workerのコンテキストは200Kトークン。タスクデータが5Kなら、195Kが空いている。ここにvision（目的、スタイル、トーン、用語、構造）を入れれば、全workerが同じビジョンを共有したまま独立に作業できる。

これは人間の組織では当たり前だ。会社にはブランドガイドラインがあり、100人のライターが別々の記事を書いても一貫性が保たれる。軍隊ではCommander's Intent（指揮官意図）が全部隊に共有され、各部隊が自律的に行動しても意図に沿った結果が出る。

**並列化の可否は「ビジョンを事前にプロンプトとして書けるか」で決まる。** 言語化できるビジョン（明示知）は並列化可能。言語化できないビジョン（暗黙知）は逐次のみ。

### Kill Web推論 — 推論を鎖から網へ

DARPA提唱のMosaic Warfare（2017年〜）のKill Web概念を推論に適用した。

従来の推論は直列の鎖（Kill Chain）だ:

```
A → B → C → D → 結論
各ステップが前のステップに依存。並列化不能。
```

だが、多くの実務的推論は鎖ではなく**網**だ。法律でいえば:

```
理論A: 不法行為 → 注意義務 → 違反 → 因果関係 → 損害
理論B: 債務不履行 → 契約義務 → 違反 → 損害
理論C: 不当利得 → 法律上の原因なし → 利得

各理論は独立。各ステップは並列検証可能:
  Worker 1: 「注意義務は認められるか？」→ ◯
  Worker 2: 「契約義務はあるか？」→ ❌
  Worker 3: 「因果関係は立証できるか？」→ ◯
  ...
  Root: ◯だけ繋いで有効なパスを発見
```

依存関係がある場合は層並列で対処する:

```
Phase 1 (並列): A, B, C の前提を検証
Phase 2 (並列, Phase 1の結果を使う): 前提が成立した理論の次のステップを検証
Phase 3: 統合
```

**幅は並列、深さは逐次。** 完全な並列ではないが、6ステップの推論が3ステップに短縮される。

---

## 第9章: 安全性の考察

◯/❌は「大量データから異常を拾うフィルタ」だ。防御にも攻撃にも使える。ただし後日気づいたのは、既存の脆弱性スキャナ（nuclei等）の方がパターンマッチで100倍速いということ。◯/❌が既存ツールに勝てるのは「意味の理解が必要な判定」だけだ。

本当の脅威は**nuclei（既知パターン）+ LLM（未知の意味的判定）の組み合わせ**かもしれない。ただしこれは攻防同じ道具で、先にやった方が勝つ。

ローカルの検閲なしLLMとの組み合わせも懸念がある。◯/❌は弱いモデルの出力を人間が扱える量に圧縮するため、検閲なしモデルの実用性を引き上げてしまう。プロトコルレベルでの防御は不可能だ。この問題は未解決であり、正直にそう書く。

## 第10章: コスト推移と結論

> 数字は嘘をつかない。495倍のコスト削減。ただし、このコスト削減と引き換えに失ったものがある。workerの能力だ。

### v1 → v2 → v3

```
                        v1           v2           v3
──────────────────────────────────────────────────────────
Round 1 cost            n × 500tk    n × 50tk     n × 50tk
中間層cost (depth=3)     90,000tk     90,000tk     0tk
Round 2 cost            —            k × 6,000tk  k × 6,000tk
統合cost                O(n²)        O(k²)        O(k²)
──────────────────────────────────────────────────────────
例: 100 workers, 5% ❌   495,000tk    91,000tk     1,000tk
──────────────────────────────────────────────────────────
```

v1からv3で**495倍のコスト削減**。情報の質は同じ。

### 何が起きたのか

caveman（4行の圧縮プロンプト）から始まって、12時間で:

1. 出力圧縮95%で並列数を増やした（v1）
2. ◯/❌で統合コストをO(n²)からO(k²)にした（v2）
3. 腕木通信から中間層のLLMを消した（v3）
4. 軍事ドクトリンで異常系を設計した（v3）
5. 自分のプロトコルで自分のコードの43件のバグを見つけて修正した

### 本質

200年前の腕木通信と、2026年のLLMスワームは、同じ問題を解いている: **帯域が限られた環境で、大量の情報を正確に伝達するにはどうするか。**

答えも同じだ: **信号を単純にして、中間層から知能を抜く。**

腕木の操作員は角度をコピーするだけ。relay層は◯を数えて❌を転送するだけ。意味を理解する必要がないから、速く、安く、正確に動く。

通信技術は帯域を広げる方向に進化してきた。LLMスワームでは、意図的に帯域を絞ることが正解だった。ボトルネックは物理帯域ではなく、受信者の認知限界だからだ。

制約を課すことで品質が上がる。◯/❌に絞ることで精読が強制される。180行に集中させることで注意力が最大化される。この逆説が、4,471行から26件のバグを見つけた理由であり、おそらく、まだ誰も気づいていない領域だ。

## 第11章: 後日談 — 実戦が理論を壊した

> ここからが本番。読者が知りたいのは「で、使えるの？」だろう。答えは「理論は死に、別のものが生まれた」。

### 迷走

v3の理論が綺麗にハマるたびに、深掘りした。腕木通信、Shannon情報理論、DARPA Mosaic Warfare。論文を読み、対応表を作り、プロトコルの数学的性質を分析した。

気づいたら**研究者になっていた**。

◯/❌の圧縮効率が劇的だった（500倍削減）ので、そっちに設計が引っ張られた。「中間層のLLMを消す」「コードブックで集計」という最適化に集中した結果、workerの能力を**削る**方向に進化が進んだ。

```
v1: workerはフルレポートを書く    → 重い
v2: workerは◯/❌だけ返す         → 軽い！500倍！
v3: 中間層からLLMを消す           → 更に軽い！0コスト！
   → workerは1行返して死ぬプロセスになった
```

効率を追求するほど、workerが何も生産しなくなった。元々の目的は「複数のAIエージェントに並列で仕事をさせる」だったのに、「◯/❌で答えるだけの読み取り専用プロセス」を最適化していた。

### orchestrai CLIの死

それでも手を動かし続けた。v3プロトコルをCLIツール（orchestrai）として実装した。`claude -p`を並列起動し、◯/❌を集計するパイプライン。TypeScript 2,800行、5フェーズ（構造チェック→分割提案→バグチェック→詳細分析→レポート）。依存分析（import先の型シグネチャ解決）、markitdown連携（PDF/Word/Excel対応）、Circuit Breaker、レート制限リトライまで入れた。

理論通り動くはずだった。

実戦投入した結果:

```
orchestrai（claude -p 26並列）:
  Phase 1（構造チェック）: 8-9/26成功、残りクラッシュ
  Phase 2（分割提案）:     全滅
  Phase 3（バグチェック）:  全滅
  Phase 4（詳細分析）:     全滅
```

`claude -p`を26本同時起動すると、Pro契約のレート制限に引っかかって85%がexit 1で死んだ。2回やっても同じだった。

一方、Claude CodeのAgent tool（セッション内並列）は:

```
swarm（Agent tool 10並列）:
  Phase 1: 10/10成功、全バグ検出
  Phase 2: 全件修正完了
```

**理論は正しかった。実装手段が間違っていた。**

```
                claude -p (CLI)             Agent tool (セッション内)
─────────────────────────────────────────────────────────────────
プロセス      独立プロセス×N本              セッション内サブプロセス
API接続      各自が独立に叩く              親セッションの接続を共有
レート制限    N本分カウント → 即死          セッション内で制御 → 安定
ツール       なし（stdout返すだけ）         Read/Write/Bash/全ツール
粘り         1発で死ぬ                     失敗→修正→再試行できる
コスト       claude -p は Pro契約内         同左
実績         26並列 → 85%クラッシュ        10並列 → 10/10成功
─────────────────────────────────────────────────────────────────
```

### 何が間違っていたか

`claude -p`は外部プロセスとしてAPIを叩く。並列起動すると、それぞれが独立したAPI呼び出しになり、レート制限のカウンタを急速に消費する。

Agent toolはセッション内のサブプロセスで、親セッションのAPI接続を共有する。同じ10並列でも、APIから見た負荷が全く違う。

### 理論が生き残った部分

CLIは死んだが、設計思想は全部生きている:

* **◯/❌プロトコル** → Agent tool swarmの信号形式として使用中
* **構造→バグの順序** → 先に構造チェックしてからバグを探す方が精度が高い（実証済み）
* **コードブック型** → BOOL/ENUM/EXTRACT、質問を変えるだけで用途が変わる
* **Circuit Breaker** → 失敗率が高すぎたら自動停止
* **relay層のLLM排除** → Agent toolの報告集計もスクリプトでやっている

### 実戦で証明されたこと

翻訳エンジン（2,352行）のリファクタリングで:

1. swarm 10 worker → 全バグ検出
2. 構造チェック → 6ファイルに分割提案
3. 後方互換性を壊さず分割実行
4. テスト全通過 → push

1セッションで完了した。通常なら分割だけで1日、バグ修正で1日、テストで半日。

### 教訓

**効率を追求するほど、workerが無能になった。** ◯/❌の圧縮効率に魅せられて、workerの能力を削る方向に最適化し続けた結果、1行返して死ぬだけのプロセスになった。

本当に必要だったのは「粘れるworker」だった。ファイルを読み、コードを書き、テストを実行し、失敗したら修正する。Agent toolはそれができる。`claude -p`にはできない。

◯/❌スキャンは「大量データの仕分け」で今も有効だ。100件のPDFからChange of Control条項を探す、200件の判例を認容/棄却に分類する。1回読んで1回答えれば済む仕事。

ただし、コードを書く仕事は1発では済まない。粘りが要る。この区別を見落としたのが、12時間の理論構築で唯一の設計ミスだった。

### 結局できたもの

迷走を経て、3つの並列処理ツールが生まれた:

```
/audit       — ◯/❌スキャン（Agent tool並列、Opus）
               構造チェック→バグチェック→修正の2フェーズ
               PDF/Word/Excel対応（markitdown連携）
               依存分析（import先の型シグネチャ解決）

/parallelize — Wave方式の並列開発
               依存関係を分析してWave分割→Agent tool並列実行
               --audit で Wave間に品質ゲートを挟む
               --safe/--surge/--beast のモード切替

/conductor   — 異質タスクの振り分け
               法律・開発・調査・事務を同時に回す
               heavy/scan/research/quick/draft の5カテゴリに自動分類
               各カテゴリに最適な実行方法を選択
```

orchestraiのCLI（`claude -p`並列）は死んだが、そこで設計した5フェーズパイプライン、コードブック型、依存分析、構造チェックは全部Agent toolベースのスキルに移植された。

理論の迷走は無駄ではなかった。腕木通信の「中間層に知能は要らない」は、Agent toolの報告集計でも同じ原則として機能している。軍事ドクトリンのROE（交戦規則）は`/parallelize`のモード設計そのものだ。Circuit Breakerは品質ゲートとして生きている。

各ピースが最終形の中に場所を見つけた。回り道をしなければ、この設計には辿り着かなかった。
