## はじめに

Notion の PRD を起点に、AI がデザインシステムのコンポーネントを使った React コードを自動生成するワークフローを作りました。この記事では、その設計と仕組みを紹介します。

## 解決したかった課題

従来のフロントエンド実装フローには、3つの課題がありました。

### コンテキストの断絶

PRD → Figma → コードの間で情報が抜け落ちます。PRD に「ローディング状態を表示する」と書いてあっても、実装時にその要件を見落とすことがあり、Figma のコンポーネント名がデザインシステムの名前と一致しない時は手探りで対応表を確認する時間が発生します。

→ このワークフローでは、PRD・Figma・デザインシステムの情報を MCP（Model Context Protocol） 経由で同時に取得し、1つのコンテキストに統合することで断絶を解消しています。

### 繰り返しの手作業

ページレイアウトの構築、import パスの確認、CSS クラスの選定。これらは毎回やることがほぼ同じで、自動化の余地が大きい領域です。

→ プロジェクト規約の読み込みとホワイトリスト照合を自動化し、これらの定型作業を AI に委譲しています。

### Figma リンクの不在

PM が PRD に Figma リンクを貼り忘れることがあります。また、バックエンド寄りの機能ではそもそもデザインが存在しないこともあります。その場合、エンジニアは既存画面の見た目を参考に、似たような構成をゼロから書くことになります。

→ デザインなしパスを用意し、PRD の機能要件からページ構成を推定して生成できるようにしています。

## Claude スキルとは

今回のワークフローは、Claude の「スキル」という仕組みで実装しています。

スキルは、`SKILL.md` というマークダウンファイルに手順を記述すると、Claude エージェントがその手順に従って自律的にタスクを実行する仕組みです。コードを書くのではなく、手順書を書くことでオーケストレーションを構成します。スキルの中から MCPサーバーを呼び出したり、サブエージェントに作業を委譲したり、ユーザーに確認を取ったりできます。

今回は、3つの MCP サーバー（Notion、Figma、社内デザインシステム）を1つのスキルで束ねて、PRD の読み取りからコード生成・検証までを一貫して実行するワークフローを作りました。

## ワークフローの全体像

エンジニアが Notion PRD の URL を渡すと、以下を自動で実行します。

1. PRD から機能要件・ユーザーストーリーを抽出
2. PRD 内の Figma リンクを自動検出（ユーザーが明示指定した場合はそちらを優先）
3. Figma があればデザインを解析し、データモデルを抽出してデザインシステムのコンポーネントにマッピング
4. Figma がなければ PRD の機能要件からページ構成を推定
5. モックデータ付きの `.tsx` ファイルをプロジェクト規約に準拠して生成
6. 型チェックと linter を通過するまで自動修正

最終出力は、コンパイルと linter を通過し、モックデータで画面が動作するコンポーネントファイルです。

## 技術構成

3つの MCP サーバーを組み合わせています。

| MCP サーバー | 役割 |
| --- | --- |
| Notion MCP | PRD ページの読み取り、Figma URL の検出 |
| Figma MCP | デザインファイルの構造解析 |
| albers MCP | デザインシステムのコンポーネント一覧・詳細情報の取得 |

MCPは、AI エージェントが外部ツールやデータソースにアクセスするための標準プロトコルです。Anthropic が策定・公開しており、Notion や Figma などのサービスに対して、AI エージェントが安全に読み書きできるインターフェースを提供します。

「albers」というのは、SUPER STUDIO社内のデザインシステムです。MCP 経由でコンポーネントの一覧や Props 情報を動的に取得できるため、ワークフローはデザインシステムの更新に追従できます。

### 処理フロー

## 各ステップの解説

### Step 0：ナレッジベース読み込み

プロジェクト規約と日本語 UI ルールを読み込み、albers MCP からコンポーネント一覧を取得します。この一覧が以降のステップで import 可能な名前のホワイトリストになります。

### Step 1：PRD 抽出

Notion MCP で PRD を読み取り、機能要件・ユーザーストーリーを抽出します。実装に不要な情報（会議メモ等）は除外し、同時に `figma.com` URL もすべて検出します。

### Step 2：Figma ソース解決

**エンジニアがプロンプトで Figma URL を渡した場合、それを最優先で使います。** 指定がなければ PRD 内のリンクを使います（決定表は後述の「スキルの構成」を参照）。PRD から取得した URL を使う場合は、PRD の目的とデザインのページタイプが一致するかも検証します。

### Step 3-4：デザイン解析とマッピング

Figma MCP で UI 階層・コンポーネント名・**データモデル**（カラム名、フィールド名、推定型、サンプル値）を抽出します。次に、Figma のコンポーネント名をマッピング表と albers MCP を使ってデザインシステムのコンポーネントに変換します。マッピング表にも MCP にもなければ CSS でカスタム実装します。

Figma MCP が利用不可の場合は、Dev Mode のコピーやスクリーンショットでも受け付けます。

### Step 5-7：コード合成・バリデーション・検証

**デザインありパス**：Figma の構造に従ってページレイアウトを構築します。  
**デザインなしパス**：PRD からページタイプを推定し、確認を取ってから生成します。どちらのパスでも、日本語固有のスタイルルールを適用します。

API 未実装でも画面が動くよう、Figma のデータモデルからモックデータを生成します。本番用の API 呼び出しは `TODO` 付きでコメントアウトして残します。ローディング・エラー・空状態の3パターンも含めます。

生成後、セルフチェックと型チェック・linter を通過するまで自動修正します。

## スキルの構成

実際のスキルは、1つのオーケストレーターファイルと、詳細手順を分離したリファレンスファイルで構成しています。

```
ui-generation/
├── SKILL.md                       # オーケストレーター本体
└── references/
    ├── albers-catalog.md          # Figma → デザインシステムのマッピング表
    ├── codebase-conventions.md    # タイポグラフィ・スペーシングの補足ルール
    ├── notion-prd-extractor.md    # PRD 抽出の詳細手順
    ├── figma-ui-inspector.md      # デザイン解析の詳細手順
    └── albers-component-mapper.md # コンポーネントマッピングの詳細手順
```

`SKILL.md` にはワークフロー全体の流れと判断ルールを書き、各ステップの詳細は `references/` に分離しています。Anthropic が公開している[スキル設計ガイドライン](https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf)に従い、Claude が必要なときだけ詳細を読み込む構成にしました。

以下、`SKILL.md` の中で設計判断が分かりやすい部分を2つ抜粋します。

### Figma ソース解決の決定表

Figma の URL がどこから来るかによって挙動を変える必要があります。スキルの中では以下の決定表で一意に解決しています。

| ユーザー提供 | PRD 内リンク | 解決 |
| --- | --- | --- |
| あり | （無視） | ユーザー提供の URL を使用 |
| なし | 0件 | デザインなしパス |
| なし | 1件 | PRD 内の URL を使用 |
| なし | 複数 | ユーザーにどれか選択させる |

エンジニアが明示的に Figma URL を渡した場合、PRD 内のリンクは一切参照しません。PRD のリンクが間違っているケースが実際にあったため、このルールにしています。

### ハルシネーション対策：動的ホワイトリスト

AI によるコード生成で最も怖いのは、存在しないコンポーネントの import です。`import { FancyTable } from "@super_studio/ecforce_ui_albers/FancyTable"` のようなコードが生成されると、コンパイルは通らず、原因の特定にも時間がかかります。

このワークフローでは、Step 0 で albers MCP から取得したコンポーネント一覧を「使ってよい名前」のホワイトリストとし、ここに存在しない名前の import を禁止します。静的なリストではなく MCP から動的に取得するため、デザインシステムにコンポーネントが追加されても追従できます。生成後のセルフチェックでも import 文をホワイトリストと突き合わせ、さらに型チェックの段階でコンパイルエラーとして検出する多段構成にしています。

## 使い方

Cursor のチャット、または Claude Code（CLI）で Notion の PRD URL を渡します。

**Cursor の場合：**

```
この PRD から UI を実装して: https://www.notion.so/your-workspace/PRD-xxxxx
```

**Claude Code（CLI）の場合：**

```
claude "この PRD から UI を実装して: https://www.notion.so/your-workspace/PRD-xxxxx"
```

PRD 内に Figma リンクがあれば自動検出されます。Figma を別途指定することもできます。

```
この PRD から UI を実装して:
- PRD: https://www.notion.so/your-workspace/PRD-xxxxx
- デザイン: https://www.figma.com/design/xxxxx
```

デザインがないことが分かっている場合は、明示的に伝えることもできます。

```
この PRD から UI を実装して: https://www.notion.so/your-workspace/PRD-xxxxx
デザインはありません
```

## 実際に使ってみた結果

直近、ecforce AIdp のファイル管理機能の UI を、このワークフローで実装してみました。一覧テーブル、検索・フィルター、アップロードモーダル、進捗アコーディオンなど7つのコンポーネントを含む画面です。入力は Notion PRD と Figma デザインの URL のみ、プロンプトは `この PRD から UI を実装して` の1行です。

### デザインと生成結果の比較

全体のレイアウトやコンポーネント配置はデザインに忠実に再現されていますが、完璧ではありません。Figma のレイヤー構造が複雑な場合（Auto Layout の深いネスト、コンポーネント名の不統一など）に AI の解析精度が落ち、細かいずれが出る箇所がありました。Figma 側のファイル構成を整えることで改善できる部分が大きいと感じています。

| 画面 | Figma デザイン | AI 生成結果 |
| --- | --- | --- |
| ファイル一覧 | Figma: ファイル一覧 | 生成結果: ファイル一覧 |
| アップロードモーダル | Figma: アップロードモーダル | 生成結果: アップロードモーダル |

### エンジニア工数の比較

同じ画面をエンジニアが手作業で実装したらどのくらいかかるか、ざっくり比較してみました。PRD とデザインの読み込みから、コードベース調査、実装、型エラー対応までを含めた純粋な作業時間です。実際にはレビュー待ちやミーティングで手が止まる時間もあるので、体感のリードタイムはもっと長くなります。

|  | AI ワークフロー | シニアエンジニア | ジュニア〜ミドル |
| --- | --- | --- | --- |
| 合計時間 | **約15分** | 約2〜4時間 | 約6〜9時間 |
| 倍率 | -- | 8〜16倍 | 24〜36倍 |

## 制約と今後の課題

### 現時点の制約

* 1回の実行で対象にできるのは1ページ分の UI のみ
* 生成されるのはページコンポーネントの骨格とモックデータであり、ビジネスロジックは手動で追加する
* モックデータは API ができたら手動でコメントを切り替える必要がある
* Figma MCP がエラー状態の場合はフォールバック（手動入力）に切り替わる

### 今後の改善候補

* デザインなしパスの精度向上（既存の類似ページを参照する仕組み）
* 複数ページの一括生成
* API スキーマからの型自動インポート（現在はモックデータ止まり）

## まとめ

PRD を読んでデザインを確認してレイアウトを組む、という繰り返しの作業を AI に委譲するワークフローを作りました。MCP サーバーを通じてデザインシステムの情報を動的に取得することで、ハルシネーションのリスクを下げつつ、デザインシステムの変更にも追従できる構成にしています。

まだ初期段階ですが、ファイル管理画面（7コンポーネント）の実装で約15分という結果が出ており、シニアエンジニアの手作業と比べても大幅な時間短縮になっています。コンパイルエラーを人が直す作業がなくなったことと、モックデータによって API 待ちなしで画面が動く状態まで一気に持っていけることが特に大きいです。

社内のデザインシステムを MCP で公開している方は、似たアプローチで自分たちのワークフローを自動化できるかもしれません。
