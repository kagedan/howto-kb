## Bug 1：Alpaca 403 Insufficient Buying Power

### 症状

Discord に `⚠️ Order FAILED — AMZN LONG: insufficient buying power` が届く。

### 原因

ポジションサイジングの equity cap の計算に `equity`（含み益込み総資産）を使っていた。

```
// 修正前
const maxByEquity = Math.floor(equity / entry);  // equity = 含み益込み
quantity = Math.min(quantity, maxByEquity);
```

複数ポジション保有中だと `equity` は大きく見えるが、実際の buying power（使える現金）は少ない。AMZN(~$200)で `equity=$10,000` なら `max=50株` → 注文額 `$10,000` → buying power が `$3,000` しかなければ 403。

### 修正

Alpaca API が返す `buying_power` フィールドをパースして cap に使う。

```
// 修正後
const capital      = buyingPowerCap ?? equity;  // buying_power を優先
const maxByCapital = Math.floor(capital / entry);
quantity = Math.min(quantity, maxByCapital);
```

`account-tools.ts` はすでに `buyingPower` を返していたが、`hermes.ts` が `equity` しか読んでいなかった。

## Bug 2：Claude が毎5分同じシグナルを評価する（∞ WAIT ループ）

### 症状

META SHORT の phantom が毎5分まったく同じ内容で記録され続ける。

```
👻 Phantom recorded — META SHORT  entry=608.51  23:12
👻 Phantom recorded — META SHORT  entry=608.51  23:17
👻 Phantom recorded — META SHORT  entry=608.51  23:22
...（7回繰り返し）
```

### 原因

META には2つの `structural` エンジンがある：

* Engine 0（SMA20/50, RSI ob=60）
* Engine 1（SMA50/100, RSI ob=65）

`engineKey = "META:structural"` で衝突していた。

```
サイクルN:
  Engine 0: SELL → WAIT → lastAction["META:structural"] = "SELL"
  Engine 1: HOLD（RSI=64 < 65）→ lastAction["META:structural"] = "HOLD"（上書き！）

サイクルN+1:
  Engine 0: prev="HOLD", action="SELL" → actionChanged=true → Claude 再呼び出し！
```

### 修正

engineKey にインデックスを追加。`"META:structural:0"` と `"META:structural:1"` で独立した状態を管理。

```
// 修正前
const engineKey = `${symbol}:${params.engineType}`;

// 修正後
const engineKey = `${symbol}:${params.engineType}:${params.engineIndex}`;
```

`getEngineConfigs()` で各エンジンに配列インデックスを付与：

```
return engines.map(({ type, score, ...overrides }, idx) => ({
  ...base,
  ...overrides,
  engineType:  type,
  engineIndex: idx,  // 追加
}));
```

## Bug 3：デイリーレポートの Win Rate が NaN%

### 症状

毎朝9時JST の Discord レポート：

```
📈 All-time
Win rate: NaN% (0 closed) · No consecutive losses
```

### 原因

`getOverallWinRate(0)` を `minTrades=0` で呼んでいたため、closed トレードが 0 件のとき `0 / 0 = NaN` が返っていた。

```
// memory.ts（修正前）
getOverallWinRate(minTrades = 5) {
  // ...
  if (closed.length < minTrades) return { winRate: null, closedCount: closed.length };
  // minTrades=0 かつ closed.length=0 のとき 0 < 0 = false → 通り抜けて NaN
  return { winRate: wins / closed.length, closedCount: closed.length };
}
```

### 修正

`closed.length === 0` を明示的にチェック。

```
if (closed.length === 0 || closed.length < minTrades)
  return { winRate: null, closedCount: closed.length };
```

## Bug 4：phantom LOSS が5件同時に通知される

### 症状

```
❌ Phantom LOSS — GBP_USD SHORT  16:00:11
❌ Phantom LOSS — GBP_USD SHORT  16:00:11
❌ Phantom LOSS — GBP_USD SHORT  16:00:11
❌ Phantom LOSS — GBP_USD SHORT  16:00:11
❌ Phantom LOSS — GBP_USD SHORT  16:00:11
```

### 原因

Bug 2（engineKey衝突）が修正される前に、VPS の `trade-memory.json` に同一内容の phantom が5件蓄積されていた。SL ヒット時に5件が同時に resolve され、それぞれ Discord 通知が送られた。

### 修正

`checkPendingOutcomes` の通知ループで `(symbol:direction:entryPrice:SL:TP)` をキーに重複抑制。

```
const notifiedKeys = new Set<string>();
for (const e of resolved) {
  const key = `${e.type}:${e.symbol}:${e.direction}:${e.entryPrice}:${e.stopLoss}:${e.takeProfit}`;
  if (e.type === "phantom" && notifiedKeys.has(key)) {
    logInfo(`[${e.symbol}] Suppressed duplicate phantom notification`);
    continue;
  }
  notifiedKeys.add(key);
  void notifyFn({ ... });
}
```

リアルトレードは重複抑制しない（安全側のデフォルト）。

---

## 修正を素早く特定できた理由

* **decision.log** に全シグナルと判断が構造化テキストで残る
* **Discord通知** がリアルタイムで何が起きているかを教えてくれる
* **equity.jsonl** で残高の異常な変動を検知できる
* **last-action.json** でシグナル状態を再起動をまたいで追跡できる

本番システムを観測可能（Observable）にしておくことで、バグの原因特定が大幅に速くなりました。
