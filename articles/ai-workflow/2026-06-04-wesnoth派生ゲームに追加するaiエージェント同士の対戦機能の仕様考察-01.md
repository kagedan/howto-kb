# はじめに

![](https://static.zenn.studio/user-upload/93afde4a9cf6-20260604.png)

本記事は、オープンソースのターン制ストラテジー『The Battle for Wesnoth』をベースに、ブラウザで気軽に遊べるように開発した[Wesnoth Tactical Puzzles](https://shisaku4go.github.io/webnoth/tactical-puzzle)（ソースコードは[こちら](https://github.com/shisaku4go/webnoth)）の機能追加に関する、AIとの壁打ちの備忘録です。

今回検討している新機能は、**「AIエージェント同士の対戦と、それを人間が観戦する機能」** です。

プロンプトや言語モデルの差がどのように勝敗を分けるのか。そして、戦略ゲームにつきものの「運（乱数）」に対してAIはどう立ち向かうのか。人間が自分で操作するのとは一味違う、**「AIの知恵比べを楽しむ」** という新しい遊び方を模索しています。(AIの題材として将棋やチェスは既にやり尽くされている部分もあります)

---

## 1. プロジェクトの背景とコンセプト

本機能のコアコンセプトは、**「人間は監督（プロンプター）となり、AIエージェントたちの知恵比べを観戦する」** というものです。

```
[ ユーザー (監督) ] ➔ プロンプト調整 / モデル選択 (Gemini, Claude等)
       │
       ▼
[ AIエージェント ] ➔ 意思決定 (移動・攻撃・スキル)
       │
  （シミュレーション）
       ▼
[ ライブ盤面 (Webnoth) ] ➔ リアルタイム実況AIが盛り上げる！
```

* **プロンプトによる戦略の差別化**: ユーザーはエージェントの性格（「慎重な防衛重視」「機動力を活かした電撃戦」など）をプロンプトで記述し、自軍の強みを引き出します。
* **LLMによるリアルタイム実況・解説**: 盤面のログやダイスロールの結果をLLMにインプットし、ドラマチックなライブ実況をリアルタイム生成します。
* **人間の介入（ピンチヒッター窓口）**: 基本は観戦ですが、時には人間が1ターンだけ直接コマンドを入力し、「ピンチヒッター」として介入できる要素も検討しています。

---

## 2. コア課題：少ユニットにおける「運（RNG）」と「知性」のジレンマ

開発初期のプロトタイプは「3〜4体のユニットによる局地戦」をベースにしていました。ここでゲームデザイン上の大きな課題が浮上します。

### なぜ運で決まってしまうのか？

1. **高い乱数（RNG）性**: Wesnothの戦闘システムは「命中率（CTH）に基づく確率的打撃」がベースです。70%の攻撃が3連続で外れることも珍しくありません。
2. **極小の戦力分母**: 3〜4体しかいない場合、1体が不運なダイスで倒されるだけで戦力が**25%〜33%減少**し、そのまま即敗北に直結します。これでは、AIがどれほど優れた戦術を考えていても、ダイスの神様の一言で全てが台無しになってしまいます。

### 解決へのアプローチ：Wesnothのアイデンティティを守る

戦闘の乱数を排して「確定ダメージ（期待値）」に変更するアイデアもありましたが、最終的に **「乱数はそのまま残す」** という決断に至りました。

* **Wesnothの面白さの継承**: 歴史あるOSSゲームであるWesnothのルールを改変しすぎると、その面白さのコアを失ってしまうため。
* **不確実性への対応力こそが「知性」**: 「運の偏り」という不確実な状況に対し、LLMがどうリカバリーし、堅実な布陣を組み直せるかを見る方が、AIの性能評価（そしてエンタメ）として圧倒的に面白いと考えたからです。将棋やチェスなどの完全情報ゲームとは異なる、本作ならではの差別化要素になります。

---

## 3. 運をコントロールし、知性を引き出す3つの設計方針

乱数ロジックには手を触れず、ゲームの「マクロなルール」を工夫することで、運ゲー化を防ぐ設計方針を立てました。

### ① マップの拡大とユニットの増量（大数の法則の活用）

* **設計**: マップサイズを `13×9`〜`15×10` にしたり、ユニット数を**片側 6〜8体**に増やすことを検討しています。
* **効果**: 1体の損失が全体の1割程度に抑えられるため、運による事故死をリカバーする余地が生まれます。また、前線の形成や後衛の保護など、AIの「フォーメーション戦略」がより顕著になります。

### ② シリーズ戦（4勝先取 / 日本シリーズ方式）の導入

* **設計**: 1戦限りの一発勝負ではなく、最大7試合を行い、4勝先取したエージェントを勝者とします。
* **効果**: 試合数を重ねることで確率が収束し、最終的には実力の高いエージェント（優れたプロンプト・頑健なモデル）が勝ち残ります。
* **エンタメ性の向上**: 試合の合間に、ユーザー（監督）が「前回の敗因」を分析し、**次の試合に向けてAIのプロンプトを微調整（ダグアウト指示）できる**ようにします。対戦格闘ゲームのインターバルのような、熱いメタ駆け引きが生まれます。

---

## 4. 提案された3つのゲームモード

事前準備の楽しさと、リアルタイムな戦術戦の楽しさを両立させるため、以下の異なるモードを検討しています。

### モードA：アリーナ・ドラフトモード（事前ビルド＆シナジー重視）

本番中の追加雇用はなく、事前にドラフトした7体のみで戦うモード。

```
[共有プール (Lv1 15体)] ➔ スネークドラフト (A➔B➔B➔A) ➔ 7体編成の決定 ➔ バトル開始！
```

* **ドラフトのメタゲーム**: 相手が「物理耐性壁のSkeleton」を取ったら、こちらは「衝撃特効のHeavy Infantryman」や「必中火炎のMage」で受ける、といった事前ドラフトの知恵比べが楽しめます。

#### 📝 ドラフトシミュレーションの例（スネークドラフト：7体ピック）

* **プール例**: Spearman, Wose, Mage, Orcish Assassin, Saurian Skirmisher, Skeleton, etc.
* **チームA（防衛シナジー型）**: `Elvish Shaman` ➔ `Wose` / `Dwarvish Fighter` ➔ `Mage` / `Spearman` ➔ `Heavy Infantryman` / `Elvish Fighter`
* **チームB（機動暗殺・魔法型）**: `Orcish Assassin` / `Saurian Skirmisher` ➔ `Dark Adept` / `Mage` ➔ `Skeleton` / `Orcish Grunt` ➔ `Troll Whelp`

---

### モードB：クラシック・タクティカルモード（インゲーム経済＆適応重視）

本家Wesnothと同様に、陣営（Faction）を選択し、ゲーム中に獲得したゴールドで動的に雇用していくモード。

* **ルール**:
  + プレイヤーは特定のFaction（Rebels, Loyalists等）を選択し、リーダー1体＋初期ゴールドでスタート。
  + マップ上の「村」を占領して毎ターンゴールド収入を得る。
  + リーダーが城（キープ）にいる時のみ、ゴールドを消費してLv1ユニットを動的に雇用可能。
* **面白さのポイント**:
  + 「戦闘の勝ち負け」だけでなく、「村を奪い合う経済戦（マクロ戦略）」が生まれます。
  + LLMは戦況や相手の出方に合わせて「今、どのユニットを雇用するのがベストか」「今はゴールドを温存すべきか」を判断する必要があり、AIの臨機応変なリアルタイム適応力試されます。

---

### モードC（ハイブリッド）：メルセナリー・ドラフトモード

上記2つのいいとこ取り。事前ドラフトで「自分専用の雇用可能プール（デッキ）」を作り、ゲーム中はそこからゴールドを消費して動的雇用するモード。

---

## おわりに

まだ仕様の検討を始めたばかりのブレインストーミング段階ですが、LLMやAIエージェントを活用して作ったゲームは増えていても、**「ゲームシステムの中核（エージェント・実況・ゲームマスター）」** として組み込む試みは、これからのゲーム開発において非常に大きな可能性を秘めていると感じています。

今後、実際にプロトタイプを動かしてみた結果や、プロンプトの工夫、LLMごとの挙動の差なども記事にしていければと思います。備忘録がてら、同様の試みをしている方の参考になれば幸いです。
