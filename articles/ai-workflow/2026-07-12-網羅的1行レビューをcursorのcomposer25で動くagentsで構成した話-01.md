## 網羅的1行レビューは使えているか？

前回の記事から、網羅的1行レビューを継続して使用しています。  
理由は効果をかなり感じられるからですね。

AIエージェントによるレビュープロセス自体はありますが、それを抜けるバグを補足する能力は単純な仕組みである割には高い気がします。

どういうワークフローで使っているかという文脈がないと、AIエージェントを使用した普通のレビューとの違いがわからないと思いますので軽くそこも説明します。

## 構築している計画実行の仕組みとレビュー

私はプロジェクトを以下のような単位に分割してワークフローを回しています。

**flow** : 達成したいマイルストーンの集合を定義  
**task** : 実際の文書作成やプログラム実装を行う

**flow**を実行可能レベルで事前設計して実行プランを計画しておきます。  
実行時は**flow**を複数ステップの**task**にあらためて分割し、小さい実行ループを回すような形ですね。

仕組み自体は特段AI向けというわけでもないですが、AIエージェントに1つの**flow**全体を回せるくらいの仕組みであることはわかってます。  
**task**は以下のようにループになっています。

* **task指示書**の提示
  + **task指示書**という文書単位でタスクを指示します。
  + **task指示書**のレビューを次の視点で行い問題があれば差し戻され修正します。
    - 実行可能性、インプット不足、完了定義(DoD)の不足、不明確さなど
  + 何回か繰り返されて指示書が完成します。
* **task指示書**の実行
  + タスク実行役が**task指示書**に従ってタスクを実行します。
  + タスク実行しつつ**taskログ**を指定フォーマットで残します。
  + **完了の定義(DoD)** を全て完了させ、**task完了報告書**を残します。
* **taskの完了判定レビュー**の実行
  + **task指示書**の**DoD**と**成果物**の実体を照合する形でレビューします。
  + レビュー担当が、「どこがDoDや設計・仕様と異なるか」という指摘だけを列挙
  + 管理役がPassかどうかを判断します
    - DoD違反ならFailとして同じtaskへ差し戻し
    - 別途あがった改善項目は次taskや別flowへの申し送り

網羅的1行レビューが役に立つのは**taskの完了判定レビュー**部分です。

## 網羅的1行をCursorのComposer2.5で構築する

最新Xのポストで**Composer2.5はいいぞ!(意訳)** というのを見かけていて既に使っていました。

とはいえ移植自体はcursorで実行させたけっか以下の感じの構成となりました。  
以前と変わったのはnode.jsを使ってローカルでできる作業は機械的に行って渡すようにした点です。

```
TASK-IDをキーにgit上の追加に対して実施
  → Node: git から差分取得
  → Node: 差分をファイル・変更行（unit）へ分割
  → Cursor: frame x ファイル数（各ファイルの変更意図・観点を生成、不要行を除外）
  → Node: unit × role を answer job に展開
  → Cursor: answer（意図逸脱・仕様逸脱を推測して判定）
  → Node: 結果を集約して Markdown レポート化
```

## sonnet5とcomposer2.5での比較

網羅的１行レビューのレビュワーではモデル性能はそこまで問われません。  
そういう意味ではcomposer2.5は丁度良く、fast指定なしでもサクサクレビュー進みます。  
composer2.5は比較的愚直に仕事してくれるためこういう用途では使いやすいです。

## 大量差分レビュー課題

最初からわかっている課題ですが1行1エージェントだと1000行の差分あると1000エージェントが起動します。  
1000エージェント利用する場合は、エージェントの起動コストのトークンの無駄と、1000エージェントを順番に動かす時間の無駄が課題となります。

### エージェント起動の初期コストにかかるトークン

エージェントは起動時にエージェントとして動くためのプロンプトが最初に読み込まれるようになっているようです。詳しく調べてはいないですがは起動しただけで数千トークンくらい使われているようです。  
5000トークンと仮定して1000エージェントでは500万トークン余計に使われます。  
さらにagents.mdとか置いているとそれも全エージェント読み込んで起動します。

API利用なら単にチャット応答なのでその無駄は掛からないのですがcomposer2.5はAPI利用はできないです。agents.mdとかは抑制することはある程度できるのですが、初期コストの削減には限界があります。

何か役に立つ500万トークンならいいですが不要なトークンなので削減するにこしたことはありません。

### エージェントが動作している時間

トークンの問題以外にも時間の問題もあります。1000エージェントまとめて動かすわけでなく8つずつ動作させるとか現実的な起動数で行わせると1つ1つのレビューは短くてもトータルでは時間がかかります。1000エージェントも動かすと環境によりますが1時間くらいはかかります。

許容できないかというと微妙なとこですが、品質が変わらないなら短いにこしたことはありません。  
composer2.5のfastにしても順番に起動すること自体に時間がかかるので多重起動自体には効きません。

## 多重起動対処と性能への影響

多重起動対処の方法はシンプルにしました。  
単純に網羅的1行レビューの本体は**起動は100エージェントまで**に絞るという対応です

性能への影響は限定的です。網羅的１行コンセプトは以下のようなものだからです。

* 1)　特に大量のコードからのバグ発見能力はモデルの性能依存で大規模モデルが有利
* 2)　小量コード(例えば1行)からのバグ発見能力やバグ推定能力は意外と小規模モデルでも可能
* 3)　小規模モデルに小量のコードを渡すとバグがあるならこういうバグみたいな推測を返す
* 4)　3)の結果を使ってもう裏取りすると1~2%くらい本当のバグが入っている
* 5)　でもAIエージェントは、**やらないことを勝手に判断するため網羅的に実施することは不得意**
* 6)　**流石に1行だけ渡せば全部やるだろう。**

5)6)が意外とエージェントが不得意なところで、愚直に処理することをやってくれません。  
特にAnthropic系のLLMはそういう傾向が強い感じがしますね。  
愚直な実行は**既存エージェント**じゃなくて**API**使って**エージェント**自作して処理するのが現状では一番良い解なのかもしれませんね。

100エージェントなら1000行差分でも1エージェント辺り10行程度変更。このくらいなら問題なく動いてくれそうです。2000～3000行くらいなら対応できそうです。

10,000行変更とかならそもそもバグ見つけるの難しいですからね。1タスクのコード変更は3000行レベル以下にしといた方が多分無難です。

## 疑義の裏取りのAIの性能

ここは完全にモデル性能依存です。  
ただしモデル性能は以下のような性能が2軸あってどちらも大事です。

* 単純なバグを探すモデルの性能（ベンチマークで評価されているような）
* 裏取りを愚直に行うモデルの愚直さの性能（ベンチマークで評価されていない）

裏取りに時間がかかるため、effortは性能とのバランスが良い「低め」を推奨します。effortを上げるほど処理時間もかなり長くなります。

個人的には、CodexとComposerを中心に使い、Fableも補助的に使う構成です。

| モデル | 評価 | 推奨effort | コメント |
| --- | --- | --- | --- |
| GPT-5.5 | オススメ | High程度 | Highくらいで十分 |
| GPT-5.6 | オススメ | Sol：Low以上 Terra：Medium以上 | 用途に応じて調整 |
| Fable5 | サブスクなら候補 | Lowでも可 推奨はMedium以上 | 補助的に使用 |
| Opue4.8 / Sonnet5 | あまりオススメしない | — | 手抜きが多く、effortを上げると時間もかかる |
| Composer2.5 | オススメ | — | 速く、意外と深く探してくる |

## 裏取りのエージェント性能の制御

自由指示のレビューだと問題なのである程度は以下のような施策でエージェント間のブレを小さくしました。

* 不具合調査エージェント動作の規定
  + どのように周辺を探していくか、どこまで辿って行くかなどの方針
* サボらないように中間成果物の規定
  + エージェント動作の規定どおりに辿っていること(サボっていること)がわかる中間成果物の残し方を規定
* 成果物評価エージェントの動作の規定
  + サボらずにどこまで見たかを確認する。
  + 報告書を見て怪しいところは疑義を出して再確認する
