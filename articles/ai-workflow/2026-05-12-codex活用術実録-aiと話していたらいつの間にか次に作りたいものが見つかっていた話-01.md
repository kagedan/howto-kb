「何か作りたいけれど、具体的なネタが思いつかない」  
そんな状態のとき、私たちはつい「世の中にない新しいもの」を探してしまいがちです。しかし今回、AIとの何気ない技術相談から、「自分が無意識に諦めていた不便」を解決するアイデアがポロッと生まれました。

その「発見のプロセス」を記録しておきます。

## 1. 「モノレポはログが混ざるもの」という諦め

現在、WindowsアプリとAndroidアプリを同じリポジトリで管理する「モノレポ」構成で開発を進めています。  
そこで感じていたのが、「コミットログが混ざり合って、履歴が追いづらい」という小さなストレスでした。 私はこれを、「モノレポという構造上、仕方のないことだ」と半分諦めていました。あるいは、自分のGit操作の習熟度が足りないせいだ、と思い込んでいたのです。

## 2. AIが教えてくれた「標準機能」と「UIのギャップ」

この悩みをAIに話してみたところ、意外な事実を突きつけられました。

ここで気づきました。「不便だ」と感じていたのは私のスキルのせいではなく、ツールが私のワークフロー（モノレポ）に最適化されていなかったからなのだ、と。

## 3. 「不満」が「プロダクトの種」に変わった瞬間

AIとの対話は、単なる解決策の提示に留まりませんでした。

**AI（Codex）**: 「モノレポで、フォルダ単位に履歴を気持ちよく読むことに特化した小さいツールは、作る価値かなりあります」

この一言で、視点が変わりました。  
「我慢して既存ツールを使う」という選択肢しかなかった頭の中に、「フォルダ単位の履歴閲覧に特化した、自分専用のビューア」という具体的なプロダクトの形が浮かび上がったのです。

## 4. アイデアの解像度をAIと上げる

作りたいものが「見つかった」後は、AIと会話するほどアイデアの解像度が上がっていきました。

これは「開発の義務」ではなく、「自分が一番欲しい道具の設計図」をAIと一緒に描いている、ワクワクする体験でした。

## 結びに：AIは「気づき」の鏡になる

今回、AIと話して再確認したのは、「自分が抱えている漠然とした違和感をぶつけてみることで、AIがそれを「技術的な課題」として整理し、客観的に「それは作る価値があるよ」と背中を押してくれる。

「何を作ればいいかわからない」というときは、今自分が触っているツールの「ちょっとだけ気に入らないところ」をAIに愚痴ってみる。そこから、あなただけの新しいプロダクトが始まるかもしれません。

GitHubを見にくるユーザー的にはWindowsアプリ、Androidアプリは別だと思うのでリポジトリ分けたほうがいいと思ってるんですよね。

でも今回WindowsアプリからAndroidアプリのコンバートがこんなにできるんだったら、一つアプリでそれが各プラットフォームごとに存在するっていうように考えた場合今の構成でもいいのかなと思たりしてます。どう思いますか？

## 🤖 Codex

悩みどころ、かなり分かります。結論から言うと、今の段階では 同じリポジトリ内で Windows版 と Android版 を別フォルダに置く構成が良い と思います。

つまり今の方向です。

```
VisualProgressTimer/
  README.md
  VisualProgressTimer/          # Windows / WPF
  VisualProgressTimerAndroid/   # Android / Flutter
  docs/
```

理由は、今回の記事やプロジェクトの文脈が「WindowsアプリをAndroidに移植した」という体験そのものだからです。GitHubを見に来た人にとっても、同じリポジトリ内に両方ある方が比較しやすいです。

たとえば読者はこういう見方をしたくなると思います。

* Windows版はどう書かれているのか
* Android版ではどう再構成されたのか
* 通知やタイマー処理はどう変わったのか
* AI移植でどこが共通で、どこがネイティブになったのか

この場合、別リポジトリにすると少し距離ができます。リンクで行き来はできますが、「ひとつの題材として見る」感じは弱くなります。

一方で、別リポジトリが向いているのはこういう場合です。

* Android版だけを独立した製品として公開したい
* Play Store向けのリリース管理をWindows版と完全に分けたい
* IssueやReleaseやCIを完全に別運用したい
* 将来的にWindows版とAndroid版の名前や方向性がズレそう

でも今の Visual Progress Timer は、「同じアプリのプラットフォーム違い」というくくりが自然です。なので、まずはモノレポで良いと思います。

おすすめは、ルートREADMEを少し整えることです。

```
# Visual Progress Timer

Visual Progress Timer is available for:

- Windows: `VisualProgressTimer/`
- Android: `VisualProgressTimerAndroid/`

The Android version was rebuilt from the Windows version using Flutter and Android native alarm APIs.
```

こんな感じで入口を作ると、GitHubを見に来た人も迷いにくいです。

個人的には、今回の価値は「Android版単体」だけではなく、Windows版とAndroid版が並んでいて、移植の考え方が見えることにもあると思います。だから今は別リポジトリより、同じリポジトリ内の別フォルダが一番おいしい形に見えます。

## 🤖 Codex

あります。モノレポでコミットログが混ざって見にくくなる問題はかなり普通に起きます。回避策は「履歴を分ける」のではなく、見える単位を揃えるのが現実的です。

一番おすすめは、コミットメッセージにスコープを付けることです。

```
windows: fix timer drawing on resize
android: add exact alarm scheduling
docs: add migration article notes
root: update repository README
```

もう少しConventional Commits寄りにするなら、

```
fix(windows): redraw timer after resize
feat(android): add exact alarm scheduling
docs(article): outline Zenn essay
chore(root): update gitignore
```

こうしておくと、GitHubの履歴でもかなり読みやすくなります。

さらに見る側はパスで絞れます。

```
git log -- VisualProgressTimerAndroid
git log -- VisualProgressTimer
git log -- README.md docs
```

GitHub上でもフォルダを開いてからHistoryを見ると、そのフォルダに関係するコミットだけ見られます。

あと、1つのコミットでWindows版とAndroid版を同時に触るのは、できるだけ避けると履歴がかなり綺麗になります。たとえば「色テーマを両方に反映」みたいな変更でも、

```
feat(windows): add green timer theme
feat(android): add green timer theme
```

と分けると、あとから追いやすいです。

もう少しきっちりするなら、ルートに CONTRIBUTING.md か docs/development.md を置いて、

```
## Commit scope

Use one of these scopes:

- `windows`
- `android`
- `docs`
- `root`

Examples:

- `feat(android): add notification permission handling`
- `fix(windows): preserve timer color after restart`
```

みたいに書いておくのも良いです。

個人的には、今回の規模なら別リポジトリに分けるより、

同じリポジトリ + フォルダ分離 + コミットスコープ + GitHubのパス別History

が一番扱いやすいと思います。履歴の混在は完全には消えませんが、「読むときに迷わない」レベルにはできます。

## 🧑‍💻 ユーザー

こういう書き方にすればフィルタリングしてコミットログを読めるって言ってますか

---

## 🤖 Codex

はい、そういう意味です。

2種類あります。

**1. コミットメッセージで探しやすくする**

たとえば毎回こう書いておくと、

```
feat(android): add exact alarm scheduling
fix(windows): redraw timer after resize
docs(article): add Zenn outline
```

あとから検索しやすいです。

```
git log --grep="android"
git log --grep="windows"
git log --grep="docs"
```

ただしこれは「メッセージにちゃんと書いてある」ことが前提です。

**2. 変更されたフォルダで絞る**

こちらはもっと確実です。

```
git log -- VisualProgressTimerAndroid
git log -- VisualProgressTimer
```

これで、そのフォルダに変更が入ったコミットだけ見られます。

GitHubでも、対象フォルダを開いて History を押すと、そのフォルダに関係する履歴だけ見られます。

なので運用としては、

コミットメッセージに android/windows のスコープを付ける + フォルダ単位で git log / GitHub History を見る

の組み合わせが一番見やすいです。

## 🧑‍💻 ユーザー

えっと自分一番はじめはGitのコマンドとかを使っていたんですけど、その後クライアントツールを使うようになたので今はgitのコマンドほぼ覚えてないんです。

こういう風にLogをフィルタリングしたり、フォルダー単位でログを表示するのってモノレポではすごく有効な気がしています。

今メインで使ってるはSourceTreeだったんですけどこういう機能無いんですよね。

単純にgitのログをフォルダ単位で見るとかフィルタリングしてみるとかそういうツールありますか？または自分すごい必要だなって思ったんで作るのってどう思いますか。

## 🤖 Codex

あります。しかも今の話、かなり「作る価値がある」タイプの不満だと思います。

まず既存ツールだと、候補はこのあたりです。

* **TortoiseGit** — Windowsならかなり相性いいです。エクスプローラーでフォルダ右クリックしてログを見る、という使い方ができます。フォルダ単位の履歴、ログフィルタ、パス関連の表示に強いです。
* **JetBrains系IDE** — IntelliJ IDEA / Rider / Android Studio など。Git Logに Paths フィルタがあり、特定フォルダのコミットだけ見られます。モノレポにはかなり向いています。
* **GitLens / GitKraken系** — VS Codeを使うならGitLensのCommit Graphがかなり良さそうです。検索対象に message / author / file / change などがあります。
* **Tower** — ファイル履歴はかなり丁寧です。ただし「フォルダ単位のモノレポ履歴ビュー」が主目的なら、事前に試した方がいいです。

ちなみにSourceTreeにも昔から「選択ファイル/フォルダのログを見る」系の操作はあるようです。File Status でツリー表示して、対象を右クリックして Log Selected... という流れです。ただ、普段の履歴ビューから自然に「このフォルダだけ」と切り替える体験ではないので、モノレポ用途だと物足りなく感じるのは分かります。

**作るのはどうか — かなりアリだと思います。**

特に作るなら、普通のGitクライアント全部入りではなく、モノレポ用の履歴ビューアに絞ると価値が出そうです。

たとえばこういう機能です。

```
左:
  windows
  android
  docs
  root

中央:
  選んだスコープに関係するコミットだけ表示

右:
  変更ファイル
  diff
  commit message
```

さらに便利そうなのは、

* フォルダ単位で履歴をワンクリック切り替え
* `feat(android):` みたいなスコープでも絞れる
* パスフィルタとメッセージフィルタを同時に使える
* 「このコミットは android と docs にまたがっている」みたいに見える
* モノレポ用のスコープ定義ファイルを置ける

みたいな方向です。

個人的には、これはAI時代にもかなり需要あると思います。AIが複数プラットフォームをまたいで変更できるようになるほど、人間側には「どの領域がいつ変わったか」を見る道具が必要になります。

まずは既存なら TortoiseGit か GitLens を試すのが良さそうです。  
でも「モノレポで、フォルダ単位に履歴を気持ちよく読む」ことに特化した小さいツールは、作る価値かなりあります。

## 🤖 Codex

はい、これは 別プロジェクトにした方がいい と思います。

理由は、このツールは VisualProgressTimer の補助機能ではなく、完全に別の価値を持つアプリだからです。

VisualProgressTimer は: タイマーアプリ  
新しく作りたいものは: モノレポ用Git履歴ビューア

なので、同じリポジトリに入れるとテーマが混ざります。GitHubで見に来た人にも「タイマーアプリを見に来たのにGitクライアントが入っている」という感じになります。

おすすめは別リポジトリです。

```
VisualProgressTimer/
  Windows版
  Android版

MonorepoHistoryViewer/
  Git履歴ビューア
```

名前案としては、

* MonorepoLogViewer
* RepoScope
* ScopeLog
* PathLog
* GitScope
* Monolog（ただし名前は少し被りそう）
* ScopeTrail
* RepoLens（これも被りそう）
* GitPathView

最初のMVPはかなり絞っていいと思います。

1. ローカルGitリポジトリを開く
2. フォルダ一覧を表示する
3. フォルダを選ぶ
4. `git log -- <folder>` 相当の履歴を表示する
5. コミットを選ぶ
6. 変更ファイル一覧とdiffを見る

これだけでもSourceTreeで感じている不満にはかなり刺さります。

技術選定は、Windows中心ならまずは次のどれかです。

* **Tauri + React/Svelte** — 軽いデスクトップアプリにしやすい、Gitコマンドを呼びやすい
* **Flutter desktop** — 今回の流れとも合う、Windows/macOS/Linuxに広げやすい
* **C# / WPF or WinUI** — Windows専用ならかなり素直、既存のWindows開発経験を活かせる

個人的には、将来的に配布しやすくするなら Tauri か Flutter desktop が良さそうです。  
ただ、今のあなたの流れだと「まずAIと一緒に一気に作って試す」なら Flutter でもかなり楽しいと思います。

これはタイマーアプリとは別リポジトリで始めるのがきれいです。タイマー側の記事では「この体験から、モノレポ履歴ビューアも作りたくなった」と後日談にできます。

## 🤖 Codex

はい、別プロジェクト・別セッションにすると、この会話そのものは基本的に引き継がれない前提で考えた方がいいです。

なので、新しいセッションの最初に 短い引き継ぎメモ を貼るのが一番確実です。こんな感じで十分です。

```
モノレポ用のGit履歴ビューアを新規プロジェクトとして作りたいです。

背景:
- SourceTreeを使っているが、モノレポでフォルダ単位の履歴を見る体験が弱いと感じた
- `git log -- path/to/folder` のような見方をGUIで簡単にしたい
- WindowsアプリとAndroidアプリが同じリポジトリにあるような構成で、
  各プラットフォームごとの履歴を見やすくしたい

作りたいもの:
- ローカルGitリポジトリを開く
- ルート直下のフォルダをスコープとして表示する
- スコープを選ぶと、そのフォルダに関係するコミットだけ表示する
- コミットを選ぶと、変更ファイル一覧とdiffを表示する
- コミットメッセージの `feat(android): ...` のようなスコープでも絞れるようにしたい

MVP:
1. ローカルGitリポジトリを選択
2. フォルダ一覧表示
3. フォルダ単位で `git log -- <folder>` 相当の履歴表示
4. コミット詳細表示
5. 変更ファイルとdiff表示

技術候補:
- Tauri
- Flutter desktop
- C# / WPF or WinUI

まずは技術選定とMVP設計から相談したいです。
```

もう少し短く始めるならこれでも大丈夫です。

```
SourceTreeの代わりというより、モノレポで「フォルダ単位のGit履歴」を見ることに
特化した小さいGUIツールを作りたいです。

MVPは、ローカルGitリポジトリを開いて、ルート直下フォルダを選ぶと
`git log -- <folder>` 相当の履歴が表示され、コミットを選ぶと
変更ファイルとdiffが見える、というものです。

まず技術選定と設計を相談したいです。
```

個人的には、新セッションには上の「短い版」を貼って、最初にこう言うのが一番スムーズです。

> まずはMVP設計を一緒に詰めてください。まだ実装は始めなくていいです。

そうすると、いきなりコード生成に走らず、アプリの形から相談できます。
