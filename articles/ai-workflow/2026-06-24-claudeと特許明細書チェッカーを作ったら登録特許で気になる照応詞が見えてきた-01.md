## この春、作ったもの

2026年3月末、日本語特許明細書の方式要件を形態素解析で検査するOSSツール [meisai-checker](https://github.com/aki0ka/meisai-checker) を作り始めた。

コードは Claude が書いている。自分はしたいことを都度伝えるだけで、日々機能を追加している。以前、正規表現のみで特許明細書をチェックする JavaScript ツールを自作したことがあった。形態素解析ライブラリ（kuromoji）を導入しようとしたが難しくてそこで止まっていた。バイブコーディングが流行しはじめたのを機に、Claude と fugashi（Python の MeCab ラッパー）で作り直してみようと思い立った。経緯は[こちらの記事](https://note.com/humble_frog2601/n/n56acc1c6fda3)に書いた。

## チェック項目

ツールがカバーする項目は大きく4系統だ。

**方式・記載要件**：請求項の照応詞（前記・当該）の先行詞の有無、図面符号と要素名の対応、サポート要件（請求項の用語が詳細説明に記載されているか）、自己引用・マルチマルチクレームなどの従属関係の誤り、明細書・願書の記録項目（様式第26・第29）。

**明確性**：係り受けの曖昧性、請求項内の「適宜」「好ましくは」などの曖昧表現、企業名・価格などの非技術的事項。明細書本文の程度副詞（「極めて」「非常に」等）はinfoレベルで検出。

**文章品質・文法**：冗長表現・意味重複・読点欠落・一文120字超・こそあど指示代名詞など（特許ライティングマニュアル準拠）、助詞の連続・「の」の過剰連鎖。

**構造・文字コード**：JIS Z 8301準拠の構造確認、段落番号・句読点・見出し番号の書式。

ツールを育てるうえで一番時間を使っているのは精度調整だ。見落とし（偽陰性）はできるだけ減らしたい。誤検知（偽陽性）はノイズになるので、削れると分かったら直す。実案件の明細書を読み込んで「なぜ警告が出るか・出ないか」を確認しながら、ルールを一つずつ修正している。

このツールはバイブコーディングに向いている題材だと思っている。外部にデータを送信せず、テキストを受け取ってパターンに合致する箇所に説明を添えて返すだけだ。副作用がなく動作が見通せるので、AIが書いたコードをそのまま使いやすい。

## 気になったパターン

弁理士として明細書を読んでいると、照応詞（「前記」「当該」）の使い方に気になるパターンがあることに気づく。ツールを作るうちに、そのパターンが言語化できるようになってきた。正直に言えば、自分の書いた請求項にも同じパターンがあった。

本記事では、そのパターンをJ-PlatPatの全文検索で実際に確認し、具体例としてツールにかけてみる。

## 前記の役割（30秒）

請求項で「前記X」と書くとき、「X」は必ず先行する文で導入されている必要がある。これを先行詞という。

```
熱電対を備え、
↓
前記熱電対の測温部が〜         ← 「前記熱電対」で参照
```

「前記X」は指示対象を**一意に固定する**。Xがすでに導入されており、かつそのXが1つしかないことを前提としている。

## J-PlatPatで件数を確認する

「複数の前記X」「各前記X」「それぞれの前記X」——「1つしかない」はずのXを複数扱いにしている書き方がある。実際にどれくらいあるか、J-PlatPatで確認してみた。

文献種別を「特許（登録済）」に絞り、請求の範囲フィールドを対象に検索した。

| 検索ワード | 全範囲・登録済（2026-06-21時点） | 2023年公開 |
| --- | --- | --- |
| `複数の前記` | **214,499件** | **12,344件** |
| `各前記` | **45,171件** | **2,419件** |
| `それぞれの前記` | **56,499件** | **2,818件** |

2023年公開分だけで「複数の前記」が1万2千件超、登録済み全範囲では20万件超ヒットする。これらがすべて問題というわけではないが、先行詞が単数で導入されているのに「複数の」「各」「それぞれの」を前置きしているケースが含まれる。

## 実例：複数の前記容器アセンブリ

具体例として、存続期間満了で消滅した外国企業の登録特許を見てみよう。

技術は「バケツと蓋のロック機構」だ。蓋を外せないようにするラチェット式のロック部材を備えた容器アセンブリの特許だ。

請求項1の構造はこうなっている：

![](https://static.zenn.studio/user-upload/8e10d6433d6b-20260624.png)

```
頂部開放型容器…および、蓋…を含む容器アセンブリにして、
（中略）
前記蓋がさらに、複数の前記容器アセンブリを積み重ねる場合に、
ロック用部材を保護する保護壁を有する容器アセンブリ。
```

「複数の前記容器アセンブリ」——ここで矛盾が生じる。

前置部（「…を含む容器アセンブリにして」）ですでに「容器アセンブリ」が単数の要素として導入されている。「前記容器アセンブリ」はその正当な照応参照だ。問題は「複数の」との組み合わせにある。「前記」は単数で導入された特定の要素を指し、「複数の」はその要素が複数存在するという前提を加える——この2つは両立できない。

英語原出願の請求項1にはこの保護壁機能は記載されていない。明細書に "multiple container assemblies"（不定の複数）として書かれていた内容が、日本での審査対応で請求項に組み込まれた際にこの形になったと見られる。

**修正するなら：**

```
前記蓋がさらに、複数の容器アセンブリを積み重ねる場合に、
ロック用部材を保護する保護壁を有する容器アセンブリ。
```

「前記」を外すだけでよい。容器アセンブリを「複数積み重ねる」のは一般的な使用状況の説明であり、この請求項で先行導入した特定の1つを指しているわけではない。

## meisai-checkerで確認する

上記のパターンは [meisai-checker](https://github.com/aki0ka/meisai-checker) が自動検出する。このテキストを読み込んでみた。

起動は `uvx meisai-checker` の一コマンドで、GUIが立ち上がる。Macではアプリ化してDockに固定する方法もある。Windowsでは `.bat` ファイルを1つ作るだけでダブルクリック起動できる（詳細はREADME参照）。txt・docx・PDFをドラッグ＆ドロップするだけで解析が始まる。

### M3：照応詞チェック

![](https://static.zenn.studio/user-upload/39209aa05f1d-20260620.png)

右パネルのM3チェック結果に「複数の前記容器アセンブリ」のWARNが出ている。書き換え候補も提示される。照応詞一覧では「前記容器アセンブリ」が「先行詞重複」として△マークで示される。

「当該X」も同じ問題を持ちうる。先行詞が明示されていなければ、読み手が「今アクティブな対象はどれか」を自力で特定するしかなく、係争で争点になりやすい。meisai-checkerは当該についても先行詞の有無を検査する。スクショ左上の「該容器 先行詞なし」がその例だ。「頂部開放型容器」として導入されているのに「該容器」で照応しようとしており、先行詞と完全一致しないためエラーになっている。

### M4：符号チェック

同じファイルを読み込んだとき、M3（照応詞）以外にも気になる検出があった。M4（符号）チェックが「符号『78』に複数の要素名が対応しています：密閉リング・スロット」をエラーとして上げてきた。

明細書本文（【0020】）には「密閉リング７８をスロット７８内に配置することができ」という記述がある。同じ番号が2つの要素に割り当てられており、英文翻訳時の付番ミスとみられる。

![](https://static.zenn.studio/user-upload/d94b0cd1718e-20260620.png)

「複数の前記」と「符号の重複」——どちらも人間の目には通り抜けやすい。

## おわりに

取り上げたパターンはいずれも特許掲載公報に多数存在する。

なぜこのような表現が残るのか。審査過程で補正できる範囲が大幅に制限されることがある。書き直しの機会が尽きた段階で登録に至ることが多い。意見書で文意を説明する対応もできるため、表現を直すインセンティブが働きにくい。

それでも、問題がないわけではない。翻訳局面では、先行詞が不明確な照応詞は訳者が意図を補完するしかなく、補完の内容が揺れる。権利行使でも、侵害判断ではクレームの各構成要件を被疑侵害品の各要素に一対一で対応させる——この時点で、照応詞の曖昧さが権利範囲の広さに直接影響する。

今は審査官が文意を補完して問題を見逃してくれることが多い。しかし、その状況が続くとは限らない。JPOも審査効率化のためにAI活用を進めており、出願人側でもAIで明細書を書く動きが加速している。両側でAIが介在するようになると、効率化が優先される以上、曖昧な表現は補完されるより切り捨てられる可能性がある。

出願前のチェックが有効なのは、書き手が意図を知っているこの段階でしか直せないからだ。

ただし、ツールの警告が正しいとは限らない。形態素解析と浅い言語学の知識をパターンとしてクレームに当てはめているだけで、文脈を読む力はない。警告は「ここを確認してみて」という目印として使い、判断は書き手がする。

meisai-checkerはOSSとして公開している。出願前の確認に使ってほしい。

また、特許・知財に興味があるひと、言語学の視点で日本語に関心があるひとにも使ってみてほしい。AIに興味があるひとは、ツールの出力をそのままAIに渡して「どう直したらいいか」「この指摘は妥当か」と聞いてみるのも面白い。外部通信は一切ないので、手元の公報で気軽に試してほしい。
