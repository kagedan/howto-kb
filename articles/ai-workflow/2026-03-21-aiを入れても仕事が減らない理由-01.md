AIを入れたのに、仕事が減らない。

議事録を整え、提案書を直し、結局いちばん時間を使っているのは自分だった。 これは、AIを使っているのに最後は自分で清書しているコンサル・企画職向けの話です。

原因はAIの性能ではない。仕事の分け方が間違っていた。

僕はコンサル業務を「考える作業」と「作る作業」に分けて、後者をClaude CodeとGitに移した。 清書と整形に消えていた時間が、月30〜40時間なくなった。

---

## この記事で分かること

* AIを入れても仕事が減らない構造的原因
* 「考える作業」と「作る作業」を分ける基準
* 月30〜40時間を削減した設計と運用法

---

## AIを入れても変わらなかった本当の理由

最初の3ヶ月、AIを使って作業量はほとんど変わらなかった。

ChatGPTに「この議事録を整理して」と投げる。出てきた文章を見て直す。フォーマットが違う。もう一度投げる。また直す。気づいたら、手で書くより時間がかかっていた。

問題はAIの性能ではない。 **仕事の設計だった。**

コンサルの業務を分解すると、大きく2種類に分かれる。

**考える作業**: 問いを立てる、仮説を検証する、論理を構築する、クライアントに何を伝えるかを決める。これは人間がやるべき仕事だ。AIに投げると、それっぽいが空洞の答えが返ってくる。

**作る作業**: 議事録を整形する、報告書のフォーマットを整える、調査結果をスライド用に変換する、進捗テーブルを更新する。これは判断が不要な仕事だ。ルールを決めれば、AIが人間より速くできる。

ChatGPTに投げていた頃、この2つが混在していた。「整理して」という指示の中に、判断（何を残すか）と整形（どう書くか）が両方含まれていた。だからAIが迷い、僕も迷い、往復が増えた。 ここが詰まりだった。

### 混在させると何が起きるか

判断と整形を一緒にAIに投げると、3つの問題が起きる。

1. AIが勝手に「何を重要か」を判断する。その基準が自分と違う
2. 修正が発生するたびに「前回どうしたっけ」から始まる
3. 成果物がセッションをまたいで積み上がらない

つまり、混在させる限りAIは「毎回リセットされる作業者」のままだ。何も蓄積しない。

## Claude CodeとGitが解決策になる理由

ChatGPTとClaude Codeの根本的な違いは、**ファイルを直接読み書きできるかどうか**だ。

ChatGPTはチャット画面の中で完結する。出てきた文章は自分でコピーして貼り付けなければならない。次のセッションでは「前回こういう方針でした」をまた説明し直す。

Claude Codeは違う。ローカルのCLAUDE.mdを起動時に自動で読む。案件フォルダの議事録も、タスクリストも、過去の成果物も全部参照できる。そして作った結果をファイルとして保存する。Gitと組み合わせれば、変更の履歴が自動で積み上がる。

**これは「AIが社内に住み込んだ」状態に近い。**

毎朝「おはようございます。今日の定例の議事録を書いてください」と言えば、Claude CodeはCLAUDE.mdからクライアントの背景を確認し、前回の議事録のフォーマットを参照し、正しい場所に保存する。背景説明はゼロだ。

### 実際の構成

僕の案件フォルダはこういう構成になっている。

```
案件/[クライアント名]/
├── CLAUDE.md              # AIへの案件指示書（背景・人物・注意事項）
├── _タスク.md             # 今週のタスク・Waiting For
├── 議事録/                # 定例・臨時MTGの記録
├── 成果物/                # 報告書・資料（MD + PPTX）
├── 調査/                  # リサーチ・分析ファイル
└── .claude/skills/        # 案件固有のSkillsファイル
```

CLAUDE.mdには、このクライアントの課題、これまでの経緯、関係者の役割、使っているフォーマットのルールを書いてある。

平均300〜400行。これを読んだClaude Codeは、初回セッションから「分かっている人」として動く。

## 暗黙知をSkillsファイルに移植する

「考える作業」と「作る作業」を分離したとして、「作る作業」をAIにやらせるには、**やり方のルール**をどこかに書いておく必要がある。

口頭で教えるか、CLAUDE.mdに全部書くか。どちらも限界がある。

最初はCLAUDE.mdに全部詰め込んでいた。800行になった。起動時の読み込みが遅くなり、指示が衝突し始めた。「どこに何が書いてあるか」が分からなくなった。

**解決策がSkillsファイルだ。**

.claude/skills/ディレクトリに、タスク別のMarkdownファイルを置く。Claude Codeは、関連するタスクを検出した時に自動でそのSkillを参照する。

### 具体例1: 議事録作成

議事録作成のSkillファイルは、こんな内容になっている。

```
## 議事録作成 Skill

## いつ使うか
ユーザーが「議事録を書いて」「MTGの記録を」と言ったとき

## アウトプットの形式
- H2: 日付・参加者・目的
- H3: 決定事項（★マーク必須）
- H3: 次回アクション（担当者・期日付き）
- H3: 論点・議事

## クライアント固有ルール
- 敬称はすべて「さん」統一
- 決定事項は必ずタスクリストにも転記
- 議事録は [案件名/議事録/YYYY-MM-DD.md] に保存
```

このファイルを作るまで、毎回「前回と同じフォーマットで」と伝え直していた。作ってからは、指示が不要になった。

### 具体例2: 週次レビュー

週次レビューも同じ構造だ。処理の順番だけ決めてある。

1. 全案件タスクノートを読む
2. 完了・未完了・持ち越しを集計
3. 催促すべき案件を特定
4. レビューファイルを自動生成

「週次レビューして」の一言で、全案件の状況が整理されたMarkdownが出る。手動でタスクを拾い集める作業が完全になくなった。

### 暗黙知の言語化プロセス

Skillsファイルを書くには、まず自分の判断基準を言語化しなければならない。

議事録なら「決定事項と次回アクションの区別をどうしているか」。報告書なら「1枚目に何を置くか」。提案資料なら「Why now、Why youをどこで出すか」。

これは最初、かなりしんどい作業だった。

自分が無意識にやっていることを、ルールとして書き下すのは難しい。最初に書いたSkillsファイルは、実際に使ってみるとAIが意図しない動作をする場面が多かった。「ここが穴だったか」と気づいて、毎回追記する。

今は、Skillsファイルを「育てている」感覚になっている。使うたびに精度が上がる。自分が仕事を離れても、ノウハウはファイルに残る。

### 失敗パターン

最初の失敗は、Skillsファイルを「理想の状態」で書いてしまったことだ。

「議事録は必ずこの5項目を入れること」「決定事項は3文以内にまとめること」——こういうルールをたくさん書いた。実際に使うと、クライアントによってフォーマットが全然違う。汎用ルールが足を引っ張った。

今は、汎用ルールはCLAUDE.md（Vault全体）に置き、案件固有のルールだけをSkillsファイルに書く。この分離をやってから、誤動作がほぼなくなった。

## CLAUDE.mdを800行から74行に削った話

CLAUDE.mdが800行になったのは、「AIに全部教えよう」という間違った思想のせいだ。

AIが知らないことが怖くて、どんどん書き足した。背景、注意事項、禁止事項、例外ルール。次第に「一番大事なこと」が埋もれた。

ある時、Claude CodeがCLAUDE.mdの中盤に書いてあったルールを完全に無視した。ログを見ると、冒頭の長い説明を読んでいる段階で指示が迷子になっていた。

**AIは人間と同じで、長い指示は後半を忘れる。**

そこで全面的に書き直した。原則は2つ。

1. **「何をするか」ではなく「どう判断するか」を書く**
2. **詳細はSkillsファイルに移す。CLAUDE.mdには参照先だけ書く**

### 74行になったCLAUDE.mdの実際の構成

削った後のCLAUDE.mdは、こういう構造になっている。

```
## 応答スタイル（4行）
## 結論先行・ファクトベース・効率重視のルール

## このVaultについて（5行）
## 案件名・クライアント背景・現在のフェーズ

## 作業分類の原則（8行）
## 考える作業 vs 作る作業の判断基準
## 「迷ったら過去成果物を参照」など

## よく使う参照先（10行）
## タスクノートのパス一覧
## 各案件のCLAUDE.mdへのリンク
## Skillsファイルの場所（.claude/skills/）

## 禁止事項（5行）
## Front Matter削除禁止・タグ上限・バックアップなし大規模変更禁止

## タスク管理コマンド（20行）
## 「タスク確認して」→ 何をするかの対応表
## 「完了した」→ 該当チェックボックスに入れてセクション移動

## Skills参照先一覧（12行）
## minutes.md（議事録）→ いつ参照するかの説明
## weekly_review.md（週次レビュー）
## briefing.md（ブリーフィング）
## note_publish.md（NOTE発信）

## 最終確認（5行）
## git pushと削除は確認を求める
## 大規模変更前はcheckpoint commit
```

合計74行。セクション数は8つ。各セクションが単一の目的を持つ。

800行版との最大の違いは、「どこに何があるか一目でわかる」ことだ。AIが指示を探す必要がなくなった。

800行を74行にした後、AIの精度が上がった。 毎回の追加指示が減り、やり直しが半分以下になった。

これは**情報量を増やすほどAIが賢くなるという思い込み**への反証だった。AIに渡す情報は、多いほどいいわけではない。適切な構造で渡す方が効く。

## エージェントを分けたら、さらに変わった

Skillsで「作る作業」を1つずつAIに移管していくと、次の壁が見えてくる。

1つのAIに全部やらせると、品質が落ちる。 「リサーチもして、分析もして、レポートも書いて」と頼むと、全部が中途半端になる。

人間のコンサルチームと同じだ。 リサーチ担当・分析担当・ライター担当が分業するから、専門性が出る。

今の僕の環境では、11のエージェントが役割分担している。問いの設計、情報収集、数値分析、洞察抽出、執筆、品質検査——それぞれが専門の仕事をこなし、1つのレポートに統合される。

この仕組みの結果、リサーチ系レポートの作成が3時間→45分に変わった。

ただ、この記事の主題はエージェント設計ではない。 大事なのは「仕事を分ける」という設計判断の方だ。エージェントの詳細は別の記事で書く。

## 実際にやってみて分かった3つの失敗

### 失敗1: 最初から完璧な構成を作ろうとした

フォルダ設計、CLAUDE.mdのフォーマット、Skillsファイルの体系——全部先に考えようとした。

結果、2週間かけて設計した構成が、実際の業務では全然使えなかった。

今は「最初はシンプルに、使いながら育てる」を原則にしている。CLAUDE.mdは最初50行で十分だ。Skillsファイルは、必要になってから作る。フォルダ構成は、案件が3つ来てから統一する。

### 失敗2: Gitのコミット粒度が大雑把すぎた

「作業完了」でまとめてコミットしていた。後から見ると、何をどう変えたか分からない。

AIが生成した内容と、自分が修正した内容が混在するので、Gitの差分が読めない状態になった。

今は「AIが生成したコミット」「自分がレビューして修正したコミット」を必ず分ける。AIの出力がどこまで信頼できるかを、Gitで追跡できるようにした。

### 失敗3: 「作る作業」に分類すべきものを手でやり続けた

分離の判断基準が曖昧なまま進めると、「これは自分がやった方が早い」という判断が増えてくる。特に慣れている作業ほど、自分でやる方が速く感じる。

でも、自分でやっている限り、Skillsファイルは育たない。

今は「この作業は今後も発生するか」を判断基準にしている。月に1回以上発生する作業は、必ずSkillsファイルを書いてAIに移管する。初回は遅くなるが、2回目からは自分の時間がゼロになる。

## まとめ

* コンサル業務は「考える作業（判断）」と「作る作業（整形）」に分けられる
* 混在させると、AIの修正が増え、蓄積もなく、時間が変わらない
* Claude Code × Git で案件情報を1箇所に集約すると、セッションをまたいで文脈が引き継がれる
* CLAUDE.mdは長ければいいわけではない。800行→74行にして精度が上がった
* Skillsファイルは暗黙知の言語化強制装置。使うたびに育つ
* エージェントを11個に分けてから、レポート作成が3時間→45分になった
* 削減工数は月30〜40時間。時給換算で月15〜20万円分の価値

## 次のアクション

今日の宿題は1つです。

自分の仕事で毎月2回以上発生する「作る作業」を3つ書き出してください。 そのうち1つだけ、期限・指標・制約・出力形式の4つに分けて言語化してみてください。

それが、最初のSkillsファイルになります。

## よくある質問

**Q. Claude Codeを使うには、プログラミングの知識が必要ですか？**

不要です。CLAUDE.mdもSkillsファイルもMarkdown（テキスト形式）で書きます。ターミナルを開いてclaudeと打てば起動します。僕自身、プログラマーではなくコンサルタントです。

**Q. ChatGPTではなくClaude Codeを使う理由は何ですか？**

最大の違いはファイルへの直接アクセスです。ChatGPTは会話の外に出られませんが、Claude Codeはローカルのファイルを直接読み書きし、結果を保存します。Gitと合わせれば「AIが何をしたか」の履歴も残る。業務で継続的に使うなら、この差が大きいです。

**Q. CLAUDE.mdは何を書けばいいですか？**

「判断基準」を書くのが正解です。①案件の背景、②フォーマットのルール、③迷ったときの優先基準、④Skillsファイルの場所。この4つから始めてください。最初は50行以内に収めること。長すぎると後半が読まれなくなります。
