# 議事録AIでは、会議は前に進まない。前回→今回→次回をつなぐ Decision Loop を作った

> **この記事の要点**  
> 定例会議を回しているチームのための、AIエージェント **Decision Loop** を作りました。会議で生まれた発言を「決定・未決・タスク・曖昧さ」に切り分け、その状態を次回会議まで運ぶことで、「前回どこまで話しましたっけ？」から始まる空回りをなくします。解決するのは、議事録の不足ではなく、発言が業務として動ける形に変換されていないこと。AIが候補と根拠と曖昧さを集めて提示し、人間が確定する。解析は Azure OpenAI（GPT-4o）を用いた非同期エージェント処理として動かしています。実際に動く様子は、5章のデモ動画でご覧いただけます。

---

## 1. はじめに：議事録はあるのに、会議は前に進まない

「あれ、前回これ誰がやることになっていましたっけ？」

定例会議の冒頭でこの言葉が出たとき、たいてい議事録は存在しています。Slackにも要約が流れています。NotionやBacklogにも会議ログがあります。

それでも、会議は前に進んでいないことがあります。

決まったはずのことが実行されていない。担当者と期限を毎回確認し直している。保留にした論点が、次の定例でまたゼロから話されている。

この問題は、記録が足りないことではありません。

会議で出た発言が、業務として動ける状態に変換されていないことが問題です。

今回参加した Microsoft Agent Hackathon 2026 のテーマは、「業務改革につながる Agentic AI」です。そこで私たちは、「どこにAIを入れるか」ではなく、「会議後の業務はどこで止まっているのか」から考えました。

作ったのが **Decision Loop** です。

Decision Loop は、議事録を要約するAIではありません。会議で生まれた決定・未決事項・タスク・曖昧さを整理し、次回会議まで持ち越すためのAIエージェントです。

この記事では、Decision Loop の動きを、桃太郎の鬼ヶ島遠征に向けた作戦会議を例に説明していきます。桃太郎・イヌ・サル・キジが何のために集まっているかは、誰もが知っています。背景を説明する必要がないぶん、「その会議で出た発言を、Decision Loop がどう次に動ける形へ変換するのか」だけに集中できます。実在の業務会議と同じように、彼らの会議にも「決まったこと」「まだ決まらないこと」「やるべきこと」「曖昧なまま残ったこと」が入り混じっています。

## 2. 本当のボトルネックは「変換」にある

会議の発言は記録されます。

しかし、記録されているだけでは業務は動きません。

たとえば、作戦会議で桃太郎がこう言ったとします。

> 船については、村の漁師に頼めば借りられると思っています。

この発言は、議事録には残ります。

しかし、タスクとして動かすには、もう少し情報が必要です。実際、会議でもキジがすぐに突っ込みます。

> 「借りられると思う」だけだと少し不安です。船がなければ、そもそも鬼ヶ島には渡れません。

整理すると、この一言には次の曖昧さが詰まっています。

* 船の確保は誰の担当なのか
* いつまでに確認するのか
* 「借りられると思う」は確証なのか、見込みなのか
* 何が分かれば「確保できた」と言えるのか
* これは正式に引き受けたタスクなのか、その場の思いつきなのか

ここが曖昧なまま、「桃太郎が船を手配する」とだけ書かれていても、タスクとして動く保証はありません。

もうひとつ、よくあるのが論点が何度も蒸し返されることです。

この作戦会議でも、出発時刻はいったん「明日の朝」で決まりかけたあと、会議の後半でサルが「早朝すぎると漁師に船を借りる段取りが難しい」と差し戻し、改めて議論し直しています。その場では時間をかけて再合意できますが、もし結論が記録されないまま次回に持ち越されれば、次の会議でまた同じ議論が最初から始まります。

会議後の業務が止まる理由は、議事録がないからではありません。発言が、次に動ける形に変換されていないからです。

NotionもBacklogもSlackも便利です。しかし、会議中の発言を「決定事項」「未決事項」「タスク」「曖昧な確認事項」に分ける作業は、まだ人間に任されています。

Decision Loop は、この変換を支援します。AIが会議ログを読み、候補と根拠を出します。人間がそれを確認し、確定したものだけを次回へ運びます。

## 3. Decision Loop：会議の「状態」を次回へ運ぶ

Decision Loop の中心にある考え方は、会議の状態を持ち越すことです。

ここでいう状態とは、次のようなものです。

* 何が決まったのか
* 何がまだ決まっていないのか
* 誰が何をやることになったのか
* どの期限や担当者が曖昧なのか
* 次回会議で何を確認すべきなのか

議事録は、過去の会話を残します。

Decision Loop は、次に進むための状態を残します。

この違いを一番分かりやすく表しているのが、「意思決定の文脈グラフ」です。

![](https://static.zenn.studio/user-upload/76548a23937c-20260601.png)

この画面では、前回の会議、今回の会議、次回の会議が横につながっています。

そこから、決定事項、タスク、未決事項、次回議題が枝分かれしています。ノード同士は、「発生」「派生タスク」「依存」「解決先」「次回へ持ち越し」といった関係で結ばれます。

たとえば、桃太郎の作戦会議は、次のように読めます。

「出発日が未確定だった」という未決事項がありました。今回の会議で「出発は今週中を目標とし、第一候補は明日の朝七つ時。ただし船・天候・風の条件を満たすこと」という、条件つきの決定に変わりました。その決定から「桃太郎が今日中に漁師へ船を確認する」というタスクが生まれました。しかし、「明日の天候や風が条件を満たすか」という最終判断は、まだ曖昧なまま次回へ持ち越されています。

このグラフを見ると、単に「何が話されたか」ではなく、「何が決まり、何が残り、何が次に進むのか」が分かります。

### この動きは、実際のデータでも観測できる

上の例は説明のためのものですが、「未決が会議をまたいで運ばれ、解決に向かう」という動きは、概念だけの話ではありません。複数回の定例をつないで解析したところ、未決事項の件数は次のように動きました。

ある回では、いくつかの未決事項が立ちました。次の回では、それらが引き継がれて再び検討され、その一部についてシステムが「すでに決着がついた可能性が高い」と判断し、解決候補として提示しました。残った未決事項は次回への引き継ぎとして積まれます。そしてさらに次の回では、未決事項の件数が収束していきました。

ここで強調したいのは、この「解決された候補」を、人間が後から会議ログを突き合わせて見つけたのではない、という点です。Decision Loop が前回の未決事項を引き継ぎ、今回の会議でその論点に決着がついていないかを自分で照合し、解決候補として差し出しました。未決事項が立ち、次の会議へ運ばれ、検討され、解決へ向かう──文脈グラフが描いているのは、まさにこの会議をまたいだ状態の移り変わりです。

普通の議事録AIは、1回の会議を要約します。

Decision Loop は、会議と会議の間に残る状態を扱います。

## 4. ワークフロー：入力 → 抽出 → 確認 → 確定 → 次回へ

Decision Loop の流れは、次の4ステップです。

1. Extract：会議ログから候補を抽出する
2. Review：人間が根拠を見て確認する
3. Commit：確定した内容を業務状態として保存する
4. Carry Forward：次回会議の入力として持ち越す

英語で書くと少し硬く見えますが、やっていることはシンプルです。

会議ログを入れる。AIが候補を出す。人間が確認する。確定したものだけが次回へ運ばれる。

この流れを、デモで見ていきます。

## 5. デモ：会議ログを業務状態に変換する

<https://youtu.be/hhk_OXZ1CMc>

ここでは、桃太郎の鬼ヶ島遠征に向けた作戦会議の文字起こしを入力として使います。

### 5.1 会議ログを入力する

まず、会議詳細画面に文字起こし済みのテキストを貼り付けます。

今回入力したのは、次のような会議の文字起こしです。

> [桃太郎] 今日は、鬼ヶ島へ向かう作戦について話し合いたいと思います。決めたいことは大きく3つです。まず、いつ鬼ヶ島へ行くのか。次に、どうやって鬼ヶ島へ渡るのか。そして、島に着いた後に誰が何を担当するのかです。  
> [イヌ] 目的は、鬼退治と、村から奪われた宝を取り返すことですよね。そこは全員一致していると思います。  
> [サル] はい。ただ、鬼ヶ島に行くこと自体は決まっていても、準備不足で向かうのは危ないと思います。鬼が何匹いるのか、島のどこに上陸できるのか、まだ分かっていません。  
> （以下、出発日・移動方法・役割分担・上陸後の方針などの議論が続く）

「AI解析を実行」を押すと、解析ジョブが作成されます。解析はバックグラウンドで実行され、画面ではステータスが `queued → analyzing → completed` と変わります。

![](https://static.zenn.studio/user-upload/7060b91590fa-20260601.png)

長い会議ログの解析は、数秒で終わるとは限りません。そのため、同期的に待つのではなく、ジョブとしてキューに積み、完了状態を画面から確認できるようにしています。

### 5.2 AIが候補と根拠を出す

解析が完了すると、AIが構造化した候補が表示されます。

単なる要約ではなく、候補・根拠・曖昧さが分かれていることです。

たとえば、次のように出ます。

**決定事項候補**

* 鬼ヶ島への移動手段は船を使用する
  + 根拠：「では、船で行くこと自体は決定でよさそうですね。」
  + 判断：移動手段について議論し、船で行くことに全員が合意
* 出発は今週中を目標とし、第一候補は明日の朝七つ時。ただし、船が借りられること、海が荒れていないこと、キジが飛べる程度の風であることが条件
  + 根拠：「出発は今週中を目標とする。第一候補は明日の朝。ただし、船・天候・風の条件を満たすことを前提にする。」
  + 判断：当初「必ず出発する」という案だったが、議論の末「目標とする」という条件つきの表現に修正されたうえで合意
* 役割分担：桃太郎が全体指揮、イヌが周囲警戒、サルが地上偵察、キジが上空確認
  + 根拠：「役割分担としては、桃太郎が全体指揮、イヌが周囲警戒、サルが地上偵察、キジが上空確認。ここは決定でよいですか？」
  + 判断：全員が賛成し、決定

**タスク候補**

* 漁師へ船の確認をする（担当：桃太郎／期限：2026-06-01）
  + 根拠：「では、TODOとして、私が今日中に漁師へ船の確認をします。」
* 明日の朝の風と天候を確認する（担当：キジ／期限：2026-06-02）

**未決事項**

* 船が本当に借りられるか
  + 根拠：「『借りられると思う』だけだと少し不安です。船がなければ、そもそも鬼ヶ島には渡れません。」
  + 状態：船の確保が未確認
* 明日の天候がどうか
* 鬼の人数や宝の場所
  + 根拠：「鬼が何匹いるのか、島のどこに上陸できるのか、まだ分かっていません。」
  + 状態：未確認

**曖昧箇所**

* タスクの期限が「明日の朝」「上陸後」などの相対的な表現のままで、具体的な絶対日付・タイミングが確定していない
* 鬼ヶ島への出発条件が天候や船の状況に依存しているが、これらの情報がまだ確認できていない
* 持ち物の詳細な分担が未決定で、誰が何を持つかの範囲が不明確

ここで重要なのは、AIが勝手に確定しないことです。AIは候補を出し、根拠を出し、曖昧な部分を出します。しかし、「これは正式なタスクです」「期限はこの日です」と一方的には決めません。確定は、次のレビュー画面で人間が行います。

### 5.3 人間が根拠を見て確定する

次に、レビュー画面で候補を確認します。各候補に対して、承認・修正・却下・保留を選べます。

![](https://static.zenn.studio/user-upload/eef07f2ec48d-20260601.png)

ここで、Decision Loop の設計思想がよく分かる2つの場面があります。

ひとつは、AIの出力が常に正しいとは限らないことです。たとえば「上陸地点の安全確認」というタスクは、会議では上空から確認するキジの担当のはずですが、AIが担当を別のメンバーとして出してくることがあります。人間はレビュー画面で根拠発言（「まず沖合で止まり、私が上空から見張りの位置、上陸できそうな浜、鬼の建物の場所を確認します」）を確認し、担当をキジに直してから承認します。AIは候補を出しますが、その割り当てが正しいかを確かめるのは人間です。

もうひとつは、曖昧箇所の扱いです。「タスクの期限が相対的な表現のまま」という曖昧箇所に対して、レビュー画面では「タスクにする」「未決事項にする」「破棄」を選べます。たとえば「明日の朝の風と天候を確認する」の期限を、その会議の日付に合わせて絶対日付（2026-06-02）に直したうえでタスクとして確定する。一方、「持ち物の詳細な分担」のように、その場で詰めきれていないものは、すぐに確定せず未決事項として次回へ残せます。

AIにすべてを決めさせると、便利に見えます。しかし、会議の発言には文脈があります。「確認します」は正式な約束かもしれませんし、単なる前向きな返事かもしれません。「明日の朝」は人によって解釈がずれることもあります。

だから Decision Loop では、AIは候補と根拠を出し、人間が確定します。AIは判断を置き換えるのではなく、人間が判断すべき場所を見つけやすくします。

### 5.4 確定した状態から、次回アジェンダを作る

レビューが終わると、確定した内容をもとに次回会議のアジェンダ候補が生成されます。

![](https://static.zenn.studio/user-upload/cc89642b02fc-20260601.png)

実際に生成されたアジェンダは、次のようなものでした。

1. 船の確保状況の確認（約10分）— 出発日程の仮決定条件に影響するため、早急に確認が必要
2. 明日の天候確認（約10分）— 出発日程の最終決定に必要な情報であるため
3. 持ち物の詳細な分担（約10分）— 準備不足を防ぐため、具体的な分担を明確にする必要がある
4. 鬼の人数と宝の場所の確認方法の検討（約10分）— 上陸後の作戦立案に必要な情報であるため
5. 出発日程の仮決定の再確認（約5分）— 船の確保状況や天候確認を踏まえ、仮決定を再確認する必要がある

注目してほしいのは、各議題に所要時間の目安と「なぜこれを話すべきか」の理由がついていることです。これにより、次回会議は「前回どこまで話しましたっけ？」から始まりません。前回の決定、残った未決事項、進行中のタスクが、次回会議の入力になります。

## 6. なぜ一発要約にしなかったのか

最初は、会議ログをそのままLLMに渡して、「決定事項、タスク、未決事項を抽出してください」と聞く形も考えました。

しかし、この方法には問題があります。1つのプロンプトに多くの目的を詰め込むと、AIは曖昧さを補完しようとします。「担当者が曖昧」と出してほしい場面で「おそらく桃太郎」と補ってしまう。「期限が不明」と出してほしい場面で、会話の雰囲気から期限を推測してしまう。

これは、会議後の業務では危険です。間違った担当者や期限が確定したように見えると、かえって混乱します。

そこで Decision Loop は、会議ログを一度に要約するのではなく、**会議を次に進めるという目的に向かって、エージェントが解析を自分で段階に分け、必要な文脈を集めながら進める**設計にしました。

### 解析を段階に分けて進める

Decision Loop は、次の判断を一つずつ順に行います。

キーワードを抽出する。前回からの未決事項を参照する。決定事項を抽出する。タスクを抽出する。曖昧箇所を抽出する。相対的な期限（「明日の朝」など）を扱う。そして次回アジェンダを生成する。

これは思想として段階に分けただけではなく、実装としても複数の呼び出しに分かれて動いています。1回の解析は、キーワード抽出・決定事項と未決事項の抽出・タスク抽出・曖昧箇所の抽出・日付変換・サマリーとアジェンダ生成といった段階に分割され、それぞれが別の処理として実行されます。

ポイントは、各段階が前の段階の結果を踏まえて進むことです。たとえば「前回からの未決事項を参照する」段階の結果が、今回の決定事項抽出やタスク抽出の判断材料になります。会議ログを一息に要約するのではなく、目的に沿って判断を分解し、段階ごとに精度を担保しています。

### 次回計画に必要な文脈を、自分で取りに行く

次回会議の計画には、その会議ログ単体だけでは足りない情報があります。そこで Decision Loop は、必要な情報をツールとして自分で取得します。

前回の作戦会議を探す。前回からの変化をまとめる。次回会議の所要時間を見積もる。

5.4で見た次回アジェンダに、議題ごとの所要時間（約10分、約5分）がついていたのは、この「所要時間を見積もる」動きの結果です。これらは「ユーザーに言われたから実行する」のではなく、次回アジェンダを組み立てるために、エージェントが必要だと判断して取りに行く動きです。会議ログを読むだけのAIと違い、前後の会議をまたいで文脈を集めにいくところに、エージェントとしての自律性があります。

実際、この自律的な動きは解析結果に記録されています。ある会議の解析では、エージェントは次の3つのツールを自分で呼び出していました（解析結果 `report_json.agent` より）。

```
"tool_calls": [
  "meeting_context-estimate_next_meeting_duration",   // 次回会議の所要時間を見積もる
  "meeting_context-search_related_past_meetings",      // 関連する過去会議を探す
  "meeting_context-summarize_previous_changes"         // 前回会議からの変化をまとめる
]
```

ここで重要なのは、これらが固定の手続きではなく、エージェントへの「次回アジェンダに必要な文脈を集めてほしい」という指示に対して、エージェント自身が関数呼び出しを選んで実行した結果だという点です。所要時間の見積もりも前回からの変化の要約も、人間が呼び出したのではなく、エージェントが必要だと判断して取りにいきました。なお、関連する過去会議の検索は現状ツールとして接続済みで、検索基盤（Azure AI Search）との本連携は今後の拡張です。

Decision Loop の Agentic AI らしさは、会議ログを一発で要約することではありません。会議を次に進めるために、判断を自分で分解し、足りない文脈を自分で集め、候補と根拠と曖昧さを分けて提示することです。

## 7. なぜAIに自動確定させなかったのか

Decision Loop は、AIが候補を出します。しかし、確定は人間が行います。ここは意図的にそうしています。

会議で生まれるタスクや決定事項は、きれいなデータではありません。言いかけの発言もあります。冗談のような返事もあります。明確な合意に見えて、実は仮置きの方向性にすぎないこともあります。

桃太郎の作戦会議でも、これは起きていました。桃太郎は一度「今週中に必ず出発する」と言い切りましたが、イヌとサルが「『必ず』と言い切って大丈夫か」と押し戻し、最終的に「今週中の出発を目標とする」という表現に落ち着きました。もしAIが最初の「必ず出発する」をそのまま決定事項として確定していたら、会議の本当の結論とずれてしまいます。決定に見えて、実は条件つきの目標だった、という機微は、文脈を踏まえなければ判断できません。

AIがそれらを自動で確定してしまうと、業務の責任も曖昧になります。誰がそのタスクを引き受けたのか。その期限で本当に合意したのか。その決定はチームとして確定したのか。これらは、人間が確認すべきことです。5.3で見たように、AIが担当者を取り違えて出すこともあります。だからこそ、根拠発言を見て最終的に確定するのは人間であるべきです。

ここで誤解されやすいのですが、人間が確定するという設計は、AIの自律性を弱める方向ではありません。むしろ逆です。

6章で見たとおり、Decision Loop のエージェントは、判断を自分で分解し、過去会議をまたいで文脈を集め、候補と根拠と曖昧さを切り分けるところまで自律的に動きます。3章で触れた「解決候補の提示」も、エージェントが前回の未決を引き継いで自分で照合した結果です。その自律的な働きを、すべて「人間が確定する一点」に収束させているだけです。エージェントが目一杯走ったうえで、最後の責任ある判断だけを人間に手渡す。自律性を捨てているのではなく、自律性の向かう先を「人間が判断すべき場所を明確にすること」に絞っているのです。

AIは、候補を出します。AIは、根拠を出します。AIは、曖昧さを出します。人間は、それを見て確定します。

この境界を引くことで、AIの便利さと、業務上の責任を両立させています。

## 8. Azure上でどう動かしているか

Decision Loop は、フロントエンド、APIサーバー、AIサービスを分離し、Azure App Service（Web App for Containers） 上でそれぞれ独立したコンテナとして動かしています。

![](https://static.zenn.studio/user-upload/2d1c700ebed4-20260601.png)

* Frontend（web）：議事録入力、レビュー、タスク管理、次回アジェンダ、意思決定の文脈グラフを表示します。
* API Server（api）：会議・決定事項・タスク・未決事項・次回議題・解析ジョブを管理します。データは Azure Database for PostgreSQL に保存します。
* AI Service（ai）：会議ログを解析し、候補・根拠・曖昧箇所・次回アジェンダを生成します。LLM には Azure OpenAI（GPT-4o）を使っています。

### 解析を非同期ジョブにしたのは、運用を見据えた判断

会議ログの解析は、すぐに終わる処理ではありません。長い会議ログでは、LLM呼び出しや後処理に時間がかかります。そのため、解析は **Azure Service Bus** を挟んだ非同期ジョブとして扱います。

ユーザーが「AI解析を実行」を押すと、APIサーバーが解析ジョブを作成し、Service Bus のキューに積みます。AIサービスがそれを受け取って処理し、完了すると結果を保存します。画面側は `queued → analyzing → completed` というステータスを確認します。

この構成は、見た目の派手さのためではなく、実運用で壊れにくくするための判断です。HTTPリクエストの中で長い解析を待つと、タイムアウトや再実行の問題が起きやすくなります。キューを挟むことで重い処理をアプリ本体から切り離し、ジョブが失敗しても状態が残るため、どこで止まったかを後から確認でき、再実行の判断もしやすくなります。さらに、解析エージェントはツール実行に失敗しても確定計算にフォールバックして解析を止めない設計にしています。

### 認証とシークレット

本番のユーザー認証は **Microsoft Entra External ID（OIDC）** を使い、ローカル開発では同等の OIDC プロバイダ（FakeAuth）に切り替えられるようにしています。Azure OpenAI のキーや Service Bus の接続文字列などの機密値は **Azure Key Vault** に置き、App Service からは Key Vault 参照で注入します。AIサービスとAPIサーバー間の内部通信は共有シークレットで保護しています。

### デプロイの自動化

デプロイは GitHub Actions で自動化しています。Azure へは OIDC フェデレーションでログインし、web/api/ai 各サービスのコンテナイメージを **Azure Container Registry（ACR）** にビルド＆プッシュ → **PostgreSQL への `prisma migrate deploy`** → **App Service を ACR の新しいイメージへ更新** → **ヘルスチェック** までを一連のパイプラインで実行します。コンテナイメージは ACR に集約し、App Service はそこから取得して起動します。手作業のデプロイに比べ、誰がいつ何を出したかが追え、壊れたときの切り分けもしやすくなります。

### 多段に分けても、1会議あたり約10円

6章で述べたとおり、Decision Loop は解析を複数の段階に分けています。段階を増やすとLLMの呼び出し回数も増えるため、コストが気になるところです。

同規模の会議ログを実際に解析して計測したところ、1会議あたりのAPI費用は約10円でした（GPT-4o、実測ベース）。解析を6つの段階に分けても、この水準に収まっています。一発要約に比べて呼び出しは増えますが、間違った担当者や期限が確定したように見えるリスクを避けられること、そして段階ごとに精度を担保できることを考えれば、十分に見合うコストだと判断しています。

過去会議の文脈参照については、現時点では会議状態の構造化、次回への引き継ぎ、文脈グラフによる可視化を主軸にしており、検索基盤との連携は今後の拡張余地として位置づけています。

## 9. できたこと、あえてやらなかったこと

今回のハッカソンで私たちが決めたのは、会議の最初から最後までを薄く全部自動化するのではなく、「会議後に業務が止まる一点」に資源を集中させることでした。

その一点とは、会議で生まれた発言を、次に動ける状態に変換する場所です。ここに集中した結果、Decision Loop は次のことを実現できました。会議ログを入力してAI解析を実行し、決定事項・タスク・未決事項・曖昧箇所を根拠つきで抽出する。人間が根拠を見ながらレビューして確定する。確定したタスクを管理し、前回・今回・次回をつなぐ意思決定の文脈グラフで状態を可視化する。そして、確定した状態から次回アジェンダ候補を生成する。これらの解析は非同期ジョブとして扱い、次回計画に必要な文脈はツールを使うエージェントとして自分で集めにいきます。

逆に、この一点に集中するために、意図的に外したものもあります。

一つは、音声の文字起こしです。会議音声から直接扱えると便利ですが、今回解きたかったのは文字起こしそのものではありません。すでにある会議ログを業務として動ける状態に変換することです。そのため、入力は文字起こし済みのテキストや手動でまとめた議事メモとしました。

もう一つは、Jira・Backlog・Notion・Slack といった外部ツール連携です。外部連携は重要ですが、連携先を増やす前に、まず「何を決定事項として扱うのか」「何をタスクとして確定するのか」「何を未決事項として次回へ持ち越すのか」を明確にする必要があります。Decision Loop の主戦場は、ツール連携の前段にあります。会議で生まれた発言を、次に動ける状態に変換することです。

## 10. 作ってみて分かったこと

作ってみて分かったのは、会議の問題は「情報量」ではなく「状態管理」だということです。

議事録は情報を増やします。要約は情報を短くします。しかし、それだけでは会議は前に進みません。

必要なのは、次の会議に渡せる状態です。何が決まったのか。何が未決なのか。誰が何をやるのか。何が曖昧なまま残っているのか。次回、何から話せばいいのか。これらが整理されて初めて、会議は次に進みます。

AIは、会議の代わりに意思決定をする必要はありません。むしろ、AIがやるべきなのは、人間が意思決定しやすい状態を作ることです。Decision Loop は、そのためのAIエージェントです。

議事録は、過去を残します。Decision Loop は、次に進むための状態を残します。

会議の成果物は、議事録ではありません。次の行動です。Decision Loop は、その行動が途切れないように、前回、今回、次回をつなぎます。
