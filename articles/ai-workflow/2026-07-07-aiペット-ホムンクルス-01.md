## title: "「負の感情でしか育たないペット」を作ったら、ステート管理が地獄だった話" emoji: "👾" type: "tech" topics: ["claude", "unity", "webgl", "aws", "個人開発"] published: true

## 「幸せになると、餌が消えて死ぬ」ペットの話

個人でブラウザゲームを1本作った。`HOMUNCULUS（ホムンクルス）` という、毒舌AIペット育成ゲームだ。

普通の育成ゲームは、エサをやって、褒めて、すくすく育てる。こいつは逆だ。**プレイヤーの「グチ」「弱音」「負の感情」を食べて育つ。** ポジティブな言葉では育たない。それどころか、飼い主が幸せになって負の感情を入力しなくなると、餌が尽きて餓死する。放置してもやっぱり餓死する。

完全にエンタメ（ジョーク作品）として作ったもので、何かが「治る」とか「効く」みたいな効能は一切ない。ただの、ちょっと性格の悪いデジタルペットだ。18歳以上向けに作ってある。

技術スタックはざっくりこんな感じ。

* フロント: Unity WebGL（ブラウザ、DL不要、完全無料）
* 応答生成: Anthropic の Claude API
* サーバ: AWS Lambda + DynamoDB
* ペット識別: 5桁のID

「AIに毒を吐かせるゲーム」という企画自体はノリで決まったが、いざ作り始めると**ゲームのコア体験を支えているのは演出ではなく地味なバックエンド設計**だった、という話を書く。技術9・宣伝1くらいの気持ちで。プロンプトの中身みたいな"種"は出さないので、そこは設計思想レベルで読んでほしい。

---

## 1. 「負の感情で育つ」を、どう"成立"させたか

最初に決めたのは「育成の燃料は負の感情だけ」というルールだった。問題は、これをどうやってゲームのステート遷移に落とすか。

やったことを抽象化すると、こうなる。

```
プレイヤーの入力テキスト
   ↓ 感情の傾きを判定（ネガ寄りか／そうでないか）
   ↓ ネガ寄りなら「栄養ポイント」に変換
   ↓ 累積ポイントが閾値を超えると成長段階が1つ進む
```

ポイントは「ネガティブ＝栄養」という変換を入れたこと。ポジティブな入力や雑談はポイントにならない（むしろ世界観的には"餌にならない無駄な発話"）。なので、ひたすら褒め続けてもこいつは1ミリも育たない。グチや弱音を吐かれて、初めて成長メーターが動く。

ここで効いてくるのが\*\*「閾値を超えた瞬間だけ段階が進む」という不可逆設計\*\*。一度進んだ段階は基本的に戻さない（戻るのは餓死ルートだけ）。これによって「育ってきた」という手応えが演出ではなくステートとして残る。

感情判定そのものの中身（どう分類しているか、どんな指示で動かしているか）はゲームの肝なので伏せる。ここで言いたいのは、**"負の感情で育つ"はキャラの演技ではなく、入力 → 分類 → ポイント → 段階遷移という一本のステートマシンとして設計しないと成立しない**ということ。演出だけでやろうとすると「褒めても育つじゃん」というツッコミで世界観が即死する。ルールはデータ側で守らせるしかなかった。

## 2. 放置で餓死する ── クライアントを1ミリも信用しない

このゲーム最大の地味な実装ポイントが、**餓死**だ。

一定時間、負の感情を与えられないとペットは弱り、やがて死ぬ。育成ゲームでありがちな「最後の餌やりからの経過時間」管理なのだが、ブラウザゲームでこれをやると地獄を見る。

最初に思いつくのは「最終接餌時刻をローカルに持って、起動時に経過時間を計算する」だ。これは**一瞬で破綻する**。理由は明白で、

* ブラウザのローカル時刻はユーザーが好きに巻き戻せる
* localStorage は消せるし書き換えられる
* 別タブ・別端末で開いたら状態が割れる
* 「死んだ直後に時計を戻して生き返らせる」が成立してしまう

「切ない世界観」を売りにしているのに、時計いじりで不老不死になれたら台無しだ。なので方針を真逆にした。

> **時間に関わる真実はすべてサーバ（DynamoDB）に置く。クライアントは"表示"しかしない。**

DynamoDB のレコードはこんなイメージ。

```
// pets テーブル（イメージ）
{
  "petId": "48213",          // 5桁ID
  "stage": 2,                // 成長段階
  "nutrition": 740,          // 累積栄養ポイント
  "lastFedAt": 1719700000,   // 最後に栄養を得たUNIX時刻（サーバ時刻）
  "isAlive": true
}
```

生死や残り時間の判定は、すべて Lambda 側がサーバの現在時刻と `lastFedAt` の差分で計算する。クライアントが送ってくる時刻は一切信じない。フロントは「サーバがこう言ってる」を絵にするだけの存在に格下げした。

これで「時計を戻して餓死を回避」は封じられた。代償として、状態が知りたいたびにサーバへ問い合わせることになるが、餓死というコア体験の整合性のためには妥当なトレードオフだったと思う。**ステートの真実を持つ場所を1箇所に集約する**、というだけの話なのだが、リアルタイム性のある育成ものだと効いてくる。

## 3. 最終形態20種の分岐 ── 確率を二層で考える

育ちきると、ペットは20種類ある最終形態のどれかになる。ここの設計は地味に悩んだ。

単純に「育成完了時に1/20で抽選」だと、何度遊んでも同じ確率なので、プレイヤーの育て方がまったく結果に反映されない。かといって完全に育て方で確定させると、今度はネタバレ攻略が出回って「正解の餌やり手順」をなぞるだけのゲームになる。引きの楽しさが死ぬ。

そこで採った"考え方"が**二層モデル**だ。

* **第1層：ベースの抽選確率**。20種それぞれにベースの出現確率を持たせる。
* **第2層：育成過程による重みの変動**。どんな負の感情をどれだけ与えてきたかという履歴に応じて、各形態の重みが上下する。

```
最終形態の出現確率 = ベース確率 × 育成履歴による重み係数
   → 全体を正規化して抽選
```

これだと「育て方によって"出やすい形態"は傾くが、確定はしない」という塩梅になる。攻略情報で「これをやれば必ずアレになる」とは言えず、でも「こういう育て方だとこっち系が出やすいらしい」という"噂"が生まれる余地が残る。個人開発のゲームにとって、この\*\*"噂が生まれる余白"\*\*はけっこう大事だと思っていて、確定テーブルにしなかったのはそこを狙った。

具体的な係数の振り方は調整中のまま出した（ここはずっと終わらない）。

## 4. 口調を成長段階で変える ── キャラ設計はパラメータの話

このゲーム、応答の生成に Claude API を使っている。ただ「AIが喋る」だけだと"ガワだけAI"になってしまうので、**成長段階に応じてキャラの口調・態度を変える**ことにした。

設計思想としては、段階ごとに「人格のパラメータ」を切り替えるイメージ。

* 第1段階（幼体）：語彙が少なく、たどたどしい。毒もまだ可愛げのある幼児的なもの。
* 第2〜3段階：知恵がつき、皮肉が増える。毒舌の解像度が上がってくる。
* 最終段階：完成された人格。成熟した、容赦のない物言い。

実装としては、現在の `stage` を生成時のコンテキストに渡して、応答のトーンを段階に紐づけている。同じ「グチを言われた」という入力でも、幼体なら無邪気に、成熟体なら冷ややかに返ってくる。**口調はキャラの飾りではなくステート（成長段階）の関数**として持たせた、というのがポイント。

具体的にどんな指示で口調を制御しているか（プロンプト設計）はゲームの核なので出さない。ここで共有したいのは「キャラの一貫性を出したいなら、口調を"その場のノリ"ではなく、ちゃんとステートに紐づくパラメータとして設計したほうが破綻しない」という一点だ。

## 5. Claude API をサーバ中継する ── キー隠蔽と安全設計

ブラウザゲームで外部のLLM APIを使うとき、絶対にやってはいけないのが**APIキーをクライアントに置くこと**。Unity WebGL のビルド成果物は実質ブラウザに全部落ちてくるので、フロントにキーを埋めたら抜かれて終わりだ。

なので構成はシンプルに「全部サーバ経由」にした。

```
[ブラウザ / Unity WebGL]
      │  入力テキストを送るだけ
      ▼
[API Gateway → AWS Lambda]   ← ここで Claude API を呼ぶ
      │  キーはサーバ環境変数。クライアントには出さない
      ▼
[Anthropic Claude API]
      ▼
[整形した応答だけをブラウザへ返す]
```

ポイントは3つ。

1. **キーはサーバ環境変数にだけ持つ**。クライアントは「テキストを投げて、整形済みの応答を受け取る」だけ。生のAPIレスポンスもそのままは返さない。
2. **入力はサーバで一度受け止める**。クライアントから来た文字列を素通しでLLMに流さず、サーバ側で検証・整形の層を一枚噛ませる。コストや悪用の制御点をサーバに集約する狙いもある。
3. **危機ワード時の安全動作**。このゲームは「負の感情を吐き出す」場なので、入力の中に深刻な内容（自傷を匂わせる言葉など）が混じる可能性がある。そういう危機的なワードを検知したときは、ペットの毒舌を返すのではなく、**運営からのまじめなポップアップ**（相談窓口の案内など）を出す動線をサーバ側に持たせた。

3つ目は技術というより設計倫理の話だが、「ネガティブな入力を受け止めるサービス」を作る以上、ここはエンタメの世界観より優先する、と最初に決めていた部分だ。ジョークゲームでも、入り口が「弱音を吐く」である以上、受け止め方の安全弁は要る。

なお、応答生成に Anthropic の Claude API を使っているのは事実だが、Anthropic と何らかの公式な提携をしているわけではない。単に一開発者として API を利用しているだけ、という点は念のため明記しておく。

## まとめ ── 「演出」だと思っていたものが、全部「設計」だった

作る前は「AIに毒を吐かせるネタゲーだし、見た目とキャラのノリが9割でしょ」と思っていた。実際に手を動かしてみると、コア体験を支えていたのは全部バックエンド側の地味な設計だった。

* 負の感情で育つ → 入力 → 分類 → ポイント → 段階遷移のステートマシン
* 放置で餓死 → 時刻の真実をサーバに集約、クライアントを信用しない
* 20種の最終形態 → 確率を二層で持って"噂の余白"を残す
* 段階で変わる口調 → 口調をステートの関数として設計
* AI応答 → キーをサーバに隠し、安全弁を一枚噛ませる

「切ない世界観」も「毒舌キャラ」も、結局はステートとサーバ設計をちゃんとやらないと一瞬で破綻する、という当たり前を改めて思い知った1本だった。同じように「LLM + ブラウザゲーム」をやろうとしている個人開発者の参考になればうれしい。

最後に。あくまでエンタメ作品で、何かに効くものではない。18歳以上向け。完全無料・ダウンロード不要、ブラウザですぐ遊べる。
