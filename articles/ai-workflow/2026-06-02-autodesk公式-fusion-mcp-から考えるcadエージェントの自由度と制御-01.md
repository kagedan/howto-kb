普段 AutoCAD 上で動作する AI エージェントを試作していて、LLM に CAD をどう操作させるかを考えています。LLM に CAD を操作させるための設計は、論文や既存プロダクトを見てもまだ決定打がなく、CAD エージェントにおける大きな設計課題だと感じています。

細かい CAD 操作を大量のツールとして用意するのか。  
それとも、LLM にもっと自由度の高い実行口を渡すのか。

そんな中で、[Autodesk 公式 Fusion MCP](https://blog.autodesk.io/fusion-mcp-server/) が出ているのを見て気になりました。

CAD ベンダー本体である Autodesk は、LLM に CAD を操作させる入口をどう設計しているのか。それを知るため、実際に Claude Desktop から接続して触ってみました。

## ツールは3つ

Claude Desktop で確認できた Fusion MCP のツールは下記の通りでした。

![](https://static.zenn.studio/user-upload/e6dd02486c49-20260518.png)  
*Claude Desktopでのツール確認画面*

| ツール | 主な役割 |
| --- | --- |
| `fusion_mcp_read` | API ドキュメント参照、スクリーンショット取得、ドキュメント参照 |
| `fusion_mcp_execute` | Python script 実行、ドキュメントの open / close / save |
| `fusion_mcp_update` | undo / redo |

この MCP の面白い点は、`create_box`、`create_sketch`、`extrude`、`fillet`、`set_appearance` のような CAD 操作用の専用ツールが大量に並んでいる構成ではないことです。

もちろん、Fusion MCP の内部実装がどうなっているかまでは分かりません。  
ただ、Claude Desktop 上で見えているツール仕様としては、CAD 操作を個別ツールとして大量に公開する構成ではなく、少数の汎用ツールを組み合わせる構成のようです。

もう少し具体的に確認すると、各ツールは次のような種類を指定して使います。

| ツール | 指定できる主な種類 |
| --- | --- |
| `fusion_mcp_read` | `apiDocumentation` / `screenshot` / `document` / `projects` |
| `fusion_mcp_execute` | `script` / `document` |
| `fusion_mcp_update` | `undo` / `redo` |

つまり Fusion MCP は、CAD 操作ごとの専用ツールを大量に公開するのではなく、API ドキュメント参照、script 実行、スクリーンショット確認、undo / redo を組み合わせる構成のようです。

## 実際の操作ループ

実際に試すと、Claude × Fusion MCP はだいたい下記のようなループで動くようです。

まず、LLM は `read.apiDocumentation` で Fusion API の仕様を確認します。  
たとえば、円柱を作るときに `CylinderFeature` や `CylinderFeatures` を調べたり、色を変えるために `Appearance`、`ColorProperty`、`Color` などを検索したりします。

そのうえで、生成した Python script を `execute.script` に渡して Fusion 上で実行します。

実行後は、`success / error` と `print()` の出力を確認します。  
ここで `print()` はかなり重要で、script 内で取得したボディ数、メッシュ数、appearance の ID、変更後のカウントなどを、LLM が読み取れる形で返すために使われます。

必要に応じて `read.screenshot` で見た目も確認します。  
撮影方向などを指定できるため、AI が実行結果を視覚的に確認できます。

失敗していれば `update.undo` で戻し、script を修正して再実行する流れです。

ここで重要なのは、`fusion_mcp_read` がモデル内の Body、Sketch、Feature、Parameter などを直接一覧する専用クエリを持っているわけではない、という点です。

モデル構造を調べたい場合も、基本的には `execute.script` で Python を実行し、その中で Fusion API を使って走査し、結果を `print()` で返す形になります。

たとえば、次のような観測 (Observe) 処理も script 側に寄ります。

* モデル内の要素を列挙する
* 条件に合う対象を抽出する
* 操作前後の状態差分を確認する
* タスク達成に必要な値や属性を読み取る

つまり Fusion MCP は、操作だけでなく、観測もかなり script 側に寄せる設計のようです。

## `execute.script` が操作の中心

Fusion MCP の中心にあるのは、`fusion_mcp_execute` の `script` です。

`script` では、`def run(_context: str):` のようなエントリポイントを持つ Python script を渡し、Fusion API を使ってアクティブな Fusion デザインを操作します。

たとえば、ビューポート調整なら次のような script になります。

```
import adsk.core

def run(_):
    adsk.core.Application.get().activeViewport.fit()
    print('fit')
```

これは単なるビューポート調整の例ですが、同じ仕組みでスケッチ作成、押し出し、フィレット、色や見た目の変更、メッシュ削減なども実行できます。

つまり Fusion MCP は、個別の CAD 操作をツールとして細かく分けるのではなく、Python script を実行する入口を中心にした構成のようです。MCP ツールとして用意されていない操作でも、Fusion API で実現できるなら、LLM がその場で Python script を書いて実行できます。

CAD のように操作種類が非常に多い領域では、これは合理的な設計だと思います。  
すべての CAD 操作を MCP ツールとして個別に定義するのは大変ですが、script 実行口があれば、少ないツール数でも広い操作範囲を扱えます。

## メリット：自由度が高い

この方式の一番大きなメリットは、自由度の高さです。

`execute.script` を使えば、Fusion API で書ける処理を広く実行できます。  
個別の操作ツールが用意されていなくても、LLM が Python script を書くことで、その場で必要な操作を組み立てられます。

CAD の操作は非常に種類が多く、形状作成、編集、マテリアル、色や見た目の変更、メッシュ、アセンブリ、ドキュメント操作などをすべて個別の MCP ツールとして定義しようとすると、大きなツール群になります。

その点、少数ツール + Python script 実行という構成では、MCP サーバ側に細かい CAD 操作ツールを大量に用意しなくても、広い操作範囲を扱えます。

もちろん、前提は **LLM が Fusion API を理解し、正しく Python script を書けること** です。  
その精度が十分なら、少数ツール + Python script 実行方式はかなり強い構成になります。

## デメリット：自由すぎる

一方で、この自由度の高さは、そのまま難しさにもなります。

特に重要なのは、**操作・観測・検証の多くが LLM 生成 script 側に寄ること**です。

Fusion MCP では、モデル構造を調べたい場合も、基本的には LLM が Python script を書き、その中で Fusion API を使って走査し、結果を `print()` で返す形になります。

つまり、操作だけでなく、観測や実行後の検証も script 側に寄ります。

たとえば、次のような判断を LLM がその都度行う必要があります。

* どの API を使うべきか
* 対象をどう列挙するか
* UI 上の単位と API 上の値が一致しているのか
* 操作後に何を再走査すれば成功と言えるのか
* screenshot で確認できることと、API で確認すべきことをどう分けるのか
* 失敗したときに undo すべきか、script を修正して続行すべきか

専用ツール方式であれば、こうした判断の一部をツール側に閉じ込められます。  
たとえば `set_appearance` という専用ツールを用意するなら、`scope`、`target_count`、`preview`、`verify` のような設計を、スキーマや実行器側に持たせることができます。

しかし script 実行方式では、それらの多くを LLM が都度 Python script として組み立てることになります。つまり、LLM が生成する script の品質に大きく依存するということです。

もちろん、破壊的な操作のリスクもあります。  
`deleteMe()` のような処理を書けば、モデル内の要素を削除することもできますし、対象の選び方を間違えると意図していないボディの色や属性まで変更してしまう可能性もあります。

ただ、CAD エージェント設計としてより本質的なのは、そこだけではありません。  
本当に難しいのは、正しく操作するための前提確認、対象選定、単位解釈、実行後検証のような処理まで、非決定論的な LLM の判断に寄ってしまう点です。

つまり、メリットは自由であること。  
そして、デメリットは自由すぎることです。

この違いを整理すると、次のようになります。

| 論点 | script 方式 | 専用ツール方式 |
| --- | --- | --- |
| 自由度 | 非常に高い | ツール定義の範囲に制限される |
| 対応範囲 | Fusion API で書ける処理を広く扱える | 用意したツールの範囲に依存する |
| ツール追加 | 必要な操作を script で組み立てられる | 操作ごとにツール追加が必要になりやすい |
| 観測 | LLM が走査処理 script を書く必要がある | 観測ツールとして標準化しやすい |
| 検証 | 毎回 script 内に検証を書く必要がある | 検証ツールとして標準化しやすい |
| 安全性 | 処理内容が script 品質に依存する | スキーマで制約しやすい |
| 再現性 | script 全文は残るが環境依存も残る | ツール実装側に知識を集約できる |
| 実務利用 | 追加の制御層が必要 | 制御しやすい |

## `success: true` はタスク成功ではない

ここからが、CAD エージェントとして重要な話です。

エージェントの処理は、基本的に 1 回でタスクを完遂する前提ではありません。  
実行結果を観測し、うまくいかなければ LLM が再思考して script を修正し、再実行する。  
この自己修復のループを前提にしています。

Fusion MCP の `execute.script` も、実行結果として `success: true` や `error` を返します。  
ただし、`success: true` で分かるのは、あくまで script の実行が正常終了したことです。これは、Python の `run()` 関数が例外なく最後まで実行された、という意味に近いです。

ユーザーが意図した CAD 操作が正しく完了したかどうかは、別途確認する必要があります。

```
【重要】 script 実行成功 ≠ CAD タスク達成
```

たとえば、script が例外なく終われば `success: true` は返ります。  
しかし、次のような失敗は普通に起こり得ます。

* 対象を間違えた
* 単位を間違えた
* 変更すべき部品の一部を取りこぼした
* 変更してはいけない部品まで変更した
* 見た目は合っているが、内部の material / appearance が意図と違う
* タイムラインに不要なフィーチャーを大量に残した

つまり、`success: true` は「Python がクラッシュしなかった」ことの確認であって、「CAD タスクが正しく終わった」ことの確認ではありません。

## screenshot だけでは足りない

スクリーンショットは便利で、見た目の確認には有効です。  
内容によっては、タスクが達成できているかを見た目から判断できる場合もあります。

![](https://static.zenn.studio/user-upload/d4214b896c89-20260518.png)  
*`read.screenshot` による見た目確認の例*

ただし、CAD エージェントの成功判定の主役にはできません。  
CAD では「それっぽく見える」と「設計データとして正しい」は別だからです。

たとえば、次のような問題は screenshot だけでは検出しにくいです。

* 色は合っているが、ベース appearance や material が違う
* 小さな部品の一部だけ変更漏れしている
* カメラに写っていない内部部品が変更されていない
* 参照関係やタイムラインが壊れている
* 単位スケールが誤っている

これは Fusion MCP が screenshot だけでタスク達成を判定しているという話ではありません。  
`read.screenshot` は、最終的な見た目を LLM に返すための補助的な確認手段として使われます。

つまり、screenshot は最後の見た目確認としては有効です。  
ただし、CAD データとして正しいかどうかは、API による再走査や対象スコープの検証で確認する必要があります。前述の通り、その確認も script 生成で行われます。

## 実務利用には Verify 層が必要

ここからは Fusion MCP に限らず、CAD エージェント全般の話です。

Fusion MCP を触っていて改めて感じたのは、CAD エージェント、特に実務向けでは単に「実行できたか」を見るだけでは足りないということです。

タスクが期待どおりに完了したのか。  
完了していないなら、どこまでできていて、どこで失敗しているのか。

実務利用では、ここまで確認できる Verify 層を用意しておきたくなります。

ただし、Verify 層を用意すること自体も簡単ではありません。CAD 操作のパターンはほぼ無限にあり、すべての操作に対して専用の検証ロジックを用意することは現実的ではないからです。

そこで、確認すべきことを次の 3 層に分けて考えます。

### 1. 個別オブジェクトの操作後チェック

対象となる各オブジェクトについて、操作後に満たすべき条件を確認します。  
たとえば、色変更なら次のような条件です。

```
body.appearance.id == expected_appearance.id
appearance の RGB が指示値と一致している
```

これは、コマンド単位の確認に近い層です。

### 2. スコープ全体の整合性

次に、「変更すべき対象がすべて変更されたか」「変更してはいけない対象を変えていないか」を確認します。つまり、操作対象の集合全体が正しい状態になっているかを見る層です。

たとえば、「黒い部品をすべて赤にして」という指示であれば、次のような確認が必要です。

* 変更対象として列挙したボディがすべて赤になったか
* 変更対象外のボディが赤になっていないか
* 列挙方法によって対象漏れが起きていないか

Fusion API では、同じモデルを調べているつもりでも、取得方法によって見える対象が変わることがあります。たとえば、モデル全体から直接たどった場合と、アセンブリ上の配置関係をたどった場合で、列挙される部品の範囲が変わることがあります。

そのため、1 つの列挙方法だけに依存すると、変更漏れや誤操作につながります。

### 3. システムレベルの不変条件

最後に、操作とは直接関係しないものの、壊れてはいけない条件を確認します。  
たとえば次のようなものです。

* アセンブリ全体の bounding box が異常に大きくなっていない
* 意図しない遠方に部品が配置されていない
* タイムラインに壊れたフィーチャーが増えていない
* 参照関係が壊れていない
* ソリッド数やボディ数の変化が想定範囲に入っている

この層は、毎回すべてを厳密に確認すると重くなるかもしれません。  
ただし、破壊的操作や一括変更の後には必要になる層だと思います。

## 自作 AutoCAD エージェントとの違い

自作している AutoCAD エージェントでは、LLM に直接 AutoCAD API のコードを書かせるのではなく、DSL (ドメイン固有言語) を生成させています。DSL といっても難しいものではなく、エージェント用に定義した「操作手順の JSON」のようなものです。

その DSL を C# 側で検証し、AutoCAD .NET API 経由で実行します。

このエージェントは、既存図面の調査・編集だけでなく、図形や簡単な形状の生成も行います。  
ただし、LLM が AutoCAD API を自由に呼び出すのではなく、いったん DSL という中間表現を通す点が、Fusion MCP の `execute.script` とは大きく違います。

script 実行方式では、LLM が Python script で必要な処理をその場で組み立てます。  
一方で、DSL 方式では、LLM が出力するものを操作手順の JSON に寄せています。

これは単に実装都合というより、CAD エージェントを安全に制御するための設計です。

### DSL を挟む理由

CAD × LLM の研究でも、LLM に直接 CAD の低レベル操作をすべて任せるのではなく、DSL のような中間表現を使う考え方は出ています。AIDL も CAD 向け DSL の一例です。

もちろん、自分が使っている DSL は AIDL そのものではありません。  
ただ、LLM に直接 API コードを書かせるのではなく、DSL を生成させ、それを実行器側で検証してから処理するという意味では、近い問題意識があります。

DSL を挟むと、Fusion MCP の `execute.script` のような自由度は落ちます。  
新しい操作を追加するには、DSL コマンドと C# 側の実行器を実装する必要があります。

一方で、次のような制御はしやすくなります。

* 存在しないコマンドを弾く
* 必須パラメータを検証する
* 対象 ID の存在を確認する
* 実行前にスキーマチェックする
* 操作ログを追いやすくする
* 危険な操作を実装しない、または確認必須にする
* 操作後の Verify を組み込みやすくする

構成の違いを単純化すると、次のようになります。

この違いを表にすると、次のようになります。

| 設計 | 強み | 弱み |
| --- | --- | --- |
| script実行 | 自由度が高い。 未知の操作にも対応しやすい | 操作・観測・検証が LLM 生成 script に寄りやすい |
| DSL方式 | 検証しやすい。ログを追いやすい。 決定論的に制御しやすい | DSL にない操作はできない。 拡張には実装が必要 |

どちらが正解という話ではありません。

script 実行方式には、少ないツール数でも広い操作範囲を扱える強さがあります。  
一方で、実務 CAD データでは、対象範囲、単位、検証、復旧をより強く制御したくなります。

そのため、AutoCAD 側のエージェントでは DSL を挟み、LLM が出した手続きを実行前に検証し、できるだけ決定論的に処理できる構成にしています。

ただし、DSL 方式だけですべてを閉じる必要もないと思っています。  
よく使う操作、危険な操作、検証が重要な操作は DSL と専用実行器で定義し、そこから外れる非定型な処理だけ script 実行に逃がす、というハイブリッド構成も考えられます。

もちろん、script に任せた部分には script 実行方式の難しさが残ります。  
対象選定、単位解釈、観測、検証、失敗時の復旧まで LLM 生成 script の品質に依存するため、どこまで script 実行を許可するかは慎重に設計する必要があります。

この違いは、CAD エージェントにおける自由度と制御のトレードオフそのものだと感じました。

## まとめ

Autodesk 公式 Fusion MCP は、細かい CAD 操作を大量の MCP ツールとして公開するのではなく、少数の汎用ツールを組み合わせる構成でした。

代わりに、API ドキュメントを読み、Fusion API を使った Python script を実行し、`print()` や screenshot で結果を確認し、必要なら undo / redo する。  
かなりシンプルで、自由度の高い構成です。

少数ツール + script 実行は、CAD のように操作種類が多い領域では合理的に見えます。  
一方で、実務 CAD データでは、対象範囲、単位、検証、復旧などの制御層も必要です。

その意味では、DSL や専用ツールで制御しやすい操作を固めつつ、そこから外れる処理だけ script 実行に逃がすようなハイブリッド構成も考えられます。

CAD エージェントでは、「何ができるか」だけでなく、「どこまで自由にさせるか」「どうやって正しくできたことを確認するか」が重要になります。

Fusion MCP は、そのバランスを考えるうえで非常に面白い題材でした。
