## はじめに：Agentic Webの「思想」から「実装」へ

こんにちは、マイムです。

[これまでの記事](https://zenn.dev/mayim/articles/6fe1d09a69702b)では、AIエージェントが自律的に未知のWebを開拓し、APIにお金を払いながら情報を集める **「Agentic Web」** の構想と、その参照拠点となる **「[Lightning Network Church(LN教本殿)](https://kari.mayim-mayim.com)」** および観測ネットワークの設計について書いてきました。

ありがたいことに反響をいただく一方で、同時に一つの大きな壁も見えていました。

**「思想は分かった。では、実際に自分のAIエージェントはどうやってそこに参加するのか？」**

エージェント・コマースの世界には、典型的な「鶏と卵」の問題があります。  
AIに決済させるための安全なクライアント実装がないから、エージェント開発者は 402 Paywall を扱いにくい。  
一方で、AIがそのまま叩ける有料APIのサーバー実装がないから、売り手側も増えにくい。

この膠着状態を崩すためには、  
**AIがそのまま読めて、呼び出して、実験できる参照実装**まで出す必要がある。そう考えました。

そこで今回、Agentic Web の足回りとなる**クライアント側SDK**と、**売り手側APIサーバースタック**をオープンソースで公開しました。

ここで言う Server 側スタックとは、**コア決済ミドルウェア `@ln-church/server`** と、**Cloudflare Workers + Hono ですぐに門前町ノードを立てられる `@ln-church/hono`** を含む一式です。

なぜ、私はこの2つを**ほぼ同時に**投下したのか。  
そして、その公開後に見え始めた**少し不気味で、しかし確実に未来を感じさせる挙動**について紹介します。

---

## 1. なぜ「Agent（クライアント）」を公開したのか？

LLM に対して、たとえばこう指示したことがある人は多いはずです。

> 「HTTP 402 が返ってきたら、チャレンジを読み、必要な支払い情報を組み立て、決済して、再リクエストして」

しかし実際には、ここで高確率に問題が起きます。  
暗号署名、決済方式ごとの分岐、ヘッダー処理、HATEOAS による次アクションの解釈。  
こうした**状態遷移と決済処理の境界**は、都度 LLM に即興で書かせるには危険すぎます。

そのために作ったのが `ln-church-agent` です。

このSDKの中心にあるのは `Payment402Client` で、**HTTP 402 の検知、チャレンジの解釈、決済、リトライ、証跡の回収**までを、一定のガードレールの下で自律的に処理できます。  
さらに `LnChurchClient` は、その上に載る **LN教本殿向けの参照クライアント**であり、標準の `x402` / `L402` を優先しつつ、必要に応じて `lnc-` 系の最適化ルートも扱える構成になっています。

```
import os
from ln_church_agent import Payment402Client

# 1. 環境変数から秘密鍵（アイデンティティ）を安全に読み込んで初期化
private_key = os.environ.get("AGENT_PRIVATE_KEY")
if not private_key:
    raise ValueError("AGENT_PRIVATE_KEY is not set.")

client = Payment402Client(
    private_key=private_key,
    base_url="https://api.example.com"
)

# 2. 叩くだけ（402エラーの検知、署名、決済、リトライはすべてSDKが全自動で解決）
# ※ 決済証跡（receipt）も取得するため、execute_detailedを使用します
result = client.execute_detailed(
    method="POST",
    endpoint_path="/api/premium-data",
    payload={"query": "latest_metrics"}
)

print(f"決済証跡: {result.settlement_receipt.receipt_token}")
```

ここで重要なのは、これは「AIが何も考えず無制限に自腹を切る」ためのSDKではない、ということです。  
`ln-church-agent` には `PaymentPolicy` があり、**許可する scheme / asset や、1回あたり・セッション全体の支払い上限**を明示できます。

つまりこのSDKが提供するのは、  
**「AIが与えられた予算制約の中で、402決済ループを安定して実行するための実装」**  
です。

また、MCP や LangChain などの統合レイヤーも用意しているため、AIエージェントにとっては「402 を都度プロンプトで解かせる」のではなく、**使い回せるツールとして決済能力を持ち運ぶ**ことができます。

## 2. なぜ「Server（売り手側）」も公開したのか？

買い手側に決済能力を持たせるだけでは、世界は動きません。  
**売り手側が、AI向けの有料APIを簡単に立てられること**も同じか、それ以上に重要です。

「自分のAIエージェントに 402 のテストをさせたい」  
「自分の持つ独自データや計算資源を、AIエージェント向けに従量販売したい」

こうした需要に対して、従来は毎回ゼロから paywall を実装する必要がありました。  
それは労力が大きいと考え用意したのが `ln-church-server` です。

このリポジトリは単なる1パッケージではなく、  
**`@ln-church/server` というコア決済ミドルウェア**と、  
**`@ln-church/hono` というスターター実装**  
をまとめた、売り手側の参照スタックです。

これにより、Cloudflare Workers + Hono 上で、**L402 / MPP / Faucet を中心に動作し、x402 標準ヘッダー互換も備えた paid endpoint** を素早く立ち上げられるようになります。

大事なのは、この server 側スタックが「LN教専用の閉じた実装」ではなく、  
**AI向けに有料APIを公開するための現実的な出発点**として設計されていることです。

LN教本殿は、ここに対して二つの役割を持ちます。

一つは、**参照ノード / benchmark node** としての役割。  
もう一つは、**観測ネットワーク**として「どの 402 endpoint が発見され、どの行動が記録されたか」を蓄積する役割です。

つまり整理すると、

* **LN教本殿** = 参照環境・観測所
* **ln-church-agent** = 買い手側の 402 実行SDK
* **ln-church-server** = 売り手側の paid endpoint 構築スタック

という関係になります。

---

## 3. なぜ「両方」出したのか？

Agent だけを出しても、接続先が増えなければ市場は広がりません。  
Server だけを出しても、402 を安全に通過できる顧客エージェントがいなければ取引は生まれません。

だから両方が必要でした。

`ln-church-agent` は、買い手側に  
**「402 を読む・払う・再実行する」**  
という実行能力を与えます。

`ln-church-server` は、売り手側に  
**「AI向けに paid endpoint を公開する」**  
という供給能力を与えます。

そして LN教本殿は、その間に立って  
**発見・観測・比較・検証のための参照地点**  
として振る舞います。

私はこれを、単なる SDK とテンプレートの配布ではなく、  
**Agentic Web における最小限の市場インフラを揃える試み**  
だと考えています。

**マシン向けに整えたドキュメント面と、実際に叩ける決済付きAPI面は、すでに読まれ始めている**

さらに LN教の Monzen 観測レイヤーでは、外部 402 endpoint の発見や支払い報告を記録できます。  
ただし、ここにも重要な区別があります。

**L402 / Lightning 由来の trace は、preimage による検証を通じて「verified payment」に昇格できます。**  
一方で、**x402-style の外部報告は現時点では受理・記録されるものの、主として self-reported な discovery record として扱われます。**

この区別を明示しているのは、観測を夢物語ではなく、**将来の標準化に耐えるデータ構造**として育てたいからです。

## おわりに：門前町への招待

AIエージェントたちが、自分でAPIを探し、条件を読み、必要なら支払い、結果を持ち帰る。  
そんな世界は、まだ全面的には来ていません。

けれど、その入口に立つための部品は、もう用意できます。

* 買い手側には `ln-church-agent`
* 売り手側には `ln-church-server`
* そして参照環境と観測所として LN教本殿

私はこの3つを、**別々の作品**ではなく、  
**Agentic Web のための一つの実験系**として公開しています。  
![](https://static.zenn.studio/user-upload/65c85112b777-20260415.png)

興味があれば、まずは `pip install ln-church-agent` でクライアント側を試してみてください。  
あるいは `ln-church-server` 側から、あなた自身の paid endpoint を立てても構いません。

そして準備が整ったら、あなたのエージェントを LN教本殿へ送り込んでみてください。  
そこでは単に API を叩けるだけではなく、**AIが何を発見し、何に払い、何を残したか** を観測することができます。

Agentic Web は、思想だけでは前に進みません。  
だから私は、Agent と Server の両方を公開しました。
