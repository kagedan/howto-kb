> 個人開発で作っている **OTOHA（オトハ）** というツールの、コアである「音声 → 5媒体のSNS原稿」を生成する部分の設計メモです。ライブラリの宣伝というより、「文字起こしで終わらせず、話者の口調を保ったまま複数フォーマットを一度に作る」ときに何が難しくて、どう解いたかを共有します。まだ実験の途中なので、つまずきも正直に書きます。

## 何を作ったか（30秒で）

* **入力**: 音声/動画ファイル（or テキスト貼り付け）
* **出力**: **X（3本）/ Instagram / 縦動画台本 / note / メルマガ** の5フォーマットの下書き
* **差別化**: 「文字起こし止まり」（Notta/Rimo等）でも「英語圏前提」（海外ツール）でもなく、**日本語SNSの型 × 話者の口調保持**に振っている

一番こだわっているのは2つ目です。ただ文字起こしを切り刻んで要約すると、誰が書いても同じ"平均的なAI文章"になり、その人の人格が消えます。ここをどう防ぐかが記事の主役です。

## 素朴な実装だと何がダメか

最初は素直に「文字起こし全文を渡して、フォーマットごとに1コールずつ生成」していました。これだと2つ問題が出ます。

1. **口調が媒体ごとにバラつく**。X はやたらフランク、note は急にお堅い、みたいに人格が揃わない。各コールが独立に「それっぽい文体」を選んでしまうからです。
2. **毎回、全文を読み直させるので冗長**。要約・トピック抽出を各フォーマットが暗黙にやり直していて、無駄が多い。

そこで **二段生成** にしました。

## 設計の肝：二段生成（前処理1コール → 並列5コール）

```
音声/動画
  │  文字起こし（Whisper系）
  ▼
文字起こし全文
  │  【前処理 1コール】要約 + トピック + 口調プロファイル を構造化抽出
  ▼
Preprocessed { summary, topics, tone }
  │  【並列 5コール】各フォーマット専用プロンプト＋「口調プロファイル厳守」
  ▼
X / Instagram / 縦動画台本 / note / メルマガ
```

ポイントは、**「話者の口調」を先に1回だけ確定させてから、それを全フォーマットに制約として配る**こと。人格を固定してから書かせるので、5媒体で"同じ人"になります。

### ステップ1：口調プロファイルを構造化抽出する

前処理コールで、要約・トピックと一緒に「口調プロファイル」を構造化して取り出します。型はこんな感じです。

```
export type ToneProfile = {
  first_person: string;    // 一人称（例: 僕 / 私 / 俺）
  sentence_ending: string; // 語尾・文末の癖（例:「〜ですね」「〜っす」）
  energy: string;          // 熱量（例: 落ち着いている / 熱い / フラット）
  expertise: string;       // 専門度（例: 初心者向け / 中級 / 専門的）
  notes: string;           // その他の話し方の特徴
};

export type Preprocessed = {
  summary: string;   // 3行要約
  topics: string[];  // 発信ネタになる主要トピック 3〜6個
  tone: ToneProfile;
};
```

前処理プロンプトは「文字起こしから summary / topics / tone を JSON で返して」という主旨のものを1本投げ、レスポンスをパースして `Preprocessed` にします（JSON はコードフェンス混入や前後の説明が混ざりがちなので、`{` 〜 `}` を素朴に切り出してからパースする保険を噛ませています）。

### ステップ2：口調を"制約文"に変換して全生成に差し込む

抽出した `ToneProfile` は、そのままだと生成に効きません。\*\*各フォーマットのプロンプト共通で差し込む「制約文」\*\*に変換します（要約すると）：

```
export const toneConstraint = (t: ToneProfile) => `
【話者の口調プロファイル（厳守）】
- 一人称: ${t.first_person}
- 語尾/文末の癖: ${t.sentence_ending}
- 熱量: ${t.energy} / 専門度: ${t.expertise}
- その他: ${t.notes}
（この話し方を保ったまま各SNSの型に落とす。別人のような整いすぎた文章にしない）
`.trim();
```

各フォーマットのプロンプトは `要約 + トピック + toneConstraint(tone) + 共通ルール + そのフォーマット固有の型` という組み立てにしていて、口調制約は毎回同じものが入ります。これで「Xでもnoteでも一人称と語尾が揃う」状態になります。

### ステップ3：5フォーマットを並列生成する

前処理が終われば、5フォーマットは互いに独立なので `Promise.all` で並列に投げます。

```
const pre = await preprocess(client, transcript); // 1コール
const outputs = await Promise.all(
  FORMATS.map(async (f) => ({
    key: f.key,
    text: await generateOne(client, f, pre), // 5コール並列
  }))
);
```

直列だと5フォーマット分の待ち時間が積み上がりますが、並列なら実質「一番遅い1コール分」で返ります。体感で初回20秒台 → 改善後17秒台くらいに収まりました（モデルは既定で `claude-haiku-4-5`。速さとコストのバランスがこの用途に合っています）。

## 共通ルールで"事故"を止める

生成AIに任せると、放っておくと勝手に盛るのが一番怖い。特にこのツールの客層は発信初心者なので、**素材に無い固有名詞や数字を創作されると信頼を失います**。そこで全フォーマット共通のルールで縛っています（抜粋）。

* 前置き・あいさつ・「以下が〜です」を書かない。**完成テキストだけ**返す
* 「# Instagram投稿案」のような**メタ見出し（表紙ラベル）を付けない**
* **文字起こしに無い固有名詞・数字・具体例を足さない**（例: 元ネタに無いプラットフォーム名・「週3回」等の数値を創作しない）
* **SNS名は収録で実際に言及されたものだけ**使う。一般化したいときは「各プラットフォーム」と書く

Haiku は放っておくと「TikTok にも展開しましょう」「3ヶ月で成果が…」みたいに、言ってもいないことを足しがちでした。共通ルールに明示してからはほぼゼロになっています。

## つまずき（Build in Public として正直に）

### 語尾を機械的に足して"非文"になる

口調保持を効かせすぎると、逆に日本語として壊れます。実際に出たのがこれ：

> 「中核になる**なのかな**」「問い直されてる**なのかな**」

`sentence_ending` に「〜なのかな」を入れると、動詞の辞書形に無理やり接続して非文を作ってしまう。対処として、口調制約に「**語尾は日本語として自然に接続する。辞書形にそのまま繋げた非文を作らない。必要なら『〜になるのかな』と整える。口調の再現より文の自然さを優先**」という一文を足しました。**「口調保持」と「自然さ」は衝突しうるので、優先順位を明示する**のが効きました。

### 「全体やり直し」をやめて、1カード単位の再生成にした

初期は気に入らないと5フォーマット全部を作り直していました。でもユーザーが直したいのはたいてい1枚だけ。全再生成は待ち時間もAPIコストも無駄なので、**カード単位の再生成**に変えました。前処理（要約・口調）は使い回すので、再生成は生成1コールだけで済みます。

### 日本語Whisperと"下手な録音"前提の設計

初心者は録音に不慣れで、雑音・小声・短すぎる音声が普通に来ます。文字起こしは Groq → OpenAI → ローカル whisper.cpp のフォールバックにしていますが、多少の誤変換は残る。**ただ二段生成はこれに意外と強い**とわかりました。前処理の要約時に Claude が文脈から誤変換を吸収してくれるので、`small` モデルの「ストロプレイナー」みたいな誤変換が後段では自然に補正される。二段構えの副次的な頑健性です。

### 長尺を1セットに畳みすぎる

9分のトークに実質6トピックあったのに、1セット5発信に圧縮してしまう問題。「1収録＝1セット」の前提が長尺で破綻するので、**収録を"発信単位の塊"に分割して複数セット作る**方向に拡張中です（口調プロファイルは収録全体で1回だけ抽出して使い回す）。ここはまだ実験中。

## スタック

* **Next.js 15 / React 19**
* **文字起こし**: Whisper系（Groq / OpenAI / ローカル whisper.cpp フォールバック）
* **生成**: Claude（既定 `claude-haiku-4-5`、環境変数で差し替え可）
* プロンプト設計は `lib/prompts.ts` に集約（口調制約・共通ルール・フォーマット定義）。生成の retry は薄いラッパーで包む

## まとめ

「音声を複数SNSに展開する」だけならツールは既にたくさんあります。差がつくのは**話者の人格を消さないこと**で、そのために

1. 口調を**先に構造化抽出**して固定する（二段生成）
2. それを**全フォーマット共通の制約文**として配る
3. 「口調保持 vs 自然さ」の**優先順位を明示**する
4. 共通ルールで**創作（盛り）を止める**

というのが今のところの答えです。まだ完成版ではなく、毎日自分の声で品質を詰めている段階です。
