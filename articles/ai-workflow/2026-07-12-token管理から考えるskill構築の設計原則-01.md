## 概要

今回の改修は、「Skillによりよい文章を書かせるにはどうすればよいか」から始まったのではない。もっと根本的な問いから始まった。**役割分担が明確に見えるSkillが、なぜ大量のTokenを消費し、使うほど保守しにくくなるのか。**

当初は、アウトライン、レビュー、タイトル調査、タイトル選択、草稿、校閲、最終稿、タグ、公開と、工程を細かく分けていた。工程を細かくし、ファイルを増やし、Agentを独立させるほど、コンテキストを節約できると直感的には思える。

しかし実際には、主コーディネーターのコンテキストを減らせても、子Agentの起動、再読、承認往復、中間成果物の保守というコストが増えることがある。

そこで問題を「あるPromptをどう圧縮するか」ではなく、次のように置き換えた。

> 1つのタスクで情報をいつ取り込み、保持し、移し、再利用し、無効化するかをどう管理すれば、総Tokenコストをタスクの複雑さに見合うものにできるか？

この記事では、そこから得たSkill構築の設計原則を記録する。焦点は特定のブログ基盤、コマンド、コードではなく、プロセス設計の考え方である。

---

## 改修前のSkill

### ファイル構成

改修前のSkillは、おおむね「総合説明、工程別Prompt、少数の専用スクリプト」で構成されていた。

```
skill/
├── SKILL.md
├── references/
│   ├── stage-prompts.md
│   ├── title-keywords.md
│   ├── publication-workflow.md
│   └── deprecated-cache-note.md
├── scripts/
│   └── topic-count.js
└── agents/
    └── openai.yaml

workspace/
├── shared/
│   ├── config/
│   ├── knowledge/
│   └── templates/
└── runs/
    └── <single-run>/
        ├── work/
        └── outputs/
```

共有キャッシュと単発成果物を分離し、決定的な処理をスクリプトに委譲し、公開手順を独立させ、実行ディレクトリに履歴を残す。方向性として正しい要素はすでにあった。

問題は「ファイルを分けたか」ではない。分割粒度がなお、「考える操作はすべて1工程に値する」という仮定に従っていたことだった。

### 文章で表すワークフロー

全体の流れは次のようなものだった。

```
要求
→ Brief
→ 3つのアウトライン案
→ 人によるアウトライン承認
→ アウトラインレビュー
→ タイトル・キーワード調査
→ 3つの多言語タイトル案
→ 人によるタイトル承認
→ 本文草稿
→ 本文レビュー
→ 最終編集
→ タグ調査
→ Metadata
→ プレビューまたは公開
```

各工程には入力ファイル、出力ファイル、役割の説明があった。この設計は追跡性が高い。どの判断も振り返れ、どの工程も個別に議論できる。

一方で、通常の執筆依頼まで完全な生産ラインとして扱う設計でもあった。

## 改修前に整理した問題と対策

### 問題はコンテキストの長さだけではなく、総コストの制御不能さにある

ワークフローのコストを、主Agentが最終的に読んだ量だけで測るべきではない。より実態に近い表現は次のとおりである。

```
総コスト
= Skill指示の読み込み
+ 子Agentの起動コスト
+ 各工程の入力
+ 各工程の出力
+ ツールと検索結果
+ 人による承認往復
+ 再試行と重複実行
```

複数のAgentへ分ければ、確かに1つのAgentが抱えるコンテキストは小さくなる。しかしAgentを1つ増やすたびに、システム指示、ツール定義、タスク説明、結果の受け渡しが増える。工程自体が軽いなら、固定の起動コストが得られる効果を上回りかねない。

したがって、「工程が多いほどTokenが少ない」とは限らない。

### 問題1：すべての依頼が完全なフローへ入る

既存記事の推敲、単純なレビュー、既存ファイルの公開準備、テーマからの記事作成では、必要な複雑さがまったく異なる。すべてを完全フローに通すと、目的に無関係な規則、工程、成果物まで持ち込んでしまう。

対策は、実行前にルーティングすることだ。まず依頼を高速修正、標準作成、調査型作成、再開、公開のいずれかに分類する。すべての能力を最初から読み込ませない。

### 問題2：工程境界が「思考手順」であり「情報境界」ではない

アウトライン、タイトル、アウトラインレビューは、いずれも記事設計に属する。同じBriefを何度も読み、同じ決定を複数のファイルで言い換えるなら、別々のAgent、ファイル、承認に分ける意味は薄い。

対策は情報境界でまとめることだ。同じ入力を使い、同種の判断を出力する作業は、できるだけ同じコンテキストで済ませる。通常の執筆なら、アウトラインと多言語タイトルを「記事設計案」として1回で承認する。

### 問題3：人による承認も往復コストである

承認は無料ではない。「A/B/Cから選んでください」というやり取りごとに会話ターンが増え、コーディネーターが前の状態を復元し直す場合もある。

対策は承認をなくすことではなく、1回の承認で完結した判断を扱うことだ。通常の記事では、構成と多言語タイトルをまとめて確認する。構成に大きな争点がある、または利用者が段階的な制御を明示的に求める場合だけ、承認を分ける。

### 問題4：決定的な作業まで言語モデルが担当している

タイトル文字数、ファイル名、Front Matter、キャッシュ鮮度、Topic検索、状態更新、Hash比較は、意味推論を必要としない。モデルに任せるとTokenを消費するだけでなく、形式も揺れやすい。

対策は決定的な処理をスクリプトへ移すことだ。モデルは、切り口の選択、論証の構成、トレードオフの説明、意味上の問題の発見といった判断だけを担う。

### 問題5：中間成果物に保存はあっても、無効化がない

ファイルに保存しただけでは再開可能とは言えない。Briefが変わったとき、以前のアウトライン、草稿、レビューはまだ信用できるのか。入力Hashがなければ、スキップできる完了工程と、作り直すべき古い結果を区別できない。

対策は実行状態を明示することだ。工程、入力Hash、成果物Hash、次工程、指標を記録する。上流が変わったら、本当に依存する下流だけを無効化する。

## 改修後の姿

### ファイル構成

改修後は、Skill本体をルーターへ絞り込み、詳細は利用場面ごとのreferenceへ分離した。決定的な規則と実行状態は、スクリプトと設定に委ねる。

```
skill/
├── SKILL.md                    # 要求ルーティング、必須規則、完了条件
├── references/
│   ├── create-workflow.md      # standard / research
│   ├── review-workflow.md      # quick / review / resume
│   ├── publish-workflow.md     # publish
│   └── output-rules.md         # タイトル、Topic、Front Matterなど
├── scripts/
│   └── external-query.js
└── agents/
    └── openai.yaml

workspace/
├── shared/
│   ├── config/                 # モード、制限、キャッシュTTL、既定値
│   ├── knowledge/              # 再利用できるキャッシュと別名
│   └── templates/              # 安定したテンプレート
├── scripts/
│   ├── create-run.js
│   ├── update-run-state.js
│   ├── check-title-length.js
│   ├── research-topics.js
│   └── prepare-output.js
└── runs/
    └── <timestamp>-<title>/
        ├── manifest.json
        ├── brief.md
        ├── plan.md
        ├── draft.md
        ├── review.json
        ├── metrics.json
        └── outputs/
            ├── article.md
            └── metadata.json
```

これは単にファイルを減らす話ではない。責務を次のように分ける設計である。

* Skillは経路を選ぶ
* Referenceはその経路を説明する
* Scriptは決定的に実行する
* Shared状態は実行をまたいで再利用する
* Run状態は1回のタスクを再開可能にする
* モデルは意味的な判断を行う

### 文章で表すワークフロー

改修後の最初の工程はコンテンツ生成ではなく、モード判定である。

```
ユーザーの要求
→ ルーティング：quick / standard / research / resume / publish
```

標準作成モードは次のようになる。

```
Brief
→ Planner：アウトライン + 自己レビュー + 多言語タイトル案
→ 記事設計を1回承認
→ Writer：本文草稿
→ Review-Finalizer：レビューと最終稿の出力
→ Script：タイトル、Topic、Front Matter、状態、指標
```

調査型モードでは、必要なときだけ資料台帳と独立レビューを追加する。高速モードは既存草稿を直接修正またはレビューし、再開モードはまずManifestを読んで次を決め、公開モードは検証、バージョン管理、デプロイだけを扱う。

重要なのは作業量を減らすことではない。複雑さをSkillの最大能力ではなく、依頼の複雑さに従わせることである。

## 1回の改修から抽象化したSkill構築の原則

### 1. Agentの役割より先に、情報のライフサイクルを設計する

Skillを作るとき、最初の問いは「Agentを何人にするか」ではない。次を問うべきである。

* この情報はいつモデルに入るか
* 本当に必要な役割は誰か
* 次工程でも保持する価値があるか
* コンテキスト、ファイル、キャッシュのどこに置くか
* 上流が変わったとき、誰を無効化するか

コンテキストはその場の判断に、ファイルは追跡可能なタスク状態に、キャッシュは再利用できる事実に向く。すべてを会話に置くことも、すべてを文書にすることも、境界が曖昧な状態である。

### 2. 子Agentを固定コストを持つ計算単位として扱う

子Agentの価値は「視点が1つ増える」ことではない。本当に独立し、十分に短いコンテキスト境界を得ることにある。

新しいAgentには少なくとも、起動Prompt、ツール定義、タスク説明、引き継ぎのコストがある。そのため、分ける価値があるのは次の場合に限られる。

* 新しい役割が前の役割と明確に異なる入力を必要とする
* 独立レビューが大きな品質向上をもたらす
* 1つのコンテキストが隔離を必要とするほど大きい
* タスクを並列に進められ、結果の重なりが小さい

それ以外では、関連する判断を同じ役割呼び出しで済ませる。たとえば「計画、自己レビュー、タイトル設計」を3つの軽量Agentに分けるより、まとめた方が経済的なことが多い。

### 3. 全機能の既定読み込みを、モードルーティングで置き換える

Skillの能力境界は広くてよい。しかし1回の依頼が、すべての能力を既定で受け継ぐべきではない。

まず、目標を満たす最も低コストなモードへ依頼を対応付ける。この原則は次のように要約できる。

> フローの複雑さは、Skillの機能総量ではなく、依頼の複雑さに従わせる。

ルーティングはTokenを節約するだけでなく、振る舞いを予測可能にする。レビューが意図せず調査を始めず、公開が意図せず新しいアウトラインを作らず、高速修正が一式の中間成果物を生まない。

### 4. 「コンパクト」を検証可能なインターフェース契約にする

「簡潔にしてください」は願いであり、制約ではない。Tokenを節約するには、これをインターフェースとして定義する必要がある。

* Briefの最大文字数
* Planの最大文字数と選択肢数
* Reviewの最大問題数
* Metadataの固定フィールド
* タイトル文字数の決定的な検査
* Topic数と形式の検査

中間成果物に境界があれば、Agent間で渡すものは無限に増える散文ではなく、有限で安定し、検証可能なプロトコルになる。

### 5. モデルには判断を、スクリプトには決定性を任せる

実用的な境界は次のとおりだ。

| 作業の種類 | 適した担当 |
| --- | --- |
| 切り口の選択、論点の発見、トレードオフの説明、意味上の問題の発見 | モデル |
| 計数、検証、Hash、パス生成、キャッシュ更新、形式の組み立て | スクリプト |
| 実行をまたいで再利用する事実 | 共有キャッシュ |
| 単発タスクの工程と依存関係 | Manifest |

これはモデルを排除するためではない。本来プログラムで決められる処理を、費用が高く不安定な言語推論で模倣しないための分担である。

### 6. 再開をフローの再生ではなく、増分計算として扱う

再開可能なSkillは「前回どこまで進んだか」だけでなく、「前回の結果が依存した入力は今も成立するか」を問うべきである。

Manifestには少なくとも次を記録する。

* 完了済みの工程
* 各成果物のHash
* 各工程の入力Hash
* 次工程
* 下流の依存関係
* コスト指標

これにより再開は、会話履歴から状態を推測する作業ではなく、小さな増分ビルドになる。入力が変わらなければスキップし、変われば影響を受ける下流だけを再計算する。

### 7. 偽の精度を作らず、総コストを観測する

実行環境が実際のToken使用量を返すとは限らない。その場合は観測できる代理指標を記録する。

* Agent呼び出し回数
* 入出力とツール結果の文字数
* キャッシュのヒットとミス
* Hashによりスキップされた工程
* 取得可能な場合のみ実Token

文字数からの推定値を正確な課金額のように見せてはならない。指標の目的は傾向と回帰を見つけることにある。変更によってAgent数が増えていないか、ある工程が全文を繰り返し読んでいないか、キャッシュが無効化されていないかを把握できる。

## 次にSkillを構築するとき、AIへどう依頼するか

次回は、AIにいきなり「完全なSkillを書いて」と頼まない方がよい。まずToken経済性を制約として、アーキテクチャを設計させる方が効果的である。

以下は再利用できるPromptテンプレートである。

```
<タスク目標>のためのSkillを作りたい。

まずSKILL.mdを直接書かず、次の順で設計してほしい。

1. ユーザー要求をコストの異なるモードへ分け、各モードの条件、最小フロー、読み込んではならないものを示す。
2. 主Agentのコンテキストだけでなく、指示読み込み、Agent起動、入出力、ツール結果、承認往復、再試行を含む総Tokenコストを目的にする。
3. 思考手順でなく情報境界でAgentを分ける。各Agentが必ず読むもの、絶対に読まないもの、固定起動コストを払って分ける価値を示す。
4. 共有状態、単発Run状態、キャッシュ、Manifest、入力Hash、成果物Hash、下流の無効化規則からなる最小実行状態を設計する。
5. 計数、形式検証、キャッシュ更新、パス生成、状態更新など決定的な処理はスクリプトにし、モデルには意味判断だけを任せる。
6. 中間成果物ごとに構造、最大長、フィールド、検証方法を定義し、「できるだけ簡潔」のような検証不能な指示を避ける。
7. 再開ロジックを設計する。入力不変なら何をスキップし、入力変更ならどの下流を無効化するかを示す。
8. コスト観測を設計する。少なくともAgent回数、文字数、ツール結果量、キャッシュヒット、スキップ回数を記録し、推定値と実Tokenを区別する。
9. 最後に、簡潔なSKILL.md、オンデマンドのreferences、スクリプト一覧、最小のforward test案を出力する。

すべての設計で次を守る。フローの複雑さは依頼の複雑さに従わせる。コンテキストには現在の判断に必要な情報だけを置く。完全な事実は外部化し、参照、要約、キャッシュで再利用する。
```

このPromptの目的は、AIに機械的にファイルやスクリプトを増やさせることではない。どの情報をモデルに入れる価値があるのか、どの役割を起動する価値があるのか、どの状態を保存する価値があるのか、そもそもモデルに任せるべきでない作業は何かを、先に答えさせることである。

## 設計段階でこれらの問いに答えられていれば、Skillは使いやすさ、再開可能性、Token経済性を同時に備えやすくなる。
