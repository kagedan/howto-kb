## はじめに

少し前に [AIへの丸投げと細部の業務設計 — サブセッション自走化で見えた明暗の分岐](https://zenn.dev/fixu/articles/subsession-self-drive-guardrails) という記事を書きました。Laravel 8→12 / PHP 8.1→8.3 / alpine 化という MW（ミドルウェア）アップグレードを 9 リポジトリ横断で進める中で、サブセッションの自走運用に踏み込むときに浮上したリスク 7 類型と、親側に組み込んだ監視ガードレールについてまとめたものです。

あれから 2 週間ほど、同じ 9 リポジトリで次のフェーズ — ステートレス化（Cache / Logs / Queue / File upload を ECS Fargate 想定で外出し）— に進みました。前作で整理した 7 類型のリスクは、起動プロンプトとガードレール強化ブロードキャストによって、実務上はかなり抑え込めるようになっていました。

ところが、その先に **「AI multi-agent（実装サブ + 親 + 第三者監査サブ + Copilot レビュー）の全層を通り抜けた重大インシデント未遂」** が起きました。発見できたのは、人間（筆者）が後から「もっと踏み込んだ regression test できそう?」と追加要請したからで、もし要請がなければ 9 リポジトリ全部で「✅ 完遂」表示の裏で、本番反映後に顧客側で問題が表面化する状態が残っていたところでした。

さらに少し遡って、前々作 [AI開発でユーザーの判断軸が効いた1日 — Claude × Copilot と作るインフラの現実](https://zenn.dev/fixu/articles/day051-ai-human-collaboration) では、「人間の判断軸 × AI の実行速度 × 複数視点のレビュー」の三層構成で品質が決まる、と書きました。あの時点では、人間がリアルタイムで Claude の暴走を訂正する役割でした。今回は構造が一段違います。**multi-agent + Copilot レビューが平常運用化してもなお、AI 群の全層が通すケースが残る** — そこに人間の批判的圧力という最終層を組み込まないと、業務設計として閉じない、という体験でした。

この記事は、

* multi-agent + Copilot レビューでコードレベルの座組はかなり確立されてきたこと（実測値つき）
* それでもなお検出できなかった構造的な blindspot が verify 経路の真正性層に存在したこと
* AI 多層防御がなぜ全層を通したのかの構造分析
* ここを「コードと同じ手触り」で塞ぐ test / audit / QA の best practice はまだ手探りで、知見を求めていること

を書き残すものです。

冒頭で結論めいたことを書いてしまうと、

> **AI multi-agent でも検出できない unknown unknowns は存在する。人間の役割は「実行」から「批判的圧力 + 不在を疑う」へ shift する。**

これが今回得た最大の手触りです。

## 1. 前作からの差分 — リスク 7 類型は抑えた、その先に何があったか

前作の続編なので、まず構造的な差分を可視化します。

| 観点 | 前作（MW upgrade フェーズ） | 今回（ステートレス化フェーズ） |
| --- | --- | --- |
| 主作業 | Laravel/PHP/alpine の major upgrade | Cache/Logs/Queue/File upload の ECS Fargate 想定外出し |
| AI 構成 | 実装サブ + 親 + Copilot レビュー | 実装サブ + 親 + **第三者監査サブ** + Copilot レビュー |
| 抑え込み対象リスク | 自走運用の構造リスク 7 類型（証跡 / 判断品質 / エスカレ漏れ / 機械的判断 / 自走副作用 / 経路依存性 / 進捗管理） | **verify 経路の真正性層** — 前作で整理しきれていなかった層 |
| 人間の主役割 | 自走運用への監視（sweep + audit + transparency） | **AI 多層防御後の最終批判圧力（「不在」を疑う）** |
| 主な発見 | 細部設計の積み重ねが明暗を分ける | 多層防御は「同じ model 由来」だと独立性が稼げない |

前作で整理した 7 類型は、起動プロンプトのチューニングとブロードキャスト配信で、実務上はかなり抑え込めるようになりました。サブセッションは scope-out 判定時に必ず Copilot へ返信を投げるようになり、9 リポジトリ統一基準の認識はブロードキャストで同期されます。証跡は残るし、エスカレも回ります。

ただ、それで「自走運用は安全に閉じる」と思った瞬間に、別レイヤの blindspot が立ち上がりました。**verify 経路の真正性 — その verify は本当に実 user 経路を通っているか? という問い** です。これは 7 類型のどれにも厳密にはハマらない、verify 設計層に固有の問題でした。

> **前作との差分**: 前作はサブの「行動」を律する話、本作はサブの「verify 設計」を律する話。

## 2. コードレベルでは座組ができてきた — Copilot レビュー平均 1.5 ラウンドの実測

まず、対策側の現状を率直に書きます。

コード変更については、Claude（実装サブ）が PR を作り、それに対して GitHub Copilot のコードレビューを入れる、という座組がかなり機能するようになりました。Copilot は同じ Claude 由来ではない、別の AI レビュアーです。同じ AI 同士でも model 系列が違えば、ある程度は独立した視点を持ち込めます。

実測値も取りました。Day062 以降の約 12 日間、9 リポジトリで merged された PR を集計すると以下です（執筆時点）。

| 指標 | 値 |
| --- | --- |
| 期間 | Day062〜（約 12 日間） |
| 集計対象 PR | 247 件（merged） |
| Copilot レビューが付いた PR | 195 件（79%） |
| **Copilot レビュー平均ラウンド数** | **1.5 ラウンド** |
| 1 ラウンドで収束した割合 | 73% |
| 2 ラウンド以上（往復が発生） | 27%（53 件） |

「ラウンド」とは Copilot レビューが PR に submit された回数です。Round 1 で指摘なし or 軽微で即 merge できた PR が 7 割、Round 1 で指摘 → 対応 → Round 2 で収束、という往復が起きたケースが 2 割強。重めの議論で 4〜6 ラウンドまで行った PR も 5% ありました。

体感としては「コードのレビュー観点では概ね座組として機能している」と言えます。Claude が単独で書いて単独で merge する状況と比べれば、確実に視点の独立性が一段増えています。前々作で書いた「人間 × AI 実行速度 × 多視点レビュー」の三層構成のうち、**多視点レビューの軸がコード側ではかなり整備された** — これは確かな進歩でした。

ところが、本記事の核はここからで、**この座組は test / audit / QA の観点には届いていない** という発見でした。

> **章末の蒸留**: コード変更の品質保証は Claude × Copilot で平均 1.5 ラウンドの座組として機能。ただしこれは「実装の妥当性」を見るレビューであって、「verify 設計の妥当性」を見るレビューではない。

## 3. それでも全層を通り抜けた事例 — verify 経路の真正性

具体的に何が起きたかを書きます。

ステートレス化フェーズの pilot リポジトリで、ファイルアップロード周りを ECS Fargate 想定の S3 ストレージに切り替える作業をしていました。実装サブが移行を完了し、続いて baseline テストの設計に進みました。baseline は 6 つの verify path で構成されており、そのうちの 1 つが「**S3 access** — ECS タスクから S3 バケットの画像群を一覧取得できるか」というものでした。

実装サブは以下のように verify しました（抽象化して書きます）。

```
# サブセッション自身の AWS profile から
aws s3 ls s3://<bucket>/photo/face/ --region <region>
# → 1099 件のファイルが返る
```

「1099 件 list 成功 → ✅ S3 access OK」と判定。baseline 6 パスすべて ✅ で完遂と報告してきました。

親側（筆者 Claude）はこれを承認しました。第三者監査サブを別途立ち上げて、ブロードキャスト前の独立 GO/NO-GO 判定をかけました。監査サブは 10 項目の観点で確認し、7 ✅ 3 🟡（cosmetic）→ 🟢 GO 判定でした。Copilot レビューも該当 PR では Round 1 で軽微な指摘のみ、merge 済み。

ここまでで AI 多層防御 4 層（実装サブ + 親 + 監査サブ + Copilot）が全層通過しています。

ところが — そして本記事のテーマはここからです — 筆者（人間）が「baseline は了解。**もっと踏み込んだ regression test できそう?**」と追加要請しました。実装サブは追加 3 ケースを設計し、そのうちの 1 つで初めて、

```
# ECS Exec で実コンテナに入り、実 app のエンドポイントを実 user 経路で叩く
curl -i http://<container>/image/<filename>
# → HTTP 404
```

**HTTP 404 が出ました**。

つまり、実装サブが「✅ S3 access OK」と判定した `aws s3 ls` の経路では list できていたのに、**実 app の HTTP 経路では 1 件も画像が返ってこない** という状態でした。仮にここで人間の追加要請がなければ、9 リポジトリ全部で「✅ 完遂」表記の裏で、本番反映後に顧客側で画像表示の経路で問題が表面化していたところでした。重大インシデント未遂です。

### 真因（簡略）

技術的にはチェーンになっていました。

1. Laravel 標準の env 名（`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`）がデプロイ先の環境変数管理に未投入で、過去の独自 env 名のみ存在していた
2. Laravel 設定は標準 env 名を参照するため空となり、AWS SDK が ECS タスクロール fallback に流れた
3. ECS タスクロールには対象バケットの S3 access policy が attach されておらず AccessDenied
4. アプリ層の Throwable catch が AccessDenied を呑み、ローカルストレージへ fallback
5. ECS Fargate の ephemeral storage は当然 0 件 → 一覧 0 件 → 個別ファイルアクセスは 404

これらが直列で繋がって、**実装サブの `aws s3 ls` はサブ自身の AWS profile（広い権限）で通っていた一方、実 app は app 経路（= 狭い ECS タスクロール）で全く通っていない** という構造でした。

### 何が問題だったか

このバグそのものは設定 1 行レベルで解消する技術的には軽微なもので、本番反映前に発見できているため実影響はありません。問題はそこではなく、**「AI 多層防御 4 層を通り抜けた」というメタな構造** にあります。

> **章末の蒸留**: 「✅ PASS」と「実 user 経路で実際に動く」は別問題。verify が **どの権限経路** で通っているかを問わない限り、PASS は false positive を含み得る。

## 4. 多層防御がなぜ通したのか — 構造分析

各層がなぜ通したのか、層別に分解します。

| 層 | 何を見ていたか | 結果 | 通した理由 |
| --- | --- | --- | --- |
| 1. 実装サブ | baseline 6 path の自己verify | ✅ PASS と判定 | サブ自身の AWS profile で list 成功 → 「S3 access OK」判定。**verify 経路の真正性は観点に入っていなかった** |
| 2. 親（Claude） | baseline 設計の承認 + 完遂判定 | ✅ 完遂 | path #5 が ✅ PASS の報告を信じた。「その verify は広い権限で bypass してないか?」を問わなかった |
| 3. 第三者監査サブ | ブロードキャスト前 GO/NO-GO 判定 | 🟢 GO | 監査プロンプトが「6 path 達成の verify」だった。**baseline 設計そのものの十分性は観点外** |
| 4. Copilot レビュー | 該当 PR のコードレビュー | 軽微指摘のみで merge | コード差分のレビューであって、verify 設計のレビューではない |
| **5. 人間（筆者）** | **追加 regression 要請** | **検出** | **「もっと踏み込んだ verify できそう?」の批判的圧力で初めて exec curl 経由の verify が組まれ、404 が露呈** |

### 同じ model 系列の AI には同じ blindspot がある

AI multi-agent を組めば、原理的には独立性が稼げるはずです。実装サブと監査サブは独立したセッションで、互いの context にアクセスしません。「複数の AI が独立に見れば blindspot は解消する」 — 直感的にはそう思いがちです。

ところが今回明らかになったのは、**「AI のセッションが独立していても、model 系列が同じであれば系統的な blindspot は共有される」** という点でした。

実装サブも監査サブも、どちらも同じ Claude 系列の model でした。両者ともに「verify は path 達成チェックリスト型に組む」「✅ PASS と書かれていれば達成扱い」という形式的 verify への傾きを共有していました。これは独立セッションであることでは解消しないクセです。**学習データ由来の系統的な傾向は、セッション独立性では拭えない**。

実装サブだけが Copilot レビューを浴びる構造になっていますが、Copilot は **コード差分** を見るレビュアーで、**verify 設計の十分性** を見る役回りには配置されていません。つまり、verify 設計の観点に対しては、独立性のある外部視点（= 同じ model 系列ではないレビュアー）が組み込まれていなかった、というのが今回の構造的な気づきです。

> **章末の蒸留**: 同じ model 系列の AI を何層重ねても、その系列の系統的な blindspot は通り抜ける。独立性は「セッション分割」ではなく「**model 系列 + 観点 + 経路**」の独立で稼ぐ必要がある。

## 5. 引き出した原則 — verify 経路の真正性と人間の批判圧力

今回得た原則は 2 つです。

### 原則 5-1: verify 経路の真正性（path authenticity）

verify は「✅ PASS」を返すだけでは足りず、**どの権限経路で通したかを必ず問う** 必要があります。チェックリスト化しました。

* **権限経路**: サブ自身 / 管理者 profile / サブの広い権限を使っていないか?（使っていれば false positive リスク）
* **実 user 経路**: 実 user が通る最小権限経路で verify しているか?
* **アプリ層通過**: 実コンテナ / 実 app プロセスを実際に通って verify しているか?（サブの local 環境からの直接 CLI / 直接 SQL は不可）
* **error path も verify**: 異常系（権限なし / リソースなし）も実 user 経路で verify

この 4 点を起動プロンプトと監査プロンプトの両方に「**必須観点**」として書き込み、verify 設計時にサブが自発的にチェックする形に倒しました。

### 原則 5-2: 人間の批判的圧力を構造に組み込む

AI 主導の verify 設計は、明示的にプロンプトされない限り、形式的 verify（path 達成チェックリスト型）に流れます。これは AI を責める話ではなく、構造として認識すべき性質です。

人間（筆者 / レビュアー）は AI の verify 設計提案を受け取ったときに、以下の批判的圧力を構造的に加えるルールにしました。

* 「その verify は管理者 profile / サブ自身の広い権限で bypass していないか?」
* 「実 user が同じ path を通ったら同じ結果になるか?」
* 「サブの権限が広すぎて、app 層の権限不足を隠蔽していないか?」

これは前々作 [Day051 記事](https://zenn.dev/fixu/articles/day051-ai-human-collaboration) の「人間の判断軸」と地続きですが、適用層が深くなりました。Day051 時点では「Claude が rolling update を提案 → 人間が本番見越して却下」のような **コード設計層** での判断軸でした。今回は **verify 設計層** に同じ構造の判断軸が必要になった、ということです。

### 監査サブのプロンプト改修

第三者監査サブを起動するとき、プロンプトに以下を必須観点として渡すように変更しました。

* baseline / regression 設計**自体**の十分性を critical に評価
* verify 経路の真正性（管理者 / サブ権限 bypass の有無）
* multi-agent 同 blindspot リスクへの自覚（「自分は実装サブと同じ model 系列であり、同じ blindspot を共有している可能性がある」）

最後の項目を入れたのが今回の小さな発明でした。**「自分の独立性を過信しないこと」をプロンプトで宣言させる**。

> **章末の蒸留**: verify 経路の真正性は AI に自発的に確認させづらい層。プロンプトでの明示的観点注入 + 人間の批判圧力の併用で抑える。

## 6. それでも test / audit / QA は best practice 未確立 — コメント募集

ここまで対策を書いてきましたが、率直に書くと、**コードレベルで Claude × Copilot の座組ができているのと同じ手触りで、test / audit / QA 観点のレビューはまだ組めていません**。

整理するとこうです。

| 観点 | コードレベル | test / audit / QA レベル |
| --- | --- | --- |
| Claude が一次成果物を作る | ✅ 確立 | ✅ できる（test 設計 / 監査観点設計） |
| 別 model 系列のレビュアーで多視点クロスチェック | ✅ Copilot レビュー（平均 1.5 ラウンド） | ❌ **未確立** |
| 数値で運用できる | ✅ ラウンド数 / 指摘件数で管理可 | ❌ 「観点の網羅度」をどう測るか不明 |
| best practice の参照先 | レビュー文化の蓄積あり | 手探り |

理由を考えると、コードレビューは「diff を見る」というスコープが明確で、Copilot のような自動レビュアーが介入する設計が業界全体で蓄積されています。一方で test / audit / QA は「**何を見るかを設計する作業の妥当性**」を問うので、レビュー対象自体が flat な diff ではなく、観点の network 構造になります。これに対して別の model でクロスチェックを組むのが、概念的には可能でも実装としては手探りです。

現時点で組み込めているのは、

* 監査サブのプロンプトに「baseline / regression 設計の十分性そのもの」を critical 観点として明示注入
* 起動プロンプトに verify 経路の真正性チェックリスト 4 点を埋め込み
* 「同じ model 系列の AI 多層防御に独立性を期待しない」という前提を全プロンプトで共有
* 人間の追加要請を **構造的なステップとして組み込む**（= AI 完遂報告後に必ず人間が「もっと踏み込んだ verify できそう?」の質問を入れる）

くらいで、これは **対症療法に近い** ことを自覚しています。

### 知見を求めています

同じ問題に取り組んでいる方にぜひ伺いたいのが、以下の点です。

1. **test / audit / QA レイヤで、別 model 系列の AI レビュアーをどう組み込んでいますか?** Copilot 相当の自動レビュアーが test 設計に対しても介入する仕組みを構築できているケースはありますか?
2. **「観点の網羅度」を数値で測る運用** をしている例はありますか? コードレビューで言うラウンド数のような、運用可能な指標が test / audit 側にも欲しい
3. **AI 同士の多層防御で「同 model 由来の系統的 blindspot」を構造的に潰す** やり方はありますか? 我々は今のところ「人間の批判的圧力を最終層に組み込む」しかできていません

「自分のところはこう組んでいる」「ここはこう設計した方がいい」という知見を、コメントで教えていただけると嬉しいです。AI 業務化はまだ業界全体で型が固まっていない領域なので、外部知見を取り込みたいというのが本音です。

> **章末の蒸留**: コードレビューのような確立された型は、まだ test / audit / QA レイヤには無い。手探りの段階を率直に開示し、共創で型を作りに行く。

## 7. まとめ — 人間の役割は「実行」から「批判的圧力 + 不在を疑う」へ

前作からの 2 週間でもっとも大きかった発見は、**AI multi-agent + Copilot レビューの座組がコードレベルで確立しても、test / audit / QA 層には届いていない blindspot が残る** ということでした。

そして、その blindspot は AI を何層重ねても抜けません。同じ model 系列の AI には同じ系統的な blindspot があり、セッション独立では拭えない。**独立性は「セッション分割」ではなく「model 系列 + 観点 + 経路」の独立で稼ぐ必要がある**、というのが構造原則として残ります。

この前提に立つと、人間の役割は前々作 [Day051 記事](https://zenn.dev/fixu/articles/day051-ai-human-collaboration) で書いた「リアルタイムで AI の暴走を訂正する判断軸の保持者」から、もう一段抽象度が上がります。

> **AI 多層防御後の最終批判層 — 「不在」を疑い、「verify が広い権限経路で bypass してないか?」を構造的に問う役**

「実行」を AI に明け渡すこと自体は前作までで成立しています。今回見えてきたのは、**「実行を明け渡した後の verify を AI 任せにしない」というもう 1 段** の業務設計です。verify は「✅ PASS」を返した瞬間に終わる作業ではなく、その PASS が **どの経路で通ったか** を問うことで初めて閉じる作業です。

おそらく 1 年もしないうちに、別 model 系列のレビュアーが test / audit / QA レイヤにも自然に入ってきて、人間の批判的圧力を構造的に補完してくれるようになると思います。それまでは、**人間の批判的圧力を業務設計に組み込み続けるしかない** — それが今回の体感でした。

「自分のところはこう組んでいる」「ここはこう設計した方がいい」という知見を、コメントでぜひ教えてください。手探りの段階を共有しつつ、業界として型を作っていけたら嬉しいです。
