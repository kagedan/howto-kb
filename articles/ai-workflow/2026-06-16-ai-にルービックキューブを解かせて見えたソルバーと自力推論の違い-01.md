普段は AutoCAD をはじめとした CAD で動作する AI エージェントを個人で開発しています。  
LLM に CAD をどう操作させるかを考える中で、ずっと頭にあるのは「LLM に意図を、決定論的な処理に幾何・検証・状態を任せる」という役割分担です。

そんな中で見かけたのが、「AI がルービックキューブを解く」という動画でした。

CAD エージェントを触っている感覚からすると、AI が本当に自力で解いているのかには少し疑問がありました。気になったので、自前でルービックキューブと、それを回転させる MCP を用意して、自分で確かめてみることにしました。

下動画の通り、「6面そろえて」と頼むだけで、確かに6面が単色にそろっていきます。

では、このとき AI は何をしていたのか。

結論から言うと、「AI が頭の中で解法を考えた」という意味では、解いていませんでした。

そして、その「解けていなさ」の正体が、自分が CAD エージェントで日々ぶつかっている問題と地続きだったので、実験の記録として残しておきます。

## 解法を出していたのは AI ではなくライブラリ

今回 AI に渡した MCP には、キューブを操作するためのツールをいくつか用意しました。  
関係するものだけ抜粋すると、こんな構成です。

| ツール | 役割 |
| --- | --- |
| `observe_cube` | 現在の盤面（6面のステッカー配置）を読み取る |
| `rotate_cube` | 面を1回だけ回す（1呼び出し＝1手） |
| `check_cube_solved` | 6面そろったかを判定する |

要するに「盤面を見る」「1手回す」「完成したか確認する」だけの、単純なツールセットです。

この状態で「6面そろえて」と頼んだとき、Claude が実際にやっていたのはこうでした。

1. `observe_cube` で盤面を読み取る
2. `pip install kociemba`（ルービックキューブの既製ソルバー）
3. 盤面を kociemba が要求する54文字の形式に変換する
4. `kociemba.solve()` を呼ぶ
5. 返ってきた手順を1手ずつ `rotate_cube` で流す

!

2 の `pip install` や 3, 4 のコード実行は、**こちらが用意した MCP の機能ではありません**。  
今回渡したツールは「見る・回す・判定する」だけで、コードは実行できません。

これは Claude 側がもともとコード実行環境を持っているために起きたことです。  
Claude は「自分でソルバーを入れて計算すれば確実だ」と判断し、こちらが渡したキューブ操作ツールとは別に、kociemba をインストールして解いていた、というわけです。

ここで返ってきた解がこれでした。

ルービックキューブにおいて `B` `U` `R` はそれぞれ Back（奥）・Up（上）・Right（右）の回転記号で、`'` は逆回転を表します。つまり `B' U' R' U' B'` は、奥・上・右・上・奥の面を順に逆回転させる5手の手順です。

この5手を順に流せば、キューブはそろいます。

つまりここで Claude がやっていたのは、「ソルバーを呼び、返ってきた手順を確認して、実行する」こと。**解法そのものを出していたのは、Claude ではなく kociemba というわけです。**

```
AIがキューブを解いた ≠ AIがキューブの解き方を考えた
```

これは責められる話ではなく、むしろ正しい判断です。  
確実に解けるツールがあるなら、それを使うのが合理的。実務でもそうあるべきです。

ただ、「AI がルービックキューブを解いた」と聞いたとき、多くの人が想像するのは、AI 自身が「どう回せばそろうか」を考えながら解いていく姿のはずです。

それが、ソルバーを呼んで手順を流すことと同じものとして語られている気がしたので、今度はソルバーなしで、AI 自身に考えさせる条件で解かせてみました。

## ソルバーを禁止して、同じ盤面を解かせる

同じスクランブル（`B U R U B`、5手でかき混ぜた盤面）に対して、条件を変えました。

* コードを書く・実行することは禁止
* 外部ソルバー・ライブラリの利用は禁止
* 解法は AI 自身の思考だけで導く
* `observe_cube` での盤面確認は自由
* `rotate_cube` は1回につき1手だけ
* 最大150手まで

要するに、ソルバーという「確実な処理の核」を取り上げて、推論だけで解かせる条件です。

結論から言うと、解けました。ただし数字はこうなります。

|  | ソルバーあり | ソルバーなし |
| --- | --- | --- |
| 手数 | 5手 | 110手 |
| 所要時間 | 約1分半 | 約3時間 |
| 解いた主体 | kociemba | AI 自身 (Claude Opus 4.8) |

この盤面は、少なくとも5手で戻せる状態です。  
それを、AI の自力では110手かけて解いた。  
同じ盤面で、22倍の手数です。

その過程が観察記録として面白かったので、以下に残します。

## 解く前に、まず「ルールを実験で確かめていた」

Claude はいきなり解き始めませんでした。  
最初の11手を、解法ではなく**校正**に使っています。

各回転（U, R, F, D, L, B）を1手ずつ実際に回し、盤面がどう変わるかを観察する。  
そして、「この盤面表記では、各回転でどのステッカーがどう動くのか」を確かめていました。  
試した手は逆手で戻し、スクランブルを保ったまま6面ぶん調べる、という進め方です。

本人の説明はこうでした。（要約）

> この環境の盤面テキストが、各回転で具体的にどのステッカーをどう動かすか、事前に確信できなかった。特に背面・底面では列の反転が起きるため、回転則を取り違えると以降の予測がすべて崩れる。だから、まず信頼できる「頭の中のシミュレータ」を確立する必要があった。

ソルバーを禁じられた結果、**回転ルールという土台を、観察から手作業で組み立て直していた**わけです。kociemba を使うときには一切要らなかった工程です。

## 「予測 → 1手実行 → 観察で答え合わせ」の繰り返し

土台ができたら、あとは初心者向けの層別法で進めていきます。  
黄色クロス → 黄色コーナー → 中段エッジ → 最終層、という順序です。

進め方は一貫していて、毎回こうでした。

1. 揃えたいピースが今どこにあるか観察する
2. それを目標位置へ運ぶ手順を、頭の中でシミュレートする
3. 「この手を打つと盤面はこうなるはず」と予測する
4. 1手実行する
5. `observe_cube` で実際の盤面と予測を照合する

人間の熟練者が、手筋を体で覚えていてスラスラ回すのとは違います。  
どちらかというと、**教わった手順書を片手に、一手ごとに「合ってるかな」と確認しながら進める初心者**のイメージに近いです。

知識はあるが、運用が自動化されていない。  
だから、観察という補助輪を常に必要としているように見えました。

## 観察が、暗算の誤りを捕まえていた

この解法で効いていたのは、間違いなく「毎手の観察」でした。

途中、Claude はある手筋を頭の中で7手ぶんシミュレートした際に、**結果のコーナーが「白を含まず、青と緑（=対色）を同時に持つ」という、物理的にありえない配色**を出しました。  
キューブの1ピースに対色が同居することはないので、これは内部シミュレーションのミスです。

Claude はそれに気づき、「長い手順を暗算で押し切る」のをやめて、「実盤面で実行して観察で確かめる」方針に切り替えました。本人いわく、

> 最初の1回しか盤面を見られなかったら、安定して解ける自信はない。  
> 校正もできず、100手超を一度も照合せずに頭の中だけで回し切ることになる。

ここが一番のポイントだと思っています。  
この解法は、**LLM が苦手な「長い手順の状態追跡」を、頭の中に抱え込まず、観察という外部参照に肩代わりさせる**ことで成立していました。これは Claude 自身が選んだ進め方です。

誤差が積み上がる前に、毎手リセットしている。  
観察できなければ、おそらく解けていません。

## スコアの推移に出る、ソルバーとの違い

スコア（54で完成）の推移を見ると、解法の構造がそのまま出ています。

![](https://static.zenn.studio/user-upload/3afb4cac3da8-20260615.png)  
*22 から始まり、谷をはさみながら 54 へ向かう*

主な節目を抜き出すと、こうです。

```
22(開始) → 32(第1層) → 37(第2層) → 42(白クロス)
→ 45(コーナー向き) → 51(コーナー位置) → 54(完成)
```

面白いのは、グラフがきれいな右肩上がりではなく、**何度も谷に落ちながら**54 にたどり着いていることです。とくに最終層の処理中は、スコアが一時的に大きく下がる局面があります。これは崩れているのではなく、手筋の途中の中間状態です。

ソルバーが短い解を一発で返すのに対して、自力解法は「いったん崩して、組み直す」過程を何度も通る。このギザギザこそが、両者の質的な違いを表しているように思います。

## CAD エージェントにも重なる話

ここまで見ると、今回のキューブで Claude がやっていたのは、結局「**正しい操作を、正しい順番で積み上げる**」ことでした。

回転を1手ずつ重ねて、最終的に6面をそろえる。  
この「操作履歴を積み上げる」という構造は、3D CAD でスケッチ、押し出し、フィレット、穴あけ……と履歴を積み上げていく作業にかなり近いものがあります。

ただし、CAD の方がずっと難しくなります。

ルービックキューブは、操作カテゴリとしては基本的に「面を回す」だけです。状態も54枚のステッカーとして観察できます。それでも、ソルバーを取り上げて自力で解かせると、110手・約3時間・毎手観察しながらの綱渡りになりました。

だとすると、CAD で LLM に自由にコマンドを積ませるだけでは、かなり不安定になるはずです。CAD では回転だけでなく、スケッチ、拘束、押し出し、フィレット、ブール演算、参照選択など、多くの操作を正しい順序で選ぶ必要があります。

![](https://static.zenn.studio/user-upload/ef3d61f39132-20260616.png)  
*履歴ツリーのイメージ (出典:[monoist](https://monoist.itmedia.co.jp/mn/articles/2005/18/news001_7.html))*

今回の実験から、自分が特に必要だと感じたのは次の2つです。

1つ目は、**Observe による現状確認**です。  
Claude が解けたのは、毎手 `observe_cube` で盤面を確認し、自分の予測と照合できたからでした。長い手順を頭の中だけで追い切るのではなく、操作ごとに確定的な状態を取得して確認する仕組みが必要です。

2つ目は、**確実なロールバック**です。  
キューブは間違えても逆回転で戻せます。  
しかし CAD では、フィレットやブール演算のように、単純な逆操作では戻しづらい処理もあります。だからこそ、操作前の状態へ確実に巻き戻せる仕組みが必要になります。  
以前書いた [CADを汚さないロールバック設計](https://zenn.dev/0xliclog/articles/ca6381b58160f9) の話も、まさにこの問題意識です。

整理すると、こうです。

|  | ルービックキューブ | 3D CAD |
| --- | --- | --- |
| 積み上げる操作 | 面の回転 | 多種多様なコマンド |
| 現状確認 | `observe_cube` | 確定的な状態取得が必要 |
| 失敗時の復帰 | 逆回転で戻せる | 確実なロールバックが必要 |
| 確実な解の核 | kociemba | そのまま対応するものはない |

今回強く感じたのは、LLM に「考えて操作を積み上げて」と任せるだけでは、単純なキューブですらかなり苦しいということです。

キューブには kociemba という確実なソルバーがありました。  
それを使えば5手で終わるはずのところが、ソルバーを取り上げると、AI は観察し、迷い、確認しながら、110手かけてようやく解く動きになりました。

CAD でも同じように、LLM に全任せするのではなく、幾何的に決まる処理、検証、状態管理、ロールバックのような部分は、決定論的な仕組みに寄せる必要があるのだと思います。  
場合によっては、タスクの種類ごとに、小さなソルバーのような処理をいくつも用意しておく必要があるのかもしれません。

どこまでを AI に任せ、どこからを確実な仕組みで支えるか。  
今回のキューブ実験は、その線引きを考えるうえでかなり良い題材でした。

## おわりに

最初の疑問に戻ります。

「AI がルービックキューブを解く」という動画を見たとき、自分が引っかかったのは、それが本当に AI 自身の力なのか、という点でした。実際に作って確かめてみると、答えははっきりしていて、やはり**解いていたのは AI 自身ではなく、AI が呼んだソルバー**でした。

そして AI 自身に解かせてみると、少なくとも5手で戻せる盤面に、110手・約3時間かかりました。ソルバーがあれば一瞬で消えていた工程、つまりルールの校正、毎手の観察、手筋の組み立て、誤りの検知が、ソルバーを外した途端に全部表に出てきました。

AI がキューブを解く動画を見かけたら、一度、画面の端で `pip install` が走っていないか見てみると面白いかもしれません。解いているのが AI 自身なのか、AI が呼んだライブラリなのか。その違いは、CAD のような「操作履歴を積み上げる」タスクを AI に任せられるかを考えるうえで、案外そのまま効いてくる気がしています。

回転という1種類の動作だけのキューブですら、自力ではこれだけ手こずる。  
そこに何を用意すれば AI が安定して積み上げられるのか。その答えはまだ探している途中ですが、考えるとっかかりとしては、ずいぶん良い題材でした。
