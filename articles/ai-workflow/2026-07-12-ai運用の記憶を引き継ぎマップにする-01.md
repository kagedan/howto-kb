# AI運用の記憶を引き継ぎマップにする

![Operational Memory Handoff Map](https://static.zenn.studio/user-upload/deployed-images/e38958604b432c68d82e8ee7.png?sha=c317e2f9fffef675a6b4c571f605e57469d09b3e)

AIを問い合わせ対応、初回相談、CRM更新、FAQ候補作成に使い始めると、運用の記録は少しずつ増えていきます。

ただ、記録が増えても、次の担当者やAIエージェントへ引き継げる形になっていなければ、同じ確認が繰り返されます。

この記事では、AI運用で残すべき記憶を **引き継ぎマップ** として整理する方法をまとめます。Miraigentでは、AI導入前の相談でも、ツール選定より先に「何を残せば次の判断が軽くなるか」を確認します。

## 想定する読者

* AI返信下書きやAI要約を小さく始めたチーム
* 担当者が変わるたびに、同じ確認や説明が発生している人
* AIエージェントに業務を渡す前に、引き継ぎ情報を整理したい人
* CRM、Notion、Googleスプレッドシート、GitHub READMEなどに運用ルールを残したい人

ここでは、AI APIの呼び出しやプロンプト改善ではなく、AIに渡す前後の業務記憶をどう設計するかに絞ります。

## 完成形

作るものは、次の3つです。

| 作るもの | 役割 |
| --- | --- |
| memory\_handoff\_map | AI運用で引き継ぐ7項目を固定する |
| memory\_record | 1件ごとの判断、例外、改善先を残す |
| review\_checklist | AIへ渡す前に、人間が見る条件を固定する |

最初から大きなナレッジベースを作る必要はありません。

まずは、次の7項目を1枚の表にします。

```
目的
  ↓
入力データ
  ↓
AIに任せる作業
  ↓
人間確認
  ↓
例外ケース
  ↓
判断ログ
  ↓
改善バックログ
```

この流れがあると、AIに何を任せるかだけでなく、止まった時にどこへ戻すかまで共有できます。

## なぜ引き継ぎマップが必要か

AI運用で詰まりやすいのは、モデルの性能だけではありません。

次のような情報が残っていない時に、確認作業が増えます。

| 残っていない情報 | 起きること |
| --- | --- |
| 目的 | AIの出力が何を満たせばよいか分からない |
| 入力データ | どの情報を参照してよいか分からない |
| 人間確認 | どこで止めるべきか分からない |
| 例外ケース | 毎回その場で判断する |
| 判断ログ | 前回なぜ止めたか分からない |
| 改善先 | FAQ、CRM、フォームへ戻らない |

引き継ぎマップは、AIを賢くするためだけの資料ではありません。

新しい担当者が読んでも、AIエージェントが参照しても、同じ運用境界を理解できるようにするための設計です。

## 7項目の設計

最小構成は、次の表です。

| 項目 | 書くこと | 例 |
| --- | --- | --- |
| purpose | 何のための運用か | 初回相談の返信下書きを早くする |
| input\_data | AIが参照してよい情報 | マスク済み相談要約、公開FAQ |
| ai\_task | AIに任せる作業 | 要約、分類、返信案の作成 |
| human\_review | 人間が見る条件 | 契約、料金、個人情報、苦情 |
| exception\_case | 通常処理から外す条件 | 返金、法務、権限不足 |
| decision\_log | 判断理由の残し方 | hold\_reason、owner、next\_action |
| improvement\_backlog | 改善へ戻す先 | FAQ、フォーム、CRM、承認ルール |

この7項目は、業務ごとに1枚ずつ作ります。

「問い合わせ対応」「無料診断」「営業メモ」「FAQ更新」のように、入口や確認者が違うものは分けた方が扱いやすいです。

## YAMLで管理する例

GitHub READMEやNotionに説明を書く前に、設定として読める形にしておくと運用しやすくなります。

たとえば、問い合わせ返信下書きなら次のように書けます。

```
workflow_id: inquiry_reply_draft
owner: support_owner
purpose: "問い合わせ返信の下書きを作り、送信前に人間が確認する"

input_data:
  allowed:
    - masked_inquiry_summary
    - public_faq
    - service_page_url
  blocked:
    - phone_number
    - contract_file
    - payment_detail
    - private_customer_note

ai_task:
  allowed:
    - summarize
    - classify
    - draft_reply
    - suggest_faq_candidate
  blocked:
    - send_to_customer
    - decide_refund
    - change_contract_status

human_review:
  required_when:
    - contains_personal_data
    - contains_contract_or_refund
    - contains_complaint
    - confidence_low
    - customer_status_unknown

decision_log:
  fields:
    - request_id
    - decision_type
    - status
    - owner
    - reason
    - next_action
    - improvement_target

improvement_backlog:
  targets:
    - faq
    - form
    - crm
    - approval_rule
    - reply_template
```

これはAIにそのまま渡すための秘密資料ではありません。

AIへ渡してよい情報と、人間だけが見る情報を分けるための運用設定です。

## memory\_record の項目

1件ごとの記録は、長文で残しすぎない方が続きます。

次のような項目で十分です。

| 項目 | 例 | 目的 |
| --- | --- | --- |
| record\_id | MEM-20260709-001 | 記録を一意に追う |
| workflow\_id | inquiry\_reply\_draft | どの業務か分ける |
| source\_id | INQ-20260709-018 | 元の問い合わせへ戻れるようにする |
| ai\_action | draft\_reply | AIに任せた作業 |
| status | hold | completed / hold / rejected |
| review\_reason | contains\_contract\_or\_refund | 人間確認の理由 |
| decision | owner\_review | 最終判断 |
| next\_action | 責任者確認へ戻す | 次にやること |
| improvement\_target | approval\_rule | どこを直すか |

ここに、顧客名、電話番号、契約書全文、社内メモ全文を入れる必要はありません。

引き継ぎに必要なのは、個人情報そのものではなく、次の担当者が同じ判断にたどり着くための理由です。

## スプレッドシートで始める列

最初は、専用システムよりもスプレッドシートで十分です。

| 列 | 入力例 |
| --- | --- |
| record\_id | MEM-20260709-001 |
| date | 2026-07-09 |
| workflow\_id | inquiry\_reply\_draft |
| source\_id | INQ-018 |
| ai\_action | draft\_reply |
| status | hold |
| review\_reason | contains\_contract\_or\_refund |
| owner | support\_owner |
| decision | owner\_review |
| next\_action | 返金条件を責任者確認 |
| improvement\_target | approval\_rule |
| note | 料金FAQの表現も見直す |

ポイントは、`review_reason` と `improvement_target` を選択式にすることです。

自由記述だけにすると、週次で集計しにくくなります。

## AIへ渡す時の最小コンテキスト

AIエージェントに渡す情報は、できるだけ絞ります。

悪い例は、過去の問い合わせ本文や社内メモをまとめて渡すことです。

良い例は、現在の状態と許可された行動だけを渡すことです。

```
{
  "workflow_id": "inquiry_reply_draft",
  "allowed_ai_actions": ["summarize", "classify", "draft_reply"],
  "blocked_ai_actions": ["send_to_customer", "decide_refund"],
  "allowed_inputs": ["masked_inquiry_summary", "public_faq"],
  "review_required_when": [
    "contains_personal_data",
    "contains_contract_or_refund",
    "contains_complaint"
  ],
  "current_record": {
    "source_id": "INQ-018",
    "status": "pending",
    "review_reason": "contains_contract_or_refund",
    "allowed_next_action": "draft_for_owner_review"
  }
}
```

この形なら、AIは次のことを判断しやすくなります。

* 顧客へ送信しない
* 返信案だけ作る
* 返金判断はしない
* 人間確認が必要な理由を残す
* FAQや承認ルールの改善候補に回す

AIに「安全にやって」と言うだけでは、業務境界は固定されません。

許可する行動、禁止する行動、確認条件をデータとして渡すことが重要です。

## 人間が確認する境界

引き継ぎマップを作っても、次の判断は人間が確認します。

| 判断 | AIの役割 | 人間の役割 |
| --- | --- | --- |
| 返信文の下書き | 候補を作る | 送信可否を決める |
| FAQ候補 | 重複や候補を整理する | 公開表現を決める |
| CRM更新候補 | 入力案を作る | 顧客状態の変更を承認する |
| 例外ケース | 条件に当てはまる可能性を示す | 対応方針を決める |
| 改善バックログ | 集計候補を出す | 優先度と担当者を決める |

特に、契約、返金、請求、法務、個人情報、顧客への最終送信は、AIだけで決めない方が安全です。

## 週次レビューの手順

引き継ぎマップは、作って終わりではありません。

週次で次の順に見直します。

1. `status = hold` の件数を見る
2. `review_reason` が多い順に並べる
3. 同じ理由が2件以上あるものを選ぶ
4. `improvement_target` を確認する
5. FAQ、フォーム、CRM、承認ルール、テンプレートのどれを直すか決める
6. 翌週に同じ `review_reason` が減ったか見る

この運用にすると、AIの失敗を単なるエラーではなく、業務側を直す材料として扱えます。

## 導入前チェックリスト

AIエージェントへ業務を渡す前に、次を確認します。

* 業務ごとの `workflow_id` がある
* 目的が1文で書かれている
* AIへ渡してよい情報が決まっている
* AIへ渡さない情報が決まっている
* AIに許可する行動が決まっている
* AIに禁止する行動が決まっている
* 人間確認が必要な条件がある
* 例外ケースの戻し先がある
* 判断ログの項目がある
* 改善バックログの戻し先がある

1つでも曖昧なら、AI連携の前に運用メモを整えた方がよいです。

止めるためではなく、任せてよい範囲を広げるためです。

## まとめ

AI運用の記憶は、ただ保存するだけでは引き継げません。

目的、入力データ、AIの作業、人間確認、例外ケース、判断ログ、改善バックログをつなげると、次の担当者やAIエージェントが参照できる運用資産になります。

最初の一歩は、1つの業務だけで構いません。

問い合わせ返信、初回相談、FAQ更新、CRMメモのどれかを選び、`workflow_id` と7項目の引き継ぎマップを作ってみてください。

無料診断でも、Miraigentは「AIに何を任せるか」だけでなく、「何を残せば次の判断が軽くなるか」を確認します。
