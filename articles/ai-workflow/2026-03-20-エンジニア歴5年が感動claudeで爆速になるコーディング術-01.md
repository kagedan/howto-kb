## 「また同じような実装、書いてる…」そう感じたことはありませんか？

エンジニアとして5年ほど働いてきて、気づいたことがあります。  
日々のコーディング業務の中で、**意外と多くの時間が「思い出す作業」や「定型的な実装」に費やされている**ということです。

* ライブラリのAPIの書き方、またググった
* バリデーション処理、毎回似たようなコード書いてる
* エラーメッセージの意味、調べるのに時間かかった
* コードレビューのコメント、どう修正すればいいかわからない

こういった「ちょっとした詰まり」が積み重なって、気づけば1日があっという間に終わっている——そんな経験、ありませんか？

私もずっとそうでした。でも、AIアシスタント「Claude」を本格的に活用するようになってから、その状況が大きく変わりました。

この記事では、**エンジニア歴5年の私が実際に使って効果を実感したClaudeの活用術**を、具体的なプロンプト例とともにご紹介します。読み終えた後には、明日から即実践できるヒントが手に入るはずです。

## この記事を読むメリット

* Claudeをコーディングに活かす具体的なプロンプト術がわかる
* 「なんとなく使っている」から「意図的に使いこなす」レベルに上がれる
* 作業時間の短縮だけでなく、コードの品質向上にもつながる考え方が身につく

## Claudeって結局、何が得意なの？

ChatGPTと同様のLLM（大規模言語モデル）ベースのAIですが、Claudeには**長いコンテキストを正確に扱う力**と**丁寧な説明を返してくれる傾向**があると私は感じています。

特にコーディング場面では以下のような用途で力を発揮します。

| 用途 | 具体例 |
| --- | --- |
| コード生成 | 関数・クラス・APIエンドポイントの雛形作成 |
| コードレビュー | バグの指摘・改善提案・可読性の向上 |
| デバッグ支援 | エラーメッセージの解析・原因特定 |
| リファクタリング | 重複排除・命名改善・設計見直し |
| ドキュメント作成 | コメント生成・README・API仕様書 |
| 学習サポート | 概念の説明・サンプルコードの解説 |

どれもエンジニアの日常業務に直結しているのが、嬉しいところです。

## 実践！Claudeを使いこなす6つのシーン

### 1. コード生成：「お題」を渡すだけで雛形ができる

最もオーソドックスな使い方ですが、**プロンプトの精度でアウトプットの質が大きく変わります。**

❌ 曖昧なプロンプト

✅ 具体的なプロンプト

```
TypeScriptで、以下の条件を満たすユーザー登録関数を作ってください。
- 引数: name(string), email(string), password(string)
- メールアドレスの形式バリデーションを含む
- パスワードは8文字以上、英数字混在
- 戻り値: { success: boolean, message: string }
- エラー処理も含めること
```

後者のように「型・条件・戻り値・エラー処理」を明示すると、そのままプロダクションに近い品質のコードが返ってきます。

### 2. デバッグ：エラーメッセージをそのまま貼る

エラーが出たとき、私がよくやるのは**エラーメッセージ＋該当コード＋状況説明をまとめてClaudeに貼る**方法です。

```
以下のエラーが発生しています。原因と修正方法を教えてください。

【エラーメッセージ】
TypeError: Cannot read properties of undefined (reading 'map')

【該当コード】
const List = ({ items }) => {
  return items.map(item => <div key={item.id}>{item.name}</div>);
};

【状況】
APIからデータを取得して表示しようとしていますが、
初回レンダリング時にエラーになります。
```

これだけで「itemsがundefinedの状態でmapが呼ばれている」という原因と、デフォルト値やオプショナルチェーンを使った修正案まで提示してくれます。StackOverflowを何分もさまよわなくて済むのは、正直かなり助かっています。

### 3. コードレビュー：自分のコードを客観視できる

プルリクを出す前に、Claudeに一次レビューをお願いするのもおすすめです。

```
以下のコードをレビューしてください。
観点：可読性・パフォーマンス・セキュリティ・命名

（コードを貼り付け）
```

チームのレビュアーに出す前に「基本的な指摘」を潰しておけるので、レビューの質が上がり、お互いの時間を節約できます。私はこれを習慣にしてから、レビューでの指摘数が体感で半分以下になりました。

### 4. リファクタリング：「なんかイケてない」を言語化してもらう

自分で書いたコードが「なんかモヤっとする」けど何が問題かわからない——そんなときにも使えます。

```
以下のコードをリファクタリングしてください。
現在の問題点も合わせて説明してください。

（コードを貼り付け）
```

問題点の説明がついてくるので、リファクタリング後のコードを読むだけでなく、**自分のコーディング力の向上にもつながる**と感じています。

### 5. ドキュメント生成：コメントもお任せ

コードを書いた後のドキュメント作業は、正直しんどいですよね。JSDocやTypeDocのコメント生成もClaudeに任せられます。

```
以下の関数にJSDocコメントを追加してください。
引数・戻り値・使用例を含めてください。

（関数コードを貼り付け）
```

ドキュメントを書くハードルが下がると、チーム全体のコードの読みやすさが向上します。これは地味ながらも、長期的に大きな効果があると思います。

### 6. 技術学習：「わかったつもり」を解消する

新しい技術を学ぶときにも、Claudeは頼りになります。

```
ReactのuseCallbackとuseMemoの違いを、
具体的なコード例を使って説明してください。
初心者でもわかるようにお願いします。
```

公式ドキュメントより砕けた説明と、すぐ試せるサンプルコードが返ってくることが多いです。「なんとなく使ってたけど、ちゃんと理解できた」という体験を何度もしています。

## 効果を実感するためのコツまとめ

| コツ | 内容 |
| --- | --- |
| 具体的に書く | 型・条件・形式を明示すると精度が上がる |
| 背景を伝える | 「〜の目的で使う」という文脈があると的確な回答が返る |
| 一度で終わらせない | 出力を見てさらに「〜を修正して」と続けるのが効果的 |
| コードを丸ごと貼る | 断片より全体のほうが精度が上がる |
| 複数観点を指定する | 「可読性・パフォーマンス両面で」など観点を絞る |

---

## 注意点：過信しすぎないことが大切

Claudeは非常に便利ですが、**出力されたコードを無条件に信用するのは危険**です。

* 最新ライブラリの仕様に追いついていないことがある
* セキュリティ面での考慮が不十分なケースがある
* ロジックが一見正しそうでも、エッジケースで壊れることがある

「最終的な判断は自分でする」というスタンスを忘れずに。あくまで**思考のパートナー・ドラフト生成ツール**として使うのが、賢い付き合い方だと思っています。
