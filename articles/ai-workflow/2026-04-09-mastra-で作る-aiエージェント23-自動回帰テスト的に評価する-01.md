[Mastra で作るAI エージェント](https://zenn.dev/shiromizuj/articles/a0a1659e9f05b6) というシリーズの第23回です。

---

前回までで、AIエージェントの「テスト」として「Eval（評価）」があること、その評価にはいろいろな評価観点と評価手法があること、いくつかの評価観点はMastraがプリセットで持っていること、独自にカスタマイズもできること、をご紹介しました

そして前回までの例では、エージェント本体に「評価」を組み込むことで、「本番実行時に」サンプリングしながら評価・観測をすることができる、という点をお話ししました。

一方で、いわゆる xUnit的、 Vitest的に自動回帰テスト、もとい自動回帰評価をすることも可能です。開発しながら評価、修正しPullRequestする前に評価、という感じで運用していきます。

# `runEvals + groundTruth` と `Datasets/Experiments`

自動で回帰的な評価をするに方法として、Mastraでは `runEvals + groundTruth` というアプローチと `Datasets/Experiments` というアプローチの２つの方法論が存在します。

私は、次の使い分けで理解しています。

* `runEvals + groundTruth`: 「その場で採点する小テスト」
  + **データ管理**: 配列をコード直書きすることが多い
  + **判定方法**: 絶対閾値判定（例: `score >= 0.8`）
  + **向いている規模**: PoC / 小規模 / 単独開発
  + テストコード内のデータをそのまま評価できる
  + 設定が少なく、導入が早い
  + 「まず品質ゲートを作る」用途に向く
* `Datasets/Experiments`: 「問題集の管理 + 実験履歴 + 差分比較まで行う評価基盤」
  + **データ管理**: JSONLやDatasetに切り出し、再利用しやすい
  + **判定方法**: ベースライン比較（例: `-0.03` 以上悪化で失敗）
  + **向いている規模**: 中〜大規模 / 複数人 / 長期運用
  + 評価データをデータセットとして独立管理できる
  + 実験履歴を残して、過去実験との差分比較ができる
  + 「継続改善・チーム運用・回帰分析」に向く

まずは `runEvals` で始め、評価運用が育ってきたら `Datasets/Experiments` に拡張するのが実践的なのではないでしょうか。

# runEvals + groundTruth + scorer を使う場合（基本）

## 推奨フォルダ構成（runEvals + groundTruth + scorer + Vitest）

`runEvals` を中心に回帰テストを作る場合は、

* 評価対象エージェント
* 評価ケース（input + groundTruth）
* scorer定義
* Vitestテストコード

を最小構成で分離すると追いやすくなります。

```
src/
  mastra/
    agents/
      weather-agent.ts                  # テスト対象エージェント
    evals/
      cases/
        weather-eval.cases.ts           # 「入力＋期待回答」の評価データ
      scorers/
        weather-eval.scorers.ts         # scorer定義

__tests__/
  mastra/
    agents/
      weather-agent.eval.test.ts        # runEvals の回帰テスト
    setup/
      vitest.setup.ts                   # 共通初期化（任意）
```

## ファイル別サンプルコード（構成図に対応）

以下は、上のフォルダ構成をそのまま実装するための最小サンプルです。  
（説明を優先し、テスト網羅性は最小限にしています）

### `src/mastra/agents/weather-agent.ts`

まずこちらが、今回の評価対象とするエージェントです。

```
import { Agent } from "@mastra/core/agent";

/**
 * 天気問い合わせを受けるエージェント。
 *
 * @remarks
 * Datasets/Experiments の `targetId` から参照できるよう、
 * `id` を固定で定義しておく。
 */
export const weatherAgent = new Agent({
  id: "weather-agent",
  name: "Weather Agent",
  instructions:
    "ユーザーの質問から都市名を解釈し、簡潔で一貫した天気回答を返してください。",
  model: "openai/gpt-5.4-nano",
});
```

### `src/mastra/evals/cases/weather-eval.cases.ts`

`runEvals` に渡す固定評価ケースです。`groundTruth` を明示することで、毎回同じ条件で比較できます。

```
/**
 * runEvals で利用する最小評価ケース。
 *
 * @remarks
 * 初期は 3〜10 件程度で始め、失敗事例を継続追加する。
 */
export const weatherEvalCases = [
  { input: "Berlin の現在の天気は？", groundTruth: "Berlin, DE" },
  { input: "Berlin, Maryland の天気は？", groundTruth: "Berlin, US" },
];
```

### `src/mastra/evals/scorers/weather-eval.scorers.ts`

採点基準（scorer）を定義します。テストコードへ直書きせず、別ファイルにすると再利用しやすくなります。

今回の例では、「模範回答と似たような回答をしているか」という評価をしたいので、`answerSimilarityScorer` を作成し、`runEvals` 側でそのまま渡せる scorer オブジェクトを提供しています。

```
import { createAnswerSimilarityScorer } from "@mastra/evals/scorers/prebuilt";

/**
 * 回答の類似度を評価する scorer。
 */
export const answerSimilarityScorer = createAnswerSimilarityScorer({
  model: "openai/gpt-5.4-nano",
});
```

### `__tests__/mastra/setup/vitest.setup.ts`

テスト全体で共通化したい初期設定を置く場所です（任意）。

```
/**
 * Vitest の共通初期化。
 */
```

### `__tests__/mastra/agents/weather-agent.eval.test.ts`

`runEvals` を直接使う最小の回帰テストです。絶対閾値で品質ゲートを作るのに向いています。

```
import { describe, expect, it } from "vitest";
import { runEvals } from "@mastra/core/evals";
import { weatherAgent } from "../../../src/mastra/agents/weather-agent";
import { weatherEvalCases } from "../../../src/mastra/evals/cases/weather-eval.cases";
import { answerSimilarityScorer } from "../../../src/mastra/evals/scorers/weather-eval.scorers";

/**
 * runEvals + groundTruth の最小回帰テスト。
 */
describe("weather-agent runEvals", () => {
  it("answer-similarity が最低ラインを下回らない", async () => {
    const result = await runEvals({
      data: weatherEvalCases,
      target: weatherAgent,
      scorers: [answerSimilarityScorer],
    });

    expect(result.summary.totalItems).toBe(weatherEvalCases.length);
    expect(result.scores["answer-similarity-scorer"]).toBeGreaterThanOrEqual(0.7);
  });
});
```

### この基本構成で押さえるポイント

* `runEvals` は「まず品質ゲートを作る」用途に最適
* 評価ケースと scorer を分離すると、変更影響を追いやすい
* ケース数が増えたら次章の `Datasets/Experiments` に段階移行しやすい

# Datasets/Experiments を使う場合（発展）

## 推奨フォルダ構成（Datasets/Experiments + Vitest）

`Datasets/Experiments` を使う場合は、

* 評価対象エージェント（本体）
* 評価データセット定義
* scorer定義
* 実験起動ユーティリティ
* Vitest（回帰テスト）

を分離して置くと運用しやすくなります。

以下は、テスト対象に `weather-agent` を含めた全体イメージです。

scorer が「採点基準」、experiment が「実行オーケストレーション」、このファイルは「問題集（評価データ）の管理」、というように責務を分離しています。

```
src/
  mastra/
    agents/
      weather-agent.ts                  # テスト対象エージェント
    evals/
      datasets/
        weather-eval.dataset.ts         # 評価用データ（≒問題集）の管理
      scorers/
        weather-eval.scorers.ts         # 採点基準
      experiments/
        weather-experiment.ts           # 実行オーケストレーション
      fixtures/
        weather-eval-cases.jsonl        # 具体的な評価用データ
      index.ts

__tests__/
  mastra/
    agents/
      weather-agent.eval.test.ts          # runEvalsでの回帰テスト
      weather-agent.dataset.test.ts       # Dataset起点の実験テスト
    scorers/
      weather-scorer.test.ts              # カスタムscorerがある場合
    setup/
      vitest.setup.ts                     # 共通初期化（env/mock等）

.mastra/
  mastra.db                               # LibSQLStore（ローカル実験履歴）
```

補足:

* `__tests__/mastra/agents/` は `src/mastra/agents/` と対応し、ガイドとして意図が伝わりやすい配置
* データ（`datasets` / `fixtures`）と採点（`scorers`）を分けると、失敗時の切り分けが容易
* 既存の `tests/*.test.ts` を残したまま、評価系だけ `__tests__/mastra/` へ段階移行しても問題ありません

## ファイル別サンプルコード（構成図に対応）

以下は、上のフォルダ構成をそのまま実装するための最小サンプルです。  
（説明を優先し、テスト網羅性は最小限にしています）

### `src/mastra/agents/weather-agent.ts`

まずこちらが、今回の評価対象とするエージェントです。

```
  instructions:
```

### `src/mastra/evals/fixtures/weather-eval-cases.jsonl`

こちらは、テストケース、というか評価ケースとしてのJSONLファイルです。入力値と期待回答例のセットがいくつか定義しているものです。

```
{"id":"wx-001","input":{"query":"Berlin の現在の天気は？"},"groundTruth":{"city":"Berlin","country":"DE"},"metadata":{"category":"normal"}}
{"id":"wx-002","input":{"query":"Berlin, Maryland の天気は？"},"groundTruth":{"city":"Berlin","country":"US"},"metadata":{"category":"disambiguation"}}
```

### `src/mastra/evals/datasets/weather-eval.dataset.ts`

「評価データを用意する層」です。このサンプルコードでは、上記で用意したJSONLファイルを読み込んで `seedWeatherEvalItems` に流しています。

```
import type { Mastra } from "@mastra/core";
import { readFile } from "node:fs/promises";
import { resolve, dirname } from "node:path";
import { fileURLToPath } from "node:url";

type WeatherEvalItem = {
  input: { query: string };
  groundTruth: { city: string; country: string };
  metadata?: { category?: string };
};

const THIS_DIR = dirname(fileURLToPath(import.meta.url));
const FIXTURE_PATH = resolve(THIS_DIR, "../fixtures/weather-eval-cases.jsonl");

/**
 * weather 評価用データセットの名前。
 *
 * @description
 * 実運用では環境ごとに suffix を付ける（例: `-dev` / `-prod`）と管理しやすい。
 */
export const WEATHER_EVAL_DATASET_NAME = "weather-eval-set";

/**
 * JSONL ファイルから評価ケースを読み込み、配列に変換する。
 *
 * @returns Dataset に投入可能な評価ケース配列
 */
async function loadWeatherEvalItemsFromJsonl(): Promise<WeatherEvalItem[]> {
  const raw = await readFile(FIXTURE_PATH, "utf-8");

  return raw
    .split("\n")
    .map((line) => line.trim())
    .filter((line) => line.length > 0)
    .map((line) => JSON.parse(line) as WeatherEvalItem);
}

/**
 * JSONL から読み込んだ評価ケースをデータセットに投入する。
 *
 * @param dataset - Mastra の dataset インスタンス
 */
export async function seedWeatherEvalItems(dataset: any) {
  const items = await loadWeatherEvalItemsFromJsonl();

  await dataset.addItems({
    items,
  });
}

/**
 * 既存データセットを取得し、なければ作成して返す。
 *
 * @param mastra - Mastra アプリケーションインスタンス
 */
export async function getOrCreateWeatherEvalDataset(mastra: Mastra) {
  const dataset = await mastra.datasets.create({
    name: WEATHER_EVAL_DATASET_NAME,
    description: "天気回答の回帰評価用データセット",
  });

  return dataset;
}
```

### `src/mastra/evals/scorers/weather-eval.scorers.ts`

評価に使う「採点基準」を定義・集約するファイルです。

前半は「runEvals + groundTruth + scorer を使う場合（基本）」で作った `weather-eval.scorers.ts` から変えていません。

後半に、`WEATHER_SCORER_IDS` を定義し、Dataset/Experiment 側で `scorerIds` として渡せる ID 一覧も提供しています。これによって「同じ評価軸」を、`runEvals` と `Dataset/Experiment` の両方で再利用できる形にしています。

```
import { createAnswerSimilarityScorer } from "@mastra/evals/scorers/prebuilt";

/**
 * 回答の類似度を測る標準 scorer。
 *
 * @remarks
 * 複数 scorer を使う場合は、このファイルで配列として集約する。
 */
export const answerSimilarityScorer = createAnswerSimilarityScorer({
  model: "openai/gpt-5.4-nano",
});

/**
 * 実験で利用する scorer ID 一覧。
 *
 * @description
 * Dataset Experiment API では scorer オブジェクトではなく ID を渡すことがあるため、
 * ここで管理しておく。
 */
export const WEATHER_SCORER_IDS = ["answer-similarity"];
```

### `src/mastra/evals/experiments/weather-experiment.ts`

評価処理の「実行オーケストレーション層」です。このファイル単体では採点ロジックや評価データの中身は持たず、実験を回す手順をまとめます。

```
import { Mastra } from "@mastra/core";
import { LibSQLStore } from "@mastra/libsql";
import { WEATHER_SCORER_IDS } from "../scorers/weather-eval.scorers";
import {
  getOrCreateWeatherEvalDataset,
  seedWeatherEvalItems,
} from "../datasets/weather-eval.dataset";

/**
 * 実験実行に使用する Mastra インスタンスを生成する。
 */
export function createEvalMastra() {
  return new Mastra({
    storage: new LibSQLStore({
      id: "mastra-storage",
      url: "file:.mastra/mastra.db",
    }),
  });
}

/**
 * weather-agent に対するデータセット実験を開始する。
 *
 * @returns 実験サマリー（experimentId を含む）
 */
export async function runWeatherDatasetExperiment() {
  const mastra = createEvalMastra();
  const dataset = await getOrCreateWeatherEvalDataset(mastra);

  await seedWeatherEvalItems(dataset);

  return dataset.startExperiment({
    targetType: "agent",
    targetId: "weather-agent",
    scorerIds: WEATHER_SCORER_IDS,
  });
}

/**
 * ベースライン実験と比較し、スコア差分を取得する。
 *
 * @param baselineId - 比較元となる実験ID
 * @param currentId - 今回実行した実験ID
 */
export async function compareWithBaseline(baselineId: string, currentId: string) {
  const mastra = createEvalMastra();

  return mastra.datasets.compareExperiments({
    experimentIds: [baselineId, currentId],
    baselineId,
  });
}
```

### `src/mastra/evals/index.ts`

Eval機能の\*\*公開窓口（バレルファイル）\*\*です。

役割は `export * from ...` で `datasets` / `scorers` / `experiments` を再公開し、呼び出し側の import 先を1本化することです。これによりテストやスクリプト側は深い相対パスを辿らず、evals 入口だけ見れば必要APIを把握できます。 `datasets` / `scorers` / `experiments` が実処理で、`index.ts` はそれらをまとめるAPIファサードだ、と位置付けることができます。

```
/**
 * Eval 関連ユーティリティの公開エントリ。
 *
 * @description

export * from "./datasets/weather-eval.dataset";
export * from "./scorers/weather-eval.scorers";
```


### `__tests__/mastra/agents/weather-agent.dataset.test.ts`

```
describe("weather-agent dataset experiment", () => {
```
