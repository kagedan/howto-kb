**気づいたら、エンジニアはコーディングをしなくなっていた。**

Anthropic は、競合他社が四半期ごとに機能を出す横で、**週単位でサービスをリリース**し続けている。  
2026年に入ってからのわずか数ヶ月で、すでに **74 のリリース**（機能・改善含む）が世に出た。  
もはやキャッチアップが追いつかないほどの速度だ。

（参考：Product CompassのClaude Shipping Calendarより）

そしてその恩恵は、AI チャットやコンテンツ生成にとどまらない。  
いま最も大きな変化が起きているのは、**ソフトウェアエンジニアリングの現場**だ。

設計図を描く、インフラを構築する、テストを書く、コードをレビューする、本番にデプロイする——  
これまで「エンジニアが手を動かす」ことが前提だった工程に、AI が入り込んできている。  
しかも「補助ツール」としてではなく、**主体的に動くエージェント**として。

この記事では、2026年4月現在における各開発工程の自動化の実態を、現場目線の具体例を交えて解説する。

## アーキテクチャ設計

システム全体をどう構成するかを決める「設計図づくり」の工程だ。  
非エンジニアの方に例えると、家を建てる前の「間取り図と構造計算書」を作るような作業。従来は経験豊富なシニアエンジニアが何日もかけて描いていた。

**Claude Code** に「こういうユーザー体験を実現したい」「この規模のトラフィックを捌く」といった要件を自然言語で伝えるだけで、**構成案＋Draw.io互換の図面**を自動生成してくれる。  
「Claude Code + Agent Skillsで、要件を投げると5分で高品質なアーキテクチャ図＋説明文が出力される」との実践報告が相次いでいる。※本記事のアーキテクチャ図はClaude とDraw.ioで出力している。draw.ioでダウンロードできるため自由に修正が可能だ。

**メリット**：非エンジニアでも図を見ながら「ここをこう変えたい」と議論できる。  
**まだ人間が必要な部分**：AIが出す案は技術的には正しいが、「このプロダクトの事業文脈（スケジュール・予算・チームスキル）に本当に合っているか」の最終判断。そこは人間の経験と事業理解が決定的に重要だ。

## IaC（インフラのコード化）

サーバー、データベース、ネットワークなどのインフラ環境を、手作業ではなく**コードで定義・管理**する手法。  
「誰かの手元でしか動かない環境」という属人化を防ぎ、誰でも同じ環境を再現できるのが最大の強みだ。

Claude Code に「このマイクロサービス用のAWS環境をTerraformで作って」と伝えると、**Terraformの設定ファイル一式**を瞬時に書き起こしてくれる。  
2026年の現場では「Claude Codeで生成したTerraformをGitでレビュー→適用」という流れが標準化しつつある。

**非エンジニア視点**：レシピ本に材料と手順を全部書いておくようなもの。再現性が高く、コスト見積もりも自動で出せる。  
**難しい点**：コスト最適化や既存システムとの連携はAIだけでは完璧にできない。自動生成されたコードをそのまま本番適用する前に、人間によるレビューが必須。

## CI/CD（自動ビルド・テスト実行の仕組みづくり）

コードが変更されるたびに自動でビルド・テストを実行し、問題がないかを確認する「自動化ライン」。  
工場で「部品が来たら即検査→組み立て→出荷」まで全自動にするイメージ。

**Claude Code Action**（GitHub Actions連携）やHeadless Modeを使うと、パイプライン設定ファイル（YAML）自体をAIが自動生成・更新してくれる。  
「Claude CodeをCI/CDに組み込み、プッシュした瞬間にテスト・セキュリティチェックまで完結」との事例がある。

**現場の変化**：人間が手作業で確認していた「動くかどうか」の大部分を機械が肩代わり。リリース頻度が劇的に上がる。  
**まだ人間が必要**：テストが薄いプロジェクトだとCIは通っても品質が担保されない。パイプラインと並行して「テスト設計」も強化する必要がある。

## テスト設計・テスト実装

バグを早期に発見するための「動作確認コード」を書く工程。  
非エンジニアには「ソフトウェアの健康診断書を作る作業」と説明できる。

Claude Code はGitHub上の差分（diff）やIssue内容を読み取り、**変更に対応したユニットテスト・統合テストを網羅的に自動生成**。  
さらにAgent Skillsで「エッジケース（特殊な状況）」も考慮したテストを提案してくれるようになった

**メリット**：従来「どんな状況でバグが出るか」を人間が1つ1つ考えていた部分が大幅に省力化。  
**難しい点**：ユーザー体験を模倣したE2Eテストや「このシステムの本当の目的に合っているか」は、AIがまだ完全には理解できない。シナリオ設計の最終責任は人間が持つ。

## ドキュメントの整備

多くの現場で長年課題だった「暗黙知の属人化」を解消する工程。  
コードは読めても「なぜこの設計にしたのか」「運用時の注意点」「アーキテクチャのトレードオフ」が書かれていないと、後から入った人が苦しむ。

Claude Code はコードや会話履歴から**自動でREADME、API仕様書、ADR（Architecture Decision Record）、運用プレイブック**を生成可能。  
NotionやConfluenceへの自動反映、MkDocs静的サイト生成もAgent Skillsで一気通貫。  
「CLAUDE.mdにプロジェクトルールを書いておくと、ドキュメント生成の精度が劇的に上がる」と指摘されている。

**非エンジニア視点**：チームの「取扱説明書」をAIが最新版に保ってくれるイメージ。新人教育期間が半減し、技術的負債も溜まりにくくなる。  
**まだ人間が必要**：生成されたドキュメントの「事業的正確性」と「読みやすさ」の最終チェック。AIは「正しい日本語」を書くが、ニュアンスや優先順位は人間の判断。

## PR レビュー

コードを本番に取り込む前に別の人が問題点を指摘する「確認作業」。  
コードは「書くより読まれる回数が多い」のが現場の現実だ。

**Claude Code Review**（2026年3月リリースの多剤システム）をCI/CDに組み込むと、PR作成と同時にAIが自動でコメントを投稿。  
スタイル、セキュリティ、ベストプラクティス違反を瞬時に指摘してくれる。  
Anthropic公式発表でも「人間レビュアーは機械的チェックから解放され、設計思想やビジネス適合性の本質議論に集中できる」と位置づけられている。

**現場の変化**：レビュー時間が1/3〜1/2に短縮。  
**難しい点**：「このコードがビジネス要件を正しく満たしているか」という深いコンテキスト判断は、今でも人間の領域。

## デプロイ

完成したソフトウェアをクラウド（AWSなど）に配置し、ユーザーが使えるようにする最終工程。  
従来は深夜対応や人為的ミスが常態化していた。

**AWS Agent Plugins**やClaude CodeのComputer Use機能を使うと、Claude自身がクラウドコンソール操作やTerraform applyを実行。  
GitHub ActionsのHeadless Modeと組み合わせれば「PRマージ→自動デプロイ→監視開始」まで完全自動化可能。

**メリット**：リリース作業が「ボタン1つ」レベルになり、ミスが激減。  
**難しい点**：障害発生時のロールバック判断（「どこまで戻すか」「サービス停止のタイミング」）は事業影響を理解した人間が最終決定。

## 最後に——まだ自動化が難しい工程

ここまで見てきたように、2026年4月現在のClaude Code（およびAgent Skills）は、開発の大部分を「主体的に」こなせるレベルに到達した。  
Zenn・Qiitaの実践記事を読むと、すでに「開発速度3〜5倍」「エンジニアはコードを書く時間より判断する時間が増えた」という声が主流だ。

一方で、依然として人間の判断が中心の領域も残っている。

* **要件定義**：「何を作るべきか」「誰のどんな課題を本当に解くのか」→ ユーザー・ビジネスへの深い理解が必要
* **インシデント対応**：本番障害時の優先順位付けと意思決定
* **プロダクトの方向性判断**：技術的負債の返済タイミングや「この機能は本当に価値があるか」

AIは「決まったゴールに向けて速く・正確に進む」のが抜群に得意になった。  
エンジニアの役割は、**コードを書くことから「正しいゴールを設定し、AIの出力を判断・責任を持つ」ことへ**、静かに、しかし確実にシフトしている。

気づいたら、エンジニアは「コーディングをしなくなっていた」。  
それは「仕事がなくなった」のではなく、**より人間らしい創造的な仕事に集中できるようになった**証拠だ。
