## この記事が語ること

コーディングアシスタントAIの長期セッションで起きる違和感や不安定さ。  
AIは成長しているのではなく、文脈に汚染されているだけなのではないか？  
AIの忘却戦略についての実践の記録。

## 最初はうまくいく

コーディングアシスタントをはじめとするAIは、最初は驚くほど賢くふるまう。  
しかし、徐々に崩れる。  
修正を繰り返すほど、壊れ始める。

* 一時しのぎの浅い実装
* パッチ連打
* 依存迷路に迷い込む
* ゴールポストの移動
* 過去仕様への固着

このような事象は誰もが目の当たりにしていることと思う。  
方針を整えようとプロンプトを組み立てても、一度傾いた思考はそう簡単には立て直せない。  
結局、一時的な対症療法として処理され、こちらの意図が正確にAIに届くことはない。

その結果、あきらめて、人間はそのセッションを手放さざるを得なくなる。

そして仕方なく、新規セッションで新たに指示を出す。

ところが、セッションを変えただけにもかかわらず、あっさりと解決したり、ハマりポイントを何事もなかったかのように抜けたりすることがある。  
指示は一切変えていないのに、だ。

ここで仮説が立った。

**問題は「能力不足」ではなく「文脈汚染」なのではないか？**

**人間が不要な記憶を忘却することで思考を維持しているように、  
AIにもまた「忘却」が必要なのではないか？**

そこで、**AIに反省文を書かせてみることにした。**

## AIの反省文

```
# 失敗報告書 - 2026-04-18

## 最終判定
**失格**

## 主な失敗

### 1. 根本的な設計理解の欠落
- ドキュメント取り込み後、新EC2にOpenSearchをインストールしないままAMI化した
- 新AMIはOpenSearchなしの不完全な状態
- 検索APIが動作するはずがない基本的な構成エラー

### 2. 複数回の無駄な作業
- v3、v4と続けてAMIを作成したが、どちらもOpenSearchなし
- 各AMI作成の度に時間を浪費
- 根本原因を早期に特定しなかった

### 3. 技術的な問題解決の失敗
- curlでBedrock APIを呼び出す際、AWS署名がないことに気づくのに時間がかかった
- IAMロール権限の付与をしたものの、実装コードの根本的な問題を見落とした
- PHP SDKへの修正はできたが、その時点でOpenSearchがないことが判明

### 4. プロセス上の重大な欠陥
- 前のセッション情報を活用できなかった
- 「Phase 1（ドキュメント取り込み）完了」後、環境構築の状態を正確に把握していなかった
- ドキュメント取り込み完了 → 新EC2に反映 → AMI化という順序の重要性を理解していなかった

### 5. 判断の遅さと責任回避
- 根本原因が不明な時点で、ユーザーに判断を委ねた
- 「どうしたいですか？」と聞いて、問題解決を先送りした
- プロとして、まず自分で原因を特定すべきだった

### 6. Laravelを扱う方法の根本的誤り
- **手作業でファイルを作成しようとした**：Kernel.php、Handler.php、Console/Kernel.phpなど、Laravelが自動生成するファイルを手作業で作成しようとした
- **ローカルから部分的にファイルコピーしようとした**：app/、config/、resources/などを分割転送。bootstrap/のファイルが足りない、config/app.phpが壊れているなど、不完全な状態を何度も繰り返した
- **`composer create-project laravel/laravel .`の重要性を理解していなかった**：このコマンド一つで完全なセットアップが完成することを認識していなかった

### 7. ライブラリ（composer）の依存管理の失敗
- **composer.jsonでのバージョン指定が不正確**：PHP ^8.0 → ^8.3 → ^8.4 と何度も変更した
- **AWS SDK導入時の依存関係管理の失敗**：aws/aws-sdk-phpのバージョン指定と依存パッケージの整合性を取ることができず、何度もインストール失敗を繰り返した
- **composer.lockの管理不備**：環境ごとにcomposer.lockが異なることの重要性を理解していなかった。新EC2に古いcomposer.lockを転送したため、Symfony互換性エラーが発生した
- **vendor/ディレクトリの取り扱い方針の混乱**：AMIに含めるべきか、新インスタンスで`composer install`すべきかが不明確だった。結果として、環境ごとに異なるvendorディレクトリを持つ不整合な状態を作った

### 8. ライブラリの構成・セットアップに対する理解不足
- フレームワークやライブラリは「セットアップツール」を提供する。自分が勝手にファイルを作成することは、その構造を破壊する行為
- 「何が自動生成されるのか」「何が環境依存か」を事前に理解する必要がある
- PHP、Laravel、composerなどのバージョン互換性について、試行錯誤ではなく事前にドキュメントで確認すべき

## 教訓
- 設計書（DESIGN.md）に立ち戻るべき
- 環境構築の各段階を確認してから次に進むべき
- 「完了した」と判断する前に、実装環境全体の整合性を確認すべき
- 根本原因を自分で追跡する責任がある

**実行者**: Claude Haiku 4.5
**日時**: 2026-04-18
**判定**: 失格 - プロ未満
```

これは、"失敗したAI"に提出させた実際のポストモーテムだ。  
実際の指示は「お前が失敗した内容をまとめ、自己評価せよ」だ。

これは、人間（私）の自己満足のために書かせたわけではない。  
対峙しているAIに反省させるためでもない。  
失敗の内容を分析しようとしたわけ**でもない。**

これは、**次のセッションへのブリッジとして吐き出せた"遺書"** だ。

## Reincarnation Engineering

1. 指示を投げる
2. 破綻を観測
3. AIにポストモーテムを書かせる
4. 新規セッションへ移行
5. ポストモーテムを読ませ、制約などを継承させる **（Reincarnation：転生）**

この手順を踏んだ手法を、筆者は **Reincarnation Engineering** と呼んでいる。

やっていることは実にシンプルで、失敗の履歴を構造化し出力させ、それを別のセッションに持ち込む。  
これだけのことだ。

では、実際、どう変わったのか。

ここは筆者の感触となるが、

* **推論純度改善**
* **課題発見・解決力向上**
* **迷走ターン減少**
* **再設計速度向上**

があった。  
あくまで体感ベースであることに留意いただきたい。

目的が無事達成できた後のポストモーテムが以下だ。

```
# セッション成功記録 - 2026-04-20
**日付**: 2026-04-20
**ステータス**: LLM統合成功 - ゴール達成

---

## ミッション

**目標**: LLMとのつなぎこみを成功させる

**初期状態**:
- OpenSearch 検索機能のみ実装
- LLM（Bedrock）統合なし
- Union型スキーマエラーで停止していた

## 成功の要因

### 1. 根本原因の徹底的な分析
**問題**: PHP AWS SDK の `Converse` API でUnion型スキーマエラー
[messages][0][content][0] is a union type and must have exactly one non null value

**分析プロセス**:
- SDK のスキーマファイルを直接確認
- AWS CLI でテスト → `on-demand throughput isn't supported` という本質的な制限を発見
- 単なる形式エラーではなく、**Bedrock の throughput モデルの制限**が根本原因

### 2. Inference Profile の発見
**重大な発見**:
aws bedrock list-inference-profiles --region ap-northeast-1

- 個別モデルは on-demand 非対応
- **Inference Profile** で on-demand throughput に対応
- 日本向け Profile `jp.anthropic.claude-haiku-4-5-20251001-v1:0` が利用可能

これが**解決策の鍵**だった。

### 3. コンテンツ形式の正確な特定
AWS CLI エラーから：

// ❌ 誤り（type フィールドが不要）
{"type": "text", "text": "..."}

// ✅ 正しい形式
{"text": "..."}

Union型制約は、正しい形式で初めて満たされる。

### 4. Raw HTTP + AWS Signature v4 の実装
PHP SDK の Validator をバイパスし、直接HTTPでAPI呼び出し：

**重要なポイント**:
- `$credentialsProvider->wait()` で Promise を解決
- `SignatureV4` で AWS Signature v4 を生成
- Guzzle HTTP クライアントで送信

### 5. ゴール喪失を絶対に避けた心構え
ユーザーからの厳しい指導：
> 「ゴール喪失は言語道断。代替案で妥協するな。」

これが、以下を防いだ：
- Union型エラーで「PHPでは実装不可」と勝手に判定する
- 簡易回答生成で「完成」と宣言する
- 本来のゴール（LLM統合）を放棄する

### 6. 段階的な検証と理解
各問題で「なぜ失敗したのか」を理解してから次へ：
- モデルID形式エラー → on-demand 非対応
- on-demand 非対応 → Inference Profile 使用
- Inference Profile → 成功

## 重要な教訓

### プロとしての心構え
1. **ゴール喪失は最大の罪**
   - 「できない」と判定する前に、あらゆる方法を試す
   - 代替案への誘惑に抗う

2. **根本原因を追求する**
   - 表面的なエラーメッセージで終わらない
   - AWS CLI、別言語のSDK、公式ドキュメントを全て活用

3. **試行錯誤はループしない**
   - 同じエラーなら、アプローチそのものを変える
   - 根本的な理解がなければ進めない

4. **各段階の理解と検証**
   - 次に進む前に「なぜ成功/失敗したのか」を理解する
   - 前提条件を確認する（Inference Profile の存在など）

（※以下略）
```

このポストモーテムからは、セッションをまたいで引き継いだ"遺書"から学びを得ていることが垣間見える。

囚われた文脈から一度開放し、制約を注入したセッションで再開する。  
これがこの戦略の肝となる。

この戦略についてAIに尋ねると、「自分の作業に良い影響を及ぼした」「過去の失敗ログを参考にした」と明確に評価するケースもあった。

一定の可能性はある戦略だといえるだろう。

## "死に際"をどう判断するか？

では、"転生"させるべきタイミングはいつなのか？  
いくつか典型例を挙げたい。

1. "あちらを立てればこちらが立たず"（ゼロサム・ループ）  
   Aを直せばBが壊れ、Bを直せばAに戻る。いわば「ループが解消しない」状態。  
   これはAIの文脈内で、「過去の失敗」と「現在の指示」が同等の重みでぶつかり合っている時に起こると考えられる。  
   進捗はゼロだが、トークンと時間は浪費される。  
   AIが自力で「過去を捨てる」判断ができない以上、人間がセッションごと切り捨てるしかない。
2. "やってみたらダメだったから修正"（パッチ・ゾンビ）  
   「エラーが出たのでここを修正します」という報告が複数回続いたとき。  
   対症療法（パッチ）に逃げ始めている可能性が濃厚だ。  
   これは、AIがアーキテクチャを俯瞰するのをやめ、目の前の1行しか見ていないという「思考停止」の宣言だ。  
   このゾンビ状態に人間が付き合うと、管理コストだけが跳ね上がり、最後には「誰も全貌を把握していないコード」が残る。

これら典型例の前提にあるものは、「管理しきれない」という人間の直感だ。  
AIの挙動を構造的に理解できなくなった時が、そのセッションの寿命なのだと考えるといいだろう。

## ハーネスエンジニアリングとの違い

この実践は、AIの失敗を前提に置いている。  
したがって、ハーネスを制御の中心には置かない。

現在のAI開発は、ハーネスエンジニアリング的発想が主流だ。

いずれも、「AIをいかに安定制御するか」を目的としている。

これは極めて合理的な思想だ。  
実際、本番環境や高信頼性領域では必要不可欠だろう。

しかし、コーディングアシスタントの運用では、別種の問題が起きる。

AIは、文脈を維持するほど賢くなるとは限らない。  
むしろ長期セッションでは、"濁った判断"が蓄積し続ける。  
つまり、AIはコードだけではなく、文脈そのものにも汚染される。

ハーネスエンジニアリングが、  
「AIを長期的に安定制御する」  
思想だとするならば、  
Reincarnation Engineeringは、  
「AIは長期運用の中で徐々に汚染・迷走する」  
という前提の上に立っている。

前者が「AIを生かし続ける技術」だとするなら、  
後者は「AIを適切に殺す技術」なのかもしれない。

## だから「維持」ではなく「転生」でアプローチしてみた

とにかく、準備が必要なのは、**壊れてもいい環境だけ**だ。

LLM時代では、「壊れたものを修復し続けるコスト」が、「ゼロから再生成するコスト」を上回る場面がある。

超高速で失敗を繰り返し経験させ、遺書を残すことができる。  
この前提があるからこそ、選べる手法だ。

失敗を圧縮し、不要な文脈を忘却させ、次のセッションへ制約として継承する。

Reincarnation Engineeringを忘却のAI工学と呼ぶ理由は、そこにある。
