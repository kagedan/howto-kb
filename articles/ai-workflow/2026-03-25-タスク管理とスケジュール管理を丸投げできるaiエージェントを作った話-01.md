## はじめに

この記事では、メールを読んでタスクを拾い上げ、スケジュールと照らし合わせながら毎朝ブリーフィングを届けてくれるAIエージェントを作った話を書きます。タスク管理とスケジュール管理を人間の手を介さず自動で回す、というのがコンセプトです。

---

作ったきっかけは育休です。

第二子が生まれたとき、まず6ヶ月の育休を取りました。新生児期は育児に集中するしかなく、タスク管理どころじゃありませんでした。

その後、子どもが9ヶ月になるタイミングで再び1ヶ月の育休を取得。今度は新生活に向けたさまざまな準備が重なって、やることや予定が一気に詰まってきました。育児で自由に使える時間は限られているのに、管理しなきゃいけないことは山積み。メールを確認してタスクを整理して……なんて時間は正直ありません。

「誰かに丸ごと管理してもらいたい」と思ったのが出発点です。

バックグラウンドで静かに動き続け、メールを読んでタスクを拾い上げ、Notionに登録し、Telegramで通知してくれるパーソナルエージェント。それを目指して作ったのが **Ambient Agent** です。

<https://github.com/eh6gac4/ambient-agent>

## Ambient Agentとは

LangChainのCo-Founder Harrison Chaseが2025年1月に提唱した概念です。

> "Ambient agents listen to an event stream and act on it accordingly"  
> — [Introducing Ambient Agents, LangChain Blog (2025)](https://blog.langchain.com/introducing-ambient-agents/)

従来のチャット型AIは「人間がメッセージを送る → AIが応答する」という形でした。対してAmbient Agentは**人間の操作を起点にせず、バックグラウンドのイベント（メールの受信、スケジュール、Webhookなど）を常時監視して自律的に動き続ける**エージェントです。

|  | 従来のチャット型AI | Ambient Agent |
| --- | --- | --- |
| トリガー | 人間のメッセージ | バックグラウンドのイベント |
| 動作タイミング | 呼ばれたとき | 常時（イベント駆動） |
| 人間の関与 | 都度必要 | 通知・確認を最小限に |

今回作ったエージェントは、この考え方を個人のタスク管理に適用したものです。

## 何をするエージェントか

朝8時になると自動で以下が流れます。時刻はすべてJST。

| 時刻 | 処理 |
| --- | --- |
| 07:55 | Gmailの未読メールをClaudeが分析 → アクション項目をNotionのタスクDBに登録 |
| 07:57 | Notionのタスクとカレンダーを同期（完了済み削除・未着手を新規登録） |
| 07:58 | 期限3日以内のmediumタスクをhighに昇格してTelegram通知 |
| 08:00 | 今日の予定と未完了タスクをClaudeが要約して日次ブリーフィング送信 |
| 08:05 | 前日のClaude APIコストレポート送信 |
| 13:00 | 未着手タスクのリマインド |

また、Telegramのチャットからリアルタイムにタスクを操作できます。URLやテキストをそのまま送りつけると、Claudeがタスクを抽出してNotionに登録もしてくれます。

### Telegramコマンド

チャットから以下のコマンドで操作できます。

```
/tasks              未着手タスク一覧
/done 3             タスク番号3を完了
/add タスク名       タスクを追加
/due 3 2026-04-01   タスクの期限を設定
/briefing           今日のブリーフィングを手動取得
/help               コマンド一覧を表示
```

## 技術スタック

| 分類 | 技術 |
| --- | --- |
| 言語 | Python 3 |
| AI モデル | Claude Haiku 4.5 |
| スケジューラ | APScheduler |
| 外部API | Gmail API, Google Calendar API, Notion API, Telegram Bot API |
| 実行環境 | Docker / Docker Compose |

Claude Haiku を選んだのは、タスク抽出・要約のような軽めの推論が中心なのでコストとスピードのバランスが最適だからです。

### なぜTelegramなのか

通知チャンネルにTelegramを選んだのには理由があります。

もともとTelegramは家族間の連絡に使っているアプリです。LINEやSlackと違って、**アプリごとに用途を分けている**ので、Telegramの通知が来た瞬間に「家族か、エージェントからの通知」だとわかります。仕事のSlackや友人とのLINEが混在するアプリだと、通知を開くまで誰からか判断できませんが、Telegramは一目でわかる。

技術的な理由もあります。TelegramのBot APIは非常に充実していて、コマンドの登録・受信・返信がシンプルに実装できます。SlackもAPIは充実していますが、Workspaceの管理や権限設定などの手間を考えると、個人用途ではTelegramの方が圧倒的に楽です。

## アーキテクチャ

エージェントは2つの並走する部品で成り立っています。**定時バッチ**（APSchedulerで朝の処理などを自動実行）と、**Telegramリスナー**（常時待機してコマンドをリアルタイムに受け付ける）です。

```
agent/
  ├── main.py              ← エントリーポイント
  │   ├── BlockingScheduler（APScheduler）で定時ジョブを管理
  │   └── Telegram ロングポーリング（daemonスレッド）
  ├── gmail_handler.py     ← 未読メール取得・本文パース
  ├── claude_agent.py      ← Claude API ラッパー
  ├── notion_handler.py    ← Notion DB への読み書き
  ├── google_calendar.py   ← タスク↔カレンダー同期
  ├── calendar_handler.py  ← ブリーフィング・リマインド生成
  ├── telegram_handler.py  ← コマンド処理
  ├── telegram_notifier.py ← メッセージ送信
  └── usage_tracker.py     ← Claude API コスト記録

scripts/
  ├── run_simplify.py      ← 週次リファクタリング（PR作成まで自動化）
  └── watchdog.py          ← コンテナ死活監視

prompts/
  ├── extract_tasks.md     ← タスク抽出プロンプト
  └── analyze_email.md     ← メール分析プロンプト
```

## こだわった実装ポイント

### 1. Gmailスレッド追跡でタスクの重複を防ぐ

同じスレッドへの返信メールが届いたとき、新しいタスクを作らず既存タスクを更新するようにしました。

`data/gmail_thread_map.json` に `threadId → Notion page_id` のマッピングを持っておいて、既知スレッドへの返信なら：

* 既存タスクにチェックリストとして追記
* 優先度を昇格
* 期限を前倒し

という処理を行います。これで「同じ件名のタスクが何個も積まれる」問題を回避できました。

### 2. プロンプトの外部化

タスク抽出のプロンプトをMarkdownファイルとして分離しています。

```
# prompts/extract_tasks.md

あなたはメールからタスクを抽出するアシスタントです。
以下のメール本文を分析し、アクションが必要な項目を抽出してください。
...
```

コードに埋め込んでしまうと、プロンプトを調整するたびにコードを触る必要があります。Markdownに分離しておくと、コードを変更せずにプロンプトだけチューニングできて便利です。

### 3. Claude API のコスト管理

各Claude呼び出しのラベル・入力/出力トークン数を `usage_tracker.py` で記録し、翌朝にコストレポートをTelegramへ送信します。

```
usage_tracker.record(
    label="extract_tasks",
    input_tokens=response.usage.input_tokens,
    output_tokens=response.usage.output_tokens,
)
```

また `config.py` で `OPERATING_START_HOUR`〜`OPERATING_END_HOUR`（デフォルト8時〜21時 JST）を設定し、深夜帯はClaudeを呼び出さないようにしてコストを抑えています。

### 4. 週次自動リファクタリング

毎週月曜に `scripts/run_simplify.py` をcronで実行し、Claude Codeの `/simplify` でコードを整理します。変更があれば自動でブランチを作ってコミット・プッシュし、PRを作成してURLをTelegramに通知するところまで全自動です。

これにはちょっとした背景があります。

AIは指示を的確にこなしてくれますが、**局所的な最適解になりがち**です。「この機能を追加して」という指示に対しては正確に動くコードを書いてくれる一方で、既存の処理との重複や全体的な整合性まで気を配ることは苦手です。機能を積み重ねていくと、似たような処理があちこちに散らばっていたり、不要なコードが残っていたりします。

定期的にコードベース全体を見渡してリファクタリングすることで、こうした問題を解消し品質を保つようにしました。エージェントがエージェントを整理する、という構成です。

### 5. コンテナ死活監視

`scripts/watchdog.py` を別のcronジョブで定期実行しており、`ambient-agent` コンテナが停止していたらTelegramに通知します。一度通知したら復旧まで再通知しない仕組みにしてあり、深夜に止まっていても次の朝に気づけます。

## 開発環境

育休中、まとまった時間はほとんど取れません。子どもが昼寝している間や、ちょっとした移動中の隙間——そういう時間を使って実装を続けるために、iPhoneからでも開発環境にアクセスできる仕組みを整えました。

自宅サーバーにVPNを構築して外出先から自宅のネットワークに接続し、SSH経由でClaude Codeを使えるようにしました。これで「机に座れないと作業できない」という制約がなくなり、隙間時間に機能を追加したりバグを直したりできるようになりました。

## セットアップ

`.env` ファイルに各サービスの認証情報を設定してDockerで起動するだけです。

```
cp .env.example .env
# .envを編集して各APIキーを設定
docker compose up -d
```

必要な認証情報:

* Anthropic APIキー
* Google OAuth2（Gmail + Calendar）
* Notion インテグレーショントークン
* Telegram Bot トークン

## まとめ

作ってみて感じたのは、**「AIを使うと自動化できる対象が変わる」** ということです。

従来のRPA的な自動化は「決まった操作を繰り返す」もので、非構造化データのメールは扱いにくい対象でした。LLMを組み合わせることで、文脈を読み取ってタスクを抽出するという「人間がやっていた判断」を自動化できるようになりました。

まだ荒削りな部分もありますが、育休中のタスク管理という当初の目的は十分に果たしてくれています。今後追加したい機能もいくつかあります。

まず**メール返信の自動下書き生成**です。今はメールからタスクを抽出して終わりですが、「この件は了解の返信が必要」と判断したメールに対して下書きを自動生成してGmailのドラフトに保存できれば、返信待ちが減ります。

次に**タスクの優先度スコアリングの改善**です。現在は期限が近づいたら昇格するルールベースの処理ですが、メールの内容・送信者・タスクの種類などを踏まえてClaudeがスコアを判断する仕組みにしたいと考えています。

また、今は育休中なので仕事のコンテキストが薄いですが、復職後は**Slackとの連携**も入れたいです。仕事のSlackメッセージも同じフローでNotionに流せれば、仕事とプライベートのタスクを一元管理できます。
