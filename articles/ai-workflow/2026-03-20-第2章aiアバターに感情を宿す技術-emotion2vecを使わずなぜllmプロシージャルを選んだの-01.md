口が動くだけのアバターは、どれだけ精巧に作っても「人形」だ。

前回の記事で、音声からリアルタイムに口を動かすリップシンクAIを作った。精度0.89、75倍速。数字の上では成功だった。

だが、実際に動かしてみると、違和感が消えない。

口だけが機械的に動き、目は死んでいる。何を喋っても同じ顔。これは「喋る人形」であって、「生きたキャラクター」ではない。

人間が相手を「生きている」と感じるのは、口ではなく**目**だ。嬉しいときに目尻が下がる。怒ったときに眉間にしわが寄る。驚いたときにまぶたが大きく開く。無意識にそれを読み取っている。

**口の動きは「何を喋っているか」を伝え、目の動きは「何を感じているか」を伝える。**

今回は、このアバターに「感情」を実装した。そして、emotion2vecのような感情認識AIモデルがあるのに、あえて使わなかった理由を書く。

## 前提知識: 3Dアバターの顔はどう動いているのか

本題に入る前に、3Dアバターの顔がどう動いているかを説明しておく。知っている人は読み飛ばしてほしい。

### 「スライダーで顔を動かす」仕組み

Webエンジニアに一番伝わる例えはこうだ。**CSSの `opacity` を0.0〜1.0で変えると透明度が変わるのと同じ要領で、顔のパーツごとに「変形の度合い」を0.0〜1.0で指定する。**

3DCGではこれを**ブレンドシェイプ**と呼ぶ。名前は仰々しいが、やっていることは「スライダー」だ。

まず「無表情の顔」をベースに、「口を開けた顔」「目を閉じた顔」「眉を上げた顔」といった変形パターンをあらかじめ用意しておく。それぞれのスライダーを動かすと、ベースの顔にその変形が合成される。

```
jawOpen:        0.8  → 口が大きく開く
mouthSmileLeft: 0.5  → 左口角が少し上がる
eyeBlinkLeft:   0.0  → 左目は開いたまま
browInnerUp:    0.3  → 眉の内側が少し上がる
```

JSONで顔のパラメータを送って、3Dモデルに反映する ── WebSocketでリアルタイムにやっているだけだ。Web APIを叩いてUIを更新するのと感覚的にはそう変わらない。

### VRMアバターには52本のスライダーがある

VRMは、Web上で3Dアバターを扱うためのフォーマットだ（Three.jsで描画できる）。Apple ARKitの定義に準拠した**52種類のブレンドシェイプ**が標準として決まっている。

リップシンクAIが主に使うのは「口・あご」の22本。**残りの30本が、今回の感情表現の主戦場だ。**

代表的なものを部位別に並べるとこうなる。

**眉（5本）** ── 感情が最も出るパーツ

| スライダー名 | 動き | 関連する感情 |
| --- | --- | --- |
| `browDownLeft/Right` | 眉を下げる | 怒り |
| `browInnerUp` | 眉の内側を上げる | 悲しみ・驚き |
| `browOuterUpLeft/Right` | 眉の外側を上げる | 喜び |

**目（14本）** ── 視線・まばたき・感情すべてに関わる

| スライダー名 | 動き | 関連する感情 |
| --- | --- | --- |
| `eyeBlinkLeft/Right` | まばたき | ─ |
| `eyeLookUp/Down/In/Out` (x4) | 視線の方向 | ─ |
| `eyeSquintLeft/Right` | 目を細める | 怒り・笑顔 |
| `eyeWideLeft/Right` | 目を見開く | 驚き・恐怖 |

**頬・鼻（4本）** ── 細かいが効果は大きい

| スライダー名 | 動き | 関連する感情 |
| --- | --- | --- |
| `cheekSquintLeft/Right` | 頬を持ち上げる | 笑顔（目が細くなる効果） |
| `noseSneerLeft/Right` | 鼻にしわを寄せる | 嫌悪 |

口・あご（22本）の詳細は前回の記事で書いたのでここでは省略する。

### VRMには「ショートカット」もある ── プリセット表情

52本のスライダーを毎回1本ずつ動かすのは大変だ。そこでVRMには**プリセット表情**という仕組みもある。

```
happy = 0.8  → 「嬉しい顔」が一発で出る
angry = 0.5  → 「怒った顔」が一発で出る
sad   = 0.3  → 「悲しい顔」が一発で出る
```

CSSでいうと、個別プロパティ（`margin-top`, `margin-right`, ...）と一括指定（`margin`）の関係に似ている。`happy = 0.8` と書けば、モデル制作者が事前に設定した「嬉しいときの眉の角度、目の形、口角の上がり具合」がまとめて適用される。

VRoid（VRMモデルの制作ツール）で作ったキャラクターには、こんなプリセットが入っている。

```
Fcl_BRW_Angry    → 怒り眉
Fcl_BRW_Fun      → 楽しい眉
Fcl_EYE_Surprised → 驚き目
Fcl_MTH_Joy      → 笑顔の口
Fcl_MTH_A/I/U/E/O → あいうえおの口
```

### 「ショートカット」と「個別指定」、どちらを使うか

|  | プリセット（ショートカット） | 52本のスライダー（個別指定） |
| --- | --- | --- |
| 手軽さ | `happy = 0.8` で完結 | 全部自分で決める |
| 自由度 | 制作者の設定に依存 | 何でもできる |
| 例 | 「怒りの表情を出して」 | 「眉だけ下げて目は細めず」 |

プリセットは楽だが、\*\*「怒りで眉だけ少し下げたいけど、目は笑ったまま」\*\*みたいな細かい制御ができない。1つの感情に対して1つの顔が固定されてしまう。

でも感情は白か黒かではない。嬉しいけど少し照れている。怒っているけど呆れてもいる。こういう微妙なニュアンスを出すには、52本のスライダーを個別に制御するしかない。

**今回は、プリセットは使わず、52本のスライダーを直接操作する方式を選んだ。** 自由度のために手軽さを捨てた。

### ここまでのまとめ

前回の記事で作ったリップシンクAI（LAM-ja）は、音声波形からこの52本のスライダー値を毎秒30フレーム分、リアルタイムに予測するモデルだった。ただし、LAM-jaが主に動かすのは口周りのスライダーだけだ。目や眉はほぼゼロ ── つまり無表情のまま。

**だから「喋る人形」に見えていた。** 52本のうち、口の22本しか使っていなかったのだ。

今回やることは、残りの30本 ── 目・眉・頬・鼻 ── を感情に応じて動かすことだ。

## 作ったもの: 感情パイプラインの全体像

先に完成形を見せる。

ポイントは、感情の処理が2つのレイヤーに分かれていることだ。

1. **感情の「認識」**（青） ── 今この瞬間、何を感じているかを決める
2. **感情の「表現」**（黄） ── その感情を、顔のどのパーツに、どれくらいの強さで反映するか

この2つは、まったく別の問題だ。そして、それぞれで「AIモデルを使わない」という判断をした。

## 感情の認識: なぜemotion2vecではなくClaude APIなのか

### 感情認識AIモデルという選択肢

音声から感情を推定する専用AI ── SER（Speech Emotion Recognition）は、この数年で急速に進化している。

**emotion2vec**は、そのSER分野の代表格だ。音声の韻律やトーンから、怒り・喜び・悲しみといった感情を分類する。実際、このプロジェクトで比較した7モデルのうちDEEPTalkは、内部でemotion2vecを使って感情条件付きの表情生成をやっている。

つまり、手元にあった。使おうと思えばすぐ使えた。

**それでも使わなかった。** 理由は3つある。

### 理由1: そもそも「声」がない

emotion2vecが解くのは「この音声はどんな感情か？」だ。

だが、今回のチャットモードでは、ユーザーの入力はテキストだ。

```
テキスト入力 → Claude API → 返答テキスト → TTS → 音声
```

音声が存在するのはTTSの出力以降。それはElevenLabsが合成した音声であって、そこに「AIの感情」は乗っていない。合成音声をemotion2vecに食わせたところで、返ってくるのはElevenLabsの音声パラメータの癖であって、会話の文脈とは何の関係もない。

**感情の源泉がテキスト会話にあるなら、テキストを理解できるモデルが感情を決めるべきだ。**

### 理由2: LLMなら「返答」と「感情」を同時に、追加コストゼロで出せる

2つのアーキテクチャを比べてみる。

決定的な差は**レイテンシ**だ。

SERアーキテクチャでは、TTSが音声を生成し終わるまで感情ラベルが確定しない。つまりアバターの表情は、音声再生が始まるまで「待ち」になる。ユーザーがメッセージを送ってから、アバターの顔が反応するまでに空白の時間が生まれる。

LLMに同時出力させれば、テキストが返ってきた瞬間に感情が決まる。音声の再生が始まる前に、表情の変化をスタートできる。

さらに、感情ラベルが先に決まるということは、**将来的にTTSのパラメータ（声のトーンや抑揚）を感情に応じて変える**ことも可能になる。SERでは原理的にこの順序が逆転できない。

### 理由3: 会話の「空気」を読める

emotion2vecは、渡された数秒の音声断片だけを見る。1ターン前に何を話していたか、会話がどんな雰囲気だったかは知らない。

Claude APIには会話履歴を渡せる。

```
trimmed = conversation_history[-MAX_HISTORY:]  # 直近20ターン
```

「さっきペットが亡くなった話をしていたから、今の返答もまだ少しトーンを落とす」── こういう判断ができるのは、会話の流れを丸ごと見ているLLMだけだ。

### 実装: たった数行のプロンプトで済む

Claude APIへのシステムプロンプトは、これだけだ。

```
必ず以下のJSON形式で返答してください:
{"text": "返答テキスト", "emotion": "感情ラベル", "emotion_intensity": 0.0〜1.0}

感情ラベルは以下から1つ選んでください:
- happy: 嬉しい、楽しい、面白い
- angry: 怒り、不満、苛立ち
- sad: 悲しい、残念、寂しい
- surprised: 驚き、意外
- neutral: 普通、落ち着いている
- disgust: 嫌悪、不快
- fear: 恐怖、不安、心配
```

Ekmanの基本6感情 + neutral の7種。強度は0.0〜1.0の連続値。

```
async def get_chat_response(conversation_history):
    client = anthropic.AsyncAnthropic(api_key=ANTHROPIC_API_KEY)
    response = await client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        system=SYSTEM_PROMPT,
        messages=conversation_history[-20:],
    )
    raw_text = response.content[0].text
    return _parse_response(raw_text)  # (text, emotion, intensity)
```

モデルは `claude-haiku-4-5`。感情ラベルの付与程度ならHaikuで十分で、レスポンスも速い。不正なラベルや、マークダウンのコードフェンスが混入した場合は `neutral` にフォールバックさせている。

### ただし、emotion2vecが正解な場面もある

公平のために書いておく。

* **ユーザーの「声色」から感情を読みたいとき** ── テキストでは伝わらない韻律情報がある。怒っているのか冗談なのか、声を聞かないとわからない場面はある。
* **マイク入力でアバターを動かすモード** ── このプロジェクトにもある。ユーザーの声をそのままアバターに反映するモードでは、感情の源泉は「ユーザーの声」なので、SERが適する。
* **会話を外から分析する場面** ── コールセンターの品質監視など。AIが「話者」ではなく「観察者」の場合。

結局、**感情の源泉がどこにあるかで、使うべきモデルが変わる**。

| 感情の源泉 | 適したアプローチ |
| --- | --- |
| AIが生成したテキスト | LLM同時出力 |
| ユーザーの音声 | SER（emotion2vec等） |
| ユーザーのテキスト | テキスト感情分析 / LLM |
| 映像（表情） | FER（顔表情認識） |

今回は「AIが喋る側」だったから、LLMが正解だった。それだけの話だ。

## 感情の表現: なぜAIモデルではなくプロシージャルなのか

感情ラベルが決まった。次は「顔のどこをどう動かすか」だ。

ここにも、DEEPTalkのような「感情条件付き表情生成AI」を使う選択肢はあった。感情ラベルを入力すれば、AIが適切な表情のスライダー値を丸ごと生成してくれる。

だが、ここもプロシージャル（ルールベース） ── つまり `if` 文と四則演算で手書きする方式を選んだ。

### Pixarのアニメーターが口を揃えて言うこと

> **感情は目に宿る。口ではない。**

目を細めるだけで「嬉しそう」に見える。眉を寄せるだけで「怒っている」と伝わる。口が笑っていても目が笑っていなければ「作り笑い」だとわかる。

この知見をコードに落とし込んだのが「Eye Mask」システムだ。

### 7感情の顔面マッピング

各感情が、52本のスライダーのうちどれを動かすかを図にした。

感情スコアが0.2以下のときはスライダーを動かさない。閾値をカットしないと、弱い感情で顔が常にピクピクして気持ち悪くなる。

実際のコードはこうなっている。ウェブエンジニアなら見慣れたJavaScriptだ。

```
const happyVal = (scores.happy || 0);
if (happyVal > 0.2) {
  const intensity = happyVal * 0.35;
  // 頬を持ち上げる → 自然と目が細くなる
  ['cheekSquintLeft', 'cheekSquintRight'].forEach((name) => {
    const idx = dict[name];  // スライダーのインデックスを取得
    if (idx !== undefined)
      mesh.morphTargetInfluences[idx] = Math.max(
        mesh.morphTargetInfluences[idx] || 0, intensity
      );
      // ↑ morphTargetInfluences = スライダー値の配列。ここに値を書き込むと顔が変形する
  });
}
```

`mesh.morphTargetInfluences` はThree.jsのAPIで、3Dメッシュのブレンドシェイプ値（＝スライダー値）を格納する配列だ。ここに数値を書き込むだけで、リアルタイムに顔が変わる。

### DEEPTalkのような感情AIモデルを使わなかった3つの理由

**1. 速度を殺したくなかった**

LAM-jaは口の動き特化で、1秒の音声を0.013秒で処理する（75倍速）。DEEPTalkは感情込みで表情を一括生成するぶん、はるかに重い。

プロシージャルの感情表現は**GPUすら使わない**。JavaScriptの `if` 文と掛け算だけで毎フレーム完結する。レイテンシへの影響は実質ゼロ。リアルタイムアバターでは、この差が体験を左右する。

**2. 「もう少し眉を上げたい」に即座に対応できる**

AIモデルの表情はブラックボックスだ。出力が気に入らなくてもモデルを再学習するしかない。

プロシージャルなら `intensity * 0.35` の `0.35` を `0.5` に変えるだけだ。ブラウザをリロードすれば反映される。アニメーターが手でパラメータを調整するのと同じ感覚で、リアルタイムに試行錯誤できる。

**3. これは「機械学習」の問題ではなく「アニメーション」の問題**

目的は「現実の人間の表情をコピーする」ことではない。「見た人が感情を読み取れる表情を作る」ことだ。

ディズニーの12原則にある「Exaggeration（誇張）」── 感情は現実より少しオーバーに表現したほうが伝わる。この「少し」の度合いは、感情ごとに違う。驚きは大げさでいいが、悲しみは控えめなほうがリアルに見える。

こういう美的判断を、データから学習させるのは難しい。人間がルールを書いたほうが速いし、結果も良い。

## 細部が「生きている感」を作る

Eye Maskだけでは足りない。人間を「生きている」と感じさせるのは、もっと些細なことだ。

### 感情で変わるまばたきの頻度

```
const EMOTION_BLINK_RATE = {
  neutral:   1.0,
  happy:     1.1,   // 少し多め
  angry:     1.3,   // 苛立ちで多め
  sad:       0.9,   // 少なめ（ぼんやり）
  surprised: 0.4,   // 目を見開いている → 大幅に減少
  fear:      1.6,   // 不安 → かなり多め
  disgust:   1.2,
};
```

不安なときにまばたきが増える。驚いたときは目を見開いて、まばたきが止まる。これは臨床研究で確認されている人間の生理反応だ。コードにすれば数行。だが、これがあるのとないのとでは、アバターの「生きている感」がまるで違う。

さらに、20%の確率でダブルブリンク（二度まばたき）も入れている。人間は機械のように等間隔でまばたきしない。このランダムさが「人形っぽさ」を消す。

### 感情の切り替わりを滑らかにする

「happy」から「sad」に感情が変わったとき、顔が一瞬でパッと切り替わると、ロボット感が出る。CSSの `transition` のように、スムーズに補間する必要がある。

ここでは指数移動平均（EMA）を使っている。ReactやVueの `transition` と同じ発想で、現在値を目標値に少しずつ近づけるだけだ。

```
const EMOTION_LERP = 0.15;

// 毎フレーム実行（requestAnimationFrameのループ内）
smoothedEmotions.happy += (target.happy - smoothedEmotions.happy) * EMOTION_LERP;
smoothedEmotions.angry += (target.angry - smoothedEmotions.angry) * EMOTION_LERP;
smoothedEmotions.sad   += (target.sad   - smoothedEmotions.sad)   * EMOTION_LERP;
```

`0.15` は、約0.5秒で目標値の90%に到達する速度。感情の「余韻」が少し残る。嬉しい話の直後に悲しい話になっても、表情がふわっと移り変わる。この「ふわっと」が、キャラクターに人間味を与える。

## 結論: AIモデルがあるからといって、使うべきとは限らない

今回のプロジェクトで一番の学びはこれだった。

| 処理 | 選んだ手法 | なぜ |
| --- | --- | --- |
| リップシンク | **AIモデル**（LAM-ja） | 音声→口の動きは複雑すぎてルールでは書けない |
| 感情認識 | **LLM**（Claude API） | 感情の源泉がテキスト会話にある。レイテンシも最小 |
| 感情表現 | **プロシージャル** | 調整の自由度、速度、アニメーション原則の適用 |
| まばたき | **プロシージャル** | 生理学パラメータをそのまま実装するだけ |
| 視線追従 | **プロシージャル** | 物理モデルで十分 |

AIモデルの出番は「入力と出力の関係が複雑で、人間がルールを書ききれない」ときだ。

感情表現は違う。「嬉しいときは頬が上がる」「怒ると眉が下がる」── 人間はこのルールを何千年も前から知っている。知っているなら、書けばいい。AIに学習させる必要はない。

**書けるものは書く。書けないものだけAIに任せる。** この判断が、リアルタイムアバターの品質と速度の両立を可能にした。

## ディズニーレベルまでの距離

ただ、正直に書いておく。今回「解決した」と言えるのは、全体のほんの一部だ。

ディズニーやPixarのキャラクターが「生きている」と感じられるのは、口・目・眉だけではない。手が動き、体が揺れ、髪が揺れ、呼吸し、重心が移動する。それが全部、1フレームの遅延もなく連動している。

今の進捗を正直に並べるとこうなる。

| 領域 | 要素 | 状態 | アプローチ |
| --- | --- | --- | --- |
| 顔 | リップシンク | **解決** | LAM-ja（75倍速） |
| 顔 | 感情表情 | **★今回** | LLM + プロシージャルEye Mask |
| 顔 | 視線・まばたき | **★今回** | サッケード + 感情連動ブリンク |
| 声 | 音声合成（TTS） | 動作中 | ElevenLabs |
