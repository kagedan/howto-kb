# AI 支援開発に「テストと監査の規律」を組み込む

AI にコードを書かせる開発スタイルが定着すると、生産量は跳ね上がる一方で、**「動くけれど壊れやすいコード」「同じ修正の貼り重ね」「秘密情報の埋め込み」** といった事故も同じ速度で増えていきます。これを人間のレビューだけで止めるのは、現実的ではありません。

そこで効いてくるのが、**テストの階層化**と**コミット前の自動監査**を、仕組みとして開発フローに埋め込むことです。この記事では、実プロジェクトで運用している「テストの目的・分類」と「監査ルール・自動ブロック」の設計を、再利用できる形で整理します。

---

## 0. なぜ「テストコード」がガードレールなのか

### 前提が変わった ― レビューは破綻する

Agent coding も Vibe coding も、**人間が検知できない領域で既存のコードを壊します**。意図したファイルの隣で、呼び出し規約の裏で、誰も見ていない分岐で、静かに挙動が変わります。

そして決定的なのは、**生成量が上がれば上がるほど、人間が全部レビューすることは不能になる**という点です。1日に数十ファイルが書き換わる世界で「全 diff を人間が読んで担保する」という前提は、もう成立しません。崩れた前提を埋めるものが必要になります。

その担い手が**テストコード**です。テストは、生成 AI が触れた広い面のうち「壊れてはいけない挙動」を機械的に押さえ続けます。人間のレビューが追えなくなった領域を、テストが代わりに見張ってくれます。

### テストの目的は「違反しないこと」ではなく「変化を検知すること」

ここで誤解してはいけません。**テストコードに違反しないことが目的ではありません。** 目的は、**実装の変更を検知すること**です。

* テストを「通すべき制約」とだけ捉えると、AI はテストを通すために実装を歪めます。最悪の場合、テスト自体を書き換えて通してしまいます。
* 正しい捉え方は逆です。テストは**実装が意図せず変わったことを鳴らすセンサー**です。グリーンが目的なのではなく、「変わったら赤くなる」ことに価値があります。

この区別は、実運用で痛い形で学ばれてきました。たとえば——

* **「テストが落ちた＝テストが正しい」とは限りません。** 実装が正しくてテスト側の前提が古い、というケースもあります。だからテストは「実装と仕様、どちらが動いたか」を問う材料であって、盲目的に緑にする対象ではありません。
* **「実施する必要のないテストを skip にする」のは不具合扱いです。** スキップは「見張りを外す」ことであり、検知能力の穴になります。

テストを「センサー」と捉え直すと、**実装コードと一緒にテストコードを生成させる**ことが必然になります。実装だけ生成してテストを後回しにすると、その瞬間に生成された広い面が「見張りのない領域」になってしまいます。だからこそ——

> **Test First は徹底した義務です。** 実装コードを生成するなら、それを見張るテストコードも同時に生成します。例外を作りません。

### これは「手戻り」への予防策である

テストコードがないと何が起きるか。**同じ作業を何度もループする羽目になります。**

* 実装を変える → どこかが壊れる → 気づかず次へ進む → 後で発覚 → 原因を一から調査 → 直す → また別の所が壊れる、という無限ループ。
* 一度直したバグが、テストで固定されていないために再発する。「前にも直したはずの現象」を二度三度と踏む。

これを断つために、運用ルールとして\*\*「バグ修正には必ず回帰テストの path を残す」\*\*ことを最低契約にしている考え方があります。狙いは次の通りです。

* **根本原因と「潰した仮説」を記録します。** 次回の調査時間を短縮するためです。記録がないと、却下したはずの仮説をまた検証してしまいます。
* **回帰テストが「このバグを守っている assertion が今も生きているか」を保証します。** テストスイートを再編しても、過去のバグへの見張りが外れていないことを検証できます。

つまりテストは、**未来の自分（と AI）が同じ穴に落ちるのを防ぐ予防接種**です。手戻りのコストは、テストを書くコストより桁違いに大きくなります。

### ガードレールは「AI が覚えていること」に依存させない

最後に、設計上いちばん重要な原則を一つ挙げます。

> **テストは「ワークフローが要求するから走る」のであって、「エージェントが覚えているから走る」のではありません。**

AI に「テストを書いてね」と頼むだけでは、忘れます。プロンプトの長い文脈の中で抜け落ちます。だから**ガードレールは AI の記憶ではなく、仕組み（フック・契約・パイプライン）に埋め込みます**。具体的には、

* ターン終了時にフックが自動で走り、テストなしのコード変更を**ブロックします**。
* レビュー済みを示す成果物（diff ハッシュ + 合否）がないと次のコミットへ進めない**契約**を置きます。AI がレビューを忘れると、コミットが永久に止まります。
* 「テストを書く」ことを善意や記憶でなく、**バイパスできない関門**にします。

この「AI が回避できないルール」という発想こそが、生成量が人間のレビュー能力を超えた時代の品質設計の核心です。**人間が全部読めない以上、読まなくても壊れたら止まる仕組みを作る。** テストコードはその仕組みの中心に座ります。

## 0.5. レビューは破綻している ― だから「ましな手段」を多層で積む

前節で「生成量がレビュー能力を超える」と書きました。これは抽象論ではなく、**設計判断として一度結論が出ている**話です。

ガードレールを設計したとき、最初に検討して**捨てた案**が「全 diff を人間が承認する方式」でした。エディタ統合型 AI によくある、変更を人間が逐一レビューして approve するスタイルです。これを不採用にした理由は単純で——

> **対話性と生成速度を犠牲にしてでも自動化したいのに、全 diff の人間レビューはそれと正面衝突します。そして、そもそもスケールしません。**

つまり出発点からして「**レビューはもう破綻している**」を前提に置いています。人間が全部読んで品質を担保する、という素朴な方法は、最初から選択肢から外しました。

### では何を選んだか ― 完全解ではなく「ましな手段」の多層化

捨てた以上、代わりが要ります。選んだのは「**完璧な検証**」ではなく、「**性質の違う、不完全な検査を複数重ねる**」設計でした。重要なのは、これを**解決策とは呼んでいない**ことです。各層は穴を持ちます。穴の位置が違う検査を重ねることで、事故率を下げる——それ以上のことは主張しません。

層はおおむね次のように分かれます。

| 層 | 性質 | 役割 |
| --- | --- | --- |
| **静的検査** | 決定論 | 速い・誰がやっても同じ判定。red-flag を機械的に弾く |
| **意味論レビュー** | **確率** | AI による設計レビュー。patchwork や共有コード無視など、静的に判定できない文脈を見る |
| **回帰テスト** | 決定論 | 実装の意図せぬ変化を検知し続けるセンサー |

ここで意味論レビューの層を、設計上はっきり\*\*「確率」\*\*と注記している点が肝です。AI レビューは見落とします。同じ diff でも判定がぶれます。**だからこれは「保証」ではありません。** 完全保証でないと認めたうえで、決定論の層（静的検査・テスト）と重ねることで、片方の穴をもう片方が拾います。

「解決ではないが、ましな手段」とは、まさにこの**多層化の割り切り**を指しています。

### 「仕様とレビューさせ続ける」を仕組みにする ― policy-as-prompt

もう一つ、過去の設計に明確に残っているのが「**仕様を、AI に継続的にレビューさせ続ける**」という発想です。

素朴にやると、レビュー観点は人間が毎回プロンプトに書くことになり、ルールが増えると抜けます。そこで——

1. **仕様・ルールを1つの正典（宣言的な定義ファイル）に集約します。**
2. その定義から、**レビュー用のプロンプトを自動生成します。**
3. コミットのたびに、そのプロンプトで AI レビューを**当て続けます。**

ルールを1行足せば、AI レビューの評価基準も自動で更新されます。**「仕様」と「レビューの観点」が同じ1ソースから派生する**ので、両者がずれません。これを **policy-as-prompt**（ポリシーをプロンプトに変換する）と呼んでいます。

この仕組みのポイントは、**仕様との突合を「人間がそのつど思い出す作業」から外す**ことにあります。仕様は正典に書いてあります。レビューは毎コミット自動で当たります。人間が偉い・覚えている・気が利く、に依存させません。

### 認めるべきこと ― これは到達点ではない

正直に書くと、この設計は**到達点ではなく、現時点で取れる最善**にすぎません。実際、「フルに LLM で仕様合成まで自動化する」のは、**未着手の野心目標**として残されています。今あるのは、

* 静的検査で**確実に弾けるもの**は弾き、
* 確率的な AI レビューで**文脈の怪しさ**を拾い、
* テストコードで**意図せぬ変化**を鳴らし続ける、

という**不完全な検査の束**です。それでも、「人間が全 diff を読む」という破綻した前提に戻るよりは、はるかにましです。

結論は次のようになります。

> **レビューは破綻しています。ですが、テストコードを着実に積み、仕様と照らしてレビューを当て続けることは、解決ではないにせよ、確実にましな手段です。** そして「ましな手段」を、AI の善意や記憶ではなく**仕組み**として回し続けることが、生成 AI 時代の品質設計の現実解です。

## 1. テストの目的を「速度」で2層に分ける

テストをすべて同じタイミングで走らせようとすると破綻します。網羅テストやカバレッジ計測は時間がかかり、デプロイを遅らせるからです。そこで **目的別に2層へ分離**します。

### 第1層: デプロイ時テスト ―「動くか・安全か」だけを確認する

デプロイ経路で走らせるのは、**構成が動作しているか**と**セキュアか**の確認に限定します。カバレッジ計測や網羅的な E2E はここには入れません。

| カテゴリ | 目的 | タイミング | 目標時間 |
| --- | --- | --- | --- |
| **smoke** | デプロイ前の基本動作確認（ゲート）。失敗したらデプロイ中止 | デプロイ pre-flight | < 2分 |
| **install** | デプロイ直後の機能動作確認。通れば「稼働」とみなす | デプロイ post-verify | ~5分 |
| **audit** | セキュリティ監査・脅威検証 | デプロイ後・非同期 | ~5分 |
| **security** | injection / ペンテスト網羅 | デプロイ後・非同期 | ~10分 |

ポイントは2つです。

* **smoke がゲート、install が検証です。** smoke はデプロイを許可するための最低限の確認です。install が通れば「サービス稼働」とみなします。
* **セキュリティ検証は非同期です。** audit / security はデプロイをブロックせず、バックグラウンドで走らせます。デプロイ速度とセキュリティ姿勢の検証を両立させるための割り切りです。

### 第2層: 夜間バッチ ―「網羅」と「カバレッジ」をまとめて

時間のかかるものは全部デプロイ経路から切り離し、夜間に回します。

| カテゴリ | 目的 |
| --- | --- |
| **coverage** | インフラ非依存テスト + カバレッジ計測 |
| **periodic** | インフラ依存テスト（DB / 認証基盤が必要）。合否のみ |
| **e2e** | 機能 E2E + ブラウザ UI E2E |
| **scripts** | デプロイ/セットアップスクリプトの契約テスト |

夜間バッチは結果を日付つきの Markdown レポートに出力します。**遅くてよい代わりに網羅する**、というのが第2層の役割です。

### 設計上の学び

* **「テストの所要時間」は設計の制約として最初から扱います。** 速いテストと遅いテストを混ぜると、速いほうが遅いほうに引きずられて全体が回らなくなります。
* **インフラ依存テストは「到達できるホスト」で実行します。** 踏み台ホストから内部ポートに届かないなら、内部ホストへ入って現地で走らせます。到達不可なら CRASH ではなく SKIPPED にして、バッチ全体を落とさないようにします。
* **マーカーは自動付与します。** テストファイルの配置ディレクトリから種別マーカーを自動で付けると、書き手がマーカー付け忘れで分類を壊す事故が消えます。

## 2. コミット前監査 ― 「コード変更には必ずテスト」を仕組み化

ルールの中核はシンプルです。

> **コード変更には必ずテストを同時に追加・更新します。**

これを口約束でなく、**コミット前プレイブック**として手順化します。

### コミット前プレイブックの流れ

```
Phase A（順次）:
  1. 変更ファイルを列挙 → 対象ゼロなら早期終了

Phase B（並列・1回で同時発火）:
  2. テスト存在確認
  3. 静的監査（red-flag パターン検出）
  4. サブエージェントによる客観レビュー

Phase C（順次）:
  5. 指摘を統合 → 修正 → 再度 Phase B（指摘ゼロまで）
  6. テスト実行
  7. 報告 → コミット判断
```

### ここでの工夫

* **静的検査と LLM レビューを並列で発火します。** サブエージェントのレビューは LLM 呼び出しで数十秒かかります。grep ベースの静的検査（数秒）と**同じメッセージで同時に投げて**パイプライン化すれば、実時間は一番遅いレビュー1本分で済みます。
* **テストが書けないときは理由を残します。** インフラ依存で書けないなら、コード冒頭に `TODO: test (reason: ...)` を入れ、コミットメッセージにも記載します。「無言でスキップ」を許しません。
* **客観レビューは graceful degradation します。** サブエージェントが使えない環境では、その旨をログに出し「人手レビュー必要」と明示したうえで、コミット自体はブロックしません。検査の有無を**報告に正直に書く**ことを優先します。

## 3. コード監査チェックリスト

各 diff について、以下を機械的に確認します。

### 3-1. fall-through（失敗の握り潰し）防止

* 空の `except`、`|| true`、`2>/dev/null` で失敗を黙らせていないか。
* シェルは `set -euo pipefail` 下で動くか。
* **落とし穴**: `set -e` のシェルで `local VAR=$(cmd)` と書くと、`local` ビルトイン自体が rc=0 を返すため、`$(cmd)` が失敗しても `set -e` が発火しません。宣言と代入を分離します。

  ```
  # BAD — silent failure
  local x=$(some_cmd)

  # GOOD — 代入の rc が set -e に届く
  local x
  x=$(some_cmd)
  ```

### 3-2. hard-code 禁止

* IP / ホスト名 / ポート / パス / パスワード / API キーを直接埋め込まない。
* 非秘密の設定値は1つの正典設定ファイル、秘密は別の秘密ファイル経由のみで読む。
* **「環境変数を読む場所は1箇所だけ」を絶対原則にします。** それ以外の場所からの env 読み込みは、すべてバグ扱いです。

### 3-3. Docker / 本番配慮

* 本番コンテナのパスとローカル開発パスを取り違えていないか。
* 本番の実行ユーザー権限で動くか。
* 依存ライブラリを追加したら依存定義ファイルを更新したか。
* ビルド成果物がデプロイ経路に乗るか。

### 3-4. patchwork 検出

* 同じ症状を別の場所で再修正していないか（根本原因を見つけたか）。
* 「とりあえず動く」分岐を足していないか。
* 根本原因をコメント/docstring で説明できるか。

### 3-5. shared code 再利用

* 類似ロジックが既存の共通モジュールに無いか。
* 共通化すべきものを別ファイルに複製していないか。

## 4. 自動ブロック ― Stop フックで「静的 red-flag」を検出する

プレイブックを人間（や AI）が守るとは限りません。だから**ターン終了時に自動で走るフック**が、静的に検出できる赤信号を見つけてコミットをブロックします。実際に組み込んでいるチェックは多岐にわたります。

| チェック | 検出内容 |
| --- | --- |
| テスト欠落 | ソースが変わったのにテストが変わっていない |
| ハードコード IP | ドキュメント用予約レンジ（RFC 5737 等）以外の生 IP |
| ハードコード秘密 | `password = "..."` 等の埋め込み（env getter は除外） |
| 握り潰し | 空 `except` / `|| true` の追加 |
| 非正典 env 読み | 正典ファイル以外からの `.env` 読み込み |
| 重複関数 | 同名の関数定義が他ファイルにも存在（再利用を促す） |
| `set -e` の `local=$(...)` | 前述のサイレント失敗パターン |
| 構文/lint | 言語ごとの構文チェック・linter |
| 型チェック | 型エラー（増分キャッシュ利用で高速化） |
| リンク切れ | ドキュメント内の相対リンクが解決するか |
| 複雑度 | 循環的複雑度が閾値超え（AI コードは過剰ネストしがち） |
| 秘密スキャン | 専用ツールでの資格情報漏れ |
| 改行コード | プラットフォーム別の CRLF/LF 不整合 |
| ファイル長 | 一定行数超で**警告のみ**（ブロックはしない） |

### 設計判断のポイント

* **ブロックと警告を使い分けます。** ファイル長のような「望ましくないが致命的でない」ものは、exit code を変えない警告に留めます。既存の大きいファイルを大量にブロックすると、道具として使い物にならなくなります。
* **検出は「追加行」だけを対象にします。** diff の `+` 行のみスキャンすれば、既存コードの違反で新規変更が止まる事故を防げます。
* **誤検知の許容リストを実データで育てます。** 重複関数チェックは、CRUD 動詞やフレームワークのライフサイクル名（`init` / `setup` / `handle` など）を許容リストに入れます。監査ログを分析したら誤検知の2割がこうした汎用名だった、というように**実測でチューニングします**。
* **エラーメッセージに「次に何をすべきか」を書きます。** ブロック時は、修正方法・抑制マーカーの書き方・参照ドキュメントまで本文に含めます。

## 5. 抑制（escape hatch）― ただし「理由」を必須にする

絶対ブロックは現場を止めます。緊急時のための抜け道は用意しますが、**いずれも理由（`:` 以降の非空文字列）を必須**にします。理由なしの裸マーカーは、抑制として扱いません。

```
HOST = "203.0.113.10"  # audit-ignore: documentation example only
```

* **行単位**: 該当行に `audit-ignore: <理由>`
* **ファイル単位**: 先頭付近に `audit-ignore-file: <理由>`
* **ワンショット完全 bypass**: 理由を書いたファイルを置くと、フックが**読んだ瞬間に削除**して恒常化を防ぐ

設計の勘所は「**抜け道を塞ぐのではなく、抜けた記録を必ず残す**」ことです。理由必須にするだけで、安易な握り潰しが「説明責任のあるトレードオフ」に変わります。

---

## 6. まとめ ― この設計の背骨

1. **テストは速度で2層に分けます。** デプロイ時は「動くか・安全か」だけ、網羅とカバレッジは夜間へ。
2. **「コード変更には必ずテスト」を口約束でなく、手順とフックで強制します。**
3. **静的検査と LLM レビューを並列で回します。** 速い検査を遅い検査が待たないようにします。
4. **自動ブロックは追加行だけを見て、ブロックと警告を使い分け、誤検知を実測でチューニングします。**
5. **抜け道には必ず理由を残させます。**

AI が書く量が増えるほど、**「人間が全部読む」前提は崩れます**。崩れた前提を埋めるのは、根性ではなく**フローに織り込まれた検査の仕組み**です。レビューが破綻したことを認めたうえで、テストの階層化とコミット前監査という「ましな手段」を仕組みとして回し続けること——それが、生成 AI 時代の品質設計の現実解と言えます。
