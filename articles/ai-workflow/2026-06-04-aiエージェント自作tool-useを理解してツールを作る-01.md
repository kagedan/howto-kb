## この記事で作るもの

Step 04〜07で実装した「ツールを使うエージェント」の仕組みを解説します。

完成後はこのような流れで動作します。

```
ユーザー：「srcフォルダにあるファイルを教えて」
   ↓
LLM：list_files を呼び出す
   ↓
エージェント：ツールを実行してファイル一覧を取得
   ↓
LLM：結果をもとに「src/ には以下のファイルがあります…」と回答
```

ただの会話AIではなく、ファイルを読んだりコードを検索したりできるエージェントへの第一歩です。

---

通常のチャットとtool\_useの違いを図で整理します。

```
通常のチャット：ユーザー → LLM → 回答

tool_use：ユーザー → LLM → ツール呼び出し要求
                      ↑          ↓
                      └── 結果を受け取って再応答
```

LLMが「このリクエストにはツールが必要だ」と判断したとき、`stop_reason` が `"tool_use"` になります。エージェント側でツールを実行し、その結果をmessagesに追加してLLMに再度渡すと、今度は `"end_turn"` で最終回答が返ってきます。

---

## ツール定義のフォーマット

LLMに渡すツールはOpenAI互換フォーマットで記述します。`src/tool_definitions.py` の `read_file` 定義がわかりやすいので引用します。

src/tool\_definitions.py

```
TOOL_DEFINITIONS = [
    {
        "type": "function",
        "function": {
            "name": "read_file",
            "description": "指定したファイルの内容を読み込む",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {"type": "string", "description": "読み込むファイルのパス"}
                },
                "required": ["path"],
            },
        },
    },
    # ...
]
```

`name` と `description` がLLMの判断に直結します。descriptionが曖昧だとLLMが正しいツールを選択できないので、「何をするツールか」を端的に書くのがコツです。

---

`src/step04_tool_use.py` のループ処理がサイクルの核心です。messagesの構造がどう変化するかを図解します。

```
① {"role": "user", "content": "今何時ですか？"}
② {"role": "assistant", "tool_calls": [...]}   ← LLMのツール要求
③ {"role": "tool", "tool_call_id": "...", "content": "2026-05-30 12:00:00"}
④ → LLMが再度応答して end_turn
```

実装はこのループで実現しています。

src/step04\_tool\_use.py

```
while True:
    response = client.chat(messages=messages, tools=TOOLS)

    if response["stop_reason"] == "end_turn":
        return response["content"]

    if response["stop_reason"] == "tool_use":
        tool_calls = response["tool_calls"]

        # アシスタントのツール要求をmessagesに追加
        messages.append({
            "role": "assistant",
            "content": None,
            "tool_calls": [...],
        })

        # ツールを実行して結果をmessagesに追加
        for tc in tool_calls:
            result = dispatch_tool(tc["name"], tc["arguments"])
            messages.append({
                "role": "tool",
                "tool_call_id": tc["id"],
                "content": result,
            })
```

ポイントは「ツール要求→実行→結果追加」を `while True` で繰り返すことです。LLMが複数ツールを連続呼び出しするケースもこのループで自然にさばけます。

## 3本のツールを実装する

### read\_file / list\_files

src/tools/file\_tools.py

```
def read_file(path: str) -> str:
    try:
        file_path = Path(path)
        if not file_path.exists():
            return f"エラー：ファイルが見つかりません: {path}"
        return file_path.read_text(encoding="utf-8")
    except PermissionError:
        return f"エラー：ファイルへのアクセス権限がありません: {path}"
    except UnicodeDecodeError:
        return f"エラー：UTF-8で読み込めないファイルです: {path}"
```

エラーを例外で投げずに文字列で返しているのがポイントです。LLMはツール結果を文字列として受け取るので、エラーも文字列にしておけば「ファイルが見つかりませんでした」とLLMが読者に伝えられます。

### search\_text

src/tools/search\_tools.py

```
MAX_RESULTS = 50
```

## ディスパッチャで自動化する

`src/dispatcher.py` の `TOOL_REGISTRY` と `dispatch()` がツール追加を簡単にする設計の肝です。

src/dispatcher.py

```
TOOL_REGISTRY: dict = {
    "read_file": read_file,
    "list_files": list_files,
    "search_text": search_text,
}

def dispatch(tool_name: str, arguments: str | dict) -> str:
    args = json.loads(arguments) if isinstance(arguments, str) else arguments

    if tool_name not in TOOL_REGISTRY:
        return f"エラー：未知のツールです: {tool_name}"

    func = TOOL_REGISTRY[tool_name]
    return func(**args)
```

新しいツールを追加するときは `TOOL_REGISTRY` に1行追加するだけです。`dispatch()` 自体には触らなくてよいので、ツールが増えても呼び出し側のコードが変わりません。

## 躓いた点・気づき

**qwen2.5-coder:7b がtool\_use非対応だった**

最初に使っていた `qwen2.5-coder:7b` は、Ollamaのtool\_use要求に対して `finish_reason=stop` でJSONをそのまま平文返却していました。tool\_useが動くのは `llama3.1:8b` や `qwen3.5:9b` です。使用するモデルで動作確認を必ずとることが大切です。

**システムプロンプトにカレントディレクトリを明示する必要があった**

`llama3.1:8b` でread\_fileを試したとき、ファイルパスを `/path/to/src/...` というプレースホルダーで返すことがありました。システムプロンプトに「カレントディレクトリは `C:\work\agent01` です」と明示したところ、実際のパスを使うようになりました。

**Windows環境でのエンコーディング問題**

Step 07でREPLに日本語を入力するとサロゲート文字化けが発生しました。`sys.stdin.reconfigure(encoding="utf-8")` をスクリプト先頭に追加することで解決しています。Step 01の `sys.stdout.reconfigure` とペアで設定するのが定番です。
