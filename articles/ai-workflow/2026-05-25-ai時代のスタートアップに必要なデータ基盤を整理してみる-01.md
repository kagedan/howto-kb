こんにちは、[@aymkbyshi](https://x.com/aymkbyshi) です。

普段はDatabricksで、スタートアップのデータ・AI活用を支援しています。

スタートアップの皆さまと話していると、生成AIやAIエージェントの活用について相談をいただく機会が増えています。

一方で、実際に話を深掘りしていくと、多くの場合、論点はモデル選びだけではありません。

「AIにどのデータを見せるのか」  
「そのデータは信頼できるのか」  
「権限や参照範囲をどう制御するのか」  
「今のフェーズで、どこまで整えるべきなのか」

こうした問いに行き着きます。

この記事では、AI時代におけるデータ基盤がこれまでと何が違うのか、特にスタートアップの視点で整理していければと思います。

## なぜ今、スタートアップにデータ基盤が必要なのか

「AI時代にはデータ基盤が重要だ」と言われる機会が増えています。

一方で、スタートアップにとって **データ基盤** と聞くと、少し大げさに感じるかもしれません。

たとえば、次のような言葉が並ぶと、どうしても大企業向けの仕組みに見えます。

* DWH
* ETL
* データレイク
* BI
* ガバナンス
* データカタログ

しかし、AI時代のデータ基盤は、従来のような大規模分析組織だけのものではありません。

初期のスタートアップでは、データはプロダクトDB、CRM、スプレッドシート、Slack、Notion、問い合わせツールなどに分散しがちです。

最初はそれでも回ります。

しかし、生成AIやAIエージェントをプロダクトや業務に組み込もうとすると、この分断がそのままAI活用の限界になります。

重要なのは、最初から完璧な基盤を作ることではありません。

**AIが安全に参照し、解釈し、活用できる形でデータを整え始めること** です。

## これまでのデータ基盤：見える化のための基盤

![](https://static.zenn.studio/user-upload/772fd975c567-20260525.png)

従来のデータ基盤は、多くの企業において **人間が状況を把握し、意思決定するための仕組み** でした。

### 主な用途

| 領域 | 見ていたデータ |
| --- | --- |
| 経営 | 売上、MRR、チャーン、ARR |
| プロダクト | 利用状況、継続率、機能別利用率 |
| 営業 | 商談数、受注率、パイプライン |
| マーケティング | CPA、CVR、広告効果 |
| CS | 問い合わせ数、ヘルススコア、解約リスク |

これらは今でも非常に重要です。

特にスタートアップでは、意思決定の速度が競争力になります。

* プロダクトが伸びているのか
* どの顧客が定着しているのか
* どの施策が効いているのか
* どこに成長機会があるのか

こうした問いに素早く答えられることは、事業成長に直結します。

ただし、従来のデータ基盤の主な目的は、あくまで **人間が見て判断すること** でした。

## AI時代のデータ基盤：AIが使うための基盤

![](https://static.zenn.studio/user-upload/cb98df769aea-20260525.png)

AI時代に大きく変わるのは、データの使い手です。

これまでは、人間がデータを見て判断していました。

これからは、AIがデータを読み、解釈し、提案し、場合によっては人間の承認を挟みながらアクションまで支援します。

### AI活用の具体例

| AIの種類 | 活用イメージ |
| --- | --- |
| 営業AI | 商談前に顧客の利用状況、契約情報、過去の問い合わせを要約する |
| CS向けAI | 解約リスクを検知し、次に取るべき対応を提案する |
| プロダクト内AI | ユーザー行動に応じてパーソナライズされた提案を出す |
| 経営向けAI | KPIの異常値や成長機会を自然言語で説明する |

このとき必要になるのは、単なるダッシュボードではありません。

必要なのは、AIが次のことをできる状態です。

* 必要なデータに安全にアクセスできる
* 正しい文脈でデータを解釈できる
* 業務やプロダクトの中で活用できる
* 根拠を追跡できる
* 権限に応じて参照範囲を制御できる

> **つまり、データ基盤は「分析のための基盤」から「AI活用のための基盤」へ変わりつつあります。**

## AIエージェントを動かすには、データ基盤が必要になる

![](https://static.zenn.studio/user-upload/a0dac966c642-20260525.png)

この変化を考えるうえで、特に重要になるのが **AIエージェント** です。

AIエージェントとは、単に質問に答えるAIではなく、目的に応じて情報を集め、状況を判断し、次のアクションを支援するAIです。

たとえば、スタートアップでは次のような使い方が考えられます。

| エージェント | 活用イメージ |
| --- | --- |
| 営業エージェント | 商談前に顧客情報、利用状況、契約情報、問い合わせ履歴を整理する |
| CSエージェント | 問い合わせ内容を読み取り、過去の対応や利用状況をもとに回答案を作る |
| オンボーディングエージェント | 顧客の進捗やつまずきを見て、次に案内すべき内容を提案する |
| 社内ナレッジエージェント | Slack、Notion、Docsなどから必要な情報を探し、要約する |
| 経営支援エージェント | KPIの変化を検知し、原因や次の打ち手を整理する |

ここで重要なのは、AIエージェントの品質は **モデルだけでは決まらない** ということです。

たとえば営業エージェントであれば、商談前に次のような流れで動くことが考えられます。

1. CRMから企業情報と商談履歴を取得する
2. プロダクトログから直近の利用状況を確認する
3. CSツールから問い合わせ履歴を確認する
4. 過去の類似顧客の成功事例を探す
5. 次回商談で確認すべき論点と提案仮説をまとめる

この流れを実現するには、顧客情報、契約情報、利用ログ、問い合わせ履歴、商談メモ、社内ナレッジがつながっている必要があります。

また、AIが参照してよい情報を権限ごとに制御し、どの情報を根拠に提案したのかを追跡できることも重要です。

つまり、AIエージェントは単体で賢く動くわけではありません。

**企業のデータとナレッジにつながって初めて、業務で使える存在になります。**

スタートアップでは、いきなり大きなAIエージェントを作る必要はありません。

まずは、営業準備、問い合わせ対応、社内ナレッジ検索、オンボーディング支援など、業務の一部から始めるのが現実的です。

> **AIエージェントを動かすには、モデルだけでなく、エージェントが参照・判断・実行できるデータ環境が必要です。**

## AIに必要なのはデータ量よりコンテキスト

![](https://static.zenn.studio/user-upload/98b9b4ce1d71-20260525.png)

AI活用でよくある失敗は、次のように考えてしまうことです。

> 社内データをつなげば、AIは勝手に賢くなる。

しかし実際には、データを渡すだけでは不十分です。

AIに必要なのは、データそのものだけでなく、**データの意味や使い方まで含めたコンテキスト** です。

### 例：利用率が下がっている顧客

ある顧客の利用率が下がっているとします。

この数字だけを見ても、理由は分かりません。

考えられる理由は複数あります。

* 解約リスクが高まっている
* 季節要因で一時的に利用が減っている
* 契約更新前に利用を見直している
* 導入初期のオンボーディングで停滞している
* 担当者変更により利用が止まっている

AIが有益な提案を出すには、次の情報も必要になります。

* 営業履歴
* CSメモ
* 契約条件
* 業界特性
* 過去の類似ケース
* 顧客の組織体制
* 直近の問い合わせ内容

つまり、AIに渡すべきなのは単なるデータではありません。

**そのデータをどう解釈すべきかという文脈** です。

そのためAI時代のデータ基盤では、データを保存するだけでなく、IDの紐づけ、メタデータ、業務定義、検索、権限、履歴を含めて設計する必要があります。

## 構造化データと非構造化データをつなぐ

![](https://static.zenn.studio/user-upload/a59d4d608218-20260525.png)

これまでのデータ活用では、主に構造化データが中心でした。

### 構造化データの例

* プロダクトログ
* 売上
* 契約
* 広告データ
* CRM
* 決済データ

しかし、AI活用ではそれだけでは足りません。

スタートアップの重要な知識は、むしろ非構造化データの中に散らばっています。

### 非構造化データの例

* 顧客との商談メモ
* Slackでの議論
* NotionやGoogle Docsの仕様書
* CSの問い合わせ履歴
* 営業資料
* オンボーディング資料
* プロダクト改善要望
* 失注理由
* 導入事例

これらは、従来のBIでは扱いにくい情報でした。

しかし生成AIにとっては、むしろ重要なコンテキストになります。

生成AIでは、こうした非構造化データを検索し、関連する情報をAIに渡すことで、より業務文脈に沿った回答や提案ができるようになります。

> **非構造化データは、単なる補足情報ではありません。AIが業務を理解するための重要な材料です。**

## スタートアップは小さく始めればいい

![](https://static.zenn.studio/user-upload/c71199810328-20260525.png)

ここで重要なのは、スタートアップがいきなり大企業のようなデータ基盤を作る必要はない、ということです。

創業初期から複雑な設計をしすぎると、むしろスピードを失います。

### 最初からやりすぎなくていいもの

* 複雑なDWH設計
* 厳密なデータカタログ
* 全社的なガバナンス体制
* 大規模な分析組織
* 完璧なデータ品質管理

必要なのは、完璧な基盤ではありません。

**成長に合わせて拡張できるデータ環境** です。

### 小さく始めるためのステップ

| フェーズ | やること |
| --- | --- |
| まず整える | プロダクトDB、決済データ、CRM、主要KPIを見られる状態にする |
| 次につなぐ | 問い合わせ履歴、広告データ、オンボーディング状況を連携する |
| 早めに決める | 顧客ID、企業ID、契約ID、ユーザーIDの紐づけ方を決める |
| 後から強化する | データカタログ、品質管理、全社ガバナンスを必要に応じて整える |

小さく始めるとは、何もしないという意味ではありません。

将来AIが使えるように、最低限のデータをつなぎ、あとから拡張できる状態を作っておくことです。

最初から100点を目指す必要はありません。

ただし、あとからAI活用に進めるように、データをバラバラに放置しすぎないことが重要です。

## 権限管理と安全性は後回しにしすぎない

![](https://static.zenn.studio/user-upload/d91619a5551d-20260525.png)

スタートアップでは、スピードを優先するあまり、権限管理やガバナンスが後回しになりがちです。

しかし、AIを導入すると状況が変わります。

AIは複数のデータソースを横断して情報を取り出します。

これは便利である一方、リスクもあります。

### 起こり得るリスク

| リスク | 例 |
| --- | --- |
| 情報漏えい | 本来見せるべきでない契約条件が回答に含まれる |
| 個人情報の露出 | 人事情報や個人情報が社内AIの回答に出る |
| 顧客向けAIの誤参照 | 顧客向けAIが内部メモを参照してしまう |
| 権限無視 | 営業担当者が見られないはずの情報をAI経由で見られる |

そのため、最初から完璧である必要はありませんが、最低限次の設計は必要です。

* 誰が何を見られるのか
* AIがどのデータを参照してよいのか
* 顧客向けAIと社内向けAIで使うデータをどう分けるのか
* 回答の根拠をどう追えるようにするのか

AI活用が進むほど、データ基盤には「つなぐ力」だけでなく、**守る力** も求められます。

特に、社内向けAIと顧客向けAIでは、参照してよいデータの範囲を明確に分ける必要があります。

## データ基盤はプロダクト競争力になる

これまでデータ基盤は、社内の分析や経営管理のための裏側の仕組みと見られがちでした。

しかしAI時代には、データ基盤そのものがプロダクトの競争力になります。

同じLLMを使っていても、裏側のデータ基盤によってユーザー体験は大きく変わります。

たとえばSaaSスタートアップであれば、次のようなAI機能が考えられます。

* ユーザーごとの利用履歴をもとに最適な提案をする
* 業務データを読み解いて次のアクションを出す
* ユーザー企業内のナレッジを検索・要約する
* 異常やリスクを自動で検知する
* 過去の操作履歴から業務改善案を提示する

こうしたAI機能の品質は、モデルだけでは決まりません。

むしろ、次のような要素によって差が出ます。

* どれだけ良いデータを持っているか
* どれだけ正しい文脈でAIに渡せるか
* 検索精度が高いか
* 権限管理が適切か
* 履歴を活用できるか
* フィードバックループがあるか

ここでいうフィードバックループとは、AIの回答に対するGood / Bad、修正内容、実際の成果などを蓄積し、次の改善に活かす仕組みです。

つまり、AI時代のスタートアップにとって、データ基盤は単なる社内ITではなく、**プロダクト価値を支えるコア資産** になります。

スタートアップがAI時代のデータ基盤を考えるとき、最初から大きな構想を描きすぎる必要はありません。

まずは、次の問いから始めるのが現実的です。

### チェックリスト

このチェックリストをすべて一度に満たす必要はありません。

まずは、自社にとって最もインパクトの大きいユースケースを一つ選び、そこに必要なデータから整えていくのがよいと思います。

## Databricksで始めるという選択肢

ここまで見てきたように、AI時代のデータ基盤では、構造化データだけでなく、非構造化データ、ガバナンス、権限管理、AI活用までを一体で考える必要があります。

その選択肢の一つとして、Databricksがあります。

Databricksは、Data Intelligence Platformとして、データ、分析、AIアプリケーションを一つのプラットフォーム上で扱える環境を提供しています。

### スタートアップにとっての利点

| 観点 | メリット |
| --- | --- |
| データ統合 | 構造化・非構造化データを一つの基盤で扱いやすい |
| AI活用 | BI、機械学習、生成AI、AIエージェントまで同じデータ基盤上で展開しやすい |
| ガバナンス | Unity Catalogで権限、監査、リネージを一元管理しやすい |
| アプリ化 | Databricks AppsやModel Servingを使って、データ・AIアプリケーションに展開しやすい |
| 成長対応 | 小さく始めて、データ量・ユーザー数・ユースケースの拡大に合わせて広げやすい |

たとえばDatabricksでは、Unity Catalogを使ってデータやAI資産に対するアクセス制御、監査、リネージ管理を一元的に扱えます。

また、Databricks AppsやModel Servingを活用することで、分析やPoCにとどまらず、データ・AIアプリケーションとして業務やプロダクトに組み込んでいくこともできます。

もちろん、すべてのスタートアップが初日から大きなデータ基盤を作る必要はありません。

ただ、AIを単なる実験で終わらせず、プロダクトや業務に組み込んでいきたい場合、データとAIを一体で扱える基盤を早い段階から意識しておくことは大きな意味があります。

> **AIを単なる実験で終わらせず、プロダクトや業務に組み込んでいきたいスタートアップにとって、DatabricksはデータとAIをつなぐ実践的な選択肢になり得ます。**

## まとめ

AI時代のデータ基盤は、大企業だけの話ではありません。

むしろ、スピードとプロダクト価値で勝負するスタートアップにこそ重要です。

ただし、必要なのは重厚長大な基盤ではありません。

重要なのは、次の3つです。

1. **AIが使える形でデータとナレッジを整えること**
2. **事業成長に合わせて拡張できる状態を作ること**
3. **権限管理や文脈設計を早い段階から意識すること**

従来のデータ基盤が **人間が見るためのもの** だったとすれば、これからのデータ基盤は **AIが使い、業務やプロダクトを動かすためのもの** です。

AI活用の成否は、モデル選びだけでは決まりません。

重要なのは、そのAIにどれだけ信頼できるデータと文脈を渡せるかです。

これからのスタートアップにとって、データ基盤は「いつか必要になるもの」ではなく、AIを競争力に変えるために、早い段階から少しずつ育てていくべきものです。
