2026年1月30日、AIエージェントフレームワーク Mastra の バージョン1.1がリリースされました（[ニュースリリース](https://mastra.ai/blog/changelog-2026-01-30)）。今回はバージョン1.1の最大の新機能「Workspace」機能を紹介・解説しつつ、なぜこのタイミングで Mastra が この機能に力を入れるのか、個人的な想像を踏まえて解説します。なおWorkspace機能の公式ドキュメントは[こちら](https://mastra.ai/docs/workspace/overview)です。

## Mastra の Workspace機能とは

Mastra の Workspace は、簡単に言うとエージェントが「ファイルを読み書きし、コマンドを実行し、そして検索することもできる」というものです。

Workspaceは以下の機能で構成されます。

* [**ファイルシステム**](https://mastra.ai/docs/workspace/filesystem/llms.txt): ファイル保存（読み取り、書き込み、一覧、削除）
* [**サンドボックス**](https://mastra.ai/docs/workspace/sandbox/llms.txt): コマンド実行（シェルコマンド）
* [**検索**](https://mastra.ai/docs/workspace/search/llms.txt): インデックスされたコンテンツに対する BM25 / ベクトル / ハイブリッド検索
* [**スキル**](https://mastra.ai/docs/workspace/skills/llms.txt): Agent Skills、つまりエージェント向けの再利用可能な指示

これまでの Mastra エージェントは ファイルシステムにアクセスしてファイルを読み書きしたり、コマンド実行したり、あるいは検索したりすることはできませんでした。いや正確には、専用のToolを実装すればできたのでしょうが、それを Mastra として正式に標準化した形でサポートするようになった、ということです。

## ざっくり、これで何ができるようになるか

例えば、ユーザのプロンプトに対して、エージェントがPython なり TypeScript なり シェルスクリプト なりをコーディングしてソースコードのファイルを Mastra サーバのどこかの領域に保存することができます。そして、そのコードを実行して、結果を確認して、ユーザに回答することができます。

また、Agent Skills をサポートするようになったので、特定の指示に対応する手順が長くて複雑な場合に、あらかじめその手順を SKILL.md に定義しておくことで、コンテキストを手順書で圧迫することなく対応することができるようになります。

ここまでの機能自体は、そんなに難しいことではないですよね。すでにVSCodeなどでは、ユーザのプロンプトを受けて必要に応じてコードを書いて自動実行したりするし、Agent Skills もサポートしています。最近のChatGPTだって、質問によってはPythonコードを書いて実行したうえで回答を返してくれます。それと同じことが、Mastra でも気軽に実現できるようになった、ということです。

### 解説

LLMだけの場合、「83個です（えっへん）」と自信たっぷりに答えていますが、明らかに誤っています。例えば、最後の「4030」がどう考えても素因数に 2や5を含んでいるのでNGです。

一方、filesystem-sandbox-agent の方は、以下のPythonコードを生成・実行したうえで、その結果を踏まえて正しく回答をしていました。

```
import itertools
import math

primes = [7,11,13,17,19]

# generate all numbers of form 7^a*11^b*13^c*17^d*19^e in [1000,9999]
nums=set()
# exponents upper bounds
max_exp=[int(math.log(9999,p)) for p in primes]
for exps in itertools.product(*[range(m+1) for m in max_exp]):
    n=1
    for p,e in zip(primes,exps):
        n*=p**e
        if n>9999:
            break
    if 1000<=n<=9999:
        nums.add(n)

nums=sorted(nums)
print(len(nums))
print(nums)
```

## 改めて、Workspace機能の意義

Mastra公式YouTubeチャンネルの [2026年1月7日のこちらの動画](https://www.youtube.com/live/8gyjaP0ZPRs?si=x26oXzc7JDcR_KY0) で、2026年のAIエージェントのトレンドとして、「ラルフ・ウィガン・ループ」と「ファイルシステムやサンドボックス」が挙げられていましたが、まさに 1月30日のバージョン1.1へのアップデートは後者についてMastra標準機能として対応したものでした。

「ラルフ・ウィガン・ループ」については、本連載の中でちょうど[前回](https://zenn.dev/shiromizuj/articles/6eb3e898b98d58)、解説しました。一言でいえば、条件を満たすまで何度も繰り返しAIエージェントに同じタスクを取り組ませ、改善に改善を重ねさせる手法でした。

「ラルフ・ウィガン・ループ」と「ファイルシステムやサンドボックス」は一見あんまり関係ない話のように見えて実は密接に関連していて、というか同じ文脈の中にあって、それはつまり「**複雑で長大なタスクを、エージェントが自律的に実行する**」という世界を実現したい、ということだと思います。「ラルフ・ウィガン・ループで何度もPDCAを繰り返す」というアプローチを基本方針としつつ、「ではその際のコンテキスト爆発にどう対応するのか」という点の答えの一つが今回のWorkspaceです。

多くのエージェント間の引き継ぎ事項を、全部メモリに載せるのではなくファイルに残し必要なものだけを読む、というアプローチが**ファイルシステム**機能につながっています。またLLMに読み込ませるにはあまりにもデータ量が多すぎるようなタスクを、スクリプトを書いて実行することでコンテキストを節約する、というアプローチが**サンドボックス**機能につながっています。複雑な手順が多数ある場合の標準的なアプローチが**Agent Skills**です。

そうなると、今回合わせてWorkspace機能として提供された**ファイル検索**機能もその文脈で理解できます。この機能は、ファイルシステム中に格納された**ファイルをインデックス化し、全文検索またはベクトル検索**することができる、というもので、**それって要するにRAGやん**、RAGはすでにMastraはサポートしてるやん、と思えます。どうしてそれが今回のWorkspace機能に含まれることになったのでしょうか。それは、「エージェントがファイルを書いて、プログラムを実行して、ファイルを溜めて、そして**検索して活用する**」というエージェント自律化計画の一角を担っているからだ、と個人的に理解しています。
