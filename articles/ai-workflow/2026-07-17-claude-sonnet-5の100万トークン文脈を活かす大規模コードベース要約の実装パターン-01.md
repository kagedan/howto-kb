## なぜ100万トークンが効くのか

従来はコンテキスト長の制約で、リポジトリ全体を一度に読ませることが困難でした。ファイルを小分けにして要約→再要約する多段パイプラインが必要で、ファイル間の依存関係が失われがちでした。100万トークン（=概ね数十万行規模のコードに相当）を一度に扱えると、ファイルをまたいだ参照解決やアーキテクチャ全体の一貫したレビューが1パスで可能になります。

## 実装：リポジトリを1リクエストにまとめる

対象リポジトリのソースを収集し、トークン数を見積もりながら1つのプロンプトに束ねます。

```python
from pathlib import Path

EXTS = {".py", ".js", ".ts", ".go", ".rs", ".java"}
IGNORE = {"node_modules", ".git", "dist", "build", "__pycache__"}

def collect_source(root: str):
    files = []
    for path in Path(root).rglob("*"):
        if path.is_file() and path.suffix in EXTS:
            if any(part in IGNORE for part in path.parts):
                continue
            try:
                files.append((str(path.relative_to(root)), path.read_text("utf-8")))
            except (UnicodeDecodeError, OSError):
                continue
    return files

def estimate_tokens(text: str) -> int:
    return int(len(text) / 3.5)
```

収集したファイルを、パス見出し付きで結合します。

```
def build_context(files, budget=900_000):
    blocks, used = [], 0
    for rel_path, source in files:
        block = f"\n\n===== FILE: {rel_path} =====\n{source}"
        cost = estimate_tokens(block)
        if used + cost > budget:
            break
        blocks.append(block)
        used += cost
    return "".join(blocks), used
```

## プロンプト構成：役割・タスク・出力形式を分離する

長文文脈では指示が埋もれやすいため、システム側で役割と出力形式を固定し、ユーザー側にコードを置きます。

```
import anthropic

client = anthropic.Anthropic()

SYSTEM = "シニアアーキテクトとして、リポジトリ全体を読み、アーキテクチャ概要・依存関係・リスク上位5件を日本語のMarkdownで出力する"

def review_repo(context: str) -> str:
    resp = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=4096,
        system=SYSTEM,
        messages=[{"role": "user", "content": context}],
    )
    return resp.content[0].text
```

## 精度とコストを両立させる3つの工夫

1. **予算を9割に抑える**：上限ギリギリまで詰めると出力用トークンが枯渇します。入力は上限の80〜90%に。
2. **重要ファイルを先頭に置く**：モデルは文脈の前後端に注意が偏りがち。エントリポイントや設定ファイルを先頭へ。
3. **出力を構造化させる**：自由記述より「ファイル:行」形式を強制したほうが後段の自動処理に回しやすい。
