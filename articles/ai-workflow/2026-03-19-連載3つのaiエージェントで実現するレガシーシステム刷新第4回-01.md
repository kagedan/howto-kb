## 実装計画 — Coding Agent for ScalarDB で実装ロードマップを策定する

---

### はじめに

前回は Architecture Redesign Agent で DDD に基づく再設計とコード生成を行いました。境界コンテキストが定義され、ターゲットアーキテクチャが描かれ、Spring Boot + ScalarDB のコード骨格まで生成されました。

しかし、コード骨格があるだけでは実装には着手できません。データモデルの詳細設計、トランザクション戦略、インフラ構成、セキュリティ、可観測性、テスト戦略、デプロイ計画——これらを漏れなく策定し、チーム全体が共有できるドキュメントにする必要があります。

今回は、この「設計と実装の間のギャップ」を埋める **Coding Agent for ScalarDB** を解説します。

---

### Coding Agent for ScalarDB とは

前の2つのエージェントとは設計思想が異なります。

| 特徴 | Refactoring Agent / Architecture Redesign Agent | Coding Agent for ScalarDB |
| --- | --- | --- |
| 実行方式 | パイプライン（自動実行） | 対話的（ユーザーと協働） |
| 入力 | コードベース | 要件（ユーザーが提供） |
| レビュー | なし | 5観点並列レビュー + 最終判定 |
| モデル戦略 | 単一モデル | マルチモデル（haiku/sonnet/opus） |
| 出力 | 分析レポート + コード | 設計ドキュメント（コード生成なし） |

最大の特徴は、各フェーズ完了時に **5つの異なる観点からの並列レビュー** が組み込まれている点です。設計ドキュメントの品質を自動的に検証し、問題があれば修正を促します。

### セットアップ

```
git clone https://github.com/wfukatsu/coding-agent-for-scalardb.git
cd coding-agent-for-scalardb
```

Python や Node.js の追加インストールは不要です。Claude Code が動作する環境であれば、そのまま使えます。

#### 前ステージの成果物を活用する

Architecture Redesign Agent の出力を参照できるようにしておくと、要件入力時に具体的な情報を提供できます。

```
# 前ステージの成果物を参照用にコピー
cp -r ../architecture-redesign-agent/reports/ ./reference/
cp -r ../architecture-redesign-agent/generated/ ./reference/
```

### 4フェーズ・13ステップのワークフロー

Coding Agent for ScalarDB のワークフローは、4つのフェーズに分かれた13のステップで構成されます。

```
Phase 1: 要件定義
  Step 01: 要件分析
  Step 02: ドメインモデリング
  Step 03: ScalarDB スコープ決定

Phase 2: 設計
  Step 04: データモデル設計
  Step 05: トランザクション設計
  Step 06: API・インターフェース設計

Phase 3: インフラストラクチャ
  Step 07: インフラ設計
  Step 08: セキュリティ設計          ─┐
  Step 09: 可観測性（Observability）設計 ├─ 並列実行可能
  Step 10: DR（災害復旧）設計         ─┘

Phase 4: 実行計画
  Step 11: 実装ガイド
  Step 12: テスト戦略
  Step 13: デプロイ・ロールアウト計画
```

Phase 3 の Step 08-10 は互いに独立しているため、並列実行が可能です。

---

### 5つのスラッシュコマンド

```
/plan                  # ワークフロー管理、次のステップの提案
/plan-step <N>         # 特定のステップを実行（例: /plan-step 01）
/plan-review <phase>   # フェーズレビュー（例: /plan-review phase1）
/plan-review final     # 最終GO/NO-GO判定
/plan-status           # 進捗ダッシュボード
```

### 実際のワークフロー — Step by Step

#### Step 01: 要件分析

エージェントが対話的に要件を聞き取ります。前ステージの Architecture Redesign Agent の出力がある場合、それを参照しながら具体化します。

```
エージェント: プロジェクトの概要を教えてください。
  - どのようなシステムですか？
  - 主要なユースケースは何ですか？
  - 非機能要件（性能、可用性）はありますか？

ユーザー: 注文管理システムのマイクロサービス化です。
  Architecture Redesign Agent で以下の境界コンテキストが定義されました：
  - 注文管理コンテキスト
  - 顧客管理コンテキスト
  - 商品管理コンテキスト
  ...
```

エージェントはヒアリング結果を構造化し、要件定義ドキュメントとして出力します。

#### Step 02: ドメインモデリング

Step 01 の要件を基に、ドメインモデルを詳細化します。Architecture Redesign Agent が出力した境界コンテキスト設計をベースに、ScalarDB での実装を見据えたモデリングを行います。

* エンティティと値オブジェクトの詳細定義
* 集約の境界確認
* ドメインイベントの設計
* コンテキスト間の統合パターン

#### Step 03: ScalarDB スコープ決定

ScalarDB をどの範囲で使うかを決定する重要なステップです。

* ScalarDB を使うサービスと使わないサービスの切り分け
* トランザクションモード（Consensus Commit / JDBC）の選択
* マルチストレージ構成の要否
* ScalarDB Analytics の必要性

ここでの判断が後続の全ステップに影響するため、慎重に進めます。

---

### Phase 1 レビュー — 5観点並列レビューの実際

Phase 1（Step 01-03）が完了したら、レビューを実行します。

このコマンドにより、**5つの異なる観点**からの並列レビューが実行されます。

```
┌─────────────────────────────────────────────────┐
│                 Phase 1 レビュー                   │
│                                                   │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐      │
│  │ 整合性    │ │ ScalarDB  │ │ 運用性    │       │
│  │ レビュー   │ │ 適合性    │ │ レビュー   │       │
│  │ レビュアー │ │ レビュアー │ │ レビュアー │       │
│  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘      │
│        │             │             │              │
│  ┌─────┴─────┐ ┌─────┴─────┐                     │
│  │ セキュリティ│ │ 実装可能性│                      │
│  │ レビュアー │ │ レビュアー │                      │
│  └─────┬─────┘ └─────┬─────┘                      │
│        │             │                             │
│        └──────┬──────┘                             │
│               ▼                                    │
│        レビュー統合レポート                           │
│        PASS / CONDITIONAL PASS / FAIL              │
└─────────────────────────────────────────────────┘
```

#### 5つのレビュー観点

| # | 観点 | チェック内容 |
| --- | --- | --- |
| 1 | **整合性** | ステップ間の矛盾はないか。用語は統一されているか |
| 2 | **ScalarDB適合性** | ScalarDB のベストプラクティスに沿っているか |
| 3 | **運用性** | 本番環境で運用可能な設計か |
| 4 | **セキュリティ** | セキュリティ上の懸念はないか |
| 5 | **実装可能性** | エンジニアがこのドキュメントで実装を開始できるか |

#### レビュー結果の例

```
=== Phase 1 レビュー結果 ===

整合性:         PASS
ScalarDB適合性: CONDITIONAL PASS
  → Step 03 のトランザクションモード選定理由が不明確。
    Consensus Commit を選んだ根拠を追記してください。
運用性:         PASS
セキュリティ:    PASS
実装可能性:     CONDITIONAL PASS
  → Step 02 の集約境界が曖昧。OrderItem の独立アクセス
    可否を明記してください。

総合判定: CONDITIONAL PASS
  → 2件の指摘事項を解消後、Phase 2 に進んでください。
```

CONDITIONAL PASS の場合、指摘事項を修正してから次のフェーズに進みます。

### Phase 2-4 の概要

#### Phase 2: 設計

```
/plan-step 04  # データモデル設計
/plan-step 05  # トランザクション設計
/plan-step 06  # API・インターフェース設計
/plan-review phase2
```

**Step 04: データモデル設計** では、ScalarDB のスキーマ設計を詳細に行います。テーブル定義、パーティションキー/クラスタリングキーの設計、セカンダリインデックスの要否など。

**Step 05: トランザクション設計** では、分散トランザクションのパターンを設計します。2PC（Two-Phase Commit）の適用範囲、サガパターンの必要性、冪等性の保証方法など。

**Step 06: API設計** では、サービス間のインターフェースを設計します。REST / gRPC / AsyncAPI の選定とOpenAPI仕様の策定。

#### Phase 3: インフラストラクチャ

```
/plan-step 07  # インフラ設計
/plan-step 08  # セキュリティ設計
/plan-step 09  # 可観測性設計
/plan-step 10  # DR設計
/plan-review phase3
```

Step 08-10 は並列実行可能です。それぞれが独立した設計ドキュメントを出力します。

#### Phase 4: 実行計画

```
/plan-step 11  # 実装ガイド
/plan-step 12  # テスト戦略
/plan-step 13  # デプロイ・ロールアウト計画
/plan-review phase4
```

**Step 11: 実装ガイド** が最も実用的な成果物です。エンジニアリングチームが「明日から実装を開始できる」レベルの詳細さを持つガイドラインを生成します。

### 最終レビュー — GO / NO-GO 判定

全フェーズが完了したら、最終レビューを実行します。

これはフェーズ単位のレビューとは異なり、**フェーズ間の整合性**を検証するクロスフェーズレビューです。

```
=== 最終レビュー ===

Phase間整合性チェック:
  Phase 1 → Phase 2: OK（ドメインモデルとデータモデルが整合）
  Phase 2 → Phase 3: OK（トランザクション設計とインフラ設計が整合）
  Phase 3 → Phase 4: OK（インフラ設計とデプロイ計画が整合）

全体的な品質:
  ドキュメントの完成度: 95%
  実装着手可能性: 高
  リスク要因: 2件（中程度）

最終判定: GO
  → エンジニアリングチームへの引き渡しを推奨します。
```

判定は **GO / CONDITIONAL GO / NO-GO** の3段階です。

### 16の調査ドキュメント — 組み込み知識ベース

Coding Agent for ScalarDB には、ScalarDB に関する **16の調査ドキュメント** が事前に組み込まれています。

| カテゴリ | ドキュメント |
| --- | --- |
| 基礎 | ScalarDB 概要、アーキテクチャ、Consensus Commit |
| パターン | データモデリング、トランザクションパターン、マルチストレージ |
| 比較 | vs CockroachDB、vs Spanner、vs YugabyteDB |
| 運用 | モニタリング、バックアップ、マイグレーション |
| セキュリティ | 認証・認可、暗号化、監査ログ |

エージェントが設計判断を行う際、これらのドキュメントを自動的に参照します。例えば、Step 05（トランザクション設計）で「Consensus Commit と JDBC トランザクションのどちらを使うべきか」を判断する際に、比較ドキュメントの情報が活用されます。

### マルチモデル戦略

Coding Agent for ScalarDB は、タスクの複雑度に応じて異なるモデルを使い分けます。

| モデル | 用途 |
| --- | --- |
| **Haiku** | 進捗管理、ステータス表示、軽量なクエリ |
| **Sonnet** | 各ステップの実行、標準的な設計タスク |
| **Opus** | レビュー統合、最終判定、複雑な判断 |

これにより、トークン消費を最適化しつつ、重要な判断にはより高性能なモデルを投入できます。
