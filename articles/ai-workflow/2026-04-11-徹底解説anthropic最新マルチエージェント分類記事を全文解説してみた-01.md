# Multi-Agent Coordination Patterns: Five patterns for coordinating agents

**ーマルチエージェント協調パターン: エージェントを束ねる5つの型ー**

こんにちは。yamamotoです。  
私は非エンジニアですが、Anthropicがマルチエージェントシステムの協調パターンを体系化した記事を公開しましたので前回に引き続き徹底解説していきます。

今回は「使うと決めたチームが、どのパターンを選ぶべきか」という実践的な設計判断がテーマです。

<https://www.anthropic.com/engineering/multi-agent-coordination-patterns>

## Key Message: ここだけ読めばわかる！5つの協調パターン

Anthropicは、マルチエージェントシステムの協調方法を5つのパターンに分類しています。  
重要なのは「洗練されて見えるパターン」ではなく、**問題に適合する最もシンプルなパターン**から始め、つまずいた箇所を観察しながら進化させることです。

| # | パターン | 一言で言うと | 使いどころ |
| --- | --- | --- | --- |
| 1 | Generator-Verifier | 作って、検品する | 品質が最重要で、検品基準を明文化できるとき |
| 2 | Orchestrator-Subagent | リーダーが部下に振る | タスク分解が明確で、サブタスクが独立しているとき |
| 3 | Agent Teams | チームで並列作業 | 長時間かかる独立タスクを並行して走らせたいとき |
| 4 | Message Bus | イベントで繋ぐ | イベント駆動でワークフローが発生するパイプラインで、かつエージェントが増える可能性があるとき |
| 5 | Shared State | 共有ボードで協働 | エージェント同士が状態や知識を共有しながら働くとき |

**【非エンジニアの補足】**  
これらのパターンは排他的な選択肢ではなく、ビルディングブロックです。実運用では複数のパターンを組み合わせます。たとえば「全体はOrchestrator-Subagentで回しつつ、協調が必要なサブタスクにはShared Stateを使う」というハイブリッド構成です。

# 以下記事本文を全文解説します。

## Pattern 1: Generator-Verifier

![](https://static.zenn.studio/user-upload/8919b7444acc-20260412.png)

### "A generator receives a task and produces an initial output, which it passes to a verifier for evaluation. The verifier checks whether the output meets the required criteria and either accepts it as complete or rejects it with feedback."

ジェネレータ（生成器）がタスクを受け取り、最初の出力を生成する。これがベリファイア（検証器）に渡され、評価を受ける。ベリファイアは、出力が要求基準を満たしているかどうかを確認し、完了として受理するか、フィードバックとともに差し戻す。

**【非エンジニアの補足】**  
最もシンプルなマルチエージェントパターンであり、最も多く導入されているものでもあります。  
仕組みは「書く人」と「赤ペン先生」の関係そのものです。

| 役割 | 何をするか | 人間の例え |
| --- | --- | --- |
| Generator | タスクに対する出力を生成する | レポートの執筆者 |
| Verifier | 基準に照らして検証し、合否を返す | 上司のレビュー |
| フィードバックループ | 不合格なら具体的な指摘とともに差し戻し | 赤入れして返却 |

### "Consider a support system that generates email responses to customer tickets. The generator produces an initial response using product documentation and ticket context. The verifier checks accuracy against the knowledge base, evaluates tone against brand guidelines, and confirms the response addresses each issue raised."

例として、カスタマーチケットに対するメール応答を生成するサポートシステムを考えてほしい。ジェネレータは、製品ドキュメントとチケットのコンテキストを使って最初の応答を生成する。ベリファイアは、ナレッジベースと照合して正確性を確認し、ブランドガイドラインに対してトーンを評価し、提起された各課題に応答が対処していることを確認する。

### Where it struggles

### "The verifier is only as good as its criteria. A verifier told only to check whether output is 'good,' with no further criteria, will rubber-stamp the generator's output."

ベリファイヤーはその検証基準の質に依存する。「良い出力かどうかだけ確認せよ」としか指示されていないベリファイヤーは、ジェネレータの出力にただハンコを押すだけになる。

**【非エンジニアの補足】**  
ここがこのパターン最大の落とし穴です。「検証ループを入れた」こと自体に安心してしまい、**検証基準を具体化しないまま運用する**ケースが多いとAnthropicは指摘しています。形式的な品質管理が、実質的な品質管理の幻想を生み出してしまうのです。

もう一つの弱点として、**生成と検証が同じ難易度の場合**（例: 創造的なアプローチの評価）、ベリファイヤーも問題を見逃す可能性があります。  
さらに、ジェネレータがフィードバックに対処できない場合、ループが収束せず振動し続けるリスクがあります。最大反復回数の設定と、フォールバック戦略（人間へのエスカレーション等）が必須です。

## Pattern 2: Orchestrator-Subagent

![](https://static.zenn.studio/user-upload/bfb81c5bc003-20260412.png)

### "Hierarchy defines this pattern. One agent acts as a team lead that plans work, delegates tasks, and synthesizes results. Subagents handle specific responsibilities and report back."

階層構造がこのパターンを定義する。1つのエージェントがチームリーダーとして機能し、作業を計画し、タスクを委任し、結果を統合する。サブエージェントは特定の責務を担い、結果を報告する。

### "Claude Code uses this pattern. The main agent writes code, edits files, and runs commands itself, dispatching subagents in the background when it needs to search a large codebase or investigate independent questions so work continues while results stream back."

Claude Codeはこのパターンを使用している。メインエージェントはコードの記述、ファイル編集、コマンド実行を自ら行いつつ、大規模なコードベースの検索や独立した調査が必要なときはバックグラウンドでサブエージェントをディスパッチし、結果がストリームバックされる間も作業を継続する。

**【非エンジニアの補足】**  
このパターンは「チームリーダーが部下に仕事を割り振る」組織構造そのものです。

| 役割 | 動作 | 特徴 |
| --- | --- | --- |
| Orchestrator | タスクを分解し、サブエージェントに委任、結果を統合 | 全体像を把握している |
| Subagent | 個別の責務を遂行して結果を返す | 自分の担当のみに集中 |

### Where it struggles

### "The orchestrator becomes an information bottleneck. When a subagent discovers something relevant to another subagent's work, that information has to travel back through the orchestrator."

オーケストレータに情報が集中することがボトルネックになる。あるサブエージェントが別のサブエージェントの作業に関連する発見をした場合、その情報はオーケストレータを経由して伝達されなければならない。

**【非エンジニアの補足】**  
すべての情報がリーダーを経由するため、やりとりが何度も行われるうちに重要な詳細が要約によって失われる、いわゆる「伝言ゲーム問題」が発生します。また、明示的に並列化しない限り、サブエージェントは逐次的に実行され、マルチエージェントのトークンコストを払いつつ速度メリットを得られないという事態にもなりえます。

## Pattern 3: Agent Teams

![](https://static.zenn.studio/user-upload/390bfd754d3a-20260412.png)

### "A coordinator spawns multiple worker agents as independent processes. Teammates claim tasks from a shared queue, work on them autonomously across multiple steps, and signal completion."

コーディネーターが複数のワーカーエージェントを**独立したプロセスとして**生成する。チームメイトは共有キューからタスクを取得し、複数ステップにわたって自律的に作業し、完了を通知する。

### "The difference from orchestrator-subagent is worker persistence. The orchestrator spawns a subagent for one bounded subtask, and the subagent terminates after returning a result. Teammates stay alive across many assignments, accumulating context and domain specialization."

Orchestrator-Subagentとの違いは**ワーカーの永続性**にある。オーケストレータは1つの限定的なサブタスクのためにサブエージェントを生成し、サブエージェントは結果を返した後に終了する。一方、チームメイトは多くの割り当てにわたって存続し続け、コンテキストとドメインの専門性を蓄積していく。

**【非エンジニアの補足】**  
Pattern 2との最大の違いは、ワーカーが「使い捨て」か「持続的」かです。

| 比較軸 | Orchestrator-Subagent | Agent Teams |
| --- | --- | --- |
| ワーカーの寿命 | 1タスクで終了 | 複数タスクにわたって持続 |
| コンテキスト | 毎回ゼロから | 蓄積・学習される |
| 適するタスク | 短く限定的なサブタスク | 長期的で多段階の作業 |

具体例として、**大規模コードベースのフレームワーク移行**が挙げられています。各チームメイトがサービスごとの移行を担当し、依存関係・テスト・デプロイ設定に習熟しながら作業を進めます。

### Where it struggles

独立性が必須要件である。Orchestrator-Subagentではオーケストレータがサブエージェント間の仲介や情報のルーティングを行えるのに対し、チームメイトは自律的に動作するため、中間的な発見の共有が容易ではない。

**【非エンジニアの補足】**  
もう一つの課題が**共有リソースの競合**です。複数のチームメイトが同じコードベースやファイルシステムを操作する場合、同じファイルを編集したり、互換性のない変更を加えたりする可能性があります。タスクの分割と競合解決のメカニズムを慎重に設計する必要があります。

## Pattern 4: Message Bus

![](https://static.zenn.studio/user-upload/cc7eabe5f5f8-20260412.png)

### "Agents interact through two primitives: publish and subscribe. Agents subscribe to the topics they care about, and a router delivers matching messages. New agents with new capabilities can start receiving relevant work without rewiring existing connections."

エージェントは2つのプリミティブを通じて相互作用する。パブリッシュ（発行）とサブスクライブ（購読）である。エージェントは関心のあるトピックを購読し、ルーターがマッチするメッセージを配信する。新たな能力を持つ新しいエージェントは、既存の接続を再配線することなく、関連する作業を受け取り始めることができる。

**【非エンジニアの補足】**  
これは「社内の掲示板」のような仕組みです。

| 概念 | 役割 | 例え |
| --- | --- | --- |
| Publish | イベントを掲示板に投稿する | 「高重大度のネットワークアラート発生」と掲示 |
| Subscribe | 特定のカテゴリの掲示を監視する | ネットワーク担当が「ネットワーク」カテゴリを購読 |
| Router | 投稿を適切な購読者に届ける | 掲示板の仕分け係 |

具体例として**セキュリティオペレーション自動化**が挙げられています。アラートが到着すると、トリアージエージェントが分類し、重大度とタイプに応じて専門の調査エージェントにルーティングします。新しい脅威カテゴリが出現しても、新しいエージェントを追加するだけで対応できます。

### Where it struggles

### "The flexibility of event-driven communication makes tracing harder. When an alert triggers a cascade of events across five agents, understanding what happened requires careful logging and correlation."

イベント駆動型コミュニケーションの柔軟性が、追跡を困難にする。1つのアラートが5つのエージェントにわたるイベントのカスケードを引き起こした場合、何が起きたかを理解するには注意深いロギングと相関分析が必要となる。

**【非エンジニアの補足】**  
Orchestratorパターンならリーダーの判断を追えばデバッグできますが、Message Busでは「誰が何をどの順で処理したか」を追跡するのが格段に難しくなります。また、ルーターがイベントを誤分類したり落としたりすると、システムはクラッシュせずに**サイレントに失敗**します。何もしないが、何も壊れない、という最も発見しにくい障害です。

## Pattern 5: Shared State

![](https://static.zenn.studio/user-upload/f6afcec1cc57-20260412.png)

### "Agents operate autonomously, reading from and writing to a shared database, file system, or document. There's no central coordinator. Agents check the store for relevant information, act on what they find, and write their findings back."

エージェントは自律的に動作し、共有データベース・ファイルシステム・ドキュメントに対して読み書きする。中央のコーディネーターは存在しない。エージェントはストアに関連情報があるか確認し、見つけたものに基づいて行動し、自らの発見をストアに書き戻す。

**【非エンジニアの補足】**  
このパターンは「共有ホワイトボード」のイメージが近いです。

| 特徴 | 説明 |
| --- | --- |
| 中央コーディネーターなし | 各エージェントが自律的に共有ストアを読み書き |
| リアルタイム共有 | あるエージェントの発見を、他のエージェントが即座に参照可能 |
| 単一障害点なし | 1つのエージェントが停止しても、他は継続 |

具体例として**リサーチ・シンセシス（調査統合）システム**が挙げられています。学術論文、業界レポート、特許、ニュースをそれぞれ調査するエージェントが、発見を共有ストアに書き込みます。学術エージェントが重要な研究者を発見すれば、その情報は即座に業界エージェントの調査に活かされます。

### Where it struggles

### "Without explicit coordination, agents may duplicate work or pursue contradictory approaches. ... The harder failure mode is reactive loops."

明示的な協調がなければ、エージェントは作業を重複させたり、矛盾するアプローチを追求したりする可能性がある。...より困難な障害モードは**リアクティブ・ループ**である。

**【非エンジニアの補足】**  
リアクティブ・ループとは、エージェントAが書いた発見をエージェントBが読み、BのフォローアップをAが読んで反応し...と、収束しないまま延々とトークンを消費し続ける現象です。

| 問題 | 既知の解決策 |
| --- | --- |
| 作業の重複 | ロック、バージョン管理、パーティション分割 |
| 並行書き込みの競合 | 同上（エンジニアリングの定石） |
| **リアクティブ・ループ** | **一級の終了条件が必須**（時間予算、収束閾値、判定エージェント） |

終了条件を後付けで考えるシステムは、無限に循環するか、あるエージェントのコンテキストが溢れたときに任意に停止する傾向があるとAnthropicは警告しています。

# パターンの選択と進化

**ーどのパターンをどう選ぶかー**

### "The right pattern depends on a handful of structural questions about the system."

適切なパターンは、システムに関するいくつかの構造的な問いに依存する。

Anthropicは前回の記事で提唱した\*\*コンテキスト中心の分解（context-centric decomposition）**の原則がここでも適用されるとしています。「どんな種類の作業をするか」ではなく、**「各エージェントがどんなコンテキストを必要とするか」\*\*で分割するという考え方です。

以下に、パターン間の選択基準を示します。

## Orchestrator-Subagent vs. Agent Teams

**ーワーカーがコンテキストを持続する必要があるか？ー**

| 判断基準 | Orchestrator-Subagent | Agent Teams |
| --- | --- | --- |
| サブタスクの性質 | 短く、焦点が明確で、明瞭な出力を持つ | 持続的で多段階の作業から恩恵を受ける |
| 具体例 | コードレビュー（各チェックが1回の呼び出しで完結） | コードベース移行（各チームメイトが担当サービスに習熟） |
| 決め手 | サブエージェントが呼び出し間で状態を保持する必要がないとき | 蓄積されたコンテキストが性能を向上させるとき |

## Orchestrator-Subagent vs. Message Bus

**ーワークフローの構造は予測可能か？ー**

| 判断基準 | Orchestrator-Subagent | Message Bus |
| --- | --- | --- |
| ワークフロー | ステップの順序が事前に分かっている | イベントからワークフローが創発される |
| 具体例 | コードレビュー（固定パイプライン） | セキュリティオペレーション（アラートの種類は予測不能） |
| 決め手 | 固定的なパイプラインで済むとき | 条件分岐ロジックがオーケストレータに蓄積し始めたとき |

## Agent Teams vs. Shared State

**ーエージェント同士が互いの発見を必要とするか？ー**

| 判断基準 | Agent Teams | Shared State |
| --- | --- | --- |
| エージェント間の関係 | 独立したパーティションで作業 | 発見をリアルタイムで共有し協働 |
| 具体例 | コードベース移行（各サービスは独立） | リサーチ統合（学術エージェントの発見が業界エージェントに即座に影響） |
| 決め手 | 最終結果のみ共有すればよいとき | 中間結果の共有がワークフローに不可欠なとき |

## Message Bus vs. Shared State

**ー作業はイベントの流れか、知識の蓄積か？ー**

| 判断基準 | Message Bus | Shared State |
| --- | --- | --- |
| 情報の流れ | 離散的なイベントがパイプラインを流れる | 知識が時間とともにストアに蓄積される |
| 中央集権性 | ルーターという中央コンポーネントが存在する | 完全に分散型 |
| 具体例 | セキュリティアラートの処理（段階的にパイプライン処理） | リサーチ統合（継続的に知識を蓄積） |
| 決め手 | イベントが「アクションをトリガーする」とき | イベントが「発見を共有する」ためのものであるとき |

**【非エンジニアの補足】**  
Message Busのエージェントが「発見を共有する」ためにイベントを発行し始めたら、それはShared Stateの方が自然なサインです。また、単一障害点の排除が優先事項であれば、Shared Stateがより完全にそれを実現します（Message Busにはルーターという中央コンポーネントが残ります）。

# まとめ

### "For most use cases, we recommend starting with orchestrator-subagent. It handles the widest range of problems with the least coordination overhead."

ほとんどのユースケースにおいて、Orchestrator-Subagentから始めることを推奨する。最小の協調オーバーヘッドで最も広範な問題に対応できるからだ。

| 状況 | 推奨パターン |
| --- | --- |
| 品質重視で、評価基準を明文化できる | Generator-Verifier |
| タスク分解が明確で、サブタスクが限定的 | Orchestrator-Subagent |
| 並列で独立した長時間サブタスク | Agent Teams |
| イベント駆動でワークフローが発生するパイプラインや、エージェントの増加 | Message Bus |
| 協調的な調査、エージェント間の発見共有 | Shared State |
| 単一障害点の排除が必要 | Shared State |

実運用ではパターンの組み合わせが一般的です。「全体のワークフローはOrchestrator-Subagent、協調が重要なサブタスクにはShared State」や、「イベントルーティングはMessage Bus、各イベント処理はAgent Teamsスタイルのワーカー」といったハイブリッド構成が有効です。

**最もシンプルなパターンから始め、つまずいた箇所を観察し、そこから進化させる。** これがAnthropicの推奨するアプローチです。
