LLM に「なぜ？」を連投して、気付いたら 1 時間が溶けていた——そんな経験はありませんか。

私自身、副業で個人開発をしているのですが、LLM との壁打ちで時間を失うパターンが何回かありました。たとえばバグ調査で「なぜこのエラーが出るのですか？」と聞いて、返ってきた説明にまた「なぜ？」を重ね、3 往復目で「**結局これ、最初に何を聞きたかったんだっけ**」と止まる、というやつです。

この手の空転は、LLM の調子の問題ではなく、こちら側の問いの組み立て方の問題でした。半年くらい自分の壁打ちログを並べて見比べた結果、うまくいった日とダメだった日の差は、感覚論ではなく **5 つの軸** に集約できることが分かってきました。

本記事は、その 5 軸を一覧で見渡せる **全体像** をお渡しします。各軸の詳細は別記事と eBook で扱いますが、まずは 5 軸の存在と相互関係を掴むのが、壁打ちを再現可能な技術に変える最初の一歩になります。

## 5 軸とは何か

5 軸は、壁打ちが空転する原因を切り分けるためのチェックリストです。

| 軸 | 内容 |
| --- | --- |
| **軸① 前提共有** | 制約 / 目的 / やりたくないこと を LLM に先に渡す |
| **軸② 問いの粒度** | 「どうすれば？」を「A / B / C のどれが？」に置き換える |
| **軸③ 答えの種類** | 結論を断り、論点を並べてもらう |
| **軸④ 往復数** | 1 往復で終わらせず、3〜5 往復に分けて深掘る |
| **軸⑤ 出力形式** | 表 / リスト / 段落の使い分けを指定する |

5 軸は独立したチェックリストではなく、**相互作用** します。

たとえば前提共有（軸①）が雑だと、問いの粒度（軸②）を整えても返答が一般論に寄ります。逆に前提共有さえ整えれば、「どうすれば？」と粗く聞いても、思ったよりまともな返答が返ってくることもあります。

本記事ではまず 5 軸を **横並びで把握する** ことを目的にし、軸同士の組合せ方は次のセクションで扱います。

## 5 軸が効く 3 つの場面

5 軸を意識的に使うと、個人開発でよく出会う 3 つの場面で効果が分かりやすく出ます。

### 場面① 設計判断 — テーブル設計をどう分けるか

「ユーザーと記事のテーブル設計、どうすればいいですかね？」と聞くと、たいてい「正規化の基本は…」という教科書的な引用から始まります。

ここで効くのは **軸②（問いの粒度）+ 軸⑤（出力形式）** です。

> 個人ブログ用に、ユーザーと記事のテーブル設計を考えています。  
> 案 A（正規化）、案 B（記事にユーザー名を非正規化）、案 C（JSON カラムで著者情報を埋める）の 3 つを、  
> 「読み取り頻度」「更新頻度」「将来の拡張性」の 3 軸で比較して、表で出してください。

選択肢と評価軸を先にこちらが指定し、出力形式を表で固定するだけで、返ってきた表をそのまま判断材料にできます。

### 場面② スタック選定 — Next.js vs Astro

「Next.js と Astro どっちがいいですか？」と聞くと、「用途次第です」という両論併記で終わります。

ここで効くのは **軸①（前提共有）+ 軸③（答えの種類）** です。

> 副業エンジニアで、稼働時間は月 20 時間ほど。  
> 次の 3 ヶ月で技術ブログとポートフォリオサイトを公開したい。  
> TypeScript は仕事で使っているが、フレームワークの設定で消耗したくない。  
> この前提で、Next.js と Astro の選定にあたって押さえるべき論点を 5 つ挙げてください。結論は不要です。

前提を渡し、結論ではなく論点を頼むだけで、返ってきた 5 つの論点から自分にとって本当に重要なのは 1〜2 個に絞れます。

### 場面③ 詰まったバグ調査 — Prisma findUnique が undefined

「Prisma で findUnique が undefined を返します。なぜですか？」と 1 往復で原因を聞くと、当てずっぽうの返答が返ってきます。

ここで効くのは **軸③（答えの種類）+ 軸④（往復数）** です。

> Prisma で findUnique が undefined を返しています。  
> 原因候補を、よくある順に 5 つ挙げてください。それぞれ「確認方法」も併記してください。

候補を 5 つ並べてもらい、上から確認しながら「2 番目を試したらこのエラーが出た」と次の往復に進む。**1 往復目を足場にする** という発想が、調査の効率を大きく変えます。

## 5 軸の使い方 — 「全部使う」ではなく「場面で 1〜2 軸」

5 軸を見て「全部毎回意識するのか」と思うかもしれませんが、実際の運用は逆です。

**場面に応じて 1〜2 軸を選ぶ** のが基本姿勢になります。

個人開発の判断は、ほとんどがトレードオフです。「コードを書く前に設計の選択肢を整理したい」「実装方針で迷っている」「バグの原因が掴めない」——どれも結論を 1 つに絞るより、判断材料を複数並べたほうが前に進みます。なので、軸③（答えの種類）と軸⑤（出力形式）を中心に組み合わせる場面が圧倒的に多くなります。

逆に、設計判断を頼むときは前提（軸①）が決定的に効きます。前提が雑だと、どれだけ問いの粒度（軸②）を整えても、返答の質はあまり変わりません。

経験的な使い分けの目安はこんな感じです。

| 場面 | 主役の軸 | 補助の軸 |
| --- | --- | --- |
| 設計判断 | 軸②（問いの粒度） | 軸⑤（出力形式） |
| スタック選定 | 軸①（前提共有） | 軸③（答えの種類） |
| バグ調査 | 軸③（答えの種類） | 軸④（往復数） |
| 仕様の整理 | 軸①（前提共有） | 軸⑤（出力形式） |
| アイデア発散 | 軸③（答えの種類） | 軸④（往復数） |

軸を意識すると、壁打ちが **再現可能** になります。

「今日は調子よく LLM と話せた」が「軸①〜② を意識した日だった」に翻訳できるようになると、翌週の自分が同じ条件を作りに行けます。これが、壁打ちを感覚芸から技術に変える最大の効用だと思っています。

## 次のステップ

5 軸の全体像を掴めたら、次は **軸の使い方を 1 つずつ深める** フェーズに入ります。

* **軸①〜③ の詳細（前提共有 / 問いの粒度 / 答えの種類）** は、近日 Zenn で詳細記事を公開する予定です。Before / After の実例つきで、それぞれの軸を 1,500 字ずつ掘り下げます（公開され次第、本記事末尾にリンクを追加します）。
* **軸④〜⑤（往復数 / 出力形式）+ 代表プロンプト 10 個** は、有料 eBook「壁打ちエンジニアリング — 個人開発者のための LLM 活用プロンプト100」ver.1.0 でお届けしています（先行価格 ¥980 / Gumroad）。
