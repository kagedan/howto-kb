## 現状分析 — Refactoring Agent でレガシーシステムの「今」を可視化する

---

### はじめに

前回は、3つのAIエージェントを連携させたレガシーシステム刷新ワークフローの全体像を紹介しました。今回は最初のステージ、**Refactoring Agent** を使った現状分析の進め方を解説します。

レガシーシステム刷新の第一歩は「現状を正確に把握すること」です。しかし、数万行、数百ファイルのコードベースを人手で網羅的に調査するのは非現実的です。Refactoring Agent は、この分析作業をAIエージェントに委託し、構造化されたレポートとして出力します。

---

### セットアップ

#### 前提条件

| ツール | バージョン | 用途 |
| --- | --- | --- |
| **Claude Code** | 最新版 | エージェント実行環境 |
| **Python** | 3.9以上 | ナレッジグラフ構築・レポート生成 |
| **Node.js** | 18以上 | Mermaid図レンダリング |

#### インストール

```
# リポジトリの取得
git clone https://github.com/wfukatsu/refactoring-agent-for-claude-code.git
cd refactoring-agent-for-claude-code

# Python依存パッケージ
python3 -m venv .venv
source .venv/bin/activate
pip install ryugraph pandas markdown pymdown-extensions radon networkx

# Mermaid CLI
npm install -g @mermaid-js/mermaid-cli
```

#### Serena MCP（推奨）

Serena MCP を設定すると、Javaコードの解析精度が格段に上がります。ASTレベルでクラス定義やメソッドの参照関係を追跡できるため、単なるテキスト検索よりも正確なドメインマッピングが可能です。

```
// .claude/mcp.json
{
  "mcpServers": {
    "serena": {
      "command": "uvx",
      "args": ["serena"]
    }
  }
}
```

├──────────────────────────────────────────┤
│  設計スキル                                 │
│  /design-microservices  マイクロサービス設計  │
│  /design-api            API仕様生成          │
│  /design-scalardb       ScalarDB設計         │
│  /design-scalardb-analytics  分析基盤設計    │
│  /create-domain-story   ドメインストーリー    │
│  /estimate-cost         コスト見積もり        │
├──────────────────────────────────────────┤

### 実行してみる

#### 分析対象の準備

Refactoring Agent にはサンプルコード（`SampleCode/`）が同梱されています。Spring Boot の注文管理システム（Java、DDD 4層アーキテクチャ）で、「よくあるレガシーの初期状態」を模したものです。

#### メインコマンドの実行

```
/refactor-system-cmd ./SampleCode
```

このコマンドを実行すると、パイプラインが順番にスキルを呼び出していきます。

```
Step 1: /init-output          出力ディレクトリの初期化
Step 2: /analyze-system       システム分析（ユビキタス言語、アクター、ドメイン対応表）
Step 3: /evaluate-mmi         MMIモジュール成熟度評価
Step 4: /map-domains          ドメインマッピング
Step 5: /design-microservices マイクロサービス設計
Step 6: /design-scalardb      ScalarDB設計
Step 7: /design-api           API仕様生成
Step 8: /create-domain-story  ドメインストーリー
Step 9: /estimate-cost        コスト見積もり
Step 10: /build-graph         ナレッジグラフ構築
Step 11: /visualize-graph     グラフ可視化
Step 12: /fix-mermaid         Mermaid図修正
Step 13: /compile-report      HTMLレポート生成
```

#### 個別スキルの実行

パイプライン全体ではなく、特定のスキルだけを実行することもできます。

```
# まずシステム分析だけ試す
/analyze-system ./SampleCode

# MMI評価だけ試す（analyze-systemの結果が必要）
/evaluate-mmi ./SampleCode
```

後半のスキルは前のスキルの出力に依存しているため、依存するファイルが存在しない場合はエラーが報告されます。

### 出力を読む — 3つの重要レポート

パイプライン完了後、`reports/` ディレクトリに成果物が生成されます。まず読むべき3つのレポートを紹介します。

#### 1. ユビキタス言語辞書

**ファイル:** `reports/01_analysis/ubiquitous-language.md`

コードベースからドメイン固有の用語を抽出し、定義と対応するコードをマッピングした辞書です。

```
| 用語（日本語） | 用語（英語） | 定義                       | コード上の表現    |
|--------------|------------|---------------------------|----------------|
| 注文         | Order      | 顧客が商品を購入する行為       | Order.java      |
| 注文明細      | OrderItem  | 注文に含まれる個別商品の情報    | OrderItem.java  |
| 注文状態      | OrderStatus| 注文のライフサイクル上の状態    | OrderStatus.java|
| 顧客         | Customer   | 注文を行う主体              | customerId      |
```

DDD の根幹は「開発者とドメインエキスパートが同じ言葉を使うこと」です。この辞書は、後続のすべてのフェーズでドメイン概念の一貫性を保証する基盤になります。

#### 2. MMI（モジュール成熟度）評価

**ファイル:** `reports/02_evaluation/mmi-overview.md`

各モジュールのマイクロサービス化準備度を4つの軸で評価します。

| 軸 | 重み | 評価観点 |
| --- | --- | --- |
| 凝集度（Cohesion） | 30% | モジュール内の機能はまとまっているか |
| 結合度（Coupling） | 30% | 他モジュールへの依存は少ないか |
| 独立性（Independence） | 20% | 独立してデプロイ・スケールできるか |
| 再利用性（Reusability） | 20% | 他プロジェクトでも使えるか |

実際のサンプルシステムでの評価結果：

```
| 順位 | モジュール          | MMI スコア | レベル      |
|:----:|---------------------|:---------:|:----------:|
| 1    | Domain Repository   | 88.0      | High       |
| 2    | Domain Model        | 64.0      | Low-Medium |
| 3    | Infrastructure      | 64.0      | Low-Medium |
| 4    | Domain Service      | 62.0      | Medium     |
| 5    | Application         | 62.0      | Medium     |
| 6    | Presentation        | 48.0      | Low-Medium |
```

「Domain Repository はリファクタリング不要だが、Presentation層は大幅な改善が必要」——こうした**定量的な判断材料**が自動で生成されます。

#### 3. ドメイン-コード対応表

**ファイル:** `reports/01_analysis/domain-code-mapping.md`

抽出されたドメイン概念と、実際のコード上のクラス・メソッドの対応関係をマッピングした表です。DDD のパターン（エンティティ、値オブジェクト、リポジトリ、サービスなど）ごとに分類されます。

この対応表があることで、「このドメイン概念を実装しているのはどのクラスか」「このクラスが担っているドメインの責務は何か」が一目で分かります。

### ナレッジグラフ — 分析結果を探索可能にする

Refactoring Agent の特徴的な機能がナレッジグラフです。分析結果をグラフデータベース（RyuGraph）に格納し、自然言語でクエリできます。

#### グラフの構築と可視化

```
# グラフの構築
/build-graph

# D3.jsベースのインタラクティブビューア
/visualize-graph
open reports/graph/visualizations/graph.html
```

#### 自然言語クエリ

```
/query-graph "Orderエンティティに関連するすべてのエンティティを表示"
/query-graph "注文作成プロセスに含まれるアクティビティの順序"
/query-graph "共有カーネルに配置すべき共通エンティティ"
```

コードを読み返すことなく、ドメイン間の関係性をグラフで探索できるのが大きなメリットです。特に大規模コードベースでは、コードを毎回読み直すよりもグラフ経由のクエリの方が圧倒的に効率的です。

#### データモデル

ナレッジグラフは10種類のノードタイプと17種類のリレーションタイプで構成されます。

| ノードタイプ | 例 |
| --- | --- |
| Domain | OrderManagement |
| Entity | Order, OrderItem |
| Method | createOrder(), save() |
| File | OrderService.java |
| Term | 注文, 顧客, 商品 |
| Actor | Customer, Admin |

### HTMLレポートで俯瞰する

```
/compile-report
open reports/00_summary/full-report.html
```

全レポートを統合したHTMLが生成されます。サイドバー目次付きで、Mermaid図のインラインレンダリングも含まれます。ライト/ダークテーマに対応しており、意思決定者への共有にも使えます。

### Serena MCP による解析精度の向上

Serena MCP が有効な場合、エージェントはASTレベルのコード解析を行います。

```
通常（テキスト検索のみ）:
  "OrderService" を grep → 出現箇所のリスト

Serena MCP 有効時:
  OrderService.createOrder() の呼び出し元を追跡
  → OrderController.create()
  → OrderFacade.processOrder()
  → BatchOrderProcessor.run()
```

テキスト検索では「OrderService」という文字列が出現するファイルしか分からないのに対し、Serena MCP ではメソッドの呼び出し関係を構造的に辿れます。依存関係の把握精度が格段に向上するため、モジュール分割の判断材料としてより信頼性の高い情報が得られます。

### Stage 2 への橋渡し — 出力をどう使うか

Refactoring Agent の出力は、次のステージ（Architecture Redesign Agent）の入力として直接利用できます。

| Refactoring Agent の出力 | Architecture Redesign Agent での用途 |
| --- | --- |
| `ubiquitous-language.md` | DDD評価の基盤。用語の一貫性チェック |
| `domain-code-mapping.md` | 境界コンテキスト再定義の入力 |
| `mmi-overview.md` | 改善優先度の判断材料 |
| `target-architecture.md` | リファクタリング目標の確認 |
| `knowledge.ryugraph` | ドメイン間関係の追加クエリ |

実際の橋渡し手順は非常にシンプルです。Architecture Redesign Agent のリポジトリに Refactoring Agent の `reports/` ディレクトリをコピー（またはシンボリックリンク）するだけで、後続のスキルが自動的に読み込みます。

```
# Refactoring Agent の出力を Architecture Redesign Agent にコピー
cp -r refactoring-agent/reports/ architecture-redesign-agent/reports/
```

---

### まとめ

* Refactoring Agent は **18スキル** でレガシーシステムの現状を網羅的に分析する
* `/refactor-system-cmd` でパイプラインを一括実行、個別スキルの単独実行も可能
* **ユビキタス言語辞書**、**MMI評価**、**ドメイン-コード対応表** が3つの重要成果物
* **ナレッジグラフ** で分析結果を永続化し、自然言語でクエリできる
* **Serena MCP** でJavaコードのAST解析精度を向上
* 出力は Architecture Redesign Agent の入力としてそのまま利用可能

次回は、Refactoring Agent の分析結果を受けて、Architecture Redesign Agent で DDD に基づく再設計を行う手順を解説します。
