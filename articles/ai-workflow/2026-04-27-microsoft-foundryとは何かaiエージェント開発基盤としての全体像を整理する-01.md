# 導入｜なぜMicrosoft Foundryを調べているのか

業務の中でMicrosoft Foundryを触る機会が出てきました。

Azureは特にAI領域に強い印象がありますが、正直なところ、Foundryは機能が多く、全体像が掴みにくいと感じています。

このままなんとなく使うのではなく、一度自分の中で整理しないと実装レベルで使いこなせないと思い、今回まとめることにしました。

また、Agent FrameworkのPython版もリリースされ、AIエージェント周りの流れもかなり早くなっています。

この流れに遅れないように、自分なりに理解を深めていきたいと思い、今回全体像を整理してみました。

# 結論｜Microsoft Foundryは何なのか（現時点の理解）

Microsoft Foundryは、単なるPaaSではなく、「モデル・エージェント・ツール・評価・ガバナンスまでを統合したAIシステムのフルスタック基盤」だと理解しています。

モデルの選択やエージェントの実行、外部ツールとの連携、さらには監視や評価といった運用面までを一体で扱うことができ、利用者はインフラを意識しすぎることなく、AIエージェントの実装に集中できる設計になっていると感じました。

# 全体像｜どんな構成になっているのか

## Foundryの全体構造

Microsoft Foundryは、複数のレイヤーに分かれた構造で設計されていると感じました。  
現時点では、以下のように整理しています。

### 1. Models（モデル層）

AIの頭脳にあたる部分です。

### 2. Agent Service（実行層）

エージェントを実際に動かすための基盤です。

* エージェントの実行・管理
* マルチエージェント
* ワークフロー

エージェントに外部との接続や知識を持たせる部分です。

* Tools（外部機能連携）
* Foundry IQ（ナレッジ）
* AI Search

### 4. Control Plane（統制・運用層）

全体の管理や監視を行うための領域です。

# 実務での使いどころ

## 仮説①：RAGの進化形

Azureを中心とした環境であれば、Foundry IQやAI Search、SharePointなどのデータと組み合わせることで、エージェントの知識基盤を比較的シンプルに構築できると感じました。

従来のRAGのように「検索して答える」だけではなく、エージェントとして情報を取得し、それをもとに処理や判断まで行える点が大きな違いだと感じています。

一方で、これらのサービスを前提とするため、ある程度Azureに寄せた構成やコストを許容できる企業である必要はあると感じました。

## 仮説②：業務自動化

Foundryでは、以下のようなAIサービスをツールとして組み合わせて利用することができます。

| サービス | 説明 |
| --- | --- |
| Speech | 音声テキスト変換、テキスト読み上げ、翻訳、話者認識。 |
| Translator | AIを利用した翻訳テクノロジを使用して、100以上の言語（使用中・危機言語・方言含む）を翻訳。 |
| Language | 業界をリードする自然言語理解機能を備えたアプリを構築。 |
| Content Understanding | さまざまなメディアコンテンツタイプを分析・理解。 |
| Document Intelligence | ドキュメントをインテリジェントなデータドリブンソリューションに変換。 |
| Vision | 画像や動画コンテンツを分析。 |
| Azure AI Search | AIを活用したクラウド検索をモバイルアプリやWebアプリで利用可能にする。 |
| コンテンツの安全性 | 望ましくないコンテンツを検出するAIサービス。 |
| Custom Vision | 画像認識をビジネス用途に合わせてカスタマイズ。 |
| Immersive Reader | ユーザーの読解を支援する機能。 |

これらのサービスをエージェントと組み合わせることで、単なるテキスト生成ではなく、「音声・画像・ドキュメントなどを扱う業務そのもの」を自動化できる可能性があると考えています。

例えば、ドキュメントの解析（Document Intelligence）と検索（Azure AI Search）、さらに翻訳（Translator）などを組み合わせることで、これまで人手で行っていた業務フローをエージェントとして構築できるイメージです。

このあたりは、従来のAPI連携ではなく、「エージェントに任せる」形で実装できる点が特徴的だと感じました。

## 仮説③：LLMの運用管理

Control Planeの機能を見ると、従来は外部ツール（LangfuseやLangSmithなど）で行っていたトレースや評価といったLLMの運用管理を、Azure内で完結できる可能性があると感じました。

特にObservabilityの観点では、エージェントの挙動やログを一元的に管理できるようになると、実運用での可視性が大きく向上する可能性があります。

また、Azure内で完結することで、セキュリティ要件が厳しい環境でも利用しやすくなる点は、大きなメリットになると感じました。

# まだ分かっていないこと

現時点では、以下の点についてはまだ理解が浅く、実務で使う上で重要になりそうなポイントとして整理しています。

## バッチ評価について

エージェントの出力を大量のデータで一括評価できる仕組みのようですが、どのように評価ロジックを設計すべきか、また実運用でどのように使われるのかはまだ理解しきれていません。

自動テストのように活用できれば、AIの品質管理の観点で大きな価値がありそうです。

<https://learn.microsoft.com/ja-jp/azure/foundry-classic/how-to/flow-bulk-test-evaluation?view=foundry-classic>

## Agent Serviceの実行基盤について

エージェントは、単体だけでなくマルチエージェント構成やツール連携など、複雑な構成になることが想定されます。

その中で、Agent Service上でどのように設計・構築していくのが最適なのか、またMCPやA2Aといった設計思想や制約をどのように考慮すべきかは、まだ整理できていません。

## Foundry IQの内部構造

Foundry IQがどのようにナレッジを管理・検索しているのかはまだ理解しきれていません。

Azureネイティブで構築する場合、SharePointやAzure AI Searchなどとどのように連携させるのが最適なのか、従来のRAGとの違いも含めて今後検討が必要だと感じています。

# まとめ

Microsoft Foundryは、単なるモデルの利用基盤ではなく、エージェントを中心に「実行・拡張・制御」までを一体で扱うためのプラットフォームだと感じました。

Azureを利用している環境であれば、モデルの選定からエージェントの構築、さらにガードレールや評価といった運用面までを一貫して扱える点は大きな強みだと思います。

まだ理解途中ではありますが、AIを「使う」から「実務に組み込む」ための基盤として、今後さらに重要になっていくと考えています。
