# 背景

普段の業務では、チームで gemini-code-assist を導入しており、プルリクエスト（PR）に対して自動でコードレビューが行われる体制を整えています。

gemini-code-assist は非常に優秀で、バグの検知や可読性の向上といった汎用的なレビューにおいては高い効果を発揮してくれます。しかし、一方で以下のような課題も感じていました。

* コンテキストが不足しがち： アーキテクチャの選定、データ構造の設計、プロダクト特有の仕様に踏み込んだレビューは、一般的なAIレビューではカバーしきれない場合がある。
* 人間のレビュアーへの依存： 結果として、人間によるレビューで設計面や仕様に関する指摘を多く受けることになる。

「せめて、過去に自分がもらった人間からの貴重なレビューをナレッジ化し、PRを出す前にセルフチェックできれば、レビューの往復を減らせるのではないか？」と考えました。

# 過去のPRから「自分専用レビュアー」を用意する

そこで、GitHub上にある「過去に自分がもらったPRのコメント」をナレッジとしてローカルに蓄積し、開発環境で事前に自分専用のコードレビューを実行できる Claudeベースのオリジナルコマンド を作成しました。具体的には、対象のGitリポジトリのディレクトリ上で以下のカスタムコマンドを実行する仕組みです。

1. レビューコメントの抽出: ghコマンドを使用し、そのリポジトリにおける過去の自分のPRから、レビュアーに指摘されたコメントを自動で取得します。
2. ナレッジの分類・蓄積: 取得したコメントをただ並べるのではなく、Claudeを用いてタイプ別に分類。その指摘がされた「背景や意図」も含めて、ローカル環境に構造化されたナレッジとして保存します。

```
# レビューナレッジ学習コマンド（/review-for-me-learn）

自分のPRに付いたレビューコメントを収集し、レビューナレッジとして蓄積します。

## 引数

- `$ARGUMENTS`: 収集対象の指定（省略可）
  - 省略 → **差分取得モード**: `index.md` の学習済みPR一覧を参照し、未学習のPRのみ収集（初回は全件取得）
  - PR番号: `123` → 特定のPRのレビューコメントを収集（学習済みでも再取得）

## 実行手順

### 1. リポジトリ情報の取得

- `git remote get-url origin` でリモートURLを取得
- owner/repo を抽出
- 自分のGitHubユーザー名を `gh api user --jq '.login'` で取得

### 2. 学習済みPRの確認

- `~/.claude/review-knowledge/{owner}/{repo}/index.md` が存在するか確認
- 存在する場合、`## 学習済みPR一覧` セクションからPR番号を抽出し、学習済みリストとする
- 存在しない場合、学習済みリストは空（＝初回実行、全件取得）

### 3. 対象PRの決定とレビューコメントの収集

# 自分が作成したマージ済みPRを全件取得
gh pr list --author @me --state merged --limit 999 --json number,title,url,mergedAt

- **引数なし（差分取得モード）**: 取得したPR一覧から学習済みPR番号を除外し、未学習PRのみを対象にする
- **PR番号指定**: 指定されたPRのみを対象にする（学習済みでも再取得して上書き）

未学習PRが0件の場合は「全てのPRが学習済みです。新しいPRがマージされたら再実行してください。」と表示して終了。

対象PRごとに以下のコマンドでレビューコメントを取得:

# 各PRのレビューコメントを取得（自分以外のコメントのみ）
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --jq '.[] | select(.user.login != "<自分>") | {body, path, line: .original_line, diff_hunk, user: .user.login, created_at, html_url}'

# PRレビュー（approve/request changes時のコメント）も取得
gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '.[] | select(.user.login != "<自分>" and .body != "") | {body, state, user: .user.login, html_url}'

レビューコメントが0件のPRはスキップするが、学習済みとして `index.md` には記録する。

### 4. レビューコメントの分析・分類

収集したコメントを以下のカテゴリに分類:

| カテゴリ | 説明 | 例 |
|---|---|---|
| security | セキュリティに関する指摘 | SQLインジェクション、XSS、認証漏れ |
| performance | パフォーマンスに関する指摘 | N+1問題、不要な再レンダリング |
| logic | ロジック・バグに関する指摘 | エッジケース、オフバイワン |
| naming | 命名規則に関する指摘 | 変数名、関数名の改善 |
| architecture | 設計・構造に関する指摘 | 責務分離、DRY原則 |
| error-handling | エラーハンドリングに関する指摘 | 例外処理、エラーメッセージ |
| testing | テストに関する指摘 | テスト不足、テストの質 |
| type-safety | 型安全性に関する指摘 | any型、型アサーション |
| readability | 可読性に関する指摘 | 複雑な条件式、コメント不足 |
| other | その他の指摘 | 上記に当てはまらないもの |

### 5. ナレッジファイルの作成・更新

ナレッジの保存先: `~/.claude/review-knowledge/{owner}/{repo}/`

#### 4.1 パターンファイルの作成

各カテゴリごとに `patterns/{category}.md` を作成・更新:

# {Category} レビューパターン

## パターン: {パターン名}

- **クリティカル度**: Critical / Warning / Info
- **前提知識**: このパターンを理解するために必要な背景知識
- **指摘内容**: 何が問題だったか
- **なぜそうすべきか**: 正しいやり方とその理由
- **出典PR**: {PR URL}
- **レビュアー**: {reviewer}
- **日付**: {date}

---

各パターンのクリティカル度は以下の基準で判定:

| レベル | 基準 |
|---|---|
| **Critical** | セキュリティリスク、データ破損、本番障害につながる |
| **Warning** | パフォーマンス劣化、保守性低下、バグの温床になりうる |
| **Info** | より良い書き方、スタイルの統一、可読性向上 |

#### 4.2 インデックスファイルの更新

`index.md` を更新:

# レビューナレッジ: {owner}/{repo}

最終更新: {日時}
総パターン数: {N}件

## カテゴリ別サマリー

| カテゴリ | パターン数 | Critical | Warning | Info |
|---|---|---|---|---|
| security | X | X | X | X |
| ... | ... | ... | ... | ... |

## 学習済みPR一覧

- #{PR番号} {タイトル} ({日付}) - {収集パターン数}件

### 6. 結果の出力

- 新規追加されたパターン数
- カテゴリ別の内訳
- 類似パターンがあった場合はマージした旨を表示

## 重要な注意事項

- 既存のパターンと内容が重複する場合は、マージして出典を追記すること
- 軽微なtypo指摘やLGTMなどのコメントは除外すること
- コードの具体例が含まれる場合、パターンとして一般化して記録すること
- 同じ指摘が複数回ある場合、頻度情報を付加してクリティカル度を上げること
```

# 蓄積したナレッジをもとに自動レビューを実行する

ナレッジの蓄積が完了したら、いよいよ実際のコードレビューに活用します。  
開発中のブランチで以下のカスタムコマンドを実行すると、先ほど蓄積した「自分専用ナレッジ」をベースに、Claudeが現在の変更差分（diff）をチェックし、GitHubのPRへ自動でレビューコメントをポストしてくれます。

人間からの過去のフィードバックを無駄にしないよう、Claudeがポストするコメントは以下のように「過去のコンテキスト」と「今回の改善策」がひと目でわかるフォーマットに統一しています。

* 📚 **前提知識**: この指摘を理解するために必要な背景知識
* 🔍 **指摘内容**: 具体的に何が問題か
* 💡 **なぜそうすべきか**: 正しいやり方とその理由
* ✏️ **修正案**: どう修正すべきか
* 📖 **学習元**: 出典元のコメントのURLなど

```
# パーソナルレビュアー PR投稿コマンド

過去のレビュー指摘から学習したナレッジに基づいてPRをレビューし、GitHubのPRにレビューコメントを直接投稿します。

## 引数

- `$ARGUMENTS`: 対象PRの指定（省略可）
  - 省略: 現在のブランチに紐づくPRを自動検出
  - PR番号: `123` → 指定PRをレビュー

## 実行手順

### 1. PR情報の取得

# 引数なし: 現在のブランチのPRを検出
gh pr view --json number,title,baseRefName,headRefName,url

# PR番号指定
gh pr view $ARGUMENTS --json number,title,baseRefName,headRefName,url

PRが見つからない場合は「PRが見つかりません。先にPRを作成してください。」と表示して終了。

### 2. リポジトリ情報の取得

- `git remote get-url origin` でリモートURLを取得
- owner/repo を抽出

### 3. ナレッジの読み込み

ナレッジ保存先: `~/.claude/review-knowledge/{owner}/{repo}/`

- `index.md` を読み込んでパターン一覧を把握
- `patterns/` 配下の全カテゴリファイルを読み込み
- **ナレッジが存在しない場合**: 「まだレビューナレッジがありません。先に `/review-for-me-learn` を実行してください。」と表示して終了

### 4. PR差分の取得

# PRの差分を取得
gh pr diff $PR_NUMBER

### 5. ナレッジベースのレビュー実行

蓄積されたパターンと変更内容を照合し、該当する指摘を抽出。

各指摘について以下の情報を整理:
- **該当ファイルパス**: コメントを付けるファイル
- **該当行番号**: コメントを付ける行（差分の中の行番号）
- **クリティカル度**: Critical / Warning / Info
- **コメント本文**: 下記フォーマットで構成

### 6. レビューコメントのフォーマット

各コメントは以下のフォーマットで構成:

**[{Critical/Warning/Info}]** {指摘タイトル}

📚 **前提知識**: {この指摘を理解するために必要な背景知識}

🔍 **指摘内容**: {具体的に何が問題か}

💡 **なぜそうすべきか**: {正しいやり方とその理由}

✏️ **修正案**:
\`\`\`{lang}
{修正コード}
\`\`\`

📖 **学習元**: {出典PR URL} ({レビュアー名}) — 過去{N}回指摘

---
_🤖 Generated by [review-for-me](https://github.com) — パーソナルレビュアー_

### 7. レビューの投稿前確認

レビュー結果をユーザーに表示し、投稿前に確認を求める:

## 投稿予定のレビュー

PR: #{PR番号} {タイトル}
投稿先: {owner}/{repo}

### 指摘一覧

1. [Critical] {ファイル}:{行} — {タイトル}
2. [Warning] {ファイル}:{行} — {タイトル}
3. [Info] {ファイル}:{行} — {タイトル}

合計: Critical X件, Warning X件, Info X件

投稿しますか？
- `all`: 全件投稿
- `critical`: Critical/Warningのみ投稿
- `no`: 投稿しない（ローカル表示のみ）
- 番号指定（例: `1,3`）: 選択した指摘のみ投稿

### 8. PRへのレビューコメント投稿

ユーザーの承認後、`gh api` を使ってレビューコメントを投稿する。

**インラインコメント（特定の行へのコメント）の場合:**

# PRレビューを作成（複数コメントをまとめて1レビューとして投稿）
gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews \
  --method POST \
  -f body="## 🤖 パーソナルレビュー結果

ナレッジベース: {パターン数}件のパターンを参照
Critical: X件 / Warning: X件 / Info: X件

---
_Generated by review-for-me — 過去のレビュー指摘から学習したパーソナルレビュアー_" \
  -f event="COMMENT" \
  -f commit_id="$(gh pr view {pr_number} --json commits --jq '.commits[-1].oid')" \
  --raw-field comments='[
    {
      "path": "{ファイルパス}",
      "line": {行番号},
      "body": "{コメント本文}"
    }
  ]'

**特定の行に紐付けられない指摘（全体コメント）の場合:**

gh pr comment {pr_number} --body "{コメント本文}"

### 9. 投稿結果の出力

## 投稿完了

PR: #{PR番号} {タイトル}
URL: {PR URL}

投稿済み: {N}件
- Critical: X件
- Warning: X件
- Info: X件

スキップ: {M}件（ユーザー選択により）

## 重要な注意事項

- **ナレッジに存在するパターンにマッチする指摘のみ行う** — AIモデルの汎用的な知識による指摘はしない
- **全ての指摘に必ず学習元（出典PR・レビュアー名）、前提知識、なぜそうすべきか、修正案を明記すること** — ソースのないレビューは投稿しない
- **投稿前に必ずユーザー確認を取ること** — 勝手にPRにコメントを投稿しない
- ナレッジにマッチするパターンがない場合は「ナレッジベースのパターンに該当する指摘はありませんでした。」と表示して終了
- 過去に同じパターンの指摘が複数回ある場合、コメント内で強調する
- レビューコメントのeventは必ず `COMMENT` を使用する（`APPROVE`や`REQUEST_CHANGES`は使わない — 自分のPRへのセルフレビューのため）
- 同じファイル・同じ行に複数の指摘がある場合、1つのコメントにまとめる
- コメントが長すぎる場合（GitHub制限65536文字）は分割して投稿する
```
