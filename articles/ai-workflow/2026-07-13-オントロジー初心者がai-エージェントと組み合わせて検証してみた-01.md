## TL;DR

* 製造トレーサビリティというテーマで「なぜこの製品は不良？」をAIエージェント（Strands + Amazon Bedrock + Amazon Neptune/SPARQL）に答えさせた。
* **オントロジーを渡す(A)** と **実データだけ(B)** の２つのパターンで、同じ質問を投げてどう変わるかを比較。
* 結果、**同じ問いに、グラフへの問い合わせ言語 SPARQL を A は4〜6回・B は14〜16回**を実行。両方正しい回答ができたが、**答えにたどり着くまでのクエリ数**に明らかに違いが出た。オントロジーが無い B は、まず「どんな項目があり、何と何がつながっているか」を調べるクエリを何本も投げてから、ようやく分析に入っていた。

## はじめに

「オントロジー」という言葉をよく聞くようになり、どうやら AI エージェントと相性がいいらしいぞということでオントロジーの勉強もしつつちょっと手触り感を見てみるために検証してみました。

一方で  
データベースにそのまま繋いで、あとはLLMがいい感じにやってくれるのでは？  
わざわざオントロジー（スキーマ・語彙の定義）を作る意味あるんだろうか？  
という思いもあります。

## 自分なりのオントロジーの理解

**「その分野に、どんな種類のモノがあり、どうつながるかを、コンピュータが扱える形で定義したルールブック」** かなと今のところ自分なりに腹落ちさせました。

ちょっと言葉の整理

* **T-Box（スキーマ）＝オントロジー**: 「`Product` は `belongsToLot` で `Lot` につながる」という**概念と関係の定義**。
* **A-Box（事実）＝実データ**: 「`P-1005` は `LOT-B` に属する」という**個々のインスタンス**。

そして、**この2つを合わせたものがナレッジグラフ**と理解。

```
ナレッジグラフ(KG)
├── オントロジー / スキーマ (T-Box)   … 概念と関係の定義
└── インスタンスデータ (A-Box)         … 個々の事実
```

### データを見ながら解説

言葉だけだと分かりにくいので、検証に使ったデータから抜き出してみます。まず **オントロジー（T-Box＝定義）** はこんな感じです。

実際のファイル（RDFの **Turtle** という記法、記法の中で一番人間に理解しやすいらしい）から抜き出すと、こうです。

```
@prefix mfg:  <http://example.com/manufacturing#> .
@prefix owl:  <http://www.w3.org/2002/07/owl#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .

# Product は「クラス（モノの種類）」である
mfg:Product a owl:Class ;
    rdfs:label "製品"@ja .

# belongsToLot は Product → Lot をつなぐ関係
mfg:belongsToLot a owl:ObjectProperty ;
    rdfs:domain mfg:Product ;   # 主語になれるのは Product
    rdfs:range  mfg:Lot .       # 目的語になれるのは Lot

# consumedMaterialLot は Lot → MaterialLot（ロットが使った原材料）
mfg:consumedMaterialLot a owl:ObjectProperty ;
    rdfs:domain mfg:Lot ;
    rdfs:range  mfg:MaterialLot .

# MaterialLot は purity（純度）という数値を持つ
mfg:purity a owl:DatatypeProperty ;
    rdfs:domain mfg:MaterialLot ;
    rdfs:range  xsd:decimal .
```

オントロジーには製品名などの具体的な個体は出てきません。「製品というものは…」「ロットというものは…」という**種類とつながりのルール**だけを決めています。

一方の **実データ（A-Box＝事実）** は、そのルールに沿った**具体的な事実**の集まりです。同じく実データから、製品 P-1005 に関する部分を抜き出します。

```
mfg:P-1005 a mfg:Product ;                    # P-1005 は 製品 である
    mfg:serialNumber  "P-1005" ;
    mfg:belongsToLot  mfg:LOT-B ;             # LOT-B に属する
    mfg:hasInspection mfg:INS-1005 .          # 検査 INS-1005 を受けた

mfg:LOT-B a mfg:Lot ;
    mfg:consumedMaterialLot mfg:ML-STEEL-002 ,   
                            mfg:ML-RESIN-001 .

mfg:ML-STEEL-002 a mfg:MaterialLot ;
    mfg:purity 91.2 ;                          
    mfg:suppliedBy mfg:SUP-BOLT .

mfg:INS-1005 a mfg:Inspection ;
    mfg:inspectedItem "strength" ;
    mfg:measuredValue 380 ; mfg:threshold 400 ;
    mfg:result "fail" .                        # 測定値380 < しきい値400 → 不合格
```

こうした事実がすべて「**主語 → 述語 → 目的語**」の3つ組（トリプル）で表されます（例:「P-1005 → belongsToLot → LOT-B」）。これをRDFという。

この2つは役割が違います。

* **T-Box（定義）**: 「製品はロットに属し、ロットは原材料を使う」という**型・つながり方**。
* **A-Box（事実）**: 「P-1005 は LOT-B で、LOT-B は純度91.2の ML-STEEL-002 を使った」という**個別の中身**。

## 検証の内容について

### お題：製造トレーサビリティの原因分析

金属加工メーカーを想定。出荷前検査で **製品 `P-1005` が「強度不足」で不合格**になりました。

> \*\*なぜ P-1005 は不合格になった？ \*\*

AI エージェントに自然言語で聞いてみます。  
実データには、製品・製造ロット・原材料ロット（純度つき）・サプライヤー・工程（設備/作業者/ライン）・部品・検査結果を入れてあります。  
データの仕込みとして、「低純度の鋼材」を材料として使ったということにしており、その鋼材を使った製品は不合格品にしています。

### 技術スタック

* グラフDB: Amazon Neptune（RDF / SPARQL）
* エージェント: Strands Agents SDK
* モデル: Amazon Bedrock（Claude Sonnet 5）
* エージェントに与えたツールは2つ
  + `describe_ontology`: オントロジー（T-Box）の語彙を返す
  + `run_sparql_query`: SPARQL(SELECT)を実行

### 検証のパターン 2つの違い

|  | エージェントが使えるツール |
| --- | --- |
| **A: オントロジーあり** | `describe_ontology` + `run_sparql_query` |
| **B: 実データだけ** | `run_sparql_query` のみ |

A と B の違いは「オントロジーにアクセスできるか」

### エージェントにセットしたシステムプロンプト

A と B に共通で、次のプロンプトを与えました。

```
あなたは製造品質の原因分析アシスタントです。
グラフデータベースに接続しており、利用可能なツールを使って自分で情報を調べ、質問に答えます。

進め方:
- まず、利用可能なツールで、扱えるデータの構造や語彙を把握する。
- 必要な SPARQL(SELECT) を組み立てて実行し、結果を確認する。仮説が外れたら問い直す。
- 断定は必ずツールで得た実データの結果に基づく（推測で決めない）。

制約:
- 最終回答は日本語で、結論とその根拠（どの結果から言えるか）を示す。
```

## 検証方法

* 同じ質問を **条件A / 条件B** でそれぞれ実行。
* 指標:
  + **ツール／SPARQL 実行回数**（＝どれだけ手数がかかったか）
  + **真因（低純度鋼材ロット）に到達できたか**（正しさ）
  + **実行ログ（トレース）**: エージェントが実際に投げたSPARQLとその結果
* LLMは出力が揺らぐので、**3回実行して傾向**を見ます。

## 結果

質問はシンプルに **「なぜ P-1005 は不合格になった？」**

### A と B の回答のまとめ部分を比較

**A（オントロジーあり）の結論**

> P-1005 の不合格は、**低純度（91.2%）の原材料ロット ML-STEEL-002（サプライヤー: XXXX Materials Inc.）を使用したことが根本原因**で、それにより強度が規格しきい値(400)を下回り(380)、強度検査で不合格となりました。同ロットを使用した他製品（P-1004も不合格、P-1006はぎりぎり合格）からも、このロットの品質問題が裏付けられます。

**B（実データだけ）の結論**

> P-1005 の不合格は、フレーム部品に使用された鋼材ロット **ML-STEEL-002（純度91.2%、やや低品質）** が原因で強度不足（測定値380 < 基準400）となったことが直接原因です。同ロット材料を使用した他製品（P-1004も不合格、P-1006はぎりぎり合格）の傾向からも、この材料ロットの品質問題が根本原因である可能性が高いと判断されます。

言い回しは違いますが、**指摘している原因も根拠もほぼ同一**で、回答の質の違いは見られませんでした。Sonnet 5 賢いですね。

### 比較サマリ（同じ質問を3回実行）

| 指標 | A（オントロジーあり） | B（実データだけ） |
| --- | --- | --- |
| グラフ探索クエリ（SPARQL）の実行回数 | **6 / 4 / 5** | **14 / 16 / 14** |

エージェントは実行ごとに多少揺らぎが発生しましたが、**A はいつも4〜6回のクエリ実行で完了、B は毎回14〜16回**（約3倍）。

### エージェントによるツール実行ログの抜粋

**A（オントロジーあり）** は、**多方向での情報収集を1本にまとめたクエリ**を書いていました。「P-1005 が **どのロットで作られ・どの工程（ライン/設備/作業者）を通り・どの部品とその原材料（純度/供給元）からできているか**」を**1クエリで一気に**取っています。非常に効率的ですね。

```
SELECT ?lot ?step ?line ?equip ?operator
       ?part ?matLot ?matType ?purity ?supplier ?supplierName
WHERE {
  mfg:P-1005 mfg:belongsToLot ?lot ;
             mfg:processedAt  ?step ;
             mfg:hasPart       ?part .
  ?step  mfg:onLine ?line ; mfg:usedEquipment ?equip ; mfg:operatedBy ?operator .
  ?part  mfg:partMadeFrom ?matLot .
  ?matLot mfg:materialType ?matType ; mfg:purity ?purity ;
          mfg:suppliedBy ?supplier . ?supplier mfg:supplierName ?supplierName .
}
```

**B（実データだけ）** は、オントロジーが無いぶん、**まずデータ自体から「どんな構造なのか」を理解しようとしていました。**

最初にやったのは、語彙（どんな項目・種類があるか）の洗い出しです。

```
-- ① どんな述語（関係・属性）があるかを網羅する
SELECT DISTINCT ?p WHERE { ?s ?p ?o }

-- ② どんな種類（クラス）があるかを網羅する
SELECT DISTINCT ?type WHERE { ?s a ?type }
```

これは、A が `describe_ontology` で一度に受け取っていた「語彙の一覧」を、**データから自力で組み立て直している**動きです。ただしこれで分かるのは「項目と種類の一覧」まで。**どの項目がどの項目とつながるのか**は、まだ分かりません。

そこで次に、**P-1005 を起点に、つながり先をたどる** ということを繰り返します。

下の③のクエリ `SELECT ?prop ?val WHERE { mfg:P-1005 ?prop ?val . }` は、P-1005 を主語にしたトリプルを全部出力しています。

| ?prop（述語） | ?val（値） |
| --- | --- |
| belongsToLot | LOT-B（← LOT-B に属する） |
| hasPart | PART-1005-FRAME / PART-1005-COVER（← 部品） |
| hasInspection | INS-1005（← 検査） |

さらにこの結果を見て、次にたどる先（LOT-B や部品）を知ります。以降の④⑤も同じ書き方で、対象のノードを P-1005 → 部品 → 原材料 … と変えながら辿っていきます。

```
-- ③ まず P-1005 を開く → belongsToLot=LOT-B, hasPart=フレーム/カバー, hasInspection=INS-1005 … が判明
SELECT ?prop ?val WHERE { mfg:P-1005 ?prop ?val . }

-- ④ ③で出てきた先を開く。たとえばフレーム部品を開くと partMadeFrom=ML-STEEL-002 が判明
SELECT ?prop ?val WHERE { mfg:PART-1005-FRAME ?prop ?val . }

-- ⑤ そこで初めて ML-STEEL-002 を開ける → purity=91.2, suppliedBy=SUP-BOLT が判明
SELECT ?prop ?val WHERE { mfg:ML-STEEL-002 ?prop ?val . }
-- … 同様に INS-1005 / 設備 / 作業者 … も開いて確認していく
```

こうして構造を把握してから、ようやく分析クエリに入っていました。

## まとめ

* 今回の検証では、オントロジーがないと正解できないというわかりやすい結果を示すことができませんでしたが、調査の効率性の違いは見ることができました。
* B の調査の仕方は、**データが少ないから成立**したやり方かもしれません。データが大規模になると、1件ずつ調べるのは現実的ではなく、**規模が大きいほど差は拡大、あるいは B は答えにたどり着けなくなる**という可能性も想像できます。
* 今回のデータは述語名が綺麗な命名になっているため（`consumedMaterialLot`など）、A-BOX だけでもその名前を見れば意味も理解できてしまうところがあります。あえてわかりにくい命名で試してみるのも興味深い実験だと思いました。
