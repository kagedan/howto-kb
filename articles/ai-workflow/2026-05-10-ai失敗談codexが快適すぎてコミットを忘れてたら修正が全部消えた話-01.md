## はじめに

最近、Codex Desktop を使って個人開発を進めています。

これがかなり快適です。

仕様を伝えると、設計して、実装して、修正して、動くところまで持っていってくれる。

以前使っていた AI エディター系のツールと比べても、「一緒に作っている」というより、かなり「おまかせで進んでいく」感覚があります。

便利です。  
かなり便利です。

ただ、便利すぎると、人間の方が油断します。

今回はその油断で、久しぶりにやらかしました。

結論から言うと、**こまめにコミットするという自分ルールを忘れたまま Codex に修正を依頼し、意図しない修正を戻そうとした結果、未コミットの変更が消えました。**

完全に自分が悪いです。

ただ、AI エージェントを使う上ではけっこう大事な失敗だったので、記録として残しておきます。

## 1. 自分で決めていた「こまめにコミットする」ルール

AI エージェントを使い始めたころ、自分の中で決めたルールがありました。  
...いろいろ失敗したので（笑）

それは、

* **ファイルの削除、ファイルの移動など、整理はAIにお願いしない**  
  ファイルを消されて...もとに戻せなくなったので...
* **こまめにコミットすること**  
  ファイルの修正を消されてもとに戻せなくなったので...
* **Git管理外のファイルは、AIに作業前にバックアップすることをお願いする**  
  AI使い始めはいろいろ試していたので、ファイルもとに戻したい時が多かったため...

理由は単純で、AI はときどき大きめの変更をします。

人間が手で少しずつ直す場合と違って、AI エージェントは複数ファイルを一気に修正します。

UI、ロジック、設定ファイル、依存関係、場合によってはビルド設定まで一気に触ることがあります。

うまくいけば最高です。

ただ、意図と違う方向に進んだ場合、戻すのが少し大変です。

なので以前は、AI に作業を頼む前に、以下のようなことをしていました。

```
git status
git add .
git commit -m "作業前の退避"
```

きれいなコミットメッセージではなくても、とにかく戻れる状態を作る。

これをやっておけば、AI が少し変な方向に進んでも、Git の履歴から戻れます。

あるいは、最低でも「この時点までは大丈夫だった」という地点を残せます。

……ということは分かっていました。

分かっていたんです。

## 2. Codexが快適すぎて、完全に油断した

今回やらかした理由は、Codex が使いにくかったからではありません。

むしろ逆です。

**Codex が快適すぎたからです。**

最近は、Codex に作業を頼むと、かなり自然に進めてくれます。

こちらが細かく会話しなくても、必要なファイルを探して、実装方針を考えて、コードを書いて、必要なら修正までしてくれる。

その流れがあまりにもスムーズだったので、いつの間にか自分の中で、

> まあ、今回もいい感じにやってくれるだろう

という感覚になっていました。

これがよくなかったです。

以前は AI エージェントを使う前に、少し緊張感がありました。

* どのファイルを触るんだろう
* 変な変更をしないかな
* 戻せるようにしておこう
* 作業前にコミットしておこう

そういう意識がありました。

でも、Codex を使い続けているうちに、うまくいくことが増えました。

うまくいくことが増えると、人間は確認を省きます。

そして、自分で作った安全ルールを忘れます。

今回の失敗は、まさにそれでした。

## 3. 「元に戻して」が、思ったより危険だった

今回、Web サイトの修正を Codex にお願いしました。

ところが、意図した修正になりませんでした。

そこで私は、軽い気持ちでこう言いました。

> 元に戻してください

人間同士の会話なら、たぶんこういう意味です。

* **今やった修正だけを取り消してほしい**
* **直前の変更をいい感じに戻してほしい**
* **関係ない作業までは触らないでほしい**
* **未コミットの別作業は残してほしい**

でも、AI エージェントにとっての「元に戻す」は、かなり危険な言葉です。

* **作業ツリーを戻すのか。**
* **特定ファイルだけ戻すのか。**
* **コミット単位で戻すのか。**
* **未コミット変更を破棄するのか。**
* **差分を見て、手作業で逆修正するのか。**

このあたりを曖昧にしたまま「元に戻して」と言うと、AI 側はそれを「リポジトリをきれいな状態に戻す」と解釈してしまう可能性があります。

そして今回、まさにそれに近いことが起きました。

**コミットしていなかった修正が消えました。**

**数時間分の作業が、きれいに消えました。**

**きれいに消えなくていいんですけどね。(苦笑）**

## 4. AIエディターとCodex Desktopの体験差

ここで感じたのは、Kiro のような AI エディターと、Codex Desktop のような AI エージェントでは、体験が少し違うということです。

AI エディターを使っているときは、比較的「今このファイルを直している」という感覚があります。

* 画面上で差分を見ながら進める。
* 場合によっては、適用前に確認が入る。
* 自分も編集の流れを追いやすい。

もちろん AI エディターでも注意は必要ですが、自分の感覚としては、まだ「共同作業」に近いです。

一方で Codex Desktop は、もっと「作業を任せる」感覚が強いです。

ターミナル操作も含めて、AI が作業単位で進めていく。

ファイルを探し、コマンドを実行し、結果を見て、次に進む。

この体験はかなり強力です。

ただし、強力な分だけ、危険な操作も自然に作業の一部として扱われる可能性があります。

たとえば、以下のような操作です。

* 未コミット変更を破棄する
* リポジトリを巻き戻す
* 履歴を変える
* force push する
* ファイルを大量に削除する
* プロジェクト範囲外のファイルを参照・変更する

このあたりは、便利な AI エージェントほど慎重に扱うべきだと思いました。

今回の件は、「AI が勝手にやった」というより、**自分が AI に十分な安全基準を渡していなかった**という方が近いです。

ここは、かなり反省しました。

## 5. グローバル安全ルールを作ることにした

今回の件があって、Codex に対してグローバルな安全ルールを設定することにしました。

プロジェクトごとのルールではなく、Codex 全体に効く前提のルールです。

内容としては、だいたい以下のようなものです。

```
# Global Safety Rules for Codex

未コミット変更・未追跡ファイル・ローカル設定ファイルは、常にユーザーの作業として扱うこと。
不要だと判断して削除・復元・上書きしてはいけない。

以下の操作は、ユーザーの明示的な許可なしに実行禁止:

- git restore
- git checkout -- <file>
- git reset / git reset --hard
- git clean
- git rebase
- git push --force / --force-with-lease
- orphan branch の作成
- .git の削除・再作成
- コミット履歴を消す、潰す、置き換える push
- リモート履歴と異なる履歴を上書きする push
- ファイルやフォルダの大量削除・大量上書き
- .env, secret, credential, token, key を含むファイルの削除・上書き
- DB reset, drop, truncate, migration rollback などデータ消失につながる操作

破壊的操作が必要になった場合は、実行前に必ずユーザーへ確認すること。

確認時には以下を提示する:

- 実行したいコマンド
- 影響を受けるファイル・ブランチ・データ
- 失われる可能性がある変更
- 代替案
- バックアップ方法

変更を戻す前には、必ず git status --short を確認すること。
必要なら git diff でパッチバックアップを作成すること。
```

特に大事だと思ったのは、この考え方です。

```
Never assume uncommitted or untracked files are disposable.
```

未コミットだから不要、ではありません。

むしろ未コミットの変更こそ、ユーザーが今まさに作っている作業です。

AI から見ると「作業ツリーの差分」に見えるかもしれません。

でも、ユーザーから見ると「数時間分の試行錯誤」です。

この認識を、AI 側に明示しておく必要があると思いました。

---

## 6. それでも、ルールを過信しすぎない

ただし、このルールを入れたからといって、完全に安全になるとは思っていません。

AI への指示は、絶対的なセーフティロックではありません。

ツール側の実装、セッションの読み込み状態、プロジェクトごとの設定、こちらの依頼の曖昧さによって、思った通りに効かない可能性もあります。

なので今後は、グローバル安全ルールを入れた上で、それでも作業前に確認するつもりです。

たとえば、別プロジェクトで作業を始めるときは、最初にこう言うようにします。

```
グローバル安全ルールを読み込んでから作業してください。
破壊的操作、履歴改変、未コミット変更の破棄、force push、ファイル削除は、明示的な許可なしに実行しないでください。
```

もう少し短くするなら、これでもよさそうです。

```
グローバル安全ルールを確認してから作業してください。
```

これだけでも、少なくともこちらの意図は明確になります。

また、AI に任せる前には、やはり自分でも最低限これを確認しておいた方がよさそうです。

そして、大きな修正前にはコミットする。

```
git add .
git commit -m "作業前の退避"
```

結局ここに戻ってきます。

AI 時代でも、バックアップは大事です。

## まとめ：AIが強くなるほど、人間側の安全設計が必要になる

今回の失敗で感じたのは、AI エージェントを使う開発では、コードの書き方だけでなく、**AI に作業させるための安全設計**が必要になるということです。

Codex はかなり便利です。

おまかせで作業してくれる。  
コードも書いてくれる。  
修正も進めてくれる。  
場合によっては、自分が想像していたよりも先回りして動いてくれる。

でも、その便利さに慣れすぎると、人間側の基本動作が抜けます。

今回の自分の場合は、完全にこれでした。

* 作業前にコミットする
* 大きな変更前に差分を見る
* 「元に戻して」と曖昧に言わない
* 破壊的操作は確認させる
* force push や履歴改変は禁止する
* 未コミット変更を AI に捨てさせない

このあたりは、AI 時代でも普通に大事でした。

むしろ、AI が強くなるほど大事になる気がします。

今回の件は、正直ちょっと痛かったです。

でも、これで Codex に対するグローバル安全ルールを作るきっかけになりました。

AI に任せる開発は、たぶんこれからもっと当たり前になります。

だからこそ、

**AI に何をやらせるか**だけでなく、  
**AI に何を勝手にやらせないか**も、ちゃんと設計しておく必要があるのだと思います。

今回の教訓を一言でまとめるなら、これです。

**Codex が快適すぎても、コミットは忘れない。**

……本当に、忘れないようにしたいです。

## 以下、ルールを作ることにしたときのセッションログです。

---

### 🧑‍💻 ユーザー

えっと例えばPushのリセットとか、勝手にリポジトリを巻き戻すとか、そういうのをやらないっていうルールをどこかに設定できませんか

---

### 🤖 AI

はい、設定できます。おすすめは 2 段です。

**1. Codex / AI 向けルールに書く**

プロジェクト直下に AGENTS.md や作業ルール用ファイルを置けるなら、こう書くのがよいです。

```
# Safety Rules

- 未コミットの変更はユーザーの作業として扱うこと。
- `git restore`, `git checkout --`, `git reset`, `git clean`, `rm -rf` など、変更を消す可能性がある操作は禁止。
- どうしても必要な場合は、対象ファイル、失われる可能性のある変更、代替案を説明し、ユーザーの明示的な許可を得ること。
- 変更を戻す場合は、先に `git diff > C:\tmp\backup.patch` のようなバックアップを作ること。
- push、force push、rebase、reset、履歴改変は禁止。ユーザーが明示した場合のみ実行すること。
```

Codex のカスタム指示欄があるなら、そこにも同じ内容を入れるのが一番効きます。

**2. GitHub 側で保護する**

リモートに対しては、GitHub の branch protection でかなり防げます。

* main / master への force push 禁止
* 直接 push 禁止
* PR 必須
* required checks 必須
* branch deletion 禁止

ただし、今回みたいなローカルの git restore は GitHub の保護では防げません。ローカル未コミット変更を守るには、AI ルール側で「戻す前に必ず確認」「必ずパッチバックアップ」が必要です。

今回の件に対する実用ルールとしては、これが一番大事です。

```
Never discard, restore, reset, overwrite, or clean uncommitted tracked files unless the user explicitly asks to discard those exact files. Before any rollback, create a patch backup and show the file list.
```

これは入れておいた方がいいです。私もこのルールを守るべきでした。

```
- 過去のコミット履歴を消す形での push は絶対禁止。
- orphan branch、履歴を持たない新規ブランチ、`git checkout --orphan`、`.git` の再作成、squash して履歴を潰した状態での push、`git push --force` / `--force-with-lease` は禁止。

- push 前には必ず `git log --oneline --decorate -5` と `git status` を確認し、通常の fast-forward / 追加コミットの push であることを確認する。

- 変更を戻す場合は、先に `git diff > C:\tmp\backup.patch` のようなバックアップを作ること。
```
