## はじめに

もともと自分は、AIとの開発セッションログを公開してきました。

なぜセッションログを公開していたのかを考えると、たぶん自分の中に少し不安があったのだと思います。

AIに開発作業を任せると、かなりの速度でコードを読んで、修正して、テストして、まとめてくれます。

それは便利です。

ただ一方で、

* AIが何を見て判断したのか
* どういう理由でその修正にしたのか
* どこまで確認したのか
* 何を確認していないのか
* 人間はどこを見ればよいのか

が後から分からないと、少し怖い感じもありました。

AIがちゃんと作業しているように見えても、その途中経過が見えないと、どこかブラックボックスっぽく感じる。

だから、自分はAIとのセッションログを公開していたのだと思います。

完成したコードだけではなく、AIがどう考えて、どう作業したのかも残しておきたかった。

そうすれば、あとから見返せるし、他の人にも説明しやすい。

そんな感覚がありました。

ただ、しばらくAIを使っていると、少し考え方も変わってきました。

最近のAIは、こちらが細かく全部を追わなくても、かなりしっかり作業してくる場面が増えています。

もちろん、確認は必要です。

でも、AIとの会話を丸ごと残さないと不安、というほどではないのかもしれない。

むしろ、あとから見返すために必要なのは、セッションログ全文ではなく、

* 何を直したかったのか
* どういう計画で進めたのか
* どこまで作業したのか
* 何を確認したのか
* 何が残っているのか

を整理した記録なのではないかと思うようになりました。

そのきっかけのひとつが、AIがすぐコードを直しに行くことでした。

「この機能をどうなってますか」

「この画面の挙動を調べてほしいです」

「このエラーの原因を把握したいです」

と伝えると、AIはすぐに関連ファイルを探して、コードを変更しようとします。

もちろん、それはAI開発の強みです。

小さな修正や試作であれば、そのスピード感はかなり便利です。

ただ、仕事として残したい作業の場合は、少し話が変わります。

いきなりコードを変更されると、何を見て、どう判断して、どこまで確認するつもりなのかが見えないまま進んでしまいます。

そこで、自分はAIに対して、最初にこう言うようになりました。

```
まだコードは変更しないでください。
まずはコードを確認して実装計画を作成してください。
そのあと、その内容に従って作業に入ってほしいです。
```

これが、今の運用のきっかけです。

最初から「Issue / Plan / Progress という運用にしたい」と明確に設計していたわけではありません。

AIがすぐコード修正に進もうとするのをいったん止めて、まず作業を整理してもらう。

そのためにIssueを書かせる。  
Planを書かせる。  
作業したらProgressを残させる。

そうしているうちに、自然と `issue.md` / `plan.md` / `progress.md` をタスクごとに残す形になっていきました。

そして、やってみて気づきました。

自分が残したかったのは、AIとの会話そのものではなかったのかもしれません。

自分が残したかったのは、**AIに何を考えさせ、どういう計画を立てさせ、どこまで作業として進めたのか** だったのだと思います。

つまり、残したかったのはセッションログではなく、

のような、作業の構造だったのだと思います。

この記事では、その形として、`issue.md` / `plan.md` / `progress.md` をタスクごとに残す運用を整理してみます。

## 計画を先に書かせると、AIの動き方が変わる

この「まだコードは変更しないでください」は、単にAIを止めるための言葉ではありません。

AIの役割を、いったん実装者から作業整理役に切り替えるための言葉だと思っています。

先にIssueとPlanを作らせると、人間が確認できるポイントができます。

* ゴールは合っているか
* 成功条件は足りているか
* 触ろうとしている範囲は広すぎないか
* 触ってほしくない場所が混ざっていないか
* 確認方法は現実的か

ここを見てから実装に入れると、AIの速さは残したまま、作業の入口だけ人間が握れる感じがあります。

正直、少し驚きました。

自分が最初から「Issue / Plan / Progress という運用にしたい」と明確に設計していたわけではありません。

むしろ、AIがすぐコード修正に進もうとするのをいったん止めて、ちゃんと計画を立ててから作業してもらおうとした結果、自然とこの形に近づいていきました。

やっている人は、すでにやっている方法だと思います。

ただ、自分にとっては、AIに「いきなり直さないで、まず作業を整理して」と頼むだけで、ここまで作業単位が見えるようになるのは、かなり大きな気づきでした。

## 実際のファイル構成

自分の場合は、作業単位ごとにディレクトリを切る形がしっくりきています。

最初は1つのMarkdownファイルにまとめてもよいと思います。

ただ、実際にAIに作業させると、事前のIssue、Plan、作業中のProgressを分けた方が見通しがよくなりました。

たとえば、次のような構成です。

```
tasks/
  2026-06-11-user-search-fix/
    issue.md
    plan.md
    progress01.md
    progress02.md
    progress03.md
```

ディレクトリ名は、日付と作業内容でもよいです。

```
tasks/
  2026-06-11-user-search-fix/
    issue.md
    plan.md
    progress01.md
```

GitHub Issueなどと紐づけるなら、チケット番号でもよさそうです。

```
tasks/
  #12-user-search-fix/
    issue.md
    plan.md
    progress01.md
```

日付とチケット番号の両方を入れてもよいと思います。

```
tasks/
  2026-06-11-#12-user-search-fix/
    issue.md
    plan.md
    progress01.md
```

この構成にすると、それぞれのファイルの役割が明確になります。

| ファイル | 役割 |
| --- | --- |
| `issue.md` | 何を直したいのか、ゴール、現状、成功条件、制約を書く |
| `plan.md` | どう調査し、どう実装し、どう確認するかを書く |
| `progress01.md` | AIが作業中に行った調査・変更・確認を記録する |
| `progress02.md` | 追加修正や次の作業単位の記録を書く |
| `progress03.md` | さらに続く修正や確認結果を書く |

自分の中では、`issue.md` と `plan.md` は**事前にAIに作成してもらうファイル**です。

一方で、`progress01.md` 以降は、**AIが実際に作業しながら残していく記録**です。

つまり、セッションログを丸ごと残すのではなく、作業ごとの進捗メモとして残します。

## issue.md に書くこと

`issue.md` は、作業の入口です。

ここには、いきなり実装方法を書くのではなく、まず「何を解決したいのか」を書きます。

たとえば、次のような内容です。

issue.md の例を見る

```
# ユーザー一覧画面でメールアドレス検索を追加する

## ゴール

ユーザー一覧画面で、名前だけでなくメールアドレスでも検索できるようにする。

## 現状

現在はユーザー名での検索には対応しているが、メールアドレスで検索しても結果に反映されない。

## 成功条件

- メールアドレスの一部で検索できる
- 既存の名前検索は壊さない
- 検索条件が未入力の場合は、これまで通り一覧表示される
- 既存のテストが通る

## 制約

- 関係ないリファクタリングはしない
- UIの大きな変更はしない
- 既存の検索仕様を壊さない

## 確認方法

- 名前で検索できること
- メールアドレスで検索できること
- 検索条件なしで一覧表示されること
- 関連テストが通ること
```

`issue.md` を作るメリットは、AIに作業の目的を明確に渡せることです。

AIにいきなり「この機能を追加して」と頼むと、意図しない範囲まで変更されることがあります。

先にIssueとして、

を書かせておくと、作業のズレが少なくなります。

## plan.md に書くこと

`plan.md` は、実装前の作業計画です。

ここでは、AIにいきなりコードを変更させるのではなく、まず調査と実装方針を整理させます。

たとえば、次のような内容です。

plan.md の例を見る

```
# 作業計画

## 調査手順

1. ユーザー一覧画面の検索処理を確認する
2. フロントエンド側の検索パラメータを確認する
3. バックエンド側の検索条件を確認する
4. 既存テストの有無を確認する

## 実装方針

- 既存の検索条件にメールアドレスを追加する
- UIは変更しない
- 既存の名前検索の挙動は維持する

## 変更対象になりそうな箇所

- UserList画面
- ユーザー検索API
- ユーザー検索条件の組み立て処理
- 関連テスト

## リスク

- 既存の名前検索の挙動が変わる可能性
- 検索条件のOR/ANDの扱いを誤る可能性
- 大量データ時の検索性能に影響する可能性

## 実装後の確認方法

- 名前検索
- メールアドレス検索
- 未入力検索
- 既存テスト
```

この段階でも、まだ実装はさせません。

まずはPlanを人間が読みます。

ここで、

「そのファイルは違う」  
「その確認が足りない」  
「そこは触らないでほしい」

という判断を入れます。

AIに任せる前に、人間が作業範囲を確認する。

これだけで、かなり安心感が変わります。

## progress.md に書くこと

`progress.md` は、AIが実際に作業しながら残す記録です。

ここには、会話ログではなく、作業結果を書かせます。

たとえば、次のような内容です。

progress.md の例を見る

```
# progress01

## 実施した作業

- ユーザー一覧画面の検索処理を確認した
- バックエンド側の検索条件を確認した
- メールアドレスを検索条件に追加した
- 関連テストを追加した

## 変更したファイル

- src/pages/UserList.tsx
- src/api/users.ts
- src/server/userSearch.ts
- tests/userSearch.test.ts

## 変更内容

- 検索キーワードをユーザー名またはメールアドレスに一致させるように変更
- 既存の名前検索の挙動は維持
- メールアドレス検索のテストを追加

## 実行した確認

- 名前検索のテスト
- メールアドレス検索のテスト
- 検索条件なしのテスト

## 確認できなかったこと

- 本番相当データ量での検索性能確認

## 残っている懸念

- 大量データ時にはメールアドレス検索用のインデックスが必要になる可能性がある
```

これがあると、次のセッションで引き継ぎやすくなります。

セッションログを全部読まなくても、`progress01.md` を見れば、

* 何をしたか
* どのファイルを変えたか
* 何を確認したか
* 何が未確認か

が分かります。

これは、自分で見返すときにも便利ですし、誰かに説明するときにも便利です。

## タスクごとにブランチを作る

この運用は、Gitのブランチとも相性がよいです。

たとえば、1つの修正単位に対して、1つのタスクディレクトリと1つのブランチを作ります。

```
task:   tasks/2026-06-11-user-search-fix/
branch: feature/user-search-fix
```

チケット番号を使うなら、こうです。

```
task:   tasks/#12-user-search-fix/
branch: feature/#12-user-search-fix
```

実際の運用イメージは次のようになります。

```
1. 修正したい内容をAIに伝える
2. AIに tasks/配下へ issue.md と plan.md を作らせる
3. 人間が issue.md と plan.md を確認する
4. タスク用のブランチを作る
5. AIに plan.md に沿って作業させる
6. AIが progress01.md に作業内容を記録する
7. 人間が diff と progress を確認する
8. 問題なければ commit / push する
```

この形にすると、作業単位がかなり明確になります。

```
Issue
↓
Plan
↓
Branch
↓
Progress
↓
Diff
↓
Commit
```

AIに作業させると、どうしても会話の流れで進んでしまいがちです。

でも、ブランチとタスクディレクトリを対応させておくと、作業の単位が崩れにくくなります。

## AIへの最初の依頼例

最初にAIへ頼むときは、次のようにしています。

最初にAIへ渡すプロンプト例

```
以下の修正をしたいです。

【やりたいこと】
ユーザー一覧画面で、メールアドレスでも検索できるようにしたいです。

まず実装はせずに、以下の構成で作業ファイルを作成してください。

tasks/2026-06-11-user-search-fix/
  issue.md
  plan.md

issue.md には以下を書いてください。

- タイトル
- ゴール
- 現状
- 成功条件
- 制約
- 影響範囲
- 確認方法

plan.md には以下を書いてください。

- 調査手順
- 実装方針
- 変更対象になりそうな箇所
- リスク
- 実装後の確認方法
- 作業タスク一覧

まだコードは変更しないでください。
```

この段階では、AIに実装させません。

まず `issue.md` と `plan.md` を作らせて、人間が確認します。

問題なければ、次にこう頼みます。

実装に入ってもらうときのプロンプト例

```
tasks/2026-06-11-user-search-fix/plan.md を読んでください。

このPlanに沿って作業してください。
変更は最小限にしてください。
関係ないリファクタリングはしないでください。

作業内容は progress01.md に記録してください。

progress01.md には以下を書いてください。

- 実施した作業
- 変更したファイル
- 変更内容
- 実行した確認
- 確認できなかったこと
- 残っている懸念
```

この形にすると、AIとの会話ではなく、**タスクディレクトリが作業の中心**になります。

次のセッションになっても、次のように頼めます。

次のセッションで引き継ぐときのプロンプト例

```
tasks/2026-06-11-user-search-fix/ を読んでください。

issue.md、plan.md、progress01.md を確認して、
現在の作業状況と次にやるべきことを整理してください。

まだコードは変更しないでください。
```

これで、セッションをまたいでも作業を再開しやすくなります。

## バイブコーディングは否定しない

ここで言いたいのは、バイブコーディングが悪いという話ではありません。

むしろ、バイブコーディングが向いている作業はかなりあります。

たとえば、

* 試しに作る
* 雰囲気を見る
* UIをざっくり形にする
* 小さなツールを勢いで作る
* プロトタイプを作る
* アイデアを動くものにする

こういう作業は、AIと会話しながら進めた方が速いことがあります。

最初から `issue.md` や `plan.md` を作ると、少し重く感じることもあります。

特に、まだ正解が分からない段階では、会話しながら作って、触って、直していく方が自然です。

一方で、仕事として残したい作業は別です。

たとえば、

* あとで説明が必要な修正
* 影響範囲がある変更
* 誰かに引き継ぐ可能性がある作業
* レビューや判断の根拠を残したい作業
* 本番に入れる可能性がある変更
* 失敗したときに原因を追いたい作業

こういうものは、会話の勢いだけで進めるより、先に `issue.md` と `plan.md` を作っておいた方が合っている気がします。

自分の中では、今後はこの2つを使い分ける形になりそうです。

| 作業の種類 | 向いている進め方 |
| --- | --- |
| 試作、UI確認、小さなツール | AIと会話しながら進める |
| 影響範囲のある修正 | Issue / Plan を作ってから進める |
| 説明やレビューが必要な作業 | Progress を残しながら進める |
| 引き継ぎが必要な作業 | タスクディレクトリを残す |

軽く作るものは、AIと会話しながら進める。

仕事として残したいものは、AIに計画を立てさせて、作業ディレクトリに残す。

この分け方にすると、最近感じていたモヤモヤが少し整理できた気がしています。

## セッションログではなく、作業ディレクトリを残す

この構成にしておくと、セッションログを丸ごと残さなくても、作業の経緯が追いやすくなります。

| 今まで | これから |
| --- | --- |
| セッションログを保存する | タスクディレクトリを残す |
| 会話を読み返す | `issue.md` / `plan.md` / `progress.md` を読む |
| AIが何をしたか会話から探す | progressに変更内容を残す |
| 次セッションで長いログを渡す | タスクディレクトリを読ませる |
| 作業の単位が曖昧 | タスクとブランチが対応する |
| AIに勢いで任せる | 先に作業範囲を確認する |
| 判断の根拠が残りにくい | 成功条件と確認結果が残る |

自分がやりたかったのは、AIとの会話を残すことではなく、**AIに任せた作業の単位を残すこと**だったのだと思います。

その意味では、タスクごとにディレクトリを切り、`issue.md` / `plan.md` / `progress.md` を分けて残すやり方は、かなりしっくりきています。

ここまで書くと、何か新しいAI開発の運用のようにも見えます。

でも、よく考えると、これは普通に仕事でやっていることに近いのだと思います。

人に作業を頼むときも、いきなり「いい感じに直しておいて」だけだと、うまくいくこともあれば、ズレることもあります。

仕事として頼むなら、

* 何をやりたいのか
* どこまでが範囲なのか
* 何を成功とするのか
* 何を触ってほしくないのか
* どう確認するのか

を伝えます。

そして、進捗を聞いたり、成果物を確認したり、必要なら方針を直したりします。

つまり、AIに頼むのも、人に頼むのも、頼み方の基本は意外と近いのかもしれません。

バイブコーディングは、ある意味では「ちょっとこれ、いい感じに作ってみて」と大きめに頼む感覚に近いです。

それはそれで速いし、向いている場面もあります。

一方で、`issue.md` / `plan.md` / `progress.md` に分けて頼むやり方は、もう少し仕事寄りの頼み方です。

先に目的を揃える。  
計画を確認する。  
作業した内容を残す。  
確認したことと、確認できなかったことを分ける。

これは、AIだから特別に必要になったというより、人に仕事を頼むときにも普通にやっていることです。

一周回って、そこに戻ってきた感じがあります。

AIがすごくなったから、人間の仕事の頼み方が全部変わるのかと思っていました。

でも実際には、AIが速く動けるようになったからこそ、最初に何を頼むのか、どう進めるのか、どこまで確認するのかを整理する意味が大きくなったのだと思います。

たぶん、こういうことはもう気づいてやっている人も多いと思います。

ただ、自分の中では、セッションログを残すところから始まって、Issue / Plan / Progress に分けて残す形に落ち着いて、そこでようやく「あれ、これ普通に仕事を頼むときと同じだな」と気づいた感じでした。

その気づきが、最近の自分にはけっこう面白かったです。
