## はじめに

きっかけは、この記事だった。

[情報漏洩に敏感な金融機関で、Claude・Gemini・ChatGPTを導入した話](https://zenn.dev/seiuchi3939/articles/b12d6746d9f187)（てくてくエンジニアさん）

漏洩経路を分解して、既存ガードレール＋許可制で制御する、という良質なガバナンス論だ。この記事を、Claude（うちでは設計・レビュー担当）とChatGPT（独立監査担当、通称「ちゃっぴー」）の両方に読ませた。問いはほぼ同じ。期待していたのは「だいたい同じ指摘が返ってきて、答え合わせができる」ことだった。

ところが、**最初に拾った場所がまるで違った**。ほぼ直交した2つの穴が返ってきて、そして本当の発見は、記事の中身でも、AIの優劣でもなかった。**人間（＝俺）の役割が、いつも言われる「承認ゲート」ではなかった**ことだった。この記事は、その実測ログ。

## 同じ記事、返ってきた穴が違う

ちゃっぴー（ChatGPT）はこう締めた。

> 総合結論：この記事で一番うちに刺さるのはセキュリティ論ではなく、「業務組込Agentの台帳管理」。

あとから振り返ると、**資産管理者の目**に近い。何が存在して、誰がどこに繋がっているか。記事の著者が金融機関のIT管理者だから、同じレイヤーで共鳴して、著者が明示した「台帳管理」に真っ先に反応した。記事に忠実な読み方。

一方、Claudeはこう締めた。

> 「特別な穴は空かない」という最も安心できて最も間違った結論に着地している。

そして、Factory-mo OSの設計原則（§13＝外部取得情報は信頼できないデータ）から見ると「防波堤が一段足りない」と評価した。Claude自身の後の言葉を借りれば、**攻撃者の目**だ。防御線のどこから抜けるか。記事が**沈黙している部分**（間接プロンプトインジェクション）に反応した。

同じ一次資料。片方は「記事が言っていること」を自分の設計に写像し、もう片方は「記事が言っていないこと」を探した。観測軸がほぼ直交していた。能力差というより、少なくとも与えていた役割・文脈の違いと整合する結果だった。

## ここで一回、両方とも外していた

大事なのはここ。物語を綺麗にする前に、事実を残しておく。

**最初のパスで、2つとも相手の穴を見落としていた。**

* Claudeは初回、「台帳が無い」に触れていない
* ちゃっぴーは初回、「間接プロンプトインジェクション」に触れていない

つまり片方だけ回していたら、もう片方の論点は残った可能性が高い。「複数AIを使えば安心」ではない。**少なくとも今回、どちらのAIも一次パスで異なる盲点を持った**、という事実がここにある。異なる役割・文脈を持つ2つの回答を並べた結果、両方の論点が見えた。ここを飛ばして「綺麗に補完した」とだけ書くと、生存者バイアスになる。

## 効いたのは「面白いよね」の一言

で、ここからが本題。

俺は2つの回答を見て、「視点の違い、面白いな」と両方に投げ返した。この瞬間、議論の対象が **AIセキュリティ** から **AIの観測軸の差** に切り替わった。

この論点遷移をやったのは、AIじゃない。俺だ。

Claudeはこう言っていた。

> 二つのAIが違う結論を出したこと自体は、あなたが統合しない限りただの分岐。

正しい。ただ、それを受けてちゃっぴーはもう一段強く言った。

> 人間の仕事はAI回答を統合することだけではない。AI間の"差"を観測し、その差自体を次の問いへ変換すること。

## 自動で直列するだけで、この差は拾えただろうか

もし完全自動化したら、こうなる。

```
記事 ─┬→ ChatGPT → AGENT_REGISTRY 追加
      └→ Claude   → Prompt Injection 対策 追加
```

一見、両方採用すればいい。でも今回実際に起きたのは、単純マージじゃなかった。

```
ChatGPT案
   ↓
人間 ── 比較・違和感 ── Claudeへ投入
   ↓
Claude反論
   ↓
人間 ── 並べて再提示 ── ChatGPTへ投入
   ↓
前提を再構成 → 新しい構造
```

成果は `A + B` じゃなくて `f(A, B, 文脈)` だった。マージではなく、**反復的な再文脈化**。

Claudeの出力を、目的も比較対象も指定せずChatGPTの入力へ流すだけなら、差分検知が走る保証のないパイプになる。もちろん「2つの回答の差異を抽出して次の検証質問に変換せよ」と組んだ差分検知ハーネスを設計すれば、この処理を模倣できる可能性はある（未検証）。ただ少なくとも今回、人間がやった「二つを並べ、差そのものを面白がり、次の問いに変える」という処理は、**単純な直列接続の設計には入っていない**。

## 着地：人間は承認ゲートじゃない

うちの分業モデルは、今までこうだった。

```
人間    = What / 品質承認
Claude  = 設計 / 壁打ち
ChatGPT = 独立レビュー
Codex   = 実装
```

今回の実測だと、人間の役割はもっと強い。

```
人間
├─ 問いを選ぶ
├─ AIを選ぶ
├─ 出力を並べる
├─ 差異を検知する
├─ 差異を別AIへ再投入する
└─ 意味を統合する
```

しかも、この一覧にはまだ「品質承認」を書いていない。承認ゲートは、この処理列の**さらに後ろ**にある。今回見えたのは、承認に至る前段で、人間が **Context Router（文脈の振り分け役）+ Integration Layer（統合層）** として動いていたことだった。承認は、そのループを一周した後の話。

うちが運用ルールにしている「**完全自動AI間ハンドオフ禁止**」は、単なる安全装置じゃなかった。人間を抜くと危険だから、だけではない。単純に自動化すると、この「**差を観測して問いに変換する処理**」が保存される保証がない。だから今のところ、ハンドオフを自動化しない。

## 抽象化すると

最後に一段だけ一般化する。

1. 今回、どちらのAIも一次パスで異なる盲点を持った。だから複数を並べる価値が出た。
2. **でも単純に自動直列するだけでは、盲点が埋まる保証はない。** 差が問いに変換されるとは限らないから。
3. 今回、人間が担っていた機能 ＝ 差の観測 → 問いへの変換 → 再文脈化。この変換を単純な直列接続に丸投げすると、保存される保証がない。
4. 「AI間ハンドオフの自動化を止める」のは、安全装置であると同時に、この変換処理を残すための設計判断。

## おわりに

面白いのは、AIの回答内容そのものより、**「視点の差」を次の問いにした瞬間**が、少なくとも今回、人間が担っていた役割の実測サンプルになっていたこと。

マルチAIを組んでいる人は、たぶん似た瞬間を毎日踏んでいる。それを「自分が統合しているだけ」と流さずに、「**差を問いに変換する処理**」として名前をつけると、自分の運用の**どこを自動化してはいけないか**が見えてくる。複数のAIを役割分担させながら、どこまで自動化するか迷っている人の参考になれば。
