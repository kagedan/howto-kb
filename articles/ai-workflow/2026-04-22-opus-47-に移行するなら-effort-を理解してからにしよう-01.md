## はじめに — AI モデルが「ダイヤル」を持つ時代に入った

Claude Opus 4.7 は 4.6 から価格据え置きのまま、14 ベンチマーク中 12 で性能が上回る。パッと見は「黙ってアップグレードすればいい」話に思える。

ところが実際に触ると、4.6 向けに書いたプロンプトの挙動が微妙に変わる。ツール呼び出しの回数が減る。トークン課金が 1.3 倍になることもある。原因のほとんどは、4.7 で追加された新 effort レベル **`xhigh`** と、effort パラメータそのものの扱いが厳格化されたことにある。

つまり 4.7 以降、AI モデルは単なる「賢さのグレード」ではなく、**タスクごとに強度を切り替える"ダイヤル"を持つ運用機器**になった。モデル ID を選ぶだけの時代は終わり、**ワークロードごとに effort を設計する**工程が必要になる。

この記事では、Opus 4.7 / 4.6 の差分を押さえた上で、実務で一番迷う「effort の使い分け」を、Playwright E2E と Amplify ビルド監視という 2 つの具体的なワークロードで整理する。最後に、Claude Code 上で effort がどう制御されるのか（結論: できない）まで触れる。

### 移行初日あるある：本番が 400 を吐き始める

先に最悪のシナリオを書いておく。4.6 時代の感覚で `temperature: 0` を指定している API 呼び出しが本番コードに 1 箇所でも残っていると、4.7 切替の瞬間から **HTTP 400 が連発**する。「決定性のために 0 にしている」というコメントと共に数年放置されてきた行が、ある日突然死ぬ。これが 4.7 の breaking change の怖さで、**モデル ID を差し替えただけで済まない**最大の理由でもある。

幸いどれも事前に気づけば潰せる。本記事の後半で 10 個のチェックリストにまとめた。

## Opus 4.7 と 4.6 の違い（3 分で押さえる）

公式の What's new を読むとかなり長いが、実務に影響するのは以下 7 点に集約できる。

| 観点 | Opus 4.6 | Opus 4.7 |
| --- | --- | --- |
| 価格 | $5 / $25（1M tokens） | **据え置き** |
| コーディング（93 タスク） | — | **解決率 +13%** |
| CursorBench | 58% | **70%** |
| 画像最大解像度 | 1,568 px（~1.15 MP） | **2,576 px（~3.75 MP）** |
| ビジョン精度 | 54.5% | **98.5%** |
| effort レベル | low / medium / high / max | **+ `xhigh` 追加** |
| 思考制御 | `budget_tokens` | **`adaptive` のみ**（budget\_tokens は 400 エラー） |

これに加えて、実装時に地味に刺さる **Breaking Change** が 3 つある。

* `temperature` / `top_p` / `top_k` を指定すると 400 エラー
* `thinking` コンテンツがデフォルトで返ってこなくなった（`display: "summarized"` で opt-in）
* トークナイザーが更新され、**同じテキストで 1.0〜1.35 倍**のトークンを消費する

つまり「モデル ID だけ差し替えれば OK」ではない。特に思考中の UI を出しているアプリは、何も対策しないとストリーミング開始までの長い沈黙が発生する。

## effort パラメータとは「頭の使い方の強度ダイヤル」

effort は、**Claude が応答にどれだけトークンを費やすか**を制御するパラメータだ。

```
response = client.messages.create(
    model="claude-opus-4-7",
    output_config={"effort": "xhigh"},  # ← ここ
    ...
)
```

ポイントは 3 つ。

### 1. 全トークンに効く

思考（extended thinking）だけでなく、**ツール呼び出しの回数・本文の長さ・コメントの充実度**まで一括で変わる。低くすれば簡潔になり、高くすれば「計画を説明してから動く」ようになる。

### 2. ハードリミットではなく行動シグナル

「このくらい頑張って」というヒントであって、厳密な上限ではない。ハード上限が欲しければ `max_tokens`（1 リクエスト上限）や `task_budget`（エージェントループ全体の目安、4.7 で追加されたβ機能）を併用する。

### 3. レベル表（Opus 4.7）

```
low < medium < high（既定）< xhigh < max
         ↑                         ↑
       コスト重視                品質最優先
```

| レベル | 位置づけ | 推奨用途 |
| --- | --- | --- |
| `low` | 最も効率的 | 短く範囲限定のタスク、サブエージェント |
| `medium` | バランス型 | 平均的ワークフロー、コスト重視 |
| `high` | **API デフォルト** | 品質とトークン効率のスイートスポット |
| `xhigh` | **コーディング/エージェントの推奨開始点** | 反復ツール呼び出し、長時間エージェント |
| `max` | 真のフロンティア問題のみ | 過剰思考リスクあり、コストが大幅増 |

## 新 effort レベル `xhigh` の正体

`xhigh` は 4.7 で新設された **`high` と `max` の中間レベル**。Anthropic は以下のようなワークロードをメインターゲットに据えている。

* 30 分以上走る長時間エージェント
* トークン予算が数百万規模のタスク
* 反復的なツール呼び出し
* 詳細な Web 検索 / ナレッジベース検索

公式推奨は **「コーディングとエージェント用途は `xhigh` から始めろ」**。`max` は「真のフロンティア問題のみ」で、過剰思考によりむしろ品質が落ちるケースがあると明記されている。

### `xhigh` で気をつけること

* `max_tokens` を **64k 以上**に設定する（サブエージェント・ツール呼び出しの余地確保）
* トークン消費は `high` より**顕著に**増える
* 並行して `task_budget` を設定すると、モデルが自己抑制しながら仕事を終える

## ユースケース別の使い分け表

ここから実務の話。私の環境（Amplify Gen2 のモノレポ、React/Vite + AWS + Stripe）で頻出する 2 つのワークロードで整理する。

### Playwright E2E の場合

Playwright は以下の特徴がある。

* `navigate` / `click` / `fill` / `snapshot` を**何度も反復**
* モーダル・非同期表示への**適応的対処**が必要
* セレクタが取れない等の**失敗リカバリ**に推論が要る

これは `xhigh` の設計意図と完全に一致する。

| シナリオ | 推奨 effort |
| --- | --- |
| 探索的テスト / 新規シナリオ作成 / デバッグ | **`xhigh`** |
| 安定した既存シナリオを 1〜2 ケース流す | `high` |
| 単純な画面遷移スモーク | `medium` |
| flaky テストの根因調査 | `max` を一時的に |

### Amplify ビルド監視の場合

Amplify のビルド監視は大きく 2 フェーズに分かれる。ここが面白いところで、**フェーズによって最適な effort が違う**。

| フェーズ | 内容 | 推奨 effort |
| --- | --- | --- |
| ステータスポーリング | `aws amplify get-job` を定期的に叩く | `low` / `medium` |
| ログ解析・原因特定 | CloudWatch / ビルドログから原因を追う | **`xhigh`** |
| CFn 循環依存などの難案件 | スタック依存を静的解析しつつ修正案を練る | `max` |

Playwright と同じく「反復ツール呼び出し＋仮説検証」フェーズが入るので、そこは `xhigh` が刺さる。一方、ただ完了を待つだけのポーリングに `xhigh` を使うのは完全な無駄遣いになる。

### 意思決定のフローチャート

```
そのタスクは
├─ ツール呼び出しを何度も繰り返すか？
│    YES → xhigh（4.7 ならここから）
│    NO
│     └─ 複雑な推論が要るか？
│           YES → high
│           NO
│            └─ 短く範囲限定か？
│                  YES → medium or low
│                  NO  → high（迷ったらこれ）
```

### 3 軸で decide する：タスク種別 × トラフィック量 × 品質要求

フローチャートはタスク単位の判断に効くが、プロダクト全体で effort を設計するときは以下の 3 軸で考えると整理しやすい。

| 軸 | 低 | 中 | 高 |
| --- | --- | --- | --- |
| **タスク種別** | 分類・要約・タグ生成 | フォーム補完・軽いコード生成 | 構造化長文生成・反復ツール呼び出し |
| **トラフィック量** | 1 日数十回（社内ツール） | 1 日数百〜数千回（プロダクト機能） | 1 日数万回以上（主力機能・自動化パイプライン） |
| **品質要求** | 多少ズレても問題なし（下書きレベル） | ユーザーに見せる前提 | SLA・ブランド直結・人命や金銭に関わる |

**組み合わせの指針:**

* 低×高×低 → `low`（高頻度・品質許容度ありのバッチ処理）
* 中×中×中 → `medium`（プロダクトの標準機能）
* 高×低×高 → `xhigh`（主力の構造化生成、少量だが品質最優先）
* 高×中×高 → `xhigh` + `task_budget`（暴走抑制しつつ品質確保）
* 高×高×高 → 設計見直し。`high` 固定 + キャッシュ、もしくは `medium` で数を捌いて不合格分だけ `xhigh` へリランなど

自分のプロダクトの LLM 呼び出しを 3 軸でマッピングすると、「ここは明らかに effort 過剰」「ここは逆に足りない」の凸凹が見えてくる。

## Claude Code では effort をどう制御するか（結論: できない）

ここが今回の記事で一番伝えたい話。Claude Code（Cursor や Zed のエージェントモード含む）でコードを書いていて、「今どの effort で動いてるんだろう？」と気になったことはないだろうか。

### 公式の立場

Anthropic ドキュメントには、Claude Managed Agents について以下の一文しかない。

> Claude Managed Agents handles effort automatically.

**アルゴリズムは非公開**。ユーザー側で指定する手段も提供されていない。

### セッション内から観測できること

| 項目 | 観測可否 |
| --- | --- |
| 現在のモデル ID | ✅ `claude-opus-4-7[1m]` 等わかる |
| **現在の effort 値** | ❌ API レスポンスにも出ない |
| effort が変わった瞬間 | ❌ 通知されない |
| 手動での effort 指定 | ❌ 不可 |

### 間接的に使えるトグル

| 操作 | 実体 |
| --- | --- |
| 通常モード | Opus 4.7（effort はハーネス任せ） |
| `/fast` | **Opus 4.6 に切り替え**（これは effort 調整ではない。モデル自体が変わる） |

つまり Claude Code ユーザーが実用上できるのは「4.7（効率も品質も）」か「4.6（速さ重視）」の 2 択だけ。effort を細かく指定したければ、**自分のコードから Messages API を直接叩く**しかない。

### 自前コードで effort を指定するケース

BMG（Lean Quest）では、LLM 呼び出しは `packages/gen2-shared-backend/amplify/custom/hono/utils/llm-with-tracing.ts` に集約されている。ここは Vercel AI SDK 経由で Messages API を叩いているので、`effort` を明示指定できる。

```
// Before（4.6 時代、adaptive thinking に乗る前）
const result = await generateObjectWithTracing({
  llmModel: "claude-opus-4-6",
  // thinking は budget_tokens で制御していた
});

// After（4.7、effort で制御）
const result = await generateObjectWithTracing({
  llmModel: "claude-opus-4-7",
  providerOptions: {
    anthropic: {
      output_config: { effort: "xhigh" }, // ビジネスモデル生成は重いので xhigh
    },
  },
});
```

ビジネスモデル生成のような長文出力タスクは `xhigh`、タイトル候補生成のような軽いものは `medium` と使い分けるのが素直。

## 実務で活かせる 6 つのポイント

ここまで理屈の話が続いたので、現場視点で「すぐ効く」使いどころをまとめる。

### 1. effort 使い分けで AI コスト 15〜30% 削減

これが一番即効性がある。特に軽量タスクを `medium` に落とすだけで費用対効果が大きい。

| 業務シーン | 最適 effort | 期待効果 |
| --- | --- | --- |
| タイトル・要約・タグ生成 | `low`〜`medium` | 速度 2〜3 倍、コスト半減 |
| 長文生成（構造化 JSON 込） | `xhigh` | 品質向上＋過剰思考回避 |
| E2E テスト・ログ解析 | `xhigh` | 失敗リカバリ精度向上 |
| チャット・Q&A 応答 | `medium` | 体感速度改善 |

月 $1,000 の AI 原価なら **$150〜300/月** の削減余地がある計算。

### 2. ビジョン精度 98.5% で新機能が現実的に

精度不足で諦めていた機能が実用域に入る。

* **画像生成物の自己検証**: 出力したスライド・UI を再度画像で読ませてレイアウト崩れを検出
* **Figma → コード精度向上**: スクリーンショット読み取りが前バージョンと桁違い
* **名刺・ホワイトボード OCR**: ユーザーアップロード画像の業務活用

解像度も 1,568 px → 2,576 px（3.3 倍面積）で、座標が 1:1 ピクセルに対応。coordinate mapping のスケール計算も不要になった。

### 3. 1M コンテキストが標準価格で使える

long-context プレミアムが撤廃されたため、「大量ドキュメント投入」系の機能がコスト現実的になる。過去の議事録・契約書・仕様書を丸ごと読ませるレビュー系エージェントは設計しやすい。

### 4. Playwright / Amplify 運用での `xhigh`

先述の通り、反復ツール呼び出しと仮説検証が混ざる系には `xhigh` が刺さる。flaky テストの根因調査や、ビルドログから循環参照を検出するような作業で体感差が出る。

### 5. `task_budget`（β）でエージェント暴走を抑制

自動修正系エージェントに「このタスクは 100k トークンまで」と予算を提示できる。自己抑制してタスクを終えるので、暴走時の請求事故を防げる。

```
response = client.beta.messages.create(
    model="claude-opus-4-7",
    max_tokens=128000,
    output_config={
        "effort": "xhigh",
        "task_budget": {"type": "tokens", "total": 128000},
    },
    betas=["task-budgets-2026-03-13"],
)
```

### 6. コーディング地力そのものの向上

93 タスクで解決率 +13%、CursorBench +12pt は素直に効く。日常の Claude Code 利用でも体感できるレベル。特に**長い修正タスクを途中で諦めなくなった**のが大きい。

## 気をつけるべき 10 のポイント

活用の裏で、4.7 は地雷も多い。本番移行前にチェックリスト的に潰しておきたい項目。

### 1. `temperature` / `top_p` / `top_k` 指定で 400 エラー

**最優先の監査項目**。`temperature: 0` で決定性を狙っていたコードは全滅する（そもそも 4.6 でも決定性は保証されていなかったが）。

### 2. `budget_tokens` 指定でも 400 エラー

Extended thinking を `budget_tokens` で制御していた箇所は `adaptive` + `effort` に置換が必要。

### 3. トークン課金が最大 1.35 倍

日本語・コード混在ワークロードで顕著。**本番切替前に 1〜2 週間の並行計測**がほぼ必須。予算運用は上振れ前提で。

### 4. 字義通り解釈で既存プロンプトが効かない

4.6 で「空気読み」に頼っていたプロンプトが止まる典型:

* 「もっと良くして」→ 何も変わらない
* 「全体を調整して」→ 1 箇所しか直さない
* サブエージェントを自動で多数起動 → デフォルト減少

推論が浅いと感じたら、プロンプトを盛るより先に effort を上げるのが定石。

### 5. thinking 非表示でストリーミング UI が沈黙

`thinking: { type: "adaptive", display: "summarized" }` で opt-in する。

### 6. `max` effort の過剰思考問題

「一番強いの入れとけばいい」と `max` に固定すると、structured output タスクで**JSON スキーマから外れる**などの品質低下が起きる。迷ったら `xhigh`、`max` は評価で頭打ちを確認してから。

### 7. Claude Code では effort を直接制御できない

Managed Agents 扱いで自動管理、観測もできない。細かく制御したい処理は自前コードで Messages API を叩く。

### 8. `xhigh` 使用時の `max_tokens` 設定忘れ

`xhigh` / `max` はサブエージェントやツール呼び出しが多いため、デフォルトの 4k / 8k だと途中で切れる。**64k 以上**に引き上げる。

### 9. セキュリティ業務で拒否が増える可能性

合法的なペンテスト・脆弱性診断でも拒否されるケースが出る。該当業務は [Cyber Verification Program](https://claude.com/form/cyber-use-case) 申請を。

### 10. 移行タイミングの罠

「黙ってモデル ID 差し替え」は危険。`staging での互換性確認` → `並行運用でコスト計測` → `本番切替` の 3 ステップを推奨。

## まとめ — 「ダイヤル設計」は新しいエンジニアリング領域

冒頭で書いた通り、Opus 4.7 以降、AI モデルは**モデル ID を選ぶだけの道具から、ワークロードごとにダイヤルを回して使う運用機器**に変わった。これはちょうど、CPU に turbo boost と power save が付いたときや、データベースに read replica という概念が追加されたときに似ている。**新しい操作軸が増えた分、設計の仕事も増えた**。

そう思って 4.7 に向き合うと、本記事の内容は下記 3 点に収束する。

* **ダイヤルを理解する**: `low` / `medium` / `high` / `xhigh` / `max` の特性、特に新設の `xhigh` の位置と `max` の過剰思考リスク
* **タスク × トラフィック × 品質** の 3 軸で、プロダクト全体の effort を設計する
* **移行時の地雷 10 個**（`temperature` 400、トークン 1.35 倍、字義通り解釈、thinking 沈黙、`max_tokens` 不足 など）を staging で潰してから本番へ

そして忘れてはいけないのが、**Claude Code 自身はこのダイヤルをユーザーに開放していない**という事実。細かく効率化したければ Messages API を自前のコードから叩くしかない。逆に言えば、ここがプロダクト開発者の差別化ポイントになる。

「速い最新モデルが出たら乗り換える」で済んだ時代は、4.7 で静かに終わった。代わりに\*\*「自分のプロダクトの LLM 呼び出しを 3 軸で棚卸しして、ダイヤルを設計する」\*\*という新しい仕事が来る。手元の LLM 呼び出しコードで、まず 1 箇所だけ `xhigh` を試してみるところから始めてみてほしい。1 つ回せば、残りの設計方針が見えてくる。
