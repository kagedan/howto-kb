## 🎯 はじめに

開発組織に AI レビューを導入・運用している。その中で、現場から繰り返し上がってくる声がある。

**「AIレビュー、結果がブレるんですけど信頼していいんですか？」**

同じPRに対して、あるときは LGTM、  
あるときは「エッジケース未処理」「N+1の懸念」といった複数の指摘が返ってくる。  
これではチームとして判断基準にできない、という懸念はもっともだ。  
ただ、この問題を掘っていくと見えてくるのは「モデルの精度」ではなく、**レビュープロセスの設計不備**である。本記事では、AIレビューがなぜブレるのかを構造的に整理し、組織として安定運用するための戦略を提示する。

---

## 🗺️ 全体像

先に結論を出す。

要点は3つ。

1. AIは「正しいか」を判断しているのではなく、**与えられた粒度で検査しているだけ**
2. 結果がブレるのはモデルの問題ではなく、**レビュー設計が属人的なまま**だから
3. レビューを**目的別に分離し、組織のプロセスとして定義する**ことで安定する

---

## 🔬 何が起きているのか

自分のチームで実際に起きたケースを示す。

### ケースA：エンジニアが自由にレビュー依頼

結果：全体を俯瞰し、明らかな構文エラーや型の不整合がないことを確認して **LGTM**。エンジニアは「AIがOKと言った」と認識する。

### ケースB：観点を指定してレビュー依頼

```
このPRの差分を以下の観点でレビューしてください。
- エッジケースの考慮漏れ
- エラーハンドリングの妥当性
- パフォーマンスへの影響
- セキュリティリスク
```

結果：同一コードに対して、以下の指摘が返ってきた。

* 入力値が空文字列の場合の未処理パス
* リトライ処理における指数バックオフの欠如
* N+1 クエリの可能性

**同じコード、同じモデル、結果だけが違う。** これが現場の混乱を生んでいた。

---

## 🧠 なぜ評価がブレるのか

「AIの精度が不安定」という整理は間違っている。原因は構造的なもので、3つに分類できる。

### 1. レビュー粒度の未定義

最大の要因。プロンプトで「何を見るか」を指定しなければ、LLM（大規模言語モデル）は効率的に全体を俯瞰する軽量スキャンを行う。観点を明示すれば、その軸に沿って深く掘る。

これは人間のレビューでも同じだ。シニアエンジニアに「ざっと見て」と頼めば表面的に確認するし、「セキュリティ観点で」と頼めば攻撃ベクトルを探す。AIだけが特別なわけではない。

### 2. コンテキストの差

LLM に渡す情報量によっても結果は変わる。差分だけを見せるのか、関連ファイル・テストコード・設計ドキュメントも含めるのかで、検出できる問題の範囲が根本的に異なる。

### 3. モデルの非決定性

LLM は同じ入力に対して毎回同じ出力を返すとは限らない（temperature パラメータによる）。ただし、これは結果の「微妙なブレ」に影響する程度で、LGTM が突然詳細な指摘に変わるような**質的な変化**の原因にはならない。

---

## 🧠 AIは「レビューエンジン」ではなく「推論エンジン」

この問題をもう一段抽象化して捉える。

AIレビューは「コードを評価するツール」ではない。本質的には、**与えられたコンテキストから推論を生成するエンジン**である。

つまり、AIは「正しいかどうか」を判断しているのではなく、

推論を行うかによって、出力が変わる。これは次のように整理できる。

* **入力**: プロンプト・コンテキスト（差分、関連ファイル、設計ドキュメント）
* **推論**: LLM の内部処理
* **出力**: レビュー結果

今回の「LGTMと指摘が共存する」という現象は、推論エンジンに対して**異なる入力を与えているだけ**である。

軽量レビューでは、コンテキストが少なく観点も曖昧なため、浅い推論が走り LGTM になる。詳細レビューでは、観点が明示されコンテキストも豊富なため、深い推論が走り指摘が出る。

この構造を理解すると、重要な結論が導ける。

> AIの出力を安定させたければ、モデルを調整するのではなく、  
> **推論条件（入力と文脈）を設計する必要がある。**

これはレビューに限らず、AIを組織に組み込む際の基本原則である。

なお、この現象は Claude Code 固有の話ではない。OpenAI Codex 系のツールでも、GitHub Copilot のレビュー機能でも、レビュー観点・渡すコンテキスト・実行タイミングが異なれば同じことが起きる。LLM を使ったレビューである以上、入力が変われば出力が変わるという構造は共通だ。

## 🔥 自分がやらかした失敗

理屈を整理する前に、自分の失敗談を書いておく。

AIレビューを導入した当初、自分はこういう運用をしていた。

**すべてのPRに対して、同じレビュープロンプトを使い回す。**

変更量が1行だろうが100ファイルだろうが関係ない。コードの種類も見ない。インフラの設定変更にも、ビジネスロジックの大幅な書き換えにも、同じプロンプトを投げていた。

結果どうなったか。

* typo 修正の1行PRに対して、2〜5分かけた詳細レビューが走り「エラーハンドリングが不足しています」と的外れな指摘が出る
* 設計変更を含む大規模PRに20〜30秒の軽量スキャンだけで LGTM が返ってきて、本番で問題が出る
* エンジニアが「AIレビューの結果、どっちを信じればいいんですか？」と聞いてくる

**レビュー結果がブレているのではなく、自分のレビュー設計がそもそも存在していなかった。**

変更の規模も種類も見ずに同じ指示を出すのは、人間のレビュアーに「とりあえず全部見て」と丸投げするのと同じだ。レビュアーによって結果が変わるのは当然である。AIも同じだった。

この失敗を経て、自分は方針を切り替えた。

---

## 💡 精度を上げるのではなく、プロセスを設計する

失敗から得た教訓はシンプルだ。

**「AIレビューの精度を上げる」のではなく、「レビュープロセスを設計する」。**

個々のエンジニアが好きなプロンプトでレビューを依頼する状態は、人間レビューで言えば「レビュー観点も粒度もレビュアー任せ」と同じだ。それでは結果がブレて当然である。

自分の失敗はまさにこれだった。必要だったのは以下の3つ。

1. **レビューの種類を定義する**（何を、いつ、どの深さで見るか）
2. **プロンプトを標準化する**（属人的な依頼をやめる）
3. **プロセスに組み込む**（CI/CD の一部にする）

---

## ⚙️ レビュー戦略の設計

レビューを目的別に3層に分離する。

### Gate 1：軽量レビュー（PR作成時）

開発のフローを止めないことが最優先。致命的な問題だけを検出する。

* **目的**: 明らかなミスの早期検出
* **トリガー**: PR 作成・更新時に自動実行
* **実測時間**: 20〜30秒（差分のみ対象）
* **判断基準**: 高深刻度の指摘があればブロック、それ以外はコメントのみ

軽量レビュー用プロンプト

```
以下の差分を確認し、明らかなバグ・型エラー・セキュリティ上の問題がないかチェックしてください。
問題がなければLGTMとだけ返してください。
```

### Gate 2：詳細レビュー（マージ前）

品質を担保するゲート。ここで時間をかける。

* **目的**: 本番品質の担保
* **トリガー**: main ブランチへのマージ前
* **実測時間**: 2〜5分（差分＋関連ファイル対象）
* **判断基準**: 中深刻度以上の指摘があればブロック

詳細レビュー用プロンプト

```
以下のコードを下記の観点でレビューしてください。
関連するテストコード・型定義も参照してください。

観点:
- エッジケースの考慮漏れ
- エラーハンドリングの妥当性
- パフォーマンスへの影響
- セキュリティリスク

各指摘には深刻度（高/中/低）をつけてください。
```

### Gate 3：横断分析（定期実行）

個別PRでは見えない、全体の劣化を検出する。

* **目的**: 技術的負債・設計の乖離を可視化
* **トリガー**: 週次・月次バッチ
* **許容時間**: 30分以上でも可
* **判断基準**: ブロックせず、レポートとしてチームに共有

## 🏗️ CI/CDへの組み込み

実際のパイプラインへの組み込み例を示す。

ci-review.yml

```
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  light-review:
    runs-on: ubuntu-latest
    steps:
      - name: 軽量レビュー
        run: |
          # PR差分のみを対象に、標準化されたプロンプトでレビュー実行
          gh pr diff ${{ github.event.pull_request.number }} | \
            claude-code review --mode light

  detailed-review:
    if: github.base_ref == 'main'
    runs-on: ubuntu-latest
    steps:
      - name: 詳細レビュー
        run: |
          # マージ先がmainの場合のみ、詳細レビューを実行
          gh pr diff ${{ github.event.pull_request.number }} | \
            claude-code review --mode detailed
```

運用の全体像をまとめる。以下は自分の環境（Claude Code、TypeScript / Go 中心のリポジトリ）での実測値であり、モデルやコードベースの規模によって変わる点に注意してほしい。

| Gate | トリガー | 対象 | 実測時間 | ブロック条件 |
| --- | --- | --- | --- | --- |
| 軽量 | PR作成・更新 | 差分のみ | 20〜30秒 | 高深刻度のみ |
| 詳細 | mainマージ前 | 差分＋関連ファイル | 2〜5分 | 中深刻度以上 |
| 横断 | 週次バッチ | リポジトリ全体 | 10〜30分 | ブロックしない |

## ⚖️ トレードオフと経営判断

この設計には明確なトレードオフがある。

* **精度を上げる** → API コスト増加・レビュー待ち時間の増加
* **速度を上げる** → 見逃しリスクの増加
* **頻度を上げる** → 月額コストの増加

「すべてを詳細レビューすればいい」は、コストと開発速度の両面で現実的ではない。CTO として求められるのは、**どのフェーズでどの精度を要求するかを明示的に決める**ことだ。

自分の組織では以下の基準で運用している。

* **PR 段階**: 致命的なバグを防ぐ。軽量で十分。開発速度を落とさない
* **マージ前**: 品質を担保する。ここは時間をかけてよい
* **定期実行**: 全体の健全性を可視化する。経営判断の材料にもなる

重要なのは、これを**暗黙知にせず、チームの合意としてドキュメント化する**ことだ。

---

## 📝 まとめ

* AIレビューの評価がブレるのは、モデルの不安定さではなく**レビュー粒度の未定義**が原因
* LGTM は「問題がない」ではなく「その粒度では問題が見つからなかった」を意味する
* レビューを**軽量・詳細・横断の3層に分離**し、プロセスとして定義することで安定する
* 組織に導入する際は、**速度・精度・コストのトレードオフを明示的に決める**必要がある

---

## 👋 おわりに

AI をチームに導入するとき、最初に聞かれるのは「精度はどうか」だ。しかし本質的な問いは精度ではない。

> **AI の出力を安定させたければ、入力と文脈を安定させるプロセスを設計せよ。**

これはレビューに限らず、AI を組織の開発プロセスに組み込むすべての場面で共通する原則だと考えている。「AIがブレる」と感じたら、まずプロセスの設計を疑ってみてほしい。

### 次にやること

* レビュー観点のテンプレートをチームごとにカスタマイズする仕組みの整備
* Gate ごとの検出率を計測し、投資対効果を定量的に評価する
* レビュー結果の蓄積と改善ループへの接続（[AI改善ループの設計](https://zenn.dev/hidetzu/articles/ai-improvement-loop-design)との統合）
