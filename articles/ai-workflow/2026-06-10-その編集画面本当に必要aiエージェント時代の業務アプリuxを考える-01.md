# エージェント時代の業務アプリケーションとは何か

## 社内業務は「操作するアプリ」から「人とAIが共同作業するワークスペース」へ変わる

生成AIやAIエージェントの進化によって、社内業務のあり方は大きく変わろうとしている。

これまでも、業務アプリケーションは社内業務を支える重要な存在だった。  
申請、承認、顧客管理、契約管理、問い合わせ対応、レポート作成、データ集計。  
多くの業務は、何らかのアプリケーション上で行われてきた。

しかし、エージェントが業務に入り込むと、アプリケーションの意味そのものが変わる。

これまでのアプリケーションは、**人間が作業するための道具**だった。  
これからのアプリケーションは、**人間とエージェントが共同作業するためのワークスペース**になる。

この記事では、特に社内業務をエージェントに置き換えていく文脈で、これからの業務アプリケーションに求められるものを考えてみたい。

---

## これまでのアプリケーションは「人が作業する場所」だった

従来の業務アプリケーションでは、人間が主役だった。

たとえば、経費精算システムであれば、人間が日付、金額、支払先、摘要、添付ファイルを入力する。  
顧客管理システムであれば、人間が顧客情報を登録し、対応履歴を更新する。  
契約管理システムであれば、人間が契約書をアップロードし、ステータスを変更し、必要な確認事項を入力する。

つまり、これまでのアプリケーションは基本的に、次のような前提で設計されていた。

* 人間が入力する
* 人間が編集する
* 人間がボタンを押す
* 人間が一覧から対象を探す
* 人間が内容を転記する
* 人間がチェックする

そのため、良いアプリケーションとは「人間が操作しやすいアプリケーション」だった。

入力フォームが分かりやすい。  
ボタンの位置が自然である。  
バリデーションが適切である。  
検索しやすい。  
一覧が見やすい。  
編集画面が使いやすい。

もちろん、これは今後も重要である。

ただし、エージェントが業務を担うようになると、アプリケーションの中心は少しずつ変わっていく。

特に大きいのは、「入力」や「編集」の前提が変わることだ。

これまでは、人間が文書やメールや音声を読み取り、必要な情報を業務アプリケーションに入力していた。  
しかしエージェント時代には、情報が含まれる文書や音声をまるごと渡し、そこから必要な情報をエージェントが抽出・構造化する流れが増えていく。

請求書PDFから金額や支払期限を抽出する。  
契約書から契約期間やリスク条項を整理する。  
問い合わせメールから顧客名や対応方針を抽出する。  
会議音声から議事録、TODO、担当者、期限を生成する。

このとき、人間の仕事は、空欄のフォームを埋めることではない。  
エージェントが抽出した情報を確認し、誤りを直し、必要なら修正指示を出すことになる。

## これまでとこれからのアプリケーション

## エージェント時代、人間の仕事は「作業」から「依頼・確認・判断」へ移る

エージェントが社内業務に入り込むと、人間がすべての作業を直接行う必要はなくなる。

たとえば、これまで人間が行っていた次のような作業は、エージェントがかなりの部分を担えるようになる。

* メールやチャットの内容を読み取る
* 必要な情報を社内文書から探す
* 申請内容の下書きを作る
* 顧客対応案を作る
* 議事録を整理する
* データを集計する
* レポートを作成する
* 契約書や規程との不整合をチェックする
* 必要なタスクを洗い出す
* ワークフローを次の担当者に回す

このとき、人間の役割は「すべてを自分で操作すること」ではなくなる。

人間の主な役割は、次の3つに近づく。

1. **依頼する**
2. **確認する**
3. **判断する**

「この顧客対応の返信案を作って」  
「この申請内容を確認して」  
「この契約書のリスクを洗い出して」  
「このデータをもとにレポートを更新して」  
「この稟議に必要な添付資料を集めて」

このように、人間はエージェントに仕事を依頼する。  
エージェントは情報を集め、処理し、下書きを作り、必要な更新を行う。  
そして人間は、その結果を確認し、必要に応じて承認したり、差し戻したり、修正を指示したりする。

つまり、人間の仕事は、**Input / Edit / Operation** から、**Intent / Review / Direction** に移っていく。

* Intent：何をしたいかを伝える
* Review：結果を確認する
* Direction：修正や次の指示を出す

ここに、エージェント時代の業務アプリケーション設計の大きな変化がある。

---

## これから重要になるのは Edit ではなく View である

従来の業務アプリケーションでは、Edit UI が非常に重要だった。

いかに入力しやすいか。  
いかに編集しやすいか。  
いかに間違えずに登録できるか。  
いかに効率よく操作できるか。

しかし、エージェントが作業を担うようになると、人間が毎回すべての項目を直接編集する必要は減っていく。

その代わりに重要になるのが、**View** である。

エージェントが作成したものを、人間が確認する。  
エージェントが変更した内容を、人間がレビューする。  
エージェントが提示した判断材料を、人間が理解する。  
エージェントが検知したリスクを、人間が見て判断する。

このとき、アプリケーションに求められるUXは「入力しやすさ」よりも、**確認しやすさ**になる。

なぜなら、エージェント時代でも、最終的な判断責任は人間や組織に残るからである。

エージェントが作ったものを、人間が理解できない。  
どこが変わったのか分からない。  
何を根拠にそう判断したのか分からない。  
どこにリスクがあるのか分からない。  
承認してよいのか判断できない。

この状態では、人間は安心してエージェントに仕事を任せられない。

だからこそ、エージェント時代のアプリケーションでは、View がUXの中心になる。

## ViewがUXの中心になる

## 良いViewとは何か

では、エージェント時代に必要な View とはどのようなものだろうか。

単に「結果を表示する画面」だけでは足りない。  
エージェントが作業した結果を、人間が責任を持って確認できる必要がある。

そのためには、少なくとも次の4つの View が重要になる。

### 1. 結果の View

まず必要なのは、エージェントが作成した成果物を分かりやすく表示する View である。

たとえば、経費精算であれば、申請内容のサマリー、合計金額、費目ごとの内訳、添付資料の一覧が見える。  
顧客対応であれば、返信案、顧客の過去問い合わせ、参照したFAQ、推奨アクションが見える。  
契約確認であれば、契約書の要約、懸念条項、確認が必要な箇所が見える。

人間が短時間で全体像を把握できることが重要である。

### 2. 差分の View

次に重要なのが、差分である。

エージェントが何を変更したのか。  
前回の内容とどこが違うのか。  
どの項目が追加され、どの項目が削除され、どの表現が書き換えられたのか。

これは、Gitのdiffや、Wordの変更履歴、Google Docsの提案モードに近い考え方である。

完成物だけを見せられても、人間は不安になる。  
特に業務では、「何が変わったのか」が分からないとレビューできない。

エージェント時代のアプリケーションには、あらゆる業務成果物に対して差分を確認できる仕組みが必要になる。

### 3. 根拠の View

エージェントの出力には、根拠が必要である。

どの文書を参照したのか。  
どの規程に基づいたのか。  
どの顧客情報を見たのか。  
どのデータを集計したのか。  
どの過去事例を参考にしたのか。

社内業務では、答えそのものよりも「なぜその答えになったのか」が重要になる場面が多い。

特に、金融、保険、医療、法務、人事、経理のような領域では、根拠のない出力は業務に使いにくい。

エージェントの判断や提案を、業務上の文脈で検証できる View が必要になる。

### 4. リスクの View

最後に重要なのが、リスクの View である。

エージェントが自信を持てない箇所。  
データが不足している箇所。  
ルールが曖昧な箇所。  
人間確認が必要な箇所。  
承認前に注意すべき箇所。

こうしたものを、アプリケーションが分かりやすく提示する必要がある。

良いエージェントアプリケーションは、単に答えを出すだけではない。  
**人間が見るべきところを浮かび上がらせる**。

これは、社内業務のエージェント化において極めて重要なポイントだと思う。

---

## 修正も「直接編集」から「修正指示」へ変わる

では、Edit UI は不要になるのだろうか。

そうではない。

ただし、Edit の意味は変わる。

これまでは、人間がフォームやエディタを使って直接編集していた。  
しかし、エージェント時代には、人間が直接編集するよりも、エージェントに修正指示を出す場面が増える。

たとえば、次のような操作である。

* 「この表現をもう少し柔らかくして」
* 「この部分は社内規程に合わせて修正して」
* 「A案ではなくB案を前提に再計算して」
* 「この顧客は例外対応として扱って」
* 「このリスクは強調して」
* 「この資料も根拠に追加して」
* 「この説明は役員向けに短くして」

これは、従来の意味での編集とは少し違う。

人間がすべての文字や項目を直接書き換えるのではなく、エージェントに対して方向性を伝える。  
エージェントが修正案を作る。  
人間が差分を確認する。  
必要ならさらに修正指示を出す。

つまり、Edit は「手を動かす編集」から「意図を伝える編集」へ変わる。

アプリケーションには、直接編集のUIだけではなく、次のような機能が求められるようになる。

* 修正指示を出す
* 修正案を比較する
* 差分を確認する
* 承認する
* 差し戻す
* 元に戻す
* 変更履歴を追跡する

このように考えると、エージェント時代のアプリケーションは、単なるチャットUIでは足りない。

チャットだけでは、業務の状態、成果物、差分、根拠、承認履歴を扱いにくい。  
必要なのは、チャットと業務オブジェクトとレビューUIが統合されたワークスペースである。

---

## アプリケーションは「状態管理装置」になる

社内業務において、アプリケーションの価値はどこに残るのだろうか。

エージェントが作業を実行できるなら、アプリケーションは不要になるのか。

私は、むしろ逆だと思う。

エージェント時代には、アプリケーションはより重要になる。  
ただし、その役割は「人間に操作させる画面」から、「人間とエージェントの共同作業状態を管理する場所」に変わる。

社内業務では、次のような状態管理が欠かせない。

* 今どの案件が進行中か
* 誰が何を依頼したか
* エージェントが何を実行したか
* どの資料を参照したか
* どの成果物が最新か
* 何が承認済みか
* 何が未確認か
* どこにリスクがあるか
* どの判断を人間が行ったか
* 監査上、どのログを残すべきか

エージェントは作業を実行する。  
しかし、その作業が業務として成立するには、状態、履歴、権限、責任、承認、監査が必要になる。

ここに、業務アプリケーションの新しい役割がある。

アプリケーションは、人間が操作する道具ではなく、**人間とエージェントの共通状態を管理するワークスペース**になる。

---

## PoCでは「編集可能なファイル」が強い

もう一つ、実務的に重要だと思うことがある。

社内業務をエージェントに置き換えるPoCを進めるとき、最初から立派な業務アプリケーションを作る必要はない。  
むしろ、最初は編集可能なファイルを中心にしたほうが進めやすい場面が多い。

たとえば、次のようなファイルである。

* Markdown
* JSON
* YAML
* CSV
* HTML
* SQL
* Mermaid
* 設定ファイル
* プレーンテキスト

これらは、人間にもエージェントにも扱いやすい。

人間は中身を読める。  
必要なら直接修正できる。  
エージェントも読み書きしやすい。  
Gitなどで差分管理しやすい。  
テンプレート化しやすい。  
他のシステムにも連携しやすい。

たとえば、業務ヒアリングの結果をMarkdownで整理する。  
業務ルールをYAMLで表現する。  
申請データをJSONで持つ。  
一覧データをCSVで扱う。  
画面モックをHTMLで生成する。  
プロセス図をMermaidで作る。

このようにしておくと、人間とエージェントが同じ対象物を見ながら共同編集できる。

![編集対象はファイルであると強い](https://static.zenn.studio/user-upload/af1d89da2bda-20260610.png)

もちろん、本番システムではDB、API、権限管理、監査ログ、ワークフローエンジンなどが必要になる。  
しかし、PoCや初期検証の段階では、編集可能なファイルを中心にしたほうが圧倒的に速い。

特にエージェントは、テキストを読み、構造化し、生成し、更新することが得意である。  
だから、エージェントと共同作業する対象は、まずテキストベースのファイルにしておくと扱いやすい。

これは、エージェント時代の業務アプリケーションを考えるうえで、意外と重要な設計原則になるのではないかと思う。

---

## 「チャットで全部できる」わけではない

エージェントというと、ついチャットUIを想像しがちである。

もちろん、チャットは重要である。  
人間が自然言語で依頼できることは、エージェント時代の大きな変化である。

しかし、社内業務を本当にエージェントに置き換えるなら、チャットだけでは足りない。

なぜなら、社内業務には成果物があり、状態があり、承認があり、履歴があり、責任があるからである。

チャットで依頼するだけでは、次のような問題が出る。

* 最新の成果物がどれか分からない
* 変更履歴が追いにくい
* 根拠資料が埋もれる
* 承認状態が分かりにくい
* 誰が何を判断したか残しにくい
* 業務オブジェクトとの紐づきが弱い
* 監査に耐えにくい

そのため、エージェント時代のアプリケーションは、単なるチャットボットではなく、ワークスペースである必要がある。

チャットは入口である。  
しかし、業務はチャットの中だけで完結しない。

必要なのは、チャット、成果物、タスク、データ、根拠、差分、承認、履歴が統合された場である。

---

## 社内業務のエージェント化で重要になる設計原則

ここまでの話をまとめると、社内業務をエージェントに置き換えるアプリケーションには、いくつかの設計原則が見えてくる。

### 1. 人間に作業させるのではなく、判断させる

アプリケーションは、人間に大量の入力作業をさせるものではなくなる。  
人間がすべきことは、目的を伝え、結果を確認し、判断することである。

### 2. Viewを中心に設計する

結果、差分、根拠、リスクを見やすくする。  
人間が安心してレビューできることがUXの中心になる。

### 3. 修正は指示として扱う

人間がすべてを直接編集するのではなく、エージェントに修正指示を出す。  
そのうえで、差分を確認し、承認する。

### 4. 業務状態を管理する

誰が何を依頼し、エージェントが何を行い、何が承認され、どこにリスクがあるのか。  
アプリケーションは、人間とエージェントの共同作業状態を管理する。

### 5. 編集可能な成果物を持つ

PoCや初期検証では、Markdown、JSON、YAML、CSVなどの編集可能なファイルを中心にするとよい。  
人間にもエージェントにも扱いやすく、差分管理もしやすい。

### 6. チャットだけで完結させない

チャットは依頼の入口として重要だが、業務には成果物、状態、承認、履歴が必要である。  
チャットとワークスペースを組み合わせることが重要になる。

---

## おわりに

エージェント時代のアプリケーションは、単に既存の画面にAIボタンを追加したものではない。

それは、人間がすべてを操作するための画面でもない。  
エージェントが裏側ですべてを勝手に処理するブラックボックスでもない。

これからのアプリケーションは、**人間とエージェントが共同作業するワークスペース**になる。

人間は目的を伝える。  
エージェントは作業する。  
アプリケーションは状態を管理する。  
人間は結果、差分、根拠、リスクを確認し、判断する。  
必要なら、エージェントに修正指示を出す。

この構造において、UXの中心は Edit から View へ移る。

いかに入力しやすいかではなく、いかに確認しやすいか。  
いかに操作しやすいかではなく、いかに判断しやすいか。  
いかに人間に作業させるかではなく、いかに人間が責任を持ってエージェントの成果を扱えるか。

社内業務のエージェント化が進むほど、アプリケーションの役割は変わっていく。

アプリケーションは「操作するもの」から「共同作業する場所」へ。  
そして、業務システムのUXは「入力と編集」から「確認と判断」へ。

この変化を前提に、これからの社内業務アプリケーションを設計していく必要がある。
