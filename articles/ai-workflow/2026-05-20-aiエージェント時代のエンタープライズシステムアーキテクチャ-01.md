# 分散・自律協調型マルチエージェント基盤の数理的定式化

## 概要 (Abstract)

本稿は、AIエージェントが企業内システムと自律的に連携し業務を遂行する次世代アーキテクチャを、論理的に構築・体系化することを目的とする。

従来の「中央集権型LLMオーケストレーター」が抱えるスケーラビリティ限界、コスト構造の硬直性、密結合のアンチパターンを指摘した上で、「処理命令がインフラ側へ移動して実行される分散型マルチエージェントシステム」を提唱する。各システムは特化型小規模言語モデル(SLM)を内包するスマートエンドポイントとして再定義され、システム間の連携は「セマンティック契約(AAPI)」による意図の動的マッチングで駆動される。

ゼロトラスト・アーキテクチャ(ZTA, NIST SP 800-207)、OAuth 2.0 Token Exchange (RFC 8693)、連続アクセス評価プロトコル(CAEP, OpenID Foundation)、および補償トランザクション(Sagaパターン)に基づく結果整合性モデルを統合することで、レガシーシステムの物理的制約を緩和しつつセキュリティと可用性を両立する基盤を提示する。

本稿で用いる数式は **(a) 厳密な数学的定義**、**(b) 設計を表現するための概念モデル**、**(c) 直感を補助する比喩** の三層に分類される。各章の冒頭で該当分類を明示する。これに加え、可観測性、コストモデル、失敗モード、移行戦略、規制対応、限界と反証可能性についても論じる。

## 第1章 — 序論: パラダイムシフトの必然性

### 1.1 HCI から ACI への移行

人間とコンピュータのインターフェース(Human-Computer Interaction, HCI)は、GUI、CLI、API という形で進化してきた。AIエージェントの自律的な台頭は、このインターフェースの主体を人間から「プログラム可能な知能体」へとシフトさせる。すなわち、Agent-Computer Interaction(ACI)の時代への突入である。

ACI において、エージェントは人間のように曖昧な指示を解釈するのではなく、構造化された意図(Intent)を機械可読な形式でシステムに伝達する。このパラダイムシフトは、従来の「人間のためのUI」最適化から「機械のためのAPI」最適化への根本的な設計思想の転換を要求する。エージェントにとって、OpenAPI/Swagger仕様書や CLI の `--help` 出力は、人間にとっての GUI に相当する。すなわち、システムの**自己記述性(Self-Descriptiveness)** が ACI の必須要件となる。

### 1.2 中央集権型オーケストレーションの限界

現在主流のアーキテクチャは、単一の巨大言語モデル(LLM)がすべての推論を担い、各システムの API を順次呼び出して結果を待つ「オーケストレーション型」である。この設計は以下の限界を内包する:

1. **コンテキスト肥大化(Lost in the Middle)**: 複数システムをまたがるタスクのコンテキストが拡大し、LLM の中間層における推論精度が低下することが経験的に報告されている。
2. **密結合(Tight Coupling)**: 中央 LLM が各システムの API 仕様をすべて知悉する必要があり、システムの変更が中央に波及する。
3. **コスト構造の硬直性**: ユーザー ID ごとに LLM リソースを確保する設計は、アイドル状態でも GPU メモリを占有する。ただし「コストがユーザー数に比例して爆発する」とまで言えるかは設計次第であり、本稿はそのリスクを指摘するに留める(定量モデルは第14章で後述)。
4. **単一障害点(SPOF)**: 中央 LLM の障害またはプロンプトインジェクションによる乗っ取りがシステム全体に影響を及ぼす可能性がある。

本稿は、これらの限界を緩和するための「分散型・自律協調型マルチエージェントアーキテクチャ」を提示する。なお、本提案が**すべての企業環境において最適である**と主張するものではなく、適用条件と限界を第17章で明示する。

## 第2章 — システムトポロジーの数理的定義

> **記法ラベル**: 本章のグラフ表現は **概念モデル** であり、設計を整理するための形式化である。確率的オートマトンとペトリネットは **厳密な数学的定義** に従う。圏論的記述は **概念モデル** として用いる。

### 2.1 有向グラフとしてのシステム空間

企業内システム全体は、有向グラフ G = (V, E) として概念モデル化される。

* **頂点集合 V**: すべての計算リソース(API エンドポイント、データベース、LLM 推論ノード、レガシーシステム)を表す。V\_{\text{LLM}} \subset V は推論基盤上の LLM 実行環境の集合である。
* **有向辺集合 E**: 頂点間の通信経路を表す。各辺 e \in E には、セキュリティ制約 \Phi(e) が付随する。

このグラフ表現は、トポロジー可視化と論理整合性チェックに有用であるが、実際のシステム挙動はマルチエッジ・動的辺生成・確率的遅延等の要素を含むため、より精密なモデル(マルコフ過程、ペトリネット)で補完する必要がある。

### 2.2 エージェントの確率的オートマトンとしての定式化

エージェント A を、状態空間 S を観測し行動空間 U から API/CLI 呼び出しを選択する **マルコフ決定過程(MDP)** としてモデル化する:

\pi: S \to \mathcal{P}(U)

ここで \mathcal{P}(U) は実行可能な API 群上の確率分布である。**ただし重要な但し書きとして**、現実の LLM ベースエージェントは厳密にはマルコフ性を満たさない(過去のコンテキスト全体に依存するため、\pi は S の履歴に依存する非マルコフ過程に近い)。MDP モデルは設計と分析の補助として有用な近似である。

複数エージェント間の並列性と同期を表現する場合、**ペトリネット(Petri Net)** PN = (P, T, F, M\_0) が有用である。プレース P はシステム状態、トランジション T はエージェントのアクションを表し、トークンの流れが制御フローを可視化する [Petri 1966]。

### 2.3 関手としてのインターフェース抽象化(概念モデル)

既存システムを CLI/API でラップする「ストラングラーフィグ・パターン」を、圏論における **関手(Functor)** として表現する:

F: \mathcal{L} \to \mathcal{M}

ここで \mathcal{L} はレガシーシステムの内部状態を表す圏、\mathcal{M} はエージェントが解釈可能な構造化データ(JSON, YAML)の圏である。

> **重要な但し書き**: 圏 \mathcal{L}, \mathcal{M} の対象と射、合成則、恒等射を厳密に定義しない限り、これは**圏論的アナロジー(概念モデル)** に留まる。厳密な関手証明を要するわけではなく、「内部実装が外部インターフェースから独立して進化できる」という設計原則を表現する**メタファ**として用いている。

この抽象化により、ラッパー内で各システムが内部ロジックを変更しても、他のシステムや中央基盤に影響を及ぼさない。これが「システムがラッパーの中で成長する」という設計原則である。

### 2.4 冪等性 (Idempotency) の代数的性質

エージェントは確率的モデル(LLM)であるため、ネットワーク遅延や推論のゆらぎにより同一 API 呼び出しを重複実行するリスクを内包する。したがって、エージェント向けに公開される API/CLI(遷移関数 f)は、**冪等性(Idempotency)** を満たすことが望ましい。状態空間 S 上の遷移関数 f: S \to S が冪等であるとは:

\forall x \in S: f(f(x)) = f(x)

これは関数論における冪等元の定義である。

API ゲートウェイ層における `Idempotency-Key` ヘッダーと分散 KVS による状態管理により、エージェントが同一操作を複数回適用してもシステム状態が発散しないことを実装的に保証する。

## 第3章 — アイデンティティと権限移譲のモデル

> **記法ラベル**: 本章の集合包含関係は **厳密な定義** に従う。

### 3.1 二重アイデンティティ構造

エージェントのアイデンティティは **ワークロード・アイデンティティ**(エージェント自身のマシン ID)と **移譲されたユーザー ID**(人間の代理としての権限)の二重構造を持つ。

* **ワークロード・アイデンティティ**: SPIFFE (Secure Production Identity Framework for Everyone) 標準規格に基づき、X.509 SVID または JWT-SVID として表現される。mTLS の基盤となる。
* **ユーザー・インパーソネーション**: OAuth 2.0 Token Exchange (RFC 8693) を活用する。

### 3.2 委譲トークンにおける権限集合の包含

ユーザー U\_i の権限集合を P(U\_i) とし、エージェント A\_j に委譲するトークン T の権限集合は、**最小権限の原則 (Principle of Least Privilege, PoLP)** に従い:

JWT の `act` (Actor) クレーム (RFC 8693 で定義) を用いて「論理的主語はユーザー U\_i、物理的実行主体はエージェント A\_j」を記録する。これにより、監査証跡(Audit Trail)においてユーザーとエージェントの両者を識別できる。

各 API が要求する権限要件 P\_{\text{required}} とトークン権限の包含関係:

P\_{\text{required}} \subseteq P(T) \implies \text{Permit},\quad P\_{\text{required}} \not\subseteq P(T) \implies \text{Deny}

これにより、権限昇格攻撃の一形態は形式的に抑止される。ただし、トークン窃取、リプレイ攻撃、SSRF 等の他の攻撃面は別途対策が必要である(第11章参照)。

### 3.3 ユーザー単位からファンクション単位へのリソースモデル転換

従来の「ユーザー ID ごとに専用エージェントを割り当てる」設計は、アイドル状態でも GPU メモリを占有する。本稿では **Function-as-a-Service (FaaS)** の概念を AI 推論に導入する:

* **ステートレスワーカープール**: エージェントは特定ユーザーに紐づく常駐プログラムではなく、機能ごとに待機するステートレスワーカーとして定義される。
* **動的ディスパッチ**: ユーザーの依頼はコンテキスト情報を持ったメッセージパケットとしてインフラ内を移動する。メッセージが特定ノードに到着した瞬間だけ最適化モデルが起動し、計算リソースを消費し、終了後に解放する。

簡略化されたコストモデル(詳細は第14章):

\text{Cost}\_{\text{total}} \approx \sum\_{f \in F} \left( \lambda\_f \cdot \tau\_f \cdot c\_f \right)

ここで \lambda\_f はファンクション f の単位時間あたり呼び出し頻度、\tau\_f は平均推論時間、c\_f は時間あたりリソース単価である。「ユーザー数 N × 巨大 LLM コスト」モデルと比較した優位性は、N \gg |F| かつ呼び出しが疎な場合に成立する(常時稼働率が高い場合は逆に不利になりうる)。

## 第4章 — ゼロトラスト・アーキテクチャの認可関数

> **記法ラベル**: NIST SP 800-207 の論理コンポーネントは規格に基づく記述。認可関数 \Phi は **概念モデル**(実装上は OPA Rego 等の述語論理として表現される)。

### 4.1 NIST SP 800-207 における論理コンポーネント

NIST SP 800-207 は、ZTA の論理コンポーネントとして以下を定義する:

* **Policy Engine (PE)**: トラスト・アルゴリズムを用いてアクセス可否を決定する判断機構。
* **Policy Administrator (PA)**: PE の決定を受けて、PEP との通信パスを確立・維持する。
* **Policy Enforcement Point (PEP)**: リソースへのアクセスを物理的に許可/拒否する執行点。
* **Policy Information Points (PIPs)**: PE が判断に用いる情報源(SIEM、CDM、脅威インテリジェンス等)。

判断機能 \Phi は概念的に以下のように表現される:

\Phi: I \times R \times C \times A \to \{0, 1\}

* I (Identity): 主語 ID 集合(エージェント ID + 移譲ユーザー ID)
* R (Resource): 対象 API/データ集合
* C (Context): 実行環境コンテキスト(IP 安全性スコア、デバイス状態、時間帯)
* A (Action): 操作種別

実装上は **Open Policy Agent (OPA)** により Policy as Code として記述され、Rego 言語の述語論理で表現される。

### 4.2 連続アクセス評価 (CAE/CAEP) の位置づけ

連続アクセス評価は、OpenID Foundation の **Shared Signals Framework (SSF)** およびその上のプロファイルである **Continuous Access Evaluation Protocol (CAEP)** として標準化されている。Microsoft Entra ID 等の実装が知られる。

NIST SP 800-207 が「継続的検証(continuous verification)」という原則を述べる一方、CAEP はそのプロトコル実装の一つに位置づけられる。両者は補完関係にあるが別系統の標準である。

CAEP により、リソースプロバイダはトークン有効期限内であっても、PE からのシグナル(セッション無効化、リスクレベル変化等)を受信してアクセスを直ちに遮断できる。

### 4.3 Bell-LaPadula モデルの情報流制御

セキュリティレベル \mathcal{L} = \{\text{Public}, \text{Internal}, \text{Confidential}, \text{Restricted}\} を半順序集合として定義する。情報流制御の基本公理:

* **No Read Up**: エージェントは自身のクリアランス以上の機密レベルを持つデータを読み取れない。
* **No Write Down**: エージェントは機密データをそれより低いセキュリティレベルの領域に書き出せない。

> **論理整合性の注記**: 第8章で論じる SLM の継続学習は、訓練データに含まれる機密情報がモデル重みに「焼き付く」可能性を持つ。Bell-LaPadula 公理を厳密に維持するには、(a) 訓練データの事前マスキング、(b) 出力検証によるメンバーシップ推論攻撃検知、(c) Differential Privacy による学習、のいずれかが必要となる。本稿の第8章 LLMOps セクションでこの対応を補足する。

エッジ SLM から中央 LLM へのコンテキスト移行には、**脱機密化関数 (Declassification Function)** の適用が必要となる。これは情報理論的に非可逆である必要はないが、攻撃者がトークン化結果から原データを復元できないことが要件である(第11章で詳述)。

### 4.4 mTLS によるトランスポート層の保護

エージェントと API 間のすべての通信は **相互 TLS (mTLS)** により暗号化される。SPIFFE によるワークロード ID が X.509 証明書の SAN フィールド(SPIFFE ID として)に埋め込まれ、エージェント間の身元保証を実現する。

これにより通信経路上での盗聴・改ざん・なりすまし(Man-in-the-Middle攻撃)を**暗号学的に困難にする**(楕円曲線離散対数問題の困難性等、暗号学的仮定に依存することに留意)。

## 第5章 — 分散型マルチエージェントアーキテクチャ

### 5.1 モバイルエージェントと計算の局所性

従来の API モデルではデータを中央 LLM に吸い上げる(Data Pull)が、分散型アーキテクチャでは処理ロジック自体がデータの存在するノードへ移動して実行される(Compute Push)。この **計算の局所性 (Locality of Computation)** により:

* ネットワーク帯域逼迫の緩和
* 情報漏洩リスクの低減(データがシステム境界を越えない)
* レイテンシの改善(往復回数の削減)

が期待される。ただし、計算の局所性は新たな課題も生む:

* 各エッジに SLM をデプロイするためのインフラ運用コスト
* 異なるエッジ間でのモデル整合性管理
* エッジ側ハードウェアの仕様統一

これらは第15章の移行戦略で論じる。

### 5.2 意図の伝播と特化型 SLM による自己解決

中央から各システムへ送られるのは具体的な API コマンドではなく、抽象的な **意図 (Intent)** のメッセージである。各システムを覆うラッパー内部には、そのシステム固有のデータスキーマと業務ロジックに特化してファインチューニングされた **小規模言語モデル (SLM)** が配置される。

SLM はメッセージを受け取ると、自律的にシステム内を操作して処理を完結し、最終的な **セマンティック・チェックポイント (Semantic Checkpoint)** を返す。これにより、中央と各システムの**疎結合 (Loose Coupling)** が達成される。

> **「疎結合」の定義の明確化**: ここでの「疎結合」とは、(a) 中央 LLM が各エッジシステムの内部 API スキーマを知る必要がない、(b) エッジシステムの内部変更が中央 LLM の再学習を要しない、という二つの条件を指す。AAPI スキーマ(第7章)による契約は依然存在し、完全な独立性ではないことに注意する。

### 5.3 確信度スコアとエスカレーション・プロトコル

SLM は出力時に自身の生成結果に対する **確信度スコア (Confidence Score)** C を算出する。

> **重要な但し書き**: LLM の出力ロジット由来の確率を「確信度」として直接用いることには既知の問題がある(キャリブレーションの欠如、過信バイアス)。実用上は、専用のキャリブレーション層(温度スケーリング、Platt スケーリング、または別途学習した不確実性推定モデル)を介して C を算出することが望ましい。

このスコアが閾値 \theta を下回る場合:

C < \theta \implies \text{Escalation}

SLM は自律処理を中断し、以下の情報を標準化された **コンテキスト・パッケージ (Context Package)** として中央 LLM へ送信する:

* Trace-ID: 分散トレーシングにおける一意のセッション識別子
* 初期の意図: ユーザーまたは上位から与えられた元のゴール
* 実行済みステップ: 成功した API 呼び出しとセマンティック・チェックポイント
* ブロック要因: エラーの内容、または確信度が低下した理由

中央 LLM は **プランの再構築と再委譲 (Re-planning and Delegation)** を行う。エスカレーション無限ループを防ぐため、最大再委譲回数 N\_{\max} と総コスト上限を設けることが実用上必須である(第13章 失敗モード参照)。

## 第6章 — 制御理論に基づく可用性保証

> **記法ラベル**: トークンバケットの微分方程式、サーキットブレーカーの状態遷移は **厳密な定義**。

### 6.1 トークンバケットによるレートリミット

エージェントが論理ループに陥り特定 API を秒間数千回叩く事態(自己 DDoS 攻撃)を防ぐため、API ゲートウェイに**トークンバケット・アルゴリズム**を実装する。バケット内トークン量 B(t) は:

\frac{dB(t)}{dt} = \begin{cases}
r & \text{if } B(t) < C \\
0 & \text{if } B(t) = C
\end{cases}

一定の補充レート r で増加し、最大容量 C でキャップされる。リクエスト到着時に B(t) \geq 1 であれば許可(消費)、不足時は HTTP 429 Too Many Requests で遮断する。

### 6.2 サーキットブレーカー(状態機械モデル)

連携先のレガシーシステムが過負荷により応答遅延を起こした場合、**サーキットブレーカー・パターン**が作動する。これは状態空間 \{\text{Closed}, \text{Open}, \text{Half-Open}\} を持つ**有限状態機械**としてモデル化される。エラー率 \epsilon が閾値 \theta\_{\epsilon} を超えると状態は Open へ遷移し、リクエストを即時エラーとして返す。一定時間経過後に Half-Open へ遷移し、少数のテストトラフィックのみを通すことでバックエンドに自己修復の猶予を与える。

> **注記**: 厳密にはサーキットブレーカーの状態遷移は時間依存とエラー履歴に依存するため、純粋なマルコフ連鎖ではなく半マルコフ過程(Semi-Markov Process)として扱うのが正確である。

### 6.3 分散トレーシングによる経路再構築

タスク開始時に一意の Trace-ID T\_{id} が生成され、エージェントの API 呼び出しの連鎖(グラフ上のパス p)を通じて伝播する:

p = (v\_1, e\_1, v\_2, e\_2, \dots, v\_n)

すべての頂点 v\_i における処理時間、エラー状態、トークン委譲情報が T\_{id} と共に時系列データベース(Tempo, Jaeger 等)に記録される。**W3C Trace Context** 仕様に基づくコンテキスト伝播により、HTTP ヘッダー(`traceparent`, `tracestate`)を通じて Trace-ID が連鎖する。OpenTelemetry はこの標準の参照実装である。

## 第7章 — セマンティック契約とエージェント・レジストリ

### 7.1 AAPI (Agent API) スキーマの定義

セマンティック契約(Semantic Contract) \mathcal{C} を以下の4要素からなるタプルとして定義する:

\mathcal{C} = \langle I, O, E, \Phi \rangle

* I (Intents): エージェントが解釈可能な「意図」の入力空間
* O (Checkpoints): 処理完了後にパブリッシュされるセマンティック・チェックポイントの出力空間
* E (Escalation): 自己解決を放棄し中央コンピュートリソースへ処理を委譲する境界条件
* \Phi (Policy): ゼロトラスト環境下で要求される権限とコンテキストの制約

> **注記**: AAPI は本稿が提案する造語であり、現時点で業界標準ではない。類似の試みとして **Agent Network Protocol (ANP)**、**Model Context Protocol (MCP)** などが存在するため、実装時には既存標準との整合を検討すべきである。

具体例(YAML 形式):

```
aapi: "1.0.0"
info:
  title: "Expense Settlement SLM"
  domain: "Finance"
  version: "2.1.0"
identity:
  spiffe_id: "spiffe://corp.local/finance/expense-slm"
intents:
  SettleTravelExpense:
    description: "ユーザーの出張費用の精算ロジックを実行する"
    required_context:
      type: "object"
      properties:
        employee_id: { type: "string" }
        receipt_semantics:
          type: "string"
          description: "OCR抽出された領収書のセマンティック表現"
checkpoints:
  SettlementCompleted:
    description: "精算完了時に発火するイベント"
    schema:
      type: "object"
      properties:
        transaction_id: { type: "string" }
        semantic_summary: { type: "string" }
escalation:
  confidence_threshold: 0.85
  max_api_retries: 3
  max_escalation_depth: 2
  fallback_routing: "spiffe://corp.local/core/dgx-orchestrator"
  declassification_rules:
    - target: "employee_id"
      method: "deterministic_tokenization"
policy:
  required_delegation_scope: "finance:expense:write"
  data_classification_limit: "Internal"
```

`max_escalation_depth` は無限再委譲を防ぐためのガードレールである。

### 7.2 分散エージェント・レジストリ

各ドメインチームが定義した AAPI スキーマは、GitOps による宣言的管理のもと、CI/CD パイプラインで型エラー、セキュリティポリシー違反を検証された後、分散 KVS (Consul/etcd) に動的登録される。

軽量ルーターはユーザーから与えられた曖昧なタスクの埋め込みベクトルに最も近い Intent をレジストリからベクトル検索(Semantic Search、例: HNSW インデックス + コサイン類似度)し、該当エージェントへメッセージをディスパッチする。

## 第8章 — LLMOps パイプライン: SLM 継続的学習の制御ループ

> **記法ラベル**: LoRA の数式は **厳密な定義**(原論文 Hu et al., 2022 に準拠)。

### 8.1 データパイプラインと LoRA による効率化

SLM の性能劣化(ドリフト)やレガシーシステムの仕様変更に伴うエスカレーション率の上昇をトリガーとして、パイプラインが起動する。ドメインチームはシステム固有の API 仕様書、エラーログ、期待される状態遷移を Git リポジトリにコミットし、パイプラインが非構造化データをインストラクション・チューニング用データセット \mathcal{D} = \{(x\_i, y\_i)\}\_{i=1}^{N} へ変換する。

学習は **LoRA (Low-Rank Adaptation)** によるパラメータ効率化ファインチューニング(PEFT)で行う。ベース汎用 SLM の重み行列 W\_0 \in \mathbb{R}^{d \times k} は固定し、低ランク行列の積として追加学習する:

W = W\_0 + \frac{\alpha}{r} BA \quad (B \in \mathbb{R}^{d \times r},\ A \in \mathbb{R}^{r \times k},\ r \ll \min(d, k))

ここで \alpha はスケーリング係数(典型値: \alpha = 2r)、r はランク(典型値: 8〜64)である [Hu et al. 2022]。

学習パラメータ数は元の d \cdot k から r(d + k) へ圧縮され、共有 GPU 上でのマルチテナント学習が現実的になる。

### 8.2 機密情報の学習取り込み防止(Bell-LaPadula 整合性)

第4.3節で論じた論理整合性の課題に対処する。学習データに含まれうる機密情報がモデル重みに焼き付くリスクを軽減するため、以下を組み合わせる:

1. **学習データの事前マスキング**: 第11.1節の決定論的トークン化を学習データにも適用。
2. **Differential Privacy (DP)**: DP-SGD によりプライバシ予算 (\varepsilon, \delta) を管理。
3. **メンバーシップ推論攻撃の検証**: 学習後の SLM に対して攻撃を試行し、訓練データ漏洩リスクを定量化。
4. **モデル分類レベルの上方化**: 学習に Restricted データを用いた SLM は Restricted 扱いとし、出力先を制限。

### 8.3 LLM-as-a-Judge による検証

学習完了した LoRA アダプターは本番デプロイ前に検証される:

| 検証次元 | 数理的基準 | 目的 |
| --- | --- | --- |
| スキーマ適合性 | \text{SLM}(x) \in O\_{\text{AAPI}} | 出力が AAPI 定義 JSON スキーマと一致 |
| 境界順守 | C < \theta \implies \text{Escalation} | 確信度低下時に正しくフォールバック |
| 情報流制御 | \text{DLP}(\text{SLM}(x)) = 0 | 機密情報の漏洩なし |
| 後退検査 | \text{Acc}\_{\text{new}} \geq \text{Acc}\_{\text{old}} - \delta | 回帰検査基準 |

> **注記**: LLM-as-a-Judge は近年いくつかの研究で信頼性のばらつきが指摘されている。自動評価の判定誤差を人間レビューで校正する体制が必要である。

### 8.4 シャドウ・デプロイメントと動的アダプター配信

検証通過した LoRA アダプター(通常数 MB〜数十 MB)はアーティファクト・レジストリに保存され、GitOps コントローラー(ArgoCD 等)により各エッジへ同期される。サービスメッシュ(Envoy 等)による **シャドウ・ルーティング** で新旧モデルの出力の発散を統計的に計測し、異常がないことを確認した上でトラフィックを切り替える。

## 第9章 — レガシーシステムのボトルネック緩和とトランザクション整合性

> **記法ラベル**: アムダールの法則は **厳密な定義**(並列計算における Speedup 上限)。CQRS と Saga パターンは **設計パターン**。

### 9.1 アムダールの法則と構造的限界

> **適用範囲の注記**: アムダールの法則は本来、並列計算における高速化上限の式であり、レガシーシステム全体への適用はアナロジーである。**より厳密な分析には待ち行列理論(M/M/1, M/M/c)** が適しているため、補完的に第14章で論じる。

直感的なアナロジーとして、システム全体の性能向上率 S は:

S = \frac{1}{(1-p) + \frac{p}{s}}

エージェント網の処理速度 s を無限大にしても、全体のパフォーマンスは \frac{1}{1-p} で頭打ちとなる。レガシーシステムの同期処理割合 1-p が大きいほど改善余地は小さい。

### 9.2 CQRS とデジタルツインによる緩和

物理的限界を緩和する3段階アプローチ:

1. **CQRS (Command Query Responsibility Segregation)**: レガシーシステムの状態空間 S\_{\text{legacy}} を、高速なインメモリデータグリッド上に読み取り専用の**射影 (Projection)** として複製し S\_{\text{fast}} を構築する。SLM の参照(Query)は S\_{\text{fast}} で完了し、レガシーシステムへの同期 Read 負荷を大幅に削減する。ただしレプリケーションラグ、整合性ウィンドウ、リカバリ時の再読み込みコストが残ることに留意する。
2. **非同期ライトビハインド**: 状態変更コマンドは Apache Kafka 等のキュー Q(t) に書き込まれ、SLM は即座に「処理受付完了」を報告する。レガシーシステムは自身のトランザクション性能に応じた処理率 \mu でキューを消化する。システム全体は**強い一貫性**を放棄し、**結果整合性 (Eventual Consistency)** を受け入れる。CAP 定理 [Brewer 2000] における可用性とパーティション耐性を優先する設計選択である。
3. **デジタルツインと無停止移行 (Strangler Fig Completion)**: SLM が対話するのは状態を模倣した**デジタルツイン**である。新しいバックエンド完成時にイベントのルーティング先を新システムへ切り替えることで、レガシーシステムを段階的にリプレースできる。

> **論理整合性の注記**: 結果整合性下で第3章の権限制御(CAE)との整合をどう保証するか。例えば、キューに残っているコマンドが、ユーザーの権限失効後にレガシーシステムに到達するケースがある。対策として、(a) コマンド実行時に権限再評価する、(b) コマンドにトークンと有効期限を同梱し期限切れは破棄、(c) 補償トランザクションで巻き戻し、を組み合わせる。

### 9.3 Saga パターンと補償トランザクション

非同期結果整合性下での論理エラー(残高不足、在庫切れ等)に対処するため、**Saga パターン** [Garcia-Molina and Salem, SIGMOD 1987] を導入する。分散エージェント環境におけるトランザクションは、複数ローカルシステムで実行される部分トランザクションの配列 T = \langle t\_1, t\_2, \dots, t\_n \rangle として定義される。

各部分トランザクション t\_i に対して **補償トランザクション (Compensating Transaction)** c\_i を定義する。これは物理的な Undo ではなく、業務ロジックとしての逆操作(例:「予約の確保」→「予約のキャンセル」)である。ステップ k で失敗が発生した場合、実行トレースは:

t\_1 \to t\_2 \to \cdots \to t\_{k-1} \to t\_k (\text{Fail}) \to c\_{k-1} \to \cdots \to c\_2 \to c\_1

補償の連鎖は **コレオグラフィ (Choreography)** として各エッジ SLM がイベント駆動で自律的に実行する。

> **失敗モードへの言及**: 補償トランザクション自体が失敗するケース(補償の補償が必要)、補償が論理的に不可能なケース(送信済みメールの取り消し等)が存在する。実用上は、補償不能な操作を Saga の最終ステップに配置する設計原則(Compensatable → Pivot → Retriable の順序付け)が知られている。

## 第10章 — 組織アーキテクチャ: 逆コンウェイ戦略

### 10.1 コンウェイの法則と分散組織

コンウェイの法則 [Conway 1968] (「システム設計は組織のコミュニケーション構造を模倣する」) を経験則として、システムの分散化は組織構造の再定義を要求する。**逆コンウェイ戦略 (Inverse Conway Maneuver)** により、組織をシステムの望ましいトポロジーに合わせて再編成する。

> **注記**: コンウェイの法則は実証的観察であり数学的定理ではない。本稿では「経験則として強く支持される設計原則」として扱う。

### 10.2 境界づけられたコンテキスト

各レガシーシステムとそのラッパー、内部の特化型 SLM は、Domain-Driven Design における **境界づけられたコンテキスト (Bounded Context)** を形成する。Team Topologies [Skelton & Pais 2019] に倣い、ドメインチーム(Stream-Aligned Team)が API ラッパーと SLM の責任を負う。

### 10.3 AI プラットフォームチームの確立

プラットフォームチームは個別業務ロジックに関与せず、共通基盤(Internal Developer Platform)を提供する:

| 提供物 | 役割 |
| --- | --- |
| Heavy Compute Pool | エスカレーション先 GPU リソースの動的プロビジョニング |
| LLMOps パイプライン | SLM の CI/CD ツール群 |
| ZTA/OPA 基盤 | セキュリティポリシー \Phi の中央管理と mTLS 証明書の自動発行 |
| Gateway (LiteLLM 等) | リクエストルーティング、レートリミット、DLP フィルタリング |

### 10.4 エージェント SRE(Site Reliability Engineering)

監視指標は CPU 使用率や HTTP エラー率に加えて、エージェントの確率的振る舞いに対する以下を導入する:

* **エスカレーション率 (Escalation Rate)**: SLM が自己解決できず中央へ委譲した割合。
* **マスキング・エラー率 (DLP Trigger Rate)**: ゲートウェイ層で機密情報漏洩がブロックされた回数。
* **状態収束時間 (Time to Convergence)**: 意図入力から最終目標状態到達までのレイテンシ。
* **補償発火率 (Compensation Rate)**: Saga における補償トランザクションの実行頻度。

## 第11章 — セキュリティ境界の多層設計

### 11.1 エッジにおけるセマンティック・マスキング

エッジ SLM が中央へエスカレーションする際、ラッパーの出口で **決定論的トークン化 (Deterministic Tokenization)** を適用する:

* **NER (Named Entity Recognition)** により機密エンティティを意味論的トークン(`<PERSON_1>`, `<CUSTOMER_ID_A>`)に置換。
* 置換写像のマッピングテーブルはエッジラッパー内部のローカルメモリにのみ保持。
* **フォーマット保持暗号 (FPE, NIST SP 800-38G)**: データ形式が論理推論に不可欠な場合に使用。

### 11.2 ゲートウェイ層での DLP 検疫

中央基盤前段のゲートウェイ層で **多層防御 (Defense in Depth)** を実施する:

\text{DLP}(X) \to \{0, 1\}

軽量パターンマッチングと統計的検出(PII 検出モデル)を組み合わせる。検知時はトランザクションを破棄し、エッジに再試行シグナルを返す。

### 11.3 コンフィデンシャル・コンピューティング (TEE)

生の機密データを中央 LLM に推論させる必要がある場合、**Trusted Execution Environment (TEE)** (Intel TDX, AMD SEV-SNP, NVIDIA Confidential Computing) を適用する:

1. 計算ノード上にエンクレーブを動的構築。
2. エッジはエンクレーブのアテステーション検証後、公開鍵で機密データを暗号化し送信。
3. エンクレーブ内部でのみ復号され推論処理が実行される。
4. 推論結果のみが暗号化されエッジへ返却される。

> **限界の明示**: TEE はサイドチャネル攻撃(Spectre 系、Foreshadow 等)への耐性が必ずしも完璧でなく、ベンダー固有の脆弱性も過去に報告されている。「現行の TEE ベンダー保証の範囲で困難」と表現するのが適切である。

## 第12章 — 設計原則のまとめ

本稿で提示したアーキテクチャの根幹は以下の通りである:

1. **インターフェースの構造化抽象化**: 既存システムをラッパー(API/CLI)を通じて構造化空間へ射影する。
2. **アイデンティティと権限の集合論的制御**: OAuth 2.0 Token Exchange (RFC 8693) と SPIFFE による二重 ID。ユーザー単位からファンクション単位へのリソースモデル転換。
3. **ゼロトラストの NIST SP 800-207 準拠**: PE/PA/PEP の三層構造に基づく認可と、CAEP による継続的検証。
4. **分散型モバイルエージェント**: 計算の局所性、疎結合、スケーラブルなリソースモデル。
5. **セマンティック契約による動的連携**: AAPI スキーマとエージェント・レジストリ。
6. **制御理論に基づく可用性保証**: トークンバケット、サーキットブレーカー、分散トレーシング。
7. **レガシーシステムの段階的克服**: CQRS、非同期ライトビハインド、デジタルツイン、Saga パターン。
8. **逆コンウェイ戦略**: システムトポロジーと組織トポロジーの整合。

ここから先の第13章〜第17章は、上記設計原則を実装・運用する上で必要となる補完的論点を扱う。

## 第13章 — 失敗モード分析 (Failure Mode Analysis)

複雑な分散システムでは、設計時に想定すべき失敗モードを網羅的に列挙し、それぞれへの対処を予め設計に組み込む必要がある。

### 13.1 SLM 起因の失敗

| 失敗モード | 症状 | 対処 |
| --- | --- | --- |
| 幻覚 (Hallucination) | 存在しない API を呼び出そうとする | スキーマバリデーション、ツールホワイトリスト |
| 過信 | 確信度が高いが実際は誤り | キャリブレーション層、サンプリングベース不確実性推定 |
| ドリフト | 時間経過で精度低下 | エスカレーション率の監視、定期再学習 |
| プロンプトインジェクション | 観測データに混入した指示で挙動が乗っ取られる | コンテキストとデータの分離、Spotlighting 技法 |

### 13.2 オーケストレーション起因の失敗

| 失敗モード | 症状 | 対処 |
| --- | --- | --- |
| エスカレーション無限ループ | A→中央→A→中央… | `max_escalation_depth` の強制 |
| サーキットブレーカー伝播失敗 | 一部のエッジで Open 状態が伝播せず過負荷継続 | 全体への状態ブロードキャスト |
| 補償トランザクション失敗 | Saga の Undo に失敗 | 補償の冪等化、ヒューマンインザループ |
| イベント二重発行 | Kafka リバランス時等 | Exactly-once セマンティクスまたは Idempotency-Key |

### 13.3 セキュリティ起因の失敗

| 失敗モード | 症状 | 対処 |
| --- | --- | --- |
| マスキング漏れ | DLP がエッジで機能せず原データが中央に到達 | ゲートウェイでの二重チェック、トランザクション破棄 |
| トークン窃取 | OAuth トークンが漏洩 | 短寿命トークン、CAEP による即時失効 |
| サイドチャネル | TEE のサイドチャネル攻撃 | ベンダー脆弱性監視、Defense in Depth |

### 13.4 カオスエンジニアリングの導入

上記失敗モードを定常的に検証するため、Netflix Chaos Monkey の流儀に倣い、**エージェント版カオスエンジニアリング**を実施する。例: ランダムに SLM を「幻覚状態」にする、特定エッジを停止する、トークンを意図的に失効させる等。

## 第14章 — コストモデルと定量分析

### 14.1 詳細コストモデル

第3.3節の簡略式を拡張し、より現実的なコスト構造を以下に示す。

総コスト \text{Cost}\_{\text{total}} は固定コストと変動コストの和:

\text{Cost}\_{\text{total}} = \text{Cost}\_{\text{fixed}} + \text{Cost}\_{\text{variable}}

**固定コスト**(エッジ常駐リソース):

\text{Cost}\_{\text{fixed}} = \sum\_{e \in E\_{\text{edge}}} \left( c\_{e}^{\text{GPU}} \cdot T + c\_{e}^{\text{storage}} \cdot V\_e \right)

**変動コスト**(中央エスカレーション):

\text{Cost}\_{\text{variable}} = \sum\_{f \in F} \lambda\_f \cdot \rho\_f \cdot \kappa\_f

ここで \rho\_f はファンクション f のエスカレーション率(エッジで自己解決できない割合)、\kappa\_f は1回あたり中央 LLM 呼び出しコスト(トークン数 × 単価)である。

### 14.2 比較分析: 中央集権型 vs 分散型

簡単のため、ユーザー数 N、ユーザーあたり1秒に1リクエスト、リクエストあたり平均トークン数 \tau を仮定する。

**中央集権型** (常時稼働の巨大 LLM、ユーザー専有 GPU):

\text{Cost}\_{\text{central}} \approx N \cdot c\_{\text{GPU}} \cdot T

**分散型** (エッジ SLM + 中央エスカレーション):

\text{Cost}\_{\text{distributed}} \approx \underbrace{|E\_{\text{edge}}| \cdot c\_{\text{edge\\_GPU}} \cdot T}\_{\text{固定}} + \underbrace{N \cdot \rho \cdot \tau \cdot c\_{\text{LLM}}}\_{\text{変動}}

エスカレーション率 \rho が十分小さく、エッジ数 |E\_{\text{edge}}| \ll N であれば分散型が有利。逆に \rho \to 1 (ほぼすべて中央処理になる) のケースでは分散型のメリットは消失する。

### 14.3 待ち行列理論による応答時間モデル

レガシーシステムを待ち行列 M/M/1 でモデル化する。到着率 \lambda、サービス率 \mu (\rho = \lambda/\mu < 1) のとき、平均応答時間:

W = \frac{1}{\mu - \lambda}

\lambda \to \mu で応答時間が発散する。CQRS による読み取り負荷オフロード(第9.2節)は、レガシー側の到着率 \lambda を \lambda \cdot (1 - q) に削減する効果がある(q は読み取り比率)。

### 14.4 損益分岐点の試算

仮想シナリオで概算する: 中堅企業(N = 5000 名)、エッジドメイン数 |E\_{\text{edge}}| = 30、エスカレーション率 \rho = 0.1、エッジ GPU 月額 $$1000$、中央 LLM 呼び出し $$0.01/\text{req}$。

* 中央集権型(専有 GPU を仮定した極端なケース): $5000 \times $1000/\text{月} = $5{,}000{,}000$
* 分散型: $30 \times $1000 + 5000 \times 86400 \times 30 \times 0.1 \times $0.01 \approx $1.3\text{B}$

> **注意**: この試算は誇張された仮定の下のものである。現実の中央集権設計は専有 GPU ではなく共有プールを用いるため、実コストは大幅に小さい。**つまり「分散型が常に安い」とは限らない**。エスカレーション率と呼び出し頻度の組み合わせ次第である。実用時には自社のアクセスパターンを計測してから判断すべきである。

## 第15章 — 段階的移行戦略

### 15.1 ブラウンフィールド導入のフェーズ分け

既存企業が本アーキテクチャに移行する際の典型的なロードマップ:

**Phase 0: 評価 (1〜3ヶ月)**

* 既存システムの API/CLI カバレッジ調査
* ZTA 成熟度評価(NIST CSF 等のフレームワーク利用)
* パイロット対象ドメインの選定(典型的には、独立性が高くデータ機密度が中程度の業務)

**Phase 1: パイロット (3〜6ヶ月)**

* 1〜2ドメインで AAPI ラッパー + SLM 配置
* 中央 LLM オーケストレーターは既存のまま
* 主要メトリクス(エスカレーション率、レイテンシ、コスト)のベースライン確立

**Phase 2: 拡大 (6〜18ヶ月)**

* 5〜10ドメインへ展開
* LLMOps パイプラインの確立
* 逆コンウェイによる組織再編成の開始

**Phase 3: 成熟 (18ヶ月〜)**

* 全ドメインへの展開
* CQRS/Saga による真の分散トランザクション
* レガシー Strangler Fig 完了

### 15.2 段階導入のためのファサード戦略

すべてを一度に置き換えるのではなく、**ファサードパターン**で既存システムを暫定的にカプセル化する:

```
[Agent] → [AAPI Facade] → [既存REST/SOAP API] → [Legacy System]
```

ファサード層は最初は単純なプロキシだが、徐々に SLM・キャッシュ・補償ロジックを取り込んでいく。

### 15.3 組織変革のリスク

* **スキルギャップ**: SRE が「エージェントの確率的挙動」を監視するスキルを獲得する必要がある
* **責任分界**: ドメインチームと AI プラットフォームチームの境界が曖昧化しやすい
* **既存ベンダー依存**: パッケージ製品はラッパー化が困難なケースがある(契約上 API 公開が制限される等)

## 第16章 — 規制対応とコンプライアンス

### 16.1 GDPR・個人情報保護法対応

* **データ最小化原則 (Data Minimization)**: 第11.1節のセマンティック・マスキングは GDPR Article 5(1)(c) の Data Minimization 原則と整合する。
* **忘れられる権利 (Right to Erasure)**: SLM が個人データを学習している場合、削除要求への対応が技術的に困難。対策として、(a) 学習データの台帳管理、(b) 個別 LoRA アダプターの破棄、(c) Machine Unlearning 技法の活用が必要。
* **自動化された意思決定 (Article 22)**: 重要な意思決定への AI 適用には人間の関与が必須(Human-in-the-Loop)。

### 16.2 業界別規制

| 業界 | 主要規制 | アーキテクチャへの影響 |
| --- | --- | --- |
| 金融 | SOX, バーゼル III, FFIEC | 監査証跡の完全性、エージェント行動のリトレース能力 |
| 医療 | HIPAA (米), 医療情報システムガイドライン (日) | TEE 必須、PHI のクロスボーダー禁止 |
| 公共 | FedRAMP, IRAP | クラウド配置制限、TEE/HSM 要件 |
| EU AI Act | リスクベース分類 | High-Risk AI System の登録、CE マーク |

### 16.3 監査証跡の設計

すべてのエージェント行動は Trace-ID と共に**改ざん困難なログストア**(Append-only ログ、または分散台帳)に記録する。最低限以下を記録:

* 実行された Intent と入力データのハッシュ
* 使用したトークンの `act` クレーム
* SLM のバージョン(LoRA アダプターのハッシュ)
* 確信度スコアとエスカレーションの有無
* 補償トランザクションの実行履歴

## 第17章 — 評価指標、既存実装との比較、限界

### 17.1 評価指標 (KPI) の体系

提案アーキテクチャの効果を測定するために以下を計測する:

**機能性指標**

* タスク成功率 (Task Success Rate)
* エスカレーション率 (Escalation Rate)
* 平均状態収束時間 (Time to Convergence)

**信頼性指標**

* MTTR (Mean Time To Recover)
* 補償発火率 (Compensation Rate)
* DLP トリガ率

**経済性指標**

* リクエストあたりコスト (/req)
* エッジ稼働率

### 17.2 既存実装との比較

| フレームワーク | アプローチ | 本稿との関係 |
| --- | --- | --- |
| LangGraph | グラフベースの状態管理 | 第2章のグラフモデルと近接。ただし中央集権型寄り |
| Microsoft AutoGen | マルチエージェント対話 | 第5章の分散性と整合するが、ZTA は外部実装 |
| CrewAI | 役割ベース協調 | ドメインチームのメタファに近い |
| Model Context Protocol (MCP) | LLM ↔ ツール接続標準 | 第7章 AAPI と類似目的 |
| Agent Network Protocol (ANP) | エージェント間通信標準 | 第7章 AAPI と相補的 |
| Service Mesh (Istio/Linkerd) | mTLS と可観測性 | 第4.4節、第6.3節と整合 |

本稿の独自性は、これら個別技術を**統合した設計原則体系**として提示する点にある。個々の要素技術は既存である。

### 17.3 本アーキテクチャが機能しないケース(限界)

正直な評価として、以下のケースでは本提案が必ずしも最適ではない:

1. **小規模組織** (N < 100 程度): 複数の SLM・LLMOps パイプライン・ZTA 基盤の固定コストが、分散化の便益を上回る。中央集権型のシンプル構成のほうが合理的。
2. **ハードリアルタイム要件** (応答 < 数ミリ秒): SLM 推論時間自体がボトルネックとなる。LLM 抜きのルールベースエンジンを推奨。
3. **均質なデータ・業務ロジック**: ドメイン特化の意味が薄い場合、SLM を分割するメリットがない。
4. **強い一貫性必須の領域** (高頻度取引、原子炉制御等): 結果整合性が許容されない領域では、本提案の前提が崩れる。
5. **AI ガバナンス未成熟な組織**: 統制体制なしに自律エージェントを多数展開することは、リスクの増幅器となる。

### 17.4 反証可能性 (Falsifiability)

本提案が実証的に「失敗した」と判定される条件を明示する:

* パイロット運用 6ヶ月後にエスカレーション率が 50% を超える → 分散の意味が薄い
* リクエストあたりコストが中央集権型を継続的に上回る → 経済性の前提が崩れる
* 重大インシデント(機密漏洩、不正アクセス)が中央集権型より多い → セキュリティ前提が崩れる
* ドメインチームが AAPI 保守に追われ業務開発が停滞 → 組織コストが想定を超える

これらが観測された場合、設計の根本的見直しが必要となる。

## 第18章 — 結論

AIエージェント時代におけるエンタープライズシステムアーキテクチャの刷新は、単なるコードの書き換えや API の追加ではなく、人間と GUI の非定型な相互作用から成る環境から、**API と CLI という構造化インターフェースによって結合された分散型マルチエージェントシステムのトポロジー** への移行を意味する。

本稿で提示した設計原則は、(1) インターフェースの構造化抽象化、(2) アイデンティティと権限の集合論的制御、(3) NIST SP 800-207 準拠のゼロトラスト、(4) 分散型モバイルエージェント、(5) セマンティック契約による動的連携、(6) 制御理論に基づく可用性保証、(7) レガシーシステムの段階的克服、(8) 逆コンウェイ戦略による組織設計、の8つに集約される。

これらは個別の要素技術としては既存のものだが、本稿の貢献はそれらを**統合された設計原則体系**として提示し、各章で適用条件・限界・整合性課題を明示した点にある。「既存システムの API/CLI 化」「ID ベースの認証」「ゼロトラスト」「分散型自律エージェント」という基本方針は、エンタープライズアーキテクチャにおける有力な選択肢の一つであるが、すべての環境に適合するわけではない。第17章で示した適用限界と反証可能性に従い、自社環境における計測と検証を通じて採否を判断することが推奨される。

今後の検証方向として、(a) 実装プロトタイプによるベンチマーク、(b) 具体的企業のケーススタディ、(c) 反証データに基づく設計修正が考えられる。
