## はじめに

「AIエージェント」という言葉が、雑に使われすぎている。

OpenClaw のスキル をcron で定期実行して、Slack に結果を流す。これを「AIエージェント」と呼んでいる人をよく見かける。正直に言えば、私も違和感ながら最初はそんなもんなのかと思っていた。

でも、自宅にMac Mini 2台を置いて、AIエージェントを数ヶ月間運用してきた結果、見えてきたものがある。それは **スクリプト+cronで十分なものと、本当にエージェントが必要なものは、根本的に別物だ** ということだ。

この記事では、私が実際に構築・運用してきたマルチエージェント環境の話を通じて、その「本質的な違い」を整理する。

---

## 「エージェントが仕事をした」は本当か？

最近よく見かける事例はこういうものだ。

> AI コーディングツールで業務自動化スキルを作った。OpenClaw のスキルで定期的に実行している。なので「AI エージェントが業務を回している」。

これ自体は素晴らしい成果だと思うし、広義では正しいのかもしれない。だが、実際は **AI が仕事をしたのは「スクリプト（スキル）を書いたとき」だけ** で、毎日動いているのは普通のスクリプトだ。cron が起動し、API を叩き、結果を返す。AI の判断は一切介在していない。

つまり構造的にはこうなっている。

```
[ 人間 ] → [ AI ツール ] → [ スクリプト（スキル）生成 ]
                                 ↓
                           [ cron + スクリプト ]  ← AI は不在
                                 ↓
                           [ 結果を Slack に投稿 ]
```

「AI がスクリプトを作った」と「AI がエージェントとして動いている」は別の話だ。前者は **AI支援の自動化** であり、後者は **エージェント** だ。  
（もちろん、AI がスクリプトを作ることはすばらしいことで、そのおかげて実現できていることも多々ある）

では、私が言う「エージェント」とは何なのか。

---

## まずやったこと——Mac Miniにエージェントを住まわせた

私が最初にやったのは、Mac Mini にAI エージェントを常駐させることだった。役割は **情報収集** ——X から最新の AI トレンド（サービス・ツール・実装ノウハウの一次情報）を 3 時間ごとに収集し、要約付きで Slack に投稿する。収集結果にはトピックごとのサマリーとソース URL が付いていて、人間がざっと目を通すだけでAI界隈の動向がわかる。

「それ、cron で十分じゃないか？」

そう思うだろう。そう、実際この段階だけなら **cron で十分** だ。

違いが見え始めたのは、運用を続けてからだった。

---

## 壁——「何が悪いのか」がわからない

数週間運用して気づいた。 **収集の精度が安定しない** 。

ある日は的確なトレンドを拾ってくるが、別の日は関連性の薄い投稿ばかりになる。しかも最初は、1 体のエージェントに「収集 → 投稿文生成」を両方やらせていた。

問題は、 **改善しようがない** ことだった。

投稿の反応が悪かったとき、原因が「そもそも収集した元ネタが的外れだった」のか「ネタは良かったが表現が悪かった」のか、切り分けられない。1 体のエージェントに全部やらせていると、フィードバックの対象が曖昧になる。

```
[ エージェント 1 体で全部やる ]

  収集 → 生成 → 投稿
                  ↓
            反応が悪い
                  ↓
        何が悪い？ → わからない
         ├── 収集が悪い？
         ├── 生成が悪い？
         └── 両方？
```

スクリプト + cron なら「当たり前」であり、「仕方がない」ことだ。なぜなら、スクリプトは「改善」しないからだ。書いた通りに動くだけで、精度を上げるという概念がそもそもない。

**この瞬間に、「自動化」と「エージェント」の分水嶺が見えた。**

エージェントには「経験を蓄積し、次に活かす」仕組みが必要だ。そして、改善を回すには、**何を改善すべきかが明確に切り分けられる構造** が不可欠だった。

---

## 分けた——専門エージェントという設計判断

そこで 2台 目の Mac Mini を用意し、投稿生成専用のエージェントを立ち上げた。

```
Mac Mini #1: 収集エージェント
  └── 3時間ごとにXから AI トレンドを収集

Mac Mini #2: 生成エージェント
  └── 収集結果を受け取り、要約を生成し投稿
```

こうすることで、改善の方向性が明確になる。

* 収集の精度が悪い → **収集エージェントのスキルを改善する**
* 投稿の表現が悪い → **生成エージェントのスキルを改善する**

![構成図](https://static.zenn.studio/user-upload/deployed-images/2b1aa617031a1d6c70f4224c.png?sha=b4089a2b72698ce30001cb92cd3ace3be6302eca)  
*↑ 現在の構成。2台の Mac Mini にそれぞれ Specialized Agent が住んでいて、Slack で繋がっている。*

これは単なるマイクロサービス的な分離ではない。 **エージェントの「学習」を機能させるために必要な分離** だ。

ここで少しアーキテクチャの話に踏み込む。

---

## なぜ「分ける」と学習が機能するのか

将来的に、各エージェントの経験を GraphRAG（グラフ構造のナレッジベース）に蓄積する設計を進めている。

GraphRAG は 3 種類のノードで構成される。

![GraphRAGノード設計](https://static.zenn.studio/user-upload/deployed-images/b073cbbbe723d2914a8fe7a1.png?sha=6f924262e016e0a36b2bc0c863668c18401f2c9f)

* **Action Node** —— エージェントが何をしたか（パラメータ含む）
* **Result Node** —— その結果どうなったか（定量メトリクス）
* **Insight Node** —— ActionとResultから導出された汎化知識

たとえば「技術解説系は画像付きだとエンゲージメント1.5倍」という Insight Node が生まれる。confidence スコアが付いていて、サンプルが増えるほど精度が上がる。

ここで、 **もし収集エージェントと生成エージェントが1体だったら何が起きるか** を考えてほしい。

収集と生成の両方をやっているエージェントの GraphRAG には、収集に関するInsightと生成に関する Insight が混在する。「エンゲージメントが低かった」という Result に対して、どちらの Action が原因なのかグラフ上で辿れなくなる。**Insight Node が汚染される** のだ。

分けていれば、収集エージェントの Action → Result の因果は収集精度だけに閉じるし、生成エージェントのそれは生成品質だけに閉じる。改善の方向がブレない。

**汎用エージェント1体より、専門エージェントの組み合わせの方が、フィードバックループの精度が高い。**

これが「分ける」理由だ。

## インフラ——1コンテナ = 1エージェント

実運用にはインフラの話も避けられない。

Mac Mini のスペックで1エージェント1台は正直オーバースペックだ。そこで Docker Compose を使い、**1台の Mac Mini 上で複数エージェントをコンテナとして隔離する構成** にした。現在は1台の Mac Mini で5体のエージェントが同時稼働している。

### 1コンテナ = 1エージェント

全エージェントが同じベースイメージ（Node.js 24-slim）から起動する。エージェントごとの差異——人格、スキル、スケジュール——はすべてボリュームマウントで注入する。

```
ホスト側                              → コンテナ内
agents/<name>/config/IDENTITY.md      → ~/.openclaw/workspace/IDENTITY.md
agents/<name>/config/SOUL.md          → ~/.openclaw/workspace/SOUL.md
rules/                                → ~/.openclaw/workspace/rules/
openclaw-skills/                      → ~/.openclaw/skills-repo/
```

人格定義（IDENTITY.md）と行動方針（SOUL.md）がエージェントの「人格」を決め、共通ルールが全エージェント共通の行動規範を与える。コンテナの entrypoint がスキルのシンボリックリンクや設定ファイルのリンクを自動で行う。

### スキルの独立設計

エージェントの「能力」は **スキル** という単位で独立させている。スキルは TypeScript で実装し、1ディレクトリ = 1スキルとして管理している。

```
skills/<skill-name>/
├── SKILL.md              # スキル定義（OpenClawが認識）
├── wrapper/index.ts      # 処理本体
└── package.json
```

スキルはエージェント本体とは別リポジトリで管理されており、ボリュームマウントで全エージェントに共有される。どのエージェントがどのスキルを使うかは、エージェント側の設定と人格定義が決める。

**エージェントは人格、スキルは職能。** 同じ「検索」スキルでも、収集エージェントと生成エージェントでは使い方が違う。人格（判断基準・優先順位）がエージェントに紐づき、職能（具体的なAPI呼び出し）がスキルに紐づく。

### セキュリティ

AI エージェントに好き放題させるのは怖い。コンテナのセキュリティは多層で考えている。

* **コンテナ隔離**: 1コンテナ = 1エージェントで、ファイルシステム・プロセスを完全分離
* **非rootユーザー**: コンテナ内は `agent` ユーザーで実行。権限昇格の経路を塞ぐ
* **設定の書き込み保護**: 共通ルールや人格定義は `:ro`（read-only）でマウント。エージェント自身が書き換えられない
* **認証情報の分離**: APIキー・トークンはGit管理外で、コンテナごとに独立した環境変数として注入

さらにセキュリティ強化として、`read_only` ファイルシステム、`seccomp` プロファイル、egress プロキシによる外部通信の deny-by-default 化をロードマップに載せている。

また、Claude Code をコンテナで動かすのは一筋縄ではいかなかった。root だと `--dangerously-skip-permissions` が使えない、非rootユーザーの作成と認証ファイルのコピーが必要、TUI が非対話環境で動かない……地味だが実運用では避けて通れないハマりどころが多い。

（このあたりの詳細は別記事で書く予定だ。）

## 見えてきたもの——Self-Evolving Agentic AI

数ヶ月の運用を通じて、全体像が見えてきた。私はこれを **Self-Evolving Agentic AI** と呼んでいる。

![6層アーキテクチャ](https://static.zenn.studio/user-upload/deployed-images/6781db4a996c385efc0d5f2d.png?sha=43f0eb8d94fab678dbc95aafb9bdd1d61aa6eb8c)

6つの層で構成される。

| 層 | 名称 | 役割 |
| --- | --- | --- |
| ① | Human-in-the-Loop | 人間による目標設定・承認 |
| ② | Orchestration Layer | タスク分解・結果評価・GraphRAG参照 |
| ③ | GraphRAG Knowledge Layer | 経験の蓄積（エピソード記憶＋意味記憶） |
| ④ | Task Queue Layer | タスクの依存管理・優先度制御 |
| ⑤ | Autonomous Skill Agents | 専門エージェント群（収集, 生成, etc.） |
| ⑥ | Tool Layer | 外部API・スキル実装 |

核心は **右側のフィードバックループ** にある。

```
エージェントが投稿を生成
  → Xに投稿
  → 反応データを収集
  → オーケストレーターが結果を評価
  → GraphRAGに知見を蓄積
  → 次のタスク生成時にInsightを参照
  → ループ継続・自動改善
```

このループが回ることで、 **運用するほど賢くなる** 。これがスクリプト + cron との本質的な違いだ。

スクリプトは書いた瞬間が最高品質で、あとは劣化するだけ。エージェントは運用するほど精度が上がっていく。 **時間が競争優位になる。**

### Self-Reflection Loop

エージェントの出力をそのまま使うわけではない。Self-Reflection のループがある。

![Self-Reflection Loop](https://static.zenn.studio/user-upload/deployed-images/b01efd3526ae9d1b886feec1.png?sha=5fd04c8e8273ef7bfd1539339de1329ae5ac7bb0)

エージェントの出力に対して LLM が Critique をかける。relevance、accuracy、engagement の多次元でスコアリングし、approve（そのまま使う）、revise（修正して再実行）、reject（破棄）の3分岐を行う。

ポイントは、この Critique の **reasoning（なぜそう判断したか）もGraphRAGに蓄積される** こと。「なぜリジェクトしたか」が次のタスク生成に反映されるから、同じミスを繰り返さなくなる。

## まだ途中——解けていない問題

正直に書くと、このシステムは **完成していない** 。特に以下は未解決だ。

### GraphRAGの階層設計

GraphRAG の知識を「誰のもの」にするか、という問題がある。ノウハウは共有すべきだがエージェントごとに思考は異なるべきだ。

* **エージェント固有の経験知識** : 「自分はこう判断して、こう失敗した」。人格に紐づくので共有すべきでない
* **スキル共有知識** : 「このAPIはこう叩くと効率がいい」。同じスキルを持つエージェントなら共有できる
* **タスク方向性で分岐する知識** : 同じ検索スキルでも、「AIトレンド収集」と「競合情報収集」では蓄積すべき知識の方向性が違う

この 3 層をどう設計するかは、実際にデータが蓄積されてから判断する方針だ。抽象論で決めても的外れになる。

### オーケストレーター層の実装

現在は cron で各エージェントを独立に駆動しているが、本来はオーケストレーターがタスクを動的に分配する設計が理想だ。これについては別プロジェクトで実験を進めており、結果が出たら改めて記事にする予定だ。

## まとめ——スクリプト+cronとの本質的な違い

数ヶ月間の運用を通じて見えたことを一言でまとめると、こうなる。

> **スクリプト + cron は「実行」を自動化する。エージェントは「判断」を進化させる。**

前者は書いた瞬間が最高品質で、環境が変われば手動で書き直す。後者は経験を蓄積し、自動で精度を上げていく。

巷で「AI エージェント」と呼ばれているもののうち、本当にエージェントと呼べるのは後者だけだ。そして後者を機能させるには、専門エージェントの分離、GraphRAG による経験蓄積、オーケストレーションによるタスク管理という設計が必要になる。

この記事で書いた内容は、あくまで私が今進行中のプロジェクトの途中経過だ。GraphRAG の階層設計、オーケストレーター層の実装、新しいエージェントの追加など、まだやることは多い。進捗は X で発信していくので、興味がある方はフォローしてもらえると嬉しい。
