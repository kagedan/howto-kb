# LLMチェーン設計におけるコンテキスト窓の最適化戦略

長時間にわたるAIタスク実行を設計する際、多くのエンジニアが直面する課題があります。

「データベースから情報取得→分析→提案生成→外部API呼び出し」といった複数ステップをLLM連鎖させると、途中で**コンテキスト窓が満杯になり、以降の推論が破綻する**現象です。

## 事例：営業提案生成の連鎖

仮に営業支援アプリで以下を実行するとしましょう。

1. 顧客情報（名前・業種・過去購買・競合情報）を DB から取得
2. LLM が顧客分析（課題特定・ニーズ仮説）を生成
3. LLM が提案書ドラフトを作成
4. LLM が LinkedIn/メール 用の個別文案を生成
5. 最終審査（一貫性チェック）

ナイーブに実装すると：

```
# ❌ 危険なパターン
context = ""
for step in [get_customer_info, analyze, draft_proposal, 
             generate_linkedin, generate_email, review]:
    result = llm.chat(context + prompt_for_step)
    context += result  # コンテキストが積み重なる
```

結果：

* Step 1 の顧客情報（8KB）→ Step 2 で分析結果（5KB）追加→ Step 3 で提案（20KB）→ Step 4 で投稿文（3KB）→ Step 5 に達する前に合計コンテキスト **30KB+ に膨張**
* Claude（200K）では余裕でも、より制約のあるモデルでは Step 3 ～ 4 で品質低下
* 履歴が長いほど「同じ情報の繰り返し」「矛盾した判断」が増加

## 対策1：ステップごとに「結果だけ」を次へ渡す

```
# ✅ パターンA：段階的な要約

def step_1_customer_analysis(customer_id):
    info = db.get_customer(customer_id)
    analysis = llm.chat(
        f"顧客情報: {info}\n課題を3つまで、20字以内で要約"
    )
    return {"summary": analysis}  # 短い結果だけ保持

def step_2_proposal_draft(analysis_summary):
    # 前ステップの「要約結果」だけを入力
    proposal = llm.chat(
        f"課題: {analysis_summary}\nに対する提案書を作成"
    )
    return {"proposal": proposal}
```

**効果**：

* Step 1 の結果（20字要約）を Step 2 に渡す ≒ 元のデータ削減 80%
* 各ステップのコンテキスト圧を均等分散

## 対策2：並列処理で「直線チェーン」を避ける

```
# ✅ パターンB：並列化で不要な依存を削減

import asyncio

async def generate_content(analysis_summary):
    # Step 3, 4, 5 を同時実行
    linkedin, email, review = await asyncio.gather(
        llm.chat(f"LinkedIn投稿を作成: {analysis_summary}"),
        llm.chat(f"メール本文を作成: {analysis_summary}"),
        llm.chat(f"内容の一貫性を検証: {analysis_summary}")
    )
    return {"linkedin": linkedin, "email": email, "review": review}
```

**効果**：

* 3 タスク × 同じ入力 = コンテキスト 1 回分で済む
* 直線チェーンより 50% コンテキスト削減

## 対策3：長期ステップでは「ファイル保存」を活用

```
# ✅ パターンC：中間ファイルで「状態管理」

def long_running_task(customer_id):
    # Step 1
    analysis = llm.chat(get_customer_prompt(customer_id))
    save_to_file(f"state_{customer_id}_analysis.json", analysis)
    
    # Step 2（ファイルから読む = LLMに全履歴を渡さない）
    analysis_loaded = load_from_file(f"state_{customer_id}_analysis.json")
    proposal = llm.chat(
        f"以前の分析: {analysis_loaded}\n...（新規入力のみ）"
    )
    save_to_file(f"state_{customer_id}_proposal.json", proposal)
```

**効果**：

* 過去ステップのフル履歴をコンテキストに乗せない
* 最新 Step のインプット（分析結果）だけに絞れる
* ディスク I/O の遅延は許容（LLM コスト削減が優先）

## 実装の落とし穴

### ① 「要約」の品質低下

**問題**: 細かい情報を 20 字に圧縮すると、次ステップで不足情報が出る

**対策**: 圧縮 + キー情報の保持

```
summary = f"""
課題TOP3:
- {top3_issues[0]}
- {top3_issues[1]}
- {top3_issues[2]}

重要タグ: {critical_keywords}
"""
# 20 字制約ではなく「構造化+最小化」に変更
```

### ② 並列実行時の矛盾

**問題**: LinkedIn と Email を同時に作らせると、トーンが違う可能性

**対策**: プロンプトに「トーン統一」指示を追加

```
base_prompt = f"""
顧客背景: {analysis}
全コンテンツのトーン: 親しみやすいが専門的
"""

await asyncio.gather(
    llm.chat(base_prompt + "LinkedIn版:"),
    llm.chat(base_prompt + "メール版:")
)
```

## まとめ：LLMチェーン設計の 3 原則

1. **要約を徹底**：各ステップの結果は短く、次へは要約版を渡す
2. **並列できる部分は並列化**：直線依存を減らす
3. **中間結果は保存**：超長期タスク（複数時間）ではファイル化して状態を外部保持

コンテキスト窓は有限な資源です。チェーン設計時には、必ずこの 3 原則を念頭に置きましょう。
