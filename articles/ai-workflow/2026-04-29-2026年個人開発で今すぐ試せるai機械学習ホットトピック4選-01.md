## はじめに

2026年のAI界隈は、研究フェーズから **個人が実際に動かせる実装フェーズ** へと急速に移行しています。

APIとオープンソースフレームワークが充実したことで、数年前なら企業の研究チームにしか触れられなかったような技術を、個人開発者が数時間で動かせる時代になりました。

本記事では、今まさに盛り上がっていて、かつ **個人開発でも実装しやすい** 4つのトピックを取り上げます。

---

## 1. MCP（Model Context Protocol）

### 何が熱いのか

Anthropicが2024年11月に策定したオープンプロトコルで、**LLMとあらゆる外部ツール・データソースをつなぐ標準規格**です。「AIのためのUSB-C」と称されることが多く、2026年現在はOpenAI・Google・Microsoftも採用し、実質的な業界標準になっています。

従来は「ChatGPT連携」「Claude連携」それぞれに個別のAPI実装が必要でしたが、MCPに対応したサーバーを一度作れば、あらゆるMCP対応クライアント（Claude Desktop、Cursor、Zedなど）からそのまま呼び出せます。

2026年2月時点でコミュニティ製のMCPサーバーは8,600を超えており、GitHub・Slack・PostgreSQL・Figmaなど主要ツールはほぼ対応済みです。

### 個人開発でできること

自分専用のMCPサーバーを作ると、LLMが「自分のローカルファイル」「自社のDB」「特定のAPI」に直接アクセスできるようになります。

具体例としては、

* 自分のObsidianノートをLLMから検索できるサーバー
* NotionやGitHubと連携した個人用タスク管理エージェント
* 社内ドキュメントに問い合わせるQAボット

などが、Pythonで数十行から実装できます。

### 始め方

```
# FastMCPを使った最小実装
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

@mcp.tool()
def search_notes(query: str) -> str:
    """自分のノートを検索する"""
    # 実装を書く
    return f"{query} の検索結果..."

if __name__ == "__main__":
    mcp.run()
```

2026年の推奨実装は **FastMCP** です。Flaskのようなデコレータ記法で、ボイラープレートをほぼ書かずに済みます。

## 2. RAG（Retrieval-Augmented Generation）の進化形

### 何が熱いのか

RAGは「LLMが知らない情報を外部DBから検索して回答に使う」技術ですが、2026年は単純なベクトル検索からさらに進化しています。

特に注目されているのが **Agentic RAG** と **GraphRAG** の2つです。

**Agentic RAG** は、AIエージェントが検索結果を評価して「情報が不十分なら再検索する」という自律的な反復ループを持ちます。従来の「1回検索 → 1回生成」より格段に精度が上がります。

**GraphRAG** は、ドキュメントをエンティティ（人名・概念など）の関係グラフとして構造化し、単純なベクトル検索では見落とす「間接的な関連性」まで捉えられます。

### 個人開発でできること

* 自分の読書ノートや論文PDFに問い合わせるパーソナルナレッジベース
* 技術ブログの記事をまとめて検索できるリサーチアシスタント
* ローカルのコードベースに対して「このバグに関連する実装はどこ？」と聞けるコード探索ツール

ベクトルDBは **ChromaDB**（完全無料・ローカル動作）から始めるのがおすすめです。LLMのAPIコスト節約には `gpt-4o-mini` や `claude-haiku` が現実的です。

### 主要ツール構成

| 役割 | 無料で使えるもの |
| --- | --- |
| フレームワーク | LangChain / LlamaIndex |
| ベクトルDB | ChromaDB / Qdrant（ローカル） |
| 埋め込みモデル | text-embedding-3-small / Ollama |
| LLM | gpt-4o-mini / claude-haiku-4-5 |

## 3. AIエージェントフレームワーク（CrewAI / LangGraph）

### 何が熱いのか

「LLMに単発で答えさせる」時代から、「複数ステップのタスクをLLMが自律的にこなす」時代へ移行しています。

2026年のエージェントフレームワーク市場は実質 **CrewAI と LangGraph の二強** です。

**CrewAI** は「役割を持ったAIのチーム」を作るイメージで、コードがシンプルで入門ハードルが低いのが特徴です。リサーチャー・ライター・レビュアーといった役割をYAMLで定義するだけで動きます。GitHubスターは45,900超（2026年3月時点）。

**LangGraph** はエージェントの動作をグラフ（ノード＋エッジ）で定義し、複雑な条件分岐・Human-in-the-Loop・チェックポイント再開など本番品質の機能が揃っています。学習コストは高めですが、予測可能性と制御性は業界最高水準です。

2026年のコミュニティで定着しつつあるベストプラクティスは、**「CrewAIでPoCを作り、本番化するときLangGraphに移行する」** というパスです。

### 個人開発でできること

* 「競合調査 → 要点まとめ → Slack通知」を自動化するリサーチエージェント
* ブログ記事のドラフト作成・推敲・SEOチェックをこなすライティングエージェント
* GitHub Issueを読んで対応方針を提案するコードレビューアシスタント

### CrewAIの最小構成

```
from crewai import Agent, Task, Crew

researcher = Agent(
    role="リサーチャー",
    goal="指定テーマの最新情報を収集する",
    backstory="Web検索と情報整理が得意なAIエージェント"
)

task = Task(
    description="LLM最新動向を調べて日本語でまとめよ",
    agent=researcher
)

crew = Crew(agents=[researcher], tasks=[task])
result = crew.kickoff()
```

## 4. ローカルLLM（Ollama）

### 何が熱いのか

Ollama の登場で、**GPU不要・MacBook上でLLMをローカル実行する**ことが一般的になりました。Llama・Gemma・Qwenなど主要なオープンソースモデルが、コマンド1行でダウンロード・起動できます。

個人開発においてローカルLLMが重要な理由は3つあります。

* **APIコストがゼロ**：プロトタイピング段階でAPIコストを気にせず実験できる
* **プライバシー**：機密データをクラウドに送らずに処理できる
* **オフライン動作**：ネット接続不要で動く

RAGやMCPと組み合わせることで、**完全ローカルで動くプライベートAIアシスタント**を構築できます。

### 個人開発でできること

* `ollama run qwen3` で日本語対応LLMをすぐに動かせる
* LangChain / LlamaIndex と組み合わせてローカルRAGを構築
* MCPサーバーのバックエンドとして、API課金なしでエージェントを試せる

### セットアップ

```
# Ollamaインストール（macOS）
brew install ollama

# モデル起動
ollama run qwen2.5:7b
```

OpenAI互換のAPIエンドポイント（`http://localhost:11434`）が立ち上がるので、既存のLangChainコードをほぼそのまま流用できます。

## まとめ：個人開発で始めるなら

| トピック | 入門難易度 | コスト | まず試すなら |
| --- | --- | --- | --- |
| MCP | ★★☆ | 低（LLM API代のみ） | FastMCP + Claude Desktop |
| RAG | ★★☆ | 低（ChromaDB無料） | LangChain + ChromaDB + gpt-4o-mini |
| エージェント | ★★★ | 低〜中 | CrewAI（無料枠あり） |
| ローカルLLM | ★☆☆ | 無料 | Ollama + Qwen2.5 |

4つの中で**一番すぐ動くのはOllama**、**「なにか作った」感を最も得やすいのはRAG**、**将来性と汎用性が高いのはMCP**という印象です。

どれも1〜2日あれば動くものが作れる成熟度になっています。ぜひ試してみてください。
