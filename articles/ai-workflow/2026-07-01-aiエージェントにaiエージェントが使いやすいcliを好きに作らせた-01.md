# はじめに

AIエージェントに作業を任せていると、よく使われるCLIがだいたい決まっていることに気づきます。

`jq`、`yq`、`mlr`、`rg`、`duckdb` などを使い、必要に応じてパイプラインでつないでデータを整形したり、検証したり、差分を見たりします。

これは便利なのですが、エージェントの文脈では少し気になる点がありました。毎回複数のCLIを組み合わせると、コマンド列も説明も長くなります。出力形式や終了コードの意味もツールごとに変わるため、エージェントがそれを解釈するためのトークンも増えます。

そこで、よく使う複数CLIの組み合わせを、契約のある一つのCLIに寄せれば、AIエージェントの出力トークンを抑えつつ、失敗の解釈もしやすくできるのではないかと考えました。

その仮説から作り始めたのが `dataq` です。

<https://github.com/koizumikento/dataq>

なお、これは完成された汎用ツールというより、あくまで仮説検証用のツールです。AIエージェントにとって本当に使いやすいCLIとは何かを試すために作っています。確証のないことをやっていることに留意してください。

最初に、AIエージェントがよく利用するCLIの洗い出しを行いました。そのうえで、「AIエージェントにとって使いやすいCLI」という方向だけを決め、実装はかなりAIエージェントに任せて好きなように作らせました。

この記事では、その過程で作ったCLIを題材に、「AIエージェントにとって使いやすいCLIとは何か」を整理します。

# 始まりはパイプラインの圧縮だった

きっかけは、AIエージェントがよく次のようなことをしていたことです。

```
yq -o=json '.jobs' .github/workflows/ci.yml | jq '...' | mlr --ijson --ojson ...
```

あるいは、CSVをJSONにして、`jq` で絞り込み、`mlr` で集計し、さらに結果を検証する、といった流れです。

人間が手元で一度だけ実行するなら、これは自然な使い方です。既存CLIを組み合わせれば、すぐに欲しい結果を得られます。

しかし、AIエージェントに繰り返し実行させると、少し話が変わります。

* どのCLIを使うかを毎回選ぶ必要がある
* パイプラインの途中形式を毎回意識する必要がある
* 失敗時にどの段が失敗したのかを読み解く必要がある
* 後続処理が期待するJSONの形を毎回説明する必要がある
* コマンド列と説明が長くなり、出力トークンを使う

ここで考えたのは、「エージェントがよく使うパイプラインには、実は定型があるのではないか」ということです。

定型があるなら、それを一つのCLIコマンドとして契約化できます。たとえば、「入力を正規化する」「フィールド概要を短く見る」「ルールで検証する」「構造差分を見る」「グループ集計する」といった操作です。

この発想が `dataq` の出発点です。

複数のCLIを否定したいわけではありません。探索では `jq` や `yq` を直接使うほうが速いです。ただ、エージェントやCIで繰り返し使う処理は、入出力と終了コードの契約を持った一つのCLIに寄せたほうが扱いやすいのではないか、という考えです。

# 作ったもの

`dataq` は、JSON / YAML / CSV / JSONL / ドキュメント入力を対象にした、決定的な前処理・検証・差分CLIです。

LLMに何かを「考えさせる」ツールではありません。LLMやエージェントの前後で、データを安定して整えるための道具です。

やることは大きく次のようなものです。

* 入力データのフィールド概要を短いJSONで把握する
* YAMLやCSVを決定的なJSONに正規化する
* JSON Schemaやルールで検証し、差分を見る
* 集計や結合を、再利用しやすい固定コマンドにする
* コマンドの契約や実行計画をJSONとして確認する

# なぜAIエージェントに好きに作らせたのか

今回、自分で細かい仕様を決めきってから実装させるのではなく、かなりAIエージェントに任せて作らせました。

理由は、人間が最初から設計しすぎると、結局「人間にとって使いやすいCLI」に寄ってしまうと考えたからです。

やりたかったのは、AIエージェントにとって使いやすいCLIの仮説検証です。であれば、AIエージェントが普段どのCLIを使い、どこで迷い、どの情報を契約として欲しがるのかを、実装にも反映させたほうがよいと考えました。

ただし、任せたかったのは「好きな機能を無制限に増やすこと」ではありません。

ざっくり分けると、次のような役割分担でした。

| 項目 | 内容 |
| --- | --- |
| 人間が決めたこと | よく使うCLIの組み合わせを一つの契約CLIに寄せる、という仮説 |
| AIエージェントに任せたこと | コマンド構成、実装方針、テスト、READMEやskillのたたき台 |
| 人間が後から見たこと | 出力契約、終了コード、過剰な機能追加になっていないか |

まだ定量的な検証はできていません。現時点では、コマンド列が短くなるか、出力JSONの形が安定するか、終了コードの解釈がしやすいか、実行前後の診断を残しやすいか、といった観点を見ています。

コマンドを短くすることよりも、エージェントが判断しなければならないことを減らしたかった、というほうが近いです。

まず、出力形式が揺れます。途中で `jq` のフィルタを変えたり、`mlr` のオプションを変えたりすると、後続処理が前提にしていたJSONの形が変わることがあります。

次に、失敗の意味が揺れます。外部ツールごとに終了コードやエラーメッセージの設計が違うため、エージェントが「これは入力ミスなのか、検証失敗なのか、内部エラーなのか」を安定して判断しづらくなります。

さらに、再現性の説明が難しくなります。AIエージェントが組み立てたパイプラインは、後から人間が見ても「どの段で何が起きたのか」を追いづらいことがあります。

この状態だと、エージェントはコマンドの実行だけでなく、途中形式の説明、失敗理由の推測、次に実行すべきコマンドの選択まで毎回行うことになります。そのぶん、出力トークンも使います。

`dataq` では、このあたりをCLI側の契約として固定する方針にしました。

# dataqの設計方針

`dataq` の中心にある設計方針は、次の4つです。

## 1. JSONを標準の入出力にする

エージェントにとって、自然文ログよりもJSONのほうが扱いやすいです。

たとえば `profile --brief` を使うと、入力データのフィールド概要をコンパクトなJSONとして取得できます。

```
printf '%s\n' '[{"team":"a","price":10},{"team":"a","price":20},{"team":"b","price":5}]' \
  | dataq profile --from json --brief --sort-fields unique_count --max-fields 10
```

出力例です。

```
{
  "record_count": 3,
  "field_count": 2,
  "truncated": false,
  "fields": [
    {
      "path": "$[\"price\"]",
      "null_ratio": 0.0,
      "unique_count": 3,
      "dominant_type": "number",
      "numeric": {
        "count": 3,
        "min": 5.0,
        "max": 20.0,
        "mean": 11.666667,
        "p50": 10.0,
        "p95": 20.0
      }
    },
    {
      "path": "$[\"team\"]",
      "null_ratio": 0.0,
      "unique_count": 2,
      "dominant_type": "string",
      "numeric": null
    }
  ]
}
```

この形式なら、エージェントは長いCSVやJSONの全件を読む必要がありません。まず `profile --brief` で形だけ確認し、必要な列や処理方針を決められます。

## 2. 終了コードの意味を固定する

`dataq` では、終了コードを次のように扱います。

| 終了コード | 意味 |
| --- | --- |
| `0` | 成功 |
| `2` | 検証不一致やポリシー違反など、期待される不一致 |
| `3` | 入力や使い方のエラー |
| `1` | 内部エラーや予期しない失敗 |

この分類は、エージェントにとって重要です。

たとえば、JSON Schema検証で `2` が返った場合、それはCLI自体の故障ではありません。「入力が期待に合わなかった」という業務上の結果です。一方で `3` なら、ファイルパスやオプション指定を見直すべきです。

エージェントが自律的に次の手を選ぶには、この違いが安定している必要があります。

## 3. 実行前に契約を確認できる

`dataq` には `contract` というコマンドがあります。

```
dataq contract --command profile
```

これは、対象コマンドの出力スキーマ名、主要フィールド、終了コード契約、注意点をJSONで返します。

```
{
  "command": "profile",
  "schema": "dataq.profile.output.v1",
  "output_fields": [
    "record_count",
    "field_count",
    "truncated",
    "returned_field_count",
    "fields",
    "missing_fields"
  ],
  "exit_codes": {
    "0": "success",
    "2": "validation mismatch is not used by this command",
    "3": "input/usage error",
    "1": "internal/unexpected error"
  }
}
```

人間がREADMEを読んで理解するのではなく、エージェントがコマンドの契約をコマンド自身から取得できます。

これは地味ですが、エージェント向けCLIとして効きそうな設計だと感じています。

## 4. 実行計画と実行時診断を分ける

`dataq` には `emit plan` もあります。

```
dataq emit plan --command aggregate \
  --args '["--input","orders.json","--group-by","team","--metric","sum","--target","price","--sort-by","metric","--order","desc","--limit","10"]'
```

出力例です。

```

  "args": [

    "team",
    "--metric",
    "sum",

    "--sort-by",
    "metric",
    "--order",
    "desc",
    "--limit",
    "10"
  ],
  "stages": [
    {
      "order": 1,
      "step": "resolve_aggregate_input",
      "tool": "rust",
      "depends_on": []
    },
    {
      "order": 2,
      "step": "read_aggregate_values",
      "tool": "rust",
      "depends_on": ["resolve_aggregate_input"]
    },
    {
      "order": 3,
      "step": "validate_aggregate_keys",
      "tool": "rust",
      "depends_on": ["read_aggregate_values"]
    },
    {
      "order": 4,
      "step": "execute_aggregate_with_mlr",
      "tool": "mlr",

```

`emit plan` は実データを読みません。あくまで静的な実行計画です。

一方で、実際にコマンドを実行するときは `--emit-pipeline` を付けられます。

```
dataq --emit-pipeline aggregate \
  --input orders.json \
  --group-by team \
  --metric sum \
  --target price \
  --sort-by metric \
  --order desc \
  --limit 10 \
  > top-teams.json \
  2> pipeline.json
```

標準出力には本来の結果だけを出し、標準エラーに実行時診断を出します。

```

```

これにより、エージェントが実行した処理について、「何を使って、どの順序で、どのバージョンで動いたか」を後から確認できます。

# 実装として見たdataq

実装はRustです。

構成は大きく次のように分かれています。

```
src/
  cmd/       CLI境界
  engine/    正規化・検証・差分などのロジック
  io/        JSON / YAML / CSV / JSONL の入出力
  adapters/  jq / yq / mlr / duckdb など外部ツール連携
  domain/    ルール、エラー、レポートなどの共通モデル
```

この分け方も、エージェントに作業させやすい構造だと感じました。

たとえば新しいサブコマンドを追加する場合、CLI引数は `src/cmd/`、中核ロジックは `src/engine/`、外部ツール呼び出しは `src/adapters/` に分けられます。変更範囲が明確なので、エージェントに「この機能を追加して、CLIテストとintegration testを足して」と依頼しやすくなります。

また、外部ツールを呼び出すときにシェル文字列を組み立てるのではなく、引数配列として実行する方針にしています。これはユーザー入力を扱うCLIとしても、エージェントが生成するコードとしても重要です。

# MCP対応も入れた

`dataq` には `mcp` サブコマンドもあります。

```
printf '%s\n' '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' \
  | dataq mcp
```

`tools/list` では、`dataq.profile` や `dataq.aggregate` などのツール定義が返ります。各ツールには `inputSchema`、例、終了コード契約が含まれます。

CLIとしても使えるし、MCP toolとしても使える、という形です。

個人的には、ここが「エージェントにとって使いやすいCLI」の先にある形だと思っています。最初はシェルコマンドとして使えます。必要になったら、そのままMCP toolとしても公開できます。

# Codex skillも同梱した

もう一つ面白い点として、`dataq` にはCodex向けのskillも同梱しています。

```
dataq codex install-skill
```

これにより、エージェント側に「dataqはどういう場面で使うべきか」「先に何を確認すべきか」という運用知識を渡せます。

CLI本体だけでなく、エージェントがそのCLIをうまく使うための説明書も一緒に配る、という考え方です。

これは、エージェントにCLIを使わせるうえで重要な設計要素になりそうだと感じました。

人間向けにはREADMEがあります。エージェント向けにはskillがあります。さらに機械向けには `contract` や `tools/list` があります。

同じCLIでも、読む相手ごとに入口を分けるイメージです。

# 使ってみたい場合

`dataq` 本体のリポジトリとは別に、Homebrewなどで配布するためのリポジトリも用意しています。

<https://github.com/koizumikento/stray-tools>

このリポジトリには `Formula/dataq.rb` を置いています。`homebrew-` 接頭辞のないリポジトリをtapとして使っているため、HomebrewではURLを指定してtapします。

```
brew tap koizumikento/stray-tools https://github.com/koizumikento/stray-tools.git
brew install koizumikento/stray-tools/dataq
```

formula側では、`jq`、`yq`、`miller`、`csvkit`、`duckdb`、`pandoc`、`xh`、`ripgrep` など、`dataq` が内部連携するCLIも依存として定義しています。

# 作ってみて感じたこと

AIエージェントにCLIを作らせるとき、機能そのものよりも「境界の設計」が大事だと感じました。

具体的には、次のような境界です。

* 標準出力に何を出すのか
* 標準エラーに何を出すのか
* 終了コードをどう解釈するのか
* 外部ツールの失敗をどう包むのか
* コマンドごとの出力契約をどう公開するのか
* 実行前の計画と実行後の診断をどう分けるのか

人間が手で使うだけなら、多少曖昧でもなんとかなります。

しかし、AIエージェントに繰り返し使わせるなら、曖昧さは失敗要因になりやすいです。逆に、契約がはっきりしているCLIは、エージェントにとって扱いやすい部品になります。

`dataq` はまだ小さなCLIですが、作ってみると「AIエージェント時代のCLI設計」の観点が少し見えてきました。

# おわりに

`dataq` は、AIそのものを実行するCLIではありません。AIエージェントがよく使う処理を、契約のある一つのCLIに寄せるとどうなるかを試すために作ったものです。

まだ定量的に検証できているわけではありません。それでも、作ってみると、AIエージェント向けの道具では機能そのものよりも境界の設計が効きそうだと感じました。

標準出力に何を出すか。終了コードをどう扱うか。実行前に契約を確認できるか。実行後に診断を残せるか。

こうした地味な部分をCLI側に寄せておくと、AIエージェントに任せる範囲を少し広げやすくなるのではないか、というのが今回の感触です。
