[Mastra](https://mastra.ai/) の [公式Blog](https://mastra.ai/blog) で公開された [Foundations](https://mastra.ai/blog/category/foundations) カテゴリの記事を、**背景知識を補いながらより詳しく**解説します。

---

## この記事が書かれた背景

AIエージェントを「作ってみたい」と思ったとき、最初にぶつかるのは「何から手をつければいいのか」という問題です。LLMを使ったチャットボットなら比較的簡単に始められますが、ツールを使い、ワークフローを組み、複数のエージェントを協調させる本格的なエージェントシステムとなると、話が違います。

Mastra チームは、何百ものチームと協力してエージェントシステムを構築してきた経験から、[Principles of Building AI Agents](https://mastra.ai/books/principles-of-building-ai-agents) という書籍も出しています。今回の元記事は、そこで培った知見を凝縮した「プロトタイプ構築の実践ガイド」です。

この記事の特徴は、抽象的なアーキテクチャ論ではなく、\*\*「実際に顧客と一緒に作ったときに、どういう順番で何を考えたか」\*\*という実務の流れを7ステップに整理している点にあります。政府機関、Fortune 500企業、投資会社など、実際の事例に基づいた内容です。

---

## ステップ1：問題をスコーピングする ― 「何を作るか」の解像度を上げる

プロトタイプ構築の最初のステップは、コードを書くことではありません。**解決したい問題を、エージェントの機能やプリミティブに対応づける**ことです。

「プリミティブ」とは、エージェントフレームワークが提供する構成要素のことです。Mastra でいえば、Agent、Tool、Workflow、Memory、Processor、Scorer などが該当します。問題を分解して「この部分にはどのプリミティブが必要か」を考えるのがスコーピングです。

### 事例：政府機関の給付金エージェント

Mastra チームが協力した政府機関は、ユーザーと対話しながら給付金の申請・取り消し・確認を行うエージェントを必要としていました。この問題を分解すると、以下のプリミティブが必要だと判明しました。

| 要件 | 対応するプリミティブ | 理由 |
| --- | --- | --- |
| 申請・取り消し・確認を振り分ける | マルチエージェントオーケストレーション | 1つのエージェントですべてをこなすより、専門エージェントに分けたほうが精度が高い |
| 個人情報の保護 | プロセッサ（PII匿名化） | 給付金の文脈では社会保障番号や住所などの機微情報を扱うため |
| 申請手続きの案内 | ヒューマンインザループ付きワークフロー | 完全自動ではなく、重要なステップで人間の確認を挟む必要がある |

### 事例：社内ミーティングレコーダー

別の企業は、会議を録音してサマリーを全参加者にメール送信するエージェントを求めていました。

| 要件 | 対応するプリミティブ | 理由 |
| --- | --- | --- |
| 会議音声の文字起こし | Speech-to-text 機能 | 音声データをテキストに変換してから要約する |
| サマリーのメール送信 | MCP 統合 | メールクライアントとの連携にMCPサーバーを利用 |
| サマリー品質の担保 | スコアラー | 自動生成されたサマリーが元の会話内容を正確に反映しているか評価する |

### なぜスコーピングが最初なのか

このステップの本質は、「エージェントに何をさせたいか」を曖昧なまま進めないことです。「AIで業務を効率化したい」という抽象的なゴールを、「この入力に対して、このプリミティブを組み合わせて、この出力を得る」という具体的な設計に落とし込む作業です。

ここが曖昧だと、後のステップでフレームワーク選定もモックデータも的外れになります。逆に、このステップを丁寧にやると「意外とシンプルな構成で済む」と気づくこともあります。

## ステップ2：フレームワークを選ぶ ― 何が標準装備で、何を自作するのか

問題をプリミティブに分解できたら、次はそれを実装するためのフレームワークを選びます。

元記事では当然ながら Mastra が推奨されていますが、重要なのはフレームワークの名前ではなく、**選定の基準**です。

### フレームワーク選定で確認すべきこと

| 観点 | 確認内容 |
| --- | --- |
| プリミティブの充実度 | ステップ1で洗い出したプリミティブが標準で提供されているか |
| カスタマイズ性 | 標準機能では足りない部分を、自分で拡張できる設計になっているか |
| 言語エコシステム | チームが慣れている言語（TypeScript / Python など）で使えるか |
| 可観測性 | トレース・eval・メトリクスなど、デバッグ・改善のための仕組みがあるか |
| デプロイメントの柔軟性 | ローカル開発からクラウドデプロイまで、スムーズに移行できるか |

「すぐに使える」機能と「自分で構築しなければならない」機能の区別は、特にエージェント開発初心者にとって重要です。ツールコール、ストリーミング、メモリ管理、ワークフローオーケストレーションなどを一から手作りするのは、プロトタイプの段階では避けたいコストです。

Mastra の場合、上記のプリミティブ（Agent、Tool、Workflow、Memory、Processor、Scorer、MCP統合など）がフレームワークに組み込まれているため、プロトタイプ段階で手作りする部分を最小限に抑えられます。

## ステップ3：モックデータでサンドボックス化する ― 統合を後回しにする技術

ここがプロトタイプ構築で最も実践的なアドバイスです。**本物のデータソースとの統合を後回しにして、モックデータで先に動かす**。

エージェントが実際に扱うデータソース（社内システム、外部API、データベースなど）は、往々にして認証が複雑だったり、アクセス権の取得に時間がかかったり、スキーマが不安定だったりします。プロトタイプの段階でこれに正面からぶつかると、「エージェントのロジック検証」という本来の目的にたどり着く前に時間を消費してしまいます。

### 事例：Fortune 500企業のインシデントマッピング

社内インシデントマッピングにMastraエージェントを使用するFortune 500企業のプロトタイプでは、以下のアプローチを取りました。

* 本番の社内システムに接続する代わりに、**CSV ファイル**でインシデントデータと従業員メタデータをモック
* エージェントのロジック（インシデントの分類、担当者の割り当て、優先度の判定など）をまず検証
* ロジックが固まってから、本番システムとの統合に移行

### 事例：投資会社のデューデリジェンス

潜在的な取引のデューデリジェンスを自動化したい投資会社は、以下でモック環境を構築しました。

| モックデータ | 実際のデータソース |
| --- | --- |
| AI生成のPDF・Excel・契約書 | 実際の投資案件のデータルーム |
| Markdown のデューデリジェンスチェックリスト | 社内の DD テンプレート |
| テキストファイルの1ページメモ | アナリストが作成する投資メモ |

### モック化のもう一つの方法：コーディングエージェントの活用

元記事では、「コーディングエージェントを使って最小限の統合とツールセットを作成する」というアプローチも紹介されています。つまり、フルスイートの統合を作る代わりに、AI（コーディングエージェント）に最小限のスタブ実装を生成させるという手法です。

これはエージェントを使ってエージェントのプロトタイプを作る、というメタ的なアプローチですが、実用的です。たとえば「このAPIのレスポンスはこういうJSONを返す」というスタブを、GitHub Copilot や Claude などに書かせれば、統合のモック化にかかる時間を大幅に短縮できます。

## ステップ4：アーキテクチャとモデルを実験する ― 「正解」は試さないとわからない

モックデータが揃ったら、フレームワークを使って実際に実装を試します。ここでのポイントは2つあります。

### アーキテクチャの実験

同じ問題に対しても、エージェントの構成パターンは複数あります。

| パターン | 特徴 | 適するケース |
| --- | --- | --- |
| 単一エージェント | シンプル、デバッグしやすい | タスクが単純、あるいはスコープが限定的 |
| スーパーバイザー＋サブエージェント | ルーティングと実行を分離 | タスクが多岐にわたり、専門性が必要 |
| パイプライン型ワークフロー | ステップを順序通りに実行 | 手続き的な処理フロー |
| 並列実行 | 複数エージェントが同時に動作 | 独立したタスクを同時処理したい場合 |

元記事では、社内チャットボットを構築するチームがいくつかのオーケストレーションパターンを試した結果、**タスクルーティングに特化したスーパーバイザーエージェント**と、**小さくスコープされたタスクを処理するサブエージェント**の構成に落ち着いた、という事例を紹介しています。

重要なのは「最初から完璧なアーキテクチャを選ぶ」のではなく、**複数パターンを試して比較する**ことです。プロトタイプ段階だからこそ、アーキテクチャの変更コストは低い。

### モデルの実験

LLM モデルの選択もプロトタイプ段階で試すべきことです。元記事では以下の使い分けが例示されています。

| タスク | 推奨モデル例 | 理由 |
| --- | --- | --- |
| コーディングタスク | Claude Opus 4.6 / GPT 5.4 | 高い推論能力が必要 |
| 単純な分類タスク | Gemini 3.1 Flash-Lite | 安価・高速で十分な精度 |

すべてのタスクに最先端モデルを使う必要はありません。むしろ、タスクごとに最適なモデルを割り当てる**モデルルーティング**は、コスト最適化と応答速度の両面で有効です。Mastra ではこのモデルルーティングをフレームワークレベルでサポートしています。

## ステップ5：v1をチームと共有する ― フロントエンドを作らずにデモする

プロトタイプが動くようになったら、チームに見せたくなります。しかし、**デモのためだけにフロントエンドを作るのは本末転倒**です。

従来のプロトタイプデモでは、エージェントの出力を見せるためにカスタム UI を構築する必要がありました。しかし、UIの出来栄えにフィードバックが引っ張られてしまい、肝心の「エージェントの品質」への評価がぼやけることがあります。

### Mastra Studio を使ったデモ

Mastra では、Studio（セルフデプロイまたはクラウドホスト）へのリンクをチームメイトに共有するだけで、以下のことができます。

| できること | 説明 |
| --- | --- |
| エージェントとチャット | Studio UI上でエージェントと直接対話 |
| ワークフロー実行の可視化 | ステップごとの実行状況をグラフィカルに確認 |
| eval・トレースの確認 | エージェントの出力品質と実行経路を検査 |
| プロンプト・ツールの調整 | Agent Editorでコードなしに設定変更 |

これにより、チームメイトは「レスポンスのストリーミング方法」や「ツールコールの表示方法」といったUI実装の細部ではなく、**エージェントそのものの品質**に集中してフィードバックできます。

非技術者のステークホルダー（営業、サポート、コンプライアンス担当など）にデモする場合にも、Studio のUIは直感的なので、フロントエンド開発なしでフィードバックサイクルを回せます。

## ステップ6：本番化への道筋を立てる ― プロトタイプから先の3つの課題

プロトタイプが動き、チームからのフィードバックも得られた。次は本番環境へのデプロイを考える段階です。元記事では、本番化にあたって3つの領域を挙げています。

### 1. 可観測性（Observability）

プロトタイプ段階では「動くかどうか」が焦点でしたが、本番では「どう動いているか」の継続的な監視が必要です。

* **トレース**：各リクエストの処理経路を追跡し、どのステップで時間がかかっているか、どこでエラーが発生したかを把握
* **実験（Experiments）**：プロンプトやモデルの変更が品質にどう影響するかをデータで検証
* **evalループ**：本番の実行ログを使って、エージェントの出力品質を継続的に評価・改善

### 2. デプロイメント

エージェントをユーザーが使える形で公開する方法です。

* 開発者向け：API エンドポイントとして公開し、既存のアプリケーションから呼び出せるようにする
* ユーザー向け：フロントエンド UI（React、モバイルアプリ、Slack など）から利用できるようにする

### 3. ホスティング

エージェントのコードを実際に実行するインフラです。エージェントは単なるAPI サーバーではなく、ツール呼び出し、ワークフロー実行、プロセッサ、メモリ管理など、多くのコンポーネントを含みます。これらを安定して動かすためのインフラ設計が必要です。

### プロトタイプ段階でも意識しておくべき理由

元記事の重要なメッセージは、**本番化の手段を決めるのは後でいいが、プロトタイプ段階から迅速にイテレーションできる環境は持っておくべき**、ということです。

Mastra を使っている場合は、Server でエージェントを公開し、Studio で可観測性を確保できます。最終的に自社インフラに移行する予定であっても、プロトタイプ段階でこれらを「とりあえず使える」状態にしておくことで、本番化への移行がスムーズになります。

## ステップ7：自信を持って構築する

7つのステップを振り返ると、以下のような流れです。

1. **スコーピング** ― 問題をプリミティブに分解する
2. **フレームワーク選定** ― プリミティブを実装できるフレームワークを選ぶ
3. **サンドボックス化** ― モックデータで統合コストを切り離す
4. **実験** ― アーキテクチャとモデルを試して最適解を探す
5. **共有** ― フロントエンド不要でチームにデモする
6. **本番化の計画** ― 可観測性・デプロイ・ホスティングの道筋を立てる
7. **構築** ― 自信を持って進む

この流れの本質は、**不確実性を段階的に減らしていく**プロセスです。最初からすべてを決めようとせず、各ステップで1つの不確実性を解消して次に進む。プロトタイプとは本来そういうものです。

## 訳註：Mastraチームの実際の顧客事例に注目

この記事はMastraの「Foundations」カテゴリに置かれています。Foundationsは、特定の機能のアナウンスではなく、エージェント開発の考え方やベストプラクティスを体系的に伝えるカテゴリです。

内容自体は「プロトタイプの作り方」として王道的ですが、注目すべきは**各ステップに実際の顧客事例が紐づいている**ことです。政府機関の給付金エージェント、Fortune 500企業のインシデントマッピング、投資会社のデューデリジェンス ― これらはMastraチームが実際に顧客と一緒に構築した経験です。

特にステップ3の「モックデータでサンドボックス化する」は、見落とされがちですが極めて実践的なアドバイスです。エージェント開発において、統合（Integration）のコストは常に予想以上に膨らみます。モックデータで先にロジックを検証し、統合は後から差し替える ― この順序を守るだけで、プロトタイプの到達速度は大きく変わります。

また、ステップ5の「フロントエンドを作らずに Studio でデモする」というアプローチは、Mastra ならではの利点です。プロトタイプ段階でフロントエンド開発にリソースを割かずに済むことで、**エージェントの品質改善に集中できる**という効果は、思った以上に大きいでしょう。
