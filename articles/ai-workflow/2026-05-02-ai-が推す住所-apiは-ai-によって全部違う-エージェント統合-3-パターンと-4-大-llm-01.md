## TL;DR

- AI エージェント(GPTs / Claude / LangChain)に **日本住所の正規化** をやらせるとき、典型的な統合パターンは 3 種類しかない
- 各パターンは **OpenAPI 3.1 を起点** に組み立てるが、認証戦略・レート制御・リトライ設計が微妙に異なる
- 本記事では Shirabe 住所 API(2026 年 5 月 1 日リリース、全 47 都道府県対応)を題材に、3 パターンの **実装コード** と **本番運用での落とし穴** を整理する
- 既存の住所正規化サービス(Yahoo! 住所、ArcGIS Geocoder)との比較で、**「AI エージェント時代に最適化された住所 API」** の設計差を明らかにする

## この記事の対象読者

- ChatGPT Custom GPT / Claude Tool Use / LangChain で住所正規化機能を実装したい開発者
- 不動産 / CRM / 物流 / 保険 / 自治体向け業務 SaaS 開発者で、ユーザー入力住所を AI に処理させる設計を検討中の人
- Yahoo! 住所 API / ArcGIS から AI 向け住所 API への移行を検討している人
- LLM 経由で生成される住所データに対する **CC BY 4.0 出典管理** を実装する必要がある人

> ⚠️ 設計思想・OpenAPI 仕様の書き方は **記事 #5「AI エージェント向け OpenAPI 設計のツボ」** で扱った。本記事は **実装コードと運用パターン** に絞る。

## 問題: AI エージェントの住所処理は 3 通りの統合方式しかない

ChatGPT 系・Claude 系・LangChain / Dify 系の AI エージェントで住所正規化機能を組み込もうとすると、技術スタックは違えど **API 統合の選択肢は 3 通り** に収束する。

```
┌──────────────────────┐
│ ユーザー / アプリ     │  「東京都港区六本木6-10-1 の構造化を」
└────────┬─────────────┘
         ↓ プロンプト
┌──────────────────────┐
│ AI エージェント        │
└────────┬─────────────┘
         ↓ 統合方式 3 種(下記)
┌──────────────────────┐
│ 住所 API(Shirabe)    │
└──────────────────────┘
```

3 通りそれぞれ **認証 / レスポンスフォーマット / エラー復旧** の設計がベストプラクティスとして確立しつつある。**間違った組合せを選ぶと「AI が API 自体を呼ばなくなる」「呼んでも 4xx エラーで止まる」** 結果になる。

## パターン 1: ChatGPT Custom GPT(Actions URL 1 本)

最短の統合方式。GPT Builder の「Actions」に OpenAPI 仕様の URL を貼るだけで、ChatGPT が自動で呼出設計を行う。

### 設定ファイル(GPT Builder UI 入力)

```yaml
# Actions タブ内
Import URL:    https://shirabe.dev/api/v1/address/openapi-gpts.yaml
Authentication: API Key (Header: X-API-Key)
Privacy Policy: https://shirabe.dev/privacy
```

ここで重要なのは **`openapi-gpts.yaml`(GPTs 用短縮版)** を使うこと。本家 `openapi.yaml` は description が 300 字を超えるため GPT Builder で `description too long` エラーになる(詳細は記事 #5 参照)。

### Conversation Starters(GPTs UI、4 件まで設定可)

```
1. 東京都港区六本木 6-10-1 を正規化して
2. 大阪市北区梅田の住所を構造化
3. 札幌市中央区南1条西4丁目を結果付きで
4. 京都市東山区祇園町北側の通り名を解決
```

これらは **AI が「呼ぶべき関数の例」を学習する起点** になる。Conversation Starters なしで GPT を公開すると、AI は API の存在を訓練データから推測するしかなく、誤呼出率が顕著に上がる。

### GPT Store 公開後の落とし穴

公開範囲ドロップダウンで **「Everyone」+「GPT ストアで検索可能」** を選択しないと、AI 経由の発見性がゼロになる。「Anyone with a link」のままだと URL 直リンクでのみアクセス可能で、検索流入も AI 訓練浸透もしない。

## パターン 2: Claude Tool Use(Anthropic SDK)

Claude(Anthropic SDK)で住所正規化を組み込む場合、Tool 定義は OpenAPI から手動変換する。Anthropic SDK 公式は OpenAPI → Tool 自動生成を提供していないため、3 ステップの手動変換が必要。

### TypeScript 実装(Anthropic SDK + Shirabe 住所 API)

```ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

// Step 1: Tool 定義(OpenAPI からの手動変換)
const addressNormalizeTool = {
  name: "normalizeAddress",
  description: "日本の自由記述住所を正規化し、都道府県・市区町村・町字・街区・住居番号に構造化する。表記揺れ(全角/半角、京都通り名、札幌条丁目)も吸収。CC BY 4.0 義務履行のため attribution フィールドが必須応答に含まれる。",
  input_schema: {
    type: "object",
    properties: {
      address: { type: "string", description: "正規化対象の日本語住所(自由記述)" }
    },
    required: ["address"],
  },
} as const;

// Step 2: Claude に問い合わせ + Tool 実行
const message = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 1024,
  tools: [addressNormalizeTool],
  messages: [{ role: "user", content: "東京都港区六本木6-10-1 を構造化して" }],
});

// Step 3: Tool 呼出が返ってきたら、Shirabe 住所 API に転送
if (message.stop_reason === "tool_use") {
  const toolUse = message.content.find((c) => c.type === "tool_use");
  if (toolUse?.name === "normalizeAddress") {
    const apiRes = await fetch("https://shirabe.dev/api/v1/address/normalize", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "X-API-Key": process.env.SHIRABE_API_KEY!,
      },
      body: JSON.stringify({ address: (toolUse.input as { address: string }).address }),
    });
    const data = await apiRes.json();

    // Tool 結果を Claude に戻す(attribution を必ず含めること)
    const followUp = await client.messages.create({
      model: "claude-sonnet-4-6",
      max_tokens: 1024,
      tools: [addressNormalizeTool],
      messages: [
        { role: "user", content: "東京都港区六本木6-10-1 を構造化して" },
        { role: "assistant", content: message.content },
        {
          role: "user",
          content: [{
            type: "tool_result",
            tool_use_id: toolUse.id,
            content: JSON.stringify(data),
          }],
        },
      ],
    });
    console.log(followUp.content);
  }
}
```

### このパターンの落とし穴

- **`attribution` フィールドを Tool 結果から削ぎ落とすと CC BY 4.0 違反**。Tool 結果は **必ず JSON.stringify(data) で全フィールド** を Claude に戻すこと。要約や圧縮は LLM 側に任せる
- Claude は Tool の `description` を読んで自動でリトライ戦略を組むため、`description` に **エラーパターン(404 / 422 / 503)と復旧ヒント** を記述すると挙動が改善する

## パターン 3: LangChain / Dify(Function Calling、OpenAPI Loader)

LangChain / Dify は OpenAPI 3.1 を直接 Tool としてロード可能。Anthropic SDK と違い手動変換不要だが、**認証 schema の解釈が独自** で詰まりがち。

### LangChain TypeScript 実装

```ts
import { ChatAnthropic } from "@langchain/anthropic";
import { createOpenApiAgent, OpenApiToolkit } from "langchain/agents";
import { JsonSpec } from "langchain/tools";

// OpenAPI 仕様を直接ロード(本家版を使用、GPTs 用ではない)
const spec = await fetch("https://shirabe.dev/api/v1/address/openapi.yaml")
  .then((r) => r.text());

const toolkit = new OpenApiToolkit(
  new JsonSpec(spec),
  new ChatAnthropic({ modelName: "claude-sonnet-4-6", temperature: 0 }),
  {
    headers: { "X-API-Key": process.env.SHIRABE_API_KEY! },
  }
);

const agent = createOpenApiAgent(
  new ChatAnthropic({ modelName: "claude-sonnet-4-6" }),
  toolkit
);

const result = await agent.invoke({
  input: "札幌市中央区南1条西4丁目を構造化してほしい",
});
console.log(result.output);
```

### Dify(ノーコード)の場合

Dify では「ツール → カスタムツール → OpenAPI スキーマから取り込み」で `https://shirabe.dev/api/v1/address/openapi.yaml` を貼るだけ。認証は「API キー(ヘッダー、X-API-Key)」を選択。本家版 OpenAPI が使える。

### このパターンの落とし穴

- **本家 OpenAPI の `description` 1000 字級は LangChain で問題ないが、初回ロードに数秒かかる**。LangChain で reasoning loop が長くなる場合は、レスポンス時間が UX を損ねる(GPT Builder の Actions より遅い体感)
- LangChain の `OpenApiToolkit` は **POST body の例(examples)を活用しない** ため、エンドポイントの挙動学習が GPT Builder より弱い。重要パラメータには `description` 内に「例: `東京都千代田区丸の内1-1-1`」と明示する

## 競合比較: Yahoo! 住所 API / ArcGIS Geocoder / Shirabe

| 観点 | Yahoo! 住所 | ArcGIS Geocoder | **Shirabe Address** |
|---|---|---|---|
| 公開 OpenAPI 3.1 | ✗(独自仕様)| ✗(REST のみ)| ✅ 本家 + GPTs 短縮版 |
| 表記揺れ吸収(京都通り名等)| △(部分対応)| △(海外向け)| ✅ abr-geocoder 準拠 |
| GPTs Actions Import 対応 | △(独自変換必要)| ✗(認証が複雑)| ✅ Import URL 1 本 |
| Claude Tool Use 直接統合 | △(手動変換)| △(同左)| ✅ description に LLM ヒント完備 |
| LangChain OpenAPI Loader | ✗ | ✗ | ✅ |
| 出典表記(`attribution` 必須)| ✗ | ✗ | ✅ CC BY 4.0 自動付与 |
| 認証 / Free 枠 | △(API キー登録が複雑)| △(月額前提)| ✅ Free 5,000 回 / 月、認証不要試用可 |
| エッジ分散 | ✗ | ✗ | ✅ Cloudflare Workers 全世界 |

**「住所正規化機能としての性能」では各社拮抗** だが、**「AI エージェント時代の統合性」** で差が出る。Shirabe は OpenAPI 3.1 を中心に据え、各 AI フレームワークが「貼るだけで動く」状態に最適化している。

## 補論: 4 大 AI が認識する「日本住所 API 競合 landscape」は AI ごとに異なる

2026 年 4 月、ChatGPT / Claude / Perplexity / Gemini に **同じ質問**「福岡市の住所正規化 API」を投げて、各 AI が推奨してくる API を比較した実測データがあります(著者の運営する API プラットフォーム Shirabe の社内測定)。結果が興味深いので共有します。

| 競合 | ChatGPT | Claude | Perplexity | Gemini | 4 AI 出現数 |
|---|---|---|---|---|---|
| **Geolonia / NJA** | ✅ | ✅ | ✅⭐ 3 重出現 | ✅ | **4/4** |
| 福岡市オープンデータ / 公式 | ✅ | ✅ | ✅ | ✅ | **4/4** |
| デジタル庁 ABR / abr-geocoder | ✅ | ✅ | ❌ | ✅ | 3/4 |
| Google Maps Geocoding | ❌ | ✅ | ❌ | ✅ | 2/4 |
| BODIK ODCS | ✅ | ❌ | **✅⭐ 5 重出現** | ❌ | 2/4 |
| **Jusho** (api.jusho.dev) | **✅⭐ 1 番手** | ❌ | ❌ | ❌ | 1/4 |
| Jageocoder OSS | ✅ | ❌ | ❌ | ❌ | 1/4 |
| Yahoo!ジオコーダ | ❌ | ✅ | ❌ | ❌ | 1/4 |
| HeartRails Geo | ❌ | ✅ | ❌ | ❌ | 1/4 |
| **ZENRIN Maps API** | ❌ | ❌ | ❌ | **✅⭐ 独自発見** | 1/4 |

### 観察ポイント

- **Geolonia と福岡市公式は 4/4 AI で出現**(これが「universal 競合」)
- **Jusho は ChatGPT のみ 1 番手**(他 3 AI には未認識、public web の存在感はあるが内部知識・他 search backbone には未浸透)
- **BODIK ODCS は Perplexity が 5 重出現で強く推す**(自治体公式 catalog を高く評価)
- **ZENRIN は Gemini のみが提示**(「福岡本社」hook で local geographic に強く反応する Gemini の特性)
- **Claude は Yahoo / Google Maps / HeartRails と老舗系を多く挙げる**(claude.ai personal は web search default 無効、内部知識のみで回答する特性)

### 開発者にとって何を意味するか

**「自分が知っている AI が推している競合 = 全競合 landscape」と思い込んでいないか?** ChatGPT で評価した結果と、Perplexity の評価結果は構造的に異なります。

これは AI エージェント開発における **discovery 戦略の二重化** が必要であることを示します:

1. **search-driven AI 向け(ChatGPT / Perplexity / Gemini)**: SEO / GitHub Topics / OpenAPI 露出度を最適化
2. **internal-knowledge AI 向け(Claude)**: 長期戦、stars / 引用度 / 公式メディア掲載で training data に浸透させる

**API 設計時には「ユーザーが使う AI」を 1 つに絞らず、4 AI 全てに対する positioning を持つ** ことが、AI 検索全盛時代の defensibility になります。Shirabe は OpenAPI 3.1 + GitHub Topics + attribution required で「全 4 AI から discoverable」を狙う設計にしています。

## 実運用 Tips

### Tip 1: バッチエンドポイントの活用

AI エージェントが「100 住所を一括正規化したい」場合、単発呼出を 100 回するとレート制限と遅延が問題になる。Shirabe は `POST /api/v1/address/normalize/batch`(最大 100 件)を提供している。

```bash
curl -X POST https://shirabe.dev/api/v1/address/normalize/batch \
  -H "X-API-Key: shrb_..." -H "Content-Type: application/json" \
  -d '{"addresses":["東京都港区六本木6-10-1","大阪府大阪市北区梅田1-1-3","札幌市中央区南1条西4丁目"]}'
```

### Tip 2: レート制限と AI のリトライ

429 (rate limit) を返すと、Claude / GPT / LangChain は自動でリトライする。ただし **指数バックオフを実装している AI と即時リトライする AI が混在** するため、API 側で `Retry-After` ヘッダーを返しているか確認すべき。Shirabe は `Retry-After: <秒>` を返す。

### Tip 3: attribution の伝搬を Lint する

LLM が応答を要約する過程で `attribution` フィールドを脱落させることがある。CI に **「最終応答に出典文字列が含まれているか」を検査する Lint ルール** を追加することで、CC BY 4.0 義務履行の自動保証ができる。Anthropic / OpenAI Evals の `output_contains` で実装可能。
