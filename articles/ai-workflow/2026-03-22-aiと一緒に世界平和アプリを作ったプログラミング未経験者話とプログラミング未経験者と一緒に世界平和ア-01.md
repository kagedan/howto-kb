世界中の人に使ってもらいたい → データベースが必要 → Supabase（無料）

```
CREATE TABLE kindnesses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  author TEXT NOT NULL,
  description TEXT NOT NULL,
  category TEXT NOT NULL,
  parent_id UUID REFERENCES kindnesses(id),
  emoji TEXT DEFAULT '🕊️',

  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

失敗の連続：

| 試行 | 結果 |
| --- | --- |
| Resendで送信 | ❌ 無料プランは本人にしか送れない |
| Brevoに切り替え | ❌ APIキーの種類を間違えた（SMTP用を使ってた） |
| 正しいAPIキーで再挑戦 | ❌ 送信者メールが未認証 |
| 送信者を認証して再挑戦 | ❌ いいね数が0のまま通知される |
| トリガー名を変更 | ✅ やっと成功！ |

5回失敗して、やっとメールが届きました。

AIがいなかったら絶対に解決できなかったと思います。毎回エラーのスクリーンショットを送って、AIが原因を分析して、次の手を教えてくれました。

ポイント：エラーが出ても諦めない。スクリーンショットをAIに送れば、大抵は解決策を教えてくれる。

技術的に面白かったところ

バックエンドなしでメール通知

通常、メール通知にはバックエンドサーバーが必要です。でも今回はSupabaseの `pg_net` 拡張を使って、データベーストリガーから直接HTTPリクエストを送る方式を採用しました。

```
-- いいねされたとき、マイルストーン（1, 5, 10, 25, 50, 100...）で通知
CREATE OR REPLACE FUNCTION notify_on_like()
RETURNS trigger AS $$
DECLARE
  current_likes int;
  milestones int[] := ARRAY[1,5,10,25,50,100,250,500,1000];
BEGIN
  SELECT likes INTO current_likes FROM kindnesses WHERE id = NEW.kindness_id;
  IF current_likes = ANY(milestones) THEN
    -- Brevo APIでメール送信
    PERFORM net.http_post(...);
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

サーバーレス、コスト0円。

トリガーの実行順序の罠

PostgreSQLでは、同じイベントに複数のトリガーがある場合、アルファベット順に実行されます。

* `on_like_change`（いいね数を更新）
* `on_like_added`（通知を送信）

`on_like_added` が先に実行されて、いいね数がまだ0のまま通知が送られてしまう問題が発生。

解決策：トリガー名を `zz_notify_on_like` に変更。アルファベット順で最後に実行されるようにしました。地味だけど、ハマるポイントです。

翻訳のキャッシュ戦略

無料のMyMemory APIにはレート制限があるため、一度翻訳した内容はlocalStorageにキャッシュ。同じ投稿を再翻訳しないようにしました。

```
// キャッシュキー: "ja:en:こんにちは" → "Hello"
const cacheKey = `${sourceLang}:${targetLang}:${text}`
```
