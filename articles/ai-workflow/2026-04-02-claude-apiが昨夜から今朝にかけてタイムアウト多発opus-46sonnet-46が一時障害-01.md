「Claude APIへのリクエストがタイムアウトする」「Claudeのレスポンスが返ってこない」「アプリに組み込んだAIが突然動かなくなった」——昨夜から今朝にかけて、Claude APIを使っている開発者や企業から障害報告が相次いだ。

AnthropicのステータスページによるとClaude Opus 4.6およびSonnet 4.6で日本時間の本日4月1日午前2時45分（UTC 2026年3月31日17:45）から午後2時52分（UTC 4月1日05:52）にかけて、リクエストのタイムアウト率が通常を大きく上回る障害が発生した。現在は解消済みだ。

さらに同日、Claude Haiku 4.5でもエラーと遅延が発生。こちらも現在は復旧している。

Claude APIを業務やサービスに組み込んでいた開発者・企業にとっては、深夜から朝にかけての数時間、AIの回答が得られない状態が続いた。本記事では今回の障害の詳細と、次回以降のAPI障害に備えるための実践的な対策を解説する。

## 今回の障害の詳細——何が起きたか

![](https://assets.st-note.com/img/1775089802-R5mNzsld0IF6BcvMytGZD17X.png?width=1200)

### 発生したモデルと時間帯

**Claude Opus 4.6・Sonnet 4.6**——日本時間4月1日午前2時45分〜午後2時52分（約12時間）にわたり、リクエストに対してタイムアウトエラーが高い割合で発生した。リクエストが正常に処理されず、APIから応答が返ってこない状態が続いた。

**Claude Haiku 4.5**——日本時間4月1日に独立したエラー・遅延の障害が発生し、午前11時14分（UTC 02:14）ごろに修正が実施され監視状態に入り、その後解消された。

**Claude.aiデスクトップアプリ**——3月31日には、Claude.aiのデスクトップアプリでコネクタ（連携サービス）が利用できなくなる別の障害も発生。組織のAllowlistからコネクタが削除される事象が確認され、こちらも現在は解消済みだ。

### 症状の具体的な現れ方

Claude APIを使ったシステムで発生した主な症状は次の通りだ。

APIへのHTTPリクエストが一定時間後に「タイムアウト（timeout）」エラーを返す。リクエスト自体は送れているが、レスポンスが返ってこない「待ち続ける」状態になる。ストリーミングモード（stream=true）で途中までしか応答が来ずに切断される。エラーコード529（Overloaded）や503（Service Unavailable）が返される。APIに組み込んだチャットボット・要約機能・文書生成などが動作しなくなる。

---

## なぜClaude APIで障害が起きるのか——技術的な背景

Claude APIはAnthropicのクラウドインフラ上で稼働している。大規模言語モデル（LLM）へのリクエスト処理は通常のWebサービスと比べて非常に大きな計算資源（GPU/TPUなど）を必要とするため、以下のような要因で障害が発生しやすい。

**急激なトラフィック増加**——多くのユーザーが同時に高負荷なリクエスト（長文生成・複雑な推論など）を送ると、処理能力の上限に達してタイムアウトが発生しやすくなる。

**インフラの更新・デプロイ**——モデルのアップデートや内部インフラの変更が引き金となって一時的な不安定状態になるケースがある。

**特定モデルへの集中**——今回のようにOpus 4.6・Sonnet 4.6という上位モデルに負荷が集中すると、Haiku 4.5などの軽量モデルへの影響は限定的になる場合がある（今回は両方に影響が出た）。

**共有インフラの連鎖影響**——認証基盤やAPIゲートウェイなどの共有コンポーネントに問題が起きると、複数モデル・複数サービスに同時に影響が出ることもある。

## Claude APIが使えないと分かったときの即時対応

### ステップ1：公式ステータスページで確認する

Claude APIへのリクエストが失敗し始めたらまず確認すべきは**Anthropicの公式ステータスページ**だ。

アドレスは **status.claude.ai** または **status.anthropic.com** だ。ここにアクセスすると、現在発生中の障害情報・影響を受けているサービス・復旧の見込みなどがリアルタイムで更新される。自分のコードやシステムに問題があるのかAnthropic側の問題なのかを最初に判断する材料になる。

### ステップ2：エラーコードで原因を判断する

Claude APIから返ってくるHTTPエラーコードは原因の判断に役立つ。

**529 Overloaded**——Anthropic側のサーバーが過負荷状態。しばらく待って再試行するしかない。**503 Service Unavailable**——サービス全体が一時的に利用不能。ステータスページを確認。**504 Gateway Timeout**——リクエストがタイムアウトした。処理が重いプロンプトの場合はプロンプトを短くすることも検討。**429 Too Many Requests**——レート制限（利用回数の上限）に達した。指数バックオフで再試行する。**500 Internal Server Error**——サーバー側のエラー。再試行で解消することが多い。

529と503はAnthropicのインフラ問題、429は自分のAPIの使いすぎで切り分けられる。

### ステップ3：フォールバック戦略を使う

障害中でも業務やサービスを継続するための「フォールバック（代替手段）」は事前に用意しておくのが理想だが、今できる応急措置として次のことを試せる。

**軽量モデルへの切り替え**——Opus 4.6が使えない場合、Haiku 4.5が動いているなら精度が多少落ちても軽量モデルで処理を続けることができる（今回は両方影響が出たが、障害のパターンによっては有効だ）。

**リクエストのキューイング**——処理を待機させてAnthropicの復旧後にまとめて実行する。急ぎでない処理はキューに積んでおき、復旧を待つ。

**ユーザーへの案内**——エンドユーザー向けのサービスに組み込んでいる場合、「現在AI機能が一時的に利用できません。しばらくお待ちください」などのメッセージを表示してユーザーを案内する。

## 次回の障害に備えるための実装対策

Claude APIを業務に組み込んでいる開発者向けに、障害時の影響を最小化するためのコーディング上の対策を整理する。

### リトライ処理と指数バックオフ

最も基本的かつ効果的な対策だ。APIが失敗したときに一定時間待ってから自動的に再試行する仕組みを入れる。待ち時間を1秒→2秒→4秒→8秒…と指数的に増やす「指数バックオフ」にすると、サーバー負荷への配慮と成功率のバランスが取れる。

```
import anthropic
import time

client = anthropic.Anthropic()

def call_claude_with_retry(prompt, max_retries=3):
    for attempt in range(max_retries):
        try:
            response = client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=1000,
                messages=[{"role": "user", "content": prompt}]
            )
            return response
        except anthropic.APIStatusError as e:
            if e.status_code in [429, 529, 503] and attempt < max_retries - 1:
                wait = 2 ** attempt  # 1秒→2秒→4秒
                print(f"エラー {e.status_code}：{wait}秒後にリトライ ({attempt+1}/{max_retries})")
                time.sleep(wait)
            else:
                raise
    return None
```

### タイムアウト値の明示的な設定

デフォルトのタイムアウト設定が長すぎると、障害時にリクエストが長時間「ぶら下がった」状態になり、後続の処理が詰まる。適切なタイムアウト値を明示的に設定しておく。

```
client = anthropic.Anthropic(
    timeout=httpx.Timeout(60.0, read=60.0, connect=10.0)
)
```

### ステータスページのアラート登録

status.claude.aiでメール通知やSlack通知を設定しておくと、障害発生時に自動でアラートが届く。「Subscribe to updates」から登録できる。

### 代替モデルのフォールバック設定

上位モデル（Opus）が使えない場合に自動的に下位モデル（Sonnet・Haiku）に切り替えるロジックを入れておくことで、障害時も精度を落としながら継続稼働できる。

## 今回の障害を振り返って——Claude APIの信頼性について

AnthropicはClaude APIのSLA（サービス水準合意）として99.9%の稼働率を目標としているとされているが、今回のように予告なしの障害が発生することはある。大規模言語モデルのAPIは、通常のWebサービスよりも一リクエストあたりの処理コストが大きいため、需要の急増やインフラ変更が障害の引き金になりやすいという特性がある。

OpenAI API・Google Gemini APIなど他のAI APIでも類似の障害は定期的に発生しており、「AI APIは時々落ちるもの」として設計上織り込んでおくことが重要だ。

重要な業務処理にClaude APIを使っている場合は、複数のAIプロバイダーを組み合わせる「マルチプロバイダー設計」も中長期的に検討する価値がある。
