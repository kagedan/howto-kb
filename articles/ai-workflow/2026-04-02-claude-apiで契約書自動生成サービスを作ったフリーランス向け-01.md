# Claude APIで契約書自動生成サービスを作った【フリーランス向け】

## はじめに：フリーランスエンジニアの契約書問題

フリーランスとして独立してから、地味にしんどいのが**契約書まわりの作業**です。

「業務委託契約書を送ってください」と言われるたびに、過去のテンプレートを引っ張り出して、社名・単価・納期・秘密保持の範囲などを手で書き換える。慣れてもこれが毎回30〜45分はかかる。法律的に問題ない表現かどうかも不安で、弁護士に確認を頼むほどでもないし…という状況。

周りのフリーランスエンジニアに聞いても「とりあえずネットで拾ったテンプレ使ってる」「クライアントに言われたものをそのまま受け入れてる」という声が多く、これは自分だけの課題ではないなと感じていました。

そこで作ったのが**LegalAI**（<https://legalai-mvp.vercel.app>）です。業務委託契約書・NDA（秘密保持契約）・利用規約を、必要事項を入力するだけで約30秒で生成するWebサービスです。バックエンドはAnthropic Claude APIを使って実装しました。

本記事では、このサービスの設計・実装・プロンプトエンジニアリングのポイントを、動くコードと一緒に解説します。

---

## アーキテクチャ概要

全体の構成はシンプルです。

```
ユーザー入力（フォーム）
    ↓
Next.js API Routes（TypeScript）
    ↓
Anthropic Claude API（claude-3-5-sonnet）
    ↓
Markdown → HTML変換 → PDF出力
```

フロントエンドはNext.js 14（App Router）、バックエンドはAPI Routesで完結しています。PDFへの変換には`react-pdf`を使用。DBは現時点では持たず、生成結果はクライアントサイドでローカルに保存する設計にしました（MVPのため）。

依存ライブラリは以下の通りです：


## プロンプト設計：法的文書生成のコツ

法的文書をLLMで生成するうえで最も重要なのは**プロンプトの構造化**です。「契約書を作って」という曖昧な指示ではなく、ドキュメントの型・条項の網羅性・文体の統一性を明示的に指定する必要があります。

実際に使っているシステムプロンプトのコア部分はこちらです：

```
// lib/prompts/contract.ts

export const buildSystemPrompt = (docType: 'nda' | 'service' | 'terms') => {
  const docConfig = {
    nda: {
      name: '秘密保持契約書（NDA）',
      requiredClauses: [
        '秘密情報の定義',
        '秘密保持義務',
        '目的外使用の禁止',
        '第三者への開示禁止',
        '契約期間および秘密保持期間',
        '契約終了後の処理',
        '損害賠償',
        '準拠法および管轄裁判所',
      ],
    },
    service: {
      name: '業務委託契約書',
      requiredClauses: [
        '業務内容の定義',
        '委託料および支払条件',
        '納期・検収',
        '権利帰属（著作権・知的財産権）',
        '再委託の可否',
        '秘密保持',
        '個人情報の取り扱い',
        '契約解除',
        '損害賠償の範囲と上限',
        '準拠法および管轄裁判所',
      ],
    },
    terms: {
      name: '利用規約',
      requiredClauses: [
        'サービスの定義と目的',
        '利用登録と資格',
        '禁止事項',
        '知的財産権',
        '免責事項',
        '個人情報の取り扱い',
        'サービスの変更・停止',
        '規約の変更',
        '準拠法および管轄裁判所',
      ],
    },
  };

  const config = docConfig[docType];

  return `あなたは日本法に精通した法務の専門家です。
以下のルールに従って${config.name}を生成してください。

【必須条項】
${config.requiredClauses.map((c, i) => `${i + 1}. ${c}`).join('\n')}

【フォーマット規則】
- 文書の冒頭に前文（当事者の定義）を置くこと
- 各条項には「第○条（タイトル）」形式の見出しを付けること
- 法的に適切な文語体（です・ます調ではなく「とする」「するものとする」）を使用すること
- 空白箇所には【　】形式のプレースホルダーを使用すること
- 最後に署名欄を設けること

【注意事項】
- 生成する文書はあくまでテンプレートであり、法的助言ではない旨を文書末尾に1行追記すること
- 日本法（特に民法・下請法・著作権法）に準拠した内容とすること`;
};
```

`requiredClauses`を配列で管理することで、文書タイプごとに必要な条項の網羅性を担保しています。これがないとモデルが条項を省略することがあります。

## API実装：Claude SDKとストリーミング

Next.jsのAPI Routeでの実装です。長文生成になるため、**ストリーミングレスポンス**を使ってUXを向上させています。

```
// app/api/generate/route.ts

import Anthropic from '@anthropic-ai/sdk';
import { NextRequest } from 'next/server';
import { z } from 'zod';
import { buildSystemPrompt } from '@/lib/prompts/contract';

const requestSchema = z.object({
  docType: z.enum(['nda', 'service', 'terms']),
  partyA: z.string().min(1).max(100),
  partyB: z.string().min(1).max(100),
  effectiveDate: z.string(),
  amount: z.string().optional(),
  deliverables: z.string().optional(),
  additionalNotes: z.string().max(500).optional(),
});

export async function POST(req: NextRequest) {
  const body = await req.json();
  const parsed = requestSchema.safeParse(body);

  if (!parsed.success) {
    return new Response(JSON.stringify({ error: 'Invalid input' }), {
      status: 400,
    });
  }

  const { docType, partyA, partyB, effectiveDate, amount, deliverables, additionalNotes } =
    parsed.data;

  const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

  const userPrompt = `以下の情報をもとに${docType === 'nda' ? 'NDA' : docType === 'service' ? '業務委託契約書' : '利用規約'}を生成してください。

- 甲（委託者）: ${partyA}
- 乙（受託者）: ${partyB}
- 契約発効日: ${effectiveDate}
${amount ? `- 委託料: ${amount}` : ''}
${deliverables ? `- 成果物・業務内容: ${deliverables}` : ''}
${additionalNotes ? `- 追加要件: ${additionalNotes}` : ''}

Markdown形式で出力してください。`;

  const stream = await client.messages.stream({
    model: 'claude-3-5-sonnet-20241022',
    max_tokens: 4096,
    system: buildSystemPrompt(docType),
    messages: [{ role: 'user', content: userPrompt }],
  });

  // ReadableStreamでフロントにストリーミング返却
  const readable = new ReadableStream({
    async start(controller) {
      for await (const chunk of stream) {
        if (
          chunk.type === 'content_block_delta' &&
          chunk.delta.type === 'text_delta'
        ) {
          controller.enqueue(new TextEncoder().encode(chunk.delta.text));
        }
      }
      controller.close();
    },
  });

  return new Response(readable, {
    headers: {
      'Content-Type': 'text/plain; charset=utf-8',
      'Transfer-Encoding': 'chunked',
    },
  });
}
```

Zodによるバリデーションを必ず挟むことで、不正な入力でAPIコストが無駄にかかるのを防いでいます。`max_tokens: 4096`は契約書の長さによっては不足することもあるので、用途に応じて調整してください。

## フロントエンド：ストリーミング受信とリアルタイム表示

API Routeからストリーミングで受け取った内容をリアルタイムで表示する実装です。

```
// hooks/useDocumentGeneration.ts

import { useState, useCallback } from 'react';

interface GenerationState {
  content: string;
  isLoading: boolean;
  error: string | null;
}

export const useDocumentGeneration = () => {
  const [state, setState] = useState<GenerationState>({
    content: '',
    isLoading: false,
    error: null,
  });

  const generate = useCallback(async (formData: Record<string, string>) => {
    setState({ content: '', isLoading: true, error: null });

    try {
      const res = await fetch('/api/generate', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(formData),
      });

      if (!res.ok) throw new Error('生成に失敗しました');
      if (!res.body) throw new Error('レスポンスボディがありません');

      const reader = res.body.getReader();
      const decoder = new TextDecoder();

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        const chunk = decoder.decode(value, { stream: true });
        setState((prev) => ({
          ...prev,
          content: prev.content + chunk,
        }));
      }

      setState((prev) => ({ ...prev, isLoading: false }));
    } catch (e) {
      setState((prev) => ({
        ...prev,
        isLoading: false,
        error: e instanceof Error ? e.message : '不明なエラー',
      }));
    }
  }, []);

  return { ...state, generate };
};
```

このカスタムフックを使うことで、コンポーネント側はシンプルに保てます。ストリーミングチャンクを受け取るたびに`setState`で`content`に追記し、リアルタイムで文書がタイプされていくようなUXを実現しています。

## 法的文書生成における注意点

技術的な話と同じくらい重要なのが、**法的文書をAIで生成する際のリスク管理**です。実装時に意識したポイントをまとめます。

**1. 免責表示の強制**  
システムプロンプトに「文書末尾に法的助言ではない旨を追記すること」と明示しています。これはUIでも別途表示しており、「このサービスは法的助言を提供するものではありません。実際の契約締結前に弁護士等の専門家にご確認ください」という文言を常時表示しています。

**2. 出力の後処理**  
生成された文書のうち、明らかに問題のある表現（例：賠償上限を設けない条項など）を検出するための簡易バリデーションを入れています。完全ではありませんが、最低限のチェックとして。

**3. 個人情報の最小化**  
ユーザーが入力した会社名・金額などはAPIに渡しますが、DBには保存しない設計（MVP段階）にしています。個人情報保護の観点から、ログにも出力しないよう注意しています。

**4. モデル選定**  
`claude-3-5-sonnet`を選んだ理由は、日本語の法律文書に必要な文語体・条文形式の精度が高かったためです。複数モデルでA/Bテストを行い、定性評価でSonnetが最も安定していました。

## コスト設計とレートリミット

APIコストは想定より安く抑えられています。1文書あたりの概算は以下の通りです：

* 入力トークン：システムプロンプト（約600トークン）＋ユーザー入力（約100〜200トークン）
* 出力トークン：契約書本文（約1,500〜3,000トークン）

claude-3-5-sonnetの価格（2024年11月時点）では、**1文書あたり約0.5〜1.5円**程度に収まります。無料枠でのサービス提供も十分現実的なコスト感です。

レートリミット対策として、API RouteにIPベースのレートリミット（1IP・1分あたり5リクエスト）を`upstash/ratelimit`で実装しています。

```
// middleware.ts（レートリミットの概略）

import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(5, '1 m'),
  analytics: true,
});

// API Routeの冒頭で呼び出し
const identifier = req.headers.get('x-forwarded-for') ?? 'anonymous';
const { success } = await ratelimit.limit(identifier);
if (!success) {
  return new Response('Too Many Requests', { status: 429 });
}
```

## まとめ

Claude APIを使った法的文書生成サービスの実装ポイントをまとめます。

* **プロンプト設計**：必須条項を配列で管理し、文書タイプごとに切り替えることで網羅性を担保
* **ストリーミング**：長文生成はストリーミングレスポンスでUXを大幅に改善できる
* **Zodバリデーション**：入力バリデーションを徹底してAPIコストの無駄遣いを防ぐ
* **法的リスク管理**：免責表示・個人情報の最小化・専門家への誘導は必須

特にプロンプトエンジニアリングの部分は、「何を含めるか」だけでなく「何を含めないか」の指示も重要で、実際に何十回もイテレーションしました。法的文書はハルシネーションが実害につながりやすいドメインなので、プロンプトの品質管理は通常のアプリ以上に慎重に行う価値があります。
