LLM に大量の構造化データを扱わせたいとき、最初にやりがちな設計があります。全部コンテキストに入れて見せる方法です。

自分も、AutoCAD 上で動作する AI エージェントを個人で開発する中で、まさにそれを試しました。ユーザーが自然言語で指示すると、図面上の線・円・文字・ブロック参照などを調査したり、条件に応じて編集したりするツールです。

![](https://static.zenn.studio/user-upload/22ce011fb42a-20260421.png)  
*AutoCAD 上で動作する AI エージェント*

開発を始めた当初、自分はかなり素朴にこう考えていました。

***図面の情報を全部 LLM に見せれば、賢く判断してくれるのでは?***

結論から言うと、この設計はやめました。LLM に大量の図面情報をそのまま見せる構成は、コスト・速度・精度の3点で厳しかったからです。

今回は、そのとき何を勘違いしていて、どう設計を変えたのかを書きます。

## まず前提: LLM は「全部覚えている賢い脳」ではない

ChatGPT 等の LLM と画面で会話していると、AI が過去のやり取りをずっと覚えていて、裏で図面ファイルもそのまま理解しているように見えるかもしれません。

ただ、アプリ開発で API として LLM を使う場合、実態はかなり違います。  
多くの場合、アプリ側が毎回こんな情報を送っています。

```
{
  "messages": [
    { "role": "system", "content": "あなたは CAD アシスタントです" },
    { "role": "user", "content": "赤い円を削除して" }
  ]
}
```

つまり、LLM 側が永続的に記憶しているというより、

* 必要なら過去履歴をアプリ側が毎回再送する
* 今回必要な情報を毎回文字列で渡す
* その入力を見て、その場で応答する

という構造です。API アプリ開発の文脈では、会話履歴を毎回渡さない限り、モデルは前回の内容を知りません。各リクエストは基本的に独立しています。

そして CAD の図面データも、そのまま LLM に「図面そのもの」として渡せるわけではありません。どんなオブジェクトがあるか、色は、半径は、レイヤは、座標は…といった情報を、最終的には**文字列として説明して渡す**必要があります。

そこで最初にやったのが、図面内オブジェクトを JSON にして全部見せる方式でした。

## 最初の構成: 図面情報を JSON で全部渡す

イメージとしてはこんな感じです。

```
{
  "objects": [
    { "id": "001", "type": "circle", "radius": 3.0, "color": "red" },
    { "id": "002", "type": "circle", "radius": 5.0, "color": "red" },
    { "id": "003", "type": "line", "length": 120.0, "color": "blue" }
  ]
}
```

これをプロンプトに含めて、

* 半径5未満の円を全部削除して
* 長い線だけ青くして
* 赤い文字を探して

のような指示を処理させようとしました。

最初は「お、いけるかも」と思いました。でも、実際の図面を扱い始めるとすぐ限界が来ました。

## 問題1: コンテキストがすぐ重くなる

実際の図面では、数個のオブジェクトでは済みません。  
数百件、数千件、場合によってはそれ以上が普通にあります。

それを全部 JSON にして送ると、入力トークン数は簡単に数万〜十数万規模まで膨らみます。

![](https://static.zenn.studio/user-upload/943cf21100a9-20260421.png)  
*AutoCAD のサンプル図面から生成したJSONの一部(1万行超)*

さらに言うと、LLM に毎回送られるのは図面データだけではありません。システムプロンプト(アプリの振る舞いを指示する文章)や、過去の会話履歴も毎リクエストに同梱されます。データ部分が重いと、それらと合わさって全体がさらに重くなる構造です。

その結果、

* 1 リクエストあたりの API コストが無視できない額になる
* レスポンスが遅くなる
* 会話を重ねるほど累積でつらくなる

「必要なものだけ送れば?」という話もありますが、**何が必要かを判断するために全体像が欲しい**場面は普通にあります。

…ただし、これはまだマシな問題でした。本当に厳しかったのは次です。

## 問題2: 条件抽出が思ったより不安定 (こちらが本題)

たとえば「半径5未満の円を全部削除して」と頼んだとき、こういう応答が返ってきたりします。

> 半径5未満の円を5件削除しました。

一見問題なさそうですが、実際に図面を数えてみると本来7件あった、みたいなことが普通に起こります。しかも LLM 自身は自信満々で「完了しました」と返してくるので、**ユーザー側からは気づきにくい**。これが厄介です。

これは LLM がダメというより、役割の問題です。

LLM は、自然言語の理解・意図の推定・要約・生成は強い。一方で、

* 大量の一覧を漏れなく走査する
* 厳密な数値条件で抽出する
* 件数を保証する

こういう処理は、決定論的な処理ほど得意ではありません。  
LLM は本質的に「それっぽく見える応答を返す」しくみなので、「7件全部抽出した」と自分で信じ込んで返してくることすらあります。

ユーザーから見ると、「**全部消してって言ったのに残ってる...**」となってしまいます。気づかないうちに取りこぼしが発生する構造は作業補佐のエージェントとして致命的です。

## 発想を変えた: LLM に全部やらせない

そこで設計を変えました。

**図面データそのものはアプリ側で保持して、LLM には全部見せない。**

代わりに、どんな要素があるかと、どう問い合わせればよいかだけを伝える方式です。

イメージとしては、AI に Web 検索をさせるときの構造に近いです。ネット上の全ページを AI に読ませているわけではなく、「検索する手段」だけを渡して、必要に応じて AI が自分で問い合わせる。それと同じで図面データも全部見せるのではなく、問い合わせる手段だけを渡す構成です。

具体的にはこんな流れです。

1. 図面情報はアプリ内で構造化して保持する (これは LLM に渡さない)
2. LLM には**どんな項目が検索できるか**だけを伝える
3. ユーザー指示に対して、LLM には**問い合わせ条件**を生成させる
4. 実際の抽出はアプリ側の決定論的な処理で行う

この流れを先ほどの例で見るとこうなります。「半径5未満の円を全部削除して」に対して、LLM は自分で対象を数えるのではなく、「type=circle かつ radius<5 に該当するものを抽出」という**条件だけ**を組み立てます。その先の抽出はアプリが厳密に処理します。

## 何が良くなったか

**軽くなった**

LLM に見せるのは全件データではなく構造だけ。  
大量の情報を持った JSON を毎回送る必要がなくなり、入力トークンは1桁以上減りました。

**正確になった**

条件判定を LLM の雰囲気処理ではなく、決定論的な検索に寄せられます。  
7件該当するなら7件、100件該当するなら100件、漏れなく探索ができるようになりました。

**役割分担が明確になった**

* LLM = 意図理解・条件組み立て
* アプリ = 厳密検索・実操作

この分担はかなり安定しました。

## 学んだこと

この経験から得た教訓は CAD やエージェントの文脈を離れても通用する一般原則だと思います。

**LLM に全部やらせようとするより、LLM が得意な部分だけ担当させるほうが強い。**

開発初期は、「LLM に全部見せれば賢くやってくれる」と思いがちです。でも実際は、**大量のデータを眺めて漏れなく抽出するタスクは LLM の得意分野ではない**。むしろ得意なのは、自然言語の要件を、構造的な問い合わせに翻訳する仕事のほうです。

今の自分のイメージは、**LLM は図面を見る人ではなく、図面に問い合わせる人** です。

この発想の切り替えは、CAD だけでなく、Excel 分析、業務マスタ検索、ログ解析、BIM / GIS 系など、構造化データを扱う多くの場面で同じように効くと思います。

## 補足: もちろん万能ではない

この方式も銀の弾丸ではありません。

* LLM が変な条件を組み立てることはあるので、検証機能や few-shot (出力例)は必要
* 項目設計(何をどう構造化するか)にはそれなりにドメイン知識が要る
* 複雑な条件(複数条件の組み合わせや高度な集計)は精度が落ちる

それでも、全部 JSON を見せる方式よりはかなり現実的でした。

同じように「AI に全部やらせれば解決する」と感じている人ほど、一度役割分担を見直してみると面白いかもしれません。

LLM は便利だけど自由すぎるので、ある程度こちら側で制御する仕組みは必要だと感じました。
