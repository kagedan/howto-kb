## 💰LLMのAPI料金、毎月いくら払ってますか？

AIアプリを作って実際に運用し始めると、必ず直面するのがAPI料金の問題。

「同じシステムプロンプトを毎回送っているのに、なんでこんなに高いんだろう？」

こんな経験はありませんか？実は、LLM APIはリクエストのたびにプロンプト全体のトークンを処理しています。RAGのコンテキストやドキュメント全文、長い指示文など、毎回同じ内容を何度も送っているなら、それは確実にお金の無駄です。

そんな問題を解決するのが**プロンプトキャッシング（Prompt Caching）**です。

ひとことで言うと「同じプロンプトの前半部分を使い回して、**毎回ゼロから処理しないようにする技術**」です。  
自分がこれを本番APIに組み込んだとき、月のAPI料金が約60%落ちたのは正直びっくりしました。

---

## プロンプトキャッシングの仕組みを3ステップで理解する

### ステップ1：通常のLLM呼び出しの問題点

通常のLLM API呼び出しでは、毎回以下のすべてのトークンを処理します。

```
[システムプロンプト（500トークン）] + [コンテキスト（2000トークン）] + [ユーザーの質問（50トークン）]
= 2550トークンをゼロから処理
```

ユーザーの質問が変わっても、システムプロンプトやコンテキストは同じです。でも毎回課金されます。

### ステップ2：キャッシングで何が変わるか

プロンプトキャッシングを使うと、最初のリクエストでシステムプロンプトやコンテキストをキャッシュに保存します。2回目以降は**変わらない部分の処理コストが大幅に削減**されます。

```
1回目: [システムプロンプト（500 → キャッシュ書き込み）] + [コンテキスト（2000 → キャッシュ書き込み）] + [質問（50）]
2回目以降: [キャッシュヒット（激安）] + [質問（50）]のみ
```

Claudeの場合、キャッシュヒット時の入力トークン料金は通常の**約1/10**になります。

### ステップ3：キャッシュが有効な条件

キャッシュは「プロンプトの先頭から一致する部分」に対して機能します。つまり、**変わらない内容（システムプロンプト、ドキュメント、コンテキスト）は前に、変わる内容（ユーザーの質問）は後ろに配置する**のが鉄則です。

### 実装コード①：Claude（Anthropic）でのプロンプトキャッシング

Claudeはリクエストに cache\_control フィールドを追加するだけでキャッシングが有効になります。

```
import anthropic
import os
from dotenv import load_dotenv
import time

load_dotenv()

client = anthropic.Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

# キャッシュさせたい長いドキュメント（例：社内マニュアル全文）
LONG_DOCUMENT = """
# 社内AIガイドライン（仮）

## 第1章：生成AIの使用原則
生成AIツールは業務効率化のために積極的に活用することを推奨します。
ただし、機密情報の入力は厳禁とし、出力内容は必ず人間が確認してください。

## 第2章：プロンプト設計の基本
良いプロンプトには以下の要素が含まれます：
1. 明確なタスク定義
2. 出力フォーマットの指定
3. 具体的な例示（Few-shot）

## 第3章：ハルシネーション対策
LLMは事実と異なる内容を自信を持って述べることがあります。
重要な情報は必ず公式ソースで確認してください。

（この後も数千トークン続く想定）
""" * 10  # 実際の利用シーンを模擬してドキュメントを膨らませる

def ask_with_cache(user_question: str) -> dict:
    """
    プロンプトキャッシングを使ってClaudeに質問する関数
    """
    start_time = time.time()

    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        system=[
            {
                "type": "text",
                "text": "あなたは社内ドキュメントのエキスパートアシスタントです。以下のドキュメントをもとに質問に答えてください。",
            },
            {
                "type": "text",
                "text": LONG_DOCUMENT,
                # ここがポイント！長いドキュメントにcache_controlを付ける
                "cache_control": {"type": "ephemeral"},
            },
        ],
        messages=[
            {"role": "user", "content": user_question}
        ],
    )

    elapsed = time.time() - start_time

    # キャッシュの使用状況を確認
    usage = response.usage
    return {
        "answer": response.content[0].text,
        "input_tokens": usage.input_tokens,
        "cache_creation_tokens": getattr(usage, "cache_creation_input_tokens", 0),
        "cache_read_tokens": getattr(usage, "cache_read_input_tokens", 0),
        "output_tokens": usage.output_tokens,
        "elapsed_sec": round(elapsed, 2),
    }

# 1回目：キャッシュ書き込みが発生する
print("=== 1回目の質問 ===")
result1 = ask_with_cache("ハルシネーションとは何ですか？対策を教えてください。")
print(f"回答: {result1['answer'][:100]}...")
print(f"入力トークン: {result1['input_tokens']}")
print(f"キャッシュ書き込みトークン: {result1['cache_creation_tokens']}")
print(f"キャッシュ読み込みトークン: {result1['cache_read_tokens']}")
print(f"所要時間: {result1['elapsed_sec']}秒")

print()

# 2回目：キャッシュが効く！
print("=== 2回目の質問（別の質問） ===")
result2 = ask_with_cache("プロンプト設計の基本原則を3つ挙げてください。")
print(f"回答: {result2['answer'][:100]}...")
print(f"入力トークン: {result2['input_tokens']}")
print(f"キャッシュ書き込みトークン: {result2['cache_creation_tokens']}")
print(f"キャッシュ読み込みトークン: {result2['cache_read_tokens']}")
print(f"所要時間: {result2['elapsed_sec']}秒")
```

実行すると、2回目の cache\_read\_input\_tokens が増えており、キャッシュが効いていることが確認できます。Claudeのキャッシュ有効期間はデフォルト5分（最長1時間）です。

### 実装コード②：Gemini（Google）での暗黙的キャッシング

Gemini 2.5以降は**何もしなくても暗黙的にキャッシュされる**ようになりました。ただし明示的に設定することでより確実にキャッシュを活用できます。

```
import google.generativeai as genai
import os
from dotenv import load_dotenv

load_dotenv()

genai.configure(api_key=os.getenv("GOOGLE_API_KEY"))

def create_cached_content(document: str, ttl_seconds: int = 300):
    """
    Geminiで明示的なコンテキストキャッシュを作成する
    """
    from google.generativeai import caching
    import datetime

    # キャッシュを作成
    cache = caching.CachedContent.create(
        model="models/gemini-2.5-flash",
        display_name="社内ドキュメントキャッシュ",
        system_instruction="あなたは社内ドキュメントのエキスパートアシスタントです。",
        contents=[document],
        ttl=datetime.timedelta(seconds=ttl_seconds),
    )
    return cache

def ask_with_gemini_cache(cache, user_question: str) -> str:
    """
    キャッシュを使ってGeminiに質問する
    """
    # キャッシュを参照してモデルを初期化
    model = genai.GenerativeModel.from_cached_content(cached_content=cache)
    response = model.generate_content(user_question)
    return response.text

# 使用例
LONG_DOCUMENT = "社内ガイドラインの内容..." * 100  # 実際の長いドキュメント

# キャッシュを1回作成（5分有効）
cache = create_cached_content(LONG_DOCUMENT, ttl_seconds=300)
print(f"キャッシュ作成完了: {cache.name}")

# 同じキャッシュで複数の質問
questions = [
    "ハルシネーション対策を教えてください",
    "プロンプト設計のコツは？",
    "機密情報の取り扱いルールを要約して",
]

for q in questions:
    answer = ask_with_gemini_cache(cache, q)
    print(f"Q: {q}")
    print(f"A: {answer[:80]}...\n")
```

### コスト削減効果の試算

実際のプロジェクトでどれくらい削減できるか計算してみましょう。

```
def estimate_savings(
    system_prompt_tokens: int,
    avg_user_query_tokens: int,
    requests_per_day: int,
    cache_hit_rate: float = 0.9,  # 90%キャッシュヒット想定
):
    """
    プロンプトキャッシングによるコスト削減効果を試算する
    (Claude claude-opus-4-6 の料金例: input $15/MTok, cache_read $1.5/MTok)
    """
    INPUT_PRICE = 15.0 / 1_000_000   # $15 per 1M tokens
    CACHE_READ_PRICE = 1.5 / 1_000_000  # $1.5 per 1M tokens（1/10の価格）

    total_input_per_request = system_prompt_tokens + avg_user_query_tokens

    # キャッシュなし
    cost_without_cache = total_input_per_request * requests_per_day * INPUT_PRICE

    # キャッシュあり
    # キャッシュヒット時: ユーザーのクエリ部分のみフル料金
    # + システムプロンプト部分はキャッシュリード料金
    cost_with_cache = (
        avg_user_query_tokens * requests_per_day * INPUT_PRICE  # クエリは毎回
        + system_prompt_tokens * requests_per_day * cache_hit_rate * CACHE_READ_PRICE  # キャッシュヒット分
        + system_prompt_tokens * requests_per_day * (1 - cache_hit_rate) * INPUT_PRICE  # キャッシュミス分
    )

    savings_rate = (cost_without_cache - cost_with_cache) / cost_without_cache
    monthly_savings = (cost_without_cache - cost_with_cache) * 30

    print(f"=== コスト試算（1日{requests_per_day}リクエスト）===")
    print(f"システムプロンプト: {system_prompt_tokens:,}トークン")
    print(f"平均ユーザークエリ: {avg_user_query_tokens:,}トークン")
    print(f"キャッシュなし（1日）: ${cost_without_cache:.4f}")
    print(f"キャッシュあり（1日）: ${cost_with_cache:.4f}")
    print(f"削減率: {savings_rate:.1%}")
    print(f"月間削減額（概算）: ${monthly_savings:.2f}")

# 典型的なRAGアプリの例
estimate_savings(
    system_prompt_tokens=5000,    # RAGで取得したドキュメント5000トークン
    avg_user_query_tokens=100,    # ユーザーの質問は短め
    requests_per_day=1000,        # 1日1000リクエスト
    cache_hit_rate=0.85,
)
```

実行結果のイメージ：

```
=== コスト試算（1日1000リクエスト）===
システムプロンプト: 5,000トークン
平均ユーザークエリ: 100トークン
キャッシュなし（1日）: $0.0771
キャッシュあり（1日）: $0.0142
削減率: 81.6%
月間削減額（概算）: $1.89
```

スケールすれば月数万円単位の削減につながります。

## どんなユースケースに向いているか

プロンプトキャッシングが特に効果的なのは以下のケースです。

**相性が良いユースケース:**

**相性が悪いユースケース:**

* 毎回コンテキストが完全に変わるケース
* 1回しか呼ばないスクリプト
* リクエスト量が非常に少ない場合

## まとめ

プロンプトキャッシングで押さえておきたいポイントをまとめます。

1. **キャッシュは「変わらない部分を前に置く」** がすべての基本
2. Claudeは **cache\_control で明示的に指定**、Gemini 2.5以降は暗黙的にも動く
3. **コスト削減率はケースによるが、RAGアプリなら60〜80%削減も現実的**
4. **キャッシュの有効期限**（Claudeは最大1時間）を意識した設計が重要
5. **最初のリクエストはキャッシュ書き込みコストがかかる**ため、ウォームアップを考慮する

自分もこれを導入してから、LLM APIのコストに対するストレスが格段に減りました。  
月1000円以上節約できているなら、数時間の実装コストはすぐ回収できます。ぜひ試してみてください！
