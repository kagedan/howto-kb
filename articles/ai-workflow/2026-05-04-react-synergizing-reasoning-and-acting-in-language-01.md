## 第1章：ReAct論文は何を解決しようとしたのか

LLMは、質問応答・要約・推論などで高い性能を示すようになりました。  
特に **Chain-of-Thought（CoT）** により、途中の考え方を出力しながら複雑な問題を解く能力が注目されました。

しかし、CoTには限界があります。

CoTは、LLMの内部知識をもとに「考える」ことはできますが、外部情報を確認しながら推論を修正することは苦手です。  
そのため、前提知識が古い・間違っている場合でも、もっともらしい推論を続けてしまい、ハルシネーションにつながることがあります。

一方で、検索・API呼び出し・Web操作など、LLMに「行動」させるアプローチもあります。  
ただし、行動だけでは「なぜその操作をするのか」「次に何を確認すべきか」といった計画や状態管理が弱くなります。  
整理すると、従来の方式には次のような課題がありました。

| 方式 | 強み | 課題 |
| --- | --- | --- |
| CoT | 段階的に推論できる | 外部情報に接地しにくい |
| Actionのみ | 検索や操作ができる | 計画・状態管理が弱い |

ReActは、この2つを組み合わせます。

```
Thought → Action → Observation
```

つまり、LLMがまず考え、必要に応じて外部ツールを使い、その結果を観測して、次の判断につなげる方式です。  
ReActの重要性は、LLMを単なる文章生成モデルではなく、**外部環境と相互作用しながらタスクを進めるエージェント**として扱った点にあります。

現在のAIエージェントで使われる、ツール呼び出し、RAG、Function Calling、MCPなどの考え方も、この「考える・行動する・観測する」というループの延長線上にあります。

この論文を理解することで、AIエージェントを**LLMにツールを接続しただけの仕組み**ではなく、**推論と行動を繰り返す制御ループ**として捉えられるようになります。

## 第2章：ReActの基本アイデア — Thought / Action / Observation

ReActの基本アイデアは、LLMに **「考える」と「行動する」を交互に実行させる** ことです。

従来のLLMは、ユーザーの質問に対して、その場で回答を生成する使い方が中心でした。  
一方、ReActでは、LLMが必要に応じて外部ツールを使い、その結果を見ながら次の判断を行います。

中心となる流れは、次の3つです。

| 要素 | 役割 | 例 |
| --- | --- | --- |
| Thought | 次に何をすべきか考える | 「まず関連情報を検索する必要がある」 |
| Action | 外部に対して行動する | 検索、API呼び出し、DB照会 |
| Observation | 行動結果を受け取る | 検索結果、APIレスポンス、DBの実行結果 |

この流れを簡単に表すと、以下のようになります。

```
Thought → Action → Observation → Thought → ...
```

例えば、質問応答タスクであれば、LLMはまず「何を調べるべきか」を考え、検索を実行し、検索結果を見て次の検索や回答方針を決めます。

重要なのは、**一度の推論で完結させない**ことです。  
外部から得られた情報をもとに、次の考え方や行動を更新していく点に、ReActの特徴があります。

この仕組みにより、LLMは内部知識だけに頼るのではなく、外部情報に接地しながらタスクを進められます。

つまりReActは、LLMを単なる回答生成モデルではなく、**観測結果を見ながら判断を更新するエージェント**として動かすための基本アイデアです。

## 第3章：Standard / CoT / Act-only / ReAct の違い

ReActを理解するには、従来のLLM活用方法との違いを整理すると分かりやすいです。

代表的な方式は、次の4つです。

| 方式 | 概要 | 強み | 課題 |
| --- | --- | --- | --- |
| Standard | 質問に対して直接回答する | シンプルで高速 | 複雑な推論に弱い |
| CoT | 推論ステップを踏んで回答する | 多段階の推論に強い | 外部情報を確認できない |
| Act-only | 検索や操作などの行動を行う | 外部情報を使える | 行動の理由や計画が弱い |
| ReAct | 推論と行動を交互に行う | 考えながら外部情報を使える | ループ制御やツール設計が必要 |

**Standard**は、もっとも基本的な使い方です。  
LLMがユーザーの質問に対して、そのまま回答を生成します。単純な質問には有効ですが、複雑な推論や外部情報の確認が必要な場合には限界があります。

**CoT**は、途中の考え方を使って回答する方式です。  
計算問題や論理問題のように、段階的な推論が必要なタスクに強くなります。一方で、LLMの内部知識だけで考えるため、前提が間違っていると誤った推論を続けてしまう可能性があります。

**Act-only**は、検索やAPI呼び出しなど、外部に対して行動する方式です。  
外部情報を取得できる点は強みですが、「なぜその行動を取るのか」「次に何をすべきか」といった計画や状態管理が弱くなりやすいです。

ReActは、これらの課題を踏まえ、**推論と行動を交互に組み合わせる方式**です。

```
Thought → Action → Observation → Thought → ...
```

LLMはまず考え、必要に応じて検索やAPIを実行し、その結果を見て次の判断を行います。  
つまりReActは、CoTの **「考える力」** と、Act-onlyの **「外部環境に働きかける力」** を組み合わせたアプローチです。  
この点が、現在のAIエージェントに近い考え方だと言えます。

## 第4章：ReActはなぜハルシネーションを減らせるのか

LLMのハルシネーションは、**外部確認をせずに、内部知識だけで回答を作ってしまう**ことで発生しやすくなります。

特にCoTでは、途中の推論ステップを生成できますが、その前提となる知識が間違っている場合、誤った情報をもとに推論を続けてしまうことがあります。

ReActでは、この問題に対して **Action** と **Observation** を使います。

```
Thought → Action → Observation → Thought
```

LLMが「何を確認すべきか」を考え、検索やAPI呼び出しを実行し、その結果を見て次の判断を更新します。

| 方式 | 回答の作り方 | ハルシネーションのリスク |
| --- | --- | --- |
| CoT | 内部知識をもとに段階的に考える | 前提が誤ると推論全体がずれる |
| ReAct | 外部情報を確認しながら考える | 観測結果で推論を補正しやすい |

つまり、ReActはLLMの推論を外部情報に接地させることで、 **「もっともらしい推測」ではなく「確認結果に基づく判断」** に近づけます。

ただし、ReActを使えばハルシネーションが完全になくなるわけではありません。  
検索結果の解釈ミス、誤ったツール選択、不十分な情報取得があれば、誤答は起こり得ます。

そのため実用システムでは、ReActに加えて、検索品質、ツール設計、根拠提示、回答検証の仕組みを組み合わせることが重要です。

## 第5章：実験結果から見るReActの効果

ReAct論文では、ReActの効果を確認するために、主に4つのベンチマークで評価しています。

| 分類 | ベンチマーク | 評価内容 |
| --- | --- | --- |
| 質問応答 | HotpotQA | 複数の情報を組み合わせて回答できるか |
| 事実検証 | FEVER | 主張が正しいかを検証できるか |
| 環境操作 | ALFWorld | テキスト環境で目的を達成できるか |
| Web操作 | WebShop | 条件に合う商品を探して購入判断できるか |

ポイントは、ReActが単に「回答精度を上げた」だけではなく、**タスクの種類によって効き方が異なる**ことです。

### 5.1 質問応答・事実検証：外部情報に接地できる効果

HotpotQAとFEVERでは、LLMがWikipedia APIを使いながら回答します。  
結果は以下の通りです。

| 手法 | HotpotQA | FEVER |
| --- | --- | --- |
| Standard | 28.8 | 52.6 |
| CoT | 36.4 | 56.8 |
| CoT-SC | 38.2 | 59.4 |
| Act | 28.2 | 57.5 |
| ReAct | 29.4 | 62.0 |
| CoT-SC → ReAct | 37.7 | 64.7 |
| ReAct → CoT-SC | 31.5 | 63.5 |
| Supervised SoTA | 67.5 | 89.5 |

この結果から、タスクごとに見え方が少し異なります。

HotpotQAでは、単体のReActは**29.4**で、CoT-SCの**38.2**を下回っています。  
つまり、**複雑な多段推論では、単に外部検索を使うだけでは十分ではなく、CoTのような深い推論も重要**です。

一方で、FEVERではReActが**62.0**となり、CoT-SCの**59.4**を上回っています。  
事実検証タスクでは、内部知識だけで推論するよりも、外部情報を確認しながら判断する効果が出やすいことが分かります。

特に注目すべきは、**CoT-SC** → **ReAct** です。  
HotpotQAで**37.7**、FEVERで**64.7**となっており、推論の強さと外部確認を組み合わせることで、より安定した性能が得られています。

ここから分かるのは、ReActはCoTを完全に置き換えるものではなく、**CoTの推論力に、外部情報への接地能力を加える考え方**だということです。

!

**HotpotQA**は、複数の文書や事実を組み合わせて答える必要がある**多段階質問応答ベンチマーク**です。  
例えば「Aという人物の出身地」と「その出身地に関係する別の情報」を組み合わせるように、1つの文だけでは答えられない質問を扱います。  
ReActでは、検索しながら必要な事実を集め、最終回答を導く力を評価するために使われています。

**FEVER**は、ある主張が**正しいか・誤っているか・判断不能か**を、根拠文書に基づいて判定する**事実検証ベンチマーク**です。  
例えば「ある人物が特定の映画に出演した」という主張に対して、Wikipediaなどの根拠を確認しながら真偽を判断します。  
ReActでは、LLMが内部知識だけで判断せず、外部情報を確認して事実に接地できるかを評価するために使われています。

### 5.2 ALFWorld：環境操作ではReActが大きく改善

ALFWorldは、テキストで表現された家庭内環境で、指定された目的を達成するタスクです。  
例えば、物を探す、移動する、加熱する、冷却する、といった操作を行います。

結果は以下の通りです。

| 手法 | Pick | Clean | Heat | Cool | Look | Pick 2 | All |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Act 平均 | 45 | 40 | 50 | 54 | 48 | 21 | 43 |
| ReAct 平均 | 71 | 60 | 82 | 75 | 38 | 30 | 59 |
| BUTLER best | 46 | 39 | 74 | 100 | 22 | 24 | 37 |

全体成功率を見ると、Actは43、ReActは59です。  
ReActはAct-onlyより **+16ポイント** 高い結果になっています。

さらに論文本文では、最良のReAct試行では平均成功率が**71%**に達し、Actの最良**45%**、BUTLERの\*\*37%\*\*を大きく上回ったと説明されています。

この結果は、ReActの強みをよく表しています。

環境操作では、単に次のアクションを出すだけでは不十分です。  
「今どこにいるのか」「何を持っているのか」「次に何をすべきか」を管理する必要があります。

ReActでは、Thoughtによってサブゴールを整理し、Observationによって環境状態を確認しながら行動を修正できます。

```
目的を理解する
  ↓
現在状態を確認する
  ↓
次の行動を考える
  ↓
実行結果を観測する
  ↓
方針を更新する
```

このため、ALFWorldのような環境操作タスクでは、ReActの効果が大きく出ています。

### 5.3 WebShop：Web操作でも既存手法を上回る

WebShopは、ユーザーの条件に合う商品をWeb上で探し、購入判断を行うタスクです。

結果は以下の通りです。

| 手法 | Score | Success Rate |
| --- | --- | --- |
| Act | 62.3 | 30.1 |
| ReAct | 66.6 | 40.0 |
| IL | 59.9 | 29.1 |
| IL + RL | 62.4 | 28.7 |
| Human Expert | 82.1 | 59.6 |

ReActのSuccess Rateは**40.0**で、Actの**30.1**を**約10ポイント**上回っています。  
また、模倣学習ベースのILや、模倣学習＋強化学習のIL+RLも上回っています。

これは重要です。

WebShopでは、商品名、価格、レビュー、条件一致など、複数の情報を見ながら判断する必要があります。  
ReActは、検索結果や商品情報を観測しながら、次に見るべき情報を考え直せるため、Act-onlyより安定してタスクを進められます。

### 5.4 数値から見えるReActの本質

実験結果をまとめると、ReActの効果は次のように整理できます。

| タスク | 数値上の特徴 | 解釈 |
| --- | --- | --- |
| HotpotQA | ReAct単体はCoT-SCを下回る | 複雑推論ではCoTの強さも必要 |
| FEVER | ReActがCoT-SCを上回る | 事実確認では外部情報への接地が有効 |
| ALFWorld | ReActがActを大きく上回る | 状態管理とサブゴール分解が効く |
| WebShop | ReActがAct / IL / IL+RLを上回る | 観測しながら判断を更新する効果がある |

つまり、ReActはすべてのタスクで常に単独最強というわけではありません。

むしろ重要なのは、**推論だけでは弱い場面、行動だけでは弱い場面を補完できる**ことです。

CoTは「考える力」を高めます。  
Actは「外部に働きかける力」を与えます。  
ReActは、その2つを組み合わせ、**考えながら行動し、観測結果をもとに判断を更新する**仕組みを作りました。

この点が、現在のAIエージェントにつながる最大のポイントです。

## 第6章：ReActから見るAIエージェントの基本アーキテクチャ

ReActの考え方は、現在のAIエージェントの基本アーキテクチャにそのままつながります。

AIエージェントは、単にLLMにツールを接続した仕組みではありません。  
本質は、**目的を理解し、次に何をすべきかを考え、外部ツールを使い、その結果を見て判断を更新する仕組み**です。

ReActの構成を、現在のAIエージェントに対応させると次のようになります。

| ReActの要素 | AIエージェントでの役割 |
| --- | --- |
| Thought | タスク理解、計画、次アクションの判断 |
| Action | 検索、API呼び出し、DB照会、コード実行 |
| Observation | ツール実行結果、検索結果、エラー、外部状態 |
| 反復ループ | 観測結果を踏まえた再計画 |
| Finish | 最終回答、レポート作成、業務処理の完了 |

この流れを図にすると、次のようになります。

重要なのは、AIエージェントが一度で答えを出すのではなく、**途中結果を見ながら処理を進める点**です。

例えば、データ分析エージェントであれば、次のような流れになります。

```
ユーザーの依頼を理解する
  ↓
必要なデータや条件を考える
  ↓
SQLを実行する
  ↓
結果を確認する
  ↓
追加集計や可視化が必要か判断する
  ↓
最終的な分析結果を説明する
```

これはまさに、ReActの **Thought → Action → Observation** のループです。  
現在のAIエージェントでは、この構造を実現するために、さまざまな技術が使われます。

| 技術要素 | ReActとの関係 |
| --- | --- |
| Function Calling | Actionを安全に構造化して実行する |
| RAG | Actionとして検索し、Observationとして関連文書を受け取る |
| MCP | 外部ツールやデータソースとの接続方式を標準化する |
| Memory | 過去の会話や作業状態を保持する |
| Guardrails | 不適切なActionや回答を制御する |
| Workflow Engine | 反復処理や業務フローを安定的に管理する |

つまり、ReActはAIエージェントの完成形ではありません。  
しかし、AIエージェントを理解するための**基本構造**を示しています。

現在のAIエージェントは、ReActの「考える・行動する・観測する」というループに対して、ツール管理、状態管理、安全制御、評価の仕組みを追加したものと捉えると理解しやすくなります。

まとめると、ReActを理解することで、AIエージェントを次のように捉えられます。

```
AIエージェントとは、LLMが外部環境と相互作用しながら、
推論・実行・観測・再計画を繰り返すシステムである。
```

この見方を持つと、RAG、Function Calling、MCP、Agent Frameworkといった技術も、個別の流行技術ではなく、ReAct的な制御ループを実用化するための部品として理解できます。

## 第7章：ReActの限界 — 実用システムでは何を追加すべきか

ReActは、AIエージェントの基本構造を理解するうえで非常に重要です。  
一方で、ReActの考え方だけで、そのまま実用システムを作れるわけではありません。

ReActは、LLMが **Thought → Action → Observation** を繰り返す仕組みですが、実運用では次のような課題が出ます。

| ReActの限界 | 実用システムで必要な追加要素 |
| --- | --- |
| どのツールを使うべきか迷う | Tool Router / Tool設計 |
| 不要なActionを繰り返す | ステップ数制限、停止条件 |
| Observationを誤って解釈する | 検証ロジック、根拠確認 |
| 危険な操作を実行する可能性がある | 権限制御、承認フロー |
| 状態を長く保持しにくい | Memory、State Store |
| 出力品質が安定しない | 評価、ログ、Human Review |

特に重要なのは、**ReActは「考えて動く」枠組みであって、「安全に・確実に・業務利用できる」仕組みまでは含んでいない**という点です。

例えば、社内システムに接続するAIエージェントでは、検索だけでなく、データ更新、チケット作成、メール送信、クラウド操作などを行う可能性があります。  
この場合、LLMの判断だけでActionを実行させると、誤操作や権限逸脱のリスクがあります。

そのため、実用システムでは次のような設計が必要になります。

```
LLMの判断
  ↓
ツール選択
  ↓
権限チェック
  ↓
必要に応じて人間の承認
  ↓
Action実行
  ↓
結果確認・ログ記録
```

また、ReActのループは便利ですが、制御しないと同じ検索を繰り返したり、不要な試行を続けたりする可能性があります。  
そのため、最大ステップ数、タイムアウト、失敗時のリトライ方針、停止条件を明確にしておく必要があります。

実用化に向けては、ReActの上に以下のような仕組みを重ねることが重要です。

| 追加設計 | 目的 |
| --- | --- |
| Tool Schema | LLMが安全にツールを呼び出せるようにする |
| Guardrails | 禁止操作・危険操作・不適切回答を防ぐ |
| Memory | 会話履歴や作業状態を保持する |
| Workflow | 決まった業務手順を安定的に実行する |
| Observability | どのThought / Action / Observationで失敗したか追跡する |
| Evaluation | エージェントの成功率・安全性・回答品質を評価する |

つまり、ReActはAIエージェントの**原型**です。  
しかし、業務システムとして使うには、周辺に制御・安全・状態管理・評価の仕組みを追加する必要があります。

まとめると、実用AIエージェントは次のように考えると分かりやすいです。

```
ReActが示したのは「考えながら行動する」基本ループ。
実用システムでは、そのループを安全に制御し、業務品質として運用できる仕組みが必要になる。
```

この視点を持つと、Function Calling、MCP、Guardrails、Memory、Workflow Engine、Agent Evaluationといった技術が、なぜ必要になるのかを理解しやすくなります。

## 第8章：ReAct以降のAIエージェント進化

ReActは、AIエージェントの完成形ではありません。  
しかし、**LLMが「考える・行動する・観測する」を繰り返す**という基本構造を示した点で、現在のAIエージェント技術の出発点として理解できます。

ReAct以降、AIエージェントは次のような方向に進化してきました。

| 進化の方向 | 内容 | ReActとの関係 |
| --- | --- | --- |
| Tool Use | LLMが検索・API・DB・コード実行を使う | Actionの実用化 |
| Function Calling | ツール呼び出しを構造化する | Actionを安全・確実に実行する仕組み |
| RAG | 外部文書を検索して回答に使う | Observationとして外部知識を取り込む |
| MCP | 外部ツールやデータソースとの接続を標準化する | Action先を広げる接続基盤 |
| Agent Framework | 推論・ツール実行・停止条件を管理する | ReActループを実装しやすくする |
| Memory | 会話履歴や作業状態を保持する | Thoughtの前提情報を増やす |
| Guardrails | 危険な操作や不適切回答を制御する | Actionの安全性を高める |

特に重要なのは、**Tool Use / Function Calling** です。  
OpenAIのTool Callingでは、モデルに利用可能なツールを渡し、モデルがツール呼び出しを返し、アプリケーション側で実行し、その結果を再びモデルに渡す、という複数ステップの流れが説明されています。これは、ReActの **Action** → **Observation** → **次の判断** に近い構造です。

また、**MCP（Model Context Protocol）** は、AIアプリケーションが外部システムに接続するための標準化された仕組みとして登場しました。MCPの公式説明では、AIアプリケーションをローカルファイル、データベース、検索エンジン、ワークフローなどに接続するためのオープン標準とされています。

さらに、LangChainのようなAgent Frameworkでは、エージェントを「言語モデルとツールを組み合わせ、タスクを推論し、使うべきツールを決め、反復的に解決へ進む仕組み」と説明しています。これは、ReActが示したループを、実装しやすい形に整理したものと捉えられます。

整理すると、ReAct以降の進化は、次のように見ると分かりやすいです。

```
ReAct
  ↓
考える・行動する・観測するループを提示
  ↓
Function Calling / Tool Use
  ↓
Actionを構造化して実行
  ↓
RAG / MCP / Agent Framework
  ↓
外部知識・外部システム・実行制御と接続
  ↓
実用AIエージェントへ
```

つまり、現在のAIエージェント技術は、ReActの考え方をベースに、**接続性・安全性・状態管理・実行制御**を強化してきたものだと言えます。

ReActを理解しておくと、RAG、Function Calling、MCP、Agent Frameworkといった技術を個別の流行としてではなく、**LLMを外部環境と相互作用させるための部品**として整理できます。

## 第9章：業務システムにReActを適用するならどう考えるか

ReActは、業務システムにAIエージェントを組み込む際の考え方としても有効です。  
ただし、業務適用では「LLMに自由に考えさせて自由に操作させる」のではなく、**実行できる行動を設計し、制御された範囲で判断させる**ことが重要です。

業務システムに当てはめると、ReActの流れは次のようになります。

```
業務依頼を理解する
  ↓
必要な情報・操作を考える
  ↓
検索、DB照会、API実行などを行う
  ↓
結果を確認する
  ↓
次の判断や追加確認を行う
  ↓
回答・レポート・業務処理を完了する
```

例えば、社内問い合わせ対応では、ReActは次のように機能します。

| ReActの要素 | 業務システムでの例 |
| --- | --- |
| Thought | ユーザーの質問意図を整理する |
| Action | 社内文書、FAQ、チケット、DBを検索する |
| Observation | 検索結果や関連文書を取得する |
| Thought | 根拠を確認し、回答方針を決める |
| Finish | 根拠付きで回答する |

データ分析業務でも同様です。  
ユーザーが「先月の異常傾向を見たい」と依頼した場合、AIエージェントは、対象データ、期間、集計軸、確認すべき指標を考え、SQLを実行し、結果を見て追加分析の必要性を判断します。

この場合、ReActは単なるSQL生成ではありません。

```
何を分析すべきかを考え、
必要なデータにアクセスし、
結果を解釈し、
次に確認すべき観点を再計画する仕組み
```

として捉えるべきです。

一方で、業務システムでは安全性と再現性が非常に重要です。  
そのため、ReActをそのまま使うのではなく、次のような設計を追加する必要があります。

| 設計観点 | 内容 |
| --- | --- |
| ツール制御 | 実行可能なAPI・SQL・操作を限定する |
| 権限制御 | ユーザー権限に応じて参照・更新範囲を制限する |
| 承認フロー | 更新・送信・削除などは人間確認を挟む |
| ログ管理 | どの判断で、どのActionを実行したか記録する |
| 根拠提示 | 回答に使った文書・データ・SQL結果を示す |
| 評価 | 回答品質、成功率、安全性を継続的に確認する |

特に、DB更新、メール送信、チケット作成、クラウド操作のような業務アクションは、LLMの判断だけで実行させるのではなく、ルール・権限・承認を組み合わせて制御する必要があります。

まとめると、業務システムにおけるReActの適用ポイントは次の通りです。

```
ReActは、業務AIエージェントを
「質問に答えるチャット」から、
「業務目的を理解し、必要な情報を取りに行き、結果を見て次の処理を判断する仕組み」
へ発展させるための基本思想である。
```

そのうえで、実用化には、ツール設計、権限制御、承認フロー、ログ、評価の仕組みを組み合わせることが不可欠です。

## 第10章：まとめ — ReActを理解するとAIエージェントの見方が変わる

ReActの本質は、LLMに「考えさせる」ことだけではありません。  
また、検索やAPIを「使わせる」ことだけでもありません。

重要なのは、次のループを作った点です。

このループによって、LLMは内部知識だけで答えるのではなく、外部情報や環境の状態を確認しながら、次の判断を更新できます。

つまりReActは、LLMを単なる文章生成モデルではなく、**外部環境と相互作用しながらタスクを進めるエージェント**として捉える考え方です。

現在のAIエージェントで使われる、RAG、Function Calling、Tool Use、MCP、Memory、Guardrailsなどの技術も、この考え方の延長線上にあります。

| 技術 | ReActとの関係 |
| --- | --- |
| RAG | 外部情報を取得し、回答を根拠に接地する |
| Function Calling | Actionを構造化して実行する |
| Tool Use | 検索・API・DBなど外部操作を可能にする |
| MCP | 外部ツールやデータソースとの接続を標準化する |
| Memory | 過去の状態や文脈を保持する |
| Guardrails | 危険な行動や不適切な回答を制御する |

ReActを理解すると、AIエージェントを「LLMにツールを接続したもの」としてではなく、**推論・実行・観測・再計画を繰り返すシステム**として見られるようになります。

この見方は、業務システムにAIエージェントを適用するうえでも重要です。

業務システムでは、単に回答を生成するだけでなく、必要なデータを探し、APIを実行し、結果を確認し、次の処理を判断する必要があります。  
そのため、ReActの考え方は、業務AIエージェントを設計するうえでの基本思想になります。

最後に、ReActの意義を一言でまとめると、次の通りです。

```
ReActは、LLMを「答えるモデル」から、
「考えながら行動し、結果を見て判断を更新するエージェント」へ発展させるための重要な考え方である。
```

この視点を持つことで、現在のAIエージェント技術を、個別の流行技術ではなく、**LLMを外部環境と接続し、業務タスクを遂行するためのアーキテクチャ要素**として整理できるようになります。
