## 概要

この動画は、Mastra の新機能 Signals を使うと、**長時間動作しているエージェントを止めずに、新しい入力や新しい文脈を途中から差し込める**ことを、短いデモで見せる内容です。扱っている API 自体は `subscribeToThread` と `sendMessage` というシンプルなものですが、動画が示しているのは単なるチャット継続の便利機能ではありません。長時間エージェント、複数ユーザー参加、外部イベント連携という、これからの実運用で避けて通れない課題に対して、Mastra が interaction model を作り替え始めたことが分かるデモです。

Alex はまず、2 つのターミナルから同じ deep search agent を見ながら、片方のユーザーが進行中の検索テーマを途中で切り替える様子を見せます。そのうえで、Signals はユーザー入力だけでなく、processor から送る reactive signals や、GitHub・監視アラートのような外部システムからの通知にも使えると説明します。短い動画ですが、Signals の価値を「request/response を少し柔らかくした API」ではなく、**long-running agent を multiplayer かつ event-driven にするための基盤**として理解できる内容です。

Agent Signals についての解説記事はこちらをご覧ください。  
<https://zenn.dev/shiromizuj/articles/e91394dd201039>

## 要点

1. Signals を使うと、長時間動いているエージェントを止めずに、新しい入力や新しい文脈を途中から与えられます
2. `subscribeToThread` と `sendMessage` が分離されたことで、エージェントを見るクライアントと、入力を送るクライアントを切り離せます
3. デモでは、2 人のユーザーが同じ deep search agent を見ながら、検索テーマを途中で変更する multiplayer 的な振る舞いを示しています
4. Signals はエージェント定義そのものを変えるのではなく、エージェントとの interaction を変える仕組みとして導入されています
5. メッセージに任意属性を付けられるため、multiplayer 環境で「誰が何を言ったか」をモデルへ渡せます
6. pub/sub を組み合わせることで、複数プロセスや複数クライアント間で同じスレッドのイベントを共有できます
7. reactive signals を使うと、processor が agent loop の状況を見ながら、必要なときだけ追加指示を差し込めます
8. notification signals によって、GitHub コメントや障害アラートのような外部イベントでエージェントを起こしたり方向修正したりできます
9. これは human-in-the-loop の改善だけでなく、より自律的な long-running agent を作る方向にもつながります

## 詳細

### Signals が変えるのは「エージェントの中身」ではなく「付き合い方」

この動画でいちばん重要なのは、Alex が何度も「Signals は agent definition を変えない、interaction を変える」と強調している点です。エージェントの instructions や tools を書き換えて、別物のエージェントを作る話ではありません。すでに動いている agent loop に対して、どうやって後から文脈を送るか、その I/O モデルを作り直しているのです。

ここは実務上かなり大きな差になります。従来の request/response 的なエージェントでは、誰かが `agent.stream()` や `agent.generate()` を呼び出した時点で、入力送信とストリーム購読がほぼ一体化していました。つまり「最初に話しかけた人」がその run の主導権も観測権も握る設計になりやすかったわけです。ですが、長時間動作するエージェントではその前提がすぐ破綻します。運用画面から様子を見たい人、Slack から補足を送りたい人、途中から状況確認したい人が別々に存在するからです。

Signals はその結びつきを外します。`subscribeToThread` でまず thread を購読し、`sendMessage` で必要なときに入力を送る。この分離だけで、長時間エージェントとの付き合い方がかなり変わります。

---

### デモの本質は「途中参加できる deep search agent」にある

動画前半では、Alex と Tyler という 2 人のユーザーが、同じ deep search agent を別ターミナルから見ています。最初に Alex が SpaceX の IPO ニュースを調べ始めると、その進行が両方の画面から見えます。そこへ Tyler が途中参加し、調査テーマを Anthropic のモデル障害へ変更します。

ここで面白いのは、入力が即座に agent loop の真ん中へ割り込むのではなく、**現在のステップが終わったあと、次のループ先頭で進行方向が変わる**ように見せている点です。これは単純に「どこからでも会話できます」というだけではなく、進行中のループと後から来る文脈を秩序立てて扱う設計を感じさせます。長時間エージェントでは、リアルタイム介入と実行一貫性を両立させる必要があるからです。

また、途中で「止まって」と伝えたり、「SpaceX について分かったことを要約して」と頼んだりできる様子も示されています。これによって、Signals が単なる継続会話ではなく、**実行中エージェントへの操舵インターフェース**だと見えてきます。

---

### subscribe と send が分かれると、multiplayer が自然になる

動画が伝えている multiplayer の価値は、2 人同時チャットができること自体ではありません。重要なのは、**あるユーザーが最初の起動に関与していなくても、途中から同じ thread に参加できる**ことです。これは運用システムではかなり本質的です。

たとえば、サポート担当が起動した調査エージェントを、数分後に SRE が見に来るケースを考えると分かりやすいです。あるいは営業担当が動かしている account research agent に、マネージャーが後から入って優先顧客の条件を追加するケースでも同じです。もし購読と送信が強く結びついたままだと、この途中参加は awkward になりがちです。Signals はそこを API レベルで自然にしています。

さらに、Alex は属性付きメッセージにも触れています。メッセージへユーザー名のような任意属性を付けられるので、モデル側で「誰が言ったか」を区別できます。これは multiplayer を単なる UI 上の多人数対応に留めず、**モデルの解釈対象になるコンテキスト**へ押し上げる重要なポイントです。

---

### reactive signals は「必要な瞬間だけ指示を差し込む」ための仕組み

動画後半では、Signals が単にユーザーから送る message だけで終わらないことが説明されます。特に面白いのが reactive signals の話です。Alex の説明では、Mastra の input processor が agent loop の各イベントを監視し、その状況に応じて signal を発行できます。

デモでは deep search agent を例に、`maxSteps` の上限が近づいたら「そろそろまとめに入れ」と動的に伝える使い方が紹介されます。これはかなり実践的です。long-running agent の失敗は、最初の system prompt へ全部の注意事項を詰め込んでしまうことから起きやすいからです。実際には、必要な指示は常に同じではなく、ある条件に達したときだけ効いてほしいことが多い。Signals はそれを、processor からの動的な介入として扱えるようにします。

この考え方は deep search だけでなく、コーディングエージェントや業務エージェントにも広く応用できます。危険な書き込みの前だけ安全指示を足す、ステップ上限直前だけ要約を促す、特定ツールの組み合わせでだけ補助知識を差し込む、といった設計が可能になるからです。

---

### notification signals によって、外の出来事でエージェントを起こせる

動画の終盤で Alex は、Signals を使うと GitHub や監視アラートのような外部イベントも agent loop へ接続できると説明します。ここは短い説明ですが、意味はかなり大きいです。

ひとつは Mastra Code の例です。pull request に CodeRabbit のコメントが付いたり、GitHub Actions からフィードバックが返ったりしたら、その出来事を signal としてコーディングエージェントへ送り、最新の状況を踏まえて次の行動候補を提示できるという話です。これは、単なる「PR を読む bot」ではなく、**PR をめぐる外部イベントを継続的に吸い込む harness** を作れることを意味します。

もうひとつは障害対応エージェントの例です。監視システムから「このサービスが落ちた」という signal を受けて起動し、調査と修復を進めている最中に、別のサーバー障害が発生したら、その追加情報をまた signal として受け取る。こうなるとエージェントは、最初に受け取った静的な問題設定で動くのではなく、**状況変化に追随しながら行動する運用主体**になります。Alex が「more autonomous long-running agents」と言っているのは、まさにこの方向です。

---

### これは request/response から thread-centered への移行だと見ると分かりやすい

この動画を announcement や docs と合わせて見ると、Mastra がやろうとしていることはかなりはっきりしています。エージェント interaction を、単発の request/response ではなく、**thread を中心にした継続的な実行モデル**へ寄せているのです。

thread があり、そこに複数クライアントが購読し、ユーザー入力も外部イベントも processor からの指示も同じ流れへ入ってくる。これによって、エージェントは「一回依頼して結果を待つ相手」から、「走り続ける作業単位」へ近づきます。Signals は派手な新モデルではありませんが、この構造変化を支える基礎部品としてかなり重要です。

短いデモ動画なのに印象が強いのは、Alex が API の説明だけで終わらせず、deep search、coding harness、incident response という 3 種類の具体例に落としているからでしょう。単なる新関数の紹介ではなく、**長時間エージェントの UX と実行モデルをどう設計するか**という論点まで見せている点に価値があります。
