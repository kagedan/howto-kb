## はじめに

LLMエージェントのベンチマークスコアは、ふつう「1回試して成功したか」（pass@1）で測られます。しかし本番運用で知りたいのは「**毎回成功するか**」のほうです。この2つの乖離を測る指標として、τ-benchが提案した **pass^k**——同一タスクをk回試行して**すべて成功する**確率——があります。

[前回の記事](https://zenn.dev/takkuhiro/articles/llm-agent-evaluation-benchmarks)でエージェント評価の設計原則を整理したとき、最初の知見に挙げたのがこのpass^kでした（未読でもこの記事だけで読めるように書いています）。今回は、それを自分の手で確かめます。

【やったこと】

* [τ²-bench](https://github.com/sierra-research/tau2-bench)（小売カスタマーサポートのエージェント評価ベンチマーク）のretailドメインを手元で動かした
* 20タスク × 8試行 × 2モデル（Claude Haiku 4.5 / Claude Sonnet 5）の計320会話を実行し、pass^kの減衰カーブを実測した
* 失敗した65会話から失敗の中身を分類した

結論の一部として、**1回なら85.6%成功するエージェントに「8回連続成功」を求めると50.0%まで落ちること**、そして**計測された「不安定さ」の約3分の1は、エージェントではなく評価システム側に起因していたこと**が分かりました。

この記事は全コマンドを掲載しているので、同じ実験を手元で再現できます（かかった費用は$62.72、ベンチマークの実行時間は合計約5時間で、詳細は記事末尾にまとめています）。

## τ²-benchとは

[τ-bench](https://arxiv.org/abs/2406.12045)（Sierra、ICLR 2025）は、ツールAPIを持つエージェントとLLMがシミュレートするユーザーとの対話でタスク完遂を評価するベンチマークです。詳しくは前回の記事に書きましたが、最大の貢献は **pass^k** ——同一タスクをk回独立に試行して **すべて成功する** 確率——という信頼性指標を提案したことです。

現在は後継の [τ²-bench](https://github.com/sierra-research/tau2-bench) リポジトリが正式版で、2026年3月の **τ³-Bench v1.0.0** で音声評価・knowledge検索ドメインの追加とともに、[75件以上のタスク修正](https://taubench.com/blog/tau3-task-fixes.html)（retailは26件）が入りました。今回使うのはこの修正済みretailドメイン（全114タスク）です。

なお日本語版としては、SB Intuitionsが2026年6月に [J-tau bench](https://github.com/sbintuitions/j-tau-bench) を公開しています（[ブログ記事](https://www.sbintuitions.co.jp/blog/entry/2026/06/19/100154)）。現時点ではtelecomドメインの翻訳ベースで、全モデルが英語版よりスコアが低く、特に小規模モデルは大幅に劣化するという結果が報告されています。retailドメインの日本語版は、私が調べた限りではまだ存在しません。

## 実験設定

| 項目 | 内容 |
| --- | --- |
| ベンチマーク | tau2-bench main（コミット `1901a30`、2026-07-01） |
| ドメイン | retail、先頭20タスク |
| 試行回数 | n=8（pass^1〜pass^8を推定。リーダーボード推奨は最低4） |
| エージェント | `anthropic/claude-haiku-4-5-20251001` / `anthropic/claude-sonnet-5` |
| ユーザーシミュレータ | `gemini/gemini-2.5-flash`（全条件で固定） |
| シード | 42、並列度4 |

設計上の判断を2つ書いておきます。

1. **ユーザーシミュレータはエージェントと別ファミリー（Gemini）に固定**しました。エージェントと同系列のモデルにユーザーを演じさせると自己選好の懸念が出るためです。安価なFlashにしたのはコスト上の判断です
2. **HaikuとSonnetの2段構成**にしたのは、「能力が上のモデルは信頼性も上なのか？」という問いを立てるためです。能力向上は信頼性をほとんど改善していないという先行研究（[arXiv:2602.16666](https://arxiv.org/abs/2602.16666)）への、ミニマムな検証でもあります

## セットアップ

```
git clone https://github.com/sierra-research/tau2-bench
cd tau2-bench
uv sync
```

`.env` にAPIキーを置きます（Python 3.12以上が必要ですが、uvが自動で用意してくれます）。

```
ANTHROPIC_API_KEY=...
GEMINI_API_KEY=...
```

動作確認は1タスク×1試行で以下のように実行できます。

```
uv run tau2 run --domain retail \
  --agent-llm anthropic/claude-haiku-4-5-20251001 \
  --user-llm gemini/gemini-2.5-flash \
  --num-tasks 1 --num-trials 1 --max-concurrency 1 \
  --save-to smoke-test --verbose-logs --seed 42
```

嬉しいことに、tau2-benchは実行ごとの **APIコストを自動計測** してくれます（スモークテストは1会話$0.06でした）。本実験の前に小さく走らせて、コストの当たりを付けてから規模を決めるのがおすすめです。

## ハマりどころ（OpenAI以外のモデルで動かす場合）

OpenAIのモデルで動かす場合、このセクションの罠は踏みません。Claude/Geminiで動かした私は3つ踏みました：

① **Claude Sonnet 5はtemperature非対応** → `--agent-llm-args '{}'` で回避（比較する全モデルで設定を統一）  
② **成功判定用のLLMが `gpt-4.1` にハードコード**されCLIから変更不可 → `config.py` を直接書き換え  
③ **判定をGeminiに差し替えるとJSONがコードフェンスで包まれ**パースに失敗 → フェンス除去パッチ

各ハマりどころの詳細と対応

**① Claude Sonnet 5はtemperatureに対応していない。** tau2-benchはデフォルトで `--agent-llm-args '{"temperature": 0.0}'` を渡しますが、Sonnet 5では `temperature` パラメータが廃止されており、全会話が `invalid_request_error` で失敗します。`--agent-llm-args '{}'` で回避できます。公平のため、比較する全モデルで同じ設定に統一してください（今回は両モデルともtemperature指定なし）。

**② 成功判定のLLMがOpenAIにハードコードされている。** retailの一部タスク（20タスク中5つ）は「NL assertions」という自然言語の成功条件を持ち、その判定にLLMを使います。このモデルが `src/tau2/config.py` に `gpt-4.1` とハードコードされていて、CLIから変更できません。OpenAIキーがない場合は直接書き換えます：

```
# src/tau2/config.py
DEFAULT_LLM_NL_ASSERTIONS = "gemini/gemini-2.5-flash"
DEFAULT_LLM_ENV_INTERFACE = "gemini/gemini-2.5-flash"
```

**③ Geminiに差し替えるとJSONパースで失敗する。** ②の差し替え後、GeminiがJSONをマークダウンのコードフェンス（```json）で包んで返すため、評価器の `json.loads()` が `Expecting value: line 1 column 1` で落ちます。`src/tau2/evaluator/evaluator_nl_assertions.py` の `json.loads(assistant_message.content)` の直前にフェンス除去を挟めば解決します：

```
content = assistant_message.content.strip()
if content.startswith("```"):
    content = content.split("```", 2)[1]
    if content.startswith("json"):
        content = content[len("json"):]
    content = content.strip()
result_data = json.loads(content)
```

（なお私が③に気づいたのはHaiku側の本実験を走らせた後だったため、影響を受けた5タスクのみ後から再実行し、集計にマージしています。集計スクリプトにマージ機能があるのはこのためです。この記事の手順通りパッチを先に当てておけば、この手間は不要です）

また③は、後処理で直すのではなく、Geminiの [Structured Output](https://ai.google.dev/gemini-api/docs/structured-output) で生成時点から正しいJSONを強制するアプローチもあります（tau2-benchはLiteLLM経由なので `response_format` を渡せるはずですが未検証です）。そもそも今回、判定モデルは「日常的に使っているから」という理由でGeminiに差し替えましたが、振り返ると、判定まわりはおとなしくデフォルトのOpenAIモデルを使っておけば②も③も踏まずに済んだ話ではあります。

なお、②と③は方法論上の注意でもあります。**成功判定のジャッジモデルを公式設定のGPT-4.1から変えているため、この実験の絶対値は公式リーダーボードと比較できません**。ただし両モデルとも同一ジャッジで判定しているので、モデル間の比較は内部的に一貫しています。前回の記事で「判定器の設計が評価の質を決める」と書きましたが、ベンチマークを動かし始めて最初に格闘する相手が判定器だとは思いませんでした。

## 本実験

```
uv run tau2 run --domain retail \
  --agent-llm anthropic/claude-haiku-4-5-20251001 --agent-llm-args '{}' \
  --user-llm gemini/gemini-2.5-flash \
  --num-tasks 20 --num-trials 8 --max-concurrency 4 \
  --save-to main-haiku --verbose-logs --seed 42

uv run tau2 run --domain retail \
  --agent-llm anthropic/claude-sonnet-5 --agent-llm-args '{}' \
  --user-llm gemini/gemini-2.5-flash \
  --num-tasks 20 --num-trials 8 --max-concurrency 4 \
  --save-to main-sonnet --verbose-logs --seed 42
```

実行時間は **Haikuが72分、Sonnetが3時間35分**。同じタスクでもSonnetは応答が長く丁寧な分、約3倍の時間がかかりました。

pass^kは実行完了時に自動で表示されますが、検算を兼ねて集計スクリプトも書きました（不偏推定量 \text{pass}^k = \mathbb{E}\_{\text{task}}[\binom{c}{k}/\binom{n}{k}] の実装。組み込み集計と一致することを確認済み）。全文を以下に折りたたんで置いておきます。この記事のグラフと表はすべてこのスクリプトの出力です。

pass^k集計と可視化スクリプト

```
"""tau2-benchのresults.jsonからpass^kを計算し、減衰カーブを描く。

使い方（tau2-benchディレクトリで実行。ラベルごとにカンマ区切りで複数ファイルをマージ可能）:
  uv run --with matplotlib python passk_analysis.py \
      "Haiku 4.5=data/simulations/main-haiku/results.json" \
      "Sonnet 5=data/simulations/main-sonnet/results.json" \
      --kmax 8 --out analysis
"""

import argparse
import json
from collections import defaultdict
from math import comb
from pathlib import Path

import matplotlib

# pyplotのimportより先にバックエンドを指定する（ヘッドレス環境対応）
matplotlib.use("Agg")
import matplotlib.pyplot as plt  # noqa: E402

SUCCESS_THRESHOLD = 0.999  # reward==1.0 を成功とみなす

COLORS = ["#2a78d6", "#eb6834"]
SURFACE = "#fcfcfb"
INK = "#1f1f1e"
INK_MUTED = "#6b6a63"

def load_run(paths: str):
    """カンマ区切りで複数のresults.jsonを受け取り、後のファイルが同一task_idを上書きする。

    （補完実行のマージ用。reward=Noneのシミュレーションは集計から除外するが、
    コストは実支出として合算する）
    """
    per_task = {}
    stats = {"agent_cost": 0.0, "user_cost": 0.0, "duration": 0.0, "n_sims": 0}
    for path in paths.split(","):
        data = json.loads(Path(path).read_text())
        file_tasks = defaultdict(list)
        for sim in data["simulations"]:
            stats["agent_cost"] += sim.get("agent_cost") or 0.0
            stats["user_cost"] += sim.get("user_cost") or 0.0
            reward = (sim.get("reward_info") or {}).get("reward")
            if reward is None:
                continue
            file_tasks[sim["task_id"]].append(float(reward) >= SUCCESS_THRESHOLD)
            stats["duration"] += sim.get("duration") or 0.0
            stats["n_sims"] += 1
        per_task.update(file_tasks)  # 後勝ちで上書き
    return per_task, stats

def pass_hat_k(per_task: dict, k: int) -> float:
    """不偏推定量: pass^k = E_task[ C(c,k) / C(n,k) ]"""
    vals = []
    for outcomes in per_task.values():
        n, c = len(outcomes), sum(outcomes)
        if n < k:
            continue
        vals.append(comb(c, k) / comb(n, k))
    return sum(vals) / len(vals) if vals else float("nan")

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("runs", nargs="+", help='"ラベル=results.jsonのパス" 形式')
    ap.add_argument("--kmax", type=int, default=8)
    ap.add_argument("--out", default="analysis")
    args = ap.parse_args()

    outdir = Path(args.out)
    outdir.mkdir(parents=True, exist_ok=True)
    ks = list(range(1, args.kmax + 1))

    results = {}
    lines = [
        "| k | " + " | ".join(r.split("=")[0] for r in args.runs) + " |",
        "|---|" + "---|" * len(args.runs),
    ]
    for run in args.runs:
        label, path = run.split("=", 1)
        per_task, stats = load_run(path)
        curve = [pass_hat_k(per_task, k) for k in ks]
        results[label] = {
            "curve": curve,
            "stats": stats,
            "per_task": {
                str(t): f"{sum(o)}/{len(o)}" for t, o in sorted(per_task.items())
            },
        }
        total = stats["agent_cost"] + stats["user_cost"]
        print(f"\n## {label}")
        print(f"  simulations: {stats['n_sims']}, tasks: {len(per_task)}")
        print(
            f"  total cost: ${total:.2f} (agent ${stats['agent_cost']:.2f} / user ${stats['user_cost']:.2f})"
        )
        print("  pass^k: " + ", ".join(f"k={k}: {v:.3f}" for k, v in zip(ks, curve)))

    for i, k in enumerate(ks):
        lines.append(
            f"| {k} | "
            + " | ".join(
                f"{results[r.split('=')[0]]['curve'][i]:.3f}" for r in args.runs
            )
            + " |"
        )
    (outdir / "passk_table.md").write_text("\n".join(lines) + "\n")
    (outdir / "passk_results.json").write_text(
        json.dumps(results, indent=2, ensure_ascii=False)
    )

    fig, ax = plt.subplots(figsize=(7.2, 4.6), dpi=200)
    fig.patch.set_facecolor(SURFACE)
    ax.set_facecolor(SURFACE)
    for (label, res), color in zip(results.items(), COLORS):
        ax.plot(
            ks,
            res["curve"],
            color=color,
            linewidth=2,
            marker="o",
            markersize=6,
            label=label,
            zorder=3,
        )
        ax.annotate(
            f"{label}  {res['curve'][-1]:.2f}",
            xy=(ks[-1], res["curve"][-1]),
            xytext=(8, 0),
            textcoords="offset points",
            va="center",
            color=INK,
            fontsize=10,
        )
    ax.set_xlabel("k (consecutive successes required)", color=INK)
    ax.set_ylabel("pass^k", color=INK)
    ax.set_title(
        "pass^k decay on tau2-bench retail (20 tasks, n=8 trials)",
        color=INK,
        fontsize=12,
        loc="left",
    )
    ax.set_xticks(ks)
    ax.set_ylim(0, 1.0)
    ax.grid(True, color="#e8e7e2", linewidth=0.8, zorder=0)
    for spine in ["top", "right"]:
        ax.spines[spine].set_visible(False)
    for spine in ["left", "bottom"]:
        ax.spines[spine].set_color(INK_MUTED)
    ax.tick_params(colors=INK_MUTED)
    ax.legend(frameon=False, loc="lower left")
    fig.tight_layout()
    fig.savefig(outdir / "passk_curve.png", bbox_inches="tight", facecolor=SURFACE)
    print(f"\nsaved: {outdir}/passk_curve.png, passk_table.md, passk_results.json")

if __name__ == "__main__":
    main()
```

## 結果: 減衰カーブが確認された

![pass^k減衰カーブ（Haiku 4.5 vs Sonnet 5、retail 20タスク、n=8）](https://static.zenn.studio/user-upload/deployed-images/b4b758f2cbf781061f1d0efd.png?sha=67ddf7a388e60fc66eb85f5a2e9484a1f38b6802)

| k | Haiku 4.5 | Sonnet 5 |
| --- | --- | --- |
| 1 | 0.738 | 0.856 |
| 2 | 0.577 | 0.754 |
| 3 | 0.471 | 0.679 |
| 4 | 0.398 | 0.622 |
| 5 | 0.345 | 0.579 |
| 6 | 0.305 | 0.545 |
| 7 | 0.275 | 0.519 |
| 8 | 0.250 | 0.500 |

**観察1: 減衰は理論通り、容赦なく起きる。** Haikuは1回なら73.8%成功しますが、「8回頼んで8回とも成功」は25.0%。奇しくも原論文のGPT-4o（pass^1約61%→pass^8 25%未満）とほぼ同じ水準です。2年経ってエージェントのpass^1は上がりましたが、「たまに失敗する」構造そのものは健在でした。

**観察2: この比較では、能力が上のモデルは信頼性も上だった。** Sonnet 5はpass^1で+12ポイント、pass^8では+25ポイントと、kが増えるほど差が開きました。単発性能の差は「連続で信頼できるか」の差として増幅されます。「能力向上は信頼性をほとんど改善しない」という先行研究の主張に対しては、少なくともこのモデルペア・このドメインでは能力差が信頼性差につながる、という結果です。ただし強調しておくと、**本実験はretailドメインの20タスクに限った小規模なもの**です。タスク単位のブートストラップでpass^1の差の95%信頼区間を出すと約+12ポイント [+0.0, +22.5] と、ぎりぎり有意といえる程度の水準でした。断定ではなく、一つの観察として受け取ってください。

**観察3: 素朴な予測より実測は高い。** pass^1が0.738なら 0.738^8 \approx 0.09 になりそうですが、実測のpass^8は0.250でした。これはタスクごとの難易度が不均一だからだと思われます（簡単なタスクは毎回成功し、難しいタスクは毎回失敗する。\mathbb{E}[p^k] \geq (\mathbb{E}[p])^k）。pass^kの減衰の傾きは「どれだけのタスクが**flaky**（成功したりしなかったりする）か」を反映します。今回、8回中1〜7回成功という中間的なタスクはHaikuで20中15、Sonnetで20中10ありました。

## 失敗した65会話の分類から分かったこと

失敗した全65会話（Haiku 42、Sonnet 23）のTrajectoryを、終了パターンや実行されたアクションと突き合わせて分類しました（分類にはAIを併用しています）。ここで予想外の発見がありました。

**失敗の35%は、エージェントのミスではありませんでした。** 典型例を日本語訳で示します（Sonnet、task 0。このタスクは8回中7回成功しています）：

> **エージェント**: 確認させてください。以下の交換を処理します。
>
> * メカニカルキーボード → 新しい商品（クリッキー、バックライトなし、フルサイズ）
> * スマートサーモスタット → 新しい商品（Google Assistant対応、黒）
> * 差額 $16.63 をクレジットカードに返金（5〜7営業日）
>
> 進めてよろしいですか？（yes/no）
>
> **ユーザー**: はい、お願いします。すべて対応してくれてありがとう！ **###STOP###**

エージェントは商品選択も差額計算も正しく、最終確認まで完璧に進めています。ところがユーザー（を演じるLLM）が「はい」と答えるのと**同時に会話終了トークンを発行**したため、エージェントは交換APIを実行する機会がないまま会話が終了。最終的なDB状態が期待と一致せず、「失敗」として記録されました。皮肉なことに、このエージェントは「不可逆操作の前にユーザーの確認を取る」という行儀の良い振る舞い——前回の記事で紹介した安全性次元 User Consent そのもの——をしたからこそ、この打ち切りに巻き込まれています。

この承諾直後の打ち切りというパターンは、失敗65件中23件。Haikuの失敗の40%、Sonnetの失敗の26%を占めていました。つまり、**pass^kが測っているのは「エージェント単体」ではなく「エージェント＋ユーザーシミュレータ＋判定系」の複合システムの信頼性**であり、今回の実験で言えば、観測された不安定さの約1/3は、私が選んだユーザーシミュレータ（Gemini Flash）の癖に由来する可能性があります。

これがGemini固有の挙動なのか、ユーザーシミュレータ一般の問題なのかは、シミュレータを差し替えて確かめる必要があります——これは次回検証します。

残りの約2/3は本物のエージェント起因でした。多かったのは、複数の変更依頼のうち1つを実行し忘れる、条件付きの要望（「クリッキーがなければバックライトなしでも可」）で誤った商品を選ぶ、返金先の支払い方法を取り違える、といった「惜しい」失敗です。全部できたのに最後の1手で崩れる様子は、まさに前回書いた「タスク成功率だけでは見えない」状況でした。

前回の記事で、LLM判定の弱点やτ²-benchの「ユーザー側の課題」（対話化で最大25ポイント低下）を紹介しましたが、ベンチマークをひとつ動かしただけで、その両方が自分の実験データの中に現れたことになります。**評価結果がどれだけ信用できるかは、測られるエージェントの出来と同じくらい、測る側——ユーザーシミュレータや判定器——の出来に左右される**ということです。

## まとめ

* pass^kの指数減衰は実在する。pass^1が0.86のエージェントでも、8回連続成功は0.50
* この実験では能力の高いモデルほど信頼性も高く、kが増えるほど差が開いた（ただし20タスクの小規模実験での観察）
* ただし、**測定された不安定さの約1/3は評価ハーネス（ユーザーシミュレータ）由来**だった。スコアを鵜呑みにせず、ハーネス起因の失敗を切り分ける仕組みが要る——導入時に失敗サンプルを点検するほか、「承諾直後の打ち切り」のような既知の癖をパターン検出して集計から分ける、タスクが監査されたVerified系ベンチマークを使う、といった対応が現実的です
* ベンチマークを動かす作業の実態は、エージェントとの格闘というより**評価系との格闘**だった（temperature廃止、ジャッジのハードコード、JSONパース）

なお、τ-bench系列の日本語版は現状J-tau bench（telecomのみ・翻訳ベース）にとどまっており、retailを含む他ドメインの日本語での評価はこれからの領域です。日本語エージェントの信頼性を測りたい場合も、当面は今回のように英語ベンチマークを動かして評価の仕組みと勘所を掴むのが現実的な入り口になりそうです。
