## はじめに

私はTRPGが好きです。

ただし、TRPGには現実的な壁があります。ゲームマスター（GM）を引き受けられる人材が必要で、参加者全員のスケジュールを合わせる必要があって、GMはセッションの準備に数時間を費やします。「やりたいな」と思ってから実際に卓が立つまでに数週間かかることも珍しくありません。

ならAIにGMをやらせればいい。

この一文から始まったプロジェクトの設計思想と実装、そして将来構想を解説します。なお動機は純粋な知的好奇心であり、実用性については追々考えます。

  

## なぜ既存のTRPGシステムを使わなかったのか

「CoC（クトゥルフ神話TRPG）やD&Dのルールブックをそのままプロンプトに突っ込めばいいのでは？」

真っ先に思いつく方法ですし、そういうアプローチを取っているプロジェクトも存在します。しかし私はそれをしませんでした。理由は主に二つです。

**一つ目は権利の問題です。**  
市販のTRPGシステムのルール文書をLLMのコンテキストに丸ごと投入することは、著作権侵害や規約違反に抵触する可能性があります。ルールを自分の言葉に書き直せばグレーゾーンを回避できるかもしれませんが、わざわざ地雷原に突っ込むくらいなら、自前でシステムを用意したほうが健全です。

**二つ目はシステム設計上の問題です。**  
既存のTRPGシステムは「人間のGMが運用すること」を前提に設計されています。人間の裁量に委ねた曖昧な記述や、対面の雰囲気に依存するルールが多く、AIに厳密に解釈させると破綻します。

最初からAIによる運用を前提としたシステムを自作する、というのが本プロジェクトの出発点です。

---

## システム全体像

本システムはDiscordボット、JSONデータベース、LLM（Claude Haiku）の三層で構成されています。

```
プレイヤーのDiscordメッセージ
  → メッセージキュー（1秒バッファリング）
  → コンテキスト構築（世界データ + プロンプト）
  → Claude Haiku API
  → コマンドパーサー（!で始まる行を抽出）
  → コマンドエグゼキュータ（JSONデータベースを更新）
  → DiscordへのWebhook送信（NPC・GMの発言）
```

プレイヤーの発言がLLMを経由して世界の状態変化に変換され、結果がDiscordに返ってくるサイクルです。

---

## 世界をJSONデータとして管理する

AIがGMを務めるうえで最大の課題は**状態管理**です。

LLMはステートレスです。セッションをまたぐと前回の出来事を覚えていません。NPCとの関係変化、プレイヤーが達成したこと、世界で起きた出来事——これらをLLMの内部記憶に依存することはできません。

解法はシンプルで、**世界の状態をLLMの外側に置く**ことです。このシステムでは世界のあらゆる状態をJSONファイルとして管理しています。

```
world/
├── meta.json              # 世界設定（時代・文化・法律・通貨など）
├── index.json             # 全データの高速参照用インデックス（自動生成）
├── characters/
│   ├── pc/                # プレイヤーキャラクター
│   └── npc/               # ノンプレイヤーキャラクター
├── locations/             # ロケーション（階層ネスト構造）
├── quests/                # クエスト
├── organizations/         # 組織・勢力
├── items/                 # アイテム（カテゴリ別）
└── scenes/                # 進行中のシーン
```

LLMはAPIコール毎にこのデータを受け取ることで、世界の現在状態を把握した上でGMを務めます。

### キャラクター（NPC）のデータ構造

NPCの一例です。personality（口調・一人称・性格）、relations（PC・NPCとの関係性）、shop（店舗在庫）、gm\_notes（GMだけが知る内部情報）などのフィールドを持ちます。

gm\_notesはプレイヤーには公開されない、GMだけが参照する設計図です。NPCの真の目的・秘密・行動原理をここに書かせることで、LLMがNPCを一貫性を持って動かせるようにしています。

### ロケーションの階層構造

ロケーションはディレクトリのネストで親子関係を表現します。

```
locations/
└── harbor/                # ハーバー区
    ├── info.json
    ├── bastian_port/      # バスティアン港
    └── cline_harbor/      # 環状線ハーバー駅
```

ファイルシステムのパスがそのままロケーションの地理的階層になります。「この建物はこの地区にある」という文脈が、ディレクトリ構造として自明に表現されます。

### インデックスによる高速参照

全JSONをそのままLLMに渡すのはトークンの無駄です。index.jsonとして世界データの概要を一枚のJSONに集約し、GMがまず全体像を把握できるようにしています。インデックスにはキャラクターの所在地・ロケーションの居住者・組織のメンバーなどが含まれており、いずれかのデータが更新されるたびに自動再構築されます。詳細が必要なときは!getコマンドで個別データを次のターンに取得できます。

## LLMの出力をコマンド言語として設計する

世界データをLLMに渡せるようになったとして、次の問題は「LLMの出力をどう世界に反映するか」です。

LLMに自然言語で返答させてそれを解釈するアプローチも考えられますが、自然言語の解釈には曖昧さが残ります。「アリスは荷物を受け取った」という文章から、JSONのどのフィールドを更新すべきかをルールベースで確実に導くのは困難です。

本システムでは、**LLMは自然言語を一切出力しません。** 返答はすべて **!**から始まるコマンドで構成されます。以下はLLMの出力の例です。

```
!chat scene042731 npc_mari "「昨夜から荷物が消えた。心当たりはないか？」"
!new quest id:missing_cargo title:消えた荷物 state:active location:harbor \
    hook:依頼人から荷物が消えたと聞いた discovered_by:[alice1234] \
    gm_notes:ラス・マレアスが横取り。倉庫B-7に隠されている。依頼人は実は密輸業者。
!set scene.scene042731.situation "マリから消えた荷物の依頼を受けた"
!add scene.scene042731.active_quests missing_cargo
!clock scene042731 2031031214
!log 2031031214 scene042731 harbor [alice1234] [missing_cargo] [] "マリから依頼発生"
```

コマンドは上から順に実行され、世界の状態を直接書き換えます。NPCが喋り、クエストが生まれ、タイムスタンプが刻まれ、ログが積まれます。LLMの「意図」が確実にデータベースに反映されます。

コマンドの種類は以下のとおりです。

* !chat — NPCのセリフ・GMの語りをDiscordに送信
* !dice — BCDiceを経由してダイスを振る
* !set / !add / !sub / !del — ドット記法でJSONフィールドを更新
* !new — NPC・ロケーション・クエスト・アイテムなどを新規作成
* !clock — シーンの時刻を進める
* !log — 出来事をイベントログに記録
* !get — コンテキスト外のデータを次のターンに取得

フィールドの指定はドット記法で行います（例：char.pc.alice1234.status.physical）。ネストされたJSONの任意の深さにアクセスでき、GMがどのデータを更新したかが判別できます。

## GMとして振る舞わせるためのプロンプト設計

コマンドを出力させるだけでは「良いGM」にはなりません。GMとして何をすべきか、何をしてはいけないかをシステムプロンプトで明確に定義しています。

**介入の閾値**  
プレイヤー同士の通常の会話ではGMは何も出力しません。NPCへの話しかけ・行動判定の発生・重要なイベント——これらのトリガーに対してのみ反応します。不要な場面で沈黙できることが、会話の自然なテンポを保つ条件です。

**PCへの不干渉**  
プレイヤーキャラクター（PC）のセリフ・行動・心理をGMが代わりにロールプレイすることは禁止しています。GMが動かしていいのはNPCだけです。明記しないと、LLMは善意でPCの行動を補完しようとします。余計なお世話です。

**ダイスの主権**  
PCに関わる行動判定は必ずプレイヤー本人に振ってもらいます。GMが振るのはNPC固有の判定とランダムイベントのみです。プレイヤーが自分でダイスを握る体験はTRPGの根幹のひとつであり、これを外すべきではないと考えています。

**クエスト管理**  
ストーリーの進行は必ずクエストとして記録させます。各クエストのgm\_notesには真相・解決方法・分岐・今後の展開を具体的に書かせます。曖昧な記述を禁じ、「PLが暗号を解読すれば正体を明かし報酬を渡す。暴力に訴えれば即退室」というレベルの具体性を要求します。これがないとLLMはNPCの行動に一貫性を持てません。

## Discordボットとしての実装

### なぜDiscordか

TRPGのオンラインセッションツールとしてはココフォリアが広く使われていますが、ココフォリアでは外部ボットを動かすことができません。本システムの根幹はボットによるリアルタイムな世界状態の書き換えであり、ボットが動かせない環境では成立しません。

DiscordはBot APIが充実しており、Webhookによる柔軟なメッセージ送信も可能です。オンセツールとしての機能は限定的ですが、本プロジェクトの要件を満たす現実的な選択肢として採用しました。

### キャラクター作成

キャラクター作成はDiscordのスラッシュコマンドで完結します。プレイヤーがコマンドを実行するとボットがDMで対話式の入力フローを展開し、名前・種族・職業・外見・性格などを入力していきます。作成したキャラクターはそのままJSONデータベースに登録され、セッションに参加できる状態になります。

### NPCの発言　Webhookによるなりすまし送信

!chatコマンドでNPCが発言する際、DiscordのWebhook機能を使い、**NPCの名前とアイコンを持つアカウントとして**メッセージを送信します。「GM\_Bot」という無機質な名義でNPCのセリフが流れるのではなく、チャンネル上に名前と顔写真付きのメッセージが表示されます。GMの語りはGMのアイコンで、各NPCはそれぞれの顔で喋ります。これにより、テキストチャットでありながらキャラクターと向き合う体験を実現しています。

### メッセージのバッチ処理

複数のプレイヤーがほぼ同時にメッセージを送信した場合、それらを1秒間バッファリングして1回のAPIコールにまとめて処理します。TRPGでは複数人が立て続けに発言することが多く、毎発言をAPIコールに変換するのはコスト・レイテンシの両面で非効率です。

### 複数シーンの並行稼働

Discordのチャンネルとシーンを1対1で対応させ、複数チャンネルを同時進行できます。Aチャンネルではプレイヤーが情報収集中、Bチャンネルでは別のプレイヤーが戦闘中——それぞれのシーンが独立してGMに処理され、出来事はすべて同一の世界DBに書き込まれます。

## なりきりチャット＋D20判定

本格的なTRPGシステムの前段階として、なりきりチャットに判定を付与しただけの軽量なシステムで検証を行います。

ダイスシステムは1D20判定を採用しています。難易度は5段階で、それぞれ成功閾値が異なります。

* 難易度1（容易）：5以上で成功 — 無抵抗の相手を脅す、など
* 難易度2（標準）：8以上で成功 — 検問を言い訳で通過する、など
* 難易度3（困難）：12以上で成功 — 武装した相手と戦う、など
* 難易度4（極難）：16以上で成功 — 主要NPCを暗殺する、など
* 難易度5（超難）：19以上で成功 — 組織を動かす、大規模な事件を起こす、など

出目20は完全成功、出目1は致命的失敗とし、得意分野への±2〜3の修正はGMが裁量で適用します。

ルール密度は意図的に低く抑えています。現段階のテスト目的は「AIがGMとして機能するか」の検証であり、複雑なルール処理の精度計測ではないからです。現在は私との1対1でのテストを行っています。思ったよりずっとちゃんとGMしています。

## 将来構想

### TRPG × MMO的運用

AIGMには人間のGMには物理的に不可能なことがあります。**24時間365日稼働**でき、**複数のシーンを同時進行**できます。

複数のプレイヤーがそれぞれ異なる場所で異なる出来事に巻き込まれながら、すべてが同一の世界DBの上で動く——これは設計としてすでに組み込まれています。あとはテスト人数を増やすだけです。

まず知り合いを数人呼んで本格的な多人数テストを行い、AIGMが複数プレイヤーを同時に捌けるか、世界の整合性を保ちながらシナリオを展開できるかを検証します。

### Foundry VTTへの展開

将来的にはFoundry Virtual Tabletop（FVTT）のボットとしても実装したいと考えています。FVTTはTRPGのオンラインセッションツールとして高機能であり、マップ・駒・ダイスロールUIなど、本システムが現状Discordで代替している部分を正面から解決できるプラットフォームです。FVTTを使えば、より高機能なボットとして実装できるようになるでしょう。

## おわりに

現段階は1対1のテストにすぎませんが、複数プレイヤーが同時に別シーンで動き、世界の状態が一枚のデータベース上で更新され続けるという設計はすでに実装済みです。

AIにTRPGのGMを任せることは、技術的には十分に実現可能だと判断しています。あとはそれが面白いセッション体験を生み出せるかどうか——その検証が次のステップです。
