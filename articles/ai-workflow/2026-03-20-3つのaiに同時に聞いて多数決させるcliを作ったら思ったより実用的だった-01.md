個人開発をしていると、「このアーキテクチャで本当にいいのか」という判断を一人で下し続けないといけない。

レビュアーがいないのは辛い。Xで聞くのも毎回できないし、ChatGPTに聞いても「いいと思います！」みたいな答えしか返ってこないことが多い。

じゃあ、複数のAIに同時に聞いて、多数決させたらどうだろう。

そう思って作ったのが **MAGI CLI** です。

## エヴァのMAGIシステムから着想した

エヴァンゲリオンに登場するMAGIシステムは、3台のスーパーコンピューターが多数決で重要な判断を下します。MELCHIOR、BALTHASAR、CASPAR。それぞれ異なる人格を持ち、意見が割れることもある。

これ、コードレビューに使えるじゃんと思いました。

| ユニット | モデル | 担当する視点 |
| --- | --- | --- |
| MELCHIOR | Gemini 2.5 Flash | 効率・パフォーマンス |
| BALTHASAR | Claude Opus | 可読性・保守性 |
| CASPAR | GPT-4o | エッジケース・新規性 |

Claude派の自分がGPTも使うのはちょっと複雑な気持ちでしたが、多様性のためには仕方ない。

## アーキテクチャ

## 実際の動き

```
$ magi "このコードのアーキテクチャを改善してください"
```

これだけで3つのLLMが並列でコードを読み、それぞれ改善案を出し、最後に多数決でVERDICTが出ます。

```
│  🧠 MELCHIOR (Gemini)   APPROVE
│     効率化の方向性は正しく、パフォーマンス影響なし
│
│  🧠 BALTHASAR (Claude)  APPROVE
│     可読性・保守性の観点でも問題なし
│
│  🧠 CASPAR (GPT-4o)     REJECT
│     テスト未整備のままリファクタリングするのは危険
│
│  ⚖️  VERDICT: APPROVED (2-1)
│
│  📋 マイノリティレポート:
│     テストが存在しない状態でのリファクタリングは、
│     既存の動作保証がなく本番リスクが高い。
```

2対1のとき、少数意見（マイノリティレポート）を必ず出すようにしました。多数決で決まったとしても、「CASPARがこういう理由で反対していた」という情報を残したかった。実際これが一番役に立ちます。

## 技術的なポイント

### 並列実行が重要

最初は順番に叩いていたんですが、これだと3倍かかる。`asyncio.gather` で並列にするのが正解でした。

```
async def run_parallel(task: str) -> list[AgentResult]:
    results = await asyncio.gather(
        run_melchior(task, []),
        run_balthasar(task, []),
        run_caspar(task, []),
    )
    return list(results)
```

体感で半分以下になりました。Claude Opusが一番遅いので、全体のレイテンシはそこに引っ張られますが、それでも十分速い。

### 人格付けが効く

最初は3つとも同じ答えを返してきて、「これ意味ないじゃん」と思ったんですが、システムプロンプトで性格を分けたら変わりました。

```
CASPAR_SYSTEM = """あなたはMAGIシステムの CASPAR-3 です。
役割: エッジケース・堅牢性・新規性の観点で判断する。
- エラーハンドリング、テスト、ドキュメント、運用上の注意点を確認する
- 見落としや代替案を必ず提示する"""
```

CASPARにこのプロンプトを渡すと、他の2つが通したものでも「テストは？」「エラーハンドリングは？」と突っ込んでくるようになります。これが地味に助かる。

### 多数決ロジックはシンプルに

```
def compute_verdict(verdicts: list[str]) -> tuple[str, str]:
    votes = [v for v in verdicts if v != "ERROR"]
    approve = votes.count("APPROVE")
    reject  = votes.count("REJECT")

    if approve >= 2:
        return "APPROVED", f"APPROVE {approve} - REJECT {reject}"
    if reject >= 2:
        return "REJECTED", f"REJECT {reject} - APPROVE {approve}"
    return "DEADLOCK", ...
```

2票以上で決定、全員バラバラならDEADLOCK。シンプルで良かった。最初はウェイトを付けたり複雑にしようとしたんですが、余計でした。

## 実際に役に立った場面

**技術選定**が一番使えました。

```
$ magi "RustとGoどちらでCLIを作るべきか。パフォーマンスより開発速度を重視したい"
```

MELCHIORとBALTHASARがGoを推し、CASPARだけRustを推した。VERDICTはGo。ただCASPARのコメントに「将来的にパフォーマンスがボトルネックになる可能性がある」とあって、それを念頭に設計するきっかけになりました。

一人でうんうん悩んでいるより、3方向から意見をもらった方が判断が早い。

## ファイルを実際に読んで修正もできる

テキスト回答だけでなく、コードを直接編集する機能も作りました。

```
TOOLS = [
    {"name": "read_file",   ...},
    {"name": "edit_file",   ...},
    {"name": "run_command", ...},
]
```

3つのエージェントがそれぞれファイルを読んで修正案を出し、APPROVED後のみ適用という流れです。間違った変更が勝手に適用されないよう、差分確認→適用を分けています。

## コストは思ったより安かった

1回の合議で約 **$0.05〜$0.15**（¥8〜22）です。Claude Opusが高いので、そこが大半を占めます。

毎日10回使っても月$30〜45くらい。3つのAPI代と考えると許容範囲かなと。

```
$ magi /cost
💰 セッションコスト: $0.23 (¥34)
```

セッション中の累計コストを `/cost` で確認できます。

## 反省点

正直、ツール機能（ファイル読み書き）をここまで作り込むつもりはなかったんですが、気づいたらv3.0になってました。個人開発あるある。

最初は「多数決でVERDICTを出すだけ」でよかったんです。シンプルに保てばよかった。

あと、Geminiのレスポンス形式がAPIバージョンで変わって、そこのハンドリングで数時間溶かしました。APIの変更は急に来るので注意。

## まとめ

* `asyncio.gather` で3LLM並列 → 直列の約1/3の時間
* システムプロンプトで人格を分ける → 同じ問いへの回答が分散する
* マイノリティレポートで少数意見を残す → これが一番実用的
* 思ったより機能が膨らんだ（反省）

一人で開発していてレビュアーがいない方、試してみる価値はあると思います。
