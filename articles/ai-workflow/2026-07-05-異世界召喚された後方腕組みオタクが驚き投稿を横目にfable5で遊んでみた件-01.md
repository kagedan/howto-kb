# はじめに

正直、LLMの性能がサチってきた感覚と共に、コスト面から見てもOpusモデルで十分では？という感覚がありました。

Twitterでは、「こんなアプリを作れるようになった！」という驚き投稿で溢れていましたが、Enterpriseの開発現場においてはそういったバイブ寄りの活用はほとんどありません。  
LLM利用コストの高騰を背景に、企業ではコスト管理の締め付けが日に日に強くなり、ここまで高コストなモデルっていつ使うのだろうと考えさせられます。  
ほとんどの企業が人件費削減を掲げているのにLLMの方が高いと悲鳴をあげていますね。

とはいえ、せっかくサブスクプランで7月7日までFable5を利用できるということで、肌触りを得るべく少しだけ遊んでみました。

# 検証

そこそこ規模が大きいかつ、自身が一定中身を理解しているリポジトリである必要があるということで、某スタートアップ政党の公開リポジトリを対象にfable5で遊んでみました。

### repository

### prompt

```
このリポジトリの大きなissueを与えます。
コードベースを詳細に調査し、「調査結果レポート」を出力してください。
なぜそういった対応が必要なのかなど、実行者が若手のエンジニアでもわかるように解説も含めてプランを組み立ててください。
** UTカバレッジ向上 **
UnitTestのカバレッジを向上させたいです。原則として以下のルールを守り、現在のコードでこれが守られていない部分についてもリファクタリング観点で言及してください。
- 純粋関数のユニットテストを増やしたい（必要に応じて純粋関数に切り出す）
- ローカルホストで動くものはモックせず、インテグレーションテストで動かす
- 外部APIは原則モックする

ultracode
```

### 感想

出力された内容の詳細は伏せますが、まず驚いたのは、約15分で$35程度のコストが飛んだという事実。高いですね。  
当然、35エージェントの並列実行処理によって生成されたアウトプット品質は高く、要修正対象の網羅性、変更妥当性の説明含めて、Opusモデルよりも考察深さが1つ2つ深い気がします。

そしてスコープ外ですが重要なセキュリティ課題を検出しレポートしてくれました。  
セキュリティ課題の検出における考察深さも、Opusモデルより優位性を感じます。

概ね、[公式ドキュメント](https://platform.claude.com/docs/ja/build-with-claude/prompt-engineering/prompting-claude-fable-5)通りの感想を得られた気がします。  
fable5、すごいですね。私も後方腕組みをほどいて、輪の中に混ざりに行こうと思いました。

# 総括

* セキュリティレビューでfableモデルを利用するのは有効そう
* 複雑な設計作業ももちろんfableモデルが品質は優位だと思いますが、消費Tokenあたりのコストを鑑みると利用は控えめになりそうか
* HITL(Human-In-The-Loop)を中心に組まれているAIDD設計が将来的にHOTL(Human-On-The-Loop)のようなloop設計になるとき、そのオーケストレーターはfableモデルが妥当そうか
