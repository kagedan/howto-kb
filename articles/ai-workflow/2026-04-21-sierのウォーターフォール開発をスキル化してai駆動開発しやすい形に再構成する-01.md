# **１．はじめに**

SIerの現場では、ウォーターフォール型の開発プロセスが長年にわたって整備されてきました。  
大きなSIerでは、開発プロセスが標準化され、豊富なドキュメント資産が存在します。

一方で、こうした標準プロセスには次のような課題があります。

* 習得コスト：標準プロセスを使いこなすまでの習熟コストが高い
* 属人化：担当者のスキルや経験によって成果物の品質にばらつきが出る
* フォーマット：標準資産の多くが Word / Excel 中心で、AI活用しづらい

特に最後の点は、AI駆動開発を進めるうえで大きな壁になります。  
人間が読むことを前提に作られたOffice文書は、そのままではAIエージェントの参照・再利用・差分管理に向いていません。

そこで本記事では、**SIerで標準化されたプロセス資産をエージェントスキルへ転用することで、ウォーターフォール開発をAI駆動開発しやすい形に再構成するアプローチ**を紹介します。

なお、仕様駆動開発や AI-DLC のような新しい開発プロセスを提案するものではありません。  
あくまで、**社内で標準化されたウォーターフォール型プロセスを、AI駆動開発に適した形へ再構成する取り組み**として読んでいただければと思います。

## **■ SIerにおけるウォーターフォール開発とは**

SIerの開発プロジェクトでは、企画・要件定義・設計・実装・テスト・リリースといった工程を順番に進める、ウォーターフォール型開発が今でも広く使われています。

各工程では、あらかじめ定められた成果物を作成し、レビュー＋承認を経て次工程へ引き継ぎます。  
この進め方自体は非常に整理されている一方で、実務では次のような課題が起こりがちです。

## **■ ウォーターフォール開発で発生しがちな課題**

標準プロセスやテンプレートが整備されていても、それを適切に参照しながら成果物を作るには**一定の経験が必要**です。  
結果として、標準化されているはずなのに、現場では**成果物の品質が人に依存してしまう**ことがあります。

| 課題 | 内容 |
| --- | --- |
| 成果物の属人化 | 担当者のスキルや経験によって、成果物の品質や粒度が変わる |
| 習熟コスト | 標準プロセスを理解し、正しく使いこなせるようになるまでに時間がかかる |

## **■ AI駆動開発に適さないドキュメント形式**

また、現場の**ドキュメントのほとんどがExcelやWordで管理**されています。  
これは**人間にとっては扱いやすい一方で、AI駆動開発の観点ではボトルネック**になりやすいです。

| 問題点 | 内容 |
| --- | --- |
| AIエージェントが直接読めない | Excel/WordのバイナリファイルはAIエージェントが直接参照できない |
| テキストとしてコピーできない | セルの結合・書式・図が多用されており、テキスト抽出が困難 |
| バージョン管理が難しい | Gitなどのバージョン管理システムとの相性が悪く、差分管理ができません |
| コンテキストとして渡せない | プロンプトやスキルのリファレンスとして直接組み込むことができません |

**標準資産が存在すること**と、**その資産をAIが扱いやすいこと**は別問題です。  
AI活用を前提にするなら、まず資産の形式そのものを見直す必要があります。

# **２．事前準備：資産をマークダウン化する**

長年にわたり整備・標準化されたプロセスをAIエージェントに理解してもらうには、まずWordやExcelといったOffice文書ベースの資産を**マークダウン形式に変換する**ことが大前提となります。  
ここで重要なのは、単なるファイル形式の変換ではありません。  
目的は、**AIエージェントが参照・再利用・差分管理しやすい形に、既存資産を再編成すること**です。

## **■ 変換方法：Office 文書操作用のエージェントスキルを活用する**

ありがたいことに、Anthropicの公式リポジトリに、PDF／PPT／Word／Excelを扱うためのエージェントスキルが公開されています。  
<https://github.com/anthropics/skills>

上記エージェントスキルを使うと、次のような簡単なプロンプトでもマークダウン形式に変換してくれます。  
図やフローのように文章だけでは扱いづらい情報も、Mermaidで表現できればマークダウン資産として再利用しやすくなります。

```
/xlsm
対象のExcel設計書テンプレートを、xlsxスキルを使ってAI駆動開発で扱いやすいマークダウンファイルへ変換してください
マークダウンファイルへ変換する中で、図・作業フローなどMermaidで表現できるものについてはMermaidで表現してください
```

# **３．エージェントスキル化の基本方針とイメージ**

ドキュメントのマークダウン化ができたら、次はそれをどうスキル化するかを考えます。

標準プロセスを見直してみると、成果物は大きく次の2種類に分けられました。

* **定義書や作成フローが明確ではない成果物**

  + 超上流の構想資料
  + 顧客との対話や複数人での議論を通じて作る成果物
  + 工程の途中で追加で必要になる成果物
* **定義書や作成フローが明確な成果物**

  + 要件がある程度固まった後に作る成果物
  + 外部設計書、内部設計書など、標準化しやすい成果物

この違いを踏まえて、スキル化の方針を次のように整理しました。

| 成果物の種類 | 例 | 方針 |
| --- | --- | --- |
| 定義書・フローなし | 超上流成果物、複数人の対話が必要な資料、急遽追加で必要になった資料 | AIエージェントとの対話を通じて、探索的に成果物を作るスキルにする |
| 定義書・フローあり | 要件定義以降で少人数でも作成しやすい成果物、外部設計書、内部設計書など | 工程別に定義・フロー・フォーマットを参照する標準スキルにする |

## **■ エージェントスキルを利用した工程の進め方のイメージ**

2種類のスキルを組み合わせると、ウォーターフォール型開発をカバーするパイプラインが構成できます。  
超上流では対話型のスキル、プロセスが標準化された工程では、標準化スキルで成果物を生成していくイメージです。  
![](https://static.zenn.studio/user-upload/41645c38a711-20260421.png)

# **４．作成したエージェントスキル**

## **■ 「定義書・フローなし」パターンのスキル**

上流工程の構想段階では、そもそも「何を書くべきか」が確定していないことが多くあります。  
また、顧客との会話やチーム内の議論を通じて、内容を徐々に固めていく必要があります。

こうした探索的な成果物では、人間だけでなく AIエージェントも議論に参加させる形が有効です。

このタイプのスキルは、Anthropic が公開している **`doc-coauthoring` スキル**を参考にしました。  
基本的にはこれをベースにし、上流工程で必要になる参考資料や観点を追加して構築しています。  
<https://github.com/anthropics/skills/tree/main/skills/doc-coauthoring>

スキルを起動すると、AIエージェントと段階的に対話しながら成果物を作成できます。

| ステップ | 内容 |
| --- | --- |
| ステップ1：コンテキスト収集 | AIがヒアリングを通じて背景・目的・制約条件を整理する |
| ステップ2：構成検討・ドラフト作成 | アウトラインを提案し、セクションごとにドラフトを生成・修正する |
| ステップ3：読者テスト | 別のエージェント視点で疑問点や改善点を洗い出す |

---

あくまで一例ですが、下記のような使い方をすると複数人＋AIエージェントを巻き込んだ成果物作成が可能です。

1. VS Code の拡張機能「Live Share」を導入する  
   ![](https://static.zenn.studio/user-upload/3fdaebf10478-20260421.png)
2. ホストの VS Code 環境（ファイル編集やターミナル操作）を共有する
3. VS Code のターミナル上でコーディングエージェントを起動する
4. スキルを実行し、複数人 + AI エージェントで議論しながら成果物を作る

このタイプのスキルは、**正解が最初から決まっていない成果物**に向いています。

## **■ 「定義書・フローあり」パターンのスキル**

一方で、成果物の定義書や作成フローがすでに存在しているものについては、工程ごとに標準スキルとして整理できます。

スキル構成は、[一般的なスキル構成](https://agentskills.io/specification)に沿って次のようにしました。

```
スキルフォルダ/
  ├── SKILL.md
  ├── references/
  │   ├── 成果物１/
  │   │   ├── 成果物１定義.md
  │   │   └── 成果物１作成フロー.md
  │   └── 成果物２/
  │       ├── 成果物２定義.md
  │       └── 成果物２作成フロー.md
  └── assets/
      ├── 成果物１フォーマット.md
      └── 成果物２フォーマット.md
```

| ファイル名 | 説明 |
| --- | --- |
| SKILL.md | スキル実行時のエントリーポイント。対象成果物に応じて、成果物定義・作成フロー・テンプレートを参照し、成果物作成を進める |
| 成果物定義.md | 成果物の目的、記載項目、記載粒度、完成条件を定義する |
| 成果物作成フロー.md | 作成手順、前提、入力情報、作成順序、確認観点を定義する |
| 成果物フォーマット.md | 成果物のテンプレートを定義する |

スキルを起動すると、エージェントは定義・フロー・フォーマットを参照しながら成果物のドラフトを生成し、ユーザーとのレビューと修正を経て成果物を完成させます。  
![](https://static.zenn.studio/user-upload/6aadddd518f0-20260421.jpeg)

このように標準資産をスキルへ内包することで、ウォーターフォール開発で起きがちな課題に対して、次のような改善が期待できます。

| 課題 | スキル化後の変化 |
| --- | --- |
| 成果物の属人化 | 成果物定義・作成フロー・フォーマットをスキル側に持たせることで、一定の品質と粒度を保ちやすくなる |
| 習熟コスト | スキルが手順や観点をガイドするため、標準プロセスへの習熟を待たずに成果物作成へ着手しやすくなる |

もちろん、これだけで完全に属人性がなくなるわけではありません。  
ただし、少なくとも**標準資産を探し回りながら手作業で読み解く負荷**は減らせます。

# **５．今後の展望**

今後は以下のような展開を想定しています。

## ■ フェーズゲートの取り入れ

現時点では、成果物のフォーマットに YAML フロントマターを持たせて、ステータスを管理しています。

```
---
status: draft          # draft -> review -> approved -> baselined
---
```

今後は、スキルにステータスチェック用のスクリプトを組み込み、プロジェクトの進行状況や成果物の状態を見ながら、次の工程に進めるかどうかを評価する仕組みを取り入れたいと考えています。

```
スキルフォルダ/
  ├── SKILL.md
  ├── references/
  │   ├── 成果物１/
  │   │   ├── 成果物１定義.md
  │   │   └── 成果物１作成フロー.md
  │   └── 成果物２/
  │       ├── 成果物２定義.md
  │       └── 成果物２作成フロー.md
  ├── assets/
  │   ├── 成果物１フォーマット.md
  │   └── 成果物２フォーマット.md
  └── scripts/
      ├── ステータス管理用スクリプト
      └── ...
```

これが実現できると、成果物生成だけでなく、**フェーズゲートを含めた開発プロセス全体の運用支援**までスキル側で扱えるようになります。

## ■ 共通のスキル実行環境の提供

現時点では、個人がローカル環境でスキルを実行して成果物を作成する使い方を想定しています。  
しかし、実プロジェクトでは複数人が別々の工程を担当し、成果物を受け渡しながら開発を進めます。

そのため、実運用を考えると、個人のローカル環境だけでなく、**複数人が共通で利用できるエージェントスキル実行基盤**があると使いやすいと感じています。

# おわりに

ウォーターフォール型の開発プロセスは「**人が作ることを前提**」とした時代の開発プロセスであるため、単純なスキル化だけでは不十分だと感じています。

一方で、長年蓄積されてきた標準資産をマークダウン化し、さらにエージェントスキルとして再構成することで、従来のプロセスを **人と AI が協調して扱える形** に変えていく余地は十分にあります。

こうした取り組みが「**人とAIが共に作ることを前提**」とした時代の新たな開発プロセス創出のきっかけにはなることを期待しています。
