# AI-DLC とは

AI-DLC (AI-Driven Development LifeCycle) は、AWSが提唱した開発手法です。

[本家の記事](https://aws.amazon.com/jp/blogs/news/ai-driven-development-life-cycle/)、および [試してみた記事](https://tech.every.tv/entry/2026/02/05/184756) が内容を詳しく解説しています。

# AI-DLC Workflow の導入

<https://github.com/awslabs/aidlc-workflows.git> を任意の場所（プロジェクトルートなど）にクローンします。

```
$ cd YOUR-PROJECT
$ git clone https://github.com/awslabs/aidlc-workflows.git
```

README では `aidlc-workflows/aidlc-rules/aws-aidlc-rules/core-workflow.md` を AGENT.md（またはCLAUDE.md）に置き替える、とありますが、単純に既存の AGENT.md 内に一行加えるだけで大丈夫でした。

```
**MUST READ**: @aidlc-workflows/aidlc-rules/aws-aidlc-rules/core-workflow.md
```

この core-workflow.md がすべての起点となっています。

# AI-DLC をはじめる

正直な話、AI-DLC がどのようなフローで成り立ってるかを十分把握している必要はありません。

ひとまず「AI-DLCに基づいた開発を始めたい。なにからやったらいい？」くらいの問いかけで、AI-DLCがスタートします。

# Inception Phase

## ワークスペース状態把握（Workspace Detection）

AI-DLCに基づいて作業が進行中なのか、新規にスタートするのかを判定します。

## ソースコード解析（Reverse Engineering）

機能追加を行う際に、まずはじめにソースコードの全容を把握します。

## 要求分析（Requirements Analysis）

なにをやりたいか、を複数回質疑を繰り返して明確にしていきます。

## ユーザストーリー作成（User Stories）

要求分析の結果必要と判断されたときに、ユーザストーリーを作成します。

私が試したときはスキップされました。

## ワークフロー計画作成（Workflow Planning）

ここで、要求分析やユーザストーリーから、今後どのようにコードを生成していくかの計画を立てます。

作業単位をユニットとして分割し、1ユニットが独立して作業できるかどうかを計画するようなイメージです。

## アプリケーション設計（Application Design）

作業ユニットを分割したので、ユニットごとの詳細を設計していきます。コンポーネント・サービス層などの設計を行います。

ユニットの作業順を計画し、要件との対応を確認します。

# Construction Phase

## 機能設計（Functional Design）

ワークフロー計画とアプリケーション設計で、何を作るかを定義しました。  
機能設計では、下記のような「どう作るか」の詳細を詰めていきます。

ウォーターフォールでも機能設計とか詳細設計とか呼ばれるフェーズです。

## 非機能要件の確認、インフラ設計

みたいなのが作られることもあるそうですが、自分が試したときはスキップでした。

## コード生成（Code Generation）

とうとうコード生成が始まります。

# 一連の動作を体験しての所感

コードはものの10分程度ですべて生成されました。

しかし実際動作させてみると、やはりうまく動かないところが多々ありました。

ここで、じっくり時間をかけてコードレビューをおこない、不具合を見つけていくのも策の一つではありますが、大変そうです。

## テスト計画書の作成

そこで、以下のような追加の手順を加えてみました。

* V字モデルに従った以下のテスト項目作成
  + 要求分析から受け入れテスト（人の手で動かして試すテストシナリオ）
  + 要件定義から総合テスト（E2Eテスト）
  + 基本設計から結合テスト（モックを使ったI/O、DB読み書きを伴うテスト）
  + 詳細設計からユニットテスト（モックを使わない関数単位のテスト）

![V字モデル](https://static.zenn.studio/user-upload/e3e374c77632-20260330.png)

膨大な量の生成コードを読み解くよりは、テスト項目に過不足が無いかをチェックするほうが幾分楽なように感じます。

テスト項目をレビュー後、これらを、下（ユニットテスト）から順に実行、NGがあったら修正、を繰り返します。

E2Eテストの不具合を直し切ったくらいで、ほぼほぼ問題なく動くクオリティにはなった印象です。

# まとめ

AI-DLCを利用して、プロダクトを作ってみました。

おおむねスムーズに進めることができましたが、やはり品質がネックになると感じました。

テスト計画のフェーズを追加し、テスト項目をレビューすることで、コードレビューより楽に品質を担保することができると思います。
