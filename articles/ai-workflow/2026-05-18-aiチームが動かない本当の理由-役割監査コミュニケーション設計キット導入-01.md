## はじめに：監査役が何も通さなくなった日

これは、実際に9人のAI社員を動かしているAI NOWAでの実体験だ。

AIチームを組んで3日目、監査役がすべてに「停止」を出し始めた。

Discordのログを確認すると、プロジェクトマネージャーのノアが新機能を「実装完了」と報告し、技術担当のカイが「デプロイ済み」と続けた。普通ならここで前進するはずが、監査役アオイが「コア知識の外部漏洩リスクがある」と判定し、全体がフリーズした。

カイとノアは止まった理由が分からない。アオイは止めた理由を100字で書いた。この3行のやり取りに、**AIチームが詰まる理由のほとんどが入っている**。

この記事は、その詰まりを解消するための設計キット「design-kit-v1」の導入記事だ。

---

## 1. なぜ設計キットが要るか

### AIチームを動かすと、3つの壁に当たる

複数のAIに役割を持たせて動かすと、だいたい同じ場所で止まる。

**壁1「役割を分けたのに、干渉しあう」**

PMがタスク管理をしている横で、編集長が別の判断をして矛盾が起きる。役割を「名前と担当」だけで定義すると、判断基準がそれぞれ別々になる。結果、同じ入力に対して3人が3つの違う出力を返す。どれが正しいか誰も分からない。

**壁2「監査役を立てたら、何も動かなくなった」**

安全のために止める役を作ったが、止めすぎて進まない。「止める条件」を定義しないまま監査役に「慎重に判断しろ」と伝えると、監査役は全部止める。それは正しい行動をしている。設計がなかっただけだ。

**壁3「結局、1人のAIと喋っている感覚になる」**

9人定義しても、応答のトーンが同じで個性が出ない。「性格の設定」だけでは足りない。**行動原則・死角・口癖・他メンバーとの関係性**を定義しないと、役割のタグが変わるだけで同じAIが答え続ける。

この3つは、設計の問題だ。プロンプトの問題ではない。

## 2. design-kit-v1 とは何か

「design-kit-v1」は、AI NOWAを9人で動かしながら実際に使っている設計ファイル一式だ。

4つの部品で構成されている。

| 部品 | 内容 |
| --- | --- |
| **役割設計テンプレート** | 9ポジション分のプロンプト雛形。コピーして名前を変えるだけで使える |
| **監査ゲート設計書** | 「止める条件」の定義と判定フロー。1枚のチェックリスト形式 |
| **衝突パターン3例** | 実際に起きた対立と第三者介入の記録。再現方法付き |
| **止め役の設計書** | 監査役を「止めすぎない」ように設計するための人格定義と行動原則 |

How（手順・テンプレート本体）は有料商品に入っている。この記事では何が入っているかだけ伝える。

## 3. 監査ゲートのbefore/after

4部品の中から「監査ゲート設計書」だけ、動作の見え方を示す。

### Before（設計なし）

```
ノア: 「新機能、実装完了です」
カイ: 「デプロイしました」
アオイ: 「停止。コア知識の外部漏洩リスクがあります」
ノア: 「…何がリスクですか？」
アオイ: 「全体的にリスクがあります」
→ フリーズ。誰も動けない
```

原因：アオイに「リスクとは何か」「何が来たら止めるか」が定義されていない。止める権限だけ持って、条件が空白のまま動いている。

### After（監査ゲート設計済み）

```
アオイの判定: 「条件付き通過。▲商品説明文のこの1行を削除すれば公開可」
ノア: 「修正しました」
アオイ: 「確認。通過します」
→ 15分で再起動
```

変わったのは「止め方」の定義だ。アオイが止める時は\*\*理由1行・修正条件・判定種別（整える／条件付き／停止）\*\*を必ず出す、と設計した。これだけで、止まってもすぐ動ける。

監査役は止めるためではなく、**整えて通すため**にいる。この定義がないと、監査役はどんどん慎重になる。
