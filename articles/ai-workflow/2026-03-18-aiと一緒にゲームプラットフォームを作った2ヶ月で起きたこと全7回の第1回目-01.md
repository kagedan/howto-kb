前回の記事「[ゲームを配信するために自分用のバックエンドサービスと組み込み型SDKを開発しはじめたお話](https://note.com/hataraiku/n/n65a0e3606323)」で、AIを使ってゲーム配信用のバックエンドを作り始めたよ、というお話をしました。

あの記事では「つくったもの」の話が中心でしたが、個人的に良かったのは「つくる過程」のほうでした。振り返りを兼ねて、今回から全7回にわたって、AIと一緒にプラットフォーム開発をした2ヶ月間で何が起きたのかを生々しく振り返ってみます。

## 2ヶ月で何が起きたのか、数字で見る全体像

まずは結果から。

数字だけ見ると「めちゃくちゃ開発してんな」ですが、正直に言うと、自分がキーボードを叩いてコードを書いた量は全体の1ミリもありません。残りのほぼ10割はAIが書いています。

じゃあ自分は何をしていたのかというと、**AIに「どう動いてほしいか」を伝え続けていました。**

## 「AIにコードを書かせる」から「AIとプロジェクトを運営する」へ

開発初期の自分のワークフローはシンプルでした。

1. AIに「こういう機能を作って」と言う
2. AIがコードを生成する
3. 動くかどうか確認する
4. 動かなかったらエラーメッセージを貼り付けて「直して」と言う

バイブコーディングですね。2025年のRoo Code記事（2025年7月）の頃は、まさにこれでした。Orchestrator → Architect → Code → Debug とモードを切り替えて、ひたすらコードを生成する。1つのタスクのAPIコストが$3超えたらだいたいダメになるので、切り替えて再スタート。

でも、バックエンドプラットフォーム開発を始めてから、このやり方だと破綻することに2週間くらいで気づきました。

### なぜ破綻したか

フロントエンドのゲームなら「動けばOK」でなんとかなる。でもバックエンドは違います。

AIが書いたコードを「動くかどうか」だけで判断していたら、いつか取り返しのつかないことが起きる。それに気づいたのは、何気なく「そういえばセキュリティチェックしてよ」と依頼した日の事。  
AIが生成したコードの中に Math.random() でIDを生成している箇所が見つかってCriticalで差し戻しがあったときでした。

あーね、、、そうね、暗号学的に安全じゃない乱数生成でユーザーIDを作ってる。動くけど、これはダメでしょう。

この瞬間から、自分のジョブが変わりました。「コードを書かせる人」から「**プロジェクトの品質を守る人**」に。

## 道具の進化 Roo Code → Claude Code → 多頭体制

ツールも大きく変わりました。初期のバイブコーディングではRoo CodeのカスタムモードでAIエージェントっぽい設計を自作して動かしてましたが、途中からClaude Codeに完全移行しています。

### Roo Code時代（〜2025年末）

これはこれで好きだったけど、従量制はとても辛かった。

* VS Code上で動く、カスタムモードで差をつける系
* Orchestrator → Architect → Code → Debug のモード切替
* APIコスト従量制。$3の壁。気がつくとコンテキストが汚染されてAIが指示を忘れはじめる
* 「全部賭けたところ、Claude-Codeが全部もっていきました」（前回記事より）

### Claude Code時代（2026年1月〜）

いつのころから、性能も使い勝手も完全に別次元。定額制なのもアツい。

* ターミナルで動く。Unix哲学。シンプルで鋭い
* MAXプランの定額制。コスト気にせずぶん回せる革命
* .claude/ ディレクトリに設定ファイルを置ける → **プロジェクトごとにAIの振る舞いを制御できる**
* Hooks（フック）でAIのツール呼び出しに介入できる → **ガードレールを物理的に設置できる**

特に .claude/ ディレクトリの存在が大きかった。「[CLAUDE.md](http://CLAUDE.md) にプロジェクトの方針を書いておけば、AIがそれを読んで行動する」という仕組みは、最初は便利なメモ帳程度の認識でした。それが2ヶ月後には **21本のルールファイルと、AIの行動を物理的に制御するフックの集合体** に成長するとは思ってもいませんでした。

### 「セカンドオピニオン」も付けました

Claude Codeだけでは不安は拭えず、あるときOpenAIのCodex CLIをセカンドオピニオンとして導入。さらに「セキュリティ専門のAIチーム」として、攻撃者役（Red Team）と防御者役（Blue Team）のサブエージェントを作りました。

加えて、Google の Jules にGitHubリポジトリを監視させて、セキュリティ上の指摘を自動通知させる仕組みも導入。Jules自体の修正コードの品質は正直それほど高くないのですが、**「何か問題がありそう」という通知のトリガー**として優秀。通知が来たら Claude / Codex にレビューさせてコード品質を整えてからマージする、というワークフローです。ちょっとしたお守り代わり。

1つのAIに全部任せるのではなく、**複数のAIに異なる視点からレビューさせる**。人間のチーム開発と同じ発想です。これが後に「セカンドオピニオン必須ルール」としてプロジェクトの憲法に組み込まれることになります。

## 2ヶ月を3つのフェーズで振り返る

振り返ると、2ヶ月間はだいたい3つのフェーズに分かれていました。

### Phase 1: 爆速開発期（1月） とりあえず思いついたら全部入れる狂気

APIを片っ端から実装していた時期。ランキング、フレンド、マッチメイキング、ウォレット、リアルタイム通信……。AIに設計文書（SPEC）を書かせて、SPECに沿って実装させる。1日に17コミットが飛ぶ日もありました。

この時期は「AIすげー！なんでも作れる！！」という万能感に満ちていました。

### Phase 2: セキュリティ覚醒期（2月） 作るより守るほうが難しい

Codexセカンドオピニオンと Red/Blue Team を導入した時期。コードレビューで次々と脆弱性が見つかる。タイミング攻撃、IPスプーフィングによるレート制限回避、XSS、弱い乱数生成……。

「動いてるから大丈夫」ではなく「**攻撃者の視点で見たらどうか**」を考えるようになったのがこの時期です。セキュリティ関連だけで15個以上のコミットが積まれました。

### Phase 3: 品質統治期（3月） AIに法律を与える

Express→Hono移行（247エンドポイント全部）、OpenAPI駆動のドキュメント自動生成、環境変数の事故対策……。この時期はもう新機能をほとんど追加していません。**既存のコードを、壊れにくい形に作り変える**ことに全力を注いでいました。

そしてこの時期に .claude/rules/ が爆発的に増えました。21本のルールファイルの大半はこの時期に生まれています。

## この連載で書くこと

全7回の構成はこんな感じです。

1. **この記事** → 全体像と、何が起きたかの概観
2. **AIに"憲法"を与えた話** → CLAUDE.mdとルールファイルの進化史
3. **セカンドオピニオンとAIセキュリティチーム** → 脆弱性はどうやって発見したのか
4. **23本のSPEC文書で247エンドポイントを移行した話** → SPEC駆動開発の威力
5. **"ベストプラクティス"の賞味期限は3ヶ月だった** → SPEC駆動・TDDへの率直な再評価
6. **ルールからフックへ — AIガードレール進化論** → エージェント時代の制御手法の最前線
7. **自律プロセスへの挑戦** → OpenClaw、Claude Code Hooks、そして収束する未来

前半4回は「作った体験の話」、後半3回は「体験を踏まえて今を見渡す話」です。第5回で「あれ、自分がやってたことって実はもう古いのかも」とちゃぶ台をひっくり返すので、そこまでお付き合いいただけると嬉しいです。

## 正直な所感

2ヶ月の開発を終えて思うのは、AIを使った開発は **「AIの性能」と「人間の設計力」の掛け算** だということです。

AIの性能は日進月歩で上がっていきます。2ヶ月前の自分が「これがベストだ」と確信していたやり方が、今日の時点ですでに古くなっている。これは怖いことでもあり、面白いことでもあります。

でも、「AIにどう動いてほしいかを設計する力」は陳腐化しません。道具は変わっても、道具の使い方を考える力は残る。この連載では、その「考え方の変遷」をできるだけ正直に書いていくつもりです。

次回は「CLAUDE.md に1行書いたところから、21本のルールファイルに分裂するまで」のお話の予定です。
