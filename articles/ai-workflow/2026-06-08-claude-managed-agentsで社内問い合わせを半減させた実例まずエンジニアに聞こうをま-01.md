社内で「これ分かる人いますか？」とエンジニアに聞く文化、ありますよね。  
便利なんですが、聞かれる側のエンジニアは作業を止められ続けます。  
飲食店向けプロダクトを開発するダイニー（Dinii）は、この「まずエンジニアに聞こう」を「まずbotに聞こう」に変えました。

使ったのは、Anthropic の Claude Managed Agents です。  
結果、エンジニアへのエスカレーションは初週の100%から、数週間で22%まで下がりました。

この記事では、その事例を初心者向けに整理します。  
「Managed Agents で何ができて、自分の環境でも試せるのか」が読み終わったときに分かるように書きました。  
元ネタは、ダイニーのまりるさんが公開した[この記事](https://zenn.dev/dinii/articles/d7be3acc43d868)です。

## まず数字から：社内問い合わせはこう変わった

  

ダイニーが社内 Slack に置いたのは、ask-anything という質問 bot です。  
2026年4月末から本格稼働して、数字はこう動きました。

* エスカレーション率（bot で解決できず人間に回った割合）：初週100% → 5月中旬の週で22.0%
* エンジニアへの質問チケット起票：週40〜50件 → 20〜30件（ほぼ半減）
* bot が回答に参加したスレッド：4週間で週24件 → 109件
* 回答時間：データ調査が10〜15分 → 1〜2分、ログ調査が15〜30分 → 1〜3分

最初は全部エンジニアに回っていた質問が、数週間でほとんど bot だけで完結するようになった、ということです。

![](https://assets.st-note.com/img/1780876681-O6thqF97B2SmXpoke8xHWQDs.png?width=1200)

この数字の裏側で起きているのは、エンジニアの「集中の取り戻し」です。  
質問のたびに作業を中断されていた人たちが、本来の実装に戻れる時間を取り戻しました。

## 何が問題だったのか：「まずエンジニアに聞こう」のコスト

社内の質問には、実は「再現性の高いもの」がかなり混じっています。

どれも答えは社内のどこかにあります。  
ただ、その「どこか」にアクセスする手順を知っているのがエンジニアだけ、という状態でした。

だから質問はすべてエンジニアに集まります。  
聞かれた側は、自分の作業を止めて BigQuery を開き、SQL を書いて、結果を整形して返す。  
1件あたり10分や30分でも、何件も積み重なれば1日が溶けます。

「人にしか聞けない」のではなく「手順を知っている人にしか聞けない」。  
ここを bot で埋めれば、エンジニアを解放できる、というのが出発点です。

## ask-anything が実際に答えてくれること

使い方はとてもシンプルで、Slack でメンションするだけです。

```
@ask-anything 昨日の◯◯店の客数を教えて
```

これだけで、bot が裏側で必要なデータを取りに行き、整形して返してくれます。

データ調査なら、bot が自分で SQL を組み立てて、コストを見積もる dry-run をかけてから実行し、結果を表にして返します。  
人間がやると10〜15分かかっていた作業が、1〜2分で返ってきます。

ログ調査も同じです。  
Cloud Logging のフィルタを組むか、SQL でログを抽出するか——その判断ごと bot に任せられます。  
時系列で読んで CSV に書き出すところまでやってくれます。

仕様やバグの確認では、bot がソースコードを読みに行きます。  
社内のコードを read-only で配置して、関連箇所を grep して要約を返す。  
「どのファイルを見ればいいか分からない」を bot が肩代わりしてくれる形です。

## 裏側の構造：7つのデータ源を read-only で束ねる

ask-anything が賢いのは、質問の種類に応じて「どのデータ源を叩くか」を自分で選ぶからです。

つないでいるのは、次の7つです。

* **BigQuery**（売上などのデータ）
* **Cloud Logging**（アプリのログ）
* **monorepo**（ソースコード）
* **Notion**（ドキュメント）
* **GitHub**（コードとPR）
* **Sentry**（エラー情報）
* **Slack**（過去のやり取り）

ここで大事なのは、すべて read-only、つまり読み取り専用でつないでいる点です。  
bot にデータを書き換えたり消したりする権限は渡していません。

「社内データを bot に触らせて大丈夫なのか」という不安は、誰もが最初に感じます。  
その答えが、この read-only という設計判断です。  
読めるけれど書けない。  
だから、最悪でも情報が壊れることはない、という安心感のうえで運用できます。

![](https://assets.st-note.com/img/1780876769-5yaWOEA9IwHtcoNpS8v1ekFD.png?width=1200)

## なぜ Managed Agents を選んだのか：自前コードが7割減った

ここが、エンジニアにとって一番おいしい話かもしれません。

ダイニーは元々、こうした bot を自前で組もうとしていました。  
ですが Claude Managed Agents に乗り換えた結果、自分たちで書き続けるコードが3種類だけに減ったそうです。

* Slack からのイベントを受け取る、薄い受け口のコード
* Claude 側のセッションを作り、やり取りを中継するコード
* bot から呼ばれる「実際の処理」のコード（BigQuery を叩く、CSV を出す、など）

体感で7割のコードが消えた、という表現が使われています。

なぜ減るのか。  
ツールを動かす実行環境（sandbox）や、外部サービスとの接続、手順書（Skill）の管理を、Anthropic 側が引き受けてくれるからです。  
自分たちは「何を聞かれたら何を返すか」という中身だけに集中できます。

## Claude Managed Agents とは（1分でわかる基礎）

ここまで読んで「そもそも Managed Agents って何？」という人向けに、最小限だけ整理します。

ひとことで言うと、ツールを登録した Claude を、Anthropic 側でホストしてくれる仕組みです。  
普通の Claude API は、こちらがリクエストを投げて、返事をもらって終わりの単発のやり取りでした。  
Managed Agents は、Claude が自分でツールを使い、考え、長く動き続けるセッションを前提にしています。

向いているのは、今回のような「複数のツールをまたいで、ある程度の時間をかけて答えを作る」タスクです。  
逆に、ひとことで返せるような軽い処理には向きません。  
そういう用途は、素の Messages API を直接叩いたほうが速くて安いです。

料金は実行時間ベースで、動いている間だけ1セッションあたり時間 約0.08ドルです。  
待機中は課金されないので、思っているより試しやすい料金設計になっています。

## 自分の環境で試すなら

「うちでも試せるか？」と思った人へ、最初の一歩を具体的に書きます。

いきなり7ツール全部をつなぐ必要はありません。  
read-only のツールを1本だけつないで、小さく始めるのが安全です。

最初にやるべきは、社内で「再現性の高い質問」を洗い出すことです。  
データ確認、ログ確認、仕様確認——毎回エンジニアに飛んでくる定番の質問を見つけます。  
そこが bot に置き換えやすい、一番おいしい部分です。

料金の感覚も持っておくと判断しやすいです。  
1セッション約0.08ドルなら、月に100回動かしても10ドル弱です。  
小さな検証なら、コストはほぼ気にしなくていいレベルから始められます。

## まだ人間が必要なところ

正直に書いておくと、これは万能ではありません。

エスカレーション率が22%まで下がった、という数字は裏を返せば「22%はまだ人間に回っている」ということです。  
文脈が深い質問や、判断をともなう相談は、これからも人間の仕事として残ります。

技術的なつまずきもあります。  
クラウドの認証情報を sandbox にどう渡すか、という設計はひと工夫が必要です。  
接続先 URL の末尾スラッシュひとつで認証が通らなくなる、といった地味な罠も報告されています。

そして、bot の回答精度そのものも運用で育てていくものです。  
手順書を整え、ツールの精度を上げていくことで、初週100%だったエスカレーションが22%まで下がりました。  
置いて終わりではなく、育てる前提で向き合うのが現実的です。

## まとめ

ダイニーの事例が示したのは、「人にしか聞けない」と思っていた社内の質問の多くが、実は「手順を知っている人にしか聞けない」だけだった、ということです。

その手順を Claude Managed Agents に肩代わりさせたら、エンジニアへの質問は半分になり、エスカレーションは100%から22%まで下がりました。

まずは read-only のツール1本から、再現性の高い質問を bot に任せてみる。  
そこから始めるのが、一番現実的な第一歩だと思います。
