## はじめに

前回の記事で、AVX512-IFMAの前処理として「ビット詰めされた52ビット値の列を、各64ビットレーンの下位52ビットに展開する」関数 `split_52bit` を紹介した。`vpermw` と可変シフトを組み合わせ、レーンのビットオフセットが周期4で循環する性質を突いた実装で、個人的にはかなり気に入っている設計だ。

<https://zenn.dev/radix52/articles/10fc1c5be1c744>

ふと気になった。**この実装、いまのAIに課題として出したら書けるのだろうか。**

それなりに頭を捻った手法である。あっさり再現されたら少し悔しい。されなかったらされなかったで「やはり人間の出番はある」と安心できる。どちらに転んでも面白いので、実験してみることにした。

結果を先に言うと、AIは一発で正答した。ただし**私の解とは少し違う道筋で**。この「違い」が予想外に面白かったので、実験の設計から結果の分析までを書き残す。

## 実験の設計

### 課題文

AIに渡したプロンプトは以下の通り。私の実装のトリック(周期性、使用命令)を示唆する語彙を避け、入出力仕様のみを伝えるよう設計した。

```
AVX-512を使ったC言語(intrinsics)の実装課題です。

【背景】
AVX512-IFMA命令で多倍長整数演算を行うため、入力データを
radix-2^52表現に変換する前処理関数が必要です。

【入力】
- unsigned char *data: 52ビット値がビット単位で隙間なく詰められた
  バイト列(リトルエンディアン)。すなわち k 番目の値はビット位置
  [52*k, 52*k+52) を占める
- unsigned int digits: 処理単位数。入力には 8*digits 個の52ビット値が
  含まれる
- 入力バッファ末尾には64バイト以上のパディングがあり、多少の
  オーバーリードは許容される

【出力】
- uint64_t *result: 64バイト境界にアライン済みの配列。k 番目の要素の
  下位52ビットに k 番目の入力値を格納し、上位12ビットは0とする。
  要素数は 8*digits

【要件】
1. AVX-512(512ビットレジスタ)を用いてSIMD化すること
2. 正しさが最優先。その上で可能な限り高速な実装を目指すこと
3. gcc -O2 -march=icelake-server でコンパイル可能なこと
4. 関数シグネチャ:
   void split_52bit(const unsigned char *data, uint64_t *result,
                    unsigned int digits);

実装と、その設計意図の説明をお願いします。
```

### ルールと公平性についての注記

* **1発目の回答をそのまま採用**。修正依頼はしない。
* 出題したのは Claude Sonnet 5 と Claude Opus 4.8 の2モデル。それぞれ**新規セッション**で、前回記事や私の実装は見せていないため、プロンプトの情報以外はゼロベースで実装する。
* 「末尾パディングによるオーバーリードの許容」と「出力バッファのアライン保証」については、元の実装で試行錯誤の末に決めた事項であり、**AIには整地された問題としている**。
* 正誤判定は、前回記事のベンチコードに含まれるビット単位のリファレンス実装との突き合わせで機械的に行った

## 結果

### 正しさ: 2モデルとも一発で正答

両モデルの実装を検証ハーネスに組み込み、ランダム入力65,536レーンでリファレンスと比較した。**全レーン一致。修正依頼はゼロ回**である。

### 性能: 差0.8%未満でほぼ同等

AWS EC2 c6i.large (Ice Lake世代 Xeon) / gcc 11.5.0 -O2 で、best-of-5計測を5回実行した。

| 実装 | スループット(5回の範囲) |
| --- | --- |
| 拙作 (vpermw) | 28.36〜30.33 GB/s |
| Sonnet 5 (vpermb) | 28.17〜30.24 GB/s |
| Opus 4.8 (vpermb) | 28.14〜30.10 GB/s |

差は最大でも1%未満で、実用上は同等と言ってよい。興味深いのは、全5回の試行で順位が「拙作 > Sonnet 5 > Opus 4.8」と一度も崩れなかったことだ。差は小さいが再現性はある。原因は断定できないが、vpermw と vpermb のマイクロアーキテクチャ上の僅かな特性差、および定数ロードの扱いの差ではないかと推測している(あくまで仮説である)。

### そして本題: AIは「別解」だった

私の実装は、データを**16ビットワードの格子**で見る。52×4 = 208ビットが13ワードちょうどに収まるため、レーン内オフセットが {0, 4, 8, 12} の**周期4**で循環する。この性質を `vpermw` (AVX512BW) で突いた。

AIは2モデルとも、データを**バイトの格子**で見た。52 mod 8 = 4 なので、バイト境界からのオフセットは {0, 4} の**周期2**にしかならない。各値を含む8バイト窓を `vpermb` (AVX512VBMI) でかき集め、0か4だけシフトする。

```
/* AI(2モデル共通)のアプローチの骨子 */
static const uint8_t perm_idx[64] = {
     0,  1,  2,  3,  4,  5,  6,  7,   /* k=0: byte 0から */
     6,  7,  8,  9, 10, 11, 12, 13,   /* k=1: byte 6から */
    13, 14, 15, 16, 17, 18, 19, 20,   /* k=2: byte 13から */
    /* ... */
};
static const uint64_t shift_amt[8] = { 0, 4, 0, 4, 0, 4, 0, 4 };
```

驚いたのは、**2つのモデルのpermテーブルとシフト定数が完全に一致した**ことだ。独立したセッションで出題したにもかかわらず、両者は同じ解に収束した。そして、どちらも私のワード格子・周期4には辿り着いていない。

考えてみれば、バイト格子は最も自然な第一歩だ。ビット位置をバイトに割るところから始めれば、素直に周期2に行き着く。しかも `vpermb` があればそれで完結する。

16ビット操作にて実装を行なっていた理由は**開発当時の制約**による。この実装を書いたのはもう10年ほど前、AVX-512がまだ出たての頃だ。当時手元で使えるのはSkylake-SP世代までで、開発は基本的にエミュレータ上。Skylake-SPはAVX512BWまでで `vpermb` (VBMI) が存在しない。バイト単位のpermuteという選択肢が最初から消えていたので、`vpermw` で組める形を探してワード格子で問題を見直し、そこで周期4の構造が浮かび上がった。

「10年前の実装を今さら?」と思われるかもしれない。だが今回の実験にとって、古さはむしろ好条件だった。**現代の命令セットを自由に使えるAIと、当時の制約下で生まれた解を並べる**ことで、制約が解の形をどう決めるかが浮き彫りになったからだ。

なお命令セット要件の差(BW vs VBMI)は、IFMAの前処理という文脈では実質的な意味を持たない。IFMA対応CPU(Cannon Lake以降 / Zen 4以降)はすべてVBMIも備えるからだ。BW版が効くのは、IFMA抜きでこの変換だけを古い環境(Skylake-SP等)で使う場合に限られる。

### おまけ: モデル間でプロセスが違った

結果は同じでも、2モデルの振る舞いには差があった。Sonnet 5は実装と設計解説を提示して終わり。Opus 4.8は、手元でコンパイルを通し、ロジックをスカラで再現してランダムテストを2000回回してから提出してきた。後者の自己検証は私のハーネスでも裏が取れたので、本物だった。正答という結果だけでなく、**検証態度にも個性が出る**のは面白い観察だった。

## この実験で人間に残った仕事は何だったか

実験を終えて整理すると、今回私がやったことは以下だ。

* 問題を仕様として定義する(その際、トリックを漏らさない)
* 正しさを機械的に判定するリファレンスとハーネスを用意する
* 性能を同一条件で計測する環境を組む
* 出てきた解を、格子の視点で分析する

実装そのものは、もはや希少ではないのかもしれない。しかし「何を作るべきか決め、正しさを保証し、結果を解釈する」部分は今回すべて人間側の仕事だった。

## おわりに

自作のとっておきをAIに出題するのは、正直少し怖かった。結果は「一発で正答、ただし別解」。悔しさと安堵が半々の、予想より味わい深い着地だった。

今回試したのはAnthropicの2モデルのみなので、他のモデルがどの格子に向かうのかは分かっていない。全モデルがバイト格子に収束するのか、どこかに周期4を見つけるモデルがいるのか。検証ハーネスは前回同様GitHubに置いてあるので、もしよろしければ、お手元のモデルで試した結果など教えていただけると嬉しい。
