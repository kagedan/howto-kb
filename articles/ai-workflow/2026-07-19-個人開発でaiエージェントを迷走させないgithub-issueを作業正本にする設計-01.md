ChatGPTで要件を整理し、Codexにコードを調べてもらい、Cursorで実装する。

AIエージェントを使うと、調査や実装を短時間で進められる場面があります。その一方で、ツールやセッションをまたぐたびに、同じタスクを少しずつ違う言葉で説明し直すことになります。

そして、しばらくすると分からなくなります。

* どの会話が正式な依頼だったのか
* どこまで変更してよいのか
* 何ができれば完了なのか
* 作られたPRは、どの依頼に応えたものなのか

自分の個人開発でも、この問題にぶつかりました。そこで、構想や会話ではなく、GitHub Issueを「正式な作業の入口」にすることにしました。

結論から言うと、IssueをTODOではなく、AIと人間が共有する軽量な「作業契約」として扱います。

この記事では、Issueへ何を書き、ObsidianやChatGPTのメモとどう分け、AI、PR、CIへどうつなぐかを説明します。読み終える頃には、次の小さな開発タスクをAIへ渡せるIssueとして書ける状態を目指します。

> この記事は、自分の個人開発における運用設計と試行を一般化したものです。定量的な生産性向上や品質改善を検証した記事ではありません。

## AIを増やすほど、作業の基準が必要になる

### 会話は便利だが、正式な依頼には向いていない

チャットは、曖昧なアイデアを広げたり、選択肢を比較したりするのに向いています。しかし、会話にはそのセッション固有の文脈があります。

ある会話では「まず調査だけ」と依頼し、別の会話では実装まで頼んでいた。途中で方針を変えたものの、別のAIは以前の前提を参照していた。

このずれはAIの性能だけでなく、正式な依頼と判断基準が、会話の外へ出ていないことによっても起こります。

### 曖昧な依頼では、差分の正しさを判定できない

たとえば、AIへ次のように依頼したとします。

> 表示名を変更できるようにして。

これだけでは、設定画面だけを変えるのか、APIやデータも変えるのか、何文字まで入力できるのか、どのテストが必要なのかを判断できません。

AIが不足した前提を補い、動くコードを作ったとしても、それが依頼どおりかは別の問題です。

コードを書き終えたことと、タスクが完了したことは同じではありません。期待する振る舞いを満たし、必要なテストが通り、対象外の変更がなく、レビューできる状態になって初めて完了を判断できます。

一人で開発していても、AIを使うほど関与する「作業者」は増えます。だからこそ、チーム開発ほど重くなくても、作業単位と判断条件を外へ出す必要があると考えています。

## GitHub Issueを正式作業の入口にする

本記事でいう作業正本とは、そのタスクの目的、対象範囲、入力、完了条件を確認するときに、最初に参照する記録です。

GitHub Issueはリポジトリの近くにあり、Branch、PR、CIと結びつけやすい場所です。背景、目的、対象範囲、対象外、受け入れ条件、レビュー観点を一つの作業単位として残せます。

**作業正本を中心にした開発フロー**

Issueは、前段の曖昧な材料を作業条件へ変換し、後段の差分と検証結果を照合するときの基準になります。

ただし、Issueへすべてを集めるわけではありません。「何をどこで確認するか」を決めます。

| 確認したいこと | 主な正本 |
| --- | --- |
| なぜ、どこまで、何をもって完了とするか | GitHub Issue |
| 今どの状態か | GitHub Projects |
| 何を変更し、どう検証したか | Pull Request |
| 長期的に正しい仕様・設計は何か | リポジトリ内の仕様書 |

Issueと仕様書、PRへ同じ文章を複製すると、どれかが古くなります。Issueには作業に必要な条件と、参照すべき正本への入口を置きます。

作業単位は、まず `1 Issue = 1 Branch = 1 PR` を目安にします。絶対的な規則ではありませんが、一つのPRで閉じられないIssueは、AIへ渡す前に分割を考えるサインになります。

## 良いIssueは、AIが「進む・聞く・止まる」を判断できる

AI向けのIssueは、長ければよいわけではありません。重要なのは、AIが次のどれかを判断できることです。

1. 必要な情報があるので着手する
2. 不足や矛盾があるので質問する
3. 権限や範囲を超えるので停止する

**AIの着手判断**

AIに期待するのは、常に前進することではありません。曖昧さを検出したときに質問し、境界を越えそうなときに止まれることも、Issueへ条件を書く目的です。

Issueへ持たせる情報を整理すると、次のようになります。

| 項目 | 役割 |
| --- | --- |
| 背景・目的 | なぜ変更し、何を実現するのかを共有する |
| Scope | 今回変更してよい機能や責務を決める |
| Out of Scope | 今回は変更しない範囲を決める |
| 受け入れ条件 | 完了を外から確認できる形にする |
| 参照資料・変更対象 | 読む資料と探索・変更範囲を絞る |
| テスト・レビュー観点 | 実行する検証と重点確認箇所を決める |
| AIへの注意事項 | 推測や独自判断を避ける箇所を示す |
| 停止条件 | 人間へ判断を戻す条件を決める |

特に重要なのは、Scope、Out of Scope、受け入れ条件、停止条件です。

受け入れ条件には実装方法ではなく、外から確認できる期待結果を書きます。停止条件には、仕様の矛盾、想定外のデータ変更、対象外の機能変更など、AIが独自判断で進まない方がよい状況を書きます。

小さな変更なら、次のテンプレートから始められます。

```
## 背景・目的

## Scope

## Out of Scope

## 受け入れ条件

## 参照資料・変更対象

## テスト・レビュー観点

## AIへの注意事項・停止条件
```

## 表示名変更をIssueからPR・CIまでつなぐ

ここからは、架空の個人開発アプリへ「プロフィールの表示名変更」を追加する例を使います。実在するIssue、PR、ファイル構成とは関係ありません。

### 1. Issueで作業の境界を決める

```
## 背景・目的

ユーザーが設定画面から表示名を変更できるようにする。

## Scope

- 設定画面
- 表示名更新処理
- 関連テスト

## Out of Scope

- メールアドレスの変更
- 認証方式の変更
- プロフィール画像の変更

## 受け入れ条件

- [ ] 1〜30文字の表示名を保存できる
- [ ] 未入力時にエラーが表示される
- [ ] 31文字以上ではエラーが表示される
- [ ] 保存前の既存表示名が初期表示される
- [ ] 保存後の表示名がプロフィールへ反映される

## テスト・レビュー観点

- 未入力と31文字以上の境界値
- 既存値を表示する処理への回帰
- Scope外の変更が含まれていないこと

## 停止条件

- データ構造の変更が必要と判明した
- 既存仕様と入力制約が矛盾する
- 認証方式の変更が必要になる
```

これでAIは「何を作るか」だけでなく、「何を変えないか」と「どの状況で止まるか」を確認できます。

### 2. AIにはIssueと作業条件を渡す

Issue本文に加え、開始時の入力と終了時の出力を短く指定します。

> 対象Issueと参照資料を読み、指定された範囲だけを変更してください。受け入れ条件に対応するテストを追加・実行し、変更内容と結果を報告してください。不足・矛盾・停止条件への該当を見つけた場合は、推測で補わず作業を止めて質問してください。

この指示は特定のAI製品専用ではありません。長い会話履歴を再現する代わりに、Issueを中心に再現できる依頼を作ることが目的です。

### 3. PRには「条件を満たしたか」を書く

PRではコードの要約だけでなく、Issueの受け入れ条件へどう対応したかを記録します。

```
## 変更内容

- 設定画面へ表示名の入力・保存処理を追加
- 入力値のバリデーションを追加

## 受け入れ条件との対応

- [x] 1〜30文字を保存できる
- [x] 未入力時にエラーが表示される
- [x] 31文字以上ではエラーが表示される
- [x] 既存表示名が初期表示される
- [x] 保存後の表示名がプロフィールへ反映される

## テスト結果

- 未入力時のバリデーションテスト: 成功
- 文字数上限のテスト: 成功
- 既存値を表示するテスト: 成功

## レビューで確認してほしいこと

- Scope外の変更が含まれていないこと
- 境界値の扱いが受け入れ条件と一致すること
```

チェックを付けたこと自体が完了の証拠ではありません。差分、テスト結果、人間または独立したレビューによって照合します。

### 4. 受け入れ条件と検証を対応させる

| 受け入れ条件 | 対応する検証 | PR・CIで見ること |
| --- | --- | --- |
| 未入力時にエラーが表示される | バリデーションテスト | テストが追加され、CIで成功している |
| 31文字以上ではエラーが表示される | 境界値テスト | 上限の前後が検証されている |
| 既存表示名が初期表示される | コンポーネントテスト | 既存表示への回帰がなく、CIで成功している |
| 保存後の表示名が反映される | 結合テストまたは手動確認 | 保存から表示までの経路を確認している |

これは対応関係の例です。すべてを自動テストにするのではなく、自動テストと人間のレビューのどちらで確認するかをPR上で明示します。

**受け入れ条件のトレーサビリティ**

CIは完了判定の一部です。最終的には、Issueに書いた条件と、差分・テスト結果を人間が照合します。

## すべての思いつきをIssueにしない

Issueを正式な作業の入口にするといっても、すべての思いつきを即座にIssueへ変える必要はありません。

自分の運用では、Obsidian、ChatGPTとの会話、音声メモを「Issue化前の素材」と位置づけました。この段階には、仮説、調査メモ、未確認のアイデア、複数の選択肢が混ざっていて構いません。

GitHub Issueへ昇格させる前に、次を確認します。

* 目的と期待する結果を説明できるか
* 必要な仕様や入力がそろっているか
* 一つの作業単位として扱える粒度か
* 依存する作業や競合する変更がないか
* 受け入れ条件と停止条件を書けるか

**Issueへの昇格ゲート**

Issueはメモの置き換えではなく、着手可能になった作業だけを通すゲートです。

個人開発では、依頼者も実装者も自分です。それでも、思いついた時点の自分と、実装を判断する時点の自分を分けることには意味があります。

## AIへ渡す文脈は、多さではなく境界で決める

Issueを書いた後も、そのURLや本文だけをAIへ渡せば十分とは限りません。

AIが読むべき資料、触れてよい範囲、完了時に返す情報をまとめます。本記事では、このまとまりを便宜上 `Context Packet` と呼びます。特定製品の正式な標準ではありません。

```
Context Packet
├─ 対象Issue
├─ 参照必須の仕様書
├─ 探索・変更してよい範囲
├─ 参照・変更しなくてよい範囲
├─ 実行すべきテスト
├─ AIが独自に決めてはいけない事項
└─ 不足・矛盾・競合がある場合の停止条件
```

文脈は多ければ多いほど安全、というわけではありません。リポジトリ全体を毎回渡すのではなく、そのタスクの判断に必要な資料を選び、不要な領域を対象外として示します。

作業終了時には、AIへ次の情報を求めます。

* 変更した内容
* 受け入れ条件への対応
* 実行したテストと結果
* 未解決事項
* PRで重点的に確認してほしい点

次のAIやレビュー担当へ渡すのは、長い会話履歴ではなく、Issue、差分、テスト結果です。利用するサービスやセッションが変わっても、同じ作業条件から確認を再開しやすくなります。

## 価値は自動化より、再開と照合のしやすさにある

ここで挙げる内容は、定量的に効果測定した結果ではありません。自分の運用設計と試行から得た、定性的な学びです。

* **着手前に不足へ気づきやすい**：目的、対象範囲、受け入れ条件をIssueへ書くことで、人間が判断する機会を作れる
* **PRで結果を照合しやすい**：受け入れ条件をPRのチェック項目として再利用できる
* **複数のAIを使っても作業正本が散らばりにくい**：正式な作業条件をIssueへ戻せる
* **作業を再開しやすい**：別のセッションでも、同じ目的と条件から実装やレビューを始められる

ここでいう再現性は、同じAIが同じコードを生成することではありません。途中で作業を止めても、別のセッションから同じ目的と条件を確認できる、という意味です。

## この運用だけでAIの誤りは防げない

Issueを作業正本にしても、AIの誤りがなくなるわけではありません。

* **Issue作成を重くしすぎない**：小さな変更では、目的、Scope、Out of Scope、受け入れ条件、停止条件から始める
* **粒度を極端にしない**：一つの責務として閉じられる大きさを目安にする
* **人間による確認を残す**：Issueは自動運転の許可証ではなく、作業と検証の共通基準である
* **並列実行時は論理的な競合も見る**：Branchやworktreeだけでなく、同じファイル、生成物、契約、データ変更への競合を管理する

## 明日から試すなら、5項目だけでいい

最初から複数AgentやCI/CDの自動化まで作る必要はありません。

次の小さなタスクを一つ選び、まずは5項目だけを書きます。

1. 目的
2. Scope
3. Out of Scope
4. 受け入れ条件
5. 停止条件

そのIssueをAIへ渡し、PRで受け入れ条件と結果を照合する。

うまくいかなければ、AIへの指示だけでなく、Issueに足りなかった条件を見直します。次のタスクでは、その学びをテンプレートへ戻します。

ObsidianやChatGPTは構想の入口、GitHub Issueは正式作業の入口。この二つを分ける小さな一周から、自分の開発に合う運用へ調整していくのがよいと考えています。
