## はじめに

以前、AIエージェントに「段取り」を持たせるためのOSSとして、Dandoriを公開しました。

前回の記事はこちらです。

リポジトリはこちらです。

公開後も、自分でDandoriを使い続けていました。

Workerへ曖昧な依頼をそのまま渡さず、Task Cardで作業範囲を限定する仕組みは、想定していたとおり機能しました。

頼んでいない修正を始めたり、関係の薄い範囲まで調査を広げたりする動きは抑えやすくなりました。

ただ、使い続けるうちに別の問題が気になり始めました。

**承認が重い。**

Task Flow Reviewに多くの情報を載せていたため、確認に時間がかかります。

調査後に少し段取りを変えるだけでも、再承認が必要になることがありました。

安全のために表示している内容なのに、長すぎて自分でも流し読みしそうになる。

これでは、承認画面を用意している意味が薄れてしまいます。

そこで、Dandoriの承認フローを作り直しました。

今回見直したのは、承認の有無ではありません。

**ユーザーが何を承認するのか**です。

## Dandoriとは

Dandoriは、GitHub Copilot Custom Agentsを使ったマルチエージェント構成に、司令塔を追加するための仕組みです。

エージェントを大きく2つに分けています。

| 種別 | 役割 |
| --- | --- |
| Orchestrator | 依頼の整理、承認、Workerへの委譲、結果確認を行う |
| Worker | OrchestratorからTask Cardを受け取り、実作業を行う |

全体像は次のようになります。

Orchestratorは、ユーザーの依頼をそのままWorkerへ渡しません。

作業範囲を整理し、ユーザーへTask Flow Reviewを提示します。

承認後、Workerが実行できる範囲をTask Cardとして切り出し、作業を委譲します。

Dandoriで一貫しているのは、次のルールです。

```
Workerは、境界付きのTask Cardなしに作業しない。
```

Task Cardは単なる依頼文ではありません。

何を行うのか、どの対象を扱えるのか、どこで完了とするのか、どの条件で止まるのかを定めたWorker向けの実行契約です。

## 最初の設計は、計画を固定しすぎていた

公開当初のDandoriでは、Task Flow Reviewに実行手順まで細かく載せていました。

たとえば、次のような内容です。

* どのWorkerを使うか
* どの順番で進めるか
* どのファイルを扱うか
* どの操作を許可するか
* 何を完了条件とするか
* どの条件で停止するか

Workerへ渡すTask Cardも、承認されたTask Flow Reviewの各ステップからはみ出してはいけない設計でした。

```
Internal Task Card ⊆ approved Task Flow Review step
```

Workerへ渡す範囲を厳しく制限できる一方で、作業開始後の調整が難しくなります。

コードを調査してみると、最初の想定どおりに進まないことはよくあります。

たとえば、不具合修正を依頼した時点では1ファイルの変更で済むと思っていても、調査後に関連するテストや呼び出し元の修正が必要だと分かることがあります。

別のWorkerへ任せた方がよいと分かることもあります。

Task Cardを分けたり、追加の確認を挟んだりした方が進めやすい場合もあります。

これらは、依頼の目的や許可範囲を変える話ではありません。

目的へ到達するまでの段取りが変わっただけです。

それでも、以前の設計では実行手順そのものを承認対象にしていたため、再承認が発生することがありました。

## 承認する内容と、内部の段取りを分けた

ユーザーが確認したいのは、主に次の内容です。

* 何をするのか
* 何が成果として返ってくるのか
* どこまで調査・変更してよいのか
* ファイル変更やコマンド実行など、何を許可するのか
* どのような場合に再確認が必要なのか

一方、次の内容はOrchestrator側の段取りです。

* どのWorkerへ任せるか
* どの順番で呼び出すか
* Task Cardを何枚に分けるか
* どのタイミングで調査や確認を挟むか

以前は、この2つを同じTask Flow Reviewへ入れていました。

今回の変更では、ユーザーが承認する作業契約と、Orchestratorが内部で管理する計画を分けました。

新しい包含関係は次のとおりです。

```
Worker execution
  ⊆ Task Card
  ⊆ Active Approved Contract
  ⊆ Approved TFR/TFC chain
```

WorkerはTask Cardを越えられません。

Task Cardは、ユーザーが承認した作業契約を越えられません。

一方で、Task CardをTask Flow Reviewの個別ステップには固定しません。

承認された契約の内側であれば、Orchestratorは次の内容を調整できます。

* Workerの選択
* 実行順序
* Task Cardの分割や統合
* 追加調査
* 検証の追加
* 承認済み上限内での対象追加

ユーザーが許可した範囲は変えず、その内側の段取りだけを組み替える設計です。

今回、この設計を **Bounded Adaptive Orchestration（境界付き適応型オーケストレーション）** と名付けました。

## Task Flow Reviewを短くした

新しいTask Flow Reviewは、詳細な実行計画ではありません。

ユーザーが今回の依頼で「どこまで任せるか」を確認するための画面です。

たとえば、次のように表示します。

```
## Task Flow Review: TFR-ab12

**目的**
指定された不具合を修正する

**成果**
原因の説明、必要な変更、実施できた検証結果

**作業境界**
- 調査: 対象リポジトリ内
- 変更: 明示された対象と、根拠を確認できた既存対象
- 自動追加: 個別に特定できる変更対象を最大5件

**許可する効果**
- observe
- change_local

**検証**
- 永続的な変更は、可能な場合に別の呼び出しで確認する

**再承認**
目的、成果、境界、許可する効果、自動追加上限、
検証水準を広げる場合のみ
```

Worker名や実行順序は表示しません。

内部のTask Cardや状態管理も表示しません。

ユーザーは、目的、成果、作業範囲、許可する効果、検証方法だけを確認します。

承認には、表示言語に依存しないトークンを使います。

説明は会話中の言語に合わせて表示します。

言語を判断できない場合は、英語を使用します。

## 再承認は、作業契約を広げる場合だけ

作業中に契約を広げる必要が出た場合は、再承認を求めます。

たとえば、次のような場合です。

* 目的や成果物を変える
* 調査・変更範囲を広げる
* 新しい種類の効果を許可する
* 自動追加できる対象数を増やす
* 検証水準を下げる
* 外部サービスへ書き込む
* 削除などの破壊的な操作を行う

再承認が必要になった場合も、Task Flow Review全体は表示し直しません。

変更点だけをTask Flow Changeとして提示します。

```
## Task Flow Change: TFC-cd34

**理由**
現在の自動追加上限を超える変更対象が必要になった

**変更**
- 自動追加上限: 5件 → 7件

**変更しないもの**
目的、成果、許可する効果、検証水準
```

承認する場合は、次の1行だけを返します。

これなら、変更されていない内容まで毎回読み直す必要がありません。

承認済みの契約にはRevisionを持たせています。

途中で契約が変わっても、古い条件で実行したWorkerの結果が、そのまま新しい契約の完了判定に使われないようにするためです。

## 発見した対象を、そのまま変更させない

Workerが調査中に新しい対象を見つけても、その呼び出しの中で勝手に変更することはできません。

たとえば、調査中に別のファイルも修正が必要だと分かった場合、Workerはそのファイルを候補として返します。

Orchestratorは、次のような情報を確認します。

* ファイルを個別に特定できるか
* 承認済みの作業境界内か
* 成果条件と関係しているか
* 根拠となる位置が示されているか
* 必要な効果がすでに許可されているか
* 自動追加上限を超えていないか

条件を満たした場合だけ、次のTask Cardで変更を許可します。

「関連していそう」「同じディレクトリにある」「一般的には直した方がよい」といった理由では追加しません。

また、ディレクトリ全体や「関連ファイル一式」を1件として扱うこともできません。

自動追加の対象は、ファイルやIssueのように、個別に特定し、変更理由と結果を1件ずつ記録できる単位に限定します。

## 操作名だけでなく、起こり得る効果を見る

コマンドを実行するといっても、何が起きるかはコマンドによって異なります。

読み取りだけで終わるものもあれば、ファイルを書き換えるものもあります。

外部APIへ書き込む可能性もあります。

そのためDandoriでは、操作名だけでなく、実行によって起こり得る効果を管理します。

```
observe
change_local
execute
affect_external
destructive
```

これらは、どれか1つを選ぶ分類ではありません。

複数の効果が発生する場合は、すべての許可が必要です。

たとえば、ファイルを変更するコマンドには、`execute`と`change_local`の両方が必要です。

```
allowed_effects:
  - execute
  - change_local
```

コマンド実行を許可したからといって、その結果として発生するファイル変更まで自動的に許可されるわけではありません。

起こり得る効果を判断できない場合は、そこで止めるか、追加承認を求めます。

## Worker側にDandoriの内部状態を持たせない

今回変更したのは、Orchestratorの設計です。

Worker側へ、Dandori固有の承認状態や契約Revision、ルーティング規則は追加していません。

Orchestratorにも、個別Workerの役割表や固定ルーティングを持たせていません。

実行時に利用可能なWorkerから候補を選び、そのWorkerの定義とTask Cardが明確に矛盾していないかを確認します。

Worker選択は実行結果の品質には影響しますが、承認範囲を広げる理由には使いません。

Workerは、渡されたTask Cardの範囲内で作業し、結果を返します。

そのため、Dandoriに同梱しているWorkerだけでなく、すでに使っているCustom AgentもWorkerとして再利用できます。

必要に応じて利用可能なAgent一覧へ追加するだけで、Orchestratorの制御ロジック自体は変更しません。

## 変更後は別のコンテキストで確認する

変更を行ったWorkerが「完了しました」と報告しただけでは、検証済みとは扱いません。

可能な場合は、変更時とは別のサブエージェント呼び出しで確認します。

確認用のTask Cardでは変更を許可せず、次の点だけを確認します。

* 成果条件を満たしているか
* Task Card外の変更がないか
* 完了を否定する事実が残っていないか

ただし、これは完全に独立した第三者レビューではありません。

同じモデルや同じCustom Agentが選ばれる可能性もあります。

そのため、Dandoriでは「独立検証」ではなく「別コンテキスト検証」と呼んでいます。

検証手段がない場合は、無理に検証済みとはしません。

変更は完了しているが、別コンテキストでの確認はできていない、と分けて報告します。

## トークン効率も見直した

前回の記事でも書きましたが、トークン効率はプロンプトを短くすることだけではありません。

不要な調査や同じ作業の繰り返しも、そのままトークン消費になります。

今回の変更では、次のような処理を減らしました。

* 長いTask Flow Reviewを毎回生成する
* 内部計画の変更で再承認する
* すべてのWorker定義を毎回比較する
* Workerの出力を巨大な固定形式へ変換させる
* 対象が明確なのに調査工程を挟む
* 進捗がないまま同じTask Cardを繰り返す
* Workerの回答全文を次のWorkerへ渡す

Task Cardも、工程ごとに細かく分割するのではなく、同じ対象と権限で実施できる作業は1枚にまとめます。

一方、対象の発見と変更、成果物の作成と検証など、権限の境界が変わる部分は分けます。

Workerを呼ぶ前には、その呼び出しで何が新しく分かるのか、何が完成するのかをOrchestrator内部で確認します。

新しい事実、成果物、認可対象、検証結果、具体的な停止理由のいずれも得られない呼び出しは繰り返しません。

## 変えなかったこと

承認フローは変えましたが、Dandoriの基本方針は変えていません。

* Orchestratorが全体を制御する
* Orchestrator自身は実作業を行わない
* WorkerはTask Cardなしに動かさない
* WorkerはTask Cardの外側へ作業を広げない
* Workerの結果を確認してから次へ進む
* Worker側にDandoriの内部詳細を持たせない

変えたのは、細かな実行手順までユーザー承認の対象にしていた部分です。

以前は、安全性を高めるために計画を細かく固定していました。

現在は、ユーザーが許可した契約を固定し、その内側の段取りはOrchestratorが調整します。

## Dandoriはサンドボックスではない

Dandoriは、OSレベルの権限制御ではありません。

Task Cardに書かれていない対象へのアクセスを、技術的に不可能にする仕組みでもありません。

Dandoriが行うのは、Workerへ渡す作業を狭くし、範囲外の行動を見つけやすくすることです。

* Workerには狭いTask Cardを渡す
* 発見と変更を分ける
* 許可した効果を明示する
* Workerが扱った対象と効果を確認する
* 契約内か判断できない場合は止める

Workerの逸脱を完全に防ぐのではなく、逸脱しづらくし、起きた場合に止めやすくするための仕組みです。

## まとめ

Dandoriを公開した当初は、Workerを承認範囲から出さないことを重視しました。

実際に使ってみると、実行手順まで細かく承認対象にしたことで、Task Flow Reviewが長くなり、再承認も増えていました。

今回のアップデートでは、ユーザーが承認する作業契約と、Orchestratorが管理する段取りを分けました。

ユーザーが確認するのは、目的、成果、作業境界、許可する効果、検証方法です。

その契約を広げない限り、OrchestratorはWorkerの選択や実行順序、Task Cardの構成を調整できます。

Workerへ渡すTask Cardは、引き続き狭く保ちます。

```
短い承認
＋
固定された作業契約
＋
状況に応じて変えられる段取り
＋
境界付きTask Card
```

最初に作ったDandoriの考え方を残しつつ、実際に使い続けやすい形へ調整しました。
