## はじめに

「自然言語でデータを分析できる」──デモでは魅力的に見えても、 **本番導入に足る品質をどう担保するか、そもそも担保しきれるのか** と疑問に感じたことはないでしょうか。

この記事では、 **自然言語でデータを分析するAIエージェントが実務で安定しにくい理由** を考察しつつ、近年話題の **セマンティックレイヤーとの組み合わせにより、そうした課題を緩和しうる可能性** について紹介します。

題材にはBigQueryの公開データセット `bigquery-public-data.thelook_ecommerce` を使います。手元で再現できるよう実テーブルをそのまま使っています。

---

## なぜ分析エージェント単体では実務で安定しないのか

AIエージェントに自然言語で質問してデータを分析する、いわゆる分析エージェントはデモ映えする一方、実務投入においてはいくつかの課題があるように思います。

### 問題1：ビジネス定義がBIツールに閉じており、AIから見えない

「売上」をどう計算するかは、テーブルのカラム名だけでは分かりません。

* `SUM(sale_price)` をそのまま合計するのか（グロス売上）
* 返品された注文を除くのか（ネット売上）
* キャンセルも返品も両方除くのか
* 割引前の `retail_price` を使うのか、実際の `sale_price` を使うのか

これは会社・チームごとに異なる **ビジネスロジック** です。多くの企業では、こうした分析指標定義はTableauのCalculated FieldやLookerのLookMLの中に蓄積されています。

AIエージェントはTableauのワークブックを読めないため、内部のLLMはこの定義をテーブル定義から推測するしかありません。結果、「今日は返品除き、昨日は返品込み」という異なる結果が返ってくるリスクがあります。「売上の数字が毎回違う」はデータ分析において致命的な問題であり、 **長年かけて磨いた指標定義がAI活用の文脈では存在しないも同然** になってしまいます。

### 問題2：テーブル変更に弱い

テーブルのカラム名やスキーマが変わるたびに、AIエージェント側のLLMプロンプトや設定（スキーマ記述・マッピング等）も修正する必要があります。修正にはメンテナンスコストもかかりますが、根深い問題点としては **古い定義のまま動き続ける可能性がある** ことだと考えています。

通常のバッチ処理等では、存在しないカラムを参照するSQLは実行時エラーになるため、その時点で比較的早期に異常に気づけるはずです。ところがAIエージェント（LLM）にSQLを生成させる経路では壊れ方が特殊で、仮に古いカラム名が使えなくなっても、似た名前の別カラムで辻褄を合わせたSQLを生成してしまうことがあります。するとエラーにはならず、ただ **間違った数字を返し続ける** ということになります。これは出力が揺れるLLMの特性も相まって問題が顕在化しにくく、誰かが「この数字、本当に合ってる？」と深追いするまで潜伏しがちです。

データ基盤の変更とAIの定義管理が別チームの別リポジトリで進む現場では問題検知のきっかけ自体が得にくく、この種の問題が起きやすい構造にあると考えています。

### 問題3：LLMのSQL生成（text2SQL）は本質的に「揺れる」

上の2つはデータ基盤の整備で緩和できますが、text2SQL特有のこの問題はより根本的です。

LLMはトークンを確率的に生成するため、同じ質問に対しても毎回まったく同じSQLが返ってくる保証はありません。その結果、クエリの書き方やフィルタ条件の解釈がわずかに変わるだけで、返ってくる数字が変わりえます。これは「たまに間違える」ではなく、 **生成AIの仕組み上、避けられない非決定性** です。

財務・EC・在庫など、数字の正確さが求められるデータ分析において、この非決定性は致命的となります。「昨日と今日で同じ質問をしたら違う数字が返ってきた」という状況は、ビジネスの意思決定の場では許容されません。

### 問題4：定義情報をコンテキストに詰め込んでも処理しきれない

出力を安定させるべく、スキーマ定義やビジネスロジックをプロンプトに渡す、という発想もありますが、これも限界があります。

実際の業務データベースには、数十〜数百のテーブル、カラムの説明、テーブル間のリレーション、指標の計算ロジック、フィルタの前提条件……と、膨大な定義情報があります。これをすべてコンテキストに詰め込むと以下のような問題が発生します。

* コンテキストウィンドウを圧迫し、会話履歴や質問の意図を処理する余地が減る
* 情報が多すぎてLLMが適切な定義を選べず、むしろ精度が下がる（Lost in the Middle問題）
* 毎回大量のトークンを消費し、コストと遅延が増大する

RAGでスキーマ情報を検索して渡す工夫もありますが、「どの定義を渡すべきか」の判断自体が難しく、漏れが生じれば結局おかしなSQLが生成されます。

上記を踏まえると、そもそも **LLMにスキーマを理解させてSQLを書かせるというアプローチ自体に限界** があるように思います。LLMが本来得意なのは「自然言語を理解し、人間と会話すること」であり、決定的なSQL実行はその外側の仕事です。

## セマンティックレイヤーの導入

### Before：意味の定義がBIツールの中に閉じている

現場では、「ネット売上」「返品率」などのビジネス定義がBIツール内の計算ロジックに蓄積され、そこに閉じているケースも多いと思います。AIエージェントはワークブックやモデルを直接読めず、同じ定義にはアクセスできません。

そのため、AIエージェントが直接DWHに問い合わせする際は、AI側がビジネス定義を推測しながらSQLを構築することになります。

### After：意味の定義をDWHの上位層に置き、全ツールから参照する

セマンティックレイヤーをDWHとアプリの間に独立して立てることで、 **指標定義の一元管理が可能** となります。

BIツール内に閉じていた計算ロジックをセマンティックレイヤーへ移し、これまで蓄積してきた指標の意味づけを載せれば、BIツールもAIエージェントも同じ定義を参照できます。例えば、「ネット売上」「返品率」「平均注文単価」などのメトリクスがここに揃っていれば、AIエージェントはテーブル構造から指標を推測する作業から解放される訳です。

dbt Labsのベンチマークでは、セマンティックレイヤーを経由しない素のSQL生成の正答率が **16.7%** だったのに対し、セマンティックレイヤー経由では **83%** まで改善したと報告されており（[Semantic Layer as the Data Interface for LLMs](https://www.getdbt.com/blog/semantic-layer-as-the-data-interface-for-llms)）、この層の有無は精度に大きな影響を与える事が知られています。

## AIエージェントにSQLを書かせない──役割分担の明確化

ここで設計上の重要な判断を述べておくと、 **AIエージェントはSQLを生成せず、人間との会話に専念させます。**

具体的には、AIエージェント（LLM）に担わせる役割は以下の通りにします：

* ユーザーとの対話により、知りたい内容を整理する
* 調べたい情報をセマンティックレイヤーに問合せする
* 結果を自然言語で要約・解釈して返す

一方、SQLの生成と実行は **セマンティックレイヤーが引き受けます** 。  
AIエージェントがセマンティックレイヤーに渡すのは「何のメトリクスを、どのディメンションで集計するか」というクエリの意図だけです。実際にどんなSQLが走るかはセマンティックレイヤーが決定します。

この分担により、 **AIエージェントが関与する部分で揺れが出ても、数字の計算ロジックには影響しない** という構造になります。AIエージェント内のLLMが多少違うメトリクス名を選んだとしても、定義済みのメトリクス以外は存在しないのでセマンティックレイヤー側でエラーになるだけです。アドホックなSQLがこっそり実行されることはありません。

### セマンティックレイヤー製品群

この設計を実現するには、セマンティックレイヤーが **APIとして外部から呼び出せる** ことが必要となってきます。

**プラットフォーム非依存型**（BigQueryを含む任意のDWHと組み合わせ可能）

| ツール | API提供 | 特徴 |
| --- | --- | --- |
| **Cube Core** | REST / GraphQL | OSS、セルフホスト可能、BigQueryネイティブ対応 |
| **dbt Semantic Layer** | JDBC / GraphQL | dbt環境があれば自然に統合できる。ただしBIツールからのセマンティック利用はdbt Cloud（有償）に限定 |
| **AtScale** | REST / MDX | エンタープライズ向け、既存BIとの併用が容易 |

**プラットフォーム内蔵型**（それぞれのDWH内でセマンティックレイヤーが完結する）

| ツール | API提供 | 特徴 |
| --- | --- | --- |
| **Databricks AI/BI（Genie）** | REST | Unity Catalog 上に Metrics View で定義・参照可能 |
| **Snowflake Cortex Analyst** | REST | Semantic Model 定義・参照可能 |

プラットフォーム内蔵型は、DWHとセマンティックレイヤーが一体化しているため運用がシンプルになりやすい一方、BigQueryのような別プラットフォームへ機能をシームレスに持ち越せるわけではありません。データや定義の移し替え、連携経路の別建てなど、追加の設計・運用コストが乗りがちです。  
任意のDWHを前提とする場合は、プラットフォーム非依存型が選ばれやすいという整理になるかと思います。

!

**補足：Open Semantic Interchange（OSI）の動き**

ツール間の差分是正に関する取り組みとして、セマンティックレイヤーの定義形式を標準化する **Open Semantic Interchange（OSI）** が進んでおり、「プラットフォーム内蔵型は移行性が低い」と単純に言い切れない面もあります。標準化がどこまで進むかはまだ流動的ですが、ツール選定の際には頭に入れておくと良いと思います。

また、LookMLやdbtにあるような **定義情報のみをファイルとして持つ構成は、AIエージェントが「参照するドキュメント」としては使えますがSQL実行の委譲はできません。** これは「LLMに定義を読ませてSQLを生成させる」形に戻ってしまい、非決定性の問題が再発します。

セマンティックレイヤーを **「AIエージェント（LLM）が呼び出せるSQL実行サービス」として立てる** ──この点がアーキテクチャの強みの部分となります。

## TheLook Ecommerceのデータ構造

BigQueryの公開データセット `bigquery-public-data.thelook_ecommerce` には、架空のアパレルECサイトの注文・ユーザー・商品データが入っています。

**参考（公式）**

この構造で「売上」の計算に対しても取り方は複数ありえます。

* `SUM(sale_price)` — グロス売上（返品・キャンセル含む）
* `SUM(sale_price) WHERE status NOT IN ('Cancelled', 'Returned')` — ネット売上
* `AVG(sale_price)` — 商品単価の平均（注文単価ではない）

どれをこの文脈の「売上」とみなすかはビジネスルールの問題で、テーブルとカラム名だけでは決まりません。この定義をCubeに書いておけば、AIエージェントは推測しなくてよくなります。

## Cube Core のモデル定義

今回のセマンティックレイヤーにあたる Cube Core モデル定義（`model/order_items.js`）の記述例です。Cube Core では事前に定義したモデルに応じて、問い合わせからSQLを構築し結果を返却してくれます。

**参考（公式）**

Cube Core モデル定義（OrderItems / Products / Users）

*※ 動作することを意図したサンプルコードです。十分な精査は行っていません。*

`model/order_items.js`

```
cube(`OrderItems`, {
  sql_table: `bigquery-public-data.thelook_ecommerce.order_items`,

  joins: {
    Products: {
      sql: `${CUBE}.product_id = ${Products}.id`,
      relationship: `many_to_one`,
    },
    Users: {
      sql: `${CUBE}.user_id = ${Users}.id`,
      relationship: `many_to_one`,
    },
  },

  measures: {
    // ネット売上（返品・キャンセルを除く）
    net_revenue: {
      sql: `${CUBE}.sale_price`,
      type: `sum`,
      filters: [{ sql: `${CUBE}.status NOT IN ('Cancelled', 'Returned')` }],
      title: `ネット売上`,
    },

    // 注文数（ユニーク注文ID）
    order_count: {
      sql: `${CUBE}.order_id`,
      type: `countDistinct`,
      title: `注文数`,
    },

    // 返品率（完了＋返品の中で返品の割合）
    return_rate: {
      sql: `
        COUNTIF(${CUBE}.status = 'Returned')
        / NULLIF(COUNTIF(${CUBE}.status IN ('Complete', 'Returned')), 0)
      `,
      type: `number`,
      title: `返品率`,
      format: `percent`,
    },

    // 平均注文単価（ネット売上 / ユニーク注文数）
    average_order_value: {
      sql: `
        SUM(CASE WHEN ${CUBE}.status NOT IN ('Cancelled', 'Returned')
                 THEN ${CUBE}.sale_price END)
        / NULLIF(COUNT(DISTINCT CASE WHEN ${CUBE}.status NOT IN ('Cancelled', 'Returned')
                                     THEN ${CUBE}.order_id END), 0)
      `,
      type: `number`,
      title: `平均注文単価`,
    },
  },

  dimensions: {
    id: {
      sql: `${CUBE}.id`,
      type: `number`,
      primary_key: true,
      title: `ID`,
    },
    status: {
      sql: `${CUBE}.status`,
      type: `string`,
      title: `注文ステータス`,
    },
    created_at: {
      sql: `${CUBE}.created_at`,
      type: `time`,
      title: `注文日時`,
    },
  },
});
```

## AIエージェント側の実装

Gemini APIのfunction callingを使い、Cubeの`/cubejs-api/v1/load`エンドポイントをツールとして定義します。

**参考（公式）**

**ディレクトリ構成例**

```
├── cube/
│   ├── docker-compose.yml
│   ├── .env                 # Cube Core 環境設定
│   └── model/
│       ├── order_items.js
│       ├── products.js
│       └── users.js
└── agent/
    ├── agent.py
    ├── requirements.txt
    └── .env                 # エージェント環境設定
```

前提条件

* Docker（Cube Core の起動に使用）
* gcloud CLI（BigQuery への ADC 認証に使用）
* Python 3.x（エージェントの実行環境）
* Gemini APIキー（[Google AI Studio](https://aistudio.google.com) で取得）
* Google Cloud プロジェクト（`bigquery-public-data` への読み取り権限が必要）

エージェント実装コード（Python）

*※ 動作することを意図したサンプルコードです。十分な精査は行っていません。*

環境変数は各ディレクトリの `.env` に記載しておきます。

`agent/agent.py`

```
import os
import json
import argparse
import requests
import google.generativeai as genai
from dotenv import load_dotenv

load_dotenv()

genai.configure(api_key=os.environ["GEMINI_API_KEY"])

def _to_plain(value):
    """Gemini function_call の args を dict/list/primitive に変換する（Protobuf対策）。"""
    if value is None or isinstance(value, (str, int, float, bool)):
        return value
    if callable(getattr(value, "items", None)):
        return {str(k): _to_plain(v) for k, v in value.items()}
    if callable(getattr(value, "ListFields", None)):
        return {field.name: _to_plain(v) for field, v in value.ListFields()}
    try:
        return [_to_plain(v) for v in value]
    except TypeError:
        return str(value)

CUBE_API_URL = os.environ.get("CUBE_API_URL", "http://localhost:4000/cubejs-api/v1/load")
CUBE_API_TOKEN = os.environ["CUBE_API_TOKEN"]
GEMINI_MODEL = os.environ.get("GEMINI_MODEL", "models/gemini-2.5-flash")

# ---- ツール定義 ----
cube_tool = genai.protos.Tool(
    function_declarations=[
        genai.protos.FunctionDeclaration(
            name="query_cube",
            description=(
                "ECサイトの売上・注文データをセマンティックレイヤー（Cube）経由で集計する。\n"
                "利用可能なメトリクス:\n"
                "  - OrderItems.net_revenue         : ネット売上（返品・キャンセル除く）\n"
                "  - OrderItems.order_count         : 注文数（ユニーク注文ID）\n"
                "  - OrderItems.return_rate         : 返品率\n"
                "  - OrderItems.average_order_value : 平均注文単価\n"
                "利用可能なディメンション:\n"
                "  - Products.category    : 商品カテゴリ\n"
                "  - Products.brand       : ブランド\n"
                "  - Products.department  : 部門（Men/Women）\n"
                "  - Users.country        : 国\n"
                "  - Users.traffic_source : 流入元\n"
                "  - Users.gender         : 性別\n"
                "  - OrderItems.status    : 注文ステータス\n"
                "  - OrderItems.created_at: 注文日時（timeDimensions で使用）"
            ),
            parameters=genai.protos.Schema(
                type=genai.protos.Type.OBJECT,
                properties={
                    "measures": genai.protos.Schema(
                        type=genai.protos.Type.ARRAY,
                        items=genai.protos.Schema(type=genai.protos.Type.STRING),
                        description="集計するメトリクス（例: ['OrderItems.net_revenue']）",
                    ),
                    "dimensions": genai.protos.Schema(
                        type=genai.protos.Type.ARRAY,
                        items=genai.protos.Schema(type=genai.protos.Type.STRING),
                        description="グループ化するディメンション（例: ['Products.category']）",
                    ),
                    "timeDimensions": genai.protos.Schema(
                        type=genai.protos.Type.ARRAY,
                        items=genai.protos.Schema(
                            type=genai.protos.Type.OBJECT,
                            properties={
                                "dimension": genai.protos.Schema(
                                    type=genai.protos.Type.STRING,
                                    description="時間ディメンション（例: 'OrderItems.created_at'）",
                                ),
                                "granularity": genai.protos.Schema(
                                    type=genai.protos.Type.STRING,
                                    description="粒度（例: 'month'）",
                                ),
                                "dateRange": genai.protos.Schema(
                                    type=genai.protos.Type.STRING,
                                    description="期間（例: 'last month'）",
                                ),
                            },
                            required=["dimension"],
                        ),
                        description="時間軸の指定（例: [{dimension: 'OrderItems.created_at', dateRange: 'last month'}]）",
                    ),
                    "limit": genai.protos.Schema(
                        type=genai.protos.Type.INTEGER,
                        description="取得件数上限（デフォルト: 10）",
                    ),
                    "order": genai.protos.Schema(
                        type=genai.protos.Type.OBJECT,
                        description="ソート順（例: {'OrderItems.net_revenue': 'desc'}）",
                    ),
                },
                required=["measures"],
            ),
        )
    ]
)

model = genai.GenerativeModel(model_name=GEMINI_MODEL, tools=[cube_tool])

# ---- Cube API 呼び出し ----
def query_cube(measures, dimensions=None, filters=None, timeDimensions=None, limit=10, order=None, **_):
    query = {"measures": measures, "limit": int(limit)}
    if dimensions:
        query["dimensions"] = dimensions
    if filters:
        query["filters"] = filters
    if timeDimensions:
        query["timeDimensions"] = timeDimensions
    if order:
        query["order"] = order

    resp = requests.get(
        CUBE_API_URL,
        headers={"Authorization": CUBE_API_TOKEN},
        params={"query": json.dumps(query)},
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()["data"]

# ---- エージェントループ ----
def run_agent(user_question: str) -> str:
    chat = model.start_chat()
    response = chat.send_message(user_question)

    while True:
        fn_calls = [
            part.function_call for part in response.parts if part.function_call.name
        ]

        if not fn_calls:
            return "".join(part.text for part in response.parts if part.text)

        # ツールを実行し、結果を返す
        tool_responses = []
        for fn in fn_calls:
            args = _to_plain(fn.args) or {}
            result = query_cube(**args)
            tool_responses.append(
                genai.protos.Part(
                    function_response=genai.protos.FunctionResponse(
                        name=fn.name,
                        response={"result": result},
                    )
                )
            )

        response = chat.send_message(tool_responses)

# ---- エントリポイント ----
if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("question", help="質問文（例: \"先月の売上トップ5カテゴリは？\"）")
    args = parser.parse_args()
    print(run_agent(args.question))
```

補足：Cube Core が実際に発行したSQL・結果テーブル

上記の実行例でCubeが裏側で発行したSQLは以下です。返品率の定義（`COUNTIF(status = 'Returned') / NULLIF(COUNTIF(status IN ('Complete', 'Returned')), 0)`）がモデル定義通りに展開されていることが確認できます。  
※[?]はバインド変数部分

```
SELECT
  `users`.traffic_source `users__traffic_source`,
  COUNTIF(`order_items`.status = 'Returned')
    / NULLIF(COUNTIF(`order_items`.status IN ('Complete', 'Returned')), 0)
    `order_items__return_rate`
FROM bigquery-public-data.thelook_ecommerce.order_items AS `order_items`
LEFT JOIN bigquery-public-data.thelook_ecommerce.users AS `users`
  ON `order_items`.user_id = `users`.id
LEFT JOIN bigquery-public-data.thelook_ecommerce.products AS `products`
  ON `order_items`.product_id = `products`.id
WHERE (`order_items`.created_at >= TIMESTAMP(?) AND `order_items`.created_at <= TIMESTAMP(?))
  AND (`products`.category = ?)
GROUP BY 1
ORDER BY 2 DESC
LIMIT 10
```

## セマンティックレイヤーを挟む設計の恩恵

### ビジネス定義の変更がAIエージェント側に伝播しない

「返品率の分母を『完了+返品』から『全注文』に変えたい」となったとき、変更するのはCubeのモデル定義のみです。AIエージェントのコード、プロンプト設定等は変える必要がありません。

### 指標の追加が容易

新しいKPI（例：リピート購入率、LTV）をCubeに追加するだけで、AIエージェントから参照できるようになります。

### 定義の一元管理ができる

既存のTableauダッシュボードで使っている指標をCubeに移植することで、 BIツールとAIエージェントで同じ定義を参照する一元管理が実現できます。「BI担当者が定義していた『ネット売上』の意味」がセマンティックレイヤーという形で明文化され、AI活用の文脈でも生きてきます。

## セマンティックレイヤー導入で難しいのは実装よりも「定義の整理」

ここまで読んで、実装自体はそれほど難しくないと感じた方も多いと思います。実際、Cube Coreのセットアップは数時間もかからず行うことが出来ました。

実際、実務で本当に難しいのは、 **何を定義するかの整理** に当たると考えています。

* 「売上」の定義は誰が持っているか
* グロスとネットの両方が社内に混在していないか
* 「返品率」の分母は注文数か、金額か
* 指標間の依存関係はどうなっているか

この整理をせずにセマンティックレイヤーを作ると、「書かれた定義と実態が乖離している」という別の問題が生まれます。

> このあたりの整理について、弊社（ギックス）では、元より提唱する分析手法「ゾクセイ」を活用した **「セマンティックレイヤー構築支援」** の提供を開始しています。詳細は以下よりご参照ください。

セマンティックレイヤーを導入したうえでも、分析エージェントが意味のある結果を安定して返せるかは、いま述べてきた「定義の整理」（データの意味づけが揃っているか）に強く依存します。**その定義づくりは、ドメインに責任を持つ人間の仕事** であり、いまのところ AIに任せきれる領域ではないと考えています。

## まとめ

* **LLMに会話・解釈だけを担わせる** → SQLの非決定性がデータ集計に影響しない
* **API型セマンティックレイヤー（Cube Core）を立てる** → SQL実行が決定的・一貫した数字を保証
* **function callingでCube APIを呼ぶ** → エージェントはSQL生成不要・結果はCube任せ
* **定義の整理が重要** → 定義の曖昧さ・重複・乖離を防ぎ、意思決定に資する指標を持たせる

自然言語で分析させるうえでの再現性と信頼性は、主として **指標の意味づけをどこに置き、SQLの組み立てを誰に任せるか** にかかっていると思います。これまで見てきた **会話（LLM）と定義・実行（セマンティックレイヤー）を分ける** という組み方は、実務でも現実的な一手だと考えています。
