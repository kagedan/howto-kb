少し前に、Palantir Foundry を「リバースドメインモデリング」という見立てで分解した投稿（[@gura105 さんの X 投稿](https://x.com/gura105/status/2077153028982133080)）を読みました。ドメインモデリングという馴染みの工程を、既存システムの物理データから逆回しにやる——オントロジーの実体は全社横断のドメイン層だ、という整理です。その投稿は最後を「今後は AI エージェントとの相性を掘っていく」と締めていました。

その「掘る」はご本人の手でも進んでいて、つい先日にはパターンの最小参照実装（[gura105/operational-ontology](https://github.com/gura105/operational-ontology)）が公開されました。読むためのセマンティックレイヤーに対し、実行するためのオペレーショナル・オントロジー——概念を最小の形で示すのがあちらだとすれば、本記事は同じパターンを**製品側から**引き取った報告です。相性を確かめるなら、作って業務で使うのが一番早い。というわけで Foundry の縮小コピー——**ミハシラ**と名付けました——を自作しました。業種ごとの業務モデルは業種テンプレート（YAML）として差し替える設計で、最初のテンプレートに中小の受託事業者（受託開発・SI）向けを選んでいます。Postgres と TypeScript と Claude Agent SDK で、オントロジー定義からエージェント・業務 UI までが動くところまで来ています。

なおちょうど、社内の用語・システム・関係をカード型の知識オントロジーにして AI に引かせる実践記（[Ubie のエンジニアによる note 記事](https://note.com/ymdpharm3/n/n8515d151e56d)）も公開されています。あちらが「会社を読む」ための参照系オントロジーだとすれば、本記事が扱うのは「業務を安全に動かす」ための操作系です。

この記事では一点だけに絞ります。**オントロジー定義からエージェントのツールへの変換**です。先に結論を書くと、この変換は完全に機械的で、設計の余地がほとんどありませんでした。それが何を意味するかを、実装で示します。

## 4 要素を 1 枚の YAML で宣言する

本家のオントロジーは、業種テンプレートという形に縮小しました。オブジェクト（顧客・商談・案件・契約・工数・請求…）、リンク、アクション、メトリクスの 4 要素を 1 枚の YAML で宣言します。受託業テンプレートの現物から、アクションを 1 つ引用します。

```
- name: markWon
  displayName: 受注確定にする
  target: Opportunity
  parameters:   # 抜粋（ほか startDate / contractType）
    - { name: contractAmount, type: money, required: true, description: "請負は総額、準委任・保守は月額" }
  preconditions:
    - { expr: 'target.stage == "negotiating"', message: 交渉中の商談のみ受注確定にできます }
  effects:
    - update: { stage: won }
    - create: { type: Project, ... }    # 案件を自動生成（実物はプロパティ列挙あり）
    - create: { type: Contract, ... }   # 契約ドラフトも自動生成
  permissions: { roles: [operator, admin], agentPolicy: confirm }
  whenToUse: >
    受注の確定連絡を受けたとき。案件と契約ドラフトが自動生成される。
    実行提案の際は生成される内容に加えてアサイン候補（空き工数のある要員）を添えること
```

商談を受注確定にすると、案件と契約ドラフトが生成される。事前条件は「交渉中の商談のみ」。人間の承認が必須（`agentPolicy: confirm`）。業務の意味はすべてここに書いてあり、この 1 枚から物理テーブル・API・業務 UI、そして本題のエージェントのツール群までが生成されます。

## オブジェクトは query\_\* / get\_\* に、アクションは action\_\* になる

Claude Agent SDK には、任意のツールをプロセス内 MCP サーバーとして生やす仕組み（`tool()` + `createSdkMcpServer()`）があります。テンプレートを読んで、オブジェクトごとに検索・取得ツールを、アクションごとに実行ツールを動的生成します。生成器の本体は、ほぼこれだけです。

```
export function buildOntologyServer(rt: ToolRuntime) {
  const tools = [
    ...rt.template.objects.flatMap((o) => queryToolsFor(rt, o)),  // query_* / get_*
    ...rt.template.actions.map((a) => actionToolFor(rt, a)),      // action_*
    approvalsToolFor(rt),   // 承認状態の照会（固定）
    bulkImportToolFor(rt),  // 一括登録（固定）
    // ほかにメトリクス・ワークリスト・チャット UI 操作の固定ツールが数本
  ];
  return createSdkMcpServer({ name: "ontology", version: "0.1.0", tools });
}
```

受託業テンプレートは 12 オブジェクト・34 アクションなので、`query_Customer` `get_Project` `action_markWon` といったツールが 58 本、この写像から出てきます。手書きのツール定義はゼロです。

パラメータのスキーマ（zod）も YAML から導出します。担当するのは `zodForProperty` という 30 行ほどの関数だけで、`integer` は `z.number().int()`、`enum` は `z.enum(values)` のように型を落とし、表示名・単位・「`ref` は参照先の UUID」「`date` は YYYY-MM-DD」といった注記を `describe()` に詰めます。退屈なくらい素直な変換です。検索ツール（`query_*`）の引数も同じ関数で、全プロパティ＋リンク列を optional な等価フィルタとして並べて作ります。`enum` のステージ名も日付の形式も、スキーマがエージェントに教えてくれるので、プロンプトで説明する必要がありません。

## この変換には、設計の余地がほとんど無い

やってみて分かったのは、ここに工夫の入り込む隙がほぼ無いということです。オブジェクトがあれば検索と取得。アクションがあれば実行。パラメータは型ごとに zod へ。迷う場所がない。これは手抜きの自慢ではなく、構造的な帰結だと思っています。ツール定義が LLM に伝えるべき情報 — 何が操作できて、どんな引数で、いつ呼ぶべきか — は、業務モデルがまともに書けていれば**すでにオントロジーの中にある**。変換器は写像であって、設計ではない。

逆に言うと、エージェント統合の品質は、ツール実装の巧拙ではなく**オントロジーの解像度で決まります**。`contractAmount` の description に「請負は総額、準委任・保守は月額」と書いてあるかどうか。事前条件の message が人間に通じる日本語かどうか。エージェントの答えの質を上げたければ、TypeScript ではなく YAML を直すことになります。

## precondition の message が「いつ呼べるか」を教える

写像の中で一番効いていると感じるのが、ツールの description の組み立てです。アクション定義の各フィールドを、そのまま焼き込みます。

```
const descParts = [
  `${action.displayName}。${action.description ?? ""}`,
  action.preconditions.length > 0
    ? `実行条件: ${action.preconditions.map((pre) => pre.message).join(" / ")}`
    : null,
  action.whenToUse ? `いつ使うか: ${action.whenToUse}` : null,
  // agentPolicy: confirm のアクションには【要承認】の注記を追加（省略）
].filter(Boolean);
```

冒頭の `markWon` からは、こういう description が生成されます（整形して表示）。

> 受注確定にする。  
> 実行条件: 交渉中の商談のみ受注確定にできます  
> いつ使うか: 受注の確定連絡を受けたとき。案件と契約ドラフトが自動生成される。実行提案の際は生成される内容に加えてアサイン候補（空き工数のある要員）を添えること  
> 【要承認】このアクションは人間の承認が必要。実行を試みると承認依頼の案内が返る（dryRun: true の試算は承認不要）

precondition の message は、もともと UI で「押せないボタンの理由」として人間に見せるために書いた文です。それがそのまま「いつ呼べるか」の教材になる。実行時にも同じ事前条件がエンジン側で再検証され、違反すれば同じ message で拒否が返るので、エージェントは理由を読んで次の一手を変えます。宣言は 1 箇所、効く場所は 3 箇所（description・UI・実行時検証）。

## まとめ — 次回は安全装置の話

* オブジェクト / リンク / アクション / メトリクスの 4 要素を YAML で宣言すると、`query_*` / `get_*` / `action_*` のツール群が写像で生成される。zod スキーマも定義から導出
* この変換に設計の余地はほとんど無い。エージェント統合の質は、ツール実装ではなくオントロジーの解像度で決まる
* precondition の message や whenToUse は description に焼き込まれ、「いつ呼べるか」をエージェントに教える。同じ宣言が UI の理由表示と実行時検証にも効く

ところで、description の末尾にあった「【要承認】」の一文。`agentPolicy: confirm` の 1 行が、エージェントの自律実行を止めて人間の承認フローに接続します。「危険な操作をしないでください」というプロンプトの祈りとは決定的に違うこの部分 — 次回は**安全装置をプロンプトではなく構造で作る**話を書く予定です。
