LangChainやLlamaIndexを使えばエージェントはすぐ動く。ただ、フレームワークに任せてしまうと「なぜ動いているのか」がわからないまま使い続けることになる。

このシリーズでは、Anthropic Python SDKだけを使ってエージェントをゼロから組んでいく。フレームワークなし、マジックなし。第1弾はReActパターンの最小実装だ。

---

## ReActとは何か

ReActは2022年にGoogleが発表した論文（Wei et al.）で提案されたパターンで、ReasoningとActingを組み合わせた造語だ。

```
Thought（思考） → Action（行動） → Observation（観察）
       ↑_______________________________________________↓
```

LLMが「何を考えて」「どう動いて」「何を見たか」を自分で繰り返す。ChatGPTのような一問一答とは違い、前の結果を踏まえて次の行動を変えられる。これがエージェントと単なるチャットの本質的な違いだ。

---

## 何を作るか

意図的にバグを仕込んだPythonファイル5本を、エージェントに順番にレビューさせる。

| サンプル | 仕込んだバグ |
| --- | --- |
| sample\_01.py | SQLインジェクション（3パターン） |
| sample\_02.py | 未使用変数4個 + 循環的複雑度超過 |
| sample\_03.py | 型エラー・例外処理なし |
| sample\_04.py | 平文パスワード・ハードコードAPIキー |
| sample\_05.py | N+1クエリ・Python側SUM計算 |

バグの正解が明確なので、どれだけ検出できたかを後で数えられる。

## ツールを3つだけ持たせる

第1弾のエージェントには `read_code`、`think`、`comment` の3ツールだけを渡す。

```
TOOLS = [
    {
        "name": "read_code",
        "description": "指定したPythonファイルのソースコードを読み込む。最初に必ず呼ぶ。",
        ...
    },
    {
        "name": "think",
        "description": "コードについて分析・思考する。実際の処理は行わず、思考内容を返すだけ。",
        ...
    },
    {
        "name": "comment",
        "description": "コードレビューコメントを生成・出力する。最後にこのツールを呼ぶ。",
        ...
    }
]
```

`think` ツールは少し変わっている。呼ばれても何も処理しない。ただ、これを用意しておくとClaudeが「まず考えを整理してからコメントを書く」という順番を守るようになる。ログに思考過程が残るのでデバッグもしやすい。Anthropicの [extended thinking](https://docs.anthropic.com/ja/docs/build-with-claude/extended-thinking) とは別の話だが、考え方は近い。

## ReActループの実装

```
MODEL = "claude-haiku-4-5-20251001"
MAX_ITERATIONS = 10

def run_react_agent(target_file: str) -> dict:
    messages = [
        {
            "role": "user",
            "content": f"次のファイルをコードレビューしてください: {target_file}"
        }
    ]

    while iteration < MAX_ITERATIONS:
        response = client.messages.create(
            model=MODEL,
            max_tokens=4096,
            system=system_prompt,
            tools=TOOLS,
            messages=messages
        )

        for block in response.content:
            if block.type == "text":
                print(f"Thought: {block.text}")
            elif block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                print(f"Observation: {result}")

        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})

        if response.stop_reason == "end_turn" and not has_tool_use:
            break
```

Claudeは状態を持たない。毎回のAPI呼び出しに会話全体を `messages` として渡すことで、前のObservationを踏まえた次のThoughtが生まれる。この配列の積み重ねがReActループの記憶になっている。

`execute_tool()` の中身はシンプルだ。

```
def execute_tool(tool_name: str, tool_input: dict) -> str:
    if tool_name == "read_code":
        content = Path(tool_input["filepath"]).read_text(encoding="utf-8")
        lines = content.split("\n")
        numbered = "\n".join(f"{i+1:3}: {line}" for i, line in enumerate(lines))
        return f"=== {filepath} ({len(lines)}行) ===\n{numbered}"

    elif tool_name == "think":
        return f"[思考記録]\n{tool_input['reasoning']}"

    elif tool_name == "comment":
        issues = tool_input.get("issues", [])
        for issue in issues:
            print(f"【問題】{issue['category']}: {issue['description']}")
        return "レビュー出力完了"
```

## 実行結果

Windowsで動かした出力がこれだ。

```
--- イテレーション 1 ---
Thought: ファイルを読み込んで分析いたします。
Action: read_code({"filepath": "review_targets/sample_01.py"})
Observation: === review_targets/sample_01.py (48行) ===
  1: # sample_01.py - SQLインジェクション脆弱性のサンプル
  ...

--- イテレーション 2 ---
Thought: コードを確認しました。では詳細に分析します。
Action: think({"reasoning": "SQLインジェクション脆弱性（全3関数）..."})
Observation: [思考記録] ...

--- イテレーション 3 ---
Thought: それでは、レビューコメントを生成します。
Action: comment({...})
```

read → think → comment の3ステップで毎回まとまった。5ファイルすべて4イテレーションで完了している。

| ファイル | 仕込んだバグ | 検出数 |
| --- | --- | --- |
| sample\_01.py | SQLi × 3 | 3件 + α |
| sample\_02.py | 未使用変数 × 5、複雑度 | 8件 |
| sample\_03.py | 型/例外エラー × 6 | 10件 |
| sample\_04.py | セキュリティ × 5 | 13件 |
| sample\_05.py | N+1 × 2、非効率 × 2 | 7件 |

全ファイルで仕込んだバグを検出できた。sample\_04.py は13件で、「ログにAPIキーを出力している」など用意していなかった問題まで指摘している。実行時間は108.3秒、合計20イテレーションだった。

横軸に4つの参加者が並んでいる。左からUser、Agent（react\_agent.py）、Anthropic API（claude-haiku）、そしてTools（read\_code / think / comment）だ。Toolsはまとめて1つの枠に入っているが、実体はPython関数の集まりで、APIとは無関係にAgentが直接呼び出す。縦軸は時間で、上から下へ4回のイテレーションが流れる。

### 起動

まずUserがAgentに依頼を出す。これがループの起点になり、Agentは最初のAPI呼び出しを準備する。

### イテレーション1：ファイルを読む

AgentはClaudeに対して `messages`（会話履歴）とTOOLS定義をセットで送る。Claudeはコードをまだ見ていないので、まず `read_code` を呼ぶべきだと判断し、`block.type == "tool_use"` として返してくる。ここで注意が必要で、ClaudeはファイルをIOしているわけではない。「`read_code` を呼んでください」という宣言を返しているだけだ。

実際にファイルを読むのはAgent側の `execute_tool()` だ。行番号付きのソースコードを文字列に変換して返す。その結果を `tool_result` としてmessagesに追記し、Agentは再びClaudeにAPI呼び出しをする。

### イテレーション2：問題点を整理する

ファイルの中身を受け取ったClaudeは今度は `think` を選ぶ。「SQLインジェクションが3か所ある」「ゼロ除算のリスクがある」といった分析を `reasoning` フィールドに書いてくる。

`think` ツールの実装は1行だ。受け取った文字列をそのまま返すだけで、何も処理しない。それでもこのステップを挟む意味がある。Claudeが思考を一度アウトプットすることで、次のcommentステップで「整理済みの分析」を前提として話を進められる。いきなりcommentを書かせるより指摘の質が上がる。

### イテレーション3：レビューコメントを出力する

思考を終えたClaudeは最後に `comment` を選ぶ。問題の一覧、重大度、総評をJSON形式で渡してくる。`execute_tool()` がそれを整形してコンソールに出力し、結果をtool\_resultとしてAgentに返す。

### イテレーション4：ループを終了する

`comment` の結果を受け取ったClaudeは、レビュー全体のまとめを自然言語で書いてからend\_turnする。このとき `block.type == "tool_use"` は含まれない。Agentはそれを検出してループを抜ける。

```
if response.stop_reason == "end_turn" and not has_tool_use:
    break
```

実行ログで4回目にThoughtだけが出てツールが動かないのはこのためだ。

### なぜ会話履歴を毎回全部渡すのか

図を見ると、イテレーション1〜4のすべてでAgentからAPIへの矢印が出ている。Claudeはステートレスなので、過去の会話を自分では覚えていない。イテレーション3のClaudeがイテレーション1で読んだファイルの内容を「知っている」のは、Agentが毎回 `messages` 配列に全履歴を積み重ねて送っているからだ。この配列こそがReActループの記憶媒体になっている。

## 今回の限界と次回

今回のエージェントはコードを読んでLLMの知識でコメントを書くだけだ。SQLインジェクションを見つけるのも、複雑度を判断するのも、全部Claude任せになっている。

次回（#2：Tool Use設計）では静的解析ツールを追加する。ASTで未使用変数を検出する `lint_check()`、McCabe複雑度を計算する `complexity_check()`、SQLiやN+1クエリをパターンマッチする `search_best_practice()` の3つだ。コードの内容に応じてどれを使うかをエージェント自身が判断するようになる。
