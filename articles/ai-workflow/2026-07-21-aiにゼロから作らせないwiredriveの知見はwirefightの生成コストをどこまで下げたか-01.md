WIREFIGHTをFable 5で制作した際の追加消費は、約17ドルでした。

正確な比較はできませんが、WIREDRIVEからの蓄積がなければ、同じところまで進むのに2〜3倍ほどかかったと思います。（既存の描画・音声基盤まで新規に生成し、過去に解決済みの環境依存エラーを再びデバッグすると仮定した概算です）

この記事では、どの知見がそのまま使えたのか、どこからが新規だったのかを振り返ります。

## 1. そのまま使えたもの

WIREDRIVE以降、WIREシリーズでは同じ描画基盤を使っています。

### カメラ投影

3D空間の線分をカメラ空間へ変換し、near面でクリップして、画面座標へ透視投影します。

最後に座標を整数へ丸めると、頂点が少し震えるPSX風の見た目になります。

```
def draw_seg(a, b, color):
    # world -> camera
    # near clipping
    # perspective projection
    # integer snapping
    ...
```

この`draw_seg`は、シリーズを通してほぼ同じ形で使えました。

### フォントとパレット

16色のパレットと、5×7ドットの自作フォントも共通です。

タイトルやUIを同じ関数で描いているため、ゲームの内容が変わってもWIREシリーズらしい見た目が残ります。

### 音声の変換

BGMとSFXはElevenLabsで生成し、ffmpegで22.05kHzのOgg Vorbisへ変換して使っています。

ここでは以前、環境に`libvorbis`がなく、変換に失敗しました。

```
ffmpeg -i input.wav -ar 22050 -c:a vorbis -strict -2 output.ogg
```

`-c:a vorbis -strict -2`を指定すれば、ffmpeg内蔵のVorbisエンコーダーで変換できます。

一度分かってしまえば短い指定ですが、知らなければ今回も同じところで止まっていました。

ここまでの処理は完全に無料になったわけではありませんが、新しく設計する必要はありませんでした。

「作り直す」のではなく、「当てはめて確認する」だけで済んだ部分です。

## 2. 再利用して発覚したバグ

> 動いているコードと、正しいコードは同じではない。

再利用するまで見つからなかったバグでした。

## 3. WIREFIGHTで新しく作ったもの

WIREFIGHTでは、Mixamoのモーションキャプチャを使いました。

ここからは前作までにない処理です。

### Colladaから関節座標を取り出す

Mixamoから、骨格とアニメーション情報を含むCollada（`.dae`）形式でモーションを取得しました。

ColladaはXMLなので、Python標準ライブラリの`xml.etree`で読めます。

骨の親子関係をたどり、各フレームでローカル行列を親から子へ掛け合わせるFK（順運動学）を実装しました。

```
world_matrix = parent_world @ local_matrix
```

全フレームの関節座標は事前に計算し、`.npz`へ保存しています。

ゲームの実行中にColladaを解析する必要はありません。

### ファイターの向きを合わせる

同じモーションを左右のファイターへ使うには、キャラクターの正面方向をそろえる必要があります。

構えモーションの左右の肩を結ぶベクトルから、モーション側の正面角度を求め、`BASE_YAW`として保存しました。

再生時は、各ファイターが相手を向くように関節座標全体をyaw回転します。

### その場で倒れるようにする

KOモーションには、倒れながら1m以上前進する動きが含まれていました。

そのまま再生すると、倒れている間にリングを横断し、相手の位置まで移動してしまいます。

そこで腰のx、z座標を、モーション開始時の値に固定しました。

高さ方向の変化だけを残すことで、前進しながら倒れるモーションを、その場で倒れる動きへ変換できます。

## 4. モーションから当たり判定を決める

ジャブやキックのヒットタイミングは、見た目に合わせて決める必要があります。

今回は感覚で設定せず、ベイクした関節座標から最大伸長フレームを調べました。

* ジャブ: 15フレーム目、リーチ0.95
* キック: 32フレーム目、リーチ1.30

手や足が前方へ最も伸びるフレームを`argmax`で求め、その値を当たり判定へ使っています。

これで、拳が届く瞬間と攻撃が当たる瞬間が一致します。

CPUは、相手との距離に応じて次の行動を確率で選びます。

プレイヤーの攻撃がヒットフレームへ到達する前に、一定確率でガードする処理も入れました。

強いAIではありませんが、対戦相手としては動きます。

ちなみに私は格闘ゲームが苦手なので、実際にプレイするとCPUが少しガードしすぎる気もします。

## 5. コストを下げたのはコードだけではなかった

今回そのまま使えたのは、次のような部分です。

一方、骨格の解析、向きの補正、ルート移動の固定、ヒットフレームの計測は新規でした。

ここで気づいたのは、再利用できるのはコードだけではないということです。

生成されたコードの行数や、消費したトークン数よりも、

> もう一度失敗しなくて済んだ回数

で考えるほうが、今回の制作コストには近い気がします。

## まとめ

* 描画と音声の基盤は、前作からほぼそのまま使えた
* 再利用したことで、WIREDRIVEでは隠れていた符号のバグも見つかった
* 骨格アニメーションとモーション補正は、WIREFIGHTで新しく作った
* 今回の新規部分は、次の作品では再利用できる基盤になる

AIに毎回ゼロから作らせるより、以前のコードと失敗した記録を渡したほうが、少ない往復で先へ進めます。

今回、Fable 5での追加消費が約17ドルで済んだのは、AIが急に賢くなったからではなく、前の作品で一度遠回りしていたからだと思います。
