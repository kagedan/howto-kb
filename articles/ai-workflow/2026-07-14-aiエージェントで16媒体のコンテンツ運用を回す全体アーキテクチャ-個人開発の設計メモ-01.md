個人プロジェクトとして、動画・記事・音声コンテンツを **16以上のプラットフォーム** に配信する運用基盤を、ほぼ **AIエージェント任せ** で回しています。この記事は「実際にどう組んでいるか」を、AIエージェントを中心に据えた全体アーキテクチャとして整理したものです。個別のスクリプトの話ではなく、**どのレイヤーをどう分けて、エージェントにどこまで任せ、人間がどこで介入するか**という設計判断が主題です。

前提として、コンテンツのジャンルや商材、運用主体の詳細はここでは伏せます。技術的な構造だけを一般化して書きます。

## 出発点: 「投稿スクリプトを1本書けば終わり」は破綻する

最初は素朴に「1本のコンテンツを作って全媒体に流すスクリプトを書けばいい」と考えていました。すぐ破綻しました。媒体ごとに前提がまるで違うからです。

* 公式投稿APIの有無・機能差がバラバラ
* 自動投稿そのものがBANリスクになる媒体がある
* 新規アカウントを初日からフル運用すると凍結される
* 同じ素材でも媒体ごとに最適なフォーマットが違う

これを無視して機械的に量産した結果、数週間で複数アカウントが止まった経験があります。そこから「媒体差を吸収するのは配信層に閉じ込め、**判断そのものはAIエージェントに寄せる**」という構造に落ち着きました。

## 全体アーキテクチャ

現在の構成を5つのレイヤーに分けています。

肝は、**オーケストレーション層のエージェントが中心にいて、知識層を読み書きしながら生成層を駆動し、配信結果がフィードバックとしてまた判断に戻ってくる**というループ構造です。生成層・配信層は「エージェントが呼ぶ道具」であって、頭脳ではありません。

## AIエージェントを「実行者」ではなく「判断者」として使う

このアーキテクチャで一番効いている設計判断がこれです。**エージェントを、決まった処理を流すだけの実行者ではなく、「今やるべきか・何を作るか」を判断する判断者として起動する**、というものです。

具体的には、スケジューラがエージェントを定期起動しますが、渡すのは「これを実行しろ」という命令ではなく「状況を見て、価値ある仕事があるかを判断し、あれば1件だけやれ。無ければ何もするな」というプロンプトです。

たとえば技術記事の枠では「直近の作業ログを見て、他の人の学びになる汎用的な知見に一般化できる出来事があったときだけ1本書く。薄い内容や焼き直しになるならスキップする」という判断をエージェント自身にさせています。**「無理に成果物を作らない」判断を人間の代わりにやらせる**のが、量産型の自動化との一番の違いです。機械的に毎日1本生成させると質が落ちて逆効果になるからです。

この「定期起動して判断させる」パターンは、コンテンツのネタ補充・投稿要否・リトライ要否など、あちこちで使い回しています。

## 知識・記憶層 (SSOT) が一貫性を担保する

エージェントは起動のたびにコンテキストがリセットされます。そのままだと毎回ゼロから判断してブレるので、**永続的な知識層（運用Wiki + エージェント用メモリ）を単一の正典 (SSOT) として置き、起動時に必ず読ませています**。

ここに入れているもの:

* 媒体ごとの制約・禁止事項（規約違反で凍結した実例つき）
* 過去にハマったバグと対処（同じ轍を踏まないため）
* 表記・読みの正規化ルール（固有名詞の誤読を防ぐ辞書など）
* 何をどの媒体にいつ出したかの履歴（重複回避）

ポイントは、**エージェントが仕事の結果や新しい学びをこの層に書き戻す**ことです。読むだけでなく書き込むことで、失敗が次回以降のルールとして蓄積されます。人間が全部のルールを先回りして書くのは不可能なので、「失敗するたびにエージェント自身が明文化する」運用にしています。

## 配信層: 媒体を性質で分類してルーティングする

媒体差はこの層に閉じ込めます。新しい媒体を足すとき、まず次の分岐で対応方針を決めます。

* **公式APIあり・BANリスク低** → 全自動。ただし「投稿はできても副次機能だけAPIが無い」欠落に何度も遭遇したので、機能単位で確認します。
* **APIが無い / 欠落** → 全自動をあきらめ、**下書きを生成してメッセージアプリに通知し、人間が数十秒でコピペ投稿**する半自動に倒します。「機械が作る・人が押す」の分業です。
* **自動投稿がグレー / BANリスク高** → 手動。反復投稿を明示的に禁止している媒体があり、そこは人間が都度言い換えて投稿します。

「全部を全自動にする」ことにこだわらず、**リスクの高い最後のワンタップだけ人間に残す**のが安定運用のコツでした。

## 生成層: パイプラインを分解して各段を差し替え可能にする

生成層は媒体非依存の素材工場です。動画系を例にすると、こう分解しています。

各段を疎結合にしておくと、TTSエンジンや画像生成モデルを差し替えても他段に波及しません。地味ですが効いているのが **「読み・表記の正規化」段** で、TTSは固有名詞や年号を平気で誤読するため、台本と発話用テキストを分けて読み仮名を辞書で当てています（誤読は知識層のルールとして貯める）。

そして最後の **品質ゲート** で、尺・禁止表現・必須要素の有無などを自動チェックしてから配信層に渡します。

## 一番効いたのは「静かな失敗」への対策

運用していて最も怖かったのは、エラーが出ないのに実は動いていないパターンでした。

* ネタが尽きたバッチが **exit code 0 のまま何も生成せず終わり続ける**
* ある媒体向けロジックが、メタデータのキー名想定違いで **特定フォーマットだけ静かにスキップし続ける**
* 通知のバグで **失敗しても誰にも通知が飛ばない**

どれも「ログを能動的に見に行かないと気づけない」種類で、数日〜数週間放置していたことがあります。対策として、**「正常に見える状態」と「本当に正常な状態」を区別するチェック層**を別立てにしました。

```
# NG: 戻り値を見ずに成功扱いしてしまう
foreach ($poster in $posters) {
    & $poster
    Write-Output "distributed: $poster"   # 失敗していても表示される
}

# OK: 戻り値を見て、失敗を握りつぶさない
foreach ($poster in $posters) {
    & $poster
    if ($LASTEXITCODE -eq 0) {
        Write-Output "OK:   $poster"
    } else {
        Write-Output "FAIL: $poster"       # ここで初めて気づける
    }
}
```

ルールは3つに集約しました。

1. 各処理は必ず戻り値（成功/失敗）を確認し、無条件の成功表示をしない
2. 「在庫」を扱うバッチは、在庫切れ自体を異常として通知する（正常終了として握りつぶさない）
3. すべての失敗・警告は、人間が毎日見る通知チャンネルに必ず流す

AIエージェントに判断を委ねるほど、この「本当に動いているかを別レイヤーで監視する」ことの重要性が上がります。判断がブラックボックス化するぶん、出力の有無と結果は機械的に検証する必要があるからです。

## まとめ

複数媒体へのコンテンツ運用を自動化するとき、本質は「投稿ロジックを書くこと」ではありませんでした。効いた設計判断は次の4つです。

* **AIエージェントを実行者ではなく判断者として起動する**（無理に作らない判断まで委ねる）
* **永続的な知識層 (SSOT) を置き、エージェントが読み書きして一貫性と学習を担保する**
* **媒体差は配信層に閉じ込め、API有無とBANリスクでルーティングする**（最後のワンタップだけ人間に残す）
* **「エラーは出ないが実は動いていない」静かな失敗を、別レイヤーで機械的に監視する**

エージェントに任せる範囲を広げるほど、周辺の「知識層」と「監視層」を厚くしておくことが、破綻しない自動化の条件だと感じています。
