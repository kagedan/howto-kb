トリビューでCTOをしている小尾（[@obiyuta](https://x.com/ob1yuta)）です。

弊社では2026年7月、美容医療アプリ「トリビュー」にAI相談機能をリリースしました。悩みごとのヒアリングから始めて、ユーザーの希望条件に沿って施術やクリニック、メニューの候補を検索・整理して提示し、そのまま予約まで行うことができるAIチャット機能です。

この記事は、医療ドメインにおいてユーザー向けのAI機能を提供するにあたり、安全性やコストを犠牲にせずベストな体験を提供するための設計やそこに至る失敗をお伝えできればと思います。

## AI相談とは

![](https://static.zenn.studio/user-upload/5a855bc894a5-20260723.png)

機能としては、悩みのヒアリングと整理から始まり、施術の理解・比較から、希望条件に沿ってクリニック・メニューの候補を検索・整理して提示し、関連する口コミの検索、予約にまでサポートします。回答には、テキストだけでなく施術情報の比較カード、メニュー、口コミ、予約ボタンなどアプリで提供している情報や機能が並びます。

この会話は行き当たりばったりに進むわけではなく、「1. 悩みを言語化する」「2. 施術を決める」「3. クリニック・ドクター・メニューを選ぶ」の3ステップとして設計しています。ユーザーが思考を整理しながら、段階的に予約まで進められるようなサポートになっています。

機能の位置づけも書いておきます。AI相談は、トリビューに掲載されている公開情報をユーザー自身の希望条件で検索・整理して表示する「**情報検索の補助ツール**」です。診断や治療の要否・適否の判断は行わず、施術を受けるかどうかは医療機関で医師に確認いただく前提で、その旨と免責をチャット画面上に常時表示しています。機能の範囲も、法令・ガイドラインを守って提供できると確認できたものに絞ってリリースしました。

## プロジェクトの与件と役割分担

「対話でユーザーの意思決定に伴走すれば、検索やレコメンドでは届けられていない体験を作れる」という仮説を検証する上で、実装には最初から与件を5つ設定していました。この与件を、バックエンドとフロントエンドで次のように分担し、実現しています。

| 与件 | 内容 | バックエンド | フロントエンド |
| --- | --- | --- | --- |
| 独自データでの差別化 | ChatGPTでできない納得感のある提案 | 自社データの検索結果に基づく提案候補の構築 | 実データの取得及びUIの構築 |
| アプリ水準のUI品質 | チャットだからテキスト中心ではない高品質のUI | 「何を見せるか」の指示データ（テンプレートJSON）を生成 | 指示データをReactで描画し、UIを構築 |
| コスト制約 | スケールしても成立するコスト構造 | 出力を指示データに絞り、プロンプトキャッシュ | 実体データをLLMの往復に載せず、自前で取得 |
| 法令・ガイドライン遵守 | 薬機法・医療広告ガイドライン・体験談の扱いに配慮した表示 | プロンプトでの出力設計 | 表示直前の統制（置換・差し替え）と観測 |
| 提案ロジックの透明性 | 提案ロジックのアカウンタビリティ | 候補を実検索データから組み立て | 実データに基づく表示 |

この記事では主に表の右列、フロントエンド領域を扱っていきます。（バックエンドは別途ご紹介予定です）

## 体験・安全・コストも🙅だったプロトタイプ

開発当初は、まず動くものを作ることを優先し、回答のカードUIもLLMにHTMLとして直接出力させていました。プロンプトでマークアップの書き方を指示し、回答ごとにカードのHTMLを生成させるような実装です。  
出力品質の不安定さに加えて、法令・ガイドラインなどの安全面でも基準を満たせておらず、特にコストに関しては、スケール時の試算に対する関係者一同の「え…？🥹」という表情が忘れられません。

| 問題 | 内容 |
| --- | --- |
| 不安定な出力 | タグの閉じ忘れなど壊れたHTMLで画面が崩れる。XSSなどの懸念も。 |
| トークン費用 | UIマークアップが冗長で、テキストの数倍の文字数に。チャットは過去の回答も入力に積まれるため、会話が続くほどコストが膨らむ。 |
| 統制不能 | 何をどう表示するかがすべてプロンプト頼みになり、表示のルールを保証できない。医療ドメインでは絶対許容不可。 |

**なにかをトレードオフにするのではなく、体験も、安全も、コストも高いレベルで実現するために、アプローチの大幅な見直しを行いました。**

## フロントエンドとバックエンドの責務を分離

やったことは非常にシンプルで、責務を分離して設計しなおしていきました。**LLMはビジネスロジック側の1コンポーネントとして扱い「何を見せるか」の決定を、「どう見せるか」はフロントエンドに寄せ切りました。**

最終的なユーザーの相談開始から実際に回答が表示されるまでの流れは以下の通りです。

![](https://static.zenn.studio/user-upload/95476b6186df-20260723.png)

**回答の見た目はリッチですが、LLMが画面を作っているわけではありません。** LLMが出すのは「何を見せるか」の指示データまでで、画面の組み立てとデータの取得はフロントエンドで行っています。

### テンプレートJSONに基づきフロントでUIを構築

実際にはフロントエンドがなにを表示すべきかを知る必要がありますので、LLMからは回答テキストに加えて、次のようなJSONを返しています。

```
{
  "templates":[
    {
      "type":"menu",
      "items":[
        {"id":12345, ...}
      ],
      "surgeryId":1
    },
    {
      ...
    }
  ]
}
```

`type`がテンプレートの種類で、カードに合わせて10前後の種類あります。

| 例 | 用途 |
| --- | --- |
| surgery | 施術 |
| clinic | クリニック |
| menu | メニュー |
| doctor | ドクター |
| ... | ... |

スキーマはシステムプロンプトで定義し、フロントエンドには対応する型とtypeごとのコンポーネントがあります。APIサーバーはこのJSONをパースせず、そのままクライアントに返すだけです。スキーマを変えたいときはプロンプトとフロントエンドだけを触れば良いようになっています。Reactコンポーネント側でUIの変更が完結するようになり、UIの改善も遥かに行いやすくなりました。

ただし、このJSONもLLMの特性上、必ず指定のフォーマットで返ってくるとは限りませんので、パーサーが欠損を確認した場合は回答全体を定型の案内に差し替え、壊れた出力が画面に表示されないようになっています。それ以外のテキスト部分もサニタイズしてから描画するなど、**LLMは信頼できないヤツという前提**で設計をしています。

### 実データの取得もフロント側で

また、テンプレートJSONに載せるデータは最小限になっており、実データの取得も徹底的にフロントエンドに寄せています。**JSONに載るのはIDと表示に使う最小限のラベルのみ**にし、フロントエンドがIDから、既存のAPIから取得した最新のデータを用いて、最終的なUIを構築しています。  
実在するIDであることがLLMからの出力では保証しきれないなどの安全面もそうですし、画像や口コミ本文などの長いテキストもLLMのI/Oに載せずに済むためコスト面での恩恵も非常に大きいです。

## 安全性、ガバナンス面での恩恵

実データをフロントで取得しなおすようにしたことで、表示ルールについてのガバナンスも効かせやすくなりました。

美容医療の情報表示には、薬機法や医療広告ガイドラインなど消費者の安全を守るための様々なルールがあります。弊社でも、効果への言及を含む口コミ抜粋は本文を表示しないなど、則ったサービス運用を行っており、今回もこの制御をフロントエンドの表示層で行っています。例えば、口コミの抜粋を表示する直前に効果への言及を含むかをキーワードで判定し、含む場合は本文を出さず「◯◯さんの口コミ」という静的な文言に置き換えるなど、いくつかのフィルタがあります。

LLMの出力に本文が含まれる構成では、プロンプトで「効果に言及しないで」と指示してもその保証はできません。**フロントエンドで取得・判定をすることでスクリプトベースでの確実な制御を構造的に実現できるようになりました。**

それでも生成AIの特性上、意図しない表現を完全には排除できません。会話ログは全件保存して継続的にモニタリングし、表示の差し替えが発生したケースはイベントとして計測して、どの施術で起きたかを分析基盤で把握できるようにしています。検知した場合は速やかに調整する運用です。**構造で防ぎ、観測で補う**二段構えにしています。

## コスト: 実測で約1/10に

残るはコストです。方式切り替えの前後を同条件で比較すると、回答の平均文字数は1,041字から510字になり、約半減。その後、回答の仕様をリッチにする変更を重ねた現在の全期間平均でも638字で、約4割減を維持しています（トークンではなくあくまで文字数の比較です）。  
チャットは毎回、過去の会話履歴を入力に載せることになるため、**1メッセージの削減は会話の長さの乗数で効果がでてきます**。文字数ベースの概算では、数往復の会話で1/8〜1/16。**実測では、API利用コストは当初の想定の約1/10になりました**（プロンプトキャッシュも同時導入したため、その効果も一定あります）。

## まとめ

AI相談のフロントエンド設計では、LLMの役割を最低限に絞り、UIの構築も実データの取得もフロントエンドが行うことで、体験、安全、コストをどれも犠牲にしない機能の提供が開始できました。

* **体験**: リッチなUI、情報の最新性・正確性、機能の保守性を実現
* **安全**: 実データのフロント取得と表示直前の統制で、頑張る運用ではなく構造で守る
* **コスト**: all LLMバージョンから実測約1/10

「LLMを何のために使うのか」「ユーザーにどんな価値を届けたいのか」を起点にアプリケーション全体を設計し、LLMでしか提供できないことを担ってもらうのがベストです。今回の開発では、エンジニアがプロダクトマネージャーやデザイナーと「最終的にユーザーへ何を届けたいか」を会話しながら実装を進められたことで、短期間で要件を満たす機能開発ができました。

**本件に限らず、体験も安全もコストもトレードオフではありません。適切に設計・実装すれば、すべて担保できるはず。均衡点を見つけるのがエンジニアの役割であり、今後の価値や存在意義なのだと思います。**
