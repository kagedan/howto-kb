LLM のコスト最適化には大きく 2 つの軸があります。1 つは「1 クエリで何を読ませるか」= **入力トークンを減らす** 設計、もう 1 つが本稿で扱う「そもそも検索・生成をするか」= **重い処理の回数を減らす** 設計です。

本稿は後者、**「毎回検索しない」** アーキテクチャの話です。RAG (Retrieval-Augmented Generation) を本番運用に乗せると、コストの主因は 1 クエリあたりの入力トークンよりも **「全クエリに対して毎回フル検索 + フル生成を回していること」** に移ります。ここを設計で削ると、月間 100 万クエリ規模で **$50,000 → $3,000（約 1/15〜1/20）** まで落ちる、という構造を整理します。

[ShintaroAmaike 氏の記事「RAG のコスト問題を 1/15 に削る ―『毎回検索しない』アーキテクチャの設計」](https://zenn.dev/shintaroamaike/articles/ecaaf3b4a8a77d) が直接の触発元です。具体的で優れた実装記事だったので、本稿はそれを教材に、筆者が **設計判断の軸** として自分なりに整理し直したものです。正確な実装は原典が一次情報なので、RAG を本番運用する方は原典と本稿を併せて読んでいただくのが前提です。

## TL;DR

* RAG 本番運用のコスト主因は「入力トークン量」より **「毎回フル検索 + フル生成を回す回数」**
* 本質は *計算配置（computation placement）* 問題 — リクエストパス上の計算を **precompute / cache / route** で外へ追い出す（DB のマテビュー・CDN・入場制御と同じ系譜）
* 原典の「想定質問インデックス」は **LLM 版マテリアライズドビューに近い**。コストは消えるのでなく生成・更新コストとして資産側へ移る、と捉えると損益が読みやすい
* 階層化（分類器→事前回答→軽量→高性能）で筆者が気になったのは **層の境界**。律速は分類器で、誤りの向き（品質劣化 vs コスト増）は非対称
* 「1/15」の効き目は構成だけでなく **クエリ分布の形** に左右される（頭でっかちなら大きく効く）→ 作り込む前に分布を測っておきたい
* 入力設計（縦）× 処理回数設計（横・本稿）の **二段ロケット**
* 前提は **静的コーパス**（社内マニュアル・FAQ 等、頻繁に更新されないデータ）

## 1. RAG のコストはどこで膨らむか

RAG の素朴な実装はこうです。

```
ユーザのクエリ
  → 埋め込み生成
  → ベクトル検索（関連チャンク取得）
  → 取得チャンク + クエリを LLM に投入
  → 回答生成
```

PoC ではこれで十分動きます。問題は本番でクエリ数が伸びたときです。上の経路は **すべてのクエリに対して「埋め込み + 検索 + 生成」をフルセットで実行** します。月間 100 万クエリなら、100 万回フル生成が走る。1 クエリの入力を入力設計の工夫で半分にしても、**回数そのものが減らなければ線形に積み上がる** わけです。

ここで効くのが、**「全クエリが同じ重さの処理を必要としているわけではない」** という観察です。

* 全体の数割は「営業時間は？」「返品方法は？」のような **定型質問** — 毎回検索する必要がない
* さらに一定割合は「そもそも RAG が不要」（雑談・範囲外） — 生成すら要らない
* 本当にドキュメント横断の推論が要る難問は **ごく一部**

にもかかわらず素朴 RAG は全部を最重量レーンに通します。**処理の重さをクエリの難易度に合わせて段階化する** のが本稿の主題です。

## 2. RAG を「計算配置問題」として見る

ここからは原典の実装を教材に、筆者が学びながら **設計の一般論** として読み解いた整理です。理解が及んでいない部分は原典の方が正確なので、原典との併読を前提に読んでいただければと思います。

筆者の整理では、RAG のコスト最適化の本質は **「リクエストパス上でやっている計算を、どこか別の場所に追い出せるか」** という *計算配置（computation placement）* の問題です。これは RAG 固有の話ではなく、DB のマテリアライズドビュー、CDN のエッジキャッシュ、Web の admission control（過負荷時の入場制御）と同じ系譜の発想です。リクエストごとに走る計算を減らす手段は、突き詰めると 3 つしかありません。

* **precompute（事前計算）**: 結果を先に作って置いておく
* **cache（再利用）**: 一度作った結果を使い回す
* **route / shed（振り分け・間引き）**: そもそも軽い経路に流す、やらない

原典「RAG のコスト問題を 1/15 に削る」が提案する仕組みは、この 3 手段を RAG に当てはめた具体例として読めます。中でも筆者が一番うまい応用だと感じたのが、原典の言う **「想定質問インデックス」** — precompute と cache を兼ねた仕掛けです。

### 2.1 「想定質問インデックス」は LLM 版マテリアライズドビューである

原典のアイデアはこうです。コーパス側から「このドキュメントに対して聞かれそうな質問」をオフラインで生成し、その回答まで先に作ってインデックス化しておく。クエリが来たら、まずこのインデックスに意味検索をかけ、ヒットすれば**検索も生成もせず**に事前生成済みの回答を返す。

筆者の見立てでは、これは DB の **マテリアライズドビューに近い** 発想です。「よく走るクエリの結果を実体化して持っておき、リクエスト時は参照するだけ」という、データ基盤では枯れた最適化を、LLM の Q&A に持ち込んだものと理解すると腑に落ちます。

この見方をすると、**コストの所在** も整理しやすくなります。マテリアライズドビューが「ビューの構築・更新コスト」を別途抱えるのと同じで、想定質問インデックスも **「質問と回答を事前生成する LLM コスト」と「コーパス更新時の作り直しコスト」** をオフライン側に抱えます。つまりこれは *コストをゼロにする* 仕掛けではなく、*コストをリクエストパスから資産側へ移し替える* 仕掛けです。回数が出るほど移し替えが得になる、という構造は前半で触れた入力設計の損益分岐と同じです。

### 2.2 「文字列キャッシュ」でなく「意味キャッシュ」にする意味と、その代償

素朴なキャッシュは「クエリ文字列 → 回答」を保存しますが、ユーザは同じことを「営業時間は？」「何時まで開いてる？」「いつ閉まる？」と無数の言い回しで聞くため、文字列一致ではほとんどヒットしません。想定質問インデックスが効くのは、**埋め込みの意味類似でヒット判定する**から、表現が違っても拾えるためです。

ただし筆者として一点付け加えたいのは、**意味キャッシュには文字列キャッシュに無い固有の事故がある**ことです。類似度のしきい値を緩めるとヒット率は上がりますが、「似ているが実は別の質問」に事前回答を返す *誤ヒット* が混じります。文字列キャッシュは完全一致なので誤答しませんが、意味キャッシュは **ヒット率（コスト削減）と誤答率（品質劣化）がトレードオフ** になる。ここのしきい値設計は、後述の分類器と並ぶ運用上の勘どころです。

## 3. 階層化の本質は「入場制御」— 境界をどう引くか

原典は具体的に 4 つの層（クエリ分類器 → 想定質問インデックス → 軽量モデルの通常 RAG → 高性能モデルのフル RAG）を提示しています。筆者はこれを、より一般的な **tiered serving + admission control（多段処理 + 入場制御）** パターンの一実装として捉えています。「全リクエストを最重量の経路に通す」のをやめ、難易度に応じて軽い経路へ振り分け、不要なものは入口で断る——CDN やロードバランサの世界では当たり前の構えを、LLM 推論コストに適用したものです。

層の顔ぶれ自体は素直なので、筆者が気になったのは **「層の境界をどう引くか」** です。原典は実装の提示に重きを置いているので、本稿ではこの設計判断の部分を自分なりに補ってみます。具体例として 4 層を再掲しつつ、各境界の論点を添えます。

| 層 | 役割 | コスト | 設計上の勘どころ（筆者の補足） |
| --- | --- | --- | --- |
| ① 分類器 | 要否・難易度を振り分け | 極小 | 全体の効率は **この精度が律速**。後述 |
| ② 想定質問インデックス | 定型質問に事前回答 | 最小 | しきい値で *ヒット率↔誤答* を調整（§2.2） |
| ③ 通常 RAG（軽量モデル） | 普通の質問 | 小 | 「とりあえず全部高性能」をやめるだけで効く |
| ④ フル RAG（高性能モデル） | 難問・横断推論 | 大 | ここに **到達させない**ことが全体最適 |

### 3.1 律速は「分類器」— 誤りの向きでコストが非対称になる

筆者がこのアーキテクチャで最も繊細だと考えるのは①の分類器です。理由は、**誤分類のコストが向きによって非対称**だからです。

* 難問を③（軽量）に落とす → **品質劣化**（誤答・取りこぼし）
* 易問を④（高性能）に上げる → **コスト増**（無駄に高い経路）

この 2 つは等価ではありません。FAQ 応答なら品質劣化の方が痛い（誤答はブランド毀損）し、社内向け下書き生成ならコスト増の方が許容できる場合もある。つまり **「迷ったら上へ倒すか、下へ倒すか」は用途で先に決めるべき方針判断** であって、技術的に一意には決まりません。分類器を作る前に「この用途は fail-up か fail-down か」を定義しておくのが、筆者の考える正しい順序です。

### 3.2 階層化は「ファネル」— 効果は通過率の積で決まる

もう一つの見方として、この 4 層は **ファネル**です。①で何割を断ち、②で何割を吸収し、③で何割を捌き、④に何割残すか。全体コストは各層の「単価 × 通過率」の総和で決まるので、**最重量の④の通過率を下げることが、単価の高さゆえに最も効きます**。逆に言うと、②③で取りこぼして④に流れる率が想定より高いと、見積もりは簡単に崩れます。設計目標は「各層の精度」ではなく「④通過率を所定値以下に保つこと」だ、と置くと運用指標が立てやすくなります。

## 4. 「1/15」を効かせるのは『分布』— まず自分のクエリ分布を測る

原典は月間 100 万クエリ規模で **$50,000 → $3,000（約 1/15）** という試算を示しています。とても大きな数字で独り歩きしやすいので、筆者の理解を一点だけ補足しておきます。この削減率が出るかどうかは、**4 層構成そのものより、クエリの分布が頭でっかち（head-heavy）かどうか**に大きく左右される、という点です。

カギは②の想定質問インデックスで「どれだけ吸収できるか」です。

* **頭でっかちな分布**（少数の定型質問が全体の大きな割合を占める）→ ②で多くを事前回答に落とせるので、原典のような大幅削減が効きやすい
* **ロングテールな分布**（質問が多様にばらける）→ ②のヒット率が下がり、削減幅はぐっと小さくなる（どの程度かは分布次第）

つまり効果の源泉はアーキテクチャ単体ではなく、**「自分のクエリ分布の形」** にある、というのが筆者の受け取りです。

だからこそ、**作り込む前に自分のクエリログの分布を測っておく**ことをおすすめします。上位の質問が全体の何割を占めるか（どれくらい偏っているか）が分かれば、②の事前生成にどれだけ投資すべきかが先に見えます。偏っていれば想定質問インデックスに重点を、ばらけていれば③の軽量化と①の棄却に重点を——という配分の判断ができます。これは前半の入力設計で「反復頻度の実測に基づいて構造化範囲を決める」と書いたのと同じで、**「測ってから作る」** が両軸に共通する作法だと感じています。

そして、「1 クエリを安くする」入力設計が **縦方向の最適化**、本稿の層別振り分けが **横方向（本数）の最適化**。両者は独立なので、掛け合わせると削減は乗算で効きます。

## 5. 前提条件と適用範囲

この設計が強く効くのは **静的コーパス** — 社内マニュアル・製品 FAQ・規程集など、**頻繁には更新されないデータ** が対象のときです。理由は明快で、想定質問インデックス（事前生成回答）は **コーパスが変わると作り直しが必要** だからです。

| コーパスの性質 | 想定質問インデックスの相性 |
| --- | --- |
| 静的（マニュアル・FAQ・規程） | ◎ 事前生成が長く使える |
| 準静的（月次更新の商品情報 等） | ○ 更新時に差分再生成 |
| 動的（リアルタイム在庫・株価 等） | △ 事前生成が陳腐化しやすい。①③④中心に |

動的データが主体なら、②の比重は下げ、①の振り分けと③の軽量化に寄せる、という調整になります。**「自分のコーパスはどの位置か」を最初に見極める** のが設計の出発点です。

## 6. 本番移行の落とし穴

ShintaroAmaike 氏の記事でも強調されていますが、ローカル PoC から本番に移す際の典型的な詰まりどころを挙げます。

### 6.1 ベクトル DB・埋め込み API・抽象化レイヤを「最初から本番想定」で選ぶ

PoC で手軽な in-memory ベクトル検索や特定 SDK 直叩きで作ると、本番でスケール要件・運用要件に当たって作り直しになります。**永続化・スケール・差し替え可能性を最初から織り込む**（埋め込み API やベクトル DB をアプリから抽象化レイヤ越しに呼ぶ）ことで、PoC → 本番の差分を最小化できます。

### 6.2 想定質問の生成品質がヒット率を左右する

②のヒット率がコスト削減率そのものなので、想定質問が現実のユーザ質問とズレていると効きません。**実際のクエリログから頻出意図を抽出し、想定質問の生成プロンプトに反映** する改善ループを回すのが定石です。

### 6.3 分類器の誤振り分けコスト

①が難問を③（軽量モデル）に送ると品質が落ち、易問を④（高単価）に送るとコストが無駄になります。**誤分類の方向ごとのコスト（品質低下 vs 単価増）を見て、分類器のしきい値を調整** します。迷ったら上位レーンに倒す（品質優先）/ 下位に倒す（コスト優先）の方針を先に決めておくこと。

### 6.4 キャッシュの陳腐化を検知できない

コーパス更新時に②の再生成を忘れると、古い回答を返し続けます。**コーパスのバージョンと想定質問インデックスのバージョンを紐づけ、不一致を検知して再生成をトリガ** する仕組みを運用に組み込みます。

## 7. まとめ — 「1 クエリを軽く」と「重いクエリを減らす」の二段ロケット

RAG のコスト最適化は、2 つの独立した軸の掛け算です。

* **縦軸（入力設計）**: 1 クエリあたりの入力を絞る（何を読ませるか）
* **横軸（本稿）**: 重い処理に到達するクエリ本数を絞る（そもそも検索・生成するか）

素朴 RAG は「全クエリ × フル処理 × 高単価モデル」という最悪の組み合わせになりがちです。これを、

1. 入口で振り分け（①）
2. 定型は事前生成で返す（②）
3. 大半は軽量モデル（③）
4. 高単価は難問だけ（④）

に再設計するだけで、月間 100 万クエリ規模で 1 桁以上のコスト差が現実に出ます。

そして重要なのは、これが **「推論時計算をオフラインに寄せる」** という一般原則の RAG 版だということです。同じ計算を「1 クエリごとに課金される場所」でやるか「1 回作れば使い回せる場所」でやるか——この問いは RAG に限らず、LLM を本番運用するあらゆる場面で効きます。**動かすこと自体が無料ではない時代**、推論をオフラインに逃がせないか、を設計の最初に問う価値があります。
