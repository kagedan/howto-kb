前の記事（[AIエージェント、何を任せて、何を残すか](https://zenn.dev/aidecodelabjp/articles/ai-agents-boundary-2026)）で、エージェントに何を任せるかの境界線を書いた。

任せる範囲が決まったら、次の論点は「**そのエージェントの良し悪しをどう測るか**」、そして「**失敗したときにどう巻き戻すか**」だ。

2026年5月、現場の実務者の判断軸が一段ずれた感覚がある。SWE-bench のスコアやベンチマーク順位ではなく、**「やり直しコスト」** でエージェントを選ぶ動きが出てきた。完璧な8割より、失敗が安い7割のほうが、本番では回る。

この記事は、評価軸の再構成とロールバック設計を、2軸で実務的に整理する。

## 従来の評価軸の限界

エージェントの評価軸として、2024年〜2025年に定着したものはおおよそ3つだった。

* **タスク完了率**（SWE-bench Verified、TAU-bench など）
* **応答品質**（MT-bench、Arena-style 投票）
* **コストパフォーマンス**（トークン単価 × 性能）

これらは「閉じた環境での性能テスト」としては有効だ。ベンチマーク間で順位がきれいに出るし、リーダーボードに乗りやすい。

しかし2026年5月時点で、本番運用の現場が直面している問題は、これらの軸では測れない。

## 実務軸:「やり直しコスト」が中心になっている

私が観察している範囲では、エージェントを長期運用しているチームの判断軸はこうずれてきている。

* **やり直しコスト**: エージェントが失敗したとき、巻き戻しに何ステップ必要か
* **ロールバック容易性**: 失敗を検知してから「失敗前の状態」に戻すまでの時間と作業量
* **失敗の警告精度**: エージェント自身が「これは自信がない」と人間に通知できるか
* **副作用の可逆性**: 外部システム（DB / API / メール送信）への副作用が取り消せるか

これらは従来のベンチマークには出てこない。理由は単純で、ベンチマークは「失敗しないこと」を測るが、本番運用では **失敗したあとどうなるか** が決定的だからだ。

### 具体例: 同じ性能でも採用が分かれる

たとえば、2つのエージェントが SWE-bench Verified で同じ73%のスコアを出したとする。従来軸ではタイ。

実務軸を当てると分かれる。

* エージェント A: 失敗したコミットを `git revert` で1コマンド取消し、ロールバック数秒
* エージェント B: 失敗時に複数ファイルに書き込み済み、人間が手動で整合性チェック、ロールバック数時間

同じ73%でも、A は本番投入される、B は試験運用で止まる。性能が同じでも、**やり直しコストが桁違い** になる。

### 評価フレームの最小構成

自社で評価フレームを作るときの最小構成は、こうなる。

1. **タスク成功率**（従来通り）
2. **失敗時の検知率**（エージェント自身がエラーを宣言できるか）
3. **ロールバック工数**（成功時の状態に戻すまでの手作業ステップ数）
4. **副作用範囲**（取り消せない外部影響の有無）

この4軸で見ると、SWE-bench スコアが30位のエージェントが、1位のエージェントより本番採用される、というのが2026年6月の実情だ。

## ロールバック設計の4パターン

ロールバック設計は「失敗してから考える」ものではない。**エージェントを導入する前に決めておく** ものだ。

2026年5月時点で観察できる、現場で使われているロールバック設計は大きく4パターンに分かれる。

### パターンA: 即時取消（Git revert型）

エージェントの操作が **追記型** で、過去の状態が保存されているケース。

代表例:

* コードコミット（Git revert）
* ドキュメント編集（バージョン履歴）
* API呼び出しで idempotent な操作

特徴:

* ロールバック = 1コマンド（数秒〜数十秒）
* 失敗の影響範囲が「変更分のみ」に閉じる
* エージェントを攻め気味に運用できる（失敗してもすぐ戻せる前提）

向いている業務: コード生成、ドキュメント作成、設定ファイル変更

### パターンB: 段階的ロール戻し（DBトランザクション型）

エージェントが **複数の操作を連鎖** させるケース。途中で失敗すると、部分的な変更が中途半端に残る。

代表例:

* 複数テーブルへの DB 書き込み
* 複数サービスをまたぐ API 連携
* ファイルシステムの一括変更

特徴:

* トランザクション境界の設計が必須
* ロールバック = トランザクション rollback または compensating action
* 失敗の影響範囲が「論理単位」に閉じる

向いている業務: 受発注処理、データ移行、複数システム間の同期

### パターンC: 人間チェックポイント挿入（HITL pause型）

エージェントが **取り消しにくい操作** を実行する直前に、必ず人間に確認を求めるケース。

代表例:

* メール送信前のレビュー
* 公開記事の publish 前承認
* 顧客への自動返信の差し戻し

特徴:

* 速度は落ちる（人間の応答待ち）
* 失敗の影響範囲を「人間の判断ミス」に絞れる
* エージェントの自律性を犠牲にして、副作用の制御を取る

向いている業務: 顧客対応、対外コミュニケーション、決裁プロセス

### パターンD: 事前合意による fail-safe ガードレール

エージェントが **完全に取り消せない** 操作をする場合の最終手段。

代表例:

* 金銭の支払い実行
* 物理デバイスへの命令送信
* 第三者に届くアクション（SMS、電話発信）

特徴:

* 「やり直し」自体が不可能なので、**事前にできること** で制御する
* 上限金額、許可されたドメイン、時間帯制限などのガードレール
* エージェントの操作前に多段チェック

向いている業務: 決済、物理制御、対外的に取り返しのつかない操作

### 4パターンの選び方

1つのシステムで4つ全部を使い分けるのが普通だ。「この業務は A で、この業務は C」と業務単位で決める。

決め方の順序はこうなる。

1. **その操作は取り消せるか?** → No なら D
2. **取り消せるなら、自動か手動か?** → 手動の判断が必要なら C
3. **連鎖操作か単発か?** → 連鎖なら B
4. **単発で追記型か?** → A

このフローを最初に通すと、エージェントの導入設計が後戻りしない。

## 判断軸: 業務クリティカリティ × ロールバック容易性

業務とロールバック設計をマッピングする2軸マトリクスを使うと、整理がきれいに進む。

### 2軸の定義

**縦軸: 業務クリティカリティ**

* 高: 失敗すると顧客 / 売上 / 信頼に直接ダメージ
* 低: 失敗しても社内で完結、リカバリ可能

**横軸: ロールバック容易性**

* 高: 数秒〜数分で失敗前に戻せる（パターンA、B）
* 低: 戻すのに数時間以上、または戻せない（パターンC、D）

### 4象限と推奨対応

**第1象限（高クリティカリティ × 高ロールバック容易性）**

* 例: 本番コードの自動修正、本番DBの自動マイグレーション
* 推奨: パターンA / B + 監視強化
* エージェントを積極的に使える。失敗してもすぐ戻せる前提。

**第2象限（低クリティカリティ × 高ロールバック容易性）**

* 例: ドキュメント生成、社内タスクの一括処理
* 推奨: パターンA、人間レビューほぼ不要
* エージェントの自律性を最大化して問題ない。

**第3象限（低クリティカリティ × 低ロールバック容易性）**

* 例: 社内通知メール、定型レポート送信
* 推奨: パターンC（送信前プレビュー）
* 「低クリティカリティだから」と油断すると、信頼低下が累積する。

**第4象限（高クリティカリティ × 低ロールバック容易性）**

* 例: 顧客向けメール送信、決済、契約書送付
* 推奨: パターンD + 人間最終承認
* **エージェントの自律実行は禁止**。人間がトリガーを引く役割を担保する。

### 「完璧な8割より、失敗が安い7割」の意味

この2軸マトリクスを使うと、SWE-bench スコアやベンチマーク順位より、**自社の業務分布で第4象限が多いかどうか** のほうが、エージェント選びの決定要素になる。

第4象限が多い業務（金融、医療、法務）では、性能が高くてもロールバック設計の弱いエージェントは採用されない。逆に第1〜2象限が多い業務（社内ツール、開発支援、社内文書）では、性能が低くてもロールバック設計が堅いエージェントが採用される。

「完璧な8割より、失敗が安い7割が回る」という言い方は、この第4象限の存在を意識したものだ。

## まとめ

2026年6月時点で、AIエージェントの選び方は「性能」から **「やり直しコスト」** へ重心が移っている。

評価軸を タスク成功率 + 失敗検知率 + ロールバック工数 + 副作用範囲 の4つで再構成し、ロールバック設計を A〜D の4パターンに分けて業務ごとに当てはめる。

業務クリティカリティとロールバック容易性の2軸マトリクスで、自社の業務分布を確認すると、必要なエージェント像が見える。

「完璧な8割より、失敗が安い7割」は、第4象限が多い現場で特に効く。性能ベンチマークだけ見て選ぶと、ここで詰む。
