## 「それ、もう本当じゃないですよ」

AI に個人開発を任せていると、こんな事故が起きます。あるセッションが、過去の自分のメモに「この API は認証済みで動いている」と書いてあるのを見つけ、それを前提に次の実装を進める。ところが実際には、数日前にその API は差し替えられていた。**メモは嘘をついていないけれど、もう本当ではない。** そして誰もそれを教えてくれない。

複数のエージェント／複数のセッションで同じプロジェクトを回すと、この手の「古い記録を現在として踏む」事故が地味に効いてきます。今日はこの問題と、そこで作った仕組みの話です。

## みんなが最初に解く問題：「矛盾」

エージェントを並行で走らせると、まず出るのが**矛盾**です。片方が認証方式やスキーマを変えたのに、もう片方は知らない。共有状態がないから、次のセッションで静かに上書きされる。

これに対する定番の答えが、**共有メモリ / versioned memory**。決定を1か所に集め、誰がいつ何を変えたかを記録する。これは正しいし、必要です。私も file ベースのメモリ層を持っています。

## でも、その次に来る壁が「staleness（陳腐化）」

共有メモリは「**いま2人が食い違っている**」を解きます。けれど、その先にもう一段の壁があります。

> 書いた瞬間は本当だった記録が、地面が動いた後もそのまま「現在の事実」として残る。

versioning が教えてくれるのは「誰が・何を・いつ変えたか」です。でも本当に知りたいのは、**「この記録された主張は、いつ本当でなくなったのか」**。ここは version 履歴には写りません。コードの挙動・本番の状態・外部サービスの仕様——こういう“状態への主張”には賞味期限がありますが、メモにはそれが書かれない。

たちが悪いのは、古いメモほど**それらしく権威的に見える**ことです。「動いている」と断言された1行に、陳腐化フラグは立ちません。次のセッションはそれを事実として受け取り、その上に積み上げてしまう。

## 私がやっている3つのこと

完全な正解ではなく、実験中の現在地です。

### ① 状態への主張は「事実」でなく「未確認の仮定」として扱う

メモに「◯◯は動いている／完了した」と書いてあっても、それを判断に使う前に、**いまのセッションで一度観測し直す**。curl を叩く、差分を見る、実際に動かす。観測できなければ「未確認」と明示する。記録を信じるのではなく、記録を**再確認のトリガー**として使う、という運用です。

### ② 主張に賞味期限を持たせる

外部仕様のように確かめるのが高くつく事実は、確認した日付と有効期限をつけて記録します。期限が切れたら、それは「知っていること」ではなく「もう一度確かめること」に戻る。永遠に true のままの記録を作らない。

### ③ 事実の家を1つにして、毎回、門番を通す

同じ事実を複数の場所に書くと、片方だけ古くなって矛盾します。だから正本（single source of truth）を1か所に決め、**セッション開始時に決定論スクリプトが差分・矛盾を必ずチェックする**。掃除は何度でも再発するので、人手でなくセッション開始のゲートに持たせる。

## つまり

メモリは「真実を置く場所」ではなく、**「あとで確かめ直すべき主張を置く場所」** だと思っています。versioning は矛盾を解く。でも陳腐化を解くのは、記録そのものを疑う仕組みのほうです。
