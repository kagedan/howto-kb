この記事は、UiPath AgentHack 向けに作った `AgentOps Case Control Room` の実装メモです。

公開デモとコードはこちらです。

## 3行でいうと

* AIエージェント、人間、RPA、API、システムの作業を1つのcaseとして扱う
* リリース例外の調査、証拠収集、ポリシーブロック、人間承認、robot work itemをまとめる
* live UiPath実行は主張せず、UiPath-readyなローカルprototypeとimport packageとして境界を分ける

## なぜチャット履歴だけでは足りないか

AIエージェントは、調査、コード確認、API呼び出し、リスク判断、提案までできます。

でも、業務では「AIがそう言いました」だけでは足りません。

必要なのはcaseです。

* 誰が作業を開いたか
* AIが何を調べたか
* robotやAPIが何を確認したか
* どの証拠を見たか
* どこでpolicy blockされたか
* 誰が承認または却下したか
* 次に誰が再開できるか

`AgentOps Case Control Room` は、AIエージェント作業をUiPath風の管理可能なcaseへ変換するデモです。

## デモシナリオ

題材は、payment serviceのproduction release exceptionです。

流れは次のようにしました。

1. 人間がrelease-risk caseを開く
2. AI coding agentが調査する
3. robotがticket、pull request、failed test、owner evidenceを集める
4. policy gatewayがregression test failureとproduction deployment riskを検出する
5. production actionをblockする
6. Action Center風taskで人間へ判断を戻す
7. service ownerがregression修正までreleaseを拒否する
8. handoff reportがevent IDを残す

ここで見せたいのは、AIがすごいことよりも、AIが動いた後に業務として扱えることです。

## 実装の中心

内部では、AgentOps event streamを中心にしています。

```
human event
ai agent event
robot event
api event
policy event
approval event
handoff event
```

これをUiPathの概念に寄せて変換します。

| UiPath concept | Local artifact |
| --- | --- |
| Maestro Case | case spec |
| Action Center task | approval task model |
| Robot queue | robot work items |
| Orchestrator transaction | transaction log |
| Decision log | action center decision log |
| Handoff | evidence-linked handoff report |

ローカルprototypeでも、case state machineを持たせています。

blocked、more-evidence、owner-signoff、rejectedなど、業務上の状態を明示します。

## 人間承認を中心に置く

AIエージェント運用で危ないのは、AIが作業を進めること自体ではありません。

危ないのは、どこで人間が止めたのか分からないことです。

このデモでは、production actionをpolicyで止め、人間の承認・却下をcaseの中に残します。

人間承認は、コメントではなく、業務イベントです。

## 検証したこと

ローカルでは次を実行します。

```
cd uipath-agenthack
bash scripts/run_uipath_local_checks.sh
```

検証結果の例です。

```
uipath_verify_ok
state_machine=ClosedRejected
transaction_events=6
action_decision_events=9
```

このように、見た目だけではなく、case data、robot work item、decision log、state machineを検証対象にしています。

## この記事で主張しないこと

このデモは、live UiPath Automation Cloud executionを主張しません。

安全な主張範囲は次です。

* verified local prototype
* UiPath-ready implementation package
* Action Center風approval model
* robot work item model
* Orchestrator transaction logの形
* evidence-linked handoff report

ハッカソンでは「できたこと」と「次に接続すること」を分ける方が、提出物の信頼性が上がります。

## 読者が自分のAI業務に使うなら

AIエージェントを業務に入れるなら、チャットではなくcaseに変換するのがよいです。

最初は次だけでも十分です。

1. `case_id` を作る
2. AI、人間、robot、APIのeventを同じ時系列にする
3. policy blockを明示する
4. human approvalをeventとして残す
5. robot work itemとdecision logを分ける
6. 最後にhandoff reportを作る

これで、次の人間や次のAIが安全に再開しやすくなります。

## まとめ

`AgentOps Case Control Room` は、AIエージェントの作業を業務caseへ変換するデモです。

AIが何をしたかだけでなく、robotが何を確認し、人間がどこで止め、どんな判断で終わったかを残します。

AIエージェントを実務に入れるなら、case、承認、証拠、handoffを最初から設計するのが重要です。
