最近よく耳にする「ハーネスエンジニアリング」という言葉。

「git workflowをbashで書き直してるだけでは？」「TDDを言い換えてるだけでは？」みたいな声もあって、ちょっとバズワードっぽく見えるのもわかります。

でも、設計思想として整理しておく意味はあるなと感じています。

私はここ数年の流れを、プロンプトエンジニアリングからコンテキストエンジニアリングへ、そしてハーネスエンジニアリングへと関心が広がってきた動きとして見ています。この記事では、そのあたりを自分なりに整理してみます。

## ハーネスエンジニアリングとは？

「ハーネス（harness）」という言葉は、馬具の「馬を制御するための装具」が語源です。AIの文脈では、**エージェントの自律動作を制御する仕組み全体**を指します。

ただし、この言葉の使われ方は各社でかなり違います（参考：[ハーネスエンジニアリングとは](https://www.youtube.com/watch?v=VGL45KZGOkM)）。

* テスト用の評価フレームワークとして語る会社
* エージェントのオーケストレーション層として語る会社
* プロンプト・ツール・ガードレールの全体設計として語る会社

定義はゆれているとはいえ、重なっている論点はあるように思います。

> **「何をさせるか」だけでなく、どんな情報・制約・検証環境を与えるかの設計が重要**

エージェントが自律的に動くほど、暴走を防ぐガードレールだけでなく、判断に必要な知識配置やフィードバックループの設計も効いてきます。

ハーネスエンジニアリングはその設計を体系化しようとする営みです。

## Anthropicの設計思想

Anthropicのエンジニアリングブログ（[Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)）は、ハーネス設計をかなり具体的に書いていて、とても参考になりました。

### 3つのエージェントを分離する

Anthropicが提唱するのは以下の3分離です。

* **仕様エージェント**：要件を整理してspecを生成する
* **開発エージェント**：specをもとに実装する
* **評価エージェント**：実装結果を検証する

単一のエージェントに全部やらせるのではなく、役割ごとに分けることでコンテキスト汚染を抑えやすくなります。各フェーズの品質も上げやすいです。

### 「ハーネスの仮説」という考え方

このブログでいちばん印象に残ったのは、この考え方でした。

> 「ハーネスのすべてのコンポーネントは、モデルが自力ではできないことについての仮説をエンコードしている。それらの仮説は検証する価値がある。間違っているかもしれないし、モデルの改善によってすぐに陳腐化しうるからだ。」

要するに、ハーネスのコンポーネントは「モデルの弱点を補う補助具」だということです。モデルが進化すれば、一部は不要になります。

実際、Opus 4.5 では必要だったコンポーネントが、Opus 4.6 ではモデルが自力でこなせるようになって外せた、という話も出てきます。

ハーネスを固定的な正解として扱うのではなく、モデル能力に応じて入れ替わる足場として見る発想です。作り込みすぎると、モデル更新後にはそのまま負債になることもあります。

## OpenAIの設計思想

OpenAIも[ハーネスエンジニアリング](https://openai.com/ja-JP/index/harness-engineering/)について公開しています。Anthropicと比べると、少し重心が違います。

### AGENTS.mdは目次として使う

OpenAIは、エージェントに最初から大量の指示を流し込むのではなく、AGENTS.mdを小さく安定した入口にして、詳細はリポジトリ内のドキュメントに辿らせる構成を紹介しています。実質的には「目次」や「入口」に近い使い方です。

これはかなり理にかなっています。AGENTS.mdが肥大化すると、重要な指示が埋もれるし、エージェントもどこを見ればいいのか迷いやすくなるからです。

### リンターのエラーメッセージに修正指示を埋め込む

OpenAIの事例で面白かったのが、リンターのエラーメッセージ自体に「どう直すか」を埋め込む手法です。エージェントがエラーを読んだ時点で、修正の方向までわかるようにしてあります。

[ハーネスエンジニアリングのベストプラクティス](https://nyosegawa.com/posts/harness-engineering-best-practices-2026/)でもこの手法は実践知として整理されており、補助線として参考になりました。

## 共通原則：ハーネスは軽量に保つ

Philipp Schmidの [Agent Harness in 2026](https://www.philschmid.de/agent-harness-2026) は、AnthropicやOpenAIの動きを踏まえて、ハーネスをどう捉えると理解しやすいかを第三者視点で整理した記事でした。これもかなり参考になりました。

この記事では、ハーネスをOSにたとえています。

> CPU = モデル、RAM = コンテキストウィンドウ、**OS = Agent Harness**、アプリケーション = エージェント

OSはCPUの能力を引き出す基盤ですが、CPUが進化すればOSも変わります。Schmidは、ハーネスも同じようにモデル進化に応じて作り替わるものだと整理しています。

個人的にいちばん刺さったのは、この指摘でした。

> 競争優位は「プロンプトではなく、ハーネスが記録する軌跡（データ）」にある

さらに、Manusが6ヶ月で5回リファクタリングした事例を挙げながら、固定化しすぎた設計はモデル更新のたびに見直しを迫られるとも書いています。

この記事から私が引き出した設計原則は3つです。

* **シンプルさ**：制御フローを複雑化しない
* **削除可能性**：モジュール設計で将来のモデル更新に対応できるようにする
* **データ資産化**：ハーネスが記録する軌跡は、将来の改善原資になりうる

「薄く保つ」という見方は、Anthropicの「仮説として検証し続ける」という思想とも相性がよさそうです。もちろん、薄すぎるハーネスはエージェントの暴走や品質低下を招くので、そのバランスは難しいのですが。

## 自分が作りたかったワークフロー

こうした設計思想を踏まえて、私が作りたかったのは次の6フェーズのワークフローです。AnthropicやOpenAIの実装をそのまま再現するというより、参考になった論点を自分なりに組み合わせたイメージです。

### フェーズ構成

| フェーズ | 内容 |
| --- | --- |
| **1. Brainstorm** | 対話型LLMに1問ずつ質問させ、アイデアを反復的に掘り下げてspec.mdを生成 |
| **2. Plan** | spec.mdを推論モデルに渡し、小さなステップに分解 |
| **3. Execute** | サブエージェントを割り当てて実装 |
| **4. Test** | フロントエンドではビジュアルテストも含む |
| **5. Review** | コードレビュー |
| **6. Loop** | レビュー修正→テスト→レビューをテスト完了まで自動ループ |

### 設計で重視したポイント

* フェーズを飛ばせないようにして、「とりあえず実装」を防ぐ
* 実装はサブエージェントに委任して、メインコンテキストを汚しにくくする
* Claude Code、Codex、Gemini などを用途に応じて切り替えられるようにする
* レビュー→修正→テストのサイクルを、人の手をなるべく入れずに回す

コンテキスト汚染が積み重なると後半の判断精度が落ちる感覚があったので、特にサブエージェント分離は重視しています。

## OSSハーネスを4つ調査した

上記のワークフローを実現するために、既存のOSSハーネスを4つ調査しました。

* **superpowers**: スキル駆動のプロセス強制型
* **takt**: YAML定義の状態マシンで制御するマルチエージェントオーケストレーション
* **everything-claude-code**: 30エージェント＋138スキルの包括的ハーネス
* **oh-my-claudecode**: 「完了するまで止めない」永続モードが特徴

それぞれの特徴と比較は、次回の記事で詳しく書くつもりです。結論だけ先に言うと、私は **takt** を採用しました。

## まとめ

ハーネスエンジニアリングを一言でまとめるなら、私は「エージェントの能力と限界を見極めながら、過不足のない実行環境と制御系を設計すること」だと理解しています。

過剰なハーネスはモデルの可能性を制限し、保守コストを生みます。不足したハーネスはエージェントの暴走や品質低下を招きます。

大事なのは、正解っぽいハーネスを一度作って終わりにしないことだと思っています。モデルが変われば、効く足場も変わります。

次回はOSS4つの詳細比較と、最終的にtaktを選んだ理由を書きます。
