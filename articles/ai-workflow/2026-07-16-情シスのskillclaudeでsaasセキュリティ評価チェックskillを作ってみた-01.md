# 【情シスのSkill】ClaudeでSaaSセキュリティ評価チェックSkillを作ってみた

## はじめに

皆さんこんにちは！

に引き続き【情シスのSkill】シリーズ第2弾です。  
今回は「このSaaS、社内で使っていいですか？」という社内の問い合わせで毎回調べていたセキュリティ評価業務を、ClaudeのSkillとして作ってみた話です。

> 想定読者
>
> * SaaSの利用申請対応・セキュリティ評価に時間を取られている情シス・CorpEngの方
> * ClaudeのSkill機能で自分の業務を定型化してみたい方
> * 「セキュリティチェックって結局何を見ればいいの？」と聞かれて言語化に困ったことがある方

## 導入：毎回ゼロから調べていた過去の私

情シスをやっていると、こういう問い合わせが飛んでくるかと思います。

「〇〇っていうツール、便利そうなので使っていいですか？」

「いいよ！」と即答できればラクなのですが、会社のデータを預ける以上そうもいかないので私は毎度

1. 運営会社どこだっけ…と公式サイトを検索
2. プライバシーポリシーと利用規約の読み込み
3. ISO 27001やSOC 2の取得状況を確認
4. データの保存先リージョンを調べ
5. 過去のインシデント履歴をググる

という調査を**毎回ゼロから、しかも毎回微妙に違う観点で**やっていました。  
所要時間は1件あたり1〜2時間。  
しかも属人的なので、「前回のツールでは学習利用のオプトアウトを確認したのに、今回は見忘れた」みたいな抜け漏れが起きたりします。

ということで「観点が毎回ブレるなら、観点そのものを固定すればいいんじゃね？」と思い、この調査業務をClaudeのSkillに落とし込むことにしました。

## Skillとは

Claudeの「Skill」は、特定のタスクの手順・チェックリスト・テンプレートをまとめたフォルダをClaudeに渡しておくと、該当するタスクのときにClaudeが自動で参照して動いてくれる仕組みです。

「毎回同じ長文プロンプトをコピペする」運用から卒業でき、手順書・参照資料・出力テンプレートを分割して管理できるのが強みです。詳しくは公式ドキュメントをどうぞ。

<https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview>

## 作ったもの：it-security-review

### 何ができるのか

「〇〇は社内で使っても大丈夫？」とClaudeに聞くと、以下のワークフローが自動で走ります。

1. **前提確認**：評価対象のプラン（Free / Business / Enterprise等）と、扱うデータ区分を確認。プランが不明なら「法人利用想定でBusiness/Enterprise相当を基準に評価」というデフォルト方針にフォールバック
2. **Web調査**：公式サイト・プライバシーポリシー・Trust Center・認証機関の登録簿など、一次情報を優先して収集
3. **チェックリスト評価**：6分野18項目を3段階（低・中・高リスク）で評価
4. **レポート生成**：日本語のMarkdownレポートを出力

### チェック内容：6分野18項目

Skillの心臓部であるチェックリストを紹介します。  
今まで頭の中と過去メモに散らばっていた観点を、6分野18項目に棚卸ししました。

| 分野 | 項目 |
| --- | --- |
| **A. 運営元と基本情報** | ①運営企業・所在地 ②インシデント履歴 ③政府開示義務（中国国家情報法、米CLOUD Act等） ④事業継続性 ⑤サポート体制・日本語対応 |
| **B. プライバシーとデータ保護** | ⑥規約・HTTPS通信 ⑦学習利用とオプトアウト ⑧保存時暗号化 ⑨データ削除・エクスポート |
| **C. データの保管場所** | ⑩保存サーバー所在地 ⑪保管国の選択可否 ⑫最終的な所在（日本国内/GDPR圏に収まるか） |
| **D. 認証とアクセス制御** | ⑬パスワードポリシー ⑭MFA/SSO ⑮操作ログ |
| **E. 外部認証** | ⑯第三者認証（ISO 27001、SOC 2、Pマーク等） |
| **F. AI機能** | ⑰AI機能と学習利用 ⑱外部送信・保持・自律行動のリスク |

分野Fは最近のSaaS事情を反映した項目です。今どきのSaaSはなんでもかんでもAI機能が生えてくるので、「入力データがモデル学習に使われないか」「AIエージェントが勝手に外部へデータを送らないか」は必須の確認観点にとしました。

### 工夫ポイント：「情報がないこと」もリスクとして扱う

気をつけた点は、**公式情報から確認できない項目は憶測で埋めさせない**ことです。

LLMに調査させると、学習データの記憶からそれっぽい答えを出してしまうことがあります。セキュリティ評価でこれをやられると誤判定に直結するので、Skillには以下を明記しました。

* 必ずWeb検索を実施し、一次情報（公式ポリシー・Trust Center・認証登録簿）を最優先する
* 確認できない項目は「公式に情報未公開」と記録し、原則として高リスク評価にする
* ただし項目の性質で調整する（データ保護の根幹に関わる項目の未公開は高リスク、サポートSLAなど運用補助的な項目は中リスクでも可）

企業利用において「情報が公開されていないこと」自体がリスクである、という情シスの肌感覚をロジックに落とし込んだ形です。

また、総合判定（推奨／条件付き許可／非推奨）は機械的なスコアリングにせず、「社外秘データが漏洩した場合に取り返しがつくか」を軸に判断させています。  
同じ中リスクでも、扱うデータが機微であるほど厳しく判定する、という運用です。

## 効果と学び

### 効果

* 現状Skillを作ったばかりなので個人的な試運転の状態です。1製品の調査は大体15分くらいで完了します。
* 評価観点が18項目に固定されたので、**担当者やタイミングによる抜け漏れ・ブレがゼロになるはずです**
* レポートに「評価実施日」「前提条件（プラン・データ区分）」が必ず残るので、後から監査的に振り返れる
* レポートで確認できなかったところはメーカーに個別に問い合わせて確認して補完することができる

15分の内訳もほぼ「Claudeの調査を待ちながらレポートをレビューする時間」なので、体感の負荷はもっと軽くなっています。

### 学び

* **情シス業務はSkill化することができるものがある。** 情シスの業務の特性上非定型なタスクが多い中Skill化して体系的にかつ自動化する業務はある。
* **LLMには「調べない自由」を与えない**。「必ずWeb検索する」「未公開は未公開と書く」を明文化しないと、記憶ベースでそれっぽく埋めてくる。性善説では運用しない。
* **出力は業務の最終形まで作らせる**。レポートで止めず、稟議システムの承認コメント案まで出すことで「あとはコピペするだけ」になり、運用を想定したアウトプットとなった。

## 今回作成したSkillを公開します

今回作ったSkillはGitHubで公開しています。

<https://github.com/asaponi-corpeng/it-security-review>

チェックリストの項目や判定基準は`references/`配下のMarkdownに分離してあるので、自社のポリシーに合わせて項目を足したり、基準を厳しくしたりのカスタマイズも簡単です。

「うちではこの項目も見てるよ」という観点があれば、IssueやPRでぜひ教えてください！

## 更に向こうへ plus ultra

まだ個人レベルでの検証なので、社内で使えるか個人で試運転を続けてみてうまくいくようでしたら以下の発展も考えられます。

* ワークフローシステムと連携し、利用申請が起票されたら自動で評価が走る仕組み化
* 承認済みSaaSの定期再評価（規約変更・インシデント発生の検知）への展開
* GoogleChrome拡張機能の評価
* 評価済みレポートの蓄積を、SaaS棚卸し・コスト最適化のインプットに接続

セキュリティ評価に限らず、「毎回やっているのに毎回微妙にやり方が違う業務」はSkill化の絶好の候補だと思います。
