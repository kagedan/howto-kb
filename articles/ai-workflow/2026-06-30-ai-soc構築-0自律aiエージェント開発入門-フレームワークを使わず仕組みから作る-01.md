## はじめに：AIが、攻撃を自力で見つけた

ハニーポットとSIEMで攻撃を観測するプロジェクトを始めて、約2か月。集めた攻撃データを分析・可視化し、週次・月次のレポートまで自動で出せる環境を作り込んできました。

ある日、ふと思いました。「このデータ、AIに自分で分析させたらどうなるんだろう？」

1日分の攻撃データへのアクセス手段だけをAIに渡して、こう頼んでみました。

> 「この中から、気になる攻撃を自分で探して報告して」

やり方は何も指示していません。なのにAIは、自分で次々と問い合わせを組み立てて実行し、最後に——**私が何週間も手作業で追いかけていた攻撃**（SSHに侵入して裏口を仕込む一連の攻撃）を、誰にも教えられずに見つけ出したのです。

思わず声が出ました。「自分で、ここまでやるのか」と。

### なぜこの記事を書いたのか

正直、私はAIエージェントを作った経験がありませんでした。"エージェント"と聞くと難しそうで、専用フレームワークを覚えないと無理だと思い込んでいました。

でも作ってみたら、印象がひっくり返りました。**エージェントの仕組みは、拍子抜けするほど単純だった**のです。フレームワークも要りません。手を動かして組み立てれば、中身が全部わかる。

次の目標は、この環境を「AI SOC」——AIが自律的に動くセキュリティ運用——へ育てることです。その第一歩として、自律AIエージェントの作り方をゼロから学びました。**この記事は、その学びの共有です。** 専門家の解説ではなく、つい先日まで「難しそう」と思っていた人間が腑に落ちた道のりを書きました。

> この記事は、**自律AIエージェントを作ったことがない入門〜中級のエンジニア**に向けています。読み終わって「これなら自分にも作れそう」と思ってもらえたら成功です。

これは「AI SOC構築」シリーズの第0回。すべての土台になる「自律AIエージェントの作り方」から始めます。

> **この記事の読み方**  
> エージェントの作り方そのもの（前半）は、特別な環境がなくても誰でも再現できます。後半で私のハニーポットのデータにつなぎますが、**この道具をあなたの環境のデータ源（社内のログ検索、データベース、API、CSVなど）に置き換えれば同じ仕組みが動きます。** 「ハニーポットが無いと試せない」ことはありません。

## 1. AIエージェントの正体は「LLM＋道具＋ループ」だけ

結論から言います。**AIエージェントの正体は、たった3つの組み合わせです。**

* **LLM**（大規模言語モデル。文章を理解して返すAI。ChatGPT や Claude の中身）
* **道具（ツール）**（LLMに使わせる関数。例：「ログを検索する」「ファイルを書く」）
* **ループ**（LLMが道具を使い、結果を見て、次の手を決める——を繰り返す）

順番にするとこうです。

1. **あなたのプログラム**が、LLMに"道具のリスト"と"やってほしいこと"を渡す
2. **LLM**が「この道具をこの引数で使いたい」と言ってくる
3. **あなたのプログラム**が道具を実行し、結果をLLMに返す
4. **LLM**が結果を見て、次の手（別の道具／最終回答）を決める
5. LLMが「終わった」と言うまで2〜4を繰り返す

ポイントは、**考えて指示するのはLLM、実際に動くのはあなたのプログラム**という役割分担です。LLMは「使いたい」と言うだけで、実行するのは必ずこちら側。だから何をやらせるかを完全にコントロールできます。

> **用語：エージェント**  
> 道具とループで、自分で何度も判断しながら自律的に動くLLMのこと。1問1答で終わらず、目的に向かって自分で複数の手を打つのが特徴です。

「LangChain のような専用フレームワークが必要では？」と思うかもしれません。多くの入門記事はフレームワークを使いますが、それだと**中で何が起きているかが見えません。** この記事ではあえて使わず、Anthropic の SDK（次章で説明）だけで、上の1〜5を**手作業で1周**回してみます。そうすれば"エージェント"の正体がはっきり見えます。

## 2. まずはAIと1往復してみる（準備とhello world）

エージェントの前に、AIと1回やりとりするところから。使うのは Anthropic 公式の **SDK** です。

> **用語：API と SDK**  
> **API** はプログラムから機能を呼ぶ窓口。Claude もAPI経由で使えます。**SDK** はそのAPIを楽に使う道具箱（ライブラリ）。今回はPython用を使います。

### 準備

```
mkdir agent-tutorial && cd agent-tutorial
python3 -m venv .venv && source .venv/bin/activate   # 仮想環境（他と隔離する箱）
pip install anthropic                                 # 公式SDK
export ANTHROPIC_API_KEY='sk-ant-...'                 # APIキー
```

`ANTHROPIC_API_KEY` は[Anthropicのコンソール](https://console.anthropic.com)で発行する鍵です。使った分だけ課金される従量制で、ChatGPTやClaudeの月額プランとは別物。学習中の実験は1回数円ほどです。

> **用語：トークン**  
> AIが文章を扱う単位（だいたい単語のかけら）。**料金はこのトークン数で決まります。** `max_tokens=200` は「返答は最大200トークン」の意味です。

### 最初のコード

```
import anthropic

client = anthropic.Anthropic()          # 環境変数 ANTHROPIC_API_KEY を自動で読む

resp = client.messages.create(
    model="claude-haiku-4-5",           # 学習中は安いモデルで十分
    max_tokens=200,
    messages=[
        {"role": "user", "content": "あなたは何ができる？1文で。"}
    ],
)

print(resp.content[0].text)
```

返事が出れば成功です。

### 1点だけ気づいてほしいこと

最後の行は `resp.text` ではなく **`resp.content[0].text`** です。AIの返答は\*\*ただの文字列ではなく「ブロックの配列」\*\*で返ってきます。今回はテキストのブロック1つだけなので、0番目の `.text` を取り出しています。

この「配列で返る」が、次章でエージェントを理解する鍵になります。今はそれだけ覚えてください。

## 3. エージェントの一番大事な部分：道具を渡して、手で1周回す

ここが本記事でいちばん大事な章です。

1章で見た流れ（1〜5）を、コードにすると**4つのステップ（①〜④）**になります。これを**手作業で1周**回せば、中で何が起きているか完全にわかります。

### 道具を定義する

道具は「3つの情報」で定義します。

```
tools = [
    {
        "name": "get_weather",                                   # ① 名前
        "description": "指定した都市の現在の天気を返す。天気を聞かれたら必ずこれを使うこと。",  # ② 説明
        "input_schema": {                                        # ③ 引数の形
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "都市名（例: 東京）"},
            },
            "required": ["city"],
        },
    }
]
```

**②の説明（description）が重要です。LLMはこれを読んで「いつ使うか」を判断します。** ③の `input_schema` は引数の型を決める **JSON Schema** という書き方です。

> **用語：JSON Schema**  
> データの"型"を表す書き方。ここでは「`city` という文字列の引数が必要」と宣言しています。

### 前半：LLMが「道具を使いたい」と言ってくる

道具を渡して質問を投げ、返ってきたものを**観察**します（まだ実行しません）。

```
resp = client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=300,
    tools=tools,
    messages=[{"role": "user", "content": "東京の天気は？"}],
)

print("stop_reason:", resp.stop_reason)
for block in resp.content:
    print(block)
```

```
stop_reason: tool_use
TextBlock(text='東京の天気を確認します。', ...)
ToolUseBlock(name='get_weather', input={'city': '東京'}, id='toolu_01W2...', ...)
```

LLMは天気を知りません。だから**自分で答えず、「get\_weather を city=東京 で使わせてほしい」と要求**してきました。`stop_reason` も `end_turn` ではなく **`tool_use`** です。2章の「配列で返る」がここで効き、配列に **`ToolUseBlock`（道具の要求）** が入っています。

> **用語：ツール使用（tool use）**  
> LLMに道具を渡すと「この道具をこの引数で使いたい」と要求してくる仕組み。要求するだけで、実行するのはこちら側です。

`id='toolu_01W2...'` を覚えておいてください。結果を返すときにこのIDを使います。

### 後半：結果を返して、1周を完成させる

道具を実行し、結果をLLMに返します（本物のAPIの代わりに固定文字列を返す関数で十分）。

```
def get_weather(city):
    return f"{city}は晴れ、22度。"

messages = [{"role": "user", "content": "東京の天気は？"}]
resp = client.messages.create(model="claude-haiku-4-5", max_tokens=300, tools=tools, messages=messages)

messages.append({"role": "assistant", "content": resp.content})

for block in resp.content:
    if block.type == "tool_use":
        result = get_weather(**block.input)              # **input は {'city':'東京'} を city='東京' として渡す（辞書の展開）
        # ③ 結果を tool_result として履歴に追加（同じIDを付ける）
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": block.id,                 # 前半で来たIDと一致させる
                "content": result,
            }],
        })

resp2 = client.messages.create(model="claude-haiku-4-5", max_tokens=300, tools=tools, messages=messages)
print("stop_reason:", resp2.stop_reason)
print(resp2.content[0].text)
```

```
stop_reason: end_turn
東京の現在の天気は晴れ、22度です。
```

結果（晴れ、22度）を踏まえて答え、`stop_reason` も `end_turn` に変わりました。

### 4つのステップの意味

なぜこの4ステップが必要か。理由は**ClaudeのAPIが「ステートレス」だから**です。

> **用語：ステートレス**  
> サーバー側が会話を覚えていない、という意味。だから毎回、会話履歴をまるごと送り直します。

* **① 返答を履歴に追加**：APIは記憶しないので履歴を自分で持ちます。しかも**道具の要求を履歴に残さないとエラー**になります（結果を返す相手が履歴に無いため）。
* **② 道具を実行**：要求された関数を動かすだけ。ふつうの処理です。
* **③ 同じIDで結果を返す**：LLMは1回で複数の道具を要求することもあるので、**どの要求への結果かをIDで判別**します。結果は `user` メッセージとして送ります。
* **④ もう一度呼ぶ**：「質問→要求→結果」と育った履歴を送り直すと、LLMが最終回答を出します。

いま手作業で回した①〜④——これが**エージェントの中核**です。今回は1周で終わりましたが、LLMが「次は別の道具を」と言えばまた①に戻る。それを繰り返すだけで、AIは自分で何手も打つ自律的な動きになります。

毎回これを手で書くのは大変です。次章で、この繰り返しを**自動でやってくれる仕組み**に置き換えます。中身は今と全く同じです。

①〜④を手で回すのは大変だし、本物のエージェントは何周も回します。このループはSDKが**自動で回してくれます。** 覚える部品は2つです。

**① @beta\_tool —— 関数を「道具」にする。** Pythonの関数に `@beta_tool` を付けるだけで、**型ヒントと docstring から道具の定義が自動生成**されます（3章の手書きJSONが不要に）。

> **用語：型ヒント / docstring / デコレータ**  
> **型ヒント**は引数の型を書き添えるしくみ（`city: str` ＝ city は文字列）。**docstring**は関数直下の説明文（`"""..."""`）。**デコレータ**は関数の上に付ける `@◯◯` で、元の関数に機能を足す仕組みです。

**② tool\_runner —— ①〜④を自動で回す。** 「履歴に追加→実行→結果を返す→もう一度呼ぶ」を、LLMが「終わり」と言うまで自動で繰り返します。

3章のコードを、この2つで書き直します。

```
import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()

@beta_tool                               # これだけで関数が「道具」になる
def get_weather(city: str) -> str:
    """指定した都市の現在の天気を返す。天気を聞かれたら必ずこれを使うこと。

    Args:
        city: 都市名（例: 東京）
    """
    return f"{city}は晴れ、22度。"

runner = client.beta.messages.tool_runner(   # ①〜④のループを自動で回す
    model="claude-haiku-4-5",
    max_tokens=300,
    tools=[get_weather],
    messages=[{"role": "user", "content": "東京の天気は？"}],
)

for message in runner:                       # 1ターンごとに message が返る
    for block in message.content:
        if block.type == "text":
            print("[Claude]", block.text)
        elif block.type == "tool_use":
            print("[道具]", block.name, block.input)
```

```
[Claude] 東京の天気を確認します。
[道具] get_weather {'city': '東京'}
[Claude] 東京の現在の天気は晴れ、22度です。
```

3章と見比べると、**`input_schema` のJSONと、①〜④の手作業が、両方とも消えています。** `@beta_tool` が型ヒントとdocstringから定義を作り、`tool_runner` が裏で①〜④を回しているからです。**docstringがそのまま道具の説明になる**ので、「いつ使うか」を書くのが大事です。

つまり\*\*`tool_runner` の中身は、3章であなたが手で動かした①〜④そのもの\*\*。「フレームワークは中が見えない」という不安はもうありません。中身を理解した上で楽をしているだけです。

この形なら、道具を増やすだけでLLMが使い分け、自分で何手も打ちます。次章では偽の天気を**本物のデータ**に置き換えます。

## 5. 偽の道具を、本物のデータにつなぐ

仕組みは完成しました。あとは**偽の天気の道具を本物に差し替えるだけ**。道具はただのPython関数なので、「天気を返す関数」を「ログを検索する関数」に変えれば、実データで動き出します。

> ここから私のハニーポットのログ検索につなぎますが、**この道具をあなたの環境のデータ源（ログ検索、DB、API、CSVなど）に置き換えれば同じ仕組みが動きます。**

私の環境では攻撃ログは **OpenSearch**（大量のログを検索するエンジン）にあります。そこに問い合わせる道具を作ります。

```
import json
from anthropic import beta_tool
from msearch import run_msearch    # 私のOpenSearchに問い合わせる関数（全文は末尾のリポジトリに）

@beta_tool
def opensearch_search(query: str) -> str:
    """ログを検索して結果(JSON)を返す。データを調べたいときに使う。集計（件数を数える等）に使うこと。

    Args:
        query: 検索クエリ本体のJSON文字列。
    """
    try:
        q = json.loads(query)              # AIが書いた文字列を辞書に変換
    except Exception as e:
        return f"ERROR: queryが正しいJSONではありません: {e}"
    try:
        result = run_msearch({"q": q})     # 実際の検索（←ここをあなたのデータ源に差し替える）
    except Exception as e:
        return f"ERROR: 検索に失敗: {e}"
    out = json.dumps(result["q"], ensure_ascii=False)
    if len(out) > 8000:                    # 結果が長すぎるとLLMが扱いにくいので上限
        out = out[:8000] + "\n...(省略)"
    return out
```

ポイントは2つ。

* **`run_msearch(...)` の1行を、あなたのデータ源に差し替えるだけ**で残りは動きます。
* **エラーは文字列で返す**：失敗してもプログラムを止めず「ERROR: ...」を返すと、**LLMがそれを読んでクエリを直して再挑戦**できます。

ここで面白いのは、**OpenSearchのクエリ文法をあなたが覚える必要がない**こと。クエリは**AIが書く**からです。

```
runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",                 # 実際のクエリを書くので賢いモデルに
    max_tokens=4000,
    tools=[opensearch_search],
    messages=[{"role": "user", "content":
        "直近1日のログを、種類別の件数で集計して、結果を日本語で要約して。"}],
)

for message in runner:
    for block in message.content:
        if block.type == "text":
            print("[Claude]", block.text)
        elif block.type == "tool_use":
            print("[クエリ]", block.input)
```

AIは**自分で集計クエリを組み立てて**実行し、件数を読んで要約します。4章では引数は `city` だけでしたが、今回はAIが**クエリそのものを設計**しています。

ただ、今は「1つ質問して1つ調べて答える」だけ。次章で、**自分で仮説を立てて何度も深掘りする**ところまで引き上げます。鍵は「システムプロンプト」です。

## 6. "1問1答"から"自分で深掘りする"へ（システムプロンプト）

今はまだ、こちらが**具体的な質問を1つ**渡しています。本物のアナリストは、「この1日分を調べて」とざっくり頼まれたら、自分で仮説を立てて何度も深掘りし、報告します。この差を生むのが **システムプロンプト** です。

> **用語：システムプロンプト**  
> LLMに「あなたは何者で、どう振る舞うか」を伝える指示文。その場の質問（messages）とは別に、**役割・ルール・手順**を定義します。

```
SYSTEM = """あなたはセキュリティのアナリストです。ログから気になる攻撃を能動的に探します。

- opensearch_search(query): ログを検索する。集計（件数を数える等）に使う。

1. まず全体像を掴む（種類別の件数、件数の多い送信元など）
2. 異常や外れ値、気になる送信元が見つかったら、追加の検索で深掘りする
3. 最後に、見つけた脅威と推奨アクションを日本語でまとめる

1回で終わらせず、複数回検索して、能動的に深掘りすること。
（フィールド名の決まりなど、環境固有のルールがあれば、ここに一緒に書いておく）"""
```

これを `tool_runner` に渡し、質問を**ざっくりした任務**に変えます。

```
runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=8000,
    system=SYSTEM,                            # ← 追加
    tools=[opensearch_search],
    messages=[{"role": "user", "content":
        "直近1日のログを調べて、気になる攻撃やパターンを深掘りして、まとめて。"}],
)
```

実行すると、出力はこんな流れになります（抜粋）。

```
[Claude] まず全体像を掴むため、種類別の件数を集計します。
[クエリ] {...種類別の集計...}
[Claude] 突出した送信元 47.238.88.184（8915件）が目立ちます。これを深掘りします。
[クエリ] {...その送信元の内訳...}
[クエリ] {...深刻度の高いアラート...}
[Claude] 重要な発見がありました。ログイン試行のパターンも確認します。
[クエリ] {...ログイン試行の内訳...}
[Claude] 調査結果をまとめます。…
```

ここが、私が伝えたかったところです。**ざっくりした任務1つから、AIが自分で「全体像→気になる送信元を深掘り→深刻度を確認→…」と何度も検索を重ねています。** 誰も「次はこれを調べろ」と言っていません。AIが自分で仮説を立て、次の一手を決めているのです。

これが"自律"の正体です。特別な仕組みはありません。**「LLM＋道具＋ループ」に、システムプロンプトで"どう動くか"を与えただけ。** ループが①〜④を何周も回し、LLMが毎周「次に何を調べるか」を判断している——それだけです。

## 7. 発見を「文章」から「データ」にする（道具をもう1つ増やす）

AIの要約は読み物としては十分ですが、**保存したり他のシステムに渡したりするには文章は扱いにくい**。そこで結果を**構造化データ**でも残します。やり方は——**道具をもう1つ増やすだけ。**

```
findings = []                                # 発見を貯めるリスト（トップレベルに置く）

@beta_tool
def record_finding(severity: str, title: str, evidence: str, recommendation: str) -> str:
    """確認できた脅威を1件記録する。調べて裏が取れたものだけ記録すること。

    Args:
        severity: 深刻度（CRITICAL / HIGH / MEDIUM のいずれか）
        title: 脅威の簡潔な名前
        evidence: 根拠となる具体的な事実（送信元IP・件数など）
        recommendation: 推奨アクション
    """
    findings.append({
        "severity": severity, "title": title,
        "evidence": evidence, "recommendation": recommendation,
    })
    return f"記録しました（現在 {len(findings)}件）"
```

あとは、(1) この道具を渡し、(2) システムプロンプトに「記録してからまとめる」と1行足し、(3) 最後に `findings` を表示するだけ。

```
tools=[opensearch_search, record_finding],

import json
print(json.dumps(findings, ensure_ascii=False, indent=2))
```

AIは調査中に `record_finding` を呼び、最後に `findings` がこう出力されます。

```
[
  {
    "severity": "HIGH",
    "title": "単一の送信元による大量スキャン",
    "evidence": "47.238.88.184 が1日で8915件のアラートを発生。全て非標準のSSHポートへのスキャン。",
    "recommendation": "当該IPをブロックし、非標準ポートへのスキャンを監視する。"
  },
  {
    "severity": "CRITICAL",
    "title": "侵入後のコマンド実行",
    "evidence": "ログイン成功後、設定改ざんと偵察コマンドが連続実行されていた。",
    "recommendation": "パスワード認証を無効化し、鍵認証へ。該当IP群をブロック。"
  }
]
```

ここで2つのことが起きています。**1つは、道具を2つ持ち、AIが「調べる道具」と「記録する道具」を使い分けるようになったこと**（4章の「道具を増やせば使い分ける」が本当になった）。**もう1つは、出力が文章から"データ"になったこと。** 文章を返すだけなら"チャット"ですが、構造化データを出せると\*\*他の仕組みに組み込める"部品"**になります。AIエージェントが"賢いチャット相手"から**"システムの一部"\*\*に変わったのです。

## 8. 毎日回せるツールに仕上げる

最後に、仕上げを3つ足します。

1. **結果をファイルに保存**（ターミナルに流れて消えないように）
2. **期間を指定**（`--days` で直近N日を選べるように）
3. **繰り返し回数に上限**（万一ループが止まらないときのため）

```
import argparse, datetime, json

parser = argparse.ArgumentParser()
parser.add_argument("--days", type=int, default=1, help="対象日数（直近N日）")
args = parser.parse_args()

final_text = ""
for i, message in enumerate(runner):
    if i > 40:                                       # 念のため上限：40ターンで止める
        break
    msg_text = "".join(b.text for b in message.content if b.type == "text")
    if msg_text.strip():
        final_text = msg_text                        # 最後の文章＝最終レポート
    for block in message.content:
        if block.type == "tool_use":
            print("[道具]", block.name)

date = datetime.date.today().isoformat()
with open(f"report_{date}.md", "w", encoding="utf-8") as f:
    f.write(final_text)
with open(f"findings_{date}.json", "w", encoding="utf-8") as f:
    json.dump(findings, f, ensure_ascii=False, indent=2)
print(f"保存しました: report_{date}.md / findings_{date}.json")
```

```
python hunt_agent.py --days 1
```

実行すると、AIが調査を進め、文章のレポート（`report_YYYY-MM-DD.md`）と構造化データ（`findings_YYYY-MM-DD.json`）が残ります。`--days 7` なら1週間分。ターミナルに消える出力から、**毎日回せて結果が残るツール**になりました。

## 9. それでも、AIを過信してはいけない

ここまで読むと「全部任せられる」と思うかもしれませんが、はっきり書いておきます。**AIは強力ですが、間違えることもあります。** 私の経験でも、集計の数字を読み違えたり、確証のないことを断定的に書いたりしました（逆に「うまく集計できませんでした」と正直に限界を認めることもあり、これは良い挙動でした）。

特に気をつけたいのは、AIが**自信たっぷりに間違った結論を書く**こと。文章が整っていると正しく見えてしまいます。だからルールはシンプルです。

> **AIの出力は必ず元のデータで確かめる。最終判断は人間が持つ。**

「この送信元が危険です」と言われたら根拠を自分の目で確認する。推奨アクションをそのまま自動実行しない。AIは手間を肩代わりしてくれる**優秀なアシスタント**であって、最終決定者ではありません。「AIに何をやらせ、どこを人間が確かめるか」を設計できることが、AIを使いこなす側に回るということです。

（この後の回で、AIの出力の確かめ方と人間のレビューの組み込み方を具体的に扱います。）

## 10. まとめ：あなたはもう、エージェントを作れる

私たちは、フレームワークを使わず、SDKだけで自律AIエージェントを作りました。

* 正体は「LLM＋道具＋ループ」だけ（1章）／①〜④を手作業で1周回して仕組みを見た（3章）
* `@beta_tool` と `tool_runner` で自動化（4章）／偽の道具を本物のデータにつないだ（5章）
* システムプロンプトで役割を与えたら自律的に深掘りした（6章）／発見を構造化データにした（7章）
* 保存・期間指定・上限を足して毎日回せるツールに（8章）／そして、AIを過信せず人間が最終判断を持つことも確認した（9章）

大事なのは、**もうフレームワークがブラックボックスではない**ことです。"エージェント"の中身は全部、この記事で手を動かしたものでした。仕組みがわかれば、ライブラリを使うのも自分で書くのも自由です。

私自身、この記事を書く前はエージェントを作ったことがありませんでした。それでも手を動かせば、ちゃんと動くものができた。だから、あなたにもきっと作れます。
