シンプレクスweb3チームは、web3のマスアダプションに向けて、[Simplex Fourth](https://www.simplex.inc/solution/fourth/)をはじめとしたソリューションの開発を行うほか、最先端のテクノロジーを活用し、新たなユースケースの創出に取り組むチームです。  
  
Simplex テックブログのEdge of Cryptoシリーズでは、その最前線で活躍するweb3チームメンバーが、最先端の業界動向やテクノロジーについての情報をお届けします。前回のEdge of Cryptoは、[「AIエージェントが自律決済する世界でのウォレット」](https://zenn.dev/simplex/articles/68ae40fd345d18)です。

## はじめに

前回の記事では、AIエージェントが自律的に支払いを行う世界において、ウォレット側で支払い可否を制御する必要性について整理しました。

本記事では、AWS WAFのAI traffic monetization機能を利用して、x402の一連のフローを体験するとともに、作成されるデータやトランザクションを確認していきます。

<https://aws.amazon.com/jp/about-aws/whats-new/2026/06/aws-waf-ai-traffic-monetization/>

## 検証するx402のフロー

x402は、HTTPステータス 402（Payment Required）を用いて支払い条件を提示し、クライアントが支払い認可の署名を付与して再リクエストすることで、有料リソースへのアクセスを可能にする決済プロトコルです。

<https://docs.x402.org/introduction>

今回確認する流れは以下の通りです。

1. AWS WAFによるAIトラフィック収益化設定を行ったコンテンツに対し、AIエージェントを模したリクエストを送信します。
2. AWS WAFがHTTP 402と支払い条件を返却します。
3. 支払い条件をもとに、支払い認可・署名を作成します。
4. 署名をリクエストヘッダに付与して再度アクセスします。
5. オンチェーンの決済トランザクションが確定し、HTTP 200とともにコンテンツが返却されます。

現在、AWS WAFのAIトラフィック収益化機能では、BaseチェーンまたはSolanaチェーン、およびこれらのテストネットでのUSDC決済に対応しています。今回は、BaseチェーンのテストネットであるBase Sepoliaで検証を行います。

## ウォレット準備

支払いを行う側（AIエージェント側）と、支払い先サービス側の2つのBase Sepoliaウォレットを用意します。前者のウォレットにはx402での支払いに足りるだけのUSDC残高を補充しておきます（後者は不要）。

今回用意したアドレスは、それぞれ以下の通りです。

* AIエージェント側：`0x7c236e45C8f1f4FfE05a038E8f7BA686478bCF8c`
* サービス側：`0xaba784F15B7e228C8335D15e66B200e8b05d5718`

## AWS WAFのAIトラフィック収益化設定

AWS WAFのAIトラフィック収益化はBot Controlのルールとして用意されているため、Amazon CloudFrontディストリビューションに関連付けられたWeb ACLでBot Controlを有効化します。続いて、ルールオーバーライドにて、CategoryAIの設定値を「Monetize」に変更すると、AIトラフィックが収益化対象となります。

具体的な収益化設定（決済を行うネットワーク、支払い先ウォレットアドレス、リクエストあたりの価格）は、Web ACLのMonetizationConfigから行います。テストネットであるBase Sepoliaを選択し、用意した支払い先ウォレットアドレスを指定します。金額は0.001 USDCから指定可能です。

![AWS WAF収益化設定](https://static.zenn.studio/user-upload/080ae190a714-20260709.png)

以上でAWS WAF側の設定は完了です。

収益化設定のより詳細な情報は、AWS公式ブログおよびデベロッパーガイドをご参照ください。

## HTTP 402と支払い条件を確認する

AIエージェントを装ったリクエストを行い、402が返却されることを確認します。

今回の検証では、Bot ControlにAI関連のbotとして分類されやすいUser-Agentを指定し、Monetizeルールが発火することを確認します。ここでは、ChatGPT経由であることを示す`ChatGPT-User`を指定して、リクエストを送信しました。

![x402_HTTP402](https://static.zenn.studio/user-upload/bdc8da39292c-20260709.png)

実際に402が返ってくることを確認できました。

支払い条件は、レスポンスヘッダの`payment-required`にBase64エンコードされた状態で送信されます。デコードすると以下のようなJSONが確認できます。

```
{
  "accepts": [
    {
      "amount": "1000",
      "asset": "0x036CbD53842c5426634e7929541eC2318f3dCF7e",
      "extra": { "name": "USDC", "version": "2" },
      "maxTimeoutSeconds": 300,
      "network": "eip155:84532",
      "payTo": "0xaba784F15B7e228C8335D15e66B200e8b05d5718",
      "scheme": "exact"
    }
  ],
  "error": "PAYMENT-SIGNATURE header is required",
  "extensions": { "payment-identifier": { "info": { "required": false } } },
  "resource": { "url": "https://<サービスのドメイン>/" },
  "x402Version": 2
}
```

各項目は以下を表しており、AWS WAFで設定した通りの支払い設定が入っていることが確認できます。

* amount: 金額。最小単位で表した整数値であり、USDCの単位は小数6桁であるため、1000 = 0.001 USDCを表す。
* asset: 支払資産（ここではUSDC）のコントラクトアドレス。
* maxTimeoutSeconds: 支払い署名の有効期限（秒）。
* network: 支払いを行うネットワーク（`eip155:84532`はBase Sepolia）。
* payTo: 支払先ウォレットアドレス。
* scheme: 支払方式。`exact`は、`amount`の金額ちょうどで1回の決済を行う方式を表す。

## 支払い認可・署名を作成する

AIエージェント側は受け取った支払条件をもとに、自身のウォレットを用いてPaymentPayloadと呼ばれる支払い認可・署名オブジェクトを作成します。

この際、「何の形式で署名するか」を判断する必要があります。ここでは詳細については省略しますが、x402では支払い条件の`network`, `scheme`, `extra`などから署名形式を決定できるようになっています。

今回作成されたオブジェクトは以下の通りです。

```
{
  "payload": {
    "authorization": {
      "from": "0x7c236e45C8f1f4FfE05a038E8f7BA686478bCF8c",
      "to": "0xaba784F15B7e228C8335D15e66B200e8b05d5718",
      "value": "1000",
      "validAfter": "1782744032",
      "validBefore": "1782744332",
      "nonce": "0xb136adec474b496af0c06172868781a9c66b8e895e783ee7ad591bdeb873aa33"
    },
    "signature": "0xfaf6f7a7b7c2edf3e039de108d7b339ba65856a6d03ec7b2acca6f5762b180cb466ebea89c9f62aa12539076771962e7b6e3e36f1018749d3e296a70233e60931b"
  }
}
```

今回のPaymentPayloadは、EIP-3009というイーサリアムの拡張規格に沿った署名形式となっています。EIP-3009とは、USDCなどのERC-20トークンに対し、オフチェーン署名での送金認可と第三者による送金実行を可能にするための規格です。

通常のERC-20トークン送金では、トークンの保有者自身が署名とトランザクション送信を行います。一方EIP-3009では、ユーザは「fromアドレスからtoアドレスに、この金額を、この期限内に送ってよい」という認可メッセージに署名し、署名を受け取った第三者が実際の送金をオンチェーンで実行できるようにしています。

このように署名者とトランザクション送信者を分けることで、署名者はガス代負担が不要となります。今回のケースでは、AWS WAFが利用するCoinbase x402 Facilitatorが送金のガス代を肩代わりしており、AIエージェント側ではガス代のETHを用意しなくとも支払いが可能であるため、UX向上に寄与しています。

## 署名付きで再リクエストする

AIエージェントは、`PAYMENT-SIGNATURE`ヘッダにBase64エンコードしたPaymentPayloadを含めて、再度サーバにリクエストを送信します。HTTP 200とともにコンテンツが返却されることを確認できました。

![x402_HTTP200](https://static.zenn.studio/user-upload/5e8782560161-20260709.png)

このとき、リクエストからレスポンスまでに3〜4秒程度のラグがありました。オンチェーンでのトランザクション確定後にレスポンスが返却されるためだと考えられます。

なお、PaymentPayloadの`authorization`は使い捨ての値である`nonce`を含むため、支払いのトランザクションが確定した後に同一の署名を用いた支払いは不可能となります。実際、決済完了後に同一ヘッダで再度サーバにリクエストすると、今度は402が返却されました。

![x402_replay_HTTP402](https://static.zenn.studio/user-upload/cd9b582010aa-20260709.png)

## 作成されたトランザクションを確認する

Base Sepoliaのエクスプローラーから、支払いトランザクションを確認してみます。

![x402_transaction](https://static.zenn.studio/user-upload/f6f708237979-20260709.png)

ERC-20トークンの移転を見ると、From / Toはそれぞれ支払い側ウォレット / サービス提供者が設定した受取ウォレットであり、0.001 USDCが正しく移転したことが確認できます。

一方で、トランザクション自体のFromを見ると、AIエージェントでも支払い先サービスでもない別のアドレスが入っています。これは、支払い認可を受け取ったFacilitatorが、オンチェーン決済を実行しているためです。ガス代のETH支払いはFacilitatorであるCoinbaseにより行われており、AIエージェントや支払先サービス側のウォレットではETHを用意しなくても良いことが分かります。

今回作成されたトランザクションは、[こちら](https://sepolia.basescan.org/tx/0x0a6d8fb853dba99c3dd1abdc3fc1320c4a0a6dcc67c3bcc3ba7e69eb30121c38)のリンクから確認可能です。

## 今後の展望

今回x402での支払いフローを試すにあたり、AIエージェント用のウォレットを用意し、支払いに使うUSDC残高をあらかじめ補充しました。しかし、AIエージェントを用いてWeb上のデータにアクセスしたいだけの一般ユーザにとっては、ウォレット準備やステーブルコイン残高管理を行うのは負担になります。

このUXを考慮すると、一般ユーザ向けのAIサービスでは、ユーザがウォレットやステーブルコインの管理を意識せずAIを利用できるように整備されていくのではないかと思います。支払い機能はAIエージェントサービス側に組み込まれ、AIエージェントの支払い分は月額利用料などの形で最終的にユーザが負担する形になるのではないでしょうか。

一方、企業のAI利用においては、支払いや残高管理が隠蔽されることが望ましくないケースも存在します。たとえば、AIによる資金利用を管理・監査する必要がある企業は、AIが利用するウォレットに適切な権限を定め、ポリシー通りに運用されていることを確認できなければいけません。こうしたケースでは、別途AIエージェント用のウォレットサービスを利用して管理すべきだと考えられます。

今後AIによる自律決済が普及するにつれ、AIエージェント周辺のサービス設計やビジネスモデルも変化していくことが想定されます。その中で、ブロックチェーン技術がどんな役割を果たせるか、引き続き注目していきたいと考えています。
