AI駆動開発スターターキットを、また少し改善しました。

今回のテーマは、かなり地味です。

でも、実務で使うならかなり大事です。

テーマは、

**長いプロンプトを、どこまで短くできるか**

です。

前回までで、cli\_text\_counter を題材に、AIに overview.md と 01\_spec.md の初期配置から、設計、実装、テスト、レビューまで進めさせました。

結果として、かなりうまくいきました。

ただし、そこで気になったことがあります。

**毎回投げていたプロンプトが長い。**

うまくいったのはよい。  
でも、そのために毎回長い指示を書かないといけないなら、スターターキットとしては少し弱い。

スターターキットにしたいなら、他の人も同じような体験をできる必要があります。

そのためには、

```
標準プロンプトに書くこと
毎回チャットで渡すこと
人間が判断すること
```

を分ける必要があると思いました。

## 長いプロンプトでうまくいっても、運用としては弱い

cli\_text\_counter の検証では、かなり丁寧にプロンプトを書いていました。

たとえば、

このあたりを、かなり細かく毎回書いていました。

それはそれで、うまくいきます。

でも、それを毎回人間が書くなら、スターターキットとしてはつらい。

本来は、

```
prompts/implement_feature.md を参照してください。

対象機能フォルダ: ...
コマンド/アプリ名: ...
対象機能名: ...
実装ファイル: ...
テストファイル: ...
作りたいもの: ...
補足条件: ...
```

くらいで進めたいわけです。

毎回、禁止事項や責務分担や tasks.md の扱いまで全部書くのは、しんどいです。

## 標準プロンプトと毎回の指示を分ける

そこで、まず整理メモを作りました。

作ったのは、

```
docs/prompt_design_notes.md
```

です。

ここで整理したのは、主に3つです。

```
1. 標準 prompts/*.md に入れるべき共通ルール
2. 毎回チャットで渡すべき作業固有情報
3. 人間が判断すべき情報
```

たとえば、標準プロンプトに入れるべきものは、毎回変わらないルールです。

```
指定外ファイルを変更しない
src/common を勝手に触らない
feature / entrypoint / 結合試験の責務を分ける
レビュー結果は専用ファイルに書く
tasks.md には詳細を書かない
```

これは、どの command/app でも変わりません。

一方で、毎回チャットで渡すべきものは、対象ごとに変わる情報です。

```
command/app 名
feature 名
実装ファイル
テストファイル
作りたいもの
入力
出力
エラー扱い
今回やらないこと
```

これは標準プロンプトに固定してはいけません。

さらに、人間が判断すべきものも分けました。

```
空文字列をどう扱うか
文字数をどう数えるか
fallback import を標準方針にするか
skipped を成功扱いしてよいか
既存ドキュメントと実装状態がズレたとき、どちらを正とするか
```

このあたりは、AIに任せるとそれっぽく進めてしまうところです。

でも、実務ではここを人間が決めないと危ない。

## how\_to\_use\_prompts.md に短い指示の書き方を追加した

次に、docs/how\_to\_use\_prompts.md を更新しました。

ここには、汎用プロンプトの使い方を書いています。

今回追加したのは、**短いチャット指示の書き方**です。

考え方はシンプルです。

標準 prompts/\*.md には、すでに共通ルールが入っている。

だから、チャット側では毎回すべてのルールを書き直さず、今回固有の情報を渡す。

たとえば、feature 実装ならこんな感じです。

```
prompts/implement_feature.md を参照してください。

対象機能フォルダ: docs/<command_or_app_name>/features/<feature_name>/
コマンド/アプリ名: <command_or_app_name>
対象機能名: <feature_name>
実装ファイル: src/<command_or_app_name>/features/<feature_name>.py
テストファイル: tests/<command_or_app_name>/features/test_<feature_name>.py
作りたいもの: <今回作りたい機能>
補足条件: なし
```

迷いやすい仕様がある場合だけ、補足条件に短く追加する。

```
補足条件:
- 空文字列は有効な入力として扱ってください。
- 文字数は Python の len() 相当で数えてください。
- レビューだけ行い、実装ファイルとテストファイルは変更しないでください。
```

このくらいなら、実際に使えそうです。

## チュートリアルにも反映した

次に、docs/tutorials/010\_simple\_calculator.md も更新しました。

このチュートリアルは、cli\_simple\_calculator を題材に、AI駆動開発スターターキットの流れを小さく体験するものです。

ここでも、短いチャット指示の考え方を反映しました。

冒頭に、

```
標準 prompts/*.md には共通ルールが入っている
チャットでは、参照するプロンプト、対象情報、補足条件を渡す
```

という説明を入れました。

また、最初のサンプル確認ステップも修正しました。

以前は、

```
1. サンプル機能を確認する
```

という見出しでした。

でも、これだと先読みする人が、

「AIに完成済みサンプルを読ませて、次の実装に流用させるのかな？」

と思いそうでした。

本来の意図は違います。

cli\_hello\_greeting は、AIに学習させるためのデータではありません。

利用者が、docs / src / tests の対応関係を理解するための完成済みサンプルです。

なので見出しを、

```
1. 完成済みサンプルで構成を確認する（任意・初回向け）
```

に変えました。

さらに、

```
これはAIにサンプルを学習させて次の実装へ流用させるためではなく、
利用者がスターターキットの構成を理解するための任意ステップです。
```

と明記しました。

すでに構成を理解している場合や、2回目以降は、ステップ2から始めてよい。

この整理は大事だと思います。

## 実際にチュートリアルを通してみた

ここまで整えたので、実際に 010\_simple\_calculator.md の流れで試してみました。

目的は、短いチャット指示で本当に同じ体験を再現できるかを見ることです。

まず、完成済みサンプル cli\_hello\_greeting の確認。

ここはうまくいきました。

AIは、overview.md、tasks.md、01\_spec.md、02\_design.md、03\_flow.md、04\_test\_plan.md、src/、tests/ の対応関係を説明してくれました。

この出力は、まさに初回利用者向けの構成説明になっていました。

つまり、

**AIに学習させるためではなく、人間が理解するための説明をAIに出してもらう**

という意図どおりです。

## 02\_design.md と 03\_flow.md はうまく進んだ

次に、cli\_simple\_calculator の関数設計を作成しました。

短い指示で、

```
prompts/create_function_design.md を参照してください。
```

と渡し、対象情報と補足条件だけ指定しました。

結果、02\_design.md が作成されました。

内容は、add\_numbers(a: int, b: int) -> int を中心にした関数設計です。

entrypoint.py はCLI引数解析・呼び出し・標準出力だけを担当し、足し算ロジックは features/calculator.py に置く。

この責務分担も守れていました。

次の 03\_flow.md も、短い指示で作成できました。

entrypoint.py から features/calculator.py の add\_numbers() を呼び出す流れで、parse\_args、main、add\_numbers の責務も整理できました。

ここまでは、かなり順調です。

## 短くしすぎると、境界が少し広がる

問題が出たのは、04\_test\_plan.md です。

最初のチュートリアル文では、補足条件をこうしていました。

```
補足条件: tests/cli_simple_calculator/ 配下に置く前提で整理してください。
```

すると、AIは tests/cli\_simple\_calculator/ 全体を見たのか、feature単体テストだけでなく、parse\_args や main の entrypoint 側の観点まで含めました。

これは完全なミスではありません。

でも、今回の方針とは少しズレています。

04\_test\_plan.md は feature 単体テストの計画です。

entrypoint の引数解析、標準出力、終了コードは、後続の entrypoint テストで扱うべきです。

つまり、短くした結果、境界が少し広がりました。

ここは大事な気づきです。

**短いプロンプトは良い。  
でも、責務境界がぶれやすい箇所だけは補足条件で明示する必要がある。**

そこで、チュートリアルの補足条件を修正しました。

```
補足条件: feature 単体テストは tests/cli_simple_calculator/features/test_calculator.py に置く前提で整理してください。entrypoint の引数解析、標準出力、終了コードは後続の entrypoint テストで扱ってください。
```

少し長くなりました。

でも、この程度なら許容範囲です。

何でも短くすればいいわけではない。

短くしてよいところと、明示すべきところがある。

## feature 実装はきれいに進んだ

次に、feature本体とfeature単体テストを作りました。

ここでは補足条件に、

```
feature 本体と feature 単体テストだけを作成してください。
entrypoint.py、entrypoint テスト、結合試験は作成しないでください。
```

という趣旨を追加しました。

結果は良好でした。

実装は、

```
def add_numbers(a: int, b: int) -> int:
    return a + b
```

だけです。

余計な入力チェックも、共通化もありません。

テストも、

を確認し、5 passed でした。

ここは、かなりスターターキットらしい流れになりました。

## tasks.md 更新が漏れた

ここで別の問題が出ました。

feature本体とテストは作れました。

でも、tasks.md が更新されませんでした。

最初は、

「あれ、tasks.md 更新ルールを prompts に入れたはずなのに」

と思いました。

でも、よく見ると理由がありました。

prompts/implement\_feature.md などには、作業後の tasks.md 更新ルールがあります。

一方で、プロンプトの変更許可では、

```
変更してよいファイルは実装ファイルとテストファイル
```

のように読める箇所がありました。

するとAIは安全側に倒します。

```
docs/ は変更禁止
tasks.md は docs 配下
だから tasks.md は更新しない
```

という判断です。

これはAIがサボったというより、**プロンプト側の矛盾**です。

ここはかなり大きな発見でした。

## entrypoint と結合試験でも同じ問題が出た

その後、entrypoint と entrypoint テストを作成しました。

実装自体は良かったです。

argparse で --a と --b を受け取り、add\_numbers を呼び、結果を標準出力に出して 0 を返す。

テストも、引数解析、正常系、引数不足、不正な整数値まで確認できていました。

しかし、また tasks.md は更新されませんでした。

理由は同じです。

```
docs は変更禁止
tasks.md も docs 配下
だから更新しない
```

さらに、結合試験計画でも同じことが起きました。

10\_integration\_test\_plan.md は作成できた。  
でも tasks.md は更新されない。

結合試験実装でも同じでした。

test\_integration\_calculator.py は作成できた。  
テストも 15 passed。  
でも tasks.md は更新されない。

ここで、完全に分かりました。

**tasks.md 更新ルールを入れるだけでは足りない。  
変更してよいファイルにも tasks.md を入れないと、AIは更新しない。**

これは実務的にはとても自然です。

AIに「指定外ファイルを変更するな」と強く言っているのだから、tasks.md が指定外に見えたら更新しない。

むしろ、安全側に倒れていて良いとも言えます。

だから、プロンプト側の書き方を直す必要がありました。

## prompts/\*.md を修正した

そこで、対象の prompts/\*.md を修正しました。

対象は12ファイルです。

```
prompts/create_overview.md
prompts/create_feature_spec.md
prompts/create_function_design.md
prompts/create_function_call_flow.md
prompts/create_test_design.md
prompts/create_review_checklist.md
prompts/create_integration_test_plan.md
prompts/implement_feature.md
prompts/implement_entrypoint.md
prompts/implement_integration_test.md
prompts/review_feature.md
prompts/review_command.md
```

修正したのは、主にここです。

```
変更してよいファイル:
- このプロンプトで指定された成果物
- 対象 command/app または feature の tasks.md
  - ただし、現在地と次に確認することを短く更新する場合に限る
```

または、

```
docs/ は原則として指定された成果物以外は変更しないでください。
ただし、対象 command/app または feature の tasks.md は、
作業後の現在地メモとして必要最小限の更新を行ってよいものとします。
```

これで、

```
成果物 + 対象 tasks.md は更新可
ただし tasks.md は現在地と次の一手だけ
```

というルールになりました。

この修正はかなり効くと思います。

## tasks\_template.md にも反映した

さらに、docs/templates/tasks\_template.md も更新しました。

なぜなら、prompts/\*.md で tasks.md の更新を許可しても、テンプレート側に何を書くべきかが弱いと、またブレるからです。

tasks\_template.md では、tasks.md は現在地メモであることを強めました。

書いてよいものは、

```
現在の状態
作業メモ
次に確認すること
引き継ぎに必要な短い注意点
```

です。

書かないものは、

```
仕様の詳細
設計の詳細
テスト計画の詳細
レビュー結果の詳細
長いテストログ
```

です。

この整理はかなり重要です。

tasks.md は便利なので、放っておくと何でも書かれます。

でも、それをやると、仕様書やレビュー結果ファイルとの役割が混ざります。

だから、テンプレート側でも役割を明確にしました。

## チュートリアルにも tasks.md を明記した

最後に、010\_simple\_calculator.md も更新しました。

各ステップの「作成または更新されるファイル」に、必要に応じて更新される tasks.md を追記しました。

たとえば、関数設計なら、

```
docs/cli_simple_calculator/features/calculator/02_design.md
docs/cli_simple_calculator/tasks.md
docs/cli_simple_calculator/features/calculator/tasks.md
```

のようにしました。

feature単体レビューでは、

```
docs/cli_simple_calculator/features/calculator/06_review_result.md
docs/cli_simple_calculator/features/calculator/tasks.md
```

command/app全体レビューでは、

```
docs/cli_simple_calculator/11_command_review_result.md
docs/cli_simple_calculator/tasks.md
```

です。

これで、利用者から見ても、tasks.md が更新対象に含まれることが分かります。

ただし、あくまで現在地メモとして必要最小限です。

tasks.md は成果物の詳細を書くファイルではありません。

## 最後までチュートリアルは通った

結果として、cli\_simple\_calculator のチュートリアルは最後まで通りました。

feature 実装、entrypoint 実装、結合試験まで進み、テストは最終的に、

```
15 passed
```

になりました。

最後の command/app 全体レビューでは、判定は **軽微な指摘あり** でした。

主な指摘は、

```
overview.md に古い記述が残っている
__init__.py の扱いが AGENTS.md の構成例とズレている
```

です。

overview.md の古い記述は修正しました。

一方、\_\_init\_\_.py の扱いは少し大きめの方針判断なので、今回は保留しました。

ここも実務っぽいです。

レビューで出た指摘を、全部その場で直すわけではありません。

今直すべきものと、方針判断として残すものを分ける。

これもAI駆動開発では大事だと思います。

## 短いプロンプトで同じ体験は再現できたか

今回の結論としては、

**おおむね再現できた。  
ただし、補足条件と標準プロンプトの整備が必要。**

です。

短いチャット指示だけでも、かなり進められました。

ただし、いくつかの課題が見えました。

```
テスト計画では、補足条件が弱いと entrypoint 側まで範囲が広がった
tasks.md 更新ルールは、変更許可リストに入っていないと効かなかった
チュートリアル側にも tasks.md が更新対象だと明示する必要があった
```

これは、かなり大きな収穫です。

つまり、

**長いプロンプトを短くするには、ただ削ればいいわけではない。**

共通ルールは標準プロンプトへ寄せる。  
作業固有情報はチャットで渡す。  
人間判断が必要な仕様は補足条件に書く。  
境界がぶれやすいところは短くても明示する。  
tasks.md のような例外更新は、変更許可にも明記する。

この整理が必要でした。

## 今回わかったこと

今回の検証で、かなり実務的なことが分かりました。

1つ目。

**AIは短い指示でも進められる。**

ただし、標準プロンプトがちゃんとしていることが前提です。

2つ目。

**短い指示にすると、曖昧なところが見える。**

今回なら、テスト計画で entrypoint 側まで広がりました。

3つ目。

**AIは指定外変更を避ける。**

これは良いことです。

ただし、その結果、tasks.md 更新のような必要な作業も止まることがあります。

4つ目。

**標準プロンプトは、実際にチュートリアルを回さないと矛盾に気づきにくい。**

机上で整えているだけだと、tasks.md 更新ルールと変更許可リストの矛盾には気づきにくいです。

実際に通して初めて分かりました。

5つ目。

**tasks.md は、prompts / template / tutorial の3点で支える必要がある。**

```
prompts/*.md
  → tasks.md 更新を許可・指示する

tasks_template.md
  → tasks.md に何を書く/書かないを定義する

010_simple_calculator.md
  → チュートリアル上でも tasks.md が更新対象だと示す
```

ここまで揃って、ようやく運用に乗る感じです。

## まとめ

今回は、AI駆動開発スターターキットで、長いプロンプトをどこまで短くできるかを試しました。

やったことは、かなり地味です。

```
prompt_design_notes.md を作った
how_to_use_prompts.md に短いチャット指示の書き方を追加した
010_simple_calculator.md を短い指示前提に更新した
実際にチュートリアルを通した
tasks.md 更新漏れを発見した
prompts/*.md の変更許可に tasks.md を追加した
tasks_template.md にも運用ルールを反映した
010_simple_calculator.md の更新ファイル一覧にも tasks.md を追加した
```

地味ですが、かなり重要です。

スターターキットは、1回うまくいけばよいものではありません。

他の人が、同じような体験をできることが大事です。

そのためには、

**長いプロンプトで成功した内容を、標準プロンプトと短い入力に分解する**

必要があります。

今回、それが少し見えてきました。

AI駆動開発は、AIに長文指示を投げる力だけでは続きません。

共通ルールをどこに置くか。  
毎回何を渡すか。  
人間がどこで判断するか。  
状態をどこに残すか。

このあたりを整理していくことが、実務で使うためにはかなり大事だと思います。

次は、残課題として見えている \_\_init\_\_.py や fallback import の扱いをどうするか、少し整理していきたいと思います。
