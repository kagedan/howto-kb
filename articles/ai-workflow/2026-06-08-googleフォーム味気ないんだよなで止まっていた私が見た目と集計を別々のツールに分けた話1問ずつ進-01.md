社内アンケートを取ることになった。

AI活用の研修をやるための、事前アンケート。 全員の「今どんな業務に困っているか」を集めたい。

Googleフォームを開いて、設問を3つ打ったところで手が止まった。

味気ない。 あの白い背景に、ラジオボタンが縦に並ぶやつ。回答する側のテンションが、最初の1問で下がるのが目に見えた。

かといって、自前でアンケートサイトを作るのは面倒だ。 回答をどこに溜めるか。データベースを立てて、サーバーを用意して、集計の仕組みを書いて。そこまでやる体力はない。

見た目を取ると集計が面倒。集計を取ると見た目が死ぬ。 私はずっと、この二択で止まっていた。

## 二択じゃなかった。両方、別々のツールに任せればいい

Claudeに相談して、構成がひっくり返った。

見た目は、Claudeに作り込んだHTMLを1枚書いてもらう。1問ずつ進む、ブランドカラーの効いた画面。 集計は、Googleフォームとスプレッドシートに丸投げする。

回答者には、洗練されたHTML画面だけが見える。 その裏で、見えないフォームがGoogleに回答をPOSTしている。回答者はGoogleフォームの存在すら知らない。ログインも不要、匿名で、回答はスプレッドシートに勝手に溜まっていく。

同じツールで両立させようとしていたのが間違いだった。 別々のツールに分けて、それぞれの得意なところだけ使う。割り切った瞬間に、二択が消えた。

## 「役割で分ける」というフレーム

あとから整理して、これは1つの型だと気づいた。

1. 体験の質（見た目・操作感）は、どのツールがいちばん作り込めるか
2. データの受け皿（集計・保存・連携）は、どのツールがいちばん手間がないか

この2つを同じツールで満たそうとすると、妥協が生まれる。 分けていいと気づくと、選択肢が広がる。今回なら体験＝Claude製HTML、受け皿＝Google。申込フォームでも診断でも投票でも、そのまま横に流用できる。

## 実装は、4つのパーツを繋ぐだけ

仕組みは難しくない。Googleフォームを「裏方の受け皿」にして、HTMLから値を送り込む。順に書きます。

ひとつ目。Googleフォームを手で作らず、スクリプト（GAS）で自動生成する。 会社アカウントのスクリプトエディタで、設問を定義して実行する。

```
const form = FormApp.create('社内AI活用アンケート');
form.addCheckboxItem().setTitle('普段の困りごと')
  .setChoiceValues(['メール返信', '議事録', '資料作成', 'その他']);
form.addParagraphTextItem().setTitle('困りごと（その他・自由記入）');
Logger.log('PREFILLED_URL: ' + form.createResponse()…toPrefilledUrl());
```

ふたつ目。実行ログに出るPREFILLED\_URLから、2つの情報を抜く。 送信先（…/formResponseで終わるURL）と、各設問に振られたentry.数字というID。このentryが、どの値をどの設問に入れるかの宛先になる。

みっつ目。HTML側の設定ブロックに、抜いたIDを配線する。

```
const FORM_CONFIG = {
  action: "https://docs.google.com/forms/d/e/XXXX/formResponse",
  entries: {
    q1: "entry.1234567",
    q1other: "entry.7654321"
  }
};
```

よっつ目。送信は、隠しiframeへのPOSTにする。 画面には出さないiframeを1つ作り、そこを宛先にしたフォームを動的に組んで送る。

```
const iframe = document.createElement("iframe");
iframe.name = "hidden_target";
iframe.style.display = "none";
document.body.appendChild(iframe);

const f = document.createElement("form");
f.action = FORM_CONFIG.action;
f.method = "POST";
f.target = iframe.name;   // 画面遷移させず裏で送る
// 各回答を input にして f に append → f.submit()
```

ここで一点だけ。fetchのno-corsで送るやり方は、ローカルファイルや社内共有だと詰まりやすい。 隠しiframeへのPOSTなら、その環境でも安定して届く。私はここで一度fetch版を試して動かず、iframe方式に切り替えた。

あとはHTMLをWebに公開すれば、URL1本で配れる。

## POSTは成功する。なのに、値が消える

テスト送信したら、回答は届いた。送信は成功している。 なのに、スプレッドシートを見たら、ある設問の値だけが空っぽだった。

その他の選択肢。HTML側では「その他（自由記入）」、Googleフォーム側では「その他」。 たった数文字の違い。これだけで、Googleはその設問の値を丸ごと捨てていた。

しかも厄介なのは、送信自体は成功扱い（200）で返ってくること。 エラーが出ないから、気づけない。

Googleフォームのラジオやチェックボックスは、選択肢の文字列が1文字でも違うと値を破棄する。setChoiceValuesで書いた文字列と、HTML側の表示ラベルを、完全に一致させないといけない。 これを知らずに本番配布していたら、私は欠けたデータで研修テーマを決めていた。ぞっとした。

## 配線を点検させるプロンプト

「完全一致」は、人間が目で追うと必ず見落とす。だから配線後に点検を挟む。

```
あなたは私のフォーム実装のレビュー担当です。

【入力】
・HTML側の全選択肢ラベル（設問ごとに箇条書きで貼る）
・Googleフォーム側の setChoiceValues の中身（同じく設問ごとに貼る）

【やってほしいこと】
1. 設問ごとに、HTML側とフォーム側のラベルを1対1で突き合わせる
2. 1文字でも違うペアを、設問番号つきで全部洗い出す
3. 「その他（自由記入）」のような表記ゆれを特に重点的に見る

【制約】
- 完全一致しているものは報告しなくていい。不一致だけ挙げる
- 直し方の提案ではなく、まず差分の検出だけする
```

人間の目は「だいたい合ってる」で流す。機械の目は「1文字違う」を見つける。 点検は、得意な方に任せればいい。

## 結果

このアンケートは、28件の回答が匿名で集まった。 「フォームが見やすかった」という声がいくつか届いた。裏でGoogleに飛んでいるとは、誰も気づいていない。

集めた28件をそのままClaudeに渡して、困りごとの傾向を分解して、研修のテーマ案まで作った。 アンケートを作る道具も、回答を分析する道具も、同じ相棒だった。

役割で分ける。得意なところだけ使う。 それだけで、ずっと諦めていたものが動き出した。
