# はじめに

本稿は「なぜ、『AIエージェント』はユーザーをバカにした不誠実なワードなのか？」という問いを扱います。ただし、これはAI業界に固有の不誠実さというより、IT業界が繰り返してきた**曖昧化**の典型例として「AIエージェント」を解剖するものです。クラウド、ゼロトラスト、サーバーレス、DXといった語と同じ系譜にありますが、その中でも特に境界操作が露骨に効いている語として扱います。

本来、エージェントとは、環境を観測し、目標に向けて行動を選択するシステムを指す言葉です。古典的な定義では、外部環境への作用を伴うことが要件とされる場合もあり、その意味では推論モデル単体を「エージェント」と呼ぶことに一定の留保はつきます。

しかし、その留保を認めたうえでもなお、現在の業界の線引きには商業的な歪みがあります。推論モデル内部の計画や検証は「モデル性能」と呼ばれ、外部にツール接続を足した瞬間に初めて「AIエージェント」と呼ばれます。この切り分けは、技術的な定義論だけでは説明しきれない**売り手都合**を含んでいます。

つまり「AIエージェント」という言葉の問題は、定義の曖昧さそのものではありません。**モデル**、**推論スタック**、**業務自動化**、**自律実行**の境界を、売り手と買い手の双方に都合のよいように動かせてしまう点にあります。

# 推論モデルはすでに半ばエージェンティックである

近年の推論モデルは、単なるLLM本体だけを意味していません。実態としては、LLMの重み、推論時トークン、隠れた思考領域、検証ループ、再試行、システムプロンプト、安全制御などを含んだ**推論スタック**です。

ユーザーから見れば、モデルに一回質問して一回回答が返ってきているように見えます。しかし内部では、問題を分解し、仮説を立て、検証し、必要に応じて考え直し、答えを整形している可能性があります。これは、かつて外部のエージェントフレームワークで明示的に組み立てていた処理に、構造としてかなり近いものです。

ここで古典的な反論があり得ます。すなわち、エージェントとは外部環境への作用を持つものであり、内部推論だけのものは推論器に過ぎない、という立場です。この立場には一定の根拠があります。しかしそれを認めるとしても、推論スタック内部にすでに**目標分解**、**仮説生成**、**自己検証**という、かつてエージェントの中核と呼ばれていた要素が存在することは事実です。

それにもかかわらず、業界はこれを「エージェント」とは呼ばず、あくまで「推論モデル」と呼びます。ここで使われる線引きは、純粋に古典的定義に従ったものというより、商品として「モデル」と「エージェント」を別売りにできる場所に都合よく合わせられています。

# 外部拡張だけをエージェントと呼ぶ奇妙さ

世の中で「AIエージェント」と呼ばれているものの多くは、LLMの外側に組まれたプログラムです。社内データベースを検索する、メールを送る、チケットを更新する、カレンダーを操作する、コードを修正する、承認フローに回す、といった処理を束ねるものです。

こうした外部制御には実務上の意味があります。企業システムでは、LLMが勝手に本番環境を書き換えたり、顧客にメールを送ったり、請求処理を実行したりするわけにはいきません。そこで**権限管理**、**監査ログ**、**承認**、**ロールバック**が必要になります。古典的なエージェント定義に照らせば、この外部作用層こそがエージェントの本体だ、という整理にも一理あります。

しかし問題は、その層が「業務システムとして安全にLLMを使うための制御層」として説明されず、「自律的な知能が業務を理解し、人間の代わりに主体的に仕事をしている」かのように呼ばれることです。古典的定義に乗っかって「外部作用があるからエージェントだ」と言うわりに、売り文句の中身は「自律知能」のニュアンスに大きく寄っています。

ここに不誠実さがあります。外部拡張は必要です。しかし必要な制御層を、あたかも高度な自律知能のように見せる命名は、ユーザーに対する説明として誠実ではありません。

# 上流企業にとって都合がよい言葉である

基盤モデルを持つ上流企業にとって、「AIエージェント」という言葉は非常に便利です。自社のモデルAPIを下流企業に使わせながら、自社でも完成品に近いエージェント製品を売れるからです。

これは特定の一社の話ではなく、基盤モデル提供者全般に当てはまる構造です。海外大手のAI企業もクラウド大手も、いずれも基盤モデルAPIを売りつつ、その上に自社ブランドの開発支援ツールや業務支援ツールを載せて売っています。下流企業が「自社AIエージェント」を作れば作るほど、上流企業のモデル利用量が増えます。一方で、上流企業自身も「自社のAIエージェント製品」を売れます。

つまり、上流企業は**部品供給者**であると同時に、**完成品販売者**にもなれます。このため、上流企業は「モデル」と「エージェント」をあえて分けておく方が得をします。モデルだけでなく、エージェント基盤、開発環境、業務支援ツール、実行環境まで市場を広げられるからです。

ここで重要なのは、上流企業がこの言葉を意図的に作ったというより、**両方売れる立場**にいる以上、境界が曖昧なままの方が得だ、というインセンティブが働く点です。誰かの陰謀ではなく、構造的に維持される曖昧さです。

# 下流企業にとっても都合がよい言葉である

基盤モデルを持たない下流企業にとっても、「AIエージェント」は極めて都合のよい言葉です。自社で巨大なLLMを作れなくても、外側に業務連携プログラムを組めば、AI商品らしく見せられるからです。

業務アプリ会社、SIer、コンサル、SaaS企業は、もともとワークフロー、権限管理、データ連携、帳票処理、承認ルート、通知、RPAのようなものを扱ってきました。そこにLLMを接続すると、それまでの業務自動化を**AIエージェント**と呼べるようになります。

業務理解やシステム連携には確かな価値があります。特に企業ごとの業務ルール、責任分界、例外処理、監査要件を扱うには、外部の実装が欠かせません。しかし、それは「LLM付き業務オーケストレーター」や「LLM付きワークフロー基盤」と呼んだ方が正確です。問題は、ただの業務フロー拡張まで「自律的なAIエージェント」と呼ばれることです。

# ユーザー企業にとっては属人性問題の隠蔽装置になる

ここで見落とされがちなのが、買い手であるユーザー企業側の事情です。「AIエージェント」という曖昧語は、売り手の押しつけだけで広まっているのではなく、ユーザー企業にも積極的にこの語を引き取る理由があります。その中心にあるのが、**属人性問題**です。

LLMは、使用者のスキルレベルで成果が大きく変わるツールです。問いの立て方、文脈の与え方、出力の検証の仕方によって、同じモデルから引き出せる成果は何倍にも変わります。これは現場で使っている個人にとっては当然の話ですが、企業全体としては非常に扱いにくい性質です。

属人的なツールは、特定部門への限定導入であれば「あの部署にはたまたま使える人がいる」で済みます。しかし全社導入を前提にすると、説明が極めて難しくなります。投資対効果を一律に語れず、教育コストも見積もれず、評価制度との整合も取れません。経営会議で「全社員のスキル次第で効果が変わります」と説明することは、実質的に稟議として通らないのと同じです。

ここで「AIエージェント」という言葉が役に立ちます。エージェントという呼称を採用すれば、「決まった手順を覚えれば、誰が使っても同じ結果が出る」という体裁を取れるからです。スキルの高い人もそうでない人も、同じ手順を踏めば同じ成果に到達する、という前提に立てます。これにより、属人性という説明困難な性質を、組織として呑み込める形に変換できます。

つまり「AIエージェント」は、ユーザー企業にとって**属人性の脱臭装置**として機能します。社内稟議でも「ワークフロー基盤の更新」より「AIエージェントの導入」の方が通りやすく、人事評価上も映え、**DX推進**や**生成AI活用**といった社内テーマに自然に乗ります。経営層に対して「全社員がスキルに依存せず使えます」と言えるかどうかは、導入可否を分ける決定的な要素です。

結果として「AIエージェント」は、上流企業、下流企業、ユーザー企業という三者すべてにとって都合のよい語として、構造的に維持されます。これは三角構造であり、誰か一人が言葉を選び直したくらいでは崩れません。曖昧さがここまで広く維持される理由は、ここにあります。

# 不誠実さの中心は境界操作にある

「AIエージェント」という言葉の不誠実さは、嘘を一つ言っていることではありません。より厄介なのは、嘘ではないが正確でもない状態を広く作っていることです。これにより、売り手は都合に応じて説明の粒度を変えられます。

モデルが賢く見える場面では、「最新の推論モデルだからできる」と言えます。業務連携が売りになる場面では、「AIエージェントが業務を自動化する」と言えます。失敗した場面では、「まだ人間の承認が必要な支援ツールです」と言えます。同じ言葉で、**能力の主張**と**責任の回避**を同時に行えるのです。

ユーザーが本当に知りたいのは、「何がどこまで自律的に判断しているのか」「どの処理がモデル内部で、どの処理が外部プログラムなのか」「失敗時の責任はどこにあるのか」「従来の自動化と何が違うのか」です。この問いに対して、「AIエージェント」という語は答えを与えるどころか、むしろ問いを溶かしてしまいます。モデル、ツール、プロンプト、ワークフロー、権限、UI、課金単位を一つの魅力的な言葉に押し込み、細部を見えにくくするからです。

価値説明の不正確さも問題です。推論コストと外部制御層のコストは、本来は別機能の対価です。それぞれ別々に課金されること自体は不当ではありません。しかし、外部制御層が「AIエージェント」という名のもとで、あたかも自律知能そのものへの対価のように見せられると、ユーザーは何に対していくら払っているのかを判別しにくくなります。

属人性の隠蔽も同じ構造に乗っています。本当はLLMの効果がスキル依存であるにもかかわらず、エージェントという呼称によって「手順化された均質なツール」のように見せかけられます。実際の現場で成果が出ない場合でも、「使い方の問題」ではなく「エージェントの精度の問題」として処理できてしまうため、属人性の本質が温存されたまま導入が進みます。

# 外部エージェント層そのものは必要である

ただし、外部エージェント層そのものを否定するのは間違いです。企業実務では、モデル内部の推論だけでは足りません。現実の業務には、権限、履歴、承認、例外処理、障害対応、監査、法務、セキュリティが必ず関わります。古典的なエージェント定義に立てば、むしろここからが本来のエージェントの仕事です。

たとえば、AIがコードを修正するだけならモデル内部の推論でも可能です。しかし、そのコードをどのブランチに反映するのか、誰がレビューするのか、テストが落ちたらどう戻すのか、本番反映してよいのかは、外部の**開発プロセス**が決める必要があります。同じように、AIがメール文を作ることはできても、誰に送ってよいのか、添付してよいファイルか、顧客情報を含んでいないか、送信前に誰が承認するのかは、外部の業務制御で扱うべきです。

したがって、批判すべきは外部制御層ではありません。批判すべきは、外部制御層を「安全な業務実行基盤」として説明せず、**自律AI**であるかのように売ることです。価値の本体は地味な制御と業務理解にあるのに、看板だけ未来的にする構造が問題なのです。

# 本当は何と呼ぶべきか

より誠実に呼ぶなら、現在「AIエージェント」と呼ばれているものは、いくつかに分解できます。推論モデルは「推論スタック」、業務接続部分は「LLM付き業務ワークフロー」、ツール実行部分は「LLM付き実行基盤」、RPA的なものは「LLM拡張RPA」と呼ぶ方が正確です。

また、モデル内部で計画や検証をしているなら、それは「エージェンティック推論モデル」と呼べます。サーバー内部で検証ループやツール使用判断をしているなら、「エージェンティック推論ランタイム」と呼べます。外部アプリケーションが業務手順を実行しているなら、「業務エージェント基盤」と呼べます。

このように分ければ、ユーザーは何を買っているのかを理解しやすくなります。**モデル能力**を買っているのか、**実行制御**を買っているのか、**業務連携**を買っているのか、**運用責任**を買っているのか、それとも**属人性の脱臭**を買っているのかが見えます。ところが「AIエージェント」という言葉は、この区別をあえて潰します。だからこそ売りやすいのですが、だからこそ不誠実でもあります。

# まとめ

「AIエージェント」という言葉が問題なのは、エージェント性そのものが存在しないからではありません。むしろ逆で、推論モデルの内部にも、提供会社のサーバー内部にも、外部の業務アプリケーションにも、確かに**エージェント性**は分布しています。古典的な定義に立てば外部作用を伴う部分こそ本来のエージェントですが、内部推論側にもエージェンティックな要素が含まれていることは否定できません。

問題は、その連続した構造を正確に説明せず、売り手に都合のよい場所でだけ「エージェント」と呼ぶ点にあります。推論モデル内部のエージェンティック性は「モデル性能」として売られ、外部の業務接続や制御層は「AIエージェント」として別に売られます。これは技術的な必然というより、商品設計上の都合に沿った線引きです。

さらに、この曖昧さはユーザー企業側の事情によっても支えられています。LLMは本来スキル依存の属人的なツールであり、全社導入を経営的に説明するのが極めて難しい性質を持っています。エージェントという呼称はその属人性を**手順化された均質なツール**のように見せかけ、稟議と説明の壁を越えやすくします。上流企業、下流企業、ユーザー企業の三者すべてにとって、この曖昧さは利益になります。

したがって、この言葉の問題は、技術革新を表していることではなく、**技術分類**、**商品分類**、**期待形成**、**責任境界**、そして**属人性**を一つの語に混ぜてしまうことにあります。ユーザーに必要なのは、未来的な呼び名ではなく、何がモデルで、何が制御層で、何が業務連携で、どこまで自律的に実行されるのか、そして実際の成果はどの程度スキルに依存するのかという、淡々とした説明です。
