## はじめに

「X（旧Twitter）の発信を続けたいけど、ネタを考えるのがしんどい」

これは多くの個人事業主・エンジニアが抱える悩みだと思います。私自身、本業の合間にXを運用する中で、**ネタ出しと文章化に時間を取られすぎて発信が止まる**問題に何度もぶつかりました。

そこで、すでに毎日書いている Notion 日記を起点にして、X投稿の生成・承認・予約・投稿までを自動化するパイプラインを構築しました。完全自動化ではなく、**承認ポイントを意図的に残したハイブリッド型**です。

本記事ではその構成と、なぜ「全自動」ではなく「半自動」を選んだのかという設計判断を書きます。

## 技術構成

### 全体像

### 技術スタック

| レイヤー | 採用技術 | 役割 |
| --- | --- | --- |
| データソース | Notion DB | 毎日書いている日記をそのまま素材化 |
| 同期 | launchd + Node.js | Notion API → Firestore に毎日12:00同期 |
| 永続化 | Firestore | posts / notion\_posts / reference\_posts コレクション |
| AI生成 | Claude API (sonnet) | 日記から投稿候補を5〜7本まとめて生成 |
| 承認UI | Discord (Webhook + Bot) | プレビュー表示、承認、日時指定をDiscord上で完結 |
| スケジューラ | Google Apps Script | Notion同期、承認通知、投稿Cronをトリガー駆動 |
| 投稿先 | X API v2 | OAuth 1.0a で投稿、自己リプライでスレッドも対応 |

### なぜ Firestore を中継したか

Notion API を直接叩いて投稿生成するのが最短ですが、以下の理由で Firestore を中継しました。

* **Notion API のレート制限と遅延**: 1秒3リクエスト制限があり、複数処理がぶつかると遅い
* **検索性**: 「未処理の投稿」「投稿済かつ未展開」のような条件抽出が Firestore のほうが速い
* **コレクション設計の柔軟性**: 参考投稿 (reference\_posts)、生成済投稿 (posts)、Notion同期キャッシュ (notion\_posts) を分けて運用できる

## 実装のポイント

### 1. バッチ生成 + 段階的承認

「1本ずつリアルタイムに生成して投稿」だと、AIの揺らぎで品質がバラつきます。逆に「全部AIに任せる」とブランド毀損のリスクがあります。

採用したのは中間案 — **週1〜2回、5〜7本まとめて生成 → 1日1本ずつ承認通知 → 承認後にスケジュール予約**というフローです。

```
// scripts/fetch-unprocessed.mjs
// Notion日記の未処理分を取得
async function fetchUnprocessedDiaries() {
  const snap = await db.collection('notion_posts')
    .where('processed', '==', false)
    .orderBy('created_at', 'desc')
    .limit(20)
    .get();
  return snap.docs.map((d) => ({ id: d.id, ...d.data() }));
}

// scripts/save-generated.mjs
// 生成投稿をサニタイズして保存
async function saveGenerated(posts) {
  const batch = db.batch();
  for (const post of posts) {
    const sanitized = sanitize(post.text); // 個人名・固有名詞の除外
    const ref = db.collection('posts').doc();
    batch.set(ref, {
      text: sanitized,
      status: 'サニタイズ済',
      target: post.target,      // 経営者向け / エンジニア向け / AI関連
      angle: post.angle,        // 切り口
      created_at: FieldValue.serverTimestamp(),
      scheduled_post_time: null,
    });
  }
  await batch.commit();
}
```

### 2. サニタイズは生成と分離する

Notion 日記には個人名、クライアント名、内省的な内容など、外部に出してはいけない情報が混ざります。AIに「サニタイズしながら書いて」と指示するだけだと取りこぼしが出ます。

そこで **生成プロンプトとサニタイズロジックを分離**しました。

* 生成プロンプト側: トーン・文字数・切り口の指示に専念
* サニタイズ関数: 固有名詞リスト、個人情報パターン、NG表現を機械的にフィルタ

これによって、生成の自由度を保ちながら、漏洩リスクを構造的に減らせました。

### 3. Discord を承認UIにする利点

承認用のWebアプリを作る案もありましたが、以下の理由で Discord に集約しました。

* **モバイルから即承認できる**: 通勤中でも電車内でもスマホで完結
* **チャンネルで用途を分離**: `#x-approval`（承認）、`#content-hub`（コピペ用）、`#task-report`（完了報告）
* **リプライで日時指定**: 「明日 7:00」「4/5 10:00」のような自然文で予約可能


### 4. 完全自動化を「あえて」やめた判断

最初は「投稿生成 → 自動投稿 → メディア展開（note/Qiita/Zenn）まで全部自動」を目指していました。が、運用してみてやめました。

| 完全自動化のリスク | 実際に起きた / 起きうること |
| --- | --- |
| note の bot 検知 | アカウント凍結（致命的） |
| AIの誤情報拡散 | 訂正コストが大きく信用毀損 |
| トーン崩壊 | 個人ブランドと乖離した投稿で違和感 |
| 個人情報の漏洩 | サニタイズ漏れが拡散される |

そこで **承認ポイントを意図的に残すC案（ハイブリッド半手動）** に方針転換しました。X投稿は承認後に自動投稿、メディア展開は「未展開検出 → 通知 → スキル起動 → 1本ずつ承認後展開」というフローです。

承認の手間は1日1〜2分程度。これで上記リスクを構造的に避けつつ、追跡漏れもゼロにできました。

## 運用してわかったこと

### 「自動化」と「自動通知」を分ける

完全自動化を目指すと、リスクの高い処理（外部投稿、課金、削除）まで自動化対象になります。これを **「自動化（実行まで自動）」と「自動通知（人間にプッシュするだけ）」** に分けると設計が一気に楽になりました。

例: メディア展開は「未展開検出 → Discord 通知」までを自動化、展開実行は人間が承認。これだけで凍結リスクを避けつつ、追跡漏れはゼロにできます。

### Firestore のコレクション設計が後で効いてくる

最初は1コレクションにフラグで区別していましたが、用途別に分離した（posts / notion\_posts / reference\_posts）ことで、後からの集計・分析・メディア展開判定が圧倒的に楽になりました。

特に `media_published` フラグを `posts` に持たせたことで、「投稿済かつ未展開」の検出が1クエリで済みます。

```
// 未展開検出クエリ
const snap = await db.collection('posts')
  .where('status', '==', '投稿済')
  .where('media_published', '!=', true)
  .orderBy('media_published')
  .orderBy('posted_at', 'desc')
  .get();
```

### Cronトリガーの分散配置

GASのCronトリガーを以下のように分散配置することで、1ジョブが落ちても他に波及しないようにしました。

| トリガー | 時刻 | 役割 |
| --- | --- | --- |
| Notion同期 | 毎日 12:00 | Notion → Firestore |
| 承認通知 | 毎日 13:00 | サニタイズ済投稿を1本 Discord 通知 |
| 投稿Cron | 10分毎 | 予約時刻を過ぎた投稿をX投稿 |
| 未展開検出 | 毎日 11:00 | 投稿済 & 未展開を Discord 通知 |

## まとめ

X投稿自動化は「全自動を目指さない」のが結果的に正解でした。重要だったのは以下の3点です。

1. **データの中継地点（Firestore）を置くことで、検索性・拡張性が劇的に上がる**
2. **生成とサニタイズは分離する。AIに両方を任せない**
3. **承認ポイントを残し、「自動化」と「自動通知」を切り分ける**

次の一手としては、Notion 日記以外のソース（読書ログ、技術検証メモ）からの自動生成、メディア展開後のエンゲージメント分析の自動化を進めています。

「完全自動化」が美しく見えても、運用リスクと品質を考えると半自動が現実解になることは多いです。設計判断の参考になれば幸いです。
