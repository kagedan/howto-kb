## TL;DR

* Claude のプレビューはブラウザの実環境と異なる。特に画像まわりで差異が出やすい
* 「完成してからデプロイ」ではなく「デプロイして確認する」を繰り返す方が効率がいい
* MinTo は数秒でデプロイ・URL発行されるので、確認環境として使いやすい
* Canva（デザイン）→ Claude（HTML生成）→ MinTo（デプロイ・確認）の流れがスムーズ

---

## 背景

自分が作っている静的サイトホスティングサービス [MinTo](https://lp.mintoai.net/) を、パソコン操作に慣れていない親に使ってもらいました。

使ったツールはこの3つです。

| ツール | 役割 |
| --- | --- |
| Canva（スマホアプリ） | デザイン作成・PNG エクスポート |
| Claude（スマホアプリ） | 画像を渡してHTML生成・修正 |
| MinTo | デプロイ・URL発行・確認 |

結果として5つのサイトが公開されました。

この過程で「Claude のプレビューを確認環境として使う」というアプローチに問題があることがわかり、MinTo を確認環境として使う方法に切り替えました。

---

## 問題：Claude のプレビューはブラウザの実環境と異なる

Claude に HTML を生成してもらい、プレビューで確認すると画像が表示されないことがありました。

原因は Claude のプレビュー機能の仕様です。Claude のプレビューは Artifact 上でのレンダリングであり、実際のブラウザ環境とは異なります。特にアップロードした画像ファイルの参照まわりで差異が出やすいです。

エンジニアなら「ローカルで確認する」という選択肢がありますが、今回のユースケースではその手段を取れませんでした。

---

## 解決：MinTo を確認環境として使う

行き着いたのが「未完成の状態でも MinTo にデプロイして、発行された URL で確認する」というアプローチです。

MinTo はデプロイが数秒で完了し、即座に URL が発行されます。その URL をブラウザで開くと実際の表示が確認できます。Claude のプレビューで表示されていなかった画像も、MinTo の URL では正しく表示されていました。

ワークフローはこうなります。

```
1. Canva でデザインを作成 → PNG でエクスポート
2. Claude に画像を渡して HTML を生成してもらう
3. MinTo にデプロイして発行された URL で確認
4. 差異があれば Claude に修正を依頼
5. 3 に戻る
```

「完成してからデプロイ」ではなく「デプロイしながら作る」に切り替えたことで、作業のフィードバックループが速くなりました。

---

## MinTo を確認環境として使うメリット

**デプロイが速い**  
数秒で URL が発行されるので、確認のたびに待たされることがありません。

**実ブラウザ環境で確認できる**  
Artifact のプレビューではなく、実際にホスティングされた URL で確認できます。CSS や画像の挙動が本番と一致します。

**何度でもデプロイできる**  
上書きデプロイに対応しているので、修正のたびに新しい URL を発行する必要がなく、同じ URL で最新の状態を確認できます。

## まとめ

Claude で HTML を生成するワークフローでは、プレビューに頼らず **「デプロイして実 URL で確認する」サイクルを回す**のが現実的です。

MinTo はデプロイの速さと手軽さからこの用途に向いています。静的サイトの生成 → 確認 → 修正のループを回す確認環境として、ぜひ試してみてください。
