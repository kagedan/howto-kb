# AIにテストさせると「浅くなる」問題

<https://qiita.com/k-sak/items/067216bed3adc7f8eac1>  
上記の記事でも書いたが、各テストプロセスでAIに中間成果物を作らせたことで、  
人間側の理解がかなり浅くなってしまい、空白領域ができた。

また、生成AIを使ってテストを自動化できないか、という話はよく聞くようになった。  
で、大体がリグレッションではなく、人間がやっている「新機能のE2Eテスト」を自動化できないか、という話だ。  
実際に試してみると、ある程度まではうまくいきそうかなと。  
テストケースを生成し、UI操作を自動化し、結果を出力することもできる。  
（その辺の話はコチラ）  
<https://zenn.dev/k_sak/articles/91c86c7a733e9c>

ただ、動かしていてすぐに違和感が出てくる。

* それっぽいケースを消化しているだけじゃん
* なんかテストっぽいことはしているけど、新しい発見がない
* 「なんかおかしい」が拾えない（OKかNGしかない）

テストは実行できているのに、テスト結果を見ても品質に対する理解が一向に深まらない。  
このあたりのAIを使った時の「浅さ」がずっと引っかかっていた。

# QAの価値は「Discovery」にある

改めて考えると、QAの価値はテストの実行そのものにはない。  
例えばケースを消化するためにテストをしたところで何のビジネス成果にも結びつかない。  
ソフトウェアテストの本質は、次のサイクルにあると思っている。

* 挙動を観察する
* 違和感を見つける
* 違和感に対する仮説を立てる
* 仮説を検証する

QAは、この「発見 → 仮説 → 検証」の繰り返しとチームへのフィードバックによって、  
プロダクトの理解が深まったり、リスクを露見させたりしている。

観点リストやテストケースは、このサイクルを回すための**起点**であって、  
ゴールではない。

つまり、AIを使ったテスト自動化においてAIに任せるべきなのは「操作」ではなく、上述の **「思考のサイクルそのもの」** だと考えた。

# 思考サイクルをエージェントに埋め込む

そこで作ったのが、探索的テストの思考をそのまま再現するエージェント。  
やっていることはシンプルで、以下のループを回している。

```
1. テストを実行する
2. 観察した事実を記録する（Finding）
3. 複数のFindingから仮説を立てる（Hypothesis）
4. 仮説を検証するテストを作る（Probe）
5. 実行してナレッジに蓄積する
→ 2に戻る
```

先述の『AIでテストプロセスを自動化する』に書いた通り、2025年にはツールからAPIで生成AIを呼ぶような構想も、ある程度は実現した。  
しかし、毎回AIが「何も知らない状態」に戻ってしまうことに手を焼いた。  
今回、幸運にもナレッジ化の方法を得たことでこの課題がクリアできた。  
（JaSST26Tokyoで複数のセッションを視聴したところ、合わせ技でアイデアが降ってきたの！）

ナレッジを生成して活用するサイクルによって、テストが単発で終わらずに  
「連続した探索」が可能になる。

# エージェントはどう動いたか

実際にこのエージェントを動かしてみた。  
対象はお馴染みの「テスト自動化練習サイト」を使わせていただいた。  
<https://hotel-example-site.takeyaqa.dev/ja/>

1つの発見からAIの探索が広がっていく

例えば、最初にこんな発見があった。  
`hiddenフィールドの値を改ざんすると、料金が0円になる`

ここからエージェントはこう動いた。

* 他のプランでも同じことが起きるのか？
* 条件を変えたらどうなるか？
* なぜこれが通ってしまうのか？

単発のバグを起票して終わりではなく、  
「このシステムはどう作られているのか」に興味が移ったようだ。

複数のFinding（気づき）が揃ったタイミングで、仮説が立つ。  
`サーバ側でのバリデーションが存在しないのではないか？`

この時点で、テストは  
バグ探しやケース消化から「構造理解のための検証」に変わる。

# ナレッジによって探索が加速する

このエージェントは、Finding（気づき）やHypothesis（仮説）を読み込んでからテストを開始する。

その結果、ナレッジの有無によりセッションごとに明確な違いが出た。

|  | Session 5（観点リストのみ） | Session 6（ナレッジあり） |
| --- | --- | --- |
| テストの起点 | 事前に与えた観点リスト | 前セッションのFinding・仮説 |
| 探索の広がり | リストに沿った実行のみ | 他プランへの横展開を試みる |
| 観点の深さ | 観点の指定範囲内に留まる | 境界値の特定・プランごとの差異を発見 |

テストをするたびに、次のテストが深く賢くなる状態が生まれたわけだ。

# レポートは「結果」ではなく「思考」

このエージェントのアウトプットは、いわゆるテストレポートだが、中身はかなり違う。

通常のPlaywrightレポートなんかだと、

といった「過去の記録の羅列」になりがちだ。  
一方で、このレポートには

* Finding（事実）
* Hypothesis（仮説）
* Probe（次の一手）
* 後続セッションへの申し送り事項

を含むようにした。この構造にしたことで、  
未解決の仮説や、次にやると良いことが具体的に書かれたテスト結果レポートになる。

結果として、レポートを読むと自然にこうなる。

「お？これって他でも起きるのでは？」  
「こんなパターンも試したいな」  
「次はここをもっと突っ込んで見てみよう」

つまり、AIが通った道の跡が、人間が実行するテストへの示唆を与えてくれる。  
実際にこのエージェントが吐いたテストレポートを初めて見た時は、テストをお願いしていた方からの報告を受けているような気持ちになった。  
**「テスターを1人追加した」** くらいの衝撃だった。

# このエージェントが変えるもの

この仕組みで一番大きいのは、テストの性質を変えることなく、まるで人間のようなAIテスターが仲間に加わることだ。

従来の人間中心のテストでは、簡単に以下のアンチパターンに陥る構造だった。

* 実行して終わる
* 経験が個人に閉じる
* ナレッジが残りにくい

一方でこのエージェントは、

ことができる。

結果として、

> テストすればするほど、ナレッジが溜まっていく

状態になる。

# まとめ

このエージェントはバグを見つけるためのものではない（バグも見つけてくるけど）。  
「なぜそれが起きるのか」を考え続けるための装置だ。

テストを重ねるほど、理解が深くなる。  
そして次のテストは、前よりも少し賢くなる。

AIによって中間成果物を生成することで、プロダクトの理解が浅くなった実感があるが、  
このエージェントは僕に「テストの楽しさ」を取り戻してくれる可能性が高い。  
それはおそらく、AI登場以前よりも格段にテスト担当者を賢くしてくれると信じている。
