## はじめに

2026年5月、Claude Opus 4.8（Anthropic）、GPT-5.5 Instant（OpenAI）、Gemini 3.5 Flash（Google）がほぼ同時にリリースされました。それぞれ得意分野が異なります。

* **Claude Opus 4.8**：エージェント型コーディングが強い（69.2%）
* **GPT-5.5 Instant**：ハルシネーション52.5%削減で高リスク領域に強い
* **Gemini 3.5 Flash**：速度と精度のバランスに優れる

「タスクごとに最適なモデルへ振り分けたい」というニーズが現実的になりました。本記事では、タスク種別に応じて3モデルを使い分ける軽量な **マルチLLMルーター** をPythonで実装します。

## 設計方針

タスクを3種類に分類し、それぞれ最適なモデルへルーティングします。

| タスク種別 | ルーティング先 | 理由 |
| --- | --- | --- |
| コード生成・エージェント | Claude Opus 4.8 | コーディングスコア最高 |
| 事実性が重要（医療/法律/金融） | GPT-5.5 Instant | 幻覚52.5%削減 |
| 高速・大量処理 | Gemini 3.5 Flash | 速度と精度の両立 |

## 実装

まずはルーティングロジック本体です。各SDKは遅延importにしています。

```
from enum import Enum
from dataclasses import dataclass

class TaskType(Enum):
    CODING = "coding"
    FACTUAL = "factual"   # 高リスク・事実性重視
    FAST = "fast"         # 高速・大量処理

@dataclass
class ModelChoice:
    provider: str
    model: str

# タスク種別 -> モデルのマッピング
ROUTING_TABLE = {
    TaskType.CODING:  ModelChoice("anthropic", "claude-opus-4-8"),
    TaskType.FACTUAL: ModelChoice("openai",    "gpt-5.5-instant"),
    TaskType.FAST:    ModelChoice("google",    "gemini-3.5-flash"),
}

def classify(prompt: str) -> TaskType:
    """シンプルなキーワード分類器（実運用ではLLM分類に置換可）"""
    p = prompt.lower()
    if any(k in p for k in ["code", "実装", "バグ", "関数"]):
        return TaskType.CODING
    if any(k in p for k in ["医療", "法律", "税", "投資"]):
        return TaskType.FACTUAL
    return TaskType.FAST

def route(prompt: str) -> ModelChoice:
    choice = ROUTING_TABLE[classify(prompt)]
    print("route ->", choice.provider, choice.model)
    return choice
```

## 統一インターフェースで呼び出す

各プロバイダを同じI/Fでラップします。

```
def call_llm(prompt: str) -> str:
    choice = route(prompt)
    if choice.provider == "anthropic":
        from anthropic import Anthropic
        msg = Anthropic().messages.create(
            model=choice.model, max_tokens=1024,
            messages=[{"role": "user", "content": prompt}],
        )
        return msg.content[0].text
    if choice.provider == "openai":
        from openai import OpenAI
        return OpenAI().responses.create(model=choice.model, input=prompt).output_text
    if choice.provider == "google":
        from google import genai
        return genai.Client().models.generate_content(
            model=choice.model, contents=prompt).text
    raise ValueError(choice.provider)
```

## フォールバックを足す

エージェント運用では「あるモデルが落ちても止まらない」ことが重要です。

```
FALLBACK_ORDER = ["anthropic", "openai", "google"]

def call_with_fallback(prompt: str) -> str:
    primary = route(prompt).provider
    order = [primary] + [p for p in FALLBACK_ORDER if p != primary]
    last_err = None
    for provider in order:
        try:
            return _call_provider(provider, prompt)
        except Exception as e:
            last_err = e
            print("fallback:", provider, "failed:", e)
    raise RuntimeError("all providers failed: " + str(last_err))
```

## まとめ

3大モデルが横並びで進化したことで、「1モデル固定」から「タスク最適でルーティング」へ設計の重心が移りつつあります。本記事のルーターは最小構成ですが、`classify` をLLMベースに、`call_with_fallback` をリトライ＋指数バックオフに拡張すれば、そのままエージェント基盤の土台になります。

「AIを使う」から「AIに任せる」へ。その第一歩は、**適材適所のルーティング** から始まります。
