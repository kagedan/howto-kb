## はじめに：16本のSaaSを1人で回している話

現在、個人で16本のSaaSを稼働させています。合計MRRは数万円規模ですが、運用に使っている時間は週に数時間程度です。

なぜそれが可能かというと、**AIエージェントに多くの「仕事」を委譲するアーキテクチャ**を採用しているからです。

この記事では、そのアーキテクチャの考え方と実装の一部を共有します。誇張なしに書くので、「全自動で億稼ぐ」みたいな話は出てきません。あくまで「1人でも複数プロダクトを無理なく維持できる仕組み」の話です。

---

## 課題：個人開発はスケールより「維持コスト」が問題

SaaSを1本リリースするのは、今やそれほど難しくありません。問題は**リリース後の維持コスト**です。

具体的には、

* ユーザーからの問い合わせ対応
* ランディングページのA/Bテスト
* エラー検知と対応
* 競合調査・市場リサーチ
* 新機能のアイデア出し

これらを人手でやっていると、2〜3本が限界です。実際に `FreelancePainPollBot`（¥4,350/月）や `WorkflowNagBot`（¥2,850/月）を並走させるようになったとき、手動運用のボトルネックを強く感じました。

---

## 解決策：「判断」ではなく「作業」をエージェントに渡す

よくある誤解として、「AIエージェントに意思決定させる」というアプローチがあります。ただ実際には、**意思決定は自分が行い、ルーチン作業をエージェントに渡す**設計のほうが安定します。

自分が採用しているのは以下の役割分担です：

| 自分がやること | エージェントに任せること |
| --- | --- |
| プロダクト戦略の決定 | ユーザーフィードバックの要約 |
| 価格・機能の判断 | エラーログの分類とレポート |
| コードの最終レビュー | LPのコピー案の初稿生成 |
| リリース判断 | 問い合わせの一次回答ドラフト |

この「渡し方」の設計がアーキテクチャの核心です。

## 実装：n8n + Claude APIで作る自律ワークフロー

実際に使っているワークフローの簡略版を示します。`AIHtmlDiffAlert`（¥1,350/月）の運用で使っているエラー検知→通知→対応ドラフト生成のフローです。

yaml

# n8nワークフロー概要（擬似コード）

trigger: webhook（Vercelのエラーアラート）

step1\_classify:  
tool: Claude API  
prompt: |  
以下のエラーログを分類してください。  
カテゴリ: [critical, warning, info]  
影響範囲: [全ユーザー, 一部ユーザー, 個人]  
推定原因: （簡潔に）  
input: "{{ $json.error\_log }}"

step2\_draft\_response:  
condition: category == "critical"  
tool: Claude API  
prompt: |  
このエラーに対してユーザーへ送る  
ステータス通知メールの初稿を書いてください。  
日本語・200字以内・謝罪トーン

step3\_notify:  
tool: Slack  
message: |  
[{{ $json.category }}] エラー検知  
原因: {{ $json.cause }}  
ドラフト: {{ $json.draft }}  
→ 送信する場合は承認ボタンを押してください

**ポイントは最後の「承認ステップ」を残していること**です。エージェントが自動送信するのではなく、自分がSlackで承認した場合にのみメールが送られます。完全自動化より、この「人間が最終確認する」設計のほうが長期的に信頼できます。

## もう一つの実装：市場リサーチの自動収集

`FreelancePainPollBot` のネタ収集に使っているのが、以下のような定期リサーチエージェントです。

python

# 週次で走るPythonスクリプト（Cloudflare Workers経由）

import anthropic

client = anthropic.Anthropic()

def weekly\_research\_summary(raw\_posts: list[str]) -> str:  
prompt = f"""  
以下はフリーランサーコミュニティの投稿です。  
「まだツールで解決されていない業務上の困りごと」を  
上位5つ、箇条書きで日本語にまとめてください。  
根拠となる投稿番号も併記すること。

```
投稿リスト:
{chr(10).join(raw_posts)}
"""

message = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": prompt}]
)
return message.content[0].text
```

これをNotionに自動保存し、月1回自分が見て次の機能開発の参考にしています。

## 正直なところ：自律収益の実態

「自律収益」という言葉を使いましたが、**完全に放置して収益が増え続けるわけではありません**。

実際の感覚としては、

* **手放せた作業**: ログ監視、フィードバック要約、一次対応のドラフト作成
* **まだ自分が必要な作業**: LP改善の判断、価格設定、新機能開発
* **エージェント化を失敗した例**: 問い合わせの自動返信（トーンがブレてユーザーに指摘された）

`SoloTaxReceiptBot`（¥1,350/月）では、オンボーディングの自動案内をエージェント化しようとして一度失敗しています。ステップ数が多すぎて、エラー時のフォールバックが機能しなかったためです。今は「最初の1ステップだけ自動化し、詰まったら人間対応」という設計に戻しました。

## まとめ：委譲できる粒度を見極めることが全て

AIエージェントへの委譲で大切なのは、\*\*「タスクの粒度と成功基準を明確にすること」\*\*です。

* ✅ 「このログをカテゴリ分類して」→ 委譲しやすい
* ✅ 「200字の通知文を書いて」→ 委譲しやすい
* ❌ 「ユーザーが満足するように対応して」→ 委譲できない

16本のSaaSを回せているのは、AIが賢いからではなく、**自分が「何を任せられるか」を丁寧に設計してきたから**だと思っています。

小さく始めて、失敗して、粒度を調整する。その繰り返しが、個人開発者にとっての現実的なAIエージェント活用だと感じています。
