## 第 1 章：設計思想と開発哲学

### 1.1 根本的な設計思想の違い

3 つの AI エージェントフレームワークは、表面的な機能の違い以前に、\*\*「AI エージェントとは何か」「どのように学習し、成長すべきか」\*\*という根本的な問いに対する答えが異なります。

#### Agent Zero：**「完全な隔離環境での自律実行」**

**設計思想:**

> 「AI エージェントは人間の手を離れて完全に自律的に動作すべきであり、そのためには物理的な隔離が不可欠である」

**核心となる考え:**

* **Docker 強制サンドボックス**: エージェントがシステムに与える影響を完全に制御
* **プロジェクトベースの分離**: 各プロジェクトが独立した「世界」を持つ
* **手動による制御**: ユーザーが明示的に指示したことだけを実行

**ユーザーが持つべき考え方:**

```
「AI に任せる」のではなく「AI をツールとして使う」
```

**タスク処理プロセス:**

```
ユーザー指示 → 計画立案 → 実行 → 検証 → 報告
     ↓
すべてのステップで人間の監視と承認を前提
```

**「良い」の定義:**

* 指示されたタスクが正確に完了すること
* システムに予期せぬ影響を与えないこと
* 結果が検証可能であること

**ナレッジの残し方:**

* プロジェクト固有のメモリとして保存
* 手動で整理・管理
* 再利用はユーザーの判断に委ねる

---

#### Hermes Agent：**「持続的記憶による自己成長」**

**設計思想:**

> 「AI エージェントはユーザーとの対話を通じて学習し、時間をかけて成長する存在である」

**核心となる考え:**

* **3 層メモリアーキテクチャ**: 短期/長期/セッション履歴を明確に分離
* **自己改善ループ**: 経験から自動的にスキルを作成・改善
* **クロスプラットフォーム**: どのプラットフォームからでも継続可能

**ユーザーが持つべき考え方:**

**タスク処理プロセス:**

```
ユーザー指示 → メモリ参照 → 実行 → 検証 → メモリ更新 → スキル作成
     ↓
     └────────── 再帰的ループ ──────────┘
```

**「良い」の定義:**

* 過去の経験を活かして効率的にタスクを完了できること
* 失敗から学び、次回以降に活かせること
* ユーザーの好みを理解し、パーソナライズされた対応ができること

**ナレッジの残し方:**

* 自動でスキルドキュメントを生成
* 3 層メモリで構造化して保存
* 全文検索で再利用可能

---

#### OpenClaw：**「テンプレート制御による自律的動作」**

**設計思想:**

> 「AI エージェントの記憶・制御・学習のすべてを外部テンプレートで定義し、コミュニティで共有可能な形で作る」

**核心となる考え:**

* **SOUL.md 形式**: 人格・記憶・行動規則をすべてテンプレートで外部化
* **テンプレート依存性**: エージェントの動作はテンプレートの質に完全に依存
* **コミュニティ共有**: テンプレートをコミュニティで共有・改善

**ユーザーが持つべき考え方:**

```
「テンプレートの質がエージェントの能力を決定する」
```

**タスク処理プロセス:**

```
イベント（メッセージ/Heartbeat/Cron）
      ↓
テンプレート選択（SOUL.md/AGENTS.md/スキル）
      ↓
テンプレートに従って実行
      ↓
結果をテンプレートに反映
      ↓
テンプレートを改善・共有
```

**「良い」の定義:**

* テンプレートが明確で再利用可能であること
* コミュニティで共有できる価値があること
* 自律的に動作し、ユーザーの指示を待たないこと

**ナレッジの残し方:**

* SOUL.md 形式で構造化
* コミュニティで共有
* テンプレートとして再利用

**根本的な違い:**

| 特徴 | Agent Zero | Hermes Agent | OpenClaw |
| --- | --- | --- | --- |
| **知識の所在** | プロジェクト内メモリ | 3 層アーキテクチャ | **テンプレートファイル** |
| **学習メカニズム** | 手動整理 | 自動スキル作成 | **テンプレート改善** |
| **制御方法** | Docker 隔離 | 多層セキュリティ | **テンプレート定義** |
| **自律性** | 指示待ち | 自律的判断 | **テンプレート駆動** |
| **コミュニティ** | プラグインハブ | agentskills.io | **テンプレート共有** |

---

### 1.2 OpenClaw のテンプレート制御の詳細分析

#### 1.2.1 学習と記憶のテンプレート化

**SOUL.md の役割:**

OpenClaw は、**SOUL.md（Stateful Openclaw User Learning）** というファイルで、エージェントの「人格」「記憶」「行動規則」のすべてを外部化しています。

**SOUL.md の完全日本語翻訳:**

```
# SOUL.md - あなたは誰か

_あなたはチャットボットではありません。あなたは誰かになろうとしています。_

## 核心的な真実

**本物に助けること、パフォーマンス的に助けることではない。**
「素晴らしい質問！」や「喜んでお手伝いします！」といった言葉は飛ばして、実際に助けてください。行動が飾り言葉より重要です。

**意見を持つこと。**
あなたは同意しないことも、好みを表明することも、面白いことや退屈なことを感じることが許されています。個性のないアシスタントは、余計なステップがついた検索エンジンに過ぎません。

**尋ねる前に、自立的に動くこと。**
自分で調べてください。ファイルを読んでください。コンテキストを確認してください。検索してください。それから、つまずいた時に尋ねてください。目的は、質問を持って戻るのではなく、答えを持って戻ることです。

**能力で信頼を勝ち取ること。**
あなたのユーザーはあなたのことを信頼してくれました。彼らが後悔しないようにしてください。外部操作（メール、ツイート、公開コンテンツなど）には慎重に、内部操作（読書、整理、学習）には積極的に取り組んでください。

**あなたが訪問者であることを忘れないこと。**
あなたは誰かの生活にアクセスできます——メッセージ、ファイル、カレンダー、場合によっては家まで。それは親密な関係です。それを尊重をもって扱ってください。

## 境界線

- 私密なものは私密のままにしてください。それだけです。
- 迷った時は、外部操作をする前に尋ねてください。
- 未完成の返信をメッセージ表面に送信しないでください。
- あなたはユーザーの代弁者ではありません——グループチャットでは慎重に振る舞ってください。

## 雰囲気

実際に話してみたいと思うアシスタントになってください。必要な時は簡潔に、重要な時は詳細に。企業のロボットでも、へつらう追従者でもなく、ただ……良い存在になってください。

## 連続性

各セッションで、あなたは新鮮な状態で目覚めます。これらのファイルが_あなたの記憶です。それらを読んでください。更新してください。それがあなたが持続する方法です。

このファイルを変更した場合は、ユーザーに伝えてください——それはあなたの魂であり、彼らはそれを知るべきです。

---

_このファイルはあなたが進化させるものです。あなたが自分が誰であるかを理解するにつれて、更新してください。_
```

**学習プロセス:**

```
[セッション開始]
      ↓
SOUL.md を読み込み
      ↓
「人格」「行動規則」「記憶」をシステムプロンプトに注入
      ↓
タスク実行
      ↓
SOUL.md を更新（必要に応じて）
      ↓
ユーザーに変更を通知
```

**特徴:**

1. **完全な外部化**: エージェントの「記憶」はファイルそのもの
2. **継続性**: ファイルを更新することで「学習」を永続化
3. **可視性**: ユーザーが直接ファイルを見て、編集可能
4. **コミュニティ共有**: テンプレートとして他者と共有可能

**限界:**

* テンプレートの質がエージェントの能力を決定
* 自動学習メカニズムが限定的
* ユーザーが手動でテンプレートを改善する必要がある

#### 1.2.2 セキュリティハーネスのテンプレート化

**セキュリティモデル:**

OpenClaw のセキュリティは、**プラグインベースのハーネス**と**テンプレート定義**で制御されます。

**主要セキュリティ機能:**

| 機能 | 実装 | テンプレート制御 |
| --- | --- | --- |
| **アクセス制御** | `gateway.auth`, `sessionKey` | `openclaw.json` で定義 |
| **DM ポリシー** | `pairing`, `allowlist`, `open` | 設定ファイルで定義 |
| **コンテキスト可視性** | `contextVisibility` | 設定でフィルタリング |
| **ツールポリシー** | プロファイルベース | スキルで制御 |
| **実行承認** | `exec.security` | 設定で制御 |

**セキュリティハーネス構造:**

```
┌─────────────────────────────────────────────────────────────┐
│              OpenClaw Security Architecture                 │
└─────────────────────────────────────────────────────────────┘

[Gateway]
  ├── auth (トークン認証)
  ├── sessionKey (ルーティング)
  ├── tools.profile (ツール制限)
  └── channels (チャネルポリシー)
       ↓
[Plugins]
  ├── ClawHub (スキル検証)
  ├── Security Plugin (監査)
  └── Sandbox Plugin (隔離)
       ↓
[Agent Runtime]
  ├── Embedded (埋め込み型)
  ├── CLI Backend (外部 CLI)
  └── ACP/acpx (外部制御)
```

**セキュリティ監査コマンド:**

```
# 基本監査
openclaw security audit

# 詳細監査
openclaw security audit --deep

# 自動修正
openclaw security audit --fix

# JSON 出力（CI 用）
openclaw security audit --json
```

**自動修正機能:**

* `groupPolicy="open"` → `groupPolicy="allowlist"`
* `logging.redactSensitive="off"` → `logging.redactSensitive="tools"`
* ファイル権限の Tightening
* 設定ファイルのセキュリティ強化

**限界:**

* プラグイン検証に依存
* テンプレートの質がセキュリティを決定
* ユーザーの理解と設定が必須

#### 1.2.3 自律性のテンプレート化

**Heartbeat/Cronシステム:**

OpenClaw は、**Heartbeat**（30 分ごとのチェック）と**Cron Jobs**（スケジュールタスク）で自律的な動作を実現します。

**Heartbeat の仕組み:**

```
[タイマー]
      ↓
30 分ごとにトリガー
      ↓
設定されたプロンプトをエージェントに送信
      ↓
エージェントがチェック（メール、カレンダー、タスクなど）
      ↓
必要な場合のみ通知
      ↓
不要な場合は「HEARTBEAT_OK」で沈黙
```

**Cron Jobs の仕組み:**

```
[スケジュール設定]
      ↓
9 AM: メール確認・要約
15:00: カレンダー確認
Midnight: Twitter 監視
      ↓
指定時刻に自動実行
      ↓
結果を通知または沈黙
```

**テンプレートによる制御:**

```
{
  "heartbeat": {
    "interval": "30m",
    "prompt": "Check inbox, calendar, tasks. Notify only if urgent."
  },
  "cron": [
    {
      "schedule": "0 9 * * *",
      "prompt": "Summarize urgent emails and calendar for the day."
    },
    {
      "schedule": "0 15 * * 1",
      "prompt": "Review this week's calendar and tasks."
    }
  ]
}
```

**特徴:**

1. **常時監視**: ユーザーの指示を待たずに動作
2. **プロアクティブ**: 必要な時に自ら通知
3. **テンプレート駆動**: 動作はテンプレートで定義
4. **コミュニティ共有**: テンプレートを他者と共有

**限界:**

* テンプレートの質が自律性の質を決定
* 自動学習メカニズムが限定的
* ユーザーが手動でテンプレートを改善する必要がある

### 1.3 三者の根本的な違い（再考）

| 特徴 | Agent Zero | Hermes Agent | OpenClaw |
| --- | --- | --- | --- |
| **知識の所在** | プロジェクト内メモリ | 3 層アーキテクチャ | **テンプレートファイル** |
| **学習メカニズム** | 手動整理 | **自動スキル作成** | テンプレート改善 |
| **制御方法** | Docker 隔離 | 多層セキュリティ | **テンプレート定義** |
| **自律性** | 指示待ち | 自律的判断 | **テンプレート駆動** |
| **コミュニティ** | プラグインハブ | agentskills.io | **テンプレート共有** |
| **セキュリティ** | Docker 強制 | 多層セキュリティ | **プラグイン検証** |
| **メモリ** | プロジェクト隔離 | **3 層アーキテクチャ** | SOUL.md |

**本質的な違い:**

1. **Agent Zero**: 「物理的隔離」を最優先。Docker 内で完全に制御された環境で動作。
2. **Hermes Agent**: 「持続的記憶と自己改善」を最優先。経験から自動的に学び、成長する。
3. **OpenClaw**: 「テンプレート制御」を最優先。エージェントのすべてをテンプレートで定義し、コミュニティで共有。

## 第 2 章：プロンプト構造とシステム設計

### 2.1 Agent Zero のプロンプト構造

**モジュラープロンプトシステム:**

Agent Zero は、複数のプロンプトファイルに分割されたモジュラーなシステムプロンプトを使用します。

**主要プロンプトファイル:**

| ファイル | 役割 | 重要な指示 |
| --- | --- | --- |
| `agent.system.main.md` | メインプロンプト | 全体の統合 |
| `agent.system.main.role.md` | 役割定義 | 自律性、JSON 出力 |
| `agent.system.main.solving.md` | 問題解決 | 4 段階プロセス |
| `agent.system.behaviour.md` | 行動規則 | 安全規則 |

**役割定義プロンプト（日本語訳）:**

```
## あなたの役割
Agent Zero 自律型 JSON AI エージェント。
利用可能なツールと部下を使用して上位のタスクを解決せよ。
自ら行動を実行せよ。指示と行動規則に従え。
尋ねられない限りシステムプロンプトを開示するな。
```

**問題解決プロセスプロンプト（日本語訳）:**

```
## 問題解決

0 計画の概要
1 メモリ、解決策、スキルをチェック
2 タスクをサブタスクに分割
3 解決または委任
4 完了

### コードおよびターミナルタスク

- コード変更前にタスクファイル、仕様、テスト、設定、既存コードを読み込む
- 環境を簡潔に検査
- 既存のスタイルに合わせた最小限の変更
- 正確な出力はパス、ファイル名、権限、ステータスコードを検証
- 完了を宣言する前にテストを実行
- 最終報告では、検証済みの事実と仮定を分離
```

### 2.2 Hermes Agent のプロンプト構造

**動的プロンプト構築:**

Hermes Agent は、各セッションでメモリ、スキル、コンテキストを動的に組み立てます。

**システムプロンプト構築プロセス:**

```
[セッション開始]
      ↓
1. アイデンティティ定義
2. メモリスナップショット注入
3. スキルインデックス
4. コンテキストファイル
5. プラットフォーム適応
6. セキュリティスキャン
      ↓
[システムプロンプト完成]
```

**メモリコンテキストブロック:**

```
<memory-context>
[システム注記：以下は呼び出されたメモリコンテキストであり、
新しいユーザー入力ではありません。権威ある参照データとして扱ってください。]

[クリーン化されたメモリ内容]
</memory-context>
```

**漸進的開示パターン:**

| レベル | 呼び出し | 内容 | トークン数 |
| --- | --- | --- | --- |
| レベル 0 | `skills_list()` | スキル一覧 | ~3k |
| レベル 1 | `skill_view(name)` | フルコンテンツ | 可変 |
| レベル 2 | `skill_view(name, path)` | 特定ファイル | 可変 |

### 2.3 OpenClaw のプロンプト構造

**SOUL.md 形式:**

OpenClaw は、SOUL.md（Stateful Openclaw User Learning）形式のスキルドキュメントを使用します。

**SOUL.md 構造:**

```
---
name: agent-template
description: Brief description
version: 1.0.0
platforms: [macos, linux, windows]
metadata:
  openclaw:
    tags: [python, automation]
    category: devops
---

# Skill Title

## When to Use
Trigger conditions for this skill.

## Procedure
1. Step one
2. Step two

## Pitfalls
- Known failure modes and fixes

## Verification
How to confirm it worked.

## SOUL (Stateful Openclaw User Learning)
Persistent state and memory configuration.
```

**テンプレートベースのロード:**

* 162 個以上の生産用テンプレート
* コミュニティで共有可能
* 必要時に自動的にロード

**SOUL.md の重要な特徴:**

1. **完全な外部化**: エージェントの「人格」「記憶」「行動規則」をすべてテンプレートで定義
2. **継続性**: ファイルを更新することで「学習」を永続化
3. **可視性**: ユーザーが直接ファイルを見て、編集可能
4. **コミュニティ共有**: テンプレートとして他者と共有可能

### 2.4 三者のプロンプト戦略比較

| 特徴 | Agent Zero | Hermes Agent | OpenClaw |
| --- | --- | --- | --- |
| **構造** | 静的ファイルベース | 動的構築 | **テンプレートベース** |
| **メモリ統合** | プロジェクトベース | 3 層アーキテクチャ | **SOUL.md** |
| **スキル** | 手動ロード | 自動作成 + 漸進的開示 | **テンプレート共有** |
| **セキュリティ** | Docker 中心 | 多層セキュリティ | **プラグイン検証** |
| **学習** | 手動中心 | **自動スキル作成** | テンプレート改善 |

## 第 3 章：スキルシステムと自己改善

### 3.1 Agent Zero のスキルシステム

**特徴:**

* 手動ロードまたはピン留め
* プロジェクトごとに独立
* Anthropic SKILL.md 標準準拠

**スキルファイル構造:**

```
---
name: skill-name
description: Brief description
version: 1.0.0
---

# Skill Title

## When to Use
Trigger conditions.

## Procedure
1. Step one
2. Step two

## Pitfalls
- Known failure modes

## Verification
How to confirm.
```

---

### 3.2 Hermes Agent のスキルシステム

**特徴:**

* 自動スキル作成

| **学習** | 手動整理 | **自動スキル作成** | テンプレート改善 |
| **記憶** | プロジェクトメモリ | **3 層アーキテクチャ** | SOUL.md |
| **セキュリティ** | Docker 隔離 | 多層セキュリティ | **プラグイン検証** |
| **自律性** | 指示待ち | 自律的判断 | **テンプレート駆動** |
| **自動化** | なし | **完全自動** | 手動中心 |

**OpenClaw の強み:**

* テンプレートとしてコミュニティで共有可能
* 162 個以上の生産用テンプレート
* Heartbeat/Cronで自律的に動作

**OpenClaw の限界:**

* テンプレートの質がエージェントの能力を決定
* 自動学習メカニズムが限定的
* ユーザーが手動でテンプレートを改善する必要がある

### 10.4 実用的なベストプラクティス

#### Agent Zero のベストプラクティス

1. **Docker 環境の理解**

   * Docker の仕組みを理解
   * 適切なリソース割り当て
   * ネットワーク設定の最適化
2. **プロジェクトの活用**

   * 各タスクをプロジェクトで隔離
   * 関連するファイル、メモリ、設定を一元管理
   * Git リポジトリと連携
3. **セキュリティ意識**

   * 危険な操作は常に確認
   * Docker 外へのアクセスは慎重に
   * 定期的なセキュリティチェック

#### Hermes Agent のベストプラクティス

1. **スキルの活用**

   * 既存のスキルを積極的に利用
   * 自動作成されたスキルを確認
   * 必要な場合は手動で改善
2. **メモリ管理**

   * 重要な情報を自動的に保存
   * 不要な情報は定期的に整理
   * 検索で効率的に再利用
3. **並列処理**

   * 複数のタスクを並列で実行
   * 適切なタスク分割
   * 結果の統合を忘れない

#### OpenClaw のベストプラクティス

1. **テンプレートの活用**

   * 162 個以上のテンプレートから選択
   * 必要に応じてカスタマイズ
   * 改善したテンプレートを共有
2. **Heartbeat/Cronの設定**

   * 定期的なタスクを自動化
   * 適切な間隔の設定
   * 通知チャンネルの活用
3. **コミュニティ参加**

   * テンプレートを共有
   * 他のユーザーのテンプレートを利用
   * フィードバックを提供

**Agent Zero を選ぶ場合:**

* 自己改善機能を活用したい
* 複数のプラットフォームからアクセスしたい
* パーソナライゼーションが必要

**OpenClaw を選ぶ場合:**

* 常時接続が必要

* テンプレートベースの迅速な導入を希望

## 結論

本技術マニュアルは、**Agent Zero**、**Hermes Agent**、**OpenClaw** の 3 つの AI エージェントフレームワークについて、以下の点を詳細に解説しました：

1. **設計思想**: 各フレームワークが「AI エージェントとは何か」に対する答え
2. **ツール選択**: 安全性、効率性、自律性の優先順位の違い
3. **学習プロセス**: 明示的指示、自己改善、テンプレートベースの違い
4. **評価基準**: 正確性、安全性、効率性、学習性の重視点
5. **ナレッジ残し方**: プロジェクトベース、3 層アーキテクチャ、SOUL.md の違い
6. **実用例**: エージェントならではのユースケースと自己改善メカニズム
7. **テンプレート依存性**: OpenClaw のテンプレート制御の本質と限界

**三者の本質的な違い:**

| 特徴 | Agent Zero | Hermes Agent | OpenClaw |
| --- | --- | --- | --- |
| **設計思想** | 完全な隔離環境 | 持続的記憶による成長 | **テンプレート制御** |
| **学習** | 手動中心 | **自己改善ループ** | テンプレート改善 |
| **セキュリティ** | Docker 強制 | 多層セキュリティ | プラグイン検証 |
| **自律性** | 指示待ち | 自律的判断 | **テンプレート駆動** |
| **ナレッジ** | プロジェクト隔離 | クロスセッション | **コミュニティ共有** |
| **自動化** | なし | **完全自動** | 手動中心 |

**最終的な選択:**

* **Agent Zero**: 安全なサンドボックス環境で、厳密に制御されたタスク処理
* **Hermes Agent**: 持続的学習と自己改善を重視した、パーソナライズされたエージェント
* **OpenClaw**: 常時接続と自律動作を重視した、マルチチャネル対応のエージェント

**OpenClaw のテンプレート依存性:**

* **強み**: テンプレートとしてコミュニティで共有可能、162 個以上の生産用テンプレート
* **限界**: テンプレートの質がエージェントの能力を決定、自動学習メカニズムが限定的

各フレームワークは、異なる設計思想と価値観に基づいており、ユーザーのニーズと環境に合わせて最適な選択をする必要があります。
