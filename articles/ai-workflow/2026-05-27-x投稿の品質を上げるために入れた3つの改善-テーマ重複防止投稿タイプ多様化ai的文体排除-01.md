# X投稿の品質を上げるために入れた3つの改善 — テーマ重複防止・投稿タイプ多様化・AI的文体排除

## この記事について

Content AgentでX投稿の下書き生成が動くようになってから、使い続けているうちに気になる問題が出てきた。同じようなテーマの投稿が続く、投稿タイプのバリエーションが少ない、生成された文章がAIっぽい。

これらを解決するために入れた改善を3つに分けて書く。動作自体はできているのに「もう一歩品質を上げる」という段階での改善なので、具体的なコードの変更点がメインになる。

**こんな人に向けて書いている：**

* LLMの生成品質を改善する実装テクニックに興味がある
* S3を使ったエージェント間のデータ共有パターンを知りたい
* LLMのプロンプトでAI的文体を排除する方法を考えている

---

## 改善1：テーマの重複を防ぐ

### 問題

最初は「今日の研究・収集データをもとにテーマを選べ」とだけ指示していた。毎日同じリサーチ情報から生成するため、AIがよく取り上げるトピック（「Claude APIのtool\_use」「LangGraphの活用」など）に偏り、1週間で似たようなテーマが繰り返された。

### 解決策

過去14日分のS3データを読んで「使用済みテーマリスト」を作り、Claudeへの指示に含める。

```
def fetch_past_themes(bucket: str, days: int = 14) -> list[str]:
    """過去N日のS3下書きデータからテーマ使用履歴を取得"""
    s3 = boto3.client("s3", region_name=REGION)
    result = []
    for i in range(1, days + 1):
        date = (datetime.now(JST) - timedelta(days=i)).strftime("%Y-%m-%d")
        key  = f"content/drafts/{date}.json"
        try:
            obj  = s3.get_object(Bucket=bucket, Key=key)
            data = json.loads(obj["Body"].read().decode("utf-8"))
            theme_name = data.get("theme", {}).get("theme", "")
            theme_type = data.get("theme", {}).get("type", "")
            if theme_name:
                result.append(f"- {date}（{theme_type}）: {theme_name}")
        except s3.exceptions.NoSuchKey:
            pass  # その日のデータがなければスキップ
        except Exception as e:
            logger.warning("S3 過去テーマ取得失敗 %s: %s", key, e)
    return result
```

これをClaudeへのプロンプトに渡す：

```
## 使用済みテーマ（過去14日間 — これらと同じテーマは選ばないこと）
- 2026-05-22（実装Tips）: Claude APIのtool_useでNotionを更新する
- 2026-05-21（仕組み解説）: Step FunctionsのwaitForTaskTokenパターン
...
```

「これらと同じテーマは選ばないこと」という禁止リストをコンテキストに入れることで、LLMが自然に回避するようになる。

## 改善2：投稿タイプのバリエーションを多様化する

### 問題

最初は曜日ごとに投稿タイプを固定していた（月: 週次目標、火: 実装Tips、水: 失敗談...など）。これだと毎週同じ曜日に同じ種類の投稿になる。

### 解決策

固定するのは月曜（週次目標）と金曜（週次報告）だけにして、残りはプールから動的に選ぶ設計に変えた。

```
FIXED_TYPES = {0: "週次目標", 4: "週次報告"}
TYPE_POOL   = [
    "実装Tips", "失敗談", "比較", "ツール紹介", "コスト公開",
    "ハマりどころ", "仕組み解説", "知らないと損する", "引用と考察", "質問企画"
]
QA_MIN_INTERVAL = 5  # 「質問企画」は5投稿以上の間隔を開ける

def select_post_type(weekday: int, past_themes: list[str]) -> str:
    if weekday in FIXED_TYPES:
        return FIXED_TYPES[weekday]

    # 過去のテーマ履歴からタイプを抽出
    used_types = [
        re.search(r'（(.+?)）', e).group(1)
        for e in past_themes
        if re.search(r'（(.+?)）', e)
    ]

    candidates = [t for t in TYPE_POOL if t not in FIXED_TYPES.values()]

    # 質問企画は最近5件以内に使われていたら除外
    if "質問企画" in used_types[:QA_MIN_INTERVAL]:
        candidates = [t for t in candidates if t != "質問企画"]

    # 最後に使われた日が遠いものを優先（LRU的な選択）
    def last_used_pos(t):
        try:
            return used_types.index(t)
        except ValueError:
            return 999  # 一度も使われていないなら最優先

    candidates.sort(key=last_used_pos, reverse=True)
    return candidates[0] if candidates else "実装Tips"
```

「一番長く使われていないものを選ぶ」ロジックで、自然なローテーションが生まれる。使われていないタイプ（`index`が見つからず999を返す）が最優先になるため、全タイプを一周するまで同じものが繰り返されにくい。

「質問企画」だけ5投稿以上の間隔という追加ルールを入れた。質問系は毎週出てくると読者が飽きるため、比較的レアな存在にしておく意図だ。

## 改善3：AI的な文体を排除する

### 問題

初期のプロンプトで生成した投稿文がいかにも「AIが書いた」感だった。「〜が重要です」「ぜひ参考にしてください」「〜のポイントは」というフレーズが頻出していた。Xに投稿するコンテンツとして、自分の言葉で書いたように見えることが重要だ。

### 解決策

generate\_drafts()のプロンプトに「トーン・文体の絶対ルール」セクションを追加した。禁止フレーズを具体的にリストアップするのがポイントだ。

```
tone_rules = """
## トーン・文体の絶対ルール（最重要）
- 実在する人間が自分の言葉で書いたように書くこと。AIが生成した感を出さない
- 物腰柔らかく、丁寧だけど堅くなく、読んでいて親しみが持てる文体
- 言葉のリズムを大切に。一文一文が自然に流れるよう、長さや語尾に変化をつける
- 内容は変えない。余計な盛りや誇張も一切しない

## 以下のAI的な定型フレーズは絶対に使わない
- 「〜のポイントは」「〜してみました」「〜してみましょう」
- 「〜が重要です」「まとめると」「ぜひ参考に」
- 「〜な方へ」「〜という方に向けて」「〜ことが大切」
- 「〜ではないでしょうか」「〜かもしれません」
"""
```

「絶対に使わない」「最重要」といった強調表現を入れることで、プロンプトのこのセクションに優先的に従うようになる効果がある。

LLMの生成品質は、禁止リストによるネガティブ制約と、「実在する人間のように」というポジティブ指示を組み合わせるのが効果的だった。「こう書いて」より「これは書くな」の方が制御しやすい部分がある。

---

## 3つの改善をまとめると

| 問題 | 解決策 | 実装のポイント |
| --- | --- | --- |
| テーマが繰り返される | 過去14日のS3履歴を禁止リストに | `NoSuchKey`例外をスキップして欠落データに対応 |
| 投稿タイプが単調 | LRU的なプール選択 + 固定ルール | `used_types.index(t)`で最後の使用位置を取得 |
| 文章がAIっぽい | 禁止フレーズリスト + 人間らしさの指示 | ネガティブ制約とポジティブ指示を組み合わせる |

---

## 改善後の変化

改善前は1週間に1〜2回「これ似たような内容を先週も見た気がする」という感覚があった。改善後は2週間以上同じテーマが出ておらず、タイプも日ごとにバラけている。

投稿文のトーンは主観的な評価になるが、「〜が重要です」「ぜひ参考にしてください」のような文末が消えた。読んでいて自分が書いたように感じる文章が生成されるようになった、という実感がある。

---

## まとめ

LLMの生成品質を上げる実装テクニックとして、禁止リストをS3の実データから動的に作る（過去テーマの除外）こと、ルールベースのローテーションで多様性を保つ（LRU的な選択）こと、プロンプトにネガティブ制約（禁止フレーズ一覧）を明示的に入れることの3つが効果的だった。

「LLMに任せるだけ」から「制約を設計してLLMの出力を絞り込む」段階に進むと、品質が安定してくる。
