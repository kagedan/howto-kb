## はじめに

前回の記事では、OpenAI / Anthropic互換APIの `base_url` を切り替えて、AI APIの接続先を小さく検証する方法を整理しました。

ただ、AI APIサービスを本番で使い始めると、接続できること自体よりも、その後の運用の方が大きな問題になることがあります。

例えば、以下のような状態です。

* 誰がどれだけAPIを使っているか分からない
* どの機能・ユーザー・顧客のトークン消費が多いか分からない
* APIキーをどこまで分けるべきか決まっていない
* モデルごとの単価差を見ずに使っている
* エラーや失敗リクエストの傾向を追えていない
* 将来、ユーザー別・チーム別・顧客別に管理したいが、最初の設計が雑になっている

この記事では、AI APIサービスを本番公開する前に確認しておきたい運用項目を、開発者向けのチェックリストとして整理します。

特定のサービスに依存しない一般的な設計メモとして読めるようにしつつ、最後にFlatkeyで確認できる範囲にも触れます。

## この記事で扱うこと

この記事では、以下を扱います。

* APIキー管理
* 利用量ログ
* トークンコスト
* モデル別の利用状況
* エラー・レイテンシ
* 本番投入前の確認チェックリスト

この記事では、未確認のプロダクト機能を前提にした説明はしません。

例えば、特定サービスで必ずクォータ制限、請求管理、チーム権限、アラート、自動課金が使える、というような前提では書きません。そうした機能は、利用するサービスの最新ドキュメントや管理画面で確認してください。

## 1. APIキーを何の単位で分けるか

AI APIを使うとき、最初は1つのAPIキーで十分に見えることがあります。

しかし、本番運用では、1つのキーにすべてを集約すると後から困りやすくなります。

よくある分け方は以下です。

| 分け方 | 向いているケース |
| --- | --- |
| 環境別 | development / staging / production を分けたい |
| 機能別 | chat / summarization / batch など用途ごとに見たい |
| ユーザー別 | ユーザーごとの利用量や停止を管理したい |
| チーム別 | 社内ツールで部署・チームごとに見たい |
| 顧客別 | 受託開発やB2B SaaSで顧客ごとに分けたい |

最初からすべてを細かく分ける必要はありません。

ただし、最低限、以下は分けておくと安全です。

* 開発環境と本番環境
* 内部テストとユーザー向け機能
* バッチ処理とリアルタイム処理

これだけでも、トラブル時の切り分けがしやすくなります。

## 2. 利用量ログで何を見るか

AI APIの運用では、単に「リクエストが成功したか」だけでは足りません。

本番後に見たくなるのは、だいたい以下です。

* いつ呼び出されたか
* どのAPIキーから呼び出されたか
* どのモデルを使ったか
* 入力トークン数
* 出力トークン数
* リクエスト数
* 成功 / 失敗
* エラー内容
* レイテンシ

さらに、アプリケーション側で以下のような情報を紐づけられると、運用しやすくなります。

* user id
* team id
* customer id
* feature name
* request id
* job id

このあたりは、APIプロバイダー側だけでは分からないこともあります。

そのため、外部APIの利用量ログと、自社アプリケーション側のログをどう紐づけるかを最初に考えておくとよいです。

## 3. トークンコストは「合計」だけで見ない

月額のAPIコストだけを見ると、原因が分かりません。

少なくとも、以下の軸で分解できるようにしておくと、改善の打ち手が見えやすくなります。

* モデル別
* 機能別
* ユーザー別
* 顧客別
* 入力トークン / 出力トークン別
* リアルタイム処理 / バッチ処理別

例えば、同じ月額コストでも、状況によって対応は変わります。

```
ケースA:
一部のユーザーだけが非常に長いプロンプトを送っている

ケースB:
全ユーザーの利用量は普通だが、単価の高いモデルを広く使っている

ケースC:
バッチ処理のリトライが増えて、失敗リクエストも含めてコストが増えている
```

合計金額だけでは、A / B / C のどれなのか分かりません。

トークンコストを見るときは、少なくとも「どこで増えているか」を分けて見られる状態にしておくのが重要です。

## 4. モデル名と用途を対応づける

AI APIサービスでは、複数のモデルを使い分けることがあります。

例えば、以下のような使い分けです。

| 用途 | モデル選定で見ること |
| --- | --- |
| チャットUI | レイテンシ、品質、ストリーミング |
| 要約 | 入力トークン上限、単価 |
| 分類 | 単価、安定性、JSON出力 |
| Agent | 複数回呼び出し時の総コスト |
| バッチ処理 | 単価、失敗時の再実行コスト |

ここで大事なのは、「この機能はなぜこのモデルを使っているのか」を後から説明できる状態にしておくことです。

モデル名をコードの中に直接散らばらせると、後から変更しづらくなります。

最低限、環境変数や設定ファイルで切り替えられるようにしておくと、検証がしやすくなります。

```
AI_CHAT_MODEL=gpt-4o-mini
AI_SUMMARY_MODEL=gpt-4o-mini
AI_BATCH_MODEL=gpt-4o-mini
```

実際に使えるモデル名やパラメータは、利用するプロバイダーのドキュメントに従ってください。

## 5. 失敗リクエストも運用コストになる

AI APIのコストを考えるとき、成功したリクエストだけに目が行きがちです。

しかし、本番運用では失敗リクエストも重要です。

例えば、以下のようなケースがあります。

* タイムアウトでリトライが増える
* レート制限に当たって再実行する
* JSONパースに失敗して再生成する
* tool callingの結果が期待と違い、再試行する
* ユーザー操作のたびに同じ処理を繰り返す

失敗そのものが課金対象になるかどうかは、プロバイダーや失敗タイミングによって異なります。

ただし、アプリケーション全体としては、失敗やリトライが増えるほど、レイテンシ・体験・コストに影響します。

そのため、少なくとも以下は記録しておくと便利です。

* status code
* error type
* retry count
* latency
* request id
* model

## 6. 本番公開前チェックリスト

AI APIを本番公開する前に、以下を確認しておくと安心です。

### APIキー

* 開発環境と本番環境のキーを分けている
* キーをコードに直書きしていない
* キーを無効化・差し替えできる
* 用途別または顧客別に分ける方針がある

### 利用量

* リクエスト数を確認できる
* 入力トークン・出力トークンを確認できる
* モデル別の利用量を確認できる
* 急に増えたときに原因を追える

### コスト

* モデルごとの単価を確認している
* 高コストになりやすい処理を把握している
* バッチ処理とリアルタイム処理を分けて見られる
* リトライや失敗の影響を見られる

### 品質・安定性

* レイテンシを測っている
* エラー率を見ている
* ストリーミング利用時の挙動を確認している
* 切り戻し手順がある

### データの扱い

* 送信してよいデータの範囲を決めている
* 個人情報や機密情報の扱いを確認している
* ログに保存してはいけない情報を決めている

## 7. 小さく始めるなら、どこから見るか

全部を最初から作り込むのは重いです。

まずは以下の3つだけでも見えるようにすると、運用の見通しがかなり良くなります。

1. APIキー
2. 利用量
3. モデル別のコスト感

この3つが見えれば、少なくとも以下の問いに答えやすくなります。

* どのキーから呼び出されているか
* どれくらい使われているか
* どのモデルがコストの中心か
* 増加の原因がユーザー数なのか、トークン量なのか、モデル選定なのか

## まとめ

AI APIサービスでは、接続できることはスタート地点です。

本番運用では、以下のような問いに答えられることが重要になります。

* 誰が使っているか
* どれくらい使っているか
* どのモデルが使われているか
* どの処理がコストを増やしているか
* 失敗やリトライが増えていないか
* 問題が起きたときに切り戻せるか

最初から大きな管理システムを作る必要はありません。

ただ、APIキー、利用量、モデル別コスト、エラー状況だけは、早い段階で見えるようにしておくと、本番後の運用がかなり楽になります。
