## はじめに — 「共通基盤 + DRY + 共通モジュール」は本当に正解か

マイクロサービス化したはずなのに、なぜか「共通基盤」の更新が全サービスのボトルネックになっていく。モジュールを DRY にした途端、ちょっとした改修にも全サービスの影響調査が必要になる。ベストプラクティスを中央で一元管理しようとしたら、誰もそれを更新したがらなくなる。

こうした「中央集権の疲弊」は、マイクロサービス時代のあるあるとして多くの組織で観測されています。

本記事の主張を先に3行で示します。

1. **中央集権的な共通基盤は、更新サイクルのミスマッチでいずれ詰まる**。低頻度・高影響と高頻度・低影響が同じ場所に同居するのは破綻の元です。
2. **補完性原理（Subsidiarity Principle）を技術設計にも適用すべき**。上位は下位ができないことだけを担い、判断と責任はエッジに寄せる。
3. **AIネイティブ時代は判断軸が変わる**。「コード重複」や「ベストプラクティス伝播遅れ」は人間工数を前提にしたデメリットであり、AIエージェントで一括対応可能な今、その重みは大幅に下がりました。

この主張を、カトリック社会教義（1891年）、Self-contained Systems（2015年）、End-to-End原則（1984年）という3つの古典的原理から読み解き、筆者が進めているプロジェクト「fixU Reboot」でのTerraform運用方針の決定を実例として紹介します。

抽象論で終わらせたくないので、実例は具体的に、ただしセンシティブな情報は一般化した形で書きます。

## きっかけ：Terraform運用方針の議論

きっかけは、9つのマイクロサービスで構成されるシステムを大阪リージョンで再構築するプロジェクトで、Terraform運用方針をどうするかという議論でした。

選択肢は大きく2軸あります。

* **軸1**: 共通基盤（infraリポジトリ）にすべてをまとめるか、各サービスリポジトリに分散させるか
* **軸2**: 共通モジュールでDRYに管理するか、各サービスで独立してTerraformを持つか

最初の直感は「infraリポジトリに全部まとめて、共通モジュールでDRYに管理すべき」でした。Terraformの教科書通りです。しかし議論を進めるうちに、この「王道パターン」が抱える構造的な問題が見えてきて、最終的には真逆の結論に至りました。

決定事項を先に表にまとめます。

| 決定事項 | 内容 |
| --- | --- |
| infraリポジトリ | 共通基盤のみ（VPC / ALBスケルトン / ECS Cluster / ECR / Route53 / IAM共通Role） |
| サービスのTerraform | 各リポジトリ内の `infra/` に独立して配置 |
| 共通モジュール | **使わない**（自由度・拡張性重視） |
| クロスstate参照 | タグベース data source（`terraform_remote_state` は使わない） |
| ALBリスナー優先度 | サービス別に数値帯を予約（例：lp=100–199、user-front=200–299） |
| Terraform backend | 既存S3+DynamoDB再利用、key=`services/{service}/{env}/terraform.tfstate` |
| IAM Task Execution Role | 共通基盤で1本（最小権限に絞る） |
| IAM Task Role | サービス別 |

なぜこうなったのか。その背景を3つの古典原理から読み解いていきます。

## 3つの古典原理 — なぜ今、読み直すのか

### 補完性原理（Subsidiarity Principle）

補完性原理のルーツは、意外にも技術ではなく社会思想にあります。

起源は1891年、ローマ教皇レオ13世が発した回勅 *Rerum Novarum*（『労働者の境遇について』）。さらに1931年、ピウス11世の回勅 *Quadragesimo Anno*（『第40年』）で明確に定式化されました。カトリック社会教義のもっとも中核的な原理のひとつです。

その定式はこう述べています。

> 下位の共同体が自力で果たしうる事柄を、上位の共同体が奪い取ってはならない。上位の役割は、下位を**助ける**（subsidium）ことにある。

この思想は20世紀に各方面へ広がりました。

* **EU基本条約 第5条**: EUが加盟国に対して行動できるのは、加盟国レベルでは目的が達成できない場合に限る、と明記
* **ドイツ連邦制**: 州（Land）の自律性と連邦政府の補完的役割
* **組織論**: Peter Drucker の分権化思想、アメーバ経営、ホラクラシー

補完性原理のエッセンスはこうまとめられます。

> **上位は下位が自力で解決できないことのみを行う。上位の存在意義は「助けること」にある。**

これをソフトウェアアーキテクチャに置き換えると、「共通基盤は、各サービスが自力で解決できないことだけを担う」という設計原則になります。ポイントは、共通基盤が「管理する」のではなく「助ける」存在である、という役割の再定義です。

### Self-contained Systems（SCS）

2015年、ドイツのコンサルティング会社 INNOQ を中心としたエンジニア集団が提唱したアーキテクチャスタイルが Self-contained Systems（SCS）です。公式サイトは [scs-architecture.org](https://scs-architecture.org/) にあります。

SCSはマイクロサービスの一種ですが、より強い自律性を求めます。定義されている9つの特徴を要約すると以下の通りです。

1. 各SCSは自律的なWebアプリケーションであり、UI・ロジック・データを所有する
2. 各SCSは単独のチームに所有される
3. SCS間の通信は非同期が望ましい。同期呼び出しがあってもUI統合レベルで行う
4. 各SCSは独自のUIを持つ（UIの統合もWebリンクやトランスクルージョンで）
5. SCSは自身のデータとロジックを他のSCSと共有しない
6. SCSは共通のビジネスコードを他のSCSと共有しない
7. 複雑なサービス呼び出しチェーンは避ける（エッジで統合）
8. チームは独立してデプロイできる
9. インフラも極力自前で持つ

マイクロサービスとの決定的な違いは、「UI・ロジック・データ・**インフラ**まで全部自前で持つ」という所有権の完全性です。単なる「機能の小分け」ではなく、**ビジネス能力の完全な垂直スライス**を目指します。

ここで重要なのは、SCS が意図的に「共通化」を拒否していることです。6番の「共通のビジネスコードを共有しない」は、DRYを重んじるエンジニアには衝撃的に映るかもしれません。しかし後述するように、これには技術的・組織的な根拠があります。

### End-to-End原則（Saltzer / Reed / Clark 1984）

補完性原理、SCSに続く3つ目の柱は、コンピュータサイエンスの世界から来ています。

1984年、Jerome Saltzer、David Reed、David Clark の3人が発表した論文 *End-to-End Arguments in System Design* は、インターネット設計の根本哲学として今なお影響を与え続けています。

原論文の主張を一言でまとめるとこうなります。

> **Intelligence at the edges, dumb network in the middle.**  
> （賢さはエッジに置き、中間のネットワークは単純に保て）

エラー訂正、暗号化、信頼性保証といった機能は、ネットワークの中間層で完全に提供しようとすると、エッジでも結局やり直す必要があり、二重の実装になってしまう。だからそれらの機能は、最終的な責任を負うエッジ（エンドポイント）に任せ、中間ネットワークは**パケットを運ぶことだけに集中すべき**だ、という論理です。

この思想は、TCP/IPアーキテクチャそのものに組み込まれ、インターネットのスケーラビリティと進化可能性を支えてきました。AWSのクラウド設計にも通じます。S3、SQS、Lambdaといったサービスが互いに知識を持たず、疎結合で連携するのは、End-to-End原則の直系の子孫です。

### 3原理の共通項

補完性原理、SCS、End-to-End原則。時代も分野も違う3つの原理は、驚くほど同じことを言っています。

* **上位は構造だけを提供し、判断と責任はエッジに寄せる**
* **下位の自律性を最大化し、上位は「助ける」役割に徹する**
* **中央集権ではなく、局所性と疎結合を重視する**

時代を超えてこの構図が繰り返し発見されるのは、おそらく**複雑系をスケールさせるための普遍的な解**だからです。生物の器官、国家の連邦制、インターネット、マイクロサービス — スケールする系はみな、この構造を持っています。

## 実例：fixU Reboot のTerraform構成

原理を紹介したところで、実際の設計に落とし込みます。

### infraリポジトリ（中央 / 共通基盤）の責務

共通基盤のリポジトリ（以下「infraリポジトリ」）が担うのは、以下のリソースだけです。

* **ネットワーク**: VPC、Subnet、Security Group のベース
* **ロードバランサ**: ALB本体と共通リスナー（L7ルーティングのスケルトン）
* **コンピュート**: ECS Cluster、ECR リポジトリ群
* **DNS**: Route53 HostedZone、ACM証明書
* **IAM**: Task Execution Role 共通1本、GitHub Actions用Role

特徴は「動かなくなったら全サービスが止まる」ほど重要度が高く、同時に**更新頻度を意図的に低く保つ**ことです。VPCやALBは一度作れば半年は触らない。この低頻度性こそが中央に置く正当化根拠になります。

補完性原理の言い方をすれば、「各サービスが自力で解決できないこと」だけを中央が担う、という役割分担です。サービス1つでVPCを作ることはできませんが、ALBのリスナールール追加はサービス1つで完結します。

### 各サービスリポジトリの `infra/`（エッジ）の責務

一方、各マイクロサービスリポジトリは `infra/` ディレクトリに自分専用のTerraformコードを持ちます。

```
fixu-lp/
├── app/                      # アプリケーションコード
├── docker/
│   ├── Dockerfile.app        # php-fpm
│   └── Dockerfile.nginx      # nginx
├── infra/
│   ├── main.tf               # provider, backend
│   ├── data.tf               # タグで共通基盤を参照
│   ├── ecs.tf                # Task Definition, Service
│   ├── alb.tf                # Target Group, Listener Rule
│   ├── logs.tf               # CloudWatch Logs
│   ├── iam.tf                # Task Role (サービス別)
│   └── variables.tf
└── .github/workflows/
    └── deploy.yml            # ECR push + ECS deploy
```

1サービスをすべて1リポジトリで完結させる。UI・ロジック・データアクセス・インフラ定義・CI/CDが同居する。まさに SCS の哲学そのものです。

「このサービスを理解するには、このリポジトリだけを見ればよい」という明確さが、認知負荷を劇的に下げます。

### クロス境界の設計

中央とエッジの間で情報をやり取りする必要があります。ここがアーキテクチャの分水嶺になります。

Terraformの世界で「別stateのリソースを参照する」方法は主に2つあります。

1. `terraform_remote_state` data source
2. タグ / 命名規則ベースの `aws_*` data source

本プロジェクトでは**後者を選択**しました。

```
# fixu-lp/infra/data.tf
data "aws_vpc" "main" {
  tags = {
    Project     = "fixu"
    Environment = var.environment
  }
}

data "aws_lb" "main" {
  tags = {
    Project     = "fixu"
    Environment = var.environment
  }
}

data "aws_lb_listener" "https" {
  load_balancer_arn = data.aws_lb.main.arn
  port              = 443
}

data "aws_ecs_cluster" "main" {
  cluster_name = "fixu-${var.environment}"
}
```

`terraform_remote_state` を使わなかった理由は、これを使うと「infraリポジトリのoutput定義」がサービスリポジトリから参照される暗黙の契約になり、infra側のリファクタリングができなくなるからです。outputひとつ変えるだけで全サービスが壊れうる。これは End-to-End 原則に反します。中央がエッジの知識を持ってしまうのです。

タグベースで参照すれば、中央と各サービスの間にあるのは「タグ規約」という薄いインターフェースだけになります。中央はタグを付けることだけを約束し、どのリソースがあるかを各サービスが**自分で発見する**。中央は賢さを持たず、ただタグというマーカーを残すだけ。まさに dumb network の思想です。

ALBリスナー優先度も同じ思想で設計しました。優先度番号を中央で一元管理するのではなく、**サービス別に数値帯を予約**しておくだけで、実際のルール定義は各サービスが自分で行います。

| サービス | 優先度帯 |
| --- | --- |
| lp | 100–199 |
| user-front | 200–299 |
| user-app | 300–399 |
| user-mypage | 400–499 |
| manage | 500–599 |
| console | 600–699 |

中央が提供するのは「このレンジを使え」という規約だけ。番号の割り当てはエッジで決まります。新しいサービスが増えたら新しい数値帯を足すだけ。共通基盤は動かしません。

## 共通モジュール化しない判断 — DRYの誘惑と5つの落とし穴

ここからが本記事でいちばん書きたかった部分です。

Terraformを書くとき、教科書は必ず「再利用可能なモジュールを作れ」と教えます。ECS Serviceのモジュール、ALB Listener Ruleのモジュール、CloudWatch Logsのモジュール。これをDRYに共通化すれば、コードは減り、ベストプラクティスは一箇所に集約されます。

しかし今回、筆者は意図的に**共通モジュール化をしない**という決断をしました。理由は以下の5つです。

### 落とし穴1: 早すぎる抽象化（Premature Abstraction）

最初の1サービス（LP）だけを見て作ったモジュールは、次のサービス（Manager）で必ず破綻します。LPはほぼ静的で軽量、Managerは動的でステートフル、ファイルアップロードあり、セッションあり、重い管理画面。この2つを同じモジュールで表現しようとすると、`variable` が10個20個と増えていき、結局**分岐まみれのモジュール**になります。

Sandi Metz の有名な格言があります。

> **"Duplication is far cheaper than the wrong abstraction."**  
> 重複は、間違った抽象よりもはるかに安い。

早すぎる抽象化は、将来のすべての変更を「モジュール側の変更」と「全利用者側の検証」の2ステップにしてしまいます。重複していれば、必要なサービスだけを変更すればよい。

### 落とし穴2: バージョン管理の地獄

共通モジュールは必ずバージョン管理されます（されなければ変更ができません）。すると各サービスは `source = "git::...ref=v1.0.0"` のように特定バージョンにピン留めする必要が出てきます。

ここから地獄が始まります。

* v1.0.0 → v1.1.0 にアップグレードしたい → **全サービスの影響調査**が必要
* あるサービスはv1.0.0、別のサービスはv1.1.0を使っている → **infraに2つのバージョンが共存**
* 古いバージョンの廃止 → **全サービスに移行期限を通知**

これはもはやマイクロサービスではなく、**内部プロダクトのバージョン管理**です。

### 落とし穴3: アーキテクチャ変更の困難さ

マイクロサービスの強みは、サービスごとに独立した技術選択ができることです。LPは nginx+php で動いていても、将来 user-line を Node.js に置き換えたいかもしれない。user-api を Go で書き直したいかもしれない。

しかし共通モジュールがあると、モジュールが前提とする技術スタックから外れる変更が急に困難になります。モジュールを拡張するか、モジュールから外れて独自実装するか、という**二者択一**を迫られる。SCSが「共通のビジネスコードを共有しない」と明言する理由はここにあります。

### 落とし穴4: インターフェース設計コスト

汎用モジュールを作るには、あらゆるユースケースを吸収するための `variable` を設計する必要があります。optional な変数、デフォルト値、validation、dynamic block、count や for\_each の組み合わせ。結局、**モジュールを書くよりモジュールのインターフェースを設計するほうに時間を使う**ことになります。

そしてそのインターフェースは、結局「最初に想定した数個のサービスのニーズ」を超えられません。

### 落とし穴5: 結合の逆転

これがいちばん深刻な落とし穴です。

共通モジュールの存在は、サービスとモジュールの間に**所有権の逆転**をもたらします。本来、サービスが自分の必要なリソースを定義するべきなのに、モジュールの仕様に合わせてサービスの設計が歪められていきます。

「このモジュールはTarget Group を自動で作るから、サービス側で Target Group を作るなら `create_target_group = false` にしないといけない」のような、**モジュールに合わせる方向の思考**が生まれた瞬間、補完性原理は破られています。

上位が下位を助けるのではなく、下位が上位に従属している状態です。

---

以上の5つの理由から、共通モジュールは作らないことにしました。代わりに「サンプル実装としての1サービス」をリファレンスとして整備し、新規サービスはそれを**コピペで開始**するスタイルを採用しています。

> コピペは負債ではなく、**選択肢の保持**である。

## IAM Resource Scope に補完性原理を適用する

補完性原理は、アーキテクチャ全体だけでなく、個別のリソース設計にも適用できます。今回いちばん手応えがあったのは、IAMポリシーの Resource スコープ設計でした。

共通のTask Execution Role にSecrets Manager の読み取り権限を追加する必要がありました。選択肢はこうです。

```
# 案A: 環境横断で読めるようにする
Resource = "arn:aws:secretsmanager:*:*:secret:fixu/*"

# 案B: 環境ごとにスコープを絞る
Resource = "arn:aws:secretsmanager:*:*:secret:fixu/${var.environment}/*"
```

案Aは楽ですが、dev 環境のタスクが prod 環境のシークレットを読める可能性を残します。案Bは、`${var.environment}` が展開されることで、環境ごとに物理的に別のポリシーが作られます。

```
resource "aws_iam_policy" "ecs_task_execution_extra" {
  name = "fixu-${var.environment}-ecs-task-execution-extra"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "ssm:GetParameters",
          "secretsmanager:GetSecretValue"
        ]
        Resource = [
          "arn:aws:ssm:*:*:parameter/fixu/${var.environment}/*",
          "arn:aws:secretsmanager:*:*:secret:fixu/${var.environment}/*"
        ]
      },
      {
        Effect = "Allow"
        Action = ["kms:Decrypt"]
        Resource = "*"
        Condition = {
          StringEquals = {
            "kms:ViaService" = [
              "ssm.${var.region}.amazonaws.com",
              "secretsmanager.${var.region}.amazonaws.com"
            ]
          }
        }
      }
    ]
  })
}
```

この設計の意味合いは単なる「IAMベストプラクティス」では終わりません。

* dev 環境のECSタスクは `fixu/dev/*` のみ読める
* stg 環境のECSタスクは `fixu/stg/*` のみ読める
* prod 環境のECSタスクは `fixu/prod/*` のみ読める
* **環境間で誤って相互アクセスする事故が物理的に不可能**

これは End-to-End 原則と補完性原理の、IAMレベルでの実装です。

* **End-to-End原則**: 各環境（エッジ）に責任を閉じ込め、中央は構造だけを提供する
* **補完性原理**: 環境別に必要な権限は環境別の抽象で完結し、上位（共通infra）はその「枠組み」だけを与える

共通infraリポジトリがやっているのは、「環境ごとにスコープが分離されるポリシーの**型**」を提供することだけ。実際のスコープ境界は、各環境のTerraform applyのタイミングで環境変数から決定されます。

こう設計しておけば、ISO27001 / SOC2 のような監査で「環境間のアクセス制御はどう保証されていますか？」と聞かれたとき、ポリシーのTerraformコードを見せるだけで答えになります。物理的に不可能な設計は、どんなドキュメントよりも強い保証を提供します。

この同じ発想は、S3バケット、CloudWatch Logs グループ、KMS Keyなどにも展開できます。

```
S3: arn:aws:s3:::fixu-${var.environment}-*
CloudWatch Logs: arn:aws:logs:*:*:log-group:/ecs/fixu-${var.environment}/*
KMS: alias/fixu-${var.environment}
```

命名規則に環境を織り込むことで、IAMポリシーのResourceセクションが自動的に環境境界を強制する — これが「補完性原理を技術で実装する」という感覚です。

## AIネイティブ時代に変わる判断軸

ここまでの議論は、実は**AIエージェントを活用する前から有効な原理**です。補完性原理は1891年、End-to-End原則は1984年、SCSは2015年。どれもAIとは無関係に確立されてきました。

しかし AIネイティブ時代は、**判断のコスト関数**を変えます。これは重要な認識です。従来「共通化すべき」という結論に至っていた判断のいくつかは、前提条件が崩れたことで結論が変わります。

### 人間工数前提のデメリットは重みが下がった

| デメリット | 人間時代のコスト | AIネイティブ時代のコスト |
| --- | --- | --- |
| コード重複 | N倍の改修工数 | AIが一括PR生成 → 1倍に近い |
| ベストプラクティス伝播遅れ | 周知会・ドキュメント・催促 | 「全リポジトリに反映して」の1コマンド |
| セキュリティパッチ全サービス適用 | 人海戦術、数日〜数週 | 一晩で全リポジトリに並列対応 |
| コーディング規約の統一 | レビューと教育の継続投資 | AIに規約を食わせて自動チェック |
| リファクタリングの一括適用 | 「手が回らない」と先送り | プロンプトで一括実行 |

かつてDRY原則がこれほど強く支持されたのは、「同じ修正をN箇所に手で当てるのは非現実的」という前提があったからです。この前提が崩れると、DRYの重みは劇的に下がります。

### 自由度・局所性のメリットは相対的に大きくなった

一方、非中央集権のメリットはAI時代にむしろ強くなります。

* **1サービスの変更が他に波及しない価値** — AIが変更を提案するとき、影響範囲が明確なほど安全に適用できる
* **AIにとってリポジトリ内完結は「文脈ウィンドウが閉じる」利点** — 共通モジュールを跨ぐ変更は、AIにも人間にも認知負荷が高い
* **技術スタック変更の自由度** — AIに「このサービスだけ Go に書き換えて」と指示するのは、共通モジュールがなければ一瞬
* **実験の容易さ** — 1サービスで新しい構成を試してダメなら捨てる、のサイクルが速く回る

### "後から共通化は容易、最初からの共通化は失敗する"

これは古くから言われてきた経験則ですが、AIネイティブ時代は**さらに極端**になります。

* **後から共通化**: 3〜5サービスで実際にパターンが現れたあと、AIに「これらの共通部分を抽出してモジュール化して」と指示すれば数時間で完了
* **最初からの共通化**: 実例が1つしかない段階で抽象を書くと、将来の要件を想像で補完することになり、ほぼ確実に間違う

つまり AI時代は、「**とりあえず重複を許容し、パターンが確立したあとで共通化する**」という Rule of Three (「3回書いたら共通化を検討する」)の判断を、より強く支持する時代になっています。

### 「小さく早く段階的に、DRYよりスピード」は精神論ではない

これらをまとめると、「小さく早く段階的に、DRYよりスピードを優先する」という方針は、単なる精神論やスタートアップ的アジリティの話ではなく、**AIネイティブ時代における判断のコスト関数の必然**だと分かります。

人間工数が支配的コストだった時代には、共通化とDRYが「正しい」最適化でした。AIがコスト関数を変えた今、自律分散と局所性が「正しい」最適化になりつつあります。原理は同じ（コストを最小化したい）で、最適解が変わっただけです。

## 関連概念の俯瞰マップ

本記事で触れた原理の周辺には、同じ方向を向いた多くの概念があります。探索の入口として紹介しておきます。

* **Federated Architecture** — 連邦型アーキテクチャ。補完性原理の直接的な実装スタイル
* **Cell-based Architecture** — AWS re:Invent で語られるセル型アーキテクチャ。障害の blast radius を局所化する
* **Choreography vs Orchestration** — サービス間協調を中央のオーケストレータで制御するか、各サービスがイベント駆動で自律的に動くか。後者が補完性原理に近い
* **Platform Engineering / Internal Developer Platform (IDP)** — 中央はプラットフォームという「土台」を提供し、各開発チームはその上で自律的に動く
* **Golden Path** — 推奨レールであって強制ではない、という考え方。中央が「助ける」役割に徹する思想
* **You build it, you run it** — 2006年のAmazon CTO Werner Vogels のインタビューで有名になった原則。作ったチームが運用まで責任を持つ
* **Two-pizza team** — チームはピザ2枚で足りる人数に収める。SCSとの親和性が高い
* **Inverse Conway Maneuver** — Conwayの法則（組織構造がアーキテクチャを決める）を逆手に取り、望ましいアーキテクチャに合わせて組織を設計する
* **Team Topologies** — Skelton & Pais の著書。Stream-aligned Team の概念は SCS の所有単位と重なる

どれも補完性原理の異なる側面を照らす概念であり、時代を問わず繰り返し再発見され続けています。

## 判断基準のまとめ — 共通化すべきか、すべきでないか

実務で使える判断フローにまとめます。

1. **そのリソースの更新頻度は？**
   * 半年に1回程度 → 中央（infraリポジトリ）へ
   * 週に1回以上 → サービス側へ
2. **サービス間で技術スタックが変わる可能性は？**
   * ある → モジュール化しない
   * ない → 3回以上同じコードを書いてから検討
3. **AIエージェントで一括対応できるか？**
   * できる → 重複してOK、DRYの重みを下げる
   * できない（レガシーな手動作業が必要） → DRYの重みを維持
4. **失敗時の blast radius はどこまで？**
   * 全サービス影響 → 中央で厳しく管理
   * 1サービスのみ → サービス側で自由に
5. **本当にすでに3回以上同じものを書いたか？**
   * No → まだ共通化しない（Rule of Three）
   * Yes → AIに「パターン抽出して共通化して」と指示

この判断軸は「共通化するな」と言っているわけではありません。\*\*「共通化の閾値を上げろ」\*\*と言っています。AIエージェントが人間工数のコストを下げた結果、以前は「1回目で共通化すべき」だったものが「3回目以降に検討」に変わるだけです。

## おわりに — 古典の再発見

補完性原理は1891年、End-to-End原則は1984年、SCSは2015年。

面白いのは、AIネイティブ時代の設計哲学として立ち上がってくる原則の多くが、**伝統的な古典の再発見**になっていることです。何か新しいパラダイムが必要だと思っていたのに、実はずっと前から語られていたことが、今こそ正しい形で実装できるようになった、という感覚です。

* 補完性原理は、100年以上前にカトリック社会思想として語られていた
* End-to-End原則は、インターネットが商用化される前に書かれていた
* SCSは、マイクロサービスがバズワード化する前に定式化されていた

これらに共通しているのは、**複雑系をスケールさせるには上位が下位を支配するのではなく助ける構造が必要**という認識です。これは生物の器官、国家の連邦制、インターネット、そしてマイクロサービス、すべてに通底する原理です。

AIネイティブ時代の設計哲学は、新しい何かを発明するというよりも、こうした古典的な原理を**コスト関数の変化に合わせて復権させる**ことに近いのかもしれません。中央集権が有利だった時代の制約が取り払われた今、私たちはようやく、先人たちが直感していた分散・自律・補完的な構造を、技術的に実現しやすい地点に立っています。

次にあなたが共通モジュールを書くかどうか迷ったら、こう自問してみてください。

> **この共通化は、上位が下位を「助けている」のか、それとも「支配している」のか？**

答えが後者なら、補完性原理がそっと首を振っているはずです。
