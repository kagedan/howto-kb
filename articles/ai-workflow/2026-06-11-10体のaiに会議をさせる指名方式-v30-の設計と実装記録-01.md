## 0. はじめに

「AI同士に会議をさせる」というアイデアは、LLMの進化とともに現実的な選択肢になってきた。しかし実際に実装すると、すぐに「全員が同時に話す」「誰も反論しない」「議論がループする」といった問題に直面する。

本稿は、Soul-Twinというプロジェクトで10体のAI（世界の指導者たち）が座談会を行う「指名方式 v3.0」の設計・実装・実験から得た知見を記録する。

---

## 1. なぜ「指名方式」が必要か

### 1-1. ラウンドロビン方式の問題

最初の実装は単純な「全員が順番に発言するラウンドロビン」だった。これには2つの問題がある。

**問題1：全員が毎ターン発言する**

10体×5ターン = 50発言。議論が深まらないまま言葉だけが積み重なる。

**問題2：反論が生まれない**

「Aが言った→Bが言った→Cが言った」という並列発言は、議論ではなくアンケート収集に近い。AとBが直接対立する構造が生まれない。

### 1-2. 指名方式の基本設計

議長（coordinator）が状況に応じて次の発言者を指名する方式。

```
議長: 「〔指名：ケネディ〕キューバ危機での判断から、AIの自律判断をどう見るか？」
ケネディ: 「不確実性の中での決断こそが人間の自由だ。AIがそれを奪うなら...」
議長: 「〔指名：チャーチル〕ケネディの言う『決断の自由』に、反論はあるか？」
チャーチル: 「決断の重さを知る者は、AIに任せる誘惑を知っている。しかし...」
```

議長が「対立軸を持つ人物」を選んで指名することで、議論が深化する。

---

## 2. Turn 0 の設計：全員発言ターン

座談会は5ターン（内部的には0〜4）構成。

\*\*Turn 0（第1ターン）\*\*は全員発言ターンとして別設計にした。

```
# Turn 0の処理フロー
1. coordinator が議題提起（最初の発言）
2. coordinator が最初の参加者を〔指名：TWIN名〕で指名
3. その参加者が発言
4. coordinator が次の参加者を指名（全員発言するまで繰り返し）
5. 全員発言完了後、coordinator が橋渡しコメント
```

**設計の肝**：Turn 0で全員に「初期ポジション」を表明させることで、Turn 1以降の指名が「この対立軸を深化させる」という目的を持てる。

---

## 3. 中間ターン（Turn 1〜final-1）の設計

中間ターンでは coordinator が自由に指名を選ぶ。

```
# 中間ターンの処理
while mid_nominee is not None:
    # 参加者が発言
    p_content = _call_participant(mid_nominee)
    
    # coordinatorが次を指名（または〔指名なし〕で終了）
    next_coord_content = _call_coordinator(
        f"残り参加者: {names_remaining}\n"
        "最も反論・新視点を加えられる参加者を〔指名：TWIN名〕で指名してください。\n"
        "必ず1人は指名すること。〔指名なし〕は使用禁止。"
    )
```

**重要な設計判断**：「発言希望者がいなければ〔指名なし〕」という仕様から\*\*「必ず1人指名すること。〔指名なし〕は使用禁止」\*\*に変更した。

理由：AIが「もう議論が出尽くした」と判断して空ターンを生成し、連続空ターン → 強制終了 → 結論なし、という問題が実際に発生したため。

---

## 4. 副議長システム

### 4-1. ループ検出

coordinator が「同じ内容を繰り返す」ループに陥ることがある。これを検出するため副議長（sub\_coordinator）を設けた。

```
def _check_coordinator_loop(prev_msgs, coordinator_id):
    """coordinatorの直近2回の発言が先頭80文字以上一致したらループと判断"""
    coord_msgs = [m.content for m in prev_msgs if m.from_twin_id == coordinator_id]
    if len(coord_msgs) < 2:
        return False
    return coord_msgs[-1][:80] == coord_msgs[-2][:80]
```

### 4-2. 議長運営評価

座談会完了後、副議長が議長の運営を5項目（指名の適切さ・進行の一貫性・テーマ忠実度・公平性・総合評価）でA〜E評価する。

これが実際の改善サイクルに役立っている。副議長が「Turn 3以降、同じ対立軸の繰り返し」と指摘すれば、次の実験でその問題を意識した設計ができる。

---

## 5. 最終ターンの「〔結論〕問題」

最も苦労した実装の一つが、最終ターンで議長が必ず「〔結論〕」で始まる発言をするという仕様だった。

### 5-1. プロンプトが効かない

```
【最終ターン — 最終ターン】
★ 発言の最初の文字は必ず「〔結論〕」でなければなりません ★
```

どれだけ強調しても、LLM（Claude Haiku）は文脈の自然な流れを優先して「〔指名：副議長AI〕」で終わってしまう。

### 5-2. 後処理での強制付与

プロンプトエンジニアリングに限界を感じ、後処理での解決に切り替えた。

```
def _clean_coordinator_summary(summary: str) -> str:
    """〔指名：〕以降を除去し、〔結論〕タグを強制付与する"""
    import re
    # 〔指名：...〕以降を除去
    cleaned = re.sub(r"〔指名[：:][^〕]*〕.*", "", summary, flags=re.DOTALL).strip()
    # 〔結論〕タグを強制付与
    if not cleaned.startswith("〔結論〕"):
        cleaned = "〔結論〕\n" + cleaned
    return cleaned
```

**教訓**：「LLMに絶対〇〇してほしい」という制約は、プロンプトよりシステム側で保証する方が確実。

---

## 6. フォールバック指名：名前部分一致

coordinator が「〔指名：ウィンストン・チャーチルAI〕」という形式ではなく「チャーチルよ。」という自然な呼びかけで指名することがある。

これを検出できないと空ターン → 強制終了になる。

```
# フォールバック：名前部分一致
if mid_nominee is None and all_regular:
    for candidate in all_regular:
        short = candidate.name.replace("AI", "").strip()  # "ウィンストン・チャーチル"
        if short and short in coord_content:
            mid_nominee = candidate
            break
```

実際のログ：

```
[roundtable] id=60 turn=2: fallback name-match nominee=ウィンストン・チャーチルAI
```

## 7. 現在の仕様（v3.0）まとめ

| 仕様 | 実装 |
| --- | --- |
| Turn 0 | 議長→全員を順番に指名→橋渡し |
| 中間ターン | 副議長ループ検知→議長が1人以上指名（〔指名なし〕禁止） |
| 最終ターン | `_clean_coordinator_summary()` で〔結論〕を強制付与 |
| フォールバック | 名前部分一致で指名を検出 |
| 副議長 | ループ検出（先頭80文字比較）・完了後に5項目評価レポート生成 |
| RAG | 副議長はデフォルト無効（過去会議の役割情報混乱を防ぐため） |

---

## 8. おわりに

実装で最も大事だったのは「LLMを信頼しすぎないこと」だった。プロンプトで「必ず〇〇してください」と指示しても、LLMは文脈の自然な流れを優先する。制御が本当に必要な部分はシステム側で保証する設計が安定する。
