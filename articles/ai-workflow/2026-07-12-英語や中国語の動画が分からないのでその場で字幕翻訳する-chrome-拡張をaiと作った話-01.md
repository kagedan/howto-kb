## はじめに

プログラミングを本格的に学んだことのない自分が、AI（Claude と Codex）と一緒に、ブラウザで再生している動画をその場で日本語字幕に翻訳する Chrome 拡張を作りました。この記事はその制作の記録です。完成品の紹介というより、「なぜ作ったか」「どこで詰まって、どう抜けたか」を残すことを目的にしています。同じように AI と何か作ってみたい人の道しるべになれば嬉しいです。

この拡張は、Chrome の横に開くサイドパネルで動きます。英語や中国語の動画を再生しながらパネルを開くと、聞こえている内容が日本語の字幕になってパネルに流れていきます。翻訳は外部の API に投げるのではなく、Chrome に内蔵された翻訳機能と、自宅の PC で動かす音声認識で、全部手元（オンデバイス）で完結します。だから外部への課金はゼロで、動画の中身をどこかのサーバーに送ることもありません。流れた字幕は、あとから Markdown（`.md`）ファイルとして保存もできます。

自分用に作った道具なので「今すぐ誰でも試せる公開版」はありませんが、そのぶんこの記事では作り方を具体的に書きます。特に、Chrome 拡張（Manifest V3）でタブの音声や字幕、常駐状態を扱おうとして踏んだ穴は、後から同じことをやる人にそのまま効くと思うので、丁寧に残します。

## 実録データ

* 制作期間: v1 はほぼ1晩で形になりました（2026-07-11）。前身にあたる試作（音声認識で字幕を出すだけの実験）はその前に作っていて、その失敗の学びを引き継いでいます
* 概算費用: ほぼ0円
  + 翻訳は Chrome 内蔵の翻訳機能、音声認識はローカルの Whisper で、外部 API は使っていません。大前提として、Claude と Codex のサブスク（各 月約3,000円）を常用していますが、この開発でそれ以外の追加費用はかかっていません
* 使った AI / 道具: Claude、Codex、faster-whisper（base モデル・CPU）、Python、WebSocket、素の JavaScript（Chrome 拡張・Manifest V3）
* 形態: Chrome にインストールする拡張機能＋自宅の PC で動く小さな音声認識サーバー（外部公開なし・接続があったときだけ起動）

## なぜ作ったか

きっかけは、英語や中国語の技術動画・解説動画を見たいのに、字幕が付いていなかったり、付いていても英語のままで頭に入ってこなかったことでした。自動翻訳字幕がある動画もありますが、無い動画も多いし、あっても手動でメニューをたどって設定するのが地味に面倒です。

自分の発信活動のテーマは「AI で、人間の面倒くささ・分からなさを少し減らす」です。今回はこの「分からなさ」――外国語の動画が分からない――を、自分のために減らしてみることにしました。ゴールは、「どの動画でも、パネルを開くだけで日本語字幕が出る」状態です。特定のサイトの機能に頼るのではなく、Chrome で再生できる動画なら汎用的に効くようにしたい、というのが最初からの狙いでした。

## 全体像 ― 字幕がある動画と、無い動画で経路を分ける

作りはじめてすぐ分かったのは、「動画にもともと字幕（キャプション）が出ているかどうか」で、取れる情報がまったく違うということでした。そこで、翻訳のもとになる文章の取り方を2通り用意して、状況に応じて切り替えることにしました。

* **字幕モード**: 動画に字幕が表示されている場合。画面に出ている字幕の文字を読み取り、Chrome 内蔵の翻訳機能で日本語にする。音声認識を挟まないので速くて正確
* **音声モード**: 字幕が無い場合。タブの音声を取り込み、自宅の PC の Whisper で文字起こしして、それを翻訳する

普段は自動で「字幕があれば字幕モード、無ければ音声モード」を選びます。翻訳の頭脳は Chrome 内蔵の翻訳機能に、音声の聞き取りはローカルの Whisper に借りて、拡張自身は「文章を集めて、翻訳に渡して、字幕として並べる」係に徹しています。

## いちばん大事にした設計判断 ― 全部を手元で完結させる

最初に決めたのが「動画の中身を外部に送らない」でした。素直に考えれば、音声や字幕を翻訳 API に投げるのが一番早くて楽です。でもそうすると、API キーの管理が必要になり、使うほどお金がかかり、そして何より「今見ている動画の内容」を外部のサーバーに送り続けることになります。

そこで、翻訳は Chrome にこの1〜2年で入った**内蔵の翻訳機能**（ページ内で完結する翻訳）を使い、音声認識は**自宅の PC で動く Whisper** を使う、と決めました。どちらも手元で動くので、動画の中身は自分の PC の外に出ません。「ほぼ0円で、キーも秘密も持たず、中身も外に出さない」という方針を、真ん中に置いて作りました。「必要のない依存（外部 API）は最初から持たない」に倒したことで、あとの設計がずいぶん軽くなりました。

## 詰まったところと、抜け方

ここからが本題です。Chrome 拡張（Manifest V3）でこういうことをやろうとすると、静かに、しかし確実にハマる穴がいくつもありました。どれも「コードは合っているのに動かない」タイプで、原因にたどり着くのに時間がかかったものです。

### 1. タブの音声を取る王道（tabCapture）が、サイドパネルでは使えない

まず音声モードで、「今開いているタブの音声を取り込む」必要がありました。Chrome にはまさにそのための `tabCapture` という仕組みがあります。ところが、この拡張のようにサイドパネルから動かす形だと、`tabCapture` が実質使えませんでした。

`tabCapture` は「ユーザーが今まさにこの拡張を操作した」という印（transient activation と呼ばれる、クリック直後の短い有効時間）が付いた呼び出しでないと許可されません。ところがサイドパネルを開く操作では、その印が音声取得の呼び出しまで届かないのです（Chrome 側の意図的な仕様で、権限を足しても解決しません）。

抜け方は、`tabCapture` をあきらめて、**画面共有の仕組み（`getDisplayMedia`）を、サイドパネル内のボタンのクリックから直接呼ぶ**ことでした。ユーザーが「このタブの音声を共有」を選ぶ、あの見慣れたダイアログです。ボタンのクリック直後なら、さっきの「操作した印」がちゃんと生きています。

```
// サイドパネル内のボタンのクリックハンドラから直接呼ぶ（操作直後＝許可が生きている）
const stream = await navigator.mediaDevices.getDisplayMedia({
  video: true,                 // 使わないのに true 必須。false にすると TypeError で弾かれる
  audio: true,
  selfBrowserSurface: "exclude",
});
```

ここには小さな罠が2つありました。ひとつは、音声だけ欲しいのに `video: true` にしないとエラーになること（映像トラックは受け取った直後に止めて捨てます）。もうひとつは、`getDisplayMedia` で取った音声は `tabCapture` と違って**元の再生がミュートされない**ので、取り込んだ音を素直にスピーカーへ繋ぎ返すと**二重再生**になってしまうことです。

```
// getDisplayMedia の音声は元の再生を止めない。
// destination へ繋ぎ返すと二重再生になるので、無音のゲイン(0)を挟んで
// 「処理はするが音は出さない」経路にする。
const source = context.createMediaStreamSource(new MediaStream([audioTrack]));
const silent = context.createGain();
silent.gain.value = 0;
source.connect(processor);
processor.connect(silent);
silent.connect(context.destination);
```

### 2. YouTube の字幕は、API で取ろうとすると「空っぽ」が返ってくる

字幕モードのために、最初は「YouTube が持っている字幕データを API（timedtext）から取ればいい」と考えていました。ところが実際にやってみると、素直にリクエストしても**中身が空のまま**返ってきました（2026年7月時点で、トークンが必須化されていて、素のリクエストは通らなくなっていました）。

そこで発想を変えて、**画面に実際に表示されている字幕の文字を、そのまま読み取る**ことにしました。YouTube の字幕は、CC ボタンを押すと画面に文字として描画されます。その文字の出入りを監視して、新しい字幕が出たら拾う、という方法です。

```
// 画面に描画された字幕(.ytp-caption-segment)の変化を監視して読み取る。
// API 経由が空を返すので、これが唯一の実用経路（CC 表示 ON が前提）。
const observer = new MutationObserver(() => {
  const text = [...document.querySelectorAll(".ytp-caption-segment")]
    .map((node) => node.textContent)
    .join(" ")
    .trim();
  if (text && text !== previous) {
    previous = text;
    emitCue(text);
  }
});
```

この方法は「ユーザーが字幕表示を ON にしている」ことが前提になります。また、画面から読んだ字幕は「何語か」の情報が付いてこないことがあるので、そこは動画のトラック情報から言語を補う、という後始末も必要でした。きれいな公式ルートが塞がっていても、画面に見えているものは読める――という逃げ道で乗り切った形です。

### 3. 拡張の「頭脳」が、30秒放っておくと勝手に死ぬ

Manifest V3 の拡張では、裏方の処理を担う service worker という仕組みが、**約30秒何もしないと Chrome に止められます**。止められると、それまで変数に覚えていたこと（例：「今どのタブを翻訳中か」）が全部消えます。

これが厄介なのは、エラーも出さずに静かに起きることでした。「さっきまで動いていたのに、少し放っておいたら、タブから来たメッセージが誰にも届かず黙って捨てられる」という形で発現して、原因に気づくのが遅れました。

抜け方は、消えたら困る状態を変数ではなく**専用の保存領域（`storage.session`）に置き、参照する前に必ず復元を待つ**ことでした。

```
// service worker はアイドル約30秒で再起動され、変数の中身は消える。
// 対象タブIDは storage.session に持ち、使う前に復元を await する。
let activeTabId = null;
const restored = chrome.storage.session.get("activeTabId")
  .then((stored) => { activeTabId ??= stored.activeTabId ?? null; });

async function relayFromContent(message, sender) {
  await restored;                       // 復元を待たずに参照すると null で取りこぼす
  if (sender.tab?.id !== activeTabId) return;
  // …中継処理…
}
```

「プログラムは動き続けている」という思い込みを捨てて、「いつ止められて記憶を失っても大丈夫な作りにする」という考え方に切り替えたのが、この一件の収穫でした。

### 4. ローカルサーバーの接続テストは、`/health` だけだと嘘をつく

音声モードで使う音声認識サーバーは、自宅の PC で動かします。無防備に開けておきたくないので、正しいトークンを持っている接続だけを受け付けるようにしました。

このサーバーがちゃんと動いているかのテストを、最初は「`/health` を叩いて 200 が返るか」で確認していました。ところがこれは**偽陽性**でした。`/health` はトークンを検証しないので、認証が壊れていても平気で 200 を返してしまい、「テストは通るのに本番でつながらない」という一番タチの悪い状態になりかけました。

抜け方は、テストを「実際の通信のやり取り（挨拶を送ったら、準備完了が返ってくるか）まで最後まで確認する」形に変えることでした。表面的な生存確認ではなく、**本当に使う経路をそのまま検証する**、という当たり前のことを徹底した形です。

### （前身の失敗）音声認識が追いつかず、字幕が歯抜けになった

実は今回の v1 の前に、一度うまくいかなかった試作があります。CPU での音声認識が動画の再生速度に追いつかず、聞き取りが間に合わなかったぶんの音声を捨ててしまって、**字幕がところどころ歯抜けになる**という失敗でした。

v1 では、音声認識のエンジンを速い実装（faster-whisper）に替え、さらに「認識が追いつかなくても、その間の音声を捨てずに順番に処理する」設計にして解消しました。速さで殴るだけでなく、「遅れても取りこぼさない」という作りにしたのが効きました。

## 常駐のさせ方 ― 使うときだけ起きる

自宅の PC はいろいろ動いていてメモリに余裕がないので、音声認識サーバーを24時間起動しっぱなしにはしたくありませんでした。

そこで、systemd のソケットアクティベーションを使い、**拡張から接続があった瞬間にサーバーが起動し、一定時間つながらなければ自分で終了する**形にしました。普段は何も動いておらず、翻訳を使いたいときに拡張がつなぎに行くと勝手に立ち上がり、しばらく使わなければ静かに消えます。

```
def systemd_activation_socket() -> socket.socket | None:
    """systemd から渡された待受ソケット(fd 3)を引き継ぐ。無ければ None（＝単体起動）。"""
    if os.environ.get("LISTEN_PID") != str(os.getpid()):
        return None
    if int(os.environ.get("LISTEN_FDS", "0")) < 1:
        return None
    return socket.socket(fileno=3)

# …接続が一定時間なければ、自分で終了して次のアクセスを待つ…
while not app.idle_exceeded(idle_exit_seconds):
    await asyncio.sleep(min(30.0, idle_exit_seconds / 2))
```

「常駐（いつでも使える）」と「省メモリ（使わないときは何も動かさない）」を両立させる、地味だけれど気に入っている工夫です。文字起こしアプリ（もじおこし）でも同じやり方を使っていて、今回はそれをローカルの音声認識サーバーにも流用しました。

## AI の使い分け

* **Claude**: 何を作って何を作らないかを詰めるラリー（相談・設計）と、この記事のように詰まりを言葉に残す整理を担当。
* **Codex**: 拡張とサーバーの実装、レビューの往復を担当。手を動かす実装を任せる相手です。今回のような「実機でしか出ない穴」は、Codex に直させてはレビューで指摘し、を何周も回して潰しました。

大まかに「相談・設計は Claude、実装・整形は Codex」という分担です。速・打やもじおこしのときと同じ組み方で進めています。

## おわりに

自分には拡張開発の知識がほとんどなく、Manifest V3 も、タブ音声の取り方も、service worker が勝手に止まることも、最初は何も知りませんでした。それでも、「外国語の動画が分からない」という自分の困りごとを起点に、AI と一緒に一つずつ穴を切り分けていったら、1晩で普段使いできる道具になりました。

この記事に書いた詰まりは、どれも AI に丸投げして一発で解けたわけではなく、「どこが原因か」を一緒に切り分けながら抜けていったものです。特に Chrome 拡張まわりの穴は、静的なテストでは出ず実機でだけ現れるものばかりで、原因の見当をつける相棒がいたのは大きかったです。

同じように「これ分からないな／面倒だな」を AI と一緒に解いてみたい人の役に立てば嬉しいです。
