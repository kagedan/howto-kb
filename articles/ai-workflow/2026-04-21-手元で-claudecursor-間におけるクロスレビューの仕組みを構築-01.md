# 課題感

自分の目で見ながら一から実装すると細部まで目を配れますが、AI Agent に実装させると、自分で修正を加える前提だとしても、どうしても細部の問題を見落としがちになるなと感じていました。

# 対策

そこで、Agent に修正を入れさせた後は、別 Agent に自動でクロスレビューをさせることにしました。  
その際、合わせてスコアリングもさせることにしました。

なお複数回ループをさせて自律的に修正をかけさせることもできたのですが、  
AI Agent がどういう指示の下で何を修正して、それに対してどういうレビューを受けたか、を見届けた上で、  
自身の判断において修正判断を下すようにしたかったため、今回ループはさせないことにしました。

# 大まかな仕組み

## Claude

```
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/cross-review.sh cursor",
            "timeout": 240,
            "statusMessage": "Cursor にレビュー依頼中..."
          }
        ]
      }
    ]
  }
}
```

<https://code.claude.com/docs/ja/hooks>

| Event | When it fires |
| --- | --- |
| Stop | When Claude finishes responding |

とあるので、Stop の発火は「回答後のタイミング」。

## Cursor

```
{
  "version": 1,
  "hooks": {
    "stop": [
      {
        "command": "hooks/cross-review.sh claude",
        "timeout": 240,
        "loop_limit": 1
      }
    ]
  }
}
```

<https://cursor.com/ja/docs/hooks>

> stop - エージェント 完了時の処理

ほぼ同じ話ですね。設定ファイルの命名は若干違いますが。

## 共通

### 前提

自分の場合は別で、Claude・Cursor に共通定義したいものは shared ディレクトリに入れてコマンドで同期できるように仕組み化しているので、共通部分は shared 配下に置いています。

### `~/shared/hooks/cross-review.sh`

シェルスクリプトの詳細は割愛しますが、やっていることは下記です。

* Claude Hooks からは引数で cursor を渡して、Cursor CLI（`agent`コマンド）にレビューさせる
* Cursor Hooks からは引数で claude を渡して、Claude CLI（`claude`コマンド）にレビューさせる
* 飽くまでレビューコメントとスコアリングの点数のみを伝えるようにして、再修正などはさせない
* Stop Hooks に引っ掛けているとコード修正をしていなくても発火してしまうので、直前の修正との間で diff がなければ再実行せずループしないようにする

シェルスクリプトは、Agent にやりたいことを伝えれば、よしなに書いてくれます。

### `~/shared/commands/review-score.md`

レビュー観点の指定・スコアリングのルールはこだわりたいポイントだったため、別でファイルを切り出して、`cross-review.sh`から読み出すようにしました。

なお、手動でも実行できるように、コマンドファイルとして定義しています。

コチラも詳細は割愛しますが、保守性・可読性の高い設計かどうかを重視したかったため、大まかには以下のような観点で点数付けさせるルールとしました。

* 要件充足：第一優先
* 堅牢性
* 保守性
  + 責務の分離ができているか
  + 早すぎる最適化がなされていないか（YAGNI原則を守れているか）など
* 可読性
  + 命名が役割・責務を明示しており、曖昧ではないか
  + 早期 Return を取り入れるなどネストを浅くできているか
  + コメントが適切か（WHAT でなく WHY を語れているか）
* セキュリティ
* パフォーマンス
* Rails：MVC それぞれの設計
  + RESTFul か
  + N+1 が発生しないか
  + Fat Controller, Fat Model 回避など

なお最後の言語・フレームワーク特有の視点は、環境が変われば変えれば良いと思っているので、特に可変的な項目として捉えています。

## 調整事項

* 上記にも記載しましたが、前回のやり取りからコードに差分あった場合のみ、レビューが走るようにしました。そうしないと Ask でやり取りしただけで毎回レビューが走ってしまうので…。
* タイムアウトの時間はプロンプトの内容とそれを受けての Agent 実行時間次第で超過するケースがあり、微調整が必要でした。
* Agent からのレビュー内容にどういう形式の文字列が返ってくるか分からないので、パースがコケるケースがあったので、ここも調整が必要でした（例：Claude では JSON の systemMessage で、Cursor では stderr に出力される、といったように）。
* クロスレビューの処理自体がコケた場合に、何も出力されないとちゃんと発火してるのかどうかすら分からないため、問題が発生した場合は次の改善に繋げるため、ちゃんとその旨を出力させるようにしました。

### オン・オフ機能

* この機能は手元に`Claude`と`Cursor`がそれぞれ使える状態でないと成り立たず、またオフにしたいタイミングもあるため、設定を追加しました。
* 各リポジトリファイル内に `.git/info/exclude` で置いている `_myname_local` ファイルの中に`cross-review-enabled`があり中身が`true`の場合のみオン。ファイルが存在しなかったり中身が`true`以外だったらオフ、という風にしました。

## まとめ

まだ運用開始したばかりですが、これから実際に運用しながら随時改善していきたいと思っています！

体感としては、毎回コード修正の度に自動でレビューを受けさせるようにしていると、Agent の意識？（コード品質）も少し改善した気がしますね。
