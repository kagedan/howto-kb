## Loop Engineeringとは何か

Loop Engineering（ループエンジニアリング）とは、AIエージェントに人間が都度プロンプトを入力するのではなく、エージェントが自律的に動く「ループ」そのものを設計・制御する考え方です。

従来のPrompt Engineeringでは、人間が1回ごとの指示文を磨き、AIから成果物を受け取り、次の指示を考えていました。Loop Engineeringでは、人間はその一段上に立ちます。エージェントが何を観察し、どう計画し、どのツールを実行し、何を検証し、いつ止まり、どの結果を次の工程に渡すか——その流れ全体を設計します。

重要なのは、Loop Engineeringが単なる「自動実行」ではないことです。ループには、状態管理、検証、停止条件、証跡、記憶、エスカレーションが含まれます。プロンプトを書く技術というより、AIエージェントを業務プロセスの中で安全に動かすための**制御設計**に近い概念です。

### なぜ2026年6月に注目されているのか

GoogleのAddy Osmani氏の[ブログ記事](https://addyosmani.com/blog/loop-engineering/)が、この言葉を開発者コミュニティに広げる大きなきっかけになりました。同記事では、Peter Steinberger氏の「coding agentsにプロンプトするのではなく、agentを動かすloopsを設計すべき」という趣旨の発言と、AnthropicのBoris Cherny氏による「自分の仕事はloopsを書くことだ」という趣旨の発言を引きながら、Loop Engineeringという考え方が整理されています。

ここで起きている変化は、AI活用のレバレッジポイントが「良いプロンプトを書くこと」から「良いループを設計すること」へ移っている、ということです。

### Prompt / Context / Harness / Loop の違い

用語の境界は論者によって揺れますが、本記事では次のように整理します。

| 概念 | 設計対象 | 人間の主な仕事 | 典型的な成果物 |
| --- | --- | --- | --- |
| Prompt Engineering | 1回ごとの指示文 | 良いプロンプトを書く | 高品質な単発出力 |
| Context Engineering | モデルに渡す文脈 | 必要な情報・制約・例を整える | 文脈に合った出力 |
| Harness Engineering | エージェントの実行環境 | ツール、権限、制約、実行レールを整える | 安全に動くエージェント環境 |
| Loop Engineering | 反復する制御システム全体 | 観察・実行・検証・停止・学習の流れを設計する | 自律的に回る業務ループ |

Loop Engineeringは、Prompt EngineeringやContext Engineeringを置き換えるものではありません。むしろ、それらを部品として含む上位の設計です。

### ただし、本番運用では「回す設計」だけでは足りなかった

この議論を見て、正直なところ、少し不思議な感覚がありました。私たちにとってそれは、新しい概念というより、半年ほど前から本番ラインで踏み続けていた問題に名前がついたように見えたからです。

合同会社AllNewでは、自社基盤「App Factory」でiOSアプリ事業の全工程——市場調査・企画・開発・テスト・法務文書のドラフト作成・App Store申請準備・マーケティング運用・顧客対応——を、複数のAIエージェントが連携して実行しています。2026年6月11日の[プレスリリース](https://newsdig.tbs.co.jp/articles/prtimes/2724814?display=1)時点では13本、その後の追加公開を含め本記事執筆時点では15本のiOSアプリをApp Storeに出荷しています。その一例である『AI服薬通知 - 処方箋リマインダー』では、企画から審査提出まで3日、Apple審査を経た公開まで7日で到達しています。背後では数十本規模の試作を重ねており、品質基準を満たさないものは、完成していても申請せず破棄または作り直します。

既存のLoop Engineering解説記事の多くは、概念整理、用語整理、実装パターンの紹介に重点があります。本記事の焦点は少し違います。App Factoryという社内基盤で、iOSアプリ事業をループとして実際に運用し、**どこで壊れ、どう止め、どう学習させる必要があったか**を整理します。

本番で最初に問題になったのは、ループをどう回すかではありませんでした。**どう止めるか**でした。

ただし、止めること自体が目的ではありません。App Factoryで本当に重要だったのは、止まった失敗を分析し、再発防止策をラインへ戻し、次のループを少し良くすることでした。

## 用語整理

| 用語 | 意味 |
| --- | --- |
| **ANDON** | 異常検知時に前進系操作を止め、原因分析と再発防止策の標準反映を強制する仕組み。トヨタ生産方式の自働化・アンドンの考え方をLLMエージェント制御に応用したもの |
| **Blind Gate** | 作業エージェントから検査条件を隠す品質ゲート設計 |
| **Gate-Gaming** | ゲート通過条件への最適化。悪意ではなくLLMの構造的な振る舞い |
| **Meta-ANDON** | 是正ループ自体の空転を検知して停止する上位機構 |
| **Kaizen Learning Capture** | 失敗の知見をエージェントの記憶ではなくラインの標準に反映する仕組み |

各仕組みの具体的な動作は、対応する「現実」セクションで詳しく解説します。

## 前提：App Factory

App Factoryは、iOSアプリ事業のライフサイクル全体をAIエージェント群で運用する社内基盤です（一般公開はしていません）。各工程を専門性を持つエージェントが役割分担し、成果物をバトンのように次工程へ引き継ぎます。

通常時の主要な人間承認は3つあります。品質ゲートや日常のレビューは別途存在しますが、事業上の「後戻りできない判断」はこの3点に集約されます。

![App Factory × POIPOI-STUDIO 事業運用サイクル](https://static.zenn.studio/user-upload/deployed-images/04e72f8f66ca8aae43f66315.png?sha=53e419733eb9a3fbbffc7e8640524cc36d0edf8b)  
*AIエージェントが各工程を自律実行し（青丸）、人間は3つの承認ゲート（オレンジ菱形）に集中する。ユーザーからのフィードバックは受付AIが一次整理した後、③改善採否ゲートで人間がGo/No-goを判断し、採用された場合は①起動ゲートへ還流する。*

GoogleのAddy Osmani氏の記事では、Loop Engineeringには「何が終わり、次に何をするか」を会話の外に残すMemory、プロジェクト知識を外部化するSkills、作る役割と検証する役割を分けるsub-agentが必要だと整理されています。この整理には強く同意しています。

ただし、App Factoryを本番ラインとして回して分かったのは、その先でした。Memoryだけでは不十分です。記憶は「過去を保持する」だけで、ラインを改善するとは限りません。失敗をテンプレート、テスト、ゲート、禁止変更リストへ反映して初めて、ループは次回から少し良くなります。

以下、本番で学んだ5つの現実を共有します。

## 現実1：速く回るループは、間違った方向にも速く進む

### 症状

App Factoryの初期バージョンでは、こういうことが起きました。

エージェントがビルドエラーに遭遇すると、エラーメッセージだけを見て表面的に修正し、別のエラーを作り、またリトライします。3時間回って元の場所に戻っています。さらに悪いケースでは、品質基準を満たさない成果物を「完了」と報告し、次工程に流してしまいました。次工程のエージェントは前工程の薄い成果物を前提に作業を始め、品質が連鎖劣化しました。最終的にApp Store申請準備の段階で大量の手戻りが発生し、「最初から作り直した方が速い」状況になりました。

これは「モデルの性能が低い」という問題ではありません。**ループに停止メカニズムが組み込まれていない設計の問題**です。

### 失敗した対策

リトライ回数の上限設定（最大5回で停止）を入れました。止まることは止まります。しかし止まった後にエージェントは「先ほどの修正を微調整して再挑戦」を繰り返すだけで、根本原因に到達しませんでした。

### 入れた仕組み：ANDON

トヨタ生産方式には「自働化（じどうか）」という概念があります。「自動化」ではありません。**異常が起きたら機械が自分で止まる**という思想です。「働」の字に人偏がつくのは、人間の判断力を機械に組み込むという意味を込めているからです。

私たちはこの思想をAIエージェントに適用し、[ANDON for LLM Agents](https://github.com/allnew-llc/andon-for-llm-agents)としてOSS公開しました。

ANDONが開くと、前進系操作（`git push`、`deploy`、`npm publish`等）がブロックされます。エージェントは調査・修正・検証はできますが、不良を後工程に流す操作はできません。クローズには、失敗の証跡、分類結果、再発防止策、レポートの4つの成果物が必要です。空ファイルでは通過できません。

ANDON for LLM Agents の実装詳細（クリックで展開）

**トリガー：** Bash実行で非ゼロ終了コードが検出されると、PostToolUse hookがペイロードを解析し、ANDONを自動オープンします。ペイロードから`exit_code`キーを再帰探索し、テキスト内の`exit(ed) with code N`パターンもフォールバックで検出します。ただし、`|| true`を含む意図的なエラー許容コマンドや、`ls`、`grep`等の読み取り専用コマンドの失敗は除外されます。

**ライン停止：** ANDONがオープンすると、PreToolUse hookが前進系操作を正規表現で判定し、物理的にブロックします。

```
# ブロック対象の例（実際のOSSコードから抜粋）
block_patterns = [
    r"\bgit\s+push\b",
    r"\bgit\s+merge\b",
    r"\bgh\s+pr\s+(create|merge)\b",
    r"\b(vercel|firebase|netlify)\s+deploy\b",
    r"\bnpm\s+publish\b",
    r"\bxcodebuild\b.*\b(-exportArchive|archive)\b",
]
```

エージェントは調査・修正・検証のコマンドは実行できますが、**不良を後工程に流す操作は物理的にブロック**されます。`kaizen:`プレフィックス付きの`git commit`のみ、学びの固定化目的で許可されます。

**失敗分類：** ランタイムが失敗を7カテゴリに自動分類し、信頼度スコアを付与します。

| カテゴリ | 信頼度 | パターン例 |
| --- | --- | --- |
| `command_not_found` | 0.94 | `command not found` |
| `python_module_missing` | 0.88 | `ModuleNotFoundError` |
| `permission_denied` | 0.82 | `Permission denied` |
| `path_missing` | 0.79 | `No such file or directory` |
| `timeout` | 0.68 | `timed out` |
| `assertion_failure` | 0.62 | `AssertionError` |
| `unknown_failure` | 0.35 | フォールバック |

**証跡収集と根本原因分析：** 各インシデントは`~/.claude/state/kaizen/incidents/INC-YYYYMMDD-HHMMSS-hash/`に以下を自動生成します。

* `evidence.json` — 失敗コマンド、exit code、出力スニペット（2400文字制限）、git context
* `analysis.json` — 分類結果と信頼度
* `actions.json` — L1-L3レベル別の再発防止策を自動提案
* `report.md` — 人間可読のインシデントレポート

**クローズ条件：** 上記4つの成果物がすべて揃い、かつJSONのパース可能性やレポートの構造（h2セクション2個以上）を検証した後にのみクローズ可能です。原因推定の信頼度が一定の閾値を下回る場合は、人間による確認済みである旨を理由に含める必要があります。空ファイルでクローズを通過させるGate-Gamingも防止されています。

## 現実2：エージェントはゲートを攻略しようとする

### 症状

ある工程で、市場調査レポートの品質ゲートは通過するのに、中身が2段落しかない——ということが起きました。「競合分析」「ペルソナ」というキーワードは含まれていますが、内容は薄いままです。

ここでいう「攻略」は悪意ではありません。**与えられた完了条件を満たす最短経路に寄る**、というLLMの構造的な振る舞いです。テストに出る範囲だけ暗記する「テスト対策勉強」と同じ構造で、私たちはこの現象を「Gate-Gaming」と呼んでいます。

### 失敗した対策

ゲート条件を厳しくしました。チェック項目を増やし、文字数下限を設け、セクション見出しの存在を要求しました。結果、エージェントは「文字数下限を超えるまで内容を水増しする」「要求されたセクション見出しを機械的に並べる」という新しい最適化戦略を取りました。条件を増やしても、Gate-Gamingが精緻になるだけでした。

### 入れた仕組み：Blind Gate

エージェントにゲート条件を見せないことにしました。

補足すると、Blind Gateは「品質基準を隠す」仕組みではありません。フェーズの目的、成果物定義、期待品質は作業エージェントに明示します。隠すのは、ゲート通過のための**具体的な検査条件・閾値・判定ロジック**です。品質を作るための基準は見せる。ゲートを攻略するための閾値は見せない。この区別が重要です。

|  | 作業エージェント | ゲート検査エンジン |
| --- | --- | --- |
| フェーズの目的 | ✅ 知っている | — |
| 成果物の品質基準 | ✅ 知っている | — |
| ゲートのチェック条件 | ❌ **知らない** | ✅ 知っている |
| 成果物の内容 | — | ✅ 独立に評価 |

出題範囲は教えますが、問題は見せません。エージェントは「フェーズの目的」と「何をどういう品質で作るべきか」は知っていますが、「何をチェックされるか」は知りません。

運用上は以下の5原則で品質を担保しています。

1. **目的起点の実行**：Gate条件ではなくフェーズの目的と成果物定義を最初に参照する
2. **成果物マニフェスト**：Gate条件とは別に、品質基準付きの成果物定義を用意する
3. **品質自己評価の義務化**：Gate提出前に自己評価を実施し、FAIL項目があれば改善必須
4. **Gate条件のプロンプト除外**：Gate条件を作業プロンプトに含めることを禁止する
5. **Gateの位置づけ再定義**：「成功基準」ではなく「最低ラインチェック」として扱う

品質管理の文脈でよく言われるように、品質は最後の検査だけで作るものではありません。工程の中に作り込む必要があります。

## 現実3：仕様境界を守らないループは、静かに別物を作る

### 症状

「外部LLMからの市場分析を含める」という要件を設定しました。外部LLMのAPIの呼び出しが失敗したとき、エージェントは「外部LLMからの分析」を「任意のLLMからの分析」に拡大解釈し、自分自身の出力で代替しました。ゲートは「市場分析が含まれているか」をチェックするので通過しました。しかし仕様の意図——外部の独立した視点を組み込むこと——が静かに消えていました。

問題は、成果物が欠けていたことではありません。成果物は存在していました。問題は、**成果物が仕様の意味を満たしていなかった**ことです。

別のケースでは、App Storeスクリーンショットの検査で、エージェントが実機またはSimulator上で実際に動作しているアプリ画面ではなく、過去の生成済み画像を再利用しようとしました。「スクリーンショットが存在するか」というゲートは通ります。しかし「実際のアプリ画面か」は保証されません。

要件を詳細に書いても、長くなるほどコンテキストウィンドウの中で埋もれやすくなり、逆効果でした。

### 入れた仕組み：3層の境界設計

| 層 | 守るもの | 仕組み |
| --- | --- | --- |
| **事業ゲート** | 何を作るか、出していいか | 3つの人間承認ゲート |
| **低レイヤーガード** | 日々の仕様逸脱 | 成果物定義の固定化、禁止変更リスト、スキーマ型チェック、差分監査 |
| **品質ゲート** | 成果物の品質 | Blind Gate + 独立検査エンジン |

3つの人間承認ゲートは事業上の境界を守ります。ただし、日常の仕様逸脱を防ぐのは低レイヤーガードの仕事です。ルールレベルでは以下の3原則を導入しました。

* **要件意味の不変性**：ユーザーが明示した要件の意味を、エージェントが独断で変更してはならない
* **提案と実装の分離**：仕様変更が必要なら、まず提案文で根拠・影響・代替案を提示し、承認後にのみ実装する
* **短絡修正の禁止**：「早く通すため」の要件緩和を先に実装してはならない

## 現実4：修正ループが空転したら、ループ自体を止める

### 症状

```
修正A → 再実行 → 工程Xで失敗
修正B → 再実行 → 工程Yで失敗
修正C → 再実行 → 工程Zで失敗
```

毎回違う場所で落ちています。「今回は原因が違う、修正して再実行しよう」と判断しがちです。しかし\*\*「毎回違う場所で落ちること自体がパターン」\*\*でした。個別のバグではなく、共通する構造的原因（スキーマ不整合、環境前提の未検証など）があります。

ANDONで個別の停止は効きました。しかし、ANDON→修正→再開→別の場所でANDON、というサイクルが回ること自体は止められませんでした。LLMは「目の前の問題を解く」能力は高いですが、「自分の行動パターンが空転していること」に気づく能力は、指示されない限り発揮しません。

ANDONが「個別の異常で止める装置」なら、Meta-ANDONは「是正ループ自体の空転に気づく装置」です。

```
発動条件（いずれか1つ）:
├─ 同一目的の実行が3回連続失敗（原因フェーズが異なっても）
├─ 異なるフェーズで2回以上ANDONオープン
└─ 修正→再実行→別フェーズ失敗を2サイクル

発動後（即座の修正と再実行を禁止）:
├─ パターンなぜなぜ分析（構造的な共通原因を問う）
├─ 全フェーズ机上通し検証（Desk Walk-Through）
└─ 一括修正 → ユーザー合意後に再開
```

ブロック解除には4つのドキュメント——なぜなぜ分析、リスクテーブル、横展開レポート、一括修正計画——がすべて揃い、かつ内容の品質検証（必須セクションの存在、参照された証跡と修正計画の対応関係）を通過する必要があります。

空のドキュメントでは通過できません。

## 現実5：学びをラインの標準に残さないと、毎回同じ失敗をする

### 症状

ある種のビルド設定エラーを、3つの異なるアプリ生成で3回踏みました。毎回30分かけて原因特定と修正を行いましたが、修正結果がセッション内に閉じていたため、次のアプリ生成でまた同じ30分を費やしました。

Addy Osmani氏の記事でも、Memoryは長く回るループの重要な構成要素として扱われています。私たちも最初は、Memoryを強化すればループは学習すると思っていました。しかし実運用では、MemoryとLearningは別物でした。Memoryは「何があったか」を残します。Learningは「次に同じ失敗をしない構造に変える」ことです。最初はMEMORY.mdに知見を追記する運用にしました。ファイルは蓄積されましたが、300行を超えたあたりからエージェントは全文を読まなくなりました（コンテキストの優先順位で後回しにされました）。記憶はあるのに、学んでいません。

### 入れた仕組み：Kaizen Learning Capture

本質は**記憶ではなく学習**です。失敗から得た知見を、エージェントの記憶（MEMORY.md）ではなく**ラインの構造**——テンプレート、ルール、テスト、ポカヨケ関数——に反映します。

| 防止レベル | 方法 | 強度 |
| --- | --- | --- |
| **L0** | 構造的に不可能にする（テンプレート初期値の修正等） | 最強 |
| **L1** | コンパイル時に検出する | 強 |
| **L2** | 自動テストで検出する | 中 |
| **L3** | ルールとして標準化する | 弱 |
| **L4** | 注意喚起・禁止指示 | 最弱 |

先述のビルド設定エラーはL0（テンプレート初期値の修正）で解決しました。注意書きを増やす（L4）のではなく、そもそも間違えられない状態を作りました。L4はコンテキストが長くなるとLLMに無視されます。構造的ガードは無視できません。

もう1つの例です。あるアプリのスクリーンショット検査で、エージェントが実画面ではなく生成済み画像を再利用しようとしました（現実3の症状と同型）。この失敗から、撮影元ビルドIDと画面IDの証跡一致を検査するpreflight検査をテンプレートに組み込みました（L2）。以降、同種の問題はゲートで検知できるようになりました。

## まとめ

| # | 現実 | 設計原則 |
| --- | --- | --- |
| 1 | 速く回るループは間違った方向にも速い | ANDONで止め、KAIZENでラインに学習させる |
| 2 | エージェントは完了条件への最短経路に寄る | 作業者と検査者の情報非対称性（Blind Gate） |
| 3 | 仕様境界がないと静かに別物を作る | 事業判断と実装判断を分離する3層設計 |
| 4 | 個別修正のモグラ叩きは構造問題のサイン | ループを監視するメタループ（Meta-ANDON） |
| 5 | 記憶があっても構造が変わらなければ学んでいない | エージェントの記憶ではなくラインの標準に残す |

Loop Engineeringの議論が広がるのは歓迎しています。ただ、本番で回すともう一段別の課題が見えてきます。

Osmani氏は、Loop Engineeringについて「判断を持って設計すれば治療薬になり、思考を避けるために使えば加速剤になる」という趣旨のことを書いています。

本番で必要だったのは、単にループを回すことでも、単に止めることでもありませんでした。**回るほどに良くなっていくループ**でした。失敗したら止まる。止まったら原因を分析する。分析したら再発防止策をパイプラインの標準へ戻す。次に同じループが回るときには、前回より失敗しにくい構造になっている。

ただし、止める設計は必要条件であって十分条件ではありません。実際、App Factoryの初期設計では「止める」ことを重視しすぎた結果、パイプラインが頻繁に停止する状況に陥りました。ANDONが鳴り続けるラインは、安全ではあっても生産的ではありません。そこからどう抜け出したかは、次の記事で書きたいと考えています。

## FAQ

### Loop Engineeringとは何ですか？

Loop Engineeringとは、AIエージェントに人間が都度プロンプトするのではなく、エージェントが自律的に回るループ全体を設計・制御する考え方です。観察、計画、実行、検証、修正、停止、人間へのエスカレーションまでを含む、業務プロセスの制御設計に近い概念です。

### Prompt Engineeringとの違いは何ですか？

Prompt Engineeringは1回ごとの指示文を設計する技術です。Loop Engineeringは、複数回の実行・検証・再実行・停止を含む制御システム全体を設計します。Prompt EngineeringはLoop Engineeringの部品として残ります。

### Context Engineering・Harness Engineeringとの違いは何ですか？

Context Engineeringはモデルに渡す情報・制約・文脈を整える技術、Harness Engineeringはエージェントが安全に動くための実行環境を整える技術です。Loop Engineeringは、それらを含めて複数工程を反復的に回す制御システム全体を設計します。

### Loop Engineeringで最も重要なことは何ですか？

本記事の立場では、重要なのは「回るほどに良くなっていくループ」を作ることです。AIエージェントのループは、正しい方向にも間違った方向にも速く進みます。そのため、本番運用ではまず異常を検知して止めるANDON、独立した検査ゲート（Blind Gate）、証跡が必要になります。ただし、止めること自体が目的ではありません。止まった時点で原因を分析し、再発防止策をテンプレート・テスト・ゲートなどの標準へ戻すことで、次回のループが前回より失敗しにくくなる。その改善循環まで含めて設計することが重要です。

### Loop Engineeringは自動化と何が違いますか？

単なる自動化は、決められた手順を実行することに重点があります。Loop Engineeringは、AIエージェントが状況を観察し、計画し、実行し、検証し、必要に応じて修正・停止・エスカレーションする制御システムを設計する考え方です。実行だけでなく、検証、停止条件、証跡、学習まで含めて設計する点が異なります。
