AIエージェントを自作しようとして、以下のような壁にぶつかったことはないでしょうか？

* 処理の安定性を求め、ノードや分岐を増やしていくうちに、システムが複雑で硬直化した\*\*スパゲッティ・ワークフロー（有向グラフ）\*\*になってしまう。
* プロンプトやモデルを少し修正しただけで、全体の連携が崩れ、予期せぬ挙動を起こす。
* AIエージェント単体の機能は作れても、それを組み合わせて大規模な業務システムへ\*\*「スケール（巨大化）」\*\*させる美しい設計が見つからない。

結論から言うと、AIエージェントの設計における最大の過ちは、**人間が「固定的な実行パス（ワークフロー）」を定義してAIをがんじがらめに縛ってしまうこと**です。

AIエージェントに真の実用性とスケーラビリティをもたらすのは、外側からのルールの強制ではありません。エージェント自身に\*\*自律した「認知プロセス」\*\*をカプセル化し、それを自己相似的（フラクタル）に結合することです。

この記事では、その設計思想を具現化した TypeScript 製の最先端 AI エージェントフレームワーク **「Synapse」** の設計思想について、抽象度の高いアプローチから解説します。

現在、バージョン 0.x のオープンベータとして、**GitHub にてすべての機能を完全無料・エージェント数無制限で Public 公開中**です！

## 1. ワークフロー（有向グラフ）型設計が内包する「限界」

従来の多くのエージェント開発（LangGraph や Dify などのノーコードツール）は、人間が「分岐・検証・リトライ」のノードを静的に線で繋ぐ **「ワークフロー型設計」** です。

このアプローチは、一見確実に見えますが、本質的な限界があります。

### 正解前提で進むため、ミスが増幅する

ワークフロー型の最大の問題は、「前のノードの出力が正しいことを前提に進んでしまう」ことです。ノードAの出力を受けてノードBが動くとき、BはAの結果を疑いません。しかし実際にはAIは普通に間違えます。「半分合っている」ような中途半端な出力がそのまま次のノードに流れると、下流に行くほどズレが大きくなり、取り返しのつかないハルシネーションを生み出します。

### 複雑性の指数関数的爆発

AIの判断ミスをカバーしようと「検証 ➔ 差し戻し」のノードを増やすほど、実行パスは指数関数的に複雑化します。成功した場合、失敗した場合、再試行、例外。それぞれにルートが必要になり、人間の認知の限界を超えたスパゲッティコードが完成します。

### AIの自律性の喪失

固定されたフローの中にAIを「ただの部品」として配置するため、想定外の入力や環境の変化に対して柔軟に軌道修正するAI本来のポテンシャルが完全に失われます。

## 2. 状態駆動（State-centric）と「自律的人格」のカプセル化

Synapse が提示する解法は、外側からフローを強制することではありません。  
1つのエージェントの中に、**「解釈/計画」➔「実行」➔「評価」という独立した認知プロセス**をカプセル化することです。

ここで、エージェントが思考するために使うのが2つの「状態（State）」です。難しいシステム的な用語を使う必要はありません。現実のオフィスで行われるコラボレーションをイメージしてみてください。

* **エージェント自身の「短期記憶（手元のメモ帳）」** (Local State):  
  各エージェントが自分専用に持っているプライベートなメモです。現在取り組んでいる自分用のタスクや、一時的な思考を保持します。
* **全員で共有する「共有ホワイトボード」** (Global State):  
  マルチエージェント環境において、全員の真ん中に置かれた大きなホワイトボードです。ここに、検証され確定した共有事実や、解決すべき共通の論点を綺麗に書き出します。

各エージェントは、最初に立てた計画に縛られることなく、「短期記憶（手元のメモ帳）」に自分なりの計画を立ててツールを実行し、その結果からエッセンスだけを取り出して、全員が見る「共有ホワイトボード」へと綺麗に書き出します。そして、**「目的を達成する情報が完全に揃う」と自己評価（ステップ3）がパスするまで、このループを自律的に何回でもぐるぐると回し続けます。**

### 💡 なぜ「1単位のエージェントへのカプセル化」が革命なのか？

「計画 ➔ 実行 ➔ 評価」というループ自体は、ReAct などの既存の概念と似て聞こえるかもしれません。しかし、\*\*「この推論ループを、外側のシステム制御（ワークフローグラフなど）ではなく『1人の自律的人格（エージェント）』の中に完全にカプセル化して閉じ込めている」\*\*という点に、Synapse の真の革新性があります。

既存の多くのフレームワーク（特にグラフ型）では、「計画を立てるノード」「実行するノード」「評価するノード」を、人間が外側から線で繋いで制御しようとします。その結果、エージェントを複数人配置して連携させようとした瞬間に、外側のネットワーク設計が複雑怪奇になり破綻します。

Synapse では、このループはあくまでエージェント単体の「脳の推論プロセス」です。  
開発者は、エージェントが裏側でどれほど泥臭く「計画 ➔ 実行 ➔ 評価」を周回しているかを一切気にする必要がありません。マルチエージェントを構築する際も、\*\*「この完結した自律思考回路を持つエージェントを新しく1人ポコッと生やすだけ」\*\*で完了します。

個々の脳の独立性が保たれ、全員の真ん中にある「共有ホワイトボード」だけを介して緩やかに、かつ時系列の文脈を保って協調するため、個別の連携管理やステート制御の手間から完全に解放されるのです。

### 生ログ完全破棄 ＆ 知見抽出のプロセス

実戦でAIが最も迷走する原因は、実行したツールの結果（長大なJSONやHTMLなど）を、生のログとして次々とプロンプトに積み上げてしまうことです。情報過多の紙クズの山に埋もれたAIは、2ステップ目にして既に溺死し、思考を迷走させます。

次の思考ステップでエージェントが読むのは、生のログ（机の上の不要な紙クズ）ではなく、**極限まで整理された「短期記憶（手元のメモ帳）」と「共有ホワイトボード」だけ**です。これにより、長大なループでもコンテキストを常にクリーンに保ち、AIが自身の人格を一貫して保ったまま自律的に思考し続けることを可能にしています。

## 3. 指示のゴミを排除する：プロンプトを徹底的に「クリーン」に保つ仕組み

AIエージェントのプロンプトを開発したことがある人なら、システムのプロンプト（指示書）が「ツールAを使った後は、ツールBを実行すること」「結果がNGならリトライすること」といった「指示のゴミ」で肥大化し、身動きが取れなくなった経験があるはずです。

Synapse は、システム的なアプローチによってプロンプトを徹底的に「クリーン」に保つ仕組みを提供します。

### 評価軸のツール側へのカプセル化

「ツールAを使った後はこう考えて…」という評価ロジックは、プロンプトではなく**ツール定義（`evaluation`）側に完全にカプセル化**して持たせます。  
この評価基準は、ツールが実際に実行された後の **「結果評価フェーズ」のコンテキストにしか一時的に注入されません**。そのため、平時に無関係なツールの指示書でプロンプトが汚れることが一切なくなります。

### 不要なツール・エージェントの「完全抹消」

Synapse は、最初の計画（1ターン目）において、AI自身が「今回のタスクの実行にはツールXは不要だな」と判断した場合、**その実行セッションのコンテキストからツールXの存在定義そのものを完全に抹消**します。  
AIから不要なツールの存在自体が完全に見えなくなるため、誤った方向に行くリスクが根底からゼロになります。

## 4. 認知プロセスの共有がもたらす「フラクタルな巨大化」

1エージェントが「解釈/計画 ➔ 実行 ➔ 評価」という自律的な認知プロセス（人格）を持っていることの真の価値は、**マルチエージェント全体でこのプロセスを「フラクタル（自己相似的）」に共有し、結合できる点**にあります。

親エージェント（Orchestrator）も、子エージェント（Specialist）も、それらを束ねる共同体（Colony）も、**すべてが全く同じ「解釈/計画 ➔ 実行 ➔ 評価」の認知プロセス**を持って動いています。

* **構造の相似性（自己相似的スケール）**:  
  個々のエージェントの認知構造と、それらを束ねたシステム全体のアーキテクチャが「相似形」になっています。そのため、どれだけ規模を巨大化させても、インターフェースの複雑さは増加せず、1つの巨大な「メタ人格」としてスケールさせることができます。
* **構造化JSONによる明確な責務分離**:  
  エージェント間の依頼や委譲は、曖昧な自然言語ではなく、厳格に構造化された JSON を使用します。これにより、マクロレベルでも推論の解釈揺れを防ぎ、自律的な分散協調システムが極めて安定して動作します。

## 5. TypeScript による Synapse の具現化

以下は、Synapse を用いて「自律的な認知プロセス（State駆動）」をエージェントに持たせる最小の実装コードです。

```
import { Network, GeminiAdapter, AgentConfig, ToolDefinition } from "@synapse-agent/framework";

// 1. 推論エンジン（LLM）の注入 (Dependency Injection対応)
const llm = new GeminiAdapter({
  apiKey: process.env.GEMINI_API_KEY,
});

// 2. エージェント（自律的人格）の設定
const agents: AgentConfig[] = [{
  name: "orchestrator",
  prompt: "高度な調査 and 分析を担うオーケストレーターです。状態駆動で自律的に計画を立て、課題を解決してください。",
  tools: ["web_search"],
}];

// 3. ツール定義と、それを自己レビューするための「評価基準」
const tools: ToolDefinition[] = [{
  name: "web_search",
  summary: "Web検索を実行します。",
  instruction: "Google等の検索エンジンを利用して最新情報を取得します。",
  schema: {
    args: [{ name: "query", type: "STRING", required: true, desc: "検索クエリ" }]
  },
  // AIが実行結果を客観評価するための基準（評価フェーズでのみAIに提示される）
  evaluation: "最新動向をカバーできているか。情報の鮮度が低い場合は、ワードを絞り込んで再検索せよ。"
}];

// 4. 実行ネットワークの構築 (関心の分離)
const network = new Network({
  llm,
  agents,
  tools,
  onToolCall: async (call, ctx) => {
    if (call.name === "web_search") {
      // 外部APIを叩くなどして結果を返却（生データはAI評価後に破棄されます）
      return { results: "..." };
    }
  }
});

// 5. 実行
await network.verifyLicense(); // オープンベータ期間中のため無制限に無料パス
network.resetStates();
const result = await network.get("orchestrator").chat("業界の最新トレンドを調査して");
console.log("最終レポート:", result.finalText);
```

## まとめ：AIエージェントの新たな標準設計へ

「プロンプトエンジニアリング」や「アドホックな有向グラフの構築」でAIの賢さに依存する開発は、もう終わりにしましょう。

エージェント自身に「解釈/計画 ➔ 実行 ➔ 評価」の自律した認知プロセスを持たせ、それらをフラクタルに共有・結合させることこそが、AIエージェントシステムを極限までシンプルに、そして無限にスケールさせるための唯一の答えです。
