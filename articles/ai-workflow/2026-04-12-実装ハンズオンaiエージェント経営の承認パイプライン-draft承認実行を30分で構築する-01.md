## はじめに — なぜ承認パイプラインが必要なのか

僕は現在、AIエージェント10部門体制で会社を経営している。開発、マーケティング、営業、法務、経理……すべてにAIエージェントを配置し、自動化率98%を達成した。

だが、ここに至るまでに**致命的な失敗**をしている。

**AIが生成した営業メールを、確認なしで自動送信してしまった。** 敬語がおかしいメールが取引先に届いた。

さらに、**AIがgit push --forceを自動実行し、ブランチが消えた。**

これらの事故を経て、僕が導入したのが「承認パイプライン」だ。**draft→承認→実行**の3ステップを挟むことで、AIに安心して仕事を委任できるようになった。

この記事では、僕が実際に運用している承認パイプラインの仕組みを、ハンズオン形式で解説する。

## 承認パイプラインの全体像

```
AIエージェント（実行者）
    ↓ draft生成
承認キュー（approval-queue.md）
    ↓ CEOが確認
承認 → 実行 / 却下 → 差し戻し
    ↓
意思決定ログ（decisions/YYYY-MM.md）に記録
```

ポイントは3つ。

1. **対外アクション（メール送信、SNS投稿、デプロイ等）は必ずdraftを経由する**
2. **内部アクション（テスト実行、分析等）は自動実行OK**
3. **すべての判断を意思決定ログに記録する**

## Step 1: 権限レベルの定義

まず、アクションを3つの権限レベルに分類する。

```
# permissions.md

## 権限レベル

### read-only（自動実行OK）
- コード分析・レポート生成
- テスト実行
- ログ収集・KPI集計

### draft（承認必須）
- メール送信
- SNS投稿
- 請求書発行
- 本番デプロイ
- git push --force ← 絶対禁止リストに入れてもいい

### execute（閾値内で自動実行）
- バグ修正（影響範囲が限定的なもの）
- ドキュメント更新
- 開発環境へのデプロイ
```

僕の実体験から言うと、**迷ったらdraftにしておけ**。自動実行にして事故るより、1回の承認クリックのほうが100倍マシだ。

## Step 2: 承認キューの実装

承認キューは、シンプルなMarkdownファイルで十分だ。

```
# approval-queue.md

## 承認待ちアイテム

| ID | 部門 | 種別 | 内容 | ドラフトパス | 作成日 |
|----|------|------|------|-------------|--------|
| AQ-001 | マーケ | SNS投稿 | 新書籍の告知ツイート | drafts/mkt/sns-001.md | 2026-04-11 |
| AQ-002 | 営業 | メール | A社への提案書送付 | drafts/sales/proposal-a.md | 2026-04-11 |
```

### キューへの追加処理

AIエージェントが対外アクションを実行しようとしたとき、直接実行せずにキューに追加する。CLAUDE.mdに以下のルールを書く。

```
## 権限制御ルール

全てのアクションは permissions.md の閾値に従う。

- **read-only:** 分析・レポート系は承認不要で自動実行
- **draft:** 対外アクション（メール送信、SNS投稿、デプロイ等）は
  必ずdraftモードで生成し、approval-queue.md に追加
- **execute:** 閾値内の内部アクションは自動実行

**重要:** 対外に影響のあるアクションを直接実行してはならない。
必ず draft → 承認 → 実行 のパイプラインを通す。
```

これだけで、AIは対外アクションを勝手に実行しなくなる。**CLAUDE.mdに書いたルールは、AIにとっての「法律」になる。**

## Step 3: 承認・却下の処理

CEOが承認キューを確認し、判断する。

### 承認時の処理

```
## /approve AQ-001 の処理フロー

1. approval-queue.md から AQ-001 を削除
2. drafts/mkt/sns-001.md の内容を実行
3. decisions/2026-04.md に記録:
   - 承認日時
   - 承認内容
   - 判断理由（任意）
```

### 却下時の処理

```
## /reject AQ-002 "提案書の価格を再検討" の処理フロー

1. approval-queue.md から AQ-002 を削除
2. decisions/2026-04.md に却下として記録
3. 営業部門に差し戻し指示を送信
4. 修正版が再度 approval-queue.md に追加される
```

## Step 4: 意思決定ログ

すべての承認・却下を月別ファイルに記録する。

```
# decisions/2026-04.md

## 2026-04-11

### [承認] AQ-001: 新書籍の告知ツイート
- **部門:** マーケティング
- **内容:** 書籍「Claude Code実践ガイド」の告知
- **判断:** 承認。内容に問題なし

### [却下] AQ-002: A社への提案書送付
- **部門:** 営業
- **理由:** 価格設定が市場相場より高い。再検討を指示
```

このログが溜まることで、**AIがCEOの判断傾向を学習する**。3ヶ月もすれば「このパターンは承認される」「これは却下されそう」という予測ができるようになる。

## Step 5: 朝の運用フロー

僕の毎朝のルーティンはこうだ。

1. Claude Codeを起動して「今日の状況は？」と聞く
2. AIが全部門のSTATE.mdを収集し、ダイジェストを生成
3. 承認待ちアイテムがあれば、内容を確認して承認/却下
4. **所要時間: 約5分**

```
📊 AI-CEO Morning Digest — 2026-04-11

## 承認待ち (2件)
- [AQ-001] マーケ: 新書籍の告知ツイート
- [AQ-002] 営業: A社への提案書送付

## 部門状態サマリ
| 部門 | ステータス | 進行中タスク |
|------|-----------|------------|
| 開発 | 🟢 正常   | API改修     |
| マーケ | 🟢 正常  | 記事3本執筆中 |
| 営業 | 🟡 注意   | 提案書差し戻し |
```

**朝の5分で全部門を把握し、承認ボタンを押すだけ。** これが承認パイプラインの真価だ。

## 実装のコツ: 絶対禁止リストを作る

承認パイプラインとは別に、**どんな状況でも実行してはいけないアクション**のリストを作っておくことを強く推奨する。

```
# 絶対禁止リスト

以下のアクションはいかなる状況でも実行してはならない:

1. git push --force（ブランチ消失の実害あり）
2. 本番DBへの直接書き込み
3. 未承認の外部API課金
4. rm -rf /（言うまでもないが念のため）
```

これは僕がgit push --forceでブランチを消した実体験から生まれたルールだ。「AIは実行に使う。判断は人間。」この原則を忘れてはいけない。

## 運用して分かったこと

### コスト感

当社の実績として、AI運用コストは月約2万円（Claude Code Max）。サーバーが月2,000円、Google Workspaceが月680円。**月5万円以下で会社が回っている。**

### 自動化率98%の内訳

* launchdジョブ17個で日次タスクを自動化
* SNS自動配信27件/日
* 記事自動公開3チャネル/日（Zenn, Qiita, note）
* 承認パイプラインのおかげで、自動化しても「暴走」しない

### 最大の学び

**「自動化は『AIの能力を制限する』ためではなく『安心して委任するための仕組み』。」**

承認パイプラインがあることで、僕はAIに大胆な仕事を任せられるようになった。メール送信も、SNS投稿も、提案書作成も。draftが来たら中身を見て、OKならボタンを押すだけ。

## まとめ

承認パイプラインの実装は以下の5ステップ。

1. **権限レベルを定義する**（read-only / draft / execute）
2. **承認キューを作る**（Markdownファイル1つでOK）
3. **承認・却下の処理フローを決める**
4. **意思決定ログを残す**
5. **朝の運用ルーティンに組み込む**

特別なツールは不要。Markdownファイルと、CLAUDE.mdへのルール記述だけで実現できる。

AIエージェント経営に興味がある方は、ぜひ試してみてほしい。
