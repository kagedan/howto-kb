## はじめに

LLMエージェントのベンチマークでは、相手役の「顧客」もLLMが演じるのが一般的です。エージェントの性能を測っているつもりでも、会話の半分はユーザー役のLLM（ユーザーシミュレータ）が作っている。では、**そのユーザー役はどこまでスコアに関与するのでしょうか**。

このテーマを調べようと思ったきっかけは、[前回の記事](https://zenn.dev/takkuhiro/articles/tau2-bench-passk-experiment)にあります。[τ²-bench](https://github.com/sierra-research/tau2-bench)というカスタマーサポート型のベンチマークでpass^k（k回連続で成功する確率）を実測したとき、失敗の分析で奇妙なことに気づきました。失敗の35%は、ユーザー役のLLMが「はい、お願いします」と承諾した直後に会話を打ち切り、エージェントが実行する間もなく終了してしまう——つまり**エージェントのせいではない失敗**だったのです。

そこで、この記事では同じτ²-benchを使って、ユーザーシミュレータのLLMを変えることでエージェントの評価スコアがどのように変化するのかを実験します。

結論は以下です。

* **ユーザーシミュレータの癖は、実際にエージェントの性能差を消しました**。Gemini Proをユーザーにすると、他のユーザーでは明確に見えるSonnet 5とHaiku 4.5の差（7〜9ポイント）が消えます。原因は「承諾直後の打ち切り」という癖と、確認を丁寧に取るエージェントの相性でした
* 「承諾直後の打ち切り」という癖は、GeminiとGPTの両ファミリーで現れました（同じファミリーでは軽量なモデルほど顕著）。現れなかったのはClaude系ユーザーだけ。ファミリーと能力の両方に依存する癖でした
* 軽量なユーザーシミュレータは「**意図の演じ損ね**」という発見しづらい失敗パターンで評価を壊します（後述）

かかった費用は合計$197.33、実行時間は約5時間です。前回と同じく、全コマンドを掲載しています。

## 実験設計

前回の設定（[τ²-bench](https://github.com/sierra-research/tau2-bench) retail、20タスク、n=8試行）を流用し、変数を1つだけ入れ替えます。τ²-benchでは、ユーザー役のLLMが「キーボードを交換したい」などと要望を伝え、エージェントがツールを使ってそれに応え、**会話終了時点のデータベースの状態がタスクの期待通りかどうか**で成否が判定されます。

| 項目 | 内容 |
| --- | --- |
| エージェント（固定） | `claude-haiku-4-5` |
| ユーザーシミュレータ（変数） | 下表の6種類 |
| タスク | retail 先頭20タスク × 8試行（seed 42〜45） |
| 成功判定 | 全条件共通（NL assertionsの判定モデルも共通） |

ユーザーシミュレータは「モデルファミリー × 能力」で6種類選びました。どちらの軸が効くのかを分離するためです。

| 条件 | ユーザーシミュレータ | 位置づけ |
| --- | --- | --- |
| Flash | gemini-2.5-flash | 軽量 × Gemini（前回と同じ） |
| Pro | gemini-2.5-pro | 強力 × Gemini |
| Haiku | claude-haiku-4-5 | 軽量 × Claude |
| Sonnet | claude-sonnet-5 | 強力 × Claude |
| GPT-4.1-mini | gpt-4.1-mini | 軽量 × GPT |
| GPT-4.1 | gpt-4.1 | 強力 × GPT |

実行コマンドは前回とほぼ同じで、`--user-llm` を差し替えるだけです。GPT系もlitellm経由でそのまま動きます（`OPENAI_API_KEY` を`.env`に追加するだけ）。

```
uv run tau2 run --domain retail \
  --agent-llm anthropic/claude-haiku-4-5-20251001 --agent-llm-args '{}' \
  --user-llm gemini/gemini-2.5-pro --user-llm-args '{}' \
  --num-tasks 20 --num-trials 8 --max-concurrency 4 \
  --save-to us-pro --verbose-logs --seed 42
```

1点だけ設定上の注意があります。Claude Sonnet 5はユーザー役でもtemperatureパラメータ非対応なので、**全条件で `--user-llm-args '{}'`（temperature指定なし）に統一**しました。このため前回のFlash条件（ユーザー側temperature 0.0）とは条件が異なり、Flash条件も取り直しています。この差の影響は後で触れます。

実験スクリプト

今回は途中でマシンをスリープさせる可能性があったため、各条件の8試行を一度に実行するのではなく、**2試行ずつの小さな実行単位（チャンク）4つに分割**し（seedは42〜45を1つずつ割り当て）、チャンク単位でチェックポイントする方式にしました。完了済みチャンクをスキップする冪等設計なので、中断しても同じスクリプトを再実行するだけで続きから走ります。

```
#!/bin/bash
set -u
cd "$(dirname "$0")/tau2-bench"

CONDITIONS=(
  "gemini/gemini-2.5-pro:us-pro"
  "anthropic/claude-haiku-4-5-20251001:us-haiku"
  "anthropic/claude-sonnet-5:us-sonnet"
  "gemini/gemini-2.5-flash:us-flash"
)

chunk_done() {
  # 評価済みシミュレーションが40件(20タスク×2試行)あれば完了とみなす
  uv run python - "$1" <<'PY'
import json
import sys
from pathlib import Path

try:
    data = json.loads(Path(sys.argv[1]).read_text())
    sims = [
        s for s in data.get("simulations", [])
        if (s.get("reward_info") or {}).get("reward") is not None
    ]
    sys.exit(0 if len(sims) >= 40 else 1)
except Exception:
    sys.exit(1)
PY
}

for CHUNK in 1 2 3 4; do
  SEED=$((41 + CHUNK))
  for COND in "${CONDITIONS[@]}"; do
    MODEL="${COND%%:*}"
    NAME="${COND##*:}"
    SAVE="${NAME}-c${CHUNK}"
    RESULTS="data/simulations/${SAVE}/results.json"
    if [ -f "$RESULTS" ] && chunk_done "$RESULTS"; then
      echo "[skip] ${SAVE} (完了済み)"
      continue
    fi
    rm -rf "data/simulations/${SAVE}"
    echo "[run ] ${SAVE} (seed ${SEED})"
    uv run tau2 run --domain retail \
      --agent-llm anthropic/claude-haiku-4-5-20251001 --agent-llm-args '{}' \
      --user-llm "$MODEL" --user-llm-args '{}' \
      --num-tasks 20 --num-trials 2 --max-concurrency 4 \
      --save-to "$SAVE" --verbose-logs --seed "$SEED" \
      > "../logs/${SAVE}.log" 2>&1
  done
done
```

ハマりどころ：tau2は `results.json` を実行中に**逐次書き込み**するため、「ファイルが存在するか」で完了判定すると、中断したチャンクを完了扱いしてしまいます。完了判定は「評価済みシミュレーションの件数」で行う必要があります（上のスクリプトの `chunk_done`）。

チャンクは「全条件の1番目 → 全条件の2番目 → …」の交互順で実行しています。途中で止まっても、全条件に部分データが残るためです。

## 結果1: ユーザーが違えば、スコアが違う

![ユーザーシミュレータ6種でのpass^k減衰カーブ（エージェントはHaiku 4.5固定）](https://static.zenn.studio/user-upload/deployed-images/c3774df98b9c3d9ddf562762.png?sha=74553b66f0b3ec116ae995526b6454ac1827b37d)

**エージェント: `Claude Haiku 4.5`（全条件で固定）**

| ユーザーシミュレータ | pass^1 | pass^8 |
| --- | --- | --- |
| Gemini Flash | 0.787 | 0.300 |
| Gemini Pro | **0.869** | 0.500 |
| Claude Haiku | **0.713** | **0.250** |
| Claude Sonnet | 0.838 | **0.550** |
| GPT-4.1-mini | 0.725 | **0.250** |
| GPT-4.1 | 0.838 | 0.450 |

エージェントは6条件とも完全に同一です。それでもpass^1は71.3%から86.9%まで開きました。この15.6ポイントという幅は、前回同じ環境で測ったエージェント間の差（Haiku 4.5 vs Sonnet 5 = 約12ポイント）に匹敵します。pass^8に至っては25.0%〜55.0%と、2.2倍の開きです。点推定で見るかぎり、「どのエージェントを評価するか」と同じかそれ以上に「誰にユーザーを演じさせるか」がスコアを動かした、ということになります。

図を見ると、6本のカーブが上下2つの群にはっきり分かれています。上の群は強力なユーザー（Gemini Pro・Claude Sonnet・GPT-4.1）、下の群は軽量なユーザー（Gemini Flash・Claude Haiku・GPT-4.1-mini）。**3ファミリーすべてで「強いユーザーほどエージェントが成功しやすい」が成り立ち、ファミリーの違いよりも能力の違いが支配的**という構図です。

ただし、この数字をそのまま鵜呑みにはできません。本実験はretailドメインの20タスクに限った小規模なもので、推定には大きな誤差が含まれます。6条件が同じ20タスクを共有するペア設計を活かし、タスク単位のブートストラップ（10,000回）で95%信頼区間を出すと：

* 最大差の Gemini Pro vs Claude Haiku（+15.6pt）は **[−0.6, +31.9]** —— 幅が広く、統計的には断定できません
* 同ファミリー内の能力差は、Gemini Pro vs Gemini Flash（+8.1pt）が **[+1.3, +15.0]**、GPT-4.1 vs GPT-4.1-mini（+11.3pt）が **[+1.3, +21.9]** で、**2ファミリーで独立に有意**でした
* 一方、Claude Sonnet vs Claude Haiku（+12.5pt）は **[−5.6, +30.6]** —— 点推定は3ファミリー中最大なのに、タスクごとのばらつきが大きく有意ではありません。「差の大きさ」と「差を言い切れるか」は別物だという実例です

信頼区間・ブートストラップの補足

**まず、表の +15.6pt は「この20タスクではこうだった」という値**です（点推定）。もし別の20タスクを選んでいたら +10ptや+20ptだったかもしれません。「真の差はだいたいどのあたりか」という**幅**を出したのが95%信頼区間で、ざっくり「真の値はこの範囲に収まっていそう」と読めます。

**なぜ同じ20タスクを6条件で共有したか（ペア設計）。** タスクには元々の難易度があります。条件ごとに違うタスクを使うと「スコア差はユーザーの違いか、タスクの引きの良し悪しか」が区別できません。同じタスクを使えばタスクごとに直接引き算でき、タスク難易度という共通ノイズが相殺されて、ユーザーシミュレータの効果だけを取り出しやすくなります。

**ブートストラップ（10,000回）は何をしている？** 本当は「実験を何度もやり直して差のブレを見る」のが理想ですが、それは高コストで不可能です。そこで手元の20タスクから**重複を許して20個ランダムに引き直す→差を計算する、を1万回繰り返し**、得られた1万個の分布の上下2.5%を切って区間とします。ここで注意したいのは、**1万回はサンプルを水増ししているのではなく、計算を安定させているだけ**という点です。情報の量は元の20タスクぶんのまま。区間が広いのは「20タスクしかない」という情報の乏しさを正直に映しています。

**数字の読み方は「0を含むか」がすべて。** 0は「差がない」を意味します。区間が0をまたぐと「差はゼロかもしれない」を否定できず（＝断定できない）、区間がまるごとプラス側にあれば「確かに差がある」（＝有意）と言えます。だから点推定が大きい Pro vs Haiku（+15.6, 区間が0をまたぐ）よりも、点推定が小さい Pro vs Flash（+8.1, 区間が0をまたがない）のほうが「確かに差がある」と言える、という逆転が起きています。差の大きさそのものより、**差とそのブレ幅のバランス**が効くわけです。

なお、ブートストラップは標準的な手法ですが、20件という小標本では区間をやや狭めに見積もりうる（本当の不確実性はもう少し大きいかもしれない）という限界もあります。本文が結論を控えめに書いているのはこのためです。

まとめると、「ユーザーシミュレータの選択は、エージェントの選択に匹敵する規模でスコアを動かしうる」ことは観測されたものの、効果の正確な大きさを確定するには20タスクでは足りません。一方、能力軸（強いユーザーほどエージェントが成功しやすい）はGeminiとGPTの2ファミリーで独立に有意だったため、この規模の実験でも比較的信頼できる傾向と言えます。

ただ、スコアの絶対値が動くこと自体は、まだ序の口です。評価の実務で本当に困るのは、次の結果でした。

## 結果2: エージェントの性能差が見えなくなる

スコア全体が一律に底上げ・底下げされるだけなら、順位は保たれるので実害は小さい。怖いのは、**エージェント間にある本物の性能差が、ユーザーの選択によって見えなくなってしまう**場合です。これを確かめるため、エージェントはHaiku 4.5とSonnet 5の2種類を用意し、各ファミリーの強力なユーザー3種（GPT-4.1 / Gemini Pro / Claude Sonnet）と組み合わせて測りました。

![エージェント2種×ユーザー3種のpass^k](https://static.zenn.studio/user-upload/deployed-images/10c5b8727db5c79b151d6eef.png?sha=3a1536d9513e0ebe41cdc38e52499b5a127a1cda)

| ユーザー | Haiku 4.5 の pass^1 | Sonnet 5 の pass^1 | エージェント差 |
| --- | --- | --- | --- |
| GPT-4.1 | 0.838 | 0.931 | **+9.4pt** |
| Claude Sonnet | 0.838 | 0.912 | **+7.5pt** |
| Gemini Pro | 0.869 | 0.844 | **−2.5pt** |

GPT-4.1やClaude Sonnetをユーザーにすると、Sonnet 5エージェントはHaiku 4.5より7〜9ポイント高く、図でも上下の群がはっきり分かれます。ところが**Gemini Proをユーザーにした瞬間、この差が消えます**（−2.5pt、ブートストラップ95%信頼区間 [−15.6, +10.6] で有意差なし）。図で言えば、Sonnet + Gemini Proのカーブだけが、Haiku群の中に落ちてくる。**どのユーザーで測るかによって、「Sonnet 5 エージェントの方が明確に良い」と「差はない」という異なる結論が出てしまう**わけです。

なお、Sonnet 5エージェントに固定して見ても、ユーザー間のスプレッドは8.7ポイント（GPT-4.1 vs Gemini Pro、95%信頼区間 [+1.9, +16.2] で有意）残っていました。強いエージェントを使えばユーザーシミュレータへの感度が消える、ということもなさそうです。

では、なぜGemini Proユーザーの下でだけエージェント差が消えるのか。答えは、失敗した会話の中身にありました。

## なぜ結論が変わるのか——3つの失敗パターン

失敗した会話（Haikuエージェントの6条件で計197件、Sonnetエージェントの3条件で計50件）を分類しました。ユーザーシミュレータが評価を壊すパターンは、1つではありませんでした。

**エージェント: `Claude Haiku 4.5`（全条件で固定）**

| ユーザーシミュレータ | 失敗数 | うち承諾直後STOP | 失敗中STOP率 |
| --- | --- | --- | --- |
| Gemini Flash | 34 | 14 | 41% |
| Gemini Pro | 21 | 8 | 38% |
| Claude Haiku | 46 | **0** | **0%** |
| Claude Sonnet | 26 | **0** | **0%** |
| GPT-4.1-mini | 44 | 11 | 25% |
| GPT-4.1 | 26 | 1 | 4% |

**失敗パターン①: 承諾直後の打ち切り:** 前回の宿題の答えです。「はい、お願いします」と同時に会話終了トークンを発行してエージェントの実行機会を奪うパターンは、Geminiユーザーの失敗の約40%を占めていました。さらに、**GPT-4.1-miniユーザーでも失敗の25%（11件）で発生**していました。一方でGPT-4.1では1件に減り、**Claude系ユーザーは強弱どちらも0件**でした。つまりこの癖は「Gemini固有」ではなく、ファミリーと能力の両方に依存します。

そしてこの癖こそが、**結果2で見た「エージェント差の消失」の正体**です。Sonnet + Gemini Pro条件の失敗25件のうち8件（32%）が、この承諾直後STOPでした。不可逆な操作の前に確認を取るという行儀の良い振る舞いをするエージェントほど、「承諾と同時に会話を切る」ユーザーに実行機会を奪われます。Gemini Proユーザーの癖が、Sonnet 5の丁寧さをちょうど打ち消す方向に働いた——**エージェントの美点が、評価ハーネスの癖によって減点に変換されていた**わけです。

**失敗パターン②: 意図の演じ損ね:** では失敗パターン①が全くなかったClaude Haikuユーザーがなぜ最下位（pass^1: 71.3%）なのか。典型例を日本語訳で示します。このタスクのユーザーは本来「クリッキースイッチ希望。クリッキー＋RGB＋フルサイズの在庫がなければ、バックライトなしでも可」という条件付きの意図を持っています。ところが——

> **ユーザー（Haiku）**: キーボードをクリッキースイッチのものに交換したいです。RGBバックライトとフルサイズはそのままで。

条件を正確に伝えないまま会話が進み、希望構成が在庫切れと判明した後、ユーザー役はタスク定義にない「白バックライト」で勝手に妥協しました。エージェントはその指示通り正確に交換を実行し、ユーザーは「完璧です！」と満足して会話は終了。**会話としては完全な成功です**。しかしDBの最終状態はタスクの期待（バックライトなし）と一致せず、失敗として記録されました。

1件だけの珍事ではありません。Haikuユーザーの失敗46件すべてを、タスク定義の期待アクション・会話・エージェントの実際の行動と突き合わせて分類しました（AIを併用）。判定基準はシンプルで、ユーザーの発言通りにエージェントが動いたのに期待とズレていたらユーザー起因、ユーザーが正しく伝えたのにエージェントの行動が誤っていたらエージェント起因としています。

結果は、**46件中40件がユーザー役の脚本逸脱**で、エージェント自身のミスは6件でした。多かったのは、タスクにない交換依頼を勝手に追加するパターンが20件、問い合わせるべき商品を取り違えるパターンが8件。つまりHaikuユーザー条件の低スコアの大部分は、**エージェントの実力を反映していない**ことになります。

これは①の「打ち切り」よりタチの悪い失敗パターンです。打ち切りはログを見ればすぐ分かりますが、意図の演じ損ねは**会話がどこも破綻していない**ため、集計値だけ見ても、ログを流し読みしても気づけません。タスク定義と突き合わせて初めて「ユーザー役が脚本を守らなかった」と分かります。

**失敗パターン③: 本物のエージェント起因:** もちろんエージェント自身のミスの場合もあります。Haikuユーザー条件で言えば46件中6件で、返品を先に処理して後続の交換を不能にする実行順序のミス、商品属性の誤案内などでした。ただし今回の実験で分かったのは、この「本物の失敗率」を測るためには、①と②のノイズをユーザーシミュレータの選定と検証で抑え込む必要がある、ということです。

## おまけ: ユーザー側のtemperature設定だけでもスコアは変わる

前回のFlash条件（ユーザー側temperature 0.0）はpass^1が73.8%、今回のFlash条件（temperature指定なし）は78.7%でした。シードも異なるため単純比較はできませんが、**ユーザーシミュレータのサンプリング設定という細部だけで、数ポイント規模の変動が起こりうる**ことを示唆しています。ベンチマークスコアの±2〜3ポイントの差を議論する前に、まず評価系の設定——ユーザーシミュレータのモデル・バージョン・サンプリング設定、成功判定モデルなど、測定対象のエージェントではなく「測る側」のパラメータ——が固定・明記されているかを疑う必要があります。

## 実務への示唆

この実験から持ち帰れることを4つにまとめます。

1. **ベンチマークスコアを読むときは、ユーザーシミュレータの構成（モデル・バージョン・サンプリング設定）を必ず確認する**。異なるユーザー設定で測られたスコア同士の比較は、エージェントの比較になっていない可能性があります
2. **自前の評価基盤では、ユーザーシミュレータを「仕様」として固定し、明記する**。API経由で最新モデルを使っていると、プロバイダ側のモデル更新で評価の基準そのものが動きます
3. **ユーザーシミュレータの「演技の忠実性」を検証する工程を持つ**。タスクの意図（特に条件付きの要望）が会話に正しく現れたかは、成功率からは見えません。失敗したTrajectoryをタスク定義と突き合わせて読む工程は、やはり省略できないようです
4. **重要なエージェント比較（モデル選定・リリース判定など）は、ユーザーシミュレータを1つに固定しただけでは足りません**。今回見たように、ユーザーの癖とエージェントの振る舞いの相性次第で差が消える（あるいは誇張される）ことがあります。結論を左右する比較は、構成の異なる複数のユーザーで測って結論が入れ替わらないかを確認するのが安全です

τ²-benchの論文は、ユーザーもツールを持つ双方向（dual-control）環境ではエージェントに推論だけでなくユーザーの誘導という課題が加わり、実際にユーザーなし環境からの移行で性能が大きく落ちる、と指摘していました。今回の実験はその手前の、もっと地味な事実を示しています。**ユーザーシミュレータは評価装置の一部であり、装置が変われば測定値が変わる**。エージェントの信頼性を測っているつもりで、実は測定器の個体差を見ているのかもしれない、という点に注意する必要があります。

## 本実験の限界

* **タスク数が20と少ない**ため、多くのスコア差の信頼区間は広めです。有意と書いた差以外は「傾向」として読んでください
* **retailドメイン1つ、ベンチマーク1つ**での観察です。telecomドメインや他のベンチマークで同じ構図になるかは未検証です
* エージェント比較は2種（Haiku 4.5 / Sonnet 5）に留まります。3種以上で順位の入れ替わり（逆転）まで起こるかは確かめていません
* 承諾直後STOPの検出は正規表現ベースで、目視でパターンの一致を確認していますが、境界的な事例の数え漏れ・数え過ぎはありえます

## まとめ

* **ユーザーシミュレータの癖は、エージェントの性能差を消す**。Gemini Proをユーザーにすると、他のユーザーでは明確なSonnet 5とHaiku 4.5の差（7〜9ポイント）が消えた。確認を丁寧に取るエージェントほど「承諾直後の打ち切り」に実行機会を奪われるため
* **スコアの絶対値も、ユーザーの選択だけで大きく動く**。エージェント固定のままpass^1は71.3%〜86.9%（最大15.6ポイント差）。ファミリーの違いよりも「強いユーザーほどスコアが高い」という能力軸が支配的だった
* **「承諾直後の打ち切り」はGeminiとGPTの両ファミリーで現れ、同じファミリーでは軽量なモデルほど多い**。Claude系ユーザーでは観測されなかった
* **軽量なユーザーは「意図の演じ損ね」で評価を壊す**。会話はどこも破綻していないのに、タスク定義とズレて失敗になるため、集計値からは見えない。Claude Haikuユーザーの失敗の87%がこれだった
* ユーザーシミュレータは評価装置の一部。実務では**構成の固定・明記・忠実性の検証**、そして**重要なエージェント比較では複数ユーザーでの確認**が、評価の再現性の前提になる
