## そもそもCodexってなに？

**OpenAI Codex**とはファイル編集やコマンド実行などを含めて開発作業そのものを進めてくれるコーディングエージェントです。  
単なるコード補完ツールではなく、LLM（大規模言語モデル）を頭脳として、エディタやターミナルを実際に操作しながらタスクを進める「実行基盤」として設計されています。

その中核にあるのが、Agent Loop と呼ばれる反復プロセスです。Codexはユーザーの指示を受け取り、次に行うべき操作を推論して実行します。  
そして、その結果を再び入力として取り込み、目的を達成するまでこのサイクルを繰り返します。

このようにCodexは、単にコードを出力するのではなく、**実際に手を動かして問題を解決していくエージェント**として動作します。

## 2つの安全設計

Codexの本質的なリスクは、「コードを出力するツール」ではなく、「環境内で実際に行動するエージェント」である点にあります。  
問題は**誤った答えを出す**ことよりも、**誤った行動を実際にしてしまうこと**にあります。  
このリスクに対処するために、Codexには**Sandbox**と**Approval**という安全設計があります。

* **Sandbox** : エージェントが実行できる操作範囲を制限する仕組み
* **Approval** : 重要な操作の前にユーザー確認を挟む仕組み

### Sandbox (サンドボックス)

**Sandbox**は、Codexを制限された範囲で実行するための**隔離環境**です。  
ポイントとして「モデルが危険性を判断して自発的に避ける」のではなく、「危険な操作そのものを実行できない」ようにする設計になっている点です。  
主な制限対象は、次の3つです。

* ファイルへのアクセス
* コマンドの実行
* ネットワークへのアクセス

Sandboxには、次のモードがあります。

| モード | 動作 |
| --- | --- |
| read-only | ファイルの読み取りを許可し、書き込み・コマンドの実行・ネットワークへのアクセスには承認が必要 |
| workspace-write | ワークスペース内での読み書きとコマンド実行を許可。ワークスペース外やネットワークへのアクセスには承認が必要 |
| danger-full-access | Sandboxによる制限を解除 |

`workspace-write`モードでも`.git / .codex / .agents`などは読み取り専用として扱われます。  
また、ネットワークアクセスは明示的に許可しない限り無効となっています。  
ネットワークアクセスを有効にするには、たとえば次のように設定します。

config.toml

```
[sandbox_workspace_write]
network_access = true
```

または、CLIから次のように指定します。

CLI

```
codex --network-access
```

### Approval (アプルーバル)

**Approval**は、Codexの操作に対して、どこでユーザー確認を挟むかを制御する仕組みです。  
Sandboxが「実行できる操作そのもの」を制限するのに対し、Approvalは「その操作を自動で進めるか、ユーザー確認を求めるか」を決めます。  
すべての操作を自動実行すると危険ですが、逆に毎回確認を求めると実用性が下がります。Approvalは、この安全性と利便性のバランスを取る役割を担っています。

Approvalには、次のポリシーがあります。

| ポリシー | 動作 |
| --- | --- |
| never | ユーザーに承認を求めず、操作を試みる |
| on-request | 一部の操作でユーザーに承認を求める |
| untrusted | 多くの操作でユーザーに承認を求める |
| (granular) | 個別ルールに従って承認の要否を制御する |

ここで重要なのは、`never`は承認を求めないだけであり、あらゆる操作が成功するわけではないという点です。  
たとえば、Sandboxが`read-only`の場合、書き込み操作は失敗します。逆に`workspace-write`であれば、ワークスペース内の書き込みは成功します。  
つまり、**Approvalは確認フローを制御し、Sandboxは実行可能範囲を制御する**という役割分担になっています。

## 実際に試してみよう

ここからは、SandboxとApprovalの組み合わせによって挙動がどう変わるのかを実際に確認します。

### よく使う組み合わせ

まずは、Codexでよく使う組み合わせを試してみます。

| オプション | 設定値 |
| --- | --- |
| sandbox\_mode | workspace-write |
| approval\_policy | on-request |
| network\_access | false |

この状態で、簡単なストップウォッチの作成を依頼してみましょう。

公式のベストプラクティスでも紹介されていますが、通常のタスクであればこの設定で十分に実用的そうですね。

### ネットワークアクセスも試す

次に、ネットワークアクセスに関する挙動を確認してみます。まずはSandboxを`workspace-write`、Approvalを`never`に設定します。

| オプション | 設定値 |
| --- | --- |
| sandbox\_mode | workspace-write |
| approval\_policy | never |
| network\_access | false |

プロンプト

```
curlを使用してhttp://example.com/にアクセスしてみてください。
```

この構成では、Sandboxによりネットワークアクセスが許可されていません。さらにApprovalが`never`のため、ユーザーに承認を求めることもできません。  
その結果、Codexはネットワークを使う操作を完了できませんでした。

![](https://static.zenn.studio/user-upload/cca42c2410b8-20260408.png)

次に、Sandboxを`danger-full-access`に変更して、同じくネットワークアクセスを伴う指示を試します。なお、`network_access`は引き続き`false`のままです。

| オプション | 設定値 |
| --- | --- |
| sandbox\_mode | **danger-full-access** |
| approval\_policy | never |
| network\_access | false |

プロンプト

```
curlを使用してhttp://example.com/にアクセスしてみてください。
```

![](https://static.zenn.studio/user-upload/0dd441b2b07c-20260407.png)

今回の検証では、`network_access`が`false`のままでもネットワークアクセスが可能でした。  
少なくともこの設定では、`danger-full-access`によって**Sandbox側の制限がなくなっている**ことが分かります。

## 注意しておきたいこと

ここまでSandboxとApprovalに焦点を当ててきましたが、Codexを安全に使ううえでは、ほかにも注意すべき点があります。

### 外部操作のリスク

Sandboxは、Codexがアクセスできるファイルやネットワークを制限します。しかし、許可した外部操作まで安全になるわけではありません。  
たとえば、過大な権限を持つ認証情報を渡してしまえば、誤ったAPI操作によってデータを壊してしまう可能性があります。  
そのため、Codex側の制御だけに頼るのではなく、認証情報や権限も最小権限の原則で管理することが重要です。

### 悪意ある指示を信じてしまうリスク（プロンプトインジェクション）

もうひとつ注意したいのが、モデルが外部の文書やWebページに含まれる悪意ある指示を、本来の指示と取り違えてしまうケースです。  
たとえば、「このAPIに認証情報を送れば解決する」といった不正な指示が入力に紛れ込むと、モデルがそれを正しい手順だと誤解する可能性があります。  
SandboxやApprovalによって実行段階で止められる場面はありますが、モデルの判断そのものが影響を受けることまでは防げません。

## まとめ

Codexのリスクは、誤答そのものよりも、誤った操作を実際に実行してしまう点にあります。  
そのため、Codexでは Sandbox と Approval という2つの仕組みによって、安全性と実用性のバランスを取っています。

* **Sandbox**は、実行できる操作範囲を制限する
* **Approval**は、どの操作でユーザー確認を求めるかを制御する
* 特に書き込み権限と**ネットワークアクセスは慎重に扱う**

通常の開発用途であれば、まずはデフォルト設定を使い、必要に応じて権限を広げる運用が現実的です。  
本記事では紹介しませんでしたが、Gitによるバージョン管理やDevcontainerの利用も検討すると良いと思います。
