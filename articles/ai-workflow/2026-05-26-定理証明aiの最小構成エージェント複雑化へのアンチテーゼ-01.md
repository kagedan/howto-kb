## 概要

数学の定理証明言語「Lean」を操るAIエージェントにおいて、複雑な既存システムと同等以上の性能を叩き出す「最小構成（Minimal）」のベースラインを提案・検証した論文です。

### 1. なぜこのテーマを選んだのか

現場でPoC（概念実証）を進める際、「とりあえず最新の複雑なアーキテクチャを組んでみよう」という罠に陥りがちです。しかし、運用コストや保守性を考えると、アーキテクチャはシンプルであればあるほど良いはずです。  
現在、AIによる数学的推論（AlphaGeometryなど）やLeanを用いた定理証明の研究は、強化学習（RL）や合成データ生成を組み合わせた重厚長大なシステムが主流になっています。そんな中、最低限の推論ループだけでどこまで戦えるのか？を愚直に検証した本論文は、実務のエージェント設計においても非常に強力な指針になると感じたためピックアップしました。

### 2. 関連調査

#### 📘 Paper

* **arXiv ID**: [2602.24273](https://arxiv.org/abs/2602.24273) (2026年2月27日)
* **タイトル**: A Minimal Agent for Automated Theorem Proving
* **著者**: Borja Requena Pozo, ら
* **概要**: Leanを用いた自動定理証明において、SOTAシステムが持つコア機能（反復的改善・ライブラリ検索・文脈管理）のみを抽出した「最小限のエージェント」を構築し、複雑なアーキテクチャと遜色ない性能を実証した研究。

#### 1️⃣ 構造化読解

* **目的**: 複雑化するAI定理証明システムに対し、公平に比較・評価するための「シンプルで強力なベースライン」を提供すること。
* **仮説**: 現在のSOTAモデルの成功の要因は、複雑なRLパイプラインではなく、「反復的なフィードバック」「記憶（Context）」「ツールアクセス（検索）」という基本要素の組み合わせに帰着するのではないか。
* **方法**:
  1. **Proposer agent**: Leanコードを生成
  2. **Review system (Compiler)**: 生成されたコードをLeanコンパイラで検証
  3. **Memory system**: エラー内容を保存し、次のイテレーションに活かす  
     という極めてシンプルなループ（Reflexionに近い構造）を構築し、様々な基盤モデルで検証。
* **結果**: PutnamBenchなどのデータセットにおいて、複雑な手法（膨大なサンプル生成やMCTSなど）を用いずとも、この最小構成エージェントが競争力のある性能を発揮。特に、シングルショットの大量生成よりも「反復的アプローチ」の方がサンプル効率とコスト効率で圧倒的に優れていた。

#### 3️⃣ 比較・検証

既存の強化学習ベースのアプローチ（例：DeepSeek-ProverやLean Copilotの高度な探索手法）と比較すると、本手法は**推論時計算量の使い方が根本的に異なります。**  
従来が「木探索で無数の候補を広げて正解を探す」アプローチだとすれば、本手法は「人間の数学者のように、コンパイルエラーを読み解いて1つの証明を粘り強く修正していく」アプローチです。結果としてAPIコール数やトークン消費量が抑えられ、エコシステムの変化（Leanのバージョンアップなど）にもプロンプトひとつで追従できる柔軟性を持っています。

### 3. 再現視点（Implementation Lens）＆ 実装アーキテクチャ

この論文の素晴らしいところは、私たちが普段使っている**LangGraph**や**Dify**などのフレームワークで即座に再現できる点です。  
論文中の3つのモジュールを、LangGraph的なフローに翻訳してみましょう。

**【実装パイプラインの概念】**

1. **`Proposer Node`**:  
   現在の問題文、これまでの失敗履歴をLLMに渡し、Leanの証明コードを生成させる。
2. **`Compiler Tool Node`**:  
   生成されたコードを抽出し、実際のローカルのLean環境（サブプロセス等）で実行する。

* 成功（Success） -> ループ終了、証明完了！🎉
* 失敗（Error） -> エラーメッセージと問題箇所をキャッチ。

3. **`Memory / Review Node`**:  
   エラー内容をフォーマットし、「なぜ間違えたのか」を要約してMemoryリストに追加する。
4. **`Conditional Edge`**:  
   最大試行回数に達していなければ `Proposer Node` に戻る。

これをPython風の擬似コードで書くと、驚くほどシンプルです。

```
# Minimal Agent for ATP (LangGraph的な擬似コード)

def proposer_node(state):
    prompt = f"""
    定理: {state['theorem']}
    過去のエラー履歴: {state['memory']}
    上記の失敗を踏まえ、Lean4で証明を書いてください。
    """
    response = llm.invoke(prompt)
    return {"current_code": response.content}

def compiler_node(state):
    code = state['current_code']
    # 外部のLeanコンパイラを実行
    result, error_msg = run_lean_compiler(code)
    
    if result == "SUCCESS":
        return {"status": "solved"}
    else:
        return {"status": "failed", "last_error": error_msg}

def memory_node(state):
    # 失敗したコードとエラー内容を記憶に蓄積
    new_memory = f"Code: {state['current_code']}\nError: {state['last_error']}"
    return {"memory": state['memory'] + [new_memory]}

# グラフの構築
# Proposer -> Compiler -> (if failed) -> Memory -> Proposer ...
```

この「コード生成 → コンパイル → エラーをプロンプトにフィードバック」というループは、定理証明だけでなく、**社内システムの自律コード生成エージェント**などにもそのまま転用可能です。

### 4. 筆者の考察

#### 💡 「引き算の設計」こそが実用化の鍵

AIエージェント界隈では、論文を通すために、意味もなく複雑なモジュールを継ぎ接ぎした「フランケンシュタイン型エージェント」が散見されます。しかし、本論文は「LLMの基礎的な推論能力（ベースモデルの強さ）＋環境からの確実なフィードバック（Leanコンパイラ）」の組み合わせが、実は最も強力なエンジンであることを証明しました。  
これは我々エンジニアにとって耳の痛い、かつ勇気をもらえる事実です。

#### 💡 環境の重要性

この最小構成が上手く機能する最大の理由は、**Leanという「絶対的に正解/不正解を判定してくれる完璧な環境」が存在するから**です。  
ビジネス領域のRAGやエージェント開発にこれを応用するなら、「LLMを賢くすること」以上に、「LLMの出力を確定的にチェックする仕組み（ルールベースのチェッカー、バリデーター、テストコード）」をエージェントのループ内に組み込むことが、精度向上の最短ルートになるという深い洞察を与えてくれます。

## 5. まとめ

1. **最小構成の強さ**: 反復的改善、記憶、検索のみのシンプルなエージェントが、複雑な定理証明AIの最先端に匹敵する性能を出した。
2. **コストと効率**: シングルショットでの大量生成よりも、エラーを読み解いて修正するループ構造の方がコスト効率が良い。
3. **現場への応用**: エージェント設計は「引き算」から始めるべき。完璧な検証器（Verifier）を用意し、シンプルなフィードバックループを回すことが最も手堅い。
