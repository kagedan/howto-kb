## DDD再設計 — Architecture Redesign Agent で未来のアーキテクチャを描く

---

### はじめに

前回は Refactoring Agent を使って既存システムの現状分析を行いました。ユビキタス言語辞書、MMI評価、ドメイン-コード対応表という3つの重要成果物が手に入りました。

今回は、これらの分析結果を土台に **Architecture Redesign Agent** を使って、DDDに基づく再設計とコード生成を行います。「現状を知る」から「未来を描く」へのステップです。

---

### Refactoring Agent との違い

Architecture Redesign Agent は Refactoring Agent の進化版です。元の18スキルをすべて含んだ上で、19スキルが追加されています。

| 機能 | Refactoring Agent | Architecture Redesign Agent |
| --- | --- | --- |
| システム調査・分析 | ○ | ○ |
| MMI成熟度評価 | ○ | ○ |
| マイクロサービス設計 | ○ | ○ |
| ナレッジグラフ | ○ | ○ |
| セキュリティ分析（OWASP Top 10） | - | ○ |
| DDD適合度評価 | - | ○ |
| 統合改善計画 | - | ○ |
| DDD再設計（境界コンテキスト） | - | ○ |
| コード生成（Spring Boot + ScalarDB） | - | ○ |
| テスト仕様生成（BDD） | - | ○ |
| インフラ設計（K8s / Terraform） | - | ○ |
| 対話的ワークフロー（`/workflow`） | - | ○ |
| パイプライン中断・再開 | - | ○ |

最も重要な違いは、Refactoring Agent が「分析と設計案の提示」で止まるのに対し、Architecture Redesign Agent は**実装可能なコードとインフラ設計まで**生成する点です。

---

### セットアップ

```
git clone https://github.com/wfukatsu/architecture-redesign-agent.git
cd architecture-redesign-agent

# Python依存パッケージ
python3 -m venv .venv && source .venv/bin/activate
pip install ryugraph pandas markdown pymdown-extensions radon networkx

# Mermaid CLI
npm install -g @mermaid-js/mermaid-cli

# Stage 1の成果物をコピー（Refactoring Agent を先に実行した場合）
cp -r ../refactoring-agent/reports/ ./reports/
```

Refactoring Agent を先に実行していない場合でも問題ありません。Architecture Redesign Agent は分析フェーズ（Phase 0-1）を自前で実行できます。

---

### 3つの実行モード

Architecture Redesign Agent には3つの入り口があります。

#### `/workflow` — 対話的に選択（推奨）

実行すると、以下の選択肢が表示されます：

```
1. 完全パイプライン実行 — 調査から設計・コード生成まで全フェーズ一括
2. 分析・評価まで実行 — Phase 0-2（調査→分析→評価→統合）
3. 設計から実行 — Phase 3-8（DDD再設計→ScalarDB設計→実装→コード生成）
4. 個別フェーズ選択 — 全フェーズから個別に選択
```

Refactoring Agent で分析が完了している場合は **「3. 設計から実行」** を選べば、評価フェーズから開始できます。

#### `/full-pipeline` — 全フェーズ一括実行

```
/full-pipeline ./SampleCode
```

Phase 0（調査）から Phase 13（レポート統合）まで、すべてのフェーズを自動実行します。

```
# オプション例
/full-pipeline ./src --skip-security          # セキュリティ分析スキップ
/full-pipeline ./src --analyze-only           # 分析・評価のみ
/full-pipeline ./src --domain=Order           # 特定ドメインに限定
/full-pipeline ./src --resume-from=phase-6    # 中断後の再開
```

#### `/refactor-system` — 分析・設計のみ（v1互換）

```
/refactor-system ./SampleCode
```

コード生成を行わず、分析→評価→設計までを実行します。Refactoring Agent の `/refactor-system-cmd` と同等の動作です。

---

### 13フェーズのパイプライン

```
【調査】 Phase 0 → 0.5 → 1 → 1.5
【評価】 Phase 2              ← 転換点
【設計】 Phase 3 → 4 → 5
【実装】 Phase 6 → 7 → 8 → 8.7
【統合】 Phase 9 → 10 → 11 → 12 → 13
```

以下、各フェーズのポイントを解説します。

---

### Phase 0-1: 調査と分析

Refactoring Agent と同等の機能です。Stage 1 で既に実行済みの場合、このフェーズの出力がそのまま使われます。

| フェーズ | スキル | 出力 |
| --- | --- | --- |
| 0 | `/system-investigation` | 技術スタック、コード構造、技術的負債 |
| 0.5 | `/security-analysis` | OWASP Top 10 対応状況、ゼロトラスト準備度 |
| 1 | `/analyze-system` | ユビキタス言語、アクター、ドメイン対応表 |
| 1.5 | `/data-model-analysis` | ER図、テーブル定義、正規化状態 |

---

### Phase 2: 評価 — パイプラインの転換点

ここからが Architecture Redesign Agent の真骨頂です。Phase 0-1 で集めた情報を使って、現行システムの品質を**定量的に**評価します。

#### DDD適合度評価

**スキル:** `/ddd-evaluation`

DDDの戦略的設計と戦術的設計の両面から適合度を評価します。

**戦略的設計の評価項目:**

* 境界コンテキストは適切に定義されているか
* コンテキスト間の関係パターンは明示的か
* ユビキタス言語はコードに反映されているか

**戦術的設計の評価項目:**

* エンティティと値オブジェクトは適切に区別されているか
* 集約の設計は正しいか
* リポジトリパターンは守られているか
* ドメインイベントは定義されているか

サンプルシステムでは、以下のような課題が検出されました：

```
検出された課題:
・値オブジェクトが一切定義されていない（Money, Address 等がプリミティブ型）
・ドメインイベントが未実装
・DTOが未使用でエンティティがそのままAPIレスポンスになっている
・集約ルートが不明確（OrderItem が独立してアクセス可能）
```

#### 統合改善計画

**スキル:** `/integrate-evaluations`

MMI評価（Refactoring Agent / Phase 2前半）とDDD評価の結果を統合し、**優先度付きの改善計画**を策定します。

```
改善優先度:
1. [高] Presentation層の分離 — MMI 48.0, DTOの導入が急務
2. [高] 値オブジェクトの導入 — DDD戦術的設計の根幹
3. [中] ドメインイベントの実装 — サービス間連携の基盤
4. [中] 集約境界の明確化 — OrderItem のアクセス制御
5. [低] リポジトリ抽象化の改善 — 既にスコア高い
```

---

### Phase 3: DDD再設計 — 境界コンテキストを描き直す

**スキル:** `/ddd-redesign`, `/map-domains`

Phase 2 の評価結果を踏まえて、DDDに基づく新しい設計を行います。

#### 境界コンテキストの再定義

既存のモノリスを、DDDの境界コンテキスト（Bounded Context）に分割します。

```
Before（モノリス）:
  com.example.order/
    ├── controller/
    ├── service/
    ├── domain/
    └── repository/

After（境界コンテキスト）:
  注文管理コンテキスト（Order Management）
    ├── 集約: Order（ルート）+ OrderItem
    ├── 値オブジェクト: Money, Quantity, OrderNumber
    └── リポジトリ: OrderRepository

  顧客管理コンテキスト（Customer Management）
    ├── 集約: Customer（ルート）
    ├── 値オブジェクト: Email, PhoneNumber, Address
    └── リポジトリ: CustomerRepository

  商品管理コンテキスト（Product Management）
    ├── 集約: Product（ルート）
    ├── 値オブジェクト: Price, ProductCode
    └── リポジトリ: ProductRepository
```

#### コンテキスト間の関係パターン

```
注文管理 ←─── Customer-Supplier ───→ 顧客管理
   │
   └──── Customer-Supplier ───→ 商品管理
```

関係パターン（Shared Kernel、Customer-Supplier、ACL等）も自動的に提案されます。

#### ドメインの分類

各境界コンテキストが、以下のどのカテゴリに属するかを分類します：

| 分類 | 説明 | 例 |
| --- | --- | --- |
| Core Domain | 競争優位性の源泉 | 注文管理 |
| Supporting Domain | コアを支援する | 顧客管理 |
| Generic Domain | 汎用的で差別化不要 | 認証・認可 |

---

### Phase 4-5: マイクロサービス設計とデータ設計

#### Phase 4: ターゲットアーキテクチャ

**スキル:** `/design-microservices`

DDD再設計の結果をマイクロサービスアーキテクチャに落とし込みます。

* ターゲットアーキテクチャ図
* サービス分割計画
* 現行→ターゲットへの段階的移行パス
* 運用計画

#### Phase 5: ScalarDB データ設計

**スキル:** `/select-scalardb-edition`, `/design-scalardb`, `/design-api`

ScalarDB を使用したデータアーキテクチャを設計します。

```
# ScalarDB エディション選定（対話形式）
/select-scalardb-edition
```

3つのエディション（OSS / Enterprise Standard / Enterprise Premium）から最適なものを選定した上で、スキーマ設計とAPI仕様を生成します。

---

### Phase 6-8: 実装とコード生成

ここが Architecture Redesign Agent 最大の目玉です。

#### Phase 6: 実装仕様

**スキル:** `/design-implementation`

AIコーディングエージェントがそのまま使える詳細仕様書を生成します。メソッドシグネチャ、例外ハンドリング、バリデーションルールまで網羅します。

#### Phase 7: テスト仕様

**スキル:** `/generate-test-specs`

BDD（振る舞い駆動開発）シナリオを Gherkin 形式で生成します。

```
Feature: 注文作成
  Scenario: 正常な注文作成
    Given 顧客ID "C001" の顧客が存在する
    And 商品ID "P001" の商品が在庫にある
    When 顧客が商品 "P001" を数量 2 で注文する
    Then 注文が作成される
    And 注文状態が "PENDING" である
    And 注文合計金額が正しく計算される
```

#### Phase 8: コード生成

**スキル:** `/generate-scalardb-code`

Spring Boot + ScalarDB のコードを自動生成します。

```
generated/
├── order-service/
│   ├── src/main/java/
│   │   ├── domain/
│   │   │   ├── model/        # エンティティ、値オブジェクト
│   │   │   ├── repository/   # リポジトリインターフェース
│   │   │   └── service/      # ドメインサービス
│   │   ├── application/      # アプリケーションサービス
│   │   ├── infrastructure/   # ScalarDB リポジトリ実装
│   │   └── presentation/     # REST コントローラ + DTO
│   └── pom.xml
├── customer-service/
└── product-service/
```

#### Phase 8.7: インフラ設計

**スキル:** `/design-infrastructure`

Kubernetes マニフェスト、Terraform モジュール、Helm チャートを生成します。AWS / Azure / GCP / OpenShift に対応しています。

---

### 3層スキルアーキテクチャ — 内部の仕組み

37のスキルは3つのレイヤーで構成されています。

```
Layer 1: Orchestration（3スキル）
  /workflow, /full-pipeline, /refactor-system
  → 他のスキルをどの順番で呼び出すかを制御

Layer 2: Phase-specific（27スキル）
  Investigation(7), Evaluation(4), Design(12),
  CodeGen(2), Estimation(2), Graph(3)
  → 実際の分析・設計・生成を担当

Layer 3: Utility（4スキル）
  /init-output, /compile-report, /render-mermaid, /fix-mermaid
  → パイプラインを補助
```

Layer 1 のオーケストレーションスキルが Layer 2 のフェーズスキルを順次呼び出し、Layer 3 のユーティリティスキルが補助するという階層構造です。

---

### サブエージェントパターン — コンテキストを守る

各スキルは **サブエージェント** を使ってコンテキストウィンドウを保護します。

```
メインエージェント（コンテキスト: 分析指示 + 要約結果のみ）
  │
  ├── Task(Explore): "パッケージ構造を調査"  → 箇条書き5行を返却
  ├── Task(Explore): "ドメイン用語を抽出"    → 用語テーブル10行を返却
  └── Task(Explore): "外部連携APIを列挙"    → リスト3件を返却
```

サブエージェントが大量のコードを読み込んで調査し、**要約結果だけ**をメインに返します。メインエージェントのコンテキストには要約だけが入るため、大規模コードベースでもコンテキストが溢れません。

---

### Stage 3 への橋渡し

Architecture Redesign Agent の出力は、次のステージ（Coding Agent for ScalarDB）の入力になります。

| Architecture Redesign Agent の出力 | Coding Agent for ScalarDB での用途 |
| --- | --- |
| `target-architecture.md` | 要件分析の基盤 |
| `scalardb-schema.md` | データモデル設計の入力 |
| `api-specifications/` | API・インターフェース設計の参照 |
| `generated/{service}/` | 実装ガイドの具体化 |
| DDD再設計ドキュメント一式 | ドメインモデリングの前提 |

---

### まとめ

* Architecture Redesign Agent は Refactoring Agent の **全機能 + 19スキル**
* **Phase 2（評価）が転換点**。MMI + DDD評価 → 統合改善計画で「何から直すか」が決まる
* **Phase 3 でDDD再設計**。境界コンテキスト、集約、値オブジェクトを再定義
* **Phase 6-8 でコード生成**。Spring Boot + ScalarDB の実装コードとインフラ設計
* 3層スキルアーキテクチャとサブエージェントパターンでコンテキストを効率管理
* 中断しても `--resume-from` で再開可能

次回は、最後のステージ——Coding Agent for ScalarDB による実装計画の策定とレビューを解説します。
