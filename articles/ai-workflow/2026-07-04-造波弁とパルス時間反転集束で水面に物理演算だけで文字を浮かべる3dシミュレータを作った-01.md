## はじめに

(この記事はFable5で作ったアプリの解説をSonnet5で作成し、私が編集したものです)  
スマホ1台・シングルHTMLファイルで「3D水面波形シミュレータ」を作りました。指でタップすれば波紋が広がり、ドラッグで視点が回転し、ピンチでズームできる——ここまでならよくあるThree.js製の水面デモです。

このアプリの本題は別のところにあります。円形の水槽の縁に並んだ**448基の造波弁**を精密に駆動し、水面の波の「干渉」だけで任意の文字列を一瞬だけ水面に浮かび上がらせる機能です。文字の形をあらかじめ高さデータとして書き込む、というような近道は一切していません。弁が波を出し、その波が水槽内を伝わり、ぶつかり合って強め合った場所がたまたま文字の形になる——正真正銘、物理演算だけの産物として文字が浮かび上がります。

## アプリの概要

* シングルHTMLファイル(React 18 + Three.js r128 + D3.js + Tailwind、すべてCDN経由でBabelがその場でトランスパイル)
* 円形の水槽を448×448グリッド上でセル分類(流体/シェル/圏外)した2次元波動方程式ソルバー
* タップで波紋、ドラッグでカメラ回転、ピンチでズーム、「一滴」「雨」などの基本操作
* 波高マップ(D3 Canvas)とエネルギー推移チャート(D3 SVG)をリアルタイム表示
* 目玉機能:テキストを入力して「発射」を押すと、1〜5文字が水面にフラッシュ表示される(全文字同時表示/1文字ずつ表示の2モード)

## 要件が生んだ制約:「本物の物理」縛り

最初の要望はこうでした。

> 造波弁を精密にコントロールすることで、水面に任意の文字を浮かび上がらせる仕組みを実装してほしい

もっとも手っ取り早い実装は、文字の形をそのまま高さデータへ書き込んでしまう「フェイク」方式です。実装は一瞬で終わり、文字も完全にシャープになります。しかし要望の本質は「造波弁を(精密に)コントロールすることで」——つまり干渉は本物の波動方程式に計算させろ、という縛りでした。このたった1行の制約が、以降のほぼすべての設計判断を支配することになります。

検討はしたものの見送った案もあります。円形膜の固有振動をBessel関数展開するモード分解方式は数学的には厳密ですが、シャープな形を再現するには多数の高次モードが必要で、モードごとに異なる共振周波数を同時駆動しなければならず調整が煩雑になります。シミュレーションを内側に組み込むadjoint最適化(真の逆問題)は理論上もっとも忠実ですが、1反復ごとに定常状態までのフル計算が必要でコストが桁違いに重くなります。極座標グリッドへの全面移行も、既に動作検証済みのCartesianソルバーを捨てるリスクに見合わないため不採用にしました。

## 第1案:位相配列ホログラフィ(Gerchberg-Saxton法)

最初に採用したのは、音響ホログラフィやフェーズドアレイと同じ原理です。円周上に並んだ点音源(造波弁)それぞれの位相を最適化して目標形状の振幅分布を再現する、計算機合成ホログラム(CGH)の標準的な手法であるGerchberg-Saxton法(GS法)を使いました。

流れは大きく4ステップです。

1. 各弁の位相をランダム初期化する(振幅は全弁一定)
2. 全弁からの寄与を目標グリッド上で重ね合わせ、複素振幅場を計算する(順伝播)
3. 各点の振幅だけをターゲット画像(`ctx.fillText()`で描いた文字)に置き換え、位相は保持する
4. 共役伝播で各弁へ逆投影し、新しい位相を抽出する

これを100〜300回繰り返すと、弁の位相配列は「目標形状を作る干渉パターン」へ収束していきます。求めた位相・振幅を`stepSim`内の境界条件(Dirichlet強制)として毎フレーム適用すれば、あとは本物の波動方程式が干渉を計算してくれる、という設計でした。

### GS法の壁:閉じたキャビティは共振する

理論上は正しいのですが、実装すると壁にぶつかりました。境界を持つ水槽は閉じたキャビティです。連続波を弁から出し続けると水槽の壁で反射した波が戻ってきて元の波に重なり、キャビティ特有の共振モードが立ってしまいます。GS法の最適化は「反射のない自由空間」を前提にしているため、狙った干渉パターンと実際の共振パターンがズレる——このズレを消すには実測ベースの較正が必要で、しかも水槽サイズや減衰係数を変えるたびにやり直しになります。「ボタンを押した瞬間にきれいな文字が出る」という体験には程遠い状態でした。

## 第2案:パルス時間反転集束

発想を転換しました。連続波をやめ、各弁は「ガウス窓付きの波束(パルス)」を1回だけ発射する方式です。

鍵は時間反転の原理です。各弁からターゲットの各ドットまでの距離と、離散格子上での群速度が分かれば、「すべての弁からの波束が、共通のフラッシュ時刻Tちょうどに全ドットへ同時到達する」ように、弁ごとの発射タイミングと位相を逆算できます。全部の波束が収束した瞬間だけ、文字列全体が水面に一斉に浮かび上がります。

この方式の強みは、GS法が抱えていた問題がそもそも起きないことです。集束は「壁で反射して戻ってくる前の、初回通過波」だけで完成します。閉じたキャビティの共振モデルとは無縁なので、較正もGS反復も不要——ボタンを押した瞬間に発射できます。数値検証では単一文字でコントラスト6.7倍、3文字同時表示でも3.4倍を確認しました(GS法時代は2〜3倍程度でした)。

### 一瞬だけ光る文字をどう「見せる」か

パルスが収束する瞬間は文字通り一瞬(±0.2秒ほど)なので、そのままでは目で追えません。そこで「長時間露光」の考え方を持ち込みました。フラッシュの捕捉窓の間だけ各セルの正のクレスト(山)の最大値を記録し、それを時定数約4.5秒でゆっくり減衰する琥珀色の残像として重ねて描画します。

正のクレストだけを捕捉するのには理由があります。すべての波の成分が位相0で同時到達する焦点は必ず正の山になりますが、焦点を取り囲む最初のリング(第1リング)は必ず負の谷になります。つまり「正だけ拾う」という単純なルールだけで、リングは自動的に消えドットだけが残るのです。数値検証で確認済みの、地味に気に入っている副産物です。

## 第3案:広帯域分散補償チャープでさらに解像度を上げる

単一周波数の波束でも文字は読めるようになりましたが、焦点の周りにJ0ベッセル関数状のリングがまだ強く残り、にじみが気になります。

そこで単一波長ではなく、ω\_lo〜ω\_hiの範囲に26個の周波数成分を重ねた広帯域パルス(分散補償チャープ)に発展させました。各周波数成分は、格子上の方向依存の厳密な分散関係

`sin²(ω·dt/2) / (c·dt)² = sin²(k·cx/2) + sin²(k·cy/2)`

をNewton法で解いて位相 k(ω,θ)·r を個別に求め、「全成分が焦点に時刻Tちょうど・位相0で同時到達する」ように波束を合成します。波長の異なる成分どうしのリングは互いに位置がずれて打ち消し合うため、焦点スポットが大きく締まります。

効果は数値検証で確認済みです。

* 焦点波形のクレスト数:5〜7個 → 3個
* 単一波長では40%残っていた第1リングが実質消滅
* リング消滅により、ドット同士の最小間隔を5.25セル→4.2セルまで詰められるように

単純にパルス幅を縮めれば同じ効果が得られるのでは、とも考えましたが、格子分散(周波数によって群速度が変わる性質)のせいで伝播中にパルスが約1秒まで間延びしてしまい逆効果になることを確認済みです。広帯域チャープは、この分散を逆手にとって「発射時点であらかじめ引き伸ばしておき、伝播中にちょうど圧縮される」ように設計しています。

## 性能との戦い

ここからは高速化の話です。文字の解像度を上げるにはグリッドを細かくするしかありませんが、素直にやると事前計算も描画も重くなります。実際にN=64から128、192、256、320、448と段階的に上げていく過程で(物理パラメータの安定性はCFL条件で頭打ちになるため、幾何スケールと物理スケールは`PHYS_SCALE`という係数で分離し、格子を細かくしても波の速さの見た目は変えずセル単位の波長だけを短くしています)、2つの大きなボトルネックにぶつかりました。

### ボトルネック1:分散関係の求解が事前計算の77%

プロファイリングの結果、パルス波形合成コストの77%が、Newton法による分散関係の求解(`solveLatticeK`)に費やされていることが分かりました。方向・周波数の組み合わせごとに毎回12回のNewton反復を回していたのが原因です。

これを、格子の対称性を使って方向を0°〜45°に折りたたんだ256分割のルックアップテーブル+線形補間に置き換えました。事前検証した最大位相誤差は3×10⁻⁴ rad(コヒーレンス許容0.1 radの1/300)で、品質への影響は無視できるレベルです。

```
// 方向を[0°,45°]に折り畳んでθテーブルを線形補間
let a = Math.abs(dgj / r), b2 = Math.abs(dgi / r);
if (a < b2) { const tswap = a; a = b2; b2 = tswap; }
const x = (Math.atan2(b2, a) / QPI) * tbins;
const bin0 = Math.min(tbins - 1, Math.floor(x)), fr = x - bin0;
```

効果は劇的でした。この改修と同時にドット予算も168個→216個へ引き上げましたが、それでもN=448構成での事前計算時間は**29秒→4.2秒**まで縮んでいます。

### ボトルネック2:頂点更新ループがフレームコストの98%

2つ目は描画側です。N=448だと格子点は約20万頂点。毎フレームJSで全頂点の変位・法線(有限差分)・色(高さ→RGBランプ+残像ブレンド)を計算し、Three.jsのBufferAttributeへ6.9MB分を再転送していました。計測すると1フレームあたり83ms——フレームコストの実に98%です。

解決策は、高さ場と残像場を浮動小数テクスチャ(`THREE.DataTexture`)としてそのままGPUへ送り、頂点シェーダ側で変位・法線・色を計算させることでした。`material.onBeforeCompile`でThree.jsの標準頂点シェーダに割り込み、`#include <begin_vertex>`などのフック位置にGLSLを注入しています。

```
material.onBeforeCompile = (shader) => {
  Object.assign(shader.uniforms, uniforms);
  shader.vertexShader = shader.vertexShader
    .replace('#include <color_vertex>', /* 高さ→色ランプ+残像を計算するGLSL */)
    .replace('#include <beginnormal_vertex>', /* 有限差分法線 */)
    .replace('#include <begin_vertex>', `
      vec3 transformed = vec3(position.x, wgHC, position.z);
    `);
};
material.customProgramCacheKey = () => 'water-gpu-displace-v1';
```

数式はCPU版`updateMeshFromHeights`と1行単位で同一にしてあるので見た目の劣化はありません。転送量もテクスチャの零コピー更新で計1.6MB/フレームまで下がり、83msかかっていたJSループはほぼ消滅しました。WebGL2が使えない環境ではCPU経路に自動フォールバックします。

### 仕上げ:Web Worker並列合成

弁ごとの波形合成は完全に独立な処理なので、並列化との相性が良い処理です。合成コアの`synthesizeValveRange`を外部依存ゼロの純粋関数として切り出し、`.toString()`でソースコードを文字列化してWeb Workerへそのまま転写しています。

```
const src = `
  ${synthesizeValveRange.toString()}
  self.onmessage = function(e) {
    const rows = synthesizeValveRange(e.data);
    self.postMessage({ ...e.data, rows }, [rows.buffer]);
  };
`;
```

メインスレッド用のフォールバック実行とWorker側の並列実行が文字通り同じ関数定義を共有するので、「Worker版とメインスレッド版で数値がズレる」というクラスのバグは構造的に起きません。`navigator.hardwareConcurrency`を見てワーカー数を決め、さらに細かくタスク分割して進捗表示とコア間の負荷分散を両立させています。Workerが使えない、あるいは初期化に失敗した環境では、自動的に従来のフレーム分割チャンク実行にフォールバックします。

## 「盛れるだけ盛る」を試した検証実験

最適化が一段落したところで、「ドット数をもっと増やせないか」を検証しました。面白い結果が出たので紹介します。

旧グリッド(N=320)のまま、最小ドット間隔4.2セルという制約を単純に3.4セルまで詰める実験をしたところ、216個のドットを確保できましたが、コントラストは2.6倍まで落ち込み、隣接するドットの第1リング同士が重なって文字が完全に融合してしまいました。一方、間隔は据え置きでグリッド自体をN=448まで拡大するアプローチでは、288個のドットを8.6倍のコントラストで確保できています。詰め方次第で明暗がくっきり分かれた形です。

原因は、最小間隔が単なるチューニングパラメータではなく、広帯域集束スポットのメインローブ半径(約2セル)そのもので決まる物理的な下限だったこと。「ドット間隔を削るな、幾何スケールを上げろ」という、数値でしか出てこない結論でした(このN=448拡大案の事前計算コスト・約49秒を4.2秒まで縮めたのが、前述のθテーブルです)。ちなみに最終構成でN=448のまま実際に「2355」(Eテレの番組名)を表示させたときのドット数も216個でした——**同じ216個でも、間隔を削ったか幾何スケールを上げたかで、コントラストは2.6倍にも9.1倍にもなる**というのが、この検証で一番面白かった発見です。

## できあがったものの数字

最終構成(コード内では「実験4」と呼んでいる、N=448・弁448基・ドット予算2倍の構成)での実測値です。

| グリッド | 448×448(流体セルのみLaplacian更新) |
| 造波弁 | 448基 |
| 4桁の数字表示("2355") | 216ドット・コントラスト9.1倍 |

## 実装まわりの小さな工夫

* **円形水槽は正方形グリッドのセル分類で実現**:極座標への全面移行はせず、中心からの距離が半径以内かどうかでセルを「流体/シェル(造波弁が直接駆動)/圏外」の3種に分類。ジオメトリのインデックス配列も圏外の三角形を除外するだけで、見た目は円形にクリップされます。極座標特有の中心特異点処理やLaplacianステンシルの再設計を避けられ、既存の検証済みロジックをそのまま流用できました。
* **固定タイムステップ+補間描画**:物理は1/60秒固定で積分し、実際の描画では前フレームと今フレームの間をalpha補間します。90Hz/120Hzディスプレイでも滑らかに見えるようにするための定番手法です。
* **文字→ドット変換**:オフスクリーンCanvasに`fillText`で文字を描画し、ブロックごとの重心(輝度加重平均)まで間引いたあと、重み順の貪欲法で最小間隔を満たす点だけを採用します。複数文字同時表示では隣の文字のドットとの間隔も同時にチェックし、文字境界でのリング干渉によるゴーストを防いでいます。
* **スロット占有率0.80**:複数文字を横に並べる際のスロット占有率です。0.94まで詰めると、5文字構成の検証で隣接文字の同じ高さのストロークが同時フラッシュ時に融合して見えることを確認したため、余裕を持たせています。

## Fable5をどう使ったか

この開発では要件定義から実装、検証まで一貫してFable5を活用しました。特に効いたと感じたのは次の3点です。

1. **実装前の設計比較**:「造波弁を精密にコントロールして文字を出す」という1行の要件に対し、位相配列ホログラフィ・モード分解・adjoint最適化・極座標全面移行など複数のアプローチのトレードオフを言語化してから実装に入る、という進め方ができました。
2. **本体を触る前のスタンドアロン検証**:GS法やパルス集束のアルゴリズムは、まずNode.js上の単体スクリプトで数値的に検証してから、シングルHTMLファイル本体への組み込みに進みました。壊れたら困る巨大な1ファイルへ、検証されていないアルゴリズムをいきなり書き込まずに済みます。
3. **ベンチマークを受け入れ基準にした最適化**:「GPU変位レンダリング→θテーブル分散→Worker並列合成」という改修の優先順位そのものが、事前に用意したベンチマークスクリプトの計測結果から導かれています。「効きそうだから直す」ではなく「実測した結果ここが支配的コストだったから直す」という進め方を一貫してできたのが大きかったです。

## まとめ

「文字の形を直接描かない」という1行の制約から出発し、GS法によるホログラフィ的アプローチ→パルス時間反転集束→広帯域分散補償チャープと、3段階でアルゴリズムを進化させました。並行してθテーブル化・GPU変位レンダリング・Worker並列化という3つの最適化で、事前計算29秒→4.2秒、フレームコスト83ms→1ms未満まで持っていけたのは、地道な検証あってのことだと思います。
