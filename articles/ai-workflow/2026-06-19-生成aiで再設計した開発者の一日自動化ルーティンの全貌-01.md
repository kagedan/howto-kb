## はじめに：なぜルーティンを見直したのか

「また同じような作業をしている」と気づいたのは、ある朝のスタンドアップ準備中だった。

毎朝やっていたこと：

* 昨日のGitログを見てタスクを思い出す
* Slackの未読を整理してサマリーを作る
* その日のTodoをNotionに手入力
* 技術ブログ・RSS feedをざっと眺める

これが毎朝**30〜40分**かかっていた。月換算で約15時間。純粋に「繰り返し作業」に溶けていた時間だ。

エンジニアとして、**「自動化できるものは自動化する」** のが信条のはずなのに、自分自身のワークフローには無頓着だった。生成AIが実用レベルになった今こそ、ここにメスを入れるべきだと判断した。

この記事では、OpenAI APIとPythonを使って自分のルーティンをどう再設計したか、具体的な実装とともに紹介する。

---

## 全体の仕組み：アーキテクチャ概要

構成はシンプルにまとめた。複雑にすると結局メンテしなくなるからだ。

```
[データソース群]
  ├── GitHub API（コミット・PR・Issue）
  ├── Slack API（未読メッセージ）
  ├── RSS Feed（技術ブログ）
  └── Google Calendar API（今日の予定）
         ↓
[Python集約スクリプト]
  └── 各APIからデータ取得 → 前処理
         ↓
[OpenAI API（GPT-4o）]
  └── サマリー生成 / 優先度付け / アクション提案
         ↓
[出力先]
  ├── Slack（朝のDM自動送信）
  └── Notion（デイリーログ自動作成）
```

トリガーはcronで毎朝7:30に実行。所要時間は**約90秒**で完了する。

### Step 2：各データソースからの情報収集

#### GitHub：昨日のアクティビティ取得

```
from github import Github
from datetime import datetime, timedelta, timezone

def get_github_activity(token: str, username: str) -> dict:
    g = Github(token)
    user = g.get_user(username)
    since = datetime.now(timezone.utc) - timedelta(days=1)

    commits = []
    prs = []
    issues = []

    for repo in user.get_repos():
        try:
            for commit in repo.get_commits(author=username, since=since):
                commits.append({
                    "repo": repo.name,
                    "message": commit.commit.message.split("\n")[0],
                    "sha": commit.sha[:7]
                })
        except Exception:
            continue

    # PRとIssueはsearchで取得
    query = f"involves:{username} updated:>{since.strftime('%Y-%m-%d')}"
    for issue in g.search_issues(query):
        item = {
            "repo": issue.repository.name,
            "title": issue.title,
            "state": issue.state
        }
        if issue.pull_request:
            prs.append(item)
        else:
            issues.append(item)

    return {"commits": commits, "prs": prs, "issues": issues}
```

### Step 3：GPT-4oへのプロンプト設計

ここが一番時間をかけた部分だ。プロンプトの質がそのまま出力の質に直結する。

試行錯誤の末、**役割・文脈・出力形式**の三段構成に落ち着いた。

```
from openai import OpenAI
import json

client = OpenAI()

def generate_daily_brief(github_data: dict, articles: list, calendar_events: list) -> dict:
    system_prompt = """
あなたはシニアエンジニアの朝のブリーフィングを担当するAIアシスタントです。
提供された情報を分析し、その日の生産性を最大化するためのサマリーを作成してください。
出力は必ずJSON形式で返してください。
"""

    user_prompt = f"""
以下の情報を元に、今日の朝のブリーフィングを作成してください。

## 昨日のGitHubアクティビティ
{json.dumps(github_data, ensure_ascii=False, indent=2)}

## 今日のカレンダー予定
{json.dumps(calendar_events, ensure_ascii=False, indent=2)}

## 注目の技術記事
{json.dumps(articles[:10], ensure_ascii=False, indent=2)}

## 出力フォーマット（JSON）
{{
  "good_morning_message": "モチベーションが上がる一言（50文字以内）",
  "yesterday_summary": "昨日の作業サマリー（3行以内）",
  "today_priorities": ["優先タスク1", "優先タスク2", "優先タスク3"],
  "recommended_articles": [
    {{"title": "記事タイトル", "reason": "なぜ読むべきか（30文字以内）", "link": "URL"}}
  ],
  "focus_suggestion": "今日の集中すべき一点（フォーカスタイム提案）"
}}
"""

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt}
        ],
        response_format={"type": "json_object"},
        temperature=0.7
    )

    return json.loads(response.choices[0].message.content)
```

`response_format: json_object` を指定することでJSON parseエラーをほぼゼロにできる。これは地味に重要だ。

### Step 4：SlackへのDM自動送信

```
from slack_sdk import WebClient

def send_to_slack(token: str, user_id: str, brief: dict):
    client = WebClient(token=token)

    blocks = [
        {
            "type": "header",
            "text": {"type": "plain_text", "text": f"🌅 {brief['good_morning_message']}"}
        },
        {
            "type": "section",
            "text": {
                "type": "mrkdwn",
                "text": f"*📝 昨日のまとめ*\n{brief['yesterday_summary']}"
            }
        },
        {
            "type": "section",
            "text": {
                "type": "mrkdwn",
                "text": "*🎯 今日の優先タスク*\n" + "\n".join(
                    [f"• {t}" for t in brief['today_priorities']]
                )
            }
        },
        {
            "type": "section",
            "text": {
                "type": "mrkdwn",
                "text": f"*⚡ フォーカス提案*\n{brief['focus_suggestion']}"
            }
        }
    ]

    # DMチャンネルをオープン
    dm = client.conversations_open(users=[user_id])
    channel_id = dm["channel"]["id"]

    client.chat_postMessage(
        channel=channel_id,
        blocks=blocks,
        text="今日の朝のブリーフィングが届きました"
    )
```

## ハマったポイント・注意点

### 1. GitHub APIのレート制限

`get_repos()` で全リポジトリを舐めると、リポジトリ数が多い場合にレート制限（5000req/hour）に引っかかる。

解決策：対象リポジトリを明示的に指定するか、直近更新順で上位N件に絞る。

```
# 直近30日以内に更新されたrepoのみ対象
recent_repos = [
    repo for repo in user.get_repos(sort="updated")
    if repo.updated_at > datetime.now(timezone.utc) - timedelta(days=30)
][:20]  # 最大20件
```

### 2. プロンプトの「出力ブレ」問題

初期は `temperature=1.0` にしていたが、JSON構造が崩れることが週1回程度あった。`temperature=0.7` に下げ、かつ `response_format` を明示することでほぼ解消した。

### 3. Slack APIのスコープ設定

DM送信には `chat:write` と `im:write` の両方が必要。片方だけだと `channel_not_found` エラーが出て30分溶かした。

### 4. コスト管理

毎日GPT-4oを叩くのでコストが気になった。実測では1回の実行で**約0.03〜0.05ドル**。月30回で約1.5ドル。許容範囲だが、節約したいならGPT-4o-miniで十分かもしれない。

```
# コスト意識するならminiも選択肢
model = "gpt-4o-mini"  # 約1/10のコスト
```

## 実際に変わったこと

導入して2ヶ月が経った。変化を正直に書く。

**良かったこと：**

* 朝の情報収集が**40分→5分**に短縮
* 「昨日何やったっけ」がなくなり、スタンドアップが明快になった
* 優先タスクを「AI視点」で整理してもらうことで、見落としに気づくことが増えた

**思ったほどでもなかったこと：**

* 記事推薦は精度が高くない。RSSのsummaryだけでは判断が難しいらしい。タイトルと本文をもう少し送るべきかもしれない
* 「good\_morning\_message」は最初は新鮮だったが、慣れると読み飛ばすようになった（笑）

## まとめ・今後の展望

生成AIは「すごい機能」を作るだけでなく、**自分自身のワークフローの摩擦を取り除く**のにこそ真価を発揮すると感じた。

今後やりたいこと：

* **夜のレビュー自動化**：その日のコミットとPRをもとに「学んだこと」をNotionに自動記録
* **週次レポート生成**：1週間のアクティビティをまとめて振り返りに使う
* **ベクトルDB連携**：過去のタスク履歴をEmbeddingして「似たような問題、以前どう解決したっけ？」を自動提示

コード全体はGitHubに公開予定（記事更新時にリンクを追記します）。

まず1ステップ目として、GitHub APIだけでも繋いでみると「あ、これ使える」と実感できるはず。ぜひ試してみてほしい。
