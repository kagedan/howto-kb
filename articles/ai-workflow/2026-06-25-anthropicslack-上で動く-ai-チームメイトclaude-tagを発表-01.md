## はじめに

2026年6月23日、Anthropic は **Claude Tag** を発表しました。これは Slack のチャネル内に常駐し、`@Claude` とメンションすることでタスクを委譲できる「AI チームメイト」です。従来の「Claude in Slack」アプリを置き換える位置づけで、Claude Enterprise / Team 向けにベータ提供が開始されました。

本記事では、Claude Tag の機能・仕組み・セキュリティ管理・導入方法について、Anthropic の公式発表をもとにまとめます。

## Claude Tag とは

Claude Tag は、チャットボット的な一問一答ではなく、**Slack チャネルに参加する1人の「同僚」** として振る舞うことを目指した機能です。チャネル内の誰でも `@Claude` を呼び出してタスクを任せ、自分は別の作業に集中できます。

<https://www.anthropic.com/news/introducing-claude-tag>

### 従来の Claude in Slack との違い

| 観点 | Claude in Slack（旧） | Claude Tag（新） |
| --- | --- | --- |
| 対話モデル | 個人ごとの一問一答が中心 | チャネル単位で1つの Claude を全員が共有 |
| コンテキスト | セッションごとにリセットされがち | チャネルを追いながら継続的に学習・蓄積 |
| 実行スタイル | 主に応答 | タスクを分解し、非同期・主体的に遂行 |

## 主な特徴

Anthropic が挙げている特徴は次の4点です。

* **マルチプレイヤー対応**: 1つのチャネルには1つの Claude が存在し、チャネル内の全員が Claude の作業状況を確認できます。誰かが途中まで進めた会話を、別の人が引き継ぐことも可能です。
* **継続的な学習**: チャネルの会話を追いながらコンテキストを蓄積するため、毎回ゼロから背景を説明する必要が減ります。
* **主体的な実行**: タスクを段階に分解し、順番に処理します。非同期で複数タスクを並行して進めることもできます。
* **スコープ化されたメモリ**: 学習したメモリはチャネル単位に限定されます（後述）。

### 仕組み

> タグを付けてシンプルな言葉でリクエストすると、Claude がそれを段階に分けて順番に処理する

タスクの結果は Slack スレッドに返却されます。チャネルだけでなく、個人へのダイレクトメッセージにも対応します。

## セキュリティとアクセス制御

Claude Tag はチームや組織での利用を前提に設計されており、システム管理者による細かな制御が可能です。

* **ツール／情報のチャネル単位の指定**: 管理者が「どのチャネルで、どのツール・情報にアクセスできるか」を指定します。
* **用途ごとの分離された Claude**: 用途に応じて別々の Claude 識別情報を作成できます。たとえば営業向けに設定した Claude が、エンジニアリング向けに設定した Claude にメモリを引き渡すことはできません。
* **メモリのチャネルスコープ化**: 学習内容は割り当てられたチャネルの範囲に限定されます。
* **支出制限**: 組織単位・個別チャネル単位でトークン支出の上限を設定できます。
* **監査ログ**: すべての実行ログと「誰が依頼したか」の情報が記録されます。

## 対応モデルと提供状況

| 項目 | 内容 |
| --- | --- |
| 対象プラン | Claude Enterprise / Team |
| 提供形態 | ベータ（2026年6月23日開始） |
| 移行 | 既存の「Claude in Slack」アプリを置き換え |
| 管理者対応 | 30日以内にオプトイン可能。対象組織には導入クレジットを提供 |

## 導入手順

Anthropic が示している導入ステップは次の4段階です。

1. **Slack ワークスペースとのペアリング**
2. **ツールへのアクセス許可の設定**
3. **月間支出上限の設定**
4. **プライベートチャネルでのテスト**

いきなり全社展開するのではなく、プライベートチャネルでの試用から始める流れが想定されています。

## Anthropic 社内での活用

Anthropic は、社内のプロダクトチームのコードの **約65%** を、Claude Tag の社内版が**生成している**と公表しています（原文: "65% of our product team's code is created by our internal version of Claude Tag"）。これは Anthropic 自身による測定値であり、一般的な利用環境での再現性は別途検証が必要です。

<https://x.com/ClaudeDevs/status/2069468900216234010?s=20>

## 技術者視点でのポイント

* **「共有された1人のエージェント」というモデル**: ユーザーごとに独立したボットではなく、チャネル単位で状態を共有するため、引き継ぎ・可視性の面でチーム開発と相性が良い設計です。
* **メモリのスコープ設計が前提**: 学習が強力なほど、情報の越境（営業 → エンジニアリング等）が懸念になります。Claude Tag はメモリをチャネルに閉じることでこれに対処しています。導入時は「どのチャネルに、どの Claude を、どのツール権限で置くか」の設計が重要になります。
* **コストガードレール**: 自律実行・非同期実行はトークン消費が読みづらくなるため、組織／チャネル単位の支出上限が標準で用意されている点は運用上の安心材料です。
* **段階的展開**: プライベートチャネルでの検証 → 権限とコスト上限の調整 → 展開、という流れを取れるため、いきなり本番チャネルへ投入する必要はありません。

## まとめ

Claude Tag は、Claude を「呼び出して使うツール」から「チャネルに常駐する同僚」へと位置づけを変える試みです。マルチプレイヤー対応・継続学習・主体的なタスク遂行を備える一方で、メモリのチャネルスコープ化、ツール権限の細分化、支出上限、監査ログといった企業利用に必要な制御も組み込まれています。

現時点では Claude Enterprise / Team 向けのベータであり、対応モデルは Opus 4.8 です。既存の「Claude in Slack」を利用している組織は、30日以内のオプトインと、権限・コスト設計を含めた移行計画を確認しておくとよいでしょう。
