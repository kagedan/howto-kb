## はじめに

とある架空のAndroidアプリ開発チームの話をします。

そのチームでは、チーム全体のAndroid技術スキルの底上げを目標に掲げていました。チームメンバーは各自が学んだことを整理し、AIエージェント向けのSkillとしてまとめることにしました。

次のSkillを見て、あなたはどのような感想を持つでしょうか？

```
---
name: architecture-review
description: Androidのアーキテクチャに関するレビューを行うスキル。
---

現在のブランチの差分の内容が下記のレビュー観点に従っているかを確認する

# Jetpack Compose のベストプラクティス
- Composable関数は、UIの構築に専念し、ビジネスロジックやデータ処理はViewModelやUseCaseに委譲する
- State Hoistingを活用して、Composable関数の再利用性を高めている
- ViewModelは、UIの状態を管理し、Composable関数に必要なデータを提供する役割を担う

# アーキテクチャ設計
- RepositoryにはInterfaceを定義し、実装クラスはRepositoryImplとする
- ViewModelはUseCaseに依存し、UseCaseはRepositoryに依存する
- domain層はdata層に依存せず、data層はdomain層に依存する
  - 参考として、 `:data:ai` が `:domain:ai` に依存している例を参照する
- domain層のコードはAndroidフレームワークに依存しないようにする

# クリーンアーキテクチャの原則
- SOLID原則を遵守し、単一責任の原則を守ること
- コードの可読性と保守性を高めるために、適切な命名規則とコード構造を採用する

# テスト
- 新規に追加されたコードには、ユニットテストが含まれている
...
```

## そのSkill、本当に意味ありますか？

学んだことをSkillとして整理し、チーム全体で共有すること自体は有効なアプローチです。AIを個人の最適化ツールとしてだけでなく、チームの知識基盤として活用していく姿勢には価値があります。

しかし、コードと同じように、Skillにも有効性や実用性を左右する設計上の要素があります。

Skillのdescription部分はセッション中常にシステムプロンプトに含まれ、本文はトリガー時にのみロードされるという仕組みになっています。

つまり、Skillはゼロコストではありません。公式ベストプラクティスでは「CLAUDE.mdは500行以下に」と推奨されており、これは**注意力の希釈（attention dilution）** の問題を示唆しています。Skillのdescriptionが増えるほど、モデルがどのSkillをいつ起動すべきかの判断精度が落ちる可能性があるのです。

同様に、OpenAIのハーネスエンジニアリングの記事でも「コンテキストは希少な資源である」と明言されており、巨大な指示ファイルはタスクやコードを圧迫し、エージェントが重要な制約を見落とす原因になると指摘されています。

> 過度なガイダンスは指針を失う。すべてが「重要」であるときには、何も重要ではなくなります。  
> ── [OpenAI "Harness Engineering"](https://openai.com/index/harness-engineering/)

## Skillの価値は「いつ・誰が・どの頻度で起動するか」に依存する

Skillの有効性を設計するときに考えるべきは、次の3つの問いです。

| 問い | 具体的に |
| --- | --- |
| **いつ？** | 設計フェーズで起動するのか、実装後のチェックとして起動するのか |
| **誰が？** | 人間が明示的に呼び出すのか、AIが自律的に起動するのか |
| **どの頻度で？** | 毎PR？毎コミット？1スプリントに1回？半期に1回？ |

この3つが曖昧なSkillは、存在しているだけでコンテキストを消費し続ける割に、効果的に使われることがありません。

### いつ？：descriptionの質がSkillの価値を決める

Skillはスラッシュコマンドによる明示的起動と、descriptionに基づくエージェントの自律的起動の両方をサポートしています。descriptionにトリガー条件を明記することが推奨されており、`TRIGGER when:` / `DO NOT TRIGGER when:` のパターンを使ってトリガー条件を明示することが効果的とされています。

いかに充実した内容であっても、適切なタイミングでフックが起動しなければSkill自体の価値は発揮されません。

冒頭のSkillのdescriptionを振り返ってみましょう。「Androidのアーキテクチャに関するレビューを行うスキルです」とだけ書かれています。何をするSkillかは伝わりますが、**いつエージェントが実行すべきか**の情報が欠けています。これでは、自律的にトリガーされるタイミングが極めて限定されます。

### 誰が？：AIが自律的に起動する価値はあるか

人間が手動で呼び出すSkillにも一定の有効性はあります。しかし、2026年現在のトレンドは、人間が意識的にSkillを起動する運用から、AIが自律的にSkillを起動しサブエージェントとして実行する方向へと移っています。

Martin Fowlerサイトの連載シリーズ "Exploring Gen AI" において、Kief Morrisはこの変化を次のように整理しています。

* **Human *in* the loop**：人間がコードの生成や変更を逐一確認する
* **Human *on* the loop**：人間はハーネスを設計・改善し、エージェントがそのハーネスの中で自律的に動く

> The difference between in the loop and on the loop is most visible in what we do when we're not satisfied with what the agent produces. The "in the loop" way is to fix the artefact. The "on the loop" way is to change the harness that produced the artefact so it produces the results we want.  
> ── [Kief Morris, "Humans and Agents in Software Engineering Loops"](https://martinfowler.com/articles/exploring-gen-ai/humans-and-agents.html)

この視点で冒頭のSkillを見直すと、「AIが自律的にアーキテクチャレビューを起動する価値があるか？」という問いが浮かびます。

ドメイン層を設けてAndroidフレームワークに依存しないドメインロジックを書くとか、クリーンアーキテクチャの依存方向とか、ComposeのState Hoistingの話は、現在のLLMが十分に理解している一般原則です。これらをわざわざSkillとして教えなくても、LLMは事前学習の時点でこうした知識を持っています。もしこうした基本的な原則をSkill起動前から意識できていないとしたら、それはSkillの問題ではなくLLMの品質の問題です。

### どの頻度で？：使用頻度とコストのバランス

冒頭のSkillは「アーキテクチャ設計が必要なタイミング」で起動されることを想定しています。では、そのタイミングはどのくらいの頻度で訪れるでしょうか。  
毎リリースごとにこれまでにない新機能を開発し、モジュールの追加が必要になるようなプロダクトなら嬉しいですが、現実的には年に数回程度の頻度になることが多いのではないでしょうか。

年に数回しか発生しない意思決定のために、数十文字とはいえdescriptionがセッション中ずっとコンテキストに載り続けます。最近はClaude Opus 4.6の1M Contextモデルをはじめ、コンテキストウィンドウは大きくなっているためトークンコストそのものは無視できるレベルかもしれませんが、前述の注意力の希釈とトリガー精度の問題は残ります。

## 設計すべきはSkill単体ではなく、フィードバックループである

ここまでSkillの有効性について考えてきましたが、それでは一体どうすればよいのでしょうか。

答えは、「Skill単体の設計ではなく、AIエージェントにとってのフィードバックループを設計すべき」です。つまり、**ハーネスエンジニアリング**の考え方が必要になります。

ハーネスエンジニアリングは、2026年2月にOpenAIが自社のCodexチームの実践をまとめた記事で打ち出した概念です。これを受けて、Birgitta BöckelerがMartin Fowlerサイトの "Exploring Gen AI" シリーズで "Harness Engineering" として整理し、Anthropicも "Effective Harnesses for Long-Running Agents" で同様のアプローチを発信しました。

ハーネスエンジニアリングの核心は、**決定論的に検証できることをLLMに判断させない**という原則です。

OpenAIの記事では、アーキテクチャの制約をカスタムリンターと構造テストによって**機械的に**適用していると述べています。ドキュメントの指示ではなく、ツールによる強制です。Birgitta Böckelerもこう書いています。

> Architectural constraints: Monitored not only by the LLM-based agents, but also deterministic custom linters and structural tests  
> ── [Birgitta Böckeler, "Harness Engineering"](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)

フォーマッター、リンター、型チェッカーのようなものは、AIに「これに従ってね」とお願いするのではなく、意識する間もなく自動的に実行される仕組みにするべきです。アーキテクチャに違反しているコードがあれば、人間がレビューで指摘するまでもなく、機械的に検出されるべきです。

このフィードバックループには二層の構造があります。

* **Inner loop（秒単位）**：linter / formatter / type checker → 即座にフィードバック
* **Outer loop（分単位）**：CI / アーキテクチャテスト → 構造的検証

Anthropicの "Effective Harnesses for Long-Running Agents" でも、テストをフィードバックループに組み込むことがハーネスの中核と位置づけられています。エージェントに対して「変更を実装したら必ずテストで検証し、テストが通るまで次に進むな」という構造を与えることで、自律的かつ信頼性の高い開発が可能になります。

## そのSkillはADR的な価値しかないのでは？

冒頭のSkillの内容をもう一度見てみましょう。「RepositoryにはInterfaceを定義する」「domain層はdata層に依存しない」「SOLID原則を遵守する」──これらは、「ある時点でチームがこうしていきたいと思っていた」という設計意思決定の記録です。

これはADR（Architecture Decision Record）に近い性質を持っています。チームの設計意思決定を機械可読な形で残す価値はあります。しかし、**宣言的なルール**をSkillに書いた場合、それはADR+αの域を出ません。

「何を実行するか」のワークフローが明確なSkill（例：PR作成やコミットメッセージ生成）とは性質が異なります。自動起動されないSkillはあくまで補完的なドキュメントであり、実際のガードレールとしての役割は果たしにくいです。

### ドキュメントは腐る

そして、ドキュメントには宿命的な問題があります。時間とともに腐ることです。

AIがコード生成速度を加速させるほど、この腐敗速度も加速します。コードベースの進化が早くなればなるほど、過去に残したドキュメントとの乖離は広がっていきます。OpenAIの記事でもこう指摘されています。

> マニュアルは即座に腐る。分厚いマニュアルは、古びた規則の墓場と化します。エージェントは真偽を見極められず、人間は更新を止め、ファイルは知らぬ間に厄介な存在となります。  
> ── [OpenAI "Harness Engineering"](https://openai.com/index/harness-engineering/)

だからこそ、ルールは**実行可能な形式**で残すべきです。

### テストを動作するドキュメントとして残す

では何を残すべきか。テストです。

テストは**動作するドキュメント**です。コードの実態と乖離した瞬間にCIで検出できます。テストの持つ「実行可能」「CIで検証可能」「ドリフト検知が自動」という特性は、AIエージェント時代においてこれまで以上に価値が高まっています。

Anthropicの記事でも、エージェントに対してテストによる自己検証を促すことが品質向上の鍵であると述べられています。

> Providing Claude with these kinds of testing tools dramatically improved performance, as the agent was able to identify and fix bugs that weren't obvious from the code alone.  
> ── [Anthropic "Effective Harnesses for Long-Running Agents"](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

### アーキテクチャテスト

冒頭のSkillが守りたかったルール──「domain層はdata層に依存しない」「domain層はAndroidフレームワークに依存しない」──を、テストとして表現する手段があります。アーキテクチャテストです。

例えば、[Konsist](https://docs.konsist.lemonappdev.com/)はKotlin向けの構造リンター（静的コード解析ツール）で、アーキテクチャの依存ルールをテストコードとして記述できます。

たとえば、「domain層は他のどの層にも依存しない」「UI層とdata層はdomain層に依存する」というクリーンアーキテクチャの基本ルールは、Konsistで次のように書けます。

```
class ArchitectureTest {
    @Test
    fun `clean architecture layers have correct dependencies`() {
        Konsist
            .scopeFromProject()
            .assertArchitecture {
                val domain = Layer("Domain", "com.myapp.domain..")
                val ui = Layer("UI", "com.myapp.ui..")
                val data = Layer("Data", "com.myapp.data..")

                domain.dependsOnNothing()
                ui.dependsOn(domain)
                data.dependsOn(domain)
            }
    }
}
```

宣言的なルールの羅列だったSkillの内容の一部が、実行可能なテストに変わりました。このテストがCIで毎回走れば、ドキュメントが腐る心配も、AIがルールを見落とす心配もありません。違反すればテストが落ちるので、エージェントは自律的に修正できます。

### テストによるガードがハーネスになる

守りたかったルールをテストとして残し、それがCIのフィードバックループに組み込まれる。これがハーネスエンジニアリングの構成要素そのものです。

Skill（advisory）とテスト（enforcement）を組み合わせることで、多層的なハーネスを構築できます。Skillは「なぜそうするか」の意図を伝え、テストは「そうなっているか」を機械的に検証する。この二層構造が、2026年現在のベストプラクティスの収束点です。

## まとめ

この記事では、具体的なSkillのレビューから出発して、ハーネスエンジニアリングという概念にたどり着きました。

Skillは有効なツールですが、設計段階で「いつ・誰が・どの頻度で起動するか」を考える必要があります。そして多くの場合、Skillに書きたくなるようなルールは、テストやリンターといった決定論的なフィードバックループに昇格させるべきです。

Human *on* the loopの世界では、エンジニアの仕事はコードを書くことだけでなく、エージェントが正しく動くためのハーネスを設計し改善し続けることに移っていくでしょう。

**あと、良いskillを作りたかったら普通にAnthropic公式の[skill-creator](https://github.com/anthropics/skills/tree/main/skills/skill-creator)を使いましょう。**
