最強と評された Mythos 5 のガードレール版として、Claude Fable 5 が登場しました。実務でこれまで最強格だった Opus 4.8 と比べて、どこまで優秀なのか。事前アナウンスの段階でトークン単価は Opus の約 2 倍とされ、今の使い方のままではコスト懸念が大きい。だとすれば問いは「速いか・賢いか」ではなく、**どの場面・どの対応で使うべきか**の見極めです。

筆者は個人の自動売買システムの開発で、この見極めを実機評価で行いました。本稿はその設計と、決め手になった一件——実装側モデルが創作した実在しない API フィールドを、Fable 5 が公式仕様を自分で取得して潰した——を、数値とともに残します。

---

## 先に数値で — Opus 4.8 と Fable 5

**前提（事前アナウンス）**: Fable 5 のトークン単価は Opus 4.8 の約 2 倍。「消費トークンが同じでもコストは倍」が出発点です。

**中難度 A/B**（同一プロンプトを両モデルへ・実装1問＋設計判断1問）

| 項目 | Opus 4.8 | Fable 5 |
| --- | --- | --- |
| 実装課題・正答 | ○（一発） | ○（一発） |
| 実装課題・所要 | 37.5 秒 | **8.2 秒** |
| 実装課題・消費トークン | 27,468 | 27,186 |
| 設計判断・所要 | 5.6 秒 | 8.1 秒 |

→ **正確性は互角・消費トークンはほぼ同等・実装は Fable が数倍速い**（37.5 秒 → 8.2 秒）。中難度の実装では対価が「単価 2 倍」だけで速度が大きく勝つ＝**額面は Fable 有利**。

**実タスク**（Fable をレビュアーに・設計＋コードレビューの 2 ラウンド）

| ラウンド | 消費トークン | ツール使用 | 所要 |
| --- | --- | --- | --- |
| 設計レビュー | 105,025 | 29 | 6.6 分 |
| コードレビュー | 124,717 | 48 | 16.2 分 |
| 合計 | **229,742** | **77** | **22.8 分** |

→ **高難度・長時間では消費そのものが重い**。単価 2 倍との掛け算でコストは無視できません。この重さに見合う価値が出るかが分岐点です。

**結論**

* 中難度では差はほぼ速度のみ。**「価値」として差が出たのは高難度の実タスク**
* そこで Fable が、実装側モデルの**ハルシネ（存在しない API フィールドの創作）を公式仕様の自走取得で検出**＝本番事故を未然に防いだ
* 採否は **採用＝用途限定**。本命は**独立アドバーサリアル・レビュアー**（実装は従来モデル × レビューは Fable）。加えて、**中難度の単発開発は Fable が速い**ので、反復の速さを取るならコスト 2 倍と引き換えに実装者として使う手もある

## 1. なぜ「実装者として」でなく評価したか

賢いモデルを「速い実装者」として使うと、評価は「同じ実装を速く正しく書けるか」になります。そこでまず、自己完結する中難度タスク（実装1問・設計判断1問）を**同一プロンプトで両モデルへ**投げる A/B を回しました。

結果は**互角**でした。両モデルとも一発で正答し、テストも揃い、消費トークンもほぼ同等。違いは速度だけで、実装課題では Fable のほうが明確に速い（前掲の表）。裏を返せば、**短い一発もの**では単価 2 倍を払う意味が薄い。差が出るとすれば、**自分が普段詰まる高難度の実タスク**だろうと当たりをつけました。

そこで角度を変えます。\*\*実装は従来モデルに任せ、新モデルは「その成果物を独立に検証するレビュアー」\*\*として、実タスクに投入しました。

---

## 2. 決定的だった一件 — 存在しない API フィールド

題材は、自動売買システムのリスク制御まわりの修正です。詳細は伏せますが、要は「株式戦略に紛れ込む ETF / ETN を、約定前に除外したい」という、ごく普通のフィルタ実装でした。

実装側モデルが書いたコードは、こうでした（趣旨を再現した擬似コード）。

```
# 実装側モデルが書いたコード(趣旨)
info = broker.get_symbol(code)
if info["SecurityType"] != "STOCK":   # ← この行が問題
    skip(code)
```

一見もっともらしい。`SecurityType`（証券種別）でフィルタする、と。

レビュアーとして投入した新モデルは、ここで止まりませんでした。**証券会社の REST API（kabu STATION API）の公式 swagger を自分で取得し、`/symbol` のレスポンス定義を grep して**、こう指摘してきます。

> `SecurityType` というフィールドはこの API の `/symbol` レスポンスに存在しない。この分岐は常に `KeyError`（または既定値）になり、フィルタは**本番で 100% 不発**になる。種別判定に使えるのは `TradingUnit`（売買単位）で、東証の通常株は 100、ETF/ETN は 1 や 10 になる。

実機で叩いて確認すると、まさにそのとおり。`SecurityType` は存在せず、`TradingUnit` が判別子でした（通常株=100、対象の ETN=1）。**実装側モデルは、それらしいが実在しないフィールド名を創作（ハルシネーション）していた**わけです。

これが恐ろしいのは、**コードもテストも「通って」しまう**点です。フィルタ対象が来なければ分岐は素通りし、ユニットテストはモック前提なので緑になる。**本番の実 API に当たって初めて死ぬ**。レビュアーがいなければ、この死にコードはそのままマージされていました。

新モデルは設計レビュー段階でも実装側の盲点を複数指摘し、コードレビュー段階では上記の Blocker に加えて軽微な指摘を数件返してきました。いずれも「ファイル:行・根拠・重大度・修正案」で構造化され、数値の検算や公式仕様の照合まで踏み込んだもので、表面的な指摘ではありませんでした。

---

## 3. なぜ「実装者」でなく「レビュアー」が効くのか

ここが本稿の主題です。同じ賢いモデルでも、置く場所で価値が変わります。

**実装者は、自分の前提を疑いません。** 実装に入ると「`SecurityType` というフィールドがありそうだ」という前提の上に話が積み上がり、自分でその前提を取りに行くインセンティブが働きにくい。これは賢さの問題ではなく**役割の問題**です。実装者が自分の成果物の盲点を見つけにくいのは、人間のコードレビュー文化が解いてきた問題と同じ構図です。

**レビュアーは、検証だけを仕事として与えられます。** 「この実装は正しいか、一次資料に当たって反証せよ」という指示は、賢いモデルの「自分でツールを使って根拠を取りに行く」能力を、ハルシネ検出という一番効く方向に向けます。実際、決め手になったのは推論の賢さそのものより、**公式 swagger を自分で取得して照合した**という行動でした。

提供元の公式ドキュメントも、この世代のモデルについて**コードレビューとデバッグ（バグ発見の再現率）の向上**に言及しています。今回の結果は、その特性を「実装の置き換え」ではなく「独立検証」に振り向けたときに最大化される、という実地の裏づけになりました。

> 一般化すると、**新しく強いモデルは「主役の交代」より「検証レイヤーの新設」に使うほうが、リスク調整後のリターンが高い**ということです。実装を 1.1 倍速くするより、本番事故を 1 件防ぐほうが効く局面は多い。

---

## 4. コスト — 重い。だが何と引き換えか

正直に書くと、レビュアーとしての Fable は**消費が重い**です（前掲のとおり 2 ラウンドで約 23 万トークン・約 23 分）。コードレビューが 16 分と最も重いのは、**外部の公式 swagger を取得し、関連テストをひととおり走らせて検証した**から。深さの裏返しです。さらにトークン単価は Opus の約 2 倍——消費が重い領域でこそ、その 2 倍が効いてきます。

判断軸は「1 トークンあたりの単価」ではなく「**タスク完了までの総コストと、防げた損失**」です。誤った API 実装を本番に投入してしまえば、不発フィルタの発見・原因切り分け・再修正・再デプロイ、最悪は実害が乗ります。それと約 23 万トークンを天秤にかければ、**重いレビューでも割に合う**。逆に、日常の軽作業や、結果が安く検証できるタスクに毎回この重さを払うのは過剰です。だから**用途を限定**します。

---

## 5. 運用への落とし込み

実機評価から、Fable の使いどころは 2 つに絞れました。どちらも「狭く」使うのがコツです。

### (1) 高難度の独立レビュアー（本命）

* 重要な実装（外部 API 連携・リスク制御・不可逆な処理）に、Fable を独立レビュアーとして 1 枚かませる
* 指示は「**一次資料に当たって反証せよ。確信が持てなければ未確定と明記せよ**」
* 実装そのものは従来モデルで十分。価値は「実装者が見落とすものを潰す」点にある

### (2) 中難度の高速実装者（速さを取るとき）

中難度の A/B では、**正確性は互角**（サンプルでは両者とも一発正答）で、**実装（コード生成）は Fable が数倍速い**（37.5 秒 → 8.2 秒）。額面どおりに読めば、これは中難度の実装作業における **Fable の明確な優位**です——唯一の対価がトークン単価 2 倍。対話的に回す開発では待ち時間こそが体感を左右するので、**コストを許容できるなら、中難度の実装はむしろ Fable を既定にしてよい**。

ただし速度差は一様ではありません。同じ中難度でも、設計判断のような「書く量が少なく考える」タスクでは逆に Opus が速い場面もありました（サンプルは小さく、タスク種別で差が出ます）。Fable の速さが効くのは**コードを書く量がある**中難度です。

ここでいう「中難度の開発」とは、**仕様が明確・スコープが限定・正しさが検証可能で、横断調査やアーキ判断をほとんど含まない**単位作業です。具体的には:

* 仕様が決まった単機能の実装（データ変換・整形、入力バリデーション、正規表現・小さなパーサ）
* 既存パターンに沿った CRUD / API エンドポイント 1 本の追加
* 再現条件が分かっているバグの修正
* ユニットテストの新規作成・拡充
* 範囲を限定したリファクタ、ボイラープレート / スキャフォールドの生成
* 単発のデータ整形スクリプト、型定義やスキーマの起こし

いずれも「何を作るかは決まっていて、あとは正確に速く書くだけ」のタスク。本稿の A/B で使った関数実装も、ちょうどこの層でした。逆に、**低難度の雑用や大量の並列ワーカー**は対象外です（コストが支配的で、速度差も価値になりにくい）。

---

この「**高難度はレビュー・中難度は高速実装**」の使い分けと、(1) の検証用スキル（マージ前に API 仕様やハルシネを潰すチェックリスト）化をセットにすると、人手のレビュー規律と同じく**再現性**が出ます。

---

## まとめ

* 賢いモデルの初手は「実装の主役交代」になりがちだが、実機評価で最も効いたのは **独立アドバーサリアル・レビュアー**としての採用
* 中難度は品質互角だが **Fable が速い**（実装で 4 倍以上）。「価値」としての差は高難度の実タスクで、**ハルシネ検出**という形で出た
* 決め手は推論の賢さより、**公式仕様を自分で取得して照合する行動**。実装側が創作した実在しない API フィールドを、本番に出る前に潰せた
* 使い分けは 2 つ: **高難度＝独立レビュー**（事故を防ぐ）／ **中難度＝高速実装**（反復を速める・例: 仕様明確な単機能実装・限定リファクタ・テスト作成）。どちらもコスト 2 倍に見合う狭い用途に絞る
* コストは重い（レビュー 1 回で約 23 万トークン）が、防げる事故と引き換えなら正当
* 強いモデルは「速い実装者」としても使えるが、**最大のリターンは「疑り深いレビュアー」に置いたとき**
