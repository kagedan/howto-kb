## はじめに：なぜ今、認証を問い直すのか

AIエージェントを使ったアプリケーションを作るとき、私たちは当たり前のように `.env` に API キーを入れています。

```
OPENAI_API_KEY=sk-...
SERVICE_API_KEY=...
```

人間がツールを直接使う前提では、この方法はシンプルで便利でした。

しかし、AIエージェントが

* 自律的に外部APIを呼び出す
* 長時間動き続ける
* 外部入力（プロンプト、Webページ、ツール結果）の影響を受ける

という前提になると、この設計に違和感が出てきます。

違和感の正体は何か。この記事ではそれを言語化しながら、**APIキーを配る代わりに、鍵ペアで認証する**というシンプルな設計を考えます。

---

## 1. Bearer Tokenの本質的な性質

多くのAPIは、いわゆる Bearer Token で認証しています。

```
Authorization: Bearer xxxxx
```

この仕組みの本質は、

> **その文字列を持っている主体が、そのまま使える**

という点にあります。

スコープや有効期限で制限できる場合もありますが、基本的な性質は変わりません。「知っている者が使える」という設計です。

人間が管理する分には問題になりにくいですが、AIエージェントに渡すと、リスクの性質が変わります。

* **プロンプトインジェクション**：外部入力（Webページの内容、メールの本文など）経由でAPIキーをログや外部サービスに送信させられるリスク
* **コンテキスト露出**：LLMのコンテキストウィンドウにAPIキーが乗ると、ツール呼び出しの結果や要約に混入する可能性がある
* **意図しない呼び出し**：エージェントが持つ権限と、実際に必要な権限のミスマッチ

これらは「セキュリティが甘い」の問題ではなく、**「誰が使うか」という前提が変わった**ことから来る構造的な問題です。

---

## 2. 別のやり方：リクエストに署名する

APIの認証は「秘密の文字列を送る」以外の方法もあります。

金融系やクラウドAPIでは、

> **リクエストそのものに対して署名を付ける**

という方法が使われています（AWS Signature V4 など）。

フローはこうです。

1. クライアントがリクエスト内容を決める
2. その内容に対して、手元の秘密情報で署名する
3. サーバーにリクエストと署名を送る
4. サーバーが公開鍵で署名を検証する

この方法では、**秘密そのものをネットワークに流す必要がありません。**

また、署名の対象に以下を含めることで、改ざんやリプレイ攻撃も防ぎやすくなります。

* `method`
* `path`
* `body` のハッシュ
* `timestamp`
* `nonce`（任意）

ここで重要なのは、発想の転換です。

> **秘密を渡すのではなく、「このリクエストに同意した」という証明を渡す**

## 3. エージェントが鍵ペアを持つ設計

ここまで来ると、シンプルな発想に至ります。

> **AIエージェント自身が鍵ペア（公開鍵・秘密鍵）を持てばよいのでは？**

構成はこうなります。

### サービス側

公開鍵（またはその識別子）を登録しておく。

### エージェント側

自分の秘密鍵でリクエストに署名する。

### 実際のHTTPリクエストはどう見えるか（イメージ）

署名や timestamp をリクエストに含める方法は、大きく二つあります。

**① カスタムヘッダーに含める（一般的な慣習）**

```
POST /api/resource HTTP/1.1
Host: api.example.com
Content-Type: application/json
X-Agent-Key-Id: agent-pub-key-001
X-Timestamp: 2026-04-08T00:00:00Z
X-Signature: <base64urlエンコードされた署名バイト列>

{"key": "value"}
```

※ `X-Signature` の値は概念的に示しています。実際には `method + "\n" + path + "\n" + bodyHash + "\n" + timestamp` のような形で正規化した文字列に対して秘密鍵で署名し、その結果をbase64urlエンコードした値が入ります。

`Authorization` ヘッダーを拡張する形もあります。

```
Authorization: AgentSig keyId="agent-pub-key-001",
               timestamp="2026-04-08T00:00:00Z",
               signature="base64url(...)"
```

AWS Signature V4 や HTTP Message Signatures（RFC 9421）はこの方向性の標準実装です。署名をヘッダーに分離することで、ボディの内容を変えずに認証情報を付与できる点が重要です。

**② ボディに含める（JSON-LD Proof など）**

```
{
  "payload": {"key": "value"},
  "proof": {
    "type": "DataIntegrityProof",
    "cryptosuite": "eddsa-rdfc-2022",
    "created": "2026-04-08T00:00:00Z",
    "verificationMethod": "agent-pub-key-001",
    "proofValue": "base64url(...)"
  }
}
```

こちらはVerifiable Credentials（W3C Data Integrity仕様）などで使われるアプローチで、ペイロードと署名を一体として扱います。REST APIより分散システム・Web3との親和性が高いです。

**一般的な選択**：REST APIであればヘッダー方式が自然です。既存のルーター・ミドルウェアとの相性がよく、ボディのパース前に認証処理を差し込めます。

### サーバー側の検証

```
verifySignature({
  publicKey,
  method,
  path,
  bodyHash,
  timestamp,
  signature,
})
```

これにより、

> **「このリクエストは、この公開鍵に対応する主体が送った」**

ことを確認できます。APIキーのような「秘密の文字列」を配らなくても、鍵の持ち主であることを証明してAPIを使えるようになります。

## 4. APIキーと鍵ペアの違いを一言で

この違いは、こう言い換えられます。

> APIキーは「権限を持った文字列」を配る設計  
> 鍵ペアは「主体そのものを識別する」設計

APIキーは**コピーできるモノ**です。流出したら、それを手に入れた誰でも使えます。

一方で鍵ペアは「その秘密鍵を持つ主体」に紐づきます。漏洩リスクがゼロになるわけではありませんが、**「誰が送ったか」というアイデンティティに署名が結びつく**点が本質的に異なります。

この違いは、AIエージェントのように**自律的に動く主体**が増えるほど重要になります。

暗号通貨ウォレットがまさにこの設計で、「秘密鍵の持ち主 = その資産を動かせる主体」という考え方はブロックチェーンの世界ではすでに当たり前です。エージェントが経済活動を行う未来を考えると、認証とウォレットが同じ鍵ペアに収束していくのは自然な方向かもしれません。

---

## 5. 鍵ペア方式の落とし穴：秘密鍵はどこに置くのか

ここまで読んで、

> 「それって昔からある公開鍵認証では？」

と思った方は正しいです。この記事で言っていることは新しい暗号技術ではありません。

ただし、前提が変わっています。

> **APIを叩く主体が、人間ではなくAIエージェントになる**

この前提で見たとき、見落としやすい問題があります。

**秘密鍵をどこに置くか、という問題です。**

APIキーを `.env` に入れる問題が、秘密鍵を `.env` に入れる問題に置き換わるだけでは意味がありません。

鍵ペア方式が意味を持つのは、**秘密鍵がエージェントのコンテキスト（＝LLMが見られる領域）から分離されているとき**です。

この問題に対する実用的なアプローチとして、以下のような選択肢があります。

| アプローチ | 概要 | 特徴 |
| --- | --- | --- |
| OS の Keychain / Secret Store | macOS Keychain、Linux の Secret Service など | ローカル環境では手軽 |
| クラウド KMS | AWS KMS、Google Cloud KMS | 署名操作を API 経由で委譲。秘密鍵をプロセスに渡さない |
| TEE / HSM | Intel SGX、ハードウェアセキュリティモジュール | 最も堅牢。ただし導入コストが高い |
| 署名専用プロセスの分離 | 署名処理だけを別プロセス・別サービスに切り出す | ソフトウェアのみで実現可能な現実解 |

特に**クラウド KMS や署名専用プロセス**のアプローチは、「秘密鍵をエージェントが直接持たない」設計を実現します。エージェントは署名を**依頼する**だけで、鍵そのものには触れない。

これは「エージェントが鍵を持つ」設計から一歩進んで、

> **エージェントは鍵を持たない。鍵を使う権限だけを持つ**

という分離です。この方向性は、エージェントの自律性が高まるほど、より自然に見えてきます。

---

## 6. これは設計の問題であり、技術の問題ではない

鍵ペア認証は新しい技術ではありません。しかし、AIエージェントという新しい主体に適用するとき、問われるのは技術的な実装よりも**設計の問いの立て方**です。

| 問い | 従来の前提 | AIエージェント時代の前提 |
| --- | --- | --- |
| 誰が使うか | 人間（または信頼できるサーバー） | 自律的に動くエージェント |
| 鍵の管理者は誰か | 人間が直接管理 | エージェントのライフサイクルに依存 |
| 流出時のリスク | 人間が気づいて無効化できる | 気づく前に使われる可能性がある |
| 権限の粒度 | API単位 | リクエスト単位で制御したい |

AIエージェントに認証を持たせるとき、「どうやって安全に使わせるか」より先に「**そもそも何を証明したいのか**」を問うべきです。

* 「このエージェントが送った」ことを証明したいのか
* 「このリクエストが改ざんされていない」ことを証明したいのか
* 「このエージェントに、このリソースへのアクセスを許可した」ことを証明したいのか

これらは微妙に異なる問いであり、鍵ペア認証はそのうちの最初の二つに応えるものです。三つ目は別途、認可の設計が必要になります。

この「認証と認可を組み合わせる」問題に対して、既存の標準仕様がすでに一つの答えを出しています。

**OAuth 2.0 の DPoP（Demonstrating Proof-of-Possession）** です。

DPoP は、アクセストークン（認可）に加えて、「このHTTPリクエストが、そのアクセストークンと紐づいた鍵の持ち主から送られた」ことを証明するJWTをリクエストごとに付与する仕組みです。

```
Authorization: DPoP <access_token>
DPoP: <signed_jwt_proof>
```

Bearer Token の弱点だった「トークンを盗んだ者が使える」問題を、鍵ペアによるリクエスト単位の署名で補強しています。RFC 9449 として標準化されており、AIエージェントの認証文脈でも参照する価値があります。

ただし DPoP はあくまで OAuth の文脈（認可サーバーが存在する前提）での解決策です。エージェント間の直接通信や、認可サーバーを介さないシンプルなAPI認証には、この記事で触れてきた鍵ペア署名の方が適合することもあります。

---

## まとめ

この記事で伝えたかったことは、

> AIエージェントにAPIキーを持たせる代わりに、**鍵ペアで認証させるという設計は、もっと自然かもしれない**

という一点です。

ただし、それだけでは不十分で、

* **秘密鍵の格納場所**（エージェントのコンテキストから分離すること）
* **署名の対象と形式**（クライアント・サーバー間での合意）
* **認証と認可の分離**（「誰か」の証明と「何ができるか」の制御は別の問題）

という三つの問いが必ずついてきます。

AIがAPIを叩くのが当たり前になるなら、「誰がそのリクエストを送ったのか」をどう証明するかも、少しずつ変わっていくはずです。

> **主体 = 鍵を持つもの**

というより、

> **主体 = 鍵を使う権限を持つもの**

という設計が、思っているよりしっくりくるかもしれません。
