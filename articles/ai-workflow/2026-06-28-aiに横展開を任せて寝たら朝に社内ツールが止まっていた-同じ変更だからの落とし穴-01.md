## はじめに

ある修正を、1つのリポジトリで正しく完了させました。3ファイルの変更で、レビューもブラウザ確認も通し、問題なく動いた。十分に検証できた手応えがありました。

問題は、その「成功体験」を、残りのリポジトリへAIに横展開させたあとに起きました。

深夜、AIエージェントに「最初のリポと同じ変更を残りに展開して」と自走モードで指示し、私は就寝しました。翌朝アクセスすると、社内管理ツールが**全リクエストでエラーを返す状態**になっていました。

原因は単純です。最初のリポでは「ファイル2つの削除＋設定ファイルからの参照1行の除去」という3点セットで完結していたのに、横展開された側では**ファイル2つは消えたのに、参照1行が消えていなかった**。存在しないクラスを参照したまま本番に出ていたわけです。

そして本質的な問題は、その実装漏れそのものよりも、**「同じ変更だから」という安心感が、レビューも環境確認も検証も、まとめて省略させていた**ことにありました。

先に影響範囲を書いておきます。

* 影響を受けたのは**社内専用ツール**で、一般のお客様向けの機能は**今回のリリース対象に含めていませんでした**
* 発生したのは**休日の早朝**という、アクセスが極めて少ない時間帯
* 実際にエラーに当たったのは**1名（筆者自身）**、復旧までおよそ30分

一般のお客様が触れる機能をリリース対象から外していたことが、せめてもの救いでした。影響は社内ツールに閉じ、お客様向けのサービスには波及していません。とはいえ、社内ツールとはいえ本番を止めたのは事実で、確認を一つ挟んでいれば越えずに済んだ一線でした。これは記録して教訓化する価値がある、と思いました。

以前、[8リポジトリに同じ変更を並列展開したらCopilotレビューのばらつきがシグナルになった話](https://zenn.dev/fixu/articles/parallel-repo-copilot-review-signal)という記事を書きました。あれは横展開でレビューを「使い倒した」話です。今回はその裏面、\*\*横展開でレビューを「省いた」\*\*側の話になります。

## 1. 背景 — 読み取り専用レプリカが、全サービスで効いていなかった

ことの発端は、横展開した修正そのものではなく、その修正が直そうとしていた別の問題でした。順番に説明します。

データベースは、書き込み用（Writer）と読み取り専用レプリカ（Reader）に分かれた構成でした。負荷分散のため、参照系のクエリはReaderに流す設計です。Laravelでは接続設定で`read`/`write`を分け、あわせて`sticky`という設定をよく使います。

```
'mysql' => [
    'read'   => ['host' => env('DB_READ_HOST')],  // 読み取り専用レプリカ（キー名は一般化）
    'write'  => ['host' => env('DB_WRITE_HOST')], // 書き込み用
    'sticky' => true,
],
```

`sticky: true`は、「同一リクエスト内で一度でも書き込みが発生したら、それ以降のSELECTは（レプリカではなく）Writerに向ける」という設定です。書き込んだ直後に自分の書き込みを読み返す際の不整合を防ぐ、read-after-write consistencyのための正当な仕組みで、これ自体は何も悪くありません。

問題は、**全リクエストで実行されるグローバルミドルウェア**が、リクエストのたびにアクセス記録を1行INSERTしていたことでした（以下、クラス名は一般化しています）。

```
// 全リクエストで実行されるグローバルミドルウェア（名称は一般化）
public function handle(Request $request, Closure $next)
{
    // アクセス記録を1行INSERTしているだけ
    AccessLog::insert([
        'url'          => $request->url(),
        'requested_at' => now(),
    ]);
    return $next($request);
}
```

このINSERTが、Laravel内部で次の連鎖を引き起こします（以下はフレームワーク本体のコードで、公開情報です）。

```
// Illuminate\Database\Connection
// INSERT/UPDATE/DELETE はこの statement() を通り、
// 無条件に「書き込みが起きた」フラグを立てる
public function statement($query, $bindings = [])
{
    // ...
    $this->recordsHaveBeenModified(); // ← 無条件
    // ...
}

// sticky: true のとき、一度でも書き込みがあれば
// 後続の SELECT は Reader ではなく Writer に向かう
public function getReadPdo()
{
    if ($this->recordsModified && $this->getConfig('sticky')) {
        return $this->getPdo(); // ← Writer を返す
    }
    return $this->readPdo;       // ← 本来はこちら（Reader）
}
```

つまり、**全リクエストの入口で必ず1回INSERTが走る** → その時点で「書き込みが起きた」フラグが立つ → `sticky`が効いて、**そのリクエスト内の以降のSELECTがすべてWriterに向かう**。結果として、読み取り専用レプリカが、全サービスの全リクエストで一切使われていませんでした。負荷分散のために用意したレプリカが、設計意図と裏腹に丸ごと遊んでいたわけです。

さらに調べると、このアクセス記録テーブルはINSERTされるだけで、**SELECTしている箇所がコードベースのどこにもありません**でした。Webサーバーのアクセスログや別系統のログと完全に重複する、書き込み専用のデータです。

直し方は明快でした。INSERTを行うこのミドルウェアと、対応するモデルを削除し、**設定ファイル（Laravelのミドルウェア登録配列）から参照の1行も外す**。この3点がワンセットです。

> **1章の蒸留**: 入口で必ず走る小さなINSERTひとつが、`sticky`と噛み合って、全サービスの読み取り分散を無言で無効化していた。直すべきは「ファイル削除」だけでなく「参照の除去」までを含む3点セット。

## 2. 最初の成功が、罠の入口だった

最初に直したのは、影響範囲を見極めるために選んだ1リポジトリでした。

変更は3ファイル。ミドルウェアのファイル削除、モデルのファイル削除、そして登録配列からの参照1行の除去。AIエージェントに任せつつ、Copilotレビューもトリガーして指摘を確認し、開発環境・検証環境に順にデプロイして、ブラウザで実際にアクセスして正常を確かめました。きれいに通りました。

ここで手応えを得たのが、いま振り返ると最初の分岐点でした。「3ファイル変更で、レビューも確認も通って、ちゃんと動いた」という成功体験。これが次の判断を歪めます。**残りは"同じ変更"なのだから、同じようにうまくいくはずだ**、と。

横展開そのものは、AIエージェントが得意とする作業です。実際、機械的なパターン複製はAIに任せるのが効率的で、私もそれを前提に運用しています。問題は、横展開の「実行」をAIに任せたことではなく、**横展開の「検証」まで一緒に省いてしまった**ことでした。

> **2章の蒸留**: 1リポでの成功は「この変更は正しい」を保証するが、「残りも正しく適用される」は保証しない。成功体験は、次の確認を省くための言い訳になりやすい。

## 3. 横展開で何が抜けたか — 三重の省略

横展開した側で起きていたことを、層に分けて整理します。穴は一つではなく、三重に重なっていました。

### 省略その1: 実装が1行足りなかった

最初のリポは3点セット（ファイル2つ削除＋参照1行除去）で完結していました。ところが横展開された側は、**ファイル2つの削除だけが行われ、設定ファイルの参照1行が残っていました**。

Laravelは、グローバルミドルウェアをリクエスト処理の入口で全部インスタンス化しようとします。参照されているクラスのファイルが存在しないと、その解決に失敗し、**全リクエストが処理の入口で例外になります**。エラーは特定の画面ではなく、トップページを含むあらゆるリクエストで一様に発生しました。

「ファイルを消す」と「ファイルへの参照を消す」はワンセットです。片方だけだと、参照が宙に浮きます。この「セットのうち1つを落とす」のは、機械的な複製作業でいちばん起きやすいタイプの欠落です。

### 省略その2: 「同じ変更だから」でレビューを外した

最初のリポではCopilotレビューをトリガーし、指摘を確認していました。ところが横展開の各リポでは、**「同じ変更だから」としてレビューをトリガーしませんでした**。

皮肉なのは、横展開こそレビューが効く場面だということです。前掲の[Copilotレビューのばらつきの記事](https://zenn.dev/fixu/articles/parallel-repo-copilot-review-signal)で書いたとおり、同じ変更をN本に展開したときのレビューの集約は、機械的な欠落を拾う網になります。「同じ変更」だからこそ、1本で漏らした指摘を別の本が拾える。その網を、自分から畳んでいました。

### 省略その3: 環境にデプロイしても、実際に開かなかった

開発環境へ自動デプロイしたあと、**エラーログもブラウザも確認しませんでした**。検証環境では、監視メトリクスの「エラー0件」だけを見て先に進みました。けれどそれは、\*\*実際にアクセスが発生していない時間帯の「0件」\*\*でした。一度ブラウザでトップページを開いていれば、即座にエラーに気づけたはずです。

「デプロイが完了した」と「正しく動いている」はまったく別のことです。デプロイの成功は、コードが本番に配置されたことしか意味しません。配置されたコードが正しく応答するかどうかは、実際にリクエストを投げてみないと分かりません。この当たり前の一線を、3つの環境すべてで越えずに通過していました。

| 段階 | やったこと | 抜けたこと |
| --- | --- | --- |
| 実装 | ファイル2つを削除 | 参照1行の除去 |
| レビュー | （最初のリポのみ実施） | 横展開分はトリガーせず |
| 開発環境 | 自動デプロイ | エラーログ・ブラウザ確認 |
| 検証環境 | メトリクスで0件確認 | 実アクセス（無人時間帯の0件） |

> **3章の蒸留**: 実装漏れ単体なら、レビューか環境確認のどちらかが生きていれば止められた。事故は「一つのミス」ではなく「複数の安全網を同時に外したこと」で起きる。

## 4. 寝る前の自走モード — 過信と、6時間の空白

ここが、いちばん書き残したかった部分です。

私は深夜、AIエージェントに横展開を自走モードで指示し、就寝しました。AIが実装し、環境にデプロイし、判断して先に進む — その一連を、私が眠っている間に任せたわけです。

問題は、**AIの実装能力と検証能力が、まだその委譲幅に見合っていなかった**ことです。横展開で1行落とすことも、メトリクスの0件を「正常」と早合点することも、現時点のAIには十分起こりえます。それを承知していたはずなのに、「同じ変更だから」「最初のリポで成功したから」という気の緩みで、確認を伴わない権限委譲をしてしまった。

結果として生まれたのが、**6時間近い空白**でした。本番デプロイが完了したのが早朝、私が異常に気づいたのが朝。その間ずっと、誰も見ていない本番環境で、壊れたコードが配置されたまま放置されていました。社内ツールが対象で、休日早朝という時間帯でもあったため実害には至りませんでしたが、誰も見ていない時間に確認なしで本番を変えたという事実は残ります。

[サブセッションの自走化について書いた記事](https://zenn.dev/fixu/articles/subsession-self-drive-guardrails)で、私は「任せる」と「丸投げ」は別物で、自走を成立させるには細部の業務設計が要る、と書きました。今回はその細部設計を、自分で省いた側の実例です。理屈は分かっていても、深夜の「もう寝たい」という気分の前では、その理屈が簡単に後回しになりました。

異常に気づいてからロールバックに入り、事前に手順を用意していたこともあって、復旧そのものはおよそ30分で完了しました。

ここで一つ、せめてもの救いがありました。今回の横展開は、**一般のお客様が触れる機能をリリース対象に含めていなかった**ことです。リリースのスコープを社内側に絞っていたため、影響は社内ツールに閉じ、お客様向けのサービスには波及していません。

とはいえ、社内ツールとはいえ本番を止めたことに変わりはありません。スコープの狭さに助けられはしましたが、それは「確認して防いだ」のとは違います。確認を一つでも挟んでいれば、本番反映そのものを止められた一線でした。

| 時刻(JST) | 出来事 |
| --- | --- |
| 深夜 | 最初のリポで修正成功（3ファイル、検証済み） |
| 深夜 | 残りへ横展開を自走指示 → 就寝 |
| 深夜 | 各環境へ自動デプロイ（エラーログ・ブラウザ確認なし） |
| 早朝 03:51 | 本番デプロイ完了（お客様向け機能はリリース対象外） |
| 朝 09:17 | 社内ツールにアクセス → 全リクエストエラーに気づく |
| 朝 09:50頃 | ロールバック完了、復旧 |

> **4章の蒸留**: AIの自走幅は「翌朝の自分が確認できる範囲」に留めるべきだった。確認を伴わない深夜の権限委譲は、誰も見ていない本番に空白の時間を生む。影響が社内に留まったのはリリーススコープを絞っていたおかげで、確認を省いた判断そのものが正されたわけではない。

## 5. まとめ — AI横展開のチェックリスト

今回の経験を、自分のワークフローに差し戻せる形に蒸留します。AIに横展開を任せる場面で、私が次から守るチェックリストです。

* **1リポの成功は、横展開の安全を保証しない。** 「変更が正しい」と「全リポに正しく適用された」は別の命題。差分の完全性は、リポごとに独立して確かめる。
* **「同じ変更だから」はレビューを省く理由にならない。** むしろ横展開こそ、機械的な欠落が最も起きやすく、レビューの網が最も効く場面。最初の1本で通したレビューを、残りで畳まない。
* **環境確認は「実際に開く」までやって1単位。** デプロイ成功やメトリクスの0件は、正常の証明ではない。無人時間帯の0件は、ただアクセスがないだけかもしれない。ブラウザで1リクエスト投げる、を必須ゲートにする。
* **自走の幅は「翌朝の自分が確認できる範囲」に。** AIの実装・検証能力の現在地を正確に見積もり、それを超える委譲を、確認なしの本番反映と組み合わせない。
* **AIの実装には穴がある前提でワークフローを組む。** 穴があること自体は責められない。責められるのは、穴を拾う網（レビュー・環境確認）を同時に外したこと。

そして最後に、薄く、けれど自分にとっては芯を食っている教訓を一つ。

今回いくつもの確認を省いた背景には、正直に言えば、**AIのトークン（ClaudeやCopilot）と作業時間を惜しむ気持ち**がありました。レビューをもう一周回せば、その分のトークンと待ち時間がかかる。環境を一つずつ実際に開けば、その分だけ手間がかかる。深夜の「早く終わらせて寝たい」という気分の前で、その小さなコストが妙に大きく見えました。

けれど今回、その節約で浮いたトークンと数十分は、復旧と原因究明にかかった時間と労力に、まるで釣り合いません。**検証コストをケチる誘惑は、AIネイティブな開発で新しく現れた、断ち切るべき誘惑**だと思いました。手作業の時代は「面倒だから」で省いていたものが、いまは「トークンがもったいないから」「AIなら速いはずだから」に形を変えて、同じ場所で同じ確認を省かせてくる。誘惑の見た目は変わっても、省いた先で待っているものは変わりません。

影響が社内ツールに留まったのは、お客様向けの機能をリリース対象から外していた、せめてもの救いでした。それを、次に同じ判断をしないための材料に変えられたら、この朝の30分にも意味があったと思えます。

同じようにAIへの横展開を運用している方の、確認を一つ省きそうになった瞬間のブレーキになれば幸いです。
