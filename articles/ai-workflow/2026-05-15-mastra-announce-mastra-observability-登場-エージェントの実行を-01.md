## 発表の概要

2026年5月13日、Mastra は **Mastra Observability** を発表しました。

アナウンス本文の冒頭を意訳すると、こうです。

> Mastra Observability は、エージェントのログ、トレース、メトリクスを検索可能な形で保持し、すべてのプロジェクトやデプロイを横断して見られるようにする。

ポイントは「**すべてのプロジェクトやデプロイを横断して**」というところです。単に「ログが見られるようになった」という話ではありません。この記事では、それが誰にとって、なぜ重要なのかを掘り下げます。

## 背景を整理する

### Observability はもともとあった

Mastra の Observability 機能は、今回初めて登場したわけではありません。Mastra 1.0 系から `PinoLogger` による構造化ログや `@mastra/observability` によるトレース収集はすでに存在しており、ローカルの Studio で見たり外部の OTEL 互換基盤へ送ったりする仕組みは以前から整っていました。

詳細は以前書いた記事をご覧ください。

<https://zenn.dev/shiromizuj/articles/71866dccab1931>

### Mastra Platform が4月にローンチした

そして先月（2026年4月）、Mastra は **Mastra Platform** をローンチしました。

<https://zenn.dev/shiromizuj/articles/ba80b4694382b3>

Platform ローンチの本質は「Studio / Server / Memory Gateway を3つの独立したプロダクトに分解した」ことでした。つまり一体型だったアプリが、「観測・デバッグ・評価」「実行・API 公開」「状態管理」という3つの責務に分解されたわけです。

その時点で、Studio には観測機能（ログ、トレース、メトリクス）がすでに含まれていました。では今回の発表で何が変わったのか。

### Studio の観測機能が持っていた制約

Platform ローンチ時点での観測機能には、2つの制約がありました。

* **観測データが Studio インスタンス単位でまとまっていた**
* **使い始めるには Studio のデプロイが必要だった**

この制約の何が問題か。それは「**複数のプロジェクトやデプロイを持ちはじめたとき**」に一気に顕在化します。

## 誰のための横断検索か

「プロジェクトやデプロイをまたいで検索できるようになった」という話は、単一のエージェントを開発・運用する人には無関係です。そしてもちろん、エージェントのエンドユーザー（システムの利用者）でもありません。では誰が嬉しいのか。

### １：マルチエージェント構成を運用している人

これが最も直接的な受益者です。そして、Mastra が強く推しているスーパーバイザーパターンを使っている人なら、この必要性を身をもって感じているはずです。

Mastra のスーパーバイザー構成では、スーパーバイザーエージェントが複数のサブエージェントをツールとして呼び出します。そのサブエージェントが別プロジェクトや別デプロイに存在することも珍しくありません。

このとき、サブエージェント内で何か失敗が起きても、スーパーバイザー側のトレースには「ツール呼び出しが失敗した」としか記録されません。サブエージェントの中で何が起きたか、どのモデル呼び出しがどういう応答を返したか、どこで時間がかかったかは、別の Studio インスタンスを開いて手動で照合しなければならない。

```
スーパーバイザーエージェント（プロジェクトA）
  ↓ ツール呼び出し
  サブエージェントB（プロジェクトB）  ← ここで失敗
  サブエージェントC（プロジェクトC）
```

これはマイクロサービス時代にエンジニアが直面してきた「**分散トレーシング問題**」と同型です。サービスAがサービスBを呼び、サービスBがサービスCを呼ぶ構成で、C側の障害がA側のログにしか現れないとき、Jaeger や Zipkin のような分散トレーシングツールが解決策として登場しました。

Mastra Observability が「プロジェクト横断・デプロイ横断でトレースを検索できる」というのは、マルチエージェントシステムに対する分散トレーシングの提供です。このフレーミングで見ると、今回の発表の意味が一段クリアになります。

### ２：複数プロジェクトを束ねるチームリーダー・プラットフォームエンジニア

複数の AI エージェントプロジェクトを横断して管理する立場の人にとっては、**コスト配賦と品質の全体把握**が切実な問題です。

* 「今月のLLMコストはどのエージェントが一番使っているか」
* 「先週からエラー率が上がっているのはどのデプロイか」
* 「新しいモデルに切り替えたプロジェクトと切り替えていないプロジェクトで、レイテンシはどう変わったか」

こういった問いは、プロジェクトをまたいだデータを横断集計しないと答えが出せません。Studio インスタンスが分散していると、確認のたびに手動でタブを切り替えながら足し算しなければならなくなります。

### ３：Observability は「OLAP」だから、共有するインセンティブが高い

Memory と Observability を対比すると、共有化の動機がより明確になります。

|  | Memory（状態管理） | Observability（観測データ） |
| --- | --- | --- |
| データアクセスのパターン | 会話ごとに低レイテンシで読み書き | 過去の大量データへの集計クエリ |
| データベースの種類 | OLTP（PostgreSQL、LibSQL） | OLAP（ClickHouse） |
| エージェントとの距離感 | 近くに置くのが自然 | 集約してこそ価値が出る |
| 共有インセンティブ | 薄い（会話コンテキストはエージェント個別） | 高い（比較対象が増えるほど insight の価値が上がる） |

OLAP 型の観測基盤には「**集積するデータが増えるほど比較の価値が上がる**」というネットワーク効果があります。モデルの比較、デプロイ間の性能差、時系列での品質推移、こういった insight はデータが分散していると出てきません。

さらに、ClickHouse のような列指向データベースは構築・運用のコストが高い。メモリ向けに Postgres を1台立てるより、観測基盤を1つ立てる方が運用のハードルが上がります。だからこそ、**複数プロジェクトで1つの観測基盤を共有するインセンティブが Memory より明確に大きい**のです。

### ４：共有 RAG のデータオーナー（補足）

少し視点を変えると、こんなケースも浮かびます。

複数のエージェント（A、B、C）が共通の RAG 知識ベースを参照している場合、その知識ベースのデータを管理しているチームは「自分たちのデータがどのエージェントにどう使われているか」を把握したくなることがあります。特に機微なデータを扱う知識ベース（個人情報、金融取引情報、医療情報など）では、複数エージェントを横断した参照の監査が求められる場面もあるでしょう。

これは主要なユースケースではないかもしれませんが、エージェント開発チームと知識管理チームが分かれている組織では、横断 Observability への需要が別の形で生まれます。

## 技術的な仕組み: OTel と ClickHouse を前提にした観測パイプライン

今回のアナウンスでは、内部実装の説明がかなり具体的です。

Mastra は内部で **OTel 形式のシグナル** を ingest 層に流し込み、Google Cloud Pub/Sub を経由して ClickHouse へ保存していると説明しています。

```
Mastra runtime
  ↓
OTel-shaped signals
  ↓
Hono ingest tier
  ↓
Google Cloud Pub/Sub
  ↓
writer
  ├─ ClickHouse に保存
  └─ billing pipeline に送信
```

\*\*OTel（OpenTelemetry）\*\*は、分散トレース・メトリクス・ログを共通フォーマットで扱うための標準仕様群です。最近のクラウド監視基盤では事実上の共通言語になっており、Datadog、New Relic、SigNoz、Langfuse、Braintrust などのツールは OTel 形式のデータを受け取れます。Mastra が独自形式に閉じず OTel を前提にしているのは、このエコシステムとつながり続けるためです。

**ClickHouse** が保存先として選ばれているのも自然です。列指向データベースとして大量のトレース・ログデータへの集計クエリと相性がよく、先日の Mastra ClickHouse サポート発表でもその特性は詳しく紹介されていました。

### `organizationId` と `resourceId` でマルチテナントを実現する

アナウンスでは、各行に `organizationId` と `resourceId` を付け、API スコープのクエリで扱うことで、テナントごとの専用クラスターを持たなくてもサブ秒でトレース検索できると説明しています。

これは **単一の大きな観測基盤をマルチテナントで共有しつつ、検索時には組織やリソース単位で素早く絞り込める** ようにするための設計です。顧客や組織ごとに物理クラスターを毎回立てると運用もコストも重くなる。かといって単一クラスタに全部を入れるなら、検索性能とアクセス境界の担保が課題になる。`organizationId` と `resourceId` を軸にした API スコープの絞り込みで、その両立を狙っています。SaaS の観測基盤としてまっとうな設計です。

## 今月初めの ClickHouse 発表と何が違うのか

今月5日、Mastra は ClickHouse ストレージアダプターの正式リリースも発表しています。

<https://zenn.dev/shiromizuj/articles/2202620087cd64>

この2つは「Observability × ClickHouse」という点で重なって見えますが、レイヤーが違います。

**5月5日の ClickHouse 発表**は、Mastra フレームワークの利用者が「自分の Observability データをどこへ保存するか」という話でした。DuckDB は本番でエフェメラルになる問題があるので、ClickHouse を観測データ向けストレージとして自前で運用できるようにした、という内容です。

**5月13日の Mastra Observability 発表**は、Mastra Platform がマネージドな観測基盤として独立した、という話です。「自前で ClickHouse を運用するかどうか」より一段上の話で、Mastra 側が観測データの保存・検索・共有体験をプラットフォームとして提供する方向に踏み出したと見るのがよいです。

ざっくり言えばこの違いです。

* **ClickHouse サポート**: フレームワーク側のストレージ選択肢（自前で運用）
* **Mastra Observability**: プラットフォーム側のマネージド観測基盤（Mastra に任せる）

## 使い始め方

コード上の入り口は比較的シンプルです。既存の Mastra インスタンスに `MastraPlatformExporter()` を加えます。

```
import { Mastra } from "@mastra/core/mastra";
import { Observability, MastraPlatformExporter } from "@mastra/observability";

export const mastra = new Mastra({
  // ...
  observability: new Observability({
    configs: {
      default: {
        serviceName: "mastra",
        exporters: [new MastraPlatformExporter()],
      },
    },
  }),
});
```

新規プロジェクトなら `npm create mastra@latest` のウィザードで Observability 有効化に `Yes` と答えるだけでも始められます。

ここで注目すべきなのは、導入方法が **storage adapter の細かな設定より exporter の追加に寄っている**ことです。以前の Observability 記事で書いた「Mastra の observability 設定の中心は exporter」という設計がそのまま活きており、ローカル確認用・外部 OTEL 互換基盤・Platform 管理基盤を、**同じ exporter という差し込み口で切り替えられる**のは Mastra の設計として筋が良いところです。

## まとめ

Mastra Observability は「ログが見られるようになった」という話ではありません。**マルチエージェント・マルチプロジェクト運用における分散トレーシングの問題を、Platform レベルの共有基盤として解く**という発表です。

受益者をあらためて整理すると、こうなります。

* **スーパーバイザー構成の運用者**: 別デプロイのサブエージェントをまたいで1本のトレースとして追いたい
* **チームリーダー・プラットフォームエンジニア**: コスト・エラー・レイテンシを複数プロジェクト横断で把握したい
* **OLAP 基盤を自前で持ちたくないチーム**: ClickHouse 運用を Mastra に任せたい
* **共有 RAG のデータオーナー**: 複数エージェントを横断してデータ参照を監査したい

エージェントが1体から複数体へ、1プロジェクトから複数プロジェクトへとスケールするとき、「それぞれのエージェントの中で何が起きているか」という観点だけでは足りなくなります。Mastra Observability は、エージェントシステムを「全体として観る」ための基盤として位置づけられた発表です。
