## AIエージェントとは何か

AIエージェントとは何かを、このセッション内での定義を記すと、

> **Agent = Observe / Act / Achieve**

とあり、AIエージェントは、単に回答を生成する存在ではなく、状態を**観察**し、自身の持つツールを使って**行動**し、目的の**達成**を試みるアプリケーションと定義していました。  
そのため、マルチエージェントについても、複数のエージェントがより大きな目的の達成を試みるためのアプリケーションと定義しています。

---

## マルチエージェント

単一のエージェントが解決できない複雑な問題を解決できるのがマルチエージェントです。エージェントで解決したい用途に合わせて、様々な構成を取ることが可能です。セッションの中では4つの種類の構成が定義されていました。

![](https://static.zenn.studio/user-upload/be251a2e165d-20260423.jpg)

1. Independent
2. Decentralized
   * エージェント同士で議論しながら、多数決で振る舞いを決定する
3. Centralized
   * 中央集権型で、サブエージェントを統率するエージェントが対応する
4. Hybrid

これらの構成は、上手に扱うことができれば強力なことは間違いありませんが、マルチエージェントを構成する中で苦労する点が多いと考えられます。セッションの中では、そういった困難のうち、84%を占めるのがシステムデザインに関わる問題と説明されていました。

![](https://static.zenn.studio/user-upload/2063bacd1449-20260423.jpg)

具体的に挙げられていたのは下記の問題です。

* Topology design issues
* Inter-agent miscommunication
* Task progress miscalculation
  + 部分的にしか実行されていないタスクを、システムが完了済みと判断してしまう

また、選んだマルチエージェントパターンを使えば他のどのような問題でも解決できるかというと、そういうわけでもありません。マルチエージェントのアーキテクチャは解きたいタスクの性質に大きく依存します。

## マルチエージェントを構築するためのチェックリスト

![](https://static.zenn.studio/user-upload/b790b067bcaa-20260423.jpg)

上図に示すエージェントアーキテクチャをベースに、マルチエージェントを構築するために考慮すべき観点は以下の通りとしています。

1. Value
   * ユースケースは明確に定義されているか？
   * 価値はどのように測定するのか？
2. Users
   * ユーザーは誰か？
   * ユーザーペルソナや目的は定義されているか？
   * エージェントやデータに対するアクセス権は特定されているか？
3. Frontend
   * フロントエンド開発の責任者は誰か？
   * ユーザージャーニーは明確に定義されているか？
   * どのような種類のドキュメントを取り込むのか？
   * 認証・認可はフロントエンドまたはその前段で処理されているか？
4. Orchestration
   * あなたのタスクは、事前定義されたワークフローで完結できるものか？それとも、\*\*オープンエンド（柔軟に展開するタイプ）\*\*のタスクか？
   * 精度をある程度犠牲にしてでも、高速またはインタラクティブな応答を優先するか？それとも、高品質な応答を重視するか？
   * 推論（Inference）コストに対する予算はどの程度か？
5. Sub-agents
   * 新しいサブエージェントを開発する必要があるか？それとも既存のエージェントを呼び出すか？
   * エージェントのガバナンスプロセスは整備されているか？
6. Tools
   * MCPサーバーは誰が開発するのか？
   * レスポンスやデータの品質をどのように管理・理解するのか？
   * 1つのエージェントに対してツールはいくつ必要か？（エージェント内に占めるツールの数で、最大トークン数の中で割くことのできるコミュニケーションキャパシティが決まる）

## エージェントのルーティング

上記のようなチェックリストで適切なエージェント構成を定義したとして、適切なエージェントにタスクが渡るか、というルーティングの問題もあります。特に簡単なタスクであればあるほど、エージェントは混同しやすいため、困難になる傾向があります。

これに対して、セッションの中では2次元空間にタスクとエージェントを埋め込み、類似したエージェントがいることを視覚的に表す方法を見せていました。

![](https://static.zenn.studio/user-upload/bd6013c2e8f7-20260423.jpg)

画像が見えにくい点はご了承ください。青色がタスク、水色がエージェントを表しています。水色が存在しない空間は新しいエージェントを作る必要性、水色が重複するような空間はエージェントの統合をそれぞれ視野に入れることができます。具体の手法やライブラリなどの言及はありませんでしたが、今回の発表資料は後日共有されるそうなので、詳細を待ちたいところですね。
