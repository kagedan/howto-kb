![](https://static.zenn.studio/user-upload/5c932e437352-20260721.png)  
生成AIを使った業務システムでは、モデルの性能が話題になりがちです。

「長い文書を読める」  
「外部ツールを呼び出せる」  
「エージェントとして自律的に動ける」

しかし、本番運用で先に決めるべきなのは、モデルに何ができるかではなく、何をしてよいかです。

社内資料を要約することと、顧客へメールを送ることでは、失敗した際の影響が違います。候補日を提示することと、相手の予定を確定することも同じではありません。

生成AIの導入を、機能の追加ではなく「権限の委譲」として考えてみます。

## 能力と権限を分離する

モデルには、文章の生成、分類、要約、情報抽出などの能力があります。

一方、データの閲覧、レコードの更新、外部送信、決済といった権限は、業務システム側が管理するものです。

モデルがメール本文を生成できるからといって、送信権限まで与える必要はありません。請求書から金額を読み取れても、支払いを実行してよいことにはなりません。

この二つを混ぜると、「精度が上がったので自動実行へ進む」という危険な判断につながります。

モデルの出力品質と、操作を任せられるかどうかは別々に評価します。

## AIへ渡す操作を4段階に分ける

業務上の操作を、影響の大きさで四つに分けます。

### Level 0：読む

文書を検索する、内容を要約する、過去の履歴を参照する段階です。

書き込みは発生しません。ただし、質問者が閲覧できない情報までAIが取得しないよう、検索時のアクセス制御が必要です。

### Level 1：提案する

問い合わせの分類候補、次の行動、確認事項などを表示します。

システムの状態は変わりません。人は提案を採用することも、無視することもできます。導入初期は、この段階でAIがどのように間違えるかを観察できます。

### Level 2：下書きを作る

メール案、申請案、顧客情報の更新案などを保存します。

状態は変わりますが、外部への影響は限定的です。人が編集または破棄できるため、比較的元に戻しやすい操作です。

### Level 3：実行する

メール送信、予約確定、発注、支払いなど、実際の副作用を伴う段階です。

誤りを完全には取り消せない操作もあります。対象、金額、送信先などに応じた承認が必要です。

生成AIの導入は、必ずしもLevel 3を目指すものではありません。提案や下書きまででも、調査や準備にかかる時間を減らせます。

## 可逆性で自動化の上限を決める

同じLevel 2の更新でも、変更履歴から戻せる操作と、他部署の処理を開始させる操作ではリスクが異なります。

操作ごとに次の点を確認します。

* 元の状態へ戻せるか
* 影響する対象を限定できるか
* 実行前に内容を確認できるか
* 誤りを後から検知できるか
* 実行者と承認者を追跡できるか
* 取り消せる期限があるか

社内Wikiの下書き作成は可逆性が高いでしょう。一方、顧客へのメール送信は、送信後に完全には取り消せません。

「どこまで自動化できるか」より、「どこから先は戻せないか」を基準に承認境界を置きます。

## モデルへ実行関数を直結しない

AIエージェントが外部ツールを使う構成でも、モデルが生成した操作をそのまま実行するべきではありません。

モデルは操作候補を作り、アプリケーションのポリシー層が実行可否を決めます。

```
type Actor = {
  userId: string;
  department: "sales" | "support" | "finance";
  roles: string[];
};

type Action =
  | {
      kind: "save_email_draft";
      recipient: string;
      body: string;
    }
  | {
      kind: "send_email";
      recipient: string;
      body: string;
      approvalId?: string;
    }
  | {
      kind: "create_payment";
      invoiceId: string;
      amount: number;
      approvalId?: string;
    };

type Decision =
  | { allowed: true }
  | { allowed: false; reason: string };
```

操作種別ごとに、必要な役割や承認を確認します。

```
function authorize(actor: Actor, action: Action): Decision {
  if (action.kind === "save_email_draft") {
    return actor.roles.includes("draft_writer")
      ? { allowed: true }
      : { allowed: false, reason: "missing_draft_writer_role" };
  }

  if (action.kind === "send_email") {
    if (!actor.roles.includes("email_sender")) {
      return {
        allowed: false,
        reason: "missing_email_sender_role"
      };
    }

    return action.approvalId
      ? { allowed: true }
      : { allowed: false, reason: "approval_required" };
  }

  if (actor.department !== "finance") {
    return { allowed: false, reason: "finance_only" };
  }

  if (!action.approvalId) {
    return { allowed: false, reason: "approval_required" };
  }

  return action.amount <= 100_000
    ? { allowed: true }
    : { allowed: false, reason: "amount_exceeds_limit" };
}
```

プロンプトへ「承認がなければ実行しないでください」と書くことは、モデルへの指示にすぎません。

ポリシー層では、許可されていない操作を実行経路から排除します。

## 承認した内容と実行内容を一致させる

`approved: true`という値だけでは、承認後に宛先や金額が変更されても検知できません。

承認証跡には、誰が、いつ、どの操作内容を承認したかを残します。

```
type Approval = {
  approvalId: string;
  actionDigest: string;
  approvedBy: string;
  expiresAt: string;
};

async function digestAction(action: Action): Promise<string> {
  const bytes = new TextEncoder().encode(JSON.stringify(action));
  const hash = await crypto.subtle.digest("SHA-256", bytes);

  return Array.from(new Uint8Array(hash))
    .map((value) => value.toString(16).padStart(2, "0"))
    .join("");
}
```

実行直前の`Action`からハッシュを作り、承認時のハッシュと照合します。

内容が変更されていた場合は、改めて承認を求めます。承認には有効期限も設け、過去の判断を使い回せないようにします。

## 検索には利用者の権限を引き継ぐ

社内検索やRAGで、AIだけが全資料を閲覧できる構成にすると、回答を通じて情報が漏れる可能性があります。

質問した利用者の権限を検索時にも引き継ぎます。

```
type SearchContext = {
  userId: string;
  allowedScopes: string[];
};

async function searchDocuments(
  query: string,
  context: SearchContext
) {
  return documentStore.search({
    query,
    filters: {
      scope: { in: context.allowedScopes }
    }
  });
}
```

生成後の回答から機密情報を取り除くのではなく、取得段階で対象外の文書を除外します。

モデルが情報を受け取らなければ、回答へ混ぜることもできません。

## 最初はシャドーモードで観察する

本番導入の初期には、AIの判断を業務へ反映しないシャドーモードが使えます。

AIは分類や操作候補を作りますが、担当者は従来どおり処理します。後から両者を比較し、次の点を確認します。

* AIの提案と担当者の判断が一致した割合
* 不一致が起きた業務条件
* AIが参照できなかった情報
* 担当者が暗黙に使っていた基準
* 提案を採用した場合に減らせる作業

十分に観察してから、提案表示、下書き保存という順で権限を広げます。

精度の平均値だけで進めるのではなく、失敗を検知できるか、影響を戻せるかを判断基準にします。

## 自動実行の停止条件を先に決める

実行を許可する条件と同時に、停止条件も必要です。

たとえば、次の状態では自動実行を止めます。

* 監査ログを保存できない
* 承認サービスへ接続できない
* 拒否される操作が急増した
* 許可していない操作種別が生成された
* 実行回数や利用量が上限を超えた

```
type RuntimeHealth = {
  auditLogAvailable: boolean;
  approvalAvailable: boolean;
  rejectionRate: number;
};

function canRun(health: RuntimeHealth): boolean {
  if (!health.auditLogAvailable) return false;
  if (!health.approvalAvailable) return false;
  if (health.rejectionRate > 0.2) return false;

  return true;
}
```

AIが正常でも、監査や承認が動いていなければ実行しない。構成要素の一部が壊れたとき、安全側へ倒れる設計にします。

## 導入計画は権限の拡張として描く

生成AIのロードマップは、機能数ではなく、委譲する権限で表現できます。

1. 利用者の権限内で情報を読む
2. 判断材料を提示する
3. 破棄できる下書きを作る
4. 承認された操作だけを実行する
5. 影響を限定できる操作を条件付きで自動化する

各段階で、責任者、承認条件、取り消し方法、停止条件を決めます。

モデルが高性能になったから次へ進むのではありません。運用側が失敗を扱えるようになったときに、権限を一段階広げます。

## まとめ

生成AIの業務導入では、能力と権限を分けて設計します。

* 読む、提案する、下書きを作る、実行するを区別する
* 戻せない操作の手前に承認境界を置く
* モデルが作った操作をポリシー層で検査する
* 承認内容と実行内容の同一性を確認する
* 質問者の閲覧権限を検索へ引き継ぐ
* シャドーモードから段階的に権限を広げる
* 監査や承認が使えない場合は自動停止する

生成AIの価値は、すべてを自律実行することだけにありません。

人が判断する直前まで情報を整えるだけでも、業務負担は減らせます。安全に委譲できる権限を見極めることが、継続運用できるAIシステムの出発点です。
