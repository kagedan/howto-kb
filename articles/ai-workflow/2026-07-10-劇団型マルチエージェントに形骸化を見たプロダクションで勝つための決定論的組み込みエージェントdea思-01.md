## **1. はじめに：なぜあなたのマルチエージェントは遅くて脆いのか**

「社長、デザイナー、コーダーのエージェントを用意し、彼らが自律的にチャットを往復させて成果物を作り上げます」  
LangChainやCrewAI、AutoGenの登場以降、こうした「人間の組織を模したマクロなマルチエージェント（以下、劇団型）」のデモをよく目にするようになりました。しかし、これらをそのまま実商用環境（プロダクション）に投入しようとすると、例外なく強烈な現実の壁にぶつかることになります。

1. **非決定論的なハルシネーションの連鎖：** 確率論的に動くLLM同士が自然言語で対話するため、一箇所のエラーやブレが後続のエージェントで増幅され、システム全体の挙動が予測不能になる。
2. **絶望的なレイテンシ：** エージェント間のコンテキスト往復（「了解しました」「修正します」といった無駄なテキスト挨拶や、冗長なメタ会話）により、ユーザーへのファーストバイトまでに数十秒〜数分を要する。
3. **コンテキストの自己汚染：** 複数エージェントの会話履歴が全域で累積・肥大化し、不要な情報がコンテキストを圧迫して中核推論の精度を低下させる。

エンタープライズなシステムや、超低レイテンシが求められるリアルタイム対話型AIエンジンにおいて、この「劇団型」はあまりにも脆弱で非効率です。  
本記事では、外部に対するインタフェースを単一のシングルエージェントとしてカプセル化し、内部では厳格な型定義と局所的な状態管理によって駆動する「決定論的組み込みエージェント（Deterministic Embedded Agent: DEA）」という設計思想を提案します。

## **2. アーキテクチャの対比：劇団か、DAWのプラグインか**

DEA思想の本質は、「疎結合な自然言語の対話」をシステム内部から排除し、「厳格なインターフェース定義による高密度な連携（Strongly Typed Pipeline）」へと昇華させる点にあります。実装レベルの密結合（緊密な依存）を避けつつ、データ型をPydantic等で固定することで決定論的な結合を担保します。  
既存の主要なマルチエージェント・アーキテクチャとのトポロジー比較を以下に示します。

### **📊 アーキテクチャ特性の比較**

| 評価軸 | 劇団型（Role-play） | 既存のState Graph（LangGraph等） | DEA（本アーキテクチャ） |
| --- | --- | --- | --- |
| **制御トポロジー** | 自律分散・プロンプトベース | 集中制御・環状/分岐グラフ（状態依存） | **線形・半線形パイプライン（静的インターフェース）** |
| **状態（State）のスコープ** | 全域（会話履歴の累積） | 共有コンテキスト（全ノードが書き換え可） | **局所スロット（Isolated Context Slot）** |
| **データ交換形式** | 自然言語（Unstructured） | 自由な辞書型（Semi-structured） | **厳格なスキーマ（Strongly Typed / Pydantic）** |
| **応答速度（レイテンシ）** | 非常に遅い（秒〜分単位） | 中〜低速（動的エッジ評価のオーバーヘッド） | **超高速（並行・並列パイプライン制御）** |
| **デバッグ・品質保証** | 困難（非決定論的なエラー連鎖） | 中難度（グラフ遷移のトレースが必要） | **容易（境界でのスキーマ検証が可能）** |

既存のState Graph（状態管理グラフ）が「すべてのノードが共有Stateを動的に書き換えながら巡回する」という自由度（非決定性）を持つのに対し、DEAは「ステートレスな関数のパイプライン結合」に近い構造を取ります。  
外からは1つの洗練されたシングルエージェントに見えるのに、内部では極限まで同期され、1つの美しい音を奏でるDAW（デジタル・オーディオ・ワークステーション）の内部プラグイン群のように機能する。これがDEAの基本トポロジーです。

## **3. 構造化データ生成コストと例外耐性（Fault Tolerance）の設計**

「テキストによる対話を廃し、JSON等の構造化データを流す」という設計を愚直に行うと、もう一つの罠に嵌ります。LLMにとって厳格なスキーマ遵守を強制するサンプリング（Structured Outputs等）は、推論コスト（計算時間）の増大を招くほか、1B〜3Bクラスの軽量モデルではパースエラーによるリトライが発生し、結果的に「劇団型」以上の遅延を招くリスクがあります。  
DEAでは、この「構造化データの生成コスト」と「パースエラー」をランタイム側で吸収するため、以下の「2段階トリアージ＆フォールバック・パイプライン」を定義します。

```
[Input Request]  
       │  
       ▼  
 ┌───────────┐  
 │ Layer 0   │ ──(Raw String Output / スキーマ制約なしの最速推論)  
 └───────────┘  
       │  
       ▼  
 ┌───────────┐  
 │ Validator │ ──(正規表現 / Pydanticによる高速パース。成功なら即次段へ)  
 └───────────┘  
       │  
       ├─ \[Parse Failure / Validation Error\]（例外発生時のみトリガー）  
       ▼  
 ┌───────────┐  
 │ Layer 0.5 │ ──(文法制御・制約付き構造化出力によるピンポイント修正)  
 └───────────┘  
       │  
       ├─ [Fatal Error / 2次失敗]  
       ▼  
 ┌───────────┐  
 │ Fallback  │ ──(決定論的なルールベース/正規表現による強制割当)  
 └───────────┘
```

1. **Layer 0（オプティミスティック・パス）：** 軽量モデルに対してスキーマ制約をプロンプト側の「推奨」に留め、自由文（Markdownコードブロック等）で最速で出力させます。
2. **高速バリデータ：** 出力をLLMではなく、ランタイム側（Regex / Pydantic）でパースします。9割以上の正常系はここで最速パスします。
3. **Layer 0.5（ペシミスティック・パス）：** パースエラー時のみ、エラー箇所をフィードバックした制約付きサンプリング（JsonformerやGrammarベース）を起動し、特定スロットのみを再生成します。2次失敗時はルールベースのデフォルト値へ強制フォールバックさせ、システム全体の停止（ハングアップ）を数学的に防止します。

## **4. 思考レイヤーにおけるモデル・スイッチングと責任境界**

「適材適所」という言葉で、単にエッジ向け軽量モデルからフロンティアモデルまでをインラインに並べるだけでは、モデル間の能力差による出力形式のブレや、モデル切り替え時のコンテキスト再注入によるオーバーヘッドで崩壊します。  
DEAでは、各レイヤーの責任境界（Boundary）と採用メトリクスを厳格に独立させます。

### **📐 責任境界とモデルマッピング・マトリクス**

| レイヤー名 | 責任境界（Boundary） | 主要メトリクス（Metric） | 採用モデル基準 |
| --- | --- | --- | --- |
| Layer 1: Triage （分類・マッピング） | 入力からユーザーのインテント・エンティティを抽出し、静的スキーマへマッピング。 | ・Time to First Token (TTFT) ・F1-Score（分類精度） | 1B - 3B クラス （量子化ローカルモデル） |
| Layer 2: Logic （論理検証・思考） | 隔離されたコンテキストスロットを基に、中核推論（思考）を実行。 | ・推論ベンチマーク（Math/Code） ・指示追従性（Constraint Adherence） | 70B - フロンティアクラス （商用API、高精度モデル） |
| Layer 3: Protocol （出力最適化） | 思考結果を、システム（TTSやモーション、外部API等）が要求する物理プロトコルにマッピング。 | ・JSON準拠率 ・処理速度（Tokens/sec） | 7B - 8B クラス （Instruction/Code特化モデル） |

### **🔄 局所状態管理（State Management）によるコンテキスト隔離**

モデル間を跨ぐ際、Layer 1の生プロンプトや内部のトリアージログ、プロトコル変換の中間テキストといった「メタな会話履歴」は、すべて次段のコンテキストから隠蔽（History Masking）されます。  
Layer 1から抽出されたデータは純粋な「構造化引数（型定義された値）」としてのみ取り出され、Layer 2の事前定義済みシステムプロンプトテンプレートへ「値渡し」で注入されます（**局所コンテキストスロット**）。  
これにより、 Layer 2のフロンティアモデルは汚染されていない純粋なコンテキストのみを処理することができ、モデル切り替えに伴う出力のブレをランタイム側で完全に吸収・遮断することが可能になります。

## **5. 結論：プロダクションファーストの思想へ**

AIを用いたシステム開発において、デモの面白さとプロダクション環境での実用性は往々にしてトレードオフの関係にあります。  
ユーザーを待たせない「脊髄反射」的なレスポンス、破綻しないコンテキスト、予測可能な例外耐性。これらを担保した上で高度な協調処理を実現するには、「シングルエージェントという堅牢な仮面（カプセル化）を被った、静的パイプライン駆動の組み込みマルチエージェント」という設計が1つの最適解となります。  
役割を演じ合わせるマクロな「劇団型」の形骸化を乗り越え、「型安全、例外耐性、コンテキスト隔離、責任境界」を極限まで追求すること。それこそが、これからのAIエンジニアリングを実戦（プロダクション）で勝利に導く、真のアーキテクチャ思想です。
