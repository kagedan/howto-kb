## なぜマルチエージェントは難しいのか

単体のAIエージェントは今や十分実用的になってきた。  
しかし複数のエージェントを連携させる「マルチエージェントシステム」になった途端、  
予期しない失敗が頻発する。

GitHubによると、マルチエージェントが特に有効なのは以下のケースだ。

* コードベースのメンテナンスと依存関係の更新
* コードレビューの自動チェックとリファクタリング
* 仕様書からの機能実装
* IssueとPull Requestのトリアージ

しかしこれらを**単一のAIエージェントで全部やろうとすると問題が起きる。**

```
# ❌ 単一エージェントに全部任せた場合の問題

class SingleAgent:
    def handle_everything(self, task):
        # 状態管理が複雑になりすぎる
        self.state = {}  # どんどん膨れ上がる
        
        # コンテキストが大きくなりすぎてLLMが混乱
        context = self.load_entire_codebase()  # 巨大すぎる
        
        # 一つのエラーが全体に影響する
        result = self.llm.process(context, task)
        # → 途中でエラーが出ると全部やり直し
```

マルチエージェントは解決策になるが、**エージェント間の通信設計が甘いと  
さらに複雑な問題が起きる。**

## 最新動向：GitHubが公開した3つの設計パターン

2026年2月24日、GitHubはマルチエージェントシステムを  
信頼性高く動かすための設計プラクティスを公開した。

### パターン1：型付きスキーマ（Typed Schema）

**問題：エージェント間でJSON を使うと壊れやすい**

```
# ❌ 型なしJSONでエージェント間通信（壊れやすい）
def agent_a_output():
    return {
        "status": "done",      # 文字列？列挙型？
        "result": 42,          # 数値？文字列？
        "data": [...],         # 構造が不定
    }

def agent_b_input(data):
    # dataの型が保証されていない
    # → フィールド名のタイポ、型の不一致、フォーマット変更で即死
    status = data["status"]    # "done"? "Done"? "DONE"?
    result = data["result"]    # int? str? float?
```

```
# ✅ 型付きスキーマを使った場合（壊れにくい）
from pydantic import BaseModel
from enum import Enum
from typing import List

class TaskStatus(Enum):
    PENDING   = "pending"
    RUNNING   = "running"
    DONE      = "done"
    FAILED    = "failed"

class AgentOutput(BaseModel):
    status: TaskStatus          # 列挙型で厳密に定義
    result: int                 # 型が保証される
    data: List[str]             # 構造が明確
    error_message: str | None = None

# エージェントA：型付きで出力
def agent_a() -> AgentOutput:
    return AgentOutput(
        status=TaskStatus.DONE,
        result=42,
        data=["item1", "item2"]
    )

# エージェントB：型付きで受け取る
def agent_b(output: AgentOutput) -> None:
    if output.status == TaskStatus.DONE:
        # 型が保証されているので安全に処理できる
        process(output.result, output.data)
    # スキーマ違反は実行前に検出される
```

**GitHubの主張：** スキーマ違反はコントラクト違反として扱い、  
無効な状態になる前にリトライまたはエスカレーションする設計にすること。

### パターン2：アクションスキーマ（Action Schema）

**問題：曖昧な指示でエージェントがバラバラな行動をする**

```
# ❌ 曖昧な指示の場合
instruction = "このIssueを分析してチームにアクションを取らせよう"

# エージェントによって解釈が分かれる
agent_1_response = "close"      # クローズする
agent_2_response = "assign"     # アサインする
agent_3_response = "escalate"   # エスカレーションする
agent_4_response = "do_nothing" # 何もしない
# → どれが正しいのかシステムが判断できない
```

```
# ✅ アクションスキーマで正確なアクションセットを定義
from enum import Enum
from pydantic import BaseModel

class IssueAction(Enum):
    CLOSE        = "close"
    ASSIGN       = "assign"
    ESCALATE     = "escalate"
    LABEL        = "label"
    COMMENT      = "comment"
    DO_NOTHING   = "do_nothing"

class ActionDecision(BaseModel):
    action: IssueAction              # 列挙型で限定
    assignee: str | None = None      # assignの場合必須
    label: str | None = None         # labelの場合必須
    comment: str | None = None       # commentの場合必須
    reason: str                      # 理由は常に必須

# エージェントの出力が必ずこのスキーマに収まる
def analyze_issue(issue: dict) -> ActionDecision:
    # LLMに構造化出力を強制する
    return llm.structured_output(
        prompt=f"Analyze: {issue}",
        output_schema=ActionDecision
    )
    # → 常に有効なアクションの一つが返ってくることが保証される
```

**GitHubの主張：** 全ステップに構造が必要なわけではなく、  
最初のステップで曖昧なアクションのセットに絞り込む設計が重要。

### パターン3：MCPによるインターフェースの統一

**問題：エージェントが使うツールのインターフェースが統一されていない**

```
# ❌ ツールごとに異なるインターフェース（混乱の元）
class GitHubTool:
    def create_pr(self, title, body, base, head):
        ...

class JiraTool:
    def create_ticket(self, summary, description, project):
        ...

class SlackTool:
    def post_message(self, channel, text, blocks=None):
        ...

# エージェントはツールごとに異なる呼び出し方を学習しなければならない
# → インターフェースの変更が全エージェントに影響する
```

```
# ✅ MCP（Model Context Protocol）で統一
# すべてのツールが同じインターフェースで呼び出せる

class MCPClient:
    def call_tool(self, server: str, tool: str, params: dict):
        """MCPで統一されたツール呼び出し"""
        return self.mcp.execute(server, tool, params)

# エージェントからの呼び出しが統一される
agent.call_tool("github", "create_pr", {
    "title": "Fix bug",
    "body": "...",
    "base": "main",
    "head": "fix/bug"
})

agent.call_tool("jira", "create_ticket", {
    "summary": "Bug found",
    "description": "...",
    "project": "PROJ"
})

# MCPのメリット：
# ・ツールの入出力スキーマが事前に検証される
# ・実行前に呼び出しを認証できる
# ・インターフェースの仕様変更に強い
# ・無効な状態が本番環境に届く前にブロックできる
```

## マルチエージェントを信頼性高く動かすための基本原則

GitHubが示した6つのプラクティスをまとめると：

| 原則 | 内容 |
| --- | --- |
| **実行前に設計する** | 動かしてから考えるのではなく、設計を先に |
| **全エージェントで認証する** | 境界だけでなく各エージェント間でも認証 |
| **アクションを事前に制限する** | エージェントに追加する前にアクションを絞り込む |
| **中間状態をログに記録する** | 途中経過を常に記録して追跡可能にする |
| **リトライと冪等な実行を想定する** | 失敗したら安全に再実行できる設計にする |
| **チャットではなくコードとして扱う** | エージェントはチャット相手ではなくシステムコンポーネント |

## 現場エンジニア目線の考察

半導体・製造業の現場では、複数のシステムが連携して  
一つのプロセスを動かすのは珍しくない。  
検査装置・制御システム・品質管理システムが互いにデータをやり取りする。

今回GitHubが示した設計パターンは、**そういった既存の分散システム設計の  
知恵と本質的に同じだ。**

製造現場でインターフェースの型定義を甘くしたまま  
複数システムを繋いだ結果、データ連携が壊れて大規模なデバッグに  
なったという経験は誰しも持っているはずだ。

AIエージェントも同じだ。「賢いから大丈夫」という発想が最も危険で、  
**むしろ賢いからこそ、曖昧な指示から予想外の行動を取りうる。**

特に注目したいのが「エージェントをチャットではなくコードとして扱う」  
という思想だ。エージェントに人間のように話しかけるのではなく、  
**APIのエンドポイントと同じように型付きのインターフェースで呼び出す。**  
この発想の転換がマルチエージェント設計の本質だと感じる。

## まとめ・次のアクション

**結論：マルチエージェントの失敗はAIの能力不足ではなく設計の問題だ。  
型・スキーマ・MCPの3つで構造を作れば信頼性は劇的に上がる。**

今すぐ取り組むべきことは以下だ。

1. **エージェント間のデータ交換にPydanticなどの型定義を導入する**
2. **エージェントが取りうるアクションを列挙型で事前に限定する**
3. **外部ツールとの連携にMCPを採用してインターフェースを統一する**
4. **エージェントの中間状態を必ずログに残す仕組みを作る**

「AIに任せれば動く」から「設計しなければ動かない」へ。  
マルチエージェント時代のエンジニアに求められる思考が変わりつつある。
