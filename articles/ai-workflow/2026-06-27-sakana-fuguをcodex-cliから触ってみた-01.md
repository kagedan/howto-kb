## はじめに

Sakana AI から **Sakana Fugu** が出ていたので、Codex CLI から既存リポジトリの調査・課題抽出・改修実装の流れで触ってみました。

今回は API を直接叩いたのではなく、公式のインストールコマンドで環境構築して `codex-fugu` から使いました。導入はかなりスムーズでした。

この記事は 2026-06-27 時点の公式情報をもとにしています。価格、モデル名、提供地域などは変わる可能性があるので、利用前に公式ページを確認してください。

## Sakana Fuguとは

Sakana Fugu は、Sakana AI が提供するマルチエージェント型のLLMサービスです。複数のモデルを裏側で組み合わせ、利用者からは1つのモデルのように扱えるのが特徴です。

詳細な仕組みや料金、提供条件は [Sakana Fugu 公式ページ](https://sakana.ai/fugu/) と [Sakana Fugu Technical Report](https://arxiv.org/abs/2606.21228) を見るのが確実です。本記事では仕組みの深掘りではなく、実際に Codex CLI から触ってみた所感を中心に書きます。

## Codex CLIから触ってみる

まず [Sakanaのダッシュボード](https://platform.torafugu.app/api-keys) からAPIキーを発行します。

次に、以下のコマンドで環境構築しました。

```
curl -fsSL https://sakana.ai/fugu/install | bash
```

発行したAPIキーは `SAKANA_API_KEY` として設定します。自分の環境では、インストール後に `codex-fugu --set-key` を実行して、発行したAPIキーを入力しました。

この方法だと `~/.codex/.env` にAPIキーが保存され、Codex側から自動で読み込まれます。シェルの環境変数として設定する場合は、以下のような形でもよいと思います。

```
export SAKANA_API_KEY=fish_...
```

その後、以下で起動します。

環境構築から起動までかなりスムーズでした。普段 Codex CLI を触っている人であれば、ほぼ同じ感覚で Fugu を試せます。

## 実際に使った用途

今回の主な利用方法は、既存リポジトリに対する以下の流れです。

1. リポジトリ全体の概要説明
2. 設計・実装上の課題点の抽出
3. 抽出した課題の改修実装

これまでは主に Claude Opus 4.8 を使って実装していました。もちろん Opus 4.8 もかなり強いのですが、Fugu に同じような文脈を渡すと、想像していたより多くの指摘が返ってきました。

## 使ってみた所感

### 指摘の量が多い

既存リポジトリの課題抽出を依頼すると、かなりの数の指摘が返ってきました。

これは単純に「たくさん出せばよい」という話ではありません。指摘が多すぎるとレビューコストも増えるため、実務では severity や対応優先度で絞り込ませる必要があります。

ただ、最初の棚卸しとしては有用でした。人間がコードを読み始める前に、気になるポイントを広めに洗い出してくれるので、調査の初速がかなり上がります。

### 日本語が自然で読みやすい

特段「自然な日本語で」「読みやすく」といった指示をしなくても、出力される日本語がかなり自然でした。

これは地味ですが、長時間使う上ではかなり重要です。技術的には正しくても、日本語が硬すぎたり、翻訳調だったり、妙に圧のある文体だったりすると、読むだけで疲れます。

Fugu の出力は説明の流れがなめらかで、長めのレビュー結果でも比較的読みやすい印象でした。リポジトリ調査や設計レビューのように、大量の文章を読む用途ではこの差が効きます。

### レスポンスが速く感じる

体感ですが、Fugu の動作やレスポンスは他のモデルに比べて速い気がしました。

もちろん、これはネットワーク状況、入力サイズ、混雑状況、選択されるエージェント構成によって変わるはずです。厳密なベンチマークではありません。

ただ、既存リポジトリを読ませながら会話する用途では、待ち時間の短さは作業リズムに直結します。「次の質問を投げるまでのテンポ」がよいのは、実装支援ツールとしてかなり大きいです。

### トークン消費はかなり早い印象

一方で、利用枠の消費はかなり早い印象です。

現在は Pro プランで利用していますが、5時間枠への到達も早く、1週間枠も丸一日 + アルファくらい使うと到達しました。同料金帯の Claude Max プランでは、自分の使い方で1週間枠に到達したことはありませんでした。

## 触ってみて気になったところ

### どのモデルが使われたかは見えない

公式FAQでは、Fugu がどの基盤モデルを選択したか、どのように協調させたかは公開されないと説明されています。

これはプロダクトとしては自然ですが、開発者目線ではトラブルシュート時に少し気になる点です。たとえば「昨日と同じプロンプトなのに出力傾向が変わった」場合に、裏側のモデル更新やルーティング変更をどこまで追えるのかは確認しておきたいです。

### データ利用の設定は最初に確認する

公式FAQでは、学習データ利用はコンソールからオプトアウトできると説明されています。業務コード、顧客データ、未公開情報を扱う場合は、APIキーを発行してすぐ試す前にこの設定を確認した方がよいです。

## どこで使いたいか

正直なところ、使い勝手だけで言えば全面的に使っていきたいです。既存リポジトリの概要把握、課題抽出、改修実装までの流れはかなりスムーズでした。

一方で、現時点ではトークン消費の速さがネックです。常用するには、どの作業を Fugu に任せるのか、どの作業は他のモデルや通常の Codex CLI に寄せるのかを見極める必要がありそうです。今後は、実装タスクの性質ごとに使い分けを模索していきたいです。

## まとめ

`codex-fugu` は導入が簡単で、既存リポジトリの調査から課題抽出、改修実装までかなりスムーズに使えました。特に、指摘量の多さ、日本語の読みやすさ、レスポンスの速さが印象的でした。

一方で、Pro プランでも利用枠の消費はかなり早く感じました。全面的に使いたいくらい手応えはありますが、実務で常用するにはトークン消費との付き合い方を探る必要がありそうです。
