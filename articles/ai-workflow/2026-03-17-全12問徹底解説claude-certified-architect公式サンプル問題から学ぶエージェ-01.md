# 1. Scenario 1: Customer Support Resolution Agent

## Question 1: ツール実行の決定論的強制

**問題**:  
エージェントが12%のケースで `get_customer` を完全にスキップし、顧客が申告した名前のみを使用して `lookup_order` を呼び出しているため、時折アカウントの誤認や誤った返金が発生していることがわかりました。 この信頼性の問題に最も効果的に対処する変更はどれですか？

A) `get_customer` が検証済みの顧客IDを返すまで、`lookup_order` と `process_refund` の呼び出しをブロックするプログラム上の前提条件を追加する。  
B) システムプロンプトを強化し、注文操作の前に `get_customer` による顧客検証が必須であると記載する。  
C) 顧客が自発的に注文の詳細を提示した場合でも、エージェントが常に最初に `get_customer` を呼び出すことを示す Few-shot サンプルを追加する。  
D) 各リクエストを分析し、そのリクエストタイプに適切なツールのサブセットのみを有効にするルーティング分類器を実装する。

正解と解説を確認する

**正解: A**

**解説と図解**:  
LLMは確率論的に動作するため、プロンプト（B, C）だけでは「100%の遵守」を保証できません。金銭的影響がある処理（返金など）では、コード側で「ゲート」を作るのが最も安全です。

## Question 2: 曖昧なツール説明の改善

**問題**:  
本番ログでは、ユーザーが注文について尋ねたとき、エージェントが `lookup_order` ではなく `get_customer` を頻繁に呼び出していることが示されています。 どちらのツールも最小限の説明しかなく、類似した識別子フォーマットを受け入れます。 ツール選択の信頼性を向上させるための最も効果的な最初のステップは何ですか？

A) 解析用の Few-shot サンプルを5〜8個追加する。  
B) 各ツールの説明を拡張し、入力フォーマット、クエリ例、エッジケース、および類似ツールとの使い分けを説明する境界条件を含める。  
C) キーワードに基づくルーティングレイヤーを実装する。  
D) 両方のツールを単一の `lookup_entity` ツールに統合する。

正解と解説を確認する

**正解: B**

**解説と図解**:  
Claudeがツールを選ぶ際の最大のヒントは「ツールの説明（Description）」です。モデルが混乱しているときは、定義を詳細にして「選択の解像度」を上げることが効果的です。

## Question 3: エスカレーションの判断基準

**問題**:  
エージェントは単純なケースを人間にエスカレーションする一方で、ポリシーの例外が必要な複雑な状況を自律的に処理しようとして、失敗しています。 エスカレーションのキャリブレーションを改善する最も効果的な方法は何ですか？

A) 自律的に解決する場合とエスカレーションする場合の違いを示す Few-shot サンプルとともに、明示的なエスカレーション基準をシステムプロンプトに追加する。  
B) エージェントに信頼度スコアを自己報告させ、低い場合にルーティングする。  
C) 過去のチケットでトレーニングされた個別の分類器モデルをデプロイする。  
D) 感情分析を実装し、不満度が高い場合にエスカレーションする。

正解と解説を確認する

**正解: A**

**解説と図解**:  
LLMの自己評価（B）は不正確なことが多いため、プロンプトで「境界線」を定義し、具体的な事例（Few-shot）を教え込むのが最も確実です。

# 2. Scenario 2: Code Generation with Claude Code

## Question 4: カスタムコマンドの配置

**問題**:  
標準的なコードレビューチェックリストを実行するカスタムの `/review` スラッシュコマンドを作成したい。このコマンドは、すべての開発者がリポジトリをクローンした時に利用できる必要があります。どこに作成すべきですか？

A) プロジェクトリポジトリ内の `.claude/commands/`  
B) ホームディレクトリ内の `~/.claude/commands/`  
C) プロジェクトルートの `CLAUDE.md`  
D) `.claude/config.json`

正解と解説を確認する

**正解: A**

**解説と図解**:  
チーム開発では、共有すべき設定はリポジトリの管理下に置きます。ラベルにスペースや記号を含む場合は引用符で囲む必要があります。

## Question 5: Planモードの適切な活用

**問題**:  
モノリシックアプリケーションをマイクロサービスに再構築する（数十ファイルにまたがる変更）。どのアプローチを取るべきですか？

A) 計画（plan）モードに入り、依存関係を理解し、変更前に実装アプローチを設計する。  
B) 直接実行（direct execution）で段階的に変更を加える。  
C) すべての指示を与えて直接実行を使用する。  
D) 直接実行で開始し、複雑さに遭遇した場合にのみ計画モードにする。

正解と解説を確認する

**正解: A**

**解説と図解**:  
大規模なリアーキテクチャでは、「いきなり書く（Direct）」のではなく「調べてから考える（Plan）」プロセスが必須です。

---

## Question 6: パス固有のルールの適用

**問題**:  
Reactコンポーネント、APIハンドラー、DBモデルで異なる規約があり、さらにテストファイル（`Button.test.tsx`など）はコードベース全体に分散しています。正しい規約を一律に適用する最も保守しやすい方法は？

A) `.claude/rules/` に glob パターン（例: `**/*.test.tsx`）を指定したルールファイルを作成する。  
B) ルートの `CLAUDE.md` にすべて記載し、推論に頼る。  
C) 各タイプに対して `.claude/skills/` を作成する。  
D) 各サブディレクトリに `CLAUDE.md` を配置する。

正解と解説を確認する

**正解: A**

**解説と図解**:  
ディレクトリ構造に縛られず、ファイル名パターンで横断的にルールを適用できるのが `.claude/rules/` の強みです。

# 3. Scenario 3: Multi-Agent Research System

## Question 7: タスク分解の不備

**問題**:  
「AIがクリエイティブ産業に与える影響」を調査。最終レポートが視覚芸術のみをカバーし、音楽や執筆を見落としていた。コーディネーターは「デジタルアート」「デザイン」「写真」の3つに分解していた。原因は？

A) 統合サブエージェントにギャップを特定する指示がない。  
B) コーディネーターエージェントのタスク分解が狭すぎた。  
C) Web検索エージェントのクエリが包括的ではなかった。  
D) ドキュメント分析エージェントが基準を厳しくしすぎた。

正解と解説を確認する

**正解: B**

**解説と図解**:  
コーディネーターによる「タスク分解（Decomposition）」が、本来の親タスクの範囲（クリエイティブ産業全体）をカバーできていなかったことが原因です。

## Question 8: 構造化エラーの伝播

**問題**:  
Web検索サブエージェントがタイムアウトした際、インテリジェントな回復を可能にする最適なエラー伝播方法は？

A) 失敗のタイプ、試行内容、部分的な結果、および代替アプローチを含む構造化されたエラーコンテキストを返す。  
B) エクスポネンシャルバックオフを実装し、最後に一般的な「検索利用不可」を返す。  
C) タイムアウトをキャッチし、成功とマークされた空の結果セットを返す。  
D) 例外を最上位へ伝播させ、ワークフロー全体を終了させる。

正解と解説を確認する

**正解: A**

**解説と図解**:  
コーディネーターが「次に何をすべきか」を判断するには、何が起きたかという詳細なコンテキストが必要です。

## Question 9: 効率的な検証ツールの付与

**状況**:  
統合エージェントが頻繁に事実確認を行う。現在は一度コーディネーターに戻ってから検索エージェントを呼んでいるため、レイテンシが高い。85%は単純な事実確認である。どう改善すべきか？

A) 単純な事実確認のためにスコープを絞った `verify_fact` ツールを統合エージェントに与え、複雑な検証は引き続き委譲する。  
B) すべてバッチにして最後に返す。  
C) 統合エージェントにWeb検索ツールをすべて与える。  
D) 検索エージェントに予測して情報を多めにキャッシュさせる。

正解と解説を確認する

**正解: A**

**解説と図解**:  
「最小特権の原則」を守りつつ効率を上げるには、頻度の高い単純作業（事実確認）だけをエージェントに持たせるのが正解です。

# 4. Scenario 5: Claude Code for CI/CD

## Question 10: 非対話型モードでの実行

**問題**:  
CIパイプラインで `claude "Analyze..."` を実行するとハングアップする。正しい使い方は？

A) `-p` フラグを追加する。  
B) `CLAUDE_HEADLESS=true` を設定する。  
C) stdin を `/dev/null` に向ける。  
D) `--batch` フラグを追加する。

正解と解説を確認する

**正解: A**

**解説と図解**:  
CI環境ではユーザー入力を待てないため、一方向の出力モードを選択します。

## Question 11: Batch APIの適材適所

**問題**:  
コスト50%削減のため、Message Batches APIへの切り替えを検討。(1) ブロッキングなマージ前チェック、(2) 翌朝確認の技術負債レポート。どう評価する？

A) 技術負債レポートにのみバッチ処理を使用し、マージ前チェックはリアルタイムを維持する。  
B) 両方をバッチに切り替える。  
C) 両方をリアルタイムにする。  
D) バッチにし、遅い場合にリアルタイムへフォールバックする。

正解と解説を確認する

**正解: A**

**解説と図解**:  
Batch APIは「安いが遅い（最大24時間）」。開発者を待たせるプロセスには不適です。テキストの視認性を高めるため、コントラストの強い色を使用します。

## Question 12: マルチパス（分割）レビュー

**問題**:  
14ファイルを変更した大規模PRのレビューが不安定（見落としや矛盾が発生）。どう再構築すべきか？

A) 局所的な問題のための「ファイル別パス」と、データフローのための「統合パス」に分割する。  
B) PRを小さく分割させる。  
C) 上位モデルへ切り替える。  
D) 独立したパスを3つ実行し、多数決で決める。

正解と解説を確認する

**正解: A**

**解説と図解**:  
一度に多くを詰め込むとモデルの注意力が低下します。「木を見て、その後に森を見る」ように段階を分けます。

**DataBlizzard流・合格のポイント**:

1. **「丸投げ」はしない**: 重要なロジックはコード（決定論）で守る。

3. **「コンテキスト」を導く**: 大規模なタスクは分解し（マルチパス）、適切なエラー情報を伝播させる。
