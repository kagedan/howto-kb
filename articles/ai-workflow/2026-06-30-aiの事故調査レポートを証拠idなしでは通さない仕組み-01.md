この記事は、`Evidence-Locked DFIR Agent` の実装メモです。

公開デモとコードはこちらです。

## 3行でいうと

* AIにセキュリティ事故調査を手伝わせるが、証拠IDのない断定は通さない
* suspicious-email caseを解析し、claim、evidence、accuracy、integrity reportを出す
* malware execution のような危険な断定は、証拠が足りなければdowngradeする

## 誰のどんな課題か

対象は、セキュリティアナリストやインシデント対応担当です。

AIはログや調査資料を素早く要約できます。

ただし、セキュリティ調査では「速くそれっぽい説明」が危険になることがあります。

* どの証拠から言っているのか分からない
* まだ分からないことを断定している
* malware executionを証明できていないのに実行済みと言っている
* 調査レポートだけが残り、根拠との対応が消える

このプロダクトは、AIの回答をそのまま信じるのではなく、証拠IDに紐づいたclaimだけを通す設計です。

## 実装の中心

流れは次のようにしています。

```
case packet
  -> AI proposes claims
  -> evidence-lock verifier checks evidence IDs
  -> unsupported certainty is blocked or downgraded
  -> analyst report
  -> accuracy report
  -> evidence integrity report
```

ポイントは、AIの文章を最後に検証することです。

AIが「これはmalware executionです」と書いても、process evidenceが実行を証明していなければ、その断定は通しません。

証拠が足りないものは、unsupportedやneeds reviewへ落とします。

## Claimをデータとして扱う

セキュリティ調査では、レポート本文だけでなく、claimそのものをデータとして扱うと管理しやすくなります。

例えば、claimには次のような情報を持たせます。

```
{
  "claim_id": "claim-003",
  "statement": "The attachment execution is confirmed.",
  "status": "unsupported",
  "evidence_ids": [],
  "risk": "false_confidence"
}
```

こうすると、「AIがどこで強く言いすぎたか」を後から調べられます。

## 検証したこと

ローカルでは次を実行します。

```
bash findevil/scripts/run_findevil_local_checks.sh
```

期待される確認項目は次です。

```
claims_total=5
exact_status_accuracy=1.0
unsupported_claims_blocked=1
false_confident_supported_claims=0
```

この数字の意味は、AIが作ったclaimを、証拠とground truthに対して確認しているということです。

重要なのは、失敗や弱い断定を隠さないことです。

## この記事で主張しないこと

このデモは、実際のSANS SIFT実行や本物のマルウェア帰属を主張しません。

また、自動隔離や本番環境での封じ込めも主張しません。

安全な主張範囲は次です。

* ローカルのsanitized case dataで動く
* 証拠IDつきclaimを作る
* unsupported claimを止める
* analyst report、accuracy report、integrity reportを出す
* 人間承認なしで危険な封じ込めへ進まない

## 読者が自分のAI調査ツールに使うなら

AIに調査を手伝わせるなら、次を入れるとかなり安全になります。

1. レポート本文とは別にclaim一覧を作る
2. claimごとにevidence IDを必須にする
3. 証拠がない断定はunsupportedに落とす
4. 危険な結論はhuman approvalへ回す
5. ground truthや期待状態と比較するaccuracy reportを作る
6. 証拠ファイルのhashやsource manifestを残す

AIの速さを活かすには、速い断定を許すのではなく、速く証拠に縛る方がよいです。

## まとめ

`Evidence-Locked DFIR Agent` のテーマは、AIに事故調査を任せることではありません。

AIが出した調査claimを、証拠ID、status、accuracy、human approvalで制御することです。

セキュリティ領域では、AIの自信より、証拠に戻れることの方が重要です。
