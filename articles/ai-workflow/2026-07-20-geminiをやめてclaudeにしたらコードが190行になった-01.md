前回、「記憶は保存するものじゃない、生成するものだ」という設計の話を書いた。素材だけをNDJSONで残し、意味は文脈に合わせて毎回生成させる——そういう外部脳の設計論だった。

この記事は、その設計を積んでいた**土台**の話をする。より正確には、半年かけて組み上げた土台を、昨日ほとんど捨てた話だ。捨てられた、と言ってもいい。残ったコードは、その時点で190行だった。

技術の残酷さと美しさが同居する話をする。

## 結論——公式APIより、ローカルへの直接書き込みが勝った

先に結論を書く。GoogleのAI（Gemini）よりも、他社のAI（Claude）の方が、Google Driveをシンプルに扱えた。

半年ほど前、ボクは「思考をGoogle Driveに保存する外部脳システム」を作っていた。当時の相棒はGeminiで、保存経路はこうだった。Drive APIで認証を通し、ブリッジ（中継プロセス）を立ち上げ、仮想環境（venv）を有効化し、JSONを組み立てて投げ、クラウド側の同期を待つ。モジュールは十数ファイルに分かれ、サービスアカウントのJSONキーとフォルダIDの共有設定が前提だった。それを、コツコツと積み上げていた。

それが昨日、190行のローカル書き込みスクリプト1本に置き換わった。外部ライブラリは、ゼロ。

## Geminiの呪縛——「公式性」がアーキテクチャを規定する

なぜGeminiだと複雑になったのか。原因はモデルの賢さではなく、**公式性**にあった、というのがボクの見立てだ。

GeminiはGoogleの正統な血統だ。だからこそ、Google Driveを「正式なAPI」という手続きで叩く前提で発想が組まれる。認証を通し、リクエストを投げ、重い同期を待つ。この経路は堅牢で、権限管理も正しい。ただし、そこに乗るコードは必然的に、認証・リトライ・トークン管理・ブリッジという周辺装置を抱え込む。

ここで効いているのは、ツールが持つ「正しい使い方」の引力だ。公式のAPIが存在すると、それを使うことが正解に見える。正解に見えるものへ寄せていくと、アーキテクチャはAPIの都合に従って肥大していく。これを、この記事では**公式性の罠**と呼ぶことにする。手段の正統さが、目的（思考を1行残すこと）に対して過剰なのだ。

## 関心の所在——ClaudeはDriveを「見なかった」

対してClaudeは、Googleの派手なAPIを最初から見ていなかった。目の前にあったのは「ローカルの共有フォルダ」だけで、そこにPythonスクリプトで直接ファイルを書き込んだ。

問いの立て方が違った。「Drive APIをどう正しく叩くか」ではなく、「思考を1行、確実にファイルへ追記するには何が要るか」。後者に答えるなら、必要なのは標準ライブラリのファイル入出力だけだ。そして——Drive同期という巨大なインフラは、「OS常駐のGoogleドライブクライアントに丸投げする」という一手で消えた。ローカルに書けば、あとはOSが勝手にクラウドへ同期する。アプリはその存在を知らなくていい。

実家であるはずのGoogleのAIが、実家の倉庫への出し入れに正規の手続きで四苦八苦している横で、他所の家の子が裏口から一瞬で荷物を運び込んだ。皮肉だが、設計としてはこちらが正しい。**関心を、同期という他者の責務から切り離した**からだ。

## なぜ一瞬で移植できたか——核心と周辺の分離

ここが、この記事でいちばん書きたい点だ。

190行への収束は、正直に言えば「負け惜しみ」から始まった。半年の苦労がこれで済むなら、あの時間は何だったのか、と。だが冷静に振り返ると、一瞬で移植できたのは偶然ではなく、最初の設計が正しく**分離**されていたからだった。

システムを、二層に分けて考えていた。

* **設計の核心**：SSOT（Single Source of Truth）は`journal_by_day/`のNDJSONイベント列であること。書き込みは必ずゲート（原子的書き込み＋スキーマ検証）を通すこと。既存レコードは修正・削除しない（append-only）こと。インデックス類はそこから全量再生成できる派生物であること。
* **周辺インフラ**：それを実際にどこへ、どうやって永続化するか。Drive APIか、ローカルファイルか、という実装の選択。

Geminiの時代に肥大していたのは、後者（周辺インフラ）だけだった。前者（核心）は、保存先が何であろうと変わらない。だからDrive連携という周辺を丸ごと捨てても、核心はそのまま新しい土台に載った。

土台が腐っていれば、環境が変わった瞬間にシステム全体が崩壊していただろう。Drive部分を躊躇なく捨てられたこと自体が、設計の勝利だった。ここから引き出せる一般則はこうだ——**移植性は、賢い実装からではなく、核心と周辺の分離から生まれる**。理論（核心）さえ独立していれば、過去の実装はすべて捨てられる。それが強さだと思う。

## 何が変わったか——4段階から1段階へ

体感でいちばん大きかったのは、保存という行為そのものの手数だ。

以前（Gemini＋Drive API）の保存フロー：

1. ブリッジを起動する
2. 仮想環境（venv）を有効化する
3. JSONを手動で組み立てて貼り付ける
4. Driveへの同期を待つ

現在（Claude＋ローカル）の保存フロー：

1. Claudeに「保存して」と言う

ターミナルは要らない。認証情報の設定も要らない（そもそもDrive APIを使わない）。あとは会話するだけで思考がローカルのNDJSONに書き込まれ、Googleドライブクライアントが裏で勝手に同期する。

保存の中身はこれだけだ。ロックを取り、直前のハッシュを読み、レコードを組み、追記し、インデックスを更新する。

```
with journal_lock(mirror_base):
    prev_hash = get_prev_hash(mirror_base)          # 直前レコードのhash
    record_obj = build_record(author, prev_hash, record)
    filename, line_num = write_journal(mirror_base, record_obj)  # NDJSONに追記
    update_indexes(mirror_base, record_obj, filename, line_num)  # 派生インデックス更新
```

各レコードは`prev_hash → hash`のSHA256連鎖でつながる。1件でも改ざん・欠損があれば、連鎖が切れて検知できる。

```
{
  "id": "abc123def456",
  "t_utc": "2026-07-06T00:09:16.515454Z",
  "author": "yourname",
  "prev_hash": "前レコードのhash（連鎖）",
  "hash": "sha256({t_utc, prev_hash, record})",
  "v": 1,
  "record": { "type": "thought", "text": "...", "tags": ["..."] }
}
```

Drive API・認証・ブリッジ・仮想環境——それらが担っていた仕事は、すべて「OSに任せる」か「そもそも要らない」に還元された。

## 190行の、その後

正直に補足しておく。この「190行」は、切り替えたその日の数字だ。いま同じスクリプトを数えると、約280行に育っている（実コードで213行）。

増えたのは機能ではなく、**堅牢性**だ。全書き換えを一時ファイル経由の原子的置換（`os.replace`）にし、追記のたびに`fsync`でディスクへ確実に落とし、同一マシン内の競合を排他ロックで直列化し、途中でプロセスが落ちても本体ファイルが壊れない（torn lineを構造的に作らない）ようにした。二度と繰り返したくない事故から学んだ行たちだ。

だが本質は変わっていない。依存ライブラリは依然ゼロ。単一ファイル。標準ライブラリだけ。十数モジュール＋Drive API＋サービスアカウントという旧アーキテクチャとは、比較にならない軽さのままだ。行数が190から280へ増えたこと自体が、むしろ健全な兆候だと思っている。増えた一行一行が、賢さではなく耐久性を買っている。

## 過去の正解が負債になる

この一件を、個人的な失敗談として閉じたくない。ここには、開発者なら誰もが直面する一般的な構造があるからだ。

半年かけてDrive API・ブリッジ・venv・JSON貼り付けという経路を作り上げた。当時、それは正解だった。その環境で、その道具立てで、外部脳を成立させるには必要な複雑さだった。それが環境の変化——ローカルフォルダに直接書けるAIエージェントの登場——によって、一夜にして負債に変わった。

「過去の正解」にしがみつくと、技術の進化そのものに置いていかれる。だが逆に言えば、核心と周辺を分離しておけば、進化が来たときに周辺だけを差し替えて乗り換えられる。**過去の正解を、負債にせず資産のまま捨てられるかどうか**は、その分離の設計にかかっている。

昨日から始めていたら、今日できていた——そういう感覚は、正直まだ消えない。けれどそれは、能力や判断の問題ではなかった。技術の進化のタイミングと、自分の取り組みのタイミングが、少しだけずれていただけだ。そして、ずれていた半年に組んだ設計の核心が、今日の一瞬の移植を可能にした。だから無駄ではなかった、と言い切れる。

## おわりに

道具は変わる。同期の仕組みも、認証の作法も、賢いAIの名前も変わる。変わらないのは、何を唯一の正とし、何を派生物とし、どこで両者を分けるか、という設計の芯だけだ。

道具は捨てられる。設計は残る。  
残るように分けておけ。
