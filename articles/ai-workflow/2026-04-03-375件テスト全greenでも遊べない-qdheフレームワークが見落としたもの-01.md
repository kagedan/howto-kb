## 前回のあらすじ

[前編](https://zenn.dev/belka_dolphin/articles/c885376e01b59d)では、AI駆動開発フレームワーク\*\*QDHE（Quality-Driven Harness Engineering）\*\*を設計し、AI人狼ゲームのバックエンドをStep 10まで完走させた。352件のテスト全GREEN、ミューテーションスコア70.95%、カバレッジ95%超え。

その後、フロントエンド（React）を実装し、バックエンドと結合してv2パイプラインを回した。テストは375件に増え、全件GREEN。

**で、実際にブラウザでゲームを起動したら、1回も遊べなかった。**

この記事は、そこから何が起きたか、なぜ起きたか、QDHEをどう改善するかの記録だ。

## まず、このゲームの規模感

「どのくらいのソフトの話なの？」という疑問に先に答えておく。

| 区分 | ファイル数 | 行数 |
| --- | --- | --- |
| ソースコード | 38 | 3,480 |
| テストコード | 33 | 9,488 |
| ドキュメント | 18 | 5,566 |
| 設定ファイル | 17 | 711 |
| **合計** | **106** | **19,245** |

テスト/ソース比率は**2.73倍**。ソースコード1行に対してテストが2.73行ある。普通のプロジェクトで1:1あれば優秀と言われるなか、QDHEのテストファーストが数字に表れている。

技術スタックはTypeScript / React 18 / Vite / Node.js / WebSocket / Claude API。pnpm monorepoで3パッケージ（shared / backend / frontend）構成。小〜中規模のリアルタイムWebアプリとしては妥当なサイズ感だ。

そして、この2.73倍のテストが全部GREENなのに、ゲームは動かなかった。

## 9件のバグ — 何が壊れていたか

ブラウザを開いてから「遊べる」状態になるまでに、9件のバグを対話的デバッグで修正した。一覧はこうなる。

| # | 概要 | 根本原因 | 深刻度 |
| --- | --- | --- | --- |
| 1 | バックエンドが即終了（`import.meta.url` Windows互換性） | 環境依存 | Critical |
| 2 | dotenvが.envを見つけられない（cwd差異） | 環境依存 | Critical |
| 3 | 画面が紺色のまま（WebSocketContext.Provider未配線） | 設計漏れ | Critical |
| 4 | ゲームが一瞬で終わる（人間入力待ちなし） | 要件漏れ | Critical |
| 5 | 昼に投票モーダルが先に表示される | 設計漏れ | Major |
| 6 | 1回発言したらAIが即しゃべり始める | 設計漏れ | Major |
| 7 | プレイヤーIDがそのまま表示される | テスト観点漏れ | Minor |
| 8 | 占い師で夜アクション画面が固着する | 設計漏れ | Critical |
| 9 | Vite proxyのポートが固定値 | 環境依存 | Major |

Critical 5件、Major 3件、Minor 1件。**Critical 5件のうち、FMEAで完全に検出できたものは0件だった。**

### 根本原因の分類

| パターン | 件数 | 該当 |
| --- | --- | --- |
| 環境依存（OS/ランタイム/モノレポ構造） | 3件 | #1, #2, #9 |
| 設計漏れ（状態遷移・配線・タイミング） | 4件 | #3, #5, #6, #8 |
| 要件漏れ（暗黙の前提が明文化されていない） | 1件 | #4 |
| テスト観点漏れ（表示の人間可読性） | 1件 | #7 |

9件中、FMEAが部分的にカバーしていたのは3件（33%）。残り6件（67%）はFMEAのスコープ自体に故障モードが存在しなかった。

## 代表的なバグを深掘りする

### 「バックエンドが即終了する」— 環境の罠

`pnpm run dev` を叩いたら、バックエンドが `exited with code 0` で即終了した。

原因はESMのエントリポイントガード。`import.meta.url === \`file://${process.argv[1]}``で「このファイルが直接実行されたか」を判定していたが、Windows + tsx環境では`import.meta.url`が`file:///D:/...`（スラッシュ3つ）、`process.argv[1]`から変換した結果が`file://D:/...`（スラッシュ2つ）になり、常に不一致。つまりメインモジュールが「自分は直接実行されていない」と判断して何もせず終了していた。

FMEAにはF-03（モジュール解決エラー）があったが、「dist未生成/パスエイリアス未設定」に限定されていた。OSごとのURL正規化差異はスコープ外。そして自動テストでは `createServer()` を直接呼び出すため、このガード行を一度も経由しない。**テストは原理的にこのバグを検出できない。**

### 「ゲームが一瞬で終わる」— 最も構造的な問題

Start Gameを押したら、一瞬でリザルト画面に飛んだ。人間は発言も投票もできない。

原因は、ゲームループが `processNightPhase → processDayPhase → processVotePhase` を同期的に呼び出し、人間の入力を待つ機構がなかったこと。GameEngineには「人間が未投票ならランダム投票する」フォールバックが実装されていたから、テスト上は正常に動作する。ただし実際のゲーム体験としては、人間が何もできないまま終わる。

これが**QDHEの構造的限界を最も端的に示すバグ**だ。個別モジュールのテストを積み上げても、ユーザー体験レベルの品質は保証されない。テストリストの手動確認項目には「ゲームが進行し結果が表示されること」はあったが、「人間が操作できること」の粒度がなかった。

### 「占い師で画面が固着する」— 状態遷移の盲点

人間が占い師のとき、夜フェーズで占いターゲット選択モーダルが表示される。だがタイムアウトでゲームエンジンがランダム選択して昼に進んでも、フロントの `actionState` が `pending` のまま残り、モーダルが消えない。昼フェーズでもテキスト入力が無効化されたまま。

`appReducer` は `actionState.status === 'sent'` のときだけリセットし、`pending` はリセットしない設計だった。フェーズ変更時のUI状態リセットという観点がFMEAに存在しなかった。

これを状態遷移図で表すとこうなる。

左が修正前、右が修正後。修正前は `pending → フェーズ変更` のパスが存在しなかった。夜にタイムアウトで昼に進んだとき、`actionState` がリセットされず画面が固着する。修正後はフェーズ変更時に `actionState` を無条件で `idle` に戻す。

このバグは、**「ゲームフェーズ」と「UI状態」という2つの状態機械のクロス分析**ができていれば事前に発見できた。FMEAの故障モードとして「フェーズ変更時にUI状態がリセットされない」を登録し、状態の組み合わせマトリクスを作れば、`pending × フェーズ変更 = 固着` というセルが空白で残る。

## なぜ375件のテストで検出できなかったか

3つの構造的原因がある。

**第一に、テストが個別モジュールの正しさしか検証していない。** `createServer()` を直接呼ぶ、コンポーネントを個別にレンダリングする。統合された状態での動作は一切検証していない。これは前編でミューテーションスコア43%→70%に上げたときも同じ構造で、「テストの数」ではなく「テストのスコープ」が問題だった。

**第二に、テスト環境と実行環境が乖離している。** テストでは `process.env` を直接設定するからdotenvのパス解決は通らない。WebSocketはモック。Reactコンポーネントは独立レンダリング。「プロセス起動→dotenv読み込み→WebSocket接続→React描画」という実環境の連鎖は、テストの中に存在しない。

**第三に、スモークテストが1本もなかった。** 「実際にプロセスを起動して、主要操作を1回実行する」レベルのテストがゼロ。前編の102件→352件の増強でもミューテーションスコアを上げるための境界値テストが中心で、統合テストの追加にはなっていなかった。

つまり、テスト/ソース比率2.73倍という数字は「個別モジュールの検証密度」を表しているに過ぎず、「システムとして動くか」の保証にはなっていなかった。

## FMEAの検出力を評価する

9件のバグに対するFMEAの検出力を改めて整理する。

| 評価 | 件数 | 割合 |
| --- | --- | --- |
| FMEAが部分的にカバー（故障モードの粒度が粗い） | 3件 | 33% |
| FMEAがスコープ外（故障モード自体が存在しない） | 6件 | 67% |
| FMEAで完全に検出可能だった | **0件** | **0%** |

厳しい結果だ。ただし、これはFMEAという手法の限界ではなく、**FMEAの適用スコープが狭すぎた**ことが原因だ。v2のFMEAは14件の故障モードを分析したが、すべてがアプリケーションロジックレベル。以下のカテゴリが体系的に欠落していた。

* **環境差異系**: OSのパス表現、モノレポのcwd挙動、ツールチェーンの実行モデル
* **フロントエンド配線系**: Context Provider、状態リセット、ランタイム依存のクラッシュ
* **状態遷移系**: フェーズ変更時のUI状態マトリクス、非同期処理のタイミング

## QDHEをどう改善するか

振り返りから導いた改善提案は5つ。すべてフレームワーク本体の修正であり、個別プロジェクトの対症療法ではない。

### 改善1: FMEAに「環境差異」カテゴリを必須追加

Step 3のFMEA分析で、以下の故障モードテンプレートを標準搭載する。

| ID | 故障モード | 検出方法 |
| --- | --- | --- |
| ENV-01 | OSごとのパス表現差異（file:// URL、パス区切り） | スモークテスト |
| ENV-02 | モノレポcwd差異（pnpm/npm workspace実行ディレクトリ） | スモークテスト |
| ENV-03 | 設定値ハードコード vs 環境変数の不整合 | 静的解析 |
| ENV-04 | ツールチェーン固有動作（tsx vs ts-node、Vite vs webpack） | スモークテスト |

### 改善2: FMEAにフロントエンド配線カテゴリを追加

| ID | 故障モード | 検出方法 |
| --- | --- | --- |
| FE-WIRE-01 | Context Provider未設定（useContextがnullでthrow） | ルートコンポーネントテスト |
| FE-WIRE-02 | フェーズ変更時のUI状態リセット漏れ | 状態遷移マトリクステスト |
| FE-WIRE-03 | コンパイル通過 + ランタイムクラッシュ | スモークテスト |

### 改善3: テストリストに「スモークテスト」を必須化

Step 4のテストリスト作成で、以下を必須項目にする。

* **SMOKE-01** 実プロセス起動: バックエンドプロセスを起動し、5秒以内にヘルスチェックが通ること
* **SMOKE-02** フロントエンド描画: ルートコンポーネントをレンダリングし、初期画面が例外なく表示されること
* **SMOKE-03** 主要ユースケース1回実行: 最も基本的なユーザー操作を実行し、期待する応答が返ること

### 改善4: 配線スコープでもE2Eシナリオを1本含める

「配線が正しい」と「繋いだ先が動く」は別の検証。配線パイプラインでも、主要ユースケースのE2Eテストを最低1本含める。今回であれば「ゲーム開始→1回発言→1回投票→結果画面表示」まで。

### 改善5: 要件定義に「ユーザーストーリー完遂条件」を明記

ユーザーの元の要求（「遊べるところまで」）から、操作レベルの完遂条件を明文化する。

> **完遂条件**: 人間プレイヤーが以下の操作を1回以上実行できること
>
> 1. ゲームを開始する
> 2. 昼フェーズでテキスト発言する
> 3. 投票フェーズで対象を選択する
> 4. 結果画面で勝敗と役職を確認する

「遊べる」の定義が「配線が繋がる」ではなく「人間が体験として成立する」であることを、要件の段階で合意する。

## 教訓: テスト全GREEN ≠ 動く

前編では「102件GREENの幻想」を語った。ミューテーションテストで43%→70%に引き上げ、テストの殺傷力を検証した。それでも今回、375件GREENで遊べなかった。

前編の幻想は「テストの質」の問題だった。今回の幻想は「テストのスコープ」の問題だ。個別モジュールをどれだけ精密にテストしても、それは統合品質の保証にはならない。

```
テストの質    → ミューテーションテストで検証（前編の教訓）
テストのスコープ → スモークテスト/E2Eで検証（今回の教訓）
テストの改ざん  → SHA256ハッシュ凍結で防止（QDHEの核）
```

QDHEのテスト凍結は「AIにテストを直させない」という非対称性でシステムを支えている。この思想自体は正しかった。だが、凍結するテストのスコープが狭ければ、凍結の意味も限定される。**良質なテストを凍結しないと、良質なロックにはならない。**

今回の改善を反映したQDHEでは、FMEAの故障モードテンプレートに環境差異とフロントエンド配線を追加し、テストリストにスモークテストとE2Eを必須化する。次にパイプラインを回すとき、同じ種類のバグは事前に検出できるはずだ。

フレームワークは、失敗から学んで進化するものだ。
