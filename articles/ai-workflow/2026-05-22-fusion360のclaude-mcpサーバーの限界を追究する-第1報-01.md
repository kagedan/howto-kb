# はじめに

ClaudeのMCPサーバーにFusion360が正式サポートされました．Fusion360をClaudeで扱うことができます．  
早速，使用感を確かめた記事がありますね．  
<https://note.com/tokyomakers/n/n35aae26aa835>

私も早速ラズパイ５のケースを作成してもらいました．ラズパイ5のCADデータをインポートしたファイルをFusion360上で開いておけばClaude Desktopが自動に認識してケースをモデリングしてくれました．  
![](https://static.zenn.studio/user-upload/cf64dbbfc566-20260522.png)

この試行で分かったことは以下です．

* 干渉を検出できる
* ここを直してと赤丸で囲った画像を見せれば直してくれる
* 指定しなければスケッチに寸法が書かれないため，再設計が難しい
* 断面にスナップフィットを作るような高度なことは難しい

使った所感としては，初心者が一からデザインする手間が省けるのと，多くのフィレットを付けたりザグリを付けるなどの単純作業をやるのに便利だと思いました．Claude自体もFusion360のAPIの知識が乏しいようで，想定していない動きに苦悩している様子が見られました．

しかし，私は設計業務の自動化を目指し，最適なプロンプトを追究したい．そして，3D CAD設計とプログラミンはとても似ている部分があり，その共通事項を言語化することを目指しています．言語化をすることで，AIによる機械設計が容易になると思っています．

# ネジの下穴を作る

次は，ネジの下穴をつくってもらう．  
ネジの下穴についての知識があり，下穴の径についてちゃんと質問してきた．  
![](https://static.zenn.studio/user-upload/84e2a6c62853-20260515.png)

![](https://static.zenn.studio/user-upload/9738e455c406-20260515.png)  
1試行でできたが，かなり悩んだ様子．矩形のスケッチが原点中心に変更されているし，矩形の中心を辺の半分の距離で指定することで割り出している．補助線が一切ない

![](https://static.zenn.studio/user-upload/8e9e514c4da0-20260515.png)

振り返りをさせてみた

```
### 全体の流れ
最初の3×3×3 mm立方体から、50×50×10 mmで中心にM3タッピング下穴の貫通板まで仕上げました。最終的にルール（寸法記載、完全拘束、ユーザーパラメータ化）を満たした設計ができています。

### 良かった点
- **ユーザーパラメータ駆動の設計が一貫して維持できた**: 初期の `cube_size` から `body_x` / `body_y` / `body_z` / `tap_hole_dia` への発展でも、寸法は常にパラメータ参照で記述。Fusion 360上で値を変えれば全体が追従する状態。
- **完全拘束を最終的に全スケッチで達成**: 最後の BodyBaseSketch と PilotHoleSketch は両方 `isFullyConstrained = True`。
- **検証を毎回入れた**: バウンディングボックス計測・円筒面の半径と高さチェックなど、APIで結果を数値検算してから次に進めたので、見た目だけでなく寸法的に正しいことを保証できた。
- **タッピング下穴径の確認を取った**: 樹脂か金属かで径が変わるため、勝手に決めず仕様確認した。

### つまずいた点と学び

**1. `addTwoPointRectangle` の自動拘束を誤認**
最初の3mm立方体作成時、水平/垂直拘束が自動付与されると想定したが、実際には付与されず完全拘束にならなかった。→ 検証スクリプトで拘束数を実測することで発見。以降は `addByTwoPoints` × 4 + 明示的な水平/垂直拘束に統一。

**2. `HoleFeature` の内部バリデーションエラー**
中心点を `sketchPoints.add` で追加して `setPositionBySketchPoint` を試みたが `logicalSelection` エラー。→ シンプルな円スケッチ + カット押し出しで代替。Fusion APIでは穴の中心点として「曲線端点」や「投影点」の方が安定する印象。

**3. `addMidPoint` がソルバー失敗**
中心配置を中点拘束で表現する自然な方法がスクリプト経由で `VCS_SKETCH_SOLVING_FAILED` 連発。→ 点-点間の距離寸法（`body_x/2`, `body_y/2`）で同等の意味を持たせて回避。意味的には完全拘束として等価。

**4. プロファイル選択ミスで板を削りすぎた**
円スケッチを上面に置くとプロファイルが2つ（円内側・円外側）生成されるが、`profiles.item(0)` が外側になっていて、最初は板そのものを削ってしまった。→ 面積最小のプロファイルを選ぶヘルパーに切り替え。

**5. undo/redoの粒度に手こずった**
エラーリカバリでundoしたら想定より深く巻き戻ってしまうケースが何度かあった。スクリプト1回 = タイムライン1エントリ単位なので、エラー時はredoで戻し、状態を都度確認してから次に進む必要があった。

### 次回に活かせること
- 完全拘束は **最後に必ず `isFullyConstrained` で確認** する（拘束を追加した時点で「成功」を信じない）
- Fusion APIでのスケッチ更新は **削除→再構築の方が、既存拘束の編集より安定** することが多い
- プロファイル選択は **常に面積最小を選ぶ** ヘルパーを持っておく（特に貫通穴系）
- 中心配置の表現は `addMidPoint` より **点-点距離寸法** の方がスクリプトでは堅牢
```

ルールに学びを追加してもらう

```
あなたが学んだことを踏まえ，私が一番最初に述べたルールに必要最小限のルールを追加して，ルールを修正してください
```

以下，Claudeが追加したルール  
スケッチについて

```
位置や中心の同定には補助線（コンストラクション線）を活用する。既存ジオメトリの破壊的編集ではなく、補助線で意図を表現する
```

スケッチ平面について（追加）

```
B-Rep（フィーチャ生成物の面・エッジ）を直接スケッチ平面として参照しない。コンストラクション平面を新規作成し、その上にスケッチを描く
他スケッチのジオメトリを参照する場合は projection で投影してから使う
```

# まとめ

今回の試行で最終的にプロンプトに記述するルールは以下になった．

```
スケッチについて
 
寸法を記載
完全拘束する
過剰拘束に注意するため，一度スクリプトを吟味し，一致させるジオメトリが正しいか確認する
位置や中心の同定には補助線（コンストラクション線）を活用する。既存ジオメトリの破壊的編集ではなく、補助線で意図を表現する
B-Rep（フィーチャ生成物の面・エッジ）を直接スケッチ平面として参照しない。コンストラクション平面を新規作成し、その上にスケッチを描く 他スケッチのジオメトリを参照する場合は projection で投影してから使う

パラメーターについて
設計パラメータとしてユーザーパラメータを作成し，外形寸法や共通化される寸法に適用する
アセンブリについて
コンポーネント単位で1責務に分割する
アセンブリ階層で共有されるジオメトリは、最上位コンポーネントで定義する（ボルトスケッチなど）
```

次回は，簡単なアセンブリを作らせて，今回得られたルールでどこまで実用的な設計ファイルを作成できるか検証する．
