# usertourの仕組みを理解する — OSSオンボーディングツールを試してみる

## はじめに

この記事では、オープンソースのユーザーオンボーディングプラットフォーム「usertour」の仕組みと構成を解説します。

単に「使い方」ではなく、「なぜこの構成でSPA・マルチページアプリ問わず動くのか」「セルフホスティング時に何が動いているのか」を理解することを目的としています。

公式リポジトリ: <https://github.com/usertour/usertour>

## 前提知識・用語整理

**ユーザーオンボーディングツールとは**  
プロダクト内でユーザーに表示するガイド（ツアー、チェックリスト、アンケート）を管理するためのシステム。エンジニアがコードを変更することなく、PMやデザイナーがGUI上でフローを設定・更新できる。

**代表的な商用SaaS**  
Appcues / Userflow / Userpilot / Userguiding / Chameleon

usertourはこれらのOSS代替として位置付けられています。

**主要な技術用語**

* **モノレポ**: 複数のパッケージ・アプリケーションを1つのリポジトリで管理する構成
* **pnpm workspace**: Node.jsのパッケージマネージャpnpmが提供するモノレポ管理機能
* **Turbo**: Vercel製のモノレポ向けビルド・タスクオーケストレーションツール

## アーキテクチャの仕組み

### モノレポ構成

usertourはpnpm workspaceとTurboを採用したTypeScriptモノレポです。TypeScriptが全体の98.6%を占めており、フロントエンド・バックエンドともにTypeScriptで統一されています。

この構成により以下のメリットがあります。

* 型定義をフロントエンド・バックエンド間で共有できる
* 依存関係の管理がシンプル
* Turboによる差分ビルドで開発サイクルが高速

### なぜブラウザアプリ全般で動くのか

usertourがSPA・マルチページアプリ問わず動作する理由は、JavaScriptスニペットの挿入による動作モデルにあります。

ユーザー側のアプリケーションに軽量なJSスニペットを1行追加するだけで、usertourのSDKが読み込まれます。このSDKがDOM操作・イベント監視を行うため、フレームワーク非依存で動作します。具体的には以下の処理が行われます。

1. SDKがusertourのサーバーと通信し、対象ユーザー向けのフロー定義を取得
2. 現在のURLやDOMの状態に基づいて表示タイミングを判断
3. ツアーのUIをオーバーレイとしてDOMに注入

この方式はAppcuesやPendoなどの商用SaaSも採用している標準的なアーキテクチャです。

### セルフホスティング時に何が動いているか

Dockerで起動すると、大きく以下のコンポーネントが立ち上がります。

* **APIサーバー**: フロー定義の管理、ユーザー属性の受け取り、分析データの蓄積
* **管理画面（ダッシュボード）**: ノーコードでフローを作成・編集するGUI
* **SDK配信**: ユーザーアプリに組み込むJSスニペットの配信

`docker compose up -d` 一発で全コンポーネントが起動するのは、Docker Composeによるサービス定義がリポジトリに含まれているためです。

## セットアップ手順（仕組みを踏まえた上で）

```
# 1. 環境変数を設定
cp .env.example .env
# .envを編集（DBの接続情報、シークレットキーなど）

# 2. 起動
docker compose up -d

# 3. ブラウザでアクセス
open http://localhost:8011
```

起動後、管理画面でアカウントを作成し、自分のアプリへのSDKスニペットを取得します。そのスニペットをアプリのHTML（`<head>`タグ内など）に貼り付けることで連携が完了します。

## ユーザーターゲティングの仕組み

usertourの特徴の一つが高度なユーザーターゲティングです。

アプリ側からSDKを通じてユーザー属性（プランタイプ、登録日、使用した機能など）を送信すると、管理画面でそれらの属性を使った表示条件を設定できます。

```
// アプリ側での属性送信例（概念コード）
usertour.identify('user-id-123', {
  plan: 'free',
  signed_up_at: '2026-01-15',
  has_completed_onboarding: false
});
```

このようにサーバーサイドのデータをSDKに渡すことで、「無料プランで登録から7日以内のユーザーにだけオンボーディングツアーを表示する」といった細かい制御が可能になります。

## ハマりポイントとその技術的な理由

### 環境変数の設定漏れ

`.env.example` に記載されている値のうち、特にシークレットキーとデータベース接続情報は必須です。設定漏れがあると起動時にエラーとなります。必ず`.env.example`の全項目を確認してから起動してください。

### ローカルとステージング・本番でのSDKエンドポイント切り替え

セルフホスティングの場合、SDKが通信するエンドポイントをusertour.ioではなく自分のサーバーに向ける設定が必要です。この設定を忘れると、ローカルで動いているように見えても実際にはクラウド版に通信してしまいます。

### SPA環境でのURL変更検知

ReactやVue等のSPAでは、ページ遷移がURLの変更ではなくJSのルーティングで行われます。usertourのSDKはこれを自動で検知する仕組みを持っていますが、カスタムルーターを使っている場合は手動でページ変更イベントを通知する必要があるケースがあります。

## まとめ

usertourは以下の点で技術的に完成度が高いOSSです。

* TypeScriptモノレポ + pnpm + Turboによるモダンな構成
* JSスニペット方式によるフレームワーク非依存の動作
* Dockerによるセルフホスティングの簡易化
* カスタム属性・イベントを使った高度なターゲティング

有料SaaSと同等の機能をセルフホスティングで実現したいチームにとって、現時点で最も実用的な選択肢の一つです。
