ローカルでAIエージェントからGitHubに接続しようとしたとき、手段としてMCPとCLIがありました。

GitHub CLIとMCPの両方を試してみたところ、自分の利用範囲ではMCPを使う場合にPAT（Personal Access Token）が必要になる場面がありました。PATは便利ですが、流出リスクを考えると少し不安があります。

一方でGitHub CLIは、PATを自分で生成しなくてもブラウザログインで利用できるため、安心して使いやすいと感じました。

そこで、AIエージェントからの利用を想定したCLIツールを、学習目的で作ってみることにしました。

CLIからAPIを呼び出すだけであれば、コマンドライン引数を受け取り、HTTPリクエストを組み立て、レスポンスを表示する、という流れはある程度イメージできます。

一方で、安全に使うことまで考えると認証まわりが一気に重要になります。ブラウザログイン後にCLIへどう戻すのか、トークンをどこに保存するのか、期限切れをどう扱うのか。このあたりの知見が少なかったため、小さなCLIとサーバーをGoで作って確認しました。

この記事では、実装したタスク管理機能そのものではなく、**AIエージェントが利用するCLIを安全に使うための認証まわり**を中心にまとめます。

## 作ったもの

今回作ったのは、`vagent` というCLIと、`vagent-server` というローカルAPIサーバーです。

```
vagent login
vagent task create --title "調査タスク" --prompt "このリポジトリの構成を調べて"
vagent task list
vagent task get <task-id>
vagent task run <task-id>
```

タスクAPIは、認証付きAPI呼び出しを試すための題材です。現時点ではタスクはインメモリ管理で、`run` も疑似的にステータスを更新するだけにしています。

## 構成

リポジトリは、CLIとサーバーを同じGoモジュールの中に置いています。

```
agent-cli-server/
├── cmd/
│   ├── vagent/          # CLIのエントリポイント
│   └── vagent-server/   # サーバーのエントリポイント
├── internal/
│   ├── api/             # APIのリクエスト/レスポンス型
│   ├── cli/             # CLIコマンドとAPIクライアント
│   ├── config/          # 設定ファイルとプロファイル管理
│   └── server/          # HTTPサーバー、認証、タスクAPI
├── README.md
└── ARCHITECTURE.md
```

## 認証フロー

ログインは、OAuth 2.0のAuthorization Code Flow + PKCEに寄せた流れにしています。

厳密なOAuthサーバーを作り込んでいるわけではありませんが、CLIツールでよくある「ブラウザを開いてログインし、CLIに戻ってくる」体験を再現しています。

全体の流れは以下のようになります。

作る前は、ブラウザログインの中で何が必要になるのかをあまり整理できていませんでした。作ってみると、特に以下の3つを押さえる必要があると分かりました。

1. ローカルコールバック
2. PKCE
3. トークン保存

### 1. ローカルコールバック

全体図では、認可URLを開いてから `GET /callback?code=...&state=...` でCLIに戻ってくるまでの部分です。

CLI側では、`vagent login` 実行時に `127.0.0.1:0` で一時的なコールバックサーバーを起動します。

ログイン操作自体はブラウザで行います。CLIはログインURLに `redirect_uri`、`state`、`code_challenge` を付けてブラウザを開きます。ブラウザで認可すると、サーバーは `redirect_uri` に `code` と `state` を付けてリダイレクトします。

今回は学習用の実装なので、ユーザーID/Passwordによるログインや、アクセス許可の同意画面は簡略化しています。本来であれば、ここでユーザー認証を行い、ログイン済みユーザーに対してCLIへのアクセス許可を確認したうえで、認可コードを発行します。

この `redirect_uri` の向き先が、CLIが一時的に起動したローカルのコールバックサーバーです。CLIは戻ってきた `state` を検証し、認可コードを受け取ります。

この時点ではまだアクセストークンは取得していません。受け取った認可コードを安全にトークンへ交換する部分が、次のPKCEです。

### 2. PKCE

全体図では、`code_challenge` を認可URLに含めるところから、`code_verifier` を使ってトークン交換するまでの部分です。

PKCEは、認可コードを盗まれても、それだけではアクセストークンに交換できないようにする仕組みです。

CLIはログイン開始時に `code_verifier` というランダムな文字列を作り、そこから `code_challenge` を作ります。認可リクエストでは `code_challenge` だけをサーバーに渡し、`code_verifier` はCLI側に保持します。

ブラウザ認可後、CLIは受け取った `authorization code` と `code_verifier` を `/oauth/token` に送ります。サーバーは `code_verifier` から `code_challenge` を再計算し、認可リクエスト時に保存した値と一致するか確認します。

つまり、認可コードだけを横取りされても、`code_verifier` がなければトークン交換できません。

CLIツールでは固定の `client_secret` を安全に埋め込めないため、PKCEで認可コードの横取りに備えることが重要になります。

| 名前 | 役割 |
| --- | --- |
| `code_verifier` | CLIだけが保持するランダムな文字列 |
| `code_challenge` | `code_verifier` から作った検証用の値 |
| `authorization code` | ブラウザ認可後にCLIへ戻ってくる一時的なコード |
| `access token` | APIを呼び出すためのトークン |

流れとしては以下のようになります。

1. CLIが `code_verifier` を作る
2. CLIが `code_verifier` から `code_challenge` を作る
3. CLIが `code_challenge` 付きの認可URLをブラウザで開く
4. ブラウザが `GET /oauth/authorize` でサーバーへ `code_challenge` を渡す
5. サーバーは認可後に `code_challenge` を認可コードに紐づけて保存する
6. トークン交換時に、CLIが `authorization code` と `code_verifier` をサーバーに送る
7. サーバーが `code_verifier` から `code_challenge` を再計算する
8. 保存済みの `code_challenge` と一致すれば、トークンを返す

### 3. トークン保存

全体図では、`access token / refresh token` を受け取った後、OS認証情報ストアと設定ファイルに保存する部分です。

トークン本体と、CLIが次回以降の実行で使うメタデータは保存先を分けています。

| 保存先 | 保存するもの |
| --- | --- |
| OS認証情報ストア | アクセストークン、リフレッシュトークン |
| 設定ファイル | active profile、server URL、user ID、有効期限 |
