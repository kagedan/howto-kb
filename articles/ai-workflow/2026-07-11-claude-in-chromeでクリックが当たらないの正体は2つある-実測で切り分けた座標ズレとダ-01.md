Claude in Chrome(Chrome MCP)でブラウザを自動操作させていると、こういう症状に当たることがあります。

* クリックしても狙った要素に当たらない
* ポインタが目標より右にズレて、何度やっても当たらない
* スクリーンショットを撮ってしばらくすると画面サイズが変わっている
* ダブルクリックしても編集状態にならず、行選択で止まる

600個以上のWebツールを運用するプロジェクトで、AIによるブラウザ操作を日常的に回す中でこの症状に繰り返し遭遇しました。当初は要素側の不具合を疑って、何時間も溶かしました。

切り分けてみると、原因は要素ではありませんでした。座標ベース操作の前提が崩れていた。それだけの話で、2つに集約されます。

## 原因1: ウィンドウ幅の確定が遅れて、座標が右にズレる

ページ読み込み直後の `innerWidth` は小さい値になっていることがあります。実測では 1664 でした。`document.readyState` が `complete` になった後も、2〜3秒かけて本来の幅(1920)へ広がっていきます。この間、`outerWidth` / `outerHeight` が 0 を返すことすらあります。

この確定前にスクリーンショットを撮り、その座標でクリックすると何が起きるか。撮影時よりも画面が広がっているため、クリック位置が右方向にズレます。1664→1920は約1.15倍なので、右側の要素ほどズレ幅が大きい。

「スクショの後に画面サイズが変わる」「ポインタがズレて当たらない」の正体はこれでした。

canvas系の特殊なアプリに限った話ではありません。遅延読み込みや動的レイアウトを持つ普通のHTMLページでも起きます。「入力欄がクリックできない」と詰まっていたケースの一部は、要素が押せないのではなく、単に座標が空振りしていただけでした。

### 診断方法

`javascript_tool` でビューポート値を読み、安定しているかを確認します。

```
({ url: location.href, readyState: document.readyState,
   dpr: window.devicePixelRatio,
   innerW: window.innerWidth, innerH: window.innerHeight,
   outerW: window.outerWidth, outerH: window.outerHeight })
```

判定基準は4つ。`outerW` が 0 ならウィンドウサイズ未確定で、まだ操作してはいけない。少し待って再度読み、`innerW` が前回と違えば確定中なので待つ。2回連続で同じ `innerW` が出て `outerW` が非ゼロなら、サイズは確定済みです。そして4つ目として、**スクリーンショットを撮った時点から `dpr`(devicePixelRatio)が変わっていないこと**。DPIの異なるディスプレイへウィンドウを移すと座標全体が1664→1920と同じ形でリスケールされますが、移動後の値は「正しい値」としてすぐ安定するため、`innerW` の一致チェックだけでは検出できません(dev.to側の記事に寄せられた指摘。文末の追記を参照)。

## 原因2: double\_clickの間隔が長すぎて、シングルクリック2回になる

`computer` ツールの `double_click` アクションは、2回のクリックの間隔がアプリ側のダブルクリック判定時間より長くなることがあります。結果、ダブルクリックではなくシングルクリック2回として解釈され、テキスト編集状態に入らずに行選択などで止まる。厄介なのは、ツール側は正しくダブルクリックを送ったつもりでいることです。

ログだけ見ても気づけません。

## 対策

### 方針A(推奨): 座標を捨てて要素ベースで操作する

普通のHTMLページなら、これが最も確実です。`computer` の座標クリックをやめ、`find`(自然言語での要素検索)や `read_page`(アクセシビリティツリー)で要素参照を取り、その参照に対してクリック・入力する。入力は `form_input` で要素参照に値を入れる。要素参照はレイアウトが多少動いても有効です。

ただし万能ではありません。参照が追っているのは**ノードの同一性**であって、位置や役割ではない。追試(2026-07、素のDOM・Reactなし)では、DOM再構築でノードが差し替えられると参照は明示エラーで死に、ノードが移動・役割変更されると**成功ログ付きで旧ノード(いまは別の位置・別の役割)を踏みました**。つまり参照は使い捨てが原則です。**操作の直前に `find` で取り直し、ページ遷移や submit をまたいでキャッシュしない。** さらに、未確定ウィンドウでは有効な参照へのクリックが無言で不発になる事象も確認しており、参照クリックも内部では座標解決を経ている可能性があります。方針Aは座標問題を大幅に縮小しますが、ゼロにはしません(文末の追記を参照)。

例外はcanvas/Flutter製のアプリ(Riveエディタなど)です。UIがHTML要素として存在しないため、`find` / `read_page` では何も取れません。その場合は方針Bしかない。

### 方針B: 座標を使うなら「確定待ち」+「高速2連クリック」をセットにする

1. 操作前にサイズ確定を待つ。上の診断スニペットで `innerW` が安定するまで待ってからスクリーンショットを撮る
2. スクショとクリックの間を空けない。撮影→その最新スクショの座標で即クリック、を1セットにする。間が空いたら座標を取り直す
3. ダブルクリックは `double_click` を使わず、同座標への高速 `left_click` 2連で作る。`browser_batch` で2つの `left_click` を連続実行すると間隔が詰まり、判定が通ります

```
browser_batch([
  { name: "computer", input: { action: "left_click", coordinate: [x, y], tabId } },
  { name: "computer", input: { action: "left_click", coordinate: [x, y], tabId } },
])
```

4. クリック後は `zoom` で対象を拡大し、期待した状態(選択・編集・ハイライト)になったかを確認してから次へ進む。空振りに気づかず進むと、以降の操作が全部ズレます

## チェックリスト

| 症状 | 最初に疑うこと |
| --- | --- |
| クリックが当たらない | サイズ確定(原因1)。未確定なら待つ |
| ダブルクリックが効かない | 高速2連クリックに変える(原因2) |
| 普通のHTMLページで詰まる | 方針A(要素ベース)へ切り替え |
| 参照クリックがエラーになる / 別の要素に当たる | 参照が stale。操作直前に find で取り直す(使い捨て運用) |
| canvas/Flutterアプリ | 方針B一択。要素は取れない |

## 制限と注意

この記事の数値(1664→1920、2〜3秒)は筆者の環境(Windows / 1920x1080)での実測です。ディスプレイ構成やDPIスケーリングで値は変わるので、持ち帰ってほしいのは数値ではなく「確定を待ってから撮る」という原則の方です。

Claude in Chromeは活発に更新されているプロダクトなので、将来ツール側がサイズ確定を待つようになれば、この対策は不要になるかもしれません。また、座標操作は本質的に壊れやすい手法です。要素ベース(方針A)が使える場面では常にそちらを選ぶのが正解です。

## 追記 (2026-07-11)

同じテーマを英語で書き下ろした[dev.to側の記事](https://dev.to/mxhlix/claude-in-chrome-keeps-missing-clicks-i-measured-why-there-are-exactly-two-causes-2m8p)のコメント欄で2点の指摘を受けました。「要素参照も rerender で stale になる」「settle判定は devicePixelRatio の変化を見落とす」。素のDOM(Reactなし)で追試したところ、参照の stale はエラーになるケースと無言で旧ノードを踏むケースの両方が再現・確認できたため、方針Aと診断基準の記述を改訂しました。要素参照が追うのはノードの同一性だけで、位置や役割ではない — 検証の詳細なやり取りは dev.to 側のコメントスレッドにあります。

なおDPI移動(モニタ間)の物理的な再現は単一ディスプレイのため未実施です。ただし追試環境自体が dpr 1.1(110%スケーリング)で、6月の検証時(innerW が 1920 に確定 = 実質 dpr 1.0)と同じ物理ディスプレイなのに CSS上の解像度が既に変わっており、`innerW` だけを見る判定が環境変化に対して脆いことは自環境でも裏付けられました。
