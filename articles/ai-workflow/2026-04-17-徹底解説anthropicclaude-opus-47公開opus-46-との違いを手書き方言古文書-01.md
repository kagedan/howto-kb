日本時間 2026 年 4 月 16 日、Anthropic は新モデル 「Claude Opus 4.7」 を公開しました。

Opus 4.7 は Opus 4.6 を置き換える後継モデルです。同社のフラッグシップ Mythos Preview が限定提供のまま残る一方で、Opus 4.7 は主要プラットフォームで本日から一般提供されています。

今回の主眼は、難しいコーディング作業を手を離しても任せやすくすることにあります。あわせて、指示を「文字通り」に実行する方向へも一段強化されました。既存プロンプトが挙動を変える可能性があるため、移行の視点も大切になります。

手元で Opus 4.6 と 4.7 を 4 つのタスクで並べて走らせたところ、指示外の情報を付け足したり、原文の微細な差分を落とすといった挙動差が観察できました。ベンチマークの数字だけでなく、こうした振る舞いの差のほうが、実務のプロンプトにとっては影響が大きいかもしれません。

この記事では、公式発表と API 仕様・価格、英語圏の反応を整理したうえで、4 つの実機検証をもとに違いを見ていきます。

## 要点：

* **コーディング性能と指示追従が、Opus 4.6 から一段強化されました。**  
  SWE 系や Terminal Bench など、公開された主要なコーディング評価の複数項目で改善が示されています。指示を「文字通り」に読むようになったため、これまで曖昧に解釈されていたプロンプトは再調整が必要になる場合があります。
* **画像入力の上限ピクセル総量が、従来モデルの 3 倍超に拡大されました。**  
  長辺は 2,576 ピクセル（約 3.75 メガピクセル）まで扱えます。日本語ウェブの密度の高いスクリーンショットや、複雑な図表からのデータ抽出を狙えるレベルに入ってきた印象です。
* **Claude Code 側に /ultrareview コマンドと新 effort 「xhigh」 が加わりました。**  
  価格は Opus 4.6 から据え置きで、入力 $5／出力 $25（100 万トークンあたり）。API モデル名は claude-opus-4-7 です。

公式発表によると、Opus 4.7 は SWE-bench シリーズ（Pro / Verified）や Terminal Bench 2.0 といった主要なコーディング評価の複数項目で、Opus 4.6 から改善を示しています。とくに難易度の高いタスクで伸び幅が大きい、という分布になっています。

### 画像理解の解像度向上

画像入力の上限ピクセル総量が、従来の Claude モデルの 3 倍を超える規模に拡大されました。長辺では 1,568 → 2,576 ピクセルへ広がり、約 3.75 メガピクセルまで扱えます。

## 【検証】Opus 4.6 と Opus 4.7 を、公式ベンチに含まれない 4 タスクで並べる

ベンチマークの数字は、指標によって上下します。実務で体感できるのは「同じ指示に対する出力の微妙な違い」のほうです。ここでは、Opus 4.7 が強みを主張している領域を中心に、4.6 と 4.7 を同一プロンプトで走らせて並べてみます。

実行環境は [AGI Cockpit](https://chatgpt-lab.com/n/nf178a3d39e1d) 経由の Claude Code terminal agent と、Claude.ai Web（Incognito chat）の両方を用い、effort は両モデルとも high に揃えました。**各タスクは 1 回ずつの実行**で、再現試験は行っていません。サンプル数は少ないので、差は「このサンプルで観察された傾向」として読んでください。

### 検証3: コーディングのセンス — ペリカンが自転車で森を走る SVG

**プロンプト（抜粋）:** 森の中を自転車で走るペリカンを、単一 HTML ファイルで、アニメーション込みで描く

**Opus 4.6 の出力**

![](https://assets.st-note.com/production/uploads/images/268014416/picture_pc_c3f6ce0d5cabb720b5840c6a947fb4fc.gif?width=1200)

**Opus 4.7 の出力**

![](https://assets.st-note.com/production/uploads/images/268014450/picture_pc_82d31e320a9c51fd7f7c261b09bf5e84.gif?width=1200)

この 1 回の実行で観察できた差分は、次のとおりです。（AGI Cockpit 経由・effort: high・各1回のみ。harness 側の要因も混じる可能性があるため、性能差の一般化には再現試験が必要です。）

* **生成時間**: Opus 4.6 が思考フェーズを含めて 約 17 分、Opus 4.7 が 3 分 19 秒。この回では 4.7 の方がかなり短時間で終わりました
* **出力サイズ**: 4.6 が 19 KB / 463 行、4.7 が 38 KB / 805 行。この回では 4.7 の方が出力量が倍近くありました
* **アニメーション方式**: 4.6 は JavaScript の requestAnimationFrame で毎フレーム再計算する手続き型、4.7 は CSS @keyframes で宣言的に動かす型。同じ「動くペリカン」でも、実装パラダイムが異なる選び方になりました
* **アクセシビリティ**: 4.7 の出力には prefers-reduced-motion への対応が含まれており、動きを抑えたいユーザー向けの配慮が入っていました
* **描画品質のヒント**: 4.7 の出力には image-rendering: optimizeQuality などの描画品質指定も含まれていました

動いているペリカンは、正直に言うと、**ぱっと見の派手な差はありません**。どちらかというと 4.6 の方が質がいいように思えます。

### 検証4: 文章のセンス — 47語だけで書く超短編

制約が強い創作タスクで、言葉の選び方 に差が出るかを確認しました。プロンプトは「生涯で使える言葉が1万語と宣告された人物が、生まれたばかりの我が子を腕に抱いている。残された言葉は47語。情景と感情の両方を立ち上げる短い一段落を書いてください」というものです。

**Opus 4.6 の出力:**

> 残りの言葉を数えることが、もう何年も前から呼吸と同じになっていた。四十七。毛布にくるまれた体は驚くほど軽く、まだ名前すら声に出していない。呼べば一語減る。歌えば減る。叱ることも、許すことも、いつか来る別れの朝に「行ってらっしゃい」と言うことも、すべてこの四十七のなかにある。小さな拳が人差し指を握った。喉元まで言葉がせり上がり、唇が開きかけ——そして閉じた。代わりに頬を寄せると、産毛の匂いがした。沈黙は語数を消費しない。それだけが、いま自分に許された、惜しみない愛の形だった。

**Opus 4.7 の出力:**

> 産院の窓から差しこむ薄い朝の光のなかで、彼は数えることをやめられなかった。腕のなかの小さな重みが、指先の内側で規則正しく上下している。四十七。残された数字を、彼は自分の肋骨の裏側に刻みつけるようにして黙っていた。看護師が何か問い、妻が微笑みながらこちらを見た。返事の代わりに、彼はただ頷いた。頷くたびに、使わずに済んだ一語が胸の奥で静かに積もっていく気がした。赤ん坊が小さく口を開け、世界にむかって最初の息を探している。その無音のあくびを見つめながら、彼は決めた。最初に呼ぶ名前だけは、必ず、声に出す。あとの四十六は、この子の一生のどこかへ、分けて渡していくのだと。窓の外で、名前を知らない鳥が一度だけ鳴いた。彼はまだ、何も言わなかった。

いかがでしょうか。どちらが上と評価するのは難しい、というのが正直な感想です。Opus 4.6の癖だった「——」の出力はなくなっているかもしれません。

effort レベルの既定値は全プランで **xhigh に引き上げ** られており、特に指定しなくても新しい既定で動きます。

### 新 effort level「xhigh」の位置づけ

Opus 4.7 から、reasoning の effort レベルに新しい選択肢 **xhigh** が加わりました。既存の level と並べると、次のような位置づけになります。

* **none / low / medium**: 軽めの処理、応答速度を優先
* **high**: API の既定値。従来モデルでは多くのケースで最初に試す値
* **xhigh（新）**: high と max の中間。**Claude Code では全プランで既定値** に引き上げられました
* **max**: 最も深い推論。時間とトークンを最大限に使う

Anthropic は Opus 4.7 について、コーディングやエージェント用途では high か xhigh から始めることを推奨 しています（公式ブログ／移行ガイド）。Claude Code 側の既定値が xhigh に引き上げられたのも、この推奨に沿った変更です。

なお、Opus 4.7 は公式ブログで「高い effort レベル、とくにエージェント用途の後半ターンで思考量が増える」と説明されています。API 利用時のコスト見積もりには、このあたりを踏まえて実トラフィックで測り直すのが安全です（詳しくは 「提供形態と API 価格」 章を参照）。

### Claude Code の新機能

**/ultrareview — 変更差分のディープレビュー**

/ultrareview は、現在のブランチの変更差分を読み込み、バグや設計上の問題を丁寧に指摘する 専用レビューセッションを起動するスラッシュコマンドです。通常のレビュー依頼より、一段深い読み込み を前提に設計されています。

* **対象プラン**: Pro / Max（それ以外は未対応）
* **無料枠**: Pro・Max の各ユーザーに **3 回分** の /ultrareview が付与
* **使い方**: Claude Code セッション中に /ultrareview と入力

![](https://assets.st-note.com/img/1776386263-v9GpskSOaNJcHKtPZBDbelzX.png?width=1200)

通常のレビュー依頼とどう違うかといった運用面は、次回以降の記事で扱う予定です。

**auto mode — Max ユーザーにも拡大**

auto mode は、Claude Code が許可の問い合わせを最小化しつつ、Claude 自身の判断で処理を進めていくモードです。完全に permissions をスキップする（dangerous skip）のとは異なり、安全側の判断を含んだままタスクを長く走らせやすくなります。

## まとめ

Claude Opus 4.7 は、Opus 4.6 を置き換える後継モデルとして、コーディング・指示追従・画像理解・Claude Code 連携 が同時に更新されました。

本記事で試した検証タスクでは、指示に含まれていない情報を付け足したり、細部の助詞・語尾を落としたりする挙動 が、4.6 と違う形で見えました。

一方、文章のセンスや SVG のぱっと見では、派手な差は出ませんでした。もはや 同じ方向の性能比較では測れない vibes の領域 に入ってきている、というのが正直な感想です。優劣より、手癖や相性で選ぶフェーズに入りつつあります。

料金が Opus 4.6 と据え置きで、主要プラットフォームで即日使えるリリースです。本番自動化に Opus を組み込んでいる場合は、テスト用のトラフィックから段階的に切り替えるのが安全な入り方になります。AGIラボでは今後、記事だけではなく講座などを通して /ultrareview や auto mode の運用ノウハウ、既存プロンプトの調整事例などを別途取り上げていく予定です。
