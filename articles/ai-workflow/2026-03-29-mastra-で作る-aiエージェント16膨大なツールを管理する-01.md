[Mastra で作るAI エージェント](https://zenn.dev/shiromizuj/articles/a0a1659e9f05b6) というシリーズの第16回です。

---

2026年2月4日にリリースされた Mastra 1.2　で **ToolSearchProcessor**（動的ツール検索）が導入されました（[ニュースリリース](https://mastra.ai/blog/changelog-2026-02-04)）。これは、AIエージェントが膨大な数のツールを抱えている場合に、「使わないのに保持しているだけでコンテキストを圧迫する」という問題に対する、Mastraによる解決アプローチです。

# 背景

## AIエージェントにおける大規模ツールライブラリの課題

モダンなAIエージェントは、多様なタスクを実行するために数十から数百のツールにアクセスする必要があります：

* **MCP (Model Context Protocol) サーバー**: 外部データソースへのアクセス
* **統合ヘビーなセットアップ**: GitHub、Slack、CRM等の多数のAPI
* **内部ツールカタログ**: 企業固有の数百のビジネスツール

しかし、従来のアプローチでは、**すべてのツール定義を最初から含める**必要があり、これが深刻な問題を引き起こしていました。

## 開発の背景

ToolSearchProcessorは、この問題を根本的に解決するために設計されました。

**コンセプト**:  
**"エージェントに必要なツールを検索させ、必要な時にロードさせる"**

従来の「すべてのツールを最初から与える」アプローチから、「必要なツールを動的に発見・ロードする」パターンへの転換です。

## 課題1. コンテキストウィンドウの消費

**問題**: ツール定義がトークンを大量に消費

```
// 従来のアプローチ
const agent = new Agent({
  name: "my-agent",
  tools: {
    // ツール1: 定義で約50-100トークン
    createGitHubIssue: createTool({ ... }),
    
    // ツール2: 定義で約50-100トークン
    sendSlackMessage: createTool({ ... }),
    
    // ツール3-100: さらに9,800トークン以上...
    // ... 合計約10,000トークン
  },
});
```

**具体的な影響**:

| ツール数 | ツール定義のトークン | 利用可能なコンテキスト | 影響 |
| --- | --- | --- | --- |
| 10 | ~500-1,000 | 99% | 問題なし |
| 50 | ~2,500-5,000 | 90-95% | やや影響 |
| 100 | ~5,000-10,000 | 80-90% | 大きな影響 |
| 500 | ~25,000-50,000 | 50%以下 | 深刻な問題 |

**例**: gpt-4o (128K context)の場合

* 100ツール定義: ~10,000トークン (約8%)
* 残りコンテキスト: ~118,000トークン
* **しかし、実際の会話には10ツールしか使わない**

→ **90ツール分の定義（約9,000トークン）が無駄**

**問題**: 大量のツール定義がLLMの判断を混乱させる

## 基本コンセプト

ToolSearchProcessorは、**オンデマンドなツール検索・ロード**を実現します：

![](https://static.zenn.studio/user-upload/e934904f7c50-20260317.png)

## 主な特徴

### 特徴1. BM25ベースのツール検索

**BM25 (Best Matching 25)**: 情報検索で広く使われる関連性ランキングアルゴリズム

```
// エージェントが「GitHubにissueを作成したい」と判断
↓
search_tools("GitHub issue creation")
↓
BM25アルゴリズムが関連度スコアでランキング:
1. createGitHubIssue     (スコア: 0.85) ← 最も関連性が高い
2. updateGitHubIssue     (スコア: 0.42)
3. listGitHubIssues      (スコア: 0.38)
4. sendSlackMessage      (スコア: 0.05)
...
100. calculateTax        (スコア: 0.00)
```

**BM25の利点**:

* キーワードマッチングが高速
* 関連度の高いツールを優先
* トークン消費なし（ローカル計算）

ToolSearchProcessorは、2つの**メタツール**をエージェントに提供します：

```
// ツール定義（自動生成）
{
  name: "search_tools",
  description: `キーワードで利用可能なツールを検索します。
  
  現在持っていない機能が必要なときに使用してください。
  一致するツールの名前と説明の一覧を返します。
  
  関連するツールが見つかったら、load_tool を使用して利用可能にしてください。`,
  parameters: {
    query: "string", // 検索キーワード
  },
}

// 使用例（エージェントが自動実行）
search_tools({ query: "GitHub issue" })
// → [ { name: "createGitHubIssue", description: "..." }, ... ]
```

```
// ツール定義（自動生成）
{
  name: "load_tool",
  description: `指定したツールをこの会話にロードします。
  
  ロードされたツールはこのスレッド内で使用可能になります。
  ロード済みのツールはこの会話の以降のターン全体で保持されます。`,
  parameters: {
    toolName: "string", // ロードするツール名
  },
}

// 使用例（エージェントが自動実行）
load_tool({ toolName: "createGitHubIssue" })
// → ツールがスレッドに追加され、次のターンから使用可能
```

### 特徴3. スレッド単位の状態管理

ロードされたツールは**会話スレッド内で永続化**されます：

```
会話の流れ:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
スレッドID: "thread-123"

ターン1:
ユーザー: "GitHubにissueを作成して"
エージェント: 
  1. search_tools("GitHub issue") を実行
  2. load_tool("createGitHubIssue") を実行
  3. "ツールをロードしました"

ターン2:
ユーザー: "タイトルはBug Reportで"
エージェント:
  1. createGitHubIssue を実行 ← すでにロード済み
  2. "Issueを作成しました"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

異なるスレッド:
スレッドID: "thread-456"
ユーザー: "GitHubにissueを作成して"
エージェント:
  → createGitHubIssueはまだロードされていない
  → 再度search/load が必要
```

**スレッド分離の利点**:

* 会話ごとに最適なツールセット
* 他のスレッドと干渉しない
* メモリの効率的な使用

### 特徴4. 94%のコンテキスト削減

公式発表によるコンテキスト使用量の削減率：**約94%**

**Before**（従来のアプローチ）:

```
100ツール定義を含む
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
リクエストトークン:
- システムプロンプト: 500トークン
- ツール定義（100個）: 10,000トークン
- 会話履歴: 1,000トークン
合計: 11,500トークン
```

**After**（ToolSearchProcessor使用）:

```
メタツール2つ + 実際に使用する3ツールのみ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
リクエストトークン:
- システムプロンプト: 500トークン
- メタツール定義（2個）: 200トークン
- ロード済みツール定義（3個）: 300トークン
- 会話履歴: 1,000トークン
合計: 2,000トークン

削減率: (11,500 - 2,000) / 11,500 = 82.6%
```

**さらに大規模な場合**（500ツール）:

```
Before: ~50,000トークン（ツール定義のみ）
After:  ~500トークン（メタツール + 実使用3ツール）
削減率: 99%
```

# 課題の解決方法

## 解決方法1. コンテキストウィンドウの効率化

**解決策**: 必要なツールだけをロード

```
// Before: すべてのツール定義を含める
const agent = new Agent({
  tools: {
    tool1, tool2, ..., tool100  // 10,000トークン
  }
});

// After: ToolSearchProcessorで動的ロード
const agent = new Agent({
  inputProcessors: [
    new ToolSearchProcessor({
      tools: {
        tool1, tool2, ..., tool100  // プロンプトには含まれない
      }
    })
  ],
  // 実際のプロンプトには search_tools と load_tool のみ（200トークン）
});
```

**効果**:

* 初期トークン: 98%削減（10,000 → 200）
* 必要なツールのみロード（300トークン追加）
* 総削減率: 94%以上

## 解決方法2. レスポンス品質の向上

**解決策**: 関連性の高いツールのみを提示

```
エージェントの判断プロセス:

Before（100ツール提示）:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 100個のツールリストをスキャン
2. 無関係なツールがノイズとなる
3. 最適なツールの判断に時間がかかる
4. 誤ったツールを選択する可能性

After（ToolSearchProcessor）:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 「GitHubツールが必要」と判断
2. search_tools("GitHub") を実行 ← BM25で関連ツールのみ
3. 関連度の高い3-5ツールのみが返される
4. 的確なツール選択
5. load_tool("createGitHubIssue")
```

**BM25検索の精度**:

* **topK設定**: 上位K件のみ返す（デフォルト: 5）
* **minScore設定**: 最低スコアの閾値（デフォルト: 0.0）

```
new ToolSearchProcessor({
  tools: { /* 100ツール */ },
  search: {
    topK: 5,        // 上位5件のみ
    minScore: 0.1,  // スコア0.1以上のみ
  }
})
```

## 解決方法4. スケーラビリティの確保

**解決策**: ツール数に依存しないアーキテクチャ

```
ツール数とトークン消費の関係:

Before（線形増加）:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
100ツール → 10,000トークン
500ツール → 50,000トークン
1000ツール → 100,000トークン（上限に達する）

After（定数）:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
100ツール → 200トークン（メタツール）+ 300トークン（実使用）
500ツール → 200トークン（メタツール）+ 300トークン（実使用）
1000ツール → 200トークン（メタツール）+ 300トークン（実使用）

```

**スケーラビリティの実現**:

* ツール登録: 何千でも可能
* 検索時間: O(n) but 高速（BM25）
* トークン: 実際に使用するツールのみ
* メンテナンス: ツール追加は既存会話に影響しない

# 利用方法

## 基本設定

**以下の設定を書くだけで有効になる**

```
import { Agent } from "@mastra/core";
import { ToolSearchProcessor } from "@mastra/core/processors";
import { openai } from "@ai-sdk/openai";

// ステップ1: ToolSearchProcessorを作成
const toolSearch = new ToolSearchProcessor({
  tools: {
    createIssue: githubTools.createIssue,
    sendEmail: emailTools.send,
    updateJira: jiraTools.update,
    // ... 数百のツールを記載可能
  },
  search: {
    topK: 5,        // 検索結果の上位5件を返す（オプション）
    minScore: 0.1,  // 最低スコア閾値（オプション）
  },
});

// ステップ2: inputProcessorsに追加
const agent = new Agent({
  name: "my-agent",
  model: openai("gpt-5.4"),
  inputProcessors: [toolSearch],  // ← これだけで有効化
  tools: {
    // 常に利用可能にしたいツールがあればここに記載
    // ロードされたツールは自動的にマージされる
  },
});

// ステップ3: 通常通り使用
const response = await agent.generate("GitHubにissueを作成して");
// エージェントが自動的に:
// 1. search_toolsで検索
// 2. load_toolでロード
// 3. 該当ツールを実行
```

**自動的に行われること**:

1. `search_tools` メタツールの生成・提供
2. `load_tool` メタツールの生成・提供
3. エージェントによる自動検索・ロード
4. スレッド内でのツール永続化
5. 既存ツールとの自動マージ

`tools`プロパティは**オブジェクト形式**で、キーがツール名、値がツールオブジェクトです：

```
import { createTool } from "@mastra/core";

// 方法1: 個別にツールを定義
const toolSearch = new ToolSearchProcessor({
  tools: {
    // キー: ツール名（エージェントが参照）
    // 値: ツールオブジェクト
    createGitHubIssue: createTool({
      id: "createGitHubIssue",
      description: "GitHubリポジトリに新しいIssueを作成します",
      inputSchema: z.object({
        title: z.string(),
        body: z.string(),
        repo: z.string(),
      }),
      execute: async ({ context }) => {
        // ツールの実装
      },
    }),
    
    sendSlackMessage: createTool({
      id: "sendSlackMessage",
      description: "Slackチャンネルにメッセージを送信します",
      inputSchema: z.object({
        channel: z.string(),
        message: z.string(),
      }),
      execute: async ({ context }) => {
        // ツールの実装
      },
    }),
    
    // ... さらに追加
  },
});

// 方法2: 既存のツールオブジェクトを使用
import { githubTools } from "./tools/github";
import { slackTools } from "./tools/slack";
import { jiraTools } from "./tools/jira";

const toolSearch = new ToolSearchProcessor({
  tools: {
    // 既存ツールの展開
    ...githubTools,  // { createIssue, updateIssue, ... }
    ...slackTools,   // { sendMessage, createChannel, ... }
    ...jiraTools,    // { createTicket, updateTicket, ... }
  },
});

// 方法3: 動的にツールを生成
const toolSearch = new ToolSearchProcessor({
  tools: Object.fromEntries(
    allToolDefinitions.map(def => [
      def.name,
      createTool(def)
    ])
  ),
});

// 方法4: カテゴリごとに整理（推奨）
const toolSearch = new ToolSearchProcessor({
  tools: {
    // GitHub関連
    createGitHubIssue: githubTools.createIssue,
    updateGitHubIssue: githubTools.updateIssue,
    closeGitHubIssue: githubTools.closeIssue,
    
    // Slack関連
    sendSlackMessage: slackTools.sendMessage,
    createSlackChannel: slackTools.createChannel,
    
    // Email関連
    sendEmail: emailTools.send,
    scheduleEmail: emailTools.schedule,
    
    // データベース関連
    queryDatabase: dbTools.query,
    updateDatabase: dbTools.update,
    
    // ... 数百のツールを整理して記載
  },
});
```

**ツール名の命名規則（推奨）**:

* **明確性**: 何をするツールか一目で分かる名前
* **一貫性**: 同じパターン（例: `create*`, `update*`, `send*`）
* **説明的**: BM25検索で適切にマッチするキーワードを含む

```
// Good: 説明的で検索しやすい
createGitHubIssue
sendSlackMessage
updateJiraTicket
searchDatabase

// Bad: 曖昧で検索しにくい
tool1
gh
msg
db
```

# 制限事項と注意点

## 注意点1. 初回検索のオーバーヘッド

**注意**: 最初のターンでは search + load が必要

```
ターン1（初回）:
- search_tools: ~500トークン（入力+出力）
- load_tool: ~100トークン
- 合計: ~600トークン追加

ターン2以降:
- search/load不要（ツールがロード済み）
- トークン: 通常通り
```

**対策**: 短い会話では効果が薄い可能性あり

## 注意点2. 検索精度の限界

**注意**: BM25はキーワードベース、意味理解はしない

```
// マッチしやすい
Query: "GitHub issue creation"
Tool: "createGitHubIssue" ← "GitHub", "issue", "create"がマッチ

// マッチしにくい
Query: "バグレポートを作成"
Tool: "createGitHubIssue" ← 日本語が含まれていない

Query: "make a ticket"
Tool: "createGitHubIssue" ← "ticket"がissueとマッチしない
```

**対策**:

* ツール説明文に同義語を含める
* 多言語対応が必要な場合は説明文を多言語化

## 注意点3. ツール数と検索速度

**注意**: ツール数が多いと検索時間が増加（O(n)）

```
10ツール: ~1ms
100ツール: ~10ms
1000ツール: ~100ms
10000ツール: ~1秒
```

**対策**: 通常は問題ないが、10,000ツール以上の場合は検討が必要

## 注意点4. サーバーレス環境での制限

**注意**: インメモリキャッシュはプロセス再起動で消失

```
サーバーレス（Vercel, Lambda等）:
- 各リクエストが新しいプロセス
- キャッシュは毎回空
- 永続化（Memory）が必須

対策: Memory設定を必ず有効にする
```

# ツールの数が膨大にならざるを得ない場合に活用を検討

ツールの数が20を超えたら使うことを考えたほうがよさそうです（私はまだ経験が無いですが）。日本語の問題など、もう少し実際に使ってみて探っていきたいです。

# コンテキスト・エンジニアリング

Mastra の Ver1.2 は、コンテキスト・エンジニアリングに正面から解決策を示した意欲的なアップデートでした。今回ご紹介した「ツール検索プロセッサ」もその一つです。
