## はじめに

**Findy DevOps × AI Agent Hackathon 2026** に、個人開発の作品「**Citify**」で参加しました。この記事は、その**参加記録**です。

Citify は、自治体の議事録・プレスリリースを若者向けに翻訳して TikTok 風の For You フィードで届けるマルチエージェント AI プロダクトです。「自分の街、自分の世代の話を、60 秒で。」をコンセプトにしています。

この記事では作品の宣伝よりも、**開発中に踏んだ「沼（ハマりどころ）」を 7 つ**、原因と対処をセットで残します。ADK・Vertex AI・マルチエージェント・ローカル TTS・Remotion あたりを触る人の遠回りが少しでも減れば幸いです。

**想定読者**

* Google ADK でマルチエージェントを組んでいる／組もうとしている人
* Vertex AI RAG Engine や Imagen を本番 Cloud Run から使う人
* ハッカソンでデモ動画まで作る予定の人（ローカル TTS・Remotion を含む）

## Citify とは

一言でいうと「役所の言葉を、あなたの年代の言葉に翻訳して届ける」プロダクトです。設計上いちばん大事にしたのは、**同じ自律エージェントの設計パターンを 2 つの領域に使い回した**ことです。

> 計画 → 並列で専門家エージェント → 統合 → 批判（悪魔の代弁者）→ 人間ゲート（自動実行はしない）

このまったく同じパターンを、

1. **Watcher（街の見張り番）** … ユーザーに合う街を AI が自分で調べて結論を出す
2. **Ops crew（`/ops`）** … 自分たちの運用（スクレイパー健全性・コスト・データ鮮度）を AI が自分で診断する

の両方に適用しました。「なぜマルチエージェントか」と「なぜ DevOps × AI か」を、1 つの設計思想に寄せる狙いです。

アーキ図（図1）は情報量が多いので、細部は追わなくて大丈夫です。まずは「同じ自律パターンを 2 ドメインに使い回した」という要点だけ掴めれば十分です。

技術スタックは Cloud Run / Cloud Run Jobs / Pub/Sub / BigQuery / Firestore、AI 側は ADK / Gemini 2.5 / Vertex AI RAG Engine / Imagen 3、フロントは Next.js、インフラは Terraform + Cloud Build です。

ここからが本題の「沼」です。

## 沼 1. Watcher の「完全自律」は本番で破綻した

最初、Watcher は**完全自律型**で作りました。エージェントが毎ステップ「次に何をするか」を LLM に決めさせる設計です。ローカルでは動きました。

ところが本番の smoke テストで破綻しました。

* 毎ステップ LLM 判断 → **逐次実行になりレイテンシが積み上がる**
* モデルの癖で**ツール選択が偏る**（同じツールばかり呼ぶ／必要な観点を飛ばす）

そこで「**計画は LLM、実行は並列**」に再設計しました。

* プランナーが最初に調査計画を立てる
* 必要なツール（統計比較・議題検索・人口推移など）を**並列で走らせる**
* 結論を出す前に **自己検証（批判）** を挟む
* カバレッジの床を保証するガードレールを入れる

![図4: Watcher の自律実行の進捗（4 専門家を並列調査 → 自己批判・悪魔の代弁者で再点検）](https://static.zenn.studio/user-upload/deployed-images/7094fc06f2377d29b1f883ed.png?sha=0fd5f81035fa17bf74ef3b730d2135440b570ba9)

再設計の前後を図にすると次の通りです。

![図5:（概念図）Watcher の自律性を「完全自律（毎ステップ LLM）」から「計画は LLM・実行は並列」へ再設計した Before / After](https://static.zenn.studio/user-upload/deployed-images/c47963eddbc1ccd740669a59.png?sha=3797dbb32767a28a8b74e69e744a6c6e60c186d0)

対話で街を探す Concierge（街さがしの対話 UI を担うエージェント）を ADK（Agent Development Kit、Google のエージェント構築フレームワーク）で組んだところ、**候補（candidates。ADK / Gemini の応答に含まれる候補フィールド）が常に 0 件**になりました。ツール自体は正しい値を返しているのにです。

私のケースでの原因は、**ツールが pydantic モデルを返すと、後段（ADK のフロー層）でそれが辿れる（キーで中身を取り出せる）dict にならない**ことでした。浅くはラップされるものの、ネストした pydantic が JSON 化しきれず、後続の応答生成で構造を辿れずに拾えていませんでした。

対処は、ツールを **dict を返すラッパで包む**ことです（`functools.wraps` で名前・シグネチャは保持）。

```
import functools

def dict_returning(func):
    @functools.wraps(func)  # __name__ / signature を保持（ADK がスキーマ生成に使う）
    def wrapper(*args, **kwargs):
        result = _jsonable(func(*args, **kwargs))  # pydantic → JSON-able に
        return result if isinstance(result, dict) else {"result": result}
    return wrapper
```

これで本番でも candidates が返るようになりました。

なお `_jsonable` は pydantic を JSON-able に変換する自作ヘルパー（実装は省略）です。また**このラッパは同期ツール用**なので、`async def` のツールには非同期版（`async def wrapper(...): result = _jsonable(await func(*args, **kwargs))`）が必要です。

## 沼 3. RAG が本番で「corpus 未構築」— 犯人はリージョン不一致

議題詳細ページの「国会の関連議題（RAG semantic search）」が、本番で「**corpus 未構築**」と表示されました。corpus はちゃんと作ってあるのにです。

（RAG は、あらかじめ用意した文書集合＝ corpus から質問に近い文書を検索して、回答の根拠に使う仕組みです。Citify では国会議事録を corpus にしています。）

![図6: 議題詳細の国会 RAG。Vertex AI RAG Engine が国会議事録から関連論戦を検索し原典リンクを併記](https://static.zenn.studio/user-upload/deployed-images/c30f4542ec9640c5a4eb7d2f.png?sha=ff7ef095016f1f34b28f8d8a7f82b237db048358)

調べると、原因は**リージョン不一致**でした。

* corpus の実体は **`asia-northeast1`** に構築済み
* しかし API の `RAG_LOCATION`（Citify 側の設定変数）の既定が **`us-central1`**（corpus が 0 件の別リージョン）

デプロイ環境に `RAG_LOCATION` を明示していなかったため、起動時の display-name lookup が「別リージョンを見て 0 件 → 未構築」と判定していました。既定を `asia-northeast1`（BigQuery や Cloud Run と同じリージョン）に直して解決しました。

確認は Vertex の REST（`:retrieveContexts`）で corpus が実際にチャンクを返すことを見てから、コードを直しました。

## 沼 4. デモ動画のナレ音声（ローカル TTS）のモデル選定が沼だった

締切直前、デモ動画を **Remotion（React コンポーネントで動画を組み立てるライブラリ）+ ローカル TTS（Voicebox）** で作ることにしました。ナレーションはローカル完結の TTS で生成する方針です。

ここでモデル選定が沼でした。

* 軽量でプリセット音声を持つ **Kokoro** を試す → **日本語が動かない**。日本語 G2P（文字を発音記号に変換する処理）に MeCab 系の形態素解析（fugashi / UniDic 辞書）が要るのに、アプリ同梱に辞書が無く初期化に失敗（`Failed initializing MeCab`）
* Qwen TTS の Base モデルは**クローン専用**（参照音声が必須）で、「自分の声を使いたくない」要件に合わない

最終的に、**Qwen CustomVoice のプリセット日本語話者**（参照音声なし・MeCab 非依存）に落ち着きました。これでクローン用の録音なしに日本語ナレを生成できました。

（余談：Windows 側でモデル DL が `[Errno 22] Invalid argument` で失敗しましたが、これはアプリ内蔵ダウンローダの既知の問題で、`huggingface-cli` での手動 DL でキャッシュに置いて回避しました）

## 沼 5. 尺合わせのスロー再生で黒画面になり、Freeze で回避した

録画がナレより短いシーンで、尺を合わせようと `OffthreadVideo`（Remotion で動画を1フレームずつ描画するコンポーネント）の `playbackRate` を 1 未満（スロー）にしたら、そのシーンが**黒画面**になりました。

ただし念のため補足すると、[公式ドキュメント](https://www.remotion.dev/docs/offthreadvideo)では OffthreadVideo のスロー再生（`playbackRate < 1`）は**サポートされています**（明示的に非対応とされているのは「逆再生」のみで、黒画面の記述もありません）。なので、これはおそらく**私の使い方に固有の問題**（`startFrom`/`endAt` でのトリミングと `playbackRate < 1` を併用した組み合わせ、もしくはバージョン依存）です。

締切が近かったので原因を深追いせず、スローは使わずに「**動画を等速で流し切って、最後のフレームを `<Freeze>`（子要素を指定フレームで固定する Remotion のコンポーネント）で保持する**」方式（holdLastFrame と命名）に切り替えました。

```
// 動画尺 < シーン尺 のとき、末尾フレームを Freeze で埋める（抜粋。holdGap/lastSrcFrame 等は事前に算出した変数）
{holdGap > 0 && (
  <Sequence from={videoEndFrame} durationInFrames={holdFrames}>
    <Freeze frame={lastSrcFrame}>
      <OffthreadVideo src={staticFile(src)} />
    </Freeze>
  </Sequence>
)}
```

結果的に、締めのナレ（payoff）が結論画面の上に乗る形になり、**むしろ良くなりました**。

![図7: holdLastFrame の考え方（概念図）。録画を等速で流し切り、末尾フレームを Freeze で保持してシーン尺を埋める](https://static.zenn.studio/user-upload/deployed-images/1f3cf93bb1005c8f893e518f.png?sha=50a598321cc0212e0fd1df1cb6a11923e26c707f)

## 沼 6. 制限環境で BQ / Vertex クライアントが動かない → REST で回避

開発を進めた環境（ローカルの制限されたサンドボックス）では、`grpc`（Google の各種 SDK が内部で使う RPC フレームワーク）が必要とするルート証明書（`roots.pem`）を読めず、BigQuery / Vertex の Python クライアントがハング／クラッシュしました。

回避策として、**grpc SDK を使わず REST で叩く**ことにしました。

* ADC（Application Default Credentials）を OAuth2 のトークンエンドポイントで手動リフレッシュしてアクセストークンを取得（この手順は `gcloud auth application-default login` によるユーザー認証の ADC 前提。Cloud Run 等でアタッチしたサービスアカウントはメタデータサーバ経由なので別手順）
* BigQuery は `jobs.query`、Vertex RAG は `ragCorpora` / `:retrieveContexts` を `urllib` で直接呼ぶ

```
# grpc を避け、トークンを手動リフレッシュして REST を叩く（要点）
token = refresh_adc_token()  # oauth2.googleapis.com/token に refresh_token を POST
req = urllib.request.Request(
    f"https://{loc}-aiplatform.googleapis.com/v1/projects/{proj}/locations/{loc}:retrieveContexts",
    data=body,
    headers={"Authorization": f"Bearer {token}"},
)
resp = urllib.request.urlopen(req)  # grpc を通らず HTTPS で完結
```

## 沼 7. 空の `package.json` で esbuild（Remotion）がビルドできない

Remotion プロジェクトをビルドしようとすると、esbuild（Remotion が内部で使う JS/TS バンドラ）が親ディレクトリの `package.json` を辿って落ちました。

```
✘ [ERROR] Unexpected end of file in JSON
    ../package.json:1:0
```

犯人は、モノレポのルートに紛れていた **空（0 バイト）の `package.json`** でした。esbuild は Node のモジュール解決の一環で親ディレクトリの `package.json` を辿るため、空ファイル（不正 JSON）に当たるとそこで停止します（[esbuild#4048](https://github.com/evanw/esbuild/issues/4048) で既知の挙動）。中身を `{}`（有効な JSON）にして解決しました。

## 全体を通しての学び

7 つの沼を並べると、共通するものが見えてきました。

* **「動く」と「本番で破綻しない」は別物。** 特に自律エージェントは、完全自律の見栄えより「判断と実行の役割分担」の設計が効きました（沼 1）。
* **締切直前の新規ツール導入（TTS・動画）は、モデル/環境依存の沼が多い。** ビルド・ダウンロード・G2P・リージョンといった「実現性」を先に潰すほど、あとが楽でした（沼 3・4・5・7）。
* **制限環境では REST が効く。** SDK が理由不明で動かない時の逃げ道になりました（沼 6）。

これから ADK / Vertex AI RAG / Remotion に触るなら、まず次を確認しておくと沼が減ります。

* ADK のツール戻り値は plain dict / JSON-able に寄せる（沼 2）
* Vertex AI RAG は **corpus のリージョン**と、アプリが参照するリージョンを一致させる（沼 3）
* ローカル TTS は「日本語 G2P の依存（MeCab 等）」と「プリセット / クローン」を先に確認（沼 4）

個人開発 × 締切ドリブンだと、こういう「小さな沼」の積み重ねが一番時間を溶かします。同じところで止まる人が減れば嬉しいです。
