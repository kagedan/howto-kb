## 概要

今回の `AI News` パートも、「最強モデル探し」より「モデルをどう使い分けるか」が重要になってきた、という話題が中心でした。表向きの主役は `Claude Sonnet 5` と `Fable 5` ですが、ホストたちが繰り返し触れているのは、単一モデルに賭ける設計の限界、計画役と実行役を分ける発想、オープンモデルを実務に組み込む現実性、そしてAI基盤の主導権をどこまで自社側に残せるか、という論点でした。

この空気感をよく表しているのが、`Sonnet 5` への反応です。新モデルの発表なのに、盛り上がり方が「すごいモデルが出た」ではなく、「本当に Opus の代わりになるのか」「実運用で安いのか」「重い仕事では結局トップモデルに戻るのではないか」という、かなり冷めた見方が中心です。これはモデル市場が成熟してきた証拠でもあります。派手な発表より、日々の開発と運用で何が得になるのかが問われる段階に入っているからです。

私個人としては、そもそも Opus 4.8 なんて高くて使っていられなくて、Sonnet 4.6 よりも優秀で安価な Sonnet 5 を歓迎しているのですが、貧乏性なのでしょうか(笑)

## 要点

1. **Sonnet 5 は“新しい主役”というより“用途限定の選択肢”として見られていた**: 軽い作業なら使えるが、高難度のコーディングや推論では Opus 4.8 を置き換えるほどではない、というのが番組のトーンでした。
2. **Fable 5 の再開は朗報である一方、制限とフォールバックが信頼を削っている**: コーディング時に別モデルへ透過的に切り替わる挙動は、開発者にとってかなり分かりにくい体験です。
3. **複数モデルを組み合わせる設計は、もはや一部の玄人芸ではない**: 計画、実装、判定を別モデルに振る `model routing` は、エージェント設計の標準パターンに近づいています。
4. **GLM 5.2 や DeepSeek は“安い代替品”ではなく、実運用で役割を持てるモデルになってきた**: とくに実装担当、観測担当、メモリ系など、役割を絞るとかなり現実味があります。
5. **コーディングエージェントの質はメモリ設計に大きく左右される**: `observational memory` 系の話題からは、モデル本体以外の層が差別化要因になっていることがよく分かります。
6. **AI に全部任せれば開発が楽になるわけではない**: コードを理解していない人が AI で量産したPRは、むしろレビュー負荷を増やします。
7. **トークン課金とAI主権の議論は、単なる思想論ではなく設計論になってきた**: どのモデルに依存するか、どこに予算を払い、どこにデータを送るかは、今後ますます重要になります。
8. **推論ハードウェアや独自チップの話題は、モデル競争が供給競争でもあることを示している**: AI企業はソフトウェアだけでなく、供給の握り方でも差をつけようとしています。

## 詳細

### Sonnet 5 が見せたのは、モデル発表そのものの“相対化”

![](https://static.zenn.studio/user-upload/6916f68fea75-20260707.png)

この回の冒頭で扱われる `Claude Sonnet 5` は、本来なら番組の主役になっていてもおかしくありません。Anthropic が「よりエージェント向きで、以前はより大きく高価なモデルが必要だった仕事をこなせる」と打ち出したからです。けれども、ホストたちの受け止め方はかなり醒めています。ベンチマーク図の見せ方に対しても、「本当に Opus より使う理由があるのか」が先に立っていました。

![](https://static.zenn.studio/user-upload/5b6c7d912437-20260707.png)

ここで重要なのは、彼らが Sonnet 5 を完全に否定しているわけではないことです。軽いタスク、メール整理、簡単なツール利用、日常的なチャットボット用途のように、「そこまで高い知能はいらないが、それなりに扱いやすいモデルが欲しい」場面なら成立しうると見ています。ただし、重いコーディングや高い推論能力を要求する仕事になると、結局 `Opus 4.8` に戻るのではないかというわけです。

この感覚は、AI市場が少し成熟してきたことの表れでもあります。2024年や2025年の空気なら、「新モデルが出た」というだけでもっと盛り上がったはずです。ところが今は、「そのモデルは何に向くのか」「既存モデルから本当に置き換えが起きるのか」「トークン単価ではなくジョブ単価で見たらどうか」という現実的な比較に、関心が移っています。発表イベントより運用実感が重くなってきた、と言っていいでしょう。

### Fable 5 の再開は、“能力”より“体験”の問題をあぶり出した

`Fable 5` が再び使えるようになった話も、この回の大きな軸です。ただし、ここで焦点になっているのは性能そのものではありません。むしろ「高性能なはずのモデルが、いざ一番使いたいタスクでは別モデルにフォールバックする」という体験のわかりにくさに、ホストたちは強い違和感を示しています。

![](https://static.zenn.studio/user-upload/320d2458dfa0-20260707.png)

番組内の整理では、Fable 5 は計画立案や長いループにおいてはまだ魅力があるかもしれません。しかし、コーディングやデバッグのような日常的な開発作業で Opus 4.8 に切り替わるなら、開発者にとっては「見えている名前」と「実際に動いている中身」がズレます。このズレはかなり厄介です。モデル選定の判断、費用感の把握、性能評価の蓄積、その全部がやや曖昧になるからです。

![](https://static.zenn.studio/user-upload/0bc696fe713c-20260707.png)

ここで出てくるのが信頼の問題です。容量制限、サブスクの扱い、ロールアウト時の説明不足、UI 上の表示とのズレ。そうしたものが積み重なると、たとえモデルそのものが優秀でも、「本番で安心して使えるか」という評価は別物になります。実際、AI開発の現場では“最高性能”より“挙動が読めること”のほうが大事な場面が少なくありません。Fable をめぐるこの混乱は、その事実をかなりよく示していました。

### いま本当に進んでいるのは、単一モデル競争ではなく `model routing`

`Devon Fusion` の話をきっかけに、番組は `model routing` を「一部の上級者だけがやるテクニック」ではなく、普通の設計に近いものとして語ります。Cognition はこれを `35% cost savings` を狙える `hybrid model harness` として出していましたが、ホストたちの関心は製品名そのものより、「session の途中でモデルを切り替えながら frontier 級の性能を保つ」という発想のほうに向いていました。

![](https://static.zenn.studio/user-upload/d1d6a7ba81ed-20260707.png)

「多くの開発者は、もう手動で routing している」とかなりはっきり言っています。あるモデルで計画を立て、別のモデルで実装し、必要なら最後に別モデルを judge に回す。名前を付けずにやっているだけで、実態としてはすでに multi-model な設計に入っている、という整理です。ホストの一人は、Fable は planner や architect として使い、コード生成は GPT や Opus に任せ、最後の完了判定だけ Fable に戻すような流れまでその場で例示していました。

![](https://static.zenn.studio/user-upload/b2fab367468f-20260707.png)

これは、エージェント設計の重心が変わっていることを意味します。以前は「どのモデルが一番賢いか」が中心でした。今は「どのモデルを、どの段階で、どの条件で使うか」が重要になりつつあります。この違いは大きいです。前者はモデル選定の問題ですが、後者はシステム設計の問題だからです。

![](https://static.zenn.studio/user-upload/5d0f54c06de4-20260707.png)

しかも、この節では `Devon` 自体への評価がやや冷めているのも重要です。ホストたちは「一発で終わる単純な仕事ならまだ送れる」「ジュニアエンジニアに振る雑務みたいな仕事なら使える」としつつ、多段の推論や重要な経路に乗る仕事は、むしろ自前の coding agent でやったほうがよいという温度感でした。以前は Devon をかなり好意的に見ていたのに、いまは deep research 的な用途でも期待がしぼんでいる、という率直な反省まで出ています。

だからこそ、この変化はコスト最適化だけの話ではありません。速さ、安定性、供給制約、課金、ガードレール、ツール利用の得意不得意まで含めて設計できるようになります。個別の router 製品が勝つかどうかより、routing という設計思想そのものが標準化していく。その意味で、このパートは「最強モデル探し」から「役割分担の設計」への重心移動を、かなりはっきり言語化していた場面でした。

### GLM 5.2 と DeepSeek が示す、オープンモデルの役割変化

このブロックで出てきたニュースは、実はそれぞれ少し性質が違います。`Zcode` は `GLM 5.2` 向けの公式開発環境が出たというツール側の話で、BYOK に対応し、Mac/Windows/Linux で使えることが紹介されました。

![](https://static.zenn.studio/user-upload/995d268fab8e-20260707.png)

`GLM 5.2` 自体については、その上で本当にコーディング実務に使えるのかという評価の話が続きます。`DeepSeek V4` は「いま見えているものは preview で、近いうちに本格版が来るらしい」「その代わり価格は少し上がりそうだ」というロードマップ寄りの話でした。Tencent については、新しい `295B` 規模の mixture-of-experts モデルが出た、という新規リリースの話です。つまりこの節は、単に“中国系オープンモデルが増えている”という雑なまとめではなく、開発環境、現場評価、次期版の予告、新モデル公開という別々のニュースが一気に並んでいた場面でした。

![](https://static.zenn.studio/user-upload/dd5ad3436b8d-20260707.png)

ここでのホストたちのテンションは、かなり具体的でした。`Zcode` が `GLM 5.2` 向けの公式開発環境として出た話題では、「見た目や体験はほぼ Codex っぽい」「かなり借りてきている感じはある」と軽口を叩きつつも、肝心のモデル評価は真面目です。`GLM 5.2 is the truth` とまで言っていて、少なくとも Mastra 側ではコーディング作業のかなり大きな部分をすでに任せられるモデルとして見ています。

![](https://static.zenn.studio/user-upload/de25e12f735b-20260707.png)

ただし、ここでも主語は「最強かどうか」ではありません。彼らが繰り返していたのは、「計画を立てるモデル」と「その計画を実装するモデル」は分けてよい、という前提です。`GLM 5.2` はまさに後者、つまり planner が描いた方針を受け取って実装を進める execution モデルとして高く評価されていました。Alibaba 側の `coding plan` なら月100ドル規模で使えること、`Z.ai` 側はやや遅いが今後改善しそうなこと、米国内なら Fireworks のような提供形態もあることまで話していて、かなり実務寄りの会話になっています。

![](https://static.zenn.studio/user-upload/b074938eba33-20260707.png)

`DeepSeek` についても、単なる「もうすぐ来る強いモデル」の話ではありません。ホストたちは `DeepSeek V4 fast` を社内でかなり使っていると明言し、とくに `observational memory` のような observer 役、つまりメモリの観測や整理、分析の補助に向いていると話していました。ここでも重要なのは、万能モデルとして持ち上げているのではなく、「観測役ならこれがハマる」と、かなり役割ベースで評価していることです。

![](https://static.zenn.studio/user-upload/3703b6efaed8-20260707.png)

Tencent の新しい mixture-of-experts モデルに対する反応も、その延長線上にあります。ベンチマーク表だけを見れば Anthropic や Z.ai を大きく上回るわけではないが、「他のオープンモデル群の中で十分競争力がある」「サイズのわりにかなり強い」という見方で、実務に入れられる選択肢が着実に増えているという空気でした。オープンモデルは frontier の完全代替ではなく、役割ごとに差し込める有力部品になってきた。ホストたちの会話は、まさにその変化をかなり生々しく示していました。

### コーディングエージェントの差は、モデルのIQよりメモリと人間介入で決まる

後半で印象的だったのが、`observational memory` まわりの話です。メモリ層を追加することで難しいタスクの成功率がかなり上がった、という報告に対して、ホストたちは強く反応しています。

ここで取り上げられていたのは、Will という開発者の「自分はしばらく、誰にも言わずに coding agent に memory extension を載せて実運用していた」というスレッドでした。ホストたちはこの“告白調”を面白がりつつも、数字のほうにすぐ食いつきます。`Pi` 系の observational memory 実装で、難しいタスク群に対して `77% relative gain`、しかも memory なしのときより `5倍` 近く多く成功した、という報告が出ていたからです。

![](https://static.zenn.studio/user-upload/b1e8ee29c8d5-20260707.png)

会話の中では、単に「メモリがあると便利」では終わっていません。Mastra Code が強い理由のひとつも memory にあるのではないか、他フレームワークでも同じ傾向が再現されているのは重要だ、というかなり踏み込んだ反応でした。さらに Tyler が「履歴サイズや memory window の切り方を調整すれば、まだ伸びる余地がある」と茶々を入れていて、メモリは思想論ではなく実際にチューニング可能な設計変数として扱われています。

![](https://static.zenn.studio/user-upload/d41fd93a8cfe-20260707.png)

そこから話題は、人間がコードを理解しなくてよいという風潮への反省に移ります。この流れがかなり生々しいです。ホストたちは「1年前には、もう PR を読まなくてもよいのではないか、という空気があった」と振り返りつつ、今はむしろ逆に戻ってきたと言います。実際、オープンソース側ではコードベースを理解しない人たちが Codex などに投げて大量の PR を送ってきており、レビュー負荷が増しているとかなり率直にこぼしていました。

![](https://static.zenn.studio/user-upload/dc6167677a91-20260707.png)

とくに印象的だったのは、「先週ついに PR スパムを初めて食らった」というくだりです。15本くらい連続で、問題設定も曖昧なまま AI 製の修正案が飛んできた、とホストは半ばうんざりした様子で話します。つまり、彼らが言いたいのは「AI を使うな」ではありません。メモリを含む設計を工夫すれば agent は強くなるが、それでも人間の理解と評価責任は消えない、ということです。後半のトーンはかなり明確にそこへ向かっていました。

### `token maxing` と `AI sovereignty` は、2026年の設計キーワードかもしれない

Alex Karp の発言をきっかけにした `token maxing` と `AI sovereignty` の議論は、少し挑発的ですが、今の空気をよく表しています。ここで言われているのは、「高い frontier model を大量に使えばいい時代ではなくなってきた」ということです。

![](https://static.zenn.studio/user-upload/5051dbc2bcc4-20260707.png)

このパートは、かなりホストの地声が出ていました。発端は Palantir の Alex Karp による長い rant で、彼が言いたかったのは大ざっぱにいえば「Anthropic や OpenAI に無自覚にトークンを流しすぎるな」「その依存は製品にも将来の選択肢にもよくない」「open model をもっと真面目に使え」ということです。番組側もこの問題提起にはかなり共感していて、少なくとも半年ほど前なら大げさに聞こえた話が、2026年の今ではだいぶ現実味を帯びてきた、という温度感でした。

![](https://static.zenn.studio/user-upload/a6fe7908eb39-20260707.png)

ホストたちは、frontier モデル企業が API やアプリを通じて大量の利用データとトレースを受け取っている以上、利用者は「どの企業に何を見せるのか」「どこまで依存するのか」を考えたほうがよいと整理します。しかも彼らは、この議論を単なる思想論ではなく、プロダクト防衛の話として見ています。スタートアップがせっかく作った体験も、モデルラボがその需要を横から見て自前機能として取り込めば、一発で厳しくなる。だから governance と choice を残すために open model や router に慣れておくべきだ、というわけです。

ただし、番組は Alex Karp の主張をそのまま礼賛しているわけでもありません。たとえば「token ではなく value で課金すべきだ」という方向の話には、ホストたちも「それは理屈としては分かるが、実際にどうやって value 課金するのかは難しい」と距離を置いていました。このあたりは、煽りを煽りとして楽しみつつ、実装可能性は別で見るというバランスです。

一番実務的だったのは、サブスク価格の話です。いまの `200ドル` 級プランは API の素の価格から見ればかなり補助されている可能性があり、極端にいえば本来 `月5万ドル` 規模でもおかしくない使い方を、多くの人が安く使えているのかもしれない。だから、使えるうちは使えばよいが、それが `1000ドル` に跳ねても困らないように Plan B を持っておけ、というのがホストたちの提案でした。Frontier サブスクに加えて、open model 側にも予算や慣れを配っておくべきだという話は、かなり具体的でした。

### 独自チップ、推論速度、global workspace。競争軸はさらに増えている

最後の短いトピック群からも、いくつか見えてくるものがあります。Anthropic の独自チップ観測、Etched のような推論ハードウェア企業、超高速推論アピール。これらは「賢いモデルを作る」競争が、そのまま「安定供給と低遅延をどう握るか」の競争でもあることを示しています。

![](https://static.zenn.studio/user-upload/a2297d841a46-20260707.png)

ここも、単なるニュースの箇条書きではありませんでした。まず `agent-cn` のような one-click のエージェントテンプレート群を見て「こういう production-ready な雛形が wild に出てくるのはよい」と盛り上がり、その流れで Anthropic が Samsung と独自 AI チップを探っているという報道に移ります。OpenAI が自前チップに向かうなら Anthropic も同じことを考えるはずで、結局 frontier ラボ自身もまた `NVIDIA 依存` を減らしたいのだろう、という整理でした。

![](https://static.zenn.studio/user-upload/d99766d58f16-20260707.png)

その前段には、Etched や Cerebras の話も出ています。Etched が `8億ドル` を調達し、`10億ドル` 相当の顧客契約を抱えているという話には素直に驚いていて、「そんな会社知らなかった」という反応から、推論ハードウェアの戦場が急速に広がっていることを実感している様子でした。Cerebras による `Gemma 3 41B` の超高速推論にも反応していて、ここでは賢さそのものより `1800 tokens/sec` 級の応答速度が製品価値になる、という感覚が前面に出ています。

![](https://static.zenn.studio/user-upload/81b372ac1092-20260707.png)

細かい雑談としては、Fable のフォールバック問題を揶揄したミームにも触れていました。Ferrari として売っておきながら、実際の routine task では Honda Civic にフォールバックする、という皮肉です。番組全体の不信感がどこにあるのかを、かなり端的に表したジョークでした。

![](https://static.zenn.studio/user-upload/ead56d3149f9-20260707.png)

締めの `global workspace` 研究では、ホストたちの関心は論文用語そのものより、「モデル内部には、表から見える reasoning とは別に、もっと潜在的な作業空間があるのではないか」という発想に向いています。動画内で出てくる `JSpace` という名前にはかなり文句を言っていましたが、概念自体には面白がっていました。そして重要なのは、彼らがすぐにこれを application layer の設計へ結びつけていたことです。モデル内部には触れられなくても、見える推論、記憶、計画、実行を分けて組むという発想は、エージェント実装側でもすでに同じ方向を向いている。最後の雑談めいた数分も、実はかなり示唆的でした。
