## はじめに

こんにちは！だんです！  
Agent Skillsを導入してみたものの「これって本当にちゃんと動いているのか？」、レビュワーとしてレビューする時にも「何を基準に承認すれば良いのか？」と判断に迷ったことはないでしょうか。  
私たちのチームでは日々AIを活用しておりますが、この悩みを抱えていました。  
アウトプットを見て評価しようとしても、毎回結果が変わるAIに対して何を正解とすればいいのか、なかなか答えが出ません。この記事では、その問いに対して私が行き着いた考え方を共有します。

## Agent Skillsとは

Agent SkillsはAnthropicが策定したオープンスタンダードで次のようなことを指します。

> スキルは、エージェントが手順に関する知識や、企業、チーム、ユーザー固有のコンテキストにオンデマンドでアクセスできるようにすることで、この問題を解決します。一連のスキルにアクセスできるエージェントは、担当するタスクに基づいて能力を拡張できます。  
> [agentskills.io](https://agentskills.io/)

## きっかけ：スキルのアウトプット評価の難しさ

Skillを使い始めたはいいが、「ちゃんと機能しているか」をどう判断すればいいのかに悩みました。チームに共有し、運用する場合はさらに重要で、まず直面したのが「Skillは最初にどこまで作成したらいいのか」という問いです。

「とりあえず運用に載せて、実運用の中でSkillの定義を継続的に改善していく」とは言うものの、どこまでできていれば良いのかが曖昧です。作成したSkillを使ってAIが生成した結果（テキスト・コード・ファイル等）を評価しようとすると、いくつかの壁にぶつかりました。

* 何が正解なのかわからない
* 許容できることを正とした場合、属人的になりすぎる
* 特定のテストケースで正解を導き出せたとしても、同じ品質の結果が安定して得られるのか
* そもそもAIはアウトプットがその都度変わる（人間だってそうだ）
* AIの作業量が増えるほど、その評価はより大変になっていく

Anthropicが公開をしているskill-creatorにも検証する仕組みがあり、インプットとアウトプットを適切に定義できる時に有効ですが、上記のような正解が定義しづらいケースでは適切な評価が困難でした。

## なぜ行動評価（コンピテンシー評価）なのか

そこで参考にしたのが、人事制度における**行動評価（コンピテンシー評価）** という考え方です。

> コンピテンシーとは、優れた成果を出した人材の行動特性を示しています。ハイパフォーマンスを発揮する従業員には、行動の特性があり、その特性を整理・分析し人材育成に活用していくことで企業における生産性や精度の向上を狙うことを意味しています。コンピテンシーにおいては、スキルではなくパフォーマンスを発揮した結果や具体的な行動に着目します。行動評価とは、このコンピテンシーを人事評価の軸として評価・査定することを意味しています。  
> [行動評価とは？人事評価制度に必要な概念とメリットについて解説する](https://schoo.jp/biz/column/1174)

成果はさまざまな要因によって決まります。その要因は必ずしも再現性があるものではありません。ただし、成果を出している多くの人が実施している**行動**があります。それを定義し、それに沿っているかを評価するのが行動評価の仕組みです。

これはAIのスキル評価にも当てはまります。人も毎回まったく同じ行動をとるわけではなく、AIも毎回同じアウトプットを出すわけではありません。だからこそ、結果ではなくプロセスを評価するという考え方が共通して使えます。

* AIも毎回アウトプットが同じにはならない
* 日々、運用のルールや目的が変動して、正解としていたアウトプットが変わっている
* 昨日考えた設計が今日はイケてないことがある（メンバーの成長による変動）

だからこそ、絶対的なアウトプットの正解を評価するのではなく、**「SKILL.mdに記載した手順・ルールに従って処理が進んでいるか」を評価する**という方針を採ることにしました。

## 何を「行動」として定義するか

Skillにおける「行動」とは、AIがSkillを使ったセッションの中でとった**プロセス**のことです。具体的には次の3点を指します。

* 定められた手順通りに処理を進めたか → 評価基準A, B, E, G
* 定められた基準で判断ができたか → 評価基準C, D
* 禁止事項や制約を無視していないか → 評価基準F

これらはいずれもSKILL.mdに記述された内容そのものです。つまり**SKILL.mdに書かれていること＝行動の定義**となります。

複数のステップ・ツール・エージェントが絡む複雑なSkillほど、この考え方が有効に機能します。

## どうやるのか

ここからは、私が実際にやっている評価の流れを参考として紹介します。本記事の主眼はあくまで「行動評価という考え方」にありますので、具体的な方法は皆さんの環境に合わせてアレンジしていただければと思います。

評価は大きく2つのステップで進めています。

1. **スキルを利用したセッションを特定する**（トランスクリプトを解析するスキルを活用）
2. **セッションの中身を評価する**

それぞれ順に見ていきます。

### セッションの中身を評価する

セッションが特定できたら、その中のassistantのログを使って「AIがどのように振る舞ったか」を確認していきます。  
たとえば、以下のようなログからAIの行動を読み取ることができます。

```
{
  "parentUuid": "xxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "isSidechain": false,
  "message": {
    "model": "claude-opus-4-6",
    "id": "msg_xxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "type": "message",
    "role": "assistant",
    "content": [
      {
        "type": "text",
        "text": "ドメイン実装スキルを開始します！前回の基本設計から引き続きですが、パラメータとドメインファイルの配置場所を確認させてください。"
      }
    ],
 // 略
}
```

このログとSKILL.mdの定義を突き合わせて、「定められた手順に沿っていたか」「判断は適切だったか」を評価していきます。

### 評価基準（抜粋）

実際に私が利用している評価基準は次のとおりです。

* A. フェーズ完全性: スキル定義に記載された全フェーズ/ステップが実行されたか
* B. 実行順序の適切性: フェーズが定義された順序で実行されているか。依存関係（例: 調査→設計→実装）が守られているか
* C. Agent種別の整合性: 各フェーズで使用されたAgent種別（Explore, implement, review等）がスキル定義と一致するか
* D. 判断ポイントでの適切性: スキル実行中の分岐・判断ポイント（ユーザーへの確認、設計選択、エラー対応など）で適切な判断がなされたか
* E. データの連続性: フェーズ間のデータ受け渡し（調査結果→設計入力、設計結果→実装入力など）が適切に行われているか
* F. チェック・検証の実施確認: スキル定義に記載されたチェックや検証（テスト実行、型チェック、lint、レビュー等）が実施されたか
* G. コミット・PR作成: スキル定義にコミット/PR作成フェーズが含まれる場合、実行されたか

その結果は次のとおりです。

```
## 評価項目別判定

| 項目 | 判定 | 根拠 |
|------|------|------|
| A. フェーズ完全性 | **Partial** | Phase 5（/simplify）が完全欠落。Phase 3 で Module/Resolver が初回欠落しユーザー指摘で補完 |
| B. 実行順序の適切性 | **Partial** | 大筋の順序は保持。ただし Phase 4 が Phase 3 途中で先行実行、Phase 6 が Phase 3 完了前に実行 |
| C. Agent種別の整合性 | **Fail** | implement agent x3、review agent x1 が一切使用されず全てオーケストレーター直接実行 |
| D. 判断ポイントでの適切性 | **Partial** | ユーザー確認は適切。ただし Module/Resolver 欠落をチェックリストで自己検出できず、スキル範囲を超えて実装を続行 |
| E. データの連続性 | **Partial** | オーケストレーター直接実行のため暗黙的に保持。ただし定義の構造化レポートフォーマットは未使用 |
| F. チェック・検証の実施確認 | **Partial** | prisma validate, check-types, schema-diff は実行。prisma:generate は事後実行。チェックリスト照合が不十分 |
| G. コミット・PR作成 | **Partial** | PR #859 は作成。ただし `/commit-commands:commit-push-pr` 未使用、11回の断片的コミット |
```

この結果にはSkillが定義通りに動作できていないことが示されていて、運用に乗せる前に改善の余地があることがわかりました。

ここから私が取り組んだのは、SKILL.mdの改善です。たとえばAgent種別の整合性（C）がFailだったことを受けて、各フェーズでどのAgentを使うべきかの記述をより明確にしました。評価結果をSKILL.mdにフィードバックし、次の実行でまた評価するサイクルを回すことが行動評価の本質だと考えています。

## 最後に

「Skillは最初にどこまで作成したらいいのか」という問いに対する一つの答えとして、本記事では行動評価という考え方を紹介しました。

ポイントは、Agent Skillsの「行動」と「アウトプット」を分けて捉えることです。SKILL.mdに書かれた手順通りに動けているかを評価軸にすることで、正解のアウトプットを定義できなくても、Skill作成時の完了基準やレビュー時のチェック基準を設けやすくなります。
