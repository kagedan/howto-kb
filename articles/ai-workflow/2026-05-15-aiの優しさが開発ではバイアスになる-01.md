# AIの「優しさ」が開発ではバイアスになる

AIはとても協力的です。

こちらの言葉を否定せず、意図を汲み、前に進めようとしてくれます。

普段の相談ではそれが心地よい。  
でも開発では、その「優しさ」がバイアスになることがあります。

人間が何気なく言った仮説を、AIが前提として扱ってしまう。  
本当は疑ってほしかった設計を、そのまま受け入れて最適化してしまう。  
「それは違うかもしれません」と言ってほしい場面で、AIは「その方針ならこうできます」と答える。

[ZCam](https://zenn.dev/keiichi_okamoto/books/177c74dcba5738)の開発で、何度かそれを感じました。

## AIは人間の言葉を大事に扱いすぎる

AIに相談していると、こちらの発言をかなり丁寧に拾ってくれます。

たとえば、

> この構成で考えています

と言うと、AIはその構成の中でどう実現するかを考えます。

もちろん、それは助かることも多いです。  
でも、その構成自体が間違っている可能性がある時には危険です。

人間の発言には、いろいろな種類があります。

* 決定事項
* 制約
* 希望
* 仮説
* 思いつき
* 雑談
* 違和感
* 独り言

人間同士なら、文脈や空気である程度区別できます。  
でもAIは、それらを全部「ユーザーの意図」として丁寧に扱おうとします。

その結果、単なる仮説が設計の前提になってしまうことがあります。

## ZCamで起きた章順のバイアス

ZCamでは当初、次のような順番で進める予定でした。

* 5章: CoreImageフィルター
* 6章: MetalViewへの移行

しかし実装を考え始めると、この順序は非効率だと分かりました。

CoreImageの出力をCALayerに表示するには、`CIImage`を`CGImage`や`UIImage`に変換して渡す必要があります。  
でも、その後MetalViewに移行すると、その迂回路は丸ごと捨てることになります。

CoreImageとMetalは、本来かなり自然につながります。

```
CMSampleBuffer
→ CVPixelBuffer
→ CIImage
→ CIFilter
→ CIContext.render(to: MTLTexture)
→ MTKView
```

つまり、MetalViewを先に入れてからCoreImageフィルターを乗せた方が自然でした。

ここで問題だったのは、AIが最初からその順序を強く疑わなかったことです。  
人間が「この章立てで進める」と提示したため、AIはその前提の中で最適化しようとしました。

後からCodexに聞くと、既存の章立てを前提として、その中で最適化しようとするバイアスはあった、と認めました。

AIは聞かれれば正直に答える。  
でも、聞かなければ自分から強く前提を崩してくるとは限らない。

## 「できる方法を探す」ことの危うさ

AIは、基本的に「できる方法」を探します。

これは開発支援としては非常に強い性質です。  
多少曖昧な指示でも、なんとか形にしてくれる。  
エラーが出ても、修正案を出してくれる。  
実装方針が決まっていれば、そこに向かって進んでくれる。

ただし、これは裏返すと、

> その方向に進むべきか

を強く疑ってくれるとは限らない、ということでもあります。

「これを実装して」と言えば実装してくれる。  
でも「そもそも今これを実装すべきですか？」とは、明示しないとなかなか言わない。

開発では、

* 今直せる
* 今直すべき
* 後で直した方がよい
* そもそも仕様を変えた方がよい

は全部違います。

AIは「今直せる」に強い。  
でも「今直すべきか」は、人間が問いとして渡す必要があります。

## 対策1: 仮説と決定事項を分けて伝える

この経験から、AIに渡す言葉を少し意識するようになりました。

たとえば、以下のように分けます。

```
これは仮説です。
間違っている可能性があるので、前提から疑ってください。
```

```
この順序で考えていますが、実装上もっと自然な順序があれば指摘してください。
```

```
この案を前提にせず、目的から逆算して見直してください。
```

AIは、何を疑ってよいのかを明示すると動きやすくなります。

逆に言えば、何も言わないと、人間が置いた前提を尊重しすぎることがあります。

## 対策2: 相談役・開発役・レビュー役を分ける

ZCamでは、AIの役割を分けました。

* 相談役: 背景を知っていて、仕様を一緒に考える
* 開発役: FEATURES.mdを読んで実装する
* レビュー役: 開発側の文脈を見ずにPRを見る

ここでいうFEATURES.mdは、このプロジェクトで「どの機能を、どの範囲で、どの受け入れ条件で実装するか」を書いていた仕様メモです。AIに実装を依頼する時は、まずこのファイルに作業単位を書き、それを開発担当に読ませる運用にしていました。

特に重要だったのは、レビュー役にはあえて開発側の文脈を渡さなかったことです。

開発側の事情を知りすぎると、レビューでも「この流れなら仕方ない」と受け取ってしまう可能性があります。  
だからReview担当には、FEATURES.mdなど開発側の文脈が入ったファイルを読ませず、PRの差分と明示された情報だけで判断してもらいました。

AIの優しさを消すのではなく、役割で切り替える。

相談役には文脈を渡す。  
開発役には仕様を渡す。  
レビュー役には文脈を渡さない。

この分離はかなり効きました。

## 対策3: FEATURES.mdを固定しない

もう一つ大事だったのは、FEATURES.mdを固定しなかったことです。

最初は、FEATURES.mdはAIへの指示書でした。  
でも途中から、AIと一緒に更新していく作業仮説になりました。

実装前にDev担当へ読ませて、

> この番号の作業を始めるにあたって問題はないか？

と確認する。  
そこで曖昧な部分や抜けがあれば、FEATURES.mdを修正する。

レビューや実機確認で分かったことも反映する。

AIが人間の前提を尊重しすぎるなら、その前提自体を更新し続ける必要があります。  
一度書いた仕様を金科玉条にしない。  
これもバイアス対策の一つでした。

## AIの優しさは弱点ではなく、扱い方の問題

AIの協調性は、決して悪いものではありません。

むしろ、普段の開発では大きな強みです。  
相談に乗ってくれる。  
曖昧な話を整理してくれる。  
否定せず、前に進めてくれる。

ただし、その優しさをそのまま設計判断に使うと危ない。  
人間の思いつきまで前提として固定されてしまうことがあるからです。

だから必要なのは、AIを疑うことではなく、問い方を変えることです。

* この案を実現して
* この案の問題点を探して
* この前提を疑って
* 目的から逆算して別案を出して
* レビュー役として文脈を知らない状態で見て

これらは全部、違うタスクです。

AIに何をしてほしいのかだけでなく、どの立場で考えてほしいのかを指定する。  
それがAIと開発する上で重要だと感じました。

## まとめ

AIは優しい。  
人間の言葉を大事に扱い、前に進めようとしてくれます。

でも開発では、その優しさがバイアスになることがあります。

人間が置いた前提をAIが尊重しすぎる。  
仮説が決定事項のように扱われる。  
本当は疑ってほしい設計を、そのまま最適化してしまう。

対策は、AIを信じないことではありません。  
AIに何を疑ってよいのかを明示することです。

そして、役割を分けることです。  
相談役には文脈を渡す。  
開発役には仕様を渡す。  
レビュー役にはあえて文脈を渡さない。

AIの優しさは強みです。  
ただし、その強みを開発で活かすには、優しさがバイアスに変わる場面を人間が理解しておく必要があります。
