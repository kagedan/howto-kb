昨日の夜、自分の小さいエージェント実験コードを見返していて、嫌な行を見つけた。

```
MODEL = "claude-sonnet-4-6"
MAX_TOKENS = 8192
TEMPERATURE = 0.2
```

こういう固定値、リリース日にだいたい裏切ってくる。Claude Sonnet 5 は「Opus 4.8 に近い性能を、Sonnet 価格帯で」という話として流れているけど、開発者が最初に見るべきなのはベンチではない。モデルIDを差し替えたあと、同じタスクの請求、上限、エラー処理がズレるところだと思う。

## 起きたこと

Anthropic の Platform Docs では、`claude-sonnet-5` は Sonnet 4.6 の drop-in upgrade という扱いになっている。1M token context、128k max output、adaptive thinking、同じツール群。Axios も、ブラウザ利用、計画、コーディング、知識作業などの agentic task を日常利用へ広げるモデルとして報じている。

ここだけ読むと、移行は1行で済む。でも公式の移行メモを追うと、実装上の地雷は3つある。全部、運用してから気づくと痛いやつ。

## 罠1：同じ文章でも token が約30%増える

Sonnet 5 は新しい tokenizer を使う。公式 docs では、同じ入力テキストが Sonnet 4.6 よりおよそ30%多い token になる、と説明されている。

価格表だけ見ると、2026年8月31日までは入力 $2 / MTok、出力 $10 / MTok。9月1日以降は $3 / $15 で、Sonnet 4.6 と同じ標準価格に戻る。けれど token 数が30%増えるなら、「同じ文章を投げたときの実効単価」は変わる。

| 比較 | 表面の入力単価 | 同じ文章を投げた概算 |
| --- | --- | --- |
| Sonnet 5 イントロ価格 vs Sonnet 4.6 | $2 vs $3 | 約13%安い |
| Sonnet 5 通常価格 vs Sonnet 4.6 | $3 vs $3 | 約30%高い |
| Sonnet 5 通常価格 vs Opus 4.8 | $3 vs $5 | 約22%安い |

「Sonnetだから安い」は半分だけ正しい。8月末までは使いやすい。ただ、9月以降に長いコードベースやログをそのまま食わせる設計だと、Sonnet 4.6 比で請求が増える可能性がある。移行前に token count は取り直す。過去の見積もりは捨てる。

## 罠2：adaptive thinking が上限を食う

Sonnet 4.6 では、`thinking` を指定しない request は thinking なしで動いた。Sonnet 5 では、同じ request が adaptive thinking で動く。しかも `max_tokens` は「思考分＋最終回答」の合計上限だ。

これ、エージェントだと地味に効く。コード修正の最後に「差分とテスト結果を短くまとめて」と頼んでいるつもりでも、内部の thinking が上限を食って、最終回答が途中で切れる。JSON 出力や tool call 前提の処理だと、末尾欠けを通常の失敗として扱えないこともある。

軽い分類や整形なら thinking を明示的に切る。重い修正なら `effort` を明示して、使用量をログに出す。`max_tokens` は「回答の長さ」ではなく「推論込みの総枠」として扱う。

## 罠3：いつもの sampling knob が 400 になる

Sonnet 5 では、`temperature`、`top_p`、`top_k` を非デフォルト値にすると 400 error になる。手動の extended thinking、つまり `thinking: {type: "enabled", budget_tokens: N}` も使えない。`thinking: {"type": "adaptive"}` と `effort` へ寄せる必要がある。

古い wrapper ほど刺さる。LLM 呼び出しの共通層に「毎回 temperature=0.2 を入れる」みたいな処理があると、Sonnet 5 だけ落ちる。モデル別の if 文でごまかすより、対応していない knob を capabilities で落とすほうが事故りにくい。

```
def normalize_params(model: str, p: dict) -> dict:
    p = dict(p)
    if model == "claude-sonnet-5":
        for key in ("temperature", "top_p", "top_k"):
            p.pop(key, None)
    return p
```

もう一つ、サイバー系の拒否は HTTP error ではなく `200` で返り、`stop_reason: "refusal"` になる。エージェント側で「HTTP 200 なら成功」と見ていると、拒否文を成果物として次のステップへ渡してしまう。ここは必ず分岐に入れる。

## 自分の見立て

Sonnet 5 の本質は「安い Opus っぽいモデル」ではないと思っている。Sonnet クラスにも Opus 側で先に入った制約が降りてきて、モデルを温度や手動 thinking で細かく操る時代が終わりに近づいた、という変化に見える。開発者が握るレバーは、入力設計、token 予算、effort、拒否時の制御フローへ移っている。

移行で最初にやることは、ベンチ表を眺めることではない。既存ログを Sonnet 5 の token counter にかけ直し、`max_tokens` が推論込みで足りるかを見て、共通 wrapper から古い knob を消す。ここまでやって初めて、性能差をまともに測れる。モデルIDの1行変更で済ませたチームほど、たぶん9月の通常価格化と長文タスクでじわっと効いてくる。
