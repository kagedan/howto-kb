## 1. 何を作ったか

Azure OpenAI 経由で **`GPT Realtime Translate`（OpenAI が 2026 年 5 月 7 日に発表した、翻訳に特化したリアルタイム音声モデル）** を使った、iPhone Safari 向けの **双方向リアルタイム音声翻訳 Web アプリ** です。

* 🇯🇵 日本語 → 🇺🇸 英語、🇺🇸 英語 → 🇯🇵 日本語（UI で切替）
* 🎙 8 種類の voice（alloy / ash / ballad / coral / echo / sage / shimmer / verse）
* 📊 マイク音量に **実音量連動** したイコライザー表示
* 🔐 Microsoft Entra ID 認証必須
* ⏱ 30 分セッション自動キャップ（コスト保護）

![アプリの起動画面（iPhone Safari）](https://static.zenn.studio/user-upload/deployed-images/b1766387e7aafd5993cc78f3.jpg?sha=9ffeeac78acbc9b0b237ea80de34c2cfc55958a0)

このアプリが「リアルタイム」と言える理由は、モデルの構造にあります。

従来の音声 AI は、多くが「音声を文字にする → 文字を AI が処理する → 文字を音声に戻す」という多段パイプラインで動いていました。段ごとに待ち時間が積み重なり、声のトーンや間合いも途中で失われがちです。

`GPT Realtime Translate` は、この中間のテキスト変換を挟まず、音声を音声のまま直接やりとりします（speech-to-speech と呼ばれる構造です）。しかも翻訳に特化していて、**入力された音声を受け取りながら推論を進め、応答も音声のまま返していく**ため、会話のテンポが途切れにくい。「話しかけたら、待たされずに返ってくる」という体験は、この構造から来ています。リアルタイム翻訳という発想が個人で組めるようになった背景には、このモデルの登場があります。

OpenAI 公式の Realtime API ではなく、**Azure OpenAI** 経由で動かしています。これは、外向きの認証や運用ルールを Microsoft の世界に乗せたかったからです。

## 2. AI が「実装」したのはコードだけじゃない

「AI にアプリを作らせた」と聞くと、コードを書かせたイメージを持つかもしれません。でも実際に AI 主導で進めた開発は、それよりかなり広い範囲をカバーしていました。

| フェーズ | AI が用意したもの |
| --- | --- |
| **1. 依頼** | チャットでの自然言語のリクエスト 1 行から |
| **2. 設計・文書化** | **仕様書・システム設計書・システムテスト仕様書兼成績書・利用者マニュアル**（Mermaid 図含む）、さらに別 AI（Codex）へ**レビューを依頼し、指摘をレビュー記録として保存** |
| **3. 実装準備** | GitHub リポジトリ作成、IaC（Terraform）コード、CI/CD ワークフロー（GitHub Actions） |
| **4. デプロイ** | `terraform apply` 実行、失敗時のアーキ刷新（13 → 6 リソース）、再 apply 成功 |
| **5. テスト・改善** | URL 認証エラー対処、GA 仕様変更（URL パス・セッションスキーマ）への追従、UI 全面リデザイン（2 ビュー構成、波形 + イコライザー） |
| **6. 完成・反映** | URL 共有、変更内容の **仕様書・利用者マニュアルへの反映**、PR 最終 push |

つまり今回 AI がやったことは、**「コードを書く」より「開発プロセス全体を回す」** に近かったです。

設計書には Mermaid のアーキテクチャ図とシーケンス図が入り、利用者マニュアルには iPhone 向けの操作手順とトラブルシューティングが整理されました。Terraform は CAF（Cloud Adoption Framework）の命名規約に沿って 6 リソースを定義し、GitHub Actions は PR 時の plan、手動の apply、フロント自動デプロイの 3 ワークフローを揃えました。

ここはチャットで人間が一行ずつ書かせたわけではなく、「**仕様書、設計書、IaC、CI/CD まで、一気通貫で揃えて**」と頼んだら勝手に揃えてくれた、という感覚です。実験のため、私は途中の作業にはあえて手を出さず、**目的を伝えることと、途中の判断だけ** を担当しました。

## 3. つまずきポイントと、そのリアル

完全にスムーズだったわけではありません。むしろ、**発表されたばかりのモデルを早速触ったからこそ踏んだ罠** が、今後同じことをやる方の参考になるかもしれません。

主な引っかかりどころ：

| # | 詰まりどころ | 対処 |
| --- | --- | --- |
| 1 | Azure Functions の **VM クォータが 0** で `terraform apply` 失敗 | Static Web App の Managed Functions に統合、構成を **13 → 6 リソース** に簡素化 |
| 2 | WebSocket の **認証方式**（純正の subprotocol が Azure では効かない） | `api-key` をクエリパラメータで渡す方式に変更 |
| 3 | 入力音声の **サンプルレートが最低 24kHz** | クライアント側で 16kHz → 24kHz にリサンプル |
| 4 | `GPT Realtime Translate` は **専用エンドポイント**（`/openai/v1/realtime/translations`）が必要。公式 how-to が汎用 Realtime のままで、繋いでも即切断（`1006`） | 公式ドキュメントではなく **OpenAI の cookbook という一次情報** で正解を特定 |
| 5 | 翻訳セッションは **原文（話した日本語）の文字起こしを返さない** 仕様 | 別モデル `gpt-realtime-whisper` の WebSocket を **もう一本** 張る二刀流で、原文と翻訳を同時表示 |
| 6 | コードは正しいのに、修正が画面に反映されない | 原因はコードではなく **配信**（HTML のキャッシュ更新パラメータが固定）。値を更新して強制再取得 |
| 7 | 翻訳セッションに **「声の選択」を混ぜたら設定ごと弾かれ**、出力が勝手に別言語化 | 翻訳セッションでは voice を渡さない（モデル仕様に合わせる） |
| 8 | クラウドが自動生成した管理用リソース名に **通知先メールが埋め込まれ**、個人情報が露出 | 検知して削除。便利機能の自動生成物まで監査する |

これらは全部、「動かしてみないと分からない」タイプの落とし穴でした。とくに **発表されたばかりのモデルは "地図がまだ無い"**。公式どおりに繋がらないことが普通にあって、そういうときは設定を疑い続けるより、サンプルコードや一次情報に当たるほうが速い、と何度も思いました。

![原文と翻訳が同時に表示される（ の別 WebSocket で実現）](https://static.zenn.studio/user-upload/deployed-images/821c4f56e5882fc5b6683350.jpg?sha=b67efea62437bdecaeffb82ee3fcf092558ea549)

事前に完璧に設計しようとすると、半年前の知識ではもう古い、というスピードで仕様が動いています。**設計書を完璧にしてから実装する** より、**動かしながら設計書を更新する** ほうが、今の AI 時代には現実的だと感じました。

## 4.「リアルタイム翻訳」という言葉 ── 最初の作り方と、本物の違い

ここは少し時系列の話をします。実は、リアルタイム翻訳に最初に挑んだときは、**汎用の会話向け realtime モデル（`gpt-realtime` 系）** を使っていました。そのとき見えた gap が、今回 `GPT Realtime Translate` を使って初めて埋まったので、その「前後」を並べて書きます。

**【最初の作り方】疑似同時通訳を手で組んでいた頃**

汎用の `gpt-realtime` は本来「会話 AI」向けの設計です。`turn_detection: server_vad`（音声活動検出）でユーザーが話し終わるのを待ち、ターンが完結してから応答を返す ── これは **UN の同時通訳** というよりも、**iPhone の Siri のような対話モデル** に近い動きでした。

「話している間も翻訳音声が流れてくる」体験を作るには、API のデフォルト挙動から外れたショートカットの積み重ねが必要でした：

* VAD を完全にオフ
* クライアント側で 1.5 秒ごとに音声バッファを手動でコミット + 応答リクエスト
* さらに音量を見て、発話があった時だけコミット（無音時に勝手に応答するのを防ぐ）

この組み合わせで「話してる間に 1.5 秒の塊で英語が流れてくる」風には見えるものの、できあがるのは **「即応性の高い逐次翻訳」であって、本物の同時通訳ではない** ── これが、手を動かして初めて言語化できた現実でした。

**【今回】`GPT Realtime Translate` は、前提から違った**

そして今回。翻訳のために設計された `GPT Realtime Translate` は、**連続ストリームを前提に、話している側から訳が返ってくる**。あの 1.5 秒コミットや音量検知のショートカットが、そもそも要りません。「同時通訳っぽく見せるための工夫」をしていた場所が、まるごと不要になりました。

![声量に連動して動くイコライザー（動作中）](https://static.zenn.studio/user-upload/deployed-images/4e2de3602c0553fc450ecf91.jpg?sha=ed8426bf87adf2196f6a16ddc90c2719e3bfde9a)

## 5. AI Agent を「成長させる」ということ

今回特に意識したのは、**AI Agent 自身が学んだことをログとして残し、次回以降に活かせる仕組み** です。

成功したタスクだけでなく、失敗したタスクも、その原因と対処をノウハウとして蓄積し、再現可能なレシピにする。

たとえば、今回出てきた 10 件以上のつまずきポイントは、すべて「エラーログ」として時系列で残し、それを **次に類似 PoC を作るときの落とし穴チェックリスト** として AI 自身が参照できる形に保管しました。

![失敗を3箇所に記録し、1日のスキル成長として要約した実際のチャット](https://static.zenn.studio/user-upload/deployed-images/179b148bbeafbcdf4c231434.jpg?sha=a7e676fa0e0f106c1191b806fb1cccaf6a7e18fe)

人間で言えば、**振り返りと書き残しの習慣** に近いものです。

AI Agent のハーネスを設計する際には、人間と同じように、**成功体験と失敗体験の両方を記録し、振り返り、改善につなげる仕組み** が大事だと感じています。

そして、もう一つ効いたのが **「いつ止めるか」を人間が握る** ことでした。深夜、勘で何度もデプロイし直しそうになったところで、「3 回ダメなら一度止める」と決めて、推測ではなく一次情報の調査に切り替えた ── AI に任せきるのではなく、人が「どこで立ち止まり、何を調べ直すか」を判断する。**実作業は AI、止めどころの判断は人間**。この分担が、今回いちばん効いたように思います。

AI を一度作って終わりにせず、**経験を蓄積しながら少しずつ賢くしていく**。

この観点は、今後 AI Agent を実務で使っていくうえで、確実に重要になっていくと思います。

## 6. これから vibe coding / ハーネスエンジニアリングをやる人へ

この記事は、これから **vibe coding**（チャットから AI に作らせる開発）や **ハーネスエンジニアリング**（AI に "仕事をする身体" を設計してあげること）に踏み出す人の、ちょっとした参考になればと思って書いています。今回つまずきながら掴んだことを、持ち帰りやすい形で残しておきます。

* **完璧な設計書を待たない**。動かしながら設計書を更新する。最新モデルは半年で前提が変わるので、固めてから作ると着いた頃には古い
* **公式どおりに繋がらない時は、設定を疑い続けない**。一次情報（cookbook やサンプル）に当たる。発表されたばかりのモデルほど「地図がまだ無い」
* **制約は正面から押さず「別の口」で回避**する。原文が返らないなら、別モデルの WebSocket をもう一本張る ── という発想の柔らかさが効く
* **失敗を必ずログ化し、次の落とし穴チェックリストにする**。これがハーネスの中核で、AI を一度きりにせず成長させる装置になる
* **「いつ止めるか」は人間が握る**。3 回ダメなら止めて調べ直す。実作業は AI、止めどころの判断は人
* **自動生成物まで目視する**。便利機能が、個人情報や規約に外れた命名を勝手に作っていることがある

## 8. おわりに

今回、iPhoneからチャットで「リアルタイム翻訳を作って」と言って、数時間後にそれが動くアプリが手元にありました。これは、ドラえもんが出すひみつ道具の感覚に、似ているように感じました。

> こんなこといいな　できたらいいな  
> あんなゆめ　こんなゆめ　いっぱいあるけど  
> みんなみんなみんな　かなえてくれる  
> ふしぎなポッケで　かなえてくれる
>
> ――「ドラえもんのうた」より

技術的には確かに近づいてきています。  
ただし「**何が起きているのか**」「**何を許可しているのか**」「**どこで失敗したのか**」 ── これらを **理解しているのは人間** です。

のび太が道具を出しても使い方を間違えて転んだように、私も落とし穴を一つずつ自分で踏んで覚えました。SKUの選択、認証の設定、双方向のPCMストリーム——「できたらいいな」と「できた」の間には、ちゃんと地続きの作業がありました。

それでも、その距離は確かに短くなっていると感じています。AIエージェントはポケットそのものではなく、ポケットまでの距離を縮める道具なのだと思います。何を取り出したいかを考えて決めるところは、これからも自分の手元に残っている気がします。

「道具は変わった。でも『こんなこといいな』と最初に思うのは、今も自分だ。」

AI に "ほんやくコンニャク" を作ってもらった日。
