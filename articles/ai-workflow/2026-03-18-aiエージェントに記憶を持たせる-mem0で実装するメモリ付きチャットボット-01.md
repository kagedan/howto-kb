## 概要

LLM はステートレスな設計です。つまり、会話セッションが終わるたびに記憶はリセットされ、次の会話はゼロから始まります。

単純なチャットボットでは問題になりませんが、長期タスクをこなすAIエージェントにとっては致命的な制約です。「先週相談した設計方針を忘れてまた同じ提案をしてくる」「ユーザーの好みを毎回聞き直す」といった問題が発生します。

この記事では、AIエージェントの**メモリアーキテクチャ**を体系的に整理し、`mem0` を使ってセッションをまたいで記憶を保持するチャットボットを実装します。

## 技術スタック

| 項目 | 内容 |
| --- | --- |
| 言語 | Python 3.x |
| メモリライブラリ | mem0 |
| LLM | OpenAI gpt-4o-mini |
| ベクトルストア | Redis（永続化） / インメモリ（デフォルト） |
| 参考ライブラリ | cognee（グラフ＋ベクトルハイブリッド） |

## エージェントメモリの4種類

AIエージェントのメモリは人間の記憶システムに倣って4種類に分類されます。

| 種類 | 保存する内容 | 例 |
| --- | --- | --- |
| **エピソード記憶** | 過去の出来事・会話履歴のシーケンス | 「前のセッションでユーザはXについて聞いた」 |
| **意味記憶** | 事実・知識の構造化情報 | 「ユーザーの名前はアリス。Pythonが好き」 |
| **手続き記憶** | やり方・スキル・ツール定義 | エージェントが使えるツール一覧 |
| **作業記憶** | 現在のコンテキストウィンドウ | アクティブなプロンプト（Short-term Memory） |

今回の実装が扱うのは主に**エピソード記憶**と**意味記憶**です。ユーザが話した内容（出来事）と、そこから抽出された事実（名前・好み）を永続化します。

## メモリの検索戦略

保存したメモリをどう取り出すかも重要な設計ポイントです。

* **キーワード検索 / BM25** — 高速・完全一致。特定の単語を含む記憶を取り出す場合に有効
* **セマンティック検索 / ベクトル検索** — 意味ベースの曖昧な検索。「旅行について話した」と聞いても「ハイキングが好き」を引ける
* **知識グラフ探索** — エンティティ間の関係を辿る。「アリスの友人の好み」のようなリレーショナルな検索が可能

### ベクトルストア vs 知識グラフ

| 観点 | ベクトルストア（mem0） | 知識グラフ（cognee） |
| --- | --- | --- |
| 向いているケース | 「関連情報を取得できればOK」 | 「関係性まで把握したコンテキスト理解が必要」 |
| 検索の性質 | 類似度ベース | グラフ探索ベース |
| 実装難易度 | 低い | 高い |

単純なパーソナライズなら mem0 のベクトルストアで十分。複雑なエンティティ関係（「Aさんが言ったことをBさんに共有した」など）が必要なら cognee が有力です。

## 実装内容

### セットアップ

```
pip install mem0ai openai python-dotenv
```

### インメモリ版（セッションごとにリセット）

#### コード

```
from mem0 import Memory
from openai import OpenAI

memory = Memory()  # デフォルトはインメモリのベクトルストア
client = OpenAI()

user_id = "alice"

def chat(user_message: str) -> str:
    print(f"Saved memory:\n{memory.get_all(user_id=user_id)}")

    # 関連メモリを検索
    relevant = memory.search(user_message, user_id=user_id, limit=3)
    context = "\n".join(r["memory"] for r in relevant["results"])

    messages = [
        {"role": "system", "content": f"あなたは親切なアシスタントです。\nこのユーザーについて知っていること:\n{context}"},
        {"role": "user", "content": user_message},
    ]
    response = client.chat.completions.create(model="gpt-4o-mini", messages=messages)
    answer = response.choices[0].message.content

    # 新しいメモリを保存
    memory.add(user_message, user_id=user_id)
    return answer

print(chat("私の名前はアリスで、ハイキングが好きです。"))
print(chat("私について何を知っていますか？"))
```

### Redis で永続化する

#### コード

```
from mem0 import Memory
from openai import OpenAI
from dotenv import load_dotenv
import os

load_dotenv()

config = {
    "vector_store": {
        "provider": "redis",
        "config": {
            "collection_name": "mem0",
            "embedding_model_dims": 1536,
            "redis_url": "redis://localhost:6379"
        }
    }
}
memory = Memory.from_config(config)
client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

user_id = "alice"

def chat(user_message: str) -> str:
    print(f"Saved memory:\n{memory.get_all(user_id=user_id)}")

        {"role": "user", "content": user_message},
    ]
    response = client.chat.completions.create(model="gpt-4o-mini", messages=messages)
    answer = response.choices[0].message.content

    memory.add(user_message, user_id=user_id)
```

### mem0 の仕組み

`memory.add()` はユーザーの発言をそのまま保存するのではなく、LLM を使って「アリス、ハイキングが好き」のような事実を抽出してベクトル化します。`memory.search()` はクエリをベクトル化し、類似する記憶を取得します。

## ハンズオンで詰まったこと

### Ollama（ローカルLLM）は embedding の精度問題で断念

最初はコストをかけずに試したかったため、Ollama でローカル実行を試みました。しかしメモリの制約からモデルが `tinyllama` 一択だったため、embedding の精度により意味のある検索結果が得られませんでした。

**結論**: mem0 の embedding には十分な品質のモデルが必要です。私のようにPCに十分な空き容量（数G〜数十G）を確保できない方は、OpenAI の `text-embedding-3-small` が現実的な選択肢です。

### OpenAI API は $5 から気軽に試せる

OpenAI の [API プラットフォーム](https://platform.openai.com/home) でサインアップし、Billing から $5 チャージするだけで利用できます。自動チャージをオフにしておけば予想外の請求を防げます。

## 学んだこと

### メモリアーキテクチャの設計判断

今回実装した `chat()` 関数は以下のフローを持ちます：

```
ユーザー入力
  ↓
memory.search() → 関連する過去の記憶を取得
  ↓
system プロンプトにコンテキストとして注入
  ↓
LLM が文脈を踏まえて回答
  ↓
memory.add() → 今回の発言から事実を抽出・保存
```

「保存」は今の対話から事実を抽出して永続化すること。「検索」は過去の保存から現在の対話に必要な情報を取り出すことです。この2ステップが揃って初めてメモリが機能します。

### 実用的なメモリシステム

とは言え、実務に際しては検討事項も：

* **矛盾する記憶のマージ**: 「Pythonが好き」と後で「最近はRustばかり書いている」という情報が入ったとき、どう扱うか
* **忘却戦略**: 古くなった・重要度の低い記憶をいつ・どう削除するか
* **どのメモリ種類をいつ使うかの判断基準**: タスクに応じてエピソード記憶と意味記憶を使い分ける設計

## まとめ

**実務への応用可能例**

| ユースケース | 活用するメモリ種類 |
| --- | --- |
| パーソナライズドAIアシスタント | 意味記憶（ユーザーの好み・プロフィール） |
| 長期コードレビューエージェント | 意味記憶（設計方針・コードベースの構造） |
| 複数エージェント間の情報共有 | 共有ベクトルストア上のエピソード記憶 |
| 集計・分析エージェント | 意味記憶（テーブル定義・集計ロジック） |

**ナレッジ**

* **まず mem0 のインメモリで試す** → 動作確認後に Redis や PostgreSQL に切り替える
* **ベクトルストアで十分か検討する** → エンティティ間の関係が必要なら cognee のようなグラフ型メモリも検討
* **メモリを省いた場合、本番エージェントが壊れる箇所**: セッションをまたいだ一貫性・ユーザー文脈の継続・過去の意思決定の根拠

LLM の「忘れる」という制約は、メモリアーキテクチャを正しく設計することで克服できます。今後のエージェント開発において、この層の設計はデータベース設計と同じくらい重要なスキルになるのではないかと思います。
