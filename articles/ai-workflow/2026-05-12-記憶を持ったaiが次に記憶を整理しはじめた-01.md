[前回の記事](https://zenn.dev/daishir0/articles/ai-agent-enterprise-personal-session)で、Claude Managed Agentsが「brain / hands / session」という3層に分離されたことを書きました。なかでも「sessionが外に出たこと」——記憶が外部ストアに永続化されたこと——が、AIエージェントの使われ方を根本から変えると書きました。

今回はその続きです。「記憶が持てる」ようになった次のステップとして、Anthropicが発表した**Dreaming**という機能を読んで考えたことをまとめます。名前が良いですよねー。

## 「記憶があること」と「記憶が使えること」は違う

sessionの外部化によってAIは記憶を持てるようになりました。しかし記憶とは、ただ積み上げるだけでは使えません。

人間でも同じです。日記を毎日書いていても、振り返って整理しなければ同じ失敗を繰り返します。逆に、うまくいったやり方を意識的に言語化している人は、次の仕事に活かせます。仕事できるひとって、この辺が上手ですよね。

で、AIも同じ構造的な課題を抱えていました。セッションをまたいで文脈は保持できるようになったけど、その記憶の品質を誰も管理していない。ノイズが増え続け、重要な洞察が埋もれていく。とほほ。過去何度か経験ありです。

Dreamingはこの問題に対する答えです。

## Dreamingとは何か

Dreamingは、エージェントが**セッションとセッションの間に**、蓄積されたログや記憶を自律的に見直してキュレーションする仕組みです。

具体的には次のようなことを行います。

* **繰り返しのミスの検出**：同じ種類の失敗が複数セッションで起きていれば、パターンとして抽出する
* **効果的なワークフローの定着**：うまくいったアプローチを記憶として強化する
* **チームエージェント間での共有学習**：複数のエージェントが協働する環境では、個別エージェントで得た知見をチーム全体の共有メモリに集約する
* **不要な記憶の整理**：古くなった情報やノイズを削除してメモリを「高品質」に保つ

Anthropicの発表では、法律AIのHarveyでDreamingを適用したところ**タスク完了率が約6倍に向上した**と報告されています。はーびっくり。単なる記憶の保持ではなく、記憶の質を継続的に改善することが、これほどの差を生みます。

名前の通り、人間が眠っている間に脳が記憶を整理・定着させる「睡眠と記憶の固定」のメカニズムに似ています。うまい名前を付けたもんですね。

## もう一つの柱：Outcomes（実行中の自己検証ループ）

Dreamingがセッション「間」の学習なら、**Outcomes**はセッション「中」の品質制御です。

ドキュメントの表現を借りると、Outcomesは「sessionを *会話* から *仕事* に格上げする」機能です。あなたが「完成とはこういう状態だ」とルーブリック（評価基準）で定義すると、エージェントはその基準を満たすまで自律的に作業と修正を繰り返します。言い方なんでしょうが、「ゴール」に近いのかな。

### 仕組み：WriterとGraderの分離

内部では **Writer（執筆者）** と **Grader（採点者）** という2つの役割が分かれています。

```
ユーザーが outcome を定義
       ↓
Writer がタスクを実行
       ↓
Grader が独立したコンテキストウィンドウで成果物を採点
       ↓
  satisfied → 完了
  needs_revision → Writer に差し戻し → Writerが修正 → Graderが再採点
  max_iterations_reached → 終了（最大反復に達した）
```

GraderはWriterとは**まったく別のコンテキストウィンドウ**で動きます。これが重要な設計上のポイントで、Writerの思考過程や「言い訳」に引きずられることなく、純粋に成果物だけを評価できます。上手ですよね。

### ルーブリックの書き方が鍵

ルーブリックは「採点できる基準」で書かなければなりません。曖昧な基準はグレーダーが全通過させるデフォルトの失敗モードに陥ります。

```
# 悪い例（曖昧すぎる）
- データが正しく見える

# 良い例（採点可能）
- CSVにpriceカラムが存在し、数値が入っている
- 過去5年分の売上データを使った収益予測が含まれている
- WACCの計算根拠が明示されている
```

ちゃんと具体的ですよね。仕事でも達成目標は明確であればあるほど、行動はしやすいもので、成果も上がるもんです（実体験）。さて、ルーブリックは`user.define_outcome`イベントで送信します。`max_iterations`はデフォルト3、最大20まで指定できます。

### 実際の動作例

Anthropicのクックブックに掲載されている事例がわかりやすいです。EVの急速充電に関するリサーチブリーフを作成するエージェントに、「各引用URLが実際にアクセス可能か」「引用文が原文と完全一致しているか」「7つのカバー項目が揃っているか」という厳格なルーブリックを設定すると、こう動きます。

```
Draft 0 → Grader評価 → needs_revision
  「需要課金の定量値（$/kW）がない。EVgoの損益がSEC提出書類ではなく
   ニュース記事を引用している」

Revision 1 → Grader評価 → satisfied
  「7項目すべてカバー済み。全引用がLIVEかつ原文と一致」
```

1回の修正で基準をクリアした例ですが、重要なのはこのフィードバックが「NG理由の具体的な説明」として返ってくる点です。「何となく不十分」ではなく「どの基準がどう満たされていないか」が明示されるため、Writerは的確に修正できます。

### DreamingとOutcomesの関係

2つの機能は時間軸が異なります。

* **Outcomes**：1回のセッション「内」で品質を担保する（iterate → grade → revise ループ）
* **Dreaming**：複数セッション「間」で学習を積み上げる（記憶のキュレーション）

両方を組み合わせると、「今この仕事をきちんとやる」と「次の仕事をもっとうまくやる」が両立します。よく考えますよねーAnthropic。現実社会の経験則を上手に組み込みます。

## claude.aiのメモリ機能とどう違うのか

ここで個人的に一つ整理しておきたいのが、Dreamingと**claude.aiのメモリ機能**の違いです。混同しやすいですが、設計思想がまったく異なります。

|  | claude.aiのメモリ | Dreaming |
| --- | --- | --- |
| 対象 | エンドユーザー（個人） | 開発者が作るAIプロダクト |
| 何を覚えるか | ユーザー自身の情報（名前・好み・文脈） | エージェントのパフォーマンスパターン |
| 誰が使うか | claude.aiを使う一般ユーザー | SDK/APIで自社AIを構築する開発者 |
| 学習の単位 | 1ユーザー × 会話の蓄積 | 複数セッション・複数エージェントをまたいだパターン抽出 |

一言で言うと、claude.aiのメモリは「**ユーザーを知る**」ための機能で、Dreamingは「**タスクがうまくなる**」ための機能です。

前者は「誰と話しているか」、後者は「どうすれば仕事がうまくできるか」。対象レイヤーが根本的に違います。claude.aiのメモリが個人秘書の記憶であるとすれば、Dreamingは組織が業務手順書を継続的に改善していくプロセスに近いイメージです。←うまいこと言った。

## 実装はこうなるはず（たぶん）

Managed Agents SDKの現在の構造と今回の発表から、実装の輪郭を推測できます。

**メモリストアはセッションから分離した永続レイヤーになる。** セッションが終了してもメモリは残り、次のセッションで参照できます。おそらく`agent.memory`のようなオブジェクトにread/writeの口が生えるイメージです。これはイメージしやすいっすよね。

**Dreamingはセッション終了後に非同期で発火する。** `dreaming_interval`のような設定でスケジュールを制御するか、プラットフォーム側が自動的にトリガーする設計になると思われます。ユーザーが意識しなくても、セッションの合間にバックグラウンドで整理が走る。Dreamingですからね、寝てる間に整理しなきゃってことで、これもイメージしやすい。

**マルチエージェント環境では共有メモリに集約される。** 複数のエージェントが並列で動いている場合、個別の学習を共有メモリに統合することで、チーム全体が同じ失敗を繰り返さない仕組みが実現します。素晴らしい。

## 「整理する」レイヤーが生まれた

前回の記事で書いたように、Managed Agentsはbrain（考える）/ hands（動く）/ session（覚える）という3層に分かれました。Dreamingはその上に、新しい第4のレイヤーとして加わったと考えています。

```
brain    ── 考える（推論）
hands    ── 動く（ツール実行）
session  ── 覚える（記憶の永続化）
outcomes ── 検証する（実行中の品質ループ）  ← 今回①
dreaming ── 整理する（記憶の自己最適化）   ← 今回②
```

レイヤが多すぎ？ま、いいんです。OSI参照モデルも参照モデルとして役割は果たして、最終的にTCP/IPなんですから。ともかく「考える・動く・覚える」AIから、「検証する・整理する」AIへってことですね。

前回書いたエンタープライズとパーソナルの文脈で言えば、両機能はどちらにも意味を持ちます。エンタープライズでは「品質基準に達するまで自律的に修正し、組織の暗黙知が形式知化される」ことを意味し、パーソナルでは「使えば使うほど自分のことをわかってくれる、かつ仕事がうまくなるAI」が実現していきます。えーーますます便利じゃん。

## おわりに

「セッションが終わったらAIはリセットされる」という前提は、今後どんどん変わっていくでしょう。ステートレスだったからこそ面倒だったけれど、毎回新しい気持ちでコントロールできるAIから変わっていきます。これって車のオートマとマニュアルの違いなのかな。

余談です。似た仕組みをafk-codeで個人的に手動実装していましたが、便利です。自分で作って楽しむだけじゃなくて、パブリックな文書化をすること、大事ですね。
