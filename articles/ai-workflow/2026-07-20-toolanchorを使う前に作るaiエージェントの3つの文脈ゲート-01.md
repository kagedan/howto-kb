AIエージェントに新ツールを渡しても、使ってくれない。モデルより、文脈の入口を分けた方が直る。定義を足してログを見ると、既存ツールへ何度も戻っていた。能力不足というより、選ぶ場面で新しい手段が見えていない。

7月14日に出た [ToolAnchor](https://arxiv.org/abs/2607.14145) は、「慣れたツールへ戻る」性質を behavioral inertia と呼ぶ。判断の節目へ、新ツールを使う反実仮想の文脈を入れる方法だ。翌15日の [AI Agents Do Not Fail Alone](https://arxiv.org/abs/2607.14275) は、役割や指示の整合性、ツールスキーマを含む7項目で文脈品質を見る。実装でやることは近い。

## ツール一覧を「常時表示」にしない

つまり、ツール定義を全部プロンプトへ積む設計は、追加に弱い。モデルは選択肢を読める。でも、今の作業と結び付かなければ選べない。

自分なら文脈を次の3ゲートに分ける。

| ゲート | 入れるもの | 落とすもの |
| --- | --- | --- |
| 受付 | タスクの目的、禁止事項 | 過去の手順全文 |
| 選択 | 今回だけ候補になるツール、入出力の例 | 無関係なツール仕様 |
| 実行 | 選んだツールの引数、検証条件 | 他ツールの説明 |

![AIエージェントの3つの文脈ゲート設計: 受付/選択/実行の役割分担](https://static.zenn.studio/user-upload/6c6a9b0b43e4-20260720.png)

この分け方だと、ツールを増やしても選択ゲートだけを更新すればいい。どの道具を使うかを、モデルが推測しなくて済む。

## 「使える」と「選ばれる」の間に判定を書く

ToolAnchorの核は、失敗した軌跡に別の判断材料を差し込む点にある。まずは失敗しやすい分岐を、コード側で明文化すると扱いやすい。

```
def choose_tool(task: Task) -> Tool:
    if task.needs_fresh_source and task.url is None:
        return web_search
    if task.needs_exact_record and task.url is not None:
        return fetch_page
    return ask_for_input
```

ここで大事なのは `web_search` を登録したことではなく、「新しい一次情報が必要なら検索」と判断を固定したこと。ツール説明に用途を長々と書くより、この分岐とテストケースを残す方が、次のモデル更新でも壊れにくい。

論文が挙げる7項目のうち、最初に見たいのは指示の整合性とツールスキーマだと思う。ツール名、用途、失敗時の戻り先が別々の場所に書かれていると、モデルには矛盾した指示に見える。エージェントが妙な遠回りをした時、推論ログより先にこの3点を見直す価値がある。

## 次に増えるのはツールではなく、入口のテスト

エージェントの品質は、搭載したツール数では伸びない。どの条件でそのツールを文脈へ入れるか、その条件を再現できるかで決まる。今後はツール追加のPRに、選択ゲートの入力例と不採用例を添えるチームが強くなるはずだ。モデルを替えても挙動が変わらない場所を、先にコードとして持てるからだ。
