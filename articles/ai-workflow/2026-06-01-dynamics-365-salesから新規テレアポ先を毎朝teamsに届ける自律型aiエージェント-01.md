## はじめに

新規テレアポは、しんどい。

私たちの会社（株式会社学情）は、新規顧客の開拓を非常に重視しています。特に新人営業は、新規テレアポを通じて顧客基盤をゼロから築いていく。これは営業としての成長に不可欠なプロセスですが、現実は厳しいものです。

* **どこに架電すればいいかわからない。** Dynamics 365上に大量の取引先データはあるが、そこから「新規で狙うべき企業」を見つけ出す方法がわからない
* **リスト作成に時間がかかる。** アプローチ先を探す作業自体が属人化しており、ベテランのやり方を見て学ぶしかない
* **断られ続けるとメンタルがやられる。** 根拠のないリストで闇雲に架電し、断られ続ける。「なぜこの企業に電話しているのか」という納得感がないまま数をこなすのは、精神的にも非効率的にも最悪のパターン

**だったら、AIがナレッジから判断して、「ここに電話してみたら？」と企業をレコメンドしてくれればいいのではないか。**

この発想から、Microsoft Agent Hackathon powered by Tokyo Electron Device にて、 **新規テレアポ先を毎朝Teamsに届ける自律型AIエージェント** を開発しました。

本記事では、実装上の工夫・アーキテクチャ・プロンプト設計について解説します。

---

## エージェントの概要

### 最大の特徴：自律駆動 × Teams配信

このエージェントの最大の特徴は、 **ユーザーが話しかける必要がない** ことです。

一般的なAIチャットボットは「ユーザーが質問する → AIが回答する」という対話型ですが、本エージェントは **Power AutomateのRecurrenceトリガー（タイムトリガー）により、毎朝自動で起動** します。

```
毎朝 [指定時刻]
  → Power Automateが自動起動
  → Copilot Studioエージェントを呼び出し
  → フィルタリング済みデータからレコメンド企業を抽出
  → 結果をTeamsチャットに自動送信
  → 営業担当者はチャットを見てリンクをクリックするだけ
```

Teamsチャットには以下の情報が届きます。

* **企業名**
* **D365の取引先企業ページへのリンク**
* **レコメンド理由**（なぜこの企業に電話すべきか）

営業担当者がやることは、 **朝、Teamsのチャットを開いて、リンクをクリックするだけ** です。新しいツールを覚える必要も、ファイルを開く必要もありません。普段使っているTeamsに、根拠付きのレコメンドが届く。

これは単なるチャットボットではなく、 **自律的に判断し、行動し、結果を届けるエージェント** です。

## アーキテクチャ

### システム構成図

### 構成要素と役割

| コンポーネント | 役割 |
| --- | --- |
| **Dynamics365 Sales / Dataverse** | 取引先企業・商談・活動履歴などの業務データを保持 |
| **Microsoft Fabric（データフロー）** | Dataverseからデータを取得し、フィルタリング条件に基づいて新規アプローチ候補を抽出。結果をExcel形式でSharePointに出力 |
| **SharePoint** | フィルタリング済みExcelの格納先。定期更新により常に最新データを維持 |
| **Copilot Studio** | SharePoint上のExcelをナレッジソースとして読み込み、条件に合致した企業をレコメンド |
| **Power Automate（Recurrenceトリガー）** | 毎朝自動でCopilot Studioエージェントを呼び出し、結果を **Teamsチャットに送信**。自律駆動の要 |

### データの流れ

```
Dynamics 365 Sales / Dataverse
    ↓  Microsoft Fabric（データフロー）で定期取得・フィルタリング
SharePoint（フィルタリング済みExcel）
    ↓  ナレッジとして読み込み
┌─── Power Automate（Recurrenceトリガーで毎朝自動実行）───┐
│  Copilot Studio エージェント                            │
│      ↓  レコメンド企業をJSON出力                         │
│  Teamsチャットに整形して送信                             │
└─────────────────────────────────────────────────────────┘
    ↓  リンクをクリックするだけ
営業担当者
```

## 実装上の工夫

### 工夫①：「自律型」にこだわった設計

多くのAIエージェントは「ユーザーが話しかけて初めて動く」対話型です。しかし、新規テレアポに悩んでいる新人営業が、自分からAIに質問できるでしょうか？ **何を聞けばいいかすらわからない** のが実態です。

だからこそ、 **エージェント側から能動的にレコメンドを届ける** 設計にしました。

Power AutomateのRecurrenceトリガーにより、毎朝指定時刻にフローが自動起動。Copilot Studioエージェントを呼び出し、レコメンド結果をTeamsチャットで配信する。営業担当者は何も操作する必要がありません。

**「使ってもらう」のではなく「届ける」。** これがこのエージェントの設計思想です。

### 工夫②：AIにデータ抽出させない設計判断

今回の開発で最も重要な設計判断は、 **「データの絞り込みをAIに任せない」** ことでした。

**❌ 悪い設計：**

> AIに全データを渡して「新規テレアポ先を選んで」と依頼する  
> → 判断理由がブレる。毎回違う基準で選定され、再現性がない

**✅ 良い設計（採用した方法）：**

> データ処理側（Microsoft Fabric）でフィルタリングを実行 → 条件に合致した企業だけをエージェントに渡す → AIは「なぜこの企業がおすすめなのか」を説明する役割に徹する

つまり、 **AIの役割を「データ抽出」から「説明」に限定** したのです。

AIが得意なのは自然言語での説明や推論であり、大量データの条件分岐処理はFabricのデータフローのようなデータパイプラインの方が正確かつ高速に処理できます。**適材適所の役割分担** がAgentic AI設計の鍵です。

### 工夫③：ナレッジソースの前処理と定期更新

Copilot Studioは大量の構造化データを直接ナレッジとして扱うことが困難です。ファイルサイズの制限もあり、全データをそのまま読み込ませることは現実的ではありませんでした。

そこで以下のアプローチを採用しました。

1. **Microsoft Fabricのデータフローでフィルタリングを実行**し、新規アプローチ候補のみを抽出
2. 結果を **Excel形式でSharePointフォルダに出力**（データ量を大幅に削減）
3. **定期更新を設定**し、常に最新のフィルタリング結果を維持
4. Copilot Studioは **SharePoint上のExcelをナレッジソースとして参照**

「Fabricデータフロー → Excel → SharePoint → ナレッジ」という構成は、 **現実的かつスケーラブルな解** でした。

### 工夫④：Teamsチャットへの直接配信

レコメンド結果の出力先として、ExcelファイルやSharePointリストではなく、 **Teamsチャットを選択** しました。

理由はシンプルです。営業担当者が毎朝必ず開くのはTeamsだからです。

Power Automateがエージェントの出力（JSON）を受け取り、Teamsチャットに以下の情報を整形して送信します。

* **企業名**
* **D365取引先企業ページへのリンク**（クリックすれば即座にD365の該当企業ページに遷移）
* **レコメンド理由**（「なぜこの企業に電話すべきか」の根拠）

**「なぜこの企業なのか」が明示される** ことで、新人営業は納得感を持ってテレアポに臨めます。闇雲に電話するのと、根拠を持って電話するのでは、断られたときのメンタルダメージがまるで違います。

また、TeamsチャットにD365リンクを埋め込むことで、 **チャット → D365 → 架電** という導線が最短になります。リストを探す・ファイルを開く・コピペするといった無駄な作業は一切発生しません。

## プロンプトの工夫

### 2段階プロンプト設計

Copilot Studio上のプロンプトは、 **2段階に分割** して設計しました。

**第1段階：ナレッジから条件を抽出**

ナレッジソース（フィルタリング済みExcel）から、候補企業の条件・特徴を抽出するプロンプトです。ここではデータの読み取りと構造化に専念させます。

**第2段階：レコメンド企業を抽出・理由付け**

第1段階の結果を受けて、具体的なレコメンド企業を抽出し、「なぜこの企業に新規アプローチすべきか」の理由を付与してJSON形式で出力します。

### なぜ分割したのか

1つのプロンプトに「データ読み取り」と「レコメンド生成」を同時にやらせると、 **処理が不安定になり、JSON出力が崩れる** 問題が発生しました。役割を分離することで、各段階の出力品質が安定しました。

## フィルタリングロジック

今回のハッカソンでは、MVP（Minimum Viable Product）として **まず動作する仕組みを優先** しました。大量の取引先データから新規アプローチ候補を絞り込むために、Microsoft Fabricのデータフロー上で以下のフィルタリング条件を設定しています。

| フィルタリング条件 | 目的 |
| --- | --- |
| **所在地が東京都23区の主要地区** | FS部員の活動エリアに合致する企業に絞り込む |
| **最終活動日から一定期間が経過している** | 長期間コンタクトが取れていない＝再アプローチの余地がある企業を抽出 |
| **過去の受注実績がない** | 既存顧客ではなく、純粋な新規開拓ターゲットに限定 |

これらの条件でデータを絞り込んだ上で、Copilot Studioエージェントが「なぜこの企業にアプローチすべきか」のレコメンド理由を生成します。

## 苦労した点

### Copilot Studioのナレッジ読み込み制限

大量の取引先データをそのまま扱うことは不可能でした。複数の方法を検討した結果、 **「Fabricのデータフローでフィルタリング・絞り込みを行い、候補データのみをExcelで渡す」** という現実解に着地しました。

### JSON出力の不安定さ

カスタムプロンプトでJSON出力を指定しても、エージェントが安定してJSON形式で返さないケースが頻発しました。プロンプトの2段階分割より改善しました。

### ほぼ未経験のチームで挑んだ開発

今回開発に関わったメンバー3名は、AIエージェント開発においてはほぼ初心者でした。それぞれPower PlatformやDynamics 365の業務経験はあるものの、Copilot Studioでエージェントを設計・構築した経験はほとんどありません。月・水・金の定例MTGで進捗を共有し、各自の得意分野で役割分担することで、限られた期間内での完成を目指しました。

## 今後の展望

* **スコアリングロジックの導入**：現在のフィルタリング（条件一致による絞り込み）を、企業規模・業種・過去の商談履歴・接触頻度などの複数指標を重み付けした **加点方式のスコアリング** に発展させ、レコメンド精度を大幅に向上させる
* **対象テーブルの拡張**：取引先企業だけでなく、取引先担当者・リードにも対象を広げ、より精度の高いレコメンドを実現
* **Dynamics365 Salesとの双方向連携**：レコメンド結果をD365のタスクとして自動登録し、架電結果のフィードバックも自動収集
* **架電結果のフィードバックループ**：「このレコメンドは当たりだった/外れだった」を記録し、フィルタリング・スコアリング精度を継続的に改善

## まとめ

新規テレアポは営業の登竜門であると同時に、最も心が折れやすい業務でもあります。

**「どこに電話すればいいかわからない」「なぜこの企業に電話するのかわからない」** ――この2つの不安を、AIエージェントが解消する。しかも、毎朝自律的に動いて、Teamsのチャットにレコメンドを届ける。話しかける必要すらない。チャットを開いて、リンクをクリックするだけ。

最も重要な設計判断は、AIに何をやらせて、何をやらせないかの線引きです。データの絞り込みはFabricのデータフローに任せ、AIには「説明」と「レコメンド理由の言語化」に徹してもらう。この役割分担こそが、Agentic AIを業務で実用するための鍵だと考えています。

ハッカソンはプロトタイプです。しかし、 **「毎朝、Teamsを開いたらAIが根拠付きでテレアポ先を届けてくれる世界」** が実現可能であることを示すには十分な筋を見せられたと思います。
