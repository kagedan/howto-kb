AIエージェントにPRを作らせる話になると、多くのチームは最後のマージボタンを人間が押すかどうかから考え始める。

もちろん、それは大事だ。だが、そこだけ見ていると設計を間違える。人間が最後にマージしたとしても、その前にどのタスクを許可し、どのファイルを触らせ、どの判断をエージェントに任せたのかが曖昧なら、リスクはあまり減っていない。

本当に嫌なのは、「AIが勝手にコードを書くこと」よりも、「誰が何を判断したのか」がログから消えていくことだ。

2026年のAI coding agent運用で設計すべき中心は、マージ権限より作業境界である。

## PRは成果物ではなく、実行ログの出口になる

これまでPRは、人間が作業した結果をレビューに出す場所だった。差分があり、説明があり、コメントがあり、最後にマージ判断がある。多少雑でも、そこには「この人がこう考えて作った」という前提があった。

AIエージェントがPRを作るようになると、この前提が崩れる。

たとえば、エージェントがIssueを受け取り、リポジトリを読み、実装方針を決め、テストを走らせ、PR説明まで書く。人間は最後に差分を見て承認する。見た目は従来のPRフローに近い。だが中身はかなり違う。

人間がレビューしているのは、もはや「自分か同僚が書いたコード」とは限らない。途中の探索、判断、失敗、再試行をすべて含んだ自動実行の最終出力である。

ここでPRだけを見ても足りない。差分が小さくても、エージェントがどの権限で何を読んだのか、どのツールを使ったのか、なぜその実装を選んだのかが見えなければ、レビューはかなり苦しい。レビュー欄は、実行過程を取り戻すには狭すぎる。

だからPRは、成果物というより実行ログの出口になっていく。出口だけ堅くしても、上流の水路が雑なら事故は起きる。

## マージ権限だけを守る設計は、少し古い

「AIにはマージ権限を渡さないので安全です」という説明をよく見る。気持ちはわかる。最後の一線を人間が持つのは自然だ。

ただ、それだけでは弱い。

マージ権限を持たないエージェントでも、危ない変更は作れる。依存を増やす。設定を緩める。テストを都合よく直す。セキュリティ境界に触る。生成物を編集する。あるいは、本来人間が議論すべき設計判断を、なめらかなPR説明で既成事実のように見せる。

人間が最後にマージする場合でも、レビューする情報量には限界がある。巨大な差分、複数ファイルの横断変更、見慣れないツール呼び出し、CIの再実行。疲れているときほど、「まあテスト通っているし」で進みやすい。

ここで必要なのは、最後の承認者を決めることに加えて、そもそもエージェントにどんな作業単位を渡してよいのかを決めることだ。

作業境界は、たとえば次のように切れる。

* 触ってよいディレクトリ
* 変更してよいファイル種別
* 追加してよい依存
* 実行してよいコマンド
* 使ってよいMCPや外部ツール
* 読んでよいシークレットや設定
* 人間の確認なしに進めてよい判断
* 必ず止まるべき条件

これは細かい運用ルールに見えるかもしれない。だが、AIエージェントにとっては実質的な設計図である。

## エージェントはチャット相手から作業キューに変わる

GitHub Copilotのcloud agent taskがAPIから起動できるようになったことは、地味だが大きい。

チャット欄から「これ直して」と頼む世界では、AIエージェントはまだ個人の道具に見える。だがAPIでタスクを開始し、進捗を追跡できるようになると、話が変わる。エージェントは開発者の横にいる相談相手ではなく、作業キューの実行者になる。

そうなると、設計対象も変わる。

どのIssueをエージェントに流すのか。どのラベルなら自動で開始してよいのか。途中でキャンセルできるのか。失敗したら誰に通知するのか。PRを出す前にどのチェックポイントを通すのか。人間が見るべきログはどこに残るのか。

これはプロンプトの書き方というより、ジョブシステムの設計に近い。

CI/CDでは、どのブランチで何が走るか、どの環境にデプロイされるか、どのステップで承認が必要かを決める。AIエージェントも同じ方向へ行く。違いは、実行されるのがシェルスクリプトだけではなく、コードを読み、判断し、変更を作るプロセスだという点だ。

だから怖い。便利だからこそ怖い。

一度キューに入れたら、エージェントはそれっぽく前進する。人間よりも迷いなく、大きく動く。だから「何を頼むか」より前に、「どこまで動いてよいか」を決める必要がある。

## MCP allowlistはセキュリティ設定ではなく、開発文化の設定である

Copilot CLIのenterprise managed pluginsやMCP allowlistのような機能は、単なる管理者向けオプションに見える。実際には、ここがかなり効くと思う。

AIエージェントの能力は、モデルだけで決まらない。どのツールを呼べるか、どのコンテキストを読めるか、どの外部システムに触れるかで、行動範囲が決まる。

同じモデルでも、ファイル編集だけできるエージェントと、Issue、PR、CI、社内ドキュメント、クラウド環境まで触れるエージェントでは、まったく別物だ。後者は便利だが、境界を間違えるとレビュー不能な作業者になる。

MCP allowlistは、「危ないツールを禁止する」ためだけのものではない。チームがAIにどう働いてほしいかを定義する設定でもある。

たとえば、調査系のMCPは許可するが、本番環境に触るMCPは禁止する。ローカルのテスト実行は許すが、依存追加は人間確認を必須にする。コード検索は広く許すが、社内Wikiの読み取り範囲はチーム単位で絞る。

こういう判断には、チームの思想が出る。速く動きたいのか、事故を減らしたいのか。個人の自由を重視するのか、組織として再現性を重視するのか。AIエージェントの設定は、開発文化をかなり露骨に映す。

少し前まで、チーム開発の標準化といえばESLint、Prettier、tsconfig、CI workflowだった。これからはそこにagent configが入る。どのエージェントに何をさせるか。どの能力を全員に配るか。どの能力は明示的に禁止するか。

これはモデル選びより地味だが、実務ではこちらのほうが効く。

## 人間が握るべきレバーを先に決める

では、実際に何から決めるべきか。

最初に決めたいのは、「AIに何をさせたいか」という大きな願望ではなく、「人間がどこで止めるか」だ。

たとえば、次のようなレバーがある。

* タスク投入前の分類: バグ修正、リファクタ、依存更新、設計変更を分ける
* 実行中の停止条件: ファイル数、差分サイズ、テスト失敗回数、未知の依存追加
* 権限の境界: 書き込み可能な範囲、利用可能なMCP、外部アクセスの範囲
* レビュー前の要約: 何を読んだか、何を変更したか、何を諦めたか
* マージ前の人間判断: 設計判断、セキュリティ境界、ユーザー影響

ここで大事なのは、全部を細かく縛ることではない。縛るべきところを明確にし、それ以外は任せる。

AIエージェントに向いている作業は確かにある。退屈な移行、機械的な修正、テスト追加、古いAPIの置き換え、軽い調査。こういう作業は、人間が全部やるより速いことが多い。

ただし、向いている作業ほどキュー化されやすい。キュー化されるほど、人間は途中を見なくなる。そこで止める条件が効いてくる。

「最後にレビューするから大丈夫」は、エージェントが少数で、差分が小さく、作業頻度も低いあいだは成立する。運用が増えると、たぶん破綻する。

## PRテンプレートより、実行テンプレートが必要になる

多くのチームにはPRテンプレートがある。変更内容、確認事項、スクリーンショット、関連Issue。これは今後も必要だ。

だがAIエージェントには、PRテンプレートだけでは足りない。PRが出てきたあとに質問しても、すでに作業は終わっているからだ。

むしろ必要なのは、実行前のテンプレートである。

```
task_type: bugfix
allowed_paths:
  - app/components/
  - app/composables/
blocked_paths:
  - server/
  - infra/
requires_human_approval:
  - dependency_change
  - auth_logic_change
  - generated_file_edit
stop_if:
  - changed_files > 8
  - tests_fail_twice
```

これは極端に単純化した例だが、方向性はこうだと思う。AIに渡す仕事を、自然文のお願いではなく、実行可能な境界として渡す。

人間が「いい感じにやって」と頼むほど、AIはいい感じに推測する。推測が当たると気持ちいい。外れると、なぜ外れたのかを追いにくい。

境界を構造化しておけば、レビューの観点も変わる。

この差分は正しいか。さらに、このタスクは許可された範囲に収まっていたか。止まるべきところで止まったか。人間承認が必要な判断を勝手に進めていないか。

AIエージェント時代のレビューは、コードの正しさだけでなく、実行の正しさを見る必要がある。

## プロンプトエンジニアリングから、agent configの設計へ

しばらく前まで、AI codingの話はプロンプトに寄りがちだった。どう頼むと良いコードが出るか。どんな言い回しが効くか。どのモデルが賢いか。

それは今でも役に立つ。だが、チーム開発ではだんだん主役ではなくなる。

個人のプロンプトは再現しにくい。チャット履歴に残るだけで、レビューしにくい。人によって品質が揺れる。昨日うまくいった頼み方が、今日の別タスクに効くとは限らない。

agent configは差分として残る。レビューできる。履歴を追える。チーム全体に効く。古くなったら直せる。

これは、AIを信用するためというより、AIを信用しすぎないための仕組みだ。

人間が最後にマージするだけでは、もう足りない。AIがどの作業に入ってよいのか、どの能力を使ってよいのか、どこで止まるのか。それを設定として持たないチームは、レビュー欄で毎回同じ事故を後追いすることになる。

AIエージェントにPRを任せるなら、まずマージボタンの前を設計したほうがいい。

作業境界を決める。権限を絞る。ログを残す。止まる条件を決める。

この地味な設計ができていないチームにとって、エージェントは賢い同僚ではなく、説明のうまい自動実行環境になる。そこに安心してマージ権限だけ残しても、守れるものは少ない。

これからのAI coding agent運用で差がつくのは、モデルを選ぶ速さよりも、エージェントの仕事をどこで始め、どこで止め、誰が判断したと記録するかである。
