AIエージェントのPRで怖いのは、AIがコードを書くことではない。

本当に怖いのは、人間がどこで判断したのかわからないPRが、普通のレビュー待ちとして流れてくることだ。

diffだけ見れば、それっぽい。説明も一応ある。テストも何かしら通っている。けれど、なぜそのissueを始めたのか、どの制約で動かしたのか、何を検証し、何を諦めたのかが見えない。そういうPRは、レビューできるようでいて、実はかなり危ない。

AIがissueからbranchを作り、修正し、PRを出す流れは、もう特別なデモではなくなりつつある。GitHub Copilot coding agent のような仕組みは、AI作業を既存のissueとpull requestの中へ押し込む。これは自然な進化だと思う。開発チームはすでにその儀式で仕事を回しているからだ。

ただし、そこにAIを入れるなら、PRの意味も変わる。

これからのPRは、単なる差分では足りない。意思決定ログでなければならない。

## issueは依頼文ではなく、実行仕様になる

人間にissueを渡すとき、多少雑でも通ることがある。

「この画面を直して」「このAPIを整理して」「このテストが落ちているので見て」くらいでも、相手がチームの文脈を知っていれば補える。過去の経緯、触ってよい範囲、暗黙の地雷、レビューで嫌われる変更。人間はそこを雑に読んで、勝手に加減する。

AIエージェントには、その勝手な加減を期待しすぎないほうがいい。

agent-readyなissueは、要望メモではない。小さな実行仕様である。

最低限、目的、触ってよい範囲、触ってはいけない範囲、検証コマンド、期待するPRの粒度くらいは書いたほうがいい。これがないと、AIはPRを作れる。しかし、人間はレビューしにくい。

たとえば「フォームのバリデーションを改善して」では広すぎる。

「signup formのemail fieldだけを対象にし、既存のvalidation helperを使い、UI文言は変えず、`pnpm test signup` と `pnpm lint` の結果をPRに残す」なら、だいぶ違う。AIが賢くなるというより、人間があとで判断できる単位に仕事が切れている。

ここが曖昧なままAIに投げると、PR descriptionが苦しくなる。何を直したかは書けるが、なぜその差分でよいのかが説明できない。レビューで本当に知りたいのは、そこなのに。

## PR descriptionは要約欄ではない

AIが作ったPRの説明は、油断すると妙にきれいになる。

変更点が並ぶ。影響範囲が書かれる。テスト結果も一行ある。見た目は悪くない。でも、読み終わっても不安が残ることがある。

何を試して、何を捨てたのかがないからだ。

人間のレビューでは、最終差分だけでなく、判断の跡を見る。なぜこの設計にしたのか。別案はなかったのか。なぜこのファイルに触ったのか。テストは本当にこの範囲で足りるのか。失敗したコマンドはあったのか。

AI PRのdescriptionは、そこを回収する場所になる。

「実装しました」では足りない。できれば、次のような情報が近くにほしい。

* issueから読み取った作業目的
* 変更した範囲と、意図的に触らなかった範囲
* 実行した検証コマンドと結果
* 失敗した試行、または採用しなかった案
* 人間に確認してほしい判断点

これは儀式を増やしたいわけではない。むしろ逆で、レビューの時間を減らすために必要になる。

diffを開いた瞬間、人間はすでに負けていることがある。変更量が多い。文脈が薄い。AIがなぜそこを触ったのかわからない。そこから全部掘り返すのは、レビューではなく考古学に近い。

PR descriptionが意思決定ログになっていれば、最初に見るべき場所ができる。そこで怪しければ閉じる。足りなければ差し戻す。納得できればdiffへ進む。

この順番を作れるかどうかが、AI PRの運用ではかなり大きい。

## AI code reviewは人間のレビューを消さない

AIがPRを作るなら、AIに先にレビューさせればいい。そう考えるのは自然だ。

実際、その方向は進むと思う。静的解析、テスト不足の指摘、危ないAPI利用、セキュリティ上の匂い、変更範囲の不一致。AIが先に見たほうがよいものは多い。

ただ、そこで人間のレビューが消えるわけではない。

むしろ、人間が最後に何を閉じるのかを、はっきり決める必要が出てくる。

AIに任せてよいのは、見落としやすいパターンの洗い出し、テスト観点の提案、差分とissueの照合あたりだろう。人間が閉じるべきなのは、目的に対してこの変更を入れるべきか、将来の保守に耐えるか、チームの設計方針と合うか、そして何より、このPRをmergeする責任を誰が持つかだ。

ここを曖昧にすると、AI reviewは便利なふりをした責任の拡散になる。

「AIがレビューしました」は、判断ではない。「AIがここを確認し、人間がここを見て、最後にこの人がmerge責任を持つ」まで決まって、ようやくレビュー設計になる。

エージェント時代のコードレビューは、人間が全部読む精神論では回らない。だが、人間が読まなくてよいという話でもない。AIに先に見せる範囲と、人間が絶対に閉じる範囲を分ける。その境界がないチームほど、レビュー待ちのPRだけが増えていく。

## OSS maintainerが嫌がるもの

AI生成コードへの反発を見ると、しばしば「AIを使うな」という話に単純化される。

それは少し雑だと思う。

多くのmaintainerが嫌がっているのは、AIそのものというより、説明できない差分を他人のレビューキューに投げ込む態度ではないか。

自分で理解していないコードを出す。デバッグしていない。プロジェクトの作法を読んでいない。落ちたテストを見ていない。なのにPRだけは作れてしまう。これはAI以前にも嫌われる行為だった。ただ、AIによって量と速度が上がった。

ここで問題になるのは、生成元ではなく負担の移動だ。

書いた本人が負うべき理解コスト、検証コスト、説明コストを、maintainer側へ押しつけている。AI PRが嫌われるとき、たぶんこの構図が一番きつい。

だから、チーム内でもOSSでも、AI PRには拒否基準が必要になる。

検証ログがないなら返す。issueの目的と差分が対応していないなら返す。説明できない変更が混じっているなら返す。生成されたかどうかではなく、レビュー可能かどうかで切る。

この基準を明文化しておくと、AIの利用を無意味に禁止しなくて済む。使ってもいい。ただし、理解していない差分を他人に読ませるな、というだけだ。

セキュリティに関わるAI PRでは、さらに話が厳しくなる。

モデルが良いコードを書くかどうかも大事だが、それだけでは足りない。説明不足、テスト不足、レビュー不全のほうが現実的なリスクになる。

たとえば、認証まわりの修正でAIが一見正しい差分を出したとする。テストも通った。しかし、そのテストが何を保証しているのか、攻撃面がどう変わったのか、既存の例外ケースを壊していないのかが説明されていない。これは怖い。

セキュリティ修正では、正しそうなdiffより、拒否できる運用のほうが重要になる。

「この検証がないならmergeしない」

「この影響範囲が説明されないなら受け取らない」

「この変更は人間の設計レビューを通す」

こういう線を引けることが安全になる。AIを信じるのではなく、AIの出力を断れる形にしておく。地味だが、ここを飛ばすと危ない。

AI PRの運用で一番まずいのは、何となく良さそうだから通すことだ。人間のレビューでも同じだが、AIの場合は特に、流暢な説明が判断を鈍らせる。文章がきれいでも、証拠がなければ採用しない。そのくらいでちょうどいい。

## agent-readyなチームは、PRの型を持っている

AIエージェントをチームに入れるとき、最初に整えるべきものはプロンプト集ではないと思う。

PRの型である。

どんなissueならagentに渡してよいのか。どの情報がなければ渡さないのか。PR descriptionに何を残すのか。AI reviewに何を見せるのか。人間はどこを見るのか。どの条件なら即座に差し戻すのか。最後に誰がmerge責任を持つのか。

この型があると、AIは急に現実的な道具になる。

逆に、型がないままAIを入れると、レビュー不能なPRが増える。コードは増える。差分も増える。だが、判断の痕跡が薄い。結局、人間があとから全部追うことになる。

それは自動化ではない。レビュー負債の生成である。

AIエージェント時代のPRレビューでは、diffを読む力だけでは足りない。diffに至るまでの判断を、PR上に残させる力が必要になる。

良いAI PRとは、AIがうまく書いたPRではない。

人間が責任を持って、通すか断るかを判断できるPRである。

これからの開発チームに必要なのは、「AIにPRを作らせる技術」ではなく、「そのPRを人間が責任を持って断れる形にする技術」だ。
