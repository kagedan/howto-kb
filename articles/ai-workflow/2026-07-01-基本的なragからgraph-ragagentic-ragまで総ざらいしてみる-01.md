## はじめに

モデルが学習できていない情報をモデルに与え、有効活用する手法として検索技術を利用したRAG（Retrieval-Augmented Generation; 検索拡張生成）があります。

このRAGの技術は2020年5月に現Metaの研究者らによって基礎概念が定義され、LLMが世に普及した2023年以降にAI活用の話題の場において主役の座に躍り出ました。

2026年現在、話題の中心の座を他のトピックに譲り、やや寄せられる衆目は落ち着いたように見えますが、活用の実態は寧ろ進んでおり、ユーザーが意識せずとも裏側でRAGやRAGに類する処理が走っているケースがほとんどかと思います。

本記事では、RAGが注目を集め始めた2023年頃から現在に至るまでのRAGに関するトピックを総ざらいし、RAGの各手法の使い分けについて分析したいと思います。

## RAGとは

RAG（Retrieval-Augmented Generation）とは、AIが回答を作成する際に、あらかじめ学習した情報だけでなく、外部のデータベースや社内文書を検索して取得し、それを基にしてテキストを生成する技術のことです。

![](https://static.zenn.studio/user-upload/deployed-images/26312659cab408929137e6bd.png?sha=ba47d762dfb3f48e6c3db6b518d551eac14aed29)

2020年に発表されたRAG原論文では、DPRという検索モデルとBARTという生成モデルを組み合わせ、検索と生成を一体で学習する形が提案されました。

現在一般的にRAGと呼ばれているものは、Embeddingモデル、検索エンジン、ベクトルデータベース、生成モデルをそれぞれ独立した部品として組み合わせる構成が多いです。つまり、RAGは単一のモデル名ではなく、外部知識を検索してLLMに渡すためのシステム設計に近いものです。

### なぜRAGが必要なのか

通常の生成AIは、過去の学習データをもとに回答するため、以下のような弱点があります。

* 最新のニュースや仕様変更に対応しづらい
* 社内文書や自社独自のルールを知らない
* 存在しない情報を自信ありげに答えてしまう

このような課題を抱えているLLMに対し、RAGでは次の流れを追加します。

1. ユーザーが質問する
2. 質問に関連する文書を検索する
3. 検索した文書をLLMに渡す
4. LLMが渡された文書を根拠に回答する

![](https://static.zenn.studio/user-upload/deployed-images/0f416262870338d72bb29e2d.png?sha=64b939a095de5393a732e84eb9cf73c32a511195)

その結果、RAGには次のようなメリットがあります。

* 事実に基づいた回答に近づけられる
* データベースを更新するだけで新しい情報を反映しやすい
* どの文書を根拠にしたかを示しやすい

既存モデルに対し、ドメイン知識を用いてFine-Tuningすることでも一定の精度向上は期待できます。ただし、追加学習や再学習には時間と手間がかかります。特に、社内文書や製品仕様のように知識が継続的に更新される場合、個人や小さなチームがFine-Tuningを都度行うのは現実的ではありません。

![](https://static.zenn.studio/user-upload/deployed-images/35cc9ce1f6e8dd652de21de5.png?sha=a5a23bd2a193e02c68d608bbbccd66503f8e816f)

### RAGのパーツ

RAGは複数の独立したパーツを組み合わせて実現する技術です。そのため、精度を上げるときには、どの部分がボトルネックになっているかを見極める必要があります。

チューニングのポイントとなる構成要素は、ざっくり次のように分けられます。

| ステップ | 構成要素 |
| --- | --- |
| データ準備/インデックス化 | ナレッジベース |
| データ準備/インデックス化 | チャンキング |
| データ準備/インデックス化 | 埋め込みモデル |
| データ準備/インデックス化 | ベクトルデータベース |
| 検索 | クエリ変換 |
| 検索 | 検索エンジン |
| 検索 | リランキング |
| 生成 | プロンプト構築 |
| 生成 | LLM |

## RAGの構成要素

### ナレッジベース

ナレッジベースは、RAGが検索する外部データの置き場です。社内マニュアル、PDF、Webサイト、議事録、仕様書、FAQ、ソースコードのドキュメントなどが対象になります。

ここで重要なのは、文書をただ集めるだけでは不十分だという点です。ファイル名、タイトル、更新日、作成者、カテゴリ、権限、出典URLのような情報がないと、検索した文書が本当に使ってよいものなのか判断しづらくなります。

例えば、同じ仕様書でも最新版と旧版が混ざっていれば、RAGは古い情報を根拠に回答してしまう可能性があります。社内向けの文書であれば、ユーザーの権限によって検索してよい範囲も変わります。ナレッジベースは、文書を保存する場所というより、検索に使える状態へ整えた知識の台帳として考える方が近いです。

### チャンキング

チャンキングは、長い文書を検索しやすい単位に分割する処理です。

LLMには一度に渡せる文量の上限があります。そのため、PDFや長いMarkdownをそのまま渡すのではなく、見出しや段落ごとに小さく分けて検索対象にします。

ただし、単純に500文字ごとに切ればよいわけではありません。文の途中で切ると意味が壊れます。仕様書、契約書、社内規程、研究論文のような文書では、章や節の構造を保ったまま分割することが重要です。→**ドキュメント構造解析**

チャンクが大きすぎると、不要な情報が混ざって回答の精度が落ちます。小さすぎると、前後の文脈が足りず、意味を取り違えやすくなります。RAGの精度調整では、このサイズと構造の設計がかなり効いてきます。

また、同じ仕様書の最新版と旧版を識別したりフィルタリングにかけたりするための次のメタデータ設計・付与がよく用いられます。→**メタデータ設計・付与**

#### メタデータ設計・付与

チャンクには、本文だけでなくメタデータも付けます。メタデータは、検索結果を絞り込んだり、回答時に出典を示したり、古い情報を避けたりするために使います。

例えば、社内規程を検索するRAGであれば、部署、公開範囲、更新日、版数がないと危険です。古い規程を根拠に回答してしまったり、本来見せてはいけない文書を検索対象にしてしまったりするためです。

代表的なメタデータは次のようなものです。

| メタデータ | 用途 |
| --- | --- |
| document\_id | 元文書とチャンクを紐付ける |
| title | 検索結果や引用時の表示に使う |
| source\_url | 参照元や原文確認に使う |
| created\_at / updated\_at | 情報の新旧を判断する |
| author / department | 文書の責任者や所属を示す |
| category | 検索対象を分野で絞る |
| access\_control | ユーザー権限に応じて検索範囲を制御する |
| page / section | 引用箇所を特定する |
| version | 古い文書の混入を避ける |

また、元文書に存在するメタデータだけでは足りない場合もあります。その場合は、チャンクごとの要約、想定質問、重要キーワードなどを追加で作ることがあります。こうした情報を足しておくと、質問文と本文の言い回しがずれている場合でも検索しやすくなります。

#### ドキュメント構造解析

ドキュメント構造解析は、文書の見出し、章、節、表、コード、図表キャプションなどを見て、どこでチャンクを切るべきかを決める処理です。

単純な文字数分割だけだと、見出しと本文が分かれたり、表の途中で切れたり、コードブロックが壊れたりします。特に仕様書、契約書、研究論文、APIドキュメントでは、構造そのものが意味を持つため、本文の並びを保つことが重要です。

構造解析で見るポイントは次のようになります。

| 構成要素 | 見るポイント |
| --- | --- |
| 見出し | どの章や節に属する内容か |
| 階層構造 | H1 / H2 / H3 の親子関係を保てているか |
| 親子チャンク | 小チャンク検索後に親セクションを戻せるか |
| テーブル | 行や列の意味を壊さずに扱えるか |
| コードブロック | 関数や設定例を途中で切っていないか |
| 画像・図表キャプション | 図の意味を検索可能なテキストにできているか |

実務では、小さなチャンクを検索に使い、回答時には親セクションや前後の段落を一緒に渡す設計がよく使われます。検索は細かく、回答は文脈を広く、という分担にすると、取りこぼしと文脈不足の両方を減らしやすくなります。

### 埋め込みモデル

埋め込みモデルは、文章をベクトルと呼ばれる数値の配列に変換するモデルです。ベクトルにすることで、文章同士の意味的な近さを計算できるようになります。

例えば、「ログインできない」と「認証に失敗する」は文字列としては違いますが、意味は近いです。埋め込みモデルを使うと、このような言い換えにもある程度対応できます。

埋め込みモデルを選ぶときは、精度だけでなく、扱う言語、コスト、速度、運用方法を見ます。日本語中心なら日本語に強いモデルを選ぶ価値があります。多言語の文書を扱うなら、多言語モデルの方が扱いやすいです。社外APIを使えない環境なら、ローカルで動かせるモデルが候補になります。

| 観点 | 具体例 | 選び方 |
| --- | --- | --- |
| 日本語中心 | ruri-v3-310m | 日本語に強いモデルを優先する |
| 多言語文書 | bge-m3、Gemini embedding-001 | 多言語対応モデルを使う |
| 精度優先 | OpenAI text-embedding-3-large、Gemini embedding-001 | 大きめのAPIモデルも検討する |
| コスト優先 | ruri-v3-310m、bge-m3 | ローカルモデルや小型モデルを検討する |
| 固有名詞が多い | bge-m3 sparse、BM25 | キーワード検索との併用を考える |

### ベクトルデータベース

ベクトルデータベースは、埋め込みモデルで作ったベクトルと元のテキストを保存し、質問に近い文書を高速に探すためのデータベースです。

有名なものには、Qdrant、Milvus、Weaviate、Chroma、Elasticsearch、OpenSearch、pgvector、MongoDB Atlas Vector Searchなどがあります。

| 用途 | 具体例 | 候補の考え方 |
| --- | --- | --- |
| 小規模な検証 | Chroma、LanceDB | ローカルの軽量構成で十分なことが多い |
| 既存DBと統合 | pgvector、CockroachDB | PostgreSQL系の環境に統合しやすい |
| 大規模検索 | Milvus、Qdrant、Weaviate、OpenSearch | データ量、更新頻度、分散構成を見て選ぶ |
| 運用を軽くしたい | MongoDB Atlas Vector Search、Weaviate Cloud、Qdrant Cloud | マネージドサービスを優先する |
| 権限や監査が重要 | OpenSearch、MongoDB Atlas、CockroachDB | メタデータフィルタやアクセス制御を重視する |

重要なのは、どのDBが最強かではありません。データ量、更新頻度、権限管理、既存システムとの接続、チームの運用能力に合っているかです。

### クエリ変換

クエリ変換は、ユーザーの質問を検索しやすい形に整える処理です。

ユーザーの質問は、検索クエリとしてそのまま使いやすいとは限りません。曖昧だったり、長すぎたり、逆に短すぎたりします。例えば「最近の変更を教えて」と言われても、最近がいつからなのか、どの製品の変更なのかが分からない場合があります。

そこで、RAGでは検索前に次のような処理を入れることがあります。

| 処理 | 目的 |
| --- | --- |
| 意図分類 | 質問、要約、比較、手順確認などを判定する |
| クエリリライト | 検索しやすい表現に言い換える |
| クエリ分解 | 複雑な質問を複数の小さな質問に分ける |
| HyDE | 仮想回答を作って検索に使う |
| Multi-query expansion | 複数の観点から検索する |
| Temporal resolution | 最近、去年、最新版などを具体化する |
| Entity extraction | 製品名、人物名、組織名、日付を抽出する |

この工程を入れると、単純なベクトル検索だけでは拾いにくい情報に届きやすくなります。

### 検索エンジン

検索エンジンは、質問に関連する文書を見つける部分です。

RAGというとベクトル検索の印象が強いですが、実務ではベクトル検索だけに頼ると失敗することがあります。固有名詞、型番、日付、エラーコード、法律名のような情報は、キーワード検索の方が強い場面があります。

| 検索方式 | 向いているもの |
| --- | --- |
| ベクトル検索 | 言い換えや意味の近さを拾う |
| キーワード検索 | 固有名詞、型番、日付、エラーコードを拾う |
| ハイブリッド検索 | ベクトル検索とキーワード検索を組み合わせる |
| メタデータフィルタ | 部署、日付、カテゴリ、権限で絞る |
| Graph検索 | エンティティや関係性を辿る |
| Multi-query検索 | 複数の検索クエリに展開する |
| Parent document retrieval | 小チャンクで探し、親文書を回答に使う |

最初から複雑な検索器を作る必要はありません。まずはキーワード検索とベクトル検索を組み合わせ、検索結果を見ながら失敗パターンを把握するのが現実的です。

### リランキング

リランキングは、検索で集めた候補をさらに並べ替える処理です。

検索エンジンが返す上位候補には、質問に似ているだけで回答には使いづらい文書が混ざることがあります。リランキングを入れると、候補文書を質問との関係で再評価し、回答に使いやすい順に並べ替えられます。

リランキングが効きやすいのは、検索候補を広めに取る場合です。まず候補を多めに集め、最後にリランカーで絞り込むことで、取りこぼしとノイズのバランスを取りやすくなります。

リランキングの考え方は、いくつかの方向に分けられます。

| 手法 | 何をするか | 向いている場面 |
| --- | --- | --- |
| クロスエンコーダ型 | 質問と候補文書をまとめて読ませ、関連度を再評価する | 精度を優先したい検索 |
| LLM rerank | LLMに候補を読ませ、回答に使う順番を判断させる | 少数候補を丁寧に評価したい場合 |
| RRF | 複数の検索結果順位を統合する | ベクトル検索とキーワード検索を併用する場合 |
| 適応型再ランキング | クエリの難しさに応じて再ランキング対象数を変える | コストと精度のバランスを取りたい場合 |
| 融合型再ランキング | 複数のクエリやサブクエリの検索結果をまとめ直す | 複雑な質問や調査系の質問 |

クロスエンコーダ型は、候補文書を一つずつ深く評価できるため精度を出しやすい一方、候補数が多いと遅くなります。最初の検索で上位50件ほどを集め、最後に10件程度へ絞るような使い方が現実的です。

RRFは、Reciprocal Rank Fusionの略で、複数の検索結果を順位ベースで統合する方法です。ベクトル検索では拾えるがキーワード検索では拾えない文書、逆にキーワード検索では拾えるがベクトル検索では弱い文書を合わせて扱いやすくなります。

LLM rerankは柔軟ですが、コストと遅延が大きくなりやすいです。そのため、すべての候補に使うより、検索結果をある程度絞った後の最終判断に使う方が扱いやすいです。

## RAGの代表的な手法

### 基本的なRAG

基本的なRAGは、文書をチャンクに分け、ベクトル化し、質問に近いチャンクを検索し、その結果をLLMに渡す構成です。

流れはシンプルです。

* 文書を集める
* 文書をチャンクに分ける
* チャンクをベクトル化する
* 質問に近いチャンクを検索する
* 検索結果をプロンプトに入れる
* LLMが回答する

この構成は導入しやすく、FAQ、社内文書検索、仕様確認のような用途では十分に役立ちます。

一方で、検索で正しい文書を拾えない場合や、拾った文書がノイズだらけの場合、回答品質はすぐに落ちます。基本的なRAGの弱点は、LLMそのものというより、検索対象の整備、チャンク設計、検索クエリ、ランキングに出やすいです。

### Advanced RAGとModular RAG

Advanced RAGは、基本的なRAGの検索品質を高めるための改善を加えた構成です。クエリ変換、メタデータフィルタ、リランキング、細かいチャンキング、親子チャンクなどがここに入ります。

![](https://res.cloudinary.com/zenn/image/fetch/s--ndvTrKdF--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/advanced_rag.png?_a=BACMTiAE)

Modular RAGは、RAGを一つの固定パイプラインではなく、複数の部品を組み替えられる構成として捉えます。検索器を複数持つ、質問によって検索先を切り替える、外部APIやWeb検索を組み込む、生成後に検証する、といった作り方です。

![](https://res.cloudinary.com/zenn/image/fetch/s--KZ-fTiaL--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/modular_rag.png?_a=BACMTiAE)

ざっくり言うと、Advanced RAGは検索品質を上げるための改善、Modular RAGは用途に応じてRAGの部品を組み替える考え方です。

### Graph RAG

Graph RAGは、文書をテキスト断片として検索するだけでなく、人、組織、製品、技術、出来事などの関係性も検索対象にする手法です。

![](https://res.cloudinary.com/zenn/image/fetch/s--icOrTEI_--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.00309v2/extracted/6113564/Figure/rag-grag.png?_a=BACMTiAE)

従来のRAGでは、質問に近いチャンクを探します。Graph RAGでは、情報をノードとエッジとして扱い、エンティティ同士の関係を辿れるようにします。

![](https://res.cloudinary.com/zenn/image/fetch/s--gY5DS8Ka--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/graph_rag.png?_a=BACMTiAE)

例えば、ある薬剤がどの疾患に使われ、どの遺伝子やタンパク質に作用するのかを調べる場合、単なる文章検索よりも関係性を辿る方が答えに近づきやすいことがあります。

![](https://res.cloudinary.com/zenn/image/fetch/s--Ajm2KepM--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.00309v2/extracted/6113564/Figure/framework.png?_a=BACMTiAE)

ただし、Graph RAGは普通のRAGの上位互換ではありません。エンティティ抽出、名寄せ、関係抽出、グラフ更新の運用が必要になります。関係性を辿る価値があるデータでなければ、コストに見合わないこともあります。

### Self-RAG

Self-RAGは、検索するかどうか、検索結果を使うべきかどうかをモデル自身に判断させる方向の手法です。

通常のRAGでは、質問が来たら常に一定数の文書を検索して回答に使います。しかし、すべての質問に検索が必要とは限りません。検索結果が質問とずれている場合、その文書を無理に使うことで回答が悪くなることもあります。

Self-RAGでは、検索の必要性、取得文書の有用性、生成した回答の妥当性を途中で評価します。これにより、不要な検索や不適切な文脈利用を減らすことを目指します。

### Corrective RAG

Corrective RAGは、検索結果が悪い場合に補正する手法です。

RAGは検索結果に強く依存します。関係の薄い文書を取ってきてしまうと、LLMはその文書を根拠にして誤った回答を作る可能性があります。

Corrective RAGでは、取得した文書の品質を評価し、低品質な場合は再検索したり、外部検索に切り替えたりします。検索結果を分解して重要な部分だけを残す設計もあります。

基本的なRAGが「検索して渡す」ものだとすると、Corrective RAGは「検索結果を疑い、必要なら直す」ものです。

### Agentic RAG

Agentic RAGは、検索計画、再検索、候補レビュー、追加調査、停止判断をAIエージェントに任せる設計です。

![](https://res.cloudinary.com/zenn/image/fetch/s--Mzj6PTTC--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/taxonomy.png?_a=BACMTiAE)

通常のRAGでは、検索の流れはあらかじめ固定されています。Agentic RAGでは、質問の内容に応じて、どの検索器を使うか、追加で何を調べるか、検索結果が足りているかをエージェントが判断します。

![](https://res.cloudinary.com/zenn/image/fetch/s--UKfokJlf--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/ai_agents.png?_a=BACMTiAE)

Agentic RAGの主な要素は次のように整理できます。

| 要素 | 役割 |
| --- | --- |
| LLM | 質問を解釈し、次の行動を決める |
| 記憶 | 会話や過去の作業文脈を保持する |
| プランニング | 複雑な質問を分解し、調査順を決める |
| ツール | ベクトル検索、Web検索、API、DB検索を実行する |
| 評価 | 検索結果や回答の妥当性を確認する |

Agentic RAGには、いくつかの構成パターンがあります。

| パターン | 概要 | 向いている場面 |
| --- | --- | --- |
| 単一エージェント型 | 一つのエージェントが検索先選択から回答まで担当する | 小規模な社内検索、個人用アシスタント |
| マルチエージェント型 | 複数の専門エージェントが分担する | 調査、分析、複数データソースの統合 |
| 階層/グラフ型 | 上位エージェントが下位エージェントを管理する | 複雑な業務ワークフロー |

![](https://res.cloudinary.com/zenn/image/fetch/s--fgFJ_nnf--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/single_agentic_RAG.png?_a=BACMTiAE)

![](https://res.cloudinary.com/zenn/image/fetch/s--ZCIPgfKb--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/multiagent_agentic_rag.png?_a=BACMTiAE)

![](https://res.cloudinary.com/zenn/image/fetch/s--gT37dO7z--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://arxiv.org/html/2501.09136v4/images/hierarchical_agentic_rag.png?_a=BACMTiAE)

Agentic RAGは、複雑な調査や業務自動化に向いています。一方で、コスト、遅延、制御の難しさも増えます。すべての質問にAgentic RAGを使うのではなく、通常のRAGで足りない質問だけ深い探索に切り替える方が現実的です。

## どの手法をいつ使うか

RAGの発展形は多いですが、名前だけで選ぶと失敗しやすいです。まず、どの失敗を直したいのかを見ます。

| 課題 | 検討する手法 |
| --- | --- |
| 最新情報や社内文書を回答に使いたい | 基本的なRAG |
| 正しい文書が検索で出てこない | Advanced RAG |
| 固有名詞や日付を取りこぼす | ハイブリッド検索 |
| 文書の関係性や全体像を知りたい | Graph RAG |
| 検索が必要か判断したい | Self-RAG |
| 検索結果が間違っている可能性が高い | Corrective RAG |
| 複数ステップの調査やツール利用が必要 | Agentic RAG |

最初からGraph RAGやAgentic RAGを作る必要はありません。まずはシンプルなRAGで失敗を観測し、検索精度、関係推論、自己評価、再検索、ツール連携のどこを足すべきかを決めるのがよいです。

## まとめ

RAGは、LLMに外部知識を与える仕組みとして説明されることが多いですが、実際には単なるベクトル検索とプロンプト拡張だけで完結するものではありません。

ナレッジベースの整備、チャンキング、メタデータ設計、埋め込みモデル、検索エンジン、リランキング、プロンプト構築、生成モデルといった複数の構成要素が連動してはじめて、実用的な品質になります。

特に重要なのは、RAGの失敗が必ずしもLLMそのものの能力不足ではないという点です。回答が不正確な場合、その原因は検索対象のデータ品質、チャンク設計、クエリ変換、検索方式、リランキング、プロンプト設計にあるかもしれません。

基本的なRAGは導入しやすく、FAQや社内文書検索のような用途では十分に有効です。一方で、検索精度や文脈理解に課題がある場合にはAdvanced RAG、関係性や全体像を扱う場合にはGraph RAG、検索結果を評価したい場合にはSelf-RAGやCorrective RAG、複数ステップの調査が必要な場合にはAgentic RAGが候補になります。

ただし、高度な手法は常に導入すればよいものではありません。Graph RAGやAgentic RAGは強力ですが、設計、実装、運用の複雑性や計算コストも大きくなります。

最終的に、RAGはLLMの知識不足を補う技術というより、外部知識をどのように収集し、構造化し、検索し、評価し、生成に接続するかという設計思想に近いです。最新情報、社内知識、専門知識、根拠付き回答を扱う必要がある限り、RAGおよびその発展形は生成AIシステムにおける重要な基盤技術であり続けると思います。
