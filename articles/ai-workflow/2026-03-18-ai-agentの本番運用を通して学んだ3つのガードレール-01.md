## はじめに：AI Agentは「動かす」より「安定させる」が難しい

こんにちは、ログラスでエンジニアしている山崎です。私は2025年11月に入社して以来、AI Agentの新規事業の開発をしています。

私たちはLangGraphでAI Agentを構築し、経営の実績データに対する分析コメント（以下、財務分析コメント）をLLMに生成させる機能を開発・運用しています。なお、本機能で利用するLLM APIは、入力データがモデルの学習に利用されないエンタープライズ向けのセキュアな環境で運用しています。

プロトタイプの段階では順調に動いていたのですが、本番環境に投入した途端、3つの壁が立ちはだかりました。

1. **100件のバッチリクエストを実行した際、ThrottlingExceptionが発生し処理が完了しなかった**
2. **生成された分析コメントにプロダクトの出力要件を満たさない表現が含まれていた**
3. **プロンプトを修正するたびにコードデプロイが必要で、改善サイクルが回らなかった**

1つ目はAPI基盤の安定性、2つ目は出力品質の保証、3つ目は改善サイクルの高速化と、それぞれ異なるレイヤーの課題です。共通しているのは、「LLMを本番環境で安定的に運用するためのガードレールが必要だった」という点です。

本記事では、私たちがFP&A（財務計画・分析）向けAI Agentの本番運用で実装した3つのガードレールを紹介します。

決定論的な仕組みと非決定論的なLLMの役割分担については、弊社の田村が「[万能なエージェントより、確実なワークフローを —自律性と信頼性のトレードオフ](https://zenn.dev/loglass/articles/08a238976938a9)」で詳しく論じています。本記事の論点2（出力品質保証）はこの設計思想を直接実装に落とし込んだものであり、論点1・論点3もLLMを本番で安定運用するために欠かせないガードレールです。

### システム構成

私たちはLangGraphベースのAI Agentを開発・運用しています。クライアントからFastAPIに対してリクエストを送り、LangGraphのワークフローが複数ノードでLLM APIを呼び出す構成です。プロンプトの管理にはLangfuseを使用しています。

---

## 3つのガードレールの全体像

3つの課題は、AI Agentシステムの異なるレイヤーに位置しています。

| レイヤー | 課題 | ガードレール |
| --- | --- | --- |
| **入口（API呼び出し）** | TPM制限超過による処理エラーの発生 | asyncio.Semaphore + 段階的チューニング |
| **出口（LLM出力）** | 出力要件を満たさない表現の発生 | 正規表現バリデーション + 条件付きエッジリトライ |
| **運用（プロンプト改善）** | プロンプト修正にデプロイが必要 | Langfuse Prompt Managementによる外部管理 |

これらの論点について見ていきましょう。

## 論点1: LLM APIのリクエスト過多を防ぐためのスロットリング制御

### 直面した技術的課題

クライアントからLLM APIに対して100件の分析コメント生成リクエストがバッチで送信されました。LangGraphのワークフロー内の複数ノードがそれぞれLLM APIを呼び出すため、TPM（Tokens Per Minute）制限を超えて`ThrottlingException`が発生し、処理が完了しない状態になりました。

LLM APIのレート制限は、従来のREST APIとは性質が異なります。制約が「リクエスト数」ではなく「トークン数（TPM）」ベースであるため、同時実行数を抑えてもトークン消費量次第では制限に抵触します。LangGraphのノードはワークフロー内で逐次実行されるため、5ワークフローが並列に動いた場合の同時リクエスト数は最大5です。しかし、各ワークフローが8ノードぶんのLLM API呼び出しを次々に行うため、短時間に大量のトークン消費が集中し、TPM上限に到達していました。

### トークン消費量を起点としたチューニング

この課題に対して、過去のトークン消費量のログを分析し、ワークフロー1回あたりの平均トークン消費量とTPM上限から逆算して、同時実行数とリトライ間隔を決定しました。

#### ステップ1: asyncio.Semaphoreで同時実行数を制御

`asyncio.Semaphore`で同時実行数に上限を設けました。

```
# バッチリクエストの同時実行数を制御し、LLM APIへの負荷を平準化する
# agent_workflow: LangGraphで構築済みのCompiledGraphインスタンス（クラス外で初期化済み）
async def generate_results(items: list[dict]) -> AsyncIterator[str]:
    # 同時実行数の上限（TPM上限から逆算した値）
    semaphore = asyncio.Semaphore(5)

    # クライアント切断時に未実行タスクを停止するためのイベント
    cancelled = asyncio.Event()

    # 各アイテムをSemaphore制御下でLangGraphワークフローに投入する
    async def run_task_with_index(index: int, item: dict) -> tuple[int, dict]:
        # チェックポイント1: Semaphore取得前にキャンセル確認
        if cancelled.is_set():
            return index, {"skipped": True, "output": None}

        async with semaphore:
            # チェックポイント2: Semaphore取得後にキャンセル確認
            if cancelled.is_set():
                return index, {"skipped": True, "output": None}

            # LangGraphワークフローを実行し、分析コメントを生成
            result = await agent_workflow.ainvoke(item)
            return index, result
```

Semaphore取得の**前後2箇所**でキャンセルチェックを入れています。取得前チェックでキューに並んでいるタスクを即キャンセルし、取得後チェックでAPI呼び出し直前にもクライアント切断を検知します。これによりAPIトークンの浪費を防止しています。

#### ステップ2: トークン消費量に基づくリトライ間隔と同時実行数の調整

ステップ1の導入後もTPM超過が解消しませんでした。過去のトークン消費ログを分析したところ、1ワークフローあたりの平均トークン消費量が想定より大きく、5並列では合計TPMが上限を超えることが判明しました。

これを踏まえて、2つのパラメータを同時に調整しました。

```
# リトライ間隔: TPMバケットの回復を待つのに十分な値に設定
retry_base_delay: float = 10.0   # 指数バックオフの基底（10秒→20秒→40秒）

# 同時実行数: ワークフロー1回あたりの平均トークン消費量 × 並列数 < TPM上限 を満たす値
max_concurrent: int = 3

semaphore = asyncio.Semaphore(max_concurrent)

async def run_task_with_index(index: int, item: dict) -> tuple[int, dict]:
    async with semaphore:
        for attempt in range(3):
            try:
                result = await agent_workflow.ainvoke(item)
                return index, result
            except ThrottlingException:
                if attempt == 2:
                    raise
                delay = retry_base_delay * (2 ** attempt)  # 10秒→20秒→40秒
                await asyncio.sleep(delay)
```

* **リトライ間隔の延長**: TPM制限はトークン単位のバケットで管理されているため、短いリトライ間隔ではバケットが回復しません。10秒を基底とした指数バックオフに変更しました
* **同時実行数の削減**: 1ワークフロー内の複数ノードが合計で消費するトークン量を見積もり、3並列でTPM上限に収まることを確認しました

この変更によってバッチ全体の処理時間は伸びますが、私たちのシステムではバッチ処理のレイテンシ要件に余裕があったため、安定性を優先する判断をしました。結果としてThrottlingErrorはほぼゼロになっています。

### この論点のまとめ

| 対策 | 効果 |
| --- | --- |
| Semaphoreによる同時実行数制御 + キャンセル機構 | 同時実行数に上限を設定し、不要なAPI呼び出しを抑制 |
| トークン消費量に基づくリトライ間隔・同時実行数の調整 | ThrottlingErrorほぼゼロ |

単発のリクエストがランダムに飛んでくる環境では、TPMのレートリミットには遭遇しないかもしれません。しかし、まとまった量のリクエストが等間隔でバッチ実行されるケースでは、意外とすぐにTPM上限に到達します。規則的なトークン消費パターンが発生するバッチ処理を実装した後は、ワークフロー1回あたりのトークン消費量の計測とモニタリングが不可欠であるという基本を改めて実感しました。

## 論点2: LangGraphの条件付きエッジによる出力品質保証 ── 禁止ワード検出と自己修正リトライ

### 直面した技術的課題

LLMが生成した分析コメントに、プロダクトの出力要件を満たさない表現が含まれるケースが確認されました。

* 「ですます調」 → 報告書は「である調」に統一すべき
* 「赤字」「黒字」 → 「予算未達」「予算達成」と表現するというプロダクトの品質基準への統一

プロンプトで「赤字・黒字という表現は使わないでください」と指示しても、LLMの非決定的な性質上、指示から逸脱してしまうことがあります。実際に100件の分析コメントを生成する検証では、数件に出力要件外の表現が含まれるケースが確認されました。

このように、非決定論的なプロンプトは「お願いベースの指示」として扱い、決定論的なコードは「明確な契約」として振る舞わせるという役割分担を行っています。

### アプローチ: 生成→検証→再生成のループ

LangGraphの`conditional_edges`を使い、バリデーション違反を検出したらループバックして再生成する仕組みを構築しました。

#### 正規表現バリデーション

検証はLLMではなく、正規表現で行います。禁止ワードの検出は「指定パターンに一致するか否か」という決定論的な判定であり、LLMの得意領域ではありません。一方、代替表現を使って文章を再構成するのはLLMが得意とする非決定論的なタスクです。**検出は決定論的なコードで、再生成は非決定論的なLLMで**という、それぞれの特性に合った役割分担をしています。

```
# 生成された分析コメントに禁止ワードが含まれていないか検証する
def validate_banned_words(comment: str) -> list[str]:
    """禁止ワードを検証して検出されたワードのリストを返す"""
    banned_patterns = {
        "赤字・黒字": r"赤字|赤字予算|赤字幅|赤字化|黒字|黒字化|黒字転換|黒字予算",
        "ですます調": r"です(?![a-zA-Zぁ-ん])|ます(?![a-zA-Zぁ-ん])"
                     r"|でした(?![a-zA-Zぁ-ん])|ました(?![a-zA-Zぁ-ん])|でしょう",
    } # 「ますます」を弾いてしまうなどの偽陽性判定は一定許容

    found = []
    for category, pattern in banned_patterns.items():
        matches = re.findall(pattern, comment)
        if matches:
            unique_matches = list(set(matches))
            found.append(f"{category}: {', '.join(unique_matches)}")
    return found
```

バリデーションノード（`validate_comment_node`）は、上記の検証結果をLangGraphのグラフ状態に書き込みます。

```
# バリデーションノード: 検証結果をグラフ状態に書き込み、後続の条件分岐で参照させる
def validate_comment_node(state: SummaryState) -> dict:
    comment = state.get("generated_comment", "")
    found = validate_banned_words(comment)
    return {
        "banned_words_detected": found,
        "validation_attempts": state.get("validation_attempts", 0) + 1,
    }
```

#### 条件分岐ルーティング

検証結果に応じて、次のノードへ進むかループバックするかを決定します。リトライ回数に上限を設けているため、無限ループは発生しません。

```
# バリデーション結果に基づいて再生成するかどうかを判定する
def should_retry_generation(state: SummaryState) -> str:
    # LangGraphが管理するグラフ状態から、前段のバリデーションノードが書き込んだ結果を取得
    banned_words = state.get("banned_words_detected")
    validation_attempts = state.get("validation_attempts", 0)
    max_attempts = state.get("max_validation_attempts")

    if not banned_words:
        return "end"           # 禁止ワードなし → 終了

    if validation_attempts >= max_attempts:
        return "end"           # リトライ上限到達 → ベストエフォートで終了

    return "generate_comment"  # リトライ → 再生成
```

リトライ時には、検出された禁止ワードとその代替表現をプロンプトに追加します。

```
# リトライ時に、前回検出された禁止ワードと代替表現をプロンプトに付加する
def add_banned_word_warning(prompt: str, banned_words: list[str]) -> str:
    warning = "\n\n## 重要: 前回の生成で禁止ワードが検出されました\n\n"
    for word in banned_words:
        warning += f"- {word}\n"
    warning += """
**必ず禁止ワードを使用せずに、代替表現で再生成してください。**

禁止ワードの代替表現:
- 「赤字」「黒字」 → 「予算未達」「予算達成」「予算を上回って着地」
- 「ですます調」 → 「である調」に変更
"""
    return prompt + warning
```

LangGraphの`add_conditional_edges`で、上記ルーティングを宣言的に定義します。

```
# LangGraphのグラフにバリデーション→条件分岐のフローを組み込む
builder.add_node("validate_comment", validate_comment_node)
builder.add_edge("generate_comment", "validate_comment")
builder.add_conditional_edges(
    "validate_comment",
    should_retry_generation,
    {
        "generate_comment": "generate_comment",  # リトライ
        "end": END,                               # 終了
    },
)
```

### この論点のまとめ

正規表現バリデーション + 条件付きエッジリトライで、出力要件外の表現の発生率を実質ゼロにできました。ここで重要なのは、「LLMに自己修正を依頼する」のではなく、**決定論的なコードで違反を検出し、非決定論的なLLMに再挑戦させる**という構造です。LangGraphの条件付きエッジはこの「検証→再試行」パターンを宣言的に書ける点で非常に相性がよいです。

## 論点3: Langfuse Prompt Managementによるプロンプトの外部管理 ── コードデプロイなしでプロンプトを更新する

### 直面した技術的課題

論点2で禁止ワードバリデーションを入れたことで、出力品質の検出精度は向上しました。しかしバリデーションはあくまで「最後の砦」であり、**そもそも1回目の生成で正しい出力が出る確率を上げる**ことが理想です。

そのためにはプロンプトの継続的な改善が欠かせません。しかし、プロンプトがソースコードにハードコードされていたため、ちょっとした文言修正でもPR→レビュー→マージ→デプロイのサイクルが必要でした。AI Agentの出力品質はプロンプトに強く依存するため、この改善サイクルの遅さがボトルネックになっていました。

### プロンプトのリリースサイクルをコードのリリースサイクルから分離する

この課題に対して、Langfuse Prompt Managementを導入しました。Langfuseの管理画面上でプロンプトを編集・公開すれば、コードデプロイなしにAgentの振る舞いを更新できます。

導入にあたっての設計上の関心事は「Langfuseが落ちたときにサービスが止まらないこと」でした。外部サービスへの依存を追加する以上、障害時の可用性を担保する仕組みが必要です。

### 3層のプロンプト管理アーキテクチャ

以下の3層構成でプロンプト管理を実装しました。

#### 1. Registry: プロンプトキーとLangfuseプロンプト名のマッピング

```
# アプリケーション内のプロンプトキーとLangfuse上のプロンプト名を対応付ける
PromptKey = Literal[
    "system_instruction",
    "report_generation",
    "summary_extraction",
    # ... 用途ごとにプロンプトを定義
]

PROMPT_REGISTRY: dict[PromptKey, str] = {
    "system_instruction": "my-agent-system-instruction",
    "report_generation": "my-agent-report-generation-v2",
    "summary_extraction": "my-agent-summary-extraction",
    # ...
}
```

コード上では`PromptKey`（Literal型）でプロンプトを参照します。Langfuse側のプロンプト名が変わっても、Registryのマッピングを変更するだけで対応できます。

#### 2. Provider: Langfuse SDKとの連携と変数展開

```
# Langfuseからプロンプトを取得し、変数展開とフォールバックを担うプロバイダ
class PromptProvider:
    def __init__(self, langfuse_client: Langfuse):
        self._client = langfuse_client
        self._registry = PROMPT_REGISTRY

    def get_prompt(self, key: PromptKey, variables: dict[str, str] | None = None) -> str:
        langfuse_name = self._registry[key]
        try:
            # Langfuseからプロンプトを取得（キャッシュあり）
            prompt = self._client.get_prompt(langfuse_name)
            compiled = prompt.compile(**variables) if variables else prompt.compile()
            return compiled
        except Exception:
            # Langfuse障害時はフォールバック
            return self._get_fallback(key, variables)
```

Langfuse SDKはMustache記法（`{{variable}}`）による変数展開をサポートしています。プロンプト内に`{{target_name}}`のようなプレースホルダを記述しておけば、実行時に実際の値に置換されます。

#### 3. Fallback Prompts: Langfuse障害時の静的プロンプト

```
# Langfuseにアクセスできない場合に使用するフォールバックプロンプト
# ※以下は説明用のサンプルです。実際のプロダクションでは業務要件に合わせたプロンプトを定義しています
FALLBACK_PROMPTS: dict[PromptKey, str] = {
    "system_instruction": """（※ここにシステムプロンプトを記述します）""",
    "report_generation": """（※ここにレポート生成用プロンプトを記述します）""",
    # ...
}
```

Langfuseがダウンしても、ソースコードに埋め込まれたフォールバックプロンプトで処理が継続されます。フォールバックプロンプトは、Langfuse上のプロンプトが安定したバージョンに達するたびに同期しておきます。

### キャッシュ戦略: Stale-While-Revalidate

Langfuse SDKにはStale-While-Revalidateのキャッシュ戦略が組み込まれています。キャッシュが期限切れになっても、古いキャッシュを即座に返しつつバックグラウンドで最新版を取得します。これにより、プロンプト取得のレイテンシがリクエスト処理に影響しません。

さらに、サービス起動時に全プロンプトをプリフェッチしています。

```
# サービス起動時に全プロンプトをキャッシュにロードし、コールドスタートを回避する
async def prefetch_all(self) -> None:
    for key in PROMPT_REGISTRY:
        try:
            langfuse_name = self._registry[key]
            self._client.get_prompt(langfuse_name)
        except Exception as e:
            logger.warning(f"Prefetch failed for {key}: {e}")
```

最初のリクエストが来る前にキャッシュウォーミングが完了するため、コールドスタート時のレイテンシ増加を回避しています。

### 論点2との接続: プロンプト改善がバリデーション通過率を上げる

論点2の禁止ワードバリデーションは「出力を検知して再生成する」事後対策です。一方、プロンプト改善は「そもそも1回目で正しい出力が出る確率を上げる」事前対策です。

Langfuse導入後のワークフローは以下のようになりました。

1. **バリデーションで禁止ワードを検出**（論点2）
2. **Langfuseのトレースで、どのプロンプトがどの禁止ワードを誘発しやすいか分析**
3. **Langfuse上でプロンプトを修正・公開**（デプロイ不要）
4. **1回目の生成品質が向上し、リトライ率が低下**

プロンプト改善のサイクルが高速に回ることで、バリデーションによるリトライが「例外処理」として機能するようになりました。

### この論点のまとめ

Langfuse Prompt Managementにより、プロンプトのリリースサイクルをコードのリリースサイクルから分離できました。3層構成（Registry / Provider / Fallback）とStale-While-Revalidateキャッシュにより、外部依存のリスクを最小化しつつ、プロンプト改善の速度を大幅に向上させています。

## 3つのガードレールによって何が変わったか

| 課題 | Before | After |
| --- | --- | --- |
| スロットリング | 100件バッチで処理エラー多発 | Semaphore + トークン消費量ベースのチューニングでThrottlingErrorほぼゼロ |
| 出力品質 | 出力要件外の表現が発生 | バリデーション + リトライで発生率ほぼゼロ |
| プロンプト改善 | 修正にデプロイ必要、改善サイクルが遅い | Langfuseで即時反映、リトライ率低下 |

3つのガードレールはそれぞれ独立して導入可能です。しかし組み合わさることで相乗効果が生まれます。スロットリング制御でAPI呼び出しが安定し、禁止ワードバリデーションで出力品質が保証され、プロンプト外部管理で改善サイクルが高速化する。さらに、プロンプト改善によってバリデーションのリトライ率が下がれば、リトライ分のAPI呼び出しも減少し、スロットリング圧力も軽減されます。

## おわりに

AI Agentを「デモ」から「プロダクション」に引き上げるには、LLMを安定的に運用するためのガードレールが不可欠です。

本記事では、スロットリング制御（入口の安定性）、出力品質保証（出口のガードレール）、プロンプト外部管理（改善サイクルの高速化）という3つの仕組みを紹介しました。これらは「LLMの力を制限する」ものではなく、「LLMが安心して力を発揮できる環境を整える」ものです。

LLMには「考える」ことに集中してもらい、安定性・品質保証・改善サイクルはコードとインフラが担う。この役割分担を設計し、実装していくことがAI Agentの本番運用品質を高めていくためには必要だと実感しています。
