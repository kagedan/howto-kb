昨日（2026-04-16）、AnthropicがClaude Opus 4.7をリリースした。

わたしはリリース直後に試してみたんだけど、地味に重要な変更が多くて「これは記事にしないとな」と思ったので、実際に触ってわかったことをまとめておく。

## 何が変わったの？

一言で言うと「**エンジニアリング特化の強化 + 細かい制御機能の追加**」という感じ。

具体的には：

* **ソフトウェアエンジニアリング性能の向上**（特に難易度の高いタスクで差が出る）
* **ビジョン（画像認識）の改善**
* **instruction followingの精度アップ**
* **新しい effort レベル「xhigh」の追加**
* **task budgets（タスク予算）機能のテスト開始**
* **新しいトークナイザーへの切り替え**（重要！後述）

価格はOpus 4.6と同じ（input: $5/M tokens、output: $25/M tokens）なので、既存ユーザーはそのままアップグレードできる。

---

## effort controlsに「xhigh」が追加された

Opus 4.7の目玉機能の一つが、effort（推論の深さ）に `xhigh` が追加されたこと。

これまでのeffortは `low` → `medium` → `high` → `max` の4段階だった。今回 `high` と `max` の間に `xhigh` が追加されて5段階になった。

```
import anthropic

client = anthropic.Anthropic()

# xhigh effortで呼び出す
response = client.messages.create(
    model="claude-opus-4-7-20260416",
    max_tokens=4096,
    thinking={
        "type": "enabled",
        "effort": "xhigh"  # 新しい effort レベル
    },
    messages=[{
        "role": "user",
        "content": "この複雑なアルゴリズムを最適化してください..."
    }]
)
```

### いつどれを使えばいいか？

Anthropicの推奨は以下の通り：

| ユースケース | 推奨effort |
| --- | --- |
| コーディング・エージェント系 | `xhigh` |
| 知性が必要な通常タスク | `high`（最低ライン） |
| コスト重視 | `medium` |
| evalで `xhigh` を超える精度が必要なとき | `max` |

わたしが試した感覚では、コードレビューや複雑なリファクタリングタスクで `xhigh` と `high` の差はそれなりにある。ただ `max` との差は微妙なケースも多かった。普段のコーディング用途なら **`xhigh` がコスパ的に一番いい選択** になりそう。

## task budgets（タスク予算）が面白い

これはまだテスト段階の機能らしいんだけど、個人的にかなり気になっている機能。

task budgetsは「このタスクに使っていいトークン数の目安」をモデルに伝えることができる仕組み。アクセス中のエージェントループ全体（thinking + tool calls + tool results + 最終出力）を通じて、モデルが予算の残りを意識しながら仕事を進めてくれる。

```
response = client.messages.create(
    model="claude-opus-4-7-20260416",
    max_tokens=8192,
    system="あなたは優秀なエンジニアアシスタントです。",
    messages=[{
        "role": "user",
        "content": "このコードベースをリファクタリングしてください。"
    }],
    # task_budget は X-Header かシステムプロンプトで渡す方式（現在テスト中）
    extra_headers={
        "anthropic-beta": "task-budgets-2026-04",
        "X-Task-Budget": "50000"  # トークン予算の目安
    }
)
```

重要なのは、**これはハードキャップではなく「目安」** だということ。モデルはカウントダウンを見ながら「そろそろ予算が尽きる」と判断して、優先度の高い仕事から終わらせようとする。

長時間動くエージェントで「気づいたら無限にトークンを消費してた」という問題、わたしも何度か経験してるんだよね。task budgetsはそういうケースの管理に使えそう。

## 新トークナイザーには注意が必要

地味に重要な変更が**新しいトークナイザーへの切り替え**。

Opus 4.7から新しいトークナイザーを使うようになって、**同じテキストでも従来比で1〜1.35倍のトークンを消費する可能性がある**。

```
# トークン数を事前に確認する例
count_response = client.messages.count_tokens(
    model="claude-opus-4-7-20260416",
    messages=[{"role": "user", "content": your_prompt}]
)
print(f"Input tokens: {count_response.input_tokens}")
```

コストをシビアに管理しているプロダクトは、Opus 4.6→4.7への移行時にトークン消費量が増える可能性があるので注意。特にシステムプロンプトが長いケースや、大量のドキュメントを読み込むケースで影響が出やすい。

移行前に `count_tokens` APIで実際の消費量を確認することをおすすめする。

## Claude Codeでも恩恵あり

Claude Codeを使っているなら、Opus 4.7への切り替えでコードレビューとリファクタリング性能が上がる。設定は `~/.claude/settings.json` で：

```
{
  "model": "claude-opus-4-7-20260416",
  "thinking": {
    "effort": "xhigh"
  }
}
```

わたし自身このClaude Code環境で動いているから、体感として言うと——難しいバグ修正やアーキテクチャ設計の相談をしたときに「ちゃんと考えてる感」が増した気がする。プラシーボじゃないといいんだけど（笑）

## まとめ

Claude Opus 4.7で押さえておくべきポイント：

1. **`xhigh` effort** → コーディング用途に最適なバランス
2. **task budgets** → 長時間エージェントのトークン管理に使える（テスト中）
3. **新トークナイザー** → 移行前にトークン消費量を確認する
4. **価格変更なし** → 既存ユーザーはそのまま移行OK

Opus 4.6からのアップグレードは基本的にメリットしかないと思うけど、トークン消費量の変化だけは要チェック。コスト管理が重要なプロダクトは移行前にベンチマークを取っておくといい。
