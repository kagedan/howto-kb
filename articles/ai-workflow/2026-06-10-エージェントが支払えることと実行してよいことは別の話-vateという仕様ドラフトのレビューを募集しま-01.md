最初に書いておくと、私はエンジニアではありません。ですが、VATEという仕様のdiscussion draftを書いていて、メンテナンスをしています。仕様もリファレンス実装もAIツールと一緒に作ってきましたが、構成と公開内容の責任は私にあります。

この記事はそのVATEの紹介と、外部レビューのお願いです。「この設計、どこが間違っているか教えてほしい」という募集です。

## エージェントが支払い始めた

ここ数ヶ月、AIエージェントとペイメントの話が一気に具体化しました。

CloudflareはAgents SDKでx402を扱えるようにして、MCPのtoolを`paidTool`として公開できる仕組みを出しています。

<https://developers.cloudflare.com/agents/tools/payments/x402/charge-for-mcp-tools/>

AWSはAmazon Bedrock AgentCore Paymentsをプレビューとして発表しました。エージェントがAPIやMCPサーバー、他のエージェントに対して自律的に支払う。ウォレット認証から支出ガバナンスまで込みです。

<https://aws.amazon.com/about-aws/whats-new/2026/04/amazon-bedrock-agentcore-payments-preview/>

GoogleのAP2はFIDO Allianceに寄贈されることが決まり、「Human Not Present」、つまり人間がその場にいない状態での自律決済まで仕様に入ってきました。

<https://blog.google/products-and-platforms/platforms/google-pay/agent-payments-protocol-fido-alliance/>

エージェントが有料リソースに当たると、サーバーが`402 Payment Required`を返し、エージェントが支払いを付けて再試行し、結果とレシートを受け取る。この流れはもう動くコードの話です。

で、ここからが本題です。

支払えることと、実行してよいことは、同じではないと考えています。

## capabilityとadmissionを分けたい

エージェントの議論はcapability、つまり「何ができるか」に寄りがちです。toolを呼べるか。認証を通せるか。402を処理して支払えるか。どれも重要ですし、上に挙げた各社の実装はまさにそこを埋めています。

ただ、リクエストを受ける側、relying partyの立場で考えると、知りたいことが少し違います。このエージェントは誰で、誰のために動いているのか。実行環境は本物で、今も新鮮なのか。このタスクに対する権限は今も有効で、広すぎないか。そして、支払いの credential があったとして、この具体的なアクションを、今、この条件で実行させてよいのか。

できるかどうかがcapability。受け手がこの条件で実行してよいと判断したかどうかがadmission。

この境界を切り出して考えるために作っているのが、VATE（Verifiable Agent Trust Envelope）です。

<https://github.com/Poke-nushi/Verifiable-Agent-Trust-Envelope>

まず範囲を明確にしておきます。VATEはdiscussion draftです。標準でもなければproduction-readyでもなく、A2Aの公式extensionでもありません。A2A、MCP、OAuth、AP2、x402、VC/DIDといった隣接プロトコルを置き換える気もありません。それらはタスクを運び、証拠（evidence）を提供する側で、VATEはその証拠を受け手がどう評価し、判断をどう記録するかだけを扱います。

問いはこれだけです。

> 外部のエージェントがリスクのあるアクションを実行しようとするとき、副作用が起きる前に、受け手は何を検証し、何を記録すべきか。

リスクのあるアクションというのは、エージェントのruntimeの外側に状態変更を起こす操作のことです。支払いはその代表例ですが、リポジトリへのmerge、アカウント設定の変更、インフラの変更、規制対象データの更新、有料toolの呼び出しなども含みます。検索や文章生成は対象外です。

## 判断は allow / attenuate / deny の3つ

VATEの現行ドラフトでは、verifier（受け手側の検証者）は意味的に次の層を順に見ます。

```
status -> identity -> runtime -> permit -> policy
```

statusは失効・停止などの現在状態、identityはactorとprincipalの識別、runtimeは実行環境の新鮮さと真正性、permitはタスクスコープの権限、policyはローカルポリシーと制約です。実装上はこの前に、構造のパース、digest照合、署名とtrust anchorの確認、鮮度チェックがfail-closedで先に走ります。

評価の結果は3つに分かれます。

allowとdenyは説明不要だと思います。面白いのは真ん中のattenuateです。

エージェントが「USD 120まで使える前提で決済したい」と言ってきたとします。提示されたpayment tokenの上限とローカルポリシーを見ると、USD 25までなら通せる。このとき現実のシステムが取る選択は「全部拒否」ではなく「狭めて通す」のはずです。それがattenuationです。

重要なのは、この「狭めた」という事実を人間向けのログの文章ではなく、機械可読なレシートとして残すことです。リポジトリの実例（[admission-attenuate-max-amount.example.json](https://github.com/Poke-nushi/Verifiable-Agent-Trust-Envelope/blob/main/examples/receipts/admission-attenuate-max-amount.example.json)）から大幅に省略して引用します。

```
{
  "receipt_type": "admission",
  "decision": {
    "outcome": "attenuate",
    "reason_codes": [
      "PAYMENT_TOKEN_LIMIT",
      "LOCAL_POLICY_MAX_AMOUNT_NARROWED"
    ]
  },
  "attenuation": {
    "mode": "narrow",
    "original_request_hash": "sha-256:2621...",
    "effective_request_hash": "sha-256:55a4...",
    "changes": [
      {
        "op": "replace",
        "path": "/constraints/max_amount/value",
        "from": "120.00",
        "to": "25.00",
        "reason_code": "PAYMENT_TOKEN_LIMIT"
      }
    ],
    "effective_constraints": {
      "max_amount": { "currency": "USD", "value": "25.00" },
      "expires_at": "2026-05-04T03:10:08Z"
    }
  }
}
```

元のリクエストのハッシュと、狭めた後の実効リクエストのハッシュを両方持つのがポイントです。何がrequestedで、何がeffectiveになったのか。どのフィールドが、なぜ、どの証拠を根拠に変わったのか。これが残っていないと、後段の実行証拠を「事前に許可された範囲」と突き合わせる方法がなくなります。

## レシートは実行前と実行後で分ける

VATEはレシートを2段階に分けています。

admission receiptは実行前の判断の記録です。allow / attenuate / denyのどれだったか、どのアクションのどのdigestに紐付くか、どの証拠を見たか、いつまで有効か。

post-execution receiptは実行後の事実の記録です。実際に実行されたのか、失敗したのか、キャンセルされたのか。transaction idは何か。実行内容はeffective request hashと一致しているか。そして元のadmission receiptにリンクバックできるか。

「実行前に許可されたか」と「実行後に何が起きたか」は関連していますが別の事実なので、混ぜると監査ができなくなります。全体像はこうです。

## A2Aに載せるのはreferenceだけ

A2Aとの関係は、A2A本体のリポジトリでも質問として出しています。

<https://github.com/a2aproject/A2A/issues/1769>

私の現在の立場は、A2Aに面する部分は小さく保つべきだ、というものです。A2Aのmetadataに載せるのはレシート本体ではなく、artifactのURI、digest、issuer、有効期限といったreferenceまで。レシートやattenuationやpermitの本体はA2A coreの外に置く。

そして、ここが一番守りたい一線なのですが、referenceはauthorityではありません。metadataに参照が載っているからといって実行してよいことにはならず、受け手は参照先をdereferenceして、署名、digest binding、scope、鮮度、replay状態を自分で検証する必要があります。この区別が曖昧になると、metadataがいつの間にか許可証のように扱われてしまう。それが一番怖いパターンです。

ペイメントの話から始めましたが、VATEをpayment protocolに寄せる気はありません。支払いは説明しやすい一例というだけで、repository mergeにもインフラ変更にも同じ境界の問題があります。VATEはx402やAP2の競合ではなく、それらが生んだ証拠を受け取って評価する側、evidence-consumingなレイヤーのつもりです。

## レビューしてほしいこと

リポジトリにはドラフト本文のほかに、JSON Schema一式、allow / attenuate / denyそれぞれのconformance corpus（v0.3）、リファレンスのrunner、外部実装の結果を比較する`compare`コマンドと実装レポートのフォーマットが置いてあります。批評に耐える程度には具体化したつもりですが、リポジトリのreference runner以外の実装から出てきたSUT resultは、まだ一件もありません。これが今いちばんの弱点です。

なので募集します。ほしいのは技術的な批評です。

* allow / attenuate / denyという判断の形は妥当か
* status -> identity -> runtime -> permit -> policyという順序は実装者として納得できるか
* admission receiptとpost-execution receiptを分ける意味はあるか
* digest-boundなreferenceは外部実装で現実的に扱えるか
* A2A / MCP / OAuth / AP2 / x402との境界線の引き方を間違えていないか

フル実装は求めていません。corpusから3ケースだけ、たとえば`allow-valid-admission`、`attenuate-max-amount`、`deny-digest-mismatch-before-policy`を眺めて、「ここが曖昧」「このフィールドは実装しづらい」「このreason codeは意味が取れない」と言ってもらえるだけで十分に価値があります。1時間でできる最小レビューの案内を用意してあります。

<https://github.com/Poke-nushi/Verifiable-Agent-Trust-Envelope/blob/main/docs/conformance/external-sut-ask-1-hour.md>

質問や部分的な結果、「指示が曖昧でできなかった」という報告も含めて、受付はこのissueでやっています。

<https://github.com/Poke-nushi/Verifiable-Agent-Trust-Envelope/issues/2>

一点だけフェアに書いておくと、`compare`が通っても、それは「あるSUTの結果が、あるcorpusのスナップショットと、定義済みの比較ルールの下で一致した」という意味しかありません。認証でも互換性の証明でもないです。ドラフト段階で言えることはこのくらい狭い方が誠実だと思っています。

## おわりに

エージェントが外の世界に副作用を起こし始めると、identityとtokenの有効性だけでは足りなくなる。この具体的なアクションを、今、この条件で実行してよいのか。狭めて通したなら、何がオリジナルで何が実効なのか。その判断はレシートとして残っているのか。VATEはこの狭い境界だけを扱うドラフトです。

形が間違っているなら、早く知りたい。ボコボコの指摘でも歓迎です。A2A、MCP、エージェント認証、ペイメントプロトコル、runtime attestation、セキュリティレビューあたりに関心のある方に届けば嬉しいです。
