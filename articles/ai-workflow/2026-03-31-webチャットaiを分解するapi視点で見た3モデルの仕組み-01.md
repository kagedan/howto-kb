生成AIはWebチャットから使い始めた人が多いと思います。今でもChatGPT、Claude、Geminiなどをメインで使っているという人も多いんじゃないでしょうか。

生成AIを使っているとよく「コンテキスト」という言葉を聞くと思います。「コンテキストが大事」「コンテキストを節約して」「コンテキスト上限に引っかかった」などと言われますが、これらは具体的に何のことでしょうか。

Anthropicの公式ドキュメントにはこう書かれています。

> 「コンテキストウィンドウ」とは、言語モデルがレスポンスを生成する際に参照できるすべてのテキストを指し、レスポンス自体も含まれます。  
> これは言語モデルが学習した大規模なデータコーパスとは異なり、モデルの「ワーキングメモリ」を表します。  
> コンテキストウィンドウが大きいほど、モデルはより複雑で長いプロンプトを処理できます。  
> コンテキストウィンドウが小さいと、長い会話にわたって一貫性を維持するモデルの能力が制限される可能性があります。
>
> 以下の図は、APIリクエストにおける標準的なコンテキストウィンドウの動作を示しています。
>
> * **段階的なトークンの蓄積：** 会話がターンを重ねるにつれて、各ユーザーメッセージとアシスタントのレスポンスがコンテキストウィンドウ内に蓄積されます。
> * **線形的な増加パターン：** コンテキストの使用量は各ターンごとに線形的に増加し、以前のターンは完全に保持されます。
> * **200Kトークンの容量：** 利用可能なコンテキストウィンドウの合計（200,000トークン）は、会話履歴の保存とClaudeからの新しい出力の生成のための最大容量を表します。
> * **入出力フロー：** 各ターンは入力フェーズ（すべての以前の会話履歴 + 現在のユーザーメッセージ）と出力フェーズ（テキストレスポンスの生成）で構成されます。

![コンテキストウィンドウの動作図](https://static.zenn.studio/user-upload/181a1cc70a88-20260331.png)  
*出典: [コンテキストウィンドウの管理 (Anthropic)](https://platform.claude.com/docs/ja/build-with-claude/context-windows)*

つまり、モデルが「今この瞬間に読める範囲」がコンテキストウィンドウです。会話が長く続くほど、コンテキストウィンドウにトークンが増えていきます。

これをAPIで書くと次のようになります。

```
response = client.chat.completions.create(
    model="gpt-5.4",
    messages=[
        { "role": "developer", "content": "あなたは親切なアシスタントです。" },
        { "role": "user",      "content": "1+1は？" },
        { "role": "assistant", "content": "2です。" },
        { "role": "user",      "content": "では2+2は？" },
    ],
)
print(response.choices[0].message.content)
# -> 4です。
```

それぞれの `role` の意味は以下のとおりです。

* `developer`: 開発者・サービス側が書く指示
* `user`: ユーザーが打つ文字
* `assistant`: AIの返答

でも、ここで疑問が生まれます。Webチャット版には「カスタム指示」「メモリー」「プロジェクト」など、もっとたくさんの機能がありますよね。これらはAPIでは再現できないのでしょうか。全部、`developer` ロールに挿入すればいいのです。

## 各社のWebチャットは何ができるのか

まず、主要3サービスの機能を整理します。

| 機能カテゴリ | ChatGPT | Claude | Gemini |
| --- | --- | --- | --- |
| 永続的な指示 | カスタム指示 | — | カスタム指示 |
| 独自AI設定 | MyGPT（完全なシステムプロンプト制御） | — | Gems（保存されたシステムプロンプト） |
| プロジェクト | プロジェクト（指示 + ファイル + メモリー） | プロジェクト（指示 + 知識 + ファイル） | — |
| メモリー | あり | あり | あり |

## APIでは全部 `developer` ロールに入れる

これらの機能は全部、APIの `developer` ロールに注入することで実現できます。例えば次のように設定します。

```
developer_prompt = """
<custom_instructions>
ユーザーはエンジニアです。簡潔に答えてください。
</custom_instructions>

<memory>
- ユーザーはPythonを使っている
</memory>

<project_instructions>
このプロジェクトでは必ず日本語で返答してください。
</project_instructions>
"""

# 1ターン目
response = client.chat.completions.create(
    model="gpt-5.4",
    messages=[
        { "role": "developer", "content": developer_prompt },
        { "role": "user",      "content": "1+1を出力するプログラムを教えて" },
    ],
)
print(response.choices[0].message.content)
# -> print(1 + 1)

# 2ターン目
response = client.chat.completions.create(
    model="gpt-5.4",
    messages=[
        # developer は毎回同じ内容を渡す
        { "role": "developer", "content": developer_prompt },
        { "role": "user",      "content": "1+1を出力するプログラムを教えて" },
        { "role": "assistant", "content": "print(1 + 1)" },       # 前の返答を自分で追加
        { "role": "user",      "content": "Rubyではどう書く？" }, # 次の質問を入力
    ],
)
print(response.choices[0].message.content)
# -> puts 1 + 1
```

Webチャットのサービスが裏でやっていることは、要するにこれです。カスタム指示・メモリー・プロジェクト指示を組み合わせて `developer` ロールに詰め込み、ユーザーのメッセージより先にモデルに渡します。

APIを使えば、同じ仕組みを自分で設計できます。Webチャットのサービスはこれらを裏で自動的にやってくれますが、APIではすべて自分で組み立てる必要があります。面倒ではありますが、逆に言えば何をどう渡すかを完全にコントロールできます。

なお、APIはステートレスなので、サーバー側は会話を覚えていません。ターンが増えるたびに、呼び出し側が「これまでの全履歴」に「新しい質問」を追加して渡します。これがコンテキストウィンドウの「会話が続くほどトークンが積み重なる」という意味です。

## 見えない最上位プロンプト

ここまでで、Webチャットの機能が `developer` ロールで再現できることが分かりました。しかし、実は `developer` よりもさらに上位の、ユーザーからは触れない**隠しプロンプト**が存在します。

各会社のプロンプトは以下の通りです。

### OpenAIの場合

OpenAIは [Model Spec (2025-12-18)](https://model-spec.openai.com/2025-12-18.html) で、developer・userの上にさらに2つの層があることを明記しています。

**Model Spec** とは、OpenAIがモデルの振る舞いと価値観を定義した公開文書です。「モデルは何を絶対にやってはいけないか」「各ロールの優先順位はどうなっているか」「ユーザーの指示とプラットフォームの指示が衝突したときにどう動くか」といった原則が書かれています。特に **Red-line principles**（越えてはならない一線）として、以下のような絶対禁止事項が列挙されています。

* 大量破壊兵器（生物・化学・核兵器）の作成支援
* テロリズム・人身売買・児童性的虐待コンテンツの生成
* 人間の自律性を損なう操作や、市民プロセスへの介入
* 個人のプライバシーの侵害

そしてこう続きます。

> "Customization, personalization, and localization (except as it relates to legal compliance) should never override any principles above the 'guideline' level in this Model Spec."  
> 「カスタマイズ・パーソナライズ・ローカライズは、このModel Specの『guideline』レベルより上位の原則を上書きしてはなりません。」（筆者訳）

つまり、どんな `developer` プロンプトを書いても、Root・System レベルの制約は上書きできません。**Root** は「誰も上書きできない絶対ルール」、**System** はOpenAIが設定し `developer` や `user` には上書きできない層です。

### Anthropicの場合

Anthropicは **"core system prompt"** という名称で、claude.aiとモバイルアプリが使うプロンプトの内容を[APIドキュメント](https://platform.claude.com/docs/ja/release-notes/system-prompts)に全文公開しています。このプロンプトには、現在の日付や応答スタイルの指示だけでなく、以下のような行動原則が直接埋め込まれています（Opus 4.6、2026年2月版より）。

* 子どもの安全への配慮（未成年に関わるコンテンツの慎重な取り扱い）
* 兵器・有害物質の製造につながる情報の提供禁止（爆発物・化学・生物・核兵器を含む）
* マルウェアや脆弱性エクスプロイトなど悪意のあるコードの作成禁止

また、どんな `developer` プロンプトを書いても、これらの原則はすでにプロンプトとして最上位に注入されており、上書きできません。

### Googleの場合

Geminiについては、OpenAIのModel SpecやAnthropicの公式リリースノートに相当する文書が現時点では確認できません。同様の隠しプロンプト層が存在する可能性は高いですが、公式に認める一次情報はない状態です。

---

これをまとめると、実際の構造はこうなっています。

* 隠しプロンプト: プラットフォームが自動注入（ユーザーには見えない）
* developer: Webチャットの各種設定
* user: ユーザーのインプット
* assistant: AIの返答

![](https://static.zenn.studio/user-upload/b209ce171e36-20260331.png)

なぜ隠しプロンプトはこれほど強いのでしょうか。訓練で「上位の指示を優先せよ」と学習していることもありますが、もう一つ理由があります。

LLMはこれら全てをフラットな1列のトークン列として読みます。そして、**先頭と末尾に置かれた情報ほど参照されやすく、中間に埋まった情報は参照されにくくなる**という特性があります。これをLost in the Middleと呼びます。隠しプロンプトはコンテキストの一番先頭に置かれるため、構造的に最も参照されやすい位置にあります。

## まとめ

Webチャットの便利な機能は、全部 `developer` ロールへの注入で実現できます。

| 種別 | 内容 | 誰が書くか |
| --- | --- | --- |
| 隠しプロンプト | 日付・安全制限・応答スタイル等 | プラットフォーム（ユーザーには不可視） |
| developer | カスタム指示・メモリー・プロジェクト指示 | サービス or ユーザー |
| user | 毎回のメッセージ | ユーザー |

APIを使えば、`developer` ロールの中身を自分で自由に設計できます。カスタム指示もメモリーもプロジェクト設定も、全部自分の手で組み立てられます。Webチャットが裏でやっていることと同じことを、自分のアプリで再現できるわけです。

![](https://static.zenn.studio/user-upload/93e0000f8714-20260331.png)

ただし、コンテキストウィンドウには上限があるので、詰め込みすぎには注意です。`developer` に何をどれだけ入れるかも含めて、設計の腕の見せどころです。

今回は、Webチャットの機能と、それをAPIで再現するにはどうすればよいかについて調べてみました。API開発をするときの参考になれば幸いです。
