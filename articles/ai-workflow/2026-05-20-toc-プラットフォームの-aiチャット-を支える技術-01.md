こんにちは、[かわそん](https://x.com/KKohey4)です。

先日、知識共有プラットフォーム「[Brain](https://brain-market.com/)」に、CS（カスタマーサービス）を担当するAIチャットを導入しました。

リリースから 47時間での利用回数は238件。

AIが答えられるべきだが答えられなかったものは 5件のみ。

**カバー率が 97.9%** という結果になりました。  
(もちろん集計期間や集計頻度によるブレはありますが、速報としての数値です)

なお、最終的に人間が対応した問い合わせも減りました。

概算で9.3通/日 の問い合わせだったのが3通/日 のため、**約67.7%の減少**です。

今回は、そのAIチャットを支える技術についてご紹介したいと思います。

## CS AI チャットのスコープ

現時点では CS AI はβ版です。β版の仕様は下記となっています。

1. サービスの利用方法に関するものに特化して回答
2. ユーザー個別の事象については回答しない

2について下記の理由によりβ版としては見送る判断をしています。

```
1. read onlyとはいえ、AIに db を直接触らせる必要がある。
2. ユーザーの情報とはいえ、db の中にはユーザーに見せたくないデータがある。それをAIの判断や仕組みに落とし込むのは、考慮すべき要素が多い。
3. Brainの機能は非常に多く、使い方や仕様に関する問い合わせが多い。そのため、それに自動応答するだけでも、ミニマムのバリューは出せると判断。
```

今後の展望として、もちろんカバーすることを見越して設計しています。

今回はこの点はスコープ外として、技術仕様を記述します。

## アーキテクチャ

インフラ、外部ストレージ、外部通信などを含むアーキテクチャの全体像は下記です。

![](https://static.zenn.studio/user-upload/802878a4a9ab-20260520.png)

mermaid はこちらです。

ユーザーがCS AIに質問すると、ALB target group の path-based routing により、chat 専用のECS Service にリクエストを分流します。

専用のECSサービスに切り出した理由については後述します。

chat 専用のECS container に おいては、mode 環境変数を task definition に埋め込むことで、Fargate上で chat 専用で振る舞わせるようにしています。

container 上で入力値のバリデーションや、ElastiCacheと連動した rate limiter による入力回数・頻度の評価を行っています。

リクエストが正当である確認できた段階で、質問回数を consume します。

この段階で初めて、RAGとして選定している外部ストレージや外部のLLMと通信を行い、回答を生成します。

この探索ロジックについても後述します。

回答できた場合は chunk として本文を返し、最後に done シグナルを返します。

下記のイメージです。

```
data: {"type":"chunk","content":"回答本文の一部"}

data: {"type":"chunk","content":"続きの本文"}

data: {"type":"done"}
```

個別問い合わせ、knowledge 不足、外部サービスエラーなどは、それぞれ error または redirect と専用 code を response に含める形で返します。

```
data: {
  "type": "redirect",
  "code": "knowledge_unavailable",
  "message": "...",
  "redirect_url": "..."
}
```

フロントが SSE を読み取り、chunk を受けるたびに assistant メッセージへ追記して画面に表示します。

redirect や error の場合は問い合わせフォーム導線やエラー表示へ分岐します。  
(redirect\_url は 定数で管理しており、それ以外の値が返らないように制御しています)

送信後、フロントは 残り回数を再確認します。このリクエストも同じくChat ECS Service に分流し、必要に応じて UI の切り替えを行います。

## 技術選定の根拠

この構成を実現するにあたっての、技術選定のポイントをまとめます。

大きく下記の3点です

```
1. APIのECSサービスと同居させるかどうか
2. SSEかWebSocketか
3. PineconeかOpenAI Vector Storeか
```

### 1. API の ECS Service に同居させるか、別にするか

下記の理由により、専用の ECS Service を切り出す判断としました。

```
1. APIのHTTP属性と違う
2. Auto Scaling の条件が違う
3. 障害を分離したい
4. デプロイを完全に切り分けたい
```

#### APIのHTTP属性と違う

チャットは SSE を利用するため、**APIと比べて比較的長めの応答を許容**します。

APIが求める response 速度と chat が求める速度が異なるためです。

しかし、ALBが共通であるため、リクエストの送信先によってタイムアウト値を変えるといった制御はできません。そのため、性質の異なるものを別のECSサービスとして切り出す構成を選定しています

1. API用ECSサービス  
   既存のALBの idle timeout 値に合わせる。それによって、既存の response から不必要に引き延ばさないようにします。
2. チャット専用ECSサービス  
   接続自体は長めに取るものの、**定期的に heartbeat を送り続ける**ことによって idle timeoutを回避する。

#### 障害の分離とオートスケーリング条件の違い

これはイメージがつきやすいので、まとめて扱います。

chat が高負荷になったからといって、APIに影響を与えたくないためです。

chatが高負荷になったら**chatだけに閉じて auto scaling する** というのが望ましい形です。

chat がアクセスを捌ききれなくなってもAPIが通常稼働してほしいため、これらを完全に切り分けるべく、Service ごと分離する判断をしました。

#### デプロイを完全に切り分けたい

chatはユーザーの入力値の幅が非常に広いです。

そのため、予期しない不具合があったとき、既存のAPIは可能な限りタッチせず、chatの方のみdeployを切り分けたいというニーズがありました。

現在はβ版なのでこのニーズが高いというのもありますが、今後AIが進んでいくにあたって、interactive な chat の需要はある程度伸びてくるはず。

そこを踏まえて、別の ECS service として切り分けるという判断を取りました。

#### 2. SSEかWebSocketか

SSEを選定した理由はシンプルで、今回の chat というのは **LLM からの streaming 用途が強い**からです。

ユーザーとAIの双方向なリアルタイム通信は必要ではないです。

ましてや一本の通信でやり取りする必要 = 双方向のコネクションを維持する必要もない。

AIからの応答を streaming してユーザーに返すといった側面の方が強いため、SSEを選定しました。

#### Pinecone か OpenAI Vector Store か

OpenAI Vector Storeを採用しました。

vector DBの tuning に関する保守運用をしたくなかったのと、いきなり定性的な話になりますが、OpenAIという巨人の肩に乗っておけば、この辺はいずれ最適化されるだろうと踏んだためです。

Pinecone は　Vector DBとしての検索性は評価しています。

しかし、今回の単純なドキュメント検索では、Pineconeの小回りの高さを活かしきることや、保守の運用に乗せることのメリットを考えると、**割に合わない**と判断したためになります。

### 探索のロジックと応答の制御

探索ロジックの全体構成は以下です  
![](https://static.zenn.studio/user-upload/38ac9784dfd6-20260520.png)

mermaid はこちらです。

全体を見ると複雑に見えますが、紐解いていくとそうでもありません。

まずはユーザーの質問に対して検索表現を作るフェーズです。言い換えや口語表現、会話履歴などを通じて、検索クエリの候補を作ります。

そして、定義してある最大クエリの数に絞って、主要クエリで検索を行います。

定義してある最大並列度に応じて、OpenAI Vector Storeを探索します。

検索結果が重複した場合は、文字列ベースで rerank することにしています。ヒット数が十分でなければ、残りの検索クエリを実行します。

ここまでやっても十分な結果が得られなかった場合に限り fallback 検索を行い、広めに候補を洗い出すというような処理をしています。

検索結果を確定した段階で、文字列ベースで順位付けを行います。要素としては、以下の通りです

```
1. 本文一致
2. ファイル名一致
3. 正規化一致
4. Bigram類似度
```

その後、LLMによる rerankが必要かどうかを判定します。

**候補が非常に強力であったり、逆に候補が少なかったり、あるいは1位が明確に強かった場合は、LLMのrerankをスキップします。**

判断が微妙な場合にのみ、LLMでrerankを行います。  
判断の信頼スコアを出しているため、閾値を下回った場合に限って次の打ち手に進むという流れです。

最終的に順位を確定し、回答生成用のコンテキストとして整備するようにしています。

## コードベースと knowledge・OpenAI Vector Storeの同期

コードベースと knowledge 、knowledgeと OpenAI Vector Storeを同期するようなCDを組んでいます。

**APIはもちろん仕様を含んでいますが、APIの変更はユーザーに見せたくないような仕様も含んでおり、どちらかというとフロントの方が仕様そのものに近い** です。

そのため、フロントはフロントのCI/CDでナレッジと同期するような仕組みを組んでいます。

knowledge はOpenAI Vector Storeを利用しており、knowledge のCI/CDでOpenAI Vector Storeへの反映を行っています。

フロントエンドのブランチを release へマージしたタイミングで、knowledge もその対応する release ブランチに対応した PR を生成し、対応環境へ昇格・反映させます。

これによって front のコードベースの変更を knowledge に反映し、その変更をさらに OpenAI Vector Store に反映するという構成をとっています。

front の更新が優先されるため、front の feature ブランチや release branch に対して変更があれば、knowledge リポジトリが OpenAI Vector Store に同期するという形をとっています。

(このあたりの lock はどうなっているんだ、という話もありますが、需要があれば解説します。現時点では情報過多のため省略します。)

### knowledge の管理単位

1 topic = 1 ファイルにしています。

* 画面単位の knowledge
* 共通仕様単位の knowledge

それぞれのファイルは yaml formatter を搭載しています。

一部抜粋します。

```
type: screen | common
id: 安定した識別子
category: 大分類
route: 画面 route # screen のみ
tags: 検索語
questions: ユーザーが入力しそうな発話
related: 関連 knowledge
sync:
  owners:
    front_files: 対応する front ファイル
    front_globs: 影響する front の glob
  change_kinds: 変更種別
last_reviewed_at: 最終確認日
```

sync 系のフィールドは検索クエリに直接効くというより、検索される knowledge をコード変更に追従させるための仕組みになります。

category, route, tags, questions, related を検索性に強く効くフィールドとして採用しています。

last\_reviewed\_at に関しては、 knowledge ないし front のリポジトリに変更があった際に、日付だけを更新するような挙動を許可しています。

これは変更があった時点で、**最新のナレッジの鮮度が保たれているという保証を記録**するためです。

## Devin を使った knowledge の更新

AIだけで解決せず、調査が必要になった場合は、社内のSlackに通知が飛んできます。

そこでCSの担当者と開発者が、必要に応じて会話を行います。

Devinはそのチャンネルにいるため、問い合わせが解決した後にスレッドでDevinを巻き込みます。

Devinはそのスレッド内容を解釈し、汎用的なナレッジとしてリポジトリに反映処理を行います。

## 課題とまとめ

今回、CSのAI化によってある程度定量的な成果を上げることができました。

しかし、AIに db を触らせない以上、やはり個別具体的な調査に関する制約が出てきます。

現時点では、これはCSの担当者と開発側がコミュニケーションを取って解決するしかありません。

AIにデータベースを触らせるかという判断については、様々な考慮要素があります。

ただ、**従来の目的はCS担当者の負荷を下げること** です。

これから先は費用対効果の話。

1. 現時点で負荷の軽減が十分であれば、実装に落とし込む必要はない
2. 負荷の軽減が不十分であれば、次の打ち手を考える必要がある

AIに db を触らせるにあたっては、**各種ガードレールや権限の整備が必要** になります。

これらを踏まえて、継続的に観察しつつ次のアクションを模索していくのが、現時点での最適解だと考えています。

今回の記事が、同様の課題を抱えている企業の一助になれば幸いです。
