## TL;DR

* 個人開発の株式情報サービス [dow-japan.jp](https://dow-japan.jp) のデータ基盤の上で、毎日 **約4,400銘柄** を処理して個別解説を自動生成するパイプラインを回している（16並列・ローカルLLMでコスト最適化）。
* その上に、短中期スイングの判断を出す **12体のマルチエージェント・リサーチチーム**。全銘柄をスクリーニング → 数銘柄に絞り込み → 深掘り → **レッドチームが必ず反証してから、PMが最終判定**する。
* すごいデモではなく、**既存の本番システム（Laravel 12 / Nuxt 3 / MySQL / J-Quants API）に統合して、毎日無人で動かしている**のがポイント。幻覚対策・コスト設計・運用まで含めた"作って終わりにしない"作り方を書く。

---

## 課題：数千銘柄から「今見るべき数銘柄」を、根拠と反証つきで出す

東証には3,000社以上が上場している。毎日の値動き・出来高・指標を全部人間が追うのは物理的に無理だ。しかも欲しいのは単なる要約ではなく、「**いま注目すべきはどれで、どういう根拠で、どこにリスクがあるか**」まで踏み込んだ判断だった。

ここで素朴にLLMへ全銘柄を丸投げすると、**コストと品質（幻覚）** が破綻する。数千銘柄 × 毎日 × 高性能モデルでは財布が持たないし、根拠のない"それっぽい嘘"を量産するだけになる。さらに、単発のLLM出力は「強気にも弱気にも、それらしく書けてしまう」。判断に使うなら、**出力をそのまま信じない仕組み**が要る。

この記事は、その3つ——**スケール**（全銘柄を毎日）／**信頼性**（反証を経た判断）／**コスト**——をどう両立させたか、という話。

---

## 手法：データ基盤 → 12体チームの「広げて・絞って・反証して・決める」

全体は2段になっている。まず **dow-japan のデータ基盤**が毎日ほぼ全銘柄ぶんのデータと解説を生成する。その上で、**12体のリサーチチーム**が「広く拾う → 絞る → 深掘る → 反証する → 決める」を回す。

```
[ データ基盤 / dow-japan ]
  J-Quants API → MySQL 正規化 → flow_score・相関メタデータ
  毎日 約4,400銘柄の個別データ/解説を自動生成（16並列・ローカルLLM）
            │
            ▼   [ 12体リサーチチーム / daily_run.sh ]
  Round1（並列）  06 マクロ ／ 02 クオンツ ／ 03 カタリスト ／ 09 振り返り
        └─→ 11 フィルター：30〜40銘柄 → 5〜7銘柄 → TOP3
  Round2（並列）  05 ファンダ ／ 04 テクニカル ／ 07 リスク（TOP3を深掘り）
  Round3          08 レッドチーム = 全提案に必ず反証（ベアケース）
  Round4          01 PM = 反証を踏まえ最終判定（GO / 打診 / 見送り）→ /proposals 保存
  別系統          10 ポジション・レビュアー（保有玉の日次管理） ／ 12 トレード・コーチ（規律）
```

1体の万能エージェントに全部やらせず、**役割を分けた専門エージェント**を並べているのがミソ。工程を分けるほど各エージェントのプロンプトは単純になり、検証もしやすくなる。

## 12体の役割

**発掘・分析班（何を・なぜ候補にするか）**

* 02 クオンツ・スクリーナー … モメンタム/需給/流動性/ボラの複合条件で機械的に候補プールを抽出
* 03 カタリスト/材料ハンター … 適時開示・決算スケジュール・レーティング変更を追い、近い将来動く銘柄を抽出
* 06 マクロ・セクター戦略家 … 市場地合い・セクターローテーションを判定し、候補のマクロ整合性を評価
* 09 振り返り（Post-Mortem） … 過去のプロセスと結果を分けて検証し、勝ち/負けパターンを抽出
* 11 フィルター … 軽量フィルタで 30〜40銘柄を 5〜7銘柄に絞る

**深掘り・検証班（本当に良い玉か・どこで入り、どこで切るか）**

* 04 テクニカルアナリスト … マルチタイムフレームでトレンド/押し目位置/出来高整合を判定
* 05 ファンダメンタルズアナリスト … 業績モメンタム・コンセンサス乖離・織り込み度を判定
* 07 リスクマネージャー … ATR/相関/許容DDからポジションサイズとロスカット水準を定量化
* 08 レッドチーム（反証）★最重要 … 全提案に必ず反論。ベアケース/見落とし/ロジックの穴/バイアスを指摘。同意は禁止

**意思決定・管理・コーチ**

* 01 PM/統合戦略責任者 … 全員の分析を統合し最終意思決定。提案を記録
* 10 ポジション・レビュアー … 保有中の玉を日次レビューし、継続/手仕舞い/部分利確/ストップ更新を判定
* 12 トレード・コーチ … 規律・実行を鍛える（チェックリスト照合・日誌レビュー・弱点矯正）

## ここからが本題：コモディティとの差になる "深さ"

「エージェント作れます」は今や誰でも言える。差がつくのは、**本番で毎日動かすために必要な、地味で面倒な部分**だ。4つに分けて書く。

### 1. 統合：既存の本番システムに"組み込んだ"

新規に作った隔離デモではなく、**すでに動いている dow-japan の本番構成にそのまま乗せている**。

* **データ**：J-Quants API から取得 → MySQL に正規化 → 33業種ごとの独自スコア **flow\_score** と相関メタデータを生成
* **オーケストレーション**：Laravel 12 のバッチ／キューと `daily_run.sh` がエージェント群の実行・リトライ・スケジューリングを担当
* **表示**：Nuxt 3 のダッシュボードに反映、Slack と X に自動投稿

つまりエージェントは「既存のデータ層・アプリ層の上に増設したモジュール」であって、**受託で言えば "お客さんの既存システムに統合する" のと同じ構図**で作っている。

### 2. 信頼性：反証してから決める（レッドチームと評価ループ）

毎日自動で判断を出す以上、**根拠のない出力を垂れ流さない仕組み**が必須だった。ここがこの基盤の一番の肝。

* **入力を事実に縛る**：生成前に、MySQL上の確定値（価格・出来高・flow\_score 等）を渡し、**"渡した数字以外を断定しない"** 制約をかける
* **08 レッドチームが必ず反証**：全提案に対してベアケース・見落とし・ロジックの穴・バイアスを指摘する。同意は禁止
* **01 PM はその反証を踏まえて判定**：レッドチームのベアケースを採用したうえで GO / 打診 / 見送りを決める
* **09 振り返り**でプロセスと結果を分けて検証し、次に反映する

この **広げる → 絞る → 反証 → 判定** のループが、単発のLLM呼び出しと決定的に違うところ。ある日は「1銘柄だけ打診（押し目限定）、他は見送り」という判定になる——**派手に銘柄を並べるのではなく、反証を通った数銘柄だけが残る**。

![Slack に毎朝自動投稿される日次要約。地合い・セクター資金流入まで自動で整形（売買判断はマスク）](https://static.zenn.studio/user-upload/deployed-images/83ecbd0e20c3e112d0731bfc.png?sha=2f6744bf8529b06bb2b5e0b80996bca530380912)

### 3. コスト設計：全部を高いモデルに投げない（Dual LLM）

数千銘柄を毎日まわすので、**処理の性質でモデルを使い分けている**。

* **大量バッチはローカルLLM**：全銘柄の個別データ/解説の一次生成は、ローカルPC上に立てた自前のLLM推論サーバーで処理。**外部APIを叩く回数そのものを減らす**
* **少量・高品質が要る所だけ上位API**：翻訳や最終仕上げなど、1日数十件レベルの用途にだけ上位のAPIモデルを使う

試算すると差が分かりやすい。1日およそ5,000件（入力5M / 出力1.5Mトークン）を上位APIモデルで毎日回すと、ざっくり試算で **月数万〜十数万円**になる（例：Gemini Flash級で月$30〜50、Claude Haiku級で月$375、Sonnet級で月$1,000超）。この大量バッチを **ローカルLLMに寄せることで、その部分を電気代だけ＝実質ゼロ**に抑え、上位APIは"少量だが品質が要る"用途にだけ使う——という二段構え（Dual LLM）にしている。作れるかどうかではなく、**コストまで設計して構成を選ぶ**。ここが本番運用で効く。

### 4. 運用：本番で毎日回っている

一番の証拠は、**これが特別な日のデモではなく、毎営業日 無人で動いている**こと。

* データ取得 → 分析データ構築 → 個別テキスト生成 → キャッシュ更新まで日次バッチで自動
* 12体チームの `daily_run.sh` も日次で走り、判断を記録
* 失敗時のリトライ／ログ、出力の自動投稿まで含めて運用に乗っている

## 成果：何がどれだけ変わったか

* 毎営業日、データ取得 → 分析データ構築 → 個別テキスト生成 → キャッシュ更新までを**全自動で完走**（一連で**約4.8時間／約17,000秒**、人は一切触らない）
* **日足・週足あわせて約7,400銘柄ぶん**（日足3,732件＋週足3,712件）の個別解説テキストを毎日生成
* テキスト生成は **16ワーカー並列で、約3,700件を約30分**
* 元データは J-Quants から日次 **約4,400銘柄**（imported 4,442 / skipped 7）を取り込み
* その上で12体チームが全銘柄から数銘柄に絞り込み、**反証を通った候補だけ**を提案として残す
* 出力はダッシュボード・Slack・X に自動で届き、**人は"結果を読むだけ"** になった

人手で毎日数千銘柄を読むのは不可能。それを「**無人で約4.8時間、毎営業日まわり続け、反証を経た数銘柄だけが残る**」状態にしたのが一番の成果。

## これは株式に限った話ではない

ここで使った要素 ——

* 大量データを **工程分割したエージェント群** で処理する
* **レッドチームによる反証 → 統合判断** で出力の信頼性を担保する
* **モデルのカスケード＋ローカルLLM** でコストを設計する
* それらを **既存の本番システムに統合** して無人運用する

—— は、株式分析に固有のものではない。**大量のデータと多工程の判断業務を抱えた、あらゆる既存システム**に同じ型で持ち込める。「PoCは動いたが本番に乗らない」「コストが読めない」「幻覚が怖くて出せない」——そこを越えるのが、自分のやってきたことだと思っている。

## まとめ

* デモではなく、**本番で毎日動くマルチエージェントを、実データの上に組んで運用している**
* 差がつくのは派手な部分ではなく、**統合・信頼性（反証してから決める）・コスト設計・運用**という地味な4点
* この型は金融に限らず、**既存システムへのAIエージェント統合**として転用できる
