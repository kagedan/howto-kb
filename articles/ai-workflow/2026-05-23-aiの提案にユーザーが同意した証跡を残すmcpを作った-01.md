![](https://static.zenn.studio/user-upload/4866ea4562fd-20260523.png)  
toCのAIエージェントを作っていると、ユーザーの同意を記録したい場面がある。

たとえばユーザーの代わりに外部サービスへの送信や登録をエージェントが代行するとき、「ユーザーがこの内容に同意した」という証跡が欲しい。

## 定型的な同意ならブラウザで開けばいい

Webサービスの利用規約のような定型文言なら、同意ページをブラウザで開いてチェックを押させる方法がある。あれは単なるUIではなく、誰が・いつ・何に同意したかの記録だ。

問題はAIエージェントの場合、提案の内容が毎回違う。エージェントが状況に合わせて提案した内容への同意も残したい。

## エージェントの中で閉じていたい

ブラウザを開かせるのは体験として重い。CLIでもチャットでも、エージェントとの会話の中で完結させたい。

エージェント側にfunctionを定義すれば実装はできる。でも同意取得のロジックをエージェントごとに書くのは非効率だし、証跡の堅牢性もまちまちになる。汎用的な仕組みとして切り出したかった。

## MCPサーバーにした

というわけで同意管理をMCPサーバーとして実装した。

![](https://res.cloudinary.com/zenn/image/fetch/s--39x1EyUs--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://github.com/tomohiro-owada/ai-consent-audit?_a=BACAGSGT)

![](https://static.zenn.studio/user-upload/4ad9a3f03447-20260523.png)

エージェントは3つのツールを順番に呼ぶだけでいい。

```
1. consent.request.fixed   ユーザーに確認文を出す
2. consent.judge.fixed     返答を判定する
3. consent.evidence.commit 証跡を保存する
```

`consent.request.fixed` を呼ぶと、ユーザーに見せる確認文が生成される。

```
以下の内容で実行してよいですか？

対象：〇〇
送信先：〇〇
含まれる情報：〇〇

同意する場合は、次の文をそのまま返信してください。

はい、上記の内容を確認し、実行に同意します。
```

ユーザーがこの文言をそのまま返信したら `consent.judge.fixed` で判定する。「まあいいか」のような曖昧な返答は通らない。判定結果は4パターン。

```
accepted  → 次のステップへ
rejected  → 実行しない
unclear   → 再確認を促す
expired   → 同意を作り直す
```

## 改ざんできないログとしてGitを使う

証跡の保存先はGitHub Private Repositoryのコミット履歴にした。

コミット履歴はappend-onlyで、後から書き換えが難しい。専用のDBやストレージを立てなくても、手元にあるGitHubがそのまま監査ログになる。

GitHubに保存されるのはこんなJSON。個人情報は含めず、ハッシュとラベルのみ。

```
{
  "consent_id": "consent_01KS...",
  "status": "accepted",
  "timestamps": {
    "created_at": "2026-05-23T10:00:00Z",
    "judged_at": "2026-05-23T10:01:00Z",
    "committed_at": "2026-05-23T10:01:03Z"
  },
  "action": {
    "label": "〇〇の実行",
    "destination_label": "〇〇サービス",
    "included_data_labels": ["項目A", "項目B"],
    "payload_hash": "sha256:..."
  },
  "user_response": {
    "text_hash": "sha256:..."
  },
  "integrity": {
    "record_hash": "sha256:..."
  }
}
```

## 判定だけでは実行させない

同意の判定が通っても、GitHubへの証跡保存が成功するまで実行を許可しない設計にした。

```
consent.judge.fixed     → accepted（判定OKだが実行はまだ不可）
consent.evidence.commit → 成功（ここで初めて execution_allowed: true）
```

ログが残らなければ動かない、という制約をサーバー側に持たせることで、エージェント側のミスや省略を防ぐ。
