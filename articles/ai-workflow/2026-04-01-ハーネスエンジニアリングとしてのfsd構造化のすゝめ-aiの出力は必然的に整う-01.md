## AIの出力は必然的に整う

FSD の構造ルールに違反するとエラーが出る。AI がそれを直す——それだけで、コードの構造が整っていく。本記事はこの「エラー修正の繰り返しで構造が収束する」メカニズムに特化した記事だ。

対象読者は AI エージェントにコードを書かせているフロントエンド中級者。FSD の2つの核心ルール（レイヤー依存方向 + スライス間分離）がなぜ AI のハーネスとして機能するのか、Clean Architecture との比較を交えて論じ、lint ツールの導入方法と導入時の Q&A を提供する。

## ハーネスエンジニアリングと FSD の位置づけ

AI エージェントにコードを書かせる時代、エンジニアの仕事は「コードを書くこと」から「AI が正しく動ける環境を設計すること」へとシフトしつつある。Mitchell Hashimoto はこのアプローチを ["Engineer the Harness"](https://mitchellh.com/writing/my-ai-adoption-journey) と名付けた。

> It is the idea that anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again.  
> — [My AI Adoption Journey](https://mitchellh.com/writing/my-ai-adoption-journey)

エージェントがミスをするたびに、そのミスが二度と起きないよう仕組みで解決する——これがハーネスエンジニアリングの基本思想だ。

では、ハーネスにはどんな種類があるのか。[Birgitta Böckeler（Martin Fowler's blog）](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)は、ハーネスを次の3カテゴリに整理している。

> **Context Engineering**: Continuously enhanced knowledge base in the codebase, plus agent access to dynamic context.  
> **Architectural Constraints**: Monitored not only by the LLM-based agents, but also deterministic custom linters and structural tests.  
> **Garbage Collection**: Agents that run periodically to find inconsistencies in documentation or violations of architectural constraints.  
> — [Harness Engineering](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)

これを図示し、本記事で注目する FSD の位置づけを示す。

本記事では、この中の **Architectural Constraints**（アーキテクチャ制約）に焦点を当てる。具体的には、フロントエンドアーキテクチャである **FSD（Feature-Sliced Design）** を lint ルールで強制し、AI の出力を構造的に制約するアプローチを解説する。

## AIエージェントが引き起こす構造的問題

AI エージェントに「この機能を実装して」と指示すると、多くの場合コードは動く。しかし、構造は壊れていく。

制約なしで AI にコードを書かせると、典型的に次のような問題が発生する。

* **無秩序なファイル配置**: 機能に必要なファイルが `components/`、`utils/`、`hooks/` などに散乱し、どのファイルがどの機能に属するか判別不能になる
* **責務の混在**: API 呼び出し、状態管理、UI ロジックが1つのコンポーネントに詰め込まれ、変更の影響範囲が予測できなくなる
* **依存関係の破綻**: あるモジュールが本来依存すべきでないモジュールを直接インポートし、循環参照や暗黙の結合が蓄積する

これらは「動くけど保守できないコード」の典型だ。そして厄介なのは、AI 自身はこの構造的劣化に気づかないということ。明示的な制約——ハーネス——がなければ、AI は毎回「動けばよい」コードを生成し続ける。

## FSD の基本構造

[FSD（Feature-Sliced Design）](https://feature-sliced.design/)は、フロントエンドプロジェクトのためのアーキテクチャ手法だ。ここではハーネスとしての議論に必要な最小限の構造を紹介する。

FSD は **レイヤー**（縦の階層）、**スライス**（レイヤー内の横の区分）、**セグメント**（スライス内の技術的分類）の3層構造で構成される。

![FSD の構造: レイヤー・スライス・セグメント](https://static.zenn.studio/user-upload/deployed-images/79277dca2642eee4b83c2f0c.jpg?sha=630a4d06d9a571c836e0a3542cfa912da3fb7473)  
*出典: [Feature-Sliced Design](https://feature-sliced.design/) / [MIT License](https://github.com/feature-sliced/documentation/blob/main/LICENSE)*

公式図では、同一レイヤー内のスライスが同じ色で表示されている。これはスライスが水平方向の区分であることを示す。レイヤーが縦の階層（app → shared の6層、`processes` 層は非推奨のため省略）、スライスがレイヤー内の横の区分（user, post 等）だ。

FSD の核心ルールは2つだけだ。

1. **レイヤー依存方向**: 上位レイヤーは下位レイヤーにのみ依存できる。`features` は `entities` と `shared` をインポートできるが、`widgets` や `pages` はインポートできない
2. **スライス間分離**: 同一レイヤー内のスライス（機能単位）は互いにインポートできない。例えば `features/auth` と `features/payment` は直接参照し合えない

この2つのルールが、後述するハーネスの基盤になる。

ただし、FSD が制約するのは**ファイルの配置場所**と**モジュール間の依存方向**だけだ。ファイル内部のコード品質（型安全性、エラーハンドリング等）は別の lint ルールで補完する必要がある。

## なぜ FSD がハーネスとして最適なのか

「アーキテクチャで AI を制約する」という発想自体は、FSD に限った話ではない。Clean Architecture（CA）でもレイヤー分離は定義されている。では、なぜ FSD なのか。筆者の実務経験と個人検証に基づき、4つの理由を挙げる。

### 1. 機械強制可能

FSD のルールは lint ツール（[steiger](https://github.com/feature-sliced/steiger) / [eslint-plugin-boundaries](https://github.com/javierbrea/eslint-plugin-boundaries)）でリンターエラーとして強制できる。「ドキュメントに書いてあるルール」ではなく「破ったら即座にエラーが出るルール」だ。AI エージェントはリンターエラーを検知すると自動的に修正を試みる。つまり、人間による指摘は不要だ。

### 2. スライス間分離とファイル配置の標準化

CA はレイヤー間の依存方向を定義するが、「同一レイヤー内のモジュール同士をどう分離するか」や「ファイルをどこに配置するか」は規定しない。CA を採用しても、`features/auth` が `features/payment` を直接インポートすることを防げず、ファイルの置き場所はチームの裁量に委ねられる。

FSD はスライス間の分離を**ルールとして**定義し、ファイル配置を `{layer}/{slice}/{segment}` の形式で**標準化**している。これにより、AI が「このコードはどこに置くべきか」を一意に判断できる。

### 3. エラーメッセージが AI への修正指示として機能

eslint-plugin-boundaries が出すエラーメッセージは、「何が違反しているか」と「どう修正すべきか」を暗示している。例えば、`features` レイヤーから `widgets` レイヤーへのインポートがあれば、「下位レイヤーが上位レイヤーに依存している」という違反が明示される。AI はこのメッセージを解釈して、依存を逆転させるか、共通のコードを `entities` や `shared` に移動するかを判断できる。

### 4. 制約の組み合わせで配置先が一意に定まる

レイヤー依存方向とスライス間分離——この2つの制約を同時に適用すると、あるコードを「どこに置くべきか」の選択肢が絞られる。`features/auth` から使いたい共通ロジックは `entities` か `shared` にしか置けない。`widgets` 間で共有したいコンポーネントは `features` 以下にしか置けない。制約の組み合わせが、**コードの配置先をほぼ一意に定める**。これが、人間のレビューなしでも AI が正しい構造に到達できる理由だ。

FSD の構造を踏まえて、CA との構造差を比較する。

CA はレイヤー間の依存方向（縦）のみ制約する。FSD はそれに加えてスライス間分離（横）も制約し、コードの配置先を2次元で絞り込む。上の図で CA 側のモジュール間に実線（可能）があるのは、同一レイヤー内の相互参照がルールとして禁止されていないため、こうした結合が実装できてしまうことを示している。FSD 側のスライス間に点線（禁止）があるのは、スライス間の直接参照がルールとして禁止されていることを示している。

| 比較軸 | Clean Architecture | FSD |
| --- | --- | --- |
| レイヤー依存方向 | ✅ 定義あり | ✅ 定義あり |
| スライス間分離 | ❌ 未規定 | ✅ ルールとして定義 |
| ファイル配置の標準化 | ❌ チーム裁量 | ✅ `{layer}/{slice}/{segment}` |
| 機械強制ツール | △ 汎用 lint で部分的 | ✅ eslint-plugin-boundaries / steiger |
| エラー → 構造収束 | ❌ 制約不足で収束しない | ✅ 2つの制約で一意に収束 |

ここで重要なのは、FSD の優位性が「eslint-plugin-boundaries というツールが存在すること」ではないということだ。CA のルールだって、やろうと思えば ESLint で部分的に強制できる。FSD の本質的な優位性は、**コミュニティが構造を標準化した**ことにある。レイヤーの名前、スライスの粒度、セグメントの分類——これらが合意済みだからこそ、ツールによる機械強制が実効性を持つ。

## 「エラーを直すだけで収束する」体験

ここまでの理屈を、筆者が実際に経験した典型的なパターンを基に、具体例で追体験してみよう。

### Before: AI が生成した構造

AI に「ユーザープロフィール編集機能を実装して」と指示した結果、次のような構造が生成されたとする。

```
src/
├── components/
│   ├── UserProfile.tsx        # UI + API呼び出し + バリデーション
│   ├── PaymentHistory.tsx     # UserProfileを直接import
│   └── UserAvatar.tsx
├── utils/
│   └── userApi.ts             # API関連ユーティリティ
└── hooks/
    └── useUser.ts             # ユーザー状態管理
```

一見動作するが、問題だらけだ。`components/` に全コンポーネントが混在し、`PaymentHistory` が `UserProfile` を直接インポートし、API ロジックが `utils/` に散らばっている。

### リンターエラーの発生

FSD + lint ツール（ここでは eslint-plugin-boundaries）を導入した状態でこのコードを lint すると、次のようなエラーが出る。

```
src/components/PaymentHistory.tsx
  error  Importing from 'features/user' is not allowed in 'features/payment'.
         Elements of type 'feature' can only import from: 'entity', 'shared'
         (boundaries/element-types)

src/components/UserProfile.tsx
  error  File is not in a recognized FSD layer.
         Expected path format: src/{layer}/{slice}/...
         (boundaries/no-unknown)
```

AI エージェントはこれらのエラーを受け取ると、メッセージの内容を解釈して修正を開始する。

### After: FSD 準拠構造

エラーメッセージに従って修正すると、次の構造に収束する。

```
src/
├── features/
│   ├── edit-profile/          # プロフィール編集機能
│   │   ├── ui/
│   │   │   └── EditProfileForm.tsx
│   │   └── index.ts
│   └── payment-history/       # 支払い履歴機能（分離済み）
│       ├── ui/
│       │   └── PaymentHistory.tsx
│       └── index.ts
├── entities/
│   └── user/                  # ユーザーエンティティ
│       ├── api/
│       │   └── userApi.ts
│       ├── model/
│       │   └── useUser.ts
│       └── index.ts
└── shared/
    └── ui/
        └── Avatar.tsx
```

元の `UserProfile.tsx` は UI・API 呼び出し・バリデーションが1ファイルに混在していた。FSD のセグメント構造に従って責務ごとに分割され、UI は `features/edit-profile/ui/` に、API・状態管理は `entities/user/` の各セグメントにそれぞれ配置された。`PaymentHistory` と `EditProfileForm` はスライス間分離により直接参照できなくなり、共通の `User` ロジックは `entities/user` から両方の feature に正当にインポートされている。

### 収束プロセス

このプロセスをフロー図で整理する。

ポイントは、人間がアーキテクチャを逐一指示する必要がないことだ。**リンターが「何が間違っているか」を伝え、AI が「どう直すか」を判断する**。このループが回る限り、最終的に FSD 準拠の構造に収束する。これが「エラーを直すだけで良い構造に収束する」体験だ。

## FSD ルールの導入方法

「エラー→収束」のループを自分のプロジェクトで実現する方法を紹介する。

### steiger で始める

FSD 公式チームが開発する [steiger](https://github.com/feature-sliced/steiger) を使えば、設定ファイルなしで FSD ルールを検証できる。

```
npm i -D steiger @feature-sliced/steiger-plugin
```

`package.json` に npm scripts を追加する。

```
{
  "scripts": {
    "lint:fsd": "steiger ./src"
  }
}
```

`npm run lint:fsd` でレイヤー依存方向、スライス間分離、セグメント構造を含む 20 以上のルールがゼロコンフィグで検証される。

### eslint-plugin-boundaries で ESLint に統合する

ESLint エコシステムに統合し、エディタでリアルタイムにエラー表示したい場合は [eslint-plugin-boundaries](https://github.com/javierbrea/eslint-plugin-boundaries) を使う。`default: "disallow"` 設定で許可されたインポートだけを通し、レイヤー依存方向とスライス間分離を同時に強制できる。FSD 公式の @feature-sliced/eslint-config は ESLint v9 flat config に未対応のため、eslint-plugin-boundaries を直接設定する。

### 使い分け

* **steiger**: ゼロコンフィグで FSD ルール全体（ファイル構造 + インポート + セグメント構成）を検証。まず試すならこちら
* **eslint-plugin-boundaries**: ESLint エコシステムに統合し、エディタ連携や CI パイプラインに組み込む場合

両者は補完関係にあり、併用するとより厳密なハーネスが構築できる。

## FSD 導入の Q&A

FSD をハーネスとして採用する際によくある疑問と対策をまとめる。

### Q1: スライス間でコードを共有したいが、クロスインポートが禁止されている

FSD のスライス間分離は意図的に厳しい。この制約が品質を担保している。

**features / widgets 層の場合**、4つの戦略がある。

1. **スライス統合**: 2つの feature が密結合しているなら、1つのスライスに統合する
2. **entities へ移動**: 共通のビジネスロジックを `entities` 層に切り出す
3. **上位レイヤーで合成**: `widgets` や `pages` で複数の feature を組み合わせる（IoC パターン）
4. **Public API 経由のみ**: どうしても参照が必要なら、参照先の `index.ts`（Public API）経由でのみアクセスする

**entities 層の場合**、FSD 公式は [`@x` 記法](https://feature-sliced.design/docs/guides/issues/cross-imports)を提供している。ただし、公式自身が次のように注意を促している。

> Some teams use @x as a dedicated cross-import surface for entities, but it should be treated as a last resort — a necessary compromise, not a recommended approach. Think of @x as an explicit gateway for unavoidable domain references—not a general-purpose reuse mechanism.  
> — [FSD Cross-imports ガイド](https://feature-sliced.design/docs/guides/issues/cross-imports)

まずは上記4戦略で解決を試みるべきだ。

参照: [FSD 公式 Cross-imports ガイド](https://feature-sliced.design/docs/guides/issues/cross-imports)

### Q2: barrel export（index.ts）が循環参照やバンドルサイズの問題を引き起こす

FSD では各スライスが `index.ts`（Public API）を持つことが推奨されるが、`export *` で全てを再エクスポートすると、循環参照・バンドルサイズの肥大化・dev server の低速化を招く。

**対策**: セグメント単位で分割してエクスポートする。

```
// features/auth/index.ts
// ❌ export * from './ui'
// ❌ export * from './model'

// ✅ 必要なものだけ名前付きエクスポート
export { LoginForm } from './ui/LoginForm';
export { useAuth } from './model/useAuth';
export type { AuthState } from './model/types';
```

参照: [FSD 公式 Public API](https://feature-sliced.design/docs/reference/public-api)

### Q3: shared/ が肥大化してカオスになる

`shared/` は全レイヤーから参照できるため、安易に何でも放り込みがちだ。

**対策**:

* `shared/` をセグメントで厳格に分割する: `shared/ui/`、`shared/lib/`、`shared/api/`、`shared/config/` 等
* **判断基準**: 2つ以上のスライスから使われるものだけを `shared/` に配置する。1つのスライスからしか使われないなら、そのスライス内に留めるべきだ

### Q4: Next.js のファイルベースルーティングと FSD の pages/app が衝突する

Next.js の `app/` ディレクトリと FSD の `app` レイヤーが名前衝突する。

**対策**: `src/` ディレクトリで FSD レイヤーを分離する。

```
project-root/
├── app/                    # Next.js App Router（ルーティングのみ）
│   └── dashboard/
│       └── page.tsx        # → src/pages/dashboard を re-export
└── src/                    # FSD レイヤー
    ├── app/                # FSD app レイヤー
    ├── pages/
    │   └── dashboard/
    ├── features/
    ├── entities/
    └── shared/
```

Next.js のルートファイルは FSD の `pages` からコンポーネントを re-export するだけの薄いラッパーにする。

参照: [FSD 公式 Next.js ガイド](https://feature-sliced.design/docs/guides/tech/with-nextjs)

### Q5: AI が eslint-disable コメントや any 型でエラーを回避する

AI エージェントはリンターエラーを「直す」代わりに、`// eslint-disable-next-line` や `as any` でエラーを黙殺することがある。ハーネスを無効化する最も危険なパターンだ。

**対策は3層で防御する**。

1. **eslint ルールで禁止**: [eslint-plugin-eslint-comments](https://github.com/eslint-community/eslint-plugin-eslint-comments) を導入し、`eslint-disable` コメント自体をエラーにする
2. **pre-commit hook で検出**: コミット時に `eslint-disable` の混入をチェックし、含まれていればコミットを拒否する
3. **AI への指示で禁止**: `CLAUDE.md` や `.cursorrules` 等のエージェント設定ファイルに「eslint-disable 禁止」を明記する。これはハーネスの一部——AI の行動制約——として機能する

### Q6: FSD はフロントエンド専用？バックエンドには使えない？

FSD は[公式サイト](https://feature-sliced.design/docs/get-started/overview)で「Architectural methodology for frontend projects」と宣言されている。レイヤー名（pages、widgets、features）は UI 概念を前提としており、バックエンドには MVC や DDD 等の確立パターンが既に存在するためだ。

ただし、FSD の**原則**——レイヤー依存方向とスライス間分離——は普遍的だ。リンターツール自体は言語やフレームワークに依存しないため、原理的にはバックエンドにも適用可能だ。筆者も BE への適用を模索中だが、現時点では FE での実践が中心だ。

## まとめ

本記事では、FSD をハーネス（制約）として AI エージェントの出力を構造化するアプローチを解説した。

核心は単純だ。**FSD の2つのルール（レイヤー依存方向 + スライス間分離）をリンターで強制すれば、AI はエラーを直すだけで FSD 準拠の構造に収束する**。人間がアーキテクチャを設計し、リンターがそれを強制し、AI が制約の中で実装する——この分業モデルが「動くけど保守できないコード」問題を解決する。
