## TL;DR

長いので、先にまとめておきます。

* 「どのAIが何を読めるか」はモデルの賢さではなく**データへのアクセス権**で決まる

* 直読できない Claude は「司令塔」に回した

* 外部AIに調べさせるときの肝はハルシネーション対策

* 自分で一次確認できたものと、AIの報告に依拠して未確認のものは、正直に線引きして書く
  + これをやらないと、URL なしで断定する“検証対象の動画と同じ穴”に落ちる

## 1. 発端：自社を名指しで批判するバズ動画

Syncable 上で、ある非営利団体が、毎年恒例の数千万円規模の大型寄付キャンペーンを期間限定で実施し、盛況のうちに終了しました。海外にもファンの多い、あるコンテンツ発の寄付です。

ところがその後、“海外の反応”系の YouTube 動画が、こう主張してバズりました。

> 海外勢が決済できず寄付できなかった。Syncable が海外決済にまともに対応しておらず、不満が爆発して寄付が激減する“悪循環”に陥った。

気になったのはコメント欄でした。誰も真偽を疑わず、賛同一色だったのです。もっともらしくまとまった動画だと、人はここまで素直に信じるのか、というのが最初の感想でした。

## 2. 運営の認識とのギャップ／なぜ社内データを使わなかったか

前提として、このキャンペーンは運営・団体ともに海外からの寄付増を見込んで注視していた背景があります。ちょうどそのコンテンツの海外向け展開が始まった時期で、海外ファンの参加が増えるかもしれない、と開始前から考えていました。だから海外決済まわりは事前に問題ないことを確認していましたし、キャンペーン中・終了時点でも、海外寄付者の割合や問い合わせには注目していました。

その上での運営・団体の共通認識は、シンプルに「決済そのものに問題はなかった」というものでした。事前に確認し、期間中も見ていて、決済まわりで大きなトラブルは起きていない。そこへ「決済できずに寄付が激減した」という動画が出てきたので、確認することにした、という流れです。

ここで社内データを見れば早いのですが、それだと“身内の論理”になってしまいます。「内部の数字ではこうなっている」と言ったところで、それを検証できるのは自分たちだけです。

さらに言えば、内部データには原理的な穴があります。映るのは「実際に決済まで到達した人」だけで、把握できないところで弾かれていた、エラーが出ていた、あるいはそもそも寄付の導線にたどり着けず離脱していた——そういう可能性はゼロではありません。「内部では問題なかった」は、その穴の外側までは保証しないわけです。

だからこそ、内部データの外側にある公開情報で、動画と同じ土俵に立って検証することに意味がある、と考えました。身内のデータで反論するのではなく、外から検証して落とす、という形にこだわりたかったのです。

## 3. 主張を3つに分解する

そもそも、主張を分解するには、まず動画の中身を正確に把握する必要があります。ところが相手は文字ではなく映像です。そして Claude は、YouTube のページは取れても、動画の中身（音声・字幕）までは読めません（＝実質“観られない”）。

そこで最初の一手として、動画の要約は Gemini に任せました。YouTube は Google 傘下なので、Gemini は字幕・文字起こしにアクセスしやすい——後述する「データアクセス権」の話は、実はここから始まっていました。Gemini の要約をもとに、主張を独立した3段に分けます。この分解自体が、後から効いてきます。

1. **減少という現象**：寄付額がピーク年から減ったか
2. **評価**：その減少は「失敗」と呼べるものか
3. **因果（核心）**：Syncable の海外決済が原因で海外勢が離脱 → コミュニティで不満爆発 → 過疎化の悪循環、という筋書きが成り立つか

分けた理由は、各段で独立に真偽が変わるからです。①が正しくても、③まで正しいとは限りません。

先に言ってしまうと、①と②はこの記事の本題ではありません。金額の増減そのものはブームの自然な推移などでいくらでも説明がつきますし、「減った＝失敗」という評価も短絡です。検証すべきは③——その減少を「Syncable の決済問題のせい」と断じた因果でした。以降はそこに絞ります。

## 4. Claude で数字を固める → Reddit の壁

まず [Claude](https://claude.ai/)（WebSearch / WebFetch）で公式キャンペーンページや団体の告知を取得し、①の事実関係（公式に掲げた目標は達成されている等）を確認しました。前章のとおり①②はこの記事の本題ではないので、ここは軽く押さえるに留め、すぐ③へ向かいます。

問題は③です。コミュニティの実態は、Reddit を直接読まないと分かりません。ところが、ここで壁にぶつかりました。

Claude の WebFetch からは、reddit.com 本体も old.reddit も JSON エンドポイントも読めない。検索経由でも本文は拾えませんでした。

ここで、「どのAIなら読めるのか」という問題に直面します。これが、今回の技術的な山場の入口でした。

## 5. 「読める/読めない」はデータアクセス権で決まる

今回いちばんの気づきがこれです。同じ問いでも、Reddit や X に構造的にアクセスできるAIを選べたかで結果が変わりました。賢い Claude が直読できず、用途によっては他のAIに委ねるほうが正しかった。これは「どのモデルが賢いか」ではなく「どのモデルがそのデータへのアクセス権を持っているか」の話です。

### Reddit は AIクローラを遮断、ただし Google は契約で例外

Reddit は2024年、公開コンテンツの利用に規約・ポリシーへの同意と契約を求める方針へ舵を切りました。Reddit 自身が公開している[公開コンテンツポリシー（Public Content Policy）](https://support.reddithelp.com/hc/en-us/articles/26410290525844-Public-Content-Policy)がこれで、データへ自動的・商用的にアクセスする側は、ポリシーに従い Reddit と合意（契約）する必要がある、という立て付けです（[利用には契約が必要になったと報じた TechCrunch](https://techcrunch.com/2024/05/09/reddit-locks-down-its-public-data-in-new-content-policy-says-use-now-requires-a-contract/)）。結果として、Google 以外の主要検索エンジンは実質的に締め出されました（[9to5Google](https://9to5google.com/2024/07/25/reddit-search-engine-block-google-deal/)）。

一方 Google だけは、2024年2月の提携拡大で、Reddit のデータAPI（リアルタイムの構造化コンテンツ）へのアクセス権を得ました（[Google 公式発表](https://blog.google/company-news/inside-google/company-announcements/expanded-reddit-partnership/)）。契約規模は年間6,000万ドルとされますが、こちらは公式発表にはなく報道ベースの数字です（[CBS News](https://www.cbsnews.com/news/google-reddit-60-million-deal-ai-training/)）。このデータは Gemini に使われます。

→ だから Reddit は Gemini（Google）経由なら読める。Claude からは直読できませんでした。

### リアルタイムの X は Grok が構造的に強い

[Grok](https://x.ai/grok)（xAI製）は、X のリアルタイムデータにネイティブにアクセスできます。これは xAI 公式も前面に出している Grok の中心的な特徴です（[xAI 公式ページ](https://x.ai/grok)）。

一方、現在の X はログアウト状態での閲覧が制限され、Google にインデックスされる投稿も大きく減りました（[TechCrunch](https://techcrunch.com/2023/06/30/twitter-now-requires-an-account-to-view-tweets/)。減少幅の具体的な数字は報道によって幅があります）。公開Web経由ではまともに読めません。

→ だから X 調査は Gemini では取得できず、Grok なら直接読める、という見立てになりました。「賢いが読めないAI」より「やや弱いが直接読めるAI」が勝つ、という典型例です。

### YouTube も実質 Gemini が有利（ここは推測を含む）

3章で触れた「最初の一手」がこれです。Claude では動画の中身までは読めなかった一方、動画本体の内容把握は、Gemini が文字起こしベースで的確に要約できていました。YouTube は Google 傘下なので、Gemini は字幕・文字起こしにアクセスしやすいとみられます。

ただし、「Gemini が直接動画を見ている」のか「Google が集めた情報を参照している」のかは、外からは断定できません。ここは観測事実（実際に的確に要約できた）と推測（Google 経由でアクセスできているとみられる）を分けて書いておきます。

### Claude は「司令塔」に回した

Claude からは Reddit も X も直読できませんでした。なので Claude を「自分で読む人」ではなく、「読めるAIに、捏造させないプロンプトで調査を発注し、返ってきた答えを別ソースで裏取りする司令塔」として使うことにしました。今回の勝ち筋は、ここでした。

## 6. Gemini で Reddit、Grok で X、Gemini を照合役に

### Reddit 調査（Gemini）

Reddit 調査は [Gemini](https://gemini.google.com/) に委託しました。ただしハルシネーションが怖いので、調査プロンプトは Claude に書かせました（ルールは次章）。

返ってきた回答は、動画の主張とは逆方向でした。

* 決済エラーに触れたコメントは、汎用的なものが1件のみ。「3Dセキュアで弾かれた」といった具体的な不満や炎上スレッドは見つからない。
* むしろ「海外カードで普通に寄付できた」「英語UIがある」「Amazon Pay 対応」といった成功報告や、達成を祝うムードが主流。

③の悪循環は、Reddit 上では支持されませんでした。

裏取りとして、英語のニュースメディアが同じ趣旨（決済トラブルによる炎上ではなく、ポジティブな成果として）を独立に報じているのを Claude 側で確認し、Gemini の回答と矛盾しないことを押さえました。

### X 調査（Grok と Gemini のクロスチェック）

残った穴は、不満ソースが Reddit ではなく X 由来である可能性でした。ここで Grok の出番です。正直、Grok を使う日が来るとは思っていませんでした。普段使わないツールが、データアクセス権という観点では唯一解になる、という体験でした。

ただ Grok の無料版は弱いモデルなので、同じプロンプトを Grok と Gemini の両方に投げてクロスチェックする方針にしました。一致すれば信頼でき、食い違えばその差分が次に掘る場所になります。

* Gemini は予測どおり X を取得できず、「アクセス不可」と正直に回答しました。これ自体が「X は Grok 優位」という事前判断の裏付けになりました。“できないと正直に言える”ことの価値も、地味に大事です。
* Grok は実データを取得。「Syncable で弾かれた／3Dセキュアで通らない」といった不満投稿は、日英の複数の検索語でも見つからない。むしろ公式が海外支援者向けの英語告知（領収書はメール送付）を出しており、達成祝い・感謝の投稿が主流でした。

裏取りとして、Grok が挙げた公式の海外向け英語告知が、Instagram にも同一内容でミラー投稿されているのを Claude 側で確認しました。少なくとも告知の存在は Grok の捏造ではない、と判断できます。一方、個別の「いいね数」などは Grok の報告のままで当方は未確認、と線引きしておきました。

## 7. 外部AIに調べさせるときのハルシネーション対策

外部AIに調査を委託する以上、最大のリスクはハルシネーション（それっぽい嘘）です。各AIに課した共通ルールは、だいたい次のようなものでした（固有名詞は伏せています）。

```
# 厳守ルール（ハルシネーション防止）
1. 実際にアクセスして読めたページ/投稿のみを根拠にする（記憶・一般論で補完しない）
2. 主張ごとに必ず出典URL（パーマリンク）を併記。URLを示せない情報は書かない
3. 重要な記述は要約でなく原文引用を最低1つ添える（引用と要約を区別）
4. 見つからなければ「確認できませんでした」と明記する（減点ではなく正しい回答だと伝える）
5. 「事実（確認できたこと）」と「推測・解釈」を別の見出しに分ける
6. 件数・金額はページ記載の数字をそのまま転記（換算時は元数字も併記）
7. 年・日付を明示し、過去年と混同しない
```

設計の意図をいくつか補足します。

* 4 が核です。「見つからなければ確認できずと書け、それは正しい回答だ」と事前に伝えるのが、ハルシネ抑制でいちばん効きました。「何か答えを出さなければ」という圧を外してやるイメージです。
* 確定済みの数字は“前提”として渡しました。AIに金額を再調査させると、換算ミスや年の取り違えでハルシネが起きやすいからです。検証対象を「定性的な実態（コミュニティの空気）」に絞らせました。
* 因果（③が①のせいか）は必ず「あなたの解釈」として明示させました。事実とAIの推論を混ぜさせないためです。
* 「散発か、多発・拡散か」を独立項目にしたのも効きました。動画の真偽は「不満が存在したか」より「激減を起こすほど多発・拡散したか」で決まります。散発が数件なら「事実を誇張」、ほぼ無いなら「ほぼ事実無根」、多発・拡散なら「動画にも一理」と、判定が分かれる設計にしました。

## 8. 結論

最終的に、独立した複数のソース（公式の Syncable ／第三者の英語報道／Reddit〔Gemini〕／X〔Grok〕／Instagram〔Claude 裏取り〕）が、すべて同じ方向に収束しました。

* **①減少という現象**：減っている部分があるのは事実（ただし本題ではない）
* **②評価**：公式に掲げた目標は達成されている。「減った＝失敗」という評価は短絡で、動画の問題は減少そのものではなく、その原因を決済問題（③）に帰した点にある
* **③因果（核心）**：全ソースで支持されず。むしろ公式は海外向けの英語告知や領収書メール対応を能動的に行っており、「海外対応していない」という前提が事実と逆

そもそも寄付額が動く理由はいくらでもあります（コンテンツのブームが自然に落ち着いてきた、など）。少なくとも確かなのは、「決済の仕組みが変わったから」ではないということです。決済は全期間ずっと Syncable のままで、プラットフォームが変わっていないのに特定の年だけ決済が主因になる、という説明は論理的に弱い。

全体としてこの動画は、「正しい数字」に「事実と乖離したストーリー」を被せた、誇張・ミスリード型のコンテンツだった、というのが私の結論です。

## 9. エピローグと、ひとつの自戒

調査結果（背景・手段・結果・まとめ・限界）は、社内の GitHub Issue にソースURL付きでコメントとして記録し、PM には Slack で簡潔に報告しました。

ひとつ、自戒があります。最初に Issue へ投稿した版は、結論を急いで URL を載せ忘れていました。読み返して「これでは、自分が検証していた動画と同じだ」と気づき、全ソース明記版に差し替えました。

URL なしで断定した瞬間に、検証する側も、される側と同じ穴に落ちます。これは今回いちばん刺さった教訓でした。

## まとめ

* 「どのAIが読めるか」は賢さでなくデータアクセス権で決まる。Reddit は Gemini、リアルタイム X は Grok、という構造的な向き不向きがある
* 直読できないAI（今回は Claude）は、読めるAIに発注して裏取りする「司令塔」として使うと、単一AIより精度高く検証できた
* 外部AIにはハルシネ対策を徹底する。とくに「確認できなければ確認できずと書かせる」のが効く
* 自分で確認できたものと、AIの報告に依拠して未確認のものは、正直に線引きする

身内のデータで反論するのではなく、外から検証して落とす。結果的に、そのプロセス自体が、いちばんの学びでした。もっともらしい動画が、誰にも疑われずに広がっていく。その怖さと、それを外形的に崩す手段の両方を、実地で体験した記録でした。
