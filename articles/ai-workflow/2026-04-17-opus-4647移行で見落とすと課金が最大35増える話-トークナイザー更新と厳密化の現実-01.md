## TL;DR

* **Opus 4.7 は `model` 名を書き換えるだけでは済まない**。トークナイザーが更新されており、同じ入力テキストが <strong>1.0〜1.35倍</strong> のトークン数にマップされる。価格据え置きでも請求額は増える可能性がある。
* **xhigh / max effort では思考量が増える**。後半ターンの推論が深くなる設計で、出力トークンも従来より多い。Task budgets（public beta）を併用しないと長時間エージェントで想定外の請求が飛んでくる。
* **Instruction following が厳密化している**。Opus 4.6 向けの「曖昧でも汲んでくれる」プロンプトは Opus 4.7 でブレる。CLAUDE.md とシステムプロンプトの棚卸しは避けて通れない。
* それでも移行すべきかと問われれば Yes。SWE-bench Verified +6.8pt、SWE-bench Pro +10.9pt、CursorBench +12pt の伸びは、同価格帯のモデルで見てきた中でも屈指。ただし<strong>移行前に自社ワークロードで実測</strong>する工程を絶対にスキップしてはいけない。

## なぜこの記事を書いたか

2026年4月16日、Anthropic が [Claude Opus 4.7](https://www.anthropic.com/news/claude-opus-4-7) をリリースした。公式発表を読むと「Opus 4.6 からの直接アップグレード」「価格据え置き」「SWE-bench Verified 87.6%」と景気のいい文言が並ぶ。

自分は普段 Claude Code で [ComfyUI](https://github.com/hiroki-abe-58/ComfyUI-Win-Blackwell) まわりのコード生成や複数記事の同時執筆をやっていて、Opus 4.6 の挙動にはそれなりに慣れている。だから「モデル名書き換えるだけでしょ」と軽く構えていたのだが、公式の [migration guide](https://platform.claude.com/docs/en/about-claude/models/migration-guide#migrating-to-claude-opus-4-7) を読み込むうちに「これは事前に注意喚起しないとまずい」と感じた項目が3つ出てきた。

特に、**トークナイザー更新とeffortレベル後半ターンでの思考量増加の組み合わせは、ベンチマークが良くなった一方で請求額が静かに跳ねる可能性がある**。この記事はそのリスクを整理し、移行する側の技術者が何をすべきかを書く。

本記事は2026年4月17日時点の情報に基づく。筆者はまだ Opus 4.7 に本番ワークロードを流していないため、数値はすべて Anthropic 公式と主要ベンダーの自社評価からの引用だ。実測はこれから各自でやる必要がある、という前提で読んでほしい。

## 罠①：トークナイザー更新 ─ 同じテキストが最大35%重くなる

これが一番静かで、一番効く。

### 何が変わったのか

Opus 4.7 は更新されたトークナイザーを採用している。公式 migration guide は次のように言っている（要約）：

> Opus 4.7 は更新されたトークナイザーを使用している。これによりテキスト処理は改善されるが、トレードオフとして、同じ入力がコンテンツタイプ次第で 1.0〜1.35倍 のトークン数にマップされる可能性がある。

### どのコンテンツで影響が大きいか

公式は「コンテンツタイプによって」としか言っていないので、以下は筆者が過去のトークナイザー更新（Claude 2→3、GPT-4→GPT-4o など）のパターンから推定したもの。数字は根拠ある実測ではなく目安として読んでほしい：

| コンテンツタイプ | 推定トークン増加率 | 根拠 |
| --- | --- | --- |
| 英文プレーンテキスト | 1.0〜1.05倍 | 過去の更新では英文はあまり変わらない |
| 日本語プレーンテキスト | 1.1〜1.2倍 | 多バイト文字のトークナイズ改善の副作用 |
| ソースコード（Python/JS） | 1.15〜1.35倍 | 記号・インデントの扱い変更の影響大 |
| JSON / YAML 構造化データ | 1.15〜1.3倍 | 構文記号のトークン分割の影響 |
| Markdown ドキュメント | 1.05〜1.15倍 | 英文ベースなら軽め |

自分のようにソースコード中心のワークロードを流している人は、**上限側（1.35倍）を想定して予算を引き直す**のが安全側の判断になる。

### 実測すべきこと

移行前に以下の手順で自社ワークロードのトークン増加率を計測することを強く勧める。

実測スクリプトの例（Python）

```
import anthropic

client = anthropic.Anthropic()

def count_tokens(model: str, text: str) -> int:
    """指定モデルのトークナイザーでテキストのトークン数を数える"""
    response = client.messages.count_tokens(
        model=model,
        messages=[{"role": "user", "content": text}],
    )
    return response.input_tokens

def measure_ratio(samples: list[str]) -> None:
    """Opus 4.6 と 4.7 のトークン数比を計測する"""
    total_46 = 0
    total_47 = 0
    for i, text in enumerate(samples):
        tokens_46 = count_tokens("claude-opus-4-6", text)
        tokens_47 = count_tokens("claude-opus-4-7", text)
        ratio = tokens_47 / tokens_46 if tokens_46 else 1.0
        print(f"Sample {i}: 4.6={tokens_46}, 4.7={tokens_47}, ratio={ratio:.3f}")
        total_46 += tokens_46
        total_47 += tokens_47

    print(f"\n全体: 4.6={total_46}, 4.7={total_47}, ratio={total_47/total_46:.3f}")

# 自社プロンプトの代表サンプルを並べる
samples = [
    open("prompts/system_prompt.txt").read(),
    open("prompts/typical_user_query.txt").read(),
    open("prompts/long_code_sample.py").read(),
    # ...
]

measure_ratio(samples)
```

1週間分の実プロンプトログを走らせて、<strong>自社ワークロード固有の増加係数</strong>を出す。Anthropic の「1.0〜1.35倍」は広い幅なので、自分のケースがどこに落ちるかは実測しないと分からない。

## 罠②：xhigh / max effort で思考量が増える ─ 後半ターンの出力トークン膨張

### 新設された effort level

Opus 4.7 では `high` と `max` の間に **`xhigh`** が新設された。Anthropic 公式はコーディング・エージェンシック用途では `high` か `xhigh` から始めることを推奨している。Claude Code では全プランでデフォルトが `xhigh` に引き上げられた。

| Effort | 位置づけ |
| --- | --- |
| `low` | 軽い応答 |
| `medium` | 通常のQA |
| `high` | 複雑な推論 |
| **`xhigh`**（新設） | **ハードコーディング・長時間エージェント** |
| `max` | 最難関タスク |

### 見落としやすい仕様変更

公式 migration guide が明記しているのはここ：

> Opus 4.7 は高 effort レベルで、特にエージェンシック設定における後のターンにおいてより多く思考する。これは難しい問題に対する信頼性を向上させるが、より多くの出力トークンを生成することを意味する。

つまり `xhigh` と `max` では、<strong>マルチターン会話の後半で前ターンを参照しながら深く考える設計</strong>になっていて、ターンを重ねるほど1ターンあたりの出力トークンが増える。

### Task budgets で安全弁を張る

同時に発表された **Task budgets（public beta）** は、この罠への直接的な対策として用意されている。自律的なエージェントに対してトークン消費のハード上限を設定できる。Claude API のドキュメント側で詳細が公開されているはずなので、Auto mode を本番で使っているなら必ずセットで導入したほうがいい。

筆者の所感としては、**Task budgets なしで Opus 4.7 の xhigh を本番 Auto mode に入れるのは、スピードリミッターなしで新型スーパーカーを深夜の高速道路に放つのと同じ**だと考えている。ベンチマークスコアが上がったことと、課金制御が効くことは別の話だ。

### とはいえ、効率は実は改善している

誤解のないよう書いておくと、トークンあたりの性能は純粋に改善している。Anthropic 内部の coding eval では全 effort レベルで「トークン使用量あたりのスコア」が Opus 4.6 より良くなっている。

さらに Hex の CTO Caitlin Colgrove は「低effort の Opus 4.7 は中effort の Opus 4.6 とおおよそ等価」と述べている。

→ つまり、<strong>移行と同時にeffortを1段下げる</strong>のが最初に試すべき運用パターンだ。Opus 4.6 で `high` を使っていたなら Opus 4.7 では `medium` から測る。同じ品質が取れていればコストは下がる、取れていなければ `high` に戻す。これだけで罠②の影響は大きく緩和できる。

## 罠③：Instruction following 厳密化 ─ "察してくれ"が通用しなくなった

### 何が変わったか

ここは罠というより<strong>仕様の向上</strong>だが、影響範囲が広い。公式はこう書いている：

> Opus 4.7 は指示の従順さが大幅に改善されている。興味深いことに、これは以前のモデル向けに書かれたプロンプトが予期しない結果を生む可能性があることを意味する。以前のモデルが指示を緩く解釈したり一部をスキップしていた箇所で、Opus 4.7 は指示を文字通りに受け取る。ユーザーはプロンプトとハーネスをそれに応じて再調整する必要がある。

要するに、Opus 4.6 の「雑に書いてもいい感じに解釈してくれる」挙動は Opus 4.7 では発動しない。**書かれていないことはやらない**。

### 典型的に踏むパターン

自分がすぐ想像できたもの：

1. **「コードを修正してください」とだけ書いてあった場合**

   * Opus 4.6：テストも暗黙に追加してくれることがあった
   * Opus 4.7：テストについて書かれていないので、テストは追加しない
2. **「簡潔に書いて」とだけ書いてあった場合**

   * Opus 4.6：コメントも含めて全体を短くしてくれた
   * Opus 4.7：「簡潔に」が何の単位で、どこまで削るのかが不明瞭なので、ブレる
3. **複数項目の指示で一部が省略されている場合**

   * Opus 4.6：文脈から全部を要求していると解釈してくれた
   * Opus 4.7：明記されている項目のみに応答する

### 再調整のチェックリスト

自分の CLAUDE.md や system prompt を見直すときに意識すること：

!

* 「してほしいこと」を全部箇条書きにする（含みを持たせない）
* 「してはいけないこと」も明示する（デフォルト挙動に頼らない）
* 暗黙の前提条件を全部書き出す（「このリポはPython製」「テストは必ず追加」など）
* 出力フォーマットを具体的に指定する（「簡潔に」より「3項目以内の箇条書き」）
* 役割・コンテキストを冒頭で与える（「あなたは〇〇の専門家として」）

この棚卸しには半日から1日を見ておいたほうがいい。本番に Opus 4.7 を入れる前の必須工程だ。

## それでも移行すべき理由 ─ ベンチマークと現場報告

ここまで罠の話を書いてきたが、筆者は「それでも移行すべき」と考えている。理由は2つある。

### 理由1：ベンチマークの伸びが大きすぎる

公式発表から抜き出した主要な数字：

| ベンチマーク | Opus 4.6 | Opus 4.7 | 差分 |
| --- | --- | --- | --- |
| SWE-bench Verified | 80.8% | **87.6%** | **+6.8pt** |
| SWE-bench Pro | 53.4% | **64.3%** | **+10.9pt** |
| CursorBench | 58% | **70%** | **+12pt** |
| OSWorld-Verified（Computer Use） | 72.7% | **78.0%** | +5.3pt |
| CharXiv（科学図表理解） | 69.1% | **82.1%** | +13.0pt |
| XBOW Visual Acuity | 54.5% | **98.5%** | **+44.0pt** |
| GPQA Diamond | 91.3% | **94.2%** | +2.9pt |
| Finance Agent v1.1 | 60.7% | **64.4%** | +3.7pt（SOTA） |

SWE-bench Pro での +10.9pt は特に注目に値する。Pro は Verified より難しく、memorization 汚染の影響が少ない上位互換のベンチマークだ。<strong>難しい問題ほど改善幅が大きい</strong>のは、単なるベンチマークハックではなく本質的な能力向上の兆候と考えている。

### 理由2：現場パートナーの報告が一致して肯定的

Anthropic 公式発表から、特に具体性のある声を引用する：

* **Rakuten（梶悠介氏）**：Rakuten-SWE-Bench で本番タスク解決数が **Opus 4.6 の3倍**、Code Quality / Test Quality 共に二桁ポイント向上。
* **XBOW（Oege de Moor CEO）**：視覚的知覚ベンチマークで Opus 4.6 の 54.5% から **Opus 4.7 で 98.5%**、これまで使えなかった用途一式が一気に実用圏に。
* **Vercel（Joe Haddad）**：Opus 4.6 より正確で完全、自身の限界について目に見えて正直。Claude 以前のモデルでは見られなかったシステムコードの証明まで行う新しい挙動。
* **Hex（Caitlin Colgrove CTO）**：Hex が評価した最も強力なモデル。データが欠落している際に plausible-but-incorrect fallback を返さず、正直に欠落を報告する。

特に最後の「plausible-but-incorrect fallback を返さない」は、エージェンシック用途における信頼性の本質だ。もっともらしい嘘をつくモデルは、長時間エージェントとして使えない。

### 逆張り気味の注意点

一方で、以下は Opus 4.7 の弱点として記録しておくべきだと思う：

| ベンチマーク | Opus 4.6 | Opus 4.7 | 競合 |
| --- | --- | --- | --- |
| BrowseComp（エージェンシックWeb検索） | 83.7% | **79.3%（後退）** | GPT-5.4 Pro: 89.3% |
| Terminal-Bench 2.0 | 65.4% | 69.4% | GPT-5.4: 75.1%※ |
| MMMLU（多言語Q&A） | - | 91.5% | Gemini 3.1 Pro: 92.6% |
| CyberGym | 73.8% | 73.1%（わずかに後退） | - |

※ GPT-5.4 の Terminal-Bench は self-reported harness で直接比較不可、と Anthropic 自身が注釈を入れている。

BrowseComp の後退は、Web検索ベースのエージェントを本番運用している場合に無視できない。自分のワークロードがここに引っかかるなら、**移行の前に必ず自社データで A/B 測定**すべきだ。

CyberGym のわずかな後退について、Anthropic は「訓練中にサイバー能力を意図的に差別的に削減する実験を行った結果」と認めている。Project Glasswing で明らかになった Mythos Preview のサイバー能力リスクを踏まえ、**Opus 4.7 はあえてこの能力を鈍らせている**。これは"弱点"というより設計思想だ。

## 現場エンジニアの処方箋

ここまでの話をまとめて、移行時にやるべきことをチェックリストにする。

### 移行前（1日〜1週間）

!

1. 自社ワークロードの代表プロンプト10〜30件を用意する
2. Opus 4.6 と Opus 4.7 の両方でトークン数を計測（上記スクリプト参照）、<strong>自社固有の増加係数</strong>を出す
3. CLAUDE.md と system prompt の棚卸し、暗黙の前提を全部明記
4. Task budgets（public beta）の設定値を決める（タスク種別ごとにトークン上限）
5. BrowseComp 系のワークロードを使っているなら、GPT-5.4 Pro との比較検討

### 移行初日

!

1. まず低〜中effortから試す。Opus 4.6 で `high` を使っていたなら Opus 4.7 では `medium`
2. 代表プロンプト10件で出力品質を Opus 4.6 と比較
3. 品質が落ちていれば effort を1段上げる（`medium` → `high`）
4. `/ultrareview` を試す（Pro/Max は3回無料）

### 移行1週間後

!

1. 1週間分の請求額を Opus 4.6 時代と比較
2. 増加率が推定レンジ内（1.0〜1.35倍）に収まっているか検証
3. 収まっていなければ、effort を下げる or Task budgets を厳しくする
4. instruction following で予期しない挙動があれば、プロンプトを再調整

## 考察：Opus 4.7 は"Mythos 公開の踏み台"だと割り切ると見え方が変わる

最後に筆者なりの見立てを書く。

公式発表のこの一文が気になっている：

> 私たちは Mythos Preview のリリースを限定したままにし、より能力の低いモデルで新しいサイバー安全対策をまずテストすると述べた。Opus 4.7 は最初のそのようなモデルだ。

Opus 4.7 は単独のフラグシップというより、\*\*Mythos クラスを広く展開するために必要な safety 学習の"最初の実戦投入"\*\*という位置づけで読める。実際、instruction following 厳密化も、出力を自己検証する挙動も、CyberGym の意図的な削減も、全部「Mythos を世に出すために先に Opus 4.7 で学ばなきゃいけないこと」の具体化に見える。

この見立てに立つと、Opus 4.7 の"地味な進化"が逆に納得できる。派手な新機能の乱れ撃ちではなく、**プロダクション運用に必要な信頼性の積み上げ**が Anthropic の当面のプライオリティなのだろう。価格据え置きで出してきたのも、"広く使ってもらってデータを集めたい"という意図と整合する。

個人的には、こういう地味で実直なリリースのほうが技術者として信頼できる。派手なベンチマークハックではなく、**本番で静かに仕事が減る**類の改善だ。

## まとめ

* Opus 4.7 への移行は、モデルID書き換えだけでは完了しない
* トークナイザー更新（+0〜35%）、xhigh/max での思考量増加、instruction following 厳密化の3つは必ず事前計測・対策する
* それでも移行すべき理由は、SWE-bench 系の大幅改善と、現場パートナーからの一貫した肯定的報告
* Task budgets を併用し、effortを1段下げるところから始めるのが安全な運用パターン
* Opus 4.7 は Mythos 公開の踏み台として読むと、リリースの設計思想が腑に落ちる

自分は来週、自社ワークロードで実測を回す予定だ。結果が出たら別記事で追う。
