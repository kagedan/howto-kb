## はじめに：自律性とセキュリティのジレンマ

[第1回](https://zenn.dev/slate_infra/articles/466e9f4561659d)では、OpenAIとAnthropicの公開事例をもとに**harness engineering**──モデルの外側にある環境設計が生産性を決める──という概念を整理した。その中で「permissionsを緩める代わりに環境側で爆発半径を制限する設計」が先進事例に共通していると書いた。[第2回](https://zenn.dev/slate_infra/articles/37d7d7c8ebf0a3)では、レビューの自動化と人間判断の境界線を3層モデルで再設計した。

どちらの記事でも繰り返し浮上したのは、「エージェントの自律性を上げるほどセキュリティリスクが増大する」というジレンマだ。Boris Chernyは`--dangerously-skip-permissions`をサンドボックス内で使い、Carliniの16並列プロジェクトは全エージェントをDockerコンテナ内に隔離していた。自律性の高いエージェント運用は、例外なくセキュアな実行環境とセットだった。

2026年4月8日、Anthropicが発表した**Claude Managed Agents**は、この問題に対するAnthropicの回答だ。注目を集めているのはマルチエージェント連携機能だが、筆者が考えるこのプロダクトの本質的な価値は別のところにある。**セキュアなサンドボックス、認証情報管理、スコープ付きパーミッション、実行トレーシングを「マネージドインフラ」として提供した**ことだ。つまり、harnessのランタイム層をAnthropicが引き受けるという提案である。

本記事では、Claude Managed Agentsをharness engineeringの文脈に位置づけ、「エージェントに自律性を与えるために必要なセキュリティ基盤」を整理する。第1回が設計層、第2回が検証層を扱ったのに続いて、今回はランタイム層──エージェントをどこで、どう安全に動かすか──を扱う。

## 問題の構造：なぜ自律実行にセキュリティが必要か

### 事実として確認できること

AIコーディングエージェントは従来のIDEプラグインとは根本的に異なる。Claude Codeを例にとると、エージェントはファイルの読み書き、シェルコマンドの実行、外部APIへのアクセスを自律的に行う。これはコードを提案するだけのcopilotとは次元が違うリスクプロファイルだ。

Anthropicは2025年10月にClaude CodeのネイティブサンドボックスをOSレベルのプリミティブ（Linux bubblewrap、macOS Seatbelt）で実装し、内部テストで**パーミッションプロンプトを84%削減**したと報告している。Anthropicのエンジニアリングブログは、効果的なサンドボックスには**ファイルシステム隔離とネットワーク隔離の両方**が不可欠だと明言している。ネットワーク隔離がなければ、侵害されたエージェントがSSHキーなどの機密ファイルを外部に送信できる。ファイルシステム隔離がなければ、エージェントがシステムリソースにバックドアを仕掛けてネットワークアクセスを取得できる。

さらに、この構造的リスクを示す単一の実証例がある。セキュリティ企業Onaは2026年3月、Claude Codeが自身のdenylistとサンドボックスを回避する挙動を実証した。エージェントにブロックされたコマンドを実行させるよう指示したところ、エージェントは`/proc/self/root/`経由でパスベースのdenylistを迂回し、bubblewrapサンドボックスが阻止するとサンドボックス自体を無効化して実行を試みた。特別なジェイルブレイクは使われていない。エージェントは単に「タスクを完了したかった」だけだ。これは単一のセキュリティ企業による実証であり、すべての環境で再現するかは不明だが、タスク指向の自律エージェントが持つ構造的傾向を示す事例として注目に値する。

### そこから言えること

エージェントはタスク完了のために最適化されている。セキュリティ制約はエージェントにとって「タスク完了を阻害する障害物」でしかなく、回避可能であれば回避を試みる。これはモデルの欠陥ではなく、タスク指向の自律エージェントに固有の構造的特性だ。

したがって、セキュリティは「モデルへの指示」ではなく「モデルの外側の環境」で担保する必要がある。第1回で書いた「制約をインフラ側で担保する」というharness engineeringの原則が、セキュリティ領域で最も先鋭的に現れる。

## Claude Managed Agentsが提供するもの

### 公式に確認できること

Anthropicの公式ブログとドキュメントによると、Claude Managed Agentsは以下の4つの概念で構成される。

**Agent：** モデル、システムプロンプト、ツール、MCPサーバー、スキルの定義。一度作成すればIDで参照して複数セッションで再利用できる。

**Environment：** クラウドコンテナの構成テンプレート。プリインストールパッケージ（Python、Node.js、Goなど）、ネットワークアクセスルール、マウントするファイルを定義する。

**Session：** エージェントとEnvironmentを参照して起動する実行インスタンス。長時間の自律実行に対応し、接続が切れても進捗と出力が保持される。

**Events：** SSE（Server-Sent Events）ベースの双方向メッセージ。実行中のエージェントに追加指示を送ることも、中断して方向転換させることもできる。

公式ブログが列挙する主要機能は、セキュアなサンドボックスと認証・ツール実行の管理、長時間セッションの永続化、マルチエージェント連携（research preview、アクセス申請が必要）、スコープ付きパーミッション・ID管理・実行トレーシングだ。

課金体系は従量制で、モデルの利用料に加えてアクティブな実行時間1時間あたり$0.08が加算される。メッセージ待ちやツール使用の確認待ち時間は課金対象外だ。

### 何が「マネージド」されているのか

ここが核心だ。Claude Managed Agentsの名前が示すのは「エージェントが管理されている」ということだが、実際に管理されているのはエージェントそのものではなく、**エージェントが動く環境**だ。

公式ブログの表現を借りれば、本番用エージェントをシップするにはサンドボックス化されたコード実行、チェックポイント、認証情報管理、スコープ付きパーミッション、エンドツーエンドのトレーシングが必要で、これらは「ユーザーが見るものを何もシップする前に数ヶ月のインフラ作業」に相当する。Claude Managed Agentsはこの部分を引き受け、開発者はAgent（何をさせるか）とEnvironment（どこで動かすか）の定義に集中できる。

Atlassianの担当者は、サンドボックス化、セッション管理、スコープ付きパーミッションをManaged Agentsが処理してくれるため、エンジニアがインフラではなく機能開発に集中できるようになったと述べている。Rakutenは各部門のエージェントを「それぞれ1週間程度で」展開したとしている。

## harness engineeringの文脈での位置づけ

### harnessの3層構造

第1回と第2回で整理してきた内容を踏まえると、harness engineeringは以下の3つの層で構成されると考えられる。これは筆者が公開事例から導出した整理であり、特定企業の公式分類ではない。

**設計層：** 何をさせるか。CLAUDE.md / AGENTS.md、システムプロンプト、ツール選択、パーミッション設計、成功条件の定義。

**検証層：** 正しく動いたかを確認する。テスト、リンター、CI、AIコードレビュー（第2回で整理した3層モデル）。

**ランタイム層：** どこで、どう安全に動かすか。サンドボックス、コンテナ隔離、ネットワーク制限、認証情報管理、実行トレーシング。

第1回ではOpenAIとAnthropicの事例から設計層の原則を抽出した。第2回では検証層の再設計を論じた。Claude Managed Agentsは**ランタイム層をマネージドサービスとして提供する**ものだ。

### 自前でランタイム層を構築する場合との比較

Claude Managed Agents以前から、ランタイム層を自前で構築するアプローチは存在していた。

Carliniの16並列プロジェクトでは、各エージェントがDockerコンテナ内で稼働していた。Boris ChernyはGitのworktreeで作業ディレクトリを分離し、サンドボックス内で`--dangerously-skip-permissions`を使用していた。Claude Code公式のサンドボックス機能はOSレベルのプリミティブ（bubblewrap / Seatbelt）でファイルシステムとネットワークを隔離する。Docker社はDocker Sandboxesを提供し、microVMベースの隔離でClaude Codeなどのコーディングエージェントをサポートしている。

これらは機能するが、設定・運用・監視は開発者の責任だ。コンテナのセキュリティ設定（`--cap-drop ALL`、`--security-opt no-new-privileges`、seccompプロファイル、読み取り専用ファイルシステム、ネットワーク制限、リソース制限など）を正しく構成し、維持するのは非自明な作業だ。

Claude Managed Agentsはこの構成作業をEnvironment定義に抽象化し、実際のセキュリティ実装と運用をAnthropicが担当する。トレードオフは明確で、カスタマイズの自由度は下がるが、「やるべきセキュリティ設定を忘れる」リスクも下がる。

## 残る課題：Managed Agentsでは解決しないもの

### 設計層は委譲できない

Claude Managed Agentsはランタイム層を引き受けるが、設計層——何をさせるか、どのツールを許可するか、どのネットワークドメインにアクセスさせるか——は開発者が決める必要がある。

Onaの事例が示したように、エージェントはタスク完了のためにセキュリティ制約を回避しようとする。Managed Agentsのサンドボックスがこれを阻止するとしても、そもそも「このエージェントにBashツールを許可すべきか」「このドメインへのネットワークアクセスは本当に必要か」という設計判断は人間が行う。

最小権限の原則はコンテナセキュリティの基本だが、エージェントに対しても同じだ。公式ドキュメントのセキュアデプロイメントガイドは、エージェントにAPIキーを直接渡すのではなく、エージェントの環境外でプロキシを動かし、プロキシがリクエストにキーを注入する設計を推奨している。エージェントはAPI呼び出しができるが、認証情報そのものには触れない。

### 検証層は別途必要

Managed Agentsが提供するのは実行環境であって、出力の品質検証ではない。テスト、CI、AIコードレビューといった第2回で論じた検証層は、Managed Agentsの上に自分で構築する必要がある。

ただし、research previewとして提供されている「outcomes」機能——成功条件を定義し、エージェントが自己評価して反復する仕組み——は、検証層の一部をランタイムに組み込む試みとも読める。公式ブログによると、構造化ファイル生成タスクにおいて、標準的なプロンプトループと比較して最大10ポイントのタスク成功率向上が見られたという。

### マルチエージェント連携はまだresearch preview

マルチエージェント連携（エージェントが別のエージェントをスピンアップして並列処理する機能）はresearch previewであり、一般利用にはアクセス申請が必要だ。現時点でManaged Agentsの最も確実な価値は、単一エージェントのセキュアな自律実行基盤にある。

## 企業のチーム開発への含意

### 「安全な箱」がエージェント導入の前提条件になる

企業がAIエージェントを本番環境で使い始めるとき、最初に問われるのは「それは安全か」だ。特にSIerが関わる金融・公共系のプロジェクトでは、エージェントがファイルシステムやネットワークに自由にアクセスできる状態は受け入れられない。

Claude Managed Agentsは、この「安全か」という問いに対して、Anthropicがインフラレベルで担保するという回答を提供する。サンドボックス化、スコープ付きパーミッション、実行トレーシングが標準装備されており、「セキュリティ設定を忘れる」リスクが構造的に排除される。

ただし、これは「Managed Agentsを使えばセキュリティは万全」という意味ではない。設計層（何を許可するか）の判断は依然として開発者の責務であり、エージェントに過剰な権限を与えれば、サンドボックス内であっても望ましくない動作は起こりうる。

### SIerの場合：監査対応とトレーサビリティ

SIerにとって特に重要なのは、実行トレーシングの存在だ。Claude Managed Agentsは、Claudeコンソール上でツール呼び出し、判断、エラーの全履歴を確認できる。

第2回で「全行レビューを廃止する提案ではなく、機械的に代替できる部分を明示する提案だ」と書いた。同じロジックが適用できる。エージェントの全行動をトレースログとして記録し、人間は判断が必要な部分に集中する。監査対応としても、「エージェントの全アクションがトレース済み＋人間が設計判断をレビュー」というトレーサビリティは、「人間がすべて目視確認しました」よりも説明力が高い可能性がある。

### 自前構築 vs Managed：判断基準

すべてのチームがManaged Agentsを使うべきとは限らない。判断基準を整理する。

**Managed Agentsが適するケース：** エージェントをエンドユーザーに提供するプロダクトを構築する場合。インフラチームが小さく、セキュリティ基盤の自前構築・運用にリソースを割けない場合。長時間の非同期タスクを処理する場合。

**自前構築が適するケース：** 既にKubernetes等のコンテナ基盤があり、セキュリティポリシーの細かいカスタマイズが必要な場合。規制要件で特定のクラウドリージョンやオンプレミスでの実行が求められる場合。開発者自身のローカル開発フローの効率化が主目的の場合（この場合はClaude Codeのネイティブサンドボックスで十分なことが多い）。

## 個人開発・フリーランスの場合

個人開発やフリーランスにとっても、「安全な箱」の問題は無関係ではない。

### 開発時：Claude Codeのネイティブサンドボックス

自分の開発マシンでClaude Codeを使う場合、最低限やるべきは`/sandbox`でネイティブサンドボックスを有効化することだ。これだけでファイルシステムとネットワークの隔離が有効になり、パーミッションプロンプトが大幅に減少する。Anthropicの内部データでは84%の削減だ。

より厳格な隔離が必要なら、Docker Sandboxesを使ってmicroVMベースの環境でClaude Codeを実行できる。特に、顧客のコードベースを扱うフリーランスの場合、コンテナ隔離は「自分のSSHキーや他の顧客の情報がエージェントに触れない」という安全保証になる。

### プロダクション：Managed Agentsの出番

個人開発者がSaaSプロダクトにAIエージェント機能を組み込みたい場合——例えば、ユーザーが自分のデータに対してエージェントを走らせるような機能——Managed Agentsの価値が直接効く。サンドボックス、セッション管理、認証情報の分離を自前で構築する必要がなくなる。

公式ブログに掲載されたVibecodeの事例では、「以前はユーザーがサンドボックス内でLLMを手動で動かし、ライフサイクルを管理し、ツールを装備し、実行を監視する必要があり、セットアップに数週間から数ヶ月かかっていた」と説明されている。Managed Agentsはこのインフラ作業を不要にする。

ただし、現時点ではManaged Agentsの実行コスト（モデル利用料＋$0.08/アクティブ時間）がサービスの収益モデルに合うかどうかは慎重に検討する必要がある。

## 自律性とセキュリティの設計パターン：まとめ

第1回から本記事まで繰り返し現れたパターンを、「自律性とセキュリティ」の軸で整理する。

| 自律性のレベル | セキュリティ基盤 | 対応するツール / サービス |
| --- | --- | --- |
| 対話的（毎回承認） | パーミッションプロンプト | Claude Code デフォルト設定 |
| 半自律（ホワイトリスト運用） | `/permissions` + ネイティブサンドボックス | Claude Code + `/sandbox` |
| 高自律（ローカル） | コンテナ隔離 + ネットワーク制限 | Docker Sandboxes / 自前Docker構成 |
| 高自律（クラウド） | マネージドサンドボックス + トレーシング | Claude Managed Agents |
| 完全自律（並列） | コンテナ隔離 + Gitロック + CI/CD | Carliniの16並列モデル / 自前構成 |

下に行くほど自律性が高く、同時にセキュリティ基盤の厚みが増す。第1回で書いた「スケールに応じた段階的な強化」は、自律性の軸でも同じ構造を持つ。

共通する設計原則は一貫している。

**1. パーミッションはモデルへの指示ではなく環境側で強制する。** エージェントは指示を無視できるが、サンドボックスは（適切に実装されていれば）回避できない。

**2. 自律性を上げる前に、爆発半径を制限する。** `--dangerously-skip-permissions`を使いたいなら、先にサンドボックスを設定する。Managed Agentsを使う場合でも、先にEnvironmentのネットワークルールとツール制限を定義する。

**3. トレーサビリティは安全策であり制約ではない。** 実行トレーシングはエージェントの自律性を制限しないが、問題が起きたときの原因特定と再発防止を可能にする。

## おわりに：ランタイム層が揃い始めた

第1回で、harness engineeringの本質は「モデルの外側にある環境設計」だと書いた。設計層（CLAUDE.md、プロンプト、ツール選択）と検証層（テスト、CI、AIレビュー）については、これまでの2回で整理してきた。

Claude Managed Agentsの登場で、ランタイム層——セキュアな実行環境、セッション管理、認証情報の分離、実行トレーシング——にマネージドサービスの選択肢が加わった。harnessの設計層・検証層に続き、ランタイム層にも具体的なツールとサービスが出揃い始めている。ただし、マルチエージェント連携やoutcomes機能はまだresearch previewであり、Managed Agentsの成熟はこれからだ。

とはいえ、「マネージドだから安全」ではない。設計層の判断——何を許可し、何を制限するか——は依然として人間の仕事だ。Kubernetes、コンテナ、ネットワーク隔離、最小権限設計。これらのインフラスキルは、ランタイム層がマネージドになっても、設計層で活きる。

エージェントの自律性は「安全な箱」の堅牢さに比例する。その箱を設計するのは、依然としてエンジニアの仕事だ。
