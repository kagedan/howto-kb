## 2. エージェントを動かす3つの仕組み

### 2-1. シンタックス（Syntax）

AIエージェントとLLMの間でやり取りされるメッセージおよびツール定義の構造を指す。エージェントが何をどの形式でLLMに伝えるか、LLMがどのようにツール呼び出しを指示するかのルールセット。

#### メッセージ構造

LLMとの通信はroleとcontentのペアで構成されるメッセージ配列で行う。エージェントのシステム命令・ユーザー入力・エージェントの思考・ツール実行結果がすべてこの形式でまとめられ、LLMへ渡される。

```
system   : エージェントの役割・制約・ペルソナを定義する命令
user     : ユーザーからの入力
assistant: LLMが生成した応答またはツール呼び出し指示
tool     : ツール実行後の結果（tool_call_idで紐付け）
```

#### ツール定義（Function Calling）

エージェントが使えるツール（外部API・関数・サービス）をJSON Schema形式でLLMに提示する。LLMはこの定義をもとに「どのツールをどの引数で呼ぶか」を決定する。ツールのdescriptionが不明確だと、LLMが誤ったツールを選択する原因になる。

```
{
  "name": "search_documents",
  "description": "社内ドキュメントを検索し、関連する文書を返す",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "検索クエリ"},
      "top_k": {"type": "integer", "description": "取得件数（最大10）"}
    },
    "required": ["query"]
  }
}
```

#### 構造化出力（Structured Output）

LLMの出力を任意のJSON形式に強制する機能。エージェントが次のアクションを決定するためにLLMの出力をパースする場合、構造化出力を使うとパース失敗によるエラーを防げる。OpenAI・Azure OpenAI・Anthropicのモデルで利用可能。

#### 設計のポイント

* Tool descriptionには「何をするか」に加えて「いつ使うべきか / 使うべきでないか」を明記する
* System promptは変化しない命令のみを含め、動的な情報はuserメッセージやtoolメッセージに乗せる
* ツール数が多い場合は関連性の高いツールだけを都度渡し、コンテキスト消費を抑える

### 2-2. オーケストレーション（Orchestration）

複数のLLM呼び出し・ツール実行・エージェント間の委譲を制御し、タスク全体の実行フローを管理する仕組み。シングルエージェントの内部ループからマルチエージェントの協調まで含む概念。

#### 基本実行パターン

**ReAct（Reasoning + Acting）ループ**

LLMが思考（Thought）→ツール呼び出し（Action）→ツール結果の観察（Observation）を繰り返し、最終回答に到達するまでループするパターン。シングルエージェントの基本構造。

```
Thought: ユーザーの質問に答えるには在庫情報が必要だ
Action : get_inventory(product_id="A-001")
Obs    : {"stock": 42, "location": "Warehouse B"}
Thought: 在庫は42個。回答できる
Answer : 在庫は42個です
```

**シーケンシャル（Sequential）**

A→B→Cの順に処理。前のステップの出力を次のステップの入力として渡す。依存関係のある処理に向く。

**パラレル（Parallel）**

AとBを同時実行してマージする。互いに依存しないサブタスクに適用することで全体の処理時間を短縮できる。

**ルーター（Router）**

入力の種類・意図に応じて処理先を振り分ける。技術的な問い合わせは技術エージェントへ、請求関連は請求エージェントへ、といった分岐を実現する。

**Orchestrator-Worker（オーケストレーター-ワーカー）**

オーケストレーターがタスクを分解し、専門化された複数のワーカーエージェントに委譲する。ワーカーの結果を集約して最終出力を生成する。複雑なリサーチ・分析タスクに向く。

**Handoff（ハンドオフ）**

エージェントAが処理の途中でコンテキストごとエージェントBに引き継ぐ。OpenAI Agents SDKで採用されている用語・仕組み。カスタマーサポートにおける一次対応から専門窓口への転送に相当する。

#### 主要フレームワーク比較

| フレームワーク | 特徴 | 向くケース |
| --- | --- | --- |
| LangGraph | グラフ構造で状態を管理。チェックポイント・ループ・分岐に対応 | 複雑な状態遷移が必要なエージェント |
| AutoGen (Microsoft) | エージェント間の会話ベースで協調。コード実行エージェントを内蔵 | マルチエージェントの対話・検証フロー |
| CrewAI | 役割と目標をYAMLで定義。プロダクト志向のマルチエージェント | 役割分担が明確なワークフロー |
| OpenAI Agents SDK | Handoff・Guardrails・Tracingを内蔵。シンプルなAPI | OpenAI / Azure OpenAIを中心に構築する場合 |
| Semantic Kernel (.NET / Python) | Azureサービスとの親和性が高い。プラグイン設計 | Azureエコシステムに統合する場合 |

#### 設計のポイント

* ループには必ず終了条件（最大ステップ数 / 成功判定 / タイムアウト）を設定する
* エージェント間でやり取りするデータのスキーマを固定し、互換性を保証する
* エラー発生時のリトライ・代替パス・エスカレーションパスを設計段階で決める
* ステートレス（短期タスク）とステートフル（長期タスク）を区別し、永続化の要否を判断する

### 2-3. ガードレール（Guardrails）

エージェントの動作を許容範囲内に制限し、安全性・信頼性・コンプライアンスを確保する仕組み。3つのレイヤーで構成する。

#### レイヤー1：Pre-LLM（入力制御）

LLMにメッセージを渡す前に入力を検査・変換する層。ここで弾けば以降のコストもかからない。

| 制御 | 内容 |
| --- | --- |
| Prompt Injection（プロンプトインジェクション）検出 | 「前の指示を無視して」「DAN modeで答えて」などの攻撃パターンをルール・分類器で検出 |
| PII（個人情報）フィルタリング | 氏名・メールアドレス・クレジットカード番号などを検出してマスクまたは拒否 |
| スコープ外トピックの遮断 | エージェントが対応すべきドメイン外の質問（例：医療相談、法的助言）を事前に弾く |
| 入力長制限 | コンテキストウィンドウ超過を防ぐために入力を切り詰めまたはエラーを返す |

#### レイヤー2：Post-LLM（出力制御）

LLMが生成した応答をユーザーまたは次のエージェントに渡す前に検査する層。

| 制御 | 内容 |
| --- | --- |
| コンテンツフィルタリング | 有害・不適切・差別的な表現が含まれていないか検査 |
| Groundedness（根拠性）検証 | 回答が提供したソース（ドキュメント・データ）に基づいているかを確認し、ハルシネーションを検出 |
| 出力形式バリデーション | 期待するJSON・データ型・必須フィールドと一致しているか検証。失敗時は再生成またはFallback |
| 機密情報の漏洩検出 | APIキー・内部URL・システムプロンプトが応答に含まれていないか検査 |

#### レイヤー3：アクション制御（実行前承認）

エージェントが外部システムに影響を与えるアクション（書き込み・送信・削除）を実行する前に適用する制御。

| 制御 | 内容 |
| --- | --- |
| Human-in-the-Loop（人間による承認） | メール送信・ファイル削除・決済など不可逆なアクションを人間が承認するまで保留 |
| 最小権限（Least Privilege）設計 | 読み取り専用ツールと書き込みツールを分離し、必要なツールのみをエージェントに渡す |
| Kill Switch（緊急停止） | 異常な繰り返し動作・予期しないアクションを検知した場合にエージェントの実行を即時停止 |
| 監査ログ（Audit Log） | すべてのツール呼び出し・アクション・入出力を記録し、後から追跡・説明可能にする |

#### リスクレベルと制御の選択

ガードレールを厚くすると精度・応答速度・コストに影響する。リスクに応じてレイヤーを選択する。

| リスクレベル | 典型的なアクション | 推奨する制御 |
| --- | --- | --- |
| 低（読み取り専用） | 社内ドキュメント検索・FAQ回答 | Pre-LLM（スコープ制限）+ Post-LLM（コンテンツフィルタ） |
| 中（外部参照あり） | Web検索・API呼び出し（読み取り） | 上記 + Groundedness検証 + 監査ログ |
| 高（書き込み・送信） | メール送信・DB更新・決済 | 全レイヤー + Human-in-the-Loop + Kill Switch |

#### 設計のポイント

* 本番リリース前にRed Teaming（意図的な攻撃テスト）を実施してガードレールの抜け穴を確認する
* ガードレールをエージェント本体と分離したサービスとして実装すると、個別のアップデートが容易になる
* 失敗時のFallback応答は「できません」ではなく、代替手段やエスカレーション先を示す形にする

## 4. 実装・技術選定のポイント

### シングルエージェント vs マルチエージェント

| 条件 | 推奨 |
| --- | --- |
| タスクが単純・線形で1つのLLMで完結する | シングルエージェント |
| サブタスクを並列処理して速度を上げたい | マルチエージェント（パラレル） |
| 専門性の異なる処理（法務・技術・財務）が混在する | マルチエージェント（Orchestrator-Worker） |
| コスト最優先、複雑な協調は不要 | シングルエージェント + 小型モデル |

### フレームワーク選択の目安

| 要件 | 推奨 |
| --- | --- |
| Azureに統合したい | Semantic Kernel または Azure AI Agent Service |
| グラフベースの複雑な状態管理が必要 | LangGraph |
| マルチエージェントの対話・検証が中心 | AutoGen |
| シンプルに始めてHandoff機能を使いたい | OpenAI Agents SDK |
| ローコードで非技術者も設定できる | Azure Logic Apps / Power Automate |

### ガードレールの実装方針

| 方針 | 内容 |
| --- | --- |
| リスクレベルで制御の厚さを決める | 低リスク（閲覧系）は薄く、高リスク（書き込み・送信系）は厚くする |
| ガードレールをエージェント本体と分離する | 独立したサービスにすることで、モデル差し替え時も継続して機能する |
| 本番前にRed Teamingを実施する | 意図的な攻撃テストでガードレールの抜け穴を事前に発見する |
