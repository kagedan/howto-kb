# ハーネスエンジニアリングとは

詳細は[元記事](https://zenn.dev/ubie_dev/articles/sec-agent-harness-eng)に譲りますが、簡単に言うと「**LLMエージェントを実用に耐える形にするための、モデル外の工夫全般**」を指します。

具体的には以下のような観点が含まれます:

* **計測観点**: エージェントの挙動を把握・評価できるようにする
* **実行設計観点**: コンテキスト効率化、実行フロー制御、安全性確保
* **実行最適化観点**: エージェント性能、レイテンシ、コスト効率の改善

プロンプトだけでLLMの挙動を制御するのは限界があり、「アーキテクチャで構造的に弱点を埋める」のがハーネスエンジニアリングの思想です。

# Warren とは

[Warren](https://github.com/secmon-lab/warren) は、Ubie社が開発しているOSS(Apache 2.0)のセキュリティアラート分析AIエージェントです。

ざっくりした動作フロー:

1. セキュリティ製品(GuardDuty、SIEM、EDR等)からアラートをWebhookで受信
2. LLM(Gemini)がアラートを読んで、要約・分類・脅威インテル照会を自動実行
3. 結果をSlackまたはWeb UIでチケット化し、アナリストが追加調査をチャットで依頼可能

重要なのは、これが「LLMにツールを繋いだだけのシンプルなエージェント」ではなく、**ハーネスエンジニアリングの考え方で緻密に設計されている** 点です。

# セットアップ手順

## 前提

* macOS(Apple Silicon / Intel どちらも可)
* Docker Desktop
* gcloud CLI
* GCP アカウント(無料トライアルでOK)

### 検証環境

* 検証日: 2026年4月18日
* macOS: Apple Silicon
* Docker Desktop: 29.1.3
* Warrenイメージ: `ghcr.io/secmon-lab/warren:latest`(検証時点)

## Step 1: 必要なツールのインストール

Homebrewがあれば一発です。

```
# gcloud CLI をインストール
brew install --cask google-cloud-sdk

# zshに設定追加
echo "source '/opt/homebrew/share/google-cloud-sdk/path.zsh.inc'" >> ~/.zshrc
echo "source '/opt/homebrew/share/google-cloud-sdk/completion.zsh.inc'" >> ~/.zshrc
source ~/.zshrc

# 確認
gcloud --version
```

## Step 2: GCPプロジェクトの準備

```
export PROJECT_ID="your-project-id"  # 自分のプロジェクトIDに置き換え

# 認証
gcloud auth login
gcloud config set project $PROJECT_ID
gcloud auth application-default login

# Vertex AI API有効化
gcloud services enable aiplatform.googleapis.com --project=$PROJECT_ID
```

**注意**: 請求先アカウントをプロジェクトにリンクする必要があります。GCPコンソールから手動でリンクしてください。

## Step 3: Warrenの起動

```
mkdir -p ~/warren-test && cd ~/warren-test

# 環境変数ファイル作成
cat > warren.env << EOF
WARREN_GEMINI_PROJECT_ID=$PROJECT_ID
WARREN_GEMINI_LOCATION=us-central1
WARREN_NO_AUTHENTICATION=true
WARREN_NO_AUTHORIZATION=true
WARREN_ADDR=0.0.0.0:8080
EOF

# Docker起動
docker run -d \
  --name warren \
  -p 8080:8080 \
  -v ~/.config/gcloud:/home/nonroot/.config/gcloud:ro \
  --env-file warren.env \
  ghcr.io/secmon-lab/warren:latest serve
```

起動確認:

ブラウザで `http://localhost:8080` を開くとダッシュボードが表示されます。

# 実際に動かしてみた

## 検証1: シンプルなアラート分析

まずはシンプルな脅威インテル調査を投げてみました。以下のIPは[Warrenの公式チュートリアル](https://github.com/secmon-lab/warren/blob/main/doc/getting-started.md)で例示されている検証用IPです。

```
curl -X POST http://localhost:8080/hooks/alert/raw/test \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Malicious IP Communication Detected",
    "description": "Outbound connection to known C2 server detected",
    "severity": "critical",
    "source_ip": "45.227.255.182",
    "destination_ip": "10.0.1.100"
  }'
```

Web UIでアラートを選択 → Create Ticket → Start Chatから以下を送信:

```
Analyze IP 45.227.255.182 using available tools
```

### 観察された挙動

エージェントは以下のフェーズを踏んで動作しました:

1. **Intent Resolution**: 「WHOISを使ってIPを調べる」と方針決定
2. **Execute**: `whois_ip` ツールを実行
3. **Reflection (Completed)**: 結果を整理してナレッジ化
4. **Final**: 脅威評価と推奨対応策を生成

特に興味深かったのは、Final回答の以下のくだりです:

> "While the WHOIS data does not directly confirm or deny C2 activity, it provides ownership details for the IP."

**事実(WHOIS情報)と仮説(C2サーバーである)を明確に区別** しています。これは元記事で触れられていたプロンプト制御:

> Never state a hypothesis as a confirmed fact. Use language that reflects uncertainty

が効いている証拠です。

## 検証2: 複雑なアラートでのタスク分割

次に、複数の観点が絡む複雑なアラートを投げてみました:

```
curl -X POST http://localhost:8080/hooks/alert/raw/test \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Suspicious process on internal host",
    "severity": "critical",
    "source_ip": "10.0.1.100",
    "destination_ips": ["IP_1", "IP_2", "IP_3"],
    "suspicious_process": "svchost.exe (unsigned)",
    "user_account": "admin",
    "unusual_activities": ["after-hours login", "privilege escalation attempt"]
  }'
```

チャットで包括的な調査を依頼:

```
Please investigate this alert comprehensively.
```

### 観察された挙動

今度は **3つのタスクが並列で実行** されました。

```
Starting...
Starting...
Starting...
🤖 Execute: `knowledge_search`
🤖 Execute: `knowledge_search`
🤖 Execute: `knowledge_search`
```

元記事の以下の主張:

> 同一フェーズ内のタスクは完全に並列実行し、待ち時間を最小化しています

これが実装レベルで確認できました。エージェントが「このアラートは複数の観点から調査する必要がある」と判断して、自動でタスクを分解して並列化しているわけです。

# ログから読み取れる5つの設計パターン

実際の動作を観察して、以下5点が特に印象的でした。

## 1. フェーズ分割

全ての実行が `Starting... → Execute → Completed → Reflection` というフェーズを経ます。単なる「LLM呼び出し」ではなく、明示的な状態遷移として構造化されています。

## 2. ツールセット分離

各タスクに必要なツールだけが渡されます。WHOISが必要なタスクにはWHOISツールだけ、ナレッジ検索が必要なタスクにはナレッジ検索ツールだけ。**不要なツールを与えないことで、LLMが目的から逸れにくくなる** 設計です。

## 3. 「解釈せず生データだけ返す」子エージェント

ログを見ると、子エージェントは生のWHOISデータをそのまま返しており、**解釈は親エージェント(最終統合フェーズ)に委ねられています**。これによりコンテキストのノイズを抑え、親エージェントの判断品質を保っています。

## 4. Knowledge Reflection(記憶の自動編纂)

タスク完了後、エージェントが自動でナレッジを生成しました:

```
💾 Saved knowledge 'WHOIS lookup reveals delegated regional registries...'
```

しかもこれ、単なる事実記録ではなく「**WHOISは地域レジストリに委譲されることがあるので、最初からそこに聞くと効率的**」という **調査テクニック** を抽出しています。元記事の「Technical Knowledge(技術知識)」の実装例です。

## 5. 確証バイアス対策の効いたプロンプト

最終レポートの文体が特徴的で、「raise suspicions」「often indicates」といった **不確実性を含む表現** が多用されます。これは元記事の以下のプロンプトが効いている証拠:

> Never state a hypothesis as a confirmed fact.

一方で、「偽りの安心」も与えません。Critical判定すべきものはきちんとCritical判定します。

# 実運用LLMエージェント開発への応用

今回の検証を通して、自分のLLMエージェント実装業務で応用できると感じた知見を整理します。

## ① BigQueryエージェントの設計

クライアント業務でよくある「自然言語でBigQueryを叩けるエージェント」を作るとき、Warrenの以下のパターンが参考になります:

* 全クエリのドライラン必須化(スキャンサイズの上限)
* 結果行数の自動制限(LIMIT必須、1000行まで)
* `SELECT *`, `OFFSET` の禁止をプロンプトで明示
* サービスアカウントは `dataViewer` 権限のみに絞る(破壊的クエリの防止)

## ② エージェントの暴走防止

**バジェット戦略** が秀逸です。単純なループ回数制限ではなく、ツール種別ごとのコストと時間コストをポイント制で管理し、ソフトリミット・ハードリミットの二段構えにする設計は応用範囲が広いです。

## ③ 継続的改善ループ

Knowledge Reflectionの考え方は、セキュリティに限らずどんなエージェントにも応用できます。「失敗の知識は成功の知識より価値がある」という方針は、検出系や推論系システムでも重要です。

# まとめ

ハーネスエンジニアリングという概念を、ただ読むだけでなく **実際に動かして確認する** ことで、以下が腹落ちしました:

1. LLM単体ではなく、周辺の構造が本番運用品質を決める
2. フェーズ分割・ツール分離・記憶の編纂という基本パターンは再利用可能
3. プロンプトは「望ましい振る舞いの指示」ではなく「アーキテクチャの隙間を埋める最後の手段」

Warrenは実装の勉強教材としても非常に優秀なOSSだと感じました。興味のある方はぜひ手を動かしてみてください。
