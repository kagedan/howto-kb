## 1. 相談のきっかけ：無意識に諦めていた「不便」の正体

きっかけは、モノレポ構成での開発中に感じていた小さなストレスでした。一つのリポジトリに複数のプラットフォームが共存していると、標準のGUIツールでは特定のフォルダだけの履歴を追うのが意外と面倒なのです。

私はこれを「モノレポの仕様上、仕方ないこと」と半分諦めていました。しかし、Codexにこの「愚痴」をぶつけてみたところ、意外な答えが返ってきました。

**AI（Codex）**: 「それは作る価値があるタイプのものだ」

AIとの対話を通じて、自分が「仕方ない」と思い込んでいた不便が、実は「解決すべき明確な課題（アイデア）」へと変わった瞬間でした。

## 2. 技術選定と設計を「丸投げ」してみる

今回は、実装を指示する前に「技術選定と設計」からAIに提案させました。自分で考えるのではなく、AIにリードを任せてみたのです。

**GUI**: Tauri + React / TypeScript  
**Git操作**: ライブラリを自前で持つより、挙動の信頼性が高い Git CLI を直接呼び出す構成  
**画面設計**: モノレポの「フォルダ単位」での閲覧に特化した3ペイン構成

Tauriを選ぶ理由は、OS標準のWebViewを利用するため配布用exeが非常に軽量になるから、という極めて合理的なものでした。

## 3. なぜバックエンドに Rust なのか？

「TypeScriptだけで完結するのでは？」という疑問に対しても、Codexは明確な役割分担を提示しました。

**TypeScript**: 画面の描画や状態管理を担当。

**Rust (Tauri)**: ブラウザ環境（TS）では触りにくい「OS側の仕事（gitコマンドの実行やファイルシステムへのアクセス）」を安全に担当。

Rustは「ビルド時にexeへコンパイルされる」ため、配布先のPCにRustをインストールする必要もありません。この「UIはWeb技術、ローカル操作はRust」という棲み分けの提案により、開発の解像度が一気に上がりました。

## 4. 「一発で動く」ことへの戸惑い

驚くべきことに、指示を出した直後、AIは900行を超えるコードを書き上げ、一発で起動するMVP（実用最小限の製品）を届けてくれました。

ここで、ある種の戸惑いが生まれます。「自分で一から試行錯誤してコードを書いていた頃の手応え」が、どこにもないのです。しかし、アプリを触り始めると、すぐに「人間にしかできない仕事」が山ほど現れました。

* 「コミットメッセージを全文読めるように折り返してほしい」
* 「パネルのリサイズができないと、結局ストレスが溜まる」
* 「特定のファイルだけの差分を切り替えたい」
* 「アイコンがあったほうが、視覚的な認知コストが下がる」
* 「スクロールバーがレイアウトの副作用で消えてしまったのを直したい」

これらは、AIが生成した「正解のようなコード」を、「自分の道具」として血を通わせる作業です。一発で動くコードが出た後の、こうした細部への「こだわり」の修正こそが、今の時代の開発における職人芸なのかもしれません。

## 5. まとめ：AI時代の個人開発における「活用術」

今回の体験を経て確信したのは、AIは単なる「コーディングツール」ではないということです。

* **不満の言語化**: 自分が感じている「なんとなくの不便」をAIにぶつけ、課題へと昇華させる。
* **設計の壁打ち**: 技術選定の妥当性をAIと議論し、最適な構成を導き出す。
* **ディレクション**: 動くものが出た後、自分の「審美眼」で徹底的に磨き上げる。

「自分で作っている感」の形は変わりました。しかし、その分私たちは「どんな道具が自分にとって最高か」を考えることに、より多くの時間を使えるようになったのです。

**おすすめ構成**

* GUI: Tauri
* UI: React + TypeScript
* Git操作: まずは git コマンド呼び出し
* 差分表示: monaco-editor か react-diff-viewer 系
* 状態管理: 最初は React state / Zustand 程度
* 配布対象: Windows優先なら Tauri はかなり相性が良いです

理由は、Git履歴ビューアは「ローカルファイル」「ローカルGit」「ネイティブ寄りの軽さ」が大事なので、Electronより Tauri の方が小さく始めやすいからです。Git操作も最初から libgit2 などを抱えるより、Git CLI に寄せた方が挙動がユーザーの期待と一致しやすいです。

**MVPの画面設計**

大きくは3ペインが良さそうです。

```
┌──────────────────────────────────────────────┐
│ Repository: C:\path\to\repo                  │
├───────────────┬──────────────────────────────┤
│ Folders       │ Commits                      │
│               │                              │
│ packages/app  │ abc123  Fix auth flow        │
│ packages/api  │ def456  Update schema        │
│ libs/core     │ ...                          │
├───────────────┴──────────────┬───────────────┤
│ Changed files                 │ Diff          │
│ src/foo.ts                    │ ...           │
│ package.json                  │ ...           │
└───────────────────────────────┴───────────────┘
```

MVPではルート直下フォルダだけにして、あとから `packages/*` や任意パス選択に広げるのが良いと思います。

**Gitコマンド設計**

最初はこのあたりで十分です。

```
# リポジトリ判定
git -C <repo> rev-parse --show-toplevel

# ルート直下フォルダ一覧
git -C <repo> ls-tree -d --name-only HEAD

# フォルダ履歴
git -C <repo> log --date=iso --pretty=format:<custom> -- <folder>

# コミット内の変更ファイル
git -C <repo> show --name-status --format= -- <commit> -- <folder>

# diff
git -C <repo> show --format= -- <commit> -- <folder-or-file>
```

ただし `pretty=format` はパースが壊れにくいように、区切り文字を工夫した方がいいです。例えば `\x1f` や `\x1e` を使うなど。

**アーキテクチャ**

Tauriなら、Git実行はフロントではなく Rust 側の command に寄せたいです。

```
Frontend React
  ↓ invoke()
Tauri Rust commands
  ↓
Git CLI
  ↓
Parsed DTO
  ↓
Frontend state
```

型はだいたいこんな感じです。

```
type RepoInfo = {
  path: string;
  root: string;
};

type RootFolder = {
  name: string;
  path: string;
};

type CommitSummary = {
  hash: string;
  shortHash: string;
  subject: string;
  authorName: string;
  authorEmail: string;
  date: string;
};

type ChangedFile = {
  path: string;
  status: "A" | "M" | "D" | "R" | "C" | string;
  oldPath?: string;
};

type CommitDetail = {
  commit: CommitSummary;
  files: ChangedFile[];
  diff: string;
};
```

**最初に決めたい設計判断**

一番大事なのは「Git履歴の意味」をどこまで正確に扱うかです。

MVPでは `git log -- <folder>` 相当でよいですが、将来的には以下が論点になります。

* リネーム追跡をするか: `--follow` は単一ファイル向きでフォルダには癖があります
* マージコミットをどう見せるか
* 巨大diffをどう扱うか
* 削除済みフォルダの履歴を見たいか
* サブモジュールを扱うか
* ワークツリーの未コミット変更も見るか

MVPではこう割り切るのがよさそうです。

* 対象は通常のローカルGitリポジトリ
* ルート直下の現在存在するディレクトリのみ表示
* 履歴は `git log -- <folder>`
* diffは1コミット分だけ表示
* 巨大diffは一定サイズで切る
* 未コミット変更は対象外

**私の提案する初期ロードマップ**

1. リポジトリを選ぶ
2. ルート直下フォルダを一覧表示
3. フォルダ選択でコミット一覧表示
4. コミット選択で変更ファイル一覧表示
5. ファイル選択またはコミット全体diff表示
6. 検索、作者、期間フィルタは後回し

技術選定としては、私は Tauri + React + Git CLI を推します。  
「早くMVPを触れる形にする」「Gitの挙動が本物と一致する」「将来Windows向けに軽く配れる」のバランスが良いです。

このプロジェクトで Rust を使っている理由は、ざっくり言うと TypeScriptだけでは安全に触りにくい"PC本体側の仕事"を担当するためです。

このアプリでは役割がこう分かれています。

```
React + TypeScript
  画面、ボタン、リスト、検索、diff表示、状態管理

Rust / Tauri

```

実際に `src-tauri/src/main.rs` では Rust が git コマンドを呼んでいます。たとえば `open_repository`、`get_folder_history`、`get_commit_detail`、`get_file_diff` みたいな関数が Rust 側にあり、TypeScript 側の `src/App.tsx` から `invoke("get_folder_history", ...)` のように呼び出しています。

なのでこのプロジェクトでの Rust は、複雑なビジネスロジック用というより、

> UIから頼まれたことを、ローカル環境で安全に実行する小さなバックエンド

という立ち位置です。

「TypeScriptで完結するのでは？」という感覚もかなり自然です。画面だけなら完結します。ただ、このアプリはローカルのGit履歴を見るので、`git log` や `git show` を実行する必要があります。そこから先はブラウザ/TypeScriptの得意範囲を少し出るので、TauriのRust側に任せています。

Electron なら Node.js を同じ役目に使うことが多いです。Tauri はそのバックエンド役に Rust を使う設計、という理解で大丈夫です。

### 🤖 Codex

たぶん思い浮かべているのは Electron だと思います。VS Code や Slack 系でよく使われるやつですね。

このプロジェクトでは README にもある通り、ビルドするとまず単体の実行ファイルが出ます。

```
src-tauri/target/release/monorepo-viewer.exe
```

なので、開発者目線では「このexeを起動すればアプリが動く」という形に近いです。

Electronとの大きな違いはここです。

```
Electron
  Chromium + Node.js をアプリに同梱する
  サイズが大きくなりやすい
  JS/TSだけでローカル処理を書きやすい

Tauri

ただし注意点もあります。

* Windows側に Microsoft Edge WebView2 Runtime が必要（多くの環境では既に入っている）
* 画像・DB・外部ツールなどを別ファイルとして持つ設計にすると、exe単体では済まない場合もある
* このアプリはローカルの git CLI を呼ぶので、利用PCに Git が入っている前提になる

今回の Monorepo Viewer で言うと、Tauriを選ぶ利点はかなりあります。

* Git履歴ビューアなので、ローカルファイルや git コマンドに触る必要がある
* Webアプリより自然にローカルPCと連携できる
* Electronより軽量な配布物になりやすい
* React + TypeScript のUI資産はそのまま使える
* Rust側は薄い「ローカル命令係」だけで済む

なので、イメージとしては、

```
Electron:
  TypeScript/Node.jsで全部やりやすいが、配布物は大きめ

Tauri:
  UIはTypeScript、PC操作はRust
  少し二言語になるが、軽くてデスクトップアプリらしく配れる
```

という感じです。今回の用途なら、Tauriはかなり相性いいです。

### 🤖 Codex

Rustは「開発・ビルドする人のPC」に必要です。ビルドすると Rust のコードは `.exe` の中にコンパイルされます。配布先では Rust コンパイラや Cargo は要りません。

  cargo build / tauri build で exe を作る

```

Tauriの場合、配布先で必要になりうるのは Rust ではなく、Windowsなら WebView2 Runtime です。これは最近のWindows環境だとだいたい入っています。

Rustの使いどころは、ざっくり言うと **「OSに近い処理」「速さが必要な処理」「壊れると困る基盤部分」** です。

たとえばこういう場所でよく使われます。

```
デスクトップアプリ:
  Tauriのバックエンド
  ファイル操作、プロセス実行、OS連携

CLIツール:
  ripgrep, fd, bat, starship など
  速くて単体バイナリで配りやすい

サーバー:
  高速API、認証、プロキシ、リアルタイム処理

インフラ/開発ツール:
  パッケージマネージャ、ビルドツール、フォーマッタ、LSP

組み込み/低レイヤー:
  OS、ドライバ寄り、WebAssembly、ゲームエンジンの一部
```

逆に、Reactの画面を作るだけ、管理画面を作るだけ、ちょっとしたWeb APIを素早く作るだけなら、Rustである必要は薄いです。TypeScript、Python、Go、C#などのほうが楽なことも多いです。

Rustの強みはこのへんです。

* **速い** — C/C++に近い性能が出やすい
* **配りやすい** — 単体exeにしやすい
* **メモリ安全** — C/C++っぽい領域を扱えるのに、危ないバグをコンパイル時に防ぎやすい
* **並行処理に強い** — 複数処理を安全に動かしやすい
* **ランタイムが軽い** — Node.jsや.NETのような大きな実行環境に依存しにくい

今回のようなTauriアプリでは、Rustは「アプリ全体をRustで書く」というより、TypeScriptから呼ばれるローカル処理係です。

```
TypeScript:
  ユーザーが見る画面を作る

Rust:
  gitを実行する
  ファイルを読む
  OSの機能を呼ぶ
  危ない入力をチェックする
```

これはかなり自然な使い方です。

Rustを学んだけど「何に使うんだろう？」となる感覚も自然です。Rustは万能な日常言語というより、道具を作るための道具っぽい性格があります。アプリの表側より、裏側のエンジン、CLI、変換処理、ローカル処理、重い処理に置くと気持ちよく効いてきます。

なので最初の実用先としては、今回みたいなものがちょうどいいです。

* React/TypeScriptでUI
* Rustで数個のTauri commandを書く
* ファイル操作やCLI呼び出しだけ任せる

Rustを全部理解してから使う、ではなく、小さいバックエンド関数を1つずつ増やすくらいがかなり学びやすいです。
