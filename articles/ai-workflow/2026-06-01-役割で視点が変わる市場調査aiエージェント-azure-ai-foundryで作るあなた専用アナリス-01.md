## 1. 背景と目的

個人が技術や市場のトレンドを把握しようとすると、いつも同じ壁にぶつかります。

**「調べる時間」と「読み解く時間」が、とにかくかかる。**

たとえば「自動運転市場は今どうなっているのか」を知りたいだけでも、

* 業界ニュース（英語記事を何十本も）
* 上場企業の財務（株価・PER・PBR・売上成長・粗利率）
* 株価のテクニカル指標（RSI・移動平均）
* 各国のマクロ経済（GDP成長率）
* 国際貿易統計（市場規模のシグナル）
* 日本企業の開示書類（有価証券報告書など）

といった情報が**バラバラのソースに散らばっており**、これらを横断的に集めて整理するだけで膨大な労力がかかります。さらに苦しいのは、集めた情報が断片的で、**「自分がどんな立場でこの情報を必要としているか」という文脈に沿って解釈するのが難しい**という点です。

> 投資家が見たい「自動運転市場」と、経営者が見たい「自動運転市場」と、学生が見たい「自動運転市場」は、まったく別物のはずです。  
> しかし、検索エンジンは誰に対しても同じ結果しか返してくれません。

本システムは、**一般ユーザーであっても、関心のある領域に対して、信頼性の高い情報を統合的かつ構造的に取得できる環境**を提供することを目的に構築しました。具体的には次の体験を狙っています。

1. **関心のあるテーマを一言入力するだけ**で、信頼性の高い一次情報源（公的統計・金融API・ニュース）を**統合的かつ構造的に**取得する。
2. **ユーザーの「役割（ロール）」を選ぶ**だけで、同じテーマでも**まとめ方・着眼点・出力フォーマットが役割ごとに変わる**。投資家には投資判断メモを、経営者には戦略ブリーフを、学生にはケーススタディを。
3. レポート中の難しい専門用語や固い文章を、**比喩を使ってその場で分かりやすく解説する**機能で、知識レベルの差を埋める。

つまり、「**自分専用のアナリストに調べさせ、自分の立場の言葉で報告を受ける**」体験を、数十秒で得られる環境を目指しました。

## 2. 設計上の工夫

### 2-1. 「役割（ロール）」を起点に、情報の集め方とまとめ方を変える

このエージェント最大の設計思想は、**「同じテーマでも、読む人の役割によって必要な情報も、まとめ方も違う」** という点をシステムの中心に据えたことです。

用意した役割は6つです。

| No | 役割 | アウトプット | 想定読了時間 |
| --- | --- | --- | --- |
| 1 | VC/PE投資家 | 投資判断メモ（Investment Memorandum） | 15分 |
| 2 | 企業経営者（CEO/CFO） | 経営戦略ブリーフ | 30分 |
| 3 | 証券アナリスト | エクイティリサーチノート（Buy/Hold/Sell） | 20分 |
| 4 | ビジネスデベロップメント（BD/M&A） | M&A評価レポート | 60日判断用 |
| 5 | 学生・アカデミック | 学術ケーススタディ（10〜15ページ） | 学習用 |
| 6 | 事業開発担当者 | トレンド・ビジネス機会レポート | 20分 |

ここで工夫したのは、**「役割ごとに、そもそも叩くデータソースを変える」** という点です。役割によって“知りたいこと”が違う以上、集める情報も変えるべきだと考えました。

```
ROLES = {
    "1": {  # VC/PE投資家：トレンド+競合財務+テクニカル+マクロTAM
        "name": "VC/PE投資家",
        "news_count": 50, "news_months": 1,
        "use_news": True, "use_finnhub": True, "use_alpha_vantage": True,
        "use_edinet": False, "use_worldbank": True, "use_comtrade": False,
        "pages": "5ページ以内（30分で読める）",
    },
    "3": {  # 証券アナリスト：個別株のカタリスト+財務+テクニカル+日本株開示
        "name": "証券アナリスト（Equity Research）",
        "news_count": 100, "news_months": 1,
        "use_news": True, "use_finnhub": True, "use_alpha_vantage": True,
        "use_edinet": True, "use_worldbank": False, "use_comtrade": False,
        "pages": "3〜5ページ（クイックノート形式）",
    },
    "5": {  # 学生：業界構造の理解が目的。長期・大量・マクロ重視
        "name": "学生・アカデミック",
        "news_count": 200, "news_months": 12,
        "use_news": True, "use_finnhub": True, "use_alpha_vantage": False,
        "use_edinet": False, "use_worldbank": True, "use_comtrade": True,
        "pages": "10〜15ページ（論文・ケーススタディ用）",
    },
    # ...
}
```

たとえば、

* **証券アナリスト**は「個別株を今買うか」が関心事なので、テクニカル指標（Alpha Vantage）と日本株の開示（EDINET）をON、マクロ（World Bank）はOFF。ニュースは直近1ヶ月に絞る。
* **学生**は「業界構造を深く理解したい」ので、ニュースは過去12ヶ月・200件、貿易統計やマクロ指標もONにして、短期テクニカル指標は不要なのでOFF。
* **VC投資家**は「30分で読める5ページ」、**学生**は「10〜15ページ」と、**出力ボリュームまで役割で規定**しています。

このように、**「役割の選択」＝「データ収集戦略の選択」＝「アウトプット形式の選択」** が一気通貫で連動するのが、本システムの設計上の核心です。

### 2-2. 知識レベルの差を埋める「分かりやすく解説」機能

役割で視点を変えても、レポートには `IRR` `CAGR` `EV/EBITDA` `RSI` といった専門用語が登場します。これは専門家には必要でも、一般ユーザーには壁になります。

そこで、**レポート内の任意の文章を選択すると、比喩を使って噛み砕いて解説する吹き出し機能**を搭載しました。専門用語を「お店の常連」「広告費」といった身近なたとえに置き換えることで、知識レベルに関わらず内容を理解できるようにしています。

### 2-3. 「データソースの幻覚」を構造的に防ぐ

AIに自由に書かせると、存在しないデータや出典を“それらしく”創作してしまいます。これを防ぐため、

* 役割ごとに\*\*「使ってよいデータソースのリスト」を明示的にプロンプトへ注入\*\*し、
* 実際にAPIから取得した**生データのみをコンテキストとして渡し**、
* 「これら以外のデータ源は創作・引用しないこと」と**制約**する、

という設計にしました（詳細は §プロンプトの工夫）。

## 3. 実装上の工夫

### 3-1. 6つの公的・金融APIを役割に応じてオーケストレーション

データ収集層は、6つの外部APIを役割設定（`ROLES`）に従って選択的に叩く構造です。

| API | 取得データ | 主な利用役割 |
| --- | --- | --- |
| **NewsAPI** | 業界ニュース・企業動向・規制・センチメント | 全役割 |
| **Finnhub** | 上場企業の財務（株価/PER/PBR/売上成長/粗利率） | 全役割 |
| **Alpha Vantage** | テクニカル指標（RSI・週次株価） | 投資家・アナリスト |
| **World Bank** | 各国マクロ（GDP成長率） | 投資家・経営者・学生など |
| **UN Comtrade** | 国際貿易統計（輸出入額＝市場規模シグナル） | 経営者・BD・学生 |
| **EDINET** | 日本上場企業の開示書類（有価証券報告書など） | アナリスト・BD |

各 `fetch_*` 関数は**例外を捕捉してログに記録した上で空を返す**設計にしており、一部のAPIが落ちても全体は止まらず、取得できたデータだけでレポートを生成します（堅牢性の確保）。

```
try:
    r = requests.get(..., timeout=10)
    # ... 整形して返す ...
except Exception as e:
    print(f"  [NewsAPI] エラー: {e}")   # ← 標準出力に記録
    return []                           # ← 落とさず空で継続（部分劣化）
```

エラーは標準出力へ出すことで、**App Service のログストリーム（`az webapp log tail`）にそのまま集約**されます。外部APIは無料枠のレート制限・一時障害が避けられないため、「**1つのAPIの失敗で全体を落とさず、取得できた分でレポートを出す（graceful degradation）**」ことを優先しました。どのソースが欠落したかはログで追跡できます。

### 3-2. 日本語テーマ → 英語クエリ・株式シンボルへの自動マッピング

ユーザーは「自動運転」「生成AI」のように**日本語で**テーマを入力します。一方、NewsAPIは英語検索が有効で、Finnhubは株式シンボルが必要です。そこで、テーマ名から**英語キーワード**と**関連銘柄**を自動で引き当てる辞書を用意しました。

```
MARKET_SYMBOLS = {
    "自動運転":   ["TSLA", "GOOGL", "GM", "INTC", "MBLY"],
    "生成AI":     ["NVDA", "MSFT", "GOOGL", "META", "AMD"],
    "半導体":     ["NVDA", "AMD", "INTC", "TSM", "QCOM"],
    # ...
    "デフォルト": ["AAPL", "MSFT", "GOOGL", "AMZN", "NVDA"],
}

# NewsAPI 用：「自動運転」→「autonomous vehicles」へ変換して英語記事を検索
search_keyword = mapping.get(jp_key, jp_key)  # 例: autonomous vehicles
```

辞書に無いテーマは「デフォルト」銘柄にフォールバックするため、**どんな入力でも必ず動く**ようにしています。

さらに、**サービス系テーマ（AI・SaaSなど）には貿易統計が存在しない**ため、UN Comtradeは物品テーマ（HSコードにマッピングできるもの）のみ対象とし、それ以外は静かにスキップします。EDINETも日本企業にマッピングできるテーマだけを対象にしています。**「無理にデータをでっち上げない」** ことを実装レベルで担保しました。

```
def fetch_comtrade(keyword: str, year: str = "2023") -> dict:
    hs = next((code for k, code in COMTRADE_HS.items() if k in keyword), None)
    if not hs:
        print(f"  [Comtrade] '{keyword}' は物品貿易の対象外（スキップ）")
        return {}   # ← サービス系は空を返す＝幻覚させない
```

### 3-3. 収集データを「AIが必ず参照する構造化コンテキスト」に整形

集めたバラバラのAPIレスポンスを、`format_context()` で**1つのMarkdown文字列**に統合します。ここで、ニュースは独自ロジックで**ポジ／ネガのセンチメント比率を算出**し、財務は**表形式**に、マクロは**直近3年推移**に整形しておくことで、AIが解釈しやすい状態にしてから渡します。

```
# ニュース見出し＋説明文からキーワードでセンチメントを集計
pos = sum(1 for a in news if any(w in (a["title"]+...).lower()
          for w in ["growth","launch","partnership","approval","record","surge","win"]))
neg = sum(1 for a in news if any(w in (a["title"]+...).lower()
          for w in ["recall","accident","fine","lawsuit","delay","ban","crash","loss"]))
lines.append(f"- Sentiment: ポジティブ {pos}件({pos*100//total}%) / ...")
```

### 3-4. リアルタイム進捗を見せる Web UI（Flask + SSE）

レポート生成は数十秒かかります。その間ユーザーを待たせきりにしないため、**Server-Sent Events（SSE）** で「今どのAPIを叩いているか」を逐次ストリーミング表示します。

* `/start` でジョブを起動し、別スレッド（`_worker`）でデータ収集〜生成を実行
* `/stream/<jid>` が `text/event-stream` で進捗イベントを配信
* 各ステップは `running / done / skip` の状態を持ち、「このロールでは使用しません」も明示

```
@app.route("/stream/<jid>")
def stream_events(jid):
    def generate():
        while True:
            msg = q.get(timeout=180)
            yield f"data: {json.dumps(msg, ensure_ascii=False)}\n\n"
            if msg.get("type") == "done":
                break
    return Response(stream_with_context(generate()), mimetype="text/event-stream", ...)
```

長時間のデータ収集処理がWebリクエストをブロックしないよう、`queue.Queue` と `threading` でジョブを別スレッド化しました。ユーザーは「現在どのAPIから何を収集しているか」をリアルタイムに把握できます。

> なお、ジョブ状態をプロセス内メモリの辞書（`jobs`）で管理しているため、**App Service上ではワーカー数を1に固定**（`gunicorn --workers 1 --threads 8`）しています。複数ワーカーにすると、SSEを購読するワーカーと処理を実行するワーカーが別プロセスになり、進捗が共有されず壊れるためです。
>
> 本来ならジョブ状態を Redis 等の外部ストアに置けばワーカーを水平スケールできますが、本アプリは\*\*「テーマを入力 → 数十秒で1本のレポートを生成」という単発・短時間のジョブが中心\*\*で、同時実行数も限定的です。そこで、**追加のマネージドサービス（＝コストと運用の複雑性）を増やすより、1ワーカー＋マルチスレッドでインメモリ完結させる**方をあえて選びました。スケールが必要になった段階で Redis へ差し替えられるよう、ジョブ管理は `jobs` 辞書1か所に局所化してあります。実装と運用構成を整合させた点も工夫の一つです。

### 3-5. 生成レポートを Markdown / PDF / Word でダウンロード

生成結果は画面表示だけでなく、3形式で出力できます。

* **Markdown**：そのままファイル化
* **PDF**：`reportlab` で生成。Markdownの見出し・箇条書き・表・太字をPDFスタイルへ変換
* **Word**：`python-docx` で見出しレベル・テーブル・インライン装飾（`**bold**` `*italic*` `` `code` ``）を再現

提出物や社内資料に流用できる「**実務で使える成果物**」として持ち帰れる点を重視しました。

#### Linuxコンテナでの日本語PDF：フォントファイルを同梱しない工夫

ReportLabでの日本語PDF生成は、**Linuxコンテナ上だとフォントファイルのパス指定でつまずきやすい**ポイントです（OSにインストールされた `.ttf`/`.otf` を絶対パスで `TTFont` 登録する方式だと、コンテナにフォントが入っていなければ文字化け・例外になる）。

本アプリでは、外部フォントファイルに依存せず、**ReportLabに標準同梱されている日本語CIDフォント `HeiseiKakuGo-W5`** を `UnicodeCIDFont` で登録する方式を採りました。これにより**フォントファイルの同梱もパス設定も不要**で、`requirements.txt` に `reportlab` を入れるだけで、App Serviceのクリーンなコンテナ上でも日本語PDFが安定して出力できます。さらに、フォント登録は `try/except` で囲み、万一登録に失敗しても `Helvetica` にフォールバックして処理自体は止めない構成にしています。

```
from reportlab.pdfbase import pdfmetrics
from reportlab.pdfbase.cidfonts import UnicodeCIDFont

try:
    pdfmetrics.registerFont(UnicodeCIDFont("HeiseiKakuGo-W5"))  # 同梱CIDフォント
    ja = "HeiseiKakuGo-W5"
except Exception:
    ja = "Helvetica"   # ← 失敗してもPDF生成は継続
```

## 4. プロンプトの工夫

本システムは「役割ごとに別人格のアナリストになる」ことが肝なので、プロンプト設計に最も力を入れました。プロンプトは**3層構造**で組み立てています。

```
# ロール固有プロンプト + 全ロール共通ルール + 出力言語の絶対命令
system_prompt = PROMPTS[role_id] + UNIVERSAL_RULES + LANGUAGE_OVERRIDE
```

### 4-1. ロールごとに「人格」と「レポート構成」を与える

各役割に、**一流ファームのプロフェッショナルという人格**と、**そのまま提出できるレポートの章立て**を細かく指定しています。

```
# VC/PE投資家のプロンプト（抜粋）
あなたは一流VC/PEファーム（Sequoia / KKR / Blackstone級）のシニアパートナーです。
読者は「資金配分するか」を15分で判断するファンドパートナー。投資判断メモを書きます。
宿題のようなテンプレートではなく、本物の投資メモとして、
すべての文章が資金投下を正当化するように書くこと。
...
■ カバーシート（要約表） ｜ 投資命題 ｜ ユニットエコノミクスとIRR分析 ｜ ...
推奨: ✅ LEAD（主導投資） / ⏸ WATCH（様子見） / 🔴 PASS（見送り）
```

```
# 証券アナリストのプロンプト（抜粋）
あなたはゴールドマン・サックス / モルガン・スタンレー級のシニア証券アナリストです。
読者はポートフォリオマネージャーで、20分で「買い/持ち/売り」を判断したい。
...
■ バリュエーション分析（DCF・マルチプル・感度テーブル）
★DCF内在価値と現在株価の乖離を必ず指摘し解釈する。
```

役割が変わると、登場するフレームワーク自体が変わります。経営者なら**3つの戦略オプション比較（提携／買収／内製）**、学生なら**Porterのファイブフォース・VRIO・Christensenの破壊的イノベーション**といった具合に、**読者の意思決定に直結する分析軸**を役割別に作り込みました。

### 4-2. 全役割共通の「8つのナラティブ原則」で品質を底上げ

役割が違っても**レポートの“質”は一定以上**に保ちたいので、共通ルール（`UNIVERSAL_RULES`）を全プロンプトに自動注入しています。

```
### 原則1: 見出しは「結論」にする（ラベルにしない）
✗「市場分析」 ✓「生成AI SaaSは$45B市場で28%シェアを握る」

### 原則2: 出典は必ず文中に併記（脚注にしない）
✓「Teslaの粗利は19.07%（Finnhub 2026-05-30）」

### 原則3: 断定的な言葉のみ（ヘッジ禁止）
✓「我々は提携を推奨する。理由は〜」  ✗「検討の価値があるかもしれない」

### 原則4: 結論を先に、根拠を後に（Pyramid Principle）
### 原則5: 各セクションは自己完結（ページ参照禁止）
### 原則6: 本文の表は6行以内（詳細は付録へ）
...
```

特に **原則2（全数値に出典・日付を併記）** は、§実装の工夫で取得した実データだけを使わせるための“縛り”として効いています。これにより\*\*「出典のない主張」を構造的に排除\*\*し、ハルシネーションを抑えています。情報の鮮度（直近のニュース）と長期トレンド（財務・マクロ指標）が競合しないよう、プロンプト内で優先順位と使い方も明示しています。

### 4-3. 出力言語の「絶対命令」で英語化を防ぐ

Azure AI Foundry上のエージェントは英語の構造に引っ張られて見出しが英語になりがちでした。そこで、**他の全命令に優先する言語命令**を最後に重ねています。

```
## 🚨 出力言語の絶対命令（最優先・他の全命令に優先する）
1. レポート本文・見出し・表のヘッダーは「すべて日本語」で書くこと。
2. セクション見出しは「日本語の結論文」にすること。
   ✗禁止: 「Executive Summary」「Rating & Target Price」
6. 【最重要】AIが書いたと分かるメタ説明を冒頭に絶対に書かない。
   ✗「以下は〜のレポートです」  ✓ いきなりリード文から始める
```

「`Executive Summary` ではなく日本語の結論文に」「`【Page 1】` のようなラベルを使わない」「`以下は〜です` のようなAI臭い前置きを書かず、いきなり記事のリード文から始める」——こうした\*\*“本物のレポートらしさ”を担保する細かな指示\*\*を積み重ねることで、提出にそのまま使える品質に近づけています。

### 4-4. 「分かりやすく解説」専用の比喩プロンプト

解説機能には、レポート生成とは別の専用プロンプトを使っています。

```
あなたは、難しい専門用語や固い文章を、身近な比喩・たとえ話を使って
「誰にでもスッと分かる」ように解説する名物講師です。
1. 必ず「比喩・たとえ話」を1つ使ってイメージで理解できるようにする
   （例:「LTV/CACは、常連が生む利益と、その客を呼ぶ広告費の比。
        1人500円かけて3000円使ってくれるなら大成功、という関係」）。
2. 全体で3〜5文程度（吹き出しに収まる量）。
```

「必ず比喩を1つ使う」「吹き出しに収まる3〜5文」と**出力の形を強く制約**することで、UIの吹き出しにぴったり収まる、分かりやすい解説を安定して得ています。

## 5. アーキテクチャ

### 5-1. 全体構成図

以下は本システムのアーキテクチャを Mermaid 記法で表したものです。

### 5-2. Microsoft Azure アプリケーション実行基盤：**Azure App Service**

アプリの実行基盤には **Azure App Service（Linux / Python 3.11 / F1 無料プラン）** を採用しました。

* **Flask アプリを `gunicorn` で起動**し、HTTPS付きの公開URL（`https://<app>.azurewebsites.net`）をそのまま審査・デモ用URLとして提供できます。
* ジョブ状態をプロセス内メモリで持つSSE構成に合わせ、**`--workers 1 --threads 8 --timeout 600`** で運用。1インスタンス・1ワーカーで進捗状態を一貫させています。
* **マネージドID（システム割り当て）**を有効化し、AI Foundryリソースへ `Azure AI User` ロールを付与。これにより**コード上にAzureの認証情報を持たず**、ローカルの `az login` から本番のマネージドIDへ、`DefaultAzureCredential` のまま無改修で移行できます。

### 5-3. Microsoft AI 技術：**Azure AI Foundry / Agent（Azure OpenAI）**

中核のAI技術として **Azure AI Foundry 上にデプロイした Agent（`MarketResearchAgent` v4）** を利用しています。アプリは Foundry の OpenAI 互換クライアントを通じて、この**エージェントを参照しながら**推論を実行します。

```
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient

client = AIProjectClient(endpoint=ENDPOINT, credential=DefaultAzureCredential())
openai_client = client.get_openai_client()

response = openai_client.responses.create(
    input=[
        {"type": "message", "role": "system", "content": system_prompt},
        {"type": "message", "role": "user",   "content": enriched_query},
    ],
    extra_body={"agent_reference": {
        "name": "MarketResearchAgent", "version": "4", "type": "agent_reference",
    }},
)
```

* **認証**：`DefaultAzureCredential` により、ローカルでは開発者の資格情報、App Service上ではマネージドIDを自動で使い分け。
* **Agentic な振る舞い**：「役割の選択 → データソース戦略の決定 → 6APIからの実データ収集 → 構造化 → 役割別プロンプトで分析・執筆」という一連の流れをエージェントとして組み立て、\*\*外部の一次情報に根拠を置いたレポート（RAG的な構成）\*\*を生成します。
* **同一エージェントの再利用**：メインのレポート生成と「比喩で分かりやすく解説」機能は、システムプロンプトを切り替えるだけで同じ Foundry エージェントを使い回しており、構成をシンプルに保っています。

### 5-4. 採用技術まとめ

| レイヤ | 採用技術 |
| --- | --- |
| 実行基盤（必須要件） | **Azure App Service**（Linux/Python 3.11, F1） |
| AI技術（必須要件） | **Azure AI Foundry / Agent（Azure OpenAI, responses API）** |
| 認証 | **Microsoft Entra ID + マネージドID**（`DefaultAzureCredential`） |
| Web / 進捗配信 | Flask + Server-Sent Events |
| 帳票出力 | reportlab（PDF・日本語CIDフォント） / python-docx（Word） |
| 外部データ | NewsAPI / Finnhub / Alpha Vantage / World Bank / UN Comtrade / EDINET |

## まとめ

本システムは、「個人が技術・市場トレンドを把握するには時間がかかりすぎる」という課題に対して、

* **役割（ロール）を選ぶだけで、データ収集戦略からまとめ方・出力形式までが切り替わる**設計、
* **6つの公的・金融APIを一次情報として統合し、出典付きで根拠あるレポートに落とし込む**実装、
* **3層プロンプトで“役割別の人格”と“一定品質”と“日本語出力”を両立**させたプロンプト設計、
* **Azure App Service × Azure AI Foundry** によるマネージドな実行・認証基盤、

によって、「**自分専用のアナリストに、自分の立場の言葉で報告してもらう**」体験を実現しました。専門用語をその場で比喩解説する機能まで含め、知識レベルを問わず誰もが市場の“今”を素早くつかめることを目指しています。
