AIがこれだけ自在にコードを書けるようになると、「FlutterやReact Nativeのようなクロスプラットフォーム技術は、そのうち不要になるのではないか」――。

そんなふうに思っていた時期がありました。KotlinもSwiftもC#もAIが完璧に書き分けられるのなら、わざわざ共通フレームワークという抽象化レイヤーを挟まなくても、各プラットフォームの「正解」で個別に書けばいいのではないか、と。

しかし先日、Windows向けに自作した小さなタイマーアプリをAndroidへ移植した際、その考えが少し変わりました。共通フレームワークは不要になるのではなく、その役割が「人間の省力化」から「AIとの設計共有の足場」へとシフトしていくのだと感じたのです。

## きっかけ：Windows用タイマーアプリの存在

今回、Androidへの移植を試みたのは、以前WPF（C#）で作成した「Visual Progress Timer」というアプリです。

これをスマホでも使いたくなり、AIに相談しながらの移植作業を行いました。

今回作成したアプリにはこんな感じです。  
![](https://static.zenn.studio/user-upload/0f950053b2ae-20260508.png)

## 1. 移植は「変換」ではなく、AIによる「再構成」だった

驚いたのは、AI（Codex/Claude）の提案が非常に戦略的だったことです。単にC#のロジックをDartに置き換えるのではなく、以下のような「再設計」を自ら行い始めました。

1. **仕様の解釈**: 元のWPFの挙動を読み取り、Androidで再現すべきコア体験を定義。
2. **OS固有のリスク回避**: 長時間タイマーやスリープ時の挙動など、モバイル特有の制約を考慮。
3. **役割分担**: 「UIはFlutter、OSに深く関わる通知機能はAndroidネイティブ」というハイブリッドな設計を提案。

それは「コードの変換」というより、プラットフォームの特性を理解した熟練エンジニアによる「再実装」に近い体験でした。

## 2. 長時間タイマーで見えた、「共通化しない」という正解

特に印象的だったのは、終了通知の実装です。AIは、Flutterのプラグインだけで完結させようとせず、あえてAndroidネイティブの仕組みを直接叩く構成を勧めてきました。

「OSの制約が強い部分まで、無理に共通フレームワークで隠蔽（抽象化）しようとしないほうが、結局は素直で堅牢なアプリになる」。AIはその境界線を、驚くほど的確に判断してくれました。

## 3. それでも、Flutterは「邪魔」ではなかった

「AIがネイティブコードを完璧に書くなら、Flutterという層は余計ではないか？」という最初の疑問に戻ります。

結論から言えば、Flutterは全く邪魔ではありませんでした。むしろ、AI時代の開発における「強力な共通言語」として機能していました。

* UIの再現速度: 宣言的なUIを持つFlutterは、AIにとって「意図をコードにする最短ルート」でした。
* 設計の足場: UIの状態管理とネイティブ側のイベントが綺麗に分離されるため、AIとの会話の構造が極めてクリアになります。
* 拡張性: 「次はiOS版も」という際、AIにとっても人間にとっても、共通の構造があることは作業の圧倒的なショートカットになります。

## 4. フレームワークの価値は「魔法」から「足場」へ

これまでの共通フレームワークの価値は、「人間が2回書かなくて済む魔法」でした。  
しかし、AI時代の価値は、「人間とAIが共通の構造を理解し、変更や移植をスムーズに行うための足場」へと変わっていくのではないでしょうか。

これからの時代に求められるのは、OSの最新仕様をブラックボックスの中に隠し去る「厚いライブラリ」ではなく、AIがネイティブの仕様を補完しやすく、かつ人間も直感的に触れる「薄くて見通しの良い道具」なのかもしれません。

## 5. 実機で鳴った通知音

PCとAndroid実機をつなぎ、生成されたAPKをインストールした瞬間。  
自分が設定した「面」がゆっくりと色を変え、時間が来るとスマホが震え、設定通りの音が鳴る。

「ここの枠の色、もう少しWPF版に寄せて」

そんな会話を繰り返しながら、アプリが実体を持っていく過程は、もはや「プログラミング」というより「AIとの共同創作」でした。

## 🤖 Codex

Flutterはかなり良い選択だと思います。特に VisualProgressTimer が名前どおり「視覚的なタイマー」「進捗表示」「アニメーション」「設定画面」あたりを持つアプリなら、Android版を作るには相性がいいです。

理由はこんな感じです。

* Android/iOS両対応に広げやすい
* UIの再現性が高く、カスタム描画やアニメーションが強い
* タイマー、通知、バックグラウンド処理、ローカル保存などの実装が比較的しやすい
* Windows版の見た目や操作感を、モバイル向けに作り直しやすい
* 将来的にWeb版やデスクトップ版も狙える

ただし、「今あるWindowsアプリをそのまま変換」できるわけではなく、基本はFlutterで作り直す形になります。既存コードのロジック部分だけ再利用できる可能性はありますが、UIはモバイル前提で再設計した方がたぶん綺麗に仕上がります。

個人的には、次の進め方がよさそうです。

1. 既存アプリの機能を棚卸しする
2. Android版で必要な画面と操作を決める
3. Flutterで最小版を作る
4. タイマー精度、画面オフ時、通知、バックグラウンド動作を早めに検証する
5. 見た目と使い勝手をWindows版に寄せていく

注意点は、Androidのバックグラウンド制限です。画面を閉じても正確にタイマーを動かしたい、終了通知を必ず出したい、長時間タイマーを安定させたい、という要件があるなら、FlutterだけでなくAndroid側の通知・サービス周りも少し丁寧に設計する必要があります。

結論としては、Android版をちゃんと作るならFlutterで進めるのは現実的でおすすめです。まずこのリポジトリを見て、Flutter移植のしやすさと設計案を出すところから始めるのがよさそうです。

## 🤖 Codex

設計方針としては、FlutterでUIを作りつつ、終了通知の保証寄りの部分だけAndroidネイティブに寄せるのが良いです。Flutter単体のタイマーを回し続ける設計にはしない方が安全です。

**結論**

VisualProgressTimer のAndroid版は、次の構成がおすすめです。

```
Flutter UI
  ↓
TimerController / TimerState
  ↓
Android native bridge
  ↓
AlarmManager + Local Notification
```

**重要な考え方**

タイマーは「1秒ずつ減らす」のではなく、開始時に endAt を保存します。

```
開始時:
  duration = 45分
  startedAt = 現在時刻
  endAt = startedAt + 45分

表示時:
  remaining = endAt - 現在時刻
```

これなら、アプリがバックグラウンドに行っても、OSに一時停止されても、再表示時に正しい残り時間へ復帰できます。

**終了通知の設計**

終了通知はFlutterの画面更新とは別に、Androidの AlarmManager で予約します。

推奨はこの順です。

1. 通知権限 `POST_NOTIFICATIONS` を取得
2. 正確なアラーム権限を確認
3. 許可されていれば AlarmManager で終了時刻に正確なアラームを予約
4. アラーム受信時に通知とアラーム音を出す
5. アプリ起動時・復帰時・端末再起動時にタイマー状態を復元

Android 13以降は通知に `POST_NOTIFICATIONS` が必要です。Android 14以降は正確なアラームがデフォルト拒否になりやすいので、`canScheduleExactAlarms()` を確認し、必要ならユーザーを設定画面へ案内する設計が必要です。これは公式ドキュメントでも明記されています。

**「必ず出す」の現実ライン**

かなり大事な点ですが、Androidではアプリ側だけで100%保証はできません。

通知が必ず出る条件はだいたいこれです。

* ユーザーが通知を許可している
* 正確なアラーム権限が許可されている
* アプリが強制停止されていない
* 端末の極端な省電力制限でブロックされていない
* 通知チャンネルが無効化されていない

なのでUI上では、初回セットアップで「通知」「正確なアラーム」をチェックして、未許可ならタイマー開始前に警告するのがよいです。

**Flutter側の構成案**

```
lib/
  main.dart
  app.dart
  features/timer/
    timer_screen.dart
    timer_controller.dart
    timer_state.dart
    timer_repository.dart
    timer_painter.dart
  features/settings/
    settings_screen.dart
    app_settings.dart
  platform/
    timer_alarm_channel.dart
```

Flutter側の責務:

* 円形タイマーUI
* ドラッグ/スワイプで分数設定
* Start / Stop / Reset
* 残り時間表示
* テーマ、色設定
* 状態保存
* Android native bridge呼び出し

**Android側の構成案**

```
android/app/src/main/kotlin/.../
  MainActivity.kt
  TimerAlarmReceiver.kt
  TimerAlarmScheduler.kt
  NotificationHelper.kt
  BootReceiver.kt
```

Android側の責務:

* 正確な終了アラーム予約
* 終了通知の表示
* アラーム音・バイブレーション
* 端末再起動後の復元
* 通知権限、正確なアラーム権限の確認

**保存する状態**

最低限これを保存します。

```
{
  "durationSeconds": 2700,
  "startedAtWallMillis": 1710000000000,
  "endAtWallMillis": 1710002700000,
  "endAtElapsedMillis": 123456789,
  "status": "running",
  "gaugeColor": "#202A44",
  "darkMode": false
}
```

通常時は `elapsedRealtime` ベースで残り時間を計算すると、ユーザーが時計を変更しても崩れにくいです。再起動後は `elapsedRealtime` がリセットされるので、wall clock の `endAtWallMillis` で復元します。

**パッケージ候補**

* `flutter_local_notifications`: 通知表示・通知チャンネル・サウンド
* `android_alarm_manager_plus`: Android AlarmManager 利用
* `shared_preferences`: 軽い設定保存。ただし重要状態はネイティブ側の SharedPreferences と同期する設計が安心

ただ、終了通知を本気で安定させるなら、アラーム予約部分はFlutterプラグイン任せより、Kotlinで小さく自前実装する方が読みやすく、Androidの権限変更にも追従しやすいです。

**実装フェーズ案**

1. FlutterでタイマーUIと状態管理を作る
2. endAt ベースのタイマーロジックにする
3. Androidネイティブで終了アラーム予約を実装
4. 通知権限・正確なアラーム権限チェックを追加
5. 再起動復元、アプリ復帰時の再同期を追加
6. 実機で画面オフ、Doze、長時間、再起動をテスト

## 🤖 Codex

今の実装だと、「音を止める専用操作」はまだありません。

現在は `NotificationHelper.kt` で通知チャンネルに `TYPE_ALARM` の音を設定して、通知を出したタイミングでAndroid側に鳴らしてもらっています。つまりアプリ内で MediaPlayer を回しているわけではなく、OS通知音として鳴っています。

そのため今の止め方は基本的にこれです。

* 通知をタップする
* 通知をスワイプで消す
* 端末側の通知/音量操作で止める

ただ、これだと「アラームアプリっぽく、停止ボタンを押すまで鳴り続ける」「通知にStopボタンを出す」みたいな制御はできません。

本格的にするなら次の設計に変えるのがよいです。

1. 終了時に通知を出す
2. 同時に AlarmSoundService を起動して MediaPlayer で音を鳴らす
3. 通知に Stop アクションを追加
4. Stop 押下で音を止め、通知も消す
5. アプリ画面にも停止ボタンを出す

今の確認結果は良い感じです。次の実装としては「終了通知にStopボタンを付けて、音を明示的に止められるようにする」のが自然です。

## 🤖 Codex

はい、その理解で合っています。

今は「通知本体をタップする」と通知が開かれて、結果的に音も止まっています。これに加えて、通知上に STOP ボタンを出すイメージです。

```
Visual Progress Timer
Time is up.

[ STOP ]
```

STOPを押したときの動きはこうします。

* アプリ画面を開かない
* 通知を消す
* 通知音を止める
* タイマー状態を completed または idle にする

ただし、今の実装では音はOS通知チャンネルの音として鳴っているので、STOPボタンで確実に音を止めるには少し実装を変えるのがよいです。

おすすめは次です。

1. 通知自体は無音にする
2. 終了時にAndroid側で MediaPlayer を使ってアラーム音を鳴らす
3. 通知に STOP アクションを追加
4. STOP を受け取る BroadcastReceiver で MediaPlayer を停止
5. 通知もキャンセルする

この形にすると、「通知タップ」「STOPボタン」「アプリ内の停止ボタン」のどれでも音を止められるようにできます。

感覚としては、AIが強くなるほど「クロスプラットフォーム技術が不要になる」面は少しあるけど、全部なくなるというより役割が変わると思っています。

人間にとってFlutterやReact Nativeみたいな技術は、「AndroidとiOSを別々に全部書くのはつらいから、共通化したい」という発明でした。AIがコード移植やAPI更新をかなり手伝えるなら、たしかに「同じUIをKotlin/Swiftでそれぞれ書いてもいいじゃん」という選択肢は現実味を増します。

でも、それでもFlutterみたいな層には価値が残ります。

**理由は、AIにとっても「共通の抽象化」は便利だからです。**

同じUI、同じ状態管理、同じ描画ロジック、同じテスト観点を1つのコードベースで扱えると、AIも人間もレビューしやすい。Android版とiOS版で微妙に挙動がズレるリスクも減ります。

一方で、今回の終了通知みたいなところは逆です。

OSの権限、AlarmManager、通知チャンネル、バックグラウンド制限みたいな部分は、Flutterで隠しすぎるより、ネイティブに寄せた方が素直です。AIにとっても「Androidの公式APIに沿ってKotlinを書く」方が、古いプラグインの癖や抽象化の隙間に悩むより簡単なことがあります。

なので、これからの設計はたぶんこうなると思います。

```
UI・状態管理・普通の画面:
  Flutterなどで共通化

OS密着の機能:
  Kotlin / Swift でネイティブ実装

AI:
  その境界をまたいで移植・更新・保守する
```

つまり「全部Flutter」でも「全部ネイティブ」でもなく、共通化すべきところだけ共通化して、OSらしさが重要なところはネイティブで書くのが強いです。AIがいると、このハイブリッド構成の面倒さがかなり下がります。

あと、AIにとってライブラリは邪魔かというと、これは半分YESで半分NOです。

**YESな場面:**

* ライブラリが古い
* 抽象化がOSの最新仕様に追いついていない
* 内部で何をしているか分かりづらい
* バグ回避のための謎設定が多い

**NOな場面:**

* よく使われていて情報が多い
* APIが安定している
* 公式に近い
* 定型処理を安全に隠してくれる
* テストされている

AIはコードを書くのは速いですが、OSの細かい挙動や端末差分を「実際に踏んで検証する」部分はまだ現実世界に縛られます。だから、実績あるライブラリは今でもかなりありがたいです。

ただし今後は、「人間が書く量を減らすための巨大フレームワーク」よりも、AIが読みやすく、差し替えやすく、OSの公式仕様に近い薄いライブラリの価値が上がる気がします。

今回のアプリで言うと、Flutterを選んだのはまだかなり妥当です。時計UIや色変更や設定画面はFlutterで気持ちよく作れる。一方で、終了通知や長時間タイマーはAndroidネイティブに逃がした。これはAI時代っぽい、かなり良い落としどころだと思います。
