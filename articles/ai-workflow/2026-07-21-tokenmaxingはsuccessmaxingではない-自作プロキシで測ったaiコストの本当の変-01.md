## 結論

先日、「[オープンモデルは価格を殴る棍棒だ](https://zenn.dev/ryok/articles/china-open-models-price-lever)」という記事を書いた。中国のオープンウェイトがクローズド勢のAPI単価を押し下げる、という話だ。だが最後にひとつ処方を置いた——「単価が下がっても、**トークンの使い方がザルなら値下げは机上の数字で終わる**」。

この記事は、その処方だけを取り出して掘る。

主張はシンプルだ。**あなたのAIコストの過半は、"どのモデルを選ぶか"ではなく、"送っていることに気づいていないトークン"で決まっている。** 私は自作のプロキシで自分の開発を1年ぶん測った。結果、成果を一切変えずに**54.7%のトークンが消えた**。モデルは変えていない。プロンプトの中身も変えていない。ただ「送らなくていいもの」を送るのをやめただけだ。

扱わないのは「どのモデルが安いか」（それは[別の記事](https://zenn.dev/ryok/articles/model-selection-pareto-frontier)で書いた）。扱うのは「何を送らないか」という設計だ。

---

## 第1部: いま起きている "tokenmaxing"

いまのAI開発の空気は、はっきり言って **tokenmaxing**（トークン最大化）だ。コンテキストは大きいほど良い、エージェントは多いほど良い、ダメなら投げ直せばいい——コンテキストウィンドウも並列エージェントもリトライも、全部「盛る」方向に進んでいる。

だが盛った量は、成果と相関しない。r/ClaudeCode の699票を集めた投稿が、この空気をひとことで刺していた——**「tokenmaxing と successmaxing は同じではない」**（[r/ClaudeCode](https://www.reddit.com/r/ClaudeCode/comments/1v0tw4s/)）。コメント欄の自虐が象徴的だ:

> 「180億トークン使って、リポジトリ名はまだ `untitled-project-final-v2`」

面白いのは、フロンティア側にも揺り戻しが出ていることだ。同じ週、OpenAI は Codex の投入コンテキストを **372k から 272k へ削減**した（[Hacker News](https://news.ycombinator.com/item?id=48965850)、259pt）。「多いほど良い」が自明でなくなってきている。

盛るのをやめる、という発想がここから始まる。ただし「性能を落として節約する」のとは違う。**成果を変えずに減らせる余地**が、実は大量にある。それを可視化するところから話を始めたい。

## 第2部: なぜ「不可視トークン」がコストを決めるのか

3つのメカニズムに分ける。

### (1) コストは "どのトークンを送るか" の分布で決まる

AIコストは乱暴に言えば「トークン量 × 単価」だ。モデル選びは単価の側をいじる。だが多くの人が見落とすのは、**トークン量の側は"何を送るか"で桁が動く**ことだ。

具体例を挙げる。エージェントに `curl` でAPIを叩かせると、生のレスポンス（巨大なJSON、HTMLの全文、HTTPヘッダの山）がまるごとコンテキストに入る。`ps aux` の全プロセス一覧、ビルドログの数千行、`git log` の全履歴——どれも「AIが必要としている情報」はごく一部なのに、ノイズごと送られる。単価をいくら下げても、ここが太いままなら効かない。

これは私だけの実感ではない。Quesma のエンジニアが「トークン節約法を研究していたら手持ちのトークンを全部溶かした」という皮肉な体験記で、同じことを別角度から測っている。彼らの報告では、**ハーネス（＝トークンをどう運ぶか）次第で最大66倍のトークン差**が出た（[Quesma](https://quesma.com/blog/custom-deep-research-pipeline/)）。66倍。モデルの単価差など誤差に見える桁だ。効いているのはモデルではなく、トークンの運び方なのだ。

### (2) ノイズは偏在する

そして重要なのは、**この無駄が全コマンドに均等に散らばっているわけではない**という点だ。ノイズは偏在する。

私の実測（後述）だと、`curl` 系のWeb取得は出力の **99%以上が冗長**で、消しても意味が変わらない。一方、ソースコードを読む操作は既に情報密度が高く、**10%程度しか減らない**。つまり「全部を圧縮する」のは筋が悪い。**どこにノイズが集中しているかを知り、そこだけ叩く**のがトークン設計だ。

### (3) フリーランチは実在する

経済学は「フリーランチはない」と言う。だが (1)(2) を合わせると、AIコストには例外的にフリーランチが存在する。**モデルを変えず、成果も変えず、コストだけが落ちる**領域だ。

なぜ可能か。高ノイズ領域（curl・ログ・ps）は、"意味を保ったまま"要約・圧縮できるからだ。APIレスポンスの必要なフィールドだけ抜く、ログのエラー行だけ残す、プロセス一覧を集計する——AIに渡る情報の質は変わらず、量だけ落ちる。これは「性能とコストのトレードオフ」ではない。ただの無駄取りだ。

## 第3部: 個人開発者はどうすべきか

概念を、再現可能な処方に落とす。一次資料として、自分の実測を出す。

### 処方1: 高ノイズ出力は「圧縮してから」渡す

私は開発コマンドのトークンを削るために、RTK（Rust Token Killer）という自作のCLIプロキシを常用している。`git status` や `curl` の出力を、意味を保ったまま圧縮してからエージェントに渡す仕組みだ。1年ぶんの `rtk gain`（実測）はこうなっている:

```
Total commands:    7,656
Input tokens:      54.1M
Output tokens:     24.5M
Tokens saved:      29.6M (54.7%)
```

累計 **29.6M トークン、54.7% の削減**。そしてコマンド別の内訳が、第2部の「偏在」を裏づける:

| コマンド | 削減率 |
| --- | --- |
| `curl -sL ...`（Web取得） | 99.5% |
| `ps aux` | 99.1% |
| `curl -s -H User-Agent ...` | 92.9% |
| `read`（ファイル読み） | 10.0% |

効いているのはモデルではない。「生の巨大出力を、要約を通してから渡す」——ただそれだけだ。RTKを使わなくてもいい。要点は、**Web取得・ログ・プロセス一覧のような"高ノイズ・低情報"の出力を、そのままAIに流さない**こと。

### 処方2: 既に痩せているものは、触るな

上の表の最終行が効いている。`read`（ファイル読み）は671回実行して**10%しか減らない**。ソースコードやテキストは、元から情報密度が高いからだ。

ここに落とし穴がある。「トークンを削る」と決めると、つい全部を圧縮したくなる。だが低ノイズ領域を削っても、労力に見合う削減は出ないうえ、情報を落として性能を下げるリスクだけが増す。**節約は偏在するのだから、最適化も偏在させる**。全体の数%を占める高ノイズ・コマンド群に集中投下するのが、費用対効果が最も高い。

しかも、雑な圧縮はむしろ逆効果になりうる。前述の Quesma は「素朴なコンテキスト圧縮がかえって請求を倍増させ得る」「キャッシュ無効化で隠れた再請求が発生する」ことも報告している（[Quesma](https://quesma.com/blog/custom-deep-research-pipeline/)）。プロンプトキャッシュが効いていた部分を下手に圧縮すると、キャッシュが無効化されて全体を再送する羽目になる。だから「とにかく短くする」ではなく、「高ノイズな所だけを、キャッシュを壊さずに」削るのが要点だ。

### 処方3: まず「測れ」。ツールは主役ではない

最後に、いちばん大事なこと。ここまでの話は全部、**測って初めて分かった**。どのコマンドが浪費源かは、`rtk gain` のような可視化がなければ永遠に見えない。見えないものは設計できない。

そしてフェアに言えば、RTKは万能ではない。`rtk proxy`（生のまま通す用途）やフォールバック経路は\*\*削減0%\*\*だ。ツールが魔法なのではなく、「**測って、偏りを見つけて、そこだけ叩く**」という習慣が本体で、RTKはその習慣を回すための計器にすぎない。プロンプトキャッシュでも、コンテキスト圧縮でも、要約通過でも、入口はどれも「まず測る」だ。

## 締め

モデルの値札は目に入る。だから私たちは「どのモデルが安いか」を熱心に比べる。一方、自分が毎日どれだけの不可視トークンを垂れ流しているかは、目に入らない。だから最適化は、見える方——モデル選び——に偏る。

でも本当の変数は、見えない方にある。

前の記事で「オープンモデルが単価を殴って下げてくれる」と書いた。その値下げを、自分の垂れ流しで相殺してしまってはもったいない。tokenmaxing をやめて、successmaxing へ。まず測ろう。削れるのは、たいてい、あなたが送っていることに気づいていないトークンだ。
