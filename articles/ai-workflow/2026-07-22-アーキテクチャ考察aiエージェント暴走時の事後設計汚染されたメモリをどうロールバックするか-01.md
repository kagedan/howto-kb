## TL;DR
* AIエージェントの暴走対策は「事前防御（どう止めるか）」に偏りがちだが、「事後設計（どう巻き戻すか）」が欠落していると致命傷になる。
* エージェントが共有メモリに誤情報を書き込んだ場合、プロセスをKillしても、すでにその情報を読み取った別エージェントによる「汚染の連鎖」は止まらない。
* ロールバックの鍵は「Writeの来歴」だけでなく、「誰がそのメモリをReadして意思決定したか」という消費記録（依存エッジ）を残すことである。

---

## 1. はじめに：事前防御の限界

現在、LLMを活用した自律型Agentの運用において、暴走（ハルシネーションや意図しない操作）を防ぐための対策が盛んに議論されている。権限の最小化、Human-in-the-Loop（承認ゲート）、プロキシによる外形監視、サンドボックス化などである。これら破壊的変更を未然に防ぐアプローチは間違いなく「第一の防衛線」である。

しかし、これらの議論には「異常が起きてもタイムリーに停止できる」という危険な前提が潜んでいる。現実には、検知から停止までに必ずタイムラグが発生する。プロセスをKillするまでの数分間に、Agentが共有ステート（メモリ）に何かを書き込んでいたらどうなるだろうか。

事前防御をどれほど強固にしても、「すでに書き込まれてしまった後」のリカバリ設計がなければ、システムには常に脆弱性が残ることになる。

## 2. プロセスをKillしても、汚染（ブラストラジアス）は止まらない

派手なDB削除のようなわかりやすい障害ではなく、「静かに嘘を書き込む」という、開発現場で極めて発生しやすいシナリオを想定してみる。

```text
# タイムライン：汚染の伝播シナリオ

14:02 [Agent-A] 調査タスク中、共有メモリに誤情報をWrite。
                "本番DBの接続先は db-legacy である" (※正しくは db-primary)
14:20 [Human]   異常を検知し、Agent-Aのプロセスを強制終了 (Kill)
14:35 [Agent-B] 共有メモリをReadし、db-legacyを前提としたマイグレーション計画を生成
15:10 [Cron]    夜間バッチがそのメモリを元に派生ドキュメントを再生成し、全社へ拡散
```

14:20の時点で `Agent-A` を停止させた運用者は、インシデント対応が完了したと錯覚しやすい。しかし、**被害は14:02に書き込まれた1行のテキストからすでに蔓延している。**

`Agent-B` はそれを事実として消費し、夜間バッチはそれを自己増殖させる。Aのプロセスを殺しても、Aが残したデータは生き続ける。
単に Git のように A が書いたファイルを元に戻すだけでは不十分だ。すでに B が生成した計画書やドキュメントは残存しており、汚染の「影響範囲（Blast Radius）」を特定して対処しない限り、真のリカバリとは言えない。

## 3. ロールバックに必須となる「3つの情報」

事後に汚染を巻き戻すためには、システムとして以下の3つを特定できるアーキテクチャが必要となる。

1. **誰が書いたか (Provenance)**
   汚染データは、どのAgentがどの実行コンテキストで書き込んだものか。
2. **以前の値は何か (Point-in-Time Recovery)**
   上書き保存（Update）では過去が消失してしまう。`Append-only + Tombstone（論理削除）` の設計にし、任意時点への巻き戻しを可能にする必要がある。
3. **誰が読んだか (Consumption Tracking) ★最重要**
   汚染データが書き込まれた後、**どの主体がそれを読み取り、意思決定に使ったか。**

1と2は、データストアをイベントソーシングやGitライクな追記型設計にすることで比較的容易に解決できる。しかし、ボトルネックとなるのは3である。
通常のアーキテクチャでは「Read操作」は状態を変更しないため、永続的なログとしては残されにくい。

## 4. 解決策：Readの消費記録を成果物に「刻印」する

すべてのRead操作をデータベース側でトラッキングしようとすると、システムがI/Oのボトルネックで崩壊する。重要なのは「意思決定に用いられたRead」だけを粗い粒度で記録することだ。

筆者が提案したいアプローチは、重厚なトレーシング基盤を作るのではなく、**Agentの出力（成果物）自体に依存メモリのIDをメタデータとして刻印させる**というシンプルなルールの徹底である。

### 実装イメージ

Agentが共有メモリのデータに基づいて意思決定をした場合、プロンプトエンジニアリングや出力スキーマの制約を通じて、必ず以下のような `sources` を持たせる。

```json
// Agent-B が出力したマイグレーション計画のJSON例
{
  "plan_name": "DB Migration",
  "target_db": "db-legacy",
  "steps": [ ... ],
  "_meta": {
    "generated_by": "Agent-B_run_987",
    "sources_referenced": [
      "mem/db-connection@14:02",  // ← 汚染源のメモリIDとバージョン
      "mem/schema-info@12:00"
    ]
  }
}
```

この設計（依存エッジの生成）を組み込んでおけば、インシデント発生時のフローは標準化される。

1. **検知**: `mem/db-connection@14:02` が汚染源であると特定する。
2. **影響範囲の特定**: ログ基盤やストレージに対して `grep sources_referenced: "mem/db-connection@14:02"` をかけるだけで、この嘘を信じて生成されたすべての成果物（Agent-Bの計画書など）を一網打尽に検索できる。
3. **ロールバック**: 原点を修正し、影響を受けたAgentのタスクを再実行（Re-run）する。

我々のチームで構築している `Memorylake`（記憶層アーキテクチャ）で最も注力しているのが、この「消費プロセスを含めたフルチェーントラッキング」である。
記憶を単なる「読み書き自由なホワイトボード」から、「改ざん不可能な台帳（Ledger）」へと昇華させることが、Agenticなシステムにおけるデータガバナンスの鍵となる。

## 5. 限界：事前防御と事後設計の役割分担

とはいえ、この事後設計（メモリロールバック）にも明確な限界がある。システム外部への副作用（Side Effects）だ。

* 送信してしまったメール
* 実行されてしまった本番環境へのデプロイ
* 完了してしまった決済

これらは、内部メモリをどれだけ綺麗にロールバックしても取り消すことはできない。
だからこそ、「事前防御（権限モデル、サンドボックス化など）」が不要になるわけでは決してない。

* **事前防御**: 不可逆な操作（外部への副作用）を食い止めるための関所。
* **事後設計**: 可逆な状態（内部メモリ）の汚染を速やかにクリーニングするための基盤。

この2つはトレードオフではなく、役割の異なる二段階の防衛線として設計されるべきである。

## 6. おわりに（筆者の所感）

本番環境でAIエージェントを稼働させるということは、「時として停止が遅れる、あるいは停止に失敗する」というリスクを受け入れることと同義である。
万が一停止できなかった時に、被害のブラストラジアスを正確に特定し、そこだけをサージカル（外科的）に切り取って巻き戻せるか。この「事後設計」の有無が、単なる冷や汗で済むか、致命的なシステム障害に発展するかを分ける。

白状すると、我々のチームの共有メモリ基盤でも、Writeの来歴管理はできているものの、Readのトラッキング（影響範囲の逆引き）についてはまだ荒削りで、人手による調査に依存している部分がある。
ひたすら事前防御を厚くして、事後設計を後回しにする。我々自身もその罠にハマっていた。

「いかにして暴走を止めるか」というアーキテクチャ図の隣には、必ず「万が一止められなかったらどうロールバックするか」の設計図を配置しなければならない。自戒を込めて、本記事を共有したい。
