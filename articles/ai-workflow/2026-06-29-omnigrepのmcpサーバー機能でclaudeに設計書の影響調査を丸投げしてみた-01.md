## はじめに

前回の記事では、Excelの図形テキスト・コメント・VBAまで横断検索して設計書の影響調査を効率化するツール「OmniGrep」を紹介しました。

https://qiita.com/OmniGrep_Studio/items/2b10802fb841ab6bf7b3

前回は「`orders` で検索 → 結果を確認」という1回の検索で完結するシナリオでした。しかし、実際の影響調査ではもう一歩踏み込む必要があります。

> 「`orders` テーブルを変更するなら、それを使っている処理はどれか？ その処理を呼び出している画面はどれか？」

これを手作業でやると、**検索 → 結果から処理IDを読み取る → そのIDで再検索**を何度も繰り返すことになります。

OmniGrepにはMCPサーバー機能があり、Claude等のAIから直接操作できます。この機能を使えば、**1回の指示で検索の連鎖を自動実行**できます。

この記事では、その具体的なやり方を紹介します。

## 手作業の「連鎖検索」で何が大変なのか

前回と同じ受発注管理システム「SmartOrder Cloud」の設計書群を例にします。

> 「`orders` テーブルのカラムを変更する。影響を受ける**画面**を特定してほしい」

### 手作業の場合

手順はこうなります。

**ステップ1: `orders` で検索して、関連する処理を特定する**

OmniGrepで `orders` を検索すると、処理設計書がいくつもヒットします。ファイル名から処理IDを読み取ります。

- `prc_PRC-ORD-001_注文登録_処理設計書.xlsx` → 処理ID: `PRC-ORD-001`
- `prc_PRC-ORD-002_合計金額計算_処理設計書.xlsx` → 処理ID: `PRC-ORD-002`
- `prc_PRC-APV-001_注文承認_処理設計書.xlsx` → 処理ID: `PRC-APV-001`
- ...（他にも `PRC-APV-002`、`PRC-PO-001`、`PRC-STK-001`、`PRC-STK-002`、`PRC-SYS-002` がヒット）

**ステップ2: 各処理IDで再検索して、呼び出し元の画面を特定する**

`PRC-ORD-001` で検索 → 画面設計書 `SCR-ORD-001` がヒット。
`PRC-ORD-002` で検索 → 画面設計書 `SCR-ORD-001` がヒット。
`PRC-APV-001` で検索 → 画面設計書 `SCR-APV-001` がヒット。
...これを**8回繰り返す**。

**問題点:**
- 処理IDを**目で読み取る**必要がある
- 処理が8つヒットしたら、**8回検索を手動で実行**する必要がある
- ベテランエンジニアならシステムの構造を知っているので見当がつくが、**途中からアサインされたメンバーには難しい**
- IDの連鎖を追うだけでは見落とす**グレーゾーン**の存在に気づけない

### MCP + Claudeの場合

Claudeに1回指示するだけです。

> 「`orders` で設計書を検索して、ヒットした処理設計書から処理IDを抽出して、その処理IDをすべてOR条件で画面設計書から検索して、影響を受ける画面を一覧にして」

Claudeが以下を**自動で実行**します。

1. OmniGrepのMCPサーバーに `orders` で検索リクエストを送信
2. 検索結果からファイル名のパターンを読み取り、処理IDを抽出
3. 抽出した処理IDをOR条件（`PRC-ORD-001 OR PRC-ORD-002 OR PRC-STK-001 ...`）で再検索
4. 結果を整理して「影響を受ける画面」の一覧を提示

**人間がやることは「指示を出す」だけ**です。

なお、OmniGrepは本体のGUI上でもOR検索に対応しています。検索バーに `PRC-ORD-001 OR PRC-ORD-002 OR PRC-APV-001` のように `OR` で繋いで入力すれば、複数キーワードの一括検索が可能です。ただし、「ordersで検索 → 処理IDを読み取る → OR条件を組み立てる」という手順は人間が行う必要があります。MCP + Claudeの場合はこの一連の流れがすべて自動化されます。

## MCPとは

<!-- MCP未経験の読者向けに簡潔に説明 -->

MCP（Model Context Protocol）は、AIモデルが外部のツールやデータソースと通信するためのプロトコルです。Anthropicが策定し、オープンな仕様として公開されています。

OmniGrepはMCPサーバーとして動作し、Claude Desktop等のMCPクライアントから検索リクエストを受け付けることができます。これにより、Claudeが「OmniGrepで検索して」という操作を自律的に実行できるようになります。

## セットアップ手順

### 1. OmniGrepのインストール

MicrosoftストアでOmniGrepをインストールします（無料）。

<!-- ストアリンクを挿入 -->

### 2. OmniGrepでプロファイルを設定

前回の記事と同様に、検索対象フォルダとファイル種別を設定します。

![プロファイル設定画面](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/4454119/26001082-d320-49ff-9f19-530df5810f12.png)



**重要: 「MCPサーバー経由でAIがファイルの内容（スニペット）を読み取ることを許可する」にチェックを入れると、検索結果のスニペット（ヒット箇所の抜粋）がClaudeに送信されるようになります。一旦今回はチェックを入れずに実施します**

このチェックはプロファイルごとに設定可能です。社内機密文書のプロファイルではOFFにして、デモ用や個人プロジェクトのプロファイルだけONにする、という使い分けができます。

### 3. Claude Desktopの設定

Claude Desktopの設定ファイル（`claude_desktop_config.json`）に以下を追加します。

```json
{
  "mcpServers": {
    "omnigrep": {
      "command": "omnigrep",
      "args": ["--mcp-server"]
    }
  }
}
```

設定後、Claude Desktopを再起動するとOmniGrepがMCPサーバーとして認識されます。

## 実演: 1回の指示で影響範囲を特定する

### Claudeへの指示

Claude Desktopで以下のように指示します。

> 「omnigrepで設計書を検索して、ヒットした処理設計書のファイル名から処理IDを抽出して、その処理IDをすべてOR条件で画面設計書から検索して、影響を受ける画面を一覧にして」

### 結果

Claudeが以下を自動で実行しました。

**Step 1: `orders` で検索 → 処理IDを抽出**

「orders」を含む処理設計書として8ファイルがヒットし、ファイル名から処理IDが自動抽出されています。

**Step 2: 処理IDでOR検索 → 影響を受ける画面を特定**

8つの処理IDをOR条件で画面設計書から検索し、**4画面**がヒットしました。

| 画面ID | 画面名 | ヒット箇所 |

ここまでが、人間が手作業でやっていた「検索 → IDを目で拾う → 再検索」の繰り返しに相当します。Claudeはこれを**数十秒で自動実行**しました。

### AIならではの付加価値

注目すべきは、Claudeが最後に出した**補足**です。

> 「SCR-ORD-002（注文一覧）、SCR-ORD-003（注文詳細）はStep 1の「orders」ワード検索でもヒットしていましたが、Step 2の処理IDによる参照は確認されませんでした。必要であれば対象に加えるかどうかご確認ください。」

これは「処理IDでは紐づかないが、ordersという単語自体は出現している画面」です。手作業で処理IDの連鎖だけを追っていたら見落とす可能性が高い「グレーゾーン」を、AIが自発的に指摘しています。

**単純な検索の自動化ではなく、人間が見落としがちな考慮点まで提示してくれる** — これもAI連携のメリットです。

### 手作業との比較

手作業の場合の所要時間を計算してみます。

画面設計書が11ファイル、検索すべき処理IDが8つとすると、

> **8 ID × 11ファイル = 88回の検索操作**

1回の検索操作（ファイルを開く → Ctrl+Fで処理IDを検索 → シートが複数あればシートごとにCtrl+F → 結果確認 → 閉じる → 結果を記録）にかかる時間は、Ctrl+Fの操作自体は5秒程度だとしても、シートの移動や結果の記録まで含めると**短く見積もっても1回あたり1分**はかかります。

> **88回 × 1分 = 88分。結果のとりまとめを加えると約2時間**

| 方法 | 計算 | 所要時間 |
|---|---|---|
| 手作業 | 8 ID × 11ファイル × 1分 + とりまとめ | **約2時間** |
| MCP + Claude | 1回指示を出すだけ | **数十秒** |

時間短縮だけでなく、**システム構造に詳しくないメンバーでも同じ品質の影響調査ができる**という点が重要です。ベテランエンジニアの頭の中にある「この処理はあの画面から呼ばれている」という知識を、AIが検索の連鎖で再現します。
「ベテランエンジニアのAさんしか把握してないから他のメンバーでは分からない！時間が掛かりすぎる！」
という属人化問題の解消にも繋がります。

## セキュリティについて

OmniGrepのMCP連携では、**プロファイルごとにAIへのデータ送信可否を制御**できます。

- チェックOFF: 検索結果のファイル名・位置情報のみ返され、ファイルの中身は送信されない
![スニペット許可OFF](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/4454119/dee75b29-ddec-4ac1-be1f-c5a163a51175.png)



- チェックON: 検索結果のスニペット（ヒット箇所の抜粋）がClaudeに送信される
![スニペット許可ON](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/4454119/c537429c-9b90-4f0c-8ea7-33c9b64715aa.png)

社内の機密設計書を扱う場合は、プロファイルを分けてチェックをOFFにすることで、情報漏洩リスクに対応できます。
