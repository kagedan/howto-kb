Anthropicが公開した Claude Mythos Preview は、単なる「次の高性能モデル」として片づけにくい性質を持っていると感じています。今回の論点は、モデルの能力が上がったという一般論ではありません。コード生成や推論の一般能力の延長として、実戦級の脆弱性探索やエクスプロイト生成にまで届き始めていることです。Anthropic がこれを一般公開せず、防御目的に限定した Project Glasswing として扱っているのは、その能力が便利さだけではなく、攻撃面にも直結しうると判断しているからだと見てよいと思います。

OpenAI 側も方向性はかなり近いように見えます。2026年4月公開の政策文書では、超知能への移行を前提に、従来の事前評価中心の発想では足りず、配備後のリアルタイム監視、監査、責任分界、threat modeling、red teaming、containment playbook を含む制度と運用が必要だと整理されています。報道ベースでは、Sam Altman が次世代モデルの事前学習完了を社内で伝えたともされていますが、少なくとも私が確認できた範囲では、そこは一次情報ではありません。そのため本稿では、OpenAI公式文書で確認できる部分を軸に扱います。

この記事の結論を先に書きます。  
Claude Mythos 級のモデルが一般公開されたときに重要なのは、全企業が一律に全開発を止めることではありません。そうではなく、重要資産を持つ企業ほど、平時のロードマップとは別に、緊急棚卸し、権限見直し、脆弱性検査、限定的なロックダウンを発動できる条件と権限を、あらかじめ経営レベルで定義しておくことです。極端に聞こえるかもしれませんが、これは「常時停止」を勧める話ではなく、「必要時に止められる状態を先に作る」話です。少なくとも、この前提をゼロとして扱うのは楽観的すぎると思います。

## なぜ Claude Mythos は「強いモデル」ではなく「脅威モデルの更新」なのか

Anthropic の公開情報で最も重いのはここです。  
Mythos Preview は、主要なOSと主要なWebブラウザ全体にまたがってゼロデイ脆弱性を見つけ、悪用可能な形に持っていけると説明されています。しかも見つけた脆弱性には、長期間見逃されていたものも含まれます。単発のクラッシュで終わるのではなく、複数の脆弱性を組み合わせて sandbox を抜ける exploit chain まで構成しているという説明です。これは、単に「強いコード生成モデル」が出たという話ではなく、攻撃側に必要だった技能と時間の一部が圧縮され始めていることを意味します。

さらに重要なのは、これは攻撃専用に特化したモデルではないことです。Anthropic は Mythos Preview を general-purpose の frontier model と位置づけています。つまり、特殊な兵器を別枠で作ったというより、一般能力の向上が一定水準を超えた結果として、防御にも攻撃にも使える危険な実力が立ち上がってきたと見るほうが自然です。

Anthropic が Project Glasswing を立ち上げたのも、その認識があるからでしょう。重要ソフトウェアを支える企業やOSS保守者、防御側の組織に先行アクセスを渡し、重要なシステムや依存関係を先に洗わせる。加えて、OSS セキュリティ向けの資金や利用支援も打ち出している。これは、「騒ぐ前に守る側が先に動け」というメッセージとして読むのが妥当です。

## 何が技術的に変わるのか

従来の防御は、かなりの部分で「攻撃者にとって面倒であること」に依存していました。  
脆弱性の発見から exploit 化までには、通常かなりの人手と時間がかかります。クラッシュを起点に read/write primitive を作り、ASLR や sandbox を越え、最終的に privilege escalation や RCE に持ち込む。この流れには経験も執念も必要でした。

しかし Mythos Preview のようなモデルが実用域に入ると、その前提が崩れます。Anthropic の記述には、JIT heap spray、複数脆弱性の chaining、local privilege escalation、サーバ側の深刻な侵害に相当する exploit の例が並んでいます。要するに、これまで「高度な人が時間をかけてやっていたこと」が、モデルの並列試行能力と継続実行能力で圧縮されるわけです。

ここで本当に問題になるのは、摩擦で守っていた防御の価値が下がることです。面倒だから諦めてもらえていた層の防御は、モデル支援の攻撃者には効きが悪くなります。一方で、KASLR や W^X のような hard barrier は引き続き重要です。つまり今後は、「なんとなく面倒にしてある」対策より、壊しにくい構造的対策の比重が上がると考えるべきです。

## だから経営と先に握るべきことがある

この変化を現場の努力だけで吸収しようとするのは危険です。  
理由は単純で、必要なのは技術だけではなく、優先順位を変更する権限だからです。

ただし、ここで言いたいのは「常に全案件を止めろ」ではありません。そうではなく、次のような条件を満たしたときに、通常開発より防御を優先できる仕組みを経営と握っておくべきだ、ということです。

* 重要資産に対する新しい脅威モデルが確認されたとき
* ベンダー側の修正待ちでは露出期間が長すぎると判断されるとき
* 権限の強いシステムや端末に攻撃波及の可能性があるとき
* 監視、封じ込め、権限縮小を先にやらないと被害が広がるとき

必要なのは全面パニックではなく、非常時の優先順位変更を事前に制度化しておくことです。ここを決めずに「現場で頑張ってください」と言うのは、さすがに無責任です。

## 公開前に作っておきたい実務チェックリスト

以下は、Claude Mythos 級モデルの一般公開を想定して、先に用意しておきたい項目です。  
重要なのは、自社管理下または許諾済み環境で、防御目的に限定して再現・検証することです。攻撃者の手口を知らずに守れるほど、状況は甘くないと思います。

### 1. 資産と攻撃面の棚卸し

* 公開中の Web アプリ、API、管理画面
* 社内端末、VDI、踏み台、VPN、EDR、MDM
* CI/CD、artifact repository、runner、secret store
* SaaS 連携、OAuth 設定、MCP/コネクタ、Webhook
* OSS 依存、社内共通ライブラリ、放置レポジトリ
* 野良スクリプト、個人運用バッチ、管理者権限付きツール

ここで抜けると、以降の話は全部甘くなります。

### 2. アプリケーション層の確認

* 認証・認可の境界が曖昧な箇所
* SSRF、IDOR、権限昇格、CSRF、XSS、SQL/NoSQL Injection
* テンプレート注入、RCE につながる unsafe deserialization
* ファイルアップロード、画像・動画・文書処理パイプライン
* PDF、Office、FFmpeg、画像変換系の処理チェーン
* ブラウザ拡張、Electron アプリ、埋め込みブラウザ利用箇所

公開情報を見る限り、ブラウザ周辺やメディア処理は軽視しないほうがよいです。

### 3. インフラ・端末・権限の確認

* ローカル管理者権限の見直し
* 長寿命トークン、共有アカウント、放置APIキーの失効
* 特権アカウントの分離と操作ログ
* OS・ブラウザ・プラグイン・拡張機能の更新統制
* 端末の USB、外部記憶媒体、開発者モード、sideload 制御
* 重要端末への application allowlist、ネットワーク分離

「そのうちベンダーが直すだろう」で待つのは、BCP の発想としてかなり弱いです。

### 4. AIエージェント経由の侵入口確認

* プロンプトインジェクション耐性
* 社内文書検索や外部検索の混線
* ツール呼び出し権限の最小化
* ブラウザ自動操作、CLI 実行、自動コード修正の範囲制限
* 秘密情報を参照できるコネクタの棚卸し
* モデル出力をそのまま実行・反映する経路の遮断
* 人間承認のない自動化チェーンの停止条件

エージェント時代は、モデル単体の安全性より、どの権限で、どの文脈につなぎ、どこまで実行させるかのほうが本体になります。

### 5. 検知・封じ込め・監査

* 重要操作の監査ログ集中
* トークン不正利用検知
* 異常な外向き通信、圧縮・分割送信の検知
* canary token、honey credential 配置
* シークレット漏えい時の即時失効フロー
* 端末隔離、アカウント凍結、接続先遮断の手順化
* 休日・夜間を含む緊急時連絡網

事故をゼロにするより、起きた瞬間に止める能力を持つほうが重要です。

### 6. OSS の扱い

これは自社開発以上に重要な場合があります。  
Anthropic も Glasswing で、第一者システムだけでなく OSS のスキャンと保護を前提にしています。現実には、自社サービスは自社コードだけで動いていません。脆弱な依存先、更新されないトランジティブ dependency、保守が薄い周辺ツールが、攻撃の踏み台になります。

そのため最低限、以下は先にやるべきです。

* SBOM の整備
* 依存関係監査
* 重要OSSのバージョン固定方針見直し
* maintainer 依存が高い要素の代替候補確認
* 署名、provenance、artifact 検証
* CI runner と build secrets の隔離

「うちは SaaS だから関係ない」は成立しません。SaaS も OSS の上に立っています。

## 場合によっては社員PCのロックダウン宣言も必要になる

ここはかなり強い表現ですが、完全に0のシナリオではないと思います。  
ただし重要なのは、これを全社一律、常時の措置として語らないことです。妥当なのは、特定部門、特定端末、特定期間に限定したBCP上のオプションとして定義することです。

たとえば、以下のような条件付き措置は十分に現実的です。

* 重要部門のみ一時的に端末設定を厳格化する
* 未承認拡張機能やローカル実行ツールを禁止する
* 特定 SaaS 連携を一時停止する
* リモート接続経路を絞り込む
* 特権操作を二重承認にする
* モデルやエージェント利用範囲を暫定的に縮退運転する

重要なのは、「ロックダウンするかどうか」そのものではなく、「いつ、誰が、どの範囲で発動するか」を平時に決めておくことです。

## まとめ

Claude Mythos Preview の本質は、「Anthropic の危険な新モデルが出た」という単純な話ではないと思います。  
本質は、フロンティアモデルの一般能力が、実戦級のサイバー攻撃能力と防御能力の両方に接続し始めたことです。Anthropic はそのために Mythos を一般公開せず、防御側へ限定配布し、業界横断で重要ソフトウェアと OSS を守ろうとしています。OpenAI もまた、超知能時代を前提に、配備後監視、監査、containment、責任分界を含む運用と制度の整備を急ぐべきだと整理しています。

ですので、Claude Mythos 級の能力が一般化したときに私が本当にやりたいことは、「全部止めること」そのものではありません。  
必要なときに、重要資産から優先的に止められることです。  
止めて、洗って、塞ぐことを、例外的かつ迅速に実行できる状態を先に作っておくことです。

そのための停止条件、権限、予算、ロックダウン判断、BCP、OSS を含む点検手順を、公開前から経営と握っておく必要があります。  
新しいモデルが出た瞬間に「何に使おうか」と盛り上がること自体は悪くありません。ですが、それより先に「どこを止めるか」「どこを閉じるか」「どこを直すか」を決めている会社のほうが、おそらくこの変化には強いはずです。
