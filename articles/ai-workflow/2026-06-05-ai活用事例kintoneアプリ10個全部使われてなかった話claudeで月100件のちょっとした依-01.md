**半年前に旗を振って kintone を入れた。アプリは10個作った。日常的に使われているのは3個だった。** 残り7個は誰も触っていない。それでもバックオフィスは毎日忙しい。月100件の「ちょっとした依頼」がメール／Slack／口頭で飛び交って、3名の総務・経理・兼任担当が消耗していく。──これは小峰製作所の社長・小峰誠太郎（55、二代目）の机から見た半年間の総括です。

2026年5月、サイボウズが kintone 公式の MCPサーバー（2025年8月リリース）の Tips を出していたのを偶然知り、Claude Desktop と繋いでみた。最初の3日で、バックオフィス3名が抱えていた月100件の依頼のうち、半分が「Claude に聞けば終わる」状態になった。アプリ10個のうち7個は "乾いたまま" だったはずなのに、Claude が間に入っただけで全部が "生き返った" 感覚です。

本記事は、業界別ケーススタディの第2弾「中小製造業バックオフィスの kintone × Claude 九部作」のオープニング（全9回）。3部作だった税理士法人（第17〜20弾）とは別の架空企業ですが、税理士四部作の顧問先として登場した **株式会社小峰製作所** を主役に格上げします。九部作を通じて、この会社のバックオフィス（総務・人事労務・経理・横断検索・ダッシュボード）が順番に変わっていく流れを、社長視点で予告するのが第21弾の役割です。

## 月100件は何に消えていたのか

「ちょっとした依頼」の月100件、内訳を社長と桐山（総務人事44）、西尾（経理48）の3名で洗い出すと、こうなります。

* **総務系の依頼**（桐山宛）：月40件前後／名刺発注・備品発注・入退社チェックリスト確認・社外向け証明書発行
* **経理系の依頼**（西尾宛）：月35件前後／経費精算ステータス確認・請求書発行依頼・支払予定日の確認・残高見込み
* **横断系の依頼**（誰宛か曖昧）：月25件前後／「あの案件、どうなってる？」「先月のA社向け売上いくらだった？」「半年前の議事録どこ？」

依頼1件あたりは3〜10分。1件で見れば短いが、累計で **月15〜30時間**。3名のバックオフィスが毎月「2〜3営業日ぶん」を依頼の取り次ぎだけで使っている計算です。

kintone を入れたとき、社長はこのうち相当量を「アプリで吸収できる」と踏んでいました。だから10個も作った。実態は違って、**「アプリは作ったが、現場が結局 Slack で聞いてくる」**。アプリの場所を覚えていない、入力フォーマットが分からない、画面遷移が面倒、入れた後の結果がどう流れるかの説明がない──そういう「使われないアプリ」の溜まり場が、バックオフィスの机の上に積み上がっていました。

## 解決アプローチ：「kintone を触らせる」のをやめて、「kintone を Claude に触らせる」

転換点は、2026年5月にサイボウズ公式ページで「kintone 公式MCPサーバー」（2025年8月28日リリース）を見つけたところからです。

MCPサーバー（Claude に外部ツールを直接触らせる仕組み）に、ようやく kintone 自身が乗ってきた。これは、これまで「kintone を従業員に触らせる」前提だった世界が、**「kintone を Claude に触らせる（従業員は Claude にだけ話す）」** という選択肢に切り替わったことを意味します。

桐山と西尾に「アプリを使ってくれ」と言って回るのをやめる。代わりに、Claude Desktop に「今月のA社向け売上は？」「鈴木さんの入社チェックリスト、未完了は？」「経費精算で7日以上滞留しているのは？」と聞くだけで、kintone を裏で読みに行ってくれる世界に切り替える。

これがオープニング回の基本設計です。次の3つを順に開示します。

1. kintone 公式MCPサーバーの接続手順（APIトークン発行＋Claude Desktop の設定）
2. 3層ガードレール（Get自動許可／Update都度確認／Delete禁止）の組み方
3. 最初のテスト：「先月のA社向け売上はいくら？」を Claude に聞いて、kintone から答えが返るところまで

技術裏付けの数字を1段落だけ。サイボウズ公式によれば、kintone MCPサーバーは「スタンダードコース以上」で利用可能、現時点で **APIトークン最大9個** まで登録できます。1アプリ1トークン運用だと、最大9アプリが Claude から直接触れる範囲です（10個目以降は運用設計で吸収する話を「失敗パターン」セクションで後述）。中小製造業 kintone 導入企業の約78%が業務効率30%以上の改善を報告している調査もありますが、これは「kintone そのもの」の数字。そこに公式MCPサーバーが乗ったいまは、上限がもう一段上に動いた局面と読んでいます。

## 実装サンプル①：kintone APIトークンを発行する

最初の関門は地味で、APIトークンを発行する画面です。kintone の各アプリの「設定」→「カスタマイズ・サービス連携」→「APIトークン」から生成します。

サイボウズ公式が推奨しているのは、**APIトークンごとに権限を最小化** すること。レコード閲覧／レコード追加／レコード編集／レコード削除／アプリ管理の5つのチェックボックスで、トークン単位に「何ができるか」を決められます。

小峰製作所では、最初に接続する9アプリのトークン設計を、社長と桐山・西尾で次のように切りました。

* 交通費申請／営業案件／入退社：閲覧＋追加＋編集（Delete OFF・アプリ管理 OFF）
* 顧客マスター／商品マスター／取引先マスター：閲覧のみ（参照用）
* 売上実績／月次会議資料／勤怠：閲覧＋追加（編集 OFF・Delete OFF）

「Delete は全アプリで OFF」が、3層ガードレールの最下層になります。AI が誤って消す経路を、APIトークンの段階で塞いでしまう。

## 実装サンプル②：Claude Desktop に kintone MCPサーバーを繋ぐ

接続側はサイボウズ公式の GitHub（kintone/mcp-server）が配布する **MCPB ファイル**（Claude Desktop 用拡張機能パッケージ）をドラッグ＆ドロップする方式がいちばん早く、Claude Desktop の「拡張機能」ページで kintone MCP サーバーが追加されます。設定ファイル（claude\_desktop\_config.json）に直接書く方式もあり、9アプリ運用ならこちらが管理しやすい。骨格はこうなります。

```
{
  "mcpServers": {
    "kintone": {
      "command": "npx",
      "args": ["-y", "@kintone/mcp-server"],
      "env": {
        "KINTONE_BASE_URL": "https://komine-mfg.cybozu.com",
        "KINTONE_API_TOKEN": "abcd****1234,efgh****5678,ijkl****9012,mnop****3456,qrst****7890,uvwx****1234,yzab****5678,cdef****9012,ghij****3456"
      }
    }
  }
}
```

ポイントは `KINTONE\_API\_TOKEN` 1個にカンマ区切りで9個までトークンを並べる仕様（サイボウズ公式の現行仕様）。Claude Desktop を再起動すると、「kintone」MCPサーバーが Claude の道具箱に追加され、`kintone-get-records`／`kintone-add-records`／`kintone-update-records`／`kintone-update-statuses` などの操作系ツールが Claude から見える状態になります（全23ツール）。

![](https://assets.st-note.com/img/1780541313-Utsuqdho9kAHnXmxBQ8P6agF.png?width=1200)

23ツールがリストで並び、各行の右側に3つのアイコン（✓＝承認不要／🖐＝承認が必要／🚫＝無効）が並んで、現在の選択状態がアクティブ表示されている

## 実装サンプル③：「今月の交通費申請を集計して」を Claude に聞いてみる

接続できたら、最初に流すテストプロンプトを1本。

```
交通費申請アプリの今月の申請を集計してください。
出力フォーマット：
- 件数（合計）
- 合計金額（税抜・税込が分かれば両方）
- 申請者ごとの内訳（人数・件数・金額）
- 金額上位3件（申請者／利用日／区間／金額）

なお、参照のみで構いません。レコードの追加・編集・削除はしないでください。
```

Claude は kintone-get-apps（アプリ一覧）→ kintone-get-form-fields（フィールド構成）→ kintone-get-records（レコード取得）を Get系自動許可ルールで連続して叩き、4個目のツールで取得結果を集計して返してくれます。所要時間およそ10秒。

これまで桐山が「ちょっと待ってください、kintone 開いて……今月の申請ってどれだっけ？」と画面を切り替えながら2〜3分かけていた作業が、社長の机で完結しました。アプリの場所を覚える必要も、検索条件を組み立てる必要もない。**「kintone はあるが、誰も画面を開かない」** 状態が、ここから始まります。

## 失敗パターン：APIトークン9個制限を知らずに、10個目を繋ごうとして詰まる

最初の3日で1つだけ詰まったのが、APIトークンの上限。kintone MCPサーバーは現時点で **APIトークン最大9個** までです。1アプリ1トークン運用だと、9アプリ目で枠が尽きます。

小峰製作所には10個のアプリがあり、「全部 Claude から触れる状態にしたい」と社長が言ったところで、私（GRIS落合）が止めました。10個目（議事録アプリ）は使用頻度が低く、Claude から触りたい場面が月数回しかない。これを「常時接続のトークン枠」に入れると、より頻度の高い経費精算・営業案件・売上実績のどれかを外す羽目になります。

対処はシンプルで、**「常時接続する重要アプリ9個」と「臨時接続する周辺アプリ」を分けてトークン枠を管理する** こと。臨時側は、必要なときだけ Claude Desktop の設定ファイル（claude\_desktop\_config.json）でトークンを差し替える運用にします。サイボウズ公式GitHub のリリースノートでも、今後の上限拡張がアナウンスされていますが、現時点（2026年6月）では「9個枠の優先順位設計」がそのまま運用の一部になります。

この他にも初期2か月で踏んだ穴は「Update都度確認の頻発で Claude が使いづらくなる」「Delete禁止し忘れてテストで大事なレコードを消しかける」の2つ。前者はヒット率の高い Update 先アプリだけ「自動許可」に格上げ、後者は **全アプリのAPIトークンで Delete OFF を初期テンプレ化** するルールで第2弾以降に潰します。
