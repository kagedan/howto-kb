## 概要

この動画で Ismail Pelaseyed が繰り返し強調しているのは、**AI によって「脆弱性を見つけて、つなぎ合わせて、攻撃可能な形にするまでのコスト」が劇的に下がった**という点です。以前なら複数の攻撃者と数週間を要した作業が、今では 1 時間程度で済むこともある。しかも必要なのは最先端モデルではなく、小さめのオープンソースモデルでも十分だと言います。

その結果として、オープンソースのサプライチェーンは非常に攻撃されやすい場所になりました。入口として典型的なのはフィッシング、次に古くなった CI/CD、そしてメンテナが深く意識していない小さな依存関係です。動画の中心メッセージは単純で、攻撃者が AI を使える時代に入った以上、メンテナ側も無料で使えるスキャン、パッケージマネージャの設定、依存関係の更新方針、信頼できるコントリビュータ管理などを、もはや「やれたらやる」ではなく日常運用として組み込まなければならない、ということです。

同時にこの動画は、防御の強化がそのままオープンソースの活力を削りかねないという矛盾にも踏み込みます。新しいパッケージが採用されにくくなり、大企業は自前実装や既存大規模パッケージに閉じていく。その先にあるのは、Ismail が言う「Darwinian GitHub」です。つまり、守るコストを払える巨大プロジェクトしか残らない未来です。

---

## 要点

1. AI によって、公開コードから脆弱性を探し、攻撃経路を組み立てるコストは大きく下がった
2. 実際の侵入経路として最も多いのは、高度なゼロデイではなくフィッシングである
3. 古い CI/CD ワークフローや設定ミスのあるリポジトリは、トークン流出や不正パッケージ公開の足がかりになりやすい
4. 攻撃者は巨大パッケージだけでなく、見落とされがちな小さな依存関係を起点にサプライチェーン全体へ広がる
5. 防御策としては、CI/CD の定期監査、PNPM への移行、CodeQL・Dependabot・Socket などの無料スキャン有効化が強く推奨される
6. 単純スキャンだけでなく、連鎖 exploit を探す adversarial agent のような攻撃者視点の検査も必要になってきている
7. 外部コントリビュータ制限や新規パッケージ導入の遅延は組織防衛には有効だが、オープンソース全体には負荷をかける
8. 個々のメンテナだけでは守り切れず、レジストリ運営企業やエコシステム全体での対策が必要である

## 詳細

### AI は「攻撃成立までの時間」を圧縮した

動画は「**今もっとも多い攻撃パターンはサプライチェーン攻撃だ**」というメッセージから始まります。

ここでいうサプライチェーン攻撃とは、アプリ本体を正面から破るのではなく、開発者や利用者が信頼して取り込んでいる依存パッケージ、配布経路、公開権限、CI/CD のような「ソフトウェアを届ける流れ」を汚染する攻撃です。たとえばメンテナや権限保持者をフィッシングして NPM や PyPI、GitHub の公開権限を奪い、その権限で依存パッケージに悪意ある更新を混ぜる。あるいはインストール時の script や workflow を悪用して、トークン窃取やデータ持ち出しにつなげる。つまり「自分のコードが直接破られる」のではなく、「自分が信頼して使ったもの経由で破られる」タイプの攻撃を指しています。

この観点に立つと、セキュリティに敏感な組織ほど抱えやすい逆説的なリスクも見えてきます。つまり、「少しでも古いパッケージは危ないからすぐ新しくする」という運用を機械的に回していると、汚染された新しい版が出たときに、それを誰よりも早く取り込んでしまう可能性があるということです。古い版を放置するのも危険ですが、最新化それ自体を安全性と同一視すると、攻撃者にとって都合のよい配布経路をこちらから太くしてしまう、という別の危うさがあります。

Ismail は AI によって攻撃の economics が変わったと説明します。公開リポジトリに過去のバグが残っていれば、以前はそれを掘り起こして攻撃可能な流れにまとめるまでに相当な人手と時間が必要でした。しかし今は、攻撃者が AI を使ってその工程を一気に短縮できるようになっています。

ここで重要なのは、Ismail が「frontier model すら不要だ」と言っていることです。つまり、最先端モデルへのアクセスを持つごく一部の攻撃者だけの問題ではありません。小さめのオープンウェイトモデルでも、使い手が分かっていれば十分に役立つ。この認識は重いです。防御側が「そこまで高度な相手は限られるだろう」と楽観しづらくなるからです。

この前提を置いたうえで、彼は「私たちが顧客と見ている攻撃の入口は意外なほど単純だ」と話を進めます。

### 入口は高度な侵入ではなく、まずフィッシングである

彼によれば、実際の攻撃の多くは組織内の誰か、あるいはプロジェクトへの権限を持つコントリビュータがフィッシングされるところから始まります。メールでも X の DM でもよく、本人の例としても、「TechCrunch から記事化の相談です」と装って cal.com に誘導するようなメッセージが挙げられています。クリックした瞬間に侵害され、その先でトークンや権限が使われていくわけです。

この話が示しているのは、AI 時代の脅威を「AI が自力でトークンを発見する」ような派手な物語として捉えすぎないほうがいい、ということです。Ismail の見立てでは、大半はもっと人間的な弱点から始まる。しかも、そこが厄介です。フィッシングは以前よりずっと見分けにくくなっているからです。送信ドメインを見れば怪しさが分かった時代とは違い、今はプロバイダ側のフィルタをすり抜けるメールも少なくありません。

そのため彼は、受信メール内のリンクを二重チェックする仕組みを自分で bot として入れていると話します。派手な対策ではありませんが、ここでのメッセージは明快です。メールをクリックする前の確認コストを、何らかの形で追加しろ、ということです。

### 次の弱点は、古くなった CI/CD と設定ミスのある自動化である

続いて彼が挙げるのは、古くなった CI/CD ワークフローです。AI 以前でもここは重要でしたが、今は事情が違います。CI/CD の中でエージェントを動かす、生成 AI を組み込む、複数のトークンや環境変数を扱う、といった構成が珍しくなくなりました。その結果、ワークフローの書き方や権限の持たせ方の甘さが、そのまま侵害の連鎖に使われやすくなっています。

Ismail は、設定ミスのあるリポジトリやワークフローの中で、トークンが漏れる、複数の脆弱性がつながる、持ち出しが成立する、最後には悪意あるパッケージが公開される、という流れを見ていると説明します。ここでのポイントは、単発の設定不備ではなく、複数の小さな欠陥を AI が短時間で組み合わせやすくなったことです。

### 攻撃者は「誰も気にしていない小さな依存関係」から入ってくる

動画の中盤では、攻撃者は最初から巨大パッケージを狙うとは限らない、という話が出てきます。むしろ狙われやすいのは、広く使われているのに、みなが無意識に安全だと思い込んでいる小さな依存関係です。月に一度だけメンテされるようなパッケージでも、サプライチェーンの深い場所に入っていれば、そこから先の被害は一気に拡大します。

多くのチームは、自分たちが直接依存している大きなライブラリには注意を払いますが、その下にぶら下がる細かい依存関係までは把握しきれません。動画はまさに、その「当然安全だと思っていた部分」が攻撃面になっていると指摘しています。

### 具体策1: CI/CD の監査を routine にする

ここで Ismail は、防御策をかなり具体的に 2 つ挙げます。1 つ目は CI/CD ワークフローの定期チェックです。ベストプラクティスを追い、定期的に見直し、必要なら skill や CodeX、Cloud Code のようなツールを使って検査する。重要なのは、これを思いついたときにだけやるのではなく、routine として組み込むことだと強調しています。

この指摘は地味ですが、本質的です。AI が脆弱性探索を加速するなら、防御側も「たまに点検する」程度では間に合わない。更新されるベストプラクティスに合わせて、ワークフロー側も継続的に手入れしなければならないからです。

### 具体策2: NPM 前提を見直し、PNPM の制御を使う

2 つ目の提案は、依存関係管理そのものの見直しです。彼はかなりはっきりと、JavaScript 開発者は NPM を使い続ける前提を考え直し、PNPM のような選択肢を使うべきだと言います。理由は、依存更新のタイミングや方式、受け入れる lifecycle hook の制御など、より厳密なポリシーを持たせやすいからです。

このあたりでコメント欄から「結局 skill 自体にハックされるのでは」という反応も拾っていますが、彼はその懸念も認めつつ、少なくとも自分の環境や公開している環境変数、GitHub Actions の設定などを理解しておくべきだと返します。要するに、ツールを入れるだけでは足りず、「何がどこで走っていて、何に触れられるのか」を把握し続ける必要があります。

### 新規パッケージをすぐ入れない方針は有効だが、副作用が大きい

司会者が紹介する「公開から一定日数が経つまでインストールしない」ポリシーについて、Ismail は組織防衛としては肯定します。露骨な問題なら、最初の数日で表面化する可能性が高いからです。ただし、彼は同時に、それがオープンソース全体にとってかなり痛いと見ています。

理由は単純で、新しいパッケージや新しいメンテナが入り込む余地を大きく狭めるからです。結果として、安全対策に十分な資本を投じられる大型プロジェクトだけが有利になり、小さな新規プロジェクトは採用されにくくなる。彼が「Darwinian GitHub」と呼ぶのは、この淘汰圧のことです。防御策として合理的であるほど、エコシステムには息苦しさが増す。その矛盾が率直に語られています。

### サプライチェーン攻撃が多い理由は、結局「人のほうが簡単に破れる」から

動画中でもっとも印象的な一言は、「人をハックするほうがコンピュータをハックするより簡単だから」という答えです。フィッシングで入口を作れれば、そのあとの実装は post-install script や pre-install script のような比較的単純なものでも成立する。Socket.dev のようなツールがすぐ検知できるほど単純な手口でも、入口が取れていれば被害は出せてしまう。

一方で、彼はもっと厄介なケースも挙げます。パッケージを更新せず、既存コード内のバグを複数つないで exploit path を作るタイプです。こちらはずっと検出しにくい。つまり、防御側は「単純な悪意ある script」と「もっと深いコードレベルの連鎖 exploit」の両方を相手にしなければなりません。

### 大企業はオープンソースを捨て、自前再実装に向かうのか

後半では、もしこの手の攻撃が続けば、大企業はオープンソース利用自体を減らし、Claude などでモジュールを書き直して自前管理に向かうのではないか、という論点が出ます。Ismail は、この動きはすでに始まっている面があると見ています。安全性だけでなく性能改善のためにも、既存のユーティリティを自分たち向けに再実装するインセンティブは強まっています。

ただし彼は、それがモデル企業には好都合でも、オープンソースには破壊的だと見ています。みなが既存資産を共有する代わりに、トークンを消費して自前依存を作り、自前で検証し直す世界になるからです。そこで失われるのは、単なるコード共有ではなく、オープンソースという協調の土台そのものです。

### 解決は個人技ではなく、エコシステム全体の協力でしか成立しない

Ismail の主張で一貫しているのは、個々のメンテナの努力だけでは限界があるという点です。サプライチェーン全体で互いを守る仕組み、必要に応じて計算資源やチェック能力を補い合う仕組みが要る。オープンソースはみなが依存している土台なのだから、防御も個人戦ではなく集合戦でなければならない、という見方です。

その文脈で、彼は GitHub や NPM のようなレジストリ運営企業にも注文をつけます。彼らはオープンソースから大きな恩恵を受けてきたのだから、トークン配布のような表面的な支援ではなく、パイプラインや公開物の検査をもっと安全にする方向で責任を果たすべきだ、というわけです。

### 実践チェックリスト: 今すぐできることはかなり多い

最後の practical tips は非常に明快です。JavaScript 開発者なら PNPM を使うこと。依存更新まわりの設定を見直すこと。CodeQL、Dependabot、Socket など無料で使えるスキャナを全部有効にすること。さらに、単なるスキャンでは拾えない連鎖 exploit を探す adversarial agent も検討すること。ここまでは比較的まっすぐです。

そのうえで彼は、誰をコントリビュータとして受け入れるかも慎重に考えるべきだと言います。Vouch のような仕組みで信頼済みの貢献者を可視化する発想は、その一例です。ただしここでも、厳しくしすぎれば新しい貢献者の入口が閉じるという問題が残ります。動画の終盤では、まさにこのジレンマがそのまま言語化されています。守るための仕組みが、そのままオープンソースの開放性を削ってしまうのです。

### この動画の本質は、AI 時代の防御が「導入するかどうか」ではなく「常時運用できるかどうか」になったこと

全体を通して見ると、この動画は単に「サプライチェーン攻撃に気をつけよう」と言っているだけではありません。もっと重要なのは、AI によって攻撃コストが下がった結果、防御策も sporadic なチェックでは足りなくなった、という指摘です。メール確認、CI/CD 監査、依存更新ポリシー、スキャナ、攻撃者視点のテスト、コントリビュータ信頼管理。これらを個別の施策として持つだけではなく、日々の開発運用の中に埋め込めるかどうかが問われています。

そして、その現実に最も強い圧力を受けるのが、資金や人手に限りのあるオープンソースのメンテナです。だからこそ Ismail は、防御を個人の自己責任に押し戻すのではなく、エコシステム全体の課題として語っていました。技術的な防御の話であると同時に、オープンソースをこの先どう維持するかという話にもなっている点が、この動画のいちばん重いところです。
