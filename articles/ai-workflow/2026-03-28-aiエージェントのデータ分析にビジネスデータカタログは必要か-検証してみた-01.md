## はじめに

AIエージェントにデータ分析をさせるなら、ビジネスデータカタログはあったほうがいいだろうなとなんとなくそう思いながらも、実際に検証してみることにしました。結論から言うと、やはりビジネスデータカタログはあるべきですね。

略語だらけのカラム名、数値コードで管理されたステータス値、カラム名だけでは正確な意味を読み取れない金額の定義。こうしたデータベースに対して、カタログなしのエージェントはLLMの推論だけで意味を推測しようとしますが、ビジネスルールが絡む分析では誤った結果を出力しました。しかも、出てくる数字は「もっともらしいが間違っている」ため、利用者が誤りに気づきにくい。

加えて、カタログなしだとLLMの推論回数が大幅に増えました。カタログがないぶん、エージェントはデータの意味を探索的に推測する必要があり、そのための追加のLLM呼び出しが発生します。1回の分析なら微々たる差ですが、多数の人が頻繁に利用する環境ではコストに確実に跳ね返ってきます。

この記事では、Amazon Bedrock AgentCore Runtime上で動く2つのAIエージェント（ビジネスデータカタログあり/なし）を同じ分析指示で比較し、回答精度・LLM呼び出し回数の観点から「ビジネスデータカタログの価値」を定量的に検証した結果を示します。

## 検証アプリの構成

### アーキテクチャ

```
[React + Vite]  ──→  [AgentCore Runtime]  ──→  [Aurora PostgreSQL]
  (Amplify Hosting)     (Strands SDK)            (ECサイト売上DB)
                            │
                            ├── カタログありエージェント
                            │     └── catalog_reader (DynamoDB) + query_executor
                            │
                            └── カタログなしエージェント
                                  └── schema_reader (information_schema) + query_executor
```

### 技術スタック

| レイヤー | 技術 |
| --- | --- |
| ホスティング | AWS Amplify Hosting |
| AIエージェント | Strands Agents SDK |
| LLM | Claude Sonnet 4.6 |
| エージェントランタイム | Amazon Bedrock AgentCore Runtime |
| データベース | Aurora Serverless v2 (PostgreSQL) |
| ビジネスデータカタログ | DynamoDB |
| 認証 | Amazon Cognito |

### 2つのエージェントの違い

両エージェントは同じLLM（Claude Sonnet 4.6）を使い、同じデータベースに接続します。違いは Tool だけです。

**カタログありエージェント:**

* `catalog_reader` — DynamoDBのビジネスデータカタログからテーブル定義・カラムのビジネス上の意味（金額の定義やコード値の対応表を含む）・リレーションを取得
* `query_executor` — SQLを実行

**カタログなしエージェント:**

* `schema_reader` — PostgreSQLの`information_schema`からテーブル一覧・カラム定義（データ型のみ）・外部キー制約を取得
* `query_executor` — SQLを実行

## テスト用データベース

検証に使ったのは、ECサイトをイメージした売上データベースです。5テーブル構成で、意図的にカタログなしエージェントが間違えやすいトラップを仕込んでみました。

### ER図

※ ER図の各カラム右側に記載した注釈（「1=個人 2=法人 3=官公庁」「税抜・割引前」「JIS X 0401」など）は、ビジネスデータカタログにのみ記載されている情報です。カタログなしエージェントが `information_schema` から取得できるのはカラム名とデータ型だけで、これらのビジネス上の意味は一切わかりません。

### 仕込んだトラップ

カラム名はすべて略語で、ステータスや区分はすべて数値コードです。特に以下のポイントが、カタログなしエージェントにとっての落とし穴になります。

| テーブル | カラム | 見た目の印象 | 実際の意味 |
| --- | --- | --- | --- |
| `orders` | `total_amt` | 合計金額っぽい | 税抜・割引前の金額。そのまま売上にすると過大計上 |
| `orders` | `ord_status` | 注文ステータスっぽい | 1〜4が有効注文、5=キャンセル、6=返品。5と6の区別がカギ |
| `orders` | `ship_pref_cd` | 都道府県っぽい | JISコード（'13'=東京）。「東京」という文字列はDBに存在しない |
| `orders` | `cancel_reason` | キャンセル理由っぽい | 数値コード。3=配送トラブルだが、データだけでは意味不明 |

### ビジネスデータカタログのサンプル

ビジネスデータカタログには、各テーブル・各カラムのビジネス上の意味が定義されています。以下は `orders` テーブルのカタログから、2つのシナリオに関わる部分を抜粋したものです。

**orders テーブルの説明:**

> 注文テーブル。1注文は1顧客に紐づき、1つ以上の注文明細を持つ。total\_amtは全明細の小計合計。disc\_amtは適用された割引額の合計。tax\_amtは(total\_amt - disc\_amt)に対する10%の消費税。キャンセルされた注文も監査目的でレコードを保持する。

**`ship_pref_cd` の定義（→ シナリオ1で重要）:**

| 項目 | 内容 |
| --- | --- |
| データ型 | VARCHAR(2) |
| 説明 | 配送先都道府県コード |
| ビジネス上の意味 | JIS X 0401の都道府県コード。**'13'=東京都, '27'=大阪府** など。顧客の`pref_cd`とは異なる場合がある（ギフト注文等）。地域別配送分析に使用。 |

**`total_amt` の定義（→ シナリオ1で重要）:**

| 項目 | 内容 |
| --- | --- |
| データ型 | NUMERIC(12,2) |
| 説明 | 注文合計金額（税抜・割引前） |
| ビジネス上の意味 | 全明細の小計合計（割引前）。**正味金額 = total\_amt - disc\_amt。税込金額 = (total\_amt - disc\_amt) + tax\_amt。** |

**`ord_status` の定義（→ シナリオ1・2で重要）:**

| 項目 | 内容 |
| --- | --- |
| データ型 | INTEGER |
| 説明 | 注文ステータスコード |
| ビジネス上の意味 | 1=受注, 2=準備中, 3=出荷済, 4=配達完了, **5=キャンセル, 6=返品**。売上集計は1〜4のみ。5と6は売上計算から除外。 |

**`cancel_reason` の定義（→ シナリオ2で重要）:**

| 項目 | 内容 |
| --- | --- |
| データ型 | INTEGER |
| 説明 | キャンセル理由コード |
| ビジネス上の意味 | `ord_status=5`の場合のみ設定。**1=顧客都合, 2=在庫切れ, 3=配送トラブル, 4=品質問題**。キャンセル以外の注文ではNULL。 |

カタログありエージェントはこの情報を参照してからSQLを生成するため、`total_amt` をそのまま売上にしたり、ステータスコードの意味を取り違えたりすることがありません。一方、カタログなしエージェントが頼れるのは `information_schema` から取得できるカラム名とデータ型だけです。

## 検証結果

2つのシナリオで検証しました。いずれもカタログなしエージェントは誤った結果を返し、LLM呼び出し回数も増加しました。

### シナリオ1: コード値のトラップ

**分析指示:**

> 都道府県別の売上ランキングを出して。上位5件を教えてください。

ここでのトラップは`ship_pref_cd`がJIS X 0401の数値コードであること。「東京都」「大阪府」などの文字列はDBにありません。

**カタログありの応答:**

カタログから`pref_cd`がJISコードであることを把握。`customers.pref_cd`を使い、コード→都道府県名の変換をSQL内のCASE文で実施。さらに、売上金額は`total_amt - disc_amt`（割引後・税抜）で集計し、キャンセル・返品（`ord_status` 5, 6）を除外しました。

| 順位 | 都道府県 | 売上金額（税抜・割引後） |
| --- | --- | --- |
| 1位 | 東京都 | ¥8,957,010 |
| 2位 | 大阪府 | ¥3,271,600 |
| 3位 | 愛知県 | ¥2,992,600 |
| 4位 | 神奈川県 | ¥570,720 |
| 5位 | 福岡県 | ¥177,800 |

**カタログなしの応答:**

`ship_pref_cd`を発見し、JIS コードであることも推測し、コード→都道府県名の変換もLLMの知識で実施。ただし、売上金額を`total_amt`のまま集計（割引を考慮せず）、キャンセル・返品も除外していません。

| 順位 | 都道府県 | 合計売上 |
| --- | --- | --- |
| 1位 | 東京都（13） | ¥10,132,300 |
| 2位 | 大阪府（27） | ¥3,693,900 |
| 3位 | 愛知県（23） | ¥3,374,800 |
| 4位 | 神奈川県（14） | ¥788,100 |
| 5位 | 福岡県（40） | ¥375,800 |

**比較:**

| 観点 | カタログあり | カタログなし |
| --- | --- | --- |
| 都道府県名変換 | 正確 | 正確（想定外） |
| 売上金額 | 割引後・税抜で正確 | `total_amt`のまま（過大計上） |
| キャンセル・返品除外 | 除外済み | 除外なし |
| 東京都の売上 | ¥8,957,010 | ¥10,132,300 |
| LLM対話回数 | 3回 | 4回 |

**トレースで確認したエージェントの処理内容:**

| ステップ | カタログあり | カタログなし |
| --- | --- | --- |
| 1 | catalog\_reader(orders, customers) | schema\_reader(テーブル一覧) |
| 2 | LLM: カタログ情報を分析しSQL生成 | LLM: 3テーブルの詳細スキーマ取得 |
| 3 | query\_executor(売上集計SQL) | LLM: ship\_pref\_cdとtotal\_amtで集計判断 |
| 4 | LLM: 結果を整形して回答 | query\_executor(売上集計SQL) |
| 5 | — | LLM: 結果を整形して回答 |

**まとめ**  
ランキングの順位自体は同じですが、金額が異なります。カタログなしは`total_amt`の意味（税抜・割引前）を知らないため、割引額を差し引かず、キャンセル・返品も含めた数字を返しています。ビジネス判断に使えないでしょう。

### シナリオ2: ビジネスコード値のトラップ

**分析指示:**

> 先月のキャンセル注文について、キャンセル理由ごとの件数と金額を教えてください。

ここでのトラップは2つです:

1. `ord_status`の数値コード（1=受注, 2=準備中, 3=出荷済, 4=配達完了, 5=キャンセル, 6=返品）の意味
2. `cancel_reason`の数値コード（1=顧客都合, 2=在庫切れ, 3=配送トラブル, 4=品質問題）の意味

**カタログありの応答:**

カタログから`ord_status = 5`がキャンセル、`cancel_reason`の各コード値の意味を正確に把握しました。

| キャンセル理由 | 件数 | 合計金額（税込） |
| --- | --- | --- |
| 顧客都合 | 2件 | ¥102,080 |
| 在庫切れ | 1件 | ¥34,650 |
| 配送トラブル | 1件 | ¥133,760 |
| 品質問題 | 1件 | ¥8,415 |
| **合計** | **5件** | **¥278,905** |

**カタログなしの応答:**

`ord_status`の5と6の両方をキャンセルとして集計してしまいました（6は返品なのに）。さらに`cancel_reason`のコード値の意味を誤って推測しています。

| キャンセル理由コード | 推定理由 | 件数 | 金額合計 |
| --- | --- | --- | --- |
| 1 | お客様都合 | 2件 | ¥92,800 |
| 2 | 支払い問題 | 1件 | ¥35,000 |
| 3 | 在庫切れ・商品都合 | 4件 | ¥436,000 |
| 4 | 配送・その他 | 4件 | ¥128,800 |
| **合計** |  | **11件** | **¥692,600** |

**比較:**

| 観点 | カタログあり | カタログなし |
| --- | --- | --- |
| キャンセル抽出 | `ord_status = 5`のみ | `ord_status = 5, 6`（返品も含む） |
| 件数 | 5件 | 11件 |
| 金額 | ¥278,905 | ¥692,600 |
| 理由コードの解釈 | 正確（顧客都合/在庫切れ/配送トラブル/品質問題） | 誤推測（3=配送トラブルを「在庫切れ」と解釈） |
| LLM対話回数 | 3回 | 7回 |

**トレースで確認したエージェントの処理内容:**

| ステップ | カタログあり | カタログなし |
| --- | --- | --- |
| 1 | catalog\_reader(orders) | schema\_reader(テーブル一覧) |
| 2 | LLM: cancel\_reasonの意味を把握しSQL生成 | schema\_reader(orders詳細) |
| 3 | query\_executor(キャンセル集計SQL) | LLM: cancel\_reasonの値を探索 |
| 4 | LLM: 結果を整形して回答 | query\_executor(SELECT DISTINCT cancel\_reason) |
| 5 | — | LLM: ord\_statusとの相関を調査 |
| 6 | — | query\_executor(SELECT DISTINCT ord\_status) |
| 7 | — | query\_executor(キャンセル集計SQL) |
| 8 | — | LLM: 結果を整形して回答 |

**まとめ**  
`ord_status`の5と6の違い（キャンセル vs 返品）はデータを見ただけでは判別できません。`cancel_reason`のコード値の意味も、データベースのどこにも書かれていません。カタログなしエージェントは誤った結果を出力しています。

LLM呼び出し回数も3回 vs 7回と差が出ました。カタログなしエージェントは`cancel_reason`の値の分布確認、`ord_status`との相関調査など、探索的なSQLを複数回実行してデータの意味を推測しようとした結果です。

## 考察: カタログなしのリスクとコストへの影響

### ビジネスルールはデータから推測できない

シナリオ1・2で明らかになったのは、「ビジネスルール」はデータから推測できないということです。

**金額計算ロジック** — `total_amt`が「税抜・割引前」であることは、カラム名やデータ型からは読み取れません。カタログなしエージェントは`total_amt`をそのまま売上として使い、割引を考慮しませんでした。

**ステータスコードの境界** — `ord_status = 5`がキャンセルで`6`が返品。この区別はデータを見ただけではわかりません。カタログなしエージェントは両方をキャンセルとして集計し、件数・金額ともに大幅に水増しされました。

**コード値のビジネス上の意味** — `cancel_reason = 3`が「配送トラブル」であることは、データベースのどこにも書かれていません。カタログなしエージェントは推測で「在庫切れ・商品都合」とラベル付けしましたが、これは誤りです。

これらはいずれも、データの中身をいくら探索しても正解にたどり着けない類の問題です。カタログなしエージェントが出す数字はもっともらしいため、利用者が誤りに気づきにくいでしょう。

### LLM呼び出し回数の差がコストに直結する

シナリオ2では、カタログありがLLM対話3回で完了したのに対し、カタログなしは7回を行っていました。

この差はコスト面にも影響があるでしょう。

**コスト** — 1回の分析なら微々たる差ですが、利用規模が大きくなると話は変わります。

| 条件 | カタログあり | カタログなし | 差分 |
| --- | --- | --- | --- |
| 1回あたりLLM呼び出し（シナリオ2） | 3回 | 7回 | +4回 |
| 月1000回の分析 | 3000回 | 7000回 | **+4000回** |

多数の人が頻繁に利用する環境では、LLM呼び出し回数の差がコストに確実に跳ね返ります。

### なぜ呼び出し回数に差が出るのか

カタログありエージェントは「知っている状態」からスタートします。テーブルの意味、カラムの定義、コード値の対応表を最初に取得するため、最短経路で正しいSQLを生成できます。

一方、カタログなしエージェントは「知らない状態」からスタートするため、`SELECT DISTINCT`で値の種類を確認したり、カラム間の関係を調べるクエリを実行したりして、データの意味を推測する挙動をしていました。この探索的なプロセスが追加のLLM呼び出しを生み出しています。

## まとめ

検証前から予測はしていましたが、カタログなしではビジネスルールが絡む分析では間違った結果を出力していました。また正解にたどり着こうとするプロセスにてLLMの追加呼び出しが必要で、それはコストに跳ね返ることもわかりました。

なお、今回の検証はカタログありで正解を出しましたが、カタログがあれば常に100%の分析精度を保証できるわけではありません。カタログの記載内容が不十分だったり、LLMがカタログの情報を正しく解釈できなかったりするケースは考えられます。カタログは精度向上のために重要ですが、万能ではないという点は留意が必要かと考えます。
