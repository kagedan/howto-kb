# はじめに

ClaudeやCodexを使って開発していると、使用量や制限時間を意識する場面があります。

毎回ブラウザで使用量ページを開いて確認するのが面倒だったので、ClaudeとCodexの使用量をデスクトップ上に常時表示するアプリを作りました。

作ったものはこちらです。

GitHub：  
<https://github.com/ma0dev0/desktop-usage-meter>

Release：  
<https://github.com/ma0dev0/desktop-usage-meter/releases/latest>

この記事では、アプリ紹介そのものよりも、Electronで小さな常駐HUDアプリを作るときの設計と実装のポイントをまとめます。

扱う主な内容は次の3つです。

* Electronで常に最前面の小さなHUDウィンドウを作る
* 隠しウィンドウでログイン済みページを読み取り、テキストを解析する
* macOS / Windows向けに配布するときの注意点を整理する

# 作ったもの

Usage Meterは、ClaudeとCodexの使用量をデスクトップ上に表示するElectronアプリです。

![](https://static.zenn.studio/user-upload/4c4ce4c7f4c7-20260628.png)

主な機能は次の通りです。

* Claudeの現在セッション・週間制限の残り使用量を表示
* Codexの5時間制限・週間制限を表示
* 小さなHUDとしてデスクトップ上に表示
* 常に最前面に表示
* 透明度の変更
* macOSではメニューバー、Windowsではタスクトレイに常駐
* 自動更新
* macOS Apple Silicon / Windows向けにビルド

技術構成はかなりシンプルです。

* Electron
* JavaScript / CommonJS
* electron-builder
* HTML / CSS / JavaScript

# なぜElectronにしたか

最初はChrome拡張で作る案も考えました。

ただ、今回欲しかったのは「ブラウザの中に表示されるメーター」ではなく、「エディタやターミナルを見ているときにも見えるメーター」でした。

Chrome拡張だと、ブラウザ内では便利です。  
一方で、VS Codeやターミナルを開いているときには表示できません。

今回必要だった条件は次のようなものです。

* デスクトップ上に小さなウィンドウを出せる
* 常に最前面にできる
* 透明なHUD風UIにできる
* タスクトレイ / メニューバーに常駐できる
* ログイン済みページをアプリ内で開ける
* macOS / Windows向けに配布できる

この条件だと、Electronが一番素直でした。

# 全体構成

大まかな構成は次のようになっています。

```
.
├── main.js              Electronメインプロセス
├── preload.js           レンダラーへAPIを公開
├── src
│   ├── claudeParser.js  Claude使用量ページのテキスト解析
│   ├── codexParser.js   Codex使用量ページのテキスト解析
│   ├── providers.js     対象サービス定義
│   └── statusSummary.js 状態コピー用テキスト生成
├── renderer
│   ├── meter.html
│   ├── meter.js
│   ├── meter.css
│   └── usagePace.js
└── package.json
```

役割を分けると、こうです。

* main.js：ページ取得、トレイ、ウィンドウ管理、設定保存
* src/\*Parser.js：ページの innerText から使用量を解析
* renderer/\*：HUDの表示
* preload.js：メインプロセスとレンダラーの橋渡し

Electronではメインプロセスとレンダラープロセスが分かれます。  
初心者向けに言うと、メインプロセスは「アプリ本体の管理」、レンダラーは「画面表示」を担当します。

## HUDウィンドウを作る

メーター画面は、通常のアプリウィンドウではなくHUD風にしています。  
実装では BrowserWindow を次のような設定で作っています。

```
meterWin = new BrowserWindow({
  width: 320,
  height: 180,
  frame: false,
  transparent: true,
  backgroundColor: '#00000000',
  hasShadow: false,
  resizable: true,
  skipTaskbar: true,
  show: false,
  alwaysOnTop: prefs.alwaysOnTop,
  webPreferences: {
    preload: path.join(__dirname, 'preload.js'),
    sandbox: true
  }
});
```

ポイントは次のあたりです。

* frame: false でOS標準のタイトルバーを消す
* transparent: true で透明ウィンドウにする
* skipTaskbar: true で通常のアプリ一覧に出にくくする
* alwaysOnTop で常に最前面にする
* preload 経由で必要なAPIだけをレンダラーへ渡す

常に最前面の切り替えは次のようにしています。

```
meterWin.setAlwaysOnTop(Boolean(prefs.alwaysOnTop), 'screen-saver');
```

HUDの位置は、初回は画面右上に出すようにしています。  
移動後は meterBounds として保存し、次回起動時に同じ場所へ戻します。

## タスクトレイに常駐させる

Usage Meterは、ウィンドウを閉じてもアプリ自体は終了しない作りにしています。

```
meterWin.on('close', e => {
  if (!app.isQuitting) {
    e.preventDefault();
    meterWin.hide();
    prefs.meterVisible = false;
    saveState();
    updateTray();
  }
});
```

閉じる操作を「終了」ではなく「隠す」に変換しています。  
トレイ側は Tray と Menu で作っています。

```
tray = new Tray(img);
tray.setToolTip('Usage Meter');
tray.on('click', toggleMeter);
tray.setContextMenu(buildTrayMenu());
```

トレイメニューには、次のような項目を入れています。

* メーター表示 / 非表示
* 再取得
* 透明度
* 常に最前面
* 自動更新間隔
* 対象サービス
* Claude / Codexへのログイン
* 起動時に自動起動
* 終了

小さなユーティリティアプリでは、メイン画面を大きく作るより、トレイメニューに設定を集約した方が扱いやすいと感じました。

## 使用量ページを隠しウィンドウで読む

このアプリの中心は、ClaudeやCodexの使用量ページをアプリ内で開いて、そのページのテキストを読む部分です。  
内部APIを直接叩くのではなく、ログイン済みの画面を開いて document.body.innerText を取得しています。

```
const EXTRACT_JS =
  "(() => ({ url: location.href, title: document.title, bodyText: (document.body && document.body.innerText) || '' }))()";
```

実際の取得は、表示しない BrowserWindow で行います。

```
const win = new BrowserWindow({
  show: false,
  webPreferences: {
    partition: PARTITION,
    backgroundThrottling: false,
    sandbox: true
  }
});

await win.loadURL(provider.usageUrl);
const data = await win.webContents.executeJavaScript(EXTRACT_JS, true);
```

ここで重要なのが partition です。

```
const PARTITION = 'persist:usage';
```

Electronの persist: パーティションを使うことで、アプリ内ブラウザのセッションを保持できます。  
つまり、一度ClaudeやCodexにログインすると、次回以降もそのセッションを使ってページを開けます。  
ログインが切れている場合は、URLからログインページに飛ばされたかを判定しています。

```
function isLoginUrl(url) {
  return /\/(login|log-in|signin|sign-in|auth)\b|auth0\.com|accounts\.google|auth\.openai\.com|openai\.com\/auth/i.test(
    url || ''
  );
}
```

この判定に引っかかった場合は、メーター上で「ログインが必要です」と表示します。

## 解析ロジックはサービスごとに分ける

ClaudeとCodexでは、使用量ページの表示文言や構造が違います。  
そのため、対象サービスを providers.js で定義し、それぞれ別のパーサーを持たせています。

```
const providers = {
  claude: {
    id: 'claude',
    name: 'Claude',
    usageUrl: 'https://claude.ai/settings/usage',
    parse: bodyText => claude.parseUsage({ fullText: bodyText })
  },
  codex: {
    id: 'codex',
    name: 'Codex',
    usageUrl: 'https://chatgpt.com/codex/cloud/settings/analytics',
    parse: bodyText => codex.parseUsage({ fullText: bodyText })
  }
};
```

パーサー側では、ページ全体のテキストを行単位に分割し、見出しやパーセント表記を探しています。  
たとえばCodex側では、5時間制限と週間制限を次のような正規表現で探しています。

```
const SECTION_DEFS = [
  { key: 'fivehour', label: '5時間の使用制限', re: /5\s*時間|5[-\s]?hour|five[-\s]?hour/i },
  { key: 'weekly', label: '週間利用上限', re: /週間|weekly/i }
];
```

この作りにした理由は、外部サービスの画面は変わる可能性があるからです。  
DOM構造に強く依存すると、class名や要素構造の変更で壊れやすくなります。  
一方で、innerText と表示文言をもとにすると、見た目のDOM変更には比較的強くなります。  
もちろん、文言が変われば壊れます。  
ここはトレードオフです。

## 取得値はローカルに保存する

設定や取得結果は、Electronのユーザーデータ領域に保存しています。

```
function stateFile() {
  return path.join(app.getPath('userData'), 'state.json');
}
```

保存しているのは、主に次のような情報です。

* 透明度
* 常に最前面にするか
* 自動更新間隔
* 表示対象サービス
* メーター位置
* 前回の取得結果

外部サーバーに送る処理は入れていません。  
使用量ページのテキストを読むアプリなので、この点は明確にしておく必要があると思いました。

## レンダラーには必要なAPIだけを渡す

Electronでは、レンダラーからNode.js APIへ自由に触れる設計にすると危険になりやすいです。  
このアプリでは contextBridge を使って、必要な操作だけを window.api として公開しています。

```
contextBridge.exposeInMainWorld('api', {
  onUpdate: cb => ipcRenderer.on('update', (e, data) => cb(data)),
  getState: () => ipcRenderer.invoke('getState'),
  refresh: () => ipcRenderer.invoke('refresh'),
  openLogin: id => ipcRenderer.send('openLogin', id),
  hide: () => ipcRenderer.send('hideMeter')
});
```

レンダラー側は、このAPIを呼んで状態取得や再取得を行います。

```
window.api.onUpdate(render);
window.api.getState().then(render);
```

小さな個人開発アプリでも、メインプロセスとレンダラーの境界は意識した方が良いです。  
特にログイン済みページを扱うアプリでは、必要以上に権限を広げない設計が大事になります。

## 使用ペースも表示する

単に「残り何％」を表示するだけだと、今の使用ペースが速いのか遅いのか分かりにくいです。  
そこで、リセット時刻と現在時刻から、今の時点でどのくらい使っているのが目安かを計算しています。  
renderer/usagePace.js では、5時間枠を次のように扱っています。

```
const HOUR_MS = 60 * 60 * 1000;
const SESSION_MS = 5 * HOUR_MS;
```

リセットまでの残り時間から経過率を出し、現在の使用率と比較して「余裕」「標準」「速い」などを表示します。  
これは厳密な予測ではありません。  
ただ、画面の端に置くメーターとしては、細かい数値よりも「このペースだと速いかどうか」が分かる方が便利でした。

## ビルドと配布

配布には electron-builder を使っています。  
package.json では、macOS Apple Silicon向けとWindows向けにコマンドを分けています。

```
{
  "scripts": {
    "dist:mac": "npm run icons && CSC_IDENTITY_AUTO_DISCOVERY=false electron-builder --mac dmg --arm64",
    "dist:win": "npm run icons && electron-builder --win nsis --x64"
  }
}
```

現時点では、macOSはApple Silicon向けのdmg、WindowsはNSISインストーラとして配布しています。  
Release：  
<https://github.com/ma0dev0/desktop-usage-meter/releases/latest>

## 署名なし配布の注意点

個人開発のデスクトップアプリを配布するときに避けて通れないのが、署名や公証の問題です。  
このアプリは、現時点では署名や公証をした配布用アプリではありません。  
そのため、macOSやWindowsで初回起動時に警告が表示される場合があります。  
READMEやBOOTHの説明では、この点を明記しています。  
OSSとして公開しているので、利用者が不安な場合はソースコードやRelease内容を確認できるようにしています。

GitHub：  
<https://github.com/ma0dev0/desktop-usage-meter>

# 詰まったところ

## 外部サービスの画面変更に弱い

一番大きな制約は、対象サービスの画面表示に依存していることです。  
今回は内部APIを直接使わず、画面の innerText を読む方針にしました。  
この方針のメリットは、実装が比較的単純で、ユーザーのログイン済み画面をそのまま利用できることです。  
一方で、サービス側の文言やページ構成が変わると解析できなくなる可能性があります。  
そのため、パーサーはサービスごとに分け、テストしやすい純粋関数にしています。

## ログイン状態の扱い

ログイン済みでなければ使用量ページは見えません。  
そこで、ログイン用のウィンドウと取得用の隠しウィンドウで同じ persist: パーティションを使うようにしました。  
これにより、一度ログインすれば、隠しウィンドウ側でも同じセッションを使えます。

## HUDの邪魔にならなさ

常に最前面のウィンドウは便利ですが、邪魔にもなりやすいです。  
そのため、次のような調整を入れました。

* ウィンドウを小さくする
* 透明度を変更できるようにする
* 閉じると終了ではなく非表示にする
* 表示対象サービスを切り替えられるようにする
* 自動更新間隔を選べるようにする  
  常駐アプリは、機能を増やすより「邪魔にならない」ことが大事だと感じました。

# 注意点

このアプリは、ClaudeおよびOpenAI / Codexの公式アプリではありません。  
個人が作成した非公式ツールです。  
また、対象サービスの画面表示を読み取っているため、サービス側の変更で動かなくなる可能性があります。

セキュリティ面では、内部APIを直接叩いたり、外部サーバーにデータを送ったりはしていません。  
ただし、ログイン済みページをアプリ内ブラウザで開く設計なので、利用者に仕組みを説明し、ソースコードを確認できる状態にしておくことが大事だと思っています。
