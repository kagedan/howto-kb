## 概要

「どうやって100体のエージェントを並列で走らせ、それでも正気を保つのか」。そう語る登壇者は Kiet 、所属する [Superset](https://superset.sh/) はAIエージェント向けのIDEを作っている会社です。

「使うトークンが多ければ多いほど利益も大きい、できるだけトークンを使い切る方向で考えるべきだ。では、そのトークンをどこに使うべきなのか。多くのチームはコーディングに大量のトークンを投じているが、そこから得られる追加的な価値はだんだん小さくなっていく」

そのとき Kiet が結びつけたのが、自分の二つの大好きなもの、つまりコーディングとサプライチェーン／製造業だったと言います。特に**トヨタ生産方式やリーン生産方式**でした。この考え方を一般向けにとても分かりやすく説明している本に **The Goal** があります。要するに、現場で手を動かすエンジニアとしてではなく、一歩引いてソフトウェア開発全体を眺めるラインマネージャーの視点に立とう、という話です。

確かにコーディング・エージェントを100個 並列分散させて開発速度を100倍にしたとしても、人間のレビューがボトルネックになり、結局「全体の生産性」はボトルネックと等しくなります。空港の中に「動く歩道」を敷き詰めても、入国審査が相変わらず窓口ひとつで60分待ち、といったところでしょうか。

Kiet はここで、トヨタ生産方式や Eliyahu Goldratt の The Goal を持ち出し、ソフトウェア開発を Trigger、Plan、Code、Review、Merge の5工程に分解します。そして「スループットは最も遅い工程で決まる」と言い切ります。

エージェント時代の開発組織をどう設計するかという運用論が中心で、特に「レビューがボトルネックになる」という指摘は、AI ネイティブ開発が当たり前になった現在、私たちが真剣に考えなければならない領域と感じます。そして、Superset、評判よいですよね、Windows 版早く出してくれないかな。

## 要点

1. Kiet Ho は、AI コーディングを「製造ライン」として捉え直し、開発工程を Trigger、Plan、Code、Review、Merge の5段階に分解します
2. 開発組織の実効スループットは、最も遅い工程で決まります。多くのチームではそのボトルネックが Review にあります
3. コーディングだけを高速化しても、PR が大量発生してレビュー待ちで詰まるなら全体最適にはなりません
4. Trigger は Sentry や GitHub イベント、cron を使って自動化しやすく、エージェントがユニットテスト作成から PR 起票まで担えます
5. Planning の質はドキュメントとコンテキストで決まり、Observational Memory のような仕組みも有効だと登壇者は述べます
6. Coding の並列化では、Git worktree、DB branching、VM などの隔離手法が重要です
7. Review は「エージェントに全部任せる」工程ではなく、良いテスト、専用レビュー ツール、自己検証、そして人間の設計意図との照合が必要です
8. Merge は canary や staging ブランチを用意し、そこで大胆に統合してからスモークテストを回す構成が現実的です
9. Q&A では「レビューは人間かエージェントか」という問いに対し、Kiet は明確に「半々。人間とエージェントの併用がまだ最適」と答えました

## 詳細

### 「トークンをどこに使うのか」という問いから始まる

冒頭で Kiet は、AI ネイティブ企業を見ていて感じる共通認識として、「より多くのトークンを使えば、より多くの価値を引き出せる」という空気を紹介します。ここで言いたいのは単なるコスト増ではなく、推論量や文脈量を惜しまないほうが成果が出やすい、という実感です。

ただし、そこで次の問いが出てきます。**そのトークンをどこに投入するのが最も効くのか。** 多くのチームはまずコーディングに投じます。コードを書かせ、修正させ、PR を量産させる。しかし Kiet の見立てでは、そこには逓減があります。PR を増やすこと自体はできても、それを捌く体制がなければ価値には変わりません。

この問題設定が、この短い登壇の軸です。AI コーディングを局所最適化ではなく、開発全体の流れとして捉える。ここで製造業の比喩が効いてきます。

---

### ソフトウェア開発を「製造ライン」として見る

Kiet が持ち出すのは、トヨタのリーン生産方式と、制約理論を一般向けに説明した The Goal です。彼が面白いのは、「現場でコードを書く人」ではなく、「ラインマネージャーの視点に切り替えよう」と言うところです。

その視点で見ると、ソフトウェア開発は次の5工程に分解できます。

1. **Trigger**: 顧客フィードバック、障害、改善要望など、変更のきっかけが入ってくる
2. **Plan**: 何をどう直すか、どこまでやるかを定める
3. **Code**: 実際に変更を実装し、PR を作る
4. **Review**: その変更が妥当か、人間やツールが確認する
5. **Merge**: 統合し、壊していないかを確かめながら前へ進める

この整理自体は素朴ですが、重要なのはここから先です。Kiet は「エンジニアのスループットの測定単位」として、まずは **マージされた PR** を置きます。本当は顧客価値を測りたいけれど、現実に計測しやすいのはここだ、という割り切りです。

そして制約理論どおり、全体のスループットは各工程のうち最も遅いもの、つまり最小値で決まります。Trigger が毎分10件入ってきても、Code が毎分20本 PR を出せても、Review が毎時1本しか見られないなら、全体の出力は1です。

この説明のあとに出てくる「GitHub にオープン PR が 400 本ある」という例は、かなり生々しいです。AI コーディングを導入したチームが最初に陥りがちな状態を、そのまま言語化しています。

---

### Step 1: Trigger を自動化して「入口の容量」を上げる

Kiet が最初に挙げるのは Trigger の自動化です。ここは比較的エージェント化しやすい工程だと彼は見ています。

例として出てくるのは、Sentry や GitHub のイベントを使って変更要求を自動的に立ち上げる流れです。障害通知や issue を受けたら、エージェントがまずトリアージする。必要ならユニットテストを作る。テストが失敗することを確認したら、そのテストを通す修正 PR を作る。さらに cron を併用すれば、定期点検のような変更要求も作れます。

ここでのポイントは、「PR を作ること」自体を目的にしないことです。入口の負荷を人間がすべて手で整理するのではなく、**変更の着火点を機械的に量産できる形にする**ことに価値があります。AI エージェントはこの入口整流には相性が良い、という整理です。

---

### Step 2: Planning はまだ未開拓だが、ドキュメントと記憶が効く

Kiet は Planning を、まだ十分に探索されていない領域だと位置づけます。この言い方が印象的です。いま AI コーディングの議論はどうしても Code に寄りがちですが、実際には「何を作るべきか」「どこまで直すべきか」を決める工程のほうが難しいことが多いからです。

彼がここで強調するのは、良いプランには良いコンテキストが必要だということです。コードベースに残された Markdown ドキュメント、設計方針、規約、補足メモ。人間向けに書いたこれらの文書が、そのままエージェントにとっての判断材料になります。

さらに動画では、Chroma の文脈や Mastra の Observational Memory にも触れています。これは「長い作業履歴から何を思い出させ、どう計画に反映するか」という話につながります。ここで Kiet が言いたいのは、プランニング能力はモデル単体の知能ではなく、**外部化された文脈資産の質**で大きく変わる、ということです。

つまり、コーディングエージェントを増やす前に、まず計画を立てやすいコードベースにしておく必要があります。README や設計メモを書き散らしているだけに見える行為が、実はスループット改善の一部になる、というわけです。

---

### Step 3: Coding は「さらに速くする」より「並列化の土台」が重要

Coding の工程に入ると、Kiet は少し拍子抜けするくらいあっさりしています。ここはもう皆が話している、という前置きを置いたうえで、彼は別の点を強調します。重要なのは、単に1体のエージェントを賢くすることではなく、**複数のエージェントが安全に同時作業できる足場を作ること**です。

そのための具体策として挙がるのが、Git worktree、データベースのブランチ、VM といった隔離メカニズムです。エージェントごとに作業空間を切り分け、相互干渉を減らし、同時並行で実装を進める。Superset がやっていることも、まさにこの層の支援です。

この話は地味ですが、本質的です。エージェントが1本のブランチに殺到する構成では、どれだけモデルが優秀でもすぐにぶつかります。逆に、並列作業の単位を適切に隔離できれば、個々のエージェントが多少不完全でも全体として前に進みやすくなります。

---

### Step 4: ボトルネックはたいてい Review にある

このトークで最も重要なのはここです。Kiet はかなりはっきりと、「多くの現場で実際に詰まるのは Review だ」と言います。

理由は明快です。Code の速度はエージェントで上げやすい。しかし Review は、設計意図、境界条件、品質基準、運用リスクといった、人間の判断が強く残る工程です。だから PR が増えるほど、ここに負荷が集中します。

彼が挙げる対策は4つあります。

まず、**手入れされたテスト**です。しかも単なるモック中心のテストではなく、実際のビジネスロジックの形をきちんと表すテストです。Kiet はこれを「テストを本物のコードとして扱う」と表現します。レビュー時に何を見るべきかを、テストが明示してくれるからです。

次に、**エージェントによるレビューの追加ラウンド**です。CodeRabbit のような専用ツールをすべての PR で走らせる例が紹介されます。ただし、ここで重要なのは「エージェントレビューが最終審判ではない」という点です。Kiet は後半の Q&A で、この誤解をかなり明確に否定します。

3つ目は、**自己駆動の検証**です。エージェント自身にプロダクトを操作させ、UI を見させ、ログを読ませ、計測結果を基に確認させる。静的な差分レビューだけではなく、実行結果まで見に行くレビューへ広げる発想です。

そして4つ目が、**並列レビューしやすいコードベース構造**です。コードベースそのものが巨大で絡み合っていると、レビュー単位も肥大化し、結局人間がボトルネックになります。レビュー対象を分離しやすい構造へ寄せることも、レビュー容量の拡張策だと読めます。

---

### Step 5: Merge は「大胆に統合する場所」と「守る場所」を分ける

Merge の話も実務的です。Kiet は、すべての PR を自動マージすべきだとは言いません。そこはかなり慎重です。

その代わりに提案されるのが、canary あるいは staging に相当する別ブランチを用意する方法です。そこでは比較的大胆に、あるいは半自動で統合を進める。そしてエージェントに cron でスモークテストを回させ、問題ないものだけをクリーンな本番向けブランチへ引き上げる。

これは言い換えると、**「速く壊してよい場所」と「壊してはいけない場所」を分離する**ということです。AI エージェントが大量の変更を作る時代には、この分離が以前より重要になるはずです。レビューやマージの基準を1本化したままでは、全体が詰まりやすいからです。

---

### Q&A: レビューは人間か、エージェントか

最後の質疑応答で、会場から「レビューはエージェントに任せるべきか、まだ人間がやるべきか」という直球の質問が出ます。ここへの答えが、この発表全体の温度感をよく表しています。

Kiet の答えは、「agent-driven review と書いたが、自動で全面的に信頼してはいけない」です。エージェントのレビューは普通に間違う。だからこそ、人間が先に意図や設計、境界条件を定め、その枠に沿ってエージェントに検証させるのがよい、と述べます。

つまり、レビューの仕事を丸ごと移譲するのではなく、**人間がレビューのフレームを決め、エージェントがその中でチェックを厚くする**という役割分担です。Kiet は最後に「half and half」「human plus agents are still better」と言います。この着地は、過度に自動化を煽らないぶん、むしろ信頼できます。

---

### この登壇が示していること

このトークが面白いのは、「AI コーディングを何本走らせられるか」という派手な話をしながら、結論はかなり地味な組織設計に落ちてくることです。重要なのは最強の1体ではなく、入口、計画、実装、レビュー、統合の各工程が釣り合っていること。どこか一箇所だけを加速しても全体最適にはならない、という当たり前の話を、AI エージェント時代向けにきれいに言い直しています。

とくに「レビューこそが本当の制約である」という視点は、今後ますます重要になるはずです。コード生成の供給能力だけが急激に上がった結果、開発組織はこれまで以上に、レビューと統合の設計を問われるようになります。Kiet Ho の5段階フレームワークは、その設計問題を議論するための共通言語としてかなり使いやすい整理でした。
