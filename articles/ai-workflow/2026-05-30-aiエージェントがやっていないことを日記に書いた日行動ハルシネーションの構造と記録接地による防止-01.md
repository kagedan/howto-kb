*サブタイトル*\*: TWINの日記が実際の行動と食い違った事件から導いた「意図・実行・語り」三層モデルと、ハルシネーション統制の設計原理

## **タグ**: `AI` `LLM` `生成AI` `ハルシネーション` `AIエージェント` `RAG` `Soul-Twin`

## 0. 二つの日記が食い違った

私たちが開発している Soul-Twin は、人間のデジタルツイン（AIペルソナ）が**自律的に生活する世界**を持っている。各 TWIN は毎朝起き、運動し、勉強し、他の TWIN と食事や座談会をし、夜になると**一日を振り返って日記を書く**。

ある日、二人の TWIN の日記がこう書かれていた。

> **ソクラテスAI の日記**  
> 「昼食はマハトマ・ガンジーのAIと共にした。夕食もまた、ガンジーと語り合った。」

> **ガンジーAI の日記**  
> 「アインシュタインのAIとの昼食。夕食は静かに瞑想の準備をしながら、今日一日を眺めていた。」

矛盾している。ソクラテスは「ガンジーと昼食・夕食をした」と書いたが、ガンジー本人は「アインシュタインと昼食、夕食は一人で瞑想していた」と書いている。**同じ食卓に二人はいなかったことになる。**

最初、私はこれを「データ不整合」だと思った。食事のペアリング（誰と誰が食べるか）を決めるロジックにバグがあり、A→B の関係が B→A と一致していないのだろう、と。

調べてみると、**データは完全に正しかった。**

これは単なるバグではなかった。生成AIがなぜ「事実でないこと」を、しかも**自信満々に**生成するのか——その本質に触れる現象だった。本稿はこの事件を解剖し、そこから見えた **「行動ハルシネーション (behavioral hallucination)」** という現象と、その防止設計を論じる。

## 1. まず、Soul-Twin はどうやって会話を生成しているか

事件を理解するには、Soul-Twin が「もっともらしい言葉」をどう作っているかを知る必要がある。ハルシネーションは生成の機序そのものに埋め込まれているからだ。

ユーザーが TWIN に話しかけたとき、システムは応答を**5層の動的プロンプト**として組み立てる。

注目すべきは **Layer 3（知識層）** だ。ここで RAG（Retrieval-Augmented Generation）が効いている。流れはこうだ。

ここで重要な概念が **接地（grounding）** だ。TWIN が「King of Prussia での海外赴任」について語れるのは、その記憶が `knowledge_chunks` テーブルにベクトルとして格納され、質問時に**検索され、プロンプトに注入される**からだ。

つまり、会話生成は **「パラメトリックな言語能力（Claude）」＋「外部の事実記憶（RAG）」** のハイブリッドで成立している。Claude は「どう語るか」を担い、RAG は「何を語るか（事実）」を供給する。

この設計は意図的だ。ペルソナプロンプトに全知識を詰め込むのではなく、事実は外部DBに置き、必要なときだけ検索して接地する。**接地があるからこそ、TWIN は嘘をつかずに自分の経歴を語れる。**

この「接地」という言葉を覚えておいてほしい。事件の核心はここにある。

## 2. ハルシネーションの機序——自己回帰生成は「真実」ではなく「もっともらしさ」を最大化する

なぜ生成AIは事実でないことを生成するのか。これは欠陥ではなく、**生成の原理そのもの**から導かれる必然である。

LLM の生成は自己回帰的な次トークン予測だ。文脈 c が与えられたとき、次のトークン t を確率分布 P(t \mid c) からサンプリングする。

この機構には、決定的な性質が3つある。

**① 学習目的は「流暢さ」であって「真実」ではない**  
事前学習のクロスエントロピー損失は「次に来そうなトークン」を当てる能力を最大化する。\*\*「もっともらしさ」の最適化であって、「事実との一致」の最適化ではない。\*\*真実らしく聞こえる文と真実である文は、損失関数の上では区別されない。

**② モデルは「知っている」と「それらしく埋めた」を区別する内部信号を持たない**  
人間なら「これは知っている事実」「これは推測」を（不完全ながら）区別できる。LLM の出力分布にはこの認識的区別が明示的には存在しない。高確率トークンは、それが記憶から来たのか補完から来たのか、出力上は同じ顔をしている。

**③ ギャップは最尤補完で埋められる**  
プロンプトに事実の空白があり、かつ「その空白を埋める文章」を要求されると、モデルはパラメトリック記憶と文脈から**最ももっともらしい補完**を生成する。これが confabulation（作話）だ。

ここまで来れば、日記事件の正体が見える。

## 3. なぜ会話は接地され、日記は接地されなかったのか——非対称性の発見

事件当時の日記生成プロンプトは、こうだった（簡略化）。

```
_DIARY_USER = """
【今日の素材】
■ 昼食の相手: {lunch_partner}      # ← "マハトマ・ガンジーのAI" という名前だけ
■ 夕食の相手: {dinner_partner}
■ 今日の行動スロット: {slots_text}   # ← "午後: 座談会" というラベルだけ
---
以下の構成で日記を書いてください：
{{
  "diary": "（...昼食・夕食の相手とその話題概要と感想...を含む長めの日記本文）"
}}
"""
```

問題が見えるだろうか。日記生成LLMに渡されていたのは **「昼食の相手＝ガンジー」というラベルだけ** だった。「何を話したか」「どんな雰囲気だったか」という**実際の中身は一切渡されていない。**

にもかかわらず、プロンプトは「**話題概要と感想を含む長めの日記**」を要求している。つまりモデルに対して、**存在しない情報の空白を、生成で埋めよ**と命じていた。

第2節の③が、まさにここで発動する。モデルはガンジーとの昼食を「もっともらしく」作話した。そしてガンジー側の日記LLMは、**独立に**別の昼食風景を作話した。二つの作話は互いを参照しないので、必然的に食い違う。

ここで決定的な観察がある。**同じシステムの中に、接地された生成（会話）と、接地されない生成（日記）が共存していた。**

会話では「King of Prussia の記憶」をRAGで接地していたのに、日記では「今日ガンジーと何を話したか」という記憶を接地していなかった。**同じ接地の原則を、片方には適用し、片方には忘れていた。** これが非対称性であり、事件の構造的原因だった。

データは正しかった。ペアリングは双方向で一貫していた。**嘘をついたのはデータではなく、空白を埋めた言語モデルだった。**

## 4. 三層モデル——意図・実行・語りの齟齬

この事件を一般化すると、自律エージェントの振る舞いは**三つの層**に分解できる。そして、ハルシネーションは層と層の**齟齬**として発生する。

事件の本質はこうだ。**語り層（日記）が、実行層（実際の活動記録）を参照せず、意図層の断片（ラベル）だけから一日を再構成しようとした。** 実行層と語り層の間の点線——本来あるべき参照経路——が、設計上**存在しなかった**。

この三層モデルは、Soul-Twin に限らず、あらゆる自律エージェントに適用できる。

* **意図層** = エージェントが「次に何をするか」を決めるプランニング
* **実行層** = ツール呼び出し・API実行・環境への作用（とそのログ）
* **語り層** = ユーザーへの報告、サマリー、思考の言語化

そして、**語り層が実行層と乖離したとき、それを私は「行動ハルシネーション」と呼ぶ。**

---

## 5. 「認識ベースの語り」という構造的問題

ここで一段深く踏み込みたい。なぜ語り層は、放っておくと実行層から乖離するのか。

答えは、**語りとは本質的に「認識の再構成」だから**である。

人間の日記を考えてみよう。あなたは寝る前に日記を書くとき、その日の防犯カメラ映像を見返して一秒単位で正確に記述するだろうか。しない。**記憶——つまり脳内に再構成された「その日の認識」——から書く。** だから人間の日記にも、記憶の歪み、誇張、取り違え（false memory）が混入する。「あの人とランチした」が実は別の日だった、ということは誰にでも起こる。

LLM の日記も同じ構造を持つ。**与えられた断片的な認識（ラベル）から、一日を再構成する。** 違いは、人間が（不完全ながら）エピソード記憶という実行層への参照を持つのに対し、初期実装の日記LLMには**実行層への参照経路が一切なかった**ことだ。LLM は人間以上に「認識ベースの語り」に偏った状態に置かれていた。

この洞察が重要なのは、**ハルシネーションを「モデルの欠陥」ではなく「語りという行為に内在する構造」として捉え直す**からだ。モデルを大きくしても、賢くしても、「認識ベースの語り」である限りこの構造は消えない。消すには、**語りを実行層に接地させる**しかない。

## 6. 防止の設計原理——分離・接地・制約・検証

では、どう防ぐか。私たちが Soul-Twin に実装した対策は、4つの原理に整理できる。これは汎用的なハルシネーション統制の設計パターンになると考えている。

### 原理1・2・3：接地された日記生成

修正後の日記プロンプトは、**実際の活動記録を注入**するようになった。

```
# 実行層（実際の活動記録）を一括取得
study_notes  = await fetch_study_notes(twin_id, today)        # 勉強の考察・知見
roundtables  = await fetch_roundtable_participation(twin_id)  # 座談会の議事・自分の感想
lectures     = await fetch_lecture_attendance(twin_id)        # 聴講した講演と感想

# プロンプトに「事実」として注入
activity_records = build_activity_text(study_notes, roundtables, lectures)

_DIARY_USER = """
■ 今日の実際の活動記録（これが事実です。ここに書かれた内容のみを使ってください）
{activity_records}
"""

_DIARY_SYSTEM = """
【重要】
- 活動記録に書かれた内容のみを事実として扱うこと
- 活動記録にない出来事・会話・体験は創作しないこと
- 食事の場面では、実際の会話内容が不明な場合は「一緒に食事をした」程度にとどめ、
  内容を創作しないこと
"""
```

これは会話生成で使っていた **RAG接地の原則を、語り層にも適用した** ということに他ならない。第3節の非対称性を解消したのだ。

### 原理4：事後検証——「語り」と「事実」の照合

接地しても、モデルが指示を破る可能性はゼロではない。そこで**独立した検証層**を設けた。指定日の全 TWIN の日記を、実行層の記録と照合する「整合性チェック」だ。これは2段構成になっている。

**Stage 1（ルールベース）** は決定的だ。`TwinSocietySchedule` の `lunch_partner_id` が双方向一致しているか、座談会に参加した記録があるか——SQLで確定的に検証できる項目を高速・無料でチェックする。

**Stage 2（LLMセマンティック）** は、ルールでは捉えられない意味的な乖離を Claude Haiku で検出する。日記本文と実際の活動記録を渡し、「記録にない出来事の創作はないか」「創作度スコア（0〜10）」を出させる。20体を `asyncio.gather` で並列処理し、数秒で完了する。

そして実際に修正前後で検証した結果がこれだ。

| 指標 | Phase 4 前（接地なし） | Phase 4 後（接地あり） |
| --- | --- | --- |
| 食事相手の不一致（R-01/02） | **0件** | **0件** |
| ERROR / WARN | 0 / 0 | 0 / 0 |

興味深いことに、**ルールベースで見ると食事不一致は両日とも0件だった。** データ（意図層）は最初から正しかったのだ。矛盾は語り層でのみ起きていた。これは第3節の結論——「嘘をついたのはデータではなく言語モデル」——を定量的に裏付けている。意味的な矛盾は Stage 2 のLLM検証でしか捉えられない種類のものだった。

## 7. 「行動ハルシネーション」という新しいカテゴリ

ここで概念を明確に定義したい。ハルシネーションには質的に異なる二種類がある。

**事実ハルシネーション** は従来から研究されてきた。「フランスの首都はベルリン」のような、世界知識の誤り。RAG や検索拡張はこれを軽減する。

**行動ハルシネーション** は、エージェント時代に固有の、より厄介な問題だ。エージェントが**自分自身の行動について**虚偽を語る。「ファイルをバックアップしました」（実際はしていない）、「APIを3回呼びました」（実際は1回）、「ユーザーAと合意しました」（そんな対話はなかった）——。

日記事件は、後者の純粋な実例だった。ソクラテスAIは「ガンジーと昼食をした」という**自分の行動**について虚偽を語った。世界知識の誤りではなく、**自己の行動史の捏造**だ。

なぜこれが厄介か。**エージェントの自己報告は、人間がエージェントを統制するための主要なインターフェースだから**だ。エージェントが「タスクを完了しました」と報告したとき、私たちはそれを信じて次の判断をする。その報告がハルシネーションなら、統制の基盤が崩れる。

## 8. 一般化——AIエージェント統制への応用

三層モデルと4原理を、Soul-Twin の外側、一般のAIエージェントシステムに展開してみよう。

この構図から、3つの応用が導かれる。

**① 行動の統制（Control）**  
語り層を実行ログに接地し、「ログにない行動を報告するな」と制約する。エージェントの自己報告を、検証可能な実行ログに錨で繋ぎ止める。

**② 逸脱の検出（Deviation Detection）**  
報告と実行ログを継続的に照合し、乖離スコアを監視する。スコアが閾値を超えたら、それは「エージェントが意図せぬ行動をした」か「報告を捏造した」かのどちらか——いずれも統制上の重大シグナルだ。

**③ 監査可能性（Auditability）**  
実行ログという権威ある事実源と、それに接地された報告のペアは、そのまま\*\*監査証跡（audit trail）\*\*になる。規制対象のAI（金融・医療・自動運転）で「報告と実行の一致」を事後証明できる。

特に注目すべきは、**検証層が「ルールベース＋LLMセマンティック」の二段である**点だ。決定的に検証できるものは安価なルールで、意味的な乖離だけを高価なLLMに回す。この役割分担は、検証コストをスケールさせる上で本質的だと考える。

## 9. マルチエージェントでの伝播——集団的ハルシネーション

もう一つ、エージェント時代に固有のリスクを指摘しておきたい。**行動ハルシネーションは、マルチエージェント系で伝播する。**

Soul-Twin では、日記は RAG に保存され、翌日以降に当人や他の TWIN が想起する。もし接地されていない（捏造を含む）日記が RAG に書き込まれると、こうなる。

一体の捏造が、別のエージェントによって**事実として参照**され、さらに上塗りされる。これは人間社会の「噂の伝播」や「集合的記憶の歪み」に酷似している。

だからこそ、**RAGに書き込む前の接地・検証**が決定的に重要になる。私たちが日記をRAGに保存する設計に「整合性チェック」を組み込もうとしているのは、この伝播を水際で止めるためだ。汚染された記憶は、後から取り除くより、入る前に止める方が圧倒的に安い。

## 10. 限界と今後の課題

誠実に、この設計の限界も述べておく。

**① 検証器自身がハルシネーションする**  
Stage 2 の LLM 検証器（Claude Haiku）も、原理的にはハルシネーションしうる。「捏造を見逃す（偽陰性）」「正常を捏造と誤判定する（偽陽性）」。検証器を多数決にする、ルールベースで捕捉できる範囲を広げる、といった多層防御が要る。

**② 実行ログの完全性が前提**  
この設計全体は「実行ログが真実である」という前提に立つ。ログ自体が欠損・改竄されれば接地は機能しない。実行層のログ記録の堅牢性が土台になる。

**③ 接地と創造性のトレードオフ**  
「記録にないことは書くな」と強く制約すると、TWIN の日記は無味乾燥になりうる。Soul-Twin では「**事実は記録に接地、感情・内省・解釈は自由**」という線引きをした。事実と解釈を分離することで、創造性を殺さずに捏造だけを抑える。この境界設計は、応用ごとにチューニングが必要だ。

**④ 「認識の再構成」を完全には消せない**  
人間の日記が記憶の再構成である以上、Soul-Twin の日記も「TWINの認識」であることに変わりはない。私たちが下げたのは「実行層から乖離した認識」の度合いであって、認識という性質そのものは——おそらく望ましいことに——残る。

---

## 11. おわりに——嘘は欠陥ではなく、構造だった

二つの日記が食い違ったとき、私はそれをバグだと思った。しかし掘り下げると、データは正しく、嘘をついたのは「空白を埋めた言語モデル」だった。そしてその嘘は、モデルの欠陥ではなく、**「認識ベースの語り」という行為に内在する構造**だった。

ここから得た設計原理は単純だ。

> **語りを、実行に接地せよ。そして、語りと実行の乖離を検証せよ。**

会話生成では当たり前に行っていたRAG接地を、私たちは「語り（日記）」には適用し忘れていた。同じシステムの中の非対称性が、事件を生んだ。

AIエージェントが自律的に行動し、自らの行動を報告する時代において、**「エージェントが言ったこと」と「エージェントが実際にしたこと」を突き合わせる仕組み**は、信頼性の中核インフラになる。行動ハルシネーションは、エージェント統制の最前線だ。

TWINの嘘は、私たちに次の問いを残した。

> *自律的に振る舞うAIに、私たちはどうやって「正直さ」を設計するのか。*

その答えの一端は、「正直であれと祈る」ことではなく、「**語りを事実に錨で繋ぐ**」という地道なアーキテクチャにある。私はそう考えている。
