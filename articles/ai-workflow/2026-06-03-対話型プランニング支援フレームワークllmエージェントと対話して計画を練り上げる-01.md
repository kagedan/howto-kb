**🎯 結論**  
LLMは複雑な推論（プランニングそのもの）は苦手ですが、「人間の曖昧な要求を形式言語に翻訳する」「プランナーが出した論理的な結果を人間にわかりやすく説明する」というインターフェース役としては圧倒的に優秀です。この論文は、LLMを適材適所でエージェント化し、従来の記号的プランナーと組み合わせることで、実用的な対話型AIシステムを作れることを実証しています！

### 概要

自動プランニングにおいて、ユーザーが自然言語で条件を調整しながら計画を練り上げられるようにするLLMマルチエージェントフレームワークの提案。目標間の競合を解決するユーザーテストにおいて、従来のテンプレートベースUIよりも高いユーザー満足度とタスク達成度を示しました 。

### 1. なぜこのテーマを選んだのか

最近、LangGraphやAutoGenなどのフレームワークの発展により、マルチエージェントシステムが流行しています。しかし、LLMだけで複雑なロジックや制約を満たす計画（Planning）を解かせようとすると、ハルシネーションを起こしたり、制約を破ったりして破綻することが多いのが現状です 。

僕は「LLMの『言語理解力』と、従来のソルバーやプランナーの『厳密な推論力』をどう組み合わせるか？」というテーマにとても興味があります。この論文は、まさにその課題に対して「LLMを計算エンジンではなく、ユーザーとプランナーの間の翻訳・説明エージェントとして配置する」というアプローチをとっており、実際のプロダクトやPoCにも直結する非常に実用的なアーキテクチャだと感じたため、今回ピックアップしました！

### 3. 論文の詳報（アーキテクチャと数式）

ここからは、論文の中身をガッツリ解説していきます！

#### 🗺️ 対話型プランニングの定式化

この論文では、プランニングを「最初から完璧な条件を投げて答えをもらう」ものではなく、\*\*「ユーザーが徐々に条件を追加・変更していく反復ステップ（Iteration Step）」\*\*として定義しています 。

各イテレーションステップ \delta\_i は次のように定義されます ：

\delta = \langle G^{ref}, G^{enf}, \pi \rangle

* G^{ref}：システムが知っているすべての目標（Reference goals）
* G^{enf}：現在のステップでユーザーが「絶対に満たしてほしい」と要求した目標の集合（Enforced goals, G^{enf} \subseteq G^{ref}）
* \pi：G^{enf} を満たす計画（プラン）。もし G^{enf} の組み合わせが矛盾していて解けない場合は空（\epsilon）になります 。

例えば、「買い物に行く(S)」「夕食を作る(C)」「子供の試合に行く(M)」という目標があったとします 。夕食と試合の時間が被っている場合、これらをすべて G^{enf} に入れると「解なし」になります 。

このとき、ユーザーは「なんでこの計画は解けないの？」や「どうすれば夕食を作る時間を確保できる？」といった質問をシステムに投げます 。

#### 🤖 マルチエージェントアーキテクチャ

ユーザーの自然言語を処理し、プランナーの論理的な結果を翻訳するために、システムは複数のLLMエージェントで構成されています 。論文内で示されている通信プロトコルをMermaidで表現してみました！

各エージェントの役割は以下の通りです。

1. **Question Translator (T\_Q)** : ユーザーの質問を分類します。例えば「なんで夕食を作れないの？」という質問を、形式的な表現 `(S-why-not, {C})` に変換します 。
2. **Goal Translator (T\_G)** : ユーザーが新しい目標を言った場合、「子供を買い物に連れて行かない」といった自然言語を、LTLf（線形時相論理）のフォーマット `□¬at(kid, store)` に翻訳し、プランナーが理解できるようにします 。
3. **Explanation Translator (T\_E)** : バックエンドのExplanation Framework（EF\_{CC}）が計算した論理的な結果（「目標CとMが競合しています」など）を受け取り、「夕食を作るには、子供の試合を諦める必要がありますよ」と人間にわかりやすく翻訳して返します 。
4. **Question Suggester (S\_Q)** : ユーザーが「何を質問していいかわからない」状態を防ぐため、現在の状況（解なしか、どの目標が満たされていないか）を判断し、「こんな質問ができますよ」と候補を3つほど提示します 。

### 4. 実装・検証・制作

この論文の面白いところは、LLMエージェントたちに渡す具体的なプロンプト構造がしっかり定義されている点です 。実際に自社で同じようなシステムを組む際のヒントとして、アーキテクチャをどう実装するか考えてみましょう。

#### プロンプトの工夫：コンテキストの分離

LLMをエージェントとして動かす際、過去の会話履歴を全部突っ込むと、LLMが混乱してハルシネーションを起こしやすくなります。この論文では、**イテレーションステップ（計画のバージョン）ごとにLLMのコンテキストメモリを分離**しています 。  
ステップ1での会話と、条件を変えたステップ2での会話を混ざらないようにすることで、正確な応答を維持しているわけですね 。

#### Question Translator の実装イメージ

実際のプロンプトでは、Structured Outputs（JSONスキーマ指定）を利用して出力を固定します 。

```
// LLMの出力スキーマ例
{
  "questionType": "S-WHY-NOT", // 質問の分類
  "questionArgument": ["夕食を作る"], // 対象となる目標
  "used": "ALREADY-USED", // 既存の目標か新規か
  "reverseTranslation": "なぜ目標「夕食を作る」は満たされないのですか？" 
}
```

このように、LLMには自然言語からJSONへのパースという得意な作業だけをさせ、実際の競合判定は裏側の最適化ソルバー（Fast Downwardなど）に投げます 。ソルバーが「MUS（Minimal Unsolvable Subset：最小の解けない組み合わせ）」や「MCS（Minimal Correction Set：諦めるべき目標の最小セット）」を計算して返してくれます 。

### 5. 筆者の考察

論文を読んでみて、特に技術的価値が高いと感じた点をまとめます。

#### 1. 「LLMに計画を立てさせない」という割り切り

最近のLLMは推論能力が上がっていますが、それでもスケジュール最適化や巡回セールスマン問題のような厳密な制約解決は苦手です 。本論文は「計算は専門のソルバー（Planner）に任せ、LLMはインターフェース（翻訳・説明）に徹する」という「Neuro-Symbolic（深層学習と記号推論の融合）」の王道アプローチをとっています 。エンタープライズ領域で生成AIを導入する際、最も安全で確実なパターンだと言えます。

#### 2. 説明（Explanation）によるユーザーの納得感

計画がうまくいかないとき、単に「エラーです」と言われるより、「AとBの予定が被っているからです。Bを外せばAを達成できますよ」と言われた方が、ユーザーは次にどうアクションすべきかすぐにわかりますよね 。  
ユーザーテストでも、テンプレートベースの決まりきった回答より、LLMが文脈に合わせて回答し、フォローアップの質問もできるLLMベースのインターフェースの方が、主観的な満足度や計画の改善度合いが有意に高かったという結果が出ています。  
「AIの判断結果をいかに人間に寄り添って説明するか（XAI）」の実装例として、非常に美しい設計です。

#### 3. 「質問を提案する（Question Suggester）」のUX的意義

一般ユーザーは、「システムにどう質問していいかわからない」という壁にぶつかりがちです 。システム側から「〇〇について聞いてみますか？」とボタンを提示する機能は、ユーザーの認知負荷を下げる上で極めて重要です 。実際のプロダクト開発でも、チャットUIの前に「サジェストチップス」を出す設計は必須級になっていますね。

### 6. まとめ

本日は、LLMを介して対話的に計画を練り上げるフレームワークの論文を解説しました！

* **要点1:** 計画（プランニング）は一発で決まるものではなく、人間とAIが条件を調整し合う反復的プロセスである 。
* **要点2:** LLMにすべてを解かせるのではなく、質問分類・翻訳・説明というエージェントに分割し、裏側の厳密なプランナーと連携させるアーキテクチャが有効である 。
* **要点3:** 対話的な説明（なぜダメなのか、どうすればいいのか）と質問のサジェストにより、ユーザーはより良い計画へ効率的にたどり着ける 。
