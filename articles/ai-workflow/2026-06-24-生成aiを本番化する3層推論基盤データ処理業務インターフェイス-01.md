2026年6月24日に公開されたOpenAIとAWSの発表から、生成AIを本番サービスとして提供するために必要な構成を整理します。

今回取り上げるのは、次の3つです。

| 層 | 事例 |
| --- | --- |
| 推論基盤 | OpenAIとBroadcomによるLLM推論向け専用チップ |
| データ処理 | AWSによる4億件以上の文書からの機密情報マスキング |
| 業務インターフェイス | Amazon Nova 2 Sonicによる医療予約音声エージェント |

## 生成AIサービスを構成する3つの層

生成AIサービスは、基盤モデルだけでは成立しません。

実際には、次のような複数の層が必要です。

| 層 | 主な役割 |
| --- | --- |
| チップ・推論基盤 | モデルを高速かつ効率的に実行する |
| AIモデル | 推論、生成、認識を行う |
| データ処理 | 入力データの移動、前処理、検索、保護を行う |
| 業務インターフェイス | 利用者との会話、認証、ツール実行を行う |
| 人間による運用 | 例外処理、承認、監査、改善を行う |

## LLM推論向けの専用チップ

OpenAIとBroadcomは、LLM推論向けの専用チップ「Jalapeño」を発表しました。

OpenAIのモデル、カーネル、推論システム、製品要件に関する知見をもとに、チップをLLM推論向けに設計しています。

主な特徴は次のとおりです。

* 現在および将来のLLMへ対応する設計
* ワット当たりの推論性能を重視
* 計算、メモリ、ネットワークのバランスを最適化
* データ移動を削減
* 複数世代にわたるプラットフォームとして展開
* ギガワット規模のデータセンター展開を想定

LLM推論では、演算性能だけを高めても十分ではありません。

モデルの重みやKVキャッシュを扱うメモリ、チップ間通信、実際の利用率、電力効率を合わせて設計する必要があります。

モデルの利用者が増えるほど、1回当たりの推論コストや電力効率の差が、サービス全体のコストへ影響します。

## 4億件の文書を処理するAWSパイプライン

Huntington Bankは、4億件以上の文書から個人情報やカード情報を検出し、マスキングするシステムをAWS上に構築しました。

当初は数年かかると見積もられた処理を、数か月まで短縮し、95%以上のマスキング精度を実現したとされています。

使用された主なサービスは次のとおりです。

| サービス | 役割 |
| --- | --- |
| AWS DataSync | オンプレミスとAWS間の文書転送 |
| AWS Direct Connect | 専用ネットワーク接続 |
| Amazon S3 | 文書と処理結果の保存 |
| AWS KMS | データ暗号化 |
| Amazon Textract | 文書からテキストや項目を抽出 |
| Amazon SageMaker | 機械学習処理 |
| AWS Step Functions | 処理フローの制御 |
| AWS Lambda | イベント処理や各処理の連携 |

この構成では、モデルの精度だけでなく、以下の要件が重要です。

* 保存時と通信時の暗号化
* 厳格なアクセス制御
* PCI DSS対象サービスの利用
* オンプレミスへの結果同期
* 大量データの並列処理
* 失敗時の再実行
* 処理状況の監視

大量データのAI処理では、1件当たりの成功率だけでなく、処理全体を完了できるワークフロー設計が必要です。

## 音声エージェントを業務へ接続する

AWSは、Amazon Nova 2 SonicとAmazon Bedrock AgentCoreを使った医療予約向け音声エージェントの構築方法を公開しました。

このエージェントは、次の処理を1つの会話内で行います。

* 患者の本人確認
* 予約内容の案内
* 予約の確定、キャンセル、変更
* 診察前情報の収集
* 人間スタッフへのエスカレーション

構成には、次のサービスや技術が使われています。

| 要素 | 役割 |
| --- | --- |
| Amazon Nova 2 Sonic | リアルタイムSpeech-to-Speech |
| Amazon Bedrock AgentCore | エージェント実行環境 |
| Strands Agents SDK | 医療予約向けツールの連携 |
| Amazon Cognito | 利用者認証 |
| Amazon DynamoDB | 予約や会話状態の保存 |
| Amazon SNS | スタッフへの通知 |
| WebSocket | 双方向の音声ストリーミング |

従来の音声AIでは、Speech-to-Text、LLM、Text-to-Speechを順番に実行します。

この方法では、各処理間で遅延が発生し、声のトーンやためらいなどの情報も失われる可能性があります。

Speech-to-Speechモデルでは、音声の特徴を保ちながらリアルタイムに応答できます。

ただし、業務利用では会話性能だけでなく、認証、データ更新、例外処理、人間への引き継ぎも必要です。

## 本番化で確認したいポイント

| 観点 | 確認事項 |
| --- | --- |
| 性能 | レイテンシーとスループットは要件を満たすか |
| コスト | 推論、データ転送、保存、処理コストを追跡できるか |
| データ | 入力データを安全に移動・保存・削除できるか |
| セキュリティ | 暗号化、認証、権限分離、監査ができているか |
| 可用性 | 大量処理やアクセス集中に耐えられるか |
| 例外処理 | AIが処理できない場合に人間へ引き継げるか |
| 観測性 | モデル、ワークフロー、業務処理を追跡できるか |

## まとめ

生成AIを本番サービスとして提供するには、モデル以外の設計が重要です。

専用チップによる推論効率、大量データを安全に扱う処理基盤、利用者と業務システムをつなぐインターフェイスを組み合わせる必要があります。

生成AI開発では、モデルの選定だけでなく、インフラ、データ、セキュリティ、業務フローまで含めて設計することが重要です。
