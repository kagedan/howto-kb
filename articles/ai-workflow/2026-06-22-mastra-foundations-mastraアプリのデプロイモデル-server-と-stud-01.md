## この記事が書かれた背景

Mastra は 2026年4月に **Mastra Platform** を正式発表し、エージェントの開発・実行基盤を**3つの独立プロダクト**に分解しました。

* **Mastra Server** ― エージェントランタイム（実行・API公開）
* **Mastra Studio** ― 可観測性（Observability）レイヤー（トレース・メトリクス・eval）
* **Memory Gateway** ― エージェントメモリの永続化サービス

この分離によって「Server はどこに置く？」「Studio は自社インフラに置くべき？それともクラウドに任せる？」という**デプロイメントモデルの選択**が現実的な設計課題になりました。

今回の元記事は、Mastra チームが顧客との対話を通じて蓄積した知見をもとに、代表的な3つのデプロイメントモデルを実例付きで整理したものです。本稿ではその内容を紹介しつつ、各モデルの技術的な背景・メリット・トレードオフをより詳しく掘り下げます。

## 前提知識：Mastra Server と Mastra Studio とは

デプロイメントモデルの議論に入る前に、Server と Studio がそれぞれ何をするのかを正確に把握しておきましょう。

### Mastra Server ― 「エージェントランタイム」

Mastra Server は、定義したエージェント・ワークフロー・ツールを**REST APIエンドポイント**として公開するランタイムです。具体的には以下を担当します。

| 役割 | 説明 |
| --- | --- |
| リクエストルーティング | エージェント・ワークフローごとにエンドポイントを自動生成 |
| ミドルウェア | 認証、ログ取得、エラーハンドリングなどの共通処理 |
| 認証 | API キーやトークンベースの認証を組み込み |
| リクエストコンテキスト | 各リクエストにセッション情報やユーザーコンテキストを注入 |
| ストリーミングレスポンス | LLM からの応答をリアルタイムにクライアントへ中継 |

開発者は `mastra build` でビルドし、`mastra start` で起動するだけで、エージェントが API サーバーとして動き始めます。OpenAPI ドキュメントと Swagger UI も自動生成されるため、フロントエンドチームや外部サービスとの連携がスムーズです。

```
mastra build            # 自己完結型のビルドを生成
mastra start            # ローカルで起動
mastra server deploy    # Mastra Platform にデプロイ
```

セキュリティ面では、システムプロンプト、ツール定義、API キーがデフォルトでストリーミングレスポンスから除外される設計になっています。

### Mastra Studio ― 「エージェントの可観測性」

Mastra Studio は、エージェントの動作を**観測・検査・改善する**ための統合ツールです。

| 機能 | 説明 |
| --- | --- |
| エージェントとのチャット | Studio UI 上でエージェントに直接話しかけて動作確認 |
| ワークフロー可視化 | ワークフローの実行状況をステップごとにグラフィカルに表示 |
| トレース検査 | モデル呼び出し・ツール実行の全履歴を時系列で確認 |
| メトリクス | コスト・レイテンシ・エラー率をダッシュボードで監視 |
| Datasets / Experiments | 過去の実行ログからリプレイ評価、プロンプト変更の A/B テスト |
| Agent Editor | コードなしでプロンプトやツール構成を変更・バージョン管理 |
| 設定調整 | モデルの切り替え、パラメータの変更をUIから実行 |

開発中は `mastra dev` でローカル起動し、本番環境では `mastra studio deploy` で URL にデプロイしてチームで共有できます。RBAC と認証による権限管理も備えています。

```
mastra dev              # Server + Studio をまとめて起動（開発用）
mastra studio deploy    # Studio を単独でクラウドにデプロイ
```

要するに、**Server は「エージェントを動かして外部に公開する」場所**、**Studio は「エージェントの振る舞いを見て改善する」場所**です。この2つをどこに配置するかが、デプロイメントモデルの選択です。

## 3つのデプロイメントモデル

元記事では、ホスティングオプションとして以下の3パターンが提示されています。

| # | モデル | Server | Studio |
| --- | --- | --- | --- |
| 1 | フルマネージド | Mastra Platform | Mastra Platform |
| 2 | セルフホストServer＋マネージドStudio | 自社インフラ | Mastra Platform |
| 3 | 完全セルフホスト | 自社インフラ | 自社インフラ |

これは Server と Studio の配置場所の組み合わせです。下に行くほど自社管理の範囲が広がり、上に行くほど Mastra に委ねる部分が多くなります。

## モデル1：フルマネージド

### 概要

Server も Studio も Mastra Platform にデプロイし、ランタイムと可観測性のすべてを Mastra に委ねるモデルです。本番データベースのホスティングにも対応しています。

![フルマネージド デプロイメントアーキテクチャ](https://res.cloudinary.com/zenn/image/fetch/s--Wu-BNF9s--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://mastra.ai/images/blog/deployment-models/fully-managed.png?_a=BACAGSGT)  
*Mastra社の記事より*

### 実例1：サーバーレス志向の小規模チーム

マルチエージェントシステムを構築したい小規模チームが、サーバーレスを維持したいと考えていました。Mastra はフルマネージド構成で支援し、以下のアーキテクチャを実現しました。

* **Mastra ホスト型 Server** がワークフローオーケストレーターとして機能し、タスクを各エージェントに振り分け
* **Mastra ホスト型 Studio** でチーム全体が可観測性ツールを利用

このチームはインフラ管理を一切行わず、エージェントのロジック開発だけに集中できています。

### 実例2：Vite + React フロントエンドを持つ企業

ユーザー向けアシスタントを構築する企業が、Mastra のホスティングサービスでエージェントとツールコールを実行しています。

* API エンドポイントが **Vite + React フロントエンド**と接続
* データ取得のために **Elastic インスタンス**と統合
* 迅速な開発イテレーションのために **Studio** を利用

フロントエンド（Vite + React）は自社で管理し、バックエンドのエージェントランタイムは Mastra に任せるという分担です。

### このモデルが適するケース

* **スタートアップや小規模チーム**で、インフラ管理に割くリソースがない
* **サーバーレスアーキテクチャ**を志向している
* スケーリングやメンテナンスを自社で行いたくない
* **速度最優先**で、エージェントのロジック開発に集中したい

### トレードオフ

* データが Mastra のインフラを通過するため、厳格なデータローカリティ要件がある場合は不適
* カスタマイズの自由度はセルフホストモデルに劣る
* Mastra Platform の料金体系（CPU 時間、データ転送量）に依存

### 訳註：「本番データベースのホスティング」について

元記事には "We also have the ability to host your prod database"（本番データベースのホスティングにも対応）と書かれています。この一文だけ読むと、Mastra Platform 上で PostgreSQL のようなマネージド DB を立てられるように聞こえますが、実際はそうではありません。

Mastra Platform の[デプロイガイド](https://mastra.ai/guides/deployment/mastra-platform)には、以下の警告が明記されています。

> Mastra platform uses an **ephemeral filesystem**, so any storage you configure (including observability storage) **must be hosted externally**. If you're using LibSQLStore with a file URL, switch to a remotely hosted database.

つまり、Mastra Platform のファイルシステムは揮発性であり、ファイルベースの libSQL（SQLite）を直接使うことはできません。ストレージは Neon や Supabase のような外部のデータベースサービスを使う必要があります。

では元記事の「本番 DB のホスティング」とは何を指しているのか。料金表には "Data storage" という項目がありますが、これはデプロイアーティファクトや一時データ用の領域です。おそらく元記事が言及しているのは、**Enterprise 契約での個別対応**（顧客ごとに DB をプロビジョニングする）や、裏側のインフラ（Railway）の DB アドオンを紐づけるような形態と考えられます。いずれにせよ、セルフサービスで「DB を立てる」ボタンのようなものがあるわけではない点に注意してください。

## モデル2：セルフホスト Server ＋ マネージド Studio

### 概要

Server は自社インフラに置き、Studio だけを Mastra Platform（クラウド）で利用するモデルです。元記事では**最も一般的なパターン**と紹介されており、既存のホスティングインフラを持つ企業が可観測性レイヤーだけを追加したい場合に適しています。

![セルフホストServer＋マネージドStudio デプロイメントアーキテクチャ](https://res.cloudinary.com/zenn/image/fetch/s--5ZTkgVhB--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://mastra.ai/images/blog/deployment-models/self-hosted-server-managed-studio.png?_a=BACAGSGT)  
*Mastra社の記事より*

技術的には、Server がエージェントの実行トレースを Mastra Platform 上の Studio に**API コールで送信**します。ランタイム自体は自社インフラで動くため、リクエスト処理やデータアクセスの制御は完全に自社管理下にあります。

### 実例1：Google Cloud Run 上のウェブデータ分析企業

あるウェブデータ分析企業は、以下の構成でエージェントを運用しています。

* **Google Cloud Run** 上にエージェントランタイムをデプロイ
* エージェントとウェブアプリの両方に **BFF（Backend for Frontend）ティア** を使用
* エージェントの実行トレースを **クラウドホスト型の Studio** に送信
* チームは Studio 上で本番エージェントの動作を確認・操作

BFF パターンを採用することで、フロントエンド向けの API とエージェント向けの API を同じティアで管理しつつ、可観測性だけはマネージドサービスに委ねるという構成です。

### 実例2：AWS EKS 上の WhatsApp チャットエージェント

サービス企業が WhatsApp チャットエージェントを提供するのを支援するチームは、以下の構成を採用しています。

* **AWS EKS クラスター**内の Docker コンテナでエージェントを実行
* Mastra の可観測性機能は気に入っているが、**可観測性インフラの管理はしたくない**
* エージェントのスケールアップに伴い、運用の焦点をランタイムに絞りたい

可観測性のためのインフラ（ストレージ、ダッシュボード、アラート、ログ管理）を自前で構築・維持するのはコストがかかります。Studio のマネージド版を使うことで、その負担をゼロにしています。

### このモデルが適するケース

* **既存のクラウドインフラ**（AWS、GCP、Azure など）を持っている
* ランタイムのデータは自社管理したいが、可観測性は外部に任せてよい
* **既存プロダクトの上にエージェント機能を追加**している
* 可観測性インフラの構築・運用コストを避けたい

### トレードオフ

* トレースデータが Mastra Platform に送信されるため、全データの自社管理は不可
* Server のスケーリング・運用は自社責任
* Server と Studio 間のネットワーク接続が必要

### なぜこれが「最も一般的」なのか

多くの企業がこのモデルを選ぶ理由は明確です。エージェントのランタイムは既存のアプリケーションと同じインフラに載せたいけれど、可観測性ツールを1から構築するのは大変だからです。

従来のアプリケーション開発でいえば、アプリケーションサーバーは自社で運用しつつ、ログ管理には Datadog や New Relic を使う、という構成と同じ発想です。エージェントの世界でも同じパターンが自然に成り立ちます。

## モデル3：完全セルフホスト（ランタイム＋可観測性）

### 概要

Server と Studio の両方を完全に自社インフラ内に配置するモデルです。外部にデータが一切出ないため、厳しい規制要件に対応できます。

![完全セルフホスト デプロイメントアーキテクチャ](https://res.cloudinary.com/zenn/image/fetch/s--baIQd-OR--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://mastra.ai/images/blog/deployment-models/self-hosted.png?_a=BACAGSGT)  
*Mastra社の記事より*

### 実例1：HIPAA 準拠が必要なヘルスケアチーム

ヘルスケア分野のチームは、**HIPAA コンプライアンス**のためにすべてのホスティングを自社インフラ内に置く必要がありました。

* Mastra エージェントとワークフローを **Kubernetes 上にセルフホスト**
* ワークフローが**並列スーパーバイザーエージェント**を実行し、タスクをルーティング
* Studio は **セルフホスト Next.js アプリ**とともに同じインフラ内に配置
* **データがプライベートクラウドの外に出ない**

HIPAA（Health Insurance Portability and Accountability Act）は、米国の医療データ保護法です。患者の健康情報（PHI: Protected Health Information）を扱うシステムでは、データの保存・転送・アクセスに厳格な要件があります。外部のクラウドサービスにトレースデータを送信すること自体がコンプライアンス違反になりかねないため、完全セルフホストが必須となります。

### 実例2：AWS GovCloud 上の地方自治体機関

ある地方自治体機関は、以下の構成で内部エージェント一式を構築しています。

* **AWS GovCloud** 内の **EKS クラスター**上にデプロイ
* 組み込みの **RBAC（Role-Based Access Control）＆ Auth** を使用
* 内部チームが **可観測性レイヤー**（Studio）でエージェントとやり取り
* すべてのトレース、ソースコード、トラフィックを**自社インフラ内に保持**

AWS GovCloud は、米国政府機関向けに設計された隔離された AWS リージョンで、FedRAMP High、ITAR、CJIS などの厳格なコンプライアンス基準を満たしています。このような環境では、外部への一切のデータ送信が許されない場合があり、完全セルフホストモデルが唯一の選択肢となります。

### このモデルが適するケース

* **規制要件**（HIPAA、FedRAMP、GDPR など）により外部へのデータ送信が制限されている
* **政府機関や金融機関**など、データ主権が最優先
* セキュリティ監査で外部サービスへの依存が問題になる
* **オンプレミス**または**プライベートクラウド**での運用が義務付けられている

### トレードオフ

* Server と Studio の**両方の運用・保守が自社責任**
* インフラのスケーリング、アップデート適用、障害対応をすべて自前で行う必要がある
* 導入・運用コストが最も高い
* Mastra Platform のマネージド機能（自動スケール、マネージド DB など）を利用できない

## 3つのモデルの比較

| 観点 | フルマネージド | セルフホストServer＋マネージドStudio | 完全セルフホスト |
| --- | --- | --- | --- |
| **Server の運用** | Mastra | 自社 | 自社 |
| **Studio の運用** | Mastra | Mastra | 自社 |
| **データの所在** | Mastra Platform | Server は自社、トレースは Mastra | すべて自社 |
| **導入の容易さ** | ★★★ | ★★☆ | ★☆☆ |
| **運用コスト** | 低（従量課金） | 中（Server 運用＋Studio 課金） | 高（すべて自社） |
| **規制対応** | △ | ○（ランタイムデータは自社） | ◎ |
| **カスタマイズ性** | △ | ○ | ◎ |
| **スケーリング** | 自動（Mastra 任せ） | Server は自社管理 | すべて自社管理 |
| **典型的な利用者** | スタートアップ | 既存インフラを持つ企業 | 規制産業の大企業・政府機関 |

## デプロイメントモデル選択の判断フロー

どのモデルを選ぶべきかは、以下のような判断基準で絞り込めます。

### 1. 規制要件を確認する

最初に確認すべきは、**データが外部のクラウドサービスに送信されてよいかどうか**です。

* HIPAA、FedRAMP、ITAR などの規制下にあり、データの外部送信が不可 → **モデル3（完全セルフホスト）**
* トレースデータの外部送信は許容されるが、ランタイムデータは自社管理 → **モデル2** が候補
* 特段の制約なし → 3つすべてが候補

### 2. 既存インフラの有無を確認する

* AWS、GCP、Azure などのインフラをすでに運用している → **モデル2 または 3**
* インフラを持っていない、または管理したくない → **モデル1（フルマネージド）**

### 3. チームの運用力を考慮する

* インフラチームがいて、Kubernetes や Docker の運用経験がある → **モデル2 または 3**
* 開発者だけのチームで、インフラ運用は最小限にしたい → **モデル1**

### 4. コストとスピードのバランス

* 最速でプロダクションに到達したい → **モデル1**
* ランニングコストを最適化しつつ、可観測性の利便性も欲しい → **モデル2**
* 長期的なコスト管理とコンプライアンスが最優先 → **モデル3**

## Mastra Platform の料金参考

デプロイメントモデルの選択にはコストも重要な要素です。参考として、Mastra Platform（モデル1・モデル2で利用）の料金体系を示します。

**Studio / Server**

|  | Free | Teams | Enterprise |
| --- | --- | --- | --- |
| 価格 | 無料 | $250/チーム・月 | カスタム |
| ユーザー / デプロイ | 無制限 | 無制限 | 無制限 |
| 可観測性イベント | 10万件 | ― | カスタム |

Free ティアでもユーザー数・デプロイメント数が無制限というのは注目に値します。小規模チームがフルマネージドモデルで開始するハードルが低い設計です。

## 「Server と Studio の分離」がもたらした設計の自由度

従来の Mastra では、`mastra dev` を実行すると Server と Studio が一体的に起動していました。この一体型アーキテクチャはシンプルで開発体験は良いのですが、本番環境では以下の問題がありました。

* Studio の負荷が Server に影響する（あるいはその逆）
* Studio だけを社内チームに公開したいが、Server ごと公開することになる
* 可観測性ツールのスケーリングがランタイムのスケーリングと連動してしまう

Mastra Platform は Server と Studio を独立プロダクトとして分離したことで、これらの問題を解消しました。それぞれを**異なるインフラに、異なるスケーリングポリシーで、異なるアクセス制御のもとに配置できる**ようになったのです。

重要なのは、**開発体験は変わらない**という点です。ローカルでは `mastra dev` で今まで通り全部まとめて起動します。分離が意味を持つのは、本番デプロイの設計においてです。

## まとめ

Mastra アプリケーションのデプロイメントモデルは、「フルマネージド」「セルフホスト Server ＋ マネージド Studio」「完全セルフホスト」の3つに大別されます。

これは「正解が1つ」という話ではなく、チームの規模、既存インフラの有無、規制要件、運用能力、コスト感覚によって最適解が変わるという設計判断の話です。

Mastra が Server と Studio を独立プロダクトとして分離したことで、この選択が「全部 Mastra か、全部自前か」の二択ではなく、**コンポーネントごとに配置先を選べるグラデーション**になったことが、今回の記事の核心的なメッセージです。

特に「モデル2（セルフホスト Server ＋ マネージド Studio）」が最も一般的だという記述は示唆に富みます。多くの企業が「ランタイムは既存のインフラに載せたいが、可観測性の構築は大変」という課題を抱えており、従来のアプリケーション開発における Datadog や New Relic の位置づけと同じように、エージェントの可観測性をマネージドサービスとして利用するパターンが自然に定着しつつあることがわかります。
