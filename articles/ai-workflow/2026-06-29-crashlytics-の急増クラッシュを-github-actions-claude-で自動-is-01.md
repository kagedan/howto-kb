## はじめに

モバイルアプリを運用していると、毎朝 Crashlytics を眺めて「急増しているクラッシュ」を探し、手で GitHub Issue を立てる……という作業が地味に発生します。見落とすし、起票は雑になるし、見る人によってバラつく。いわゆる **トイル（価値を生まない繰り返しの手作業）** です。

そこで「**急増クラッシュを毎朝自動で見つけて、原因の見当をつけた Issue を勝手に立てる**」仕組みを作りました。この記事は、コードの写経よりも **仕組みの考え方** を中心に紹介します！

## 全体像

毎朝、ざっくりこう動きます。

ポイントはひとつだけ。**「起票するかどうか」は SQL で機械的に決め、「中身を調べて書く」ところだけ Claude に任せた** ことです。

## キモ：判定とライティングを分業する

|  | SQL | Claude |
| --- | --- | --- |
| 役割 | 起票する / しないを**判定** | 中身を調べて**文章化** |
| 強み | 毎回同じ基準でブレない・ほぼタダ | 柔軟な調査・自然な文章 |
| 苦手 | 文章は書けない | 判定がブレる・お金がかかる |

なぜ分けるのか。全部を Claude に任せると、起票基準が日によってブレるうえ、毎日動かすとコストがかさみます。逆に「機械的に決められる門番」は SQL がいちばん得意で、しかも激安です。**得意なものを得意な担当に振った**、という話です。

## 仕組みを順番に

### ① クラッシュは BigQuery に溜める

Firebase Console で Crashlytics の **BigQuery エクスポート** を ON にすると、クラッシュが BigQuery のテーブルに溜まっていきます（反映は最大1日遅れ）。

なぜ Crashlytics を直接見ないのか

Crashlytics 自体には「**過去データを自由に集計する API**」がありません。コンソールは人間が目で見る用で、「前日比3倍のクラッシュだけ抽出」のような条件はプログラムから書けないからです。

BigQuery に流すと、クラッシュが **ただのテーブル** になります。そうすれば「直近24時間が前日平均の3倍」のような条件を SQL で自由に書けます。即時通知が必要なら Firebase Functions という手もありますが、有料プラン・デプロイ・Secret 管理が要るため、「毎朝まとめて拾えればいい」今回は BigQuery が最も軽いと判断しました。

### ② SQL で「ヤバいやつ」だけ抜く

溜まったクラッシュから、毎朝この2タイプだけを抽出します。

* **急増タイプ**：直近24時間で10件以上 **かつ** 前日平均の3倍以上
* **影響大タイプ**：影響を受けたユーザーが5人以上

ここで地味に大事なのが **「いつからの24時間か」の基準** です。BigQuery エクスポートは最大1日遅れるため、「今の時刻」を基準にするとまだ届いていない時間帯を見て空振りします。

```
        昨日                  今         未来
  ───────┼────────────────────┼──────────▶ 時間
         │   データはここまで　  │
         │   しか届いてない  　  │
         └────────────────────┘ ← 最大1日の遅れ

  ✗「今の時刻」から24時間  → まだ空のゾーンを見て空振り
  ✓「データ最新時刻」から24時間 → 中身のある範囲を評価できる
```

なので基準は「現在時刻」ではなく **「届いている中で最新の時刻」** にします。

### ③ 候補ゼロの日は Claude を動かさない

SQL で抜いた候補が **0件なら、そこで終了**。クラッシュが落ち着いている日は LLM を起動せず、コストは0円です。「動かす価値がある日だけ動かす」ための、地味だけど効く工夫です。

### ④ Claude が「人間のデバッグ」を再現する

候補があった日だけ、Claude が登場します。やっているのは、人間がやるデバッグ手順そのままです。

Claude には **決まったツールだけ** を渡し、その範囲で自走させます。

| 渡したツール | Claude がやること |
| --- | --- |
| BigQuery 参照 | スタックトレースや発生バージョンを取得 |
| ソース検索・閲覧 | エラー箇所を手がかりに該当コードを探す |
| Issue 一覧の検索 | すでに立っている Issue は飛ばす（重複防止） |
| Issue 作成 | 原因仮説つきの本文で起票 |

アプリのコードが原因なら「たぶんこの行」まで書き、SDK や OS が原因なら「ネイティブ層」と明記して **断定しない**。このさじ加減もプロンプト（手順書）で指示しています。

## 安全に運用するための工夫

無人で Issue を立てる仕組みなので、暴発しないよう仕込みを入れています。

* **手動はお試しがデフォルト**：自分で試すときは、うっかり本番 Issue を立てないよう「起票しない」が既定。定時実行のときだけ本当に起票します。
* **重複は立てない**：Issue 本文にクラッシュ ID を埋め込み、起票前に既存 Issue を検索。あれば飛ばします。
* **最小権限**：Claude に渡す権限は「Issue を書く」だけ。触れるツールも上記の4つに限定しています。

## つまずきポイント

仕組みづくりで踏んだ落とし穴です。同じことをやる人向けに残します。

1. **24時間の窓は「データ最新時刻」基準**にする（バッチ遅延で空振りするため）
2. **スケジュール実行はデフォルトブランチに置かないと有効化されない**（feature ブランチに置いても動かない）
3. **iOS と Android で Firebase プロジェクトが分かれている**と、両方を見にいく設定が必要
4. リポジトリ同梱の設定ファイルが、実際のクラッシュ送信先と食い違っていることがある（要確認）
5. 公式の Crashlytics MCP は、無人 CI のサービスアカウント認証だとデータ取得が通らない場合がある（今回 BigQuery を選んだ理由のひとつ）

## まとめ

「**判定は SQL で決定的に、調査と起票だけ Claude に**」という分業と、GitHub Actions による無人実行で、急増クラッシュを毎朝自動で Issue 化できました。LLM は候補がある日だけ動くのでコストも最小限です。

クラッシュ対応の **繰り返し作業（トイル）削減** に、同じ考え方が役立てば幸いです！
