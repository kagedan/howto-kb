Sakana AI が「Fugu」を出した。触る前にドキュメントとベンチを読んだ感想を先に言っておくと、これは「賢い新モデル」ではなく「賢い束ね方を1モデルの皮で包んだ」プロダクトだ。OpenAI 互換の API を1本叩くと、裏では複数のモデルがチームを組んで答えを返してくる。呼ぶ側からは普通の1モデルにしか見えない。

最初に押さえたいのは、面白さの在りかだ。「複数モデルを合議させる」こと自体は新しくない。Mixture-of-Agents もアンサンブルも前からある。Fugu が一歩踏み込んでいるのは2点で、ひとつは束ね方を人手のワークフローではなく学習で獲得していること、もうひとつはそれを「1つのモデル」として商品化しきったことだ。この2つが噛み合って初めて、マルチエージェントが研究のデモから日常で叩ける製品になる。

## まず数字を見る

Sakana の公表値だと、上位版の Fugu Ultra は SWE-Bench Pro で 73.7、GPQA-D で 95.5。公開されているフロンティア（Opus 4.8、Gemini 3.1 Pro、GPT 5.5）を上回り、非公開の Fable 5 と Mythos Preview と肩を並べる、という主張になっている。下のグラフが公開モデルとの比較だ。

![Fugu Ultra と公開フロンティアモデル（Opus 4.8 / Gemini 3.1 Pro / GPT 5.5）の SWE-Bench Pro・GPQA-D スコア比較の棒グラフ](https://static.zenn.studio/user-upload/6106d2c792ef-20260622.png)

ひとつ注意がいる。Anthropic の2モデルは Fugu の内部プールには入っていない。「裏で Fable や Mythos を呼んで勝っている」わけではない、と明記されている。単体モデルの賢さで殴ったのではなく、手持ちのモデルの束ね方で公開モデルの上を取った。これがこの数字の読みどころだ。ベンチの額面はいつも割り引いて読むべきだが、主張の構造自体が「束ね方が効く」側に寄っているのは押さえておきたい。

## 束ね方を「学習する」：TRINITY と Conductor

中身は2本の研究が土台になっている。the-decoder によると、どちらも ICLR 2026 で発表されたものだ。

TRINITY は、軽量な「進化で得たコーディネータ」が、Thinker・Worker・Verifier という役割を、コーディング・数学・推論・知識といったタスクごとに各モデルへ割り当てる。誰が考え、誰が手を動かし、誰が検証するかを、その場のタスクに合わせて振り分ける係だと思えばいい。

Conductor はもう一段メタだ。強化学習で「自然言語の協調戦略」、つまりエージェント同士の通信パターンそのものを学習する。個人的にはここが一番効いていると思う。よくある自作マルチエージェントは、プロンプトのチェーンや会話の段取りを人間が手で書く。Fugu はその「どう会話させるか」を学習対象にした。固定のワークフローではなく、協調の仕方をポリシーとして持つわけだ。

しかもモデルプールは入れ替え可能（swappable）になっている。新しいモデルが出てもプールを差し替えれば、束ね方のロジックはそのまま使い回せる。モデルは消耗品、オーケストレーションは資産、という割り切りだ。全体像は下の図にまとめた。

## 本当の勝ち筋は「アーキを隠したこと」

ここからが2段目で、たぶん見落とされやすい。Fugu の効きどころは、アーキテクチャそのものより「アーキテクチャを隠したこと」にある。技術より、技術の畳み方だ。

理由は3つある。

ひとつ、単一の OpenAI 互換 API で、SDK の移行もいらない。マルチエージェントを使うのにフレームワークを覚える必要がない。採用障壁がほぼゼロになる。

ふたつ、課金が単一レートになっている。複数のエージェントが動いても、課金は「関わった最上位モデルのレート1本」だけ。これが地味に本丸だ。もし内部で5モデル呼んだら5倍課金、という設計だったら誰も使わない。マルチエージェントが経済的に成立する条件を、課金の設計側で作りにいっている。

みっつ、ルーティングは設計として非公開だ。どのモデルが答えを書いたかは見えない。特定モデルを外す opt-out はできるが、選ぶことはできない（Ultra はプール固定）。

この3つを並べると、別の勝ち筋が見えてくる。「賢いモデルを作る」競争とは別に、賢い束ね方を1つのエンドポイントの裏にどれだけ薄く畳めるか、という競争がある。Fugu はその最初の量産品に見える。

ここまでを表で整理しておく。

| 観点 | 単一モデル | 自作オーケストレーション | Fugu 型（system as a model） |
| --- | --- | --- | --- |
| 束ね方 | なし | 人手のワークフロー | 学習で獲得（TRINITY / Conductor） |
| 導入コスト | 低い | フレームワーク学習が要る | 互換 API のみ・SDK 移行なし |
| 課金 | トークン単価 | 呼んだモデルぶん積み上がる | 最上位モデルの単一レート |
| 中身の可視性 | 高い（1モデル） | 自分で組むので全部見える | 非公開（opt-out のみ） |
| 強み | 単純で予測しやすい | 完全に作り込める | 束ね方を学習・運用を肩代わり |

自作オーケストレーションの一番の摩擦は、課金が積み上がることと、組み込みの手間だった。Fugu はそこを単一レートと互換 API で潰している。代わりに差し出しているのが「中身の可視性」だ。トレードオフの形がきれいに見える。

## 叩く側に効く3つの変化

リクエストが内部でどう流れるかを図にしておく。呼ぶ側は1本の API だが、中ではチーム編成と検証ループが回っている。

![](https://static.zenn.studio/user-upload/e36014d19b34-20260622.png)

で、これがアプリを作る側に何を意味するか。「自分が叩く1モデル・1エンドポイントは、もう中身がマルチエージェントかもしれない」を前提にすると、設計の前提が3つ変わる。

1. レイテンシのばらつきが大きくなる。Thinker だけで返る軽いタスクと、フルチーム＋検証ループが回る重いタスクとでは、内部の重さがまるで違う。p50 と p99 の差が開く前提でタイムアウトやストリーミングを組まないと、固定の閾値で事故る。
2. 再現性とコスト見積もりの単位が変わる。同じ入力でも内部ルーティングが変われば出力は揺れうる。課金も「トークン単価」ではなく「タスク単価（最上位レート）」で考えることになる。原価計算の前提が、入力長ベースから難易度ベースへ寄っていく。
3. ブラックボックス・ルーティングは新しいロックインになる。どのモデルが答えたか分からないと、ベンダーをまたいだ評価や監査がやりにくい。制御の手綱が opt-out しかない、というのは運用上けっこう重い。

裏返すと、ここで効いてくるのは結局「自前の小さな eval と実行検証の harness」を持っているかどうかだ。中身が見えないなら、出口で測るしかない。評価の軸が「モデルを測る」から「オーケストレーションを測る」へ寄る、と言ってもいい。

コードで言うと、差し替え自体は1行で済む。互換 API だから接続先を変えるだけだ。引っ越せないのは、その後ろにある検証の前提のほうだ。

```
# 差し替えは1行（OpenAI 互換なので base_url を変えるだけ）
client = OpenAI(base_url="https://api.fugu.example/v1", api_key=...)

# 引っ越せないのは「評価の前提」。中身が見えない以上、出口で締める。
def ask_with_gate(prompt, check):
    out = client.chat.completions.create(
        model="fugu-ultra",
        messages=[{"role": "user", "content": prompt}],
    )
    text = out.choices[0].message.content
    if not check(text):          # 自前の実行検証・アサーション
        raise VerificationError  # 通らなければ落とす、もしくは再試行
    return text
```

`check` の中身、つまり何をもって成功とするかは、ベンダーは書いてくれない。ここを持っているチームだけが、中身の見えないエンドポイントを安心して本番に流せる。

## おわりに

整理すると、「モデル」という単位が、訓練済みの重みという成果物から、オーケストレーションを隠した1つのエンドポイントへと移りつつある。Fugu のベンチが示すのは能力の出どころのシフト（単体の賢さではなく束ね方）で、互換 API と単一レート課金が示すのは、それを商品に変えるための条件だ。技術と、技術の隠し方が、別々のレバーとして同時に効いている。

作る側の持ち帰りは1つでいい。次に叩く API は「中身が見えないチーム」かもしれない、という前提でレイテンシ・再現性・原価・評価を組み直す

![](https://static.zenn.studio/user-upload/97cb72d407bf-20260622.png)

こと。中身を覗けない以上、自分の出口に検証を置く以外に打ち手はない。

見立ても置いておく。この「system as a model」が標準化すると、競争の軸はモデル単体のベンチから、束ね方の質と、それをどれだけ薄く隠せるかへ移る。賢さがコモディティ化した次に来るのは、たぶん束ね方のコモディティ化だ。そのとき差がつくのは、束ねる側ではなく、束ねられたものを正しく測れる側だろう。
