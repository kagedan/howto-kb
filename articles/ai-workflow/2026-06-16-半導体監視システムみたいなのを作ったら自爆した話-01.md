# 結論から

副業にしようと思って、日本の半導体メーカー50社のニュースを毎日自動で監視するシステムを作りました。Redisキャッシュ、並列処理、統計的な異常検知、GitHub Actionsでの自動実行。我ながら、わりと本格的に作り込んだ方だと思います。  
そして一度も動かさないまま、力尽きました。  
正確に言うと、力尽きる前にClaudeに相談して、相談した結果、作ったものの大半が「動かす意味あるんですか?」という状態だと判明しました。  
この記事は、個人開発で機能を盛りすぎて自爆した話と、その過程で気づいた「一番痛いところ」の話です。同じように「作るのは好きだけど、なぜか完成品が誰にも届かない」人に届けばうれしいです。技術的な学びはほぼありません。心当たりがある人向けです。

# そもそも何を作ろうとしてたか(前提を3行で)

* 日本の半導体メーカー(信越化学、レーザーテック、DISCOとか)は、世界的に重要なのに英語圏のニュースでほぼ報じられない
* なら、その「死角」を毎日監視してデータにすれば、投資家とかに価値があるのでは?
* それを全自動でやるシステムを個人で作る

「Western media blind spots(欧米メディアの死角)」を突く、みたいな格好いい構想でした。構想は。

# 作ったもの

最初はシンプルでした。ニュースを取ってきて、LLMに「これサプライチェーン的にやばい?」と聞いて、点数をつける。それだけ。  
それだけだったはずなんですが、作っているうちにどんどん盛れてくる。エンジニアなら分かってもらえると思うんですが、作ってる最中って一番楽しいんですよ。で、気づいたらこうなってました。

# 実際に出来上がったもの

```
ニュース取得(リトライ + キャッシュ + レート制限)
  → LLM分析(信頼度スコア付き)
  → ベイズ更新
  → Thompson Sampling(探索と活用のバランス)
  → 指数移動平均で平滑化
  → アンサンブルで複数シグナル統合
  → Zスコアとigで異常検知
  → アラート発火
  → Slack/Discord通知
  → Google Driveに二重保存
  → GitHub Actionsで1日2回自動実行
```

Thompson Sampling。アンサンブル。異常検知。  
……誰のために?  
いや、必要だと思って入れたんです。精度が上がると思って。実際、コードとしてはちゃんと動く(はずの)ものが書けた。1,700行くらいになりました。認証まわりまで作り込んだバージョンもあります。  
問題は、この時点で顧客がゼロ人だったことです。いや、ゼロどころか、一人も「これ欲しい」と言った人間に会ったことがなかった。

# で、Claudeに相談した

さすがにそろそろ売り方を考えるか、と思ってClaudeに相談しました。「市場規模を増やすにはどうすればいい?」「価格はAとBどっちのフレーミングがいい?」「A/Bテストしたいんだけど」と。  
そしたら、まあ、ことごとく冷静に潰されまして。  
「A/Bテストは、あなたの想定トラフィックだと統計的に有意な結果が出る確率が6割を切ります。半分近くは『何も分からないまま6ヶ月が終わる』可能性があります」  
「市場規模(SAM)は約3.12M。これは大きく稼ぐ市場ではありません」  
「機能を足すより、まず3人の見込み客に『これに金を払うか』を聞くべきです」  
全部、正しい。グウの音も出ないくらい正しい。実際にモンテカルロシミュレーションまで回して数字で見せてきた。

~~Claude、お前が「やれ」って言ったこと全部やったあとで「それ意味ないですよ」って梯子外すのやめてくれない?~~  
……まあ、よく読み返すと「やれ」とは一言も言ってなくて、勝手に作り込んだのは全部こっちなんですけど。冷静になると、相談する前から薄々わかってたことを、ちゃんと数字で突きつけられただけでした。逆ギレする相手を間違えてるのはわかってるんすよ。

# で、これに何の意味があるのか

ここからが、自分にとっての本題です。  
Claudeに一通り潰されたあと、一番ブーメラン刺さった指摘がこれでした。

* 製品を磨くのは楽しい。でもそれは「売れる証拠」から逃げる行為でもある。

これ、過去の自分に全部刺さるんですよ。  
私、これが初めてじゃないんです。認証機能まで作り込んだ別のプロダクトもある。学生の頃はブログ的なものをやってて、それなりに数字も出てた。でも毎回、作り込むところまでは行くのに、「他人に見せて評価される」一歩手前で止まる。  
今回もそうでした。Thompson Samplingを実装する時間はあったのに、知り合いの一人に「こういうの作ってるんだけど、どう思う?」と聞く時間は作らなかった。異常検知のZスコアは計算したのに、自分のビジネスが当たるかどうかの「検知」からは目を逸らしていたんですよね。  
機能を足すのは、前に進んでる感覚があるんです。コミットログも伸びるし、コードも賢くなる。達成感がある。でもそれは、一番怖い質問(「これ、誰か金払うの?」)に当たらずに済む、いい言い訳でもあった。  
作り込みは、逃避だった。少なくとも今回の私にとっては。

# 留保と振り返り

数字や反省が都合よく見えるときほど自分を疑うことにしているので、正直なところを書いておきます。

1. 「作り込み=悪」ではない  
   技術を突き詰めること自体はいいことだし、その過程で身についたものは無駄にならない(Thompson Samplingの実装、勉強にはなった)。問題は「作り込み」そのものじゃなく、それを検証から逃げる口実に使っていたこと。ここを混同すると、ただの「手を動かすな」という話になって、それは違う。~~というかはやくGeminiもここまで作れるようになってくれ!いやほんとにClaude高いんじゃ~~
2. このシステムが無価値だと確定したわけではない  
   まだ誰にも見せてないので、本当に売れないのかは分かりません。「売れない」も「売れる」も、まだ仮説です。確定したのは「顧客に一度も聞かずに半年作り続けていた」という、私の動き方の問題だけ。製品の評価はこれからです。(だすとは言ってない)
3. Claudeが常に正しいわけでもない(知ってた)  
   冷静で正しい指摘ではあったけど、AIは構造的にリスクを低く見積もって保守的な結論に寄りがちです。独創的なことをやろうとすると、だいたい「やめとけ」寄りの分析が返ってくる。鵜呑みにすると何も挑戦できなくなる。今回は私の動き方が明らかにおかしかったから刺さっただけで、毎回従うべきという話ではないと思っています。

# おわりに

というわけで、半導体監視システムは今、一度も本番稼働しないまま私のリポジトリで眠っています。動かすのに必要なコストは月20ドルくらいだと分かったんですが、動かすより先に、やることがあるなと。  
次は、まず人に見せます。この記事もその一歩のつもりです。

まぁアレですね。やっぱり、最終的に責任を取るとか判断するっていう立場は、絶対に人間に残るんだなと。もう痛い目にあったので、二度とやりたくないですけど。ほんとに。  
同じように「作るのは得意なのに、なぜか完成品が世に出ない」って人、わりといる気がしてます。もしいたら、どうやってその壁を越えたか(or 越えられてないか)、話せたらうれしいです。こっちはまだ越えてる途中なので。
