## 概要

この動画は、Sanity の Rostislav Melkumyan が、同社の Content Agent を作る中で得たエージェント設計の学びを紹介する講演です。主張は明快で、「エージェントがツールをうまく使えないとき、それはツール不足ではなく、コンテキストの渡し方の問題であることが多い」というものです。

Sanity は、構造化されたコンテンツ、スキーマ、参照関係、GROQ というクエリ言語を持つため、エージェントにとって扱いやすい環境です。ただし、扱う対象が数百のコンテンツタイプや数万のドキュメントに及ぶと、すべてをモデルのコンテキストに入れることはできません。そこで登壇者は、モデルに「全部」を見せるのではなく、まずデータの「形」を見せ、必要になったときだけ詳細へズームインし、最後に実行するという三段階の設計を提案します。

講演の中心にあるのは、スキーマ概要、セット（Sets）、コードによる検証、慎重なサブエージェント利用、そして良い指示設計です。どれも派手なマルチエージェント構成ではありませんが、実運用でエージェントを安定させるためのかなり実践的な話です。

## 要点

1. エージェントのツール利用が悪いとき、根本原因はツール数ではなくコンテキストの形であることが多いです
2. Sanity では、モデルに生の巨大な JSON を渡すのではなく、まずスキーマの圧縮ビューを渡します
3. 「概要を先に、詳細は必要になったときだけ」という設計により、モデルは正しい対象を見つけやすくなります
4. Sets は、大量のドキュメントをコンテキストに入れずに扱うための圧縮された作業メモリです
5. Sets はエージェントだけでなく、ワークフローとユーザー確認にも効くため、信頼のプリミティブとして機能します
6. 統計処理、CSV 生成、構造的な検証などは、プロンプトではなくコードで処理した方が安定する場合があります
7. GROQ クエリはモデルに下書きさせ、コードで検証・補正・実行するという分担が有効です
8. 複雑なマルチエージェント構成は、サブエージェント起動時にコンテキスト損失を起こしやすいです
9. より多くのツールを足すより、明快で重複のない指示と、エージェントに与えるメンタルモデルの設計が重要です
10. 最終的な設計パターンは、Shape、Detail on demand、Execution の三段階です

## 詳細

### 問題は「ツール呼び出し」ではなく「何を見せるか」

Rostislav Melkumyan は冒頭で、エージェントがツールをうまく使えない場合、それはツール自体の問題ではなく、コンテキストの問題であることが多いと述べます。

Sanity の Content Agent は、コンテンツの監査、生成、編集、一括更新を支援するエージェントです。ここでいう「コンテンツ」は、動画や電子書籍のような配信メディアそのものではなく、企業の Web サイトやドキュメントサイト、EC サイト、アプリで使われる構造化データを指します。たとえばブログ記事、製品情報、ヘルプ記事、著者プロフィール、マーケティング用のページ本文、アプリ内で表示する説明文や設定データのようなものです。こうした作業では、モデルにツールを呼ばせることよりも、モデルに何を見せるか、どの解像度で見せるかの方が難しかったと説明します。

発表全体は、優れたエージェント設計を三つのレベルで捉えています。

1. **Shape**: まず対象の大まかな形を見せる
2. **Detail on demand**: 必要なときだけ詳細にズームインする
3. **Execution**: 確信を持ててから高価または危険な処理を実行する

これは、モデルにすべてを渡して賢く判断してもらう設計ではありません。むしろ、モデルが最初に見るべき情報を絞り、判断に必要な形だけを先に与える設計です。

### Sanity はエージェントに向いた環境だが、それでも失敗する

Sanity は、コンテンツを構造化して扱うためのプラットフォームです。Studio でスキーマとコンテンツを扱い、Content Lake に構造化 JSON としてデータを保存し、GROQ でクエリします。

この環境はエージェントに向いています。コンテンツは構造化され、参照関係は明示的で、クエリ言語も表現力があります。さらに Sanity は長年パブリックに開発されてきたため、ドキュメント、サンプル、スキーマ、GROQ クエリがインターネット上に多く存在します。モデルはすでに Sanity や GROQ に触れている可能性が高く、完全に未知のシステムを学ばせるよりも有利です。

それでもモデルは、GROQ でフィールドを入れ忘れたり、間違ったフィルタを書いたり、存在しない関数を作ったりします。環境がエージェント向きであっても、コンテキストの渡し方を間違えると失敗する、というのがこの講演の出発点です。

### 「ツールを増やす」はすぐ限界に達する

Sanity チームも最初は、エージェントがうまく動かないたびにツールを追加する方向へ進みました。しかし、ツールが増えるほど境界は曖昧になり、似たようなツールが増え、モデルはユーザーの問題ではなく「どのツールを選ぶべきか」に認知負荷を使うようになります。

登壇者の結論は、能力を細かいツールに切り分け続けるのではなく、ツールをより広くし、境界を明確にし、重複をなくし、指示を改善することでした。

ここで重要なのは、「ツールが少ない方が常に良い」という単純な話ではありません。モデルが何を見て、何を判断し、どこから先をツールやコードに任せるのかを設計する必要がある、という話です。ツール設計は、コンテキスト設計の一部として考えるべきものです。

### まずスキーマの「圧縮ビュー」を見せる

コンテンツエージェントにとって、スキーマ理解は非常に重要です。しかし実際のスキーマは巨大です。多数のドキュメントタイプ、ネストされたオブジェクト、配列、参照、バリデーション、再利用可能な型などをすべてコンテキストに入れると、モデルは大量の詳細を見ますが、全体像をつかみにくくなります。

そこで Sanity では、まずスキーマの圧縮ビューを渡します。どんなドキュメントタイプがあるか、重要なトップレベルフィールドは何か、どの型が何を参照し、どこから参照されているかを、巨大な JSON ではなくグラフに近い形で見せます。

講演では、blog post、author、person の例が挙げられます。人間なら、ブログ記事が参照するのは author だと分かるかもしれません。しかしモデルには person も author も正しそうに見えます。ユーザーが「Sam は何本ブログを書いたか」と尋ねたとき、弱いエージェントは person を探して blog post と結びつけようとし、結果ゼロを返してしまうかもしれません。

一方、圧縮ビューで「blog post は author を参照している」「author には blog post から多数の incoming references がある」と分かれば、モデルは正しい経路を選びやすくなります。

このパターンは、まず概要を見せ、必要なところだけ詳細にズームインする、という設計です。モデルが company に contacts 配列があると分かれば、必要なときに contacts の中身へ進み、さらに必要なら person.profile のキーへ進む、という形です。

### Sets は大量コンテンツを扱うための作業メモリ

次の課題はスケールです。数百、数千、数十万のドキュメントを、すべてメインコンテキストに入れずにどう扱うか。

ここで登場するのが Sets です。登壇者は Sets を「圧縮された作業メモリ」と説明します。完全なデータ本体ではなく、クエリ結果への再利用可能なハンドルです。そこには、ドキュメント数、型の内訳、数値や日付の範囲、元のクエリ、タイトル、説明などのメタデータが付いています。

エージェントは、結果全体を見るのではなく、まず Sets の形を見ます。件数は多すぎないか、含まれているドキュメントタイプは意図通りか、日付や数値の範囲は妥当かを確認できます。

これは、リブランドの影響を受ける全ドキュメントを探す、古い API 名を使っている記事を探す、大量の記事からパターンを分析する、数百件のドキュメントを安全に更新する、営業やサポート向けに CSV を作る、といった実務に向いています。どれも一つのドキュメントではなく、ドキュメント集合の問題だからです。

### Sets は「信頼」のためのプリミティブでもある

Sets はエージェントのためだけの仕組みではありません。ワークフローにとっては次の処理対象になり、ユーザーにとっては確認可能な UI オブジェクトになります。

ユーザーは UI 上で Set を開き、エージェントが正しいコンテンツ集合を対象にしているか確認できます。つまり Set は、技術的なプリミティブであると同時に、信頼のプリミティブでもあります。

大きな処理や高価な処理を始める前に、エージェントは Set の形を検査できます。間違っていれば早く止められます。正しそうなら、小さなサブセットで処理し、ユーザーが承認してから全体に広げられます。コスト見積もりも、ユーザーが処理を確定する前に提示できます。

登壇者は、この流れを「Set を検証し、アクションを検証し、それからスケールアップする」と整理しています。

### すべてをプロンプトで解かない

Sets によって大きな作業対象を持てるようになると、クエリだけでは表現できない問題も扱えるようになります。たとえば、誤字があるドキュメント、古く感じる記事、ブランドらしくない商品説明、古い表現を使っている記事を見つける、といったタスクです。

こうしたタスクには推論が必要ですが、大きな Sets に対する推論は高価になりがちです。そこで Sanity では、全件処理、チャンク処理、サンプリング、コード実行を使い分けます。

ここで重要なのは、エージェントが分析をいつもプロンプトで解く必要はないという点です。統計的な処理や CSV 生成、レポート作成のような作業は、整えたデータに対してコードを実行した方が安定し、安く、速い場合があります。

流れとしては、まず Set を作り、結果の形を理解し、どう作業するかを決め、その後で深い分析、並列処理、エクスポートに進みます。一つの巨大なプロンプトにすべてを詰め込むより、はるかに扱いやすい設計です。

### モデルが提案し、コードが強制する

GROQ クエリの生成では、モデルは最初のドラフトを書く役割を担います。モデルは GROQ に慣れているため、ユーザー意図をそれなりに良いクエリへ変換できます。しかし、「それなりに近い」だけでは実運用には足りません。

たとえば、Sanity の多くのフローでは ID フィールドと type フィールドが必要です。モデルがこれを忘れると、どのドキュメントを扱っているのか分からず、ユーザーを正しい場所へリンクできません。

そこでコードが介入します。GROQ クエリをパースし、検証し、AST を調べ、必要なフィールドを追加し、シリアライズして実行します。コードで直せない場合は、モデルが自己修正できるように意味のあるエラーを返します。

登壇者は、この分担を「モデルが提案し、コードが強制する」と表現します。AI 時代でも、品質の多くはまだコードから来ます。プロンプトやツールを増やすだけでなく、コードで検証し、修復し、制約を守らせることが、信頼できるエージェントには必要です。

### マルチエージェントはコンテキストを失いやすい

次に登壇者は、複雑なマルチエージェント構成への警戒を述べます。サブエージェントは図では魅力的に見え、デモでもうまく動くことがあります。しかし実運用では、メインエージェントが長く積み上げた文脈が、サブエージェントに渡るときに短い指示へ圧縮されて失われがちです。

たとえばブログ記事のトーン、構成、意図を会話の中で丁寧に詰めてきたのに、サブエージェントには「この新製品についてブログを書いて」とだけ渡るかもしれません。これでは、作業のいちばん価値ある文脈が消えてしまいます。

そのため Sanity の現在の設計は、一つの強いメインエージェント、少数の強いツール、そして効果が明確なところだけに置く専門サブエージェント、という方向です。

例外として、mutation には専門サブエージェントが有効だと述べています。Sanity のドキュメント構造は深くネストされ、ライブコラボレーションを支えるために機械向けの形をしています。そのため変更処理では、関連するスキーマ型、編集対象のフィールドパス、Studio コンテキスト、ドキュメントコンテキストを起動時に絞って渡す専門エージェントが役立ちます。

ここでも目的は、すべてのコンテキストを渡すことではありません。正しいコンテキストだけを渡すことです。

### より良い指示は、より多いツールに勝つ

登壇者が特に強調するのが、指示設計の重要性です。これはマルチエージェントや新しいツールほど派手ではありません。しかし、指示が曖昧で、重複し、矛盾していると、ツールが技術的に正しくても壊れているように見えます。

良い指示は、明快で、具体的で、重複がなく、行動に基づいている必要があります。また、モデルに自分の仕事領域を理解しているというメンタルモデルを与える必要があります。

「あなたはツールを呼べるエージェントです」と言うのと、「あなたはコンテンツシステム、大規模なコンテンツ集合、スキーマ関係、構造化分析を理解しています」と言うのでは、モデルの振る舞いが変わります。後者の方が、より良い計画とツール利用につながると説明されています。

ツールの使い方を改善したいなら、ツール自体だけでなく、エージェントに与えているメンタルモデルも再設計すべきです。

### Shape、Detail、Execution の三段階

講演のまとめとして、登壇者は最終的な設計パターンを三段階で整理します。

**Level 1: Shape**

エージェントが向きを定めるための、ぼんやりした概要です。スキーマ要約、Set のメタデータ、件数、範囲、型の内訳、関係のヒントなどがここに入ります。

**Level 2: Detail on demand**

正確な構造は、必要になったときだけ取得します。スキーマパスを掘る、フィールドを深く見る、関連ドキュメントを読む、必要なリッチフィールドだけ展開する、といった形です。

**Level 3: Execution**

エージェントが十分に確信を持ってから、高価またはリスクのある作業に進みます。Set 全体の処理、多数のドキュメント変更、コード実行、ファイルエクスポートなどです。

この設計により、メインエージェントは単純に保たれ、コンテキストは汚れにくくなり、ユーザーにとっても信頼しやすいシステムになります。

### Q&A: データの最適な形と Sanity の位置づけ

Q&A では、エージェント実行コストと価値のバランスについて質問がありました。登壇者は、Sanity が人々のビジネスを構造化されたコンテンツとしてモデル化することを目指しており、エージェントはそれを速く進めるための手段だと答えます。Slack、Linear、Microsoft Teams、Telegram、メールなど、既存の場所からエージェントにアクセスできるようにすることにも触れています。

また、エージェントに渡すデータの最適な形については、「自分たちのデータに合わせた形」がよいと述べています。重要なのは、モデルにとって馴染みのあるパターンを使うことです。Sanity では TypeScript の型表現に似た形を使うと、モデルがすぐに理解したそうです。

成功時とエラー時で返すデータ形状が違う場合も、スタック全体で同じパターンを繰り返し、モデルを混乱させないことが重要だと説明しています。

最後に、知識グラフや Neo4j のようなグラフデータベースとの関係も問われました。登壇者は、必要なら伝統的なデータベースを使い、Sanity から参照すればよいとしつつ、コンテンツは Sanity に置くことで、人間とエージェントが同じ最新情報を扱える利点があると述べます。Sanity 自体も多くの参照関係を持つため、ある意味ではグラフデータベースのようにも使える、という整理です。
