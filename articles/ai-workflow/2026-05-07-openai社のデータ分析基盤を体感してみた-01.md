## 背景

データ分析基盤にメタデータは欠かせません。  
ただ、メタデータの情報だけではデータ分析が捗りません。これは人間もエージェントも同じです。  
その解決策を見事に言語化しているブログを見つけたので、少しだけ真似してみました。

## 参考にしたOpenAIの記事

<https://openai.com/ja-JP/index/inside-our-in-house-data-agent/>

OpenAIの記事では、データエージェントが正しく分析するためには、単なるスキーマ情報だけでは足りないと説明されています。

いわゆるメタデータはLayer #1 にあたります。これが6層までありますが、今回は3層まで再現しています。Layer #2, #3と続きますが、それぞれ人間の理解とコードの理解です。

今回のローカル実装では、これらを次のように対応させました。

| OpenAI記事の層 | 今回の実装 |
| --- | --- |
| Layer #1: Table Usage / Metadata | `manifest.json`, `catalog.json`, dbt model SQL, DuckDBでの実データ確認 |
| Layer #2: Human Annotations | `knowledge/*.md` の business context |
| Layer #3: Codex Enrichment | dbt SQLを読んで抽出した grain, joins, filters, aggregations, caveats |

### 質問1

「顧客ごとの累計売上を出して」

**回答**

```
select
  customer_id,
  first_name,
  last_name,
  number_of_orders,
  customer_lifetime_value as cumulative_sales_aud
from customers
order by cumulative_sales_aud desc nulls last, customer_id
```

思考過程

### 質問2

「顧客LTV上位10名。ただし返品注文は除外」

**回答**

```
with customer_ltv as (
  select
    customer_id,
    sum(amount) as ltv_excluding_returned_orders,
    count(*) as included_order_count
  from orders
  where status <> 'returned'
  group by customer_id
)

select
  c.customer_id,
  c.first_name,
  c.last_name,
  cl.included_order_count,
  cl.ltv_excluding_returned_orders
from customer_ltv cl
join customers c
  on cl.customer_id = c.customer_id
order by cl.ltv_excluding_returned_orders desc, c.customer_id
limit 10
```

思考過程

> `customers.customer_lifetime_value` は既に集計済みですが、返品注文を除外する条件を後から適用できません。そのため、注文粒度の `orders` から `status <> 'returned'` で返品済み注文を除外し、`customer_id` ごとに `sum(amount)` しました。集計後に `customers` とJOINしているため、注文行による顧客属性の重複は起きません。

注意点

> `return_pending` は「返品予定」であり、今回は `returned` のみを返品注文として除外しました。`return_pending` も除外する場合は条件を `status not in ('returned', 'return_pending')` に変更します。

### 質問2

「顧客LTV上位10名。ただし返品注文は除外」

**回答**

```
with customer_ltv as (
  select
    customer_id,
    count(*) as order_count_excluding_returns,
    sum(amount) as ltv_excluding_returns
  from orders
  where status not in ('returned', 'return_pending')
  group by customer_id
)

select
  l.customer_id,
  c.first_name,
  c.last_name,
  l.order_count_excluding_returns,
  l.ltv_excluding_returns
from customer_ltv l
left join customers c
  on l.customer_id = c.customer_id
order by l.ltv_excluding_returns desc, l.customer_id
limit 10;
```

思考過程

> * 粒度: 顧客ごと
> * 指標: LTV、ただし返品注文を除外
> * 集計単位: `customer_id`
> * 期間条件: 指定なし
> * `orders` knowledgeで `return_pending` も返品扱いで売上から除外とされているため、`returned` と `return_pending` の両方を除外
> * `orders` を先に顧客単位へ集計してから `customers` に結合し、join duplication を回避

注意点

> `customers.customer_lifetime_value` は集計済みですが、条件付きLTVでは粒度と条件を明示するため `orders` から再集計しています。今回のLTVは `status in ('placed', 'shipped', 'completed')` の注文売上合計です。

## 結果

Layer1のみ VS Layer2,3 の回答の主な違いは以下でした。  
質問2において、Layer2,3ではビジネス定義に合わせたステータスを選択しています。

```
with customer_ltv as (
  select
    customer_id,
    count(*) as order_count_excluding_returns,
    sum(amount) as ltv_excluding_returns
  from orders
- where status <> 'returned'
+ where status not in ('returned', 'return_pending')
  group by customer_id
)
```

これは「人による確認が必要」で定義を固めた部分なので、それを理解したLayer2,3が正しいです。

## 所感

Layer1は優秀な部下に仕事をお願いし、曖昧なところを補完してくれた感じです。  
それに対し、Layer2,3は仕様書を確認して細かい部分も詰めてきました。  
優秀な人材を生かすにはきちんとした定義と権限が必要ですね。

また、検証の限界もあります。  
jaffle\_shopのシンプルなテーブルで試したので、正直Layer1だけでも十分な分析をしてくれています。実際質問1はどちらも同じ回答に至っています。  
ビジネスに利用されているような複雑な実テーブルならもう少し差が生まれたかもしれません。
