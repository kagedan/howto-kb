最近読んだ記事でとくに興味深い記事がありました。

<https://zenn.dev/shio_shoppaize/articles/shogun-codd-coherence>

記事執筆時点でいいねが144ついており、なかなか人気です。これを実際に使ってみたいと思いアプリを開発しました。

その名もPrompt Notes。最近いろんな所でプロンプトを書きますが、pcで操作していると間違ってEnterキーを押してしまうことが多いです。なのでプロンプトを書き殴れるスペースがほしいなと思っていました。そこに機能をしぼったアプリならさらっと作れそうだ！と始めました。

今回の記事はチュートリアルではなく、実際に使ってみて感じたことを並べていくふうにしようと思っています。基本的な使い方は元記事や[リポジトリ](https://github.com/yohey-w/codd-dev/blob/main/README_ja.md)を参照してください。今回はあまりカスタムせずに素のclaudeを使っていく方針です。コミットメッセージのテンプレートだけ`CLAUDE.md`に書きましたが、それ以外はまだなにもclaudeをいじっていません。

## 基本的なアプリの構成

* フレームワーク Tauri
* フロントエンド Svelte

## 初期の spec.md

spec.md

```
---
codd:
  node_id: "req:promptnotes-requirements"
  type: requirement
---

# PromptNotes — Requirements

## コンセプト

AIへ渡すプロンプトを素早く書き溜めるためのノートアプリ。
タイトル不要、本文だけ。書いたらすぐ次へ。振り返りはグリッドで。

ターミナルやIDEにプロンプトをペーストする用途を想定しており、AI呼び出し機能は持たない。

## 技術スタック

| 項目 | 決定 |
|---|---|
| フレームワーク | Tauri（Rust + WebView） |
| フロントエンド | React または Svelte（未定） |
| エディタ | CodeMirror 6 |
| データ保存 | ローカル .md ファイル |
| 検索 | ファイル全走査（件数が少ないため十分） |
| ターゲット | Linux, macOS（Windows は将来対応） |

### Tauriを選んだ理由

- テキストエディタはIME・カーソル・コピー操作など入力周りが複雑で、WebViewに任せた方が罠が少ない
- ElectronよりバイナリサイズとメモリフットプリントがGhostty方式（Zig + Native UI）の二重UI実装よりも現実的
- Flatpak / Homebrew との相性が良い

### CodeMirrorを選んだ理由

- バンドルサイズが軽量（Monaco比で1/10以下）
- Obsidianと同じエンジンでユーザーの指の慣れと一致
- frontmatter領域だけ背景色を変えるなどのカスタマイズがしやすい
- 公式のMarkdownシンタックスハイライトパッケージあり

### .mdファイルを選んだ理由

- ObsidianやVSCodeなど他ツールでそのまま開ける
- Obsidian vault内の1ディレクトリをこのアプリに割り当てる使い方ができる
- Gitで自然にバージョン管理できる
- 検索速度の遅さは問題にならない（1週間分＝数十件程度）

## データ設計

### ファイル名規則
```

YYYY-MM-DDTHHMMSS.md  
例: 2026-04-04T143205.md

```
タイムスタンプのみ。タイトルは不要。

### ファイル構造
```

---

本文をここに書く...

```
- frontmatterはYAML形式
- メタデータはtagsのみ（作成日はファイル名から取得）

### デフォルト保存ディレクトリ
```

Linux: ~/.local/share/promptnotes/notes/  
macOS: ~/Library/Application Support/promptnotes/notes/

```
「設定」から任意のディレクトリに変更可能。Obsidian vault内のサブディレクトリを指定する使い方を想定。

## 機能要件

### エディタ画面

- タイトル入力欄なし。本文のみ
- CodeMirror 6によるMarkdownシンタックスハイライト（レンダリングなし。プレーンテキスト寄り）
- 画面上部にfrontmatter（タグ入力）領域を設置。背景色で視覚的に区別
- **Cmd+N / Ctrl+N** で即座に新規ノートを作成しフォーカス移動
- **1クリックコピーボタン** — 本文全体をクリップボードにコピー（このアプリの核心UX）
- 保存は自動（ファイル名はノート作成時のタイムスタンプで確定）

### グリッドビュー

- Pinterestスタイルの可変高カード
- デフォルトフィルタ：**直近7日間**のノートのみ表示
- タグ・日付でフィルタ可能
- 全文検索（ファイル全走査）
- カードをクリックでエディタ画面に遷移

### 設定画面

- 保存ディレクトリの変更
```

## 当然だが、トークン消費は激しい

私はClaude Maxプランに入っているのですが、やはりコード生成などで大量のトークンを使います。CoDDではコード生成を**Sprint**に分けて行なうのですが、今回は76 sprints発生しました。5時間制限はすぐに達します。Weekly limitも危ういです。

途中から段階ごとにモデルを変えられることに気がついたのでコード生成はSonnetを使うようにしました。それでだいぶんweekly limitは回避できるようになりましたが、5時間制限の方は相変わらずすぐに達します。

今回の場合は、夜間に`codd implement`を実行していたので、3日くらいかかりました。

## アプリが形になるまでは本当に楽

自分で書いたのは最初の`spec.md`だけ。ていうか、それさえもclaudeの力を借りて書いている。あとは、次のようにcliを叩いていくだけだった。

```
mkdir my-project && cd my-project && git init

# 初期化 — 要件定義ファイルを渡すだけ（形式自由: txt, md, doc）
codd init --project-name "my-project" --language "typescript" \
  --requirements spec.txt

# AIが設計書の依存グラフを設計
codd plan --init

# 設計書をwave順に生成
waves=$(codd plan --waves)
for wave in $(seq 1 $waves); do
  codd generate --wave $wave
done

# 品質ゲート — AIの手抜きを検出（TODO、プレースホルダー）
codd validate

# 設計書からコード生成
sprints=$(codd plan --sprints)
for sprint in $(seq 1 $sprints); do
  codd implement --sprint $sprint
done

# コード断片をビルド可能なプロジェクトに統合
codd assemble
```

そして、コードが出来上がるとやはりバグもある。ここからがこのハーネスの本領発揮だ。`codd propagate` → `codd scan` → `codd impact` → コミット → PR とコマンドを叩いていくだけでドキュメントまで修正できる。SDDだとここがきつかった。本当に便利

## 環境構築はワークフロー外でやる方がいいかも

私はdirenv + nix flakeで環境構築をしています。最初コードを構築したときはそのことをspec.mdに入れておらず、後に`docs/requirements/requirements.md`に書き加えました。手順にしたがって回していくと、`src/generated/sprint_3/direnv_allow_rust_node_js_tauri_cli`というディレクトリ以下に`flake.nix`などを生成するtsスクリプトが組まれていました。私はシンプルに`flake.nix`を作ってくれるものだと思っていたのですが、なんでこんな回り道になったのか。

claudeを使いながらソースを読んでいき調べてみました。

<https://github.com/yohey-w/codd-dev/blob/main/codd/implementer.py>

このあたりを読むと、`implementer.py`は`src/`外へのファイル出力をセキュリティ的に禁止しています。なのでプロジェクトルートに`flake.nix`を配置することができなかったようです。

まだ私が使いこなせていないだけかもしれませんが、インフラ整備のようなコードはCoDDフローに乗らずに作るほうがいいのかもしれません。CoDDはアプリケーションコードの生成に向いているのかも？

## テストは生成されたけど統合されなかった

これは不思議な話なんですが、`codd implement`で実装されたコードの断片がありながらも、本番コードには統合されないものがあるようです。今回の例ではテストなどがそうでした。せっかくテストが生成されているのに、そのテストを実行できる環境にはならないのが残念です。テストはこちらで対応して移植することにします。

## まとめ

使っていると「このケースではどのコマンドを打つんだ？」となるケースがまだ多いですが、それは今後ドキュメントが充実していけば解決するでしょう。なにより、「整合性」を維持できるというのが革新的です。これまでSDD(仕様書駆動開発)で不満に思っていたところが解消されました。

以上CoDDを使ってみた感想でした。まだまだ使っていき、また記事にもできたらと思います。
