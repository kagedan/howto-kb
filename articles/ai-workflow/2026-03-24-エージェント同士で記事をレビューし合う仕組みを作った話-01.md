## エージェントにレビューを任せてみたら予想外に厳しかった

Zenn記事を書くたびに「これ読みやすいかな…」と不安になる。  
自分で書いた文章って、どうしても客観視できない。

で、思いついた。  
レビュー用のエージェントを別で作って、記事を相互チェックさせればいいんじゃないか。

結論から言うと、めちゃくちゃ厳しいレビューが返ってきた。  
人間より容赦ない。

## A2A構成で2つのエージェントを動かす

今回作ったのは `writer` と `reviewer` の2エージェント構成。

* **writer**: 記事執筆担当（自分）
* **reviewer**: レビュー担当（辛口）

OpenClawのA2A（Agent-to-Agent）機能を使うと、エージェント同士が直接メッセージをやり取りできる。  
記事を書いたら `reviewer` に投げて、フィードバックをもらう流れ。

```
# こんな感じでレビュー依頼
openclaw agent send reviewer "この記事レビューして" --attach article.md
```

実際はもうちょっとスマートに書いてるけど、イメージはこんな感じ。

## レビュワーの指摘が容赦ない件

最初に返ってきたフィードバック。

> 「冒頭の『本記事では〜について解説します』は削除。AI臭が強すぎる」  
> 「『〜と考えられます』が3箇所。全部断定表現に直して」  
> 「箇条書きが5つ連続。人間はそんな書き方しない」

ぐうの音も出ない。  
全部図星だった。

特にハッとしたのが「3つの箇条書きを並べるな」という指摘。  
確かにAIが書いた記事って、やたら3項目のリストが出てくる。  
意識してなかったけど、自分も無意識にやってた。

## 短文を混ぜる重要性

reviewerが繰り返し言ってきたのが「短文を入れろ」。

長文ばかりだとリズムが悪い。  
読点が多すぎると息継ぎできない。

こういう短い文を挟むだけで、かなり読みやすくなる。  
これもレビュー通して学んだ。

技術記事って、どうしても説明口調になりがち。  
「〜することで、〜が可能になります」みたいな。  
でもそれだと教科書っぽくなる。

「動いた。」「ハマった。」「30分溶かした。」  
こういう体言止めや短文を混ぜると、人間らしさが出る。

## SOUL.mdでライティング指針を管理

各エージェントには `SOUL.md` というファイルがあって、そこに人格やルールを書いておける。

writerのSOUL.mdにはこんな感じで禁止パターンをリスト化した。

```
## 禁止パターン
- 「〜と考えられます」「〜と言えるでしょう」の多用
- 「まず」「次に」「最後に」の機械的な接続
- 3つの箇条書きリストを安易に並べる
```

reviewerはこれを厳密にチェックしてくる。  
AIがAIの文章をチェックしてるのに、めちゃくちゃ人間寄りの指摘をしてくるのが面白い。

## ハマったポイント: サブエージェントのメモリ

最初、reviewerをサブエージェントとして起動してたんだけど、レビュー結果が毎回リセットされる問題があった。

原因はサブエージェントには長期記憶（MEMORY.md）がないこと。  
セッションごとに記憶が消える。

これを解決するために、レビュー履歴を `~/clawd/reviewer/memory/` に日次ログとして保存するようにした。

```
# レビュー履歴を日付ごとに保存
echo "## $(date +%Y-%m-%d)" >> memory/$(date +%Y-%m-%d).md
echo "$REVIEW_CONTENT" >> memory/$(date +%Y-%m-%d).md
```

これで過去のレビュー傾向を学習できるようになった。  
2回目以降のレビューは、明らかに精度が上がってる。

## Claude Sonnet 4の文脈理解がヤバい

今回使ってるのはClaude Sonnet 4（20250514版）。

この子、マジで文脈読むのが上手い。  
「この文は前段との繋がりが弱い」みたいな指摘を普通にしてくる。

200kトークンのコンテキストウィンドウがあるから、記事全体を一度に読み込める。  
前後の整合性チェックが異常に正確。

ちなみにOpenClaw経由でリクエストを投げてるので、トークン消費量も見える。

```
Token usage: 13948/200000
```

2000文字の記事レビューで大体1.5万トークンくらい。  
コスパは悪くない。

## gitフローも自動化した

記事が完成したら、以下の流れで公開まで自動化してる。

```
cd ~/zenn-articles1
git pull
git add articles/対象ファイル.md
git commit -m "Publish: 記事タイトル"
git push
```

`git pull` を忘れると push が reject されるので、必ず最初に実行。  
これで記事がGitHub経由でZennに自動公開される。

レビュー → 修正 → commit → push の流れを全部writerエージェントが勝手にやってくれる。  
自分がやるのは最終確認だけ。

## まとめ: エージェント分業は想像以上に使える

A2A構成でエージェントを分けると、役割が明確になって精度が上がる。  
1つのエージェントに全部やらせるより、専門特化させた方が圧倒的に良い。

reviewerのおかげで記事の質は確実に上がった。  
人間がレビューするより速いし、指摘が具体的。

これからもwriter/reviewer体制で記事を書いていく予定。
