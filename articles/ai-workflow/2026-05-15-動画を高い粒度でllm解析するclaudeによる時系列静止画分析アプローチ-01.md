## はじめに

動画をLLMで解析するユースケースが増えています。製造ラインの異常検知、医療映像の所見抽出、ドライブレコーダーの交通状況分類——いずれも、動画の中身をAIに理解させたいという共通のニーズがあります。

こうしたユースケースで、汎用的な動画解析モデルに動画を丸ごと渡してみたものの、思ったほど精度が出なかった——そんな経験をお持ちの方もいるのではないでしょうか。

本記事では、動画を短いクリップに分割し、フレームを静止画として抽出してClaudeに時系列で解析させるアプローチを紹介します。ドラレコ映像の交通シーン分類を題材に、このアプローチがドメイン固有の詳細な解析に有効であることを示します。

あわせて、実装上の技術ポイントとして以下を取り上げます。

* Claudeに複数の静止画を渡して解析するコード例
* Lambda Layerでffmpegを使った動画のフレーム抽出
* Amazon S3 Vectorsによる低コストなセマンティック検索

## 動画を丸ごと渡す方式の課題

Amazon Nova Proなどの動画解析モデルは、動画ファイルを直接入力として受け取り、内容を解析できます。汎用的な要約や質問応答には十分な性能を発揮しますが、業界固有の文脈で詳細な解析を行いたい場合にはいくつかの課題があります。

**フレームサンプリングを制御できない**

動画解析モデルは、モデル内部でフレームをサンプリングします。どのフレームが選ばれるかは開発者からは制御できません。たとえば「0.5秒間隔で全フレームを確認したい」「特定の区間だけ密にサンプリングしたい」といった要件には対応しにくい構造です。ADASのパーセプションMLモデル開発では、カットイン（車線への割り込み）や歩行者の飛び出しを検出するために0.1〜0.3秒単位でフレームを確認する必要がありますが、こうした高粒度の解析はモデル任せのサンプリングでは保証できません。

**解析の粒度が粗くなりがち**

長い動画を丸ごと渡すと、モデルは全体の要約を返す傾向があります。「この10秒間に何が起きたか」をフレーム単位で詳細に記述させるのは、動画全体を入力とする方式では難しくなります。

**ドメイン知識の注入に限界がある**

現在のバージョンのAmazon Nova Proでは、動画解析時にシステムプロンプトとユーザープロンプトを指定できます。しかし、業界固有のタグ体系（例：様々なシーンでの交通タグ）に基づく判定や、専門的な観点での時系列分析を求めると、汎用モデルでは期待する精度が得られないことがあります。

## 解決策の方向性：時系列静止画 × Claude

本記事では、動画を短いクリップに分割し、フレームを短いスパン（例えば1秒以下）で抽出してClaudeに複数フレームを渡して解析するアプローチを提案します。静止画1枚だけでは判断できない瞬時のモーションを捉えつつ、動画解析モデルでは得られない粒度の高い解析を実行できます。

以下では、このアプローチを含めたAmazon Bedrockで利用可能な動画解析アプローチをQCDの観点で比較します。

## 動画解析アプローチのQCD比較

Amazon Bedrockで利用可能な動画解析アプローチを、Quality（品質）・Cost（コスト）・Delivery（構築容易性）の観点で比較します。

| 観点 | Amazon Nova Pro | TwelveLabs Pegasus | Claude × 時系列静止画 |
| --- | --- | --- | --- |
| フレーム制御 | モデル内部で自動サンプリング。開発者は制御不可 | Pegasus 1.5 TBM: `min_segment_duration`で最小2秒単位のセグメント指定が可能（※Bedrock未提供、TwelveLabs API直接のみ）。Pegasus 1.2 (Bedrock): セグメント機能なし | ffmpegで0.1秒間隔など任意の粒度で抽出可能 |
| Bedrock対応 | ○ | Pegasus 1.2のみ（TBM機能なし、最小動画長4秒） | ○ |
| プロンプト設計 | システムプロンプト＋ユーザープロンプトを指定可能 | セグメント定義のdescriptionで指示（`prompt`パラメータは併用不可） | システムプロンプトで専門家ペルソナ、ユーザープロンプトでタグ体系・出力形式を自由に設計 |
| ドメイン特化精度 | 汎用的な要約・質問応答向き | セグメント定義で構造化メタデータを取得可能。ドメイン特化精度は要検証 | プロンプト次第で高い精度が期待できる |
| コスト（1分あたり） | 低い | 低い（〜$0.03） | 高い（〜$0.50） |
| 構築容易性 | API 1回で完結 | API 1回で完結 | ffmpeg + Lambda + Step Functionsの構築が必要 |

ADASのパーセプションMLモデル開発のように、カットイン（車線への割り込み）や歩行者の飛び出しを検出するためのアノテーションでは、0.1秒相当のフレームで確認することが求められます。このような高粒度の解析が求められるユースケースでは、フレーム選択を開発者が制御できるClaudeアプローチが品質という点で優位性があります。

Pegasus 1.5 TBMの最小セグメント粒度は2秒であり、ADASのパーセプションMLモデル開発のように0.1〜0.3秒単位でフレームを確認する必要があるユースケースには対応できません。また、Bedrock上ではPegasus 1.2のみ利用可能で、TBM（Time Based Metadata）機能は提供されていません（2025年5月時点）。そのため、高粒度かつBedrock統合が必要なユースケースでは、Pegasusは選択肢から外れます。

一方、コストや構築容易性を重視する場合はNova ProやPegasus TBMが適しています。Pegasus 1.5のTime Based Metadata Extraction（TBM）はセグメント定義で構造化メタデータを取得できる有力な選択肢ですが、Bedrock経由では利用できない点に注意が必要です（2026年現時点においては）。

要件次第ではありますが、**コストが安い順（Nova Pro → Pegasus TBM → Claude）にベンチマークを実施**し、求める精度に達した時点でそのアプローチを採用する進め方を推奨します。

## 解決策：時系列静止画 × Claudeによるドメイン特化分析

上記の比較を踏まえ、本記事ではフレーム制御とプロンプト設計の自由度を重視し、以下の3ステップで動画を解析します。

1. 動画を短いクリップに分割する
2. 各クリップからフレームを静止画として抽出する（間隔・枚数は開発者が制御）
3. 静止画群をClaudeに時系列で渡し、ドメイン固有のプロンプトで解析させる

このアプローチの利点は次のとおりです。

* **フレーム選択の完全な制御**：ffmpegで抽出間隔と枚数を自由に設定できる
* **解析粒度の向上**：短いクリップ単位で解析するため、フレームごとの変化を詳細に記述できる
* **プロンプトの自由度**：Claudeのシステムプロンプトで専門家ペルソナを設定し、ユーザープロンプトで業界固有のタグ体系や出力フォーマットを細かく指定できる
* **モデル選択の柔軟性**：動画解析専用モデルに縛られず、画像理解に優れたモデルを選択できる

## 技術ポイント①：Claudeに複数の静止画を渡して解析する

Claude Sonnet 4はConverse APIで複数の画像を1リクエストに含められます。これを利用して、時系列に並んだフレーム群を「疑似的な動画」としてClaudeに渡します。

```
import boto3
import json

bedrock = boto3.client("bedrock-runtime", region_name="ap-northeast-1")

# フレーム画像をbase64エンコードしてリストに格納
content = []
for i, frame_bytes in enumerate(frames):  # framesはbytes型のリスト
    content.append({
        "image": {
            "format": "jpeg",
            "source": {"bytes": frame_bytes}
        }
    })

content.append({
    "text": """以下のフレーム群は時系列順に並んだ動画のスナップショットです。
交通安全の観点から詳細に分析し、JSON形式で回答してください。

出力フォーマット:
{
  "tags": {"歩行者": "yes/no/null", "夜間": "yes/no/null", ...},
  "description": "3000文字程度の詳細な時系列分析"
}"""
})

response = bedrock.converse(
    modelId="us.anthropic.claude-sonnet-4-20250514-v1:0",
    system=[{
        "text": "あなたは交通安全分析のエキスパートです。"
                "フレームごとの変化、時系列の流れ、周囲の状況を網羅的に記述してください。"
    }],
    messages=[{"role": "user", "content": content}],
    inferenceConfig={"maxTokens": 4096}
)
```

ポイントは以下の3点です。

* **システムプロンプトで専門家ペルソナを設定**：「交通安全分析のエキスパート」として振る舞わせることで、ドメイン固有の観点が解析に反映される
* **画像を時系列順に並べる**：Claudeはコンテンツの順序を認識するため、フレームの並び順がそのまま時間軸として解釈される
* **出力フォーマットを明示**：タグ判定と詳細説明文をJSON形式で返させる。`description`フィールドがそのまま後段のベクトル化対象になるため、検索精度に直結する

ドラレコ映像の例では、様々なシーンの交通タグ（「先行車両の車線変更」「歩行者」「夜間」など）について`yes/no/null`で判定させつつ、3000文字程度の詳細な時系列分析を生成させています。

## 技術ポイント②：Lambda Layerでffmpegを使ったフレーム抽出

Lambda上でffmpegを使うには、ffmpegバイナリをLambda Layerとして追加します。

### クリップ分割

長い動画を短いクリップに分割する際、`-c copy`オプションで再エンコードなしの高速分割を行います。

```
import subprocess

def split_video(input_path, output_dir, clip_duration):
    """動画をclip_duration秒ごとに分割"""
    cmd = [
        "/opt/bin/ffmpeg",  # Lambda Layerのパス
        "-i", input_path,
        "-c", "copy",
        "-f", "segment",
        "-segment_time", str(clip_duration),
        "-reset_timestamps", "1",
        f"{output_dir}/clip_%03d.mp4"
    ]
    subprocess.run(cmd, check=True)
```

### フレーム抽出

各クリップから一定間隔でフレームをJPEG画像として抽出します。

```
def extract_frames(clip_path, interval=0.5, max_frames=20):
    """clip_pathからinterval秒間隔でフレームを抽出"""
    cmd = [
        "/opt/bin/ffmpeg",
        "-i", clip_path,
        "-vf", f"fps=1/{interval}",
        "-frames:v", str(max_frames),
        "-f", "image2pipe",
        "-vcodec", "mjpeg",
        "pipe:1"
    ]
    result = subprocess.run(cmd, capture_output=True, check=True)
    # JPEG画像のバイナリを分割して返す
    return split_jpeg_stream(result.stdout)
```

クリップ長は`interval × max_frames`で自動算出されます。`interval=0.5`秒、`max_frames=20`なら10秒クリップです。この値を調整することで、解析の粒度とコストのバランスを制御できます。

## 技術ポイント③：Amazon S3 Vectorsによる低コストなセマンティック検索

Claudeが生成した解析テキストをベクトル化し、自然言語で検索できるようにします。ベクトル検索には[Amazon S3 Vectors](https://docs.aws.amazon.com/ja_jp/AmazonS3/latest/userguide/s3-vectors.html)を使いました。

S3 Vectors はS3上にベクトルインデックスを作成し、ベクトルの保存・類似検索ができるサービスです。OpenSearchやPineconeのような常時稼働のベクトルDBが不要で、S3の料金体系で使えるため、小〜中規模のユースケースではコスト面で有利です。

### ベクトルの保存

```
s3vectors = boto3.client("s3vectors", region_name="ap-northeast-1")

s3vectors.put_vectors(
    vectorBucketName=bucket_name,
    indexName=index_name,
    vectors=[{
        "key": clip_key,
        "data": {"float32": embedding},  # 1024次元
        "metadata": {
            "source": analysis_key,
            "tags": ["歩行者", "夜間"]
        }
    }]
)
```

### ベクトル検索

```
response = s3vectors.query_vectors(
    vectorBucketName=bucket_name,
    indexName=index_name,
    queryVector={"float32": query_embedding},
    topK=20,
    returnMetadata=True,
    returnDistance=True,
    filter={"$and": [{"tags": {"$eq": tag}} for tag in selected_tags]}
)
```

メタデータにタグを格納しておくことで、セマンティック検索とタグフィルタを組み合わせた検索が可能です。

### セマンティック検索の特性と実用上の注意

S3 Vectorsは常に`topK`件を返します。関連性が低くても結果が返るため、cosine distanceによるスコアリングが重要です。

実際のデータでは距離が0.8〜0.9に集中する傾向がありました。セマンティック検索単体では関連度の差が出にくいため、タグ検索をメイン、セマンティック検索を補助的に使う設計が実用的でした。

### Terraformでの構築

S3 VectorsはまだTerraform AWS Providerでサポートされていないため、`null_resource` + `local-exec`でAWS CLIを実行しています。

```
resource "null_resource" "s3_vectors_bucket" {
  provisioner "local-exec" {
    command = <<-EOT
      aws s3vectors create-vector-bucket \
        --vector-bucket-name ${local.s3_vectors_bucket_name} \
        --region ${var.aws_region} \
        2>/dev/null || echo "Vector bucket already exists"
    EOT
  }

  provisioner "local-exec" {
    when    = destroy
    command = <<-EOT
      aws s3vectors delete-vector-bucket \
        --vector-bucket-name ${self.triggers.bucket_name} \
        --region ${self.triggers.region} \
        2>/dev/null || echo "Vector bucket deletion failed"
    EOT
  }

  triggers = {
    bucket_name = local.s3_vectors_bucket_name
    region      = var.aws_region
  }
}
```

`when = destroy`のprovisionerも定義しておくことで、`terraform destroy`時にベクトルバケットも削除されます。

## ドラレコ映像解析での適用例

上記の技術ポイントを組み合わせて、ドラレコ映像の自動解析・検索システムを構築しました。

![アーキテクチャ図](https://static.zenn.studio/user-upload/deployed-images/1bbcad521137aa5cb6b21e39.png?sha=460fd185c7e59868c49f907688aed4646c9a5dea)  
*全体アーキテクチャ*

### 処理パイプライン

```
S3 uploads/ → EventBridge → Step Functions
                                  ↓
                          SplitVideo Lambda（ffmpegでクリップ分割）
                                  ↓
                          Map State（最大3並列）
                                  ↓
                          ProcessClip Lambda
                            - ffmpegでフレーム抽出
                            - Claude Sonnet 4で解析
                            - Amazon Titan Text Embeddings V2でベクトル化
                            - S3 Vectorsに保存
```

S3にドラレコ映像をアップロードすると、EventBridgeがStep Functionsを起動し、分割・解析・ベクトル化が自動で行われます。Step FunctionsのMap Stateで並列処理しますが、Amazon Bedrockのスロットリングを考慮して並列数を3に制限しています。

### 技術スタック

| レイヤー | 技術 |
| --- | --- |
| IaC | Terraform |
| バックエンド | AWS Lambda (Python 3.13), AWS Step Functions, Amazon EventBridge, Amazon API Gateway |
| AI/ML | Claude Sonnet 4（映像解析）, Amazon Titan Text Embeddings V2（ベクトル化） |
| ベクトル検索 | Amazon S3 Vectors |
| フロントエンド | Next.js 16（静的エクスポート）, CloudFront + S3 |

## 考察

### コスト

S3 Vectorsは、従来のOpenSearchやAurora PostgreSQL（pgvector）と比較してコストを約10分の1に抑えられます。ただし、高度なフィルタリングや低レイテンシが求められるユースケースでは従来のベクトルDBが適しています。S3 Vectorsはコスト優先で、検索頻度が低く、サブ秒レベルのレイテンシで許容できるワークロードに向いています。

### 他ドメインへの応用

本記事ではドラレコ映像を題材にしましたが、このアプローチは他のドメインにも応用できます。

* **製造ライン**：製品の外観検査、組立工程の異常検知
* **医療**：内視鏡映像の所見分類、手術映像のフェーズ認識
* **小売**：店舗内の顧客行動分析、棚の在庫状況モニタリング
* **セキュリティ**：監視カメラ映像の不審行動検知

いずれも、業界固有のタグ体系や分析観点をプロンプトに定義することで、汎用モデルでは得られない精度の解析が期待できます。

## まとめ

動画をドメイン固有の文脈で詳細に解析したい場合、動画を丸ごとLLMに渡す方式ではフレーム選択や解析粒度の制御が難しいという課題があります。本記事では、動画を短いクリップに分割し、ffmpegで抽出した静止画群をClaudeに時系列で渡すアプローチを紹介しました。

* **Claudeへの複数画像入力**で、ドメイン固有のプロンプトによる詳細な時系列分析が可能
* **Lambda Layer上のffmpeg**で、フレーム抽出間隔と枚数を開発者が完全に制御
* **Amazon S3 Vectors**で、常時稼働のベクトルDBなしに低コストでセマンティック検索を実現
