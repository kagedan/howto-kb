## はじめに

プログラミングを本格的に学んだことのない自分が、AI（Claude と Codex）と一緒に、デスクトップ向けのタイピング練習アプリ「速・打（そくだ）」を作りました。この記事はその制作の記録です。完成品の紹介というより、「どこで詰まって、どう抜けたか」を残すことを目的にしています。同じように AI と何か作ってみたい人の道しるべになれば嬉しいです。

速・打は、ひらがなを見てローマ字で入力するタイピング練習アプリです。短文・中文・長文などのモードを選び、KPM・正確さ・ミス傾向をローカルに記録します。サーバーや外部アカウントは使いません。

## 実録データ

* 制作期間: 基本の部分は2日ほどでほぼ形になりました。そのあとは自分で毎日使いながら、何日もかけて少しずつ改善しています（2026-05-31 着手・現在も継続中）
* 概算費用: ほぼ0円
  + 大前提として、Claude と Codex のサブスク（各 月約3,000円）を常用しています。この開発でそれ以外の追加費用はかかっていません。
* 使った AI / 道具: Claude、Codex、Electron、React、Babel
* 配布形態: インストール不要のブラウザ版と、ソースから起動するデスクトップ版（Electron・Windows / Mac / Linux 共通）。ビルド済みインストーラーは未提供

## なぜ作ったか

タイピング練習を、静かに・記録を手元に残しながら・広告や外部アカウント無しでやりたい、というのが動機でした。よくできたタイピング練習サイトは他の方々がすでにたくさん公開されています。そこへ「オフラインのデスクトップアプリとして、記録が自分の手元だけに残る」という形を一つ足してみたかった、というのが出発点です。

## 全体像 ― ブラウザ試作からデスクトップアプリへ

最初はブラウザ上の試作でした。React を CDN から読み、JSX を `type="text/babel"` でその場で変換して動かす、手軽な構成です。これを「サーバー無しで起動する1つのデスクトップアプリ」にしたい、と考えました。

最終的な構成はこうなりました。

* 画面は React（`src/` に JSX）
* `build.js` が `src/` から、Electron が読み込む `renderer/` を生成
* `main.js`（Electron のメインプロセス）が `renderer/index.html` を開く
* 記録はブラウザ互換のローカルストレージ（サーバー・外部アカウント無し）

## 詰まったところと、抜け方

### 1. JSX の事前コンパイルで起きたグローバル衝突

ブラウザ試作では、JSX は実行時に変換していました（`type="text/babel"`）。この方式では各ファイルが eval 実行で互いに隔離され、共有はすべて `window` 経由でした。

デスクトップ版では実行時変換をやめ、`build.js` で JSX を Babel によって事前コンパイルし、ふつうの `<script>` として読み込む形にしました。ところがこれで、それまで隔離されていた各ファイルの top-level の `const` / `function` がグローバルで衝突するようになりました。たとえば `app.jsx` の `const { Home } = window` と、`Home.jsx` の `function Home` がぶつかります。

抜け方は、コンパイル後の各ファイルを IIFE（即時実行関数）で包み、元の「ファイルごとに隔離される」挙動を再現することでした。

```
// build.js より（抜粋）
const result = babel.transformSync(code, {
  filename: f,
  presets: [['@babel/preset-react', { runtime: 'classic' }]],
});
// 事前コンパイルすると top-level が衝突するので IIFE で包んで隔離を再現する
const wrapped = `(function () {\n${result.code}\n})();\n`;
```

この包み方ひとつで、事前コンパイルしてもブラウザ試作と同じ隔離が保てるようになりました。「動く仕組みの前提（ここでは eval による隔離）は、方式を変えると一緒に失われる」というのを実感した箇所です。

### 2. 完全オフラインで動かす

「実行時に外部ネットワークへ依存しない」を目標にしました。`build.js` のビルド時に、次を行っています。

* React / ReactDOM を `renderer/vendor/` に同梱する
* Google Fonts の CSS とフォント本体をダウンロードしてローカル化する
* 生成する `index.html` の CSP を `default-src 'self'` に絞り、すべてローカルの `<script src="...">` で読む

結果として、初回ビルドのときだけネットが必要で、ビルド後のアプリ本体は `file://` だけで完結してオフラインで動きます。

### 3. GPU の取り合いでアプリが落ちる

このPCでは、Electron の GPU プロセス起動が失敗してアプリが落ちることがありました（`error_code=1002` → GPU FATAL）。原因は、同じPCで動かしている別のプログラムと GPU を取り合っていたことでした。

タイピングのUIに GPU アクセラレーションは要りません。そこで `main.js` で明示的にソフトウェア描画へ固定し、GPU プロセスに依存しないようにして解決しました。

```
// main.js より（抜粋）
app.disableHardwareAcceleration();
app.commandLine.appendSwitch('disable-gpu');
app.commandLine.appendSwitch('disable-gpu-compositing');
```

「必要のない機能は最初から使わない」に倒すと、環境依存の不安定さをまるごと避けられることがある、という学びでした。

### 4. 公開リポジトリの整理（clean import）

元の作業フォルダには `node_modules/` や、ビルドで生成される `renderer/`・`dist/` が入っていました。公開にあたっては、これらを `.gitignore` で除外し、「再生成できる source だけ」を管理する形にしました。誰が clone しても `npm install && npm run build` で同じものが作れる、という状態を公開の条件にしています。

## ブラウザでも遊べるようにした ― 配布をどうするか

最初は「ソース公開だけ」で始めました。デスクトップアプリを配るとなると、ふつうは OS ごとのビルド済みインストーラー（Windows の `.exe`、Mac の `.dmg` など）を用意します。ただ、署名なしのアプリを配ると、ダウンロードした人の OS が「開発元を確認できません」といった警告を出します。これを消すには有料の署名証明書が要り、しかも OS ごとに別々のビルド（Mac 版は Mac 実機）が必要です。ほぼ0円で作りたい趣味の道具には、これは重すぎました。

そこで気づいたのは、速・打の画面（`renderer/`）は、実は Electron 固有の API を一切使っていない、ふつうの Web アプリだということでした。ネットワークもサーバーも要らず、記録はブラウザのローカルストレージに入るだけ。Electron の `main.js` は、その Web 画面を開く「窓枠」でしかありません。

ということは、ビルドした `renderer/` をそのまま Web ページとして置けば、誰でもインストールなしにブラウザで遊べます。記録は遊ぶ人それぞれのブラウザの中だけに残り、こちらのサーバーには何も送られません。実際にポートフォリオサイトへ web 版として置きました。

デスクトップで使いたい人のために、ソースからの起動も残しています。Electron は Windows / Mac / Linux 共通なので、`npm install && npm run build && npm start` はどの OS でも動きます（同梱の `launch.sh` は Linux / Mac 向けの起動ヘルパーで、Windows では `npm start` を使います）。「まずブラウザで手軽に、気に入ればソースからデスクトップ版」という二段構えにしました。

「必要のない前提（インストール・OS ごとの署名）は、そもそも要らない形に寄せられないか」を考えると、配布のハードルがまるごと消えることがある、という学びでした。

## AI の使い分け

* **Claude**: UI の試作と、改善方針の相談。「どんな画面構成にするか」「この詰まりをどう切り分けるか」を一緒に考える相手として使いました。
* **Codex**: Electron 化、公開用リポジトリの整理、README の作成など、手を動かす実装を任せました。

大まかに「相談・設計は Claude、実装・整形は Codex」という分担で進めています。

## おわりに

正直なところ、自分はアプリ開発の知識がまったくなく、Electron も React も知りませんでした。今でも全然わかっていません。それでも、ざっくりとした理解を少しずつ進めていって、セキュリティにさえ気をつけていれば、AI が急速に進化したこの時代に、自分の作りたいアプリを作ることは驚くほど簡単でした。

詰まった点はどれも、AI に丸投げして一発で解けたわけではなく、「どこが原因か」を一緒に切り分けながら少しずつ抜けていったものです。それでも、専門知識の穴を AI が埋めてくれるおかげで、ここまで形にできました。
