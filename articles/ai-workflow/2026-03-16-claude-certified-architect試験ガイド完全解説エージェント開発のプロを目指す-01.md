Anthropicから、Claudeを活用した高度なエージェントシステム開発のスキルを証明する公式資格\*\*「Claude Certified Architect – Foundations」\*\*の試験ガイドが公開されました。

この記事では、試験ガイドから読み取れる「Claudeネイティブなアーキテクチャ」の核心部分を図解を交えて徹底解説します。

---

# 1. 試験の概要：何が問われるのか？

この試験は、単なるプロンプトエンジニアリングの知識だけでなく、\*\*「現実世界の複雑な課題をClaudeを使ってどう解決するか」\*\*という実践的な設計能力を問うものです。

## 出題ドメインと比率

試験は以下の5つのドメインで構成されています。

| ドメイン | 概要 | 比率 |
| --- | --- | --- |
| **1. Agentic Architecture** | エージェントループ、マルチエージェント設計 | 27% |
| **2. Tool Design & MCP** | MCPサーバー統合、ツールインターフェース設計 | 18% |
| **3. Claude Code & Workflows** | Claude Codeの設定、CLAUDE.md、CI/CD連携 | 20% |
| **4. Prompt Engineering** | 構造化出力、Few-shot、JSONスキーマ | 20% |
| **5. Context Management** | コンテキストウィンドウ管理、信頼性、エスカレーション | 15% |

---

# 2. ドメイン別の重要キーワード解説

## Domain 1: Agentic Architecture & Orchestration (27%)

エージェントシステムの心臓部です。特に**ハブ＆スポーク型のマルチエージェント設計**が重視されています。

### コーディネーター・サブエージェントパターン

コーディネーター（親）がタスクを分解し、専門のサブエージェント（子）に委譲する流れを理解する必要があります。

---

## Domain 2: Tool Design & MCP Integration (18%)

Model Context Protocol (MCP) を通じた外部システムとの連携能力です。

* **ツールの説明（Description）**: LLMがツールを選択する唯一の手がかり。境界条件やエラー時の挙動を詳細に書くことが求められます。
* **MCPの階層**:
  + `.mcp.json`: プロジェクト共有（リポジトリ管理）
  + `~/.claude.json`: ユーザー個人用
* **構造化エラー**: `isError: true` フラグとともに、再試行可能かどうか（`isRetryable`）をモデルに伝える設計が重要です。

---

## Domain 3: Claude Code Configuration (20%)

ターミナル専用ツール「Claude Code」を使いこなす能力です。

* **CLAUDE.md**: プロジェクト固有のルールを定義。`@import` によるモジュール化や、`.claude/rules/` によるパス固有の制約が鍵となります。
* **計画（Plan）モード vs 直接実行**:
  + **Planモード**: 複数ファイルに及ぶ変更、大規模リファクタリング用。
  + **直接実行**: 単一ファイルのバグ修正、単純なタスク用。

---

## Domain 4: Prompt Engineering & Structured Output (20%)

出力を制御し、システムの信頼性を高める技術です。

* **JSONスキーマ**: ツール呼び出しでの構造化出力が最も推奨されます。
* **Few-shot**: 曖昧なケースでの判断基準をモデルに教え、ハルシネーションを抑制します。
* **Message Batches API**: 大量処理を50%コスト削減。ただしマージ前チェックなどのレイテンシ重視タスクには向きません。

---

## Domain 5: Context Management & Reliability (15%)

長い会話での精度を維持する技術です。

* **case factsブロック**: 重要な事実（顧客ID、金額等）を要約で消失させないよう、常に先頭付近に配置するテクニック。
* **エスカレーション**: LLMの信頼度スコアは「不正確」なことが多いため、明確なエスカレーション基準を数ショットで示すのが正解です。

---

# 3. サンプル問題に挑戦！

試験の難易度を体感するために、ガイド掲載のサンプル問題を1問見てみましょう。

サンプル問題：ツール選択の改善

**問題**: エージェントが `lookup_order` と `get_customer` を頻繁に呼び間違えています。説明文がどちらも最小限である場合、最初にとるべき対策は？

A) Few-shot サンプルを5〜8個追加する  
B) ツールの説明を拡張し、入力フォーマットや境界条件を詳細に書く  
C) キーワードベースのルーティングレイヤーを自作する  
D) 2つのツールを統合する

**正解: B**  
**解説**: ツールの説明文はLLMがツールを選ぶ主要なメカニズムです。Few-shotの追加よりも、説明文の充実が「低労力・高効果」な最初のステップとなります。

# 4. DataBlizzardの分析：この資格の価値とは？

この試験ガイドを読み解くと、Anthropicが「単なる開発者」ではなく、\*\*「システムの挙動に責任を持てるアーキテクト」\*\*を求めていることが分かります。

* **「確率論」から「決定論」へのブリッジ**: サンプル問題にもある通り、重要な処理ではプロンプトによる指示（確率論）ではなく、プログラムによる前提条件のチェック（決定論）が推奨されています。この「LLMに任せる部分」と「コードで固める部分」の切り分けこそが、プロの腕の見せ所です。
* **MCPの真価**: 単なる外部ツール連携ではなく、LLMがシステムの構造（スキーマやリソース）を理解するための「セマンティックなインターフェース」としてMCPを位置づけています。
* **運用を考慮した設計**: `case facts` ブロックや構造化エラーなど、本番環境での長期運用における「コンテキストの劣化」や「エラーからの回復」に焦点が当てられています。

# 5. 学習のススメ：実践的な準備

この試験は、実機での演習が必須です。ガイドの内容を深く理解するために、私たちは以下のステップを推奨します。

1. **MCPサーバーの構築**: 既存のAPI（GitHub等）をMCP化し、Claude Codeから操作してみる。
2. **マルチエージェントの実装**: Cord[エージェント] -> Sub[エージェント] の委譲フローを Agent SDK で実装し、コンテキストがどう引き継がれるか（または引き継がれないか）を検証する。
3. **CLAUDE.md の最適化**: 自分のプロジェクトに `.claude/rules/` を導入し、パスごとのコーディング規約を自動適用させる。
