# プロンプトエンジニアリング実践ガイド：Claude/ChatGPTから最高の出力を引き出す技術

「AIに頼んでも思った通りの結果が出ない」という経験はありませんか？問題の9割はプロンプトの書き方にあります。この記事では、実際の業務で使えるプロンプトパターンをビフォー/アフターで解説します。

---

## 基本原則：AIに「文脈」を与える

AIは文脈がないと平均的な回答しか返しません。

```
# NG
メールを書いて

# OK
あなたはB2Bソフトウェア企業の営業担当です。
以下の状況でフォローアップメールを書いてください。

状況：
- 先週、初回提案ミーティングを実施した
- 先方担当者: 田中部長（IT部門責任者）
- 提案内容: 社内ナレッジ管理ツールの導入
- 先方の反応: 興味はあるが、コストと導入工数が懸念
- 目的: 次回デモの日程調整

条件：
- 200字以内
- 懸念点に軽く触れつつ前向きなトーン
- 件名も含める
```

## テクニック1：ロールプロンプティング

AIに役割を与えると専門性が上がります。

```
あなたは15年のキャリアを持つシニアデータアナリストです。
統計の素人にも理解できるように、以下のデータの傾向を説明してください。

[データ]
月,売上,顧客数
1月,1,250,000,45
2月,980,000,38
3月,1,580,000,52
```

**ロール例一覧：**

| 目的 | 効果的なロール |
| --- | --- |
| コードレビュー | 「Googleのシニアエンジニア」 |
| 文章校正 | 「出版社の敏腕編集者」 |
| 法律確認 | 「企業法務の専門家（ただし最終確認は弁護士へ）」 |
| マーケ文章 | 「コピーライティング歴20年のプロ」 |
| ビジネス分析 | 「元McKinseyのコンサルタント」 |

## テクニック2：出力フォーマットを指定する

フォーマットを明示すると、そのまま使える出力が返ってきます。

```
以下の会議メモから議事録を作成してください。

[メモ]
田中：新機能のリリースは来月末を予定
佐藤：テスト工数が足りないかも、2週間必要
田中：じゃあ中旬に延ばす。告知は田中が担当

出力形式：
## 決定事項
- （箇条書き）

## アクションアイテム
| タスク | 担当 | 期限 |
|--------|------|------|

## 次回確認事項
- （箇条書き）
```

**出力フォーマット指定の例：**

```
# JSON形式で欲しい場合
以下を分析して、JSONで出力してください：
{
  "sentiment": "positive/negative/neutral",
  "key_points": ["ポイント1", "ポイント2"],
  "action_required": true/false
}

# Markdown表形式
比較表をMarkdownのテーブルで出力してください。

# 番号付きリスト
手順を番号付きリストで、各ステップは50字以内で。
```

## テクニック3：Chain of Thought（思考の連鎖）

複雑な問題は「考える手順」を示すと精度が上がります。

```
# NG（直接答えを求める）
この施策は成功しますか？

# OK（思考プロセスを指示）
以下の新規事業案を評価してください。

評価の手順：
1. まずターゲット市場の規模を推定してください
2. 次に主要な競合と差別化ポイントを分析してください
3. 収益化モデルの実現可能性を検討してください
4. リスクを3つ挙げてください
5. 最後に総合評価（5段階）と理由を述べてください

[事業案]
ビジネスパーソン向けのAI議事録自動生成サービス。
月額2,980円。Zoom/Teams/Google Meetと連携。
```

## テクニック4：Few-Shot（例示）

例を見せると出力スタイルが安定します。

```
商品レビューをポジティブ/ネガティブ/ニュートラルに分類してください。

例：
入力: 「配送が早くて助かりました！商品も期待通りです」
出力: ポジティブ

入力: 「説明書が分かりにくく、設定に2時間かかった」
出力: ネガティブ

入力: 「普通の商品です。特に不満はありません」
出力: ニュートラル

では以下を分類してください：
入力: 「値段の割にはまあまあですが、耐久性が心配です」
出力:
```

## テクニック5：制約を明示する

「やらないこと」を指定すると出力がブレなくなります。

```
以下の技術仕様書の要約を作成してください。

制約：
- 500字以内
- 技術的な専門用語は使わない（非エンジニア向け）
- 箇条書きのみ使用（文章は書かない）
- コスト・期間には言及しない
- 現状の問題点ではなく、解決後のメリットにフォーカス

[仕様書テキスト]
...
```

## 業務別：すぐ使えるプロンプトテンプレート集

### テンプレート①：提案資料のアウトライン作成

```
あなたは経験豊富なコンサルタントです。
以下の情報をもとに、提案資料のアウトラインを作成してください。

提案先: {顧客の業種・規模}
提案内容: {サービス・製品名}
顧客の課題: {具体的な課題}
我々の強み: {競合との差別化ポイント}
目標: {この提案で達成したいこと}

アウトライン形式：
1. エグゼクティブサマリー（1スライド）
2. 課題の整理（2-3スライド）
3. 解決策の提案（3-4スライド）
4. 導入効果・ROI（2スライド）
5. 実施スケジュール（1スライド）
6. 投資対効果・料金（1スライド）
7. 次のステップ（1スライド）

各セクションに含めるべき具体的なコンテンツを箇条書きで示してください。
```

### テンプレート②：コードレビューコメントの生成

```
あなたはGoogleのシニアエンジニアです。
以下のコードをレビューして、改善点をコメントしてください。

レビューの観点：
1. バグ・潜在的なエラー
2. パフォーマンス
3. 可読性・保守性
4. セキュリティ

コメントの形式：
- 深刻度（Critical/Major/Minor）を明示
- 問題のある行番号を指定
- 改善後のコードを提示
- 理由を1行で説明

[コード]
{コードをここに貼る}
```

### テンプレート③：週次進捗レポートの自動生成

```
以下の箇条書きメモをもとに、上司向けの週次進捗レポートを作成してください。

今週のメモ：
{やったことを箇条書きで雑に書く}

レポートの形式：
## 今週の成果（3点以内）

## 課題・リスク（あれば）

## 来週の計画（3点以内）

条件：
- 全体で300字以内
- ポジティブなトーンを維持
- 数値があれば必ず含める
- 受け身の表現（〜されました）は使わない
```

### テンプレート④：競合分析

```
{競合他社名} と {自社サービス名} を比較分析してください。

分析軸：
1. 価格帯
2. 主要機能
3. ターゲット顧客
4. 強み
5. 弱み
6. 市場ポジション

出力：
- Markdown形式の比較表
- 各項目に対する5行以内の考察
- 最後に「自社が勝てる領域」を3点

注意：公開情報のみをベースにし、推測は「（推定）」と明示してください。
```

## Claude特有のテクニック

### XMLタグで構造化する

ClaudeはXMLタグで囲むと指示と入力データを明確に区別できます。

```
以下のドキュメントを要約してください。

<document>
{ここに長い文書を貼る}
</document>

<instructions>
- 要約は200字以内
- 箇条書き3点
- 専門用語は避ける
</instructions>

<output_format>
## 要約
- 
- 
- 
</output_format>
```

### 長い処理はステップ分割

```
これは複数ステップのタスクです。各ステップを完了したら「次のステップに進む準備ができました」と言ってください。

ステップ1：以下の契約書を読んで、重要な条項を抽出してください。
[契約書テキスト]
```

## よくある失敗パターンと修正

| 失敗パターン | 原因 | 修正方法 |
| --- | --- | --- |
| 出力が長すぎる | 文字数を指定していない | 「〇〇字以内で」を追加 |
| 抽象的すぎる | 具体例がない | Few-Shotで例を3つ示す |
| 毎回結果が違う | 条件が曖昧 | 判断基準を明示する |
| 余計な前置きが多い | フォーマット未指定 | 「前置き不要、本文のみ出力」を追加 |
| 嘘をつく | 知識の限界を超えている | 「わからない場合は『不明』と答えて」を追加 |
