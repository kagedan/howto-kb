# Mastraをデプロイする様々な方法

Mastraを本番環境にデプロイする方法がいくつか存在します。今回は代表的な構成をいくつか紹介します。

# **1. スタンドアロンサーバー構成**

![](https://static.zenn.studio/user-upload/c36f5d856514-20260326.png)

**特徴:**

* MastraをHTTPサーバーとして起動
* フロントエンドは `@mastra/client-js` でAPI呼び出し
* Mastra StudioはStudio（開発/テスト用）で、本番はAPI経由

**解説:**

Mastraは、`mastra build`をすると**Honoを搭載したMastraサーバー**が起動するようになっています。これを任意のプラットフォームにデプロイ可能です。HonoとはTypeScript/JavaScript向けの軽量・高速なWebフレームワークで、ルーティングやミドルウェアの仕組みがあります。

最も代表的なMastraの利用方法は、この**Mastraサーバーを起動してバックエンドとし、フロントエンドのアプリからAPIで呼び出す**構成でしょう。フロントエンドは上図のようにReact/Next.js が代表的ですが、もちろんVue.jsでも何でも、APIさえ呼べればよいのでHTML＋ピュアJavaScriptでも動きます。

ただし、Mastraサーバー自体はアプリケーションロジックに集中しているので、**認証・認可やレート制限は外側で担保する必要**があります。API Gateway、Nginx、Cloudflare、WAFなどでトークン検証・レート制限をするのが最も一般的でしょう。（Mastra ServerはHono上に載っているため、ミドルウェアで認証・レート制限を追加する方法も不可能ではありません）

Mastraは、「**MastraClient SDK**」を公開しており、公式SDKを使うと型安全・ストリーミング・認証ヘッダーなどが簡単になります。もちろん、SDKは必須ではなく、SDKなしでもREST APIを直接呼べれば Mastraの利用は可能です。

**実装例:**  
以下の実装例はクライアント側からMastraのHTTP APIを呼ぶ例です。MastraClient SDKを使ってMastraのAPIを簡単・安全に呼んでいます。

```
import { MastraClient } from "@mastra/client-js";

const client = new MastraClient({
  baseUrl: "<https://your-production-domain.com>",
  headers: { "Authorization": "Bearer YOUR_TOKEN" }
});

const response = await client.getAgent("weatherAgent").stream("天気は？");
```

# **2. Next.js統合構成 (フルスタック)**

![](https://static.zenn.studio/user-upload/d984dadb45a6-20260326.png)

**特徴:**

* Next.js App RouterのAPI Routesで `handleChatStream()` を使用
* サーバーレスフレンドリー
* フロントエンドとバックエンドが同じプロジェクト

**解説:**

この構成はNext.js前提です。「**Next.jsのAPI RoutesにMastraを組み込む**」というのがポイントで、純ReactやVueだけではこの構成になりません。※ 公式ドキュメントの[こちらのページ](https://mastra.ai/docs/deployment/web-framework)ではNext.js/Astroのみが明記されていてNuxtは未記載ですが、[Guides & Migrations](https://mastra.ai/guides/getting-started/nuxt)にはほぼ同じ考え方でNuxtでの実現方法も紹介されています。

Next.jsは「Reactで作るWebアプリに、サーバー機能も一緒に持たせられるフルスタックフレームワーク」です。普通のReactはブラウザ側の画面だけ作りますが、Next.jsは同じプロジェクト内にAPI（サーバー側の処理）も作れるのが特徴です。

この構成での通信イメージは次の通りです。

* Frontend（画面）はブラウザで動く
* 画面から「質問」などを送ると、Next.jsのAPI Routes（同じアプリ内のサーバー側）が受け取る
* API Routesの中でMastraが動き、AIエージェントを実行
* 結果をAPI Routesが返し、Frontendが表示する

つまり、「画面 → 同じアプリ内のAPI → Mastra → 画面」という流れで、別のMastraサーバーを立てなくても1つのNext.jsアプリ内で完結するのがこの構成です。

この構成の場合、認証・認可／レート制限はNext.js側で担保する考え方になります。

* Next.jsのAPI Routes（＝Mastraが動く入口）に認証・認可とレート制限を入れるのが基本です。例えば、ミドルウェアでトークン検証、APIルート内でユーザー判定と制限をします。
* もっとも、Vercel/Cloudflareなどのエッジ/ゲートウェイ側で制限する方法も一般的です。
* いずれにせよ、独立したMastra ServerではなくNext.jsアプリ内に組み込む構成なので、「Mastra Serverの外側」＝「Next.jsの前段/内側」で守る、という整理になります

**実装例:**

こちらの実装例は、Next.jsのAPI Routes（App Routerのroute.ts）側の実装です。つまり Next.jsアプリ内のサーバー側で、Mastraを呼び出してストリーム応答を返す処理になります。

```
// app/api/chat/route.ts
import { handleChatStream } from '@mastra/ai-sdk';
import { createUIMessageStreamResponse } from 'ai';
import { mastra } from '@/src/mastra';

export async function POST(req: Request) {
  const params = await req.json();
  const stream = await handleChatStream({
    mastra,
    agentId: 'weatherAgent',
    params,
  });
  return createUIMessageStreamResponse({ stream });
}
```

# **3. アダプター統合構成 (Express/Fastify)**

![](https://static.zenn.studio/user-upload/d164aa7cb049-20260326.png)

**特徴:**

* `@mastra/express` や `@mastra/fastify` アダプターを使用
* 既存のExpressアプリに統合
* カスタムミドルウェア・認証と組み合わせ可能

**解説:**

これは、**Express/FastifyアプリにMastraを**「**組み込む**」構成です。Mastraを独立サーバーとして立てるのではなく、APIサーバー内にMastraのハンドラを同居させます。

Mastra HandlersからMastraアプリを呼ぶ方法はHTTPではなく（つまりAPI呼び出しではなく）、**関数を直接呼ぶイメージ**です。Mastraは「アプリ内の一機能」として動きます。

既存のExpress/Fastifyアプリの中にMastraをマウントされるので、共通の認証・ログ・レート制限・CORSなどを同じミドルウェアで一元管理できます。

すでにExpress/FastifyでAPI基盤があったり、認証・監査・ログ・レート制限などを既存方式で統一したい、1つのサーバーで「既存API + AI機能」をまとめたい、というニーズがある場合に採用を検討するとよいでしょう。

**実装例:**

こちらの実装例は、Mastra Handlers（Adapter）の部分です。

```
// Express本体
import express from 'express';
// ExpressにMastraを組み込むためのアダプター（MastraServer）を読み込み
import { MastraServer } from '@mastra/express';
// Mastraアプリ（エージェント設定など）を読み込み
import { mastra } from './mastra';

// Expressアプリのインスタンスを作成
const app = express();
// JSONリクエストボディをパースできるようにする
app.use(express.json());

// ExpressアプリにMastraを組み込み（Mastraのハンドラをアプリに登録する準備）
const server = new MastraServer({ app, mastra });
// Mastraのルートやハンドラを実際に初期化して登録
await server.init();

// --- サンプル: エージェントを呼ぶAPIを追加 ---
app.post('/chat', async (req, res) => {
  // ここでは仮のエージェントIDを使った例
  const agent = mastra.getAgent('weatherAgent');
  const result = await agent.generate({
    messages: [
      { role: 'user', content: req.body.message ?? '天気は？' },
    ],
  });
  return res.json(result);
});
// --- ここまでサンプル ---

// Expressアプリをポート4111で起動
app.listen(4111);
```

# **4. サーバーレス構成 (AWS Lambda/Vercel)**

![](https://static.zenn.studio/user-upload/96af0f3d64b2-20260326.png)

**特徴:**

* Lambda Web Adapterを使ってコンテナ化
* Vercel/Netlifyにもデプロイ可能
* 自動スケーリング、従量課金

**解説:**

この構成は、AWS Lambda にMastraをデプロイし、フロントエンドから LambdaのFunction URLにHTTPリクエストを送ります。Lambda自体がMastra Serverとして振る舞い、FrontendはそのHTTP APIを呼ぶ、ということです。案１のMastra Server（案1と同じ中身）をコンテナ化してLambdaで動かしている、と考えてよいです。

違いとしては、Lambda特有のメリデメが入る、ということです。

**メリット**は以下の通りです。

* **自動スケールが簡単**
* **インフラ構築・運用が軽い**
  + サーバを構築・運用する手間が省けるのが最大のメリット

**デメリット**は以下の通りです。

* Lambdaは**最大15分で強制終了**
  + 長時間の推論、重いRAG、複数ステップのワークフローには不向き
  + 常駐プロセスが持てないのでスケジュール処理や常時監視のような用途は苦手
  + ストリーミングも制約あり（実質「短いストリーム」に限られる）
* **永続ディスクは原則なし**（一時領域のみ）
  + 起動ごとにキャッシュやインデックスが消える
  + RAGインデックスやファイル保存は外部ストレージ必須（S3等）
  + 重いファイルI/Oは遅くなりがち
* **コールドスタートが発生**
  + 特にAI依存ライブラリや大きなモデル初期化が重い
  + 「最初の1回が遅い」問題が出やすい

# **5. Mastra Cloud構成 (マネージド)**

**特徴:**

* Mastra Cloudがビルド・デプロイ・ホスティングを管理
* GitHubリポジトリと連携、自動デプロイ
* ストレージ・観測性が統合済み

**解説:**  
こちらはMastraが公式に運営する「Mastra Cloud」へのデプロイです。  
「Mastra Cloud」については、**別途解説記事**を掲載します。

**利用方法:**

1. Mastra Cloudでプロジェクトを作成
2. リポジトリを接続
3. "Enable Deployments"で自動デプロイ開始
4. 生成されたドメインでAPI利用

# **まとめ比較表**

| 構成 | 用途 | メリット | 適したケース |
| --- | --- | --- | --- |
| スタンドアロン | 独立したAIサービス | シンプル、型安全なクライアントSDK | マイクロサービス、社内API |
| Next.js統合 | フルスタックアプリ | SSR/SSG対応、サーバーレス | WebアプリケーションのAI機能 |
| Express/Fastify | 既存システム統合 | 既存コードベースに追加しやすい | レガシーシステムのAI化 |
| サーバーレス | スケーラブルなAPI | 自動スケーリング、低コスト | 変動的なトラフィック、低頻度利用 |
| Mastra Cloud | マネージド環境 | 運用不要、簡単デプロイ | 迅速なプロトタイピング、スタートアップ |
