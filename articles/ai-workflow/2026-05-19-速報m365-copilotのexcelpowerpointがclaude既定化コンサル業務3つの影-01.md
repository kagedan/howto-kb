2026年5月4日、Microsoftが M365 Copilot の Excel と PowerPoint の既定モデルを Anthropic Claude に切り替えた。同社のメッセージセンター通知 MC1269241 で、商用クラウドの大多数の地域で「自動オン」となっており、Word も2026年夏に追従予定。本記事では、コンサル現場の視点で何が変わるのか、3つの実務インパクトと、いま打つべき初動対応を整理する。

## 1. M365 CopilotのClaude既定化とは？リリース概要

Microsoft が4月初旬に予告し、5月4日に発効した変更は、M365 Copilot の Researcher、Agent Mode、document agent の既定モデルを OpenAI 系から Anthropic Claude に切り替えるもの。対象アプリは Excel と PowerPoint で、Word は2026年夏。EU・EFTA・UK テナントは既定オフで残るため、それ以外の地域は管理者が明示的にオフにしない限り、もう Claude が動いている。

なお Anthropic は Microsoft Foundry にも Claude Opus 4.5／Sonnet 4.5／Haiku 4.5 を投入済みで、M365 Copilot だけでなく Copilot Studio、Power Platform、Researcher エージェント全体に展開している。

## 2. 何が変わったか：Excel・PowerPointの3つの新挙動

### Excel：Researcher と Agent Mode が Claude 経由に

セル参照を含む自然言語クエリに対する回答が、論理ステップを明示的に列挙する傾向が強くなった。複数シート横断の集計依頼で、Claude は「どのシートのどのセル範囲を見たか」をフットノートで返す比率が上がっている。

### PowerPoint：document agent が Claude

スライド構成の提案や、長文の議事録から要約スライドを生成するタスクで、Claude のアウトプットはアクションタイトル化（結論を主語に置く形）に寄りやすい。

### Word：2026年夏に順次切替

Word は同年夏に追従予定。提案書や報告書のドラフト生成は段階的に Claude に切り替わっていく。

## 3. コンサル現場でどう活きるか：3つの具体シーン

ひとつ目は、月次レビュー資料。Excel Researcher が出してくる差異分析サマリーで、引用元セルが明示されるため、レビュー時のファクトチェックが速くなる。

ふたつ目は、議事録→スライド変換。PowerPoint document agent が、議事録テキストから「論点／結論／次アクション」の3スロット構成スライドを返しやすくなった。

みっつ目は、競合分析シート。Excel Agent Mode で「この比較表を読み取って、勝ち筋・負け筋を3点ずつ」と聞くと、Claude は「データから言える範囲」と「言えない範囲」を分けて返す傾向がある。

## 4. 独自フレーム：「マルチモデル時代の実務3層構造」

私はこの局面を3層に分けて整理している。

第1層「モデル無意識層」：エンドユーザーは、裏で何のモデルが動いているかを意識せず使う。今回 Excel と PowerPoint で起きたのはまさにこれ。

第2層「モデル指定層」：チームリーダーや上級メンバーが、タスクに応じて「これは Claude」「これは GPT」と明示的に振り分ける。

第3層「モデル監査層」：管理者が、モデル別の挙動・精度・コストを比較計測し、運用ガイドラインに反映する。

2025年は「どのモデルを選ぶか」が論点だったが、2026年は「モデルを選び続けない設計」が論点に変わる。

## 5. クライアント現場の具体ケース：4日で挙動が変わった

中堅製造業 E 社（売上320億円・営業企画15名）で M365 Copilot を全社展開していた最中、4日で Excel の既定モデルが Claude に切り替わった。月次レポート更新フローでは Researcher の引用元フットノート精度が上がり、レビュー時間が30%短縮。一方で Excel Agent Mode の VBA マクロ提案では、出力フォーマットの差が混乱を生んだため、社内 Wiki に「Claude のときはこう聞く」プロンプトガイドを1週間で整備した。

意外なのは、ユーザーから「速くなった」より「説明が丁寧になった」というフィードバックの方が多かったこと。ハルシネーション体感の減少が、AI への信頼感を地味に底上げしていた。

## 6. 反直感的な視点：マルチモデル化はむしろリスクを下げる

「複数モデルを混ぜると業務が混乱する」と言われがちだが、実際は逆だ。単一モデル前提で運用してきた組織ほど、モデルの癖が業務プロセスに溶け込んでしまい、ベンダー乗り換え時に総取っ替えが必要になる。今回 Microsoft がやったのは、製造業の「マルチサプライヤー戦略」と同じ構造で、単一サプライヤー依存リスクの分散にあたる。

## 7. 日本テナントの初動：3つの確認ポイント

日本テナントは商用クラウド扱いなので、原則「既定オン」群。管理者は Microsoft 365 管理センター → Copilot → Settings → AI providers operating as Microsoft subprocessors で状態を確認できる。

第1に、現状の有効状態（オン／オフ／管理者ロール）を確認する。第2に、Claude 利用時の UI 表示（モデルバッジ）の有無を社内に周知する。第3に、Excel・PowerPoint 経由で扱う情報の機密区分を再点検する。

## 8. よくある疑問（Q&A）

### Q. 既存のプロンプト資産は使えなくなる？

A. ほぼ流用可能。ただし「ステップを箇条書きで返して」のように出力形式を強く指定するプロンプトは、Claude の方が指示への追従が高めに出る傾向がある。微調整の手間は少ない。

### Q. データはどこに行く？

A. Microsoft のテナント境界内で処理され、Anthropic 側は Microsoft の subprocessor として規定されている。詳細は Microsoft Learn の subprocessor ページで確認できる。

### Q. オフに戻せる？

A. 管理者ロール（AI Administrator）からオフへ切り替え可能。ただしオフにすると Researcher・Agent Mode・document agent の挙動が元の OpenAI 系に戻る。

### Q. 日本語精度に違いはある？

A. 公開ベンチは限定的だが、現場体感では「議事録→要約」「数値→自然言語」のタスクで Claude の方が落ち着いた敬体に寄りやすい。

## 9. まとめ

M365 Copilot の Claude 既定化は、表面的には「裏のモデルが切り替わっただけ」だが、構造的にはマルチモデル時代の到来を、エンタープライズ標準ツールが正式に追認した出来事だ。コンサル現場が打つ初動は、(1) 自社・クライアント先テナントの設定確認、(2) プロンプト資産の小規模検証、(3) チームメンバーへの周知、の3点。「モデルを意識しないで使えるか」が、これからの導入成功条件になる。
