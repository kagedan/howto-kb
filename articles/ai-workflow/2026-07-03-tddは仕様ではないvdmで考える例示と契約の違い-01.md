## はじめに

TDDは強力な開発手法です。

失敗するテストを書き、最小の実装で通し、リファクタリングする。このサイクルは、実装を小さく進め、設計をフィードバックで育てるうえで非常に有効です。

ただし、ここで混同してはいけないことがあります。

**テストは仕様の例示ではありますが、仕様そのものではありません。**

VDMのような形式手法を考えると、この違いがはっきり見えてきます。

## TDDでは、例から始める

たとえば `sort` 関数をTDDで作るとします。

```
test('空列はそのまま', () => {
  expect(sort([])).toEqual([]);
});

test('1要素はそのまま', () => {
  expect(sort([5])).toEqual([5]);
});

test('整列済みは変わらない', () => {
  expect(sort([1, 2, 3])).toEqual([1, 2, 3]);
});

test('逆順を並べ替える', () => {
  expect(sort([3, 2, 1])).toEqual([1, 2, 3]);
});

test('重複を保つ', () => {
  expect(sort([3, 1, 3])).toEqual([1, 3, 3]);
});

test('負数も扱える', () => {
  expect(sort([-1, 5, -10])).toEqual([-10, -1, 5]);
});
```

これらのテストは、読みやすいです。

`sort([3, 1, 3])` が `[1, 3, 3]` になる。これは、形式仕様を知らない人にも直感的に理解できます。

TDDの強さはここにあります。  
例は、人間にとって圧倒的に読みやすい。

## しかし、テストは有限個の点でしかない

では、次の実装はどうでしょうか。

```
function sort(l: number[]) {
  const r = [...l].sort((a, b) => a - b);
  return r.length <= 3 ? r : r.slice(0, 3);
}
```

この実装は、先ほどの6つのテストをすべて通ります。

しかし明らかに、正しいソートではありません。4要素以上の配列を渡すと、結果が3要素に切り詰められてしまいます。

つまり、テストが固定しているのは有限個の入力と出力の対応だけです。

一方で、私たちが本当に言いたかった仕様は、おそらく次のようなものです。

```
sort: seq of int ==> seq of int
sort(xs) == ...
post IsOrdered(RESULT) and IsPermutation(RESULT, xs);
```

これは特定の例ではなく、すべての入力に対して次を要求しています。

* 結果は順序づけられている
* 結果は元の列の順列である
* 要素が消えたり増えたりしない

TDDのテストは、この仕様のいくつかの例を示しているにすぎません。

## 実装者は、テストから「真の仕様」を推察している

現実の開発者は、先ほどのような切り詰め実装を普通は書きません。

なぜなら、テストだけを見ているのではなく、その背後にある意図を読んでいるからです。

`sort` という関数名を見る。  
空列、1要素、逆順、重複、負数というテストケースを見る。  
そこから「これは一般的なソートを実装してほしいのだろう」と推察する。

つまりTDDでは、テストの書き手と実装者のあいだに、文化・常識・文脈の共有があることを前提にしています。

人間同士のチームでは、この前提はかなりうまく働きます。

しかし、これは明示された仕様ではありません。  
あくまで、読み手が復元している仕様です。

## VDMで重要なのは「仕様を明示する」こと

VDMでは、関数や操作の性質を事前条件、事後条件、不変条件として明示します。

たとえばソートなら、「いくつかの例で期待通りに動く」ではなく、「任意の入力に対して、結果は整列済みであり、元の要素を保存している」と書きます。

ここで大事なのは、VDMが単にテストを増やす道具ではないということです。

VDMには実行可能な仕様を書き、テストやトレースで動作確認できる面があります。  
一方で、VDMの本質は、仕様を数学的に明示し、その仕様から証明責務を導けるところにあります。

つまり、次の2つは別物です。

* 仕様を実行してテストできる
* 仕様や設計が性質を満たすことを証明する

「VDMは仕様をテストできる」という説明だけを強調すると、TDDの延長のように見えてしまいます。

しかし、VDMがTDDと決定的に違うのは、有限個の例ではなく、全称的な性質を仕様として扱う点です。

## TDDとVDMは対立しない

ここまでの話は、TDDを否定するものではありません。

TDDには明確な価値があります。

| 観点 | TDDの例示 | VDMの形式仕様 |
| --- | --- | --- |
| 読みやすさ | 高い | 慣れが必要 |
| 固定できる範囲 | 有限個の例 | すべての入力 |
| 意図の伝達 | 文化や常識に依存 | 記号の定義に依存 |
| フィードバック | 実装に対して速い | 仕様・設計段階から得られる |
| 検証 | 点ごとの確認 | 証明責務や全称的な検証につながる |

TDDは、実装のフィードバックループとして優れています。

VDMは、その前段で「何が正しいのか」を明示するために使えます。

実践的には、次のように組み合わせるのが自然です。

1. VDMで重要な仕様を明示する
2. 仕様から境界条件や不変条件を洗い出す
3. それをもとに実装テストを書く
4. 実装後も、テストと仕様の両方でズレを検出する

TDDは「例で進める」ための手法です。  
VDMは「性質を固定する」ための手法です。

役割が違います。

## AIエージェント開発では、この違いがより重要になる

人間同士の開発では、テストから意図を読むことができます。

しかし、AIエージェントがコードを書く状況では、この前提は弱くなります。

モデルが違う。  
バージョンが違う。  
学習している文脈が違う。  
同じテストを見ても、同じ意図を復元するとは限りません。

テストは有限個の点しか見ていないため、意図のズレはテストの隙間をすり抜けます。

一方で、形式仕様は「読み手の推察」ではなく「契約」として機能します。

もちろん、形式仕様を書けばすべてが自動的に正しくなるわけではありません。  
仕様そのものが間違っていれば、間違った契約を正確に実装してしまいます。

それでも、少なくとも何を正しさとみなすのかを明示できます。

これは、複数のAIエージェントが協調して開発する時代には重要です。

## まとめ

TDDは、例示によって実装を導く強力な手法です。

しかし、テストは仕様そのものではありません。  
テストは有限個の例であり、その背後にある真の仕様は、読み手が文脈から推察しています。

VDMの価値は、ここを明示することにあります。

* TDDは、例を通じて実装を育てる
* VDMは、性質を仕様として固定する
* VDMで仕様をテストできることと、仕様を証明できることは別である
* 形式仕様は、推察ではなく契約として意図を共有する

人間中心の開発では、例示と常識の共有で多くの場面を乗り切れました。

しかし、AIエージェントが実装を担う比率が上がるほど、「何を正しいとするのか」を明示する重要性は増していきます。

TDDからVDMへ、という単純な置き換えではありません。

例示によるフィードバックと、形式仕様による契約。  
この2つを役割分担して使うことが、これからの開発では重要になるはずです。
