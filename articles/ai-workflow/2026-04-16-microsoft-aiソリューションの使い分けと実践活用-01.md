## はじめに

皆さんこんにちは！私はNTT DATAにて、Copilot StudioやPower Platformを活用したAIエージェントの普及・展開を支援しています。実際の現場では、「どのMicrosoft AIソリューションを選択すべきか」という相談を受ける機会が増えてきました。

Microsoftは非常に幅広いAIソリューションを提供している一方で、それぞれの役割が分かりにくく、結果として過剰設計やミスマッチが発生するケースも少なくありません。

本記事では、主に以下の3つのソリューションを対象に整理します。  
・Microsoft 365 Copilot  
・Copilot Studio  
・Microsoft Foundry

そして後半では、これらのエージェントをどのように管理していくべきかという観点で、Agent365の考え方についても触れます。

## Microsoft AIソリューションとは

MicrosoftのAIは、スキルレベルと用途に応じて3つのレイヤーで整理できます。これらは競合するものではなく、役割分担として理解することが重要です。

図：Microsoft AIソリューション全体像  
![](https://static.zenn.studio/user-upload/8e80612e90de-20260318.png)

出典：<https://jp.ext.hp.com/techdevice/ai/ai_explained_13/>  
※注釈：一部製品のロゴは図が作成された時点のものであり、現在のロゴとは異なります。

**●ノーコード：Microsoft 365 Copilot**

Microsoft 365 Copilotは、Word、Excel、Teamsなどに統合された生成AIであり、ユーザーはプロンプトを入力するだけで業務を効率化できます。

例えば、会議の要約、提案書のドラフト作成、Excelデータの分析など、日常業務の多くが対象になります。  
既存業務の延長線で活用できるため、最も導入ハードルが低く、最初に検討すべきレイヤーです。

**●ローコード：Copilot Studio**

Copilot Studioは、業務特化型のAIエージェントを構築するためのプラットフォームです。

単なる生成AIの活用にとどまらず、ナレッジや業務フローを組み込むことで、問い合わせ対応や業務支援といった「業務そのもの」を担うエージェントを構築できます。

Power Automateと組み合わせることで、AIによる判断と業務プロセスの自動化を一体化できる点が特徴です。

**●プロコード：Microsoft Foundry**

Microsoft Foundryは、より高度なAIシステムを構築するための開発基盤です。

RAG構成によるナレッジ検索、複数エージェントの連携、基幹システムとの統合など、エンタープライズ要件に対応した柔軟な設計が可能です。

一方で、設計・開発・運用の難易度は高くなるため、適用範囲の見極めが重要になります。

## Microsoft AIソリューションの使い分け

これらのソリューションを使い分ける上で重要なのは、「機能比較」ではなく「技術選定の順序」です。

●技術選定の考え方

実務では、以下の順序で検討することを推奨します。

図：意思決定フロー  
![](https://static.zenn.studio/user-upload/c0f2f6b8cc3a-20260416.png)

まずはMicrosoft 365 Copilotで実現できないかを検討します。多くのユースケースは、個人業務の効率化という形でこのレイヤーで完結します。ここをスキップすると、不要に複雑な構成になりやすい点に注意が必要です。

次に、複数ユーザーでの利用や業務プロセスとの統合が必要な場合は、Copilot Studioを検討します。この段階では、AIは単なる補助ではなく、業務の一部として機能するようになります。

それでも要件を満たせない場合、初めてMicrosoft Foundryによる開発を選択します。ここでは柔軟な設計が可能になる一方で、コストや運用負荷も増えるため、全体の構成や運用負荷を考慮し、段階的な選択肢の一つとして整理しておくことが重要です。

このように「上から順に検討する」ことで、スピードとガバナンスの両立が可能になります。

## エージェントを管理していくためのAgent365活用

AIエージェントの活用が進むと、新たな課題として「管理」が顕在化します。

エージェントが部門ごとに乱立したり、参照するデータが統制されていなかったりすると、セキュリティや品質の問題につながります。

この課題に対する考え方が、Microsoft 365を基盤としたAgent365です。

図：Agent365イメージ  
![](https://static.zenn.studio/user-upload/81fb82ea908a-20260318.png)  
出典：Microsoft Agent 365: The control plane for AI agents | Microsoft 365 Blog

Agent365では、エージェントを個別に管理するのではなく、既存のMicrosoft 365基盤と統合して管理します。

Entra IDによる認証・権限管理、Purviewによるデータガバナンス、そして利用状況の可視化を組み合わせることで、安全かつ持続可能な運用が可能になります。

エージェントは作って終わりではなく、継続的に改善していく対象であるため、初期段階から管理を前提とした設計が重要です。

## まとめ

MicrosoftのAIソリューションは、ノーコード・ローコード・プロコードというレイヤーで整理すると理解しやすくなります。

```
・Microsoft 365 Copilot：個人の生産性向上
・Copilot Studio：業務への組み込み
・Microsoft Foundry：高度なシステム化
```

そして、これらを活用する上で最も重要なポイントは次の通りです。

まずCopilotで検討し、必要に応じてStudio、Foundryへと段階的に拡張する

AI活用はPoCの段階を終え、実運用のフェーズに移行しています。  
今後は「どのツールを使うか」ではなく、「どのレイヤーで設計するか」が重要になります。

本記事が、実務におけるMicrosoft AIソリューション利活用の一助になれば幸いです。
