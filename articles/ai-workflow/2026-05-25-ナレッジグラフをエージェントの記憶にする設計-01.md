# ナレッジグラフをエージェントの「記憶」にする設計

> この記事は独立して読めます（約10分）
>
> この記事はオントロジー設計の技術詳細です。意思決定者の方は「[ツールを100個並べてもAIエージェントは賢くならない](https://zenn.dev/knowledge_graph/articles/kg-agent-skill-layer)」「[AIエージェントを本番に出せない本当の理由](https://zenn.dev/knowledge_graph/articles/kg-agent-production-safety)」に先に進んでも理解できます。

「[AIエージェントが毎回データを取りに行く設計の限界](https://zenn.dev/knowledge_graph/articles/kg-agent-memory-first-design)」では、scatter-gather問題の設計回答として「事前統合型（Memory-first）」アーキテクチャを示しました。ナレッジグラフを「推論が始まる前にデータが揃っている状態」の基盤として使う設計です。

しかし「ナレッジグラフを使う」と決めた後に、多くのエンジニアが2つの壁にぶつかります。本記事ではこの2つを順番に解説します。

---

## 問題：KGはデータを入れるだけでは「記憶」にならない

**壁その1 - スキーマ設計**: ノードとエッジをどう定義すればLLMが推論しやすいか分からない

Neo4jやAmazon Neptuneを立ち上げ、既存データベースからデータを流し込む。技術的にはすぐできます。しかしこの状態のKGはLLMにとって「使いやすい記憶」にはなっていません。

`user_id: 10045` というノードと `account_cd: ACC-003` というノードが別々に存在していて、それが同じ会社を指すとLLMには分からない。カラム名が内部コードのままで意味が読み取れない。関係性のエッジがなく、孤立したノードが並んでいるだけ。こういう状態では、KGに問い合わせても意味のある回答は返ってきません。

**壁その2 - 名寄せ**: 「Acme Corp」「Acme」「acme.com」が同一か別か、自動で判断できない

複数システムからデータを統合すると、同じ実体が別の名前・別のIDで存在します。この問題を放置するとKGのノードが重複し、エージェントは「Acme Corpに関する情報」と「Acmeに関する情報」を別物として処理します。

この2つを解決しないと、KGはただのデータダンプになります。どれだけ多くのデータを入れても、エージェントの推論精度は上がりません。

## ナレッジグラフはどう貢献するか

### オントロジー設計：データ構造ではなく意味構造を作る

KGのスキーマ設計で最初に意識すべき転換があります。**「データをどう格納するか」ではなく「ドメインの意味構造をどう表現するか」で設計する**ことです。

テーブル設計の感覚でKGを作ると、`user_tbl`ノードと`account_master`ノードができます。LLMがクエリを生成するとき、このノード名から「これは顧客のことだ」と解釈するのは困難です。業務語彙に合わせた命名（`顧客`、`チケット`、`商品`など）にするだけで、LLMが生成するCypherクエリの精度が上がります。

**設計原則1：ドメインの「概念」をノード、「関係」をエッジにする**

* ノード = 業務上の意味を持つ実体（顧客、商品、担当者、チケット…）
* エッジ = 実体間の動詞的な関係（担当する、購入した、起票した、所属する…）

エッジの向きと名前が、エージェントの推論経路を決めます。`担当する`と`担当された`は別のクエリパスを開きます。

**設計原則2：推論に必要な関係性だけを持つ**

全てのデータをKGに入れる必要はありません。「このKGで何を推論させたいか」から逆算して、必要なノードとエッジだけを設計します。最初は3-5ノード、数本のエッジで十分です。

### 3つのドメインでの例示

#### ECサイトのオントロジー

「この顧客が最近よくレビューしているカテゴリの新着商品を教えて」というエージェントの推論が、このグラフ構造があれば1クエリで取得できます。

#### 開発チームのオントロジー

「先週のスプリントで誰がどのIssueをクローズしたか」「このリポジトリに最も貢献している開発者は誰か」といった推論がクエリ1本になります。

#### カスタマーサポートのオントロジー

「この顧客から来たチケットと同じ問題を過去に報告した別の顧客は？」「その問題の対応担当者は誰か？」という推論がこの構造から可能になります。

### 「どこまで細かく設計するか」の判断基準

スキーマ設計で詰まる原因の多くは「完璧なオントロジーを最初から作ろうとすること」です。

判断基準はシンプルです。**「このエッジがなければ答えられない推論はあるか？」** という問いで各エッジの必要性を評価します。答えがYesなら入れる。Noなら省く。

最初から全てのエンティティと関係をカバーしようとすると、KGが複雑になりすぎてLLMが使いにくくなります。3-5ノード、数本のエッジから始めて、エージェントの使われ方を見ながら徐々に拡張するのが現実的な進め方です。

---

### Entity Resolution（名寄せ）：同一エンティティを同一ノードに統合する

スキーマ設計の次に立ちはだかる問題がEntity Resolutionです。

複数システムから同じ実体が異なる名前・IDで流れ込んできます。

| システム | 識別子 | 表記 |
| --- | --- | --- |
| Salesforce | ACC-10045 | Acme Corporation |
| Zendesk | org\_887 | Acme Corp |
| kintone | 取引先\_234 | 株式会社Acme |

これら3つが同一企業であることをKGが認識できなければ、エージェントは「Acme Corporationに関連する全チケット」を正確に返せません。

#### 同一エンティティを識別するための属性セット

名寄せには「どの属性の組み合わせで同一性を判断するか」を定義することが必要です。

* **確定マッチング**: ドメイン（acme.com）が完全一致 → 同一エンティティと断定
* **ファジーマッチング**: 社名の表記ゆれ（Acme / Acme Corp / 株式会社Acme）を類似度スコアで評価 → 閾値を超えたら候補として扱う
* **複合マッチング**: 社名が類似かつ電話番号が一致 → 同一と判断

#### 「分からない場合」の設計

名寄せで最も見落とされるのが「判断できないケース」の扱いです。

類似度スコアが閾値ギリギリで判断できない場合、2つの選択肢があります。

* **保留ノード**: 仮に別エンティティとして扱い、レビューキューに積む
* **人間レビューフロー**: 担当者が確認して統合・分離を決定し、その結果をKGに反映する

「分からないものを強制的に統合する」設計は危険です。別会社のAcmeが同一顧客として扱われ、無関係な会社の情報が推論に混入します。

#### 推論時データ収集型（Model-first）でも名寄せは解決しない

「MCPやFunction Callingで各システムからその場でデータを取ってくれば、LLMがつなぎ合わせてくれるのでは？」という考え方があります。しかしこの設計でも名寄せの問題は消えません。

SalesforceとZendeskを別々に叩いて取得した結果に「Acme Corporation」と「Acme Corp」が混在していたとき、LLMはその場で同一性を判断しなければなりません。判断の根拠となるデータ（ドメイン情報・電話番号・契約ID）がコンテキストに含まれていれば確認できますが、含まれていなければLLMは「おそらく同じ会社だろう」と推測します。これがハルシネーションの温床です。

名寄せは推論のたびに解決する問題ではなく、データを統合するときに一度解決しておく問題です。事前にEntity Resolutionが済んでいるKGであれば、エージェントは「Acme」というノードを参照するだけで済み、同一性の推測をLLMに委ねる必要がなくなります。

#### 名寄せの誤りがエージェントに与える影響

Entity Resolutionの失敗はKGの上で静かに広がります。別ノードにしてしまった場合、エージェントは「Acme Corporationに関連する全チケット」を正確に返せません。同一ノードに誤って統合した場合、関係のない企業のデータが推論に混入します。

スキーマがどれだけ綺麗でも、名寄せが壊れていればKGの推論精度は上がりません。この順番が重要で、**スキーマ設計が先、名寄せは後**です。良いスキーマがあって初めて、何を同一と見なすべきかの判断基準が決まります。

**設計上の帰結**: Entity Resolutionが事前に済んでいるKGでは、エージェントの回答が名寄せミスで揺れることがなくなります。同じ質問をすれば同じノードに辿り着き、同じ回答が返る。この安定性は「推論のたびにLLMが同一性を推測する設計」では得られません。

### ナレッジグラフだけでは不十分な領域

* **入力データの品質**: Entity Resolutionの精度は、ソースデータの品質に直接依存します。スキーマが良くても、ソースデータが「Acme」「acme」「ACME」と揺れていれば名寄せの精度は落ちます。データ品質の管理はKGの外側の問題です
* **フィードバックループ**: エージェントが誤回答を返したとき、それが名寄せの失敗なのかスキーマ設計の問題なのかを検知し、KGを改善に戻す仕組みが必要です。この継続的なフィードバックループはKG単体では実装できません
* **スキーマのバージョン管理**: ビジネスの変化に合わせてオントロジーを更新するとき、既存のクエリや既存のエッジが壊れないように管理する仕組みが必要です

これらのうち、権限制御・監査・ロールバックへの応用は[AIエージェントを本番に出せない本当の理由](https://zenn.dev/knowledge_graph/articles/kg-agent-production-safety)で詳しく扱っています。

## 参照実装：DevRev Computer のオントロジー設計

DevRev Computerでは「プロダクトオントロジー」として、カスタマーサポートドメイン向けに事前定義済みのセマンティックモデルを持っています。上記の例3（顧客→チケット→既知の問題→Feature→Developer）に相当する関係が設計済みで、即利用可能です。

ただし、**これはDevRevのドメインに特化した1実装です**。ECサイトや製造業、金融機関では異なるオントロジーが必要になります。

DevRevの設計で参照実装として興味深い点は、6種類のクエリエンジンの統合です。

| クエリエンジン | 用途 |
| --- | --- |
| Graph DB | エンティティ間の関係走査 |
| Search | 全文検索・セマンティック検索 |
| SQL | 集計・フィルタリング |
| Time Series | 時系列データの分析 |
| Data Warehouse | 大規模分析クエリ |
| Workflow | 自動化・ステータス管理 |

「関係性を辿る」クエリはGraph DBで処理し、「最近1週間のチケット数」はTime Seriesで処理する。この使い分けを1つのMemoryレイヤで統一することで、エージェントは「どのエンジンに投げるか」を意識しなくて済みます。

Entity Resolutionについては、AirSyncによる同期時に自動実行される設計になっており、個別実装が不要です。[AIエージェントが毎回データを取りに行く設計の限界](https://zenn.dev/knowledge_graph/articles/kg-agent-memory-first-design)で触れた同期インフラが、KGの品質を担保するための仕組みの一部として機能しています。

## 今日から始める1ステップ

次にやるべきことは1つです。**Mermaidで自分のドメインのオントロジーを3-5ノードで書いてみる**。

以下は最小例です。あなたの業務に合わせて言葉を入れ替えてください。

書いてみたとき、「このエッジはどちら向きが自然か」「このノードは業務でどう呼ばれているか」という問いが自然に出てきます。その問いこそがオントロジー設計の本質的な作業です。

Entity Resolutionは最初は省略して構いません。まずノードとエッジの定義に集中し、「このKGで何を推論させたいか」を具体化することが先です。
