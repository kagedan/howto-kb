「メモリを毎ステップ読み書きする」。7月7日に公開された論文 *Memory in the Loop* は、言語エージェントのこの設計を正面から扱っている。これが意味するのは、RAGの検索精度を上げる話より先に、検索をどの実行経路に置くかを決めないと、エージェント全体の待ち時間が壊れるということだ。

論文では、ネットワーク越しのストアは数十〜数百ミリ秒かかりうるため、重い検索をループ内に入れると、エンドツーエンドの時間が最大83倍まで膨らむとしている。出典は一次資料の [arXiv:2607.05690](https://arxiv.org/abs/2607.05690)。モデルの推論より、ベクトルDBへの往復やJSONの復元待ちでログが止まる。実装していると見覚えがあるやつだ。

## 「一度検索して考える」では足りなくなる

普通のRAGは、最初に関連文書を引いて文脈として渡す。ところがツールを使うエージェントは、観測、判断、実行、結果確認を何度も回す。途中で「さっき失敗した理由」や「前に決めた条件」を参照したくなる。

メモリを周回の外に置けば、古い文脈で次の行動を選ぶ。毎周回、外部ストアへ取りに行けば遅い。検索を増やせば賢くなる、という足し算が崩れる地点だ。

メモリは、長期保存だけを担う倉庫ではない。エージェントが今この瞬間に使う作業メモでもある。CPUのキャッシュを毎命令ディスクへ読みに行かないのと同じで、近い状態を近くに置く設計が必要になる。

## 83倍を避けるには、メモリを3層に分ける

自分はこの論文を、メモリを増やす話ではなく「待ち時間の予算を層ごとに決める話」として読んだ。次の3層を分けると考えやすい。

| 層 | 置くもの | 読み出すタイミング | 許容する待ち時間 |
| --- | --- | --- | --- |
| ホット状態 | 現在の目標、直前のツール結果、失敗理由 | 毎ステップ | プロセス内で即時 |
| 作業メモリ | 今回のタスクに関係する要約・候補 | 必要なステップだけ | 小さく、上限を設ける |
| 長期記憶 | 過去案件、文書、ユーザー設定 | 周回の境目か非同期 | ネットワーク往復を許容 |

ホット状態は、実行中のプロセスに持つ。ここに置くのは検索結果の全文ではなく、次の判断に効く小さな事実だけでいい。作業メモリは、長期記憶から引いた内容を圧縮して置く場所になる。長期記憶への問い合わせは、行動の前後に必ず行う処理から外す。

この分け方をしないと、検索結果の質を少し上げるたびに、全ステップのレイテンシに課金する構造になる。推論モデルを速くしても、外部往復が支配的なら体感は変わらない。

## 検索は「毎回」ではなく、状態が変わったときに走らせる

実装の入口は、検索を必須処理から条件付き処理へ落とすことだと思う。例えば、直前のツール結果が失敗だった、目的が切り替わった、ホット状態に必要なキーがない。このどれかが起きたときだけ作業メモリを補充する。

```
def next_step(state, event):
    if event.goal_changed or event.tool_failed:
        state.working = retrieve_and_compact(state.goal)

    context = {
        "goal": state.goal,
        "last_result": state.last_result,
        "working": state.working,
    }
    return planner.decide(context)
```

この形なら、`planner.decide()` のたびに検索しない。もちろん、必要な事実を見落とす危険はある。だからキャッシュの有効期限、検索が走らなかった回数、失敗から検索までの時間はログに残したい。いつの状態で判断したかも記録しないと、原因が追えないんだよね。

別の話だが、エージェントの評価で「必要な文書を取れたか」だけを見るテストも少し危うい。待ち時間を無視した正答率は、実運用の並列数が上がった瞬間に別のシステムを測っていることになる。

## 先に測るべきなのは検索精度ではない

論文の83倍は、すべての構成で起きる値ではなく、検索が高コストなときの最大値だ。それでも設計レビューには十分強い数字だと思う。検索を1回増やす変更を出すなら、「何件当たるようになるか」と同じ表に、「1タスクで何回往復が増えるか」を置くべきだ。

まずは、1タスクあたりの検索回数、検索待ち時間の合計、検索なしで復旧できた失敗の割合を取る。ここが見えれば、検索対象を削るのか、要約をキャッシュするのか、ストアをプロセスに寄せるのかを選べる。モデル選定より前に効く計測だ。

エージェントのメモリは、たくさん覚えているほど強いわけではない。次の一手に必要な状態が、必要な瞬間に手元にあることが強さになる。今後は長期記憶の容量より、ホット状態と外部検索の境界をどう切ったかが、エージェントの実行速度を分けるはずだ。検索の精度改善を続けるほど、この境界の雑さは隠せなくなる。だから、まず1周回の待ち時間から見る。
