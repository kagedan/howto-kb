OpenAI が6月18日に出した Codex の「Record & Replay」、名前だけ見て「ああ、RPA の焼き直しね」で片付けた人が多いと思う。自分も最初そう思った。でもドキュメントを読んで考えを変えた。これは録画再生じゃない。発想としてはむしろ逆に立っている。

ついでに同じ週、VS Code チームが「GitHub Copilot のトークン効率を改善した」という地味な記事を出している。一見すると無関係だ。片方はキラキラした便利機能、片方は裏方の最適化。ところがこの2本を並べると、いまエージェント界隈で起きている一番大きな変化が見えてくる。今日はそこを実装目線で掘っていく。

## 「Record & Replay」は録画再生ではない

まず事実から。Codex アプリ v26.616（macOS 限定、Computer Use が必須、EU・UK・スイスは未提供）に入った機能で、やることは「作業を一度だけ実演して見せると、再利用できるスキルになる」。OpenAI のデモでは、YouTube 動画をサムネ・字幕・メタデータ付きでアップロードする手順を見せていた。

![Codex Record & Replay の発表(引用元: the-decoder.com)](https://static.zenn.studio/user-upload/410025ee7209-20260621.png)

ここで多くの人が「画面操作を録画して、同じ座標を再生するんでしょ」と想像する。RPA やマクロ録画のイメージだ。ところが録画している中身がそれじゃない。

OpenAI のドキュメントはこう書く。録画を止めると Codex が「キャプチャしたワークフローを点検してスキルを下書きする。スキルには、いつ使うか・どんな入力が要るか・どの手順を踏むか・結果をどう検証するか、が書かれる」。保存されるのは操作のテープじゃない。手順を言語化した仕様書なんだよね。しかも「結果をどう検証するか」まで最初から仕様に入っている。

再生時を見るともっとはっきりする。次に使うときは新しいスレッドを開いて、「今回だけ違う値」（アップロードするファイル、作る issue、レポートの日付範囲）を渡す。すると Codex はそのスキルをコンテキストとして読み、Computer Use やブラウザ操作で「いまの環境にあるツール」を使って作業を進める。毎回その場の画面に向き合って、ボタンを選び直して動く。録画した座標を叩き直すわけじゃない。

つまりこういう構造だ。実演は1回きりの「教師例」。そこからモデルが「パラメータ付きの仕様」を起こして、本番では現環境に対して再グラウンディングする。RPA が「固定された操作の再生」だとすれば、こちらは「意図の蒸留と、その場での再実行」。名前に引っ張られて録画再生だと思い込むと、設計をまるごと読み違える。

![RPA と Codex Record & Replay の違い](https://static.zenn.studio/user-upload/410025ee7209-20260621.png)

この違いは実務でかなり効く。RPA の一番の弱点は、UI が少し変わるとセレクタや座標がずれて静かに壊れることだった。再グラウンディング型はそこに強い。ボタンの位置が動いても、モデルが画面を見て「アップロードはこれ」と判断し直す。代わりに弱点が入れ替わる。毎回その場で判断する以上、挙動は完全には決まらない。同じスキルが昨日と今日で微妙に違う動きをしうる。

だから自動生成される「検証手順」が地味に効いてくる。非決定的な再実行を信用できるものに変えているのは、最後の検証ゲートなんだよね。ここを最初から持つぶん、検証を持たない素の RPA より堅牢にできる余地がある。OpenAI が「安定した手順・明確な成功条件のものに向く」「録画は短く完結させ、パスワードなどの機密は含めるな」と注意しているのも、この性質の裏返しだと読める。

## なぜ今これが出たのか：従量課金が「最適化の狙い」を変えた

表向きは「便利機能が増えました」で終わる話だ。でも一段下を掘ると、別の力学が見えてくる。

ヒントが冒頭のもう1本、Copilot のトークン効率の記事にある。書き出しがいい。「従量課金に移行したいま、エージェントセッションの1トークンすべてが効いてくる。クレジット、レイテンシ、そして残りのコンテキスト窓に。」課金が定額から使った分だけに変わった瞬間、最適化したい対象そのものが変わった、という話だ。

具体策も載っている。プロンプトキャッシュの保持時間を伸ばす（`prompt_cache_retention` を 24h に）。安定した境界に最大4つのキャッシュ区切りを置いて、Anthropic 系モデルのエージェント用途で約94%のキャッシュヒット率。ツール定義を毎回プロンプトに全部載せず、名前と説明だけの軽いメタデータにして必要なときだけ展開する（OpenAI 側でターンあたり約9〜11%、Anthropic 側で11〜18%のトークン削減）。

![ツール定義の遅延ロードによるトークン削減率](https://static.zenn.studio/user-upload/410025ee7209-20260621.png)

![実演→スキル化→再実行+検証ゲートの三層パイプライン](https://static.zenn.studio/user-upload/d2f5ff133868-20260621.png)

要は「同じ文脈を毎ターン読み直して、そのたびに課金される」のをやめにいっている。

ここで Record & Replay と線がつながる。片方は、繰り返す作業を毎回ゼロからプラン立てするのをやめる。もう片方は、同じ文脈を毎回読み直すのをやめる。狙いはまったく同じだ。一度やった思考を再利用して、二度払わない。

自分はこれを「推論の償却（amortize cognition）」と呼んでいる。定額課金の時代、賢さは「毎回どれだけ深く考えられるか」で測られた。長く考えるほど偉い。従量課金はその前提をひっくり返す。繰り返しのタスクで毎回プランを立て直すのは、ただの二重払いだ。だから向かう方向は直感と逆になる。本番では考える量をむしろ減らし、初回の思考を成果物に固めて使い回す。

プログラミングで言えば、インタプリタからコンパイラへの移動に近い。意図を毎回その場で解釈し直すのをやめて、再利用できる中間表現に一度コンパイルしておく。Record & Replay の「スキル」も、Copilot のキャッシュ済みプレフィックスも、見た目はまるで違うが、立ち位置は同じ「コンパイル済みの何か」だ。

## 自分のエージェントにどう効かせるか

ここからが本題。この設計思想は、OpenAI の機能解禁を待たなくても、自分のエージェントに今日から入れられる。3つに分けて考えている。

![実演→スキル化→再実行+検証ゲートの三層パイプライン](https://static.zenn.studio/user-upload/410025ee7209-20260621.png)

![](https://static.zenn.studio/user-upload/d2f5ff133868-20260621.png)

1つ目、トレースを捨てない。エージェントが一度こなしたタスクは、ツール呼び出しの列・触ったファイル・実行したコマンドを構造化ログとして残しておく。これが「実演」にあたる素材になる。多くの実装はここでログをだだ流しにして捨てている。正直もったいない。

2つ目、トレースから「パラメータ付きの仕様」を起こす。生のトレースをそのまま再生すると、Record & Replay が避けたはずの RPA の罠に逆戻りしてしまう。やるべきは、トレースの中で「毎回変わる値」（対象ファイル、日付、ID）を変数スロットとして抜き出し、残りを手順として固める作業だ。Codex が使っている「いつ使う／入力／手順／検証」というスキーマは、そのまま雛形に流用できる。ここはモデルに一度起こさせて、人間がレビューする形が現実的だと思う。

3つ目、再実行には「機械的に判定できる検証」を必ず付ける。ここが一番効くのに、一番手を抜かれる。非決定的な再実行を本番に出すなら、成功条件をアサーションに落とす。「ファイルが N 件アップロードされたか」「ステータスは 200 か」「行数は想定どおりか」。検証が通らなければ止める、もしくはその箇所だけモデルに考え直させる。決定的な骨格に、非決定的な穴埋めを挟み、最後をまた決定的な検証ゲートで締める。この三層が、速さと堅牢さの両取りになる。

擬似コードで書くと、こんな分担になる。

```
# 1. 実演を1回キャプチャする
trace = capture(run_agent("動画を字幕つきで公開して"))

# 2. トレース → パラメータ付きスキル(モデルに起こさせる)
skill = induce_skill(trace)
# skill = {
#   "when":   "動画を公開するとき",
#   "inputs": ["file", "thumbnail", "subtitles"],
#   "steps":  [...],                 # 変数スロット入りの手順
#   "verify": "uploaded_count == len(inputs['subtitles'])",
# }

# 3. 本番: 現環境に再グラウンディング + 検証ゲート
def replay(skill, values):
    result = run_agent(skill["steps"], values)   # 座標再生ではなく、その場で再実行
    assert check(skill["verify"], result)        # 通らなければ止める
    return result
```

肝は2つ目と3つ目の役割分担だ。スキルは決定的な「やることリスト」を渡すが、実行は毎回その場の画面に合わせる。そして検証だけはまた決定的に戻す。RPA は1つ目を座標で固めて検証を持たない。素の LLM エージェントは毎回プランから考え直して、検証も曖昧なまま流す。Record & Replay 型はちょうどその中間に陣取っている。

余談だが、自分がいま触っているエージェント用のハーネスも、ツール定義を最初に全部渡さず「名前で検索して、必要なぶんだけ後から展開する」方式に寄ってきている。実際に使うと、プロンプトのプレフィックスが安定してキャッシュが効くのが体感でわかる。Copilot の記事で読んだ最適化が、現場の手触りとぴったり重なった瞬間で、ちょっと笑った。

## で、結局どういうことか

Record & Replay を「便利な録画機能」として消費すると、たぶん一番大事なところを取りこぼす。起きているのは、エージェントの賢さの測り方が「毎回どれだけ考えるか」から「考えた結果をどれだけ使い回せるか」へ移ったことだ。そして従量課金が、その転換を後ろからぐいぐい押している。

見立てを2つ置いておく。ひとつ、これからのエージェント基盤は「実演 → スキル化 → 再実行 + 検証」を一級市民として組み込むようになる。プロンプトをその場で書く時代から、成果物をコンパイルして貯めていく時代へ寄っていく。もうひとつ、勝負どころが検証ゲートの設計に移る。非決定的な再実行を作るのは誰でもできる。それを安心して本番に流せるかは、機械的に判定できる成功条件をどれだけ丁寧に書けるかで決まる。

自分のエージェントが吐いているログ、いま捨てていないか。そこを一度見直すだけで、次の半年の作り方がけっこう変わると思う。
