# なぜ今「MCP」と「A2A」を混同したまま使ってはいけないのか!

最近、社内のDX推進の場でも「MCP対応しました」「A2Aで連携します」という言葉をよく聞くようになった。ただ、実際に話を聞いてみると、この2つがかなり曖昧に、時には同じ意味であるかのように使われているケースが多い。

結論から言うと、この2つは全く別のレイヤーの話をしている。MCPは「AIエージェントが道具をどう使うか」の規格で、A2Aは「AIエージェント同士がどうやり取りするか」の規格だ。似ているようで、守備範囲がまったく違う。  
![](https://static.zenn.studio/user-upload/4ffb3547ef82-20260713.png)

この記事では、この違いを人間の組織に置き換えて直感的に理解できるようにしたうえで、実際にTypeScriptで最小構成のMCPサーバーを動かすところまでやってみる。ビジネスサイドの人は前半だけ読めば十分だし、エンジニアの人は後半のコードまで見てもらえると、実装イメージが掴めるはずだ。

## 例え話:AIエージェントは「新しく入った専門スタッフ」である

![](https://static.zenn.studio/user-upload/749c52e3842c-20260714.png)

話を進める前に、ひとつ前提を置きたい。

AIエージェントを、優秀だけど入社したばかりの中途社員だと考えてみてほしい。前職での実力は申し分ない。文章を書かせても、コードを書かせても、驚くほど早い。ただし、この中途社員にはひとつ厄介な特性がある。**会社に入るたびに、社内のルールも、使えるツールも、他部署の連絡先も、記憶が全部リセットされた状態でスタートする**、ということだ。

この中途社員が会社で機能するためには、最低でも2つのものが必要になる。

1. 社内のツールをどう使えばいいかが書かれた「マニュアル」
2. 他部署や社外の人とどうやり取りすればいいかの「ビジネスマナーと連絡先リスト」

前者にあたるのがMCP、後者にあたるのがA2Aだ。以下、順番に見ていく。

## MCPとは「新人に渡す業務マニュアルとツールの使い方書」

![](https://static.zenn.studio/user-upload/886f9f50cca2-20260713.png)

MCP(Model Context Protocol)を一言で言うと、AIエージェントが会社の道具──社内DB、ファイルサーバー、Slack、外部API──を安全かつ統一的な手順で使えるようにするための規格だ。

人間の新人にたとえるなら、これは「業務マニュアル」そのものにあたる。マニュアルには、こう書いてある。

* このツールを使うときはこの手順を踏むこと
* このデータにアクセスできる権限とできない権限
* このフォーマットで申請すればこの結果が返ってくる

MCPが優れているのは、この「マニュアルの書式」を業界標準として統一したことだ。以前は、AIにSlackを使わせたいならSlack用の連携コードを書き、Notionを使わせたいならNotion用の連携コードを書き、というように、ツールの数だけ個別対応が必要だった。これは、会社ごとに全く違うフォーマットのマニュアルを、新人が来るたびにゼロから作っているようなものだ。

MCPは、この「マニュアルの書き方」自体を標準化した。サーバー側(ツールを提供する側)が一度MCPの作法でマニュアルを用意しておけば、どのAIエージェント(クライアント側)が来ても、同じ手順でそのツールを使いこなせる。新人研修を1回設計すれば、どの新人が来ても使い回せる状態を作った、というイメージだ。

もともとMCPは2024年11月にAnthropicが発表した仕様で、当初は「一社が管理するオープン仕様」という位置づけだった。だが2025年12月、Linux Foundationが新設したAgentic AI Foundation(AAIF)にMCPの管理が移管され、状況が変わった。一部署が使っていたローカルルールが、いつの間にか全社共通の公式マニュアルに格上げされたようなものだ。特定の1社の都合で仕様が変わる心配が薄れ、OpenAIやGoogle、Microsoftといった他社も安心して採用を進められる体制になった、というのがこの移管の実質的な意味になる。

## A2Aとは「他部署・他社の担当者とやり取りするための名刺交換とビジネスマナー」

一方、A2A(Agent-to-Agent)が扱っているのは全く別の問題だ。

MCPが「個人と道具の関係」を扱うのに対して、A2Aが扱うのは「個人と個人(あるいは組織)の関係」だ。

新人スタッフの例えを続けると、こういう場面を想像してほしい。自分の部署だけでは完結しない仕事が来た。経理部の担当者に確認を取らないといけないし、場合によっては取引先の担当者にも連絡しないといけない。このとき必要になるのは、マニュアルではない。

* 相手が誰で、何ができる人なのかを知る手段(名刺交換)
* 「この作業をお願いできますか」と依頼する手順(業務依頼のフォーマット)
* 依頼した仕事の進捗や結果を受け取る方法(報告のルール)  
  ![](https://static.zenn.studio/user-upload/1dcbeb672d4f-20260714.png)

これがA2Aの役割にあたる。AIエージェント同士が、お互いの「役割」や「できること」を確認し合い、タスクを依頼し、結果を受け取る。この一連のやり取りを標準化したのがA2Aだ。

重要なのは、A2Aで会話する相手は、必ずしも自分と同じ会社(同じベンダー、同じフレームワーク)のエージェントとは限らないという点だ。人間の世界で、取引先の担当者がどこの会社の名刺を持っていようと、名刺交換のマナー自体は共通しているのと同じで、A2Aは異なる開発元・異なる実装のエージェント同士が協業するための「共通のビジネスマナー」を目指している。

### 名刺の中身:Agent Card

では、実際に「名刺交換」はどう実装されているのか。ここで登場するのがAgent Cardだ。

Agent Cardは、A2A対応のエージェントが公開しているJSON形式のメタデータで、自分の名前・できること・連絡先・話しかけ方をまとめたものにあたる。多くの場合、`/.well-known/agent-card.json` という決まった場所に置かれていて、誰でも取得できるようになっている。会社の受付に、誰でも自由に持ち帰れる名刺立てが置いてあるようなものだと思ってもらえばいい。

中身はこんな形になる。

```
{
  "name": "invoice-checker-agent",
  "description": "請求書の内容をチェックし、不備があれば指摘するエージェント",
  "url": "https://agents.example.com/invoice-checker",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true
  },
  "skills": [
    {
      "id": "check-invoice",
      "name": "請求書チェック",
      "description": "PDFの請求書を受け取り、金額・宛先・振込先の不備を検出する"
    }
  ],
  "securitySchemes": {
    "bearerAuth": {
      "type": "http",
      "scheme": "bearer"
    }
  }
}
```

他のエージェントは、話しかけたい相手のAgent Cardをまず取得し、

* 「このエージェントには何を頼めるか(`skills`)」
* 「どこに連絡すればいいか(`url`)」
* 「どう認証すればいいか(`securitySchemes`)」

を確認してから、実際のタスク依頼に進む。名刺を受け取ってから初めて「この人に見積もりの相談をしていいんだな」と分かるのと、流れは変わらない。

![](https://static.zenn.studio/user-upload/89d6f4ea356a-20260714.png)  
ここまで分かると、MCPとの役割分担もより鮮明になる。  
MCPのマニュアルは **「自分が使う道具の説明書」** であるのに対し、Agent Cardは **「自分自身の説明書を、他人に読んでもらうために公開したもの」** だ。  
矢印の向きが逆になっている、と捉えると整理しやすい。

## 比較表:MCPとA2Aは「上下関係」ではなく「守備範囲が違う」

ここまでの内容を整理すると、こうなる。

| 観点 | MCP | A2A |
| --- | --- | --- |
| 何を繋ぐか | エージェントと道具(ツール・データ) | エージェントとエージェント |
| 人間業務での対応 | 業務マニュアル・社内SOP | 名刺交換・業務依頼のプロトコル |
| やり取りの向き | 基本は一方向(道具を使う) | 双方向(依頼と報告のキャッチボール) |
| 主な関心事 | 権限管理、ツールの安全な呼び出し | 役割の把握、タスクの委譲、進捗管理 |
| 認証認可 | プロトコルにOAuth 2.1を内蔵し標準化 | 名刺(Agent Card)に必要な認証方式を宣言するだけで、発行プロセスは協定の外 |

この2つは競合する規格ではなく、そもそも見ている層が違う。実際のAI活用の現場では、1体のエージェントがMCPで社内ツールを使いこなしつつ、A2Aで別のエージェントに仕事の一部を振る、という形で両方が同時に動くことになる。

「うちはMCP対応してるからA2Aはいらない」という話をたまに聞くが、これは「マニュアルを整備したから名刺交換はしなくていい」と言っているのに近い。両者は置き換え関係ではなく、積み上げの関係にある。

## 最小構成で動かす:TypeScriptで作る「マニュアルを読めるAI」

ここからはエンジニア向けに、実際にMCPサーバーを最小構成で書いてみる。お題は単純にして、「今日の日付を返すだけのツール」を1つ公開するサーバーを作る。

```
npm install @modelcontextprotocol/sdk zod
```

```
// server.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "date-tool-server",
  version: "1.0.0",
});

// 「マニュアルの1ページ」にあたる部分。
// このツールで何ができるか、どんな引数を受け取るかを宣言する。
server.registerTool(
  "get_today",
  {
    title: "今日の日付を取得する",
    description: "指定したタイムゾーンで今日の日付を返します",
    inputSchema: {
      timezone: z.string().default("Asia/Tokyo"),
    },
  },
  async ({ timezone }) => {
    const today = new Date().toLocaleDateString("ja-JP", { timeZone: timezone });
    return {
      content: [{ type: "text", text: `今日は${today}です` }],
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
```

たったこれだけで、AIエージェント側から見ると「日付を聞けば教えてくれるツール」が1つ増えたことになる。ポイントは、`inputSchema` と `description` の部分だ。ここが、新人に渡すマニュアルの「このツールはこう使ってください」という説明にあたる。人間が読む説明書ではなく、AIが読んで理解できる説明書を書いている、と捉えるとしっくりくる。

クライアント側(このツールを使うAIエージェント)は、接続時にこのサーバーに問い合わせて「どんなツールが使えるか」を取得し、必要に応じて `get_today` を呼び出す。この「マニュアルを自分で読みに行って、使い方を理解してから使う」という部分こそが、MCPが解決している核心の課題だ。

実務では、これがSlack投稿だったり、社内DBへの問い合わせだったり、ファイルサーバーへのアクセスだったりするわけだが、規格としてやっていることの本質は変わらない。

## この2つを理解すると、DX推進の何が変わるのか

ここまでの内容を、もう一度ビジネスサイドの目線で整理したい。

「AIを導入するかどうか」という議論は、ともすると精神論や期待値の話になりがちだ。だが、MCPとA2Aという2つの軸を持つと、この議論を具体的な組織設計の話に変換できる。

* **MCPの整備状況を問う** = 「うちの会社のツールやデータは、AIが安全に使える形でマニュアル化されているか?」という問い
* **A2Aの整備状況を問う** = 「複数のAIエージェント、あるいは複数部署のAI活用を、誰がどう連携させるルールになっているか?」という問い

この2つが整っていない状態でAIエージェントを導入するのは、マニュアルもなく、他部署の連絡先も渡さずに新人を現場に放り込むようなものだ。優秀な人材を採用しても、活躍できないのは当然だろう。逆に言えば、この2つの整備状況を可視化するだけで、AI活用の議論がぐっと具体的になる。

## もっと実装に踏み込みたい人へ

この記事では、MCPとA2Aの違いを比喩で整理し、最小構成のMCPサーバーまで動かした。

実務でここから先に進むと、次のような課題にぶつかる。

* 複数のAIエージェントが役割分担して働く構成(自分の場合は9つのエージェントに分けている)を、MCP・A2A的な考え方でどう設計するか
* エージェント間でどこまで情報を渡し、どこで区切るか(ハンドオフの設計)
* セキュリティ面で、エージェントに渡す権限をどう最小化するか
* 複数エージェントを運用する上でのコスト管理

この続きは、より実装に踏み込んだ形で別途まとめる予定なので、気になる方は今後の記事も見てもらえたら嬉しい。
