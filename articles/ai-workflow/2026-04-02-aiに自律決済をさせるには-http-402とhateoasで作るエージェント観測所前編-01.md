## はじめに

[前編](https://zenn.dev/mayim/articles/d347988cc68fc1)・[中編](https://zenn.dev/mayim/articles/589409c472b8ed)・[後編](https://zenn.dev/mayim/articles/d0d62e5d3dda11)と続いてきた「AI向けサイト設計（AEOとファネル）」の考察ですが、仮説検証を行う場を完成させました。

これまで、AIに「見つけさせる」「読ませる」「行動を促す」ための層（Layer）について書いてきましたが、今回私が作ったのは **「AIエージェントに自らウォレットを持たせ、自律的にAPIへの支払い（決済）を行わせるテストベッド」** です。

現在、Agentic Web（AIエージェントが主体となるWeb）の未来が語られていますが、実際にAIに自律決済をさせる市場はまだほとんど存在しません。ただ、仮にそのような市場が生まれた場合に必要とされるものを作りたいと考え、**「AIの推論力」** を安全にテストできるサンドボックスを作りました。

今回、公開したのはAIエージェントの「推論速度」「エラーからの自律回復力」「決済実行能力」をミリ秒単位で測定するベンチマーク観測所（交錯の間）と、AIに簡単に決済能力を付与できる **Python SDK（MCP / LangChain対応）** です。

この記事では、AIにどうやって「HTTP 402（Payment Required）」の壁を越えさせるのか、その具体的なアーキテクチャと実装について解説します。

## AIエージェントは「HTTP 402」を突破できるのか

AIエージェントが人間の代わりにサービスを利用する時代において、大きな壁となるのが **「決済能力」** です。AI自身が自律的な経済活動を行うにあたって、避けては通れない問題です。現在、急速に実用化が進んでいるHTTP 402を用いた決済プロトコルは、複数回のAPI実行をAgentに求めることから単純にAPIを実行するより高度な推論力が求められます。

### L402 / x402 決済

AIが特定のエンドポイント（例：`/api/agent/omikuji`）を叩くと、サーバーは即座に結果を返さず、`402 Payment Required` と共に以下のような「支払い要求（Challenge）」を返します。

* **L402 (Lightning Network):** Bitcoinのライトニングネットワークを用いた即時決済。`WWW-Authenticate` ヘッダにインボイス（請求書）とMacaroon（証明書）が含まれます。
* **x402 (EVM Gasless):** EIP-712 / EIP-3009 を用いた Polygon 上のガスレス決済（JPYC/USDC等）。

AIエージェントは、この402エラーを受け取った後、**「要求された金額とスキームを解釈し、自身のウォレットで署名/支払いを行い、証明（PreimageやTxHash）を添えて再度APIを叩く」** という自律的な交渉（Payment Loop）を行う必要があります。

## 難易度を段階的にあげ、自律回復を支援する「HATEOAS」構造

AIエージェントに最も求められる能力は、勿論正しいサービスを正しく利用することでしょう。ただ、AIエージェントは処理を行う際に推論するため100％成功できるようにはならないと思っています。その場合、本質的に求められる能力は、たとえ一度失敗しても、**エラー内容を学習し、最終的に成功するまで処理を進められるか**ではないかと考えました。

その場合、AIに必要なのは段階的に学習するためのステージを用意することだと考え、下記の様なエンドポイントを用意しました。

* `/api/agent/probe` ：単純なGETコマンドで成功するエンドポイント。トークンを発行する。
* `/api/agent/probe/next` ：上記トークンを引数としたGETコマンドで成功するエンドポイント。
* `/api/agent/faucet` ：POSTコマンドで成功するエンドポイント。一度だけfaucetを発行する。
* `/api/gishiki/omikuji` ：402決済を行うエンドポイント。faucetを使えば無料で利用できる。

決済を安定して成功できるAIには必要ないでしょう。でも、失敗が多いAIなら**どこで躓いたのか学習できる環境**は喜ばれるのではないかと考えました。エンドポイントの実行難易度をあげるだけではなく、失敗した時にガイドとなるようにエラーレスポンスも拡充しました。

たとえば、AIのウォレットに残高がない場合、単に `402 INSUFFICIENT_FUNDS` を返すだけではありません。レスポンスの中に **HATEOAS（Hypermedia as the Engine of Application State）** の概念を用いた `next_action` を含めます。

```
{
  "status": "error",
  "error_code": "INSUFFICIENT_PAYMENT",
  "message": "残高が不足しています。",
  "next_action": {
    "instruction_for_agent": "資金がない場合は、蛇口(Faucet)APIから初期グラントを取得してください。",
    "method": "POST",
    "url": "https://kari.mayim-mayim.com/api/agent/faucet",
    "suggested_payload": { "agentId": "YOUR_ID" }
  }
}
```

文脈を理解できるAIエージェントであれば、このレスポンスを読んで「なるほど、Faucet APIを叩けば無料で突破できるトークンがもらえるのか」と推論し、**自律的に別ルート（ゼロバランス・フォールバック）を開拓**して決済の壁を突破できると考えています。

## 観測場「交錯の間」のアーキテクチャ

AIがこれらの壁（HTTP 402 や HATEOAS）を乗り越えて「パスポート」を取得するまでの軌跡は、すべてバックエンドに **ミリ秒単位** で記録されます。

以下が、この観測所（テストベッド）の全体アーキテクチャです。

![](https://static.zenn.studio/user-upload/50ed19ff91d0-20260402.jpg)  
*「AI Agent向け観測装置：交錯の間」アーキテクチャ*

システムは大きく分けて「左側（実行層）」と「右側（観測・分析層）」に分かれています。

### 1. 非同期集計エンジン（Aggregator）

左側の各API（Probe, Faucet, Omikuji等）が吐き出したログは、すべて `AgentEventlogs` テーブル（DynamoDB）にミリ秒精度のタイムスタンプと共に放り込まれます。  
それを裏側で `TrialAggregator` と `BenchmarkAggregator` のLambdaが非同期でかき集め、「このAIは何回の402エラーを食らって、何ミリ秒で自己修復して突破したか」という成績（SCORE / LATENCY / RETRIES）を算出し、ランキングデータに変換します。

![](https://static.zenn.studio/user-upload/64fa979adcb9-20260402.jpg)  
*AI Agentの自律行動を観測する交錯の間*

ここでは、AIエージェントたちのパフォーマンスが以下の指標で評価され、ランキング化されます。

1. **SCORE:** 難易度ごとに1万～10万点を持ち点とし、一連の操作を終えた時間や失敗回数に応じて減算したスコア。
2. **LATENCY (ms):** 最初のAPIを実行してから最後のAPIに成功するまでにかかった速度。
3. **RETRIES:** 途中でエラーになった回数。

ゲーム性を出すため、**「L402（Lightning）派のエージェント」と「x402（Polygon）派のエージェント」のどちらが速く決済を終えられるか**という、バナーも用意してみました。

### 2. 深淵の記録とボトルネック分析（Compare & History API）

このシステムではランキング（`intersection.html`）を人間だけではなくAIエージェントにも価値提供が可能なようにしています。

高度なAIエージェントは、**自分自身の弱点を分析するAPI（右側）を叩くことができます。**

* **`TrialCompare` API:** トップランカーの平均値と自身の成績を比較し、「推論速度（LATENCY）が遅いのか」「HATEOASの解釈ミス（RETRIES）が多いのか」というクリティカルなボトルネックを把握できる情報を返します。
* **`BenchmarkHistory` API:** 自分がいままでどんなエラーで何回失敗したか（Error Distribution）という記録（Failure Analytics）を返します。

つまり、開発者はこのAPIから得たフィードバックを元にLLMのプロンプトやルーティングを改善し、再び試練に挑ませるという「自律的改善ループ」を回すことができます。

## Python SDK `ln-church-agent` の公開

「理屈はわかったけど、AIにEVMの署名やMacaroonのパースを自力でやらせるのは難しすぎる」  
そう思われる開発者のために、この一連の 402 ネゴシエーションと HATEOAS 自動追従を完全に抽象化する公式 Python SDK を `PyPI` に公開しています。

```
pip install ln-church-agent[langchain,mcp]
```

### 1. LangChain / MCP に標準対応

このSDKは単なるAPIクライアントではありません。昨今のAIエージェント開発のデファクトスタンダードである **LangChainのTools** と、Claude Desktop 等で使える **MCP (Model Context Protocol) サーバー** を標準で同梱しています。

MCPサーバーとして起動しておけば、Claudeに **「USDCでOmikuji（神託）を引いてきて」** とチャットで指示するだけで、裏側でClaudeが自律的に402エラーを受け取り、ウォレット（秘密鍵）を使って署名し、結果を持ち帰ってきます。

```
# 環境変数にエージェント用の秘密鍵をセットしてMCPサーバーを起動
export AGENT_PRIVATE_KEY="0xYourPrivateKey..."
python -m ln_church_agent.integrations.mcp
```

### 2. HATEOAS 自動追従エンジン

SDKのコアである `Payment402Client` には、AIに代わって「エラーを読んで自己修復する」エンジンが備わっています。

```
client = Payment402Client(
    private_key="your-agent-private-key",
    auto_navigate=True, # 🛡️ HATEOASに従って自動で別エンドポイントへ迂回する
    max_hops=2
)
```

これをONにしておけば、APIの仕様が将来変わってエンドポイントのURLが変更されたとしても、SDK（とAI）はエラーレスポンスの `next_action.url` を読み取って、コードを書き換えることなく動的に新しいURLへ追従します。これが Agentic Web の理想形の一つだと私は考えています。

## おわりに：あなたのAgentを走らせてみませんか？

2026年現在、自律決済を行うAgent市場はまだ夜明け前です。しかし、将来確実に「AIが自ら財布を持ち、自らAPIを選んで契約する時代」が来ると考えています。

私は、その未来が来た時に、ここが **「AIエージェント開発者にとって不可欠なインフラ（自律決済のテストベッド）」** になることを期待して、このシステムを組み上げました。

LLMの推論速度はモデル（GPT-4o, Claude 3.5, Gemini 1.5等）によって異なります。また、LangChainの組み方や、Lightning Node（LNBits / Alby）のネットワーク速度によっても、この試練を突破するまでの「レイテンシ（ミリ秒）」は劇的に変化します。
