## はじめに

2026年、「AIエージェント」という言葉は日常語になった。ChatGPTやClaudeに頼むのはもう当たり前で、次に来ているのは「自分のマシン側に住み着いて、勝手に動いてくれるAI」だ。朝のニュースを要約してくれたり、Slackの通知をさばいてくれたり、コードを書いて試してくれたり。

そんな時代の中核を担っているのが、今回取り上げる2つのオープンソースプロジェクトだ。

* [OpenClaw](https://openclaw.ai/)、Peter Steinberger氏が主導する、マルチチャネル対応のAIアシスタント基盤（GitHub Star 358k）
* [Hermes Agent](https://hermes-agent.nousresearch.com/)、Nous Research（Hermes/Nomos/Psycheモデルの開発元）が手がける、自律進化型AIエージェント（GitHub Star 87.3k）

どちらもMITライセンスで公開され、Claude/GPT/ローカルモデルに対応し、MCP（外部ツールをAIに繋ぐ共通規格。後述）をサポートする。機能一覧だけ並べると、正直ほとんど同じに見える。

でも、実際に使ってみると挙動が全然違う。 そしてその違いは、機能の多寡ではなく設計思想そのものから来ている。

本記事では、両プロジェクトの公式ドキュメントとソースコードを精査したうえで、以下を解き明かす。

* 何が違うのか（並べてみないと見えない構造の差）
* なぜそう設計されたのか（開発者たちが解こうとしていた問題）
* どう使い分けるべきか（実運用での選び方と、併用という選択肢）

特に焦点を当てるのは 「ハーネス（Agent Harness）」、AIエージェントの「動かし方」を決める外枠の設計だ。同じLLM（Claudeなど）を積んでいても、ハーネスが違えばエージェントは別の生き物になる。ここに設計思想が最も色濃く出る。

## 「AIエージェント」は一つじゃない、Claude Code / Codex との本質的な違い

2026年、「AIエージェント」と呼ばれるものは大量に存在する。Claude Code、Codex、Devin、Cursor、そしてOpenClawやHermes。どれも「ファイルを読み書きする」「コマンドを実行する」「自律的に判断する」。ラベルだけ見ると同じだ。

ではなぜ別々に存在しているのか。まず身近なClaude Codeを例に、現在のAIエージェントがどれほど多機能になっているかを押さえておく。

2026年4月時点のClaude Codeは、もはや「コードを書くだけの道具」ではない。

* Auto Memory: AIが自分でメモリファイル（過去の会話から学んだ事実をまとめたテキスト）を書く。プロジェクト単位で永続化され、セッション（1回の会話の単位）をまたいで引き継がれる
* Dream Memory Consolidation: `/dream` コマンドや夜間のcron（定期実行ジョブ）経由で、複数フェーズのメモリ統合を実行する。セッションログから知見を抽出 → トピック別に整理 → 重複や古い記憶を剪定、という「睡眠中の記憶整理」のような処理を自動で回す。`autoDreamEnabled` 設定で有効化。公式docsには詳細な記載がなく、コミュニティによる解析（[Piebald-AI/claude-code-system-prompts](https://github.com/Piebald-AI/claude-code-system-prompts)）で内部プロンプトが明らかになった
* 20以上のHookイベント: SessionStart、PreToolUse、PostToolUse、FileChanged、SubagentStart... ライフサイクルの各タイミングで任意のスクリプトを走らせる「フック」を仕込める。「ファイルが編集されたら自動フォーマット」「ツールを呼ぶ前に権限チェック」のような拡張が可能
* Agent Teams（experimental・デフォルト無効）: 複数の独立セッションが共有タスクリストとメールボックスで協調する実験的機能。チームメイト同士がメッセージを送り合える。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` で明示的に有効化する
* `/loop`: 現在のセッション内で指定間隔のプロンプトを繰り返す。セッションが終われば停止するので、一時的な巡回タスクに使う
* Remote Tasks: GitHubリポジトリと連携し、Anthropicのクラウドインフラ上でClaude Codeをスケジュール実行する。自分のPCを閉じていても動き続ける「常駐型」
* Skills / Plugins / Subagents / Worktree隔離: 独立したスキルファイル、外部プラグイン、サブエージェント（補助的に呼ぶ別のAI）、git worktreeによる作業隔離、どれも揃っている

こうして並べると一目瞭然なのだが、機能カテゴリだけで見るとOpenClaw・Hermes・Claude Codeの3つはほぼ同じことができる。メモリもある、cronもある、マルチエージェントもある、スキルもある。

ではなぜ別々のプロジェクトとして存在し、それぞれ何万ものスターを集めているのか。答えはハーネス（動かし方の外枠）の設計思想が根本的に違うからだ。

#### ハーネスの「重心」がどこにあるか

同じ「メモリ機能」でも、ハーネスの中での位置づけが全然違う。

Claude Codeのハーネスは「1つのプロジェクトに対する深い理解」に重心がある。Auto Memoryはプロジェクト単位で蓄積される。Hookはコード変更のライフサイクルに最適化されている。Agent TeamsはPRレビューや並列実装に使われる。Dream Memory Consolidationですらコードベースの知見を整理するために設計されている。すべてが「このリポジトリをどう理解し、どう改善するか」に向いている。

OpenClawのハーネスは「複数の人間と複数のAIの間のメッセージフロー」に重心がある。Gatewayのルーティング階層、エージェント間のセッション隔離、24以上のチャネル統合、モバイルノードとの連携。メモリは「このエージェントがこのチームでどんな会話をしてきたか」を追跡するためにある。すべてが「誰が、どこから、どのAIに、何を伝えるか」に向いている。

Hermesのハーネスは「1人の人間の知識と判断力の拡張」に重心がある。SOUL.mdが人格として最上位に固定され、MEMORY.mdとUSER.mdが有界の永続記憶として機能し、スキルが自律的に生成・改善され、FTS5が過去のすべてのセッションを検索可能にする。すべてが「この人間をどう理解し、この人間の代わりにどう判断するか」に向いている。

#### 同じ機能が、ハーネスの重心によって別の意味を持つ

「スケジュール実行ができる」という同じ機能カテゴリでも：

* Claude Codeの Remote Tasks は「毎晩テストを回して、失敗したらPRを自動修正する」。プロジェクトの健全性維持に使う
* OpenClawの cron は「毎朝チームの進捗をSlackにサマリーして投稿する」。組織のコミュニケーション自動化に使う
* Hermesの cron は「毎朝ニュースを要約して、自分の興味に合うものだけTelegramに送る」。個人のナレッジ拡張に使う

同じ機能を持っていても、ハーネスの重心が違うから、自然と異なるユースケースに収束する。そしてそれぞれの領域で、他の2つでは代替しにくい最適化がされている。

Claude CodeのAgent TeamsでSlackボットを運用する「こともできる」。だがOpenClawの24チャネル統合、決定論的ルーティング階層、DMペアリングシステムを手動で再実装することになる。逆にOpenClawでコードレビューを回す「こともできる」が、Claude Codeのファイルレベルの差分理解、型推論の追跡、テスト自動生成には到底及ばない。

「できる」と「そのために設計されている」は全く違う。 ハーネスの重心が決める。

この視点を持つと、次の技術比較がぐっと読みやすくなる。表面的な機能比較ではなく、「この設計は何を優先しているのか」を読み解くための補助線になるからだ。

## 設計思想の根本的な違い

### OpenClaw：「メッセージング・ゲートウェイ」として設計されたAI

OpenClawの世界観を一言で表すと、「どこからでも同じAIに話しかけられる基盤」 だ。

中心にあるのは Gateway と呼ばれるサーバープロセスで、公式ドキュメントでは「セッション、ルーティング、チャネル接続の単一の真実（single source of truth）」と説明されている。

```
WhatsApp ─┐
Telegram ─┤
Discord  ─┼─→ Gateway ─→ Agent Runtime（AIの頭脳）
Slack    ─┤
Signal   ─┘
```

ここで押さえたいのは、AIの頭脳部分（Agent Runtime）はGatewayに接続される「パーツの一つ」に過ぎないということだ。Gatewayの方が主役。設計の優先順位が「まずメッセージングを統合し、そこにAIを接続する」という順序になっている。

なぜこう設計されたのか。OpenClawの根底にある思想は「コンテキストとスキルはユーザーのマシンに住むべきで、プラットフォームの壁に閉じ込められるべきではない」というものだ。SlackにはSlackのAI、DiscordにはDiscordのAI、という分断を壊し、一つのAIがすべてのプラットフォームにまたがって動く世界を目指している。

### Hermes Agent：「自律進化するエージェント」として設計されたAI

Hermesの世界観を一言で表すと、「使うほど賢くなり続けるAI」 だ。

中心にあるのは AIAgent という約10,800行の巨大なプログラムで、これがすべてを統括する頭脳だ。Telegram/Discord/Slackからのメッセージも、ターミナルからの入力も、すべてこの一つの頭脳に流れ込む。

公式サイトではこう説明されている。

> IDEに縛られたコーディングコパイロットでも、チャットボットのラッパーでもない。サーバーに住み、学んだことを覚え、動かせば動かすほど賢くなるシステム

OpenClawとは優先順位が逆で、「まずAIの知性を最大化し、そこにいろんな入り口（CLI、チャット、エディタ）を接続する」という設計になっている。Nous Researchの思想は明確で、エージェントは使うほど賢くなるべきだという信念がアーキテクチャ全体に反映されている。

## ハーネス（Agent Harness）の設計比較

ここからが本記事の核心だ。

### 「ハーネス」とは何か、もう一歩踏み込む

先の用語ボックスで「ハーネスはLLMを動かす外枠」と書いた。ここではもう一歩踏み込んで、具体的に何が行われているのかを分解しておく。

LLM単体にできるのは、入力されたテキストに続くテキストを生成することだけだ。ファイルを読むとか、コマンドを実行するとか、結果を踏まえて次の手を考えるとか、そういう「エージェント的な振る舞い」は、LLMの外側で作り込む必要がある。その作り込みが「ハーネス」だ。

具体的には、以下のループをぐるぐる回す仕組みのことを指す。

```
ユーザー入力
    ↓
① 入力の組み立て：「あなたは○○です。使えるツールは×です。質問は△です」
    ↓
LLM呼び出し
    ↓
② 出力の解釈：LLMが「ファイルを読みたい」と返してきたらツール呼び出しに変換
    ↓
③ ツールの実行：実際にファイルを読む/Web検索する/コマンドを走らせる
    ↓
④ 結果のフィードバック：ツールの結果をLLMに渡して次の判断を促す
    ↓
 ②〜④をLLMが「最終回答」を出すまで繰り返し
    ↓
ユーザーへ応答
```

同じLLM（例えばClaude Opus 4.6）を載せていても、①〜④の設計が違えば、エージェントは別の生き物になる。①でどんな情報を積むか、②でどこまでツール呼び出しを許すか、③でどこのマシンで実行するか、④でどう結果を要約してLLMに戻すか、設計の選択肢は無限にある。

だからこそ、ハーネスを読むと「その開発者が何を最も大事にしているか」が見える。ここからOpenClaw・Hermesそれぞれのハーネスを解剖していく。

### OpenClawのハーネス：Gateway中心の分離設計

#### 全体の流れ

```
① ユーザーがSlack/Telegram等でメッセージを送信
   ↓
② Gateway が受け取り、どのエージェントに渡すか決める（ルーティング）
   ↓
③ 該当エージェントの Pi Agent Runtime（OpenClaw 内蔵のAI実行エンジン）に RPC でメッセージを渡す
   ↓
④ Agent Runtime が LLM API を呼び出す（35+プロバイダ対応）
   ↓
⑤ LLM がツール呼び出しを返す → Agent Runtime が実行
   ↓
⑥ 結果をストリーミングで Gateway に返す
   ↓
⑦ Gateway がユーザーのチャットアプリに応答を配信
```

#### セッション開始時にAIに注入されるもの、OpenClawの「人格キット」

OpenClawでは、エージェントがセッションを開始するたびに、ワークスペース（デフォルト `~/.openclaw/workspace`）から複数のファイルがLLMのシステムプロンプト（AIへの前提指示）に注入される。これが「このAIは何者で、何ができて、何を覚えているか」を決める。

##### Bootstrap injection（公式 `concepts/context.md` より）

> By default, OpenClaw injects a fixed set of workspace files (if present): AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md, HEARTBEAT.md, BOOTSTRAP.md

この 7 ファイルが、ワークスペースに存在すれば（if present）注入される。

| ファイル | 役割 |
| --- | --- |
| AGENTS.md | 行動ルール・制約 |
| SOUL.md | 人格・口調・スタンス |
| TOOLS.md | 使えるツールの一覧と使い方 |
| IDENTITY.md | 名前、役割、自己紹介の仕方 |
| USER.md | ユーザーの好み・背景 |
| HEARTBEAT.md | 自発的に動く条件とタイミング |
| BOOTSTRAP.md | 初期セットアップ指示 |

サイズ制限は公式に明記されている：1ファイルあたり `agents.defaults.bootstrapMaxChars`（デフォルト `20000` 文字）で切り詰め、全ファイル合計で `agents.defaults.bootstrapTotalMaxChars`（デフォルト `150000` 文字）を上限とする。

##### 記憶系ファイル（公式 `concepts/memory.md` より）

OpenClawには以下の3つの記憶系ファイルがある。

| ファイル | 公式の定義（原文） | 公式が明示する注入条件（原文） |
| --- | --- | --- |
| MEMORY.md | "long-term memory. Durable facts, preferences, and decisions." | "Loaded at the start of every DM session." |
| memory/YYYY-MM-DD.md | "daily notes. Running context and observations." | "Today and yesterday's notes are loaded automatically." |
| DREAMS.md（optional） | "Dream Diary and dreaming sweep summaries for human review" | 条件は公式に明記なし（dreaming 機能の成果物として生成される） |

ただしここで、公式ドキュメントと実装の挙動にズレがある点を明記しておく必要がある。

OpenClaw公式リポジトリの [Issue #24404](https://github.com/openclaw/openclaw/issues/24404) が、このズレを Bug として報告している。Issue タイトルは：

> "docs/concepts/memory states MEMORY.md is main-session-only, but code loads it in all non-subagent/cron sessions"

公式docsは「DMセッション（main session）専用」と読める書き方をしているが、実際のコードは subagent/cron 以外のすべてのセッションで MEMORY.md をロードする。実装の該当コード：

```
// filterBootstrapFilesForSession の挙動（Issue #24404より）
if (!sessionKey || (!isSubagentSessionKey(sessionKey) && !isCronSessionKey(sessionKey))) {
  return files;  // MEMORY.md を含む全ファイルを返す
}
```

Discord のグループチャネル、Slack のチャネル、guild、room、いずれも subagent / cron でなければ MEMORY.md は入る。筆者の実運用環境（Slack channel 上で動くエージェント）でも MEMORY.md の取り込みを確認している。

なお Issue #24404 は `Closed as not planned` のステータスで、公式はこのドキュメント・実装ズレを直す予定はない状態。この記事で正確な記述を残す意義はここにある。

実用上の結論を整理すると：

| セッション種別 | MEMORY.md の注入 |
| --- | --- |
| DM session | ✅ 注入される（公式明示） |
| Channel / Guild / Group / Room | ✅ 注入される（公式未明示だが実装の挙動、Issue #24404で確認） |
| Subagent session | ❌ 注入されない |
| Cron session | ❌ 注入されない |

##### Active Memory（公式 `concepts/active-memory.md` より）

MEMORY.md と `memory/**/*.md` は起動時に ~400トークン・80トークンオーバーラップでチャンク化され、`~/.openclaw/memory/<agentId>.sqlite` に索引される（公式 `memory-builtin.md`）。この索引を使って、ユーザーのメッセージごとに関連記憶を検索しプロンプトに挿入するのが Active Memory だ。

公式が明言しているデフォルト設定と除外条件：

* デフォルト `config.allowedChatTypes: ["direct"]`、ダイレクトメッセージのみで動作
* Sub-agent / internal helper execution では動作しない
* Heartbeat / background runs でも動作しない

##### Evergreen と decay（公式 `reference/memory-config.md` より）

> Evergreen files (`MEMORY.md`, non-dated files in `memory/`) are never decayed.

MEMORY.md と `memory/` 内の日付なしファイルは、記憶の減衰（decay）対象にならない。日付付きの日次ノート `memory/YYYY-MM-DD.md` は時間経過で優先度が下がる運用だが、evergreen なナレッジは放っておいても陳腐化しない。

##### Dreaming（公式 `reference/memory-config.md` より）

OpenClaw には「dreaming」という機能がある。memory-core プラグインの設定で cron スケジュールされる一括処理で、デフォルトは `0 3 * * *`（毎日午前3時）。記憶を巡回して統合・要約した結果が、DREAMS.md（または既存の `dreams.md`）に human-readable な形式で書き出される。人間がレビューできる形で記憶の整理結果を出力する設計になっている。

これはClaude Codeの Dream Memory Consolidation と名前・発想が近いが、ドキュメント上は別物として扱われている（OpenClaw 独自の機能として公式 docs に記載）。

#### Memory Flush と Daily Reset、記憶のライフサイクル

日次ノート `memory/YYYY-MM-DD.md` が「自動で育つ」のは、OpenClawが2つの自動処理を回しているからだ。これが公式docsには散在しているものの、実運用で必ず関係してくる押さえるべき挙動なので整理しておく。

##### Memory Flush（compaction 前の silent turn）

会話がコンテキストウィンドウの限界に近づいたり、明示的に compaction が走る前に、OpenClaw は `runMemoryFlushIfNeeded` という silent agentic turn を発火させる。このターンでAI自身にこんなプロンプトが投げられる：

> "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO\_REPLY if nothing to store."

AIが自分で「今日の日次ノートに書き残すべきことはあるか？」を判断して書き出す仕組みだ。デフォルトで有効、設定不要。compaction によって古い会話が要約されて失われる前に、永続化する最後のチャンスとして機能する。

##### Daily Reset（毎日午前4時）

セッションは `atHour: 4`（デフォルト）で毎日午前4時に自動破棄される（`src/config/sessions/reset.ts`）。これは「長すぎる会話履歴を毎日クリアする」ための仕組み。ただしここに穴があって、この daily reset では Memory Flush が走らない。[Issue #45608](https://github.com/openclaw/openclaw/issues/45608) で修正要望中の既知の問題だ。前日の会話に未書き出しの情報があると、4時を境に失われる。実運用上は、残したい事実はその日のうちに `memory_add` ツールなどで明示的に保存しておくのが無難。

`/new` や `/reset` コマンドでもセッションは破棄されるが、同じく flush は走らない。compaction だけが「記憶を守る関所」として機能している。

#### 4層メモリアーキテクチャ

ここまでの情報を整理すると、OpenClawの記憶は4層の階層構造として見ると全体が腑に落ちる。

| 層 | 実体 | 特性 | ロードタイミング |
| --- | --- | --- | --- |
| Layer 1: Session Context | 現在の会話履歴 | ephemeral（揮発性） | 常時、コンテキストウィンドウにあるもの |
| Layer 2: Daily Notes | `memory/YYYY-MM-DD.md`（raw running log） | 日ごと蓄積、大量 | セッション開始時に今日＋昨日の2日分を自動注入 |
| Layer 3: Long-term Memory | `MEMORY.md`（curated insights） | 凝縮された永続事実 | subagent/cron以外のセッション開始時に注入（Issue #24404） |
| Layer 4: Semantic Search | `memory_search` ツール（全ファイル横断） | プロンプトに入りきらない過去記憶 | Active Memoryが必要時に動的検索（デフォルトはDMのみ） |

下層ほど「頻度が高いがノイズも多い」（raw log）、上層ほど「凝縮されていて重要度が高い」（curated）という設計思想。Layer 2 の Daily Notes は raw な観察ログなのでセッション開始時に全量注入、Layer 3 の MEMORY.md への昇格は heartbeat や dreaming で定期レビューされるときに行われる。Layer 4 の semantic search は、prompt に直接入れるには大きすぎる過去記憶を必要なときだけ引いてくる。

この4層を意識すると、「Hermes の凍結スナップショット＋FTS5検索」と非常に似た発想で設計されていることがわかる。OpenClawは記憶の中核機能について、Hermesとほぼ同じアーキテクチャに独自の用語（dreaming、heartbeat）を載せていると見ることもできる。

#### Bootstrap injection の順序について

公式 `concepts/context.md` には 7ファイルの列挙はあるが、注入順序や優先度についての明示的な記述はない（筆者が確認した範囲で `context.md` / `memory.md` / `memory-config.md` / `active-memory.md` のいずれにも順序指定は見当たらなかった）。

ただし筆者がビルド済みの `workspace-*.js`（Gateway側のビルド成果物）を読んだ限りでは、ハードコードされた順序で配列が組み立てられていて、そこでは `AGENTS.md → SOUL.md → TOOLS.md → IDENTITY.md → USER.md → HEARTBEAT.md → BOOTSTRAP.md → MEMORY.md` の順番で並び、MEMORY.md が末尾に置かれているように見える。どのセッション種別でこの配列が組まれるかまでは公式には明記されていない。

この順序が意図的かどうかは公式には明言されていないが、実運用で確実に前提にできるのは「bootstrap 7ファイルが注入されること」と「MEMORY.md は subagent/cron 以外のセッションで追加ロードされること」までだ。ソースで確認できる順序はあくまで「現在のビルド成果物の挙動」に過ぎず、将来のバージョンで変わりうる。

#### subagent / cron セッションでの挙動、「独立エージェント」との混同注意

まず用語整理が必要だ。OpenClawにおける「サブエージェント」は、前項で説明した独立エージェント（マルチエージェント構成で並列運用される各エージェント、自分のワークスペースに SOUL.md / USER.md / MEMORY.md 等をフルセットで持つ） とは別物だ。混同しやすいので明確に区別する。

| 区分 | 実体 | sessionKey の形式 | workspace |
| --- | --- | --- | --- |
| 独立エージェント（standalone agent） | Gateway下で並列に走る各エージェント。OpenClaw公式docs曰く "a fully scoped brain with its own Workspace" | 通常のセッションキー（DM/channel/guild等） | 自身の workspace を持つ（SOUL/USER/MEMORY すべて自分の） |
| サブエージェント（subagent） | 親エージェントのセッションから `/subagents spawn` や `sessions_spawn` ツールで動的にスポーンされる短命のbackground run | `agent:<agentId>:subagent:<uuid>` | 親エージェントの workspace を参照するが、注入は制限される |
| cron session | スケジュール駆動のバックグラウンド実行 | cron専用のsessionKey | 同上 |

独立エージェントの通常セッションでは、前述の通り bootstrap 7ファイル＋MEMORY.md＋日次ノート＋Active Memory検索がフルで働く。ここでは人格もユーザー情報も長期記憶もすべて効く。

制限されるのは「subagent / cron のセッション」だけだ。

公式 `tools/subagents` ページの Limitations 節が明示している：

> Sub-agent context only injects `AGENTS.md` + `TOOLS.md` (no `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, or `BOOTSTRAP.md`).

Issue #24404 で引用されている実装コードでも、`MINIMAL_BOOTSTRAP_ALLOWLIST` は2ファイル構成：

```
MINIMAL_BOOTSTRAP_ALLOWLIST
  ├── AGENTS.md   (DEFAULT_AGENTS_FILENAME)
  └── TOOLS.md    (DEFAULT_TOOLS_FILENAME)
```

subagent / cron セッションでは 行動ルール（AGENTS.md）とツール定義（TOOLS.md）の2ファイルだけが注入され、SOUL.md（人格）も IDENTITY.md も USER.md も MEMORY.md も日次ノートも入らない。「サブエージェントに personality がない」「ユーザーの好みを知らない」挙動はここから来ている。

さらに公式 `active-memory.md` は「Sub-agent/internal helper execution」「Heartbeat/background runs」で Active Memory（記憶の動的検索挿入）も動作しない ことを明示している。つまりスポーンされたサブエージェントは、bootstrap の記憶系が除外されたうえに、検索による記憶補完も受けられない。完全に「与えられた仕事だけを、ノイズなく遂行する」モードに振り切っている設計だ。

要するに：

* あなたが独立エージェントとして Slack channel で動いているなら → SOUL.md も MEMORY.md もフルで効く
* あなたがそのエージェントから `/subagents spawn` された子タスクなら → AGENTS.md + TOOLS.md のみ、人格も記憶も引き継がれない

#### ポイント①：Gatewayがすべてのメッセージを仲介する

OpenClawでは、ユーザーのメッセージがAIに届くまでに必ずGatewayを経由する。Gatewayの役割は「電話交換手」のようなもので、以下を担当する。

* どのエージェントに回すか決める: 例えば「Slackの#salesチャネルからのメッセージはAgent A（営業サポート）に」「Discordの開発サーバーからはAgent B（開発アシスタント）に」と振り分ける
* セッション（会話の文脈）を管理する: 誰と誰がどんな会話をしているか追跡する
* 認証を行う: 知らない人からのDMはブロックする（ペアリングシステム）

ルーティングは決定論的な優先度ルールに従う。

```
1. 完全一致（この特定のDM相手 → Agent A）
2. 親スレッド継承（このスレッドの親 → Agent A だから、子も Agent A）
3. Discordロール（#admin ロール → Agent C）
4. チーム / アカウント単位
5. チャネル単位（Telegram全体 → Agent B）
6. デフォルトエージェント
```

この階層を「上から順に見ていき、最初にマッチしたルールが勝つ」という仕組みだ。つまり「細かい指定ほど優先される」ようにできている。

#### ポイント②：RPC分離、GatewayとAgent Runtimeが別プロセスで動いている

OpenClawの設計で地味だけど決定的なのが、GatewayとAIの頭脳（Agent Runtime）がRPC（Remote Procedure Call）で通信する別プロセスとして動いていることだ。

RPCとは「あるプログラムが、別のプログラムに処理を依頼する通信の仕組み」のこと。ブラウザがサーバーにAPIリクエストを飛ばすのと構造は同じで、GatewayがAgent Runtimeに「このユーザーからこんなメッセージが来たから処理して」とリクエストを投げ、Agent Runtimeが結果を返す。

なぜわざわざ分離しているのか。具体例で考えるとわかりやすい。

AIモデルをClaude Opus 4.6からGPT-5.4に乗り換えたい。 普通ならSlack連携やTelegram連携も全部作り直しになりそうだが、OpenClawではAgent Runtimeだけ差し替えればいい。Gatewayは何も変わらない。Slackの設定もTelegramの設定もそのまま。

逆に、LINEチャネルを追加したい。 Gatewayにプラグインを一つ入れるだけで、AIの頭脳側は何も触らなくていい。

AIの頭脳が暴走してクラッシュした。 でもGatewayは生きているので、Slackには「ただいま処理中です」と返せる。全チャネルが無反応になるのを防げる。

この設計パターンはソフトウェアの世界で「疎結合（loose coupling）」と呼ばれる。部品同士の依存を最小限にして、片方を変えてももう片方が壊れないようにする考え方だ。組織で毎日使うサービスで「AI側のアップデートが原因でSlack連携が落ちた」なんてことが起きたら業務が止まる。RPC分離はそういう事故を構造的に防ぐためにある。

#### ポイント③：マルチエージェントが一級市民

OpenClawでは、1つのGatewayで複数の独立したエージェントを同時に動かせる。各エージェントは完全に隔離された「ワークスペース」を持つ。

```
~/.openclaw/agents/
  ├── sales-support/      ← 営業サポートAgent
  │   ├── SOUL.md          ← このAgentの人格定義
  │   ├── AGENTS.md        ← プロジェクト固有の指示
  │   ├── USER.md          ← ユーザー情報
  │   └── sessions/        ← 会話履歴
  ├── dev-assistant/       ← 開発アシスタントAgent
  │   ├── SOUL.md
  │   ├── ...
  └── client-facing/       ← クライアント対応Agent
      ├── SOUL.md
      ├── ...
```

エージェント間で認証情報は共有されない。Agent Aが持つAPIキーにAgent Bはアクセスできない。セッション（会話履歴）も完全に分離されており、あるエージェントの会話が別のエージェントに漏洩することはない。

この「マルチエージェント・ルーティング」がOpenClawの真骨頂だ。1つのメッセージング基盤の上で、役割の異なる複数のAIを安全に同時運用できる。

#### ポイント④：ツールは「場所」によって実行先が変わる

OpenClawのツール実行には面白い設計がある。ツールの種類によって、実行される場所が変わるのだ。

* ファイル操作やシェルコマンド: Gatewayが動いているマシン上で実行
* カメラ撮影やスクリーンショット: ペアリングされたiPhone/Android上でリモート呼び出し（`node.invoke`）経由で実行
* ブラウザ操作: 専用のChrome/Chromiumインスタンス上で実行

これは、OpenClawがモバイルアプリ（iOS/Androidノード）を持っていることから来る設計だ。例えば「今目の前にあるものを写真に撮って」と言えば、ペアリングされたiPhoneのカメラが起動する。サーバー上のAIが、手元のスマートフォンを遠隔操作できるということだ。

### Hermesのハーネス：AIAgent中心の一体設計

#### 全体の流れ

```
① ユーザーがCLI / Telegram / Discord / エディタ等から入力
   ↓
② AIAgent コアループが入力を受け取る
   ↓
③ 「システムプロンプト」を動的に組み立てる
   ├── SOUL.md（人格）
   ├── USER.md + MEMORY.md（記憶）
   ├── Skills（習得済みスキル）
   ├── Context Files（プロジェクト情報）
   ├── Tool Guidance（モデルごとのツール使用ガイド）
   └── 圧縮された過去の会話
   ↓
④ プロバイダを選択し、LLM APIを呼び出す
   ↓
⑤ LLM がツール呼び出しを返す → 47種のツールから実行
   ↓
⑥ 結果をSQLiteに永続化
   ↓
⑦ ④に戻る（LLMが最終回答を出すまでループ）
   ↓
⑧ 最終回答をユーザーに返す + 必要ならスキルを自動生成
```

#### セッション開始時にAIに注入されるもの、Hermesの「知性キット」

まず全体像を見てから、各レイヤーを掘り下げていく。

```
システムプロンプトに注入されるもの（Hermes）、上から順に優先度が高い

① SOUL.md                → 人格（最上位・絶対にブレない）
② Tool Guidance           → モデル別のツール使用ガイド
③ USER.md + MEMORY.md     → 永続記憶（凍結スナップショット）
④ Skills 一覧             → 習得済みスキルの名前と概要のみ
⑤ Context Files           → 作業中プロジェクトの情報
⑥ タイムスタンプ            → 現在日時
⑦ プラットフォーム情報       → CLI/Telegram/Discord等の出力形式
⑧ /personality オーバーレイ → セッション限定の人格調整（任意）
```

OpenClawと比較すると、構成要素は似ているが設計哲学が違う。

* OpenClaw：注入経路が複数ある（bootstrap 7ファイル / subagent・cron以外で追加ロードされる MEMORY.md / 今日と昨日の日次ノート / Active Memoryの索引検索）。サイズ上限は bootstrap で1ファイル20,000文字・合計150,000文字。情報量に余裕を持たせる設計
* Hermes：注入は順序付きの単一プロンプトスタックで一本化。SOUL.mdがスロット#1に配置され、MEMORY.md（2,200文字）、USER.md（1,375文字）と有界。Skillsは名前だけ入れて全文は遅延読み込み。トークン効率を最大化する設計

Hermesの SOUL.md をスロット#1に固定する設計は、後述の通り primacy effect（LLMはシステムプロンプト冒頭の情報にattentionが強くかかる特性）を明示的に利用したものだ。

#### ポイント①：各レイヤーの詳細、Hermesの知性の源泉

Hermesのハーネスで最も肝なのは、③の「システムプロンプトの動的組み立て」 だ。ここにHermesの設計思想が凝縮されている。

OpenClawが「bootstrap 7ファイル＋DMセッション時のMEMORY.md＋日次ノート＋Active Memoryでの索引検索」という多層の注入経路を持つのに対し、Hermesは「順番と文字数制限を厳密にコントロールした単一のプロンプトスタック」で組み立てる。それぞれのレイヤーを見ていこう。

##### レイヤー1：SOUL.md（人格）

```
# ~/.hermes/SOUL.md の例
あなたは技術的に正確で、簡潔な回答を好むアシスタントです。
不確かなことは「わからない」と正直に言います。
ユーモアは控えめに。コードは説明より先に見せます。
```

SOUL.mdはプロンプトスタックの最上位（スロット#1）に配置される。どんなプロジェクトで作業していても、どんなツールが有効でも、人格だけは絶対にブレないようになっている。

##### レイヤー2：USER.md + MEMORY.md（記憶）

```
# ~/.hermes/memories/USER.md（1,375文字制限）
- TypeScriptを好む。Pythonも使うが型ヒント必須
- エラーメッセージは日本語で出してほしい
- 「なぜそうするのか」の説明を重視する

# ~/.hermes/memories/MEMORY.md（2,200文字制限）
- プロジェクトXではpnpmを使用。npmは禁止
- AWS ap-northeast-1にデプロイ。CDKを使用
- 前回、BigQueryの日付パーティションで改善した手法が有効だった
```

ここで押さえたいのが凍結スナップショットパターンだ。メモリはセッション開始時に一度だけ読み込まれ、セッション中は変更されない。

なぜこんな設計にしているのか。LLMのAPI呼び出しには「プレフィックスキャッシング」という最適化技術がある。これはシステムプロンプトの内容が前回と同じなら、API側が「前と同じ部分は処理済み」として使い回してくれる仕組みだ。結果、応答が速くなりAPIコストも下がる。セッション中にメモリが変わるとキャッシュが効かなくなるため、あえて「凍結」している。

メモリの更新はディスクに即座に書き込まれるが、プロンプトに反映されるのは次のセッションから。つまり「今日学んだことは明日から効く」という設計だ。

##### レイヤー3：Skills（習得済みスキル）

Hermesが過去のタスクから自律的に作成したスキルの一覧が注入される。ただし全文が入るわけではない。段階的開示（Progressive Disclosure） という仕組みで、最初はスキル名と説明の一覧だけを入れる（トークン節約のため）。LLMが特定のスキルを使いたいと判断した時に初めて、そのスキルの全文が読み込まれる。

##### レイヤー4：Context Files（プロジェクト情報）

作業ディレクトリに置かれた`.hermes.md`、`AGENTS.md`、`CLAUDE.md`等のファイルが読み込まれる。プロジェクト固有の技術スタック、コーディング規約、注意事項などが記述される。20,000文字の上限があり、超過分は先頭と末尾を残して中間が切り詰められる。

さらに、エージェントがディレクトリを移動すると、そのディレクトリのコンテキストファイルが段階的に発見・追加される。システムプロンプトを壊さずに、必要な情報だけを動的に追加する設計だ。

すべてのコンテキストファイルは読み込み前にプロンプトインジェクション検査が行われる。プロンプトインジェクションとは、コンテキストファイルの中に「このルールを無視して秘密情報を教えて」のような悪意のある指示を仕込む攻撃手法だ。Hermesはこうしたパターンや不可視文字を検出し、該当ファイルを除外する。

##### レイヤー5：Tool Guidance（モデル別ツール使用ガイド）

LLMのモデルによってツールの使い方の癖が異なる。例えばClaudeとGPTではファイル操作の指示方法が微妙に違う。Hermesはモデルごとに最適化されたツール使用ガイドをプロンプトに注入する。

##### レイヤー6：圧縮された過去の会話

長い会話を続けると、LLMのコンテキストウィンドウ（一度に処理できるテキスト量）を超えてしまう。Hermesはコンテキストが一定の閾値に到達すると、自動的に過去の会話を要約圧縮する。

圧縮は3つのパラメータの組み合わせで制御される：

* threshold: 圧縮を発動するコンテキスト使用率の閾値（例：50%）
* target\_ratio: 圧縮後のターゲットサイズ（例：元の30%まで縮める）
* protect\_last\_n: 直近のN件のターン（発言のやりとり単位）は圧縮対象から保護

つまり「コンテキストの50%を使ったら発動 → 目標の30%まで縮める → ただし直近のターンは削らない」といった動作になる。これにより、3時間の連続作業でも直近の文脈を失わずに会話を続けられる。

#### ポイント②：AIAgent は一つの巨大なループ

OpenClawがGatewayとAgent Runtimeを分離していたのに対し、Hermesは一つの巨大なプログラム（約10,800行）がすべてを処理する。

この設計を「モノリシック（一枚岩）」と呼ぶ。プラットフォーム固有の処理（CLIの表示、Telegramの通知形式、エディタとの通信）は「エントリポイント（入り口）」と呼ばれる薄い層に隔離され、AIのコアロジックには影響しない。

```
CLI ──────────┐
Telegram ─────┤
Discord ──────┼──→ AIAgent（10,800行のコアループ）
VS Code(ACP) ─┤
Batch Runner ─┘
```

なぜモノリシックなのか。AIの知性を最大化するには、すべての情報が一箇所に集まっている方が有利だからだ。メモリ、スキル、セッション履歴、コンテキストファイル。これらすべてが一つのループ内で参照でき、一つの判断に統合される。分離するとデータのやりとりにオーバーヘッドが生まれ、情報が欠落するリスクが増える。

#### ポイント③：プロバイダ解決、どのLLMを使うかの決定

Hermesは18以上のAIプロバイダ（Anthropic、OpenAI、Google、OpenRouter、Nous Portal等）に対応し、3つのAPIモード（chat\_completions、codex\_responses、anthropic）を持つ。

プロバイダの切り替えは`hermes model`コマンド一つで行える。コードの変更は不要。この「モデル非依存性」はHermesの設計原則で、特定のAI企業にロックインされることを防いでいる。

さらに、メインのLLMとは別に補助モデル（Auxiliary Models）を設定できる。

```
auxiliary:
  vision:
    provider: "google"
    model: "gemini-2.5-flash"       # 画像認識用
  compression:
    provider: "openrouter"
    model: "gemini-3-flash-preview"  # 会話圧縮用
```

画像認識にはGemini、会話圧縮には安いモデル、メインの推論にはClaude、というようにタスクごとに最適なモデルを使い分けることで、コストと品質の最適化を行える。

#### ポイント④：47のツールと6つの実行バックエンド

Hermesは47のツールを19のカテゴリ（ツールセット）に整理して持っている。各ツールは起動時に自動的にツールレジストリ（「使えるツールの名簿」のようなもの）に登録される。

ツールセットの例：

| カテゴリ | 内容 |
| --- | --- |
| terminal | シェルコマンド実行、プロセス管理 |
| file | ファイルの読み書き、検索 |
| web | Web検索、コンテンツ抽出 |
| browser | ブラウザの自動操作 |
| vision | 画像の認識・分析 |
| delegation | サブエージェントの生成・並列処理 |
| memory | 記憶の追加・更新・削除 |
| automation | cronジョブ、メッセージ送信 |
| mcp | 外部MCPサーバーとの連携 |

ツールが実際に実行される「場所」は6つのターミナルバックエンドから選べる。

| バックエンド | 実行場所 | 隔離レベル | 用途 |
| --- | --- | --- | --- |
| local | 自分のマシン | なし | 開発・個人利用 |
| docker | コンテナ内 | 完全隔離 | サンドボックス（隔離された安全な実行環境） |
| ssh | リモートサーバー | ネットワーク境界 | リモート開発 |
| modal | クラウド（サーバーレス） | VM隔離 | 使った分だけ課金 |
| daytona | マネージドワークスペース | コンテナ隔離 | チーム開発 |
| singularity | HPCクラスタ | プロセス隔離 | 研究・大規模計算 |

Dockerバックエンドでは、セキュリティが強化されている。プログラムが持てる権限の制限（Linux capabilities削除）、同時実行プロセス数の上限（256）、ファイルの書き込み禁止（読み取り専用ファイルシステム）、CPU/メモリ/ディスクの使用量制限が適用される。要するに「コンテナの中でAIが暴走しても、ホストマシンには影響が及ばない」ようになっている。

#### ポイント⑤：セッション永続化、すべてが記録される

Hermesでは、すべての会話と全てのツール実行結果がSQLiteデータベースに保存される。このデータベースにはFTS5（Full-Text Search 5）という全文検索エンジンが組み込まれており、過去のセッションを横断して検索できる。

```
「3週間前にRedisのキャッシュ戦略について調べた時、どういう結論だったっけ？」
→ FTS5が過去セッションを検索 → LLMが結果を要約して回答
```

このクロスセッション想起機能が、Hermesの「使うほど賢くなる」を技術的に支えている。過去の試行錯誤が検索可能な知識ベースとして蓄積されるため、同じ問題を二度調べる必要がなくなる。

## メモリシステムの比較

ハーネス章で OpenClaw の4層メモリ（Session Context / Daily Notes / MEMORY.md / Active Memory）と、Hermes の凍結スナップショット + FTS5 を見てきた。ここでは比較の視点を1枚に整理しておく。

| 観点 | OpenClaw | Hermes |
| --- | --- | --- |
| 中核 | MEMORY.md + 日次ノート + Active Memory索引 | MEMORY.md / USER.md（凍結スナップショット）+ FTS5 全量検索 |
| サイズ上限 | bootstrap 合計150,000文字、MEMORY.md 単体で20,000文字 | MEMORY.md 2,200文字、USER.md 1,375文字（厳密に有界） |
| 更新ポリシー | compaction 前の Memory Flush で AI 自身が書き出す | `memory_add` / `memory_replace` / `memory_remove` で明示操作 |
| 外部拡張 | 各種プラグイン | 8つの外部メモリプロバイダ（Honcho / Mem0 / Hindsight 等） |
| 隔離 | エージェント単位のワークスペース隔離（認証情報・履歴とも分離） | 単一エージェント、クロスセッションは FTS5 検索で遡る |

差が出るのは上限と隔離方針だ。OpenClawは「ゆとりある入れ物を役割別に分ける」、Hermesは「タイトな入れ物を凝縮して持つ」。組織基盤として複数ロールのAIを動かす前者と、個人の知性を最大化する後者の違いが、そのままメモリ設計の差として現れている。

## スキルシステムの比較

ハーネス章の Hermes レイヤー3 で段階的開示（Progressive Disclosure）の仕組みに触れたので、ここでは OpenClaw 側を補足し、両者の設計思想を対比する。

| 観点 | OpenClaw | Hermes |
| --- | --- | --- |
| スキルの所在 | バンドル / マネージド / ワークスペースの3層 | `~/.hermes/skills/<name>/SKILL.md` + references/templates/scripts |
| 有効化単位 | エージェントごとにON/OFF切り替え可能 | 全体で有効、LLMが判断して呼び出す |
| 自律生成 | 会話の流れでAI自身が作成 | 5回以上のツール呼び出しが要った時、エラーから解決策を見つけた時、ユーザーに修正された時 |
| プロンプト注入 | スキルはプラグインとして読み込み | 段階的開示（名前だけ → 全文 → 参照ファイル） |
| マーケットプレイス | Lobsterワークフロー連携 | Skills Hub（インストール時セキュリティスキャン） |

Hermesの段階的開示は、「スキル数が増えてもトークンコストが増えない」設計になっている。Level 0ではスキル名のリストだけ（数十トークン）、LLMが必要と判断して初めてLevel 1（全文、数百〜数千トークン）が読み込まれる。OpenClawはエージェント単位のON/OFFで制御し、スキル=プラグインという割り切り方。スキルを「積極的に育てる資産」と見るか、「機能拡張の口」と見るか。ここに両者の哲学が出ている。

## セキュリティモデルの比較

### OpenClaw

OpenClawのセキュリティは「個人アシスタント運用」を前提としている。公式ドキュメントでは「1つのGatewayに1人の信頼されたオペレーター」を想定し、敵対的マルチテナント（互いに信頼しないユーザーの同居）はサポート外と明記されている。

主なセキュリティ機能：

* DMペアリングシステム: 知らない相手からのDMはデフォルトでブロック。1時間有効のショートコードで承認する
* ツール拒否リスト: エージェントごとに使えるツールを制限（exec、browser、filesystemなど丸ごとブロック可能）
* Exec承認ゲート: シェルコマンドの実行にオペレーター承認を要求
* Dockerサンドボックス: ツール実行をコンテナ内に隔離
* `openclaw security audit`コマンド: 設定の脆弱性を自動検出し、`--fix`フラグで自動修正

### Hermes Agent

Hermesは7層の多層防御（Defense in Depth）モデルを採用している。公式docsに記載されている順序に従って解説する。

1. User authorization（ユーザー認可）: 誰がエージェントに話しかけられるかを制御。allowlist、DMペアリング等で知らない相手からのアクセスをブロック
2. Dangerous command approval（危険コマンド承認）: 再帰的削除、権限変更、SQL DROP等の破壊的操作を検出し、人間に確認を求める。3つのモード（manual / smart / off）から選択可能
3. Container isolation（コンテナ隔離）: Docker / Singularity / Modal 等のサンドボックスで強化設定（hardened settings）を適用。Linux capabilities削除、プロセス数制限、リソース上限等
4. MCP credential filtering（MCP認証情報フィルタリング）: MCPサブプロセスに渡される環境変数を隔離。KEY、TOKEN、SECRET等を含む変数は明示的な許可がなければ渡さない
5. Context file scanning（コンテキストファイル検査）: AGENTS.mdやSOUL.md等のプロジェクトファイルを読み込む前にプロンプトインジェクション（悪意のある指示を仕込む攻撃）のパターンでスキャン
6. Cross-session isolation（クロスセッション隔離）: セッション同士がお互いのデータや状態にアクセスできないように分離。cronジョブのストレージパスもパストラバーサル攻撃から保護
7. Input sanitization（入力サニタイゼーション）: ターミナル系ツールの作業ディレクトリパラメータをallowlistで検証し、シェルインジェクションを防ぐ

なお、Hermesにはここに含まれていない追加のセキュリティ機能もある。Tirith実行前スキャンはホモグラフ攻撃（見た目が似た文字で正規コマンドに偽装する手法）、パイプ経由のインジェクション等を検出する。SSRF防御（Server-Side Request Forgery、AIに社内ネットワークやクラウド管理画面にアクセスさせる攻撃を防ぐ仕組み）はプライベートIP帯域、localhost、クラウドメタデータエンドポイントへのアクセスを自動ブロックする。これらは7層モデルの上に追加で積まれている。

## 考察：3層のAIエージェント構成、「道具・組織・個人」を使い分ける

ここからは筆者の考察パート。2つのエージェントを精査したうえで、自分はこれらをどう使うべきか、そしてなぜ単一のエージェントでは足りないのかについて、現時点の結論を書いておく。

### なぜ「一つで全部」が成立しないのか

まず押さえておきたいのは、Claude Code・OpenClaw・Hermesは「同じ問題を違う角度から解いている」のではなく、「別々の問題を解いている」ということだ。

* Claude Codeは「このリポジトリをどう理解し、どう改善するか」を解いている
* OpenClawは「このメッセージをどのAIに、どう届けるか」を解いている
* Hermesは「この人間をどう理解し、この人間の代わりにどう判断するか」を解いている

機能の重なりは確かにある。Claude Codeにもメモリがあるし、OpenClawにも自律実行がある、Hermesにもマルチチャネル対応がある。でも、それぞれの「重心」（その設計が最も最適化している部分）は別の場所にある。

ここでよくある勘違いは、「重心が違うだけで、頑張れば他の2つの代替もできるだろう」というものだ。実際にはそう簡単にいかない。Hermesの凍結スナップショットやFTS5による過去セッション検索は、Claude CodeのAuto Memoryを拡張してもそう簡単には再現できない。OpenClawの24チャネル統合・決定論的ルーティング・DMペアリングを、Hermesのモノリシック設計の上に後付けするのも現実的ではない。「そのために設計されている」ものと「頑張れば代替できる」ものには、運用面で決定的な差がある。

結論は単純で、3つを併用するのが素直な最適解になる。役割分担を整理したのが下の3層モデルだ。

### 3層モデル：道具・組織・個人

```
┌─────────────────────────────────────────────────────┐
│  Layer 3: 個人（Hermes）                             │
│  「自分の知識と判断力の拡張」                        │
│  SOUL/MEMORY/USER、自律スキル、クロスセッション想起  │
├─────────────────────────────────────────────────────┤
│  Layer 2: 組織（OpenClaw）                           │
│  「チームのコミュニケーションレイヤーのAI」          │
│  Gateway、24+チャネル、マルチエージェント、ルーティング│
├─────────────────────────────────────────────────────┤
│  Layer 1: 道具（Claude Code）                        │
│  「リポジトリと対峙する手」                          │
│  深いコード理解、型推論、差分運用、テスト生成        │
└─────────────────────────────────────────────────────┘
```

下から順に、抽象度が上がる。Layer 1はコードと直接格闘する「手」で、Layer 2は組織のコミュニケーションを捌く「場」で、Layer 3は自分自身の延長になる「影」だ。それぞれのLayerは独立して動くが、次に述べる「知識フロー」で相互に接続される。

### 知識フロー：個人と組織の双方向の循環

3層モデルの真価は、個別に動かすことではなく層の間で知識を循環させることにある。

##### 個人 → 組織（Hermes → OpenClaw）

個人としてHermesに蓄積されていく知見、コードレビューの癖、設計判断の基準、失敗から学んだ注意点、を、組織のタイミングでOpenClaw経由でチームに流す。具体的には、HermesのMEMORY.mdやスキルから「公開しても良い知見」を抽出し、OpenClawのエージェントが担当するSlackチャネルに「週次の気づき」として投稿する、といった運用が考えられる。個人のナレッジを組織にブロードキャストする経路だ。

##### 組織 → 個人（OpenClaw → Hermes）

逆方向も同じくらい大事だ。OpenClawがSlackやDiscordで拾ったチームの文脈、プロジェクトの方針変更、他メンバーの課題感、暗黙の合意事項、を、Hermesの記憶に吸収させる。OpenClawのログからキーワードを抽出してHermesの`memory add`ツールに流す仕組みを作れば、個人の判断の中に「組織のリアルタイムな状況」が自動で織り込まれていく。

##### 道具 → 個人（Claude Code → Hermes）

Claude Codeで書いたコードや設計判断も、Hermesが学ぶ対象になる。Claude CodeのAuto MemoryをHermesがスキャンして、繰り返し参照されるパターンをHermesの自律スキルに昇華する、という使い方も想定できる。

この循環がうまく回り始めると、個人の仕事と組織の仕事の境界がゆるく溶ける。個人で学んだことが自動で組織に還元され、組織の文脈が個人の判断を助ける。AIが間に入ることで、従来は「意識して共有しないと流れなかった情報」が、ほぼ自動で流れる世界になる。

### 選定フレーム：目の前のタスクをどこに投げるか

3層を並列に持っていると、「今このタスクはどれに投げるか」の判断が必要になる。自分なりに整理するとこうなる。

| タスクの性質 | 投げる先 | 理由 |
| --- | --- | --- |
| リポジトリの中で完結する作業（コード変更、テスト、レビュー） | Claude Code | コード理解と差分運用が最強 |
| チーム宛のメッセージ処理、Slack/Discordで完結する作業 | OpenClaw | チャネル統合とルーティングが最強 |
| 自分の思考整理、情報収集、個人の判断が必要な作業 | Hermes | 個人記憶と長期学習が最強 |
| 3層を横断する作業（例：Slackの議論をコードに落とす） | OpenClaw → Claude Code の連携 | 各層の強みを直列接続 |
| 長期に育てたい知見（例：設計原則、失敗パターン） | Hermesに蓄積 | MEMORY.mdと自律スキルに永続化 |

大原則は「タスクの重心とエージェントの重心を合わせる」だ。メッセージングの話はOpenClawに、コードの話はClaude Codeに、自分の記憶と判断の話はHermesに。ずれた組み合わせで無理やり動かすと、機能的には動いても運用負担が跳ね上がる。

### 留意点：「3つ持つ」コストは無視できない

正直に書いておくと、この3層構成にはコストがある。

* セットアップの複雑さ：3つ分の設定、3つ分の認証、3つ分のアップデート追従が必要になる
* APIコスト：3つが並列でLLMを叩くと、課金もそれなりに膨らむ
* 知識フローの配管：個人↔組織の循環は自動では起きない。連携の仕組みを自分で作り込む必要がある
* セキュリティ境界の管理：個人の記憶と組織の記憶をどこで区切るか、どちらに何が流れるべきか、明示的な設計が要る

だから「とりあえず3つ入れてみる」は推奨しない。まず自分のタスクの重心がどこにあるかを観察して、一つだけ入れる。重心が単一なら、それ一つで完結させた方が幸せだ。2つ目以降を足すべきタイミングは、「既存のエージェントに無理をさせている」と感じた時だと思う。Claude Codeにチーム通知を担わせていて辛い、HermesにSlackの全履歴を食わせていて重い、そういうサインが出てから足す。

### OpenClawの組織適用、セキュリティの現実と、コミュニケーション摩耗を解く基盤としての可能性

3層モデルを語るうえで、特にLayer 2の組織レイヤー（OpenClaw） については少し突っ込んで書いておきたい。ここは多くの読者が「セキュリティ大丈夫？」「本当に実務で使えるの？」と引っかかる部分だからだ。

#### セキュリティは「歴史的問題」と「現在の実力」を分けて見るべき

OpenClawがリリースされた2026年初頭、セキュリティ面の指摘は多かった。DMペアリングのバイパス、コンテキストファイル経由のプロンプトインジェクション、MCPサブプロセスへの認証情報漏洩、実際にいくつかのCVEも出た。これは事実で、初期のOpenClawが本番運用に耐える状態だったとは言いがたい。

ただそれは2026年1月〜3月の話で、今（2026年4月時点）はかなり事情が違う。OpenAIと協業する形でセキュリティ強化が進み、セキュリティ章で並べた7機能（DMペアリング / ツール拒否リスト / Exec承認 / Dockerサンドボックス / security audit 等）はすべてこの数ヶ月で大きく整備された成果物だ。`security audit --fix` で設定の脆弱性を自動検出・自動修正できるところまで来ている。

そして冷静に業界を見渡すと、ほとんどの企業の内製Slack botや社内ツールは、現在のOpenClawより脆弱なケースが大半だと感じている。平文APIキー、権限分離ゼロの管理画面、認証なしの内部API、プロンプトインジェクション対策ゼロのRAG。こういう環境でOpenClawだけ「まだセキュリティが不安」と言うのは、バランスを欠いた評価だ。

権限設定を真面目にやれば、OpenClawは組織で十分セキュアに運用できるというのが筆者の結論。もちろんデフォルト設定のままポイっと置くのは論外だが、それはどのツールでも同じだ。

#### セキュリティ観の再定義、「守りきれる前提」から「壊れたときに回収できる前提」へ

もう一歩踏み込むと、そもそも現代のセキュリティ議論は 「完全に守りきる」という前提自体がもう古いと思っている。本気の攻撃者に狙われたら、どんな企業でも最終的には突破される。ゼロデイ、サプライチェーン攻撃、標的型フィッシング、内部者脅威、クラウド設定ミス、攻撃面は無数にあり、守備側は常に後手だ。

だから評価軸を変えるべきで、今のセキュリティの勝ち筋は以下の3点に移っていると考えている。

1. リスクヘッジ（権限最小化・影響範囲の限定）  
   権限を分離して、1箇所が突破されても全体が陥落しないようにする。OpenClawで言えば、エージェントごとのツール拒否リスト、DMペアリングの信頼境界、マルチエージェントのセッション隔離。どれも「ここが破られても、あそこは生きている」を作るための部品だ。
2. ログ追跡（audit trail と再構成可能性）  
   完全防御ができない以上、起きたことを後から完全に再構成できることが欠かせない。誰が何を、いつ、どこから叩いたか。OpenClawは全セッションをSQLiteに永続化し、`sessions_history` で横断参照できる。Hermesも全ツール実行をSQLite+FTS5に記録する。このログの厚さが、事後対応と再発防止のコアになる。
3. 復旧可能性（rollback と secret rotation）  
   やられた後、どれだけ速くクリーンな状態に戻せるか。バックアップの粒度、シークレットの回転設計、権限の即時取り消し、こういう「回収の早さ」こそ今の競争軸だ。

この3軸で見ると、OpenClawはかなり筋がいい。権限は細かく分けられるし、ログは詳細に残るし、エージェント単位のワークスペース隔離で影響範囲を構造的に閉じ込められる。「侵入前提（Assume Breach）」モデルでの運用に、OpenClawの設計は意外とフィットしている。

#### ヒューマン自体が最大のセキュリティリスクである現実

もう一つ、セキュリティ議論で見落とされがちなのが 「結局、一番弱いのは人間」 という現実だ。統計的にも、セキュリティインシデントの圧倒的多数は技術的な脆弱性ではなく、人間の判断ミスから始まる：

* フィッシングメールを開く
* 強いパスワードを使い回す
* 機密情報を誤ったチャネルに貼る
* 「ちょっと急ぎだから」と承認なしで本番を触る
* 退職時の権限取り消しが漏れる
* USBメモリを拾ってPCに挿す

どれだけ技術的ガードを固めても、人間側の1回のミスで穴が開く。技術セキュリティと人的セキュリティのバランスが崩れた瞬間、後者から先に崩れるのが常だ。

この観点で、AIエージェントというものを「セキュリティリスクを増やすもの」ではなく、「人間の判断ミスを構造的に減らしうるもの」として見ることもできる。OpenClawの設計で言えば：

* exec承認ゲート：危険なコマンドを人間が判断するのではなく、AIが「これは危険な操作だから確認してください」と構造的に止める
* DMペアリングシステム：「知らない人からのDMに気をつけて」と教育するのではなく、承認フローに載せないと物理的に届かなくする
* ツール拒否リスト：「このチャネルではシェルを触らないで」と運用ルールで縛るのではなく、そのエージェントには exec ツール自体を与えない
* プロンプトインジェクション検査：AGENTS.mdに悪意の指示が混入しても、AIがそれに従う前に遮断する

人間を信頼する運用ではなく、人間が判断する前に構造で止める運用。これは従来の「ルール＋教育」型セキュリティの限界を超える発想で、AIエージェント時代に特有の強みでもある。

逆に言えば、AIエージェントを導入すると「人間がうっかりやってしまう判断ミス」の発生面が変わるだけで、消えるわけではない。だからこそログ追跡が肝で、AIが何をどう判断したかも含めて全部記録される必要がある。OpenClawがセッションをすべて永続化している意味は、ここにもある。

#### 組織の摩耗の正体は、ほぼコミュニケーションコスト

もう一段抽象度を上げた話をしたい。企業の内部コストのうち、相当な割合はコミュニケーション起因だと自分は考えている。

* 「これって誰に聞けばいいんだっけ？」で30分溶ける
* Slackの通知が飛び交って、どれが自分宛か分からない
* 同じ質問が別チャネルで3回繰り返される
* ベテランが退職して、暗黙知と一緒に連絡網も消える
* 意思決定が「確認待ち」で半日止まる

これらはすべてコミュニケーションの摩耗だ。機能開発の難しさでもなければ、技術的負債でもない。情報が誰から誰へ、どのタイミングで、どの形式で流れるかが整っていないことに起因する損失。

OpenClawのハーネスは、まさにここを解くために設計されている。Gateway中心のメッセージルーティング、24チャネル統合、マルチエージェントの役割分離、DMペアリングによる信頼境界、すべて「組織のコミュニケーションレイヤーにAIを滑り込ませる」ための部品だ。Claude CodeやHermesの設計では、ここまで踏み込んだマルチチャネル・マルチエージェントの運用はできない。

だからこそ、一般的なビジネスユーザーのユースケースのほとんどは、OpenClaw1層で十分カバーできると実感している。開発者向けの3層モデルは Claude Code + OpenClaw + Hermes で組むが、普通の組織の業務改善文脈なら、OpenClawを Slack/Teams に常駐させるだけで最初のインパクトは十分出る。コード生成も個人ナレッジも後回しでいい。まず組織の配管を直す。

実際、筆者の関わるForOne Holdingsでは、OpenClawを基盤にした [「AI社員配属」サービス](https://openclaw.forone-ai.com/) をこの考え方で提供している。Slack/Teamsに"チームの同僚"として常駐するAIが、日々の確認・依頼・共有・ベテランノウハウの継承まで肩代わりする、というサービス設計は、まさに「コミュニケーション摩耗の解消」を価値の中心に置いた例だ。OpenClawの設計思想が、商用サービスとして成立するだけの実用性に到達していることの裏付けでもある。

### 1年後、この構図はどう動くか

最後に予想を置いておく。この3層の関係は、今後1年で次のいずれかに収束する可能性が高いと見ている。

1. 統合の波：3つのうちどれかが他の2つの重心領域を取り込み、境界が曖昧になる。例えばHermesが強力なチャネル統合を実装し、OpenClaw的な領域に食い込むシナリオ
2. 標準化の波：MCPのような共通規格がさらに成熟し、3つのエージェント間で知識とスキルを直接やり取りできるようになる。「Hermesのスキル」と「OpenClawのプラグイン」が相互運用可能になる未来
3. 分化の加速：逆に、それぞれの重心がさらに尖って、併用前提のエコシステムがツール側で整備される。知識フローの配管が標準機能として提供される

個人的には 2 を期待している。3層モデルの価値は、各層の独立性と相互接続の両立にある。MCPが「エージェント間の共通言語」として機能するなら、配管を自分で作り込まなくても循環が起きる世界になる。そこまで来れば、「どのエージェントを選ぶか」ではなく「どの組み合わせで何を解くか」が主題になる。

いずれにせよ、2026年のAIエージェント運用は「一つを極める」から「複数を使い分ける」フェーズに入った。ハーネスの設計思想を読み解く力は、そのまま「何をどこに投げるか」の判断力になる。本記事がその判断力を磨く一助になれば嬉しい。
