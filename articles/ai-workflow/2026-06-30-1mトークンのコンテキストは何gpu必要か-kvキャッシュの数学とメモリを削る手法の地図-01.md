## はじめに

1Mトークンのコンテキストウィンドウが珍しくなくなりました。Claude Opus 4.8 は1M全域がフラット課金（900kトークンのリクエストも9kと同じ単価）で提供されています。利用者から見ると「とにかく全部放り込めばいい」と錯覚しがちですが、これをナイーブに自前で動かそうとすると、必要なGPUメモリは現実離れした量になります。

この記事では、(1) なぜコンテキストがGPUメモリを食うのか、(2) どれくらい食うのか、(3) それを削るためにどんな手法があるのか、を整理します。最後に「GPUを持たない利用者（API利用）」の視点でも何ができるかをまとめます。

## なぜコンテキストがGPUメモリを食うのか

推論時のGPUメモリは大きく「モデル重み + KVキャッシュ + アクティベーション + オーバーヘッド」に分かれます。このうちコンテキスト長に効くのはほぼ **KVキャッシュ** だけです。重みはコンテキストに依存せず固定で、アクティベーションはFlashAttention系で抑えられます。

ここで重要な誤解があります。アテンションの**計算量**は系列長に対して O(n²) ですが、KVキャッシュの**メモリ**は O(n)、つまり**線形**です。FlashAttention系はアテンション行列全体をメモリに実体化しないため、保持するのは各層のキー K とバリュー V のテンソルだけで、これは系列長に比例します。

KVキャッシュのサイズは次の式で表せます。

\mathrm{KV} = 2 \times L \times h\_{kv} \times d\_{head} \times s \times b \times \mathrm{bytes}

* 2 … K と V
* L … 層数
* h\_{kv} … KVヘッド数（MHA か GQA/MQA かで激変します）
* d\_{head} … ヘッド次元
* s … 系列長（＝コンテキスト長。注目の変数）
* b … バッチサイズ
* \mathrm{bytes} … 要素あたりバイト数（BF16で2、FP8/INT8で1）

## どれくらい食うのか

具体例で見ていきます。Llama 3 70B（GQA, h\_{kv}=8, L=80, d\_{head}=128, BF16）の場合、1トークンあたりのKVキャッシュは次のようになります。

```
2 × 80 × 8 × 128 × 2 = 327,680 bytes ≈ 320 KiB/token
```

128Kコンテキスト・バッチ1で約 **40 GiB** です。8Bモデルでも128Kで約16 GiBになります。系列長に対して完全な直線なので、128K → 1M でKVキャッシュは約8倍になります。

では1M・フロンティア級を仮に見積もってみます（層96・h\_{kv}=16・d\_{head}=128・BF16という**あくまで一例の仮定**です）。1トークンあたり約768 KiB、1Mで **KVキャッシュだけで約786 GB** になります。重みを600BパラメータのBF16とすれば1.2 TB、合計で約2 TB。NVIDIA B200（192GB）で**約11枚**、収容するだけでこの規模になります。

ここで読み取るべき本質は、**1Mスケールではバッチや系列が伸びるとKVキャッシュがモデル重みに匹敵・凌駕する**ことです。重みは固定ですが、KVは s \times b で膨らみます。ですから長コンテキスト最適化＝KVキャッシュ最適化と言ってよいでしょう。

## メモリを削る手法の地図

ほとんどの効率化手法は、上の式の**どの項を叩くか**で整理できます。

| 削減する項 | 手法 | 代表的な論文 |
| --- | --- | --- |
| h\_{kv} を減らす | MQA / GQA / MLA | DeepSeek-V2 MLA (arXiv:2405.04434) |
| \mathrm{bytes} を減らす | KVキャッシュ量子化 | KIVI (2402.02750) / KVQuant |
| s（実効）を減らす | eviction / sparsity | H2O (2306.14048) / StreamingLLM (2309.17453) |
| L を共有 | 層間KV共有 | CLA / xKV (2503.18893) |
| 式ごと消す | SSM / 線形Attention | Mamba (2312.00752) |
| 式の外側 | メモリ管理 | PagedAttention / vLLM (2309.06180) |

### h\_{kv}：アーキテクチャでヘッドを減らす

最も枯れた道です。MQA（Shazeer 2019）と GQA（Ainslie 2023）はKVヘッドを共有してキャッシュを縮めますが、性能を犠牲にしがちです。DeepSeek-V2 の **Multi-head Latent Attention (MLA)** はこれを置き換える低ランクKV同時圧縮で、MHAより高性能でありながらKVキャッシュを小型モデルで約14%、大型MoEで約4%にまで縮めます。最近のモデルが軒並みGQA以降を採用しているのはこのためです。

### \mathrm{bytes}：KVキャッシュを量子化する

**KIVI** はキーをper-channel、バリューをper-tokenで量子化するのが最適という分析から、チューニング不要の2bit量子化を提案しています。重み込みで2.6倍少ないピークメモリ、最大4倍のバッチ、2.35〜3.47倍のスループットを報告しています。**KVQuant** はper-channelキー量子化と専用CUDAカーネルで、LLaMA-7Bを単一A100-80GBで最大100万トークン、8GPUで1000万トークンまでサービングできるとしています。ただし2bitは難タスク（MATH, GPQA等）で精度が落ちるため、ビット幅は品質とのトレードオフになります。

### s（実効）：重要なトークンだけ残す

**H2O** はアテンションスコアの大半をごく一部のトークン（heavy hitters）が占めるという観測に基づき、直近トークンとheavy hitterを残すevictionを提案しています。heavy hitterを2割残す構成でスループット最大29倍を達成します。**StreamingLLM** は別角度で、系列冒頭のトークンが意味に関わらず性能維持に効く「attention sink」現象を発見し、sink＋直近ウィンドウで無限長を有界メモリで処理します。SnapKV、TOVA、PyramidKVなども同系統です。

### L：層間でKVを共有する

Cross-Layer Attention (CLA) や **xKV** は隣接層のKVの冗長性を使って共有します。xKVはMLAの上にさらに3〜3.5倍の追加圧縮を、精度を落とさず達成すると報告しています。

### 式ごと消す：SSM / 線形Attention

**Mamba** の発想は明快です。「アテンションは文脈を一切圧縮しない（＝全KVを保持する）から非効率なのであって、有限状態の再帰モデルなら定数メモリ・線形時間の推論になる」というものです。Mambaはそもそもキャッシュが無いため大きなバッチが使え、同規模Transformerの4〜5倍のスループットを出します。実用では純粋SSMより hybrid（Jamba等）が主流で、64Kで full transformer の10〜15%のKVしか使いません。ただし固定状態への圧縮は損失を伴うため、全履歴を参照できるアテンションに比べると非構造データからの情報抽出が苦手という本質的な代償があります。

### 式の外側：PagedAttention

**PagedAttention / vLLM** は総量を減らすのではなく無駄を消すアプローチです。OSの仮想記憶・ページングに着想を得てKVキャッシュをブロック単位で管理し、断片化・予約過多による浪費をほぼゼロにして、同レイテンシで2〜4倍のスループットを実現します。いまや vLLM・TGI・TensorRT-LLM に採用される業界標準です。

## 落とし穴：圧縮率と正確性のトレードオフ

これらの手法には共通のリスクがあります。**圧縮は均一には劣化しません**。eviction系は「普段ほぼ注目されないが生成時に必須になる休眠トークン」で壊れます。最近の研究は、資格情報（credential）の検索のようなタスクで、アテンションスコアに頼る既存手法が攻撃的な予算下で軒並み0%の検索率になると報告しています。2bit量子化は難タスクで落ち、SSMはretrievalで弱くなります。

つまり「圧縮率を上げると、稀だが決定的な情報が静かに落ちる」のが中心的なリスクです。意味チャンク単位で残すChunkKVのように、何を残すかを賢くする方向の研究が続いているのはこのためです。

## 利用者（API利用）側からは何ができるか

GPUを持たない利用者は、上の物理層（量子化・MLA・SSM・paging）には触れません。これらは基盤側の選択です。しかし**コンテキストそのもの**は減らせます。ここで効くのは「物理層（KVテンソル、ホワイトボックス）の手法を、論理層（トークン列、ブラックボックス）に持ち上げる」発想です。

論理層でも同じ3操作が再現できます。

* **drop（間引く）** … 重要トークンだけ残します。物理層のH2O/StreamingLLMに対応します。ただしアテンションが見えないので、recency・埋め込み類似度・perplexityなどの**意味的プロキシ**で代用します。プロンプト圧縮の **LLMLingua** が代表で、小さな言語モデルのperplexityで低情報トークンを除去し、最大20倍の圧縮をブラックボックスLLMに適用できます。
* **compress（圧縮する）** … 冗長なスパンを要約で凝縮します。物理層のMLA/量子化に対応するテキスト版ですが、トークンは離散なので「要約」が現実的な手段になります。
* **externalize（外に出す）** … 増え続ける履歴を固定サイズの外部メモリに退避し、必要時にrecallします。物理層のSSM（O(1)状態）に対応します。**MemGPT** はOSの階層メモリ・ページングに着想を得て、外部ストアとの間でコンテキストをページイン/アウトします。

おもしろい対応として、PagedAttentionが「KVテンソルのためのOSページング」だったのに対し、MemGPTは「トークン窓のためのOSページング」になっています。同じメタファが1層上で回っているわけです。

加えて利用者側だけが持つ強力なレバーが **retrieval（RAG）** です。eviction が「入れた後に減らす」のに対し、retrievalは「そもそも入れない」。これは入力レベルのスパース化と言えます。

### よくある誤解

ここで重要な注意があります。コンテキストの**サイズ**を実際に減らす操作と、減らさない操作を取り違えやすいのです。

* **プロンプトキャッシュはサイズを減らしません**。キャッシュは再計算（prefill）を安くするだけで、トークンは依然として窓を占有します。
* **配置の分離（安定部を前、可変部を後ろ）もサイズを減らしません**。これはコストとキャッシュの最適化であってメモリ削減ではありません。
* サイズを実際に減らすのは、**常駐情報を書き減らす・参照ボディをretrievalに出す・処理中の情報（ツール結果など）を都度トリムする**、の3つに集約されます。

## まとめ

* コンテキストがGPUメモリを食う本体はKVキャッシュで、系列長に対して**線形**に増えます。1Mスケールではバッチや系列が伸びるとKVキャッシュが重みを凌ぎます。
* 削減手法は「式のどの項を叩くか」で地図化できます：h\_{kv}（MQA/GQA/MLA）、\mathrm{bytes}（量子化）、s（eviction）、L（層間共有）、式ごと消す（SSM）、式の外側（PagedAttention）。
* どの手法も「圧縮率と正確性」のトレードオフを抱えます。休眠トークンの取りこぼしが代表的な失敗モードです。
* 利用者側は物理層に触れなくても、論理層で **drop / compress / externalize / retrieval** を実装できます。ただし「サイズを減らす操作」と「コストを減らす操作」を混同しないことが肝心です。

1Mは「天井」であって「作業予算」ではありません。最大コンテキストより、必要十分に絞った **bounded context** のほうが、メモリ・コスト・品質のすべてで有利になります。
