* 従来のサポート業務は複数ツールをまたぐ**コンテキスト収集の非効率**が

## 技術サポートの何がボトルネックだったのか

技術サポートエンジニアの仕事は「回答する」ことより、  
**「回答に必要な情報を集める」ことに時間がかかる。**

従来のサポート対応フローはこうだった。  
顧客からエラー報告  
↓  
Datadogでログを確認（ツール1）  
↓  
Slackで過去の類似スレッドを検索（ツール2）  
↓  
Notionで仕様書・手順書を確認（ツール3）  
↓  
Linearでエンジニアチームのチケットを確認（ツール4）  
↓  
顧客へ返信を作成

**問題は「どのボタンは何を制御しているのか？」という問いに  
答えるために、フロントエンドのロジック・バックエンドの  
ポリシーチェック・ドキュメント全体を別々のツールで  
調べなければならなかったこと。**

関連するリポジトリを1つのワークスペースにまとめることで、  
1つのスレッドで調査が完結するようにした。

Anysphereが公開したサポートフローは以下の構成だ。

### MCPサーバーで情報源を一元化

# Cursor サポートセッション：接続MCPサーバー一覧

現在、以下のプラットフォームおよびサービスがMCPサーバーとして接続されています。

| No. | カテゴリ | 接続対象 | 主な取得データ・情報 |
| --- | --- | --- | --- |
| 01 | 顧客管理 | 顧客データベース | サブスクリプション情報、チーム設定、プライバシー設定 |
| 02 | 運用・分析 | ストリーミングイベントログ | 利用サービス、テレメトリ、ネットワーク詳細 |
| 03 | コミュニケーション | Slack | スレッド履歴、顧客の関与状況（エンゲージメント） |
| 04 | 開発管理 | エンジニアリングチケット | 類似問題のトラッキング、チーム横断的なバグ調査 |
| 05 | ナレッジベース | 社内ドキュメント | 利用手順書、トラブルシューティングガイド |

これら6つの情報源を統合することで、サポートエンジニアは  
**複数ツールを行き来せずに必要な情報を1セッションで収集**できる。

## 4つのサブエージェントによる自動化フロー

サポートプロセスの各ステップを並列・高速実行するために、  
**4つの専門サブエージェント**が連携する。

# エラー対応ワークフロー：自動化されたサポート解決プロセス

| ステップ | ツール名・内容 |
| --- | --- |
| ① | **LogInvestigator** Datadogでログをトレース → 障害ポイントと周辺の証跡を収集 |
| ↓ |  |
| ② | **KnownIssueMiner** SlackとNotionをスキャン → 過去の類似スレッド・ 　ワークアラウンドを抽出 |
| ↓ |  |
| ③ | **TicketWriter** 収集した証跡を整形 → 完全なエスカレーション 　チケットを自動生成 |
| ↓ |  |
| ④ | **CustomerReplyDrafter** 社内情報の詳細をもとに → 顧客への返信メッセージを生成 |

```
4つのサブエージェントの結果を1つの出力にまとめ、
```

**サポートエンジニアが最終レビューして送信するだけ**になった。

## Rules・Skills・スラッシュコマンドで繰り返し作業を自動化

繰り返し発生する定型作業には**スラッシュコマンド**を定義して運用している。

| コマンド例 | 役割 |
| --- | --- |
| `/create-ticket` | サポートチケットの作成 |
| `/reply-customer` | 顧客への返信案の生成 |
| `/find-known-issue` | 過去の類似事例検索 |
| `/search-logs` | ログの検索・分析 |

さらに\*\*Rules（ルール）とSkills（スキル）\*\*を組み合わせて  
以下のプロセスの一部を自動化している。

* 顧客向け返信案の精査（安全に送れない返信内容を自動チェック）
* 適切なバグチケットの作成（必要情報を整理した状態で生成）
* 既知の問題の調査（SlackとNotionを横断して自動検索）

## 現場エンジニア目線の考察

半導体の現場でも、技術サポートや不具合対応は  
**「情報を集めること」に時間の大半が消える**という経験は共通だ。

装置のログ・仕様書・過去の不具合報告・設計変更履歴──  
それぞれが別のシステムに散在していて、  
ベテランエンジニアしか「どこに何があるか」を知らない。

Anysphereが実践したアプローチで特に参考になるのは  
\*\*「サブエージェントに明確なスコープと出力形式を持たせる」\*\*という設計だ。

これは前回紹介したGitHubのマルチエージェント設計原則と完全に一致している。  
「アクションを事前に限定する」「中間状態をログに残す」という構造が  
あるからこそ、サポートエンジニアが安心してAIの出力をレビューできる。

一方で懸念もある。

* **顧客データをMCPで広く参照する**構成は  
  情報漏洩リスクと権限管理の精度が問われる
* **AIが生成した返信をそのまま送る**リスク──  
  最終レビューが形骸化すると顧客対応の品質が下がる
* **自社ツールで自社のサポートをする**というのは  
  ドッグフーディングとして理想的だが、  
  同じ構成を他社が再現するには相応の設計コストがかかる

## まとめ・次のアクション

**結論：AIエージェントがサポートを5〜10倍速にした本当の理由は  
「AIが賢いから」ではなく「情報源を統合してエージェントに  
明確な役割を与えたから」だ。**

今すぐ参考にできることは以下だ。

1. **自分のチームの繰り返し作業をリストアップする**  
   （スラッシュコマンド化できるものが必ずある）
2. **情報が散在しているツールをMCPで繋ぐことを検討する**  
   （Slack・Notion・Jira・Datadogはすでに対応済み）
3. **サブエージェントに「役割・スコープ・出力形式」を明確に定義する**  
   （曖昧な指示では効果が出ない）
4. **最終レビューを形骸化させない仕組みを作る**  
   （AIの出力を人間が確認するプロセスを必ず残す）

「Cursorで作業する」から  
「CursorにCursorを使わせる」へ。  
自社ツールを使い倒す姿勢がそのままプロダクトの改善につながっている。
