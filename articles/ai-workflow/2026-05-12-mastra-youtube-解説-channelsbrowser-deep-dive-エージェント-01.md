## 概要

Mastra で最近追加された `browser` と `channels` は、単に「ブラウザ操作ができます」「Slack につなげられます」という機能紹介だけでは終わらないプリミティブです。

`browser` と `channels` の過去の紹介記事はこちら  
<https://zenn.dev/shiromizuj/articles/45319f8282b6e4>  
<https://zenn.dev/shiromizuj/articles/b09eee651a06e9>

この動画では Mastra CTO の Abby Ayer が、その2つを「カーソル」と「受信箱」にたとえながら、なぜエージェントに両方が必要なのか、なぜ既存のブラウザ自動化やチャット連携をそのまま使うだけでは不十分なのかを、かなり丁寧に分解して説明しています。

前半はブラウザ編で、HTML をそのまま読ませる方式やスクリーンショット頼みの方式がなぜ限界を持つのかを踏まえつつ、アクセシビリティツリーを使ったページ理解、Stagehand や Browserbase などの既存ツールを Mastra のプリミティブとしてどう包んだかを解説します。後半は Channels 編で、Slack 連携に見える裏側の「transport」「会話の継続性」「添付ファイルや承認 UI」といった泥臭い問題を列挙したうえで、それを1行の設定で扱える形にする意味を示します。最後は両者を組み合わせ、Slack 上でフライトを見つけて承認付きで予約させるデモまで見せる、実装者向けの濃いワークショップでした。

## 要点

1. **エージェントが現実世界で動くには「外へ出る手段」と「外から呼ばれる窓口」の両方が必要**。その2つを Mastra では browser と channels として切り出している。
2. **ブラウザ自動化そのものは既に存在するが、エージェントに自然に使わせるには別の統合問題がある**。ブラウザインスタンスの寿命、共有、可視化、状態保持などは単なる Playwright 呼び出しでは片付かない。
3. **ページ理解の中心は生 HTML でもスクリーンショットでもなく、アクセシビリティツリーにある**。意味が圧縮され、要素を選択しやすく、LLM にとってトークン効率が良いからだ。
4. **Mastra は特定のブラウザ実装を押し付けるのではなく、抽象クラスとして Browser Primitive を用意している**。Stagehand、Agent Browser、Browser Use などを同じ土台で扱えるようにしている。
5. **ブラウザエージェントの基本ループは observe → reason → act の繰り返し**。しかしそのたびにシステムプロンプトを書き換えると、キャッシュ効率を壊して無駄に高くつく。
6. **そのため Mastra は browser state を system prompt ではなく system reminder として積み増す**。ブラウザの状態は変わるが、土台のプロンプトは安定させる設計だ。
7. **Channels の本質は「Slack に返信できること」ではなく、各メッセージング基盤ごとの面倒な差分をエージェントから隠すことにある**。Webhook、再送、認証、スレッド継続、ユーザー識別、添付、ストリーミング更新などをまとめて吸収する。
8. **チャット空間では複数人が同時に話すため、誰が何を言ったかをエージェントが識別できないと会話は破綻する**。Channels は sender ID と thread の対応づけまで面倒を見る。
9. **ツール承認 UI をチャット面に埋め込めることが、browser と channels を組み合わせたときの強み**。Slack をそのままエージェントの実行 UI にできる。
10. **この動画の価値は新機能の紹介よりも、プリミティブ設計の勘所を言語化している点にある**。ブラウザもチャットも「つなげば動く」ではなく「状態と責務をどこで持つか」が本題だった。

## 詳細

### 「カーソルと受信箱」というたとえ

Abby はこのワークショップを、「エージェントにはカーソルと受信箱が必要だ」というたとえから始めます。ブラウザは、エージェントが外の世界に手を伸ばすためのカーソルです。ページを開き、クリックし、入力し、情報を抜き出す。逆に channels は、世界の側からエージェントを呼び出すための受信箱です。Slack や Discord や Telegram のような、すでに人が居る場所にエージェントを置く。

この整理が面白いのは、browser と channels を「別々の機能」ではなく、エージェントの外界接続の両面として見せているところです。API や Studio の中に閉じたエージェントは、賢いチャット UI に過ぎません。そこから一歩進めて、外へ動き、外から呼ばれる形にしたいときに必要なのが、この2つのプリミティブだというわけです。

既に browser 単体や Slack bot 単体の紹介記事を読んでいたとしても、この動画を見ると「ああ、Mastra はここを一つの設計問題として捉えているのか」がかなり明確になります。

---

### ブラウザ編の出発点: Playwright があっても問題は終わっていない

ブラウザ自動化それ自体は、いまさら Mastra が発明したものではありません。Playwright、Puppeteer、Chromium ベースのテストや RPA ツールは既に豊富にあります。Abby もその点は率直で、「ブラウザ操作はある意味もう解決済み」と認めています。

ただし、**それをエージェントに使わせる**段階に入ると、急に問題の質が変わります。動画ではその統合上の論点が順番に挙げられます。

* ブラウザはどの寿命で起動すべきか。ツール呼び出しごとか、会話単位か。
* 複数エージェントが同じブラウザを触るとき、インスタンスは共有か分離か。
* ヘッドレスで動かすとしても、人間が途中経過を確認できる必要はないか。
* エージェントは今どの URL にいて、何をクリック済みなのかをどう覚えるのか。

つまり、単に `goto` と `click` を呼べることと、「エージェントにとって自然なブラウザプリミティブ」があることは全く別です。動画の良さは、この違いを「便利なSDKがあります」で流さず、状態管理と可視性の問題としてちゃんと切り出している点にあります。

---

### HTML 直読でも画像認識でもなく、アクセシビリティツリーが主役になる

この動画で最も印象に残る説明の一つが、**エージェントはどうやって Web ページを読むのか**というパートです。Abby はまず、素朴な3つの案を並べて、それぞれの限界を示します。

#### 1. HTML をそのまま読む

最初に思いつくのは `page.content()` のように HTML を丸ごと取得して LLM に読ませる方法です。小さなページなら何とかなるかもしれません。しかし現代的な Web アプリの HTML は巨大で、意味の薄い `div` やクラス名だらけです。レイアウト情報は大量にあっても、「どれが押せるボタンか」「どれが入力欄か」は、LLM にとって案外読み解きにくい。

しかもコンテキスト消費が重く、ページ全体をその都度読ませるとすぐに高コスト化します。単純にトークンが大きいだけでなく、意味密度が低いのです。

#### 2. スクリーンショットを見る

次に思いつくのが、スクリーンショットを撮ってビジョンモデルに見せる方法です。ヘッダーに何が書いてあるか、今どんな画面が見えているか、といった認識には有効です。

ただし、画面が「見える」ことと、「どこをどうクリックすべきか」が一貫して分かることは別問題です。画像だけでは、操作対象を安定して指定し続けるのが難しい。視覚的理解はできても、そこから精度良くアクションに落とすのが弱いわけです。

#### 3. アクセシビリティツリーを使う

そこで本命として紹介されるのがアクセシビリティツリーです。動画では `A11y` を強調しつつ、「昔はフロントエンドの人間もそこまで気にしていなかったけれど、いまになってこれがとても効いている」と語ります。

アクセシビリティツリーの利点は次の通りです。

* ボタンはボタン、テキストボックスはテキストボックスとして意味が落ちてくる
* スタイル用のノイズが消え、重要な要素だけが残る
* DOM 全体よりはるかにコンパクトで、トークン密度が高い
* 要素の識別と操作に直接つなげやすい

この説明は、browser support の記事を読んだだけでは分かりにくかった「なぜ semantic selector 的な世界観が重要なのか」をかなり腑に落ちる形で言い換えています。単に AI っぽいからではなく、**意味の圧縮と行動可能性の両立**が理由なのです。

### Stagehand だけではなく、Mastra は「差し替え可能なブラウザソケット」を作っている

動画では Stagehand、Browserbase、Browser Use、Agent Browser など複数のツールが紹介されます。ここでのポイントは、「Mastra の browser は一つの実装名ではない」ということです。

Abby は Mastra Browser を「ソケット」あるいは抽象クラスのようなものとして説明します。つまり、Mastra はブラウザ制御の世界を自前で完全再実装したいわけではなく、既存の優れたツールをエージェントに自然に接続するための土台を作っている、という立ち位置です。

動画内での整理はだいたい次のようなものでした。

* **Agent Browser**: アクセシビリティツリーを強く活かす決定論寄りのアプローチ
* **Stagehand**: 自然言語でブラウザ操作を指示しつつ、内部ではページ理解をうまくやるアプローチ
* **Browser Use**: Python 圏で先行してきたブラウザエージェント系の流れ
* **Browser Viewer**: CLI ベースのブラウザツールを Mastra Workspace で扱うための橋渡し

この説明が良いのは、Mastra が「どのブラウザツールが勝つか」を賭けているのではなく、**上位のエージェント抽象化に責任を持つ**という姿勢が見える点です。ユーザーから見ると browser primitive だが、その裏では provider を入れ替えられる。これは長期的にかなり賢い構えだと思います。

### observe → reason → act のループと、見落とされがちなキャッシュ設計

ブラウザエージェントの基本ループとして Abby が説明するのは非常にシンプルです。

1. **Observe**: 今見えているページ状態を取得する
2. **Reason**: 目的に対して何をすべきか考える
3. **Act**: クリック、入力、遷移、抽出などを実行する
4. 必要なら再び observe に戻る

ただ、この説明だけなら他のブラウザエージェントの話でも出てきます。この動画が一段深いのは、そのループを長く回すときの**コスト設計**まで話しているところです。

ブラウザ状態はどんどん変わります。URL も変わるし、前に押したボタンも変わる。だからといって、そのたびにシステムプロンプトを書き換えると、プロンプトキャッシュを毎回捨てることになります。これは以前の Mastra ワークショップでも繰り返し出てきたテーマですが、この動画ではそれを browser primitive の文脈に引きつけて説明しています。

Mastra が取っている手は、ブラウザの状態を「システムそのもの」ではなく、**system reminder 的な追加メッセージ**で渡すことです。つまり、エージェントの原則や役割は安定させたまま、変動するブラウザ状態だけを後ろから差し込む。これにより、ブラウザのような状態変化の激しいタスクでも、土台のプロンプトキャッシュを壊しにくくなります。

機能紹介記事では見落としがちな話ですが、実際に運用する人にとってはかなり本質的です。ブラウザ操作はただでさえループ回数が増えやすいので、ここを雑にするとあっという間に高くつきます。

### フライト検索デモが見せたもの: 「動いた」以上に「途中が見える」ことが重要

動画前半のデモでは、エージェントがフライト検索サイトを開き、SFO から Paris への片道便を探します。内容としてはシンプルなデモですが、見どころは結果よりもプロセスです。

Abby は headless に近い形で動いているブラウザに screencast を付け、Studio 上でエージェントの操作を追えるようにしています。これによって、observe と act のループが本当に回っていること、そして必要なら人間が途中で画面に割り込めることが見えるようになります。

このとき彼女が強調しているのは、ブラウザエージェントは完全自動がすべてではない、という点です。途中で CAPTCHA やログインが出ることもあるし、人間が「そこじゃない」と介入したいこともある。だから視覚的に追跡できること自体が、プリミティブとして重要だと分かります。

単に「ブラウザ操作できます」ではなく、「エージェントの行動を人が観察し、必要なら割り込める」まで含めて設計しているところが、このワークショップの現実感でした。

### Channels 編が本当に面白いのは、Slack bot を作るときの面倒を全部言語化しているところ

後半は channels の話に移ります。ここがまたよくできていて、いきなり「Slack につながります」と言わず、**自分で Slack bot をつなごうとすると何が起きるか**を一つずつ列挙していきます。

#### まず transport が面倒

Slack を例に取ると、イベント受信、インタラクション、Webhook、応答制限時間、署名検証、リトライ、重複排除など、最初の入口だけでも相当な作業量があります。HTTP 的には単純に見えても、実際には「届いたのに失敗扱いされた」ケースや再送の扱いが厄介です。

#### 次に conversation が面倒

DM と mention は同じなのか、thread reply はどうつなぐのか、どこまで履歴を backfill するのか。Slack のスレッドは会話単位に見えて、実際にはかなりカオスです。人が割り込み、話題がずれ、bot に話しているのか人に話しているのかも曖昧になる。

#### 最後に rich surface が面倒

テキストだけでなく、添付ファイル、画像、URL、絵文字リアクション、承認ボタン、ストリーミング時のメッセージ更新など、現代のチャットは表現面が広い。しかも「途中経過をストリーミングするなら同じメッセージを書き換えたい」など、UI 的な作法までプラットフォームごとに違います。

この列挙が効いているのは、channels が「Slack 対応」の別名ではなく、**チャット面をエージェントの実行環境にするための抽象化**だと伝わるからです。単なる adapter ではなく、エージェントが会話面で壊れずに動くための設計層になっています。

---

### 「1行でつながる」が価値を持つのは、その裏にある複雑さが大きいから

Mastra の channels は、動画中では `channels.adapters` に Slack adapter を足すイメージで紹介されます。Abby は「10分くらいで自分の agent を Slack に持ち込めた」とデモ前にかなり強調します。

この手の話は marketing 的に見えがちですが、動画全体の流れを見たあとだと印象が変わります。彼女は先に transport / conversation / rich surface の複雑さを全部見せているので、「1行でつながる」がどれだけ作業を圧縮しているかが分かるのです。

つまりここでの価値は、初期セットアップが短いことそのものより、**複雑さの所在を agent author から primitive 側へ押し戻した**ことにあります。エージェント作者は「どういうふるまいをさせたいか」に集中し、チャット基盤との glue code は channels が受け持つ。この分業がはっきり見えます。

---

### マルチプレイヤー会話では「誰が言ったか」を分かるだけでも難しい

channels の説明の中でとても良かったのが、Slack を「single-player world ではない」と言い切るところです。これは実際そうで、業務チャットは複数人が好きなタイミングで割り込みます。

たとえば bot に対して Alice が質問している途中で、Bob が「それもう解決したよ」と横から言うこともあるし、Carol が「Bob の言う通り」と被せることもある。こういう状況で、bot が最後の1メッセージしか見ていなかったり、誰が誰かを区別できなかったりすると、簡単に変な応答になります。

動画では、Mastra 側で sender ID と名前をきちんと渡し、thread との対応も維持することで、エージェントに「複数人が参加している」ことを伝える話が出ます。さらに system reminder によって、「毎回しゃべる必要はない」「他の人が対応中なら黙るべき」というルールも与えられます。

ここが面白いのは、channels が単なる入出力 I/O 層に見えて、実は**会話マナーの実装層**にもなっているところです。業務チャットに bot を置くとき、技術的な接続以上に「出しゃばりすぎないこと」が大切なので、この話はかなり実務的でした。

---

### browser と channels をつなぐと、Slack がそのままエージェント UI になる

ワークショップ後半の山場は、ブラウザプリミティブと channels を組み合わせたフライト予約デモです。Slack 上で bot に話しかけ、フライト URL を渡し、ブラウザで内容を確認させ、最後に承認 UI を出して予約させる。

ここで重要なのは、Slack が「エージェントに命令を送るチャット欄」ではなく、**エージェントの実行画面そのもの**に変わっていることです。

* 進行中のツール呼び出しが見える
* 承認ボタンが出る
* 応答はスレッドにまとまる
* 必要なら人間が止められる

この構成は、Agentic UI の一つの現実的な形だと思います。専用のアプリを毎回作らなくても、人がすでに1日中開いている Slack の中に、ブラウザ付きエージェントを住まわせられるからです。

もちろん Abby 自身も「ツールコールを全部そのままスレッドに出すのが理想とは限らない」と言っています。これはその通りで、実運用では見せ方をもっと整理したくなるはずです。ただ、このデモが示したかったのはそこではなく、**承認付きアクションをチャット面で完結できる**という構造の方でしょう。

### Q&A で見えた現実的な論点

最後の Q&A も、単なる補足ではなく面白い観点がいくつも出ていました。

#### Serverless との相性

channels は serverless 的なルートとして比較的扱いやすいが、browser はクラウド側のブラウザ実行基盤が必要になることが多い、という整理が出ます。これは実際その通りで、チャットイベントの受け口とブラウザ実行環境は別の制約を持っています。

#### 長いスレッドとメモリ

100件を超えるような thread にどう対応するか、という質問に対しては、channels だけでは解決せず、最終的には memory の問題になると答えています。これも重要で、channels は会話面を接続するが、長期文脈管理そのものは Mastra の memory primitives 側の仕事だ、という境界が見えます。

#### マルチエージェントの難しさ

「複数の agent を同じ thread に放り込むとかなりカオスになる」と Abby が正直に話していたのも印象的でした。multi-agent をやれるかどうかではなく、**会話面に複数 agent を同時参加させる設計はまだ荒れやすい**という現実をちゃんと認めています。こういう正直さはむしろ信用できます。

#### UI テストへの応用

Mastra Code から browser を呼んでスライドや UI を検証してみせた補足デモも良かったです。browser primitive はエージェントのためだけでなく、開発フロー上の smoke test や UI 検証にも使える、という含みが出ています。

## まとめ

この動画は browser と channels の「使い方」動画であると同時に、Mastra がどこをプリミティブとして切り出しているのかを説明する設計論の動画でもありました。単にブラウザを動かしたいなら Playwright を呼べばいいし、Slack bot を作りたいなら各プラットフォーム SDK を触ればいい。けれどエージェントにそれを**自然に、繰り返し使わせる**には、状態管理、可視化、会話継続、承認、ユーザー識別、キャッシュ効率といった別の問題が出てきます。

Abby はその「別の問題」をちゃんと列挙した上で、browser と channels を導入していることです。紹介済みの機能をなぞる動画ではなく、なぜその抽象化が必要だったのかを1時間かけて言語化したワークショップとして見ると、かなり見応えがありました。Mastra を使っていなくても、エージェントに現実世界への入出力を与えるとき何が面倒になるのかを学ぶ教材として十分価値があります。
