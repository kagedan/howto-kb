# 導入｜なぜAIエージェントは止まらないのか

AIエージェント連載の前回では、Function Callingによって、LLMが外部ツールを呼び出し、「行動」を実行できるようになる仕組みを整理しました。

しかし、ここで一つの限界があります。

Function Calling単体では、処理は一度実行されると、その場で終了します。つまり、それは「実行」はできても、「業務を進める存在」にはなりません。

実際の業務は、一度の処理で完結するものではなく、

* 状況を確認し
* 判断し
* 行動し
* 結果を受けて再び判断する

という連続した意思決定の積み重ねで成り立っています。

では、AIエージェントはなぜ、このような処理を「止まることなく」継続できるのでしょうか？

その答えは、LLMの性能ではなく、「状態を持って判断を繰り返す構造」にあります。

本記事では、AIエージェントを支えるControl Flow（ループ）の観点から、この仕組みの本質を整理していきます。

# よくある誤解｜Function Callingだけでは業務は回らない

Function Callingは、LLMが外部ツールを呼び出し、「行動」を実行するための仕組みです。

しかし、それ単体では業務を完結させることはできません。

Function Callingはあくまで「一回の行動」に過ぎず、呼び出して結果を返した時点で処理は終了します。

例えば、「在庫を確認する」という処理を考えてみます。

在庫確認という行動自体には意味がありますが、本来の目的はその先にあるはずです。

このように、結果に応じて次の行動を変えることで、初めて業務として成立します。

しかし、Function Calling単体には、

* 状態を保持する仕組み（State）
* 次に何をすべきかを判断する仕組み（Think）
* 実行結果をもとに再評価する仕組み（Observe）

が存在しません。

そのため、「一度の行動」はできても、「業務を進め続けること」はできません。

つまり、Function Callingは「行動」は実現できても、「意思決定の連続」は実現できないのです。

では、AIエージェントはどのようにして、この継続的な意思決定を実現しているのでしょうか。

# 本質｜AIエージェントは「ループ」である

AIエージェントは、単発の処理ではありません。

状態を持ち、判断し、行動し、その結果を受けて再び判断する。この一連の流れを繰り返すことで、業務を継続的に進めていきます。

この構造は、以下のように整理できます。

* State：現在の状況を保持する
* Think：次に何をすべきかを判断する
* Act：実際に行動する（Function Calling）
* Observe：結果を受け取り、状態を更新する

そして、この一連の流れが繰り返されることで、エージェントは業務を進め続けることができます。

これは、業務そのものが「状態に応じた意思決定の連続」であることに対応しています。

つまり、AIエージェントとは「状態を持ったループ構造」そのものです。

このループ構造こそが、AIエージェントにおけるControl Flowの本質です。

# ループ構造の分解｜State / Think / Act / Observe

では、それぞれの要素を整理していきます。

Stateは、現在の状況を表し、エージェントが次の行動を判断するための前提となる情報を保持する役割を持ちます。

Thinkは、Stateをもとに次に何をすべきかを判断する役割を持ちます。ここで、エージェントの意思決定が行われます。

Actは、Thinkによって選択された行動を実行する役割を持ちます。具体的には、Function Callingなどを通じて外部ツールやシステムを呼び出します。

Observeは、Actの実行結果を受け取り、Stateを更新する役割を持ちます。これにより、エージェントは次の判断に必要な最新の状況を把握することができます。

この一連の流れが、エージェントにおけるループの1サイクルとなります。

# Control Flowとは何か｜ループを成立させる仕組み

ここまで見てきた通り、AIエージェントは「ループ構造」によって動作します。

では、そのループはどのように制御されているのでしょうか。

このループの流れを定義し、制御する仕組みがControl Flowです。

つまりControl Flowの設計によって、エージェントが「どのように振る舞うか」が決まります。

Control Flowでは、主に以下のような観点を扱います。

* 終了条件（いつループを終えるか）
* 分岐（次にどの行動を選択するか）
* 例外処理（エラー時にどう振る舞うか）

これらを設計することで、エージェントは単なるループではなく、「意図を持って制御された処理」として動作するようになります。

例えばLangGraphでは、ノードとエッジによって状態遷移を明示的に定義することで、このControl Flowを表現します。

どの状態からどの行動に遷移するのかを明確に設計できるため、エージェントの挙動を制御しやすいという特徴があります。

# 実装イメージ｜最小のエージェントループ

ここでの目的は、AIエージェントのループ構造が、実際にはどのようにコードとして表現されるのかを見ることです。

題材は、在庫確認と発注を行う最小の在庫管理エージェントです。

* 在庫を確認する
* 閾値を下回っていれば発注する
* 結果を状態に反映する
* 必要なら次の判断に進む

この流れをコードにすると、エージェントループは次のように表現できます。

※前回の記事で扱ったコードを再掲し、ループ構造の観点から整理します。

第一回の一部抜粋

```
state = InventoryState(item_id="item-001")

loop_count = 0
max_loops = 3

while not state.done and loop_count < max_loops:
    loop_count += 1

    history = ChatHistory()
    history.add_user_message(
        f"""
あなたは在庫管理エージェントです。
次の業務を実行してください。

目的:
- inventory.get_stock を使って在庫を確認する
- 在庫が reorder_threshold を下回っていれば inventory.place_order を使って 20 個発注する
- 最後に結果を JSON のみで返す

現在の状態:
- item_id: {state.item_id}
- stock: {state.stock}
- reorder_threshold: {state.reorder_threshold}
- last_order_result: {state.last_order_result}
"""
    )

    result = await chat_completion.get_chat_message_content(
        chat_history=history,
        settings=settings,
        kernel=kernel,
    )

    parsed = json.loads(str(result))

    state.stock = int(parsed["stock"])
    state.done = bool(parsed["done"])

    if parsed.get("ordered"):
        state.last_order_result = parsed.get("summary", "ordered")
```

やっていることは一貫しています。

* State：現在の状態を保持する
* Think：次に何をすべきかを判断する
* Act：必要なツールを呼び出して実行する
* Observe：結果を受けて状態を更新する

そして、この一連の流れをwhileで繰り返しているだけです。

このwhileこそが、エージェントにおけるControl Flowの最小単位です。

また、state.doneやmax\_loopsは、ループを終了させるための終了条件として機能しています。

つまり、AIエージェントの本体は、複雑な魔法ではなく「状態を持ったループ構造」にあります。

# なぜループが業務を完結させるのか

では、なぜループ構造が業務を完結させるのかを整理します。

バックオフィス業務やオペレーター業務の多くは、単発の処理ではなく「判断の連続」によって成り立っています。

ある状態を確認し、その結果に応じて次の行動を決める。

そして、その行動の結果を受けて、さらに次の判断を行う。

つまり業務とは、「状態に応じた意思決定を繰り返す構造」として捉えることができます。

AIエージェントは、この判断の連続をループとして実装することで、業務そのものを処理できる存在になります。

例えば在庫管理では、

* 現在の在庫状況を確認する（State / Observe）
* 在庫が足りないかを判断する（Think）
* 発注を実行する（Act）

という流れになります。

この一連の流れは一度で終わるものではなく、結果を受けて再び状態を確認し、次の判断に進みます。

この「状態 → 判断 → 行動 → 再評価」のループが繰り返されることで、初めて業務が完結します。

だからこそ、業務を実行するためには、このループ構造が不可欠なのです。

# 実務への接続｜LangGraph / Semantic Kernelとの関係

ここまで見てきたように、AIエージェントはループ構造によって動作します。

では、このループは実際のフレームワークではどのように扱われるのでしょうか。

ポイントは、「ループをどのように制御するか」にあります。

代表的なアプローチとして、LangGraphとSemantic Kernelを比較してみます。

## LangGraph｜ループを明示的に制御する

LangGraphでは、エージェントの処理をノードとエッジで表現し、状態遷移をグラフとして定義します。

つまり、

* どの状態から
* どの行動に遷移し
* 次にどこへ進むのか

を明示的に設計します。

このため、

* 業務フローが明確に決まっている
* 分岐や例外処理を厳密に制御したい

といったケースでは、挙動をコントロールしやすく、検証・運用がしやすいという特徴があります。

## Semantic Kernel｜ループをLLMに委ねる

一方でSemantic Kernelでは、

* Function Calling（Act）
* 状態管理（State）
* プロンプトによる意思決定（Think）

を組み合わせることで、エージェントを構成します。

Control Flowの一部はコードで制御されますが、「次に何をするか」という意思決定はLLMに委ねられます。

そのため、

* 柔軟な判断が必要な業務
* あらかじめフローを固定できないタスク

に適しています。

ただしその分、エージェントの挙動はプロンプト設計に強く依存するため、「どのように判断させるか」を明確に定義することが重要になります。

つまり、ループをどこまで明示的に制御したいかによって、選択すべきフレームワークは変わります。

# よくある落とし穴｜ループ設計を間違えると壊れる

ループ設計を誤ると、AIエージェントは正常に動作しなくなります。

特に注意すべき問題は、大きく3つあります。

## ① 無限ループによる制御崩壊

終了条件が適切に設計されていない場合、エージェントは処理を止めることができず、ループを繰り返し続けます。

これは単なるバグではなく、Control Flowの設計ミスによるものです。

## ② API呼び出しの増加によるコスト問題

AIエージェントのループは、Azure OpenAIやAmazon Bedrock、Vertex AIなどのモデル呼び出しを繰り返す構造になります。

そのため、ループが想定以上に回ると、APIコールが増加し、コストが急激に膨らむ可能性があります。

ループが1回増えるごとに、モデル呼び出しも1回増えるため、コストはループ回数に比例して増加します。

業務改善のために導入したはずが、逆にコストを押し上げてしまうケースも珍しくありません。

## ③ 状態の不整合による意思決定の暴走

ループの中でStateが正しく更新されない場合、誤った状態を前提に判断が繰り返されます。

その結果、

* 本来不要な処理を繰り返す
* 同じ行動を何度も実行する
* 意図しない判断を続ける

といった「意思決定の暴走」が発生します。

これらの問題はすべて、「ループ構造をどう設計するか」に起因します。

AIエージェントはループによって業務を完結させますが、同時にループ設計を誤ると簡単に壊れる構造でもあります。

そのため、Control Flowの設計は、エージェント開発において最も重要なポイントの一つになります。

# まとめ｜AIエージェントは「動き続ける構造」である

AIエージェントの本質は、「動き続ける構造」にあります。

LLM単体では、Function Callingによって外部ツールを呼び出すことはできても、それは単発の処理にとどまります。

一方でAIエージェントは、ループ構造を持つことで、

* 状態を保持し
* 判断し
* 行動し
* 結果をもとに次の判断を行う

という一連の流れを繰り返し、継続的に意思決定を行うことができます。

これにより、単なる応答生成ではなく、「業務を進め、完結させる」ことが可能になります。

つまりAIエージェントは、「業務を実行する主体」として機能するようになります。

AIエージェントとは、「状態を持ち、意思決定を繰り返すループ構造」である。

また、フレームワークによってループの制御方法は異なります。

* 明示的に制御する（LangGraph）
* LLMに意思決定を委ねる（Semantic Kernel）

この違いを理解した上で、自分たちの業務特性に合わせて適切に選定することが重要です。
