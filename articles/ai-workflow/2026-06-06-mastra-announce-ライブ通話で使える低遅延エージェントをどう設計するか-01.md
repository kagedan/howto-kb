## 今回の話をひとことで言うと

今回 Mastra が紹介したのは、新しい API の発表そのものではありません。もっと実務的な話で、**ライブ通話中に人間を支援するエージェントでは「賢いが遅い」よりも「十分賢く、しかも新鮮なまま返せる」ことが重要だ**、という設計原則です。

題材になっているのは、Mastra ユーザーの Micro が Recall.ai のリアルタイム文字起こしを使って構築した、通話中の営業・商談支援エージェントです。記事によれば、目標は往復 3 秒未満です。ここでいう往復時間は、会話内容が流れ込んできてから、人間がその助言を実際に使える形で UI に出るまでの時間だと考えると分かりやすいでしょう。

この種のプロダクトでは、単に LLM を速くするだけでは足りません。重要なのは次の 4 点です。

1. 会話全文を毎回入れず、直近の重要部分だけを渡す
2. 文字起こし表示と推論結果表示を別レーンで流す
3. すべての partial 発話に反応せず、適切に間引く
4. 遅い補助処理は並列化し、間に合わなければ切り捨てる

要するにこれは、**低遅延エージェントの本質はモデル選定だけでなく、コンテキスト設計と入出力パイプライン設計にある**、という話です。

## 背景: なぜ音声・ビデオはレイテンシの「最終ボス」なのか

テキストチャットであれば、5 秒待たされてもギリギリ許容される場面があります。調査系エージェントなら 20 秒や 30 秒待つことすらあります。しかし、通話中の支援はそうはいきません。

人間同士の会話はターンテイキングで成り立っています。相手が質問し、こちらが数秒以内に反応しないと、「考えている」のではなく「答えられていない」ように見えてしまいます。しかも営業やサポートの通話では、返答が遅れること自体が不信感につながります。

ここで問題になるのは、単発のモデル推論時間だけではありません。

* 音声をテキスト化するまでの時間
* partial transcript が final transcript に固まるまでの時間
* 新しい文脈をプロンプトへ組み立てる時間
* 補助データ検索の時間
* LLM の生成時間
* UI がそれを描画し直す時間

つまり、ライブ通話支援は「1 回の LLM 呼び出し」ではなく、**連続して流れ込む入力イベントに対して、鮮度を落とさず助言を返し続けるストリーム処理**です。ここをテキストチャットと同じ設計で作ると、すぐ破綻します。

## 記事の核心 1: 文字起こしを「会話ログ」ではなくイベントストリームとして扱う

元記事で最初に強調されているのは、ライブ transcript を event stream として扱うことです。Recall.ai からは `transcript.partial_data` と `transcript.data` が飛んできます。これは要するに、

が別々に届くということです。

ここでやってはいけないのは、発話が少し増えるたびに「会議全文」を再構成し、それをそのまま次の推論へ渡すことです。全文を積み増していくと、プロンプトサイズも処理時間も単調に増えます。しかもライブで欲しいのは、多くの場合「今まさに何が話題になっているか」です。

そのため Micro の構成では、文字起こし自体は即座に UI へ流しつつ、推論用には別のループで直近範囲だけを切り出します。この記事では、クライアント側で直近約 60 発話、サーバー側でさらに直近約 4,000 文字に絞る、という多段の窓制御が紹介されています。

この設計は、Mastra で以前から強調されてきた**コンテキスト・エンジニアリング**の考え方そのものです。重要なのは「どれだけ多く入れるか」ではなく、**何を捨て、何を残すか**です。長い会議の全文を抱え込むより、直近の話題と圧縮済みの継続状態を持つ方が、むしろ正確で速くなります。

---

## 記事の核心 2: 長時間の継続性は「全文保持」ではなく compact state で持つ

とはいえ、ライブ通話が長くなると「直近だけ見ればよい」とも言えません。たとえば 45 分前に顧客が挙げた懸念点、10 分前に営業担当が約束したフォローアップ、前半で合意した要件などは、会話後半でも重要です。

元記事では、こうした長期の継続性が必要なら、次のような compact state を持つべきだとしています。

これはかなり重要です。低遅延システムでありがちな誤解は、「長期記憶が必要なら、とにかく過去履歴を全部突っ込むしかない」というものです。しかし実際には、必要なのは全文そのものではなく、**将来の判断に効く構造化された持ち越し情報**です。

Mastra のドキュメントでも、長い会話で raw message history をそのまま増やすとコンテキストウィンドウを圧迫し、性能劣化を招くと説明されています。そこで Mastra は `Memory` と `Observational Memory` を用意し、古い履歴を密な observation に圧縮して、長期記憶を保ちつつ入力サイズを抑える方向へ進んでいます。

ライブ通話支援の文脈で言い換えると、役に立つのは「過去 1 時間の全文」ではなく、たとえば次のような状態です。

```
type MeetingState = {
  customerGoals: string[];
  openQuestions: string[];
  commitments: string[];
  risks: string[];
  latestTopic: string;
};
```

こういう状態を別管理しておけば、ライブ推論には「直近の発話ウィンドウ + compact state」を渡せばよくなります。全文再送よりはるかに軽いし、何より論点の鮮度を保ちやすいです。

## 記事の核心 3: 文字起こし更新とエージェント出力を別チャネルに分ける

元記事では、出力を 2 つのチャネルに分けています。

* WebSocket: transcript 更新
* SSE: agent output

この分離は理にかなっています。文字起こし側は、partial / final の更新が高頻度で飛び、双方向のイベント処理とも相性がよいため、WebSocket が自然です。一方で、サーバーからクライアントへ一方向に流すだけの生成テキストは SSE で十分な場面が多いです。

面白いのは UI 側の見せ方です。まず raw streamed text を即座に見せ、その後に最終パース結果が届いたら cards / chips のような構造化 UI に差し替えます。

これは UX 的にかなりうまいやり方です。理由は 2 つあります。

1. まず「何か返ってきている」ことを最短で見せられる
2. 最終的には読みやすい形へ整えられる

リアルタイム UI では、最初から完璧な構造化データを待つと遅くなります。逆に、生テキストだけだと後から参照しづらい。なので、**最初は速度優先、最後に整形**という二段構えが効きます。

Mastra の streaming docs でも、`.stream()` は `textStream` による増分出力を返し、`text-delta`、`tool-call`、`tool-result` などのイベントを逐次扱えます。つまり今回の記事で語られている思想は、Mastra のストリーミング API が向いている問題設定とかなり一致しています。

---

## 記事の核心 4: partial ごとに推論しない

ライブシステムを初めて作ると、入力イベントが来るたびに即推論したくなります。ですが、それをやるとほぼ確実に無駄が増えます。

元記事では、

* 最初の数発話で起動
* その後はおよそ 5 発話ごと
* あるいは 30 秒ごと

という粒度で meeting intelligence ループを回しています。

ここでのポイントは、**イベントドリブンに見えて、実際には debounce / batching が入っている**ことです。partial transcript は途中で言い換えや修正が起きるので、そこへ毎回推論をぶつけても、すぐ古くなる中間結果を大量生産するだけになりやすいです。

低遅延設計では「毎回最速で反応する」ことより、「価値のある単位で間引いて反応する」ことの方が重要です。これはコストだけでなく、キャンセル制御にも効きます。古い入力を元に進んでいる推論を、新しい文脈が来た時点で中断できるからです。

---

## 記事の核心 5: enrich 処理は parallel + timeout + cancel が前提

元記事の終盤でとても大事なのが、遅いサイドタスクの扱いです。参加者検索、連絡先検索、vault 検索のような enrichment は並列で投げ、短い上限時間までだけ待ち、返ってきたものだけで先へ進みます。しかも、より新しい transcript 文脈が届いたら、古いリクエストは abort します。

これは「完全性」より「鮮度」を優先する設計です。

ライブ会話支援では、10 秒後に完璧な助言が返るより、2 秒後に 80 点の助言が返る方が価値があります。しかも 10 秒前の文脈で出した完璧な助言は、今の会話にはもうズレている可能性があります。つまり遅延は単なる UX 劣化ではなく、**意味の古さ**を生みます。

この観点から見ると、元記事の次の判断はどれも筋が通っています。

* 高速なサブエージェントモデルを使う
* `maxOutputTokens` を絞る
* 応答は完了待ちせずストリーミングする
* enrichment は並列化する
* 間に合わない enrich は諦める
* stale request は中断する

ここまでくると、もはや「推論を速くする」ではなく、**システム全体で freshness budget を守る**という設計になります。

## Mastra の機能とどうつながるか

今回の元記事は、Mastra の新機能紹介というより設計思想の共有です。ただ、Mastra の最近の機能群と並べると、かなりきれいにつながります。

### 1. Streaming

Mastra の `.stream()` はテキストを増分で返せますし、`text-delta`、`tool-call`、`tool-result` といったイベントも扱えます。ライブ UI では、最終結果を待たずに出し始められることが重要なので、ここは直接的に効きます。

### 2. Memory / Observational Memory

長い会話をそのまま抱え込まず、圧縮された observation を長期記憶として扱う発想は、今回の「rolling window + compact state」と非常に相性がよいです。ライブ経路では短い窓、長期継続性は別メモリ、という分離がしやすくなります。

### 3. Background Tasks

今回の記事では「遅いサイドタスクは live path を止めない」が重要でした。Mastra の Background Tasks は、長いツール呼び出しを会話の主ループから切り離し、必要なら `streamUntilIdle()` で後から結果を織り込めます。ライブ通話支援でも、非必須の enrich や後追い処理を分離する設計に応用できます。

### 4. Supervisor / Multi-Agent

高速な助言専用の軽量エージェントと、深掘り用の重いエージェントを分ける設計は、Mastra の multi-agent パターンと相性がよいです。ライブの fast path と、後追いの slow path を別エージェントに切り分けると理解しやすいでしょう。

## 既存手法・関連パターンとの位置づけ

今回の話は、単なる Mastra ローカルのテクニックではなく、リアルタイム AI システム全般に通じる設計パターンです。

### 全履歴プロンプト方式との違い

最も分かりやすい対比はこれです。

* 素朴な方式: 会話全文を毎回プロンプトへ積み増す
* 今回の方式: 直近ウィンドウ + compact state

前者は実装が簡単ですが、時間がたつほど重くなり、鮮度も落ちます。後者は state 設計が必要なぶん少し複雑ですが、ライブ性のある場面では明らかに有利です。

### 単一チャネル方式との違い

* 素朴な方式: すべて WebSocket か、すべて REST で処理する
* 今回の方式: transcript は WebSocket、生成結果は SSE

すべてを 1 本の仕組みで済ませたくなりますが、入力イベントと出力イベントは性質が違います。今回の分離は、システムの都合ではなくデータの性質に合わせてパイプを分けています。

### 完全性重視システムとの違い

* 素朴な方式: enrich が全部返るまで待つ
* 今回の方式: timeout で打ち切り、返ってきた分だけ使う

通話中支援では、後者の方が正しい場面が多いです。これは検索品質の問題ではなく、タスク定義の問題です。タスクが「最も正確なレポートを出す」ではなく、「人が次の一言を言えるようにする」だからです。

## 実装イメージをコードで捉えるとこうなる

元記事そのものは短いですが、概念をコード化すると次のような構成だと理解しやすいです。

```
type TranscriptEvent = {
  kind: "partial" | "final";
  text: string;
  speaker: string;
  at: number;
};

const rollingBuffer: TranscriptEvent[] = [];
let compactState = {
  summary: "",
  openQuestions: [] as string[],
  commitments: [] as string[],
};

function onTranscriptEvent(event: TranscriptEvent) {
  rollingBuffer.push(event);
  trimToLastUtterances(rollingBuffer, 60);
  broadcastTranscriptOverWebSocket(event);

  scheduleMeetingIntelligence();
}

async function runMeetingIntelligence() {
  const recentWindow = toRecentPromptWindow(rollingBuffer, 4000);

  const enrichments = await Promise.race([
    Promise.allSettled([
      lookupAttendees(),
      searchVault(),
    ]),
    timeout(3000),
  ]);

  return streamAdvice({
    recentWindow,
    compactState,
    enrichments,
  });
}
```

ポイントは、AI へ渡す情報の作り方がすでに「設計」であって、モデル呼び出しはその最後にあるということです。ライブエージェントでは、プロンプト以前の前処理が支配的になる場面が多いのです。

## まとめ

今回の Mastra 公式記事は短いですが、リアルタイム AI エージェントを作るうえで非常に重要な原則を端的に示しています。

特に重要なのは次の 5 点です。

1. **ライブ transcript は全文ログではなくイベントストリームとして扱う**
2. **推論には rolling window を使い、長期継続性は compact state へ逃がす**
3. **UI は stream-first で作り、最初は生テキスト、最後に構造化する**
4. **partial ごとの推論を避け、価値のある単位で間引く**
5. **遅い補助処理は並列・タイムアウト・中断を前提にする**

ライブ通話支援では、遅延は単に「待たせる」だけではありません。**会話文脈を古くし、助言の意味そのものを劣化させます。** だからこそ、低遅延設計はモデルのベンチマーク比較ではなく、入力イベント、コンテキスト圧縮、出力ストリーミング、補助処理の打ち切り戦略まで含めた全体設計として考える必要があります。

Mastra が最近強化している Streaming、Memory、Background Tasks、Supervisor 系の機能も、こうしたリアルタイム要件を支える部品として読むと、かなり立体的に見えてきます。今回の記事は新機能発表ではありませんが、**本番で使えるエージェントをどう設計するか**という意味では、むしろかなり本質的な内容だったと言えるでしょう。
