## はじめに

最近、AIコーディング文脈で「ハーネスエンジニアリング」や「ループエンジニアリング」という言葉をよく見かけるようになりました。

私は以前から、生成AI（主に Codex CLI）にいきなり「この機能を実装して」とはあまり頼まず、まずはAI自身に Goal Doc を作らせ、その Goal を満たすまで `/goal` で自走させる、というやり方をしています。

Goal Doc には、単なるタスク一覧ではなく、次のような内容を入れるようにしています。

* 達成したい最終状態
* 今回やらないこと
* 調査フェーズ
* 検証条件
* ADRを作る条件
* Skill化する条件
* 停止条件
* 人間レビューが必要な箇所

また、タスクが重い場合は実装自体をいきなり Goal にせず、まず「仕様を作ること」や「調査して設計判断を整理すること」を **Goal** にしています。

この記事では、最近の文献や公式ドキュメント参照しつつ、この私のやり方をあらためて整理してみようと思いました。

単なる Prompt Engineering ではありません。  
また、単なる Harness Engineering でも、単なる Loop Engineering でもありません。

言うなれば、次のようなものです。

```
AIエージェント用のハーネスの上で、Goal Docを中心に仕様駆動のループを回す開発フロー
```

ただ、最初に注意点を置いておきます。

これは「このやり方が最高あるいはベストプラクティスである」と主張する記事ではありません。  
ソフトウェアエンジニアの間ではすでに、Harness Engineering、Loop Engineering、Goals、ExecPlans、Spec-Driven Development、Agent Skills などがそれぞれ議論されています。

私のやり方は、それらを Goal Doc 中心に束ねた実践に近い、という位置づけです。

## AIコーディングは「プロンプトを書く」段階から「ループを設計する」段階へ移っている

従来のAIコーディングでは、人間が毎回こう指示していました。

```
この関数を直して
次はテストを書いて
エラーが出たから修正して
この設計だとまずいから変えて
もう一回テストして
```

これは一見AIに仕事をさせているように見えますが、実際には人間が細かいディレクションを続けています。

この状態では、人間がボトルネックになってしまっています。

Addy Osmani は Loop Engineering について、次のように書いています。

> “Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead.”  
> 出典: [Addy Osmani, “Loop Engineering”](https://addyosmani.com/blog/loop-engineering/)

つまり、Loop Engineering の本質は「人間が毎回プロンプトする係であり続ける」のをやめることです。  
人間が毎回 `next action` を指定するのではなく、AIが目的に向かって反復する仕組みそのものを設計する。

同じ記事では、loop を次のようにも説明しています。

> “A loop here can be thought of a recursive goal where you define a purpose and the AI iterates until complete.”  
> 出典: [Addy Osmani, “Loop Engineering”](https://addyosmani.com/blog/loop-engineering/)

これは、私の `/goal` 運用とかなり近いです。

私がやっているのは、AIに「次に何をするか」を毎回指示することではありません。  
まず Goal Doc で完了条件を定義し、その完了条件に到達するまでAIに反復させることです。

この意味で、私のやり方の中心はほぼ Loop Engineering です。

ただし、ここで重要なのは「どんな**Goal**を渡すか」です。

雑なGoalを渡せば、AIは雑にループします。  
曖昧なGoalを渡せば、AIは曖昧なまま自走します。  
検証不能なGoalを渡せば、AIは「完了したように見えるもの」を作ります。

したがって、AIにループさせる前に、Goalの品質を上げる必要があります。

## Prompt-first の限界

Microsoft は Spec-Driven Development の記事で、AIによってソフトウェア開発は速くなったが、速さだけでは良い成果は保証されないと述べています。

> “AI has made software delivery faster, but speed alone does not guarantee better outcomes.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

同記事では、Spec-Driven Development を、構造化されたspecを人間とAIの shared source of truth にするものだと説明しています。

> “Spec-Driven Development (SDD) addresses this by making structured specs the shared source of truth for both humans and AI.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

また、prompt-first な進め方について、次のように対比しています。

> “Instead of prompting first and aligning later, teams align first and let AI accelerate execution from a clear spec.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

これはかなり重要だと思います。

AIコーディングでありがちな失敗は、AIの性能不足だけではありません。  
人間側が「何を成立させたいのか」「何を壊してはいけないのか」「何で成功確認するのか」を明示していないことが多いです。

Microsoft は同じ記事で、曖昧さについてもかなり厳しく書いています。

> “AI can accelerate those steps, but it cannot correct ambiguity that was never resolved.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

AIは曖昧な要求を高速に処理できます。  
しかし、曖昧な要求を正しい要求に変換してくれる保証はありません。

むしろ、曖昧な要求は、曖昧なまま高速に実装されます。  
その結果、コードは動くが、意図からズレてしまうことがよくあります。

GitHubのSpec Kit紹介記事にも、AIコーディングで起きる問題が端的に書かれています。

> “looks right, but doesn’t quite work”  
> 出典: [GitHub Blog, “Spec-driven development with AI”](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)

私の経験でも、これはよくあります。  
AIが生成したコードは、構文上は正しく、テストも一部通り、見た目もそれっぽい。  
しかし、ドメイン上の制約、既存設計との整合性、将来の変更容易性、境界の切り方が間違っている。

この状態で「AIすごい」と言ってマージすると、あとで負債になります。

だから、私はまず **Goal Doc** を作ります。

## ハーネスエンジニアリングとは何か

OpenAI は “Harness engineering: leveraging Codex in an agent-first world” で、AIエージェント時代のエンジニアの役割を次のように表現しています。

> “Humans steer. Agents execute.”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

同じ箇所で、エンジニアの仕事が「コードを書くこと」から、環境・意図・フィードバックループの設計に移ると説明しています。

> “design environments, specify intent, and build feedback loops”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

AIにコードを書かせるだけなら、誰でもできます。  
問題は、AIが安全に作業できる環境をどう作るかです。

私の理解では、ハーネスエンジニアリングは次のようなものです。

```
AIが安全かつ再現性高く作業するための足場を作ること
```

具体的には、次のようなものです。

* AGENTS.md
* docs/
* design docs
* ADR
* test
* lint
* CI
* verification command
* observability
* benchmark
* local dev environment
* Skill
* tool permission
* stop condition
* review gate

OpenAIの記事では、巨大なAGENTS.mdを百科事典のように使うのではなく、短いAGENTS.mdを「地図」として使い、深い知識は構造化されたdocsに置く話も出ています。

> “give Codex a map, not a 1,000-page instruction manual.”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

また、構造化された `docs/` を system of record として扱う例も紹介されています。

> “The repository’s knowledge base lives in a structured `docs/` directory treated as the system of record.”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

この考え方は、私のGoal Doc運用とも接続できます。

私の中では、

```
AGENTS.md = プロジェクト全体の作業憲法
docs/ = 参照すべき知識ベース
Goal Doc = タスク単位の完了契約
ADR = 重要な設計判断の記録
Skill = 再利用可能な作業手順
```

という位置づけです。

## ループエンジニアリングとは何か

Loop Engineering は、ハーネスの上で、AIが**Goal**を満たすまで反復する仕組みだと理解しています。

Addy Osmani の言葉を借りるなら、Loop Engineering は「人間がプロンプトする係であること」をシステムに置き換えるものです。

> “Loop engineering is replacing yourself as the person who prompts the agent.”  
> 出典: [Addy Osmani, “Loop Engineering”](https://addyosmani.com/blog/loop-engineering/)

ここで重要なのは、ループは単なる無限実行ではないことです。

AIが勝手に走り続けることが重要なのではありません。  
重要なのは、**目的、検証条件、停止条件を持ったループにする**ことです。

OpenAIのCodex Goalsのドキュメントでは、Goalは次のように説明されています。

> “Goals are persistent objectives in Codex that keep a thread working toward a defined outcome across turns.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

さらに、Goalは単なる大きなプロンプトではなく、完了条件・検証・制約を持つものだと説明されています。

> “A Goal gives Codex a completion condition: what should be true, how success should be checked, and what constraints must stay intact.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

この定義は、私のGoal Docの考え方とほぼ一致します。

つまりGoalは、

ではなく、

を定義するものです。

OpenAIの同ドキュメントでは、Goalと通常のPromptの違いも明確です。

> “A normal prompt says: do this next thing. A Goal says: keep working until this outcome is true.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

通常のプロンプトは「次にこれをやれ」です。  
Goalは「この状態が成立するまで進めろ」です。

私のやっている `/goal` 運用はこちらにあたります。

## 私のやり方：まずAIにGoal Docを作らせる

私は、AIにいきなり実装させないようにしています。

まず、AIにGoal Docを作らせます。

ただし、ここで重要なのは、Goal Docの採用判断は人間が持つことです。  
AIが作ったGoal Docは初稿でしかありません。 それをそのまま実行契約にしてはいけません。

OpenAIのCodex Goalsのドキュメントにも、Goalは user-controlled な completion contract だと書かれています。

> “A Goal is not background autonomy without boundaries. It is a scoped, user-controlled completion contract.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

私の理解では、Goal DocはAIに対する「作業契約」です。

```
この状態を目指す
この範囲でやる
これを壊してはいけない
これで成功確認する
ここで止まる
ここは人間に確認する
```

を明文化します。

OpenAIのドキュメントでは、強いGoalに必要な要素として、outcome、verification surface、constraints などが挙げられています。

> “The strongest Goals usually define six things: Outcome, Verification surface, Constraints, Boundaries, Iteration policy...”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

また、タスクは明確だがGoalが明確でない場合、CodexにGoal自体を書かせる二段階ワークフローも説明されています。

> “When the task is clear but the Goal is not, Codex can help write the Goal itself.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

これは、私の運用にかなり近いと思いました。

つまり、次のように進めます。

```
1. まずAIにGoal Docの初稿を作らせる
2. 人間がGoal Docをレビューする
3. 成功条件・検証条件・制約・停止条件を締める
4. `/goal` でGoalを満たすまで走らせる
```

ここでAIに渡しているのは、単なる作業指示ではありません。作業の契約です。

## Goal Docに含めているもの

私のGoal Docには、最低限以下を入れます。

```
# Goal

## Desired end state
この作業が完了したときに成立しているべき状態。

## Non-goals
今回やらないこと。

## Research phase
実装前に調べるべきこと。

## Verification
成功を確認する方法。
例: unit test, integration test, E2E, benchmark, screenshot, log, manual checklist

## Constraints
壊してはいけないもの。
例: public API, DB schema compatibility, existing UX, security boundary

## ADR trigger
以下に該当する場合はADRを作る。
- アーキテクチャ境界を変更する
- 永続化モデルを変更する
- 外部依存を追加する
- 認証・認可・課金・セキュリティに影響する

## Skill trigger
同じ作業が再発しそうな場合、またはAIが同じミスを繰り返した場合はSkill化する。

## Stop conditions
以下の場合は停止する。
- 要件が矛盾している
- 既存仕様と衝突する
- 検証不能である
- 人間判断が必要である

## Human review gates
人間が見るべき箇所。
例: goal doc, ADR, migration plan, security-sensitive diff, public API change
```

この構成にしている理由を分解します。

### Desired end state

AIには「何をやるか」よりも「何が成立すれば完了か」を渡すべきです。

OpenAIもGoalについて、measurable outcome、verification surface、constraints を持たせるべきだと説明しています。

> “Write a Goal with a measurable outcome, a verification surface, and constraints.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

Desired end state は、AIにとっての完了条件です。

### Non-goals

AIは優秀なので、勝手に周辺の問題まで直そうとします。  
しかし、AIの「ついでに直しました」は危険です。

特に既存仕様、外部API、DBスキーマ、認証認可、UI挙動に関わる場合、スコープ外の変更は事故になります。

OpenAIはGoalを scoped な completion contract と説明しています。  
私はこの “scoped” を、Non-goals と boundaries で実装しています。

> “a scoped, user-controlled completion contract”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

### Research phase

AIに調べさせずに実装させると、モデルの事前知識や推測で進みます。それは速いですがかなり危険です。

特に、ライブラリの最新仕様、既存コードの設計意図、社内ルール、既存のADR、外部API仕様などは、事前に確認させるべきです。

Microsoftは、AIは未解決の曖昧さを修正できないと述べています。

> “AI can accelerate those steps, but it cannot correct ambiguity that was never resolved.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

Research phase は、AIに推測で進ませないための工程です。

### Verification

Verification は、Goal Doc の中でも最重要だと考えています。

検証条件が弱いGoalは、AIに「完了したように見えるもの」を作らせます。

OpenAIのCodex Goalsでは、Goalが完了したかは concrete evidence に照らして判断すべきだと説明されています。

> “A Goal should not be marked complete because the model believes it is probably done.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

さらに続けて、ファイル、テスト、ログ、ベンチマーク、生成物などの具体的な証拠に照らして確認すべきだと説明されています。

> “It should be complete only after the objective is checked against the relevant files, tests, logs, benchmark output, generated artifacts, or other concrete evidence.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

私の運用では、Verificationにはできるだけ具体的な確認方法を書きます。

```
## Verification

- pnpm test が通る
- pnpm lint が通る
- 対象ユースケースのE2Eが通る
- 既存の public API response が変わっていない
- 変更対象のスクリーンショットを取得して差分確認する
```

「確認する」では弱いため、「何を実行し、何が通れば成功か」まで書きます。

### ADR trigger

AIに実装させると、設計判断がコード差分に埋もれます。

これは危険です。

設計判断がコードだけに残ると、あとから「なぜこうなっているのか」が分からなくなります。  
AIが書いたコードの場合、この問題はさらに強くなります。人間が頭の中で意思決定していないため、暗黙知としても残らないからです。

MicrosoftのSpec Kit深掘り記事では、Spec-driven development を「思考のバージョン管理」として表現しています。

> “version control for your thinking”  
> 出典: [Microsoft, “Diving Into Spec-Driven Development With GitHub Spec Kit”](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)

また、設計判断や制約を explicit, reviewable, evolvable にすることの重要性も述べられています。

> “explicit, reviewable, and evolvable”  
> 出典: [Microsoft, “Diving Into Spec-Driven Development With GitHub Spec Kit”](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)

私は、アーキテクチャ境界、永続化モデル、外部依存、認証認可、課金、セキュリティに影響する判断が出た場合、ADRを作らせます。

ADRは万能ではありませんが、少なくとも「なぜこの判断をしたのか」をコード差分の外に出せます。

### Skill trigger

同じ手順や同じ失敗が繰り返される場合は、Skill化を検討します。

Anthropic は Agent Skills を次のように説明しています。

> “organized folders of instructions, scripts, and resources that agents can discover and load dynamically”  
> 出典: [Anthropic, “Equipping agents for the real world with Agent Skills”](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)

Claude Platform のドキュメントでも、Skillは instructions、metadata、optional resources をパッケージ化するものだと説明されています。

> “Each Skill packages instructions, metadata, and optional resources”  
> 出典: [Claude Platform Docs, “Agent Skills”](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)

私の理解では、Skillは「再利用可能な手続き知識」です。

たとえば、毎回同じように以下を指示しているなら、それはSkill候補となります。

```
- このリポジトリでのテストの実行方法
- フロントエンド変更時のスクリーンショット確認方法
- ADRを書く判断基準
- DB migration 作成時のチェックリスト
- OpenAPI / ogen / gRPC 生成時の手順
- React Router v7 での loader/action 追加手順
```

毎回プロンプトで貼るのは無駄です。個人的な感覚では、3 度同じように指示をした際にはSkill化を検討します。  
固定できるのであれば、プロジェクト固有の手順としてSkill化した方がよいです。

### Stop conditions

Stop conditions は、AIを安全に止めるために必要です。

AIは、曖昧な状態でも何かしら進めようとします。しかし、明確に進めるべきでない状態があります。

たとえば、次のような場合です。

```
- 要件が矛盾している
- 既存仕様と衝突している
- public APIを壊す必要がある
- DB migrationの影響範囲が不明
- セキュリティ境界に影響する
- テストでは検証できない
```

OpenAIのCodex Goalsでも、budget limit や blocker がある場合には停止し、進捗・ブロッカー・次の入力を報告すべきだと説明されています。

> “When the budget is reached, Codex should stop substantive work, summarize progress and blockers, and identify the next useful step.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

Stop condition がないGoalは危険です。  
AIが前進しているように見えて、実際には間違った方向に掘り進めることがあります。

### Human review gates

私は、AIに下位の反復を任せます。  
しかし、上位判断は人間が持つべきだと考えています。

OpenAIの “Humans steer. Agents execute.” という表現は、この分担を表しています。

> “Humans steer. Agents execute.”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

人間が見るべきものは、たとえば次でしょう。

```
- Goal Doc
- ADR
- migration plan
- public API change
- security-sensitive diff
- billing / auth / permission change
- data migration
- irreversible decision
```

AIに任せてよいのは、下位の反復です。  
AIに完全に渡してはいけないのは、問題設定、制約、優先順位、リスク許容度です。

---

## `/goal` でGoalを満たすまで自走させる

Goal Docを作ったら、それをもとに `/goal` で走らせます。

ここでやりたいのは、タスク一覧を順番に消化させることではありません。  
**Goalを満たすまで、AIに実装・検証・修正・調査を反復させること**です。

OpenAIのCodex Goalsでは、Goalがactiveな状態では、Codexはテストやベンチマークなどの証拠を見て、目的が満たされたかを判断し、満たされていなければ続行できると説明されています。

> “After a turn finishes, it can inspect the current evidence and decide whether the objective is satisfied.”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

また、実用上のメンタルモデルとして、PromptとGoalの違いが以下のように整理されています。

> “Prompt: ask -> work -> result -> wait”  
> “Goal: work -> check -> continue or complete”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

この違いは大きいと思います。

Promptは1ターンの依頼ですが、Goalは継続的な作業状態です。

```
Goalを読む
↓
必要なら調査する
↓
実装する
↓
検証する
↓
失敗したら原因を調べる
↓
修正する
↓
再検証する
↓
成功条件を満たしたら完了報告する
↓
満たせない場合はブロッカーと必要な人間判断を報告する
```

これが、私の考える **Goal Doc 駆動の Loop Engineering** です。

---

## 重いタスクでは「仕様作成自体」を先にGoalにする

重いタスクを、いきなり実装Goalにするのは危険です。

たとえば、次のような作業です。

* 認証認可の境界変更
* DB schema変更
* 外部API連携
* 決済・課金
* gRPC / OpenAPI の境界設計
* DDDの集約境界変更
* 大きめのリファクタリング
* フロントエンドのルーティング構造変更
* 既存仕様の整理が必要な機能追加

こういう作業でいきなりAIに実装させると、AIはもっとも実装しやすい形に問題を変形することがあります。

そのため、私は重いタスクではGoalを分割します。

```
Research Goal
↓
Spec Goal
↓
ADR Goal
↓
Implementation Goal
↓
Verification Goal
```

OpenAIのExecPlansは、この考え方です。

OpenAI Cookbookでは、ExecPlanを次のように説明しています。

> “an execution plan ("ExecPlan"), a design document that a coding agent can follow to deliver a working feature or system change”  
> 出典: [OpenAI Cookbook, “Using PLANS.md for multi-hour problem solving”](https://developers.openai.com/cookbook/articles/codex_exec_plans)

また、ExecPlanを書くときは、調査しながら正確な仕様に肉付けするように説明されています。

> “When creating a spec, start from the skeleton and flesh it out as you do your research.”  
> 出典: [OpenAI Cookbook, “Using PLANS.md for multi-hour problem solving”](https://developers.openai.com/cookbook/articles/codex_exec_plans)

さらに、実装時には毎回ユーザーに次の手順を聞かず、次のマイルストーンに進むように書かれています。

> “do not prompt the user for "next steps"; simply proceed to the next milestone.”  
> 出典: [OpenAI Cookbook, “Using PLANS.md for multi-hour problem solving”](https://developers.openai.com/cookbook/articles/codex_exec_plans)

私の場合、重いタスクでは以下のように分けます。

### Research Goal

まず調査だけをGoalにします。

```
/goal この機能追加に必要な既存コード、関連ADR、既存仕様、外部API仕様を調査し、実装前に確認すべき論点を整理する。コード変更はしない。
```

このGoalでは、実装させません。  
調査結果と論点整理だけを成果物にします。

### Spec Goal

次に仕様作成をGoalにします。

```
/goal Research結果をもとに、実装可能なSpec Docを作成する。成功条件、非目標、制約、検証方法、未解決論点を明記する。コード変更はしない。
```

ここでも、まだ実装しません。

### ADR Goal

アーキテクチャ境界や永続化モデルに影響する場合は、ADRをGoalにします。

```
/goal Spec Docで発生した設計判断についてADR案を作成する。選択肢、決定、理由、トレードオフ、将来の撤回条件を明記する。コード変更はしない。
```

### Implementation Goal

ここまで来てから実装Goalにします。

```
/goal 承認済みSpec DocとADRに従って実装する。public API、DB schema、既存UX、認証認可境界を壊さない。各milestoneごとにテストを実行する。
```

### Verification Goal

最後に検証だけをGoalにします。

```
/goal 実装結果がSpec Docの成功条件を満たしているか検証する。テスト、lint、E2E、ログ、スクリーンショット、差分確認を実施し、確認済み・未確認・ブロッカーを分けて報告する。
```

この分割により、AIの暴走範囲を狭められます。

1つの巨大なGoalで全部やらせるより、各フェーズの成果物をレビューできます。これにより、人間が介入すべきポイントも明確になります。

## ADR・Skill・ResearchをLoopに含める理由

私のGoal Doc駆動では、Loopを「コード生成の反復」だけにはしません。

Research、ADR、SkillもLoopの対象にします。

### Researchを入れないと、AIは推測で進む

AIは、知らないことでもそれっぽく進められます。

これは強みでもありますが、ソフトウェア設計では危険です。

特に、既存コードの境界、ドメインルール、チームの設計方針、外部APIの最新仕様、依存ライブラリの破壊的変更などは、事前調査が必要です。

Microsoftは、shared artifact がないと、各handoffがinterpretation stepになると述べています。

> “Without a shared artifact that preserves intent, every handoff becomes an interpretation step.”  
> 出典: [Microsoft, “Spec-Driven Development: A Spec-First Approach to AI-Native Engineering”](https://developer.microsoft.com/blog/spec-driven-development-ai-native-engineering)

私の運用では、Research結果も共有成果物にします。  
AIの推測ではなく、参照したコード、仕様、ドキュメント、未解決論点を残します。

### ADRを入れないと、設計判断がコード差分に埋もれる

AIが設計判断をした場合、それをコードだけに残すのは危険です。

コードを読めば分かる、というのはある意味においては間違っていると思います。  
コードから「何をしたか」は分かりますが、しかし「なぜそうしたか」は分からないからです。

Spec-driven developmentの文脈では、仕様や設計判断を明示的でレビュー可能なものにする重要性が語られています。

> “explicit, reviewable, and evolvable”  
> 出典: [Microsoft, “Diving Into Spec-Driven Development With GitHub Spec Kit”](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)

私は、以下の場合はADRを作るようにしています。

```
- アーキテクチャ境界を変える
- 永続化モデルを変える
- 依存ライブラリを追加する
- 認証認可に影響する
- 課金・決済に影響する
- セキュリティ境界に影響する
- 将来の撤回が難しい判断をする
```

これは、AIの判断を人間がレビュー可能にするためです。

### Skillを入れないと、同じ失敗を繰り返す

AIに何度も同じことを教えているなら、それはプロンプトではなく資産化すべきです。

AnthropicはSkillsについて、手続き知識を共有・再利用するものとして説明しています。

> “capturing and sharing their procedural knowledge”  
> 出典: [Anthropic, “Equipping agents for the real world with Agent Skills”](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)

Claude Platform Docsでも、Skillsはconversation-levelの一回限りの指示ではなく、必要に応じて読み込まれる再利用可能なfilesystem-based resourcesだと説明されています。

> “Skills are reusable, filesystem-based resources”  
> 出典: [Claude Platform Docs, “Agent Skills”](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)

私の運用では、以下の場合にSkill化を検討します。

```
- 同じ手順を3回以上説明している
- 同じ種類のミスをAIが繰り返している
- 特定リポジトリ固有の実行手順がある
- 特定技術スタック固有の制約がある
- チームの設計方針を毎回説明している
```

ただし、Skillは万能ではありません。

誤ったSkillを作ると、誤った慣習が固定化されます。  
したがって、Skillは作って終わりではなく、常にレビュー・検証・更新が必要です。

---

## この方法で何を解決するのか

**Goal Doc駆動のLoop Engineering**で解決したいのは、主に以下です。

### 1. 人間の逐次ディレクションを減らす

人間が毎回「次はこれ」と指示し続けるのはスケールしません。

Loop Engineeringの考え方では、人間がプロンプトする係ではなく、その役割を果たすシステムを設計します。

> “You design the system that does it instead.”  
> 出典: [Addy Osmani, “Loop Engineering”](https://addyosmani.com/blog/loop-engineering/)

私の運用では、Goal Docがその中心になります。

### 2. 完了条件を明確にする

AIにとって難しいのは、コードを書くことではなく、いつ完了したと判断するかです。

OpenAIのCodex Goal use caseにも、重要なのは contract であり、Codexは開始前に “done” の意味を知っているべきだと説明されています。

> “Codex should know what ‘done’ means before it starts.”  
> 出典: [OpenAI, “Follow a goal”](https://developers.openai.com/codex/use-cases/follow-goals)

Goal Docでは、Doneの定義を明確にします。

### 3. 調査・設計・実装・検証を分離できる

重い作業では、全部を一つの「実装して」に押し込むべきではありません。

OpenAIのExecPlansでも、複雑なタスクは research、design、implement を含むものとして扱われています。

> “research, design, and implement”  
> 出典: [OpenAI Cookbook, “Using PLANS.md for multi-hour problem solving”](https://developers.openai.com/cookbook/articles/codex_exec_plans)

私の運用では、これをGoal単位で分けます。

### 4. AIの作業をレビュー可能な成果物に変換できる

AIの出力をコードだけにすると、レビューが難しくなります。

Goal Doc、Research Doc、Spec Doc、ADR、Verification Report、Skillとして成果物を分けることで、人間がレビューしやすくなります。

GitHubのSpec Kit記事では、specをliving, executable artifactsとして扱うと述べられています。

> “living, executable artifacts”  
> 出典: [GitHub Blog, “Spec-driven development with AI”](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)

私の運用では、Goal Docも同じように「生きた作業契約」として扱います。

### 5. チームで再利用できる知識が残る

Goal Doc、ADR、Skillを残すと、AIとのやり取りが一回限りで消費されません。これは、チームにとって重要です。

属人的なプロンプト術ではなく、再利用可能な知識になります。

## 逆に危険なところ

ここまで書くと、Goal Doc駆動のLoop Engineeringはかなりよさそうに見えます。

しかし、危険もあります。

### 1. AIが作ったGoal Docが人間の意図からズレる

最大のリスクはこれです。

AIにGoal Docを作らせると、AIは実装しやすい形に問題を再定義することがあります。

人間が本当に解くべき問題ではなく、AIが解きやすい問題に変換される。  
これは危険です。

したがって、Goal Docの初稿はAIに作らせてもよいですが、採否は人間が持つべきです。

OpenAIのCodex Goalsが user-controlled completion contract と表現している点は重要です。

> “user-controlled completion contract”  
> 出典: [OpenAI, “Using Goals in Codex”](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)

自走させることと、制御を手放すことは違います。

### 2. 検証条件が弱いと、完了したように見えるだけになる

AIは「完了した」と言えます。  
しかし、それが本当に完了しているかは別です。

GitHubのSpec Kit記事の “looks right, but doesn’t quite work” という表現は、ここに鋭く刺さります。

> “looks right, but doesn’t quite work”  
> 出典: [GitHub Blog, “Spec-driven development with AI”](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)

Goal DocにVerificationがない場合、AIはそれっぽく完了報告します。

そのため、Goal Docには必ず検証条件を書きます。

### 3. ADRが後付けの言い訳になる

ADRをAIに作らせると、実装後にそれっぽい理由を後付けするだけになる危険があります。

これは避けるべきです。

ADRは、意思決定ログです。実装後の言い訳ではありません。

そのため、重要な設計判断が必要になった時点でADRを作らせ、人間がレビューすべきです。

### 4. Skillが誤った慣習を固定化する

Skill化は強力ですが、その反面、危険でもあります。

間違ったSkillを作ると、AIは毎回その間違った手順を再利用します。

したがって、Skillは以下を満たすべきだと考えます。

```
- 目的が明確である
- 適用条件が明確である
- 非適用条件が明確である
- 実行手順が検証可能である
- 定期的に見直される
```

Skillは「便利なメモ」ではなく、AIが実行時に参照する手続き知識です。雑に作ると一気に危険になります。

### 5. 人間が握るべき判断までAIに渡すと破綻する

AIに任せるべきなのは、下位の反復です。任せてよいものは、たとえば次でしょう。

```
- 既存コードの探索
- テストの追加
- 小さな実装
- lint / test / build の反復
- エラー原因の調査
- 仕様草案の作成
- ADR草案の作成
- Skill草案の作成
```

逆に、人間が握るべきものは次です。

```
- 何を解くべきか
- 何を解かないべきか
- どのリスクを許容するか
- どの設計判断を採用するか
- セキュリティや課金に関する判断
- public APIやデータモデルの破壊的変更
```

OpenAIの表現を借りるなら、Humanはsteerし、Agentはexecuteします。

> “Humans steer. Agents execute.”  
> 出典: [OpenAI, “Harness engineering: leveraging Codex in an agent-first world”](https://openai.com/index/harness-engineering/)

ここを間違えると、AI活用ではなく、責任の所在が曖昧な自動化になります。

---

## 私の現在の結論

もっと正確に言うと、

```
Harnessを整え、その上でLoopを回し、その中心にGoal Docを置く。
```

です。

Prompt Engineering は不要ではありません。ただし、それだけでは足りません。

Harness Engineering も重要です。ただし、ハーネスは足場であり、それだけではAIは目的達成まで反復しません。

Loop Engineering も重要です。ただし、ループはGoalの質に強く依存します。

だから、私はGoal Docを中心に置こうと思います。

```
AGENTS.md / docs / tests / tools / skills / ADR
          ↓
        Harness
          ↓
        Goal Doc
          ↓
        /goal Loop
          ↓
   implementation / verification / update
```

この構造にすると、AIに渡すものが「お願い」ではなく「作業契約」になります。

---

## まとめ：PromptではなくGoalを設計する

AIに「何をするか」を逐次指示するのではなく、「何が成立すれば完了か」を渡す。

これが、私の考えている **Goal Doc 駆動の Loop Engineering** です。

ソフトウェアエンジニアの間では、以下のような概念がそれぞれ議論されています。

* Harness Engineering
* Loop Engineering
* Goals
* ExecPlans
* Spec-Driven Development
* Agent Skills

私のやり方は、それらをGoal Doc中心に束ねた実践に近いと思います。

これから重要になるのは、AIにコードを書かせる力だけではありません。

AIが安全に自走できるように、

* 何を達成するか
* 何を壊してはいけないか
* 何で成功確認するか
* どこで止まるか
* どの判断を人間が持つか
* どの知識をSkill化するか

を設計する力でしょう。

つまり、プロンプトを書く力ではなく、Goal、Harness、Loopを設計する力です。
