## 今回の発表をひとことで言うと

Mastra のブラウザ操作に、**実行を録画してあとから再生できる仕組み**が入った、というのが今回の発表です。これまでは Studio 上でライブの様子を見るのが中心でしたが、そこに「終わったあとに見返す」「チームで共有する」「失敗の瞬間を特定する」という運用が加わりました。

録画を有効にすると、エージェントには `browser_record` と `browser_record_caption` の 2 つが増えます。前者は録画の開始・状態確認・停止を担い、後者は録画中に短い字幕を差し込むためのものです。出力形式は Motion-JPEG の `.avi` で、`ffmpeg` のようなネイティブ依存に頼らずに扱える点も、導入のしやすさにつながっています。

---

## 背景: なぜこの発表が必要だったのか

Web を相手にするエージェントは、API を叩くだけの処理と違って、ページ遷移やフォーム入力、待機タイミングのズレといった、目に見えるけれど再現しにくい失敗が起こりやすいです。しかもその失敗は、テキストログだけでは「どの瞬間に崩れたのか」が読み取りにくい。実行が終わったあとに、同じ画面、同じ順序、同じ視点で見返せないことが、調査を難しくしていました。

Mastra のブラウザ機能は、これまで Studio 上のライブ確認、つまり実行中のスクリーンキャストが中心でした。これはリアルタイムで追うには便利ですが、実行が終わると証跡が残りません。今回の Browser Recording は、この欠点を埋めるために、**映像と字幕を持つ時系列ログ**を標準で扱えるようにしたものです。短いデモの保存にも、長めの自動化フローのデバッグにも効きますし、プロンプトを変えた前後の比較もしやすくなります。

---

## 何が追加されたのか（機能の中身）

この機能はデフォルトでは無効で、必要なエージェントだけ `recording` を設定して使います。`StagehandBrowser` でも `AgentBrowser` でも同じ考え方で有効化でき、`outputDir` だけは必須です。実運用では、失敗しやすいエージェントや、チームでレビューしたいフローだけを録画対象にするのが自然です。

```
import { join } from "node:path";
import { Agent } from "@mastra/core/agent";
import { StagehandBrowser } from "@mastra/stagehand";

export const browserAgent = new Agent({
  id: "browser-agent",
  name: "Browser Agent",
  model: "openai/gpt-5.5",
  // 録画時の字幕がそのまま人間向けの説明になるので、日本語で書いておく
  instructions: `
あなたはWeb操作アシスタントです。
録画が要求されたら、作業開始前に browser_record を start してください。
主要操作のたびに browser_record_caption で短い説明を追加してください。
完了直後に browser_record を stop して保存先パスを返してください。
`.trim(),
  browser: new StagehandBrowser({
    headless: false,
    recording: {
      outputDir: join(process.cwd(), "browser-recordings"),
      maxDurationMs: 60_000,
      maxWidth: 1280,
      maxHeight: 720
    }
  })
});
```

録画の操作は `browser_record` で行います。`start` で録画を始め、`status` で現在の状態を確かめ、`stop` で `.avi` を保存します。`outputPath` を指定する場合は、設定した `recording.outputDir` の下にある絶対パスでなければなりません。安全のための制約ですが、実際にはここが重要で、保存先を勝手に外へ逃がせないようにする仕組みでもあります。

字幕は `browser_record_caption` で入れます。ここで残るのは、いわゆる open captions の焼き込みです。単なる画面録画ではなく、「今なにをしているのか」「なぜそこで止まったのか」を短い言葉で残せるので、後から見た人が流れを追いやすくなります。

出力の既定値も押さえておくと、形式は Motion-JPEG AVI、既定の最大録画時間は 30 秒、ただしハード上限は 120 秒、最大フレームサイズは 1024 x 720、字幕は 1 件 80 文字まで、同時録画は 1 process につき 1 本です。docs では beta と alpha の表現が混在していますが、要するにまだ安定化前で、導入するならバージョン固定と段階的な展開が安全、という理解でよいです。

## 以前は何が課題で、今回の機能で何が嬉しいのか

以前の課題は、ブラウザ操作が「見ている間だけ」成立していたことでした。ライブで追っていれば動きはわかるものの、後から同じものを再生できないので、失敗したときにどこで崩れたのかをチームで共有しづらかった。しかも、見ていた人の記憶に依存しやすいので、レビューが属人的になりやすい。プロンプト改善の A/B 比較も、結果だけでなく途中経過を比べたいのに、その材料が残りませんでした。

今回の機能で嬉しいのは、その逆ができることです。実行は動画として残り、字幕でエージェントの意図も添えられるので、原因分析が速くなります。保存した録画はそのまま不具合調査の一次資料になりますし、チーム内で同じものを見ながら話せるので、再現手順や修正方針のすり合わせもやりやすくなります。改善前後を同じ条件で比べやすい、というのも地味ですが大きい変化です。

ブラウザ自動化を運用するチームほど、この差は効いてきます。単発のデモではなく、失敗録画を見返してプロンプトや待機戦略を少しずつ詰めていく、という改善サイクルに乗せやすくなるからです。

## 既存手法・関連機能・競合文脈での位置づけ

Mastra にはすでに Studio 上の screencast がありました。こちらは実行中の観察に強く、いま何が起きているかをライブで見るには向いています。一方で Browser Recording は、実行後の検証や共有に向いた機能です。両者は置き換え関係ではなく、観察フェーズと検証フェーズを分担する関係だと考えると理解しやすいです。

Playwright 単体でも動画録画はできますが、Mastra の今回の価値は、録画がエージェントツールとして統合されているところにあります。エージェント自身が `start`、`caption`、`stop` をワークフローの一部として扱えるので、単なるブラウザ制御の外付け機能では終わりません。StagehandBrowser と AgentBrowser のどちらでも同じ発想で使えるため、プロバイダーが変わっても運用の考え方を揃えやすいのも実務では重要です。

---

## 実務での導入パターン（すぐ効く運用）

導入のコツは、長時間を常時録画することではなく、目的を絞って短く撮ることです。失敗再現が多いエージェントだけを録画対象にして、主要ステップにだけ短い字幕を入れ、停止後は保存パスを返す流れを標準化しておくと、あとで見返しやすい素材が自然に残ります。そこから週次で失敗録画を見返して、プロンプトや待機戦略を少しずつ改善していくと、録画が単なるログではなく改善サイクルの一部になります。

---

## まとめ

Browser Recording は、Mastra のブラウザエージェントに「再生可能な実行ログ」を与えるアップデートです。ライブで見るだけだった実行を、あとから追える・共有できる・比較できる対象に変えたことで、ブラウザ自動化の運用が一段実践的になりました。

今回の発表の意味は、単に録画ができるようになったことではありません。実行を動画として残せること、字幕で意図まで記録できること、Stagehand / AgentBrowser の運用に自然に乗ること、そしてエージェント改善を再現可能な検証サイクルに持ち込めること。この 4 つがそろって初めて、「ブラウザを動かせる」段階から「ブラウザ実行を品質管理できる」段階へ進んだと言えます。
