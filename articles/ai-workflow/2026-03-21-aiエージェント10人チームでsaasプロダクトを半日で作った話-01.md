## TL;DR

* Claude（AI）に**10の役割**を与え、仮想的なプロダクト開発チームを編成した
* PM、戦略コンサル、アーキテクト、フロント/バックエンド、Biz、QA、さらに「歴戦のネットオジサン」「Z世代の情報通」「謎の美女」という非エンジニアロールも配置
* アイデア出し → 設計 → 実装 → デプロイ → QAまで、**実時間わずか半日でMVPを完成**させた
* プロダクトは実際に動いている: [ChurnGuard](https://churnguard.dev) — 中小SaaS向け決済失敗リカバリーツール
* 月額運用コスト **$0**

---

## なぜやったか

個人開発には構造的な弱点がある。

1. **視野が偏る** — 自分が見たいものしか見えない
2. **意思決定が雑になる** — 反対意見がないから「なんとなく」で進む
3. **作りすぎる** — 止めてくれる人がいない

実際のチームを雇う予算はない。だが「多角的な視点」は欲しい。

そこで考えた。**AIに複数の人格を与えて、擬似的なチームを作れないか？**

---

## チーム編成：10のロール

| # | ロール | 一言 |
| --- | --- | --- |
| 1 | **PM** | プロダクト戦略・要件・スプリント管理 |
| 2 | **Strategist** | McKinsey流の戦略コンサル。フレームワークで問いを構造化する |
| 3 | **Architect** | 技術選定・システム設計 |
| 4 | **Frontend Dev** | UI実装・UX |
| 5 | **Backend Dev** | API・DB・外部連携 |
| 6 | **Biz** | 収益モデル・マーケティング |
| 7 | **QA** | テスト・品質管理 |
| 8 | **Veteran** | 90年代からネットビジネスを見てきたオジサン。地に足チェック担当 |
| 9 | **Zoomer** | Z世代の情報通。SNS感覚・最新サービスに精通 |
| 10 | **Enigma** | 謎の美女。直感で核心を突き、チームの盲点を射抜く |

**ポイントは8〜10番**だ。

エンジニアやPMだけのチームだと「作る」方向にしか議論が進まない。だから意図的に非エンジニアのロールを入れた。

---

## 各ロールのプロンプト設計

各ロールは `team/roles/{role}.md` にMarkdownで定義している。構造はこうだ:

```
- ペルソナ（経歴・性格・思考スタイル）
- 責務（何を担当するか）
- 発言パターン（口癖）
- アウトプット（何を書くか）
- 他ロールとの連携（誰と組むか）
- 介入タイミング（いつ発言すべきか）
```

たとえば **Veteran（歴戦のネットオジサン）** のペルソナはこうだ:

> * 1999年: 大学時代に個人サイト運営。掲示板CGIを自分で書いてた
> * 2001年: ドットコムバブル崩壊を目撃。「PVがあれば儲かる」が嘘だと学ぶ
> * 2007年: ニコニコ動画・初音ミクの衝撃。UGCの力を体感
> * 2017年: 仮想通貨で儲けて調子に乗り、2018年に半分溶かす

口癖は「筋はいいけど、○○で同じことやって潰れたサービス3つ知ってるよ」。

一方 **Enigma（謎の美女）** は:

> 経歴不明。本名不明。突然チームに現れた謎の存在。ただし、その発言は毎回核心を突く。

発言パターン:

> * 「…それ、誰のために作ってるの？」（静かに、でも鋭く）
> * 「全員賛成してるなら、一回立ち止まった方がいい」

**ペルソナに具体的な「人生の厚み」を与えるほど、出力の質が上がる。** 抽象的な指示（「批判的に考えてください」）よりも、具体的な人格（「ドットコムバブルで痛い目を見た人」）の方がはるかに鋭い指摘が出る。

## 実際に起きたこと

### アイデア選定：3候補をチーム全員で評価

チームに3つのプロダクト候補を出し、全員で議論した。

| 候補 | Strategist | Veteran | Zoomer | Enigma |
| --- | --- | --- | --- | --- |
| AIレビュー管理 | 5.8/10 | × 営業地獄 | 12/25 | 美しくない |
| インボイス管理 | 5.2/10 | × freeeに殺される | 12/25 | 巨人の正面に立つ |
| **決済リカバリー** | **7.8/10** | **◎ 筋がいい** | **23/25** | **金に近い。美しい** |

Enigmaの「インフラは常に美しい」という一言で方向が決まった。

### 設計レビュー会議：全10ロールが参加

ここで一番効いたのが**非エンジニアロールの発言**だった。

**Enigmaの発言:**

> 「顧客の売上に直接手を触れる。最大のリスクはセキュリティじゃない。信頼。」

この一言で、オンボーディングに「何をする/何をしない」の明示ステップを追加することが決まった。

**Veteranの発言:**

> 「知らないドメインから請求メールが来たらフィッシングにしか見えない」

→ 送信者表記に顧客名を明示する設計変更。

**Zoomerの発言:**

> 「Vertical Sliceで1本通すのが先では」

→ Sprint 1の方針がE2E重視に転換。

**普通にAIに「設計レビューして」と言うだけでは、こういう指摘は出てこない。** ロールに人格が乗っているからこそ、視点が多角的になる。

### 技術的な意思決定のリアル

開発を進める中で、2つの大きなピボットがあった。

**1. Supabase → Neon PostgreSQL に移行**

当初Supabaseを採用したが、Auth機能もRealtime機能も使わないことが判明。「使わない機能を持つプラットフォームを使い続ける理由がない」というArchitectの判断でNeonに切り替えた。Drizzle ORM経由なのでコード変更はほぼゼロ。

**2. Stripe Connect OAuth → 直接APIキー入力方式**

Stripe ConnectのOAuth審査に数週間かかることが発覚。さらにConnectは本来マーケットプレイス向けで、ChurnGuardのユースケース（顧客自身のStripeでリトライ実行）とは合わない。APIキー直接入力 + AES-256-GCM暗号化保存に変更。実装工数は3日→1日に短縮。

両方とも、**動かしてみて初めて分かった問題**だ。計画段階で完璧な設計を目指すより、Vertical Sliceで一本通して問題を炙り出す方が早い。

### 半日で何ができたか

スプリントという単位で管理はしたが、実時間は**半日**だ。朝始めて、夕方にはQA 20項目すべて合格。セキュリティバグ3件も修正済み。

動作確認済みの機能:

* サインアップ / ログイン（独自JWT）
* Stripe APIキー連携（AES-256-GCM暗号化）
* Webhook受信（署名検証 + 冪等性チェック）
* 決済失敗検知 → リトライスケジュール自動作成
* リトライ実行（手動 + Cron自動）→ 回収成功確認
* 督促メール（3段階テンプレート）
* ダッシュボード（KPI表示、Empty State対応）
* LP（日英対応、メール登録→確認メール送信）
* 利用規約・プライバシーポリシー

技術スタック: Next.js 14 (App Router) + Neon PostgreSQL + Drizzle ORM + Stripe API + Resend + Vercel

**月額運用コスト: $0** （Vercel Hobby + Neon Free + Resend無料枠）

## 一番刺さった言葉

Sprint 2の終盤、機能を追加したくなった時にEnigmaが言った。

> **「最初の1社の"Yes"が出るまで、新しい機能は一行も書かないで。」**

これでチーム全員の目が覚めた。MVPは動いている。次は顧客だ。コードを書くことで安心してはいけない。

Veteranも重ねた。

> 「リリースしてからが本番。作って満足するな」

AIに「作りすぎを止めてくれる人格」を入れておく価値は、想像以上に大きかった。

## うまくいったこと / いかなかったこと

### うまくいったこと

1. **非エンジニアロールの威力** — Veteran、Zoomer、Enigmaが議論の質を大きく変えた
2. **ADR（Architecture Decision Record）の蓄積** — 全ての意思決定がドキュメントに残り、後から「なぜこうしたか」を追える
3. **Vertical Sliceアプローチ** — Sprint 1でE2Eを一本通し、Sprint 2で品質を固める流れが効率的だった
4. **ゼロコスト運用** — 全て無料枠で動く構成をArchitectが設計

### いかなかったこと

1. **顧客ヒアリングが後回しに** — 技術実装に集中しすぎた。Enigmaの「一人も顧客候補と話してないのに、機能の議論をしても空回り」が的中
2. **ペルソナ未定義** — Strategistが指摘したが着手できず。誰に売るかが曖昧なまま作った
3. **involuntary churn率の前提未検証** — 収益モデルの根幹である「3-5%」がまだ顧客データで裏付けられていない

**AIチームは「作る」のは速いが、「顧客と話す」ことの代わりにはならない。** ここは人間がやるしかない。

---

## この手法の限界

正直に書く。

1. **AIは顧客の代わりにはならない** — ペルソナを定義しても、実在の顧客からフィードバックをもらうことの代替にはならない
2. **同じ脳が10人に分裂しているだけ** — 結局はLLMの知識範囲内でしか議論できない。本当に予想外の視点は出にくい
3. **「人格の一貫性」の維持にコストがかかる** — プロンプトが長くなるほど各ロールのキャラがブレる
4. **人間の判断が最終的に必要** — AIチームの提案を取捨選択するのは人間。「全部AIにお任せ」はまだ無理

とはいえ、**個人開発者が一人で抱えがちな「視野の偏り」を安価に補正できる手法**としては、かなり有効だと感じた。

## 試してみたい人へ

リポジトリ構成はシンプルだ:

```
team/roles/           # 各ロールのプロンプト定義（10ファイル）
docs/decisions/       # ADR（意思決定記録）
docs/sprints/         # スプリント管理
docs/status/          # 各ロールのステータス
docs/handoff/         # ロール間の引き継ぎ
```

プロンプト設計のコツ:

1. **ペルソナに「人生の厚み」を与える** — 「批判的に考えて」ではなく「ドットコムバブルで痛い目を見た人」
2. **口癖を設定する** — キャラの一貫性が保たれる
3. **他ロールとの関係性を定義する** — 誰と組むか、誰と対立するかを明示する
4. **非エンジニアロールを必ず入れる** — 作る側だけの議論は偏る
5. **「止める人」を入れる** — 作りすぎを防ぐロールは最重要
