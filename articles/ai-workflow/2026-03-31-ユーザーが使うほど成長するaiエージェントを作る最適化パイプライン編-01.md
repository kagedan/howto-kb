## LangChainとDSPyを組み合わせる

AIエージェント開発において、**プロンプトエンジニアリングの属人化**という課題があります。  
そこで、LangChainの豊富なエコシステムと、DSPyの自動プロンプト最適化を組み合わせることを試みました。

LangChainとDSPyはそれぞれ異なる設計思想を持つフレームワークであり、「互換性がない」と評されることもありました。  
しかし、LangChainのグラフ構造を見直し、ルールを設けることで、2つのフレームワークを無理なく組み合わせることができました。

## 4層のAIエージェントアーキテクチャ

具体的には、LangGraphのグラフ構造を以下の4層に分離しました。

```
┌─────────────────────────┐
│    Service Layer        │  ← 外部システムとの接続
├─────────────────────────┤
│    Agent Layer          │  ← AIの推論ロジック (DSPy導入可能)
├─────────────────────────┤
│    Node Layer           │  ← システムの状態管理
├─────────────────────────┤
│    Graph Layer          │  ← フロー制御
└─────────────────────────┘
```

各層の依存関係にルールを設けることで、責務が明確になり、**Agent層にDSPyのモジュールをそのまま配置できる**構造を実現しました。  
これにより、LangChainのエコシステムを維持しながら、DSPyを活用できるようになりました。

```
┌─────────────┐                  ┌─────────────────┐
│ Agent Layer │── uses(tools) ──▶│  Service Layer  │  ← 外部システムとの接続
└─────┬───────┘                  └────────┬────────┘
      ▲                                   ▲
      │ calls                             │ calls
┌─────┴───────────────────────────────────┴────────┐
│                   Node Layer                     │  ← システムの状態管理
└───────────────────────┬──────────────────────────┘
                        ▲
                        │ calls
┌───────────────────────┴──────────────────────────┐
│                   Graph Layer                    │  ← フロー制御
└──────────────────────────────────────────────────┘
```

## FastAPIによるアプリケーション層の分離

AIエージェントシステム（`ai/`ディレクトリ）は、内部をパッケージ化して外部に公開する口だけを絞ることで、外部から隠蔽しました。  
そして、`ai/`と同じ階層に`app/`ディレクトリを設け、**FastAPI**を使ったAPIサーバを別で用意しました。

```
src
├── ai        # AIエージェントシステム（パッケージ化）
├── app       # FastAPIアプリケーション
│   ├── main.py
│   ├── schemas
│   └── service
│       └── ai_chat_service.py      # ai/ を呼び出すサービス
├── config
├── infra
├── log
└── optimizer
```

`app/service/ai_chat_service.py` が `ai/` のグラフを呼び出し、FastAPIのエンドポイントがそれをラップする、シンプルな構造です。

前回はここまでで、**「プロンプトの最適化」はまだ実装していない**状態でした。  
今回はいよいよ、この最適化部分を実装していきます。

# データパイプライン

## チャットデータの保存

AIエージェントが推論した結果は、後続の最適化のためにDBへ保存します。

保存するテーブルは2つです。

```
chat_sessions          # チャットのセッション単位
├── id (UUID)
└── created_at

chat_messages          # セッション内の個々のメッセージ
├── id (UUID)
├── session_id         # chat_sessions への外部キー
├── order_index        # セッション内での順番
├── assignee           # 担当エージェント名
├── message            # メッセージ本文
├── score              # メッセージの品質スコア
└── created_at
```

`assignee` カラムが重要で、後の最適化でエージェント別にデータを取り出す際のキーになります。また `score` は初期値 NULL で保存し、後述の評価ステップで書き込まれます。

## バックグラウンドでの保存

チャットのレスポンスを返した後、保存処理をバックグラウンドで非同期実行します。FastAPIの `BackgroundTasks` を使うと、レスポンスを返しながら後続処理を走らせることができます。

```
# app/main.py
@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest, background_tasks: BackgroundTasks):
    # ai/ディレクトリを呼び出し、レスポンスを返す
    response_content, full_result = ai_chat_service.chat(request.message)
    # バックグラウンドで保存
    background_tasks.add_task(save_chat_data, full_result)

    return ChatResponse(response=response_content)
```

`save_chat_data` は `app/` 側のサービスとして実装し、LangGraph から返ってきた `messages` リストを整形してDBへ書き込みます。

## 最適化タスクのキュー投入

`call_optimize` では、チャットに登場したエージェントタイプを特定し、対応する最適化タスクを Celery のキューに投入します。

```
# app/service/optimize.py
def call_optimize(messages: List[Any]):
    assignees = set()
    for message in messages:
        raw_content = getattr(message, 'content', '')
        try:
            parsed = json.loads(raw_content.strip())
            assignee = parsed.get('assignee')
            if assignee and assignee in VALID_AGENT_TYPES:
                assignees.add(assignee)
        except (json.JSONDecodeError, AttributeError):
            pass

    for agent_type in assignees:
        self_improve.delay(agent_type=agent_type)  # Celeryに非同期投入
```

`.delay()` がポイントです。FastAPI側からこの最適化トリガーを叩きますが、この時.delay()の呼び出し自体はタスクをキューに積んだ瞬間に完了するため、FastAPIは次の処理を実行できます。  
最適化の実際の実行は、別コンテナで動く最適化エンジンが非同期で引き受けることで、バックグラウンドで処理されるようになります。  
これにより、長時間の稼働が見込まれる最適化処理は、実際にユーザに直接関わるFastAPIの実行(パフォーマンス)を妨げません。

```
# app/main.py
@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest, background_tasks: BackgroundTasks):
    # ai/ディレクトリを呼び出し、レスポンスを返す
    response_content, full_result = ai_chat_service.chat(request.message)
    # バックグラウンドで保存
    background_tasks.add_task(save_chat_data, full_result)
    # ここで最適化タスクのキューを投入する
    background_tasks.add_task(call_optimize, full_result)

    return ChatResponse(response=response_content)
```

## 非同期処理

最適化は重い処理なので、FastAPIとは別のコンテナで実行します。タスクキューに **Celery**、ブローカーに **Redis** を採用しています。

```
# optimizer/_celery.py
celery = Celery(
    "tourist_ai",
    broker=f"redis://{host}:{port}/{db}",
    backend=f"redis://{host}:{port}/{db}",
    include=["optimizer.main"]
)
```

コンテナ構成はシンプルで、FastAPIコンテナが Redis にタスクを投入し、Celery ワーカーコンテナがそれを取り出して実行します。

```
┌─────────────────┐   .delay()   ┌─────────┐   pop task   ┌──────────────────────┐
│   FastAPI       │ ──────────▶  │  Redis  │ ──────────▶  │  Celery Worker       │
│  (app container)│              │ (broker)│              │ (optimizer container)│
└─────────────────┘              └─────────┘              └──────────────────────┘
```

## フィードバック制御（スコア閾値によるゲート）

無制限に最適化を走らせるのは非効率です。そこで「平均スコアが閾値を下回っているときだけ最適化を実行する」フィードバック制御を設けています。

```
# optimizer/main.py
@celery.task(bind=True)
def self_improve(self, agent_type: str):
    # スコアの付いていないメッセージを評価してスコアをDBに書き込む
    EvaluateNode().process(state={})

    # 平均スコアが閾値以上なら最適化をスキップ
    average_score = AgentRepository().get_field(agent_type, "average_score")
    if average_score is None or average_score < SCORE_THRESHOLD:
        graph = SelfImproveGraph()
        return graph.execute(agent_type)

    return {"status": "skipped", "reason": f"average_score={average_score:.1f} >= {SCORE_THRESHOLD}"}
```

スコアの評価自体も AI（`EvaluateNode`）が行います。後述しますが、`DeclarativeEvaluationAgent` が各メッセージを採点し、エージェントごとの平均スコアを `agents` テーブルに書き戻します。

# 最適化ロジック

## 評価フェーズ：スコアの自動付与

最適化の前に、まだスコアが付いていないメッセージを評価します。評価エージェントも DSPy で実装しています。

```
# ai/agent/evaluation/evaluation_agent.py
class EvaluationSignature(dspy.Signature):
    input: str = dspy.InputField(desc="Evaluation context including answer")
    score: float = dspy.OutputField(desc=EVALUATION_PROMPT)

class DeclarativeEvaluationAgent(dspy.Module):
    def __init__(self):
        super().__init__()
        self.lm = dspy_openai()
        self.agent = dspy.Predict(EvaluationSignature)

    def forward(self, input: str) -> Dict[str, float]:
        with dspy.context(lm=self.lm):
            result = self.agent(input=input)
        return result.score
```

`EvaluateNode` は未採点のメッセージを取得して評価エージェントでスコアを付け、エージェントごとの平均スコアも更新します。

```
# ai/node/evaluate/evaluate_node.py（抜粋）
messages = get_non_scored_specific_agent_messages()
evaluator = DeclarativeEvaluationAgent()

for message in messages:
    score = evaluator(input=message.message)
    update_message_score(message.id, score)

assignees = set(m.assignee for m in messages)
for assignee in assignees:
    update_agent_average_score(assignee)
```

## 最適化フェーズ：Optimizer の選定

DSPy には複数の Optimizer がありますが、今回は **MIPROv2**（Multi-prompt Instruction Proposal Optimizer v2）を採用しました。選定に至るまでの検討を順に説明します。

### LabeledFewShot ではダメなのか

最初に検討したのは **LabeledFewShot** です。DBに蓄積した会話ログ（入力と応答のペア）をそのまま Few-Shot 例としてプロンプトに差し込む、最もシンプルな方法です。

しかしこの方法には2つの問題があります。

* **指示文（Instruction）は変わらない** — Few-Shot 例を差し込むだけで、プロンプトの指示文自体は最適化されません。
* **どの例が良いか評価しない** — trainset の例をそのまま使うため、質の低い会話が混入してもフィルタリングされません。

### BootstrapFewShot ではダメなのか

次に検討したのは **BootstrapFewShot** です。LLM に実際にタスクを実行させ、品質の高い出力を Few-Shot 例として自動選別してくれます。

しかしこちらも **指示文の最適化はできません**。Few-Shot 例の選択にとどまります。

### SignatureOptimizer / BayesianSignatureOptimizer ではダメなのか

これらは**指示文の最適化に特化**した Optimizer です。複数の指示文候補を生成・評価して最適なものを選びます。逆に Few-Shot 例の最適化は行いません。

指示文だけ、あるいは Few-Shot 例だけを最適化しても、プロンプト全体としての品質向上は限定的です。

### MIPROv2 を選んだ理由

MIPROv2 は **指示文と Few-Shot 例を同時に最適化できるハイブリッドな Optimizer** です。

| Optimizer | 概要 | ラベル付きデータ |
| --- | --- | --- |
| **LabeledFewShot** | 既存の入出力ペアをFew-Shotとして直接使用 | **必須** |
| BootstrapFewShot | 実行結果から高品質な例を自動選択 | 不要 |
| MIPRO | 指示文とFew-Shotを同時最適化 | 不要 |
| **MIPROv2** | MIPROの改良版（探索戦略が向上） | 不要 |
| BayesianSignatureOptimizer | ベイズ最適化で指示文を最適化 | 不要 |

ラベル付きデータがなくても動作し、実際の会話ログを trainset として与えるだけで LLM が自律的に最適なプロンプトを探索してくれます。

MIPROv2 の仕組みは大まかに以下の通りです。

```
1. LLMで複数の指示文候補を生成（num_candidates個）
2. 各候補に対して trainset から最適な Few-Shot 例を選択
3. valset（事前に用意した評価データ）でスコアを評価
4. 最高スコアの（指示文 × Few-Shot 例）の組み合わせを採用
```

実装は以下です。

```
# ai/node/optimize/optimize_node.py（抜粋）
def optimize_agent(self, agent_type: str, agent_instance: Any, ...):
    # DBからトレーニングデータを準備（人間の入力とエージェントの応答のペア）
    trainset = self.prepare_training_data(agent_type)

    # バリデーションデータ（事前定義の評価用データセット）
    valset = create_agent_valset(agent_type)

    lm = dspy_openai()
    dspy.configure(lm=lm)

    optimizer = MIPROv2(
        metric=metric,          # 評価関数（後述）
        prompt_model=lm,        # 指示文生成に使うLM
        task_model=lm,          # タスク実行に使うLM
        init_temperature=1.0,
        num_threads=num_threads,
    )

    optimized_agent = optimizer.compile(
        student=agent_instance,
        trainset=trainset,
        valset=valset,
    )

    # 最適化済みエージェントをJSONに保存
    optimized_agent.save(f"{self.output_dir}/{agent_type.lower()}_agent_miprov2.json")
```

## 評価メトリクス

MIPROv2 がプロンプトの良し悪しを判断するために、メトリクス関数が必要です。ここでも `DeclarativeEvaluationAgent` を再利用しています。

```
# ai/node/optimize/metric.py
_evaluator = DeclarativeEvaluationAgent()

def metric(example, prediction, trace=None):
    try:
        output = prediction.get("output", "") if isinstance(prediction, dict) else prediction.output
        score = _evaluator(input=output)
        return float(score) / 100.0  # 0.0〜1.0に正規化して返す
    except Exception:
        return 0.0
```

評価に用いた AIエージェントを利用することで、一貫した品質基準で最適化が進む設計になっています。

## トレーニングデータの準備

DBに蓄積された会話履歴から、（人間の入力, エージェントの応答）のペアを取り出してトレーニングデータにします。

```
# ai/node/optimize/optimize_node.py（抜粋）
def prepare_training_data(self, agent_type: str):
    message_pairs = get_message_pairs_by_assignee(agent_type)

    train_examples = []
    for human_msg, agent_msg in message_pairs:
        if human_msg.message and agent_msg.message:
            example = dspy.Example(
                input=human_msg.message,
                output=agent_msg.message
            ).with_inputs("input")
            train_examples.append(example)

    return train_examples
```

`get_message_pairs_by_assignee` は `assignee` カラムを使って、特定エージェントが応答したメッセージとその直前の人間メッセージをペアで取得します。データパイプラインで `assignee` を丁寧に保存しておいた意味がここで出てきます。

## 全体の最適化フロー

ここまでをまとめると、最適化の一連のフローは以下のようになります。

```
チャット終了
    ↓
[バックグラウンド] save_chat_data  →  DB (score=NULL で保存)
    ↓
[バックグラウンド] call_optimize   →  Celery に .delay() でキュー投入
    ↓
[Celery Worker] self_improve タスク実行
    ↓
EvaluateNode  →  未採点メッセージを評価 → score & average_score を更新
    ↓
average_score < 80.0 ?
    ├── NO  → スキップ（十分な品質）
    └── YES → SelfImproveGraph 実行
                  ↓
              OptimizeNode (MIPROv2)
                  ↓
              最適化済みプロンプトを JSON に保存
```
