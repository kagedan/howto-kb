先日、日本経済新聞が **「社員と化すAIエージェント、費用は人件費？ 人事とITの境界溶かす」** という記事を出した。

読んで、思わず「それ、もう設計思想として実装しています」と声が出た。

私たちは **TechHive Agent（以下 THA）** という、**「AIエージェントを"社員"として企業に派遣する BPO プラットフォーム」** を作っている。日経が"これから起きること"として書いた論点は、私たちにとっては"どう作るかを決めた選択"そのものだった。

この記事は、日経が投げかけた3つの問い——**①費用は人件費なのか ②中身は何なのか ③人事とITの境界は本当に溶けるのか**——への、実装側からのアンサーである。

---

## 問い①：費用は「人件費」なのか？

答えは **Yes、比喩ではなく課金モデルとしてそう作っている。**

多くの AI プロダクトは「ツール」として売られる。席数（シート）や API コール数、機能アンロックで課金する SaaS モデルだ。THA はここで明確に舵を切っている。

> **ツールを売らない。人（AIエージェント）を派遣する。**

採用担当エージェント、経理担当エージェント——専門ペルソナを「社員」として企業に置き、**業務量に対して人月で契約する**。価格が"席数"ではなく"こなした業務"に紐づく。これは料金表の都合ではなく、プロダクトの芯（＝絶対に曲げない不変条件）として最初に固定した。

日経が「費用は人件費？」と問うたとき、市場はまだ「SaaS なのか人材なのか」を迷っている。私たちの回答は、**課金の単位を人件費側に寄せた瞬間、プロダクトの作り方が全部変わる**、という実装上の事実だ。席課金なら「機能をどれだけ載せるか」を競うが、人月課金なら「その"社員"が実際に業務を終わらせたか」しか価値にならない。**Doing over Talking**——手順書を返すのではなく、実際に操作して業務を反映する。ここが BPO を名乗る前提になる。

---

## 問い②：その"社員"の中身は何なのか？

答えは **中身は全部 Claude。私たちはエージェントを"自作"しない。**

ここがエンジニア読者に一番伝えたい設計判断だ。THA のアーキテクチャは「脳・手」で綺麗に割れている。

- **脳 = Claude（Managed Agents）** — LLM 推論・ブラウザ操作・判断は全部クラウド上のマネージドなエージェント実行基盤が担う。
- **手 = 自社アプリ** — DB 書き込み、外部 API 呼び出し、トリガー受信、セッションのライフサイクル管理。役割はこの3カテゴリに徹する。

なぜ自前でエージェント（推論ループやプロンプトオーケストレーション）を抱え込まないのか。**モデルの進化速度に乗り続けるため**だ。基盤モデルが賢くなるたびに"社員"が勝手に有能になる構造にしておけば、こちらは業務インターフェースの品質だけに集中できる。逆に推論ループを自作した瞬間、それが技術的負債になり、モデル進化の恩恵を自分で塞いでしまう。

「基盤ベンダーの新プリミティブは貪欲に取り込む。でも推論エンジンの主軸は乗り換えない」——プリミティブ採用と主軸乗り換えを混同しないこと。これも芯として明文化している。

> 余談：この判断を守るために、私たちは日次で回る**自己改善スカウトエージェント**を別に走らせている。外部の AI エージェント／BPO ニュースと基盤（Anthropic）の更新を毎日集め、「プロダクトの芯を壊さずに取り込めるものだけ」を選別して提案させる。今回この日経記事を拾ったのも、そのエージェントだ。この記事自体、ある意味 AI エージェントが見つけてきた話題への人間側のアンサーでもある。

## 問い③：「人事とITの境界」は本当に溶けるのか？

答えは **溶ける。ただし"溶かしてはいけない境界"を同時に引かないと、ただの事故になる。**

日経の見出しの核心はここだと思う。従来「AI導入」は情シス（IT）の仕事だった。でも THA が狙うのは **非エンジニアのバックオフィス担当者が、日本語で"社員"に仕事を頼める**世界だ。ツールの設定でも API 連携でもなく、**採用・オンボーディングの体験**として AI を迎える。1ペルソナ = 1エージェント = 1業務フロー。「経理の子を1人増やす」感覚に近い。この意味で、人事とITの境界は確かに溶ける。

**が、ここからが実装の本題**だ。"社員"に全権委任してはいけない。境界を溶かすのと、統制を捨てるのは別物だ。私たちは逆に、**溶かしてはいけない境界を技術で固く引いている**：

- **取り消し不能・外部作用のある操作には人間承認（HITL）を挟む** — 送信・支払い・確定のような後戻りできない行為は、リスク階級に応じて人間のゲートを通す。"社員"は提案するが、引き金は人が引く。
- **ツールへのアクセスは単一のゲートに集約する** — エージェントが DB や外部 API を直接触ることを許さない。全ての操作を1つの制御層（MCP）経由に強制することで、テナント分離と監査を担保する。
- **マルチテナントは行レベルで越境を封じる** — 複数企業の"社員"が同じ基盤に相乗りしても、データが org を跨がない設計を土台に置く。

つまり THA の答えは、**「境界を溶かす」＝「無秩序にする」ではない**、ということだ。人事とITの壁は溶かすが、**"AIに任せていい範囲"と"人が握る範囲"の境界は、むしろ従来より明示的に引く**。この線引きこそが、BPO＝業務を預かる責任の中身だと考えている。

## まとめ：これは未来予測ではなく、もう下した設計判断

日経の記事は「これから起きること」として書かれていた。実装している側からすると、3つの問いの答えはこうなる：

| 日経の問い | 実装側の回答 |
|:---|:---|
| 費用は人件費か？ | **Yes。課金の単位を人件費側に置くと、プロダクトの作り方が全部変わる** |
| 中身は何か？ | **全部 Claude。エージェントを自作せず、基盤の進化に乗り続ける** |
| 人事とITの境界は溶けるか？ | **溶ける。ただし"人が握る境界"は技術で固く引き直す** |

「AIエージェントの費用は人件費」は、キャッチコピーではなく **一度選ぶと後戻りできない設計上の分岐点**だ。SaaS として売るのか、"社員"として派遣するのか。私たちは後者を選び、その帰結として脳（Claude）・手（アプリ）・統制（HITL/ゲート/テナント分離）を今の形にした。

日経が市場に問いを投げてくれたのは追い風だと思う。あとは、**問いに実装で答え続けられるか**が私たちの仕事だ。
