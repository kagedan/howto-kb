## はじめに

今回はWebサイトのビジュアルテストをやりたくてAIお願いしたときの話になります。

Webアプリとか作ってると開発中なども見た目のレイアウト崩れとか確認するためにビジュアルテストとかやるときあります。

ビジュアルリグレッションテストって言ってしまうと、それはそれで基本となる画像と今回の差分をとってデザインの崩れてないかっていうの検出させたりするんで、ちょっと大掛かりになってきますので...

実用ベースで考えて...AIにWebサイトの全画面キャプチャーをPlaywriteで作ってもらいました。今回はCodexデスクトップ版を使って一発で出来るのかちょっとやってみました。  
以前別のAIを使って作ってもらったときも一瞬でできたのでCodexでも大丈夫だと考えてました。

結果....

* ビジュアルテスト用のスクリプトも作ってくれて...
* かつ実行して全画面キャプチャまで撮ってくれました...
* しかも下部までスクロールしてキャプチャするFULLページのキャプチャでした...

以前のAIは「スクリプトを実行してください」って感じでしたが...今回は実行までやってしまってます。うーん...このあたりはいいのか...わるいのか...

また、AIの回答を見るとその後ビジュアルリグレッションテストに発展させることも考えてくれている感じでした。

ビルドは通りました。スクショ実行はテスト自体ではなくPlaywrightのChromium本体が未インストールで止まったので、追加した `test:visual:update-browsers` を使ってブラウザを入れてから再実行します。

全50ケース、desktop/mobile両方で通ってスクリーンショットも生成されました。最後に変更差分と出力数だけ確認して、使い方が分かる状態に整えます。

Playwrightで組み込みました。今でもこの用途ならPlaywrightがかなり素直です。まずは「全ページを人間レビュー用にキャプチャする」形にして、必要になったら差分検知型の visual regression に育てられます。

追加したもの:

* `package.json` に `test:visual` と `test:visual:update-browsers`
* `playwright.visual.config.ts` に desktop/mobile の2 viewport設定とVite自動起動
* `sitemap-screenshots.spec.ts` で静的ページ、`/template/*`、`/episode/*` を既存データから自動列挙

使い方:

```

npm.cmd run test:visual
```

### 🤖 AI

この用途だと、だいたい次の呼び方です。

* **Visual Regression Test / ビジュアルリグレッションテスト**: 以前のスクショと今回のスクショを比較して、見た目の差分を検出するテスト。
* **Screenshot Test / スクリーンショットテスト**: 今回作ったように、ページを開いてスクショを保存・確認する広めの呼び方。
* **Visual Testing / ビジュアルテスト**: レイアウト崩れや見た目の確認全般。人間レビュー込みでも使いやすい言葉です。

今回のツールは厳密にはまだ「差分比較」まではしていないので、名前としては スクリーンショットE2E や ビジュアル確認用スクリーンショットテスト が近いです。将来的に基準画像と比較するようにしたら、はっきり ビジュアルリグレッションテスト と呼べます。
