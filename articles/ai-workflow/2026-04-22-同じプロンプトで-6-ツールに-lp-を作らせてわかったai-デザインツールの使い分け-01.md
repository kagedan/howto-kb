## はじめに

2026-04-17、Anthropic が Claude Design（Research Preview）を発表した。[claude.ai](https://claude.ai/) のキャンバス上で、対話しながらデザインを作れる機能である。触ってみて最初に出てきた画面を見て、素直に「これワンショットで LP 作れるやつだな」と感じた。

で、そこから思考が一歩進んだ。今の AI 時代、デザインを作る手段はどれだけ増えていて、結局どれを使うのがいいのか。思い当たる選択肢を並べてみると、手数がやけに多い。

* [Google Stitch](https://stitch.withgoogle.com/)（Gemini 系モデル内蔵のデザイン生成サービス）
* Claude Design（前述）
* [Pencil](https://www.pencil.dev/)（デザインファイルを直接編集できるエディタ、MCP 連携あり）
* Claude Code / Gemini CLI / Codex CLI にデザイン原則を読ませてのワンショット HTML 生成

全量を網羅しきれているわけではないが、この 6 つを同じ条件で比べれば、少なくとも「どの方向に行けばいいか」の当たりはつくはずだ。本記事はその検証ログである。

結論を先に書いておく:

* ビジュアルの磨きで選ぶなら **Claude Design**
* 実装まで含めた総合力なら **Claude Code にデザイン原則を読ませる構成**
* 同じ原則を渡してもモデルで出力はまるで別物になる（ここが一番の収穫だった）

以下、検証の設計から順に見ていく。

## 検証の設計

### 題材: 架空 B2B SaaS「Kaiwa」の LP

題材は B2B SaaS の Landing Page にした。情報階層・タイポ・コピー・CTA 設計・ビジュアルが同時に問われるので、ツールの総合力が見やすい。

プロダクトは架空の「Kaiwa」を用意した:

* カテゴリ: Meeting Intelligence（会議音声を自動で議事録 / ToDo 化）
* One-liner: 会議が終わる頃には、議事録も ToDo も出来ている
* ターゲット: 日本の 10〜50 人規模スタートアップの PM / マネージャー

### 統一プロンプト（全文）

全ツールに同じプロンプトを渡した。コピー・セクション構成・デザイン方向性まで同一にしてある。

統一プロンプト全文を見る

```
# ランディングページ生成依頼

## プロダクト情報

- プロダクト名: Kaiwa
- カテゴリ: B2B SaaS / 会議インテリジェンス
- キャッチコピー（H1 候補）: 会議が終わる頃には、議事録も ToDo も出来ている
- 言語: 日本語

## ターゲットユーザー

- 主な対象: 10〜50 名規模スタートアップの PM・マネージャー層
- 課題: 会議後の議事録作成・ToDo 漏れ・決定事項追跡に時間が奪われている

## 主要機能（3 つ）

1. 会議音声の自動文字起こし + 要約
2. 決定事項・アクションアイテムの自動抽出
3. Slack・Notion・Linear への自動連携

## デザイン方向性

- トーン: 落ち着いていて信頼感がある。誠実でプロフェッショナル
- 参考トーン: Linear・Notion・Stripe のような情報密度の低い余白重視スタイル
- 避けること: 派手なグラデーション、絵文字の多用、立体的なスタック画像
- 配色: ニュートラルな背景色（白またはオフホワイト）+ 1 色のアクセントカラー
- 文字ウェイト: 見出しは太め・本文は細め〜中程度のコントラストで読みやすく

## 必須セクション（この順序で）

1. Hero — キャッチコピー、サブコピー、メイン CTA
2. Problem — 読者の課題に共感するセクション（3 項目程度）
3. Features — 主要機能 3 つの説明
4. How it works — 利用開始〜成果までの 3 ステップ
5. Testimonials — 3 名の導入事例コメント
6. Pricing — 3 プラン（フリー・スタンダード・プロ）
7. FAQ — 5〜7 問
8. Final CTA — クロージングとアクション誘導

## コピーライティング指定

各セクションに下記のコピー案を使用すること（文言変更は最小限に）。

（※ 記事では省略。実際には Hero / Problem / Features / How it works /
Testimonials / Pricing / FAQ / Final CTA の各セクションに
H1・H2・サブコピー・CTA ラベル・本文まで完全指定している）
```

ツール固有の微調整（Stitch には "Desktop 1440x900"、HTML 生成系には "single HTML file" 指定など）は最小限に留めた。

### 評価軸（8 軸）

全軸とも 5 = 優秀、1 = 不足。軸 8 の「実装コスト」も "リリース到達までの修正量が少ない = 高スコア" に揃えている。

1. AI 臭のなさ — 「AI が作った感」が強くないか
2. 情報設計の質 — セクション階層・情報密度
3. タイポ・余白の精度 — Linear / Notion 級の精度で組まれているか
4. コピーへの忠実度 — 与えたコピーを改変せずに使えているか
5. 開発ハンドオフ適性 — そのまま実装に流せるか
6. 編集自由度 — 細部調整がどれだけ効くか
7. 再現性 — 同じプロンプトで再生成した時に安定するか
8. 実装コスト — リリース品質に到達するまでの修正工数（少ないほど高スコア）

## 6 ツールの結果

### 3-1. Claude Design（Anthropic Research Preview）

キャンバス上で対話しながらデザインを作る Research Preview。2026-04-17 発表。これが今回の検証のきっかけになったツールでもある。

![](https://static.zenn.studio/user-upload/d27f8cc384b7-20260423.png)  
![](https://static.zenn.studio/user-upload/a1669f26d56f-20260423.png)

結果は素直に一番綺麗だった。B2B SaaS らしい trust 感、余白の呼吸、タイポの重み——どれも丁寧で、「このまま叩き台として使える」画面が 1 発で出る。Claude 系特有の warm tone + editorial な寄せ方はあるが、Kaiwa の題材にはむしろ味方してくれた。

懸念は 2 つ。Research Preview なので仕様が流動的で、コードエクスポート経路の実態までは今回検証しきれなかったこと。そして対話型で磨き込む前提の UI なので、反復するほどトークン消費が膨らむ。ワンショットで済む題材ならいいが、毎回細かく詰めたい用途では運用コストが気になる。

### 3-2. Stitch（Google Labs、内部 Gemini 3.0）

ブラウザで動くデザイン生成サービス。内部は Gemini 3.0 系のモデル。

![](https://static.zenn.studio/user-upload/204670c86fbc-20260423.png)  
![](https://static.zenn.studio/user-upload/d7c9b077ecaa-20260423.png)

画像崩れ問題を除けば、Claude Design にタメを張るレベルの完成度だった。Hero 付近で「プロダクト UI プレビュー風の画像」を生成しようとしてテキストが文字化けするのが第一印象をだいぶ損ねるが、それ以外は整っている。色味は Gemini 系らしく cool / clean 寄りで、Claude 系とは明確に別方向の味を出す。

叩き台としてはかなり優秀。画像崩れは Stitch のモデルグレードを上げれば改善する可能性があるので、この軸は流動的だと思う。

### 3-3. Pencil（MCP 経由）

デザインファイル（`.pen`）を直接編集できるエディタ。今回は Claude Code から MCP 経由で操作して生成した。

![](https://static.zenn.studio/user-upload/267478f3205d-20260423.png)  
![](https://static.zenn.studio/user-upload/5fcf6cf6cac7-20260423.png)

出てきたものはシンプル、そして不揃い。Hero 付近の UI プレビュー要素が欠落し、全体的に「必要最低限の枠組みが並んでいるだけ」の印象になった。色味は Claude 系寄りに落ち着いた（Claude 内部で構築しているため、これは自然な結果だ）。

そもそも Pencil の MCP 操作は「要素単位で組む」モデルで、LP 1 枚でオペレーションが 150 を軽く超える。これを 25 operation ごとに分割しながら積み上げる構造になっていて、正直「ワンショットで仕上げる」用途には向いていない。デザイン経験者が手を入れる前提のツールだと割り切るのが正しい使い方だと思う。

### 3-4〜6. CLI にデザイン原則を読ませて HTML 生成（Claude / Gemini / Codex）

ここから 3 つは、同じデザイン原則（Linear / Notion / Stripe を参照した 4px グリッド・borders-only depth・Serif × Sans × Mono の 3 層タイポを規定するもの）を、異なる CLI + モデルに読ませて比較したものである。

ただし厳密には 3 者で原則の「読ませ方」は揃っていない。Claude Code は Skill 機構で発動させ、Gemini / Codex は各 CLI の仕様に合わせて原則文書をプロンプトに含める形にした。完全に同条件の比較ではない点、"共通の原則をどのモデルに読ませるか" の比較として見てほしい。

#### 3-4. Claude Code + デザイン原則

![](https://static.zenn.studio/user-upload/b8633ed60d9c-20260423.png)  
![](https://static.zenn.studio/user-upload/c01598b5dccf-20260423.png)

Claude Design に迫るビジュアル完成度で、しかも HTML 直出力なので実装ハンドオフがほぼゼロだった。4px グリッド厳守、Feature カード 3 種の内部構造変化（transcript 風 / ToDo リスト風 / integration grid 風）、Hero 直下の UI 実プレビュー（静的スクショ風）まで、原則が隅々に効いている。

ビジュアルの磨きだけ見れば Claude Design の方がわずかに上。ただし実装物として使える価値を含めると、本検証の総合 1 位はこちらになった。

#### 3-5. Gemini + デザイン原則

![](https://static.zenn.studio/user-upload/f672d674d7cb-20260423.png)  
![](https://static.zenn.studio/user-upload/06cb856d3bb1-20260423.png)

破綻はしていない。ただシンプル過ぎる。4px グリッドは守るのに情報密度が低く、B2B SaaS としての説得力が弱い。色味は Stitch に近くて、Gemini 系モデルがスパース寄りに寄るバイアスは単体でも出る。

Mock 用途なら許容範囲。ただし「そのままリリース」には届かない。主観では 60 点。プロンプトを相当補強すればもっと上げられそうな感覚はあるが、デフォルトの出力はこんなものだった。

#### 3-6. Codex + デザイン原則

![](https://static.zenn.studio/user-upload/a4d673777a94-20260423.png)  
![](https://static.zenn.studio/user-upload/a4b8b27e3475-20260423.png)

レイアウトが崩壊した。ワンショットで出てきた HTML は、正直そのままでは使い物にならない。色々な箇所が崩れていて、修復に入るなら「書き直した方が早い」レベル。Codex は設計判断・デバッグ・深い推論には本当に優秀なのだが、美的判断を伴う UI 生成はやはり別タスクなのだと再確認した。

この「同じ原則・同じプロンプトでここまで変わる」という事実が、次の節で最大の気づきとして効いてくる。

## 主要な発見

### 発見 1: 同じ原則を渡しても、モデルで出力は別物になる

今回の検証で一番印象に残ったのがこれだ。

同じデザイン原則を Claude / Gemini / Codex に読ませて LP を生成した結果、出力はまったく別物になった。

* Claude: Claude Design に迫る完成度（合計 39 点）
* Gemini: 破綻はないがシンプル過ぎ（合計 30 点）
* Codex: レイアウト崩壊（合計 23 点）

「プロンプトや原則を揃えれば品質も揃う」——そういう期待をつい抱くが、実際にはモデル側のデザインセンスがかなり支配的だった。

原則やプロンプトは、モデルの振る舞いを調整する道具ではあっても、モデルの能力そのものを書き換える魔法ではない。整理すると:

* モデル選択が依然として最重要の変数
* プロンプト工学や原則設計は「モデルが持っている能力を引き出す」方向で効く
* 逆に「モデルに無い能力を足す」ことはできない
* 「Codex は賢いから原則さえ読ませれば Claude と同じ出力が出るはず」という仮説は、今回の範囲では成立しなかった

### 発見 2: モデル系列による「色味バイアス」

同じプロンプト・同じデザイントークンを指定したのに、出力の色味・密度感がモデル系列で明確に分かれた。

| モデル系列 | 色味・密度の傾向 | 例 |
| --- | --- | --- |
| Claude 系 | Warm tone + editorial、密度あり | Claude Design / Claude Code / Pencil |
| Gemini 系 | Cool / clean、スパース寄り | Stitch / Gemini |
| Codex | デザイン判断できず、レイアウト崩壊 | Codex |

Pencil は Claude 内部で構築しているので Claude 系の色味に落ち、Stitch は内部 Gemini なので Gemini 系に寄る。つまり "ツール" ではなく "backing model" が色味を決めていた。

これは運用時に地味に効く事実で、「狙ったテイストに合わせてモデルを選ぶ」という逆引きの使い方が成立する。cool な色味が欲しければ Gemini 系、editorial な warmth が欲しければ Claude 系、という形で最初のツール選択を切り替えればいい。

### 発見 3: ワンショットで 100 点は出ない

どのツールでも共通する「ワンショット限界」があった。

* 画像内テキストの文字化け（特に Stitch で顕著。Hero UI プレビューの再現が崩れる）
* 複雑な UI 要素の解釈の弱さ（Pencil で Hero の UI プレビューが抜け落ちる）
* テンプレ的な構図の残存（全ツール共通で、「AI が作った感」がゼロにはならない）

これを前提に運用を組み立てる方が現実的だ。1 発で完璧を狙うより、「叩き台を早く作って磨き込む」フローの方が結果的に速いし綺麗に仕上がる。

## 運用ワークフロー提案

検証結果を踏まえた、現実的な 3 フェーズモデル。

```
┌─────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│ Phase 1         │    │ Phase 2          │    │ Phase 3          │
│ Concept         │ →  │ Coded Artifact   │ →  │ Polish           │
├─────────────────┤    ├──────────────────┤    ├──────────────────┤
│ Claude Design   │    │ Claude Code      │    │ 手動コード編集    　│
│ or Stitch       │    │ + デザイン原則     │    │ or Pencil        │
└─────────────────┘    └──────────────────┘    └──────────────────┘
```

### Phase 1: Concept / First Draft

ワンショットで叩き台を作るフェーズ。

* Claude Design: 一番綺麗な叩き台が出る。ただし対話型なのでトークン消費には注意
* Stitch: Gemini 系の色味で別方向性を見たい時に並走させる
* いずれも 1〜2 発で方向性を決めてしまうのが良い

### Phase 2: Coded Artifact

デザインをコードに落とし込むフェーズ。

* Claude Code にデザイン原則を渡して HTML 生成。Phase 1 のスクショ or 構造説明を input として渡す
* 4px グリッド、borders-only depth、Serif × Sans × Mono の 3 層タイポといった原則がそのまま適用される
* 成果物は single HTML file として即運用できる

### Phase 3: Polish

細部を調整するフェーズ。

* 非デザイナーは、手動で HTML / CSS を編集するのが素直
* デザイナーが入れるなら、Pencil に取り込んで視覚的に磨く（MCP ではなく Pencil アプリで直接編集する方が速い）

### 避けるべき構成

検証から出てきた、避けた方がいい組み合わせ。

* ❌ Codex でデザインを生成する — レイアウト崩壊が常態化する。Codex はコード実装・デバッグ・深い推論に専念させる
* ❌ Gemini 単体で LP デザインを起こす — 60 点止まりで Mock から脱出できない
* ❌ 非デザイナーが Pencil ワンショットで仕上げる — 不揃いが目立って修復コストが高い

### トークン消費の現実的な落としどころ

Claude Design は対話型なので、磨き込むほどトークンが膨らむ。実運用を考えた時の代替ルートとして、こんな選択肢もある:

* Stitch で叩き台 → Figma or Pencil で微調整 → Claude Code でコード化
* Claude Code にデザイン原則を渡してワンショット生成 → 手動で磨き込み

題材とコスト許容度によって Phase 1 のツールを切り替えるのが実務的だ。

## まとめ

今回の検証で残った要点を並べると、こんな感じになる。

ビジュアル単体で勝負するなら Claude Design。実装まで含めた総合力なら Claude Code にデザイン原則を渡す構成。同じ原則を渡しても、モデルで出力はまるで別物になる。モデル系列で色味と密度感が寄るので、狙ったテイストに合わせてモデルを選ぶ逆引きができる。ワンショットで 100 点は出ないので、早く叩き台を作って磨き込む方が結果的に速い。そして、Codex をデザインに使うのは今のところ厳しいし、Gemini 単体は Mock 止まり、Pencil は非デザイナー単独だと仕上げきれない。

AI デザインツールの選択は、「ツール」単位で考えるのではなく「モデル + ツール + 原則」の三位一体で見る必要があると感じた。そして三位の中でも、モデル選択の重みが圧倒的に大きい。

プロンプトエンジニアリングや原則設計に時間を使う前に、「このタスクにはどのモデルが合うか」を先に問う。この順序の方が投資対効果が高い、というのが今回の一番の学びだった。もちろんモデルは進化するので、半年後に同じ検証をしたら順位がひっくり返っている可能性も普通にある。そういう前提で、この記事は 2026-04 時点のスナップショットだと思って読んでほしい。
