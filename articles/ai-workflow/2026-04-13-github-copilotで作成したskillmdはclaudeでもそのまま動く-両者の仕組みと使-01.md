## はじめに

先日、GitHub Copilotの『SKILL.md育成RPG』という記事を書きました。SKILL.mdを段階的に育てることで、AIエージェントのコードレビュー精度が劇的に変わるという検証です。

記事を公開した後、ふと気になったのが——**「これ、Claudeでも同じことできるの？」**

GitHub CopilotもClaudeも、どちらもAgent Skillsに対応しています。SKILL.mdのフォーマットも共通仕様。じゃあ実際に試したらどうなるのか。そして仕組みとして何が違うのか。

この記事では以下を扱います：

1. **実際にClaude環境でSKILL.mdを使ってコードレビューを実行した結果**
2. **GitHub CopilotとClaudeの技術的な違い**
3. **どう使い分けるべきか**

※ いずれもモデルは Claude Opus 4.6 を利用しています。

## Claudeの実行環境について

まず前提として、Claudeには複数の実行環境があります。

| 環境 | 概要 | スキルの管理方法 |
| --- | --- | --- |
| **Claudeデスクトップアプリ** | macOS/Windows向けのネイティブアプリ。Chat・Coworkモードを搭載 | Settings → Capabilities からZIPアップロード、または会話内で作成。Claude Code連携ONでローカルの `.claude/skills/` も認識 |
| **Claude Code（CLI）** | ターミナルで動くコーディングエージェント | `.claude/skills/` にファイル配置で自動認識。`/skill-name` でスラッシュコマンド呼び出しも可能 |
| **claude.ai（ブラウザ）** | Web版チャット | デスクトップアプリと同様にSettings → Capabilitiesからスキルアップロード |
| **Claude API** | プログラム呼び出し。CI/CD連携向き | システムプロンプトとして渡す |

今回は**Claudeデスクトップアプリ**で検証しました。デスクトップアプリを選んだ理由はシンプルで、GitHub Copilot（VS Code上のGUIツール）と比較するなら、同じくGUI環境であるデスクトップアプリが最も公平な比較対象だからです。

デスクトップアプリでは、カスタムスキルをZIPファイルとしてアップロードするか、会話内で `skill-creator` スキルを使って対話的に作成できます。一度登録すれば、関連するタスクをClaudeが検知した時点で自動的にスキルが呼び出されます。この「descriptionベースの自動選択」の仕組みはGitHub Copilotと同じです。

さらに、デスクトップアプリには**Claude Code on Desktop**という連携機能があります。これをONにすると、デスクトップアプリのGUIからClaude Codeの機能（ローカルファイルの読み書き、bashコマンド実行、Git操作）が使えるようになります。この場合、ローカルの `.claude/skills/` に配置したスキルも自動認識されます。

## 実際にやってみた：SKILL.md構成の移植

前回の記事で使ったGitHub Copilot用の構成を、Claudeデスクトップアプリ用に移植しました。ここでは**具体的にどのファイルを変更し、何を新規作成したか**を明示します。

### GitHub Copilot版の元構成（前回の記事）

```
.github/
├── agents/
│   └── bug-hunter.agent.md       ← エージェント定義
├── skills/
│   └── code-review/
│       ├── SKILL.md               ← スキル本体
│       ├── references/
│       │   └── checklist.md       ← チェックリスト
│       ├── scripts/
│       │   └── static-analysis.mjs ← 静的解析スクリプト
│       └── assets/
│           └── report-template.md  ← 出力テンプレート
└── copilot-instructions.md        ← プロジェクト全体の指示書
```

### Claude版で変更・作成したファイル一覧

| ファイル | 操作 | 変更内容 |
| --- | --- | --- |
| `SKILL.md` | **そのまま流用** | 変更なし。YAMLフロントマター（name, description）も本文もそのまま |
| `references/checklist.md` | **そのまま流用** | 変更なし |
| `assets/report-template.md` | **そのまま流用** | 変更なし |
| `scripts/static-analysis.mjs` | **そのまま流用**（※） | 変更なし。Claude Code連携ONならbash経由で実行可能。Chat単体では実行不可 |
| `bug-hunter.agent.md` | **不要** | Claudeにはエージェント定義ファイルの概念がない。SKILL.mdのdescriptionでスキルが自動選択される |
| `copilot-instructions.md` | **不要** | Claude Code利用時は `CLAUDE.md` が対応する役割を持つ（後述）。デスクトップアプリのChatモードではカスタム指示やプロジェクト指示で代替 |

**スキルの中核ファイル（SKILL.md + references + assets）は一切変更なしで動きました。**

変更が必要だったのは「周辺の仕組み」だけです：

* **エージェント定義**（`.agent.md`）：GitHub Copilot固有の概念。Claudeではdescriptionベースの自動選択に任せるため不要
* **プロジェクト指示書**：GitHub Copilotの `copilot-instructions.md` に対応するのが、Claude Codeでは `CLAUDE.md`。どちらもプロジェクト全体のルール（使用言語、コーディング規約、技術スタックなど）を記述するファイルで、AIが最初に読み込む「前提知識」として機能する。デスクトップアプリのChatモードでは `CLAUDE.md` は読み込まれないため、プロジェクト指示やカスタム指示で代替する
* **スクリプト実行**：GitHub CopilotはVS Codeのターミナル経由で実行可能。デスクトップアプリ単体ではスクリプト実行不可だが、Claude Code連携をONにすれば実行可能

### Claudeデスクトップアプリへの登録手順

1. `SKILL.md`、`checklist.md`、`report-template.md` をまとめてZIP化
2. デスクトップアプリの Settings → Capabilities → カスタムスキルからアップロード
3. チャットでコードを貼り付けて「このコードをレビューしてください」と依頼

または、会話内で `skill-creator` を使い「コードレビュースキルを作りたい」と伝えて対話的に作成することもできます。

## レビュー結果：Claudeでも同じブレイクスルーは起きるのか

前回の記事と同じ10個のバグ入りExpressサーバーを使い、Claudeデスクトップアプリで段階的にスキルを強化しました。

| Lv | 追加要素 | 検出数 | 構造化出力 | 修正コード例 | 行番号 |
| --- | --- | --- | --- | --- | --- |
| 0 | なし（素のClaude） | 9-10/10 | ❌ バラバラ | ほぼあり | 一部 |
| 2 | 手順書追加 | 10/10 | やや統一 | あり | 一部 |
| 3 | チェックリスト追加 | 10/10 | 統一 | あり | あり |
| 5 | **出力テンプレート追加** | **10/10** | **✅ 完全** | **全件** | **全件** |
| 7 | 自己評価ループ追加 | 10/10 | ✅ 完全 | 全件 | 全件 |

**前回の記事と同じ結論が出ました。** Lv.5（出力テンプレート追加）が最大のブレイクスルーです。

「フォーマットが思考を規定する」——この法則はGitHub Copilotに限った話ではなく、AIエージェント全般に当てはまる普遍的な特性でした。どのモデル、どのツールを使っても、出力テンプレートを定義した瞬間にレビュー品質が跳ね上がります。

## GitHub Copilot vs Claudeデスクトップアプリ — 技術的に何が違うのか

ここからが本記事の核心です。SKILL.mdの共通性を踏まえた上で、両者のアーキテクチャの違いを整理します。

### コンテキストの取得方法

GitHub Copilotは**エディタが自動的にコンテキストを提供**します。開いているファイル、カーソル位置、ワークスペースのファイルツリー、ターミナルの出力——これらが暗黙的にAgentに渡されます。開発者は何も意識する必要がありません。

Claudeデスクトップアプリ（Chatモード）では**チャットに貼り付けたコード、またはアップロードしたファイル**がコンテキストになります。プロジェクト機能のナレッジに事前登録しておくことも可能です。ワークスペース全体を自動認識する仕組みはないため、レビュー対象は明示的に渡す必要があります。

ただし、Claude Code on Desktopを有効にすれば、ローカルのプロジェクトフォルダを丸ごと読み込んでレビューすることも可能です。この場合、GitHub Copilotに近い使い勝手になります。

### スキルの呼び出し方法

GitHub Copilotでは `/bug-hunter` のようにスラッシュコマンドで明示的にエージェントを指定できます。

Claudeでもスラッシュコマンドによるスキル呼び出しが可能です。Claude Code（CLI）ではスキルとスラッシュコマンドが統合されており、`.claude/skills/code-review/SKILL.md` を配置すれば `/code-review` で呼び出せます。デスクトップアプリでもClaude Code連携をONにしている場合は同様にスラッシュコマンドが使えます。

Chatモード単体（Claude Code連携OFF）の場合は、自然言語で依頼するとdescriptionベースの自動選択で適切なスキルが呼び出されます。

### プロジェクト指示書の対応関係

GitHub Copilotには `copilot-instructions.md`、Claude Codeには `CLAUDE.md` があります。どちらも「プロジェクト全体の前提知識・ルール」を記述するファイルで、AIがタスクに取りかかる前に必ず読み込みます。

両者の役割はほぼ同じですが、管理場所が異なります。`copilot-instructions.md` は `.github/` 配下、`CLAUDE.md` はプロジェクトルート直下に配置します。同じリポジトリに両方置いておけば、それぞれのツールが自分用のファイルだけを読み込みます。

### スキルの管理方法

GitHub Copilotではスキルをリポジトリの `.github/skills/` に配置し、Gitでバージョン管理します。チームメンバーがリポジトリをcloneすれば、全員が同じスキルセットを使えます。

Claudeデスクトップアプリでは、カスタムスキルをZIPでアップロードするか、会話内で作成します。個人のアカウントに紐づくため、チーム共有にはTeam/Enterpriseプランの「Provision skills」を使うか、各メンバーが同じZIPをアップロードする必要があります。Claude Code連携をONにしている場合は、リポジトリの `.claude/skills/` からも読み込まれるため、Gitベースの共有も可能です。

### スクリプト実行

GitHub CopilotはVS Codeの統合ターミナルを通じてスクリプトを実行します。SKILL.mdの `scripts/` に置いた分析スクリプトを直接呼び出せるのは大きな強みです。

Claudeデスクトップアプリ（Chatモード単体）では、スクリプトファイルの直接実行はできません。ただしClaude Code連携をONにすれば、ローカルのbashコマンド実行が可能になり、GitHub Copilotと同等のスクリプト実行能力が得られます。

### 出力先と後続アクション

GitHub Copilotのレビュー結果はエディタ内のチャットパネルに表示されます。コード差分とのインライン表示、PRコメントへの変換など、**開発ワークフローへの統合**が強みです。

Claudeデスクトップアプリのレビュー結果はチャット画面に表示されます。ファイルとしてダウンロードも可能ですが、エディタやGitとの直接的な連携はありません。Claude Code連携をONにしている場合は、レビュー結果を踏まえてコードの修正やコミットまで自律的に実行することも可能です。

### 比較まとめ

| 観点 | GitHub Copilot | Claudeデスクトップアプリ |
| --- | --- | --- |
| SKILL.md互換性 | ✅ | ✅（中核ファイルはそのまま） |
| コンテキスト取得 | エディタが自動提供 | チャットに貼付 or ファイルアップロード |
| スラッシュコマンド | `/bug-hunter` でエージェント呼び出し | Claude Code連携ONなら `/code-review` で呼び出し可能 |
| 自動スキル選択 | descriptionベース | descriptionベース（同じ） |
| プロジェクト指示書 | `copilot-instructions.md` | `CLAUDE.md`（Code連携時）/ カスタム指示（Chat時） |
| スキル管理 | Gitリポジトリで共有 | ZIPアップロード / Code連携時は `.claude/skills/` |
| スクリプト実行 | ✅（統合ターミナル） | △（Claude Code連携ON時のみ） |
| エディタ統合 | ✅（最大の強み） | ❌ |
| エージェント定義 | `.agent.md` で明示 | 不要（descriptionで自動選択） |
| チーム共有 | Git pushで全員に適用 | 各自アップロード / Code連携時はGitで共有可 |
| 非エンジニアの利用 | △ | ✅ |
| MCP連携 | ✅ | ✅ |

## 結論：どう使い分けるか

**SKILL.mdの中核ファイル（SKILL.md + references + assets）は共通資産として1つ作れば、どちらでも使える。** これが最も重要なポイントです。

その上で、実行環境は用途で選びます。

**「エディタの中でサクッとレビューしたい」→ GitHub Copilot**  
開いているファイルのコンテキストが自動で渡り、スラッシュコマンドで即呼び出し。日常的な開発フローに組み込むならこちら。スクリプト実行やPR連携もシームレスです。

**「GUIで手軽にレビューしたい / 非エンジニアも使う」→ Claudeデスクトップアプリ**  
コードを貼り付けるだけ。VS Codeの知識が不要なので、PM・デザイナー・QAメンバーなどがコードレビューに参加するケースに向いています。Claude Code連携をONにすれば、スラッシュコマンドでのスキル呼び出し、修正やコミットまでの自律実行も可能です。

**最強の構成はハイブリッド運用です：**

| シーン | ツール | 理由 |
| --- | --- | --- |
| 日常のコードレビュー | GitHub Copilot | エディタ統合の速さ |
| 深いレビュー + 自律修正 | Claude（Code連携ON） | 一気通貫の実行力 |
| 非エンジニアのレビュー参加 | Claudeデスクトップアプリ | セットアップ不要の手軽さ |
| スキルファイルの管理 | Gitリポジトリに一元管理 | `.github/skills/` と `.claude/skills/` を併置すれば両方から認識される |

## おわりに

前回の記事で検証した「SKILL.mdを育てるほどレビュー精度が上がる」という法則は、GitHub Copilotに限らずClaude環境でも完全に再現できました。

そして今回わかったのは、**SKILL.mdはツールを超えた共通言語である**ということ。一度作り込んだスキルファイルは、GitHub CopilotでもClaudeでもGemini CLIでも使い回せます。エージェント定義やプロジェクト指示書といった「周辺の仕組み」はツールごとに異なりますが、スキルの中核ファイルは変更なしで移植できました。

どのツールで実行するかは用途で選ぶ。でもスキルの育成に投資した時間は、どの環境でも無駄になりません。

あなたのプロジェクトにも、まずは1つSKILL.mdを作ってみてください。
