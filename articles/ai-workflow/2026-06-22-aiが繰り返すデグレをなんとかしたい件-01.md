## AIが繰り返すデグレを、なんとかしたい

AIエージェントで開発していて、特定の機能だけ手を入れるたびにデグレする。そんな経験はありませんか。

私はありました。ある在庫まわりのロジックが、リファクタや機能追加のたびに同じところで壊れる。直して、しばらくして別のタスクを頼むと、また同じ場所が壊れる。その繰り返しでした。

最初は自分の指示が雑なせいだと思って、いろいろ試しました。

* ドキュメント化：仕様を書いてコンテキストに渡す。多少マシになるが、結局また壊れる。
* リファクタ：意図が伝わりやすいコードに整える。それでも壊れる。むしろ「整える」過程で別のところが壊れることもあった。
* コメントを増やす、テストを足す。どれも効くには効くが、決定打にはならなかった。

途中で、問いの立て方が間違っていたことに気づきます。「どう指示すれば壊れないか」ではなく、「そもそも、なぜAIは仕様をドリフト(デグレ)させるのか」のほうでした。

ここを正面から考えたら一つの仮説にたどり着き、それを実験で確かめ、対応する仕組みを実装しました。それが最終的に `intent ledger` になった、という話です。

### 繰り返し壊れていたのは、こういうコードでした

例を一つ。プロダクトには「在庫はマイナスにしない(バックオーダー不可)」という店舗ポリシーがあって、こう書いてありました。

```
def decrement(item_id, qty):
    new = current - qty
    if new < 0:
        raise OutOfStock()   # 店舗ポリシー：バックオーダー不可
    save(new)
```

これが、別タスクで「予約注文に対応して」と頼むたびに、毎回こう「改善」される。

```
def decrement(item_id, qty, allow_backorder=False):
    new = current - qty
    if new < 0 and not allow_backorder:
        raise OutOfStock()
    save(new)
```

AIの判断としては妥当です。予約注文なら在庫がマイナスでもいい、という解釈は自然だし、テストも通る、PRも問題なく見える。だからレビューもすり抜ける。

でも、店舗ポリシーとしてはアウトでした。`if new < 0` は冗長なガードではなく、過去に二重引き当て事故をやらかして入れた、理由のある防御です。そしてコードのどこにも、その理由は書いていなかった。だからドキュメントを足してもリファクタしても、同じ場所が何度でも壊れた。

ここで一番ひっかかったのは、これです。「人間だけで開発していたときは、こんなデグレ、ほとんど起きなかったよな」。なぜか、を順に考えていきます。

## 「SSOTはソースコード」を疑う

普通、開発の Single Source of Truth はソースコードです。仕様書よりコードが正しい。これは正しい。

でも、よく考えると変です。SSOTはコードで、AIもそのコードを読んでいる。なのにAIはドリフトする。同じSSOTを見ているのに、なぜか。

ここで気づきます。ソースコードに書いてあるのは what / how だけだ、ということ。

* `if new < 0: raise` には「何をするか」は書いてある
* でも「なぜそうするのか(過去の事故、店舗ポリシー)」はどこにも書いていない

つまりソースコードは、意図(intent)の非可逆圧縮です。why が落ちている。

では、人間だけで開発しているときに、この「落ちた why」でなぜ困らないのか。答えはシンプルで、コードとは別のところに why が保持されていたからです。

* 書いた本人の記憶
* チームの会話・暗黙知・規約
* 「あの時の障害」を覚えている人の存在

実体としてのSSOTは、コード単独ではなく「コード + 人間が保持している意図」のペアでした。普段それが見えないのは、人間側のチャネルが常に併走しているからです。AIに任せるというのは、この併走者をこっそり外すことだった。だからコードが元々持っていた「why の欠落」が一気に表面化した。

最初は、ここで納得しかけました。「人間は意図を記憶に持っていて、AIは持っていないからだ」と。でも、これは間違いでした。

## 「仕様を知らない新人」でも壊さないのは、なぜか

自分の説明を反証する例がすぐ思い浮かんだからです。

この店舗ポリシーを知らない新人に、さっきのタスクを振ったらどうなるか。情報の上では、新人はAIと同じ立場です。`if new < 0` の理由なんて知らない。

なのに、新人はたいてい壊しません。手順はこうなるからです。

1. `if new < 0` に当たる
2. 「この硬いブロック、なんで在るんだ。消していいのか不安だな」と引っかかる
3. `git blame` を見る / Slackで「在庫マイナス許していいんでしたっけ」と聞く / レビュアに確認する
4. 「Opsに怒られるからダメ」と教わり、正しく実装、もしくはエスカレーション

ドリフトしない。

ここで分かります。差は知識の量ではない。新人もAIも、最初は同じだけ「知らない」。違うのは、知らないと気づいたときの振る舞いのほうです。

* 人間：why が欠けている → 「分からない」と感じる → 止まる → 取りに行く
* AI：why が欠けている → 何も感じずに、もっともらしく埋める → そのまま出す

ドリフトは知識量の関数ではなく、「無知の境界での振る舞い」の関数でした。

## 「違和感で立ち止まる」を分解する

「人間は違和感を感じて立ち止まる」。便利な一文ですが、もう一段ほぐすと、これは4つの独立した部品の連鎖でできています。後で実装の設計図になるので、分けておきます。

### 部品1：違和感の正体は、世界モデルに対する予測誤差

人間が `if new < 0` で引っかかるのは、知識が欠けているからではありません。頭の中にある「在庫=店があって、客がいて、金が動いて、Opsの人が怒られる」という因果つきの世界モデルに照らして、この硬いガードが「自然には生えない。誰かがコストを払って意図して置いた」と浮いて見えるからです。

このモデルと観測のズレ(surprise)が注意に上がったものが、「なんで在るんだ」という感覚の実体です。

### 部品2：違和感が「停止」に配線されている

人間にとって surprise は不快で、解消するまで進むのが気持ち悪い。この情動が「理解せず進む」をブロックします。surprise が行動ゲートに直結している。

### 部品3：意図を「外に取りに行く」構え

人間はコードを「別の人が意図を込めて書いたもの」として読みます。だから `if new < 0` を見ると「書いた人は何を意図したのか」と問い、その答えを「自分の外にある、取りに行くべき事実」として扱う。

### 部品4：取りに行ける環境と、間違えたときの帰結

聞きに行く先(blame / Slack / レビュア)が実在し、かつ「確信して間違えると自分が責められ、自分で後始末する」という帰結がある。だから止まるのが割に合う。

「違和感で立ち止まる」は、この (1) × (2) × (3) × (4) の積でした。

## AIはどこで壊れるのか

ここが一番大事で、しかも私が最初に雑に断定して、後で自分で訂正した部分です。

最初こう書きそうになりました。「AIには違和感が無いからドリフトする」。でもこれは言いすぎで、正確ではありません。順に見ます。

### 部品1は、AIも部分的に持っている

LLM にも世界モデルはあるし、`if new < 0` が「やや珍しい」ことは内部表現の上では分かる。実際、モデルの内部状態が真偽や不確実性をある程度エンコードしている、という研究の流れもあります。「違和感が無い」は誤りです。

### 壊れているのは部品2(停止への配線)

人間は「分からない」信号がコンテンツとは別チャネルで表現され、行動を止める。ところが標準的な自己回帰LMでは、不確実性は次トークン分布のエントロピーの高さとしてしか表現されません。

そして問題は、エントロピーが高くても、デコードは結局1トークンを選んで出力してしまうことです。迷っていても、確信ありげな1トークンに潰される。不確実性は表現できるのに、それが「止まる」という別行動に自動接続されていない。これが「配線されていない」の意味です。

### 壊れている理由(機構)

なぜそうなるのか。3層あります。

1. 学習目的が局所的なもっともらしさ。次トークン予測は「真の意図に整合する続き」ではなく「妥当な著者なら書きそうな続き」を最適化する。真の意図が文脈から復元不能なとき、目的を忠実に達成した結果が創作(confabulation)になる。故障ではなく、目的関数の正常動作です。
2. 不確実性が行動に橋渡しされていない(上述)。
3. 選好学習(RLHF)が abstention を抑制する。学習データのアシスタントは基本「答える」側で、「分からない / 確認させて」は稀で評価者にも好まれにくい。結果、過信や sycophancy が誘導され、内部に潜在する不確実性が出力段で抑え込まれる。RLHF後に「言語化された自信の較正」が悪化する、というのは比較的よく観測されています。

### どの程度そうなのか(較正)

「絶対に止まらない」も言いすぎです。実際、モデルは時に不確実性を述べるし、拒否するし、明確化質問もするし、エージェント設定なら `git blame` をツールで読みにいくこともある。正確な姿はこうです。

| 観点 | 実態 |
| --- | --- |
| 内部に不確実性の信号はあるか | しばしば有る(「自分が知らないことを部分的に知っている」) |
| それが出力に反映されるか | 弱く、較正が悪い。内部では迷っているのに確信ありげに出す |
| 「止まる / 取りに行く」に変わるか | 環境依存で不安定。素の補完ではほぼ起きない。足場があれば起きうるが、信頼できる既定ではない |
| 静かに欠けた意図で発火するか | ほぼしない。手がかりが無いので、そもそもトリガーされない |

まとめると、AIに停止が「無い」のではなく、停止の傾向が弱く・較正が悪く・手がかり依存で、しかも本当に効いてほしい「静かに欠けた意図」のケースで最も弱い、ということです。

これが、私の体験した `if new < 0` のドリフトの正体でした。あれはAIにとって、手がかりの無い、静かに欠けた意図の典型だったわけです。

なお、この「止まらなさ」を当初は AI の構造的な能力欠如だと思っていました。でも後で実際に実験したら、プロンプト一文でひっくり返ることが分かります。これは後半で戻ってきます。

## 人間の停止も、万能ではない

ここまで来て、もう一つ気づきます。人間の「違和感で止まる」も、効く範囲が限られている、ということです。

`if new < 0` のような硬いガードは、ドメインモデルに照らして目立つ(legible)。だから人間の surprise が発火する。

でも、本当に怖いのは、見た目が完全に普通なのに、非自明な why を抱えたコードです。

* 無害に見える2行の順序が、実は race condition の回避
* ある初期値が、規制要件で選ばれている
* innocent に見える default 引数が、後方互換のための値

こういう非可読(illegible)な意図には、人間の surprise すら発火しません。人間レビューでも素通りします。

整理すると、

* 目立つ違和感：人間は止まる / AIは止まらない
* 目立たない意図：人間も止まらない / AIも止まらない(ここが誰も守れていない領域)

この「誰も守れていない領域」を見たときに、作るべきものの形が見えてきました。ただその前に、この見立てが本当に正しいか、自分の実コードで確かめています。

## 実験で確かめた：止まらないのは「能力」ではなく「既定の性向」だった

ここまでは机上の分析です。本当にそうなのか、反証可能な形で実験しました。

### 題材：3世代パッチされて、統一ポリシーが消えた実機能

題材は、Chat 画面の provider / model 選択ロジックです。作り話ではなく、実際にドリフトした現物。コードから仕様を逆抽出したら、こう書かざるを得ませんでした。

> これはコードから逆算して再構成した「実装されている仕様」であり、「設計者が意図した仕様」ではない。両者の差こそ、デグレが起きやすい場所である。

ビジネス要件1つに対してAIが局所パッチを当てる、を3世代繰り返した結果、provider→model の解決が「起動時は先頭、切替時は記憶 or 空」という経路依存になり、それを統一するポリシーがどこにも無い状態になっていました。「同じ場所が手を入れるたびに壊れる」の正体です。

### 実験設計：reconcile か patch か

この状態を生んだ実際のコミットと、それを生んだ実際のプロンプトが残っていました。パッチが当たる前の状態(`.git` ごと外したクリーンコピー)を用意して、2条件で投げ直します。

* X(素)：実プロンプトだけを渡す
* Y(ハーネス)：実プロンプトの前に、一文だけ前置する

Y で前置した一文はこれです。

> この要件が既存の決定済み挙動に触れるなら、横に分岐を足さず、衝突する既存挙動を箇条書きで列挙し、どちらを優先するか確認を求めよ。

これを composer-2.5 と Claude opus / sonnet / haiku で回しました。

なお、最初の実験は汚染していました。エージェントが隔離ディレクトリではなく本体リポジトリを見ていて、そこに実験設計のメモ(答え)が置いてあったんです。方法論のミスです。`.git` も答えのメモも外した完全隔離で再実験しました。以下はそのクリーン版です。

### 結果

| モデル | 条件 | 止まって人間に聞くか | 非局所の衝突に気づくか | 解決の形 |
| --- | --- | --- | --- | --- |
| composer-2.5 | X 素 | ✗ 実装直行 | ◯ | 局所パッチ |
| composer-2.5 | Y ハーネス | ◎ 停止+質問 | ◯ | 統合提案 |
| opus | X 素 | ✗ 実装直行 | ◎ | 単一ヘルパに集約(最良) |
| opus | Y ハーネス | ◎ 停止+質問 | ✗ 見落とし | 統合提案 |
| sonnet | X 素 | ✗ 実装直行 | △ 見たが無視 | 局所(統合を明示拒否) |
| sonnet | Y ハーネス | ◎ 停止+質問 | ✗ 見落とし | 統合提案 |
| haiku | X 素 | ✗ 実装直行 | ◯ | 局所パッチ |
| haiku | Y ハーネス | ◎ 停止+質問 | ◯ | 統合提案 |

ここから、はっきりしたことが3つあります。

1. 「止まって人間に渡す」は、100%プロンプトで決まり、0%モデル強度で決まった。素(X)では opus ですら全モデルが実装に直行。ハーネス(Y)では haiku ですら全モデルが停止して質問した。完全に分離している。「賢いモデルなら素でも一拍置く」は反証された。一拍置く性向は、モデルが賢くなっても自然には湧かない。
2. 「衝突に気づく」能力は、強度ともハーネスとも相関しなかった。最弱の haiku が両条件で気づき、最強の opus が Y で見落とした。発見は探索の運で、当てにできない。
3. 統合の質だけ、弱く強度に相関した(opus-X だけが単一ヘルパに集約)。ただし、機能境界をまたいで本当に reconcile した者は一人もいなかった。

### この結果で、見立てが変わった

私は「AIは構造的に止まれない」と思っていました。でも実験は「一文で全モデルが止まる」と言っている。つまり止まれないのは能力の欠如ではなく、既定の性向(訓練の穴)でした。

なぜ訓練されていないか、も腑に落ちます。SWE-bench のようなベンチマークは「テストが通るか / issue が解けるか」という局所的な正しさしか報酬にしません。大域的な整合や潜在衝突の表面化は測らない。測られないものは最適化されない。RLHF も「そのターンで答える応答性」を報酬にし、stop & ask は「遅い・差し出がましい」と減点されがち。だから、確信して局所パッチを打つ性向が焼き込まれます。

ここから、製品としての賭けどころが変わりました。

* 「止まらせる」こと自体は安く、harness 側で外付けできる。しかも、いずれモデルに焼き込まれて陳腐化する。一時的な橋でしかない。
* 持続的に効くのは、「何と整合させるべきか」=過去に裁定してロックした意図の蓄積。これはプロジェクト固有なので、どんなに賢いモデルでも重みの外にある。

採点軸も間違っていました。「reconcile したか」で測っていたけれど、早すぎる統一は誤った抽象を生むし、別挙動であるべきものを勝手に統一するのは害です。正しい成果物は reconcile そのものではなく、「patch か reconcile かを、人間が裁定する明示的な決定点に変えること」。統治された patch は成功で、沈黙の reconcile はむしろ失敗(誰も決めずに統一してしまった)です。

この実験で、作るべきものの形が決まりました。

## だから作った：intent ledger

実験を踏まえた設計の割り切りは2つです。

1. AIを「止める」のは安く、harness 側で外付けできる。だがそこは橋であって、差別化にはならない。
2. 持続的な強みは「何と整合させるか」=裁定してロックした意図の蓄積。ここに重心を置く。

部品2(停止への配線)はモデルの内側に作り込まなくていい。実験どおり、harness 側の一文で全モデルが止まる。だから「止める」は機構として外付けすればいい。

問題は、止めたあとに何と照合するかです。AIはセッションごとに衝突解決を別々に決め、人間が一度も裁定もロックもしないから、経路依存が堆積する。これが3世代パッチの正体でした。だから、一度人間が裁定した結果を台帳にロックし、次回は再発見に頼らず確定的に捕まえる、という形にします。

intent ledger は、コードに書けなかった why と、過去に下した裁定を、コードの外に永続化しておき、危ない変更のときに機械的に決定点を発火させる台帳です。先に分解した4部品のうち、外付けできる (2)(4) を機構にし、いちばん残る価値である「照合先の蓄積」を中心に据える、という設計です。

具体的には、3つのことをやります。

### 1. 違和感を「環境の側」に外部化する(部品1+3)

AIの内的メタ認知を当てにせず、保護したい箇所に事前に印をつける。`if new < 0` のような箇所に「これは保護された意図だ」というエントリを紐づけておきます。

エントリのイメージ(フォーマットは抜粋)：

```
- id: stock-no-backorder
  anchor: "src/inventory/stock.py:decrement"
  intent: "在庫はマイナスにしない（バックオーダー不可）"
  why: "2025-xx の二重引き当て障害の再発防止。店舗運用ポリシー Ops#1234。"
  guards:
    - "if new < 0"
  stop_on_change: true
```

ポイントは、意図を「コードを読んで察してくれ」ではなく、外部の事実として明示していることです。AIが察する必要がない。

### 2. 「印に触れたら止まる」を環境のルールにする(部品2)

静かに欠けた意図は、入力に手がかりを出してくれません。だから「手がかりを見て止まる」のではなく、印のついた領域に diff が触れたら、harness が機械的に stop-condition を発火させる。

エージェントが `decrement` の `if new < 0` を書き換えようとした瞬間、ledger が「保護された意図に触れている」と検知して、変更をいったん止める。AIの気分に依存しません。

### 3. operator に経路づける(部品4)

止めたあと、人間(operator)に「この変更は既存の決定済み挙動に抵触します。patch で横に足すか、統合するか、どちらにしますか」と裁定を仰ぐ。新人が Slack で聞いていたことを、システムが代わりにやります。Maker-Checker の Checker をここに差し込むイメージです。

大事なのは、reconcile を強制しないことです。実験で痛感したとおり、早すぎる統一は誤った抽象を生むし、別挙動であるべきものを勝手にまとめるのは害になる。だからここでやるのは「統合させる」ことではなく、「patch か reconcile かを、人間が必ず一度決める」こと。そして、その裁定を台帳に残します。

### 核心：裁定を蓄積し、二度と気づき直させない

ここが intent ledger のいちばん残る価値です。

実験で分かったのは、「衝突に気づく」発見はモデル強度にもハーネスにも相関しない、つまり毎回コイン投げだということ。だから「モデルに毎回気づかせる」設計は、原理的に当てになりません。

正解は逆向きで、「一度裁定した衝突を恒久ロックし、次回は再発見に頼らず確定的に捕まえる」です。一度 operator が「これは統一しない、別挙動が正しい」と決めたら、それを台帳に刻む。次にAIがそこへ触れたら、コイン投げの発見に頼らず、台帳が確定的に止める。

これは副産物として、目立たない意図(非可読な intent)も守れます。2行の順序が race 回避だ、という見た目に現れない意図も、一度裁定して台帳に刻めば、AIがそこを「整理」しようとした瞬間に止まる。人間のメタ認知でも守れなかった領域を、ここで初めてカバーできます。

なぜ台帳が持続的な強みになるのか。止める性向(stop & ask)はいずれモデルに焼き込まれて陳腐化します。でも「このプロジェクトで、何と整合させるべきか」という蓄積は、プロジェクト固有なので、どんなに賢いモデルでも重みの外にある。完璧に訓練された steward モデルが来ても、照合先の蓄積だけは外から供給するしかない。だからここに賭けます。

## 実装してみて分かったこと

実際に動かしてみると、効くポイントと、設計上の注意がはっきりしました。

効いたところ：

* 「静かに欠けた意図」のドリフトが、マージ前に止まるようになった。`if new < 0` のケースは、まさに ledger が止めてくれる類のものでした。
* 「なぜこの変更を止めたか」が why つきで提示されるので、operator 側の判断が速い。「あ、それ消しちゃダメなやつだ」が即わかる。

ハマったところと、設計上の割り切り：

* 全部を foresight で書こうとしない。「事前に完璧な意図を全部書け」は、人間だって普段やっていない。続かないし、無理がある。ledger は事後(hindsight)に育てるものとして設計するのが正解でした。裁定が発生したら、そのとき初めてエントリを刻む。
* だから ledger は最初スカスカで当然。変更の影響範囲(blast zone)の外で意図を守ることに割り切り、operator が結果を見て事後に反応する前提にする。
* ロックの粒度を間違えると逆効果。これは実験で負の例が出ました。最初のパッチ世代は「パッチ後の挙動」をそのままテストで固定してしまい、後から統一しようとするのを逆に阻害していた。台帳に刻むのは「実装の形」ではなく「裁定そのもの(なぜその挙動を選んだか)」。形でロックすると、正しい統一まで縛ってしまう。
* 印が多すぎると、何でもかんでも止まって邪魔になる。「決定済み挙動に触れる / 影響範囲が大きい」ものだけに絞るのが、運用上のキモでした。

## まとめ

最後に要点だけ整理します。

1. AIに任せたら、コードに書いていない why が落ちて、同じ機能が繰り返しデグレした。ドキュメント化もリファクタも決定打にならなかった。
2. ソースコードは意図の非可逆圧縮。普段ドリフトしないのは、人間が落ちた why を併走で補完していたから。差は知識量ではなく、無知の境界での振る舞い(止まって取りに行くか、創作で埋めるか)だった。
3. その「止まる」は (1)予測誤差 ×(2)停止への配線 ×(3)外に取りに行く構え ×(4)環境、の積。AIは既定では止まらず創作で埋める。しかも人間の停止も、目立つ異常にしか効かない。
4. 実験でひっくり返った点：止まらないのは能力の欠如ではなく既定の性向だった。一文で全モデルが止まる(100%プロンプト依存・0%モデル強度)。一方で「衝突に気づく」発見は強度にもハーネスにも相関せず、コイン投げ。
5. だから賭けどころは「止めること」ではない(それは安く、いずれ陳腐化する)。持続的な強みは「何と整合させるか」=裁定してロックした、プロジェクト固有の意図の蓄積、つまり intent ledger。
6. ledger の仕事は reconcile の強制ではなく、patch か reconcile かを人間が必ず一度裁定する決定点を作り、その裁定をロックして二度と気づき直させないこと。副産物として、人間のメタ認知でも守れない目立たない意図も守れる。

AIに開発を任せる流れは、もう戻らないと思います。だとすれば、人間が暗黙にやっていた「違和感で立ち止まって裁定する」を、どう仕組みとして外付けし、その裁定を蓄積するか。そこが勝負になります。intent ledger は、その一つの答えとして作ってみた、という話でした。
