こんにちは、かわそんです。  
toC 向けプラットフォームの開発・運営をしています。

今回、これまで人手でやっていたコンテンツ審査を、非同期パイプラインに切り出してAIに移管しました。  
結果、1日100件超を数時間かけて捌いていた業務が、1件あたり数十秒〜数分で処理できるようになっています。

**実装・検証に要した期間は実質2営業日**。

平均値で**6倍のスループット改善**を実現しました。  
(人間の稼働との比較なので、パーセンタイルではなく平均を採用)

本稿では、このシステムのアーキテクチャと技術選定の背景、および品質を継続的に担保するための設計方針についてまとめます。

## 背景と課題

運営サービスにおいては、ガイドラインを守っている限り比較的自由にコンテンツを投稿できますが、その自由度は品質リスクと表裏一体です。  
ユーザーベースが拡大するにつれ、スパム・低品質コンテンツの流入は構造的に増加します。

（余談ですが、6年前のリリース当初は審査機構自体が存在しませんでした。）

その対策として審査フローを整備してきたものの、人手によるオペレーションはスケールアウトに向きません。  
処理量に対して人件費と待ち時間が線形に増加する構造は、プロダクトの成長に対してそのままボトルネックになります。

この「**線形スケールしか取れない審査**」という制約を解消するため、審査ロジックそのものを offload する設計に全面移行しました。

## ソリューション

審査処理はアプリケーションの同期フローから完全に分離し、Pub/Sub ベースの非同期パイプラインとして構成しています。

判定エンジンには Claude API を使用していますが、肝はLLMの選定そのものではありません。

「**重い審査処理を非同期かつ安全に流せるインフラを先に設計すること**」

が前提であり、LLMはあくまでそのパイプライン上の worker の一つです。  
本質は「審査システム全体の再設計」にあります。

### ECS → Lambda（非同期連携）

ECSからLambdaへの処理委譲は direct call を避け、SNS + SQS を介した loose coupling な非同期連携にしています。

目的は spike 耐性の確保と、将来的な fan-out（複数審査ロジックの並列適用）への対応余地の確保です。

審査は**リアルタイム性を要求しない**ユースケースであり、わずかなレイテンシよりも系全体の安定性を優先するトレードオフを取っています。

SQS は DLQ と合わせてモニタリングしており、キュー滞留や処理失敗は即時検知・復旧できる状態を維持しています。

### Lambda の処理内容

Lambda は LLM に渡す前の preprocessing を一括して担当させています。

投稿コンテンツは数千〜数万文字規模に加えて画像・動画を含むケースがあり、そのまま payload として渡すとトークン制限やコストの観点で即座に破綻します。

そのため、以下の処理を分割して実行しています。

```
- DB 参照のみで完結する事前判定ロジックの優先実行
- HTML stripping による pure text 抽出
- 画像・動画コンテンツの分離
- chunking による分割リクエスト
- 複数チャンク結果の merge・集約
```

また、特定のケースについては意図的に人間審査への fallback を設けています。  
これは精度上の限界への対処というより、責任分界とリスクコントロールを目的とした設計判断です。

#### Lambda 選定の根拠

処理時間とワークロード特性を精査した上での採用です。  
実測では最大サイズのコンテンツでも数分以内に収まっており、**Lambda の 15 分制限に対して十分なマージンがあります**。

また、審査処理はリアルタイム性を要求しないため、cold start のオーバーヘッドも許容範囲内です。

常時起動コンテナを維持するより、イベント駆動で必要な分だけ実行するほうがコスト・運用の両面で合理的という判断です。

なお、冪等性の担保については専用の DynamoDB などのidempotency store を別途設けることはせず、既存ロジック側の排他制御に委ねています。

審査の業務特性上、重複実行がユーザー体験に直接影響するケースが限定的であり、システムを不必要に複雑化しない選択をしています。

### Lambda → S3（審査基準の外部化）

審査基準はS3上に配置し、Lambda から都度 fetch する構成にしています。

ただし、この審査基準の変更頻度は低いこともあり、LLM へのリクエスト時にはキャッシュを噛ませています。

vector search（OpenAI Vector Store、Pinecone 等）の採用も検討しましたが、今回は見送っています。

対象データが審査ガイドラインと社内規定という限定されたセットであり、semantic search の複雑性を持ち込む必要がないためです。

外部サービスへの依存を増やさず AWS 内で完結させることで、レイテンシと運用コストの双方を抑制しています。

このユースケースではオブジェクトストレージが最適解という結論です。

### Lambda → RDS（コンテンツデータ取得）

Lambda から RDS への接続は、前段に **RDS Proxy を配置**しています。

Lambda は spike 的に並列実行されるため、直接 DB に接続するとコネクション枯渇が発生します。  
connection pool を共有できる RDS Proxy を介在させることでこのリスクを回避しています。

既存構成でも RDS Proxy の運用実績があり、新規の接続戦略を持ち込まず実績ある構成に寄せる判断をしています。

### SQS → ECS（審査結果の反映）

Lambda + LLM による審査結果は SQS に送信し、ECS の sidekick job で long polling して取得しています。  
審査結果の確定とそれに伴うビジネスロジックの実行はこの consumer 側で行います。

この分離により、「AI による判定」と「サービス側の状態更新」が loose coupling になります。  
審査処理が詰まっても、アプリケーションコアへの影響が遮断される構造です。

なお、RDS Proxy の接続切断タイミングで取得処理が失敗するケースがあるため、接続状態を検知して retry・再接続するハンドリングを実装しています。

（DLQ が積み上がってこの考慮漏れに気づいた、というのが正直なところです。）

### 補足：アーキテクチャを崩す実装について

我々の組織は人材の流動性が比較的高く、それが技術的な制約になるケースがあります。

「Claude で審査を自動化して」という粒度の指示を出すと、局所的には動くが全体最適を崩した実装が出てくることがあります。  
LLM に素朴なプロンプトで実装を生成させると、ECS 上で処理を完結させる monolithic な構成が提案されることが多いです。

```
Claude を使って、既存の審査フローを自動化して。審査基準はこれです
```

この構成は spike 耐性もスケーラビリティも持たず、重い処理を載せた瞬間にサービス全体に波及します。

**非同期で分離すべき処理をメインプロセスに混在させた時点**で、ユーザー体験の毀損は時間の問題です。

「**実装の前に設計レイヤーで担保する**」という判断軸を持てるかどうかが、このような局面では決定的な差になります。

## 品質安定の仕組み

人間の審査判断を LLM で完全に再現することは、現実的な目標ではありません。

同一入力に対して常に同一出力を保証できない点は LLM の特性として受容する必要があります。  
これは人間の判断も同様で、コンテキストや状態によってブレが生じます。

**「完全な一貫性」ではなく「許容可能なブレ幅に収める」設計**が現実的なゴールです。

そのため、明文化された審査ガイドラインと、担当者の暗黙知としての判断基準の両方をシステムに取り込み、hybrid に精度を引き上げる方針を取っています。

### 品質改善ループ

品質の継続改善は、人間の審査結果を起点にした feedback loop で実現しています。

審査担当者には、下記2つの作業をしてもらっています：

1. 定期的な承認コンテンツの確認
2. 人間判断を要するコンテンツの審査

```
確認日時　2026/03/20 13:00
https://example.com/u/hoge/a/sample - 外部誘導が目的
https://example.com/u/fuga/a/sample2 - 有料部分が貧弱
```

ローカルの /tune-for-review skill でフィードバックループを回します。

/tune-for-review(抜粋)

各コンテンツについて、**ユーザーが提供した否認理由を起点に**以下を分析する：

1. **否認理由の深掘り**: ユーザーの否認理由（例：「外部誘導がメイン」「有料部分が2行しかない」）を踏まえ、コンテンツ本文のどの部分がその理由に該当するかを具体的に特定する。否認理由は人間の審査者の直感的な判断なので、それを criteria に落とし込める具体的な特徴に分解する
2. **承認された理由**: ユーザーの否認理由の観点で、現行 criteria のどの条文・方針が原因で承認されてしまったのか。具体的に criteria のどの部分が甘いのか、またはどの観点が欠落しているのかを特定する
3. **パターン分類**: 複数コンテンツの否認理由に共通するパターンがあれば分類する（例：「外部誘導系」「コンテンツ量不足系」「誇張表現系」など）

分析結果をユーザーに提示する。否認理由ごとに整理し、「この否認理由を criteria でカバーするにはどうすべきか」を明確にする。

分析結果をもとに、criteria\_1.txt と criteria\_2.txt それぞれの修正プランを作成する。

修正プランには以下を含める：

* **追加する基準**: 新たに否認対象とすべきケースの具体的な条文案
* **修正する基準**: 既存の「許容」ルールで緩すぎる部分の改善案
* **変更の根拠**: なぜこの変更が必要かの説明（どのコンテンツのどの問題に対応するか）

複数の事例から共通パターンを抽出し、基準の追加・修正案を作成し、それを実審査で検証することで、基準とモデルの双方を段階的に改善していきます。

そして、人間と判断がずれたコンテンツ数 / 人間チェックのスパン をベンチマークとし、この数が収束してきたことを持って、人間とAIの基準が近づいた、と判断しています。

## まとめ

本取り組みは単なる AI 導入ではなく、審査業務全体を非同期パイプラインとして再設計した上でLLMを組み込んだものです。

AI は一要素に過ぎず、本質は「水平スケール可能な審査インフラの構築」と「継続改善サイクルを回せる基準設計」にあります。

今後も、悪用リスクと品質担保のトレードオフに対して、AIと人間の責任範囲を適切に分離しながら、プラットフォームの健全性を維持していく方針です。
