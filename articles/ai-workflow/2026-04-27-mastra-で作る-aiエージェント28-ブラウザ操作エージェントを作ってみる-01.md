[Mastra で作る AI エージェント](https://zenn.dev/shiromizuj/articles/a0a1659e9f05b6) というシリーズの第28回です。

---

2026年4月24日、Mastra は **Browser Support** を正式発表しました。エージェントにブラウザを持たせ、Web ページのナビゲーション・フォーム入力・ボタンクリック・データ抽出を自然言語で指示できるようになる機能です。

公式デモ動画では、Mastra Studio のチャット画面にブラウザのライブ映像が埋め込まれ、エージェントがページを操作する様子がリアルタイムで表示されています。「これは面白い」と思って Mastra v1.27 の手元のプロジェクトに組み込んでみたのですが、そのライブビューが映るようになるまでにはいくつかの壁がありました。この記事では、Stagehand ベースのブラウザエージェントを構築する過程で踏んだ落とし穴と、最終的な構成を記録します。

---

## ブラウザが稼働する場所にご注意

まず、Mastraでブラウザが使えるようになる、ということのアーキテクチャをしっかり確認しておきましょう。以下の絵のように、**Mastraサーバーの中でエージェントがブラウザ操作**をします。その様子を、エンドユーザは**ライブビューで眺める**ことができます。ときどきユーザが介入して、例えばボタンを押すこともできますが、**あくまでブラウザの実体はMastraサーバが稼働しているマシンに存在**します。ユーザの端末ではありません。

![](https://static.zenn.studio/user-upload/b9e2d4c06577-20260426.png)

ローカル開発（npm run dev）ではMastraサーバが稼働するマシン＝自分の PC なので問題になりにくいですが、Mastra Platform やクラウドにデプロイした場合は、ブラウザはクラウドサーバー上で動くことになり、外部サイトから見える IP はクラウドのグローバル IPであり、場合によっては接続先システム等の IP ホワイトリストにクラウドの IP を追加する必要が出てきますし、LAN 経由でしかアクセスできない社内サイトには到達できなかったりします。

## ブラウザプロバイダの選定

Mastra のブラウザ機能には 3 つのプロバイダがあります。

| プロバイダ | パッケージ | 特徴 |
| --- | --- | --- |
| AgentBrowser | `@mastra/agent-browser` | Playwright ベース。アクセシビリティツリーで要素を特定する決定論的アプローチ |
| Stagehand | `@mastra/stagehand` | Browserbase 製。AI で要素を認識し、自然言語で操作を指示できる |
| BrowserViewer | `@mastra/browser-viewer` | CLI プロバイダ。agent-browser 等の CLI ツールを CDP 経由で接続する |

今回メインで使うのは **Stagehand** です。「ログインボタンをクリックして」「メールアドレス欄に入力して」といった自然言語の指示がそのまま通るのが魅力で、セレクタやアクセシビリティ ref を意識する必要がありません。比較検証用に BrowserViewer と AgentBrowser のエージェントも並行して作成しました。

## Stagehand

### 最初のコード — そしてすぐエラー

最初に書いたコードはシンプルでした。`StagehandBrowser` のインスタンスを作り、`Workspace.browser` に渡す構成です。

```
const browser = new StagehandBrowser({
  headless: true,
  model: 'openai/gpt-5.4-mini',
});

const workspace = new Workspace({
  filesystem: new LocalFilesystem({ basePath: './agent-workspace' }),
  sandbox: new LocalSandbox({ workingDirectory: '.' }),
  browser,  // ← ここに渡した
});

export const browserStagehandAgent = new Agent({
  // ...
  workspace,
});
```

起動すると **`WorkspaceError INVALID_CONFIG`** が返ってきます。

原因は、ブラウザプロバイダの「種類」によって渡す先が異なるという仕様でした。Mastra のブラウザプロバイダには SDK プロバイダと CLI プロバイダの 2 系統があります。

* **SDK プロバイダ**（AgentBrowser, Stagehand）→ `Agent.browser` プロパティに渡す
* **CLI プロバイダ**（BrowserViewer）→ `Workspace.browser` に渡す

Stagehand は SDK プロバイダなので、`Workspace` ではなく `Agent` に直接持たせる必要があります。修正後のコードはこうなります。

```
export const browserStagehandAgent = new Agent({
  id: 'browser-stagehand-agent',
  model: 'openai/gpt-5.4',
  browser,       // ← Agent に直接渡す
  workspace,     // ← workspace には browser を含めない
  // ...
});
```

一方、BrowserViewer は CLI プロバイダなので従来通り `Workspace.browser` に渡します。この使い分けはドキュメントを読まないと気付きにくいポイントでした。

---

### 「ブラウザは使えません」と言われる

構成エラーは解消できましたが、次の壁は LLM 側にありました。Studio でエージェントに「Zenn のトップページを開いて」と指示すると、「申し訳ありませんが、ブラウザを操作する機能を持っていません」と返ってきます。ブラウザツール自体は登録されているのに、エージェントが使おうとしないのです。

原因はエージェントの `instructions` にありました。初期のインストラクションにはブラウザツールの存在を含めていなかったため、LLM は自分がブラウザを操作できることを「知らない」状態でした。

対処として、`instructions` に利用可能なツール一覧（`stagehand_navigate`, `stagehand_act`, `stagehand_extract` など）を明示し、Studio ライブビューの説明やユーザー介入の案内方法まで含めるようにしました。これでエージェントはブラウザツールを積極的に使うようになります。

ツールの説明をいちいちエージェントのインストラクションに書かなければならない、というのは本来的にはおかしい話なので、これはむしろ Mastra 側で近いうちに改善されることでしょう。

---

### headless: false で別ウィンドウが立ち上がる

ブラウザ操作は動くようになりました。しかし Studio 上にライブビューが表示されません。「`headless: false` にすれば見えるのでは？」と思って試すと、デスクトップに Chrome ウィンドウが別途立ち上がるだけで、Studio の画面には何も埋め込まれませんでした。

![](https://static.zenn.studio/user-upload/ec8f2c5376aa-20260426.png)  
*headless: false にすると別ウィンドウが立ち上がる*

`headless` オプションは「OS 上にブラウザウィンドウを表示するかどうか」を制御するもので、Studio のライブビューとは別の仕組みです。ライブビューは CDP（Chrome DevTools Protocol）のスクリーンキャスト API を使い、ブラウザのフレームを WebSocket 経由で Studio のフロントエンドに配信しています。`headless: true` でもスクリーンキャストは動作します。

> 「別に、Studioとは別のウィンドウでもいいじゃないか。エージェントがブラウザを操作して、それをエンドユーザが確認できているのだから問題ない」と考える方もいるかもしれません。しかし、今回 npm run dev で開発サーバを起動しているから問題ないように思えるだけで、これをそのままクラウド上の本番環境にリリースするとどうでしょう。いま、Studioとは別のウィンドウで起動したブラウザ。これは「エンドユーザのマシンのブラウザ」が起動した、ということではなく、「Mastra サーバーの存在するマシンのブラウザ」が起動した、ということです。いまは「エンドユーザのマシン＝Mastraサーバーの存在するマシン」なので両者は同じことですが、クラウド上の本番環境にリリースした場合ですとサーバー側で OS の GUI ウィンドウを開こうとし、場合によっては起動に失敗します。エンドユーザーの PC にウィンドウが飛んでくることは 絶対にないでしょう。

---

### WebSocket パッケージの不在

ここからが最も時間のかかった調査でした。ブラウザ操作は問題なく動くのに、Studio にライブビューのパネルがまったく表示されません。

サーバー側のコードを調べると、Studio UI の `main.js` にはスクリーンキャスト表示用のコンポーネントがちゃんと含まれています。WebSocket で `/browser/:agentId/stream?threadId=...` に接続し、フレームデータを `<img>` タグに流し込む仕組みです。フロントエンドは準備できています。ではなぜ動かないのか。

`@mastra/server` パッケージのコードを掘っていくと、サーバー側のブラウザストリーミング機能（`ViewerRegistry`）が CDP の `Page.screencastFrame` イベントをキャプチャし、WebSocket で配信する構造になっていることがわかりました。そして、この WebSocket サーバーの実装に **`ws` と `@hono/node-ws`** という 2 つのパッケージが必要だったのです。

公式ドキュメントの [Browser Overview](https://mastra.ai/docs/browser/overview) を確認すると、こう書かれています。

> Screencast requires WebSocket support. Install these packages in your project:  
> `npm install ws @hono/node-ws`
>
> These packages are not included by default because they are incompatible with serverless environments like Cloudflare Workers. If they are not installed, screencast is disabled but all other browser functionality works normally.

**サーバーレス環境との互換性のために、WebSocket パッケージはデフォルトで含まれていない。** 未インストールの場合、スクリーンキャストだけが静かに無効化され、他のブラウザ機能には影響しません。エラーメッセージも出ない。これが「ブラウザは動くのにライブビューだけ映らない」という現象の正体でした。

```
pnpm add ws @hono/node-ws
```

これだけで解決します。しかしエラーが出ないぶん、原因の特定に時間がかかりました。

### screencast オプションの設定

WebSocket パッケージをインストールしたら、ブラウザプロバイダにスクリーンキャストの設定を追加します。

```
const browser = new StagehandBrowser({
  headless: true,
  model: 'openai/gpt-5.4-mini',
  screencast: {
    format: 'jpeg',
    quality: 80,
    maxWidth: 1280,
    maxHeight: 720,
  },
});
```

`screencast` オブジェクトを渡すだけでスクリーンキャストが有効になります。`enabled: true` のようなフラグは存在しません。実際、最初 `enabled` プロパティを含めて書いたところ TypeScript の型エラーになりました。`ScreencastOptions` 型（`@mastra/core/browser/screencast/types.d.ts`）が受け付けるのは `format`, `quality`, `maxWidth`, `maxHeight`, `everyNthFrame`, `threadId` の 6 フィールドのみです。オブジェクトの存在自体がスクリーンキャスト有効化のトリガーになっています。

![](https://static.zenn.studio/user-upload/c6341ddeb81e-20260426.png)  
*Studio でライブビューが表示される*

### セキュリティ上の注意点

実際にブラウザを操作しているとき、エージェントが Google のログインページに遷移しようとした際にこんな画面になりました。

「このブラウザまたはアプリは安全でない可能性があります」というエラーです。

![](https://static.zenn.studio/user-upload/ec211c93f918-20260426.png)  
*Googleによる安全でないブラウザの警告*

Playwright の自動化ブラウザは通常のブラウザではないため、Google 等のサービスが不審なアクセスとして弾くことがあります。エージェントに認証を伴う操作をさせる場合は、Browserbase のようなクラウドブラウザサービスへの接続（`env: 'BROWSERBASE'`）や、セッションの引き継ぎなど、別途対策が必要です。上記で触れたように「headless: false」 で別ウィンドウでブラウザ表示する、というのも一つの対策ですね。

### 最終的な構成

紆余曲折を経て、最終的なエージェント構成はこうなりました。

```
import { Agent } from '@mastra/core/agent';
import { Workspace, LocalFilesystem, LocalSandbox } from '@mastra/core/workspace';
import { StagehandBrowser } from '@mastra/stagehand';
import { Memory } from '@mastra/memory';

const browser = new StagehandBrowser({
  headless: true,
  model: 'openai/gpt-5.4-mini',
  screencast: {
    format: 'jpeg',
    quality: 80,
    maxWidth: 1280,
    maxHeight: 720,
  },
});

const workspace = new Workspace({
  filesystem: new LocalFilesystem({ basePath: './agent-workspace' }),
  sandbox: new LocalSandbox({ workingDirectory: '.' }),
  // browser を含めない（SDK プロバイダは Agent に渡す）
});

export const browserStagehandAgent = new Agent({
  id: 'browser-stagehand-agent',
  name: 'Browser Agent (Stagehand)',
  instructions: `
あなたはWeb自動化アシスタントです。stagehand ツールを使ってページを操作してください：
- stagehand_navigate: URL への遷移
- stagehand_act: 自然言語で記述された操作の実行
- stagehand_extract: ページから構造化データを取得
- stagehand_observe: ページ上で実行可能なアクションを一覧

Mastra Studio でこのエージェントを使用すると、ブラウザのライブビューが表示されます。
操作前に確認が必要な場合は、ユーザーに問い合わせてください。
  `.trim(),
  model: 'openai/gpt-5.4',
  browser,      // SDK プロバイダは Agent に直接渡す
  workspace,
  memory: new Memory({
    options: {
      lastMessages: 40,
      observationalMemory: {
        model: 'openai/gpt-5-mini',
        temporalMarkers: true,
        observation: { previousObserverTokens: 10_000 },
      },
    },
  }),
});
```

## AgentBrowser

### Playwright 実行ファイルの不足

Stagehand とは別に、`@mastra/agent-browser` を使った Browser Agent も試しました。ここでぶつかったのが、Playwright の実行ファイルが入っていない問題です。

`@mastra/agent-browser` は内部で `playwright-core` を使いますが、`playwright-core` はブラウザバイナリを同梱しません。そのため環境によっては、次のようなエラーで起動に失敗します。

![](https://static.zenn.studio/user-upload/0a55f3bcdc59-20260426.png)

* `Executable doesn't exist ... chrome-headless-shell.exe`
* 案内: `npx playwright install`

今回は pnpm 環境だったため、`npx playwright install` では CLI が見つからず失敗しました。そこで `.pnpm` 配下の Playwright CLI を直接実行して Chromium をインストールしたところ解決しました。

```
& "node_modules\.pnpm\node_modules\.bin\playwright.ps1" install chromium
```

インストール後に Studio を再起動すると、AgentBrowser 側でもブラウザ起動と操作が正常に動作しました。

![](https://static.zenn.studio/user-upload/021a1e01457d-20260426.png)

### 最終的な構成

AgentBrowserをつかったエージェントの最終的な構成はこちらです。

```
import { Agent } from '@mastra/core/agent';
import { Workspace, LocalFilesystem, LocalSandbox } from '@mastra/core/workspace';
import { AgentBrowser } from '@mastra/agent-browser';
import { Memory } from '@mastra/memory';

const browser = new AgentBrowser({
  headless: true,
  screencast: {
    format: 'jpeg',
    quality: 80,
    maxWidth: 1280,
    maxHeight: 720,
  },
});

const workspace = new Workspace({
  filesystem: new LocalFilesystem({
    basePath: './agent-workspace',
  }),
  sandbox: new LocalSandbox({
    workingDirectory: '.',
  }),
});

export const browserAgentBrowserAgent = new Agent({
  id: 'browser-agent-browser-agent',
  name: 'Browser Agent (AgentBrowser)',
  instructions: `
あなたは AgentBrowser（Playwright ベース）を使ったブラウザ操作に特化したアシスタントです。
アクセシビリティツリーの ref を使って要素を特定し、決定論的にブラウザを制御します。

## Studio ライブビュー
Mastra Studio 上ではブラウザのライブスクリーンキャストが自動的に表示されます。
ユーザーは Studio のブラウザパネルを通じて、あなたの操作をリアルタイムで確認できます。
また、ユーザーはブラウザに直接マウスやキーボードで介入（CAPTCHA 入力など）できます。
ユーザーの介入が必要な場面では、Studio のブラウザパネルで直接操作するよう案内してください。

## 操作の基本フロー
1. **browser_snapshot** でページのアクセシビリティツリーを取得する
2. スナップショットに含まれる ref（@e1, @e2 など）を確認する
3. ref を指定して **browser_click**, **browser_type** 等で要素を操作する
4. 操作後は再度 **browser_snapshot** で結果を確認する

## 利用可能なブラウザツール
- **browser_goto**: URL に移動する
- **browser_snapshot**: ページのアクセシビリティツリーを取得する（ref 付き）
- **browser_click**: ref を指定して要素をクリックする
- **browser_type**: ref を指定してテキストを入力する
- **browser_press**: キーボードキーを押す（Enter, Tab, Escape など）
- **browser_select**: ドロップダウンからオプションを選択する
- **browser_scroll**: ページまたは要素をスクロールする
- **browser_hover**: 要素にホバーする
- **browser_back**: ブラウザの戻るボタンを押す
- **browser_dialog**: ダイアログを処理する（確認、キャンセルなど）
- **browser_wait**: 要素や条件を待機する
- **browser_tabs**: タブの管理（一覧、作成、切り替え、閉じる）
- **browser_drag**: 要素をドラッグ＆ドロップする
- **browser_evaluate**: JavaScript を実行する
- **browser_close**: ブラウザを閉じる

## 利用可能なワークスペース機能
- **ファイルシステム**: ファイルの読み書き、一覧表示
- **サンドボックス**: シェルコマンドの実行
- **スキル**: 学習した手順の保存・再利用

## ガイドライン
- 操作前に必ず browser_snapshot で現在のページ状態を確認する
- ref（@e1, @e2 など）を使って要素を正確に指定する
- 操作後は再度 browser_snapshot で結果を検証する
- フォーム入力は browser_type で ref を指定し、必要に応じて browser_press で Enter を送信する
- ページ遷移後は browser_snapshot で新しいページの状態を取得する
- 抽出した結果はワークスペースにファイルとして保存できる
- スキルとして手順を保存し、同様のタスクで再利用する
- エラーが発生したら browser_snapshot でページの状態を確認してリトライする
- ログインなどユーザー本人の操作が必要な場合は、一時停止して Studio ブラウザパネルでの操作を案内する
`,
  model: 'openai/gpt-5.4',
  browser,
  workspace,
  memory: new Memory({
    options: {
      lastMessages: 40,
      observationalMemory: {
        model: 'openai/gpt-5-mini',
        temporalMarkers: true,
        observation: {
          previousObserverTokens: 10_000,
        },
      },
    },
  }),
});
```

## 落とし穴の整理

落とし穴を整理すると、次の 5 点になります。

1. **`browser` は `Agent` に渡す**（SDK プロバイダの場合）。`Workspace.browser` に入れると `INVALID_CONFIG` エラー
2. **`ws` と `@hono/node-ws` をインストール**しないとスクリーンキャストが無効化される（エラーは出ない）
3. **`screencast` オブジェクトを設定**する。`enabled` フラグはなく、オブジェクトの存在自体がスクリーンキャスト有効化のトリガー
4. **`instructions` にツール一覧を明示**する。少なくとも記事執筆時点では、LLMはブラウザ系のツールの使い方を詳しく理解できていなかった
5. **AgentBrowser 利用時は Playwright の実行ファイルを用意**する。pnpm 環境では `npx playwright install` が使えない場合があり、その際は `.pnpm` 配下の CLI を直接実行する
