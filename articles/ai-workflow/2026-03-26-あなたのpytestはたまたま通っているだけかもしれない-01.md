## あなたのpytestは何を見逃しているのか？

最近、TDDスキルを作成してLLMにテストを書いてもらう運用をしているのですが、ビジネスユースケースごとの観点が抜けていることに気づいて追加する → また別の観点が漏れている → 追加する……といたちごっこになっています。手動でテストケースを書く（あるいはLLMに書かせる）アプローチには構造的な限界があります。 **開発者が思いついたケースしかテストされない** という限界です。

例えば、文字列のサニタイズ関数をテストするとき、あなたはいくつのパターンを書きますか？ 5個？ 10個？ 仮に50個書いたとしても、Unicode結合文字、ゼロ幅スペース、サロゲートペアを含む文字列はテストしましたか？

Property-Based Testing（プロパティベーステスト）は、この「想像力の限界」を突破するアプローチです。Pythonでは **Hypothesis** というライブラリで実現でき、pytestとシームレスに統合できます。

### Property-Based Testingの適用範囲

本記事で紹介するProperty-Based Testingは、 **純粋関数のユニットテストを強化する枠組み** です。入力と出力が明確な関数（文字列処理、数値計算、エンコード/デコード等）に最も威力を発揮します。

なお、Hypothesisには `RuleBasedStateMachine` によるステートフルテスト（DB操作やセッション管理の状態遷移検証）や、APIパラメータの自動生成による統合テストへの応用も考えられます。ただし、これらの領域については私自身まだ十分に検証できていないため、本記事では純粋関数のユニットテストに焦点を絞ります。

## Property-Based Testingとは何が違うのか？

Property-Based Testingは、具体的な入力値ではなく「常に成り立つべき性質（プロパティ）」を定義するテスト手法です。HaskellのQuickCheckに着想を得たPythonライブラリです。

| 比較項目 | Example-Based | Property-Based |
| --- | --- | --- |
| テスト入力 | 開発者が手動で選定 | ライブラリが自動生成（デフォルト100回） |
| カバー範囲 | 想像力に依存 | 型の空間を網羅的に探索 |
| 失敗時の出力 | 指定した入力で失敗 | **Shrinking**で最小反例を自動特定 |
| 保守コスト | ケース追加が手動 | プロパティ1つで数千ケースをカバー |

## Hypothesisの基本はどう使うのか？

HypothesisはPythonのProperty-Based Testingライブラリで、`@given`デコレータとStrategy（データ生成戦略）を組み合わせて使います。pytestとの統合は追加設定なしで動作します。

```
from hypothesis import given, strategies as st

@given(st.text())
def test_upper_preserves_length(s):
    """大文字変換は文字列長を変えないはず"""
    assert len(s.upper()) == len(s)
```

この1つのテスト関数で、Hypothesisは空文字、ASCII、日本語、絵文字、制御文字など **数千パターンの文字列を自動生成** してテストします。

### Shrinkingとは何か？

Shrinkingは、テストが失敗した入力を自動的に最小化する機能です。例えば、1,000文字の入力で失敗した場合、Hypothesisは「同じ失敗を再現する最も短い入力」を探索します。これによりデバッグ時間が劇的に短縮されます。

## 実際にProperty-Based Testingでバグは見つかるのか？

ここからは具体的なコード例で、Example-Based Testingでは見逃しやすいバグをProperty-Based Testingで発見するケースを紹介します。

### ケース1: 文字列サニタイズの抜け穴

```
import re

def sanitize_input(text: str) -> str:
    """HTMLタグを除去するサニタイズ関数"""
    return re.sub(r'<[^>]+>', '', text)

# Example-Based: これは通る
def test_sanitize_basic():
    assert sanitize_input("<script>alert('xss')</script>") == "alert('xss')"
    assert sanitize_input("hello") == "hello"
    assert sanitize_input("<b>bold</b>") == "bold"

# Property-Based: 想定外のケースを発見
from hypothesis import given, strategies as st

@given(st.text(alphabet=st.characters(blacklist_categories=('Cs',))))
def test_sanitize_no_angle_brackets(s):
    """サニタイズ後にHTMLタグ（< >の組み合わせ）が残らないこと"""
    result = sanitize_input(s)
    # 入力に<>が含まれていた場合、出力から除去されているか？
    assert not re.search(r'<[^>]+>', result)
```

このテストを実行すると、`<`が閉じられていない不正なHTML（例: `<img src=x onerror=alert(1)`）で正規表現が期待通りに動かないケースが見つかることがあります。

### ケース2: 数値計算の境界値

```
def calculate_discount(price: int, rate: float) -> int:
    """割引価格を計算（小数点以下切り捨て）"""
    return int(price * (1 - rate))

# Property-Based: 割引後価格のプロパティを検証
@given(
    price=st.integers(min_value=0, max_value=1_000_000),
    rate=st.floats(min_value=0.0, max_value=1.0, allow_nan=False)
)
def test_discount_properties(price, rate):
    result = calculate_discount(price, rate)
    # プロパティ1: 割引後は元の価格以下
    assert result <= price
    # プロパティ2: 割引後は0以上
    assert result >= 0
```

浮動小数点の丸め誤差により、`rate`が0に極めて近い値のとき `int()` の切り捨てで `result > price` になるケースや、`rate`が1に近い値で負数が出るケースを発見できます。

### ケース3: JSON/APIレスポンスの異常系

```
import json

def extract_username(response_body: str) -> str:
    """APIレスポンスからusernameを抽出"""
    data = json.loads(response_body)
    return data["user"]["name"]

# Property-Based: 不正なJSONでクラッシュしないか検証
@given(st.text())
def test_extract_username_no_crash(response_body):
    """不正入力でも未ハンドルの例外が出ないこと"""
    try:
        extract_username(response_body)
    except (json.JSONDecodeError, KeyError, TypeError):
        pass  # 想定内のエラーはOK
    # それ以外の例外（AttributeError等）は失敗 = バグ
```

## Claude Code × Hypothesisで何ができるのか？

ここからが本記事の核心です。Property-Based Testingの最大の課題は **「どんなプロパティを定義すべきか」を考えるのが難しい** ことです。この思考プロセスをClaude Codeに委任できます。

### AIにプロパティを列挙させる

Claude Codeに以下のようなプロンプトを投げます：

```
このPython関数のプロパティ（常に成り立つべき性質）を列挙してください。
Hypothesisの@givenデコレータで検証可能な形式で出力してください。

def encode_base64(data: bytes) -> str:
    import base64
    return base64.b64encode(data).decode('ascii')
```

Claude Codeからは、このような出力が期待できます：

```
from hypothesis import given, strategies as st
import base64

@given(st.binary(max_size=1000))
def test_base64_roundtrip(data):
    """エンコード→デコードで元データに戻ること"""
    encoded = base64.b64encode(data).decode('ascii')
    decoded = base64.b64decode(encoded)
    assert decoded == data

@given(st.binary(max_size=1000))
def test_base64_output_is_ascii(data):
    """出力がASCII文字のみで構成されること"""
    encoded = base64.b64encode(data).decode('ascii')
    assert all(ord(c) < 128 for c in encoded)

@given(st.binary(max_size=1000))
def test_base64_length_relation(data):
    """出力長は入力長の約4/3倍（パディング込み）"""
    encoded = base64.b64encode(data).decode('ascii')
    expected_len = 4 * ((len(data) + 2) // 3)
    assert len(encoded) == expected_len
```

人間が「ラウンドトリップ」は思いついても、「出力長の数学的関係」まで網羅するのは難しいです。Claude Codeは関数のセマンティクスから体系的にプロパティを導出できます。

### 実践ワークフロー

## CI/CDにどう組み込むのか？

HypothesisはpytestプラグインとしてCI/CDに追加設定なしで組み込めます。以下はGitHub Actionsの設定例です。

```
# .github/workflows/test.yml
name: Tests with Property-Based Testing

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          pip install pytest hypothesis

      - name: Run tests (including property-based)
        run: |
          # CI環境ではデフォルト100回
          pytest tests/ -v

      - name: Run extended property tests (nightly)
        if: github.event_name == 'schedule'
        run: |
          # Nightly buildでは試行回数を増やす
          pytest tests/ -v --hypothesis-settings=ci_extended
```

```
# conftest.py — CI用のHypothesis設定
from hypothesis import settings

settings.register_profile(
    "ci_extended",
    max_examples=1000,       # 通常の10倍
    deadline=None,            # タイムアウト無効化
)
```

**ポイント** : 通常のCIではデフォルト（100回）で高速に回し、Nightly buildで試行回数を増やす二段構えがおすすめです。Hypothesisはデータベースに過去の失敗ケースを記憶するため、一度見つかったバグは以降のCIでも必ず再テストされます。

## まとめ — 「たまたま通っている」テストを脱却するために

Property-Based Testingは、手動テストの構造的な盲点である **開発者の想像力の限界** を補う手法です。Hypothesisを導入すれば、既存のpytestに数行追加するだけで、数千パターンの自動生成テストが手に入ります。

さらにClaude Codeと組み合わせることで、「どんなプロパティを検証すべきか」というテスト設計自体をAIに委任できます。

### やることリスト

1. `pip install hypothesis` して既存プロジェクトに導入する
2. 最もクリティカルな関数1つに `@given` テストを追加する
3. Claude Codeに「この関数のプロパティを列挙して」と依頼してみる
4. CI/CDにHypothesisを組み込み、Nightlyで試行回数を増やす

「テストが全部グリーンだから安全」ではありません。 **テストされていない入力パターンにこそ、本当のバグは潜んでいます。**
