### ◆業務フロー診断プロンプト

**✅目的：**  
　現在の業務フローを整理し、  
　無駄・重複・自動化できる箇所を洗い出す。

**✅いつ使うか：**  
　「なんとなく忙しい」と感じているとき。  
　業務改善の第一歩として。

```
# ロール設定
あなたは業務改善コンサルタントです。
依頼者の業務フローを客観的に診断し、
コスト・時間・リソースの無駄を
容赦なく指摘してください。

# 背景・目的・対象
・対象業務：[例：週次の売上レポート作成]
・現在の手順：[例：Excelでデータ集計→グラフ作成→メール送信]
・関わる人数：[例：担当1名、確認者2名]
・所要時間：[例：毎週3時間]
・課題感：[例：時間がかかる・ミスが出る]

# 指示内容
以下の4点を診断してください。

1. 業務フローの中で「なくせる・減らせる」工程
2. 自動化・ツール化できる工程（具体的なツール名も提案）
3. 担当者を変えた方が効率が上がる工程
4. 改善した場合の時間短縮・コスト削減の試算

# 制約条件
・現実的に実行できる改善案を優先する
・ツール導入コストも考慮する
・改善の優先順位（高・中・低）をつける

# 出力形式
診断結果を見出しつきで整理し、
改善案を優先度順に箇条書きで出力。

# 入力データ
業務の詳細：[ここに具体的な内容を入力]
```

**✅具体的な使用方法：**  
　「対象業務」と「現在の手順」に  
　自分の仕事内容をそのまま書くだけ。

　「毎週やってる面倒な作業」から  
　試してみてください。

**✅読者への一言：**  
　診断されることを恐れないでください。

　見えていなかった無駄が見えることが、  
　改善の第一歩です。

### ◆コスト構造見直しプロンプト

**✅目的：**  
　業務にかかっているコスト（人件費・時間・ツール費用）を  
　整理し、削減ポイントを特定する。

**✅いつ使うか：**  
　「なんとなくコストが高い気がする」  
　「利益率を改善したい」と感じているとき。

```
# ロール設定
あなたは中小企業向けの
コスト改善の専門家です。
依頼者のコスト構造を分析し、
削減・最適化できる箇所を
具体的に提示してください。

# 背景・目的・対象
・業種・事業内容：[例：ECサイト運営]
・主なコスト項目：[例：広告費・梱包費・外注費・人件費]
・現在の利益率（おおよそ）：[例：売上の15%程度]
・改善したい目標：[例：利益率を20%以上にしたい]

# 指示内容
1. コスト項目を「固定費」「変動費」に分類する
2. 各コスト項目の削減余地を評価する（高・中・低）
3. 削減余地が高いものから具体的な改善策を提示する
4. 改善後の利益率の試算（概算）

# 制約条件
・事業の継続性を損なう削減案は出さない
・短期（1ヶ月以内）・中期（3〜6ヶ月）の改善策を区別する
・数字で表現できるものは数字で示す

# 出力形式
コスト分類表＋改善策リスト（優先順位つき）

# 入力データ
コスト詳細・事業状況：[ここに入力]
```

**✅具体的な使用方法：**  
　「主なコスト項目」に  
　月次で出ている費用を  
　ざっくり書くだけでOKです。

　正確な数字がなくても、  
　AIが概算から改善案を出してくれます。

**✅読者への一言：**  
　コスト削減は「ケチること」ではありません。

　「使うべき場所に集中させること」です。

　このプロンプトで、  
　その仕分けをAIと一緒にやりましょう。

### After（本記事のプロンプトを使用）

```
# ロール設定
あなたは業務改善コンサルタントです。

# 背景・目的・対象

　各部門からExcelファイルを収集
　→手作業でデータを集約

　→PDF化してメール送信

・所要時間：毎週3時間
・課題感：集計ミスが出る・時間がかかる

# 入力データ
改善したい業務：上記のとおり
```

**AIの回答（例）**：

>  【削減できる工程】

>  →Googleスプレッドシート＋

>  自動グラフ化

>  【改善後の試算】  
>  毎週3時間→約20分に短縮  
>  年間削減時間：約130時間

「ロール設定」で  
AIを業務改善の専門家として起動し、

**実際に実行できる改善案が出てきました。**

AIの回答は現場に即した

## 実践プロンプト集

## 1. 業務の「見えない残業」を発見する

「定時に帰れない理由が  
　はっきりしない」

そう感じているなら、  
問題は「業務量」ではなく  
「業務の見えない無駄」かもしれません。

水槽に小さな穴が開いていたら、  
どれだけ水を注いでも  
水位が上がらないように、

小さな非効率の積み重ねが  
時間を静かに奪い続けています。

AIにその「穴」を探してもらいましょう。

---

### ◆見えない残業の原因診断プロンプト

**✅目的：**  
　「なぜ時間が足りないか」を  
　業務レベルで分解し、  
　具体的な改善策を見つける。

**✅いつ使うか：**  
　「忙しいのに成果が出ない」  
　「毎日残業なのに終わらない」  
　と感じているとき。

```
# ロール設定
あなたは生産性改善の専門家です。
依頼者の1日・1週間の業務内容を
詳細に分析し、
「やめていい仕事」
「減らせる仕事」
「人に任せるべき仕事」
を分類してください。

# 背景・目的・対象
・職種・役割：[例：中小企業の営業マネージャー]
・1日の主な業務（箇条書きで）：
　[例：朝礼30分・メール確認1時間・顧客対応2時間・
　　報告書作成1.5時間・会議2時間・雑務1時間]
・特に「時間がかかると感じる業務」：
　[例：報告書作成・社内会議]
・理想の勤務時間：[例：8時間以内]

# 指示内容
1. 業務を「価値を生む仕事」と
　「価値を生まない仕事」に分類する
2. 「やめる・減らす・任せる・自動化する」
　4軸で各業務を仕分けする
3. 最も優先的に改善すべき業務を3つ選び、
　具体的な改善案を提示する

# 制約条件
・完全に廃止できない業務には
　「削減案」を提示する
・AIツール・テンプレート・外注など
　現実的な手段を挙げる
・改善後の時間削減効果を試算する

# 出力形式
業務分類表＋優先改善案3点（効果試算つき）

# 入力データ
1日の業務詳細：[ここに入力]
```

**✅具体的な使用方法：**  
　「1日の主な業務」に  
　昨日やったことを  
　思い出しながら書くだけ。

　正確でなくても大丈夫です。  
　ざっくりで十分診断できます。

**✅読者への一言：**  
　「やめていい仕事をやり続けること」  
　ほど、もったいないことはありません。

　AIに棚卸しを手伝ってもらいましょう。

## 2. 繰り返し作業を自動化の候補に変える

毎週同じ作業を  
繰り返していませんか？

コピー・貼り付け・集計・送信。

まるで工場のベルトコンベアのように  
同じ動作を繰り返す仕事は、  
自動化の「宝の山」です。

**繰り返しがあるということは、  
パターンがあるということ。**

パターンがあるものは、  
必ず自動化できます。

---

### ◆繰り返し作業の自動化設計プロンプト

**✅目的：**  
　定型・反復業務を洗い出し、  
　自動化の具体的な手順と  
　ツールを設計する。

**✅いつ使うか：**  
　「毎回同じことをやっている」  
　と気づいたとき全般。

```
# ロール設定
あなたは業務自動化（RPA・ノーコードツール）の
専門家です。
依頼者の反復業務を分析し、
自動化の優先順位と
具体的な実装方法を提案してください。

RPA：Robotic Process Automationの略。
人間が行う定型作業を
ソフトウェアが代わりに行う技術のこと。

# 背景・目的・対象
・繰り返している作業の内容：
　[例：毎週月曜に各部門の数字をExcelに手入力し、
　　グラフを更新してSlackに投稿する]
・頻度と所要時間：[例：毎週・約90分]
・使っているツール：
　[例：Excel・Slack・Google スプレッドシート]
・ITスキルレベル：[例：初心者・中級者]

# 指示内容
1. この作業の自動化可能な部分を特定する
2. 自動化に使えるツールを
　難易度別に3つ提案する
   （無料・有料それぞれ）
3. 最も簡単な方法での
　自動化手順をステップ形式で説明する
4. 自動化後の週次・年次の時間削減効果を試算する

# 制約条件
・ITの専門知識がない人でも実行できる方法を優先する
・ツールの初期設定に必要な時間も明示する
・無料でできるものは無料を優先する

# 出力形式
自動化マップ（作業フロー図）＋ツール比較＋
実装ステップ

# 入力データ
作業の詳細・使用ツール：[ここに入力]
```

**✅具体的な使用方法：**  
　「毎回やっている作業」を  
　手順通りに書き出すだけ。

　「こんな細かい作業、自動化できる？」  
　と思うものほど、実は簡単に自動化できます。

**✅読者への一言：**  
　自動化は「エンジニアだけがやること」  
　ではありません。

　今は、文章で説明するだけで  
　AIが手順を設計してくれます。

　やってみると、驚くほど簡単です。

## 3. 会議の無駄を削ぎ落とす

「この会議、必要だったかな…」

会議が終わった後に  
そう感じたことはありませんか？

日本企業の会議は  
平均して業務時間の約30%を占める  
という調査もあります。

逆に言えば、  
会議を最適化するだけで  
**業務時間の3割を取り戻せる**  
可能性があります。

料理のレシピで言えば、  
「無駄な煮込み時間をなくす」と同じ。  
火が通るなら、それ以上は不要です。

---

### ◆会議設計最適化プロンプト

**✅目的：**  
　会議の目的・時間・参加者・アジェンダを最適化し、  
　「やって良かった会議」に変える。

**✅いつ使うか：**  
　会議を設計する前・  
　または「最近会議が多すぎる」と感じたとき。

```
# ロール設定
あなたは組織の会議改革コンサルタントです。
依頼者の会議の現状を診断し、
「本当に必要な会議」だけが残る
会議設計を提案してください。

# 背景・目的・対象
・現在の会議一覧（週次）：
　[例：朝礼30分・部門MTG1時間・全社報告会1時間・
　　プロジェクト定例1時間]
・会議の参加人数（おおよそ）：[例：平均5名]
・感じている問題点：
　[例：決定事項がない・話が脱線する・資料準備に時間がかかる]
・1週間の総会議時間：[例：約6時間]

# 指示内容
1. 各会議を「必要・縮小・廃止・非同期化」で仕分けする
   （非同期化＝リアルタイムでなく
   　メッセージや資料共有で代替できること）
2. 廃止・縮小できる会議の代替手段を提案する
3. 残す会議のアジェンダテンプレートを作成する
4. 改善後の週次・年次削減時間を試算する

# 制約条件
・意思決定に必要な会議は残す
・代替手段はSlack・Notionなど
　一般的なツールで対応できるものにする
・実行のしやすい改善順に並べる

# 出力形式
会議仕分け表＋アジェンダテンプレート＋効果試算

# 入力データ
会議の詳細・課題：[ここに入力]
```

**✅具体的な使用方法：**  
　「現在の会議一覧」を  
　曜日・時間・参加者でざっくり書くだけ。

　会議を「すべてなくせ」というわけではなく、  
　「必要なものだけ残す」設計をしてもらえます。

**✅読者への一言：**  
　会議は「集まることが目的」ではありません。

　「決めることが目的」です。

　この視点を持つだけで、  
　会議の質は劇的に変わります。

## 4. PDCA高速化でチームを進化させる

改善は、一度で終わりません。

「やってみて、測って、直す」  
このサイクルを速く回せる組織が  
最終的に勝ちます。

でも多くの現場では、  
P（計画）とD（実行）はするけど、  
C（確認）とA（改善）が抜けている。

これは意欲の問題ではなく、  
**「振り返りの仕組みがない」**  
ことが原因です。

筋トレで言えば、  
鍛えた後にタンパク質を摂らないと  
筋肉がつかないのと同じ。

PDCAも「実行後の振り返り」が  
最も重要なのです。

---

### ◆PDCA高速化振り返りプロンプト

**✅目的：**  
　業務改善の施策を実行した後、  
　効果を測定・分析し、  
　次の改善アクションを設計する。

**✅いつ使うか：**  
　改善施策を実行してから  
　1〜4週間後の振り返りタイミング。

```
# ロール設定
あなたは組織開発とKPI管理の専門家です。
依頼者の改善施策の振り返りを支援し、
次のアクションを具体的に設計してください。

KPI：Key Performance Indicatorの略。
目標達成のための重要な指標のこと。

# 背景・目的・対象
・実施した改善施策：
　[例：週次レポートをGoogleスプレッドシートで
　　自動化した]
・施策の目的（達成したかったこと）：
　[例：レポート作成時間を3時間から30分に削減]
・実施前の状態：[例：毎週3時間かかっていた]
・実施後の状態：[例：約1時間に短縮。まだ手作業が残っている]
・うまくいったこと：[例：集計は自動化できた]
・うまくいかなかったこと：[例：グラフ形式が崩れることがある]

# 指示内容
1. 目標達成度を評価する（数値化できるものは%で）
2. 「うまくいった要因」と「改善が必要な要因」を分析する
3. 次の改善アクションを3つ提案する（優先順位つき）
4. 1ヶ月後のゴール設定を提案する

# 制約条件
・感想ではなく事実・数字を根拠に分析する
・実行可能な改善案を優先する
・改善のモメンタムが続くような前向きな設計にする

# 出力形式
振り返りレポート形式（評価・分析・次のアクション）

# 入力データ
施策の詳細・結果：[ここに入力]
```

**✅具体的な使用方法：**  
　「実施前の状態」と「実施後の状態」を  
　比べて書くだけ。

　うまくいかなかったことを書くのが  
　特に重要です。  
　そこにこそ次の改善のヒントがあります。

**✅読者への一言：**  
　振り返りが苦手な理由は、  
　「何をどう振り返ればいいかわからない」  
　からだと思います。

　このプロンプトが、  
　その「型」を作ってくれます。

## 5. 利益率改善を組織全体に根付かせる

「上の人間だけが改善を言っても、  
　現場は動かない」

これは多くの職場で起きている  
リアルな問題です。

改善文化が定着しない理由は、  
**「やり方」ではなく「仕組み」がないから**  
です。

川の流れを変えるには、  
岩を一個ずつ動かすのではなく  
水が自然に流れる「水路」を作ることが必要。

改善文化も同じで、  
「やって当たり前」と感じる仕組みを  
AIと一緒に設計しましょう。

---

### ◆改善文化定着プロンプト

**✅目的：**  
　業務改善を個人の努力に頼らず、  
　組織の仕組みとして定着させる  
　設計を行う。

**✅いつ使うか：**  
　チームリーダー・マネージャーが  
　改善活動を継続させたいとき。

```
# ロール設定
あなたは組織変革と業務改善の
専門コンサルタントです。
依頼者の組織に「改善が続く仕組み」を
設計してください。

# 背景・目的・対象
・組織の規模：[例：10名のチーム]
・業種・事業内容：[例：Webマーケティング会社]
・現在の改善活動の状況：
　[例：個人が気づいたときだけ改善している・
　　定期的な見直しの仕組みがない]
・うまくいっていないこと：
　[例：改善案が出ても実行されない・
　　誰がやるか決まらない]
・目指したい状態：
　[例：月1回、全員が改善提案を出せるチームにしたい]

# 指示内容
1. 改善文化を定着させるための
　「仕組みの設計」を提案する（3〜5つ）
2. 各仕組みの導入手順を
　ステップ形式で説明する
3. 改善活動の継続を促す
　「評価・承認の仕組み」を提案する
4. 3ヶ月後に現れるべき変化の指標を設定する

# 制約条件
・コストをかけずにできることを優先する
・現場の負担を増やさない仕組みにする
・小さく始めて拡張できる設計にする

# 出力形式
仕組み設計（一覧＋手順）＋
3ヶ月後の変化指標

# 入力データ
組織の現状・課題：[ここに入力]
```

**✅具体的な使用方法：**  
　「現在の改善活動の状況」と  
　「うまくいっていないこと」を  
　正直に書くのがポイントです。

　きれいごとではなく、  
　現実を書くほど  
　実用的な設計が出てきます。

**✅読者への一言：**  
　改善は「頑張る文化」では続きません。

　「やらなければいけない仕組み」と  
　「やりたくなる仕組み」の両方が要ります。

　AIがその設計を  
　一緒に考えてくれます。
