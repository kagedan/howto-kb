## こんな悩みありませんか？

「CodexもClaudeも使えるようにしたけど、結局どっちに何を頼ればいいか分からない」

正直、最初の頃は僕もそうでした。「とりあえずClaudeに聞いてみる」→うまくいかない→「Codexで試してみる」の繰り返し。どちらも強力なツールなのに、使い方がぼんやりしているせいで半分も活かせていなかった。

**結論：役割を明確に分けて2刀流で使うと、PdM業務の生産性が劇的に上がります。**

Codexは「手」、Claudeは「頭」。この一言を軸に整理すると、迷いがなくなります。

## 基本の違い

まず「何が本質的に違うのか」を押さえます。

Codexはコードを実際に「動かして」結果を出すツール。Claudeは言語と論理を使って「考えて」答えを出すツール。この違いが使い分けの起点になります。

## PdM実務での具体的な使い分け例

ここが一番大事なところ。抽象的な説明より、「あの業務はどっち？」を見た方が理解が早いです。

これを見ると、「考えること・言語化すること」はClaude、「動かすこと・作ること」はCodexという軸が見えてくるはずです。

---

## ハイブリッド活用

2つを組み合わせると、さらに強力になります。実務でよく使うパターンを3つ紹介します。

### パターンA：Claude → Codex（設計から実装へ）

「何を見るべきか分からない状態でCodexに投げても意味がない」と気づいてから、このパターンを使うようになりました。Claude で問いを立ててから Codex で動かす、この順番が大事。

### パターンB：Codex → Claude（データから洞察へ）

数字が出た後の「で、どうする？」をClaudeに任せると、施策提案まで一気通貫で進みます。

### パターンC：並列活用

独立したタスクは同時並行で進めて時短できます。

## 判断フロー

迷ったときは、このフローに沿って考えてみてください。

```
新しいタスクが発生
        │
        ▼
「コードを書く / 実行する / ファイルを操作する」が必要？
        │
       YES ──────────────────────► Codex
        │
        NO
        │
        ▼
「考える / 整理する / 文書を作る / 壁打ちする」が中心？
        │
       YES ──────────────────────► Claude
        │
        NO（両方必要）
        │
        ▼
   Claude で設計・仮説整理
        │
        ▼
   Codex で実装・実行・検証
        │
        ▼
   Claude で解釈・文書化
```

9割のタスクはこれで判断できます。

## やりがちな失敗と対策

使い始めの頃、僕自身がハマった失敗パターンをまとめました。

特に「コンテキストの引き継ぎ」は見落としがちです。Codexで出したデータをClaudeに渡すとき、「このデータの背景・目的」も一緒に渡さないと精度が落ちます。

## まとめ

**Codexは「手」、Claudeは「頭」**

PdM業務の本質は、「何を作るか決める（戦略）」と「それを実現する（実行）」の往復です。

この2つを正しい場面で使い分けることで、一人PdMでもエンジニアリングと戦略の両側をカバーできるようになります。

まず1週間、「迷ったら判断フローに当てはめる」だけ試してみてください。思った以上に業務のスピードが変わります。
