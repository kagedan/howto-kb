2026年4月頃にリリースされた Opus 4.7 あたりから、Claude と細かく会話しながら開発することが少し苦痛になってきました。指示や会話を積み重ねるよりも、最初に依頼と境界を決めて、あとは構造化された手順の中で実行してもらう方が自然なのではないか、と感じるようになりました。

そんな中で Claude から出てきた `ant` コマンドが `kubectl` っぽいと感じ、Argo を思い出しながら、**「AI と会話しない」** を目標に AI パイプラインの CLI を作ることにしました。

今回、あえて**構造化パイプライン**を CLI にすることで、「コスト」や「不確実性」に対して安心して回せる開発サイクルを作ることができました。備忘録の意味も込めて書いておきます。

## spec-runner

spec-runner は、`request.md` を入力にして、設計（design）、レビュー、実装、検証、PR 作成までを決められたステップで実行する CLI です。設計のやり方自体を固定するものではありません。ステップの中に `design` はありますが、これは設計プロセス全体を肩代わりするものではなく、あくまで実装方針を整理するためのステップです。

設計は人間が考えても、別の AI や別のプロセスを使っても構いません。軽量な依頼から始めてもよく、重要なのは、その時点での意図や境界を `request.md` としてパイプラインに渡すことです。

<https://github.com/color4pen/spec-runner>

### 初期セットアップ

spec-runner は、npmパッケージとしてインストールして使います。各リポジトリの設定、検証コマンド、ルール、実行履歴と一体で動く CLI なので、継続利用する場合は devDependency として固定する方が再現性が高いと考えています。グローバルインストールでも動きますが、この記事では project-local に入れる前提で説明します。

このプロジェクトは Bun で開発しているため例も Bun で書いていますが、npm や pnpm でも利用できます。

```
bun add -D @color4pen/specrunner
```

その後、対象リポジトリで初期化します。

初期化すると、`specrunner/` 配下に設定ファイルや作業ディレクトリが作られます。Pull Request の作成や merge 状態の確認には GitHub 認証が必要なので、初回は GitHub へのログインも行います。

spec-runner の `login` は、spec-runner が GitHub API を利用するための認証です。Pull Request の作成、merge 状態の確認、archive 時の確認などに使います。

一方で、実装ステップで動く agent には git 操作を任せていません。ブランチ作成、コミット、PR 作成などの git / GitHub 操作は spec-runner 側で行います。agent は、与えられた作業範囲の中でファイルを編集するだけです。

また、Claude / Codex など agent 側の認証は spec-runner では管理しません。ローカルで `claude` や `codex` のコマンドを起動できる状態になっていれば、その環境を使って実行できます。

初期化後は、生成された config に verification コマンドや各ステップで使うモデルを設定します。以下は自分のリポジトリで使っている設定例です。

```
{
  "version": 1,
  "verification": {
    "commands": [
      { "name": "build", "run": "bun run build" },
      { "name": "typecheck", "run": "bun run typecheck" },
      { "name": "test", "run": "bun run test" },
      { "name": "lint", "run": "bun run lint" }
    ]
  },
  "steps": {
    "defaults": {
      "model": "claude-sonnet-4-6",
      "maxTurns": null,
      "timeoutMs": null
    },
    "design": {
      "model": "claude-opus-4-8[1m]"
    }
  },
  "archive": {
    "mergeWaitTimeoutMs": null
  }
}
```

ターン数やタイムアウトの制限機能は便宜上つけています。自分の運用では、パイプライン側の停止条件とエスカレーションを信頼しているため、無制限を表す `null` にしています。ただし、チーム運用や従量課金環境では、まず上限を設定して様子を見る方が安全だと思います。

`run` に必要な入力は、何をやりたいかを書いた `request.md` だけです。このファイルには「要件」「スコープ外」「受け入れ基準」を記入します。

```
# 依頼の draft を作成する
bun run specrunner request new my-feature

# パイプラインを実行する
bun run specrunner run my-feature

# PR の merge を確認して作業ファイルをアーカイブする
bun run specrunner job archive --with-merge my-feature
```

`run` コマンドの中では、次の 10 ステップが実行されます。

```
request.md
  ↓
① request-review    依頼の妥当性を検証
  ↓
② design            ブランチ作成、仕様ファイル生成
  ↓
③ spec-review       仕様のレビュー（不合格なら修正ループ）
  ↓
④ test-case-gen     テストケース定義の生成
  ↓
⑤ implementer       コードの実装
  ↓
⑥ verification      ビルド・型チェック・テスト・lint（不合格なら修正ループ）
  ↓
⑦ code-review       コードレビュー（不合格なら修正ループ）
  ↓
⑧ conformance       設計物に対するチェック
  ↓
⑨ adr-gen           技術的決定の生成
  ↓
⑩ pr-create         GitHub PR の作成
  ↓
Pull Request
```

各ステップは SDK 経由で実行されます。ローカルで `claude` や `codex` のコマンドを起動できる状態になっていれば、その環境を使って agent を実行できます。

ただし、課金条件は CLI / SDK / 契約形態によって変わります。特に Claude の場合、`-p` まわりの従量課金の扱いには注意が必要そうです。実際に使う場合は、各ベンダーの最新仕様を確認してください。

## 止まることを前提にしている

パイプラインは、特定の条件で実行を中断することがあります。たとえば、依頼が曖昧だったり、実装や設計の中でスコープを逸脱しそうになった場合です。AI が勝手に解釈して進むのではなく、人間の判断が必要なところで止まります。

パイプラインが一時停止したとき、コマンドのオペレーターはエスカレーションの内容を読んで判断します。必要な判断・作業を行った上で `resume` コマンドを実行すると、そこから再開できます。

```
# 止まったジョブを再開
bun run specrunner job resume my-feature

# 特定のステップにプロンプトを渡して再開する
bun run specrunner job resume my-feature --prompt "認証まわりは JWT に統一して"
```

個人的には、この「どこで止まるか」の精度が、このパイプラインの良いところだと感じています。つまり、オペレーターがやることは 3 つだけです。

**依頼を書く、PR を確認する、詰まったら判断する。**

## 作る時に工夫したこと

### 1. AI に合否を言わせない

判断を AI に委ねたくない、というのがそもそもの動機でした。ただ、「AI に判断させない」と言っても、単にプロンプトで禁止するだけでは不十分です。そこで、AI が判断する場面を構造から消す方向で設計しました。

レビュー系のステップ、たとえば `spec-review`、`code-review`、`conformance` などで agent が返すのは、最終的な合否ではありません。返すのは、あくまで**レビュー指摘の一覧**です。一件ごとに、次の情報を付けます。

* 重大度: `critical` / `high` / `medium` / `low`
* 対応種別: `fixable` / `decision-needed`
* 該当する `file:line`

プロンプトの中で agent に**最終判定**を返させません。最終判定は、CLI 側の純関数がレビュー指摘から導出します。

イメージはこれだけです。

```
function deriveFinalJudgement(findings, ok) {
  if (!ok) return "escalation";
  if (findings.some(f => f.resolution === "decision-needed")) return "escalation";
  if (findings.some(f => f.severity === "critical" || f.severity === "high")) return "needs-fix";
  return "approved";
}
```

`decision-needed`、つまり人間の判断が必要な指摘が一件でもあれば、人間にエスカレーションします。`critical` / `high` の指摘があれば、修正ループに入ります。どちらもなければ合格です。

これは順序が決まっているだけの、テストしやすい関数です。

さらに、合格を止める指摘が参照している `file:line` は、CLI の実行時に実在を検証します。存在しない行を指していたら、それ自体をエスカレーションにします。agent が「直すべき場所」を捏造して合格を演出する経路も塞いであります。

人間判断が必要な指摘には、選択肢を 2 つ以上付けることも強制しています。二択を言語化できないなら、それは判断が要る問題ではなく、単に直せる問題、つまり `fixable` だという線引きです。

つまり、agent には「自分の仕事を自分で合格にする」経路が構造的に存在しません。

この分離の対価は大きく、**判定の形式がモデルの能力に依存しなくなります**。ステップごとにモデルを差し替えても、安いモデルに落としても、契約は CLI 側にあるので壊れません。

判断の質は agent に求めますが、判断の権限は渡しません。同じ原理は、ほかの場所にも出てきます。reviewer の起動条件は LLM ではなく、変更ファイル一覧との glob 照合で決まります。ステップの完了も、宣言された出力契約を機械的に検証して確かめます。

agent が判断する場面を一つずつ消していくことが、不確実性への根本対策でした。プロンプトにルールを足し続けるのは、対症療法でしかない。これが、このパイプラインを書いていて一番腹落ちしたところです。

### 2. 状態はファイルで管理する

CI でも動かせるようにしたかったので、着手時に Docker のような daemon を持たせたり、面倒なプロセス管理をしたりしないと決めていました。

ジョブの状態は `events.jsonl` というジャーナルに 1 行 1 レコードで追記していきます。状態を読むときは、全レコードを畳み込みます。書き込み途中でクラッシュして最終行が壊れていても、その行は `JSON.parse` に失敗するので黙って捨てます。`state.json` はそこから導かれた表示用の状態にすぎません。一次情報はあくまでジャーナルとブランチ上のファイルです。

これによりプロセスにも、セッションにも、メモリにも一次情報はありません。なので、無人実行時は常駐プロセスである必要がありません。定期的に起動される短命プロセスが、毎回「今の状態から次にやるべき操作」を再導出するだけでよくなります。

クラッシュ、再起動、スリープからの復帰に専用の回復処理は要りません。通常動作とまったく同じコードパスを通るからです。言い換えると、回復経路が常にテストされ続けている状態になります。

副次的な効果もあります。文脈をセッションではなくファイルから毎回組み直すので、どのモデルが来ても「読めば続きができる」状態になります。これはモデル価格の変動に対するヘッジになっています。

また、同じリポジトリ構造や作業コンテキストが繰り返し読まれるため、Claude API のプロンプトキャッシュが効きやすい構造になっています。実際に `usage.json` を集計すると、Claude-only の一発完走ジョブでは cache read 比率の中央値が 92.1% でした。自動修正で収束したジョブや、人間判断で復帰したジョブでも、cache read 比率の中央値は 92% 台に収まっています。

ただし、`usage.json` に記録されているのは token 数であり、請求額そのものではありません。そのため、この記事では金額を扱う場合、実測請求額ではなく、公開単価ベースの試算として明示します。

### 3. フォローアッププロンプトで規律やルールを注入する

ステップに規律を足したくなったとき、コードやプロンプトを書き足したくありませんでした。新しいレビュー観点が欲しいたびにステップを増やしていたら、結局その場その場で agent に任せる複雑なフローに戻ってしまいます。

そこで、作業ターンが成功で終わった後、**同じセッションに追加のプロンプトを順番に投げる**仕組みにしました。主な狙いは規律の注入です。

`specrunner/rules/<step>/` に markdown を 1 枚コミットしておくと、その内容が作業後の追加プロンプトに変換されます。そして、「いま終わらせた作業をこの規約に照らして確認しろ」というターンとして、同じセッションに流れます。

CLI が付ける枠は 3 つだけに固定してあります。

* 修正範囲: この規約に関連するファイルだけ触る
* 停止条件: 違反がなければ何も変えず終了する
* 意図解釈: 言葉を機械的に当てるのではなく、規約の意図を汲む

規律を 1 つ増やすのに、セッション数は増えません。markdown を置くだけです。

これは、工夫 1 の帰結でもあります。判定の契約が CLI 側に標準化されているので、reviewer ごとに変わるのはプロンプト素材と設定だけになります。拡張のためにコードを開く必要がありません。

この仕組みは、agent が返却する JSON が定義した形になっていない場合にも使っています。agent がツールを呼ばずにターンを終えたり、レスポンスの JSON が空だった場合、「呼ばれていない」「どのフィールドが足りない」を名指しして、返却が壊れていたらフォローアップで直させます。

ちなみに、`resume --prompt` で人間が渡す指示も、この同じ仕組みに乗っています。

## 実行結果

spec-runner は、spec-runner 自身の開発にも使っています。機能の多く、たとえばカスタムレビューの並列実行、マルチプロバイダ対応、パーミッションスコープ、判断台帳などは、spec-runner 自身のパイプラインで実装・レビュー・マージしてきました。全過程は `specrunner/changes/archive` に残っています。

アーカイブ済みの 361 件について、`events.jsonl` と `usage.json` から実行経路を集計しました。分類は次のようにしています。

* 一発完走: fixer も escalation / resume も含まない
* 自動修正で収束: fixer を含み、human resume を含まない
* 人間判断で復帰: escalation / resume を含む

人間判断で復帰したジョブの中に fixer を含むものもあります。その場合は「人間判断で復帰」に分類しています。

| 経路 | 件数 | 割合 |
| --- | --- | --- |
| 一発完走 | 198 | 54.8% |
| 自動修正で収束 | 113 | 31.3% |
| 人間判断で復帰 | 50 | 13.9% |

結果として、**86.1% は人間判断なしで完走し、13.9% は人間判断が必要なところで止まった**ことになります。

ここで重要なのは、AI が常に正しかったことではありません。むしろ、AI が間違えることを前提に、直せるものは修正ループに回し、人間が決めるべきものは止める、という分岐が実際に機能していたことです。

一発完走したものは、追加の修正ループや人間判断を挟まずに PR 作成まで到達しています。自動修正で収束したものは、review や verification の指摘を受けて fixer が動き、そのまま人間判断なしで完走しています。

一方で、人間判断で復帰した 50 件は、パイプラインが勝手に進まず、一度止まった上で `resume` によって再開しています。

### コストについて

もう一つ見たかったのは、小さい作業でもこの安全装置を現実的なコストで回せるかです。

`usage.json` には請求額そのものは記録していません。記録されているのは `inputTokens`、`outputTokens`、`cacheCreationInputTokens`、`cacheReadInputTokens` です。そこで、Claude の公開単価を使い、input / output / cache creation / cache read を分けて積算しました。この値は請求実績ではないため、ここでは `estimatedUsd` と呼びます。

GPT モデルを含む 9 件は、単価を揃えられないため `estimatedUsd` の集計から除外しています。Claude-only の一発完走ジョブ 180 件では、`estimatedUsd` の中央値は $3.93 でした。

| 指標 | 値 |
| --- | --- |
| 一発完走ジョブ数 | 198 |
| usage あり | 181 |
| Claude-only estimatedUsd 対象 | 180 |
| tokens p50 | 5.3M |
| estimatedUsd p25 | $2.84 |
| estimatedUsd p50 | $3.93 |
| estimatedUsd p75 | $6.06 |
| estimatedUsd p90 | $11.10 |

さらに、tokens が 1M 以下の軽い一発完走ジョブに絞ると 6 件あり、`estimatedUsd` は $0.48〜$1.15、中央値は $0.73 でした。

| 条件 | 件数 | estimatedUsd min | estimatedUsd p50 | estimatedUsd max |
| --- | --- | --- | --- | --- |
| 一発完走かつ tokens ≤ 1M | 6 | $0.48 | $0.73 | $1.15 |
| 一発完走かつ tokens ≤ 1.5M | 9 | $0.48 | $1.04 | $1.79 |

もちろん、小さい作業をすべて $1 で回せるわけではありません。一発完走全体の中央値は $3.93 なので、「一発完走は $1 で回る」とは言いません。

ただ、軽い一発完走ジョブの中には、設計・レビュー・検証・PR 作成まで含めたパイプライン全体を、公開単価ベースの試算で $1 前後で完走しているものがありました。

自分にとって重要だったのは、AI の実行コストを最小にすることではなく、コストを観測できる形にすることでした。どのジョブが重く、どのジョブが軽く、どこで修正ループや人間判断が発生したのかを、後から見返せる。これによって、「AI に任せたらいくらかかるか分からない」という不安をかなり減らせました。

この結果を見ると、パイプラインで重要なのは「AI が常に正しいこと」ではありませんでした。AI が間違えることを前提に、直せるものは直し、人間が決めるべきものは止める。そのために、判断を agent に名乗らせず、CLI 側で導出し、状態をファイルに残して何度でも再開できるようにしました。

この**止まり方と戻り方**を設計したことが、361 件を回し続けられた一番の理由だったと思います。

## 所感

設計レイヤにはまだ考えるべきことが多い一方で、設計や方針がある程度決まった後の実装・検証・PR 作成は、先に構造化できると考えました。

最近の AI coding agent や AIDD は、設計から実装までを一気通貫で扱う方向に進んでいます。その中で実装に境界を設けた spec-runner は、少しニッチな切り口をしているのかもしれません。

ただ、自分が欲しかったのは「AI にすべてを任せる体験」ではなく、AI が働く範囲と止まり方を明確にした実行基盤でした。そう考えると、このニッチさは弱みというより、安心して使うために意図的に選んだ制約だったと思っています。

自分の開発の基盤を整えられたという意味で、個人的には過去一で意味のあるアウトプットになりました。今後はこの CLI を使いながら、さらに開発サイクルを回していきたいと思っています。

このリポジトリをメンテナンスしつつ、今回作ったステップ実行の仕組みを、他のプロジェクトでも再利用できる形に抽象化していきたいです。

また、今回は設計や方針がある程度決まった後の実装レイヤを主な対象にしましたが、その手前にある設計レイヤをどう扱うかも考えていきたいと思っています。
