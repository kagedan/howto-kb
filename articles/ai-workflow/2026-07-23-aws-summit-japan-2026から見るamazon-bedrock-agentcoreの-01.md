## 1. はじめに

生成AIやAIエージェントを業務で活用する動きが広がる中で、最近は「AIエージェントをどう作るか」だけでなく、**AIエージェントをどう本番運用するか**が重要なテーマになってきています。

PoCでは、LLMにツールを呼び出させたり、RAGと組み合わせたりすることで、比較的短期間で動くものを作ることができます。

一方で、本番利用を考えると、次のような課題が出てきます。

* エージェントをどこで安全に実行するのか
* ユーザーやエージェントの認証・認可をどう扱うのか
* 外部SaaSや社内APIへのアクセス権限をどう制御するのか
* データ、ナレッジ、ツール、業務ルールをAIが理解・活用しやすい形に整え、安全にアクセスできるAI Readyな環境をどう構築するのか
* 会話履歴やユーザーごとの業務コンテキストをどのように記憶・活用するのか
* 実行ログ、トレース、評価結果をもとに、AgentOpsとして継続的な改善サイクルをどう回すのか
* プロンプトインジェクションや過剰な権限付与、意図しないデータアクセスをどう防ぐのか

AWS Summit Japan 2026の各セッションを見ると、Amazon Bedrock AgentCoreは、こうした課題に対して **AIエージェントを本番運用するための基盤**として位置付けられていることが分かります。

本記事では、AWS Summit Japan 2026のAWSセッションをもとに、Amazon Bedrock AgentCoreに関するセッションと、そこから見えるトレンドを整理します。

## 3. Amazon Bedrock AgentCoreとは何か

今回のセッション群を見ると、Amazon Bedrock AgentCoreは、AIエージェントを本番環境で安全かつ大規模に運用するための基盤として紹介されています。

AIM342では、Amazon Bedrock AgentCoreについて、任意のモデル、任意のフレームワーク、任意のIDプロバイダー、MCPやA2Aなどと組み合わせてAIエージェントを本番運用するための基盤として整理されています。

AgentCoreには、以下のような機能が含まれます。

| 機能 | 役割 |
| --- | --- |
| Harness | コンソール上でモデル、指示、ツール、スキル、メモリなどを設定し、ローコードでAIエージェントを構築するための機能 |
| Runtime | 任意のフレームワークやモデルで作成したAIエージェントやツールを、セキュアかつスケーラブルに実行するためのインフラ環境 |
| Gateway | API、Lambda、OpenAPIスキーマ、MCPサーバーなどを、AIエージェントが利用できるツールとして安全に公開・管理するための接続レイヤー |
| Identity | ユーザーやエージェントの認証・認可、外部サービスへの委任アクセス、トークン管理を扱う機能 |
| Memory | 会話履歴やユーザーごとの業務コンテキストを短期・長期で保持し、複数セッションにまたがる文脈理解を支える機能 |
| Policy | エージェントが実行できる操作やアクセスできるリソースを、自然言語またはCedarなどのポリシーで制御する機能 |
| Built-in tools | ブラウザ、コードインタープリター、Knowledge Base、Web Searchなど、エージェントから利用できる組み込みツール群 |
| Payments | エージェントが有料API、MCPサーバー、有料コンテンツなどへアクセスする際の支払い、利用上限、支払い状況の可観測性を扱う機能 |
| Registry | 組織内のエージェント、MCPサーバー、ツール、スキルなどをカタログ化し、検索・承認・再利用しやすくするための仕組み |
| Observability | エージェントの実行ログ、トレース、メトリクスを可視化し、デバッグや運用監視を支える機能 |
| Evaluations | エージェントの回答品質、ツール選択、振る舞いなどを評価し、継続的な改善につなげる機能 |

このように見ると、AgentCoreは「エージェントを実行するためのランタイム」にとどまらず、**構築、実行、接続、認証認可、記憶、評価、監視、改善までを一連で支える基盤**と捉えるのがよさそうです。

特にHarnessを利用すると、インフラやエージェントループの実装を細かく意識せずに、モデル、指示、ツール、メモリなどを組み合わせてAIエージェントを構築できます。一方で、既存フレームワークや独自実装を活かしたい場合はRuntimeを利用できます。

また、Gateway、Identity、Policy、Observability、Evaluationsなどを組み合わせることで、AIエージェントが必要なデータやツールに安全にアクセスし、その実行内容を観測・評価しながら改善していく流れを作ることができます。

つまり、Amazon Bedrock AgentCoreは、AIエージェントをPoCで終わらせず、企業システムの中で本番運用していくための機能群が揃ったサービスと捉えると分かりやすいです。

## 4. AgentCoreから見るAIエージェント本番化のトレンド

ここからは、AWS Summit Japan 2026の各セッションをもとに、Amazon Bedrock AgentCoreから見えるAIエージェント本番化のトレンドを整理します。

今回のセッション群を見ると、AIエージェントは「PoCで動かす」段階から、「業務システムに安全に接続し、継続的に評価・改善しながら運用する」段階へ進みつつあるように感じます。

その中で重要になるのが、実行環境、認証認可、ツール接続、メモリ、評価、可観測性、ポリシー制御を一連で扱うAgentCoreのような基盤です。

### 4.1 AIエージェントはPoCから本番運用の段階へ

AIM342では、プロトタイプから本番環境に進む際の課題として、パフォーマンス、スケーラビリティ、セキュリティ、ガバナンスが挙げられています。

PoCでは、エージェントがツールを呼び出し、期待通りの回答を返せば一定の成果になります。しかし、本番利用ではそれだけでは不十分です。

本番運用では、以下のような観点が必要になります。

* ユーザー単位で安全に実行できること
* セッションや記憶を分離できること
* ツールやAPIへのアクセスを制御できること
* 実行ログやトレースを追えること
* 品質を継続的に評価できること
* 問題があればロールバックできること
* 組織として再利用可能な標準基盤にできること

AIM342では、AIエージェントを本番運用するためのベストプラクティスとして、以下のような内容が整理されています。

![](https://static.zenn.studio/user-upload/ace261d2099e-20260720.png)  
*出典：AWS Summit Japan 2026 AIM342「本番運用を見据えた AI エージェント - Amazon Bedrock AgentCore を活用したベストプラクティス」*

この点から、AgentCoreの価値は、AIエージェントを「試す」段階から「継続的に運用する」段階へ進めることにあると考えられます。

### 4.2 エージェントの権限委任を安全に管理する

AIエージェントが業務システムやSaaSにアクセスする場合、重要になるのがIDとアクセス制御です。

SEC353では、AIエージェントを組織全体で活用する際の課題として、主に以下の3点が整理されています。

| 観点 | 課題 |
| --- | --- |
| セキュリティ | エージェントに必要最小限の権限だけを安全に委任したい |
| UX | ユーザーに毎回同意を求めず、自然に使える体験にしたい |
| 開発速度 | 認証・認可やトークン管理を個別実装せず、素早く本番適用したい |

たとえば、営業支援エージェントがCRM、財務システム、メールにアクセスして会議準備を支援するケースを考えると、単に「エージェントがシステムにアクセスできる」だけでは不十分です。

* 誰がエージェントを呼び出しているのか
* そのユーザーはそのエージェントを利用してよいのか
* エージェントは誰の代理で外部サービスにアクセスするのか
* どのツールに、どの範囲の権限を渡すのか
* 誰が何に同意し、何が実行されたのかを後から追えるのか

こうした論点を支えるのが、**Amazon Bedrock AgentCore Identity** です。

![](https://static.zenn.studio/user-upload/eea3de416b38-20260720.png)  
*出典：AWS Summit Japan 2026 SEC353「エージェンティック AI アプリのセキュリティ・UX・開発速度を同時に実現」*

AgentCore Identityは、大きく **Inbound Auth** と **Outbound Auth** の2つで整理できます。

| 観点 | 役割 |
| --- | --- |
| Inbound Auth | エージェントを呼び出すユーザーやアプリケーションを検証する |
| Outbound Auth | エージェントがユーザーの代理で外部サービスへアクセスするための認証情報を扱う |

Inbound Authでは、既存のIdPが発行したトークンを検証し、ユーザーやグループなどの情報をもとにエージェントを実行してよいかを判断できます。

Outbound Authでは、外部サービスへの初回アクセス時にユーザーの同意を取得し、その後はToken Vaultに保存されたトークンを利用することで、毎回同意を求めずにアクセスできます。また、Workload Access Tokenにより、ユーザーとエージェントの組み合わせをもとに認証情報を分離できるため、別ユーザーのトークンが誤って使われるリスクを抑えられます。

AIエージェントの本番化では、モデルやプロンプトだけでなく、**誰が、誰の代理で、どのリソースに、どの権限でアクセスするのか**を設計する必要があります。

AgentCore Identityは、そのための認証、権限委任、トークン管理、監査を支える中核機能と捉えることができます。

### 4.3 データエージェントでは多層防御が必要になる

AIM344では、SaaSデータエージェントの設計において、データエージェントを4つのレイヤーで守る考え方が紹介されています。

![](https://static.zenn.studio/user-upload/567c0696b1b3-20260720.png)  
*出典：AWS Summit Japan 2026 AIM344「Amazon Bedrock AgentCore による堅牢な SaaS データエージェントの設計」*

| レイヤー | 主な論点 |
| --- | --- |
| LLM層 | 入出力、プロンプト、ガードレール |
| 呼び出し層 | ユーザーやテナントの実行コンテキスト |
| ツール層 | ツール呼び出し、パラメータ検証、業務ロジック |
| リソース層 | DB、ナレッジベース、メモリ、ストレージへのアクセス制御 |

データエージェントでは、ユーザーの自然言語に応じてデータを取得・分析します。そのため、従来のBIのように事前定義されたSQLだけを実行する構成とは異なり、柔軟である一方、リスクも大きくなります。

たとえば、以下のようなリスクが考えられます。

* PIIや機密情報の漏えい
* ユーザーや組織をまたいだ不適切なデータ参照
* 業務外トピックへの応答
* 非効率なクエリの実行
* プロンプトインジェクション
* LLMが生成するSQLを制御できないこと
* ツール呼び出しの認可がLLM任せになること

特に、マルチテナントSaaSでは注意が必要です。ここでいうテナント越境とは、本来はA社のユーザーがA社のデータだけを参照すべきところ、AIエージェントが誤ってB社のデータまで取得・回答してしまうようなケースを指します。

重要なのは、**LLMに「tenant\_idをWHERE句に入れて」と指示するだけでは、テナント分離の保証にはならない**という点です。

テナントIDやユーザーIDのような重要な制御情報は、LLMの推論に任せるのではなく、JWTなどの実行コンテキストから機械的に取得し、ツール層やリソース層へ伝搬する必要があります。

AIエージェントのセキュリティは、プロンプトやガードレールだけで完結するものではありません。ID、ツール、API、データベース、ストレージまで含めた多層防御が必要になります。

### 4.4 Agentic RAGではMemoryとContext Engineeringが重要になる

AIM412では、AgentCoreを活用したエンタープライズAgentic RAGの実装が紹介されています。

従来のRAGでは、質問に関連する文書を検索し、LLMへ渡して回答させる構成が一般的です。

一方、Agentic RAGでは、エージェントがツールやメモリを使いながら、必要な情報を取得し、会話の文脈も踏まえて回答します。

RAGシステムの現場では、以下のような課題が起きやすいです。

* ナレッジベースから検索しても、ユーザーの意図や文脈を理解できず、関連性の低い情報を返してしまう
* 会話履歴をすべてコンテキストに含めると、トークン数が膨大になり、コスト増・応答遅延・精度低下を招く
* PoCではうまく動いても、エンタープライズ環境へ展開できない
* エージェントが過去の会話やユーザーの好みを記憶できず、毎回ゼロからのやり取りになる

ここで重要になるのが、**Memory** と **Context Engineering** です。

コンテキストは「多ければよい」わけではありません。不要な情報やノイズが増えると、回答品質、コスト、レイテンシーに悪影響が出ます。

![](https://static.zenn.studio/user-upload/4b28743dd91b-20260720.png)  
*出典：AWS Summit Japan 2026 AIM412「Amazon Bedrock AgentCore を活用したエンタープライズ Agentic RAG の実装解説」*

Agentic RAGでは、以下を設計することが重要になります。

* どの情報を検索するか
* どの情報をコンテキストに含めるか
* どの情報をメモリとして保持するか
* どの情報を要約・圧縮するか
* どのタイミングで過去の情報を取り出すか
* ユーザーやセッションごとに記憶をどう分離するか

また、Agentic RAGを業務で使うためには、AIが必要なデータやナレッジを理解・検索・活用しやすい形に整えることも重要です。これは、AI Readyな環境を整えることにもつながります。

つまり、Agentic RAGでは、単に検索精度を上げるだけではなく、ナレッジの整備、メタデータ付与、アクセス制御、メモリ設計、コンテキスト設計を組み合わせて考える必要があります。

### 4.5 MCPとGatewayにより、エージェントとツールの接続が標準化される

AIエージェントは、LLM単体では業務を完結できません。

データベース、SaaS、社内API、分析基盤、チケット管理、ワークフローなど、さまざまなツールと接続する必要があります。

一方で、AIエージェントとツールの接続を個別に実装していくと、エージェント数とツール数の掛け算で統合が複雑になります。

そこで重要になるのが、MCP（Model Context Protocol）やAgentCore Gatewayのような標準化された接続レイヤーです。

![](https://static.zenn.studio/user-upload/b8b9435b0e75-20260720.png)  
*出典：AWS Summit Japan 2026 ANT310「AWS Analytics MCP サーバーで実現するエージェント型データエンジニアリング」*

MCPやGatewayを使うことで、AIエージェントに対して、社内API、SaaS、MCPサーバー、Knowledge Baseなどをツールとして提供しやすくなります。

![](https://static.zenn.studio/user-upload/e520f21b3b57-20260720.png)  
*出典：AWS Summit Japan 2026 ANT301「エージェンティック AI のためのデータ活用実践ガイド」*

今後、AIエージェント開発では、個別にツール連携を作り込むよりも、**安全に、再利用可能に、標準化された形でツールを提供する**ことが重要になりそうです。

特にエンタープライズ環境では、以下のような観点が重要になります。

* 組織で承認済みのツールカタログを用意する
* ツールの説明、入力、出力、エラー処理を明確にする
* 既存MCPサーバーを再利用する
* 新規作成ツールもMCPで公開する
* ツールごとに認証認可や監査を設計する
* 危険な操作には人間承認を挟む

AIエージェントの品質は、モデルだけでなく、接続するツールの設計品質にも大きく依存します。

### 4.6 評価と可観測性がAgentOpsの前提になる

AIエージェントは、従来のアプリケーションよりも挙動が変動しやすく、単体テストだけでは品質を十分に担保しにくい領域です。

そのため、本番運用では、作って終わりではなく、実行ログ、トレース、評価結果、ユーザーフィードバックをもとに継続的に改善する仕組みが必要になります。

このような継続的な改善サイクルは、AgentOpsの重要な要素と捉えられます。

![](https://static.zenn.studio/user-upload/62fcd6343587-20260720.png)  
*出典：AWS Summit Japan 2026 AIM329「社内 AI エージェント展開の勘所 - AI の能力を組織で最大限引き出すための課題と打ち手」*

AIM342では、オブザーバビリティを初日から設定すること、エージェントの評価を自動化することがベストプラクティスとして示されています。

AIエージェントを本番運用するには、以下のような観点を継続的に確認する必要があります。

* どの入力に対して、どのツールが使われたか
* どのツール呼び出しで失敗したか
* 回答にハルシネーションが含まれていないか
* レイテンシーやコストが悪化していないか
* エージェントが想定外の操作をしていないか
* 品質が時間とともに劣化していないか
* チームやユーザーごとの利用状況はどうか

AIエージェントの本番運用では、「動いたか」だけでなく、以下のような観点で継続的に評価する必要があります。

| 観点 | 評価例 |
| --- | --- |
| 正確性 | 事実に基づいた回答になっているか |
| 一貫性 | 同じ意図の質問に安定して回答できるか |
| ツール選択 | 適切なツールを選択できているか |
| セキュリティ | 権限外の情報や操作に触れていないか |
| ガードレール | 禁止トピックや不適切な出力を抑制できているか |
| レイテンシー | 応答時間が許容範囲に収まっているか |
| コスト | ツール呼び出しやトークン消費が過剰になっていないか |
| ユーザー体験 | 業務上使いやすい回答になっているか |

AgentCore ObservabilityやEvaluationsを活用することで、AIエージェントの挙動を可視化し、評価し、改善につなげる流れを作りやすくなります。

つまり、AIエージェントの本番運用では、**観測できること、評価できること、改善できること**が前提になります。

### 4.7 AIエージェントの設計はArchitecture・Context・Toolsで考える

AIM314では、AIエージェントの挙動が不安定になる原因を、Architecture、Context、Toolsの3つの観点で整理しています。

| 観点 | 主な論点 |
| --- | --- |
| Architecture | Single Agent、Graph、Agents as Tools、Swarmなど、どの構成を選ぶか |
| Context | LLMに渡す情報が肥大化していないか、必要最小限に保てているか |
| Tools | LLMが使いやすいツール設計になっているか |

![](https://static.zenn.studio/user-upload/10d4826cda15-20260720.png)  
*出典：AWS Summit Japan 2026 AIM314「AI エージェント精度改善のポイント - Architecture・Context・Tools の設計方針」*

特に重要なのは、**自律性の高いアーキテクチャほど制御が難しくなる**という点です。

たとえば、少数ドメインのタスクであればSingle Agentでも実現できます。一方で、複数ドメインにまたがる分析では、ContextやToolsが肥大化し、挙動が不安定になる可能性があります。

その場合、以下のような設計が考えられます。

* ドメインごとにエージェントを分ける
* Graphで処理手順を事前定義する
* Orchestratorが専門エージェントをToolとして利用する
* 必要に応じてマルチエージェント構成を採用する

ただし、マルチエージェントにすれば必ず良くなるわけではありません。エージェント間の調整、デバッグ、責任分界、評価が難しくなるため、ユースケースに応じてできるだけシンプルな構成から始めることが重要です。

AgentCoreを使う場合も、まずは以下を整理するのがよさそうです。

* エージェントに任せるタスクは何か
* どの程度の自律性が必要か
* 処理手順は固定か、不定か
* 必要なContextは何か
* 使うべきToolsは何か
* どこをコードやワークフローで決定論的に制御するか

![](https://static.zenn.studio/user-upload/fced364f62e7-20260720.png)  
*出典：AWS Summit Japan 2026 AIM314「AI エージェント精度改善のポイント - Architecture・Context・Tools の設計方針」*

AIエージェントの本番化では、モデルの性能だけでなく、Architecture、Context、Toolsをどう設計するかが安定性と運用性を左右すると感じます。

## 5. まとめ

AWS Summit Japan 2026の各セッションをAmazon Bedrock AgentCoreの観点で見ると、AIエージェントは「PoCで動かす」段階から、**業務システムに安全に接続し、継続的に評価・改善しながら運用する**段階へ進みつつあると感じました。

今回整理したポイントをまとめると、以下のようになります。

* AIエージェントはPoCから本番運用の段階へ移りつつある
* 本番利用では、実行環境、認証認可、ツール接続、メモリ、評価、可観測性を一連で考える必要がある
* AgentCore Identityにより、誰が、誰の代理で、どのリソースにアクセスするのかを管理することが重要になる
* データエージェントでは、LLM層だけでなく、呼び出し層、ツール層、リソース層まで含めた多層防御が必要になる
* Agentic RAGでは、MemoryやContext Engineeringに加えて、AIが理解・活用しやすいAI Readyな環境づくりが重要になる
* MCPやGatewayにより、エージェントとツールの接続は標準化・再利用の方向へ進む
* ObservabilityやEvaluationsを活用し、AgentOpsとして継続的に改善していくことが重要になる
* Architecture、Context、Toolsの設計が、エージェントの安定性と運用性を左右する

一方で、最初からAgentCoreのすべての機能を使う必要はありません。

まずは、業務上の課題が明確で、効果を確認しやすいユースケースを1つ選び、小さく始めることが重要だと思います。たとえば、社内ナレッジ検索、問い合わせ対応、営業支援、データ分析補助など、利用者・対象データ・実行させたい操作が比較的明確な領域から始めると進めやすそうです。

そのうえで、実際に使いながら、回答品質、ツール選択、権限制御、レイテンシー、コスト、ユーザー体験を継続的に確認し、改善していく流れを作ることが重要になります。AIエージェントは一度作って終わりではなく、ログ、トレース、評価結果、ユーザーフィードバックをもとに育てていくものだと考えています。

また、AIエージェントに任せる部分と、人が判断する部分を分けることも重要です。

たとえば、情報検索、要約、候補提示、定型的なツール実行はAIエージェントに任せやすい領域です。一方で、重要な業務判断、権限の強い操作、外部への送信、取り消しが難しい処理などは、人による確認や承認を挟む設計が必要になります。

Amazon Bedrock AgentCoreは、AIエージェントを「試す」ためだけのサービスではなく、企業システムの中で安全に接続し、観測し、評価し、改善しながら運用していくための基盤と捉えると分かりやすいです。

今後、AIエージェントを業務に組み込んでいくうえでは、モデルやプロンプトだけでなく、Identity、Gateway、Memory、Observability、Evaluations、Policyなどを組み合わせ、AIと人がそれぞれ得意な役割を担える設計がますます重要になっていきそうです。
