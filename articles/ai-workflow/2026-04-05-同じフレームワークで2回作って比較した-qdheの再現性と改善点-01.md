## これまでのあらすじ

[前編](https://zenn.dev/belka_dolphin/articles/c885376e01b59d)でAI駆動開発フレームワーク **QDHE（Quality-Driven Harness Engineering）** を設計し、AI人狼ゲームのバックエンドをStep 10まで完走させた。

[続編](https://zenn.dev/belka_dolphin/articles/3702746398a933)でフロントエンドと結合したら、375件テスト全GREENなのに1回も遊べなかった。9件のバグを分析し、5つの改善提案を出した。

今回は、**改善提案を意図的に反映せず**、同じQDHEフレームワークでもう一度AI人狼ゲームをゼロから作った。ただし1点だけ条件を変えた——「バックエンドからUIまで一貫して作成し、ユーザーが遊べるものを作って」というプロンプトにした。前回は「バックエンドを作って」→「フロントエンドも作って」と2段階だったのを、1回で全部やらせた。

2回の実行結果を比較することで、QDHEの再現性と、まだ残っている構造的な弱点を明らかにする。

## 2回の実行結果を並べる

まず数字で全体像を掴む。

| 指標 | 1回目 | 2回目 |
| --- | --- | --- |
| テスト件数 | 375 | 204 |
| テスト/ソース比 | 2.73倍 | 1.13倍 |
| ソースコード行数 | 3,480 | 3,345 |
| ドメイン層ミューテーション | 70.95%（全体） | 88〜93%（ドメイン層） |
| 統合層ミューテーション | （未分離） | 33% |
| UXバグ（実プレイ発覚） | 9件 | 5件 |
| ロジックバグ | 0件 | 0件 |
| 凍結解除回数 | 1回 | 1回 |
| 開発時間 | 1h(BE) + v2別途 | 3h45m（全部込み） |
| FMEA故障モード | 48件 | 44件 |
| EARS要件数 | 約40件 | 53件 |

### 一貫して強かったところ

**ロジックバグが両方とも0件。** これがQDHEの最大の成果だ。2回やって2回とも、ゲームの勝利判定、投票集計、役職配分、占い結果、夜行動処理——純粋なドメインロジックにバグがない。FMEAで故障モードを洗い出し、テストを先に書いて凍結し、実装はテストを通すためだけに書く。この構造が効いている。

**凍結→凍結解除のガバナンスが機能。** 両回とも凍結解除は1回で、正当な理由（テスト設計バグの修正）でのみ実施された。AIが勝手にテストを書き換えて通す、という事態は起きていない。

**トレーサビリティの保持。** 要件→設計→FMEA→テスト→実装の連鎖が両回とも保持されている。FMEAのRPN上位が全てP0テストに落ちている点は2回目のほうがより明確で、前回の経験が暗黙的にプロンプトの質に反映されている可能性がある。

### 改善された部分

**テスト件数が半減したのに、ドメイン品質が上がった。** 1回目は375件でミューテーション70.95%（全体）。2回目は204件でドメイン層88〜93%。テストの数ではなく質が品質を決める、という前編の教訓が数字で裏付けられた。

**ミューテーションスコアの層別評価。** 1回目は全体一律で70%閾値を設けていた。2回目はドメイン層（純粋関数）と統合層（WebSocket、GameController等）を分けて計測し、ドメイン層90%超 / 統合層33%という切り分けができている。テストを追加すべき場所が明確になる。

**Property-Based Testingの導入。** 2回目ではfast-checkによるプロパティベーステストが含まれている。「役職配分で全プレイヤーにユニークな役職が割り当てられる」のような不変条件を性質として記述し、ランダム入力で検証する。境界値テストを手動で列挙するより効率的にエッジケースを発見できる。

**UI込みの一貫実行。** 1回目はバックエンド→フロントエンドと2段階で実行し、結合で9件のバグが出た。2回目は最初から一貫して実行したことで、型定義の共有やWebSocketメッセージ型の整合がStep 6（テストコード作成）の時点で確保された。

### 同じ穴に落ちたところ

**UXレベルのバグが再発した。** 2回目でも以下のバグが実プレイで発覚した。

* 昼フェーズで人間の発言を待たずに投票フェーズへ進む
* 夜フェーズでPhaseChanged(Night)をブロードキャストしない
* 人間が占い師/狼でも夜行動を待たない
* ゲーム終了後に再開できない

続編のBug 4「ゲームが一瞬で終わる」とほぼ同じ構造だ。「人間の入力を待つ」という要件が曖昧なまま設計・実装が進み、テストは「エラーなく完走する」ことしか検証していない。

**これが最も重要な発見だ。** 続編で改善5「ユーザーストーリー完遂条件を要件定義に明記する」を提案したが、**改善提案はレポートに書いただけで、フレームワーク本体のテンプレートには反映していなかった。** だから2回目の実行でも同じ穴に落ちた。

## 改善内容 — フレームワークに焼き込む

続編で出した5つの改善提案と、2回目の実行で新たに見えた問題を統合して、QDHEフレームワーク本体に反映する改善を整理する。

### 改善1: 要件テンプレートに「ユーザーストーリー完遂条件」を必須セクション化

**問題**: 2回連続で「人間が操作できない」バグが実プレイまで発覚しなかった。  
**原因**: 要件定義に「ユーザーが1ゲーム通して遊ぶ体験」がシナリオとして明文化されていない。

**対策**: 要件テンプレート（`requirements-template.md`）に以下のセクションを必須追加する。

```
## ユーザーストーリー完遂条件
 
以下の操作シナリオを人間が1回以上実行できること:
1. [操作1: アプリを起動する]
2. [操作2: 主要な入力を行う]
3. [操作3: 結果を確認する]
 
※ 各操作で「人間の入力を待つ」タイミングを明記すること
```

Plannerエージェントは、このセクションが埋まらない限りStep 1を完了報告できない。

### 改善2: FMEAテンプレートに「環境差異」「フロントエンド配線」カテゴリを標準搭載

**問題**: 1回目で環境依存バグ3件、フロントエンド配線バグ4件がFMEAスコープ外だった。  
**原因**: FMEAの故障モードテンプレートがアプリロジックに限定されていた。

**対策**: FMEAテンプレート（`fmea-template.md`）に以下のカテゴリを追加する。

```
## 必須分析カテゴリ
 
### ENV: 環境差異
- ENV-01: OSごとのパス表現差異（file:// URL、パス区切り）
- ENV-02: モノレポcwd差異（pnpm/npm workspace実行ディレクトリ）
- ENV-03: 設定値ハードコード vs 環境変数の不整合
- ENV-04: ツールチェーン固有動作（tsx vs ts-node、Vite vs webpack）
 
### FE-WIRE: フロントエンド配線
- FE-WIRE-01: Context Provider未設定（useContextがnullでthrow）
- FE-WIRE-02: フェーズ変更時のUI状態リセット漏れ
- FE-WIRE-03: コンパイル通過 + ランタイムクラッシュ
 
### ST: 状態遷移クロス分析
- ST-01: 複数の状態機械の直積で未定義の組み合わせがないか
```

Analystエージェントは、各カテゴリに対して最低1件の故障モードを分析しない限りStep 3を完了報告できない。

### 改善3: テストリストテンプレートにスモークテストとE2Eを必須化

**問題**: 2回とも「テスト全GREEN → 実際には動かない」が発生した。  
**原因**: テストが個別モジュールの正しさしか検証しておらず、統合動作とユーザー操作のテストが0件だった。

**対策**: テストリストテンプレート（`test-list-template.md`）に以下を必須項目として追加する。

```
## 必須テスト（スキップ不可）
 
### SMOKE: スモークテスト
- SMOKE-01: 実プロセス起動 → ヘルスチェック応答
- SMOKE-02: フロントエンド描画 → 初期画面が例外なく表示
- SMOKE-03: 主要ユースケース1回実行 → 期待する応答
 
### E2E: エンドツーエンド
- E2E-01: ユーザーストーリー完遂条件の全操作を1回実行
```

Plannerエージェントがこれらを含まないテストリストを提出した場合、Orchestratorが差し戻す。

### 改善4: ミューテーションスコアをリスクベースの層別基準に変更

**問題**: 1回目は全体一律70%閾値で、250件のテスト追加の大半が低リスク箇所の境界値テストに費やされた。  
**原因**: 全体スコアを均一に上げようとすると、FMEA高リスク箇所も低リスク箇所も同じコストをかけることになる。

**対策**: ミューテーションスコアの閾値を層別にする。

```
## ミューテーションスコア閾値
 
| 層 | 閾値 | 対象 |
|----|------|------|
| ドメイン層（純粋関数） | 85% | FMEA RPN≥80 の実装ファイル |
| 統合層（Controller、Server） | 50% | WebSocket、API層 |
| UI層（React コンポーネント） | 計測のみ | スモークテストで補完 |
```

2回目の実行でドメイン層88〜93% / 統合層33%という実測値が出ているので、この基準は現実的だ。ドメイン層に集中投資し、統合層はスモークテストとE2Eで補完する。

### 改善5: Step 6完了条件に型チェック必須化

**問題**: 2回目でWriterが型チェック未実行のまま完了報告し、コンパイルエラー4件が残った。  
**原因**: Step 6の完了条件が「テストファイルの生成」だけで、コンパイル可能性が含まれていなかった。

**対策**: Step 6の完了条件に `tsc --noEmit` の成功を追加する。

```
## Step 6 完了条件
 
1. 全テストファイルが生成されている
2. `npx tsc --noEmit` がエラー0件で通過する
3. 全テストが「Not implemented」エラーで失敗する（実装前の状態確認）
4. SHA256ハッシュが生成・記録されている
```

### 改善6: Reviewerの観点にモック適合性チェックを追加

**問題**: 2回目でテスト設計バグ2件が発覚。MockRandomProvider（恒等シャッフル）で統計テストをやろうとしていた。  
**原因**: Reviewerがモックの決定性と、テストが要求する非決定性の整合をチェックしていなかった。

**対策**: Step 5の観点レビューチェックリストに以下を追加する。

```
## モック適合性チェック
 
- [ ] モックの決定性（固定値を返すか、ランダムか）がテストの前提条件と整合しているか
- [ ] 統計的テストに決定的モックを使っていないか
- [ ] モックのフォールバック挙動がテストシナリオの前提を壊さないか
```

2回目のAnalystがこの観点を自発的に提案して`.harness/decisions.md`に記載していた。これをフレームワーク標準に昇格させる。

## 改善のまとめ — テンプレートに焼き込む

ポイントは、**改善がテンプレートに入ることで、次の実行から自動的に適用される**ということだ。プロンプトに「前回の反省を踏まえて」と書く必要はない。テンプレートがフレームワークの「記憶」として機能する。

## QDHEの現在地

3本の記事を通じて、QDHEの強みと弱みが明確になった。

**強み（2回とも一貫して機能）**:

* FMEAによる故障モード事前洗い出し → ロジックバグ0件
* SHA256ハッシュ凍結 → AIのテスト改ざん防止
* ミューテーションテスト → テストの殺傷力検証
* トレーサビリティ → 要件からテストまでの追跡可能性

**弱み（2回とも破れた）**:

* UXレベルの要件が曖昧なまま進む
* 統合・E2Eテストの欠如
* テスト環境と実行環境の乖離

**今回追加した改善**:

* 要件テンプレートにユーザーストーリー完遂条件
* FMEAテンプレートに環境差異・配線・状態遷移カテゴリ
* テストリストにスモークテスト・E2E必須化
* ミューテーションをリスクベース層別評価に変更
* Step 6完了条件にtsc必須化
* Reviewerにモック適合性チェック追加

次にQDHEパイプラインを回すとき、これらの改善が自動的に適用される。同じ穴に3回落ちることはないはずだ。

フレームワークは、失敗を記憶して進化するものだ。2回失敗して、ようやくその「記憶の仕組み」ができた。
