「オントロジー（Ontology）」という言葉を、AI ネイティブな組織設計や Forward Deploy Engineer（FDE）の文脈で耳にする機会が増えました。とはいえ、RAG とどう違うのか、ナレッジグラフと何が違うのか ── きちんと整理できている人は意外と少ない印象です。

背景としては、Palantir Foundry の中核技術として、そして RAG の限界を埋める「知識構造」として、この概念が新しい意味を持って再評価されています。Palantir が利益を4倍に伸ばした背景に、このオントロジー基盤の参入障壁があるという指摘も見られます。

<https://x.com/s3atoshi/status/2051451582995399030>

RAG で議事録を要約するところまではできる。しかし「今週納期のタスクで、まだ進行中のまま、レビュアーが決まっていないものを全部出して」のような問いに来ると、急に怪しくなります。情報が Notion / Slack / GitHub に散らばっていて、関係をまたいで意味を保つ必要があるからです。

本記事の前半では、オントロジーの基礎から BI / RAG との違い、Palantir Foundry の動的オントロジーまでを整理します。後半では、ここからもう一段踏み込んで、自分が考えている3つの主張を書きます。

* **オントロジーは AI エージェントの「世界モデル」である**（認識）
* **オントロジーは AI の「行動仕様」と「権限」を統合的に定義する**（行動）
* **オントロジーは AI と人間が共通言語で動くための「業務 OS」になる**（組織）

この3点が、AI エージェントが本格的に業務に入ってくる時代に、設計の中心になっていくと考えています。

## オントロジーとは何か

オントロジーには形式的な学術定義もあるのですが、本記事ではこう言い換えて扱います。

> オントロジーは、データを「現場の人間が使う言葉と関係性に揃え直した、意味の地図」である。

これは厳密な定義というより、設計実装の文脈で扱いやすくするための言い換えです。

少し抽象的なので、誰でもイメージしやすい例で見てみます。タスク管理ツール（Notion / Linear / Jira など）で扱う概念を並べてみると、こうなります。

* メンバーはチームに所属する
* チームはプロジェクトを担当する
* プロジェクトは複数のタスクに分解される
* メンバーはタスクを担当する
* タスクはステータス（未着手・進行中・完了）を持つ
* プロジェクトには期限がある

書かれているのはタスクのタイトル一覧でも、メンバーの名簿でもありません。書かれているのは「メンバー」「チーム」「プロジェクト」「タスク」といった **モノ（Object）** と、「所属する」「担当する」「分解される」といった **コト（Relation）** の定義です。

つまりオントロジーは、情報の中身ではなく、**情報をどう理解するかの設計図** なのです。

データベースが「テーブルとカラム」でデータを表現するのに対して、オントロジーは「意味を持つオブジェクトとその関係性」でデータを表現する、と覚えてください。

## データのバラバラ問題

なぜわざわざ「設計図」を作るのか。理由は、現実の組織データが派手にバラバラだからです。

AI 開発組織を例に取ると、データの居場所はだいたいこうなります。

* GitHub には `issue-1248` という Issue がある
* Slack には `#product-dev` のスレッドがある
* Notion には `2026Q2-roadmap` というページがある
* Linear には `PROJ-512` というチケットがある

これは「ツールの言葉（Issue ID、チャンネル名、ページタイトル、チケット ID）」であって、「組織の言葉」ではありません。メンバーが「あの改善要望どうなった？」と話している言葉と、ツールの中の `issue-1248` がつながっていないのです。

オントロジーはここを翻訳します。

| ツールの言葉 | 組織の言葉 |
| --- | --- |
| `issue-1248` | 「機能改善要望」 |
| `#product-dev` のスレッド | 「設計議論」 |
| `2026Q2-roadmap` ページ | 「リリース計画」 |
| Linear の `PROJ-512` | 「進行中タスク」 |

そしてさらに重要なのが、**関係性の定義** です。

* 「機能改善要望」は「リリース計画」に紐づく
* 「リリース計画」は「進行中タスク」に分解される
* 「設計議論」は「機能改善要望」を生み出す

モノとコトを定義することで、ツールの中に散らばっていた断片に **組織側の意味の構造を貼り直す** ことができます。これが、いまデータプラットフォームの文脈で「オントロジー」と呼ばれているものです。

## BI ツールとの違い

ここで多くの人が疑問に思う「BI ツール（ダッシュボード）と何が違うの？」を整理します。

従来の BI ツールは、ETL でデータを整え、グラフで眺めて、気づきを得るところまでが守備範囲でした。「Agent のエラー率が上がっている」と気づいたとして、実際にモデルを切り替えるには別の管理画面にログインし直す必要がある。データを見る場所と仕事をする場所が分断されているわけです。

オントロジー基盤のアプローチは、ここが根本的に違います。データが「Agent」「ユーザー」「タスク」というオブジェクトとして画面上に存在しており、そのオブジェクトに対して **直接アクション（Write-back）が可能** です。「Agent のエラー率が閾値を超えた」と気づいたら、その Agent オブジェクトをクリックして、その場でフォールバックモデルに切り替える ── という動き方ができる。

| 観点 | 従来の BI ツール中心 | オントロジー中心（Palantir 等） |
| --- | --- | --- |
| 主な目的 | 検索と分析 | 意思決定とアクション |
| ユーザー体験 | 気づいて、別画面へ移動 | 気づいて、その場でアクション |
| データの流れ | 読み取り専用（Read Only） | 書き込み可能（Read / Write） |

この「見る」と「やる」の分断という観点は、channnnsm 氏の Zenn 記事で簡潔に整理されていて参考になります。

<https://zenn.dev/channnnsm/articles/035b612ac973ab>

## RAG との違い

もうひとつ、最近よく比較されるのが RAG（Retrieval-Augmented Generation）です。

RAG をひとことで言うと「**質問に関係ありそうな文書を検索して、LLM に読ませて回答させる仕組み**」。一方、オントロジーは情報そのものではなく、意味と関係性の設計図でした。

| 観点 | RAG | オントロジー |
| --- | --- | --- |
| 主目的 | 文書検索 + 回答生成 | 知識構造の定義 |
| 扱うもの | テキスト、PDF、Notion、Slack 等 | 概念・関係・属性・制約 |
| 強み | 導入が速い。既存文書をそのまま使える | 意味関係を厳密に扱える |
| 弱み | 関係性や整合性に弱い | 設計コストが高い |

RAG は「探して読ませる」仕組み、オントロジーは「意味と関係を定義する」仕組みです。

冒頭の問い ── 「今週納期のタスクで、まだ進行中のまま、レビュアーが決まっていないものを全部出して」── に RAG 単体で答えるのはかなりつらい。納期は Notion、進捗ステータスは別ツール、レビュアー情報は GitHub や Slack、という具合に情報が散らばっていて、それを **関係でつなぐ** 必要があるからです。

実務での共通理解になりつつあるのは、**「RAG だけでも、オントロジーだけでもダメ。両方を組み合わせる」** ということです。構造は Knowledge Graph で、文脈は RAG で。

## Palantir Foundry の動的オントロジー

ここで Palantir Foundry に触れておきます。以下、Palantir 公式の Architecture Center「The Ontology system」ページを参照しながら整理します。

<https://www.palantir.com/docs/foundry/architecture-center/ontology-system>

Palantir が20年以上かけて磨いてきた「動的オントロジー（Dynamic Ontology）」は、データ・ロジック・アクション・セキュリティの4つを一体で扱う、業務 OS そのものと言えるものです。

### 4軸の統合（Data / Logic / Action / Security）

公式ドキュメントは、オントロジーが何をしているかを端的にこう書いています。

> The Ontology models decisions through the four-fold integration of data, logic, action, and security.
>
> （出典：Palantir Architecture Center「The Ontology system」）

オントロジーは、データ・ロジック・アクション・セキュリティの4要素の統合として、企業の意思決定をモデル化する ── これが Palantir の言い分です。それぞれの軸を整理するとこうなります。

| 軸 | 内容 |
| --- | --- |
| Data | ERP / CRM / 自社開発の業務システム / 産業データベース / 地理空間データ / リアルタイムセンサー / 文書ストア など、あらゆるソースを統合し、意味のある Object / Property / Link に変換する |
| Logic | アクションや属性計算の裏側にあるロジック。単純なビジネスルールから、機械学習モデル、LLM 関数、複数エンジンをまたぐ多段オーケストレーションまで含む |
| Action | 単純なトランザクションから、複数システムをまたいで本番に書き戻す多段更新までを含む「動詞」。公式の言葉でいうと「Semantics（名詞）と Kinetics（動詞）が揃ってはじめて意思決定がモデル化できる」 |
| Security | 行・列レベルの権限、ユーザーやプロジェクト単位のスコープ、AI エージェント単位の制約まで、全アクセスに対する細粒度ポリシー |

ここで重要なのは、Data・Logic・Action・Security が **層ごとに別管理されているのではなく、ひとつのオントロジー基盤の中で一体として扱われている** 点です。AI 時代に向けて Action と Security を後から接ぎ木しようとして詰む、というのが各社が直面している壁ですが、Palantir はこれを最初から統合設計しています。

### 業界横断の使われ方

公式ドキュメントは、Foundry のオントロジーが実装されている業界の例として、こう挙げています。

> An airline might model flights, aircraft, crew manifests, scheduling optimizers, and other fragmented enterprise assets into their ontology, to power day-of flight operations and longer-range planning.
>
> A hospital system might instead model patients, nurse schedules, medical supplies, bed capacities, and other elements that often shift in real-time...
>
> （出典：Palantir Architecture Center「The Ontology system」）

整理するとこうなります。

| 業界 | モデル化される Object 例 |
| --- | --- |
| 航空 | フライト / 機材 / クルー編成表 / スケジュール最適化 |
| 病院 | 患者 / 看護師シフト / 医療資材 / 病床容量 |
| 軍事 | 前方展開部隊の即応情報 / 偵察 / 目標選定 |
| 医療製造 | 製造工場 / 作業指示 / 顧客情報 / 入出荷パッケージ |

業界によって扱う Object はまったく違いますが、**「現場の業務概念をそのまま Object 化して、Action と Security を一体で運用する」** という構造は共通です。製造業の例で「ロット」「品質検査」「出荷」と並んでいたのと同じ抽象度のものが、各業界に存在する、ということです。

公式は、オントロジーをただの「意味の薄い層」ではないと明言しています。

> The fourfold integration and operationalization of data, logic, action, and security cannot be accomplished with a thin "semantic layer" or a monolithic design.
>
> （出典：Palantir Architecture Center「The Ontology system」）

その代わりに、3つのコンポーネントから成る多モーダルなシステムとして説明されています。

| コンポーネント | 役割 |
| --- | --- |
| Ontology Language | Object / Link / Property といった意味側と、Action / Automation・実際のロジックを記述する記述言語 |
| Ontology Engine | 高スケールの SQL クエリ、状態変化のリアルタイム購読、原子的なトランザクション、Change Data Capture などの読み書きアーキテクチャ |
| Ontology Toolchain | OSDK（Ontology SDK）と DevOps ツールチェイン。実際の業務アプリ（自動車組立・山火事対応・海軍ロジスティクスなど）を構築する基盤 |

ただの ER 図ではない、ということです。「業務概念の定義」と「それを動かす実行基盤」と「その上にアプリを建てる SDK」が、一体のシステムとして提供されています。

### 動的（Dynamic）の意味

このオントロジー基盤が、顧客の業務理解の進展に応じて **動的に** 更新されていく。これが「動的」たる所以です。データを移すのではなく、データの意味の層だけを動かす、と表現するとイメージしやすいかもしれません。

Palantir はこの動的オントロジーをめぐって複数の特許を取得しており、それを意識した日本企業のベンチマーク特許開発も進んでいるという観察もあります。

<https://x.com/TakefumiY/status/2052678321931243869>

<https://x.com/yasu_xaion/status/2053618939192905990>

ここまでが、いま語られている「オントロジー」の整理です。ここから先は、自分が考えている AI エージェント時代のオントロジーの捉え方を3つ書きます。

## 主張1：オントロジーは AI エージェントの「世界モデル」である

ここまでオントロジーを「データを人間の言葉で再定義した地図」「BI を超える基盤」「RAG を補完する Graph」と説明してきました。これらは間違っていません。ただ、AI エージェント時代に入って、自分はもう一段大きな意味づけが必要だと考えています。

それを言い表すのに一番近い言葉が「世界モデル（World Model）」です。もともとは強化学習やロボティクスで使われてきた用語で、エージェントが「世界はいまどうなっていて、自分が動くとどう変わるか」を内部表現として持っていることを指します。LLM の弱点として Yann LeCun が繰り返し指摘してきたのも、この世界モデルの欠如です。LLM は言語の上ではなめらかに振る舞えるけれど、世界の構造そのものは持っていない、というやつです。

自分はこの概念を、物理的な世界だけでなく、組織や業務といった抽象的な世界にも拡張できると考えています。そう見ると、オントロジーが世界モデルそのものになります。

> オントロジーは、AI エージェントが「世界」をどう見ているかの設計図である

人間は世界を理解するために言語を持っています。「机」「椅子」「コップ」という単語と、それらの間の関係性のおかげで、世界を整理して扱える。言語のないところに、複雑な思考は立ち上がりません。

AI エージェントにとっての言語が、オントロジーです。

LLM は **言語の中で完結** しています。これは強力ですが、それだけでは「世界の構造」を持ちません。LLM 単独のエージェントは「文書を要約する」「会話する」までは強いが、「組織を動かす」「資産を運用する」「業務を実行する」となると急に弱くなる。原因は、LLM が世界の中の **意味の関係** を直接持っていないからです。

ここにオントロジーが入ると、何が起きるか。

エージェントは LLM で **推論** はできるが、推論するための **対象の構造** はオントロジーが提供する。両者がそろってはじめて、エージェントは世界に対して意味のあるアクションを取れます。

Palantir Foundry の動的オントロジーは、これを企業の業務領域に対して実装したものです。製造業の「ロット」「品質検査」「出荷」というオブジェクトと関係性が、AI が扱える形で揃っているからこそ、AI による現場最適化が成立する。

逆に言うと、これからの AI エージェントの能力は、モデルサイズや推論性能よりも、**それが参照できるオントロジーの解像度** で決まる、というのが自分の仮説です。同じ GPT を使っていても、世界の構造をしっかり持つエージェントと、持たないエージェントでは、できることがまるで違う。

## 主張2：オントロジーは AI の「行動仕様」と「権限」を統合的に定義する

ここまでオントロジーを「世界を見るための地図」「文書をつなぐための関係」として語ってきました。これは半分しか合っていないと考えています。

Palantir Foundry の動的オントロジーをよく見ると、Object（モノ）と Relation（コト）の上に、もうひとつ大事な層が定義されていました。

> **アクション**：ロットを出荷する / 品質検査を再実施する / 不良品ロットを保留する

このアクション層こそが、検索基盤や BI を超える本当の差分です。オントロジーは「状況を表示する」だけでなく、**そこに対して何ができるか** を一緒に定義している。さらにそれと **不可分の形で「誰が何をやっていいか」のポリシー** も定義できる。Action と Policy はセットで扱われてはじめて、AI エージェントを業務に投入できる構造になります。

### Action として定義する

LLM のツールユース、Function Calling、MCP のような仕組みは「AI に道具を渡す」発想です。ただ、道具を渡すだけでは AI はどの道具を使えばいいかわからない。文脈がないからです。オントロジーは、これを構造的に与える仕組みです。

* どの Object に対して
* どの Action が実行可能で
* 実行すると関係性がどう変わるか

を、まとめて与えられる。タスク管理を例にするとこうなります。

| Object | 実行可能な Action |
| --- | --- |
| タスク | ステータス変更 / 担当者変更 / 期限延長 / レビュー依頼 |
| プロジェクト | アーカイブ / フェーズ更新 / メンバー追加 |
| メンバー | 役割変更 / 稼働率更新 |

### Policy として定義する

ここから抜けがちなのが「誰がどの Action を実行していいか」のレイヤーです。AI エージェントの導入が止まる原因のかなりの部分は、ここに対する答えがないことだと感じています。

個別実装の if 文や別の RBAC 設定で散らかすのではなく、**Policy 自体をオブジェクトとしてオントロジーに含める** のがスマートです。

| Object | 主な属性 |
| --- | --- |
| Agent | id / 所属チーム / 能力範囲 |
| Policy | 対象 Agent / 許可 Action / 禁止 Action / 承認ステップ |

これがあると、AI エージェントは「自分がいま操作できる Object はどれか」「どの Action が許可されているか」「どの Action は人間の承認を必要とするか」を構造的に把握できる。

自然文の社内規定を RAG で読ませる方法もありますが、その都度のテキスト解釈は事故のもとです。ポリシーをオントロジーの上に乗せれば、AI はそれを **検索ではなく構造として** 参照できます。

オントロジーは「AI が世界を見るための地図」であると同時に「AI が世界に働きかけるための API カタログ」であり、その API カタログには **権限と承認フロー込みで** 仕様が書かれている。これが AI エージェント時代における Action 設計の現実的な形だと考えています。

## 主張3：オントロジーは AI と人間が共通言語で動くための「業務 OS」になる

3つめの主張は、もう少し組織寄りの話です。オントロジーは AI のためだけでなく、**人間と AI が同じ言葉で業務を語れるようにする土台** になります。これを自分は「業務 OS」と呼んでいます。

これまで多くの組織では、業務概念があちこちに散っていました。

* 社内 Wiki に説明文がある
* ER 図にデータ構造がある
* 業務フロー図にプロセスが書かれている
* Slack の社内規定チャンネルにルールが流れる
* スプレッドシートに権限表がある

それぞれが別の文書として独立していて、しかも更新タイミングがバラバラ。新人が入ってきたら全部を読み合わせて理解する、というのが現実でした。

オントロジーが効くのは、これらを **単一の構造に集約** できる点です。

| いままで散らかしていたもの | オントロジー上での位置 |
| --- | --- |
| 用語集（Wiki） | Object 名とその定義 |
| ER 図 | Object とその Property |
| 業務フロー | Relation と Action の連鎖 |
| 権限表 | Policy オブジェクト |
| 役割マトリクス | Agent / Member の属性 |

これがあると、人間が業務を学ぶ場所と、AI が業務を理解する場所が **一致** します。新人エンジニアが読むドキュメントと、AI エージェントが参照するスキーマが、同じ実体を別の表示形式で見ているだけ、という状態に近づく。

これは細かい話ではなく、組織にとっては大きな変化です。

* ルールが変わったら、修正箇所は一箇所
* 人間と AI が同じ業務語彙で会話できる
* 新人もエージェントも、入ってきた瞬間に「何ができて何ができないか」を構造的に把握できる
* 業務の透明性が上がり、意思決定の根拠も追跡できる

「AI が組織を動かす」のではなく、「組織のルールを AI と人間が共通の言語で読める形に整え、その上で両者が動く」というアプローチです。前者は不透明で危ういが、後者なら検証可能で、人間も AI も同じ構造を共有できます。

オントロジーは検索精度向上のためのテクニックではなく、**組織が AI と一緒に動くための土台** になっていく。これが3つ目の主張です。

## 実装パターンと最小スキーマ

ここまでで主張の3本柱は終わりです。最後に少しだけ実装の話を書いておきます。

RAG + オントロジーの基本構造はこうです。

取り込みは2系統に分岐し、質問時には Query Router がベクトル検索とグラフ検索を振り分けます。

最小スキーマなら Postgres + pgvector で十分です。Neo4j のような専用 Graph DB は、関係を多段にたどる複雑なクエリが頻発するようになってから検討すればいい。

schema.sql

```
-- モノ
CREATE TABLE entities (
  id UUID PRIMARY KEY,
  type TEXT NOT NULL,                -- Agent / Wallet / Contract / Task など
  canonical_name TEXT NOT NULL,
  properties JSONB,
  created_at TIMESTAMP DEFAULT now()
);

-- 別名（名寄せ用）
CREATE TABLE aliases (
  id UUID PRIMARY KEY,
  entity_id UUID REFERENCES entities(id),
  alias TEXT NOT NULL                -- OpenClaw / openclaw / OC Agent など
);

-- コト
CREATE TABLE relations (
  id UUID PRIMARY KEY,
  from_entity_id UUID REFERENCES entities(id),
  relation_type TEXT NOT NULL,       -- OWNS / EXECUTES / SIGNS など
  to_entity_id UUID REFERENCES entities(id),
  properties JSONB,
  confidence FLOAT,
  source_chunk_id UUID
);

-- RAG 用の文書チャンク
CREATE TABLE chunks (
  id UUID PRIMARY KEY,
  document_id UUID,
  text TEXT,
  embedding VECTOR(1536),
  metadata JSONB
);

-- 文書とエンティティの紐づけ
CREATE TABLE evidence_links (
  id UUID PRIMARY KEY,
  entity_id UUID REFERENCES entities(id),
  chunk_id UUID REFERENCES chunks(id),
  relation_id UUID REFERENCES relations(id)
);
```

これだけで、最小の RAG + オントロジー基盤は組めます。

## 実装で意識したい5つのこと

最後に、実装に取り組むときの作法を5つ整理します。基本の論点は channnnsm 氏の Zenn 記事でも整理されていますが、AI Agent 開発の現場に置き換えるとどう効くかという視点で書きます。

### ① テーブルではなく「モノ」で考える

`tbl_users` / `tbl_user_wallets` / `tbl_kyc_status` を別テーブルに分けるのが DB 脳。これらを束ねて「ユーザー」という1つのオブジェクトにするのがオントロジー脳です。開発メンバーが「あの Agent の挙動どうなってる？」と聞くなら「Agent」オブジェクトが必要、というぐらいシンプルです。

### ② 見るだけでなく「アクション」を設計する

「この異常検知が出たら、次に何をする？」を必ず一緒に設計する。フォールバックモデル切替、担当者の自動アサイン、ステータスの一括更新 ── Write-back のアクションまで含めないと、ただのリッチなダッシュボードで終わります。

### ③ AI のための説明文（Semantics）をサボらない

`field_a` のような匿名的な名前を避け、`is_agent_active` のような意味のある名前にする。Description には「Agent の実行回数」とだけ書くのではなく、「過去30日間の成功実行回数。失敗とリトライは含まない」のように、ロジックの前提まで書く。新入社員に伝えるつもりで書くと、AI の精度が大きく変わります。

### ④ 名寄せ（Entity Resolution）が技術的な肝

`OpenClaw` / `openclaw` / `Open Claw` / `OC Agent` を同じプロダクトとして扱う仕組みが必要です。alias テーブルと Entity Resolver を組み合わせて運用します。

```
def resolve_entity(entity):
    # 1. 完全一致を探す
    candidates = search_existing_entities(entity.name, entity.type)
    if exact_match(candidates):
        return candidates[0]

    # 2. 文字列類似度が高いものを探す
    if high_similarity_match(candidates):
        return candidates[0]

    # 3. 別名テーブルを引く（OC Agent → OpenClaw）
    if alias_match(entity.name):
        return alias_table[entity.name]

    # 4. 見つからなければ新規登録
    return create_new_entity(entity)
```

名寄せを後回しにすると、見た目はカッコいい Knowledge Graph ができても、実務検索には使えない代物になります。

### ⑤ スモールスタートでユースケースから作る

「組織全体の完璧なオントロジー」を最初から設計しようとすると失敗します。「来月リリース予定の Agent のリスク確認」のような具体的な問いに絞って、必要なオブジェクトだけ作る。動いて成果が出たら、隣の領域に拡張する。最初の Entity は 8〜10 種類で十分です。

## まとめ

オントロジーは、データを人間の言葉と関係性で定義した地図です。BI ツールが「見る」までを担うのに対して、オントロジーは「やる」までを含む。RAG が文書を探す仕組みであるのに対して、オントロジーは意味を定義する仕組み。実務では両者を組み合わせるのが強い、というのが現代的な共通理解です。

ただし AI エージェントの時代に入ったいま、オントロジーは別の意味を持ち始めていると考えています。本記事で書いた3つの主張に集約すると、こうなります。

* オントロジーは AI エージェントの「世界モデル」である。エージェントの能力は、参照できるオントロジーの解像度で決まる
* オントロジーは AI の「行動仕様」と「権限」を統合的に定義する場である。Action と Policy はセットで扱われてはじめて、AI を業務に投入できる
* オントロジーは AI と人間が共通言語で動くための「業務 OS」になる。散らかった社内資料を単一の構造に集約し、人間も AI も同じ業務語彙で会話できるようになる

「データをどう貯めるか（Data Lake）」から「データをどう定義し、AI に理解させるか（Ontology）」へ。さらに「AI と人間が共通言語で動くための業務 OS」へ。オントロジーが担う役割は、確実に大きくなってきています。

Komlock lab でも、AI エージェントを業務に組み込むための最小オントロジーの設計を進めています。続きはまた別の記事で書きます。
