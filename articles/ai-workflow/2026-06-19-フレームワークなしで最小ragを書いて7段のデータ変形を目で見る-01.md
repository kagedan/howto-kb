## はじめに

RAG は LangChain や LlamaIndex を使えば数十行で動く。便利だが、便利さの代償として「各段で何が起きているか」がほぼ全部抽象化の裏に隠れる。`VectorStore.from_documents(...)` の一行の内側で、テキストがどう刻まれ、どんな形のベクトルになり、何と何の距離を測って上位を引いているのか。動くものは作れても、挙動が崩れたときに当たりをつけられない。

そこで、フレームワークを一切使わず素の Python で RAG を1本通した。ingestion → chunking → embedding → index → retrieval → context組み立て → generation の **7段**を順に実行し、各段の境界で中間データを `print` して「データがどの段でどう変形するか」を目視する。検索のコサイン類似度も numpy を使わず手書きにして、計算そのものを露出させた。

目的は本番品質の RAG を作ることではなく、**自分の言葉で各段を説明できるようになること**。この記事はその学習ログをまとめたもの。

リポジトリ構成はこう。

```
rag.py     7段を順に実行する単一スクリプト
corpus/    トイ・コーパス（英語）。事実が離散的で一意な3文書
NOTE.md    学習履歴（疑問・観察・ゴール）
```

生成は Claude（Anthropic Messages API）、埋め込みは Voyage AI を使う。Anthropic は embeddings API を提供していないので、ベクトル化は外部に出す形になる。

コード一式はこちら（この記事のコード引用はすべてここから）。

<https://github.com/yut496/rag-lab/tree/main/minimal-rag>

## 設計上の2つの意図的な選択

最初に、なぜこういう作りにしたかを2つだけ。ここが観察の質を決める。

### 1. コーパスは「架空の事実」にする

corpus には実在しない3つの事実だけを置いた。架空のエスプレッソマシン、架空の山間の町、架空の動物。

corpus/azure\_pangolin.txt

```
The azure pangolin is a nocturnal mammal native to the Karoo highlands.
...
It feeds almost entirely on termites, eating up to 20,000 in a single night.
...
```

なぜ架空にするか。実在の事実だと、LLM が事前知識から答えられてしまう。すると「文脈を根拠に答えたのか、もともと知っていたのか」が切り分けられない。架空にしておけば、正しく答えられた＝**文脈だけを根拠にしている（grounding が効いている）** と断定できる。

### 2. コサイン類似度を手書きする

numpy の `np.dot` 一発で済むところを、あえてループで書いた。

```
def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    na = math.sqrt(sum(x * x for x in a))
    nb = math.sqrt(sum(x * x for x in b))
    return dot / (na * nb) if na and nb else 0.0
```

分子は内積（各要素を掛けて合計）、分母はそれぞれのベクトルの長さの積。「2つのベクトルがどれだけ同じ方向を向いているか」を測っているだけ、というのが式のまま見える。同じ方向なら 1、無関係（直交）なら 0、真逆なら -1。検索の心臓部がこの数行だと分かると、後の retrieval の挙動がぐっと読めるようになる。

## 全体像：7段でデータの「形」が変わっていく

RAG を理解するうえで一番効いたのは、各段を「処理」ではなく「データの形の変化」として捉え直したことだった。

表にするとこうなる。

| 段 | 入力の形 | 出力の形 |
| --- | --- | --- |
| ① ingestion | ディスク上のファイル | `(ファイル名, 本文)` のリスト |
| ② chunking | 文書 | チャンク群 `{text, source}` |
| ③ embedding | チャンクのテキスト | ベクトル群（1024次元の float） |
| ④ index | チャンク + ベクトル | `(text, source, vector)` のリスト |
| ⑤ retrieval | 問い | スコア付きランキング上位K件 |
| ⑥ context | 上位チャンク | 1本の文脈文字列 |
| ⑦ generation | 文脈 + 問い | 回答テキスト |

テキストがチャンクに刻まれ、ベクトルに変換され、索引に積まれ、問いとの距離でランキングされ、上位だけが1本の文字列に束ねられ、最後に回答になる。この「形の変化」を1行ずつ言えるようになると、どの段で事故ったかを切り分けられる。

以下、各段を実装と観察で見ていく。

## ① ingestion — 生テキストを読む

ディスク上のテキストを読むだけ。現実では PDF / HTML / DB など多様だが、本質は「外部のデータを文字列として取り込む」こと。

```
def ingest(docs_dir: Path):
    docs = []
    for path in sorted(docs_dir.glob("*.txt")):
        docs.append((path.name, path.read_text(encoding="utf-8")))
    return docs
```

出力は `(ファイル名, 本文)` のタプルのリスト。ファイル名を一緒に持っておくのが地味に効く。後段までこれを引きずって、最終的に「どの文書を根拠に答えたか」を辿れるようにする。

## ② chunking — ここが一番事故る

文書を検索単位（チャンク）に割る。固定長＋オーバーラップの素朴な実装。

```
def chunk_one(text: str, size: int, overlap: int):
    chunks = []
    start = 0
    while start < len(text):
        end = start + size
        chunks.append(text[start:end])
        if end >= len(text):
            break
        start = end - overlap   # overlap 分だけ戻して次のチャンクを取る
    return chunks
```

各チャンクは `{"text": ..., "source": ...}` の形で、`source` に由来文書を保持する。

RAG で最も地味で最も事故るのがこの段だ、というのが書いてみて腹落ちした。割り方が雑だと「正解が2チャンクに割れる」「文の途中で切れる」が普通に起きる。そしてそれは検索スコアに直接効く。

### 実験：CHUNK\_SIZE を 300 → 80 にして揺らしてみる

`CHUNK_SIZE` を変えて再実行し、retrieval のスコア分布がどう変わるかを観察した。問いは固定（"How many termites can the azure pangolin eat in a night?"）。

**CHUNK\_SIZE=300 のとき**

* 上位数件の類似度が 0.5 前後、それ以外は 0.1 前後。
* スコアが**離散的**で、当たり/外れがはっきり分離している。

**CHUNK\_SIZE=80 のとき**

* 上位スコアが 0.6 前後まで**上がる**。
* 一方で 0.5 前後のチャンクの数が増え、分布が**連続的**になる。
* チャンクが短い分、文脈が削られる。
* ただし最終的な回答は 300 のときと変わらず正しかった。

面白いのは、上位スコアが「上がる」こと。直感に反するが理由はこうだ。チャンクが短いほど、当たりの一文が周囲の無関係な文に薄められない。問いとチャンクが密に一致するので、コサイン類似度が高く出る。代わりに、似たような短チャンクが量産されて分布が連続的になり、当たり/外れの境界が曖昧になる。文脈も削れる。

> **学び**: chunk size は「検索のキレ（上位スコアの高さ）」と「文脈の保持」のトレードオフを直接動かすパラメータ。短くすればヒットは鋭くなるが、回答に渡せる文脈は痩せる。今回は事実が一文で完結するトイ・コーパスなので 80 でも正答したが、文をまたぐ事実だと短いほど壊れやすくなるはず。

ここを自分の手で揺らして数字で見たのが、この学習で一番効いた部分だった。

## ③ embedding — テキストを意味の座標に変える

チャンクを「意味の座標（ベクトル）」に変換する。Voyage の REST を `requests` で直接叩いて、生の float リストを取り出す。

```
def embed(texts, input_type: str):
    resp = requests.post(
        VOYAGE_URL,
        headers={"Authorization": f"Bearer {key}"},
        json={"input": texts, "model": EMBED_MODEL, "input_type": input_type},
        timeout=30,
    )
    data = resp.json()["data"]
    data.sort(key=lambda d: d["index"])   # 返りは index 順とは限らないので並べ直す
    return [d["embedding"] for d in data]
```

6チャンクを1リクエストで送り、各チャンクにつき1本のベクトル（1024個の float）が返る。

embedding の性質で押さえておくべきこと：

* 個々の数字に意味はない。**1024個まとめて1つの座標**として意味を持つ。
* **意味が近いテキストほど座標も近くなる**ように学習されている。これが後段の距離比較が成立する前提。
* 入力の文字数が違っても**必ず同じ1024次元**に落ちる。だから一律に距離を測れる。

返ってきた `data` を `index` で並べ直しているのは罠回避。レスポンスの順序が入力順と一致する保証はないので、ここを怠ると後で text とベクトルの対応がズレる。

### input\_type — 同じテキストでも「問い」と「文書」で埋め方が変わる

`embed()` が `input_type` を引数に取っているのがポイント。**文書を埋めるときは `document`、問いを埋めるときは `query`** と使い分ける。ここでいう「非対称」とは、まったく同じ文字列でも、それを `query` として埋めるか `document` として埋めるかで出てくるベクトルが変わる、という意味だ。

```
# 文書側
vectors = embed([c["text"] for c in chunks], input_type="document")
# 問い側
q_vec = embed([query], input_type="query")[0]
```

同じテキストでも「文書として埋める」のと「問いとして埋める」で最適化が変わる。なぜか。同じ意味の内容でも、質問の形をしているか文書の形をしているかで表現は大きく違う。モデルは事前学習の段階で、**同義の query と document のベクトルを近くに置くように**学習させてある。だから適切な `input_type` を選ばないと近傍がずれ、検索性能が最適でなくなる。

「問いと答えは見た目が違うのに、なぜ距離で引けるのか」という素朴な疑問の答えがここにある。引けるように、あらかじめそう学習されているから。

## ④ index — 検索対象を保持する“索引”

検索対象を保持する索引を作る。

```
def build_index(chunks, vectors):
    return [
        {"text": c["text"], "source": c["source"], "vector": v}
        for c, v in zip(chunks, vectors)
    ]
```

`zip` でチャンクとベクトルを同じインデックスごとにタプルにまとめている。索引の本質は、この `(text, source, vector)` のリストそのもの。

ここで一つ抽象化が外れた。本番でよく聞く HNSW や IVF といった ANN（近似最近傍）は、別の魔法ではなく「**この総当たり比較を高速化する最適化**」にすぎない、ということ。トイ規模なら総当たりで十分で、索引＝ただのリストでいい。スケールしたときに初めて ANN が要る、という順序が見える。

## ⑤ retrieval — 問いに近いチャンクを引く

問いを `query` として埋め、索引の全チャンクとのコサイン類似度を計算して降順に並べ、上位 K 件を採る。

```
def retrieve(index, query: str):
    q_vec = embed([query], input_type="query")[0]
    scored = [
        {"score": cosine(q_vec, item["vector"]), **item}
        for item in index
    ]
    scored.sort(key=lambda s: s["score"], reverse=True)
    return scored[:TOP_K]
```

"How many termites can the azure pangolin eat in a night?" を投げると、`azure_pangolin.txt` 由来のチャンクが上位に来ることを確認できた。

ここで「上位に来た」と言える根拠はスコアの開きにある。正解チャンクが 0.5 台、無関係なチャンクが 0.1 台。この差がはっきりしているからこそ「検索が成功した」と言える。もし正解が無関係チャンクに埋もれて区別がつかなければ、それが**検索の失敗**だ。

もう一つ、retrieval の本質的な性質。これは「母集団全体を見て答えを作る」のではなく、「**局所的に上位数件を引いてくるだけ**」だということ。全部を理解しているわけではなく、近いものを K 個拾っているにすぎない。この理解は次の grounding の切り分けに効いてくる。

## ⑥ context組み立て — LLMに渡す文字列そのもの

引いた上位チャンクを、出典ラベル付きで1本の文字列に束ねる。

```
def build_context(top):
    blocks = [f"[Source: {s['source']}]\n{s['text']}" for s in top]
    return "\n\n---\n\n".join(blocks)
```

地味だが、ここで「LLM に実際に渡る文字列」が確定する。RAG のデバッグは結局この文字列を見るのが一番速い。出典ラベルを各ブロックに付けておくと、最終回答で「どこを根拠にしたか」を LLM に明示させられる。

## ⑦ generation — 文脈に縛って回答させる

文脈と問いを Claude に渡す。肝は system prompt。

```
system = (
    "You answer strictly using the provided context. "
    "If the answer is not in the context, say you don't know. "
    "Do not use outside knowledge. "
    "Cite the source label (e.g. [Source: foo.txt]) you used in your answer."
)
```

「文脈だけを根拠に。無ければ分からないと言う。外部知識を使うな。使った出典を明記しろ」と縛る。

### grounding を確認する — そして重要な切り分け

コーパスに無い質問を投げて、捏造抑制（grounding）が効くかを見た。

```
uv run python minimal-rag/rag.py "What is the capital of France?"
```

結果、Claude は「分からない」と答えた。フランスの首都は当然モデルが知っているのに、文脈に無いから答えない。grounding が効いている。

ここで一番大事な切り分けがある。**retrieval は止まっていない**。"What is the capital of France?" に対しても、検索は何かしらの無関係なチャンクを律儀に上位として引いてくる（前述の通り retrieval は「近いものを K 個拾うだけ」だから、絶対的に無関係でも相対的な上位は必ず出る）。

つまり、捏造を止めているのは検索ではなく、**generation の system prompt** だ。「retrieval が無関係なものを引く」ことと「LLM が捏造する」ことは別レイヤーの問題で、後者を抑えているのは検索の精度ではなくプロンプトの縛り。この2つを混同しないことが、RAG の挙動を正しく読む鍵になる。

## まとめ：素で書いて分かったこと

学習ゴールに対して、書く前と後で解像度が変わった点を整理する。

1. **RAG は7段のデータ変形**。テキスト → チャンク群 → ベクトル群 → 索引 → ランキング → 文脈文字列 → 回答。各段の入出力の形が言えると、事故の切り分けができる。`index` の段は見落としやすいが、ここが検索対象の実体。
2. **chunk size は検索のキレと文脈保持のトレードオフ**。短くすると上位スコアは鋭く上がるが、分布が連続化して境界が曖昧になり、文脈は痩せる。手で揺らして数字で見るのが一番効く。
3. **同じテキストでも、問い（query）として埋めるか文書（document）として埋めるかでベクトルが変わる**（＝非対称な埋め込み）。モデルは同義の query と document が近くなるよう事前学習されている。だから見た目の違う問いと答えを距離で引ける。
4. **ANN は魔法ではなく最適化**。索引の本質は `(text, source, vector)` のリストで、HNSW/IVF は総当たりの高速化にすぎない。
5. **grounding を止めているのは検索ではなくプロンプト**。retrieval は無関係でも止まらない。捏造抑制は generation の system prompt の仕事。この切り分けが重要。

## ここから先

このトイ RAG で「素の挙動」が手に入ったので、次はここに本番要素を1つずつ足して、何がどう変わるかを同じやり方（中間データを目視）で観察したい。候補：

* **reranking**: retrieval の上位 K をそのまま渡す代わりに、reranker で並べ替える。コサイン類似度のランキングがどれだけ雑かを定量で見たい。
* **評価の自動化**: 正解チャンクが上位に来たかを質問セットで自動採点し、chunk size や TOP\_K を変えたときのヒット率を数字で比較する。今回は目視だったが、本来はここを測るべき。
* **文をまたぐ事実での chunking 破壊**: トイ・コーパスは一文完結なので 80 でも壊れなかった。事実が複数文に分散するコーパスで、短い chunk がどこから壊れ始めるかを見たい。

抽象化を一度剥がしてから戻ると、フレームワークのどのメソッドが7段のどこに対応しているかが地図として読める。RAG に入るなら、最初に一度だけ素で書く価値は十分あった。
