## エージェント駆動開発とは

0→1 の MVP 開発で使う手法

PM / Dev / QA の3つの AI エージェントがサイクルを回して、人間は設計と意思決定に集中する

ただ「速く作る」だけではなく、**最初から保守性を意識して、次のフェーズ（ドキュメントファースト開発）にスムーズに移行できる土台を作る**のが目的

速さだけなら Vibe Coding でいいですが、それだと MVP ができた後に「テストもドキュメントもない、何がどう動いてるかわからないコード」になりがち

## 全体フロー

ポイントは **開発に入る前に3ステップの準備がある** こと、意思決定は多めだけど、ここを飛ばして「とりあえず作って」とやると、後から全部やりなおすよりまし

## Step 1: UI/UX ファースト - まず画面を作る

コードを書く前に、モックアップを作ってアプリのイメージを固める

これがあるとないでは全然違う

* テスト戦略が見える（「画面遷移が複雑だから E2E が要るな」とか）
* 仕様の抜け漏れが減る（「この画面のエラー時ってどうなる？」とか）
* AI が完成イメージを理解して実装できる

> 自分がやってる工夫としては、HTML + Tailwind でサクッと作って、Netlify に上げてスマホで見る。この段階では動かなくていい。見た目だけ  
> Netlifyは特定のリポジトリのそのディレクトリを指定して、自動デプロイしてくれるので、適当に /mockup みたいなディレクトリを作ってそこに放り込んでもらってます

## Step 2: 仕様の整理 — 何を作るかを決める

モックアップをもとに仕様を整理する

```
# {アプリ名} 仕様書

## 概要
（何を作るか、一言で）

## ターゲット
（誰が使うか）

## 機能一覧
| # | 機能名 | 概要 | MVP? |
|---|-------|------|------|

## データモデル
（ER 図 or テーブル定義）

## 画面フロー
（mermaid flowchart or 画面遷移図）

## 技術スタック
| レイヤー | 技術 | 理由 |
|---------|------|------|
```

ここまでは「何を作るか」の話。次の Step 3 が「どう品質を守るか」の話

## Step 3: 品質基盤の決定 - ここが一番大事

MVP でもちゃんと品質を考えたいので、**「このシステムでは何を重視するか」を最初に決める**

### テスト戦略の決定

ISO/IEC 25010 の品質特性をベースに、このシステムに必要なテストを選定する

やり方はシンプルで、まず **ドメイン特性から重視すべき品質特性を決める**

品質特性が決まったら、そこから必要なテストが自動的に導出される

| 品質特性 | → 必要なテスト |
| --- | --- |
| 機能適合性（ほぼ全て） | Unit テスト、Integration テスト |
| セキュリティ | 入力検証テスト、認証テスト、シークレット検出 |
| 信頼性 | 状態遷移マトリクス、エラーハンドリングテスト |
| 性能効率性 | 負荷テスト、レスポンスタイム計測 |
| 使用性 | E2E テスト、スナップショットテスト |

例えば「AI 分析アプリ（個人利用）」なら:

```
## テスト戦略

### 重視する品質: 機能適合性 > 性能効率性

### 採用するテスト
- [x] Unit テスト — ドメインロジック
- [x] Integration テスト — API エンドポイント
- [x] E2E テスト — メインフロー1本
- [ ] 負荷テスト → 個人利用なので MVP では不要
- [ ] セキュリティテスト → API キー管理のみ

### 採用しない理由
- 負荷テスト: 同時アクセスを考慮する規模ではない
- Contract テスト: 外部 API は Claude のみ、モックで十分
```

こうやって **カタログから選んで理由をつける** ことで、「なぜこのテストを書くのか / 書かないのか」が明確になる

### セキュリティ方針の決定

0→1 だと全部をガチガチにする必要はない。大事なのは **「MVP ではここまで、本開発ではここを追加」を明確にすること**

| 観点 | MVP での対応 | 本開発で追加 |
| --- | --- | --- |
| 認証 | IP 制限 + パブリック | Google SSO + JWT |
| データ保護 | 最小限 | 暗号化 + アクセス制御 |
| API キー | 環境変数 | シークレットマネージャー |
| 入力検証 | Zod バリデーション | Zod + WAF |
| CORS | 開発 URL のみ | フロントオリジンのみ |
| レートリミット | なし | ユーザー単位で制限 |

認証を IP 制限で済ますのか SSO まで入れるのかを**最初に決めておく**。後から「あれ、認証どうするんだっけ」にならない

### ドキュメント計画

「MVP 完成後にまとめてドキュメント書く」は地獄。**いつ何を書くかを最初に決めて、タスクの中で段階的に作る**

| ドキュメント | 作成タイミング | 誰が |
| --- | --- | --- |
| API 仕様書 | API を追加するタスクごと | Dev |
| 状態遷移図 | 状態を追加するタスクごと | Dev |
| ADR | 技術選定があったとき | PM |
| アーキテクチャ図 | MVP 完成時 | Dev |
| ER 図 | MVP 完成時 | Dev |

最初に「この一覧で漏れはないか？」を確認しておく

## タスクサイクル - PM → Dev → QA

Step 3 の品質基盤が決まったら、タスク単位で開発を回す

### PM: タスクの定義

PM の出力には、品質基盤で決めた方針が反映される

```
## タスク: セッション結果の保存 API

### 完了条件
- [ ] POST /api/sessions で結果を D1 に保存できる
- [ ] 不正な入力に 400 を返す

### テスト要件（テスト戦略より）
- [ ] Unit: バリデーションロジック
- [ ] Integration: POST の正常系・異常系

### ドキュメント要件
- [ ] API 仕様書に POST /api/sessions を追加
```

テスト要件とドキュメント要件が**最初から完了条件に含まれている**のがポイント

### Dev: TDD + ドキュメント更新

ルール

* テストを先に書く
* 完了条件に書かれていない機能は実装しない
* タスク完了時に該当するドキュメントも更新する

### QA: 品質基盤に基づくレビュー

QA のチェック項目は Phase 0 で決めた品質基盤から自動的に決まる

| チェック | 内容 |
| --- | --- |
| 完了条件 | PM の条件をすべて満たしているか |
| テスト | テスト戦略で決めた種類のテストがあるか |
| セキュリティ | セキュリティ方針を満たしているか |
| ドキュメント | 該当するドキュメントが更新されているか |
| コード品質 | lint + typecheck が通るか |

QA が毎回「何を見るか」を考える必要がない。品質基盤が決まっていれば、チェック項目も自動で決まる

不合格なら具体的な理由をつけて Dev に差し戻し

## 人間の介入ポイント

| タイミング | 何をするか | スキップ可能? |
| --- | --- | --- |
| 品質基盤の承認 | テスト戦略・セキュリティ方針・ドキュメント計画 | **不可** |
| タスクごと | PM の完了条件の確認 | 小タスクなら可 |
| QA 合格後 | 差分レビュー・動作確認 | **不可** |

## ドキュメントファースト開発への移行

この手法で MVP を作ると、完成時点で以下が揃っている:

```
✅ テスト（テスト戦略に基づいた適切な種類）
✅ API 仕様書（タスクごとに更新済）
✅ 状態遷移図（あれば更新済）
✅ ADR（技術判断があれば記録済）
✅ セキュリティ方針（MVP vs 本開発の選択済）
✅ アーキテクチャ図 + ER 図（MVP 完成時に作成）
```

これが揃っていれば、次の機能追加から**ドキュメントファースト開発**にそのまま移行できる

「MVP 作ったけどドキュメントがない、テストもどこまであるかわからない」にはならない

## まとめ

エージェント駆動開発は「速く作る」だけの手法じゃない

1. **UI/UX ファースト** — モックアップでアプリイメージを固める
2. **仕様の整理** — 機能一覧・データモデル・画面フロー・技術選定
3. **品質基盤の決定** — テスト戦略・セキュリティ方針・ドキュメント計画を ISO 品質特性ベースで決める
4. **タスクサイクル** — PM → Dev（TDD + ドキュメント更新）→ QA（品質基盤ベースのチェック）
5. **MVP 完成時にドキュメントファースト開発に移行できる状態になっている**

「保守性のある MVP を高速に作る」。それがエージェント駆動開発
