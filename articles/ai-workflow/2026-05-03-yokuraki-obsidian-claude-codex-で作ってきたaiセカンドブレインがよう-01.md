Obsidian + Claude / Codex で作ってきた「AIセカンドブレイン」が、ようやく完成形に近づいてきた。

一言でいうと、自分専用のAI図書館を作る話。
Obsidianに貯めたメモや資料を、Claude / Codexが整理し、つなげ、壊れていないか点検しながら、あとから何度でも使える知識ベースに育ててくれる。
AIに毎回聞くのではなく、AIと一緒に「自分の知識の地図＝セカンドブレイン🧠」を作っていくイメージです。

ちょっとテクニカルな話になりますがその実現手順です：

これまでは、Obsidianに情報を貯めて、必要なときにAIに読ませる運用だった。
でも今回、Karpathyの「LLM Wiki」の考え方を参考にして、単なるメモ管理から 自己成長型AI知識ベース に近い形へ進化させた。
ポイントは、RAGのように毎回ゼロから検索するのではなく、AIがMarkdown Wikiを継続的に育てること。

具体的には、

Schema：Wiki更新ルールを明文化
Ingest：Inbox / Projects / Daily Note から知識化
Query：Wikiを起点に根拠付き回答
Sanity Check：リンク切れ・重複・frontmatter不備を点検
Skill Library：うまくいった作業手順を再利用
という仕組みを入れた。

実際にSanity Checkを走らせたら、壊れたWikiリンク27件、frontmatter不備26件を検出。
修復後はどちらも0件になった。
これは単なる「AIメモ術」ではなく、かなり多くの知的職業に応用できると思っている。

研究者の場合
研究者にとって一番大変なのは、論文・メモ・仮説・実験結果がバラバラに散らばること。
LLM Wiki型のAIセカンドブレインを使うと、
- 論文PDFやWeb記事を読み込む
- 研究テーマごとに概念ページを作る
- 先行研究ごとの比較表を自動更新する
- 似た仮説・矛盾する結果を検出する
- 実験ログと文献メモをつなげる
といったことができる。
単に「この論文を要約して」ではなく、読んだ論文が既存の研究テーマ・人物・手法・仮説に統合されていく。
つまり、文献レビューが一回限りの作業ではなく、育ち続ける研究Wikiになる。

システムエンジニアの場合
エンジニアの場合、日々の調査・設計判断・エラー対応・実装メモが重要な資産になる。
この仕組みを使うと、
- エラー対応ログをナレッジ化する
- 設計判断の理由をWikiに残す
- 過去の障害対応と現在のバグを照合する
- ライブラリやAPIの調査結果を更新し続ける
- プロジェクトごとの技術的決定をMOC化する
ことができる。
たとえば、あるAPI連携でハマったとき、単なるチャット履歴で終わらせず、
「このAPIは認証方式に注意」
「このエラーは環境変数の読み込み順が原因」
「このライブラリはバージョン差異に注意」
という形で、次回使える知識に変換できる。
AIが単なるコーディング補助ではなく、チーム内の技術Wiki編集者になるイメージ。

Webマーケティング担当者の場合
Webマーケティングでは、施策・顧客理解・競合調査・広告結果・SEO記事・SNS投稿が大量に発生する。
LLM Wiki型にすると、
- 顧客ペルソナを継続更新する
- 競合調査を比較ページとして蓄積する
- 成果が出たLPや広告文のパターンを抽出する
- SEO記事のテーマクラスターを管理する
- SNS投稿の反応から勝ちパターンを記録する
ことができる。
毎回「この商品の訴求を考えて」とAIに聞くのではなく、過去の施策・顧客の反応・成功パターンを踏まえた上で、AIが提案してくれるようになる。
これは、マーケティングの属人知をWiki化していく使い方に近い。

弁護士・行政書士・士業の場合
法律系の仕事では、案件ごとの事実関係、条文、判例、行政手続き、過去の相談対応が重要になる。
この仕組みを使うと、
- 案件ごとの論点を整理する
- 条文・判例・通達をテーマ別にまとめる
- 類似案件の対応履歴を参照する
- 相談内容から必要書類や手続きの流れを整理する
- よくある質問をナレッジ化する
ことができる。
もちろん、最終判断は専門家が行う必要がある。
でも、過去に調べた論点や、何度も出てくる相談パターンをAIが整理してくれるだけでも大きい。
特に行政書士なら、ビザ、会社設立、補助金、許認可など、制度・書類・手続きが多い領域で相性がいい。
「このケースでは以前どんな資料を見たか」
「似た案件では何が論点になったか」
「必要書類のチェックリストはどこか」
を、毎回ゼロから探さずに済む。
