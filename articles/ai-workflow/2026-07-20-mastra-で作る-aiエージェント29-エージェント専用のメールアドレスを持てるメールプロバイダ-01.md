## AgentMail とは

**AgentMail** は、AIエージェント専用に設計されたメール API プラットフォームです。  
「エージェントに自分専用のメールアドレスを持たせる」ことを目的として作られており、従来の Gmail / Outlook のような人間向けメールインフラとは根本的に設計思想が異なります。

### 従来のメールプロバイダーとの違い

| 比較項目 | Gmail / Outlook | AgentMail |
| --- | --- | --- |
| 受信ボックス作成 | 手動のみ | API で即時作成 |
| 送信レート制限 | 厳しい | 大量送信向けに緩和 |
| 認証方式 | OAuth（複雑） | API キーのみ |
| リアルタイム通知 | 非公式 | Webhook / WebSocket 標準対応 |
| 開発者体験 | 悪い（Gmail API は複雑） | API ファースト設計 |

AgentMail には **Free ($0/月)** プランがあり、カード登録なしで少量の inbox やメール送受信を試せます。加えて、スタートアップ向けの無料月オファーも用意されています。

### できること（エージェントの能力）

* ユーザーとのメール対話（カスタマーサポート・営業フォローアップ等）
* 企業のメール業務自動化
* 第三者サービスへの認証用アカウントとしての利用
* エージェント同士のメール通信

## リソース階層

AgentMail の構造は以下の階層になっています。

```
Organization（あなたのビジネスアカウント）
  └── Pod（テナント・顧客単位の分離コンテナ ※オプション）
        ├── Domain（独自ドメイン）
        └── Inbox（メールアカウント 1 件）
              ├── Thread（会話スレッド）
              └── Message（個別メール）
                    └── Attachment（添付ファイル）
```

## ドメイン（@の後ろ）はどこまで変更できるか

### デフォルトドメイン

アカウント作成直後は **`@agentmail.to`** が使えます。  
開発・検証目的には十分ですが、本番環境での利用には独自ドメインの設定が推奨されています。

### カスタムドメインの設定

自社所有のドメインを登録し、DNS レコード（MX / SPF / DKIM / DMARC）を設定すれば、独自ドメインでの送受信が可能になります。

それぞれの役割は、MX が「どこにメールを配送するか」を決める設定、SPF が「どの送信元サーバーを正規とみなすか」を示す設定、DKIM が「メールに改ざん検知用の署名を付ける」設定、DMARC が「SPF / DKIM の結果をどう扱うか」を決める運用ルールです。これらは通常、社内の情シス・インフラ・メール運用担当が把握していますが、実際の設定値は DNS 管理会社や外部ベンダーが管理している場合もあります。

```
// 例: example.com を登録
const domain = await client.domains.create("example.com");
// → DNS レコードが返ってくるのでプロバイダーに登録して認証
```

認証完了後は `example.com` ドメインの Inbox を自由に作成できます。

**企業ユースで特に重要な考慮点：**

* **送信レピュテーション**はドメイン単位で構築されます。1 ドメインから大量送信するよりも、複数ドメインに分散させる方が届達率が上がります（例: `company.com`, `usecompany.com`, `companyapp.com` など）。
* ドメインは **API 経由で何個でも登録できます**。ドメインごとに独立した評判管理が可能。

AI エージェントの送信元アドレスは、社員とまったく同じ見た目にするよりも、`agent.companyname.co.jp` のようなサブドメインや `support@companyname.co.jp` のような役割用アドレスに分ける方が実務上は扱いやすいです。受け手には会社由来だと分かりやすくしつつ、運用・監査・停止判断を人間のメールと切り分けられるためです。

### サブドメインの対応

`subdomains_enabled: true` を設定すると、ワイルドカード MX レコード（`*.example.com`）を 1 つ追加するだけで、任意のサブドメインのInbox を個別登録なしに作れます。

```
// サブドメイン有効化
await client.domains.update("example.com", { subdomainsEnabled: true });

// 任意のサブドメインで Inbox を作成（bot.example.com を別登録不要）
const inbox = await client.inboxes.create({ username: "agent", domain: "bot.example.com" });
// → agent@bot.example.com
```

> **注意**: サブドメイン Inbox は親ドメインの DKIM / 評判を共有します。評判を完全分離したい場合はサブドメインを個別ドメインとして登録する必要があります。

## Inbox（@の前）は何の単位で作るか

### Inbox = 1 つのメールアカウント

`Inbox` は AgentMail の中核リソースで、**ユニークなメールアドレス 1 件 = Inbox 1 つ**です。

作成時に `username`（@の前）と `domain`（@の後）を指定します。省略すると AgentMail がランダムアドレスを自動生成します。

```
// username + domain を明示して作成
const inbox = await client.inboxes.create({
  username: "support",
  domain: "example.com",
  displayName: "サポートチーム",
});
// → support@example.com が作られる

// 省略した場合はランダムアドレスが生成される
const inbox2 = await client.inboxes.create();
```

### 何単位で Inbox を作るべきか

会員顧客へのメール業務を想定した場合、代表的な設計パターンは以下のとおりです。

| パターン | Inbox の粒度 | 向いているケース |
| --- | --- | --- |
| **役割単位** | `support@`, `sales@`, `newsletter@` | 部署・機能ごとにエージェントを分ける |
| **エージェント単位** | エージェント 1 体 = Inbox 複数 | 1 エージェントに複数の送信 Inbox を持たせて分散送信（届達率向上） |
| **顧客テナント単位（Pod 活用）** | テナントごとに Pod → その中に Inbox | SaaS プロダクトで顧客ごとに完全分離が必要な場合 |

> **Pod（ポッド）** は、テナント間のデータ分離を行うオプションのコンテナです。1 Pod = 1 顧客テナントとして設計すると、Inbox / Domain / Thread がテナントごとに完全分離されます。

### メタデータで内部 ID を紐付ける

Inbox には任意のキー・バリューメタデータを付与できます。自社システムのユーザー ID や顧客 ID を紐付けておくと管理が楽になります。

```
await client.inboxes.create({
  username: "agent-001",
  domain: "example.com",
  metadata: {
    tenant_id: "customer-abc",
    agent_role: "customer_support",
  },
});
```

## Mastra エージェントでメールを送受信する

AgentMail には **Mastra 専用の公式インテグレーションページは現時点では存在しません**。  
ただし、`agentmail` npm パッケージを使って Mastra の **Tool** として組み込むことができます。

### 基本的なアーキテクチャ

```
Mastra Agent
  ├── Tool: sendEmail        → AgentMail SDK → client.inboxes.messages.send()
  ├── Tool: replyToEmail     → AgentMail SDK → client.inboxes.messages.reply()
  ├── Tool: listThreads      → AgentMail SDK → client.threads.list()
  └── Tool: readThread       → AgentMail SDK → client.threads.get()

受信トリガー（2 択）
  ├── Webhook（外部公開 URL が必要）
  └── WebSocket（公開 URL 不要、接続維持型）
```

## 企業の会員向けメール配信における設計指針まとめ

| 検討項目 | 推奨アプローチ |
| --- | --- |
| **送信ドメイン** | 自社ドメインをカスタムドメイン登録。大量送信は複数ドメインに分散 |
| **Inbox の粒度** | 役割単位（`support@`, `newsletter@`）または送信量分散のため複数 Inbox |
| **テナント分離が必要な場合** | Pod を活用し、顧客ごとに Inbox / Domain を分離 |
| **受信処理** | 本番環境では Webhook（リアルタイム）を推奨。公開 URL 不要なら WebSocket |
| **二重送信防止** | `client_id`（冪等キー）または `Idempotency-Key` ヘッダーを使用 |
| **人間チェックを挟む場合** | Draft API で下書き作成 → 承認後に送信（Human-in-the-loop） |
| **Mastra との統合** | `agentmail` npm パッケージを Mastra Tool に組み込む |

## 顧客向け AI エージェントをメールチャネルで運用するときの注意事項

### 1. 認証は「差出人アドレス一致」だけで判断しない

`From` が登録済みアドレスと一致していても、それだけで本人確認はできません。メールは差出人名や表示アドレスを偽装しやすく、返信メールや転送メールでは元の送信者情報も見えにくくなります。

そのため、本人確認はメールアドレス単体ではなく、**件名の引き継ぎ、過去スレッドの連続性、ワンタイムトークン付きリンク、会員ページでの再認証、社内CRM 上の顧客 ID 照合**などを組み合わせるのが安全です。重要な操作ほど、メール本文だけで完結させない方がよいです。

### 2. ガードレールは「会員ごと分離」ではなく「文脈と権限の分離」で作る

1 つのエージェントが全会員を相手にする場合、最重要なのは**他会員の情報を絶対に混ぜないこと**です。

実装上は、会員 ID ごとに取得できるデータを API 層で厳密に制限し、プロンプトへ渡す前にマスキングと最小化を行います。会話履歴も、全件をそのまま渡すのではなく、**その会員に紐づく履歴だけ**を要約して与える設計が基本です。プロンプトインジェクション対策として、メール本文をそのまま指示として扱わず、命令文とデータを分離して解釈させる必要があります。

### 3. メモリは「全部残す」より「要約して残す」方が事故が少ない

過去のやり取りを保持するのは有効ですが、全文を長期記憶に入れると、誤参照・古い前提の再利用・個人情報の過剰保持が起きやすくなります。

おすすめは、**短期メモリには直近のスレッド、長期メモリには会員ごとの要約と確定情報だけ**を置く方法です。たとえば「連絡希望時間」「契約プラン」「未解決の要件」「過去のクレーム有無」などの事実だけを保存し、元メール本文は必要最小限にします。保存期限や削除手続きも最初から決めておくべきです。

### 4. 受信取りこぼしに備えて、Webhook だけに依存しない

Webhook は速い一方で、ネットワーク障害やタイムアウト、設定ミスで取りこぼす可能性があります。したがって、**Webhook 受信 → キュー投入 → 非同期処理**を基本にしつつ、別系統で**定期ポーリングの照合ジョブ**を持つと安全です。

具体的には、最後に処理した message\_id や thread\_id を記録し、一定間隔で未処理イベントを再取得します。加えて、一定時間返信が生成されない場合は担当者へアラートを飛ばし、必要なら自動で「受領しました。確認中です」の一次返信を送るなど、顧客が放置された状態にならない歯止めを入れます。

### 5. 複数メールの同時受信は「会員単位の直列化」で崩れにくくする

同じ会員から複数の話題がほぼ同時に届くと、古い会話の続きを新しい話題に誤って混ぜる危険があります。ここは、**会員単位またはスレッド単位でロックをかけて順番に処理する**のが基本です。

完全に 1 スレッドへ集約できない場合でも、少なくとも「同一会員の未解決案件は同時に 1 件だけ進める」「新しいメールは既存ケースに紐付けるか、新規ケースとして分岐するかを先に決める」というルールを持たせると混乱しにくくなります。

### 6. ほかに押さえておきたい注意点

* **送信失敗時の再送制御**: リトライは必要ですが、二重送信を避けるために冪等キーを必ず使います。
* **送信前の承認**: 契約変更、返金、解約、個人情報開示のような高リスク案件は、下書き止まりか人間承認にします。
* **監査ログ**: 何を根拠に返信したかを追えるように、取得した文脈、参照したナレッジ、送信結果を残します。
* **機密情報の扱い**: 本文、添付、署名欄に個人情報や認証情報が含まれる前提で、保存・暗号化・アクセス制御を設計します。
* **営業・サポートの境界**: 自動返信で済ませてよい問い合わせと、人間に引き継ぐ問い合わせを最初から分けます。
* **営業時間と期待値**: 24 時間応答なのか、営業時間内のみなのかを明示し、一次受付と最終解決を分けて考えます。
* **スパム・悪意ある入力**: HTML メール、長文の引用、添付ファイル、外部リンクは危険源になりやすいので、検査とサニタイズを入れます。
* **差出人名の明示**: AI が送るなら、表示名や署名で自動応答であることを分かる範囲で示した方が、誤認とトラブルを減らせます。
* **法務・規約**: 自動応答の利用規約、個人情報保護、同意取得、国外送信の扱いは、実装前に確認しておきます。

### 7. マネージドサービスのコンプライアンス保証を契約前に確認する

AgentMail にメールサーバ運用を委託する場合、個人情報保護とカード情報の観点は「公開情報の確認」と「契約時の確認」を分けて判断するのが安全です。

公開情報ベースでは、AgentMail は **SOC 2 Type I / Type II の取得**を明示しています。アクセス制御、MFA、TLS、保存時暗号化、KMS、監査ログ、バックアップ運用などの説明も公開されています。

一方で、例えば **FISC とか PCI DSS といったセキュリティ基準への準拠の明示は公開ページ上で確認できませんでした**。そのため、例えばクレジットカードデータを扱う可能性がある業務では、メール本文・添付でカード情報を受け付けない設計を前提にし、決済は PCI DSS 準拠の決済事業者へ分離するのが実務上の基本です。

導入前には、少なくとも次を確認するのが望ましいです。

* 最新の SOC 2 Type II 報告書（監査期間・除外事項・補完統制）
* DPA（個人情報処理契約）の締結可否、再委託先、越境移転の取り扱い
* データ保管リージョン、削除 SLA、バックアップ保持期間
* インシデント通知 SLA、監査証跡の提供範囲
* 業界特有の セキュリティ基準 の適用範囲（対象外の場合はその明示）

### 8. 実装の分け方の目安

ここでいう分割は、まず **責務分割** を指します。つまり、**受信・認証・文脈抽出・応答生成・送信・監査・再処理** という業務上の責務を分けることが先で、その実装手段として「ツール分割」「サブエージェント分割」「ワークフロー分割」を選びます。

したがって、スーパーバイザーエージェント配下にサブエージェントをぶら下げる設計は有効です。ただし、全工程をサブエージェント化するより、次のような **ハイブリッド構成** の方が運用しやすくなります。

* 決定的に処理すべき工程（受信正規化・認証判定・送信・再処理）はツールまたはワークフローで固定化する
* LLM 判断が必要な工程（文脈抽出・応答生成・監査の一部）をサブエージェントに担わせる
* スーパーバイザーはオーケストレーションに専念し、外部 I/O や副作用を直接持ちすぎない

この形にすると、柔軟性を保ちながら、再現性・監査性・障害時の切り分けやすさを両立できます。ひとつの関数や単一エージェントに全責務を詰め込むと、取りこぼし対応やガードレール修正が難しくなります。
