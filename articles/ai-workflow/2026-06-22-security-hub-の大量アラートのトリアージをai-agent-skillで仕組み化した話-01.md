## はじめに

AWSでのセキュリティ対策としてAWS Security Hubを有効にしている会社も多いかと思います。

この記事では、Security HubのCSPM（Cloud Security Posture Management）を有効にした後に直面した「アラート過多で対応が回らない」という課題と、それを**Claude の Agent Skill**を利用し仕分けを効率化した取り組みを紹介します。

「Security Hub を入れたはいいが放置気味になっている」という方の参考になれば幸いです。

## 課題 : Findings多すぎ問題

Security HubのCSPMを有効にすると、AWSのベストプラクティス（AWS Foundational Security Best Practicesなど）に照らした設定不備や、Amazon Inspectorによる脆弱性スキャン結果などがfindingsとして集約されます。

しかし、弊社の場合、以下のような想定を大きく超えた量の警告が発生してしまいすべてを人力で対応しきれる状況ではありませんでした。

* ECRにプッシュしたコンテナイメージのスキャンで、CVEが大量に検出される
* 設定検出系のルールが、環境ごと・リソースごとにfindingsを生成する
* 一度対応しても、新しいイメージや構成変更でまた発生する

重要度 `CRITICAL`, `HIGH` だけに絞っても数百件の警告が発生していたため、上記のような本質的ではないエラーに埋もれて本当に危険な警告を見逃す恐れもあり、対策を考える必要がありました。

## 解決策 : Agent Skill作成

Skillの役割としては以下の2点になります

* ECRスキャンの警告の自動Close
* 上記を除いた警告のコンテキストを収集し、判断材料を人間に提示する

### ECRスキャンの警告の自動Close

findingsの中で最もノイズになっていたのが、Amazon Inspectorによる**ECR コンテナイメージのスキャン結果**でした。Inspectorはもう使っていない古いイメージに対してもスキャンを走らせ、findingsを出し続けます。

結果として同じCVEが、バージョン違いのイメージから何十件も重複して検出されてしまい、findingを埋めてしまう、という状態になってしまっていました。

そこでこの種の警告は、**最新のイメージ以外の findings をまとめて `RESOLVED` に更新する** という方針で機械的に処理することにしました。

### 残った警告は、コードと突き合わせて方針を提案する

ノイズを除いたあとに残った警告に関してはきちんと調査をしてトリアージを行います。  
Claudeに以下のような情報を集め、推奨アクションを提案してもらいます。

* **警告の内容**：どんな CVE / 設定不備なのか、攻撃が成立する前提は何か
* **アプリケーションコード**：その脆弱なコンポーネントは実際に使われているのか
* **Terraform のインフラ設定**：指摘された設定は意図的なものか、代替の防御策があるか

推奨アクションとして用意しているのは、次のような方針です。この提案を元に人間が判断し、findingsのステータスを更新、対応チケットの作成などを指示します。

| 方針 | どんなときに選ぶか |
| --- | --- |
| 対応する | 実際にリスクがあり、修正が必要 |
| 受容する | 仕様上どうしても出るが、許容できる |
| 解決済みとして閉じる | コード・設定上、すでにリスクが立たない |
| 保留する | 判断材料が足りず、次回に持ち越す |

## まとめ

Security Hub のアラート過多という、多くのチームがぶつかるであろう課題に対して、Agent Skillを利用した取り組みを紹介しました。このスキルを運用に乗せた結果、数百件あったfindingsの量を圧縮でき、トリアージ判断の質と再現性、効率を大幅に向上させることが出来ました。

Security Hubに限らず、「アラートは出るが捌ききれない」という課題を抱えている方の、ひとつのヒントになれば幸いです。
