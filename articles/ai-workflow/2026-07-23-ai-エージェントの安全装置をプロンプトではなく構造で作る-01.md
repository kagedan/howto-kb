前回、YAML のオントロジー定義からエージェントのツール群を機械的に生成する話を書きました。今回はその続きで、一番書きたかったこと——**エージェントに業務データの書き込みを任せるとき、安全をどこで担保するか**——を書きます。

## 「危険な操作をしないで」という祈りの限界

まず思いつく安全対策は、システムプロンプトに禁止事項を書くことです。「確認なしに削除しないで」「請求操作は慎重に」——しかし LLM の指示追従は保証ではなく傾向です。しかも入力には外部由来のデータ（顧客名、メモ欄、取り込んだ CSV）が混ざり、悪意ある指示が紛れ込む可能性（プロンプトインジェクション）を排除できません。プロンプトによる禁止は**祈り**です。

ミハシラ（中小向けの業務基盤）では、安全装置をすべて「エージェントが破りようのない場所」＝エージェントの外側の構造に置きました。その4つの装置を実装で見ていきます。

## 装置1: 書き込みは Action Engine だけを通る

土台は**書き込み単一経路**です。UI のボタンも、エージェントの `action_*` ツールも、外部連携の MCP も、データを変更する手段はすべて同じ `invokeAction` 関数（Action Engine）に合流します。API 層以外に DML 権限を持つ DB ユーザーは作りません。抜け道が最初から存在しないので、「エージェントだけ検証が甘い」という事故は構造的に起きません。

`invokeAction` は 1 トランザクションで次を通します: ①アクション定義の解決 ②権限検証（roles / agentPolicy）③パラメータ検証（未知キー拒否・required・enum・型・参照先の存在）④対象のロック取得と precondition 評価 ⑤effects 適用（状態遷移を強制）⑥追記専用の監査ログへ記録。

重要なのは、失敗が例外ではなく**型付きの結果**として返ることです。

```
export type ActionFailureCode =
  | "action_not_found" | "target_not_found" | "forbidden"
  | "requires_approval" | "validation_failed"
  | "precondition_failed" | "invalid_transition" | "unsupported_effect";
// 失敗時: { ok: false; code: ActionFailureCode; message: string }
```

拒否は会話を殺すエラーではなく、**エージェントが次の一手を考えるための情報**です。これが後で効いてきます。

アクション定義には、エージェントが自律実行してよいか（`auto`）、人間の承認が要るか（`confirm`）を1行で宣言します。受託業テンプレートの「受注確定」はこうです。

```
- name: markWon
  displayName: 受注確定にする
  target: Opportunity
  preconditions:
    - { expr: 'target.stage == "negotiating"', message: 交渉中の商談のみ受注確定にできます }
  effects:
    - update: { stage: won }
    - create: { type: Project, ... }    # 案件と契約ドラフトを自動生成
    - create: { type: Contract, ... }
  permissions: { roles: [operator, admin], agentPolicy: confirm }
```

ステージを1段進める `advanceStage` は `auto`、受注確定・失注・案件中止など影響の大きいものは `confirm`。この宣言を実行時に強制するのが、Agent SDK の `canUseTool` コールバックです（前々回のサブスク認証記事でも軽く触れましたが、ここが本丸です）。

```
// packages/agent/src/runtime.ts（簡略化）
if (action?.permissions.agentPolicy === "confirm" && input["dryRun"] !== true) {
  const approval = await createApprovalRequest(pool, {
    actionName: action.name, params: input, requestedBy: agentActor.id,
  });
  return {
    behavior: "deny",
    message: `「${action.displayName}」は人間の承認が必要です。承認リクエスト ${approval.id} を` +
      `作成しました。実行はせず、承認してほしい内容を人間に分かる形で提示してください`,
  };
}
```

エージェントが `confirm` アクションを呼ぶと実行前に遮断され、**承認リクエストが DB に永続化**されます。人間は承認インボックスで対象とパラメータを見て決裁し、承認されると**決裁した人間の actor** で `invokeAction` が実行されます（エージェントの権限ではなく）。なお試算（dryRun — 全パイプラインを実行して最後に ROLLBACK）は何も確定しないため `confirm` でも素通しで、「受注確定したらどうなる？」には承認なしで答えられます。

このゲートは二重です。仮に `canUseTool` をすり抜けても、Action Engine 自身が `actor.type === "agent"` かつ `confirm` の呼び出しを `requires_approval` で拒否します。承認が SDK のコールバック実装「だけ」に依存していないことが要点で、外部から MCP 経由で叩かれても同じ Action Engine に落ち、同じ承認フローに入ります。

## 装置3: 状態機械は「理由付き」で拒否する

オブジェクトには状態機械を宣言できます。商談は `lead → qualifying → proposing → negotiating → won/lost`。遷移表に無い状態変更は、どのアクションからでも拒否されます。

```
// packages/engine/src/actions.ts — update effect の遷移チェック（抜粋）
const allowed = status.transitions.some(
  (t) => t.from === from && t.to === value && t.via === action.name,
);
if (!allowed) {
  return { ok: false, code: "invalid_transition",
    message: `${objectDef.displayName} の状態を ${from} から ${value} へ ` +
      `アクション「${action.displayName}」で遷移させることはできません` };
}
```

ここで「失敗を型付きで返す」設計が効きます。エージェントには「拒否されたら、拒否理由を使って原因と次の一手を説明する。同じ呼び出しを繰り返さない」と教えてあります。まだ提案中の商談に受注確定を頼むと、「交渉中の商談のみ受注確定にできます」という拒否理由が返り、エージェントは「先にステージを交渉中まで進めますか？」と**次の一手を変えて**提案してきます。拒否メッセージは YAML に人間向けの日本語で書かれているので、そのままユーザーへの説明になります。順序違反を防ぐことと、エージェントが行き詰まらないことは普通はトレードオフですが、拒否に理由を付けて返す構造にすると両方が同時に手に入ります。

## 装置4: bulk\_import — エージェントに SQL を与えない

「7月の平日、佐藤さんの工数実績を毎日8時間で入れて」のような依頼では、22件の定型登録が発生します。アクション22連打は遅く、ターン上限にも当たる。では SQL や直接 INSERT をエージェントに生成させるか？　明確に却下しました。理由は2つ。

第一に、SQL を通した瞬間、precondition・権限・状態機械・監査という安全装置が**構造ごと消える**からです。第二に、こちらが本質ですが、**承認とは人間が意味を理解して判断できること**だからです。UUID の並んだ INSERT 文を見せられて「承認しますか？」と聞かれても、人間にはレビューできません。承認フローは形式的に存在しても、実質は壊れています。

採用したのは `bulk_import` ツールです。エージェントが行の配列を組み立て（日付の展開・値の割付は LLM の得意仕事）、①ドライラン検証——行エラーはエージェントに差し戻し、承認リクエストは作らない ②全行 OK のときだけ承認リクエストを作成 ③人間は受信箱で「佐藤の7月平日分の工数実績 22件、8h/日」という**業務の言葉の要約**と行プレビューを見て1回で決裁 ④実行は CSV コネクタと同じ `importRecords` が担い、リンクの名前解決（「アクメ商事」→ Customer の id）・行検証・監査ログまで通常の取込と同一です。

LLM に任せるのは行の組み立てまで。検証と書き込みは決定的なコードが行う。この分担なら、大量登録でも安全装置は1つも減りません。

## プロンプトインジェクションの被害半径

以上を重ねると、プロンプトインジェクションへの答えが構造から出てきます。このエージェントは `tools: []` で SDK の組み込みツール（ファイル・Bash 等）を全無効化し、オントロジー由来のツールだけを見ています。読み取ったデータに悪意ある指示が混ざり、エージェントが完全に騙されたとしましょう。それでも実行できる操作は、**YAML に定義済みのアクションの集合**から一切広がりません。任意のクエリも、任意のファイル操作も、手段そのものが存在しない。そのうえ破壊的なアクションには `confirm` で人間の承認が挟まり、すべての実行が追記専用の監査ログに残ります。

つまり被害半径は「定義済みアクション × agentPolicy × ロール権限」に**上から拘束**されます。プロンプトをどれだけ上手に守っても得られないのは、この「最悪ケースの上限」です。祈りは破られる前提で、破られても越えられない壁を構造で引いておく。エージェントに業務を任せる条件は、これだと考えています。

## まとめ

* 禁止のプロンプトは傾向であって保証ではない。安全装置はエージェントの外側の構造に置く
* 書き込みは UI・エージェント・MCP すべて Action Engine の単一経路。抜け道を作らない
* 承認は `agentPolicy: confirm` の1行で宣言し、`canUseTool` ＋エンジン側検査の二重ゲートで強制。実行は決裁した人間の権限で
* 状態機械と precondition は理由付きで拒否し、拒否理由がエージェントの「次の一手」の材料になる
* 大量登録でも SQL は与えない。承認とは人間が意味を理解して判断できること
* 被害半径は「定義済みアクションの範囲」に構造で拘束される
