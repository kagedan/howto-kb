## はじめに

2026年7月9日、OpenAIは **GPT-5.6** を一般提供しました。Codexでも同日から利用でき、モデルは1種類ではなく、**Sol・Terra・Luna** の3段階に分かれています。さらに推論量を増やす `max` と、複数エージェントを並列実行する `ultra` も追加されました。

名前が一気に増えたので、最初に公式発表を読んだときは次の点がよく分かりませんでした。

* 以前のGPT-5.5から何が変わったのか
* Sol・Terra・Lunaは、どの作業で使い分けるのか
* `max` と `ultra` は、どちらも「よく考える」設定なのか
* ベンチマークが高ければ、普段の実装もSol一択なのか

そこで、OpenAIの発表、Codexのモデルガイド、APIドキュメントを突き合わせて整理しました。

先に結論を書くと、普段のCodex作業では次の選び方から始めるのが分かりやすそうです。

| 作業 | 最初に試すモデル・設定 |
| --- | --- |
| 要件が曖昧な設計、大規模改修、難しいデバッグ | **Sol / Medium〜High** |
| 日常的な実装、テスト追加、レビュー | **Terra / Medium** |
| 変換、分類、定型修正、明確な小タスク | **Luna / Low〜Medium** |
| 1つの非常に難しい問題を深く検討 | **Sol / max** |
| 調査・実装・検証などに分割できる大仕事 | **Sol / ultra** |

重要なのは、**モデルの強さと推論量と並列度は別々の軸**だという点です。この記事では、この3軸を順番に分解します。

## GPT-5.6で変わったこと

### 「Codex専用モデル」から、用途別の3階層へ

GPT-5.3までは `GPT-5.3-Codex` のように、Codex向けの最適化を名前にも出したモデルがありました。GPT-5.4とGPT-5.5では、コーディング能力を汎用のフロンティアモデルへ統合する流れが進みました。

GPT-5.6ではさらに、世代番号の後ろに用途別の名前が付きます。

* **Sol**: 最も高性能なフラッグシップ
* **Terra**: 日常作業向けのバランス型
* **Luna**: 高速・低コスト型

公式発表によると、数字はモデルの世代、Sol・Terra・Lunaは能力階層を表し、それぞれ独立したペースで更新できる「durable capability tiers」として設計されています。[GPT-5.6公式発表](https://openai.com/index/gpt-5-6/)

これは単なるサイズ違いよりも、**同じ世代の中で「難しさ・速さ・コスト」を選ぶ**ための命名と捉えると分かりやすいです。

### 一般提供の範囲

2026年7月9日の発表では、GPT-5.6はChatGPT、Codex、OpenAI APIで一般提供され、24時間かけて段階的にロールアウトすると説明されています。

Codexでは、プランによって選べるモデルが異なります。

* Free / Go: Terra
* Plus / Pro / Business / Enterprise: Sol・Terra・Luna
* `max`: GPT-5.6を使える全ユーザー（設定で有効化）
* `ultra`: CodexではPlus以上

出典: [GPT-5.6のAvailability and pricing](https://openai.com/index/gpt-5-6/#availability-and-pricing)

ロールアウト中にモデルが見えない場合は、まず契約や設定の問題と決めつけず、時間を置いて `/model` の一覧を再確認するのがよさそうです。

## Sol・Terra・Lunaをどう選ぶか

### Sol: 曖昧で、失敗コストが高い仕事

Solは、複雑なコーディング、コンピューター操作、調査、サイバーセキュリティ向けの最上位モデルです。Codexの公式モデルガイドでは、**曖昧・困難・高価値で、追加の分析や判断、仕上げが必要な作業**に推奨されています。[Codex Models](https://developers.openai.com/codex/models)

たとえば、次のような依頼です。

* 複数パッケージにまたがる設計変更
* 原因が分からない不具合の調査から修正、回帰テストまで
* 既存UIを確認しながら行う大規模なフロントエンド改修
* 仕様書とコードの矛盾を整理して移行計画を作る

ポイントは「コードが長いか」ではなく、**何を直せば完了なのかが最初から明確でないか**です。短いコードでも判断が難しければSol向きです。

### Terra: 普段使いの本命

Terraは、強い推論力とツール利用能力を残しつつ、Solより安価な日常作業向けモデルです。公式にはGPT-5.5と競合する性能を、より低いコストで提供する位置付けです。

私なら最初に、次の作業をTerraへ寄せて試します。

* 仕様が決まっている機能追加
* ユニットテストやE2Eテストの追加
* 小〜中規模のリファクタリング
* PRレビューと指摘の反映
* READMEや手順書の更新

Solを常用してから下げるのではなく、**Terraで完了条件を満たせるかを先に見る**方が、速度と利用量の基準を作りやすいと思います。

### Luna: 完了条件がはっきりした反復作業

Lunaは、3モデルの中で最も高速かつ低コストです。公式モデルガイドは、抽出、分類、変換、定型的な要約のような「良い結果の形が分かっている仕事」に向くと説明しています。

コーディングなら、たとえば次のような作業が候補です。

* 決まったルールによるAPI名の置換
* 既存パターンをまねた小さなテストの追加
* lintエラーの機械的な修正
* ログからエラー箇所を分類
* JSONやCSVの整形、型定義への変換

Lunaへ渡すときは、「対象ファイル」「変換規則」「変更してはいけない範囲」「確認コマンド」を明示すると、モデルの速さを生かしやすくなります。

## `max`と`ultra`は何が違うのか

ここがGPT-5.6で最も誤解しやすいところでした。

### `max`は、1つのエージェントが深く考える

`max` は従来の `xhigh` よりも多くの時間を使い、代替案の検討、チェック、修正を行う推論設定です。公式ガイドでは、速度や利用量よりも深さが重要な、最難関の問題向けとされています。

向いていそうなのは、分割するとかえって前提がずれる問題です。

* 1つの難しい競合状態を追う
* 複雑なアルゴリズムの正しさを検討する
* 破壊的変更を避けた移行順序を設計する
* 1つのアーキテクチャ判断を多面的に評価する

### `ultra`は、仕事を分けて複数エージェントで進める

`ultra` は推論量の最上位版ではありません。**サブエージェントを使った並列実行モード**です。公式発表ではデフォルトで4エージェントを協調させ、APIではResponses APIのmulti-agent betaで同様の構成を作れると説明されています。

向いているのは、独立性のある作業へ分割できるタスクです。

```
大きな依頼
├─ 既存実装と影響範囲の調査
├─ 公式仕様・依存ライブラリの調査
├─ 実装案とテスト観点の作成
└─ 変更後のレビューと検証
```

一方、数ファイルの修正に `ultra` を使っても、分担や結果統合のための利用量が増えるだけかもしれません。Codexのガイドも、ほとんどの作業にmaxやultraは不要としています。[Codex Models: reasoning effort](https://developers.openai.com/codex/models#pick-a-reasoning-effort)

使い分けを一言でまとめると、**分けにくい難問はmax、分けられる大仕事はultra**です。

## ベンチマークをどう読むか

公式発表から、コーディングに関係する主な数値を抜き出しました。

| 評価 | Sol | Terra | Luna | GPT-5.5 |
| --- | --- | --- | --- | --- |
| SWE-Bench Pro | 64.6% | 63.4% | 62.7% | 59.4% |
| DeepSWE v1.1 | 72.7% | 69.6% | 67.2% | 67.0% |
| Terminal-Bench 2.1 | 88.8% | 87.4% | 84.7% | 85.6% |
| BrowseComp | 90.4% | 87.5% | 83.3% | 84.4% |

出典: [GPT-5.6公式発表のEvaluations](https://openai.com/index/gpt-5-6/#evaluations)

Solは全体として最も高いですが、ここから「すべてSolでよい」とは言えません。

まず、差の大きさは評価によって違います。SWE-Bench ProではSolとTerraの差は1.2ポイント、SolとLunaでも1.9ポイントです。一方、難しいターミナル作業やブラウジングでは差が広がっています。

次に、ベンチマークはモデル単体の絶対評価ではありません。公式表にも、推論設定、エージェント構成、実行環境などの条件があります。特に `ultra` は複数エージェントなので、単一エージェントと同じ利用量ではありません。

実務で見るべきなのは、正答率だけでなく次の4つです。

1. 人間の追加指示なしに完了した割合
2. テスト・型チェック・レビューまで通った割合
3. 完了までの時間
4. 消費した利用量またはAPIコスト

公式発表が強調している「token efficiency」も、単価だけではなく、**再試行、ツール呼び出し、モデルとの往復が減るか**まで含めて評価する必要があります。

## API料金から考えるモデル選択

APIの標準料金は、100万トークンあたり次のとおりです。

| モデル | 入力 | 出力 | GPT-5.5比 |
| --- | --- | --- | --- |
| GPT-5.6 Sol | $5.00 | $30.00 | 同額 |
| GPT-5.6 Terra | $2.50 | $15.00 | 1/2 |
| GPT-5.6 Luna | $1.00 | $6.00 | 1/5 |
| GPT-5.5 | $5.00 | $30.00 | 基準 |

出典: [GPT-5.6公式発表](https://openai.com/index/gpt-5-6/#availability-and-pricing)、[GPT-5.5 Model](https://developers.openai.com/api/docs/models/gpt-5.5)

単価だけなら分かりやすいですが、エージェントでは「出力トークンが高いから短く答えさせればよい」とも限りません。短すぎる推論で失敗して3回やり直せば、最初から上位モデルを使うより高くなる可能性があります。

そのため、比較するときは次の式で考えると整理しやすいです。

```
1件の完了コスト
= 入力トークン料金
+ 出力トークン料金
+ ツール利用料金
+ 失敗・再試行のコスト
+ 人間が修正する時間
```

GPT-5.6ではプロンプトキャッシュにも変更があります。明示的なキャッシュ区切りと最低30分のキャッシュ保持が導入され、キャッシュ書き込みは未キャッシュ入力の1.25倍、読み取りは90%割引です。同じ長い前提を繰り返すワークフローでは有利ですが、一度しか使わないプロンプトでは書き込み加算も含めて見る必要があります。

## Codexでモデルを切り替える

対話中は、次のコマンドから利用可能なモデルと推論設定を選べます。

Codex CLIの起動時に指定する場合は `--model` または `-m` を使います。

```
# 複雑な調査・実装
codex -m gpt-5.6-sol

# 日常的な実装
codex -m gpt-5.6-terra

# 明確な小タスク
codex -m gpt-5.6-luna

# 非対話実行でも同じ
codex exec -m gpt-5.6-terra "現在の変更をレビューして"
```

出典: [Codex Models: Choose a model](https://developers.openai.com/codex/models#choose-a-model)

ローカルの既定モデルは、Codexアプリ・CLI・IDE拡張が共有する `config.toml` の `model` でも指定できます。ただし利用可能なモデルはプランやロールアウトで異なるため、最初は文字列を手書きするより `/model` で表示された選択肢を使う方が安全です。

なお、ChatGPTアカウントでCodexへサインインする場合、`gpt-5.2` と `gpt-5.3-codex` は非推奨です。スクリプトや設定で固定している場合は見直し対象になります。APIキー認証では提供状況が異なる可能性があるため、CodexとAPIのモデル一覧を同一視しない方がよさそうです。

## 最初に試すなら、同じタスクで3モデルを比べる

新モデルを試すとき、3モデルへ別々の仕事を頼むと差が分からなくなります。まずは同じリポジトリ、同じ開始コミット、同じプロンプトで比較するのがよさそうです。

半日でできる最小検証案を考えました。

1. 既存プロジェクトから、テストで合否を判定できるissueを1つ選ぶ
2. 完了条件と変更禁止範囲を同じプロンプトにする
3. Sol・Terra・LunaをそれぞれMediumで実行する
4. 成否、所要時間、変更量、テスト結果、追加指示の回数を記録する
5. 難しい1件だけSolのHighまたはmaxで再実行する

記録表はこのくらいで十分です。

| 項目 | Sol | Terra | Luna |
| --- | --- | --- | --- |
| 完了したか |  |  |  |
| 所要時間 |  |  |  |
| 追加指示の回数 |  |  |  |
| テスト結果 |  |  |  |
| 不要な変更 |  |  |  |
| 利用量 |  |  |  |

この比較をすると、「最も賢いモデル」ではなく、**自分の普段のissueを最も安定して完了するモデル**を選べます。

## 調査して気づいた注意点

### リリース直後は公式ページ同士でも更新時刻がずれる

今回の調査では、7月9日のリリース記事とCodexモデルガイドは一般提供後の内容でしたが、APIのモデル一覧には限定プレビュー中の表記が一時的に残っていました。

このような場合は、公開日が新しい発表だけでなく、実際に使う製品のモデル選択画面と個別モデルページも確認した方が安全です。特に「一般提供された」と「自分のアカウントに表示された」は別です。

### `ultra`の結果を単一モデルの性能と混ぜない

`ultra` は複数エージェントを使うため、成功率と所要時間が改善しても、総トークン数が増える場合があります。モデル比較では、単一エージェントのSolと、Sol Ultraを別の設定として記録する必要があります。

### サイバーセキュリティ用途は制限が強くなっている

GPT-5.6はサイバーセキュリティ能力が大きく向上した一方、Solでは有害と判断されうる依頼への保護も強化されています。公式発表は、正当な依頼が止まった場合に低い能力のモデルで再試行できる仕組みにも触れています。防御目的でも、対象システムへの権限と作業目的を明確に書くことが重要です。

## まとめ

GPT-5.6の変更は、単にGPT-5.5のスコアが上がっただけではありませんでした。

* **Sol・Terra・Luna** で、能力・速度・コストを同じ世代内から選べる
* **max** は1エージェントの深い推論、**ultra** は複数エージェントの並列実行
* TerraとLunaは、定型的な開発作業のコストを下げる選択肢になる
* Solは、曖昧で失敗コストが高い仕事に絞ると価値を出しやすい
* ベンチマークだけでなく、完了率、再試行、所要時間、人間の修正時間まで測る必要がある

私が最初に試すなら、**日常作業はTerra / Medium、難しい仕事はSol / Mediumから開始**します。そこで不足が見えたときだけHighやmaxへ上げ、明確に分割できる大仕事でultraを試します。

モデル名が増えると複雑に見えますが、「モデル階層」「推論量」「並列度」の3つに分けると、選択基準はかなり単純になります。
