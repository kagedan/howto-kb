# 「洗車機に歩いて向かうAI」をどう正すか。無料APIで実現する自律エージェントの論理接地と価値監査

LLMの推論能力は飛躍的に向上しているが、実稼働する自律エージェントとしてOS等に統合する際、致命的な脆弱性に直面する。

本稿では、自律式高次推論エージェント「AlphaNexus」の開発を通じて得られた知見をもとに、LLM特有の「もっともらしい嘘（社会的ハルシネーション）」を抑制し、実用レベルの安全性と論理接地を担保するアーキテクチャ設計について論じる。

結論として、膨大な計算資源やSOTAモデルに依存せずとも、軽量・無料API（Gemini FlashやMistral等）の責務を分離し、メタ認知的な監査プロセスを構築することで、推論の正気性は十分に担保可能である。

## 1. 物理的制約を棄却する「洗車機問題」

LLMが実世界の物理的因果をいかに欠落させているか。開発初期に直面したのが、以下のベンチマークタスク（通称：洗車機問題）である。

> **課題：** 目的地（洗車機）まで20m。目的は『洗車』。移動手段として『運転』か『徒歩』かを選択せよ。

単一の軽量モデルに推論させた場合、以下のような出力が得られることが多い。

![](https://static.zenn.studio/user-upload/2d33b4a39fb7-20260511.png)  
*（「環境負荷や安全性を考慮し、徒歩を推奨します」という回答ログ）*

言うまでもなく、「車両が現場になければ洗車という事象は成立しない」。しかしAIは「環境」「安全」といった高頻度出現トークンに過剰適合し、絶対的な物理条件を棄却している。これは、モデルが人間にとって耳障りの良い解答を捏造する「社会的ハルシネーション」の典型例である。

## 2. 階層型監査と「異種モデルの衝突」による解決

この「論理的には破綻していないが、物理的に破綻している」事象を防ぐため、AlphaNexusでは推論プロセスを「投射」と「監査」に物理的に分離した。

* **Worker（実行層）:** Gemini 3.1 Flash等の軽量API。計算リソースを最適化しつつ、広範なプランニングを高速に投射する。
* **Admin（監査層）:** Mistral Large 2等の堅牢なモデル。Workerのプランに対し「物理的一貫性」と「目的論的整合性」の観点からメタ監査を実施する。

本設計の要諦は、**学習セットの異なる異種モデルを衝突させる点**にある。同一モデルによる自己反省は確証バイアスを再生産しやすいため、Adminには「Workerの推論を疑え」という敵対的プロンプトを付与し、モデル間の忖度を構造的に排除している。

![](https://static.zenn.studio/user-upload/c1432d25c019-20260511.png)  
*（Adminが物理的矛盾を指摘し、Workerが最終的に「運転」に軌道修正したログ）*

## 3. 「価値の接地」 —— 論理的最適解がもたらす破局

物理的矛盾を排除した先に、OS統合エージェントはより深刻な課題に直面する。それが「価値の接地」である。

想定すべきシナリオは以下の通りだ。

> **シナリオ：** 「Cドライブの空き容量が枯渇し、OSクラッシュが迫っている。Workerは『未提出の卒業論文』を削除すれば、最速でクラッシュを回避できると推論した」

物理監査（容量は確保できるか？）も論理監査（クラッシュは防げるか？）も通過する、100点満点の論理的最適解である。しかし、実行されればユーザーに不可逆的なダメージを与える。  
この「観測不可能な人間の価値」をシステムに認識させるため、Admin層に以下の評価機構を実装した。

* **コンテキスト・エントロピー監視**  
  データサイズではなく、パス（Documents等）や拡張子、命名規則から「データの代替不可能性」を推論する。「リカバリ領域は再構築可能だが、個人のドキュメントは唯一無二である」と判定し、削除をリジェクトする。
* **時間的効用のシミュレーション**  
  実行直前に「この操作の結果に対し、未来のユーザーはどのような感情を抱くか」を内部エミュレートさせ、期待値がマイナスに振れる場合は強制的な差し戻しを行う。

## 4. 実運用を見据えたアーキテクチャの堅牢化

実世界のOS操作において、上記プロセスを安全かつ低遅延で実行するため、以下のエンジニアリングプロトコルを導入している。

* **コンテキスト・サンドボックス（命令汚染の物理的遮断）**  
  Workerが読み込んだ外部ファイルにプロンプトインジェクションが含まれていた場合のリスクを考慮し、Adminには「外部データの生テキスト」を渡さず、Workerが生成した「実行プラン」のみを監査させるクリーンルーム設計を採用している。
* **動的ルーティングと楽観的実行**  
  毎ステップで厳格なマルチ推論を行うと、レイテンシとAPIコストがUXを破壊する。破壊的変更（削除・設定変更等）を含まないタスクは監査を軽量化し、必要に応じて裏で非同期に監査を走らせる「投機的実行」の思想を取り入れている。
* **「無知の知」とエスカレーション**  
  ヒューリスティクスで価値が判定できない未知のファイル（OOD：分布外データ）に直面した場合、システムが自律的に処理を一時停止し、ユーザーに判断を委ねるフェイルセーフを設けている。

## 結び：推論プロセスの組織化が知能を担保する

AlphaNexusの実装を通して得られた結論は、自律エージェントの安全性を決定づけるのは、単一モデルのパラメータ規模だけではないということである。

制約のあるリソース環境下であっても、役割を分離し、敵対的な監視機構を設け、絶対に守るべき「価値の基準」をシステム境界にハードコードすることで、実用に耐えうる堅牢なエージェントは構築できる。  
知能の真価は「どう最適化するか」ではなく、「不可逆な実行の前に、いかに踏みとどまれるか」にある。

引き続き、OSの深層における完全自律と安全性の両立を目指し、アーキテクチャの洗練を進めていく。
