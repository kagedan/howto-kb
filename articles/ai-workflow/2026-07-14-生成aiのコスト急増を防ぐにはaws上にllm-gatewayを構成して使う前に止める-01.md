## 1. はじめに

生成AIの業務利用が広がる中で、最近は「AIをどう活用するか」だけでなく、**AI利用にかかるコストをどう管理するか**も重要なテーマになってきています。

たとえば、以下のようなケースです。

* 気づかないうちにLLMのトークン利用料が増えている
* 個人・チーム・アプリごとの利用量が見えづらい
* 高額なモデルを誰でも自由に使えてしまう
* 予算上限を超えてから初めて気づく
* 特定のLLMプロバイダーやモデルが混雑・制限されたときに代替手段がない

自分も当初、AWS（Amazon Web Services）であれば以下のような仕組みで、ある程度対応できるのではないかと考えていました。

* Amazon BedrockのCountTokens APIで、推論前に入力トークン数を確認する
* Amazon BedrockのModel Invocation Loggingで、モデル呼び出しログを取得する
* AWS Budgetsで、予算超過・予測超過を通知する
* Cost Anomaly Detectionで、異常なコスト増を検知する
* AWS FinOps Agentで、コスト異常の原因調査やSlack/Jira連携を行う

これらは有効な仕組みではあります。一方で、**これらは主に監視・分析・通知・事後調査のための仕組み**です。以下のような課題に対しては**リアルタイムに制御する方法が必要**となります。

* 「一定の予算を超えたら、その場でLLM呼び出しを止めたい」
* 「高額モデルへのリクエストを送信前に拒否したい」
* 「1分あたりのリクエスト数やトークン数を超えたら即時ブロックしたい」

その解決策として本記事で紹介するのは、「**Multi-Provider Generative AI Gateway on AWS**」です。  
<https://docs.aws.amazon.com/solutions/multi-provider-generative-ai-gateway-on-aws/>

## 2. AWSの標準機能での対応

生成AIのコスト管理という観点では、AWSにはいくつか有効な仕組みがあります。たとえば、Amazon BedrockのCountTokens APIを使えば、推論前に入力トークン数を確認できます。また、Request Metadataを使うことで、リクエスト単位でユーザーやチームなどの情報を付与できます。さらに、Model Invocation Loggingを有効化すれば、モデル呼び出しログをCloudWatch LogsやS3に出力し、後から利用量を分析できます。

加えて、AWS BudgetsやCost Anomaly Detectionを使えば、予算超過や異常なコスト増を検知できます。AWS FinOps Agentを活用することで、コスト異常の原因調査やSlack/Jiraへの連携といった運用面の効率化も期待できます。

| 機能 | 主な役割 | できること |
| --- | --- | --- |
| CountTokens API | 推論前の入力トークン確認 | 入力プロンプトのトークン数を事前に見積もる |
| Model Invocation Logging | モデル呼び出しログの取得 | 入出力トークン数や利用状況を後から分析する |
| CloudWatch / S3 / Athena | ログの蓄積・分析 | アプリ別・ユーザー別・モデル別の利用量を可視化する |
| AWS Budgets | 予算管理 | 予算超過や予測超過を通知する |
| Cost Anomaly Detection | 異常検知 | 通常とは異なるコスト増を検知する |
| AWS FinOps Agent | コスト異常の調査支援 | 異常原因の調査やSlack/Jira連携を支援する |

### CountTokens API：推論前に入力トークン数を確認する

Amazon Bedrockでは、CountTokens APIを使うことで、推論前に入力トークン数を確認できます。

これにより、たとえば以下のような制御が可能になります。

* 入力トークン数が大きすぎるリクエストを事前に検知する
* 入力トークン数をもとに、おおよそのコストを見積もる
* アプリケーション側で入力上限を設ける

ただし、CountTokens APIで確認できるのは、基本的には推論前の入力トークン数です。最終的なコストには出力トークン数も影響しますが、出力トークン数は実際にモデルが応答を生成するまで確定しません。

そのため、CountTokens APIは非常に有効ですが、単体で「最終的な利用料金を完全に制御する仕組み」ではありません。

なお、CountTokens API はすべてのモデルで同じように利用できるわけではありません。一部の Anthropic Claude モデルなどでは、bedrock-runtime の CountTokens がサポートされない場合があります。実際に利用するモデル・リージョンでサポート状況を確認しておく必要があります。

Amazon Bedrockでは、推論リクエストごとにRequest Metadataを付与できます。  
たとえば、リクエスト時に以下のような情報を付与します。

* user
* team
* application
* environment
* feature
* project

InvokeModel 系のAPIでは X-Amzn-Bedrock-Request-Metadata ヘッダーを使い、Converse 系のAPIでは requestMetadata フィールドを使って、リクエスト単位のメタデータを付与できます。  
これにより、Model Invocation Loggingに出力されるログに、リクエスト単位のメタデータを含めることができます。

たとえば、リクエストに user や team を付与しておけば、CloudWatch Logs InsightsやAthenaで以下のような集計ができます。

* ユーザー別の呼び出し回数
* チーム別の入力トークン数
* チーム別の出力トークン数
* モデル別の利用量
* アプリケーション別の利用傾向
* ユーザー・チーム単位の推定コスト

ただし、ここで算出できるのは、あくまでログ上のトークン数とモデル単価をもとにした推定コストです。

Request Metadataは、リクエストログの分析には有効ですが、Cost ExplorerやCost and Usage Reportのコスト配分タグとしてそのまま扱われるものではありません。そのため、請求上の正確な金額確認には、Cost and Usage Reportなどの請求データとあわせて確認する必要があります。

また、Request Metadataには個人情報や認証情報などの機微情報を入れないように注意が必要です。ユーザーを識別したい場合も、メールアドレスなどを直接入れるのではなく、社内ID、ハッシュ化したID、アプリケーション側のユーザーIDなどを使う設計が望ましいです。

### Model Invocation Logging：モデル呼び出しログを取得する

Amazon Bedrockでは、Model Invocation Loggingを有効化することで、モデル呼び出しログをCloudWatch LogsやS3に出力できます。

これにより、後から以下のような分析ができます。

* どのモデルがどれだけ使われたか
* どのアプリケーションから呼び出されたか
* 入力トークン数・出力トークン数がどの程度だったか
* 利用量の多いユースケースは何か

### AWS Budgets：予算超過を検知する

![](https://static.zenn.studio/user-upload/e1b5a8c673bb-20260711.png)  
AWS Budgetsを使うと、AWS利用料が設定した予算を超過した場合や、予算を超過しそうな場合に通知できます。たとえば、以下のような使い方が考えられます。

* 月額のLLM関連予算を設定する
* 予算の80%に到達したら通知する
* 予算の100%に到達したら通知する
* 特定のアカウントやタグに絞って予算を管理する

LLM利用が広がると、気づかないうちにコストが増えていく可能性があります。そのため、AWS Budgetsによる予算アラートは基本的な対策として有効です。

ただし、AWS Budgetsは基本的には請求情報や利用実績に基づく通知の仕組みです。つまり、予算超過を検知して通知することはできますが、LLMリクエストが送信される瞬間に、そのリクエストを止める仕組みではありません。

なお、AWS Budgets には Budget Actions もあり、しきい値を超えた場合に IAM ポリシーや SCP の適用、特定の EC2 / RDS インスタンスに対するアクションを実行できます。ただし、これは LLM リクエスト単位で、プロバイダーに送信する直前に許可・拒否を判定する仕組みではありません。そのため、生成AI利用をリアルタイムに制御したい場合は、アプリケーション側や Gateway 側での制御が必要になります。

### Cost Anomaly Detection：異常なコスト増を検知する

![](https://static.zenn.studio/user-upload/7b6f852071d4-20260711.png)

Cost Anomaly Detectionを使うと、通常とは異なるコスト増を検知できます。  
たとえば、以下のようなケースに有効です。

* ある日突然、Bedrock関連のコストが急増した
* 特定のアカウントで想定外の利用が発生した
* 通常の利用傾向から外れたコストが発生した
* アプリケーションのバグにより大量のLLM呼び出しが発生した

通常の予算管理では、予算超過に達するまで気づきづらい場合があります。一方で、Cost Anomaly Detectionを使うことで、普段と異なるコスト増に早く気づける可能性があります。

ただし、これも基本的には異常が発生した後に検知する仕組みです。異常なリクエストそのものを、LLMに送信される前にブロックするものではありません。

また、Cost Anomaly Detectionはリアルタイム検知ではありません。Cost Explorerのデータを利用するため、利用発生から異常検知までに最大24時間程度かかる可能性があります。

### AWS FinOps Agent：コスト異常の調査・通知を支援する

![](https://static.zenn.studio/user-upload/8a73eb7434eb-20260711.png)  
AWS FinOps Agent（Preview）は、コスト異常の調査や運用連携を支援するエージェントです。

コスト異常を検知した後に、関連する情報を調査し、原因を整理したり、Slack通知やJira起票のような運用アクションにつなげたりする用途が考えられます。これは、生成AI利用が増えた後のFinOps運用において有効です。

一方で、FinOps Agentも基本的には、発生したコスト異常を調査・通知・運用連携するための仕組みです。LLMリクエストの入口で、1件ごとに許可・拒否を判定する仕組みではありません。

### AWSの標準機能でできること／できないこと

ここまでの内容を整理すると、AWS標準機能を使うことで、LLM利用に対して多くの管理が可能です。

* 推論前に入力トークン数を確認する
* リクエスト単位でユーザー・チーム・アプリなどのメタデータを付与する
* LLM呼び出しログを取得する
* ユーザー別・チーム別・アプリ別・モデル別の利用量を後から分析する
* 入出力トークン数をもとに推定コストを算出する
* 予算超過や予測超過を通知する
* 異常なコスト増を検知する
* コスト異常の原因調査や運用連携を支援する

一方で、これらは主に監視・分析・通知・事後調査のための仕組みです。以下のような制御には、これだけでは不十分です。

* このリクエストをLLMに送る前に止める
* チーム予算を超えた瞬間に、その後のリクエストを拒否する
* 高額モデルへのアクセスをGateway側で拒否する
* RPMやTPMを超えるリクエストを即時ブロックする
* バグや無限ループによる大量呼び出しを入口で止める

つまり、AWS BudgetsやCost Anomaly Detection、FinOps Agentは「**使いすぎに気づく**」**ための仕組みとして有効**です。また、Request MetadataとModel Invocation Loggingを組み合わせることで、「誰が、どのモデルを、どれだけ使ったか」を可視化することもできます。

しかし、使わせる前に止めるためには、**LLMプロバイダーにリクエストが届く前に判定する仕組みが必要**です。

そこで次に登場するのが、Multi-Provider Generative AI Gateway on AWSのようなGateway構成です。

## 3. Multi-Provider Generative AI Gateway on AWS

Multi-Provider Generative AI Gateway on AWS は、複数のLLM プロバイダーへのアクセスを共通の入口に集約し、認証、モデルアクセス制御、利用量管理、レート制限、監査、可観測性を一元的に扱うためのリファレンスアーキテクチャです。AWS のマネージドサービスとして提供される単体サービスではなく、LiteLLM を中心にAWS 上へ構成するGateway パターンとして理解するとよいです。

### Multi-Provider Generative AI Gateway on AWSとは

Multi-Provider Generative AI Gateway on AWSは、AWS上にLLM Gatewayを構築するためのリファレンスアーキテクチャです。  
アプリケーションからAmazon BedrockやOpenAI、Anthropic、Vertex AIなどのLLMプロバイダーを直接呼び出すのではなく、**Gatewayを経由して呼び出す**構成になります。

これにより、複数のLLMプロバイダーを統一的なAPIで利用しつつ、利用量管理、コスト管理、モデルアクセス制御、レート制限、ログ取得、ルーティングなどを一元的に実装できます。

ポイントは、**LLMを呼び出す入口をGatewayに集約すること**です。  
これにより、各アプリケーションが個別にLLMを直接呼び出すのではなく、Gatewayで共通のルールを適用できます。

### システム構成図

システム構成図は以下の通りです。この構成では、利用者やアプリケーションからのリクエストは、Route 53やCloudFront、AWS WAF、Application Load Balancerを経由して、ECSまたはEKS上で動作するLiteLLMベースのGatewayに到達します。

Gateway は、Amazon Nova を含む Amazon Bedrock 上の基盤モデルや、Amazon SageMaker AI 上のモデルに加えて、OpenAI、Anthropic、Vertex AI などの外部 LLM プロバイダーにも接続できます。

![](https://static.zenn.studio/user-upload/2e461372bd85-20260712.png)

各AWSサービスの役割を整理すると、以下のようになります。

| 構成要素 | 主な役割 |
| --- | --- |
| Route 53 / CloudFront | Gatewayへのアクセス入口 |
| AWS WAF | Web攻撃や不正アクセスへの防御 |
| Application Load Balancer | Gatewayコンテナへのリクエスト振り分け |
| ECS / EKS | LiteLLM Gatewayをコンテナとして実行 |
| Amazon Bedrock / Amazon Nova / SageMaker | AWS上のLLM・モデル実行基盤 |
| OpenAI / Anthropic / Vertex AIなど | 外部LLMプロバイダー |
| Amazon RDS | Virtual API KeyやGateway設定の永続化 |
| ElastiCache Redis | キャッシュや利用量制御の高速化 |
| AWS Secrets Manager | 外部プロバイダーのAPIキーなどを安全に管理 |
| Amazon S3 | Gatewayのログ保存・分析 |

### Gatewayでリアルタイムにブロック

Gatewayが**LLMプロバイダーへリクエストを送る前に判定する**ため、必要に応じてリアルタイムにLLMへの通信をブロックできます。

AWS BudgetsやCost Anomaly Detectionは、基本的には利用後のコストや請求データをもとに検知します。一方で、Gateway構成では、リクエストの入口で以下のような判定を行います。

* ユーザーやチームは利用可能か
* 指定されたモデルを使ってよいか
* チームやAPIキーの予算を超えていないか
* RPMやTPMの上限を超えていないか
* 並列リクエスト数を超えていないか

条件を満たさない場合、Gatewayがその場で拒否します。つまり、**LLMプロバイダーにリクエストが届かないため、推論コストが発生しない**という点が重要です。

### リアルタイムブロックでできる制御

Gatewayを使うことで、以下のような制御が可能になります。

| 制御 | 内容 | ブロック例 |
| --- | --- | --- |
| APIキー制御 | Gatewayが発行したVirtual API Key単位で利用を管理 | 無効なキーや期限切れキーを拒否 |
| ユーザー制御 | ユーザー単位で利用量や予算を管理 | 個人予算を超えたら拒否 |
| チーム制御 | 部署・プロジェクト単位で共有予算を管理 | チーム予算を超えたら拒否 |
| モデルアクセス制御 | 利用可能なモデルを制限 | 一般ユーザーの高額モデル利用を拒否 |
| RPM制限 | 1分あたりのリクエスト数を制限 | 短時間の大量リクエストを拒否 |
| TPM制限 | 1分あたりのトークン数を制限 | 大量トークン消費を拒否 |
| 並列数制限 | 同時リクエスト数を制限 | バースト的な呼び出しを拒否 |
| ルーティング制御 | 条件に応じてモデルやプロバイダーを切り替え | 高額モデルではなく軽量モデルへ切替 |

以降では、特に重要な制御をいくつか見ていきます。

### APIキー単位で制御する

Gateway構成では、利用者やアプリケーションにLLMプロバイダーの本物のAPIキーを直接渡すのではなく、Gatewayが発行する**Virtual API Key**を使わせます。この方式にすると、Gateway側で以下を管理できます。

* APIキーごとの利用量
* APIキーごとの予算
* APIキーごとの利用可能モデル
* APIキーの有効期限
* APIキーの無効化
* APIキーごとのレート制限

たとえば、検証用アプリには安価なモデルだけを許可し、月額上限を小さく設定するといった制御ができます。

```
key: app-dev-key
team: poc-team
max_budget: 50 USD / month
allowed_models:
  - amazon-nova-lite
  - claude-haiku
rpm_limit: 30
tpm_limit: 50000
```

このキーが月額上限を超えた場合、GatewayはLLMプロバイダーにリクエストを転送せずに拒否します。

### チーム・プロジェクト単位で予算を制御する

企業利用では、個人単位だけでなく、チームやプロジェクト単位の予算管理が重要です。  
たとえば、以下のようにプロジェクトごとに月額上限を設定します。

Project Aの累計利用額が上限に達した場合、Project Aに紐づくAPIキーやユーザーからのリクエストを拒否できます。

この場合、リクエストはGatewayで止まるため、LLMプロバイダー側の推論コストは発生しません。

2章で紹介したAWS Budgetsは、予算超過を検知して通知する仕組みです。一方、Gatewayによる予算制御は、**予算を超えた後のリクエストを入口で拒否する**という点が異なります。

### 高額モデルへのアクセスを制御する

LLMコスト管理で特に重要なのが、**モデルアクセス制御**です。

すべてのユーザーが高額モデルを自由に使える状態だと、想定以上にコストが膨らむ可能性があります。

たとえば、以下のようなルールを設定できます。

| 利用者区分 | 利用可能モデル |
| --- | --- |
| 一般ユーザー | 軽量モデルのみ |
| 開発チーム | 中価格帯モデルまで |
| 承認済みチーム | 高性能モデルも利用可 |
| PoC環境 | 安価なモデルのみ |
| 本番環境 | 承認済みモデルのみ |

図にすると以下です。

一般ユーザーが高額モデルを指定した場合、Gatewayがその場で拒否します。  
この制御により、利用者が意図せず高額モデルを大量に使ってしまうリスクを減らせます。

### RPM / TPM / 並列数で制御する

LLM利用では、月額予算だけでなく、短時間に大量リクエストが発生することも問題になります。  
たとえば、アプリケーションのバグや無限ループにより、LLM呼び出しが急増するケースです。  
このような場合に有効なのが、以下の制限です。

* RPM: Requests Per Minute
* TPM: Tokens Per Minute
* 最大並列リクエスト数

たとえば、以下のような設定イメージです。

```
team_id: project-a
rpm_limit: 100
tpm_limit: 100000
max_parallel_requests: 10
```

この設定では、直近1分間のリクエスト数やトークン数、同時実行数が上限を超える場合、Gatewayが即時に拒否します。これにより、**バグや想定外の大量呼び出しによるコスト急増を抑えやすく**なります。

### 残予算や状況に応じてモデルを切り替える

Gatewayの価値は、単にリクエストを止めることだけではありません。  
残予算や混雑状況に応じて、別のモデルや別のプロバイダーへルーティングすることもできます。  
たとえば、アプリケーション側では以下のような論理モデル名だけを指定します。

Gateway側で、実際にどのモデルへ流すかを決めます。

これにより、以下のような制御が可能になります。

* 通常時は高性能モデルを使う
* 残予算が少ない場合は安価なモデルに切り替える
* 障害時や混雑時は別プロバイダーへフェイルオーバーする
* PoC環境では軽量モデルに固定する
* 本番環境では承認済みモデルだけに限定する

このように、Gatewayを挟むことで、アプリケーション側からLLM利用ポリシーを切り離せます。

### 注意点：最終コストを完全に事前確定できるわけではない

ただし、Gatewayを使えばすべてのコストを完全に事前確定できる、というわけではありません。

LLMの最終的な利用料は、主に以下で決まります。

* 入力トークン数
* 出力トークン数
* 使用モデル
* キャッシュ有無
* プロバイダーごとの料金体系

入力トークン数は事前に見積もりやすいですが、出力トークン数は実際にモデルが応答を生成するまで確定しません。たとえば、残予算がほとんどない状態で、最大出力トークン数が非常に大きいリクエストを許可すると、最終的に予算を超える可能性があります。そのため、実務では以下のような設計が現実的です。

* 入力トークンが大きすぎるリクエストは拒否する
* `max_tokens` を必ず設定する
* 残予算が少ない場合は高額モデルを使わせない
* 残予算が少ない場合は軽量モデルに切り替える
* 予算上限の90%到達時点で制限を強める
* チーム単位・APIキー単位でハードリミットを設ける

つまり、Gatewayによるリアルタイム制御は、最終コストを1円単位で完全に保証する仕組みというより、**ポリシー制御によってコスト急増を防ぐ仕組み**として捉えるのが現実的です。

### どのようなケースでGateway構成を検討すべきか

すべてのケースでMulti-Provider Generative AI Gatewayが必要になるわけではありません。小規模なPoCや単一アプリであれば、アプリケーション側で以下のような制御を実装するだけでも十分な場合があります。

* ユーザーごとの月次上限
* モデルの固定
* `max_tokens` の強制
* 1日あたりの利用回数制限
* 入力文字数・トークン数制限

一方で、以下のようなケースではGateway構成の価値が大きくなります。

* 複数の部署でLLMを使う
* 複数の業務アプリからLLMを呼ぶ
* Amazon Bedrock以外のLLMも使う
* チームごとに予算を分けたい
* 高額モデルの利用を統制したい
* 監査ログを一元化したい
* 将来的にモデルやプロバイダーを切り替えたい
* リトライやフォールバックを共通化したい

生成AIの活用が一部のPoCから全社利用へ広がるほど、「どのLLMを使うか」だけでなく、**誰が、どのモデルを、どれだけ、いくらまで使えるか**を制御する仕組みが重要になります。

その意味で、**Multi-Provider Generative AI Gateway on AWSは、企業でLLM利用をスケールさせる際の有力な選択肢になる**と考えています。

## 4. まとめ

生成AIの業務利用では、トークン利用料や高額モデルの使いすぎが課題になりやすいことを背景に、本記事ではリアルタイムにLLM利用のコストを制御する仕組みを紹介しました。

LLM利用のコストをリアルタイムに制御したいなら、請求データを見るのではなく、LLMの入口に改札を作る必要があります。その改札の役割を担うのが、Multi-Provider Generative AI GatewayのようなLLM Gateway構成となります。

生成AIの社内利用が広がるほど、今後は「どのモデルを使うか」だけでなく、誰が、どのモデルを、どれだけ、いくらまで使えるかを制御する仕組みが重要になっていくと考えます。
