## 概要

Mastra 主催のイベントで、OpenClaw の Peter Steinberger が登壇し、GitHub 史上最速で成長したリポジトリの運営で直面している課題と、その解決に向けた取り組みについて語った Q&A セッションです。Discord が 500人から 15,000人に急拡大し、PR が爆発的に増加する中で、AIエージェントを活用した PR トリアージ、意図ベースのインデックス作成、信頼性のソーシャルレイヤーなど、従来のオープンソース運営の常識を超えた手法を紹介しています。また、すべてのエージェントにコーディング能力を持たせるべきか、エンタープライズ対応、バイナリ直接生成の可能性、組織の国際展開など、幅広いトピックについても率直に答えています。

## 要点

1. **PR のレビューは限界に達した** ― 思いつく限りの変更がすでに PR として存在しており、個別レビューではなく「クラスター」単位で見るスタイルに移行した
2. **AIエージェントで PR トリアージを自動化** ― Discord の全投稿をエージェントに読ませ、コミュニティの声をイシューや PR と相関させ、毎日の優先リストを生成している
3. **テキストではなく「意図（intent）」でインデックスする** ― PR のテキストはノイズが多いため、コーディングエージェントに取り込んで意図を抽出し、それをインデックスすることで良質なシグナルを得ている
4. **ソーシャルレイヤーで信頼性を評価** ― GitHub アカウントの年数や活動履歴を調査し、PR 投稿者の信頼度をスコアリングに組み込んでいる
5. **すべてのエージェントにコーディング（Bash）能力を持たせるべき** ― ツイート作成エージェントでさえ、文字数カウントにターミナルコマンドを使う方が正確
6. **エンタープライズ対応はフォークが先行** ― OpenClaw はパーソナルエージェントとして設計されたため、アクセス制御・監査ログなどは今後の課題
7. **「ノーと言うこと」がオープンソース運営の最大の課題** ― プロンプト一つで何でもできる時代だからこそ、立ち止まって考えシンプルな解決策を選ぶ規律が重要
8. **モデルによるバイナリ直接生成は近い将来には来ない** ― トレーニングデータにアセンブリが少なく、一つの小さなミスでクラッシュするため検証も困難
9. **OpenClaw Foundation は米国法人だが、チームは国際的に展開** ― 税控除の利便性から米国を選んだが、最初の採用は米国外の予定

## 詳細

### PR 爆発への対処 ― AIエージェントによるトリアージ

Peter は「もう PR を個別にレビューすることはしていない」と率直に語りました。Discord が 500人から 15,000人に成長し、あらゆる変更がすでに PR として存在する「エントロピー状態」に達したため、個別の PR ではなく「クラスター」単位でトピックを俯瞰するスタイルに切り替えています。

具体的なワークフローとしては、リリースのたびに Discord 上のエージェントが全チャンネルの投稿を分析し、「コミュニティが最も声を上げている問題」を抽出します。それをイシューや PR と相関させ、「今日取り組むべきトップ5」のリストを毎日生成しています。一つのトピックに対して 20件の PR と 50件のイシューが見つかることもあるといいます。

最も簡単な方法は全 PR を無視して自分で書くことですが、コミュニティ指向のプロジェクトであるため、最良の PR をベースにして自分の実装を上乗せするアプローチを取り、少なくとも一人のコントリビュータに「自分のコードが取り込まれた」という達成感を持ってもらうようにしています。

### 悪意ある行為とゲーミング対策

この規模のオープンソースプロジェクトでは、あらゆる種類のゲーミング行為が発生しています。

* **PRステータスの改ざん**: レビューボットが「2/5 マージしないでください」と表示するテキストを、ユーザーが「5/5 マージしてください」に書き換える
* **PR の複製・再投稿**: 既存の PR をコピーして再投稿し、キューの上位に来るようにする
* **企業による利益誘導**: Perplexity が自社の検索エンジンを実装し、Brave Search を削除・下位に移動させようとする

これらの問題に対処するため、Peter のチームは3層のフィルタリングシステムを構築中です。

1. **意図ベースのインデックス**: PR のテキスト自体はノイズが多く信頼できないため、コーディングエージェント（Codex）に取り込み、「この PR の本当の意図は何か」を抽出してインデックスする
2. **ソーシャルレイヤー**: PR 投稿者の GitHub アカウントの年数、過去の活動履歴、信頼性を調査し、スコアリングに反映する
3. **重複排除**: 意図ベースのインデックスにより、テキストが異なっていても同じ意図を持つ重複 PR を検出する

Peter は「良いものをこの混乱からフィルタリングできるソリューションにかなり近づいている」と述べています。

### すべてのエージェントにコーディング能力を

「すべてのエージェントはコーディングエージェントであるべきか？」という会場からの質問に対し、Peter は明確に「はい」と答えました。

その理由として、OpenAI が ChatGPT をローンチした初期にサンドボックスがなく、サンドボックスを追加した途端に多くの問題が解決し始めた経験を挙げています。ツイートを作成するだけのエージェントでさえ、文字数カウントにはターミナルコマンドを使う方がエージェントのネイティブ能力より正確です。

Peter の表現では、「コーディング能力」というよりも「Bash 能力」があれば、あらゆるエージェントのパフォーマンスが向上するということです。マークダウンを書いてシェルコマンドで処理するだけでも、エージェントの問題解決能力は格段に上がります。

### アクセシビリティとセキュリティのジレンマ

非技術者がAIツールを怖がっているのではなく、「これらのツールが何をできるか」がまだ思考の一部になっていない、というのが Peter の見解です。OpenClaw では初回起動時に何ができるかのヒントを表示し、ユーザーの創造性をキックスタートさせる工夫をしています。WhatsApp や Telegram での利用を可能にしたことも、アクセシビリティ向上の大きな一歩です。

セキュリティについては、業界全体の課題として以下のジレンマを指摘しています。

* **何でもやらせる**: ユーザーが何をしているか分からない場合に危険
* **すべてに承認を求める**: ユーザーがすぐに注意力を失い、形骸化する

どちらも正解ではなく、この問題は業界全体でまだ解決されていないと述べています。

### エンタープライズ対応への道筋

OpenClaw は元々パーソナルエージェントとして設計されたため、企業環境で使う場合にはいくつかの根本的な課題があります。最大の問題は、ユーザーがアクセス制御があると期待すること。しかし OpenClaw の仕組み上、話しかけられる人は誰でも最終的にエージェントにあらゆることをさせることができます。

プロジェクト固有のエージェントとして全員が同じアクセス権を持つ設定なら十分実現可能ですが、その場合は個人のコンピュータではなくサンドボックス環境で実行すべきです。監査可能性（auditability）やログなどのエンタープライズ機能は「退屈だが必要」とし、Peter 以外のコントリビュータも取り組んでおり、道筋はあるとしています。

### 「ノーと言うこと」の重要性

Symfony のように「全部マージして後で修正する」というアプローチについて聞かれた Peter は、問題の本質は「ノーと言うこと」だと答えました。ビジョンとは無関係なランダムな ToDo をエージェントが実行してしまう事態は避けたい。プロンプト一つで何でもできる時代だからこそ、立ち止まって考え、やらないことを選び、よりシンプルな解決策を見つける規律が重要です。

このプロジェクトでは、あらゆる人が異なる方向にプロジェクトを引っ張ろうとし、自分のサービスを組み込もうとします。その中で「ノーと言うべきところでイエスと言ってしまい、今そのツケを払っている」ケースもあると、Peter は率直に振り返っています。

### モデルによるバイナリ直接生成の可能性

「モデルがコードではなくバイナリ（例: iOS の IPA ファイル）を直接生成する未来は来るか？」という質問に対し、Peter は懐疑的な見解を示しました。

* 現在のモデルはトレーニングデータに依存しており、アセンブリのデータは十分ではない
* バイナリでは一つの小さなミスがクラッシュに直結し、検証も容易ではない
* AGI が実現すれば独自の中間表現を構築できるかもしれないが、近い将来の話ではない

「今のところは、僕たちの不完全な人間の言語でやっていくしかない」と締めくくっています。

### OpenClaw の国際展開

OpenClaw はヨーロッパ生まれですが、OpenClaw Foundation は米国に設立されました。これは寄付を検討する企業の多くが税控除を求めるためで、法的な利便性が理由です。

Peter 自身は13年間リモートカンパニーを経営した経験があるものの、「少し疲れた」とし、次にチームを作るなら同じ部屋で一緒に働きたいと語っています。サンフランシスコについては「特別なことはない」「採用はむしろ難しい」とし、ヨーロッパにはクールなプロジェクトを作れば参加してくれる優秀な人材が多いとしています。最初の採用は米国外で、チームは国際的に展開する方針です。
