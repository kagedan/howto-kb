# 作ったもの

GitHubのユーザー名を入れると、公開プロフィールを**パワプロ風の選手能力カード**に変換するサイトです！

![Produce_2.gif](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3802319/457dbe83-336c-4a48-afa4-1118dcdb5117.gif)
**https://gitpawa.toromonja.com**

スター数・フォロワー数・使用言語・活動量といった公開統計を、ミート／パワー／走力／肩力／守備力／捕球の6能力（S〜Gランク）56種類の特殊能力、そして総合ランクに変換します。

たとえば `torvalds`（Linuxカーネル開発ニキ） を査定するとパワーS・走力S・肩力Sで総合Sランク、金特に「世界の王」「鉄人」が付きます。

技術構成は Next.js 16 + Cloudflare Workers、OGP画像の生成だけ Deno Deployで実装してます。

---

# ステータスロジック

パワプロの選手能力画面を模していて、表示される要素はそれぞれGitHubの統計に対応しています。

| カードの項目 | 査定のもとになる統計 |
| --- | --- |
| ミート | 直近30日の活動日数、アカウント歴 |
| パワー | 合計スター、最大スター、リポジトリ数 |
| 走力 | 直近30日のイベント数、push数 |
| 肩力 | フォロワー数、合計フォーク数 |
| 守備力 | フォーク数、PR・Issue活動 |
| 捕球 | 1年以内に更新したリポジトリの割合 |
| 弾道 | 最大スター数 |
| ポジション | メイン言語 |
| 背番号・投打 | ユーザー名のハッシュ |
| 年間成績 | 打率・本塁打・打点・盗塁から変換 |

能力値・特殊能力の各項目にはホバーで査定根拠が出ます。「なぜ走力がGなのか」がその場で分かるようにしてあります！

# 全体構成

```
Gitパワ査定 (Cloudflare Workers)
    ├ サイト本体・査定ロジック
    └ /api/player/:user  … 査定結果JSON
    
Deno Deploy
    └ /card/:user … PNGを生成
```

査定ロジックは本体（Cloudflare Workers）に置き、Deno側は `/api/player` のJSONを取ってきて描くだけに徹しています。


## GitHub APIから公開情報を抽出

```ts
const [user, repos, events] = await Promise.all([
  ghFetch<GhUser>(`/users/${username}`),
  ghFetch<GhRepo[]>(`/users/${username}/repos?per_page=100&sort=pushed`),
  ghFetch<GhEvent[]>(`/users/${username}/events/public?per_page=100`).catch(
    () => [] as GhEvent[],
  ),
]);
return buildPlayerCard(user, repos, events);
```

Fine-grained token は5,000リクエスト/時 可能になります。公開データしか読まないのでトークンに権限は要りません。

## 能力値1〜99に変換

スター数やフォロワー数には上限がありません。この数値をそのまま能力値に比例させると、torvaldsニキ のような突出した人が100になり、それ以外のほぼ全員が0になってしまいます。そこで、まず対数（log10）で桁の差を圧縮して、指数関数で1〜99の範囲に収めています。

```ts
const log10 = (v: number) => Math.log10(v + 1);

const softCap = (raw: number) =>
  Math.max(1, Math.min(99, Math.round(99 * (1 - Math.exp(-Math.max(0, raw) / 62)))));
```

パワーの素点はこうなっています。合計スターが主、最大スターとリポジトリ数が従です。

```ts
raw: log10(totalStars) * 26 + log10(maxStars) * 13 + log10(ownRepos.length) * 7
```

100は「伝説級の条件」を満たしたときだけ別枠で与えます。

```ts
{ key: "power", legend: totalStars >= 500000 },
{ key: "arm",   legend: user.followers >= 150000 },
{ key: "defense", legend: totalForks >= 80000 },
```

ミート・走力・捕球は活動量ベースで、100に近づくことは理論上難しいので特にロジックは追加していません。

## 特殊能力は56種類

金特6・青特31・緑特5・赤特14。作ってて気づいたんですが、意外と赤特って量少ないんですねー

```ts
// 金特
if (totalStars >= 50000) add("世界の王", "合計スター5万超えのレジェンド");
else if (maxStars >= 1000) add("アーチスト", `スター${maxStars}の特大アーチ級リポジトリ保有`);
if (accountYears >= 10 && daysSinceLastActivity <= 14)
  add("鉄人", `${Math.floor(accountYears)}年選手でいまだ無傷の現役`);

// 赤特
if (ownRepos.length <= 1) add("扇風機", "公開リポジトリがほぼない");
else if (ownRepos.length < 3) add("三振", "公開リポジトリが3未満");
if (totalStars >= 100 && starConcentration >= 0.9)
  add("一発", `スターの${Math.round(starConcentration * 100)}%が1リポジトリ頼み`);
```

野球の語彙に寄せた条件がいくつかあります。低レイヤ言語（C/C++/Rust/Zig）が3割を超えると「ローボールヒッター」、フロント言語が6割超で「ハイボールヒッター」、Haskell や OCaml のようなニッチ言語があると「アウトコースヒッター」。この辺の条件はAIと壁打ちしながら決めました。

## 総合評価

6能力の平均に、金特・青特を加点、赤特を減点して合成します。青特と赤特は個数を増やせるので、加点・減点は頭打ちにしています。

```ts
const blueBonus = Math.min(12, specials.filter((s) => s.type === "blue").length * 2);
const redPenalty = Math.min(12, specials.filter((s) => s.type === "red").length * 3);

const overallRaw =
  abilities.reduce((s, a) => s + a.value, 0) / abilities.length +
  specials.filter((s) => s.type === "gold").length * 6 +
  blueBonus - redPenalty;
```

総合の100も特別枠で、スター・フォロワー・フォークの3つとも振り切った者だけが到達します。

## 年間成績とハッシュ

打率は「月間活動率7割＋リポジトリ保守率3割」を .120〜.380 に変換します。保守率を混ぜないと動いている人が全員 .380 のイチローだらけになります。

背番号と投打はユーザー名のハッシュで決めます。ここはもう少しいいロジックがあったかもしれないですねー。
気が向いたら改修します。

```ts
const hash = [...user.login].reduce(
  (h, c) => (Math.imul(h, 31) + c.charCodeAt(0)) >>> 0,
  7,
);
```

# OGP画像の生成

## Cloudflare Workersでは現実的に難しい

`next/og` の `ImageResponse` は内部で satori（JSX→SVG）と resvg（SVG→PNG）を動かします。1200×630のPNGをラスタライズするのに数百ミリ秒〜数秒。10msに対して桁が2つ足りず、`error 1102`（CPU time limit exceeded）になります。

`og:image` は別ドメインを指してよいので、**描画だけを Deno Deploy に逃がす**構成にしました。

| | 無料枠 |
| --- | --- |
| Cloudflare Workers | CPU **10ms / リクエスト** |
| Deno Deploy | CPU **合計15時間 / 月**、100万リクエスト、下り20GB |

リクエストごとの上限ではなく月間の総量制なので、1枚2秒でも月27,000枚描けます。

```tsx
import satori from "npm:satori@0.10.13";
import { initWasm, Resvg } from "npm:@resvg/resvg-wasm@2.6.2";

const MAT_W = Math.round((CARD_H * OG_W) / OG_H);

const wasmReady = fetch(RESVG_WASM_URL)
  .then((r) => r.arrayBuffer())
  .then((buf) => initWasm(buf));

export async function renderCard(data: PlayerCardData, font: ArrayBuffer) {
  await wasmReady;

  const svg = await satori(
    <div style={{ display: "flex", width: MAT_W, height: CARD_H, background: "#cdedf7",
                  justifyContent: "center", alignItems: "center" }}>
      <div style={{ display: "flex", flexShrink: 0, width: CARD_W, height: CARD_H }}>
        <Card data={data} />
      </div>
    </div>,
    { width: MAT_W, height: CARD_H,
      fonts: [{ name: "MPLUSRounded", data: font, weight: 800 }] },
  );

  return new Resvg(svg, { fitTo: { mode: "width", value: OG_W } }).render().asPng();
}
```

出力は **800×420** です。X がタイムラインで表示する実寸は約500×260なので、800幅あれば許容範囲です。


## Workers側のJSONもキャッシュする

Deno側が毎回叩く `/api/player` は GitHub API を3回呼ぶので2.1秒かかります。その分だけ画像生成が遅れるので、Cloudflare側でも明示的にキャッシュします。

重要なのは、**Workers のレスポンスは `Cache-Control` ヘッダを付けただけではエッジにキャッシュされない**ことです。`caches` に自分で入れる必要があります。

```ts
const cache = await caches.open("gitpawa-player-v1");
const hit = await cache.match(key);
if (hit) return hit;

const res = NextResponse.json(await getPlayer(username));
res.headers.set("cache-control", CACHE_CONTROL);
getCloudflareContext().ctx.waitUntil(cache.put(key, res.clone()));
return res;
```

これで1.3秒 → 0.35秒になりました！
