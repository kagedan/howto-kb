## はじめに

社内セキュリティ教育楽しめてますか？

年1回の動画研修、月1回のメルマガ。いつのまにか配信が止まり、年1回教育をするだけ、、

私がやりたかったのは、シンプルなことでした。

> **「日々セキュリティに触れる機会を、しかも簡単に持ちたい」**

そのためには、**形骸化させない**ことと **参加人数を減らさない**ことが要る。配信側が消耗せず、社員側が「ちょっと楽しみ」と思える仕組みです。

弊社（WiseVine）では、Claude のスキル機能 + Slack Workflow Builder を組み合わせて、**コードを書かずに社内セキュリティクイズを作成・継続運用する仕組み**を作りました。

設計と運用ノウハウを、まるごと共有します。

## 1. 全体構成

3つのコンポーネントで動いています。

| コンポーネント | 役割 | 場所 |
| --- | --- | --- |
| **問題作成スキル** | Claude 上で対話的に問題を作成、Notion DB に登録 | Claude（スキル機能） |
| **出題 Workflow** | フォームから問題を入力してチャネルに投稿 | Slack Workflow Builder |
| **回答収集 Workflow** | リアクションを集計用に記録 | Slack Workflow Builder |

Claude のスキル定義（マークダウン）と Slack Workflow Builder の GUI のみで完結。**コードは1行も書きません**。コーポレートIT・人事の担当者が、自分でメンテできる粒度に収めることを意識しました。

## 2. 問題作成スキル：壁打ちで問題作成

Claude には **「スキル」** という機能があります。マークダウンで設計書を書くと、その設計通りに Claude が振る舞ってくれる仕組みです。

セキュリティクイズ作成スキルは、こんなフローで対話します。

### 対話のイメージ

```
User: 「最近フィッシングメールが流行ってるから何か作って」

Bot:  テーマ案、こんなのどう？
      1. 業務メール風フィッシングの見分け方
      2. パスワード再入力を促すフィッシング
      3. 経営層を装った緊急依頼（BEC）

User: 1で

Bot:  [問題文 + 4択 + 解説 のドラフト]

User: 選択肢2をもっとそれっぽくして

Bot:  [改善版]

User: OK

Bot:  登録した → URL返却
```

ユーザーは「テーマ選択 → 微調整 → OK」だけ。**1問あたり5〜10分**で完成します。

## 3. 楽しませる工夫：各種理論を組み込む

普通に問題を出題しても、つまらない・飽きる・参加されない。これを避けるため、スキル定義に **行動経済学・心理学の理論** を必須ルールとして組み込んでいます。

ここが他のクイズ自動生成系の取り組みと一番違うところです。

### ① 損失フレーミング

問題文や解説に「気づかなければ〜になる」という**損失視点**を必ず1つ入れる。

* ❌「強いパスワードを使えば安全です」
* ✅「このパスワードでは、攻撃者に30秒で突破され、アカウントを乗っ取られる」

人間の脳は「得をする話」より「損をする話」に強く反応します（プロスペクト理論）。**「あ、これ自分のことかも」と止まる**理由が、この一言で生まれます。

### ② 社会的証明

解説文に **「実際の被害データ」や「多くの人が陥るパターン」** を一言入れる。

* 例：「この手口は、国内フィッシング被害の約7割を占めるとされる」
* 例：「IPA調査では、従業員の約6割が業務外でも同じパスワードを使い回しているとされる」

「みんなやってる」「みんな陥る」を見せることで、**「自分も対象だ」と肚落ちする**。

⚠️ 数字が不明な場合は「多くの被害事例で確認されている」など定性表現でOK。**数字を捏造しない**ことだけは守る、というルールも入れています。

### ③ ひっかけ設計（認知バイアスの埋め込み）

不正解の選択肢は「**それっぽいけど間違い**」にする。スキル定義では、以下4つのバイアスを **1問に1〜2個意識的に埋め込む** ルールにしています。

| バイアス | 選択肢の例 |
| --- | --- |
| **権威バイアス** | 「上司・CEOから頼まれたから大丈夫」 |
| **確証バイアス** | 「いつもこうしているから安全なはず」 |
| **正常性バイアス** | 「うちの会社には関係ない」「自分は狙われない」 |
| **緊急性バイアス** | 「急いでいるから確認を省略しても大丈夫」 |

そして解説でバイアス名に軽く触れる。「これは権威バイアスという罠」と名指しで言うと、**間違えた人が「自分が引っかかった理由」を理解できる**。これが学習として一番効きます。

### ④ ホスピタリティ（参加が温かい体験になる設計）

問題文・解説に、参加者へのねぎらいや応援の一言を必ず入れる。

* 問題文の最後：「答えてくれるだけで、チームのセキュリティが1段階上がる」
* 解説の冒頭：「間違えた人も安心して。今気づけたのが大事」
* 解説の最後：「今日から使える：〜するだけでOK」

スキル定義に **「Bot はジャッジしない、応援するキャラクター」** という基本スタンスを明文化して、Claude がそこから外れないようにしています。これがあると、不正解だった社員も「次もまた答えよう」と思える。

## 4. キャラ付き Bot で参加率を上げる

問題文を機械的なフォーマットで送ると、誰も読みません。**Bot に明確な人格を付ける**だけで、読まれ方が変わります。

スキル定義の冒頭に、Bot の **キャラクター設定**を細かく書いておきます。

* 名前・年代・職業（架空のプロフィール）
* 性格（熱血漢／落ち着いた語り口／ユーモア重視 など、コンセプトを決める）
* 話し方の語尾と決めゼリフ
* 「どんな反応も応援する」という基本スタンス

スキル起動中、Claude は常にそのキャラとして振る舞います。問題文の冒頭・解説の語り口・登録報告のセリフまで、すべて統一感のあるキャラとして出てくる。

これだけで、配信メッセージが「IT部からの通達」ではなく **「毎週おなじみのキャラからのお便り」** に変わります。

## 5. Slack Workflow Builder で出題と回収

問題が Notion に溜まったら、Slack Workflow Builder で配信します。

### 出題ワークフロー

* フォームに **問題文 / 選択肢1〜4 / 答え / ID** を入力
* 指定チャネルに整形してポスト

### 回答収集ワークフロー

* チャネル投稿に **1️⃣ 2️⃣ 3️⃣ 4️⃣ のリアクション** が付くとトリガー
* 回答ログ（Googleスプレッドシート）を追記（あとで集計できるように保存）

回答が **「リアクションだけでOK」** なのが重要です。コメントを書かせると参加ハードルが一気に上がる。リアクションなら3秒で完了します。

このあたりは Slack Workflow Builder の GUI で完結するので、コードは1行も書いていません。

## おわりに

社内教育で大事なのは、「**正しい情報を一回送る**」ことではなく、「**参加してくれる体験を継続的に作る**」ことだと、運用して改めて感じました。

**ノーコード × AI × 各種理論のフレーム**を組み合わせると、コードを書かない部署の人でも「楽しんで参加される教育コンテンツ」を継続運用できる。これが本当の意味でのコーポレートIT × AI の効能だと思っています。
