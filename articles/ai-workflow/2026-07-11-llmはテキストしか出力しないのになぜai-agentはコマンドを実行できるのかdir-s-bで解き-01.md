AI Agentを理解し始めたころ、どうしても引っかかる疑問がありました。

> LLMは文章を予測して出力しているだけなのに、なぜAgentになるとファイルを読んだり、コマンドを実行したり、ツールを呼び出したり、実際の環境に影響を与えられるように見えるのか？

表面だけを見ていると、Agentを「自分で手を動かすモデル」だと思い込みがちです。でもこの理解は、モデル・Runtime・ツール実行層・Shell・権限環境・コンテキストコストといった重要な境界線を、全部「実行した」というブラックボックスに押し込めてしまいます。

その後、僕はこう理解するようになりました。

```
テキスト + プロトコル + Runtime/ツール実行層 + 権限環境 = 実行可能な振る舞い
```

LLM自体は今も出力を生成しているだけです。ただ、その出力は人が読む自然言語とは限らず、システムが読み取る構造化されたツール呼び出しの意図でもありえます。その意図を現実の動作に変えているのは、Agent Runtimeがその意図を解析・検証・認可・ルーティングし、さらに下層のツールやOS環境が実行するプロセスです。

この記事では、ごく小さなWindowsコマンドを使って、この一連の流れを分解してみます。

```
dir /s /b /a:-d "C:\Work\demo"
```

見出しやタイトルを簡潔にするため、以降（タイトルを含む）ではこのコマンドを `dir /s /b` と略記します。ただし本文中のすべての例・出力・議論は、`/a:-d`（ディレクトリを除外し、ファイルだけを対象にする）を含むこの完全なコマンドを前提にしています。

ここでは `cmd.exe` の文脈での `dir` を例にします。これはPowerShellのネイティブコマンドではありません。`/s`、`/b`、`/a:-d` といった引数も、`cmd` の `dir` コマンドの文法の中で解釈されるものです。

これは特定の製品のチュートリアルではなく、Agentのコマンド実行の仕組みを僕なりに分解したメモです。目的は「Agent＝魔法」というイメージを、「Agentはプロトコル化され、検証可能で、権限に制約された実行システムである」という理解に修正することです。

## まずこのコマンドが何をしているか

`dir` はWindowsの `cmd.exe` でよく使われるディレクトリ一覧コマンドです。単体で `dir` を実行すると、カレントディレクトリのファイルとフォルダを一覧表示します。

一方このバージョンは：

```
dir /s /b /a:-d "C:\Work\demo"
```

大まかに言うと、`C:\Work\demo` から再帰的にすべてのファイルパスを一覧表示する、と理解できます。

それぞれの引数は次の通りです。

| 引数 | 意味 |
| --- | --- |
| `/s` | サブディレクトリまで再帰的に入り、その中身も一覧に含める |
| `/b` | bare formatを使い、見出しや集計などの余計なテキストを出さず、パスだけを簡潔に出力する |
| `/a:-d` | 属性でフィルタし、directoryを除外する。つまりファイルだけを対象にし、フォルダは対象にしない |
| `"C:\Work\demo"` | スキャン対象のディレクトリ |

なので、こんな結果が返ってくる可能性があります。

```
C:\Work\demo\README.md
C:\Work\demo\src\main.ts
C:\Work\demo\src\utils\path.ts
C:\Work\demo\docs\intro.md
```

ここで重要なのは、LLM自身がディスクをスキャンしているわけではない、という点です。

LLMが突然 `C:\Work\demo` を読み取る能力を持つわけではありません。LLMがしているのは、せいぜいある時点で「ユーザーの質問に答えるには、このディレクトリのファイル一覧が必要だ。だからファイルスキャンのコマンド実行をリクエストすべきだ」と判断することだけです。

この判断はツール呼び出しの意図として表現されます。その意図が実行できるかどうか、誰が実行するのか、どの権限の下で実行するのか、結果がどう返ってくるのかは、すべてモデルの外側にあるシステムの責務です。

このコマンドには、工学的なリスクも自然に伴います。

* パスが存在しない可能性
* 現在のプロセスに読み取り権限がない可能性
* ディレクトリが非常に大きく、大量のパスを出力する可能性
* コマンドの実行に長い時間がかかる可能性
* 出力を切り詰めたり、ページングしたり、エラーとして包む必要が出てくる可能性

こうしたリスクが、まさにAgent Runtimeの中核的な責務へとつながっていきます。

僕が最初に混同しやすかったのは、モデルが「ツールを呼び出した」のを見て、モデルが本当にツールを実行したと思い込むことでした。

でもTool Callingは、モデルが構造化されたリクエストを出力しただけ、と捉えるほうが近いです。プラットフォームによって実際のプロトコルはメッセージオブジェクトだったり、構造化レスポンスだったり、内部フォーマットだったりして、必ずしもJSONテキストとしてそのまま露出するとは限りません。以下の例はあくまで概念的な表現です。

```
{
  "tool": "shell_command",
  "arguments": {
    "command": "dir /s /b /a:-d \"C:\\Work\\demo\"",
    "timeout_ms": 10000
  }
}
```

この構造自体は、何のプロセスも起動しません。これはRuntimeが読み取れるリクエストにすぎません。

* どのツールを使いたいか
* どんな引数を渡したいか
* 何のコマンドを実行してほしいか
* タイムアウトや作業ディレクトリなどの制約を宣言しているかもしれない

つまりTool Callingは「実行の意図」であって、「実行という動作」そのものではありません。

この境界線は次の図で分けられます。

この図は僕にとって役に立ちました。モデル側が出力するのはテキスト化・プロトコル化された意図で、ホストとツール側がその意図を実際の動作に変える責任を負っている、ということです。

だから僕が「Agentがコマンドを実行した」と言うとき、それは実は省略された言い方です。正確に言うなら：

> LLMがツール呼び出しの意図を生成し、Agent Runtimeがその意図を受理してルーティングし、Tool Executorが `cmd.exe` を呼び出し、そして `cmd.exe` がコマンドを実行し、結果が再びAgentに包まれて返ってくる。

少し長くなりましたが、境界がずっとはっきりします。

## Agent Runtimeが間で何をしているか

Runtimeはこの一連の流れの中で鍵となる中間層です。成熟した、あるいは本番環境レベルの仕組みは、単に「コマンドを見たら実行する」わけではなく、1回のツール呼び出しをめぐって一連の制御動作を行います。

僕はこれを1回の呼び出しのライフサイクルとして理解しています。

1. 解析：モデルが出力したツール名と引数を読み取る
2. 検証：引数の構造がツールのschemaに沿っているか確認する
3. 認可：この呼び出しが許可されているか判断する
4. ルーティング：対応するツール実行層を見つける
5. 監視：タイムアウト、終了コード、例外、出力サイズを処理する
6. 包装：stdout、stderr、エラー情報などを、モデルに返せる結果として整理する

ユーザーのリクエストから最終的な回答までをつなげると、だいたいこうなります。

ここで特に強調したいのが「認可」のステップです。

権限はLLM自身が持っている能力ではありません。LLMは `dir` コマンドを書けるからといって、任意のディレクトリを読み取る権限を自動的に得るわけではありません。実際に効いているのは、Runtimeとホスト環境がツール実行層に与えている権限の境界です。

例えば、あるAgent環境ではカレントの作業領域の読み取りは許可されているけれど、システムディレクトリへのアクセスは許可されていないかもしれません。読み取り専用コマンドの実行は許可されているけれど、ファイルの書き込みやネットワークアクセス、削除操作にはユーザーの承認が必要かもしれません。コマンドにタイムアウトや出力サイズの上限が設定されているかもしれません。

だからコマンドが失敗したとき、僕は「モデルがコマンドを正しく書けなかったのでは」とだけ疑うべきではありません。むしろこの流れに沿って原因を探るべきです。

* tool\_callの引数構造は正しいか？
* Runtimeはこのツールの呼び出しを許可しているか？
* カレントの作業ディレクトリとパスは正しいか？
* プロセスの権限で対象ディレクトリを読み取れるか？
* Shellはこのコマンドを解釈できるか？
* 出力が大きすぎて切り詰められていないか？

こうして各レイヤーを分解してみると、Agentの振る舞いはそれほど神秘的ではなくなります。

## 本当にコマンドを実行しているのはExecutorとcmd.exe

続いて、Tool Executorと `cmd.exe` を切り分けます。

Tool Executorは、Runtimeの下にあるツールのアダプタ層と理解できます。Runtimeの呼び出しリクエストを、何らかの具体的な動作に変換する役目を持ちます。shell系のツールの場合、この動作は通常、Shellプロセスを1つ起動し、コマンド文字列をそのプロセスに渡すことです。

`cmd.exe` は、さらに下層にあるコマンド解釈とプロセス実行の環境です。`dir /s /b /a:-d "C:\Work\demo"` の中の `dir`、`/s`、`/b`、`/a:-d` は、`cmd` のコマンド文法の中で解釈されるものであって、LLMやRuntimeがそれぞれの引数の意味を自前で実装しているわけではありません。

この順序をシーケンス図で整理してみます。

この図から気づいたのは、Agentの「知能」は「実行能力」と同じではない、ということです。

LLMの価値は、目的を理解し、手順に分解し、ツールを選び、引数を生成し、結果を読み、回答を組み立てることにあります。一方で、実際にファイルシステム・プロセス・ネットワーク・データベースとやり取りしているのは、ツール層と実行環境です。

これは、同じモデルでも異なるRuntimeの上では能力が変わって見える理由でもあります。shellツールを持たない環境では、コマンドを実行できません。読み取り専用の権限しか与えられていない環境では、ファイルを書き込めません。特定のディレクトリへのアクセスが許可されていない環境では、正しいコマンドを生成できても失敗します。

僕にとってこれは、Agent製品の違いを理解する鍵です。多くの場合、僕らが目にしているのは「モデルが急に賢くなった」のではなく、「Runtimeが異なるツール・権限・サンドボックス・結果の返し方を提供している」だけなのです。

## 最小構成のAgent Runtime疑似コード

ここまでの概念にコードとしての実感を持たせるため、非常に簡略化した疑似コードを書いてみます。特定のプラットフォームに対応するものではなく、あくまで大まかな境界を表現するものです。

```
type ToolCall = {
  tool: string;
  arguments: Record<string, unknown>;
};

async function handleToolCall(call: ToolCall) {
  const tool = registry.find(call.tool);

  if (!tool) {
    return { ok: false, error: "unknown_tool" };
  }

  const validation = tool.schema.safeParse(call.arguments);
  if (!validation.success) {
    return { ok: false, error: "invalid_arguments" };
  }

  const allowed = await policy.canRun(tool.name, validation.data);
  if (!allowed) {
    return { ok: false, error: "permission_denied" };
  }

  try {
    const rawResult = await withTimeout(
      () => tool.execute(validation.data),
      tool.timeoutMs
    );

    return {
      ok: true,
      result: packResult(rawResult, {
        maxOutputChars: 20000
      })
    };
  } catch (error) {
    return {
      ok: false,
      error: normalizeError(error)
    };
  }
}
```

実際のシステムはもっと複雑で、監査ログ、ユーザー確認、サンドボックス、リソース制限、並行制御、機密引数のフィルタリングなども備えているはずです。それでも、僕にとって一番重要な境界は変わりません。

* LLMが呼び出しの意図を生成する
* Runtimeが受理するかどうかを決める
* Tool Executorが具体的な実行を担う
* Runtimeが結果をコンテキストに整理して戻す
* LLMがその結果をもとに回答を生成する

この一連の流れがはっきりするほど、Agentの能力をモデル本体の能力だと誤解しにくくなります。

## トークンコストはどこで発生するか

ここでついでに、よくある誤解を1つ整理しておきます。OS上でコマンドを実行すること自体は、LLMのトークンで駆動されているわけではありません。

例えば `cmd.exe` がこれを実行するとき：

```
dir /s /b /a:-d "C:\Work\demo"
```

このプロセスはOS上で動き、ディレクトリをスキャンし、ファイル名を読み取り、パスを出力します。この部分は主に計算リソース、ファイルシステムへのアクセス、プロセスの実行時間の問題です。

一方でトークン課金とコンテキストコストは、具体的なプラットフォームがツール呼び出しと結果をどう記録し、どう返し、どう注入するかに左右されます。一般的に僕が気にしているのは、最終的に何がモデルのコンテキストに入ったか、という点です。

対象ディレクトリにファイルが10個しかなければ、stdoutは短く、Runtimeがその結果をLLMに戻してもコストはわずかです。でももしそのディレクトリに5万個のファイルがあり、コマンドが大量のパスを出力するなら、本当にコストがかかるのは `cmd.exe` が「たくさん行を出力したこと」ではなく、それらの行が会話のコンテキストに詰め込まれることで、大量のトークンを消費することです。

僕は今、これを3つに分けて考えています。

| 項目 | トークンを主に消費するか | 説明 |
| --- | --- | --- |
| LLMがtool\_callを生成 | する | モデルがコンテキストを読み、構造化された呼び出しを出力する |
| Shellが `dir` を実行 | 通常しない | OS上での実行自体はLLMのトークンで駆動されていない |
| ツール結果をLLMに戻す | する | stdout、stderr、エラー情報がコンテキストに入ると消費される |

なのでこの記事では、1つの核心的な判断だけ覚えておけば十分です。

> コマンドの実行そのものは、通常トークンコストの主要な発生源ではない。ツール結果がコンテキストに書き戻されることこそが、モデルが読む必要のあるトークンになる。

大きな出力をどう制御するか、どう要約するか、どうページングするか、どうコンテキスト予算を組むか——これはまた別の、もっとまとまったテーマなので、ここでは深掘りしません。

## 責務の境界：結局誰が何をしているのか

ここまで分解してきたところで、Agentのコマンド実行の流れを1つの責務表にまとめてみます。

| レイヤー | 主な責務 | 誤解してはいけないこと |
| --- | --- | --- |
| LLM | 要求を理解し、タスクを分解し、ツールを選び、引数を生成し、結果を解釈する | ファイルシステムやShellの権限を直接持っていること |
| Agent Runtime | tool\_callの解析、引数の検証、認可、ルーティング、監視、結果の包装 | モデルの出力を無条件にすべて実行すること |
| Tool Executor | 具体的なツールに適応し、呼び出しを実行可能な動作に変換する | 上位のタスク目標を自分で決めること |
| `cmd.exe` / Shell | `dir` などのコマンドを解釈し、実際の実行を起動する | ユーザーの意図やAgentの計画を理解していること |
| 権限環境 | プロセスがアクセスできるパス・リソース・能力を決める | プロンプトによってその場で作り出されるもの |
| コンテキスト | ユーザーのリクエスト、ツールの説明、ツール結果、モデルの回答を保持する | 無限容量の実行ログ倉庫であること |

この表は、問題を切り分けるときにも役立ちます。

コマンドが実行されなかった場合、Runtimeがtool\_callを受理しなかったか、そのツールが存在しない可能性があります。

権限不足のエラーが出た場合は、プロンプトだけをいじるのではなく、ホスト環境・作業領域・サンドボックス・ユーザー認可を確認すべきです。

コマンドは実行されたのに結果がおかしい場合は、Shellの文法・パス・エンコーディング・作業ディレクトリを疑うべきです。

回答が遅くなったりコンテキストが膨らんだりした場合、コマンド自体が遅いのではなく、ツール結果が大きすぎて、LLMに戻される内容が肥大化している可能性があります。

これは、Agentエンジニアリングにおいて初心者がまず身につけるべき考え方だと思います。すべての問題をモデルのせいにしないこと。Agentは1本の連鎖であり、問題は連鎖のさまざまな位置で起こりえます。

## 最初の問いに戻る

ここで、冒頭の問いに立ち返ります。

> LLMはテキストしか出力できないのに、なぜAI Agentはコマンドを実行できるのか？

僕の答えはこうです。

LLMは今も出力を生成しているだけです。ただ、その出力はツール呼び出しのプロトコルに従い、Runtimeが解析できる実行の意図を表現できます。Runtimeは権限環境の中でこの意図を検証しルーティングし、Tool Executorが `cmd.exe` を呼び出し、`cmd.exe` が初めて `dir /s /b /a:-d` のようなコマンドを実際に実行します。実行結果は再びRuntimeによって包装され、モデルのコンテキストに戻り、最後にLLMがユーザー向けの説明を生成します。

だから、この核心の公式をもう一度使い回せます。

```
テキスト + プロトコル + Runtime/ツール実行層 + 権限環境 = 実行可能な振る舞い
```

ここでの「テキスト」はLLMから来ます。「プロトコル」はその出力を解析可能な tool\_call に変えます。「Runtime/ツール実行層」は意図を実際の実行環境にルーティングします。「権限環境」がそれが実際に何をできるかを決めます。

これから先、Agentが「現実の動作を実行した」のを見たときは、まず次の5つを問うようにします。

1. モデルはどんなツール呼び出しの意図を出力したか？
2. Runtimeはこの意図をどう解析し検証したか？
3. 権限環境はこの実行を許可しているか？
4. 実際の実行者は誰か？
5. ツール結果はどうコンテキストに戻されたか？

こう見ると、Agentはもう魔法ではなく、プロトコル化され、検証可能で、権限に制約された1つの実行システムです。

初心者にとって、この境界を一度自分の中に作れば、この先Tool Calling、MCP、Context Engineering、Agent Runtimeを学んでいくときも、ずっと見通しがよくなるはずです。モデルは思考と表現を担い、ツールは実行と力仕事を担い、Runtimeはその両者を安全につなぐ役目を担っています。
