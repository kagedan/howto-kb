最近はもっぱら「自分専用のAIエージェント」をいかに快適に、かつコストを抑えて動かすかに心血を注いでいます。世の中には ChatGPT や Claude といった優秀なツールが溢れていますが、やはり技術者としては「自分の手元で、自分の命令に忠実に、勝手に裏側で動いてくれる環境」を構築したいものです。

今回は、最近私が構築した **「WSL2 + Docker + OpenClaw」** によるローカルAIエージェント実行基盤の話をしようと思います。情報の収集・整理・構造化を自律的にこなす「リサーチ・ハブ」を、**Google AI Studio の無料枠を活用して、1円もかけずに運用する**ための泥臭いテクニックを共有します。

※本システムは Google AI Studio の無料枠を利用するため、規約上、入力データがモデルの改善に利用される可能性があります。そのため、機密情報を扱うビジネス用途ではなく、あくまで「個人のリサーチ、学習、趣味のプロジェクト」を加速させるための **パーソナルな情報整理基盤** としての活用を前提としています。

## 💰 自律型エージェントと「API破産」の現実的なリスク

私がここまで「無料枠」と「ローカルLLM」の組み合わせに執着する最大の理由は、自律型エージェントが持つ **「API破産」のリスクを物理的に遮断するため** です。

2026年現在、LLMの性能向上と共に、APIの消費トークン量も増大しています。特に、自分で考え、検索し、何度もLLMに問いかけながらタスクを完遂する「自律型エージェント（Autonomous Agent）」を走らせる場合、そのトークン消費は予測不可能です。  
もし、エージェントが情報の海で無限ループに陥ったり、予期せぬ挙動で巨大なドキュメントを何度も読み込み続けたら？  
従量課金のAPIを使っていれば、翌朝には数万円、あるいはそれ以上の請求が来ていてもおかしくありません。

個人開発者にとって、好奇心が家計を破壊するリスクになる環境は不健全です。だからこそ、私は **「Google AI Studio の無料枠」** をメインに据えました。  
無料枠なら、どれだけエージェントが暴走しても、止まるのはAPIの制限だけで、財布にダメージはありません。この「失敗しても金銭的損失がない」という心理的な余裕こそが、個人がエージェントを自由に使い倒し、実験を繰り返すための絶対条件でした。

## 😵 ローカルLLMの壁：知能と速度のトレードオフに泣く

当初、私は「すべてをローカルで完結させる」というロマンを追い求めていました。**Ollama** を使い、最新モデルを何個か試した時の記録です。

### 1. Gemma 4: E4B (2026年4月リリース) の限界

Googleからリリースされたばかりの **Gemma 4: E4B** は、レスポンスが驚くほど速いです。しかし、エージェントとして「ツールを組み合わせて複雑なリサーチをしろ」と命じると、途端に推論の精度が不安定になります。  
単純な命令への回答速度は魅力的なのですが、ステップ数の多いタスクでは指示の意図を汲み取りきれず、途中で処理が止まってしまうことが多々ありました。自律的な運用を任せるには、推論の堅牢性がまだ不十分という印象です。

### 2. Gemma 4: 26Bと qwen3.5:9b の「リソースの壁」

一方で、**Gemma 4: 26B** や **qwen3.5:9b** クラスになると、知能自体は非常に高く、指示への追従性も申し分ありません。  
しかし、私の **RTX 5070 (VRAM 12GB)** では、これらの大型モデルを動かすのは非常に酷です。特に、リサーチのために数千文字のウェブページ内容をコンテキストに流し込んだ瞬間、VRAM不足により推論速度が落ち込みます。  
一つのタスクを完了するのに数十分と待たされることになり、自律型エージェントとして連続的に動作させるには、私の環境ではスペック不足でした。

結局、十分なコンテキスト窓を確保しつつ、高い精度で推論を回すには、ローカル環境では限界がありました。

そこで、私は「ロマン」を捨てて「実利」を取ることにしました。**Google AI Studio** の無料枠へのシフトです。

## 🛠️ アーキテクチャ：WSL2 + Docker で構築する安全な砂場

セキュリティと環境のクリーンさを保つため、エージェントの実行環境はすべて **Docker** 内に閉じ込めています。

### なぜ Docker なのか？

自律型エージェント（OpenClaw）は、指示に応じて Python や Ruby のスクリプトを勝手に実行します。「ウェブを検索してこい」と言えばブラウザを制御し、「ファイルを整理しろ」と言えばディレクトリを走査します。

これをホストOSで直接やるのは、いくら自分の管理下にあるコードとは言え少し怖いです。

* エージェントが生成したコードに致命的なバグがあったら？
* 意図しないディレクトリのファイルを削除してしまったら？
* 外部から取得した情報に悪意あるスクリプトが含まれていたら？

WSL2 上の Docker コンテナとして動かすことで、万が一エージェントが暴走しても、影響範囲はコンテナ内（およびマウントしたワークスペース）に限定されます。いわば「安全な砂場」の中で、エージェントを自由に遊ばせているわけです。

## 💰 徹底的に無料で使い倒す：Gemini API のフォールバック戦略

今回のシステムの最大のこだわりが、Google AI Studio の無料枠を「限界まで、かつ安定して」使い倒すための **「インテリジェント・フォールバック」** です。

2026年現在、**Gemini 3.1 Flash Lite** は無料枠ながら **RPDが 500** と非常に強力です。  
しかし、一分間に投げられる回数 (RPM) や、一度に送れるトークン量 (TPM) には制限があります。

### 優先度チェーンの実装

私は `.env` ファイルで以下のような優先度設定を行っています。

```
MODEL_PRIORITY=gemini/gemini-3.1-flash-lite-preview,gemini/gemini-2.5-flash,gemini/gemini-2.5-flash-lite,ollama/gemma4:26b
```

この優先度に基づき、独自のラッパースクリプト (`run_openclaw_agent`) が以下の動作を自動で行います。

1. **第1候補 (3.1 Flash Lite) で試行**: 最も賢く、1日の上限も多いモデルです。
2. **レート制限エラー (429) の検知**: APIから「ちょっと待ちなさい」と言われたら、即座にそれを検知します。
3. **モデルの動的切り替え**: `sed` コマンドを使い、コンテナ内の `openclaw.json` を書き換えて、第2候補 (2.5 Flash) にバトンタッチします。
4. **指数バックオフ**: 全てのクラウドモデルが制限に達した場合、最終手段としてローカルの **Ollama (Gemma 4:26b)** に切り替えるか、あるいは指数関数的に待機時間を増やしながら（60秒、120秒...）リトライを繰り返します。

## 🚀 実践：自律型「技術リサーチ助手」への挑戦

このプロジェクトで私が最初に目指したのは、特定の技術トピックについて、\*\*「放置しておくだけで網羅的なテクニカルレポートを完成させてくれる助手」\*\*を作ることでした。

正直なところ、この機能だけであれば **Google の NotebookLM** などの既存サービスでもかなりのレベルまで代替可能です。しかし、あえて自律型エージェントで自分専用の基盤を構築することには、以下の明確なメリットがありました。

1. **ツールの自由な組み合わせ**: Playwright による最新ウェブスクレイピングや、自前のDB連携など、エージェントに「手足（スキル）」を自由に増やせる。
2. **出力フォーマットの完全制御**: Zenn の Markdown 形式や、自分の Scrapbox 用のフォーマットなど、構造化のルールを自分好みに徹底できる。
3. **多段階の推論パイプライン**: 検索→選別→要約→批判→補強といった、「人間の思考プロセス」を模倣した多段階の連鎖を自分で設計・調整できる。

いわば、\*\*「NotebookLM 的な情報統合体験を、完全に自分好みにチューニングできる砂場」\*\*を作ることが真の目的です。

## 📜 9フェーズからなる自律型リサーチ・パイプライン

エージェントに「適当に情報を集めてまとめて」と一言で頼んでも、深いリサーチ結果は得られません。私はこれを **9つのフェーズ** に分け、連鎖的に実行させることで、1回のリクエストでは到達できない深さを実現しました。

### Phase 1-3: 徹底的な「調査」と「ファクトチェック」

まず、エージェントは Google 検索を多角的に行います。

* **Phase 1 (Search)**: キーワードを変えて20以上のURLをピックアップ。
* **Phase 2 (Select)**: その中から公式サイトや信頼できる技術ドキュメントを厳選。
* **Phase 3 (Extraction)**: Playwright を使い、実際に各ページの中身をスクレイピング。生のテキスト情報を収集し、一つの巨大な「リサーチレポート」にまとめます。

### Phase 4-5: 情報の構造化と詳細化

ここが最大のポイントです。**膨大な情報を一度に処理させてはいけません。**

* **Phase 4 (Structure)**: リサーチ結果に基づき、論理的なレポートの構成案を作成。
* **Phase 5 (Synthesis)**: **「トピックごとに個別に」** 内容を詳細化させます。項目ごとにプロンプトを投げ、前項の要約を渡しつつ整理を進めることで、AIはコンテキストを節約しながら密度の高いレポートを生成できます。

### Phase 7-8: 整合性の調整と自己批判ループ

収集した情報に矛盾がないか、論理が通っているかを検証します。

* **Phase 6 (Coherence)**: 各項目の繋がりを自然にし、用語を統一。
* **Phase 7 (Critique)**: シニアAIアーキテクトとしての役割でレポートを厳しくレビューさせ、「不足している情報の再調査」や「論理の補強」を指示させます。
* **Phase 8 (Final Polish)**: 指摘事項を元に、必要に応じて**再度追加調査を行い**、最終的なリサーチレポートを完成させます。

### Phase 9: 要約とメタデータの生成

最後に、整理された内容の全体要約や、タグ付けを行って終了です。

## ✍️ テクニカルレポートの品質を制御するプロンプト

収集した情報を整理させる際、私はエージェントへの指示（プロンプト）に、かなり細かい「出力ルール」を組み込んでいます。

```
**レポート整理のルール:**
- 結論から述べ、論理的な構成を徹底する。
- 筆者の主観や推測が必要な場合は、事実（Fact）と明確に区別して記述する。
- AI特有の「まず」「次に」といった機械的すぎる接続詞を避け、文脈の繋がりを重視する。
- 箇条書きを多用せず、技術的な背景や理由を論理的な文章で説明する。
- 参照元の情報に忠実であること。情報が不足している場合は勝手に補完せず、追加調査が必要な旨を記述する。
```

## 💡 構築中の苦労話：AIとの「意思疎通」の難しさ

このシステムを完成させるまでには、試行錯誤がありました。

### 1. 「429エラー」への過剰反応

APIのレート制限に触れた際、エージェントが「エラーが出たので処理を終了します」と勝手に諦めてしまうことがありました。  
これを防ぐために、シェルスクリプト側でエージェントの戻り値をチェックし、エラーが出た場合は **「エージェントにエラーを知らせずに、こっそりモデルを切り替えて再試行する」** という隠蔽工作を行っています。

### 2. 日本語の「揺れ」

Gemini は非常に日本語が上手いですが、時々「です・ます」と「だ・である」が混ざることがあります。これは Phase 6 の整合性フェーズで、専用のプロンプトを使って一括補正させることで解決しました。

## 📝 あとがき：結局、何が一番良かったのか

今回は自分専用のリサーチ・ハブを構築してみましたが、一番の収穫は\*\*「最新技術を追いかける際の心理的な摩擦が、想像以上に減った」\*\*ということです。

もちろん、NotebookLM などの完成されたサービスを使えば手軽に同じようなことはできます。しかし、自分のローカルPCと Gemini の無料枠を組み合わせて、「API破産を絶対に起こさないセーフティネット」を自分で設計し、パイプラインを調整していくプロセスは、純粋に楽しい時間でした。

「ローカルLLMだけで完結させたい」というロマンも捨てがたいですが、現実的には Gemini のような強力なAPIを無料枠の範囲内で賢く、かつ安全に「借りる」のが、2026年現在の個人開発者が知的好奇心を満たすための、最もバランスの良い遊び方ではないかと感じています。

セットアップには少し手間がかかりますが、一度自分だけの「実験場」を構築してしまえば、情報の波に飲まれるストレスはかなり軽減されます。もし「APIの請求額が怖くてエージェントに手を出せていない」という方がいたら、ぜひ WSL2 + Docker で自分好みの砂場を作るところから始めてみてください。
