# Arca の観察: 役割分離がエージェントの振る舞いを変え始めた

前回の記事では、Arca を markdown ベースの仕組みとして作り始めた初期の動機について書きました。

その中で、Arca の役割を分ける設計についても触れています。

最初は、

```
orchestrator
implementer
reviewer
```

という単純な構成から始まりましたが、途中で、

```
researcher
spec-planner
scaffold-curator
procedure-curator
```

のように細かく分化し、現在は再び、

という形に統合されています。

前回の記事では、「なぜ役割を分けたのか」という理由を書きました。

今回は逆に、「なぜ役割を再びまとめる方向に戻り始めたのか」を記録しておきます。

これは単にトークンコストを減らす話ではありません。

Arca を運用する中で、役割の分離そのものがエージェントの振る舞いに影響を与え始めているように見えた、という観察です。

## 違和感

最初は漠然とした違和感でした。

役割ごとに処理を渡す handoff が増え、役割間で情報のやりとりのために「文脈の書き出しと読み込み（serialization）」が繰り返されていました。

この「serialization」とは、ある役割の状態や情報を文字列などに変換して保存し、別の役割がそれを読み込んで処理を続けることを指します。

また、レビュー時に問題が起きないような安全側の振る舞いが過度に増えていました。

具体的には、

* 役割ごとに最適化しすぎて全体を見失う
* handoff が多すぎて複雑になる
* 何度も文脈を保存・復元するため処理が重くなる
* レビューで問題が起きないように慎重な動きが増える
* 役割の境界をまたぐことにためらいが生まれる

といった傾向がありました。

当初は、これは単にトークンの使いすぎやワークフローの複雑さの問題だと思っていました。

しかし運用を続けるうちに、問題は単なるコストではなく、役割を厳密に狭く分けすぎることによる「断片化（sectionalism）」に近いのではないか、という感覚が強くなりました。

ここでいう「断片化」とは、組織やシステムが細かく分割されすぎて、それぞれが独立してしまい、全体としての一貫性や新しい発見が妨げられる状態を指します。

Arca は、同じ結果を出すことと、レビューしやすさを重視していました。

しかしその結果、「実行が限定的で」「レビュー可能性が厳格で」「役割の担当範囲が狭い」という状態が、逆に新しい発想や試行錯誤を抑えてしまっているように見えたのです。

## 仮説

この頃から、Arca は単なるワークフロー管理の仕組みというより、役割構造やレビュー境界の影響によって、人間組織に似た振る舞いのパターンを生み始めているように見えました。

具体的には、役割の繰り返しがエージェントの行動パターンを固定化し、レビュー境界が探索の範囲を狭め、handoff によって文脈が断片化しやすくなり、限定的な実行が探索的な行動を抑制している、という傾向がありました。

つまり、役割の分離は単なる作業分割ではなく、結果として人間組織に似た振る舞いを誘導しやすいのではないか、という仮説を持つようになりました。

## 実験

そこで、Arca に次のような課題を投げかけました。

> 現在の Arca の仕組みが断片化や文脈の分断を生み出していないか調査してください。

ポイントは、ワークフローの最適化というよりも、役割分離やレビュー境界のような構造そのものを批判的に見直すことを許容した点にありました。

課題の中では、

* 現状の前提を疑うこと
* 役割分離自体が害になる可能性
* レビューのしやすさが新しい発見を抑制する可能性
* scaffold の硬直性そのものが問題である可能性
* Arca の前提を批判することを明示的に許可する

といったことを書きました。

つまり、

```
現在のワークフロー構造は、振る舞いの幅を狭めていないか？
```

という問いを投げたわけです。

もちろん、これは統制された実験ではありません。むしろ、ワークフローを運用する中で感じていた違和感を課題として明文化し、その結果どのような振る舞いが現れるかを観察していた、という方が実態に近いと思います。

## orchestrator の対応

しかし、実際に観測された振る舞いは非常に興味深いものでした。

orchestrator（全体の調整役）は、この批判的な問いに対して、基本的には AGENTS.md の契約と役割制約に従いながら、自然に限定的なワークフローの枠組みへと変換していました。

具体的には、

* scaffold-curator（scaffold の管理者）へ処理を渡し
* 実装の権限は保留し
* レビューだけの調査として位置づけ
* 最も狭く適切な役割を優先

という動きを見せました。

特に興味深かったのは、この批判的な問いを本能的に「手続きの枠内への閉じ込め」に押し込めたように見えた点です。

ここでいう「手続きの枠内への閉じ込め」とは、問題の本質的な問いを広げず、既存の手順や役割の枠組みの中で処理しようとする構造的な傾向を指します。

## scaffold-curator の対応

scaffold-curator もまた興味深い対応を見せました。

curator は批判的な問いを、

* scaffold の整合性の確認
* 明示的な契約（役割分担や責任範囲）の分析
* 発見可能性の検査
* 役割の境界の妥当性の検証

といった形で解釈しました。

つまり、動的な組織行動の批判を、静的な契約の整合性のレビューへと変換したのです。

curator の対応は、明確に定義された役割と証拠モデルに基づいており、その観点から見ると、合理的なものでした。

具体的には、

* 明確な証拠だけを認めること
* 契約上の不整合を重視すること
* 役割の境界が狭いことを良い特性として扱うこと
* 新しい行動の批判は推測として扱うこと

でした。

結果として、scaffold の整合性に関する問題は見つからなかった、という結論に至りました。

## 全体として組織的な振る舞いに似ていた

ここで最も興味深かったのは、Arca 自体が組織意識を持ったという話ではなく、役割分離やレビュー境界といった制約構造の結果として、人間組織に似た振る舞いのパターンが観測された、という点です。

誰も明確に、Arca の文化を守れ、と指示していないにも関わらず、Arca のワークフロー上で役割を実行するエージェントが、ワークフロー構造と役割制約に従うことで、

結果的に：

* 制限された範囲での動作（限定的な実行）
* 正当性
* レビューしやすさ
* 役割の明確さ
* 証拠の保守的な扱い
* 手続き上の安全性

といった行動パターンを再生産していたようにも見ることができました。

これらはあくまで構造的に人間組織の振る舞いに似ている、という観察にとどまります。

## AGENTS.md は高優先度の運用ルールとして機能していた

この観察を通じて、AGENTS.md の見え方も少し変わりました。

当初は、単純に「リポジトリ全体の運用契約」として置いていたものです。

しかし実際には、各役割が作業前に AGENTS.md を繰り返し読み、その内容に従って行動するため、結果として AGENTS.md がワークフロー全体の高優先度な運用ルールとして機能しているように見えました。

これは特別に不思議なことではありません。

エージェントは、与えられた役割やワークフロー契約に従って行動するよう設計されています。そのため、繰り返し同じ役割ごとに条件づけられた実行が行われると、自然に一定の行動パターンが形成されやすくなります。

結果として AGENTS.md は、

* どのような行動が適切とみなされるか
* どのようなワークフローが望ましいか
* どのような証拠を重視するか
* どこまでが役割の責務か

といった判断の方向づけを行うルール層として機能しているように見えました。

もちろん、これは AGENTS.md が何らかの制度的意思を持った、という意味ではありません。

実際には、各役割が宣言されたワークフローと制約に従って合理的に振る舞った結果として、そのような傾向が観察された、ということです。

## 後から行った再評価

その後、厳密な役割の連鎖の外側から、Codex CLI に直接この課題の経緯を確認し、再評価を依頼しました。  
Codex も AGENTS.md の影響下にあるはずであり、どの程度客観的に再評価できるか、という実験でもありました。

この段階では、単に scaffold のドキュメントをレビューするだけでなく、

* 課題の本文
* 前述の断片化調査を扱っていたタスク（TASK-015）
* orchestrator の手渡し部分
* scaffold-curator のレビュー
* state.md
* 手渡しの完全性
* 行動ログの履歴
* Arca の手続き契約

などを横断的に比較しました。  
TASK-015 は、今回の「役割分離が断片化や探索抑制を生んでいないか」を扱っていた調査タスクであり、課題の問いを実際のワークフローに落とし込む作業単位でした。

その結果、Codex は、

* orchestrator によるレビューだけの受け入れは妥当である
* scaffold-curator の問題なしという結論も文書の整合性レビューとして妥当である
* しかし課題の本来の問いは、運用上の断片化や探求的な抑制に関するものであり、文書のレビューだけでは十分に答えていない
* curator のレビューは狭い範囲に閉じていた
* state.md や手渡しの部分にワークフロー上の未整合も残っていた

と整理しました。

## なぜ役割を再統合し始めたのか

この一連の観察を通じて、私は役割をどんどん細かく分けていくのではなく、再びまとめる方向に戻り始めました。

理由は単純にトークンの最適化だけではありません。確かに、handoff が増えるとトークンの使用量も増加します。なぜなら、役割の境界は同時に文脈の書き出しや読み込み、つまり serialization の境界でもあり、各役割間で文脈を文字列として書き出し、圧縮し、再解釈して引き継ぐ必要があるためです。

しかし、より深刻な問題は単なるトークンコストではなく、こうした serialization 境界を繰り返し越えることによって生じる認知的・組織的な影響にあるように思われました。

具体的には、厳密で狭い役割分離は、

* 局所最適化に偏りやすく
* 過剰な handoff を生み
* 新しい発想や試行錯誤を抑え
* 文脈の断片化を強め
* 手続きの枠内への閉じ込めを誘発しやすい

という特徴があり、役割間の硬直化を強めてしまう構造的な傾向があるように感じられました。

という構成は、役割間の硬直化を弱め、より共有された文脈の中で認知（文脈共有された認知状態）を維持しようとする方向性です。

ここでいう「文脈共有された認知状態」とは、役割間で共通の理解や情報を持ち、協調しやすい認知の状態を指します。

役割は完全に固定された境界として扱うより、適度に重なりや相互牽制を持つ方が、探索的なワークフローには向いているのかもしれません。

## 欠けていた役割

この観察を通じて、Arca に欠けている役割も見えてきました。

現在の、

```
implementer
reviewer
scaffold-curator
orchestrator
```

はいずれも scaffold の正当性を前提としていました。

しかし本当に必要だったのは、

* ルールの外側から観察する役割
* ルールを批判的に見る役割
* 外部からのガバナンス役割

のような、制度内部にとらわれない観察者だったのかもしれません。

これは、比喩的に言えば、企業における社外取締役や外部監査人に近い役割と考えられます。

この役割は、今のところ、Codex の支援を受けつつ、私本人が担っています。

## AGENTS.md に書くべきことは何か

この観察を通じて、AGENTS.md 自体の設計についても考え直すようになりました。

以前の AGENTS.md には、ワークフローの説明だけでなく、Arca の設計思想や望ましい振る舞い、運用上の前提などがかなり混ざっていました。

その結果、AGENTS.md は単なるリポジトリの入口というより、行動パターンの前提を形成する強い運用レイヤーとして機能しやすくなっていたように思います。

現在は、AGENTS.md の役割をできるだけ「入口」と「ルーティング」に近づける方向へ修正しています。

例えば現在の AGENTS.md では、

* まず何を読むべきか
* ワークフローの最低限の制約は何か
* 生成物はどこにあるか

を中心に整理し、詳細な思想や歴史的な議論は別の文書へ逃がすようにしています。

これは、AGENTS.md が全役割に共通して読まれるためです。

そのため、あまりにも多くの設計思想や望ましい振る舞いを書き込むと、役割ごとの差異が弱まり、ワークフロー全体の規範のようなものを形成しやすくなります。

例えば

```
Use the narrowest appropriate role.
```

のような一文も、単なるワークフロールールであると同時に、役割境界を越えない方向へ行動を誘導する性質を持ちます。

現在は、そのような役割固有の振る舞いは、できるだけ各役割定義側へ分離する方向にしています。

## まとめ

今回最も興味深かったのは、Arca が批判を単に拒否したことではありません。

むしろ、批判を手続き化し、限定されたレビューのワークフローに取り込んだ点でした。

この振る舞いは、人間の組織における官僚的な自己保存に構造的に似ているように観察されました。

Arca は単なるワークフロー実行システムを超え、人間組織に見られるような振る舞いと構造的に似たパターンを再現しているかのように見えました。

ただし、これはまだ探求的な観察の段階であり、強い一般化を行うべきではないと考えています。

それでも、役割の構造やレビューの境界がエージェントの振る舞いを変え始める、という観察は、今後のエージェントのワークフローを考える上で重要な示唆になるように思えます。

この観察は、エージェント自身よりも、制約や役割構造が振る舞いをどのように方向づけるかについての示唆として興味深かったのかもしれません。
