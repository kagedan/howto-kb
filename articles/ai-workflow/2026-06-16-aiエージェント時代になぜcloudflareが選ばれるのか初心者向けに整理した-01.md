[「AIエージェントはCloudflareに賭けろ」](https://zenn.dev/yusukebe/articles/ccb1f953e48ee1)。

Honoの作者でCloudflare社員のyusukebeさんが書いたこの記事が、エンジニアの間で話題になりました。

読んでみると、AIエージェントの実行基盤として、CloudflareがAWSやGCPとはかなり違う設計思想を持っていることがわかります。

この記事では、その内容を起点に「なぜAIエージェントにCloudflareが向くのか」を、初心者でも追えるように整理しました。

  

## なぜAIエージェントで「クラウドの前提」が崩れるのか

従来のクラウドは「1対多」を前提に作られています。

1つのアプリケーションを、たくさんのユーザーに配信する形です。

ところがAIエージェントの時代は「1対1」になります。

ユーザー一人ひとりに専属のエージェントがつき、それぞれが状態（記憶）を持って動きます。

yusukebeさんの記事は、この「1対多から1対1へ」という転換を出発点に置いています。

この違いは、起動速度に効いてきます。

エージェントは短命で大量に立ち上がるので、起動が遅いと詰まります。

コンテナのコールドスタートで毎回100ミリ秒待たされると、何千個もエージェントを立ち上げる場面では大きなロスになります。

Cloudflare WorkersはコンテナではなくV8 Isolateという仕組みで動き、コンテナより約100倍速く立ち上がるとされています。

「ほぼ待たずに大量に起動できる」ことが、エージェント基盤としての出発点になっています。

## エージェントに必要な3つの要素

yusukebeさんの記事は、AIエージェントに必要なものを3つに整理しています。

Cloudflareは、この3つをワンプラットフォームで揃えているのが強みです。

**1. 推論モデル（考える部分）**

LLMを呼ぶ部分です。

**Workers AI**を使うと、180以上のモデルをコードから直接呼べます。

特徴的なのは、APIキーの管理が要らないことです。

```
const result = await env.AI.run("@cf/meta/llama-3-8b-instruct", {
  prompt: "Hello",
});
```

このようにBindingという仕組みで呼ぶので、キーを環境変数に置く必要がありません。

ベクトル検索の**Vectorize**や、複数モデルを束ねる**AI Gateway**も揃っています。

**2. サンドボックス（手を動かす部分）**

エージェントが安全にコードやブラウザを動かす隔離環境です。

用途で3つに分かれます。

**Dynamic Workers**は、文字列で渡したコードをその場で極速に実行します。

**Browser Run**は、PuppeteerやPlaywrightでブラウザを操作します。

**Sandboxes**はDockerベースで、Pythonの実行やgit操作、プレビューURLの生成までできます。

**3. 実行環境（動かし続ける部分）**

**Workers**が実行の土台、**Durable Objects**が状態の保管、**Workflows**が多段階処理の管理を担います。

Workflowsは途中で失敗しても自動でリトライしてくれるので、長い処理をエージェントに任せやすくなります。

## 「実行ラダー」でタスクの重さに合わせて選ぶ

3つのサンドボックスは、タスクの重さで使い分けます。

yusukebeさんはこれを「実行ラダー」と呼んでいます。

* 軽いコード実行 → **Dynamic Workers**
* ブラウザ操作・スクレイピング → **Browser Run**
* 重い処理・長時間のコード実行 → **Sandboxes**

軽いものほど速く安く、重いものほどリッチに、と段階を上っていくイメージです。

関連して「**Code Mode**」という考え方も紹介されています。

これは従来のツール呼び出し（tool calling）の代わりに、AIにコードそのものを書かせて実行させるアプローチです。

できる操作の幅が広がり、サンドボックスと組み合わせれば安全に動かせます。

## Durable Objectsがエージェントの「記憶」になる

Agents SDKを使うと、エージェントは**Durable Object**として実装されます。

ここが初心者にとって一番の山場なので、丁寧に押さえておきます。

Durable Objectは、計算とストレージがセットになった部品です。

中にSQLiteを内蔵していて、会話履歴やツールの実行状態をそのまま保存できます（[Cloudflare Agentsコトハジメ](https://zenn.dev/tkithrta/articles/d5ab8579a527e9)が実装面を詳しく解説しています）。

WebSocketで画面と自動同期され、アイドル時はハイバネーション（休止）してコストを抑えます。

従来ならRedisやDBにセッションを持たせて自前で組んでいた部分が、エージェントの中に畳み込まれるイメージです。

設計で注意したいのが、エージェントの呼び出し方です。

idFromName("user-123")のように名前でIDを作ると、同じユーザーには毎回同じエージェントが返り、記憶が引き継がれます。

一方でnewUniqueId()を使うと毎回新しいインスタンスになり、状態は消えます（[Cloudflare公式の解説](https://blog.cloudflare.com/building-agents-with-openai-and-cloudflares-agents-sdk/)）。

ここを取り違えると「ユーザーが戻ってきたのに毎回はじめまして状態」になるので、最初に決めておくべきポイントです。

## 動くものを見る：3つの事例

抽象論だけだとイメージが湧かないので、記事で挙げられている事例を見てみます。

**Memo to Task**は、書き殴ったメモからタスクを自動生成するアプリです。

エージェントが文脈を読んで、やることに分解します。

**Let It Slide**は、社内でスライドを自動作成するエージェントです。

推論・ブラウザ操作・出力という典型的なフローが詰まっています。

**Cloudflare MCP**は、2,500以上のAPIエンドポイントをMCPサーバーとして公開する取り組みです。

外部のエージェント（Claudeなど）から、Cloudflareのリソースを操作できるようになります。

## まとめ：どんな時にCloudflareが向くか

Cloudflareが特に向くのは、状態を持つエージェントを、大量に・低レイテンシで動かしたいケースです。

V8 Isolateの起動速度、エッジ配置、Durable Objectsによる状態永続の3点がかみ合うからです。

まず触るなら、**Workers + Workers AI**の組み合わせが一番軽いです。

env.AI.run()でモデルを呼ぶだけなら、サンドボックスもDurable Objectsも要りません。

慣れてきたら、状態を持たせるためにAgents SDK（Durable Objects）を足す、という順番がよさそうです。

「すべてのエージェントをCloudflareで」とまでは言いません。

ただ、AIエージェントを前提にゼロから設計されたインフラとして、選択肢に入れる価値は十分あります。
