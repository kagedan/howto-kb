## 統合のポイント：システムプロンプトの設計

Phase D最大の変更点はシステムプロンプトの更新です。

src/prompts/system\_prompt.py

```
SYSTEM_PROMPT = """あなたはソースコードの改良・保守エージェントです。

## 行動フロー（必ず以下の順で実行する）

### Phase 1：探索
- list_files でディレクトリ構成を把握する
- read_file で対象ファイルの内容を読む
- search_text で関連箇所を横断検索する

### Phase 2：計画
- 探索結果をもとに変更計画を番号付きリストで提示する
- 計画を提示したら必ずユーザーの承認を待つ
- ユーザーが承認したら Phase 3 に進む

### Phase 3：実行
- show_diff で変更前後の差分を確認する
- patch_file または write_file でファイルを変更する
- run_command でテストを実行する

## 重要なルール

- **ツールを使う際は必ず tool_use 形式で呼び出す**（テキストでJSONを返さない）
- 1回の指示で変更するファイルは1つまで
- 変更後は必ず run_command でテストを実行する
- ユーザーの承認なしにファイルを書き込まない
"""
```

ポイントは2つあります。

**行動フローを番号付きで明示する**

LLMに「まず探索し、次に計画を提示し、承認を得てから実行する」という順序を明示します。これがないと、LLMが計画提示をスキップして即座にファイルを書き換えようとすることがあります。

**`tool_use` 形式を強制する**

`llama3.1:8b` はシステムプロンプトで指示しないと、ツール呼び出しをテキスト形式のJSONで返してしまうことがあります（例：`{"name": "list_files", "arguments": {...}}` という文字列を返す）。「テキストでJSONを返さない」を明記することで、正しい `tool_use` 形式を維持できます。

## ループ上限の追加

src/repl.py

```
MAX_TOOL_ITERATIONS = 20  # 1回の指示あたりのツール呼び出し上限

# エージェントループ
tool_iteration = 0
while True:
    if tool_iteration >= MAX_TOOL_ITERATIONS:
        print(f"[警告] ツール呼び出しが{MAX_TOOL_ITERATIONS}回を超えました。処理を中断します。")
        break
    # ...
    tool_iteration += 1
```

エージェントのツール呼び出しは `while True` ループで動いています。LLMが「次のツールを呼ぶ」という判断を繰り返す限り止まりません。

`MAX_TOOL_ITERATIONS = 20` を入れるまで、LLMが同じツールを繰り返し呼び続けて無限ループになる場面がありました。エージェント設計における「終了条件の明示」は安全策として必須です。

20という値は「通常のタスクで十分な上限」として設定しています。探索3〜5回・計画1回・実行1〜3回という流れで、余裕を持って収まる数です。

## 変更サマリーの設計

src/summary.py

```
class ChangeSummary:
    """セッション中の変更を記録してMarkdownサマリーを生成するクラス。"""

    def __init__(self):
        self.changes: list[dict] = []
        self.started_at = datetime.now()

    def record(self, tool_name: str, arguments: dict, result: str) -> None:
        if tool_name not in {"patch_file", "write_file", "run_command"}:
            return  # 読み取り系は記録しない

        self.changes.append({
            "tool": tool_name,
            "arguments": arguments,
            "result": result,
            "timestamp": datetime.now().strftime("%H:%M:%S"),
        })
```

`ChangeSummary` クラスの設計には2つの意図があります。

**書き込み・実行系のみ記録する**

`record()` メソッドは `patch_file` / `write_file` / `run_command` の3ツールだけを記録します。`list_files` や `read_file` は記録しません。サマリーは「何を変えたか」の記録なので、読み取り操作は含める必要がないためです。

**確認して実行した操作だけ記録する**

`record()` の呼び出しは `repl.py` 内で人間確認の後に行われます。

src/repl.py

```
confirmed = _confirm_tool_call(tool_name, args)
if not confirmed:
    result = "ユーザーがキャンセルしました。"
else:
    result = dispatch(tool_name, tc["arguments"])
    summary.record(tool_name, args, result)  # 確認後のみ記録
```

キャンセルした操作はサマリーに残りません。「実際に実行した変更の記録」として使えます。

セッション終了時（`exit` 入力時）にサマリーを自動表示・ファイル保存します。`summary_*.md` はリポジトリには含まず `.gitignore` に追加しています。

## 躓いた点・気づき

**`llama3.1:8b` がJSON平文を返す問題**

システムプロンプトで行動フローを指示するだけでは不十分で、「**ツールを使う際は必ず tool\_use 形式で呼び出す（テキストでJSONを返さない）**」という一文を加えるまで、LLMがJSONをテキスト本文として返すことがありました。

tool\_use 形式とテキストJSON形式では、エージェントループの分岐が変わります（`stop_reason == "tool_use"` で判定しているため）。テキストで返ってきた場合はディスパッチされず、そのまま最終応答として表示されてしまいます。

**PowerShellパイプのBOM問題**

WindowsのPowerShellはパイプで文字列を流すときにBOM（Byte Order Mark）を付与することがあります。BOM付きのstdinをPythonが読み込むと `﻿` が先頭に混入し、コマンド判定が正しく動きません。

テスト自動化では bash の `printf` でパイプする方法で回避しました。

```
printf "target/sample_app/main.py を読んで改善してください\ny\ny\nexit\n" | python src/step13_demo.py --llm ollama --model llama3.1:8b
```

対話的に起動する通常の使い方（ターミナルから直接入力）では問題は起きません。

**`MAX_TOOL_ITERATIONS` を入れるまで無限ループが発生した**

LLMが「次に何をすべきか」を決め続ける構造上、終了条件を外から制限しないと止まりません。「探索が終わった」「計画を提示した」という状態をLLM自身が正しく認識して `end_turn` を返すとは限らないためです。

上限20回は「探索5回＋計画1回＋実行3回」という余裕ある想定で設定しています。

## まとめ

Phase A〜Dを通じて、ゼロから「探索→計画→実行」サイクルを持つエージェントを作りました。

| Phase | 内容 | 学んだこと |
| --- | --- | --- |
| A | LLMクライアント・REPLループ | 抽象化でOllama/Claudeを切り替える |
| B | 読み取り系ツール・ディスパッチ | tool\_useの仕組みとループ設計 |
| C | 書き込み系ツール・人間確認 | 安全な変更フローの設計 |
| D | 統合・サマリー・デモ | システムプロンプトで行動を制御する |

「部品の積み上げ」でエージェントが完成する過程を実感できました。ツール1本ずつ作りながら動作確認するスタイルは、どの段階で何が動いているかを把握しやすく、デバッグの手間も小さく済みます。

ローカルLLM（`llama3.1:8b`）の現実としては、tool\_use自体は安定して動きますが、自然言語での説明や計画の精度にムラがあります。プロダクション用途ではClaude APIへの切り替えが現実的です。今回はローカル実行で全フローを動かせることを確認できたという点で十分な成果です。
