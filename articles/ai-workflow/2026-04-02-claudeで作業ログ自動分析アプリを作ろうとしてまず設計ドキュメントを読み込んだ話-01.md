## はじめに

みなさんは、自分が1日にどんな作業をしていたか、正確に振り返れますか？

「今日なにやったっけ…」とSlackの履歴やGitのコミットログを遡る日々に終止符を打つべく、**PCの活動ログをClaude（AI）で自動分析して、作業日記を生成するアプリ**を開発することにしました。

本記事では、そのプロジェクトの初日——設計ドキュメントとコードを読み込んで全体像を把握した過程で得た学びを共有します。

## 本日の作業内容

### やったこと（約15分間の記録）

| カテゴリ | 時間 | 割合 |
| --- | --- | --- |
| ドキュメント確認・タスク管理 | 10分 | 66.7% |
| プログラミング（コードリーディング） | 3分 | 20.0% |
| Web調査・学習 | 0.5分 | 3.3% |
| その他（離席等） | 1.5分 | 10.0% |

短い時間ではありますが、\*\*"まず全体を把握してから手を動かす"\*\*というアプローチで進めました。

### 具体的に見たもの

1. **`task.md`** — プロジェクト全体のタスク一覧と進捗
2. **`architecture.md`** — システムアーキテクチャの設計書
3. **`requirements.md`** — 要件定義ドキュメント
4. **`claudeAnalyzer.ts`** — Claude APIを使った分析ロジック
5. **`handler.ts` / `ActivityLoggerStack`** — AWSサーバーレス構成の実装

## 学んだこと

### 1. アーキテクチャ設計の読み方

`architecture.md`を読み込むことで、このプロジェクトが以下のような構成であることを把握しました。

```
PCログ収集 → S3保存 → Lambda（handler.ts）→ Claude API分析 → 結果保存・表示
```

AWS CDKでインフラを定義し、Lambdaで処理を実行するサーバーレス構成です。設計ドキュメントがしっかり書かれていると、コードを読む前に**メンタルモデル**が構築できるので理解速度が段違いでした。

### 2. Claude APIを使った分析ロジックの構造

`claudeAnalyzer.ts`では、PCの活動ログ（ウィンドウタイトルやアプリ名など）をプロンプトに組み立てて、Claude APIに投げる処理が実装されていました。

```
// イメージ（簡略化）
async function analyzeActivity(logs: ActivityLog[]): Promise<AnalysisResult> {
  const prompt = buildPrompt(logs); // ログからプロンプトを構築
  const response = await callClaudeAPI(prompt); // Claude APIへリクエスト
  return parseResponse(response); // レスポンスを構造化データに変換
}
```

ポイントは、**ログの前処理（buildPrompt）とレスポンスの後処理（parseResponse）を明確に分離**している点です。AIの入出力は不安定になりがちなので、この分離は保守性の面で非常に重要だと感じました。

### 3. AWS CDK + Lambdaの実装パターン

`ActivityLoggerStack`では、CDKを使ってLambda関数やS3バケットなどのリソースを定義しています。

```
// イメージ（簡略化）
const analyzerFunction = new lambda.Function(this, 'AnalyzerFunction', {
  runtime: lambda.Runtime.NODEJS_20_X,
  handler: 'handler.main',
  timeout: Duration.minutes(5),
  environment: {
    CLAUDE_API_KEY: secret.secretValue.toString(),
  },
});
```

Claude APIの呼び出しは時間がかかる場合があるため、**Lambdaのタイムアウトを長めに設定**しているのが実践的なポイントでした。

### 4. 要件定義 → タスク分解の流れ

`requirements.md`で定義された要件が`task.md`にタスクとして落とし込まれており、**要件 → 設計 → タスク → 実装**の流れが明確に追えました。個人開発でもこの流れを作っておくと、途中で「何をやるんだっけ？」と迷わなくなります。

## 設計ドキュメントを先に読むべき3つの理由

今日の作業を通じて改めて実感したことをまとめます。

1. **コードリーディングの速度が上がる** — 全体像がわかっていれば、各ファイルの役割を推測しながら読める
2. **実装の優先順位が見える** — アーキテクチャを理解していれば、どこから着手すべきかが明確になる
3. **手戻りが減る** — 設計意図を理解せずにコードを書くと、あとで大きな修正が発生しがち

特にAIを組み込んだアプリケーションでは、**プロンプト設計・API呼び出し・結果のパース**など考慮すべきレイヤーが多いため、先に設計を把握する価値は大きいです。
