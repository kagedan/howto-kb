## Yahooの広告AI基盤に見る、Spanner GraphとBigQuery Graphの使い分け

生成AIの活用は、いま大きな転換点にあります。これまで多くの企業AIは、レポートを作る、要約する、候補を提案する、といった「判断を支援するシステム」として使われてきました。しかし、AIエージェントの登場によって、AIは単に助言するだけでなく、在庫を確認し、契約条件を照合し、候補を評価し、承認フローを通し、実際の業務を実行する存在になりつつあります。

Google Cloudが公開したYahooの事例は、この変化を非常に分かりやすく示しています。Yahooは、デジタル広告のメディアバイイング業務をAIエージェントで自動化するために、Spanner GraphとBigQuery Graphを組み合わせた「信頼できるSystem of Action」を構築しました。Google Cloudの記事では、YahooのSeller Agentが、従来は数週間かかっていた手作業中心の広告キャンペーン運用を、ガバナンス付きのライブキャンペーンとして数秒で実行できるようにする構想として紹介されています。

この記事では、そのアーキテクチャの要点を整理しながら、特に重要な問いである **「なぜ同じGraph Dataを扱うのに、Spanner GraphとBigQuery Graphという2つのDBを使うのか」** を中心に解説します。

---

## 1. Yahooが解こうとしている課題

Yahooの対象領域は、デジタル広告のメディアバイイングです。

広告主がキャンペーンを出したいとき、裏側では多くの確認作業が必要になります。たとえば、広告在庫はあるか、対象オーディエンスは適切か、契約上その広告枠を使えるか、価格条件は合うか、ブランドセーフティに反しないか、地域や同意条件、規制を満たしているか、といった判断です。

従来、このような高価値で複雑な広告キャンペーンは、人間の引き継ぎ、スプレッドシート、手作業の分析に依存していました。Google Cloudの記事では、YahooがこのプロセスをAIエージェントによって数秒単位に圧縮しようとしていると説明されています。

ただし、ここにLLMをそのまま置けばよいわけではありません。

広告取引では、AIが「それっぽい提案」をするだけでは不十分です。存在しない在庫を提案したり、契約上使えない広告枠を選んだり、規制やポリシーを見落としたりすれば、実際のビジネス上の損失につながります。Google Cloudの記事も、リアルタイム在庫、価格ルール、ビジネス制約を決定論的に理解できないエージェントは、ハルシネーションにより重大な取引ミスを起こし得ると指摘しています。

つまり、Yahooが解こうとしている本質的な課題は、単なる業務自動化ではありません。**AIエージェントに実際の予算・契約・広告配信を扱わせるために、どのように事実に基づき、説明可能で、監査可能な実行基盤を作るか** という課題です。

---

## 2. 「System of Intelligence」から「System of Action」へ

この事例を理解するうえで重要なのが、System of Intelligence と System of Action の違いです。

**System of Intelligence** は、データを分析し、人間の意思決定を支援するシステムです。たとえば、売上予測、広告効果分析、レポート生成、キャンペーン候補の推薦などが該当します。

一方、**System of Action** は、分析や提案にとどまらず、実際に業務を進めるシステムです。Yahooの例で言えば、キャンペーン要件を受け取り、広告在庫や契約条件を照合し、候補パッケージを作り、ポリシーを確認し、必要なら人間にエスカレーションし、承認後にキャンペーンを実行します。

Google Cloudの記事では、企業がエージェント型AIを導入するにつれて、受動的なSystem of Intelligenceから、文脈・性能・説明責任を備えたSystem of Actionへ移行する必要があると説明されています。

ここで重要なのは、AIが「答える」だけなら、多少の曖昧さを人間が補えます。しかしAIが「実行する」場合、その曖昧さは直接リスクになります。

そのためSystem of Actionには、少なくとも次の3つが必要になります。

| 必要なもの | 意味 |
| --- | --- |
| 事実への接地 | AIの判断が、リアルタイムな業務データ・契約・在庫・ポリシーに基づいていること |
| 説明可能性 | なぜその判断をしたのかを後から説明できること |
| 監査可能性 | どの候補を検討し、どのポリシーを適用し、誰が承認したのかを追跡できること |

Yahooの設計では、この3つを実現するための中核として、2種類のグラフが使われています。

---

## 3. Yahoo Seller Agentの全体像

YahooのSeller Agentは、Google Cloud上で動くマルチエージェントシステムです。

購入側からのリクエストは、GKE上で稼働するplanning supervisor agentに入り、GoogleのAgent Development Kit、ADKによってオーケストレーションされます。このsupervisor agentが、在庫探索、オーディエンス照合、予測、価格分析、パッケージ推薦、ガバナンスレビュー、実行といった専門タスクへ処理を分解します。エージェント同士はA2Aプロトコルで連携し、Gemini Enterprise Agent Platformがモデル、埋め込み、予測、グラフ学習を支える構成として説明されています。

単純化すると、次のような構造です。

```
広告主・Buyer Agent
        |
        | キャンペーン要件
        v
Planning Supervisor Agent
        |
        |--- 在庫探索エージェント
        |--- オーディエンス照合エージェント
        |--- 価格分析エージェント
        |--- 予測エージェント
        |--- ガバナンスレビューエージェント
        |--- 実行エージェント
        |
        v
Knowledge Graph  : 今この瞬間の業務事実
Context Graph    : 判断履歴・監査証跡・学習データ
```

この構成の核心は、AIエージェントそのものではなく、エージェントを支える dual-graph foundation にあります。Google Cloudの記事では、Yahooのプラットフォームは「行動するために最適化されたKnowledge Graph」と、「記憶し学習するためのContext Graph」という2つのグラフに支えられていると説明されています。

---

## 4. 1つ目のグラフ：Spanner Graph上のKnowledge Graph

YahooのKnowledge Graphは、Spanner Graph上に構築されています。

このKnowledge Graphは、広告商品、広告掲載面、オーディエンスセグメント、広告在庫、契約、ガバナンス制御、ポリシーなどをノードとエッジで表現します。重要なのは、ポリシーがアプリケーションコードの奥に埋め込まれるのではなく、バージョン管理された関係としてグラフの中に置かれる点です。Google Cloudの記事では、この設計により、商品、契約上の義務、同意要件、規制制約を単一のグラフトラバーサルで評価できると説明されています。

たとえば、ある広告主が次のようなキャンペーンを依頼したとします。

```
目的     : 新商品の認知拡大
対象     : 日本在住の20〜30代
予算     : 1,000万円
地域     : 日本
制約     : ブランドセーフティ必須、同意条件を満たすこと
```

このときAIエージェントは、Knowledge Graphをたどることで、次のような問いに答えます。

* このオーディエンスに合う広告枠はどれか？
* その広告枠は今使えるか？
* 契約上、その広告主に提供してよいか？
* 価格ルールに合っているか？
* ブランドセーフティ上の制約を満たしているか？
* 地域・同意・規制条件を満たしているか？
* 過去実績から見て妥当な候補か？

このような判断は、単純なRAGやLLMの推論だけでは危険です。LLMが過去の一般知識から「この広告枠が良さそうです」と言っても、その広告枠が今空いているか、契約上使えるか、現行ポリシーに適合しているかは分かりません。

Knowledge Graphの役割は、AIの判断を **現在有効な業務事実** に接地させることです。

---

## 5. 2つ目のグラフ：BigQuery Graph上のContext Graph

一方、Context GraphはBigQuery Graph上に構築されています。

こちらは、AIエージェントが何を考え、何を比較し、どの判断を下したかを記録するためのグラフです。Google Cloudの記事では、Seller Agentの各アクションがBigQuery Agent Analytics pluginでキャプチャされ、Yahooのdecision-trace ontologyに沿って、BigQuery Graphに型付き・問い合わせ可能なContext Graphとして保存されると説明されています。

Context Graphが記録するのは、単なるログではありません。

通常のログであれば、次のようなイベント列になります。

```
10:00:01 キャンペーン要件を受信
10:00:03 在庫検索
10:00:04 候補生成
10:00:05 スコアリング
10:00:06 ポリシーチェック
10:00:07 実行
```

これだけでは、「なぜこの候補を選んだのか」「どの候補を却下したのか」「どのポリシーが効いたのか」を後から説明するのは難しくなります。

Context Graphでは、判断過程そのものが関係として構造化されます。

```
Campaign Brief
  ├─ Candidate Package A
  │    ├─ Forecast Score
  │    ├─ Applied Policy
  │    ├─ Governance Review
  │    └─ Approved
  │
  ├─ Candidate Package B
  │    ├─ Forecast Score
  │    ├─ Applied Policy
  │    └─ Rejected Reason
  │
  └─ Final Execution
       ├─ Delivery Result
       └─ Outcome Signal
```

これにより、広告主や監査人が「なぜこの広告パッケージが選ばれたのか」と聞いたときに、キャンペーンbriefから候補、スコア、ポリシー、ガバナンス判断、実行結果までをたどれるようになります。Google Cloudの記事でも、Context Graphにより、判断点、候補パッケージ、ポリシー評価、専門エージェントへの委任、実行結果が接続された証拠グラフになると説明されています。

ADKのBigQuery Agent Analytics plugin自体も、LLM呼び出し、ツール利用、ユーザー入力、モデル応答などをBigQueryに記録し、デバッグ、リアルタイム監視、オフライン評価に使うための仕組みとして説明されています。さらに、BigQuery Agent Analytics SDKは、イベントログを超えて、エージェントが扱ったリクエスト、検討した選択肢、選んだ結果をBigQueryのproperty graphとしてmaterializeできるとされています。

## 6. なぜSpanner GraphとBigQuery Graphを分けるのか

ここが、このアーキテクチャで最も重要なポイントです。

一見すると、同じ「Graph Data」を扱うなら、Spanner GraphかBigQuery Graphのどちらか一方に統一したほうが保守しやすそうに見えます。DBが1つなら、接続先も、権限管理も、スキーマ管理も、運用手順も少なくなります。

しかしYahooのケースでは、2つのグラフは「同じデータの二重管理」ではありません。より正確には、**グラフという同じ表現方法を、性質の異なる2種類のワークロードに適用している** と見るべきです。

| 観点 | Spanner Graph | BigQuery Graph |
| --- | --- | --- |
| Yahoo事例での役割 | Knowledge Graph | Context Graph |
| 主な性格 | OLTP / Operational | OLAP / Analytical |
| 扱うもの | 現在の業務事実 | 判断履歴・監査証跡・学習データ |
| 重要な特性 | 低レイテンシ、強整合、リアルタイム性 | 大規模分析、履歴、集計、監査 |
| 代表的な問い | 「今この広告枠を使ってよいか？」 | 「なぜその判断をしたのか？」 |
| たとえ | 業務ルール付きの台帳 | ドライブレコーダー兼監査台帳 |

Google Cloudの比較ドキュメントでも、BigQuery Graphは大規模な深い分析向け、Spanner Graphはリアルタイム運用向けと整理されています。BigQuery Graphは大規模グラフに対する複雑クエリ、グローバルパターン、履歴トレンド、巨大データセット上の関係発見に最適化され、Spanner Graphは小さな範囲のk-hop queryを即時に実行するアプリケーション、ミリ秒単位の不正検知、ライブ推薦などに最適化されていると説明されています。

さらに同ドキュメントの比較表では、BigQuery Graphの主なワークロードはオフライン／バッチの大規模集計、Spanner Graphの主なワークロードはオンライン／リアルタイムの高頻度・低レイテンシ読み書きとされています。クエリレイテンシも、BigQuery Graphは秒から時間、Spanner Graphはミリ秒から秒という整理です。

つまり、Yahooの設計はかなり自然に言えば、こうです。

```
Spanner Graph
= OLTP寄り
= 今この瞬間の業務状態を見て、AIエージェントが行動するためのグラフ

BigQuery Graph
= OLAP寄り
= 過去の判断履歴を分析し、説明・監査・学習するためのグラフ
```

## 7. 1つのDBに集約しなかった理由

保守性の観点では、「DBを1つにする」は確かに魅力的です。しかし、ワークロードの性質が大きく違う場合、1つにまとめることで別の複雑性が生まれます。

たとえば、すべてをSpanner Graphに寄せた場合、リアルタイムの在庫・契約・ポリシー判断を担うOperational Graphに、膨大なエージェント実行ログ、候補、スコア、却下理由、成果データ、監査クエリまで載せることになります。すると、本番の低レイテンシ処理と、重い履歴分析が同じ基盤上で混在します。

逆に、すべてをBigQuery Graphに寄せた場合、リアルタイム在庫、契約、価格ルール、同意条件、規制制約を即時に確認しながら、AIエージェントがユーザー向け業務を実行することになります。BigQuery Graphは大規模分析には向いていますが、Google Cloudの比較でも、リアルタイムの高頻度・低レイテンシ読み書きはSpanner Graph側の領域として整理されています。

1DB化によって表面上の構成は単純になりますが、次のような隠れた複雑性が発生しやすくなります。

| 1DB化で起きやすい問題 | 内容 |
| --- | --- |
| 性能干渉 | 実行時処理と分析処理が同じリソースを取り合う |
| 責務の混在 | 現在の業務状態と過去の判断履歴が同じ設計思想に押し込められる |
| スキーマ肥大化 | トランザクション用データと監査ログ用データが混ざる |
| 保持期間の衝突 | 現在データと長期監査ログでライフサイクルが異なる |
| 運用リスク拡大 | 分析クエリの失敗や負荷が本番実行に影響し得る |
| コスト最適化の困難 | 低レイテンシ処理と大規模スキャンでコストモデルが異なる |

したがってYahooの構成は、単に技術を増やしたのではなく、**実行系と分析・監査系を分けた** と理解するのが適切です。

これはソフトウェア設計で言えば、CQRSに近い考え方です。

```
Command / Operational 側
  -> Spanner Graph
  -> 現在の業務事実を見て、実際のアクションを実行する

Query / Analytical 側
  -> BigQuery Graph
  -> 判断履歴を分析し、説明・監査・学習に使う
```

## 8. 2つに分けても保守性を保つための工夫

もちろん、Spanner GraphとBigQuery Graphを併用すれば、それだけで保守性が高まるわけではありません。むしろ設計を誤れば、複雑性は増えます。

重要なのは、2つのDBを無秩序に併用するのではなく、責務の境界を明確にすることです。

```
Spanner Graph
  = Operational Truth
  = 現在有効な業務事実の正

BigQuery Graph
  = Decision Lineage
  = 判断履歴・監査・分析の正
```

Google Cloudの統合Graph Solutionの記事では、Spanner Graphがアクティブな運用を担い、BigQuery Graphが大規模分析を担うという整理が示されています。同記事では、SQLの世界でデータベースとデータウェアハウスが用途に応じて分かれているように、グラフでもワークロードごとに特化したツールが必要だと説明されています。

また、Google Cloudの比較ドキュメントでは、SpannerからBigQueryへのforward ETL、BigQueryからSpannerへのreverse ETL、BigQueryからSpannerデータを直接参照するfederated queryといった連携方法も示されています。

さらに、Data Boostを使うことで、BigQueryからSpanner Graphのデータを直接問い合わせつつ、トランザクションワークロードの性能に影響を与えずに、Spannerのリアルタイム運用データとBigQueryの履歴分析データを組み合わせた「virtual graph」を作れると説明されています。

つまり、Google Cloudの狙いは「2つのDBを別々に頑張って運用する」ことではありません。むしろ、

* 物理的にはワークロードに応じて分ける
* ただし、グラフモデル・クエリ言語・データ連携はできるだけ統一する

という方向性です。

この設計により、実行系と分析系を分離しながら、開発者やデータチームが扱う概念はできるだけ揃えることができます。

## 9. キャンペーン実行時に何が起きるのか

Yahooのキャンペーン実行フローは、2つの並行した流れとして理解できます。

**1つ目は、Knowledge Graphを使って実際に行動する流れです。**

Buyer AgentがAd Context Protocol、AdCPを通じてキャンペーンbriefを提出します。Seller AgentはKnowledge Graphを問い合わせ、関連する在庫、オーディエンス、契約上の可用性、過去実績、適用ポリシーを取得します。その後、候補パッケージを作り、予測モデルでスコアリングし、governance agentが同意、ブランドセーフティ、規制制約を確認します。ポリシー閾値内であれば自動承認され、必要であれば人間レビューにエスカレーションされ、承認後にメディアバイが実行されます。

**2つ目は、Context Graphを使って説明・監査・学習する流れです。**

実行パイプラインが進む一方で、検討された候補、付与されたスコア、適用されたポリシー、ガバナンス判断、実行結果が、キャンペーンセッションに紐づく接続済みレコードとしてContext Graphに保存されます。さらに、配信結果、アトリビューション、成果シグナルが戻ってくると、それらが元の判断に結び付けられ、将来の推薦を改善する学習データになります。

この流れを図にすると、次のようになります。

```
[Buyer Brief]
      |
      v
[Seller Agent]
      |
      | 1. 実行時判断
      v
[Spanner Graph: Knowledge Graph]
      |
      | 在庫・契約・価格・ポリシー・同意条件を確認
      v
[候補生成・スコアリング・ガバナンス確認]
      |
      | 承認または人間レビュー
      v
[Campaign Execution]
      |
      | 2. 判断履歴の記録
      v
[BigQuery Graph: Context Graph]
      |
      | 候補・スコア・ポリシー・判断・結果を保存
      v
[監査・説明・分析・学習]
```

この構成の重要な点は、AIエージェントの実行がブラックボックスにならないことです。AIが何を根拠に判断し、どの選択肢を比較し、どのルールを適用し、なぜ最終案を選んだのかが、後から問い合わせ可能な形で残ります。

## 10. この設計から学べること

Yahooの事例から学べる最大のポイントは、**AIエージェントを本番業務で動かすには、LLMだけでは足りない** ということです。

LLMは自然言語理解や推論、候補生成には強力です。しかし、企業業務で本当に重要なのは、モデルが「何を言えるか」だけではありません。むしろ重要なのは、モデルが「何を根拠に行動しているか」「その行動が許可されているか」「後から説明できるか」です。

Yahooの設計では、そのために次の2つのグラフが使われています。

```
Knowledge Graph
  -> AIの判断を現在の業務事実に接地させる
  -> Spanner Graph
  -> OLTP / Operational

Context Graph
  -> AIの判断履歴を説明・監査・学習可能にする
  -> BigQuery Graph
  -> OLAP / Analytical
```

Google Cloudの記事の中で、Yahooの担当者はこの構成を、Spanner Graphのoperational truthとBigQuery Graphのdecision lineageとして表現しています。これは非常に示唆的です。

AIエージェントが安全に自律実行するには、現在の真実と、過去の判断履歴の両方が必要です。前者がなければ、AIは誤った事実に基づいて行動します。後者がなければ、AIがなぜそう動いたのかを説明できません。

---

## 11. 他業界への応用

このパターンは、広告業界に限りません。

Google Cloudの記事でも、Yahooのアーキテクチャは金融取引やサプライチェーン物流など、高リスクな意思決定を扱う業界にも応用できるブループリントだと説明されています。

たとえば金融であれば、Knowledge Graphには顧客、口座、取引、KYC、制裁リスト、商品ルール、リスクスコア、規制条件が入り得ます。Context Graphには、AIがなぜ取引を承認したのか、なぜ保留したのか、どの規制ルールに該当したのか、誰がレビューしたのかが記録されます。

サプライチェーンであれば、Knowledge Graphには部品、サプライヤー、在庫、契約、輸送経路、代替調達先、地政学リスク、規制が入り得ます。Context Graphには、AIがなぜその調達先を選んだのか、なぜ代替輸送ルートを採用したのか、どのリスク評価が影響したのかが残ります。

医療、保険、法務、製造、通信インフラなどでも同じ考え方が使えます。

```
AIが高リスクな業務を実行する
      |
      v
現在の業務事実に基づく必要がある
      |
      v
判断履歴を後から説明できる必要がある
      |
      v
Knowledge Graph + Context Graph が必要になる
```

## 12. 実装時の注意点

この設計は強力ですが、すべての企業が最初からSpanner GraphとBigQuery Graphを併用すべきという意味ではありません。

小規模なシステムや、リアルタイム実行の要件が弱いシステムでは、BigQuery Graphだけで十分な場合があります。逆に、主な用途がリアルタイム判断で、履歴分析や監査ログの規模が小さい場合は、Spanner Graph側に寄せたほうがシンプルな場合もあります。

重要なのは、技術選定を「Graphだからどちらでもよい」と考えないことです。判断基準は、Graphかどうかではなく、**ワークロードがOLTP寄りか、OLAP寄りか** です。

| ケース | 向いている構成 |
| --- | --- |
| リアルタイムな業務判断が中心 | Spanner Graph |
| 大規模な履歴分析・監査が中心 | BigQuery Graph |
| AIが実行し、その判断を監査・学習にも使う | Spanner Graph + BigQuery Graph |

併用する場合は、次のような設計原則が重要になります。

* Spanner GraphをOperational Truthの正とする
* BigQuery GraphをDecision Trace / Analyticsの正とする
* 同じエンティティIDで両者を接続する
* スキーマ進化の責任範囲を明確にする
* 実行判断は常にSpanner側の現在データを参照する
* BigQuery側はappend-onlyに近い判断履歴として扱う
* 分析結果をSpannerへ戻す場合は、明示的なreverse ETLとして管理する
* 監査ログに含まれる機密情報・認証情報・個人情報を制御する

特に最後の点は重要です。ADKのBigQuery Agent Analytics pluginは、LLMプロンプト、ツール引数、認証関連イベントなどの詳細なイベントペイロードを記録し得るため、機密情報の扱いには注意が必要です。ドキュメントでも、認証済みツールを使う場合、client secret、access token、API keyなどのセンシティブな値がログに含まれる可能性があることが明記されています。

## まとめ：AIエージェントの競争力は「モデル」から「業務グラフ」へ移る

Yahooの事例は、AIエージェント時代の企業アーキテクチャを考えるうえで非常に示唆的です。

この事例の本質は、単に「広告業務にAIを使った」ことではありません。重要なのは、AIエージェントが本番業務を実行するために、**現在の業務事実を支えるKnowledge Graph** と、**判断履歴を残すContext Graph** を組み合わせたことです。

```
Spanner Graph
= 現在の業務事実を扱うOperational Graph
= AIが今、正しく行動するための基盤

BigQuery Graph
= 判断履歴を扱うAnalytical Graph
= AIがなぜそう判断したかを説明・監査・学習するための基盤
```

この分担は、言い換えれば **Spanner GraphをOLTPに、BigQuery GraphをOLAPに使う設計** です。

DBを2つ使うことは、表面的には複雑に見えます。しかしYahooのように、AIエージェントが予算・契約・広告配信といった高リスク業務を実際に動かす場合、実行系と分析・監査系を分けることは、むしろ責務を明確にし、性能干渉を避け、説明可能性を高めるための合理的な設計です。

これからの企業AIで重要になるのは、どのLLMを使うかだけではありません。Google Cloudの記事でも、基盤モデルがコモディティ化するにつれて、企業の競争優位は、自社の業務運用とガバナンスされた履歴を表す独自のグラフへ移っていくと述べられています。

AIが「考える」だけでなく「動く」時代には、モデルの賢さだけでは足りません。

* AIが何を根拠に動いたのか。
* その行動は許可されていたのか。
* なぜその選択肢を選び、別の選択肢を退けたのか。
* その結果から次に何を学ぶのか。

これらを支えるデータ基盤こそが、AIエージェント時代の本当の競争力になります。
