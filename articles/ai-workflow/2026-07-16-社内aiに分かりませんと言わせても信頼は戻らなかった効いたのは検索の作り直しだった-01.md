#### 3行まとめ
- 社内AIが嘘（ハルシネーション）を返し、数日で誰も使わなくなった
- 「知らないことは知らないと言え」とプロンプトに書いたが、**信頼は戻らなかった**
- 効いたのは**ハイブリッド検索＋リランキング＋根拠リンク**。回答採用率が **35% → 62%** に上がって初めて人が戻ってきた

#### 最初にやった「ポエム対策」は効かなかった

社内SaaS「Atlas」に問い合わせAI「ナビ」を入れたら、それっぽい嘘を返して1週間で空気になった。

```
Q: 経費精算の締め日は？
A: 毎月25日です！（※実際は月末。ナビの創作）
```

よくある対策として、まず SYSTEM_PROMPT にこう書いた。

```python
SYSTEM_PROMPT = """
根拠が無いことは推測で答えず「分かりません」と言うこと。
"""
```

嘘は減った。でも**信頼は戻らなかった**。理由は単純で、今度は**答えられるはずの質問にも「分かりません」を連発する**ようになったから。「嘘つき」が「使えないやつ」に変わっただけだった。

問題はキャラ（性格）ではなく、**AIが何を根拠に答えているか＝検索**の側にあった。

#### 原因：検索が「それっぽいけど関係ない文書」を渡していた

`ChatGPTを一発叩くだけ` の構成では社内データを知らないので、RAG（検索してプロンプトに混ぜる）を入れる。ここまでは普通。

問題は検索の質だった。当時のナビはベクトル検索一本で、こういう取りこぼしをしていた：

- 「`TASK-1234` の状況」→ ID完全一致に弱く、無関係な文書が混ざる
- 上位50件をそのままプロンプトに詰める → ノイズに埋もれて重要文書を見落とす（**lost-in-the-middle**）

つまり、**LLMに渡す前の材料が汚れていた**。ここを3段階で作り直した。

#### 対策1：ハイブリッド検索（キーワード × ベクトルの両建て）

固有名詞・型番・エラーコードはキーワード検索が強く、言い換え・意味の近さはベクトル検索が強い。**どちらか一方では必ず取りこぼす**ので両方を並列で叩いてマージする。

```python
def hybrid_search(query: str, limit: int = 20) -> list[dict]:
    keyword_hits = sqlite_search(query, limit=limit)      # 数ミリ秒・無課金
    vector_hits = chromadb_search(query, limit=limit) if len(query) >= 2 else []
    return merge_by_id(keyword_hits, vector_hits)[:limit] # ID重複排除＋スコア合算
```

スコアのマージは重み付き平均でも動くが、スケールの違う2スコアを正規化なしで混ぜたいなら **RRF（Reciprocal Rank Fusion／順位の逆数を足す）** が定番。

#### 対策2：リランキング（50件 → 5件に絞る）← 一番効いた

ここが本命。1回目の検索は**取りこぼさない**ことを優先して30〜50件取り、2回目で**精度優先**で5件に絞る。絞り込みには軽量LLM（Claude Haiku）を使う。

```python
def rerank_with_haiku(query, candidates, top_k=5):
    listed = "\n\n".join(f"[{i}] {c['text'][:300]}" for i, c in enumerate(candidates))
    res = anthropic.messages.create(
        model="claude-haiku-4-5", max_tokens=256,
        messages=[{"role": "user", "content":
            f"質問: {query}\n関連性が高い上位{top_k}個を番号のJSON配列で返す。\n候補:\n{listed}"}],
    )
    idx = json.loads(extract_text(res.content))
    return [candidates[i] for i in idx if 0 <= i < len(candidates)]
```

Haikuは1リクエスト $0.001 程度。50件のリランキングで**1円も使わない**のに、最終的にLLMが見る文書の質が跳ね上がる。

**実測：リランキング導入前後で回答採用率（thumbs up率）が 約35% → 約62%。** SonnetをOpusに上げるより、これを入れる方が遥かに安く効いた。

#### 対策3：根拠リンク（Citations）で「疑える状態」をなくす

回答は合っていても、**どこ由来か分からないと人は信用しない**。各文書に番号タグを振り、回答に根拠を残させる。

```python
system = """
参考資料を根拠に回答し、引用箇所に [ref:N] タグを残すこと。
資料に無いことは推測せず「該当箇所がありません」と答えること。
固有名詞・数字・日付は必ず資料から引用すること。
"""
```

フロントで `[ref:N]` を元Wikiへのリンクに変換すれば、ユーザーは根拠をワンクリックで辿れる。 **「信用していい根拠が画面にある」** 状態が信頼を底上げする。

#### 「分かりません」を捨てずに次の改善へ回す

そして最初の対策の伏線を回収する。検索が空ヒットした事実を、 **そのまま明日のナレッジ追加候補** として記録する。

```python
def search_knowledge(query, **kw):
    results = hybrid_search(query, **kw)
    if not results:
        record_knowledge_gap(query=query, detection_method="tool_empty")  # 不足知識キューへ
    return {"results": results}
```

「分かりません」が謝罪で終わらず、**不足知識キュー**に溜まって埋まっていく。検索の失敗が改善に直結するループが完成する。

#### まとめ

- 「分からないと言え」は**必要条件だが十分条件ではない**
- 信頼を戻したのは、**RAG（ハイブリッド検索）＋リランキング＋根拠リンク＋分からないと言う設計** の合わせ技
- 体感で一番効いたのはリランキング（採用率 35%→62%）
- 「LLM API は 10% の工夫、検索戦略は 50% の工夫」——モデルを上げる前に検索に手をかける価値がある
