AIエージェントを開発に組み込むチームで、ADR(Architecture Decision Record)の運用が増えています。設計判断をチームでディスカッションして決め、ADRとして導入する。複数人でレビューして承認する体制も整ってきました。

ただ、そのレビューにひとつ観点が抜けがちです。**ADRの読者が変わった**ことです。昔はADRを読むのは人間でした。今は人間も読みますが、AIに読み込ませてコーディング時の判断基準にさせる使い方が主流になりつつあります。人間のレビューを通ったADRでも、AIにとって曖昧なままだと、承認された瞬間から「モデルごとに違うルール」として動き始めます。

この記事では、ADR導入前に挟むべき「AIによる解釈テスト」と、解釈ミスが起こりにくい書き方をまとめます。

## 人間の読み方とAIの読み方は違う

人間はADRを読むとき、チームの文脈で行間を補完できます。「大きなファイルは分割する」とあれば、このチームでの「大きい」の相場感を経験から埋められるし、迷ったら次の機会に聞けば済みます。

AIは毎回まっさらな状態で文面だけを読みます。相場感は持っていないのに、質問せずにそれらしい解釈を確定させて実装を進めます。しかも解釈は一度で終わらず、セッションのたび・モデルのたびに読み直されます。曖昧な文面は「読むたびに違うルールになりうる文面」です。

人間だけが読者だった時代、ADRの曖昧さは「後で聞けば済む保留」でした。AIが主読者になった今は「無言で分岐する仕様バグ」です。

## 導入前に「解釈テスト」を挟む

承認前のADRをAIに読ませると、解釈のズレは主に3パターンで現れます。

* **誤検知のズレ**: あるモデルは違反と判定し、別のモデルは問題なしと判定する
* **理解の取り違え**: ルールの主語や適用範囲を取り違えて、意図と逆の判断をする
* **拘束力の分岐**: 「原則」「推奨」「できるだけ」を、絶対禁止と読むモデルと努力目標と読むモデルに分かれる

これを導入前に検出する手順は単純です。

1. ADRのドラフトを、普段の開発で使うモデルを含む複数のモデルに読ませる
2. 同じ質問をする——「このADRから読み取れるルールを箇条書きで復唱して。判断に迷うグレーケースも挙げて」
3. 「このコードはこのADRに違反している?」という判定問題を、OK例・NG例の両方で数問出す
4. 回答を突き合わせて、復唱がズレた箇所・判定が割れた箇所を文面の曖昧さとして修正する

見るのはモデルの正答率ではなく、**回答が一致するかどうか**です。全モデルが同じ誤読をするなら文面がそう読める証拠だし、割れるなら文面が解釈を確定させていない証拠。どちらもADR側を直します。

かかるコストは、問い合わせ・突き合わせ・文面の手直しまで込みで1本あたり30分程度。全ADRに課すのが重ければ、AIのコーディング判断に直接使われるADRに絞るのが現実的です。導入後に曖昧なADRが数週間動くと、そのADRを読んだAIの判断がコードベースに蓄積されて、文面を直しても書かれたコードのばらつきは残ります。導入前の30分のほうがずっと安上がりです。

## 書き方1: 具体例は必ず両側を書く

具体例を1つ書いて満足すると、境界線をモデルが勝手に引きます。

```
NG例しかないADR:
「巨大な関数は分割する。例: 300行の関数は分割対象」
```

300行が分割対象なのは全モデルが一致します。割れるのは80行、120行、200行です。NG側の例しかないので、「どこからセーフか」を各モデルが独自に補完します。

```
両側を書いたADR:
「関数は100行以下に保つ。
 OK例: 95行の関数はそのままでよい
 NG例: 101行の関数は分割する」
```

OK側とNG側の両方に値を置けば、境界線は文面が引いたことになり、モデルの補完余地が消えます。同期/非同期、公開/非公開のような2分類ルールも同じで、片方の条件だけ書いて「それ以外はもう一方」と読ませず、**両方の分類に該当する具体値をそれぞれ明記**します。

## 書き方2: 相対語をやめて、両側を名指しする

「上位」「新しい」「大きい」のような相対語は、基準点を読者が持っている前提の言葉です。

```
相対語のADR:
「上位レイヤーから下位レイヤーへの依存のみ許可する」
```

チームメンバーは自分たちのレイヤー構成を知っているので迷いません。AIは「このプロジェクトで何が上位か」をコードから推測することになり、プレゼンテーション層とユースケース層のどちらが上か、といった判断がモデルごとに揺れます。

相対語を使うなら基準を固定します(「レイヤーの上下は本ADR末尾の図の通り」など)。より確実なのは、**相対語自体をやめて、包含する側・包含される側を具体名で名指しする**ことです。

```
名指ししたADR:
「service層(包含する側)は repository層(包含される側)を import してよい。
 repository層から service層への import は禁止。
 OK例: order_service.py → order_repository.py
 NG例: order_repository.py → order_service.py」
```

「上位/下位」という相対関係を固有名のペアに置き換え、許可される方向と禁止される方向の両方を書く。書き方1と同じ構造です。関係にも両側があり、片側だけ書くと逆方向の扱いをモデルが補完します。

名指しの代償は汎用性です。レイヤー名を変えたらADRも直すことになります。それでも名指しに倒すのは、ADRの目的が「判断の再現」だからです。読み手によって再現されない汎用原則より、このプロジェクトで確実に再現される固有ルールのほうが価値があります。

## 書き方3: 確定に効かない文章は削る

両側の例を書くとADRは長くなりますが、長さが無制限に許されるわけではありません。ADRはコード・他のADR・作業指示と一緒にコンテキストへ積まれるので、長さはそれ自体がコストです。

* **コンテキスト予算の圧迫**: ADRが10本、各3,000字あれば、毎セッション3万字がルール読解に消える
* **中間部の減衰**: 長い入力の中間は先頭・末尾より参照されにくい(いわゆる lost in the middle)。決定の本体が長い経緯説明の後ろに埋まると、経緯は読まれたのに決定が抜ける
* **ルールの希釈**: 読ませる指示の総量が増えるほど、1本あたりの遵守率は下がる

原則は一つで、**増やしてよいのは境界を確定する文章だけ、確定に効かない文章は削る**です。

* 1つのADRには1つの決定だけ。混ざったら分割する
* 具体例は各側1つで境界が確定するなら、それ以上並べない
* ディスカッションの経緯や却下した代替案の詳細は本文から切り離し、リンク先に置く(人間は辿れて、AIの毎回のコンテキストには載らない)
* AIに常時読ませるADR本文は1本あたり数百字〜千字程度が目安

経緯を切り離すと「なぜこの決定に至ったか」の説得力は本文から薄れます。人間の新メンバーには経緯こそが価値なので、リンクは必ず残してください。
