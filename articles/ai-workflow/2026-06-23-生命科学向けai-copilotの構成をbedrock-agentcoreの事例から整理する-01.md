AWSとOpenAIが2026年6月23日に公開した事例から、生命科学や研究業務でAIを利用する際の構成と役割分担を整理します。

今回取り上げるのは、次の事例です。

| 事例 | 内容 |
| --- | --- |
| AWSのタンパク質研究Copilot | 自然言語、タンパク質Embedding、ベクトル検索、生成AI要約を統合 |
| OpenAIの免疫学研究事例 | GPT-5 Proが研究者による3年前の実験結果の再検討を支援 |
| AgentCoreのマルチテナント構成 | 医療AIエージェントを複数組織へ提供する際のデータ分離とコスト追跡 |

## タンパク質研究Copilotの構成

AWSは、Amazon Bedrock AgentCoreを使ってタンパク質研究Copilotを構築する方法を公開しました。

このシステムは、次の3つの機能を組み合わせています。

1. 自然言語から検索条件を抽出
2. タンパク質Embeddingを使ったベクトル類似検索
3. 検索結果の科学的な要約

全体構成は次のように整理できます。

記事では、Strands Agents SDKを使って複数の専門ツールを1つのエージェントから呼び出しています。

| 役割 | 技術要素 |
| --- | --- |
| エージェント実行 | Amazon Bedrock AgentCore |
| ツールオーケストレーション | Strands Agents SDK |
| Embedding生成 | ESM-C 300MをSageMaker AIへデプロイ |
| ベクトル検索 | Amazon Aurora PostgreSQL、pgvector |
| 要約 | Amazon Bedrockの基盤モデル |

自然言語で「デングウイルスの特定ペプチドに似た配列を10件探して」と入力すると、エージェントが件数や対象配列を構造化し、検索ツールを呼び出します。

専門分野向けAIでは、LLMだけに回答を生成させるのではなく、専門モデルや検索基盤を組み合わせることが重要です。

## RAGとは少し異なる専門検索

一般的なRAGでは、文書をEmbeddingへ変換し、意味的に近い文章を検索します。

今回の事例では、検索対象が文書ではなくタンパク質配列です。

そのため、汎用的なテキストEmbeddingではなく、タンパク質配列向けの専門モデルを使用しています。

専門業務向けの検索では、次の観点を整理する必要があります。

* 何をEmbeddingへ変換するか
* 対象データに適したEmbeddingモデルは何か
* ベクトル類似度だけで十分か
* どのメタデータで絞り込むか
* 検索結果を誰が評価するか

今回の構成では、ベクトル類似検索とメタデータフィルタリングを1つのクエリで組み合わせています。

類似度が高いことと、研究目的に適していることは同じではありません。

そのため、AIが候補を絞り込み、研究者が結果を評価する役割分担が必要です。

## AIが研究仮説の再検討を支援する

OpenAIは、免疫学者Derya Unutmaz氏がGPT-5 Proを使い、3年前の実験結果を再検討した事例を公開しました。

研究テーマは、グルコースがT細胞の発達や分化へ与える影響です。

過去の実験では、研究チームが予想していた結果と実際の結果が一致せず、説明できない状態が続いていました。

GPT-5 Proへ実験条件や結果を提示し、異なる仮説を検討したことで、新しい説明と追加実験につながったとされています。

この事例の流れは、次のように整理できます。

重要なのは、AIの出力がそのまま研究成果になったわけではない点です。

研究者がAIの提案を評価し、実験によって確認しています。

AIは、既存知識を横断して候補を出すことには強い一方で、最終的な妥当性の判断には専門知識と実証が必要です。

## マルチテナント環境では分離を設計する

研究支援AIを複数の研究機関、病院、企業へ提供する場合、マルチテナント設計が必要になります。

AWSは、複数の病院やクリニックへ医療AIエージェントを提供する例を使い、AgentCoreで共有基盤とテナント分離を両立する構成を紹介しています。

主な要件は次のとおりです。

* テナント間のデータ分離
* Knowledge Baseの分離
* Memoryの分離
* モデルアクセスの制御
* サービスプランごとの機能差
* テナント単位のコスト追跡
* ユーザー単位の認可

共有基盤を使う場合でも、Knowledge Base、Memory、モデルアクセス、コスト追跡の各層でテナントIDを維持する必要があります。

特に医療や研究データでは、誤回答だけでなく、別テナントのデータが検索結果へ混ざることも重大な問題です。

## 実装時のチェックポイント

研究支援AIを設計する際は、以下の観点を確認するとよさそうです。

| 観点 | 確認事項 |
| --- | --- |
| データ | 配列、論文、実験結果など、検索対象を明確にする |
| Embedding | 対象データに適した専門モデルを選択する |
| 検索 | ベクトル検索とメタデータ条件を組み合わせる |
| 要約 | 検索結果と生成された説明を区別する |
| 検証 | 専門家によるレビューと実験を必須にする |
| 権限 | テナント、ユーザー、ツールごとに制御する |
| 監査 | 検索条件、参照データ、出力、ツール実行を記録する |
| コスト | モデル、推論基盤、データベースの利用量を追跡する |

## まとめ

生命科学や研究業務でAIを活用する場合、LLMだけでは十分ではありません。

専門モデル、ベクトル検索、メタデータ、AIエージェント、専門家による検証を組み合わせる必要があります。

また、本番環境では、テナント分離、権限管理、監査、コスト追跡も必要です。

AIは研究者を置き換えるものではなく、探索できる候補や検討できる仮説を広げるための道具として設計することが重要だと考えます。
