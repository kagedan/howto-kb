## はじめに

ケラケラというバンドに「笑ったもん勝ち」という曲があります。落ち込んだ時こそ笑え、というポジティブなメッセージの一曲で、最近知り合いから教えてもらって良い歌だなと思いました。

そこから「これをゲームにできないか？」が始まりです。AmiVoiceは笑い声検出ができるので、**早く笑った人が勝ち** みたいなパーティーゲームを想像していました。

ところが調べていくと、**笑い声を判定根拠にしたゲームは「笑ったら勝ち」より「笑ったら負け」のほうが圧倒的に作りやすい**ことが分かってきます。最終的にできあがったのは、お題を真顔で朗読してもらい、笑ったら陥落する **真顔チャレンジ** という、当初のコンセプトと真逆のゲームになりました。

実装中に向き合った技術的な課題は主に2つです。

* **AmiVoice使う意味は？** → お題テキストとの「進捗」と「余分」を分けて判定根拠にした
* **音量検出の閾値が決まらない** → 固定→移動平均→固定baseline、と試行錯誤した

この記事ではこの2つを中心に、実装の流れとハマりどころを紹介します。

## 作ったもの

完成形はこんな感じです。

<https://youtu.be/GTYg33ejWt8>

1. スタートを押すと、Claude APIが生成した「真顔で読むほど笑える」お題が表示される
2. プレイヤーは声に出して朗読する
3. 笑った瞬間に💥が出て「○○秒で陥落」と表示される
4. 最後まで真顔で読み切れたら🏆完走

「真面目な口調で読むほど笑える」お題が次々生成されるので、何度遊んでも展開が違うのが特徴です。

## 全体アーキテクチャ

技術スタックは以下のとおりです。

* **フロント**: HTML + Vanilla JS（Web Audio APIでマイク取得 → 16kHz PCM変換）
* **バックエンド**: Node.js
* **音声認識**: AmiVoice API
* **生成AI**: Claude API

ゲーム性質上、認識結果が **即時に返ってくる必要がある** ためWebSocketインタフェースを使っています。お題は短いテキスト生成なので、速くて安いHaiku 4.5を選びました。

## 1. AmiVoiceは笑い声を文字に起こさない、を発見するまで

最初の構想はシンプルでした。

> AmiVoiceで朗読を音声認識して、認識テキストに「あはは」「ふふ」「ぷっ」みたいな笑いトークンが現れたら陥落とする

これなら認識結果が判定根拠そのものになる、と。実装して動かしたら、認識結果がこうなりました（実機のデバッグログより抜粋）。

```
[2.69s] U <- "本日も..."
[3.69s] U <- "本日もご利用ありがとう..."
[4.27s] A <- "本日もご利用ありがとうございます。"
[6.29s] U <- "違いのメンテナンス予定日を..."
[7.02s] A <- "稚貝のメンテナンス予定日は未定"
[9.51s] A <- "月が綺麗ですね。"
[10.27s] e ok
```

朗読部分は綺麗に拾われていますが、笑った瞬間の音声はテキストとして残っていません。AmiVoiceの音声区間検出（VAD）から見ると、笑い声は「無音」「ノイズ」扱いになるか、フィラーとして除去されているようでした。

> **AmiVoice使う必要あった？**

最大の問題は技術的な失敗そのものより、この問いに答えられなくなることでした。

### 突破口: 「お題テキスト」というアンカーを使う

ここで方針転換しました。「笑い声を直接検出する」のをやめて、**お題テキストとの差分で間接的に検出する** ことにします。

朗読中の認識テキストは、ふつうはお題テキストとほぼ一致するはずです。ところが笑ってしまうと…

* **朗読が止まる** → 認識テキストの進捗が更新されなくなる
* **言い間違える** → お題にない単語が混ざる
* **笑い声が漏れる** → ごく稀に「ハ」「フ」みたいなトークンが入る

これらは全部、「**お題と認識テキストのズレ**」として観測できます。検出の主役を「笑い声トークン」から「**お題との差分**」に変えれば、AmiVoiceの認識結果が判定の主軸として使えると考えました。

## 2. お題との「進捗」と「余分」を分けて検出する

お題テキストと認識テキストを文字単位で照合して、

* **progress（進捗）**: お題の何文字目まで一致したか
* **extra（余分）**: お題に含まれない余分な文字列

の2つに分離しています。

```
// お題と認識テキストを比較
//   promptNorm:  正規化済みのお題テキスト(カタカナ→ひらがな、句読点除去など)
//   recognized:  AmiVoiceから返ってきた認識テキスト(累積)
// 戻り値:
//   progress: お題の何文字目まで一致したか
//   extra:    お題にない余分な文字列(笑い声のかけらや言い間違いがここに集まる)
//   done:     お題を最後まで読み切ったか
function diffWithPrompt(promptNorm, recognized) {
  const r = normalize(recognized);
  let pIdx = 0;
  const extraChars = [];
  for (const c of r) {
    if (pIdx < promptNorm.length && promptNorm[pIdx] === c) {
      pIdx++;                    // お題と一致 → 進捗を進める
    } else {
      extraChars.push(c);        // お題にない → extra に積む
    }
  }
  return {
    progress: pIdx,
    extra: extraChars.join(''),
    done: pIdx >= promptNorm.length,
  };
}
```

正規化（カタカナ→ひらがな、句読点・空白除去）を挟んでいるのは、AmiVoice側の出力ゆれを吸収するためです。

この `progress` と `extra` を使って、3種類の判定を回します。

| 判定 | 仕組み |
| --- | --- |
| **笑い声テキスト検出** | `extra` に「あは」「ふふ」等のパターンが含まれたら陥落 |
| **朗読つっかえ検出** | `progress` が2.2秒以上更新されず、まだ最後まで読めていなければ陥落 |
| **音量急上昇検出** | （次章）baselineの1.8倍を超えたら陥落 |

`extra` に対してだけ笑いパターン照合をかけるのがポイントです。お題本文に「は」が多いと「笑ったか朗読中の『は』か」の区別がつきませんが、**お題と一致した部分は extra に入らない**ため、誤検出が大きく減ります。

例えばお題「明日、君の心に大雨が降ります。傘はありません。」を読みながら笑った場合、

```
進捗: 「明日君の心に大雨が降ります傘」までヒット (progress=18)
extra: "あはは"                                      ← ここに笑いが残る
→ extra に /[はハ]{2,}/ がマッチ → 陥落
```

という流れになります。

### つっかえ検出も組み合わせる

笑った瞬間に必ず「あはは」みたいな音が出るとは限りません。**口を押さえて笑いをこらえる**と、朗読が単に止まります。これを検出するのが「つっかえ判定」です。

「開始直後は判定無効」と「1文字も進んでない時は判定しない」のチェックがそれぞれ大事で、これがないと「マイクをONにしただけで陥落」になります。

## 3. 動的baseline方式の音量検出を補助に

テキストベースの検出だけでは、**AmiVoiceからのレスポンスが返ってくるまでのタイムラグ（数百ms〜）** で「笑った瞬間に即陥落」の演出ができません。そこで音量検出をローカルで並行させて、即時判定の役割を持たせています。

ここに辿り着くまで、閾値方式は3回作り替えました。

### 試行錯誤の経緯

**第1案: 固定閾値** `rms > 0.15`  
→ 環境やマイク感度でまったく刺さらない人と過敏な人に分かれてしまった。

**第2案: 移動平均ベースの動的閾値** `rms > 直近2秒平均 × 2.2`  
→ 笑い声が続くとbaselineが追従して感度が下がり、結局陥落しない。

**第3案: 開始直後の固定baseline** `rms > 開始1.5秒の平均 × 1.8`  
→ これでようやく安定。

```
// 開始から CALIBRATION_MS の間、RMS を蓄積
if (inCalibration) {
  calibSum += rms;
  calibCount++;
} else {
  // キャリブレーション完了時点で baseline を固定
  if (baseline === 0 && calibCount > 0) {
    baseline = calibSum / calibCount;
  }
  const isLaughLike = rms > MIN_RMS
                   && baseline > 0
                   && rms > baseline * LAUGH_RATIO;   // 1.8倍超え
  // ...
}
```

`baseline` を**固定**するのが鍵で、移動平均だと笑いが0.5秒くらい続いただけでbaselineが追従してratioが下がってしまいます。「最初の朗読音量」を**変動しない基準** として持つほうが、今回は音量の「急上昇」が綺麗に観測できました。

### confidenceは使えなかったのか？

[以前書いた記事](https://zenn.dev/krishna_data/articles/897d73700e62fc)でAmiVoice の `confidence` を使ったので、今回も似たことはできないか考えました。

理屈上は、笑った瞬間は発音が乱れて confidence が下がる、というのは起こりそうです。ただ実機ログを見ると、笑い声が **そもそも認識テキストとして出てこない**（無音扱いになる）ことが多く、トークン自体が存在しないので confidence も観測できない、というケースが大半でした。

朗読中の confidence は普通に高いままなので、今回は使っていません。

## 4. Claude APIでお題を動的生成する

お題はゲーム性に直結します。最初はハードコードしていましたが、すぐに飽きるので都度生成するようにしました。生成1回あたり数百トークンなのでHaikuで十分速くて安いです。

```
const PROMPT_GENRES = [
  { name: 'シュール一行系',   hint: '不条理で間の抜けた状況描写' },
  { name: '無感情指令系',     hint: '感情のない口調で読むと逆に面白い文章' },
  { name: '丁寧すぎる説明書', hint: '取扱説明書風だが内容が3割おかしい' },
  { name: '逆ギレ独白',       hint: '冷静な独白から突然強気になる構成' },
  { name: '誤読誘発系',       hint: '漢字や専門用語を真面目に読み上げると間が抜ける' },
  { name: 'AI口調指令系',     hint: '関西弁のAI、ツンデレAI、丁寧なヤクザ等、口調縛り' },
];

const msg = await anthropic.messages.create({
  model: 'claude-haiku-4-5-20251001',
  max_tokens: 300,
  messages: [{
    role: 'user',
    content: `あなたはお笑い作家。「真顔チャレンジ」のお題を1つだけ生成してください。

【ジャンル】${genre.name}
【ジャンルの説明】${genre.hint}

【条件】
- 80〜120文字、声に出して15〜25秒で読み切れる長さ
- 笑いのピークが文の中盤〜後半に来る構成
- 下ネタ・差別・政治・暴力・特定の実在人物のネタはNG
- 真面目な口調で読むほど笑える内容に

【出力形式】お題テキストのみ。前置き・引用符・解説・改行不要。`,
  }],
});
```

ジャンルなしだと出力が均質化してくる（似たトーンのお題ばかり）のですが、6ジャンルからランダム選択するだけで、多様化しました。

* リアルタイム判定（音量＋テキスト差分）と動的なお題生成（Claude）でゲームが成立する
