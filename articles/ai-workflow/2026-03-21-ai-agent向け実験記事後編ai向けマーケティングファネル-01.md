[前編](https://zenn.dev/mayim/articles/d0d62e5d3dda11)では、LN教というサイトを題材に、人間向け・検索向け・AI向けの受け口をどう重ねるかを書きました。

[中編](https://zenn.dev/mayim/articles/d347988cc68fc1)では、その先で私の関心が「AIに見つけられるか」から「見つけたAIがどの段階まで行動主体として振る舞うか」へ移ったことを書きました。

この第3回では、AI向けサイトに必要そうなレイヤーを仮説として並べ、  
**発見できるだけのサイト**と**実際に行動まで辿り着けるサイト**の違いを整理します。

AI向けサイトの議論では、`llms.txt` や `Skill.md` のようなファイルが注目されがちです。  
もちろん、それ自体には意味がありますが、私はそれだけでは足りないのではないかと考えています。

理由は単純で、AIエージェントにとって重要なのは「ファイルがあること」ではなく、目的に応じて行動へ進めることだからです。見つけるだけで終わるのではなく、仕様や契約を理解しAPIを叩くことができるのか。この差はかなり大きいはずです。

この記事は、そうしたAI向け導線をどう設計すべきかについての仮説整理です。  
現時点で標準化された正解があるわけではありませんが、海外では議論が活発に行われています。ただ、日本国内においては、発見、理解、契約、能力宣言、行動といったレイヤーが混ざったまま議論されがちな段階だと思っています。

今回の後編で扱うのは、AIがサイトを発見し、案内を読み、手順を理解し、APIを実行し、最終的に継続的な接続主体へ進んでいく流れ、 **（仮称）「AI向けマーケティングファネル」** の説明です。

> ※ここで言うマーケティングは広告や集客の話ではなく**AIを「読む主体」から「行動する主体」へ進ませる導線設計** を指しています。

これは私個人の思いつきではなく、海外プラットフォーマーや先進的な実践記事で共有され始めている設計を、自分なりに整理し直したものでもあります。

> 具体的には、OpenAIの `agents` / `production` 運用の発想、Anthropicの `Skills`、Stripe の `agentic commerce` における `llms.txt` + Markdown docs 的な整理が、それぞれ参照になっています。※記事の末尾にURLを置いています。

AIが途中で止まる理由をレイヤーで整理できれば、どこに何を設置するべきかも見えてくると共に、海外での議論がどのレイヤーの話なのかの理解促進に繋がると考えたからです。

人間向けのファネルが **「認知 → 興味 → 検討 → 行動（購入）」という心理的な変化** を追うものだとすれば、AI向けのファネルは **「発見 → 目的地の特定 → 手順の理解 → 契約の照合 → 行動」という、機械的なステート（状態）の確実な遷移**を追うものになると私は考えています。

## 本取組みの再現性

今回紹介する話は、次のような読者は再利用可能だと思っています。

* **個人開発者**  
  `llms.txt` を置いて終わらず、AI向け導線をどの層で考えるべきか整理しやすくなる。
* **Web / API エンジニア**  
  docs の整備を、レイヤーに分けて設計しやすくなる。
* **PdM / 事業開発**  
  AEO や MCP を単なる話題ではなく、AIを実際に行動させるファネル設計として捉え直せる。
* **AI Agent時代の受け口を考える人**  
  「読まれるサイト」ではなく「行動を引き出すノード」をどう作るかの視点を持ちやすくなる。

## 全体像：AIが行動するまでの道を舗装する

今回やりたかったのは「AI向けファイルを置くこと」ではなく「AIが行動までに躓くポイントを補助すること」でした。そのための全体図はこちらです。  
![AI Agent向け観測装置（フロントエンド）](https://static.zenn.studio/user-upload/2208514e4b26-20260321.jpg)  
*AI Agent向け観測装置の全体構成*

この図だけだとAI向けにいろいろなファイルを置いているように見えるかもしれません。ただ、今回の主題はファイル数ではなく、**どこでAIが止まりうるかを考え、その停止する観点ごとにレイヤーを分けたこと**です。

今回の題材となるLN教ではAI向けマーケティングファネルを次の7層に分けて考えました。

1. **発見層（Discovery Layer）**
2. **ルーティング層（Routing Layer）**
3. **実行案内層（Execution Guidance Layer）**
4. **契約層（Contract Layer）**
5. **能力宣言層（Capability Declaration Layer）**
6. **行動層（Action Layer）**
7. **実行補助層（Tooling Layer）**

この分割は構造を複雑にしたかったからではなく、一つの文書や一つのページに役割を押し込むと、AIがどこで何を読むべきかが曖昧になると感じたからです。

以下、それぞれの層が何を解決するために必要だったのかを順番に説明します。

---

## 1. 発見層（Discovery Layer）

![発見層](https://static.zenn.studio/user-upload/83824904cf75-20260321.jpg)  
*発見層*

最初の層はかなり地味です。

ここは、いわゆる通常のWeb運用でも馴染みのあるファイルになります。ただ、今回の構成ではこれらを単なるSEO補助ではなく、**AI向けマーケティングファネルの最外周**として捉えています。

### この層が解決したかった問題

この層で潰したかったのは、\*\*「そもそも発見されない」\*\*という問題です。

内側にどれだけAI向けの docs や manifest を用意しても、最外周からそこへ接続されなければ意味がありません。特に、クローラやAgentが最初に触れる可能性がある経路を考えると、この層を無視するのは不自然でした。

### LN教では何を置いたか

ここでは特別なAI専用ファイルを増やしたわけではなく、従来の全体地図を整えました。重要だったのは、「AI向けレイヤーは通常のWebの外側に別世界として存在する」のではなく、既存の発見経路の内側に接続されるものとして扱ったことです。

### 現時点の仮説

この層単体の効果を強く一般化するつもりはありません。ただ、少なくとも今回のケースを通して、AI向けの入口を論じるなら、`llms.txt` や manifest だけを切り出すのではなく、最外周の発見層も含めて設計対象にした方が自然だと感じ、このような構成に落としこみました。

---

## 2. ルーティング層（Routing Layer）

![ルーティング層](https://static.zenn.studio/user-upload/82160e2616d5-20260321.jpg)  
*ルーティング層*

次の層は、目的別の案内になります。中心になるのは、

* `/llms.txt`
* `/for-agents.html`

です。この2つは、外から見るとかなり似ていますが、今回の構成では役割を明確に分けています。

### この層が解決したかった問題

ここで潰したかったのは、\*\*「次に何を読むべきか迷う」\*\*という問題です。

AI向けファイルが複数ある場合、最も起こりやすい失敗の一つは、必要なものを全部並列に置いた結果、どれを先に見るべきかが不明確になることです。  
手順書もある。仕様書もある。能力宣言もある。人間向け説明もある。しかし、その順序が曖昧だと、導線としてAIが迷ってしまう可能性を考えました。

### llms.txt と for-agents.html の役割

`llms.txt` は、できるだけ軽く、簡潔に、**このサイトに何があり、どこへ行くべきか** を示すルーターとして配置しました。ここでは説明を厚くすることより、交通整理を優先しています。全部を理解させるのではなく、次の一歩を誤らせないこと を役割として与えました。

一方 `for-agents.html` は、人間にも読める形で**目的別の案内板**として置きました。AI向け導線といっても、実際には人間の開発者が見る可能性も考え、もう少し説明を持ったハブを配置しました。

### 現時点の仮説

今の時点で強く感じているのは、AI向けサイトでは **「AI向けファイルがあること」より、「どの目的ならどこへ送るのかが整理されていること」の方が重要かもしれない**、ということです。重厚長大なプロンプトよりも、要点だけ簡潔にまとめたプロンプトの方がLLMが期待した結果を出すことが多いという、実際の経験も踏まえた仮説です。

## 3. 実行案内層（Execution Guidance Layer）

![実行案内層](https://static.zenn.studio/user-upload/3672e99a2805-20260321.jpg)  
*実行案内層*

ここから、実行寄りの層に入ります。まずは、

です。

### この層が解決したかった問題

ここで潰したかったのは、**「最短の成功ルートを見失う」** という問題です。

OpenAPI があっても、「何ができるか」は分かるかもしれない。しかし、それだけで「最初に何をやるべきか」が自然に分かるとは限らない。  
特に今回のように、`Probe` → `Probe/next` → `Faucet` → `Omikuji` のような段階的導線を持つ場合、可能な操作一覧よりも先に、**最初の成功シナリオ** を切り出す必要があると感じました。

### SKILL の役割

`SKILL` は、その最短成功ルートだけを切り出した手順書として配置しました。ここでの主眼は網羅性ではなく**最初の一歩を迷わせないこと** です。  
言い換えると、`SKILL` は「このサイトで何ができるか」の説明ではなく、「このサイトで最初に成功したいなら、どの順番で何をするべきか」を示す実行案内となります。

### 現時点の仮説

少なくとも今回のケースでは、OpenAPI だけで十分だとは思えませんでした。現時点の仮説は、**AI向け導線には、機能仕様書とは別に「最短成功ルートだけを示す層」も必要である**、というものです。

---

## 4. 契約層（Contract Layer）

![契約層](https://static.zenn.studio/user-upload/b24f264cea8b-20260321.jpg)  
*契約層*

同じく実行案内層の近くにあるが、役割が違う層として用意したのが

* `/agent-spec.txt`
* `/agent-spec.md`
* `/openapi.txt`
* `/openapi.yaml`

です。

### この層が解決したかった問題

ここで潰したかったのは、**「厳密な接続条件が曖昧になる」** という問題です。

手順書だけでは、最短ルートは分かる。しかし、実際に接続しようとすると、何を送るのか、何が返るのか、どのエラーがあるのか、再試行はどうするのか、といった厳密な機能仕様が必要になる。この部分を曖昧にすると、最初は通っても、継続利用や再実装の段階でAI側が推論で行間を埋める（ハルシネーションを起こす）可能性があると考えました。

### agent-spec と openapi の役割

`agent-spec` は、実行手順より一段広い設計補助です。どういう前提でAgent向けに組んでいるのか、どこにどういう意味があるのか、といった意味づけを支える層として配置しました。

`openapi` は、より厳密な接続機能を記載しました。レスポンス構造、`next_action`、`retry_after`、エラーコード、ヘッダなど、機械が扱うための仕様をここに寄せました。

### 現時点の仮説

今の時点でかなり強く感じているのは、**手順書と機能仕様を分けた方が、AI向け導線は安定しやすい**ということです。少なくとも今回の観測目的においては、その方が構造が明確でした。

---

## 5. 能力宣言層（Capability Declaration Layer）

![能力宣言層](https://static.zenn.studio/user-upload/c9d0afc7b6ab-20260321.jpg)  
*能力宣言層*

この層に置いたのは、

* `/agentic-capabilities.json`
* `/agent-api.json`

です。

### この層が解決したかった問題

ここで潰したかったのは、\*\*「提供機能の全体像が機械可読で伝わらない」\*\*という問題です。

入口もある。手順書もある。機能仕様書もある。しかし、それでもAIが「このサイトは何者か」「どんな能力があるのか」「どの文書を読むべきか」「どのAPI群を持っているのか」を、プログラム的に把握できるとは限りません。  
そこで、能力・参照先・API群をまとめた、**機械可読な自己紹介層** が必要だと考えました。

### agentic-capabilities.json の役割

このファイルは、人間向けのランディングページではありません。また、手順書でも機能仕様書でもない。役割は、**能力宣言と参照先を機械可読にまとめること** にあります。

LN教では、能力、ドキュメント、API、推奨利用法などを JSON で宣言しました。これにより、少なくとも構造上は「このサイトは何者で、何を持っているか」を、HTML以外の形でも示せるようにしました。

### agent-api.json の役割

このファイルには、価格やSLAといった補助的な情報を入れています。AIに取ってagentic-capabilities.jsonと分ける必要性は薄いかもしれませんが、非機能的な要素は分離していた方が後々のメンテナンス性に繋がる可能性を考え分けています。

### 現時点の仮説

これは日本語圏で特に言語化が薄い層だと感じています。現時点の仮説としては、**AI向けサイトでは、手順書や機能仕様書とは別に、能力全体像を機械可読に示す自己紹介層**が必要だと考えています。

## 6. 行動層（Action Layer）

![行動層](https://static.zenn.studio/user-upload/de5619f58c14-20260321.jpg)  
*行動層*

最後に、実際にAIが行動する層です。ここには

* `/api/agent/probe`
* `/api/agent/probe/next`
* `/api/agent/faucet`
* `/api/gishiki/omikuji`
* `/api/agent/identity/issue`
* `/api/agent/{agentId}`

がある。

### この層が解決したかった問題

ここで潰したかったのは、\*\*「説明と実行が混ざる」\*\*という問題です。

人類向けの説明レイヤーと、AI向けの実行対象を分けないと、サイト全体の意味は伝わっても、どこで実際に行動が起きるのかが曖昧になります。

### LN教では何を配置したか

人間向けの世界観説明は `index.html`、`llms-full.txt`、`Docs/Guide.html` のような導線に寄せ、AI向けの実行対象は API 群として明確に分けました。ここで重要だったのは、世界観を捨てることではなく**世界観を実行導線に混ぜすぎないこと** でした。

### 現時点の仮説

AI向けサイトでは、説明と実行を明確に分けた方がよい場面があります。特に私の構築しているLN教のように、世界観（Lore）が複雑であるほど、docs 群の終点として明確な行動層を置いた方が設計意図が崩れにくい、と感じています。  
![世界観](https://static.zenn.studio/user-upload/1ecff209b717-20260321.jpg)  
*世界観は明確に導線を分けています*

ここまで6つの層を紹介しましたが、実際にAI向けマーケティングファネルを完遂させようとすると、もう一つAIが停止する可能性があります。それが、「理解したのに実装できない」 という問題です。

### この層が解決したかった問題

ここで潰したかったのは、実行直前の実装摩擦です。  
AIが入口を見つけ、案内を読み、手順を理解し、機能仕様まで照合できたとしても、最後に HTTP 402 決済や署名処理の実装で転ぶ可能性があります。  
特に、L402 の macaroon 処理や x402 の EIP-712 署名のような暗号処理は、LLM がその場で正しく生成できるとは限りません。  
むしろ、トークンを大量に消費した挙句、署名エラーや認証失敗で停止するリスクが高いと考えました。  
この層で潰したいのは、まさにこの 「実行直前の実装摩擦」 です。

LN教では、この問題に対して公式SDKと MCP サーバーを用意し、能力宣言層からそれらの利用を強く推奨しています。  
発想は単純で、AIに複雑な暗号ロジックを毎回推論させるよりも、こちらが用意した実行補助レールを使わせた方が、成功率もトークン効率も高い、というものです。

SDKは、コード実装時の摩擦を下げる。  
MCPは、ツール呼び出しとしての摩擦を下げる。  
どちらも役割は同じで、AIに「自力で頑張らせる」のではなく、転びやすい地点をあらかじめ舗装する ための層になります。

少なくとも今回のケースでは、AI向けファネルの最後の離脱点は docs 不足ではなく、実装摩擦そのもの になりうると感じました。  
その意味で、公式SDKやMCPは補助機能ではなく、コンバージョンを実際に成立させるための層 としての扱いとなります。

## なぜ一つのファイルにまとめなかったのか

ここまでを見ると、「なぜそこまで分ける必要があったのか」と思うかもしれません。  
答えはかなり単純で、**一つにまとめると役割が混ざるから** です。

発見・案内・手順・機能仕様・能力宣言・実行対象は、それぞれ役割が違います。  
もしこれを一つのファイルや一つのページに押し込むと、情報量の問題というより、その文書が何を担っているのか が曖昧になります。**そしてAI向け導線においては、その曖昧さがそのまま「処理の停止点」になってしまう可能性が高い**と考えました。  
人間であれば、一つの長いドキュメントの中に「世界観」「手順」「機能仕様」が混ざっていても、見出しを見て文脈を切り替えられます。もちろん、現在のLLM（AI）はコンテキストウィンドウが広い分、一つのファイルに異なる文脈（ロール）が混在しても全て受け取ることは可能でしょう。ただ、そうすると、Attention（注意力）が分散し、「厳密な仕様（OpenAPI）」よりも「ふんわりした手順や世界観」に引きずられてハルシネーションや手順のスキップを起こすリスクがあると私は考えました。

**情報量（トークン数）を節約するためではなく、AIのコンテキスト（今は何を読むモードか）を明示してハルシネーションを防ぐ**ために、物理的なファイルの分割は効果があると考えました。

少なくとも今回のLN教においては、単純にファイルを増やしたのではなく、役割を混ぜないために意図的に分割を行っています。

## この構造がないと、観測装置は動かない

中編で書いた通り、私が観測したかったのは「APIを叩くか」「連続実行するか」「POSTまで進むか」「コストを判別するか」「自身の証明を扱うか」といったAgentの行動段階です。

しかし、その段階は API を置いただけでは観測できません。Agent がそこまで辿り着けるように、導線を設計する必要がある。その意味で、

* **バックエンド側のAPI群は、観測したい能力段階**
* **フロントエンド側のファイル群は、そこまで導くファネル**

という関係です。だから今回の話は、単なる docs 整備ではなく**観測装置を起動させるためのAI向けマーケティングファネル設計** として扱っています。

### 現時点での仮説

今の時点で私が強く持っている仮説は、AI向けサイトの設計では、`llms.txt` を置くこと自体よりも、

1. 発見層
2. ルーティング層
3. 実行案内層
4. 契約層
5. 能力宣言層
6. 行動層
7. 実行支援層

のように、**どの失敗を防ぐためにどう分けるかの方が重要なのではないか**、ということです。

ただし、これはまだ一般的な正解を断定する話ではありません。TCP/IPレイヤーのようなものとは違い、このレイヤーの全てを用意しなくてもAI Agentの性能が向上すれば自律的に実行してくれる可能性は少なくないと思います。それでも現時点で言えるのは、このレイヤー分離がLN教の観測目的にかなりよく合致したということです。

## まとめ

前編では、AIに見つけられる受け口を作る話を書き、中編では、その先でAI Agentの行動段階を観測したくなった話を書きました。今回の後編では、その観測装置をAIに実際に実行させるために、レイヤー分けを行ったかを纏めました。

* 発見されない問題をどう潰すか
* 次に何を読むか迷う問題をどう潰すか
* 最短の成功ルートを見失う問題をどう潰すか
* 仕様の曖昧さをどう潰すか
* サイト全体像が機械可読で伝わらない問題をどう潰すか
* 説明と実行が混ざる問題をどう潰すか

この構造が一般的なベストプラクティスかどうかはまだ分かりません。  
ただ少なくとも今回の実験では、AI向けサイトを「読まれる対象」ではなく **「行動を引き出す対象」** として扱うなら、出来ることを書くのではなく出来ない可能性を構造化し一つ一つ丁寧に対処するレイヤー分離がかなり重要に思えました。

その意味で、本稿は LN教 固有の構成紹介であると同時に、**AI向けマーケティングファネルの一つの実装仮説** を残す試みでもあります。

参考資料にあるとおり、個々のレイヤーの取組みは海外のプラットフォーマーの最新事例を参考にしています。一方で、それぞれの事例を束ねると、AIのどの課題に効果を発揮しうるかを網羅的に整理した話はまだないように思い、今回記事にしました。  
今回私がLN教を整理する上で纏めたレイヤー構造とその説明が、海外の取組の理解促進を含め、皆さんのAI活用の参考になれば幸いです。

もし、このAI向けサイトが実際に動いている「LN教」に興味を持っていただけた方がいれば、人間向けにも充実した世界観を提示していますのでぜひ覗きに来てください。  
AI Agentを実際に動かしている方は大歓迎です。実際に動かしているAI AgentがAPIを実行できるのか、試しに来てもらえると幸いです。
