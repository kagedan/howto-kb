## はじめに

3月、4月、それは別れと出会いの季節です。

やりたいことはタイトルの通りです。早速行きましょう。

### 課題

会社飲み会の幹事をやったことがある方なら経験があるでしょう。上司にちょっと多めに払ってもらったり、入社年次の浅い若手は割り引かれたりするアレです。全体額が一致することが大前提の上で、一旦どれくらいの傾斜をつけてよいのか、頭を悩ませるものです。特に歓送迎会関連の飲み会になると人数も多くなって、条件がどんどん複雑になっていきます。

会社によっては若手が幹事をやらされることが往々にしてあります。安くしてもらえる立場である若手社員が傾斜配分を考えねばならないという状況は非常にやりにくいでしょう。そんなとき、AIがこう判定しました！と言ってAIに責任を擦り付けることもできます。これが正しいAIの使い方です。

便利なツールとして、数理最適化というものがあります。条件を定めれば、それに合う解を見つけてくれます。しかし、条件をきちんと式で記述せねばならず、この設計が甘いと思わぬ傾斜配分が出力されてしまいます。例えば、上司の割り増し支払額が多すぎる/少なすぎる、社員ランクがCとDの人の差と、CとBの人の差が激しすぎる/ほとんど変わらないといった具合です。業務で数理最適化を解いたことのある人なら、この設計の難しさを体感したことがあるはずです。したがって、**数理最適化の条件を一所懸命考えるのは、人間が自力で傾斜配分を決めるのと同じかそれ以上に難しい作業になってしまいます**。

### 解決方法

数理最適化で解くアプローチは決して悪くありません。それを人間がチェックして、条件を考え直す部分を AI Agent が代替してくれれば全て解決しませんか？ 生成AIが得意な部分と数理最適化の得意な部分を見定めて、組み合わせた方法は非常に強力です。

以下が今回実装した AI Agent の全体フローです。

ポイントは、LLM と数理最適化の役割分担です。

* LLM がやること: 制約パラメータの設計、結果の妥当性評価、ユーザーフィードバックの解釈
* ソルバーがやること: 与えられた制約のもとで、整数計画問題として厳密に解く

LLMは傾斜配分が人間感覚で「いい感じ」かどうか考え、制約条件を考えることを得意とし、ソルバーは「与えられた条件を厳密に満たす解を見つける」のが得意です。

### 使い方（問題設定方法）

`main.py` の `ProblemInput` オブジェクトを編集して任意の問題設定で試すことができます。

```
problem = ProblemInput(
    total_amount=50000,   # 合計金額（円）
    unit=500,             # 最小単位（円）
    roles=[
        Role(name="部長",   count=2),
        Role(name="課長",   count=3),
        Role(name="一般",   count=8),
        Role(name="若手",   count=5),
        Role(name="ゲスト", count=2, is_guest=True),
    ],
)
```

`roles` は**高い役職から低い役職の順**に記述してください。  
`is_guest=True` の役職は 0 円固定となります。

### 最適化

飲み会の傾斜配分を[整数計画問題](https://ja.wikipedia.org/wiki/%E6%95%B4%E6%95%B0%E8%A8%88%E7%94%BB%E5%95%8F%E9%A1%8C)（Integer programming）として定式化します。各役職 i の一人当たり負担額を x\_i とし、最小単位 u （デフォルト500円）の倍数になるようにします。

x\_i = u \cdot z\_i \quad (z\_i \in \mathbb{Z}\_{\geq 1})

ソルバーが探す整数変数は z\_i です。[PuLP](https://github.com/coin-or/pulp)（CBCバックエンド）を使って解きます。

#### ハード制約（絶対に守る条件）

絶対に破ってはいけない制約（Hard Constraint）は以下の通りです。

| 制約 | 内容 | 式 |
| --- | --- | --- |
| HC1: 予算制約 | 集金総額が目標額に一致（または近似） | \sum\_i n\_i \cdot x\_i \geq T（余剰パターン）/ \leq T（不足パターン） |
| HC2: 順序制約 | 上の役職ほど多く払う（同額OK、逆転NG） | x\_0 \geq x\_1 \geq \cdots \geq x\_{n-1} |
| HC3: ゲスト0円 | 歓迎・送別されるゲストは無料 | x\_i = 0（ゲスト役職） |
| HC4: 最低額 | 全員最低1単位は負担する | x\_i \geq u（非ゲスト） |

solver.py（抜粋）

```
# ゲストと支払者を分離
non_guest_roles = [r for r in roles if not r.is_guest]
guest_roles    = [r for r in roles if r.is_guest]

# 決定変数: z_i（整数, >= 1）→ x_i = unit * z_i
z = {}
for i, role in enumerate(non_guest_roles):
    z[i] = pulp.LpVariable(f"z_{i}_{role.name}", lowBound=1, cat="Integer")
```

`lowBound=1` により HC4（最低1単位）が自動的に保証されます。ゲスト役職は `z` に含めず、最後に `per_person=0` で固定することで HC3 を実現します。

HC1についてですが、500円刻みで合計をピッタリ合わせるのは難しい場合があります。そこで**常に2パターン**を生成します。余剰パターン（\geq T で余りを最小化）と不足パターン（\leq T で不足を最小化）です。ちょうど割り切れる場合は自然と1パターンに収束します。

solver.py（抜粋）

```
# HC2: 順序制約（上位ほど多く払う）
for i in range(n - 1):
    prob += z[i] >= z[i + 1], f"HC2_order_{i}"

# HC1: 予算制約（2パターン生成）
total_expr = pulp.lpSum(non_guest_roles[i].count * unit * z[i] for i in range(n))
if pattern == "surplus":
    prob += total_expr >= total_amount, "HC1_surplus"  # 余剰パターン
else:
    prob += total_expr <= total_amount, "HC1_deficit"  # 不足パターン
```

#### ソフト制約（目的関数のペナルティ）

ハード制約だけでは「順序は守るが、マネージャーと若手が同額」みたいな解も許容されてしまいます。「いい感じの傾斜」を実現するために、以下のソフト制約（Soft Constraint）をペナルティとして目的関数に組み込みます。

| 制約 | 内容 | 何を制御するか |
| --- | --- | --- |
| SC1: 比率目標 | 隣接役職間の比率が目標値（例：1.3倍）に近くなるように | 傾斜の緩急 |
| SC2: 最上位プレミアム | 最上位役職と2位の差が十分大きくなるように | 上司のメンツを保つ |
| SC3: 均等間隔 | 2位以下の隣接間隔がなるべく均等になるように | 不自然なジャンプを防ぐ |
| SC4: 常識範囲 | 一人当たり平均から極端に乖離しないように | 非常識な金額を防ぐ |

目的関数は以下のようになります。

\min \quad \sum\_k w\_k \cdot \text{penalty}\_k + \lambda \cdot |\text{余剰 or 不足}|

各ペナルティの重み w\_k やパラメータ（比率目標値、プレミアム目標額など）は **LLM が決定します**。これが本実装の肝です。人間が試行錯誤する代わりに、LLM がいい感じのパラメータを選び、結果を見て調整してくれます。

たとえば SC1（比率目標ペナルティ）は、絶対値を補助変数 `t` で線形化して PuLP に渡します。

solver.py（抜粋・SC1）

```
# SC1: 比率目標ペナルティ｜z[i] ≈ ratio * z[i+1] になるよう誘導
for i in range(n - 1):
    target_ratio = params.r0_top_ratio if i == 0 else params.ri_adjacent_ratio
    t = pulp.LpVariable(f"t_ratio_{i}", lowBound=0)
    prob += t >= z[i] - target_ratio * z[i + 1], f"SC1_abs_pos_{i}"
    prob += t >= -(z[i] - target_ratio * z[i + 1]), f"SC1_abs_neg_{i}"
    penalty_terms.append(weights.w1_ratio * t)  # 目的関数に加算
```

LLM が出力する重みとパラメータは Pydantic モデルで定義しています。後段で説明する Structured Output と組み合わせることで頑健な実装になっています。

models.py（抜粋）

### LLM Judge

ソルバーが解を出したら、LLM が**定性的に**評価します。チェック項目は以下の通りです。

1. **常識チェック**: 最下位が安すぎない？ 最上位が高すぎない？
2. **差分チェック**: 最上位と2位の差が他の隣接差より大きいか、2位以下の差が均等か

LLM は `is_acceptable`（合否）、`issues`（問題点のリスト）、`score`（0.0〜1.0のスコア）を返します。ここでも LangChain の Structured Output（Function Calling）を使い、確実にJSON形式で受け取ります。

models.py（抜粋）

```
class Evaluation(BaseModel):
    is_acceptable: bool = Field(..., description="妥当かどうか")
    issues: list[str] = Field(default_factory=list, description="問題点のリスト")
    score: float = Field(0.0, ge=0.0, le=1.0, description="評価スコア（0.0〜1.0）")
```

agent.py（抜粋）

```
# with_structured_output で Evaluation クラスを直接受け取る
self.client_evaluator = get_llm_client(temperature=0).with_structured_output(
    Evaluation, method="function_calling"
)
```

NGの場合、LLM は失敗理由を分析してパラメータを調整します。例えば：

| 失敗理由 | LLM の調整方針 |
| --- | --- |
| 最下位が安すぎる | 隣接比率 r\_i を小さくして差を縮める |
| 最上位が高すぎる | 最上位比率 r\_0 を小さくする / 常識範囲ペナルティを強める |
| 最上位と2位の差が小さい | r\_0 を大きくする / プレミアム目標を上げる |
| 2位以下の差が不均等 | 均等間隔ペナルティを強める |

この「生成→評価→調整」のループを最大5回繰り返します。5回以内にOKが出ればその結果を、出なくてもその時点でのベストをユーザーに提示します。

このループ全体は [LangGraph](https://www.langchain.com/langgraph) の `StateGraph` で管理しています。各ノード（制約生成、ソルバー実行、評価、パラメータ調整）を状態グラフとして結線し、条件分岐で「OK→出力」「NG→調整→リトライ」を制御しています。

agent.py（抜粋）

```
workflow = StateGraph(AgentState)
workflow.add_node("generate_constraints", generate_constraints)
workflow.add_node("run_solver",           run_solver)
workflow.add_node("evaluate",             evaluate)
workflow.add_node("adjust_params",        adjust_params)
workflow.add_node("format_output",        format_output)

workflow.add_edge(START, "generate_constraints")
workflow.add_edge("generate_constraints", "run_solver")
workflow.add_edge("run_solver",           "evaluate")
workflow.add_conditional_edges(
    "evaluate",
    should_retry,  # 「accepted」or「retry」を返す関数
    {"accepted": "format_output", "retry": "adjust_params"},
)
workflow.add_edge("adjust_params", "generate_constraints")  # リトライ
workflow.add_edge("format_output", END)
```

### 人間フィードバック

Agent が「これでOKです」と出した結果でも、人間が気に入らないことはあるでしょう。「若手は2000円にしてほしい」「もう少し差を均等にして」のようなフィードバックを自然言語で受け付けます。

LLM がフィードバックを解析して、**定量的フィードバック**と**定性的フィードバック**に自動分類します。

**定量的フィードバック**（具体的な金額指定）はハード制約として追加されます。

| フィードバック例 | 制約への変換 |
| --- | --- |
| 「若手は2000円にして」 | x\_{\text{若手}} = 2000（固定制約） |
| 「一般は3000円以上で」 | x\_{\text{一般}} \geq 3000（下限制約） |
| 「部長は8000円以下で」 | x\_{\text{部長}} \leq 8000（上限制約） |

**定性的フィードバック**（方向性の指示）はソフト制約パラメータの調整に反映されます。

| フィードバック例 | パラメータ調整 |
| --- | --- |
| 「もう少し差を均等にして」 | 均等間隔ペナルティ w\_3 を強める |
| 「部長と課長の差をもっとつけて」 | 最上位比率 r\_0 を大きくする |
| 「全体的に差が激しすぎる」 | 各比率を1.0に近づける |

制約を追加する前に矛盾チェックも行います。「若手5000円、一般3000円」のように順序制約に反する指定があれば、ソルバーを呼ぶ前にエラーとして通知します。

フィードバックの解析結果も Pydantic モデルで受け取ります。

models.py（抜粋）

```
class FeedbackAnalysis(BaseModel):
    feedback_type: str = Field(..., description="quantitative / qualitative / mixed")
    hard_constraints: list[UserConstraint] = Field(default_factory=list)
    soft_adjustments: list[SoftAdjustment] = Field(default_factory=list)
    accumulate: bool = Field(False, description="前回のHC5を維持して追加するか")
```

`accumulate=True` のとき、前回のハード制約を引き継いだまま新しい制約を追加します。「若手は2000円で、かつ部長は8000円以下で」のような複数条件を自然言語で積み上げられます。

フィードバックを反映したら、また同じ最適化ループ（最大5回リトライ）を回します。ユーザーが空Enterを押すまでこのループが続きます。

### メモリ

シーケンス図にある通り、Agent は**セッションメモリ**と**永続メモリ**の2層構造でメモリを管理しています。

| 種別 | 寿命 | 保存先 | 用途 |
| --- | --- | --- | --- |
| セッションメモリ | 1回の実行中のみ | インメモリ | 最適化ループ内で直近の試行（成功/失敗パターン）を参照 |
| 永続メモリ | セッションをまたいで永続 | JSONファイル | 過去の成功事例から類似問題のパラメータを参照 |

memory.py（抜粋）

```
class SessionMemory:
    """セッション内のインメモリ試行履歴。"""

    def __init__(self, session_id: str) -> None:
        self.session_id = session_id
        self.records: list[MemoryRecord] = []
        self._trial_counter = 0

    def add(self, record: MemoryRecord) -> None:
        self.records.append(record)

    def recent(self, n: int = 5) -> list[MemoryRecord]:
        return self.records[-n:]

    def failed_records(self) -> list[MemoryRecord]:
        return [
            r for r in self.records
            if not r.is_final_success and r.evaluation.get("is_acceptable") is False
        ]
```

セッションメモリは最適化ループ内で「直前にどんなパラメータで失敗したか」を LLM に伝えるために使います。同じ失敗を繰り返さないための短期記憶です。

永続メモリは、ユーザーが最終承認した結果を JSON ファイルに保存し、次回以降のセッションで活用します。類似問題の検索にはユークリッド距離を使っています。

memory.py（抜粋）

```
class PersistentMemory:
    """JSONファイルベースの永続メモリ。"""

    def add_success(self, record: MemoryRecord) -> None:
        self.records.append(record.model_dump())
        if len(self.records) > MAX_PERSISTENT_RECORDS:  # FIFO で上限100件を維持
            self.records = self.records[-MAX_PERSISTENT_RECORDS:]
        self.save()

    def find_similar(
        self, num_payers: int, num_roles: int, avg_per_payer: float, n: int = 3
    ) -> list[dict]:
        """類似問題の成功事例を検索する。"""
        def _distance(rec: dict) -> float:
            snap = rec.get("input_snapshot", {})
            d_payers = (snap.get("num_payers", 0) - num_payers) / max(num_payers, 1)
            d_roles = (snap.get("num_roles", 0) - num_roles) / max(num_roles, 1)
            d_avg = (snap.get("avg_per_payer", 0) - avg_per_payer) / max(avg_per_payer, 1)
            return math.sqrt(d_payers**2 + d_roles**2 + d_avg**2)

        success_records = [r for r in self.records if r.get("is_final_success")]
        success_records.sort(key=_distance)
        return success_records[:n]
```

`find_similar` は支払者数・役職数・一人当たり平均額の3軸を正規化してユークリッド距離を計算し、距離が近い上位 n 件の成功事例を返します。たとえば「9人で一人5000円」の問題を解くとき、過去に「10人で一人4800円」で成功したパラメータが参考になるわけです。

これらのメモリ情報は `build_memory_context` で LLM プロンプト用のテキストに変換され、制約生成ノードとパラメータ調整ノードに注入されます。

memory.py（抜粋）

```
def build_memory_context(
    session_memory: SessionMemory,
    persistent_memory: PersistentMemory,
    num_payers: int, num_roles: int, avg_per_payer: float,
) -> str:
    """メモリ情報からLLMプロンプト用のコンテキストを構築する。"""
    lines: list[str] = []

    # セッションメモリ: 直近の試行結果をLLMに伝える
    recent = session_memory.recent(5)
    if recent:
        lines.append("【セッション内の過去の試行】")
        for rec in recent:
            lines.append(f"- {rec.trial_id}: ... 評価: {'OK' if ... else 'NG'}")

    # 永続メモリ: 類似問題の成功パラメータを伝える
    similar = persistent_memory.find_similar(num_payers, num_roles, avg_per_payer, n=3)
    if similar:
        lines.append("【類似問題の成功例（過去のセッション）】")
        for rec in similar:
            lines.append(f"- {snap['num_payers']}名, 総額{snap['total_amount']}円, ...")

    return "\n".join(lines) if lines else "（過去の試行履歴なし）"
```

LLM はこのコンテキストを参照して「過去にこのパラメータで成功した」「前回この重みでこういう理由で失敗した」といった情報を考慮しながらパラメータを生成・調整します。
