環境構築が終わったので、次はデータベースの準備です。

Supabaseというサービスを使います。世界中の個人開発者がよく使っていて、Postgresというデータベースをブラウザから簡単に操作できるサービスです。

### まずテーブルを3つ作る

最初に作ったのはこの3つでした。

* **profiles**: ユーザー情報
* **posts**: 投稿
* **likes**: いいね

Claudeが書いてくれたSQLをコピペして、Supabaseの「SQL Editor」で実行するだけです。

sql

```
create table profiles (
  id uuid references auth.users on delete cascade,
  username text unique,
  ...
);
```

実行したら Success. No rows returned と表示されて、Table Editorを開いたら3つのテーブルが並んでいました。データの入れ物ができた瞬間です。

### 「RLS」という初耳の概念

ここで、Claudeから一つ提案がありました。

> 「次はセキュリティ設定をやりましょう。ポリシーなしだとデータが誰でも読み書きできる状態です」

RLS(Row Level Security)、初めて聞く言葉です。「RLSってなんですか?」と聞いたら、こう返ってきました。

> 「ファイアウォールのルールと同じ考え方です。通常のDBは接続できれば全データ読み書き可能、RLSを有効にするとルールに合致したデータだけアクセス可能になります」

ファイアウォール、という言葉が出てきて、そういえばそういう仕組みがあったなと思い出しました。

データベースにも「誰が何を読み書きできるか」を行(レコード)単位で制御できる仕組みがある、ということですね。言われてみれば当たり前の話でした。

### ルールを書いて閉じる

実際に設定したルールはシンプルでした。

* **profiles**: 全員が閲覧可能・自分のものだけ編集可能
* **posts**: 全員が閲覧可能・自分のものだけ作成・編集・削除可能
* **likes**: 全員が閲覧可能・自分のものだけ追加・削除可能

「読むのは全公開、書くのは本人のみ」という、SNSとして基本的なルールです。これもSQLをコピペで実行しました。コードは忘れていましたが、昔SQL文も叩いていた覚えがあるので、なんとなく読めるのが助かりました。念のためTable Editorでテーブルを開いて、ルールが反映されているか目で確認しました。

sql

```
create policy "誰でも閲覧可能" on posts
  for select using (true);

create policy "自分のみ投稿可能" on posts
  for insert with check (auth.uid() = user_id);
```

### ログイン時にprofilesを自動で作るトリガー

もう一つ、後から効いてくる仕掛けを入れました。

ユーザーがGoogleでログインした瞬間に、profilesテーブルに自動でレコードを作るトリガーです。これがないと、ログインはできてもプロフィール情報が空のまま、という状態になってしまいます。

「ログイン後にprofilesに自動でレコードを作りたい」と伝えたら、トリガー用のSQLが返ってきて、コピペで完了。正直、トリガーという仕組みも初めてちゃんと使いましたが、「ログインしたら自動で動く」という説明でイメージは掴めました。

### できたこと、と気づいたこと

この日でできたのはこれだけです。

* ✅ テーブル3つ作成
* ✅ RLSポリシー設定
* ✅ ログイン時のprofiles自動作成トリガー

地味ですが、後で振り返るとこの日の作業が「アプリの土台」になりました。
