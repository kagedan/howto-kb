実務の中で、Claude にドキュメントを読ませる方法を調べていると、APIの`document`ブロックで渡すやり方を見つけました。最初読んだ時は、RAG検索（検索して関連箇所を取ってくるやり方）と何が異なるのかをわかりませんでした😢  
そのため色々少し調べてみたのでその備忘録的な感じで記事にしました〜

まず、結論から言うと、この2つは**同じ土俵で比べるものではありません**。役割が違うので、片方しか使わない場面もあれば、両方を組み合わせる場面もあります。この記事では、その違いと使い分けを整理します

## まず、これは「対立する技術」ではない

質問に答えるまでの流れに置いてみると、2つの立ち位置がはっきりします

* **APIにドキュメントを渡す** … Claude に中身を**読ませて答えさせる**部分
* **RAG** … 大量の文書から**渡すべき箇所を探し出す**前段の部分

つまり「探す」のがRAG、「読ませる」のがドキュメント受け渡し、という分担です。

## ① APIにドキュメントを渡す

`document`ブロックに中身を入れて渡すと、Claude はそれを丸ごと読んで回答します。`citations.enabled` を `true` にすると、**回答のどの部分が文書のどこを根拠にしているか**を構造化して返してくれます

```
{
  "type": "document",
  "source": { "type": "text", "media_type": "text/plain", "data": "本文..." },
  "citations": { "enabled": true }
}
```

渡せるドキュメントは3タイプあり、引用の単位（粒度）と位置の表し方が異なります

| タイプ | 向いている用途 | 分割のされ方 | 引用位置の表し方 |
| --- | --- | --- | --- |
| プレーンテキスト | 普通の文章・散文 | 文単位に自動分割 | 文字インデックス（0始まり） |
| PDF | テキストを含むPDF | 文単位に自動分割 | ページ番号（1始まり） |
| カスタムコンテンツ | リスト・議事録など細かく制御したい場合 | 追加分割なし（渡した塊のまま） | ブロック番号（0始まり） |

ここで重要なのは、**この機能に「検索」はない**という点です。何を渡すかは自分（アプリ側）が決め、Claude は渡されたものを読むだけです

引用機能には、自前のプロンプトで引用させる方式と比べて次の利点があるとされています

* 引用テキスト（`cited_text`）は出力トークンに**カウントされない**ためコストを抑えられる
* 引用が文書への有効なポインタであることが保証され、信頼性が高い
* 評価上、関連性の高い箇所を引用する精度が高い

## ② RAG

RAGは、何千ページもあって**コンテキストに収まらない**文書群を相手にする手法です。あらかじめ文書をベクトル化（エンベディング）してデータベースに入れておき、質問が来たら関連する数チャンクだけを検索して取り出します

ざっくり言えば、RAGの本質は「何を渡すべきかを自動で選ぶ仕組み」。そのぶん、ベクトルDBや類似度検索といった追加の仕組みが必要になります

## 結局、どう使い分けるか

判断の軸は「文書がコンテキストに収まる量かどうか」です

| 状況 | 適した方法 |
| --- | --- |
| 文書が数個〜コンテキストに収まる量 | APIに直接渡す（RAGは不要） |
| 文書が膨大で全部は載せられない | RAGで絞り込んでから渡す |

そして実運用では**併用が王道**です。RAGで取り出したチャンクを、APIのドキュメントブロックに載せて引用機能を効かせます。公式ドキュメントでは、チャンクの渡し方を次のように使い分けると説明されています

* チャンク内の**特定の文**を引用させたい → プレーンテキストとして渡す（文単位で再分割される）
* **これ以上分割したくない**／自分で塊を制御したい → カスタムコンテンツとして渡す

## つまずきやすい注意点

* 引用機能と**構造化出力（Structured Outputs）は併用できません**。両方を有効にするとAPIは400エラーを返します
* `.csv` `.xlsx` `.docx` `.md` `.txt` は`document`ブロックとして直接は渡せません。プレーンテキストに変換して本文に含める必要があります
* 引用は Haiku 3 を除く現行モデルで利用できます

## まとめ

「ドキュメントを渡すAPI機能」は読ませて引用させる部分、RAGは膨大な文書から渡すべき箇所を探す部分です。文書が少なければ前者だけで足り、多ければRAGで絞ってから渡す。役割の違いがわかれば、どちらを使うかは自然と決まります
