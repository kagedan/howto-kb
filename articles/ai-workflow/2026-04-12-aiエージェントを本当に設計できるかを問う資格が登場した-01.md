2026年3月、Anthropicがひっそりと、しかし業界に衝撃を与える資格をローンチした。

その名も「Claude Certified Architect – Foundations（CCA-F）」。LLM専門の設計者向け認定資格としては業界初。60問・120分・合格ライン720/1000という本格的なプロクター付き試験だ。

「またAI系の簡単な認定資格か」と思ったなら、少し待ってほしい。

この試験、配点の構造を見た瞬間に「本気だ」とわかる。エージェント設計とClaude Code設定の2領域だけで全体の47%を占めている。つまり、AIをどう動かすかではなく、どう設計し、どう統制するかを問う試験なのだ。

## この記事でわかること

* 試験の全体構造と合格ラインの現実
* 配点47%を占める2大ドメインの核心
* 合格者が口をそろえて言う「最重要コンセプト」
* 試験に頻出するアンチパターン集
* 無料で使える学習ロードマップ

  

## 試験の基本情報

まず試験のスペックを整理しておく。

* 正式名称：Claude Certified Architect – Foundations（CCA-F）
* レベル：〜301（中上級）
* 問題数：60問（シナリオベース多肢選択）
* 制限時間：120分（一気通貫、休憩禁止）
* 合格スコア：720 / 1,000
* 受験料：$99（Claude Partner Network早期登録者5,000名は無料）
* 結果通知：2営業日以内（ドメイン別スコア付き）
* 形式：完全クローズドブック。AI利用・外部資料参照・休憩はすべて禁止

前提条件として、Claude API・Agent SDK・Claude Code・MCPの実務経験6ヶ月以上が求められる。デモを触ったことがある程度では太刀打ちできない設計だ。

## 5ドメインの配点構造

出題ドメインと配点は次のとおりだ。

* ドメイン1：Agentic Architecture & Orchestration ——— 27%（最重量）
* ドメイン2：Tool Design & MCP Integration ——— 18%
* ドメイン3：Claude Code Configuration & Workflows ——— 20%
* ドメイン4：Prompt Engineering & Structured Output ——— 20%
* ドメイン5：Context Management & Reliability ——— 15%

ドメイン1とドメイン3だけで合計47%。この2領域を制した者が試験を制する。

シナリオは6種類用意されており、そのうち4つがランダムに出題される。カスタマーサポート解決エージェント、マルチエージェントリサーチシステム、CI/CDパイプライン統合、構造化データ抽出など、いずれも現場直結の設定だ。各シナリオは複数ドメインを横断してテストする設計になっており、「ドメイン別に丸暗記」という勉強法が通じない。

## ドメイン1の核心：「動くエージェント」と「壊れないエージェント」の違い

Anthropicが繰り返し強調する設計原則がある。

> 「最もシンプルな解決策から始め、必要な時にのみ複雑さを増す」

ワークフローは事前定義されたコードパスでLLMとツールをオーケストレーションするシステム、エージェントはLLMが自律的にプロセスとツール使用を動的に制御するシステム——この区別がドメイン1の出題思想の根幹にある。

### 試験で必須の5つのワークフローパターン

1. プロンプトチェイニング 各LLM呼び出しの間にプログラム的な品質ゲートを挟む逐次処理。文書のアウトライン生成→評価→各セクション執筆のような、固定サブタスクに分解できるケースで使う。
2. ルーティング 入力を分類して専門ハンドラーに振り分けるパターン。カスタマーサポートの請求・技術・一般問い合わせを振り分ける構成が典型例だ。
3. 並列化 セクショニング（タスクを独立サブタスクに分割）とボーティング（同一タスクを複数回実行して多数決）の2サブパターンがある。
4. オーケストレーター・ワーカー 中央のLLMが動的にサブタスクを決定し、ワーカーLLMに委任して結果を統合する。並列化との決定的違いは、サブタスクが事前定義ではなくオーケストレーターによって動的に決定される点だ。
5. エバリュエーター・オプティマイザー 一方のLLMが生成し、他方が評価・フィードバックするループ。明確な評価基準が存在する場合に威力を発揮する。

エージェントループの本質は「コンテキスト収集 → アクション実行 → 結果検証 → 繰り返し」だ。Anthropicはこれを端的に「LLMがフィードバックに基づいてツールを使うループ」と定義し、「インテリジェンスはモデルに宿り、ループにはない」と強調する。

## マルチエージェントの「本当のコスト」

Anthropicの内部評価が示すデータがある。オーケストレーター・ワーカーパターン（リードエージェントにClaude Opus 4、並列サブエージェントにClaude Sonnet 4）を採用したマルチエージェント構成は、単体のClaude Opus 4を90.2%上回るパフォーマンスを発揮した。

ただしトークン消費は別の話だ。

* チャット：基準（×1）
* エージェント：約4倍
* マルチエージェントシステム：約15倍

ここで重要な知見がある。「トークン予算を2倍にするよりモデルをアップグレードする方がゲインが大きい」という事実だ。Sonnet 3.7からSonnet 4への切替がトークン倍増より効果的——これはコスト設計の意思決定に直結する。

### サブエージェントの設計思想

サブエージェントは独自のコンテキストウィンドウで動作する専門AIアシスタントで、最終結果のみが親エージェントに返される。中間的なツールコールと結果はサブエージェント内に留まる。

この設計により、数万トークンの探索を行いながら1,000〜2,000トークンの凝縮されたサマリーのみを返すという効率的なコンテキスト管理が実現する。

Claude Codeには組み込みサブエージェントが3つある。

カスタムサブエージェントはYAMLフロントマター付きMarkdownファイルで定義し、.claude/agents/（プロジェクトレベル）または~/.claude/agents/（ユーザーレベル）に配置する。

## ツール設計が、エージェントの運命を決める

Anthropicが断言する言葉がある。

> 「ツール設計に費やした時間はプロンプト設計より大きなリターンをもたらす」

Anthropicは実際、SWE-benchでの成績向上においてもプロンプトよりツールインターフェースの改善に多くの時間を投じたという。

ツール設計のベストプラクティスをまとめると次のようになる。

* ツール説明は最低3〜4文の詳細な記述
* 意味のある名前空間（github\_list\_prs、slack\_send\_message）
* 関連操作の統合（list\_users + create\_event → schedule\_event）
* 検索指向の設計（list\_contacts ではなく search\_contacts）
* 高シグナル情報のみを返すレスポンス設計

また、全エージェントに全ツールへのアクセスを与えることは典型的なアンチパターンだ。18ツールは選択精度を劣化させ、4〜5ツールが最適とされている。

## 試験合格者が「最もテストされる概念」と証言したもの

複数の合格者が口をそろえて言う最重要コンセプトがある。

「プログラム的強制 vs プロンプトベースのガイダンス」

コンプライアンスや安全性に関わる制約は、プロンプト指示ではなくAgent SDKフック（PreToolUse、PostToolUse）やプログラム的な事前条件チェックで実装すべき——という原則だ。プロンプト指示は確率的、フックは決定論的。この違いを本番環境で理解しているかどうかを、試験は徹底的に問う。

### 試験頻出のアンチパターン集

これらは「もっともらしいが本番で壊れる」選択肢として出題される。

* コンプライアンスにプロンプト指示のみを使う → フックで強制すべき
* エスカレーションルーティングに自己報告の信頼度スコアを使う → 決定論的チェックを使うべき
* 全ワークフローをBatch APIに回す → SLA保証なし
* 注意力散漫の修正に大きなコンテキストウィンドウを使う → コンパクションやサブエージェントで対応
* サブエージェント失敗時に空結果を黙って返す → エラーを明示的に伝搬すべき
* 全エージェントに全ツールを与える → 4〜5ツールが最適

経験が浅いうちは引っかかりやすい。意識して覚えておこう。

## ドメイン3の鍵：CLAUDE.mdという「AIへの業務引き継ぎ書」

CLAUDE.mdはClaude Codeのメモリだ。セッション開始時に読み込まれるプレーンMarkdownファイルで、チームの文化・コーディング規約・ビルド手順・プロジェクトの意図を書き込んでおく場所だ。

### 4段階の階層構造

* マネージドポリシー：組織全体に適用（OSの専用ディレクトリ）
* プロジェクトレベル：./CLAUDE.md（チーム共有、ソース管理経由）
* ユーザーレベル：~/.claude/CLAUDE.md（個人設定、全プロジェクト共通）
* ローカル：./CLAUDE.local.md（個人かつプロジェクト固有、.gitignore推奨）

Claude CodeはCWDからディレクトリツリーを上に辿り、見つかったすべてのCLAUDE.mdとCLAUDE.local.mdを上書きではなく連結する。プロジェクトルートのCLAUDE.mdはコンパクション後も永続するが（ディスクから再読み込み）、ネストされたものは自動再読み込みされない点に注意が必要だ。

### 効果的なCLAUDE.mdを書く5原則

1. 200行以下を目標に保つ
2. 「コードを適切にフォーマット」ではなく「2スペースインデント」のように検証可能な具体性を持たせる
3. WHAT（技術スタック）・WHY（目的）・HOW（ビルドコマンド）を含める
4. コードスニペットを埋め込むよりfile:line参照を優先する
5. リンターで対応可能なスタイル規則は重複して書かない

## フックシステム：「絶対に守らせる」仕組みの実装

フックはプロンプト指示とは根本的に異なる。プロンプトは確率的、フックは決定論的なライフサイクルトリガーで、実行が保証される。

21以上のイベントタイプをサポートし、SessionStart・PreToolUse・PostToolUse・Stop・SubagentStart/Stopなどが利用できる。フックタイプはCommand・HTTP・Prompt・Agentの4種類だ。

試験で問われやすいのが「exit code 2の二重の意味」だ。

同じコードが文脈によって逆の意味を持つ。これは試験の典型的な引っかけポイントだ。

実践パターンとしては、PostToolUseでprettierを自動実行するフォーマッター、PreToolUseでrm -rfをブロックするセーフガード、SessionStartでgit statusをコンテキスト注入する初期化処理などがある。

## スキル・CLAUDE.md・サブエージェント・フック：何が違うのか

この4つの使い分けは試験の定番出題パターンだ。一言で整理すると次のようになる。

* スキル → 特定タスクの手順（オンデマンドで読み込む）
* CLAUDE.md → プロジェクトの空気（常時有効）
* サブエージェント → 別の担当者（独立した実行環境）
* フック → 絶対に守らせるルール（決定論的トリガー）

スキルはYAML frontmatter付きのSKILL.mdファイルとして.claude/skills/に置き、Claudeがdescriptionフィールドを見て関連性を判断して自動読み込みする。

## コンテキストを「有限リソース」として扱う

Anthropicが定義するコンテキストエンジニアリングの核心は次の一文に集約される。

> 「望ましい結果の可能性を最大化する、最小限の高シグナルトークンセットを見つけること」

コンテキストロット（トークン増加に伴う情報想起精度の低下）への対処法は3つある。

1. コンパクション：コンテキスト限界近くで会話を要約して再開
2. 構造化ノートテイキング：NOTES.mdなど外部ストレージへのメモ書き
3. サブエージェントアーキテクチャ：専門サブエージェントが集中探索し、凝縮サマリーのみを返す

Claude Codeはこのハイブリッド戦略を採用している。CLAUDE.mdを事前にコンテキストに投入しつつ、grepやglobでジャストインタイムな環境ナビゲーションを行う。人間が「百科事典を全暗記するのではなく索引システムを使う」のと同じ原理だ。

## 合格するための現実的な道筋

Redditで985/1000を報告した合格者は「日常的にClaudeを使い込んでいた経験が決定的だった」と証言している。複数の合格者が「模擬試験で900以上を取ってから本試験に臨むべき」と助言する。

### 優先して取り組むべき学習コース（すべて無料）

1. Building with the Claude API（8.1時間） 最重要コース。認証・ストリーミング・ツール使用・RAG・プロンプトキャッシング・エージェントパターンを網羅する。
2. Claude Code in Action Claude Codeの設定と開発者ワークフローを実践的に扱う。ファイル操作・コマンド実行・コンテキスト管理・GitHub統合・MCP統合を含む。
3. Introduction to Model Context Protocol PythonでMCPサーバー・クライアントをゼロから構築する内容。
4. Introduction to Subagents / Introduction to Agent Skills ドメイン1・3の深掘りに不可欠な2コース。

学習期間の目安は20〜30時間。公式ドキュメント（docs.anthropic.com）の精読で90%をカバーできるという声も多い。

## この試験が問うのは「設計思想」だ

試験内容の62%は設計原則（寿命数年）、38%は実装仕様（寿命数ヶ月）をテストするという。特にClaude Code設定とMCP統合は急速に変化する領域であり、公式ドキュメントの最新版を常に確認する姿勢が不可欠だ。

試験はまだ日本語未対応で、Claude Partner Networkへの加入が前提条件となる。しかし全学習リソースが無料という条件を考えると、AIエージェント開発に関わるエンジニアにとって投資対効果の高いチャレンジと言える。

この試験が最終的に問うのは3つのことだ。

* シンプルさから始めているか
* プログラム的強制を優先しているか
* コンテキストを有限リソースとして扱っているか

これを自分の設計に落とし込めているかどうか——それが合否を分ける。
