みなさん、記事のサムネイル画像作ってますか。私は作っています。  
自分の書いた記事だけではなく、メンバーが作成した記事のサムネイル作成を請け負うこともあります。

ただ、この体制だと、メンバーが「今すぐ出したい」というタイミングにPRがボトルネックになりかねません。

とくに危惧したのが、1か月間メンバーが記事を書きまくるお祭り「note matsuri」の時です。日替わりで25記事分の記事が予定されており、私のサムネ作成がボトルネックになることは自明でした。

このままではマズイと作ったのが、サムネメーカーβ版（仮称）でした。中身は、Googleフォームに必要情報を入力し、それをトリガーにしてあらかじめテンプレートを設定したGoogleスライドを新規作成、入力者にそのGoogleスライドのURLをメールで返し、ユーザはGoogleスライドから必要な画像をpngでダウンロードできる…といった挙動です。  
  
結果は好評だったのですが、改善点もいくつかありました。ひとつめは、仕様や挙動が複雑な割にはアウトプットがあまりクオリティが高くないこと。2つ目は、Googleスライドでは使えるフォントが限られてしまうこと。3つ目は決まった位置での画像の挿入を自動でおこなうのが難しいことなどがあげられます。  
  
そこで、もっと機能を洗練させて、高いクオリティのサムネメーカーをClaudeと一緒に作りました。

ただ、つくったときはかなりClaudeにおんぶにだっこでコードを書いてもらったので、その意味や使っている技術をあまり理解しておらず…このnote では、反省もかねてコードを見直し、どんな設計で作ったのかを自分なりに学習しながら言葉にしてみました。

## 技術の概要

このツールの描画部分はブラウザに標準で組み込まれているCanvas APIで動いています。JavaScriptから図形、画像、テキストをピクセル単位で配置できる機能で、背景画像の上に著者画像を重ねてタイトルや著者名を乗せる処理をすべてここで行っています。  
  
コードはTypeScriptで書いていて、ビルドツールはViteを使いました。完成したファイルはCloudflare Pagesでホスティングしています。  
  
処理の流れを大まかにいうと、Cloudflare PagesのサーバーにアクセスするとページとともにTypeScriptをビルドした[main.js](http://main.js)やbg.jpgなどのファイルがブラウザに届きます。

![](https://assets.st-note.com/img/1777600349-3Ec78UhfJGnYMk6Cuo0Oga4W.jpg?width=1200)

「bg.jpg」に指定している画像。これがすべてのサムネのベースデザインとなる。

ブラウザが背景画像を読み込んだあと、ユーザーが著者画像をアップロードしてタイトルや著者名を入力すると、Canvas APIがそれらをキャンバスに重ね描きしてプレビューを生成します。スライダーを動かすたびにキャンバスが再描画され、調整内容がリアルタイムで反映される仕組みです。

## 実装の詳細

## 1. Cloudflare Pagesへのデプロイ

一般的に、メンバーがブラウザ上で使うにはどこかのサーバーに置いてURLでアクセスできる状態にする必要があります。その際、ファイルをサーバーに置いてインターネット経由でアクセスできるようにする作業をホスティングと呼びます。  
  
今回はサーバ側で動かすプログラムがなく、いわゆる静的ファイルで構成されているので、Cloudflare Pagesが使えます。おまけに無料です。ありがたい。  
  
CloudflareはGitHubと連携させているため、Github上のmain ブランチへのpushで自動デプロイされます。  
なので、package.json に以下を書いておくだけで、Cloudflare PagesのBuild commandに npm run build を指定すると、pushのたびにCloudflare側が自動でビルドして公開してくれます。

```
json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc && vite build"
  }
}
```

dist/ フォルダへの出力、TypeScriptのJavaScriptへの変換、bg.jpg のコピーまでこの1コマンドでやってくれるので、こちらでいじる手間も少なく事故が起こりにくいのがうれしみポイントでした。

## 2. マスクする位置の把握と指定

![](https://assets.st-note.com/img/1777600466-wKYhTf25VGsuHkM1S0vJAOta.png?width=1200)

背景画像の右側には著者画像をいれるエリアがあり、座標をコードに指定する必要があります。  
  
そこで、XYH軸の指定がだるかったので、Claudeに元画像をよませてグレーの座標を特定してもらいました。  
  
PythonコードでF5F5F5のピクセルが現れるX座標を調べてカードの左端、上端を特定してくれて、実測値は x=685, y=120, w=539, h=429 とわかりました。ついでに、Claudeにその値でマスクが動くように指示をいれて、以下を出力してもらいました。

```
const cardX = 685, cardY = 120, cardW = 540, cardH = 430
function drawAuthorImage(ctx: CanvasRenderingContext2D) {
  const cardCX = cardX + cardW / 2 + authorOffX  // カードの中心X座標
  const cardCY = cardY + cardH / 2 + authorOffY  // カードの中心Y座標ctx.save()
  roundRect(ctx, cardX, cardY, cardW, cardH, 30)  // cardXはここで使われる
  ctx.clip()
  ctx.drawImage(authorImg, cardCX - baseW/2, cardCY - baseH/2, baseW, baseH)
  ctx.restore()
}
```

## 3. 著者画像のドラッグ＆ドロップアップロード

著者画像はクリックでのファイル選択とドラッグ＆ドロップの両方に対応しています。ブラウザに標準で組み込まれているFileReader APIという機能を直接使っていて、ユーザーがファイルをドロップしたときにその中身をJavaScriptから読み取れるようにしてくれます。

```
authorUpload.addEventListener('drop', e => {
  e.preventDefault()
  const file = e.dataTransfer?.files[0]  // ドロップされたファイルを取り出す
  if (file?.type.startsWith('image/')) processAuthorImage(file)
})
```

e.dataTransfer はドラッグ＆ドロップで運ばれてきたデータの入れ物で、その中の files にドロップされたファイルの一覧が配列として入っています。  
[0] で先頭の1枚目を取り出して、画像ファイルかどうかを file.type.startsWith('image/') で確認してから処理に渡しています。

## 4. キャンバスへの描画

背景画像の読み込みが終わり、著者画像とテキストの入力が揃ったところで、Canvas APIがキャンバスへの描画を行います。render() 関数が描画のすべてを担っていて、呼ばれるたびに clearRect() でキャンバスを白紙に戻してから、背景画像、著者画像、テキストの順で重ね描きします。

```
function render() {
  ctx.clearRect(0, 0, W, H)
  ctx.drawImage(bgImg, 0, 0, W, H)   // 背景
  if (authorImg) drawAuthorImage(ctx) // 著者画像
  drawTexts(ctx)                      // テキスト
}
```

著者画像を角丸カードの外にはみ出さないようにするため、クリッピング機能を使っています。ctx.save() でCanvasの状態を保存して、roundRect() でカードの形のクリッピングパスを作ってから ctx.clip() を呼ぶと、この後に描画した内容がクリッピングパスの外側に出なくなります。著者画像を描画したあと ctx.restore() でクリッピング状態を解除します。

```
ctx.save()
roundRect(ctx, cardX, cardY, cardW, cardH, 30)
ctx.clip()
ctx.drawImage(authorImg, cardCX - baseW/2, cardCY - baseH/2, baseW, baseH)
ctx.restore()
```

テキストの描画では、Canvas APIの fillText() が自動で折り返しをしてくれないので、自前で実装しています。measureText() でテキストの幅を測りながら指定幅を超えたら次の行に送る処理を文字単位でまわしています。改行コードを split して段落単位で処理しているので、テキストエリアで改行した内容もそのまま反映されます。

```
function wrapText(ctx, text, maxW) {
  const lines = []
  for (const paragraph of text.split('\n')) {
    let line = ''
    for (const word of paragraph.split('')) {
      const test = line + word
      if (ctx.measureText(test).width > maxW && line) {
        lines.push(line)
        line = word
      } else {
        line = test
      }
    }
    if (line) lines.push(line)
  }
  return lines
}
```

## 5. スライダーによるリアルタイムプレビュー

タイトルの文字サイズ、著者画像の位置、著者名の上下左右をスライダーで調整しながらプレビューをリアルタイムで確認できます。スライダーの値が変わるたびに scheduleRender() が呼ばれ、requestAnimationFrame を通じて render() が実行されます。

```
sliders.forEach(([id, update]) => {
  document.getElementById(id)!.addEventListener('input', () => {
    update()           // スライダーの値を変数に反映
    scheduleRender()   // 次のフレームでrender()を呼ぶ
  })
})

let renderScheduled = false
function scheduleRender() {
  if (renderScheduled) return
  renderScheduled = true
  requestAnimationFrame(() => {
    renderScheduled = false
    render()  // キャンバスを再描画
  })
}
```

requestAnimationFrame はブラウザに「次に画面を描画するタイミングでこの処理を実行してください」と伝える仕組みです。  
  
スライダーを素早く動かすと input イベントが1秒間に何十回も発火します。これを素直にすべて実行させるととんでもないことになります。  
なので、renderScheduled フラグで「すでに次のフレームで描画が予約されているなら新たに予約しない」という制御をしています。  
  
スライダーをどれだけ速く動かしても1フレームに描写するのは1回だけなので負荷が少なくなります。

### 詰まったポイント：オンラインプレビューがうごかない

Cloudflare Pagesにデプロイして確認したところ、プレビューキャンバスが真っ白のまま何も表示されないという問題が起きました。  
  
Claudeに状況を伝えると、「背景画像のパスが本番環境で解決できていない可能性があります」と指摘がありました。

main.ts の中で背景画像（bg.jpg）を /bg.jpg というパスで読み込んでいました。Viteのビルド出力では public/ 以下のファイルはルートに配置されるため、このパスで問題なく動くかなと思っていたのです。  
ただ、実際に調べてみると、GitHubリポジトリには public/bg.jpg がコミットされていませんでした。

そこで原因をClaudeに聞いてみると、.gitignore の設定で画像ファイルが除外されていることがわかりました。なので、.gitignore の除外対象から bg.jpg を外してコミットし直すことに。

こうして無事本番でも正しくプレビューが表示されるようになりました。このへんの挙動は全然理解できていなかったので、Claudeがサクッと原因を特定して対応してくれたので助かりました🙏
