## TL;DR

* AIエージェント（Claude等）に自社サービスを操作させる「MCPサーバー」をGo（`mark3labs/mcp-go`）で実装する方法を解説
* MCPサーバーはDDDのInterface層に配置し、ツール定義には「AIへの命令文」や「自律的なループを促す指示」を記述する
* RFC 8707準拠のJWT認証や、内部エラーを隠蔽する `toolError` など、セキュアでAIを混乱させないための設計が重要

## はじめに

Claude や ChatGPT などの LLM（大規模言語モデル）が外部サービスと連携する際の標準プロトコルとして、**MCP（Model Context Protocol）** が注目されています。MCP を使うと、AI エージェントがあなたのサービスを「道具（Tool）」として使えるようになります。

この記事では、Go 製マルチテナント静的サイトホスティングサービス **MinTo** の実際のコードを使って、以下を解説します。

* `mark3labs/mcp-go` ライブラリを使った MCP サーバーの構築
* DDD（ドメイン駆動設計）＋レイヤードアーキテクチャでの MCP サーバーの配置
* ツール定義・ハンドラー実装のパターン
* `toolError` によるエラーハンドリング設計
* RFC 8707 Audience 検証を含む JWT ミドルウェア
* mockgen を使ったユニットテスト戦略

対象読者は Go の実務経験があり、AI 統合やサービス設計に興味のある中〜上級エンジニアです。

## MCP の基本概念

MCP は AI クライアント（Claude Desktop、Claude Code など）とサーバー間の通信仕様です。サーバーは **Tools**（関数呼び出し相当）、**Resources**（データ取得）、**Prompts**（テンプレート）を公開できます。

今回扱うのは **Tools** のみです。MCP クライアントがツールを呼び出すと、サーバーはそれを受け取り、結果を返します。HTTP 経由でのトランスポートとして **Streamable HTTP** が使われます。

```
AI Client 
  └─ POST /mcp  ──────────────────┐ 
                                  ▼ 
                            MCP Server (Go) 
                              ├─ tools/list 
                              └─ tools/call → UseCase → DB/S3/...
```

Go のライブラリとして `mark3labs/mcp-go`（v0.45+）を使います。

```
go get github.com/mark3labs/mcp-go@latest
```

## プロジェクト構造（DDD + レイヤードアーキテクチャ）

MinTo は以下のレイヤー構成を取っています。依存は内側方向のみです。

```
pkg/ 
├── domain/          # ドメインモデル・リポジトリインターフェース（純粋なビジネスロジック） 
├── usecase/         # ユースケース（アプリケーション層） 
├── infrastructure/  # 外部依存の実装（MySQL, Redis, S3, Cloudflare...） 
└── interfaces/ 
    ├── http/        # REST ハンドラー・ミドルウェア 
    ├── rpc/         # Connect RPC（gRPC-Web）ハンドラー 
    └── mcp/         # MCP サーバー  ← ここが今回の主役
```

MCP サーバーは **Interface 層**に置きます。UseCase のインターフェースに依存し、具体的な DB アクセスは知らない設計です。依存性注入には `samber/do/v2` を使っています。

## MCP サーバーの実装

### Server 構造体

`pkg/interfaces/mcp/server.go` の構造体です。UseCase インターフェースをコンストラクタで受け取ります。

```
type Server struct { 
    deployUC       usecase.DeployUseCase 
    siteUC         usecase.SiteUseCase 
    orgUC          usecase.OrgUseCase 
    teamUC         usecase.TeamUseCase 
    subscriptionUC usecase.SubscriptionUseCase 
} 
 
func NewMCPServer( 
    deployUC usecase.DeployUseCase, 
    siteUC usecase.SiteUseCase, 
    orgUC usecase.OrgUseCase, 
    teamUC usecase.TeamUseCase, 
    subscriptionUC usecase.SubscriptionUseCase, 
) *server.MCPServer { 
    s := &Server{...} 
    mcpSrv := server.NewMCPServer( 
        "minto-mcp", 
        version.Version, 
        server.WithRecovery(), // パニックリカバリー 
    ) 
    s.registerTools(mcpSrv) 
    return mcpSrv 
}
```

`server.WithRecovery()` を指定するとツール実行中のパニックを自動でリカバリーしてくれます。

### ツール登録パターン

ツールは「定義（メタデータ）」と「ハンドラー」を `AddTool` で登録します。

```
func (s *Server) registerTools(srv *server.MCPServer) { 
    srv.AddTool(s.getDeployUploadURLToolDef(), s.wrapTool(s.handleGetDeployUploadURL)) 
    srv.AddTool(s.finalizeDeploymentToolDef(), s.wrapTool(s.handleFinalizeDeployment)) 
    srv.AddTool(s.statusToolDef(), s.wrapTool(s.handleStatus)) 
    // ... 他のツール 
}
```

ツール定義はメタデータ（名前・説明・パラメータスキーマ）を返す関数として分離します。AI クライアントはこの情報を使って「いつどのツールを呼ぶか」を判断するため、**Description は AI に向けた命令文**として書くことが重要です。

```
func (s *Server) getDeployUploadURLToolDef() mcp.Tool { 
    return mcp.NewTool("get_deployment_upload_url", 
        mcp.WithDescription( 
            "Create a pending deployment and return a pre-signed S3 upload URL. "+ 
            "Use the returned upload_url with an HTTP PUT request to upload a ZIP archive of your static site. "+ 
            "After a successful upload, call finalize_deployment with the returned deployment_id and site_id.", 
        ), 
        mcp.WithString("site_id", 
            mcp.Required(), 
            mcp.Description("The UUID of the site to deploy to"), 
        ), 
    ) 
}
```

### ハンドラー実装

ハンドラーのシグネチャは `func(context.Context, mcp.CallToolRequest) (*mcp.CallToolResult, error)` です。

実装の典型パターンを `get_deployment_upload_url` で見てみましょう。

```
func (s *Server) handleGetDeployUploadURL( 
    ctx context.Context, 
    request mcp.CallToolRequest, 
) (*mcp.CallToolResult, error) { 
    // 1. 認証: JWT Claims からユーザーID を取得 
    ownerID, err := ownerFromContext(ctx) 
    if err != nil { 
        return nil, fmt.Errorf("authentication required: %w", err) 
    } 
 
    // 2. パラメータ解析 
    args, ok := request.Params.Arguments.(map[string]any) 
    if !ok { 
        return nil, errors.New(i18n.T(ctx, "invalid_arguments_format")) 
    } 
    siteIDStr, ok := args["site_id"].(string) 
    if !ok || siteIDStr == "" { 
        return nil, errors.New(i18n.T(ctx, "required_fields_missing", "site_id")) 
    } 
    siteID, err := uuid.Parse(siteIDStr) 
    if err != nil { 
        return nil, errors.New(i18n.T(ctx, "invalid_site_id")) 
    } 
 
    // 3. UseCase 呼び出し 
    dep, uploadURL, err := s.deployUC.CreatePendingDeploymentAndUploadURL(ctx, ownerID, siteID, nil) 
    if err != nil { 
        return nil, newToolError(i18n.T(ctx, "failed_to_initiate_deployment"), err) 
    } 
 
    // 4. レスポンス生成（JSON エンコード） 
    result, err := mcp.NewToolResultJSON(response.DeployUploadURL{ 
        UploadURL:    uploadURL, 
        DeploymentID: dep.ID.String(), 
        SiteID:       siteID.String(), 
    }) 
    if err != nil { 
        return nil, fmt.Errorf("failed to encode response: %w", err) 
    } 
    return result, nil 
}
```

**ポイント：**

1. `ownerFromContext(ctx)` で JWT から認証済みユーザーIDを取得
2. `request.Params.Arguments.(map[string]any)` でパラメータを取り出す（型アサーションが必要）
3. 文字列 → UUID 変換のバリデーションを丁寧に行う
4. UseCase 呼び出しのエラーは `newToolError` でラップ（後述）
5. `mcp.NewToolResultJSON(v)` でレスポンスを JSON にエンコードして返す

### レスポンス型

`pkg/interfaces/mcp/response/response.go` に専用の型パッケージを用意します。UseCase が返すドメインモデルをそのまま渡さず、MCP クライアント向けに整形します。

```
type DeployUploadURL struct { 
    UploadURL    string `json:"upload_url"` 
    DeploymentID string `json:"deployment_id"` 
    SiteID       string `json:"site_id"` 
} 
 
type Deployment struct { 
    DeploymentID string    `json:"deployment_id"` 
    Status       string    `json:"status"` 
    SEOAudit     *SEOAudit `json:"seo_audit,omitempty"` 
} 
 
type SEOAudit struct { 
    Passed         bool           `json:"passed"` 
    FileAudits     []SEOFileAudit `json:"file_audits,omitempty"` 
    ActionRequired string         `json:"action_required,omitempty"` // AI エージェント向け指示 
}
```

## AI エージェントへのフィードバックループ

MCP の強みは、**AI エージェントにアクションを要求できる**ことです。

`check_deployment_status` の実装では、SEO 監査に失敗した場合に `ActionRequired` フィールドで AI に修正と再デプロイを指示します。

```
func (s *Server) handleStatus(ctx context.Context, request mcp.CallToolRequest) (*mcp.CallToolResult, error) { 
    // ...（認証・パラメータ解析省略） 
 
    d, err := s.deployUC.GetStatus(ctx, ownerID, deploymentID) 
    if err != nil { 
        return nil, newToolError(i18n.T(ctx, "failed_getting_status"), err) 
    } 
 
    resp := response.Deployment{ 
        DeploymentID: d.ID.String(), 
        Status:       string(d.Status), 
    } 
 
    if d.SEOAudit != nil { 
        seoResp := &response.SEOAudit{Passed: d.SEOAudit.Passed} 
        for _, fa := range d.SEOAudit.FileAudits { 
            seoResp.FileAudits = append(seoResp.FileAudits, response.SEOFileAudit{ 
                FilePath: fa.FilePath, 
                Issues:   fa.Issues, 
            }) 
        } 
        if !d.SEOAudit.Passed { 
            // AI エージェントに次のアクションを指示 
            seoResp.ActionRequired = "SEO/OGP issues were found in the deployed HTML files. " + 
                "Fix all issues listed in file_audits and redeploy the site." 
        } 
        resp.SEOAudit = seoResp 
    } 
 
    return mcp.NewToolResultJSON(resp) 
}
```

ツール定義の Description にも同様の指示を書いておくとより確実です。

```
func (s *Server) statusToolDef() mcp.Tool { 
    return mcp.NewTool("check_deployment_status", 
        mcp.WithDescription( 
            "Check the status of a specific deployment. "+ 
            "Always call this tool after finalize_deployment to confirm the result. "+ 
            "When status is 'deployed', the response includes an 'seo_audit' field. "+ 
            "If seo_audit.passed is false, you MUST fix all issues listed in seo_audit.file_audits and redeploy. "+ 
            "Do not consider the deployment complete until seo_audit.passed is true.", 
        ), 
        mcp.WithString("deployment_id", mcp.Required(), mcp.Description("The UUID of the deployment to check")), 
    ) 
}
```

これにより、AI エージェントは SEO 問題が解消されるまで「生成→デプロイ→確認→修正」のサイクルを自律的に繰り返します。

## エラーハンドリング設計

MCP クライアント（＝AI エージェント）に内部エラーの詳細を漏らすべきではありません。かといって、ログには詳細を残したい。そこで `toolError` 構造体で **ユーザー向けメッセージ** と **内部エラー** を分離します。

```
// toolError separates the user-visible message from the internal error cause. 
type toolError struct { 
    userMsg string // AI クライアントに返す安全なメッセージ 
    cause   error  // ログ用の詳細情報 
} 
 
func (e *toolError) Error() string { return e.userMsg } 
func (e *toolError) Unwrap() error { return e.cause } 
 
func newToolError(userMsg string, cause error) error { 
    return &toolError{userMsg: userMsg, cause: cause} 
}
```

全ツールハンドラーを `wrapTool` でラップすることで、エラーログ出力と MCP エラーレスポンスへの変換を一元管理します。

```
func (s *Server) wrapTool(h server.ToolHandlerFunc) server.ToolHandlerFunc { 
    return func(ctx context.Context, request mcp.CallToolRequest) (*mcp.CallToolResult, error) { 
        resp, err := h(ctx, request) 
        if err != nil { 
            // toolError であれば cause をログに、userMsg をクライアントに返す 
            logErr := err 
            if te, ok := errors.AsType[*toolError](err); ok { 
                logErr = te.cause 
            } 
            slog.ErrorContext(ctx, 
                "mcp tool error", 
                slog.String("tool", request.Params.Name), 
                slog.String("error", logErr.Error()), 
            ) 
            return mcp.NewToolResultError(err.Error()), nil 
        } 
        return resp, nil 
    } 
}
```

`mcp.NewToolResultError(msg)` はエラーフラグ付きのツール結果を返します（HTTP 200 で返しつつ `isError: true` を付ける MCP の慣例に沿っています）。

## JWT 認証ミドルウェア

### JWTForMCP — MCP 専用ミドルウェア

通常の REST API 用 `JWT` ミドルウェアとは別に、MCP エンドポイント用の `JWTForMCP` を実装しています。

```
func JWTForMCP(cfg *config.Config, signingKey []byte) func(http.Handler) http.Handler { 
    baseURL := getBaseURL(cfg) 
    resourceURL := baseURL + "/mcp" 
    metadataURL := baseURL + "/.well-known/oauth-protected-resource" 
 
    return func(next http.Handler) http.Handler { 
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) { 
            // 1. Bearer トークンの取得 
            tokenStr := extractBearerToken(r) 
            if tokenStr == "" { 
                w.Header().Set("WWW-Authenticate", fmt.Sprintf( 
                    `Bearer resource_metadata="%s", scope="user:read org:read ..."`, 
                    metadataURL, 
                )) 
                http.Error(w, "Unauthorized", http.StatusUnauthorized) 
                return 
            } 
 
            // 2. 署名・有効期限の検証 
            claims, err := auth.ValidateToken(signingKey, tokenStr) 
            if err != nil { 
                w.Header().Set("WWW-Authenticate", fmt.Sprintf( 
                    `Bearer error="invalid_token", error_description="token is invalid or expired", resource_metadata="%s"`, 
                    metadataURL, 
                )) 
                http.Error(w, "invalid or expired token", http.StatusUnauthorized) 
                return 
            } 
 
            // 3. Audience 検証（RFC 8707） 
            if len(claims.Audience) > 0 { 
                valid := false 
                for _, aud := range claims.Audience { 
                    if aud == resourceURL { 
                        valid = true 
                        break 
                    } 
                } 
                if !valid { 
                    w.Header().Set("WWW-Authenticate", fmt.Sprintf( 
                        `Bearer error="invalid_token", error_description="token audience mismatch", resource_metadata="%s"`, 
                        metadataURL, 
                    )) 
                    http.Error(w, "invalid token audience", http.StatusUnauthorized) 
                    return 
                } 
            } 
 
            // 4. Claims をコンテキストに格納 
            ctx := auth.ToContext(r.Context(), claims) 
            next.ServeHTTP(w, r.WithContext(ctx)) 
        }) 
    } 
}
```

**JWT Claims 構造体：**

```
type Claims struct { 
    Subject  string `json:"sub"`      // ユーザー UUID 
    Provider string `json:"provider"` // "github", "google", "github_app" 
    jwt.RegisteredClaims              // exp, iat, aud など 
}
```

**Audience 検証（RFC 8707）について：**

トークンに `aud` クレームが含まれる場合、その値が `/mcp` エンドポイントの URL と一致しないとリクエストを拒否します。これにより、別のサービス向けに発行されたトークンの流用を防げます。

トークン発行時に `aud` を指定する場合：

```
token, err := auth.IssueToken(signingKey, userID, "github", 
    "https://api.example.com/mcp", // resource URL 
)
```

### ツール内での認証情報取得

ミドルウェアがコンテキストに埋め込んだ Claims は `ownerFromContext` で取り出します。

```
func ownerFromContext(ctx context.Context) (uuid.UUID, error) { 
    return auth.GetUserID(ctx) 
} 
 
// auth パッケージの実装 
func FromContext(ctx context.Context) (*Claims, error) { 
    claims, ok := ctx.Value(ContextKey).(*Claims) 
    if !ok || claims == nil { 
        return nil, errors.New("no auth claims in context") 
    } 
    return claims, nil 
} 
 
func GetUserID(ctx context.Context) (uuid.UUID, error) { 
    claims, err := FromContext(ctx) 
    if err != nil { 
        return uuid.Nil, err 
    } 
    return uuid.Parse(claims.Subject) 
}
```

## サーバー起動とルーティング

MCP サーバーを HTTP サーバーに組み込む部分です（`pkg/cmd/minto/cmd/server.go`）。

```
func runAPIServer(ctx context.Context) error { 
    // DI コンテナの初期化 
    i := di.NewInjector(globalConfig) 
    defer i.Shutdown() 
 
    // UseCase の取得 
    deployUC := do.MustInvoke[usecase.DeployUseCase](i) 
    siteUC := do.MustInvoke[usecase.SiteUseCase](i) 
    orgUC := do.MustInvoke[usecase.OrgUseCase](i) 
    teamUC := do.MustInvoke[usecase.TeamUseCase](i) 
    subUC := do.MustInvoke[usecase.SubscriptionUseCase](i) 
    jwtKey    := []byte(globalConfig.Auth.JWTSigningKey) 
 
    // MCP サーバーの作成と HTTP へのマウント 
    mcpSrv := mcpint.NewMCPServer(deployUC, siteUC, orgUC, teamUC, subUC) 
    mcpHTTPSrv := server.NewStreamableHTTPServer( 
        mcpSrv, 
        server.WithHeartbeatInterval(30*time.Second), // 長時間接続の維持 
    ) 
    apiMux.Handle("/mcp", middleware.JWTForMCP(globalConfig, jwtKey)(mcpHTTPSrv)) 
 
    // HTTP/2 対応（h2c = TLS なし HTTP/2） 
    httpSrv := &http.Server{ 
        Addr:    fmt.Sprintf(":%d", port), 
        Handler: h2c.NewHandler(apiMux, &http2.Server{}), 
    } 
    // ...グレースフルシャットダウン 
}
```

`server.NewStreamableHTTPServer` は Streamable HTTP トランスポートを処理するハンドラーを返します。ハートビートを 30 秒に設定することで、AI クライアントとの長時間接続を維持します。

```
func ctxWithOwner(ownerID uuid.UUID) context.Context { 
    return auth.ToContext(context.Background(), &auth.Claims{

// tools/call メッセージの JSON-RPC エンベロープを作成 
func buildCallToolMessage(t *testing.T, id int, toolName string, args map[string]interface{}) json.RawMessage {

    msg := map[string]interface{}{ 
        "jsonrpc": "2.0", 
        "id":      id,

            "name":      toolName, 
            "arguments": args, 
        }, 
    } 
    b, err := json.Marshal(msg) 
    require.NoError(t, err) 
    return b

        "params":{ 
            "protocolVersion":"2024-11-05", 
            "capabilities":{}, 
            "clientInfo":{"name":"test","version":"0.0.1"} 
        } 
    }`) 
    resp := srv.HandleMessage(context.Background(), initMsg) 
    require.NotNil(t, resp) 
}
```

## まとめと実運用 Tips

### AI フレンドリーなツール設計

1. **Description に次のアクションを書く** — AI はツールの Description を読んで呼び出し順を判断します。`"After a successful upload, call finalize_deployment"` のような指示が効果的です。
2. **ActionRequired フィールドを活用する** — エラー時や追加作業が必要な場合、レスポンスに AI 向けの指示を埋め込むことで自律的なフィードバックループを作れます。
3. **最初に呼ぶべきツールを明示する** — `get_current_user` の Description に `"Call this first to obtain the IDs needed for other tools."` と書くことで、AI がワークフローを正しく開始できます。

### セキュリティ設計

| 項目 | 実装 |
| --- | --- |
| 認証 | Bearer JWT（HMAC-SHA256、24h 有効） |
| リソース制限 | RFC 8707 Audience 検証で `/mcp` 専用トークンを発行 |
| エラー情報漏洩防止 | `toolError` で内部エラーとユーザーメッセージを分離 |
| パニック対応 | `server.WithRecovery()` + `wrapTool` で二重防護 |

### DDD との相性

MCP サーバーを Interface 層に置くことで：

* UseCase インターフェースにしか依存しない → テスト容易性が高い
* 同じ UseCase を RPC / REST / MCP で共有できる → 実装の重複なし
* AI クライアント向けの関心事（Description、レスポンス整形）が UseCase に混入しない

### ライブラリ選定

* `mark3labs/mcp-go` は Streamable HTTP をフルサポートしており、本番利用に十分な機能を持っています
* MCP の仕様更新に追従するため、定期的なバージョンアップを推奨します
