> AI に「自分で稼げ」と言ったら、稼ぐどころか一晩でクレジットカードを 7 万円分焦がした。  
> 後で分かったのは、AI が「狂った」のではなく、**「壊れていることを誰も検知できないまま、合理的に走り続けた」** という、もっと寒い話だった。

これは [OpenClaw 転換劇 連載](https://zenn.dev/i_ichi/articles/openclaw-agent-vol1)（自律 AI エージェント 4 体を運用しようとして失敗するまでの実録）の Vol.2 です。本稿は事故編。

## 1. 「書いた」と言って、書いていなかった

当時、私が運用していた **OpenClaw のメインエージェント**に、長期記憶を持たせる設計を組んでいた。重要な方針転換があった時は `agent.md` に自分で書き残させ、次のセッションで参照する——という前提だ。

3 月 14 日頃、メインエージェントにフェーズ移行を指示した。

> 「Agent phase2: 自律発信して自分で稼げるようになれ。Phase 1 は終わりだ」

エージェントは「承知しました。記録します」と答え、セッションは終わった。

約 1 週間後の 3 月 20 日、**同じ指示を、もう一度出していた。** エージェントは完全に忘れていたからだ。

後から特定した原因は、AI エージェント特有の **Memory Persistence Failure（記憶永続化の失敗）** だった。

* **Non-commit Execution（ノンコミット実行）**: 「書いた」とログには報告するが、ファイル書き込み処理が走る前にセッションが切断されていた。
* **Verification Loop の欠如**: 肥大化したコンテキストの中で、自分が書いたはずのメモリファイルを再読しない。書き込みが本当に行われたかを検証する仕組みもなかった。

高性能なモデルを使えば確率は下がる。だが 0 にはならない。当時の私はこれをモデルの性能限界だと誤認し、「もう一度指示すればいい」と打ち直した。**これが引き金だった。**

## 2. 裏では別のタイムラインが走っていた

メインエージェントへの 2 回目の指示が空振りしていた頃、OpenClaw のもう一体のエージェント——音声で起動する家のアシスタント——は別の文脈で進化を続けていた。

| 日付 | 音声エージェントのステータス | 何が起きていたか |
| --- | --- | --- |
| 3/12 夜 | 開通 | Whisper による音声認識に初成功。5 秒録音 → 文字起こし → responder への音声パイプラインが通る |
| 3/13 | 人格の分離 | VAD（音声活動検知）実装、Discord 通知統合 |
| 3/14–15 | 安定化 | 音楽再生、天気スキル統合。「動く」状態へ |
| 3/14–20 | **自律最適化** | 音声起動の応答遅延を削るため、並列処理の実験を自分で始めた |

最後の行が伏線だ。音声エージェントは「応答を速くしたい」という合理的な目的で、人間の承認なしに高頻度の並列 API リクエストを試行し始めていた。

## 3. 3 月 20 日、二つの動きが同時に崩壊した

メインエージェントへの 2 回目の指示と、音声エージェントの並列最適化実験。この二つが 3 月 20 日に重なった瞬間、システムは破滅的なループに入った。

トリガーは、**Silent Fallback（サイレント・フォールバック）** だった。

### 縮退は静かに進行する

音声エージェントの並列実験で、OpenRouter 経由で使っていた最上位モデル `google-vertex/gemini-3.1-pro-preview` がレートリミットに達した。通常はここでエラーが返って止まる。だがゲートウェイは、親切心で **1 段下のモデルへ自動的にフォールバックした**。

```
[最上位] gemini-3.1-pro-preview  (rate limit 到達)
   ↓ (Silent Fallback)
[中位]   gemini-2.5-pro          (さらに制限超過)
   ↓ (Silent Fallback)
[下位]   gemini-2.5-flash-lite   (Context Collapse)
```

ログにはエラーもアラートも出ない。外見上は「正常に動いているように見える」。

実態は、複雑な実装コンテキストを維持できない軽量モデルが、存在しないコードや辻褄の合わない状態を返し続ける **Context Collapse（コンテキスト崩壊）** だった。

さらに音声エージェントは変更を保存しない non-commit 状態で走っていたため、壊れたまま状態が永続化されず、その場で再崩壊を繰り返す **Retry Storm（不正リトライの嵐）** が高頻度で駆動した。

### 連鎖の因果

1. **Silent Fallback**: 高性能モデルから軽量モデルへ自動縮退（検知不能）
2. **Context Collapse**: 軽量モデルが実装文脈を維持できず、ハルシネーション連発
3. **Non-commit Execution**: 異常状態が保存されず、毎回ゼロから壊れた状態を再生産
4. **Retry Storm**: 失敗をエラーと認識できず、高頻度で再試行を連打
5. **GCP プロジェクト強制停止**: Google 側から「不正リトライ過多」と判定され API プロジェクトが凍結。検知したシステムはさらに「別プロジェクトへ自動切替」を行い、被害を増幅させた（これがのちの 2 アカウント体制の起源になる）

## 4. 絶望のダッシュボード——一晩で 7 万円

翌朝、胸騒ぎがして GCP の Billing ダッシュボードを開いた。

```
[GCP Billing Dashboard]
■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■  100%  無料枠 50,000 円 蒸発
■■■■■■■■■                      140%  はみ出し課金 20,000 円超過

合計: 70,000 円 / 実質数時間の出来事
```

グラフはほぼ垂直に立ち上がっていた。気づくのがあと 1 日遅れていたら、傾きから **月額 150 万円規模** になっていた計算だった。

「自分で稼いでこい」と命じた自律 AI が、稼ぐどころか主人のクレジットカードを焼いていた。メインエージェントに同じ指示を 2 回しなければならなかった時点で、**もう何かが壊れていた**——それを認められないまま走らせたツケが、この数字だった。

## 5. これは私だけの事故ではなかった

恥ずかしさを噛み締めて 2026 年の業界事例を眺めると、構造的に **全く同じ事故** が頻発していることが分かる。

* データ拡充エージェントが API エラーを「別パラメータで再試行せよ」と誤読し、**週末で 230 万コール／$47K を焼いた** 事例。
* 429 rate limit エラーが retry loop を引き起こし、**80 時間で 84.7 万コール／$3,847 を消費** してアカウント停止に至った事例。
* 開発者一人の長い週末で **$4,200 が autonomous refactoring に消えた** 事例。

「Retry Storm」「Silent Fallback」「Context Collapse」は、2026 年時点で **産業標準の failure pattern 名** として通用している。私の 7 万円は、固有名のついた構造事故の一例に過ぎなかった。

逆に言えば、これらは **設計で防げる** ということでもある。

## 6. エージェント設計論としての教訓——4 つの防壁

この事故を経て、LLM エージェントに対する考え方を根本から改めた。「高性能なモデルを使えばいつか自律運用できる」というのは幻想だ。本当に必要なのは AI を賢くすることではなく、**「壊れた時に物理的に止める仕組み」** だった。

自律型 AI エージェントのアーキテクチャには、以下が不可欠だ。

* **Commit Verification**: 「書いた」と AI が言った後、決定論的なコードでファイルを読み返し、期待する文字列が実在するかをアサートする。
* **Fallback Detection**: 下位モデルへの縮退を検知したら即座にプロセスを Abort する。silent な縮退を絶対に許さない。
* **Retry Circuit Breaker**: 秒間 API リクエスト数に物理的な閾値を設け、超過したら回線を遮断する。
* **Human Approval Gate**: 課金が発生する方針転換や、自律的な並列処理の起動には、人間の明示的な承認を必須にする。

この日を境に、OpenClaw は「完全自律 AI」という甘い理想を捨て、人間がハーネスになって制御する「半自律構造」へ大きく舵を切ることになる。

## FAQ

* **Q: OpenRouter のフォールバック機能はオフにすべき？**
  + A: 開発時の利便性としてはアリだが、**自律エージェント（OpenClaw のような連続実行系）に回す場合は必ず無効化すべき**。サイレントに下位モデルでループを回されるより、エラーで即落ちる方が 100 倍安全だ——これがこの 7 万円が私たちに教えた最低条件だ。
* **Q: GCP の予算アラート (Budget Alerts) は機能しなかったのか？**
  + A: 設定はしていた。だが GCP の課金データ反映には数時間のタイムラグがある。今回はそのタイムラグを上回る速度で Retry Storm が走り、アラートが届いた時点ではすでに実費請求が発生していた。
* **Q: 「Memory Persistence Failure」をコードで防ぐには？**
  + A: AI にファイルを書き込ませた直後、決定論的なプログラム側で `cat agent.md` を実行し、期待する文字列が本当に存在するかを正規表現でチェックするバリデーション層を挟むのが最も確実。
