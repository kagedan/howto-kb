## この記事について

LLM を使ったエージェントを作っていると、最初は「1つのエージェントにツールを持たせて全部やらせる」形から始めます。ですが規模が大きくなると、それだけでは精度・速度・コストのどこかが破綻します。

そこで効いてくるのが\*\*「複数のエージェントをどう並べて動かすか」= 実行戦略**です。この記事では、実装で使える実行戦略を **基本4つ + 応用8つ = 12パターン**に整理し、それぞれ「仕組み・使いどころ・注意点」をまとめます。最後に**選び方のフローチャート\*\*も置きました。

新しくエージェントを設計するときの「引き出し」として使ってもらえればと思います。

## 前提: なぜ「実行戦略」を意識するのか

1つのエージェントに何でもやらせると、次の問題が出ます。

* **精度**: 1回の推論で複雑な判断を全部やろうとすると質が落ちる
* **速度**: 逐次的に長い処理をすると遅い
* **コスト**: 常に高性能モデルを使うと高くつく
* **デバッグ性**: どこで失敗したか分からなくなる

実行戦略とは、これらを**役割分担・並列化・段階化**で解決するための型です。以下、単純なものから並べます。

## 基本の4戦略

### 1. simple（単一エージェント）

* **仕組み**: 1つのエージェントとして動作する。外部ツールは呼べるが、他のエージェントは管理しない。
* **使いどころ**: 小規模・単一目的のタスク。まずはここから始めるのが正解。
* **注意点**: タスクが複雑化してきたら早めに次の型へ。「なんでも1エージェント」は破綻の入り口。

### 2. dispatcher（司令塔型）

* **仕組み**: ユーザー入力を分析し、どのエージェント/ツールを使うかを選んで振り分ける。マネージャー型。
* **使いどころ**: 扱う領域が複数に分かれるとき。例えば root エージェントが、技術系・市場調査系・SNS系のサブエージェントの中から必要なものだけに依頼する。
* **注意点**: 振り分けの判断自体が難しいと、ここがボトルネックになる。ルーティングは可能な限り決定的に（ルールで書けるならLLMに任せない）。

### 3. parallel（並列型）

* **仕組み**: 複数エージェントを同時に実行し、結果が揃ったら統合して返す。
* **使いどころ**: 高速に多視点の分析が欲しいとき（同じ入力を複数の観点で同時分析する等）。
* **注意点**: 統合（マージ）の設計が肝。結果が矛盾したときにどう扱うかを決めておく。

### 4. self-critic（自己批評型）

* **仕組み**: 自分自身、あるいは批評専用エージェントが出力を検証し、改善を繰り返す。
* **使いどころ**: 高品質な最終応答が必要なとき。
* **注意点**: 批評のループは回すほど遅く・高くなる。停止条件（回数上限や十分性の基準）を必ず設ける。

## 応用の8戦略

### 5. sequential（逐次依存型）

* **仕組み**: 複数エージェントを順に呼び、前の出力を次の入力に渡す。
* **使いどころ**: 工程間に強い依存があるタスク（抽出 → 整形 → 検証 のような流れ）。
* **注意点**: 前段の誤りが後段に伝播する。段ごとに検証を挟むと安定する。

### 6. iterative refinement（反復精緻化型）

* **仕組み**: 生成 → 評価 → 修正を複数回繰り返す。self-critic の強化版。
* **使いどころ**: 一発では決まらない、磨き込みが必要な成果物（文章・設計・コード）。
* **注意点**: 「どこで止めるか」が全て。改善幅が小さくなったら打ち切る。

### 7. hierarchical（階層型）

* **仕組み**: root → 子 → 孫 と階層的に管理する。
* **使いどころ**: 大規模・複雑な問題を分割統治したいとき。
* **注意点**: 階層が深いほど文脈の受け渡しでロスが出る。各層の責務を明確に。

### 8. dynamic selection（動的選択型）

* **仕組み**: 実行中に、どのエージェントを呼ぶかをその都度判断する。
* **使いどころ**: 入力によって必要な処理が変わり、無駄な呼び出しを避けたいとき。
* **注意点**: dispatcher との違いは「実行中に動的に決める」点。柔軟な反面、挙動が読みにくくなるのでログを厚めに。

### 9. committee / ensemble（委員会型）

* **仕組み**: 複数エージェントが独立に答えを出し、投票や加重平均で最終結果を決める。
* **使いどころ**: 多視点のバランスを取りたい、単一の答えに偏らせたくないとき。
* **注意点**: 単純に n 倍のコストがかかる。票が割れたときの決定ルールを事前に決める。

### 10. fallback / cascading（段階的フォールバック型）

* **仕組み**: まず軽量エージェントに問い合わせ、十分でなければ強力なエージェントにエスカレーションする。
* **使いどころ**: コスト最適化。多くのケースは安いモデルで十分、難しいケースだけ上位モデル。
* **注意点**: 「十分かどうか」の判定が要。判定自体が高コストだと本末転倒。

### 11. budgeted / cost-aware（コスト意識型）

* **仕組み**: API コストや実行時間の制約を踏まえて、エージェント選択を調整する。
* **使いどころ**: 予算・レイテンシに上限がある本番運用。
* **注意点**: 予算を使い切ったときの挙動（打ち切り or 縮退運転）を設計しておく。

### 12. parallel with pruning（並列＋剪定型）

* **仕組み**: 複数エージェントを並列実行し、有望な結果だけを残してさらに深掘りする。
* **使いどころ**: 探索空間が広く、効率的に絞り込みたいとき（ビームサーチ的な発想）。
* **注意点**: 「有望さ」の評価関数が探索の質を決める。ここが甘いと良い枝を切ってしまう。

## どう選ぶか

まずはシンプルに始めて、困った軸（精度・速度・コスト）に応じて型を足すのが実践的です。

一覧で見るとこうなります。

| 戦略 | 特徴 | ユースケース |
| --- | --- | --- |
| simple | 単一エージェント | 小規模タスク |
| dispatcher | 必要に応じて呼び分け | 司令塔・マネージャー |
| parallel | 複数を同時実行 | 高速・多視点分析 |
| self-critic | 出力を自己評価・改善 | 高品質回答 |
| sequential | 順番に呼び出す | 依存関係の強い処理 |
| iterative refinement | 反復的に改善 | 精緻化が必要な成果物 |
| hierarchical | 階層構造で管理 | 大規模・複雑タスク |
| dynamic selection | 実行時に柔軟選択 | 無駄な呼び出しの削減 |
| committee / ensemble | 投票・合議制 | 多視点のバランス |
| fallback / cascading | 段階的に強化 | コスト最適化 |
| budgeted / cost-aware | 制約内で選択 | リソース制限下 |
| parallel + pruning | 並列実行後に絞り込み | 探索の効率化 |

## まとめ

* エージェント設計は **simple から始めて、破綻した軸に応じて型を足す**のが基本。
* 精度なら self-critic / iterative refinement、速度なら parallel、コストなら fallback / budgeted、規模なら hierarchical。
* どの型でも共通して効くのは、**停止条件・統合ルール・ルーティングの決定性**を先に決めておくこと。

これらは排他的ではなく、実際には組み合わせて使います（例: dispatcher で振り分けた先が sequential、その各段が self-critic、など）。まずは手元のエージェントがどの型に当たるかを言語化してみると、次に足すべき型が見えてきます。
