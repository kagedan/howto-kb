## はじめに

本記事は **Kiro × 設計開発シリーズ**の 4 本目です。

![](https://static.zenn.studio/user-upload/f83c28a37c73-20260512.png)

本シリーズでは、AWS の AI コーディングアシスタント「**[Kiro](https://kiro.dev/)**」を使って、自然言語の指示だけで製造業向けの設計・解析タスクを進めています。

ここまでは「AI が CAD 編集する → AI が強度検証する」という設計プロセスを扱ってきました。今回はその続きで、**FreeCADで出来上がったロボットアームを、 NVIDIA Isaac Sim / Isaac Lab に移して、立方体のピックアップを学習させる**という「動かす側」の話です。

具体的には、4096 体のロボットを 1 つの GPU 上で物理シミュレーションで並列に走らせ、PPO（Proximal Policy Optimization）という強化学習アルゴリズムで吸盤ピックアップを習得させました。学習初期のランダム動作から、最終的には以下の 2 段階の成功率を達成しています:

* **立方体を床から 10cm 以上持ち上げた時間割合（lifted 率）: 91.5%**  
  — 「つかんで持ち上げる」動作ができている
* **持ち上げた上で、空中の目標位置から 10cm 以内に保持できた時間割合: 77.9%**  
  — そのうち 8 割弱は目標位置まで運べている

> 注記：本記事では、Kiro CLI の基盤モデルとして Claude Opus 4.7 を使用しています。モデルのバージョンによって生成されるコードの品質や挙動が異なる場合があります。

## 今回の流れ

```
Step 1: CAD 編集編の STEP に吸盤 pad を追加 → URDF 変換
Step 2: AWS 公式ワークショップの CFn を使って Isaac Sim 環境をデプロイ
Step 3: Kiro が EC2 サイジングや URDF インポーター問題を自律解決
Step 4: Isaac Lab 内蔵 Lift タスクを派生させて吸盤ピックアップタスク定義
Step 5: 4096 envs 並列で PPO 訓練（約 5 時間、120,000 step）
Step 6: 訓練済みモデルで推論動画を撮影、定量評価
```

* **Step 1** は [CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) で生成した STEP を起点に、[CAE 解析編](https://zenn.dev/aws_japan/articles/a5cd987ecb59ff) と同じアルミ 6061-T6 材料パラメータを再利用します。
* **Step 2〜3** は後述の「環境構築」章で詳しく扱います。

## 環境構築 — ワークショップの CFn をベースに Kiro が調整

Isaac Sim / Isaac Lab の基本環境は、AWS 公式ワークショップ [**NVIDIA Omniverse on AWS: An Isaac Lab Deployment Lab**](https://catalog.us-east-1.prod.workshops.aws/workshops/075ce3fe-6888-4ea9-986e-5bdd1b767ef7/en-US) で配布されている CloudFormation テンプレート（`IsaacLabEnvSetupHumble.yaml`）をベースにデプロイしました。

CFn が自動構築する内容:

| リソース | 内容 |
| --- | --- |
| EC2 | 初期 g6.12xlarge（NVIDIA L4 GPU ×4）、Amazon DCV リモートデスクトップ付き |
| ソフトウェア | Isaac Sim 4.5 Docker イメージ、Isaac Lab リポジトリ、skrl |
| GPU スタック | NVIDIA ドライバ 570、Vulkan（Isaac Sim のレンダラ用） |
| ストレージ | S3 バケット、ECR レジストリ |
| 分散訓練 | AWS Batch ジョブ定義、IAM Role |

このスタックを再利用したうえで、本記事（強化学習編）の「吸盤グリッパーで立方体ピックアップを 4096 envs 並列で RL 学習させる」という目的を Kiro に伝えたところ、Kiro はデプロイ後の環境状況を見ながら**自律的に以下の調整と問題解決をしてくれました**。

### EC2 サイジングの自律エスカレーション

> **用語の補足**
>
> * **NVIDIA L4**: データセンター向け GPU の 1 種で、AI 推論や小〜中規模の訓練に向く。Isaac Sim / Isaac Lab の並列物理シミュレーションも L4 ×1 枚で 4096 env 程度まで現実的に回せる。
> * **EC2 の `g6.*xlarge` ファミリー**: L4 を搭載した AWS の GPU インスタンス群。`g6.4xlarge` は L4 ×1、`g6.12xlarge` は L4 ×4 のように、サイズが大きくなると CPU・メモリだけでなく GPU 枚数も増える。
> * **InsufficientInstanceCapacity**: 「そのインスタンスタイプは今このリージョン / AZ の物理サーバーに空きがない」という AWS のエラー。一時的なキャパシティ不足で発生し、別タイプや別 AZ を試すと解決することが多い。

CFn初期設定の g6.12xlarge（L4 ×4）は us-east-1a で `InsufficientInstanceCapacity（空きがない）` となり起動できませんでした。Kiro は次のように判断してリサイズを自動実行:

> 「4 枚 GPU は本記事の学習規模では過剰で、L4 ×1 + 多並列 env（Isaac Lab の 4096 envs は単一 GPU で回る規模）で十分。AWS は EC2 の物理サーバープールをインスタンスタイプごとに分けて管理しているため、別サイズ（L4 ×1 系の g6.16xlarge など）なら別プール扱いで空きがあるかもしれない」

結果、以下のように 4 段階でエスカレーションし、最終的に g6.16xlarge で起動に成功しました:

| 試行 | タイプ | L4 GPU 数 | vCPU / メモリ | 単価 (us-east-1) | 結果 |
| --- | --- | --- | --- | --- | --- |
| 1 | g6.4xlarge | 1 | 16 / 64GB | $1.58/h | Insufficient |
| 2 | g6.8xlarge | 1 | 32 / 128GB | $2.01/h | Insufficient |
| 3 | g6.12xlarge | 4 | 48 / 192GB | $4.60/h | Insufficient |
| 4 | **g6.16xlarge** | **1** | 64 / 256GB | **$3.40/h** | **✅ 起動** |

### Isaac Sim 4.5 の URDF インポーター欠落問題

> **用語の補足**
>
> **URDF (Unified Robot Description Format)**: ロボットの関節構造・リンクの形状・質量・慣性などを XML で記述する、ロボティクス業界で標準的に使われるファイル形式。ROS、Isaac Sim、Gazebo など多くのツールが URDF を読み込んでロボットを配置できる。今回は [CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) で作った STEP（CAD 形状）を URDF に変換して Isaac Sim に持ち込みます。

[CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) の STEP から URDF を作って Isaac Sim に読み込ませようとしたところ、CFn で展開された Isaac Sim 4.5 には **URDF インポーター拡張（`isaacsim.asset.importer.urdf`）が入っていない**ことが判明しました。Kiro は次のように調査・解決:

1. コンテナ内拡張を調査 → `exporter` しか無いことを確定
2. 本家 `nvcr.io/nvidia/isaac-sim:4.5.0` を pull して確認 → 同じく無し
3. PyPI で `pip install isaacsim-asset-importer-urdf` → 該当なし
4. 代替 `yourdfpy` はパースのみで USD 変換機能なしと判定
5. 仮説: 「Isaac Lab 公式 Docker（`nvcr.io/nvidia/isaac-lab:*`）なら入っている可能性」
6. `nvcr.io/nvidia/isaac-lab:2.0.0` を pull → **URDF インポーター拡張を確認、変換成功**

この「原因特定 → 代替候補列挙 → 検証 → 採用」を、Kiro が詰めてくれました。

## STEP → URDF 変換、そして中空化による軽量化

### まず: 吸盤 (サクション) にした理由

当初は [CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) のロボットアームに **2 本指グリッパー**（物を挟む方式）を付けて学習させようとしていました。理由は Isaac Lab の標準サンプル `Isaac-Lift-Cube-Franka-v0` が Franka Panda ロボット（2 本指）を使っているため、「同じ構成にすれば参考実装が最大限流用できる」という判断でした。

![](https://static.zenn.studio/user-upload/05e37108b699-20260512.png)

> Franka Panda ロボット（2 本指）

ところが初期の訓練で、2 本指だと「挟む強さ・指の開閉タイミング・接触摩擦」の 3 要素を同時に policy が学ばなければならず、大変でした。検討した結果、以下理由で吸盤に変更しました。

* **6 軸アーム + 吸盤は産業現場でよくある構成**（物流倉庫の段ボール移載、食品工場の惣菜詰め、半導体パネル搬送、自動車ガラス搬送など）
* フランジは ISO 9409-1 規格で、もともと「エンドエフェクタは用途で付け替える」前提の設計
* RL 学習的にも **吸盤の方が ON/OFF の 2 値で済むので学習が劇的に楽**

### 吸盤 pad の追加設計

[CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) で生成した `robot_arm_edited.step`（10 パーツ構成）に、**吸盤 pad を 1 パーツ追加**して計 11 パーツにしました。吸盤は CadQuery でパラメトリックに設計し、既存フランジの ISO 9409-1 ボルト穴に固定する形です。

**【画像】吸盤 pad を追加したロボットアーム**

> Python (numpy-stl + matplotlib) で URDF の順運動学から計算した姿勢で描画。6 軸 + 吸盤 pad の合計 11 リンク、中空化により合計質量 20.4 kg。先端のオレンジがフランジ、赤が新規追加した吸盤 pad。姿勢は訓練時の初期姿勢 (j2=0.6, j3=1.5, j5=0.9 rad)。

![](https://static.zenn.studio/user-upload/177ab6934db2-20260512.png)

### 軽量化: 中空化で 85 kg → 20 kg に

[CAE 解析編](https://zenn.dev/aws_japan/articles/a5cd987ecb59ff) で使ったロボットアームは「中身まで材料で詰まったアルミ無垢モデル」で、**合計 85 kg** もありました。実機の産業用ロボット（FANUC M-20iA 20 kg、安川 MH24 30 kg など）と比べると 3〜4 倍重く、これをそのまま Isaac Sim に持ち込んだところ、**重すぎて学習が全く進まない**という問題が発生しました。([CAE解析編で無垢モデルを選択した理由はこちら](https://zenn.dev/aws_japan/articles/a5cd987ecb59ff#:~:text=%E3%81%95%E3%82%8C%E3%81%BE%E3%81%99%E3%80%82-,%E6%9C%AC%E8%A8%98%E4%BA%8B%E3%81%AE%E4%BD%8D%E7%BD%AE%E3%81%A5%E3%81%91%EF%BC%88%E3%83%A2%E3%83%87%E3%83%AB%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6%E3%81%AE%E6%B3%A8%E8%A8%98%EF%BC%89,-%E4%BB%8A%E5%9B%9E%E8%A7%A3%E6%9E%90%E3%81%99%E3%82%8B%EF%BC%89))

Isaac Lab の公式学習事例はほぼすべて 35 kg 以下のロボット（Franka Panda 18 kg、Kinova Gen3 8 kg、Unitree G1 ヒューマノイド 35 kg など）で、85 kg は実績レンジ外でした。

そこで以下の 3 ステップを踏んで、**85kg→20kgの軽量化**をしました。

1. [CAE 解析編](https://zenn.dev/aws_japan/articles/a5cd987ecb59ff) の安全率 1,690 倍（10 kg ペイロードに対して応力余裕が 1,000 倍超）を根拠に、**肉厚 5 mm の中空構造でも安全率 100 倍は維持される**と判断
2. CadQuery のスクリプトを改造して、各リンクを中空化（`hollow_cyl`, `hollow_box` 関数を新設）
3. 慣性モーメントも中空形状で再計算

**Franka Panda (18 kg) クラスに近づき、Isaac Lab 実績圏内ど真ん中**の重量になりました。肉厚 5 mm は実機産業用アームと整合する典型値で、設計としても妥当です。これをやって初めて、RL 学習が現実的に回るようになりました。

### URDF 生成の流れ

Kiro に書いてもらった処理:

1. **CadQuery でパーツごとに STEP 出力**（11 パーツ：base, swivel, shoulder, upper\_arm, elbow, forearm, wrist\_j4/j5/j6, flange, suction\_pad）、中空化版
2. **FreeCAD Python API で各 STEP を STL に変換**（Isaac Sim は STL を読める）
3. **URDF XML を生成**。リンクごとに質量・慣性モーメントを計算

### なぜ質量・慣性モーメントが必要か

物理シミュレーションでロボットを現実的に動かすには、各パーツの「質量」と「慣性モーメント（回転のしにくさ）」が必要です。例えば上腕の重量がわからないと、シミュレータは腕を振ったときの加速度・反力を計算できず、重力や外力に対する挙動がデタラメになります。

そこで上記 3 では、[CAE 解析編](https://zenn.dev/aws_japan/articles/a5cd987ecb59ff) と同じアルミ 6061-T6 の密度 2700 kg/m³ を使い、CadQuery が中空形状から計算してくれる**パーツ体積**から、**質量 = 密度 × 体積**、**慣性モーメント = 形状と質量分布から自動算出** を URDF に書き起こしています。シリーズ 1〜3 と材料パラメータを統一することで、同じロボットが一貫した物理性質で動く形を保っています。

## Isaac Sim でタスク定義 — Isaac Lab 内蔵 Lift タスクを派生

### Isaac Sim / Isaac Lab とは

**Isaac Sim** は NVIDIA が提供するロボット向け物理シミュレータで、GPU 上で多数のロボットを並列に走らせて学習用データを量産できるのが特徴です。

**Isaac Lab** はその上で動く**ロボット学習フレームワーク**で、強化学習、模倣学習、運動学ベースの制御、demo データ収集など、ロボットの動作を学ばせるさまざまな手法に対応しています。本記事では強化学習（RL）を使いますが、他にも「人間のデモ動作を真似させる」模倣学習や「VLA（Vision-Language-Action）モデルのファインチューニング基盤」など、用途は広いです。

標準で Franka Panda ロボットを使った Lift（物を持ち上げる）、Reach（目標位置に手を伸ばす）、Push（物を押す）などのサンプルタスクが用意されています。本記事ではそのうちの `Isaac-Lift-Cube-Franka-v0`（立方体を持ち上げて目標位置まで運ぶタスク）をベースに、Franka の代わりに [CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) + 吸盤の自作ロボットアームを使った**吸盤ピックアップタスク**を新規作成しました（`Isaac-Blog4Suction-H-v0`）。

### 吸盤の物理モデル — 自作 SimpleSuction

Isaac Sim には本来 `SurfaceGripper` という吸盤用の API が用意されているのですが、**本記事のロボットでは動作に至りませんでした**（原因と撤退の経緯は後半の苦労話章を参照）。そこで Kiro に独自の吸着ロジック `SimpleSuction` を書いてもらいました。

動作を噛み砕くと以下のとおりです:

```
毎ステップ:
  1) AI (policy) は「吸盤を動かしたい位置」に加えて、
     「今この瞬間、吸盤を ON にするか OFF にするか」を出す
  2) 吸盤先端と立方体との距離 d を計算する
  3) AI が "ON" を出していて、かつ d < 5cm なら → 吸着成立とみなす (state=1)
     (遠すぎたら吸い付かない、これは実機の吸盤と同じ挙動)
  4) 吸着成立中は、立方体の位置と姿勢を吸盤先端に "物理的に貼り付いている" 形で
     毎ステップ更新する (1:1 追従)
     → 吸盤が動けば立方体もそのまま付いてくる。吸着中は "外れる" ことは起きない。
```

実機の電動吸盤ポンプ（真空吸引 ON/OFF で物を掴む / 放す）の動作をシミュレーション内で単純化したものになります。

### 観測・行動・報酬 — 強化学習の 3 要素

強化学習では、AI（policy と呼ぶニューラルネットワーク）が次の 3 つとやり取りしながら学習します:

* **観測 (observation)** = AI が **毎ステップ見る情報**。ロボットの現状とタスクの状況。
* **行動 (action)** = AI が **毎ステップ出す命令**。ロボットへの入力。
* **報酬 (reward)** = 行動の良し悪しを教える**数値**。プラスなら「良いことをした」、マイナスなら「悪いことをした」。AI はこの合計値を最大化しようと学習する。

本記事の設定は以下のとおりです:

**観測（19 個の数値）**

* 6 つの関節の角度（今どう曲がっているか）
* 6 つの関節の角速度（今どれくらいの速さで曲がっているか）
* 立方体の 3 次元位置（x, y, z）
* 吸着状態（ON=1 / OFF=−1）
* 目標位置（target）の 3 次元位置（x, y, z）

**行動（7 個の数値）**

* 6 つの関節に出す「次に向かってほしい角度」の指令
* 吸盤の ON / OFF 指令（正の値 = ON、負の値 = OFF として解釈）

**報酬（5 項目の合計）** — Isaac Lab の Franka Lift タスクの設計を参考にしています:

| 項目 | 意味 | 係数 |
| --- | --- | --- |
| reaching | 吸盤が立方体に近づくほどプラス | × 1.0 |
| grasping | 吸着成立 (state=1) しているだけでプラス | × 3.0 |
| lifting | 立方体が床から 10cm 以上持ち上がっていたらプラス | × 15.0 |
| goal tracking | 立方体が目標位置に近づくほどプラス | × 16.0 |
| curriculum penalty | 関節の動きが急すぎると**マイナス** | 学習段階で強弱切替 |

**curriculum penalty** は「罰則の強さを学習途中で変える」仕組みです。Isaac Lab 公式の Franka Lift サンプル（`franka_lift_env_cfg.py`）に倣っています。具体的には:

* 学習初期（0 〜 10,000 ステップ）: 罰則を極めて弱く（−1e−4）  
  → AI が大胆にいろんな動きを試せる（探索）
* 学習後半（10,000 ステップ以降）: 罰則を 1000 倍強く（−1e−1）  
  → 急な動きはマイナス評価なので、AI は**滑らかな動作**に収束する

これが無いと、AI は速く目標に到達するため関節を振り回す policy を学習し、結果として「振動する / カクつく」動作になります。逆に最初から罰則を強くすると、AI が動き方を探索する前に諦めて「動かないのが一番得」と学習してしまいます。**ちょうど良いタイミングで罰則を切り替える**のがポイントです。

## 4096 体の並列強化学習

### なぜ 4096 体を並列に動かすのか

強化学習は「試行錯誤の回数」が命です。1 体だけで学習させると、1 回の試行（1 episode = 10 秒）を順にこなすしかなく、十分な経験を積むのに時間がかかります。

ここで GPU の並列計算力を活かすと、**同じロボットを 4096 体、1 つの GPU 上で同時に動かす**ことができます。それぞれのロボットは独立な初期条件（立方体の位置、目標位置）でタスクに取り組むので、

* 1 ステップで 4096 通りの経験が集まる → **学習が 4096 倍速**
* 立方体位置や目標位置のバラつきが広く、一度の学習で多様な状況に対応できる policy になる
* 1 体だけだと「たまたま簡単な条件」に偏って過学習するリスクがあるが、4096 体なら平均化される

**「4,096 体」という数字は Isaac Lab の標準サンプルで事実上のデフォルトです**。中途半端な数に見えますが以下理由があります。

* `4096 = 2^12` で **GPU の並列計算アーキテクチャ（16/32/256 単位のスレッドブロック）と相性が良い** 2 の冪乗。1,000 や 10,000 のような 10 進キリ番だと、内部でパディング（余分な計算）が発生して効率が落ちます。
* L4 GPU 1 枚の VRAM（24GB）に**物理シミュ + ニューラルネットの両方が収まる現実的な上限付近**が 4,096〜8,192 envs。
* Isaac Lab 公式の Franka Lift サンプルも `--num_envs 4096` で学習するのが標準で、「4,096 envs で 15 分 〜 1 時間で Pick 動作が出る」という実績ベースでチューニングされています。

### 初期条件のバラつき（ドメイン・ランダマイゼーション）

訓練時、毎 episode（10 秒）のリセットで以下をランダムに設定します:

| 項目 | 範囲 |
| --- | --- |
| 立方体の初期位置 x | 0.80 〜 0.86 m （ロボット前方、6cm の幅） |
| 立方体の初期位置 y | −0.03 〜 0.03 m （左右 6cm の幅） |
| 立方体の初期位置 z | 床上 0.02 m 固定 |
| 目標位置 (target) x | 0.70 〜 0.90 m （20cm の幅） |
| 目標位置 (target) y | −0.05 〜 0.05 m （左右 10cm の幅） |
| 目標位置 (target) z | 0.30 〜 0.50 m （高さ 30〜50cm の幅、空中） |

4096 envs で**それぞれ独立にこの範囲からサンプル**されるため、学習 1 ステップで「立方体のいろんな位置 × 目標のいろんな位置」の組み合わせを同時に経験できます。

### 訓練設定

| 項目 | 値 |
| --- | --- |
| 並列環境数 | 4096 envs |
| 総ステップ数 | 120,240 step (4,096 × 120,240 ≈ 5 億回の試行経験) |
| PPO ライブラリ | skrl（Isaac Lab 標準） |
| 1 episode 長 | 10 秒 (600 step) |
| 訓練時間 | 約 5 時間（g6.16xlarge、L4 ×1） |
| コスト | 1 回訓練あたり約 $17 |

### 学習の進行 — 動画でデバッグしながら進める

訓練中の step 0 / 30,000 / 60,000 / 90,000 / 120,000 の時点で、それぞれ 4096 envs のうち 1 env の動画を自動で記録するように Kiro に指示しました。

なぜ動画が必要かというと、**学習時に取れる数値ログ（reward の平均、loss の値など）だけでは、ロボットが実際にどう動いているかが分からない**からです。reward が順調に上がっていても、動画を見ると「腕が変な方向に振り回されているだけ」「立方体が地面に埋まっている」といった異常はよくあります。**人間や Kiro が動画を目で見て、挙動を直接確認しながらデバッグする**のは RL 開発で欠かせません。

学習開始直後 (Step 0)と学習完了時(Step 120,000)の比較動画は、本記事冒頭のYoutube動画をご参照ください。  
最初のデタラメな動きから、学習完了時にはほとんどのロボットでピックアップ成功していることがわかると思います。  
ただしピックアップ成功後にいくつかのロボットでは小刻みな振動が発生しています。この振動の正体は後半の苦労話章で掘り下げます。

### 推論定量結果

最終的な挙動の詳細評価は、訓練済みモデル（best agent = step 72,000 時点の checkpoint）を使って、**カメラを追従させながら 4 env を 300 ステップ動かした推論**で計測しました:

![](https://static.zenn.studio/user-upload/c88ca8222c82-20260512.png)

| 指標 | 値 | 意味 |
| --- | --- | --- |
| **lifted 率**（cube\_z > 10cm） | **91.5%** | 立方体を床から 10cm 以上持ち上げた step の割合 |
| **target 10cm 以内到達率** | **77.9%** | 立方体を target 座標の 10cm 圏内に保持した step の割合 |
| **吸着率**（吸盤 state=1） | 92.2% | 吸盤が立方体に吸着していた step の割合 |
| **最小距離 d\_cube\_target** | 2.2 cm | 立方体 → target 最接近距離 |
| **平均 pick up 高さ** | 45.2 cm | 床からの立方体平均高度 |

> **「4 env で評価」としている理由**: 学習がうまくいったかどうかの **集約的な判定** は、訓練中に 4096 envs 全体の「平均 reward」「平均エピソード長」を Isaac Lab が自動で TensorBoard に出してくれるので、そちらで見ます（本記事では到達するべき最終 reward まで到達したことを事前に確認済みです）。一方、**挙動の細部を目視 + 数値で細かく見る** 用途には env 数が多いと動画も見切れないので、代表として 4 env × 300 ステップ（= 5 秒分 × 4 体 = 20 秒相当のデータ）を取ります。これは Isaac Lab 公式サンプルの評価手順と同じ流れです。

> **「10cm 以内」という精度の意味**: 産業用ロボットの位置決め精度は一般に ±0.05〜±0.5 mm 程度で、本記事の 10cm は実機実運用の目線ではかなり粗い指標です。ただし **強化学習の研究・教科書的な pick & place タスク**では 5〜10cm の到達率が標準的な評価指標で、Isaac Lab 公式の Franka Lift サンプルも同じしきい値です。「RL で pick & place 動作を習得できたか」の確認には十分、というのが業界感覚のようです。実機に載せる段階では Sim2Real 転送 + ビジュアルサーボ等で精度を 1 桁以上引き上げるのが通常の流れです。

## 強化学習は一発で決まらなかった — 15 回の試行錯誤

吸盤ピックアップを成功させるまでに 15 バージョンの試行錯誤がありました。ハイライトを紹介します。

### 大きく躓いたポイント

**① Isaac Sim 本家の吸盤 API (SurfaceGripper) が動かなかった**

Isaac Sim には本来 `SurfaceGripper` という吸盤用の API が用意されていて、これを使えば吸盤挙動を実装できるはず…だったのですが、本記事のロボットでは機能させられませんでした。

SurfaceGripper の内部仕様は、**吸盤から真下方向に「光線（ray）」を飛ばして、一定距離内に物体があれば吸着する**という「**raycast 方式**」を採っています（実機の吸盤センサが真空圧で検知するのと近い考え方を、シミュレータ上で光線の当たり判定で代用している形）。ところが本記事のロボットでは、raycast が立方体にヒットしない、あるいはヒットしても吸着用の拘束（`D6Joint`）を動的に生成する処理で失敗する、という挙動になりました。

USD 上の物理設定、joint のドライブ、ClearanceOffset（吸盤と物体の許容距離）、吸盤の向き（Z+ と Z− のどちらを向いているか）、ボディ質量、14 項目以上を Kiro が調査して改修しましたが、**根本原因は最後まで特定できず**に終わりました。大きな要因としては、

* Franka Panda など公式サンプルのロボット形状と違って、本記事のカスタムロボット + カスタム吸盤形状の組み合わせが API の想定する幾何関係から外れていた可能性
* Isaac Sim 4.5 の SurfaceGripper 拡張の内部実装がバージョン間で変化しており、ドキュメントに載らない暗黙の前提条件があった可能性

が考えられます。9 バージョンで撤退し、自作の `SimpleSuction`（距離判定だけで吸着を成立させる単純な方式、前述）に方針転換しました。

**② 吸盤は成立したが立方体が運べない**

SimpleSuction で吸着（state=1）は 100% 達成したものの、立方体が target 位置に運ばれない。動画では腕が静止したまま吸着を維持しているだけ、という挙動でした。

Kiro が reward の内訳を数値分析した結果、原因は**初期の lifting reward の設計が素朴すぎた**ことでした。当時の設計は「立方体が床から 10cm 以上持ち上がっていたら +15、未満なら 0」という**ON/OFF（step function）型**の報酬で、一度持ち上げてしまうと AI にとって「下げる」動機が全く無い構造になっていました。上げた状態を維持しているだけで +15 ずつ入り続けるので、AI が学習する最適解は「**立方体を高く掲げたまま腕を動かさない**」になります。

> **なぜこの素朴な設計にしたのか**: この step function 型の lifting reward は、先に紹介した Isaac Lab 公式の Franka Lift サンプル（`franka_lift_env_cfg.py`）の標準設計をそのまま移植したものでした。Franka Lift では **2 本指グリッパーが物理的に立方体を挟んで保持**するため、運搬中に指が開いて立方体が落ちるリスクがあり、自然と「運ぶための動き」が policy に要求されます。ところが本記事の SimpleSuction は「吸着中は立方体が吸盤に 1:1 で貼り付く」物理モデルで、**立方体が絶対に落ちない**ので、「上げっぱなしで動かない」が楽な最適解として成立してしまいました。**同じ報酬設計でも、物理モデルの違いで成立する最適解がひっくり返る**、という物理 × 報酬設計の相互作用の教訓でした。

本来は「target 位置まで運べ」という報酬を、「上げっぱなしの報酬」より十分大きくする必要があるのですが、初期設計ではこのバランスが完全に逆で、**上げっぱなしのほうが稼げる**状態になっていました。RL の報酬設計は「**AI が見つけうる最悪のショートカットは何か**」を想像しながら組むべし、という教科書的な学びです。

### 学習完了後も振動が一部ロボットで残った理由

これは AI や物理シミュ特有のいくつかの要素が**閉じた輪を作る**ことで生じる現象で、以下のような流れです。

1. **AI の命令に、毎ステップ少しだけランダムな揺れが残っている**（強化学習では「全部振り切る」と学習できないので、policy が確率的に命令を出すのが普通）。学習後半でも完全にはこの揺れはゼロにならず、残っている。
2. 本記事では **「前のステップと比べて急に命令が変わったら罰則」** というルール（action rate penalty）を入れている。ただしこれは 1 ステップ前との差分しか見ていない。つまり「+0.05 → −0.05 → +0.05 → −0.05」のように **2 ステップで元に戻るような交互パターンは罰則をすり抜ける**。AI は最適解としてこの交互パターンに収束する。
3. 関節レベルでのこの小さな交互動作が、振り子の先端のように **アーム先端で拡大**して数 cm 幅の揺れになる。
4. 本記事の吸盤は立方体に **そのまま 1:1 で貼り付く** 仕様なので、アーム先端が揺れれば立方体も同じだけ揺れる。
5. 立方体位置は報酬計算に使われ、特に目標位置の 5 cm 以内で急激に値が変わる「精密追従」項目がある。そのため 1 cm 刻みで立方体が揺れると報酬もガタガタ変動し、**AI はその変動に反応してさらに命令を揺らす**、という**自己強化ループ**ができる。
6. この全体を抑える**唯一のブレーキ**は、「関節の動き自体が大きいとマイナス報酬」（joint velocity penalty）という仕組み。これがきちんと効いていると振動は抑えられるが、弱めると抑えきれない。

### 振動しているのに「タスク成立」と判断した理由

本記事の評価指標は「立方体が目標位置の 10cm 以内にいる時間割合」で、これが 77.9% でした (RL 研究における標準的な合格ラインは概ね 70-80%)。  
つまり AI は **「カクカクしながらも、ほとんどの時間、目標位置の近くに立方体を保持できている」** 状態です。

振動の性質を数字で見ると、

* 振幅: 1 ステップあたり **2.7 cm** (目標位置の許容範囲 10cm の 1/4 程度、つまり振動しても許容範囲内に収まる)
* 反転率: **63%** (プラス方向とマイナス方向にほぼ半々で行き来している)
* その結果、1 秒あたりの**実質的な移動距離はわずか 7 mm** (振動が行ったり来たりして打ち消し合うため)

例えるなら、手品師が手のひらの上でコップを載せて歩いている状態に近く、**手は細かく揺れているけれど、コップは落ちないし、ほぼその場所に留まっているような状態です**。

### 振動を完全に消せばもっと良いのでは?

その通りですが、そこまで至りませんでした。**振動を抑えるブレーキ (joint\_vel penalty) を強くしすぎると**、AI が「じっとしていた方が罰則を受けないから得」と判断してしまい、低い目標位置の env では **立方体を動かさない policy に学習が収束**してしまいます (いわゆる局所最適解)。結果、到達率が下がります。

**「振動しない × 動かす」の両立がこのタスクの難しさの核心**で、本記事の設定はそのバランスを取った結果 「許容範囲内の振動 + 合格水準の到達率」 で落ち着いた、というのが実情です。より高精度を狙うなら、報酬設計をさらに精査したり、policy のアーキテクチャを変えたりする余地があります。

## Kiroへの任せ方について

筆者はロボットの強化学習に詳しい人間ではありません。でも Kiro ならば Isaac Lab のドキュメントや既存実装を調べて、手順を立てて、コードを書いて、実行して、結果を見て次の手を提案するまで一通りやってくれます。  
だから最初の方は、**「任せておけばうまくやってくれるだろう」と手抜きして**、Kiro の提案に乗っかる形で試行を回していました。訓練が失敗しても Kiro が「原因はここ、次はこう直します」と言うので、その通りに再訓練を回す、という流れです。

ところが、だんだん違和感が出てきます。

* 「今回こそ解決」と言われて訓練を走らせてもまた失敗する
* 原因の説明が毎回違う、あるいは前回言っていた仮説と矛盾している
* パラメータを 3 つ一度に変えた版で失敗して、「何が効いたのか」が切り分け不能

普通に考えれば「実験は 1 変数ずつ変えろ」は基本です。私もそれは頭では分かっていたはずなのですが、「片手間試行だし楽したい」「細かい制御パラメータの話は Kiro の方が詳しいはず」と**自分の判断を預けてしまっていました**。

途中から**痺れを切らして、Kiro に状況を根掘り葉掘り問い詰める**ようになります。「その変更の根拠は? 数字出せ」「ログのどこにその証拠がある?」「訓練を回す前に既存 checkpoint で確認できる方法があるだろ」。  
やり取りの中で私自身もだんだん Isaac Lab の仕組み、PPO の振る舞い、報酬設計のセオリーを少し理解していき（まだ全然不十分ですが）、そこから **「これはこうするべきじゃないのか」** と方向性を自分で提案するようになりました。

* **浅い推測で「原因はこれ」と断定するな、実測データで切り分けろ**  
  — 「多分 X が原因だと思う」で方針を決めて走らせて、外してまた仮説を立てる、の繰り返しを避ける
* **重すぎる / 大きすぎるモデルは RL が回らない可能性、軽量化を早めに検討**  
  — 85 kg で粘らず、中空化で 20 kg にしたら学習が急に通った。
* **訓練を回す前に、今あるもの (既存 checkpoint、ログ、TensorBoard) でどこまで分かるかを先に詰めろ**  
  — 1 回の訓練は 5 時間 / $17。それを回すより先に、**0 コストで検証できる方法**が残っていないかを必ず確認する

Kiro は計算・コーディング・調査のスピードはとても速いですが、適当に何回も非合理な試行を繰り返すことが多々あります。  
当たり前ですが、**今回のレベルの試行を現状の性能のAI に全部任せて良い訳ではなく、大枠の方針や「そもそもそれで合っているか」を人間が問い詰めないと**回り道になってしまいます。

## まとめ

本記事では、以下を実証しました:

* **[CAD 編集編](https://zenn.dev/aws_japan/articles/45fb5d3130035d) の STEP + 吸盤 pad を URDF 化し、Isaac Sim 4.5 + Isaac Lab で物理シミュレーション化**

* **振動現象を理論的に分析**（PPO ガウシアンノイズ × action\_rate penalty の 2-step 合法化）、物理的には target 保持に影響なしと判断

本シリーズを通じて、**設計 → CAD 編集 → CAE → 強化学習** という製造業エンジニアリングの大きなワークフローが、Kiro への自然言語指示だけで最初から最後まで回せることを示しました。
