2026年6月9日、Mastra は **Code Mode for Agents** を発表しました。ひとことで言えば、**複数ツールをまたぐ分析や集計を、ツール呼び出しの往復ではなく「その場で生成した TypeScript 関数」1本に閉じ込める仕組み**です。

今回の発表は一見すると「ツール呼び出しを少し賢くした」程度に見えます。ですが実際には、エージェント実装で頻繁にぶつかる 3 つの壁にかなり正面から手を入れています。

1. 複数ツールをまたぐ問い合わせで、agentic loop が何度も回って遅くなること
2. 各ターンのツール出力がそのままコンテキストに積まれ、推論精度とコストを悪化させること
3. 集計や比較のたびに専用ツールを増やすと、アプリ側の API 設計が分析クエリに引きずられていくこと

Code Mode は、この 3 つをまとめて緩和します。しかも単なる「コード実行」ではなく、**既存ツールはそのまま活かしつつ、オーケストレーションだけをコード化する**設計になっています。ここが重要です。

---

## 今回の発表をひとことで言うと

Code Mode は、Mastra の `createCodeMode()` が返す専用ツールです。モデルはそのツールを使うと、ユーザーの問い合わせに合わせた TypeScript 関数をその場で書き、許可された既存ツールを `external_*` 関数として呼び出しながら、必要な集計や変換を行って 1 回で結果を返します。

通常の tool calling では、モデルは「どのツールを呼ぶか」を 1 回ずつ決め、結果を読み、次のツールを決める、というループを繰り返します。Code Mode では、そのループの一部をあらかじめ関数の中へ押し込みます。たとえば「売上上位 5 商品を取り、その各商品の平均評価を出す」といった処理なら、モデルは先に段取りを決めてからコードを書き、そのコードが複数ツールをまとめて呼びます。

このとき Mastra の設計がうまいのは、**モデルが書くコードはサンドボックスで動くが、実際のツール本体はホスト側でそのまま動く**ことです。つまり、ツールの入力バリデーション、request context、trace、既存の権限設計は維持されます。モデルが直接なんでも触るわけではありません。

---

### 問題は「ツールが使えること」ではなく「何ターンで答えにたどり着くか」

2024年ごろまでのエージェント実装では、「LLM が必要に応じてツールを呼べる」だけで十分に新しかったはずです。ですが、2026年の実務ではそれだけではかなり足りません。理由は、ユーザーが求めるのが単発の事実確認ではなく、**複数のデータ源を横断した分析・比較・集計**になってきたからです。

たとえば EC の運用エージェントに次の質問を投げる場面を考えます。

* 売上上位 5 商品を教えて
* その 5 件について各商品の平均評価も出して
* 返品率が高い順にも並べ替えて
* VIP 顧客の購入比率が高いものには印を付けて

この問いに対して、素朴な tool calling では次のような往復が起こります。

1. まず売上上位商品を取る
2. その結果をコンテキストへ入れる
3. 各商品について評価ツールを呼ぶ
4. 各結果をまたコンテキストへ入れる
5. 必要なら返品データを追加で取る
6. 最後に LLM が文章としてまとめる

この方式は正しく動くことも多いですが、構造的に弱点があります。

* **ラウンドトリップが増える**: ツールをまたぐたびにモデルの再判断が必要になる
* **コンテキストが膨らむ**: 生のツール結果が毎回まるごと流入する
* **算術が不安定になる**: 合計、平均、比率、フィルタ条件の適用をトークン予測に頼りやすい
* **分析クエリに API が引っ張られる**: よくある問い合わせごとに専用の集計エンドポイントを作りたくなる

Code Mode は、このうち特に「分析の途中で必要になる reduce / aggregate / filter / branch を、会話ターンではなくコードの中で済ませる」点が効きます。平均や並び替えやグルーピングは、LLM に文章として頑張らせるより、JavaScript として走らせたほうが当然安定するからです。

### 「専用集計ツールを増やせばよい」ではない理由

もちろん従来でも、`getTopProductsWithAverageRating` のような専用ツールを作れば似たことはできました。ですが、それをやりすぎると今度はアプリ側の API が「将来ユーザーが聞きそうな質問」を先回りした塊だらけになります。

本来、ツール設計は次のような粒度であるほうが再利用しやすいはずです。

* `getOrders`
* `getCustomer`
* `getProduct`
* `getProductRatings`
* `getReturns`

Code Mode の狙いは、こうした**意味の小さい、焦点の絞られたツール群**を残したまま、問い合わせごとの組み立てをモデル生成コードへ寄せることです。これは API 設計の責務を崩さずに、分析の自由度だけを上げるアプローチだと言えます。

## Code Mode はどう動くのか

基本形はかなりシンプルです。`createCodeMode()` に、呼び出しを許可したいツール群と sandbox を渡します。すると Mastra は 2 つのものを返します。

* 実行用の `tool`
* そのツールの使い方をモデルへ教える `instructions`

最小例は次のようになります。

```
import { Agent } from "@mastra/core/agent";
import { createCodeMode } from "@mastra/core/tools";
import { LocalSandbox } from "@mastra/core/workspace";

import { getOrders } from "../tools/get-orders";
import { getCustomer } from "../tools/get-customer";
import { getProduct } from "../tools/get-product";

const { tool, instructions } = createCodeMode({
  tools: { getOrders, getCustomer, getProduct },
  sandbox: new LocalSandbox(),
});

export const codeModeAgent = new Agent({
  id: "code-mode-agent",
  name: "Code Mode Agent",
  instructions: [
    "あなたは EC ストア向けの分析アシスタントです。必要なツールを組み合わせて、計算結果を正確に返してください。",
    instructions,
  ],
  model: "anthropic/claude-sonnet-4-5",
  tools: {
    execute_typescript: tool,
  },
});
```

ここでのポイントは 3 つあります。

### 1. デフォルトでは `execute_typescript` という 1 つのツールになる

`id` を指定しない場合、生成される Code Mode ツール名は `execute_typescript` です。つまりエージェントから見ると、「必要なら TypeScript でオーケストレーションしてよい」という新しい高レベルツールが 1 個増えるイメージです。

### 2. 許可されたツールだけが `external_*` として見える

Code Mode の instructions には、指定したツールごとに型付きの `external_<toolId>` 宣言が埋め込まれます。モデルが書けるコードは、その allow-list の範囲に限定されます。何でも自由に呼べるわけではありません。

### 3. サンドボックスは必須だが、ツール本体はホスト側で実行される

これは誤解しやすいところです。Code Mode は「エージェントに shell 権限を与える機能」そのものではありません。サンドボックスで動くのは、**モデルがその場で生成したオーケストレーションコード**です。一方で `external_getOrders()` のような実際のツール呼び出しは、ホスト側のツール実装へブリッジされます。

そのため、既存ツールに載っている Zod バリデーションや認可、request context、tracing がそのまま生きます。これは「コードを実行させたい」と「既存ツールの安全性や監査性は壊したくない」を両立させる設計です。

---

## なぜサンドボックスが必要なのか

Mastra の docs はここをかなり明確に書いています。Code Mode は、**モデルが書いたコードを実行する機能**です。であれば、実行境界を曖昧にしてはいけません。だから sandbox が要求されます。

ローカル開発なら `new LocalSandbox()` を渡せます。ただしこれはホストの `node` プロセスとして動くため、権限もホスト相当です。ドキュメントも「trusted or local development only」と釘を刺しています。本番では E2B、Modal、Daytona などのマネージドサンドボックスが推奨されます。

この前提は、2026年春から Mastra が強く押し出してきた Workspaces / Remote Sandboxes の流れともつながっています。Mastra はこの数か月、エージェントに「ファイルへ触る環境」「コマンドを動かす環境」「LSP でコードを解釈する環境」を与える仕組みを一気に整備してきました。Code Mode はその延長で、**モデル生成コードをどこで走らせるかを明示的に管理する**機能だと理解すると位置づけが見えます。

## 何がうれしいのか: before / after で見る

公式 docs で示されている代表例はかなり分かりやすいです。たとえば「売上上位 5 商品と、それぞれの平均評価を出して」という問い合わせでは、Code Mode は概ね次のような処理を書きます。

```
const top = await external_getTopProducts({ limit: 5 });

const ratings = await Promise.all(
  top.products.map((product) =>
    external_getProductRatings({ productId: product.id }),
  ),
);

return top.products.map((product, index) => {
  const scores = ratings[index].ratings.map((rating) => rating.score);
  const average = scores.reduce((sum, score) => sum + score, 0) / scores.length;

  return {
    name: product.name,
    sales: product.totalSales,
    averageRating: Math.round(average * 100) / 100,
  };
});
```

この例の本質は、単にコードが書けることではありません。

* 商品ごとの評価取得を `Promise.all` で並列化できる
* 平均値計算を JavaScript に任せられる
* エージェント本体には集約済みの結果だけを戻せる
* 途中の巨大なツール出力を何ターンも会話履歴へ積まなくて済む

つまり Code Mode は「コード実行ツール」ではなく、**多段ツール実行の中間結果を現場で消化してから、エージェントへ小さく返すためのバッファ層**として効きます。

## 既存手法との位置づけ

Code Mode を理解しやすくするために、近いものと比べます。

| 手法 | 何をするか | 強み | 弱み |
| --- | --- | --- | --- |
| 通常の tool calling | モデルが 1 手ずつツール選択する | 単純で分かりやすい | 多段分析で往復とコンテキスト膨張が起きやすい |
| 専用の集計ツールを大量に作る | よくある分析を API 化する | 実行結果は安定しやすい | 問い合わせごとにツールが肥大化しやすい |
| 汎用 Code Interpreter | その場でコードを書いて計算する | 計算や変換に強い | 既存アプリの業務ツールや認可と統合しづらい |
| Mastra Code Mode | 既存ツールを `external_*` としてコードから合成する | 既存ツール資産を活かしつつ計算をコード化できる | sandbox 設計と allow-list 設計が必要 |

ここで特に大事なのは、Code Mode が「汎用の Python notebook 的な実験環境」ではないことです。Mastra の中で定義済みのツールを、**型付き・制限付きで、その都度必要なだけ組み合わせる**ための機能です。Code Interpreter 的な自由さを、プロダクトのツール境界の中へうまく閉じ込めた設計だとも言えます。

## 実務で効くユースケース

Code Mode が強いのは、次のように「複数ツール + 集計 + 条件分岐」が同時に必要なケースです。

### 1. BI ほどではないが、1 問ごとに切り口が変わる分析質問

たとえば社内アシスタントに対する次のような問いです。

* 今月の失注案件のうち、金額上位 10 件を担当者別に並べて
* その担当者の過去 90 日の返信速度も横に出して
* ただしエンタープライズ顧客だけに絞って

これを全部専用エンドポイントで持つのはつらいですが、焦点の小さいツール群なら持てます。Code Mode はその間をつなぐのに向いています。

### 2. コーディングエージェントの集約処理

Workspace や LSP、検索結果など複数の情報源からデータを集めるとき、中間結果を丸ごと会話へ戻すより、その場で整理してから返したほうがよいことが多いです。たとえば「このリポジトリで auth に関係する実装を洗い出し、未使用コードだけ別枠でまとめる」といった処理は Code Mode と相性がよいはずです。

### 3. 数値の正確性が重要な問い合わせ

合計、平均、中央値、割合、ランキング、重み付け、閾値判定などは、LLM の自然言語推論だけに任せないほうが安定します。Code Mode は、この種の処理を「LLM が考えた計画」に基づきつつ、実際の計算はコードとして実行させる中間解です。

## 設計上の注意点

便利そうに見えて、何でも Code Mode に流せばよいわけではありません。docs を読むと、うまく使うコツもかなり明確です。

### 1. ツールは小さく、責務を絞る

Mastra 自身が推奨している通り、Code Mode は「それぞれが 1 つの仕事をうまくやる focused tools」と相性がよいです。もともと巨大な万能ツールを渡してしまうと、Code Mode の恩恵は薄れます。

### 2. 並列化できる問い合わせで特に効く

`Promise.all` で複数呼び出しをまとめられるケースは分かりやすく効きます。逆に、厳密に逐次依存するフローだけなら通常の tool calling でも十分なことがあります。

### 3. allow-list の切り方が重要

`createCodeMode()` は複数回呼べます。つまり、エージェントに 1 個だけではなく、役割ごとに別々の Code Mode ツールを持たせることもできます。たとえば営業分析用と在庫分析用で別の tool subset を渡せば、least privilege を保ちながら prompt surface も小さくできます。

### 4. LocalSandbox を本番の前提にしない

開発時には便利ですが、モデル生成コードをホスト権限で動かすことになるため、本番設計では実行境界をきちんと分けるべきです。ここは Code Mode そのものの良し悪しではなく、サンドボックス設計の問題です。

### 5. すべての問い合わせを Code Mode へ逃がさない

単純な 1 ツール呼び出しや、説明だけで済む質問では通常の tool calling のほうが素直です。Code Mode は「multi-tool computation」が本当にある場面で使ってこそ効きます。

## この発表の意味

Mastra は 2026 年に入ってから、Observational Memory、Workspaces、Remote Sandboxes、Browser、Background Tasks、Signals と、長時間かつ実務的なエージェントを構成するための部品をかなりの速度で積み上げています。今回の Code Mode は、その中でも特に **tool orchestration の粒度を 1 段引き上げる発表**です。

これまでは「ツールを呼べるエージェント」が中心でした。ここから先は「ツールを何度も呼ぶ必要がある問いを、どこまで 1 回の計算へ圧縮できるか」が重要になります。Code Mode は、その圧縮を専用ツールや ad-hoc なバックエンド実装に押しつけず、Mastra のエージェント実行モデルの中に取り込んだ、と見ると価値がはっきりします。

特に、既存ツールのバリデーションや trace を壊さず、分析部分だけをモデル生成コードへ任せる設計はかなり実務的です。自由すぎるコード実行ではなく、かといって固定的すぎる集計 API でもない。その中間を狙った、うまい落としどころだと思います。

## まとめ

Code Mode は、Mastra エージェントに「その場で生成した TypeScript で複数ツールを合成し、中間結果を整理してから返す」能力を与える機能です。普通の tool calling が苦手だった、複数データ源をまたぐ分析質問や集計処理に特に効きます。

ポイントは、単にコードを実行することではなく、**既存のツール境界・バリデーション・tracing を保ったまま、計算とオーケストレーションだけをコード化できる**点です。Mastra の Workspaces / Sandbox 群の文脈で見ると、この機能はかなり自然な次の一手でした。

今後、BI 的な問い合わせ、社内分析アシスタント、コーディングエージェントの結果集約などで、かなり使われるはずです。導入時は `LocalSandbox` で試しつつ、本番ではサンドボックス境界と allow-list をきちんと設計するのがよさそうです。
