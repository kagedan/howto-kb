## はじめに

7gardenでDX責任者をしているdaikiです。  
弊社はホテルの企画・開発・運営を行う会社で、僕は直近ではマーケティングチームのリードにつきながら、並行してエンジニアとしてAI推進・開発をしています。その中で棟数拡大を見据えた生産性の向上を実現するため、「マーケちゃん」というAIエージェントの開発を進めています。

![マーケちゃん](https://static.zenn.studio/user-upload/deployed-images/0e9b5d0b1b825e95fc9c97a2.png?sha=d600f6229b82823553a034f2c25e4aee0524fa60)

これがマーケちゃんのアイコンです。これまでSlackやNotionといったインターフェースを中心に、複雑なデータ分析やレポーティングの自動化といった機能を追加してきました。今回はこのマーケちゃんの拡張として、「**MTGが終わったら、ネクストアクションが勝手にNotionに起票され、実行可能なものは数値確認や要因分析まで（労力的に）ほぼ0コストで自動実行される**」という状態を目指したパイプラインを作った話を書きます。

## 背景

弊社のマーケチームでは毎週の定例MTGで、多数の「ネクストアクション（以下NA）」が議論されます。全運営施設が議論スコープとなるので絶対数が多い一方で、議論の流れの中で出てくる"生きた"論点が多いです。たとえば「**ある施設で価格変更後に予約は伸び始めているが、これは価格変更の効果か季節要因なのかを切り分けて分析する**」「**別の施設で韓国ゲストが減り台湾・日本人が増えているが、これは自社のプロモーション施策の影響なのか市場側の変化なのかを分析する**」のように、その場の文脈に強く紐付いた"すぐ動かしたい"類のものです。

ホテルマーケは未来の予約まで常に数字が動き続ける性質上、こうしたNAは **鮮度高く実行してチームに共有し、次の打ち手まで繋げるところをセットで進める** ことが重要です。判断が1〜2日遅れるだけで、対象期間の数字や打ち手の効きが変わってしまうこともあります。

一方で、NAの遂行はメンバーにとって明確な **「カットイン業務」** でもあります。当初予定していたタスクの遂行を圧迫するため、放置すればPDCAは止まり、その場で全部こなせば予定タスクが押し出される、という板挟みが日常的に起きていました。「議論で出る → Notionに起票する → 実施する → 結果をチームに共有する」のサイクルを、手をできるだけ止めずに高い鮮度で回したい、という葛藤がありました。

前提としてもう1つ。弊社ではAI議事録サービスの**Circleback**を導入しています。ミーティングはGoogle Meetを使用していますが、CirclebackはオンラインMTGに対して議事録やアクションアイテムを自動で抽出してくれるツールで、抽出結果をwebhookで外部システムに送信できます。今回のパイプラインは、このwebhookを起点に組み立てました。

## マーケチームの体験

理想とする見え方はシンプルで、**MTGが終わって席に戻る頃には、議事録から起こされたチケットがNotionに並び、自動実行に適したものはSlackに「自動実行の候補」として届いている**、という状態です。メンバーはそのSlackメッセージのボタンを1クリックするだけで実行が走り、結果はスレッド返信で戻ってきます。

**▼ MTG終了後に届くメッセージ例**

![Slackに並ぶ実行候補のSlackメッセージ](https://static.zenn.studio/user-upload/deployed-images/c4ee7cb357ac3e527f3761b4.png?sha=ed1cd238248715902c79f2cdb350b50230e3fa4e)

実際の施設名や数値を含むため、一部マスクしています。  
メンバーから見える流れを時系列に並べると、こんな感じです。

| タイミング | 起こること |
| --- | --- |
| MTG終了直後 | Circlebackが議事録からNAを抽出してwebhookで送信 |
| 数十秒〜数分後 | Notionにチケットが起票される |
| さらに数分後 | 自動実行できるかがLLMで判定され、対象になったチケットがSlackに届く |
| ボタンを押した直後 | そのSlackメッセージへのスレッド返信として実行結果が投稿される |

体感としては『Slackを覗いたら自動実行可能なNAが並んでいて、「実行」ボタンをクリックしたものだけ走る』というイメージです。

冒頭で「ほぼ0コスト」と書いたのは、メンバーの手がほぼかからないという意味です。これまで人手だった「チケットを起票する」「数値を確認・分析する」「結果をチームに共有する」という一連の工程が、ボタン1クリックに置き換わります。チーム共有も、結果メッセージのスレッドで共有したい相手をメンションするだけで終わるのが地味ですが効くポイントです。

さらに、実行結果のスレッド内でマーケちゃんにメンションをつけると、そのチケットの文脈を引き継いだまま追加分析や軌道修正を依頼できます。「自動実行で出てきた結果のうち、気になる切り口だけそのまま深掘りする」という動きも、同じスレッド上で完結します。

## 最初は、ボタンの無い全自動パイプラインだった

ここまで読んで「わざわざ人間にボタンを押させなくても、起票から実行まで一気に自動化すればいいのでは？」と思うかもしれません。実は最初の実装には、このボタンは存在しませんでした。起票されたチケットをそのまま実行まで流す、完全自動のパイプラインとして作り始めました。

その際はトークンコストの観点を考慮していました。というのが起票されたチケットを全件実行に回すと、1回のMTGで数十件の分析エージェントが走ることになり、コスト的に現実的ではありません。そこで **「自動実行する優先度」をLLMに判定させ、優先度の高いものだけを実行に回す** 設計にしました。ここまでは順調に見えていました。

ところが実際のMTGで試運転してみると、違和感がありました。このMTGは僕自身がファシリテーターとして参加しますが、「今週はこのテーマに注力したい」「ここの数字が知りたい」という温度感が残ったまま自動実行のアウトプットを見ると、「**あれ、さっき自動実行してほしいと思ったのはそれじゃないんだよな**」と感じる場面がありました。LLMが「優先度が高い」と選ぶチケットと、議論を経たチームが「これを動かしてほしい」と思っているチケットが噛み合っていない。Circlebackから抽出された議事録テキストだけでは、議論の温度感や「今週はここが勝負」といったチームの文脈までは載ってこないためです。

この乖離を受けて出した結論が、**判定は「実行候補のフィルタリング」に役割を絞り、最終トリガは人間のSlackボタンに任せる**という現在の形です。Slack通知はマーケチームが日常的に見るチャネルに流れるので、ボタン1クリックの手間はほぼ気にならないだろうという事前の議論もありました。さらに作ってみると、「どのチケットを誰が実行させたか」という情報自体がチーム内の文脈として共有されるという副産物もありました。「誰が・どういう意図で・どのチケットを実行したか？」がスレッドを覗けばわかる状態のため、最終的な施策の意思決定に至る議論も進めやすい、というメリットを感じています。

## 全体アーキテクチャ

上記の再設計を経て、最終的に構築したパイプラインの全体像です。

![全体パイプライン](https://static.zenn.studio/user-upload/deployed-images/fe8b293f8d91cc39433c074d.png?sha=f363edc20f5403000792ba3107eeff0fbb7b074a)

主要な技術スタックは以下です。

| 項目 | 内容 |
| --- | --- |
| LLM | Claude Haiku 4.5（分類・判定）/ Claude Sonnet 4.6（深掘り分析） |
| 言語 | TypeScript（Node.js） |
| インフラ | AWS Lambda / EventBridge / API Gateway |
| 可観測性 | Langfuse |

「議事録抽出 → 起票 → 実行可否判定 → 実行」の各段を **EventBridgeで繋ぎ、"実行可否の判定"と"実行"の間にSlack上でHuman in the Loop（以下HITL）を挟む**ようにしています。

判定と実行を別のEventBridge・別ハンドラに分けてあるのは、両者が独立したステージだからです。疎結合にしておけば、再実行や処理の差し込みによる拡張性も高まる、という意図です。前章のHITLも、ちょうどこの「分けてある間」に差し込んだ形になります。実装としては、判定はClaude Haiku 4.5を1ターンで安価に動かし、実行はClaude Sonnet 4.6で複数ツールを叩く重めの処理として走らせています。

余談ですが、Circlebackには"Insights"と呼ばれる機能があり、MTG終了時の議事録抽出にあたって**カスタムプロンプトや構造化出力を指定できます。** この機能を知った時「これは議事録サービスであると同時に、あらゆる自動化パイプラインの起点として使える」と感じたのがCircleback導入の決め手でした。現状、自動実行できるのは数値確認・データ分析の範囲のチケットですが、今後はマーケのコア業務の1つである販売価格の調整（="プライシング業務"）まで自動実行に含めていきたいと考えています。

ここからは、上の図のパイプラインを処理順にたどっていきます。

## 1. 決定論的に判定にかける対象を絞り込む

最初のステップは、Circlebackから構造化出力で抽出されたチケットのうち「判定にかけるもの」を絞り込むことです。  
前提として、Circlebackから抽出されるチケットは、上述のInsights機能の構造化出力により、タスクカテゴリ（分析、確認、プロモーション、レポーティング、プライシングなど）が付与される設定になっています。現段階ではこのタスクカテゴリの中には、LLM判定を持ち込むまでもなく実行不可能なものが存在します。それらを事前に決定論的に除外することで、無駄なトークンコストの発生を回避し、明らかに実行不可能なものが実行可能と誤判定される可能性を排除しています。

具体的には、Circlebackパイプラインから判定イベントを発行する段で、以下の絞り込みを入れています。

1. **タスクカテゴリ絞り込み**: 現在は `分析` / `確認` / `レポーティング` の3カテゴリに限定（対応カテゴリは今後徐々に広げていく予定）
2. **件数上限**: `TICKET_JUDGE_AUTO_LIMIT=20` で1 MTGあたりの判定起動件数に上限

```
// 絞り込みのイメージ
const eligible = createdTickets.filter(
  (entry) =>
    entry.group !== undefined && JUDGE_TARGET_TASK_CATEGORIES.has(entry.group),
);
const targets = eligible.slice(0, getAutoJudgeLimit()); // default: 20
```

現状では実行上限20件を決め打ちでスライスしていますが、運用する中でどの20件を選択させるか？というLLM判定を入れるなども検討したいと考えています。

## 2. 判定処理: 実行候補をSlackに流す

絞り込まれたチケットに対して、判定処理が走ります。Notionページのタイトルとプロパティを入力にして、以下のスキーマで判定結果を返します。

```
// 判定時にLLMに指定するzodスキーマ
const judgeOutputSchema = z.object({
  executable: z.boolean(), // 自動実行可能か
  executionType: z.enum([
    // 実行種別
    "numeric-check", // 数値・KPIの現状確認
    "root-cause-analysis", // 「なぜ」「原因」を深掘り
    "manual-only", // 人間判断必須（候補メッセージには出さない）
  ]),
  confidence: z.number().refine((v) => v >= 0 && v <= 1), // 判定の信頼度 (0〜1)
  reason: z.string(), // 判定理由（候補メッセージにも表示する）
  suggestedInputs: z.object({
    // 実行処理に渡す入力の推測値
    facilityName: z.string().nullable(), // 対象施設名
    targetMonth: z.string().nullable(), // 対象年月 (例: "2026-06")
    metric: z.string().nullable(), // 見るべき指標
    analysisFocus: z.string().nullable(), // 分析の観点
  }),
});
```

実行種別は「数値確認（`numeric-check`）」「要因分析（`root-cause-analysis`）」「人間判断必須（`manual-only`）」の3つです。`manual-only` と判定されたチケット（現段階ではマーケちゃんが実行不可能なタスク）は、そもそもSlackに出すとノイズになるため、Notionに起票されたチケットにスキップ理由のメモを記載した上で終了します。それ以外は判定結果を専用チャネルに「実行候補」のSlackメッセージとして投稿します。

![Slackに並ぶ実行候補のメッセージ一覧](https://static.zenn.studio/user-upload/deployed-images/3d1e0ba0814a2c73c200c755.png?sha=b9af59ef0bdf12f0b48662ab3289de745333b100)

結果、このようにSlack上に通知されます。

### ▼ Slackメッセージに判定結果を埋め込んで持ち運ぶ

ボタンが押されたあと、後述のSlack Interactivityが発火し、用意したAPI Gateway × LambdaにHTTPリクエストが飛びます。このAPI側では判定結果（どの実行種別か / どの施設・対象月か など）がすぐに必要になります。これらの情報を効率よく持ち回すために、Slackの **message metadata** 機能を使って判定結果をそのメッセージにそのまま埋め込んでいます。

`message metadata` は、Slackのメッセージに任意の構造化データを埋め込める仕組みです。今回の候補メッセージにはこんな感じのオブジェクトを付与しています。

```
// Slackの各メッセージに埋め込むmessage metadataの構造
{
  event_type: "ticket_judge_result", // metadataの種別（受け側のディスパッチ用）
  event_payload: { // 判定結果の本体
    pageId: "...", // 対象チケットのNotionページID
    executionType: "numeric-check", // 判定された実行種別
    suggestedInputs: { facilityName: "...", targetMonth: "2026-06", ... }, // 実行処理に渡す推測入力
    judgeReason: "...", // 判定理由
    judgeConfidence: 0.92, // 判定の信頼度 (0〜1)
  },
}
```

ボタンが押されたら、受け側はこの `event_payload` を取り出して実行イベントに詰めるだけで、判定の文脈をそのまま引き継げます。判定結果をDBなどから取り直す必要もありません。完全に余談ですが僕は以前はLINE公式アカウントのAPI（= Messaging API）を拡張したSaaSの開発をしていました。このMessaging APIにも似た仕組みがあったため、Slackの`metadata`機能を利用した設計もやりやすかったです。

## 3. ボタンクリックを受け取る（Slack Interactivity）

![候補メッセージに付いている「そのまま実行 / 補足して実行」ボタン](https://static.zenn.studio/user-upload/deployed-images/752567c4c2927a9d0a141f1d.png?sha=832627d9b47a5715f4cd84ea0e49f62b6a746c16)

実行候補のSlackメッセージが流れたあと、メンバーがこのボタンを押すと処理が再開します。

Slackには、メッセージ上のボタンが押されたときなどに、設定したURLにpayloadをPOSTしてくれる**Interactivity**という仕組みがあります。「誰がどのメッセージのどのボタンを押したか」と、前述の `message metadata` が、HTTPリクエストとしてAPIサーバーに届きます。

実行候補のボタンクリックも、この経路でInteractivityイベントが今回用意したAPIエンドポイントに届きます。エンドポイント上では署名検証 → metadataから判定結果を取り出し → 実行イベントを発行し、Lambdaが起動するという流れです。

```
// Lambda上の処理（ごく一部）
try {
  const result = await processSlackInteraction(parsed.data);
  console.log("Slack interaction processed", { result });
} catch (err) {
  console.error("Slack interaction handler failed", {
    error: err instanceof Error ? err.message : String(err),
  });
}
return reply.status(200).send();
```

SlackのInteractivityは **3秒以内に応答を返さないとタイムアウトしてユーザー側にエラーが見えてしまう** 仕様ですが、EventBridgeへのpublishは数百msで完了して制限内に余裕で収まるため、**publishの完了をawaitしてから応答を返す** 形にしています。失敗はサーバログとLangfuseで追跡しています。

学びとして、当初は3秒制限を意識してチケット自動実行のEvent発火をfire-and-forget（戻り値を待たずvoidで読み捨てる）にしていました。ところが実際のSlack上での動作確認時に、ボタンを押しても実行イベントが発火しないケースがありました。**LambdaはHTTPレスポンスを返した時点で実行環境が凍結するため、fire-and-forget方式ではEventBridgeへのイベント発行が完了する前に処理が止まってしまう**ことが原因でした。Lambda上でHTTPサーバーを動かす際は、重要な処理は応答前に確定させるのが安全という学びがありました。

### ▼ 補足プロンプトを付けて実行する

![「補足して実行」ボタンを押したときに開くモーダル](https://static.zenn.studio/user-upload/deployed-images/76657403cb5499f0a2414548.png?sha=f0889ec2ed56e83f37ea8a3021d238a69c301298)

候補メッセージには「▶ そのまま実行」だけでなく「📝 補足して実行」ボタンも用意しています。後者はクリックするとモーダルが開き、追加のプロンプト（「○○の観点で見てほしい」「△△を踏まえて」など）を書き込んで実行に渡すことができます。判定結果がそのまま使えるケースは前者、もう少し意図を加えたいケースは後者、というふうに使い分ける想定です。

## 4. 実行処理: 数値確認と要因分析

ボタンクリックで発行された実行イベントを受けて、実行処理が走ります。判定結果の `executionType` に応じて2種類のエージェントを使い分けます。

| 種類 | 用途 | モデル |
| --- | --- | --- |
| 数値確認（numeric-check） | 数値・KPIの現状確認系 | Claude Haiku 4.5 |
| 要因分析（root-cause-analysis） | 「なぜ」「原因」を深掘り | Claude Sonnet 4.6 |

実行が完了すると、結果は元のSlackメッセージへのスレッド返信として投稿されます。判定 → 実行 → 結果の文脈がSlackのスレッド単位でまとまるので、振り返り時の追跡もしやすいです。

加えて、実行時のセッションIDをそのスレッドに紐付けているため、後からスレッド内でマーケちゃんにメンションして追加の分析や軌道修正を依頼するときも、チケットの文脈を引き継いだまま深掘りできる作りにしています。

## おわりに

最終的に目指したいのは **ホテルマーケティングの「自動運転化」** で、本機能はそれを構成する一つと捉えています。今後の課題は、**既存機能の精度を落とさずに、自動実行可能な領域を広げていく**ことです。現在の自動実行の対象は `分析` / `確認` / `レポーティング` の3カテゴリのみですが、`プロモーション` / `プライシング` のような施策に直接踏み込むカテゴリも、安全に自動化できる範囲から徐々に広げていきます。

弊社ではこうした「ホテル×AI」を強化するためエンジニア・マーケターの仲間を積極採用中です！少しでもピンときた方、まずはカジュアルにお話ししましょう。
