私は AI エンジニアとして、AI SaaS を開発しています。開発していると、デモは動いたけれど、AI モジュールの精度がまだ安定しない、という場面によく遭遇します。

どこで間違えたのか。なぜ間違えたのか。prompt を直すべきなのか、schema を変えるべきなのか、後処理を足すべきなのか。次の実行で本当に良くなったのか。

この確認は地味に重いです。精度が足りないたびに、サンプルを流し、結果を見て、元データと照らし合わせ、失敗理由を考え、prompt やコードを直して、また流す。何度か繰り返しているうちに、ふと思いました。

AI モジュールを開発しているのだから、その改善プロセス自体も、AI に手伝わせられないだろうか。つまり、こういうループです。

GT を使って現在のモデルを自動評価する。失敗ケースを AI に読ませて原因を分析させる。改善案を出させる。必要なら prompt やコードを直す。そしてもう一度 eval を回す。最近よく聞く **loop engineering** にも近い発想です。目標を与えて、ループを回し、システム自身に改善させる。これがうまくいけば、私は一つひとつの case を見ながら prompt をいじるのではなく、システムに目標を与えて、精度が上がる方向へ進められるはずでした。

結論から先に書きます。**loop は私を手放してくれませんでした。** むしろ、私の判断を先に仕組みに埋め込まないと、loop はうまく回らないことが分かりました。私は「放置できる機械」を作っているつもりで、実際には自分がどこを曖昧に考えていたのかを映す仕組みを作っていた、という話です。

この記事は、eval pipeline の作り方を説明するものではありません。loop engineering を推す記事でもありません。`eval -> analyze -> improve -> eval` というループを本当に閉じようとすると、どこで割れるのか。その割れ目から何が見えたのか、という話です。

少しだけ背景を書きます。具体的な業務内容は伏せますが、対象は業務文書からの構造化抽出です。入力文書から複数の target field を抽出し、GT と比較します。eval セットは大きくありません。人手で確認済みの文書 GT が 14 件、合計 275 item（1 文書あたり最小 1 item、最大 64 item、中央値は約 9.5 item）。スコアリングはルールベースで、field value・numeric value・unit のような正規化可能な値を比較します。なお、反復の速さを優先して、実際の試行では eval セット全体ではなく 8〜12 件の小さな batch を回しています。

最初に思い描いていた流れは、かなりきれいでした。

1. サンプルを流し、現在バージョンの eval 結果を得る。
2. AI に失敗ケースと mismatch の詳細を読ませる。
3. AI に失敗原因を分析させる。
4. AI に原因に応じて prompt またはコードを修正させる。
5. もう一度 eval を回し、スコアと失敗分布が本当に改善したかを見る。

いかにもエンジニアリングらしい流れです。ただ実際に回してみると、最初に露呈したのはモデルの弱さではなく、私自身の「正しさ」に対する理解の粗さでした。

## 1つ目の割れ目: GT も疑う必要がある

最初の私はかなり単純に考えていました。

`model output != GT` なら、モデルが間違っている。だから prompt を直せばいい。

しかし最初の batch を回してすぐ、そう単純ではないことに気づきました。すべての mismatch がモデルの誤りとは限りません。ときには GT 自体を見直す必要があります。

機械的な入力作業を減らすため、GT は完全な空白から手入力しているわけではありません。まず一度抽出結果から skeleton を作ります。モデルが抽出した item、各 item の key field、numeric field、関連 field を `<document>.gt.json` に入れる。ただし、この skeleton はあくまで出発点であり、正式な答えではありません。正式に eval に使う GT は、校正用の HTML を開き、元文書と照らし合わせて一つずつ確認・修正し、reviewed として保存したものです。

それでも、GT が pipeline に入ると、心理的には「正解」として扱ってしまいやすい。スコアを見て `model != GT` が出ると、最初の反応はどうしても「モデルが間違えた、prompt を直そう」になりがちです。

でも元文書に戻って確認すると、GT item 自体が少しずれていたり、field の境界が曖昧だったり、そもそも文書側に解釈の余地があったりします。

たとえば、ある key field は一見短い値を読むだけに見えます。しかし実際の文書には、手書きの印、修正、途中で切れた文字、行をまたぐ対応関係、同じページ内の似た値が存在します。numeric field も同じで、一つの文書に複数の候補値が現れることがあります。eval がモデルに抽出してほしい「target value」はどれなのか。ここを先に決めないと、そのまま自動化したときに、誤った前提のまま結果だけが積み上がっていきます。

現実的な手戻りもあります。自動化を止めて、元文書を自分で確認する必要がある。prompt ではなく GT を修正することもある。GT を修正したら、その前に出したスコアは同じ土台では比較できません。そこで私は GT の content hash を eval metadata に入れ、各スコアがどの GT version に紐づくか追跡できるようにしました。

これが一つ目の割れ目です。私は loop をそのまま自動で進ませたかった。でも `model != GT` をそのまま「モデルの誤り」として機械に渡すことはできません。まず、モデルの問題なのか、GT の問題なのか、問題定義の問題なのかを分ける必要があります。ここを信頼できる形で分流できない限り、後続の improve は安心して任せられません。

## 2つ目の割れ目: AI の改善案は局所最適になりやすい

次の問題は、もっと分かりにくいです。最初は「成功」に見えるからです。

私の想定では、AI はスコアだけでなく失敗ケースも読み、prompt の修正案を出します。この方向自体は有用です。AI は mismatch からパターンを見つけられますし、prompt の曖昧な箇所を指摘することもあります。

ただし、AI は目の前の失敗ケースに強く引っ張られます。結果として、一見もっともらしいけれど、実際にはかなり局所的な修正案を出しがちです。

たとえば prompt にこういうルールを足そうとします。

> この layout のときは、特定の位置の値を優先する。

> 近くに特定のテキストがある場合は、隣の値を抽出しない。

> ある field が特定の領域に出てきたら、後続 item に継承する。

こうした提案の一部は、たしかに問題を捉えています。一方で、目の前の一つか二つの sample を直すための patch に見えるものもあります。AI が意図的に「暗記」しているわけではありません。ただ、与えられた文脈が失敗ケースなので、その case を説明し、その case を直す方向へ自然に寄っていきます。

そこで私は、この段階でよく止めます。

「この sample だけに効くルールではなく、もっと一般的な原則にして」

「表面的な特徴ではなく、背後にある抽出方針を書いて」

「ここではどうするか、ではなく、同種の曖昧な状況でどう判断するかを書いて」

ただ、それでもリスクは消えません。手元のデータが少なく、失敗分布が十分に広くない。AI の分析は現在見えている失敗 sample に引っ張られやすい。そして prompt は、そもそもルールを積み上げやすい媒体です。気づくと「原則」ではなく「patch」になっています。

これは素朴な過学習です。ただ prompt engineering の皮をかぶると、「bad case に対する丁寧な改善」に見えやすい。

だから私は、AI の修正案をそのまま信じません。修正したら必ず元の base dataset を回し、degradation が起きていないことを確認します。

実際、ある 8 sample の batch を 2026-06-12 に数回連続で回した記録があります。途中で prompt だけでなく抽出・評価ロジックも触っているので、厳密な A/B ではありません。それでも、「スコアが上がった」ことをどう見るべきかの例にはなります。

なお、以下の field A は比較的読みやすい key field、field B / C は境界が曖昧で「どこまでを target として数えるか」が難しい field だと思ってください。

| run | field A F1 | field B recall | field B precision | field B F1 | field C F1 | unit accuracy |
| --- | --- | --- | --- | --- | --- | --- |
| 20260612-173333 | 0.53 | 0.54 | 0.17 | 0.26 | 0.13 | 1.00 |
| 20260612-174749 | 0.37 | 0.42 | 0.12 | 0.18 | 0.15 | 1.00 |
| 20260612-183134 | 0.93 | 0.51 | 0.35 | 0.41 | 0.38 | 0.95 |
| 20260612-184322 | 0.93 | 0.82 | 0.76 | 0.78 | 0.56 | 1.00 |

この数字が教えてくれるのは、「スコアが上がったら AI を信じてよい」ではありません。むしろ逆です。改善は field ごとにかなり偏って現れます。3 回目では field A の F1 が 0.37 / 0.53 から一気に 0.93 まで上がっていますが、field B の F1 は 0.41、field C の F1 は 0.38 のままです。

つまり、一部が良くなったからといって、抽出全体が良くなったとは限らない。

そのため、improve の step は明確に止めるようにしました。AI は提案してよい。でも、無条件に適用してはいけない。私の中でのルールは 2 つです。

1つ目は、変更後に必ず base dataset を回し、degradation がないことを確認することです。新しい case で良くなっても、base が落ちるなら、それは improve ではなく prompt を局所に寄せただけです。

2つ目は、prompt の変更を、単独の失敗行ではなく、同じ root cause を持つ失敗群から出すことです。そのために eval の alignment logic も、key field で対応付けてから fallback する形にしました（詳細は次の節で書きます）。key field の読み違い一つで numeric value・unit・related field まで全部ずれてしまうと、失敗原因が汚れてしまうからです。

## 3つ目の割れ目: 直すべきなのは prompt ではなく自分のロジックかもしれない

いちばん苦しい失敗は、prompt でも GT でもなく、自分が組んだ logic にあります。

最初は、失敗を見ると自然に prompt に向かいます。モデルがうまく抽出できない。では prompt が曖昧なのではないか。しかし何度も回していると、prompt をいくら直しても本質的には解決しない問題があります。直すべきなのは pipeline 側の logic です。

ここでいう pipeline logic には、モデル出力後の後処理だけでなく、eval 側で item をどう対応付けるか、値をどう正規化するか、mismatch をどう分類するかも含めています。つまり、モデルの出力そのものではなく、**出力をどう解釈し、どう評価するか**の部分です。

つまり、AI が分析しているのは元文書そのものではなく、pipeline を通った後の mismatch / score / report です。`eval pipeline` や `post-processing` が粗いと、AI が見る mismatch も粗くなります。その結果、本当は pipeline logic の問題なのに、AI は prompt の問題として改善案を出してしまいます。

たとえば numeric field で、GT が `1000`、model output が `1,000.0` だったとします。文字列として比較すれば mismatch です。でも人間から見ると、これは同じ値です。

この mismatch をそのまま AI に渡すと、AI は「数値をもっと正確に出すよう prompt を強めましょう」と言うかもしれません。しかし本当に直すべきなのは prompt ではなく、評価側の normalization logic です。

unit も同じです。unit だけを独立して評価するのではなく、numeric value がまず match していて、そのうえで unit accuracy を見るべきです。related field も同様で、ページ単位の情報、行単位の情報、継承が必要な情報、後処理で補うべき情報があります。これらをすべて prompt に押し込むのには無理があります。

同じことは alignment にも起きます。item の対応付けが粗いと、1 つの key field の読み違いが、後続の numeric value・unit・related field の mismatch に見えてしまいます。そこで、まず双方で unique かつ non-empty な key field で対応付け、残ったものだけを同じページ内の順序で fallback するようにしました。これは prompt の修正ではなく evaluation logic の修正です。しかし、mismatch がどう分類されるかを直接変えます。

このとき気づいたのは、**pipeline は中立な観測者ではない**ということです。pipeline は、AI が失敗をどう分析するかを形作ります。そして、私が次に何を直したくなるかも形作ります。

alignment logic が key field のずれを numeric value や unit の誤りとしても罰してしまうと、AI は root cause を誤解します。結果が missed と spurious を分けていなければ、モデルが保守的に漏らしているのか、積極的に余計なものを抽出しているのか分かりません。レポートに GT version や状態が残っていなければ、異なる実行結果を比較してよいのかも分かりません。

ある sample では、field A は 11/11 で一致し、F1 = 1.00 でした。一見かなり良さそうです。しかし field B は 15 TP、7 FN、45 FP で precision は 0.25。field C も 9 TP、2 FN、34 FP で precision は約 0.21 でした。

もし eval が「このページは OK / NG」しか返さなければ、私は field A の問題だと思ったかもしれません。あるいは局所的な認識能力をさらに調整しようとしたかもしれません。しかし内訳を見ると、根本は「読めない」ことではなく、target field B/C として何を数えるべきかの境界が緩いことに近い。

最初は少し嫌でした。AI に improve を手伝わせるはずが、結局また自分が logic を直している。しかし今は、それも必要な発見だったと思っています。こういう問題では、いったん止まって、prompt ではなく自分の logic が粗すぎないかを見る必要があります。モデルをもう少し従順にしても、解けない問題はあります。

## 作っていたのは、放置できる機械ではなかった

3つ目の割れ目まで来て、この取り組みの見え方が変わりました。

私は「eval の自動化」を作っているつもりでした。しかし実際には、人間の判断を増幅するための**分流システム**を作っていました。このシステムには、少なくとも 3 つの入口があります。

この 3 つを全部まとめて「スコアが低いから prompt を直そう」にしてしまうと、最初にやりたかった自動化は簡単に壊れます。人間が loop に出てくるのは、自動化の失敗ではありません。むしろ、そこが最後まで残る中核でした。

自動 loop には、現実的なコストもあります。たとえば上の 8 sample の run は約 1.97 ドル。12 sample に広げた run は約 4.06 ドルでした。金額自体は大きくありません。ただ意味ははっきりしています。improve の判断を間違えると、その間違いは一つの提案で止まらず、**請求とレポートに載った形で残ります**。

もう一つ重要なのは、失敗を分析する AI 自体も間違えることです。GT の問題をモデルの問題として扱うこともあります。pipeline logic の問題を prompt の問題として扱うこともあります。それを全部信じると、loop はだんだん裁判官の偏りに合わせて最適化されます。頑張れば頑張るほど、変な方向へ進みます。

そして、いつ終わりにするか。今の停止条件は「スコアがいくつになったか」ではなく、次のような状態を見ています。

1. 見えている dataset のスコアが大きく動かなくなっている。
2. base dataset に degradation が出ていない。
3. 残っている失敗の主因が、GT の誤り・文書の曖昧さ・pipeline logic の境界であり、モデルが明らかにできていないわけではない。

残差が主に data や logic boundary によるものなら、さらに prompt を締めても過学習に近づきます。そのときは prompt を責めるのではなく、GT を直す、pipeline を直す、sample を増やすべきです。

## おわりに

最近、loop engineering という言葉をよく見ます。そこで少し不思議な既視感がありました。私が最初にやろうとしていた `eval -> analyze -> improve -> eval` も、かなり近い発想だったからです。

当時の私は、loop がうまく回れば、人間の介入は少しずつ減っていくと思っていました。でも実際に一周してみると、起きたことは少し違いました。loop は私を手放してくれませんでした。むしろ、私の判断を先に仕組みに埋め込まないと、うまく回らないことが分かりました。

以前は prompt の一文一文の中で、その場で判断していました。今はその判断を、GT・pipeline logic・alignment strategy・停止条件に前もって埋め込む必要があります。そして一度間違えると、機械はそれを高速に何度も繰り返します。いちばん難しいのは、何を正解とするか、どの boundary case が重要か、いつ prompt を直し、いつ GT を直し、いつ pipeline logic 自体を疑うべきかです。

結局、私は prompt を調整していたつもりでした。しかし本当は、このタスクに対する自分の判断を少しずつ明確にさせられていたのだと思います。loop を作ろうとして、結局いちばん整理されたのは、自分が何を「正しい」と見なしているのかでした。
