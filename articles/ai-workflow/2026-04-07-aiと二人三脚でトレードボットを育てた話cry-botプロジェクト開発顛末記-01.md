## 「Geminiでできるなら、Claudeでもできるはず」

きっかけは、QiitaでみつけたAI駆動の暗号資産トレードボットの記事だった。GeminiをAIエンジンに据えて相場を判断させ、自動売買する——そのアイデアが頭から離れなかった。

自分はクリプトファンとして数年、仮想通貨に興味深々なのである。しかし手動トレードは感情に左右されるし、仕事の合間にずっとチャートを見ているわけにもいかない。ならば、巷で騒がしいAIに判断を委ねてみたらどうか。

ただ——自分はUI/UXデザイナーだ。コードは書けるが、インフラ構築やAPIの深いところはまだ手探り。そこで選んだのが、Claudeをアーキテクトにする、という方法だった。単にボットのコードを書くのではなく、設計から一緒に考えてもらう。そのセッションの記録が「CRY-BOT-Project」だ。

## Day 1 — まず「制約」と向き合う

最初の壁は、想像より早く訪れた。API利用を目的に準備を進めてきたOKCoin Japanはスポット取引専用で、CFDや信用取引に対応していない。つまり「空売り」ができない。

Claudeとの対話の中で、この制約の意味が明確になっていった。双方向のポジションを取れないということは、ボットのロジックを「買い→売り」の一方向サイクルに特化させる必要がある。弱気相場での稼ぎ方は根本から別の設計になる、と。

もうひとつの制約：JPY残高がゼロだった。保有はBTC・XRP・SHIBのみ。そこで「Phase 0」という概念が生まれた。保有資産の一部を売却してJPYを確保し、そこからBOT運用を開始する初期化フェーズだ。ただし、XRPとSHIBは長期HODLの方針で「触れない」と決めた。

こうして、BTC/JPY単一ペアでのスポットトレードという、明確なスコープが定まった。

## Day 2–3 — アーキテクチャの解剖

技術スタックはClaudeとの議論の中で固まっていった。

| レイヤー | 採用技術 |
| --- | --- |
| 取引所 | OKCoin Japan（スポットAPI） |
| AIエンジン | Claude API（Haiku / Sonnet） |
| インフラ | AWS EC2 + Docker |
| シークレット管理 | AWS SSM Parameter Store |
| 開発環境 | GitHub Codespaces |

APIキーをどう安全に管理するか——これが最初の実装上の課題だった。環境変数をそのままGitHubに置くのはリスクがある。Claudeが提案したのはAWS SSM Parameter Storeを使う方式で、EC2がインスタンス起動時にSecureStringとして暗号化されたキーを取得する構成だ。

```
登録したSSMパラメータ（5種類）:
/trading-bot/anthropic-api-key
/trading-bot/okcoin-api-key
/trading-bot/okcoin-secret-key
/trading-bot/okcoin-passphrase
/trading-bot/github-token
```

GitHub TokenをSSMに入れる発想は最初なかった。EC2がプライベートリポジトリをcloneする際にも認証が必要なことを、実装中にClaudeから指摘されて気づいた。こういう「実際に動かすと当たる壁」を事前に拾ってくれるのが、AIとのペアプロの強みだと実感した瞬間だった。

途中、`awscliv2.zip`を誤ってGitにコミットしてしまうというミスも起きた。これも検知→修正をClaudeと一緒に進めた。

## 設計の核心 — 「AIに何を判断させるか」を絞り込む

ボットの判断フローで最も議論に時間をかけたのが、Claude APIの呼び出しコストの問題だった。60秒ごとに相場データを取得し、そのたびにSonnetへ投げ続けると——膨大なトークンが消費され、API費用がとんでもないことになる。

ここで普段の仕事の感覚が役に立った。GUIアプリ開発でデバイス信号処理の場面で似たような問題に頻繁にぶつかる。センサーから流れてくる大量のストリームデータをそのまま全部処理しようとすると、CPUが悲鳴を上げる。受信・判定・制御をむやみに繰り返せば、肝心なデータの読み飛ばしや処理遅延が起きる。

海で波を読む感覚に近い。風で立った細かい波紋（風浪）はただのノイズだ。でも沖から来る大きなうねりは、遠目にもその予兆がある。サーファーは全ての波に反応するのではなく、乗るべき一本を見極めて待つ。相場も同じで、60秒ごとの小さな揺れに振り回されても仕方ない。問題はどこで、どのタイミングでAIの判断を呼ぶか——だ。

解決策として採用したのが**二段階フィルタリング**だ。

**Stage 1 — ルールベースフィルター（Claude呼び出しなし）**

MA整合チェック、RSI > 65、スプレッド > 0.30%、PANICレジームの検出。この段階でトレード条件を満たさない場合、Claudeは呼び出さない。風浪はここで弾く。

**Stage 2 — AI判断（モデルをコストで使い分ける）**

* ウォッチモード（NORMAL相場 / ポジションなし）→ **Claude Haiku**
* アクティブモード（TREND/BREAKOUTレジーム or ポジション保有中）→ **Claude Sonnet**

うねりの予兆を感じたときだけ、Sonnetが動く。

この設計により、推定月額API費用は当初の$30–50から\*\*$8–23\*\*へと大幅に圧縮できた。AIを判断の主軸に据えながら、コストは人間がコントロールする——このバランスが重要だった。

イグジット（売り判断）もルールベース化した。以下の5条件をチェックする独立したモジュールとして実装している。

1. RSI > 70（買われすぎ）
2. MAアライメント崩れ
3. PANICレジーム検出
4. 含み損 -2% 超
5. 含み益 +1.5% 超かつBEARISHバイアス転換

## 番外編 — デプロイ前夜のどんでん返し

EC2へのデプロイ直前、AWSアカウントが一時停止された。支払い検証の問題だった。

ボットのコードは書き上がっている。SSMのパラメータも登録済み。EC2インスタンスは起動中で、ElasticIPも割り当て済み——なのにアカウントが凍結。焦りながら調べると、国内大手クレジットカードではAWSの自動決済が通らないケースがあるらしい。

幸い、非銀行系のクレジットカード（同じVISAブランドなのに）では問題なく認証が通過。手続き完了まで約30分。インフラは万全なのに、決済という完全に別のレイヤーで止まる——個人開発ならではのリアルな落とし穴だった。

その後も予期せぬ出来事は続いた。運用開始後、AnthropicのAPIクレジットが切れてBOTが静かに停止した。**Auto-rechargeを最初から設定しておかなかったことを後悔した**。「インフラの冗長性」はコードの外にも存在する。

## 現在地 — dry\_run=True、まだ始まっていない

執筆時点（2026年4月）、ボットはまだ本番取引を一度も実行していない。`dry_run=True`のまま稼働し、判断ログを積み上げている段階だ。

意図的な選択だった。「動くコードがある」と「信頼できるロジックがある」は別の話だ。ドライランで実際の相場データに対する判断を観察し、意図した動作をしているかを確認してから、はじめてライブ資金を投入する。

この言葉が、このプロジェクト全体を象徴している気がする。

## おわりに — AIとペアプロしてわかったこと

このプロジェクトを通じて、AIとの共同開発のリズムが少しわかってきた。Claudeは「答えを出してくれる」存在ではなく、「問いを整理してくれる」存在だ。制約の意味を明確にすること、設計の選択肢を並べること、実装上の盲点を指摘すること——そういう思考のパートナーとして機能した。

デザイナーとして仕事をしていると、「ユーザーの代わりに考える」という訓練は積んでいる。でもエンジニアリングでは、「システムの代わりに考える」視点が弱い。そこをAIが補ってくれた感覚がある。

CRY-BOTは、まだ眠っている。でも、ログを見るたびに、AIがどう相場を読んでいるかがわかってきた。次の記事では、ドライランの判断ログを分析してみたい。
