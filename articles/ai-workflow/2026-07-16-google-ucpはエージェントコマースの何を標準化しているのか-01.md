こんにちは！ブロックチェーンエンジニアの山口夏生（[@0x\_natto](https://x.com/0x_natto)）です。  
ブロックチェーン×AI Agentで自律経済圏を創る開発組織Komlock labでCTOをしています。

2026年1月にGoogleが発表したUCP（Universal Commerce Protocol）は「AIエージェントが買い物を代行する時代のコマース共通言語」と紹介されていますが、実際の通信で何がどう流れるのかまで踏み込んだ日本語の説明はほとんど見当たりません。この記事では仕様の原文だけを頼りに、`.well-known/ucp` のJSONの中身、サーバが積集合から決めるcapability negotiation、チェックアウトの状態機械、AP2 mandateが差し込まれる位置までを順に開いていきます。

## UCP とは何か

UCPは、Googleが2026年1月11日、小売業界の年次カンファレンスNRF 2026の基調講演で発表したエージェンティックコマースのオープン標準です。Sundar Pichaiは「オープンかつ特定技術に依存せず、Shopify、Etsy、Wayfair、Target、Walmartと共に作り、20社以上が支持している」と述べています。発表時の報道では、VisaやMastercard、Stripeといった決済側のプレイヤーも支持者に名を連ねています。

<https://blog.google/company-news/inside-google/message-ceo/nrf-2026-remarks/>

解決したい課題はシンプルです。エージェントごとにEC事業者が個別接続を作ると、接続数が爆発します。Googleの発表文の表現を借りると「個々のエージェントごとに固有の接続を要求する代わりに、UCPはすべてのエージェントが容易にやり取りできるようにする」。いわゆるN×N問題の解消です。ここまでは既存記事で散々語られているので、この記事ではこれ以上掘りません。

先に、ペルソナが持ちがちな誤解を3つ潰しておきます。

| 誤解 | 実際 |
| --- | --- |
| UCP ≠ 決済実行プロトコル | 決済手段や資金移動レールそのものは定義しません。credentialの受け渡し規約だけを決め、処理はpayment handlerという差し替え可能な部品に委譲されます（後述） |
| UCP ≠ AP2の後継・置き換え | AP2はUCPの拡張（`dev.ucp.shopping.ap2_mandate`）としてチェックアウトの上に合成されます。競合ではなく合成関係です |
| UCP ≠ 中立団体の標準 | ライセンスはApache 2.0ですが、A2AやMCPのようなLinux Foundationへの移管は発表されていません。Google主導です |

プロトコル群の中での位置づけを整理するとこうなります。AP2が「誰が認可したか」、x402が「支払い要求と支払い証明をどう取り交わすか」だとすると、UCPは「何を、どのカートで、どう発見・交渉して買うか」の層です。

| 層 | プロトコル | 役割 |
| --- | --- | --- |
| 通信 | MCP / A2A | エージェントとツール・エージェント間の会話 |
| コマース | UCP / ACP（Agentic Commerce Protocol、OpenAI） | 商品発見からチェックアウトまでの取引 |
| 決済承認 | AP2 | 「誰が何を認可したか」の証跡（mandate） |
| 支払い実行 | x402 など | 支払い要求の提示と支払い証明、サービス提供の結び付け |

Googleの発表文も、UCPはA2A、AP2、MCPといった既存プロトコルと互換だと明記しています。

<https://blog.google/products/ads-commerce/agentic-commerce-ai-tools-protocol-retailers-platforms/>

もう1つ、仕様が定義する登場人物を押さえておきます。UCPは4つの役割で書かれています。

| 役割 | 何をするか | 例 |
| --- | --- | --- |
| Platform | capabilityを消費する側。ユーザーや組織を代理して取引を組み立てる | AIショッピングアシスタント、スーパーアプリ、B2B調達システム |
| Business | capabilityを公開する側。取引の文脈では**Merchant of Record**として金融責任と取引の所有権を保持する | 小売、航空会社、ホテル、サプライヤー |
| Credential Provider | ユーザーの決済手段や住所を安全に預かり、トークンを発行する | Google Walletなどのデジタルウォレット |
| Payment Service Provider | 事業者側で決済を処理し、カードネットワークと通信する | Stripe、Adyen、PayPal |

「エージェントが買う」文脈で語られがちですが、PlatformとBusinessはcapabilityの流れる向きで決まる役割で、業種を問いません。仕様はB2Bの調達やエージェント同士の取引にも同じ枠組みが使えると明記しています。

ここからが本題。入口の発見から順に、この共通言語の中身を見ていきます。

## 入口はただのJSON：`.well-known/ucp` を開く

UCPの入口は拍子抜けするほど地味です。事業者が自分のサイトの `https://<host>/.well-known/ucp` に置く、静的なJSON1枚。これだけです。このJSONはプロファイルと呼ばれ、エージェントはGETするだけで次の5つをすべて解決できます。

1. 事業者が対応するプロトコルバージョン
2. APIエンドポイントの場所
3. 対応しているcapabilityの一覧
4. 受け入れる決済ハンドラ
5. メッセージ署名を検証するための公開鍵

実物の構造を見てみます。Google公式ガイドのサンプルから、幹になる部分を抜き出したものです。

.well-known/ucp（公式サンプルの抜粋）

```
{
  "ucp": {
    "version": "2026-04-08",
    "services": {
      "dev.ucp.shopping": [
        {
          "version": "2026-04-08",
          "transport": "rest",
          "endpoint": "https://business.example.com/ucp/v1",
          "spec": "https://ucp.dev/specification/overview",
          "schema": "https://ucp.dev/2026-04-08/services/shopping/rest.openapi.json"
        }
      ]
    },
    "capabilities": {
      "dev.ucp.shopping.checkout": [
        {
          "version": "2026-04-08",
          "spec": "https://ucp.dev/specification/checkout",
          "schema": "https://ucp.dev/2026-04-08/schemas/shopping/checkout.json"
        }
      ],
      "dev.ucp.shopping.fulfillment": [
        {
          "version": "2026-04-08",
          "spec": "https://ucp.dev/specification/fulfillment",
          "schema": "https://ucp.dev/2026-04-08/schemas/shopping/fulfillment.json",
          "extends": "dev.ucp.shopping.checkout"
        }
      ]
    },
    "payment_handlers": {
      "com.google.pay": [
        {
          "id": "8c9202bd-63cc-4241-8d24-d57ce69ea31c",
          "version": "2026-01-23",
          "spec": "https://pay.google.com/gp/p/ucp/2026-01-23/",
          "config": { "environment": "TEST", "...": "..." }
        }
      ]
    }
  },
  "keys": [
    {
      "kid": "key_2026",
      "kty": "EC",
      "crv": "P-256",
      "x": "WbbXwVYGdJoP4Xm3qCkGvBRcRvKtEfXDbWvPzpPS8LA",

```

<https://developers.google.com/merchant/ucp/guides/ucp-profile>

構造はきれいに2階建てです。`ucp` オブジェクトが「何ができて、どこに繋げばいいか」を宣言し、ルート直下に並ぶ `keys` がJWK形式の公開鍵を配ります（RFC 7517のJWK Setとしてそのまま使える形式です）。capabilityの宣言と検証鍵の配布が、同じ1ファイル。これが、事前のAPIキー交換なしに接続を始められるpermissionless onboardingの技術的な土台です。仕様の原文はこう説明しています。

> any platform with a discoverable profile can interact with any business without prior registration.

事前のAPIキー発行も共有シークレットの交換もなし。お互いが相手のプロファイルの公開鍵を見て、RFC 9421（HTTP Message Signatures）でメッセージ署名を検証し合う設計です。エージェント側も自分のプロファイルを公開し、後述の `UCP-Agent` ヘッダで居場所を伝えます。

認証がこの署名方式だけ、という意味ではありません。既知の相手とはOAuth 2.0、APIキー、mTLSといった事前登録型の認証も併用できます。事前登録なしで始められるのはRFC 9421の署名だけ、という整理です。

そして、この1ファイルには地味に厳格な配信ルールが課されています。仕様のMUST/MUST NOTを拾うとこうなります。

| 要件 | レベル |
| --- | --- |
| HTTPSで配信する | MUST |
| プロファイルのエンドポイントで3xxリダイレクトを使わない | MUST NOT |
| `Cache-Control` に `public` と `max-age` 60秒以上を含める | MUST |
| `private`、`no-store`、`no-cache` で配信しない | MUST NOT |
| 認証なしで公開アクセスできる | MUST |

<https://ucp.dev/2026-04-08/specification/overview/>

リダイレクト禁止は狙いが明確です。プロファイルは鍵を配る信頼の起点なので、転送で別オリジンに飛ばせてしまうと成りすましの入口になります。キャッシュ必須なのは、多数のエージェントから繰り返し参照されるファイルだから。毎回オリジンへ取りに行かせない前提です。信頼のアンカーとスケーラビリティの両方を、この静的ファイル1枚に背負わせています。

![エージェントが.well-known/ucpをGETして事業者のプロファイルを読む図](https://static.zenn.studio/user-upload/deployed-images/bdb316c640098e26520f1f23.png?sha=805bb9091970d5499b7174442dc78bcbda67b191)

## capability negotiation：サーバが積集合から決める

プロファイルを読んだだけでは取引は始まりません。エージェント側にも「自分は何ができるか」の宣言があり、両者の擦り合わせが必要です。UCPはこれをcapability negotiationと呼び、設計として一番特徴が出ているのがここです。

仕組みはこうです。エージェントはリクエストのたびに、自分のプロファイルURLを `UCP-Agent` ヘッダで提示します。

```
UCP-Agent: profile="https://agent.example/profiles/shopping-agent.json"
```

形式はRFC 8941のDictionary Structured Field。なお、UCPの呼び出し方には後述する4種のトランスポートがあり、MCPトランスポートの場合はHTTPヘッダの代わりに、`tools/call` の引数内の `meta` オブジェクトに同じ情報を入れます。

事業者側はこのプロファイルを取得し、自分の宣言との積集合を計算して、有効になるcapabilityを確定します。仕様はこれをserver-selectsアーキテクチャと呼びます。原文を引きます。

> Capability negotiation follows a server-selects architecture. The business determines the active capabilities by computing the intersection of its own declared capabilities with those in the platform's profile.

決めるのは常に事業者側。アルゴリズムは4ステップです。

1. 名前で積集合を取る。候補になるのは双方が宣言しているcapabilityだけ
2. バージョンを選ぶ。双方の配列に共通するバージョンのうち最新の日付を採用し、共通がなければ除外
3. 孤立した拡張を刈る。親capabilityが積集合に残らなかった拡張を除去
4. 除去がなくなるまで3を繰り返し、拡張の連鎖に対応する

<https://github.com/Universal-Commerce-Protocol/ucp/blob/main/docs/documentation/core-concepts.md>

言葉だけだと掴みにくいので、仕様に忠実な作例で追ってみます。数値入りの完全な例は原文にないため、これは自分で構成した例です。

![エージェントと事業者の宣言の積集合から有効capabilityが決まる図](https://static.zenn.studio/user-upload/deployed-images/322c53afcf4b11b370250845.png?sha=f9d85bd9cf0423aa1e1cc83ad4b024d170adb376)

ap2\_mandateは事業者が宣言していないので1で落ち、fulfillmentは共通バージョンがないので2で落ちます。結果、このペアで有効になるのはcheckoutだけ。注目すべきは、AP2による決済承認が使えるかどうかすら、このnegotiationの結果で決まるという点です。

### 名前空間：ドメインを持てば拡張を定義できる

capabilityの名前は `dev.ucp.shopping.checkout` のようなreverse-domain形式です。仕様が例示している実名を並べます。

| capability | 種別 |
| --- | --- |
| `dev.ucp.shopping.checkout` | コア（購入セッションの開始と完了。拡張群の親） |
| `dev.ucp.shopping.cart` | コア（チェックアウト前のカート管理） |
| `dev.ucp.shopping.catalog.search` | コア（カタログ横断の検索） |
| `dev.ucp.shopping.catalog.lookup` | コア（商品IDでの取得） |
| `dev.ucp.shopping.order` | コア（注文ライフサイクルのイベント） |
| `dev.ucp.common.identity_linking` | コア（アカウント連携、OAuth 2.0 + PKCE） |
| `dev.ucp.shopping.fulfillment` | 拡張（checkoutをextends。配送・受け取り） |
| `dev.ucp.shopping.discount` | 拡張（checkoutとcartをextends。割引コード） |
| `dev.ucp.shopping.ap2_mandate` | 拡張（checkoutをextends。AP2接続点） |
| `dev.ucp.shopping.buyer_consent` | 拡張（checkoutとcartをextends。明示的同意の取得） |

このreverse-domain命名が、ガバナンスの仕組みそのものです。仕様は、capabilityのspec/schema URLのオリジンが名前空間の権威ドメインと一致することをMUSTとし、一致しないcapabilityの拒否を要求しています。`dev.ucp.*` はUCPのTech Councilが管理する予約名前空間で、`com.example.*` を名乗れるのはexample.comの持ち主だけ。ベンダーは誰の承認も待たずに独自capabilityを定義でき、negotiationが常にopt-inなので、それが有効になるのは双方が宣言したときだけです。

capabilityの上にはserviceという束があり、`dev.ucp.shopping` のようにvertical単位でcapability群をまとめます。将来ホテル予約のような新しいverticalが増えるときは、`dev.ucp.hotels` のようなservice名前空間の追加で拡張される設計です。ここまでの階層関係は公式の全体構成図が分かりやすいので貼っておきます。

![UCPの全体構成。Services、Capabilities、Extensions、Transportsの4層をPlatformとBusinessが挟む](https://static.zenn.studio/user-upload/deployed-images/66f4cc4da620e41aa1e03d5c.jpg?sha=3534cdbb04b8591e7939d319dde16112ce3026f9)  
*UCPの全体構成（出典: [Google / UCP公式ドキュメント](https://ucp.dev/documentation/core-concepts/)）*

バージョンは `2026-04-08` のような日付形式で、capabilityごとに独立して進みます。旧バージョンを併売する事業者は `supported_versions` フィールドでバージョン別プロファイルを広告できます。トランスポートはREST、MCP、A2A、EP（Embedded Protocol）の4種があり、同じcapabilityを複数トランスポートで公開できます。プロトコルの意味論を1つに保ったまま、呼び出し方だけを環境に合わせて選ばせる構えです。

## チェックアウトの状態機械

negotiationが済むと、いよいよ取引です。UCPのチェックアウトはセッションとして表現され、6つの状態を持つ状態機械として定義されています。

*遷移の矢印は仕様の状態図に基づきますが、仕様原文は遷移条件のラベルを付けていないため、図中のラベルは補足です。*

| 状態 | 意味 |
| --- | --- |
| `incomplete` | 必須情報が不足。`messages` を見てPUTで解消を試みる |
| `requires_escalation` | API経由で埋められない情報がある。人間へハンドオフ |
| `ready_for_complete` | 全情報が揃い、プログラムから確定できる |
| `complete_in_progress` | 事業者がComplete要求を処理中 |
| `completed` | 注文成立。以後は不変（immutable） |
| `canceled` | 無効または期限切れ。任意の状態から到達 |

<https://ucp.dev/2026-04-08/specification/checkout/>

この状態機械で押さえるべき設計判断は、状態を決めるのが常に事業者側だということです。原文は一文で言い切っています。

> The business sets the status; the platform receives messages indicating what's needed to progress.

エージェントにできるのは、セッション作成のPOST、情報を足すPUT、確定を要求するCompleteの3つだけ。状態遷移そのものは指示できず、返ってきたstatusと `messages` を読んで次の手を選ぶ観測者です。AP2のmandate駆動やx402の「402が返ったら支払って再送」というモデルと比べると、この非対称は際立ちます。取引の主導権は最後まで事業者に。merchant-authorityの徹底です。

`messages` 配列がエージェントへの指示書になります。errorには `severity` が必須で、これが次のアクションを決めます。

| severity | 意味 | エージェントの対応 |
| --- | --- | --- |
| `recoverable` | API経由の修正で解決できる | PUTで直して再試行 |
| `requires_buyer_input` | APIでは得られない買い手の入力が必要 | 人間へハンドオフ |
| `requires_buyer_review` | 買い手のレビューと承認が必要 | 人間へハンドオフ |
| `unrecoverable` | 対象リソースがもう存在しない | 新セッションで再試行 |

`path` フィールドにはRFC 9535のJSONPathが入り、例えば `$.items[1]` のようにどのフィールドの問題かを機械可読に指せます。セッションのTTLは指定がなければ作成から6時間です。

### 人間へのハンドオフの2経路

`requires_buyer_input` や `requires_buyer_review` で人間の出番になったとき、UCPには2つの継続経路があります。

1つ目はリダイレクト。事業者は `requires_escalation` を返すとき、`continue_url` の提供がMUSTです。エージェントは買い手をそのURLへ誘導し、人間がブラウザで残りを済ませます。

2つ目がEmbedded Checkout Protocol（ECP）です。名前が紛らわしいのですが、これは先ほど4トランスポートの1つとして挙げたEPの上に定義された、チェックアウト専用の仕様です。ホストであるエージェントアプリが事業者のチェックアウトUIをiframeに埋め込み、postMessage上のJSON-RPC 2.0で双方向にやり取りします。単なる画面の埋め込みではなく、委譲の契約が3段階で絞り込まれる設計です。事業者が許可する範囲 `config.delegate`、ホストが要求するURLパラメータ `ec_delegate`、ハンドシェイク `ec.ready` で確定する範囲が、それぞれ前段の部分集合になります。決済credentialの取得のような敏感な操作はこの契約で明示的に委譲された場合だけ可能で、仕様には「トリガーがEmbedded Checkout起点の場合、サイレントなトークン化は固く禁止」とあります。

<https://ucp.dev/2026-04-08/specification/embedded-checkout/>

エスカレーションを失敗扱いにせず、状態機械とUI委譲のプロトコルまで用意して受け止めています。エージェント決済の実務で一番つまずくのは「全自動で終わらないケース」なので、ここが最初から仕様化されているのは実装者として助かるはずです。

ここまでがUCPのコアです。事業者はプロファイルで機能を公開し、エージェントとの積集合から有効なcapabilityを決め、チェックアウトを事業者主導の状態機械として進める。決済と認可は、この上に外部仕様として合成されます。それを次で見ます。

## 決済はどこにいる：payment handler と AP2 mandate の接点

ここまで決済の話がほとんど出てきませんでした。それもそのはずで、UCPのコア仕様は決済手段を定義していません。Google Developers Blogの原文がこの設計を一言で表しています。

> UCP models a unique payments architecture, separating what consumers use to pay (instruments) from payment handlers (payment processors)

<https://developers.googleblog.com/under-the-hood-universal-commerce-protocol-ucp/>

消費者が何で払うかを表すinstrumentと、誰が決済を処理するかを表すhandlerを分離し、コアはその接続規約だけを定めます。具体的な決済方式は、各プロバイダが自分のドメインで handler 仕様を公開します。Google Payなら `com.google.pay`、Shop Payなら `dev.shopify.shop_pay`。事業者はプロファイルとチェックアウト応答で「受け入れるhandler」を広告し、エージェントはその中から選んで、そのhandlerのspecに従ってcredentialを作ります。

この裏で動くのが、冒頭の4役のうちCredential ProviderとPayment Service Providerです。仕様はこの決済モデルを3つの二者間信頼で説明します。事業者とCP/PSPの間には契約とAPIキーという既存の関係がある。プラットフォームはCPと対話してトークンを得る。プラットフォームはその結果だけを事業者に渡す。決済実行の全体をシーケンスにするとこうなります。

導入の観点を1つだけ先に置いておくと、CPと直接対話してトークンを取得するのは主にプラットフォーム側です。事業者側で新しく必要になるのは、受け入れるpayment handlerの宣言と、そのhandlerが発行するcredentialを処理できるPSPまたは決済ゲートウェイの統合。イシュアーや国際ブランドまで話が広がるのかどうかは、AP2のmandateの分業を見てから整理します。

チェックアウト確定時のcredential提出はこうなります。

Complete Checkout リクエスト（抜粋）

```
{
  "payment": {
    "instruments": [
      {
        "id": "pi_gpay_5678",
        "handler_id": "8c9202bd-63cc-4241-8d24-d57ce69ea31c",
        "type": "card",
        "selected": true,
        "credential": {
          "type": "PAYMENT_GATEWAY",
          "token": "examplePaymentMethodToken"
        }
      }
    ]
  }
}
```

`handler_id` が事業者の広告したhandlerを指し、`credential` の中身はプロトコルからは不透明です。解釈方法はhandler仕様だけが知っています。credentialはプラットフォームから事業者への一方向にだけ流れ、事業者が応答にcredentialを書き戻すことは禁止されています（MUST NOT）。さらにbindingという考え方で、credentialは対象の事業者と取引コンテキストに束縛されることが想定されています。UCPコアはcredentialの中身を解釈しないので、束縛の具体的な暗号方式を定義するのはhandler側です。狙いは、事業者A宛のcredentialを横取りして事業者Bで使うリプレイ攻撃を防ぐこと。

### AP2 mandate が差し込まれる正確な位置

AP2を知っている読者にとっての本命はここです。拡張 `dev.ucp.shopping.ap2_mandate` がcheckoutに合成されると、フローの2箇所に署名が差し込まれます。

1つ目は事業者側の署名です。checkout応答の `ap2.merchant_authorization` に、JWS Detached Content形式（`<ヘッダ>..<署名>`）で「この条件で売る」という事業者の署名が入ります。署名対象はcheckoutペイロードから `ap2` フィールド自身を除いたものをJCS（RFC 8785）で正準化したもの。アルゴリズムはES256系で、検証鍵はプロファイルの `keys` から `kid` で引きます。

2つ目がユーザー側の承認です。プラットフォームはComplete要求の `ap2.checkout_mandate` に、SD-JWT+kb形式のverifiable credentialを入れます。kbはkey bindingの略で、提示者が秘密鍵を所持している証明が付きます。このmandateは、事業者署名込みの完全なcheckout応答を内包することがMUST。つまり「事業者がこの条件で売ると署名したもの」に対して「ユーザーが買うと承認した」という二重の証跡が、1つのcredentialに畳み込まれます。

そして強制力もあります。negotiationでAP2拡張が有効化されたセッションでは、mandateなしのComplete要求は `mandate_required` エラーで拒否されます。高額取引だけの追加オプションではなく、AP2が有効になったチェックアウトセッションでは常に必須です。

<https://ucp.dev/2026-04-08/specification/ap2-mandates/>

AP2にはもう1つ、Payment Mandate、つまり支払い自体の承認があります。公式ドキュメントの整理では、プラットフォームはComplete要求に2つのmandateを提出し、検証者が分かれます。Checkout Mandateを検証するのは事業者、Payment Mandateを検証するのは決済プロセッサです。Payment Mandateは `ap2` フィールドではなくpayment instrumentのcredentialとして運ばれ、UCPからは不透明なまま決済側へ流れます。UCPがプロトコル層で意味を定義するのは、何を買うかの承認であるCheckout Mandateまで。この分業が見えると、冒頭の「UCPは決済プロトコルではない」が具体的な線引きとして腑に落ちるはずです。効果も明快で、mandateはチェックアウトの状態に暗号的に紐づくため、別取引への転用や署名後の条件改変を検知できます。再送や重複実行まで防ぐには、実装側での一回限りの消費制御が別途必要です。

<https://ucp.dev/documentation/ucp-and-ap2/>

### mandateはイシュアーまで届く必要があるのか

カードレールにこの仕組みを載せるとき、関係者はどこまで広がるのか。鍵は先ほどの分業です。Checkout Mandateの検証は事業者、Payment Mandateの検証は決済プロセッサ。そしてPSPがPayment Mandateを検証した後も、カード決済は従来どおり3DSとイシュアーのオーソリを通せます。つまり初期導入では、イシュアーや国際ブランドがmandateを認識する必要はありません。

2つの判定は役割が違います。mandate検証が確かめるのは「このエージェントに、この支払いを試行する権限があるか」。イシュアーのオーソリが確かめるのは「このカード口座から、実際に支払ってよいか」。両方を通過条件にするので、mandateの導入で本人認証や与信判断が緩むことはありません。既存カードレールの前段に、エージェントへの委任確認が1段追加されるだけです。

ただし、この構成ではmandateの効力は事業者とPSPの範囲に閉じます。イシュアーにはmandateが届かないので、委任の上限やスコープをオーソリ判断に使うことも、mandateを根拠に3DSを省略することも、責任分界を変えることもできません。判定が二重になるぶん成功率も下がり得ます。3DSチャレンジが出れば、自律フローは人間へのハンドオフに切り替わります。導入は2段階で考えるのが正確です。

| 段階 | ブランド・イシュアー対応 | できること |
| --- | --- | --- |
| PSPゲート型 | 不要 | mandateを検証した後、従来どおりカード決済を実行する。既存レールの前段に委任確認を追加できる |
| ネットワーク・イシュアー連携型 | 必要。オーソリ電文やトークン仕様の拡張を伴う | mandate由来の情報をイシュアーのリスク判断・追加認証・責任分界に使える |

UCPとAP2を使い始めるだけならPSPゲート型で成立します。ブランドやイシュアーの出番は、mandateを前段チェックではなく、カードネットワーク上の正式な認証や責任移転の根拠として扱いたくなった段階です。導入の律速は自社のAPI実装よりも、対応ウォレットと対応PSPの広がり、その先はネットワーク側のルール整備だと自分は見ています。

比較のためにOpenAI側も見ておくと、ACPの決済はStripeのSharedPaymentToken方式です。買い手の決済手段への権限を、1事業者・1通貨・上限額・有効期限でスコープしたトークンとして発行し、事業者はStripeのPaymentIntentで償還します。credentialを取引に束縛するという防御の考え方はUCPのbindingと共通ですが、UCPがプロセッサ中立のhandler差し替え式なのに対し、SharedPaymentTokenのドキュメントはStripe APIを前提に書かれています。決済の主導権をどこに置くかの設計差がはっきり出ている部分です。

x402との関係も整理しておきます。UCPの仕様原文にx402への直接の言及は見当たりません。接続点はAP2側にあります。x402はHTTPの402応答で支払い要求を提示し、支払い証明と引き換えにリソースを提供するプロトコルで、AP2には最初期の拡張としてx402が組み込まれています。GoogleがCoinbase、Ethereum Foundation、MetaMaskらと共同開発したA2A x402拡張がそれで、mandateで認可した支払いをステーブルコインで実行できる建て付けです。スタックとして積むと、UCPが取引を確定し、AP2が認可の証跡を作り、x402が暗号資産レールでの支払い実行を担う、という分担になります。UCPのpayment handlerはプロセッサ非依存なので、設計上はステーブルコイン決済もhandler仕様として差し込める余地がありますが、そうしたhandlerの実装例は現時点で確認できていません。

<https://cloud.google.com/blog/products/ai-machine-learning/announcing-agents-to-payments-ap2-protocol>

## 設計思想と残る課題

ここまでの解剖で見えた設計を、共同開発者であるShopifyのエンジニアリングブログが自ら言語化しています。曰く、モノリシックなプロトコルは複雑さでいずれ崩壊する。責務を分離し、明確なAPIを定義し、合成を可能にした層状のプロトコルが生き残る。TCP/IPがそれを証明しており、UCPは同じパターンを適用した、と。

<https://shopify.engineering/UCP>

名前空間ガバナンスについての表現も明快です。「ドメインを所有すれば名前空間を所有する。セキュリティは官僚制ではなく名前空間バインディングで担保する」。標準化委員会の投票を待たずに拡張が生えていくバザール型の進化を、最初から意図しています。人間へのエスカレーションについても「一部の取引には常に人間の参加が必要だ。ただし今日のエスカレーションは明日のAPIコールになるかもしれない」と、過渡期の現実を仕様に織り込んだことを認めています。

きれいな設計です。ただ、批判もすでに具体的です。独立系のレビューとしてまとまっているDevesh Shetty氏の分析から、自分が重要だと思う4点を拾います。

<https://deveshshetty.com/blog/universal-commerce-protocol/>

1つ目はエスカレーションのパラドックス。欧州ではPSD2が強力な顧客認証SCAを要求するため、実際のチェックアウトの多くが `requires_escalation` に落ちます。「完全自律チェックアウト」のハッピーパスがむしろ少数派になりかねません。同氏はこれを「自動化の約束とエスカレーションの現実の緊張関係こそ、このプロトコル最大の未解決問題」と呼んでいます。

2つ目はレート制限の不在。仕様はHTTP 429に言及するだけで、標準のクォータもバックオフも定義していません。事業者ごとに独自ルールが生えれば、せっかく潰したN×N問題が運用面で再発します。

3つ目は並行性制御。PUTによるセッション更新は全体置換で、ETagのような楽観ロックがありません。埋め込みUIで買い手が配送方法を変えた瞬間にエージェントが商品数を更新したら、後勝ちでどちらかが消えます。マルチアクター前提のプロトコルとしては素朴な穴です。

4つ目が一番刺さりました。採用の必然性の問題です。OAuthが普及したのは「対応しないとユーザーのアカウントにアクセスできない」という存立に関わるレバレッジがあったから。UCPが提供するのは「すでにできることが少し楽になる」という漸進的な利便性で、構造が違います。UCPとACPの両対応コストを払う事業者が十分なボリュームを生まなければ、「標準は今まで通り各社の独自REST API」という25年来の現状が続くだけ、という同氏の指摘は、プロトコルの筋の良さと普及が別問題であることを正確に突いています。

最後に、2026年7月時点で確認できる現在地を置いておきます。

稼働状況。米国では適格な事業者が、Google検索のAI ModeとGeminiアプリ内でのチェックアウトを有効化できる状態です。決済はGoogle Payで、PayPal対応が予定と発表されています。初期パートナーにはNike、Sephora、Target、Ulta Beauty、Walmart、Wayfairが並び、ShopifyマーチャントからはFentyやSteve Maddenが参加しています。Googleは複数事業者を横断するUniversal Cartも発表しており、2026年夏に米国のSearchとGeminiで展開開始、YouTubeとGmailが続くとされています。

<https://blog.google/products-and-platforms/products/shopping/google-shopping-cart/>

地理的には米国先行で、数か月内にカナダとオーストラリア、その後に英国という順序が発表されています。公式ロードマップには、その先の段階展開としてインド、インドネシア、中南米などが挙がっており、機能面ではロイヤルティと会員特典、クロスセル用のcapabilityが、vertical面ではFood（レストランのメニュー発見からチップ・配達指定込みのチェックアウトまで）とLodging（ホテルの部屋選択・予約・支払いスケジュール）の新仕様が業界パートナーと開発中とされています。日本への言及は現時点で見つけられませんでした。

<https://ucp.dev/documentation/roadmap/>

仕様の側は、v2026-01-11、v2026-01-23、v2026-04-08と3リリースが出ており、v2026-04-08が最新です。cart capabilityの追加や拡張の複数親対応、payment handlerの強化が入っています。発表からわずか3か月で2回のバージョンアップという速度感です。

<https://github.com/Universal-Commerce-Protocol/ucp/releases>

なお、Shopifyが全ストアでUCPをデフォルト有効化したという情報が複数の媒体で報じられていますが、Shopify公式ページでの一次確認は取れていません。OpenAIが2026年3月にChatGPTのInstant Checkoutを終了しACPを商品発見側に軸足を移したという件も、報道ベースです。この2つは裏取りの温度感込みで受け取ってください。

自分がウォッチしているポイントは2つです。

1つ目は、GoogleとShopify以外からの拡張が出てくるか。名前空間ガバナンスの設計が正しかったかどうかは、`dev.ucp.*` 以外のcapabilityが実際に流通するかで判定できます。兆候はすでにあって、NVIDIA公式のリファレンス実装集であるAI Blueprintsのリテール版が、独自にUCP仕様のドキュメントを保持し始めています。

2つ目は、エスカレーション率です。実トラフィックのうち何割が `requires_escalation` に落ちるのか。この数字が下がっていくなら「今日のエスカレーションは明日のAPIコール」というShopifyの見立てが正しく、下がらないならUCPは人間のチェックアウトへの誘導路として使われ続けることになります。エージェント決済を作る側にとっては、ここが事業設計の分水嶺です。

## まとめ

タイトルの問いに答えます。UCPが標準化しているのは決済レールではなく、エージェントが事業者を発見し、双方の機能を擦り合わせ、購入条件を確定し、人間へのハンドオフを含めて注文を成立させるまでの、取引の境界です。設計の中心は4点。

* 発見。事業者は `.well-known/ucp` で対応サービス・capability・payment handler・検証鍵を自己記述し、HTTPS必須・リダイレクト禁止・キャッシュ必須の配信規律で信頼の起点を守る
* 交渉。有効になる機能とバージョンは、双方の宣言の積集合から事業者が選ぶ。AP2による決済承認が使えるかどうかもこの結果で決まる
* 取引。チェックアウトは事業者が状態を握る状態機械で、エージェントはstatusとmessagesを読んで次の手を選ぶ観測者
* 分離。決済処理と認可はコアの外に置かれ、payment handlerとAP2 mandateとして合成される

概念記事の「共通言語」で止まっていた理解が、この機構に結像していれば、この記事の目的は達成です。設計は筋が通っています。残る問いは普及で、実取引のうちどれだけがエスカレーションに落ちるのか、事業者が独自APIに加えてUCPへ対応するだけの取引量が生まれるのか。この2点を引き続き見ていきます。
