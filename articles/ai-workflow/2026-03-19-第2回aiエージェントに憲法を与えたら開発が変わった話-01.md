前回は「AIと一緒にゲームプラットフォームを2ヶ月で作った」全体像をお伝えしました。今回はその中核にあった仕組み、AIに**ルールを与えて行動を制御する**という体験について書きます。

## CLAUDE.md という名の1行メモから始まった

Claude Codeには .claude/ というディレクトリにプロジェクトの設定を置ける仕組みがあります。その中心にあるのが CLAUDE.md。AIがセッション開始時に最初に読むファイルで、いわば「このプロジェクトではこう振る舞ってね」という指示書です。

最初の CLAUDE.md はこんな感じでした。

> TypeScript monorepo。pnpm lint:ts で型チェック。

……これだけ。

「まぁ、あとはAIが空気読んでくれるでしょ」くらいの気持ちです。実際、最初のうちはこれで十分でした。AIは賢いので、コードベースを読めばだいたい文脈を理解してくれます。

問題は、プロジェクトが大きくなるにつれて「空気を読む」では足りなくなったことです。

## 事故のたびにルールが"生えた"

ルールファイルが増えた経緯を時系列で振り返ると、ほぼすべて **「やらかし」の直後** に生まれていることがわかります。

### 事件1: サイレントフォールバック問題

ある日、本番環境でユーザーのデータが一部欠落しているのに気づきました。調べてみると、AIが書いたコードの中に「エラーが起きたらnullを返す」という処理があった。

```
// AIが書いたコード（イメージ）
try {
  return await fetchUserData(id);
} catch (e) {
  return null;  // ← これが地獄への入口
}
```

エラーが起きても黙ってnullを返すので、問題が起きていることに誰も気づけない。ログにも残らない。本番で静かにデータが消えていく。

これはAIコーディングで最も頻繁に発生するアンチパターンだと後から知りました。AIは「エラーハンドリングしておきますね」と言って、善意でcatch句を書いてくれるのですが、その結果エラーが握り潰される。

この事件の翌日に生まれたのが no-silent-fallback.md というルールファイル。

**「エラーを黙って握り潰すな。このエラーが本番で起きたとき、我々は気づけるか？ 気づけないなら、そのフォールバックを削除しろ」**

### 事件2: コピペロジックの罠

プラットフォームにBAN機能を実装したときのこと。AIに「ユーザーをBANする機能を作って」と依頼したら、きちんと動くコードが生成されました。

ところが後日、別のファイルでも同じBAN処理が必要になり、AIは最初のファイルからロジックをコピーして新しいファイルに貼り付けました。問題は、コピー元のコードがその後アップデートされていたこと。キャッシュ無効化のステップが追加されていたのですが、コピー先にはそれがない。

結果、片方のファイルからBANすると正常に機能するけど、もう片方からBANするとキャッシュが残ったまま……というバグ。

no-copy-paste-logic.md が生えました。

**「2つ以上のファイルで同じロジックを使うなら、共有モジュールに抽出しろ」**

### 事件3: 環境変数の全消失

これが一番派手な事故でした。

Cloud Runにデプロイする際、AIが --set-env-vars というフラグを使いました。このフラグは「指定した環境変数をセットする」のですが、実は既存の環境変数をすべて上書きするという挙動をします。つまり、新しく追加したい環境変数1つをセットした結果、それ以外の環境変数が全部消えた。

暗号化キー、Firebase設定、CORSオリジン、管理者APIキー、Redis接続先……すべて吹っ飛びました。

怒りのあまり、 cloudrun-env-safety.md が生えました。

**「--set-env-vars は絶対に使うな。--update-env-vars を使え。デプロイ前にスナップショットを取れ。本番環境の変更はサンドボックスで検証してから」**

## ルールの効果 — そして限界

ルールを整備した効果は明確でした。

**良かったこと:**

ルールファイルは「事故の記憶をプロジェクトに定着させる仕組み」として非常に効果的でした。人間のチームでも「ポストモーテム → 再発防止策」のサイクルを回しますが、AIエージェント相手でもまったく同じ構造が機能する。

でも、限界もあった。

ルールは「AIにお願い」しているだけです。AIは賢いのでほとんどの場合ルールに従ってくれますが、従わないこともある。コンテキストが長くなると忘れることもある。ルールの数が増えるほど、AIが全部を把握しきれなくなるリスクも高まる。

21本のルールファイルを前にして、ある時ふと思ったんです。

**「これ、ルールを書くんじゃなくて、ルール違反を物理的に不可能にしたほうがよくない？」**

この問いに対する答えが、後の回（第6回）で語る「Hooks（フック）によるガードレール」に繋がっていきます。でもそれはまだ先の話。

## 「ルールを書く」はAI開発の新しいスキル

2ヶ月の開発で一番の収穫は、「AIにルールを書く」こと自体がスキルだと気づいたことかもしれません。

良いルールには共通する特徴があります。

1. **具体的であること** — 「セキュリティに気をつけて」ではなく「Math.random() を使うな、crypto.randomUUID() を使え」
2. **理由が書いてあること** — なぜこのルールが必要なのか。事故の経緯が書いてあると、AIは文脈を理解して応用できる
3. **短いこと** — 長すぎるルールはAIが読み飛ばす。1ルール1ファイル、核心だけを書く

コードを書く力、設計する力と並んで、「AIへのルール設計力」が開発者にとって重要なスキルになりつつある。少なくとも自分の2ヶ月間はそうでした。

## ルールだけで守れない領域

ただし、正直に書くと、ルールだけでは不安な領域がありました。特にセキュリティ。

「セキュリティに注意しろ」というルールを書いても、AIは既知のパターンには対応できますが、自分が知らない攻撃手法には対応できません。だってAI自身もそのパターンを生成してしまう側なのだから。

そこで必要になったのが「**別のAIに攻撃させる**」というアプローチ。次回はその話、CodexセカンドオピニオンとAIセキュリティチームの実践についてお伝えします。
