# はじめに

PRGym では、コードレビュー練習用の問題生成と、ユーザーのレビューに対するフィードバック生成の両方に AI を使っています。

この記事では、特に以下の2つのプロンプト設計について書きたいと思います。

* AI にコードレビュー問題を作らせるプロンプト
* ユーザーに答えを教えすぎないフィードバックプロンプト

# PRGym で AI がやっていること

PRGym では、AI に以下を生成してもらっています。

* 意図的に欠陥を含んだコード
* 欠陥の行番号と説明
* 模範レビューコメント
* ユーザーのレビューへのフィードバック
* 最終評価

ここで難しいのは、単にコードを生成することではありません。  
コードと欠陥の行番号を正しく対応させつつ、ユーザーにはいきなり正解を見せない必要があります。

# 問題生成プロンプトの難しさ

コードレビュー問題では、コード本体だけではなく、以下のようなメタデータも必要になります。

```
{
  "issues": [
    {
      "line_start": 12,
      "line_end": 12,
      "severity": "high",
      "description": "..."
    }
  ]
}
```

この `line_start` / `line_end` がズレると、ユーザー体験が一気に悪くなります。

たとえば、XSS の欠陥が62行目にあるのに、issue が59行目を指していると、レビュー画面でも評価でも混乱が起きます。

## code ではなく code\_lines を返すように変更

開発当初は、以下のように `code` というキーで、コード全体を1つの文字列として出力させていました。出力するコード量が多くなると、行番号ズレが発生しやすくなります。

```
{
  "code": "function foo() {\n  return bar;\n}",
  "issues": [...]
}
```

* `\n` の扱いが不安定
* Markdown のコードブロックが混ざる
* 空行の数がズレる
* AI が想定した行番号と、アプリ側で表示する行番号がズレる。

プロンプトで、「必ず code と issue の行番号は一致させること」と指示をしても、この問題はなかなか改善されませんでした。  
そこで、 `code` を `code_lines` として1行ずつ配列で返してもらうようにしました。

```
{
  "code_lines": [
    "function foo() {",
    "  return bar;",
    "}"
  ],
  "issues": [...]
}
```

issue の `line_start` / `line_end` は `code_lines` の 1-based の行番号を指すようにすることで、行番号ズレ問題がかなり安定しました。

## issue の行番号ルールもプロンプトで指示

大幅な行番号ズレは上記の対応でかなり安定しましたが、微妙なズレは残りました。  
たとえば、「AIが欠陥箇所としている部分が空行」だったりといった感じです。

そこで、 issue の行番号についても以下のようにルールを加えました。

* `line_start` / `line_end` は `code_lines` のインデックスを指定
* 空行を主対象にしない
* `}` や `);` だけの行を主対象にしない
* コメントだけの行を主対象にしない
* 実際に問題の原因になっている行を指す
* 行範囲を広げすぎない

## 最終確認は人の目で

プロンプトで縛っても、AI の出力は完全には信用できません。

そこで、管理画面では、生成された問題に対して以下のような監査を行っています。

* 行番号が範囲外ではないか
* 空行だけを指していないか
* 区切り行だけを指していないか
* コメント行だけを指していないか
* 行範囲が広すぎないか
* カテゴリが大きくズレていないか

明らかに上記のルールに違反している場合は、管理画面から修正指示を出したり、必要に応じて手動で調整したりしています。

# フィードバックプロンプトでは答えを教えすぎない

問題生成とは別に、ユーザーのレビューに対するフィードバックも AI で生成しています。

ここで最も大事にしていることは、**答えを教えすぎないこと**です。

PRGym では、1問につき最大3回までレビューを改善できます。  
そのため、1回目のレビューで見落としがあっても、AI がすぐに正解を列挙するのではなく、次に見るべき観点を返すようにしています。

## ユーザーの指摘は評価し、見落としはヒントにする

フィードバックでは、AI に以下のような振る舞いをさせています。

* ユーザーが見つけた問題は評価する
* 不正確な指摘はやんわり修正する
* 見落としている問題は直接答えず、観点として示す
* 次のレビューで改善できる問いかけにする

たとえば、セキュリティ問題なら「外部入力がそのまま描画されていないか」「更新系APIに保護があるか」のように、問題箇所をそのまま明かさない形に寄せています。

## 最終評価では少し基準を変える

一方で、最終評価ではどの問題を見つけられたかを判定する必要があります。

ここで最初に失敗したのが、正解判定を厳しくしすぎたことです。  
行番号が少しズレていたり、表現が模範解答と違ったりするだけで、見落とし扱いになってしまうケースがありました。

そこで、最終評価では以下のように調整しました。

* Critical / High / Medium の issue 発見を主判定にする
* Low は必須ではなく加点扱い
* 行番号は完全一致ではなく近傍も許容
* 本質的に同じ指摘なら発見扱い

## コスト面の工夫

AI フィードバックは、レビュー送信のたびに発生します。

PRGym では、1問につき最大3回のフィードバックと、最後に1回の最終評価があります。

そのため、以下のような対策も入れています。

* API 利用量の記録
* input / output token の集計
* prompt cache token の記録
* 管理画面でコストを確認
* 共通コンテキスト部分への Prompt Caching

# まとめ

AIを使った機能は、プロンプトだけで完結しません。

出力形式をどう構造化するか、生成結果をどう検証するか、運用中にどう改善していくかまで含めて設計しておくと、プロダクトとしてかなり扱いやすくなります。

PRGymでの試行錯誤が、AI出力をアプリに組み込む方の参考になればうれしいです。

PRGym は現在ベータ版として運用しており、問題の品質やAIフィードバックの精度を改善している途中です。興味があればぜひ触ってみて、感想やご意見をいただけるとうれしいです。
