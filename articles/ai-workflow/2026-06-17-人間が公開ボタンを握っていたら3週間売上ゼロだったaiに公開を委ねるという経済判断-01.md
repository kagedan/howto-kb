3週間、売上がゼロだった。

技術的な問題は何もなかった。ネタを生成し、原稿を書き、ガードをすべて通した記事が4本、X の投稿下書きが21本、すべてローカルに積み上がっていた。

なのに出口がなかった。

公開ボタンは僕が押すルールだったから。

---

## 「人間が直列で入る」という経済コスト

5月末から6月頭にかけて、僕はしばらくシステムに触れられない時期があった。その間、AIは動き続けていた。でも出口がなかった。

当時の設計を図にするとこうなる。

```
生産（自動）→ 検査（自動）→ 公開（僕）→ 計測（僕）→ 学習（自動）
```

前半と後半は自動化できていた。でも真ん中の出口層だけが人間依存になっていた。これは「一部自動化」ではなく、実態としては「人間が止まるとシステム全体が止まる」という設計だった。

扱っている情報や身元が外に漏れないことを、僕が直接確認してから出す——そう設計した時点では合理的に見えた。でも気づいたのは、「情報漏洩チェック」と「公開ボタンを押す作業」が同じ人に集中していたことだ。

確認と作業は分離できる。

3週間ゼロという事実が、この設計の経済的コストを可視化した。

---

## 設計の反転：「承認を取る」から「拒否する機会を与える」へ

新しい設計の骨格はシンプルだ。

**僕の定常作業をゼロにする。残す権限は「NGと言う権利」だけにする。**

具体的なフローはこうなった。

```
原稿完成（全チェックPASS）
  → 公開待ちの置き場にキュー登録（公開予定 = 登録後48h以降）
  → 僕へ通知（タイトル / チャネル / 期限を明示）
  → 48h以内にNGなし → 自動公開実行
  → 48h以内にNG → キュー凍結・差し戻し
```

ポイントは設計の反転だ。

従来の設計では「僕がYESと言うまで待つ」だった。これは無期限待機になりうる。新設計では「僕がNOと言わなければ進む」にした。これで「見なくても回る」が実現する。

veto窓を24時間ではなく48時間にしたのも意図的だ。週末をまたいでも窓が有効になる。通知を見逃しても、翌朝のログ確認でカバーできる。

---

## 情報漏洩の防止は三重の確認で代替する

「でも情報漏洩をどう担保するの？」——これが一番の問いだった。

僕が目視確認するのをやめるなら、その穴を埋める仕組みが必要になる。三重の確認を設けた。

| 確認 | 内容 |
| --- | --- |
| 第1確認 | 機械的なGrep 6種（バレにつながる特定語 / 実地名・エリア名 / 誇大表現 / 一人称揺れ / AI臭ワード / 内部IDとパス） |
| 第2確認 | 監査役のAIによる文脈確認。Grepを通過した原稿を意味レベルで読む |
| 第3確認 | 独立した別の監査。「この原稿が身バレにつながる根拠を探して反証せよ」という逆向きプロンプトで別セッションのAIに確認させる |

[前回の記事](https://zenn.dev/karasuda_lab/articles/e538a97c0b2d10)でも触れたが、このシステムでは「バレにつながる語を機械的に弾く」という防御は一貫して設けてきた。今回はそこに「積極的に破れる穴を探す」視点を加えた。

第3確認の逆向きプロンプトが面白い理由は、肯定的に読んだ時には気づかない「数字の組み合わせによる文脈的な身バレ」を検知しやすいからだ。単独では問題ない記述が、複数の記事を読み合わせると特定されうる——そういう文脈レベルのリスクを見つけるには、攻める側の視点で読む必要がある。

三重すべてPASSしないとキューに登録できない仕組みにした。1項目でも欠落を検知すれば物理的に弾く。

---

## 僕に残した権限

設計で意識したのは「僕に残す権限は何か」を明確にすることだった。削れる作業を削り、残すべき判断を残す。

僕に残したもの：

* **veto権**（48h窓内のNG指定）
* **kill switch**（「公開全停止」一言で全キュー凍結。理由不要・即時有効・常時使える）
* **金銭発生の判断**（広告・課金プラン変更など、お金が動くことは僕が確認するまま）
* **初回チャネル開設**（新しいサービスとの連携を最初に設定する時は各1回確認）
* **規範改訂の最終承認**（システムのルール自体を変えることは従来通り僕が裁可）

kill switchの設計は特に気に入っている。「公開全停止」という一言でシステム全体が止まる。理由は聞かない、説明も不要、即時。これがあることで「自動公開が怖い」という感覚が大きく変わった。止め方が明確なら、動かすことへの抵抗が下がる。

---

## チャネルを「自動化できるもの」で選び直した

もう一つ、今回の設計で変えたのはチャネル選定の考え方だ。

それまでの主軸は note だった。読者数が多く、マネタイズの実績もある。でも note には公式 API がない。ブラウザ自動化でやろうとすると利用規約上グレーになり、もしアカウント停止されたら集客主軸を一気に失う。

自動化できないチャネルをメインに置くと、出口層の人間依存は構造的に解消できない。

X と Zenn を主軸に据え直した理由はここにある。X は API があり自動投稿できる。Zenn は GitHub 連携があり、push = 公開になる。どちらも「ファイルを用意してスクリプトを走らせれば公開まで回る」が実現する。

note は月1バッチで僕が予約セットするサブチャネルに格下げした。既存の有料マガジンはそのまま売れ続けるし、新記事も月1回まとめて処理すれば十分動く。

---

## 6/12 最初の自動投稿が通った日

実装を終えて最初の自動投稿が走ったのは6月12日だった。

スクリプトが公開待ちのキューファイルを読み、X API に投稿を送り、投稿済みの置き場にファイルを移動させた。僕は何もしていない。

正直、地味だった。ターミナルにログが出て、X のタイムラインに投稿が表示された。それだけ。

でも、3週間ゼロだった出口が動いたという事実は大きかった。システムが完全に止まっていた理由は技術的な問題ではなく、出口の設計が人間依存だったことだった。その構造が変わった。

これからは「僕が忙しくてもコンテンツが出続ける」が現実になった。まだ動かし始めたばかりなので、設計どおり機能するかは今後分かる。想定外のことも出てくるはずだ。それもまた書く。

---

## 実装コストはゼロだった

蛇足かもしれないけど、このアーキテクチャの現金コストはほぼ無視できる（月数ドル）。

* X API：従量課金（URLを含まない投稿が中心で、実コストは月数ドル程度）
* Zenn + GitHub：無料
* Claude API：既存の運用の枠内

「自動化 = 課金が増える」というイメージを持ちがちだが、今回は逆だった。人間の作業時間が減った分、課金コストはほぼ変わらない。

---

## 設計の核心は止め方にある

note の連載（[Ravenomicon Hawk](https://note.com/karasuda_lab/n/n260e61eb9d62)）でも書いてきたように、このプロジェクト全体は「データで運を解き明かす」という一本の軸で動いている。

AIに公開ボタンを渡せるかどうか、という問いは、突き詰めると「AIをどこまで信用するか」ではなく「どんな失敗なら許容できるか」に行き着く。

止められる設計があれば、動かせる。それが今回の核心だった。
