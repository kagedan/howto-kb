## はじめに

近年、生成AIの進展により、ソフトウェア開発の進め方は大きく変化しつつあります。特に、コード作成が必要とされるいわゆるM(製造)・UT(単体テスト)工程での支援においてAIを活用する取り組みが広がり、従来の開発プロセスとの組み合わせが注目されています。

その中でも、テスト駆動開発（TDD）とAIを組み合わせたアプローチは、品質と開発効率の両立という観点から関心が高まっています。

本記事では、AI支援型開発フレームワーク「Tsumiki」を用いて、Vibe Codingによる簡単なアプリケーション機能を実装してみました。

### この記事でわかること

* Tsumikiの概要
* テスト駆動開発（TDD）の考え方と進め方

### 想定読者

* AIを使った開発に触れてみたい方
* テスト駆動開発（TDD）に興味はあるが経験が少ない方
* Tsumikiというツールを初めて知った方

## Tsumikiとは

「Tsumiki」は、クラスメソッド株式会社が公開した、AI支援型のテスト駆動開発（TDD）を実現するためのオープンソースフレームワークです。

本フレームワークは、Anthropic社のAIコーディングツール「Claude Code」 を主対象としつつ他ツールでも利用可能で、AIを用いた開発プロセスを体系化・効率化する役割を担います。

### 特徴

Tsumikiの最大の特徴は、AIとテスト駆動開発を組み合わせた「AITDD（AI-assisted Test-Driven Development）」というアプローチにあります。

このアプローチでは、以下の原則に基づいて開発が進められます。

* 仕様書ファースト
* テストファースト
* 品質ファースト

これにより、単に「動くコード」を生成するだけでなく、「仕様が明確で、テストによって保護された、保守可能なシステム」を構築することを目的としています。

### 開発プロセス

Tsumikiでは、AIを活用しながら以下のような一連の開発プロセスを実行します。

1. 要件定義
2. 設計
3. タスク分割
4. テスト駆動開発（TDD）による実装

特にTDDの工程では、以下のサイクルが採用されています。

* Red：テストの作成（期待する動作の定義）
* Green：テストを満たす最小実装
* Refactor：コードの改善
* Verify：要件との整合性確認

この一連の流れをAIが支援することで、開発効率と品質の両立を図る仕組みとなっています。

### コマンドについて

#### 概要

Tsumikiは以下の3つのコマンドで構成されています。

* kairo - 要件定義から実装までの包括的な開発フロー
* tdd - テスト駆動開発（TDD）の個別実行
* rev - リバースエンジニアリングのコマンド

今回は「kairoコマンド」を以下の手順で実行し、時計表示機能を作成していきます。

1. init-tech-stack
2. kairo-requirements
3. kairo-design
4. kairo-tasks
5. kairo-implement

## セットアップ

Tsumikiを動かすために必要な資材の配置を行ないます。

```
# 1. Tsumikiリポジトリのクローン
git clone --depth 1 https://github.com/classmethod/tsumiki.git /tmp/tsumiki

# 2. Tsumikiのコマンドを .rulesync/commands/ に配置
cp -r /tmp/tsumiki/commands/* .rulesync/commands/

# 3. コマンドでないファイルを削除
rm .rulesync/commands/tech-stack.md .rulesync/commands/test-optimization-patterns.md

# 4. GitHub Copilot 用に変換
npx -y rulesync generate \
    --targets copilot \
    --features commands \
    --simulate-commands
    
# 5. コマンドの重複候補を避けるためディレクトリ名を変更して移動
mv .rulesync/commands .rulesync/commands-source
```

この手順により、以下のようなディレクトリ構造となりました。

```
プロジェクトルート
├── .rulesync/
│   └── commands-source/
│       ├── *.md           ← Tsumiki からコピーしたファイル
│       └── dcs/
│           └── *.md
└── .github/
    └── prompts/
        └── *.prompt.md     ← GitHub Copilotが読み込むファイル
```

## 実行結果

Kairoコマンドは全てGitHub Copilotのエージェントで実行していきます。

### 1. init-tech-stack

実行後、init-tech-stack.prompt.mdの中身が読み込まれ、アプリケーション作成に必要な技術スタックを選定するためにいくつか質問をされます。  
一部の設問を除いてほとんどが選択式で回答することができます。

> Q：プロジェクトのタイプを教えてください  
> A：Webアプリケーション- ブラウザで動作するアプリケーション

> Q：開発チームの規模を教えてください  
> A：個人開発- 1人で開発

> Q：予定している開発期間はどの程度ですか？  
> A：数時間で作成できるような簡易的なアプリケーション。ストップウォッチやアラーム機能が付いた時計などを作成したい。

・・・

全ての質問への回答後、回答の要件に基づいて使用するフロントエンドやバックエンド、DB、開発環境を選定し、技術スタックが一覧化された定義ファイル（.md）が作成されます。  
私は、構築になるべく時間をかけず、簡単な時計のアプリケーションを作成するよう回答しました。

### 2. kairo-requirements

```
/kairo-requirements 時計表示機能
```

実行後、init-tech-stackと同様にエージェントから要件を固めるための質問をされます。

> Q：この要件の作業規模について教えてください  
> A：軽量開発 - 必要最小限の要件定義、基本的なユーザーストーリーのみ、主要な受け入れ基準のみ、非機能要件は最低限

> Q：時計表示で必要な機能を選択してください（複数選択可）  
> A：12時間制 - 12時間制表示（AM/PM付き）,24時間制 - 24時間制表示,秒表示 - 秒まで表示,日付表示 - 日付表示

> Q：時計の表示スタイルは？  
> A：アナログ表示 - アナログ時計のようなビジュアル

・・・

今回の検証では時計表示機能のみを実装対象としました。

全ての質問に回答後、要件定義書が作成されます。

### 3. kairo-design

実装対象を時計表示機能に限定するため、以降は /[コマンド名] の後に「clock-display」を指定しています。

```
/kairo-design clock-display
```

実行後、設計書を作成する上で必要となる情報をエージェントが聞いてきます。

> Q：この設計の作業規模について教えてください  
> A：軽量設計 - 必要最小限の設計、基本的なアーキテクチャとデータフロー、主要な型定義のみ

> Q：既存実装の詳細分析が必要ですか？  
> A：不要 - 要件定義と設計文書のみで設計を作成

> Q：コンポーネントの分割方針は？  
> A：複数コンポーネント - ClockFace、ClockHands、ControlPanel等に分割

・・・  
回答後、アーキテクチャ設計書・データフロー図（.md）が作成されました。

以下はデータフロー図ですが、軽量設計を指定したにもかかわらず、比較的詳細に書かれている印象です。  
![](https://static.zenn.studio/user-upload/17cfe6fe101f-20260327.png)

### 4. kairo-tasks

```
kairo-tasks clock-display
```

実行後、実装タスクの分割粒度や優先度に関する質問がなされます。

> Q：このタスク分割の作業規模について教えてください  
> A：詳細タスク分割（推奨） - 詳細な実装手順、包括的なテストケース、完全な依存関係分析、UI/UX要件含む

> Q：タスクの粒度について、1日（8時間）単位で問題ないでしょうか？  
> A：1日単位で問題ない - 8時間単位のタスク分割

> Q：実装の優先順序について教えてください  
> A：基盤→機能→UI - 環境構築→コア機能→UI実装の順

・・・

回答後、10個のタスクが洗い出され、各タスクファイル（.md）が作成されました。

#### Phase 1：基盤構築（2タスク）

1. TASK-0001：プロジェクト初期設定（Vite + React + TypeScript + Tailwind CSS）
2. TASK-0002：型定義とユーティリティ関数の実装

#### Phase 2：コア機能実装（2タスク）

3. TASK-0003：useClockカスタムフック実装
4. TASK-0004：時間計算ユーティリティ実装

#### Phase 3：UI実装（5タスク）

5. TASK-0005：Clockメインコンポーネント実装
6. TASK-0006：ClockFaceとClockNumbers実装
7. TASK-0007：ClockHands（時針・分針・秒針）実装
8. TASK-0008：DateDisplay実装
9. TASK-0009：TimeFormatToggle実装

#### Phase 4：統合とテスト（1タスク）

10. TASK-0010：統合テストとリファクタリング

作成されたタスクファイルを見てみると、必要なパッケージのインストールや作成するモジュールの中身まで細かく書かれています。  
![](https://static.zenn.studio/user-upload/5c9d0547b8b1-20260327.png)

### 5. kairo-implement

kairo-tasks で作成された10個のタスクを1から順に実行していきます。  
本記事では TASK-0001 のみ記載し、他のタスクは割愛します。

```
/kairo-implement TASK-0001
```

実行後、エージェントが TASK-0001.md の内容に沿って実装を進めてくれます。  
基本的にソースコードはエージェントが書いてくれますが、パッケージインストールやディレクトリの新規作成/移動といったターミナルでの実行が必要なコマンドはチャットにて実行可否を問われます。  
テスト時にエラーが出た際は原因を解析し、特定したソースコードの修正と内容に応じてTASKファイルの中身もエージェントが修正してくれます。

エージェントチャット上では各タスクをさらに細かい粒度で分割したTo-Doが一覧化され、進行状況を確認できます。（以下はTASK-0006）  
![](https://static.zenn.studio/user-upload/dd9a506921eb-20260327.png)  
テスト駆動開発（TDD）の基本的な考え方に基づき、 Red → Green → Refactor → Verify の順に実装しています。

### アプリケーションの完成

![](https://static.zenn.studio/user-upload/bd3ec4817311-20260327.png)  
時計表示のアプリケーションを作成することができました。  
所要時間は約4時間です。  
日時と曜日が表示されており、1秒ごとに秒針が動くいわゆる時計の挙動をしています。

![](https://static.zenn.studio/user-upload/4e6d5675c2ff-20260327.png)  
また、「表示形式」のトグルを切り替えると、24h表示にもすることができます。（デジタル時計ならともかく、アナログ時計では実用上必要ではありませんでした。）

## まとめ

Tsumikiで提供されている5つ（init-tech-stack含む）のKairoコマンドをGitHub Copilotのエージェントで実行し、テスト駆動開発（TDD）で時計表示機能のあるアプリケーションを作成しました。

## 感想

今回のようなTsumikiを用いた開発では、要件定義→設計→タスク分割→テスト→実装というプロセスが明確に分離されているため、一般的なAIエージェントによる開発（プロンプトベースで直接コード生成）と比較して、開発の進め方が構造化されている点が大きな違いだと感じました。  
特に、テストを起点に実装を進めることで、「動くコード」ではなく「要件を満たすコード」を意識した開発になりやすい点は、TDDとAIを組み合わせるメリットだと感じました。

また、本検証ではTsumikiリポジトリのMarkdownファイルをそのまま活用しましたが、プロジェクト毎の品質要求に応じてMarkdownファイルのみをカスタマイズすることで、外部・内部品質のさらなる向上が見込めるのは大きな利点だと感じました。

一方で、1. init-tech-stack（技術スタックの選定）実行時、「新しい技術の習得についてどう考えますか？」という問いに「積極的に新技術 - 最新技術でチャレンジしたい」と回答したにもかかわらず、エージェントが選定した技術スタックのバージョンが最新ではありませんでした。今回はそのまま開発を進めましたが、init-tech-stack実行後に作成されるtech-stack.mdを確認して必要に応じて中身を修正する、等の手作業が必要になります。

また、24h表示のアナログ時計が作成されてしまった原因として、kairo-requirement実行後のエージェントからの質問に対し、「24h表示」かつ「アナログ時計」と誤って指定しながらも出力された要件定義書を確認しなかったことにあります。

そのため、生成AIによって作成された成果物は必ず人間の目で確認する、ということを怠ってはいけないなと実感しました

本検証のように小規模なアプリケーション開発ではなく、大規模なシステム開発への導入はまだまだ難しいと思いますが、将来的なAI-nativeな開発の実現に可能性を示すことができたのではないかと考えます。
