```

```

## 環境構築でハマった問題

ここからは、実際に動かして直面した問題たち。**公式ドキュメントには書いてない**ので注意。

### 問題 1: エージェントが重複する

**現象**:

```
生成されたエージェントを確認すると、同じ人物が複数登録されている：
- 「田中太郎」
- 「田中太郎（社長）」
- 「田中社長」
```

**原因**:  
GraphRAG によるエンティティ抽出時、**同じ実体を別々のノードとして認識**してしまう。

例えば：

これらが別々のエンティティとして扱われ、結果的に**エージェントが重複**する。

**影響**:

* シミュレーション精度が低下
* トークン消費量が増加
* レポートの一貫性が損なわれる

**現状**:  
この問題は GitHub で既に報告されており、修正が提案されています。

**回避策**:

1. 入力データを事前に正規化（人名・役職を統一）
2. 小規模シミュレーションでテスト
3. 重複チェックを手動で実施

### 問題 2: Zep 知識グラフ構築で 404 エラー

**ここが一番ハマりました。**

**背景：なぜローカル Zep を試したか**

Zep Cloud の無料枠は月間 1,000 リクエスト程度。  
MiroFish のグラフ構築は 1 回で数百リクエストを消費するため、**数回試すだけですぐに枯渇**してしまう。

「これでは開発できない」と考え、Docker で OSS 版をローカル起動した：

```
docker run -d -p 8000:8000 zepai/zep:latest
```

ローカルが動けば、コストを気にせず開発できる——そう思っていた。

**現象**:

```
Graph Build 実行時に 404 エラー：
headers: {'content-type': 'text/plain; charset=utf-8', ...}, 
status_code: 404, body: 404 page not found
```

エラー発生箇所：`client.graph.create(name=graph_name)`

**トラブルシューティング（原因究明）の過程（3 回の試行錯誤）**:

#### 第 1 試行：API パス suffix の修正

`.env` の設定：

```
ZEP_API_URL=http://localhost:8000
```

`http://localhost:8000/api/v2/health` は存在するが、`/api/v1` は存在しない。  
SDK に `/api/v2` プレフィックスを明示する必要があるのでは？

**操作**:

```
ZEP_API_URL=http://localhost:8000/api/v2
```

**結果**: 失敗。エラー変わらず。

---

#### 第 2 試行：HTTP リクエストをキャプチャ

`test_zep_trace.py` スクリプトを作成し、httpx のデバッグログを有効化して SDK の通信を追跡。

**発覚した事実**:  
Zep Cloud SDK が\*\*仕様がかなり強引（クセが強すぎる）\*\*だった。

* `base_url` に何を渡しても、内部で自動的に `/api/v2/` を結合してしまう（繋げてしまう）
* 第 1 試行の修正により、実際の URL がこうなった：

```
POST http://localhost:8000/api/v2/api/v2/graph/create
HTTP/1.1 404 Not Found
```

**二度手間**でした。

---

#### 第 3 試行：パスを元に戻して Zep CE コンテナを分析

`ZEP_API_URL` を `http://localhost:8000` に戻して再キャプチャ：

```
POST http://localhost:8000/api/v2/graph/create
HTTP/1.1 404 Not Found
```

URL は正しくなったが、依然 404。

`docker ps` と `docker inspect` で確認：

* 実行中のイメージ：`zepai/zep:latest`（Zep Community Edition、Zep CE）

**根本原因**:  
MiroFish のバックエンド（`graph_builder.py`）は `zep-cloud==3.13.0` Python SDK に深くバインドされている。

* **Zep Cloud**: 自動知識グラフ抽出・進化（GraphRAG）機能あり
* **Zep OSS 版（ローカル）**: 基礎ベクトルメモリ機能のみ、`/api/v2/graph/create` インターフェースなし

**重要**: これは MiroFish の制限ではなく、Zep 自体の仕様。  
Zep 公式も「グラフ機能はクラウド版を利用」を推奨しており、OSS 版はあくまで開発・テスト用。

**私のケース**: クラウドの無料枠がすぐ枯渇するため、ローカルで試そうとしてハマった。

---

#### 最終解決策

ローカルオープンソース版（OSS 版）ではグラフ機能を提供できないため、**Zep Cloud 環境に切り替える**しかない。

**操作**:

1. `backend/app/config.py` を修正し、未設定時に `http://localhost:8000` に接続するハードコードを削除
2. `.env` から `ZEP_API_URL` 変数を完全削除

   * SDK をデフォルトで `https://api.getzep.com` に接続させる
3. ユーザーに Zep 公式サイトで無料登録を依頼

   * 取得した API キーを `.env` の `ZEP_API_KEY` に設定

```
# .env の最終設定
ZEP_API_KEY=your_real_zep_api_key  # Zep 公式サイトで取得
# ZEP_API_URL は削除（SDK がデフォルトでクラウドに接続）
```

**結果**: 環境回復、プロジェクトがクラウド環境でシームレスにグラフシミュレーションを実行可能に。

**教訓**:

* MiroFish は Zep のグラフ機能を使用（クラウド推奨）
* Zep 公式もクラウド版を推奨しており、OSS 版は開発・テスト用
* クラウド無料枠はすぐ枯渇するため、本格的に使う場合は課金検討が必要

**関連ファイル**:

* `backend/app/graph_builder.py` - グラフ構築ロジック
* `backend/app/config.py` - Zep 設定

### 問題 3: API レート制限

**現象**:

```
ステップ 4 のレポート生成時にタイムアウトエラーが発生
```

**原因**:  
Zep Cloud の API レート制限に到達。

**対策**:

1. 無料枠でもシンプルなシミュレーションは可能
2. 40 ラウンド未満から始める
3. エラー発生時は再試行ではなく、キャッシュを確認

---

### 問題 4: 504 Gateway Timeout

**現象**:

```
最後のエージェントインタビュー時に"抱歉，发生了错误：Request failed with status code 504"
```

**原因**:  
LLM API のレスポンスが遅すぎる、またはトークン制限超過。

**対策**:

1. モデルを軽量なものに変更（qwen-plus → qwen-turbo）
2. シミュレーションラウンド数を減らす
3. タイムアウト設定を調整

## まとめ：これは何を変えられるか

MiroFish の本質は、**「未来を当てる」ことではなく、「未来を事前に体験する」こと**だと思う。

従来の予測モデル：

> 「データからすると、70% の確率で X が起こります」

MiroFish：

> 「この条件下で、人々がどう行動するかを 1000 回シミュレーションしました。結果、このようなパターンが見られました」

**違いは「プロセスの可視化」**。

政策立案者、研究者、クリエイター——誰でも「もしも」を安全に試せる。これが MiroFish の真の価値だ。

**ただし、現在は初期段階**。エンティティ重複や Zep Cloud 依存などの問題はあるものの、アクティブに開発が続いている。今後の進化に注目したい。
