最近、OpenClaw を使って営業管理用のAIエージェントを作っています。

ざっくり言うと、Slack から自然文で話しかけると、Notion のSFA/CRMを更新したり、商談チャンネルを作ったり、アポ獲得・商談後整理・成約・失注・請求まわりの処理を進めてくれるエージェントです。

たとえば、Slack でこう書きます。

* 「アポ取れた。来週火曜13時。事前調査もしておいて」
* 「商談終わりました。次は見積提示、金額は100万円」
* 「成約した」
* 「請求書を作って」
* 「このPDFを請求書DBに入れて」

すると、取引先・商談・案件・請求書などのNotion DBを更新し、必要ならSlackチャンネルやCanvasも作り、結果を元のスレッドに返します。

![](https://static.zenn.studio/user-upload/3866b36885e0-20260526.jpg)

一見すると「Slackで動く営業AI」なのですが、作っていく中で強く感じたのは、営業管理のような副作用だらけの業務では、LLMを賢い実行者として使いすぎないほうがいいということです。

LLMは自然言語を読むのが得意です。  
でも、Notion DBを更新する、Slackチャンネルをアーカイブする、請求書ステータスを変える、という処理を毎回その場で考えさせると怖い。

この記事では、OpenClaw上にSFA/CRMエージェントを作りながら考えた、営業管理AIエージェントの設計について書いていきます。

## 営業管理は自然言語と相性がいい

営業管理は、かなり自然言語と相性がいい領域だと思っています。

現場で発生する情報は、だいたい人間の言葉で飛んできます。

* 「A社と初回アポ取れました」
* 「B社は予算が合わず見送り」
* 「C社は来月から契約開始」
* 「D社の提案書はこのURLです」
* 「E社の請求、入金済みにして」

これを毎回SFAのフォームに入れるのは面倒です。  
特に、Slackで営業の会話が進んでいる場合、わざわざNotionやCRMに移動して項目を更新するのは地味に摩擦があります。

なので、Slackで言ったことがそのまま営業管理に反映される体験はかなり気持ちいいです。

ただし、便利さの裏側に怖さもあります。

* 同じ依頼でも、毎回処理が少し変わる
* 商談と案件を取り違える
* 失注理由がないのに失注処理してしまう
* Slackチャンネルを間違ってアーカイブする
* Notionに「やった風」のメモだけ残って、実際のDBは更新されない

営業管理は、ただのチャットではありません。  
外部システムに状態を持つ業務システムです。

ここをLLMの自由実行に寄せすぎると、実用では不安が残ります。

## OpenClawは入口、実行はスクリプトに寄せる

今回の構成では、OpenClawを「Slackなどの入口からエージェントを呼び出す土台」として使っています。

一方で、実際の業務処理はスクリプトとして固定しています。

エージェントは、いきなりNotionやSlackを自由に操作するわけではありません。

まず自然文を読み、何の依頼なのかを分類します。  
その後、決められたルーターが、決められたスクリプトを、決められた引数で呼びます。

たとえば、現在は次のようなintentを扱っています。

* `appointment`: アポ獲得、商談作成、必要なら事前調査
* `prepare_deal`: 商談準備
* `meeting_result`: 商談後整理、メモ追記
* `close_won`: 成約、案件化、案件Slack作成、商談チャンネルアーカイブ
* `close_lost`: 失注、理由記録、商談チャンネルアーカイブ
* `invoice_create`: 請求書作成
* `invoice_pdf`: PDF請求書の取込
* `invoice_update`: 請求書ステータス更新
* `sales_dashboard_update`: KPIダッシュボード更新
* `crm_search`: CRM横断検索

LLMに任せているのは、主に「このSlack発話は何をしたいのか」を読む部分です。

実行の組み立てはコード側に寄せています。

ここがかなり重要でした。

## Slack自然文はまずキューに積む

最初は、Slackから来た依頼をその場で処理すればよさそうに見えます。

でも実際に運用すると、Slackイベントには厄介な点があります。

* Slackイベントが再送されることがある
* 同時に複数の依頼が来る
* トップレベル投稿とスレッド返信で文脈が違う
* 処理に時間がかかると返信先が混ざりやすい
* 途中で失敗したときに再実行したい

そこで、Slackからの自然文依頼は、まず `channel_id + message_ts` 単位で永続キューに積むようにしました。

```
Slack event
  -> enqueue-slack-request.mjs
  -> state/slack-request-queue/jobs/*.json
  -> process-slack-request-queue.mjs
  -> handle-slack-message.mjs
  -> business script
  -> Slack reply
```

Slackイベントは再送されることがあるので、同じ依頼を二重に処理しない仕組みにしています。

この設計にしてから、かなり安心感が出ました。

## LLMは分類、固定ルーターは実行

このエージェントでは、Slack自然文をまず分類します。

分類結果は、ざっくりこういうJSONです。

```
{
  "intent": "appointment",
  "confidence": 0.92,
  "should_execute": true,
  "fields": {
    "company_name": "株式会社サンプル",
    "meeting_date": "2026-06-02",
    "meeting_time": "13:00",
    "prepare": true
  },
  "missing": [],
  "reason": "アポ獲得と事前調査の依頼"
}
```

大事なのは、このJSONをそのままコマンドとして信じないことです。

ルーター側で、intentごとに呼べるスクリプトを固定しています。

```
const SCRIPT_BY_INTENT = {
  appointment: 'capture-appointment.mjs',
  prepare_deal: 'prepare-deal-from-slack-channel.mjs',
  meeting_result: 'capture-meeting-result.mjs',
  close_won: 'close-won-deal.mjs',
  close_lost: 'close-lost-deal.mjs',
  invoice_create: 'create-invoice.mjs',
  invoice_update: 'update-invoice.mjs',
  crm_search: 'search-crm.mjs',
};
```

LLMがやるのは、自然文から構造を取り出すところまで。

実際にどのスクリプトを呼ぶか、どの引数を許可するか、どの条件なら止めるかは、コードで決めます。

たとえば、失注処理では失注理由がない場合は止めます。

```
「失注した」
  -> close_lost intent
  -> reason がない
  -> Notion更新もSlackアーカイブもせず、理由だけ確認
```

成約や失注のような危険操作は、現在のメッセージに明示語がある場合だけ実行します。  
前の投稿に「成約」と書いてあったから今回も成約扱い、のような継承はしません。

このあたりは、プロンプトで「気をつけて」と言うより、ルーターで物理的に制限したほうが安定します。

## Program Skillを業務単位に分ける

OpenClaw上では、SFA/CRMエージェントの能力を大きく2つのスキルに分けています。

1つ目は、Slack連携・オーケストレーション用の `sfa-crm`。

* 商談Slackチャンネル作成
* 案件Slackチャンネル作成
* オーナー招待
* Canvas作成
* チャンネルアーカイブ
* ステージに応じたチャンネル名同期

2つ目は、Notion CRUD用の `sfa-crm-notion`。

* 取引先作成・検索
* 商談作成・更新
* 案件作成・更新
* 商談後整理
* 問い合わせDB登録
* 請求書DB更新
* KPIダッシュボード更新

この分離は地味ですが効きます。

Notionを更新する処理と、Slackチャンネルを作る処理を混ぜると、どこで何が起きるのか追いづらくなります。

営業管理では、対象も状態も多いです。

```
取引先
  ├─ 商談
  │   └─ 商談Slackチャンネル
  └─ 案件
      └─ 案件Slackチャンネル
```

この構造をコードにも反映させています。

商談を作るなら、まずNotionの商談ページを作る。  
その後、商談ページIDと取引先ページIDを使ってSlackチャンネルをセットアップする。

成約なら、商談を成約にして、案件を作り、案件Slackチャンネルを作り、商談チャンネルをアーカイブする。

業務フロー単位でスクリプトを固定しておくと、LLMは「どのフローか」を選ぶだけで済みます。

## 1商談 = 1 Slackチャンネルにする

このエージェントでは、1商談につき1つのSlackチャンネルを作る設計にしています。

チャンネル名は、Notion DBのID名を先頭に入れます。

```
#deal-{ID名}-{取引先名}-{ステージ番号}-{ステージ名}
#case-{ID名}-{取引先名}-{サービス概要}-{ステータス}
```

たとえば、こういう形です。

```
#deal-deal-0123-sample-04-提案フェーズ
#case-case-0456-sample-notion顧問-契約準備中
```

ポイントは、ID名を手打ちしないことです。

チャンネル作成時には、NotionページからID名・取引先名・ステージを取得してチャンネル名を組み立てます。

Slackチャンネル名は、人間にとって見やすい一覧でありつつ、システムにとってもNotionと対応づけられる識別子になります。

これにより、Slackチャンネルから商談を逆引きできます。

「このチャンネルで `事前調査して` と言われたら、どの商談ページを更新すべきか」が決まります。

逆に言うと、SlackチャンネルとNotion商談の紐づきがない場合は止まります。  
これは不便に見えますが、営業管理では正しい挙動だと思っています。

対象が曖昧なままCRMを更新するほうが怖いからです。

## 返答はスクリプトの結果だけにする

副作用のある処理で、もうひとつ大事だったのが返答です。

LLMに自由に返答させると、「完了しました」と言っているのに実際にはNotion更新に失敗している、ということが起きえます。

そこで、Slackに返す文章は、基本的にスクリプトの実行結果から出すようにしています。

```
business script
  -> JSON result
  -> slack_reply / post_to_slack
  -> worker posts it
```

スクリプトが `slack_reply` を返したら、それをそのままSlackに投稿します。  
OpenClaw側で勝手に言い換えません。

成約処理や商談後整理のようなフローでは、特にこのルールが効きます。

営業管理でユーザーが知りたいのは、内部ログではありません。

* 何が更新されたか
* どのNotionページを見ればよいか
* 追加確認が必要か

これだけです。

なのでSlack返答には、コマンド、stdout、JSON全体、内部ステータスを書かないようにしています。

「AIがそれっぽく報告する」のではなく、「処理結果が報告される」状態に寄せています。

## 記憶はMarkdown、業務状態はNotionとstateに分ける

AIエージェントに記憶を持たせるとき、何でも長期記憶に入れたくなります。

でも、SFA/CRMでは「記憶」と「業務状態」を分けたほうが扱いやすいです。

このエージェントでは、恒久的な方針や運用ルールはMarkdownに置いています。

* `SOUL.md`: エージェントの役割
* `USER.md`: 誰のために動くか
* `MEMORY.md`: 恒久コンテキスト
* `memory/YYYY-MM-DD.md`: 日次メモ
* `skills/*/SKILL.md`: 操作判定ツリーと業務ルール

一方で、営業の正しい業務状態はNotion DBに置きます。

* 取引先DB
* 商談DB
* 案件DB
* 問い合わせDB
* 請求書DB
* メンバーDB
* 議事録DB

さらに、処理途中の状態はローカルの `state/` に置きます。

* Slack依頼キュー
* 商談リマインドの送信履歴
* 問い合わせ通知スキャン履歴

この分離にすると、頭の中がかなり整理されます。

Markdownは、人間とAIが共有する運用契約。  
Notionは、業務の正本。  
stateは、実行システムの作業メモ。

全部をベクトルDBやLLMの記憶に入れないほうが、営業管理では安定しました。

## 定期処理もエージェントに寄せすぎない

SFA/CRMには、定期的にやりたい処理もあります。

* 停滞商談のリマインド
* 期限超過した次アクションの通知
* KPIダッシュボード更新
* 日報・週報
* 問い合わせ通知のスキャン
* Slackチャンネル名とNotionステージの同期

これも、LLMに「毎朝いい感じに見ておいて」と任せるより、cronやlaunchdでスクリプトを動かすほうが安定します。

現在は、Slack自然文のworkerはlaunchdで常駐させています。

```
process-slack-request-queue.mjs
  --watch true
  --interval-ms 5000
  --limit 5
```

また、リマインドやダッシュボード更新はcron想定のスクリプトとして分けています。

LLMが必要なのは、自然文の解釈や要約など、一部の判断が必要なところだけです。

定期巡回、重複抑止、対象DBの検索、Slack投稿のような処理はプログラムで固定したほうがいい。

ここでも、LLMとプログラムを混ぜすぎないことが効いています。

## 現状の実装

この記事を書いている時点では、SFA/CRMエージェントのスクリプトは48本あります。

主な処理は、だいたい一通り動くようになっています。

* Slack自然文のキューイング
* Slack自然文のintent分類
* 固定ルーターによるスクリプト実行
* 取引先・商談・案件の作成/検索/更新
* 問い合わせ登録
* アポ獲得から商談化
* 商談準備メモの生成
* 商談後整理
* 提案書・契約書URLの記録
* 成約から案件化
* 失注処理
* 請求書作成
* PDF請求書の取込
* 入金済みなどの請求書更新
* 案件ステータス同期
* KPIダッシュボード更新
* 日報・週報生成
* 担当者引き継ぎ

もちろん、まだ荒いところはあります。

特に難しいのは、Slackの文脈解決です。

Slackでは、トップレベル投稿、スレッド返信、未メンション返信、Botの過去返信、貼られたNotion URL、添付ファイルなどが混ざります。

「このメモはどの商談に入れるべきか」を間違えると致命的です。

そのため、商談チャンネル内ならチャンネルIDから商談を逆引きする。  
一般チャンネルのスレッドなら、スレッド内のNotion商談リンクを探す。  
それでも一意に決まらないなら確認待ちにする。

このあたりは、営業管理AIの肝だと思っています。

## 作って分かったこと

良かった点。

* Slackから自然文でSFA/CRMが更新できると、入力の摩擦がかなり減る
* キュー化したことで、Slackイベント再送や同時投稿に強くなった
* LLMを分類に寄せ、実行を固定スクリプトにしたことで、挙動をレビューしやすくなった
* Notion DBを正本にすると、AIが何をしたかを人間が確認しやすい
* Slackチャンネルと商談/案件を1対1で結ぶと、会話と業務レコードが自然につながる

難しかった点。

* Slackの文脈境界が思ったより難しい
* 「ありがとう」などの雑談と、実行依頼の見極めが必要
* アポ獲得と問い合わせ登録が同じメッセージに混ざる
* 添付PDFやSlackファイルURLは、権限や取得方法まで含めて考える必要がある
* 成約・失注・案件終了などの危険操作は、明示語と対象特定の両方が必要

今後やりたいこと。

* dead jobの原因を分類して、確認待ちに寄せる
* SlackチャンネルとNotionページの紐づき検査を強化する
* 商談準備メモの品質を上げる
* KPIダッシュボードを、手動ビューではなくより自動生成に寄せる
* ルーターのdry-runを日常的に使える検証ツールにする

## まとめ

営業管理AIエージェントを作ってみて、いちばん強く感じたのは、AIエージェントは業務システムそのものではなく、業務システムの自然言語インターフェイスとして設計したほうが安定するということです。

LLMは、曖昧なSlack発話を読めます。

でも、Notion DBをどう更新するか、Slackチャンネルをいつ作るか、失注理由がない時に止めるか、同じSlackイベントを二重実行しないか、という部分はプログラムで固定したほうがいい。

今回のSFA/CRMエージェントでは、だいたい次の分担にしています。

* OpenClaw: Slackなどの入口とエージェント実行基盤
* LLM: 自然文の分類と一部の判断
* 固定ルーター: 実行可能なintentとスクリプトの対応
* Program Skill: Notion/Slackへの安定した副作用
* Notion: 営業管理の正本
* state: キューやリマインドの実行状態
* Markdown: 人間も読める運用契約と記憶

「AIに営業管理を任せる」というより、「営業管理システムの前に、自然言語を理解する薄い層を置く」。

このくらいの距離感が、実用のAIエージェントにはちょうどいいのかもしれません。
