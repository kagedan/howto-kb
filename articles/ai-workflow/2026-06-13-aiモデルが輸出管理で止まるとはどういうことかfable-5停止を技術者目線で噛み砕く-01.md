使っていたAIモデルが、ある日エラーを返すようになりました。「選択中のモデルに問題があります。存在しないか、アクセス権がない可能性があります」。障害かと思って調べたら、原因は障害でも値上げでもなく、輸出管理でした。

2026年6月12日、Anthropic が Claude Fable 5 と Mythos 5 を、全顧客向けに無効化しました。米政府の輸出管理指令に従うためです。この「輸出管理でモデルが止まる」という事態は、技術者にとって馴染みが薄いので、何が起きたのかを噛み砕いて整理します。そのうえで、自分の本番環境をどう守るかまで書きます。

## 何が起きたか（短く）

先に事実だけ並べます。

- Fable 5 と Mythos 5 は6月9日に公開された
- 6月12日の東部時間17時21分、米政府から輸出管理指令が届いた
- 指令の内容は「いかなる外国籍者によるアクセスも禁止」。米国内外を問わず、Anthropic 自身の外国籍従業員も含む
- Anthropic はリアルタイムで国籍を判別して遮断できないため、全顧客向けに両モデルを完全停止した
- Opus 4.8 など他のモデルは影響を受けていない

ここまでが、ほぼ争いのない事実です。

## そもそも「輸出管理」とは何か

ここがいちばん噛み砕きたいところです。

輸出管理（export control）は、普段は武器や特定の半導体など、国境を越える移動を国家が管理したい「モノ」に対して使う法的な枠組みです。誰に・どこへ渡していいかを国が制限して、場合によってはライセンス（許可）がないと渡せない。技術者の感覚に置き換えると、特定の暗号技術や軍事転用可能な部品が、輸出規制リストに載って自由に持ち出せなくなる、あの仕組みです。

今回それが、展開済みのAIモデルの能力に適用されました。つまり国は、このモデルを「国境を越えて渡してはいけないモノ」として扱った、ということです。実際、報道によれば、商務省の書簡ではこれらのモデルの輸出・再輸出・国内移転にライセンスが必要とされています。ソフトウェアのアクセス権が、規制対象の貨物と同じ枠で語られている。ここが新しい点です。

## なぜ「外国籍者の禁止」が「全停止」になるのか

エンジニアとして引っかかるのは、ここだと思います。指令は「外国籍者のアクセス禁止」なのに、なぜ全員が使えなくなるのか。

理由は実装上の制約です。Anthropic は、米国ユーザーの中から外国籍者だけをリアルタイムに判別して弾く、ということができません。国籍は IP やアカウント情報から確実には分からないからです。指令の対象は「米国内外のあらゆる外国籍者」なので、確実に遵守しようとすると、誰が外国籍か判別できない以上、全員を止めるしかない。選択的に遵守しようとすれば、自社の外国籍従業員まで含めて広範囲を遮断する必要が生じます。だから、技術的に「全員停止」が唯一の確実なコンプライアンスになった、というわけです。

これは、コンテンツフィルタや地域制限とは性質が違います。地域制限なら IP ベースで切れますが、「国籍」はリクエストからは判定できない属性です。判定できない条件で遮断を求められたら、安全側に倒して全部止める、というのは、実装としては理解できる帰結です。

:::note warn
ここで起きたのは、ベンダーの都合でも、あなたのコードのミスでもありません。地政学による停止です。しかも、たいていのSLAや利用規約は、この種の「政府命令による利用不能」をカバーしていません。可用性の前提が、技術や商習慣の外側から崩れた、という点が新しいリスクです。
:::

## なぜ政府が止めたのか（背景・最小限）

理由の部分は、まだ確定していません。ここは深掘りせず、要点だけ置きます。

Anthropic の説明では、政府は Fable 5 のジェイルブレイク手法が見つかったと考えているとのことです。ただし同社は、それは他の公開モデルでも回避なしに見つかる軽微な既知の脆弱性にすぎず、ユニバーサルなジェイルブレイクは見つかっていない、と反論しています。同社はこの状況を誤解と呼び、アクセス復旧に取り組んでいるとしています。

なお、Axios は「別の企業が Mythos をジェイルブレイクしたと主張したのが発端」と報じていますが、これは単一ソースの報道で、Anthropic は確認していません。事実として扱わないでください。政治的・前例的な深掘りは長くなるので、別の記事に分けます。この記事の主眼は、技術者として何が起きて、何ができるか、です。

## 技術者として、何ができるか

起きたことが「自分の依存するモデルが、外部要因で突然消える」だとすると、対処は素直です。モデルを差し替え可能にしておく、これに尽きます。

考え方は、データベースやクラウドのベンダーロックイン対策と同じです。特定のモデル名を、コードのあちこちに直書きしない。1か所に集約して、設定で差し替えられるようにする。そして、差し替え先を事前に決めて、実際に動くか試しておく。

汎用的な擬似コードで示します。まず、モデルの選択を設定に追い出します。

```python
# config: モデルをコードに直書きせず、優先順とフォールバックを設定で持つ
MODEL_CHAIN = [
    {"provider": "anthropic", "model": "primary-model"},   # 普段使うモデル
    {"provider": "anthropic", "model": "fallback-model"},  # 同ベンダーの代替
    {"provider": "other",     "model": "cross-vendor"},    # 別ベンダーの代替
]
```

呼び出し側は、特定のモデル名を知りません。チェーンを上から試して、使えないものは次へ送るだけにします。

```python
def call_model(messages):
    last_error = None
    for entry in MODEL_CHAIN:
        try:
            client = get_client(entry["provider"])
            return client.complete(model=entry["model"], messages=messages)
        except ModelUnavailableError as e:
            # モデルが存在しない・権限がない・停止された場合に進む
            last_error = e
            log.warning("model unavailable, falling back: %s", entry["model"])
            continue
    # すべて失敗したときの最終手段（縮退運転やキューイング）
    raise AllModelsUnavailable(last_error)
```

効いてくるのは、`ModelUnavailableError` の扱いです。今回のように、昨日まで使えたモデルが「存在しない・アクセス権がない」エラーを返すケースを、一時的な失敗としてフォールボールに乗せる。例外を握りつぶさず、次の候補へ確実に送る。ここを最初から作っておくと、モデルが1つ消えてもサービスは止まりません。

実際、今回 Opus 4.8 などが無事だったおかげで、フォールバック経路を持っていた統合は、壊れずに縮退で済みました。逆に `fable-5` のような特定モデルに固定していた実装は、その場で止まりました。差は、チェーンを持っていたかどうかだけです。

最後に、フォールバック経路は必ずテストしてください。書いただけで一度も通っていないフォールバックは、いざというとき動きません。普段のテストで、わざと先頭モデルを落として、2番手・3番手が応答するかを確認しておく。これをやっておくと、「ある朝モデルが消えていた」が、設定変更1つで済む話になります。

## 次の自分に渡すメモ

今回の件で、自分の中の前提が1つ変わりました。フロンティアモデルは、基盤ではなく依存だ、ということです。基盤は自分の足元にあって動きませんが、依存は外の都合で抜かれます。そして今回、その「外の都合」に、地政学と輸出管理という、これまで可用性設計で考えてこなかった軸が加わりました。

だから、特定のモデルに固定しないこと。差し替えを設定変更で済むようにして、フォールバックを事前に試しておくこと。これは、賢いモデルを選ぶ話ではなく、どのモデルが消えても自分のサービスが立っていられるようにする、という設計の話でした。3日で公開されて3日で消えたモデルを見て、いちばん残しておきたい教訓は、これです。
