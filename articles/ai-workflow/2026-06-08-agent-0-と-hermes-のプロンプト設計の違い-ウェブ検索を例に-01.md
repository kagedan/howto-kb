## はじめに

OSS のエージェントフレームワークを使っていると、同じ「ウェブで調べる」というタスクでも、フレームワークによって体感の精度がかなり違うことに気づきます。

その差がどこから来るのかを、両者のリポジトリにある実際のプロンプトとツール定義まで降りて比較したのが本記事です。題材は「ウェブページを探すルーチン」に絞ります。

## ウェブページを探すルーチン

まず、両者が「検索 → ページ本文の取得」をどう組み立てているかを整理します。

### Agent 0

1. `search_engine` で検索し、URL・タイトル・説明のリストを得る（バックエンドは SearXNG）。
2. 開きたいページは `document_query` で本文や OCR テキストを取得する。
3. browser 系ツール（`browser_open` など）はリポジトリに存在するが、ファイル名が `._py` になっており**既定では無効**。検索とページ取得だけで完結させる前提。
4. 本格的な調査が必要なときは `researcher`（"Deep ReSearch"）プロファイルに委譲する設計。

### Hermes

1. `web_search` で検索し、URL・タイトル・説明を得る（バックエンドは Exa / Parallel / Firecrawl / Tavily などから自動選択）。
2. 開きたいページは `web_extract` で Markdown 化して取得する（PDF も可）。
3. クリックやフォーム入力などページ操作が必要なときだけ `browser` ツールを使う。`web_extract` が失敗したときの**フォールバック**先でもある。

ルーチンの骨格は似ていますが、**それぞれのツール説明やシステムプロンプトに書かれている「指示の濃さ」が違います**。

## プロンプト設計の違い

ウェブ検索に関わる部分を、プロンプト/ルーチンの観点でまとめます。

| 観点 | Agent 0 | Hermes |
| --- | --- | --- |
| 検索ツール名 | `search_engine` | `web_search` |
| クエリ整形の指示 | **強い**。「自然文の質問にしない」「3〜10 語の高シグナル語」「フィラー語を除く」を明記し、悪い例 / 良い例の対比まで提示 | **弱い**。`site:` `filetype:` `-term` `"完全一致"` 等の演算子が「使える場合がある」と案内する程度。整形ルールや例示なし |
| 既定の取得件数 | 最大 **10 件**（`SEARCH_ENGINE_RESULTS = 10`） | 既定 **5 件**（`limit` で 1〜100 に変更可） |
| 検索バックエンド | SearXNG（自前ホストのメタ検索） | Exa / Parallel / Firecrawl / Tavily から自動選択 |
| ページ本文の取得 | `document_query`（URL・文書・画像 OCR） | `web_extract`（Markdown 化、約 5000 字で要約、1 回最大 5 URL、PDF 対応） |
| browser の方針 | 既定で無効（`._py`）。検索＋本文取得で完結 | 「単純な取得は `web_search` / `web_extract` を優先（速く安い）」「`.md` `.json` raw 等は `curl` か `web_extract`」「操作が要るときだけ browser」とツール説明に明記 |
| 「いつ検索するか」の指示 | 汎用の問題解決手順に内包。検索固有の発火条件は薄め | システムプロンプトで「現在の事実（天気・ニュース・バージョン）は `web_search` で」と明示。「不足情報は推測せず検索」とも指示 |
| 空振り・失敗時の指示 | 「失敗を受け入れず再試行」という一般則 | 専用ブロックで「空 / 部分結果なら別クエリで再試行」「完了かつ検証まで呼び続ける」と明記 |
| 精度を底上げする仕組み | `researcher` プロファイルが多源クロス検証・全文精読・出典信頼度評価を強制 | `skills/research/*`（arxiv 等の手順スキル）＋検証 / 再試行ブロック |

### Agent 0 ─ クエリ整形をツール説明で強制する

Agent 0 の特徴は、**検索ツールの説明文そのものにクエリの作り方が書き込まれている**ことです。要点は次の通りです。

* キーワードで書く。自然文の質問や文章にしない。
* 「何を」「誰が」「教えてください」「〜について調べて」のようなフィラーを落とす。
* 3〜10 語の高シグナルな語を使い、再現率が上がるときだけ代替語を足す。

そして、悪い例と良い例が並べて示されています。

agent.system.tool.search\_engine.md（要点）

```
bad:  What is the latest LiteLLM release and what changed?
good: LiteLLM latest release notes changelog
```

ツール呼び出しは JSON で行われ、`thoughts` → `headline` → `tool_name` → `tool_args` という構造に整形されたうえで `query` が渡ります。つまり **「モデルが投げる検索語が、検索エンジンに届く前に短く絞られる」設計**になっています。

### Hermes ─ 演算子の案内に留め、整形はモデルに委ねる

一方 Hermes の `web_search` 説明は、検索演算子（`site:` `filetype:` `intitle:` `-term` `"完全一致"`）が**バックエンドによっては効く**ことを案内するに留まります。クエリを短くせよ・自然文を避けよ、といった整形ルールや例示はありません。クエリの良し悪しは実質モデル任せです。

その代わり Hermes は、**ルーチンの堅牢さ**をシステムプロンプト側で担保しています。

* `mandatory_tool_use`: 現在の事実（天気・ニュース・バージョン）は記憶で答えず `web_search` を使う。
* `tool_persistence`: 結果が空・部分的なら別クエリで再試行し、タスク完了かつ検証が済むまでツールを呼び続ける。
* `missing_context`: 情報が取得可能なら推測せず取りにいく。

ページ取得側の `web_extract` も作り込まれており、5000 字未満はそのまま Markdown、大きいページは LLM 要約して約 5000 字に圧縮、2M 字超は拒否、失敗時は browser へフォールバック、という分岐を説明文に持っています。

## なぜ Agent 0 のほうが当たりやすく感じるのか

プロンプトを読む限り、体感には根拠があります。

1. **クエリが構造的に絞られる**  
   Agent 0 はツール説明で「自然文を捨ててキーワードにする」ことを強制し、悪い例 / 良い例まで与えます。結果として検索エンジンに届く語が高シグナルになり、ヒットの質が上がりやすい。Hermes は演算子を案内するだけで、整形はモデルの裁量に委ねられます。
2. **既定の取得件数が多い**  
   Agent 0 は既定 10 件、Hermes は既定 5 件です。候補が多いほど、目的のページが上位に含まれる確率は上がります。

つまり「ウェブ検索の精度差」は、モデルの賢さよりも先に、**プロンプトがクエリと件数をどれだけ規定しているか**で説明できる部分が大きい、ということです。

## Hermes 側で差を縮めるには

Hermes は OSS で設定変更が容易なので、Agent 0 の「効いている部分」を移植する形で寄せられます。

1. **クエリ整形ルールをシステムプロンプトに足す**  
   「`web_search` のクエリは自然文の質問にせず、3〜10 語のキーワードにする。`what` `who` `〜について教えて` などのフィラーは落とす」といった指示と、悪い例 / 良い例の対を 1 組入れるだけでも効果が見込めます。
2. **既定の取得件数を上げる**  
   `web_search` の `limit` を 5 から 10 程度に引き上げ、候補数を Agent 0 と揃えます。
3. **`web_search` → `web_extract` の連結を明文化する**  
   「検索結果の上位 N 件は `web_extract` で本文を確認してから結論を出す」とルーチンを固定すると、スニペットだけで判断する事故が減ります。

## まとめ

* Agent 0 と Hermes は、ウェブページを探すルーチン自体（検索 → 本文取得）はよく似ている。
* 差はプロンプト設計に出る。**Agent 0 は検索ツール説明にクエリ整形ルールと例示を埋め込み、既定 10 件を返す**。**Hermes は演算子の案内に留めて整形はモデル任せ、既定 5 件**。
* 「Agent 0 のほうが当たりやすい」という体感は、このクエリ整形と件数の差で説明がつく。
* Hermes 側は、クエリ整形ルールの追記・`limit` 引き上げ・検索と本文取得の連結明文化で寄せられる。

エージェントのウェブ検索精度を上げたいとき、まず触るべきはモデルではなく**検索ツールの説明文に書かれたクエリの作法**だ、というのが両者を読み比べての結論です。
