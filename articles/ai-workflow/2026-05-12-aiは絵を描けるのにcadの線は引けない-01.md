## まず前提: 画像生成AIと LLM は別物

多くの人は「AIはどんどん進化している」と漠然と捉えていて、「絵が描けるなら、同じAIにアスキーアートも、CADも頼めば、同じ水準でできるだろう」と思いがちです。

でも実は、**画像生成とテキスト生成は、全く別のアーキテクチャのモデル**で動いています。

* ChatGPT や Gemini のような LLM: テキスト(トークン列)を扱うモデル
* DALL-E や Nano Banana、Stable Diffusion のような画像生成AI: 画像を扱うモデル

ChatGPT や Gemini が画像を作るとき、裏では対話モデルとは別の画像生成モデルが動いています。LLM がそのまま絵を描くのではなく、画像生成モデルに指示を出している構成です。

実際、Gemini に画像生成を頼むと、「Nano Banana 2 を読み込んでいます」と表示されます。

![NanoBanana読み込み](https://static.zenn.studio/user-upload/3ba373ad6abd-20260422.png)  
   
同じブランドのサービスから出てくるので「同じAI」に見えるだけで、中では別の機構が動いている。ここが出発点です。

## 仕組みの違い: 「1次元を順に書く」vs「2次元を同時に整える」

少し仕組みの話を噛み砕きます。

**◆ LLM の動き方**

LLM は「次に来る確率が高いトークンを1つ選ぶ」という動作をひたすら繰り返しています。  
トークンというのは、文字のかたまりのことです。  
たとえば「今日は良い」の次には「天気」とか「日」が来やすい、という具合です。

やっていることはシンプルで、

1. これまでの出力を見る
2. 次に来る1トークンを確率的に選ぶ
3. それを出力に追加する
4. また1に戻る

これをひたすら繰り返します。

つまり LLM は、**左から右に、1つずつ、順番に**文字を生成していく機械です。  
一度書いたものは基本的に修正しません。

これは文章を書くときには非常に自然に機能します。  
文章は本質的に「1次元の列」なので、順に書いていけば意味が通ります。

**◆ 画像生成AIの動き方**

一方、画像生成AI(特に拡散モデル系)は全く違います。

1. ランダムなノイズから始める
2. 画像**全体**を少しずつ「らしく」していく
3. すべてのピクセルを**同時に**調整する
4. 何十回〜何百回の反復で完成させる

画像生成AIは、**2次元空間を2次元のまま全体で最適化する**機械です。「左目と右目の対称性」「鼻と口の位置関係」「陰影のバランス」を画像全体で見ながら調整できます。

この **「1次元を順に」か「2次元を同時に」か** の違いが、得意・不得意の根本原因です。

## 「2次元空間を1次元に押し込むフォーマット」は両者の弱点

ここで問題になるのが、**アスキーアート**と**CADコマンド列**です。  
これらには共通する性質があります。

見かけ上は文字列(1次元)、実体は2次元空間の情報。

アスキーアートで顔を描くとき、「3行目の5文字目と5行目の5文字目が、目の位置として左右対称になっているか」といった**空間的な整合性**が必要になります。でも LLM は左から右へ順に書くだけなので:

* 2行目を書いている時点では、10行目のことを決めていない
* 10行目を書く時点では、2行目を修正できない
* 出力全体を2次元図形として厳密に検証しながら、過去の出力まで含めて調整するのが苦手

結果、アスキーアートは「何かの形っぽいもの」にはなるけれど、**誰の顔かは判別できないレベル**で破綻します。先ほどの Gemini の出力が、まさにその症状です。

一方、画像生成AIに同じ指示を出せば、ピクセルを同時に最適化するので、アインシュタインらしさ(白髪、口ひげ)がしっかり再現されます。得意分野が違うというのはこういうことです。

## で、これが CAD でも起きる

ここからが本題です。

CAD の図面は、線を引く、円を描く、といった**描画命令の列**として記述できます。たとえば長方形は、4本の線分の始点と終点を順に指定すれば描けます。「次の線の始点は、前の線の終点と同じ座標」にしておけば繋がる、という単純な世界です。

では、LLM にこの命令列を書かせたらどうなるか。  
間取り図の画像を渡して、「これと同じ図面を描くための命令を書いて」と頼みました。

![間取り図](https://static.zenn.studio/user-upload/c58d96770295-20260421.png)  
*LLM に渡した間取り図(これをCADで再現させたい)*  
   
自分は AutoCAD 上で動作する AI エージェントを開発していて、LLM と CAD を繋ぐ部分を自作しています。そのため LLM は好きな座標に、好きなだけ線を引ける状態にしています。

ここで重要なのは、**LLM には十分な情報と自由度を与えている**ことです。

* 元の間取り図は画像として渡していて、LLM は図面全体を「見て」いる
* 線は好きな座標にいくつでも引ける
* 制約らしい制約は何もない

それでも、LLM が書いた命令を実行した結果はこうなりました。

![LLMが書いた図面](https://static.zenn.studio/user-upload/77dd3f90e37a-20260421.png)  
*LLM が生成した描画命令を AutoCAD 上で実行した結果*  
外形や建具の意図は、かろうじて見て取れます。でも元の間取り図と比べると明らかに線の数が少なく、部屋の区切りや水回りの設備の表現もほとんど抜け落ちています。残っている線も正確には繋がっておらず、図面として使える状態ではありません。

**これもアスキーアートで起きたのと、本質的に同じ問題**です。

LLM は命令を**頭から順に1つずつ**生成していきます。問題は2つあります。

ひとつは、そもそも**全体を出し切れない**こと。間取り図のようにたくさんの線が必要な対象を、LLM は「どこまで書けば十分か」を俯瞰して判断できません。結果、**途中で力尽きる**ような形で命令列が終わってしまいます。今回の結果で線の数が少ないのは、おそらくここが原因です。

もうひとつは、**書いた線同士が正確に繋がらない**こと。1本目の線の終点を `100,0` と書いたあと、2本目の線の始点にも厳密に `100,0` と書かなければ、線は繋がりません。でも LLM は次のトークンを確率的に選ぶので、ときどき `99.998,0` とか `100,0.1` のように微妙にズレます。さらに 10本目、20本目と命令を書き進めるうちに、前に書いた線との位置関係を正確に保ち続けるのも難しくなっていきます。

結果、閉じるべき図形が閉じない。交わるはずの線が交わらない。

情報は渡してある。自由度もある。それでも破綻する。  
これは**LLM がサボっているわけではなく、構造的にそうならざるを得ない**というだけの話です。CAD の命令列もまた、「2次元空間の情報を、1次元の命令列に押し込んだフォーマット」なので、アスキーアートと同じ苦手領域にぴったりハマります。

## 最新モデル × Fusion MCP で同じことを試したら

ここまで書いた話は、AutoCAD 上で Gemini 3.0 Flash に線を引かせた結果がベースになっています。そこで、2026年5月時点で最新クラスの Claude Opus 4.7 と、Autodesk が公開した Fusion 360 公式の MCP 連携を使って、同じ間取り図を渡したらどうなるかを試してみました。

![Fusion MCP + Opus 4.7 の結果](https://static.zenn.studio/user-upload/d3c2d45e3af6-20260429.png)  
*左: 生成された図面 / 右: 各線の頂点を表示*  
画像を1枚渡して、1回目の回答結果がこれです。(思考は数十分ほどでした)  
外形・主要な間仕切り壁・玄関タイル・扉の開き・水回りの設備記号まで、画像1枚からここまで出てきます。AutoCAD で試したときの「途中で力尽きる」感じはなくすばらしい結果です。

ただし、これは AutoCAD での結果と Fusion MCP の結果をそのまま横並びで比較できる検証ではありません。**前述の検証とは条件がいくつも違います**。

* CAD が違う (AutoCAD → Fusion 360)
* モデルが違う (Gemini 3.0 Flash → Claude Opus 4.7)
* LLM が呼んでいるものが違う
  + AutoCAD: LLM が指定座標を指示することで線分を作成
  + Fusion MCP: LLM がスケッチ作成の Python コードを一括で動的に生成

モデル差は確かに大きい要因のひとつですが、他にもいろいろな違いがあるので「LLM が進化したから描けるようになった」と一概には言えません。Claude → Fusion MCP では LLM の処理の裏側で OpenCV 利用もしていたのでそれが利いている可能性もあります。  
(いずれ AutoCAD 側のモデルを Opus 4.7 にした場合の結果も調査しようと思います)

──と、ここまでは「すごい」で終わる話。  
ただ、CAD を触っている人が見ると、もう少し違うところに目が行きます。

## 「絵としては描ける」と「CADデータとして使える」は別

上の図、見た目はかなり元の間取り図に寄っています。でも右側の頂点表示をよく見ると、気になる点がいくつかあります。

* **複数の線が微妙にズレて重なっている**箇所がある
* **1本の直線で済むはずの壁が、複数の線分に分割されている**箇所がある
* 結果として、**頂点の数が必要以上に多い**

ズームしてみると、これがよく分かります。

![](https://static.zenn.studio/user-upload/df2558376839-20260429.png)  
*Claude Opus 4.7 × Fusion MCP で生成された結果の拡大図*  
壁の輪郭をなぞる線が複数本ズレて走っていたり、本来1本で済む壁が短い線分に細かく分割されていたりするのが見て取れます。青く塗られているのは線で閉じられた領域ですが、線のズレのせいで意図しない形の閉領域ができてしまっています。

これは下流工程で効きます。面積が二重カウントされる、ハッチが閉領域として認識されない、壁を一つの単位として動かせない、別の CAD に渡したときに不正要素扱いされる ── CAD で図面を扱ったことがある人なら想像できると思います。

つまり、**絵としては正しいけれど、CAD データとしては怪しい**、という状態。  
ぱっと見では「すごい」で終わるところに、CAD 屋からすると「使えない」が見えてきます。

冒頭の話に戻ると、これは画像生成AI のアインシュタインの絵と似ています。  
写真として見れば完璧でも、黒板の数式は実在の方程式ではない。**「絵として正しい」と「中身まで正しい」は別の話**で、CAD でも同じことが起きています。

LLM の精度が上がれば線のズレは減っていくかもしれません。でも「ここからここまでが1本の壁」という**意味のある単位として図面を組み立てる**ところは、精度の話ではなく、CAD の意味論を扱える形にツール設計する必要があります。

## じゃあどうするか

自分の AutoCAD エージェントも、最初は「LLM に図面を任せてみる」方向で試していました。でも検証を重ねるうちに、**LLM は意図を理解して命令を組み立てる役**、**実際の描画や幾何的な検証は決定論的な処理**、という分担に落ち着きました。

最新モデルなら、見た目としての図面はかなり描けるようになってきています。ただ、中身を CAD データとして見ると、線のズレや意味のない分割が残っていて、そのまま下流工程に流せる状態にはなっていません。**「絵として描ける」と「CAD データとして使える」の間にはまだギャップがある**、というのが今回の検証で見えたことです。

「ここからここまでが1本の壁」といった**意味のある単位**を LLM に扱わせるには、その単位を CAD 側のツールとして用意しておく必要がある ── というのが、AutoCAD エージェントで自分が採用してきた方針でした。

ただ正直、Fusion MCP の結果を見た今は、その方針が最適だったのか少し分からなくなっています。AutoCAD 側の結果が良くないのが、モデルの性能なのか、自分のツール設計の問題なのか、その両方なのか、まだ切り分けられていません。**LLM と決定論的な処理のどこに線を引くか**は、これから手を動かしながら詰めていく問題だと思っています。

絵が描けるからといって、CAD の線が引けるとは限らない。  
そして、CAD の線が引けたからといって、CAD データとして使えるとは限らない。AI の得意な形を見極めて、そこに頼む。その線引きこそが、自分の課題になりそうです。
