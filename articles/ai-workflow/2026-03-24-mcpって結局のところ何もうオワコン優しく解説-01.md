# 背景

AI活用の文脈で良く出てくるMCP。すごく流行ったかと思えば、ここ最近は「[MCP is dead](https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html)」なんて言われ始めており、話題に事欠かない存在です。本当に頻出ワードなのですが、実はよくわかってない人が多いんじゃないかなと思います。MCPサーバーを実際に使っているエンジニアも、なんとなく使っているだけで原理の部分は曖昧なんじゃないかなと。（かくいう僕がそうでしたので）

この記事ではなるべく平易に、わかりやすくMCPについて解説します。特に以下4点が伝えたい点です。

* MCPとはルールのこと
* MCPは「外部ソフトウェアを使うための補助輪」
* MCPの存在意義は揺らいできている
* それでもMCPには一定の意義がある

この記事で少しでも皆さんの理解が深まり、MCPの効果的な活用に繋がれば幸いです。

※この記事でAIと書いてある部分は、基本的にLLMのことを指します。ロボットが倒れないようにリアルタイムで姿勢を制御するAIなどは想定していません。

# MCPとはルールのこと

いきなりですが、誰かとコミュニケーションを取る際、重要なこととはなんでしょうか。表情や身だしなみも大事ですが、一番重要なのは「どんな音の組み合わせが、どんな概念を表すのか」のルールです。要は単語や文法ですね。これが無いと、何も伝わりません。「りんご」という音声と「赤い木の実」という概念を結び付けるルールが無ければ、「りんご」はただ「り」と「ん」と「ご」が連続して発音されただけになってしまいます。

こういったコミュニケーションのためのルールは、人間同士の会話に限らず必要です。人間とコンピューター、コンピューターとコンピューター、どんなコミュニケーションであれそこにはルールが存在します。そして、そのルールをIT業界では「プロトコル」と言います。

MCPはModel Context Protocol（モデルコンテキストプロトコル）の略です。直訳すると「AIのモデルが外部のツールと文脈を共有するためのルール」がMCPということになります。もっと簡単に言えば「AIが外部のツールと連携するためのルール」です。

現場では「MCPを立てよう」といった会話がされがちなのですが、語義通りに捉えると「ルールを立てよう」になるため、混乱すると思います。MCPサーバー（後述）というものがあるのですが、それを立てることを簡略化して「MCPを立てる」と言っている場合が多いです。

# MCPが必要とされる理由

## AIはデータの送受信しかできない

GPT、Gemini、ClaudeなどAIには色々ありますが、共通するのは「外界に直接影響を与えることはできない」という点です。送信されたデータを読み、そのデータに応じて考え、結果をデータで返す。AIがこれ以上のことをすることは基本的にできません。

しかしながら、世の中にはAIを中核に据え、ファイル編集等を行うAIソフトウェアが沢山あります。（Cursor、GitHub Copilot、Codex、Claude Codeなどなど…）データの送受信しかできないはずのAIが、どうやってタスクをこなすのでしょうか？

## AIソフトウェアの仕組み

AIソフトウェアは、コンピューター内のファイルを読み込んだり、編集したり、外部に通信を飛ばしたりする能力を、**AIとは無関係に**備えています。AIが能力をソフトウェアに与えるのではなく、ソフトウェアが元々能力を持っているのです。

そして、AIソフトウェアはその能力の情報と、ユーザーからの入力、これまでの会話などを全てまとめて特定の形式の文字列でAIに渡します。それをもとにAIは考えて、「こういう能力を使ってこんなことをしよう」という結果を導き出し、それを特定の形式の文字列でAIソフトウェアに返します。

## 外部のソフトウェアの能力を使いたいときはどうする？

さて、タスクをこなすにあたり、外部のソフトウェアの力を使いたいことがあります。ブラウザを使うとか、Webアプリを利用するとかですね。こんな時はどうすればよいのでしょうか。

### 方法１：使い方を教える

方法の一つ目は、AIに外部のソフトウェアの使い方を教えることです。ここではエンジニアが良く使うWebアプリとしてGitHubを例に出します。

| GitHubのコメントを取得する方法を教える |
| --- |
|  |

GitHubにどのような通信をすればよいのかをユーザーの指令で教えました。これによってAIソフトウェアは、GitHubのコメント内容を取得することができます。

このような使い方を教える手法のメリットは、「簡単である」ことです。使い方を教えればいいだけなので特別な設定は必要ありません。

一方で、デメリットは「指示が煩雑になりがち」なことです。コメント以外にもGitHubには様々な機能があり、それらすべてに関して取得・編集の方法を命令に含めるとなると、文字数は膨れ上がります。AIも人間も、指示はなるべくシンプルな方が動作が安定するため、これはあまり望ましくありません。

### 方法２：MCPサーバーを使用する

方法の二つ目は、MCPサーバーを使うことです。MCPサーバーは、MCPというプロトコル（コミュニケーションのルール）に従って、外部のソフトウェアとAIソフトウェアとの間を仲介してくれるサーバーです。

まず、AIソフトウェアがMCPサーバーに、「何ができる？」と聞きます。すると、MCPサーバーはあらかじめ定められている「できること」の一覧を返却します。

| できること一覧の取得 |
| --- |
|  |

その後、AIソフトウェアの能力には、MCPサーバーから返却されたできることが追加されます。

| 能力にMCPサーバーのできることが追加 |
| --- |
|  |

AIがMCPサーバーの能力を使おうとすると、AIソフトウェアはそれをMCPサーバーに連絡します。

| MCPサーバーの能力使用 |
| --- |
|  |

そしてMCPサーバーは、あらかじめプログラムされた方法でGitHubからコメントを取得し、AIソフトウェアに返却します。さらにその内容をAIに返します。

| MCPサーバーによる中継 |
| --- |
|  |

一連の流れの中で、AIソフトウェアとMCPサーバーの間で交わされるやり取りの形式を定めたルールのことを、MCPと言います。

さて、前項の方法１（使い方を教える）のデメリットはこのようなものでした。

> デメリットは「指示が煩雑になりがち」なことです。コメント以外にもGitHubには様々な機能があり、それらすべてに関して取得・編集の方法を命令に含めるとなると、文字数は膨れ上がります。AIも人間も、指示はなるべくシンプルな方が動作が安定するため、これはあまり望ましくありません。

MCPサーバーはこのデメリットを解消してくれます。人間が「コメントを取得するにはリクエスト××を送って…」みたいなことを言わなくても、MCPサーバーにあらかじめプログラムされた仕組みがリクエスト××を送ってくれます。AIはGitHubの扱い方を意識せず、単に「コメント取得」を行うだけです。動作は格段に安定します。

MCPサーバーはこのように「外部ソフトウェアを使うための補助輪」となります。

加えて、MCPというプロトコルに世の中の主要なAIツールやプラットフォームが対応しているため、一度GitHub用のMCPサーバーを作成すれば、どんなAIツールでもGitHubを安定的に使用することができるようになります。MCPという共通規格に従えば、どんなソフトウェアでもAIとの協働が可能になります。これが理由で、多くのソフトウェアがそのソフトウェア専用のMCPサーバーを開発し公開しています。

# MCPの存在意義は揺らいできている

さて、MCPは便利なプロトコルですが、良い面ばかりではありません。最近は悪い面が目立つようになってきました。それはなぜかというと、ひとえにAIの性能が上がってきたからです。

## 自走力の大幅な向上

ここ数ヶ月で、AIの自走力は目を見張るほどに成長しました。ドキュメントを自分で確認してコマンドを打ったり、リクエストを飛ばしたりするばかりか、エラーが出ればそのエラーの意味を解釈し行動を変えることもできるようになりました。

> AIはGitHubの扱い方を意識せず、単に「コメント取得」を行うだけです。動作は格段に安定します。

前項でこのように書きましたが、これは半分本当で半分嘘です。今のAIなら、MCP無しでGitHubを用いた複雑なタスクを余裕でこなせます。動作の安定性向上は古いモデルでは顕著でしたが、最近はあまり感じられなくなってきています。

## コンテキストの圧迫

AIが解釈できる文章の量（コンテキスト）には制限があります。MCPによって追加された様々な能力は、AIが読み込むことができる文章の量を確実に削ります。

| MCPによるコンテキスト圧迫の極端なイメージ |
| --- |
|  |

AIの能力が低かったときはコンテキストを削ってでもMCPを使う利点がありましたが、現在はむしろコンテキストを圧迫することのデメリットの方が目立つようになりました。

特にGitHubのような有名サービスではこの傾向は顕著で、そもそもGitHubの使い方がAIの学習データに含まれていたりします。こうなると、MCPの能力定義は完全に冗長です。それが無くても、AIはどんなリクエストをGitHubに飛ばせばコメントの取得ができるのかわかるし、仮にエラーが出ても、どのようにトラブルシューティングすればいいのか知っているのです。

## リソースの浪費

MCPサーバーは使用するソフトウェアごとに作成する必要があります。また、サーバーの立ち上げにはCPUの計算能力を使います。MCPサーバーの立ち上げは往々にして不安定で、再起動等を余儀なくされることも多いです。

# それでもMCPには一定の意義がある

最近になり欠点が目立つようになってきたMCPですが、全く無意味になったのかと言えばそんなことはありません。「以前ほど万能ではないが、用途を絞れば今も有効」になったと言えます。最近の論調を見ても、批判の中心は「MCPという規格そのもの」より、「何でもMCPで包もうとする使い方」に向いています。

実際、Anthropic自身も、ツール定義を最初から大量に読み込むとコンテキストを圧迫し、効率が落ちることを認めています。そのうえで、コード経由でMCPを扱う方法や、必要なツールだけを後から探索する方式を提案しています。つまり「MCPを捨てる」のではなく、「[全部を一度に背負わせない](https://www.anthropic.com/engineering/code-execution-with-mcp?utm_source=chatgpt.com)」方向に進んでいるわけです。

一方で、標準化の流れ自体はむしろ強まっています。2025年末にはMCPがLinux Foundation傘下の[Agentic AI Foundationに寄贈](https://www.anthropic.com/news/donating-the-model-context-protocol-and-establishing-of-the-agentic-ai-foundation?utm_source=chatgpt.com)され、Anthropic、OpenAI、Google、Microsoft、AWSなどが関与する形に広がりました。OpenAIも公式ドキュメントでMCPサーバー構築や接続方法を案内しており、少なくとも業界標準候補としての地位はまだ失っていません。

結局のところ、MCPは「AIが未熟だから必須のもの」から、「複数ツールを安全かつ共通の形式でつなぐための標準」へと役割が変わりつつあるのだと思います。GitHubやブラウザのように、AIがすでに扱い方をよく知っている相手には、CLIやAPI直叩きのほうが軽くて速い場面も多いでしょう。逆に、社内システムや独自SaaS、認証や権限管理をきちんと揃えたい環境では、MCPのような共通ルールが今なお役立ちます。最近の「MCP is dead」という言説は、MCPの完全な死を意味するというより、「補助輪を常時付ける時代は終わった」という指摘として読むのが正確です。MCPは消えるのではなく、万能薬から適材適所の道具へと落ち着いていくのではないでしょうか。

# まとめ

* MCPとはAIソフトウェアとMCPサーバーとの間のやり取りをするルールのこと
* MCPは「外部ソフトウェアを使うための補助輪」だが、最近は必要な場面が限られてきている
* それでもMCPには社内システムや独自SaaSにおいて一定の使用意義がある

**注意**  
この記事ではわかりやすさの為、AIやMCPに関して簡略化したり省いたりしている要素があります。
