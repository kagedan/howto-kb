# AIエージェント導入前に、判断ログを設計する

AIエージェントを導入する時、最初に注目されやすいのは自動実行です。

問い合わせを読む。  
返信案を作る。  
CRMを更新する。  
FAQ候補を出す。  
タスクを次の担当者へ渡す。

どれも便利です。

ただ、現場で本当に詰まるのは、実行そのものではなく、その前後にある判断です。

これはAIに任せてよいのか。  
この返信は送ってよいのか。  
この情報は外部サービスへ渡してよいのか。  
誰が承認すればよいのか。  
保留にした理由は何か。  
前回似たケースでは、どう判断したのか。

これらが残っていないままAIエージェントを入れると、自動化は動いても、運用は安定しません。

この記事では、AIエージェント導入前に作るべき **判断ログ** の設計を整理します。

## 想定する読者

* 問い合わせ対応、無料診断、初回相談をAIで効率化したい人
* AIエージェントを導入したいが、どこまで任せるべきか迷っている人
* スプレッドシート、CRM、Notionなどで業務状態を管理している人
* 全自動化より先に、安全な承認フローを作りたい人

この記事では、AI APIの呼び出しやエージェントフレームワークの比較は扱いません。

その前段にある、判断を記録する設計を扱います。

## 判断ログがない状態で起きること

AIを入れる前の現場には、暗黙の判断が多くあります。

* これはいつものFAQだから、そのまま返信してよい
* この顧客は既存契約中なので担当者へ戻す
* 返金や解約を含むので責任者確認にする
* 個人情報が含まれるのでAIへ本文を渡さない
* 料金表が古いので返信前に確認する

こうした判断は、経験のある人の頭の中では自然に処理されています。

しかし、記録されていない判断は、AIにも新しい担当者にも引き継げません。

結果として、次の問題が起きます。

| 起きること | 影響 |
| --- | --- |
| 同じ確認が毎回発生する | 対応速度が上がらない |
| 判断理由が残らない | 後から改善できない |
| 承認者が曖昧になる | 責任範囲がぼやける |
| 例外が口頭で処理される | FAQやルールに反映されない |
| AIに任せる範囲が揺れる | 運用品質が安定しない |

AIエージェントは、曖昧な業務をそのまま渡すと、曖昧なまま速く処理します。

速くする前に、どの判断を残すかを決める必要があります。

## 判断ログの位置づけ

判断ログは、タスク管理でも議事録でもありません。

タスク管理は「何をやるか」を追います。  
議事録は「何を話したか」を残します。

判断ログは、「何を、なぜ、誰が決めたか」を残します。

```
問い合わせ / タスク / CRMイベント
  ↓
判断が必要かを判定
  ↓
判断ログを作成
  ↓
承認 / 保留 / 差し戻し / 却下
  ↓
次アクションへ進める
```

AI導入では、この判断ログが人間確認ゲートになります。

AIは、判断ログを見て「今は下書きまで」「これは人間確認」「これはFAQ改善候補」といった扱いを変えられます。

## 最小構成

最初から複雑なワークフローを作る必要はありません。

小さく始めるなら、次の項目で十分です。

| 項目 | 目的 |
| --- | --- |
| decision\_id | 判断を一意に追う |
| source\_id | 元の問い合わせやタスクとつなぐ |
| decision\_type | 何の判断かを分類する |
| status | 判断待ち、承認、保留などを固定する |
| owner | 誰が判断するかを明確にする |
| reason | なぜ判断が必要かを残す |
| decision | 決定内容を残す |
| next\_action | 判断後に何をするかを残す |
| decided\_at | いつ決まったかを残す |

大切なのは、判断結果だけではありません。

理由と次アクションを残すことです。

理由が残ると、あとから分類ルールを改善できます。  
次アクションが残ると、承認されたのに誰も動かない状態を避けられます。

## decision\_typeを決める

判断種別は、現場の迷いに合わせて決めます。

たとえば問い合わせ対応なら、次の5つから始められます。

| decision\_type | 判断すること |
| --- | --- |
| ai\_transfer | AIに情報を渡してよいか |
| reply\_approval | 顧客返信を送ってよいか |
| exception\_handling | 例外ケースとして人間が見るか |
| faq\_update | FAQやテンプレートへ反映するか |
| crm\_stage\_change | CRM上の状態を進めるか |

この分類は、AIエージェントの「できること」を決める材料にもなります。

たとえば、ai\_transfer が未承認ならAIは本文を読まない。  
reply\_approval が pending ならAIは送信せず下書きで止める。  
faq\_update が approved ならFAQ候補として別タスクを作る。

このように、判断ログはAIの行動範囲を制御するルールになります。

## statusを固定する

判断ログで特に重要なのは、状態を固定することです。

おすすめは、次の6つです。

| status | 意味 |
| --- | --- |
| pending | 判断待ち |
| approved | 承認済み |
| rejected | 却下 |
| hold | 保留 |
| returned | 差し戻し |
| done | 次アクション完了 |

状態名が増えすぎると、AIも人間も解釈に迷います。

「確認中」「対応中」「要相談」「見ておく」などを自由入力にすると、運用が揺れます。

保留や差し戻しも、失敗ではありません。  
正常な判断状態として扱います。

## AIエージェントに渡す時の考え方

判断ログがあると、AIエージェントに渡す情報を絞れます。

AIに渡すべきなのは、すべての会話履歴ではありません。

必要なのは、現在の状態、判断理由、次にできる行動です。

```
decision_id: DEC-20260603-001
source_id: INQ-20260603-014
decision_type: reply_approval
status: pending
owner: 責任者
reason: 返金と解約を含むため、顧客返信前に責任者確認が必要
allowed_ai_action: draft_only
blocked_action: send_to_customer
next_action: 返信案を作成し、人間承認へ回す
```

この程度の情報でも、AIは次のように振る舞えます。

* 顧客へ送信しない
* 返信案だけ作る
* 判断理由を要約する
* 承認者に確認依頼文を作る
* FAQ化候補としてメモを残す

逆に、判断ログがないと、AIは「この返信を送ってよいか」を毎回プロンプト内の曖昧な指示から推測することになります。

## 判断ログは業務記憶になる

判断ログは、単なる承認履歴ではありません。

積み上がると、業務記憶になります。

* どの種類の問い合わせがよく止まるか
* どの判断が保留になりやすいか
* どのルールが曖昧か
* どのFAQが不足しているか
* どの承認が特定の人に集中しているか

この情報は、AI導入後の改善に使えます。

プロンプトを直す前に、判断ログを見れば、どの業務ルールが足りないか分かります。

## 小さく始める手順

最初の1週間は、AI連携を入れなくても構いません。

1. 問い合わせやタスクの中で、人間判断が必要になったものだけ記録する
2. decision\_type と status を固定する
3. 判断理由と次アクションを必須にする
4. 1日1回、pending と hold を確認する
5. 週に1回、よく止まる判断をFAQや運用ルールへ反映する

これだけでも、AI導入前の詰まりが見えます。

AIエージェントは、その後に入れた方が安定します。

## まとめ

AIエージェント導入前に必要なのは、すべてを自動化することではありません。

まず、判断を見えるようにすることです。

判断ログがあれば、AIに任せる範囲、人間が見る範囲、保留にする理由、次にやることが明確になります。

AIは、判断を消すためのものではありません。

判断すべき場所を見えるようにし、繰り返し判断しなくてよい部分から自動化していく。

この順番で進めると、小さなチームでもAI導入を安定して育てられます。
