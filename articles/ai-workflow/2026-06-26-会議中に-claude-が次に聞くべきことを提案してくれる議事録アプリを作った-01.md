商談や会議の発言をリアルタイムに文字起こしし、その場で Claude が「次に確認すべきこと」を提案、終了時に議事録まで自動生成する Web アプリ **AI-Giziroku** を作りました。

このアプリの一番の売りは、**会議中にリアルタイムで“次の一手”を提案してくれるロール別フィードバック** です。商談の最中に「営業」「エンジニア」など複数の専門視点が同席して耳打ちしてくれる感覚で、議事録を後から作るだけのツールとは決定的に違うところです。

この記事では、実装の背景から、音声処理・話者分離（ダイアライゼーション）・LLM 連携といった要素技術の解説、そして機能ごとの処理フローまでをまとめます。技術的なチャレンジとしては **「1本のミックス音声から複数話者をどう分離するか」**、プロダクトの主役としては **「会議中に効くロール別フィードバック」**、この 2 つを軸に書きます。

## なぜ作ったか

商談や打ち合わせでは「言った・言わない」が後で問題になりがちで、議事録作成も地味に手間です。既存の文字起こしツールは多くありますが、自分が欲しかったのは次の 3 つを **その場で同時に** こなすものでした。

1. 自分と相手の発言をリアルタイムに文字起こしする
2. 会話の流れを読んで「確認漏れ」「次に聞くべき質問」を **会議中に** 提案する
3. 終了したら議事録を自動でまとめる

特に 2 番が肝で、議事録は会議が終わってから作るものですが、提案は会議中にこそ価値があります。そこで AmiVoice（音声認識）と Claude（LLM）を組み合わせ、Next.js の 1 アプリにまとめました。

## 機能の全体像

* 🌟 **ロール別フィードバック（主役）**: 会議中に Claude が営業/エンジニア等の複数視点で「次に聞くべきこと」をリアルタイム提案
* **会議の作成**: タイトル・会議の目的（メタ情報）・フィードバックを求めるロール（営業/エンジニア等）を設定
* **リアルタイム文字起こし**: 自分のマイクと、相手側（オンライン会議のシステム音声）を別系統で取り込み
* **複数話者の自動分離**: 相手側に複数人いても話者ダイアライゼーションで「相手1・相手2…」に分離。誤りは手動修正可能
* **議事録の自動生成**: 終了時にバックグラウンドで Markdown 議事録を生成（議中の提案メモも反映）

## アーキテクチャ

全体構成は次の通りです。音声認識はリアルタイム性を優先し、**サーバを経由せずブラウザから直接 AmiVoice の WebSocket へ** 接続します。確定した発言テキストだけを自前の API 経由で DB に保存し、LLM 呼び出し（提案・議事録）はサーバ側で行います。

なぜサーバを挟まないのか。理由は 2 つあります。1 つは往復遅延の削減（リアルタイム性）。もう 1 つは、**Next.js の Route Handler が WebSocket をサポートしていない** ためです。AmiVoice の APP\_KEY はクローズド運用（社内 LAN / VPN 前提）としてブラウザに配布する割り切りをしています。公開運用するなら短期トークンを返す方式に差し替える必要があります。

## 音声取り込みの仕組み

### 2 系統で「誰の声か」を分ける

最初の設計では、話者の区別を **音源** で行いました。

* **自分の声** = `getUserMedia()` で取得したマイク入力 → `speaker = self`
* **相手の声** = `getDisplayMedia()` で取得した画面共有のシステム音声（ループバック） → `speaker = partner`

オンライン会議では、相手の声はスピーカーから出る＝システム音声として拾えます。Chrome の仕様上 `getDisplayMedia` は video が必須なので、取得後に video トラックは即破棄し、audio トラックだけを使います。

```
export async function acquirePartnerStream(): Promise<MediaStream> {
  const display = await navigator.mediaDevices.getDisplayMedia({ video: true, audio: true });
  display.getVideoTracks().forEach((t) => t.stop()); // video は不要
  const audioTracks = display.getAudioTracks();
  if (audioTracks.length === 0) {
    throw new Error("相手音声が共有されていません。「音声を共有」を ON にしてください。");
  }
  return new MediaStream(audioTracks);
}
```

### AudioWorklet で PCM に変換

AmiVoice のリアルタイム API は PCM 系フォーマットを受け取ります。そこで Web Audio API の `AudioWorklet` を使い、マイク/ループバックの Float32 サンプルを **16kHz / 16bit / モノラル / リトルエンディアン** の PCM に変換します。`ScriptProcessorNode` は非推奨かつメインスレッド負荷が高いため避けました。

worklet 側のポイント:

* 入力サンプルを 16kHz にダウンサンプル（線形間引き）
* `Int16Array` にスケール（`clamped * 0x7fff`）
* 20ms（320 サンプル）を 1 フレームとし、**100ms（5 フレーム）まとめて** メインスレッドへ転送（Transferable で渡してコピーを避ける）

```
// public/worklets/pcm-worklet.js（抜粋）
emitSample(sample) {
  const clamped = Math.max(-1, Math.min(1, sample));
  this.currentFrame[this.frameIdx++] = (clamped * 0x7fff) | 0;
  // ... 5フレーム溜まったら port.postMessage(buffer, [buffer])
}
```

メインスレッドでは、受け取った PCM の先頭に 1 バイト `0x70`（ASCII の `p`）を付けて WebSocket にバイナリ送信します。これが AmiVoice の音声データフレームの書式です。

### AmiVoice WebSocket プロトコル

AmiVoice のリアルタイム API は、テキストフレームでコマンド、バイナリフレームで音声をやり取りします。

| 種別 | 内容 |
| --- | --- |
| 開始コマンド | `s LSB16K -a-general authorization=<KEY> resultUpdatedInterval=400` |
| 音声送信 | バイナリ先頭 1 バイトに `p`(0x70) + PCM |
| 終了 | テキスト `e` |
| 受信 `s` | 開始 ACK（`code` が非空ならエラー） |
| 受信 `U` | 中間結果（未確定） |
| 受信 `A` | 確定発話（最終結果） |
| 受信 `e` | 終了 ACK |

`LSB16K` は「16kHz / 16bit / LE PCM」を表す単一トークンです。中間結果（`U`）は画面の最終行に上書き表示し、確定発話（`A`）のみを DB に保存します。

## 1本のミックス音声から複数話者を分離する

### 当初の設計の限界

「音源で話者を分ける」方式には致命的な穴がありました。**オンライン会議で相手側に 3 人いても、全員が 1本のループバック音声にミックスされて届く** のです。つまり相手側は何人いても `partner` 一括りにしかならない。これは音源で分ける方式の構造的な限界です。

### 解決策: AmiVoice の話者ダイアライゼーション

調べたところ、AmiVoice のリアルタイム WebSocket API は **話者ダイアライゼーション（話者分離）** に対応していました。開始コマンドに `segmenterProperties` を加えるだけで有効化でき、認識結果のトークンごとに `speaker0` / `speaker1` … というラベルが付きます（最大 20 話者）。

そこで **相手側（partner）の WebSocket セッションのみ** ダイアライゼーションを有効化しました。自分のマイクは単一話者なので不要です。

```
let startCmd = `s LSB16K ${ENGINE} authorization=${APP_KEY} resultUpdatedInterval=400`;
if (diarize) {
  // 値にスペースを含むため AmiVoice の書式に従い二重引用符で囲む
  startCmd += ` segmenterProperties="useDiarizer=1 diarizerAlpha=${diarizerAlpha}"`;
}
```

確定発話（`A`）の JSON は `results[].tokens[].label` に話者ラベルを持ちます。1 発話の中に複数ラベルが混ざることもあるため、**最頻ラベル** を採ってその発話の話者としています。

```
function extractFinal(payload: unknown): { text: string; label?: string } {
  const p = (payload ?? {}) as AmiVoiceFinal;
  const counts = new Map<string, number>();
  for (const r of p.results ?? []) {
    for (const tk of r.tokens ?? []) {
      if (typeof tk.label === "string" && /^speaker\d+$/.test(tk.label)) {
        counts.set(tk.label, (counts.get(tk.label) ?? 0) + 1);
      }
    }
  }
  // 最頻ラベルを採用
  let label: string | undefined, best = 0;
  for (const [k, n] of counts) if (n > best) { best = n; label = k; }
  return { text: p.text ?? "", label };
}
```

### 話者キーの設計

アプリ内では「音源（source）」と「話者キー（speaker key）」を分けて扱うことにしました。

* `source`: `self` / `partner` … どの認識パイプラインか（ステータス表示や中間結果の振り分けに使う）
* `speaker key`: 最終的に発言へ付与する話者識別子
  + 自分 → `self`
  + 相手 → ダイアライゼーションラベルを変換した `partner-0` / `partner-1` …

ラベル変換と表示名・配色は `lib/speakers.ts` に集約しました。表示名は未設定なら `自分` / `相手1` / `相手2`… の既定名を返します。

```
export function diarizerLabelToKey(label?: string | null): string {
  const m = label ? /^speaker(\d+)$/.exec(label) : null;
  return m ? `partner-${m[1]}` : "partner-0";
}

export function defaultSpeakerName(key: string): string {
  if (key === "self") return "自分";
  const i = partnerIndex(key);             // "partner-2" → 2
  return i !== null ? `相手${i + 1}` : key; // → "相手3"
}
```

## 自動分離の弱点を、感度調整と手動修正で埋める

1本の混合音声からの推定なので、ダイアライゼーションは万能ではありません。声質が近い／同時発話が多いと精度が落ちます。実際にテストすると **「3 人目が出てこない（別人が同じ扱い）」** ことも **「1 人が複数に割れる」** こともありました。これは UX で吸収する方針にしました。

### 感度調整スライダー

ダイアライゼーションの `diarizerAlpha` は「新しい話者の出やすさ」を制御するパラメータです。大きいほど新話者が出やすくなります（AmiVoice 既定は `1e-10`）。これを **録音画面のスライダー** で `α = 1e^{exp}` として調整できるようにしました。設定は `localStorage` に保存して次回も引き継ぎます。

* 同じ人が別人に分かれすぎる → 「少なめ」へ
* 別人が同じ扱いで人数が足りない → 「多め」へ

### ハマりどころ: 録音中に感度を変えると画面共有ダイアログが再表示される

`diarizerAlpha` は WebSocket の **開始コマンドで決まる** ため、変更には再接続が必要です。しかし単純に「相手認識を止めて再開」すると、`getDisplayMedia` が再度走り、**画面共有の選択ダイアログが会議中に再表示** されてしまう。これは致命的に邪魔です。

解決策は「ストリーム取得」と「認識セッション開始」の分離でした。取得済みの `MediaStream` を保持し、認識セッション（WebSocket + AudioWorklet）だけを貼り直します。stop 時にストリームのトラックを止めない `keepStreamOnStop` フラグを用意し、再適用時はストリームを使い回します。

```
const reapplyPartner = useCallback(async () => {
  const h = partnerHandleRef.current;
  if (!h) return;
  partnerHandleRef.current = null;
  await h.stop().catch(() => {});  // keepStreamOnStop によりストリームは生存
  await startPartner();            // 保持済みストリームを再利用
}, [startPartner]);
```

### 手動修正

それでも誤分離は残るので、最後は人が直せるようにしました。

* **話者名のリネーム**: 入力欄で表示名を変更すると全発言に反映（`PATCH /api/meetings/[id]` で `speakerLabels` を更新）
* **発言ごとの話者付け替え**: 各発言のセレクトで別話者を選択。`＋ 新しい話者` で新規 `partner-N` も発行できる（`PATCH /api/transcripts/[id]`）

これらの UI 部品（話者バッジ・名前編集・付け替えセレクト）は共通化し、**録音画面と会議終了後の詳細画面の両方** で使えるようにしています。

## 機能ごとのフロー

### 会議の開始から終了まで

## 会議中に出るロール別フィードバック（このアプリの主役）

複数話者の分離は「正確に記録する」ための機能ですが、このアプリで一番価値があると思っているのは **会議中に Claude が“次の一手”を提案してくれる** ことです。議事録は会議が終わってからでも作れますが、提案は **その場で出てこないと意味がない**。ここを丁寧に作り込みました。

### 何が嬉しいのか

商談中、頭の中は「次に何を聞くべきか」「条件を詰め忘れていないか」でいっぱいになります。これを **複数の専門視点** で同時に補佐してくれるのがロール別フィードバックです。

* **営業**: クロージング、関係構築、提案の説得力の観点
* **エンジニア**: 技術的実現性、工数、リスクの観点
* **経営/PM/マーケ/デザイナー**: それぞれの専門視点

会議の場に「営業の先輩」「技術の相談相手」が同席して、リアルタイムに耳打ちしてくれるイメージです。録音画面で「フィードバック取得」を押すと、選んだロールごとに **タブ切り替え** で提案が表示されます。  
![エンジニアタブに切り替えると、技術視点での確認事項・リスク・次の一手が表示される](https://static.zenn.studio/user-upload/8a787de2dc24-20260626.png)

### 仕組み: ロールの「視点」をシステムプロンプトに注入する

ロールは `lib/roles.ts` に `id / label / perspective`（その職種ならではの観点）として定義しておき、選択されたロールの perspective をシステムプロンプトに差し込みます。出力フォーマットもロールごとのブロックに固定し、画面側でタブにパースしやすくしています。

```
// lib/roles.ts（抜粋）— 選択ロールの視点を箇条書きで注入
const roleList = roles.map((r) => `- ${r.label}: ${r.perspective}`).join("\n");
return `あなたは商談・会議に同席する優秀なアシスタントです。
直近の会話を読み、指定された各ロールの視点から、会議を有利に進めるための
簡潔なフィードバックを返してください。${contextSection(description)}

${roleList}

### {ロール名}の視点
- **確認不足 / 次に聞くべき質問**: 1〜2個
- **リスク**: 該当があれば（なければ「特になし」）
- **次の一手**: そのロールとして取るべきアクション（1〜2個）`;
```

ロールを 1 つも選ばなければ、従来形式（ロール非依存）の汎用提案にフォールバックします。また会議作成時に入力した **会議の目的（メタ情報）** も `contextSection()` でプロンプトへ差し込み、的外れな提案を減らしています。

### リアルタイムに効かせるための工夫

* **直近ログだけを渡す**: 毎回 全文を投げると遅く・高コストになるため、`recentLimit`（既定 40 発話）で直近の会話に絞って送ります。会議が長くなっても提案の応答性が落ちません。
* **プロンプトキャッシュ**: システムプロンプトに `cache_control: { type: "ephemeral" }` を付与。会議中は同じシステムプロンプトで何度も提案を取得するため、キャッシュが効いてレイテンシ・コストを抑えられます。
* **話者名を反映**: プロンプトに渡す会話ログは話者キーを表示名へ変換するので、「相手1 が予算に言及」のように **誰の発言かを踏まえた** 提案になります。

取得した提案は `ClaudeFeedback` として履歴に残し、**会議終了時の議事録生成にも“議中メモ”として渡します**。会議中の気づきが最終成果物にも引き継がれる設計です。

## 議事録のバックグラウンド生成

議事録生成は時間がかかるので、ユーザを待たせない設計にしました。Next.js の `after()` を使い、**レスポンス（202）を返した後** に生成を走らせます。状態は `Meeting.summaryStatus`（`processing` / `done` / `error`）で管理し、詳細画面はポーリングで完了を検知して自動表示します。

```
// app/api/claude/summary/route.ts（抜粋）
await prisma.meeting.update({ where: { id }, data: { summaryStatus: "processing" } });

after(async () => {
  try {
    const summaryText = await requestSummary(transcriptInput, feedbackInput, {
      roleIds, description, speakerLabels,
    });
    await prisma.meetingSummary.create({ data: { meetingId, summaryText } });
    await prisma.meeting.update({ where: { id }, data: { summaryStatus: "done" } });
  } catch (e) {
    await prisma.meeting.update({ where: { id }, data: { summaryStatus: "error" } });
  }
});

return NextResponse.json({ status: "processing" }, { status: 202 });
```

プロンプトに渡す会話ログは、話者キーを **表示名に変換** してから整形します。これにより議事録や提案でも「相手1」やリネーム後の名前が使われます。

```
function transcriptsToText(transcripts, labels) {
  return transcripts.map((t) => `${speakerName(t.speakerType, labels)}: ${t.text}`).join("\n");
}
```

## データモデル

話者まわりで増えたのは Meeting の `speakerLabels`（話者キー→表示名の JSON 文字列）です。発言の話者は `Transcript.speakerType` に `self` / `partner-N` として持ちます。

話者名を別テーブルにせず Meeting の JSON 列にした理由は、SQLite + Prisma の軽量運用で十分だったこと、そして「会議単位の話者名マップ」という性質が JSON と相性が良かったためです。発言の付け替えは `speakerType` の UPDATE、リネームは `speakerLabels` の UPDATE で完結します。

## 設計判断と学び

* **音源での分離は早いが浅い**: 自分/相手の 2 値なら音源で十分。しかし「相手が複数」になった瞬間に破綻する。最初から「話者は何人でも増える」前提でキー設計しておくべきだった（途中で `"self" | "partner"` 型を文字列キーに作り替えた）。
* **自動 + 手動のハイブリッドが現実解**: 1本のミックス音声からの分離は原理的に限界がある。完璧を目指すより、感度調整と手動修正で人が直せる導線を用意するほうが実用的。
* **リアルタイム UX の細部**: 「再接続のたびに画面共有ダイアログが出る」のような細部が体験を壊す。ストリームと認識セッションのライフサイクルを分離して回避した。
* **待たせない LLM 連携**: 重い生成は `after()` でバックグラウンド化し、状態をステータス列で管理すると UX が安定する。

## まとめ

このアプリの主役は **会議中にリアルタイムで効くロール別フィードバック** です。ロールの「視点」をシステムプロンプトに注入し、直近ログだけを渡し、プロンプトキャッシュで応答性を確保することで、商談の最中に複数の専門視点が耳打ちしてくれる体験を実現しました。

そして、その提案を支える「正確な記録」のために、「自分と相手」しか分けられなかった文字起こしを AmiVoice の話者ダイアライゼーションで複数話者対応にし、自動分離の弱点を感度スライダーと手動修正で埋めました。ブラウザから直接 WebSocket でリアルタイム認識し、LLM 連携はサーバ側でバックグラウンド処理、という役割分担に落ち着いています。

同じように「会議中に LLM の補佐を効かせたい」「オンライン会議のシステム音声を解析したい」「リアルタイム文字起こし + LLM をやりたい」という方の参考になれば幸いです。
