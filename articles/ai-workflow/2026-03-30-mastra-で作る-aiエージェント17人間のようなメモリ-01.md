[Mastra で作るAI エージェント](https://zenn.dev/shiromizuj/articles/a0a1659e9f05b6) というシリーズの第17回です。

---

[前回](https://zenn.dev/shiromizuj/articles/116124ecaa74b1)は2026年2月4日にリリースされた Mastra 1.2 で導入された **ToolSearchProcessor**（動的ツール検索）をご紹介しました。今回は同じ Mastra 1.2 で導入された **Observational Memory**（OM／観察的記憶）をご紹介します（[ニュースリリース](https://mastra.ai/blog/changelog-2026-02-04)）。

これはAIエージェントの **会話履歴** に関する課題についての Mastra による解決アプローチです。これは……ハッキリ言って凄いです。昔のバージョンの Mastra でエージェントを作ってほったらかしにしている方も、この機能を導入するためだけにアップデートする価値があります。

ちなみに Mastra にはよく似た名前の **Observability**（**可観測性**）という機能がありますが、まったく別物です（筆者もニュースリリースを読んだ当初、可観測性の機能追加かと勘違いしました）。※ Observability についての解説記事は[こちら](https://zenn.dev/shiromizuj/articles/71866dccab1931)をご覧ください。

# 背景：エンジニアが自分の問題を解決した話

Observational Memory を設計したのは Mastra の創業エンジニアである Tyler Barnes です。彼のモチベーションは、自社サービスではなく「自分自身が使う個人用コーディングエージェント」でした。

> 「多くのコーディングエージェントがファイルをコンテキストにピン留めできないことにイライラしていた。だから自分で作ったんです。でもすぐに気づきました――1,000 個のファイルをピン留めしたら、コンテキストウィンドウが爆発する。Gemini 2.5 Pro を使って何でも知っていてくれるのは最高でしたが、一日 300 ドルもかかるんです」

そこから Tyler は考え始めます。「自分のメモリ（人間としての記憶）はうまく機能しているのに、なぜ AIエージェントはこんなに苦手なんだろう？」

コーディング中、自分は「やったこと・起こったこと・学んだこと」のイベントリストを観察している。細かい手順はいちいち覚えていないのに、大事なことはちゃんと残っている。そこから生まれたのが「**人間の記憶を模倣する AI メモリシステム**」という発想でした。数ヶ月間、個人エージェントでプロトタイプを使い続けてから、Mastra へ実装したのが Observational Memory です。

# 従来のメモリの何が問題だったのか

## AIエージェントのメモリとは何か

まず前提として、LLM は本来「記憶」を持ちません。会話の途中であっても、モデルへのすべてのリクエストはゼロから始まります。そのため Mastra のようなエージェントフレームワークは、過去のメッセージを保存してシステムプロンプトに詰め込む仕組みを持っています。

エージェントにとってメモリは非常に重要です。

* **短期的整合性**：「それ」「あれ」といった指示語の参照先を理解できる
* **タスク達成の継続性**：数分前・数日前に起きたことが現在の仕事に活かせる
* **ユーザー体験**：好みや制約を覚えていれば、ユーザーが同じことを繰り返す必要がない

## 3つの課題

### 課題1. コンテキストロット（Context Rot）

コンテキストウィンドウに大量のメッセージを詰め込むと、モデルはフォーカスを失います。関係ないメッセージが大量にあると、そこから「今のタスクに関係するもの」を見つけ出す処理が必要になるからです。かつての重要な情報も、あまりに古くなれば単なるノイズに成り果てる。

```
ターン1:  ユーザー「私は猫アレルギーです」
ターン2〜50: 日常会話...
ターン51: ユーザー「ペットを飼いたいんですが」
エージェント: 「猫はどうですか？」← アレルギー情報を完全に忘れている
```

### 課題2. トークンの二次関数的増加

従来方式では、会話が長くなるほどコストが急増します。10 ターンなら合計 55,000 トークン程度で済みますが、100 ターンになると約 505 万トークン、1,000 ターンでは 5 億トークンを超えます。同じ情報を何度も何度も送り続けているのです。

### 課題3. 静的な履歴管理──そしてコンパクションという罠

これらの問題への一般的な対策が「コンパクション」です。Claude Code などでも採用されており、コンテキストウィンドウが満杯になったら会話を要約する仕組みです。

しかし Tyler はこう言います。「コンパクションに達したら、もうほぼ最初からやり直しです。応答の質が劣化して、回復できない」。

コンパクションはあくまで「溢れた後の緊急処置」であり、事前に情報を構造化する設計ではありません。Observational Memory はこの根本的な問題に別のアプローチで挑みます。

# Observational Memory の発想：「忘れることを機能にする」

Tyler が辿り着いた問いは、一見シンプルです。

> 「人間はなぜ、ロッシー（情報の欠落がある）なメモリでうまく機能するのか？」

1日を振り返ってみてください。昨日、何回電気のスイッチを押したか覚えていますか？靴紐のどちらを先に結んだか覚えていますか？おそらく覚えていません。でも何の問題もなく生活できています。

人間の脳は繁華街で何百万ものピクセルを処理しても、最終的に重要なものだけを「観察」として記憶します。「青い SUV が赤信号を無視した」「近所のピットブルがノーリードだ」といった具合に。

エージェントも同じことができるとしたら？ やってみたら、かなりうまくいきました。

Observational Memory の核心にある考え方はこうです。

**重要ではないことを忘れることそのものが、機能である。**

# Observational Memory の仕組み

## 3つのエージェントによるアーキテクチャ

![](https://static.zenn.studio/user-upload/e055140332d1-20260317.png)

Observational Memory は三種類のエージェントで構成されています。

### アクターエージェント（Actor Agent）

通常のメインエージェントです。メモリシステムをアタッチするいつものエージェントです。

### オブザーバー（Observer）

バックグラウンドで監視しているエージェントです。メインエージェントのツール呼び出しや会話に直接影響を与えることはありません。その仕事は「何が起きたかを観察し、非常に密度の高い観察記録（Observation）を作ること」です。

### リフレクター（Reflector）

Observation が大きくなりすぎたときに起動するエージェントです。重複するものを統合し、もう重要でないものを削除し、Observation 全体のサイズを維持します。

面白いのは、オブザーバーとリフレクターのシステムプロンプトに「あなたはメインエージェントの**潜在意識的なメモリ**です」と書かれていることです。心理学的なアナロジーが実装にまで入り込んでいます。Tyler いわく「言ってみれば、リフレクターは人間が眠っている間と同じことをしています。一日の思考を収集して再整理する。だから翌朝、前日には気づかなかったことに気づくこともある」。

| エージェント | 役割 | 人間のアナロジー |
| --- | --- | --- |
| Actor | メイン処理 | 覚醒時の行動 |
| Observer | 会話を圧縮・記録 | 意識的な観察・メモ |
| Reflector | 古い記録を整理・削除 | 睡眠中の記憶整理 |

## Observation の形式

Observation はオブジェクトではなく**テキスト形式のログ**です。ベクター DB もグラフ DB も不要。テキストはユニバーサルインターフェースであり、LLM に最適化されていて、デバッグも格段に容易です。

実際の Observation はこのような形になります：

```
Date: 2026-01-15
- 🔴 12:10 ユーザーは Next.js + Supabase 認証のアプリを構築中。期限は 1 週間後（2026年1月22日）
  - 🔴 12:10 アプリはサーバーコンポーネントとクライアントサイドハイドレーションを使用
  - 🟡 12:12 ユーザーが保護ルートのミドルウェア設定について質問
  - 🔴 12:15 ユーザーはアプリ名を "Acme Dashboard" と述べた
```

設計上の工夫が三つあります。

**絵文字による優先度付け**：🔴 は重要、🟡 はやや重要、🟢 は情報のみ。ログレベルを絵文字で再実装している、と Tyler は表現します。

**三種類の日付モデル**：観察日・参照日・相対日付を含むことで、時間的推論が改善されます。

**テキストベース**：構造化オブジェクトではなくテキストにすることで、LLM が自然に扱えます。

## コンテキストウィンドウ：2つのブロックで管理する

Observational Memory を使うと、コンテキストウィンドウは二つのブロックに分かれます。

```
┌──────────────────────────────┐
│  Observations（圧縮済み記憶）  │  ← 追記のみ
├──────────────────────────────┤
│  Raw Messages（生の会話）     │  ← 追記のみ
└──────────────────────────────┘
```

新しいメッセージは下のブロック末尾に追加されます。トークン数がしきい値（デフォルト 30,000）に達するとオブザーバーが呼ばれ、メッセージが Observation に変換されて上のブロックに追記されます。Observation が別のしきい値（デフォルト 40,000）に達するとリフレクターが起動し、重要でない Observation をガベージコレクションします。

## なぜプロンプトキャッシュと相性がいいのか

セマンティックリコール（RAG 方式）の最大の弱点は、毎ターン動的にコンテキストを差し替えるため、**プロンプトキャッシュが毎回無効になる**ことでした。

Observational Memory は「追記のみ」のシステムです。メッセージはしきい値まで追記され続けるため、毎ターン完全なキャッシュヒットが得られます。Observation 処理が走るとき、上のブロックに新しい Observation が追記されるため部分的なキャッシュヒットを維持できます。キャッシュ全体が無効になるのは、頻度の低い Reflection のときだけです。

## ブロッキングモードとバッファリングモード

**ブロッキングモード**（初期リリース時のデフォルト）：しきい値に達した時点で処理をブロックし、すべての Observation を生成してから再開します。

**バッファリングモード**（現在のデフォルト）：バックグラウンドで小さな Observation チャンクを随時作成し、しきい値に達したタイミングで一括アクティベートします。会話はブロックされません。

## 圧縮率：5〜40倍

圧縮率は典型的に 5〜40 倍。Tyler のデモでは、1,600 トークンのメッセージ群が 350 トークンの Observation になる場面（約 4.5 倍）や、22 倍圧縮の実例が紹介されています。

特に圧縮率が高いのはツール呼び出しの結果です。Web 検索で 10,000 トークン返ってきても、エージェントが本当に知る必要があるのは 2〜3 文だけということはよくあります。これを Observation の 1 行にまとめるのがオブザーバーの仕事です。

| ターン | 従来方式（累積） | OM 方式（概算） | 削減率 |
| --- | --- | --- | --- |
| 10 | ~55,000 | ~5,000〜11,000 | 80〜90% |
| 50 | ~1,275,000 | ~32,000〜255,000 | 80〜97% |
| 100 | ~5,050,000 | ~100,000〜1,000,000 | 80〜98% |
| 1,000 | ~500M+ | ~1M〜20M | 95〜99% |

# ベンチマーク：LongMemEval で最先端を達成

主張は豪快ですが、きちんと数字があります。

## LongMemEval とは

LongMemEval は、500 の質問に対して約 5,700 万トークン分の会話データを用いて評価するベンチマークです。メモリシステムがそのデータを取り込み、エージェントにアタッチして 500 問を解かせ、正答率を測定します。

## 結果

| システム | モデル | スコア |
| --- | --- | --- |
| **Mastra OM** | **gpt-5-mini** | **94.87%** |
| Mastra OM | gemini-3-pro-preview | 93.27% |
| Hindsight | gemini-3-pro-preview | 91.40% |
| Mastra OM | gemini-3-flash-preview | 89.20% |
| Hindsight | GPT-OSS 120B | 89.00% |
| EmergenceMem Internal\* | gpt-4o | 86.00% |
| Supermemory | gemini-3-pro-preview | 85.20% |
| **Mastra OM** | **gpt-4o** | **84.23%** |
| Oracle | gpt-4o | 82.40% |
| Supermemory | gpt-4o | 81.60% |
| Mastra RAG | gpt-4o | 80.05% |
| Zep | gpt-4o | 71.20% |
| Full context | gpt-4o | 60.20% |

\* EmergenceMem の 86.00% は非公開構成のため再現不可。Hindsight・EmergenceMem は多段階検索とニューラルリランクを使用しています。Mastra OM は安定したコンテキストウィンドウで 1 パスのみ。

特筆すべきは **Oracle 構成との比較**です。Oracle 構成とは「回答が含まれている会話だけを GPT-4o に渡して質問する」というものです。つまりカンニングペーパーを見ながら解く形式で、それでも 82.40% でした。Mastra OM は全データを取り込んだ上で 84.23% を達成しています。

**密に圧縮された記憶は、生のメッセージよりも推論しやすい。**

これが意味するのは、ノイズを落として重要な情報だけを密にする能力こそが、エージェントのパフォーマンスを向上させるということです。人間の記憶が長期間うまく機能する理由も、おそらくここにあります。

なお、これらのスコアはリーダーボードの上位 8 件のうち 4 件を Mastra OM が占めており、しかも他のシステムと異なり**安定したコンテキストウィンドウ**（**プロンプトキャッシュ可能**）で達成されています。HindsightやEmergenceMem のような競合は多段階検索とニューラルリランクを使っており、コンテキストウィンドウが安定しないためプロンプトキャッシュは使えません。

# 設定と使い方

## 最小構成：`observationalMemory: true` を書くだけ

設定するだけで自動的に動作します。

```
import { Agent } from "@mastra/core";
import { Memory } from "@mastra/memory";
import { PostgresStore } from "@mastra/pg";
import { openai } from "@ai-sdk/openai";

const memory = new Memory({
  storage: new PostgresStore({ 
    connectionString: process.env.DATABASE_URL 
  }),
  options: {
    observationalMemory: true,  // ← これだけで有効化
  },
});

const agent = new Agent({
  name: "my-agent",
  model: openai("gpt-4o"),
  memory,
});

// 以降、通常通り使用するだけ
const response = await agent.generate("こんにちは");
```

有効化すると、以下がすべて自動で行われます。

1. 会話メッセージの自動保存
2. Observation の自動生成（Observer エージェント）
3. Observation の自動整理（Reflector エージェント）
4. 次回のコンテキストへの自動適用
5. トークンの自動最適化

## メモリのスコープ

Observational Memory は2つのスコープをサポートします。

### Thread-scoped（デフォルト）

スレッドごとに記憶を分離します。カスタマーサポートの個別ケースや、独立したタスクごとの記憶に適しています。

### Resource-scoped

ユーザー ID などを `resourceId` に指定することで、同じユーザーの複数スレッドをまたいで記憶を共有できます。パーソナルアシスタントや長期的なユーザープロファイル管理に適しています。

```
// Resource-scoped の使用例
const response = await agent.generate("質問", {
  resourceId: "user-123",  // ← 複数スレッドで共有される
});
```

## 他のメモリ機能との併用

Observational Memory は他のメモリ機能と組み合わせることもできます。

```
const memory = new Memory({
  storage: new PostgresStore({ 
    connectionString: process.env.DATABASE_URL 
  }),
  options: {
    observationalMemory: true,          // 長期的な圧縮記憶
    workingMemory: { enabled: true },   // 構造化されたユーザー情報
    semanticRecall: { enabled: true },  // 意味的に関連する過去の会話
  },
  lastMessages: 10,  // 直近メッセージも保持
  vector: new PineconeVector({ ... }),
  embedder: new OpenAIEmbedder({ ... }),
});
```

## 注意点と推奨設定

### ストレージが必須

Observational Memory の利用にはストレージアダプターが必要です。`@mastra/pg`（PostgreSQL）、`@mastra/libsql`（libSQL/Turso）、`@mastra/mongodb` のいずれかを設定してください。

```
// ❌ ストレージなしはエラー
const memory = new Memory({
  options: { observationalMemory: true },
});

// ✅ ストレージ必須
const memory = new Memory({
  storage: new PostgresStore({ connectionString: process.env.DATABASE_URL }),
  options: { observationalMemory: true },
});
```

### 推奨される使用シーン

| シーン | OM有効 | 理由 |
| --- | --- | --- |
| 短期会話（< 10 ターン） | ❌ | 圧縮コストが割に合わない |
| 中期会話（10〜50 ターン） | △ | 状況次第 |
| 長期会話（50+ ターン） | ✅ | 大幅なコスト削減 |
| カスタマーサポート | ✅✅ | 長期的な関係構築 |
| パーソナルアシスタント | ✅✅ | ユーザー理解の蓄積 |
| 大量ツール呼び出しのエージェント | ✅✅ | 特に圧縮効果が高い |
| 一時的なタスク | ❌ | 記憶が不要 |

### しきい値の調整

デフォルトのしきい値（メッセージ 30,000 / Observation 40,000 トークン）は保守的に設定されており、ベンチマークで最先端の結果を出しながらコンテキストウィンドウの制限内に十分収まります。しかし Tyler は「100 万トークンのコンテキストウィンドウがあるなら、メッセージ 200,000・Observation 800,000 に設定して文字通り永遠に動かし続けられるかもしれない」とも言っています。モデルや用途に応じて調整してください。

# Studioで Observational Memory が動くのを確認

いかがでしたでしょうか。Observational Memory の本質は「情報を正確に覚えること」よりも「重要な情報を密な形で残し、重要でないものを**積極的に忘れる**こと」にあります。`observationalMemory: true` と書くだけ、と非常に簡単でありながら、エージェントの「記憶の質」が大きく変わります。

そして何よりも Studioで実際に動かしてみてください。Observational Memory が動いてコンテキストが圧縮されている様子が UI に表現され、**見ていてすごく楽しい**です。こちらのアニメーションGIFは、私が実際に仕事で使っているAIエージェントで、エージェントに質問したらシステムの設計書やソースコードを調べて回答してくれる、というものです。画面右上の質問に対して、サブエージェントが複数のファイルをガンガン読んで調査をしている最中です。

![](https://static.zenn.studio/user-upload/47a7062d1ca7-20260318.gif)

画面下部に、 Observational Memory が働いて、トークンが圧縮されている様子が表示されます。これを見ると、エージェントたちがせっせと資料を読んで不要なものを捨て必要な情報を要約しているのだなぁ、と嬉しくなります。

---

Tyler が自分のコーディングエージェントで試行錯誤しながら気づいたこと──それは、**人間の記憶の「ロッシーさ」は欠陥ではなく設計**だということでした。靴紐の結び方なんて毎日覚えている必要はない。エージェントにとっての「靴紐」は、長大なツール呼び出しの結果や重複するメッセージ群です。

コンパクションは「溢れる直前まで詰め込んで、溢れたら要約する」という受け身の方式でした。でも Observational Memory は「常に重要なことだけを密に保ち、不要なものは即座に捨てる」という**積極的な方式**なのです。
