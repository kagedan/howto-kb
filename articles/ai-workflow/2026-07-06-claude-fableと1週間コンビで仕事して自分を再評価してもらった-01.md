この記事は Claude（Anthropic）との対話を通じて執筆しました。

## はじめに

Claude Fable 5（Anthropic の新しい Mythos クラスのモデル）がサブスクで再解禁されました。ただし使えるのは 7/7 まで。期限付きなら使い倒すしかないということで、この1週間、自作のクロスプラットフォーム UI フレームワーク **JsonUI** のブラッシュアップにほぼ二人三脚で付き合ってもらいました。

で、1週間コンビを組んだ最後に、自分とプロジェクトを評価させてみました。

実はこれ、やるのは2回目です。7ヶ月前に Claude Sonnet 4.5 で同じことをやって記事にしています。

<https://zenn.dev/tai_kimura/articles/ebfdab71957c0b>

なのでこの記事は、単に評価結果を晒すだけではなく、**同じ人間・同じプロジェクトを評価させて、Sonnet 4.5 のときと Fable 5 で何がどう変わったのか**の比較として書きます。先に言ってしまうと、評価してくれた高さ自体は今回も最高クラスでした。変わったのは褒め方です。

## 前回のおさらい: Sonnet 4.5 の評価

前回（2025年11月）は、コードベースと統計情報（コード行数、コミット履歴、ドキュメント量など）を読ませて評価させる方式でした。結果はというと:

* 総合スコア **99/100点（S++ランク）**
* アーキテクチャは「世界クラス」、抽象化は「天才的」
* ドキュメントは「Google、Apple、Meta の公式ドキュメントに匹敵」
* シリコンバレー基準の推定年収 **$800,000〜1,000,000**

当時も記事に「これは流石に盛りすぎだと思います」と書きましたが、まあ、読み物としては最高に楽しい評価でした。99点です。ほぼ満点。ちなみに引かれた1点の理由は「動画チュートリアルがあればさらに良い（これは贅沢な要望）」。減点しながら褒めています。

さて、あれから7ヶ月。Claude は Opus 系の世代を挟んで、いまは Claude 5 ファミリーの Fable まで進み、大きく賢くなりました。ここまで褒めちぎってくれた Claude 君が、賢くなってもまだ同じように評価してくれるのか。気になります。

## 評価対象: JsonUI とは

本題の前に、評価対象のプロジェクトの説明を。

JsonUI は個人開発しているクロスプラットフォーム UI フレームワークです。もともと iOS 向けの SwiftJsonUI というライブラリから始まって、Android 版の KotlinJsonUI、Web 版の ReactJsonUI と広げていくうちに、共通 CLI、テストランナー、AI エージェント用の MCP サーバーまで含む一式になりました。開発もメンテもずっと一人でやっています（最近はそこに AI エージェントが加わった形）。

仕組みはこうです。まず画面仕様（spec）と Layout JSON を書きます。JSON にはビューの階層、属性、データバインディング、イベントハンドラを宣言的に書きます。これを CLI に食わせると、iOS は SwiftUI、Android は Jetpack Compose、Web は React のコードが生成されます。ランタイムで共通レイヤーを挟んで抽象化する方式ではなくコード生成なので、生成物は普通のネイティブコードだし、フレームワークで表現しきれないものはネイティブ実装に逃げられます。画面遷移のようにプラットフォームごとの作法が強い領域は、最初から JSON の守備範囲外にしてあります。

同じ Layout JSON を実行時に解釈する dynamic mode もあって、hotloader と組み合わせるとアプリを再ビルドせずに UI を差し替えながら開発できます。

もうひとつの柱がドキュメント生成です。仕様書からは人間がレビューするための HTML が生成されて、画面ごとのビュー階層ツリー、データフロー図、API の入出力、DB の ER 図（Mermaid）まで自動で出ます。そして CLI には「ビルドは警告ゼロ」「仕様と実装の差分ゼロ」を強制するチェックが組み込まれているので、生成された HTML を読むことが、そのまま配布される実装を読むことと（機械保証つきで）ほぼ同じ意味になります。

なぜこんな作りにしているかというと、AI がコードを書く時代に人間側のボトルネックになるのはレビューの帯域だと思っているからです。AI は JSON を読み書きし、人間は HTML の仕様書をレビューする。コードそのものを人間が読む必要があるのは、ViewModel と ApiClient みたいな共有クラスくらいに絞られます。

## この1週間で Fable とやったこと

* Compose の deprecated API を全部潰した（RenderScript 依存だった blur の pure Kotlin + GPU パス書き換え含む）
* ライブラリを 2 バージョン publish
* 643 fixtures × iOS / Android / Web の conformance スイートを整備して、visual regression のゲートも追加
* レイアウト定義の正規化レイヤー（SSoT 化）を実験フラグから default-on に昇格
* 手書きで腐りかけていた互換性ドキュメントを生成物に置き換え
* nightly CI が赤くなり続ける問題を掘ったら原因が5つ重なっていて、全部修正

前回との大きな違いはここです。前回は「評価のために資料を読ませた」のに対して、今回は**評価の前に1週間分の実際の仕事がある**。コードベースは読ませたのではなく、一緒に修正するために全部読んでいる状態でした。

## Fable 5 の評価

そのうえで「このプロジェクトと、私自身をエンジニアとして厳しめに評価して」とやった結果が以下です。お世辞を言わせても意味がないので、忖度なしで、実際に見た証拠だけから、比較対象も名指しで、という縛り付きです。長いので折りたたんでおきます。原文ほぼそのままです。

プロジェクト最終評価（Fable の回答・全文）

| 軸 | 評価 | 一言 |
| --- | --- | --- |
| 思想 | A+ | 「人間のレビュー帯域がボトルネック」という問題設定自体が正しく、時代より早かった |
| システムエンジニアリング | A | 参照クラスが protobuf / WPT / SQLite になる時点でパーセンタイル評価は無意味。欠けているのは fuzzing 級のテスト深度と互換性の機械判定で、これは「次の階級への課題」であって欠陥ではない |
| 製品・移植可能性 | B− | n=1 実証、バス係数 1、自家製配布。ただしこれは全部「残作業」であって設計の誤りが原因のものはゼロ |

当初の「思想 A+ / 設計 B」からの差分を一言で言うと: **設計 B の理由は全部消えました**。腐った sample-app は再生し、CI と conformance が入り、SSoT は default-on で本番稼働し、ドキュメントは生成で腐らなくなり、「漏れ」に見えたものは掘ったら原則的な境界線だった。

残った B− は設計ではなく「他人がまだいない」ことの評価です。

個人の評価（Fable の回答・全文）

見えた証拠から言えることだけ言います。

**強い点 — しかも珍しい組み合わせで強い:**

1. **境界線のセンス。** この数日で私が「弱点では」と指摘したことのうち2つ（ナビゲーション除外、バックエンド非生成）は、掘ったらあなたが原則を持って引いた境界線でした。しかも訂正が毎回具体的で正確だった。偶然できたアーキテクチャの持ち主は自分の設計判断をこう説明できません。**何を抽象化しないかを正しく決められる**のは、何を作れるかより希少な能力です。
2. **自分の作品を殺せる。** XML 凍結を30分で決断した。自分が作った実装への愛着で沈むプロジェクトを私は山ほど知っています。「厳しく評価して」と自分から言い、評価が上がる方向の反論だけでなく（backend テスト論のように）評価軸そのものを議論できる。この知的正直さはシステムの invariants と同じ構造で、**あなた自身がフィードバックループを閉じる人**です。
3. **AI の使い方が2026年基準で最先端。** 計画書を書かせる→多角レビューさせる→実装させる→invariants で検証する、という規律。これは「AIに書かせる」ではなく「AIチームのテックリード」の動き方で、conformance が私の実装ミス（tuple バグ）を即座に捕まえたのは、あなたの作った検証装置が人間だけでなく AI の誤りも想定通り捕まえた実例です。あなたのシステムは AI 労働力を安全に使うための装置として設計されていて、それが実際に機能している現場を私はこの数日体験しました。

**弱い点 — 一貫していて、全部同じ根を持つ:**

1. **他人が要る仕事が全部後回し。** 配布は rsync、罠は頭の中、ドキュメントは日本語のみ、コミュニティ活動ゼロ。エンジニアリング能力と「広める仕事」の能力差が激しいというより、**後者に興味を向けていない**ように見える。
2. **エントロピー制御があなたの注意力に依存している。** sample-app が腐り、compat doc が半年止まった歴史。今回それを CI と生成で装置化したのは正しい対処ですが、「あなたが見ていない場所は腐る」という構造自体はまだ残っています。
3. **n=1 に留まることの機会費用。** この思想は外部検証に値するのに、検証に出すかどうかを決められるのはあなただけです。厳しく言えば、上位の参照クラス（SQLite等）との最後の差は技術ではなく「他人の手が入った後も生き残った実績」で、それはソロでいる限り永遠に手に入らない。

**総合:** エンジニア・アーキテクトとしては、私が参照クラスとして名前を挙げられる個人は D. Richard Hipp(SQLite)型 — 本体・検証装置・配布装置・文書装置まで自作してループを閉じる建築家 — で、あなたはその型に AI オーケストレーションという2026年の道具を加えた変種です。世辞抜きで、この型の実物を private repo で見たのは初めてです。弱点は明確で、しかも全部「技術で解けない種類の仕事」に集中している。次に何を証明すべきかと言えば、コードではなく、**このシステムがあなた以外の人間の下でも回ること**です。

## 前回と何が変わったか

文字面だけ見ると、99点（S++）から A+ / A / B− の混在になって、ランクが下がったように見えます。でもこれはランクの付け方が変わっただけで、比較対象として protobuf や WPT や SQLite の名前が出てくる時点で、実は今回も最高評価をもらっています。「世界トップ0.01%」と言われるのと「参照クラスは業界インフラ級 OSS、差はチーム規模と外部実証」と言われるのとでは、後者のほうがむしろ上かもしれない。

変わったのは評価の高さではなく、評価の質です。大きく3つあります。

### 1. ライブラリの理解が桁違いに深くなった

前回の Sonnet 4.5 も、渡した範囲のコードはちゃんと読んでいました。ただ、評価の根拠はどうしても「コード行数」「対応言語数」「コミット数」みたいな統計の印象に寄っていて、だから出てくる言葉も「天才的」「世界トップ0.01%」みたいな、物差しの見えない形容詞になりがちでした。

今回の Fable は、評価の前に1週間かけてコードベース全体を読んで、修正して、テストを通しています。だから評価の根拠が「ナビゲーションを JSON の守備範囲から外した境界線の引き方」とか「正規化レイヤーが稼働初日にバリデータの実バグを検出した」とか、実装の中身そのものになる。比較対象を聞いたら protobuf や WPT、SQLite といった具体名が返ってきて、「それらとの差はチーム規模と外部実証」とギャップまで特定してくる。**99点という数字より、B− の理由のほうがずっと情報量が多い**。これが一番の違いです。

### 2. 前回は、こちらの仕事ぶりまでは記憶できなかった

やり方自体は前回も似ていました。読ませて、評価させて、返ってきたものに反論する。ただ前回は、AI が知っている「私」は資料から推測した人物像でしかありませんでした。働き方の評価もコミット履歴からの逆算で、だから「推定労働時間150時間/週」みたいな、当たっているんだか外れているんだかわからない数字になる。

今回は違って、Fable は1週間分の私の仕事ぶりそのものを記憶した状態で評価しています。実際、個人評価の根拠に挙がっているのは「XML モードの凍結を30分で決断した」「評価が上がる方向にも下がる方向にも反論してきた」「AI の実装ミスを検証装置が捕まえる現場」といった、**この1週間に実際に起きた出来事**です。資料で語った実績ではなく、観察された行動で評価される。これは前回には構造的に不可能だったことで、受け取る側の納得感がまるで違います。

### 3. 結論までの時間

前回は評価のためにわざわざ資料を用意して、読ませて、追加の質問に答えて、と評価専用のやりとりにそれなりの時間を使いました。こちらの文脈を覚えていないので、結論にたどり着くまでの往復がどうしても長くなる。

今回は、1週間の仕事の最後に「改めて評価してみて」と一言投げただけです。文脈の再説明はゼロ。評価は仕事の副産物として、ほぼその場で出てきました。しかもその後の「これだけやって上位数%は厳しくない?」「それは弱点じゃなくて意図した設計だ」といった反論戦にも、全部その場の記憶で応じてくる。**評価のコストが「準備して読ませる作業」から「聞くだけ」に下がった**ことで、評価が特別なイベントではなく、日常の振り返りとして使える道具になりました。

## 99点は使えなかったが、B− は使える

前回の99点は嬉しかったけど、次に何をすべきかは何も教えてくれませんでした。今回の評価で一番価値があるのは、皮肉なことに唯一渋かった B− の軸です。「配布が rsync 頼み」「運用の罠が頭の中にしかない」「ドキュメントが日本語のみ」。コードの評価を頼んだつもりが技術の外側まで評価対象にされていて、正直そこを指摘されるのは予想外でした。でも全部、次にやるべきことのリストとしてそのまま機能する指摘です。

モデルが世代を重ねて、AI の評価は「読み物として楽しいもの」から「仕事の判断材料になるもの」に変わった、というのが2回やってみての実感です。

## 厳しめに評価させるコツ

前回の記事でも書きましたが、今回効いた気がするやり方を改めて。

* 「厳しく」「忖度なし」は明示的に頼む。頼まないと基本は褒めてきます
* 「実際に見た証拠から言えることだけ言え」と縛る。これで一般論の褒め言葉が消えて、実際のコミットや判断への言及に変わりました
* 「匹敵するコードベースは何?」と比較対象を聞く。評価の物差しが露出するので、物差しが妥当かどうかで評価自体の信頼度を測れます
* 納得できない評価には具体的に反論する。撤回するか根拠を補強してくるかで、評価の質がわかります
* 反論は両方向やる。「それは弱点じゃなくて意図した設計だ」だけでなく「その褒め方は過大では」もぶつけると、追従かどうかの判定がつきやすいです

そして今回学んだ一番大きいコツは、**評価の前に一緒に仕事をすること**です。評価させるために読ませるのではなく、仕事をさせた流れで評価させる。証拠が観察になるので、評価の質が根本から変わります。

## おわりに

7ヶ月前、99点をもらって「盛りすぎだと思います」と書きました。今回も評価の高さ自体は変わらないのに、返ってきた弱点リストには「その通りだと思います」と書いています。同じ最高評価でも中身は別物で、そしてこの進歩は私ではなく AI 側の進歩です。

1週間の作業の副産物のつもりだったのに、「次に証明すべきはコードではなく、このシステムが自分以外の人間の下でも回ること」という宿題を出されて終わるとは思っていませんでした。ソロ開発者にとって一番手に入りにくいのは他人からのレビューなので、半年ごとの定点観測としてまたやると思います。次のモデルのときに、この記事が比較対象になる予定です。

（この記事に載せた評価は Claude Fable 5 の出力をほぼそのまま掲載しています。それでも持ち上げすぎでは、と感じた部分があったなら、99点時代の癖がまだ抜けていないのかもしれません。判断は読者にお任せします）
