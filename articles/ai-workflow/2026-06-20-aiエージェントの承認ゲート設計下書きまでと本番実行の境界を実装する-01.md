自動化エージェントを運用していると、「下書きを作るだけのはずが投稿まで走った」という事故が必ず一度は起きる。原因はほぼ同じで、「実行してよい操作」と「人間確認が要る操作」の境界が実装レベルで定まっていないこと。

このまとめは、Claude Agent SDK（claude-sonnet-4系）を使った日次バッチ自動化の中で整備した、承認ゲートの実装パターン。

## 承認ゲートが必要な理由

AIエージェントはプロンプトを書き換えるだけで動作が変わる。テスト時に「下書き保存まで」と指示していても、プロンプトが変わったり会話コンテキストが汚染されると「公開まで」走ってしまう。

コードレベルで「この操作は承認なしに実行できない」という制約を持たせるのが、承認ゲートの役割。

## 設計の基本：操作を3種類に分類する

すべての操作を3つのクラスに分類することから始める。

| クラス | 操作例 | 承認 |
| --- | --- | --- |
| 読み取り専用 | ファイル読み込み、リスト取得、検索 | 不要 |
| ローカル書き込み | 下書き保存、ログ更新、レポート出力 | 不要 |
| 外部送信・公開 | メール送信、SNS投稿、本番デプロイ、DB更新 | 必須 |

外部送信・公開クラスは**承認フラグが明示的に true になるまで実行しない**、これだけ守れば大半の事故は防げる。

## 実装パターン

### 1. AgentOutput に requires\_approval を含める

エージェントの返り値に「人間確認が必要なもの」を明示的に含める。

```
type AgentOutput = {
  // 自動実行済み（読み取り・下書き保存のみ）
  autoExecuted: ExecutedAction[];
  // 人間確認待ち
  requiresApproval: ApprovalItem[];
  // 実行しなかった理由
  skipped: SkippedAction[];
};

type ApprovalItem = {
  id: string;
  action: "send_email" | "post_sns" | "deploy" | "publish";
  summary: string;       // 人間が読む1行サマリー
  draftPath?: string;    // 承認後に使う下書きのパス
  approved?: boolean;    // コントローラが埋める
};
```

エージェントは `requiresApproval` に積むだけで実行しない。コントローラが承認フラグを埋めて、次のステップを実行する。

### 2. コントローラ層で承認フラグを管理する

```
async function runWithApprovalGate(
  agent: Agent,
  task: TaskSpec,
  autoApprove: string[] = [],  // 明示的に自動承認する操作IDのリスト
): Promise<AgentOutput> {
  const output = await agent.run(task);

  // 承認ゲート処理
  const approved: ApprovalItem[] = [];
  const blocked: ApprovalItem[] = [];

  for (const item of output.requiresApproval) {
    if (autoApprove.includes(item.id) || autoApprove.includes(item.action)) {
      approved.push({ ...item, approved: true });
    } else {
      blocked.push({ ...item, approved: false });
    }
  }

  // 承認済みのもののみ実行
  for (const item of approved) {
    await executeApprovedAction(item);
  }

  return { ...output, requiresApproval: blocked };
}
```

`autoApprove` は空配列がデフォルト。スケジュールタスク（夜間自動実行）では何も渡さず、人間がレビューするタイミングでのみ `autoApprove` に操作IDを渡す。

### 3. 下書きは承認前に必ずローカル保存する

承認ゲートを安全に運用するには、「承認前に下書きがローカルに存在する」状態を保証する必要がある。

```
async function saveDraftAndQueue(draft: Draft): Promise<ApprovalItem> {
  // まず下書きをローカルに保存
  const draftPath = await writeDraft(draft);

  // 承認キューに積む（実行はしない）
  return {
    id: generateId(),
    action: draft.targetPlatform,
    summary: `[${draft.type}] ${draft.title} → ${draft.targetPlatform}`,
    draftPath,
    approved: false,
  };
}
```

こうすると、承認が遅れても下書きは残る。承認後に `draftPath` から読み込んで実行するため、「承認した瞬間に内容が変わっている」リスクもない。

```
## 承認ゲート実装チェック

### 操作分類
- [ ] 全ツールを読み取り/書き込み/外部送信で分類した
- [ ] 外部送信系ツールを別モジュールに分離した
- [ ] エージェントに直接渡すツールから外部送信を除いた

### 返り値設計
- [ ] requiresApproval フィールドを AgentOutput に含めた
- [ ] 下書きはローカル保存してパスを返している
- [ ] autoExecuted と requiresApproval を明確に分けた

### コントローラ
- [ ] 承認フラグは明示的にコントローラから渡す設計
- [ ] 自動承認リストは空がデフォルト（夜間バッチなど）
- [ ] 承認済み操作の実行ログを残している

### テスト
- [ ] dry_run モードで全フローが動作確認済み
- [ ] 承認なしで外部送信が走らないことをテストした
- [ ] 途中失敗から再実行しても冪等に動くことを確認した
```

## 失敗パターン

**パターン1：エージェント内部に承認ロジックを書いた**  
→ エージェントのプロンプトが変わると承認ロジックも変わってしまう。承認はコントローラ層（外部）に書く。

**パターン2：`dry_run` フラグだけで制御しようとした**  
→ フラグの渡し忘れで本番実行が走る。フラグなしでも外部送信が起きない構造（ツール自体を持たせない）が確実。

**パターン3：承認後に下書きを再生成した**  
→ 再生成すると内容が変わるリスクがある。承認した内容と実行する内容を一致させるため、承認前に保存した `draftPath` を使う。

## まとめ

承認ゲートの本質は「エージェントに権限を与えないこと」。プロンプトで「絶対に投稿しない」と書くより、投稿ツール自体をエージェントに渡さない方が確実。外部送信・公開はコントローラが明示的に承認した場合のみ実行する、この1点を構造として持たせることで、自動化の安全性は大幅に上がる。
