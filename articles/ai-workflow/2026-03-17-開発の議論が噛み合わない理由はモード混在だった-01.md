# 計画はレビューではない。レビューはリリースではない。

> **計画はレビューではない。レビューはリリースではない。**

この一行を意識するだけで、開発の会話はかなり整理されます。以下はその理由と、三つのモードの説明です。

---

## 開発の議論が噛み合わない理由

例えば次の三つは、**別の種類の話**です。

* 「この機能、本当に必要？」 … 何を作るか（計画）
* 「この実装、バグありそう」 … ちゃんとできているか（レビュー）
* 「とりあえずマージしてデプロイしよう」 … 本番反映（リリース）

同じ頭で同時にやろうとすると、議論が噛み合いません。

---

## 三つのモード（ここだけの定義）

| モード | やること | 思考 |
| --- | --- | --- |
| Plan | 何を作るか決める | 価値・スコープ |
| Review | 作ったものをチェック | バグ・設計 |
| Release | 本番反映を実行する | 手順実行 |

流れは次のとおりです。

* **計画** … 作るものを決めます。ここで「このコード、バグありそう」と言い出すとレビューに逸れます。逆にレビュー中に「そもそもこの機能必要？」と言い出すとレビューが終わりません。
* **レビュー** … 出してよいか判断する**まで**です。ここで「じゃあマージしてデプロイしよう」と進めると、判断と実行が混ざります。
* **リリース** … main 同期・テスト・PR・デプロイなどを、手順通りに実行するフェーズになります。

---

## 例：ログイン画面

| モード | やること |
| --- | --- |
| Plan | ログイン画面を作る。パスワードリセットは次スプリント。 |
| Review | 二重送信対策、エラーメッセージをチェック。 |
| Release | main にマージ → CI → デプロイ。 |

これを混ぜないだけで、開発は整理されます。

---

## この思想をツールにしたものが gstack

[gstack](https://github.com/garrytan/gstack)（[Garry Tan](https://www.ycombinator.com/people/garry-tan)（Y Combinator の CEO）による Claude Code 向けワークフロースキル集）は、**Plan → Review → Release** をコマンド単位で分離します。

私はCursorでも使っています。  
[Cursor 向けセットアップ](https://53able.github.io/docs-site/docs/gstack.html#cursor) を参照してください。

| モード | 主なコマンド | やること |
| --- | --- | --- |
| Plan | `/plan-ceo-review` | スコープ・「10スター製品」を問い直す（ファウンダー視点）。 |
| Plan | `/plan-eng-review` | アーキテクチャ・データフロー・テストを固める。 |
| Review | `/review` | 差分をレビューし、本番で起きそうなバグや設計の穴を指摘。 |
| Release | `/ship` | main 同期・テスト・PR・デプロイを手順通りに実行。 |

このほか QA 用の `/qa`・`/browse`、振り返り用の `/retro` などもあります。[gstack の紹介ページ](https://53able.github.io/docs-site/docs/gstack.html) を参照してください。

**機能追加からリリースまでの例**

1. **Plan** … `/plan-ceo-review` でスコープを問い直し、`/plan-eng-review` でプランを固めます。
2. **実装** … 通常の Agent でコードを書きます。
3. **Review** … 差分ができたら `/review` で指摘を潰します。
4. **Release** … 問題なければ `/ship` で main 同期・テスト・PR・デプロイまで任せます。

「計画のときは計画のコマンド、レビューのときはレビューのコマンド」と分けるだけで、会話と作業が同じ流れに乗ります。

---

## まとめ

* **Plan** … 何を作るか
* **Review** … ちゃんとできているか
* **Release** … 本番反映

モードが混ざると混乱します。思い出したいときは、冒頭の一行に戻ってください。
