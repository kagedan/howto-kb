Claudeを使い始めて、最初に思うこと。

「で、これ何に使えるの？」

チャットで質問できるのはわかった。Claude Codeも試してみた。「スキル」っていう機能があるらしいから1個入れてみた。

で、そこで止まってる人がすごく多い。

GitHubに「awesome-claude-skills」っていうリポジトリがあって、コミュニティが作ったスキルが50個以上並んでる。「ドキュメント生成」「Web調査」「コードレビュー」「セキュリティチェック」。どれも便利そう。

でもリストを見て「へー」って思って閉じてしまう。

なぜかというと、スキルを1個入れるだけじゃ、仕事のやり方は変わらないから。

スキルが本当に効くのは、複数のスキルとエージェントを組み合わせて「業務のフロー全体」を設計した時。1個の便利ツールじゃなくて、仕事の仕組みそのものを作り変える。

ラーメン屋で例えると、スキル1個は「いい包丁を買った」。ワークフロー設計は「仕込みから提供までの厨房オペレーション全体を組んだ」。いい包丁だけ持っててもラーメン屋は回らない。

## Claudeの世界は4つのゾーンに分かれてる

まず全体像を整理する。Claudeの周辺ツールは4つのゾーンに分かれてる。

ゾーン1: 公式ツール。Claude.ai（チャット）、Claude Desktop（デスクトップアプリ）、Cowork（コワーク。仕事を任せるモード）、Claude Code（プログラミング向けのターミナルツール）。Anthropic（アンソロピック。Claudeを作ってる会社）が公式に出してるもの。まずはここから。

ゾーン2: スキル。Claudeに「こういう作業はこう動け」って教える設定ファイル。レシピカードみたいなもので、作業の手順とルールを書いておくと、毎回同じ品質で動いてくれる。GitHubにコミュニティ製のスキルが大量にある。

ゾーン3: ツール・ユーティリティ。Claudeの使い勝手を上げる周辺ツール。トークン（やりとりの量）を監視するツールとか、GUIとか。

ゾーン4: エージェント・自動化。Claudeに「自分で判断して動け」って任せる仕組み。サブエージェントを作って専門の役割を持たせたり、複数のClaude Codeを並列で動かしたりする。

ほとんどの記事はゾーン1の使い方か、ゾーン2のスキルの作り方を解説してる。

でも、業務を回すワークフローは「ゾーン2のスキル × ゾーン4のエージェント × CLAUDE.md（全体のルールブック）」の掛け算で成り立ってる。

この「掛け算の設計図」を公開してる記事が、調べた限りほとんどない。

この記事ではその設計図を出す。

ここから先では、業務別のワークフロー設計図（4パターン）、スキル×エージェント×CLAUDE.mdの連携設計、スキルを育てる運用方法、よくある設計ミスを解説していきます。

「スキルを1個入れた」で止まってる人が、「業務をスキルで回してる」に変わるための記事です。本編で会えたらうれしいです。

※執筆時点（2026年3月）の情報です。Claude Code、スキル、エージェントの仕様は進化が速いので、最新の仕様とは異なる場合があります。

## 第3章: 業務別ワークフロー設計図

スキルを「どう組み合わせるか」は、業務の種類によって変わる。

4つのパターンに分けて、それぞれの設計図を出す。

### パターン1: コンテンツ制作ワークフロー

記事、SNS投稿、動画台本を定期的に作る人向け。

フロー: ネタ収集 → 記事執筆 → SNS投稿化 → 品質チェック

各ステップにスキルを割り当てる。

ステップ1「ネタ収集」: 収集スキルを作る。「どこから」「何を」「どういう基準で」集めるかをスキルに書いておく。毎回「今日のネタ探して」って言うだけで、スキルに書いた基準で集めてくれる。

ステップ2「記事執筆」: 記事スキルを作る。文体のルール、構成のテンプレート、保存先。ここが一番ルールが多くなる。

ステップ3「SNS投稿化」: 記事を要約してSNS用に変換するスキル。記事スキルとは別に作る。文体が違うから。記事はです・ます調でも、Xの投稿はカジュアル、みたいな使い分けをスキルで分離する。

ステップ4「品質チェック」: ここでエージェントを使う。「Verifier」みたいな検証専門のサブエージェントを作って、ファクトチェックや文体チェックを委任する。本体のClaude Codeが書いたものを、別の視点でチェックさせる。

大事なのは、各スキルを独立させること。「記事を書いて、ついでにSNS投稿も作って、ついでに品質チェックもして」って1つのスキルに詰め込むと破綻する。1スキル1役割。

### パターン2: SNSマーケティングワークフロー

X、TikTok、Instagram等でコンテンツを定期的に発信する人向け。

フロー: トレンド調査 → コンテンツ制作 → 投稿最適化 → パフォーマンス分析

ステップ1「トレンド調査」: 自分のジャンルで今何がバズってるかを調べるスキル。検索先、判断基準、レポートの形式をスキルに書いておく。

ステップ2「コンテンツ制作」: プラットフォーム別にスキルを分ける。Xの投稿とTikTokのスライドでは文体も構成も全然違う。1つのスキルで全プラットフォームをカバーしようとすると、どれも中途半端になる。

ステップ3「投稿最適化」: ハッシュタグの選定、投稿時間の提案、フックの強化。ここは独立したスキルにしてもいいし、制作スキルに含めてもいい。量が多いなら分離。

ステップ4「パフォーマンス分析」: 数字を見て改善策を出すスキル。ここにAnalyst（分析専門）のサブエージェントを使うと、本体の記憶を圧迫せずに深い分析ができる。

### パターン3: リサーチ・分析ワークフロー

市場調査、競合分析、情報収集を定期的にやる人向け。

フロー: 調査設計 → データ収集 → 分析 → レポート化

ここでサブエージェントが最も活きる。

「Market」エージェント: 市場調査の専門家。競合の目次、価格帯、空白地帯を調べる。

「Analyst」エージェント: データを受け取ってパターン抽出、仮説生成を行う。

本体のClaude Codeは「司令塔」として、どのエージェントに何を任せるかを判断するだけ。調査の実作業はエージェントがやる。

各エージェントは .claude/agents/ フォルダにテキストファイルで定義する。「あなたは市場調査の専門家です。以下の手順で調査してください」って日本語で書くだけ。

### パターン4: 定型業務の自動化ワークフロー

週報、月報、議事録、メール返信など、毎回似たような作業を繰り返す人向け。

フロー: テンプレート化 → 入力を受け取る → 生成 → 確認

ここはスキル1つで完結することが多い。エージェントまで使わなくていい。

ポイントは「テンプレートの精度」。週報スキルなら「何を書くか」「どの順序で」「どのフォーマットで」を全部スキルに書いておく。毎週「週報書いて」の一言で同じ品質のものが出てくる。

定型業務こそスキル化の効果が一番わかりやすい。「毎回15分かけてた作業が、スキルに任せたら2分で終わる」みたいな即効性がある。

## 第4章: スキル × エージェント × CLAUDE.md の連携設計

ワークフローを構成する3つの柱がある。

CLAUDE.md = 全体のルールブック。プロジェクト共通のルール、文体、保存先、やってはいけないこと。すべてのスキルとエージェントがこれを読む。

スキル（SKILL.md）= 個別作業の手順書。1つのスキルが1つの作業を担当する。

エージェント = 専門の部下。特定の役割に特化した存在で、本体から委任を受けて動く。自分専用のコンテキスト（記憶の箱）を持ってる。

この3つの関係は、会社組織に似てる。

CLAUDE.md = 社訓・就業規則。全社員が従う。

スキル = 業務マニュアル。部署ごとに違う。

エージェント = 各部署の専門社員。得意分野がある。

### 設計の順序

最初にCLAUDE.mdを書く。ここに書くのは「全作業に共通するルール」だけ。文体の好み、保存先のルール、やってはいけないことのリスト。個別の作業手順はここに書かない。

次にスキルを作る。業務ごとに1つ。最初は2-3個から始める。完璧にしようとせず、最低限のルールだけ書いて使い始める。

エージェントは必要になったら作る。最初から全部揃えなくていい。「この作業、毎回本体に頼むと記憶がいっぱいになるな」と感じた時がエージェントを作るタイミング。

### ファイル構成

```
プロジェクト/
├── CLAUDE.md          ← 全体のルールブック
├── .claude/
│   ├── skills/        ← スキル置き場
│   │   ├── 記事執筆/
│   │   │   └── SKILL.md
│   │   ├── SNS投稿/
│   │   │   └── SKILL.md
│   │   └── 収集/
│   │       └── SKILL.md
│   └── agents/        ← エージェント置き場
│       ├── market.md
│       └── verifier.md
```

スキルは .claude/skills/ の中にフォルダを作って、その中にSKILL.mdを置く。エージェントは .claude/agents/ の中にMarkdownファイルを置く。

中身は全部日本語のテキスト。プログラミングの知識は不要。「こういう手順でやって」「こういうルールを守って」って書くだけ。

## 第5章: スキルを「育てる」運用

スキルは作って終わりじゃない。使いながら育てるもの。

「育てる」っていうのは、フィードバックを受けてスキルを改善し続けるサイクルのこと。

### サイクルの回し方

スキルを使って成果物を作る。

成果物を確認する。「ここが違う」「ここがズレてる」があれば修正を指示する。

修正が入った時がチャンス。「なぜこの間違いが起きたのか」を考えて、スキルにルールを追加する。

スキルのSKILL.mdに「経験値」セクションを作っておくのがおすすめ。修正が入るたびに「何が起きて、何を学んで、どうルールを変えたか」を記録していく。

```
## 経験値

- 2026-03-15: 文体が堅すぎた。「です・ます」で統一していたが、
  読者から「距離が遠い」とフィードバック。話し言葉に変更
- 2026-03-20: 導入文が毎回同じパターンになっていた。
  テンプレートを削除し、方向性だけ残す形に変更
- 2026-03-31: 例え話の前振り（「わかりやすく言うと」）が
  読者を子ども扱いしてる印象。前振りなしで自然に溶け込ませる
```

この経験値が蓄積されるほど、スキルの精度が上がっていく。1ヶ月後には最初とは別物のスキルになってる。

大事なのは「学びを発生元に直接書く」こと。共有ファイルに分散させるんじゃなくて、そのスキルのSKILL.mdに直接書く。次にそのスキルが呼ばれた時に、Claude Codeが経験値を読んで同じ間違いを避けてくれる。

### メンテナンスのタイミング

週1回、使ったスキルのSKILL.mdを見直す。

チェックポイント: 矛盾するルールがないか。古くなったルールがないか。「このルールいらなかったな」ってものがないか。

スキルは「育てる」だけじゃなく「剪定する」も必要。ルールが増えすぎると、Claude Codeがどのルールを優先すべきか迷って出力がブレる。「足す」だけじゃなく「引く」も大事。

## 第6章: よくある設計ミスと直し方

### ミス1: CLAUDE.mdに全部詰め込む

CLAUDE.mdは「全作業に共通するルール」だけを書く場所。個別の作業手順まで書くと、すべての会話で不要な情報が読み込まれて、コンテキスト（記憶の箱）を圧迫する。

直し方: 「この作業でしか使わないルール」はスキルに移す。CLAUDE.mdには「文体」「保存先」「やってはいけないこと」レベルの共通項だけ残す。

### ミス2: 1つのスキルに複数の作業を詰め込む

「記事を書いて、SNS投稿も作って、画像プロンプトも出して」を1つのスキルにまとめると、スキルが巨大化して精度が落ちる。

直し方: 1スキル1役割。「記事スキル」「SNS投稿スキル」「画像プロンプトスキル」に分ける。分けるほど各スキルの精度が上がる。

### ミス3: スキルのdescriptionが曖昧

スキルが呼ばれるかどうかは、SKILL.mdの冒頭にあるdescription（説明文）で9割決まる。Claude Codeは「どのスキルを使うか」をこの説明文で判断してる。

「便利なスキル」みたいな説明だと、いつ使えばいいかClaude Codeにはわからない。

直し方: 「○○の時に使う」「○○と言われたらトリガー」みたいに、発動条件を具体的に書く。

```
悪い例:
description: 記事を書くスキル

良い例:
description: >
  英語圏でバズったAI活用事例を初心者向け記事に変換する。
  「AI活用記事書いて」「記事作って」と言った時に使用する。
```

### ミス4: エージェントの役割が広すぎる

「なんでもできるエージェント」は、何もうまくできないエージェントになる。

直し方: 1エージェント1専門。「市場調査エージェント」「品質検証エージェント」「データ分析エージェント」。役割を絞るほど精度が上がる。

### ミス5: 作ったスキルを放置する

第5章で書いた通り、スキルは育てるもの。作って放置すると、業務の変化に追いつかなくなって、だんだん使われなくなる。

直し方: 修正が入るたびに経験値を追記する。週1で見直す。「最近このスキル使ってないな」と思ったら、不要なら削除する。ファイルが残ってるだけでClaude Codeの判断を邪魔することがある。

## 読み終わったあなたがやること

今日やるのは2つだけ。

1つ目。自分の業務で「毎回同じことを繰り返してる作業」を1つ見つける。週報でも、メール返信でも、SNS投稿でも。

2つ目。それをスキルにする。.claude/skills/ にフォルダを作って、SKILL.mdに「手順」「ルール」「保存先」を書く。完璧じゃなくていい。3行でもいい。使いながら育てる。

1個のスキルが回り始めると「次はこれもスキルにしよう」が自然に出てくる。そこからワークフローが始まる。
