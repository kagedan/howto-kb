## 概要

Mastraが新たにリリースした「**メトリクスとログ（metrics and logs）**」機能の解説ワークショップ。前半はStudioのデモとコードの解説、後半はEricによる可観測性システムの設計思想と実装の舞台裏が語られる。なぜOTEL（OpenTelemetry）ではなくカスタムソリューションを選んだのか、なぜOLAPデータベース（DuckDB/ClickHouse）でなければならないのか、といった「Why」をアーキテクチャレベルで掘り下げた内容になっている。

## 要点

1. **新機能「メトリクスタブ」**: Studioに追加された新しいタブで、エージェントごとのレイテンシ・トークン使用量・コスト・スコアを集計・時系列で可視化できる。
2. **「ログ」機能**: 単なるログリストではなく、ログエントリからトレースやスパンにシームレスにジャンプでき、ユーザーのジャーニーを自然な流れで追える。
3. **コンポジットストア必須**: libSQLはメトリクス・ログをサポートしない。可観測性データにはDuckDB（またはClickHouse）を割り当てるコンポジットストア設定が必須。
4. **OTELを使わなかった理由**: AIエージェントは「長時間・止まり止まり」という特性を持ち、実行が終わるまでデータを送れないOTELの制約と根本的に相性が悪い。強型化された属性も必要だったため、カスタムのイベント駆動アーキテクチャを構築した。
5. **Prometheusでは高カーディナリティに対応できない**: ユーザーIDのような高カーディナリティのラベルはPrometheusでは保存できない。ユーザーごとの課金やコスト追跡にはこれが必要不可欠。
6. **可観測性バス（observability bus）**: スパン・メトリクス・ログ・フィードバック・スコアがすべてこのバスを通り、DuckDB/ClickHouse（デフォルト）・Mastra Cloud・サードパーティ（LangFuse/DataDog/BrainTrustなど）に同時ファンアウトされる。
7. **OLAPデータベースを選ぶ理由**: 1回の実行で200〜500行のデータが生成される。カラム指向ストレージ・日単位パーティション・スパースデータへの最適化により、インデックスなしでも高速なフィルタリングが可能。
8. **スコアラー・評価のトレーシングが新たに追加**: 従来はエージェント・ワークフローのみだったトレーシングが、スコアラーや評価の実行にも対応。
9. **今後の展望**: カスタムダッシュボード・保持ポリシー設定・ディープダイブ（グラフの特定データポイントからトレースへのジャンプ）・AIイシューサーフェシング（エージェントが自律的にバグを発見）・ノートブック機能が計画されている。
10. **グラフDBの可能性**: トレース・メトリクス・ログを「人間向けの分類」から「イベントのグラフ」として再設計することで、エージェントがより自然にこのデータを消費できるようになるという仮説をEricが語った。

## 詳細

### メトリクスとログの有効化方法

Mastraアプリケーションで新しい可観測性機能を有効にするには、ふたつの大きな変更が必要になる。

**1. コンポジットストアの設定**

libSQLやPostgresのような従来のリレーショナルDBは、可観測性データの保存に適していない。メトリクス・ログ機能を使うには、可観測性ドメイン専用にOLAPデータベースを割り当てたコンポジットストアを設定する必要がある。

```
// index.ts（概略）
const mastra = new Mastra({
  storage: {
    default: new LibSqlStore({ url: '...' }),     // メッセージなどの通常データ
    observability: new DuckDBStore({ path: '...' }), // トレース・ログ・メトリクス用
  },
  // ...
});
```

DuckDBはベクターストア（RAG利用時）と可観測性専用。libSQLは引き続きメッセージ等の通常データに使う。

**2. 可観測性オブジェクトの初期化とデフォルトエクスポーター**

可観測性オブジェクトを初期化し、スパンをどこに保存するかを指示するデフォルトエクスポーターを設定する。Mastra Cloudに送るクラウドエクスポーターも設定できる（2026年4月時点では、トレーシングのみ対応。その他シグナルは近日中に追加予定）。

**3. ログの書き方**

ワークフローのステップなどでロガーを使う場合、pinoロガーもMastraの定義ファイルで有効化する必要がある。これにより、ログは通常のpinoパイプライン（コンソール等）とDuckDBの両方に同時書き込みされる。

```
// ワークフローステップ内
const logger = useLogger();
logger.info('fetch weather step is running');
```

ログエントリをStudioでクリックすると、そのログが属するトレース・スパンにジャンプできる。これにより「どのステップで何が起きたのか」を具体的に追跡できる。

### なぜOTELを使わなかったのか

Ericがこの設計判断を下す経緯は、AIエージェントの本質的な特性から来ている。

OTELはHTTPリクエストのような「短時間・同期的」な処理のために設計されている。最大でも30秒程度のトレースを想定しているが、AIエージェントは：

* **分・時間・日単位**で実行されることがある
* **ヒューマン・イン・ザ・ループ**で一時停止・再開するため「止まり止まり」になる
* 実行が終わる前からリアルタイムでデータを確認したい（デバッグ目的）
* エージェント・ワークフロー・ツールコール・モデルで「保存したいアトリビュート」が全く異なる

特に致命的なのが「実行終了後にしかデータを送れない」という制約だ。ライブデバッグが必要なAIエージェントの開発現場では、これは大きな障壁になる。

業界他社の動向を見ると：

* **LangFuse**: 当初はカスタムソリューション → v4でOTELに移行
* **BrainTrust**: いまもカスタムソリューション
* **LangSmith**: いまもカスタムソリューション
* **Arize**: OTELベース（ただしOpen Inferenceというレイヤーを追加）

Mastraはカスタムのイベント駆動アーキテクチャ（開始イベント・終了イベントのシリーズ）を採用した。スパンの開始時に入力・属性を記録し、終了時に出力・エラー・メタデータを保存する形で、システム全体のイベントツリーを形成する。

### なぜOLAPデータベースなのか

リレーショナルDBがAI可観測性データに向かない理由は、データ量とデータ構造の両面にある。

**データ量の問題**：1回の実行で200〜500行ものデータが生成される。

* 実行ごとに30〜100件のスパン
* モデル呼び出しがあれば使用量とコストのメトリクス行が5〜10行追加
* スパンごとにデュレーション行
* 各種ログイベント

**データ構造の問題**：このデータは非常に幅が広く（カラム数が多い）、かつスパースだ（多くのカラムがNULL）。リレーショナルDBではNULLに対してもストレージを予約する必要があるが、OLAPのカラム指向ストレージはスパースデータを効率よく扱える。

**カーディナリティの問題**：Prometheusのような集計中心のシステムでは、ユーザーIDのような高カーディナリティの値を保存できない。エンタープライズ向けにユーザーごとの使用量追跡・課金をするには、これが根本的に必要になる。

**OLAP（DuckDB/ClickHouse）の利点**：

* カラム指向ストレージによる大幅なストレージ削減
* 型を知った状態でのカラム直接イテレーション（ID検索が高速）
* インデックスなしでも大量カラムにわたるフィルタリングが可能
* **日単位パーティション**によりデータ保持の管理が非常に簡単（古いパーティションをドロップするだけ）

### 可観測性バスのアーキテクチャ

可観測性バスは、Mastraの可観測性システムの中核となる設計パターンだ。

```
[エージェント/ワークフロー/スコアラー]
    ↓（スパン・メトリクス・ログ・フィードバック・スコアイベント）
[可観測性バス（バッチ＋インターバルフラッシュ）]
    ↓ ファンアウト
    ├── デフォルトエクスポーター → DuckDB / ClickHouse
    ├── クラウドエクスポーター  → Mastra Cloud
    ├── サードパーティ          → LangFuse / DataDog / BrainTrust / LangSmith / Arize など12〜13種
    └── OTELブリッジ           → 既存のOTELパイプライン
```

サーバーレス環境ではステップ終了前に`flush()`を明示的に呼び出してデータを確実に送信できる。

評価（evaluation）を実行すると、スコアはバスを通じてすべてのエクスポーターにファンアウトされるため、LangFuseのスパン・BrainTrustのスパン・Mastraのデフォルトストレージに同時にスコアが記録される。

### 保存されるフィールドとエンティティ階層

各スパン・ログ・メトリクスには以下が保存される：

| フィールド | 説明 |
| --- | --- |
| trace\_id | トレースID |
| span\_id | スパンID |
| tags | トレースに割り当てられたタグ |
| user\_id | ユーザーID（高カーディナリティ対応の核心） |
| organization\_id | 組織ID |
| resource\_id | リソースID |
| entity\_type | 現在のエンティティ型（agent / tool\_call / model / workflow\_step） |
| entity\_name | エンティティ名 |
| entity\_id | エンティティID |
| parent\_entity\_id | 直接の親エンティティID |
| root\_entity\_id | ルートエンティティID（最上位） |
| environment | 環境（production / test / CI） |
| service | Mastraサービス名 |
| experiment\_id | 実験ID（データセット×エージェント実行） |

エンティティ階層により、「このイベントがどこで起きているか」というパンくずリストが形成される。モデル→エージェント→プロセッサという階層をたどってフィルタリングが可能。

### スコアとフィードバックの違い

両者は似ているが異なる概念だ。

**スコア（score）**: 機械駆動の評価値。エージェントの実行スナップショット（トレース）に対して、LLMアズジャッジなどが自動的に計算する「この実行はどれほど良かったか」という指標。

**フィードバック（feedback）**: 人間からのアノテーション・コメント。「この応答は良かった（👍）」「このトレースは怪しい（あとでレビュー）」という開発者・ユーザー双方からの評価を含む。数値スコアも付けられる。

どちらも同じ集計・フィルタリングシステムを通じて管理されるため、スコアとフィードバックをまたいだクエリも可能。

### 会場Q&Aより（抜粋）

**Q: ユーザーごとのコストを追跡できるか？**  
→ できる。user\_idでメトリクスを集計して、入力トークンコストと出力トークンコストをそれぞれ取得して合計する。APIまたはクライアントパッケージから実現可能。

**Q: DataDogなど外部ツールにエクスポートできるか？**  
→ できる。現在12〜13種のサードパーティエクスポーターをサポート。OTELブリッジも利用可能。

**Q: LangFuseとMastra Cloudでデータの完全性に差はあるか？**  
→ すべてのエクスポーターに同等のデータを送信している。LangFuse側での処理内容（クリーンアップ・トランケーション等）は不明。

**Q: ローカルのデータを本番コンテキストのデータセットとして使えるか？**  
→ できる。ローカルでデータセットを作成してMastra Cloudにエクスポートできる。さらにCIパイプラインに組み込んでスコアによるデプロイゲートとしての利用も可能。

**Q: 従量課金をこの上に構築できるか？**  
→ できる。エージェントがMastra APIを自ら呼び出してユーザーの使用量を確認し、制限を判断するという面白い実装が可能。
