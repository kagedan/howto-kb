## 何が起きたか — お手本の言葉が、出力に混入した

AIに歌詞を書かせていたとき、妙なことが起きました。

テーマを変えても、出てくる言葉が似る。失恋でも、卒業でも、夏の思い出でも、同じような情景や言い回しに戻ってくる。

調べてみると、原因はモデルの性能ではなく、**自分が渡していたプロンプト側**にありました。品質を上げようと、プロンプトの中に「良い例」を書いていたんです。

```
例: 「水たまりに映る 二人の影」
避けるべき表現: 「指先が震える」
```

生成結果を大量に集めて見比べて、ぞっとしました。お手本に書いた **「水たまり」が、ほぼ全部の出力に出てくる**。しかも笑えることに、**「避けてください」と書いたはずの「指先」まで頻発する**。AIは「これは良い例」「これは避ける例」という意図はある程度読みます。それでも、そこで提示された具体語そのものに強く引っ張られることがある。良い例も、避ける例も、です。

さらに、AIが放っておいても勝手に寄っていく **“手癖ワード”** もあります。自分の生成結果では、都市・夜・雨・光・身体感覚に寄った語彙——アスファルト、鼓動、月明かり、残響、ガラス越し——が繰り返し出やすい傾向がありました。学習データの偏りそのもので、何も指示しなくても湧いてくる。

「じゃあ全部禁止リストに入れよう」とやって、もう一段ハマりました。**禁止語を並べたリストが、それ自体AIの“語彙カタログ”のように働いてしまう**——少なくとも、自分の生成結果ではそう見えました。実際、いまの自分のコードのコメントにはこう書いてあります。

> 具体語の列挙が多すぎるとAIの語彙カタログとして逆効果になる。

禁止すら、具体語で書けば汚染源になりうる。

## これは「水たまり」の話ではない

念のため言うと、この記事で言いたいのは「水たまり」という単語の話ではありません。

重要なのは、**AIが具体例を「参考」ではなく「正解への近道」として扱うことがある**、という点です。そして、これは歌詞に限らず、プロンプト・レビュー・Skill・エージェント設計のすべてに共通します。

## なぜ起きるのか — LLMは渡された文脈に強く反応する

LLMは、与えられた文脈を手がかりに出力を組み立てます。だから、プロンプト内に置いた具体語・例・評価観点は、単なる補足ではなく、**出力を方向づける強い信号**になる。具体例を強く渡せば、出力はその周りに寄ります。

いわゆる「AIの忖度」（こちらの期待に答えを寄せる）も、「無難で似た答えに寄る」のも、根は近い——どれも、渡された文脈に強く反応している現れです。すべてをこれだけで説明できるわけではありませんが、**自分が踏んだ失敗の多くは、この形**で出ました。

## 同じ構造が、プロンプト・レビュー・エージェントで出る

この反応の強さは、用途を問わず顔を出します。それぞれ、実際にやりがちな“悪い渡し方”で示します。

**プロンプト**

```
切ない曲にしたい。例：「夜の街、ひとりの帰り道」みたいな雰囲気で。
```

→ テーマを変えても、夜・街・ひとり…と、見せた例の周辺に戻ってくる。

**レビュー**

→ その観点は見てくれるが、それ以外の壊れ方を見落とす。「ここ問題ないよね？」と誘導すれば、問題があっても寄せてくる（忖度）。

**エージェント**

```
必ず A → B → C の順番で確認してください。
```

→ 順序に従うこと自体が目的化し、状況に応じた探索をしなくなる。

## 具体例は、毒にも薬にもなる

ここで誤解されがちですが、**具体例そのものが悪いわけではありません**。few-shot（例を見せる手法）は今も強力です。Anthropic も、few-shot を有効な手法として扱いつつ、**エッジケースを大量に詰め込むのではなく、多様で代表的な例を厳選するべき**だと説明しています。出力の形式・口調・構造を揃えたいときは、むしろ例を見せたほうが安定します。

問題になるのは、こういうときです。

* **発散させたい場面**で、狭い具体例を強く渡してしまう
* エッジケースや禁止語を**大量に並べて**、モデルの注意をそこへ集めてしまう

線引きはシンプルです。

| やりたいこと | 渡し方 |
| --- | --- |
| **発散させたい**（創造・アイデア・文脈依存の判断） | 原則と理由を渡す（自由度は高く） |
| **収束させたい**（フォーマット・分類・定型処理） | 厳選した具体例で型を固める（自由度は低く） |

前回書いた「プロンプトを“変わらない型”としてキャッシュする」話は、完全に収束側。一方、アイデアを出させたい場面で具体例を盛ると、思考はそこに固まります。

## Anthropic の文脈でいう「高度」と「自由度」

この考え方は、Anthropic が説明している設計思想とも近いです（同じことを言っているというより、**近い**という距離感で）。

ひとつは、エージェント設計の「**right altitude（ちょうどいい高度）**」（[Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）。複雑なロジックをガチガチに書き込むと脆くなり、逆に曖昧すぎる指示では信号が弱い。**具体的すぎず・曖昧すぎず、強いヒューリスティックを与える高度**を狙え、という整理です。

もうひとつは、Skill を書くときの「**自由度（degrees of freedom）の調整**」（[Skill authoring best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)）。タスクの壊れやすさや変動性に応じて、Claude に与える自由度を変える。文脈依存の判断が必要なタスクには高い自由度を、壊れやすく一貫性が必要な処理には具体的な手順（低い自由度）を与える、という考え方です。

これはプロンプト単発の話だけではありません。Skill のように、AIが必要に応じて読み込む知識にも、同じことが起きます。Skill は単なる長いプロンプトではなく、Claude が必要なときに読み込む**再利用可能な専門知識のまとまり**です。だからこそ、そこに何を書くか・どこまで具体例を入れるか・どの自由度で判断させるかが効いてきます。

## では、どう書き換えたか

「なぜ起きるか」だけでは使えないので、実際の書き換えを3つ置きます。共通する勘所は、**狭めすぎず・誘導しすぎず、ちょうどいい高度で渡す**ことです。

**歌詞の生成（発散させたい）**

```
Before（最初に書いていたもの）:
例: 「水たまりに映る 二人の影」
避けるべき表現: 「指先が震える」

After:
切ない情景は、天候・場所・身体感覚・時間帯のいずれかで表現する。
ただし同じモチーフに偏らないよう、テーマごとに違う具体物を選ぶ。
```

さらに、システムプロンプトには“考え方”そのものを置きました（実際の一部）。

```
禁止語を「避ける」のではなく、
このテーマの中にしか存在しない言葉を探せ。
```

**コードレビュー（誘導を避け、観点を明示する）**

```
Before:
この修正で問題ないか確認してください。

After:
この修正について、仕様逸脱・既存機能への副作用・セキュリティ・テスト不足の
観点から、壊れる可能性を優先して確認してください。
```

ポイントは、観点を1つに絞ることではなく、**壊れ方のカテゴリを複数渡す**ことです。

**エージェントに任せる（自由度は高く、最終チェックは漏らさない）**

```
Before:
必ず A → B → C の順番で確認してください。

After:
まず影響範囲を特定し、壊れやすい箇所から確認してください。
最終的に A / B / C の観点は漏らさないこと。順序は状況に応じて変えて構いません。
```

## まとめ

* LLMは、渡された文脈に強く反応する。だから渡し方を誤ると、出力の探索範囲が狭まったり偏ったりする
* 具体例は、毒にも薬にもなる
* **発散させたいなら「原則＋理由」（高自由度）／収束させたいなら「厳選した具体例」（低自由度）**
* 最初に決めるのは、ただ一つ。**このAIに、思考を狭めてほしいのか・広げてほしいのか**

AIに何かをさせて「なんか毎回似てる」「思ったのと違う方向に寄る」と感じたら——モデルや運のせいにする前に——一度、自分が渡した具体例と、誘導するような言い回しを疑ってみてください。

## 次回

この「渡し方で出力が変わる」を定量的に測ろうとして、作詞アプローチの切り替え機能を2ヶ月かけて改良しました。

ところが、ある日とんでもないことに気づきます。**その機能、一度もバックエンドに届いていませんでした。** 2ヶ月の最適化が、まるごと無意味だった話を次回に。
