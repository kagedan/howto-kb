実は、**たった3つのコマンドで動きます**。Ollama をインストールして、モデルをプルして、OpenClaw のオンボーディングを実行するだけ。5分後には、完全にローカルで動作するAIアシスタントが手に入ります。

この記事では、「とにかく動かす」ことに特化して、OpenClaw × Ollama の環境構築から実践的な活用パターンまでを解説します。コスト削減、プライバシー保護、低レイテンシの恩恵を受けながら、クラウドLLMとの賢い使い分けも紹介します。

## 最初に結論: 3つのコマンドで動く

複雑そうに見えるローカルLLM環境ですが、実は以下の3コマンドで完結します。

```
# 1. Ollama インストール
curl -fsSL https://ollama.com/install.sh | sh

# 2. モデルをプル
ollama pull qwen3:8b

# 3. OpenClaw セットアップ（シンプル版）
openclaw onboard --auth-choice ollama
```

たったこれだけ。5分後には完全にローカルで動作するAIアシスタントが手に入ります。

この記事の流れは以下の通りです：

1. **セットアップ**（Step 1〜3）: Ollama → モデルプル → OpenClaw
2. **基本チャット**（Step 4）: ローカルで動作確認
3. **応用設定**（応用1〜2）: ハイブリッド設定、パフォーマンスチューニング

## OpenClaw × Ollama とは？

### OpenClaw

TypeScript製のマルチチャネルAIアシスタントフレームワークです。33万GitHubスターを獲得しており、WhatsApp、Slack、Discord等、20以上のプラットフォームに対応しています。重要なのは、**Ollamaを公式サポート**している点です。ハック的な統合ではなく、ネイティブに動作します。

<https://openclaw.ai>

### Ollama

Go製のローカルLLMランタイムです。llama.cpp をベースに、GGML形式のモデルを実行します。`ollama pull` コマンドでモデル管理ができ、セットアップが非常に簡単です。

<https://ollama.com>

### なぜこの組み合わせか？

OpenClaw には**オンボーディング・ウィザード**が用意されており、Ollama との統合を自動で設定してくれます。また、ネイティブOllama API統合により、OpenAI互換エンドポイント（`/v1`）の罠を回避できます。さらに、ローカル+クラウドのハイブリッド設定が簡単に構築できるのも魅力です。

### 最終アーキテクチャ

| コンポーネント | 実行場所 | 理由 |
| --- | --- | --- |
| OpenClaw Gateway | ローカル（npx） | 軽量、インストール不要 |
| Ollama | ローカル | GPU/CPU 統合メモリ活用 |
| ローカルLLM | Ollama 経由 | プライバシー、コストゼロ |
| クラウドLLM | フォールバック | 高精度タスク対応 |

## ローカルLLMのメリット

### コスト削減

クラウドLLMは従量課金です。頻繁に利用すると、トークンあたり $0.001〜0.1 のコストが積み重なります。例えば、開発用チャットボットで何度も試行錯誤していたら、気づいたら月3万円を超えていた...という話もあります。

一方、ローカルLLMは電気代のみです。実質ゼロコストで使い放題です。

### プライバシー保護

クラウドLLMでは、入力したデータが外部サーバーに送信されます。社内ドキュメント、機密情報を処理する場合、セキュリティポリシー違反やコンプライアンスの問題（GDPR、金融・医療規制）が生じる可能性があります。

ローカルLLMなら、データは完全にローカルで処理されます。外部送信なしで安全に利用できます。

### レイテンシはハードウェア次第

ローカルLLMのレイテンシは、使用するハードウェアに大きく依存します。GPU環境（RTX 4090等）なら数秒以内の応答も期待できますが、CPU環境では全く異なります。

クラウドAPIの応答（1〜3秒）と比べると体感で10〜60倍遅くなることも珍しくありません。**十分なスペックがない環境では、ローカルLLMはむしろ遅い**と理解しておきましょう。

### オフライン動作

インターネット接続が不要です。オフライン環境でも安定して動作します。

### 比較表

| 判断基準 | クラウド（OpenAI等） | ローカル（Ollama） |
| --- | --- | --- |
| **コスト** | $0.001〜0.1 / 1K tokens | $0（電気代のみ） |
| **プライバシー** | データ送信 | 完全ローカル |
| **精度** | 高い（最新モデル） | 中程度 |
| **レイテンシ** | 1〜3秒 | 数十秒〜数分（環境次第） |
| **インターネット** | 必須 | 不要 |

## Step 2: モデルをプル

Ollama でモデルをダウンロードします。

### 推奨モデル: Qwen3 8B

Qwen3:8b を推奨する理由は以下の通りです：

* tool calling ネイティブ対応（エージェント用途に必須）
* 多言語対応（日本語対応が良好）
* メモリ使用量が比較的少ない（4〜6GB）

### その他のオプション

```
ollama pull llama3.3          # Meta LLaMA 3.3
ollama pull mistral           # Mistral 7B
ollama pull deepseek-r1       # DeepSeek Reasoning
```

### モデル一覧の確認

### モデル選択の基準

| モデル | パラメータ数 | メモリ | 応答速度 | 精度 | 用途 |
| --- | --- | --- | --- | --- | --- |
| qwen3:1.7b | 1.7B | 1〜2GB | 数秒（CPU可） | 低〜中 | 速度優先・低スペック環境 |
| qwen3:8b | 7B | 4〜6GB | 数十秒〜1分（CPU） | 中 | 汎用（GPU推奨） |
| llama3.3 | 7B | 4〜6GB | 数十秒〜1分（CPU） | 中〜高 | 汎用（GPU推奨） |

7B モデルは高速で、一般的なタスクに十分な精度を持ちます。70B モデルは高精度ですが、48GB以上のVRAMが必要です（GPU必須）。

## Step 3: OpenClaw セットアップ

OpenClaw をセットアップします。最もシンプルな方法は以下のワンライナーです。

```
openclaw onboard --auth-choice ollama
```

これだけで、Ollama との連携が自動設定されます。`~/.openclaw/openclaw.json` が生成され、プルしたモデルが自動検出されます。

### または、ウィザード形式で詳しく設定したい場合

ウィザードが起動し、対話的に設定できます：

1. **プロバイダー選択** → **「Ollama」を選択**
2. **認証方式** → **「Local」を選択**
3. **Ollama URL入力** → デフォルト `http://127.0.0.1:11434` のまま Enter
4. **モデル選択** → `qwen3:8b` を選択
5. **完了**

### CI/CD環境向け（非インタラクティブ）

CI/CD環境など、完全に自動化したい場合：

```
openclaw onboard --non-interactive \
  --auth-choice ollama \
  --custom-base-url "http://127.0.0.1:11434" \
  --custom-model-id "qwen3:8b" \
  --accept-risk
```

## 設定ファイルをカスタマイズ

OpenClaw の設定ファイルを編集して、動作をカスタマイズできます。

### 設定ファイルの場所

```
~/.openclaw/openclaw.json
```

### 基本構造

```
{
  agents: {
    defaults: {
      model: {
        primary: "ollama/qwen3:8b",
        fallbacks: ["ollama/llama3.3"]
      },
      workspace: "~/.openclaw/workspace"
    }
  }
}
```

### 主要な設定項目

* `model.primary`: デフォルトで使うモデル
* `model.fallbacks`: プライマリが失敗した時のフォールバック
* `workspace`: エージェントの作業ディレクトリ

### モデルの切り替え

高精度モデルに変更する場合:

```
{
  agents: {
    defaults: {
      model: {
        primary: "ollama/llama3.3:70b"  // 高精度モデルに変更
      }
    }
  }
}
```

### 複数エージェントの定義（マルチユーザー向け）

用途別に複数のエージェントを定義できます。

```
{
  agents: {
    list: [
      {
        id: "fast",
        model: { primary: "ollama/qwen3:8b" },
        workspace: "~/.openclaw/workspace/fast"
      },
      {
        id: "accurate",
        model: { primary: "ollama/llama3.3:70b" },
        workspace: "~/.openclaw/workspace/accurate"
      }
    ]
  }
}
```

## 応用1: ハイブリッド設定（詳細版）

ここからは応用編です。ローカルLLMとクラウドLLMを組み合わせた**ハイブリッド戦略**を解説します。

### ハイブリッド戦略とは？

ローカルLLM（Ollama）を優先し、失敗時にクラウドLLM（OpenAI/Anthropic）にフォールバックする設定です。両方の長所を活用できます：

* **ローカルLLM**: 速度、プライバシー、コストゼロ
* **クラウドLLM**: 高精度、複雑なタスク対応

実際にOpenClawとOllamaを組み合わせた実装例も、エンジニアコミュニティで活発に共有されています。

<https://x.com/DataChaz/status/2034373172825592095>

### 使い分けのロジック

1. まずローカルLLMで処理を試す
2. タイムアウト or エラー → クラウドLLMにフォールバック
3. 複雑なタスク → 最初からクラウドLLMを指定

### 設定例1: ローカル優先、クラウドフォールバック

```
{
  models: {
    providers: {
      ollama: {
        baseUrl: "http://127.0.0.1:11434",
        apiKey: "ollama-local",
        api: "ollama"
      }
    }
  },
  agents: {
    defaults: {
      model: {
        primary: "ollama/qwen3:8b",       // ローカル優先
        fallbacks: [
          "ollama/llama3.3:70b",               // ローカル高精度
          "openai/gpt-4o-mini",                // クラウド（低コスト）
          "anthropic/claude-3-5-sonnet-20241022" // クラウド（高精度）
        ]
      }
    }
  }
}
```

この設定では、以下の順序で試行されます：

1. `ollama/qwen3:8b`（高速）
2. 失敗 → `ollama/llama3.3:70b`（ローカル高精度）
3. 失敗 → `openai/gpt-4o-mini`（クラウド低コスト）
4. 失敗 → `anthropic/claude-3-5-sonnet-20241022`（クラウド高精度）

### 設定例2: タスク別の使い分け

```
{
  agents: {
    list: [
      {
        id: "quick-chat",
        model: { primary: "ollama/qwen3:8b" },  // 雑談・軽いタスク
        workspace: "~/.openclaw/workspace/quick"
      },
      {
        id: "deep-thinking",
        model: {
          primary: "anthropic/claude-3-5-sonnet-20241022",  // 複雑なタスク
          fallbacks: ["ollama/llama3.3:70b"]
        },
        workspace: "~/.openclaw/workspace/deep"
      }
    ]
  }
}
```

### 設定例3: コスト最適化

```
{
  agents: {
    defaults: {
      model: {
        primary: "ollama/qwen3:8b",       // ローカル（$0）
        fallbacks: [
          "openai/gpt-4o-mini",                // 低コスト（$0.15/1M tokens）
          "openai/gpt-4o"                      // 高精度（$2.5/1M tokens）

```

## パフォーマンスの現実とスペック要件

ローカルLLMを快適に動かすために、スペックとモデルサイズの関係を正直に整理します。

### パフォーマンスに影響する3つの要素

OpenClawのローカル運用でボトルネックになるのは、ほぼ常に以下の3つです。

**1. モデルの重みサイズ**

モデルは全体をメモリ（VRAM/RAM）に載せて使います。量子化で圧縮できますが、限界があります。

| モデルサイズ | FP16（無圧縮） | INT4（4bit量子化） |
| --- | --- | --- |
| 7B | 約14GB | 約3.5GB |
| 13B | 約26GB | 約6.5GB |
| 70B | 約140GB | 約35GB |

**2. KVキャッシュ**

生成中に過去トークンの情報を保持するバッファで、コンテキスト長に比例して増えます。7Bモデルで 128K コンテキストを使うと、重みとは別に約5GB消費します。OpenClawはエージェント用途で長いコンテキストを必要とするため、ここが意外な落とし穴になります。

**3. CPU vs GPU**

Apple Silicon の場合、統合メモリをVRAM代わりに使えますが、GPU専用VRAMを持つマシンより帯域が劣ります。RTX 4090等の専用GPUがあれば、同じ7Bモデルで数十倍速くなります。

### スペック別の現実

「体感でストリーミングが追いつかないレベル」の目安は **100 tok/s 以上**です（人間の読書速度が約60〜100 tok/s のため）。

| 構成 | 概算コスト | 推定速度 | 70Bモデル | OpenClaw向き |
| --- | --- | --- | --- | --- |
| MacBook Pro 16GB（CPU） | 手持ち | 約1分/応答 | ❌ | ❌ |
| RTX 3060 Ti 8GB | 〜5万円 | 12〜35秒 | ❌（VRAM不足） | △（コンテキスト枯渇） |
| RTX 4090（24GB） | 〜30万円 | 30〜60 tok/s | ❌（量子化で13B止まり） | △ |
| Mac Studio M4 Ultra 192GB | 〜80万円 | 30〜50 tok/s | ✅（量子化） | ✅ |
| 2× RTX 4090（48GB合計） | 〜60万円 | 60〜100 tok/s | ✅（量子化） | ✅ |
| Radeon AI PRO R9700 | 〜40万円 | **120 tok/s**（実測） | ✅（35B） | ✅ |
| NVIDIA H100（80GB） | 〜500万円 | 200+ tok/s | ✅（FP16） | ✅✅ |

!

**個人で買える現実的な上限**: Mac Studio M4 Ultra 192GB（〜80万円）か 2× RTX 4090（〜60万円）が「ローカルLLMを本当に快適に使いたいならここから」というライン。70Bモデルを量子化で動かせ、OpenClawの12,000トークン消費を気にしなくて済む十分なコンテキスト余裕があります。

**参考実測（OpenClaw公式ガイドより）**: Qwen 3.5 35B A3B を Radeon AI PRO R9700 で動かすと約120 tok/s、入力1万トークンの処理が約4秒。

### 「それならクラウドでいいのでは？」

正直に言うと、コストだけで比較すれば：

* Mac Studio M4 Ultra 80万円 ≒ OpenAI API を **約800万円分**使える費用
* Claude Sonnet 4.6 の出力コスト: $15 / 100万トークン

ローカルLLMが経済的に合理的なのは、**プライバシー要件でクラウドに出せないデータを大量に処理する場合**に限られます。単なるコスト削減目的なら、スペックを上げるより用途を絞った方が合理的です。

## 応用2: モデル選択とパフォーマンスチューニング

モデルサイズとパラメータを調整して、パフォーマンスを最適化します。

### 7B vs 70B モデルの比較

| モデルサイズ | メモリ | 応答速度 | 精度 | 推奨用途 |
| --- | --- | --- | --- | --- |
| 1〜2B | 1〜2GB | 数秒（CPU可） | 低〜中 | 速度優先・低スペック |
| 7B | 4〜6GB | 数十秒〜1分（CPU） | 中 | 汎用（GPU推奨） |
| 13B | 8〜12GB | 数分（CPU） | 中〜高 | バランス型（GPU推奨） |
| 70B | 48GB+ | GPU必須 | 高 | 複雑なタスク、推論 |

### メモリ使用量の確認

macOS の場合、以下のコマンドでメモリ使用量を確認できます。

```
ollama ps

# 応答例:
# NAME              ID           SIZE    PROCESSOR    UNTIL
# qwen3:8b     abc123       5.2 GB  100% GPU     4 minutes from now
```

### パフォーマンスチューニング設定

```
{
  models: {
    providers: {
      ollama: {
        models: [
          {
            id: "qwen3:8b",
            contextWindow: 128000,    // 最大コンテキスト
            maxTokens: 4096,          // 出力トークン制限（短くすると速い）
            params: {
              numCtx: 8192,           // 実際に使うコンテキスト
              temperature: 0.7,       // 創造性（0〜1）
              topP: 0.9               // サンプリング
            }
          }
        ]
      }
    }
  }
}
```

### パラメータの意味

* `contextWindow`: モデルが対応する最大トークン数
* `maxTokens`: 1回の応答で生成する最大トークン数（短くすると速い）
* `numCtx`: 実際に使うコンテキストサイズ（小さくすると速い）
* `temperature`: 0に近いほど決定的、1に近いほど創造的
* `topP`: サンプリングの範囲（0.9推奨）

### コンテキストウィンドウとメモリのトレードオフ

コンテキストウィンドウを大きくすると KV cache のメモリ使用量が増加します。

| contextWindow | KV cache (概算) | 合計メモリ (7B モデル) |
| --- | --- | --- |
| 8,192 | ~0.3 GiB | ~5 GiB |
| 32,768 | ~1.3 GiB | ~6 GiB |
| 131,072 | ~5.2 GiB | ~10 GiB |

**推奨設定**:

* 8GB VRAM: `numCtx: 16384`（OpenClaw最低要件。ただし後述の問題あり）
* 16GB RAM: `numCtx: 16384` または `32768`
* 32GB+ RAM: `numCtx: 32768` または `131072`

### 落とし穴: OpenClaw 自体が約12,000トークンを消費する

!

**設定コンテキスト ≠ ユーザーが使えるコンテキスト**

OpenClaw はエージェント動作のために、ユーザーの入力前から以下のコンテキストを消費します：

* System prompt（役割・制約の定義）
* Tool schema（利用可能なツールの定義）
* Hooks（起動・終了処理）
* Memory（過去の記憶）

この合計が **約12,000トークン**。16K コンテキストを設定した場合、ユーザーが実際に使えるのは残りの **約4,000トークン**（2〜3往復分）です。

RTX 3060 Ti 8GB / 8Bモデル / 16Kコンテキストの実測では、5問でコンテキストの77%（13k/16k）を消費し、10問のテストを完走できませんでした。**単純なチャットと異なり、エージェントは1回の指示に対して思考→ツール計画→実行→結果解釈→応答と複数回LLMを呼び出すため、トークン消費はチャットの2〜5倍になります。**

この構造的な問題から、OpenClaw のローカル運用では **最低でも32K以上のコンテキスト**が推奨されます。しかし32Kに増やすとメモリ消費とKVキャッシュが増加し、GPU環境でも動作が遅くなります。

### 構成別パフォーマンス比較（実測ベース）

| 構成 | 応答時間 | コンテキスト持続 | 実用性 |
| --- | --- | --- | --- |
| CPU / 7B / 32K（MacBook 16GB） | **約1分** | 数十往復 | ❌ 遅すぎ |
| GPU 8GB / 8B / 16K（RTX 3060 Ti） | 12〜35秒 | **5往復で枯渇** | ❌ すぐ破綻 |
| GPU 8GB / 4B / 32K（RTX 3060 Ti） | **最大20分** | 改善 | ❌ さらに遅い |
| GPU 16GB以上 / 7B〜13B / 32K+ | 数秒〜十数秒 | 数十往復 | △ 実用域 |
| GPU 48GB+ / 35B / 128K+（AMD R9700等） | 数秒（実測4〜20秒） | 大量 | ✅ 快適 |

この表が示すのは、**設定で解決できる問題ではなく、VRAMの物理的上限に起因する構造的なトレードオフ**が存在するということです。「8GB VRAMでも動く」という情報は技術的には正確ですが、「使える」かどうかは全く別の話です。

### GPU/CPU の使い分け

GPU が検出されていれば自動的にGPUを使用します（高速）。GPU がない場合はCPUで実行されます（遅いが動作する）。

確認方法:

```
ollama show qwen3:8b
# → "Processor: GPU" または "Processor: CPU" と表示
```

### 速度優先の設定

応答速度を最優先する場合:

```
{
  models: {
    providers: {
      ollama: {
        models: [
          {
            id: "qwen3:8b",
            maxTokens: 1024,          // 短い応答（速い）
            params: { numCtx: 4096 }  // 小さいコンテキスト（速い）
          }
        ]
      }
    }
  }
}
```

### 精度優先の設定

精度を最優先する場合:

```
{
  models: {
    providers: {
      ollama: {
        models: [
          {
            id: "llama3.3:70b",       // 大きいモデル
            maxTokens: 8192,          // 長い応答
            params: { numCtx: 32768 } // 大きいコンテキスト
          }
        ]
      }
    }
  }
}
```

## トラブルシューティング

実際のセットアップ中に遭遇した問題を時系列で記録します。

### Ollama インストール編

#### 1. Ollama が起動しない

**原因**: ポート 11434 が既に使用中

**解決策**:

```
# 使用中のプロセスを確認
lsof -i :11434

# 既存のプロセスを終了
pkill ollama

# Ollama を再起動
ollama serve
```

#### 2. GPU が認識されない

**原因**: GPU ドライバーが古い、または CUDA 未インストール

**解決策**:

* macOS: Apple Silicon なら自動認識される
* Linux: CUDA ドライバーを最新版にアップデート
* Windows: NVIDIA GPU ドライバーを更新

確認方法:

```
ollama show qwen3:8b
# → "Processor: GPU" と表示されれば OK
```

### モデル取得編

#### 3. モデルが見つからない

**原因**: ローカルにモデルをプルしていない

**解決策**:

```
# モデル一覧を確認
ollama list

# モデルをプル
ollama pull qwen3:8b
```

#### 4. モデルダウンロードが遅い / タイムアウト

**原因**: ネットワーク接続が不安定

**解決策**:

```
# タイムアウト時間を延長
OLLAMA_DOWNLOAD_TIMEOUT=600 ollama pull qwen3:8b
```

### OpenClaw セットアップ編

#### 5. `openclaw: command not found`

**原因**: npx でのインストールが失敗している

**解決策**:

```
# Node.js のバージョン確認（18.x 以上必要）
node --version

# npx で再実行
npx openclaw@latest onboard
```

#### 6. オンボーディングがフリーズする

**原因**: Ollama が起動していない

**解決策**:

```
# Ollama の状態確認
curl http://localhost:11434/api/tags

# 応答がない場合は Ollama を起動
ollama serve
```

#### 7. モデル自動発見が失敗する

**原因**: 明示的なプロバイダー設定が干渉している

**解決策**:

```
# 設定ファイルを削除して再セットアップ
rm ~/.openclaw/openclaw.json
npx openclaw onboard
```

### 実行編

#### 8. `Connection refused`

**原因**: Ollama が起動していない、または URL が間違っている

**解決策**:

```
# Ollama を起動
ollama serve

# URL を確認（デフォルトは http://127.0.0.1:11434）
curl http://localhost:11434/api/tags
```

#### 9. ツール呼び出しが動作しない

**原因**: `/v1` OpenAI 互換エンドポイントを使用している

**解決策**:

* ネイティブ Ollama API URL を使用（`http://host:11434`、パスなし）
* 設定ファイルで `baseUrl` を確認:

  ```
  {
    models: {
      providers: {
        ollama: {
          baseUrl: "http://127.0.0.1:11434"  // /v1 を付けない
        }
      }
    }
  }
  ```

#### 10. 応答が異常に遅い（10秒以上）

**原因**: CPU のみで実行している

**解決策**:

```
# GPU が検出されているか確認
ollama ps

# GPU が使われていない場合は Ollama を再インストール
# macOS: brew reinstall ollama
# Linux: GPU ドライバーを更新後、Ollama を再インストール
```

### メモリ・パフォーマンス編

#### 11. OOM（Out of Memory）エラー

**原因**: モデルサイズが VRAM/RAM を超えている

**解決策**:

* 小さいモデルに変更（70B → 7B）
* コンテキストウィンドウを縮小:

  ```
  {
    models: {
      providers: {
        ollama: {
          models: [
            {
              id: "qwen3:8b",
              params: { numCtx: 4096 }  // 8192 から縮小
            }
          ]
        }
      }
    }
  }
  ```
* SWAP を増やす（Linux）

#### 12. コンテキストウィンドウの最低要件エラー

**原因**: `numCtx` が OpenClaw の最低要件（16000）を下回っている

**解決策**:

```
{
  models: {
    providers: {
      ollama: {
        models: [
          {
            id: "qwen3:8b",
            contextWindow: 128000,
            params: { numCtx: 16384 }  // 最低 16000 以上
          }
        ]
      }
    }
  }
}
```

#### 13. `100% context used` アラート

**原因**: コンテキストウィンドウが小さすぎる

**解決策**:

* コンテキストウィンドウを拡大:

  ```
  {
    models: {
      providers: {
        ollama: {
          models: [
            {
              id: "qwen3:8b",
              params: { numCtx: 32768 }  // 8192 から拡大
            }
          ]
        }
      }
    }
  }
  ```
* 十分なメモリがあれば 128K まで拡大可能（7B モデルで約 12 GiB 必要）

### モデル選定編

#### 14. モデルがツール呼び出しに対応していない

**原因**: モデルがネイティブ tool calling をサポートしていない

**解決策**:

* ツール対応モデルに変更:
  + ✅ `qwen3:8b`, `qwen2.5:14b`
  + ✅ `llama3.3`, `llama3.1`
  + ❌ `gemma3:12b`（コミュニティ版 `orieg/gemma3-tools` なら可能）

確認方法:

```
# モデル情報を確認
ollama show qwen3:8b

# "tool_use" または "function_calling" が含まれているか確認
```

### デバッグ方法

問題が発生した場合、以下のコマンドで状態を確認します。

```
# Ollama の状態確認
curl http://localhost:11434/api/tags

# OpenClaw のログ確認
openclaw gateway --verbose

# モデルの詳細情報
ollama show qwen3:8b

# メモリ使用量の確認
ollama ps
```

## 学び

実際にローカルLLM環境を構築してみて、以下の知見を得ました。

### Ollama はローカルで動かすべき

Docker コンテナ内で Ollama を動かすこともできますが、以下の理由でホストOS上での実行を推奨します：

* **メモリ効率**: Docker Desktop on Mac は Linux VM 経由で動作するため、メモリ割り当てに制限がある
* **GPU アクセス**: Apple Silicon の統合メモリを活かすには、ホスト側で動かす方が効率的
* **パフォーマンス**: VM のオーバーヘッドがない分、応答速度が向上する

Docker を使う場合は、OpenClaw Gateway のみをコンテナ化し、Ollama はホスト側で動かすハイブリッド構成がベストです。

### ローカルLLMでのモデル選定は「ツール対応」が最重要

エージェント用途では tool calling（function calling）対応が必須です。人気モデルでも対応していない場合があるため、事前確認が重要です。

| モデル | ツール対応 | 備考 |
| --- | --- | --- |
| Gemma 3 (12B/27B) | ❌ | コミュニティ版 `orieg/gemma3-tools` なら可能 |
| Qwen 3.5 (9B) | ✅ | ネイティブ対応 |
| Qwen 2.5 (14B) | ✅ | ネイティブ対応 |
| Llama 3.1 (8B) | ✅ | ネイティブ対応 |

### コンテキストウィンドウとメモリのトレードオフ

コンテキストウィンドウを大きくすると KV cache のメモリ使用量が増加します。

| contextWindow | KV cache (概算) | 合計メモリ (7B モデル) |
| --- | --- | --- |
| 8,192 | ~0.3 GiB | ~5 GiB |
| 32,768 | ~1.3 GiB | ~6 GiB |
| 131,072 | ~5.2 GiB | ~10 GiB |

Mac のメモリに余裕があるなら大きく取って問題ありませんが、メモリが限られている場合は慎重に設定する必要があります。

**筆者の環境（16GB RAM）では**: 7Bモデル + 32K コンテキストで動作しました。128Kコンテキストは約10GBメモリを使うため、他のアプリと併用するとギリギリです。

### エージェントはチャットの2〜5倍トークンを消費する

OpenClaw はチャットツールではなく**エージェント**です。1回のユーザー指示に対して内部で複数回 LLM を呼び出します：

```
ユーザー指示
  → 思考（LLM呼び出し1回目）
  → ツール計画（LLM呼び出し2回目）
  → ツール実行
  → 結果解釈（LLM呼び出し3回目）
  → ユーザーへの応答
```

さらに OpenClaw はエージェント動作のために約12,000トークンを system prompt 等に使用します。**16K コンテキストで設定しても、ユーザーが実際に使えるのは残りの約4,000トークン（2〜3往復）です。**

### 小さいモデルはシステムプロンプトの遵守が弱い

7B〜9B モデルだと「セッション開始時に MEMORY.md を読んで記録せよ」のような複雑な指示を自律的に守らない場面があります。明示的に「メモリに保存して」と指示すれば実行はできますが、完全な自律性は期待できません。

より大きなモデル（32B+）なら改善が期待できますが、メモリ要求も増加します。用途と環境に応じて適切なモデルサイズを選びましょう。

### OpenClaw の設定のコツ

* **JSON5 形式**: コメント、末尾カンマが使えるため、設定ファイルの可読性が高い
* **ネイティブ API URL を使う**: `/v1` エンドポイントは使わない（ツール呼び出しが不安定）
* **フォールバック戦略**: ローカル優先、クラウドフォールバックの設定を最初から組み込んでおくと安心

## ローカルLLMの実用性と限界

実際に動かしてみて感じた、ローカルLLMのリアルな評価を記します。

### 正直なところ、使い所は難しい

ローカルLLMには以下の課題があります：

* **遅い**: クラウド API と比べると体感で大幅に遅い。筆者の環境（16GB RAM / Apple Silicon / CPU推論）では、7B モデル（qwen3:8b）で**約1分**かかりました。RTX 3060 Ti 8GB のGPU環境でも 12〜35 秒、4B モデルを 32K コンテキストで動かすと**最大20分**かかったという実測報告もあります。
* **コンテキストがすぐ枯渇する**: OpenClaw 自体が約12,000トークンを消費するため、16K コンテキスト設定でも実際に使えるのは約4,000トークン（2〜3往復）。5〜6問でコンテキストが尽きます
* **VRAMの構造的ジレンマ**: 8GB VRAMでは「速度とコンテキストのどちらかを犠牲にする」トレードオフを設定で解決できません
* **小さいモデルは自律性が弱い**: エージェントとしての自律的な振る舞い（メモリ保存、ルール遵守）が弱い

### ただのチャットツールなら既成品で十分

ローカルLLMでチャットしたいだけなら、Open WebUI や Jan などの既成ツールの方がセットアップも簡単で UI も洗練されています。OpenClaw をわざわざ構築する意味は、エージェントとしてのツール実行や自律的なタスク処理にあります。

### OpenClaw がハマるユースケースは？

正直、まだ「これだ」というユースケースが見つかっていませんが、以下のような用途には向いていそうです：

* **定期的な Web サイト巡回・リサーチ**: スケジュール定義し、エージェントが自律的に情報収集してメモリに記録する
* **長期間にわたるプロジェクトの文脈保持**: workspace の記憶機構により、セッションをまたいだコンテキストを維持できる
* **セキュリティが重要な環境での AI エージェント**: 完全ローカルで動かし、外部アクセスを制限できる

逆に、単発の質問応答や簡単な自動化には明らかにオーバースペックです。OpenClaw の真価は「エージェントに人格と記憶を持たせて、長期的に育てていく」という使い方にあると思います。今後、より大きなモデルやクラウド API と組み合わせたときに、本領を発揮するのかもしれません。
