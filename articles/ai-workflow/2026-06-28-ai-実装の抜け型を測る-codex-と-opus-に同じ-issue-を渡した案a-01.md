Opus に実装を任せていると、「機能は動くのに要件が漏れている」という事象が繰り返し起きた。

バグではない。エラーは出ない。デモもちゃんと動く。でも後日確認すると、指定したはずの挙動の一部が実装されていない。私たちはこれを「抜け型」と呼んでいる。

原因の仮説が2つあった。

1. Opus の推論品質の問題（複雑な要件を網羅できない）
2. issue の仕様書の問題（暗黙の前提が多すぎて、モデルが察知できない）

この2つは意味がまったく違う。モデルの問題なら乗り換えを検討できる。仕様書の問題なら、精緻化すれば済む話だ。

切り分けたかった。

## 実験デザイン

### 補修 PR の逆算

実験の評価基準をどう作るかが先に問題になった。「AI が書いたコードを人間が読んで採点する」方式は主観が入りすぎる。

そこで使ったのが**補修 PR の逆算**だ。

「実装 → 抜け発覚 → 補修 PR」という構造を持つ issue では、補修 PR の差分が「当時の実装が取りこぼした要件そのもの」になる。この差分を逆算すれば、客観的なチェックリストが作れる。

題材に選んだのは過去に抜けが起きた 3 つの機能だ。

* **機能 A**：固有名詞辞書（音声認識への語句注入）
* **機能 B**：AI 補正 UI（文字起こし後の誤字検出パネル）
* **機能 C**：低確度語句編集 UI（信頼度の低い語句のハイライトと修正）

それぞれチェックリストを用意した（N = 15 / 16 / 11）。

### cold-start 再現

各 issue の「実装コミットの親コミット」からブランチを切り直した。当時の開発者が着手したときと同じコードベースを再現する。

### 入力の同一化

Codex と Opus に**まったく同じプロンプト**を渡す。issue 本文と着手前の調査コメントだけを蒸留したセット。チェックリスト（答え）は渡さない。

## 案 A の結果

案 A は「履歴の実 Opus vs Codex」の比較だ。

Opus の列は過去の実装履歴から確定している。補修 PR が存在するということは、当時の実装がその要件を取りこぼしていたことを意味する。Codex だけ新規に実装させた。

| 機能 | 全要件 N | Opus 抜け | Codex 抜け |
| --- | --- | --- | --- |
| 機能 A（固有名詞辞書） | 15 | 3 | 1（軽微） |
| 機能 B（AI 補正 UI） | 16 | 5 | 0 |
| 機能 C（低確度語句編集 UI） | 11 | 7 | 0 |
| **合計** | **42** | **15** | **1** |

Opus 15 抜け、Codex 1 抜け。

## 「抜け型」の質的分析

数字より、**どう抜けたか**の方が本質だと思う。

Opus の抜けには 3 つのパターンがあった。

### パターン 1：新パラメータの「全経路横展開」漏れ（機能 A）

固有名詞辞書の主経路（初回アップロード時の語句注入）は完全に実装した。問題はそこから先だった。

このシステムには、ジョブの再実行経路が 3 種類ある。①失敗ジョブの手動再実行、②タイムアウト時の自動リトライ、③フォールバック経由の再試行。3 つすべてで語句を渡し忘れた。各経路が別のレイヤ（フロントエンド API・バックエンド Python・UI コンポーネント）に散っているため、それぞれに横展開する必要があった。

issue には「再実行時も語句を引き継ぐこと」とは書いていなかった。「機能を追加したら全経路に通して当然」という暗黙の前提で落ちた。

実害は大きい。初回の文字起こしでは固有名詞補助が効いているのに、再実行した瞬間に無言で消える。エラーは出ないので気づきにくく、後日の調査で発覚した。

### パターン 2：「複数レイヤの接続」つなぎ忘れ（機能 B）

「A が B を呼ぶ」UI は完成した。しかし「B のエンドポイント自体を作る」という一段下のレイヤが抜けた。

バックエンドの API エンドポイントとフロントエンドのルート、2 層が未作成のままクローズされた。呼び出しを行っても 404 が返る。UI は完成して見えるのに、実際には何も起きない。

issue には「UI を実装する」とあったが「そのUI が叩くサーバー側も作る」は書いていなかった。接続の両端を揃える責任が暗黙の前提になっていた。

### パターン 3：スコープの後半放棄（機能 C）

issue はバックエンド処理・データベース・フロントエンドの 3 層すべてを明示的に要求していた。バックエンドと DB は完全実装してクローズした。フロントエンドには手をつけなかった。

パターン 1・2 と違い、これは暗黙の前提ですらない。明記されていたスコープの後半をそのまま放棄した。

### 3 パターンに共通する構造

```
主経路（happy path）を完成させる
    ↓
デモが動く
    ↓
「当然届いているはず」の経路が実は届いていない
    ↓
後日の調査・再実行・別経路での利用で発覚
```

Opus は「機能を作る」という仕事はできていた。できていなかったのは「その機能が届くべき全範囲（surface area）を網羅する」という、明示されないまま当然とみなされる仕事だった。

### Codex の実装の特徴

Codex の唯一の抜けは機能 A の「メトリクス記録の粒度不足」だ。語句数を数値で確認しづらいが機能は完全に動く。重大度は低く、そもそも issue 本文に記載のない要件だった。

機能 A で Codex が書いたコードを見ると、辞書語句をコンポーネントスコープで一度算出しクロージャ経由で全経路から参照する構造を選んでいた。Opus が踏んだ「呼び出し忘れ」というクラスのバグを、設計で構造的に消していた。

## 誠実な留保

案 A の数字を見て「Codex に乗り換えよう」と判断するのは早い。

**入力品質の差**という交絡がある。

Codex には蒸留済みのプロンプト（issue 本文 + 着手前コメントのみ）を渡した。一方 Opus の列は当時の実際の開発で、複数コミットにわたる漸進実装とフィードバックループの産物だ。「Codex の抜けが少ない」の一部は「入力が整理されていたから」かもしれず、純粋なモデル差と切り分けられない。

その他の留意点：

* n = 3 は小標本。3 本の傾向が偶然の可能性は否定できない
* Opus 列は cold-start でなく履歴なので条件が非対称
* 測定の落とし穴：1 つの実験で `git add -A` がツールファイル 80 万行を巻き込んだ。着手地点の `.gitignore` に除外行が未設定だったのが原因で実装能力とは無関係だが、古い base で実験するとこういうノイズが混入する

## なぜまだ結論が出ないか

決定的なテストは**案 B**だ。

案 B では、準備文脈を持たないまっさらな Opus セッションに Codex と同じ蒸留済みプロンプトだけを渡して実装させる。入力条件を揃えた上で結果を比べる。

* **案 B の Opus も抜けが少なければ** → 当時の抜けの主因は入力の暗黙前提。乗り換えでなく仕様精緻化が正解
* **案 B の Opus が依然抜ければ** → 主因はモデル。そのとき初めて移行が選択肢になる

案 A は「どこに問題があるか」を絞るための予備結果だ。ただ案 A 単体でも言えることがある。

## Opus の欠点を知ると活用が変わる

Opus の強さは「主経路の設計と実装」にある。問題は「surface area の網羅」という暗黙の仕事を自力でやりきれないケースがあることだ。

この欠点の正体を知ると、使い方が変わる。

* 「このパラメータが通るべき全経路を列挙して確認してほしい」と実装後に明示的に指示する
* issue に surface area を書き出す（「この機能が触れるべき全レイヤ」を列挙する）
* 「実装 → レビュー → 補修」の 2 段構成を前提にフローを設計する（1 発完成を期待しない）

「Claude 最強」という記事はたくさんある。でも私が実際に使ってきて感じるのは、最強というより「何が得意で何に暗黙前提が弱いかを知っている人だけが使いこなせる」ツールという感覚だ。

Codex が上回った、という結果が出た。それはそれで正直に書く。でも同時に、Opus の抜け方のパターンを理解できたのは収穫だったと思っている。「なぜ抜けるか」がわかれば、対策は「乗り換え」だけじゃないからだ。
