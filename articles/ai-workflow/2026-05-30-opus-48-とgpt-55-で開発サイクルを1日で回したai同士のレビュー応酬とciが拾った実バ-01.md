# Zennウェブエディタ用：Opus 4.8 技術編

> 弁護士・税理士の太田垣（[@yoshiki\_otagaki](https://twitter.com/yoshiki_otagaki)）です。  
> 士業の現場で実際に動かしている AI 活用・業務自動化を発信しています。  
> Next.js + Supabase + Anthropic API + 独自 MCP サーバで法律事務所向け SaaS「LegalFlow」を1人で開発・運用中。

本記事は、note 版「[Opus 4.8 と1セッションで、法務SaaSの設計・実装・レビュー・マージまで丸ごと回した記録](https://note.com/yoshiki_otagaki)」の**技術編**です。「AIと開発サイクルを回した」物語の側はそちらに譲り、本記事では**実装の技術的論点**を掘ります。

題材は、私が1人で開発・運用している弁護士向け SaaS「LegalFlow」の改修。Opus 4.8 で設計・実装、GPT-5.5 でコードレビュー、人間が裁定、CI で検証、というラリーを1セッションで回しきった記録から、**他の開発でも転用できそうな論点**を抜き出します。

## TL;DR

* **PR を機能ごとに4本に分割**（フェーズ機能・Teams 連携・UI/UX P0・型ホットフィックス）。1つのセッション内でも、PR の単位は守るべき
* **認証方式選択（アプリ権限型 vs 委任型）はAIに決めさせない**。業務実態を握っている人間の判断領域
* **CI の整備自体をAIが提案**し、それが直後に**実バグを検知**して救った（クライアント／サーバー境界違反）
* **AI 同士のコードレビュー応酬**（Claude 実装 → GPT レビュー → 人間裁定 → Claude 修正）で、セキュリティ重要指摘が拾えた
* **AI のミスより人間のマージミスのほうがリスキー**。コンフリクト解決ミスで `main` を壊した。**ホットフィックスPRと CI で復旧**

## 1. PR の単位設計

セッション内で実装が長くなると、AI は「ひとつのブランチで全部やりきろう」とする傾向があります。これに乗ると、レビューが破綻し、ロールバックも困難になる。

明示的に\*\*「PR は機能単位で分けて」\*\*と最初に指示したうえで、以下の4本に分割しました。

| PR | 内容 | サイズ感 |
| --- | --- | --- |
| #9 | 事件フェーズ機能（DB マイグレーション・型・サーバー処理・UI） | 中〜大 |
| #10 | Microsoft Teams / SharePoint 連携の基盤（OAuth・トークン保存・非同期ジョブ） | 大 |
| #11 | UI/UX P0 改修（ボード新設・ヘッダー整理・設定メニュー集約） | 中 |
| #12 | 型定義のホットフィックス | 小 |

PR #9 は本来もっと細かく分けるべきでしたが、フェーズ機能は **DB・型・UI が密結合**しており、分割するとレビュー難度が上がるため、敢えて1本にまとめました。**「分割して整理しやすくなる場合のみ分割する」**——この判断もAIに丸投げせず、人間が握りました。

## 2. 認証方式の選択：AIに決めさせない領域

Microsoft Graph 連携で必ず出てくる論点が、認証方式の選択です。

| 方式 | 仕組み | 向き |
| --- | --- | --- |
| **アプリ権限型**（Application Permissions） | Azure AD の管理者が一括で権限を承認、サーバーアカウントとして動作 | 中〜大規模組織。一元管理が必要 |
| **委任型**（Delegated Permissions） | ユーザーごとに OAuth でサインインさせ、ユーザーの権限で API を呼ぶ | 個人ユーザー・小規模・「自分の Teams を操作する」用途 |

両者は技術的にもセキュリティモデルも違います。**AIはこの選択を独断せず、選択肢と論点を私に提示してきました**。

LegalFlow の現状ユーザーは「1弁護士＝1 Microsoft アカウント」が圧倒的多数。事務所単位での一元的なテナント管理者承認は、現実的に取りにくい。一方、将来的にマルチテナント SaaS として外販するなら、アプリ権限型のほうが拡張性が高い。

**委任型を選択**しました。マルチテナント拡張性を一部犠牲にしてでも、今のユーザー実態に合わせた選択です。判断の論理は明確にして、後から見て理由がわかる形で記録しています。

> **教訓**：AIは選択肢を整理してくるが、**業務の実態に照らして決める**のは人間の仕事。AIに「どっちがいい?」と聞くと、それっぽい回答が返ってきてしまい、判断の責任が曖昧になります。**選択肢と論点をAIに整理させ、判断は自分でする**——この役割分担を明示するのが重要。

## 3. トークンの暗号化保存と本番鍵必須化

委任型 OAuth では、各ユーザーのリフレッシュトークンを永続化する必要があります。**これを平文でDB保存すると、DB漏洩 = 全ユーザーの Microsoft アカウントが乗っ取られる**という、致命的なリスクが生じます。

最初の Claude の実装は、**この点が甘かった**。トークンを平文相当でDB保存していました。

ここを救ったのが、後述する**GPT-5.5 によるコードレビュー**です。指摘内容:

1. トークンは**サーバー側で対称鍵暗号化**してから DB に保存すべき
2. 本番環境では**暗号化鍵が環境変数に未設定だと起動できない**ようにすべき（フェイルセーフ）
3. 開発環境では緩めても良いが、本番モードでは厳格チェック必須

これを Claude に投げ返して修正。最終的に、

* AES-GCM ベースの暗号化ユーティリティ
* 環境変数からの鍵読み込みと、本番モードでの存在検証（未設定なら起動失敗）
* DB スキーマレベルで、トークンカラムを「暗号化済み前提」と明示

という構造になりました。**セキュリティ重要事項を、AI 単独の実装で見落とすリスクが顕在化した**好例です。

## 4. テナント越境防止のガード

もう一つ、GPT-5.5 から強烈な指摘が来たのが**テナント越境**でした。

具体的には、ユーザー A の OAuth トークンを使って、誤って別テナントのリソース（ユーザー B の Teams）を操作してしまうリスク。SaaS としては絶対に起きてはいけない事故です。

Claude の初期実装では、リクエスト処理層で「現在ログイン中のユーザーのトークン」を取得して使う、というシンプルな構造でした。これだけでは、

* ジョブキュー経由の非同期処理で、誤ったユーザーコンテキストが混じる可能性
* マルチセッション環境で、トークンの取り違えが起きる可能性

が残ります。

修正後は、

* **すべてのトークン使用箇所で `tenant_id` と `user_id` を明示的に渡す**
* API クライアント層で、**トークンとペアのユーザー識別子が一致しない場合は即エラー**
* 非同期ジョブのペイロードに、**実行コンテキスト（誰の依頼で動いているか）を必ず含める**

という構造に。これらは「動作には影響しないが、事故時の被害を局所化する」防御層です。AI実装で抜けがちで、レビューで拾うべき典型例。

## 5. CI が拾った実バグ：クライアント／サーバー境界

今回のセッションで一番ヒヤッとしたのが、**CI が直後に拾った実バグ**でした。

CI（GitHub Actions でビルド・型チェック・Lint を自動実行）の設定は、今回のセッションで Claude が「**PR を投げたらビルドを自動で走らせるべき**」と提案して入れたものです。

これを入れた直後の PR で、CI が早速エラーを出しました。**サーバー専用 API を、クライアント側（ブラウザで実行されるコード）から import している箇所**があった。

Next.js 系のフレームワークでは、Node.js 専用モジュール（ファイルシステム API など）はクライアントバンドルに含めるとビルドが壊れます。これは型チェックだけでは拾えず、**ビルドを実際に走らせて初めてわかる**バグでした。

もし CI なしでマージしていたら、本番でブラウザを開いた瞬間に**白い画面**が出ていたはずです。CI が安全網として機能した瞬間でした。

**教訓**：AIに任せる速度が上がるほど、CIの価値は相対的に増します。「ビルドが通る」「型が通る」「Lint が通る」だけでも、AI実装の事故率は劇的に下がる。**CI自体の整備をAIに頼める**時代でもあるので、最初のセッションで必ず入れることをおすすめします。

## 6. AI 同士のレビュー応酬：実際の運用フロー

今回のセッションで最も新しい体験だったのが、**Claude（Opus 4.8）の実装に GPT-5.5 でコードレビューをかける**ラリーでした。

実際の運用フローは以下:

```
1. Claude: PR を作成（実装＋自己レビュー）
2. 人間: PR の diff を GPT-5.5 にコピペし、レビュー依頼
   「セキュリティ・パフォーマンス・保守性で厳しく」
3. GPT-5.5: 数十項目の指摘を返す
4. 人間: 指摘を一つずつ「受諾／反論」を判断
5. 受諾分を Claude に投げ返す
   「これらの指摘に対応してください」
6. Claude: 修正コミットを追加
7. CI を走らせる → 通れば 8 へ、落ちたら 6 に戻る
8. マージ
```

3〜5 ラウンドで収束します。重要なのは、**人間がジャッジを握ること**。GPT-5.5 の指摘の中には、

* セキュリティ・本物の指摘（受諾すべき）
* 設計意図を理解せずに「ベストプラクティスに反する」と言ってくる過剰指摘
* 文化的・スタイル的な好みの違い

が混在しています。**全部受諾すると、AIの好みに振り回されて本来の設計が崩れる**。**全部却下すると、本物の指摘を逃す**。人間が要不要を見極めるのが、このラリーの肝です。

私が体感した受諾／却下の比率は、\*\*ざっくり「受諾4：却下6」\*\*くらい。受諾した指摘の質は高く、却下した指摘の中にも「論点として一理ある」ものが含まれていました。**1人開発のレビュー不在問題を、別AIが部分的に埋めてくれる**——この体験は明らかに開発の質を上げます。

## 7. 私のミス：マージコンフリクトで `main` を壊した

正直に書きます。**今回最も大きなミスは、AIではなく私が起こしました**。

PR #11 をマージするとき、PR #9 と #10 のマージ後の状態とコンフリクトしていて、私が手動でコンフリクト解決をしました。このとき、**両ブランチの修正を統合すべき箇所で、片方を消してしまった**。

結果、`main` のビルドが壊れました。CI が即座に赤旗を上げ、本番デプロイは止まりました。

対応は素早くできました。

1. CI の失敗ログから、欠落している型定義を特定
2. ホットフィックス用のブランチを切る
3. Claude に「この型定義が欠落したので戻して」と指示
4. PR #12 として最小diffで修正
5. CI グリーンを確認してマージ

10分くらいで復旧。本番ユーザーには影響なし。

**教訓**：AI の作業速度が上がると、**人間のマージ操作のミスが相対的にボトルネックになる**。「コンフリクトの解決もAIに任せる」「マージはGitHub UI でなく自動マージツールに任せる」など、人間の手作業を減らす方向に進めるべき。

## 8. まとめ：これから AI 開発を本格化する人へ

今回のセッションから得た、実用的な指針です。

1. **PR は機能単位で分ける**。AI に丸投げすると1ブランチで全部やろうとするので、明示的に指示する
2. **業務実態に関わる判断は AI に決めさせない**。選択肢の整理はさせ、判断は自分でする
3. **CI を最初に整備する**。AI が CI 整備も書いてくれる。これが安全網の中核
4. **別AI でコードレビューさせる**。1人開発のレビュー不在問題に強烈に効く。Claude vs GPT は相性が良い
5. **人間のマージミスを警戒する**。AI の作業より、人間のGit操作のほうがバグを生みやすい
6. **セキュリティ系（暗号化・テナント分離・本番鍵必須化）はレビューで拾う**。AI の初期実装では甘くなりがち
7. **修正サイクルが速くなると、できることが質的に変わる**。3週間→1日のスピード変化は、開発体験を根本から変える
