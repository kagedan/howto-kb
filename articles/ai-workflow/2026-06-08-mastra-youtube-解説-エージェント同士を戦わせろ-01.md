## 概要

「設定候補が多すぎて、どのエージェント構成が本当に良いのか人間には追い切れない」。この動画で ZeroEntropy の Nicholas Pipitone が提示する答えは明快で、**候補を全部並べて競わせ、リーダーボードで最適なものを選ぶ**というものです。マルチエージェント構成、RAG パイプライン、埋め込みモデル、推論バッチング戦略など、選択肢が爆発的に増える領域では、手作業のチューニングより競争ベースの探索を前提にしたほうが速い、という立場です。

発表の面白いところは、この「戦わせる」という比喩が単なる演出ではなく、検索品質の改善、GPU 推論最適化、さらには任意の最適化タスクにまで一貫して適用されている点です。文書同士をクエリに対する有用性でペア比較し、その結果を Elo 風スコアに落とす。推論ログを見たエージェント群に、どの GPU へどのバッチを送るべきか競わせる。さらに、知識ベースを持つエージェント群に「隠された最適化関数」を解かせ、どの埋め込みモデルが最も早く正解に近づけるか比較する。講演全体が、**探索を競技化することで最適化を前進させる**という一本の思想で貫かれています。

## 要点

1. ZeroEntropy は、エージェントや検索システムの設計候補を総当たりで試すのではなく、候補同士を競わせて順位付けするアプローチを取っています
2. 検索では、クエリに対して2文書をペア比較させる形で LLM に判定させ、その結果をもとに Elo 風の訓練信号を作ります
3. この訓練信号から、実運用ではクエリと文書だけで高速にスコアリングできる 0.6B 級のリランカーへ蒸留しています
4. GPU 推論最適化でも同じ発想を使い、複数エージェントがログと負荷状況を見ながら最適なバッチ分割や GPU 配置を競います
5. その結果、P99 レイテンシを約 70% 改善できたと説明しています
6. デモでは 1500 本の arXiv 論文を知識ベース化し、ZeroEntropy、OpenAI、Cohere の埋め込みモデルで「どれが最適化タスクを最も良く支援するか」を比較しました
7. エージェントが最適化対象そのものを見られないよう、秘密の目的関数は別マシンに置き、評価だけを返す構成にしています
8. 論文検索エージェント、サンドボックス実行、インデックス作成、OCR などの部品は Mastra と Vercel AI SDK で比較的少ないコード量で組めると紹介しています
9. Q&A では PDF 取り込み時は Markdown 化が重要で、DeepSeek OCR と従来型 OCR をコストに応じて使い分けていると述べました
10. うまくいった知見は巨大な Markdown ベースの内部知識庫や公開論文として蓄積し、再利用可能な形へ寄せていると説明しています

## 詳細

### 設定空間が広すぎるなら、人間が選ぶのではなく競わせる

Nicholas の出発点はかなり現実的です。いまエージェントを作ろうとすると、構成の選択肢が多すぎます。シングルエージェントにするのか、マルチエージェントにするのか。パイプライン型か、並列型か。どのモデルを使うか。RAG の前処理や検索戦略はどうするか。これらを人間がひとつずつ試し、主観と少数のベンチマークだけで決めるのは限界があります。

そこで彼らは「let them cook」と言います。候補を大量に生成し、同じ問題に対して一斉に走らせ、リーダーボードで勝敗を見ていく。重要なのは、単にたくさん試すことではなく、**比較可能な形にして順位付けする**ことです。勝ち負けが見えると、探索の方向がはっきりします。これはエージェント開発を勘や雰囲気から、継続的な最適化へ引き戻す考え方だと言えます。

---

### 検索では「文書同士を戦わせる」とリランカーの訓練信号になる

この発表の中核にあるのが、検索におけるペアワイズ比較です。巨大なコーパスを前にすると、どの文書がクエリに対して本当に有用かは一目では分かりません。そこで、クエリと2本の文書を LLM に見せ、「より役立つのはどちらか」を判定させます。

この仕組みの良いところは、LLM に複雑な絶対評価をさせるのではなく、相対比較に落としている点です。相対比較なら判断しやすく、その結果を大量に集めることで Elo 風のスコアを作れます。Nicholas は、この訓練方法を Zelo という Elo 着想の手法として説明しています。

そして実運用では、毎回その場で文書総当たりの比較をするわけではありません。比較結果を訓練信号として、クエリと文書の組だけでスコアを出せる小型モデルへ蒸留する。動画では、**0.6B のリランカーが単一 H100 上で毎秒 100 万トークン近い速度で動き、数千文書を走査できる**と語られています。つまり、競争は学習時に集中的に行い、推論時には安く速く使う構成です。

---

### GPU 推論の最適化も、実は同じ「競争問題」として扱える

発表が面白くなるのは、同じ発想を検索だけに閉じないところです。ZeroEntropy では推論基盤の最適化にも競争の仕組みを持ち込みます。顧客ごとにリクエスト量も文書長も異なる中で、どの GPU に、どのバッチサイズで、どの順序で投げるのが最適かは簡単ではありません。

Nicholas は、推論ログを材料に複数のエージェントがスケジューリング戦略を競う仕組みを説明します。短い文書群は大きなバッチで一部の GPU に送り、長い文書群は小さなバッチで別 GPU に送る。8 枚の H100 ができるだけ同時に仕事を終えれば、全体の待ち時間を減らせます。ここでも評価指標は P50、P90、P99 といったレイテンシ指標で、リーダーボードの勝者がより良い戦略として採用されます。

最終的にその戦略は Python ファイルとして出力され、実運用へ組み込める形になると話しています。発表では、この方法で **P99 を約 70% 下げた**と述べており、競争ベース探索がかなり直接的な性能改善につながっていることを示しています。

---

### デモは「隠された最適化関数」を相手に、埋め込みモデルの良し悪しを比較する

後半のデモは、競争型最適化をさらに一般化した「auto optimize」の紹介です。ここでは正体を隠した最適化関数があり、エージェントはその関数を直接見ることができません。別マシンに隔離された評価器へ候補コードを投げ、返ってきたスコアだけを頼りに改善を続けます。

その際、エージェントには 1500 本の arXiv 論文からなる知識ベースが与えられます。ただし、長大な論文群をそのまま全部コンテキストへ入れることはできない。実際に参照できるのは全体の 0.1% 程度です。だからこそ、どの論文を拾ってこられるか、つまり検索と埋め込みの質が最適化性能を左右します。

デモでは ZeroEntropy、OpenAI、Cohere の埋め込みモデルを切り替えながら、どのモデルがより早く目的関数へ収束できるかを比較していました。特定の論文を引けた直後にスコアが改善したなら、その論文は有効だったと推定できる。さらに、その論文を他のエージェントにも注入して探索を加速できる。ここでも重要なのは、**探索の履歴自体が次の探索の材料になる**という点です。

---

### Mastra と Vercel AI SDK で、周辺部品も含めて素早く組める

Nicholas は実装の複雑さについても触れています。論文検索エージェント、キーワード変換、埋め込み検索、ハイブリッド検索、Python サンドボックス実行など、各コンポーネントは 50 行未満のコードで組めると説明していました。もちろん、実際の運用にはインデックス作成、チャンク化、OCR など周辺機能も必要ですが、それらを含めても Mastra と Vercel AI SDK によって試行錯誤しやすい構成を取れていると言います。

この話が示しているのは、Mastra を使う利点が単なる「エージェントを呼び出しやすい」ことではない、という点です。多数の構成候補を並列で試し、知識ベースや外部評価器をつなぎ、最終的に比較可能な結果へ落とし込むような**実験駆動のエージェント開発**に向いていることが見えてきます。

---

### Q&A では OCR と知識再利用の実務感が出ていた

質疑応答で印象的だったのは、かなり実務寄りの話が多かったことです。PDF パーサーについては、DeepSeek OCR のように Markdown へきれいに変換できる方法と、Google OCR のような従来型 OCR をコストに応じて使い分けていると答えています。そして、変換先としては Markdown が最善だと明言していました。埋め込みモデル、リランカー、下流 LLM のいずれにとっても扱いやすいからです。

また、「同じ総当たり探索を毎回やり直さなくていいのか」という質問に対しては、arXiv や Hugging Face、さらには社内の巨大な Markdown リポジトリを通じて、見つかった知見を再利用可能な知識として貯めていると説明しました。これは競争型探索の弱点を補う話でもあります。毎回ゼロから総当たりするのではなく、**過去の勝ち筋を知識ベースとして蓄積し、次の探索の初期条件を良くする**わけです。

最後に、公開予定のコードや ZeroEntropy の API にも触れ、聴衆が試せる導線を示して締めくくりました。短い登壇ですが、検索、推論、評価、知識再利用をひとつの考え方でまとめていて、かなり密度の高い内容だったと思います。
