# 製薬R&DでAIエージェントを使うなら、自己改善とContext Rot対策が先に来る

2026年のAI活用を見ていると、だいぶ空気が変わってきたと感じます。  
以前は「うまく答えてくれるLLM」をどう使うかが中心でしたが、今は「実際の仕事をどこまで任せられるか」に関心が移っています。

この変化は、製薬R&Dの現場とかなり相性がいいです。

なぜかというと、製薬の仕事には、単発の質問で終わらないものが多いからです。  
プロトコルを読む。SAPを解釈する。解析仕様に落とす。コードを書く。QCする。結果を文書に反映する。レビューコメントを直す。次の試験で同じ失敗を避ける。  
実際には、こういう長い流れの中で仕事が進みます。

ここで効いてくるのが、ただ賢そうに答えるAIではなく、文脈を保ちながら何ステップも仕事を進められるAIエージェントです。

ただし、そこで必ず出てくる問題が2つあります。

1つは、エージェントが使うたびに少しずつ賢くなるかどうか。  
もう1つは、長い仕事を続けるうちに前提や根拠が崩れていかないかどうかです。

前者が**自己改善型エージェント**で、後者が**Context Rot**です。

この2つは別々の話に見えて、実はかなりつながっています。

## まず、なぜ製薬R&Dはエージェント向きなのか

製薬R&Dの仕事は、AIにとって扱いやすい部分と、慎重であるべき部分がはっきり分かれています。

扱いやすいのは、形式が決まっていて、あとからレビューできる仕事です。  
たとえば、こんなものがあります。

* protocol
* SAP
* mock shell
* TLF仕様
* ADaM metadata
* 解析ログ
* QC結果
* CSR文案
* 規制当局への回答案

こうした仕事では、「良いか悪いか」を比較的はっきり判定できます。

* コードは動くか
* derivation は仕様に合っているか
* shell と出力は整合しているか
* endpoint 定義は文書間で一致しているか
* 同じ結果を再現できるか
* 根拠を説明できるか

この性質は大きいです。  
AIの改善は、改善したかどうかを測れないと安定しません。製薬R&Dには、その判定をしやすい仕事がかなりあります。

一方で、全部を自動化できるわけではありません。

estimand の妥当性、intercurrent event の扱い、欠測に対する前提、感度解析の優先順位、規制当局にどう説明するのが適切か、といった論点は今でも人間の判断が要ります。

なので、製薬でAIを使うときに狙うべきなのは、  
**人間の判断を消すことではなく、検証できる部分を機械化して、判断の密度を上げること**です。

## 自己改善型エージェントは「記憶するAI」ではない

自己改善型エージェントというと、つい「前の会話を覚えているAI」みたいなイメージになりがちです。  
でも実際は、もう少し実務寄りの話です。

ポイントは、前回の失敗やレビュー結果を、次回の仕事のやり方に反映できるかどうかです。

たとえば、こんな流れです。

このループが回ると、エージェントは単に出力を返すだけでなく、チームの癖や失敗パターンを学習した作業基盤になっていきます。

製薬の現場では、同じ種類のミスがかなり繰り返されます。

* population flag の定義漏れ
* baseline window のズレ
* visit windowing の解釈違い
* responder rule の文言と実装の不一致
* footnote と program logic のズレ
* 以前の試験の前例をそのまま持ち込んでしまう

こうしたものは、個人の記憶力に頼るより、エージェント側に「次はここを先にチェックする」という形で残したほうが強いです。

## 製薬R&Dで、どこに効くのか

エージェントの使いどころはたくさんありますが、特に効きやすいのは「毎回そこそこ頭を使うのに、毎回ゼロからやる必要はない仕事」です。

### Protocol / SAP を実行単位に分解する

protocol や SAP は、そのままでは実行しにくい文書です。  
読む人が、endpoint、population、欠測処理、主要比較、サブグループ、感度解析、TLF出力を解釈して、実装単位に落とし込む必要があります。

このときエージェントは、長い文書から例えば次のようなものを抜き出せます。

* estimand の候補
* population rule
* derivation が必要な変数
* TLFごとの入力要件
* 曖昧な記述
* レビューで確認すべき open issue

ここで大事なのは、AIが正解を断定することではありません。  
**曖昧な場所を曖昧なまま、きちんと浮かび上がらせること**です。

製薬で危ないのは、よく分からないところを、もっともらしく埋めてしまうことです。

### ADaM / TLF 実装とQCの支援

標準化された解析基盤があるチームほど、エージェントは使いやすくなります。

たとえば、

* 標準マクロ
* shell template
* ADaM derivation rule
* metadata
* QC checklist
* reviewer comment の蓄積

が揃っていれば、エージェントはそのルールに沿って下書きやレビュー支援をかなり安定して出せます。

できることも分かりやすいです。

* 仕様書から derivation candidate を出す
* TLFごとの program skeleton を作る
* shell とコードの不整合を点検する
* 差分レビューの観点を並べる
* reviewer comment を次回テンプレートに反映する

ここでの価値は、単純なスピードよりも、**毎回同じ品質基準で仕事を回せること**にあります。

### シミュレーションやモデリング

simulation、PK/PD、数理モデリングの仕事では、自己改善はさらに噛み合いやすいです。  
理由は単純で、改善したかどうかを測りやすいからです。

* 実行時間
* 収束率
* bias
* coverage
* power
* type I error
* 再現性
* 異常終了率

こうした指標があると、「今回の変更は本当に改善か」を判定しやすくなります。

この領域では、エージェントは派手な文章生成よりも、むしろ

* simulation plan の整理
* 実験条件の記録
* seed や設定値の追跡
* モデル診断のチェック項目整理
* 棄却したモデルの理由の保存

のような、地味だけれど重要な仕事で効いてきます。

## でも、本当に難しいのはContext Rot

自己改善より先に、Context Rot の話をしたほうがいいかもしれません。  
実際には、こちらで失敗するケースのほうが多いからです。

Context Rot は、長い会話でAIが疲れる、みたいな曖昧な話ではありません。  
もっと実務的な問題です。

時間が経つにつれて、

* 重要な前提が埋もれる
* 古い版と新しい版が混ざる
* 一時的な議論と正式決定が混ざる
* 出所のないメモが重要そうに見えてくる
* 役割の違う情報が同じメモリ空間に溜まる

といったことが起きて、推論がじわじわ腐っていく状態です。

製薬の現場は、これが起きやすい条件が揃っています。

* draft SAP と approved SAP が両方ある
* amendment で定義が変わる
* shell の版が複数ある
* 過去試験の前例が参照される
* exploratory と confirmatory の議論が並走する
* implementation note と scientific rationale が別管理されていない

これを全部まとめてAIに放り込むと、たいてい良くなりません。  
コンテキストが豊かになるのではなく、判断が濁ります。

## ありがちな失敗は「全部読ませれば分かるはず」

AI活用でよく見る失敗がこれです。

「関連ありそうなものを全部入れれば、あとはモデルがうまく整理してくれるだろう」

実際には逆で、全部入れるほど、

* 正式版とドラフトの区別が弱くなる
* 重要なルールがノイズに埋もれる
* 途中の議論が最終方針のように扱われる
* 他試験の前例が過剰に効く

といった問題が起きやすくなります。

つまり、必要なのは長大なコンテキストではなく、**設計されたコンテキスト**です。

## Context Rot を防ぐには、情報を「設計」する

やるべきことはそこまで複雑ではありません。  
ただ、かなり地味です。

### 1. 全文ではなく、まず索引を渡す

いきなり全部の文書を読ませるのではなく、まずは文書インデックスだけを見せます。

* Protocol v3.0 approved
* SAP v2.1 approved
* TLF shell pack final
* ADaM spec draft
* reviewer decision log
* standard macro library

この状態で、エージェントに「今のタスクに必要な文書だけ開く」ようにさせます。

これだけでも、かなり安定します。

### 2. メモリを役割ごとに分ける

同じ「記憶」でも、置き場所を分けたほうがいいものがあります。

* scientific intent memory
* implementation memory
* quality / validation memory
* submission / traceability memory

たとえば、主要評価項目の解釈と、前回このマクロで warning が出たという話は、どちらも重要ですが、同じ層に置くべきではありません。

### 3. 重要な判断には provenance を付ける

製薬では、「何を決めたか」だけでは足りません。  
「どの版の、どの文書の、どの箇所を根拠に決めたか」が必要です。

なので、エージェントが保持する重要情報には少なくとも次を付けたほうがいいです。

* source document
* version
* section / page
* effective date
* owner / approver
* confidence
* superseded flag

これがない記憶は、あとでだいたい問題になります。

### 4. 文書要約ではなく、意思決定単位に変換する

「SAPを3ページに要約しました」だけだと、実務では弱いです。  
必要なのは、文書の要約というより、判断単位への変換です。

* endpoint definition
* analysis population rule
* censoring rule
* missing data handling
* subgroup rule
* sensitivity analysis trigger
* table-specific note

この粒度まで落ちると、エージェントも人間もかなり扱いやすくなります。

## じゃあ、どこから始めるのが現実的か

いきなり fully autonomous agent を目指す必要はありません。  
むしろ、段階を分けたほうが失敗が少ないです。

### Phase 1: Read-only 支援

* 文書検索
* 論点抽出
* 定義差分の検出
* reviewer question の生成

### Phase 2: Draft generation

* 仕様書ドラフト
* code skeleton
* QC checklist
* CSR文案の下書き

### Phase 3: Controlled execution

* sandbox 内でのコード実行
* simulation batch 実行
* 標準化された QC run
* 承認付きの再実行

### Phase 4: Self-improvement loop

* reviewer feedback の蓄積
* 頻出エラーへの自動チェック追加
* study type 別 playbook の改良
* 品質指標の継続監視

この順番のいいところは、組織がAIの失敗モードを理解しながら適用範囲を広げられることです。

## 最後に

製薬R&DでAIエージェントを使うとき、いちばん大事なのは「どれだけ賢そうに見えるか」ではありません。

大事なのは、

* 同じ失敗を繰り返さないか
* 根拠を追えるか
* 長い仕事の途中で文脈が腐らないか
* 人間が最後に責任を持てる形になっているか

です。

自己改善型エージェントは、うまく使えば、チームの学習を蓄積する装置になります。  
一方で、Context Rot を放置すると、長い仕事ほど不安定になります。

だからこの2つはセットで考えたほうがいいです。

製薬でAIを入れるかどうか、という段階はもう過ぎつつあります。  
これから問われるのは、**どの業務を改善ループに乗せるのか、どの知識を腐らない形で持つのか、どこに人間の判断を残すのか**です。

この設計がうまい組織ほど、速くなるだけでなく、品質も説明責任も強くなります。
