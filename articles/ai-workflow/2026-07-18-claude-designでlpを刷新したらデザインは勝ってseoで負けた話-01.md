## はじめに

個人開発で複式簿記の家計簿SaaS「[kurofukubo（黒福簿）](https://kurofukubo.com/?utm_source=zenn&utm_medium=article)」を運営しています。

<https://kurofukubo.com/?utm_source=zenn&utm_medium=article>

このサービスのLPは、最初からClaude（AI）に作らせたものでした。それなりに整っていて、公開当時は満足していました。

ところがある日、競合サービスのLPを見て凍りつきました。**構図がそっくり**なのです。白いスティッキーヘッダー、中央寄せのヒーロー、黒見出しの中に緑のキーワード強調、緑の丸ボタンと白アウトラインの副ボタン、淡いグレー背景。相手が真似したわけではありません。**「AIに何も指定せずLPを作らせると出てくる定番構図」を、お互いが使っていただけ**でした。AIでLPを量産できる時代は、裏を返せば**何も指定しないと量産品の顔になる**時代です。うちのLPは、その量産品の1つでした。

そこで今度は **Claude Design（Claudeのデザインエージェント）に、制約を明示してLPを刷新させました**。同じAIでも、渡すものを変えれば結果は変わるのか——という実験でもあります。結論だけ先に書くと:

* **デザインとCVR設計は明確に勝ち**。実画面スクショ主役の構成で、競合との差別化に成功
* **でもAIはtitleタグから主戦場の検索キーワードを消した**。気づかずデプロイしていたらSEOが後退していた
* AIの成果物は「そのまま使う」と事故る。**機械が読む部分（title/JSON-LD/座標指定）は人間がレビューする**のが正しい分担だった

この記事は、プロンプト設計 → 出力（自己解凍HTML）の中身 → 静的サイトへの移植 → レビューで見つけた事故ポイント → LPの役割論、までの実録です。

これが刷新前のLP。AIに「家計簿SaaSのLPを作って」と頼むと、おそらく誰がやってもこの顔になります。

![刷新前のLP。AIが無指定で出す定番のSaaSテンプレ構図](https://static.zenn.studio/user-upload/e2f44195d4ef-20260718.png)  
*刷新前：黒見出し＋緑キーワード強調＋中央寄せヒーロー＋CSSで描いた偽ダッシュボード。これも Claude 製*

## 1. プロンプト設計 — 「変えないもの」を先に書く

デザインを任せるとき、一番効いたのは**やってほしいことより、壊してほしくないものを先に列挙した**ことです。実際に渡したプロンプトの骨子:

```
## 絶対に変えないもの（制約）
- コピー・文言・セクション構成の意味内容
- SEO要素：<title>/meta/canonical/OGP/JSON-LD を1文字も壊さない
- AdSense審査コードと自前アナリティクス
- 単一HTMLファイル構成。外部依存はGoogle Fontsのみ可
- CTAの数と行き先

## やってはいけないこと
- 競合LPの構図・配色・要素配置を参照・模倣しない
- 緑グラデーションの多用、パステル淡色背景＋白カードの定番SaaS構図
- 画像素材の外部URL参照

## 検証基準
- 競合と並べたとき、配色・構図・ボタン形状の3点すべてが異なる
```

「検証基準」を書いたのもポイントで、**完成の定義をプロンプト側が持っている**と、出力のブレが減ります。

…と書きつつ、後述の通り「SEO要素を壊さない」は守られませんでした。制約に書いても、**生成物を機械的に検証しない限り守られたかは分からない**。これがこの記事の教訓その1です。

## 2. 出てきたデザイン — 11セクション構成

Claude Designが出してきた構成は、テンプレSaaS顔から完全に離れていました。

1. ファーストビュー：H1「使ったお金だけでなく、お金の流れまで見える家計簿。」＋**実ダッシュボードのスクショ**（ブラウザフレーム）＋スマホ画面
2. 一般家計簿との比較：「コンビニで500円」の流れ図（左右対比カード）
3. ダッシュボード注釈：実画面に①②③のピン（今月の収支／純資産／資産構成）
4. 入力3ステップ：クイック入力「食費 1200 現金」
5. 口座管理：科目チップ＋カード締め日・引落日の自動仕訳
6. 複式簿記のメリット3点＋仕訳帳の実画面
7. 利用シーン4カード
8. はじめ方3ステップ
9. 安心・信頼：**連携なし＝認証情報を預からない設計**、E2E暗号化
10. FAQ 6件
11. 最終CTA（深緑背景）

![刷新後のLPファーストビュー](https://static.zenn.studio/user-upload/5662a10bec97-20260718.png)  
*刷新後：実ダッシュボードのスクショが主役。アイボリー基調＋Zen Kaku Gothic Newで別物の顔に*

デザイントークンも一貫していました。深緑 `#2f6e55`、白とアイボリー `#f7f6f1` の交互背景、Zen Kaku Gothic New（本文）＋ IBM Plex Mono（数字）。

特に感心したのは方針の2つ:

* **架空UIを描かない**。実スクリーンショット7枚を主役に据えた（提供していない画面はストライプのプレースホルダーで「ここに画像」と明示してきた。勝手に捏造しない）
* **未実装機能を「ない」と明記して反転訴求**。銀行自動連携が無いことを隠さず「認証情報を預からない安全設計」として売りにした

コピーの誠実さはAdSense等の審査観点でもプラスに働きます。

## 3. 出力は「自己解凍HTML」だった

さて、成果物を受け取って驚いたのがファイル形式です。**10.1MBの単一HTML**で、開くと「Unpacking...」と出てからページが表示される。中身はこうなっていました。

```
<script type="__bundler/manifest">
  { "9ec40fe0-...": { "mime": "font/woff2", "data": "<base64>" },
    ...505エントリ... }
</script>
<script type="__bundler/template">
  "<!DOCTYPE html>..."  ← 本体HTMLがJSON文字列で埋まっている
</script>
<script>
  // 実行時にmanifestをBlob URL化してtemplate内のUUID参照を差し替える
</script>
```

つまり**フォントサブセット499個＋画像6枚をbase64で内蔵し、実行時にJSで展開する**自己完結バンドル。オフラインで1ファイル配布するには合理的ですが、静的サイトとしてはこのまま使えません（SEO的にもJS展開前は空、Blob URLはクロール不能）。

### 静的HTMLへの移植

やることは3つです。Node.jsで20行程度:

```
const html = fs.readFileSync('黒福簿LP（スタンドアロン）.html', 'utf8');
const manifest = JSON.parse(html.match(/<script type="__bundler\/manifest">([\s\S]*?)<\/script>/)[1]);
const template = JSON.parse(html.match(/<script type="__bundler\/template">([\s\S]*?)<\/script>/)[1]);

// 1. 画像6枚をデコードしてファイル化（→ その後WebP変換して16〜56KBに）
for (const [uuid, entry] of Object.entries(manifest)) {
  if (entry.mime === 'image/png') {
    fs.writeFileSync(`img/${names[uuid]}.png`, Buffer.from(entry.data, 'base64'));
  }
}

// 2. template内のUUID参照を実パスに置換
let t = template;
for (const [uuid, path] of Object.entries(imgMap)) t = t.split(uuid).join(path);

// 3. @font-face 505個（41万文字）を削除して Google Fonts の <link> 1本に置換
```

フォントは `Zen Kaku Gothic New` と `IBM Plex Mono` のサブセット群だったので、Google Fontsの`<link>`に戻すだけで**HTMLが512KB→35KBに**なりました。画像はcanvas経由でWebP化（quality 0.85）して6枚合計190KB。

### SEOインフラの再注入

移植したHTMLには、既存LPが持っていたSEO資産がありませんでした（これはデザインツールの責務外なので当然です）。以下を手で注入:

* AdSense審査コード
* OGP / Twitter Card / favicon群
* JSON-LD（SoftwareApplication + FAQPage — FAQPageは新デザインの可視FAQ 6問と同期させる）
* 自前アナリティクスのビーコン

**JSON-LDのFAQと可視FAQの同期**は忘れやすいポイントです。構造化データと画面の内容がズレているとGoogleのガイドライン違反になります。

## 4. レビューで見つけた事故ポイント

デザインは良い。でも「そのままデプロイ」していたら4つの事故が起きていました。

### 事故1（最重要）: titleから検索キーワードが消えていた

旧LP: `kurofukubo — 純資産まで見える複式簿記の家計簿`  
新LP: `黒福簿（kurofukubo）— お金の流れまで見える家計簿`

コピーとしては新しい方が柔らかくて良い。しかし、このサイトの検索流入は「家計簿 複式簿記」「純資産 いくらあれば」系のクエリで育ち始めたところでした。**その主戦場キーワード「複式簿記」「純資産」がtitleとh1から両方消えた**。

Search Consoleで表示回数が立ち上がり始めた時期にこれをやると、育ちかけた順位を自分で折ることになります。デプロイ後に相互レビューして気づき、titleだけ「黒福簿（kurofukubo）— 複式簿記で純資産まで見える家計簿」に修正しました。**ヒーローの見た目のコピーと、titleタグは別物として管理していい**。人間が読む文とクローラが読む文の分離です。

### 事故2: 注釈ピンがピクセル絶対配置だった

実画面スクショに①②③のピンを重ねる演出、デスクトップでは完璧でした。しかし実装は:

```
<div class="pin" style="top:81px;left:632px">1</div>
```

画像は可変幅なのにピンはpx固定。**タブレット以下で全ピンがズレ、モバイルではピンが無関係な円グラフの上に浮いていました**。

![モバイルでピンが破綻している様子](https://static.zenn.studio/user-upload/071da2d1a752-20260718.png)

*モバイル実機幅（390px）。ピン②が何も指していない場所に浮いている*

修正は2段階:

```
/* %配置＋中心基準に変換 */
.pin { transform: translate(-50%,-50%); }
/* style="top:8.4%;left:66%" のように%指定へ */

/* モバイルではピン自体を消す（下の番号付きリストが同内容を持つ） */
@media(max-width:640px){ .pin{ display:none } }
```

![修正後、タブレット幅でもピンが正しい位置に](https://static.zenn.studio/user-upload/ce68ec134227-20260718.png)

*%＋translate化後のタブレット幅（768px）。①収支・②総資産・③資産構成に正しく追従*

これはデザイン側の反省会でも挙がった点で、「最初から画像に対する%＋アスペクト比固定コンテナにすべきだった。**画像を差し替える前提のLPでは、注釈は画像の外（番号付きリスト）に置く方が保守的**」という結論になりました。実際、ピンの下に同じ内容の番号付きノートを置く二重化構成だったので、モバイルでピンを消しても情報は失われませんでした。

### 事故3: プレースホルダーの残存

「はじめ方」セクションのSTEP1に、ストライプ模様の「ここにゲスト開始直後の画面を配置」というダミーがそのまま残っていました。

![STEP1にプレースホルダーが残ったままの状態](https://static.zenn.studio/user-upload/9338db6d7ef6-20260718.png)  
*STEP1だけストライプのプレースホルダー。「画像を捏造しない」誠実さの裏返しだが、残したまま公開すると事故*  
これは前述の「架空UIを描かない」方針の裏返しで、**提供していない画像は捏造せずプレースホルダーにする**という誠実な挙動なのですが、そのままデプロイすると事故です。全ページのテキストを`grep`して「ここに」「placeholder」「TODO」を検出するチェックをデプロイ前に挟むべきでした。

### 事故4: 例文とスクショの不整合

複式簿記の説明セクションに「コンビニ ¥720 → 食費 +¥720 ＋ 現金 −¥720」というフロー図があり、その直下に「1件の取引が2つの面から記録される」として仕訳帳のスクショが置かれていました。ところが**スクショの仕訳帳にコンビニ¥720の取引は存在しない**（実データは「外食・レジャー ¥6,800」）。

![フロー図の例とスクショの中身が食い違っている](https://static.zenn.studio/user-upload/14554bcdeb07-20260718.png)  
*「コンビニ ¥720」のフロー図の直下に置かれた仕訳帳スクショ。中にその取引はない*

これはユーザー（＝自分）がローカルプレビューを目視して気づいたものです。例文をスクショに実在する取引に合わせて修正しました。**実画面スクショを使う戦略は証拠力が強い分、周辺テキストとの整合性検証が必須になる**という副作用があります。

### その他、デザイン側の反省より

* **画像差し替えの往復が多かった**。スクショの命名規則（バージョン管理）を最初に決めていれば、`width:380%;left:-58%` のようなマジックナンバーの切り出し再調整が減った
* **縦長画像への対応が後手**。途中でダッシュボードのスクショが縦長になり、`object-fit:cover` で切り出す応急処置になった。**掲載画像の推奨アスペクト比を先に指示書として出すべき**だった

## 5. で、LPを刷新すると集客は増えるのか

ここが一番書きたかった部分です。忖度なしの結論: **LP刷新は短期の集客数をほぼ変えません**。

うちのサイトの実数（CloudFrontログを自前でbot判定して集計）:

* 人間の訪問: **7〜8人/日**
* ゲスト開始（アプリを触った人）: 14日で6件 ≈ **0.4件/日**

仮に新LPでCVRが2倍になっても **+0.4件/日**。統計的に有意な差を検出するのに1〜2か月かかる規模です。

なぜか。**LPは流入を生まない**からです。流入を決めるのは検索順位とSNSで、LPが動かせるのは「来た後の行動」だけ。だから評価軸を分離する必要があります:

| 軸 | 旧LP | 新LP |
| --- | --- | --- |
| 独自性（競合との区別） | ✕ テンプレ構図 | ◎ 実画面主役 |
| 証拠力（実物が見えるか） | ✕ CSSで描いた偽UI | ◎ 実スクショ7枚 |
| CVR設計 | △ | ◎ 比較図解→証拠→信頼→FAQ→CTA |
| SEO（title/h1のキーワード） | ◎ | ✕→修正で◎ |

新LPの価値は「今日の訪問者数」ではなく、(1) 来た人が離脱しない確率、(2) 競合と間違われない独自性、(3) 実体のあるサービスとしての信頼シグナル——**全部、効くのに時間がかかる資産**です。個人開発でLPに凝りたくなったとき、「これは流入施策ではなく転換施策」と自覚してからやると、期待値を間違えません。

## 6. まとめ — AIデザインの分担表

今回の経験をそのまま分担表にします。

**Claude Designに任せてよかったもの**

* 構図・配色・タイポグラフィの一貫したデザインシステム
* セクション構成とコピーの流れ（比較図解で簿記用語を使わず複式簿記を説明したのは自分では書けなかった）
* 「未実装機能を書かない」「架空UIを描かない」という誠実さ（プロンプトの制約が効いた）

**人間（実装側）が守るべきだったもの**

* **titleタグと検索キーワードの整合**（見た目のコピーと分離して管理する）
* 構造化データ・OGP・広告コードなどの機械可読インフラ
* レスポンシブでの座標指定の破綻（px絶対配置の検出）
* プレースホルダー残存・例文とスクショの不整合チェック

AIに「SEOを壊すな」と書いても、壊れていないことの検証はしてくれません。**制約はプロンプトに書き、検証はパイプラインでやる**。デザインの好みではなく、この分担を間違えないことが、AIデザイン活用の実務的な核心だと思います。

最後にもう一度。旧LPも新LPも、作ったのは同じClaudeです。違ったのは**渡した制約だけ**。「AIに作らせると量産品になる」のではなく、「何も指定しないと量産品になる」——差別化の責任は、プロンプトを書く側にあります。
