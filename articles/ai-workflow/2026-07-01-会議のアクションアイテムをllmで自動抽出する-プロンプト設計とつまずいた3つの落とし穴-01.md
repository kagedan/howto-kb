個人で議事録まわりのツールをつくっていて、いちばん「効く」と実感した機能が、会議の文字起こしから**アクションアイテム（担当者・期限つきのタスク）を自動抽出する**処理でした。

要約そのものは、正直どのLLMでもそれなりに出ます。でも「誰が・何を・いつまでに」を、抜け漏れなく・幻覚なく・実務で使える粒度で取り出すのは、思ったより設計が要ります。この記事では、実装するなかでつまずいた点と、それをどう解いたかを、プロンプト設計を中心に共有します。特定の製品の宣伝ではなく、同じものを実装する人が踏まないで済むように、という趣旨です。

## そもそも「アクションアイテム」とは何を指すか

抽出処理を書く前に、定義をコードに落とせるくらい具体化しておく必要がありました。私は次の3条件を満たすものだけをアクションアイテムとして扱う、と決めました。

1. **担当者が特定できる**（「誰か」ではダメ。個人 or 明確な役割）
2. **実行内容が動詞で書ける**（「〜について」ではなく「〜を作成する」）
3. **期限、または期限の手がかりがある**（「次回まで」も可）

この3条件を満たさない発言は、どれだけ重要そうでも「議論」や「決定事項」に分類して、アクションアイテムからは外す。この線引きをプロンプトに明記しただけで、出力の質がかなり安定しました。

## 落とし穴1：全部を「タスク」にしてしまう

最初のナイーブなプロンプト（「この議事録からアクションアイテムを抽出して」）だと、LLMは会議で出た話題を片っ端からタスク化してきます。「A社の件は難しそう」みたいな感想まで `- [ ] A社の件を検討する` になる。

効いた対策は、**除外条件を先に書く**ことでした。「担当者が明示されていない発言、単なる感想・情報共有、すでに完了した報告は、アクションアイテムに含めない」と明示する。LLMは「やること」を足すのは得意でも、「やらないこと」の判断は指示しないと弱い、という感触です。

## 落とし穴2：担当者名の幻覚（hallucination）

文字起こしには話者ラベル（Speaker A など）しか無い場合が多く、そこでLLMに担当者を推測させると、会議に出ていない人名を創作することがありました。これは実務では致命的です。

対策は2つ。ひとつは、\*\*「文字起こしに登場しない人物名を出力してはならない。担当が不明なら担当者フィールドは『未定』とする」**と制約すること。もうひとつは、担当者を自由記述にせず、**「発言者ラベル、または本文中に明示された氏名のみ許可」\*\*とスキーマ側で縛ること。曖昧なときに空欄を許すほうが、間違った断定より圧倒的に使えます。

## 落とし穴3：出力が毎回ブレて後処理できない

自由テキストで返させると、箇条書きになったり表になったりして、カレンダー連携やDB保存の後処理が安定しません。ここは構造化出力（JSON / tool use）で固定するのが結局いちばん早かったです。

Claude の tool use を使う場合、こんなスキーマに寄せました。

```
{
  "action_items": [
    {
      "assignee": "string | 未定",
      "task": "string（動詞で終わる一文）",
      "due": "string | null",
      "source_quote": "string（根拠となった発言の引用）"
    }
  ]
}
```

ポイントは `source_quote`（根拠の引用）を必須にしたことです。「なぜそれを抽出したか」を本文から引かせると、幻覚がさらに減りました。人間がレビューするときも、引用があると確認が一瞬で終わります。抽出とは要するに「本文のどこを指しているか」なので、指し先を必ず持たせる、という発想です。

## プロンプトの骨格（抜粋）

```
あなたは会議の文字起こしからアクションアイテムを抽出するアシスタントです。

# アクションアイテムの定義（すべて満たすもののみ）
- 担当者が特定できる
- 実行内容が動詞で書ける
- 期限、または期限の手がかりがある

# 含めないもの
- 担当者が不明な発言 / 単なる感想・情報共有 / 完了済みの報告

# 制約
- 文字起こしに登場しない人物名を創作しない。不明なら assignee は「未定」。
- 各項目に source_quote（根拠となった本文の引用）を必ず付ける。
```

短いですが、「定義」「除外」「制約」の3ブロックに分けるだけで、実務に耐える出力になりました。

## まとめ

* アクションアイテムは「担当・動詞・期限」の3条件でコード的に定義する
* 「含めないもの」を先に書くと過剰抽出が止まる
* 人名の幻覚は「登場しない名前は出さない・不明は未定」で防ぐ
* 構造化出力＋根拠引用（source\_quote）でレビューと後処理が安定する

アクションアイテムそのものの書き方・管理方法（AI以前の運用面）については、別途[アクションアイテムとは何か・書き方の型](https://kaigi-ai.com/blog/action-item-toha)に整理してあるので、運用設計の参考にどうぞ。実装の議論のたたき台になれば幸いです。
