AWS MCPを使い始めるとき、最初に目につくのは `aws-api-mcp-server` だ。AWS CLIコマンドを網羅し、S3もLambdaもCloudWatchも、一つで何でも操作できる。便利に見える。しかしそれは、AWSがMCP以前から「アンチパターン」と定めてきた構成に近い。

本稿では、AWSの原則に立ち返りながら、AWS MCPのサーバー選択と権限設計をどう考えるべきかを整理する。

---

## AWSの原則：長期トークンと広権限はアンチパターン

AWSのIAMベストプラクティスは一貫している。

* **アクセスキーの長期運用は避ける**。IAMユーザーの静的アクセスキーは、漏洩しても気づかなければ無期限に悪用される。AWS STSによる一時的な認証情報（有効期限付きトークン）を使うことが推奨されている
* **最小権限の原則**。必要な操作にだけ権限を付与する。「とりあえず広い権限を渡しておく」は、侵害時の被害範囲を不必要に拡大させる

これはMCPとは無関係に、AWSがずっと言ってきたことだ。

---

## AWS MCPが50種以上に分割されている理由

AWSが公開しているMCPサーバーは2026年6月時点で50種以上ある。CloudWatch専用、ECS専用、IAM専用、Serverless専用——細かく分割されている。

`aws-api-mcp-server` のREADMEには、率直にリスクが列挙されている。

* ファイルシステムへの無制限アクセス
* 確認プロンプトなし
* サンドボックスなし
* すべてのAWS CLIコマンドへのアクセス

その一方で、用途別サーバーはデフォルトが安全側に設計されている。

| サーバー | デフォルト | 書き込み有効化 |
| --- | --- | --- |
| aws-serverless-mcp-server | read-only | `--allow-write` で明示的に有効化 |
| ecs-mcp-server | `ALLOW_WRITE=false` | 環境変数で明示的に有効化 |
| cloudwatch-mcp-server | ログ取得専用 | 書き込み操作を持たない |

つまり、AWSが用途別にサーバーを分割しているのは、**デフォルトを安全側に倒すため**だ。「何でもできる」サーバーを一つ入れるより、「必要なことだけできる」サーバーを組み合わせる——それがAWSの意図だと読める。

## 自分のユースケースで選んだ3種

筆者はLambdaとS3を使ったデータ収集パイプラインを運用している。APIやRSSフィードを毎日収集し、S3に蓄積する構成だ。

Claude DesktopからAWSを操作したいユースケースは3つに絞られた。

**cloudwatch-mcp-server**：Lambdaのログを確認する。収集パイプラインが失敗したとき、CloudWatch Logs InsightsをAIに直接クエリさせると原因特定が速い。

**aws-serverless-mcp-server**：Lambdaのメトリクスを確認する。最初にこれだけ入れたが、ログが見えないことに気づいて `cloudwatch-mcp-server` を追加した。

**aws-api-mcp-server（READ\_OPERATIONS\_ONLY）**：S3に蓄積した収集データをAIに参照させる。書き込みは不要で、read権限だけで十分だと判断した。

aws-apiを使っているのは、CloudWatchやServerlessの専用サーバーではカバーできないS3操作が必要だったからだ。ただし `READ_OPERATIONS_ONLY=true` を設定し、IAM Roleも読み取り専用に制限した。

```
{
  "aws-api": {
    "command": "<path-to-venv>\\Scripts\\awslabs.aws-api-mcp-server.exe",
    "env_keys": {
      "READ_OPERATIONS_ONLY": "mcp-launcher/aws-api/READ_OPERATIONS_ONLY"
    }
  }
}
```

aws-apiのREADMEには注記がある。「`READ_OPERATIONS_ONLY` はCLIコマンドの分類に基づく制限であり、ファイルシステムへの書き込みは防がない。IAMが一次防衛線だ」と。権限を絞り込んだIAM Roleを用意した上で、環境変数はあくまで補助として設定した。

## 実際に使ってみてわかったこと：S3ファイルの読み取り

aws-api経由でS3のファイルを取得しようとして、一つ詰まった点がある。

`s3api get-object` はファイルをMCPサーバーのワークディレクトリに書き出す設計になっている。デフォルトのワークディレクトリは `%TEMP%\aws-api-mcp\workdir`（Windowsの場合）だ。ここに書き出されたファイルは、filesystemサーバーからは読めない——filesystemサーバーの許可ディレクトリと異なるからだ。

解決策は `AWS_API_MCP_WORKING_DIR` をfilesystemサーバーの許可ディレクトリ配下に設定することだ。`claude_desktop_config.json` の `env` フィールドで静的な値として渡せる。

```
"aws-api": {
  "command": "<path-to-mcp-launcher.exe>",
  "args": ["aws-api"],
  "env": {
    "AWS_REGION": "ap-northeast-1",
    "AWS_API_MCP_WORKING_DIR": "<filesystemサーバーの許可ディレクトリ>\\aws-workdir"
  }
}
```

こうすることで、`get-object` で取得したファイルをfilesystemサーバー経由で読めるようになる。

なお、`launcher.json` の `env_keys` はすべてキーストアのキーとして解釈される。固定値の環境変数は `claude_desktop_config.json` 側の `env` で渡すのが正しい。

## MCPが追いついていないギャップ：長期トークン問題

AWSの原則に照らすと、もう一つのギャップがある。**トークンのライフサイクル管理**だ。

Claude DesktopはMCPサーバーを起動時に一度だけ生成し、常駐させる。MCPサーバーのプロセスには、起動時に注入されたトークンがセッション中ずっと保持される。

ここで長期の静的アクセスキーを渡していると、Claude Desktopを再起動するまでそのキーが生き続ける。AWSが「避けよ」と言っている長期トークンを、AIが常駐プロセスとして抱え込む形になる。

この状態でプロンプトインジェクションが成立した場合——AIが読み込むS3のデータや外部コンテンツに悪意ある指示が埋め込まれていた場合——トークンが漏洩したときのダメージは時間的に無制限になる。

AWSの原則（短命トークン）とMCPの現状（常駐プロセスへの静的トークン注入）の間にギャップがある。このギャップを埋めるため、筆者はmcp-launcherにAWS STSによる自動ローテーションを実装した。詳細は以下の過去記事に譲る。

## 選ぶときの考え方

AWS MCPのサーバー選択は、AWSのIAM設計と同じ問いに帰着する。

* **何をAIに操作させる必要があるか**を先に決める
* その操作に特化したサーバーが存在するなら、aws-apiより専用サーバーを選ぶ
* aws-apiを使う場合は、IAM Roleで権限を最小化し、`READ_OPERATIONS_ONLY=true` を設定する
* トークンには有効期限を設ける。長期の静的アクセスキーをMCPプロセスに渡さない

「便利だから」で選ぶのではなく、「これだけ必要だ」から選ぶ。MCPの選択も、IAMの設計も、出発点は同じだ。

---

## まとめ

* AWSはMCP以前から「長期トークン＋広権限」をアンチパターンとしている
* AWS MCPの用途別分割は、デフォルトを安全側に倒すための設計判断だ
* aws-apiは「何でもできる」サーバーであり、使う場合は権限の絞り込みが必須
* MCPが常駐プロセスにトークンを保持する構造は、長期トークンのリスクをそのまま引き継ぐ。プロンプトインジェクションと組み合わさると被害が甚大になる
* AWSの原則をMCPレイヤーでも守ることが、最低限の自衛だ
