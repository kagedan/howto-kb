## TL;DR

* **AIエージェントに「目」と「手」を与える**: ExcelやPowerPointなどのOfficeファイルやPDFをOn-the-flyでMarkdown変換・画像化し、エージェントが自律的に読み取れるようにするMCPサーバを開発。
* **「Agentic RAG」をローカルで実現**: 複雑なベクトルDBは不要。エージェントが自ら検索ツールを駆使し、数千件のドキュメントから必要な情報を秒速で深掘り。
* **圧倒的な解像度向上**: PPTの構成図をPNG化してマルチモーダルで理解するなど、コピペでは不可能な情報の活用を可能に。
* **結論、力技は正義**: 精度よりも「手間ゼロ」を優先し、必要な時に動的にデータを整備するアプローチで、日々のデスクワークを劇的に効率化。

## はじめに（経緯）

昨年夏ごろからAIエージェントのCLIツール（普段はGemini-CLIを使用しています）を使い始め、今では本当に日常的に活用するようになりました。開発そのものというよりは、仕様調査やインシデント分析、資料作成やタスク管理などのデスクワーク全般に使うようになっており、デスクトップには常時Gemini-CLIのコンソールが開いているという状況になっています。

通常のデスクワークで避けて通れないのが、大量のOffice系ドキュメントです。PowerPointやExcelの設計書、社内外から送られてきた各種PDFなどなど。これらは直接LLMが解釈できないため、そのままでは活用できないのが悩みの種でした。（正確には、マルチモーダルなLLMであればPDFファイルなどは読むことができますが、ここでは割愛します）

一方で、「生成AIにデータを食わせて正しい結果を得るには、それに適した形式でデータを整備しなければ・・・」と言われることもありますが、そんな作業を人間がゼロからやっていてはどう考えてもコスパが合いません。

「精度には一旦目をつむるとして、手間を最小限にして各種ドキュメントをLLMに食わせたい・・・」  
「Embeddingとかベクトル検索とか言い始めると収拾がつかなくなるのでキーワード検索でどうにか・・・」

というわけで、（かなり力技ではありますが）LLMが読む必要があるタイミングで、各種のOfficeファイルを **On-the-flyでMarkdownに変換する MCPツール（MCPサーバ）「DocTools」** を作ってみました。さらに、全文検索の仕組みを組み込むことで、Agentic RAGのように自律的なドキュメント検索・利用も可能にしてみました。

今回は、この「DocTools」の紹介をしてみます。

なお、「手間を最小限に」と書きましたが、DocToolsを作る手間はそれなりにかかってます。まぁそういうことです。楽しければいいのです。

### コンセプト

DocToolsのコンセプトはシンプルです。 **「AIエージェントに『目』と『手』を与えること」** を目的としています。  
AIエージェントがローカルにあるOfficeファイルを「あ、これはExcelなのでCSVにして中身を確認しますね」とか「パワポのスライドからテキストを抽出して要約します」といった動作を、人間がいちいち指示しなくても暗黙のうちに自動的に行えるようにするためのツール群です。

処理の流れをシーケンス図にすると、以下のようになります。エージェントが「読む必要がある」と判断した瞬間に、裏側で動的に変換が行われます。

### 前提条件

* **Python**: Pythonおよび依存ライブラリ（Pandas, python-pptx, PyMuPDFなど）が利用できること。
* **Office**: Microsoft Officeがインストールされており、COM (Component Object Model) が利用できること。（PowerPointやExcelのデータの画像化する場合）

### 制約条件

* **OS**: 一部の機能はWindows上でしか動作しません。多分。恐らく。

### メリットとデメリット

* **メリット**:
  + エージェントが自律的にOfficeファイルを「読み、解析し、処理」できるようになります。
  + Markdownに変換することで、LLMが構造を理解しやすくなります。
  + 全文検索（Whooshインデックス）により、数千件のドキュメントから秒速で情報を探し出すことができます。
* **デメリット**:
  + レイアウトが複雑すぎるPDFやExcelは、Markdown化の過程で情報が欠落する場合があります。
  + 画像の抽出や変換には、それなりの計算リソースと時間が必要です。
  + 変換後のMarkdownファイルがあちこちに残存します。

DocToolsは、Pythonの豊富なエコシステムを活用して、複雑なドキュメント処理を実現しています。MCP（Model Context Protocol）を通じて、エージェントが利用可能な具体的な「ツール（メソッド）」を多数提供しており、エージェントは状況に応じてこれらを自由に選択して実行します。

READMEは以下。

### 主要な利用ライブラリ

### On-the-flyでのMarkdown化

パワポやPDFを、その場でLLMが理解しやすいMarkdown形式に変換します。単にテキストを抽出するだけでなく、表構造なども維持したまま変換することで、エージェントの正確な理解を助けます。

* **`pdf_extract_markdown`**: PDFからテキストを抽出し、Markdownとして保存します。
* **`pptx_extract_markdown`**: スライドの内容や表を抽出し、Markdown化します。
* **`xlsx_extract_markdown`**: 特定のシートをMarkdownテーブル形式で抽出し、ファイルとして保存します。
* **`html_extract_markdown`**: HTMLを構造解析し、クリーンなMarkdownに変換します。

### スプレッドシートの操作

Excel（XLSX）やCSVの内容を、LLMが扱いやすい形式で効率的に読み取ります。巨大なファイルでも、コンテキストを節約しながら必要な部分だけをピンポイントで抽出できます。

* **`xlsx_list_sheets`**: Excelファイル内の全シート名をリストアップします。
* **`csv_read_cells`**: 指定した行・列の範囲（矩形領域）を読み取ります。
* **`csv_search_values`**: CSV内を文字列検索し、ヒットしたセルの位置と値を返却します。

### 図表の視覚的理解（マルチモーダル支援）

テキストだけでは説明が難しい図解、グラフ、フローチャートなどを画像として切り出します。マルチモーダルなエージェントはこの画像を見て内容を解釈できます。

* **`pdf_extract_images`**: PDFの指定ページをPNG/JPG画像として保存します。
* **`pptx_extract_images`**: スライド全体を画像化します。システム構成図などの理解に威力を発揮します。
* **`xlsx_extract_images`**: Excelのシートを画像化します。

### 高度なテキスト・ファイル処理

大規模なログファイルの効率的な読み込みや、文字コードのトラブル解決、アーカイブ処理をエージェント自身が行えるようにします。

* **`text_grep`**: 正規表現を用いた高速なファイル内検索を実行します。
* **`text_read_head / text_read_tail`**: ファイルの先頭や末尾だけを読み込み、不要なトークン消費を抑えます。
* **`text_convert_encoding`**: Shift-JISなどの文字コードをUTF-8に変換し、文字化けを解消します。
* **`zip_files / unzip_file`**: ファイルの圧縮・解凍を行います。

### 大量ドキュメントの全文検索（Agentic RAG）

数千件のドキュメントから必要な情報を秒速で探し出す「検索能力」をエージェントに与えます。

* **`search_documents`**: インデックスを使用してOffice文書を高速に横断検索します。Boolean検索（AND/OR/NOT）にも対応しています。
* **`list_indexes`**: 利用可能な検索インデックスの一覧と概要を取得します。

最近のトレンドでも、**[「Keyword Search Is All You Need」 (arXiv:2602.23368)](https://arxiv.org/abs/2602.23368)** という論文が話題になりました。これは、複雑なベクトル検索に頼らなくても、エージェントがキーワード検索ツールを自律的・反復的に使いこなすことで、従来のRAGに匹敵する性能を発揮できることを示唆しています。DocToolsはこの「検索を通じたエージェントの推論」をローカル環境で実現します。

## 使い方

DocToolsをAIエージェントに組み込む手順は非常にシンプルです。前提に記載した環境があればすぐに利用を開始できます。

### インストール

まず、以下のGithubリポジトリからソースコードをダウンロードし、そのディレクトリで `pip install` を実行してパッケージをインストールします。

<https://github.com/snaga/doctools-mcp>

```
pip install -r requirements.txt
pip install .
```

これにより、システムに `doctools-mcp`（サーバー本体）と `index_ctl`（インデックス管理ツール）の2つのCLIコマンドが登録されます。

### セットアップ

次に、検索インデックスの保存先を設定します。プロジェクトのルートディレクトリに `.env` ファイルを作成し、インデックスを格納するディレクトリを指定してください。

```
# 複数のインデックスを格納するベースディレクトリを指定（必須）
WHOOSH_INDEX_BASE_DIR=C:\path\to\your\index_base
```

環境設定ができたら、検索対象となるドキュメントが格納されているフォルダに対して、インデックスを構築します。

```
# 「docs」という名前でドキュメントフォルダのインデックスを作成
index_ctl build C:\path\to\your\documents docs --description "主要なプロジェクト資料"
```

### Gemini-CLIでの登録

Gemini-CLIからDocToolsを利用するには、ターミナルで以下のコマンドを実行すれば登録が完了します。

```
# カレントプロジェクトに登録する場合
gemini mcp add doctools doctools-mcp --trust

# ユーザー設定としてグローバルに登録する場合
gemini mcp add --scope user doctools doctools-mcp --trust
```

`--trust` オプションを付与することで、各ツール実行時の確認プロンプトをスキップできるようになり、エージェントとのやり取りがよりスムーズになります。

### プロンプトの例

準備ができたら、エージェントに対して自然言語で指示を出してみましょう。DocToolsが状況に応じて適切なツールを自動的に選択し、処理を実行してくれます。

* **ドキュメントの横断検索**:  
  「『API仕様』に関連するドキュメントを `docs` インデックスから検索して。」
* **PowerPointの解析**:  
  「`system_design.pptx` の3ページ目を画像として抽出し、そこに描かれているシーケンス図の内容を説明して。」
* **Excelデータの活用**:  
  「`sales_data.xlsx` の『Q4実績』シートをMarkdown形式で読み込んで、前年比の伸び率を分析して。」
* **PDFのMarkdown化**:  
  「`user_manual.pdf` の全内容をMarkdownに変換して `output/manual.md` として保存して。」

## まとめ

というわけで、今回は自作したMCPサーバ「DocTools」を紹介してみました。

ここ数カ月間、このツールを業務で常用していますが、自分自身のAIエージェント体験は劇的に進化しました。これまで人間が手作業でやっていた、「ファイルを開いて中身を確認し、コピペして渡す」という不毛な作業を、今ではエージェントが自律的に、かつ必要な時に瞬時に行ってくれています。

日々、数千本のドキュメントを対象に調査や分析をしていますが、人間なら心が折れそうになる量のドキュメントにDeep Diveして情報収集してくるその姿には圧倒されます。また、従来あまりうまく扱えていなかった「PPTに記載されたシステム構成図」なども、PNG化することでマルチモーダルLLMで理解できるようになり、LLMに与えられる情報の解像度もかなり上がりました。

このツールが提供しているのは一見単なるファイル変換機能に見えますが、その実、 **「Agentic RAG」をローカル環境で圧倒的なクオリティで実現するもの** であり、AIエージェントに「自ら資料を探索し、理解し、活用する能力」を与えることなのではないかと感じています。

もし、日々の業務で「大量の資料を目の前にして途方に暮れた経験」があるのであれば、ぜひ一度DocToolsを試してみていただければと思います。

技術的にみると、「都度ファイル変換して検索＋全文検索」という **「ザ・力技」** みたいなアプローチではありますが、AIエージェントが自分の『目』と『手』を使って資料の山を縦横無尽に駆け巡る様子は、一度目にすると圧倒されます。

ビバ、力技！！💪
