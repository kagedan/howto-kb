## 1. はじめに

こんにちは！  
株式会社うぐいすソリューションズでエンジニアをしているKoukiです。

ChatGPTやClaudeのようなLLMを業務に組み込むとき、多くの現場で最初に立ちはだかるのが「個人情報（PII: Personally Identifiable Information）を外部APIに送ってよいのか」という問題です。クレーム対応の下書き、見積メールの返信、応募者情報の要約——こうしたユースケースでは、氏名・電話番号・住所・マイナンバーといった生の個人情報が入力テキストに自然と混ざり込みます。

そこで本記事では、**Microsoft Presidio** を使って日本語テキストから個人情報を自動検出・マスキングし、マスク後のテキストだけをClaude APIに送るチャットボットを作った話を書きます。特に、Presidioは英語前提の作りになっている部分が多く、日本語対応には相応のカスタマイズが必要でした。その勘所を中心に紹介します。

ソースコードは以下のリポジトリで公開していますので、ある程度記事を読んでいただいたら、ご自身の環境でClaudeなどのAIに解析させた方が手っ取り早く理解していただけると思います。  
<https://github.com/kouki6951/pii-masking-chat>

## 4. Presidioとは（アーキテクチャ）

[Microsoft Presidio](https://github.com/microsoft/presidio) は、テキストや画像からPIIを検出・匿名化するためのOSSフレームワークです。大きく2つのエンジンに分かれています。

### 4.1. presidio-analyzer（検出）

`AnalyzerEngine` がテキストを解析し、`RecognizerResult`（エンティティ種別・開始/終了位置・信頼度スコア）のリストを返します。Analyzerは内部に次の要素を持ちます。

* **NlpEngine**: spaCyやStanzaなどのNLPライブラリのラッパー。トークン化・固有表現抽出（NER）を担当する。
* **RecognizerRegistry**: 個々の認識器（Recognizer）の集合。
* **Recognizer**: 実際にPIIを見つける部品。`PatternRecognizer`（正規表現＋文脈語ベース）と、NER結果を受け取る`SpacyRecognizer`などがある。

検出の流れは「NlpEngineがテキストをNLP処理 → 各Recognizerがその結果や正規表現を使ってPII候補を出す → スコア閾値で足切り」というものです。

### 4.2. presidio-anonymizer（匿名化）

`AnonymizerEngine` が検出結果を受け取り、`replace`（固定文字列に置換）・`mask`（`****`化）・`hash`・`redact`（削除）・`encrypt` といった**オペレータ**で匿名化します。検出と匿名化が疎結合になっているのがPresidioの設計上の強みで、検出器だけ差し替えたり、匿名化方式だけ変えたりが容易です。

```
from presidio_analyzer import AnalyzerEngine, RecognizerRegistry
from presidio_analyzer.nlp_engine import NlpEngineProvider

nlp_engine = NlpEngineProvider(nlp_configuration={
    "nlp_engine_name": "spacy",
    "models": [{"lang_code": "ja", "model_name": "ja_core_news_lg"}],
}).create_engine()
```

## 5. なぜ日本語だと難しいのか

Presidioの標準認識器は英語を前提にしており、日本語にそのまま適用するといくつかの罠を踏みます。実装中に遭遇したものを挙げます。

### 5.1. 罠1: 単語境界 `\b` が効かない

標準の `EmailRecognizer` や `CreditCardRecognizer` は正規表現に単語境界 `\b` を使っています。ところがPythonの正規表現では日本語（漢字・かな）も「単語文字」として扱われるため、PIIが日本語に直接隣接すると `\b` が成立せず**検出に失敗**します。

```
カードは4111-1111-1111-1111です  →  標準認識器では検出されない
```

「4」の前が「は」という単語文字なので、`\b`（単語と非単語の境界）が立たないのです。これは英語圏では起きませんが、日本語では文中にPIIが埋め込まれるのが普通なので致命的です。

### 5.2. 罠2: 数字列の意味が文脈依存

電話番号・郵便番号・マイナンバー・運転免許証番号・クレジットカードは、いずれも「数字の並び」です。桁数や区切り文字だけでは判別しきれません。`045-678-9012` の一部 `678-9012` は郵便番号にも見えます。

### 5.3. 罠3: spaCyのNERが一般語を誤検出する

`ja_core_news_lg` のNERは優秀ですが、ビジネス文の一般語（「お客」「担当」「出席者」「弊社」など）を人名・組織名・地名と誤判定することがあります。また、数字列を高スコアで `DATE_TIME` と判定しがちです。

## 6. 日本語対応のためのカスタマイズ

これらに対して、以下のように手を入れました。

### 6.1. lookaroundで `\b` を置き換える

`\b` の代わりに\*\*先読み・後読み（lookaround）\*\*で「前後が数字・英字でないこと」を表現した独自の `PatternRecognizer` を用意します。メールアドレスならこうです。

```
Pattern(
    name="jp_email",
    regex=(r"(?<![A-Za-z0-9._%+\-])"
           r"[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,}"),
    score=0.9,
)
```

`(?<!...)` は「直前がこの文字クラスでない」という意味で、日本語が隣接していてもマッチします。クレジットカードも同様に `(?<![0-9０-９])...(?![0-9０-９])` で前後を数字以外に限定しています。半角・全角の数字（`[0-9０-９]`）や各種ハイフン（`-‐–—ー－` 等）を許容する文字クラスを定義しておくのも日本語ならではのポイントです。

### 6.2. 日本の定型PIIを正規表現で検出

電話番号・郵便番号・マイナンバー・住所・パスポート・運転免許証・銀行口座を、それぞれ `PatternRecognizer` として定義しました。たとえば電話番号は携帯・固定・フリーダイヤルを別パターンで持ちます。

```
# 携帯: 070/080/090-XXXX-XXXX
Pattern(name="jp_mobile", regex=r"0[789]0-?\d{4}-?\d{4}", score=0.7)
# 固定: 市外局番-市内局番-加入者番号
Pattern(name="jp_landline", regex=r"0\d{1,3}-\d{1,4}-\d{4}", score=0.65)
```

固定電話のスコア（0.65）を郵便番号（0.6）より高くしているのは、`045-678-9012` のような番号で「電話番号全体」が「郵便番号の部分一致」に勝つようにするためです。住所は47都道府県名をアンカーにして、市区町村〜丁目・番地・号までを大まかに拾います。完全な構文解析は正規表現では難しいので、過不足を許容する割り切った設計にしています。  
![](https://static.zenn.studio/user-upload/87b02470397e-20260605.png)

### 6.3. 文脈語（context）とスコア設計

`PatternRecognizer` には `context` 引数で**文脈語**を渡せます。検出箇所の近くに文脈語があると、Presidioがスコアを加点してくれる仕組みです。銀行口座やマイナンバーのような「ただの数字列」は基準スコアを低め（0.3）に設定し、「口座」「振込」「マイナンバー」などの語が近くにあるときだけ閾値を超えて検出されるようにしています。

```
PatternRecognizer(
    supported_entity="JP_BANK_ACCOUNT",
    patterns=[Pattern(name="jp_bank_account", regex=r"...\d{7,8}...", score=0.3)],
    context=["口座", "振込", "普通", "当座", "預金", "銀行", "支店"],
)
```

ここで日本語特有の落とし穴がありました。文脈語は**形態素単位**で照合されるため、「郵便番号」は「郵便」「番号」に分割されることがあります。そのため `context=["郵便番号", "郵便", "番号"]` と分割後の語も登録しておく必要があります。

### 6.4. 敬称による氏名補強

NERが拾い切れない氏名を補うため、**敬称の直前**を氏名候補とする認識器を追加しました。先読みを使い、敬称自体は置換対象に含めません。

```
name = r"[一-龥々ぁ-んァ-ヶ]{2,10}"
honorific = r"(?=(?:さん|様|氏|くん|先生|部長|課長|社長))"
Pattern(name="jp_person_honorific", regex=name + honorific, score=0.45)
```

「田中さん」から「田中」だけを氏名として検出できます。

### 6.5. 重複検出の解決

同じ箇所に複数のエンティティが当たることがあります（spaCyが `150-0043` を `LOCATION`、郵便番号認識器が `JP_POSTAL_CODE` と検出するなど）。そこで重複区間を次の優先順で1つに絞り込む処理を自前で書きました。

1. スコアが高いものを優先
2. 同点なら専用ラベル（`JP_*`）を汎用ラベル（`LOCATION`/`DATE_TIME`）より優先
3. それも同じなら長い区間を優先

これにより、spaCyが数字列を `DATE_TIME` と誤判定しても、専用の電話番号・郵便番号検出が勝ちます。絞り込んだ集合をそのまま置換に使うことで、画面の検出一覧とマスク後テキストのラベルが必ず一致します。

### 6.6.　denylistで過検出を抑える

「お客」「担当」「弊社」「会議室」などビジネス文の頻出語を `NER_DENYLIST` に列挙し、これらがPERSON/ORGANIZATION/LOCATIONと判定されても除外します。個人情報保護では「見逃すより過剰にマスクする」方が安全なので閾値は低め（0.4）に置きつつ、明らかな一般語だけはピンポイントで落とす方針です。

## 7. 可逆マスキング（送信前確認と復号）

当初はPresidio標準の `AnonymizerEngine` で全件を `<氏名>` のような固定文字列に置換していました。しかしこれだと、複数人が登場したとき「どの`<氏名>`が誰か」が区別できず、応答を**元に戻せません**。

そこで、プレースホルダを `<氏名_1>` `<氏名_2>` のような**一意の連番付き**にし、`プレースホルダ → 元の値` のマッピングを保持するよう変更しました。置換は `masker.py` 内で手動実装し、同じ値には同じトークンを再利用、別の値には連番を進めます。会話をまたいで採番を継続することも可能です。

![](https://static.zenn.studio/user-upload/229171cd271d-20260604.png)

復号は逆向きの単純な置換です。長いトークンから順に戻すことで `<氏名_1>` と `<氏名_11>` の接頭辞衝突を防ぎます。

```
def demask(text: str, mapping: dict[str, str]) -> str:
    for token in sorted(mapping, key=len, reverse=True):
        text = text.replace(token, mapping[token])
    return text
```

UI側では、(a) 送信前にマスク内容と検出一覧を提示して承認を求め、(b) Claudeの応答は復号前（プレースホルダのまま）で表示し、ユーザーが「復号する」を押したときだけ元の値へ戻す、という二段階の確認フローにしました。これで「何が外部に出るか」「いつ元の情報に戻すか」をユーザーが完全にコントロールできます。

## 8. まとめ

Presidioは検出と匿名化が疎結合で、認識器を差し替えやすい優れたフレームワークですが、日本語に適用するには次のカスタマイズが要点でした。

* 単語境界 `\b` をlookaroundに置き換える
* 日本の定型PIIを正規表現の `PatternRecognizer` で実装する
* 文脈語とスコア設計で「ただの数字列」の曖昧さを解消する（形態素分割にも注意）
* 重複解決とdenylistで誤検出・過検出を抑える
* 一意プレースホルダ＋マッピングで可逆マスキングを実現する

LLMを業務に導入する際の「個人情報をどう扱うか」という共通課題に対し、Presidioは現実的な選択肢になります。本番運用では対象ドメインのサンプルで閾値とパターンをチューニングするのがおすすめです。
