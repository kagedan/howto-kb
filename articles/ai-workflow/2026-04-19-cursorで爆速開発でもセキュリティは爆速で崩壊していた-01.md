## TL;DR

* AIコーディングはコード生産速度を上げるが、レビュー・セキュリティのボトルネックも同時に増大する
* 修正パッチのタイミングが一番危ない。最初の緊張感はAIには引き継がれない
* 非エンジニアがコードを読むのは現実的じゃない。だからセキュリティエンジニアをSKILLに憑依させた
* SKILL.mdをGitHubに公開したので持ってけ

---

## 他人事じゃなかった

先日、NYTにこんな記事が出た。

> AIコーディングツールを導入した金融企業で、月のコード生産量が2.5万行から25万行に10倍化。結果、レビュー待ちのコードが100万行以上積み上がった。

「AIで効率化したのに人手不足」という皮肉な話だ。

読んで即思った。**これ、自分のことだ。**

自分はGCPとVertex AIでAIエージェントを作っている。Cursorをメインに、Claude Codeも使う。アプリの追加・修正は日常茶飯事だ。

そしてある日、Googleからアラートが来た。

シークレットキーの露出。

---

## なぜ起きたか

誤解しないでほしいのだが、最初からセキュリティを軽視していたわけじゃない。

* Secret Managerを使っていた
* `.env`は`.gitignore`に入れていた
* 認証まわりもちゃんと設計していた

それでも起きた。

原因は**修正パッチ**だった。

新しい機能を追加するとき、AIは既存のコードパターンを参照する。でも「参照する」と「正しく引き継ぐ」は別の話だ。AIは「動くコード」を生成する。「安全なコード」を生成するとは限らない。

最初の実装は緊張しながら確認する。でも修正パッチは流れで入れる。そこに穴が開く。

しかもCursorを使えば修正の頻度自体が上がる。つまり**修正頻度が上がるほど、リスクの総量も増える**という構造になっている。

---

## 従来のチェックリストが通用しない

もう一つ厄介なのが、AIアプリ特有の攻撃面が増えていることだ。

従来のセキュリティチェックリストには載っていない問題がある。

**プロンプトインジェクション**  
ユーザー入力がシステムプロンプトに直接連結されていると、攻撃者がシステムプロンプトを書き換えられる。AIアプリ特有のリスクだ。

```
# NG：ユーザー入力をシステムプロンプトに直接連結
prompt = f"あなたはアシスタントです。ユーザーの指示: {user_input}"

# OK：ロールを分離する
messages = [
    {"role": "system", "content": "あなたはアシスタントです。"},
    {"role": "user", "content": user_input}
]
```

**SSRF（サーバーサイドリクエストフォージェリ）**  
AIエージェントが外部URLを叩く処理を持っている場合、allowlistなしだと内部ネットワークへのアクセスに悪用される。

**コスト爆発**  
レートリミットなし・`max_tokens`未設定のままループ内でLLM APIを呼び出すと、請求が爆発する。セキュリティインシデントと同等に扱うべき問題だ。

これらは「6ヶ月前には誰も想像しなかったリスク」だとNYTの記事でも指摘されている。

---

## 解決策：セキュリティエンジニアを憑依させる

非エンジニアがコードを読んでレビューするのは現実的じゃない。

では誰がやるのか。

**AIにやらせる。ただし、ちゃんとしたやつに。**

Claude CodeのSKILL機能を使って、シニアセキュリティエンジニアを憑依させた。

SKILLは「呼び出し型」だ。毎回コンテキストに入るCursorのルールと違い、「セキュリティレビューして」と言ったときだけ読み込まれる。トークンの無駄がない。

---

## SKILL.mdの構成

最初は5軸で作った。

シークレット露出・認証抜け・SSRF・プロンプトインジェクション・コスト爆発。自分が実際に遭遇したリスクを中心に組んだ。

でも完成してから、セキュリティエンジニア視点で自分でレビューしてみた。

**穴だらけだった。**

SQLインジェクションがない。XSSがない。CSRFがない。セッション管理がない。ファイルアップロードの検証がない。OWASPが10項目ある理由はここにある。「よくあるチェックリスト」を作ったつもりが、「自分が知っているリスクのリスト」になっていた。

「セキュリティエンジニアを憑依させる」と言うなら、自分の知識の範囲で作っても意味がない。プロが実際に使う基準で作り直す必要があった。

---

結果、**5カテゴリ・18サブ軸**に再設計した。OWASP Top 10・OWASP LLM Top 10・CVSS v3.1準拠。

**カテゴリA：入力・出力の安全性**  
SQLi / NoSQLi / コマンドインジェクション、XSS（LLM出力のHTML埋め込みを含む）、出力エスケープ。AIは生SQLを文字列連結で作ることが今でも普通にある。

**カテゴリB：認証・認可**  
認証の実装、セッション管理（有効期限・ローテーション・ログアウト時の無効化）、CSRF、IDOR（直接オブジェクト参照）。5軸版では「認証の有無」しか見ていなかったが、「認可の粒度」まで見ないと他ユーザーのデータが取れる穴が残る。

**カテゴリC：外部通信・データ保護**  
シークレット管理、SSRF（プライベートIPブロック含む）、CORS・セキュリティヘッダー、ファイルアップロード・ストレージ権限。

**カテゴリD：運用・可観測性**  
ログの機微情報漏洩（PII・決済情報）、依存関係・サプライチェーン（Typosquatting対策・SBOM）、レートリミット・コスト制御、環境分離。

**カテゴリE：AI固有のリスク（OWASP LLM Top 10）**  
プロンプトインジェクション、LLM出力の取り扱い（`eval`禁止含む）、エージェントの権限・暴走防止、RAGのデータ境界。

加えて、**Top 10クリティカルチェック**を先頭に置いた。レビュー開始時にまずこの10項目を確認して、🔴が見つかった時点で即報告する設計だ。全軸を順番に読む前に致命的な問題を早期検出するため。

**コミット可否の判定ロジックも明文化した。**

```
🔴 1件でもあり → 🚫 NG（コミット不可）
🟠 3件以上    → 🚫 NG
🟠 1〜2件    → ⚠️ 要修正後
🟡 のみ      → ✅ OK（修正推奨）
```

感覚で「まあ大丈夫か」と判断させない。これが重要だ。

作ったセキュリティーレビューSKILLはこちら（ご参考まで）  
[security-review/SKILL.md](https://github.com/nekoai-lab/claude-skills/blob/main/security-review/SKILL.md)

## 使い方

SKILLはGitHubで公開している。プロジェクトの `.claude/skills/` にコピーして使う。

```
mkdir -p your-project/.claude/skills/security-review
curl -o your-project/.claude/skills/security-review/SKILL.md \
  https://raw.githubusercontent.com/nekoai-lab/claude-skills/main/security-review/SKILL.md
```

Claude Codeから呼び出す場合：

または差分を貼って：

```
このコードのセキュリティレビューして
[コードを貼る]
```

レビュー結果は深刻度別（🔴致命的 / 🟠高 / 🟡中 / 🟢問題なし）で返ってくる。最後に「コミットしてよいか」の判定もつく。

## AI特有のリスクパターン早見表

差分レビュー時にこれだけ見ればいい。

| パターン | リスク | 確認ポイント |
| --- | --- | --- |
| デバッグ用print/console.log追加 | シークレット出力 | ログ出力の内容を目視確認 |
| 環境変数の取り扱い変更 | ベタ書きへの退行 | 変更前後のキー管理を比較 |
| 新しいAPIエンドポイント追加 | 認証なしで公開 | 認証ミドルウェアの適用を確認 |
| 「動作確認のため一時的に」 | 認証を外したまま残る | 認証チェックが無効化されていないか |
| パッケージ追加 | 脆弱性のある依存関係 | `npm audit` / `pip audit` を実行 |
| リトライ処理追加 | コスト爆発 | 上限・バックオフの有無を確認 |

---

## まとめ

* AIコーディングは「生成ボトルネック」を解消するが、「レビューボトルネック」を顕在化させる
* 修正パッチのタイミングが最も危ない。緊張感はAIに引き継がれない
* AIアプリにはプロンプトインジェクション・SSRF・コスト爆発という新しい攻撃面がある
* 解決策はセキュリティエンジニアをSKILLに憑依させること

コードは動いている。でも安全かどうかは別の話だ。
