AIエージェントへ開発を頼むたびに、長いプロンプトを書き直していないでしょうか。

対象ファイル、実装手順、参照する設計書、テスト方法、完了条件、触ってはいけない範囲。必要な情報を一つの指示へ足していくと、最初は便利でも、やがて似ているのに少しずつ違うプロンプトが増えていきます。

私も個人開発でAIエージェントを使う中で、手順と個別タスクの条件を同じ場所へ書く運用に限界を感じました。そこで、AIが毎回たどる手順を **Command**、そのタスクだけに属する条件を **Task Definition** として分けました。

自分の開発では、開始、実装、PR作成、レビューなどの共通手順をCommandへ残し、対象ファイルや受け入れ条件をタスクごとのDefinitionへ移しました。これにより、共通手順を直す変更と、個別タスクの条件を直す変更を分けて扱えるようになりました。

結論から言えば、次のように責務を分けます。

* Commandは「どう進めるか」を持つ
* Task Definitionは「今回は何を、どの条件で行うか」を持つ
* Issueは「なぜ行うか」と、人間を含む合意を持つ
* TemplateはIssueやPRの表示形式を揃える

この分離は、AIエージェントを手順、入力、検証条件で囲うハーネス設計の一部として捉えられます。ただし、この記事が扱うのはハーネス全体ではなく、指示と作業条件を管理する範囲です。

また、この分離によってAIの誤りがなくなるわけではありません。それでも、指示のどこを変更すべきかが明確になり、不足した入力の検出や完了条件の照合にも同じDefinitionを使えるようになります。

## 巨大プロンプトでは、手順と作業条件が一緒に変わる

たとえば、AIへ次のような依頼を渡すとします。

> Issueを確認し、関連する設計書を読んで実装してください。対象は設定画面です。APIの契約は変更せず、既存テストを壊さないでください。実装後はテストを実行し、PR本文へ変更点と結果を書いてください。

一度だけなら、これでも十分です。問題は、同じ流れを別のタスクにも使い始めたときに起きます。

「Issueを読む」「設計書を確認する」「テストする」「結果を報告する」は、多くのタスクに共通する手順です。一方、「設定画面が対象」「API契約は変更しない」は、そのタスクだけの条件です。

両者を一つの文章で管理すると、タスクが変わるたびに全文を複製し、共通手順まで編集することになります。その結果、古いプロンプトだけテスト手順が更新されていない、あるタスクでは報告形式が違う、といった版ずれが生まれます。

また、必須の設計書が指定されているかを確かめるにも、自然言語の全文を解釈しなければなりません。

問題はプロンプトの長さそのものではなく、**変更理由の異なる情報が同居していること**です。

## Commandは手順、Task Definitionは作業条件

分離後のCommandには、タスクが変わっても維持したい流れだけを残します。

```
1. Task Definitionを読み込む
2. 必須入力と参照先を検証する
3. 対象範囲の中で変更する
4. 指定された検証を実行する
5. 受け入れ条件と結果を照合する
6. 変更内容、検証結果、未解決事項を報告する
```

一方、Task Definitionには、個別タスクによって変わる値を置きます。以下は説明用に簡略化した架空の例です。

```
task:
  id: settings-display-name
  objective: 設定画面で表示名を変更できるようにする

inputs:
  issue: "#123"
  docs:
    - docs/settings.md

scope:
  include:
    - src/features/settings/
    - tests/settings/
  exclude:
    - src/api/contracts/

outputs:
  - 設定画面の実装
  - 関連テスト

acceptance_criteria:
  - 保存後に新しい表示名が表示される
  - 既存の設定画面テストが成功する

stop_conditions:
  - 必須の設計書が存在しない
  - API契約の変更が必要になる
```

ここでの`outputs`は出力先のファイル名ではなく、タスクで残す成果物の種類を表しています。

重要なのはキー名ではなく、Commandから「今回だけ変わる情報」を外へ出すことです。分離したDefinitionは、実行時に読むだけでなく、着手前の検証と終了時の照合にも使います。

CommandとTask Definition以外にも、IssueとTemplateがあります。それぞれの責務と変更頻度を整理すると、次のようになります。

| 要素 | 主な責務 | 変更頻度 |
| --- | --- | --- |
| Command | 実行順序、共通チェック、結果報告の形式 | 低い |
| Task Definition | 対象、入力、対象範囲、受け入れ条件、停止条件 | タスクごと |
| Issue | 背景、目的、合意、進捗への入口 | タスクごと |
| Template | IssueやPRの人間向け表示形式 | 低い |

Issueは、人間とAIが背景や目的を共有する作業正本です。ここでいう作業正本とは、正式な作業条件を確認するときの基準を指します。

一方、Definitionは実行と検証に使う構造化データです。内容が一部重なっても、Issueは説明と合意、Definitionは機械的な読み取りを主な責務にすると整理しやすくなります。

TemplateもDefinitionとは別物です。Issue TemplateやPR Templateは、人間が読みやすい表示形式を揃えます。Definitionは、処理に必要な値を安定して取り出すために使います。

## Definitionを作業の前後で再利用する

Task Definitionを単なる「AIに読ませる設定ファイル」にすると、分離の効果は限定的です。同じDefinitionを作業の前後で使うと、価値が大きくなります。

### 着手前：不足した入力を検出する

実装前には、少なくとも次を確認できます。

* 必須項目が存在するか
* 指定されたIssueや設計書を参照できるか
* 対象範囲と対象外が矛盾していないか
* 受け入れ条件が空になっていないか
* 人間の判断が必要な停止条件に該当していないか

たとえば必須の設計書が存在しない場合、AIに推測で実装させず、そこで止めて不足を返します。構造化の利点は、AIが賢く解釈できることより、**実装を始めてよい状態かを先に判定しやすいこと**にあります。

### 実行時：必要な参照情報だけを組み立てる

Definitionの`inputs`と`scope`を使えば、タスクに必要なIssue、設計書、対象ディレクトリを選んでAIエージェントへ渡せます。毎回リポジトリ全体や長大な運用ルールを渡す必要はありません。

ただし、これによるトークン削減や精度向上は、実際に計測して判断すべきです。構造を分けただけで、自動的に参照情報が最適化されるわけではありません。

### 終了時：同じ条件で完了を照合する

実装後には、`outputs`と`acceptance_criteria`をチェックリストとして再利用できます。

* 必要な成果物が作られたか
* 指定されたテストが成功したか
* 対象範囲外の変更が含まれていないか
* 停止条件に該当する問題を未報告のまま進めていないか

開始時と終了時に同じDefinitionを使うことで、「依頼時に書いた完了条件」と「完了報告で確認した条件」が離れにくくなります。

## すべてのタスクにDefinitionを作らない

構造化にはコストがあります。数分で終わる文言修正までYAMLへ落とすと、実装よりDefinitionを書く時間の方が長くなります。

私なら、次のいずれかに当てはまるタスクから導入します。

* 同じCommandで繰り返し実行する
* 複数の入力資料や成果物がある
* 対象範囲外の変更による影響が大きい
* 受け入れ条件を機械的に確認したい
* 入力不足や仕様衝突時に、確実に止めたい
* 複数のAIエージェントやセッションへ引き継ぐ

反対に、変更範囲が明白で、受け入れ条件もIssue本文から誤解なく読める小さなタスクなら、Issueだけで十分です。

最初から詳細なスキーマや実行基盤を作る必要もありません。まずは`objective`、`inputs`、`scope`、`acceptance_criteria`、`stop_conditions`の5項目程度から始めます。

その後、実際に検証や引き継ぎへ使う項目だけを増やす方が運用しやすくなります。

## 注意点

CommandとTask Definitionを分けても、曖昧な要件が正しくなるわけではありません。「使いやすくする」のような条件をYAMLへ移しても、判定可能にはなりません。

受け入れ条件は、観測できる結果として書く必要があります。

また、Issue、Definition、設計書へ同じ情報を重複して書くほど、更新漏れが起きやすくなります。それぞれの正本を決め、他の文書からは参照する形が基本です。

Definitionの項目を増やしすぎることにも注意が必要です。使われない項目は入力作業を増やすだけでなく、「すべて埋まっているから安全」という誤解を生みます。

項目を追加するときは、誰が、どの段階で、何の判定に使うかまで決めます。

なお、同じ分離は実装だけでなく、レビュー、PR作成、ドキュメント更新などの手順にも適用できます。

## まとめ

AIエージェントへの指示を再利用したいなら、巨大なプロンプトを上手に書く前に、変更理由の異なる情報を分けます。

* 変更頻度の異なる「共通手順」と「今回の作業条件」を分ける
* Task Definitionを着手前の検証と終了時の照合にも使う
* 高リスクまたは繰り返し実行するタスクから段階的に導入する

この設計の価値は、AIへ渡す文章を短くすることだけではありません。実装を始めてよい条件と、完了したと判断する条件を、同じ場所から取り出せることにあります。

まずは、繰り返し実行する中規模のタスクを一つ選び、現在のプロンプトを「毎回同じ手順」と「今回だけの条件」に色分けしてみてください。その境界が、最初のCommandとTask Definitionになります。
