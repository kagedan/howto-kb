## 「Sunoの曲、楽譜にできないかな」から始まった

Suno AIでクラシック風の弦楽曲を生成したら、思いのほか良い曲ができた。バロック〜古典派っぽい雰囲気で、弦5パート（Vn1, Vn2, Va, Vc, Cb）が絡み合ってる感じ。G major、135-140 BPM、4/4拍子。

問題は楽譜がないこと。Sunoは音声しか出さない。MIDIもない。「この曲を演奏したい」と思っても、耳コピで2分の弦楽5パートを書き起こすのは地獄でしかない。

AIに音声を聴かせて楽譜にできないか。そう思って始めたプロジェクトが、結局7回の作り直しを経ることになった。

## まずは正攻法でやってみた（v2-v3）

最初に試したのは、いかにも教科書的なパイプライン。SpotifyのBasic Pitch（音声→MIDI変換のMLモデル）でMIDIに変換して、音域ベースで5パートに振り分けて、Claude Code SDKでパートごとに補正する。

楽器ごとの演奏特性をプロンプトに埋め込んで、「この楽器らしい音長に直して」と頼む設計にした。

```
"cb": (
    "Contrabass（コントラバス）。最低声部。チェロと同じ動きを"
    "することが多い（1オクターブ下）。基本的に長い音（2分音符"
    "〜全音符）でバスラインを支える。16分音符の速いパッセージ"
    "はまず出てこない。音の入りはやや遅れがち（発音が遅い楽器）。"
),
```

結果は散々だった。Basic Pitchが吐くMIDIが0.25拍（16分音符）だらけの不自然な音の羅列で、LLMで補正しても平均0.8拍にしかならない。そもそもの音高とタイミングが間違ってるのに、その上から「楽器らしくして」と言ったところで限界がある。

じゃあ個別補正じゃなくて、全5パートを一度にClaudeに渡して「編曲」させたらどうか。v3でそれを試したけど、出てきたのは音楽的にはそれっぽいが元曲からかなり離れたもの。別の曲になっちゃった。

ここで得た教訓は明確だった。**LLMは「磨く」ことはできても「聴く」ことはできない。** 転写の精度がボトルネックである限り、後段でいくら頑張っても無駄。そして「編曲」は自由度を与えすぎる。元曲を尊重する「補正」でないとダメ。

## 「LLMに直接聴かせたらどうなる？」（v4-v5）

ここで発想を変えた。Basic Pitchを捨てて、Geminiに音声を直接渡して楽譜を書かせたらどうなるか。Geminiはマルチモーダルモデルだから、音声入力を受け付ける。

```
uploaded = client.files.upload(file=audio_path)
response = client.models.generate_content(
    model="gemini-3-pro-preview",
    contents=[
        types.Content(parts=[
            types.Part.from_uri(
                file_uri=uploaded.uri, mime_type="audio/mpeg"),
            types.Part.from_text(text=prompt),
        ])
    ],
)
```

音声ファイルをAPIに投げて「Violin Iパートを聴き取ってJSON出力して」と頼むだけ。

v4ではGemini 3 Flashで最初の30秒だけ試した。粒度は粗かったけど、出てきた楽譜が音楽的に自然で正直驚いた。Basic Pitchの「信号処理としての音声認識」と、Geminiの「音楽として聴いている」感覚の違いがはっきり出た。

v5ではFlashのまま2段階方式にした。Stage 1で曲全体の構造を分析させて、Stage 2でその分析を参照しながらパート別に転写。全曲カバーできるようになったけど、精度は4〜5割止まり。Flashの聴き取りには限界がある。

```
v2-v3: 音声 → Demucs分離 → Basic Pitch → 声部分離 → LLM補正
v4-v5: 音声 → Gemini（直接聴き取り）
```

この時点で、パイプラインの構造自体が根本的に変わった。「音声を信号として処理する」から「音楽として聴かせる」へ。

## Gemini 3 Proで一気に化けた（v6-v7）

v5と全く同じ手法で、モデルだけ `gemini-3-pro-preview` に変えた。これが転機だった。

精度が一気に上がった。特にVn1のメロディラインとCbのバスラインが「楽器らしい」。Stage 1の分析テキストも詳細で、各パートの役割・音域・時間的な動きを正確に言語化してた。

2段階の設計が効いてると思う。まず全体像を掴んでから細部を転写する、というのは人間が耳コピするプロセスにも近い。

さらにv7では、Geminiの転写結果をClaude Code SDKで補正した。Claudeは音声を「聴く」のは苦手だけど、MIDIデータを読んで「楽器としての自然さ」を判定するのは得意。得意分野の分業。

| パート | ノート数 | 平均音長(拍) | 所感 |
| --- | --- | --- | --- |
| Vn1 | 258 | 1.01 | メロディらしい動き |
| Vn2 | 296 | 0.84 | 刻みパターンあり |
| Va | 91 | 2.85 | 持続音中心 |
| Vc | 168 | 1.55 | バス+旋律 |
| Cb | 104 | 2.54 | 長い音でバスを支える |

Va（2.85拍）とCb（2.54拍）の平均音長が、内声・低音パートは長い音で支えるという弦楽の特性をちゃんと反映してる。v2の「全パート0.25拍だらけ」と比べると、楽器ごとの個性が出ているのがわかる。

## 3つのアプローチを振り返る

|  | 従来ML (v2-v3) | LLM直接転写 (v4-v6) | ハイブリッド (v7) |
| --- | --- | --- | --- |
| 転写 | Basic Pitch | Gemini 3 Pro | Gemini 3 Pro |
| 補正 | Claude | なし | Claude |
| 精度 | 2-3割 | 5-6割 | 6-7割 |
| 音楽性 | 低い | 高い | 最も高い |
| APIコスト | 低い | 中 | 高い |

```
v7: 音声 → Gemini（転写）→ Claude（補正）
```

結局たどり着いたのは「Geminiが聴いて、Claudeが直す」というシンプルな分業だった。従来のML（Basic Pitch等）は音声のスペクトログラムを見ている。LLMは音楽を聴いている。この違いが決定的だったと思う。

ただしLLMも万能じゃなくて、生成と検証を別のモデルでやる構成が一番うまくいった。コーディングで「生成AI + 別AIでレビュー」がワークするのと同じ構図かもしれない。

## 雑感

正直、まだMVPの域を出ていない。人間が聴いて「これが正しい楽譜だ」と言えるレベルには遠い。でも「8割くらいの下書き」としてMuseScoreで開いて手修正する前提なら、ゼロから耳コピするよりは圧倒的に早い。

Gemini 3 Proが音声を「聴き取れる」ことの驚きが一番大きかった。マルチモーダルLLMの能力を正直甘く見てた。音声ファイルを渡して「Violin Iパートを書き起こして」で、それなりのものが返ってくる時代。
