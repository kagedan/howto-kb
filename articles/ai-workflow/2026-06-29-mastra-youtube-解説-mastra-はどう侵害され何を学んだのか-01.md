## 概要

この動画は、**Mastra チームが実際に受けたサプライチェーン攻撃を、自分たちの口でかなり率直に振り返る**回です。単に「こんな被害がありました」と報告するだけではなく、どういう導線で侵害されたのか、初動で何に手間取ったのか、なぜいまのオープンソースは攻撃対象になりやすいのかまで話が広がっていきます。

特に印象的なのは、攻撃の技術的な派手さよりも、**ソーシャルエンジニアリングとパッケージ公開基盤の弱さが組み合わさると、かなり現実的に被害が成立してしまう**という点です。信頼された開発者が侵害される。悪意あるパッケージは本物らしく擬装される。レジストリの対応は遅い。スキャナは役に立つが、時に雑に拡散する。そして、プロジェクトが成功すればするほど標的価値が上がる。

Shane、Abhi、そして Superagent の Ismael の会話は、それぞれ少しずつ立場が違いながらも、最終的には同じ現実を指しています。つまり、**いまのオープンソースは「善意の協業の場」であるだけでなく、「常に観測され、試され、狙われる供給網」でもある**ということです。

その意味で、この回の価値は Mastra の失敗談にあるのではなく、失敗をかなり具体的に言葉にして共有した点にあります。セキュリティの話としても、オープンソース運営の話としても、かなり示唆の多い動画でした。

サプライチェーン攻撃についての基本的な解説が必要な方は、ちょうど昨日公開したこちらの記事も併せて読んでいただくと、より一層理解が深まると思います。

## 要点

1. 攻撃の入口は高度なゼロデイではなく、Microsoft Teams に見せかけた通話リンク経由のソーシャルエンジニアリングでした。
2. 侵害された端末に残っていた NPM トークンと、未設定だった二要素認証が被害拡大の決定打になりました。
3. 攻撃者は easy-day-js という見た目の整った悪意あるパッケージを公開し、postinstall 系の仕組みで利用者側を侵害しようとしていました。
4. 封じ込め自体は数時間でできたものの、NPM の運用ツールやサポート体制の弱さが初動を難しくしました。
5. Ismael は、Mastra の成長や GitHub Stars の伸びそのものが「標的としての価値」を高めたと見ています。
6. いま重要なのは、「信頼された開発者も侵害される」前提で CI/CD、PR レビュー、権限設計を見直すことだと語られています。
7. スキャナや通知ツールは有用ですが、メンテナへの伝え方や誤情報の広がり方には大きな課題があります。

## 詳細

### 成功すると狙われる。それが今回の出発点だった

**(Shane)** 先週は大きな出来事がありました。言ってしまえば、北朝鮮案件です。なので今回は特別編の Security Corner をやります。Mastra ではもちろんセキュリティをかなり重視していますが、それでも起きるときは起きます。先週起きたこと、その背景、そして業界全体で何が起きているのかを話したいと思います。そこで Superagent の Ismael に来てもらいました。

まずは先週 Mastra で実際に起きたことから話します。そのうえで Ismael に、外から見てどうだったか、業界では何が見えているかを話してもらいます。まだ知らない人のために言うと、Mastra はサプライチェーン攻撃を受けました。かなり巧妙でした。

どうやってやられたのか。そこも順に話せますが、やられたのは事実です。調査の結果、北朝鮮系のハッカーにつながっている可能性が高そうだという話も出ています。もちろん、どう検証されたかまでは断定できません。ただ、以前の axios 系の攻撃パターンとも一致していて、そう考える理由はあるようです。

### 入口は Teams に見せかけた通話リンクだった

**(Shane)** きっかけは、LinkedIn でそれなりに知られた人物のアカウントが乗っ取られ、うちのエンジニアに「AI エンジニアリングについて話そう」と連絡してきたことでした。うちのエンジニアは普段からユーザーや顧客、同じ分野の人たちと気軽に会って話します。だからその相手とも普通に通話の約束をしたんです。送られてきたのは Microsoft Teams のリンクに見えるものでした。

**(Abhi)** 正確には teams.lineapp.com でした。つまり本物の Microsoft Teams のリンクではありません。

**(Shane)** でも、クリックすると見た目は Teams そのもので、挙動もほとんど同じです。違ったのは、何かをインストールするかダウンロードするかを求めてきた点でした。エンジニアは音声がうまく動かないと思い込み、指示どおりボタンを押してしまった。そこからマシンに何かが入り、認証情報にもアクセスされました。

### NPM トークンと二要素認証の抜けが被害を決めた

**(Abhi)** そこから NPM の認証情報が奪われました。こちら側の大きなミスは、NPM の組織メンバー全員が二要素認証を有効化していると思い込んでいたことです。実際には、この人だけ有効になっていませんでした。最初は全員が二要素認証を使っている前提だったので、そこが攻撃経路だとは考えませんでした。Gmail までやられたのではないかと疑ったほどです。かなり時間を無駄にしました。

実際には、攻撃者がそのアカウントを乗っ取り、マシン上に残っていた強力な NPM トークンを使って、NPM アカウントのメールアドレスを変更していました。そのあと、私たちをパッケージの管理権限から追い出し始めたんです。本当に冷や汗ものでした。

**(Shane)** 全部のパッケージではなく、一部のパッケージだけが対象でした。

**(Abhi)** でも、かなり焦りました。ただ、後で自分たちを再度追加し直せると分かって、そこは助かりました。まず攻撃用のパッケージを unpublish し、問題のあるものを deprecated にしようとしたんですが、NPM の操作は本当に厳しかったです。OTP がすぐ切れるし、レートリミットにも引っかかるし、非常時にツールが壊れるのは最悪です。

### easy-day-js は「いかにも本物らしい」悪意あるパッケージだった

**(Abhi)** 攻撃者は easy-day-js というパッケージを公開していました。見た目は dayjs のクローンのようで、不審に見えにくい作りです。中には postinstall script のようなものが仕込まれていたようでした。Ismael が後でもっと詳しく話してくれると思います。

要するに、その時間帯に Mastra をダウンロードしていたら、暗号資産ウォレットや鍵を探す処理が走って、そのまま侵害される可能性がありました。幸い、かなり素早く封じ込めることはできました。数時間かかりましたが、もっとツールが良ければさらに速く終わっていたはずです。その後、正常なバージョンを出し直して、いまは復旧しています。

GitHub Issue で何が起きたかを説明し、有料顧客にも、Mastra を応援してくれている人たちにも、全員に連絡を入れました。Shane と Sam がそのコミュニケーションをかなりやってくれました。

### 数時間で封じ込めたが、基盤側の弱さでかなり消耗した

**(Abhi)** そのあとで「そもそも何が起きたのか」を掘り始めました。でも、こういうソーシャルエンジニアリングは本当に自然に見えるんです。該当エンジニアにも「ここ数週間で変なことはなかった？」と聞いたのですが、最初は思い当たらないと言っていました。失敗した Zoom や Teams の通話なんて、普通は記憶に残りません。そこで、ここ数週間に起きたことを一つずつたどっていって、ようやく「あれはおかしかった」と気づいたんです。

Telegram のようなところから来たメッセージに応じるべきではなかったし、Microsoft ではない変な通話リンクに入るべきでもありませんでした。ただ、これは誰にでも起こりえます。チームの他のメンバーも、「自分でも同じように引っかかったかもしれない」と言っていました。だからそのマシンはいったん死んだものとして扱い、完全に初期化し、個人用と業務用の認証情報を全部再発行しました。いまはそのマシンも復旧しています。

### Ismael は「かなり組織的だった」と見ている

**(Ismael)** 外から見ていて興味深かったのは、その少し前に Slack で、すでに怪しいコントリビュータが PR に暗号資産向けのバックドアを仕込もうとしていた、という話があったことです。偶然かもしれませんが、私は単なる偶然には見えません。どちらにも暗号資産の文脈があるからです。

私がその週に見ていたのは、Mastra が GitHub Stars などでかなり盛り上がっていて、出荷も増え、SNS でもよく見かけるようになっていたことです。そこも一因でしょう。ハッキングされるのは、何らかの意味で成功しているからです。成功が見えた瞬間に、標的にもなる。PR のバックドア未遂と、その後のフィッシングを並べると、裏で 2 週間くらいかけて準備されていたとしても不思議ではありません。

つまり、これはその辺の誰かが適当に AI にコードを書かせて突っ込んできたような話ではなく、かなり組織的だということです。ここ数か月、こうしたフィッシングやサプライチェーン攻撃は本当に増えています。誰かがトークンを奪い、NPM パッケージに postinstall のようなライフサイクルスクリプトを仕込み、それが実行されて環境変数を抜いたり、システム全体を侵害したりする。攻撃そのものは単純です。

だからこそ腹立たしいんです。こういう細工は検出がとても簡単です。package.json を一つ見れば気づけるような話で、極端に言えば小さなモデルでも判定できるレベルです。それなのに、パッケージマネージャ側で十分な対策ができていない。Abhi が言うように、NPM はセキュリティ面で大きくこぼしてきたと思いますし、GitHub にも同じことを感じます。

ただ、最近は彼らも自分たちの役割を少しずつ認め始めていて、セキュリティ面でも開発者体験の面でも、公開側に優しいソフトウェアにしていかなければならないと分かってきているようには見えます。でも、攻撃者は一回当てればいいだけで、こちらは常に全部を守り続けなければならない。そこが根本的に不利です。

しかも彼らは AI をいくらでも使えます。フィッシングリンクを作ることも、社内の情報やユーザーを探ることも、かなり高度にできます。

**(Shane)** Microsoft Teams そっくりのアプリを作ることだって、昔ならかなり UI を作り込む必要があったのに、今なら一発でそれっぽいものを作れてしまいます。だからフィッシングは以前よりもずっと本物らしく感じられるようになっています。

**(Ismael)** 私は、あの怪しい PR のコントリビュータも追跡してみました。その人物は他のもっと大きなオープンソースプロジェクトにも似た PR を送っていました。つまり、それらも標的リストに入っている可能性があります。もし自分がその立場なら、まず CI/CD パイプラインを全部見直します。問題は、信頼されたコントリビュータが侵害されたらどうするのか、ということです。

CI/CD は、誰かが侵害された前提で設計されているでしょうか。PR に見える異常を検知できるでしょうか。そうした PR は、相手が CI/CD の権限や挙動を探るためのプロービングかもしれません。何ができて、何ができないかを測っている可能性がある。だから、そうしたシグナルは本気で受け取ったほうがいいです。

### いま必要なのは「侵害された信頼者」を前提にした設計

**(Ismael)** 今日聞いた別の話では、あるオープンソースベンダが、公開リポジトリを引っ込めて非公開にし始めたそうです。

**(Shane)** slot issue や slot PR の問題に加えて、本物のセキュリティ問題まであるので、オープンソースのメンテナンスは今まで以上に難しくなっています。

**(Ismael)** ちょうど IBM が、AI 時代のオープンソースの未来を作り直すために 50 億ドルを投じる、という話も出ていました( [アナウンス](https://newsroom.ibm.com/2026-05-28-ibm-and-red-hat-commit-5-billion-to-redefine-the-future-of-open-source-in-the-ai-era) )。

![](https://static.zenn.studio/user-upload/f2383ee775e3-20260627.png)

Red Hat は Linux の世界で長くこうしたセキュリティをやってきましたが、その知見が一般のオープンソース全体にはまだ十分に降りてきていません。だから今は、もっと大きなプレイヤーも、小さなプレイヤーも、ダウンロード数の多い有名メンテナも、全体のサプライチェーンを強化するための組織的な取り組みを始めるべきだと思います。どこか一か所が壊れれば、全員が危険にさらされるからです。

### レジストリもスキャナも必要だが、今の形には歪みがある

**(Abhi)** NPM に対しては本当に不満があります。あの日のサポートチケットも、返事が来たのは 6 時間後でした。publisher 向けの SLA がなくても、あれだけ多くのダウンロードを動かしているなら、何らかの対応保証があるべきです。影響を受けるのは API キーだけではありません。利用者のマシンにはもっといろいろなものが載っているんです。

**(Ismael)** しかも easy-day-js はいまだにレジストリ上に残っていました。2 週間も前から存在しているのに、そのままです。

**(Abhi)** 私もその日に報告しましたが、まだ残っている。

**(Ismael)** これは NPM だけの問題ではなくて、PyPI でも似たようなものです。私が 6 か月前に報告した悪意あるパッケージにも、返答が来たのは 2 週間前でした。要はビジネスモデルの問題だと思います。持続可能な形でセキュリティに投資できていない。

**(Shane)** ある程度のダウンロード規模に達したら、こちらとしては有償でもいいから強力な運用ツールや、いざという時に対応してくれる人たちへのアクセスを買いたいと思います。NPM のダウンロード統計ですら、Mastra の利用状況を見るのに使っているのに、平気で数日落ちることがあります。インフラとして弱いんです。

もちろん、こうしたパッケージ管理システムが大きな価値を提供してきたこと自体には感謝しています。でも、これだけ普及していて重要な基盤になっている以上、もっと高い期待を持たざるを得ないし、現状はその期待に届いていません。

**(Abhi)** しかも、攻撃者は見た目までかなり本物らしく整えます。バージョン番号も 0.0.1 のような怪しいものではなく、長く存在しているパッケージのように見せる。ドキュメントも多言語でそろえる。dayjs からいろいろ盗みつつ、正当なパッケージのように見せてきます。

![](https://static.zenn.studio/user-upload/509f13985e57-20260627.png)

**(Ismael)** さらにおかしいのは、レジストリ画面には Socket で解析するボタンがあるのに、すでに危険フラグが立っていること自体は前面に出ていない点です。そこまでできているなら、せめて「これは危険です」と表示すべきでしょう。

**(Shane)** 一方で、今回それに気づけたのは Socket のようなツールのおかげでもありました。定期スキャンをしているベンダがいて、その通知で問題を知ることができました。私が X で見たのも、発生から 1 時間後くらいです。アメリカ時間では深夜で、Abhi はフランスで寝ていましたが、それでもチームを起こして、できるだけ早く動けました。こういう通知システムは、オープンソースにとって本当に価値があります。Abhi は、あの日は本当に飛び起きたそうです。

**(Abhi)** 嫌な予感で目が覚めて、スマホを見ると Shane から「やられた」と連絡が来ていた、と。

**(Ismael)** その日の X を見ていて感じたのは、セキュリティ企業の売り方には恐怖の増幅がかなりあるということです。人はハックされるのが怖い。だから、誰かが被害に遭うと、それを大きく拡散して販売につなげる。ビジネスとしては理解できます。でも実際に見たのは、かなりの誤情報でした。何が起きたのか、どう起きたのか、誰が侵害されたのか。その理解が雑なまま広まっていたんです。

**(Shane)** たとえば「もう活動していない古いアカウントがやられた」といった説明も見ましたが、実際にはその人は今も現役のチームメンバーで、その日もコードを書いていました。つまり、現役の信頼されたコントリビュータが侵害されたわけです。

**(Ismael)** 他のメンテナからも似た話を聞きます。スキャナが検出して公開すること自体は悪くない。でも、そのやり方次第では、夜に趣味で OSS を守っているようなメンテナに大きなストレスと悪評を押しつけることになる。もっと滑らかに、丁寧にできるはずです。売りたい事情があるのは分かるけれど、それでもやり方はある。

**(Shane)** こちらとしても、投稿する前に一声かけてくれてよかったのに、とは思います。もちろん公表自体は必要ですし、そういうスキャナやツールが存在すること自体は重要です。ただ、どう伝えるかは別問題です。

**(Abhi)** 怪しい PR が来ていたので、今は Superagent を使っています。今日の PR でも、P1 と P2 のセキュリティ問題をちゃんと拾ってくれました。CodeRabbit も好きだけれど、Superagent はこうしたセキュリティ観点にかなりフォーカスしています。
