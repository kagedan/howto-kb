ClaudeなどのAIエージェント主導の実装方法を導入してしばらく経ち、思い知らされることがあります。

それは、**「人間にとって読みやすいコード」と「AIにとって読みやすいコード」は、必ずしも一致しない**ということです。

これまで私たちは未来の自分や同僚が読めることが重要視され、そのために可読性を意識してきました。  
しかしAIエージェントが自律的にコードを読み書きする今、リポジトリの最大の読み手はAIになりつつあります。

本記事では、AIのハルシネーションを防ぎ、期待通りのコードを生成させるための「AI向けコード設計・リポジトリ構成」のプラクティスをいくつか紹介します。

## 1. 命名規則は「短さ」より「説明」へ

人間は前後の文脈やファイルツリーから「この関数が何をするか」を推測できますが、AIはプロンプトに渡されたテキストが全てです。

少し冗長でダサく見えても、名前自体に「ドメイン知識」が詰め込まれている方が、AIは関連ファイルを正確に検索し、意図を汲み取ったコードを生成してくれます。

```
// ❌ 従来（人間向け）：短いが、これだけ渡されてもAIは推測が必要
function processData(data: any) { ... }
class UserManager { ... }

// ⭕️ AI向け：関数名やクラス名がそのままプロンプトになる
function calculateTotalOrderPriceIncludingTaxes(orderItems: OrderItem[]) { ... }
class UserAuthenticationAndSessionManager { ... }
```

名前が長いことは、IDEの保管機能などを使えば人間にとってもそこまで苦になりません。「関数名＝AIへの指示書」という意識を持つことが重要です。

## 2. 抽象化（DRY原則）の罠

「同じコードは2度書かない（DRY原則）」はコーディングの基本ですが、AI駆動開発においては、これが仇になることがあります。

処理を共通化するために継承を深くしたり、ファイルを細かく分割しすぎたりすると、AIが1つの機能を理解するためにあちこちのファイルを参照しなければなりません。

結果として、コンテキストの上限に達したり、途中で文脈を見失ったりします。

「多少コードが重複していても、**1つのファイルまたはコンポーネント内でロジックが完結している**」方が、AIはバグのない完璧な差分を生成しやすい傾向にあります。

## 3. ドキュメントは「Notion」ではなく「コードの真上」に書く

AIにプロジェクトの仕様を理解させる際、外部のWikiツール（NotionやConfluenceなど）に仕様書をまとめていると、AIエージェントがそこまで情報を拾いに行けない。あるいは最新状態を同期しづらい。という問題が発生します。

「コードのすぐ隣に、**自然言語のコンテキストを置いておく**」ことで、AIはコードの意図を正確に読み取ります。

### JSDoc / Docstrings をフル活用する

関数の真上に仕様を書くことで、AIがその関数を利用・修正する際の精度が向上します。

```
// ⭕️ AIはコメントも「コンテキスト」として読み取る
/**
 * ユーザーの決済処理を行い、失敗した場合は3回までリトライする。
 * Stripe APIのエラーハンドリングを必ず行うこと。
 * * @param userId - 決済を行うユーザーのID
 * @param paymentMethodId - 登録済みの決済手段ID
 * @returns 決済成功時はTransactionオブジェクトを返す
 */
async function processPaymentWithRetry(userId: string, paymentMethodId: string) { ... }
```

### ディレクトリ直下の `README.md`

また、機能ごとのディレクトリ（例: `/src/features/auth`）の直下に `README.md` を配置し、そのモジュール特有のルールやアーキテクチャをテキストで残すのも、AIエージェントに全体像を把握させるための手段です。

## 4. 型（Type）は最大のプロンプト

TypeScriptなどの静的型付けは、人間にとっての安全網である以上に、「AIに対する強めの制約」として機能します。

`any` や `Record<string, unknown>` を放置することは、AIに「自由に解釈していいよ」と丸投げしているのと同じです。

```
// ❌ AIが取り得る値を推測してしまう（ハルシネーションの元）
let orderStatus: string;

// ⭕️ AIに「この4つの状態しか存在しない」ことを強制する
type OrderStatus = 'pending' | 'shipped' | 'delivered' | 'canceled';
let orderStatus: OrderStatus;
```

厳密な型定義、ユニオン型、リテラル型を使って「取り得る値の境界」を明確に示しておくことで、AIの生成コードの品質はかなり安定します。

## 5. 【おまけ】テストコードはAIのための仕様書

人間にとってテストコードは「壊れていないことの証明」ですが、AIにとっては「**そのコードがどう振る舞うべきかの具体的なユースケース**」になります。

AIに機能の修正やリファクタリングを依頼する際、充実したテストコードが存在していれば、AIは「これらのテストを通るようにコードを書けばいいんだな」と明確なゴールを持つことができます。  
TDD（テスト駆動開発）は、人間以上に「AIエージェントを駆動させる」ために有効なアプローチとなるのかなと思います。

## 6. AI-Readableなコードのポイント(まとめ)

私たちがAIを使って実装をする際、意識すべき設計の要点をまとめると

* 命名: 「短さ」よりも「説明力」。名前そのものをAIへのプロンプトにする。
* 構造: DRY（共通化）にこだわりすぎず、AIが文脈を追いやすい「凝集度」を優先する。
* 情報: 仕様は外部ツールではなく、コード内のコメントやREADMEに記述し、AIに認識させる。
* 制約: 厳密な型定義とテストコードを「ガードレール」として、AIの生成品質を安定させる。

## おわりに

これからのコード実装は「AIに対するプロンプトの一部」へと役割が変わっていくのかなと考えています。少なくとも数年先の未来ではそうなっていると思います。

**AIが自走しやすいリポジトリ**（＝**AIがコードを読みやすい環境**）を設計することが、結果的に人間にとっても認知負荷の低い、堅牢なシステムへとつながるはずです。
