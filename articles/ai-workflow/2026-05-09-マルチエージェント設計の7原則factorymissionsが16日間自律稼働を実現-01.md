## TL;DR

* **人間の注意力がボトルネック**：今日の LLM は 50 のタスクを解決できる水準に達しているが、その監督に使える人間の帯域が足りない（意訳、`[01:28]`）
* **3 役割分担が基盤**：Orchestrator（計画）・Worker（実装）・Validator（検証）の責務境界を厳密に分離し、構造化ハンドオフで状態を引き継ぐ
* **検証コントラクトは実装前に書く**：「実装後に書かれたテストは決定を追認するだけ」（意訳、`[06:25]`）。Orchestrator がコントラクトを生成し、Validator が独立評価する 2 フェーズ分離が必須
* **Validator は 2 系統に分ける**：Scrutiny Validator（静的解析・コードレビュー）と User Testing Validator（UI 操作・E2E）を分離し、実装コードを事前に見せない敵対的設計でドリフトを防ぐ
* **書き込みは逐次、読み取りは並列**：競合状態を設計段階で排除し、正確性を複利的に積み上げる逐次実行戦略が数日間のタスクでは決定的に有効
* **Droid Whispering で訓練バイアスを回避**：役割ごとに最適な LLM を選択し、検証フェーズでは実装フェーズと異なるプロバイダーを使う（意訳、`[11:51]`）
* **オーケストレーションはプロンプト駆動 700 行**：ハードコード状態機械ではなくプロンプトとスキルで定義することで、4 文の変更で実行戦略を大幅に変えられる（意訳、`[15:08]`）

**対象：**

* LangChain / LangGraph / CrewAI / AutoGen などを触ったことがあり、マルチエージェントシステムを自作・業務適用したいソフトウェアエンジニア・ML エンジニア

* モデルのベンチマーク比較・コスト詳細

## 背景：なぜ単一エージェントでなくマルチエージェントか

### 人間の注意力がボトルネックになっている

> 「今日のモデルはこれら 50 のタスクすべてを解決できるほど賢いですが、その実装を監督するための人間の帯域が不足しています。」（意訳、`[01:28]`）

この発言の含意は重い。問題は LLM の能力ではなく、**人間の監督帯域**だという主張だ。

単一エージェントに長時間タスクを任せると、人間は「ちゃんと動いているか」という監視役に張り付かなければならない。しかしその監視コストが高いため、委譲できる作業量に上限が生じる。マルチエージェント構成は、この監視役の一部をシステム内部に内製化することで、人間の注意を解放する設計思想だ。

### マルチエージェントの 5 類型

Factory の Luke Alvoeiro は、マルチエージェントシステムを以下の 5 パターンに分類している（意訳、`[02:04]`）。

| パターン | 概要 | 主な課題 |
| --- | --- | --- |
| **委譲 (Delegation)** | 親エージェントが子エージェントにタスクを依頼する | サブタスクの粒度設計 |
| **作成者-検証者 (Creator-Verifier)** | 実装者とは異なるエージェントが検証する | バイアスのない分離設計 |
| **直接通信 (Direct Communication)** | コーディネーターなしでエージェント同士が通信する | 状態の断片化 |
| **交渉 (Negotiation)** | 共有リソースをめぐってエージェントが調整する | デッドロック・競合 |
| **ブロードキャスト (Broadcast)** | 1 つのエージェントが多数に情報を一斉送信する | 一貫性の維持 |

Missions はこのうち **委譲・作成者-検証者・ブロードキャスト・交渉の 4 類型を統合**した設計だ。自分のシステムがどの類型の組み合わせかを明示することは、設計ドキュメントの初期整理として有効だ。

## 全体アーキテクチャ：Orchestrator / Worker / Validator

### 3 役割の責務境界

各役割の責務を整理する（意訳、`[04:03]`）。

**Orchestrator（計画担当）**

* 要件を明確化し、マイルストーンに分割する
* 各マイルストーンに対して「検証コントラクト」を生成する
* 検証失敗を受けて是正フィーチャーを計画し、再度 Worker に渡す

**Worker（実装担当）**

* クリーンなコンテキストで単一機能の実装に集中する（他のマイルストーンの状態を引き継がない）
* 実装完了後に Git でコミットする
* 後続エージェントへの「構造化ハンドオフ」を記録する

**Validator（検証担当）**

* 実装コードを事前に参照しない「敵対的」設計で検証する
* Scrutiny Validator と User Testing Validator の 2 系統に分かれる（後述）

## 実装・主要ステップ

### ステップ 1：検証コントラクトを実装前に書く

> 「実装後に書かれたテストはバグを捕まえません。それらは決定を追認するだけです。」（意訳、`[06:25]`）

この問題は TDD を実践しているチームでも再現する。実装と同一の開発者がテストを書く限り、テストは実装者の認知バイアスを引き継ぐ。

Missions が採用するのは、**Orchestrator が計画フェーズで検証コントラクトを生成し、Worker の実装開始前にその内容を確定する**という設計だ。

```
# 検証コントラクト（疑似スキーマ）の概念例
verification_contract:
  milestone_id: "feature/user-authentication"
  assertions:
    - type: behavior
      description: "ログイン成功時に JWT トークンが返却される"
      expected_output: "status: 200, body.token: <non-empty string>"
    - type: behavior
      description: "不正パスワードでログイン失敗時に 401 が返却される"
      expected_output: "status: 401, body.error: <non-empty string>"
    - type: coverage
      min_line_coverage: 90
    - type: static_analysis
      no_lint_errors: true
      type_check_passes: true
  created_by: orchestrator
  created_at: <plan_phase>   # Workerが実装を開始する前に確定
```

2 フェーズ分離のポイントは「コントラクトを生成するのは Orchestrator、評価するのは Validator」という主体の分離にある。同一エージェントが生成と評価を兼ねると、バイアスの問題が再発する。

### ステップ 2：Scrutiny Validator と User Testing Validator を分離する

2 種類の Validator はそれぞれ異なるレイヤーを担当する（意訳、`[06:06]`）。

**実装上の重要な制約：Validator に実装コードを事前に見せない**

「敵対的」と呼ばれる設計の本質は、Validator が実装フェーズの思考過程に汚染されないことだ。実装上の分離方式として以下が考えられる。

* **コンテキスト分離**：Validator のセッションには Orchestrator から受け取った検証コントラクトと Git コミット後のコードのみを渡す（実装 Worker の会話履歴を引き継がない）
* **別セッション起動**：Worker セッションとは独立した新規セッションで Validator を起動する
* **検証コントラクト参照の強制**：Validator のシステムプロンプトに「コードを読む前に検証コントラクトを確認し、期待される振る舞いを把握してから評価せよ」と明示する

### ステップ 3：構造化ハンドオフの 5 フィールドを設計する

Worker はタスク完了時に後続エージェントへの構造化ハンドオフを記録する（意訳、`[08:15]`）。

```
# 構造化ハンドオフ（疑似スキーマ）の概念例
structured_handoff:
  completed:
    - "ユーザー認証エンドポイント (/api/auth/login) の実装"
    - "JWT トークン生成ロジックの追加"
    - "単体テスト 12 件の追加（カバレッジ 94%）"
  not_completed:
    - "リフレッシュトークンのローテーション処理（次のマイルストーンのスコープ）"
  commands_executed:
    - "npm test -- --coverage"
    - "git commit -m 'feat: add user authentication endpoint'"
  issues_found:
    - "bcrypt のハッシュ化コストが開発環境では低すぎる設定になっている（本番設定は別途確認要）"
  procedure_compliance:
    - "検証コントラクトの全アサーションを実装前に確認した"
    - "単一機能のスコープ外の変更は行わなかった"
```

このスキーマが長期稼働の信頼性に貢献するのは、**各フィールドが後続エージェント（Validator と次の Worker）が必要とする情報を完全に網羅**しているからだ。`not_completed` フィールドが特に重要で、スコープ外の処理を明示することで、次の Worker が知らずに重複実装したり、未処理のまま進んだりするリスクを排除する。

## 検証・落とし穴

### 落とし穴 1：並列実行でコンテキストが競合する

並列 Worker を起動してスループットを上げようとすると、以下の問題が発生しやすい。

* **マージコンフリクト**：2 つの Worker が同じファイルを同時期に変更する
* **状態不整合**：一方の Worker が作成したインターフェースを、もう一方の Worker が古い仕様のまま呼び出す
* **デバッグ困難性**：どの Worker の変更が問題を引き起こしたか特定しにくくなる

Missions が採用する原則は「**書き込み操作は逐次、読み取り操作は並列**」だ（意訳、`[09:21]`）。

```
# 逐次 vs 並列の選択基準
逐次実行すべき操作:
  - 機能の実装（ファイル書き込み、コミット）
  - 検証コントラクトの評価
  - ハンドオフの記録

並列実行してよい操作:
  - コードベースの検索・読み取り
  - API ドキュメントの調査
  - テストケースの列挙（書き込み前の設計段階）
```

「紙の上では遅く見えるが」という表現が講演中に登場する（意訳、`[09:21]`）。しかし数日間にわたるタスクでは、エラー率の低下が複利的に効いてくる。1% のエラー率が 100 ステップあれば、全工程が成功する確率は 36% に落ちる。逐次実行でエラー率が 0.1% になれば、同じ 100 ステップでも 90% 以上の成功率を維持できる。

### 落とし穴 2：検証は一度で成功しない前提を持たずに設計する

Slack クローンの構築事例では、**検証は一度で成功することはほとんどなく、常にフォローアップの是正フィーチャーが必要になった**（意訳、`[13:06]`）。

この事実を設計に組み込まずに「検証が通ったら完了」という線形フローで実装すると、是正フィーチャーの発生時にシステムが止まる。

Missions の自己修復ループは以下のように動作する。

重要なのは、このループが例外処理ではなく**通常フローとして設計されている**点だ。時間の 60% が実装に、残りが検証・是正サイクルに費やされるという実績からも（意訳、`[13:06]`）、是正ループを前提に工数を見積もる必要がある。

### 落とし穴 3：検証コントラクトをテストコードと混同する

検証コントラクトは「何が正しい状態か」のアサーション集であり、テストコードそのものではない。テストコードは Worker が実装するが、検証コントラクトは Orchestrator が計画フェーズで生成する。

この区別を曖昧にすると、Worker がコントラクトの意図を解釈してテストを書くことになり、「決定の追認」に戻ってしまう。Orchestrator が生成したコントラクトをそのまま Validator に渡し、Validator がそれを基準に独立評価するという経路を崩さないことが重要だ。

### 落とし穴 4：Validator に実装 Worker のコンテキストを渡す

Validator を「強力なエージェントなので何でも渡せばよい」として Worker のコンテキスト全体を渡すと、敵対的設計が崩壊する。実装 Worker が「このコードはこういう意図で書いた」という文脈を持っていると、Validator はその文脈に引きずられて検証基準を甘くしてしまう。

実装上の対策：Validator のシステムプロンプトには「Workerの実装意図の説明は参照しない。検証コントラクトと最終成果物のみを評価の根拠とせよ」と明示する。

## Droid Whispering：役割別モデル選択

> 「このスキルを我々は内部で『ドロイド・ウィスパリング』と呼んでいます。」（意訳、`[11:51]`）

計画・実装・検証といった各役割には、それぞれ異なる能力を持つ LLM が最適だという観点だ（意訳、`[11:22]`）。

この設計には 3 つの実装上の論点がある。

**1. 訓練データバイアスの回避**

同一プロバイダーのモデルで実装と検証を行うと、そのモデルが学習したコードパターンのバイアスを両方が共有する可能性がある。検証フェーズで実装フェーズとは異なるプロバイダーを使うことで、訓練データバイアスに基づく検証スルーを構造的に回避できる。

**2. コスト最適化**

役割ごとに必要な能力が異なる。

```
役割別モデル選択の考え方（例）:
  Orchestrator:
    要件: 長文理解・計画立案・コントラクト生成
    優先: 推論能力 > コスト
  Worker:
    要件: コード生成・特定言語の慣用的パターン
    優先: コーディング特化 > 汎用推論
  Scrutiny Validator:
    要件: コードレビュー・バグ発見・セキュリティ分析
    優先: 実装とは異なるプロバイダー
  User Testing Validator:
    要件: UI 操作・自然言語での動作確認
    優先: マルチモーダル能力（スクリーンショット解釈等）
```

**3. モデル非依存アーキテクチャの採用**

特定プロバイダーに依存したアーキテクチャは、モデルの性能更新時に全体を再設計するコストが発生する。役割別にモデルを差し替えられる構造を持つことで、部分的なアップグレードが可能になる。

## プロンプト駆動 700 行アーキテクチャ

> 「このシステムの失敗の分解方法やハンドリング方法は、すべて約 700 行のテキストに記述されており、そのうちの 4 文で実行戦略を劇的に変えることができます。」（意訳、`[15:08]`）

Missions のオーケストレーションロジックのほとんどはハードコードされた状態機械ではなく、プロンプトとスキルで定義されている。

この設計の実装上の含意は 2 点ある。

**メンテナンスコスト**：失敗ハンドリングや実行戦略の変更がコードの変更ではなくプロンプトの変更として実施できる。新しいモデルへの対応も、プロンプトの調整で吸収できる範囲が広い。

**規模感の参考値**：「700 行・4 文の変更で戦略が変わる」という数値は、プロンプトエンジニアリングの工数見積もりにおける参照点になる。ただしこれは Missions の実装値であり、自分のシステムの規模に直接あてはまるわけではない。

## 実績数値の整理

Slack クローン構築ミッションで確認された数値（意訳、`[13:06]`・`[08:15]`）。

| 指標 | 値 | 備考 |
| --- | --- | --- |
| 最長稼働期間 | 16 日間 | 人間の介入なしの自律稼働実績 |
| テストコード比率 | 50% | 最終的なコードベース全体に占める割合 |
| テストカバレッジ | 90% | Slack クローン構築ミッション |
| プロンプト規模 | 約 700 行 | オーケストレーションロジック全体 |
| 実行戦略変更に必要な変更量 | 4 文 | 講演時点での実績値 |

## まとめと次に試すこと

### 設計原則の再整理

Missions の設計から再利用できる原則を整理する。

1. **役割分離の厳密化**：Orchestrator・Worker・Validator の責務境界を設計ドキュメントに明文化してから実装に入る
2. **検証コントラクトの時系列を逆転させる**：実装前にコントラクトを確定し、生成者（Orchestrator）と評価者（Validator）を分離する
3. **書き込みを逐次化する**：機能実装のアトミックな単位を 1 Worker に担当させ、競合状態を設計段階で排除する
4. **是正ループを通常フローに組み込む**：検証失敗を例外ではなくフローの一部として扱い、工数見積もりに反映する
5. **コンテキスト汚染を防ぐ**：Validator には Worker のコンテキストを渡さず、検証コントラクトと成果物のみを渡す

### 次に試す 3 ステップ

1. **検証コントラクトをひとつ書いてみる**：次の開発タスクで、実装を開始する前に「完了状態のアサーション集」を箇条書きで作成し、実装後の自己評価に使う
2. **構造化ハンドオフの 5 フィールドを実装する**：既存のエージェントに `completed` / `not_completed` / `commands_executed` / `issues_found` / `procedure_compliance` の 5 フィールドを出力させ、ログを確認する
