kaji（舵）をオープンソースとして公開しました。

リポジトリはこちらです。

<https://github.com/apokamo/kaji>

`kaji` は、AI駆動開発における設計・実装・検証の流れを分断せずに回しやすくするためのハーネスです。ここでいうハーネスは、単なる便利スクリプト集ではなく、開発フローを一定の型に乗せ、再実行ややり直しも含めて扱いやすくするための実行基盤を指しています。

この記事では、`kaji` を作った背景、何を解決したかったのか、どんな構成で設計したのかを紹介します。

## なぜ作ったのか

最初に作っていたのは、もっと限定的なものでした。もともとは GitHub Issue を起点に、調査、設計、実装、レビュー、PR 作成までを段階的に進める AI 駆動のバグ修正ワークフローを回していました。

ただ、実際に運用してみると難しかったのは「各ステップで AI に何をさせるか」よりも、「ステップ間をどうつなぐか」でした。

* 調査の結果を次の設計にどう渡すか
* レビュー結果をどの形式で受け取り、次の実装や再修正にどう反映するか
* 途中で失敗したときに、どこから再開できるようにするか
* ツールやモデルが違っても、フロー全体をどう安定して回すか

初期の設計でも、この問題はかなり前面に出ていました。特に大きかったのは、ステートごとの入出力やレビュー判定のインターフェースが揃っていないと、ワークフロー全体がすぐ不安定になることでした。

つまり必要だったのは、強い AI モデルを個別に呼ぶことではなく、設計・実装・検証の流れを一定の契約で接続し、途中失敗ややり直しを含めて扱える土台でした。`kaji` はそのために切り出したハーネスです。

## `kaji` は何をするハーネスか

いまの `kaji` は、Claude Code / Codex / Gemini CLI のようなエージェント系 CLI を、ワークフロー YAML に従って順番に実行する軽量なハーネスです。

役割をかなり意図的に絞っています。

* ハーネスは「何をどの順で実行するか」を管理する
* 各ステップの実作業はスキルに持たせる
* スキルのロード、プロジェクト文脈の読解、実際のツール呼び出しは各 CLI に委譲する

この分離にしている理由は、ハーネス側が賢くなりすぎると、すぐに巨大なオーケストレーターになって保守しづらくなるからです。`kaji` はあくまでフロー制御、状態管理、再開、判定結果の受け渡しに責務を絞っています。

その結果、`kaji` は単独で何かを解決するツールというより、

* ワークフローを定義する YAML
* 各ステップの実務を担うスキル
* 実際に動く AI CLI

の間をつなぐ実行基盤になっています。

## どんなフローをつなぐのか

典型的には、Issue 駆動の開発フローをつなぎます。

たとえば `kaji` では、次のような流れを 1 つのワークフローとして表現できます。

1. Issue を作る
2. 作業用 worktree を切る
3. 設計書を書く
4. 設計をレビューする
5. 実装する
6. コードレビューする
7. 指摘があれば修正する
8. 検証して PR を作る

重要なのは、これを単なる直列実行ではなく、`review -> fix -> verify` のようなループを含めて扱えることです。AI 駆動開発では、一発で完成するよりも、レビュー結果を受けて修正し、必要なら設計まで戻る、という往復のほうが本質的です。

`kaji` はその往復を YAML で宣言し、各ステップの verdict を次の遷移に接続できるようにしています。これによって、設計・実装・検証を別々の作業として並べるのではなく、1 本の再実行可能なフローとして扱えます。

また、フロー全体を単一のツールに固定しないのも `kaji` の特徴です。たとえば調査は Gemini、レビューは Codex、実装は Claude Code のようにステップごとに混在させられます。さらに、モデルや reasoning effort、resume 継続の有無もステップ単位で切り替えられるので、工程ごとに必要な性質を揃えやすくしています。

## 設計上の考え方

`kaji` を設計するときに意識したのは、AI を中心に据えることよりも、壊れにくいフローを作ることでした。

特に重視したのは次の4点です。

### 1. ハーネスは遷移だけを見る

ハーネス自身がスキルの中身まで理解し始めると、すぐに責務が膨らみます。そこで `kaji` では、ハーネスはワークフロー定義を読み、次にどのステップへ進むかだけを担当します。

設計書を書く、コードを直す、レビューする、といった実作業はスキルに寄せています。これによって、ハーネス本体はできるだけ薄く保ちつつ、プロジェクトごとの差分はスキル側で吸収できます。

### 2. CLI が持つ機能を再実装しない

Claude Code や Codex のような CLI は、もともとコンテキスト管理、スキル読み込み、ツール呼び出しの仕組みを持っています。ここを外側から Python で再実装し始めると、追従コストが高くなります。

そのため `kaji` は、CLI のネイティブ機能をできるだけそのまま使う方向に寄せました。ハーネスは外から順序を制御し、各ステップの実行は CLI に任せる、という分担です。

### 3. 途中失敗を前提に、再開可能にする

AI 駆動開発では、途中で止まること自体は珍しくありません。レビュー差し戻し、出力フォーマットの揺れ、実装途中での見直しなど、むしろ途中停止が普通に起こります。

なので `kaji` では、一回で最後まで流れることよりも、どこで止まったかがわかり、必要な地点から再開できることを重視しています。ワークフローを一発実行の自動化ではなく、再実行可能な運用フローとして扱う設計です。

このあたりはログ設計にも強く効いています。出力の取りこぼしや verdict 判定の失敗を追うために、かなりの回数テストしながらログの出し方を詰めました。現行の `kaji` では、`stdout.log` に生の出力、`console.log` に人が追いやすいテキスト、`run.log` に workflow 単位の JSONL を分けて出していて、あとから原因を追えることをかなり重視しています。

このサンプルワークフローでは、`review` と `verify` の役割を分けています。`verify` は新規の指摘を積極的に探す工程ではなく、直前のレビューで出た指摘が解消されているかを確認する工程として寄せています。毎回ゼロから再レビューさせると、AI が新しい指摘をひねり出してループが終わらなくなることがあったためです。

そのため reviewer 側は、`review` と `verify` を同じコンテキストで継続できるほうが自然だと考えています。こうした切り方を workflow 側で調整しやすいのも `kaji` の強みです。

### 4. AI の出力揺れを前提に、多段フォールバックで判定する

ステップ間をつなぐには、判定結果の形式をある程度揃える必要があります。一方で、AI の出力はどうしても揺れます。そこに過度に厳密な parser だけを置くと、現実の運用ではすぐ壊れます。

そのため `kaji` では、verdict という共通の出力契約を置きつつ、判定は一発で決め打ちしないようにしています。まずは厳密に取り、それで拾えなければ緩めに拾い、それでもだめなら AI に `VERDICT` を再出力させる、という多段のフォールバックを前提にしています。

理想的な出力だけを前提にせず、現実に起こる delimiter の揺れやフォーマット崩れを吸収しながら、次に進めるか、どこで止めるかを判断できることを重視しました。

要するに `kaji` の設計思想は、AI を賢く使うことより、AI を含む開発フローを継続的に運用できる形にすることにあります。

## OSSとして公開した理由

正直に言うと、`kaji` 自体はまだしょぼいです。完成されたプロダクトというより、実運用の中で必要になって作ってきた叩き台に近いです。

でも、こういうものは手元だけで閉じて磨いていても限界があります。しょぼいうちに外へ出して、いろんなすごい人たちに見てもらって意見をもらったほうが前に進めると思いました。

なので今回は、完成してから公開するのではなく、しょぼい段階でも OSS として出して、フィードバックをもらいながら育てていくことを優先しました。

そのうえで、同じように AI 駆動開発の運用フローをどう作るかで悩んでいる人にとって、何かの叩き台になればいいなとも思っています。

## 使い方

最小構成で使うなら、やることは大きく3つです。

1. 対象プロジェクトに `.kaji/config.toml` を置く
2. `workflows/*.yaml` でフローを定義する
3. 各ステップで使う skill を `kaji` の入出力契約に合わせる

`kaji` は、workflow と skill を組み合わせて使います。skill は新規に作ってもいいですし、既存のものを寄せていく形でも構いません。

`workflow` の書き方は `workflow-authoring.md` が一番まとまっています。  
<https://github.com/apokamo/kaji/blob/main/docs/dev/workflow-authoring.md>

skill 側の入出力契約や `VERDICT` のルールは `skill-authoring.md` を見るのが早いです。  
<https://github.com/apokamo/kaji/blob/main/docs/dev/skill-authoring.md>

最小構成でも、`review-code` と `verify-code` の reviewer 文脈をつなぐために `resume` を入れておくと、`kaji` らしい使い方が見えやすくなります。

最小構成のイメージはこんな感じです。

```
.claude/skills/
  implement/
    SKILL.md
  review-code/
    SKILL.md
  fix-code/
    SKILL.md
  verify-code/
    SKILL.md

.agents/skills/
  review-code -> ../../.claude/skills/review-code
  verify-code -> ../../.claude/skills/verify-code
```

実体は `.claude/skills/` に置き、`.agents/skills/` はそれを参照するシンボリックリンクとして扱うのが基本です。skill を agent ごとに二重管理するのではなく、1つの実体を共有するイメージです。ハーネスは skill の中身を読みません。`agent` と `skill` の組み合わせからパスだけを解決し、実行自体は各 CLI に委譲します。ただし、skill のファイル内容自体を解釈しない一方で、実行結果として出る `VERDICT` の出力契約には依存します。

最小の `workflow` は、たとえばこんな形です。コードを書いて、レビューし、指摘があれば修正し、同じ reviewer 文脈で再確認する流れだけに絞ると、`resume` のイメージも掴みやすいと思います。

```
name: minimal-code-review
description: "最小のコードレビュー付きフロー"
execution_policy: auto

steps:
  - id: implement
    skill: implement
    agent: claude
    on:
      PASS: review-code
      ABORT: end

  - id: review-code
    skill: review-code
    agent: codex
    on:
      PASS: end
      RETRY: fix-code
      ABORT: end

  - id: fix-code
    skill: fix-code
    agent: claude
    on:
      PASS: verify-code
      ABORT: end

  - id: verify-code
    skill: verify-code
    agent: codex
    resume: review-code
    on:
      PASS: end
      RETRY: fix-code
      ABORT: end
```

`resume` は、同じ agent の前段ステップのコンテキストを引き継いで続きから実行するための指定です。たとえば上の例では、`verify-code` が `review-code` の reviewer 文脈を引き継いだまま確認を続けます。

この最小例でも、`claude` と `codex` を混在させつつ、reviewer 側だけは `resume` でコンテキストを継続する形にしています。

また、skill 側では単に作業を指示するだけでなく、最後に verdict を stdout へ出せるようにしておく必要があります。ここが揃っていないと、ハーネスは次の遷移を判断できません。

たとえば `review-code` の `SKILL.md` はこんなイメージです。

```
---
name: review-code
description: "コードレビューを実施し、verdict を返す"
---

# Review Code

(ここにレビュー手順を書く)

---VERDICT---
status: PASS
reason: |
  レビュー対象のコードは要件を満たしている。
evidence: |
  - 主要な差分を確認した
  - テスト結果を確認した
suggestion: ""
---END_VERDICT---
```

`fix-code` 側は、レビュー結果を受けて修正し、再確認へ進めるかを返す skill になります。

```
---
name: fix-code
description: "レビュー指摘をもとにコードを修正する"
---

# Fix Code

(ここに修正手順を書く)

---VERDICT---
status: PASS
reason: |
  指摘された問題を修正した。
evidence: |
  - 対象コードを修正した
  - 必要なテストを更新した
suggestion: ""
---END_VERDICT---
```

最小の実行はこれです。

```
kaji run workflows/minimal-code-review.yaml 57
```

途中から再開したい場合は、次のようにステップを指定できます。

```
kaji run workflows/minimal-code-review.yaml 57 --from fix-code
```

つまり `kaji` の導入は、「workflow ファイルを1つ書く」というより、「自分の開発フローに合わせて workflow と skill を一緒に調整する」に近いです。

このように `kaji` は、各 CLI を統一的に抽象化するというより、AI 駆動開発の各工程に対して、適切な agent と設定、適切な skill を割り当てながらフローを継続的に回すための土台として使うのが基本になります。

今回のサンプルワークフローでは、Issue 本文を仕様の起点、Issue コメントを長期記憶や履歴として使っています。ただし、これは `kaji` 全体の固定仕様ではありません。workflow と skill の設計次第で、ローカルの設計書や GitHub 上の仕様ファイルを起点にすることもできます。どの仕様ソースを正とするかをフロー側で決められる柔軟性も、`kaji` の強みのひとつです。

### 参考: 実際の `feature-development` ワークフロー

たとえば、実際に使っている `feature-development` ワークフローは次のような形です。設計レビューとコードレビューを分けつつ、レビュー系は Codex、修正系は Claude Code に寄せています。

```
name: feature-development
description: "設計→実装→コードレビューの標準フロー"
execution_policy: auto

cycles:
  design-review:
    entry: review-design
    loop:
      - fix-design
      - verify-design
    max_iterations: 3
    on_exhaust: ABORT
  code-review:
    entry: review-code
    loop:
      - fix-code
      - verify-code
    max_iterations: 3
    on_exhaust: ABORT

steps:
  - id: design
    skill: issue-design
    agent: claude
    model: opus
    effort: high
    on:
      PASS: review-design
      ABORT: end

  - id: review-design
    skill: issue-review-design
    agent: codex
    model: gpt-5.4
    effort: high
    on:
      PASS: implement
      RETRY: fix-design
      ABORT: end

  - id: fix-design
    skill: issue-fix-design
    agent: claude
    model: opus
    resume: design
    on:
      PASS: verify-design
      ABORT: end

  - id: verify-design
    skill: issue-verify-design
    agent: codex
    model: gpt-5.4
    effort: high
    resume: review-design
    on:
      PASS: implement
      RETRY: fix-design
      ABORT: end

  - id: implement
    skill: issue-implement
    agent: claude
    model: opus
    on:
      PASS: review-code
      RETRY: implement
      BACK: design
      ABORT: end

  - id: review-code
    skill: issue-review-code
    agent: codex
    model: gpt-5.4
    effort: high
    on:
      PASS: doc-check
      RETRY: fix-code
      BACK: design
      ABORT: end

  - id: fix-code
    skill: issue-fix-code
    agent: claude
    model: opus
    on:
      PASS: verify-code
      ABORT: end

  - id: verify-code
    skill: issue-verify-code
    agent: codex
    model: gpt-5.4
    effort: high
    resume: review-code
    on:
      PASS: doc-check
      RETRY: fix-code
      ABORT: end

  - id: doc-check
    skill: issue-doc-check
    agent: claude
    model: opus
    on:
      PASS: pr

  - id: pr
    skill: issue-pr
    agent: claude
    model: sonnet
    on:
      PASS: end
      RETRY: pr
      ABORT: end
```

この例で見てほしいのは次の点です。

* `agent` で `claude` と `codex` を混在できる
* `model` や `effort` をステップごとに変えられる
* `resume` で同一 agent の前段セッションを継続できる
* `cycles` で設計レビューとコードレビューの往復を宣言できる

## まとめ

`kaji` は、AI に何かを1回やらせるためのツールではなく、設計・実装・検証の流れを繰り返し回せるようにするためのハーネスとして作っています。

AI 駆動開発では、モデルの性能そのもの以上に、どの工程をどう接続し、どこでレビューし、失敗時にどう戻るかが重要になります。`kaji` はその部分を YAML、skill、CLI の分離で扱えるようにしたものです。

もし同じように、AI を使った開発を単発の補助ではなく運用可能なフローとして整えたいと思っているなら、`kaji` がその叩き台になるかもしれません。使ってみて気づいた点や改善案があれば、ぜひフィードバックをもらえると嬉しいです。
