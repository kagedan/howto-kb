## 🚀 はじめに

2026年4月8日、Anthropicは[Claude Managed Agents](https://claude.com/blog/claude-managed-agents)をパブリックベータとしてリリースしました。Anthropicが「**モデルAPI提供者」から「エージェント実行基盤提供者」へとポジションを転換する**戦略的な一歩と思われます。

従来のClaude APIが「一問一答の頭脳」だとすれば、Managed Agentsは「**自分でファイルを作り、コマンドを叩き、Webを調べ、成果物を納品してくれる作業者**」です。エージェント開発者がこれまで自前で構築していた——セキュアな実行環境、状態管理、ツール実行レイヤー、エラーリカバリ——これらすべてをAnthropicが引き受けるサービスです。

この記事では、公式エンジニアリング記事「[Managed agents: from prototype to production](https://www.anthropic.com/engineering/managed-agents)」と[公式APIドキュメント](https://platform.claude.com/docs/en/managed-agents/overview)をベースに、アーキテクチャの全貌を解説します。

## ⚠️ なぜManaged Agentsが必要なのか — エージェント開発の「見えないコスト」

### 🔍 エージェント開発の現状

LLMエージェントの開発において、**モデルのプロンプト設計はタスクの一部に過ぎません**。実際のエンジニアリング工数の大半は、モデルの周辺にあるインフラ層に費やされています。

以下の図は、エージェント開発における工数の内訳を示しています。開発者が意識するプロンプト設計やツール定義は全体のごく一部であり、実際の工数の大半はセキュリティ、状態管理、エラーリカバリといった「見えないコスト」に費やされます。

この図のポイントは、プロンプト設計とツール定義は全体の約20%に過ぎないという点です。残りの80%は、セキュリティ、状態管理、エラーリカバリ、認証、スケーリング、モニタリングといった、エージェントを本番運用するために不可欠なインフラ層です。

Managed Agentsは、この「見えないコスト」の80%を肩代わりすることで、開発者がプロンプト設計とツール定義に集中できる環境を提供します。「エージェントの頭脳を作る」ことに全力を注げる世界を実現するサービスです。

### 📊 Messages APIとの根本的な違い

ここで、既存のMessages APIとManaged Agentsの責任範囲の違いを整理しましょう。以下の図は、開発者とAnthropicの責任分担がどう変わるかを示しています。

この図を見ると、Messages APIでは開発者がエージェントループからエラーリカバリまですべてを自前で実装する必要があるのに対し、Managed Agentsでは開発者の責任がAgent定義とツール選択に限定され、残りはすべてAnthropicが引き受けることが分かります。

以下の表で、両者の違いをさらに具体的に比較します。

| 比較項目 | 📡 Messages API | 🤖 Managed Agents |
| --- | --- | --- |
| **位置づけ** | モデルへの直接アクセス | プリビルトのエージェントハーネス + マネージドインフラ |
| **最適用途** | カスタムエージェントループ、きめ細かい制御 | 長時間実行タスク、非同期作業 |
| **エージェントループ** | 自前実装が必要 | 組み込み済み |
| **ツール実行** | 自前実装が必要 | セキュアコンテナで自動実行 |
| **状態管理** | 自前実装が必要 | サーバーサイドで永続化 |
| **最適化** | 手動（キャッシュ等） | 自動（プロンプトキャッシュ、コンパクション） |
| **障害復旧** | 自前実装が必要 | セッションベースの自動復旧 |

> 💡 **ポイント：** Messages APIが「劣っている」わけではありません。エージェントループを完全にカスタマイズしたい場合はMessages APIが適切です。Managed Agentsは「そのカスタマイズが不要な場合に、インフラコストを丸ごと省略できるサービス」という位置づけです。

## 🏗️ 3コンポーネントアーキテクチャ — Harness / Sandbox / Session

Anthropicのエンジニアリング記事「[Managed agents: from prototype to production](https://www.anthropic.com/engineering/managed-agents)」は、内部アーキテクチャの核心を詳細に説明しています。Managed Agentsは**3つの仮想化されたコンポーネント**を明確に分離する設計です。

> 📖 用語について： エンジニアリング記事の原文では、3つのコンポーネントは **Harness（ハーネス）**、**Sandbox（サンドボックス）**、**Session（セッション）** と呼ばれています。本記事では直感的に理解しやすいよう、Harnessを🧠 Brain（脳）、Sandboxを🤲 Hands（手）と表記する場合があります。

### 🧩 3コンポーネントの概要

以下の図は、3つのコンポーネント間のデータフローを示しています。Harness=Brain（脳）が推論を担当し、Sandbox=Hands（手）がツール実行を担当し、Session（セッション）がすべてのイベントを永続的に記録します。

この図のポイントは、3つのコンポーネントの相互作用です。BrainはHandsに対してツール実行を依頼し、結果を受け取ります。同時に、すべてのイベントはSessionに記録され、Brainは必要に応じてSessionからイベント履歴を読み込むことができます。

各コンポーネントの特性をまとめると以下のようになります。

| コンポーネント | 🎭 役割 | 💎 特性 | 🔧 障害時の挙動 |
| --- | --- | --- | --- |
| **🧠 Brain** | Claudeによる推論・ツール呼び出し判断 | ステートレス | 新しいハーネスで再起動、セッションから状態復元 |
| **🤲 Hands** | コード実行・ファイル操作・外部API呼び出し | 交換可能 | エラーをツール呼び出し失敗としてBrainに返却 |
| **📋 Session** | 全イベントの永続的記録 | Append-only | 唯一の永続層、他コンポーネントの復旧基盤 |

### 🐄 設計の核心 — 「Cattle, not Pets」

この設計の本質を理解するには、DevOpsの世界で有名な「**Cattle, not Pets**」の概念が鍵になります。ペット（大切に育てる1台のサーバー）ではなく、家畜（いつでも交換可能な多数のサーバー）として扱う——BrainとHandsをまさにこの「家畜」として設計したのが、Managed Agentsの革新です。

> 🐄 エンジニアリング記事では、HarnessとSandboxに「Cattle, not Pets」パターンを適用した設計が紹介されています。コンテナの障害は「ツール呼び出しの失敗」としてClaudeに返され、Claudeが自律的にリトライを判断します。新しいHarnessは`wake()`で再起動し、Sessionからイベントログを取得して中断箇所から再開します（[Anthropic Engineering](https://www.anthropic.com/engineering/managed-agents)、筆者要約）。

以下のシーケンス図は、Brain #1がクラッシュした後、Brain #2が自動的に起動してセッションから状態を復元する流れを示しています。これがCattle設計の実際の動作です。

この図から分かるように、Brain #1がクラッシュしても、新しいBrain #2がSessionからイベント履歴を読み込むだけで、何事もなかったかのように処理を再開できます。状態はすべてSessionに記録されているため、Brainが何台クラッシュしても情報は失われません。

この「Cattle」設計により、以下の3つの重要な特性が実現されます。

* **🔀 Brainの水平スケーリング**: ステートレスなので、任意の数のハーネスを並列起動できます。負荷が高まったら、単にBrainの数を増やすだけです
* **🔄 Handsの交換**: コンテナが失敗しても、エラーをツール呼び出し失敗としてClaudeに返し、Claudeがリトライロジックを自分で判断します。人間のエンジニアがエラーを見て対処するのと同じ自律性です
* **🛡️ セッションの耐障害性**: Append-onlyのイベントログは「巻き戻し」「再開」「コンテキスト再読み込み」をすべてサポートします。永続化層が1つだけなので、障害ポイントも明確です

### 🧩 セッションが解決する「コンテキストウィンドウ問題」

LLMエージェント開発で避けて通れないのが、**コンテキストウィンドウの容量制限**です。長時間実行するタスクでは、会話履歴がウィンドウを埋め尽くし、過去の重要な情報が失われるリスクがあります。

従来のアプローチでは、コンテキストが溢れたら**不可逆的なコンパクション**（要約による情報損失）を行う必要がありました。Managed AgentsのSessionは、この問題に根本的に異なるアプローチを取ります。

以下の図は、従来のアプローチとManaged Agentsのアプローチの違いを示しています。

この図のポイントは、Managed Agentsではイベントログがコンテキストウィンドウの**外側**に存在する点です。コンテキストウィンドウは依然として128Kトークンの制限がありますが、Brainは`getEvents()`インターフェースを使って**位置ベースのスライシング**で必要な範囲のイベントだけを選択的に読み込めます。

つまり、トークン制限を超える長時間の作業でも、過去のイベントを「忘れる」のではなく「参照しに行く」ことができるのです。これは、人間がノートを取りながら作業するのと同じ——すべてを頭に入れておく必要はなく、必要なときにノートを見返せばいいのです。

## 📦 4つのコアコンセプト — Agent / Environment / Session / Events

[公式ドキュメント](https://platform.claude.com/docs/en/managed-agents/overview)では、開発者が操作する4つのコアコンセプトが定義されています。以下の図は、開発者がこの4つのコンセプトをどのような順序でAPI操作するかを示しています。

この図のポイントは、操作の順序が明確に番号付けされている点です。まずAgent（設計図）を作り、次にEnvironment（実行環境）を設定し、SessionでそれらをバインドしてからEventsで実際のやり取りを始めます。AgentとEnvironmentは**再利用可能**なリソースであり、一度作成すれば何度でもSessionから参照できます。

### 1️⃣ Agent — エージェントの「設計図」

Agentは**一度作成して複数セッションで再利用する**リソースです。モデル選択、システムプロンプト、利用可能なツールを定義します。人間の「職務記述書（ジョブディスクリプション）」のようなものだと考えると分かりやすいでしょう。

create\_agent.py

```
import anthropic

client = anthropic.Anthropic()

# エージェント定義を作成（IDで再利用可能）
agent = client.agents.create(
    model="claude-sonnet-4-6",
    instructions="あなたはPythonのデータ分析エキスパートです。"
                 "ユーザーのリクエストに基づいてデータ分析を行い、"
                 "グラフや要約レポートを作成してください。",
    tools=[
        {"type": "bash"},
        {"type": "file_editor"},
        {"type": "web_search"}
    ]
)
print(f"Agent ID: {agent.id}")
```

Agentに含まれる設定を以下にまとめます。

| 設定項目 | 説明 | 具体例 |
| --- | --- | --- |
| `model` | 使用するClaudeモデル | `claude-sonnet-4-6`, `claude-opus-4-6` |
| `instructions` | システムプロンプト | 「あなたはPythonのデータ分析エキスパートです」 |
| `tools` | 組み込みツール | Bash、ファイルエディタ、Web検索等 |
| `mcp_servers` | MCP経由の外部ツールプロバイダ | 社内API、Slack、GitHub連携 |
| `skills` | Claude Codeスキル | カスタムワークフロー |

> 💡 **設計のポイント：** AgentはIDで管理されるため、同じAgent定義を複数のSessionで再利用できます。たとえば「データ分析エージェント」を一度定義すれば、異なるデータセットに対して何度でもSessionを作成して分析を実行できます。

### 2️⃣ Environment — 実行環境の「テンプレート」

Environmentは、エージェントが動作するコンテナの設定を定義します。人間のエンジニアに例えるなら、「どのツールがインストールされた作業用PCを渡すか」に相当します。

create\_environment.py

```
# ※ パラメータ名は公式リファレンスで最新仕様を確認してください
# https://platform.claude.com/docs/en/managed-agents/quickstart
environment = client.environments.create(
    packages=["python3", "nodejs", "pandas", "matplotlib"],
    network_access="allowlisted",
    allowlisted_domains=["api.github.com", "pypi.org"],
    files=[
        {"path": "/workspace/config.yaml", "content": "..."}
    ]
)
```

セキュリティ設計のポイントとして、エンジニアリング記事では認証情報の取り扱いが詳しく説明されています。以下の図は、エージェントが外部サービスにアクセスする際のセキュリティアーキテクチャを示しています。

この図のポイントは、エージェントが生成したコードが認証情報に**直接触れない**設計です。すべての外部アクセスは認証プロキシを経由し、トークンはセキュアボールトで管理されます。これにより、たとえエージェントが悪意のあるコードを生成しても、認証情報が漏洩するリスクが最小化されます。

具体的なセキュリティメカニズムは以下の2つです。

* **🔑 Git認証**: サンドボックス初期化時にアクセストークンがバンドルされ、エージェントは認証情報を直接扱いません
* **🔐 MCP/OAuth**: トークンはセキュアボールトに保存され、プロキシが認証情報の取得と外部呼び出しを行います。ハーネスは認証情報に依存しない設計です

### 3️⃣ Session — 実行インスタンス

SessionはAgentとEnvironmentを指定して起動する、実際の「作業セッション」です。人間で例えるなら、「特定のスキルを持つエンジニア（Agent）に、特定のツールが揃ったPC（Environment）を渡して、作業を開始してもらう」ことに相当します。

create\_session.py

```
session = client.sessions.create(
    agent_id=agent.id,
    environment_id=environment.id
)
```

Sessionには、従来のステートレスなAPI呼び出しにはない重要な特性があります。

| 特性 | 説明 | 🎯 メリット |
| --- | --- | --- |
| **💾 永続化** | サーバーサイドでイベント履歴が保持 | クライアント側の状態管理が不要 |
| **🔌 切断耐性** | ネットワーク切断後も処理継続 | 不安定なネットワーク環境でも安心 |
| **🔄 再開可能** | セッションIDで後から再開 | 長時間タスクを中断・再開できる |
| **🔀 フォーク可能** | セッションを分岐して異なるアプローチ | A/Bテストや試行錯誤が容易 |

> 💡 注目ポイント： セッションの「フォーク」機能は特に強力です。たとえば、データ分析の途中で「別のアプローチも試したい」と思ったとき、現在のセッションをフォークして異なる分析手法を試し、結果が良い方を採用する——といったワークフローが可能です。

### 4️⃣ Events — SSEによるリアルタイムストリーミング

ユーザーメッセージをイベントとして送信し、Claudeの応答をSSE（Server-Sent Events）でストリーミング受信します。以下のコードは、基本的な送信・受信パターンです。

stream\_events.py

```
# ユーザーメッセージを送信し、レスポンスをストリーム受信
with client.sessions.stream(
    session_id=session.id,
    event={
        "type": "user_message",
        "content": "sales_data.csvを分析して、月次売上トレンドをグラフにしてください"
    }
) as stream:
    for event in stream:
        if event.type == "agent_message":
            print(event.content)
        elif event.type == "tool_use":
            print(f"🔧 ツール実行: {event.tool_name}")
        elif event.type == "tool_result":
            print(f"✅ 結果: {event.content[:100]}...")
```

Claudeは以下のように自律的に動作します。以下のシーケンス図は、ユーザーがデータ分析を依頼してから結果が返るまでの一連のフローを示しています。

この図から分かるように、Claudeは1つのユーザーメッセージに対して、**自律的に複数のツール呼び出しを連鎖**させます。「データを見る → スクリプトを書く → 実行する → 結果を報告する」という一連の流れを、人間の介入なしに実行します。

> 💡 **便利な機能：** 実行途中で**追加のユーザーイベントを送信して方向転換**させることも可能です。たとえば、分析の途中で「棒グラフではなく折れ線グラフにして」と追加指示を送れば、Claudeはその場で方針を変更します。

## ⚡ パフォーマンス最適化 — TTFTを最大90%削減

エンジニアリング記事で特に注目すべきは、**コンテナプロビジョニングの遅延化**によるパフォーマンス改善です。この最適化は、Brain/Hands/Sessionの3コンポーネント分離がもたらした直接的な恩恵です。

### 😰 問題 — コンテナ起動のレイテンシ

従来の設計では、セッション開始時にコンテナを即座にプロビジョニングしていました。コンテナの起動には数秒〜数十秒かかる場合があり、これがTTFT（Time To First Token — ユーザーが最初の応答を受け取るまでの時間）のボトルネックになっていました。ユーザーが質問を送ってから、コンテナが起動するまで何も応答が返らない——これは体験として非常にストレスフルです。

### ⚡ 解決 — Brain/Handsの分離と遅延プロビジョニング

Brain（推論）とHands（実行環境）を分離したことで、画期的な解決策が生まれました。Claudeはコンテナの準備を**待たずに推論を開始**できるようになったのです。

以下の図は、改善前と改善後のフローの違いを示しています。

この図のポイントは、改善後ではセッション開始と同時にBrain（推論）が即座に動き始め、Hands（コンテナ）はバックグラウンドで並行起動される点です。Brainが最初に行うのはユーザーの質問を理解し、計画を立てることなので、コンテナが準備完了する前にすでに有益な応答を返し始めることができます。ツール実行が必要になるタイミングには、大抵コンテナの起動が完了しています。

この改善の結果は劇的でした。

| 📊 指標 | 📉 改善幅 | 💡 解説 |
| --- | --- | --- |
| **p50 TTFT** | **約60%低下** | 中央値：半数以上のリクエストで60%速く |
| **p95 TTFT** | **90%以上低下** | 最悪ケース：ほぼすべてのリクエストで劇的に高速化 |

> 🎯 **なぜp95の改善が重要か：** p50（中央値）はすでに速いケースが多いですが、p95（最悪ケースの5パーセンタイル）は「ユーザーが体験しうる最も遅い応答」を表します。これが90%以上改善されたことは、**ほぼすべてのユーザーが一貫して速い応答を体験できる**ことを意味します。

### 🔧 組み込みの最適化機能

TTFT改善以外にも、Managed Agentsにはパフォーマンス最適化が組み込まれています。

| 最適化 | 説明 | 🎯 効果 |
| --- | --- | --- |
| **🗄️ プロンプトキャッシュ** | 繰り返し使用されるプロンプト部分を自動キャッシュ | API呼び出しコストの削減 |
| **🗜️ 自動コンパクション** | コンテキストウィンドウが溢れそうな場合の自動管理 | 長時間実行の安定性 |
| **📋 セッションイベントスライシング** | 必要なイベントだけを選択的に読み込み | メモリ効率の向上 |

これらはすべて自動で動作するため、開発者が個別に設定する必要はありません。

### 📊 判断フローチャート

この図に沿って判断すると、Managed Agentsが最も効果を発揮するのは「長時間実行タスク」と「サンドボックスが必要だがインフラリソースがない」場合です。

### ✅ Managed Agentsが向いているケース

| 📋 条件 | 💡 理由 |
| --- | --- |
| **🚀 PoC・実証段階** | インフラ構築のオーバーヘッドなしで素早くプロトタイプ可能 |
| **⏱️ 非同期処理（分〜時間）** | 長時間実行セッションがネイティブサポート |

| **🔌 マルチツール統合** | MCP経由で外部サービスと容易に連携 |

### ⚠️ 自前実装を継続すべきケース

| --- | --- |

| **🖥️ 独自GPU・専用環境** | Anthropicインフラ上の制約 |

| **⚡ ミリ秒単位の応答** | エージェントモデルは非同期向き |

## 🛠️ クイックスタート

### 📋 前提条件

### 🐍 Python でのセットアップ

以下は、データ分析エージェントを最小構成で構築する完全なコードです。

quickstart.py

```
import anthropic

client = anthropic.Anthropic()

# 1️⃣ エージェント定義
agent = client.agents.create(
    model="claude-sonnet-4-6",
    instructions="あなたはPythonのデータ分析エキスパートです。"
                 "ユーザーのリクエストに基づいてデータ分析を行い、"
                 "グラフや要約レポートを作成してください。",
    tools=[
        {"type": "bash"},
        {"type": "file_editor"},
        {"type": "web_search"}
    ]
)

# 2️⃣ 環境設定
environment = client.environments.create(
    packages=["python3", "pandas", "matplotlib", "seaborn"]
)

# 3️⃣ セッション開始
session = client.sessions.create(
    agent_id=agent.id,
    environment_id=environment.id
)

# 4️⃣ メッセージ送信 & ストリーム受信
with client.sessions.stream(
    session_id=session.id,
    event={
        "type": "user_message",
        "content": "以下のデータを分析してください: [1,4,2,8,5,7,3,6]"
    }
) as stream:
    for event in stream:
        if event.type == "agent_message":
            print(event.content)
        elif event.type == "tool_use":
            print(f"🔧 ツール実行: {event.tool_name}")
```

### ⚠️ 重要な注意点

> ⚠️ 全エンドポイントに**ベータヘッダー `managed-agents-2026-04-01`** が必要です（SDKは自動設定）

> ⚠️ Research Preview機能（Outcomes、Multi-agent、Memory）は別途[アクセスリクエスト](https://claude.com/form/claude-managed-agents)が必要です

> 💡 `max_budget_usd` を設定して、想定外のコストを防ぐことを強く推奨します

## 🔮 Anthropicの戦略的ポジション転換

### 📊 「モデルAPI」から「実行基盤」へ

Managed Agentsの発表は、Anthropicの戦略的転換を象徴しています。以下の図は、Anthropicのプロダクトポートフォリオがどのように拡大しているかを示しています。

この図のポイントは、Anthropicが単一のAPI提供者から、4つのプロダクトを持つ「エージェント実行基盤」提供者へと進化している点です。Messages API（モデル直接アクセス）、Managed Agents（マネージドエージェント）、Claude Code（ローカル開発エージェント）、claude.ai上のWebエージェント——これらは異なるユースケースに対応しつつ、共通のモデル基盤の上に構築されています。

これは、OpenAIのAssistants API、GoogleのVertex AI Agent Builder、AWSのBedrock Agentsと同じ「エージェントプラットフォーム」の競争領域に、Anthropicが本格参入したことを意味します。

### 🏛️ エンジニアリング記事が示す設計哲学

エンジニアリング記事の以下の引用は、Managed Agentsの設計哲学を端的に表しています。

> 🏛️ エンジニアリング記事では、メタハーネスの設計方針として **"unopinionated about the specific harness that Claude will need in the future"**（将来Claudeが必要とする具体的なハーネスについて仮定を置かない）と述べられています（[Anthropic Engineering](https://www.anthropic.com/engineering/managed-agents)）。

この設計原則は、モデルの進化に合わせてハーネスの種類（Claude Code、タスク特化エージェント等）を柔軟に追加できることを意味しています。インターフェース（Harness/Sandbox/Sessionの境界）は固定しつつ、実装（どのモデルが何台、どのコンテナがどこで動くか）は自由に変えられる——これはまさにソフトウェアエンジニアリングにおける「インターフェースに対してプログラムせよ」の原則です。

## 📋 まとめ

### 🎯 Claude Managed Agentsの本質

この記事で解説したManaged Agentsの本質を5つのポイントにまとめます。

1. **🏗️ 「モデルAPI」から「実行基盤」への転換**: Managed Agentsは単なるAPI群ではなく、エージェント実行に必要なインフラ全体をマネージドサービスとして提供するプラットフォームです
2. **🧩 Brain / Hands / Sessionの3コンポーネント分離**: ステートレスなBrain、交換可能なHands、永続的なSessionの分離により、耐障害性・スケーラビリティ・長時間実行を実現しています
3. **⚡ TTFT 90%以上削減**: Brain/Handsの分離と遅延プロビジョニングにより、最悪ケースのレイテンシが劇的に改善されました
4. **📋 コンテキストウィンドウ問題の解決**: セッションのイベントログがコンテキストウィンドウの外側に存在し、位置ベースのスライシングで必要な情報を選択的に読み込めます
5. **🐄 「Cattle, not Pets」の設計**: BrainもHandsも使い捨て可能。クラッシュしてもセッションから完全に復旧できる堅牢な設計です

### ⚠️ 注意事項

Claude Managed Agentsは2026年4月9日時点でパブリックベータです。API仕様やベータヘッダーは変更される可能性があります。本記事のコード例は公式ドキュメントに基づいていますが、最新のAPI仕様は[公式リファレンス](https://platform.claude.com/docs/en/api/beta/sessions)を必ず参照してください。
