GitHub Copilotを本格的に利用し始めて自分なりに経験したこと、感じたこと、必要だと思ったことを整理して見ることにしました。

## 開発するにあたって方針を整理する

!

**注釈**

* 執筆時点(2026年4月26日)で諸々共有されているような開発手法は取っていません。
* あくまで私がGitHub Copilotと1:1で向き合った知見をまとめた備忘録になります、使用感や感じたことも私の率直な意見として**優しく**受け止めてください。
* 現在開発途中のものも含んでいます。

基本的に以下のように開発を進めています。

1. Planモードで壁打ち後に仕様を確定
2. 仕様をAGENTS.mdに落とし込む
3. Agentモードに切り替えて開発開始
4. 提案を許可しつつ完成を待つ
5. レビューと修正を行う

ちゃんとした開発となると1行のプロンプトを投げるだけという訳にはいかないと思い、ここはキッチリ仕様を固めることにしました。仕様を固めた後は基本的に3~5の繰り返し作業となります。  
マシンスペックは以下になります。

* OS: macOS Tahoe 26.4.1
* CPU: M5 Pro
* Memory: 48GB
* SSD: 1T

開発環境と使用ツールは以下になります。

* VS Code 1.117.0
* GitHub Copilot Pro+
* GPT-5.4 Xhigh, Claude Opus4.6

以降特に変更がない限りGPT-5.4 Xhighを5.4、Claude Opus4.6をOpus4.6と表記します。  
執筆中に4.6が4.7に置き換わってたり、GPT-5.5が使えるようになってたりしてますが、これらにはまだ使用していないので本稿内で述べることはありません。

## LLMの特徴がハッキリとした

先にも述べていますがPro+をサブスクライブしたこともありせっかくなのでハイスペックモデルを使用しました。以降の内容は私が実際に開発を行った、行っている際の所見となります。

### Opus4.6は素直な優等生

[前回の記事](https://zenn.dev/teo_k/articles/8a629a876bf41a)ではOpus4.6を使用しました。こちらは非常に素直にAGENTS.mdに沿って開発してくだました。特にこちらが介入することなく最後まで走り抜けてくれました。ただ、精度には些か不満があったためこちらは要改善ということでissueを標記し、今後改善して行く予定です。モデルはどうしましょうか…Opus4.7を使ってみたい気持ちはありますがプレミアムリクエスト(以下PR)7.5xに若干びびっております。

### 5.4キミは…

まず前提としてOpus4.6を利用した開発とはかなり規模が違うことをお伝えしておきます。というか今も絶賛開発中なので単純な比較として見ずに**あくまで使用感**として認識してください。  
何というか、シンプルに**賢すぎる**と言えばいいでしょうか？ただ賢すぎるが故の弊害がチラチラと垣間見えました。

#### AGENTS.mdに沿ってる？

賢すぎるが故に、先回りして実装していってくれている印象がありました。`次のステップはこうです`、`次私ならこう進めます`といった提案に沿って進めて行きましたが、そのまま任せていると、こちらが望んでいない改善や修正を行おうとしたりすることがありました。最初は「賢いな〜」と思いながら進めていましたが、ある時気づきました。「これ、AGENTS.mdに記載されている内容？」と疑わしくなったので改めて5.4に調査させました。**結果は…NOでした😂**AGENTS.mdのスコープを逸脱しハレーション傾向にあったので以下のような指示を出しました。  
![](https://static.zenn.studio/user-upload/803d4efbf5cd-20260426.png)  
*図1.本来の道筋に戻す指示*

#### あれ?暴走してる？？

現在開発中のものが非常にメモリを食うため、メモリ管理は注意するように指示はしておりました。複数の検証を行う必要がありました。まあ、必要ならやるしかないよねと軽い気持ちで許可をしたら…**カーネルパニックでOSが落ちました😭**原因はまあ、メモリです、はい。ただメモリ管理には注意するように指示を投げていたにも関わらず、**勝手に**、**無闇矢鱈とサブエージェントを立てて**、**無駄にメモリ食い散らかしてたら**、そりゃ落ちるでしょと。これに関して私ちょっと**怒**でした。  
![](https://static.zenn.studio/user-upload/41601d146010-20260426.png)  
*好き勝手するのはAIだとしても流石に怒るよ*

#### SKILLはやっぱり必要？

開発が進むにつれて反復作業が増えてきたせいか、5.4が自己判断でディレクトリ内にSKILL.mdが存在するか探し始めました。  
![](https://static.zenn.studio/user-upload/2c6b88a7000d-20260426.png)  
*SKILL.mdを検索するようになった*  
正直ここまで作業が肥大化する想定ではなかったため、**これは私の見積もりの甘さによるミス**となります。人もAIも面倒な反復作業は避けたいのが本音のようです。現在の開発状況が落ち着いたら、どのようなSKILLが必要か5.4と相談することにします。

## ここまでで得られた自分なりの知見

まだ開発途中のものもありますが少なからず得られた知見はありますので以下にまとめてみます。

* 基本的に提案通りに任せても大丈夫
* サブエージェントの導入には注意が必要
* SKILLはあった方が良い
* ハーネスエンジニアリングとどう向き合うか

### 基本的に提案通りに任せても大丈夫

これに関してはLLMが賢いほど人が口出しするのはナンセンスだと私は考えています。とはいえ好き勝手ばかりさせていると、先にも述べたような事が少なからず起きるので注意や監視は必要だと思います。セキュリティとかまさに当てはまりますね。この辺はSKILLやhookでの制御の検討すべきです。

### サブエージェントの導入には注意が必要

開発を早く効率的に進めたいとか今回のようなメモリを大量に消費するような場合を除き、私はサブエージェントを利用しなければいけないとは思っていません。PRの使用量の不足や、場合によっては週制限に引っ掛かることも考慮しなければいけないからです。導入する際はインストラクションの整備が必要かなと思ってます。CLIで開発を行う場合は便利かもしれませんが、私自身この辺のキャッチアップがまだできていないので個人的な課題としておきます。

### SKILLはあった方が良い

単純作業や繰り返し作業が増えてくると効果を発揮するので、仕様を固めた後に導入するか相談すべきでした。ここは私の詰めの甘さというか開発規模が膨らんでいくことを想定出来ていなかった力量不足が露呈しました。以下公式のドキュメントやSKILLが公開されているので参考にしてみます。  
<https://docs.github.com/ja/copilot/concepts/agents/about-agent-skills>  
<https://awesome-copilot.github.com/>  
<https://microsoft.github.io/skills/>

### ハーネスエンジニアリングとどう向き合うか

最近話題に上がるハーネスエンジニアリング、これ人によって解釈が異なるから実に厄介だと感じています。[Anthropicの定義](https://www.anthropic.com/engineering/harness-design-long-running-apps)と[OpenAIの定義](https://openai.com/ja-JP/index/harness-engineering/)がきのこ・たけのこ戦争よろしくといった感じにならないことを日々祈っております。まあ冗談はさておき、長時間自立で開発をマルチエージェントで協業させたいなら前者、開発環境を整えエージェントに正しく作業させたいなら後者といったイメージでしょうか？私としては**人がハーネスを握り、AIエージェントが正しく開発手順を踏めるようにガードレールを引いてあげる役割も人が担う**と考えております。そういった意味で私のハーネスエンジニアリングの思想はOpenAIの定義に近いと思いました。いかに人が介入せず質の高いプロダクトを作り上げるか今後も議論の中心になりそうです。

## 最後に

GitHub Copilotを使用して本格的な開発を始めて日は浅いですが、それでも多くの知見を得ることが出来ました。CLIの方がまだ手を出せていないのでこちらも早く触って見たいです。今後もキャッチアップを継続し自分なりに得た知見を共有していきますのでお付き合いいただければ幸いです。
