## この記事で伝えたいこと

AIにアプリのUIを指示するとき、**「画面パーツの正式名称」を使えるかどうかで出力精度が段違いに変わる**という話と、その学習コストを限りなく下げるために作った「触って覚えるUI用語集」の設計意図を書きます。

LLMにUI実装を依頼する開発者・非エンジニア問わず、「指示のブレを減らしたい」人向けです。

## 問題：曖昧な指示でも「1回は」出る。でも再現しない

最近のLLMは優秀なので、こういう指示でもそこそこのUIを返してくれます。

```
上の方にメニューっぽいのをつけて、
真ん中に四角いのが並んでる感じで、
良い感じにおねがいします！
```

![](https://static.zenn.studio/user-upload/049b56f8ded4-20260324.png)

実際に試すと、ちゃんとヘッダー＋カードグリッドっぽいものが出てきます。LLMが文脈から意図を推測して「最善の解釈」をしてくれるからです。

**問題は再現性です。**

同じ指示をもう一度投げると、別のレイアウトが出てくる。「上の方にメニューっぽいやつ」がヘッダーになったりサイドバーになったりタブバーになったりする。1画面だけなら問題ないですが、複数画面にわたるアプリで「さっきと同じヘッダーで」が通じないのは致命的です。

再現性を担保する方法はシンプルで、**UIパーツの正式名称を使う**ことです。

```
ヘッダーにナビバーを固定。
メインエリアにカードをグリッドで3列配置。
各カードにはサムネイル画像（上）、タイトル、説明文（2行clamp）。
```

こう書けば、何回実行しても同じ構造が返ってきます。名前が仕様として機能するからです。

## なぜ「名前」が効くのか

LLMの学習データには膨大なUI関連のドキュメント・コード・チュートリアルが含まれています。それらは当然、正式名称で書かれています。

* 「ヘッダー」→ `<header>`, `position: fixed`, `z-index`といった実装パターンに直結
* 「モーダル」→ `dialog`, `overlay`, `backdrop`, `aria-modal`の文脈が呼び出される
* 「アコーディオン」→ `details/summary`, `max-height transition`, `aria-expanded`

つまり、**正式名称は「LLMが学習済みの実装パターン」へのキーワード**として機能します。「開く窓」では曖昧なクエリにしかなりませんが、「モーダル」なら一意に実装パターンが特定できる。

## 作ったもの：触って覚えるAI開発 用語集

<https://laf-create.jp/tools/ai-dev/ui-glossary.html>

全36語、7カテゴリ。すべての用語に**インタラクティブなミニデモ**がついています。

![](https://static.zenn.studio/user-upload/a84a4b0625fc-20260324.png)

### なぜ「触れるデモ」にしたか

UI用語の既存リソース（MDN、Material Design Guidelines等）は網羅性が高い一方、テキスト＋静止画での説明が中心です。

しかしUIパーツの本質は\*\*「動き」と「状態変化」\*\*にあります。

* モーダル → 開く・閉じる・背景暗転
* タブ → 切り替え時のコンテンツ遷移
* ドロップダウン → 展開・選択・折りたたみ
* スクロール領域 → 固定ヘッダーとコンテンツの独立スクロール

これをテキストで理解するのと、実際に触って「あ、これか」と体感するのでは、学習定着率がまったく違います。

### 各デモの実装例

いくつかピックアップします。

#### ヘッダー / サイドバー / フッター

ミニモック内にスクロール領域を設け、ヘッダーとフッターは `position: sticky` で固定。コンテンツ部分だけがスクロールする動作を実体験できます。

![](https://static.zenn.studio/user-upload/9fa7f5227cd9-20260324.png)

#### モーダル

ボタンクリックで `display` が切り替わり、半透明オーバーレイ＋中央パネルが出現。閉じるボタンまたはオーバーレイクリックで閉じる。背景の暗転が「注意を集中させる」UIパターンであることが直感的にわかります。

![](https://static.zenn.studio/user-upload/e689d81c549b-20260324.png)

#### タブ

クリックでアクティブタブの切り替え＋対応コンテンツの表示。カテゴリ分けされた情報を同一画面内で切り替える典型的なパターン。

![](https://static.zenn.studio/user-upload/c924206946d1-20260324.png)

#### ドロップダウン

クリックで選択肢が展開、選択するとラベルが更新されて折りたたまれる。フォーム入力やフィルタリングUIの基本。

![](https://static.zenn.studio/user-upload/d815a3cc4beb-20260324.png)

#### アコーディオン

見出しクリックで中身が開閉。FAQ、設定画面、長い情報の折りたたみに頻出。`details/summary`タグとの対応も意識しやすいデモ構成にしました。

![](https://static.zenn.studio/user-upload/e2219433d512-20260324.png)

## 実践：用語を使ったAI指示のテンプレ

用語を覚えたら、実際のAI指示にどう活かすか。

```
【画面構成の指示テンプレ】

■ ヘッダー（固定）
  - 左：ロゴ
  - 右：ナビバー（ホーム / 設定 / プロフィール）

■ メインエリア
  - タブ（すべて / お気に入り / 完了）
  - カードグリッド（2列）
    - 各カード：サムネイル + タイトル + ステータスバッジ

■ フッター（固定）
  - 追加ボタン → モーダル（入力フォーム）
```

この指示を渡せば、ChatGPTでもClaudeでもほぼブレなくUIを組んでくれます。

さらに正確に配置まで伝えたい場合は、**モックツール**で座標テキストを出力し、上記のUI用語と組み合わせて渡す方法もあります。

<https://laf-create.jp/tools/ai-dev/mock-tool.html>

座標テキスト＋UI用語の組み合わせは、現状もっともLLMに意図が伝わりやすいUI指示フォーマットだと実感しています。

![](https://static.zenn.studio/user-upload/41181219f275-20260324.png)

## 設計で意識したこと

### 滞在時間を稼ぐ構造

各用語にインタラクティブデモを仕込んでいるため、「読む→触る→次の用語→触る」のループが自然に発生します。用語集ページの性質上、直帰率が課題になりがちですが、デモの操作体験が「もう1用語見てみよう」を誘発する設計です。

### 回遊導線

用語集ページから以下への導線を配置しています：

* **モックツール** → 用語を覚えた人が「次にやること」
* **要件定義フォーム** → アプリ全体の設計をAIに伝えるためのツール
* **QAページ** → よくある疑問の解消
* **活用例ギャラリー** → モックツールの使い方イメージ

用語集は**入口**、他ツールが**出口**。この導線を意識して配置しています。

* LLMへのUI指示は、パーツの正式名称を使うだけで精度が段違いに上がる

* 用語集→モックツール→要件定義フォームの回遊導線で、AI開発の一連のフローをカバー
