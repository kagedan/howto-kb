## はじめに

フリーランス・法人でランサーズを使っていると、**案件を探す時間そのものがコスト**になります。

検索ページを開いて、案件を一つずつ確認して、金額やクライアント情報を見て、「これはSESだな」「これは予算が低すぎる」「これは良さそう」と判断して、良さそうなものには提案文を書く。この一連の作業を週に何度も繰り返している。

やっていること自体はパターン化できる。ならば自動化できるはず。

そう思って、**ランサーズの案件を定期的にスクレイピングし、AIで自動スクリーニングして、提案文まで生成するツール**を作りました。

## 全体の流れ

```
1. 週2回（月・木 9:00）、ランサーズの案件一覧を自動取得
2. 多段階のスクリーニングで応募価値のある案件だけを抽出
3. Slackに承認ボタン付きで通知
4. 「提案する」ボタンを押すとAIが提案文を自動生成
5. 生成された提案文をコピペしてランサーズで提案
```

## スクリーニングの仕組み

「どの案件に応募するか」の判断基準は、自分の中では明確にあります。それをコードに落とし込みました。

### 1. 除外フィルター

まず、そもそも応募対象にならない案件を弾きます。

**SES・準委任案件の除外**

ランサーズには受託開発の案件に混じって、実質SES（常駐・準委任）の案件が多数あります。これらは以下のパターンで自動判定しています：

* 「精算」「常駐」「出社」「参画」「面談」などのキーワード
* 月額単価表記（`○○万円/月`）
* 週○日の稼働指定
* 発注者が人材系企業

```
SES_KEYWORDS_BODY = [
    "精算", "140H", "180H", "出社", "常駐",
    "勤務地", "現場", "アサイン", "参画", "面談",
]
```

**モラルリスク案件の除外**

転売ツール、自動購入ツール、スクレイピングによる不正データ取得など、法的・倫理的にリスクのある案件も除外します。

**金額の下限フィルター**

10万円未満の案件は工数に見合わないため自動除外。

### 2. 多軸評価

除外されなかった案件に対して、3つの軸で評価します。

**金額妥当性（A〜D）**

案件の内容（API連携、新規開発、管理画面構築など）からカテゴリを推定し、そのカテゴリの市場相場と比較してA〜Dで評価します。

```
BUDGET_BENCHMARKS = [
    (["API連携", "GAS", "スプレッドシート"], (50000, 150000)),
    (["改修", "機能追加", "カスタマイズ"],    (200000, 500000)),
    (["管理画面", "管理システム"],           (300000, 800000)),
    (["新規開発", "システム開発"],           (500000, 1500000)),
    (["マッチング", "ECサイト"],             (3000000, 10000000)),
]
```

**クライアント信頼性（高/中/低）**

評価件数、平均評価、発注率、本人確認の有無から総合判定。過去の評価が10件以上で平均4.5以上、発注率60%以上、本人確認済みなら「高」。

**推奨アクション（応募推奨/要検討/見送り）**

金額妥当性がB以上かつクライアント信頼性が「中」以上なら「応募推奨」。提案数が50件を超えている場合は競争率の注記も付きます。

### 3. Slack通知

スクリーニングを通過した案件だけが、評価結果と一緒にSlackに通知されます。「応募推奨」「要検討」の案件には**承認ボタン**が付きます。

## 提案文の自動生成

「提案する」ボタンを押すと、Claude APIが案件の概要を読み取って提案文を生成します。

![提案文生成](https://static.zenn.studio/user-upload/deployed-images/8dc85441e9927550a9496386.png?sha=077bea68dce24c373c8d934ae8fbe5639c9c5be3)  
*ボタンを押すと数秒で提案文が生成される。そのままコピペで使える*

### プロンプト設計のポイント

提案文の生成で特にこだわったのは、**そのままコピペで使える完成形**を出力することです。

```
SYSTEM_PROMPT = """\
- 提案者は「合同会社SOUSHU」（認定ランサーの法人）
- 冒頭で「合同会社SOUSHUの沖口と申します」と名乗る
- 400〜600文字程度で簡潔に
- 案件の要件に対する理解を示す
- 具体的な技術スタック・実装方針を提案
- プレースホルダーは絶対に使わない
"""
```

最後の「プレースホルダーを使わない」が重要です。AIが`[ここに実績を記入]`のような穴埋めを出力すると、結局手動で埋める作業が発生します。具体的に書けない項目はそもそも省略させるようにしています。

## アーキテクチャ

構成はやや変則的です。

```
GitHub Actions cron (月・木 9:00 JST)
  → AWS Lambda でスクレイピング+スクリーニング
    → Slack Bot API で通知（承認ボタン付き）

Slack「提案する」ボタン押下
  → GCE FastAPIサーバー
    → Claude API で提案文生成
    → Slackに提案文を表示
```

### なぜスクレイピングにAWS Lambdaを使うのか

ランサーズはAWS WAF（Web Application Firewall）を使っており、**GCPやGitHub ActionsのIPレンジからのアクセスをブロック**しています。試行錯誤の結果、AWS LambdaのIPレンジは通過することがわかったため、スクレイピング部分だけLambdaで実行しています。

### なぜ提案の自動投稿ではなく手動コピペか

GCEからランサーズへのアクセスもWAFでブロックされるため、提案文の自動投稿は断念しました。ただ実際に使ってみると、**提案文を目視確認してからコピペする方が安心**なので、これはむしろ良い制約だったかもしれません。

## 技術スタック

| 役割 | 技術 |
| --- | --- |
| スクレイピング | Python + httpx + BeautifulSoup |
| スクリーニング | Python（ルールベース） |
| 定期実行 | GitHub Actions cron → AWS Lambda |
| Webhook受信 | GCE + FastAPI + Nginx + systemd |
| 提案文生成 | Claude Haiku API |
| 通知 | Slack Bot API + Interactive Messages |
| デプロイ | GitHub Actions + Workload Identity Federation |

## まとめ

このツールを使い始めてから、**案件探しに費やす時間がほぼゼロ**になりました。

週2回、Slackに通知が届いたら内容を確認して、良さそうなものは「提案する」ボタンを押すだけ。提案文もそのままコピペで使えるので、案件探しから提案までの所要時間は**1件あたり1〜2分**です。

フリーランスや法人の案件獲得において、**「探す」と「書く」は本質的な仕事ではない**。本質は「良い仕事をすること」であって、それ以外の作業はできる限り自動化したい。そういう思想で作ったツールです。
