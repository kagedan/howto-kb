第1弾ではReActループの基本を作った。コードを読んで、考えて、コメントを書く。動くことは動くが、問題がある。SQLインジェクションを見つけるのも複雑度の判断もすべてClaudeの知識頼みで、静的解析は何もしていない。

第2弾では専門ツールを3つ追加し、コードの内容に応じてエージェント自身がどれを使うか判断するようにする。

---

## 第1弾からの変更点

第1弾のツールは `read_code`、`think`、`comment` の3つで、エージェントは毎回同じ順番でこれらを呼んでいた。

第2弾では `think` を廃止して代わりに3つの専門ツールを追加する。

| ツール | 役割 |
| --- | --- |
| `lint_check` | AST解析で未使用変数・命名規則・docstringを検出 |
| `complexity_check` | McCabe複雑度を関数ごとに計算 |
| `search_best_practice` | SQLiやN+1クエリなどのパターンをコードから検出 |

第1弾との本質的な違いは「フローが固定ではない」ことだ。SQLインジェクションが疑われるコードには `search_best_practice(focus="security")` を、深いネストがあるコードには `complexity_check` を使う。何を使うかをエージェントが自分で決める。

---

## ツールの実装

3つのツールはすべて標準ライブラリの `ast` モジュールだけで実装している。外部ライブラリは使わない。

### lint\_check

```
class LintChecker(ast.NodeVisitor):
    def visit_Assign(self, node):
        for target in node.targets:
            if isinstance(target, ast.Name):
                self.assigned_names[target.id] = node.lineno
        self.generic_visit(node)

    def visit_Name(self, node):
        if isinstance(node.ctx, ast.Load):
            self.used_names.add(node.id)
        self.generic_visit(node)
```

ASTを走査して「代入されたが一度もLoadされていない変数」を未使用変数として検出する。`visit_Assign` で代入を記録し、`visit_Name` で使用を記録して差分を取るだけだ。

### complexity\_check

McCabe複雑度は「関数の基本複雑度1 + 分岐の数」で計算する。

```
def _count_complexity(self, node) -> tuple[int, list]:
    count = 1  # 関数そのもの
    for child in ast.walk(node):
        if isinstance(child, (ast.If, ast.IfExp)):
            count += 1
        elif isinstance(child, (ast.For, ast.While)):
            count += 1
        elif isinstance(child, ast.ExceptHandler):
            count += 1
        elif isinstance(child, ast.BoolOp):
            count += len(child.values) - 1  # and/or の分岐
    return count, branches
```

複雑度10超えでリファクタリング推奨、16超えで警告を出す。

### search\_best\_practice

SQLインジェクションの検出はASTで `cursor.execute()` の引数を見る。

```
def visit_Call(self, node):
    if (isinstance(node.func, ast.Attribute) and
            node.func.attr == "execute" and node.args):
        arg = node.args[0]
        if isinstance(arg, ast.JoinedStr):        # f-string
            self.matches.append(...)
        elif isinstance(arg, ast.BinOp) and isinstance(arg.op, ast.Add):  # + 結合
            self.matches.append(...)
        elif isinstance(arg, ast.BinOp) and isinstance(arg.op, ast.Mod):  # % フォーマット
            self.matches.append(...)
```

f-string、文字列結合（+）、%フォーマットの3パターンをそれぞれ検出する。さらに `query` という変数名で動的構築されている場合も前後のコードを見て判断する。

## ツール選択の仕組み

エージェントへのシステムプロンプトには、コードの特徴とツールの対応関係を書いておく。

```
SYSTEM_PROMPT = """
## ツール選択の判断基準

- DB操作・SQL文字列結合・認証処理を見つけたら
    → search_best_practice(focus="security")
- ループ内でDB呼び出し・全件取得してPythonでフィルタを見つけたら
    → search_best_practice(focus="performance")
- 複数の if/elif/for のネストが深い関数を見つけたら
    → complexity_check
- 変数の代入後に未使用・命名規則が怪しいと感じたら
    → lint_check

問題がなさそうなツールは呼ばなくて構いません。
"""
```

**「問題がなさそうなツールは呼ばなくて構いません」** という一文が重要だ。全ツールを毎回呼ぶとAPIコストが増える。必要なものだけ使う判断をClaudeに委ねる。

## 実行結果

実際のログを見ると、エージェントが内容に応じてツールを切り替えているのがわかる。

### sample\_01.py（SQLインジェクション）

```
イテレーション2:
  search_best_practice(focus="security")  → SQLi 3件検出
  lint_check                              → 0件（lint問題なし）
```

SQLが絡んでいると判断してセキュリティ検査を選んだ。lintも念のため呼んでいるが0件だった。

### sample\_02.py（未使用変数・複雑度超過）

```
イテレーション2:
  complexity_check  → calculate_discount の複雑度 16
  lint_check        → 未使用変数 5件
```

深いネストを見てcomplexity\_checkを選び、lint\_checkと組み合わせた。`search_best_practice` は呼んでいない。SQLもN+1もないと判断したからだ。

### sample\_03.py（型エラー・例外処理なし）

```
イテレーション2:
  lint_check            → 0件
  complexity_check      → 全関数の複雑度1（問題なし）
  search_best_practice  → 0件
```

3ツールすべてを呼んだが全部0件だった。静的解析では捕まらない種類の問題（例外処理の欠如）はClaudeが読んで自力で指摘している。

### sample\_05.py（N+1クエリ）

```
イテレーション2:
  lint_check                                   → 未使用import 1件
  complexity_check                             → 問題なし
  search_best_practice(focus="performance")   → N+1クエリ 2件

イテレーション3:
  search_best_practice(focus="security")      → 0件
```

パフォーマンス系の問題を見つけた後でセキュリティも念のため確認している。

### ツール使用回数の集計

```
処理ファイル数:   5
合計イテレーション: 21（第1弾は20）
実行時間:        87.5秒（第1弾は108.3秒）

read_code:           5回（全ファイル必須）
search_best_practice:5回
lint_check:          5回
comment:             5回
complexity_check:    3回（SQLi系ファイルでは不要と判断）
```

`complexity_check` が5回ではなく3回になっているのが面白い。sample\_01と sample\_04はSQLインジェクションやパスワード問題が主役なので、複雑度チェックは省いたと判断したようだ。

イテレーション2を「パターンA」と「パターンB」の2ブロックに分けて破線枠で囲みました。同じイテレーション2でも、コードの内容によって呼ぶツールが変わる——これが第2弾の核心なので、図の中で一番目立つ位置に置いています。

パターンA（sample\_02系）：complexity\_check + lint\_check を同時に呼ぶ。1回のAPI呼び出しでtool\_useブロックが2つ返ってくるので矢印も2往復になっています。  
パターンB（sample\_01、05系）：search\_best\_practice を focus パラメータ付きで呼ぶ。

## 第1弾との比較

| 観点 | 第1弾 | 第2弾 |
| --- | --- | --- |
| ツール数 | 3 | 5 |
| フロー | 固定（read→think→comment） | コード内容に応じて変化 |
| 検出手段 | LLMの知識のみ | 静的解析 + LLM |
| 実行時間 | 108.3秒 | 87.5秒 |
| イテレーション | 20回 | 21回 |

実行時間が短くなったのは `think` ツールをなくしたことと、不要なツールを省いたことが効いている。イテレーションが1回増えたのはsample\_05で5イテレーション使ったぶんだ。

## 今回の限界と次回

静的解析ツールを追加したことで精度は上がった。ただしすべての問題を事前に「パターン」として定義しなければならないという限界がある。sample\_03の例外処理欠如は静的解析では0件だった。結局ClaudeがコードをAI的に読んで指摘している部分も残っている。

もう一つの限界は「ローカルファイルしか読めない」ことだ。実際の開発現場でレビューが必要なのはGitHubのPull Requestだ。

次回（#3：MCP入門）ではその問題を解決する。自前のMCPサーバーを実装してGitHub APIをラップし、PRのdiffを自動取得してレビューするパイプラインを作る。
