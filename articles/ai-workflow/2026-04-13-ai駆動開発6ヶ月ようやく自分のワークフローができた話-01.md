## はじめに

プログラミングの勉強と AI 駆動開発を始めて、気がつけば 6 ヶ月が経っていました。

いい歳なので正直に言うと、「何をしたか書いておかないと思い出せない」のが最大の動機です。作ったものは覚えていても、なぜその設計にしたのか、何に詰まったのかが翌週にはもう曖昧になっています。

そんな状況でようやく「これなら続けられそう」というワークフローが固まってきたので、備忘録として残そうと思いました。対象読者は、個人開発を始めて半年くらいの、自分と同じような立場の人です。

---

## 以前の問題：1AI・1セッション 全部盛り

最初の数ヶ月はこうでした。

これを**全部ひとつのセッションでやっていました**。

どうなるかというと、最初のうちは快調でも、やり取りが重なるにつれて AI の出力が「なんかずれてきたな」という状態になります。コンテキストが爆発して、AI がどのファイルのどの実装を見ているのかが自分でも追えなくなる。

同時に、自分の理解力も限界を迎えます。AI が「この変更が必要です」と言っても、なぜそれが必要なのか、何と整合しているのかが追えなくなってくる。その結果、頭から煙を出しながら過去ログを読み漁る展開が続きました。

先人のブログを読み漁って、少しずつ対策を考えるようになりました。

---

## 解決のアプローチ：セッションを使い捨て、文脈をドキュメントで引き継ぐ

**① セッションを使い捨てにする**

要件定義・実装計画セッションとは別にコード生成・レビューセッションを立てることで、コンテキストを汚さない。セッションが終わったら捨てる前提で使う。1つのセッション内で全部解決しようとしない。

**② ドキュメントで文脈を引き継ぐ**

セッションをまたぐときは、ドキュメントが橋渡しをする。「この Issue の背景は〜」「この Plan で合意した方針は〜」が書いてあれば、次のセッションで改めて説明し直す必要がなくなる。

**③ サブエージェントはあえて使わない**

Claude Code にはサブエージェントという機能があります。複数の AI が並列に動く仕組みで、効率は上がるはずです。ただ正直に言うと、自分には何が起きているか追えませんでした。理解できないものは使いこなせないので、今のところ使わない選択にしています。周回遅れを自覚した上での判断です。

## ワークフロー詳解：Issue → Plan → Summary の流れ

実際のフローはこうなっています。

| ステップ | スキル / コマンド | 作られるもの |
| --- | --- | --- |
| 要求を整理する | `/create-spec` | GitHub Issue + `docs/specs/issue-123.md` |
| 実装計画を作る | `/create-plan` | `docs/plans/issue-123.md` |
| **自分が plan をレビューして承認する** | ー | 承認チェック（ここは自分の目） |
| 実装する | `/execute-plan` | コード変更・PR |
| コードレビューする | `/CodeReview_...` | 指摘リスト |
| レビュー指摘を直す | `/fix-review` | 修正コミット |
| **自分が動作確認して merge する** | ー | merge（ここも自分の目） |
| 作業を記録する | `/work-summary` | `docs/work-summary/YYYY-MM-DD.md` |

3 種類のドキュメントに役割を割り当てたのがポイントです。

**Issue（要求仕様書）= なぜ作るか**

「この機能が欲しい、なぜなら〜」を GitHub Issue として残します。実装の前に「そもそも何のためにやるのか」を言語化しておくと、実装が脱線しにくくなります。同時に `docs/specs/issue-123.md` に背景・スコープ・受け入れ条件を書いておきます。

**Plan（実装仕様書）= どう作るか**

`docs/plans/issue-123.md` に「どのファイルを触るか」「どういう順序で実装するか」「リスクは何か」をまとめます。**ここで自分がレビューして承認する**のが重要です。AI に全部任せると「なんかできた」で終わってしまうので、承認するまでは実装しない。

Plan のテンプレートにはこんな項目を用意しています：

```
## 方針
## ブランチ / PR 判断
## 実装手順
## 検証
## リスク・注意点
## 承認
- [ ] ユーザー承認済み
```

**Summary（作業記録）= 何をしたか**

セッション終了後に `docs/work-summary/YYYY-MM-DD.md` を作ります。実施内容だけでなく「なぜそうしたか（設計判断）」「未解決のこと」「次回の起点」を書くのがポイントです。これがないと翌週に「えーと、どこまでやったっけ」が確実に発生します。

## ファイル構成と CLAUDE.md の設計

リポジトリはこういう構造にしています。

```
app-template/
├── CLAUDE.md            ← 短く。詳細は docs を読め、とだけ書く
├── AGENTS.md            ← 同上
├── README.md            ← ワークフロー早見表
└── docs/
    ├── workflow.md      ← フロー詳細
    ├── ai-common.md     ← AI への共通方針
    ├── project-overview.md  ← プロジェクト概要（プロジェクトごとに書く）
    ├── specs/           ← 要求仕様書 (issue-123.md)
    ├── plans/           ← 実装計画書 (issue-123.md)
    ├── templates/       ← 各ドキュメントのひな形
    └── work-summary/    ← 日次作業記録 (YYYY-MM-DD.md)
```

**CLAUDE.md を短くした理由はトークン節約です。**

Claude Code は会話のたびに CLAUDE.md を読み込みます。ここに長々と書くと、毎回そのトークンが消費される。なので CLAUDE.md には「まず何を読め」だけを書いて、詳細は `docs/` に置く設計にしました。AI が必要なときだけ参照する形です。

実際の CLAUDE.md はこんな感じです（ほぼこれだけ）：

```
# CLAUDE.md

## まず読むもの
- README.md
- docs/project-overview.md
- docs/ai-common.md
- docs/workflow.md

## このプロジェクトで特に重視すること
- 変更は小さく行う
- 既存構成を尊重する
- 破壊的変更は確認を優先する

## 開発手法
スキーマ駆動開発で進める。詳細は `docs/project-overview.md` を参照。
```

## GitHubテンプレートリポジトリとして整理した

このワークフローを毎回ゼロから作るのは面倒なので、GitHub の「テンプレートリポジトリ」機能を使ってまとめました。「Use this template」ボタンを押すだけで、ディレクトリ構造とドキュメントのひな形が一式揃った状態で新しいリポジトリが作れます。正直この機能を知ったのも最近で、「こんな便利なものがあったのか」と感動しました。

ディレクトリ構造はこうなっています：

```
my-app/
├── CLAUDE.md            ← 短く。詳細は docs を読め、とだけ書く
├── AGENTS.md            ← 同上（Codex など他の AI ツール向け）
├── README.md            ← ワークフロー早見表
└── docs/
    ├── workflow.md          ← フロー詳細（Issue → Plan → Summary）
    ├── ai-common.md         ← AI への共通方針
    ├── project-overview.md  ← プロジェクト概要（プロジェクトごとに書く）
    ├── specs/               ← 要求仕様書 (issue-123.md)
    ├── plans/               ← 実装計画書 (issue-123.md)
    ├── templates/           ← 各ドキュメントのひな形
    └── work-summary/        ← 日次作業記録 (YYYY-MM-DD.md)
```

各ひな形はシンプルにしています。たとえば plan のひな形はこんな感じです：

```
# Issue #000 実装計画

## 方針
## ブランチ / PR 判断
## 実装手順
1. 
## 検証
- [ ] 
## リスク・注意点
## 承認
- [ ] ユーザー承認済み
```

work-summary のひな形：

```
# YYYY-MM-DD ワークサマリー

## 実施内容
## 設計判断（なぜそうしたか）
## 検証
## 未解決・TODO
## 次回の起点
```

項目が多すぎず、少なすぎず、「これだけ書けば来週の自分が困らない」というラインを意識して作りました。

## 半年やってみての正直な感想

良かったことは「なんでこうなったか」が追えるようになったことです。Plan に承認チェックを入れたことで、実装前に自分の理解が追いついているか確認する習慣がつきました。Summary を書くことで、1 週間後の自分でも「あ、そういえばここは〜という理由でこうしたんだ」と思い出せます。

正直なところを言うと、最新のトレンドには全然追いついていません。マルチエージェントでの並列処理、MCP サーバーの活用、エージェントに自律的にタスクをこなさせる、みたいな話は読んでいても「すごそうだな」で終わっています。理解できないものは実装できないし、実装できないものは使いこなせない。

でも今はそれでいいと思っています。**自分が理解できて、使いこなせるワークフローが一番良い。**
