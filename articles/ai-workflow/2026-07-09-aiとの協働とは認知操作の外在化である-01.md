## はじめに

最近、「AIと協働する」という言葉をよく聞く。

AIエージェント。  
Loop Engineering。  
マルチエージェント。  
RAG。  
プロンプトエンジニアリング。

それぞれは別の技術や手法として語られることが多い。

しかし私は、これらを眺めているうちに、一つの見方を持つようになった。

> **AIとの協働とは、人間がこれまで頭の中で行ってきた認知操作を、AIへ外在化しているだけではないか。**

ここでいう認知操作とは、難しい話ではない。

!

* 仮説を立てる
* 選択肢を出す
* 比較する
* 制約と照らし合わせる
* 役割を分ける
* レビューする
* 修正する
* 探索をやり直す

といった、人間が普段から頭の中で行っている思考の単位である。

AIとの協働を考えるとき、「AIに何ができるか」よりも先に、考えるべきことがあると思っている。

> **自分はこれまで、どのような認知操作を行ってきたのか。**  
> **そのうち、どの部分をAIへ外在化できるのか。**

この記事では、その視点からAI活用、AIエージェント、Loop Engineering、マルチエージェントを捉え直してみたい。

なお、冒頭の「だけ」には、後半で自分自身の修正を入れる。

外在化は、単なるコピーではないからだ。

## 人は昔から頭の中でループを回している

設計でも、研究でも、文章を書くときでも、何かを考えるとき、人は自然とループを回している。

たとえば設計であれば、

```
1. まず案を考える
2. その案の問題点を見る
3. 別の案を考える
4. 制約と照らす
5. 目的に合っているか確認する
6. 必要なら案を修正する
```

ということを繰り返す。

これは特別な技術ではない。

最初に出てきた案をそのまま信じるのではなく、他の可能性を考え、捨て、戻り、修正する。  
ある程度納得できるところまで、それを繰り返す。

この繰り返しが、思考の中に存在している。

最近、Loop Engineeringという言葉を見かけるようになった。

一般には、AIエージェントに対して、

```
* 実行する
* 結果を確認する
* 修正する
* 再実行する
```

という反復を行わせる考え方として語られることが多い。

その説明は正しい。

しかし私は、それをもう少し抽象化して見ている。

Loop Engineeringとは、AI時代になって突然生まれた新しい思考法ではない。

むしろ、

> **人間が昔から頭の中で行っていた「仮説・評価・修正・再試行」という認知ループを、AIとの協働プロセスとして外在化したもの**

なのではないか。

AIが考えているように見える。  
AIが自律的に作業しているように見える。

しかし、その根底にある構造は、人間が昔から行ってきた探索の外在化なのではないかと思っている。

## AIは「答えを出す機械」ではなく「認知操作を肩代わりする機械」である

一般的なAIの使い方は、こうである。

これは便利である。

メールを書いてもらう。  
文章を要約してもらう。  
コードを書いてもらう。

それだけでも十分に価値がある。

しかし、この使い方ではAIは「答えを返す機械」として扱われている。

一方で、AIをもう少し深く使う人は、AIに答えそのものを求めていない。

たとえば、

!

* 可能な設計案を複数出させる
* それぞれの長所と短所を比較させる
* 前提条件を変えて再検討させる
* 見落としているリスクを挙げさせる
* 別の専門家の視点からレビューさせる

という使い方をする。

これは「答えを聞いている」のではない。

人間が頭の中で行っていた認知操作の一部を、AIへ割り当てているのである。

AIは最終判断者ではない。

探索、比較、要約、レビュー、別視点の提示といった認知操作を肩代わりする存在になる。

## AIエンジニアがやっていることも、認知操作の外在化に見える

たとえばRAGは、単なる検索技術ではない。

人間が行っていた、

「必要な情報を思い出す」  
「過去の資料を探す」  
「文脈に関係する情報を参照する」

という認知操作を外在化している。

プロンプトエンジニアリングも同じである。

あれは単に「AIへの命令文を工夫する技術」ではない。

「何を前提にするか」  
「どの役割で考えるか」  
「どの観点から評価するか」  
「出力形式をどう制約するか」

という思考の枠組みを、言語として外へ取り出している。

AIエージェントも同じである。

人間が作業するときには、頭の中で自然に手順を組んでいる。

「まず調査する。次に方針を決める。その後に実装する。最後に確認する。」

AIエージェントは、この作業順序を外部化し、AIに実行させる仕組みである。

つまり、AIエンジニアがやっていることは、AIに魔法をかけているのではない。

人間がこれまで内側で行っていた認知操作を分解し、外へ取り出し、AIが扱える形に再構成している。

私はそう捉えている。

## マルチエージェントは「組織的な認知」の外在化である

マルチエージェントとは、複数のAIに役割を与えて協調させる仕組みである。

たとえば、

```
* 調査担当
* 設計担当
* 実装担当
* レビュー担当
* テスト担当
```

のように分ける。

これを見て、私はこう思う。

> **これは組織運営やチーム設計を、AIの上に外在化しているのではないか。**

部下を持ったことがある人。  
プロジェクトを分解して担当を割り当ててきた人。  
品質保証プロセスを設計してきた人。  
工場のラインや作業工程を設計してきた人。  
ソフトウェアアーキテクチャを考えてきた人。

こうした人たちは、普段から次のようなことを考えている。

「誰が何を担当するか。」  
「どの成果物を誰に渡すか。」  
「どこでレビューするか。」  
「どこで責任を分けるか。」

つまり、役割分担そのものを認知操作として行っている。

だから、AIを見ても自然にこう考える。

「調査はこのエージェントに任せる。」  
「設計案は別のエージェントに作らせる。」  
「レビューは第三のエージェントにさせる。」  
「最終判断は人間が行う。」

私自身も、設計・実装・検証実行・敵対的レビューという役割をAIに割り当てて開発を回している。

これは新しい発想を得たというより、これまで頭の中で組んでいた工程設計を、そのままAIの上へ移しただけである。

もちろん、これは仮説である。

正直に言えば、この仮説は反証しにくい形をしている。「何らかの役割分担の経験」なら、ほとんど誰にでもあるからだ。

厳密に言い直すなら、こうなる。

役割分担を明示的に設計してきた経験の深さが、エージェント構成の設計に現れる。

実際の組織であれ、ソフトウェア構造であれ、工程設計であれ、頭の中の仮想チームであれ、である。

ここまでの対応関係を整理すると、こうなる。

| AI技術・手法 | 外在化されている認知操作 |
| --- | --- |
| プロンプトエンジニアリング | 前提の設定、役割の設定、評価観点の固定 |
| RAG | 記憶の想起、資料の参照 |
| AIエージェント | 作業手順の分解と実行 |
| Loop Engineering | 仮説・評価・修正・再試行の探索ループ |
| マルチエージェント | 役割分担、組織的な認知 |
| レビューエージェント | 検査、批判的思考 |

## 私がAIに割り当てているのは「探索」である

私自身は、AIに正解を求めていない。

求めているのは、探索の補助である。

私は、自分の直感をそのまま信じない。

直感は大事である。  
しかし、直感だけでは採用しない。

「この方法がよさそうだ」と感じたとしても、すぐには信じない。

他の方法を考える。  
別の目的ならどうなるかを考える。  
理論的な裏付けを探す。  
目的に対して最も合っているかを確認する。

そして初めて、

「今回の目的なら、この方法が最も妥当だ」

と判断する。

つまり私は、結論を信じているのではない。

> **探索を経た選択を信じている。**

AIが登場して変わったのは、この探索の一部をAIへ任せられるようになったことである。

以前なら、自分で十通りの方法を考えていた。

今はAIに十通り出させることができる。

その中には、使えない案もある。  
前提が違う案もある。  
現実的ではない案もある。

しかし、それでよい。

重要なのは、AIが一発で正解を出すことではない。

選択肢が広がること。  
比較対象が得られること。  
自分の直感を検証する材料が増えること。

だから、私はAIを過信しない。

しかし、それはAIだからではない。

自分の直感も、同じように扱っているからである。

直感も仮説。  
AIの出力も仮説。

どちらもそのままでは採用しない。  
裏付けを取る。他案と比較する。目的に照らす。必要なら捨てる。

AIを特別扱いしているわけではない。

自分の直感に対して行ってきたことを、AIにも適用しているだけなのである。

## 「結果」ではなく「選択構造」を見たい

私は、結果だけを見て素晴らしいかどうかを判断できる人間ではない。

天才なら、完成品を見ただけで本質を見抜けるのかもしれない。

しかし私はそうではない。

だから、結論だけを提示されても困る。

「この方法が最適です」

と言われても、それだけでは信じられない。

知りたいのは、その前である。

!

* どのような選択肢があったのか
* なぜ他の案では駄目だったのか
* どの目的に対して、この案が優れているのか
* どの制約が効いているのか
* どの理論で裏付けられるのか
* どの条件なら別案の方がよくなるのか

こうした選択構造があって、初めてその方法の優位性が示される。

10種類の方法を考えた。  
そのうち9種類は、別の目的では優位だったかもしれない。  
しかし今回の目的には合わなかった。  
だから、この1つを選ぶ。

この構造があるから、選択を信じることができる。

AIの出力も同じである。

AIが出した答えを信じるのではない。

AIによって探索範囲を広げ、その中から目的に合うものを選ぶ。

その選択構造を信じるのである。

## 外在化は、単なるコピーではない

ここで、冒頭の「外在化しているだけではないか」という言い方に、自分で修正を入れておきたい。

「だけ」と書いたが、外在化は等価なコピーではない。

頭の中の認知操作は、本人にしか見えない。  
再現できない。  
他人に渡せない。  
昨日の思考を、そのまま再実行することもできない。

外在化された認知操作は違う。

プロンプトとして残る。  
手順として残る。  
ログとして残る。

つまり、観察でき、再実行でき、比較でき、改善でき、共有できる。

構造は同じでも、性質が変わる。

思考が、成果物になる。

そして、外在化して改善したループは、再び頭の中へ戻すこともできる。

AIに回させていた探索の型が、いつのまにか自分の考え方の型になっている。

外在化は、一方通行ではない。

なお、この見方自体は、私の発明ではない。

認知科学では「分散認知」「拡張された心」「認知的オフローディング」といった枠組みで、道具や環境へ認知を預ける現象が以前から論じられてきた。

メモも、図面も、チェックリストも、認知の外在化である。

AIは、その外在化先として現れた、桁違いに柔軟な器だと私は見ている。

## AI活用の差は、認知操作の差として現れる

同じAIを使っていても、人によって成果が違う。

その理由は、プロンプトの上手さだけではないと思う。

その人が、もともとどのような認知操作を行ってきたか。

それをどれだけ自覚しているか。

そして、それをAIへどのように外在化できるか。

ここで差が出る。

質問することに慣れている人は、AIを質問応答に使う。

文章を書くことに慣れている人は、AIを文章作成に使う。

設計案を比較してきた人は、AIを探索と比較に使う。

組織や工程を設計してきた人は、AIをマルチエージェントやワークフローとして使う。

品質保証を考えてきた人は、AI出力を検査対象として扱う。

つまり、AIの使い方は、その人の認知操作の写像として現れる。

AIは、使う人の思考構造を映す。

ただし、この写像は固定ではない。

外在化されたループを回すうちに、新しい認知操作を後から獲得することもある。

だからこれは、「経験のない人にAIは使えない」という話ではない。

自分の認知操作を自覚することが、AI活用の起点になる、という話である。

AI活用を深めるには、AIそのものを見るだけでは足りない。

自分がどう考えているのかを見る必要がある。

## おわりに

AIは、人間にまったく新しい思考法を与えたのだろうか。

私は、少し違うと思っている。

AIはむしろ、人間が昔から行っていた思考を、外へ取り出せるようにした。

認知操作を外へ出し、AIと分担し、観察できる成果物に変えられるようにした。

だから私にとって、AIとの協働とは、AIに仕事を任せることではない。

AIに答えを出してもらうことでもない。

> **自分がこれまで頭の中で行ってきた認知操作を分解し、外在化できる部分をAIへ委譲し、役割として再設計すること。**

探索、比較、仮説生成、レビュー、再試行。

これらは外在化できる。

一方で、目的の設定と、選択の責任は、外在化の対象ではないと私は考えている。

何を外に出し、何を手元に残すか。

その線引きこそが、役割設計である。

AIを高度に使うとは、AIに正解を出させることではない。

自分が何を考え、どこで迷い、何を比較し、どのように選択しているのかを理解し、その一部をAIへ割り当てることなのだと思う。
