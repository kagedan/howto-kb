前回、SPEC駆動開発やTDDの賞味期限の話をしました。そして「ルールでAIにお願いする」こと自体の限界にも触れました。

今回はその先の話。AIエージェントの制御手法は、いま大きなパラダイムシフトの真っ只中にあります。この手法も「今さら？」と言う方は多いと思いますが……それはそれとしてお付き合いください。

## ガードレールの3世代

AI開発における「AIの行動を制御する方法」は、ここ1年ほどで急速に進化しています。自分の体験と業界の動向を合わせて整理すると、大きく3つの世代に分かれます。

### 第1世代: ルール（Rules）— 「お願い」する

CLAUDE.md や .claude/rules/ にテキストで方針を書く。AIはセッション開始時にこれを読み、その内容に「従おうとする」。

```
# .claude/rules/no-silent-fallback.md
エラーを黙って握り潰すな。catchブロックでnullを返すな。
```

メリットは、簡単に書ける。人間にも読みやすい。  
限界は、AIの「判断」に依存する。忘れる。無視する。コンテキストが長くなると埋もれる。

自分のプロジェクトの21本のルールファイルはこの世代。効果はあったけど、「従わないこともある」という不安は常にありました。

### 第2世代: フック（Hooks）— 「物理的に不可能にする」

Claude Code の Hooks は、AIがツールを呼び出す前後にシェルコマンドやプロンプトを自動実行する仕組みです。

```
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "python check-dangerous-command.py"
    }]
  }
}
```

PreToolUse: ツール実行前に発火。exit 2 でブロック  
PostToolUse: ツール実行後に発火。フォーマット、リント、テスト実行など

ルールとの決定的な違いは**AIの判断を介さない。例えば** --set-env-vars が含まれるBashコマンドが来たら、AIが何を考えていようが物理的にブロックする。ルール違反を「注意する」のではなく「不可能にする」。

### 第3世代: 自律ガバナンス — 「ツールが自分で判断する」

OpenClaw の Heartbeat や、Policy-as-Prompt のような、AIエージェント自身がポリシーを読み込んで自律的にガバナンスを実行する世代。これは後ほど詳しく。

## Hooks の三層構造

Claude Code の Hooks は複数のハンドラを持っています。これが非常にうまく設計されているので、上手に使えるようになるとカッコいい！

### Command Hook（コマンドフック）

シェルスクリプトを実行する。マイクロ秒単位で動作。パターンマッチに最適。  
使いどころは「このファイルは編集禁止」「このコマンドはブロック」といった、Yes/Noで判定できるルール。

### Prompt Hook（プロンプトフック）

LLMプロンプトを実行する。セマンティック（意味的）な評価ができます。  
使いどころは「この変更は設計方針に合っているか？」「セキュリティ上の懸念はないか？」といった、文脈に依存する判断。

### Agent Hook（エージェントフック）

サブエージェントを起動する。ツールアクセス権を持ち、深い分析ができると謳っています。  
使いどころは「このコミットの変更全体を見て、統合テストを実行して」「関連するドキュメントも更新が必要か確認して」といった、複数ステップの検証。

つまり、

* Command 速い・決定的・シンプル
* Prompt 中速・セマンティック・柔軟
* Agent 遅い・深い・包括的

この3層を適材適所で使い分けることで、**「速さ」と「深さ」のバランスを取れます。**

## 「ルールをフックに書き換える」実践

自分のプロジェクトの21本のルールをフックの視点で見直すと、こうなります。

### 即座にフック化すべきだったもの

cloudrun-env-safety.md（環境変数安全ルール）:

* ルール: 「--set-env-vars を使うな」
* フック化: PreToolUse の Command Hook で --set-env-vars をパターンマッチ → exit 2 でブロック
* 効果: 事故が「起きてから対処する」から「起きえない」に変わる

no-silent-fallback.md（サイレントフォールバック禁止）:

mandatory-review-auto-trigger.md（レビュー自動実行）:

### ルールのまま残すべきもの

conventions.md（開発規約）:

* 「タイムゾーンはUTC」「Firestoreの初期化はハンドラ内で」
* これらは「禁止する」ものではなく「方向性を示す」もの
* フック化しても検出が難しい。ルールとしてAIに読ませるほうが合理的。

skill-subagent-usage.md（スキル・サブエージェントの使い分け）:

**線引きの原則: 「Yes/Noで判定できるか？」**

あとはセキュリティチームを最後に起動するエージェントHookを入れるとか、ルールをそのままワークフローに落とし込む手法はいろいろ考えられますね。

## エージェントツールの設計思想を理解する

フックの話をしましたが、そもそも各エージェントツールはどういう思想で設計されているかを理解しておくと、ツールの育て方が見えてきます。  
この辺は公式ドキュメントにゆずるとして、結論は **1つのツールに全部賭けない。**です。

2025年の記事（Roo Code回）で「全部賭けたところ、Claude-Codeが全部もっていきました」と書きましたが、今は全く違います。1つのツールに全部賭けるのではなく、各ツールの設計思想を理解した上で、場面に応じて使い分けるのが今のところの最適解。

## 人間がツールを育てて見守る設計

ここまでの話を総合すると、**AI開発の本質は「AIにコードを書かせること」ではなく「AIが活動する環境を設計すること」になりつつあります。**

ルールを書く。フックを設定する。ツールを選定する。エージェントの権限を設計する。レビュー体制を構築する。

コードを書くのはAI。環境を設計するのは人間。

そしてこの「環境」は固定的なものではありません。AIの性能が上がれば、フックの一部は不要になる。新しいツールが登場すれば、ワークフローを組み替える。ガードレールも進化させ続ける必要がある。

ツールを育てて、見守る。剪定して、選定して、もう庭の手入れです。🌲🌷🌲Garden maintenance Engineering 🌲🌻🌲ですわ。

これがSPECをガリガリ書いていた3ヶ月前の自分と、今の自分の、最大の考え方の違いです。
