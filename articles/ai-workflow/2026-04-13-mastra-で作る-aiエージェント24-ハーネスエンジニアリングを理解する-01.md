# 定義があいまいな「ハーネス・エンジニアリング」

ハーネス・エンジニアリングという言葉の誕生は2025年11月26日の[Anthropicの記事](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)ではないかと思います。そしてこの言葉がグッと広まったのは 2026年2月11日の[OpenAIの記事](https://openai.com/ja-JP/index/harness-engineering/)がきっかけでしょうか。それ以来、日本を含む世界中のAI技術者やエンジニアの間で、AIエージェントの性能を最大化するための環境設計として、ブログや技術系メディアで盛んにこの概念が整理され始めました。

正直、正確な定義は無いので、結構みんなバラバラな定義づけをしている、または自身のポジションや思いの強さに従って話の力点が変わってきたりもしています。[Mastra社も定義を試みています](https://zenn.dev/shiromizuj/articles/ff2ff5e6967e4d)が、やはりAIエージェント・フレームワーク・ベンダーとしての視点で語っている印象でした。

今回は、私なりの「ハーネス・エンジニアリング」の理解を書いてみますが、これもまた一人のAIエージェント・フレームワーク「ユーザー」としての視点ということで、多少の偏りはご容赦ください（笑）

# 「コンテキスト・エンジニアリング」は分かりやすかった

第12回で扱った「コンテキスト・エンジニアリング」は、概念として分かりやすかったです。「**コンテキストとはLLMの短期記憶のことであり、そこに格納するデータの量と質をどうするかという設計がコンテキスト・エンジニアリングだ**」ということでした。

<https://zenn.dev/shiromizuj/articles/e1e0717a663845>

この**一本、軸のある課題感・テーマ**があって、そこにまつわるあれこれがコンテキスト・エンジニアリングでした。すなわち、プロンプトであり、ツールの入出力であり、RAGであり、会話履歴。これらはすべて「コンテキストを圧迫しないように実現するにはどうしたらよいか」という工夫が続けられています。

Mastraの Observational Memory はまさにその一例でしょう。あるいはAgent Skills のように、必要な情報を段階的に提供してコンテキストを節約するプログレッシブ・ディスクロージャーという手法もコンテキスト・エンジニアリングの一環でした。ワークスペース上にファイルの形で引き継ぎ文書を書くことで短期記憶を解放する工夫も然り。マルチエージェント化によるコンテキスト分離や、エージェント間のコンテキスト引き継ぎをどう設計するか、という取り組みも同じ文脈でした。

# 世間の定義する「ハーネス・エンジニアリング」の最大公約数と最小公倍数

それに比べると、「ハーネス・エンジニアリング」は、「コンテキスト・エンジニアリングも含め、AIエージェントが**長時間自立して行動**するにあたって、**脱線せずに**振る舞えるようにするための**あらゆるもの**」といわんばかりです。この程度の定義だと、「今後生まれる概念すらハーネスの一部」ということになりそうです。

## 最大公約数

それでも、およそ世間の定義で、みなさんが共通してハーネス・エンジニアリングに含める概念というものはあるのではないか。ということで、Zenn、Qiitaでハーネス・エンジニアリングの定義について考察している**技術ブログ30本をピックアップし、その内容を精査・集計**してみました。その多くで言及があったのは以下の7項目です。

* **ガードレールと安全性設計**：エージェントが有害な動作や意図しない出力を行わないようにする仕組み
* **評価・テストパイプライン**：エージェントの品質を継続的に計測・改善するための基盤（本シリーズで扱った eval がその例）
* **プロンプト設計**：システムプロンプトや few-shot の工夫など、AIエージェントへの「指示の品質」を高めること
* **ツール設計**：AIエージェントが利用するツール（関数・API）の入出力を、LLMが扱いやすい形に整えること
* **マルチエージェント・オーケストレーション**：複数のエージェントを連携させるアーキテクチャ全体の設計方針
* **コンテキスト・エンジニアリング**：RAGやメモリ、会話履歴の管理を含む、コンテキストウィンドウの最適化
* **エラー処理とリトライ設計**：LLMの不確実な出力に対し、適切にリカバリする仕組みを持つこと

とりわけ上位2項目のガードレールと評価テストは、30本ほぼすべてで言及がありました。ガードレールが「事前に逸脱を防ぐ制約」、評価テストが「事後に動作を確認する検証」——両者合わせると「**AIを制御し、正しく動いているかを確認するループ**」を担う概念であり、「**ハーネス**（**手綱・馬具**）」という言葉の本意にも最も直結しているとも言えます。

## 最小公倍数

一方で、ハーネス・エンジニアリングを広く定義した場合にのみ含まれる傾向があったのが以下の3項目です。

* **インフラとデプロイ設計**：デプロイ環境、スケーリング、サンドボックス管理を含む実行基盤の整備
* **観測性とデバッグ**：エージェントの動作をトレース・ロギングし、問題の発見と修正を容易にすること
* **コスト・レイテンシの最適化**：モデルルーティングやキャッシュなど、実用的な運用コストを抑える工夫

## これまで本連載で書いてきたことは、ぜんぶハーネス・エンジニアリング

ということで、過去23回の連載で書いてきた記事の内容は、ほとんどすべてハーネス・エンジニアリングに含まれる、と言えそうです。そして、今後公開される記事もそうなのでしょう。

# フレームワークとハーネスの関係

MastraはAIエージェント・フレームワークです。AIエージェント・フレームワークと言えば、他にもLangChainなどが有名です。

フレームワークとハーネスの関係は何でしょうか？　フレームワーク＝ハーネス？　一方が他方の一部になっている？　それとも一部が重複した関係？

## ハリウッドの原則

コンピュータ・システムにおける「フレームワーク」とは、**共通部品の一種**です。よく似た言葉に「ライブラリ」がありますが、ライブラリは業務アプリから必要なときに呼び出す部品です。フレームワークはその逆で、**処理の骨格をフレームワーク自身が持ち、必要なタイミングで業務アプリを呼び出します**。リクエストはまず必ずフレームワークが受け付け、そこから個別の業務アプリへと処理が渡される仕組みです。

![](https://static.zenn.studio/user-upload/a5f09623db37-20260404.png)

この考え方を「**ハリウッドの原則**（Hollywood Principle）」と呼びます。スローガンは「Don't call us, we'll call you（我々を呼ぶな。必要なときは、我々が呼ぶ）」。ハリウッドの役者（業務アプリ）がプロデューサー（フレームワーク）に売り込みの電話をかけるのではなく、プロデューサーが必要なときに役者を呼ぶ——という比喩です。

## AIエージェント・フレームワークの責務

AIエージェント・フレームワークも同じです。AIエージェントを開発する際の「骨格」を担うのがフレームワークで、一般に以下のような機能を提供します。

**フレームワーク側が担う領域：**

* LLMとのやり取り（会話管理・コンテキスト保持）
* エージェントループ（思考→ツール呼び出し→観察の繰り返し）
* ツール呼び出しのルーティング
* ワークフロー・マルチエージェントの制御フロー
* メモリ管理（短期・長期・セマンティック検索）
* RAGパイプライン（埋め込み・チャンク・検索）
* モデルルーティング（プロバイダー切り替え・フォールバック）
* ストリーミング・構造化出力の制御
* 可観測性（トレーシング・ログ・メトリクス）基盤
* 評価・スコアリング基盤

その上に、個別の業務ロジックとしてのAIエージェントが実装されます。

**開発者（ユーザー）が担う領域：**

* エージェントのゴール・役割・人格の定義
* システムプロンプトの設計
* 使用するツールの選定と実装
* どのLLMモデルを使うかの選択
* メモリスコープの設計（何を覚えさせるか）
* ワークフロー・サブエージェントの構成
* ガードレール（入出力の安全フィルタ）の設定
* 評価基準（何をもって良い応答とするか）の定義

## フレームワークとハーネスの関係

AIエージェントにおける「ハーネス」は、「一般的な業務アプリにおけるソフトウェア・アーキテクチャ」のようなものだと言えるかも知れません。

そして、ハーネスを実現するのは「フレームワークか、個別業務アプリか」と考えるとどうでしょう。

フレームワークには、フレームワークを作った人たちがいるわけで、その人たちの「思い」が込められています。「AIエージェントはこうやって作るべきだ」という思想です。つまり、大なり小なり、フレームワークにはハーネスが一部含まれています、またはハーネスを誘導している部分もあります。

とはいえ、どちらかというと**フレームワーク ユーザーが、フレームワークの上で個別のAIエージェントを開発する際にハーネスが作られることの方が多い**ように思います。

そのあたりの塩梅は、AIエージェント・フレームワークによって、少しずつ変わってきます。

## 主要なAIエージェント・フレームワークのポジション（制約と自由度）

フレームワークが「どこまでをフレームワーク側で担うか」は製品によって大きく異なります。一軸で単純に整理することはできませんが、ここでは「**フレームワーク側の関与の度合い**」つまり「制約が多いか、自由度が高いか」という軸でざっくりと並べてみます。

```
↑ フレームワークが構造・制約を多く提供（opinionated）
   ＝ 少ないコードで動くが、フレームワークのやり方に乗る必要がある

  AutoGen（Microsoft）
    └ エージェント同士の会話パターンやロール分担を強くフレームワーク側で規定

  CrewAI
    └ 「クルー（チーム）＋ロール設計」というパターンを前提とした構造

  LangGraph（LangChain）
    └ ステートマシン・グラフ構造でワークフローを定義。DAG的な制御が中心

  Mastra
    └ エージェント・ワークフロー・ツール・メモリ・RAG・可観測性・評価まで
      フルスタックに統合。開発者体験を重視した「バッテリー付属」型

  LangChain
    └ チェーン・ツール・メモリ等の豊富なコンポーネントを提供。
      組み合わせ方の自由度が高い分、設計判断が多い

  LlamaIndex
    └ RAGパイプラインに強みを持つ。エージェント機能も拡張中

  Vercel AI SDK
    └ フロントエンドとのシームレスな統合に特化。
      エージェントループの制御はシンプルでUIとの親和性が高い

  OpenAI Agents SDK
    └ 薄いラッパー。OpenAI APIをそのまま活かしつつ、
      ツール呼び出し・ハンドオフを最小限のコードで実現

↓ 薄いラッパー・プリミティブ寄り（unopinionated）
   ＝ 自由に設計できるが、ハーネスのほとんどは自前で組む必要がある
```

各フレームワークのポジションについて補足します。

**AutoGen / CrewAI** は「マルチエージェントのロール設計」という特定のパターンにコミットしており、そのパターンに乗れば少ないコードで複雑な協調動作が実現できます。一方、そのパターンから外れた設計はやりにくい面もあります。

**LangGraph** はグラフ（有向非巡回グラフ）でワークフローを定義するというモデルを採用しており、複雑な分岐・ループ・状態管理を明示的にコントロールしたい場合に向いています。

**Mastra** はエージェント・ワークフロー・ツール・メモリ・RAG・可観測性・評価といった機能をひとつのフレームワークで提供する「フルスタック型」です。フレームワークとしての意見（opinionated）は持ちながらも、各コンポーネントの差し替えやカスタマイズの自由度も高く設計されています。TypeScriptネイティブで開発者体験を重視している点も特徴的です。

**LangChain** は豊富なコンポーネントライブラリとして長く使われてきており、ツール・チェーン・メモリ等の選択肢が広い反面、設計の自由度が高い分だけ開発者が判断すべきことも多くなります。

**LlamaIndex** はRAGパイプラインに特に強みを持ちます。エージェント機能は後から拡張されてきた経緯があり、データのインジェストと検索が中心的なユースケースに向いています。

**Vercel AI SDK** はNext.js等のフロントエンドとの親和性を重視した設計で、ストリーミング応答のUIへの反映などを簡潔に実現できます。エージェントとしての機能はシンプルで、バックエンドの複雑なオーケストレーションよりもUI統合が主眼です。

**OpenAI Agents SDK** はOpenAI公式の薄いラッパーで、ツール呼び出しとエージェント間のハンドオフを最小限のコードで実現するプリミティブ寄りのSDKです。OpenAI APIへの依存度が高い反面、余分な抽象化が少なくシンプルです。

## 主要なAIエージェント・フレームワークのポジション（ハーネスへの関与）

上記の「opinionated ↔ unopinionated」軸と「**フレームワーク自身がどれだけハーネスを作りこんでいるか**」という軸は、似ているけど別です。ハーネスへの関与、という軸で整理し直してみます。

```
【A】ハーネスを作りこんでいる
   ＝ 可観測性・評価・メモリ・RAG・ガードレール等がフレームワークに統合済み

  Mastra
    └ エージェント・ワークフロー・メモリ・RAG・可観測性・評価・ガードレールまで
      フルスタックで統合。フレームワークがハーネスの大部分を「内蔵」している

────────────────────────────────────────

【B】ハーネスの「武器」はそろっているが、組み合わせはユーザが設計する
   ＝ コンポーネントは豊富だが、ハーネスの設計・組み立ては開発者の責任

  LangChain / LangGraph
    └ ツール・メモリ・RAG・エージェントループ等の部品は豊富。
      ただしそれらをどう組み合わせてハーネスを作るかは開発者が判断する

  LlamaIndex
    └ RAGパイプラインの部品は非常に充実。
      他のハーネス要素（評価・可観測性等）は外部との組み合わせが前提

  AutoGen / CrewAI
    └ マルチエージェント協調の「型（パターン）」はフレームワークが提供。
      ただしRAG・可観測性・評価といったハーネスの他要素は別途用意が必要

────────────────────────────────────────

【C】ハーネスはユーザが一から開発する必要がある
   ＝ フレームワークはAPIの薄いラッパーや実行基盤の提供にとどまる

  OpenAI Agents SDK
    └ ツール呼び出し・ハンドオフの仕組みのみ。
      可観測性・評価・メモリ・RAG等はすべて自前またはサードパーティで実現

  Vercel AI SDK
    └ ストリーミング・UIバインディングが主眼。
      エージェントとしてのハーネス機能は最小限で、残りは自前設計が必要
```

**【A】Mastra** は「ハーネスへの関与度」という軸では群を抜いています。可観測性・評価・RAG・メモリ・ガードレールといったハーネスの主要要素が最初から統合されており、フレームワークの上で個別エージェントを開発するだけで、かなりの部分のハーネスが「自動的に」手に入ります。その分、Mastraの思想・設計方針に乗ることになりますが、関与度の高さはフルスタックならではの強みです。そして、ユーザはMastraの提供するフレームワークの上で、細やかなハーネスのチューニングをすることが可能です。

**【B】LangChain / LangGraph / LlamaIndex** は豊富なコンポーネントを提供しており、「武器庫」としての充実度は高いです。ただしそれらをどう組み合わせてハーネスとして機能させるかは開発者の設計力に委ねられます。**AutoGen / CrewAI** は「マルチエージェント協調」という特定ドメインのハーネスは強固ですが、その外側（RAG・評価・可観測性等）は自前またはLangChain等との組み合わせに頼ることが多いです。

**【C】OpenAI Agents SDK / Vercel AI SDK** はハーネスの提供より「APIアクセスのシンプルさ」や「UI統合」を優先した設計です。ハーネスを本格的に構築するには、周辺のツール群（LangFuse・BrainTrust等の可観測性プラットフォームなど）を別途組み合わせる必要があります。

---

先ほどの「opinionated ↔ unopinionated」マップと比べると、**MastraはBゾーン付近に位置していましたが、ハーネスへの関与という軸ではAに分類されます**。これが「2つの軸は別物」と述べた理由です。「設計の自由度が高い（unopinionated寄り）でも、ハーネスの機能を豊富に内蔵している」——これがMastraの特徴的なポジションと言えます。
