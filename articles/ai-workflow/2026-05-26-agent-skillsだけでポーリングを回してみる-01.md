## Agent Skillsでポーリングを試してみる

では、さっそくAgent Skillsで状態監視型のポーリングを行ってみたいと思います。  
流れとしては、`status-json-poll-demo` というスキルを作成し、その中で`/demo/status.json` の状態を監視します。`state` の値に応じて、待機・完了・修正要求・停止を切り替えることで、エージェントによるポーリング処理を試します。ポーリング処理は5秒おきで、最大120秒ポーリングするようにしています。ポーリングした際のログは`/demo/logs`以下に吐き出しておきます。

/skills/status-json-poll-demo/SKILL.md

```
---
name: status-json-poll-demo
description: demo/status.json を監視して、PENDING、DONE、FIX、BLOCKED、TIMEOUT の状態を分類し、ローカルのポーリングログを書きながら、ネットワークに依存しない安全なエージェントのポーリングループをデモする。シンプルな status.json ポーリングデモを行いたい場合に使う。
---

# status.json ポーリングデモ

## 目的

`demo/status.json` を外部サービスの代替となるローカルファイルとして扱う。デモ用ジョブが終端状態に到達するまでポーリングし、各観測結果を `demo/logs/` に記録する。

## デモファイル

- `demo/status.json`: 監視対象の状態ファイル。
- `demo/polling-config.json`: ポーリング間隔、タイムアウト、ログ出力先。
- `demo/logs/`: ローカルのポーリングログ。

どちらかの JSON ファイルが存在しない場合は、ポーリング前に次のデフォルト形で作成する。

```json
{
  "state": "PENDING",
  "message": "デモジョブはまだ実行中です。state を DONE、FIX、または BLOCKED に変更してください。"
}
```

```json
{
  "intervalSeconds": 5,
  "timeoutSeconds": 120,
  "logDirectory": "demo/logs"
}
```

## 状態ルール

- `PENDING`: `intervalSeconds` 待ってからポーリングを続ける。
- `DONE`: 成功として停止し、最後の `message` を報告する。
- `FIX`: ポーリングを停止し、対応が必要であることを報告する。
- `BLOCKED`: ポーリングを停止し、不足している判断または外部アクションをユーザーに求める。
- `TIMEOUT`: 経過時間が `timeoutSeconds` を超えたら停止する。
- 未知の `state`、不正な JSON、必須フィールド不足は `BLOCKED` として扱う。

ログと最終サマリーでは、状態名を必ず大文字の文字列としてそのまま保持する。

## ワークフロー

1. `demo/polling-config.json` を読む。
2. `demo/status.json` を読む。
3. 1 回の実行につき、次のログファイルを 1 つ作成する。
   ```text
   demo/logs/<YYYYMMDDHHMMSS>-status-json-poll-demo.md
   ```
4. 各ポーリング観測結果として次を追記する。
   - ローカル時刻
   - 経過秒数
   - 観測した `state`
   - 観測した `message`
   - 判断結果 (`continue`、`done`、`fix`、`blocked`、または `timeout`)
5. `state` が `PENDING` の場合は、`intervalSeconds` 待ってから繰り返す。
6. 終端状態または blocked 状態を観測した場合は、すぐに停止して結果を要約する。
7. 経過時間が `timeoutSeconds` を超えた場合は、`TIMEOUT` を記録して停止する。

## ログ形式

ログは日本語で書く。ただし、状態名、パス、コマンド、タイムスタンプはそのまま保持する。

次の構造を使う。

```markdown
# status.json ポーリングデモログ

## 設定

- 状態ファイル: demo/status.json
- intervalSeconds: 5
- timeoutSeconds: 120

## ポーリング履歴

| 時刻 | 経過 | 状態 | メッセージ | 判断 |
| --- | ---: | --- | --- | --- |

## 最終状態

- state: DONE
- 理由: status.json が DONE に到達した

```
```

ステータスは以下のようにしておきます。

/demo/status.json

```
{
  "state": "PENDING",
  "message": "デモジョブはまだ実行中です。state を DONE、FIX、または BLOCKED に変更してください。"
}
```

スキルを実行して、ポーリングを試してみます。  
以下のような内容が表示され、うまく動いてステータスを監視しているのがわかります。  
※ Codex CLIから実行しました

> 35 秒時点ではまだ PENDING です。DONE、FIX、BLOCKED のいずれかに変わるまで継続し、変更がなければ 120 秒超過で TIMEOUT として停止します。

ステータスを以下に変更してみます。

/demo/status.json

```
  {
    "state": "DONE",
    "message": "デモジョブが完了しました。"
  }
```

ポーリングしたタイミングで以下の内容が表示され、無事に処理が終了しました。

> demo/status.json が DONE に更新されたため、ポーリングは正常終了しました。最後にログの最終状態と作業ツリー上の見え方だけ確認します。
>
> • status-json-poll-demo を実行しました。demo/status.json は開始時 PENDING で、55 秒後の 2026-05-26 16:33:46 に DONE を検知して正常終了しています。  
> ログ: demo/logs/20260526163251-status-json-poll-demo.md

今回の構成では、状態を以下のように分類し、エージェントが次に取るべき行動に変換しやすい形にしています。

* PENDING: まだ待つ
* DONE: 正常終了
* FIX: 修正が必要
* BLOCKED: 人間の判断が必要
* TIMEOUT: 監視を打ち切る

## 実際のユースケース

スキルで状態監視型のポーリングを簡単に試すことができたので、次は実際のユースケースでこの仕組みを使った例を見ていきます。  
私が実際に活用しているユースケースは以下です。

> GitHubのPRで行われるcodexのbotのレビュー結果をポーリングで監視し、指摘されたら修正してpushを行い、再度botにレビューを依頼しポーリングの監視を継続する。レビューが通れば、ポーリング処理を終了する。

修正する際に、ローカルでサブエージェントを使ったレビューの修正ループは行っていますが（`内側のループ`）、この状態監視型のポーリングを活用することで、GitHub PRへcodexのbotのレビューの修正ループ（`外側のループ`）を実現しています。  
codexのbotのレビューではレビューのステータスとして、レビュー中は「👀」、問題がなければ「👍」のスタンプが付くので、これを監視してポーリングを行っています。

```
  外側のループ: GitHub PR のレビュー監視

  ┌──────────────────────────────────────────────────────┐
  │                                                      │
  │  ┌──▶ [ PR 状態確認 ] ───▶ [ 状態分類 ] ───┐            │
  │  │                                         │         │
  │  │                                         ▼         │
  │  │                            [ PENDING（👀）: 待つ ] │
  │  │                                         │         │
  │  └─────────────────────────────────────────┘         │
  │                                                      │
  │  [ FIX: 修正が必要 ]                                   │
  │          │                                           │
  │          ▼                                           │
  │    ┌────────────────────────────────────────┐        │
  │    │ 内側のループ: ローカル修正とレビュー         │        │
  │    │                                        │        │
  │    │ ┌──▶ [ 修正 ] ─▶ [ 検証 ] ─▶ [ review ]─┐         │
  │    │ │                                      ││        │
  │    │ └──── [ 指摘あり: 再修正 ] ◀─────────────┘│        │
  │    │                                        │        │
  │    │        [ No findings ]                 │        │
  │    └───────────────┬────────────────────────┘        │
  │                    │                                 │
  │                    ▼                                 │
  │             [ commit / push ]                        │
  │                    │                                 │
  │                    ▼                                 │
  │    [ @codex review（PRにレビュー依頼のメッセージを送信）]   │
  │                    │                                 │
  └────────────────────┴─────────────────────────────────┘
                       │
                       ▼
                  外側ループへ戻る
         DONE（👍） の場合は外側ループを抜けて完了
```

## メリット・デメリット

Agent Skills でポーリングするメリットの1つは、「手軽さ」だと思います。  
専用のジョブ基盤やワーカーを用意しなくても、状態監視のポーリングを実現できるからです。

どこまで安定して動いてくれるかという懸念はありましたが、実際に試してみると思っていたよりも安定して動いてくれました。

一方で、デメリットは、バックグラウンドで動き続ける常駐ジョブではない点です。  
そのため、エージェントのセッションを動かしておく必要があります。  
長時間の監視や、高い信頼性が求められる用途には不向きだと思います。
