## 【この記事でわかること】

* 乱立するAI開発ツールを5つのレイヤーで整理した全体像
* OpenAI、Claude、Gemini などに加え、Bedrock / Vertex AI / Azure OpenAI も含めた **6つの推論APIプラットフォーム** を実装して見えた共通構造
* LLMのツール呼び出し（Function Calling / Tool Use）におけるラウンドトリップの仕組み
* なぜ **stateless な推論API** が長く主流だったのか、そして今なぜ **stateful な会話API** も増えているのか

---

2025年、生成AIを用いた開発ツールやフレームワークは爆発的に増加し、まさにカオスな状態になっています。技術選定において「どれを使えばいいのか？」と迷う前に、まずはエコシステムの全体像と、その根底にあるLLM APIの設計を理解することが重要です。

今回、フレームワークに頼らず、あえて一番下のレイヤーである推論APIの近くを6プラットフォーム横断で実装・比較検証する機会がありました。そこから見えてきたのは、各社で細かな差分はあっても、**ツール呼び出しの基本構造はかなり共通している** 一方で、**会話状態の持ち方は必ずしも共通ではない** ということです。

勉強のため、「税金計算シミュレータ」という簡単なAIエージェントを作成しました。ざっくりと税金の計算をチャット形式で行えうことができます。

## 1. カオスなAIプラットフォームを整理する「5つのレイヤー」

現在乱立しているAI関連ツールやサービス群を、抽象度に応じてレイヤー分けした全体像は以下のとおりです。

![](https://static.zenn.studio/user-upload/e456f991d43e-20260317.png)

**Layer 1: 推論API（モデル提供元API / クラウド提供API）**  
OpenAI API、Claude API、Gemini Developer API など、モデルへの直接アクセスを提供する層です。加えて、Bedrock / Vertex AI / Azure OpenAI のように、クラウド経由でモデル推論を提供する層も、実務上はこのレイヤーで捉えると整理しやすいです。

**Layer 2: エージェントSDK / フレームワーク**  
OpenAI Agents SDK、Google ADK、LangChain など、コードファーストでエージェントを構築・管理するための層です。

**Layer 2.5: LLMOps / ゲートウェイ**  
LiteLLMやLangSmithなど、本番運用に必要なルーティング・監視・評価を行う層です。

**Layer 3: マネージドプラットフォーム**  
AWS Bedrock AgentCoreやAzure AI Agent Serviceなど、インフラと実行環境ごとホスティングする層です。

**Layer 4: ノーコード / ローコード**  
Difyやn8nなど、GUIベースでフローを構築し、コードなしで使える層です。

> ※ このレイヤー分けは、あくまで理解のための整理です。実際の製品は複数レイヤーにまたがることも多く、厳密な標準分類ではありません。

## 2. Layer 1 を6プラットフォーム横断で実装してみた

今回はこの全体像の中で、最も基礎となる Layer 1 を、以下の6プラットフォームで直接実装して比較しました。

* OpenAI
* Claude
* Gemini Developer API
* Bedrock
* Vertex AI
* Azure OpenAI

ここで1つだけ補足すると、これら6つを全部まとめて「ネイティブAPI」と言い切るのは少し雑です。

* OpenAI / Claude / Gemini Developer API は、モデル提供元のAPIに近い
* Bedrock / Vertex AI / Azure OpenAI は、クラウド事業者が提供する推論アクセスレイヤー

なので、より正確に言うなら **「6社のネイティブAPI」ではなく、「6つの推論APIプラットフォーム」** です。

ただ、そこを踏まえてもなお、同じAIアプリケーションを横断実装してみると、**決定的に共通している部分** が見えてきます。

## 3. そもそも、モデル自身は外の世界を直接実行できない

共通点について語る前に、大前提となるモデルの性質を確認しておきます。

最近のモデルは、もはや「テキストを入れて、テキストを返すだけ」ではありません。画像や音声などを扱えるモデルも増えています。

重要なのは、**モデル自身は外部APIを叩いたり、任意コードを実行したりする手足を持っていない** ということです。

なので、今回実装した「税額シミュレータ」のように、ユーザーの対話とツール（関数）呼び出しが絡む場合、裏側では必ず次のような通信の往復が発生します。

1回目のループ：ユーザーの入力 → モデルが「Tool 呼び出し」を要求  
2回目のループ：（1回目のやり取り + サーバーで実行した Tool の結果）を入力として渡す → モデルがユーザー向けの最終回答を生成

各社のAPIを触っていて痛感したのは、**「私たちはLLMそのものを直接操作しているわけではない」** ということです。  
開発者にできるのは、**APIに input を投げ、返ってくる output を監視し、もしそれがツール呼び出し要求なら hook して処理を差し込む** ことです。

---

## 4. ツール呼び出しの本質は「ラウンドトリップ」にある

このラウンドトリップ構造は、OpenAI の Function Calling、Anthropic の Tool Use、Gemini / Vertex の Function Declarations、Bedrock の Tool Use など、名前やレスポンス形式に差はあってもかなり共通しています。

イメージとしては、ざっくりこんな感じです。

```
// stateless な chat/messages 系APIでの実装イメージ
async function runChatCompletion(messages, tools) {
  for (let round = 0; round < MAX_TOOL_ROUNDS; round++) {
    const response = await api.create({ messages, tools });
    const msg = response.message;

    messages.push(msg); // 履歴を自前で追加

    if (msg.tool_calls) {
      const result = await executeTool(msg.tool_calls);
      messages.push({ role: "tool", content: result });
      continue;
    }

    return msg.content;
  }
}
```

つまり本質は、

* モデルがツール要求を返す
* アプリ側がツールを実行する
* 結果を履歴に追加して、もう一度モデルに渡す

という往復にあります。

なお、いまは SDK や一部のAPIが built-in tools や自動処理を持っているので、開発者が毎回このループを全部ベタ書きするとは限りません。  
ただ、抽象化の下にある基本構造は、やはりこのラウンドトリップです。

## 5. ただし、LLM API は「全部ステートレス」とは言えない

以前は「LLM API は全部ステートレス」と整理したくなる場面も多かったのですが、少なくとも2026年3月時点では、その言い方は正確ではありません。

### stateless な代表例

以下のようなAPIでは、会話履歴をクライアント側で持ち、必要に応じて再送する設計が基本です。

* Anthropic Messages API
* Gemini Developer API の `generateContent`
* Vertex AI の `generateContent`
* Bedrock Converse API
* OpenAI Chat Completions API

このタイプでは、ツール実行のラウンドトリップを回すときも、毎回必要な履歴をアプリ側で管理して渡すことになります。なので、ユーザーとのやりとりを全てアプリ側で保持しておき、全ての会話の記録をinputとしてLLMに与えることで、あたかもStatefulであるかのような振る舞いを実現していました。

### stateful な代表例

一方で、現在はサーバー側で会話状態を引き継げるAPIも増えています。

* OpenAI Responses API（`previous_response_id` など）
* Azure OpenAI Responses API（`previous_response_id`）
* Gemini Interactions API（`previous_interaction_id`）
* Bedrock の OpenAI 互換 Responses API

このタイプでは、**毎回フル履歴を送り直さなくても会話を継続できる** ため、長い会話やエージェント用途ではかなり扱いやすくなっています。

さらに、SDK / フレームワーク側でも stateful な設計を前提にしたものが増えています。たとえば Google ADK は `Session` / `State` / `Memory` を中核概念として持ち、会話スレッドを `SessionService` で管理します。永続化が必要なケースでは、`DatabaseSessionService` や `VertexAiSessionService` のような persistent session services を使う設計になっています。これは「基盤の推論APIが stateless かどうか」とは別に、**アプリケーション側では state を明示的に扱う設計が主流になってきている**ことを示しています。  
同じ傾向は Strands Agents にも見られます。Strands は Session Management を備えており、会話履歴やエージェント状態を保持するための `FileSessionManager` や `S3SessionManager` を公式に提供しています。つまり、ネイティブAPIの世界では stateless な設計が今も重要である一方で、実際のエージェント実装では stateful な体験を前提にしたSDKや実行基盤がかなり増えてきています。

なので、今の時点でより正確に言うなら、

> **stateless な推論APIが長く主流だったが、現在は stateful な会話API/SDKも増えている**

です。

## 6. それでも stateless な設計がデファクトスタンダードだった理由

毎回全履歴を送るのは一見非効率に見えますが、各社が long time で stateless な設計を採用してきたのには、モデルプロバイダーの戦略があったようにおもえます。

### 圧倒的なスケーラビリティ

数百万ユーザーからのリクエストを処理する際、サーバー側で個々の会話状態を保持すると、スケールや運用が一気に難しくなります。stateless にしておけば、空いているどのサーバーでもリクエストを処理しやすくなります。

### 障害への強さ

サーバー側セッションへの依存が薄いので、リトライや再実行の設計が比較的シンプルになります。

### コスト透明性

送信したトークン数 = 課金額と言い切れませんが、毎回全メッセージを送信するため、消費トークン数がリクエストの内容から算出しやすいという点もあります。ステートフルな場合、「サーバー側で保持されているコンテキスト」分のトークンが不透明になり、予期しないコスト増につながりやすいです。

## おわりに

Agent SDK（Layer 2）などの便利なフレームワークを使えば、この「状態管理」や「ループ処理」は裏側でかなり自動化されます。

ただ、その根底にある Layer 1 の設計を理解していないと、Vercel などのサーバーレス環境や各種フレームワークと組み合わせたときに、思わぬアーキテクチャの衝突に悩まされることがあります。

今回いちばん言いたかったのは、次の2点です。

1. **ツール呼び出しの本質は、モデル出力とアプリ側実行を往復させるラウンドトリップにある**
2. **会話状態の扱いは、プロバイダごと・APIごとに違い、今は stateless と stateful が併存している**

つまり、今理解しておくべきなのは「LLM API は全部ステートレスだ」という単純な話ではなく、

> **どのAPIが、どこまで状態を持ち、どこから先をアプリ側で管理するのか**

という境界です。

ツールがどんどん抽象化されていく今だからこそ、その下にある推論APIの設計を理解しておくことは、技術選定においてかなり強い武器になると感じました。
