## 背景 — RAGは動かせても、品質とコストは自明ではない

ドキュメントを読み込んで質問に答えるRAGは、ライブラリや素のSDKを使えば比較的短い時間で動かせます。一方で、動いたRAGが「正しいページを引けているか」「誤った断定をしていないか」「1クエリにいくらかかっているか」は、動作している事実だけからは分かりません。

そこで、これらを実際に測定することにしました。対象は自作のRAG [tech-docs-rag](https://github.com/Hal-Hanami/tech-docs-rag) です。Claudeの公式ドキュメント66ページを1,290個の引用可能なチャンクに分割し、質問に対して各主張へ出典 `[n]` を付けて回答します。ドキュメントが裏付けられない質問には、推測せず「分かりません」と回答します。

この記事では、このRAGについて次の3点を、リポジトリの評価ハーネスが出した実測値で報告します。

1. 検索品質 — 正しいページをどの程度引けているか（recall@1 / recall@3 / MRR）
2. ハルシネーションの抑制 — 誤った断定をどう抑え、回答を控える挙動（abstention）をどう評価したか
3. コスト — 1クエリあたりの費用と、その内訳および削減

あわせて、測定の過程で確認された測定そのものの限界（run ごとにぶれる faithfulness 指標）と、その限界を踏まえて主張をどの指標に限定したかについても述べます。

回答の実際の出力は図1のとおりです。各文に出典番号が付き、その下にモデル別のコストとレイテンシのトレースが表示されます。

*図1：デモ画面の回答ボックス（質問・`[n]` 引用つき回答・出典リンク・モデル別トレース）*

この、回答・出典・コストのトレースが1画面に表示される状態を出発点として、以降で検索品質・ハルシネーションの抑制・コストを順に報告します。

## 検索品質の測定

RAGの回答品質は、生成の前段にある検索（retrieval）の品質に大きく依存します。生成器がどれだけ優れていても、誤ったページを渡せば誤った回答になります。そのため、まず検索の品質を数値化しました。

リポジトリには `python -m rag eval` という評価ハーネスがあります。バージョン管理された質問セット（質問と、期待される出典URLの組）に対して、recall@1 / recall@3 / MRR を測定します。質問は難易度で2つのスライスに分けています。

* **core（22問）**: 一般的な意味検索で引ける質問。
* **hard（12問）**: 完全一致のキーワード（`bash_20`・`str_replace`・`pause_turn` などのツール名）や、兄弟ページの曖昧性（web-fetch と web-search のどちらか）を含む、検索が外れやすい質問。

測定は同一インデックス・同一セッションで構成をA/B比較しています。`--dense-only` / `--no-rerank` フラグにより、密検索のみのベースライン（M2）と製品構成（ハイブリッド＋リランク）を同じ条件で比較します。日をまたいだ測定は条件が揃わないため行っていません。この点は後述のコストの議論にも関わります。

測定結果は次のとおりです。

| slice | metric | dense | hybrid (RRF only) | **hybrid + rerank** |
| --- | --- | --- | --- | --- |
| **core**（22問） | recall@1 | 86.4% | 72.7% | **86.4%** |
|  | MRR | 0.932 | 0.827 | **0.932** |
| **hard**（12問） | recall@1 | 66.7% | 58.3% | **91.7%** |
|  | MRR | 0.833 | 0.764 | **0.944** |

![図2 検索品質の測定表](https://static.zenn.studio/user-upload/deployed-images/c02492278d0095ff214ba1fe.png?sha=c57f060292b27a6d5806a8a3a5c27b9f16b7b025)  
*図2：リポジトリのREADMEに掲載している検索品質の測定表*

この表から、測定によって確認できた2つの非自明な結果を示します。

### 結果1：素のRRFは順位をかえって悪化させる

ハイブリッド検索（密検索＋BM25）は密検索より良い、という前提はよく見られます。しかし、密検索が強く効く意味的なコーパスでは、BM25のキーワードヒットを素のRRFで融合すると、正解ページが1位から落ちました（core の recall@1 が 86.4% → 72.7%）。キーワードの表層一致が、意味的にはより適切なページを押し下げてしまうためです。

ここで効果を持つのがリランカーです。融合後の候補をクロスエンコーダで採点し直すと、RRFで一度落ちた順位を回収し、ベースラインと同水準に戻りました（72.7% → 86.4%）。つまりリランカーはハイブリッド検索の安全装置として働いており、「ハイブリッドだから無条件に良い」わけではありません。これは数値を取らなければ確認できませんでした。

### 結果2：リランクが効くのは hard スライスのみ

リランカーは一様な底上げではなく、効くべき箇所にのみ効きました。hard スライス（完全一致キーワード・兄弟ページの曖昧性）では recall@1 が 66.7% → 91.7% と大きく改善します。一方 core スライスでは密検索がすでに上限（86.4%）に達しており、リランクを加えても変化しませんでした。

これは妥当な結果だと考えます。易しい質問で密検索がすでに正解を1位に出しているなら、リランカーが順位を直す余地はありません。難しい質問でのみ順位が改善するという挙動は、スライスを分けて測定したことで確認できました。

なお recall@3 / recall@5 はすべての構成で100%でした。正解ページは常に候補プールには含まれています。したがってリランカーの役割は recall（含まれているか）ではなく rank（生成器が最初に見る順位）の改善にあると整理できます。この観点は、後述のコストの議論に直接つながります。

## ハルシネーションの抑制と abstention の評価

検索品質を確認したら、次は誤った断定（ハルシネーション）を抑える設計です。本RAGでは抑制を4つの層で構成し、最後の層を数値による測定としました。

1. **根拠限定プロンプト**（`rag/generate.py`）：生成器には検索した出典のみを渡し、システムプロンプトで外部知識の使用を禁止し、各主張に `[n]` 引用を必須としています。モデルが持つ知識ではなく、渡された出典だけで回答する状態を作ります。
2. **回答を控える（abstention）**：出典が回答を裏付けないときは、推測せず固定文字列 *"I don't know based on the provided documentation."* を返します。「答えない」ことを正規の結果の一つとして扱います。RAGで最も避けたいのは、根拠がないままもっともらしく答えてしまうことだからです。
3. **良い証拠を先に置く**：前節のハイブリッド＋リランクにより、正しいページを生成器の入力の先頭に置きます。検索品質の向上は、そのままハルシネーション対策になります。
4. **測定**：評価セットにおいて、コーパス外の質問4問はすべて回答を控えました（**abstention rate 100%**、4/4）。一方、コーパス内の質問はすべて回答しています（**false abstention 0**）。「答えるべき質問には答え、答えるべきでない質問には回答を控える」が数値として確認できています。

実例として、デモには *"What is a good recipe for sourdough bread?"*（サワードウブレッドのレシピ）という質問を用意しています。Claudeのドキュメントに該当する情報はないため、この質問には回答を控えます。答えられない質問に答えないことを、デモ上で確認できます。

![図3 sourdough の質問に回答を控えるデモ画面](https://static.zenn.studio/user-upload/deployed-images/7cbe1eb09ab3047e97ee903d.png?sha=9e68401f92376dec2e74cd8c9a6dbb0746eded47)  
*図3：sourdough の質問に対し "I don't know based on the provided documentation." と返すデモ画面（abstention）*

引用 `[n]` は検索結果と1対1で対応しているため、faithfulness（回答が出典に忠実か）をLLM-as-judgeで採点できる形にはなっています。ただし、この faithfulness の測定には看過できない限界があります。次節で述べます。

## 測定上の限界 — faithfulness の扱い

faithfulness を採点するLLM-as-judge（コストの安い Haiku を使用）は、run ごとに ±20pt 程度ぶれます。同一の密検索パイプラインが、ある回では 86.4%、別の回では 63.6% と採点されました。入力もコードも変更していない状態でのことです。

これは、faithfulness を「ハルシネーションを●●%抑えた」という主要な数値として提示すると、その主張が測定ノイズに左右されることを意味します。別の回に測れば異なる数値が出る以上、20pt以内の差を「改善した」と述べることは、ノイズを成果と取り違える危険があります。

そこで本プロジェクトでは、faithfulness を方向性の参考にとどめ、すべての主張を決定的（deterministic）な指標に限定しました。recall・MRR・abstention rate は、同じ入力に対して同じ値を返します。したがって「品質が落ちていない」と述べるときは、これらの指標が不変であることを根拠とします（次節のコスト最適化で、実際にこの方針を用います）。

この限界をそのまま記述することも、本プロジェクトの一部だと考えています。リポジトリのREADMEにも同じ趣旨の注記（honesty note）を記載しており、コードと記述が一致していること、数値を誇張していないことを担保しています。

## コストの測定と最適化

最後に、1クエリあたりのコストを測定します。最適化の前に、まず費用を可視化します。

本リポジトリでは、1リクエストごとにトレースを取得し、トークン数と金額をモデル別に分離します（`rag/observe.py`）。

* 生成：`claude-opus-4-8`
* 判定（judge）：`claude-haiku-4-5`
* 埋め込み・リランク：Voyage（`voyage-4-lite` / `rerank-2.5-lite`）

料金は暗記せず、リリースごとにソースから確認します（Claude は `claude-api` スキル、Voyage は公式の料金ページ）。記憶した料金は古くなり、誤りの原因になるためです。

| 役割 | モデル | 料金（per 1M） |
| --- | --- | --- |
| 生成 | `claude-opus-4-8` | $5 in / $25 out |
| 判定 | `claude-haiku-4-5` | $1 in / $5 out |
| 埋め込み | `voyage-4-lite` | $0.02 |
| リランク | `rerank-2.5-lite` | $0.02 |

モデル別に分離した結果、支出の構造が明確になりました。生成（Opus）が全体の約90%を占め、Voyage の埋め込みとリランクは合計でも全体の40分の1程度です。したがって、コスト削減の対象は「生成器に渡す文脈量」に絞られます。

ここで前節の「recall@3 = 100%」が手がかりになります。リランク後、正解は必ず rank 3 までに含まれています（recall@3 が飽和している）。つまり、生成器に渡すチャンクの4〜5枚目は、トークンを消費するだけで網羅性には寄与していません。そこで、生成に渡す文脈を top-k 5 → 3 に削減しました。

結果は次のとおりです。

| 指標（eval core・同一セッション） | k=5（前） | k=3（後） | Δ |
| --- | --- | --- | --- |
| recall@1 / @3 / MRR（決定的ガード） | 86.4% / 100% / 0.932 | **不変** | **0** |
| abstention rate / false abstention | 100% / 0 | 100% / 0 | 0 |
| **総コスト/クエリ**（生成＋判定） | $0.0303 | **$0.0219** | **−28%** |
| Opus入力トークン | 88,499 | 54,197 | **−39%** |
| レイテンシ p95 | 13.0s | 10.2s | −22% |

![図4 コスト前後の比較表](https://static.zenn.studio/user-upload/deployed-images/1379d99aafa38a4e78fec35e.png?sha=63e4c27c50609b29e8925384fca8faa8d5ecc5d6)  
*図4：top-k 削減によるコスト前後の比較（同一セッションで測定）*

1クエリあたりのコストは −28%、Opus入力トークンは −39%、レイテンシ p95 は −22% となりました。一方で、検索品質を示す決定的指標は変化していません。

品質を落とさずに削減できた理由は、リランカーの構造にあります。リランカーは `k` に依存せず全候補を再採点してから上位 k を取り出すため、k を 5 から 3 に減らしても、リランカーが評価する候補プールは変わりません。すなわち、検索の品質はそのままに、生成器へ渡すトークンだけを削減できます。これが上表の左側（決定的ガード）が動かない理由です。

そして前節で述べたとおり、「品質が落ちていない」という主張は、ぶれる faithfulness ではなく、不変の recall・MRR・abstention rate に基づいています。

なお、プロンプトキャッシュは使用していません。一見するとコスト削減に有効に見えますが、本RAGは静的なprefixが小さく、出典ブロックがクエリごとに変わるため、キャッシュの採算が合わないと測定により判断しました。機能を「使わない」という判断にも、測定を根拠としています。

## システム構成と公開方法

最後に、このRAGの構成と、デモを常時無料で公開している仕組みを述べます。

データストアは1つのSQLiteファイルに集約しています。密ベクトル（sqlite-vec）・BM25（FTS5）・引用メタデータを同一の `index.db` に同居させているため、1つのサービスとしてデプロイできます。パイプラインは次のとおりです。

```
question
  → embed (Voyage voyage-4-lite)
  → dense kNN (sqlite-vec)  ∪  BM25 (FTS5)  → RRF で融合
  → rerank (Voyage rerank-2.5-lite)         （クロスエンコーダで rank を調整）
  → generate (claude-opus-4-8)：出典のみから回答／[n] 引用／根拠がなければ "I don't know"
  → answer + citations + trace（各段のレイテンシ・モデル別トークンコスト）
```

公開URLを常時無料で運用できているのは、キャッシュ済みデモの構成によります。実パイプラインを1回だけ実行した出力（回答・引用・実測コスト。コーパス本文は含みません）を `examples.json` に保存し、公開アプリはそれを配信します。そのため公開URLは、APIキー不要・依存は Streamlit のみ・`index.db` 非同梱で動作します。結果として $0 で常時稼働し、コーパスの再配布も発生しません（Anthropic のドキュメント本文は再配布しない方針です）。

この構成にかかった費用は、デモ生成（bake）を1回実行した際の $0.1967 のみで、以降は $0 で運用しています。

品質の担保として、LLM と HTTP の境界を Protocol で fake 化しており、APIキーとネットワークなしで 42個のオフラインテストが実行できます。検索・生成・判定・集計のロジックと、コストおよびレイテンシの計算は、すべてオフラインで検証しています。

## まとめ

本記事では、自作の技術文書RAGについて、検索品質・ハルシネーションの抑制・コストの3点を実測値で報告しました。

* **検索品質**：recall@1 / @3 / MRR で測定し、同一セッションのA/B比較により「素のRRFは順位を悪化させる」「リランクは hard スライスでのみ効く」ことを確認しました。
* **ハルシネーションの抑制**：4層で構成し、最終層を abstention rate 100% / false abstention 0 という測定値としました。faithfulness は run ごとにぶれるため、主要な数値としては用いていません。
* **コスト**：1クエリあたりの費用をモデル別に可視化し、recall@3 の飽和を根拠に top-k を削減して総コストを28%下げました。検索品質の決定的指標は不変です。

測定を通じて一貫させたのは、すべての主張を再現可能な指標に紐づけること、そして再現できない指標（faithfulness）は主張に用いず、その限界を明記することです。
