これは[16媒体に自動配信するコンテンツ基盤の全体アーキテクチャ](https://zenn.dev/mameresearcher/articles/ai-agent-multi-channel-content-ops)の深掘りシリーズの1本です。[前回](https://zenn.dev/mameresearcher/articles/ai-agent-persistent-memory-ssot)は「索引+詳細ファイルに分けた永続知識層」という設計思想を書きました。今回はその続きとして、**実際にどんなファイル形式で、どんな判定基準で運用しているか**を、抽象化せずに書き下します。

前回の記事は「考え方」でした。今回は「実装」です。

## 詳細ファイルのフォーマット: 4フィールドで固定する

詳細ファイル1件は、必ず次の4フィールドを持つ小さなテキストファイルにしています。

```
---
name: {ファイル名と対応するスラグ}
description: {何についてのファイルか。索引に出す1行サマリと同一}
metadata:
  type: {4分類のどれか}
---

{本文: 事実 + なぜ + どう使うか}
```

ポイントは\*\*`description`を索引の1行とファイル内フロントマターの両方で共有する\*\*ことです。索引側だけ別に文章を書くと、時間が経つにつれて「索引の説明」と「ファイルの中身」がズレていきます。1箇所を直せば両方に反映される構造にすることで、このズレを構造的に防いでいます。

## 4分類、それぞれの本文テンプレート

前回「知識を性質で分類する」と書きましたが、分類ごとに**本文の書き方そのものを変えている**のが今回の具体化ポイントです。

### 分類1: 方針・フィードバック — 「ルール + 理由 + 適用条件」の3行

```
{ルールそのものを1行で}

**理由**: {なぜそのルールが必要になったか。多くは過去の失敗や、ユーザーが強く訂正した経緯}
**適用場面**: {いつ、どんな場面でこのルールを思い出すべきか}
```

ここで一番重要なのは**理由を必ず書く**ことです。ルールだけを書くと、後で似ているが微妙に違う場面に遭遇したときに、そのルールを適用すべきか判断できません。「なぜ」まで書いておけば、エッジケースでも「この理由に当てはまるか」で判断できます。

### 分類2: 進行中の事実 — 「事実 + 理由 + 効いてくる場面」

```
{今こうなっている、という事実}

**理由**: {何がその状態を引き起こしているか。締め切り・制約・ステークホルダーの要望など}
**適用場面**: {この事実を踏まえてどう提案を変えるべきか}
```

方針・フィードバックとテンプレートは似ていますが、**性質が違います**。方針は「エージェントの振る舞い方」についての恒久ルール、進行中の事実は「今のプロジェクトの状態」についての一時情報です。同じ3行構造でも、**この事実は腐る(1〜2ヶ月で陳腐化する)という前提で扱う**かどうかが分類上の分かれ目になります。

### 分類3: 参照先 — ポインタのみ、内容は書かない

これは意図的に薄く書きます。「詳細はここを見ろ」という**ポインタの役割に徹し、内容そのものを二重管理しない**のが目的です。参照先の実体(外部システムの中身)は変わり続けるので、ここに詳細を書いてしまうと真っ先に陳腐化します。

### 分類4: 恒久的なやり方 — 「手順・トラップ + 発見の経緯」

```
{癖・トラップ・手順}

**発見の経緯**: {どういう失敗/検証でこれが分かったか}
```

3つの中で最も長寿命な分類です。コードのバグではなく「環境やツールの構造的な癖」を書くため、月単位・年単位で有効です。ただし「発見の経緯」を書いておかないと、後から見た時に「本当にまだこの癖は残っているか」を検証しづらくなります。

## 索引 (index) の1行フォーマット

```
- [{タイトル}]({ファイルパス}) — {60〜150字のフック}
```

このフックは\*\*「なぜこれが重要か」ではなく「これを見た瞬間に思い出すべき最短の手がかり」\*\*を書きます。重要度の説明ではなく、検索性を優先するということです。

索引には**200行を超えたら古いエントリを切り詰める**というハード制限を設けています。理由は単純で、索引自体が常にコンテキストに載る前提なので、索引が肥大化すると本末転倒になるからです。切り詰める基準は「直近で参照されていない」「対応するプロジェクトが既に終了している」の2点です。

## 書き込みタイミングの判定フローチャート

前回「書き込みはユーザーの訂正時と想定外事実の判明時」と書きましたが、実際にはもう一段階、判定が挟まっています。

この中で一番効果が大きかったのが\*\*「Q1: 今後も再利用されるか」で一度フィルタする\*\*ことでした。これを入れる前は、その場限りの作業メモまで永続層に書いてしまい、索引がすぐに肥大化していました。「これは半年後の別のセッションでも役に立つか」を毎回問うだけで、書き込み量が大きく減り、質が上がりました。

## 「時点のスナップショット」を守る具体的な運用

前回、知識層の記述は書いた瞬間のスナップショットに過ぎないと書きました。これを徹底するために、**推奨事項を含む詳細ファイルには「これは書いた時点の状態」という前提を明示し、行動に移す前に現物確認を挟む一手間を運用ルール化**しています。

具体的には、詳細ファイルの中で「〇〇というファイル/機能がある」という断定を書く場合、それが実際に今も存在するかどうかは**参照した時点で改めて検証する**という前提で扱います。ファイルパスや関数名など、リネームや削除で簡単に食い違う情報ほど、この検証をスキップしたくなる誘惑が強いのですが、そこを飛ばした瞬間に永続層が「誤情報の生産源」に転落します。

## まとめ

前回の「索引+詳細ファイル」という設計思想を、実際に運用可能なレベルまで具体化すると、次の4点に集約されます。

* 詳細ファイルは4フィールド固定(スラグ/サマリ/分類/本文)、サマリは索引と共有し二重管理を避ける
* 4分類それぞれに専用のテンプレートを持たせ、「恒久ルール」と「一時的な事実」を書き方のレベルで区別する
* 書き込み前に「再利用性 → 重複チェック → 再導出可能性」の3段フィルタを通す
* 記述は常に「時点のスナップショット」であり、行動前の現物確認を運用ルールとして明文化する

抽象的な設計思想は再利用しやすい一方、実装まで踏み込まないと「結局どう書けばいいか」で毎回迷います。今回書いたテンプレートとフローチャートは、そのまま別のプロジェクトに移植できる形にしたつもりです。
