## 0. 問い

あなたはLLMエージェントに機能を追加した。プロンプトを改善した、RAGを導入した、文脈情報を渡すようにした。

**「これ、本当に効いているか？」**

多くの場合、この問いへの答えは「なんとなくよくなった気がする」で終わる。評価が難しいからだ。LLMの出力は確率的で、毎回違う。人間が評価すると主観バイアスが入る。「効いている」の定義すら曖昧になる。

本稿は、EDD（Entity-Driven Development）会議という独自の仕組みで生まれた具体的な問い——「**システム地図（全体機能説明書）をAIに与えると、本当に要件の質が上がるのか？**」——に対して、科学的な制御実験を設計・全自動実装した話を記録する。

---

## 1. 検証したい仮説

EDD会議では、AIたち（デジタルツイン）がシステムの改善要件を会議で決める。各AIの発言生成時に**システム全体の構造を説明した「地図」をプロンプトに注入**すると、要件の質が上がるという仮説がある。

過去の対照実験では「地図ありのほうが診断的な発言になった」という観察があった。しかし、これは：

* 試行が1〜2回で統計的根拠に乏しい
* AIの状態（記憶・直前の会話）が試行ごとに異なる
* 評価者が仮説を知っていた（盲検ではない）

これを科学的に検証し直す。

```
H0（帰無仮説）: システム地図の有無は要件候補の質に差をもたらさない
H1（対立仮説）: 地図あり群は地図なし群より具体的・接地的・実装可能な要件を出す
```

---

## 2. 設計の難所：LLMエージェント実験ならではの問題

通常の心理学実験や薬効試験に使う実験設計をそのまま適用できない理由がいくつかある。

### 2-1. 被験者（TWIN）の状態が実験中に変わる

通常の実験では被験者が時間とともに変化しても問題ない（介入前後の比較が目的だから）。しかし今回は：

* 実験中も本番システムが稼働し、AIたちは日記を書き、ニュースを消費し、会話を積み重ねる
* 試行1（朝）と試行10（翌日）では、同じAIの「記憶状態」が変わっている

これは統計的ノイズになる。**試行間で被験者の初期状態を均一に保てない**。

### 2-2. DBリセットだけでは不十分

EDD会議を実施すると、AIたちのRAG（知識ベース）に会議の記憶が追加される。リセットで会議のレコードを削除しても：

```
試行1で「システム地図」を知ったAIが、
試行2で「地図なし」条件に入っても、
前回の地図の記憶がRAGに残っていれば...汚染が起きる。
```

EDD実験チャンクは `meeting_id` で識別して削除できるが、**試行間のRAK汚染リスク**は残る。

### 2-3. LLM出力の評価が難しい

要件候補の「質」をどう測るか。人間評価は：

* コストが高い（10試行×2条件×複数指標）
* 評価者バイアス（仮説を知っていると甘く評価する）

LLM評価は盲検にできるが、評価LLM自体の一貫性が問われる。

---

## 3. 解決策：TWINクローン + Within-Subject設計

```
「本番TWINを使わない。スナップショットのクローンで実験する」
```

### 3-1. TWINクローン方式

```
# clone_twins_for_experiment.py（一部）
clone = Twin(
    owner_id=owner_id,
    name=f"[実験] {src.name}",
    persona_prompt=src.persona_prompt,
    society_profile=src.society_profile,
    # ... 実験に必要なフィールドをコピー
)
db.add(clone)
await db.flush()

# knowledge_chunks（RAG記憶）も複製
for ch in source_chunks:
    db.add(KnowledgeChunk(
        twin_id=clone.id,
        content=ch.content,
        embedding=ch.embedding,
        metadata_=ch.metadata_,
    ))
```

これで：

* **全10試行が同一の初期状態からスタート**（試行間の状態ドリフトなし）
* **本番TWINへの影響ゼロ**（観察値が変化しない）
* **実験後はCASCADE削除で完全消去**（クリーンアップが容易）

### 3-2. Within-Subject設計

同じ7体のクローンが、Treatment（地図あり）とControl（地図なし）の両方を経験する。

```
1試行 = Treatment会議（地図あり）→ 測定 → DBリセット
       + Control会議（地図なし）  → 測定 → DBリセット
```

条件の順序は試行ごとにランダム化（順序効果の打ち消し）。

Between-Subject（グループを分ける）と比べると：

* 個体差（TWINのペルソナ差）が消えるため **統計的感度が高い**
* 半分のTWIN数で同等の検出力

### 3-3. DBリセットの実装

各試行後にEDD実験で追加されたデータのみ削除する。

```
# 会議に紐づいたRAGチャンクを特定して削除
EXPERIMENT_CHUNK_TYPES = {"meeting_material", "system_manual", "meeting_briefing", "meeting_minutes"}

await db.execute(
    delete(KnowledgeChunk).where(
        KnowledgeChunk.metadata_["meeting_id"].astext == str(meeting_id),
        KnowledgeChunk.metadata_["chunk_type"].astext.in_(EXPERIMENT_CHUNK_TYPES),
    )
)
# EDD会議レコードはCASCADE削除
await db.delete(meeting)
```

クローンは本番稼働に参加しないため、削除後のRAKは**クローン生成時の初期状態に完全に戻る**。

## 4. 評価設計：LLM盲検評価

人間評価は量的に無理なので、**Claude + Gemini に盲検で評価させる**。

```
def evaluate_with_claude(treatment_cands, control_cands):
    # ランダムにA/Bを割り当て（評価LLMは地図あり/なしを知らない）
    if random.random() < 0.5:
        a_cands, b_cands = treatment_cands, control_cands
        a_is_treatment = True
    else:
        a_cands, b_cands = control_cands, treatment_cands
        a_is_treatment = False

    prompt = f"""
    【グループA の要件候補】
    {format(a_cands)}

    【グループB の要件候補】
    {format(b_cands)}

    以下のJSONで採点してください（0〜5点）:
    {{"group_a": {{"concreteness": ..., "groundedness": ..., "feasibility": ..., "system_understanding": ...}},
      "group_b": {{...}}, "better_group": "A" or "B"}}
    """
    # → 採点後、A/Bをtreatment/controlに戻す
```

**Claude（実装者・アーキテクト）と Gemini（発案者）の2体で評価**することで：

* 相互チェックができる
* 評価者バイアスを分散できる
* どちらが「システム地図あり」かは知らされない

## 5. 定量指標：自動計測可能な5指標

LLM評価に加えて、自動計測できる定量指標も設計した。

| 指標 | 定義 | 期待 |
| --- | --- | --- |
| 接地スコア | 発言1件あたりの証拠数の平均 | 地図ありで高い |
| 証拠タイプ多様性 | 使用された evidence\_type の種類数 | 地図ありで高い |
| システム参照率 | 「RAG」「Celery」「embedding」等の用語出現率 | 地図ありで高い |
| 候補description文字数 | 要件候補の具体性の代理指標 | 地図ありで高い |
| 投票集中度 | 最多得票/総投票数（合意しやすさ） | 地図ありで高い？ |

## 6. 統計分析：Paired t-test + Cohen's d

10試行の結果に \*\*対応のあるt検定（Paired t-test）\*\*を使う。

```
diffs = [treatment_scores[i] - control_scores[i] for i in range(10)]

t_stat, p_value = scipy.stats.ttest_1samp(diffs, popmean=0)
cohens_d = np.mean(diffs) / np.std(diffs, ddof=1)

# 判定: p < 0.05 かつ d > 0.5 → H1採択
```

Between-Subject と違い、個体差が引き算で消えるため p値が出やすい（同じn数で高い検出力）。

## 7. 全自動・冪等設計

疑義が来たとき、**第三者が同じコマンドで再実験できる**ことを要件にした。

```
# 手順1: クローン生成
python scripts/clone_twins_for_experiment.py \
    --twin-ids 1 2 3 4 5 6 7 \
    --output results/experiment_config.json

# 手順2: 実験実行（設定から自動取得）
python scripts/run_edd_experiment.py \
    --config results/experiment_config.json \
    --trials 10 --seed 42

# 手順3: 評価レポート生成
python scripts/evaluate_edd_experiment.py \
    --results results/edd_experiment_YYYYMMDD.csv

# 手順4: クリーンアップ
python scripts/cleanup_experiment_twins.py \
    --config results/experiment_config.json
```

`--seed 42` で乱数を固定し、条件の順序を完全再現できる。

## 8. 見つけたバグ：実装の落とし穴

実験設計中に**重大なバグ**を発見した。

```
# 修正前: skip_manual=True（Control群）でも地図を常に注入していた
manual_section = (
    "【システム全体の地図】...\n"
    f"{manual}\n\n"
) if manual else ""  # ← skip_manualを見ていない！

# 修正後
manual_section = (
    "【システム全体の地図】...\n"
    f"{manual}\n\n"
) if (manual and not meeting.skip_manual) else ""
```

RAG配布は `skip_manual` でゲートしていたが、プロンプトへの直接注入を見落としていた。これでは Control群も Treatment群と同じプロンプトになってしまい、実験が成立しない。

**設計書を書いてから実装すると、こういった見落としを実装前に気づきやすい。** 今回は実装後に発見できたが、実験を走らせてから気づいていたら10試行分が無駄になっていた。

## 9. まとめ：LLMエージェント実験設計の教訓

| 課題 | 解決策 |
| --- | --- |
| 被験者（TWIN）の状態変化 | クローン方式（スナップショット固定） |
| 試行間のRAK汚染 | meeting\_idでタグ付け→精確なDBリセット |
| 評価の主観バイアス | LLM盲検評価（A/Bをランダム提示） |
| 統計的検出力 | Within-subject + Paired t-test |
| 再現性・冪等性 | 全自動スクリプト + 乱数シード固定 |
| 実装の落とし穴 | 条件分岐の抜け漏れ（設計書→レビュー→実装） |

LLMエージェント開発において「機能が効いているか」を問う需要は今後増えるはずだ。薬効試験・心理学実験の知見を持ち込みながら、**「LLMならではの難所（確率的・状態依存・評価困難）」に対して工学的に解を当てる**アプローチが求められている。
