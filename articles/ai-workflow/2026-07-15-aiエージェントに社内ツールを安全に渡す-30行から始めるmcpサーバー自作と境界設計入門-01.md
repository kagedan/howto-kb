## この記事のゴール

「Claude や Cursor に、うちの社内DBや社内APIを触らせたい。でも関数をそのまま呼ばせるのも、APIキーを丸ごと渡すのも、正直こわい」

僕（あきらパパ）自身、AIエージェントに実務ツールをつなぐたびに、この「こわさ」と向き合ってきました。結論から言うと、その不安の多くは **MCP（Model Context Protocol）サーバーを“供給側”として自分で設計する** ことでかなり解消できます。

この記事では、次の3つを一本の流れで扱います。

* **設計思想**: なぜ関数の直接呼び出しではなく、MCPという「境界」を挟むのか
* **動く最小実装**: TypeScript SDK で30行ほどの MCP サーバーを作る
* **安全に渡す設計**: 権限・入力検証・監査ログという3つの境界の引き方

## なぜ「関数を直接渡す」ではなくMCPなのか

AIエージェントにツールを渡す方法として、一番手軽なのは「自作の関数をそのままツール定義に載せる」やり方です。小さな検証ならこれで十分です。ですが、実務に持ち込むと3つの壁にぶつかります。

1. **N×M問題**: エージェント（Claude / Cursor / 自作アプリ）が M 個、社内ツールが N 個あると、素朴には N×M 本の接続を書く羽目になります。
2. **境界が消える問題**: 関数を直接渡すと、「AIが呼べる操作」と「アプリ内部の実装」が地続きになり、権限やログの差し込みどころが無くなります。
3. **再利用できない問題**: あるエージェント向けに書いたツール接続を、別のエージェントで使い回せません。

MCP は、ツール提供側を「サーバー」という独立プロセスに切り出します。これにより、**1つのMCPサーバーを書けば、MCP対応のどのホスト（Claude Desktop, Cursor, VS Code など）からも同じツールが使える**ようになります。N×M が N+M に畳まれる、というのが一番大きな設計上の利点です。

## MCPの構造を最短で理解する

MCP の登場人物は3つだけです。

* **Host**: LLMを動かすアプリ本体（IDE、チャットUIなど）
* **Client**: Host の中にいる、サーバー1つにつき1つの接続役
* **Server**: 実際に能力（ツールなど）を公開するプロセス。ローカルなら **stdio**、リモートなら **Streamable HTTP** といったトランスポートで接続します

そしてサーバーが公開できる「プリミティブ」は3種類です。この使い分けが設計の肝になります。

| プリミティブ | 役割 | 誰が主導して使うか |
| --- | --- | --- |
| **Tools** | LLMが呼び出す関数（API実行・DB照会・計算など） | モデル（AI）が主導 |
| **Resources** | 文脈として読ませるデータ（ファイル内容・レコードなど） | アプリ／ユーザーが主導 |
| **Prompts** | 引数付きの再利用テンプレート（定型ワークフロー） | ユーザーが主導 |

## 30行で動く最小MCPサーバーを作る

まずは手を動かしましょう。公式の TypeScript SDK と、入力検証用に zod を入れます。

```
npm install @modelcontextprotocol/sdk zod
```

以下が、stdio トランスポートで動く最小のMCPサーバーです。ここでは「社内の在庫数を返す」ダミーのツールを1つだけ公開しています。

```
#!/usr/bin/env node
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "inventory-server", version: "1.0.0" });

// ダミーの社内データ（本来はDB照会などに置き換える）
const STOCK: Record<string, number> = { "A-001": 42, "B-002": 0 };

server.registerTool(
  "get_stock",
  {
    title: "在庫照会",
    description: "商品コードから在庫数を返す",
    // 公式SDKの inputSchema は「Zodの生シェイプ」を渡す（z.object() で包まない）
    inputSchema: { sku: z.string().describe("商品コード 例: A-001") },
  },
  async ({ sku }) => {
    const qty = STOCK[sku];
    if (qty === undefined) {
      return { content: [{ type: "text", text: `未登録の商品コード: ${sku}` }], isError: true };
    }
    return { content: [{ type: "text", text: `在庫: ${qty}` }] };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
console.error("inventory-server started (stdio)");
```

Claude Desktop などのホスト設定にこのサーバーを登録すると、AI側から `get_stock` が見えるようになります。動作確認は、次のような自然文プロンプトで行えます。

> **プロンプト例1（正常系）**:  
> 「商品コード A-001 の在庫を確認して、発注が必要か教えて」

> **プロンプト例2（異常系の確認）**:  
> 「存在しないコード Z-999 の在庫を照会したら、どう振る舞う？そのまま試して」

> **プロンプト例3（複数ツール前提の設計確認）**:  
> 「get\_stock だけを使い、在庫0の商品だけを一覧にして。DBを直接書き換える操作は絶対にしないで」

このように「AIに直接ツールを叩かせて挙動を観察する」ところまでが、MCPサーバー開発の最小ループです。

最小実装ができたら、次は「何を Tool にし、何を Resource / Prompt にするか」の線引きです。僕がいつも使っている判断基準はシンプルです。

* **副作用（書き込み・外部実行）があるか？** → あるなら Tool、しかも最小権限で
* **ただ読ませたいだけの文脈か？** → Resource
* **毎回同じ指示を繰り返しているか？** → Prompt テンプレート化

Resource は URI テンプレートで動的に公開できます。

```
import { ResourceTemplate } from "@modelcontextprotocol/sdk/server/mcp.js";

server.registerResource(
  "product",
  new ResourceTemplate("product://{sku}", { list: undefined }),
  { title: "商品情報", description: "商品コードから商品メタ情報を返す", mimeType: "text/plain" },
  async (uri, { sku }) => ({
    contents: [{ uri: uri.href, mimeType: "text/plain", text: `商品 ${sku} のメタ情報...` }],
  })
);
```

「在庫を照会する」は副作用こそ無いものの、内部で外部DBを叩くので僕は Tool にしています。一方、「商品マスタの説明文をただ読ませる」なら Resource が適切です。**この線引きをサボると、AIが“読むだけでいいもの”まで能動的に叩きにいって、無駄なAPIコールとログ汚染が増えます。**

## 「安全に渡す」ための3つの境界設計

ここが本題です。MCPサーバーは「AIに実務ツールを渡す最後の関所」なので、次の3つの境界を必ず自分の手で引きます。

### 1. 権限の境界（最小権限ラッパ）

ツールの実処理は、必ず「許可された操作か」を確認する薄いラッパを通します。

```
type Action = "read" | "write";

function guard(action: Action, allowed: Action[]) {
  if (!allowed.includes(action)) {
    throw new Error(`この操作は許可されていません: ${action}`);
  }
}

// 読み取り専用サーバーとして起動したいなら allowed=["read"] を注入する
const ALLOWED: Action[] = ["read"];

server.registerTool(
  "delete_product",
  { title: "商品削除", description: "商品を削除する（書き込み）", inputSchema: { sku: z.string() } },
  async ({ sku }) => {
    guard("write", ALLOWED); // ← ここで write が許可されていなければ即座に弾く
    return { content: [{ type: "text", text: `削除しました: ${sku}` }] };
  }
);
```

「危険な操作はサーバー起動時のフラグで丸ごと無効化できる」状態にしておくと、検証環境では読み取り専用、本番でだけ書き込み許可、という運用が安全に回せます。

### 2. 入力の境界（スキーマ検証）

`inputSchema` に zod を使うのは「型のため」だけではありません。**AIは平気で想定外の値を渡してくる**ので、境界で弾くのが目的です。

```
inputSchema: {
  sku: z.string().regex(/^[A-Z]-\d{3}$/, "形式は X-000"),
  limit: z.number().int().min(1).max(100).default(20),
}
```

### 3. 監査の境界（誰が何を叩いたかを残す）

```
function audit(tool: string, args: unknown) {
  // 個人情報・秘密情報はログに残さない。SKUなど非機微な識別子だけを記録する
  console.error(JSON.stringify({ ts: new Date().toISOString(), tool, args }));
}
```

## 落とし穴と運用チェックリスト

最後に、実際に詰まりやすいポイントを畳んでおきます。

よくある落とし穴（クリックで展開）

* **stdout を汚す**: `console.log` でのデバッグ出力がJSON-RPCを壊す → ログは `console.error` へ
* **Tools に寄せすぎ**: 読むだけのデータまで Tool 化 → Resource へ逃がす
* **権限フラグが無い**: 破壊的操作を常時有効 → 起動時フラグで無効化できる設計に
* **入力を信用する**: AIの渡す引数を検証せず実行 → zod で境界検証
* **仕様バージョンの取り違え**: MCP仕様は日付でバージョニングされる（例: 2025-06-18 など） → 依存する機能はバージョンを明記

運用に載せる前のチェックリストはこの5つです。

1. 破壊的ツールは起動フラグで無効化できるか
2. すべてのツール入力にスキーマ検証があるか
3. 監査ログに機微情報が漏れていないか
4. Tool / Resource / Prompt の役割分担が説明できるか
5. 依存するMCP仕様のバージョンを固定・明記したか

## おわりに

MCPサーバーを自作する価値は「便利だから」ではありません。**AIに実務ツールを渡すときの“境界線”を、自分の言葉で引けるようになる**ことです。関数を直接渡していた頃の漠然とした不安が、「権限・入力・監査」という3つの具体的な設計判断に変わる。ここが一番のリターンだと感じています。

まずは今日の30行のサーバーを動かして、AIに叩かせてみてください。「AIが想定外の値を渡してきた」瞬間に、境界設計の必要性が肌でわかるはずです。
