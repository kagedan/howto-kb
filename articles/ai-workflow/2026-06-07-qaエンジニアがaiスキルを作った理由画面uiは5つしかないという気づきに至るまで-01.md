## はじめに

私はQAエンジニアです。開発者ではありません。

コードを書くことは仕事ではありませんが、  
「品質を保証するためにコードの構造を理解する必要がある」と考えました。  
特に既存機能の改修が入った時、「この改修はどこに影響するのか」を把握することが、  
テストの網羅性に直結します。

この記事は、その問いを解決しようとしてAIスキルを設計した過程と、  
設計中に気づいた「画面UIは5つしかない」 という考え方に至るまでの思考の記録です。

---

## 発端：エンジニアの調査時間を肩代わりしたかった

既存機能の改修では、エンジニアがまず「現在のコードがどう作られているか」を調査します。  
この調査には、慣れたエンジニアでも数時間かかることがあります。

そこで自分ができることは、**この調査を事前に終わらせておくことだ**と思い立ち、  
エンジニアが改修を始める前に「この画面はこういう構造になっていて、これらのファイルが関係します」という情報を渡せれば、調査時間を短縮できるのではなかろうか！  
そうに違いない・・そう思いました。

---

## 最初のアプローチ：「全体マップを作ろう」

最初に考えたのは、対象画面のコードファイルを俯瞰する「全体マップ」を作ることでした。

```
画面名とルートパスを入力
    ↓
関連するコードファイルを一覧化
    ↓
Mermaidのマインドマップで可視化
```

フロント側はコンポーネント・状態管理・API呼び出し、バック側はRoute・Controller・Service・Modelという技術的な分類で整理しようとしました。

しかしここで問題が出てきました。

---

## 転換点：「操作起点」で考える

そこで発想を変えました。技術軸ではなく、  
**「ユーザーが取れる操作」を起点にする**という考え方です。

```
保存する / 削除する / 画像を追加する / 公開を切替える
```

これならQAもエンジニアも直感的に分かる。テスト単位とも一致する。

さらに重要な発見がありました。  
操作を起点にすると、その操作の「起点から終点まで」辿れるということです。

```
往路：保存ボタンを押す → DBに保存される
復路：保存結果が返る  → 画面に反映される
```

改修の事故は往路だけでなく、**復路（保存後に画面がどう変わるか）で起きることが多い**。  
操作起点で辿ることで、往路と復路の両方が自然にカバーできます。

---

## もう一つの問題：ネストの深いファイルが見えない

操作起点で辿るアプローチには、別の問題がありました。

**「どこまで辿るかがAI任せになる」**

コードは呼び出しが何層にも連なっています。

```
Controller
  └─ ServiceA
       └─ ServiceB
            └─ ServiceC
                 └─ Toggle::active('xxx')  ← ここに埋まっている
```

上から辿るアプローチでは、深いところにあるコードに届かない可能性がある。  
特に**他機能のフィーチャートグル**がこの深い場所に潜んでいることがあり、  
これが過去にバグの原因になった実体験がありました。

> 自分の改修範囲のことはエンジニアから教えてもらえる。  
> でも**他機能の領域に絡むトグルは、誰も教えてくれない**。

---

## 根本的な解決策：「辿る」から「逆引き」へ

ここで発想の転換が起きました。

上から辿るのではなく、**「下から逆引きする」** という考え方です。

例えば「保存する」操作でDBのあるカラムが更新されるとします。そのカラムを**参照しているコードを全部探す**。

```
products.status が更新される
    ↓
products.status を参照しているコードを全件grep
    ↓
・商品一覧画面
・在庫管理画面
・受注画面
・検索インデックスの同期処理
・夜間バッチ
```

これなら**深さに関係なく、そのカラムを参照している箇所が全部出てくる**。何層深くても、ファイルシステムを横断するgrepには深さという概念がない。

---

## 最大の気づき：「画面UIは5つしかない」

逆引きの発想に至ったとき、次の問いが生まれました。

**「逆引きの起点は何か」**

保存される値はDBカラム。でも画面にはDBに保存されるもの以外もある。表示文言、機能の出し分け、権限による表示の違い……。

これらを整理していくと、ある結論に辿り着きました。

| # | 種別 | 内容 |
| --- | --- | --- |
| 1 | **DBカラム** | 保存・表示されるデータ |
| 2 | **翻訳キー** | 言語切替で変わるテキスト |
| 3 | **トグル** | 機能のオン/オフの切り替え |
| 4 | **権限** | ユーザーが何をできるか |
| 5 | **設定値** | config/環境変数による出し分け |

最初はこの主張を疑いました。「ボタンはどうなの？」という反論も来ました。

でも考えてみると、ボタンは\*\*「5つのどれかを変化させるトリガー」\*\*であって、それ自体が独立した要素ではありません。ボタンを押した結果は必ず5つのどれかに収まる。

```
「保存」ボタンを押す  → DBカラムが変わる
「公開」ボタンを押す  → トグルまたはDBカラムが変わる
「言語切替」を押す   → 翻訳キーが切り替わる
```

この5つは、そのまま**逆引きの起点**になります。

```
改修で何が変わったかを5つに分類する
    ↓
該当するキーで逆引きする
    ↓
影響範囲が出る
```

---

## 5つの分類の強さ

この分類が強い理由は、**どんな機能にも適用できる**ことです。

例えば「CSVダウンロード機能がどこで実装されているか調べたい」という場合。一見5つの分類に収まらなそうですが——

> CSVダウンロードボタンは画面に見えている。  
> 画面に見えているということは翻訳キーを持っている。  
> その翻訳キーで逆引きすれば、どこで実装されているか全部出てくる。

**画面に見えているものは必ずキーを持っている。キーがあれば逆引きできる。逆引きできれば影響範囲が出る。**

この3段論法が、あらゆる機能に対して成立します。

---

## 「AIへの指示書」としての5つの分類

この分類には、もう一つ重要な役割があります。

AIが「何を探すべきか」を迷わずに動けるようにする、**「AIへの指示書」** としての役割です。

「何でもいいから調べて」ではなく、**「この5つを探せ」** という明確な探索の軸を与える。

しかも5つそれぞれに「見つけたらこうする」が決まっています。

```
トグルが見つかった    → トグル名で全体grep
DBカラムが見つかった  → カラム名で逆引き
翻訳キーが見つかった  → キー名で逆引き
権限が見つかった      → 権限名で逆引き
設定値が見つかった    → 設定名で逆引き
```

**探す → 見つける → 逆引きする** という一貫したフローがAIに与えられる。これがAIを確実に動かすための設計です。

---

## このスキルで出せるテスト観点

最終的にこのスキルで出せるものを整理すると——

**正常系**

* 各操作が正しく動くか
* 各言語で正しく表示されるか（翻訳キー）
* 権限ごとに正しく出し分けられるか

**異常系**

**状態**

**影響範囲**

* 他機能への波及
* 共有コンポーネントへの影響
* 他機能のトグルとの絡み

---

## まとめ

このスキルの設計で得た最大の気づきは2つです。

### 1. 「辿る」より「逆引き」の方が深さに強い

上から辿るアプローチはネストの深さという問題が常につきまとう。逆引きは深さという概念がそもそも存在しない。

### 2. 画面UIは5つしかない

DBカラム・翻訳キー・トグル・権限・設定値。この5つが逆引きの起点になり、テスト観点の網羅性を保証する根拠になる。
