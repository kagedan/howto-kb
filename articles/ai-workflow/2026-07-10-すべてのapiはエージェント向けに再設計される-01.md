MCPは、エージェントと外部システムを接続する問題を解決しつつあります。しかし、接続しただけで、エージェントがAPIを正しく使えるようになるわけではありません。

丁寧に設計され、OpenAPIも整備されたREST APIをMCPで公開しても、エージェントは間違ったエンドポイントを選び、1回で済む処理に複数の呼び出しを使い、ときには確認なしで破壊的な操作を実行します。

APIの品質が低いのではありません。人間の開発者を前提に設計されているのです。

これまでAPIを使うのは、目的とシステムの構造を理解した開発者でした。一方、エージェントは、与えられたインターフェースから目的や操作手順を組み立てなければなりません。

利用者が変われば、インターフェースも変わります。この記事で主張したいのは、次のことです。

**エージェントが本番環境で利用するAPIは、既存APIのラッパーにとどまらず、エージェントネイティブな操作を中心に作り直されていきます。**

すべてのエンドポイントやプロトコルが消えるという意味ではありません。エージェントに任せるプロダクトレベルの操作が、リソース中心のAPIから、目的、状態、副作用、承認、復旧を含む契約へ変わっていくということです。

## MCPが解くのは「接続」まで

MCPは、Claude、ChatGPT、VS Code、Cursor、Codex、Copilotなどで利用できる、[クライアント横断の標準](https://modelcontextprotocol.io/docs/getting-started/intro)になりつつあります。接続、認証、通信、人間による確認、長時間実行されるタスクなどは、着実に標準化されています。

一方、MCPにおけるツールは、基本的には名前、説明、スキーマを持つ関数です。次のようなドメイン固有の設計までは決めません。

* どのような状態遷移があるか
* ツールをどの順番で使うべきか
* どの操作が危険か
* どの操作に承認が必要か
* 失敗したときにどう戻すか

プロトコルの責務としては、それで正しいと思います。ただし、既存のAPIをMCPで公開できることと、そのAPIがエージェント向けになることは別です。

ここには、**動詞のギャップ**があります。

既存のAPIは、リソースとCRUDを提供します。しかし、エージェントが必要としているのは、目的や副作用まで含んだ操作です。

MCPは接続を標準化します。操作の意味と安全性は、API側で設計する必要があります。

## なぜラッパーだけでは足りないのか

その兆候は、すでに見えています。

GitHubの[公式MCPサーバー](https://github.com/github/github-mcp-server)は、よくできています。それでもGitHubは、[デフォルトのツール数を減らし](https://github.blog/changelog/2025-10-14-github-mcp-server-now-supports-github-projects-and-more/)、Pull Request関連の操作を、より少数で高機能なツールへ統合しています。

APIのエンドポイントを1対1でツールに変換すると、エージェントが選ぶべき操作が増え、推論が難しくなるためです。

研究も同じ方向を示しています。[AutoMCP](https://arxiv.org/abs/2507.16044)では、OpenAPI仕様からMCPサーバーを生成し、約5,000のエンドポイントを対象に、ツール呼び出しの成功率を76.5%から99.9%まで改善しました。ただし、この改善は、元となるAPI仕様を修正した後の結果です。

また、[エージェント向けOpenAPIドキュメントに関する研究](https://arxiv.org/abs/2605.14312)では、構造的に正しいAPIであっても、エージェントが計画やツール選択に継続的に失敗することが示されています。

APIが仕様どおりに動くことと、エージェントが安全に使えることは別です。ラッパーは既存システムへの接続には有効ですが、元の契約までは変えられません。

プロダクトとして運用するには、操作そのものを再設計する必要があります。

## エージェントネイティブAPIに必要なもの

エージェントネイティブなAPIとは、単にエージェントから呼び出せるAPIではありません。エージェントが、操作の目的と結果を理解できるAPIです。

各操作には、少なくとも次の情報が必要です。

* 何を達成する操作か
* 現在どの状態にいるか
* どのような副作用があるか
* 承認が必要か
* 失敗した場合にどう戻すか

たとえば、次のような操作です。

```
inspect_pull_request
→ 生データではなく、現在の状態と重要な情報を返す

find_merge_blockers
→ マージできない理由を、統合された事実として返す

preview_write_action
→ 実行した場合に起きる変更を事前に返す

commit_approved_action
→ 明示的に承認された変更だけを実行する

explain_missing_permissions
→ 実行できない理由と、必要な対応を返す

rollback_last_action
→ 直前の操作を戻す
```

重要なのは、単にエンドポイントの粒度を大きくすることではありません。粗い操作でも、副作用が不透明で、承認の仕組みがなく、元に戻せなければ安全とはいえません。

エージェントネイティブな操作では、事前条件、期待される結果、不確実性、承認、復旧方法が契約に含まれます。`preview` と `commit` を分ければ、変更内容の確認と実行を分離できます。`rollback` を用意すれば、復旧も後付けではなく、最初から操作の一部になります。

## APIは状態モデルから設計される

現在のREST APIは、多くの場合、データベースから外側へ設計されます。テーブルがリソースになり、カラムがフィールドになり、CRUDが基本操作になります。

しかし、エージェントが知りたいのは、現在の状態と、次に実行できる操作です。その意味で、エージェントネイティブなAPIは、APIの形をした状態機械に近づきます。

今後は、設計の順番が逆になるかもしれません。まず状態遷移、ガード条件、実行可能な次の操作を定義し、その状態モデルを正としてAPIやストレージを設計します。

データベースの構造からAPIを作るのではなく、ドメインの正しい遷移からAPIを作るのです。

私が作ってみたいのは、この形です。

## 人間向けAPIと分ける必要はない

エージェント向けAPIを、人間向けAPIとは別に持ち続ける必要はないと思います。

実行前のプレビュー、影響範囲の確認、分かりやすい権限エラー、操作の取り消し、実行可能な次のアクション。これらは、開発者にとっても便利な機能です。

エージェント専用のAPIを作るというより、エージェントを主要な利用者として設計することで、結果的に人間にとっても扱いやすいAPIになります。

時間がたてば、エージェントネイティブな設計が、人間向けの利用も含むようになるはずです。

これは、[前回の記事](https://dev.to/gyu07/rigor-compresses-why-ai-agents-need-graphs-not-more-context-5404)で書いた話にもつながります。

よい操作は、複数の危険な手順を、一つの確かな動詞に圧縮します。厳密さが情報を圧縮するように、よい操作は手順を圧縮します。

## CLIもエージェント向けに変わる

これはREST APIだけの話ではありません。エージェントが操作する、あらゆるインターフェースの話です。

コーディングエージェントは、ターミナル上で `git` や `kubectl`、ビルドやデプロイのコマンドを実行します。これらのCLIも、人間がフラグや実行順序を理解していることを前提に設計されています。

CLIでは、出力にも問題があります。色、プログレスバー、自由形式のログ、最後の要約行など、多くの出力は人間が目で読むためのものです。エージェントは、それらを解析しなければなりません。

そのため、エージェントネイティブなCLIでは、次のような機械可読性が重要になります。

* `--json`
* 構造化ログ
* 安定した終了コード
* 次の対応を示すエラー
* 変更内容のプレビュー
* 明示的な非対話モード

呼び出し側には、意図を含んだ操作が必要です。戻り値側には、解釈が必要な文章ではなく、解析可能な状態が必要です。

API、CLI、ログ、出力形式。人間が受け手であることを前提に作られたインターフェースは、受け手がエージェントになれば見直されます。

## 安全性は操作に組み込む

MCPクライアントは、ツールの名前、説明、スキーマをモデルのコンテキストへ渡します。そのため、悪意のある説明による誘導、過剰な権限、権限を持つ主体が第三者に悪用される confused deputy のような問題が発生します。

さらに、エージェントは設定されたツールを自律的に使います。読み取り操作は許可リストである程度制御できますが、人間向けに設計されたAPIのまま、安全な書き込み操作を提供するのは簡単ではありません。

書き込みは、たとえば次のように再設計する必要があります。

```
preview
→ approve
→ commit
```

実行前に変更内容を確認し、承認された操作だけを実行します。失敗した場合に戻せることも重要です。

安全性は、APIの外側に追加するラッパーではありません。操作そのものに組み込む性質です。

## 移行期は3つの層が共存する

この変化は、すぐには完了しません。当面は、次の3つの層が共存すると思います。

| レイヤー | 内容 | 位置づけ |
| --- | --- | --- |
| **直接API・SDK・CLI** | ファイル、シェル、Git、テストなどへの直接操作 | ローカル性が重要な場所では残ります |
| **既存APIをラップしたMCP** | 人間向けAPIをツールとして公開します | 移行のための橋です |
| **エージェントネイティブAPI** | 状態、承認、復旧を含む目的中心の操作です | 最終的に収束する場所です |

既存システムでは、まずMCPによるラッパーを使い、現在ある機能をエージェントへ接続します。そのうえで、利用頻度やリスクの高い操作から、契約そのものを作り直していくことになります。

新規開発であれば、最初からエージェントネイティブに設計したほうがよいと思います。後から書き直す前提の、人間専用の契約を新しく作る理由は多くありません。

## よくある反論

#### すべてのAPIを書き直せるのか

レガシーは残ります。ただし、残ることと、新しくその形で作ることは別です。

SOAPはいまも動いていますが、新しいシステムをSOAPから始めることは多くありません。人間向けAPIもすぐには消えませんが、新規設計の基準ではなく、ラップして段階的に移行する対象になっていくと思います。

#### BFFとの違いは何か

技術の一部は新しくありません。変わったのは利用者です。

BFFは、人間が設計した固定的な画面フローを支えます。一方、エージェントは自分で計画を組み立てるため、操作側が事前条件、危険性、承認、復旧方法まで示す必要があります。

また、対象は一つのフロントエンドではなく、プロダクト全体の操作です。

#### モデルが賢くなれば解決するのか

読み取り操作については、そうかもしれません。しかし、モデルの能力が上がるほど、安全性の問題は大きくなります。

高性能なエージェントが、保護されていない破壊的操作を正確に実行できることは、解決ではなくリスクです。

#### ドメインごとの再設計はスケールするのか

これは重要な懸念です。

ここでの前提は、AIによってドメイン固有の実装コストが下がり、操作の再設計が現実的になることです。この前提が外れれば、移行には長い時間がかかります。

それでも、進む方向は変わらないと思っています。

## APIの契約は「リソース」から「意図」へ

これまでのAPIは、次の前提で設計されていました。

*人間が意図を持ち、APIは材料を返します。*

これからは、次の形になります。

*エージェントが目標を持ち、APIは状態、制約、次の安全な操作、そして戻る方法を返します。*

競争軸も、エンドポイントの数から、エージェントが安全に目標を達成できるかどうかへ変わります。

まだ、そのための正しい形式は存在していないと思います。RESTやGraphQLは、エージェントとは異なる利用者を前提に作られました。当面はOpenAPIやZodのような正確なスキーマが重要ですが、実装として正しいことと、エージェントが操作しやすいことは別の軸です。

後者には、いずれ独自の形式が生まれるのだと思います。

最近、プロンプトに次のような指示を書くことが増えました。

> 書き込む前にプレビューしてください。  
> 破壊的な操作の前には確認してください。  
> 実行できない場合は理由を説明してください。

エージェントを信用していないわけではありません。渡している操作のほうが曖昧なのです。

操作そのものに、プレビュー、承認、復旧が含まれるようになれば、こうした注意書きは必要なくなります。

操作が安全に設計されていれば、エージェントに「気をつけて使ってください」と頼む必要はありません。
