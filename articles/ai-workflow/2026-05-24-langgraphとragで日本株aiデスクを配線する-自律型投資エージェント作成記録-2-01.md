# LangGraphとRAGで日本株投資AIの配線を作る

前回の記事では、LTCMがなぜ崩壊したのかを、自分が作っている自律型投資エージェント `quant-agent` に二度と踏ませたくない失敗として整理した。

前回の主題はかなりシンプルだった。

**AIに任せれば勝てる、ではなく、AIを使って自分の過信を止めたい。**

今回はその続きとして、もう少し技術寄りの話を書く。

テーマは、

**日本株投資AIを、LangGraph、RAG、投資データ取得ツール、検証ゲートでどう配線するか**

である。

いきなり全市場を扱うと話が大きくなるので、今回は日本株デスクだけを例にする。

---

# 今回作るもの

今回考えているのは、日本株の投資判断を支援するAIデスクである。

たとえば、投資をしたことがない人でも、次のような言葉は聞いたことがあると思う。

* 高配当株
* 増配
* 自社株買い
* 決算が良かった会社
* 業績上方修正
* 円安で強い輸出企業
* 新NISAで買われやすい大型株

こういう投資テーマを、AIにただ雑に聞くのではなく、

1. どういう投資テーマなのかを分解する
2. 必要な情報をRAGから取り出す
3. 株価や開示情報を見る道具を使う
4. 候補銘柄を出す
5. 根拠が弱いものを落とす
6. 判断理由を記録する
7. リスクや昇格条件をゲートで確認する

という流れにしたい。

図にすると、最初の全体像はこうなる。

ここで大事なのは、AIが一発で「この株を買え」と言う構造にしないことだ。

AIには判断をさせる。  
でも、その判断は必ず、根拠、道具の出力、検証、ゲートを通るようにする。

---

# LangGraphとRAGとは？

この記事では、LangGraphとRAGという言葉が何度も出てくる。

AI開発をしている人にはよく聞く言葉かもしれないが、投資の話として読む人には少し分かりにくいと思う。

なので、先にかなり簡単に整理しておく。

---

# LangGraphとは何か

LangGraphは、AIの処理を「グラフ」としてつなぐための仕組みである。

ものすごく簡単に言うと、

```
AIに一発で答えさせるのではなく、
AIの仕事をいくつかの工程に分けて、
順番や分岐を管理するためのもの
```

だと考えている。

普通のAIチャットは、だいたいこういう形になる。

これはシンプルで便利だ。

でも、投資判断には少し怖い。

たとえば、

と聞いたとき、AIが一発でそれっぽい銘柄を出してきたら、見た目は便利に見える。

しかし、本当はその前に、

* 高配当の理由を確認する
* 業績が悪化していないか見る
* 増配が続いているか見る
* 株価が下がったせいで利回りが高く見えているだけではないか疑う
* 過去に似た条件で失敗していないか見る
* リスク条件を満たすか確認する

という複数の工程が必要になる。

LangGraphを使うと、このような工程をノードとして分けられる。

つまり、自分にとってLangGraphは、

**AIに投資判断を一発で言わせないための配線図**

である。

AIの頭の良さだけに頼るのではなく、

「どの順番で考えるか」  
「どこで道具を使うか」  
「どこで検証するか」  
「どこで止めるか」

を明示するために使う。

---

# RAGとは何か

RAGは、Retrieval-Augmented Generationの略である。

日本語にすると少し硬いが、簡単に言えば、

```
AIが答える前に、
必要な資料や過去の記録を検索して、
それを根拠にして答える仕組み
```

である。

普通のAIチャットは、AIがもともと学習している知識や、その場の会話だけで答える。

一方で、RAGを使うと、AIが答える前に、自分で用意した資料やデータベースを検索する。

投資AIでいうと、RAGには次のようなものを入れておきたい。

* 投資戦略のメモ
* 過去に失敗した条件
* リスクルール
* 使ってよいデータの説明
* 使ってはいけないデータの注意点
* 過去の判断ログ
* なぜ候補から外したかの記録

たとえば、「高配当株を探して」と言われたとき、AIが単純に配当利回りだけを見ると危ない。

配当利回りが高い理由が、

ではなく、

かもしれない。

だからRAGから、

```
高配当だけで判断しない。
業績悪化による見かけの高配当を疑う。
増配の継続性と利益の裏付けを見る。
```

という過去のルールを取り出して、AIに思い出させる。

自分にとってRAGは、

**AIを賢くするためだけではなく、AIに過去の失敗を忘れさせないための仕組み**

である。

---

# LangGraphとRAGの役割の違い

LangGraphとRAGは、似ているようで役割が違う。

自分の理解では、かなり雑に言うとこうだ。

| 仕組み | 役割 | 投資AIでの使い方 |
| --- | --- | --- |
| LangGraph | 考える順番を作る | 調査、検索、検証、判断、記録、ゲートをつなぐ |
| RAG | 必要な記憶を取り出す | 投資ルール、失敗記憶、過去ログ、設計方針を参照する |
| Tool | 外部データを見る | 株価、決算、配当、自社株買い、出来高などを取得する |
| Gate | 危ない判断を止める | 根拠不足、リスク超過、過去の失敗パターンを止める |

もう少し直感的に言うと、

```
LangGraph = AIの仕事の流れを決める配線
RAG = AIが参照する記憶
Tool = AIが外のデータを見るための道具
Gate = AIの判断を止めるブレーキ
```

である。

この4つを分けることが、自律型投資エージェントではかなり大事だと思っている。

AIに全部を一気にやらせるのではなく、

* LangGraphで流れを決める
* RAGで過去のルールを思い出させる
* Toolで現在のデータを見る
* Gateで危ない判断を止める

という形にする。

今回の日本株デスクでは、この考え方を使って、投資判断の流れを作っていく。

つまり、今回作りたいのは、AIが勢いで銘柄を挙げるチャットボットではない。

**AIが投資テーマを分解し、必要な記憶を検索し、データを見て、根拠を検証し、危ない判断を止めるための配線**

である。

# なぜLangGraphを使うのか

普通にLLMを使うだけなら、流れはこうなる。

でも、投資判断でこれは危ない。

たとえば、利用者がこう聞いたとする。

LLMは、それっぽい答えを作ることができる。

でも、本当に必要なのはそれっぽい答えではない。

必要なのは、少なくとも次のような確認だ。

* 本当に高配当なのか
* 配当利回りだけが高くて業績が悪くないか
* 増配が一時的ではないか
* 利益が伸びているか
* 自社株買いや業績上方修正などの材料があるか
* 株価がすでに上がりすぎていないか
* 流動性は十分か
* 過去に似た失敗パターンがないか
* リスク上限を超えていないか

これを1回のLLM呼び出しでやるのは危ない。

だから、処理をノードに分ける。

LangGraphを使う理由は、AIの判断を一発回答ではなく、状態を持ったワークフローとして管理できるからだ。

---

# 日本株投資AIのLangGraph設計

日本株投資AIでは、ざっくり次のようなグラフを考えている。

それぞれの役割はこうだ。

| Node | 役割 |
| --- | --- |
| Planner | ユーザーの依頼を分解する |
| Retriever | RAGから関連ルールや失敗記憶を取る |
| Tool Router | 使う投資データツールを決める |
| JP Scanner | 条件に合う日本株候補を探す |
| Evidence Builder | 候補ごとの根拠をまとめる |
| Verifier | 根拠不足や危険な候補を落とす |
| Decision Maker | 投資判断のドラフトを作る |
| Trace Writer | 判断理由を保存する |
| Gate Reviewer | リスクや昇格条件を確認する |
| Output | 利用者に説明する |

ここでのポイントは、AIの判断を「会話」で終わらせないことだ。

AIが何を見て、どの道具を使って、なぜその候補を残したのかを、後から追えるようにする。

---

# RAGはAIを賢くするためだけではない

RAGというと、AIに外部知識を与える仕組みだと思われがちだ。

それは正しい。

ただ、自分が投資AIでRAGを使いたい理由は、それだけではない。

むしろ重要なのは、

**AIが勝手に都合の良い判断をしないように、過去のルールや失敗を思い出させること**

である。

日本株デスクのRAGには、次のような情報を入れる。

たとえば「高配当株を探して」と言われたとき、AIが単純に配当利回りランキングだけを見ると危ない。

配当利回りが高い会社の中には、株価が大きく下がったせいで見かけ上の利回りが高くなっているものもある。

だからRAGから、次のような注意点を取り出したい。

* 高配当だけで判断しない
* 業績悪化による高配当化を疑う
* 増配の継続性を見る
* 営業利益やキャッシュフローも見る
* 自社株買いや業績上方修正と組み合わせる
* 過去に高配当だけで失敗したケースを確認する

RAGは、AIを自由にするためだけのものではない。

**AIを過去のルールと失敗に縛るためのもの**

でもある。

---

# 投資データツールをどう使うか

AIに投資判断をさせるなら、当然データを見る必要がある。

日本株デスクでは、次のようなツールを考えている。

| Tool | 役割 |
| --- | --- |
| price\_reader | 株価や出来高を見る |
| disclosure\_reader | 決算や適時開示を見る |
| dividend\_reader | 配当や増配を見る |
| buyback\_reader | 自社株買いを見る |
| financial\_reader | 売上や利益を見る |
| risk\_checker | リスク上限を見る |
| trace\_writer | 判断理由を記録する |
| gate\_monitor | 投資判断を次に進めてよいか確認する |

大事なのは、AIにツールを使わせること自体ではない。

大事なのは、ツールの出力をどう扱うかである。

たとえば `dividend_reader` が、

と返しても、それだけでは十分ではない。

`financial_reader` で利益も見る。  
`price_reader` で株価が上がりすぎていないか見る。  
`risk_checker` でリスクを確認する。  
`RAG` で過去の失敗パターンを確認する。

つまり、ひとつのツール結果だけで判断を完結させない。

この流れは、次のように配線する。

AIの判断を強くするというより、AIの判断を分解して、疑えるようにする。

ここが一番大事だと思っている。

# 具体例はシンプルにする

この記事では、難しい投資理論よりも、投資をしていない人でも聞いたことがある例で考える。

たとえば、次のような依頼だ。

```
日本株で、増配していて、業績も悪くなさそうな会社を探して
```

この依頼に対して、AIがいきなり銘柄名を出すだけでは危ない。

理想はこういう流れだ。

このときAIには、たとえば次のように答えてほしい。

```
候補は出たが、増配だけを根拠にはしない。

確認した点は次の通り。

1. 配当が増えているか
2. 利益が極端に悪化していないか
3. 株価がすでに急騰しすぎていないか
4. 出来高が少なすぎないか
5. 過去の失敗パターンに似ていないか

現時点では、投資候補として監視する価値はある。
ただし、リスク条件を満たさない場合は除外する。
```

ここで重要なのは、AIが強い口調で「買いです」と言わないことだ。

投資AIに欲しいのは、派手な断言ではない。

**判断を分解する力**  
**根拠不足を根拠不足と言う力**  
**危ない候補を落とす力**

である。

たとえば、日本株の判断に暗号資産のリスク設定が混ざると危ない。

AIが間違える以前に、データの配線が混ざっていたら、正しい判断はできない。

# 配線チェックで見ること

今のフェーズで重要なのは、新しい戦略をどんどん足すことではない。

むしろ、今ある配線が正しいかを見ることだ。

見るべきポイントは次の通り。

* 日本株のリクエストが `jp_equity` として処理されているか
* 日本株の候補に、他市場のデータが混ざっていないか
* RAGが正しい投資ルールと失敗記憶を取れているか
* ツールの出力がEvidence Bundleに入っているか
* Verifierが危険な候補を落としているか
* Decision Traceに判断理由が残っているか
* Gate Monitorがリスク条件を見ているか
* Dashboardで候補だけでなく、除外理由も見えるか

これを図にすると、次のようなチェックになる。

投資AIで本当に怖いのは、明らかなエラーではない。

静かに配線がズレているのに、AIがそれっぽい説明を出してしまうことだ。

だから今は、AIの回答を強くするよりも、AIの判断経路を疑えるようにしたい。

---

# 今回の設計で大事にしていること

今回の日本株投資AIでは、次の原則を置いている。

## 1. AIに一発で買い判断をさせない

AIは候補を出す。  
根拠を集める。  
検証する。  
説明する。

しかし、ひとつの回答で投資判断を完結させない。

## 2. RAGで過去の失敗を思い出させる

AIは新しいアイデアを出すのは得意だ。

でも、過去に失敗した理由を忘れやすい。

だから、RAGには成功パターンだけでなく、失敗パターンも入れる。

## 3. ツールの出力をまとめて検証する

ひとつのツール結果だけで判断しない。

株価、出来高、決算、配当、自社株買い、リスク条件をまとめて見る。

## 4. Decision Traceに残す

AIの判断をチャットで終わらせない。

なぜ候補にしたのか。  
なぜ除外したのか。  
どのデータを見たのか。

これを後から追えるようにする。

## 5. Gateで止める

AIが良さそうと言っても、それだけで進めない。

リスク条件、根拠の強さ、過去の失敗パターンを見て、止めるべきものは止める。

# まとめ

今回は、日本株投資AIを例に、LangGraph、RAG、投資データツール、Decision Trace、Gate Monitorをどう配線するかを整理した。

作っていて感じるのは、自律型投資AIで難しいのは、予測モデルそのものだけではないということだ。

本当に難しいのは、

* どの情報を見たのか
* どの道具を使ったのか
* どの候補を残したのか
* どの候補を落としたのか
* なぜその判断をしたのか
* その判断は後から検証できるのか

をつなぐことだ。

AIに「どの株を買えばいい？」と聞くのは簡単だ。

でも、自分が作りたいのはそこではない。

自分が作りたいのは、

**AIが投資テーマを分解し、RAGで過去のルールと失敗を参照し、ツールで根拠を集め、LangGraphで判断過程を配線し、Gateで危ない判断を止める日本株投資AI**

である。
