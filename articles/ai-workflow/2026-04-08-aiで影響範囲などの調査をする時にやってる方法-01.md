## はじめに

レバテック開発部の今井です。

業務システムの開発で神経を使う工程の一つが**影響範囲調査**です。エンティティ間のリレーションが複雑で、外部サービスとの連携も多い業務システムでは、1つの変更が思わぬ箇所に波及します。

影響範囲調査を人手でやると時間がかかりますし、漏れる可能性があります。かといって、AIでやろうとしてもそのままでは漏れてしまいます。

この記事では、影響範囲チェックシートを複数のAIエージェントに並列で調査させ、結果をスクリプトで機械的に検証する仕組みを紹介します。

!

* この記事では、影響範囲調査の中でセキュリティなどの影響範囲と呼ぶべきか微妙なものも影響範囲として扱っています。これには大きな理由はないです。
* レバテック開発部は全員がClaude Codeを利用できる環境であることもあり、この記事はClaude Codeを前提に書いています。他のツールを使ってる方は、各々で読み変えてください。

## この仕組みがやること

この記事で紹介する仕組みは、**事前に用意した影響範囲チェックシートをAIエージェントに渡して、コードを読みながら全項目を埋めさせる**仕組みです。

やっていることはシンプルです。

1. 人間が「何を確認すべきか」を影響範囲チェックシートとして事前に定義しておく
2. 影響範囲チェックシート1枚につきAIエージェント1体を割り当て、並列でコードを調査させる
3. 全エージェントの結果を1つのファイルに統合し、スクリプトで漏れがないか検証する

つまり、**「何を調べるか」は人間が決め、「実際に調べる」のはAIがやり、「漏れがないか」はスクリプトが保証する**という役割分担です。

## 全体像

![](https://static.zenn.studio/user-upload/646e78640006-20260405.png)

全体の流れは以下のとおりです。

この流れによって、事前に定義した影響範囲チェックシートに基づいて検証します。

1. **影響範囲チェックシート**（複数ファイル）を用意しておく
2. 影響範囲チェックシートの数だけ**AIエージェントを並列起動**し、コードを読みながら全項目を埋める
3. 全エージェントの結果を**1つの調査結果ファイルに統合**する
4. スクリプトでID採番や網羅性チェックなどの機械的検証を行う

## 影響範囲チェックシートの設計

### 影響範囲チェックシートとは

影響範囲チェックシートは、「この変更をするとき、何を確認すべきか？」をMarkdownファイルに書き出したものです。

例えば以下のようなファイルを用意します。

| 影響範囲チェックシート | 調査観点の例 |
| --- | --- |
| ドメイン | 各種のドメイン知識をもとにした調査 |
| データ基盤 | ナビゲーション、テーブルJOIN、外部連携 |
| セキュリティ | 権限・ロール、個人情報 |
| ユーザビリティ | フォームバリデーション、CSS |

### チェック項目と調査結果

各項目にはIDで一意な項番を振っておきます。AIがコードを読んで全項目を判定すると、項目IDがそのまま引き継がれた調査結果ができあがります。

![](https://static.zenn.studio/user-upload/c22f344c5e4f-20260405.png)

IDをテンプレートに固定しておくことで、「AIが違うタスクで調査しても、必ず同じ質問項目をカバーする」ことが保証されます。

## AIエージェントの並列実行

### アーキテクチャ

Claude Codeの同時に複数のエージェントを動かせる機能である `Agent Teams` を使い、**影響範囲チェックシートの数 + 1体**のエージェントを同時に起動します。

```
調査開始
    │
    ├── ドメイン影響範囲チェックシート担当エージェント
    ├── データ基盤シート担当エージェント
    ├── セキュリティシート担当エージェント
    ├── ユーザビリティ影響範囲チェックシート担当エージェント
    ├── ...
    └── AI独自調査エージェント（影響範囲チェックシートを見ない）
    │
    ↓ 全エージェント完了後
    結果を1ファイルに統合
```

### 影響範囲チェックシートエージェント

各エージェントは担当する影響範囲チェックシート1枚を受け取り、全項目をコードを読みながら判定します。

プロンプトの例はこちらです。

```
対象機能: {ユーザーから指定された機能名}
影響範囲チェックシート: {影響範囲チェックシートのファイルパス}

## 調査手順
1. 影響範囲チェックシートを読み込む
2. 全項目を [x] にし、必ず「影響あり」「影響なし」「要確認」のいずれかを明記する
```

重要なルールとして、影響がない場合も必ず「影響なし」と記載させるようにしています。これによって、すべての項目を確実にチェックさせます。

### AI独自調査エージェント

もう1体、**影響範囲チェックシートを一切読まない**エージェントを用意します。

このエージェントを用意する理由は、**影響範囲チェックシートは過去の知見の蓄積であり、未知の観点はカバーできない**からです。影響範囲チェックシートに縛られないエージェントを入れることで、「既知の観点の網羅性」と「未知の観点の発見」を両立させています。

AI独自調査エージェントの具体例も紹介します。AIモデルの力を活かすために、私は以下のように簡単な指示だけを渡しています。このように簡単な指示からしていって、必要に応じて指示を増やすのが良いと思います。

```
チェックシートを一切見ずにコードから自由に影響範囲を調査してください。
```

## 項番によるトレーサビリティと機械検証

この仕組みのもう一つのポイントが、**IDによる追跡と機械検証**です。

AIの調査結果に対して、スクリプトが「影響あり」の問題に結果IDを自動付番します。そして、項目IDと結果IDの両方を使って網羅性を機械的に検証します。

![](https://static.zenn.studio/user-upload/062c358f8990-20260405.png)

ポイントは以下のとおりです。

* **AIは結果IDを付けません。** 付番はスクリプトが機械的に行います。機械的にできることは機械的にやることで、重複や不整合を防いでいます
* **テンプレートの項目IDが全て結果にあるか**をスクリプトが照合します。AIが項目をスキップしていたら即座にFAILになります
* **影響範囲チェックシートの項目ID**と**調査結果の結果ID**という2種類のIDで、質問から問題まで一気通貫で追跡できます

## AIとスクリプトの役割分担

この仕組みの設計思想は、**AIにしかできないこと**と**スクリプトで確実にやるべきこと**を明確に分けていることです。

| 役割 | AI | スクリプト |
| --- | --- | --- |
| 影響の有無の判定 | コードを読んで判断 |  |
| 問題の具体的な記述 | 何がどう変わるか |  |
| 影響範囲チェックシートにない観点の発見 | AI独自調査 |  |
| 問題番号（ID）の付番 |  | 一意性を保証 |
| 全項目の網羅性チェック |  | テンプレートID照合 |

## この手法のメリット・デメリット

### メリット

* **再現性がある** — 影響範囲チェックシートのIDが固定されているため、異なるAI、異なるタスクで実行しても同じ質問項目が必ずカバーされます。属人化しません
* **網羅性が機械的に保証される** — 「全項目チェック済みか」「テンプレート項目を網羅しているか」をスクリプトが検証します。AIが項目をスキップしたら即座にFAILになります
* **未知の観点も拾える** — AI独自調査エージェントが影響範囲チェックシートに縛られずに自由に調査することで、テンプレートがカバーしていない影響を発見できます
* **トレーサビリティがある** — 全ての問題にIDが振られるため、調査結果 → 考察 → 仕様書と下流工程に問題を追跡できます
* **コンテキストが一定以下になる** — 各影響範囲チェックシートごとにAIエージェントを分けるため、1体あたりのコンテキストを一定以下に抑えられます

### デメリット

* **チェックシートの初期構築コストがかかる** — ドメイン知識を質問項目に落とし込む作業は人間がやる必要があり、最初はそれなりの工数がかかります
* **チェックシートにない観点は保証されない** — AI独自調査で補っていますが、チェックシートの質が調査の質を左右します。継続的に育てていく運用が必要です
* **AIの判定精度に依存する** — 「影響なし」とAIが判定しても本当に影響がない保証はありません。重要な変更では人間のレビューも併用すべきです

## おわりに

影響範囲調査は、AIに丸投げしてもうまくいかないことがよくあります。そのシステムに熟練していなければ、人間がやってもよく漏れます。  
この状況を解決するために、影響範囲調査チェックシートを作り、それを成長させていくのが大事であると考えました。

今回の仕組みでは、この影響範囲調査チェックシートをもとに属人化せずに対処できます。  
また、項番を振っていくのがAI時代にも変わらず重要と考えて、そのように仕組みを構築しました。

この記事を読んで、皆様の影響範囲調査に少しでも役立てられたら幸いです。
