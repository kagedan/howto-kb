> 本記事は、AIエージェント（Claude / 自作のマルチエージェント環境）との対話を通じて設計・実装した内容を、筆者が再構成したものです。AIとの協働およびAIによる校正の利用を開示します。記載した数値・仕様は2026年7月時点のもので、末尾に一次情報のリンクをまとめています。

## 発端：Sonnet 5 を手で切り替えたときの面倒くささ

きっかけは Claude Sonnet 5 のリリースでした。

将軍・家老・足軽からなる階層型のマルチエージェント環境（[前回記事](https://zenn.dev/masafumi_heijo/articles/791fbea6eddd68)）では、各エージェントのモデルを `config/settings.yaml` で管理しています。Sonnet 5 が出たのを知って、それまで Sonnet 4.6 を指定していた箇所を手で書き換えました。

書き換え自体は一度で済む作業です。ただ、そのとき「これを新版が出るたびに毎回やるのか」と面倒に感じました。

* 新版が出たことに気づくのが遅れる（自分でドキュメントを見に行かないと分からない）
* 気づいても、「本当に切り替えて大丈夫か」の確認は結局手作業になる
* エージェントの数だけ、この手間が積み重なる

「気づいたら自動で切り替わってほしい」——最初はそう考えました。しかし検討を進めるうちに、**全自動での適用はそれ自体が事故のもとになる**と分かり、方針を転換しました。

## 全自動アップデートは事故る——リスクの棚卸し

「新版が出たら即座に、自動で `settings.yaml` を書き換えて反映する」という設計を検討したときに洗い出したリスクは、次のとおりです。

| リスク | 内容 |
| --- | --- |
| 破壊的変更・挙動変化 | モデルの版によって対応機能や挙動が変わりうる。切り替えた直後にエージェントが想定外の動きをしうる（Sonnet 5 の具体例は表の下に後述） |
| コスト暴発 | 単価や導入価格（intro pricing）の条件を確認せずに切り替えると、想定外の課金増につながりうる（Sonnet 5 の具体例は表の下に後述） |
| ティア誤爆 | Haiku → Sonnet のような**ファミリーをまたぐ**誤爆で、意図しない上位モデルに切り替わってしまう |
| 未提供IDで起動失敗 | 案内されたIDがまだ全アカウントに展開されていない等の理由で、起動に失敗する |
| スクレイピング経由のインジェクション | 検知処理がWebページの内容をLLMに解釈させる設計だと、ページ内の悪意ある文字列がプロンプトインジェクションの経路になりうる |
| セッションの強制リセット乱発 | 自動切り替えのたびに稼働中のエージェントのセッションを切ると、作業中の文脈が失われる |

これらはどれも、「気づいたら自動で最新化されている」という体験の裏側で発生しうるコストです。**検知（気づくこと）と適用（切り替えること）を同じ処理に混ぜてしまうと、リスクまで一緒に自動化してしまう**——これが検討の結論でした。

### 実例：Sonnet 4.6 → Sonnet 5 で実際に起きる破壊的変更（公式ドキュメント確認済み）

上の表のうち、最初の2行には、Sonnet 5 で実際に確認できる具体例があります。

**破壊的変更・挙動変化。** Sonnet 5 では、大きく2つの変更がありました。

ひとつは、manual な Extended thinking の非対応化です。Sonnet 4.6 では使えた手動の thinking 指定が、Sonnet 5 では 400 エラーを返すようになりました。もうひとつは、新しいトークナイザーの採用です。これにより、同じテキストでもトークン数が約30%増加します。

なお、本システム（`config/settings.yaml`）を確認したところ、現時点では manual な Extended thinking の明示的な有効化や、temperature／top\_p／top\_k といった非デフォルトの sampling parameter の設定を行っているエージェントは存在しませんでした（将軍から足軽7号まで、すべて未設定）。そのため、これは本システムで実際に起きた事故ではありません。

ただし、これらのパラメータを明示的に使う構成では、非対応のモデルへ切り替えた直後に API リクエストがエラーとなり、処理が止まりうる点に変わりはありません。

**コスト暴発。** Sonnet 5 は、導入価格（入力 $2／出力 $10、いずれも 1MTok あたり）が 2026年8月31日まで適用され、9月1日からは標準価格（入力 $3／出力 $15、Sonnet 4.6 と同額）に切り替わります。この期限を把握せずに運用を続けると、切替日をまたいだ時点で想定外のコスト変動が生じます。

さらに注意したいのは、標準単価が Sonnet 4.6 と同額（据え置き）であっても、前述の新トークナイザーによってトークン数が約30%増えるため、同じ処理をさせたときの実コストは最大で3〜4割増になりうる点です。「単価が据え置き＝コストは変わらない」ではありません。

## 設計原則：検知と適用の分離（notify-only）

そこで採用したのが、\*\*「検知は自動、適用は人間の承認を経て人間が実行する」\*\*という分離です。

* 新しいモデル版が出たかどうかは、決定論的なスクリプトが定期的に（起動時に）確認する
* 見つかっても、通知するだけ。`settings.yaml` の書き換えや、実際の切り替えコマンドの実行は一切しない
* 切り替えるかどうか、いつ切り替えるかは、通知を受けた人間が判断する

この仕組みが `scripts/model_update_check.py` です。

## 検知器の実装の勘所

### LLMを介さない決定論スクリプト

検知処理は Python の正規表現ベースで完結しており、LLMにWebページの内容を解釈させる工程がありません。これにより、仮にページの内容が悪意あるものだったとしても、それがエージェントへの指示として実行される経路（プロンプトインジェクション）を作らずに済みます。

### 情報源は一次情報である公式ドキュメントの .md に固定

情報源は、ハードコードした1つのURL（Anthropic公式ドキュメントの `overview.md`）のみです。任意のURLを踏みに行くことはありません。

### 脚注汚染の罠

公式ドキュメントの表には、Bedrock／Google Cloud の列などに脚注番号が付いている箇所があります。単純な文字列抽出だと、モデルIDと脚注番号がくっついた文字列（例: モデルIDの末尾＋脚注番号）を、誤ってバージョン番号として解釈してしまう恐れがあります。

`model_update_check.py` では、「`Claude API (ID|alias)` を含む行だけ」から抽出したうえで、日付サフィックスや `-vN` を除去し、`^claude-(family)-(major)(?:-(minor))?$` という厳格な正規表現で検証することで、この誤認識を防いでいます。

### 同ティア限定／fixed:true はスキップ／重複通知の抑制

* 比較は同じファミリー（例: sonnet 同士）の中だけで行い、ファミリーをまたぐ提案はしない
* `settings.yaml` 側で `fixed: true` を付けたエージェントは、検知の対象から除外する
* 一度通知した候補は state ファイル（`logs/model_update_check.state.json`）に記録し、同じ組み合わせを再通知しない

## 起動時トリガ：PCを常時起動しない前提での工夫

このマルチエージェント環境は、PCを常時起動しているわけではありません。そのため、cron の定時実行はそのままでは機能しません（PCが起きていない時間帯には実行されないため）。かといって、毎回のシェル起動で無条件に検知処理を走らせるのも重すぎます。

そこで `~/.bashrc` に、**ブートごとに一度きり**実行されるガードを追加しました。

* `/proc/sys/kernel/random/boot_id` を比較し、前回の実行時と同じブートなら何もしない（取得できない環境では、`/proc/uptime` から近似したブートエポックにフォールバックする）
* 対話シェル（`$-` に `i` が含まれる場合）でのみ動作させ、非対話シェルやスクリプト実行の邪魔をしない
* `nohup ... &` ＋ `disown` でバックグラウンド実行し、シェルの起動をブロックしない
* 万一失敗しても、`|| true` でシェルの起動自体は壊さない

`sudo` や `/etc/wsl.conf` の変更は避けています。システム全体の設定に手を入れるのは影響範囲が大きく、今回の目的（起動時に一度だけ軽い検知処理を挟む）に対しては、リスクが見合わないと判断したためです。

## これをマルチエージェントで作った話

この仕組みそのものも、将軍 → 家老 → 足軽 → 軍師QC という普段のワークフローに乗せて作りました。将軍が方針（検知・通知のみ、適用は人間の承認）を決め、家老がタスクに分解し、足軽が実装し、軍師が品質をチェックする——という一連の流れです。この体制についての詳細は、[1本目の記事](https://zenn.dev/masafumi_heijo/articles/791fbea6eddd68)を参照してください。

## 現状：まだ「初めての新版」を待っている段階

正直に書いておくと、この検知器は、Sonnet 5 を**手で切り替えた後**に「次からは楽をしたい」と思って整備したものです。つまり、実運用で新版を検知して通知を出した実績は、この記事を書いている時点ではまだありません。次に Anthropic が新しいモデル版を出したときが、この仕組みにとって初めての実戦になります。

現時点で確認できているのは、既知のモデル一覧を入力として与えたときに、検知ロジックが意図どおりに振る舞うか——同じファミリー内の新版だけを候補として拾い、`fixed: true` のエージェントを除外し、一度通知した組み合わせを再通知しないか——という、ロジックの動作までです。「実際の新版のリリースで正しく発火するか」は、次の機会に検証することになります。

## 教訓：『適用しない』設計の価値

自動化を検討するとき、つい「全部を自動でやってくれる」ことをゴールにしがちです。しかし今回のケースでは、**危険な最後の一歩（実際にモデルを切り替える操作）だけを、意図的に人間に残す**ことが、最も安全で納得感のある設計でした。

* 検知：頻度が高く、間違えても実害が小さい → 自動化する価値が高い
* 適用：頻度が低く、間違えると実害が大きい（コスト・挙動変化） → 人間の判断を残す価値が高い

「どこまで自動化するか」ではなく、「どこを自動化“しない”ままにするか」を先に決める。今回は、この順番のほうがうまくいきました。

## まとめ

モデルの更新への追随は、検知と適用を分けることで、利便性と安全性を両立できます。

* 追随（気づくこと）は自動化してよい。ただし、決定論的で、任意のコンテンツを解釈させない設計にする
* 適用（切り替えること）は自動化しない。通知を受けた人間が、価格や破壊的変更を確認したうえで実行する

「追随は自動、決断は人間」——この線引きが、今回の一番の収穫でした。
