## やったこと

Claude Designを使い、同じくClaude Designで生成した架空のランニングコミュニティ（せいうちランニングクラブ）の完成済みサイト一式を解析させて、**PC表示（1280px想定）のグレースケール・ワイヤーフレーム**を生成した。

AIで作ったサイトなので、最初から設計図が存在しない。完成形だけがある状態だ。将来の拡張を考えたとき「骨格を可視化しておきたい」と思い、Claude Designに逆算させてみた。

渡したのはHTMLファイル一式が入ったフォルダだけ。プロンプトは以下。

```
添付したHTMLファイルを解析して、PCサイト向けのワイヤーフレームを作成してください。

【作成条件】
- 既存のページ構成・コンテンツ・ナビゲーション構造を忠実に読み取ること
- PC表示（横幅1280px想定）のワイヤーフレームとして作成すること
- 画像・色・装飾は省いたグレースケールのワイヤーフレーム形式にすること
- 各セクションにラベルとレイアウトの意図を簡単に注記すること
- まずはトップページ（index.html）のワイヤーフレームから始めること

【出力してほしいもの】
1. トップページのワイヤーフレーム（全セクション）
2. 主要ページがあれば、そのワイヤーフレームも続けて作成
3. ページ間の遷移・リンク構造のメモ
```

特別なことは何も言っていない。「忠実に読み取って、グレースケールで、注記つきで」という条件だけだ。

![](https://static.zenn.studio/user-upload/3ffb086f597a-20260629.png)  
*Claude Designでは既存の他プロジェクトやフォルダを読み込ませ、新しいデザインやワイヤーフレームを生成することも可能。*

## 最初に驚いたこと——ファイルを見ただけで構造を読んだ

フォルダを渡してまず返ってきたのが、ファイル構成の解釈だった。

```
assets/
├── css/
│   ├── main.css
│   └── extra.css
├── fonts/
│   ├── Saira_600_normal_001.woff2
│   ├── ...
│   ├── Zen_Kaku_Gothic_New_400_normal_121.woff2
│   └── ...（合計500超）
└── images/
```

`fonts/` に500個以上のwoff2ファイルが並んでいるのを見て、Claude Designは「Google Fontsをサブセット化してセルフホストしている、和文に注力したサイト」と判断した。HTMLは1ファイル（`index.html` / 570行）のみ。これだけで「シングルページ構成、ナビはアンカーリンクのはず」と当たりをつけていた。

本文を読む前に、ディレクトリ構造から設計の輪郭を読む——自分でやれば数分かかる作業が、ほぼ即座に終わっていた。

---

## セクション構成の解析

続いてHTMLの `id` を辿り、全体像をテーブルに整理してくれた。

| # | セクション | id | 役割 |
| --- | --- | --- | --- |
| – | Hero | `#top` | 100vh フルスクリーン、キャッチコピー |
| 01 | About | `#about` | 活動理念（2カラム） |
| – | 区切りバンド | – | アクセント全面＋マーキー |
| 02 | Activity | `#activity` | コース紹介（地図カード×2） |
| 03 | Members | `#members` | メンバー紹介（背景反転・グリッド） |
| 04 | Gallery | `#gallery` | 写真2枚 |
| 05 | Schedule | `#schedule` | 予定リスト（タブ切替） |
| – | Footer | – | 巨大アウトライン文字＋ナビ再掲 |

さらに「各セクションの見出しが『番号＋英語＋区切り線＋日本語』という共通フォーマットで組まれている、これはデザインのリズムなのでワイヤーフレームにも反映させる」というコメントが添えられた。こういう"設計の意図の読み取り"は、人間に頼んでもできる人とできない人がいる仕事だ。

---

## 一番驚いたのはここ——「見えない状態」まで拾っていた

HTMLの末尾にあった状態管理のJavaScriptまで読んだのが、正直一番驚いたポイントだった。

```
class Component extends DCLogic {
  state = { scheduleTab: 'upcoming', lightbox: null,
            menuOpen: false, selectedMember: null };
  schedule = [
    { date: '2026-05-07', title: 'TOKYO SPEED RUN', loc: '明治神宮外苑', type: '大会' },
    ...
  ];
}
```

このコードから、画面に表示されていないインタラクションの仕様を全部拾い出してきた。

* **Schedule**：タブで「今後の予定 / 過去の実績」を切替、TODAY基準で自動振り分け
* **Members**：アバタークリックで詳細モーダルが開く、左右キー・Escで操作
* **Members詳細**：Instagramリンクは任意項目（現状は未設定）
* **モバイル**：ハンバーガーメニューで全画面メニューに展開

スクリーンショットを渡すと「見えている1状態」しか拾えない。コードを渡すと「起こりうるすべての状態」が設計に入る。この差は思っていたより大きかった。

## ワイヤーフレームの出力方針

Claude Designが自律的に決めて実行した方針が3つある。

**① 凡例を最初に定義**  
UIブロック・画像プレースホルダー・テキストバー・セクション番号・注記の5種類に記号を統一。「凡例があることで読み手の認知負荷が下がる」という判断をこちらから指示せず、自分でやっていた。

**② 実コピーを活かす**  
「忠実に」という指示を受けて、ダミーテキストは使わず実際の見出し（「速さは、目的じゃない。」など）をグレーで残し、本文だけプレースホルダーのバーにした。

**③ 注記を付箋スタイルで**  
各セクションの設計意図（「2カラム：左＝見出し＋理念リスト／右＝本文＋引用」など）を手書き風フォントの黄色い付箋で添えた。グレースケールの図面に対して注記だけ色がある、という判断も自動で行っていた。

最終的なキャンバスには3フレームが並んだ。

1. トップページ全セクション（1280px）
2. メンバー詳細モーダルの状態
3. リンク・遷移構造マップ＋構成メモ

モーダルと遷移を本体スクロールとは別フレームに切り出した理由も、「独立した構造なので分けた方が伝わる」ときちんと説明してくれた。

---

## 遷移構造の整理

```
グローバルナビ / ロゴ
   └─ すべてページ内アンカー（スムーズスクロール）
        #about → #activity → #members → #gallery → #schedule

外部リンク
   ├─ 地図      → Google Maps 埋込（Activity内）
   └─ メンバー  → Instagram（任意・現状未設定）

ページ内アクション
   ├─ Schedule タブ切替（今後 / 過去）
   ├─ Members アバター → 詳細モーダル（←→ / Esc）
   ├─ Back to top
   └─ モバイル：ハンバーガー → 全画面メニュー
```

「画面遷移図ではなく状態遷移図になった」というコメントも自動で付いていた。的確すぎる。  
![](https://static.zenn.studio/user-upload/5bf186666360-20260629.png)  
*Claude Designが出力したワイヤーフレーム。左ペインにトップページ全セクション（1280px）、右ペインに遷移・リンク構造マップとメンバーモーダルの状態が並ぶ。*

## かかった時間とトークン数

| フェーズ | 時間 |
| --- | --- |
| フォルダ渡し〜解析完了 | 約3分 |
| ワイヤーフレーム初稿生成 | 約2分 |
| 注記・遷移構造の仕上げ | 約3分 |
| **合計** | **約8分** |

消費トークンの参考値：

* 入力（HTML＋フォント定義など）：約20,000〜25,000 tokens
* 出力（ワイヤーフレーム＋注記＋遷移メモ）：約10,000〜13,000 tokens
* **合計：約30,000〜38,000 tokens 程度**

Claude Proの週次制限に対してそれなりの割合を使う。HTMLが重いサイトは入力トークンが増えるので、複数ページを一気に渡すよりも1ページずつ確認しながら進める方が安全だと思った。

---

## やってみて思ったこと

正直、「ワイヤーフレームを出してくれる」という事実よりも、**出力の判断の精度**に驚いた。

凡例をつける、実コピーを活かす、注記を付箋スタイルにする、モーダルを別フレームに切り出す——これらはどれも、こちらから指示していない。「ワイヤーフレームとして正しい判断」を自分でしていた。

スクリーンショットではなくHTMLを渡したことで、見えていない状態（タブの裏・モーダル・モバイルメニュー）まで設計に入れられたのも大きい。結果として、「画面を見て描き起こしたワイヤー」より情報量が多い成果物になった。

AIが作ったサイトの設計図が存在しない問題は、AIに逆算させると解決できる。そういうループが自然に回り始めている。

## まとめ

* Claude Designにフォルダを渡したら、約3分でワイヤーフレームが出てきた
* ファイル構成を見るだけで設計の輪郭を読む——本文より先に構造を把握する
* JavaScriptのstate管理まで読んで「見えない状態」をワイヤーに組み込む
* 凡例・注記・フレーム分割の判断を自分でしていた

完成品をいったん設計図へ戻すと、そのデザインがどれだけ意図的に組まれていたかが透けて見える。リバースは、いいデザインへのいちばん丁寧な読書だと思う。

デモとして単一のHTMLファイルを読ませただけなのでワイヤーフレームの意義がほぼないが、将来的には画面遷移のながれなどもまとめさせたい。(まずはアプリをつくらないと、、)
