## はじめに

Claudeでアプリ開発のプロジェクト立案から実装、テストまでをやっていたときにやりたいことを無限に同じチャットでやっていたらチャットのやり取りが長くなりすぎて**応答が返ってこなくなった**という経験をしました。

途中まで積み上げたコンテキストが全部消えたような絶望感…。でもその失敗を機に「Claude Projectsをちゃんと使いこなそう」と試行錯誤した結果、かなり快適に開発できるようになったのでまとめます。

---

## Claude Projectsとは

Claude Projectsは\*\*「目的別に文脈を固定する作業空間」\*\*です。通常のチャットとの違いは以下の3点：

|  | 通常チャット | Claude Projects |
| --- | --- | --- |
| 前提条件の引き継ぎ | 毎回説明が必要 | プロジェクト指示に一度書くだけ |
| ファイル・仕様書 | 毎回アップロード | ナレッジに登録して常時参照 |
| 複数チャット | 独立している | 指示・ナレッジを共有 |

**重要：** チャット間で会話履歴は共有されません。共有されるのは「プロジェクト指示」と「ナレッジ（アップロードしたファイル）」だけです。

---

## チャットが壊れたときの対処法

長くなりすぎたチャットが動かなくなってしまったとき、私が試した方法です。

### 1. コードファイルを直接アップロード

一番確実な方法。作成済みの `.swift` / `.kt` / `.tsx` などをプロジェクトのナレッジにアップロードするだけで、Claudeがコードを読んで現状を把握してくれます。

### 2. ブラウザからテキストをコピー

チャットが固まっていても、ブラウザ上でスクロールしてテキストを手動コピーできることが多いです。重要な部分だけ新チャットに貼れば引き継ぎOK。

### 3. Gitリポジトリに集約する

実はこれが一番再現性が高かった方法です。すべての成果物をGitにコミットしておけば、新しいチャットでリポジトリの差分（diff）やREADMEを渡すだけで引き継げます。

### 4. 最低限のメモを手書き

完璧じゃなくていい。こんな感じのメモで十分引き継げました：

```
- iOSのTodoアプリ（Swift + SwiftUI）
- 一覧・詳細・設定画面まで完成
- CoreDataのiCloud同期で詰まっている
```

## プロジェクト指示のテンプレート

最初にこれを書いておくだけで、毎回の説明が不要になりました。

```
## アプリ概要
- アプリ名：〇〇アプリ
- 種類：iOS / Android / クロスプラットフォーム
- 目的：〇〇ユーザーが〇〇できるアプリ

## 技術スタック
- 言語：Swift / Kotlin / React Native
- フレームワーク：SwiftUI / Jetpack Compose
- バックエンド：Firebase / Supabase

## 現在の進捗
- 完成している機能：〇〇, 〇〇
- 現在作業中：〇〇

## コーディングルール
- コメントは日本語で書く
- 変更後は必ずビルドが通ることを確認してから報告する
- 変更点は箇条書きで説明する

## 禁止事項（iOSの場合）
- .pbxprojファイルは絶対に変更しない（Xcodeで手動追加する）
```

**iOSユーザーへ：** `.pbxproj` をAIに変更させると、プロジェクトが破損して何時間も溶けます。絶対に指示に明記してください。

---

## チャット分割の戦略

**「1つのチャットに全部詰め込まない」** これが一番大事なことでした。

私は以下のように分割するようにしました：

* **チャット①** 設計・仕様決め（spec.md作成）
* **チャット②〜** 機能ごとに1チャット（一覧画面、詳細画面、API連携…）
* **チャット：バグ修正専用** 別枠にしておくと混乱しない

新しいチャットを始めるときは冒頭に一言添えるだけでOKです：

> 「前回の続きです。〇〇機能を実装した状態で、次は〇〇を作りたいです」

## ナレッジに入れると効果的なもの

| ファイル | 効果 |
| --- | --- |
| 画面設計書・仕様書 | どの画面が何をするかをClaudeが把握 |
| API仕様書 | エンドポイントの使い方を即答してくれる |
| データモデル定義 | 整合性の取れたコードを生成してくれる |
| 既知バグ・制限事項メモ | 同じミスを繰り返させない |
| 依存関係一覧（Package.swiftなど） | ライブラリバージョンの齟齬を防ぐ |

何でも入れればいいわけではありません。**目的に直結するものだけに絞る**とClaudeの回答精度が上がります。

## 開発中に気づいたTips

**計画から始める**  
いきなりコードを書かせず「まず設計を考えて」と伝えてから始めると、成果物の品質が全然違います。

**「シニアエンジニアのように実装して」と添える**  
これを入れるだけでハードコーディングや雑な回避策が減りました。

**変更のたびにビルドさせる**  
「変更後は必ずビルドして確認してから報告すること」と指示に書いておかないと、動かないコードが積み重なっていきます。

**デバッグログを必ず入れさせる**  
「Loggerを追加してください」と毎回言うより、プロジェクト指示に書いておく方が楽です。

## まとめ

**「チャットは使い捨て、ナレッジは資産」** という考え方に切り替えてから開発効率が大きく変わりました。

* チャットが長くなったら迷わず新しいチャットを作る
* 引き継ぎに必要な情報はコードファイルとナレッジで管理する
* プロジェクト指示に一度書いておけば毎回説明しなくてよくなる

最初にちゃんと設定する手間が、後から何倍もの時間節約になります。
