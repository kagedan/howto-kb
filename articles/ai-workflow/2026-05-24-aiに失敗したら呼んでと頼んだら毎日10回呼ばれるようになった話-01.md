# AIに「失敗したら呼んで」と頼んだら、毎日10回呼ばれるようになった話

副業で AI に bot のコードを書かせていた最初の月、私は AI に「エラーが出たら止まって人間を呼んで」をデフォルトにしていました。

3週間くらい、毎日10回くらい呼ばれていました。

朝、コーヒーを淹れたら呼ばれている。昼休みに戻ったら呼ばれている。夜、テレビを見ていたら呼ばれる。寝る前にスマホを見たら呼ばれている。

中身を開くと、ほとんどが「ImportError」「型ヒント漏れ」「テストの fixture 名のタイプミス」みたいな、5秒で直るやつでした。

> これ、自分でやらなくてよくない？

そう思って、エラーが出たときの振る舞いをルールに書き出したら、呼ばれる回数がかなり減りました。副業で AI を使い始めて起きたことを、いったん書き残しておきます。

## 「1回失敗したら呼ぶ」をデフォルトにしていた理由

「エラーが出たら止まって人間を呼ぶ」に設定していた理由は、AI に勝手に直されて事故るのが怖かったからです。

副業で bot を動かしている。何かおかしいことが起きたとき、自分の目で確認したかった。「勝手にいろいろ直されると、何が起きたか追えなくなる」という気持ちでした。

結果として、5秒で直せるエラーまで全部自分が見ることになりました。

* import 漏れ → import を追加
* 型ヒント漏れ → 型を足す
* テストの fixture 名タイポ → 直す
* 非同期処理の await 漏れ → await を足す

私が毎回「もう1回」と打っていました。AI に頼んで bot を作らせているのに、私が AI の手足になっていた3週間でした。

「自分で確認したい」と「全部自分が直すのは嫌」が矛盾していたんですが、その矛盾に気付くのに3週間かかりました。毎日10回呼ばれながら、当時は「まあ仕方ないかな」と思っていました。

## git diff を見ても分からなくなったことがある

3週間の終わりごろ、bot が書き換えの途中で止まったことがありました。

いつもどおり「呼ばれた」と思ってログを開いたら、何かがおかしい状態でした。直す前の状態に戻そうとしたんですが、「どこまで書き換えが終わっていたか」が分からない。

git で差分を見たんですが、どこからが AI の変更で、どこからが自分が手で直した部分か、という境目が曖昧になっていました。AI に呼ばれるたびに私が少し手を入れていたので、git log が「AI のコミット」と「私が直したあとのコミット」で交互に並んでいる。どの時点の状態に戻せばいいか分からない。

「戻したいだけなのに、まず戻す対象を探すところから始まった」という状態で、1時間くらいかかりました。ファイルを戻しているだけで時間が溶けていく感覚は、あまりよくないです。

「こういう状態にならないようにするほうが大事だな」と、そのとき初めて思いました。戻し方を考える前に、戻せる状態を作っておく、ということの意味が、その1時間で分かりました。

## 毎回「もう1回」を打つのを諦めた

3週間後に諦めたのは、「1回失敗で人間が見る」という運用でした。

エラーが出るたびに自分が確認する、というのが続かなかった。続かない運用を続けようとしていただけで、5秒で直るエラーまで全部見ようとしていた私が、状況を複雑にしていたんだと思います。

ルールを雑に書き出しました。

> エラーが出たら、まず原因を分析する。同じアプローチで2回まで直してリトライ。それでもダメなら別アプローチに切り替える。3アプローチ全滅で初めて人間を呼ぶ。

「1回失敗で呼ぶ」から「3アプローチ全滅で呼ぶ」に変えただけです。

書く前は AI が勝手に直すのが怖かったんですが、毎回「もう1回」を私が打ち続けている現状の方が、たぶん事故率は高かった気がします。私が手で直しても、AI が直しても、直してリトライしているだけ、という意味では同じでした。

## 「同じことを3回やる」も止めないといけない

retry のルールを入れた直後、別の問題が出ました。AI が同じエラーで3回 retry を回すことがありました。

> curl が失敗した → 別の URL で curl → ヘッダー変えて curl → また別の URL で…

明らかに通らないやつをぐるぐる回していました。retry だけ書いていて、「同じアプローチで2回失敗したら諦める」を書いていなかったから。

なので、切り替えのルールも一緒に書きました。

* CLI がダメなら API で試す
* API がダメなら Python から直接試す
* それも全部ダメなら人間に「代替手段を聞く」

retry とスピン防止は、片方だけ書くと壊れます。両方書いてようやくなんとかなる、という感じでした。これも一発では書けなくて、何回か詰まりながら書き足していきました。

## retry してはいけないものを先に決めた

retry のルールを入れたあと、もう一個書き足したのが「retry しない」のリストでした。

* 認証エラー（API キー不在、トークン切れ）
* 設計判断が必要な失敗（仕様が曖昧で実装できない）
* DB スキーマの破壊的変更が必要なケース
* 同じエラーで2回失敗済み

特に最初のやつが、書き出すまで踏み続けていました。API キーが入っていないのに retry を回し続けて、3アプローチぶん時間を使ってから「キーがない」と気付く、というのを何回かやっていました。

> 「これは retry しても直らない」を先に書いておくしかない

これを書いてから、こういう詰まり方はなくなりました。エラーを見て「これは認証系だ」と判断したら即エスカレーション、それ以外は retry、というだけです。

## timeout は「人間に戻す境界」として使う

timeout の設計でハマったのが、「時間が来たら失敗にする」という発想で書いていたことでした。

AI の処理が時間かかる理由は2種類あります。

ひとつは、処理が本当に重いやつ。長く待てば終わる。これは timeout を切るより、バックグラウンドで起動して「終わったら通知」の方が楽でした。「30秒以内に返ってこなければ失敗」じゃなくて、「終わったら教えて」。

もうひとつは、構造的に詰まっているやつ。認証情報を要求してループしている、人間の確認を待ってブロックしている、設計判断が必要で先に進めない。これは即エスカレーションです。

> timeout は「失敗の判定」じゃなくて、「人間判断のトリガー」

これに気付いたのも、何回か timeout で「失敗しました」と来て、中身を開いたら「API キーが設定されていないだけ」というケースが続いてからでした。「失敗しました」より「API キーが設定されておらず、進められません」と書いてある方が、次に何をすればいいか分かります。

## rollback は「戻せる状態を先に作っておく」

あのファイル書き換えで1時間溶けた件があってから、書き出したルールは2つだけでした。

* テストが通ったらすぐコミット（実装が全部終わるのを待たない）
* DB スキーマや本番データへの変更みたいな「戻せない変更」は、どんなに小さくても人間確認が必要

戻せる変更だけを自律実行の対象にして、戻せない変更は自律でやらない、というだけです。

「コミット前にコミットする」って変な日本語ですが、やっていることは「git log から戻れる地点を細かく作っておく」というだけです。これがないと、中間で止まったときに「どこまで終わったか」が分からなくなります。git diff を見ても境目が分からなくて、1時間かけて戻すことになる。そういう朝を何回も繰り返したくなかった、というのが動機です。

## 「呼ばれる回数」だけ見ていれば分かる

これだけ書いて運用してみたら、呼ばれる回数がかなり減りました。

数値は数えていません。ただ、朝コーヒーを淹れてから Slack を開くまでの間に少し時間が置けるようになった、というのが体感です。前は「淹れたら呼ばれる」が定番で、コーヒーを飲む前から作業が始まっていました。

呼ばれてくる内容も変わりました。前は「import 漏れです」「型ヒント漏れです」が多かったんですが、今は「設計判断が必要です」「認証情報を更新してください」が中心です。私しか判断できないやつだけが残った感じです。

AI に「全部やってもらう」のは無理です。私が判断するしかない部分はある。ただそれ以外、5秒で直る ImportError まで毎回自分が見る必要はなかった、というのに気付くのに3週間かかりました。

途中から、朝コーヒーを淹れながら「今日は呼ばれてないといいな」と思うのが習慣になっていました。意味があるかは分かりません。

## 同じ症状の話を聞きます

最近、似た話を聞くことが少しずつ増えました。

「AI からの『失敗しました』通知に、毎回起こされている」  
「コードのちょっとしたエラーで毎回呼ばれて、副業の時間を全部削られる」  
「自分が AI の手足になってる気がする」

私もそう思いながら、3週間、毎日10回呼ばれていました。「自分で確認したい」という気持ちでその設定にしていたので、自分がその状態を作っていたわけです。5秒で直るエラーまで全部自分が見る、という運用にしたのは AI じゃなくて私でした。

壊れ方を先に決めておかないと、結局全部自分が呼ばれ続ける。それに気付くのに3週間かかったので、たぶん他の人も、しばらく同じ状態で回しているんじゃないかと思っています。

## 朝呼ばれたくないだけ

retry ルール / スピン防止 / 認証系は retry 禁止 / timeout の使い分け / rollback できる状態の作り方、自分用に1ページに置いてあります。

reliability 設計したいとか、信頼性アーキテクチャを組みたい、というより、**朝呼ばれたくない**、というのが正直な動機です。立派な設計の話を書きたいわけじゃない。

誰かが読むなら有料記事として出してもいいですが、中身は雑なルールと、よく出るエラー別の振り分けくらいです。整った設計書ではありません。

これが最終解だとは思っていません。たぶん、また別の場所で「呼ばれすぎ」が出てきます。まだ踏みます。でも今のところは、毎日10回から、ちょっとだけ減りました。コーヒーが少し冷めにくくなったくらいの変化です。設計したいというより、朝呼ばれたくないだけです。
