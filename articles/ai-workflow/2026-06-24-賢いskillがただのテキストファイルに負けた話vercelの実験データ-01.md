AI コーディングエージェントに賢いツールを持たせれば、賢く働いてくれる。なんとなくそう思っていました。Vercel が公開した検証データは、その逆を示しています。プロジェクトに置いただけのプレーンなテキストファイルが、エージェントが必要に応じて呼び出す「賢い」Skill に勝った。それも僅差ではありません。

実はデータを見る前から、自分でもうっすら同じことを感じていました。だからこの結果はかなり刺さりました。何が起きたのか、なぜ単純なほうが勝ったのか、そして自分が変えた習慣をひとつ書きます。

## 検証の中身

Vercel が測ったのは、エージェントがどれだけ正しく **Next.js 16** のコードを書けるか、です。しかもモデルがほぼ見たことのない API ばかりを選んでいます。学習データのカットオフより後に出たものだからです。`connection()`、`'use cache'` ディレクティブ、`cacheLife()` / `cacheTag()`、`forbidden()` / `unauthorized()`、`proxy.ts`、非同期の `cookies()` / `headers()`、`after()` あたり。わざと新しい領域だけにして、暗記した知識が効かないようにしています。

同じ難しいテストを、4つの条件で解かせました。

| 条件 | 正答率 |
| --- | --- |
| ドキュメントなし（ベースライン） | 53% |
| Skill（デフォルト挙動） | 53% |
| Skill ＋「使え」と明示的に指示 | 79% |
| `AGENTS.md` のドキュメント索引 | **100%** |

この表は二度見してください。Skill を渡して「必要なら使っていいよ」と任せた結果は、**何も渡さなかったときと完全に同じ 53%** です。強めに「使え」と指示してようやく 79%。そしてプロジェクトのルートに置いただけの `AGENTS.md` が満点でした。

## Skill が伸びなかった理由

Skill は一見、賢い選択肢に見えます。普段は邪魔をせず、必要そうなタスクのときにエージェントが自分で引っ張ってくる。効率的に聞こえます。問題は、その「判断する」ところにあります。

**56% のケースで、Skill は一度も呼ばれませんでした。** ドキュメントは手元にあるのに、エージェントが「今こそ取りに行くべきだ」という瞬間を認識できなかった。これは自分が普段ぶつかっていたことと同じです。ちゃんとした Skill を用意しておいて、いざそのための作業が来ると、エージェントがそれを素通りしていく。体感で十回に六回くらい。自分の設定が悪いんだろうと思っていました。でも、判断のステップそのものが穴だったわけです。

しかも、もっと脆い。Vercel は指示の言い回しを少し変えるだけで挙動が大きくぶれることも見つけています。

* **「必ず Skill を呼べ」** と書くと、エージェントはドキュメントに引っ張られてプロジェクト側を見失った。`page.tsx` は直せたのに、必須だった `next.config.ts` の変更を丸ごと落とした。
* **「まずプロジェクトを調べてから Skill を呼べ」** と順序を指定すると、先に状況を把握してからドキュメントを見にいって、正答率が上がった。

同じツールで、文を一つ並べ替えただけで出力が変わる。ここまで言い回しに敏感な仕組みは、実運用では信用しづらい。

## ただのファイルが勝った理由

`AGENTS.md` は「単純」なほう。プロジェクトのルートに置いたテキストファイルが、毎ターンただそこにある。取りに行くステップがない。Vercel 自身の表現がそのまま核心です。

> `AGENTS.md` なら、エージェントが「これは調べるべきか？」と判断する瞬間そのものが存在しない。情報は最初からそこにある。

取りに行くかどうかを決めるステップが、まるごと消えます。そこは Skill が 56% こけた場所でした。決める瞬間がなければ、見落としようがない。毎ターン文脈に入っているので、呼ぶタイミングを逃すこともないし、「先にコードを見ろ」と順序を気にする必要もない。最初からコードの隣にあるからです。

ここで効いていたのは、`AGENTS.md` が特別に賢いことではなく、そもそも「読むかどうか」をエージェントに任せていない、という一点でした。Skill は呼ばれなければ無いのと同じですが、ルートのファイルは最初から目に入っている。その差が、そのまま結果に出たように見えます。

## 見落とされがちな点：圧縮していた

真似したいなら、ここが大事です。Vercel はドキュメント全文を `AGENTS.md` に貼ったわけではありません。索引に**圧縮しています。元の 40KB を 8KB まで、8割削った**うえで、スコアは 100% のままでした。このファイルはドキュメントそのものではなく、「どの内容がどのドキュメントの、このプロジェクトのバージョンに対応するか」を示す短い地図です。本を丸ごと机に広げると、肝心の作業スペースが消える。だから索引にとどめる。

## やってみる（Next.js）

Next.js なら、一行でセットアップできます。バージョンを検出して、対応するドキュメントを取得し、索引に圧縮して `AGENTS.md` を書き出すところまでやってくれます。

```
npx @next/codemod@canary agents-md
```

Next.js でなくても考え方は同じです。プロジェクトのルートに `AGENTS.md` を置いて、散文ではなく索引として、自分の規約を短く書く。どのフォルダに何があるか、lint とテストのコマンド、実際に使っているパターン。そして先頭付近に、エージェントを自分の記憶から引き剥がす一文を入れる。「過去の知識に頼る前に、プロジェクトとこれらのドキュメントを確認すること」くらいの一文です。この一押しが、53 と 100 を分ける一部になっています。

## Skill はまだ使う、用途を絞っただけ

Skill を捨てるわけではありません。向ける先を変えました。エージェントが毎回必要とする退屈な基礎は `AGENTS.md` に置く。Skill は自分が明示的に呼び出す重い作業、「Next.js のバージョンを上げて」「App Router に移行して」みたいなものに取っておく。Vercel も同じ線を引いています。自分がやっていた間違いは、日常的な文脈まで Skill に頼って、エージェントが取りに行ってくれるのを期待していたこと。ほとんど取りに来ませんでした。

## 一つだけ引っかかっていること

実際に自分の手元で試してみて、頷ける部分は多かったです。ただ、これがどこまで一般化できるのかは、正直まだ分かりません。今回のテストは Next.js のコード生成という、わりと型のはっきりした領域の話です。自分が関わっている業界ドメインやチームの性質が変われば、結論も変わるんじゃないかと思っています。たとえば前提知識が文書化しづらい業務や、毎回かなり違う判断が要る領域だと、「索引を常に出しておく」やり方がそのまま効くとは限らない気がします。なので自分は、この結果を「正解」ではなく「自分の環境で一度確かめる価値のある仮説」くらいに受け取っています。

## 自分がここから持ち帰ったこと

面白いのはファイル形式ではなく、呼ばれなかったほうの事実です。以前は、エージェントの自律性を上げて、賢い検索や使えるツールを増やせば出力が良くなると思っていました。この検証で、そこを少し信じなくなりました。「今ドキュメントを見にいくべきか？」も結局、半分以上の確率で間違えるほうの判断だったわけです。結局、自分が変えたのはかなり地味なところでした。エージェントに「必要なら見に行って」と渡すのをやめて、最初から目に入る場所に置く。今はそのくらいで十分だと思っています。

なお、この実験も数字もすべて Vercel のものです。元記事は読む価値があります：[AGENTS.md outperforms skills in our agent evals](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals)。
