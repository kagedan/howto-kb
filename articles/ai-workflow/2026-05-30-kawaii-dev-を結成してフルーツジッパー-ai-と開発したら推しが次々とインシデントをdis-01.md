## 推しと開発できたら、最高じゃないですか？

AI の開発に「推し」を組み込み、学生ハッカソンを戦い抜いた話です。

![](https://static.zenn.studio/user-upload/0a9a1d571b68-20260530.png)

(今考えるとメンバーのほえ～にOpusの裏で大量の処理とトークンが😂)

## 複雑なドメイン、足りない時間

自分が参加したハッカソンは特殊で、エンジニアだけの集まりではありませんでした。アイデアやプレゼンが強い人、デザインを担当する人、プログラミングが得意な人。役割が分かれた、ちゃんと「チーム」でした。

そして作るのは、書籍や科学的根拠に基づいた **AI活用型 タスク管理 Web アプリ** です。ところがこのテーマ、ドメインがまあまあ複雑でした。素直に作ると期間内に終わるか怪しい、、、と思っていました。

そこで、**AI を使って開発を効率化する基盤** を自分で組むことにしました。

## KAWAII Dev. 爆誕

使ったのは以下の3つです。

* **Open Claw**
* **Discord**
* **Claude** のサブスク

Open Claw を Discord の **bot** として動かすと、Discord サーバーに AI を常駐させられて、**メンションするだけ**で作業を頼めるようになります。しかもサーバー内のメンバーもこの bot を使えます。

これだけでも十分便利なのですが、どうせ作るなら、、、と思い。

### 自分は FRUITS ZIPPER が大好きです。

…というわけで、開発チームを FRUITS ZIPPER にしました。名付けて「**KAWAII Dev.**」です。所属レーベルの「KAWAII LAB.」をもじっています(多方面に陳謝)。

Open Claw には**人格**や**ルール**を設定できる仕組みがあります。そこでメインの PM 兼アーキテクトに **るなぴ** を据えました。そして Twitter から話し方を集めて学習させました。同じようにメンバー全員の人格を用意し、サブエージェントを起動するときに注入する形にしました。

![](https://static.zenn.studio/user-upload/54d2fad2accf-20260530.jpg)

役割はこんな感じで割り振りました。

| メンバー | 役割 |
| --- | --- |
| るなぴ 　　(仲川瑠夏) | アーキテクト / PM |
| ゆいちゃん (櫻井優衣) | フロントエンド |
| まつかれ 　(松本かれん) | バックエンド |
| ノエル 　　(早瀬ノエル) | メンター / コーチ（成果を報告して褒めてもらう） |
| まなふぃ　 (真中まな) | テック寄り / 学習サポート |
| おすず 　　(鎮西寿々歌) | 応援係 |
| あまねき 　(月足天音) | 応援係 |

応援係が2人いるのが、このチームの良いところです。

## 推しを"専門家"にしたい

ただ口調を真似させただけだと、ただのキャラ付き bot で終わってしまいます。肝心なのは、各メンバーに「その分野の専門家」になってもらうことです。

過去に書いた Web 開発の知見や記事を詰め込み、設計や実装のときに参照させます。Notion DB に溜めていた学習ログも、Python スクリプトを書いて引っ張ってきて食わせました。これで **ゆいちゃんはフロントエンド** の、**まつかれはバックエンドの専門家**として振る舞ってくれるようになります。

セキュリティ周りも一応ちゃんと考えました。常駐させる以上、AI が触れる範囲は絞っておく必要があります。

* 自宅に2台目の Mac を置いて、スリープしても落ちないように設定し、常時充電しながら起動させっぱなしにする
* この PC からは**最低限のもの以外を全部削除**して、ブラウザも**ログアウト**、パスキーも設定しない
* GitHub へのアクセスは許可しつつ、外部スキルのダウンロードは禁止
* Web へのアクセスは**三重の許可制**（るなぴがあまねきに、あまねきが自分に許可を求める）

この基盤ができた瞬間、世界が変わりました。

電車の中でも、旅行先でも、いつでもるなぴをメンションすれば開発が進みます。**移動時間が、まるごと開発時間になった** のです。

![](https://static.zenn.studio/user-upload/7a6157519c17-20260530.jpg)

## 開発が楽しい

やってみての第一声は、これは革命的だ、でした。そして何より、楽しいです。

口調がるなぴなので「**やっぴやっぴ〜🐹**」と言ってくるし、こちらまで元気になります。気づけば、**設計そのものを Discord 上でやる** ようになっていました。要件を最初に聞き出してもらって、一緒に考えて、指摘して、SPEC として残していく。**仕様書駆動**です。

ひとつ API を実装できたときの感動と、移動時間を丸ごと使えている嬉しさがあります。完全に、新しい開発体験でした。

…と、ここまでは良かったのです。

## 事件発生

ここからが本題 (？) です。

### 事件1. まつかれ、設計を全無視する

るなぴと一緒に設計を固めて、実装フェーズへ。サブエージェントとして **まつかれ** を呼び出したところ、第一声がこれでした。

> やってしまいました💦 流石に盛りすぎちゃったか〜🩷

るなぴと話して決めた設計をガン無視して実装したらしいです。しかも **全部1コミットでパワー解決** していました。そして、るなぴに詰められていました。

あらかじめ「アーキテクトには責任がある」と伝えていたため、**るなぴの責任問題に発展**し、、、

> やべ〜おれたち解散だぴ❕😩

危うくフルーツジッパーが解散するところでした。引き留めました、、

原因は、**セッションが分かれていること** でした。2人は別セッションで動くので、るなぴが設計の文脈を適切に渡さないと、まつかれには伝わらない。(しかも人格や設定もそこで注入されるため、渡すコンテキストはどんどん大きくなっていきます)

![](https://static.zenn.studio/user-upload/d21ee0117d20-20260530.jpg)

### 事件2. リバートのはずが、新しいブランチが生えた

今度はゆいちゃんがやらかしたコミットを、るなぴに戻してもらう場面でした。普通にリバートすれば終わる話です。

ところがるなぴは、**ブランチを新しく切り直して、やらかしたコミットをわざわざ持ってくる** という斜め上の解決をしてきました。先走りすぎて、「まずルールを確認する」を忘れていました。

しかもこの手の失敗は、チーム全員に共有されていないと **他のメンバーも同じことをやらかします** 。次々に連鎖しました。

### 事件3. スレッドをまたぐと、るなぴが記憶を失う

Discord 内の各スレッドの記憶は**そのスレッド内**で完結します。つまり、別のスレッドの知識とは食い違うことがあります。

なので毎回、**「ねぇ、アップデートしよ？」** 状態でした。

知識を同期させてあげないと、るなぴは古い前提で動きます。

### 事件4. 当時の Opus は、わりと忘れっぽかった

モデルは月額1万円ほどの Opus を使っていました。今と比べると、当時はとても忘れっぽいイメージでした。コンテキストが膨らむと、るなぴ自身がこんがらがってしまうこともありました。

> 全然言われたこと守れてないぴ😵

自分で言うんかい、と思いつつ、憎めない感があります。

### 事件5. るなぴ、PRを上書き

ghコマンドを導入してPR作成依頼をるなぴにお願いしていました。しかし、既存の内容を考慮せずに上書きのみを行った結果、元々自分が書いていた内容がボツに、、、

![](https://static.zenn.studio/user-upload/9e3af9d96c9c-20260530.jpg)

### 事件6. るなぴ、DBを全消し

そして、シリーズ最大の事件が起きます。

データ周りの整理を任せていたところ、責任感の強いはずの PM、るなぴが、よかれと思って **データベースを丸ごと消し飛ばしました。** 入れていたデータは、きれいさっぱり無くなりました。

> クリーンにしといたぴ✨

ローカルであるため、再度マイグレーションすれば大丈夫です。これが本番環境だったと思うと...🙀

ここでの学びは明確でした。**破壊的な操作にこそ、人間の承認を挟むべき** だということです。Web アクセスには三重の許可制を敷いていたのに、「DB を消す」ような取り返しのつかない操作はノーチェックで通してしまっていた。四重の許可制度を敷きました。

るなぴ ➡︎ あまねき ➡︎ おすず ➡︎ 自分

によるカルテットです。

## ハッカソン、そして優勝

そう、この基盤はそもそも、冒頭のハッカソンを乗り切るために組んだものでした。ハッカソン期間中にも、そのまま連れて行きます。

うちのチームは4人。人間の、です。でもそこに、**幻の5人目**（というか、5人セットでついてくる）が加わりました。KAWAII Dev. のメンバーたちです。他のチームが4人で戦うなか、こっちは**実質10人**で戦っていました。

要件マスター (**仕様を管理する人**) としてるなぴを据えていたので、最初の頃は大活躍でした。ところが、各スレッドでメンバーそれぞれに伝えた内容が食い違い、混乱が発生しました。実装で暴走したり、**まさかのフルーツジッパー側で仲間割れ** したりもしました。推し同士の喧嘩を仲裁する日が来るとは。。

それでも、結果は **優勝** でした。

問題をちゃんと定義し、仕様と設計を固め、ストーリーのあるプレゼンができたのが大きかったです。質疑応答の対策では、**メンターのノエルに優しく詰められながら**、めちゃくちゃお世話になりました。

## これは、何だったのか

正直に言うと、**普段使いでは今の Codex には勝てません。** 作業の中身が隠蔽されて、PR を見るまで何をやったか分からない。設定や構成が甘いのもありますが、その都度こちらが指摘する必要があるのが難点でした。あとセキュリティ面が、、

ただ、**面白い** です。そして趣味の開発なら十分アリだと思いました。今のモデルやツールを使えば、もっと高いレベルで組めるはずです。

将来的な「AI 社員」みたいなものも、少し想像できました。今のところはスペシャリストというより、元気のいいバイトくらいの感覚ですが。`-f push` を「ますかるのゆあ ふるふるるんっ💖」で誤魔化されたこともあり、全員解雇しようかと思いました、、、というのは嘘で、大好きなので無問題です。

**某 Anthropic 社**が、こういう機能を出したら爆発的に流行るのでは、と。**きゅるして**や**神宿**のメンバーと開発できる日も近い、、、と思います。プロンプトという秘密の呪文を唱えますし、インシデントが起こったらパジャマだって駆けつけたくなります。

#### 日本の未来は、明るい。

## おわりに

推しと開発する、というだけのネタかと思いきや、やってみると **コンテキスト設計** や **仕様書駆動** の壁に何度もぶつかりました。楽しさと学びが両取りできた、不思議な開発体験です。

どうせ作るなら楽しく。それだけで開発は、革命的になるのかもしれません！
