長期稼働のAIエージェントを組んでいると、ある日突然こんな経験をします。  
「さっきまでちゃんと動いてたのに、なんか応答がおかしくなってきた...」。  
コンテキストウィンドウはまだ余裕があるはずなのに、出力の質がじわじわ落ちていく。  
僕も自前でClaude Agent Teamを組んでいて、まさにこの問題に頭を抱えました。

そのモヤモヤを解消してくれるのが、AnthropicがリリースしたCompaction API（beta）です。

  

## そもそもcontext rotって何？

「context rot（コンテキスト劣化）」という言葉、聞いたことありますか？  
直訳すると「文脈の腐敗」。長期間エージェントを動かしていると、会話履歴に古い情報・失敗した試行・冗長なやり取りがどんどん蓄積されていきます。  
それが精度劣化を引き起こす現象のことです。

どゆこと？って思いますよね。  
もう少し具体的に話します。たとえばエージェントが「タスクAを試みたが失敗した」という記録を持ち続けると、その失敗体験が後の判断に悪影響を与えることがあります。  
あるいは、初期の指示が後続の大量の会話に埋もれて、モデルが「そもそも何をしたかったんだっけ」状態になる。  
これがcontext rotです。

## 「1Mコンテキストウィンドウ = 無敵」は誤解です

ここで正直に言わせてください。  
「コンテキストウィンドウが1Mトークンもあれば、長期エージェントでも余裕でしょ？」という認識、僕も最初そう思っていました。  
でも違うんです。

技術的には、LLMはコンテキスト全体をアテンション（注意機構）で処理します。  
ウィンドウが広くなるほど計算コストが上がるだけでなく、遠い過去のトークンへのアテンションウェイトが薄まる傾向があります。  
つまり「入ってはいるが、ちゃんと参照できていない」という状態が起きやすいんです。  
さらに、古い失敗や冗長な中間ステップが「ノイズ」として作用し、モデルの判断を歪めます。  
ウィンドウの広さと、ウィンドウの中身の質はまったく別の問題なんです。

## 従来の回避策とその限界

エンジニアたちはこれまで、context rotに対していくつかの方法で対処してきました。

ひとつは「要約プロンプトの手動挿入」。  
一定のタイミングで「これまでの会話を100字で要約してください」とプロンプトを差し込み、コンパクトな記録を作る方法です。  
ただ、この要約のタイミングや品質をコントロールするのが地味に難しい。僕がAgent Teamを組んだときも、要約プロンプトのチューニングだけで丸一日溶かしたことがあります。

もうひとつは「会話リセット設計」。一定のトークン数に達したら会話をリセットし、重要情報だけを引き継いで再起動する設計です。  
これはシンプルですが、「何を引き継ぐか」の判断をコードで書かなければならず、設計コストが高い。  
そして引き継ぎに失敗したときのデバッグがとにかく辛い。

## Compaction APIの仕組みを理解する

そこに登場したのがAnthropic公式のCompaction API（beta）です。  
対象モデルはClaude Opus 4.6とSonnet 4.6。  
これを使うと、会話のトークン数がデフォルト150Kトークンを超えたタイミングで、自動的にコンテキストを圧縮してくれます。

圧縮後の会話履歴は「コンパクションブロック」という特殊なブロック形式で格納されます。  
「今までの会話を圧縮した要約のかたまり」として会話履歴に組み込まれ、そこからエージェントは会話を継続できるイメージです。  
重要な文脈は保持しつつ、冗長なやり取りはそぎ落とされる。

トリガーは「自動」と「手動」の2パターンがあります。  
デフォルトは自動で、150Kトークンを超えると勝手に圧縮が走ります。  
手動トリガーを使えば、任意のタイミングで圧縮を発動させることも可能です。  
「このフェーズが終わったら圧縮する」という設計制御ができるのが、エンジニア的には嬉しいポイントです。

```
# 自動コンパクションを有効にした会話例（イメージ）
# 実際のbeta識別子・パラメータは公式ドキュメントを参照
client = Anthropic()

response = client.beta.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8096,
    betas=["auto-compaction-2025-xx-xx"],  # 実際の識別子は公式docs参照
    messages=conversation_history
)
# 150Kトークン超えたら自動でコンパクションブロックが生成される
```

## 長期エージェント設計の思想が変わる

これ、単なる「便利機能の追加」じゃないと思っています。  
Compaction APIが意味することは、「エージェントの寿命設計」そのものの変革です。

これまでは「いつかリセットが必要になる」という前提でアーキテクチャを組む必要がありました。  
リセットのタイミング管理、状態の引き継ぎロジック、それにまつわるバグ対応。  
これが長期エージェント開発の最大のコスト要因のひとつでした。

Compaction APIが成熟すれば、エージェントを「有限の会話を持つシステム」ではなく「継続的に稼働するプロセス」として設計できるようになります。  
ちょうどOSのメモリ管理みたいなイメージです。  
アプリケーションが直接メモリを意識しなくてもOSがうまいことやってくれるように、エージェントがコンテキストを直接管理しなくても、APIレイヤーが最適化してくれる世界観です。

## 実装時のハマりどころ

とはいえ、betaゆえの注意点もあります。

まず「圧縮精度の限界」。Compaction APIがコンテキストを圧縮するとき、何を「重要」と判断するかはモデル任せです。  
エージェントの判断に必要だった細かいニュアンスが圧縮で失われるケースは、まだあり得ます。  
特に複雑な推論チェーンを扱うエージェントでは、圧縮前後で挙動が変わる可能性を念頭に置いておくべきです。

次に「手動トリガーのタイミング設計」。自動コンパクションは便利ですが、「このタスクの途中で圧縮が走ってほしくない」というケースもあります。手動トリガーを使うなら、タスクの区切りを明示的に設計する必要があります。

そして「beta APIである」という事実。  
インターフェースが変わる可能性があります。  
本番運用に組み込む前に、変更ログの定期確認を習慣づけておくのをおすすめします。

## まとめ：で、結局どうなの？

* **context rot（コンテキスト劣化）は「ウィンドウが広ければ解決」ではなく、中身の質の問題**  
  1Mコンテキストウィンドウがあっても、古いノイズが蓄積すれば精度は落ちる。
* **Compaction APIは、長期エージェントのコンテキスト管理をAPIレイヤーに委ねる設計思想の転換点**  
  150Kトークン超えで自動圧縮、コンパクションブロックで会話継続が可能に。
* **betaゆえの圧縮精度の限界は残る**が、手動トリガーと組み合わせた設計で、これまでの要約プロンプト/リセット設計より確実にシンプルになる。

エージェントを「一回こっきりのタスク実行者」ではなく「長期稼働するシステム」として育てていくなら、このCompaction APIの動向は追いかけておく価値があります。僕はさっそくAgent Teamのアーキテクチャを見直し始めています。
