こんにちは、株式会社フェズのデータテクノロジー開発部でエンジニアをしています石原です。  
普段は **Urumo BI** という、生成AIを活用した購買データ分析ソリューションの開発に携わっています。

本記事は、AIエージェント開発手法を比較検証していく連載の最終回・第7回です。

今回は6回にわたって実装してきた各手法を横並びで比較し、「どんな場面でどれを選ぶか」という実践的な答えを出して連載を締めくくります。

## この連載でやってきたこと

連載では、同じエージェントタスク（`inputs/` フォルダのファイルを読み取り・検索・要約する）を異なる手法で実装し続けました。

| 回 | 手法 | 言語 | ひとことまとめ |
| --- | --- | --- | --- |
| 第1回 | フルスクラッチ | Python | フレームワークなし。LLM呼び出しからループまで全部自分で書く |
| 第2回 | LangChain | Python | ツール実行をLLMに委ねる基盤。行動ループをフレームワークが管理 |
| 第3回 | LangGraph | Python | ワークフローをグラフとして明示。Human-in-the-loopも組み込める |
| 第4回 | LangGraph + RAG | Python | ベクトル検索を追加。「ファイル一覧からLLMが選ぶ」から「意味検索」へ |
| 第5回 | Agno | Python | 宣言的なAPI。RAG・マルチエージェントを最小のコードで実現 |
| 第6回 | Mastra | TypeScript | TypeScript-first。型安全なワークフローとDev Playgroundが特徴 |

コードは全てGitHubに掲載しています。

[ai-agent-implementation-playground（GitHub）](https://github.com/yoshihisa-ishihara-fez/ai-agent-implementation-playground)

## 全手法の比較

まず数値で整理します。

### コード量の変化

| 手法 | ファイル数 | コード量（概算） | 依存パッケージ数 |
| --- | --- | --- | --- |
| フルスクラッチ | 1 | 約140行 | 1 |
| LangChain | 1 | 約140行 | 4 |
| LangGraph | 5 | 約290行 | 4 |
| LangGraph + RAG | 6 | 約460行 | 8 |
| Agno | 3 | 約230行 | 4 |
| Mastra | 10 | 約370行 | 8 |

コード量だけ見ると、Agnoが最もコンパクトです（フルスクラッチとLangChainはRAGをつけなかったため同条件ならAgnoが最もコンパクト）。LangGraphにRAGを追加した第4回が最も多くなっていますが、同等の機能をAgnoで書き直すと約半分になりました。

なお、コード量で比較しているLangGraph+RAGとAgnoは、同じ機能（ベクトル検索によるRAG + マルチエージェント）を実装したものです。使っているEmbeddingモデル（`paraphrase-multilingual-MiniLM-L12-v2`）とベクトルDB（ChromaDB）も同一のため、実装量の差はほぼそのままフレームワークの記述コスト差として読むことができます。

今回の比較要件（ファイルの読み取り・検索・要約）はシンプルな単一タスクのため、手法間のコード量の差は大きくありません。しかし、要件が複雑化したとき（複数の条件分岐、複数エージェントの協調、複数ドメインのRAGなど）にはこの差が拡大する傾向があります。フルスクラッチやLangChainではフロー制御や状態管理のコードが線形に増えていくのに対し、AgnoやMastraはフレームワークが吸収する部分が大きいため、要件の増加に対するコード量の増加が緩やかです。LangGraphはグラフ定義が増える分コード量は増えますが、複雑なフローほどグラフによる明示性の価値が高まります。

コード量はあくまで一側面です。次に、設計思想の違いを整理します。

### 設計思想の比較

| 観点 | フルスクラッチ | LangChain | LangGraph | Agno | Mastra |
| --- | --- | --- | --- | --- | --- |
| フロー制御 | 人間がコードで書く | フレームワーク内部（暗黙） | グラフで明示 | Teamが自動調整 | スーパーバイザー（Tool定義で明示） |
| 状態管理 | 変数 | messagesに埋もれる | TypedDictで明示 | フレームワーク管理 | TypeScript型で保証 |
| RAG | 自前実装 | 自前実装 | 自前実装 | Knowledge組み込み | Workflowで実装 |
| マルチエージェント | 手動 | 手動 | グラフに手動追加 | Team標準対応 | スーパーバイザーパターン |
| Human-in-the-loop | 実装が大変 | 困難 | interrupt()で容易 | 対話ループで自然に | interrupt()で対応 |
| デバッグ | printデバッグ | LangSmith | LangSmith / LangGraph Studio | show\_members\_responses | Dev Playground（ブラウザUI） |
| 学習コスト | 低（Pythonが分かれば書ける） | 中 | 高 | 低〜中 | 中 |
| 言語 | Python | Python / JavaScript | Python / JavaScript | Python | TypeScript |

!

この表から読み取れる重要な傾向が2つあります。

1つ目は、フロー制御の方向性の違いです。LangChainはLLMに行動を委ねる方向、LangGraphは人間がグラフで設計する方向、AgnoとMastraはその中間（委譲しつつもコードで追える）という位置づけになっています。

2つ目は、RAGとマルチエージェントの扱いです。LangGraph以前はどちらも自前実装が必要でしたが、Agnoではどちらも組み込み機能として数行で利用できます。Mastraは型安全なWorkflowとして明示的に実装する設計です。

## 各手法の特徴・メリット・デメリット

### フルスクラッチ

エージェントの本質は「観測 → 判断 → 行動」のループ構造にある、ということを実感するための手法です。フレームワークを使わないことで、ツール呼び出しの仕組み・LLMへのプロンプト設計・JSON解析の課題がすべて自分ごとになります。

メリット:

* 依存関係がほぼゼロ
* 処理の中身が完全に見える
* LLM APIさえあれば動く

デメリット:

* ループ・分岐・状態管理をすべて自前で書く必要がある
* ツールが増えるほどコードが複雑化する
* RAGやマルチエージェントへの拡張は現実的ではない

向く場面:

* AIエージェントがどう動いているかを理解したい（学習目的）
* 小規模・1回限りのスクリプトで外部依存を排除したい

### LangChain

LLMとツールを接続し、「どのツールをいつ呼ぶか」をLLMに委ねる基盤を提供するフレームワークです。エージェントのループ制御がフレームワーク内部に移ることで、開発者はツールの定義と目的の記述に集中できます。

メリット:

* ツール定義が簡潔（`@tool` デコレータ1つ）
* LangSmithと連携したトレーシング・評価が充実
* ベクトルストア・各種LLMとの接続ラッパーが豊富
* コミュニティが大きく、情報が多い
* JavaScript版（`langchain` npm）も存在し、TypeScript環境でも利用できる

デメリット:

* フレームワーク内部がブラックボックスになりやすい
* 分岐・ループ・状態管理を明示的に制御したいときに限界がある
* 抽象化レイヤーが厚く、予期しない挙動のデバッグが難しいことがある
* ライブラリが肥大化していると指摘されることがある

向く場面:

* 既存のLangChainエコシステム（LangSmith等）を活用したい
* ツールを接続してLLMに処理を委ねる、シンプルなエージェントを作りたい
* 多数のLLM・データストアへの対応が必要でラッパーを活用したい

### LangGraph

ワークフローをノードとエッジのグラフとして定義し、「エージェントの流れを人間が設計する」フレームワークです。LangChainでは暗黙的だった制御構造を、宣言的なグラフとして表現できます。

メリット:

* フロー全体が `graph.py` 1ファイルで把握できる
* 条件分岐・ループが `add_conditional_edges()` で明示的
* `interrupt()` でHuman-in-the-loopが容易
* 状態を `TypedDict` で構造化できる
* LangGraph Studioで視覚的なデバッグが可能

デメリット:

* 学習コストが高い（StateGraph・Node・Edge・interrupt・Command・checkpointerなど覚えることが多い）
* シンプルなエージェントでもファイル数・コード量が増える
* LangChainとの依存関係がある

向く場面:

* フローが複雑で、分岐・ループを厳密に制御したい
* 承認フロー（Human-in-the-loop）を細かく設計したい
* 処理の中間状態を構造化して管理したい
* 将来的にマルチエージェントへの拡張を見越している

### Agno

Python製フレームワークです。`Agent` クラスに役割・ツール・Knowledge・指示を宣言するだけで動きます。マルチエージェント（Team）とRAG（Knowledge）が標準機能として組み込まれており、コード量が最も少ない選択肢のひとつです。

メリット:

* コードが直感的で読みやすい（3ファイルでLangGraph+RAGと同等の機能）
* RAGが `knowledge=knowledge, search_knowledge=True` の設定だけで有効になる
* Teamを使えばマルチエージェントが数行で実現できる
* 依存パッケージが少ない
* 開発体験が良く、プロトタイプから本番まで同じコードベースで進めやすい

デメリット:

* `coordinate` モードはLLMがタスク分解・委譲を判断するため、フロー制御の再現性は保証されない
* LangGraph比でグラフとして処理の流れを可視化する機能はない
* LangChain/LangGraphに比べてコミュニティ規模がまだ小さく、エラー時の情報収集が難しいことがある

向く場面:

* Pythonチームで素早くマルチエージェント＋RAGを実装したい
* プロトタイプを素早く作り、検証しながら育てたい
* コード量を最小限に抑えて保守性を高めたい

### Mastra

TypeScript製のAIエージェントフレームワークです。Zodによる型安全なツール・ワークフロー定義と、`mastra dev` コマンドで即座に起動するDev Playgroundが特徴的です。

メリット:

* ZodスキーマによってツールとワークフローのI/Oが型保証される
* ワークフローの `.then().then().commit()` チェーンが宣言的でわかりやすい
* Dev Playgroundでエージェントの動作をブラウザから確認できる
* TypeScriptチームがフロントエンドと型定義を共有できる
* スーパーバイザーパターンがツール定義として明示されているため追跡しやすい

デメリット:

* Agnoと同等の機能で約1.5倍のコード量になる（型定義の記述コスト）
* ChromaDBをHTTPサーバー経由で使う必要があり、構成が少し複雑
* 2024年後半からの新興FWのため、APIが安定途上にある（バージョン固定推奨）
* データサイエンス・機械学習ライブラリとの親和性はPythonに劣る

向く場面:

* TypeScriptをフロントエンドからバックエンドまで統一しているチーム
* 型安全を重視した堅実な実装が必要なプロダクト
* フロントエンドとAIエージェントのバックエンドで型定義を共有したい
* Dev PlaygroundによるGUIデバッグ・デモを活用したい

## どれを選ぶか

実装してきた経験から整理すると、判断軸は主に「チームの言語」と「プロダクトのフェーズ・要件」の2軸になります。

### Pythonチームであれば Agno

現時点でPythonを使っているチームが新しくAIエージェント機能を追加する場合、Agnoが最も現実的な選択肢です。RAG・マルチエージェントを最小のコードで動かせることは、素早くプロトタイプを作り、PDCAを回しながら改善していく上で大きなアドバンテージになります。

Urumo BIのバックエンドはPythonで構成されている部分が多く、もし今日から新しくAIエージェント機能を追加するなら、個人的にはAgnoを選びたいと考えています。記述量が少ないことは単にコードが短いということではなく、チームが機能を素早くプロダクトに反映し、保守し続けられることを意味します。

### TypeScriptチームであれば Mastra、あるいはLangChain（JavaScript版）

フロントエンドからバックエンドまでTypeScriptで統一されているチームや、型安全な実装を特に重視するプロダクトであれば、Mastraが良い選択肢になります。

ただし、TypeScript環境でのAIエージェント開発はLangChainにもJavaScriptパッケージがあります。LangChainのエコシステムや既存の知見を活かしたい場合はこちらも有効です。プロダクトとチームの状況に応じて選ぶのが実践的です。

### 厳密なフロー制御が必要であれば LangGraph

承認フロー・複雑な条件分岐・処理のステップを明示的に制御したい要件がある場合は、LangGraphが適しています。特に「どの条件でどのパスを通るか」をコードレビューで確認できることが重要な場合、グラフ定義による明示性は大きな価値を持ちます。

### フレームワーク選定に正解はない

この連載でひとつ言えることは、「どれが一番優れているか」という問いに答えはない、ということです。

各フレームワークには向く文脈があり、それはチームの言語・規模・プロダクトのフェーズ・要件によって変わります。今日Agnoが最適な選択でも、数ヶ月後には別の答えになっているかもしれません。重要なのは、選択肢の違いを実際に手を動かして理解しておくことで、その時々の状況に合わせて判断できるようになることです。

## Urumo BIでのLLMフレームワーク選定の経緯

Urumo BIでは、LLM利用の基盤としてLangChainを採用し、現在はLangSmithによるトレーシングやLangGraphによるワークフロー制御も組み合わせて利用しています。ここでは、なぜLangChainエコシステムを選んだのかの背景と、現在の状況を共有します。

### 当時の選定理由

LangChainを採用したのは2024年、ChatGPT 3.5の時代でした。当時の最大の課題は「どのモデルを使うべきか」でした。OpenAI一択で良いのか、他のモデルも検証すべきか、精度として実用に耐えるかどうか——モデル選択自体が最大のリスクだった時期です。

そのような状況で、各モデル用SDKを直接使って個別実装するか、LLMへの依存を透過的に扱える中間ライブラリを選ぶかを検討しました。LangChainを選んだのは、モデルの差分を吸収してくれることの価値が大きく、それが選択を正当化できる状態だったからです。

AgnoやMastraといった選択肢が現れたのは2025年です。2024年の時点ではLangChainが実質的に唯一の現実的な選択肢でした。

### 現在の状況

現時点では、LangChainを使い続ける積極的な理由はなく、移行を検討しています。当時はモデル間の差分吸収が大きな価値でしたが、各モデルのAPI仕様が収斂してきた現在、その価値は相対的に薄れています。

今後については、チームで相談しながら、その時点で最適なフレームワークへの移行を進めていきたいと考えています。それがAgnoになるかもしれませんし、この連載で取り上げなかったフレームワークになるかもしれません。技術選定に「永遠の正解」はなく、「その時点での最適解を選べる状態を維持すること」の方が大切だと感じています。

## この連載を通じた気づき

バックエンド・フロントエンド・インフラはある程度経験してきましたが、正直なところAIエージェントを自分で実装することはこれまであまりイメージしてきませんでした。連載を始めたのは「他のエンジニアが使っているフレームワークの中身を理解したい」という動機からです。

最も印象的だったのは第1回、フルスクラッチで書いたときの「力技感」でした。LLMを呼ぶ、JSONをパースする、ファイルを読む、もう一度LLMを呼ぶ——それを全部 `if` と `for` で制御していたあのコードが、LangChainでは `@tool` デコレータひとつに変わります。

「フレームワークって偉大だな」と改めて感じました。先人が積み上げた抽象化の上に乗ることの価値——いわゆる「巨人の肩に乗る」ということの意味が、実装を通じてより具体的に理解できました。

また、AIにコードを書かせることが当たり前になってきた2025〜2026年の現在、開発の速度感はさらに上がっています。この環境下では、最小規模のチームで高速にプロトタイプを作り、素早くPDCAを回してプロダクトに反映していく力がより重要になります。そういう観点でも、記述量が少なく直感的に書けるAgnoのようなフレームワークの価値は高まっていると感じています。

## まとめ

6回にわたって実装してきた比較を振り返ると、以下のように整理できます。

| 手法 | ひとことで |
| --- | --- |
| フルスクラッチ | エージェントの本質を理解するために一度はやる価値がある |
| LangChain | 成熟したエコシステム。モデル非依存が必要な場面で今も有効 |
| LangGraph | フローを明示的に設計・制御したいときの選択肢 |
| Agno | Pythonチームで素早く・シンプルに作りたいなら現時点で最有力 |
| Mastra | TypeScriptチームで型安全なエージェントを作るなら |

AIエージェント開発のフレームワークは、2024年から2025にかけて急速に多様化しました。そしてこれからも変わり続けます。

大切なのは特定のフレームワークに依存することではなく、エージェントの本質である「観測・判断・行動のループ構造」を理解した上で、その時々の文脈に合った道具を選べるようになることだと思います。この連載がその判断をする上での参考になれば幸いです。
