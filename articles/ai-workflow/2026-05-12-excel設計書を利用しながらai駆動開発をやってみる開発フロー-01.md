## はじめに

AI コーディングを実務に入れると、実装速度は上がります。

一方で、SIの開発現場には次のような前提が残っていることが多いです。

* Excel 設計書が正式成果物になっている
* Excel 大好き❤️
* Excel がないと仕事できない❤️

この記事では、Excel 設計書を業務仕様の正として残しつつ、cc-sdd v3 と Windsurf Skills に渡しやすい形へ変換する開発フローを整理します。

方針は次の通りです。

* Excel 設計書はアプリケーションとは別の場所で管理する
* Excel 設計書は先行して更新する
* アプリケーションリポジトリの Git 管理外の作業フォルダに Excel を全量コピーする
* ユーザが今回の spec で使う Excel ファイルを指定する
* Python で指定された Excel から Markdown を生成する
* AI が生成 Markdown を `ai-input.md` として整形する
* 整形した `ai-input.md` を cc-sdd v3 / Windsurf Skills に渡す
* Excel 設計書は、基本設計、テーブル設計、メッセージ定義、コード定義のみ存在する

## 結論

Excel 設計書と cc-sdd v3 を併用する場合は、次の流れにします。

```
管理元の Excel を更新
  ↓
@excel-design-to-ai-input で ai-input.md を作る
  ↓
cc-sdd v3 / Windsurf Skills で requirements / design / tasks を作る
  ↓
実装・テスト
  ↓
実装結果と Excel 設計書の差分を確認
```

ポイントは、Excel を直接 AI に読ませないことです。

Excel は業務仕様の置き場として扱い、AI には今回の変更に必要な範囲だけを Markdown 化して渡します。  
設計書の管理元は Git リポジトリである必要はありません。  
Git 管理されていない共有フォルダや文書管理システムでも、参照元と取得時点を追えるようにします。

## 前提にする設計書

この記事では、Excel 設計書は次の 4 種類だけ存在する前提にします。

```
基本設計書.xlsx
テーブル設計書.xlsx
メッセージ定義書.xlsx
コード定義書.xlsx
```

そのため、AI input では次のように役割を補います。

| Excel 設計書 | 主な内容 | AI input で補う内容 |
| --- | --- | --- |
| 基本設計書 | 機能概要、業務ルール、入出力、処理概要 | 変更対象、非対象、受け入れ条件 |
| テーブル設計書 | テーブル、カラム、型、制約 | DB 変更有無、SQL 条件、既存データ影響 |
| メッセージ定義書 | メッセージ ID、文言 | エラー処理、バリデーション、テスト観点 |
| コード定義書 | コード値、コード名称 | enum、分岐条件、表示名、テスト観点 |

Excel に存在しない詳細は、cc-sdd v3 の `design.md` や `tasks.md` で実装用に整理します。

## 責務を分ける

最も重要なのは、Excel 設計書、cc-sdd v3 spec、ソースコードの責務を分けることです。

```
Excel 設計書:
  業務仕様の正
  顧客・PM・QA 向け
  仕様レビューと保守資料

AI input Markdown:
  Excel から抽出した今回変更分の作業入力
  AI に渡すための中間ファイル

cc-sdd v3 spec:
  AI 実装用の作業仕様
  requirements.md / design.md / tasks.md

ソースコード:
  稼働実態の正
```

業務仕様としては、管理元にある Excel 設計書を正とします。  
一方で、実際に稼働している挙動の確認元はソースコードです。

cc-sdd v3 spec は、Excel を実装可能な粒度へ変換した作業用の成果物として扱います。  
Excel、spec、ソースコードに差分が出た場合は、仕様変更なのか実装不備なのかをレビューで判定します。  
仕様変更であれば Excel を更新してから、AI input と spec を再生成または修正します。  
実装不備であれば、Excel と spec を変えずにソースコードを修正します。

人間と AI の責任範囲は次のように分けます。

| 作業 | 人間 | AI |
| --- | --- | --- |
| Excel 設計書の更新 | 実施する | 実施しない |
| 設計レビュー | 実施する | （SDDの文脈では）実施しない |
| 対象 Excel の指定 | 実施する | 判断しない |
| Excel から Markdown 生成 | 実行指示・結果確認を行う | スクリプト実行と整形を支援する |
| 生成 Markdown と `ai-input.md` の対応確認 | 実施する | 対応表の作成を支援する |
| cc-sdd v3 spec 作成 | 承認する | 草案を作成する |
| ソースコード実装 | レビューする | 実装する |
| DDL 作成 | 承認する | 作成する |
| DB 変更の適用 | 実施する | 実施しない |
| 差分判定 | 仕様変更か実装不備かを判定する | 差分整理を支援する |

## 全体フロー

設計書先行の現場では、次の順番に固定すると運用しやすいです。

```
1. 管理元の Excel を更新する
2. 更新した Excel をレビューする
3. @excel-design-to-ai-input で Excel から `ai-input.md` を作る
4. @kiro-discovery で作業方針を整理する
5. @kiro-spec-init で spec を作る
6. @kiro-spec-requirements で要件を作る
7. @kiro-spec-design で設計を作る
8. @kiro-validate-design で設計を検証する
9. @kiro-spec-tasks で実装タスクに分解する
10. @kiro-impl で実装する
11. @kiro-validate-impl で実装差分を確認する
12. 実装結果と Excel 設計書の差分を確認する
```

Excel を先行して更新することで、AI が参照する仕様の起点を固定できます。  
Excel のコピー、対象 Excel の指定、Markdown 化、固定スナップショット作成、`ai-input.md` 整形は `@excel-design-to-ai-input` にまとめます。  
次の「2. 管理元から作業フォルダへコピーする」から「5. AI が Markdown を ai-input.md に整形する」までは、`@excel-design-to-ai-input` の中で行う処理の内訳です。

## ディレクトリ構成

`ai-input` は spec ごとに存在する入力証跡です。  
そのため、アプリケーションリポジトリ側では `.kiro/specs/<spec名>/` の配下に `ai-input/` を置きます。

```
app-repository/
├── .gitignore
├── .work/
│   └── excel-design/
│       ├── source/
│       │   ├── basic/
│       │   │   └── basic-design.xlsx
│       │   ├── db/
│       │   │   └── table-design.xlsx
│       │   └── definitions/
│       │       ├── message-definition.xlsx
│       │       └── code-definition.xlsx
│       ├── generated/
│       │   └── excel/
│       └── manifest.md
├── requirements.txt
├── scripts/
│   └── excel_to_ai_input.py
└── .kiro/
    └── specs/
        └── <spec名>/
            ├── ai-input/
            │   ├── ai-input.md
            │   ├── README.md
            │   ├── excel-manifest.md
            │   └── generated-markdown/
            │       ├── basic/
            │       │   └── basic-design.md
            │       ├── db/
            │       │   └── table-design.md
            │       └── definitions/
            │           ├── message-definition.md
            │           └── code-definition.md
            ├── requirements.md
            ├── design.md
            └── tasks.md
```

`.work/excel-design/` は、管理元から Excel を全量コピーする Git 管理外の作業フォルダです。  
Python はこの作業フォルダに置かれた Excel のうち、ユーザが指定したファイルだけを読みます。  
業務仕様の保管場所は管理元の Excel なので、作業フォルダの Excel を業務仕様の正として扱わないようにします。  
作業フォルダ側に古いファイルや新しいファイルが残るリスクは許容し、参照元情報と `ai-input` の証跡で追跡します。

Git 管理の方針は次です。

```
Git 管理する:
  .kiro/specs/<spec名>/ai-input/ai-input.md
  .kiro/specs/<spec名>/ai-input/README.md
  .kiro/specs/<spec名>/ai-input/excel-manifest.md
  .kiro/specs/<spec名>/ai-input/generated-markdown/**
  requirements.txt

Git 管理しない:
  .work/excel-design/**
  Excel 本体
```

少なくとも `ai-input.md`、`README.md`、`excel-manifest.md`、固定スナップショット用の `generated-markdown/` は spec の入力証跡として残します。  
Excel 本体は Git 管理しません。  
`requirements.txt` は Python スクリプトの依存関係として Git 管理します。

## 1. Excel 設計書を先行して更新する

まず、管理元の Excel を更新します。

```
design-repository/
├── basic-design.xlsx
├── table-design.xlsx
├── message-definition.xlsx
└── code-definition.xlsx
```

この時点で決めておくべきことは次です。

```
- 対象機能
- 変更理由
- 変更前仕様
- 変更後仕様
- 対象テーブル
- 追加・変更するカラム
- 使用するメッセージ ID
- 追加・変更するメッセージ文言
- 使用するコード値
- 追加・変更するコード名称
- 正常系
- 異常系
- 非対象範囲
```

Excel 更新後に、設計レビューを通します。  
Markdown 化の対象にするのは、レビュー済みの Excel にします。

## 2. 管理元から作業フォルダへコピーする

この節は、`@excel-design-to-ai-input` の中で行う処理の内訳です。

管理元の Excel を、アプリケーションリポジトリの Git 管理外の作業フォルダへ全量コピーします。

例：

```
mkdir -p .work/excel-design/source

rsync -av --update \
  ../design-repository/ \
  .work/excel-design/source/
```

`rsync` は、あるフォルダの内容を別のフォルダへ同期コピーするためのOSコマンドです。  
ここでは、管理元にある Excel 一式を、アプリケーションリポジトリ内の作業フォルダへコピーするために使います。

この例では、`../design-repository/` が管理元、`.work/excel-design/source/` が作業フォルダです。  
`--update` を付けることで、管理元側が新しい Excel だけを作業フォルダへ反映します。  
一方で、削除されたファイルを作業フォルダから消すわけではありません。  
このフローでは、作業フォルダを業務仕様の正として扱わず、今回使った Excel と生成 Markdown の証跡を残すことで追跡します。

作業フォルダは Git 管理しません。

コピー時には、コピー元で確認できた Excel のファイルパス、サイズ、更新日時、SHA-256 を `.work/excel-design/manifest.md` に作成・更新します。  
作業フォルダは業務仕様の正ではないため、作業フォルダ側に古いファイルや新しいファイルが残ること自体は許容します。  
ただし、今回の spec で使用した生成 Markdown は、後続の `ai-input/` 配下に固定スナップショットとして残します。  
これは、管理元の Excel が Git 管理されていない場合でも、後から同じ入力内容をレビューできるようにするためです。  
Excel 本体は Git 管理しません。

manifest は参照 Excel の証跡です。  
`.work/excel-design/manifest.md` はコピー元で確認できた Excel の一覧として扱います。  
今回の spec で使用した Excel の行だけを、`.kiro/specs/<spec名>/ai-input/excel-manifest.md` にコピーして Git 管理します。

```
| 種別       | パス                                                | 更新日時              | サイズ   | SHA-256    | 備考         |
| ---------- | --------------------------------------------------- | --------------------- | -------- | ---------- | ------------ |
| 基本設計書 | `.work/excel-design/source/basic/basic-design.xlsx` | `2026-05-05 10:00:00` | `123456` | `<sha256>` | `<対象機能>` |
```

コピーするときは、対象バージョンが分かるように `README.md` へ参照情報を残します。

```
# 参照元

- 設計書管理元: `../design-repository`
- 作業フォルダ: `.work/excel-design/source/`
- コピー日: 2026-05-05
- 対象機能: `<対象機能>`
- 対象 Excel: ユーザが指定したファイルのみ
```

URL、コミット SHA、文書管理システム上の版番号、更新日時を使える場合は、それも残します。  
後から、どの時点の設計書を AI に渡したか追えるようにするためです。

## 3. ユーザが対象 Excel を指定する

この節は、`@excel-design-to-ai-input` の中で行う処理の内訳です。

次に、ユーザが今回の spec で使う Excel ファイルを指定します。  
Python は設計書の要否を判断しません。

指定例です。

```
basic/basic-design.xlsx
db/table-design.xlsx
definitions/message-definition.xlsx
definitions/code-definition.xlsx
```

選択基準は次です。

```
基本設計書:
  対象機能の要件・業務ルール・入出力に関係する場合に指定する

テーブル設計書:
  対象機能が参照・更新するテーブルがある場合に指定する

メッセージ定義書:
  対象機能で表示・返却するメッセージ ID がある場合に指定する

コード定義書:
  対象機能で使用するコード値・区分値・enum 相当の値がある場合に指定する
```

対象 Excel の選択は人間の責務です。  
Python は、指定された Excel を作業フォルダから読むだけにします。

## 4. Python で指定 Excel を Markdown 化する

この節は、`@excel-design-to-ai-input` の中で行う処理の内訳です。

次に、ユーザが指定した Excel を Python で Markdown に変換します。

Excel をそのまま AI に渡すより、Markdown にした方が次の点で安定します。

* 対象範囲を絞れる
* 空欄や備考欄の扱いを明示できる
* AI が表構造を読み違えにくい
* PR で差分を確認しやすい
* cc-sdd v3 の requirements / design / tasks に展開しやすい

Python スクリプトの要件の例は次です。

```
- 対象フォルダは `.work/excel-design/source/` の 1 つ
- その配下に Excel が多数存在し、サブフォルダにまたがる
- Python が読む Excel は、ユーザが指定したファイルだけにする
- `--excel` は `.work/excel-design/source/` からの相対パスだけ許可する
- Excel の更新日時を見て、Markdown 再生成対象を絞る（時短のため）
- 1 セル内の取り消し線付き文言は Markdown に含めない
- `.work/excel-design/generated/excel/` に Markdown を作成する
- 特定のシートは Markdown 化の対象から除外する
```

画像や図形は、表データとして Markdown に変換しきれません。  
そのため、スクリプトでは画像や図形の存在を検出し、Markdown 側に「このシートには画像または図形がある」と注記します。  
図そのものの意味は、必要に応じて人間が `ai-input.md` に補足します。

<summary>Python スクリプト例</summary>

```
"""
README

このスクリプトは、ユーザが指定した Excel だけを Markdown 化する前処理用のサンプルです。
業務仕様の正は管理元の Excel であり、`.work/excel-design/` 配下の Excel は作業用コピーです。

入力:
  - `.work/excel-design/source/` 配下の Excel
  - `--excel` で指定された source/ からの相対パスのみ

出力:
  - `.work/excel-design/generated/excel/` 配下の Markdown

運用:
  - 生成 Markdown は AI が `ai-input.md` に整形する前の中間ファイルです。
  - 今回の spec で使った生成 Markdown は、後から追跡できるように
    `.kiro/specs/<spec名>/ai-input/generated-markdown/` へ固定スナップショットとして残します。
"""

from __future__ import annotations

import argparse
import json
from pathlib import Path

from openpyxl import load_workbook

DEFAULT_EXCLUDED_SHEETS = {"改訂履歴", "表紙", "目次"}

def load_state(state_path: Path) -> dict[str, float]:
    if not state_path.exists():
        return {}
    return json.loads(state_path.read_text(encoding="utf-8"))

def save_state(state_path: Path, state: dict[str, float]) -> None:
    state_path.parent.mkdir(parents=True, exist_ok=True)
    state_path.write_text(json.dumps(state, ensure_ascii=False, indent=2), encoding="utf-8")

def should_convert(path: Path, state: dict[str, float], force: bool) -> bool:
    if force:
        return True
    mtime = path.stat().st_mtime
    return state.get(str(path)) != mtime

def cell_to_markdown(cell) -> str:
    if getattr(cell.font, "strike", False):
        return ""

    value = cell.value
    if value is None:
        return ""

    if not hasattr(value, "__iter__") or isinstance(value, str):
        return str(value)

    parts: list[str] = []
    for run in value:
        text = getattr(run, "text", str(run))
        font = getattr(run, "font", None)
        if font is not None and getattr(font, "strike", False):
            continue
        parts.append(text)
    return "".join(parts)

def sheet_has_images_or_shapes(sheet) -> bool:
    has_images = bool(getattr(sheet, "_images", []))
    has_drawings = getattr(sheet, "_drawing", None) is not None
    return has_images or has_drawings

def row_to_markdown(values: list[str]) -> str:
    escaped = [value.replace("\n", "<br>").replace("|", "\\|") for value in values]
    return "| " + " | ".join(escaped) + " |"

def sheet_to_markdown(sheet) -> list[str]:
    rows = [
        [cell_to_markdown(cell) for cell in row]
        for row in sheet.iter_rows()
    ]
    rows = [row for row in rows if any(cell.strip() for cell in row)]

    lines: list[str] = [f"## {sheet.title}", ""]

    if sheet_has_images_or_shapes(sheet):
        lines.extend([
            "> このシートには画像または図形があります。Markdown 化できない情報が含まれる可能性があります。",
            "",
        ])

    if not rows:
        lines.append("_有効な表データなし_")
        lines.append("")
        return lines

    width = max(len(row) for row in rows)
    normalized = [row + [""] * (width - len(row)) for row in rows]
    header = normalized[0]
    body = normalized[1:]

    lines.append(row_to_markdown(header))
    lines.append(row_to_markdown(["---"] * width))
    for row in body:
        lines.append(row_to_markdown(row))
    lines.append("")
    return lines

def excel_to_markdown(excel_path: Path, output_path: Path, excluded_sheets: set[str]) -> None:
    workbook = load_workbook(excel_path, data_only=False, rich_text=True)
    lines: list[str] = [f"# {excel_path.name}", ""]

    for sheet in workbook.worksheets:
        if sheet.title in excluded_sheets:
            continue
        lines.extend(sheet_to_markdown(sheet))

    output_path.parent.mkdir(parents=True, exist_ok=True)
    output_path.write_text("\n".join(lines), encoding="utf-8")

def resolve_excel_path(source_dir: Path, excel: str) -> Path:
    path = Path(excel)
    if path.is_absolute():
        raise ValueError(f"--excel must be relative to source/: {excel}")
    if ".." in path.parts:
        raise ValueError(f"--excel must not contain '..': {excel}")

    resolved = (source_dir / path).resolve()
    resolved.relative_to(source_dir.resolve())
    return resolved

def main() -> None:
    parser = argparse.ArgumentParser()
    parser.add_argument("work_dir", type=Path, help=".work/excel-design")
    parser.add_argument("--excel", action="append", required=True, help="source/ からの相対パス")
    parser.add_argument("--exclude-sheet", action="append", default=[])
    parser.add_argument("--force", action="store_true")
    args = parser.parse_args()

    source_dir = args.work_dir / "source"
    generated_dir = args.work_dir / "generated"
    excel_markdown_dir = generated_dir / "excel"
    state_path = generated_dir / ".excel_to_markdown_state.json"
    excluded_sheets = DEFAULT_EXCLUDED_SHEETS | set(args.exclude_sheet)

    state = load_state(state_path)

    for excel in args.excel:
        excel_path = resolve_excel_path(source_dir, excel)
        if not should_convert(excel_path, state, args.force):
            continue
        relative = excel_path.relative_to(source_dir.resolve())
        output_path = excel_markdown_dir / relative.with_suffix(".md")
        excel_to_markdown(excel_path, output_path, excluded_sheets)
        state[str(excel_path)] = excel_path.stat().st_mtime

    save_state(state_path, state)

if __name__ == "__main__":
    main()
```

</details>

実行例です。

```
python scripts/excel_to_ai_input.py .work/excel-design \
  --excel basic/basic-design.xlsx \
  --excel db/table-design.xlsx \
  --excel definitions/message-definition.xlsx \
  --excel definitions/code-definition.xlsx
```

実務では、除外シート名、対象外フォルダ、画像の扱い、更新日時の記録方法をプロジェクトごとに設定ファイル化すると運用しやすくなります。

## 5. AI が Markdown を ai-input.md に整形する

この節は、`@excel-design-to-ai-input` の中で行う処理の内訳です。

Python が生成した Markdown を、そのまま cc-sdd v3 に渡すのではなく、AI に `ai-input.md` として整形させます。  
整形先は `.kiro/specs/<spec名>/ai-input/ai-input.md` です。  
また、今回の spec で使った生成 Markdown は `.kiro/specs/<spec名>/ai-input/generated-markdown/` にコピーして、固定スナップショットとして Git 管理します。  
これは、管理元の Excel が Git 管理されていない場合でも、spec の入力内容をアプリケーションリポジトリ側で追跡できるようにするためです。

`ai-input.md` には、次の情報を入れます。

```
# AI input: <対象機能>

## 参照元

- 設計書管理元: `../design-repository`
- 設計書の版: `<commit-sha、文書管理番号、更新日時など>`
- コピー日: 2026-05-05
- Excel manifest: `.kiro/specs/<spec名>/ai-input/excel-manifest.md`
- 対象機能: `<対象機能>`

## 対象 Excel

- 基本設計書: `.work/excel-design/source/basic/basic-design.xlsx`
- テーブル設計書: `.work/excel-design/source/db/table-design.xlsx`
- メッセージ定義書: `.work/excel-design/source/definitions/message-definition.xlsx`
- コード定義書: `.work/excel-design/source/definitions/code-definition.xlsx`

## 固定スナップショット

- 基本設計書 Markdown: `.kiro/specs/<spec名>/ai-input/generated-markdown/basic/basic-design.md`
- テーブル設計書 Markdown: `.kiro/specs/<spec名>/ai-input/generated-markdown/db/table-design.md`
- メッセージ定義書 Markdown: `.kiro/specs/<spec名>/ai-input/generated-markdown/definitions/message-definition.md`
- コード定義書 Markdown: `.kiro/specs/<spec名>/ai-input/generated-markdown/definitions/code-definition.md`

## 対象範囲

- 基本設計書: `<対象機能>` に関係するシート、章、行
- テーブル設計書: `<対象機能>` で参照・更新するシート、テーブル、行
- メッセージ定義書: `<対象機能>` で使用するシート、メッセージ ID、行
- コード定義書: `<対象機能>` で使用するシート、コード値、行

## 生成 Markdown との対応確認

| ai-input.md の項目 | 生成 Markdown                              | 対象シート   | 対象行     | 人間確認 |
| ------------------ | ------------------------------------------ | ------------ | ---------- | -------- |
| `<要件名>`         | `generated-markdown/basic/basic-design.md` | `<シート名>` | `<行番号>` | `OK`     |

## 非対象範囲

- 今回の変更対象外の画面・API・バッチ
- 今回の変更対象外のテーブル
- Excel 設計書に記載がない業務ルール
- Excel 設計書に記載がないメッセージ ID・コード値
```

まず、使用した生成 Markdown を `.kiro/specs/<spec名>/ai-input/generated-markdown/` にコピーします。  
AI は固定スナップショット化した生成 Markdown を参照して、spec に必要な内容だけを `.kiro/specs/<spec名>/ai-input/ai-input.md` に整理します。  
`ai-input.md` には、対象シート、対象行、固定スナップショットのパスを必ず記載します。  
最後に、人間が `generated-markdown/` と `ai-input.md` の対応を確認します。  
確認では、対象シート・対象行が `ai-input.md` に反映されているか、AI が対象外の行や Excel にない仕様を混ぜていないかを見ます。

cc-sdd v3 / Windsurf Skills に渡す入力は、この `ai-input.md` です。

## 6. cc-sdd v3 の spec を作る

`ai-input.md` を前提に、Windsurf Skills で cc-sdd v3 の spec を作ります。  
ここから先は、通常の cc-sdd v3 の流れです。

まず、今回の変更が新規 spec なのか、既存 spec の拡張なのかを整理します。

新規 spec にする場合は、次を実行します。

```
@kiro-spec-init <対象機能>を追加・変更する
```

その後、次の順に進めます。

```
@kiro-spec-requirements
@kiro-spec-design
@kiro-validate-design
@kiro-spec-tasks
@kiro-impl
@kiro-validate-impl
@kiro-verify-completion
```

`requirements.md` には、Excel 由来の機能要件や受け入れ条件を書きます。  
`design.md` には、Excel にはない実装レイヤ、変更ファイル、トランザクション境界、SQL 方針を書きます。  
`tasks.md` には、実装タスク、テストタスク、完了条件を書きます。

DB 変更がある場合、AI は DDL を作成し、確認観点を整理します。  
ただし、実際の DB 変更コマンドの実行、マイグレーション適用、本番・検証環境への反映は行いません。  
DDL は人間が承認し、ユーザが実行します。  
`tasks.md` には、AI が実行しない DB 作業を「ユーザ実施」として明記します。

## 7. 実装後に Excel 設計書との差分を確認する

実装後は、次の 3 つを確認します。

```
1. 実装が cc-sdd v3 spec を満たしているか
2. cc-sdd v3 spec が ai-input.md と矛盾していないか
3. 差分が仕様変更なのか実装不備なのか
```

差分がある場合は、レビューで仕様変更なのか実装不備なのかを判定します。  
仕様変更の場合は、アプリケーションリポジトリ側で Excel を直接直すのではなく、管理元の Excel を更新します。  
実装不備の場合は、Excel と spec を変えずにソースコードを修正します。

```
実装中に差分を発見
  ↓
レビューで仕様変更か実装不備かを判定
  ↓
仕様変更の場合は管理元の Excel を更新
  ↓
設計レビュー
  ↓
Git 管理外の作業フォルダへ再コピー
  ↓
Markdown 再生成
  ↓
ai-input.md / cc-sdd v3 spec を更新

実装不備の場合はソースコードを修正
```

この順番にしておくと、業務仕様と稼働実態のどちらを直すべきかを判断しやすくなります。  
設計書が Git 管理されていない場合も、管理元の版番号、更新日時、ファイルパス、取得日を `ai-input/README.md` に残します。

## プロジェクト固有の前処理 Skills として定義する

毎回同じ手順を手作業で説明しないために、Windsurf Skills 側には「Excel を AI input に変換してから cc-sdd v3 に渡す」というプロジェクト固有の前処理ルールを定義します。  
開発フローでは、Excel 更新と設計レビューが終わった後に `@excel-design-to-ai-input` を実行します。

これは cc-sdd v3 本体の skill ではなく、プロジェクト固有の前処理 skill として作ります。  
`@excel-design-to-ai-input` は、Windsurf に登録したこの固有 skill を呼び出すための名前です。

この skill に、次の一連の作業を閉じ込めます。

```
Excel を作業フォルダへコピーする
  ↓
manifest を作成・更新する
  ↓
ユーザが対象 Excel を指定する
  ↓
Python で Markdown 化する
  ↓
生成 Markdown を spec 配下へ固定スナップショットとして残す
  ↓
AI が ai-input.md に整形する
  ↓
人間が generated-markdown と ai-input.md の対応を確認する
```

定義手順は次です。

```
1. アプリケーションリポジトリに `.windsurf/skills/excel-design-to-ai-input/` を作る
2. その配下に `SKILL.md` を作る
3. `SKILL.md` に目的、前提、手順、禁止事項を記載する
4. Excel から Markdown を生成する Python スクリプトのパスを明記する
5. manifest と固定スナップショットの保存先を明記する
6. 人間が確認する項目を明記する
7. 開発時は `@excel-design-to-ai-input` を呼び出してから `@kiro-discovery` へ進む
```

```
.windsurf/skills/
└── excel-design-to-ai-input/
    └── SKILL.md
```

`SKILL.md` の例です。

```
# excel-design-to-ai-input

## 目的

アプリケーションとは別の場所で管理されている Excel 設計書を作業フォルダへ全量コピーし、ユーザ指定の Excel だけを Markdown 化して、生成 Markdown を固定スナップショットとして残し、AI が `ai-input.md` に整形できる状態にする。

## 前提

- Excel 設計書はアプリケーションとは別の管理元に存在する
- Excel 設計書は基本設計書、テーブル設計書、メッセージ定義書、コード定義書のみ
- メッセージ定義書には、メッセージ ID と文言のみ定義されている
- コード定義書には、コード値とコード名称のみ定義されている
- Excel 設計書を先行して更新する
- アプリケーションリポジトリでは、`.work/excel-design/` に全量コピーした Excel を作業用入力として扱う
- Excel コピー時に `.work/excel-design/manifest.md` を作成・更新する
- 今回の spec で使用した Excel の manifest 行は `.kiro/specs/<spec名>/ai-input/excel-manifest.md` に残す
- Python が読む Excel は、ユーザが指定したファイルだけにする
- Excel 本体は Git 管理しない
- 生成 Markdown は `.kiro/specs/<spec名>/ai-input/generated-markdown/` に固定スナップショットとして残す
- 業務仕様の正は管理元の Excel とする

## 手順

1. 対象機能名と spec 名を確認する
2. 管理元から Excel を `.work/excel-design/source/` に全量コピーする
3. Excel コピー時に `.work/excel-design/manifest.md` を作成・更新する
4. ユーザが今回の spec で使う Excel ファイルを指定する
5. 今回指定した Excel の参照元、取得時点、生成 Markdown の記録方法を確認する
6. 今回指定した Excel の manifest 行を `.kiro/specs/<spec名>/ai-input/excel-manifest.md` にコピーする
7. Python が指定された Excel だけを作業フォルダから読む
8. `scripts/excel_to_ai_input.py` を実行し、指定 Excel だけを Markdown 化する
9. 使用した生成 Markdown を `.kiro/specs/<spec名>/ai-input/generated-markdown/` に固定スナップショットとしてコピーする
10. AI が固定スナップショット化した生成 Markdown を読んで `.kiro/specs/<spec名>/ai-input/ai-input.md` に整形する
11. 人間が `generated-markdown/` と `ai-input.md` の対応を確認する
12. `ai-input.md` を前提に `@kiro-discovery` へ進む

## 禁止事項

- Excel 設計書全体をそのまま AI に渡さない
- 作業フォルダにコピーした Excel を業務仕様の正として扱わない
- Python に対象 Excel の要否を判断させない
- 人間による `generated-markdown/` と `ai-input.md` の対応確認を省略しない
- Excel より先に cc-sdd v3 spec を正として更新しない
- DB 変更コマンドを AI に実行させない
- 基本設計、テーブル設計、メッセージ定義、コード定義以外の設計書がある前提で進めない
- メッセージ定義書に表示条件や制御仕様が書かれている前提で進めない
- コード定義書に enum の実装名や分岐仕様が書かれている前提で進めない
```

cc-sdd v3 本体の skill を直接書き換えるより、プロジェクト固有の前処理 skill として分ける方が運用しやすいです。

## AI input に含める内容

`ai-input.md` には、Excel の内容をそのまま貼るだけでなく、AI が判断に使う前提を明示します。

```
- 参照元の管理場所
- 参照元の版、更新日時、コミット SHA など
- コピー日
- Excel manifest のパス
- 対象機能
- 対象 Excel
- 対象シート
- 対象行
- 固定スナップショットとして残した生成 Markdown のパス
- 対象外範囲
- 基本設計から抽出した要件
- テーブル設計から抽出した DB 情報
- メッセージ定義から抽出したメッセージ ID と文言
- コード定義から抽出したコード値とコード名称
- 未決事項
- AI に判断させてはいけない事項
```

特に、次は明示した方がよいです。

```
AI に判断させてはいけない事項:
  - DB カラム名の新規命名
  - メッセージ ID の新規採番
  - コード値の新規採番
  - Excel にない業務ルールの追加
  - 対象外機能の実装
  - DB 変更コマンドの実行
```

AI に任せるのは、Excel に書かれた仕様を実装可能な粒度に分解する作業です。  
仕様そのものを勝手に増やす作業ではありません。  
DB 変更が必要な場合も、AI は DDL を作成しますが、適用と実行は行いません。  
DDL は人間が承認し、ユーザが実行します。

## PR で確認すること

PR では、コードだけでなく、AI input、Python 依存関係、cc-sdd v3 spec の対応も確認します。

```
- 管理元の Excel が先に更新されているか
- `ai-input/README.md` にコピー元 Excel と、ユーザが指定した対象 Excel の参照情報が残っているか
- `ai-input/excel-manifest.md` に対象 Excel のパス、更新日時、サイズ、SHA-256 が残っているか
- Python で生成した Markdown を元に `ai-input.md` が整形されているか
- 使用した生成 Markdown が `ai-input/generated-markdown/` に固定スナップショットとして残っているか
- `requirements.txt` に Python スクリプトの依存関係が明記されているか
- ai-input.md の対象範囲が明確か
- ai-input.md に対象シート、対象行、固定スナップショットのパスが明記されているか
- 人間が `generated-markdown/` と `ai-input.md` の対応を確認したか
- requirements.md が ai-input.md と矛盾していないか
- design.md に File Structure Plan があるか
- tasks.md にテストタスクが含まれているか
- DB 変更がある場合、AI の作業範囲が DDL 作成と確認観点の整理までになっているか
- DB 変更の実行、適用、反映がユーザ実施として明記されているか
- 実装が tasks.md の Boundary を越えていないか
- 実装中に判明した仕様差分を Excel へ戻したか
```

レビュー時の優先順位は次です。

```
1. 管理元の Excel
2. ai-input.md
3. cc-sdd v3 spec
4. ソースコード
```

業務仕様としては Excel が正で、稼働実態としてはソースコードが正です。  
仕様と実装がずれた場合は、仕様変更なのか実装不備なのかをレビューで判定します。  
仕様変更であれば Excel から更新し直し、実装不備であればソースコードを修正します。

## アンチパターン

### Excel をそのまま AI に渡す

避けた方がよいです。

不要なシート、古い仕様、備考欄、空欄を AI が拾う可能性があります。

### Excel より先に spec を正として更新する

今回の前提では、管理元の Excel 設計書が業務仕様の正です。

cc-sdd v3 spec を先に更新すると、後から Excel へ戻す作業が漏れやすくなります。

### コピーした Excel を業務仕様の正として扱う

作業フォルダの Excel は、あくまで AI input を作るためのコピーです。

業務仕様の正である設計書は管理元にあります。

### 基本設計だけで DB 変更を判断する

この記事の前提では、テーブル設計書が存在します。

DB 変更がある場合は、基本設計だけでなくテーブル設計書も必ず AI input に含めます。  
AI は DDL を作成しますが、DB 変更の適用と実行はユーザが行います。

### メッセージ文言を AI に自由生成させる

メッセージ定義書が存在するなら、メッセージ ID と文言はそこを正にします。

AI には、どの条件でどのメッセージを使うかを整理させます。

### コード定義から enum 実装を決め打ちする

コード定義書にあるのは、コード値とコード名称だけです。

enum のクラス名、定数名、配置場所、変換処理、分岐仕様は、コード定義書だけでは決められません。  
それらは `design.md` で既存コードの構成に合わせて設計します。

## まとめ

Excel 設計書を正式成果物として残す現場でも、cc-sdd v3 / Windsurf Skills は併用できます。

運用の軸は次です。

```
Excel 設計書:
  業務仕様の正
  アプリケーションとは別の管理元で管理
  先行して更新する

作業フォルダの Excel:
  Git 管理外の全量コピー
  業務仕様の正ではない

Markdown:
  Python が指定 Excel から生成する中間ファイル
  作業フォルダ側は Git 管理外
  spec で使用した分は固定スナップショットとして Git 管理

Excel manifest:
  参照 Excel の証跡
  パス、更新日時、サイズ、SHA-256 を記録する

ソースコード:
  稼働実態の正

メッセージ定義:
  メッセージ ID と文言のみ

コード定義:
  コード値とコード名称のみ

cc-sdd v3 spec:
  AI 実装用の作業仕様
  requirements / design / tasks に分ける
```

流れとしては、次の形に固定します。

```
Excel を先行更新
  ↓
@excel-design-to-ai-input で ai-input.md を作る
  ↓
cc-sdd v3 / Windsurf Skills で spec 化
  ↓
実装・テスト
  ↓
Excel との差分確認
```

Excel を正、Markdown を入力、cc-sdd v3 を作業仕様として分けることで、既存の設計書文化を残しながら AI コーディングを導入しやすくなります。
