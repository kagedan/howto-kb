以前、実装のほとんどをAIに任せてチャットボットのプロトタイプを作ってみたところ、次のような問題に直面しました。

* **出来上がった成果物(チャットボットのコード)がブラックボックス化してしまう**
* **ユニットテストがすべてOKのはずが、実際に動作させてみると不具合が残っていた**
* **想定していた契約プランとは少し認識が違うプランを選択していた**

この反省を踏まえ、AIへの指示の出し方を見直して実装しなおした際の記録です。

# 実装方法の検討

## チャットボットを組み込む環境

* 自社ホームページ(astroで作成)
* 公開サーバー(開発：AzureWeb、本番：さくらサーバー)

## チャットボットの実装方法

自社HPへのチャットボット実装方法について、組み込み環境をAIに伝えたうえで、**実装を進める前に**AIに**複数候補**を出してもらいました。**ポイントは**、最初から一つの実装方式に絞り込まず、選択肢を並べて比較することです。

**A. ノーコードSaaSウィジェットを埋め込む(最も手軽)**  
→ 各社から提供されているウイジェットを自社HPに埋め込む

**B. フロントエンドのみのルールベースFAQボット(サーバー不要)**  
→よくある質問とキーワードを対応させたJSON/JSデータを用意し、キーワードマッチして回答する方式

**C. LLM APIを使った簡易バックエンド連携**  
→ClaudeやGPTのAPIを使い、より自然な言い回しの揺らぎにも対応したいケース。  
　簡単なAPIエンドポイントを作りAstroの静的フロントからfetchする構成

**D.B（ルールベース）とC（LLM API）のハイブリッド**  
→よくある質問の回答はパターン化した回答を返し、それ以外はAIに任せる方式

## チャットボットのAIを選定する際に重要なこと

## 使用するモデルの選定

「C. LLM APIを使った簡易バックエンド連携」を使用する場合、  
チャットボットの使用量によって費用がかかります。(従量課金)  
使用するモデルを選定する際、以下のようなことをAIと相談しながら選定を行いました。

**1.　AI提供元の選択**

* Claude
* OpenAI(GPT)
* Azure OpenAI(GPT)

**2.　推論処理のプロンプト実行場所**  
  
Azure OpenAIを使う場合、**推論処理のプロンプト実行場所**を「国内限定」にするかどうかで、使用可能なモデルが変わります。

## モデルごとの料金比較

自社HPのFAQ用チャットボット利用を想定した月額試算を、モデルごとにAIに算出してもらいました。

**前提条件**

* 使用するAIはAzure OpenAI(GPT)
* 推論処理のプロンプト実行場所を「国内限定」にする
* リージョンはJapanEast
* 1回のやり取り:  
  システムプロンプト+FAQ集など入力500トークン + ユーザー質問50トークン  
  ≒ 入力550トークン、回答200トークン
* 月間利用回数: 1,000回(合計 入力55万トークン、出力20万トークン)

| モデル | 入力コスト | 出力コスト | 月額合計（円） |
| --- | --- | --- | --- |
| GPT-4o mini | 約13円 | 約19円 | 約33円 |
| GPT-4.1-mini | 約36円 | 約52円 | 約87円 |
| GPT-4o | 約223円 | 約324円 | 約547円 |

計算式: `月額(USD) = 入力単価×0.55 + 出力単価×0.20`（単位: $/1Mトークン × 百万トークン）、円換算は1USD＝162円で計算。

> 注意: 上記はあくまで単発QA形式を前提とした試算です。

## 不正アクセス対策

### 自社HP側(フロントエンド/Astro)でできる対策

| 対策 | 内容 | 効果 |
| --- | --- | --- |
| 文字数制限 | 入力欄に`maxlength`属性を設定し、長文の入力自体をブラウザ側で防ぐ | 誤操作・簡易ないたずら防止 |
| 送信ボタンの連打防止 | 送信中はボタンを無効化、直前の送信から数秒は再送信不可にする | 誤操作による連投防止 |
| reCAPTCHA / hCaptcha導入 | 送信前に人間かどうかをスコア判定し、怪しい場合は追加認証を要求 | ボットによる自動連投の防止に効果大 |
| 簡易的なクライアント側フィルタ | 明らかな空文字、絵文字の羅列、同一文字の連続などをJSで弾く | 軽微ないたずら投稿の削減 |

> 注意: フロントエンド側の対策は、開発者ツールなどでJavaScriptを無視して直接APIを叩かれると回避可能です。あくまで「正規利用者の誤操作防止」「手間のかからない攻撃の削減」が目的であり、本格的な防御はAPI側で行う必要があります。

### API側(Azure Functions)でできる対策

| 対策 | 内容 | 効果 |
| --- | --- | --- |
| サーバー側の文字数制限 | リクエストボディの文字数をサーバー側でも再検証する | フロント側の制限を回避されても弾ける |
| IPアドレスごとのレート制限 | Azure Table StorageやRedis等に直近のリクエスト回数を記録し、一定時間内の上限を超えたら拒否する | 大量連投・自動攻撃への直接的な防御 |
| CORS制限(ALLOWED\_ORIGIN) | 本番ドメイン以外のブラウザからのリクエストをブロックする | 第三者サイトへの埋め込み・ブラウザ経由の無断利用を防止 |
| reCAPTCHAトークンのサーバー側検証 | フロントから送られたCAPTCHAトークンをAPI側でGoogleに照会し、正当性を確認する | フロント側のCAPTCHAを実効性のあるものにする |
| プロンプトインジェクション対策 | システムプロンプトを無視させる指示や、FAQと無関係な用途(翻訳・文章生成など)への転用を検知・制限する | LLMの悪用によるトークン消費・意図しない出力の防止 |
| APIキーの管理 | Azure OpenAIのキーはApplication Settingsの環境変数直置きではなく、Azure Key Vault経由で参照する | キー漏洩リスクの低減 |
| OpenAI側の月間利用上限設定 | Azure/OpenAI管理画面で使用金額の上限(ハードリミット)を設定する | すり抜けが発生しても金銭的被害を上限内に抑える安全網(ただし反映にタイムラグがある場合があるため過信しない) |
| ログ監視・異常検知 | Azure FunctionsのログやApplication Insightsで異常なリクエスト数を監視し、アラートを設定する | 攻撃の早期発見 |

> 補足: CORS制限は「ブラウザがどのオリジンからのリクエストを許可するか」を制御する仕組みであり、curlや自作スクリプトなどブラウザを介さない直接のAPIコールは防げません。あくまで正規のWebページを経由しない不正利用への一次的な防止策と捉え、実質的な防御はレート制限やAPIキー管理などサーバー側の対策で担保する必要があります。

# 前回からの反省点と対策

対策：  
成果物の**機能仕様書**をAIに頼んで作成してもらい、仕様が想定通りかどうかを確認するようにしました。自分のイメージとは違う実装になっている箇所や、余計な機能が追加されていることが判明し、何度も修正を重ねました。

AIが作成したコード・ユニットテストは、**現状想定できる環境**を対象に作成されています。  
サーバー環境・セキュリティ設定による影響や、通信時間のタイムアウト時の動作など  
環境によって異なる動作をすべて網羅してくれるわけではありません。

テストの作成については、  
**ローカルPCの環境で想定できるテストを作成**  
開発環境サーバーにデプロイ後は、  
**アクセス制限、環境設定ミスのチェックや、タイムアウト時のテスト追加**  
など、現場では、AIと何度もやりとりしてテストの精度を上げました。

# さいごに

AIが最初に自動生成してくれたコード・テストは、あくまでプロトタイプという認識でいるべきです。  
そこから本番環境に耐えうる精度に近づけるには、

* 事前の検討(複数案から選定)
* 仕様の明確化(人間が見やすい形でドキュメント化)
* 環境の差異(開発・本番)によって動作に差が出ることを想定したテスト

が必要であることを実感しました。  
新しいAIツールも、結局は使う側が根拠を持って選び、検証してこそ活きるもんです。  
便利な道具ほど「なぜそう動くのか」を手放したらあかんのやと思います。
