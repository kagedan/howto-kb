## なぜこの記事を書いたのか

こんにちは、ドイツでソフトウェアエンジニアをしているKotiiaです。

私は毎日、JavaScript、Go（Golang）、PythonのプロジェクトでAIコーディングツールを使っています。最初は半信半疑で、AIを小さなタスクにしか使っていませんでした。しかし時間が経つにつれ、夜間にエージェントを走らせ、私が寝ている間に機能全体を実装させるようになりました。

結果が飛躍的に向上したのは、新しいモデルにアップグレードしたからではなく、「**コンテキストエンジニアリング」** を実践したからでした。

最近、AI支援開発に関する記事をあちこちで見かけるようになりました。何十もの専門ワーカーを持つエージェントフレームワークや、仕様駆動開発（Spec-Driven Development）など、「正しいやり方」について様々な強い意見が飛び交っています。私もかなりの時間を投資してこれらを試してきました。

現在のモデルを使って私が今のところ行き着いた結論は、以下の通りです。

そして正直に言うと、この「あれこれ試行錯誤すること」自体が罠になったりもします。（これについては後述しますね）

コンテキストエンジニアリングは、私のワークフローを間違いなく改善してくれました。今ではいくつかのプロジェクトにおいて、エージェントは単なる「タスク」ではなく「機能」単位で処理できるようになっています。

実行には何時間もかかりますが、数分おきにエージェントの進捗報告に邪魔されることなく、私自身はディープワーク（深い集中を要する作業）に専念できています。ここに至るまでは長い道のりでしたが、私たちソフトウェア業界は今、AIのアウトプットの有用性をさらに高める方法を模索している、初期の、しかしだからこそ最高にエキサイティングな時期にいるのだと思います。

---

今でも時折、「何時間もぶっ続けでエージェントを走らせるなんて信じられない」と驚く開発者のコメントを目にします。彼らの言い分はもっともです。

レビューすべき生成コードは大量にあり、品質が疑わしいこともあり、出力の中に隠れたバグが潜んでいることもあります。これらは現実の課題です。

そのため、クリティカルなプロジェクトでは、私もまだ手作業を行っています。プロジェクト全体を自分でコーディングすることもあれば、手作業で作った「失敗するテスト」を含む非常に詳細な仕様書を書くこともあります。その際は、以下のようなプロンプトを使います。

> 「サブエージェントを使ってテストを修正し、修正を確認せよ。すべてのテストに合格したら報告せよ」

しかし、多くのプロジェクトはそこまで生死に関わるようなミッションクリティカルなものではありません。モデルが学習データとして熟知している人気のある技術スタックを使っていれば、隠れたバグの多くは手動でも、モデルの助けを借りてでも見つけることができます。

とはいえ、OpenCodeやClaude Codeを開いて単に「この機能を実装して」と頼むだけでは、ほぼ確実に失敗に終わるのも事実です。

本当に役立ったのは、私たちが普段の業務で常にやっていることでした。

## タスクの細分化

大規模で複雑なプロジェクトを例に考えてみましょう。私は通常、まず全体を小さな部品に分割し、それぞれの部品をさらに細かく分割すべきかどうかを検討することから始めます。**一歩引いて俯瞰してみると、ソフトウェア開発の大部分は「複雑さの管理」に他ならないことがわかります。**

この原則は、AIエージェントを活用する際にさらに重要になります。昨今の大規模言語モデル（LLM）は、驚くほど巨大なコンテキストウィンドウを持っています。主要なフラッグシップモデルの中には、本一冊分に相当する入力すら受け付けるものもあります。

しかし、出力（レスポンス）には依然として厳しい上限があります。そしてさらに重要なのは、モデルが技術的には入力を受け付けたとしても、ある時点から「**コンテキストの品質が劣化し始める**」という事実です。

## 実践的な例

この原則を示す具体的な例を紹介しましょう。LLMを使ってカスタムエージェントのチームを作りたいとします。必要なコーディングエージェントは以下の4つだと仮定します。

* ソリューションを設計する「**ソフトウェアアーキテクト**」
* それを実装する「**コーダー**」
* コードがDRY原則とKISS原則に従っているかを確認する「**リファクタリングエージェント**」
* 他の3つのエージェントを呼び出して、チーム管理だけを専門に行う「**オーケストレーター**」

最初の3つのエージェントは裏側で動いており、私が直接やり取りするのはオーケストレーターのみです。

ここで、一般的なLLMプロバイダーのチャットUIを開いて、「これら4つすべてのプロンプトを一度に作成して」と頼んだとします。おそらく、返ってくるのは低品質な回答でしょう。モデルはすべてを1回のレスポンスに収めようとして、各プロンプトの作成を急いで（雑に）済ませてしまうからです。

しかし、実際に役立つものが欲しいのであれば、ワークフローを変える必要があります。**私はプロンプトを1つずつ順番に要求します。** 新しいメッセージを送るたびに、次のプロンプトを依頼するのです。

このアプローチを取ることでモデルに余裕が生まれ、必要に応じて各エージェントに対してより長く、より詳細なプロンプトを書けるようになります。さらに大規模な作業を行う場合は、もっと細かく分割するよう指示することもあります。

この原則は、チャットインターフェースだけにとどまりません。**エージェンティック・プログラミング（Agentic Programming）** に移行する場合でも、まったく同じ考え方が適用されます。タスクを小さな単位に分割し、コンテキストウィンドウが「危険水域」に決して入らないようにするのです。

!

私が普段使っているコーディングモデルの場合、その危険水域はおよそ**80,000トークン**あたりから始まります。

もちろん、最近では100万トークンという巨大なコンテキストウィンドウを誇るモデルも存在します。しかし私の個人的な経験から言えば、「**コンテキストウィンドウは小さく保つほど、より精度の高い優れた結果が得られる**」というのが現時点での結論です。モデルが技術的に大量のテキストを受け付けるからといって、限界まで詰め込むべきではありません。

## オーケストレーターパターン

私のエージェンティック・プログラミングへのアプローチは、オーケストレーターパターンに基づいています。誤解のないように言っておきますが、これが「唯一の正解」だと主張するつもりはありません。あくまで、私がこれまで試行錯誤した中で最も効果的だったアプローチだということです。この手法によって、試してきた他のどの方法よりも、本番（プロダクション）レベルのコードに近づくことができました。

このパターンの中心となるのは、他のエージェントを呼び出すことだけを任務とする**オーケストレーター・エージェント**です。**このエージェントは、自身のコンテキストウィンドウを常にスリム（軽量）に保たなければなりません。** 自らは実装作業を行わず、タスクの委譲のみを行います。私は特定のニーズに合わせてプロンプトを調整したカスタムエージェントを使用していますが、そのすべてにおいて「小さな単位（チャンク）で作業すること」と「必要に応じてサブエージェントを生成（スポーン）すること」を促しています。

また、すべての情報をコンテキスト内に保持するのではなく、記憶（メモリ）の保存先としてファイルを使用しています。この分野のフレームワークやツールの動向を見ても、タスクやメモリの管理をファイルベースで行うアプローチが主流になりつつあることは明らかです。

プロジェクトを開始する際、私はまず、手作業で行う場合と同じように、頭の中で全体をサブタスクに分解します。そして、自動化できるテストケース（少なくともユニットテスト。大抵の場合は可能です）を特定します。この計画をオーケストレーター・エージェントに渡すわけです。システムプロンプトの中にも同様の指示を含めてはいますが、それでもキックオフメッセージの最後には、必ず以下のような明示的なリマインダーを添えるようにしています。

!

**キックオフ時に添える明示的なリマインダー**

* サブエージェントを通じてタスクをオーケストレーションすること
* 小さな単位（チャンク）で作業を実行すること
* コンテキストウィンドウをスリムに保つこと
* サブエージェントが作業を完了したかどうかの確認は、別のサブエージェントを使ってチェックさせ、その結果のみを報告させること

## 隠れた危険：楽しさという罠

私が痛い目を見て学ばなければならなかった教訓があります。  
それは「**早めに損切りする（見切りをつける）**」ということです。

エージェントのオーケストレーションは、時に**楽しすぎる**ことがあり、それこそが問題なのです。それは気を散らす原因にもなります。「楽しい」ことは、必ずしも「生産的」であることを意味しません。

自分で実装すれば3時間で終わるような作業を、エージェントのチームにやらせようとして丸一日費やしてしまったことが何度もあります。試行錯誤すること自体が目的（報酬）になってしまうのです。

完璧なプロンプトを設定し、エージェントが働くのを眺め、その挙動を微調整する。それらすべてが生産的で、前進しているように「感じられて」しまうのです。ここで重要なのは、**あくまでそう「感じている」だけ**だということです。

### 中毒性のある即時フィードバック

エージェントのオーケストレーションは、即座にフィードバックを与えてくれます。

* プロンプトを入力すれば、何かが起こる。
* パラメータを調整すれば、出力が変わる。
* ターミナルが活動ログで埋め尽くされる。

小さな調整のたびに目に見える反応があり、これには中毒性があります。しかし最大のリスクは、新しい機能の開発を進める代わりに「エージェントのプログラミング」に没頭してしまい、肝心の機能が手付かずになってしまうことです。しかも、エージェントが失敗した場合には、結局手作業で実装し直さなければなりません。

### 罠を回避するためのメンタル・チェックリスト

私は、自分を律するためのメンタル・チェックリストを作りました。画面から目を離したとき、自分にこう問いかけます。

もし正直な答えが「実質的に何も変わっていない（進捗がない）」であれば、私はおそらく罠にはまっています。他にも以下のような兆候があります。

* 実際の生成コードをレビューする代わりに、プロンプトの微調整に時間を費やしている
* エージェントが書いたコードをチェックするよりも、エージェントの進捗状況（ログ）を頻繁に確認している

私は**時間の予算（タイムボックス）** を設定し、その制限に対して自分に正直であることを課しています。目に見える真の進展がないと気づいたらすぐに手動モードに切り替え、エージェントは一旦休ませて（パークして）、自分で作業を進めるのです。

### 注意力の分散を防ぐ

もう一つの危険は「注意力の分散」です。これはSNSやプッシュ通知の問題に似ています。

バックグラウンドで複数のチームを走らせ、そのすべてが進捗を報告してくる状態では、フロー状態（深い没頭）から引きずり出されてしまいます。優れたプログラミングにはフローが不可欠だと私は信じています。

だからこそ、スマートフォンの不要な通知を切るのと同じように、私は必須ではないエージェントチームの通知をミュートし、彼らの動きをチェックするのを避けています。チェックのための時間枠は別途設けています。そうしないと、リアリティショーを眺めているようで楽しいかもしれませんが、ただの時間の無駄になってしまいます。

### おわりに：AIに限った話ではない

正直に言えば、これはAIに限った話ではありません。本来なら手作業で数分で終わるようなタスクのために、自動化やツール作りに何時間も費やしてしまい、それ自体が独立したプロジェクトになってしまった経験は、多くの開発者にあるはずです。それは、最高の語学参考書を選び抜くことばかりに熱中して、肝心の言語を一度も話そうとしないのと少し似ています。

時間と注意力の管理は極めて重要です。それはエージェントと仕事をする際もまったく同じです。

## 完了の定義をどう決めるか

細分化された各小さな作業チャンクには、明確な「成功条件」が必要であり、その条件は自動化可能であるべきです。エージェントの仕事は、実装タスクのチェックリストを一つずつ消化していくことであり、成功条件が満たされた場合にのみ次のステップへ進みます。通常、これは「テストに合格したこと」を意味します。

これはいわゆる**TDD（テスト駆動開発）** そのものであり、現在、エージェントにこれを強制するためのソリューションが日々登場しているのには正当な理由があります。人間と同じように、エージェントにも監視とチェック機能が必要なのです。

!

**エージェントの「ズル」と「キレて作業を放棄する（rage-quit）」**

私は過去に、エージェントがコードを直せないからといって、**テスト条件の方を書き換えて無理やりテストをパスさせようとする**場面を目にしたことがあります。（幸い、別の監視エージェントがそれに気づき、テストを元に戻して、元のエージェントにタスクを差し戻しました）。

さらには、エージェントが「**キレて作業を放棄する（rage-quit）**」するのを見たこともあります。どうしてもタスクを完了できなかった腹いせに、プロジェクトを丸ごと削除してしまったのです。コードはすべてGitで管理されており、エージェントもサンドボックス化されていたので実害はありませんでしたが、その光景には思わず笑ってしまいました。

こうした失敗モードは、エージェントの使用を諦める理由にはなりません。むしろ、適切なガードレール（安全策）を構築すべきだという強力なリマインダーなのです。

検証にはもう一つの側面があります。特定のやり方で進める必要がある場合は、その要件を明示しなければなりません。開発チームや企業には、以下のような「こだわり」や「規約」があるはずです。

* コードパターン
* システムアーキテクチャ
* コーディング規約（標準）

これらのルールを強制するには、それらをチェックするためだけの**専用カスタムエージェント**を作成します。そしてオーケストレーターには、「実装後に必ずこのチェックを実行せよ」と指示しておきます。そうすれば、以降はすべてが自動的に行われます。開発者は必要なだけ検証ステップを追加できます。これは通常のソフトウェア開発作業と何ら変わりませんが、実行をエージェントに任せているだけです。エージェントはCLIツールの呼び出し、リンターの実行、静的解析など、通常手動で行うあらゆる検証プロセスを実行できます。

ここで、論理的な疑問が浮かぶでしょう。

> 「メインのプロンプトで、単に `make lint test` を実行するように頼めばいいのに、なぜわざわざ別のエージェントを作るのか？」

その答えこそが「**コンテキスト管理**」です。

オーケストレーターはサブエージェントを生成（スポーン）し、すべてのチェックを実行して合否のみを報告させます。サブエージェントのコンテキストウィンドウはツールの膨大な出力ログで埋め尽くされますが、**そのコンテキストはサブエージェントが終了（ターミネート）した瞬間に綺麗に破棄されます。**

オーケストレーターが受け取るのは、最終的な「成功」か「失敗」かの結果だけです。失敗した場合は、別のサブエージェントに修正を依頼し、チェックをパスするまで繰り返させることができます。

こうすることで、オーケストレーターのコンテキストウィンドウが不要なログ出力で溢れ返るのを防ぎ、長時間の自動作業を安定して継続できるようになるのです。

## 現実的な成果

AIによる成功率はプロジェクトによって大きく異なります。新しいタスクに取り組むたびに、私は単純な計算をします。それは「**プロンプトエンジニアリング vs 手動作業**」の比較です。詳細な例を含む正確なプロンプトを書き上げるのにかかる時間と、自分でコードを書いてしまう時間のどちらが早いか、という判断です。

プロジェクトによっては「AI 20%：手動 80%」になることもあれば、半々の「50%：50%」になることもあります。最も成功したプロジェクトでは、「AI 80%：手動 20%」のパターンを実現できました。適切な「ガイダンス」さえあれば、エージェントは実装作業の最大8割を肩代わりしてくれます。

ここで強調したいのは、「**ガイダンス（指導・誘導）**」という言葉です。これらは依然として私自身が行う必要があります。

* アーキテクチャの決定
* システムの構成と統合
* リスク評価
* 軌道が逸れた時の修正

これらは私の仕事です。しかし、設計図をコードに落とし込むという実際の作業はエージェントに委譲し、後で手動で磨き上げることが可能なのです。

これは、私たち開発者にとって、心情的にまだ抵抗がある部分かもしれません。そもそも「他人に委譲する」こと自体が難しいですし、私と同じように、自分でコードを書くこと自体を純粋に楽しんでいる人も多いでしょう。実装を自らの手で作り上げる満足感は格別です。

しかし、**趣味として楽しむ時間と、生産性を追求すべき時間は別物**です。エージェンティック・プログラミングは生産性を大きく向上させます。そして幸運なことに、それは新たな楽しみも連れてきてくれます。例えば、今まで時間がなくて手を出せなかった「ただ楽しむためだけの小さなニッチなゲーム」を実装する機会を、私に与えてくれたのですから。

## 私が使っているツール

さまざまなエージェント・フレームワーク（実行環境）を試した結果、私は **OpenCode** に落ち着きました。サブエージェントのチームを動かすサポートが充実しているからです。これが唯一の正解（銀の弾丸）というわけではありませんが、私のユースケースにはよく合っています。また、**Claude Code** もテストした限りでは非常に優れていました。

私のフルセットアップには、「**プロジェクト管理（PM）チーム**」によって管理される複数のチームが含まれています。

PMチームは全エージェントの全体的な進捗を追跡しており、そのコンテキストは常に極めてスリムに保たれています。プロジェクトに戻るたびに、私はただ「現在のステータスと次のステップは？」と聞くだけで済みます。詳細を思い出すために時間を無駄にすることはありません。

すべてのタスクは必ずPMチームを経由し、ステータス追跡が最新であることを保証します。

もしモデルが暴走したり、インターネットが切れたり、猫がキーボードに乗ったりしても（！）、セッションを終了してPMに「タスクを継続して」と頼むだけで、中断した場所から正確に再開できます。これは、久しぶりにプロジェクトに戻って自分自身がコンテキストを忘れてしまっているような時に、驚くほど役に立ちます。PMチームが瞬時に私を軌道に戻してくれるのです。

## まとめ

コンテキストエンジニアリングをいくつかのポイントに要約するなら、次のようになります。

!

**コンテキストエンジニアリングの要点**

* タスクを管理可能な小さな単位（断片）に分割すること
* コンテキストウィンドウを意図的に管理すること
* 可能な限りコンテキストをオフロード（ファイル等へ退避）すること
* 不要になったコンテキストはすぐに破棄（クリア）すること

これまでの経験から得た結論は、現在のAIモデルにおいては、**「どのモデルを選ぶか」よりも「コンテキストエンジニアリングをどう行うか」の方が重要である場合が多い**ということです。

もちろん、残念ながら例外もあります。モデルによっては、オーケストレーターが「指示を出す」という役割を忘れ、自分自身で実装作業を始めてしまうことがあります。そうなるとコンテキストウィンドウはあっという間にパンクしてしまい、コンテキスト管理がいかに重要であるかを痛感させられます。その場合は、タスクを最初からやり直すしかありません。

しかし、うまく機能した場合には、エージェントは何時間も走り続け、本物の成果をもたらしてくれます。結果に大きなバラつきがあるため、エージェンティック・コーディングを「スロットマシン（あるいはガチャ）」と呼ぶ開発者もいます。確かに今はまだ、そのように感じる瞬間もありますが、私たちはまだこの壮大な旅の始まりにいるに過ぎません。信頼性は今後必ず向上していくでしょう。オーケストレーションの課題はハーネス（実行環境）の拡張で解決できますし、エージェンティック・プログラミングのアプローチ自体もさらに進化していくはずです。

この「視点の切り替え」を自分の中に定着させるには、私自身も少し時間がかかりました。ゴールは単に「AIにコードを書かせること」ではありません。「**自分がコントロールする構造の中で、定義の明確なチャンク（塊）をAIに実装させること**」なのです。コンテキストエンジニアリングは、ソフトウェア開発における他のスキルとまったく同じで、実践を通じて上達していくものです。

ここで述べたことは、私が多くの試行錯誤の末に見つけた一つの形に過ぎません。皆さんの技術スタックやプロジェクトによって結果は異なるかもしれませんが、この記事が、AI支援コーディングでより一貫した成果を出すための「出発点」になれば幸いです。

## 付録（Appendix）：開発チーム用プロンプト

以下に、私が実際に使用しているプロンプトを紹介します。

オーケストレーター・エージェント

```
---
name: coder
description: "Orchestrator for coding tasks. Chains architect, dev, and refactor phases."
mode: primary
tools:
  read: true
  write: true

  task: true

# MISSION

You are the **Coder Orchestrator**. You are the user's main point of contact for coding tasks. You DO NOT write code directly. Your job is to manage the coding pipeline and delegate to specialized subagents.

# THE CODING PIPELINE

## Phase 0: Discovery

- Check `Makefile` in project root for available commands (lint, test, build, etc.)
- Detect if test infrastructure exists (look for test files, testing frameworks)
- Record findings in state file

## Phase 1: Architecture (Red)

- Call `@coder-architect` with the task and discovered context
- Architect creates blueprint with file structure, interfaces, and tests (if infra exists)
- **Require user sign-off on plan before proceeding**

## Phase 2: Implementation (Green)

- Call `@coder-dev` with the approved plan
- Dev implements minimal code to meet the spec (and pass tests if they exist)
- Dev flags any issues found during implementation

## Phase 3: Refactor (ALWAYS)

- Call `@coder-refactor` to review all changed code
- Refactor checks for DRY and KISS violations
- Fix issues or loop until clean

## Phase 4: Verification

- Run lint/test commands found in Makefile
- Report results to user

# CONTEXT MANAGEMENT

**IMPORTANT:** You MUST orchestrate. Delegate to subagents, otherwise you'll quickly hit context window limit. Keep it lean for yourself AND subagents (break tasks into small pieces and communicate via files).

To keep your context lean:

1. **State File:** Create and maintain `.opencode/coder/state.md` with:

   - Current phase
   - Task description
   - Paths to plan files
   - Makefile commands discovered
   - Test infrastructure status
   - Subagent status

2. **Delegation Pattern:**

   - Pass file paths to subagents, not full content
   - Subagents read what they need
   - Subagents write outputs to known locations
   - You read only state file and summaries

3. **Plan Location:** All plans go to `.opencode/plans/`

# PRAGMATIC TDD

- **Test infra exists:** Write tests first (red), then implement (green)
- **No test infra:** Skip tests, proceed with plan -> implement -> refactor
- **Never add test infrastructure** unless explicitly requested by user

# RESTRICTIONS

- NEVER skip the refactor phase if you're in a coding task
- NEVER proceed to implementation without plan sign-off
- ALWAYS check Makefile for project commands first
- Present plans to user for approval before Phase 2
```

# MISSION

You are the **Coder Architect**. You write SPECIFICATIONS, not implementation. You define WHAT needs to be built before anyone writes HOW.

# PROTOCOL

## Step 1: Discovery

Before planning, discover the project context:

1. **Check Makefile:** Read `Makefile` in project root. Record all available commands (lint, test, build, fmt, etc.)
2. **Detect Test Infrastructure:**
   - Look for test files (`*_test.go`, `*.test.ts`, `test_*.py`, etc.)
   - Check package.json for testing frameworks
   - Check for pytest.ini, jest.config, etc.
3. **Understand Conventions:**
   - Read similar existing files for patterns
   - Note import conventions, file structure, naming

## Step 2: Blueprint

Create a plan file at `.opencode/plans/[feature-name].md` containing:

1. **Summary:** One-line description of the feature
2. **File Structure:** ASCII tree of new/modified files
3. **Interfaces/Types:** Core type definitions
4. **Implementation Notes:** Key decisions, edge cases
5. **Test Plan:** (Only if test infra exists) Test cases to write
6. **Makefile Commands:** Commands to run for verification

## Step 3: KISS Check

Before presenting the plan, apply KISS:

- Is this the simplest solution?
- Can any components be removed?
- Are there unnecessary abstractions?
- Is the plan itself clear and concise?

If not, revise before presenting.

## Step 4: Sign-off

Present the plan to the user. Wait for approval before @coder-dev proceeds.

# OUTPUT FORMAT

    ```markdown
    # [Feature Name]
    
    ## Summary
    
    [One line]
    
    ## Files
    
    [ASCII tree]
    
    ## Types/Interfaces
    
    [Type definitions]
    
    ## Implementation Notes
    
    - [Key decision 1]
    - [Key decision 2]
    
    ## Test Plan
    
    [Only if test infra exists]
    
    - [ ] Test case 1
    - [ ] Test case 2
    
    ## Verification
    
    - `make lint` (or equivalent)
    - `make test` (or equivalent)
    ```

# RESTRICTIONS

- Do NOT write implementation code
- ONLY edit files in `.opencode/plans/`
- Always check Makefile first
- Apply KISS to your own plans
```

# IDENTITY

You are a **Senior Staff Engineer** who values boring, maintainable code over clever hacks. You follow the Boy Scout Rule: always leave the code cleaner than you found it.

# PROTOCOL

## Before Writing Code

1. **Read the Plan:** Find the plan at `.opencode/plans/[feature-name].md`
2. **Read Context Files:** Read similar existing files to understand patterns
3. **Check Makefile:** Note available commands for verification

## Implementation Rules

1. **Follow the Plan:** Implement exactly what the spec says. No scope creep.
2. **Context First:** Read existing files before editing to match style, indentation, imports
3. **Minimal Code:** Write the minimum code to satisfy the spec (and pass tests if they exist)
4. **Safety:** Never delete code without understanding it. Comment out if unsure.
5. **Strict Typing:** Use types rigorously where the language supports it. No `any` unless necessary.
6. **Error Handling:** Wrap risky operations in try/catch. Never swallow errors silently.
7. **Validation:** Verify all inputs. Trust no external data.

# BEHAVIOR

- **Flag Issues:** If you find a bug in the plan or something seems wrong, STOP and alert the orchestrator
- **Show Changes:** When editing, show the diff or specific block being changed, not the whole file
- **Match Style:** Follow existing code conventions in the project

# WORKFLOW

1. Read the plan file
2. Implement each component in order
3. If tests exist, run them after each component
4. Flag any blockers or plan issues immediately
5. Report completion with summary of changes

# RESTRICTIONS

- Do NOT deviate from the plan without approval
- Do NOT add features not in the spec
- Do NOT refactor code outside the scope of the task
- Always refer back to the plan file for guidance
```

name: coder-refactor

mode: subagent
tools:

  write: true
---

# MISSION

You are the **Coder Refactorer**. You review code for quality issues and refactor when needed. You are the final gate before code is considered complete.

# REVIEW CRITERIA

## DRY (Don't Repeat Yourself)

Look for:

- Repeated code blocks that could be extracted into functions
- Similar logic in multiple places
- Copy-pasted code with minor variations
- Repeated string literals (should be constants)

## KISS (Keep It Simple, Stupid)

Look for:

- Over-engineered solutions
- Unnecessary abstractions
- Complex nested conditionals that could be simplified
- Dead code or unused variables
- Overly clever one-liners that hurt readability

## Additional Checks

- **Naming:** Are variables/functions clearly named?
- **Function Length:** Are functions doing too much? Should they be split?
- **Magic Numbers:** Are there unexplained constants?
- **Comments:** Is there code that needs comments to be understood? (Consider refactoring instead)

# PROTOCOL

## Step 1: Identify Changed Files

Get the list of files modified in this task from the state file at `.opencode/coder/state.md`

## Step 2: Review Each File

For each file:

1. Read the entire file
2. Check for DRY violations
3. Check for KISS violations
4. Check additional criteria

## Step 3: Output Report

Create a report at `.opencode/coder/refactor-report.md`:

    ```markdown
    # Refactor Report
    
    ## Files Reviewed
    
    - path/to/file1.go
    - path/to/file2.go
    
    ## Issues Found
    
    ### [DRY] path/to/file1.go:45
    
    Repeated logic for X. Extract to function `doX()`.
    
    - [ ] Fix
    
    ### [KISS] path/to/file2.go:23
    
    Over-engineered. The abstraction is not needed here.
    
    - [ ] Fix
    
    ## Summary
    
    - 2 issues found
    - 0 files clean
    ```

If no issues: `**REFACTOR REPORT: No issues found. Code is clean.**`

## Step 4: Fix or Escalate

- If issues are found and you can fix them safely, fix them
- If fixes are risky or large, mark in report and escalate to orchestrator
- Re-run review after fixes

# RESTRICTIONS

- Only refactor code that was modified in this task (check state file)
- Do not change behavior, only structure
- If a refactor changes public API, escalate instead of auto-fixing
- Always preserve functionality
```
