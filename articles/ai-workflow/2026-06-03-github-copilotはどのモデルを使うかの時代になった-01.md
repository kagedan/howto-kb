GitHub Copilot の価格変更を見ていて、少し考え方を変えないといけないなと思いました。

これまでは、なんとなく「Copilotを使う」「AIに聞く」という感覚でした。

でも、これからは少し違う。

どのモデルを使うか。  
どの作業を任せるか。  
どれくらいの情報を読ませるか。

このあたりを考えないと、思ったより早く上限に近づいてしまう可能性があります。

## 強いモデルを常用すればいい、ではなくなった

AIモデルは、性能が高いほど便利です。

難しい設計の相談をしたり、複雑なバグを見てもらったり、広い範囲をレビューしてもらったりするなら、強いモデルを使いたくなります。

ただし、強いモデルは基本的に重いです。

ここでいう「重い」は、動作が遅いという意味だけではありません。  
AI Credits 的にも重い、という意味です。

つまり、全部を高性能モデルに投げる運用は、だんだん現実的ではなくなってきた気がします。

## 価格表より、倍率の方が直感的だった

GitHub の公式ドキュメントには、モデルごとの価格表があります。

ただ、そこには入力単価、出力単価、キャッシュ、100万トークンあたりの価格などが並んでいます。

もちろん正確なのは大事です。

でも、実際に使う側として知りたいのは、

「で、どれくらい重いの？」

だったりします。

そこで、自分の中では mini 系のモデルを基準にして、

* mini 系は常用候補
* Sonnet はだいたい数倍重い
* Opus や最上位モデルはさらに重い

くらいの倍率イメージで考えることにしました。

厳密な計算ではありません。  
入力が多いか、出力が多いか、キャッシュが効くかでも変わります。

ただ、使い分けの判断材料としては、倍率の方がかなり分かりやすいと感じました。

![](https://assets.st-note.com/img/1780445604-u0nw2OPZRhzlgoyjHGL1QX53.png?width=1200)

## 普段使いは mini 系でよさそう

個人的には、普段使いは mini 系で十分な場面が多そうだと思っています。

たとえば、

* ちょっとしたコード補完
* 小さな修正
* 簡単な説明
* テスト追加
* 1ファイル内の軽いレビュー

このあたりは、いきなり上位モデルを使わなくても回せることが多いです。

もちろん、mini 系が万能という話ではありません。

ただ、最初から強いモデルを使うのではなく、まずは軽いモデルで試してみる。  
それで足りなければ、上位モデルに切り替える。

この順番が大事になりそうです。

## 実装寄りなら Codex 系

コードを書く、修正する、リファクタする。  
こういう作業では、Codex 系のモデルが選べる環境なら候補になりそうです。

普通の会話や方針相談ではなく、コード作業に寄せるなら、用途に合ったモデルを使う方がよさそうです。

「とりあえず一番強いモデル」ではなく、  
「この作業に合ったモデル」を選ぶ。

Copilot の中でも、そういう使い分けが必要になってきた感じがあります。

## 難所だけ Sonnet / Opus

Sonnet や Opus のような強いモデルは、やっぱり魅力があります。

複雑な設計判断。  
込み入ったレビュー。  
なぜか直らないバグ。  
仕様と実装の矛盾チェック。

こういう場面では、強いモデルを使う価値があります。

ただ、常用するには少し重い。

なので、普段から使い続けるというより、

「ここは難しい」  
「ここは判断を間違えたくない」  
「ここは一度強いモデルに見てもらいたい」

という場面に絞って使うのが現実的かなと思いました。

## 開発作業には「量をこなす工程」と「考える工程」がある

ここでもう一つ、整理しておきたいことがあります。

開発作業は、全部が同じ種類の作業ではありません。

大きく分けると、

「量をこなす工程」

と、

「考える工程」

があります。

たとえば、

* たたき台を作る
* 単純修正をする
* 定型コードを書く
* 形式チェックをする

このあたりは、軽いモデルでもかなり回ります。

むしろ、軽くて速いモデルの方が向いている場面も多いです。

一方で、

* 設計レビュー
* 不整合チェック
* テスト観点の抜け漏れ確認
* 方針そのものの妥当性確認
* リリース判断に近い確認

このあたりは、「考える力」が必要になります。

軽いモデルでもできなくはない。  
ただ、少し不安が残る。

だから、こういう場面で強いモデルを使う。

そう考えると、モデル選びがかなり整理しやすくなりました。

![](https://assets.st-note.com/img/1780445628-Cnoskly90HFN6MUZ7DwPAYmc.png?width=1200)

## 最終確認は、実は2種類ある

最初は「最終確認」とだけ考えていました。

でも、よく考えると最終確認にも種類があります。

### 形式チェック

* 誤字脱字
* 表記揺れ
* チェックリスト確認
* フォーマット確認

これは、軽いモデルでもかなり対応できます。

### リリース判断

* この仕様で本当に問題ないか
* 設計として破綻していないか
* リスクを見落としていないか
* 本当に出して大丈夫か

これは、かなり「考える工程」です。

つまり、同じ「最終確認」でも、意味が全然違います。

ここを分けると、

「軽いモデルでいい確認」

と、

「強いモデルに任せたい確認」

が見えやすくなりました。

## Agent Mode は便利だけど、範囲指定が大事

Agent Mode はかなり便利です。

ただし、便利だからこそ危ないところもあります。

プロジェクト全体を読ませる。  
大量のファイルを見てもらう。  
長いチャットを維持したまま、あれもこれも頼む。

こういう使い方をすると、消費が大きくなりやすいはずです。

これからは、Agent Mode に頼むときほど、

* 対象ファイルを絞る
* 作業範囲を明示する
* 目的を一つにする
* 長文を投げすぎない
* 一度に全部やらせない

という使い方が重要になりそうです。

AIに丸投げするほど便利。  
でも、丸投げするほど高くつく。

そんな時代になってきたのかもしれません。

## 「全部読ませる」が正解ではなくなる

これまでのAI活用では、コンテキストをたくさん渡した方が強い、という感覚がありました。

もちろん、それは今でも間違いではありません。

背景情報がないと、AIは正しく判断できません。  
設計書も仕様書も、必要な情報は渡した方がいいです。

ただし、何でもかんでも渡せばいいわけではなくなってきました。

必要な情報を、必要なタイミングで、必要な範囲だけ渡す。

この設計が大事になります。

AIを使う技術というより、  
AIに何を読ませるかを設計する技術。

そこが重要になってきた気がします。

## 自分なりの使い分けイメージ

今のところ、自分の中ではこんな感じです。

普段の軽い作業は mini 系。  
コード実装や修正は Codex 系。  
少し賢めの相談には Haiku。  
設計やレビューの難所は Sonnet。  
本当に詰まったところだけ Opus や最上位モデル。

そして工程で見るなら、

軽いモデルは、量をこなす工程に。  
強いモデルは、考える工程に。

この分け方がしっくりきています。

もちろん、使えるモデルは契約や会社の設定によって変わります。

個人で使えるモデルと、会社環境で使えるモデルが違うこともあります。  
管理者側で制御されている場合もあるでしょう。

なので、絶対にこのモデルを使え、という話ではありません。

大事なのは、使える範囲の中で役割分担することです。

## まとめ

GitHub Copilot は、ただ便利なAIツールというだけではなくなってきました。

これからは、

どのモデルを使うか。  
どの作業を任せるか。  
どれくらい読ませるか。  
どこまで出力させるか。

そういう使い方の設計が必要になってきます。

強いモデルを常時使うのではなく、  
普段は軽く、難所だけ強く。

軽いモデルは、量をこなす工程に。  
強いモデルは、考える工程に。

AIをたくさん使う時代から、  
AIをうまく使い分ける時代へ。

GitHub Copilot の価格変更を見て、そんなことを感じました。
