## はじめに

twitter [0xmamedai](https://x.com/0xmamedai/)です。最近200ドルプランがすぐレートリミットになって辛い日々を送っています。

autoresearchが67,000スター。Ralph Loopで開発を回す記事も出てきた。「LLMをループで回して一晩研究させてみた」「パフォーマンスチューニングを自動化してみた」——個別ツールのHow-toはかなり充実してきました。

ただ、ふと思ったんですよね。「そもそもAgent Loopってどういう種類があるの？」って。

autoresearchとRalph Loopって同じ仕組みなんでしょうか、それとも全然違うものなんでしょうか。他にどんなパターンがあるのか。自分のやりたいことにはどの型が合うのか。

意外とそういう俯瞰的な整理がなかったので、自分でやってみました。約220件の論文・OSSリポジトリ・フレームワークを調査して、Agent Loopを**7つの型**に分類した結果をまとめます。

ちなみにこの調査自体もAgent Loopで回しました。その話は最後に。

## Agent Loopってなに？

この記事でいうAgent Loopとは、**LLMが自律的に「判断→実行→観測」を繰り返す反復構造**のことです。

aider、autoresearch、Ralph Loop——名前もやっていることも全然違うように見えますが、根っこはみんなこの構造の上に乗っています。でも「ループ」と一口に言っても、中身はけっこう違う。その違いを整理するのがこの記事の目的です。

## 3つのグループと7つの型

220件ほど見ていくうちに、Agent Loopは大きく**3つのグループ**に分かれることが見えてきました。その中に計7つの型があります。

### 全体像

```
Agent Loop の 7つの型

I. 逐次実行型 ── 目の前を1個ずつ解く
   ├─ A: 線形ReAct     （aider, Claude Code, Ralph Loop）
   └─ B: Event Stream   （LangGraph, OpenHands）

II. 仮説検証型 ── 複数試して良い方を残す
   ├─ C: 木探索         （SWE-Search, Tree of Thoughts）
   └─ D: 制約付き実験   （autoresearch, POPPER）

III. 自己進化型 ── 仕組み自体が育つ
   ├─ E: 段階ゲート     （Agent Laboratory）
   ├─ F: Self-Evolving  （Live-SWE-agent, Voyager）
   └─ G: Co-Evolutionary（Absolute Zero Reasoner）
```

ポイントは3つあります。

1. **全部「ループ」だけど、やっていることが根本的に違う**
2. **組み合わせ（ハイブリッド）もある** — 実際のプロジェクトは複数の型を重ねていることが多いです
3. **目的で使い分ける** — 万能な型はありません

以下、各型を代表的なOSSと一緒に紹介します。

---

## グループ I: 逐次実行型

「目の前のタスクを、1ステップずつ順番に片づける」グループです。最もシンプル。

### A型: 線形ReAct

**思考→行動→観測の直列反復。**

一番ベーシックなAgent Loopです。LLMが「次に何をしようかな」と考え、ツールを叩き、返ってきた結果を見てまた考える。これを延々繰り返します。

**代表プロジェクト: [aider](https://github.com/paul-gauthier/aider)** (42K+ stars)

aiderの場合は、コード編集→lint/test→結果をLLMに返す、というサイクルを1つの会話セッション内で回しています。

```
aiderの基本ループ:
  ユーザーの指示を受け取る
  → LLMがコード編集を提案
  → 編集を適用
  → lint / test を実行
  → エラーがあれば結果をLLMに返して再修正
  → 成功するまで繰り返す
```

```
典型的なプロジェクト構成:
my-react-app/
├── src/           # LLMが編集する対象
├── tests/         # 自動テスト（observeの源泉）
├── .aider/        # aiderの設定・履歴
└── CONVENTIONS.md # LLMへの制約（コーディング規約等）
```

**向いている場面**: バグ修正、コード編集、調査タスク。普段使っているClaude CodeやCursor、Clineも基本はA型です。Ralph LoopはこのA型セッションをcronで繰り返し起動する仕組みですし、autoresearchの内側で動いているのもA型のReActです（外側は後述のD型）。

**限界**: 1本道なので探索の幅が狭いです。「他にもっといい方法があるかも？」を試せない。

### B型: Event Stream

**すべての行動がイベントとして記録される。再生・分岐・監査が構造的に保証される。**

A型と似ていますが、決定的に違うのは「何が起きたかを全部覚えている」ところです。途中で落ちても再開できる、過去の分岐に戻れる、あとから何が起きたか監査できる。プロダクション環境で長時間走らせるときに効いてきます。

**代表プロジェクト: [LangGraph](https://github.com/langchain-ai/langgraph)** (10K+ stars)

```
LangGraphの基本構造:
  StateGraph を定義（ノード + エッジ + 条件分岐）
  → Checkpointer が各ステップの状態を自動保存
  → 失敗時はチェックポイントから再開
  → 循環（ループ）がfirst-classでサポートされる
```

```
典型的なプロジェクト構成:
my-langgraph-agent/
├── graph.py       # StateGraph定義（ノード・エッジ・条件）
├── state.py       # TypedDict + Annotated による状態スキーマ
├── nodes/         # 各ノードの処理ロジック
├── checkpoints/   # 状態の永続化先
└── events/        # イベント履歴（再生・監査用）
```

**向いている場面**: 長時間稼働するエージェント、途中で人間の承認が必要なワークフロー、「落ちたら困る」プロダクション環境。

**限界**: 設計がけっこう重いです。「ちょっとスクリプト回したいだけなんだけど」という場面にはオーバースペック。

---

## グループ II: 仮説検証型

「複数の候補を試して、数値で良い方を残す」グループです。A型が1本道なのに対して、こちらは**幅を持って探索**します。

### C型: 木探索

**複数の仮説を並列に展開し、評価関数で枝を刈る。**

チェスAIが数手先を読むのと同じ発想です。LLMに複数の候補を出させて、それぞれを評価して、有望な方向を深掘りします。

**代表プロジェクト: SWE-Search**（SWE-agentチームによる研究）

SWE-Searchの面白いところは、gitのcommit treeをそのまま探索木として使うところです。各ブランチが1つの仮説で、MCTS（モンテカルロ木探索）で有望な方向を選んでいきます。

```
SWE-Searchの基本ループ:
  issue を受け取る
  → 複数の修正方針を候補として生成
  → 各候補をブランチに切って実装
  → value agent が各ブランチを評価
  → 有望なブランチをさらに深掘り
  → 最も評価の高いブランチを最終解とする
```

```
典型的なプロジェクト構成:
my-tree-search/
├── search.py          # MCTS / BFS / DFS の探索ロジック
├── evaluator.py       # 各ノード（候補）の評価関数
├── candidate_gen.py   # LLMによる候補生成
└── branches/          # 各候補の状態（git branch等）
```

**向いている場面**: 正解が1つとは限らない問題です。複数のアプローチを比較検討したいとき、数学の証明、複雑なバグ修正なんかに向いています。

**限界**: 計算コストがかなり大きいです。あと、評価関数が微妙だと探索自体が迷子になります。

### D型: 制約付き実験

**固定された評価指標に対して、変更→測定→keep/revertを繰り返す。**

autoresearchがまさにこれです。「何かを変える → スコアが上がったら採用、下がったら巻き戻す」。シンプルですけど、これがけっこうちゃんと回る。

**代表プロジェクト: [autoresearch](https://github.com/karpathy/autoresearch)** (42.9K stars)

```
autoresearchの基本ループ:
  program.md（規約）を読む
  → train.py を編集
  → 学習・評価を実行
  → スコアが改善 → git commit（keep）
  → スコアが悪化 → git reset --hard（revert）
  → 繰り返す
```

```
autoresearchのプロジェクト構成:
my-autoresearch/
├── program.md   # 人間が書く規約（LLMは読むだけ）
├── train.py     # LLMが変更する唯一のファイル
├── eval.py      # 固定された評価関数
└── .git/        # gitが世代管理を担う
```

注目してほしいのはその制約の強さです。編集できるファイルは1つだけ。評価指標は固定。gitのcommitが世代状態。たった630行。この「あえて狭くする」設計が、700回実験を回しても結果を比較できる理由になっています。

**向いている場面**: 明確な評価指標があるタスクです。モデル学習、パフォーマンスチューニング、ベンチマーク改善など。

**限界**: 改善が局所的になりがちです。ハイパーパラメータの微調整は得意ですが、「そもそもアプローチを変える」みたいな飛躍は起きにくい。

## グループ III: 自己進化型

ここからがちょっとSFっぽくなってきます。I・IIが「問題を解く」のに対して、IIIは「解く仕組みそのものを育てる」グループです。

### E型: 段階ゲート

**stage→gate→stageの連鎖。段階ごとに成果物と評価基準が変わる。**

研究のプロセスをイメージするとわかりやすいです。文献調査→仮説立案→実験→論文執筆——各段階で出てくるものはまったく別物ですし、「次に進んでいいか」の判断基準もそれぞれ違いますよね。

**代表プロジェクト: [Agent Laboratory](https://github.com/SamuelSchmidgall/AgentLaboratory)**

```
Agent Laboratoryの基本ループ:
  Literature Review（文献調査）
  → gate: 十分な先行研究を把握したか？
  → Experimentation（実験）
  → gate: 結果は有意か？mle-solverで独立評価
  → Report Writing（論文執筆）
  → gate: レビュアーの評価をパス？
```

```
典型的なプロジェクト構成:
my-staged-pipeline/
├── stages/
│   ├── 01_literature/   # 文献調査の成果物
│   ├── 02_experiment/   # 実験コード・結果
│   └── 03_report/       # 論文ドラフト
├── gates/
│   ├── feasibility.py   # 実現可能性チェック
│   └── quality.py       # 品質評価
└── pipeline.py          # 段階遷移の制御
```

**向いている場面**: 段階ごとに違う種類の成果物を積み上げるタスクです。研究自動化、データ分析パイプライン、複雑なドキュメント生成など。

**限界**: パイプラインの設計自体がけっこう大変です。段階の切り方をミスると、全体がうまく回りません。

### F型: Self-Evolving

**エージェントのartifact（コード、プロンプト、スキル等）を版管理して、外部検証で改善していく。**

D型と似ていますが、決定的な違いがあります。D型は「タスクの出力」を改善しますが、F型は「エージェント自体の構成要素」を改善します。自分自身のコードを書き換えて、より賢い自分になっていく。

**代表プロジェクト: [Live-SWE-agent](https://github.com/OpenAutoCoder/live-swe-agent)**

100行のPythonエージェント（使えるツールはbashだけ）からスタートして、タスクを解きながら**自分自身の実装を進化させていく**んです。結果としてSWE-bench Verified 79.2%という、オープンソース最高クラスのスコアを叩き出しています。

```
Live-SWE-agentの基本ループ:
  SWE-benchのissueを受け取る
  → 現在の自分（Python scaffold + bashツール）で解く
  → 解けた → 自分自身のコードを改善できないか検討
  → 自分のツールや関数を書き換え
  → ベンチマークで改善を確認
  → 次のissueへ（より強くなった自分で）
```

```
典型的なプロジェクト構成:
my-self-evolving-agent/
├── agent.sh           # エージェント本体（これ自体が進化する）
├── tools/             # 自作ツール（runtime中に追加される）
├── versions/          # 過去バージョンの保存
├── benchmark.py       # 外部ベンチマーク（採択判定に使用）
└── evolution_log.md   # 何をどう変えたかの記録
```

**向いている場面**: エージェントの能力そのものを継続的に上げたいとき。プロンプト最適化、ツール自動生成、ワークフロー改善など。

**限界**: meta-overfittingという落とし穴があります。「ベンチマークのスコアだけ上がって実用性は下がった」みたいなことが起きうる。あと、自分自身を書き換えるので安全面の設計も必要です。

### G型: Co-Evolutionary

**課題を作る側（proposer）と解く側（solver）が共進化する。**

F型との違いは、**練習問題すら自分で作る**ところです。人間がデータセットを用意する必要がない。

**代表プロジェクト: [Absolute Zero Reasoner](https://github.com/LeapLabTHU/Absolute-Zero-Reasoner)**

名前がかっこいいですね。その名の通り「ゼロ」——人間が作った学習データをまったく使わずに、自分で課題を作って自分で解いて、推論能力を伸ばしていきます。それでいてSOTA（最高性能）を達成しているのがすごいところです。

```
Absolute Zero Reasonerの基本ループ:
  solver の現在の実力を評価
  → proposer が「ギリギリ解けるレベル」の課題を生成
  → solver が課題を解く
  → code executor が正解を検証
  → 結果をもとに solver と proposer の両方を更新
  → より難しい課題 × より強い solver へ
```

```
典型的なプロジェクト構成:
my-coevolution/
├── proposer/         # 課題生成エージェント
│   └── generator.py  # frontier難度の課題を作る
├── solver/           # 課題解決エージェント
│   └── reasoner.py   # 推論・解答
├── verifier/         # 外部検証（code executor等）
├── curriculum/       # 生成された課題のアーカイブ
└── training_log.md   # 共進化の記録
```

**向いている場面**: 学習データが足りない、あるいは既存データでは天井にぶつかっている場合。推論能力の強化、新しいスキルの獲得など。

**限界**: 検証可能な環境が必須です。コード実行のように「正解かどうか機械的に判定できる」タスクでないと成立しません。あと、課題が単調になる「curriculum collapse」のリスクもあります。

## 組み合わせ（ハイブリッド）もある

ここまで7つの型を個別に見てきましたが、実際のプロジェクトは1つの型だけで動いているとは限りません。むしろ組み合わせの方が多いくらいです。

### A+D: ReAct + 制約付き実験

autoresearchの中を覗いてみると、1回の実験サイクルの中ではLLMが「考えて→コードを書いて→結果を見る」というA型のReActを回しています。D型のkeep/revertはその外側で管理している。

つまり**autoresearchは「外側D型 × 内側A型」の二重構造**なんですよね。

### F+A: Self-Evolving + ReAct

Live-SWE-agentは「目の前のタスクを解く」ときはA型で動いて、「自分自身を改善する」ときはF型に切り替わります。仕事しながら自分も鍛える、みたいな。

### F+G: Self-Evolving + Co-Evolutionary

MinecraftのAIエージェントVoyagerは、スキルライブラリ（F型のartifact）を育てつつ、自動カリキュラム（G型のproposer）で新しい課題を生成し続けます。遊びながら勝手にうまくなっていく感じです。

## この分類自体もAgent Loopで行った

最後にちょっと自己言及的な話を。

この記事の元になった約220件の調査、実は手作業じゃないんです。**4時間ごとにClaude Opusを自動起動するcronジョブ**を組んで、約3週間・108セッションかけて論文やOSSの探索・分類をやらせました。

taxonomy.mdというファイルがartifact（成果物）で、セッションのたびに知見が追加・再構成されていきます。これ、まさに**F型（Self-Evolving）のナレッジ蓄積モード**なんですよね。Agent Loopの分類をAgent Loopでやった、という入れ子構造です。

やってみて実感したのは、**ループの型を意識して設計するかどうかで安定性がぜんぜん違う**ということ。最初はなんとなくA型で回していたんですが、知見があちこちに散らばって収拾がつかなくなりました。途中でF型の考え方——artifactを版管理する、整合性チェックを入れる——に切り替えてから、108セッションを安定して積み上げられるようになった。

型を知っていると、ループがうまく回らなくなったときに「これはA型の限界だから、D型に切り替えてみよう」といった判断ができるようになります。それがこの分類の実用的な価値だと思っています。

## まとめ

* **Agent Loopは1種類じゃない。** 大きく3グループ・7つの型があります
* **組み合わせもある。** 実際のプロジェクトはハイブリッドが普通です
* **目的で使い分ける。** 万能な型はありません

autoresearchもRalph Loopも、7つの型のうちのひとつに過ぎません。個別ツールのHow-toを読む前に、こういう全体像を頭に入れておくと、自分のやりたいことに合ったループを選びやすくなるんじゃないかなと思います。
