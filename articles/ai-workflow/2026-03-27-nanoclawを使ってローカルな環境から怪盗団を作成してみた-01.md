## はじめに

「各キャラが独立した思考を持ち、本当に異なる視点から意見を言い、議論ログが残る」

そんなAIエージェントチームを、ペルソナ5の怪盗団をモチーフに作ってみました。

Discord上でモルガナ、ジョーカー、竜司、杏がそれぞれ別のアバターで発言し、タスクについて議論してから実装に入る。  
ファンとしては見ているだけでも楽しい仕上がりになっています。

## 技術スタック

| 技術 | 用途 |
| --- | --- |
| **NanoClaw** (OSS) | メッセージングチャネルとClaude Agent SDKを繋ぐオーケストレーター |
| **Claude Agent SDK** | エージェントのコア。コンテナ内で動作 |
| **Discord Bot + Webhook** | チャネル。Webhookでキャラごとにアバターを切り替え |
| **Docker** | エージェント実行環境の隔離 |
| **GitHub MCP** | コンテナ内からPR作成・レビュー等のGitHub操作 |
| **SQLite** | メッセージ、セッション、経験ログの永続化 |

NanoClawはClaude Agent SDKを使ったパーソナルアシスタントのOSSで、WhatsApp・Slack・Discord等のメッセージングチャネルからClaude Agentを呼び出せるようにするものです。  
エージェントはDockerコンテナ内で実行されるため、Bashアクセスも安全に許可できます。

## 怪盗団メンバー

| キャラ | 役割 | 視点 |
| --- | --- | --- |
| **モルガナ** | 司会進行・ナビゲーター | 全体の方向性、過去の経験との照合 |
| **ジョーカー** | アーキテクト・最終決定者 | 技術設計、リスク、スケーラビリティ |
| **坂本竜司** | 実装担当・コーダー | 実装の現実性、工数、シンプルさ |
| **高巻杏** | UI/UXデザイナー | ユーザー体験、使いやすさ、直感性 |

各キャラは `send_message` の `sender` パラメータで切り替わり、Discord Webhook経由で**それぞれ別のユーザーとして表示**されます。  
1つのWebhookでユーザー名とアバターを動的に変えているだけなので、Bot自体は1つです。

## 独立思考の仕組み

最初は1つのエージェントに「4人を演じ分けさせる」アプローチを取っていました。  
しかしこれだと同じ脳が考えているので、本当の意味で異なる視点は出てきません。  
賛成も反対もすべて1つの思考から生成されるため、議論が形骸化します。

そこで**各キャラを独立したAgent（サブエージェント）として逐次起動する方式**に変更しました。

```
discussion.md 作成（モルガナ）
  ↓
Agent(ジョーカー) → discussion.md 読む → 意見追記 → Discord投稿
  ↓
Agent(竜司) → discussion.md 読む → 意見追記 → Discord投稿
  ↓
Agent(杏) → discussion.md 読む → 意見追記 → Discord投稿
  ↓
Agent(ジョーカー) → discussion.md 読む → 最終判断 → Discord投稿
  ↓
実装フェーズへ
```

共有ファイル `discussion.md` がホワイトボード的な役割を果たし、各Agentは前の人の発言を読んだ上で自分の専門的視点から意見を述べます。  
議論ログはそのままファイルとして残るので、後から振り返ることもできます。

### 議論ルールのポイント

ここで重要なのは **「無理に反論させない」** ことです。

最初は「異なる視点からの反論を出す」ことを目標にしていましたが、実際にはそれだと各キャラが義務的に反対意見を捻り出すようになります。  
良い案に対しても無理に穴を見つけようとする。

最終的に落ち着いたルールはシンプルです：

> 賛成なら「いいと思う」でいい。反対なら理由を添えて反対。取り繕うな。ただし根拠を持て。

「自分の専門領域の観点から正直に思ったことを言え」これだけです。

## 作業フロー

タスク依頼を受けると、怪盗団は以下のフローで動きます。

```
ユーザー: 「この機能作っといて」
  ↓
モルガナ: タスク受付 → discussion.md に議題を設定
  ↓
【作戦会議フェーズ】
  ジョーカー → 竜司 → 杏 → ジョーカー(最終判断)
  ↓
【実装フェーズ】
  竜司: feature ブランチ → 実装 → テスト → PR作成
  ↓
【レビューフェーズ】
  ジョーカー: コードレビュー（FE変更なら杏もUIレビュー）
  ↓
モルガナ: 「マージしていいか？」（承認依頼）
  ↓
ユーザー承認 → マージ → 経験記録 → 完了報告
```

マージは必ずユーザーの承認を挟みます。  
「OK」と言えばマージ、「ここ直して」と言えば竜司が修正してジョーカーが再レビュー。  
完全にお任せではなく、最後のジャッジは人間が握る設計です。

## 見ているだけで楽しい

正直に言うと、このシステムの一番の価値は**見ていて楽しい**ことかもしれません。

Discordの通知が来て、モルガナが「ワガハイが分析してやるぜ」と言い、ジョーカーが冷静に方針を示し、竜司が「っしゃ！やってやるぜ！」と実装を始め、杏が「ちょっと待って、ユーザー的にはこっちのほうが自然じゃない？」とツッコむ。

キャラクターが独立したAgentとして動いているので、本当にそれぞれ違う切り口で意見を出してきます。竜司が「面倒だからこっちでよくね？」と言えば、杏が「でもユーザーの動線を考えると...」と返す。こういうやり取りが自然発生するのが面白い。

ペルソナ5ファンとしては、怪盗団がDiscord上でワイワイやっているのを眺めているだけでニヤニヤできます。

## 課題と懸念

### トークン消費量

これが最大の懸念です。1つのタスクに対して：

* モルガナ（本体）の思考
* ジョーカーのAgent起動 → discussion.md読み書き → Discord投稿
* 竜司のAgent起動 → 同上
* 杏のAgent起動 → 同上
* ジョーカーの最終判断Agent起動
* 竜司の実装Agent起動
* ジョーカーのレビューAgent起動

...と、**最低でも5-7回のAgent起動**が走ります。  
それぞれがClaude Agent SDKのセッションを消費するため、簡単なタスクでもトークン使用量はかなりのものになります。

現時点では**遊び場**として検証している段階で、本格的な開発ワークフローに組み込むにはコスト面の検討が必要です。  
議論フェーズをスキップできるオプション（単純なタスクなら竜司に直接実装させる等）を入れるのが現実的な落とし所かもしれません。

### NanoClawのOSS改修問題

NanoClawは便利なOSSですが、今回のカスタマイズではOSS側のコードをかなり直接的に修正しています：

* `src/channels/discord.ts` — Webhook送信、キャラクター定義、アバターキャッシュ
* `src/ipc.ts` — sender伝搬、二重送信防止のトラッキング
* `src/index.ts` — IPC送信済み判定によるストリーミング出力スキップ
* `src/types.ts` — インターフェース変更
* `container/agent-runner/` — MCP サーバー読み込み、許可ツール拡張

NanoClawのアップストリームがアップデートされた場合、これらの変更とのマージ作業がかなり大変になることが予想されます。スキルシステム（ブランチベースの機能追加）である程度は吸収できますが、コアファイルへの変更はコンフリクトが避けられません。

この辺はフォーク運用の宿命ではありますが、アップストリームの更新頻度が高いOSSだと追従コストが積み上がっていく点は認識しておく必要があります。

## まとめ

ペルソナ5の怪盗団をAIエージェントチームとして実装してみました。

* **独立Agent方式**で各キャラが本当に異なる思考を持つ
* **discussion.md**で議論ログが蓄積・振り返り可能
* **Discord Webhook**でキャラごとに別アバターで表示
* **見ているだけで楽しい**（これ重要）

トークン消費とOSS改修の追従コストは課題ですが、AIエージェントの「チーム」としての可能性を探る実験としては面白い取り組みになったと思います。

好きな作品のキャラクターをAIエージェントとして動かす、という切り口は単純にモチベーションが上がるので、個人プロジェクトとしてはおすすめです。  
怪盗団じゃなくても、好きな作品のチーム構成を当てはめてみると、意外と「役割分担」がうまくハマるかもしれません。
