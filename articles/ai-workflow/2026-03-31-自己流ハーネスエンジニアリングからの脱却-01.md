本記事では、AIエージェントの性能を決める「ハーネス」の設計手法について、ざっくり解説します。

株式会社ナレッジセンスは、生成AIやRAGを使ったプロダクトを、エンタープライズ企業向けに開発しているスタートアップです。

## この記事は何

この記事は、AIエージェントのハーネス設計を自然言語で統一的に記述する手法「**NLAH（Natural-Language Agent Harnesses）**」の論文について、日本語で簡単にまとめたものです。

<https://arxiv.org/abs/2603.25723>

## 本題

### ざっくりサマリー

「NLAH」は、エージェントのハーネス設計を、自然言語で統一フォーマットに書こう、という提案です。 清華大学・ハルビン工業大学の研究者らによって2026年3月に発表されました。

![](https://static.zenn.studio/user-upload/be0825e8b233-20260330.png)

最近、「**ハーネスエンジニアリング**」が話題です。

ハーネスとは、AIエージェントを制御する技法のことです。**AIエージェントを自分の意図通りに行動をさせるためのロジック全般**は、「ハーネス」です。

ただ残念ながら、**現状のハーネス設計は、開発者ごとにバラバラで、自己流です**。これだとハーネスの品質を図るのが難しかったり、別のプロジェクトで再現性がなかったりなど、問題があります。

そこで、今回紹介する「NLAH」は、**「ハーネスを自然言語で明示的に記述しよう」**、という提案をしています（結構新しいパラダイムな予感がします）。これにより、ハーネスの移植だったり比較が簡単になります。

### 問題意識

「自己流ハーネス」には、大きく2つの問題があります。

**問題点①：移植性がないこと。**  
現状のハーネスは、開発現場それぞれで、暗中模索で行われています。普通にコードベースの中に制御ロジックが埋もれている状態です。これでは、あるプロジェクトで上手くいったハーネスを、別のプロジェクトに持っていくことはできません。

**問題点②：比較・検証ができないこと。**  
「これはいいハーネスなのか？」というのを比較して検証するのが難しい状況です。統一フォーマットがあれば、どの処理が違うのか、ということが明示的でわかりやすくなります。（学術的には、ベンチマーク評価がグッとしやすくなります）

### 手法

NLAHは、「ハーネスの制御ロジックを、コードではなく自然言語で、統一されたフォーマットで書く」という提案が大きいポイントです。

![](https://static.zenn.studio/user-upload/f4085a65a8fd-20260330.png)

まず、NLAHが提示している「コア要素」は以下です。

**【NLAHのコア要素】**

1. **コントラクト（Contracts）**

   * 上の図の真ん中の緑エリアの例で言うと、「Contract: must output valid Python code」の部分と、「If tests pass → STOP」の部分。出力制約と停止ルール
2. **ロール（Roles）**

   * 例えば、各ステージに書かれた「Role: Planner」「Role: Solver」「Role: Debugger」
3. **ステージ構造（Stage Structure）**

   * 「1. PLAN → 2. EXECUTE → 3. VERIFY → 4. REPAIR (if failure)」という全体のワークフロー構造そのもの
4. **アダプターとスクリプト（Adapters & Scripts）**

   * 「Run tests on solution.py」の部分。テスト実行など決定論的な処理の部分
5. **状態セマンティクス（State Semantics）**

   * 「Generate the code file: solution.py」「Overwrite solution.py」の部分など。ステップ間で何がファイルとして永続化され、次のステージにどう渡されるかの記述
6. **失敗分類（Failure Taxonomy）**

   * 末尾の3行。「format\_error → ...retry once」の部分。失敗の種類ごとにリカバリ方法を明示

ここまでが、今回提案されている「自然言語で記述したハーネス」のフォーマットです。

**【IHRについて】**

* 上記までで書かれたハーネスを、IHRという仕組みで実行していく（上記の図の右側に当たる部分）
  + →LLMが上記のフォーマットを解釈して実行していくイメージ。このプロセス自体もハーネスに当たる
* 各ステップで、(i) ハーネスの記述、(ii) 現在の状態と環境、(iii) ランタイムチャーター（共有ポリシー）を読み、コントラクトと予算の範囲内で次のアクションを選択

### 成果

コーディングベンチマーク（SWE-bench Verified）とコンピュータ操作ベンチマーク（OSWorld）で、3つの観点から評価されています。

![](https://static.zenn.studio/user-upload/6da644f276dd-20260330.png)

* ハーネスの適用により、ツール呼び出し数やLLM呼び出し数が大幅に変化。ハーネスは単なる「プロンプト」ではなく、エージェントの行動を実際制御できていることを示す
* モジュール単位の追加・除去が簡単に。特に自己進化（Self-Evolution）モジュールが最も効果的で、SWE-benchで+4.8pt
* コードで書かれたハーネスをNLAHに移行した結果、OSWorldで30.4→47.2に性能が向上

## まとめ

弊社では普段、AIエージェントの開発を行っています。

**AIエージェントの実装において、ハーネスエンジニアリングは、非常に重要な要素**です。ただ、今後「ハーネスエンジニアリングの標準化」が重要になると考えています。

普段からエンタープライズ企業向けにAIエージェントを実装していると、「エージェントの制御ロジックがコードに埋もれていて、精度向上・評価が難しい」という課題が出てきます。なので、NLAHのように、ハーネスの設計パターンを明示的なフレームワークとして管理する手法は、とても有用です。

みなさまが業務でAIエージェントを開発する際も、参考にしていただければ幸いです。今後も、AIの活用精度を上げるような工夫や研究について、記事にしていこうと思います。我々が開発しているサービスは[こちら](https://chatsense.jp/)。
