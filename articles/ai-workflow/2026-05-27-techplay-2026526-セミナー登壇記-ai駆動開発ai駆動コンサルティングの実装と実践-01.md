## TL;DR

Techplayさんのセミナー(<https://techplay.jp/event/994601>)で「AI駆動開発・AI駆動コンサルティングの実装と実践」というテーマで登壇してきました。AIエージェント活用、AI駆動開発の優位性と、それを活用したデモ動画をこちらで公開します。

## デモに入る前に話したこと

**今 業界で何が起きているのか / AIをどう位置付けるべきか / デモの裏側で使っているテクニック** を先に共有しました。

### 1. AI時代の地殻変動 — コンサル・ITはどう変わるか

戦略・企画から保守運用までのプロセス全体で、**人員削減と期間圧縮が同時に進んでいる**話をしました。  
![](https://static.zenn.studio/user-upload/f7932c2877e5-20260527.png)

> **キーメッセージ**: エンジニアもコンサル的・業務的素養を求められる時代になった

* 上流（戦略・企画・要件定義）に人が寄っている
* 下流（実装・テスト・運用）はAI + 少人数で回るようになる
* **上流と下流の境界が溶けている** — 「エンジニアだから業務は知らなくていい」「コンサルだからコードは書かなくていい」は通用しない
* アクセンチュア社内でもこの動きは実感ベースで進行中

### 2. AI活用 vs AI駆動 — 補助から主役へ

次に、**AI"活用"とAI"駆動"は別物**という話をしました。今日のメインメッセージです。  
![](https://static.zenn.studio/user-upload/1a9261b6ca24-20260527.png)

|  | AI活用（人間主導・AI支援） | AI駆動（AI主導・人間承認） |
| --- | --- | --- |
| 主役 | 人間 | AI |
| AIの役割 | 補助ツール | 一次成果物の生成者 |
| 人間の役割 | 作業者 | レビュアー / 方向付け |
| プロセス | 既存プロセスにAIを"足す" | プロセス自体を**再設計** |

### 3. Officeファイル vs Markdown/HTML — AIフレンドリーな形式

もう一つ、地味だけど重要な話をしました。**AIと協働する速度は、ファイル形式で決まる**という話です。  
![](https://static.zenn.studio/user-upload/dc5d84226745-20260527.png)

今回のデモで「業務フロー図がリアルタイムに育っていく」のも、成果物がMermaid + Markdownだから出来ています。**もしこれをPowerPointでやろうとしていたら、絶対に成立しません**。

## 本題のデモ

> 「会議の **前・中・後** で、Claude をフル活用すると何が起きるか」を、1つのデモで完結させています

| フェーズ | 普通のやり方 | デモでやったこと |
| --- | --- | --- |
| 会議**前** | ヒアリング用メモをエクセルで自作（半日） | Claude に質問リストを15個生成してもらう（5分） |
| 会議**中** | メモを取る・あとで整理 | 発言をその場で投入 → **業務フローが目の前で成長** |
| 会議**後** | 議事録 → 整理 → 課題抽出 → 改善案、で1週間 | **会議終了時点で全部出来てる** |

## デモシナリオ

「コールセンター運営会社への初回ヒアリング」を5〜6分の音声で再生し、私はそれを聞きながら、ライブで Claude に発言を投げ込んでいく、という構成です。デモ動画はこちらをご参照ください。  
<https://www.youtube.com/watch?v=0zKsrgOHFbo>

### 設定

* B2Cコールセンター運営会社
* 業務3種: **料金 / 技術 / 苦情**
* **50席 / 24時間体制**
* クライアントには来月、社長への中間報告がある

### クライアント（AI音声）が話してくれた内容

* IVRで①料金 / ②技術 / ③苦情に分岐
* 料金は1次完結。技術は8割が2次エスカレ。苦情は必ず管理者。
* **夜勤帯は2次スタッフがいない** → 待たせる → クレーム増
* ベテランの1次完結率90%、新人は50%
* 新人が応対すると2次エスカレ必須
* 「夜勤 × 新人 × 技術問合せ」の組み合わせが最悪パターン

## 目の前で育っていった業務フロー

デモで見せたかったのは、**クライアント発言を聞いてclaudeに投げるたびに段階的に成長していくMermaidフロー図と議事メモ**でした。最終的にこうなりました。

### フローの成長プロセス（6段階）

| # | 投入したキーワード | フローへの反映 |
| --- | --- | --- |
| 1 | 業務3種: 料金/技術/苦情。50席24時間 | 基本3分岐が出現 |
| 2 | IVRでプッシュ番号で振り分け | IVRノード追加 |
| 3 | 料金=1次完結 / 技術=80%が2次エスカ / 苦情=必ず管理者 | エスカレパス追加 |
| 4 | 夜勤帯は2次不在 → 翌朝コールバック → クレーム増 | 夜間例外パス追加 |
| 5 | ベテラン1次完結率90% vs 新人50% → 夜勤と重なり悪循環 | 人員特性ノード追加 |
| 6 | 来月: 社長中間報告で見える化＋改善方向性 | ヘッダーに納期情報反映 |

**普通だったらホワイトボードと付箋で1〜2時間かかる作業**を、ヒアリングしながらリアルタイムで仕上げています。

## 会議終了時に出来ていた成果物

デモの最後に画面に映したのは、業務フロー図だけではありませんでした。  
**会議中に裏で同時生成されていた**、課題仮説と改善案ドラフトもセットで出ています。

### 構造的要因（5つに分解）

| # | 要因 | 内容 |
| --- | --- | --- |
| 1 | 2次の夜勤不在 | テクニカルサポートが夜勤帯にいない |
| 2 | 新人の1次完結不可 | 新人応対は2次エスカレ必須運用 |
| 3 | 新人比率の高さ | 席の約50%が新人 |
| 4 | スキル分布の偏り | ベテラン90% vs 新人50%の二極化 |
| 5 | 夜勤×新人シフトの重なり | 最悪パターンが運用上発生 |

### 改善案ドラフト

* 夜勤帯の2次サポート体制見直し（オンコール / リモート対応 / 翌朝コールバック運用整備）
* 1次解決率向上策（FAQ・ナレッジ整備、頻出技術問合せのスクリプト化）
* クレーム増加の要因分析（時間帯別 / 種別別の発生傾向の可視化）

## 何が伝えたかったのか

「AI でメモを取る」「AI で議事録を書く」ではなく、今回のデモで見せたかったのは、その先です。

* ヒアリング **しながら** 構造化される
* ヒアリング **しながら** 課題が仮説化される
* ヒアリング **しながら** 改善案ドラフトが出る
* **会議終了 = 中間成果物完成**

コンサルティングの「ヒアリング → 整理 → 仮説 → 提案ドラフト」というサイクルが、1つの会議の中で1周回り切る。これがAI駆動コンサルティングの肝だと思っています。

## 会場からの質問

セミナーの途中、参加者の方からこんな質問をいただきました。

> 「AIで代替可能/不可能な業務を分別するための、**システム横断での判断軸**をどう定義すればよいか迷っています。出力の質をもとに判断するにも、プロンプトが悪かったのかな、とか、  
> 同業務でも他のシステムでは違うのでは、横展開してもよいのでは、といった迷いがあります。  
> ACNさんではどのような基準を設けられているか、可能であればご教示ください」

私の回答はこうでした。

> プロンプトの良し悪しが成果物の質に影響することは最近なくなってきている印象です。  
> 要はテキトーなプロンプトでもそれなりにイケてるものは出来てしまう。  
> 一方で、適切なゴール設定や壁打ちが出来ないと、わけのわからないものが出来てしまいます。  
> その前提において、**システム横断での判断軸というものをACNで持っている認識はありません**。

すると、さらに踏み込んだ質問が来ました。

> 「個別業務毎においても、AIで代替可能/不可能な業務を分別するための、定量・定性的な判断基準は設けられていないでしょうか？  
> 例えば、エンタープライズにおいて、AIが書いたコードでこの業務はこれからは良い、と  
> **誰がどのような基準で判断**すれば良いのか、と思いまして」

これ、本質的な良い問いだと思いました。「組織として軸が無いなら、現場の意思決定者は何を拠り所にすればいいのか？」という、実務寄りの問いです。

その場でお答えした内容と、後から整理し直したものをまとめると、以下のようになります。

### ① 業務特性での切り分け（定性）

| 観点 | AI寄せやすい | 人間に残すべき |
| --- | --- | --- |
| 可逆性・影響範囲 | 戻せる / 影響限定 | 社外公開・基幹データ更新・金銭処理 |
| 判断の性質 | ルールベースで検証可能 | 価値判断・利害調整・例外処理 |
| 正解の有無 | 正解と照合可能（要約・分類・コード生成＋テスト） | 正解が無い（顧客との合意形成等） |

### ② AIが書いたコードを業務利用してよいかの判断

原則として「AIが書いたから特別扱い」ではなく、  
**従来のコードレビュー・テスト・本番リリース基準を満たすかで判断**、というのが現場の落としどころです。

* 一次成果物はAI、最終的なアカウンタビリティは人間（レビュアー / PM）
* レビュー観点も従来と同じ（セキュリティ・性能・保守性・テストカバレッジ）
* リスクの高い領域（認証・課金・個人情報）は、人間が書いたコードと同様に追加レビューを課す

### ③ 誰が判断するか

新しい役割を作るより、**既存のガバナンス（システムオーナー / 開発リード / セキュリティ）に「AI利用時のチェック観点」を一行追加**する形が動かしやすい、と考えています。  
「AI製コードを誰が承認するか」という新ポストを作ると、それ自体がボトルネック化しがちです。

---

要するに、**「AI専用の判断軸」を新設するより、「既存のリスク・品質基準にAI観点を載せる」アプローチが現実的**、というのが私の考えです。ACNとしての公式見解というより、現場でやってきた実感としての回答ですが、参考になれば幸いです。
