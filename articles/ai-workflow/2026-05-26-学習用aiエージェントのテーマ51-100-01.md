# ブロック詳細説明書（テーマ 51〜100）

> 各テーマのMermaid図に登場するブロックの役割・実装技術・入出力を説明します。  
> ⚠️ マークのあるテーマは実装上の問題点があり、設計を変更しています。

---

## 共通ブロック用語集（再掲）

| ブロック名 | 意味 |
| --- | --- |
| `LLM Agent` | OpenAI API (gpt-4o / gpt-4o-mini) または Azure OpenAI を呼び出す中核処理 |
| `Orchestrator` | 複数エージェントに仕事を割り振る司令塔。LangGraph `StateGraph` で実装 |
| `Supervisor` | Orchestratorと同義。どのサブエージェントを次に呼ぶか判断する |
| `Tool` | エージェントが呼び出せる外部機能。LangChain `@tool` デコレータで定義 |
| `Vector DB` | ChromaDB/FAISS（ローカル）または Azure AI Search（クラウド） |
| `Graph API` | Microsoft 365にアクセスするREST API（`msal` でOAuth2認証） |
| `Human-in-the-loop` | 処理途中で人間の確認・承認を挟む仕組み |
| `Mock API` | 本番APIの代替として開発中に使うFastAPI製のスタブサーバー |

### Theme 52: Autonomous Financial Planning Agent

**設計変更**: ⚠️ **SAP / Oracle ERP API → Mock ERP（SQLite + FastAPI）に変更**

#### 変更理由

`SAP API`・`Oracle ERP API` はエンタープライズ契約（数百万円/年）が必要であり、ポートフォリオ開発環境では利用不可。Adapter Pattern を用いて開発時はSQLiteモック、本番ではSAP/Oracleに切り替えられる設計に変更。

#### 変更後のMermaidダイアグラム

| ブロック | 実装技術 | 役割 | 入力 | 出力 |
| --- | --- | --- | --- | --- |
| `CFO / 経理担当` | Streamlit UI | FP&A処理の開始トリガーと承認を行う | テキスト指示 | 処理リクエスト |
| `FP&A Orchestrator` | LangGraph `StateGraph` | 予算→実績→分析→提案の順序を管理 | 処理リクエスト | 各エージェントへの指示 |
| `予算策定 Agent` | LangChain + gpt-4o + `pandas` | 前年実績と目標から予算案を策定 | 過去実績 + 目標値 | 予算DataFrame |
| `実績収集 Agent` | LangChain + ERP Adapter | Mock/実ERPから実績データを取得・整形 | 対象期間 | 実績DataFrame |
| `差異分析 Agent` | LangChain + gpt-4o + `pandas` | 予算対実績の差異を計算し要因を特定 | 予算 + 実績 | 差異レポート |
| `改善提案 Agent` | LangChain + gpt-4o | 差異要因から実行可能な改善アクションを提案 | 差異レポート | アクションプラン |
| `ERP Adapter` | Python Abstract Class + FastAPI | ERPアクセスを抽象化。環境変数でDev/Prodを切替 | データリクエスト | 財務データ |
| `Mock ERP DB` | SQLite + サンプルCSV（経費・売上データ） | 開発環境のERPデータソース | SQLクエリ | 財務テーブルデータ |

**実装ポイント**:

* `ERPAdapter` 抽象クラスを定義し、`MockERPAdapter(SQLite)` と `SAPERPAdapter` を実装
* `ENVIRONMENT=dev` の環境変数でMockを自動選択
* サンプルデータは `pandas` で生成した3年分の月次P&Lデータを使用

**設計変更**: ⚠️ **Twitter/Instagram/LinkedIn APIの制約により代替データソースに変更**

#### 変更理由

* **Twitter/X API v2**: 無料プランは読み取り専用で月1,500ツイート制限（2023年以降大幅制限）。`snscrape` はTwitterの利用規約上グレーゾーンで2023年以降ブロック多発
* **Instagram Graph API**: ビジネスアカウント必須＋Meta Business認証が必要。個人アカウントのデータ取得は不可
* **LinkedIn API**: 採用・広告用途以外のデータ取得は認証困難（企業パートナーシップ必要）

#### 変更後の設計（代替データソース）

| ブロック | 実装技術 | 役割 | 入力 | 出力 |
| --- | --- | --- | --- | --- |
| `スケジューラー` | `APScheduler` | 日次でデータ収集を起動 | 時刻トリガー | 処理開始 |
| `Orchestrator` | LangGraph `StateGraph` | 3収集エージェントへの並列指示 | 監視キーワード | 各エージェントへの指示 |
| `Reddit収集 Agent` | `PRAW`（Reddit API, 無料） | ブランド名・製品名のメンションをサブレディットから収集 | キーワード | 投稿テキスト + スコア |
| `ニュース収集 Agent` | `NewsAPI`（無料1000req/日）+ `feedparser` | ブランド関連ニュースを収集 | キーワード | 記事一覧 |
| `レビューサイト収集 Agent` | `BeautifulSoup` + `Playwright` | G2/Trustpilotの公開レビューを収集 | 製品名 | レビューテキスト + 評価 |
| `感情分析 Agent` | LangChain + gpt-4o-mini | 収集テキストの感情・トピックを分析 | テキスト一覧 | 感情ラベル + スコア |

| `コスト管理 DB` | SQLite（`agent_name`, `tokens_used`, `cost`, `timestamp`） | OpenAI API利用コストをエージェント別に追跡 | 利用ログ | コスト集計 |

**実装ポイント**:

* Semantic Kernel の `KernelPlugin` でエージェントをプラグイン化して疎結合に管理
* `prometheus-client` でメトリクスを収集、Grafanaで可視化（本番）/ Streamlitで可視化（MVP）
* エージェント追加はAgent RegistryにDB登録するだけで動的に対応可能な設計
