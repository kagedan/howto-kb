こんにちは。母の次ぐらいにAIを信頼しているサンウンです。

## はじめに

最近、AIレビューからオートマージまでつなげたリリースパイプラインで生産性を上げた、という記事をよく見かけます。（本記事では単に「AIレビュー」と呼びます。）自分のチームや会社でもこの流れが強く、導入の準備を進めているところです。

この記事は、チームにAIレビューを入れる前に、まず**自分自身に対して1次レビューを自動で走らせるAIパイプラインを組んでみた記録**です。1週間ほど運用してみて感じたこと・学んだことを共有します。同じような試みをしている方の参考になれば幸いです。

先に結論から言うと、2つです。

ひとつ目は、**AIレビューの本当の効果は、リリース速度ではなくレビューの質の向上**だったということ。始めるときはオートマージへの中間ステップとして考えていたのですが、回してみるとオートマージを目標にしない方がいい、という結論に近づきました。

ふたつ目は、**AIに1次レビューを任せること自体よりも、その結果を人間がどう扱うかを決めておくこと**が導入の肝だったこと。ここが決まっていないと、トークンだけ消費して生産性の邪魔になる道具になります。

以下、何を作り、どう動かし、何に気づいたかを順に書いていきます。

## 作ったもの

こんなパイプラインを組みました。

<https://github.com/Sangun-Kang/pre-auto-review>

Claude Codeのヘッドレスモードを使って、ローカルで1次レビューを走らせるツールです。5分おきにlaunchdが`poll.sh`を起動し、指定のGitHubリポジトリから「レビュー必要」ラベルが付いたPRを取得して、ヘッドレスのClaudeに投げて1次レビューをマークダウンで書かせます。結果はローカルダッシュボードとSlack DMに同時に出ます。

状態管理はシンプルな`state.json`ひとつです。

```
{
  "reviewed": ["org/repo#1234@a1b2c3d", "org/repo#1235@e4f5g6h"],
  "completed": ["org/repo#1234@a1b2c3d"]
}
```

`reviewed`は「AIが1次レビューを終えた」、`completed`は「人間が確認を終えた」という意味です。次のポーリングで`reviewed`にあるPRはスキップされるので、同じPRに対する重複レビューは起きません。

キーにPR番号だけでなく**head SHA**まで入れているのが細かいポイント。同じPRでも作成者が新しいコミットをpushすればSHAが変わり、自動で新しい1次レビューが作られます。これは「再レビューは任せない」という方針と矛盾するのではなく、同じ論理です ― 新しいコミットがpushされた時点は人間の会話が本格的に始まる前なので、その時点の検証はまだ1次レビューの領域だから。

セットアップ方法とlaunchdのplist例はリポジトリのREADMEを参照してください。

## なぜヘッドレスモードなのか、なぜローカルなのか

CIやクラウドジョブの上でAIレビューを組んだ事例は多くあります。自分は意図的に**ローカルでClaude Codeのヘッドレスモード + launchd**の組み合わせを選びました。理由は3つ。

### 1. ローカルgit + ファイル探索の自由度

PR diffが大きかったり、関心事が分離していたりするケースは確かに存在します。これ自体が必ずしも悪いわけではないのですが、AIレビューの観点からは確実に毒になります。**diffの外側にある周辺コードが暗黙知になってしまうから**です。

きちんとレビューするには、同じhelperを呼んでいる別の呼び出し元、過去のコミットメッセージ、近くのテストファイル、似たパターンの既存コードを合わせて見る必要があります。これをAPIでファイル1つずつ取ってくるボット比で、ローカルにあるClaude Codeは`ripgrep`一発 + `git log`で桁違いに速く、探索的です。活発なチームほど「これ、もともと何でこうなってるんだっけ？」を確認しないとまともなレビューにならないのですが、これはローカルチェックアウトなしではほぼ不可能です。

### 2. パーソナライズされたClaude Code設定 = 自分のローカル文脈がそのままレビュー観点になる

現代のAIの最大の武器のひとつは**パーソナライズ**です。これはチーム開発にAIを適用するときも同じ。

user-levelの`CLAUDE.md`（言語、トーン）、projectの`CLAUDE.md`（チーム規約）、custom skills、hooks、保存されたmemoryが、ヘッドレスセッションにすべて引き継がれます。APIボットは毎回プロンプトに全部詰め込まないといけないものを、ローカルのClaude Codeはファイルシステムからタダで手に入れる。特にprojectの`CLAUDE.md`がきちんと整備されているリポジトリで効果が大きいです。

### 3. ローカルデータとMCPツールという潜在力

2番の延長で、ローカルに設定したMCPや`~/src`配下のメモのようなリソースを自然に引っ張ってこられます。正直に言うと今のプロンプトはMCPの参照を積極的に指示しているわけではありません。ただ**必要になったときに簡単に繋げられること**自体が、ローカルヘッドレス構成の強みです。CIパイプラインに同じものを載せようとした瞬間、設定の難易度があらゆる方向で跳ね上がります。

今は使っていないけれど今後足せる余地が開いている、という次元として置いています。

### 安全性はどこで回収したか

ヘッドレスcron自動化の厄介なところは権限です。`poll.sh`は`--dangerously-skip-permissions`を使います。PRごとに権限ダイアログを人間が押すわけにはいかないので。

その代わり、安全性は**プロンプト層で回収**しました。`review-prompt.md`に次の制約を明示的に入れています。

* `NEVER` GitHubにコメントやレビューを残さない（`gh pr comment`、`gh pr review` 禁止）
* `NEVER` PRブランチをチェックアウトしない（現在の作業ブランチを保護）
* `NEVER` リポジトリ内のいかなるファイルも修正しない（出力ファイルだけ例外）

つまり、権限レイヤーを開ける代わりに、**出力の副作用をプロンプトで縛った**形です。AIができるのはコードを読むことと、指定されたマークダウンファイルにレビューを書くことだけ。この設計のおかげで「もしかして自動でPRに何か投稿されるんじゃ？」という心配をしなくて済みます。

## 1サイクルがどう回るか

レビュー結果は2チャンネルに出ます。まずSlack DMに短い要約が届きます。

```
PR #1234のレビューを完了しました。

要約:
- PR: feat(user): ユーザー検索エンドポイントを実装
- 判定: APPROVE
- レビューファイル: /Users/your_user/.claude/auto-review/reviews/review-pr1234.md

主な評価:
- 既存のFindサービスメソッドを活用した簡潔な実装
- Contextの伝播とエラーハンドリングがプロジェクトルールを遵守
- 不要なHandlerの削除でデッドコードを整理
- シナリオテストが成功ケースと404エラーケースをカバー
```

詳細レビューは次のテンプレートでローカルのマークダウンに保存されます。

```
# Review: feat(user): ユーザー検索エンドポイントを実装

- PR: https://github.com/example/backend/pull/1234
- Reviewed at: 2026-04-17T09:00:00+09:00

## Summary
クエリパラメータによるユーザー検索機能を追加する作業。
判定: APPROVE

## Blocking issues
大文字小文字の扱いが既存の検索APIと異なっているが、仕様とも取れる。
作成者に確認してみるとよい。

## Suggestions
1. ページネーションのデフォルト値をconstantに切り出す (優先度: 低)
2. エラーメッセージのi18nキーを既存パターンに揃える (優先度: 低)

## Questions
なぜソート順のデフォルトを `created_at desc` ではなく `id asc` にしたのか
確認してみるとよい。
次の作業である一覧画面のUIが、このAPIのレスポンス形状に依存するため...

## Good points
1. 既存のFindServiceを素直に再利用していて、重複実装を避けている
2. 404のハンドリングがプロジェクト既存のエラーラッパーに合わせてある

## 判定
APPROVE
```

APPROVEのときも「AIが理解しきれていない部分は作成者に質問してみるのもよい」といったコメントを残します。**人間がPRを素早く理解できるよう、ブリーフィングする役割**です。

ダッシュボードは127.0.0.1:8477で動きます。アクティブセクションのカードを見ながら実際のPRレビューを終えたら「完了」ボタンを押して、同じPRが再処理されないようにします。再レビューが必要なら「再レビュー」ボタンでアクティブに戻せます。

## 1週間回してみた結果

1週間で処理したPRは12件。サンプルとしてはとても小さいですが、コメントレベルで見ると数字がもう少し立ってきます。  
判定の内訳はAPPROVE 10件 (83%)、Blocking含み2件 (17%)、12件すべて最終的にマージされました。

* **AIが生成したレビュー項目: 55件**（Blocking / Suggestions / Questionsの合算。褒めは除外）
* **そのうち実際にPRに投稿したもの: 13件 (24%)**
* **残りの42件 (76%) は意図的に捨てた**
* **投稿した13件のうち、マージ前に実際に反映されたもの: 10件 (77%)**

つまり、**AIは過検出器、自分はキュレーター**という構造で回っていて、この構造はそもそも意図したものでした。この比率は単に「AIの性能が足りない」という話ではない、というのがこの記事の核になる観察です。

もうひとつの観察は、AIが掴んだ指摘が実際に自分が投稿したレビューの中でかなり大きな比重を占めていた、ということ。1週間で自分が残したactionableなレビュー投稿19件のうち**14件 (74%) がAI draftに基づいて**いて、残り5件はAIが取りこぼしたものを自分が独立に見つけて追加したものでした。

### どんな種類の指摘を拾ったか

反映まで至った10件がどんな性格の指摘だったかを抽象化するとだいたいこんなものです。

* **トランザクション境界まわりの問題** ― 隣接コードで使われているパターンと比べて、現在の実装がズレていることを指摘
* **必須値バリデーションの欠落** ― 外部仕様書でREQUIREDと定義されたフィールドのバリデーションが抜けていることを指摘
* **テストカバレッジの具体的なギャップ** ― このエラーケース、このedge caseのテストが必要だという具体的な提案
* **エラーハンドリングの一貫性違反** ― 同じ種類のエラーが違う層で違う扱われ方をしているパターン差を指摘

これらの共通点は、**「コード + 隣接パターン + 外部仕様」を同時に見ないと出てこない種類**だということ。静的解析では拾えず、人間が見ると結構時間がかかる領域です。先ほど触れた「ローカルgit + ファイル探索の自由度」がこの領域で直接効いてきます ― マネージドサービスやAPIボットでは、同じ深さの探索は難しかったはず。

### 逆にAIが構造的に取りこぼしたもの

一方で、AIが完全に構造的に取りこぼす領域もはっきり見えました。1週間の集計対象12件の中で、**AI draftには載らなかったがPRに投稿することになった指摘が5件**あります。どれも自分が独立に見つけたもので、内容的にはこんな性格のものでした。

* 同じクエリが別のサービスレイヤーでも発生していて、重複がcall graphを見ないと気づけないケース
* バリデーションの配置レイヤーがアーキテクチャ上の層と噛み合っていないケース
* 認証フローの前提が、この画面では保証されていないケース（ドメイン境界の飛び越し）
* リリース順序の依存 ― このPRをマージする前に、別リポジトリの互換性対応が先に必要だったケース
* 既存の運用スクリプトとのインターフェース契約が暗黙に変わるケース

これらの共通点は、**コードそのものより、コードの外縁を見る判断**だということ。AIはPR diffとその周辺の静的な情報は読めますが、他リポジトリ、運用スクリプト、過去のアーキテクチャ決定、デプロイ順序といった**外側の文脈**は原理的に射程の外にあります。この死角については、あとの「AIの最大の死角はコードじゃない」でもう一段深掘りします。

### 意外だった点: draftの大半は捨てられた

興味深いのは、**AI draftの76%がPRまで辿り着かなかった**こと。そのままPRに上げていたら、相手はそれなりの量のコメントに困惑していたはずで、もしAIが直接コメントを残す構成だったら、そのまま投稿されていたはず。

捨てた42件を事後的に分類するとこんな内訳。

| カテゴリ | 件数 | 比率 |
| --- | --- | --- |
| スタイル/好み（間違いではないが優先度が低い） | 18 | 43% |
| ドメイン知識（仕様/RFCの解釈問題） | 11 | 26% |
| 社内規約/暗黙知 | 7 | 17% |
| AIの判断ミス | 2 | 5% |
| 重複/ノイズ | 2 | 5% |
| その他 | 2 | 5% |

正直、この分布は予想と違いました。回す前は「AIが拾えないのは会社の文脈/暗黙知のせい」だと思っていたのですが、実測してみると**優先度判断のミス（スタイル/好み 43%）が一番大きいカテゴリ**で、暗黙知系（規約 + ドメイン）を合わせてやっと並ぶくらい。この違いは解決策が別物で、「導入するときに気をつけること」で詳しく扱います。

#### ここが一番の観察: "ハルシネーション"は5%しかなかった

そしてこの表で**最も押さえておきたいのは、AIの判断ミス ― いわゆる「ハルシネーション」 ― がわずか5% (55件中2件) に留まった**ということです。

世間のAIレビューへの印象は「的外れなことを言いがち」「コードを誤解する」に強く寄っています。自分も回す前は、捨てる理由の大半はこれだと予想していました。結果は真逆でした。

少なくともこの1週間のデータでは、**AIは間違えて使えないのではなく、優先度を選別できなくて使えない**に近い。この差は導入戦略をまるごと変えます。

* **ハルシネーション問題だと仮定すると** ― モデルサイズを上げる、プロンプトでhallucination抑制を強める、事実確認レイヤーを追加する、といった方向になる。根本解決は難しく、リスクは構造的に残り続ける。
* **優先度判断問題だと仮定すると** ― プロンプトに「Blocking / Medium以上だけ上げる」といったsuppressionルールを足すだけでかなり減る。文脈層（`CLAUDE.md`）の整備と組み合わせれば、さらに下がる。

つまり**AIレビュー導入のハードルは、多くの人が想定しているより低い可能性がある**。ハルシネーションは怖い問題ですが、優先度判断は扱いやすい問題です。同じ「76%が捨てられる」でも、その中身次第で打ち手はまるで違う、という点は強調しておきたい。

APPROVE判定のケースでも、AIが残した改善ポイントを見て実際のレビューに反映して修正依頼をしたケースがありました。**承認は無条件の承認ではない**という感触を得ました。

### どれくらい速く回ったか

タイムスタンプ上で観察されたパターンもいくつかありました。タイムスタンプは厳密な作業時間ではなく「確認タイミング」に近いですが、いくつかは一貫していました。

**AI完了 → 人間の最初のレビューコメントまでの間隔が非常に短かった**です。人間がAI完了以降にレビューを始めたケースの中央値は**20〜48分**、最短で4分。AIがSlack DMで完了を知らせる設計が**レビューのトリガー役**を果たしているということです。レビューを「始める」までの心理的なハードルが、AIの成果物というtriggerのおかげで下がる体感はありました。

**低複雑度のPRはAI完了 → 承認まで1〜9分以内**でした。事実上、draftをちゃんと読まずに承認している状態で、この区間は先に挙げた「draft全量廃棄6件」とちょうど重なります。低複雑度PRにAIを回すこと自体がコストの無駄、という後ろの主張はこの数字から来ています。

全体のレビュー作成時間は計測不能ですが、体感ははっきりしています。真っ白な画面から14件のactionableコメントを直接書くなら、1コメントあたり5〜10分、合わせて1〜2時間。AI draftから選んで書き直すと30〜60分程度。ざっくり**レビュー作成時間が50%以上短縮**された体感。これはAI OFFの対照群なしには厳密に検証できません。後ろの「この数字自体の限界」とも整合する話です。

## やってみて ― オートマージへの情熱が冷めた

最初の目的は「AIレビュー + オートマージ = リリース爆発、生産性向上、全員ハッピー」というストーリーを描くパイプラインの地ならしでした。ところが、AI単独のレビュー結果を自分の手で触ってみると、この道具は**リリースの速度ではなく、レビューの質を上げる方向にずっと効く**と気づきました。

* AIがAPPROVEをくれたけれどレポートを見てもう一度改善したケース
* AIがブロッキングをくれたが実はオーバーエンジニアリングで捨てたケース
* AIが拾えなかったチーム規約レベルの指摘を、人間が追加で掴んだケース

このトリアージに人間が直接介入して判断する過程が生まれたことで、むしろレビューが精緻になりました。なので、**オートマージを主目的ではなく補助目的に、速度と質の向上を主目的に**置く方がよく見えました。オートマージは結果として出るかもしれないし出ないかもしれない選択肢にすぎず、**stagingの北極星にしてはいけない**。北極星にした瞬間、「人間のレビューを減らす方向」にすべての意思決定が寄っていって、レビューが持つメンタリングとオーナーシップ伝播の機能が犠牲になり始めます。逆に「チームのAI-readabilityとレビュー品質を一緒に引き上げる」を北極星に据えれば、自動化は結果としてついてくるもついてこないも、どちらでもチームは得をします。

## 導入するときに気をつけること

ここまで読むと「じゃあどう導入すればいい？」という問いが自然に出ます。1週間回して見えたこと中心にまとめます。

### AIレビューの精度ではなく、キュレーション構造を設計する

AIの出力が直接PRに上がる道具なら**精度が肝**です。ノイズがそのまま作成者に届くから。市場のAIレビューツールがfalse positive率をセリングポイントにしているのは、この前提があるから。

自分の構成は出力がPRに上がりません。上げるのは自分がキュレーションした結果だけを、自分の名前で。そうするとAIに保守的に振る舞わせる理由がなくなります。むしろ\*\*「疑わしければまず上げろ」が合理的\*\*になる。精度を犠牲にして再現率を取るトレードです。

これが先ほどの「76%が意図的に捨てられる」の本当の理由です。AIは過剰招集、人間はキュレーション。**ノイズを作成者に送らず、レビュアーが吸収する構造**なら過検出が資産になります。逆にノイズをそのままPRに上げる構造だと、同じプロンプトが災害になります。

導入前に決めるべき一番目は、**AIの出力が誰に、どんな加工を経て届くか**。ここが決まれば精度/再現率のトレードオフも自動で決まります。

### 捨てられる理由はひとつじゃない ― 優先度判断と暗黙知は別の問題

前述の「捨てた42件」の分解表でもうひとつ大事な事実があります。最大のカテゴリが**スタイル/好み (43%)**、次が**ドメイン知識 (26%)**、**社内規約 (17%)** の順で、AIの判断ミスは5%に留まっていたこと。

これが意味するのは、AIが「間違えて」捨てられるのではなく、\*\*「間違ってはいないけれど、この場で上げるものじゃなくて」\*\*捨てられているということ。そしてこの「その場に合わない理由」もまた2種類に分かれます。

**ひとつは優先度判断のミス（スタイル/好み 43%）**。「マジックナンバーを定数に切り出そう」「変数名はもう少し説明的だとよい」みたいなもの。間違いではないけれど、このPRで上げるものではない。これはプロンプト層で解ける問題です ― 「優先度の低いスタイル指摘はしない」「Blockingまたは中優先度以上だけ上げる」といったsuppressionルールを足せば減ります。

**もうひとつは暗黙知（規約 17% + ドメイン 26% = 43%）**。こちらはsuppressionでは解けない。AIに「間違ったことを言うな」と命じても、本人は間違っていることを知らないから。これは文脈の注入で解く領域 ― チーム規約、関連RFCの解釈、アーキテクチャ決定の歴史を`CLAUDE.md`やreferenceドキュメントに入れてやれば減ります。

2つのカテゴリを「プロンプトチューニングで解こう」とまとめて扱うと、それぞれの最適戦略が混ざって両方とも解けにくくなります。混ぜずに切り分けて、それぞれのやり方で当たる方がマシです。

### 暗黙知の構成要素 ― 時間軸と空間軸

暗黙知系43%の中身をもう少し覗くと、その内側にも2つの軸が見えます。

ひとつは**時間軸** ― チームが選んだアーキテクチャと、意図的に残した負債の歴史。AIはリポジトリのスナップショットを見ているだけで、意思決定の歴史は見えない。あるパターンの違いが意識的な差なのか放置されたドリフトなのかは、会議室にいた人だけが知っています。

もうひとつは**空間軸** ― diffの外への染み出し。AIが「ブロッキング」と刺したものが、実際には「このPRで直すのは過剰で、別でリファクタするのが正しい」ケースが多い。

2軸を合わせると、\*\*「この指摘が、ここで、今、このPRという単位で解決されるべき問題なのか」\*\*になります。AIは「問題かどうか」は判定しても、「ここで、今、この単位で」は判定できない。

### AIの最大の死角はコードじゃない

「逆にAIが構造的に取りこぼしたもの」で挙げた5件の性格を一歩引いて見ると、AIレビューの死角は大きく2層あることがわかります。

1. **コードの外のコード生態系** ― call graph、アーキテクチャ階層、ドメイン境界、他リポジトリとの契約、デプロイ順序
2. **コードの先のプロダクト責任** ― 機能がリリースされたときにユーザーの信頼モデル、運用責任、lifecycleに与える2次効果

1はツールを増やせば減る可能性があります。MCPで他リポジトリを覗かせたり、call graphを抽出するツールを渡したりすれば、AIの射程はある程度広げられる。

2はツールの問題ではない。**組織がそのPRをリリースしたときに背負う責任の範囲**を、AIは原理的に知らない。「この機能が入ったとき、過去のユーザーの期待をどう裏切るか」「運用オンコールが誰に集中するか」「ロールバック時の影響範囲はどこまで波及するか」― こういう問いは組織の文脈そのもので、ドキュメントに書き出した時点で既に遅い(その判断が必要な瞬間の、最新の組織状態は誰も書き下ろしていない)。キュレーション層を抜けない理由はここにあります。

### 低複雑度PRにはAIを回さない方がいい

数字から見えたもうひとつの事実があります。対象PR 12件のうち**6件はAI draftが全量廃棄**されました（投稿0）。そしてこの6件は**例外なくすべて低複雑度PR** ― trivialなchore、数行のドキュメント修正、依存関係のパッチバージョン更新 ― でした。この6件は前述の「AI完了 → 承認1〜9分」区間とほぼ重なる。事実上、AI draftは読まれずに通り過ぎています。

このゾーンはAIを回すこと自体がトークンの無駄。レビューキュー全体に一律AIをかけるより、**複雑度ゲート**を置いて変更行数/ファイル数/キーワードベースでフィルタし、複雑なPRだけに回す方がコスト回収率が良い。ダッシュボードを見ながら「これ回す必要なかったな」というPRが溜まってきたら、それがゲートルールになるべきシグナルです。

### チームサイズが効いてくる

先にことわっておくと、**このセットアップを試したのは小規模チーム**です。プロセスを重く整備しなくても、Slackやミーティングでリアルタイムに会話が閉じる規模。この記事で書いた観察の一部は、この規模に強く依存しています。

小規模チームでは、**暗黙知はSlack/ミーティングでほぼリアルタイムに同期されている**。別途ドキュメント化しなくても「AIが見えない領域」をメンバー間で共有している構造。AIが半分くらい外しても、外した部分をキュレーションするのに必要な知識をメンバーがすでに持っている。

チームが大きくなると、同じ暗黙知が人数分に分散します。知っている人と知らない人の境界が出てきて、同期には会議やドキュメント化のような明示的な装置が必要になる。つまり**AIが埋められない暗黙知の空白が、大きいチームでは人間同士でも均等には埋まらない**。同じ道具を大きいチームにそのまま持っていくと回らない、というのはこういう理由です。

大きいチームに持っていこうとすると、少なくとも3つの課題が追加で出てくると予想しています。

1. **暗黙知の外在化コストが人数に比例して膨らむ**。`CLAUDE.md`やreferenceドキュメントに何をどこまで書くか、誰がメンテするかが常時の負担になる。小規模チームでは（せめて自分のチームでは）即時コミュニケーションをすることで費用が少ないと思いました。
2. **キュレーション層が分散する**。自分のPRのAI draftを自分で吸収するのか、指名レビュアーが代わりに吸収するのか、それともチーム共通のAIボット1体に集約するのか ― 選択が意味を変える。自分の構成は暗黙に「自分のPRのdraftは自分で吸収」を採用していますが、他の選択肢にも合理性があり、大きいチームではどれを選ぶかが設計決定になる。
3. **指標の設計が必要になる**。小規模チームは"体感"で十分でしたが、大きいチームではAI投入がレビュー品質に実際に効いているのか、単にノイズで摩耗しているだけなのかは、明示的に測らないと判断できなくなります。

ただし**方向性自体は規模に関係なく有効**だと思っています。AIはコミュニケーション以前の段階に留まり、人間はコミュニケーション段階を担う、という役割分担自体はチーム規模の関数ではない。チームが大きくなったときに追加で解くのは「各自のAI出力をチームでどう扱うか」という調整問題であって、「AIがコミュニケーション参加者になるべきか」という問いではない、と思っています。

### この数字自体の限界

最後に押さえておきたいのは、この記事で出した数字は**集計者もAIだった**こと。マッチング/分類の判定にAIが挟まる以上、自己正当化の方向へのバイアスはあり得ます。

実際、集計結果で「人間がAIなしで独立に発見した」と分類された1件は、自分がレビューコメントを曖昧に書いたせいで、集計AIがAI draftとの関連を拾えなかったケースでした。具体的には**2つの別関数で重複クエリが発生しているという指摘**だったのですが、AI draftですでに同じ観察があったにもかかわらず、自分が投稿する際に表現を変えたせいで、集計上は「AIが見逃した領域」に入っていた。つまり実際にはAIが拾っていたのに、集計は逆に振れた。この1件だけでも、指標全体に過適合の可能性があることは否定できません。

もっと長い期間、もっと多いサンプル、できれば別の人の集計があって初めて、この数字が本物かが確かめられます。この記事はn=1、1人の環境、1週間分のデータに基づく**自分から見える風景の報告**であって、一般化された主張ではないです。

### まとめ

* **AIは1次の分類者、人間はキュレーター。** 2人が同じ仕事をしないようにする。
* **出力が直接PRに行かない構造ならプロンプトは強めに書く。** 過検出が資産になる。
* **権限は開ける代わりに、副作用はプロンプトで縛る。** `NEVER`制約を明示。
* **北極星はレビュー品質、オートマージは副産物。** 順番を逆にするとメンタリングが犠牲になる。
* **捨てられる理由は分解して見る。** 優先度判断ミスはsuppression、暗黙知は文脈注入。
* **"ハルシネーション"は思ったより少ない。** 導入のハードルの高さは世間の印象より低い可能性がある。
* **コードの先のプロダクト責任はAIの死角と仮定する。** キュレーション層は抜かない。
* **低複雑度PRには回さない。** 複雑度ゲート。
* **チームサイズによってキュレーション層の設計は変わる。** そのまま持っていかずに、設計し直す。

## QnA

### `/loop`があるのでは？

Claude Codeの`/loop`はセッションの中で周期的にプロンプトを再実行する機能です。「PRレビューも`/loop`でいいのでは？」という問いは自然に出る。見かけは似ていますが、解いている問題が違います。

**`/loop`はセッションの中で生きています。** イテレーション間の対話履歴が積み重なる。1回目で「ビルドがまだ途中だ」と見たことを2回目が覚えていて、「今回は終わってるね」と反応する。状態を会話の中に置く設計です。

**自分の構成は正反対。** `claude -p`は呼び出しごとに独立プロセスで、前の呼び出しを知らない。状態は会話の外 ― `state.json`、`reviews/*.md`、Slack通知 ― に外在化されます。それぞれのtickは「新しく生まれてやるべきことをやって死ぬ」ワーカーであって、「生き続けて観察する」エージェントではない。

この違いが、2つの実用的な結果を生みます。

**観測可能性**: launchdがOSレベルで管理するので再起動後も自動で走り、結果はファイルシステムに残るので、自分がいなくても誰かが状態を読める。`/loop`はセッションが死ねば一緒に死ぬ。

**コストカーブ**: 呼び出しごとにfresh contextで始まるのでコストがO(1) × nでflat。`/loop`は累積コンテキストをイテレーションごとに再処理するのでO(n) × nで上がる。価格表ではなく**コスト関数の形**が違う、ということ。

もちろん`/loop`が強い領域もはっきりあります。**イテレーション間の状態維持が資産になる作業** ― 「前回この部分を指摘した、今回のpushはそれを直したか？」のような連続性が肝心のとき。

なので**再レビューの場合には`/loop`の方が合います**。1次レビューは「コードベース + PR diff」で自己完結していますが、2次/3次は「前回のレビュー + それ以降の変化」が必須ですから。

### じゃあ、なぜ再レビューはAIに任せないのか？

再レビューは技術的には`/loop`でうまく解ける作業かもしれません。でも意味的には**人間の社会的な儀式**です。作成者とレビュアーが「これ直した？」「あ、忘れてた、直しました」「いいね」のやりとりをする過程。ここにAIを挟み込むのは、その儀式を侵すこと。

逆に1次レビューは**まだ人間の会話が始まる前の検証**です。AIが入っても儀式は壊さない。むしろその会話の入力の質を上げる役。

そして意外にも、再レビューは思ったより頻繁には起きませんでした。**再レビューが繰り返し発生する時点で、それはPR以前の設計や作成者自身の整理に問題がある合図**かもしれない。この場合はAIツールで解く問題ではなく、PRを上げる前の段階で解くべき問題です。

## 最後に ― チームの開発パイプラインに載せる前に持ち帰りたいこと

自分が次にやりたいのは、この1週間の観察をそのままチーム全体の開発パイプラインに接続することです。そのときに備えて、この1週間から持ち帰っておきたい原則を整理しておきます。自分のための備忘であり、似た道を進もうとしている方への引き継ぎでもあります。

* **"AIの性能"ではなく"キュレーション構造"を先に設計する。** AIの精度は後から上がりますが、出力が誰に、どんな加工で届くかの設計をサボると、精度が上がっても無駄が増えるだけ。チーム導入でもここを最初に決めたい。
* **暗黙知の注入経路を最初から分けて用意する。** プロンプト層（優先度の低い指摘はsuppress）と文脈層（`CLAUDE.md`、referenceドキュメント）は別々に扱う。混ぜて「プロンプト改善」にまとめると両方解けなくなる。
* **複雑度ゲートを初期から入れる。** 全PRに回すのはトークンの無駄で、むしろ"AIを回さない判断"の方が価値が高いケースが普通にある。AIレビューが「いつでも全部に走る」ものだと思った瞬間、導入コストだけが累積します。
* **オートマージは目標ではなく、起きたら嬉しい副産物として扱う。** 北極星にした瞬間にレビュー設計が歪み、メンタリングとオーナーシップ伝播が犠牲になる。
* **"効いている"を体感で判断できるのは小規模チームだけ。** チームに広げる前に、AI投入がレビュー品質に実際に効いているか、ノイズで摩耗しているだけかを測れる仕掛けを先に作っておく。Adoption率、AI draftの吸収率、反映率 ― このくらいは最低限。
* **キュレーションを誰が担うかを明示的に決める。** 小規模チームでは暗黙に「自分のPRは自分で」で済みましたが、大きくなった瞬間この前提は壊れる。PR作成者？指名レビュアー？共通ボット？選択次第で、AIがチームに与える影響の形が変わります。

1ヶ月、3ヶ月と回すうちに、このリストはおそらく増えるし、書き換わります。そのときはまた別の記事として共有したいと思います。もし同じようなセットアップを試している方、あるいは大きいチームで類似の構造を運用している方の観察もあれば、ぜひ教えてください。
