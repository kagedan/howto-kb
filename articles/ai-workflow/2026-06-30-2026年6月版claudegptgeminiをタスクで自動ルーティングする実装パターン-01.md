## はじめに

2026年6月、フロンティアモデルが一気に出揃いました。Anthropicの **Claude Fable 5**（100万トークン文脈・常時思考・128K出力）、OpenAIの **GPT-5.6 Sol Preview**（コード/科学/セキュリティ強化）、Googleの **Gemini 3.5 Flash / Omni**（フラッグシップ級の知能をFlash速度で）。

ベンチマークが収束した今、エンジニアにとっての論点は「どれが一番賢いか」ではなく「タスクごとにどう振り分けるか」です。本記事では、タスク特性に応じてモデルを自動選択する軽量なルーターを Python で実装します。Claude Fable 5 が公開3日で一時停止した事例を踏まえ、フォールバック設計も入れます。

## 設計方針

ルーティングの判断軸はシンプルに3つです。第一に「入力の長さ」。超長文（数十万トークン）なら長文に強いモデルへ。第二に「タスク種別」。コーディング/エージェント、長文要約、マルチモーダル生成のどれかを見ます。第三に「可用性」。第一候補がダウン（規制・障害）したら次点へフォールバックします。

## ルーター実装

まずはタスクを表現するデータクラスと、利用可能なモデルを定義します。

```
from dataclasses import dataclass
from enum import Enum

class TaskKind(Enum):
    LONG_CONTEXT = "long_context"   # 長文要約・契約書・大規模コード読解
    CODING = "coding"               # エージェント的コーディング
    MULTIMODAL = "multimodal"       # 画像・動画など生成
    GENERAL = "general"             # 一般的なチャット

@dataclass
class Task:
    kind: TaskKind
    prompt: str
    approx_input_tokens: int = 0
    needs_video: bool = False

@dataclass
class Model:
    name: str
    max_context: int
    strengths: set
    available: bool = True

MODELS = [
    Model("claude-fable-5", 1_000_000, {TaskKind.LONG_CONTEXT, TaskKind.GENERAL}),
    Model("gpt-5.6-sol", 400_000, {TaskKind.CODING, TaskKind.GENERAL}),
    Model("gemini-3.5-omni", 1_000_000, {TaskKind.MULTIMODAL, TaskKind.GENERAL}),
]
```

ルーティング本体です。タスク種別と入力長で候補を絞り、優先順位順に返します。

```
def rank_models(task, models):
    def score(m):
        fits = m.max_context >= task.approx_input_tokens
        strong = task.kind in m.strengths
        video_ok = (not task.needs_video) or (TaskKind.MULTIMODAL in m.strengths)
        return (int(fits), int(strong), int(video_ok), m.max_context)

    candidates = [m for m in models if m.available]
    return sorted(candidates, key=score, reverse=True)
```

## フォールバック付き実行

Claude Fable 5 のように「最強だが突然使えなくなる」ケースに備え、優先度順に試して最初に成功したものを採用します。

```
class ModelUnavailable(Exception):
    pass

def call_model(model, task):
    if not model.available:
        raise ModelUnavailable(model.name)
    return f"[{model.name}] -> {task.prompt[:40]}..."

def run_with_fallback(task, models):
    for model in rank_models(task, models):
        try:
            return call_model(model, task)
        except ModelUnavailable:
            continue
    raise RuntimeError("利用可能なモデルがありません")
```

## 動作確認

```
# 50万トークンの契約書要約 -> 長文に強いClaudeが選ばれる
t1 = Task(TaskKind.LONG_CONTEXT, "契約書を要約して", approx_input_tokens=500_000)
print(run_with_fallback(t1, MODELS))   # [claude-fable-5] -> ...

# Claudeが規制で停止中 -> 別モデルへ自動フォールバック
MODELS[0].available = False
print(run_with_fallback(t1, MODELS))   # [gemini-3.5-omni] -> ...

# エージェント的コーディング -> GPT-5.6 Solが選ばれる
t2 = Task(TaskKind.CODING, "このリポジトリをリファクタして")
print(run_with_fallback(t2, MODELS))   # [gpt-5.6-sol] -> ...
```

タスク特性に応じて妥当なモデルが選ばれ、第一候補が落ちても自動で次点に切り替わることが確認できます。

## まとめ

2026年6月、Claude Fable 5・GPT-5.6 Sol・Gemini Omni が出揃い、選択の論点は「賢さ」から「ルーティング」へ移りました。タスク種別・入力長・可用性の3軸で振り分ければ、十数行のルーターでも実用に足ります。そして Claude Fable 5 の一時停止事例が示すように、フォールバックは贅沢ではなく必須です。1モデル依存は事業リスクになります。

実運用では各社SDKの呼び出し・レート制限・コストを call\_model に実装し、available をヘルスチェックから動的に更新するとよいでしょう。それでは、よいモデルルーティングを。
