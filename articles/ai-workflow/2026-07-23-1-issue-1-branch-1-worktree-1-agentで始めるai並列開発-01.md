複数のAIエージェントへ同じリポジトリの作業を同時に任せたい。そう考えたとき、最初に困るのはモデルの性能ではなく、作業ディレクトリの扱いです。

一つのディレクトリでAIエージェントAが画面を変更している途中に、AIエージェントBが別ブランチへ切り替える。未コミットの差分が残ったままテストが実行され、どちらの変更による失敗か分からなくなる。生成ファイルや依存環境も共有され、作業境界が崩れていきます。

この問題に対して、自分の個人開発では次の対応を基本にしました。

> **1 Task Issue = 1 Branch = 1 worktree = 1 Worker AI**

Git worktreeを使うと、一つのリポジトリから複数の作業ディレクトリを作り、それぞれで異なるブランチをチェックアウトできます。AIエージェントごとに専用の作業場所を渡せるため、ブランチ切り替えや未コミット差分の混線を避けやすくなります。

ただし、worktreeは万能な競合回避機能ではありません。作業ディレクトリを分けても、同じAPI契約、DBスキーマ、共有設定、生成物へ影響する変更は論理的に競合します。

この記事では、worktreeが分離できるものとできないものを区別し、Issue、ブランチ、worktree、AIエージェントの対応関係、並列実行前の判定、作成から削除までの運用を整理します。

> この記事は、自分の個人開発リポジトリで確認できた運用設計を一般化したものです。並列化による開発速度や競合削減率を定量検証した記事ではありません。

## 一つの作業ディレクトリを複数のAIで共有しない

通常のGit操作では、一つの作業ディレクトリで現在のブランチを切り替えます。人間が順番に作業するなら問題になりにくい一方、複数のAIエージェントを同時に動かすと、同じディレクトリの状態を奪い合います。

たとえば、次のような混線が起きます。

* AIエージェントAの未コミット差分がある状態で、Bがファイルを変更する
* Bがブランチを切り替え、Aが参照していたファイル状態が変わる
* 片方が生成したファイルを、もう片方が自分の成果物として扱う
* 同じ開発サーバー、キャッシュ、ポートを使い、テスト結果が干渉する
* どのIssueの変更が現在の差分に含まれるか追跡できない

AIへ「他の作業を壊さないで」と指示するだけでは、共有ディレクトリの状態は分離されません。並列実行したいなら、まずOS上の作業場所を分ける必要があります。

## worktreeが分けるのは作業場所とチェックアウト

Git worktreeを使うと、同じGitリポジトリの管理情報を共有しながら、複数の作業ディレクトリを持てます。

worktreeごとに分けられるものは、主に次のとおりです。

* チェックアウトしているブランチ
* 追跡対象ファイル（tracked file）の内容
* 未コミットの変更
* worktree内に作られた未追跡ファイル（untracked file）
* AIがコマンドを実行するカレントディレクトリ

一方で、Gitリポジトリのobject databaseやrefsなど、一部の管理情報は共有されます。また、作業ディレクトリを分けても、外部サービス、DB、ポート、コンテナ名、グローバルキャッシュまで自動的に分離されるわけではありません。

つまりworktreeは、**Gitの作業場所を分ける仕組み**です。開発環境全体のサンドボックスではありません。

## 1対1対応で作業単位を追跡する

自分の運用設計では、並列作業を原則として次の単位へ揃えました。

```
Issue #123
  = Branch task/123-settings
  = worktree issue-123
  = Worker AI A
```

この対応には、次の利点があります。

* Issueを見れば、目的と受け入れ条件が分かる
* ブランチを見れば、どのIssueの差分か分かる
* worktree名を見れば、どの作業場所か分かる
* AIへ渡す参照情報を一つのIssueとworktreeへ限定できる
* PRを一つのIssueとブランチへ対応させやすい

一つのworktreeへ複数Issueを混ぜると、差分、テスト、完了条件の単位が再び崩れます。小さな修正をまとめたくなっても、少なくとも同時実行中は一つのworktreeを一つの作業へ固定します。

名前にはIssue番号や短いタスク識別子を含めると追跡しやすくなります。

```
worktrees/
  issue-123-settings/
  issue-124-docs/

branches:
  task/123-settings
  task/124-docs
```

命名規則そのものより、Issue、ブランチ、worktreeを相互に特定できることが重要です。

## worktreeを作成してAIへ渡す

以下はBashで、`origin/main`を起点に新しいブランチとworktreeを作る例です。パス、ブランチ名、Issue番号は説明用であり、プロジェクトの規則に合わせます。

```
git fetch origin
git worktree add -b task/123-settings \
  ../worktrees/issue-123-settings origin/main
```

すでに存在するブランチをworktreeへ割り当てる場合は、`-b`を付けません。

```
git worktree add ../worktrees/issue-123-settings task/123-settings
```

同じブランチが別のworktreeですでにチェックアウトされている場合、Gitは通常、新しいworktreeへの割り当てを拒否します。既存worktreeを確認し、同じブランチを複数のAIへ割り当てる運用は避けます。

作成後は一覧を確認します。

AIへ渡す情報も、worktreeに合わせて限定します。

* 対象Issue
* 作業するworktreeのパス
* 対象ブランチ
* 参照する設計書
* 変更対象と対象外
* 実行するテスト
* 停止条件

AIには、原則として一つのworktreeだけを作業場所として渡します。別worktreeのファイルを直接変更させると、分離した意味がなくなります。

## 標準フローを固定する

worktreeの作成だけを自動化しても、その後の確認がばらばらでは運用が安定しません。私は次の流れを一つの作業単位として考えます。

1. IssueとTask Definitionを確認する
2. 並列実行してよいか判定する
3. ブランチとworktreeを作成する
4. 一つのAIへworktreeを割り当てる
5. 実装とテストを行う
6. 差分と受け入れ条件を照合する
7. pushしてPRを作成する
8. マージ後にworktreeを削除する

### 作業開始前

* Issue、ブランチ、worktree、AIが1対1になっている
* 起点ブランチが最新である
* 依存タスクが完了している
* 同時編集禁止ファイルが重複していない
* ポート、DB、コンテナなど外部資源の分離方法が決まっている

### 作業中

* AIが割り当て外のworktreeを変更していない
* 予定外の対象ファイルへ変更が広がっていない
* 新しい依存関係や論理的競合が判明していない
* 生成物や共有環境が他の作業へ影響していない

### PR作成前

* `git status`と差分が対象Issueだけに対応している
* 必須テストが対象worktree内で実行されている
* 対象外変更が含まれていない
* 依存ブランチの未マージ差分を暗黙に前提としていない

## worktreeだけでは論理的競合を防げない

二つのAIが別ファイルを変更していれば、安全に並列化できるとは限りません。

たとえば、AIエージェントAがAPIのレスポンス形式を変更し、BがそのAPIを使う画面を古い契約のまま実装している場合、Git上のマージ競合がなくても、統合後に不整合が起きます。

| 競合の種類 | worktreeで分離できるか | 例 |
| --- | --- | --- |
| 未コミット差分の混線 | できる | AIごとに別の作業ディレクトリを使う |
| ブランチ切り替えの干渉 | できる | worktreeごとに異なるブランチをチェックアウトする |
| 同一行のマージ競合 | 防げない | 同じファイルの近い行を変更する |
| 共有契約の不整合 | 防げない | API、型、イベントスキーマを別々に変更する |
| DB・外部環境の干渉 | 防げない | 同じDB、ポート、コンテナ名を使う |
| 生成物の上書き | 条件次第 | 同じ出力先や共有ディレクトリへ生成する |

worktreeの役割は物理的な作業分離です。並列実行の可否は、変更の意味と外部資源まで見て判断します。

## Task Definitionへ並列制御を書く

並列化の判定をAIの推測だけに任せず、Task Definitionへ依存関係、排他対象、競合リスクを持たせます。以下は説明用の簡略例です。

```
task:
  id: settings-display-name

scope:
  include:
    - src/features/settings/
    - tests/settings/

parallel_control:
  depends_on:
    - api-profile-contract
  exclusive_files:
    - src/api/profile-schema.ts
  shared_resources:
    - dev-database
    - port-3000
  conflict_risk: medium
  notes:
    - API契約変更タスクと同時実行しない
```

各項目の役割は次のとおりです。

| 項目 | 表すこと |
| --- | --- |
| `depends_on` | 完了を待つタスク |
| `exclusive_files` | 同時編集を避けるファイルや生成先 |
| `shared_resources` | DB、ポート、外部環境などの共有資源 |
| `conflict_risk` | 競合可能性を確認するための目安 |
| `notes` | 機械的な項目だけでは表せない注意 |

これらの項目は競合を検出する材料であり、排他制御そのものではありません。実行前にDefinition同士を比較し、重複があれば直列化するか、資源をworktreeごとに分けます。

たとえば、同じworktree識別子からポート、DB名、Composeプロジェクト名を決めると対応を追跡しやすくなります。

| worktree | ポート | DB名 | Composeプロジェクト名 |
| --- | --- | --- | --- |
| issue-123 | 3101 | app\_issue\_123 | app-issue-123 |
| issue-124 | 3102 | app\_issue\_124 | app-issue-124 |

この表は命名例です。実際に分離するには、各アプリケーションやコンテナの設定へ値を反映する必要があります。

`conflict_risk: low`だから安全と断定するのではなく、並列実行前の確認を絞り込む材料として使います。

## 並列化しない方がよいタスク

worktreeを作れることと、並列実行すべきことは別です。私は次のような作業を、原則として直列化するか、人間の確認後に開始します。

* 同じDBスキーマやmigrationを変更する
* 同じ公開APIや共有型を変更する
* 認証、認可、権限モデルへ影響する
* 同じlock fileや大きな生成物を更新する
* 一方の成果物がないと、もう一方の受け入れ条件を判定できない
* 変更予定ファイルが不明な大規模リファクタリング
* 本番環境や共有データへ接続する必要がある

小規模な個人開発では、常にAIの数を最大化する必要はありません。競合リスクが低く、作業境界が説明できるタスクだけを並列化する方が、後の統合作業を抑えやすくなります。

## マージ後に安全に片付ける

worktreeは、作業が終わっただけで自動的に消えるわけではありません。PRがマージされたことと、ローカルの作業ディレクトリを削除してよいことも分けて確認します。

削除前のチェック項目は次のとおりです。

* PRがマージ済み、または作業中止が決定している
* 必要なコミットがremoteへpushされている
* `git status --short`で未保存の変更がない
* ローカルだけに必要なログや調査結果が残っていない
* 対象worktreeでプロセスや開発サーバーが動いていない

メインのworktreeから一覧を確認し、対象を削除します。

```
git worktree list
git worktree remove ../worktrees/issue-123-settings
```

次に、メインworktreeにも未保存の変更がないことを確認してから、基準ブランチを更新します。以下は`main`へマージする運用の例です。

```
git status --short
git switch main
git pull --ff-only
git branch -d task/123-settings
```

`git branch -d`が削除を拒否した場合は、マージ状況、未pushのコミット、PRのマージ方式を確認します。squash mergeやrebase mergeでは、変更が取り込まれていても元のコミットが`main`の履歴へ残らず、`-d`が拒否する場合があります。

その場合も、すぐに強制削除するのではなく、ホスティングサービス上のPR状態と、残すべき差分・コミットがないことを確認してから、プロジェクトの削除方針に従います。

ディレクトリをファイル操作で先に削除した場合など、不要なworktree管理情報が残っているときは、状態を確認したうえで整理します。

`--force`によるworktree削除を通常手順にしないことも重要です。削除を拒否された場合は、未コミット差分がないかを先に確認します。

## 注意点

worktreeを分けても、AIの権限、外部サービス、秘密情報、実行中プロセスまで隔離されるわけではありません。必要に応じてコンテナ、環境変数、DB、ポート、認証情報もタスク単位で分けます。

また、複数worktreeはディスク使用量、依存関係のインストール時間、開発サーバーの管理、後片付けを増やします。小さなタスクをすべて並列化すると、実装より管理コストが大きくなる場合があります。

本記事の1対1対応は、並列開発の混線を抑えるための原則であり、Gitの必須規則ではありません。調査だけを行うAIや、コードを変更しないレビューでは、読み取り専用の参照情報で足りる場合もあります。

## まとめ

AI並列開発でworktreeを使う目的は、AIの数を増やすことではなく、作業単位を混ぜないことです。

* `1 Issue = 1 Branch = 1 worktree = 1 Agent`で対応関係を揃える
* 一つのAIには原則一つのworktreeだけを渡す
* worktreeが分けるのはGitの作業場所であり、開発環境全体ではない
* 依存、排他ファイル、共有資源、競合リスクを別途確認する
* 共有資源はworktree識別子を使って分離する
* 競合リスクの高いタスクは直列化する
* マージ後も未保存差分とブランチ状態を確認してから削除する

まずは、互いに独立した二つの小さなIssueを選び、別ブランチ・別worktreeへ割り当ててみてください。同時に、変更予定ファイルと共有資源を紙やDefinitionへ書き出します。

worktreeを作れるかではなく、二つのタスクを独立した作業契約として説明できるか。それが、並列実行してよいかを判断する出発点です。
