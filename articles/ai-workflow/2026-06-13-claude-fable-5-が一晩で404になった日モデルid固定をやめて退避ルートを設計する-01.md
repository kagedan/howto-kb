通説では、使っているモデルが使えなくなるときは猶予がある。ベンダーが deprecation を予告して、数ヶ月の移行期間があって、その間に新しいモデル名へ差し替えればいい。少なくとも自分はそう思って、`model="claude-fable-5"` をコードに直書きしていた。

実際は一晩だった。

6月12日、Anthropic は米政府の輸出管理指令を受けて、Fable 5 と Mythos 5 へのアクセスを止めた。公式声明によると、指令が届いたのは現地時間の17時21分。理由は「国家安全保障上の権限（national security authorities）」とだけ書かれていて、具体的な懸念の中身は開示されていない（[Anthropic 公式声明](https://www.anthropic.com/news/fable-mythos-access)、[日本経済新聞](https://www.nikkei.com/article/DGXZQOGN130K40T10C26A6000000/)）。

指令の対象は、正確には「外国籍の人物（米国の内外を問わず、外国籍の Anthropic 従業員を含む）によるアクセス」。ただコンプライアンスを確実にするため、Anthropic は結果的に全ユーザーで両モデルを無効化した。他のモデルは影響を受けない。Simon Willison は API が太平洋時間18時59分に `404`（"Claude Fable 5 is not available"）を返し、Opus 4.8 へ誘導されたと記録している（[Simon Willison](https://simonwillison.net/2026/Jun/13/us-government-directive-to-suspend-access/)）。

## これは deprecation じゃない

技術者として引っかかるのは、ここなんだよね。

deprecation なら、こちらが備えられる。告知があって、移行ガイドがあって、旧モデルは一定期間は動く。可用性はベンダーの「ロードマップ」という、予測できる変数の上にあった。

今回はそれが全部すっ飛んだ。前触れゼロ、移行期間ゼロ、しかも止めたのは技術的な不具合じゃなくて、開発者が一切コントロールできない規制の判断だ。可用性が「ロードマップ」ではなく「地政学」の上に乗っていた、という事実が一晩で可視化された。

しかも、これは単発の事故じゃない。その数日前、Anthropic は Fable 5 に対して説明なしの絞り込み（invisible throttling）をかけていたことを認めて謝罪している（[The Verge](https://www.theverge.com/ai-artificial-intelligence/948280/anthropic-claude-fable-invisible-distillation-guardrail)、[The Decoder](https://the-decoder.com/claude-fable-5-anthropic-admits-wrong-tradeoff-after-invisibly-throttling-rival-ai-researchers/)）。

つまり、自分がコードに固定した1つのモデルIDの上には、独立した二つの力が乗っていた。ひとつはベンダーの裁量。見えないガードレールやレート調整で、こちらに気づかれないまま挙動や可用性を変えられる。もうひとつは政府の指令。輸出管理で、予告なくゼロにできる。どちらが動いても、結果は同じ。あなたの本番が止まる。

固定したモデルIDは、見た目はただの文字列なんだけど、実態としては「自分の制御外にある二つの力に直結した単一障害点」になっていた。

## 404 は「待っても直らない」エラーだ

実装側で具体的に効くのは、エラーの種類を取り違えないことだと思う。

多くのクライアントは、API が失敗したらとりあえずリトライする。429 や 5xx はそれで正しい。一時的な混雑やサーバ側の問題だから、少し待って同じモデルに投げ直せば回復する。

でも今回返ってきたのは `404`、しかも「そのモデルは存在しない」という意味の404だ。これは何回叩いても直らない。恒久的に消えたものを、一時的な障害として扱ってリトライし続けると、待ち時間とコストを無駄に積み上げるだけで、ユーザーにはずっとエラーが返る。

エラーを「待てば直る」か「待っても直らない」かで分けて、後者は即フォールバックに分岐させたい。

| 返り値 | 意味 | 正しい対応 |
| --- | --- | --- |
| 429 | レート制限。一時的 | 指数バックオフで同じモデルに再試行 |
| 5xx | サーバ側の障害。一時的 | 数回までリトライ |
| 404 / model not available | モデル自体が消えた。恒久的 | リトライせず、すぐ次の候補へ |

## モデルを「賭ける対象」から「差し替え可能な依存」へ

対策はそんなに難しくない。モデルIDを1か所に固定して賭けるのをやめて、優先順位つきの候補リストにする。先頭がダメなら次へ落ちる、ただの退避ルートだ。

```
MODELS = ["claude-fable-5", "claude-opus-4-8"]  # 優先順の候補

def complete(messages):
    last_err = None
    for model in MODELS:
        try:
            return client.messages.create(model=model, messages=messages)
        except NotFoundError as e:    # 404: そのモデルは消えた → 次の候補へ
            last_err = e
            continue
        except RateLimitError:        # 429: 待って同じモデルで再試行
            backoff_and_retry()
    raise last_err
```

ポイントは、コードが直接ぶら下がる先を「モデル名」から「必要な能力」に一段ずらすこと。必要なのは `claude-fable-5` そのものじゃなくて、たとえば「ある精度以上のコード生成」だったりする。それを満たす候補が複数あれば、1つ消えても次に落ちるだけで、本番は止まらない。今回 Anthropic 自身が誘導先に Opus 4.8 を挙げていたのも、結局はこの考え方だ。

候補はできれば別ベンダーまで混ぜておきたい。今回みたいに会社まるごと止まる指令が来ると、同じ会社の中で候補を並べても全部巻き込まれる。

## で、結局どうなるか

今回の件を「Anthropic の不運な一日」で片付けると、たぶん次も同じところで詰まる。本質は、モデルIDがいつの間にか揮発する依存になっていたことだ。パッケージのバージョンを1点に固定したまま放置すれば、いつか supply chain で痛い目を見る。それと同じ構造が、モデルの世界でも当たり前になった。

だから自分の予測はこうだ。これからは「どのモデルが一番賢いか」を選ぶのと同じくらい、「そのモデルが消えたとき、何秒で別ルートに逃がせるか」が設計の評価軸になる。ベンダー側も、固有のモデル名ではなく能力ベースの間接参照（best-coding のようなエイリアス）を増やしてくるはずだ。

復旧の見通しはまだ立っていない。Anthropic は「誤解だと考えていて、できるだけ早く復旧させる」としているが、それを待つ前に、自分のコードから単一障害点になっているモデルIDを1つ消しておくのが、今日できる一番安いリスクヘッジだと思う。
