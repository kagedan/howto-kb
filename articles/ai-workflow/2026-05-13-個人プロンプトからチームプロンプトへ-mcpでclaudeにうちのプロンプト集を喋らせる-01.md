## 「あのリファクタ用プロンプトどこ？」を毎日2回言ってた

Claude や Cursor を毎日使うようになって、気づいたら **プロンプトがあちこちに散らばっていました**。

* Apple Notes に「リファクタ依頼の定型文」
* Raycast Snippets に「コードレビューチェックリスト」
* `~/prompts/*.md` に「PRサマリ生成プロンプト」
* iCloud Notes に「議事録要約プロンプト」
* Slack の自分宛 DM に「あ、あれ便利だった、保存しとこ」

そして毎日2回くらい「**あのプロンプトどこに保存したっけ**」と検索する時間が発生していました。

さらに **チームでプロンプトを共有する** となると詰みます。Apple Notes も Raycast Snippets も完全に個人専用。Notion に貯めると重いしコピペが面倒。Slackに貼ると流れる。

> 「Claude が自分のプロンプト集を **直接読みに来てくれたら** いいのに」

これ、**[MCP](https://modelcontextprotocol.io/)（Model Context Protocol）** で実現できます。今日の記事はその実例です。

## やりたいこと

Claude Desktop に、自分（とチーム）のプロンプト集に直接アクセスさせます。

```
私「リファクタリング用のプロンプトでこのコード整理して」

Claude「Memolから "リファクタリング" プロンプトを取得しました…
       〔リファクタリング用の指示テンプレートを適用して整理〕」
```

「リファクタリング用のプロンプト」の本体は Memol（チーム向けノート SaaS）に保存してあって、Claude が MCP 経由で勝手に検索してきます。コピペ不要。

## 仕組みは超シンプル

```
┌─────────────────┐
│ Claude Desktop  │
└────────┬────────┘
         │ "リファクタ用プロンプト探して" と言うと…
         ▼
┌─────────────────────────────┐
│ MCPサーバー                  │
│ (Memol が提供している)        │
└────────┬────────────────────┘
         │ ノートを横断検索
         ▼
┌─────────────────────────────┐
│ あなたのプロンプト集         │
│ - リファクタテンプレ          │
│ - コードレビュー観点          │
│ - PR説明文ジェネレータ        │
│ - エラーメッセージ → 対処法   │
└─────────────────────────────┘
```

[Memol](https://memol.app) は B2B 向けのチームノート SaaS で、MCP サーバーを標準で提供しています（**全プラン無料で MCP 利用可**）。今回はこれを「プロンプト管理用 DB」として使い倒します。

## なぜ Notion ではなく Memol か（個人的な選定理由）

最初は Notion でやろうとして挫折しました：

* Notion は **重い**。プロンプト 1 つコピーするのに 2 秒待つのが地味にストレス
* Notion の公式 MCP は存在するが **設定が煩雑**（OAuth周りで詰まる）
* ページ階層が深くなりがちで「フラットなプロンプト DB」に向かない

Memol を選んだ理由：

* **APIキー1枚** で MCP 接続完了（OAuth不要）
* ノートがフラット & 軽量（Notion の重さがない）
* **チーム共有** ができる（Apple Notes / Raycast Snippets との決定的な差）
* MCP が **全プラン無料**（料金で悩まない）

「個人ナレッジから抜け出してチームナレッジに進化させたい」という人にハマります。

## セットアップ（3ステップ・5分）

### 1. Memol に登録して API キー発行

[memol.app](https://memol.app) で無料登録 → 設定 → 「API / MCP連携」から API キーを発行。`memol_` で始まる文字列が出てきます。

### 2. Claude Desktop に設定追加

`~/Library/Application Support/Claude/claude_desktop_config.json` に追記：

```
{
  "mcpServers": {
    "memol": {
      "type": "http",
      "url": "https://memol.app/api/mcp",
      "headers": {
        "Authorization": "Bearer memol_YOUR_KEY_HERE"
      }
    }
  }
}
```

Claude Desktop を再起動。

### 3. プロンプトを Memol にメモする

新しいノートを作って、タイトルに `リファクタリング用プロンプト` みたいに分かりやすい名前を付け、本文にプロンプトを貼るだけ。タグやフォルダで分類しておくと検索精度が上がります。

これで準備完了。

## 裏で何が起きているのか（MCPプロトコルの中身）

「Claudeがプロンプトを取りに来る」と言われても具体的に何が起きているのか、開発者としては気になるところです。MCPは JSON-RPC 2.0 over HTTP なので、中身は完全に追えます。

Claude が `tools/list` を投げると Memol はこう返します：

```
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "search_notes",
        "description": "ノートをキーワードで全文検索する",
        "inputSchema": {
          "type": "object",
          "properties": {
            "query": { "type": "string", "maxLength": 500 },
            "limit": { "type": "integer", "maximum": 30 }
          },
          "required": ["query"]
        }
      },
      { "name": "list_notes",   "description": "..." },
      { "name": "get_note",     "description": "..." },
      { "name": "list_folders", "description": "..." },
      { "name": "create_note",  "description": "..." },
      { "name": "update_note",  "description": "..." }
    ]
  }
}
```

ユーザーが「リファクタリング用プロンプトを呼んで」と話しかけると、Claude は自動でこんなリクエストを組み立てます：

```
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "search_notes",
    "arguments": { "query": "リファクタリング", "limit": 5 }
  }
}
```

Memol 側のレスポンス：

```
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [{
      "type": "text",
      "text": "見つかったノート (2件):\n\n1. リファクタリング用プロンプト (id: ckxy...)\n   このコードを以下の観点でリファクタリングしてください:\n   1. 早期 return で nest を浅く...\n\n2. ReactリファクタTips (id: ckab...)\n   ..."
    }]
  }
}
```

Claude はこの応答からノートIDを拾って `get_note` で全文を取得 → そのプロンプトを「自分への指示」として再解釈 → ユーザーのコードに適用、という流れです。

権限エラーは JSON-RPC のカスタムエラーで返ってきます：

```
{
  "jsonrpc": "2.0",
  "id": 3,
  "error": {
    "code": -32003,
    "message": "Forbidden: this folder is restricted"
  }
}
```

Claude Desktop はこれを `"You don't have permission to access this folder"` のようにユーザー向けに翻訳して見せてくれます。

ちなみに [MCP Inspector](https://github.com/modelcontextprotocol/inspector) を使うと、上記のやり取りを GUI で全部確認できます。デバッグ時に便利です。

## 実例 1: リファクタリング用プロンプト

Memol に `リファクタリング用プロンプト` というノートを作って、こんな本文を保存しておきます：

```
このコードを以下の観点でリファクタリングしてください:

1. 早期 return で nest を浅く
2. 命名は意図が読めるもの (foo, bar 禁止)
3. マジックナンバーは定数に
4. 副作用とビジネスロジックを分離
5. JSDoc を関数の頭に付ける

リファクタ後に「変更点」のサマリをbullet pointで出力してください。
```

Claude にこう話しかけます：

> 「**Memolにある「リファクタリング用プロンプト」を読んで、このコードに適用して**」

```
// 雑なコード
function p(u, t) {
  if (u) {
    if (t > 0) {
      if (t < 100) {
        return u.n * t * 1.1
      }
    }
  }
  return 0
}
```

Claude は MCP 経由でプロンプトを取得 → そのルールに従ってリファクタ：

```
const TAX_RATE = 1.1
const MAX_QUANTITY = 100

/**
 * 商品の税込合計金額を計算する
 */
function calculateTaxIncludedTotal(
  user: { unitPrice: number } | null,
  quantity: number,
): number {
  if (!user) return 0
  if (quantity <= 0 || quantity >= MAX_QUANTITY) return 0

  return user.unitPrice * quantity * TAX_RATE
}

// 変更点:
// - 早期returnでnestを排除
// - p/u/t → 意味のある名前に
// - 1.1 → TAX_RATE 定数化
// - JSDoc を追加
```

**ポイント**: プロンプトの本文を毎回コピペしなくていい。プロンプトを改善したいときは Memol のノートを書き換えるだけ。

## 実例 2: コードレビュー観点リスト

```
タイトル: コードレビュー観点リスト

【セキュリティ】
- ユーザー入力はサニタイズしているか
- 認可チェックは抜けていないか
- 機密情報をログに出していないか

【パフォーマンス】
- N+1 クエリになっていないか
- 不要な useEffect の依存はないか
- 大きな配列を毎レンダーで生成していないか

【可読性】
- 関数は 30 行以内か
- 命名は意図が読めるか
- 例外パスにテストがあるか

このコードを上記観点でレビューして、見つかった問題を 観点 × 該当箇所 × 修正案 で出力してください。
```

PRが上がってきたら：

> 「**Memolの「コードレビュー観点リスト」で、この差分をレビューして**」

→ Claude が観点を一個ずつ漏れなく確認してくれます。**チームでレビュー観点を統一できる** のが地味に強い。

## 実例 3: エラーメッセージ → 対処法メモ

過去にハマったエラーを Memol に貯めておきます：

```
タイトル: Next.js Hydration Error の対処法

症状:
- Hydration failed because the server rendered HTML didn't match the client
- とくに <html> のクラス名で出やすい

原因パターン:
1. theme-provider の suppressHydrationWarning を root html に付け忘れ
2. dayjs/Date のフォーマットが SSR と CSR で異なる
3. window/document をトップレベルで参照している
4. 動的importしたコンポーネントがSSRしようとしている

対処手順:
1. <html suppressHydrationWarning> を確認
2. useEffect の中に window アクセスを移す
3. dynamic(() => import("..."), { ssr: false }) で逃がす
```

Cursor で開発中にエラーが出たら：

> 「**Memolにこのエラーの対処法のメモあるか探して**」

→ 一瞬で過去の自分の対処メモが出てきます。**過去の自分はチームの一員** 説。

## 実例 4: コミットメッセージ生成プロンプト

```
タイトル: コミットメッセージ生成プロンプト

以下の規約に従って、git diff から Conventional Commits 形式の
コミットメッセージを生成してください:

- feat / fix / chore / docs / refactor / test のいずれか
- subject は 50 文字以内
- 日本語で書く
- 本文は不要、subject 1行のみ
- 「○○を追加」「○○を修正」のように動詞で終わる
```

Cursor の Composer や Claude にこう投げる：

> 「**Memolのコミットメッセージ生成プロンプトで、この diff のメッセージ作って**」

→ チーム全員が同じフォーマットでコミットできるようになります。

## 実例 5: チームでプロンプトを共有する（本題）

ここまでは個人で完結する話でしたが、Memol は **ワークスペース単位で共有** できます。

例えば 5 人のエンジニアチームで、共通のプロンプト集を持つとどうなるか：

| ノート | 効果 |
| --- | --- |
| 「PR説明文ジェネレータ」 | 全員のPRが読みやすくなる |
| 「インシデント対応テンプレ」 | 障害時の Slack 投稿が統一される |
| 「新人オンボーディングプロンプト」 | 新人がClaudeに聞きながら自走できる |
| 「リリースノート生成」 | リリースのたびに同じ品質で出せる |

これ、プロンプトという「ナレッジ」をチームで蓄積する文化が生まれます。「Claude を使い倒している人のプロンプトを **誰でも使える**」状態になる。

個人で Apple Notes に貯めていたものを、Memol に移して **チームに公開** するだけで、**チーム全体の AI 活用レベルが底上げ** されます。

## 「フォルダ単位の権限」も効く

「経営層向けプロンプト」「営業向けプロンプト」を分けたい場合、Memol は **フォルダ単位で閲覧権限** を設定できます。MCP も同じ権限を尊重するので、新人エンジニアの Claude からは経営層フォルダのプロンプトは見えません。

```
📁 公開プロンプト集（全員）
   ├ リファクタリング用
   ├ コードレビュー観点
   └ PR説明文ジェネレータ

📁 営業チーム専用（営業ロールのみ）
   ├ メール返信テンプレ
   └ 提案書ドラフトプロンプト

🔒 経営層専用（OWNER/ADMINのみ）
   └ KPIレビュー観点プロンプト
```

権限設計を MCP 越しでも崩さない、というのは B2B っぽい設計判断です。

## 「PromptLayer / Langfuse / Helicone じゃダメなの？」

ここまで読んで「プロンプト管理なら PromptLayer や Langfuse があるじゃん」と思った方へ。これは **解いてる問題が違う** 話です。

| ツール | 主な用途 | 想定ユーザー |
| --- | --- | --- |
| **PromptLayer / Langfuse / Helicone** | プロンプトの **バージョン管理・A/Bテスト・コスト計測・実行ログ** | LLMアプリを **本番運用** する開発チーム |
| **Memol（今回の用途）** | プロンプトの **書き溜め・共有・呼び出し** | **日常業務で Claude / Cursor を使う人** |

PromptLayer 系はプロダクションの LLM 呼び出しを観測するための SaaS で、UI も SDK もそれ向けに作られています。**「個人がClaude Desktopで使うプロンプトを溜める」用途には完全にオーバースペック** で、しかも Claude / Cursor から直接呼ぶ手段がありません（API として呼ぶことはできるが MCP 対応はしていない）。

逆に Memol は LLM 観測ツールではないので、**実行ログ・コスト集計・A/Bテストといった機能はありません**。プロダクションで動かす LLM パイプラインの管理には向きません。そこは PromptLayer 等を使うべきです。

棲み分けはこんなイメージ：

```
🛠 本番LLMアプリの運用            → PromptLayer / Langfuse
👨‍💻 日々のClaude/Cursor利用       → Memol
📝 個人メモ＋共有ナレッジ          → Memol
```

**「プロンプトを書き溜めて、Claude から呼んで、チームに共有する」という日常的なワークフローを最短で実現したい人** に Memol は刺さります。逆に「100種類のプロンプトをバージョン管理しながら A/B テストしたい」なら PromptLayer を使ってください。

ちなみに Memol のノートには **編集履歴** が残るので、軽い意味での「バージョン管理」は無料でできます。「先週バージョンのリファクタプロンプトに戻す」くらいなら問題ありません。

## まとめ

* プロンプトを Apple Notes / Raycast Snippets に貯めるのは **個人ナレッジで終わる**
* MCP で AI に直接ノートを読ませると **コピペが消える**
* Memol を使えば **チームでプロンプトライブラリを共有** できる
* セットアップは **APIキー貼るだけで5分**

「あのプロンプトどこ？」を最後に言った日を、今日にしませんか。
