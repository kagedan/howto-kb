前回（第3弾）ではMCP経由でGitHubと繋がるAgentを作りましたが、一つ気になる点がありました。レビューするたびにAgentはまっさらな状態から始まります。

現実のレビュアーなら「このファイルは先週も同じ指摘をした」「このプロジェクトはPEP8より社内規約を優先するルールがある」といった文脈を持っています。それがAgentにはない。

今回はその問題を解決するため、短期メモリ（会話履歴）と長期メモリ（JSONファイル）の2層構造を実装します。

---

## メモリを2種類に分けて考える

まず整理しておきたいのが、「何を短期で持ち、何を長期で持つか」という設計の切り口です。

```
┌──────────────────────────────────────────┐
│           短期メモリ（会話履歴）            │
│  messages配列 → LLMのコンテキストウィンドウ │
│  セッション終了で消える                    │
└──────────────────────────────────────────┘
         ↕ セッションをまたいで引き継ぐ
┌──────────────────────────────────────────┐
│        長期メモリ（JSONファイル）           │
│  project_rules / file_history /         │
│  known_patterns                         │
│  永続化・セッションをまたいで保持          │
└──────────────────────────────────────────┘
```

| 種別 | 用途 | 実装 | 寿命 |
| --- | --- | --- | --- |
| 短期メモリ | 1セッション内の思考の流れ | `messages`配列 | セッション中のみ |
| 長期メモリ | プロジェクトルール・レビュー履歴 | JSONファイル | 永続 |

短期メモリはAnthropicのAPIに渡す`messages`配列がそのまま機能を果たします。長期メモリは今回シンプルにJSONファイルで実装します。

```
{
  "project_rules": ["このプロジェクトはPEP8より社内規約を優先する"],
  "file_history": {
    "review_targets/sample_01.py": [
      {
        "timestamp": "2025-04-01T10:00:00",
        "comment_summary": "SQLインジェクションの脆弱性を検出..."
      }
    ]
  },
  "known_patterns": []
}
```

## 長期メモリの「見せ方」

長期メモリをAgentに認識させる方法として、今回はシステムプロンプトへの注入を採用しました。セッション開始時にJSONを読み込み、文字列に変換してプロンプトに埋め込みます。

```
def build_memory_context(memory: dict, filename: str) -> str:
    lines = []

    if memory["project_rules"]:
        lines.append("## プロジェクトルール（記憶済み）")
        for rule in memory["project_rules"]:
            lines.append(f"- {rule}")

    if filename in memory["file_history"]:
        history = memory["file_history"][filename]
        lines.append(f"\n## {filename} の過去レビュー履歴（{len(history)}回）")
        for h in history[-3:]:  # 直近3件だけ渡す
            lines.append(f"- [{h['timestamp'][:10]}] {h['comment_summary']}")

    return "\n".join(lines) if lines else "（記憶なし）"
```

全履歴をそのまま渡すとコンテキストウィンドウを圧迫するので、直近3件に絞っています。この数は実際のプロジェクト規模に応じて調整が必要です。

```
long_term_memory = load_long_term_memory()
memory_context = build_memory_context(long_term_memory, filename)

system_prompt = f"""あなたはPythonコードレビューの専門家エージェントです。

以下は記憶済みの情報です：
{memory_context}

...(以下省略)
"""
```

## ツール設計：読み書きを明示的に

今回のAgentには4つのツールを持たせました。`read_code`と`post_comment`は前回と同じですが、メモリ操作のために`remember_rule`と`recall_history`を追加しています。

**remember\_rule**：Agentがプロジェクトルールを自分で長期メモリに書き込む

```
{
    "name": "remember_rule",
    "description": "プロジェクトルールを長期メモリに記憶する",
    "input_schema": {
        "type": "object",
        "properties": {
            "rule": {"type": "string", "description": "記憶するルール"}
        },
        "required": ["rule"]
    }
}
```

```
elif tool_name == "remember_rule":
    rule = tool_input["rule"]
    if rule not in long_term_memory["project_rules"]:
        long_term_memory["project_rules"].append(rule)
        save_long_term_memory(long_term_memory)
    return f"ルールを記憶しました: {rule}"
```

ユーザーが「このプロジェクトはf文字列禁止ね」と一言伝えると、Agentがこのツールを呼んで次回以降のレビューに反映します。従来はプログラマ側がメモリを更新していたところを、Agent自身が判断して書き込む設計です。

**recall\_history**：レビュー前に過去の履歴を参照する

```
elif tool_name == "recall_history":
    filename = tool_input["filename"]
    history = long_term_memory["file_history"].get(filename, [])
    if not history:
        return f"{filename} のレビュー履歴はありません（初回レビュー）"
    result = f"{filename} のレビュー履歴 ({len(history)}件):\n"
    for h in history[-5:]:
        result += f"  [{h['timestamp'][:10]}] {h['comment_summary']}\n"
    return result
```

**post\_comment**：コメント投稿と同時に履歴を記録する

```
elif tool_name == "post_comment":
    filename = tool_input["filename"]
    comment = tool_input["comment"]
    print(comment)
    update_file_history(long_term_memory, filename, comment)
    save_long_term_memory(long_term_memory)
    return "コメントを投稿・記録しました"
```

投稿と記録を1つのツールにまとめることで、記録し忘れという状況が起きません。

## ReActループの実装

短期メモリの管理はシンプルです。`messages`配列にAgentの返答とツール結果を順番に積み上げていきます。

```
short_term_memory = [{"role": "user", "content": f"{filename} をレビューしてください。"}]

while True:
    response = client.messages.create(
        model=MODEL,
        system=system_prompt,  # 長期メモリが注入済み
        tools=TOOLS,
        messages=short_term_memory  # 短期メモリ
    )

    short_term_memory.append({"role": "assistant", "content": response.content})

    if response.stop_reason == "end_turn":
        break

    tool_results = [execute_tool(...) for each tool_use block]
    short_term_memory.append({"role": "user", "content": tool_results})
```

ツール結果も`messages`に含めて次のターンに渡すのが重要です。これによってAgentは「自分が何を調べて何を判断したか」を一つの会話として追えます。

## 実行結果

### 初回レビュー

`sample_01.py`（SQLインジェクション）の初回レビューでは、Agentは`recall_history`→`read_code`→`post_comment`の順に3ツールを使い、3ループで完了しました。

```
--- ReActループ 1回目 ---
  → ツール呼び出し: recall_history  # 初回なので「履歴なし」が返る
  → ツール呼び出し: read_code       # コードを読み込む（247文字）

--- ReActループ 2回目 ---
  → ツール呼び出し: post_comment    # 分析完了→コメント投稿＋記録

レビュー完了（3回のループ）
短期メモリのターン数: 6
```

生成されたコメントの構成は、問題点の指摘・攻撃例の提示・修正コードの3段構成でした。

```
セキュリティリスク: SQLインジェクション脆弱性

問題点
query = f"SELECT * FROM users WHERE username = '{username}'"

ユーザー入力を直接SQLクエリに埋め込んでいるため、SQLインジェクション攻撃の対象になります。

優先度
1. 高: SQLインジェクション脆弱性（即座に修正が必須）
2. 中: リソースリーク対策（try-finallyまたはwith文を使用）
3. 中: エラーハンドリングの追加
```

### 2回目のレビューで何が変わったか

同じファイルを再実行すると、ReActログに明確な変化が現れました。

```
--- ReActループ 1回目 ---
  → recall_history: review_targets/sample_01.py の履歴1件を取得
  → read_code: review_targets/sample_01.py (247文字)

--- ReActループ 2回目 ---
  → post_comment: 「2回目指摘」として投稿
```

| 項目 | 1回目 | 2回目 |
| --- | --- | --- |
| `recall_history` の結果 | 「履歴なし（初回）」 | 「履歴1件を取得」 |
| コメントのタイトル | `SQLインジェクション脆弱性` | `SQLインジェクション脆弱性（2回目指摘）` |
| 攻撃例の数 | 1パターン | 2パターン（`admin' --` と `OR '1'='1`） |

Agentは過去の履歴を参照した結果、タイトルに「2回目指摘」と追記し、攻撃例も1つ増やしていました。プログラマが何も指示しなくても、記憶に基づいてコメントの内容を変えています。

### 生成されたJSONファイル

実行後の`memory_store/long_term_memory.json`の中身です。

```
{
  "project_rules": [],
  "file_history": {
    "review_targets/sample_01.py": [
      {
        "timestamp": "2025-04-19T...",
        "comment_summary": "セキュリティリスク: SQLインジェクション脆弱性..."
      },
      {
        "timestamp": "2025-04-19T...",
        "comment_summary": "セキュリティリスク: SQLインジェクション脆弱性（2回目指摘）..."
      }
    ],
    "review_targets/sample_02.py": [ ... ],
    "review_targets/sample_04.py": [ ... ]
  },
  "known_patterns": []
}
```

次のセッションではこのJSONが読み込まれ、システムプロンプトに注入されます。セッションをまたいだ「記憶」の実体は、このファイルです。

## 処理の全体像：シーケンス図で見る

実装の全体像を5つの参加者（ユーザー・Agent・Claude API・短期メモリ・長期メモリ）のシーケンス図で整理します。

### 初期化フェーズ

セッション開始時、AgentはまずJSONファイルから長期メモリを読み込みます（`load_long_term_memory()`）。ここで取得した`project_rules`や`file_history`をシステムプロンプトに文字列として埋め込みます。その後、ユーザーからのレビュー依頼を短期メモリ（`messages`配列）の最初のメッセージとして追加します。

### ReActループ1回目

APIに`messages`と`tools`を渡すと、AgentはまずReActの「考える」フェーズに入ります。最初のループではツールを2本続けて呼びます。`recall_history`で過去履歴を確認し（初回は「履歴なし」が返る）、続いて`read_code`でコードを読み込みます。各ツールの結果は`tool_result`として短期メモリに積み上げられ、次のAPIコールに引き継がれます。

### ReActループ2回目

コードの内容を把握したAgentは、`post_comment`でレビューコメントを投稿します。このツールの実装の中で`update_file_history()`が呼ばれ、コメントの先頭200文字がタイムスタンプとともに長期メモリへ書き込まれます。ここが短期と長期の境界線で、**このループだけが長期メモリへの書き込みを行います**。

### ReActループ3回目（完了）

APIが`stop_reason: end_turn`を返すとループを抜け、Agentはユーザーへレビューコメントのサマリーを返します。

### 次回セッションでの変化

図の最下部が今回の核心です。次のセッションで同じファイルをレビューすると、`load_long_term_memory()`が前回の記録を含む`file_history`を返します。この情報がシステムプロンプトに注入された結果、APIが生成するコメントのタイトルは「2回目指摘」付きになります。コードを1行も変えなくても、長期メモリの内容によってAgentの出力が変わる——それがこの設計の要点です。

### 2種類の操作を色で区別する

シーケンス図では長期メモリへの操作を青、短期メモリへの操作をオレンジで示しています。青い矢印は常に`memory_store/long_term_memory.json`への読み書きを意味し、オレンジの矢印はインメモリの`messages`配列への追加を意味します。この2種類の操作が交互に現れるのがReActループの構造です。
