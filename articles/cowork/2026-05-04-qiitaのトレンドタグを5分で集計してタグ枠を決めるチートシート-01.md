> Claude Cowork を社内AXに使っている私の実践ログです。社内固有名・個人名は伏せています。

Qiita で記事を書くたびに「タグ枠5つ、何にしよう」で5分悩むのが地味に嫌でした。直近1週間で伸びているタグを定量で把握できれば、迷いはゼロにできるはず。私は Claude にプロンプトを投げる前段で、Qiita API → bash + jq → Python の3段で**60秒以内に週間トレンドタグTOP10を出すチートシート**を仕込んでいます。本記事はそのコピペ可能な雛形をまとめたものです。

## 結論(60秒で動かす)

- 直近7日 × stocks 10超 の記事を `/api/v2/items` で 100件まとめ取り
- `jq` でタグだけ抽出
- Python の `Counter` で件数 × 重み(likes+stocks)を集計
- TOP10 を見て、自分の記事内容と整合する2〜3個を5枠目までに採用

これで「Qiitaタグの当たり感覚」を勘ではなく数字で運用できます。

## 1. Qiita API で母集団を取る

私は Qiita のトレンドAPIに頼らず、**`created` と `stocks` で十分な精度の母集団を作る**運用にしています。理由は後述の「ハマったところ」参照。

```bash
SEVEN_DAYS_AGO=$(date -u -d '7 days ago' +%Y-%m-%d)

curl -sS \
  "https://qiita.com/api/v2/items?query=created:%3E${SEVEN_DAYS_AGO}+stocks:%3E10&per_page=100" \
  -o /tmp/qiita_trend.json

jq 'length' /tmp/qiita_trend.json
```

`stocks:>10` を入れることで、「読まれた記事」だけに絞れます。`>` は URL エンコードで `%3E` です。生の `>` を渡すとシェルがリダイレクトとして食ってしまうので、ここは必ずエンコードして渡します。

## 2. jq でタグだけを抜く(中身の確認用)

集計前に**API のスキーマを目で確認するクセ**をつけたいので、私は必ず一段噛ませます。

```bash
jq -r '.[].tags[].name' /tmp/qiita_trend.json | sort | uniq -c | sort -rn | head -15
```

これだけでも単純カウントの TOP15 は読めます。ただし「件数1だが大バズ」が拾えないので、本番は Python で重み付けします。

## 3. Python で重み付け集計

```python
import json
from collections import Counter

data = json.load(open("/tmp/qiita_trend.json"))

tag_count = Counter()
tag_weight = Counter()  # likes + stocks

for item in data:
    w = (item.get("likes_count") or 0) + (item.get("stocks_count") or 0)
    for t in item.get("tags", []):
        tag_count[t["name"]] += 1
        tag_weight[t["name"]] += w

print("# count")
for k, v in tag_count.most_common(10):
    print(f"  {k}: count={v}, weight={tag_weight[k]}")

print("# weight")
for k, v in tag_weight.most_common(10):
    print(f"  {k}: weight={v}, count={tag_count[k]}")
```

`count` と `weight` 両方を出すのが地味に効きます。`count` だけだと「1記事で大量ストックされた特殊バズ」を拾えず、`weight` だけだと「みんな付けてるが平凡な汎用タグ」を見落とします。私は**両方の TOP10 の交差**を眺めて、自分の記事と相性の良いタグを2〜3個選んでいます。

## 4. 5枠の埋め方(チートシート)

私は次の方針で機械的に埋めています。

| 枠 | 候補 | 根拠 |
|---|---|---|
| 1 | `ClaudeCode` 等のブランドタグ | weight TOP常連 |
| 2 | アカウントの主要ブランド | 連続性 |
| 3 | 当日 weight TOP の中核(例: `AIエージェント`) | 露出 |
| 4 | 記事内容と整合する2位帯のタグ | 検索ヒット |
| 5 | ニッチ or ブランド統一(例: `業務自動化`) | 差別化 |

避けるタグも決め打ちにしています。

- `生成AI`(競合過多で埋もれる)
- `AI` 単独(同上)
- 内容と無関係に乗っかる流行タグ

## 5. 自動化メモ

私はこの集計を Claude のスケジュールタスクに組み込んでいます。記事生成プロンプトの先頭に「Step A: 上の3段の集計を実行し、TOP10を Markdown 表で本文に貼る前段に使う」と書いておくだけ。Claude を使うと、得られた TOP10 と当日の記事内容を見比べて 5 枠を選ぶ判断まで一気通貫でできます。「Claude にタグ選定をさせる」というより、「私が決めたルールを Claude に毎日同じ精度で実行させる」感覚です。

## ハマったところ

私が実際に詰まった順に5つ挙げます。

1. **`>` をエンコードし忘れる**
   `query=created:>2026-04-27` のままだと bash が `>` をリダイレクトに食う。`%3E` が正解。最初これで「件数0」を量産しました。
2. **`per_page=100` が API の上限**
   100超を渡すと 400 が返ります。1週間×stocks10超 程度なら 100件で十分でしたが、人気タグ単独で取りに行くと 100件超でも普通に発生するので、ページネーションが必要になります。
3. **`stocks_count` が `null` の場合がある**
   そのまま足し算すると `TypeError`。私は `(item.get("stocks_count") or 0)` で潰しています。
4. **`jq` のドットチェーン(`.tags[].name`)**
   記事配列の各要素にネストしたタグ配列をフラット化したい時、`.[] | .tags[].name` でも動きますが `.[].tags[].name` の方が短くて読みやすいです。
5. **「件数」と「重み」のどちらで判断するか問題**
   片方だけ見るとミスリードします。私は両方を出して交差を眺めるのを基本動作にしています。**count トップだが weight が低いタグは「みんな付けてるけど刺さってない」**ので、5枠目には選びません。

## まとめ

- Qiita の `/api/v2/items` + `created:` + `stocks:` でトレンドの母集団は60秒で取れる
- 集計は `count`(件数)と `weight`(likes+stocks)の両軸で見る
- 5枠は「ブランド × トレンド上位 × 内容整合 × ニッチ」で機械的に埋める
- `>` のエンコード、`null` の握り潰し、`per_page` 上限の3点でだけ詰まりがちなので先にコピペ雛形を作っておく
- 一度雛形ができれば、毎日の記事ごとのタグ選定は完全に脳内コストゼロで運用できる
