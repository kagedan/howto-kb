AI エージェントを開発していると、どこかで必ず「**bash をどこまで握らせるか**」という判断にぶつかります。

ユーザーが自然言語で指示すると、LLM がそれに合わせて bash コマンドを組み立てて実行する ── この仕組みができると、エージェントの**できることの幅が一気に広がります**。  
ファイルの整理、プログラムの実行、ネットワーク操作、ツールのインストール。LLM の柔軟さと bash の万能さが組み合わさると、ほんとうに何でもできそうな気になります。

ただ、同じ仕組みで、たとえば、

のような象徴的なコマンドも実行できてしまう世界に踏み込みます。  
これは直訳すると「ルートディレクトリ配下の全てを再帰的に強制削除する」という意味で、**マシンの中身を根こそぎ消しに行く破滅のコマンド**として古くから知られています。

最近の Linux では `/` そのものは安全装置で守られていますが、`rm -rf ./work/*` や `$TARGET` の変数展開ミスで、許可された範囲のファイルを一気に消してしまう事故は普通に起こります。

つまり、bash を LLM に握らせるというのは、こういう **「一行で重大な破壊が起きる可能性」** を常に抱えた状態、ということです。

強さと危うさが同じ顔で同居している。この問題にどう向き合うかで、エージェントの設計がけっこう変わる、というのが最近の自分の実感です。

## なぜ bash を握らせたくなるのか

少し寄り道して、そもそも**なぜ bash を LLM に握らせたくなるのか**を考えてみます。

bash は Linux/Mac で標準的に使われているシェルで、Windows で言うコマンドプロンプトや PowerShell にあたるもの、と思ってもらえれば大きく外しません。  
この手のシェルの強さは、ざっくり言うと **"何でもできる"こと** です。

* ファイルを作ったり、消したり、移動したり
* プログラムを実行する、止める、再起動する
* インターネットからツールをダウンロードする
* ネットワーク越しに別のマシンを操作する

これらを**ひとつの言葉の体系**で扱える。  
コンピュータの世界で長く磨かれてきた、ものすごく強力なインターフェースです。

ここに LLM が組み合わさると、何が起きるか。

ユーザーが「この PDF の中身を整理して、Excel にまとめて」と書くと、LLM が必要なツールを判断して、裏で `pdftotext` や Python を走らせてくれる。  
「このプロジェクトのセットアップをして」と言えば、`git clone` して、`npm install` や `pip install` で必要なライブラリを揃えて、設定までやってくれる。

**自然言語と bash のパワーが繋がった瞬間**、エージェントはいきなり"使える"存在に化けます。  
ここが握らせたくなる一番の理由だと思っています。

## ただし、危うさも同じ顔で付いてくる

bash の"何でもできる"は、当然ながら **"致命的なこともできる"** と表裏一体です。

冒頭の `rm -rf` 系もその一つですが、怖いのはそれだけではありません。

* ネットから拾ってきたスクリプトを、そのまま実行する(`curl ... | sh`)
* ライブラリのインストールを装って、裏でマルウェアを入れる
* 認証情報や SSH 鍵が置いてある場所を、黙って外部に送る
* 本番の DB に、意図しない範囲で UPDATE や DELETE をかける

この手の操作が**全部一行のコマンドで書ける**というのが bash の世界です。  
それは裏を返せば、LLM が**たった一行で、取り返しのつかない状況を作ることができる**ということでもあります。

問題は、LLM が**悪意を持ってそれをやる**わけではない、という点です。

LLM にとっての判断は、だいたいこんな感じに進みます。

* 「ユーザーはディスク容量を増やしたがっているようだ」
* 「キャッシュを消すのが良さそう」
* 「それなら `rm` で消せる」
* 「念のため広めに消しておこう」

この流れの途中にある小さなズレが、**最後のコマンドで爆発**します。  
悪意はない。でも、結果として破壊が起きる。

bash を LLM に握らせるというのは、**この「悪意なき破壊」の可能性を、常に机に置いたまま作業する**ということを意味します。

## Anthropic の解 ── bash の実行を仮想環境に閉じ込める

ここで面白いのが、**Anthropic 自身がこの問題にどう向き合っているか**です。

Anthropic が出している [Cowork](https://claude.com/product/cowork) というツールでは、コードやシェルコマンドの実行を**仮想環境(VM: Virtual Machine)の中で行う**かたちを取っています。

今回の話は「bash などのコード実行をどう扱うか」なので、その部分に絞って見ていきます。

作りとしてはこんな感じです。

* シェルコマンドやコード実行は、**Linux VM**の中で行われる  
  (ざっくり言うと、手元の PC の中にもう一つ別の PC を用意するようなもの)
* その VM からは、**ユーザーが許可したフォルダ**にしか触れない
* ネットワーク通信は、**設定された許可範囲**に制限される
* VM の中の一時的な作業領域は、セッションが終われば基本的に捨てられる  
  (手元の PC のフォルダなどに書き出した成果物は、当然そこに残ります)
* VM 内で問題が起きても、**共有していない OS 領域には直接波及しにくい**

VM の中で何が実行されるかは、**基本的に自由**です。`rm` を打つかもしれないし、`curl` で何か取ってくるかもしれない。bash の力はそのまま残しています。

でも、その影響範囲は **VM という箱の中**に収まっている。しかも箱自体の作業領域は使い捨てに近い扱いなので、変なことが起きても、**作業場を丸ごと捨てて作り直せばいい**、という発想です。

つまり Cowork のコード実行部分は、

* **bash の自由は潰さない**
* **でも被害範囲は物理的に制約する**
* **さらに作業場は使い捨て扱いにできる**

という三段構えで、さきほどの「強さと危うさのセット」をうまく切り分けています。「LLM の判断が揺らぐなら、動ける場所を狭めて、使い捨てにしてしまえばいい」という発想です。

この考え方は、実は AI エージェント特有のものではありません。  
ブラウザの JavaScript サンドボックス、Docker、OS の権限モデルなど、**「自由に動かしつつ、影響範囲で縛る」** という発想自体は、コンピュータの世界で長く使われてきたパターンです。  
Cowork も、その系譜を LLM エージェントに応用した設計と言えそうです。

## ただし、Cowork 方式が使えない領域もある

Cowork の設計は、とてもよくできています。  
ただ、このアプローチが**うまく機能する前提**があります。

それは、***「壊れても、やり直せる」世界で動いている、ということです。***

VM の中で `rm -rf` が走っても、VM ごと作り直せばいい。  
コードのビルドに失敗しても、もう一度やればいい。  
ツールのインストールで変なものが入っても、セッションごと捨てればいい。

**作業結果のリカバリが効く領域**では、この「箱に閉じ込めて、ダメなら捨てる」という発想はものすごく強いです。ファイル整理、コード生成、調査レポート作成、プロトタイプ構築 ── こういう作業なら、多少変なことが起きてもやり直せる。時間は失うけど、**資産は失わない**。

問題は、**やり直しが効かない領域**があることです。

たとえば、業務の本番データを直接扱うケース。  
生産 DB、顧客情報、図面、会計データ。  
こういうものは、VM の中に閉じ込めて作業する、という発想自体が成り立ちません。**現物そのものを触らないと意味がないし、触った瞬間に資産に影響してしまう**。

しかも厄介なのは、この手の資産では**間違いに気づけないことがある**、という点です。  
ちょっとした書き換えが紛れ込んでも、パッと見では正しそうに見えてしまう。気づくのは数日、数週間後、場合によってはもっと先。その頃にはバックアップも上書きされていて、戻せない。

Cowork のような「範囲で囲う」方式は、**VM の中で起きた問題が、共有していない領域へ直接波及しにくい**という防御層を提供します。これは強力です。  
でも、**手元の PC のフォルダに書き出された成果物そのものが壊れているかどうか**までは、仕組みとしては保証できません。

確率が著しく低いことと、ゼロであることは違う。  
**取り返しがつかない領域では、この差が致命的になり得ます**。

## AutoCAD エージェントの悩み ── CAD 世界の bash

自分がいま開発しているのは、AutoCAD 上で動く AI エージェントです。  
ユーザーが自然言語で指示すると、図面を調査したり編集したりしてくれるツールです。

![](https://static.zenn.studio/user-upload/68328951ab7d-20260424.png)

この開発を進める中で、実は**まったく同じ構造の悩み**にぶつかっています。

AutoCAD には **AutoLISP** という、古くから使われているスクリプト言語があります。  
図面内のオブジェクトを調べたり、作成・編集したり、AutoCAD のコマンドを呼び出したりできる、CAD 作業を自動化するための強力な手段です。CAD の世界における**事実上の bash**、と言ってもそこまで外していないと思います。

もし LLM にこの LISP を**自由に書かせて実行させる**ことができれば、エージェントのできることはかなり広がるはずです。「この寸法を揃えて」「この部分を別のレイヤに移して」「似た図形をまとめて整理して」──こういった要求に対して、LLM がその場で LISP を組み立てて CAD に流し込む、という動き方になります。

もちろん、これは可能性の話です。  
実際にやってみるとどこまで実用的になるかは正直わかりません。  
複雑な CAD 操作が bash の一行ほど素直に書けるとは限らないし、試せば試すほど「思った通りに動かない LISP」が出てくることも十分にあり得ます。

それでも、**方向性としては強力になり得る**という感触は確かにあります。  
bash の話と構造は同じです。

![](https://static.zenn.studio/user-upload/aee4fdd13319-20260424.png)  
*図面内の全オブジェクトを削除する LISP ( bash で言う `rm -rf` 的な例)*

ただ、自分はまだこれに踏み切れていません。

理由は、AutoCAD の世界では **Cowork のような「箱に閉じ込める」発想が、基本的に使えない**からです。

CAD の編集対象は、ユーザーが実際に業務で使っている**本物の図面**です。VM の中にコピーを取って作業しても、最終的にはその結果をどこかで本物に反映する必要がある。反映した時点で Cowork 的な「外へ直接波及させにくい」防御層だけでは守りきれなくなります。

そして、前のセクションで書いた通り、CAD の図面には「**間違いに気づけない**」という性質があります。寸法値の変化、レイヤ属性の変化、補助線の変質 ── パッと見では分からない変化が、数日後、数週間後、場合によっては製造工程に入ってから発覚することがあります。

99.9% の確率で意図通りの LISP が生成されたとしても、残りの 0.1% で図面が壊れる可能性をどう扱うか。そしてその 0.1% の「壊れ」に、ユーザーは気づけるのか。

この問いに自分の中で納得できる答えが出るまで、**LISP を LLM に自由に書かせる**という方向には踏み切れない、というのが今の立ち位置です。

## 判断軸: 「取り返しがつくか」で選ぶ

ここまでの話を整理すると、自分の中ではこういう軸にたどり着きます。

* **取り返しがつく作業**なら、Cowork のように「範囲で囲う」発想は強い
* **取り返しがつかない作業**なら、動的なコード実行をどこまで許すか慎重に考える必要がある

プログラムのコードなら Git が差分もバックアップも残してくれるので、少々変なコードが混ざっても大きな事故にはなりにくい。でも、図面や本番 DB のように、**気づけないまま書き換えられて戻せない**領域もある。この差はけっこう大きいです。

「LLM に bash(や LISP)を握らせるべきか」という問いに一言で答えるのは、たぶん無理です。  
ただ、**「その作業は取り返しがつくか?」** という問いを先に挟むと、設計の方針はけっこう見えてきます。

## 最後に

この問題に明快な正解はまだないと思っています。ドメインの性質、作る人の良識、どこまで守りたいかという意識 ── 絡む要素が多すぎて1つの答えに収まらない、というのが今の感触です。

たとえば 99.9% の確率で成功するなら十分だ、と考える人もいれば、0.1% の失敗で何が起きるかを見たら、とても許容できない、と考える人もいる。どちらも間違ってはいません。  
**扱うドメインと、その人が守りたいものによって、許せる確率のラインが変わる**だけです。

同じような設計の悩みを抱えている人がいたら、

**「この作業は取り返しがつくか?」**

という問いを一度挟んでみると、意外と見通しが変わるかもしれません。
