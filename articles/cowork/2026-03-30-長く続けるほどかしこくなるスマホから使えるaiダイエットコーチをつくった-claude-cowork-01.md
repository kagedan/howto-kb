## コ。 〜これまでの食事管理について〜

私自身食事管理はかなり初心者なのですが、これまでは主に2つの方法を試し、それぞれのメリデメを感じながらも **「なんかちゃうな」** と思い続けてきました。

### エントリーナンバー①：あすけん

* Good
  + **Appleヘルスケアとの連携**！自宅の体重計がヘルスケアに連動するので、運動量・体重をわざわざ記録する必要がない
  + バーコード読み取り機能など、記録を楽にしようという気概を感じる
  + 食材マスタをしっかり管理していそうなので、誤差・ブレが少なそう
* More
  + **「名もなき料理」の記録しにくさ**。。基本的には食材ごとに分量を登録する必要があるので、自宅で余っている野菜で豚汁なんか作ろう日にゃ「味噌」「大根」「舞茸」「玉ねぎ」...と、全具材登録する羽目になるのです。
  + 「Myメニュー」という、複数の食材を一定の組み合わせで混ぜた料理自体を登録する機能もあるが(例：ポテトサラダ)、結果的に同じ飯を作って食うのが楽になってしまい、**食事の楽しみが減りがち**だった

### エントリーナンバー②：ChatGPT (プロジェクト機能)

* Good
  + **Web検索**！不明な食材があった時、必要に応じてWeb検索して、当該商品、または類似食材をぴたりと当ててくれる。
  + **メモリ機能**！一度食べた食材のPFCを (いったんは)覚えてくれるので、「納豆」と言ったらこいつが普段食う納豆はこれだな、と当ててくれる
  + あいまいに入力した時、幅を持って出してくれること。例えば「豚肉」だったら、「脂身の量によって脂質が5~20gで変動します」と言ってくれるので、その後「赤身多めだったので8gくらいで」と言って計算してくれるなど、**対話を通じて幅を絞れる** のがよい。
* More
  + 検索時間が長すぎ。精度を担保するため、Extended~Heavy Thinkingで回していたのだが、検索時間も含めると1回のやり取りで5分くらいかかっていた記憶がある
  + ログ・メモリ・チャット履歴が全て一体になっており、**1.5ヶ月くらいでチャットが激重になった**こと。結果的にプロジェクト内の別のチャットを利用することになったが、一部のメモリやログが使えず、せっかく続けたのに。。と意気消沈してしまった。

## 今回やりたかったこと (≒要件)

ということで、今回やりたかったこと(≒要件) はこのあたりです。

1. フローとストックを分離したい：あくまでチャットは入力・対話用として、食事記録などのストック情報と分離したい
2. 使うたびに「かしこく」なってほしい：自分の目標体重や設定カロリー、過去に検索した食材などを記憶し、随時アップデートしていってほしい
3. スマホから入力したい ：とにかく手軽に、かつ、できればラベル画像のOCRしたりWeb検索かけたり、曖昧な指示にも推論で答えられるよう、AI経由で入力したい

つまり **「とにかく長く食事管理を続ける」** ことに全振りする仕組みを作りたかったわけです。

そこで今回、最近登場したClaudeをフル活用しつつ、  
サクッと「AIダイエットコーチ」をつくってみました。

こんな感じで動きます。

> ①スマホから「昼、牛丼」とClaude Dispatchに送る。

> ②Dispatch経由でCoworkに接続。

> ③Cowork側で：
>
> * references/foods.mdに牛丼の構成があるか確認
> * なければWeb検索してPFC算出 → foods.mdに追記
> * logs/2026-03.mdに記録
> * status.mdの目標と比較して残カロリー・PFC余裕を返答

> ③一連の処理が終わったあと、スマホに「昼：約520kcal / P32g / F18g / C55g。残り夜で P40g・600kcal目安」みたいに返ってくる。

## ポイント1. フローとストックの分離

上記の要件を実現するため、Claude Cowork側では以下のようなフォルダ構成を設定しました。  
Claude Coworkはローカルのフォルダにアクセスできます。この特性を活かして、チャットとデータを分離する以下のような構成を考えました。

```
~/diet-tracker/              ← Cowork Projectフォルダ
├── INSTRUCTIONS.md           ← フォルダ指示（Claudeの行動ルール）
├── references/
│   ├── foods.md              ← 食材PFCマスタ（ストック）
│   └── status.md             ← ダイエット目標・進捗（随時更新）
└── logs/
    ├── 2026-03.md            ← 月別食事ログ
    └── 2026-04.md
```

イメージはこんな感じです。  
![](https://static.zenn.studio/user-upload/50097aaa6f45-20260331.png)

設計思想はシンプルで、**チャットは「今この食事を記録して」という一時的なやり取り（フロー）に徹し、記録はすべてファイル（ストック）に書く** というものです。

これにより、チャットをどれだけ重ねても、あるいは新しいセッションを始めても、ファイルさえ読めば状態を復元できるようになり、

ChatGPTでの「チャットが重くなる」問題から解放されることに...！感謝です。

## ポイント2. 使うたびに「かしこく」なる

自分の目標体重や設定カロリー、過去に検索した食材などを記憶し、随時アップデートする仕組みがほしいと思い、以下の3つの仕組みを仕込んでいます。

### 使うほど育つ指示書 — INSTRUCTIONS.md

INSTRUCTIONS.mdはいわばAIダイエットコーチの指示書ですが、あくまで静的な設定ファイルではなく、**ユーザーとの対話を通じて進化する**のがポイントです。

例えば、初期のINSTRUCTIONS.mdでは食事記録後のフィードバックは「累計と残り目安を簡潔に返す」だけでした。

```
### 1日サマリーと残り目安
✅ 昼記録しました
累計: 650kcal / P42g / F12g / C85g
残り目安: 約850kcal / P50g前後 / F20g程度
```

何度か使ってみて「毎食Good/Moreのフィードバックがほしい」と思い、Coworkに「このルールをINSTRUCTIONS.mdに追記して」と伝えました。すると、こんな感じで以降の応答を書き換えることができました。

```
### フィードバック（毎食必須）
✅ 〇〇記録しました

実績: XXXXkcal / PXXX.Xg / FXX.Xg / CXXX.Xg
目標: 2,050kcal / P135g   / F55g   / C255g
差分: -XXXkcal  / -XX.Xg  / -XX.Xg / -XX.Xg

**Good**: その食事やここまでの選択で良かった点
**More**: 改善できるポイント（次の食事へのアドバイス含む）
**応援メッセージ**: モチベーションが上がる一言
```

Coworkではルール自体がファイル上で永続化されるため、新しいセッションでも同じ振る舞いが保証される点が素晴らしいです。**「使い方を覚えさせる」のではなく、「使い方をファイルに書く」** という点がポイントです。

status.mdも同様で、体重が変わったり目標を見直したければ書き換えてしまえばOKです。

### 食材DBがどんどん学習する仕組み — foods.md

foods.mdは食材のPFCマスタデータで、以下のようなテーブルで構成されています。

```
## コンビニ・市販品

| 食材 | 基準量 | kcal | P(g) | F(g) | C(g) | 出典 |
|------|--------|------|-------|-------|-------|------|
| ローソン サラダチキン スモーク | 1袋 (115g) | 125 | 26.9 | 1.7 | 0.7 | ローソン公式 |
| セブン 砂肝おろしポン酢 | 1パック | 111 | 13.1 | 3.1 | 7.7 | ラベル読取 |
```

処理する流れはこんな感じで、**一度調べた食材は二度と検索しない** ようにできています。  
使えば使うほどfoods.mdが充実し、応答が速くなる (はず) です。

1. ユーザーが「昼：ローソンのサラダチキン」と送る
2. Coworkはまずfoods.mdを検索する（INSTRUCTIONS.mdの指示どおり）
3. **ヒットすればそのままPFC計算に進む**（Web検索なし → 高速）
4. ヒットしなければWeb検索し、結果をfoods.mdに追記する

※毎回ザバスのPFCを検索していたあの時間とさよなら...！

## ポイント3. スマホから記録・操作できる (Claude Dispatch)

平日はほぼ出社しており私用PCを開けない私、かつ食べた直後でないと記録を忘れてしまう私にとっては、スマホから記録・操作できることが条件でした。

そこで、最近登場した**Claude Dispatch**を使うことで、  
スマホからClaude Coworkに接続し、食事記録ができるようにしました。

[https://support.claude.com/ja/articles/13947068-coworkのどこからでもタスクをclaudeに割り当てる](https://support.claude.com/ja/articles/13947068-cowork%E3%81%AE%E3%81%A9%E3%81%93%E3%81%8B%E3%82%89%E3%81%A7%E3%82%82%E3%82%BF%E3%82%B9%E3%82%AF%E3%82%92claude%E3%81%AB%E5%89%B2%E3%82%8A%E5%BD%93%E3%81%A6%E3%82%8B)

**記録の流れ:**

1. スマホのSlackから「昼：サラダチキンとおにぎり」と送る
2. Dispatchがメッセージを受け取り、Coworkと同じINSTRUCTIONS.mdに従って処理する
3. foods.mdを参照 → PFC計算 → logs/に記録 → フィードバックをスマホのDispatchチャット側に返信

PCでもスマホでも、同じフォルダ・同じ指示書・同じ食材DBを参照するため、どこから入力しても一貫した記録が残ります！(前述のスクショの通りですね)

**本当に便利！**

## まとめ・学び

これだけのこだわり(というか己の怠惰との向き合い方)を詰め込んだAIダイエットコーチなんですが、なんと**1日で完成**してしまいました。びっくり。

もちろん普段からClaude Coworkを使い倒しており、基本的な知識があったというのはあるのですが、なんとなくやりたいことをClaudeに伝えたらいい感じのアイデアを一発でくれて、すぐCoworkで実現できちゃう。。

![](https://static.zenn.studio/user-upload/a9cb8c17c46f-20260331.png)

なんか本当に...いい時代ですね...

ちなみにこの一連の仕組みをClaudeに解説させたのですが、以下のようなコメントをいただきました。

```
### 技術的なポイント

この仕組みの本質は「LLMエージェントのステート管理をファイルシステムで行う」というアーキテクチャにある。
INSTRUCTIONS.mdは振る舞いの定義、foods.mdは知識の蓄積、status.mdは状態の管理、logs/は履歴の永続化。
それぞれ分離されているので、どれか1つを変えても他に影響しない点が特徴である。

LLMを「賢い対話相手」としてだけでなく、**「ファイルを読み書きできるエージェント」として設計する**と、チャットの限界を超えた実用的なシステムが作れる。
食事管理に限らず、日報、学習ログ、プロジェクト管理など、
「継続的に記録し、蓄積した知識を再利用する」タスク全般に応用できるパターンだと思う。
```

確かにと思いつつ、この仕組み自体は**企業の在庫管理など色々と応用が効くアイデア**だな〜とも感じました。  
(使うほどかしこくなる仕組みはAIチャットにはない強いですし、需要高いですよね。)

**とにもかくにも、自分が食事管理を続けることで、この仕組みをアップデートしていこうと思います〜！**
