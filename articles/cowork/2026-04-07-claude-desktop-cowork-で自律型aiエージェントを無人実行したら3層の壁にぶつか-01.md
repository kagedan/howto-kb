# はじめに

「毎朝9時に、前日のAI活用作業ログを自動でYAMLサマリーにして共有ドライブに保存する」

シンプルなアイデアです。Claude Desktop の Cowork（ローカルエージェント）機能にはスケジュールタスク機能があり、これを使えばAIが毎日自律的に動いてくれます。

しかし実際に動かしてみると、3日間で3種類の異なる障壁にぶつかりました。

* **壁①** スクリプトがVM内に存在しない（環境・パス問題）
* **壁②** Google Drive への書き込みが VM から不可（セキュリティ制約）
* **壁③** 処理済み記録ファイルが毎回リセットされる（状態管理の論理バグ）

それぞれが「インフラ層」「セキュリティ層」「アプリロジック層」と異なるレイヤーの問題で、1つ解決したら次が出てくるタマネギ構造でした。本記事はその3日間のトラブルシューティング記録です。

あわせて、AIが自律的に環境の不備を検知してホストへのアクセスを要求するという、想定外の「セルフヒーリング的挙動」も観察できたので、そちらも共有します。

---

# システム構成の全体像

まず何を作ろうとしていたかを整理します。

```
【ホスト Windows PC】
  ├── Claude Desktop (Cowork)
  │     └── Scheduled Task（毎朝9時）
  │           └── Cowork セッション起動
  │                 └── SKILL.md を読んで自律実行
  │
  ├── ~\.claude\projects\       ← Claude Code のセッションログ
  ├── %LOCALAPPDATA%\...\       ← Cowork のセッションログ
  ├── %APPDATA%\Code\...        ← GitHub Copilot のログ
  │
  └── G:\共有ドライブ\AI活用\   ← Google Drive Desktop（保存先）
```

Cowork はスケジュールタスクを実行するとき、**Linuxベースの VM を起動してその中でエージェントを動かします**。VM 内のファイルシステムは、ホストの一部ディレクトリを FUSE でマウントする形で構成されています。

これが今回の問題の根源でした。

---

# 壁①: VM 内でスクリプトが見つからない

## 何が起きたか

2026-03-23 の朝9時（UTC 00:00）にスケジュールタスクが起動しました。タスクは SKILL.md を読んで `extract_sessions.py` を実行する手順になっています。しかし AI のログにはこう書いてありました。

```
[2026-03-23T00:07:14.640Z] ASSISTANT:
`extract_sessions.py` スクリプトがスキルフォルダに存在しないことを確認しました。
スクリプトを作成してから処理を続けます。
```

さらに直後：

```
[2026-03-23T00:07:44.343Z] ASSISTANT:
スクリプトが存在せず、ログディレクトリのアクセスも必要なため、
Windowsホストのディレクトリへのアクセスを要求します。
```

## なぜ起きたか

Cowork の VM が `.skills` ディレクトリとしてマウントするのは、スケジュールタスクに設定したスキルフォルダの **直下のファイルのみ** でした。今回のスキルはこの構造になっています。

```
daily-ai-work-summary/
├── SKILL.md          ← マウントされた
└── scripts/
    └── extract_sessions.py  ← マウントされなかった
```

SKILL.md は VM 内の `/sessions/<id>/mnt/.skills/skills/daily-ai-work-summary/` に見えていましたが、`scripts/` サブディレクトリは含まれていませんでした。AI は SKILL.md の手順通りにスクリプトを呼ぼうとしても、対象ファイルがないため実行できません。

また `~/.claude` は VM 内では**空のディレクトリ**として見えます。ホスト側の `%USERPROFILE%\.claude\` とは別物です。ログも設定ファイルも何もありません。

## AIが自律的にアクセスを要求した（セルフヒーリング的挙動）

ここで興味深いことが起きました。AIは行き詰まった後、自律的に `request_cowork_directory` を使ってホスト側のホームディレクトリへのマウントを要求しました。

これは「スクリプトが見つからない → ホスト側に存在するはず → アクセスを要求する」という推論をAI自身が行い、不足リソースを補完しようとした挙動です。人間のオペレーターが介在せず、VM環境の制約を認識した上で自律的に対処しようとしたという点で、AIエージェントの自律性として印象的でした。

ただしこれは **無人実行時には機能しません**。`request_cowork_directory` はユーザーの承認が必要なダイアログを出すため、スケジュール実行では止まってしまいます。

## 解決策

スケジュールタスクで使うスキルの **スキルフォルダ直下に必要なファイルをすべて配置する** のが根本解決です。

```
# Before: サブディレクトリが対象外
daily-ai-work-summary/
├── SKILL.md
└── scripts/extract_sessions.py  ← VM内でアクセス不可

# After: スキルフォルダ直下にも配置（またはスキルフォルダ=scriptsフォルダ）
```

あるいは、スケジュールタスクのプロンプトを **Claude Code CLI 経由** で実行する形に変えることも有効です（後述）。

# 壁②: VM から Google Drive に書き込めない

## 何が起きたか

壁①を乗り越えてログ解析が完了し、YAML サマリーの生成まではできました。しかし保存先の `G:\共有ドライブ\AI活用\...` への書き込みで失敗しました。

```
[2026-03-23T06:29:59.318Z] ASSISTANT:
Google DriveのG:ドライブはVMからアクセス不可のため、ローカルに保存します。
YAMLを生成します。
```

## なぜ起きたか

Cowork の VM がホスト側からマウントできるのは限られたディレクトリのみです。セッションログの観察から、以下の構成が確認できました。

| ディレクトリ | VM内からのアクセス |
| --- | --- |
| `~/.claude/` | ✅ マウント可（読み書き可） |
| `.skills/` | ✅ マウント可（読み取り） |
| `uploads/` | ✅ マウント可 |
| `G:\共有ドライブ\` (Google Drive Desktop) | ❌ アクセス不可 |
| `%USERPROFILE%\` 全体 | ❌ 直接アクセス不可 |

Google Drive Desktop が Windows 上に仮想ドライブ（`G:\`）としてマウントしているディレクトリは、Cowork VM の FUSE 構成には含まれていません。

つまり **Cowork で動くエージェントは、設計上 Google Drive に直接書き込めません**。

## AIのフォールバック挙動

SKILL.md にはフォールバック処理として「Google Drive が見つからない場合は YAML をコンソールに出力し手動保存を案内する」と記載してありました。AIはこれに従い、YAML をセッションの出力テキストとして吐き出しました。保存はできていませんが、データは失われていません。

3/25 のセッションでは、フォールバック先として `.claude` 配下のローカルパスに保存するよう改善されていました。

```
[2026-03-25T00:17:37.952Z] ASSISTANT:
Google Drive への書き込み不可 — Cowork の VM は G:\ にアクセスできません
（FUSE マウントが .claude・.skills・uploads のみに限定）。
代わりに以下のフォールバック先に保存しました: ...
```

## 解決策

根本的な解決策は2つあります。

**案A: Claude Code CLI 経由でスケジュール実行する**

Claude Code CLI（`claude` コマンド）はホスト側で直接動作するため、Google Drive も含めたファイルシステムにフルアクセスできます。Cowork の Scheduled Task の代わりに、OS のタスクスケジューラ（Windows タスクスケジューラ / macOS launchd）で `claude -p` を定期実行します。

```
REM Windows タスクスケジューラで実行するコマンド例
claude.exe --dangerously-skip-permissions --print ^
  "daily-ai-work-summary スキルを実行。未処理日を処理し G:\共有ドライブ\AI活用\your-name\ に保存"
```

`--dangerously-skip-permissions` は非対話実行時にパーミッション承認ダイアログをスキップするためのオプションです。自動化環境でのみ使用してください。

**案B: Cowork の中間ファイルを `.claude` 配下に書いて同期する**

`.claude` 配下はホストと共有されているため、Cowork 側で `.claude/tmp/` に YAML を書き出し、別のジョブ（または次回の Cowork セッション）で Google Drive へ転送するという2段階の構成も取れます。ただしこの方法は設計が複雑になるため、案A の方がシンプルです。

# 壁③: 処理済みファイルが毎回リセットされる

## 何が起きたか

3/24 には Cowork のセッションが**4本**起動していました。

```
# 2026-03-24 のセッションログ
Session local_a95c4219 [02:27 UTC] → 認証エラー（OAuth token expired）で即終了
Session local_e3821b1b [00:03 UTC] → スクリプト未配置で処理断念
（他2セッション）
```

同じ日のスケジュールタスクが複数回実行されたのは、Cowork の「キャッチアップ機能」が関係しています。Claude Desktop を起動したときに「前回実行できなかったスケジュールタスクが溜まっている」と判断されると、まとめて起動されます。

そして別の問題も見つかりました。3/23 の処理で `daily-summary-done.json`（処理済み日付の記録ファイル）に 3/22 を処理済みとして書き込んでいましたが、そのファイルが**VM内のパス**に書かれていました。

```
VM内で書き込んだパス:
/sessions/<session-id>/mnt/.claude/skills/daily-ai-work-summary/daily-summary-done.json

期待していたホスト側のパス:
%USERPROFILE%\.claude\skills\daily-ai-work-summary\daily-summary-done.json
```

## なぜ問題になるか

セッションが終わると VM は破棄されます。`/sessions/<id>/` 以下のデータは消えます。`.claude` マウントへの書き込みがホスト側に正しく反映されていれば問題ないのですが、セッション ID が変わる（= 別の VM が立ち上がる）たびに done.json の内容が見えなくなるケースが発生していました。

結果として以下の誤動作が起きます。

```
# done.json が正しく引き継がれていない場合

シナリオA（再処理）:
  「3/22 は処理済みのはず」→ done.json が見えない → 未処理と判定 → 再度処理

シナリオB（処理漏れ）:
  「まだ処理してないはず」→ 誰かがdone.jsonを書き込んでいた → 処理済みと誤判定 → スキップ
```

3/24 の「未処理日なし」誤判定も、このシナリオBに当たります。前のセッションが中途半端な状態で done.json に書き込んでいたため、後続セッションが「もう終わった」と誤認したと考えられます。

## 解決策

**done.json の書き先をホスト側の確実なパスに固定する**のが根本解決です。

スクリプト内でパスを決定する際、`~/.claude` のようなシェル変数ではなく、`--claude-dir` で明示的に渡したパスを使うように実装します。

```
# extract_sessions.py の --mark-done 処理（設計の考え方）

# NG: スクリプト自身の相対パスや ~ に依存する
done_file = Path("~/.claude/skills/.../daily-summary-done.json").expanduser()

# OK: 呼び出し元から --claude-dir で受け取ったパスを使う
done_file = Path(args.claude_dir) / "skills" / "daily-ai-work-summary" / "daily-summary-done.json"
```

Claude Code CLI から実行する場合は `--claude-dir ~/.claude` を明示的に渡すことで、ホスト側の正しいパスが使われます。

加えて、done.json の設計として **冪等性（同じ処理を何度やっても結果が同じ）** を担保するロジックが重要です。

```
# 冪等な done.json の更新ロジック（概念）
def mark_done(date: str, claude_dir: str):
    done_path = Path(claude_dir) / "skills" / "daily-ai-work-summary" / "daily-summary-done.json"

    # 既存のデータを読む（なければ空リスト）
    existing = []
    if done_path.exists():
        existing = json.loads(done_path.read_text())

    # 追加（重複は除外）
    if date not in existing:
        existing.append(date)

    # 書き込み
    done_path.write_text(json.dumps(sorted(existing), indent=2))
```

重複排除と `sorted()` による正規化を入れることで、複数セッションが並行して書き込んでも壊れにくくなります。

# 3日間のトラブルの整理

| 日付 | 何が起きたか | 根本原因 |
| --- | --- | --- |
| 3/23（1回目） | スクリプト未配置、ホストへのアクセス要求発生 | VM の `.skills` マウントにサブディレクトリが含まれない |
| 3/23（2回目） | ログ解析は成功、Google Drive 書き込み失敗 | VM の FUSE マウントに `G:\` が含まれない |
| 3/24 | 複数セッション起動、「未処理日なし」誤判定 | done.json がVM破棄後に引き継がれない |
| 3/25 | スクリプト未配置だが AI が直接解析で補完、フォールバック保存 | 同上（根本は未解決、フォールバックで対処） |

# 解決策のまとめ

3つの壁それぞれに対する対処をまとめます。

## ① スクリプトを `.skills` 配下に収める

スケジュールタスクに使うスキルフォルダには、必要なファイルをすべてフラットに配置するか、スクリプトを含む構成にします。

または最もシンプルな解決として、**Cowork の Scheduled Task を使わずに OS ネイティブのタスクスケジューラ + Claude Code CLI** に切り替えます。

```
# 移行前（Cowork Scheduled Task）
Cowork VM → .skills/ マウント → python3 extract_sessions.py → 壁①②③

# 移行後（OS タスクスケジューラ + Claude Code CLI）
Windows タスクスケジューラ
  → claude.exe --dangerously-skip-permissions --print "..."
  → ホスト側で直接実行
  → Google Drive も ~/.claude も全部アクセス可
```

## ② Google Drive への保存はホスト側プロセスで行う

Cowork VM からは Google Drive に書けません。Claude Code CLI をホスト側で動かせば解決します。

Cowork でどうしても実行したい場合は、`.claude/` 配下に中間ファイルを書き出す2段階フローを取ります。

## ③ done.json の書き先を明示的に指定する

`--claude-dir` で受け取ったパスを使い、`~` や相対パスに依存しない実装にします。また done.json の読み書きは冪等になるよう設計します。

# おわりに

今回の一番の収穫は「AIが自律的に環境の不備を検知してリソース要求する」という挙動を観察できたことです。スクリプトが存在しない → ホスト側にあるはず → アクセスを要求する、という一連の推論を人間の指示なしに行いました。

これは AI エージェントの可能性として興奮するものがありますが、一方で **無人実行環境では自律的なリカバリーが機能しない** という現実もあります。ユーザーの承認が必要な処理は、誰もいないと止まります。

AI エージェントの自律化を進めるほど、「AIが止まったときに誰も気づかない」「間違った処理が静かに継続される」というリスクも高まります。

設計で気をつけるべきことは、結局のところシンプルです。

* **実行環境の制約を先に調べる**（VMなのか、ホストなのか）
* **外部リソースへのアクセスは事前に確認する**（パスが通っているか）
* **状態管理ファイルの書き先は明示的に決める**（暗黙のパス解決に頼らない）
* **フォールバックを設計する**（完全に動かなくても、データだけは残す）

AIが賢くなっても、インフラの壁はAIの賢さでは乗り越えられません。エージェントを設計する人間が、実行環境の制約を正確に把握する必要があります。

「AIに任せれば全部やってくれる」ではなく、「AIが動ける環境を人間が整える」——これが今の AIエージェント開発の現実だと感じた3日間でした。
