# はじめに

脆弱性診断の前工程として、Webアプリケーションを一通り操作し、診断対象となる画面・機能・HTTPリクエストを洗い出す作業があります。

本記事では、この作業を便宜的に **「画面巡回」** や **「巡回」** と呼びます。

画面巡回では、だいたい次のようなことをサイト全体を見るまで延々と行います。

- 画面を開く
- メニューを押す
- フォームに正常系の値を入れる
- 送信する
- Burp Suite のHTTP履歴を見る
- どの操作でどのリクエストが発生したかを記録する

診断の品質にはかなり効く一方で、地味で時間がかかります。

そこで今回は、**AIエージェント + Playwright MCP + Burp MCP** を使い、AIにWebアプリを巡回させて、診断対象一覧をどこまで作れるか試しました。


# 実施したこと

## 巡回対象
対象は **OWASP Juice Shop** です。

SPAで画面遷移とAPIリクエストが分離しており、画面上のURLだけ見ても実際の診断対象APIが分かりにくいので、今回の検証にはちょうどよい題材だと考えたためです。

## 検証対象
比較したのは以下の3モデルです。加えて、同じ対象に **ZAP Spider + Ajax Spider** も実行し、AIエージェントの結果と比べました。

今回はAIエージェントで実施させるのが主目的ですが、AIエージェントを使わず機械的に実施した場合と比較をしたく、ZAPの通常SpiderとAjax Spiderをコンテキスト認証付きで回しています。

| Claude | Opus 4.7 | Playwright MCP + Burp MCP |
| Codex | GPT-5.5 High | Playwright MCP + Burp MCP |
| ZAP | Spider + Ajax Spider | Spider/Ajax Spider結果を集計 |

## 巡回の流れ

AIエージェントには同じメインの巡回プロンプトを渡し、CSVで結果を出力させました。ZAPは通常Spiderの結果CSVとAjax SpiderのHTTP履歴CSVを後から集計しています。

また、AIエージェントは1回で漏れなく巡回できるとは思っていなかったため、初回実行後に追加プロンプトを何度か投げています。



要するに、**1回で「完了」と言われても鵜呑みにせず、「前回までに見つけたもの」と「今回新たに見つけたもの」を毎回CSVに分けて出させながら、漏れがなくなるまで巡回を繰り返す** 形です。本記事ではこのやり方を「差分クロール」と呼びます。

:::note warn
本記事は厳密なベンチマークではありません。
AIの判断、セッション状態、キャッシュ、ログイン状態、画面表示タイミングなどで結果は揺れます。
「診断前の画面巡回にAIを使うと、実務上どんな癖が出るか」を観察した記録として読んでください。
:::

:::note alert
本検証は自分が管理するローカル検証環境に対して行っています。
許可のない環境に対して同様の操作を行わないでください。
:::

# 使用したプロンプトの方針

プロンプト全文は長いので最後に記載しますが、重要視したのは以下です。

## 1. 画面から辿れるものだけを見る

Juice Shopには既知のエンドポイントがたくさんありますが、今回は **インターネット上の事前知識を使わない** ことを明示しました。
これは、画面から辿れたものを起点にしたブラックボックス寄りの巡回にしたかったためです。

ただし、プロンプト上では対象が Juice Shop であること自体は明示しています。そのため、対象名まで伏せた完全なブラックボックス検証ではなく、**既知エンドポイントや攻略情報に基づく推測を禁止した検証** として見てください。

今回の巡回範囲は、あくまで画面上に表示されたリンク、ボタン、メニュー、フォームから辿れる範囲にしています。

## 2. ブラウザURLではなく、実際のリクエストURLを書く

SPAでは画面上のURLが `/#/basket` のようになっていても、裏では `/rest/basket/6` や `/api/BasketItems/` のようなAPIが発生します。

診断対象として欲しいのは後者なので、プロンプトでは以下のように指示しました。

```text
ブラウザに表示されるURL（/#/...）ではなく、
リクエスト実行時のURL（/api/...、/rest/... 等）を記載する
```

## 3. 遷移方法は操作手順として書く

診断者が後で再現できるように、操作手順を `>` でつないで書かせました。

良い例:

```text
TOP>Account>Login>Log in
ログイン後TOP>Your Basket>Checkout>Add new card>Submit
ログイン後TOP>Open Sidenav>Customer Feedback>Submit
```

悪い例:

```text
ログイン処理
カード追加処理
フィードバック送信
```

後者だと「どこを押せばその処理に到達できるのか」が分からないため、**診断対象一覧**としては扱いづらくなります。

### 4. 機微情報はCSVに含めない

Cookie、JWT、パスワード、CSRFトークンなどはCSVに含めないよう指示しました。

また、今回は巡回であって診断ではないため、SQLi/XSSなどの診断用ペイロードは送らないようにしています。

# 結果サマリ
## 集計
最終的なCSVを集計した結果は以下です。

ここでの「最終検出数」は、各AIエージェントが最終CSVに出力した行数、またはZAPログの行数です。「URL+メソッドのユニーク数」は、CSV/ログ上のURLとメソッドの組み合わせを素朴に集計しています。

「エンドポイント数」は、クエリ文字列を除いたパス単位のユニーク数です。「コミット処理候補数」は、Socket.IOを除いた `POST` / `PUT` / `PATCH` / `DELETE` の行数として数えています。実行済みだけでなく、SonnetのようにURLとメソッドを記録したうえで「実行せず・推定」と備考に残した行も含みます。

| 指標 | Sonnet 4.6 | Opus 4.7 | GPT-5.5 High | ZAP Spider+Ajax |
|---|---:|---:|---:|---:|
| 最終検出数（CSV/ログ行数） | 135 | 139 | 243 | 1255 |
| URL+メソッドのユニーク数 | 118 | 114 | 117 | 138 |
| エンドポイント数（パス単位） | 101 | 90 | 109 | 136 |
| コミット処理候補数（行数） | 32 | 26 | 45 | 0 |
| コミット先エンドポイント数 | 28 | 24 | 8 | 0 |
| 操作経路のユニーク数 | 101 | 117 | 135 | N/A |
| 除外記録（スコープ外として記録された行） | 21 | 11 | 30 | - |

ZAPは操作経路を生成しないため、操作経路のユニーク数は比較対象外として `N/A` にしています。

ぱっと見ると、ZAPの1255行とGPT-5.5 High の 243件 が多く見えます。

ZAPはURL+メソッドのユニーク数が138件で、これはAI 3モデルより多いです。ただし、その増分の多くは `/ftp` 配下や静的/ディレクトリ探索由来のURLで、画面操作による業務フローとは性質が違いますし、診断対象としては不十分です。

Sonnet 4.6 が Opus 4.7 よりエンドポイント数とコミット先エンドポイント数で多いのは、FTP配下、`reset-password`、`user update`、`continue-code`、レビュー更新、クーポン適用、ファイルアップロードなど、周辺機能や高リスク寄りの処理を広めに記録していたためです。一方で、Sonnetには `{id}` のようなプレースホルダURLや「実行せず・推定」と書かれた行も混ざるため、そのまま実行済みリクエスト数として読むのではなく、診断対象候補として見るのがよさそうです。

一方GPT-5.5 High は「他のモデルより多く巡回ができた」というより、**商品詳細やAdd to Basketなどを商品単位で細かく記録したため行数が増えた** という見方が近いです。

コミット処理候補数でも同じ傾向が見えます。GPT-5.5 High はコミット処理候補の行数が45件で最多ですが、コミット先エンドポイント数は8件です。つまり、書き込み系の種類が圧倒的に多いというより、同じ種類の処理を商品IDやBasketItem IDごとに細かく確認していた、と読むのが自然です。

## 各結果からいえること
これらの結果を見ると、AIエージェントに巡回を任せる場合は、初回で完了と判断せず、追加確認を何度か行うことがかなり有用だと分かります。

# モデル別の所感

## Sonnet 4.6
### 広く拾うが、CSVの形が少し独特

Sonnet 4.6 は、機能の横幅が広いです。

- reset-password
- user update
- review POST/PUT
- Web3/BEE Haven 関連

一方で、CSVとしては少し癖がありました。

本来 `遷移方法` 列には `TOP>Account>Login` のような操作経路を書いてほしかったのですが、Sonnetは `自動(画面表示)` や `ボタンクリック` のような種別を入れていました。

ただし、実際の操作経路は `フォーム/ボタン名` 側にかなり入っています。なのでそのままだと機械的な比較や統合がしづらい、という話です。

Opus 4.7 は、3モデルの中で一番人間が読みやすいCSVを作りました。

# Burpログと突き合わせてみた

AIが出したCSVだけを信じるのは怖いので、Burpログとも突き合わせました。

ここで言う「動的リクエスト」は、画像・CSS・JS・フォントなどの静的アセットを除いた、APIや画面遷移に伴うリクエストを指します。

| 指標 | Sonnet 4.6 | Opus 4.7 | GPT-5.5 High |
|---|---:|---:|---:|
| Burpログ総行数 | 647 | 632 | 658 |
| 同一ホスト行数（Juice Shop宛のみ） | 645 | 631 | 658 |
| 動的リクエスト行数（静的アセット除外後） | 302 | 297 | 316 |
| BurpにあるがCSVにない（CSV取りこぼし） | 13 | 7 | 2 |
| CSVにあるがBurpにない（CSV側の幻 or 経路ずれ） | 26 | 2 | 3 |

この観点では、GPT-5.5 High が最もBurpログと近かったです。

ただし、これは「探索範囲が狭めだったため差分が小さい」とも読めます。

OpusはCSVの読みやすさとBurpログ一致性のバランスが良く、SonnetはCSV側にBurpで照合できない候補が多めでした。

# やってみて分かったこと

## 1. AIエージェントは「画面巡回」にある程度は有用

少なくとも、ログインして、メニューを開き、フォームに値を入れ、送信し、発生したリクエストをCSVにする、という流れは十分に任せられました。

手作業でゼロから台帳を作るよりは、かなり楽になります。

## 2. ただし、1回では漏れる

これはかなり重要です。

1回目で「完了しました」と言っていても、追加で確認させると普通に増えます。

今回もどのモデルでも最終的には増えました。

AIに任せるなら、最初から **何回か再確認させる前提にする** べきだと思います。

## 3. 「検出数が多い = 網羅性が高い」ではない

GPT-5.5 High は243件で最多でしたが、行数だけで網羅性は判断できません。

中身を見ると、GPT-5.5 High は商品詳細やAdd to Basketを商品単位で細かく拾っていたため、件数が大きく増えていました。これは有用ではありますが、「機能の横幅が一番広い」とは別の話です。

見るべきなのは、単純な行数だけではなく、

- URL+メソッドのユニーク数
- エンドポイント数
- 機能領域別の件数
- コミット系メソッドの件数
- コミット先エンドポイント数
- Burpログとの一致
- 未実行行の扱い

あたりです。

## 4. モデルごとに得意な巡回の癖が違う

同じプロンプトを投げても、結果はかなり違いました。

- Sonnet 4.6 は、Score Board、FTP、continue-code など横方向の探索が強い
- Opus 4.7 は、操作経路が読みやすく、診断対象一覧のベースとして使いやすい
- GPT-5.5 High は、商品/バスケット系をID単位で細かく拾う

今回の結果を見る限り、1モデルだけで完璧な診断対象一覧を作るのは難しそうです。

また特定のモデルが駄目と言った結果でもなく、それぞれ一長一短であることから、組み合わせを行うとより良い結果が得られそうです。

# まとめ
今回の検証では、

- Sonnet 4.6 は機能の横幅が強い
- Opus 4.7 は診断台帳として読みやすい
- GPT-5.5 High は商品/バスケット系と差分管理が強い
- ZAPは通常SpiderでFTP/ディレクトリ探索が伸びるが、今回の設定では業務フロー巡回は弱い

という結果でした。

今後やるなら、次は以下を試したいです。

- モデルの組み合わせた巡回の実施

# 実運用を考えてみて

AIエージェントに「巡回作業そのもの」を**全任するのはまだ慎重に考えるべき**ですが、少なくとも **巡回の下書き作成** としては、かなり実用性があると感じました。
下書きとして使うだけでも、0から作る必要がないわけですから、すべて手動で巡回を行うより圧倒的に楽になります。


# プロンプト
以下に今回の検証で使用したプロンプトを記載します。
前記の重要視したことを手で書き、ChatGPT5.5 Thinkingに肉付けさせて作成しています。
## メインプロンプト
````
Playwright MCP, BurpMCPを使って、以下のWebアプリケーションを巡回してください。

## 目的
- 脆弱性診断の観点で、このWebアプリケーションを診断するうえで診断対象になるすべての処理の遷移方法およびリクエストが知りたいです。
- 巡回結果をCSV形式でファイル出力する
- サイト内のすべてのリクエストが知りたいです。

## 起点URL & スコープ
- http://192.168.1.33:3000/

## 必須事項
- これはJuiceShopですが、絶対にインターネットなどの事前知識を使わないでください。
- 必ず画面から辿れると確認したボタン/リンクに限定して巡回を行ってください。
- ブラウザのタブに表示れているURLではなく、ボタン/リンク実行時のリクエストのURLが欲しいです。
- ただしブラウザのタブに表示されるURLとボタン/リンク実行時のリクエストが同一の場合はそのURLで問題ありません。

## 巡回ルール
- ログイン画面が表示されたら、以下の認証情報でログインを試行する
  - username: a@a.com
  - password: password
- TOP画面を「TOP」として扱う
- ログイン後の画面は「ログイン後TOP」として扱う
- ログインの成功関わらず、ログインしなくても見れる箇所については巡回する。
- ログインに成功したら、ログイン後の画面からも巡回を続ける
- ログインに失敗した場合は、失敗した旨を記録し、ログイン前に見える範囲のみ巡回する
- スコープ外のURLにはアクセスしない
- 画面上のリンク、メニュー、フォーム、ボタンを確認する
- JavaScriptで表示される要素も可能な範囲で確認する
- スクリーンショットは不要
- 送信、変更、削除などのコミット処理は行って問題ない
- 入力値は正常系の無害な値のみ使用してください
- ファイルアップロードがある場合は、無害なテキストファイルや画像ファイルなど、適宜ファイルを用意して試してください
- Cookie、セッションID、パスワード、トークン、CSRFトークン等の機微情報はCSVに含めないでください
- その他エラーは再度試し、うまくいかないと判断した場合はその旨を記録してください
- 進めなかった場合は遷移方法を記載し、URLは「未実行」と、備考欄に進めなかった理由を書いてください
- 画面からたどれない画面については見る必要はありません
- 脆弱性診断ではなく巡回なので、決して診断用シグネチャは送信しないでください
- SPA（Angular等）でハッシュルーティングのアプリの場合、画面遷移ごとに発生する `/api/...`、`/rest/...` 等のバックエンドAPIコールも漏れなく1行ずつ記録すること

## 出力形式
- 最終結果はMarkdownの表ではなく、CSV形式でファイルに出力する
- ヘッダは次にする
  `遷移方法,URL,メソッド,遷移種別,フォーム/ボタン名,ステータスコード,備考欄`
- その他、脆弱性診断者目線で有益な情報があれば、上記ヘッダの右側に列を追加してよい（例: 認証要否、入力データ概要 等）
- 巡回が終わったら、発見した処理一覧をCSV形式でファイルに出力する。画面にわざわざ表示しなくて良い

## 遷移方法の命名ルール
- **画面に表示されている文言（ボタン名・リンク名・メニュー名・画面タイトル）をそのまま使う。日本語に訳さないこと。**
- 階層は「>」で表現する
- 起点は「TOP」とする
- ログイン後トップは「ログイン後TOP」とする
- **遷移方法はユーザーが押したボタン/リンクの軌跡をそのまま連結する**
  - 例: TOP画面で `A` を押したら → `TOP>A`
  - その画面で `B` を押したら → `TOP>A>B`
  - さらに `C` を押したら → `TOP>A>B>C`
  - 例えばTOPから`Login`を押下すると → `TOP>Login`
- 同じ画面で発生する複数のAPIコールはそれぞれ1行ずつ書き、遷移方法は到達した時点の経路で揃える
- 遷移方法が違うが画面が同じものは原則1行にまとめる
- ただし裏の処理が異なっていそうであれば別行として記載する
- クエリパラメータ違いで機能が異なる場合は別行にしてよい
- アンカー（`#`以降のハッシュ）のみ違うURLは同一画面として扱う
- 1遷移で複数リクエストが取れる場合は遷移方法をそのままに、URL/メソッドを別行で書く
- 未実行の場合も、遷移方法は単独の名前ではなくボタン押下までの経路として記載する
  例: `ログイン後TOP>Account>Privacy & Security>Request Data Erasure>Delete User Data`

## URLの命名ルール
- ブラウザに表示されるURL（`/#/...`）ではなく、リクエスト実行時のURL（`/api/...`、`/rest/...` 等）を記載する
- URLのみを記載し、その他情報は含めない
- 未実行の場合は「未実行」という文字列のみ記載し、URL自体は推測しない

## メソッドの命名ルール
- GET、POST、PUT、DELETE等のHTTPメソッドのみ記載する

## 遷移種別の例
画面表示 / リンク押下 / メニュー押下 / フォーム送信 / ボタン押下 / JavaScript操作 / ファイルアップロード / リダイレクト / 非同期通信

## 備考欄に記載してほしいこと
- ログイン成功/失敗
- 入力した正常系データの概要
- 遷移できなかった理由
- 同一画面だが別処理と思われる理由
- JavaScriptにより発生したリクエスト
- リダイレクトが発生した場合の概要
- 401/403/404/500等のエラーが発生した場合の概要
- 診断時に確認対象になりそうな処理の補足（IDOR / マスアサインメント / SSTI / SSRF / Open Redirect / NoSQLi / CSRF 等の観点）
- ただし、Cookie、セッションID、パスワード、トークン、CSRFトークン等の機微情報は含めない

## 最終出力例
```
遷移方法,URL,メソッド,遷移種別,フォーム/ボタン名,ステータスコード,備考欄
TOP>ログイン,http://192.168.1.38/login.php,GET,画面表示,,200,ログイン画面
TOP>ログイン>Log in,http://192.168.1.38/login.php,POST,フォーム送信,Log in,302,ログイン成功。認証情報やトークンは記録しない
TOP>ログイン後トップ,http://192.168.1.38/index.php,GET,画面表示,,200,ログイン後トップ画面
TOP>ログイン後トップ>Customer Feedback,http://192.168.1.38/rest/captcha/,GET,非同期通信,Customer Feedback,200,Customer Feedback画面表示時に発生したCaptcha取得
TOP>ログイン後トップ>Customer Feedback>Submit,http://192.168.1.38/api/Feedbacks/,POST,フォーム送信,Submit,201,フィードバック送信
```

## その他
- アクセスは行ったがcsvファイルに記載しない場合、別のcsvファイルを用意し、同様のフォーマットに記載してください。
- その際に除外理由を必ず記載してください。
````
