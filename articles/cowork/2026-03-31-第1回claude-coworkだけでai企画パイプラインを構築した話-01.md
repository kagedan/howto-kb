## はじめに

新しいサービスのアイデアを思いついたとき、あなたはどうやって企画を固めていくだろうか。

メモを書く、競合を調べる、資料を作る、誰かにレビューしてもらう——これを一人でやると時間がかかるし、どうしても視野が狭くなる。

そこで試してみたのが**Claude Coworkを使ったAI企画パイプライン**だ。複数のAIエージェントに役割を持たせ、バトンを渡しながら企画の精度を上げていく仕組みを作った。

この記事では、そのパイプラインの設計と運用について記録する。

## パイプラインの全体像

企画フェーズに必要な役割を整理すると、こうなる。

```
Researcher → Strategist → Prototyper → Writer → Reviewer
```

これをそのままCoworkのプロジェクトとして作成した。

プロジェクト役割アウトプット01\_Researcher市場・競合調査01\_research.md02\_Strategist事業設計02\_strategy.md03\_PrototyperUIデモ作成03\_demo.jsx04\_Writer企画書作成04\_企画書.pptx05\_Reviewerフィードバック05\_feedback.md

## フォルダ構成とバトンの渡し方

各エージェントはフォルダ内のファイルを読んでアウトプットを保存する。人間はバトンを渡すだけ。

```
ProjectName/
├── [PJ_NAME].md     ← プロジェクト定義（人間が書く）
├── CLAUDE.md        ← プロジェクト名を定義
├── 01_research.md   ← Researcherのアウトプット
├── 02_strategy.md   ← Strategistのアウトプット
├── 03_demo.jsx      ← Prototyperのアウトプット
├── 04_企画書.pptx   ← Writerのアウトプット
└── 05_feedback.md   ← Reviewerのアウトプット
```

サブフォルダは作らず、フラットな構成にすることで各エージェントがファイルを見つけやすくなる。

## 汎用設計の肝：[PJ\_NAME]という考え方

インストラクション内のプロジェクト名を[PJ\_NAME]というプレースホルダーにしておく。そしてCLAUDE.mdでプロジェクト名を定義する。

**CLAUDE.mdの例**

```
このプロジェクト名は〇〇です。
各インストラクションの[PJ_NAME]は〇〇に読み替えてください。
プロジェクト定義ファイルは〇〇.mdです。
```

これだけで**新しいPJはCLAUDE.mdを書き換えるだけで同じパイプラインが使える**。

## 各エージェントのインストラクション設計

柔軟性を持たせるために、各エージェントは「ファイルが存在する場合」という条件付きで読み込む設計にした。

**例：02\_Strategistのインストラクション**

```
あなたは事業企画の専門家です。

【進め方】
1. [PJ_NAME].md を読み込む（存在する場合）
2. 01_research.md を読み込む（存在する場合）
3. 利用可能な情報をもとに企画を設計する

【アウトプット】
02_strategy.md としてプロジェクトフォルダ直下に保存する
```

「存在する場合」という条件を入れることで、**どのエージェントも単独で動ける**。必ずしもフルパイプラインを通す必要がなく、途中から始めたり、特定のステップだけ繰り返したりできる。

## パイプラインのパターン

状況に応じて使い分けられる。

```
フルパターン：
[PJ_NAME].md → Researcher → Strategist → Prototyper → Writer → Reviewer

調査スキップ（調査済みの場合）：
[PJ_NAME].md → Strategist → Writer → Reviewer

企画書のみ：
[PJ_NAME].md → Writer

レビューのみ：
既存ファイル → Reviewer
```

## 複数回パイプラインを回す

一度回して終わりではなく、Reviewerのフィードバックを受けて[PJ\_NAME].mdを更新し、また回す。これを繰り返すことで企画の精度が上がっていく。

```
[PJ_NAME].mdを更新
　　↓
パイプラインを回す
　　↓
Reviewerがフィードバック
　　↓
[PJ_NAME].mdを更新 → 2周目へ
```

毎回Reviewerが新しい視点を持ち込んでくれるので、人間一人で考えるより確実に精度が上がる。

2周目・3周目は変更がないステップをスキップできる。たとえば調査内容が変わっていなければResearcherをスキップし、Strategist以降だけ回せばいい。

## CoworkとCodeの使い分け

このパイプラインはCoworkをメインに使っているが、将来的にはCodeで自動化することも想定している。

やること向いているツールドキュメント読み書き・資料作成Coworkコード生成・実行・Git連携Codeパイプライン全体の制御（自動化）Code

現時点では**人間がバトンを渡す手動パイプライン**だが、Codeをオーケストレーターとして使えば完全自動化も可能になる。

## やってみて分かったこと

**良かった点**

役割が明確だから指示が楽。「Researcherに調査させる」「Reviewerにレビューさせる」という明確な役割があるので、何を頼めばいいかが直感的にわかる。

汎用パイプラインとして使い回せる。[PJ\_NAME]という設計のおかげで、次のプロジェクトでも同じパイプラインがそのまま使える。テンプレートの資産が積み上がっていく感覚がある。

**注意点**

Coworkはまだローカル保存のみ。プロジェクトのファイルはPC上にローカル保存される。複数PCでの共有やクラウド同期は現時点では非対応。

各プロジェクトのフォルダ指定を確認する必要がある。起動時に**[PJ\_NAME]**フォルダを認識しているか確認が必要な場合がある。

## まとめ

Claude Coworkで企画パイプラインを構築するポイントをまとめると：

1. **役割を分ける**：Researcher・Strategist・Writer・Reviewerの4役が基本
2. **フラットなフォルダ構成**：サブフォルダを作らず直下にファイルを置く
3. **[PJ\_NAME]で汎用化**：CLAUDE.mdでプロジェクト名を定義する
4. **条件付き読み込み**：「存在する場合」にすることでどのエージェントも単独で動ける
5. **複数回回す**：Reviewerのフィードバックを受けて精度を上げていく

このパイプラインは企画フェーズだけでなく、開発・運用・改修フェーズにも同じ設計で展開できる。
