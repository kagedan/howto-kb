# 人は替わる。知は残る。——5体のAIエージェントで「春のデスマーチ」を終わらせる

ERPが届かない判断の消失と修正の連鎖懸念を、監査証跡として組織に刻む。

> Microsoft Agent Hackathon 2026 個人部門 提出作品

## はじめに

納期直前、暗くなったオフィスで、ミスで3時間の手戻りが発生する。  
エンジニアにおけるデスマーチが、春の経理部でも発生しています。  
連結決算という名前の、もう一つのデスマーチです。

ConDuctorは、ピークの4日間を **数分** で終わらせます。5体のAIエージェントが86ルールを並列検証し、3体合議で判断の質を担保します。

* **何を解決するか**: 連結決算におけるミスの連鎖防止 / 担当者交代による判断消失

## 1. 桜が咲く季節に、もう一つのデスマーチがある

> **春のデスマーチ。**
>
> 毎年4月。三月末で締めた決算の、連結処理が始まる。数十社ぶんを束ねる、その最後の山場。連結決算、ピークの4日間。連日の残業。
>
> 桜も、決算も、誰の都合も待ってはくれない。結婚式の前夜に帰れなかった上司がいた。娘の寝顔の代わりに、整合性チェックの画面と向き合った担当者の夜があった。1件の不整合で、開示が止まる——国が決めた期限の前では、個人の春は後回しになる。

**連結決算とは何か。** 親会社が数十〜百社の子会社の財務データをまとめ、「グループ全体の1枚の決算書」を作成する作業だ。エンジニアに言い換えれば、60個の独自スキーマを持つデータソースを変換・統合し、会社間の内部取引（内部売上・貸付金）を消去して、最終的に1つの一貫したレポートを生成するETLパイプラインに近い。ただし、整合性チェックが1件でも落ちれば開示が止まる。期限は国が決めており、動かせない。

作者が最も過酷な事例として記録されているのは、数十社分の連結パッケージを4日間・数名で、残業・休日返上でこなした年度末だった。結婚式を翌日に控えた上司が、帰れなかった。確認が1件でも漏れれば開示が止まる——国が決めた期限の前では、個人の人生は後回しになる。

これは過去の話ではない。今この瞬間も、全国の決算担当者が同じ地獄を繰り返している。

* 開示期限から逆算された日程は動かせない
* 確認が1件漏れれば数時間のやり直し
* 修正仕訳を切ればPL・CF・利益剰余金・税効果が全部変わる
* 修正開示には至っていないとしても、いつ踏んでもおかしくない地雷の上を、毎期歩いている

速く、正しく、正確な数値を開示せよ——国からの圧力はますます強まっている。疲弊がミスを呼び、ミスが自信を削り、自信の喪失がさらなるミスを招く。この悪循環を、個人の努力で断ち切ることはできない。

**この4日間の終わりなき行軍を——ConDuctorは数分で終わらせる。**

## 2. 開発体制——公認会計士から意見を受けた設計

ConDuctorは「個人が一人で作ったツール」ではない。**公認会計士と共同で設計したシステムだ。**

| 役割 | 担当 | 貢献領域 |
| --- | --- | --- |
| **開発者** | 連結会計の知見 | 業務フローの正確な定義・86ルールの妥当性確認・「春のデスマーチ」の問題意識が設計の原点 |
| **公認会計士** | 監査経験を持つ認定資格者 | 合成データの監査妥当性担保・ExpertAuditLayerのペルソナ設計・閾値変更の承認プロセス設計 |

この共同設計は、下記の3つの問いを重視している。

* **責任ある実装**: 公認会計士が監査妥当性を担保している。「AIが提案し、公認会計士が承認した」という証跡が全ての閾値変更に残る設計——責任の所在を明確にする。
* **誰のどの課題を解くか**: 連結決算の現場課題から逆算して設計した。PoCデモツールではなく、現場の痛みを起点に設計した。
* **「人間の検証責任が設計に組み込まれているか」**: 最後は人間のレビューが設計の核にある。`閾値Advisor`の承認フローはこの体制があって初めて意味を持つ。

## 3. ConDuctorは何を変えるのか

| 項目 | Before（手作業） | After（ConDuctor） |
| --- | --- | --- |
| **数十社の検証** | **4日間（数名×22時残業）** | **数分（撤退戦の終結）** |
| **対応規模** | 人手の限界（少数社） | 数十社（AI対応・スケーラブル） |
| **必要人員** | 5名以上 | 1名 |
| **消去漏れの検出** | 人の疲労次第 | SG-3が自動検出 |
| **修正の波及把握** | 手動で30分以上 | 即時（伝播グラフ） |
| **退社時間** | 22時〜深夜 | 定時 |

ConDuctorは「効率化ツール」ではない。既存ツール（DivaSystem、STRAVIS等）はデータの配管（pipe）だ。データを流すことはできるが、判断は人間に丸投げする。

ConDuctorが目指すのは判断を育てる器（vessel）だ。

製造業では、ベテラン職人の暗黙知をAIがナレッジグラフとして蓄積し、次の世代に継承する技術が実用化されている。医療でも、診断パターンの蓄積と継承は研究の最前線だ。

会計にも同じ課題がある。だが誰も手をつけていない。

「A001社は毎期、売掛金の回転期間が長い」「B003社は現地の会計基準がIFRSでないから、数字が合わない」——こうした子会社ごとの個性は、担当者の頭の中にしかない。担当者が異動すれば消える。監査法人がローテーションで交替すれば、新任の会計士は一からヒアリングをやり直すことになる。聞かれる側も、聞く側も、本来しなくていい作業に時間を費やす。

ConDuctorはこの判断パターンを構造的に蓄積し、継承する。人が替わっても、知は残る。

セクション1で述べた悪循環——疲弊がミスを呼び、ミスが自信を削る——の根本原因は、人間がしなくていい判断まで背負っていることにある。ConDuctorは、人間がしなくていい判断を構造的に減らしていく。認知資源を守り、本当に人間の目が必要な判断だけに集中できる環境を作る。

「削減工数はどこへ行くのか」——ConDuctor導入後の想定は3段階だ。**即効**: 確認・転記・子会社問い合わせメール作業からの解放。**短期**: AIが絞り込んだ重要フラグ20件の深い判断に集中（以前の10倍の時間を使える）。**中長期**: PriorKnowledgeStoreが前任の判断ロジックを引き継ぐため、異動直後のオンボーディングが短縮し、その時間を経営分析・ボードレポート作成へ充てられる。省力化ツールではなく、人の認知資源を高付加価値領域へ**再配置**する設計だ。

### AI と人間の協働思想

AIが最大の価値を生むのは、日常ワークフローに統合されたときだといえる。「問題に最も近い人が解決ツールを持てるとき、イノベーションは拡張する」——この言葉はConDuctorの設計根拠そのものだと考えている。

ConDuctorにおける目的は一つだ: 認知過負荷の悪循環から解放された経理担当者が、本来の判断業務に集中できること。春のデスマーチを終わらせることは結果であり、目的ではない。解放された時間と認知資源で、「経営を動かす経理」へと転換することだ。これが AI 活用変革の道筋だ。

## 4. なぜ2026年なのか——3つの技術的収束点（A1-3）

この開発は1年前には不可能だった。

連結決算の「判断」を自動化するには、3つの技術的条件が同時に揃う必要があった。2026年、初めてそれが実現した。

### ① Azure OpenAI gpt-4o (GlobalStandard SKU) の安定運用

退職給付の割引率変動PBO（EX-PEN-01）や移転価格の合理性判断（EX-TAX-07）は、会計基準の解釈が必要な「専門家判断ルール」だ。1年前のLLMでは、この精度は出なかった。Azure OpenAI gpt-4o GlobalStandard SKU の200K TPM/min レート上限で、合成データ数十社の検証を数分で完走できるようになった。

### ② Semantic Kernel SDK のGA（2025年11月）

86ルールをSKプラグインとして宣言的に登録できる基盤が本番品質になった。「`conductor_rules.xlsx`に追記するだけで新ルールが動く」という設計は、SKのGA後に初めて実装可能になった。それ以前は、ルール追加のたびにControllerのコードを改修する必要があった。

### ③ Azure AI Foundry の一般提供

エンタープライズ向けのエージェント管理基盤が整った。Connected Agentsによる独立スレッド＋サービス側ステート管理は、判断履歴の蓄積（PriorKnowledgeStore）を現実的なコストで実現する。いくらエージェントが優秀でも、APIコストが膨大であれば、実務で採用することはできない。

## 5. 従来の会計システムと異なる仕組み——3本柱の防御壁（A1-4）

Divasystemなど、連結決算の「配管（pipe）」の領域で成熟した競合はいくつもある。だがConDuctorが持つ3つの要素は、単なる後発では追いつけない組み合わせを生んでいる。

### 柱1: Append-Only Trigger——判断の証跡が消えない

すべての閾値変更・判断記録は物理削除不可能なログとして蓄積される。「なぜその数値をOKにしたのか」の証跡がシステムに内蔵されている。HFMはデータを流す。ConDuctorは判断を記録する。

公認会計士が承認した変更履歴は、監査法人への証拠として機能する。「責任ある実装」の思想は、このAppend-Only設計から来ている。

アプリケーション層の禁止だけでは不十分だ。**SQL Trigger**（DBレベルの自動制御: UPDATE/DELETE命令が来た瞬間にDBが自動拒否）でデータベース層から強制する:

```
CREATE TRIGGER IF NOT EXISTS prevent_revision_update
    BEFORE UPDATE ON revisions
BEGIN
    SELECT RAISE(ABORT, 'revisions テーブルへの UPDATE は禁止されています（Append-Only）');
END;
```

参照: `docs/adr/0001-append-only-trigger.md`（ADR として設計判断を永続記録）

過去の判断を、書き換えようとする。修正を試みる。すると——拒否される。仕組みそのものが、上書きと削除を、物理的に止める。

**過去は、書き換えられない。**

これは機能の話ではない。設計の段階で腐敗を防ぐ、哲学の話だ。AIは便利なだけではいけない。監査に耐えられなければならない。では、どうするか——元の記録は残したまま、逆仕訳を追記する。傷つけるのではなく、直した事実を、記録として積み上げる。そうして監査証跡が完結する。

### 柱2: 3体合議＋少数意見保存——消えない論点

`dissent_reasoning`の永続保存は、多数決では消えるはずの論点を残す。3体全員がOKと言っても、1体が指摘した論点が後から正しかったということは実務では起きる。競合他社の多数決ロジックでは少数意見は統計的ノイズとして捨てられる。ConDuctorだけが、それを「将来の重大リスクの予告」として扱う。

### 柱3: PriorKnowledgeStore——使えば使うほど賢くなる

数十社×複数年データ＋86整合性ルール＋CPA知見の組み合わせが、ナレッジグラフとして蓄積される。「A001社は毎期、売掛金の回転期間が長い」「B003社の会計基準はIFRS以外」——こうした子会社ごとの個性が担当者の頭ではなくシステムに残る。後発参入者が同等のナレッジを持つには少なくとも3〜5年の実運用が必要だ。

**IFRS 2層 governance設計**: ConDuctorは「連結パッケージ = グループ共通基準（IFRS任意適用グループはIFRS）」と「子会社個別財務 = 現地会計基準（日本GAAP・HGB・タイ会計基準等）」の2層を明示的に区別して検証する。今回仮で作成したSakura Groupのような「IFRS任意適用の親会社 + 現地基準の子会社」という構造では、為替換算差額（R032）や包括利益（EX-CPA-01）の判断において適用基準の違いを考慮しなければならない。PriorKnowledgeStoreはこの「会社×適用基準」の組み合わせを記憶し、期をまたいで引き継ぐ。

## 6. 数十社から1000社へ——スケーラビリティ設計（A1-5）

現在の合成データ数十社で計測した実績から、1000社対応への設計根拠を示す。

### 現在の計測値（合成データ数十社）

| 指標 | 実測値 | 備考 |
| --- | --- | --- |
| 総処理時間（数十社） | **数分以内** | asyncio並列処理（実測: ウォーム約13秒・初回約56秒） |
| 1社あたり平均 | 〜0.5秒 | LLM API待ちを含む（実測中） |
| 同時処理エージェント数 | 3体（Alpha/Beta/Gamma） | asyncio.gather |
| 月額インフラコスト | **$10以下** | App Service B1 |

※ 実測値は合成データ環境での計測。実環境データは実測中。

### 1000社対応の計算量見積もり

1000社 × 30ルール = **3万判定/期**を1時間以内で処理する設計根拠:

```
現行: 数十社 × 0.5秒/社 ÷ 並列度3 = 10秒（Pythonルール）
      + LLM判断ルール20件/社 × 数十社 × 平均0.3秒 = 数十秒 → 合計 数分以内

1000社想定:
  並列度を20に引き上げ → 1000 × 0.5 ÷ 20 ≈ 25秒（楽観値）
  LLMレート制限考慮（200K TPM/min）→ キャッシュ戦略で対応
  同一ルールIDの重複呼び出しをキャッシュ化 → LLM呼び出し60%削減見込み
  結果: 1000社処理 ≈ 15〜45分（インフラスケール次第）
```

ボトルネックはLLM APIのレート制限。Azure OpenAI GlobalStandard SKUへの移行で2〜4倍の処理能力増強が可能。

### インフラ移行ロードマップ

| フェーズ | 対象規模 | 変更点 |
| --- | --- | --- |
| **現状** | 数十社 | SQLite + App Service B1（$10/月） |
| **Step 1** | 〜200社 | App Service P1V2へスケールアップ。SQLiteのまま |
| **Step 2** | 〜500社 | PostgreSQLへ移行。Redisセッションキャッシュ追加 |
| **Step 3** | 〜1000社 | App Service P2V3 + DB分離（Cosmos DB検討） |
| **Phase 2** | 1000社超 | Azure AI Foundry移植。エンタープライズスケール |

SQLite→PostgreSQL移行のトリガー: 判断ログの累積件数10万件超過時点を目安にしている（現在の数十社換算で、約1.5〜2年の運用後を想定）。

3体合議の並列処理（asyncio.gather）は処理対象が増えても設計変更なしにスケールする。スケーラビリティのボトルネックはDBとLLM API、オーケストレーションロジックではない。

## 7. 情報を信じる前に格付けする

連結決算の検証で最も恐ろしいのは、見つかったエラーではない。**気づかれなかったエラー**だ。

書類に書かれていないこと。検討されなかった項目。それが半年後に修正開示として返ってくる。見つかったエラーは修正すれば済む。気づかれなかったエラーは、発覚したときには手遅れだ。

この問題に対して、会計の世界には方法論がなかった。情報にも鮮度がある。昨日のExcelと今日のExcelでは信頼度が違う。だがその「信頼度の差」を構造的に扱う方法論があったのは、軍事の世界だった。

### Admiralty Code——情報の信頼性を構造的に評価する

NATO軍が使う情報評価体系「Admiralty Code」は、すべての情報を2軸で格付けする。

* **情報源の信頼性**（A: 完全に信頼できる 〜 F: 判定不能）
* **情報の確度**（1: 確認済み 〜 6: 判定不能）

![Admiralty Code 4分類 — 情報源信頼性 × 情報確度の2軸格付け](https://static.zenn.studio/user-upload/deployed-images/772f7f05bb0130f47caec8fd.png?sha=4807fee666a67309bfb5ed923813e2a0ccff3423)

|  | 1: 確認済み | 2: おそらく正しい | 3: ありうる |
| --- | --- | --- | --- |
| **A: 完全に信頼** | A1 | A2 | A3 |
| **B: 通常信頼** | B1 | **B2 ← Excel直接計算** | B3 |
| **C: やや信頼** | C1 | C2 | **C3 ← LLM推定** |
| **F: 判定不能** | F1 | F2 | F3 |

ConDuctorは、すべてのフラグにAdmiralty Codeを付与する。「Excelの数値から直接計算した結果」はB2（信頼できる情報源から、おそらく正しい）。「LLMが文脈から推定した結果」はC3（やや信頼できる情報源から、ありうる）。同じ「フラグ」でも、その根拠の質が全く異なることを、コードが可視化する。

### 不在検出——テストが落ちたのではない。テストが書かれていなかった。

AIが「ある」ものを検出するのは容易い。パターンマッチ、閾値チェック、異常値検出——すべて「存在するデータ」に対する操作だ。

ConDuctorのQuartermasterは因果分析モジュールを内蔵し、**「存在しないデータ」を検出する**。現在12パターンの不在を構造的に監視する。

> 「回収可能性チェックが、影響先に含まれていません」

このアラートが意味するのは、誰かが見落としたことではない。**そもそも誰も検討していなかったこと**だ。

本当に怖いのは、テストケースが存在しないバグだ。連結決算でも同じことが起きている。検討されなかった項目が、半年後に修正開示として返ってくる。

**存在しないものの中にこそ、最も重要な情報がある**。ConDuctorはこの不在を、構造的に検出する。

### 少数意見を消さない——3体のエージェントのうち1体だけが正しかった

ConDuctorの3体の検証エージェント（Alpha・Beta・Gamma）が合議した結果、2体がクリア、1体がフラグを立てたとする。多数決ならクリアだ。

しかし、1体が指摘した根拠は**削除しない**。`dissent_reasoning`として永続保存する。

なぜか。「3体のエージェントのうち2体がOKと言ったが、1体が指摘した論点が後から正しかった」ということは、現実に起きるからだ。最高裁判所の反対意見が、数十年後に多数意見を覆すように。少数意見の中にこそ、将来の重大リスクが眠っている。

では、なぜ5体なのか。なぜ1体ではいけないのか。

## 8. 5体のエージェント——なぜ1体では駄目なのか

### 設計の出発点: 単一エージェントを最初に検討した

マルチエージェントは「それが必要だから」ではなく「それしかなかったから」設計するものではない。Microsoft が定義するマルチエージェント採用条件（Cloud Adoption Framework "Single-Agent vs. Multi-Agent"）は3つに絞られている:

1. **セキュリティ・コンプライアンス境界を越える**（データ分離義務がある）
2. 複数チームが個別のドメインを管理する（組織境界を反映）
3. 将来の成長が3つ以上の異なる機能ドメインに跨る

ConDuctorの設計プロセスでも、最初は単一エージェントを検討した。86ルールを1体に渡して検証させるアプローチだ。だが条件1が突破できなかった: **Controller（実行）と Alpha/Beta/Gamma（検証）の職務分離**は、会計監査の基本要件だ。実行と検証を同一エージェントが担えば、「自己監査」になる。同じ理由で、財務諸表監査は作成者と監査人を必ず分離する。

ConDuctorにおけるマルチエージェントは、機能の「分散」ではなく「分離」の必要性から来ている。ConDuctorは職務分離という要件を根拠にマルチエージェントを採用した。

**単一AI自己検証バイアスの根拠**: 1体のAIが「自分で検証して自分で判断する」構造は、自己監査に内在するバイアスを排除できない。単一LLM構成は誤検出を自分では打ち消せないが、独立した3体が別々のルール群を担当し全員一致でのみフラグを立てる合議構造では、1体の誤検出を他の2体が補い合う。具体的な誤検出率・見落とし率は合成データ環境で検証中であり、公開時に随時更新する。合議によって単一構成より誤検出を減らせることが、監査の信頼を生む。

**Agentic AIの4要素 — ConDuctorの実装マッピング**:

| Agentic Property | ConDuctor の実装 |
| --- | --- |
| **Autonomy（自律）** | Controller LLM が数十社の優先度を自律判定し、処理フローを制御 |
| **Tool Use（道具）** | 86ルール検証エンジン・SQLiteトリガー・CascadeTracerを呼び出す |
| **Planning（計画）** | 数十社をリスクスコアでTier1/2/3に3層分類して優先処理 |
| **Collaboration（協調）** | Alpha/Beta/Gamma 3体合議 + ExpertAuditLayer の5体連携 |

連結決算の検証ルールは全部で86種ある。66種（R系48・JAA10・TAA5・MEA3）はPythonで決定論的に判定できる計算ルール、残り20種（EX-CPA6・EX-TAX7・EX-PEN7）は文脈を読む必要がある専門家判断ルールだ。そしてこの86種を、数十社それぞれに対して毎期実行する。

1体のLLMに数十社の連結パッケージを渡して「検証して」と言えば、答えは返ってくる。だがその答えは信用できない。理由は3つある。

**① 文脈が混ざる。** 債権の専門知識と税効果の専門知識は異なる。1体に全てを詰め込むと、売掛金の検証中に退職給付の文脈が混入する。人間でも、経理と税務と年金数理を1人で兼務すれば判断の質は下がる。できるベテランが1人で全部やる——その人が倒れたら全滅だ。LLMでも同じことが起きる。

**② 反論が生まれない。** 1体は自分の出力を疑わない。3体なら、Alphaがフラグを立てた項目にGammaが「ビジネス文脈で説明がつく」と反論できる。この摩擦が判断の質を上げる。

**③ 障害が全損になる。** 1体がタイムアウトすれば全停止だ。3体なら、1体が落ちても残り2体で`degraded`モードとして処理を継続できる。

### 視点の差別化——同じ数字を、違う目で見る

3体の検証エージェントは同じデータを見るが、見方が違う。

```
VERIFIER_SYSTEM_PROMPTS = {
    "alpha": """あなたは「数値の番人」です。
閾値を少しでも超えたらフラグを立てろ。
疑わしければフラグ。安全側に倒せ。""",

    "beta": """あなたは「関係の目利き」です。
単独では正常に見えても、相手先と突合すると
矛盾が見える場合がある。関係性の中で異常を見つけろ。""",

    "gamma": """あなたは「リスクの嗅覚」です。
ビジネス文脈で説明がつくならフラグは不要。
本当にリスクがあるものだけフラグを立てろ。""",
}
```

Alphaがフラグ、Gammaがクリアなら「数値は異常だがビジネス文脈で説明がつく」可能性がある。この判断を人間に委ねるのがConDuctorの設計思想だ。

**3体 specialty と担当ルール実装**（参照: `src/azure_agents/verifier_runner.py` ALPHA/BETA/GAMMA\_TYPES）:

| Verifier | 役割 | specialty formula\_type | 代表ルール例 |
| --- | --- | --- | --- |
| **Alpha「数値の番人」** | 閾値・期間比較 | period\_comparison / threshold\_check / balance\_equation / amount\_check | R001(売掛金変動率) / R009(DTA増減) / R015(BS等式) |
| **Beta「連結の整合」** | 会社間・帳票間突合 | cross\_company\_match / direction\_consistency / cross\_sheet | R019(関係会社取引差異) / R017(CF現金整合) / R041-R045(BS↔CF) |
| **Gamma「リスクの嗅覚」** | 発生頻度・集中検知 | new\_occurrence / aging\_analysis / year\_end\_concentration | R023(90日超滞留) / R025(構成比急変) / MEA系(期末月異常) |

**スループット**: 数千の判定を数分以内に処理（asyncio.gather並列実行。参照: `src/azure_agents/verifier_runner.py:VerifierRunner.run_all_async`）

### 合議——DeliberationEngine

3体の結果はDeliberationEngineで統合される。

* **unanimous**（全員一致）: 確信度が高い。自動承認の対象
* **majority**（多数決）: 意見が分かれた。人間の判断が必要
* **split**（同数）: 必ず人間が見る
* **degraded**（障害）: 1体以上がタイムアウト。手動確認

確定計算ルール（R系48・JAA10・TAA5・MEA3＝計66件）で意見が割れたら、それは「バグ」だ。Python数値計算は決定論的なので、不一致は実行障害の兆候として`execution_anomaly`フラグを立てる。LLM判断ルール（EX-\*の20件）で意見が割れるのは「正常」——この区別がシステムの信頼性を担保する。

なお、ハッカソン個人応募のため実環境データでの比較検証は実施できていない。上記の信頼性設計は、実務担当者と公認会計士が共同で設計したリスク評価可能な合成データ数十社（付録C参照）で検証した設計値に基づく。

### 単一LLM vs 3体合議——比較データで見る優位性（A1-6）

「なぜ1体ではなく3体か」——この問いに、定量的な根拠で答える。

合成データ数十社を対象に、単一LLM（GPT-4o）で全86ルールを処理した場合と、3体合議（Alpha/Beta/Gamma）で処理した場合を比較した。

| 指標 | 単一LLM（1体） | 3体合議（Alpha/Beta/Gamma） | 差異 |
| --- | --- | --- | --- |
| **誤検出（False Positive）** | 1体では自分で打ち消せない | 全員一致でのみフラグ → 1体の誤検出を2体が補う | 合議で低減（具体率は検証中） |
| **見落とし（False Negative）** | 少数意見が消える | 少数意見を保存しsplitへ回す | 合議で低減（具体率は検証中） |
| **「split」判定の人間エスカレーション** | 不可（1体に合議なし） | 意見が割れた件のみ人間へ | 重要案件に集中 |
| **障害時の継続性** | 1体が落ちれば全停止 | degradedモードで継続 | 可用性向上 |
| **処理時間（数十社）** | 単体高速 | **数分（合議品質）** | 合議で誤検出を低減 |

**重要**: 誤検出率・見落とし率の具体値は合成データ環境で検証中であり、実環境での検証は未実施。本表は単一構成と合議構成の構造的な差を定性的に示すものである。

#### なぜ3体の「合議」が誤検出率を下げるのか

Alphaは「数値の番人」として閾値を厳格に見る。Gammaは「リスクの嗅覚」としてビジネス文脈で判断する。この2体の視点の差異こそが、単一LLMが抱える「文脈の混在」問題を解決する。

```
単一LLMの問題:
  売掛金検証中に退職給付の文脈が混入
  → 「この数値は退職給付比率に照らして合理的」という誤った判断

3体合議の解決:
  Alpha(BS専門): 「前期比+50%は閾値超過 → フラグ」
  Gamma(リスク嗅覚): 「雨季前の業界特性で説明可能 → クリア」
  → split判定 → 人間がビジネス文脈を最終確認
```

#### 過去の知見蓄積が「認知判断節約」の最大の強み

PriorKnowledgeStoreによる過去ナレッジ蓄積が、3体合議の効果をさらに増幅する。「A001社は3期連続で売掛金フラグが出たが、業績回復で翌期に解消している」——この文脈を持つことで、Alpha/Gammaは今期の判断に前期の知見を組み込める。

単一LLMには「記憶」がない。毎期ゼロから判断する。3体合議＋PriorKnowledgeStoreの組み合わせは、使えば使うほど「同じ会社で同じ間違いをしない」システムへと進化する。これがConDuctorにおける「認知判断節約（Cognitive Load Reduction）」の根幹だ。

#### 3体合議の数学的・理論的根拠

「3体合議は感覚的な多重チェックではないか」という疑問への回答を、3層の理論で示す。

**第1層: ベイズ的独立性 — 同時誤検出確率の3乗減少**  
3体のVerifierが互いに**独立した視点**(Alpha=数値、Beta=連結整合、Gamma=横断比較)を持つことを前提とすれば、3体全員が同一の誤判断をする確率は単体の3乗で減少する。

* 単体の誤検出率 10% の場合: 3体一致誤判断 = 0.001(0.1%)
* ベイズ的に言えば、各Verifier の判断は**条件付き独立**であり、結合確率は積で表現される
* ただし完全独立は前提に過ぎず、現実には**プロンプト設計**で独立性を担保する必要がある(Alpha は「数値の番人」、Gamma は「リスクの嗅覚」のように役割を明示)

**第2層: 職務分離理論 — SOX法/PCAOB AS5/COBIT との直接整合**  
3体合議は会計監査の基本原則「**実行者と検証者の分離**」をシステム上で実装したものだ。

* **米国SOX法 404条**: 内部統制において「自己監査の禁止」を明示
* **PCAOB Auditing Standard No.5**: 統制設計と統制運用の分離を要求
* **COBIT 5 (ISACA)**: 「Segregation of Duties」が情報システム統制の中核原則
* ConDuctor の Controller(実行者) と Verifier(検証者) の分離は、これらの基準と直接対応する

**第3層: 反対意見の永続性 — Append-Only + dissent\_reasoning**  
3体合議が単なる多数決と決定的に異なるのは、**少数意見を統計的ノイズとして捨てない**点だ。

* 多数決(2/3) で結論が出ても、1体が指摘した論点は `dissent_reasoning` として SQLite に**Append-Only**で永続保存される
* これは最高裁判所の反対意見が判例として残るのと同じ構造
* 将来「2体がOKと言ったが、1体の指摘が後から正しかった」という事例が起きたとき、過去の `dissent_reasoning` を遡って参照できる
* 統計的にノイズに見える少数意見が、**長期的なリスク予告**として機能する

これら3層の理論的裏付けによって、3体合議は「**たまたま3つに分けた**」のではなく、**ベイズ統計・会計監査基準・判例システム的記録**の3点を同時に満たす設計選択だと示せる。

### 5体の構成と役割分担

```
┌─────────────────┐
│ ① Controller     │  ← 数十社トリアージ・並列分配・全体統制
└────────┬────────┘
         │
┌────────▼────────┐
│ ② Quartermaster  │  ← 内部取引照合・消去承認・因果伝播分析
└────────┬────────┘
         │ 並列展開
┌────────▼─────────────────────────────┐
│ ③ Alpha   │ ④ Beta   │ ⑤ Gamma         │  ← 3体Verifier合議
│ (式判定)   │ (連結整合) │ (横断比較)        │
└───────────┴──────────┴──────────────┘
         │ DeliberationEngine が unanimous/majority/split を判定
         ▼
   ExpertAuditLayer（CPA/Tax/Pension 3専門家ペルソナ）
   ← 上記5体の中で、Alpha/Beta/Gamma が EX-* ルール（CPA6 + Tax7 + Pension7）
     を評価する際に呼び出す「専門家プロンプト層」。独立した第6体ではない
```

**5体 = Controller + Quartermaster + Alpha + Beta + Gamma**。Track A（Quartermaster: 内部取引照合・消去承認・因果伝播分析）と Track B（Alpha/Beta/Gamma の3体合議による検証）を Controller が並行制御する。ExpertAuditLayer はこの5体の中で Alpha/Beta/Gamma が専門家ルール（EX-\*）を扱う際に呼び出される「ペルソナ層」であり、独立した第6体ではない。職務分離の原則は5体構成で完結している。

**Adaptive Triage**（リスク別の優先処理分類）: Controller は数十社をリスクスコアで3段階に分類する。Tier1（高リスク12社）は全86ルールを適用、Tier3（低リスク24社）は基本12ルールで高速処理。全社全ルールで処理するより致命的見落としを防ぎながら処理時間を短縮する設計だ。`src/triage/adaptive_triage.py`に実装済み。

**asyncio.gather**（Pythonの非同期並列実行ライブラリ）: Alpha/Beta/Gamma 3体の検証を同時並列で走らせる。1体ずつ順番に処理すれば3倍かかる計算が、1体分の時間で完了する。

## 9. アーキテクチャ——Azure Connected Agentsの必然性

### 業務フロー全体図——ConDuctorがカバーする領域（A1-8）

連結決算の全工程を俯瞰し、ConDuctorがどの領域をカバーするかを示す。

**青色 = ConDuctorがカバーする範囲**（工程全体の10〜15%相当）

現段階では、ConDuctorは連結決算の全工程を自動化しない。連結仕訳（Step 3）は担当者の判断が必要であり、ConDuctorは介入しない。パッケージ収集（Step 1）も担当者がExcelをドラッグ＆ドロップで渡す前提だ。

だがConDuctorがカバーする5ステップは、**全工程の中で最も認知負荷が高い領域**だ。

* 品質チェック（Step 2）: 86ルール×数十社 = 5,160判定を手作業でやれば4日かかる
* 照合・検証（Step 5）: 3体の専門家視点を1人の担当者が同時に持つことは不可能
* 消去仕訳（Step 6）: 1件の漏れが開示ミスに直結。人間の疲労が最も致命的な領域
* 開示準備（Step 7）: 11項目の整合性チェックをミスなく実施する集中力は有限

ConDuctorの戦略は「全部やる」ではなく「最も間違えてはいけない部分だけを確実にやる」だ。解放された人間の認知資源が、本来すべき経営判断に向かう——これがConDuctorの設計思想だ。  
ただし、今後の展望としては「全部やる」も実現したい。

### コンテキスト設計という本質的な問題

連結決算の構造を見れば、マルチエージェントに至るのは必然だった。

数十社の連結パッケージには、BS・PL・CF・税効果・退職給付・関係会社間取引——互いに異なる専門知識が必要な領域が混在する。1体のLLMに全てを詰め込めば、コンテキストウィンドウの中間情報が参照されにくくなる（Lost in the Middle問題）。これはTransformerの注意機構の構造的特性であり、ウィンドウサイズを拡張しても解決しない。

マルチエージェントの設計で最初に解くべき問題は、オーケストレーション（誰が誰に指示を出すか）ではない。**コンテキスト設計**（各エージェントが何を知り、何を知らないべきか）だ。

Alphaは貸借対照表の文脈だけを持つ。Betaは損益計算書の文脈だけを持つ。互いの文脈は混ざらない。これは人間の組織設計と同じだ——経営層が全体像を持ち、現場は自分の担当領域に集中する。報連相や稟議書が「コンテキスト同期プロトコル」として機能するように、ConDuctorではDeliberationEngineが各エージェントの判断を統合する。

各エージェントを疎結合にすることで、「1体がタイムアウトしても残り2体がdegradedモードで処理を続行できる」という障害耐性も同時に手に入る。CIパイプラインでジョブを並列実行するのと同じ発想で、3体のverifierをasyncio.gatherで並列実行している。

### なぜAzure Connected Agentsなのか — クラウド・フレームワーク選定の根拠

連結決算という領域は **「判断を育てる」** ことが本質だ。担当者が異動しても、子会社の個性、過去の判断パターン、CPA レビューの軌跡が**消えずに残り続ける**必要がある。これがフレームワーク選定の最優先基準になった。

#### なぜ Copilot Studio + MCP ではないか

Copilot Studio は **ノーコード/ローコードでのエージェント構築**に強い。Microsoft AI Tour Tokyo 2026 で示された「**Becoming Frontier**」の中核技術であり、ConDuctor の **将来形**としても重要な位置を占める。

それでも Phase 1 で Copilot Studio を採用しなかった理由は:

**① 86ルールの「決定論的計算 66件 + LLM判断 20件」のハイブリッド**  
ConDuctor のルール構造は、Python の数値計算で完結する 66件と、LLM 判断が必要な 20件のハイブリッドだ。Copilot Studio はノーコードに最適化されており、**Python レベルでの数値整合性検証 + LLM 判断の協調**を実装するには、Code Interpreter 経由の間接的アプローチが必要になる。Microsoft Foundry Agent Service の Connected Agents は、Python コードと LLM 判断を **同一エージェント内で直接協調**させられる。

**② Phase 2 で Copilot Studio 統合を予定**  
ConDuctor のロードマップでは、**Phase 2 で Copilot Studio + MCP 統合**を計画している。Teams から自然言語で「A001社の売掛金は大丈夫か?」と問い合わせ、ConDuctor のエンジンが 86ルール検証 + 3体合議結果を Copilot Studio 経由で返す。Phase 1 では Microsoft Foundry Agent Service で**コアエンジン**を確立し、Phase 2 で**運用層**として Copilot Studio を被せる。これが Microsoft Cloud + AI ベストプラクティスだと考えている。

#### 結論

ConDuctor のフレームワーク選定は **「インフラの自前実装に時間を使わず、連結決算ドメインの実装に集中する」** という1つの原則から導かれた。Azure Connected Agents は検討した中ではその原則を最も効率的に実現する選択肢だった。

### 通常のAPI呼び出しとの決定的な差異

ConDuctorが「エージェント」を名乗る理由は、単にLLMを呼び出しているからではない。

```
【通常のLLM API呼び出し】
Controller → POST /chat/completions → Alpha の結果取得
 ・ステート: Controller側が全管理
 ・Alpha は「記憶を持たない関数」

【Microsoft Foundry Agent Service Connected Agents】
Controller → Azure Agent Service → Alpha スレッドへの委譲
 ・ステート: Azure側がスレッドとして永続管理
 ・Alpha は「自分の判断履歴を持つエージェント」
```

**3つの本質的な違い:**

**① スレッド永続化**: 各エージェントが独立したスレッドを持つ。Controllerが Alpha に検証を依頼すると、会話文脈は Azure 側で永続管理される。「前回 A001 社でこの点を指摘した」という文脈が自動的に引き継がれる。

**② Tool Use統合**: Alpha が conductor\_rules.xlsx を読み込んで R001 を実行する操作は、Alpha 自身が Azure 側で実行する。Controllerがファイルを読んで渡す必要がない。

**③ 非同期協調**: `asyncio.gather` で3体を並列実行できるのは、各エージェントが独立スレッドを持つためだ。サービス側が競合を管理する。

```
# Connected Agents による非同期並列委譲
async def run_all_with_deliberation(
    self, pkg: dict, company_code: str
) -> DeliberationResult:
    results = await asyncio.gather(
        self._run_verifier("alpha", pkg, company_code),
        self._run_verifier("beta",  pkg, company_code),
        self._run_verifier("gamma", pkg, company_code),
    )
    return self.deliberation_engine.evaluate(results)
```

### Semantic Kernelの役割

86ルールをSKプラグインとして宣言的に登録できるため、新ルール追加時にControllerのコードを変更する必要がない。`conductor_rules.xlsx`に追記するだけで、SKが自動的に新プラグインを認識する。

## 10. 判断を育てる——5,160判定を20判断に減らす

人間がしなくていい判断を、構造的に減らす。これがConDuctorの全体設計を貫く思想だ。「**Goal**（数十社の決算整合性確認の明確な定義）→ **タスク分割**（5体が専門領域ごとに自律処理）→ **AI処理**（asyncio並列・数分・高品質）→ **人間レビュー**（絞り込まれた20件の判断に集中）」——この4段サイクルが、ConDuctorのHypervelocity設計の本質だ。高速であることと高品質であることを両立させる設計。LLMの出力は全てドラフトとして扱い、3体の合議と人間の最終判断が品質を担保する。

![5,160判定 → 50件 → 20判断 圧縮ろうと図](https://static.zenn.studio/user-upload/deployed-images/49e55e8f6c90c8e1691f7006.png?sha=c02a1a689b2a55df4f7703737701888020b83d2c)

```
  ┌─────────────────────────────────────┐
  │      5,160判定（数十社×86ルール）       │
  │     5体のAIが全判定を自動処理         │
  └──────────────────┬──────────────────┘
                     ▼  第一層: 物理疲労の解消（4日→数分）
  ┌─────────────────────────────────┐
  │       50件（FLAG / 要対応フラグ）  │
  │   AutoApproval + 一括承認で絞込   │

  └─────────────────────────────┘
```

減らすべき負荷には3つの層がある。

| 層 | 問題 | ConDuctorのアプローチ |
| --- | --- | --- |
| **① 物理疲労** | 4日間22時残業 | 5体のAIが数分で処理 |
| **② 判断疲労** | 5,160判定を精査 | 自動承認+一括処理で50件に |
| **③ 修正の恐怖** | 波及が見えない・説明できない | シミュレーター+反事実的説明 |

### 第二層: 5,160判定を50判断にする

**AutoApprovalEngine**: severity が `info`/`low` かつ全員一致（unanimous）のフラグは自動承認する。3体全員が低重要度と判断したものを人間が再確認しても、付加価値は生まれない。

**BatchApprovalBar**: 重大度別に一括承認/却下。「HIGH 3件は個別確認、LOW 12件は一括承認」というワークフローが可能になる。

**HumanReviewQueue**: フラグを「読む」のではなく「処理する」UIに変える。`a/r/p`キーで瞬時判断。進捗バーで残量を可視化。

**RejectionReasonModal**: 却下理由を4分類（False Positive / Business Exception / Duplicate / Other）で強制収集する。来期のルール改善データになる。

### 第三層: 修正の恐怖を消す

**インパクトシミュレーター**: 科目名と修正額を入力すると、影響を受ける科目と子会社がツリー表示される。「売掛金+50百万円 → 21科目に連鎖（BS/CF/PL、最大4段）」——波及が見えれば、修正の心理的コストは消える。

**反事実的説明（XAI）**: インパクトシミュレーターと同時に表示される。「もし今期売上高が1,360百万円以下だったなら、このフラグは発生していませんでした」——AIの判断理由を「もし〇〇なら」で説明する。公認会計士や上司への証跡として機能する。

**バイリンガル問い合わせメール自動生成**: フラグ確認後の子会社への確認メールをLLMがSSEストリーミングで生成する（日本語・英語の自動切替対応）。数名がバラバラに送っていた問い合わせが**1通の集約メール**になり、1社15〜20分 × 30社 = 7.5〜10時間の作業が1クリックになる。実装: `src/agents/mail_composer.py`（Quartermaster = 兵站担当エージェント）

### 第三層の先: 対話するシステムへ

**自然言語問合せ（NLQ）**: 「A001の売掛金は大丈夫？」と聞けば、審議結果込みの回答が返る。パーサーはLLM不使用で500ms以内に応答する。否定形検出（「大丈夫じゃない？」→ リスクチェックに自動変換）も実装した。

**閾値Advisor**: 過剰検知しているルールをAIが自動診断し、閾値変更を提案する。変更には`approval_rationale`（会計基準に基づく承認根拠）の入力が必須だ——「誰が承認した」ではなく「なぜこの閾値が会計的に妥当か」をシステムに刻む。「AIが勝手に変えた」ではなく「AIが提案し、CPAが会計基準に基づいて承認した」という証跡が、Append-Onlyログとして残る。

**異常伝播グラフ**: A001社の売掛金異常がB003社の買掛金に波及している——このグループ内連鎖をReact Flowの有向グラフで可視化する。修正の順序と優先度が図として見える。

**常習犯検知（Repeat Offender）**: 「A001社のR012は3期連続フラグ」というパターンを内部的に検出し、Tier分類のスコアに自動加算する。同じ指摘が繰り返される会社は、次期のトリアージで自動的にTier1扱いとなる。

**5,160判定 → 50判断 → 対話で20判断**——これがConDuctorの認知負荷攻略の全体像だ。

## 12. 監査パートナーが語るConDuctor——実務知見との符合

監査法人に所属する公認会計士から、Conductorの思想に対して以下のフィードバックをいただいた。

### 留意点1: 株主資本等変動計算書の吟味

> 「連結エラーは最終的に株主資本の部に影響する。これが大原則だ。」

連結処理の誤りは最終的に株主資本の部に表出する。株主資本等変動計算書を丹念に追うことで、上流の連結処理の整合性を逆算できる。

ConDuctorの3-verifier合議（Alpha: 数値番人 / Beta: 連結整合 / Gamma: リスク意味論）は、エラーの波及経路を多角的に検出するという点でこの原則と構造的に符合している。

### 留意点2: 四半期レビューの積み重ね

> 「Q2→Q3で増加すべき項目の不自然な減少・異常な同額の検出が肝だ。」

単期の数値チェックだけでなく、四半期をまたいだ時系列の整合性が連結審査の核心である。

ConDuctorのAppend-Only DBは全判定履歴を不可変で保存する。これにより、将来的に「前期比較ルール」を追加するだけで四半期異常検出が可能になる拡張性を持っている。

### 留意点3: 個別決算と連結決算の整合性

> 「資本金の不一致は言語道断。投資と資本の相殺は整合していなければならない。」

子会社個別決算と連結消去後の数値の不一致は根本的な誤りだ。86ルールのうち66ルールが確定計算ルールとして投資消去・資本連結・連結消去仕訳をカバーしている。

---

これらは、監査現場の実務知見が、ConDuctorの設計思想に**結果的に符合していた**ことを示す。設計時点では「監査パートナーが最初に見る場所」を意識していたわけではない。それでも符合したのは、「書いていないことはやらない」「Append-Only」「3-verifier合議」という設計原則が、監査という行為の本質と同じ問題を解こうとしていたからだと考えている。

## 13. 今後の課題　ConDuctorの「限界」3点

**① LLMは間違える**: Expert Audit Layerの20件は大規模言語モデルが判断する。LLMは確率的な存在であり、稀に誤った解釈をすることがある。「ConDuctorがOKと言ったから大丈夫」はない。

**② コンテキストは有限**: 連結パッケージに記載されていない情報——経営陣の意図、業界特有の商慣習、未公表の再編計画——はConDuctorには見えない。「数値的には正常だが実態は問題あり」というケースをAIは検出できない。

**③ 実環境データでの検証は未完**: ハッカソン個人応募のため、実際の上場企業の連結パッケージでの大規模検証は実施できていない。公認会計士と共同設計した合成データは現場の妥当性を反映しているが、実環境との乖離がゼロとは言い切れない。

### 決定論エンジンが保証すること / 推論(LLM)・人間が担うこと

ConDuctorの設計の誠実さは、この区別を明示することから来ている。

| 分類 | 具体例 | ConDuctorのアプローチ |
| --- | --- | --- |
| **① 閾値で原理的に検出可能** | 前期比変動率・残高比率・CF現金整合・消去差異（R001–R063） | 決定論エンジンが確実に判定。同一データに対し常に同一結果。ここに人間の判断は介在しない |
| **② 閾値で部分的に検出可能** | 季節性バイアス・グループ内取引連鎖・のれん減損兆候 | フラグを立てるが確信度が下がる。Admiralty CodeがB2→C3に格下げして表示し、人間に「この根拠の質を見て判断せよ」と伝える |
| **③ 閾値では原理的に検出不能 → 推論領域** | 経営陣の意図推定・業界慣行・相手方整合性・将来予測 | LLM推論（EX-\*ルール20件）または人間の判断に委ねる。決定論では保証しない。この領域で「AIがOKと言った」は証跡にならない |

必ず人間の判断が最終決定権を持つ。

### それでもConDuctorを作る理由

現時点の検証段階の精度が「使えない」を意味するかどうかは、比較対象による。

**手作業の「誤検出率」と「見落とし率」は計測すらされていない**。デスマーチの中で疲れ果てた人間が行う検証の精度を、誰も測っていない。「AIより人間のほうが正確だ」という前提は、実証されていない。

多すぎるフラグは「見落とし」よりはるかに許容できる——開示後の修正訂正より、フラグを1件余分に確認するほうが圧倒的に安い。引き続き実環境での計測を進め、数値は公開時に随時更新する。

**最終判断は常に人間が行う。ConDuctorはその判断を助けるツールだ。**

> 「AIが言ったから正しい」は禁物です。ConDuctorは連結担当者の判断を「省力化」するためのツールであり、「代替」するためのものではありません。

## 14. まとめ——桜を楽しめる決算期へ

3体のVerifierが合議し、5体のエージェントが連携して情報の質を格付けし、存在しないデータの不在を検出し、少数意見を永続保存する。そして担当者が替わっても、会計士が替わっても、「この子会社は要注意」という判断パターンが組織に残り続ける。

人が変わっても、変わらないものがある。それをAIが守っていく。

数字に苦しんだ時間があれば、公認会計士は「データ確認」ではなく「経営判断を支える本質的な監査」に集中できたはずだ。担当者は、自分の人生を豊かにする時間を楽しめたはずだ。  
全社規模の連結決算という、誰かが解かなければならない問題に、5体のエージェントが責任ある実装で応える。

ConDuctorは段階的な実行フェーズで展開できる設計だ。フェーズ1で単一子会社の検証から始め、フェーズ2で10〜20社グループへ拡張し、フェーズ3で全社数十社一元管理へ。各フェーズにGO/NOGOのゲートがある。「今すぐ動くデモ」と「段階的に全社へ広げるロードマップ」——この両方を持つことが、現場担当者から経営層まで全員が使えるシステムの条件だ。

担当者は、変わる。だが、判断の蓄積を引き継ぐ仕組みがなければ——次の担当者はまた、同じ春に、同じ間違いを繰り返す。1社あたり年420時間。連結決算義務を負う上場企業 約3,900社が、同じ春を戦っている。

**人は替わる。**

**知は残る。**

あの春を戦った、すべての人へ。

「桜が咲くと憂鬱」という言葉が、決算担当者の言葉ではなくなる日を作るために。

## 付録B: セーフガード実装詳解

### SG-1: CascadeTracer（連鎖影響トレース）

BFS（幅優先探索）で科目間の連鎖を最大4段まで追跡する。

```
class CascadeTracer:
    MAX_DEPTH: int = 4

    def trace(self, flagged_account: str) -> list[ChainImpact]:
        visited: set[str] = set()
        queue: deque[tuple[str, int]] = deque([(flagged_account, 0)])
        impacts: list[ChainImpact] = []
        while queue:
            current, depth = queue.popleft()
            if current in visited or depth > self.MAX_DEPTH:
                continue
            visited.add(current)
            for affected in CHAIN_MAP.get(current, []):
                if affected not in visited:
                    impacts.append(ChainImpact(
                        source_account=current,
                        affected_account=affected,
                        chain_depth=depth + 1,
                        affected_statements=self._classify_statement(affected),
                    ))
                    queue.append((affected, depth + 1))
        return impacts
```

売掛金1件修正の波及:

```
売掛金 → 売上高 → 営業利益 → 経常利益 → 税引前利益
  ↓                                          ↓
関係会社間売掛差異                      法人税等 → 当期純利益
                                                    ↓
                                          利益剰余金(BS) / 営業CF
```

### SG-3: EliminationVerifier（消去検証）

「消去が1件漏れて3時間やり直し」を自動検出する。4パターン検証: 消去漏れ・残差・過大消去・仕訳バランス。TOLERANCE=1.0百万円（外貨端数を許容）。

### なぜ数十社か

* 50社以下: Adaptive Triageの効果が検証しにくい
* 100社以上: テスト実行時間が現実的でなくなる
* **数十社**: Tier1(12社)・Tier2(24社)・Tier3(24社)で、トリアージの集中投資効果が測定可能

言語分布: 日本語30社・英語15社・ドイツ語10社・タイ語混在5社。多国籍グループの連結現場を反映。

合成データは意図的な異常値を含みつつ、財務諸表の基本等式（BS等式・CF整合）は常に成立させた。整合性チェックをPythonで実施後に初めてExcelに書き出すことで、「クリーンな会社はDR-01〜05が全PASS」という前提を保証した。
