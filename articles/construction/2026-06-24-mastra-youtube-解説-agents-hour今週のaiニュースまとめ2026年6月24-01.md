## 概要

今回の AI News パートを一言でまとめると、「Fable 後の世界で、どのモデルをどう使い分けるか」が一気に現実的な論点になってきた回でした。番組の冒頭でまず触れられるのは、Anthropic の Fable / Mythos をめぐる停止騒動の余波です。ここでは単に「危険なモデルが止められた」という話ではなく、モデル企業自身が危険性を強く訴えてきた結果、政治や規制の介入を自ら呼び込みかねない、という皮肉が語られます。

そのうえで番組は、OpenAI の GPT 5.5 Cyber、オープンウェイトの GLM 5.2、Cursor の新展開、Vercel の agent framework、Slack をインターフェースにしたエージェント、Midjourney の医療機器構想、そして複数モデルを束ねる orchestration API まで、かなり幅広い話題を一気に扱います。ただし散らばって見えるニュースにも共通の流れがあります。それは、単一の最強モデルを信じる時代から、複数モデル・複数UI・複数実行環境を前提に設計する時代へ移りつつある、という流れです。

## 要点

1. **Fable / Mythos 騒動は、モデル性能の話から規制とアクセス制御の話へ移った。** 危険性を強く打ち出した frontier model は、政治や国家安全保障の文脈に巻き込まれやすいことが改めて見えました。
2. **OpenAI には追い風が吹いている。** GPT 5.5 Cyber の発表に加え、Anthropic の信頼性不安が「本番投入できる代替」を探す動きを強めています。
3. **GLM 5.2 は、オープンウェイトが frontier 級へ肉薄してきた象徴として扱われた。** 最高性能そのものより、十分高性能で安く使えることの価値が強調されていました。
4. **応答速度はこれからの重要指標になる。** token/sec のような指標が、人間が loop に張り付く使い方ではそのまま体感品質を左右します。
5. **Cursor は IDE を超えて、Git・mobile・独自 frontier model まで広げようとしている。** coding agent 企業が開発ワークフロー全体を取りに行く流れが鮮明です。
6. **Vercel Eve や flu 1.0 beta からは、agent framework の主戦場が developer experience に移っていることが見える。**
7. **Claude Tag や Replit for Slack は、Slack が agent UI になる流れを後押ししている。** チームの会話空間がそのままエージェントの操作面になる、という見立てです。
8. **Midjourney Medical は、AI 企業が model API を越えて real-world device に踏み込む兆候として面白い。** 一方で、医療機器ゆえに慎重な見方も必要です。
9. **Sakana Fugu のような multi-agent orchestration API は、1モデル万能論より routing の時代を感じさせる。** ただし benchmark をどう読むかは難しくなります。

## 詳細

### Fable / Mythos の余波は、もはやモデル評価ではなく規制と統治の話になっている

ニュースパートの冒頭では、前週から続く Fable / Mythos 騒動の続報が扱われます。番組内では、NSA の機密系に Mythos が侵入したという SNS 投稿が紹介されますが、Shane 自身もその時点で「確認は取れていない」とかなり慎重です。ここで重要なのは、個別の真偽よりも、この手の話が出た時点で frontier model が一気に国家安全保障の文脈へ入ってしまう、という点です。

<https://x.com/kimmonismus/status/2068605229965234238>

ホストたちは、Anthropic が自社モデルを「非常に危険で高性能だ」と強く語ってきた以上、何らかの incident が起きたときに政府が動くのは不思議ではない、と述べます。これは少し意地悪な言い方をすると、「危険性をセールストークに使ってきたら、本当に危険物として扱われる局面が来た」ということでもあります。

ここで押さえておきたい周辺知識は、frontier model の規制論が単に safety の話では終わらないことです。実際には、誰がアクセスできるのか、API 提供者がどこまで利用者を識別できるのか、国籍や地域による制限をどう実装するのか、といった運用設計の問題へ直結します。番組でも、将来的には身分証の提示や、API レベルでのより厳しい access control が必要になるかもしれない、という見方が出ていました。AI の議論はすぐ性能競争に寄りますが、本番運用の世界では「誰に使わせるのか」が同じくらい重いテーマです。

### OpenAI の GPT 5.5 Cyber は、性能より「代替になりうる」という意味で大きい

次に扱われるのが OpenAI の GPT 5.5 Cyber です。番組内では CyberGym で Mythos preview を上回ったと紹介され、少なくとも benchmark 上は「Fable / Mythos クラスのサイバー能力に対抗できるモデルが、もう別の会社から出てきている」構図が示されます。

<https://x.com/sama/status/2069121360744550796>

ただしホストたちが本当に重視しているのは、単なるベンチマーク勝敗ではありません。Anthropic 側の供給不安や partial outage が続くなかで、OpenAI が「高性能で、しかも本番へ投げられるかもしれない代替」として浮上していることです。番組中でも、顧客が「Anthropic を使いたいが reliability の面で production に入れづらい」と話していた、という生々しいエピソードが出ます。

<https://x.com/OpenAI/status/2069104283824640023>

これはかなり重要です。2025 年ごろまでの AI 議論では、「どのモデルが一番賢いか」が中心でした。しかし agent の本番運用では、それだけでは足りません。止まらないこと、fallback を設計しやすいこと、価格や SLA を読みやすいことのほうが、実は重要になる場面が多いからです。特に長時間走る coding agent や security workflow では、モデルがたまに数分落ちるだけでもシステム全体の設計が面倒になります。今回の AIニュース は、その現実をかなり率直に映していました。

<https://x.com/mark_k/status/2069169916432003105>

また番組では、OpenAI の次モデルに関するリークめいた観測や、GPT 5.6 の遅延、DeepMind 3.5 Pro 側の遅れも話題に出ます。このあたりは確定情報というより業界観測に近いですが、少なくとも「各社とも次の大きな更新に苦戦しているらしい」という空気感は伝わってきます。Fable が見せた水準にすぐ全員が追いつくわけではない一方で、差が永続的でもなさそうだ、というのがホストの見立てです。

<https://x.com/synthwavedd/status/2069432791184650426>

### GLM 5.2 は、オープンウェイトの価値を「代替品」から「本命候補」へ押し上げた

今回もっとも熱量高く語られていたニュースのひとつが GLM 5.2 です。Shane と Abby はかなり率直に「これはかなり良い」と評価しており、Opus 4.8 に完全一致ではないにせよ、その近くまで来ていると見ています。番組では agentic coding のスコア、Design Arena での順位、単発バグ修正の anecdotal benchmark などが紹介されます。

<https://x.com/Zai_org/status/2066938937344495629>

ここで重要なのは、「Opus 4.8 と全く同じか」ではありません。むしろ番組の論点は、十分強く、しかも安いオープンウェイトが出てきたことで、用途によっては closed frontier model を常用する必要が薄れてきた、というところにあります。Abby は GLM 5.2 をかなり使っている様子で、長時間放置する大きな仕事では Opus の方が楽な場面もあるが、人間が loop に入りながら細かく指示を出す使い方なら相当戦える、という評価でした。

<https://x.com/Designarena/status/2066940737011560652>

この差は、agent の利用スタイルの違いとして理解すると分かりやすいです。大きく分けると、AI の使い方には「fire-and-forget 型」と「human-in-the-loop 型」があります。前者は数十分から数時間走らせてあとで結果を見る使い方で、後者は人間がすぐ次の指示を返しながら短い反復を回す使い方です。前者では多少遅くても賢いモデルが有利になりやすく、後者では応答速度やコストが強く効きます。GLM 5.2 は後者でかなり魅力的だ、というのが番組のトーンでした。

<https://x.com/cline/status/2069171146994729078>

加えて、one-bit 版を Mac Studio 上でローカル実行できるかもしれない、という話も出てきます。ここでいう one-bit は、ざっくり言えば極端に量子化して、巨大モデルをより少ない計算資源で扱いやすくする方向の話です。もちろん品質や速度にはトレードオフがありますが、「かなり強いモデルを手元に置けるかもしれない」というだけで意味は大きいです。特に企業利用では、コスト、データ持ち出し、待ち時間の3点でローカル実行に強い魅力があります。

### token/sec が重要になるのは、モデルが賢くなったからではなく、人間が待てなくなるから

GLM 5.2 の話の中で面白かったのが、token/sec という指標への言及です。従来は benchmark score や price per token が話題の中心でしたが、これからは「1 秒あたりどれだけトークンを出せるか」が user experience をかなり左右する、という話が出ます。

この指標が効くのは、human-in-the-loop の局面です。もしモデルが数百、数千 token/sec の速度で返せるなら、人間は別タスクに逃げず、その場で対話を続けられます。つまり、複数ループを並行で回して待ち時間を埋める必要が減り、「今見ているこの session の中で思考と操作が完結する」感覚に近づきます。

<https://x.com/UnslothAI/status/2069418532375564484>

これは単なる快適性の問題ではありません。操作のリズムが変わると、どういうタスクを AI に渡せるかも変わります。速いモデルは、より短い計画単位で人間と協調しやすくなります。逆に遅いモデルは、長時間走らせる前提の task decomposition や job queue に寄った設計のほうが向きます。今後の agent 設計では、「賢さ」と同じくらい「待ち時間の短さ」がアーキテクチャ選択に影響するはずです。

### Cursor は IDE ではなく、agent 時代の開発基盤を丸ごと取りに行こうとしている

<https://x.com/cursor_ai/status/2067012220832329782>

Cursor Compile の話題では、Origin という GitHub 競合、mobile から desktop cursor を遠隔操作する機能、そして独自 frontier model の訓練という3点が特に取り上げられます。番組で面白いのは、これを単なる機能追加ではなく、「coding agent 企業がワークフロー全体を垂直統合しに来ている」と見る視点です。

<https://x.com/swyx/status/2066928345246470204>

Origin は、agent workload に最適化された Git / code collaboration surface を狙うものとして紹介されていました。ここでの論点は、「人間が書くコードを前提に作られた GitHub のUIや操作体系が、そのまま agent 主体の開発にも最適とは限らない」ということです。もし PR の大半を agent が作り、merge conflict の解消や code review も agent 支援前提になるなら、source control の操作面そのものを再設計する余地があります。

<https://x.com/cursor_ai/status/2069149296436330776>

さらに mobile から agent を動かす方向も、単なるおもしろ機能ではありません。要するに coding agent を「IDE を開いたときだけ使う道具」ではなく、「どこからでも仕事を投げられる非同期 worker」に近づけたいわけです。そして自前 frontier model の訓練まで進めるなら、Cursor は editor company ではなく、data・model・workflow を束ねる full-stack AI developer platform を目指していると見るほうが自然です。

### Vercel Eve と flu 1.0 beta は、agent framework の差別化軸が DX にあることを示している

番組中盤では、Vercel Eve と flu 1.0 beta も短く触れられます。ここでの見どころは、どちらが勝つかよりも、agent framework の競争軸がかなり developer experience 寄りになっていることです。

<https://x.com/vercel/status/2067180054979936413>

Vercel Eve については、CLI からの立ち上がりの良さや、bundling を含む「とりあえず動く」体験の滑らかさが評価されていました。これは agent framework にとってかなり大事です。実際、エージェント開発では最初の数分で sandbox、tool wiring、deployment path まで全部つながるかどうかが、そのまま採用率に響きます。一方で、そうした magical な bundling は複雑な要件に入ると噛み合わなくなることもあり、番組でもその点は少し警戒されていました。

<https://x.com/FredKSchott/status/2066962296119959581>

flu 1.0 beta のくだりも含めて見えてくるのは、もはや「エージェントを作れます」だけでは差別化にならない、ということです。今後は、どれだけ少ない摩擦で始められるか、どれだけインフラやデプロイ先と自然につながるか、どれだけ routing・tooling・workflow を無理なく拡張できるかが勝負になります。フレームワークの競争は、モデルそのものより、周辺体験の総合点で決まる局面に入っています。

<https://x.com/OpenAIDevs/status/2067681320281723113>

### Slack は、社内エージェントにとって最有力 UI になりつつある

<https://x.com/claudeai/status/2069468693017268244>

Claude Tag と Replit for Slack の話題は地味に見えて、かなり重要です。番組内では、Claude が Slack のチームメンバーとして参加し、タグ付けで仕事を振れること、Replit も Slack 内で build の依頼を受けられることが紹介されます。

<https://x.com/Replit/status/2067661213278900402>

これは「チャットが新しい IDE になる」という話ではありません。もう少し正確には、「チームの意思決定や依頼が生まれる場所に agent を常駐させる」流れです。社内業務の多くは、実は最初から structured form で始まりません。雑談、依頼、障害報告、確認、仕様の相談といった曖昧なやりとりから始まります。Slack が重要なのは、その曖昧さを含んだまま work request を取り込みやすいからです。

この流れが進むと、agent の価値は単体の賢さだけでは測れなくなります。チャンネル履歴をどう読むか、どこまで tool access を与えるか、誰の承認で何を実行するか、といった権限設計が重要になります。言い換えると、Slack agent の本質は chat bot ではなく、社内 workflow の入り口を握る coordinator です。今回の回では、その方向がかなりはっきり見えていました。

### Midjourney Medical は、AI 企業の拡張先が「より良いモデル」以外にもあると示している

今回いちばん異色だったニュースは Midjourney Medical です。番組では、全身スキャナのような装置を spa などへ設置し、短時間で身体の異常を検出できるようにする構想として紹介されていました。Shane も Abby も最初は半信半疑で、雰囲気としては SF 映画のプロップに近い見た目だと盛り上がっています。

<https://x.com/midjourney/status/2067421950314688759>

ここで注目したいのは、画像生成で知られた企業が、医療機器や physical infrastructure の方向へ飛び出している点です。AI 企業の進化先として、私たちはつい「より強い foundation model」や「より便利な agent platform」を想像しがちですが、実際には sensor、robotics、device、clinical workflow のような物理世界との接点も大きなフロンティアです。

もちろん、この手の話には慎重さが必要です。医療文脈では、demo の印象と clinical validity のあいだに大きな隔たりがあります。番組内でも Theranos を連想するような冗談が出ていましたが、それは健全な反応です。AI が医療へ入るときは、モデルが面白いかより、偽陽性・偽陰性、規制承認、責任の所在、導入現場のワークフローまで含めて見ないといけません。とはいえ、AI ネイティブ企業が「画面の中だけ」で終わらず physical product にまで踏み出していること自体は、業界の裾野の広がりを感じさせます。

### Sakana Fugu が示すのは、1モデル時代ではなく routing 時代の到来

終盤で扱われる Sakana Fugu は、複数モデルを背後で使い分ける multi-agent orchestration system として紹介されます。番組中でも OpenRouter や Factory の routing 的な動きと並べて語られており、ホストの結論はかなり明快です。今後は「1つの最強モデルですべてを処理する」より、「タスクの性質ごとに最適モデルへ振り分ける」ほうが現実的になる、ということです。

<https://x.com/SakanaAILabs/status/2068861630327443966>

この考え方は、すでに多くの現場で自然に起きています。例えば、難しい設計判断は高価な frontier model、定型的な情報抽出は安価な open model、会話要約は高速モデル、といった具合です。これを人間が都度選ぶ代わりに、ルータや compound system に自動化させるわけです。

ただし、この構成は benchmark の読み方を難しくします。もし system が「ベンチマークでこのタスクに強いモデル」を都度選ぶなら、全体スコアは当然上がりやすくなります。だから「単一モデルとして強い」のか、「上手く振り分けているから強い」のかを分けて見る必要があります。これは悪いことではなく、むしろ実務には近いです。ただ、leaderboard の数字だけを見てモデルそのものの進歩だと解釈すると、少し誤読しやすくなります。

今回の Agents Hour を通して見えてくるのは、モデル選択がますます orchestration の問題になっていることです。どのモデルが一番賢いかより、どの仕事をどのモデルに任せ、その切り替えをどこで管理するかのほうが重要になってきています。エージェント時代の競争は、モデル単体の IQ から、システム全体の配車設計へ移っているのかもしれません。
