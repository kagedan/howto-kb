## はじめに

こんにちは！株式会社フォトラクションCTOの中村です。

株式会社フォトラクションでは、建設生産支援クラウド「Photoruction」を通じて、建設現場の写真・図面・検査記録・書類など、現場に蓄積されるさまざまな情報を扱っています。

建設現場には、日々多くの情報が蓄積されます。

写真、図面、検査メモ、チェックリスト、報告書。  
これらはすべて重要な情報ですが、現場で本当に難しいのは、それらを単に保存することではなく、必要なタイミングで判断に使える形につなげることです。

たとえば、複数の情報を見比べながら、**「今、何を確認すべきか」「次に誰へ何を依頼すべきか」「この判断にはどのような根拠があるのか」** を整理する必要があります。

今回、Microsoft Agent Hackathon powered by Tokyo Electron Deviceに応募した理由も、まさにこの課題意識があったからです。

ハッカソンのテーマである **業務現場の課題をAIエージェントでどう解決するか** は、私たちが日々向き合っている建設現場の課題と非常に相性が良いと感じました。

生成AIは、単に質問に答えるだけでなく、業務データを読み取り、判断を補助し、次のアクションにつなげる可能性を持っています。

特に建設現場のように、写真・図面・書類・現場判断が複雑に絡み合う領域では、AIエージェントが **情報をつなぎ、人の判断を支援する** 余地が大きいのではないかと考えています。

そこで今回、建設現場における「確認・判断・依頼」のプロセスを **AIエージェントでどこまで支援できるか、まず試してみる** ために、**Genba Live Agent** というプロトタイプを開発しました。

題材にしたのは、**翌日にコンクリート打設を控えた配筋検査** です。

建設業界では、鉄筋を組む工程を「配筋」、組まれた鉄筋の間にコンクリートを流し込む工程を「打設」と呼びます。  
打設後は鉄筋が見えなくなるため、打設前の配筋検査は、やり直しが効きにくい重要な工程です。

今回のプロトタイプでは、この判断プロセスを支援するために、7体の専門AIエージェントで役割を分担し、さらにAIの出力品質を確認しながら改善する **Live Workspace Agent（Meta-Orchestrator）** という仕組みも試しました。

なお、**今回作ったものはあくまでハッカソン向けのプロトタイプ** であり、現場業務でそのまま使えるレベルにはまだまだ磨き込みが必要です。

ただ、図面・写真・検査メモ・チェックリストといった複数の情報をAIエージェントがつなぎ、確認すべき観点や次のアクションを整理するアプローチを実際に組んでみたことで、**今回試した要素を磨き上げていけば、建設現場の判断支援に活かせる可能性が見えた** という手応えがありました。

このブログでは、その取り組みで何を試し、何が見えてきたかをご紹介します。

## 作ったもの

**Genba Live Agent** は、現場の図面・写真・検査メモ・チェックリストを統合し、チャット主導の **Live Workspace** として体験できるWebアプリケーションのプロトタイプです。

主な特徴は次の通りです。

* チャット1ラウンドごとにWorkspaceが動的に変化する
* 図面上に番号付きリスクピンを表示し、クリックで詳細を確認できる
* 写真の判定バッジ、関連写真、確認依頼文、レポートが会話に応じて更新される
* 7体の専門AIエージェントが、図面解析・写真解析・証跡突合・リスク整理・依頼文生成を分担する構成を試作
* Meta-OrchestratorがAI出力の品質を客観指標で測り、必要に応じて再生成するアプローチを試行

題材として、RC造共同住宅新築工事の構造図PDFを使い、2階躯体打設前の配筋検査を再現しました。

このデモで見せたかったのは、AIが単に自然文で回答することではありません。

**AIが業務データとつながり、図面・写真・リスク・確認依頼を同じ画面上で組み立てていく体験**、そして **AIが自分の出力品質を確認するアプローチ** が、業務応用にどう活かせそうかを試作・検証するのが今回のテーマでした。

## なぜ作ったのか

建設現場では、写真・図面・メモ・帳票が日々大量に発生します。

これらの情報は、ひとつひとつを見るだけでも価値があります。

しかし、本当に重要なのは、それらをつなげて判断することです。

たとえば、以下のような判断が必要になります。

* この写真は図面のどの部分を写しているのか
* 図面で求められている確認項目は、写真で十分に確認できているのか
* 打設前に追加で確認すべきリスクは何か
* 現場担当者には、何をどのように依頼すればよいのか
* 後から見たときに、なぜその判断をしたのか追えるか

一方で、現実には以下のような課題があります。

* 情報が複数の場所に分散していて、関連付けが見えにくい
* 確認すべきポイントが担当者の経験に依存しやすい
* 問題発見後の確認依頼や報告作成に時間がかかる
* 後から「なぜその判断をしたのか」を追いにくい
* AIに依頼しても、その出力が業務判断に耐えるかを評価しにくい

特に最後の点が、今回のプロトタイプで一番試したかったテーマです。

建設現場では、AIが「問題ありません」と言い切ったとしても、それをそのまま信じるわけにはいきません。  
打設後は見えなくなる情報があるからこそ、AIの生成物には **根拠と品質の手がかり** が必要だと考えました。

そこで本プロトタイプでは、AIが出力するだけでなく、AI自身が **Coverage / Grounding / Actionability** の3軸で出力品質を測り、必要に応じて改善するというアプローチを実装し、その手応えを確かめてみました。

## アーキテクチャ

Azure Container Apps（FastAPI）と Azure Static Web Apps（Next.js）をベースに、Azure AI Document Intelligence、Azure OpenAI Service、Azure Cosmos DB、Azure Blob Storage を組み合わせた Azure ネイティブ構成を採用しました。

![](https://static.zenn.studio/user-upload/877787e045cb-20260601.png)

### 使用したAzureサービス

| サービス | 役割 |
| --- | --- |
| Azure Static Web Apps | Next.js 14 フロントエンドのホスティング |
| Azure Container Apps | FastAPIバックエンドのホスティング |
| Azure Cosmos DB | プロジェクト・解析結果・チャット履歴の永続化 |
| Azure Blob Storage | 図面PDF・写真・チェックリスト・成果物の保存 |
| Azure AI Document Intelligence | 図面PDFからテキスト・表・位置情報を抽出 |
| Azure OpenAI Service | Vision / Function Calling / 各AIエージェントの推論 |
| Bicep | Infrastructure as Code |

### 処理の流れ

大まかな処理の流れは以下です。

1. 図面PDF・写真・チェックリスト・検査メモをAzure Blob Storageに保存
2. Azure AI Document Intelligenceで図面から構造化情報を抽出
3. 7体の専門AIエージェントが並列で初期解析
4. 各エージェントの完了タイミングでSSEによりフロントエンドへリアルタイム配信
5. Web UI上で、図面・写真・AIの判断結果・リスク・確認依頼文を連動表示

SSEは Server-Sent Events の略で、サーバーからブラウザへ一方向にイベントを送る仕組みです。  
今回のように、AIエージェントの処理結果を「終わった順に画面へ反映する」用途では、WebSocketよりもシンプルに扱えます。

## マルチエージェント設計

本プロトタイプの大きな特徴は、単一のLLM呼び出しで完結させず、**7体の専門AIエージェントが協調動作する設計** を試したことです。

| # | Agent | 役割 |
| --- | --- | --- |
| 1 | Drawing Intelligence Agent | 図面PDFから部位・符号・確認項目を抽出 |
| 2 | Photo Intelligence Agent | 写真ごとに施工状況・証跡候補・黒板情報を抽出 |
| 3 | Evidence Mapping Agent | 図面の確認項目と写真証跡を突合し、supported / weak / missingで判定 |
| 4 | Evidence Evaluator Agent | AI出力に対し Coverage / Grounding / Actionability の3軸の客観指標を算出 |
| 5 | Action Composer Agent | 不足証跡に対し、現場担当者向けの追加撮影指示文を生成 |
| 6 | Risk Composer Agent | 評価結果から打設前リスクカードと確認依頼文を集約生成 |
| 7 | Conversation Planner Agent | ユーザー発話を解釈し、次に実行すべきアクションを決定 |

なぜ単一LLMで済ませず、7体に分割したのか。理由は3つあります。

### 1. 責務分割によるプロンプトの簡潔化

図面を読む、写真を読む、証跡を突合する、依頼文を作る、出力品質を測る。  
これらを1つの巨大なプロンプトに詰め込むと、指示が複雑になり、出力が不安定になります。

そこで、各Agentの責務を分けることで、プロンプトを短く保ち、出力品質を安定させる方向を試しました。

### 2. 並列化による性能向上

図面解析と写真解析は、互いに独立して実行できます。  
そのため、Drawing Intelligence Agent と Photo Intelligence Agent は並列に起動し、初期解析時間を短縮しています。  
ハッカソンのデモでは、精度だけでなく「待たされすぎない体験」も重要だと感じました。

### 3. 必要なAgentだけを再実行できる

チャット中の再評価では、毎回すべてのAgentを動かす必要はありません。  
たとえば、図面と写真の初期解析が終わっていれば、Conversation Planner、Evidence Mapping、Action Composerだけを再実行すればよいケースがあります。

必要なAgentだけを動的に呼び出せる構造にすることで、応答速度と再利用性の両立を狙いました。

なお、現状の実装ではプロトタイプとして最低限の連携を組んだ段階で、Agent 間の役割分担をさらに磨き込む余地は多くあると感じています。

## AI出力に客観指標を付与する仕組み

建設現場では、AIが誤って「問題なし」と言い切ることは避けなければなりません。  
そこで本プロトタイプでは、**Evidence Evaluator Agent** がAI出力に対して3つの客観指標を **算出** するアプローチを試しました。

| 指標 | 意味 |
| --- | --- |
| Evidence Coverage | 図面から抽出した確認項目のうち、写真証跡で支持されている割合 |
| Grounding | AIが行った主張のうち、入力データに根拠があるものの割合 |
| Actionability | 出力された指示が、現場担当者の次の行動につながる具体性を持っているか |

Evidence Evaluator はあくまで **「測る」役割** に徹しています。  
これらの数値があることで:

* 現場担当者は「AIがどれだけ根拠を持って判断したか」を一目で把握しやすくなる
* **次に登場する Meta-Orchestrator が「品質基準に達しているか」を機械的に判定する材料を得る**

つまり Evidence Evaluator は、AI出力の客観品質を可視化する **「ものさし」** の役割です。  
そのものさしを使って「再生成するかどうか」を判断する役割は、後述の Meta-Orchestrator が担います。  
**「測る」と「制御する」を分離** するアプローチをとることで、AIが自分で品質を改善するループを試作することができました。

なお、3つの指標自体も今回はシンプルな算出ロジックにしており、業務利用に向けてはより精緻な定義・閾値調整が必要だと考えています。

## Live Workspace Agent：AIの出力品質を確認するアプローチを試す

今回のプロトタイプで特にこだわって試したのが、**Live Workspace Agent（Meta-Orchestrator）** です。

Genba Live Agent では、7体の専門Agentが図面解析、写真解析、証跡マッピング、確認依頼文の生成などを分担しています。

ただし、それだけだと「複数のAI処理を順番に実行するワークフロー」に近くなります。  
建設現場の確認業務で重要なのは、AIが何かを生成した後に、**その出力が判断材料として十分かどうか** を確認することだと考えました。

たとえば、確認依頼文が生成されたとしても、

* 根拠となる写真や図面に十分に基づいているか
* 現場担当者が次に何をすべきか具体的に書かれているか
* 不足している情報がある場合、それが明示されているか

といった観点が欠けていると、実際の業務では使いにくくなります。

そこで、AIの出力品質を観察し、必要に応じて再生成や再評価を行う役割として、Live Workspace Agentを試作しました。

一言でいうと、**AIが自分の出力をそのまま出しっぱなしにせず、品質を確認してからユーザーに届けるアプローチ** をどこまで成立させられるかを試した仕組みです。

ここで、「ユーザーがチャットで『もっと良くして』と頼めば同じでは？」と思うかもしれません。

しかし、ChatとMeta-Orchestratorは役割が異なります。

| 観点 | Chat | Meta-Orchestrator |
| --- | --- | --- |
| 起動条件 | ユーザーの指示で起動 | AI側の判断で自動起動 |
| 判定指標 | 自然文・主観に寄りやすい | Coverage / Grounding / Actionabilityなどの客観指標 |
| ループ性 | 1往復で完結しやすい | 必要に応じて複数回反復 |
| 目的 | ユーザーの意図に応答する | AI出力の品質を確認し、必要なら改善する |

Chatは、ユーザーの依頼に応えるための入口です。  
一方でMeta-Orchestratorは、AIが生成した結果を見て、品質基準に達しているかを確認し、足りなければ必要なAgentを再実行する役割を持ちます。

ユーザーは依頼文を読んで「なんとなく良さそう」と感じることはできます。  
しかし、Coverageが十分か、Groundingに問題がないか、Actionabilityが基準に達しているかを毎回判断するのは簡単ではありません。

そのため、Meta-Orchestratorがユーザーの主観判断を待たずに品質を確認し、必要なら再生成を行うアプローチを狙いました。

役割を分けると、以下のようなイメージです。

> Chatは「人間の指示に応えるAI」。  
> Meta-Orchestratorは「AI出力の品質を確認し、必要なら改善するAI」。

### 「AI側の判断」とは具体的に何か

Meta-Orchestratorでは、各反復で大きく3つの判断を行います。

#### 1. 起動するかどうか

初期解析や依頼文生成の結果を見て、品質指標が基準に達していなければMeta-Orchestratorを起動します。  
すでに基準を満たしていれば、追加の改善ループは回さず終了します。

#### 2. 次に何を実行するか

品質指標と過去の反復履歴をもとに、次にどのAgentを呼び出すべきかを選択します。

たとえば、以下のような判断です。

* Coverageが低い場合は、Evidence Mappingを再実行して証跡の対応関係を見直す
* Actionabilityが低い場合は、Action Composerを再実行して撮影指示や確認依頼文を具体化する
* 依頼文を改善した後は、Evidence Evaluatorを再実行して品質指標を再計算する
* これ以上改善できない場合は、無理に続けずcompleteを選ぶ

ここでは、LLM Function Callingを使って、現状の指標、過去の実行履歴、実行済みアクションをもとに次のアクションを選ばせています。

#### 3. 続けるか止めるか

改善ループは、ずっと回し続ければよいわけではありません。  
Meta-Orchestratorは、反復履歴を見ながら、目標達成、進捗なし、時間上限、最大反復回数などをもとに継続・停止を判断します。

特に建設現場では、AIだけでは埋められない情報もあります。  
その場合は、無理に「問題なし」と言い切るのではなく、現地で追加確認が必要であることを明示して終了する方向で設計しました。

つまり、Meta-Orchestratorは単にループを回すだけではなく、**何を改善すべきか、もう十分か、これ以上は現場確認が必要か** を判断する役割を担います。

ここを丁寧に設計できれば、単なる固定ワークフローと差別化できる手応えを得ました。  
ただし、今回のデモでは判断ロジックや指標の閾値はまだシンプルなレベルにとどまっており、本格的に業務で使うにはさらに作り込みが必要だと感じています。

### 実装イメージ

Meta-Orchestratorは、以下のような流れで動作します。

```
[初期解析]
Drawing → Photo → Mapping → Action → Eval
                ↓
[Live Workspace Agent / Meta-Orchestrator]
observe  : 現状のCoverage / Grounding / Actionabilityを取得
decide   : Function Callingで次に実行するAgentやActionを選択
dispatch : 必要な専門Agentを呼び出して再生成
publish  : SSEでUIに進捗を配信
                ↓
最大反復回数、時間上限、目標達成条件により終了
```

たとえば、確認依頼文を生成したときにActionabilityが十分でなければ、Meta-OrchestratorがAction Composer Agentを再実行し、撮影方法や確認観点をより具体化します。  
その後、Evidence Evaluator Agentで再評価し、基準を満たしていればユーザーに提示する、という流れです。

ユーザーから見ると、単に確認依頼文が表示されているだけに見えるかもしれません。  
しかし裏側では、AIが自分で品質をいったん確認してから出力する、という体験を作ろうとしています。

この **「生成して終わり」ではなく「評価して、必要なら改善する」** というループが、今回のプロトタイプで最もAgenticに振った部分です。

### 暴走を防ぐガードレール

自律ループには便利さがある一方で、同じ処理を繰り返してしまうリスクがあります。  
そのため、最低限のガードレールを入れています。

* 直前のアクションで改善しなかった場合、同じアクションを繰り返さない
* 実行済みアクションの履歴を参照する
* 複数回改善が見られない場合は早期終了する
* 最大反復回数と時間上限を設ける
* 最終的に追加情報が必要な場合は、無理に改善せず「現場確認が必要」と明示する

AIに自律的な改善を任せつつ、無限ループや過剰生成を避ける方向を意識しました。  
このあたりも、業務利用に向けてはさらに磨いていきたい部分です。

## UI / UXで工夫したこと

今回のデモでは、チャットだけでなく、図面を中心としたLive Workspaceとして体験できるようにしました。

### 集中モード

画面は大きく、チャットと図面ビューアで構成しています。

```
┌──────────────┬─────────────────────────────────────┐
│              │                                     │
│              │           Drawing Viewer            │
│  ChatPane    │   図面 + 番号付きリスクピン          │
│  履歴 + 入力欄 │                                     │
│              │   確認依頼文 / レポート              │
│              │                                     │
└──────────────┴─────────────────────────────────────┘
```

図面を画面の主役にし、チャットは操作と説明の入口にしました。  
リスクピンをクリックすると、関連写真、推奨対応、図面根拠を含むリスクカードモーダルが表示されます。

### Workspaceが会話に応じて変化する

チャットの各ラウンドに応じて、Workspaceの表示が動的に変わります。

| Round | チャット入力 | Workspaceで起こること |
| --- | --- | --- |
| 1 | 6枚の写真をOK/リスク/再撮影に判定 | 図面に番号付きリスクピンが点灯し、リスクカードを生成 |
| 2 | リスクありの写真を深掘り | 対象ピンが強調され、関連写真と根拠を表示 |
| 3 | 再撮影指示を作成 | 再撮影対象のピンを強調し、撮影指示文を生成 |
| 4 | 確認依頼文を作成 | Meta-Orchestratorが裏で発火し、品質確認後に依頼文を表示 |
| 5 | レポートにまとめる | 打設前確認レポートをモーダルで表示 |

この構成により、ユーザーはAIの回答を読むだけでなく、図面や写真と結びついた判断プロセスを追える体験を狙いました。

ただし、現状のUIはあくまでデモ向けに最適化したもので、本番の業務UIにするには項目の選定や情報密度をさらに調整していく必要があります。

## プロンプトの工夫

### 1. 役割を明確にする

各Agentには、汎用アシスタントではなく、建設現場の検査補助を行う専門エージェントとして振る舞うように指示しています。

```
あなたは建設現場の配筋検査を支援するAIエージェントです。
図面、現場写真、検査チェックリスト、現場メモをもとに、
打設前に追加確認すべきリスクと確認依頼文を整理してください。
```

### 2. 判断の前提情報を分けて渡す

図面情報、写真情報、チェックリスト、現場メモを明確に分けて渡すことで、AIが根拠を整理しやすくしています。

```
# 図面情報
- 対象部位:
- 図面上の指示:
- 関連する符号・寸法:

# 写真情報
- 写真ID:
- 撮影箇所:
- 写真から読み取れる状況:

# チェックリスト
- 確認項目:
- 期待される状態:

# 現場メモ
- 担当者メモ:
- 懸念事項:
```

### 3. 出力形式をJSONで固定する

AIの回答が毎回ばらつくと、UI連動が不安定になります。  
そのため、出力形式をJSONで固定し、バックエンドとフロントエンドの両方で型検証できるようにしました。

これにより、AIの回答をそのまま以下のUIに展開できます。

* チャット欄
* 図面上のリスクピン
* リスクカード
* 確認依頼文
* レポート
* 品質評価パネル

Meta-Orchestratorでは、同じアクションを繰り返さないよう、履歴を参照して次のアクションを決めるようにしています。

```
判断ルール:
1. history と actionsAlreadyTried を参照する
2. 直前のアクションでCoverageが改善しなかった場合、同じアクションは選ばない
3. 全アクションを試しても改善しない場合はcompleteを選ぶ
4. Coverageが目標を超え、Actionabilityがhighならcompleteを選ぶ
```

LLMに判断を任せつつ、コード側でも履歴や終了条件を持たせることで、安定した自律ループを目指しました。  
プロンプト設計についても、まだ磨き込みの余地が大きい領域だと感じています。

## Azure AI Document Intelligenceの活用

建設図面は、一般的な文書とはかなり性質が異なります。

文字が小さく、記号や符号が多く、表、注記、図形、寸法線が混在しています。  
さらに、図面上の位置関係そのものが意味を持ちます。

そのため、単にテキストを抽出するだけでは不十分です。  
重要なのは、**どの情報が、どの図面上のどの位置にあるのか** を扱うことです。

今回のプロトタイプでは、Azure AI Document Intelligence の `prebuilt-layout` を活用し、図面に含まれるテキスト・表・位置情報を抽出しました。  
抽出したポリゴン座標を後続の Evidence Mapping Agent が利用することで、図面上の確認項目と写真証跡を紐づけ、より現場文脈に近い回答を生成できるアプローチを試しています。

建設図面そのものは多様なため、本番運用ではさらにカスタムモデル化や前処理パイプラインの整備が必要になる領域です。

## パフォーマンスへの取り組み

本プロトタイプでは、以下を設計目標にしました。

* 初期解析：60秒以内
* チャット応答：10秒以内
* SSE初回イベント：5秒以内

これを達成するため、以下の最適化を行いました。

| 最適化 | 内容 |
| --- | --- |
| Drawing × Photo 並列起動 | 独立した処理を同時実行し、初期解析時間を短縮 |
| 写真解析の並列度制御 | 並列数を制御し、レート制限を回避 |
| Vision入力の最適化 | 速度とコストのバランスを取る |
| Cosmos DBへの状態保存 | 再起動後も解析状態を復元しやすくする |
| 指数バックオフ | 一時的なAPIエラーに対してリトライする |
| APIキーマスキング | ログ出力時に機密情報が残らないようにする |

特にSSEによる段階的な更新は、体験面で効果がありました。  
すべての処理が終わるまで待たせるのではなく、完了した解析結果から画面に反映することで、AIが作業を進めている様子を自然に感じられるようにしています。

## 苦労した点

### 1. AI出力の「品質」を客観的に測ること

最初の難所は、AIの出力が打設前判断に耐えるかをどう測るかでした。  
AIが「主筋本数を確認しました」と出力したとしても、それが本当に判断材料として十分なのかは別問題です。

人間の主観だけに頼るのではなく、Coverage、Grounding、Actionabilityの3軸を品質指標として定義し、AI自身に算出させる仕組みをEvidence Evaluator Agentとして実装しました。  
ただし、指標の閾値や定義はまだプロトタイプレベルで、実業務に合わせた校正が必要です。

### 2. Agenticと呼べる設計にすること

7体のAgentを順番に動かすだけでは、単に複数のLLM呼び出しを制御するWorkflowに近くなります。

Agentらしさを出すには、AI自身が状態を観察し、次に何をすべきかを判断し、必要なツールやAgentを選択する必要があります。  
そこでLive Workspace Agent（Meta-Orchestrator）を導入し、初期解析後にAIが自分で品質を測り、必要に応じて再生成するループを組み込みました。

「複数のLLM呼び出しを順番に流すワークフロー」を、少しでも **AIがプロセスの一部を自律的に制御するMulti-Agent Systemに近づける** ことを目標に設計しました。  
完全な自律にはまだ距離がありますが、その方向性を体験として示せた手応えはあります。

### 3. デモの「動的さ」を体験として伝えること

3分の応募動画で、AIが自律的に動いていることをどう見せるかも悩みました。  
最終的には、以下の3段階でストーリーを構成しました。

* チャット1ラウンドごとにWorkspaceが変化する
* AIが必要に応じてリスクや確認観点を深掘りする
* 確認依頼文生成時にMeta-Orchestratorが裏で発火し、品質確認のプロセスを経て出力される

「対話 → 自律改善のアプローチ → 品質確認を経た出力」という流れを、短いデモの中で伝えられるように構成しました。

## 今回の取り組みで見えてきたこと

短いハッカソン期間でのプロトタイプとして取り組んだ結果、いくつかの手応えと、まだ磨き上げが必要な点が見えてきました。

### 手応えを感じた部分

* 図面・写真・チェックリスト・メモを **1つのワークスペースに統合する** ことで、現場担当者が判断するための情報の塊が作れる
* AI出力に **客観指標を付与する** ことで、出力をそのまま信じるかどうかの手がかりが生まれる
* **Meta-Orchestrator的な仕組み** を入れると、「AIが生成して終わり」ではなく「品質を確認してから出す」体験を作れる可能性がある
* チャットと図面・Workspace を連動させると、AIとのやりとりが **業務の判断プロセスに近い体験** になりそう

### これから磨いていきたい部分

* 品質指標（Coverage / Grounding / Actionability）の **定義と閾値** を、実業務の確認観点に合わせて精緻化
* Meta-Orchestratorの **判断ロジックとガードレール** を、現場のさまざまなパターンに耐えるよう拡張
* Drawing / Photo Intelligence の **精度** を、実際の図面・写真にあわせてチューニング
* UI / UX を、ハッカソン向けデモ最適化から **本番の業務利用** に向けて整え直す
* Photoruction本番データとの連携、Microsoft Teams / Copilot Studio / Power Platform との接続

今回試した要素はあくまで第一歩ですが、これらを磨き上げていくことで、**建設現場の確認・判断・依頼のプロセスにAIエージェントを組み込んでいく** ための足がかりになりそうだと感じています。

ハッカソン応募作品としての位置づけを離れて、現場業務への適用を考えると、以下のような方向性があり得ます。

## まとめ

Microsoft Agent Hackathon 2026 では、建設現場の確認・判断・依頼プロセスを支援する **Genba Live Agent** をプロトタイプとして開発しました。

今回試したポイントは以下です。

* 7体の専門AIエージェント + Meta-Orchestratorによるマルチエージェント構成
* Coverage / Grounding / Actionabilityの3軸でAI出力を測る仕組み
* AIが自分で出力品質を確認・改善するアプローチ（Agentic Loop）の試行
* チャットとMeta-Orchestratorの補完関係の整理
* チャット1ラウンドごとにWorkspaceが動的に変化するLive Workspace UX
* Azure Container Apps / Static Web Apps / Cosmos DB / Blob Storage / Document Intelligence / Azure OpenAIによるAzureネイティブ構成

特に手応えを感じたのは、**AIが自分の出力に客観指標を付け、必要なら自分で改善する** というアプローチです。  
ユーザーがチャットで「もっと良くして」と頼むだけでは代替しにくい、人間の主観判断を待たない品質確認の流れを実装してみることで、Agent らしい動きの輪郭が見えてきました。

建設現場のように、やり直しが効きにくい判断が求められる領域では、AIの出力に客観的な品質評価を付ける発想がより重要になると考えています。

今回のプロトタイプはまだ第一歩で、業務でそのまま使うには磨き上げが必要な要素もたくさんあります。  
それでも、今回試した要素を一つひとつ磨き上げていけば、**建設現場の確認・判断・依頼のプロセスを、AIと人間が役割分担しながら進められる可能性** が見えてきた、というのが正直な手応えです。

今後も、建設現場の実務に寄り添ったAIエージェントの活用に、引き続き取り組んでいきたいと思います。
