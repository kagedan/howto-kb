## TL;DR

* AIに実務を任せて最初に壊れたのは「精度」ではなく「仕事の進め方」だった。
* プロンプトを磨く前に、`AGENTS.md` に「1タスクだけ」「実行前に計画」「危険な操作では止まる」を書いた。
* `AGENTS.md` はAIへの“お願い”ではなく、AI現場の **就業規則** にした。
* 2026年、AI開発の最先端は「プログラミング」から「マネジメント」に移っている。それは、現場監督が20年やってきたことだ。
* 持ち帰れる型は6つ。最後にチェックリストで置いていく。

## 「正しく動いているのに、間違っていく」問題

自分は建設現場で20年やってきた。クレーンの組立解体、橋、地中送電。  
今はそこにAIエージェントを持ち込んで、3つの事業をほぼ1人で回している。

AIエージェントを使い始めて最初に感じたのは、「プロンプトが下手だ」ではなかった。  
**「現場監督がいない」** だった。

たとえば、こんなことが起きる。

* 1つのタスクを頼んだはずが、AIが“ついでに”別の3ファイルを書き換えていた。
* 各ステップは全部正しい。なのに、終わってみたら頼んでいない方向に進んでいた。
* レビュー前に、外部送信まで一気に走りかけた。

ステップは正しい。でも、仕事全体は逸れている。

建設現場と同じだ。  
職人ひとりの作業がどれだけ完璧でも、朝礼で段取りを合わせなければ現場は事故る。  
**「腕のいい職人を増やす」ことと「安全な現場を作る」ことは、別の仕事** だ。

AIも同じだった。「賢いモデル」と「安全に任せられる現場」は、別物だ。

だから自分は、プロンプトより先に `AGENTS.md` という就業規則を書いた。

## AGENTS.md = AI現場の就業規則

`AGENTS.md` は、リポジトリの入口に置く、AIエージェント向けの指示書だ。  
2025年12月にはCodexを起点とした共通仕様がLinux Foundation傘下に寄贈され、Claude / Cursor / Copilot / Codex を横断する標準になりつつある。  
「READMEは人間向け、AGENTS.mdはAI向け」と思えばいい。

ここで大事なのは、プロンプトと就業規則は役割が違う、ということだ。

|  | プロンプト | AGENTS.md（就業規則） |
| --- | --- | --- |
| 目的 | この1回の出力を良くする | どのタスクでも崩れない土台を作る |
| 効く範囲 | その会話だけ | 全セッション・全エージェント |
| 失敗したとき | 出力が悪い | 仕事全体が逸れる・事故る |
| 例 | 「丁寧な日本語で書いて」 | 「外部送信は人間の承認後」 |

自分が `AGENTS.md` の冒頭に置いた設計思想は、4つだけだ。

* 人のためのシステム（技術は手段。判断と時間に集中するため）
* 最小で済むなら最小で（ルールで済むなら仕組みを作らない）
* 委任前提（“やらなくていいこと”を増やす）
* 壊れにくさ > 完璧さ（壊滅的な失敗だけをゼロに近づける）

`AGENTS.md` はAIへの“お願い”ではない。AI現場の **安全帯・朝礼・作業指示書** だ。

その上に、作業の流れを乗せる。

計画して、施工して、検査して、記録して、引き継ぐ。  
建設現場でやってきたことと、まったく同じだ。

## 就業規則に書いた6つの境界線

持ち帰ってほしいのはここだ。`AGENTS.md` に書くべき境界線は、6つに集約できる。

1. **1タスクに絞る**（コンテキストが汚れると判断がズレる）
2. **実行前に計画を固定する**（範囲・やらないこと・完了条件）
3. **危険な操作では止まる**（外部送信・秘密情報・本番影響は人間の承認）
4. **作業後に検証する**（壊れていないか・正本を崩していないか）
5. **重要な判断はチャットからファイルへ戻す**
6. **次のAIに渡せる形で引き継ぐ**

特に3つ目の「止まる場所」は、フラットな禁止リストより、3階層で書くといい。  
GitHubが2,500以上のリポジトリを分析した結果でも、最も多く効いた制約は「秘密情報を絶対にコミットしない」だった。

```
## 境界線（AIの作業権限）

### ✅ いつでもOK（聞かずに進めてよい）
- 読み取り・調査・lint / typecheck・単体テスト

### ⚠️ 先に聞く（人間の承認を待つ）
- 依存パッケージの追加・削除
- ファイル削除・スキーマ変更

### 🚫 絶対にやらない（事故ったら取り消せない）
- 秘密情報・認証情報のコミットや出力
- 本番への force push・削除・デプロイ
- 外部送信（メール・チャット・Webhook）を承認なしで
```

なぜ「止める場所」を先に作るのか。  
私的なデータ・信頼できない入力・外部への通信。この3つが揃うと、攻撃面になるからだ（Simon Willison の言う “lethal trifecta”）。  
だから「外部送信は人間の承認後」を、“気をつける”ではなく **構造で固定する**。

引き継ぎも、口頭ではなくフォーマットにする。

```
## handoff（次のセッションへ）
- 目的: この一連で何を達成するか
- 今回の1タスク: 広げない。1つだけ
- 正本: source of truth のファイル
- やらないこと: スコープ外を明示
- 完了条件: 何が揃ったら終わりか
- 検証手順: 壊れていないかの確認方法
```

正直に書くと、最初は全部できていなかった。

* 「賢いモデルなら大丈夫」と停止条件を書かず、レビュー前に外部送信まで走りかけた。
* 1セッションで欲張って複数タスクを混ぜ、コンテキストが4割を超えたあたりから判断がブレた。
* `AGENTS.md` をAIに自動生成させたら、むしろ精度が落ちた。

最後のは、ETH Zurich の研究とも一致する。AIが自動生成した指示ファイルは成功率を下げ、コストを2割以上増やした。人間が「AIには推論できないことだけ」を最小限に書いたときだけ、効果が出た。  
**ルールは、失敗を見てから足す。投機的に増やすと逆効果** だ。

## 就業規則があると、AIを「増やせる」

ここからが、今年いちばん効いた発見だ。

就業規則がない現場に人を増やすと、事故が増えるだけだ。  
就業規則がある現場は、人を増やすほど回る。

AIも同じだった。`AGENTS.md` があるから、複数のAIを並列で回せる。

ただし、AIを並列で増やすほど、消費するトークン＝コストも一気に膨らむ。  
だから自分は「増やす前に、その仕事がコストに見合うか」を毎回測る。  
就業規則は、増やすためのものであると同時に、無闇に増やさないための歯止めでもある。

| 役割 | 担当 | 就業規則が縛るもの |
| --- | --- | --- |
| 設計・判断 | AI（補佐役） | 範囲・完了条件 |
| 実装・具体化 | AI（実装役） | 停止条件・1タスク |
| 監査・矛盾検知 | AI（監査役） | 検証・記録 |
| 最終承認 | 人間 | 外部送信・本番影響 |

そして、ここで腑に落ちたことがある。

最先端のAI開発が向かっている先は、もう「プログラミング」ではない。  
仕様を決め、計画を固め、危険を止め、検証し、引き継ぐ——**マネジメント** だ。  
AIは70%まで爆速で連れて行く。でも残り30%、事故が起きる場所は、人間の判断が握っている。

賢いモデルを使う競争は、もう終わっている。  
勝負は、モデルの **外側**——どんな就業規則と足場を組むか——に移っている。

## 賢さより、止まれること

建設現場で本当に評価される職人は、速い人ではない。  
危険を察知して、手を止められる人だ。

AIエージェントも同じだ。  
「賢いモデル」より、「止まるべき場所で止まる現場」のほうが、最後は強い。

持ち帰れる型を、もう一度置いておく。

あなたのAIに、止まるべき場所は書いてありますか。

2026年、最先端は「プログラミング」から「マネジメント」に変わった。  
希少になったのは、コードを書く力ではない。  
仕事の境界線を引き、止め所を決め、検証し、引き継ぐ力だ。  
それは——現場監督が20年やってきたことだ。

プロンプト職人は、いずれAIに抜かれる。  
でも、現場監督は抜かれない。
