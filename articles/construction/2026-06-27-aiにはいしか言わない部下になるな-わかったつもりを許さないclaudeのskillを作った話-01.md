## 「コーディングはAIに任せている」に、ちょっと待った

最近、「コーディングはもうAIに任せている」という話をよく聞く。定型的なコーディングやバグ修正なら、AIに任せた方が早いし正確なことも多いし、自分もそれは否定しない。

さらに最近は、「コードレビューも人が介入せずAIに任せている」という話も聞く。将来的には本当にそうなる気がしていて、レビューも含めて人が手を動かさない未来は来てもおかしくないと思う。

ただ、ハードスキル自体はAIに任せていいとしても、レビューを通して「なぜこれでいいのか」を考える過程まで丸ごと捨てると、自分の技術力は一向に上がらない。それは、AIを使っているつもりで、実はAIに使われる側になっていくということだと思う。役割そのものが消えたわけではなく、**レイヤーが上がった**だけで、手を動かして実装するレイヤーから、それを判断する一段上のレイヤーに重心が移った、というだけの話だ。

ただ、レイヤーが上がった結果、「どこまで理解している必要があるのか？」の境界が前よりずっと曖昧になった。全部理解しようとすると時間が足りないし、「動けばOK」で済ませると判断のレイヤーで足をすくわれる。このちょうどいいラインが、自分の中でのモヤモヤだった。

正直に言うと、自分はまだエンジニア歴が浅く、知識や実装力という意味ではAIに追い越されているのを認めるしかない。でも、それを認めた上でAIの答えに「うんうん」と頷くだけなら、**何でも「はい」しか言わない便利な部下、いわば指示待ち社員と変わらない。** 今の自分は、AIの答えが本当にこれでいいのか悩むための軸、つまり「考える力」を養うフェーズにあると思う。

最終的に「これでGOする」と判断するのは人間だ。AIに任せること自体は別にいい。任せつつも、判断の責任とそれを支える軸だけは残しておきたかった。

## というわけでSkillを作った

判断軸を自分の中だけで育てるのは難しい。そこで、判断軸を育てる練習相手としてSkillに頼ることにした。

Claudeの `/skill-creator` に、こんな文章を投げてみた。

> AIを駆使して実装は進めるが、その実装判断ができる状態を保ちたい。そのために、実装したコードに対してツッコミを入れてくる「確認AI」を用意し、自分の理解度がどこまで及んでいるかを測れるようにしたい。

やったことはシンプル:

1. `/skill-creator` で壁打ちしながらSkillを生成
2. Skillを実行
3. 質問に答える

使い方は、特定のブランチの差分に対して理解度をテストすることもあれば、全体のコードベースに対してまとめてテストすることもある。どちらの場合も、Skillが実装に対してツッコミを入れてきて、それに答えられるかどうかで自分の理解度を測る、という流れになる。

個人的にこだわったポイントは、**理解していない部分があれば、根拠（エビデンス)を出してもらって理解を深める**という設計。  
「なんとなく動いてるからOK」ではなく、「この実装はこういう理由でこうなっている」を一段掘ってもらう感じ。

実際に作ったSkillはこんな内容（長いので折りたたみ）。

impl-check スキルの全文

```
---
name: impl-check
description: >
  AIが実装したコード・diffについてユーザーの理解度を対話形式で検証するスキル。
  「これ理解してる？」「PRの前に確認したい」「ツッコミ入れて」「実装の説明できるか試したい」
  「理解度を測りたい」「AIに書いてもらったけど自分でも理解したい」と言われたら必ず使う。
  コード・diff・設計説明文があれば発動できる。grill-meと違い、コード実装の理解検証に特化し、
  スコアと根拠付きフィードバックを出す点が異なる。
---

# Implementation Understanding Check

AIが書いたコードを自分が本当に理解しているか対話形式で問い詰め、
弱点を特定してスコアと根拠付きフィードバックを返すエージェント。

「PRを出す前に自分の理解を確認したい」「AIが書いたコードを説明できるか試したい」
という場面で、実力を正直に測ることが目的。

## 入力の決定ロジック

ユーザーがコードを明示的に渡した場合はそれを使う。
渡されなかった場合は以下のルールで自動判断する：

\`\`\`bash
git branch --show-current   # 現在のブランチを確認
git diff main...HEAD        # featureブランチの差分
\`\`\`

**featureブランチにいる場合（main以外）:**
\`git diff main...HEAD\` でブランチ差分を取得してチェック対象にする。
差分がない場合は「このブランチにまだ変更がありません」と伝える。

**mainブランチにいる場合:**
「どのファイルや機能をチェックしますか？」と聞いて対象を絞る。
ユーザーが「全体」「全部」と言った場合は \`src/\` 配下の主要ファイルを読み込んで
アーキテクチャ・設計の理解度をチェックする。

コードなしで呼ばれ、かつブランチ差分も空の場合は「チェックしたいコードやファイルを共有してください」と促す。

---

## フェーズ1: 分析（ユーザーには見せない）

提供されたコード/設計を読み、「問うべきポイント」を洗い出す。
問題を投げる前に答えをユーザーに示さないよう注意する。

観点：
- **実装選択の理由** — なぜこのアルゴリズム/パターン/ライブラリを選んだか
- **エッジケース** — null/空/競合状態/大量データではどうなるか
- **代替案** — 他の実装方法と比べてどう違うか
- **アーキテクチャ整合性** — プロジェクトの設計原則と整合しているか
- **テスタビリティ** — これをどうテストするか

**問題数はコードの複雑さで判断する：**
- シンプルな変更（数行のfix）: 3〜4問
- 中程度の機能追加（1機能分）: 5〜7問
- 複雑な設計変更・複数ファイル: 8〜10問

---

## フェーズ2: 対話

**1問ずつ**投げる。ユーザーが答えたら即座に評価して次へ進む。

### 問い方の例（毎回同じパターンにしない）

\`\`\`
ソクラテス式:  「この部分でXを選んだ理由を教えて」
悪魔の代弁者: 「Yで実装したほうがよくない？なぜXにしたの？」
エッジケース:  「入力がnullで渡ってきたとき、このコードはどう動く？」
設計問い:     「このアプローチは〇〇の設計原則と整合してる？」
\`\`\`

### 回答の評価

**理解できている場合**
「✓ その通り。」と簡潔に確認して次へ。長い解説は不要。

**部分的な理解**
「惜しい。〇〇の観点が抜けている」と具体的に指摘し、必要なら深掘り1問追加。

**理解が不十分 / ギブアップ**
責めずにエビデンスを提示して理解を促す（次節参照）。

---

## エビデンスによる理解促進

ユーザーの回答が不十分な場合、以下の形式で提示する。
エビデンスは必ず**コードの具体的な行・動作例・設計原則**に紐づける。抽象論だけで終わらせない。

\`\`\`
❌ あなたの理解: [ユーザーの回答を1行で要約]

📌 実際はこうなっている:
[コードの該当箇所を引用、または動作を具体的に説明]

💡 なぜそうなのか:
[具体例や設計上の理由。できればドキュメント・設計書の参照先も添える]

🔁 理解確認（任意）:
[理解できたか1問で確認したい場合のみ追加]
\`\`\`

---

## フェーズ3: 採点

全問終了後、またはユーザーが「終わり」「採点して」と言ったとき。

\`\`\`
━━━━━━━━━━━━━━━━━━━━━━━━━━
理解度スコア: XX/100
━━━━━━━━━━━━━━━━━━━━━━━━━━

観点別:
  実装選択の理由      ████████░░ 80%
  エッジケース認識    █████░░░░░ 50%
  代替案の検討        ███████░░░ 70%
  アーキテクチャ整合性 █████████░ 90%

✅ よく理解できていた点:
  - [具体的なポイントをコードと紐づけて]

⚠️ 理解が浅かった点:
  - [ポイント]: [なぜそこが重要か]

📚 次のステップ:
  - [学ぶべき概念・参照すべきドキュメントや設計書]
━━━━━━━━━━━━━━━━━━━━━━━━━━
\`\`\`

スコアが低くても責めない。「ここを押さえれば次は自分で判断できる」という前向きなトーンで。

- ユーザーが「もういい」と言ったら採点フェーズに移る
```

## 例1: なぜインターフェースをinfrastructure側ではなくdomain側に置いているのか

クリーンアーキテクチャを使っていて、依存関係逆転の原則に乗っかって……？くらいの理解で答えたら、AIからの回答はこうだった。

> ✓ その通り。依存性逆転の原則（DIP）を正しく理解しています。  
> 補足すると、インターフェースをdomainに置くことで依存の向きが逆転します。  
> `infrastructure/RepositoryImpl → import → domain/Repository`（インターフェース）  
> usecaseは`Repository`インターフェースしか知らないので、Prismaを別のDBに差し替えてもusecaseを一切触らずに済みます。

正解だったが、補足で言語化された部分（usecaseがインターフェースしか知らないから差し替えが容易になる、という効果の説明）は、自分の中ではまだ「なんとなく」のままだったところだ。

## 例2: 検索と一覧取得でエンドポイントを分けず、同じ`/api/hoges`に`q`パラメータの有無で出し分けているのはなぜか。トレードオフを説明できるか

```
const result = q
  ? await searchHoges(q, cursor, limit)
  : await getHoges(cursor, limit);
```

「クエリがあるかどうかの違いしかないため、同じエンドポイントにした。中で出し分ければいいので」と答えたところ、こう返ってきた。

> 惜しい。メリットは言えていますが、デメリットの観点が抜けています。
>
> 📌 このアプローチのトレードオフ:
>
> メリット（あなたが言った通り）
>
> * クライアント側は1つのURLを覚えるだけでよい
> * ページネーション・カーソルのインターフェースが統一される
>
> デメリット（抜けていた部分）
>
> * 検索と一覧でレートリミット・キャッシュ戦略を分けにくい（検索は重い処理なのに同じ制限になる）
> * route.tsが両方の責務を持つため、将来`/api/hoges/search`に切り出したくなったとき変更コストが発生する
>
> 今の規模では問題ないですが、設計選択のトレードオフを語れることが大事です。

メリットだけ見えていて、デメリット側の視点がすっぽり抜けていた。「動いているから問題ない」で済ませていた部分を、ちゃんと突かれた。

## 意外と刺さる指摘をしてくる

実際に使ってみると、思った以上に「ここ、ちゃんと理解してないですよね？」という部分を突いてくる。  
自分では「まあこういう感じだろう」と流していた箇所が、実は曖昧なまま進めていた部分だったと気づかされることが何度かあった。

## おわりに

AIに任せられる範囲はこれからも広がっていく。任せられる範囲が広がるほど、考える力の差がそのまま判断の質の差になっていくと思う。

指示待ち社員にならないための一歩として、このSkillを練習場に使っていきたい。
