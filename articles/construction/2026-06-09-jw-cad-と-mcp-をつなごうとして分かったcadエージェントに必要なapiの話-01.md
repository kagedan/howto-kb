普段は AutoCAD 上で動く AI エージェントを個人で開発しています。  
ただ、制御は全部自前で持つ独自実装で、MCP は使っていません。

国内では **Jw\_cad** というフリーソフトの 2D CAD が建築・土木・電気の図面で広く使われていますが、こちらは AI と組み合わせる事例があまり見かけません。**Jw\_cad も AI と連携させてみたい**と前から思っていたものの、AutoCAD のように一から作り込むのはかなり時間がかかります。

そこで思い立ったのが **MCP サーバー** での実装でした。  
手早く試せそうですし、CAD と MCP の組み合わせは自分でも触ったことがなかったので、ちょうど良い題材になりそうです。

ところが、**ファイル単位の連携は問題なく動く**ものの、API がないために**外から能動的に Jw\_cad を動かす手段がない**ことが見えてきて、設計の段階で行き止まりに当たりました。

その記録です。

## 前提知識

両者の前提が揃うように、軽く整理しておきます。

### ◆ MCP サーバーとは

MCP (Model Context Protocol) は、AI と外部ツールをつなぐための仕様です。  
2024 年に Anthropic が公開し、今は OpenAI なども含めて広く使われ始めています。

簡単に言うと Claude Desktop のような AI アプリから「このツールを呼んでね」とお願いできる仕組みで、ツール側を MCP サーバーとして実装すれば AI と外部システムが連携できます。

たとえば、最小構成の MCP サーバーは Python だとこんな感じになります。

```
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("ping-server")

@mcp.tool()
def ping() -> str:
    """pingを送るとpongが返ってくる"""
    return "pong"

if __name__ == "__main__":
    mcp.run()
```

これを Claude Desktop 側に登録しておくと、AI が「ping ツール使うよ」と判断したタイミングで `ping()` が呼び出され、結果(`"pong"`)が AI に渡される、という流れです。

つまり、理屈の上では、これと同じように **Jw\_cad を操作する処理**をツールとして用意すれば、AI がそれを呼び出して Jw\_cad を動かせるはずです。

### ◆ Jw\_cad の外部変形とは

Jw\_cad には、ユーザーが自前のプログラムで図形操作を拡張するための仕組みとして **外部変形** という機能が用意されています。

動き方は次のとおりです。

1. ユーザーが Jw\_cad で図形を選択し、外部変形メニューからバッチを起動
2. Jw\_cad が `JWC_TEMP.TXT` というテキストファイルを書き出す  
   (中には選択された図形の座標などの情報が入っている)
3. バッチが指定された外部プログラムを呼び出し `JWC_TEMP.TXT` を読み込んで処理  
   (外部プログラムとしてはPythonやRubyなどが利用されることが多い)
4. 処理結果を `JWC_TEMP.TXT` に書き戻す
5. Jw\_cad がそのファイルを読み込んで図面に反映

![](https://static.zenn.studio/user-upload/e31b1be0eb39-20260425.png)

つまり、**テキストファイル経由でやりとりする**形の拡張機構です。  
公式 API は存在せず、これが Jw\_cad における事実上唯一の拡張口になっています。

## MCP サーバーに組み込む設計を考える

外部変形を経由すれば、Jw\_cad の図形を Python で読み書きすることはできます。  
問題は、**この処理を MCP サーバーのツールとして呼び出せるようにできるか**、という点です。

理想を言えば、こういう動きにしたい:

1. ユーザーが Claude Desktop で「線を 100mm 右に動かして」と指示
2. Claude が MCP ツール「move\_lines」を呼ぶ
3. MCP サーバーが Jw\_cad を操作し、線が動く

AutoCAD なら .NET API や COM があるので、こういう動きは普通に組めます。  
ところが Jw\_cad の場合、**外部変形の根本的な仕様**がここで問題になります。

外部変形は **Jw\_cad 側からしかトリガーできません**。  
ユーザーが Jw\_cad で外部変形メニューを開き、バッチを選んで実行する、という操作を経由しないと動かない仕組みです。外から「外部変形を実行して」と命令する API は存在しません。

ということは、MCP サーバー側で「線を動かして」というツール呼び出しを受けても、**サーバー側から Jw\_cad に処理を依頼する手段がない**わけです。

ここで一度、**予約型のアーキテクチャ**で回避できないか考えました。

* MCP サーバーがツール呼び出しを受けると、`pending_job.json` のようなファイルに「やってほしい処理」を書き出す
* ユーザーが Jw\_cad で外部変形を実行する(汎用ディスパッチャのバッチを選ぶ)
* ディスパッチャは `pending_job.json` を読み、対応する処理を実行する

理屈としては動きます。  
ただ、これだと結局 **ユーザーが Jw\_cad で外部変形を毎回手動で起動する必要がある**ので、AI が能動的に Jw\_cad を操作する、という当初の絵にはなりませんし、毎回ユーザーがバッチを選ぶなら、**直接そのバッチを起動するのと変わらない**、という根本的な矛盾に当たります。

## 詰みポイント: 能動的トリガー手段がない

整理すると Jw\_cad を綺麗に MCP サーバー化しようとしたときの問題は一点だけです。  
**外部から Jw\_cad に「これを実行して」と命令する手段がない**、ということです。

外部変形は強力な拡張機構ですが、起動のトリガーは Jw\_cad のメニューに固定されています。  
コマンドライン引数で外部変形を直接呼ぶオプションも、リモート実行用のインターフェースもありません。

これは MCP の前提とは相性が悪い構造です。  
MCP は「AI から外部システムに能動的に働きかける」ための仕組みですが、Jw\_cad はその「働きかけを受ける口」を提供していない、ということになります。

予約型アーキテクチャは、この問題を**ユーザー側の手動操作で穴埋め**しているだけで、本質的には解決していません。

## 残る選択肢: RPA 的なアプローチ

「外から Jw\_cad を能動的に動かす」を諦めなければ、**力技**は残っています。

具体的には、**WinAPI や UI Automation を使って Jw\_cad のウィンドウを外部から操作する**方法です。キーストロークやマウス操作をプログラム的に送り込み、外部変形メニューを擬似的にユーザー操作として開かせる、というアプローチになります。

技術的には不可能ではありません。  
RPA ツールでよくあるやり方ですし、似たようなことをやっている事例も探せばあります。

ただ、これを **本気のプロダクトに組み込むには無理がある**、というのが正直な感触です。

* ウィンドウ位置やフォーカスの状態に依存する
* Jw\_cad のバージョンやウィンドウサイズで挙動が変わりうる
* ユーザーが Jw\_cad を触っている最中に、勝手にメニューが開く動きになる
* 失敗時のリカバリが書きにくい

要するに、**動くときは動くが、動かないときが普通にある**、という品質になりがちです。

特にリカバリは厄介です。AutoCAD なら .NET API の Transaction で失敗を検知して巻き戻せますが、UI 操作を送り込む RPA 方式では、途中でモーダルウィンドウや想定外の挙動に出くわすと操作シーケンス自体が破綻し、制御を失って手の打ちようがなくなります。

このあたりのロールバック設計については、以前の記事でも触れています。

<https://zenn.dev/0xliclog/articles/ca6381b58160f9>

これを「Jw\_cad 用の MCP サーバー」として実用品質で出せるかというと、自分の感覚ではかなり厳しいかなと思っています。(ただ、**うまく動かないこともあるくらいの気持ちで個人的に使う分**には、この手も十分アリかなと思います)

## 結論

軽く触ってみた範囲での結論はこんな感じです。

* 外部変形を経由した外部プログラムとの連携自体は問題なく動く
* ただし、外部から Jw\_cad を能動的に動かす手段がないため、MCP サーバー化を含む自動操作の設計は壁にぶつかる
* WinAPI や UI Automation で力技を使えば**できなくはない**が、安定性や品質の面で本気のプロダクトに組み込むのは厳しい

Jw\_cad の MCP サーバー化は、**個人で遊ぶならアリ、人に配るには厳しい**、というあたりが落としどころになりそうです。

今回触ってみて改めて感じたのは、CAD エージェントを作るなら、**エージェントから能動的に呼べる API が公開されている CAD が前提になりそうだ**、ということです。

外から操作を依頼できること。  
失敗したときに安全に巻き戻せること。  
この 2 つを土台として提供しているかどうかで、設計のしやすさは大きく変わります。

Jw\_cad の場合は、外部変形という強力な拡張口はあるものの、その入口がユーザー操作に固定されているため、MCP サーバーとして能動的に操作するには一歩届きませんでした。

とはいえ、Jw\_cad そのものに可能性を感じていないわけではありません。  
国内の建築・土木・電気の現場で広く使われている CAD なので、ここに AI エージェントを載せられたら面白いはずだという気持ちは変わっていません。

能動的なトリガー手段にせよ、RPA 方式を実用品質まで持っていく工夫にせよ、どこかで活路が見つかれば、Jw\_cad のエージェント開発にも改めて挑戦してみたいと思っています。
