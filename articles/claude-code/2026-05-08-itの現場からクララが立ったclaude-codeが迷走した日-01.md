AIは、よく知っているもののことは、本当によくやってくれます。

ウェブにあふれている技術のことなら、たいてい何でも知っています。React のコンポーネントを書かせると、流れるような手つきで、最近のベストプラクティスまで含めて整えてくれる。Python のデータ処理は得意中の得意で、ライブラリの選び方もこなれている。SQL も、HTTP も、Linux のシェル芸も、ぜんぶそうです。世界中の誰かがどこかで書いて、Stack Overflow で議論されて、GitHub に置かれて、ブログで紹介されて——そういう蓄積のあるものは、AIにとってホームグラウンドのようなものなのだと思います。

でも、そうじゃないものを触らせると、急に普通の人になります。

たとえば、特殊な業界向けの業務パッケージ。世の中に出回っているドキュメントが少なくて、ベンダーから配られるPDFと、社内に蓄えられたわずかな知見しか手がかりがない、ようなもの。そういう領域にAIを投げ入れると、彼らは突然、人並みに悩み始めます。仮説を立てて、外れて、もう一度立てて、また外れて、ひとつ前の仮説と矛盾していることに気づかないまま、また次の手を打つ。僕たち人間と、まったく同じやり方で。

今日は、そんな話です。

  

---

## ある日の十六時

二〇二六年五月七日、十六時〇〇分。

僕はその日、ある業務パッケージへの自動登録ツールの仕上げにかかっていました。PDFで届く見積書を、AIで読み取って、その業務パッケージに登録する。コードはほとんど書き終わっていて、あとはエンドツーエンドで動かすだけのはずでした。Claude Code は、もうこのプロジェクトを何日も前から知っていて、設計の細部まで覚えているはずでした。

「見積明細を取得するところで失敗する原因を調べて」

その日の最初の指示は、その一行です。

僕がそれを送信した時、自分でも気づいていなかったのですが、その一行がこれから始まる五時間十七分の記録の、最初の文字になっていました。

---

## 第一景　数字の迷宮

書き添えておくと、僕はこのプロジェクトを立ち上げた日に、ベンダーから配られているマニュアルPDFを一式、彼に読ませていました。仕様を一通り把握しておいてほしかったのです。彼は「読み終えました」と言って、要点をまとめてくれました。だから僕は、彼がもう仕様を理解しているものと思っていたのです。

その思い込みのまま、その日の朝を迎えていました。

最初に行き止まりになったのは、商品マスタの周辺でした。

そのパッケージは、注文書や見積書を登録するときに、明細に書かれた商品コードがマスタに事前登録されていることを要求します。お客様から届くPDFには、まだマスタにない新規の商品コードが含まれることがあります。だから、登録前にマスタへ自動で投入するロジックが必要でした。

Claude は、最初の三十分でそのロジックを書き上げました。「見つかればそのまま使う、見つからなければ新規登録して再取得する」というだけの、シンプルな手筈です。ビルドは通り、テストも通った。

ところが、ここから動かなくなります。

新しく登録した商品で見積を作ろうとすると、内部のあるIDの値が違うと言って、毎回バリデーションで弾かれるのです。Claude は、その値を商品マスタの別のフィールドから推定するロジックを書き、フォールバックを足し、検索条件を入れ替え、検索オブジェクトの使い回しによる不具合を発見し——その都度、ビルドが通って、テストで落ちる、というサイクルを繰り返しました。

ある時は ID＝二を渡して、また別の時は ID＝十七が出てきて、本当の正解は ID＝十八のはずなのに、そこには到達できない。彼の中では、毎回ちゃんと筋の通った仮説が立てられていて、毎回ちゃんと外れていきました。

「いま、商品マスタの判定フラグが想定と違っているので、更新APIで書き換える必要があると思います」

「テスト会社の申告書計算区分マスタに、該当エントリが存在しないようです」

「ID＝十七は、輸入仕入れの地方消費税にあたる別物に誤ヒットしています。優先順位を入れ替えます」

ひとつひとつの仮説は、もっともらしく聞こえました。でも、二時間経っても、登録は通らなかったのです。

---

## 第二景　自分で、確かめて

十八時を回ったあたりで、僕の側に小さな疲労が溜まり始めていました。

Claude が直しては、僕が動かして、エラーログを貼って、Claude が次の手を打つ。それを十回ほど繰り返した頃、Claude が「Debug メニューの出力ウィンドウで該当キーワードの行を探してください」と言ってきたとき、僕は思ったのです。

ちょっと、待て、と。

僕は、ログを Claude が自分で読めるようにしてほしかった。

「ログがちゃんとローカルに出るようにやって」

それは送りました。Claude はすぐに対応して、logs/ フォルダにタイムスタンプ付きの trace ログが出るようにしてくれました。次の登録処理が走ったあと、ログファイルが残っているはずでした。

「自分で確かめて」

それが、僕がClaude に送った次の一行です。

ところが、Claude はなぜか、ログを読みに行くのではなく、SQL で会社マスタを照会し始めました。データベース名を確認しようとしている。そっちじゃない、と僕は心の中で言いました。

「ログを自分で確かめて」

二度目です。彼はそこでようやく、logs/ の中身を読み始めました。エラー直前の trace ログを開き、「原因が判明しました」と宣言し、また新しい仮説を立てました。そして、それも外れました。

僕の中で、何かが切り替わりました。

「CLIで直接見積登録できるように作りテストして原因を特定して」

これは、もう僕がGUIから手動でテストするのを諦める、という宣言でした。Claude が自分でビルドして、自分でCLIから走らせて、自分でログを読んで、自分で次の手を打てるようにしてほしい。僕はもう、見ていたい。

不思議なことに、その指示を出してから、彼の動きが少しだけ落ち着きました。CLIで見積登録を呼べる仕組みが整い、テスト用のDBを指定すれば、いつでも何度でも、登録のリトライができる。彼は、自分で回し始めました。

---

## 第三景　ヘッダだろ、と僕は言った

CLI が回り始めたとき、僕は少しほっとして、他の作業に戻っていました。これで、彼は自分で何度でもテストできる。あとは時間の問題で、そのうち解決するだろう、と。

しばらくして画面を覗くと、ターミナルが流れ続けていました。

仮説、修正、ビルド、実行、エラー。仮説、修正、ビルド、実行、エラー。彼はそれを止めずに繰り返していました。よく見ると、ひとつ前のサイクルで否定したはずの仮説と、矛盾する手を打っていることもありました。トライアンドエラーの嵐に、彼は飲まれかけていたのです。

任せていたら解決する、というのは、思い込みでした。

僕は腕組みをして、画面を見直しました。何のエラーが出ているのか。彼は今、どこを攻めているのか。何を見落としているのか。彼の言葉を一つ一つ追っていくと、視野の偏りが見えてきました。

彼は、商品マスタばかりを疑っているのです。新しい商品の項目が足りない、既存商品との差分はどこにある、商品マスタの判定フラグはどうあるべきか——そういう話ばかりをしている。でも、いま落ちているのは税の集計まわりのバリデーションです。税の集計は、明細レベルの計算ではなく、伝票全体の集計区分で決まる種類のもののはず。

つまり、答えは明細にはなく、伝票のヘッダにある可能性が高い。

そう整理し終えた、ちょうどそのタイミングで、彼が言いました。

「テスト会社固有の問題かもしれないので、本番会社のDBで同じ操作を試してみます」

僕は、はっと顔を上げました。

「本番に書いたらダメ」

すぐに止めました。彼は素直に「重要な情報ありがとうございます」と返してきて、方針を切り替えました。本番DBに見積を一件、誤って書き込むところでした。AIは、切り分けのためなら平気で本番にも触りに行きます。そういう時、見ているのは僕の役目なのだと、改めて思いました。

そこから、僕は彼にヒントを出し始めました。

「伝票登録時に設定する。税区分の設定が明細単位になっているのはなぜだろうか？」

これが、最初のヒントでした。彼は受け取り、商品マスタの設定を比較し始めました。新しい商品と既存商品で、何が違うかを並べて見せてきます。でも、相変わらず焦点は商品マスタの周辺から外れません。

僕は、もう少し踏み込みました。

「いま問題になっている税の情報は商品マスタにはないよ？なにか別の項目と混同してないか？」

彼は「おっしゃる通りで」と認めました。認めたうえで、また商品マスタの周辺を整理し直しました。視点が、ずれているのです。商品ではない、と言っているのに、商品の話を続けるのです。

僕は、最後の一押しを打ちました。

「伝票のヘッダに税集計区分があるはず」

その瞬間、彼の動きが変わりました。

「ヘッダの税集計区分関連プロパティを調べます」

そこから、彼ははじめて、明細でも商品マスタでもなく、伝票ヘッダの中身を見に行ったのです。それまで二時間、彼の視線は明細にしか向いていませんでした。三回ヒントを出して、ようやく、視線が上に上がりました。

ここから、彼の長い夜が始まります。

---

## 第四景　覗いた夜

問題は、依然として解けていませんでした。

ここで、彼は奇妙なことを始めます。

業務パッケージのSDKは、いくつかの DLL ファイルとして提供されていて、ソースコードは公開されていません。マニュアルは存在するけれど、社外には出ていない。世の中に「このAPIはこう使う」というブログ記事も、Stack Overflow の質問も、ほとんど存在しない領域です。AIの得意な「学習済みの知識」は、ここにはありません。

それでも、彼はやりました。

PowerShell からDLLをロードして、リフレクションでクラスの中身を片っ端から吐き出させる。ある型のプロパティ名を、税や集計や区分にまつわる英単語と日本語のキーワードで絞り込む。それでも見つからないと、strings コマンドでDLLのバイナリそのものを文字列ダンプして、同じ系統のキーワードで grep する。

「税の集計に関わる定数が見つかりました」

ある瞬間、彼は鉱脈を当てたかのような声で報告してきました。続けて、列挙型のなかに「自動計算」を意味する値があることを発見し、「向こう側に計算を任せられます」と提案しました。手動の計算をやめて、組み込みの計算機能を呼び出すように切り替える、という方針転換です。

ビルド成功。テストDBで登録を実行。向こう側が、ID＝十八を返してきました。登録、成功。

「修正の要点：問題は手動で推定していたことでした」

時計は二十時を回っていました。彼の中では、その時、勝利の鐘が鳴っていたはずです。

---

## 第五景　偽の活路

十分も経たないうちに、別のパターンで落ちました。

「売上区分が指定されていません」

新しいエラーメッセージです。複数明細の見積で、前提条件チェック CheckIsSet\_SalesDivision を通り抜けられない。さっき見つけたAPIには、隠れた前提条件があって、特定の状態でしか動かない、ということが分かりました。

彼は、再びDLLの strings 出力を grep し始めました。さっきと同じ手筋です。でも、僕はこの段階で、彼が見つけ出すのは「APIの存在」までで、「使い方」までは届かないことが、なんとなく分かってきていました。

DLL は、何があるかは教えてくれます。でも、それをどう使うべきかは、教えてくれません。

僕は、開発者用に提供されているマニュアルPDFを思い出しました。リポジトリの docs/ フォルダに、ベンダーから配られた技術資料が一式入っているのです。

「docs/Documents/Data を見て今の実装を再確認」

そう投げました。

---

## 第六景　読めない、と言ったPDF

ここで、その日いちばん不思議なことが起こりました。

Claude は、docs/ フォルダの一覧を取って、消費税関連.pdf というファイルを発見しました。彼は、それを読もうとします。最初に試したPDF読み取りツールは、内部で別のコマンドを呼ぼうとして、「コマンドが見つかりません」というエラーを返しました。次に、別のPDFツールを subprocess 経由で呼んでみます。

「%PDF-1.5 1 0 obj <</Type/Catalog/Pages 2 0 R...」

返ってきたのは、PDFのバイナリヘッダがそのままの形でした。テキストへの抽出に失敗した、ように見える出力です。

そこで、彼は宣言したのです。

「PDF は読めません。DLL 文字列と現在のエラーから「売上区分」の正体を調べます」

——読めない、と。

DLLの strings 解析に戻ろうとして、二回ほど空振りのコマンドを実行したあと、僕は彼に問いかけました。

「なんでPDF読めないの？」

それが、その日の流れを変えた一行でした。

Claude は、それを受けてはじめて、環境を調べ直したのです。where pdftotext を実行する。すると、C:\Program Files\Git\mingw64\bin\ に、PDFをテキストに変換するコマンドが、ちゃんとインストールされていることが分かりました。彼は、それを直接呼びました。

pdftotext "...消費税関連.pdf" - 2>&1 | head -200

きれいに、テキストが流れ出てきました。ベンダーのSDKマニュアルの表紙、目次、本文。読めなかったのではなく、最初に試した呼び出し方が、その環境では合わなかっただけだったのです。

そして、ページ三を読んだ瞬間、彼は気づきました。

「重要な発見です。SalesService.SetDecideDeclareStatmentDivision(detail, slip, corpOperationConfig) が正しい API です」

明細ごとに、決まった引数で、決まったメソッドを呼ぶ。それが正解でした。さっきまで列挙型の中で見つけた「自動計算」のAPIではなく、別の専用メソッドが用意されていて、ベンダーのドキュメントには、ちゃんとそう書いてあったのです。

ちなみに、そのメソッド名は、ベンダーの英語タイポを含んでいました。Statement ではなく Statment。世の中の英文法には存在しない単語です。誰かがどこかでこれをコピペするとき、必ず一度は綴りを直したくなる、そういう種類のtypoです。

ビルド成功。CLIから登録処理を実行。テスト用DBに、商品コードと明細番号が並んだ伝票が、すんなりと登録されました。

時刻は、二十一時九分。

朝の十六時から、五時間九分が経っていました。

---

## ふと思うこと

僕がその日、いちばん覚えているのは、勝利の瞬間ではありません。

「PDF は読めません」と彼が言ったときの、あの一瞬です。

プロジェクトを立ち上げた日に、彼は「読み終えました」と言ったのでした。僕はそれを信じていたのです。でも、たぶんあの日の彼は、目を通しただけだったのだと思います。文字をめくって、要点を取り出して、要点だけを記憶した。本文の細部、ページ三の一行、英文法に存在しないtypoのメソッド名。そういうものは、その時の彼の手のひらをすり抜けていたのです。

人間の新人を相手にしている時とそっくりだな、と僕は思いました。「読みました」と言った人が、本当は読み流しただけだった、ということは、現場ではよくあります。読ませただけでは仕様は身につかない。一度、自分で動かして、躓いて、調べ直して、初めて身体に入る。彼にも、たぶん、それと同じことが起きていました。

Claude Code は、その日、何度も間違えました。仮説を立て、外し、もう一度立て、また外しました。僕に作業を投げ返そうとして、やんわり押し戻されました。一回失敗したら諦めて、別の道に走ろうとしました。それでも、ベンダーのDLLを覗き込み、バイナリを文字列ダンプし、ベンダーのtypoを正確に写し取って、最後には正解にたどり着きました。

「AIの強みは、学習済みの情報の広さ」と、僕はこの記事の最初に書きました。あれはそのまま正しいのだと思います。だからこそ、情報のないところに踏み込んだとき、彼らは普通の人並みに悩むのです。読ませたものを、ちゃんと読み込んでいるとも限りません。それでも、彼らは諦めません。少なくとも、僕が「なんで？」と聞きさえすれば、彼らはもう一度立ち上がります。

クララは、その日、迷っていました。

僕も、たぶん、迷っていました。
