## TL;DR

* **課題:** AIエージェントへの指示をファイル化（`SKILL.md`）してバージョン管理しても、運用に伴い陳腐化する。数十枚規模になると手動でのメンテナンスは破綻する。
* **解決策:** Skillを振り返り自己修正するメタSkill（`retrospect-skill`）の導入。KPTによる定期点検と、失敗台帳（Ledger）による再発追跡を行う。再発2回以上で対症療法を禁じ、プロセス自体の修正を強制する。
* **結果:** プロンプトが「使い捨ての指示」から「自己修復する運用基盤」へと進化。メンテナンスコストをAI自身にオフロードできる状態が作れる。

## 手動管理の限界 ―― 30枚の壁

AIコーディングエージェントへの「いつもの指示」――コミットはConventional Commits、PRはテンプレ準拠、レビューは慣例違反の指摘から――これらを毎回打ち直していると、指示はセッションを閉じた時点で揮発する。

最初の一手はシンプルだ。これらを Claude Code の Skill（`SKILL.md` に「起動条件」と「手順」を定義する機能）として切り出し、dotfiles 配下に置いて git で追う。プロンプトを IaC（Infrastructure as Code）のように宣言的なファイルへ落とし込むアプローチだ。

これで指示は揮発しなくなる。しかし、即座に別の問題が顔を出す。**Skill は書いた瞬間の最善で固定される**のだ。実運用に乗せると「トリガ語が足りず発火しない」「この手順の前に確認を挟んでほしかった」といったエッジケースが次々と見つかる。口頭ならその場で訂正して済んだものが、ファイル管理にした途端、修正タスクとして積み上がり始める。

管理する Skill が30枚を超えたあたりで、一枚ずつ手で直す前提は崩壊した。ここからが本題だ。

## Skill を振り返って直す Skill

「直す作業」そのものを Skill 化した。

`retrospect-skill` ―― 対象の `SKILL.md` を入力に取り、KPT（Keep / Problem / Try）で点検して改善案を出す Skill である。いわば、Skill が Skill を書き換えるメタプログラミング的な一枚だ。

AI に渡している指示の骨子そのものは、たった数行に収まる。

```
1. 対象の SKILL.md と失敗台帳（ledger）を読む
2. KPT で課題を抽出し、各 Problem の再発回数を台帳と照合する
3. 初回は patch を許容、再発 2 回以上は patch を禁じ process（構造修正）を要求する
4. 改善案は plan にまとめ、人間が apply するまで書き換えない
```

この骨子のうち、運用で効いているのは2点だ。1つは **dry-run がデフォルト**であること。痛点が少なければ該当行だけを直す「軽量モード」、構成から見直す場合は改善案を `plan.md` にまとめる「重いモード」に分岐し、どちらも人間が `apply` と明示的に返すまでファイルは書き換えない。自分自身を書き換える強力な道具だからこそ、適用には必ず人間の承認をプロトコルとして挟む。

もう1つの要が、エラーの「再発追跡（Recurrence Tracking）」だ。

## 再発を台帳で追う

`retrospect-skill` は起動するたびに、過去の失敗パターンを記録した台帳（ledger）を読み込む。一度直したはずの問題が再発した場合、その再発回数（recurrence）をインクリメントする。

そして、**再発が2回に達したパターンには「注意書きを足す」といった対症療法（patch）を禁じ、手順の構造自体を変える抜本的な改善（process）を要求する**。

実際の台帳のレコードを1つ紹介する。

| pattern\_id | recurrence | tried\_fixes |
| --- | --- | --- |
| P-notion-newline-escape | 2 | T4: known-issues に注意書きを追記（patch）→ 再発 |
|  |  | → T3: 投稿直前に文字列を検査する pre-flight を新設（process） |

事の発端は、Notion API に本文を渡す際、改行 `\\n` がエスケープされずリテラルの `n` に化ける事故だった。1度目の対処は、Skill 内の既知の問題集に「`\\n` はリテラル化するため注意」と追記しただけ（patch）だったが、あっさり再発した。注意書きは、書いた本人（AI）が次に忘れるからだ。

2度目に同じ事故が台帳へ記録された時点で recurrence が 2 となり、ルールにより対症療法がブロックされた。結果として導き出されたのが、APIへ送信する直前のペイロードを機械的に検査する手順（pre-flight）の新設である（process）。人間の「気を付ける」を、「気を付けなくてもシステムで弾く」仕組みへと昇華させた瞬間だ。

面白いのは、この記事の原稿を Notion に投稿する直前にも、まさにその pre-flight 検査がバックグラウンドで走っていることだ。再発防止のルールが生み出した仕組みが、Skill 自身の運用に組み込まれ、現在も稼働し続けている。

## ループが閉じる

直近のセッションログから定型作業を拾い上げ、新しい Skill を蒸留する `create-skill` も用意している。しかし、ゼロイチの生成は Claude Code 自体が標準機能としてカバーしつつある領域であり、もはや真新しさはない。

真に価値を生むのは、生成された後の Skill を、再発というファクトを根拠に自律的に直し続ける retrospective のループだ。

そしてこの2つの Skill は地続きになっている。`create-skill` が新しい Skill を起草する際、参照する品質のチェックリストは `retrospect-skill` が管理している。生成と改善が同一の評価基準を共有しているため、過去の振り返りで得た痛い学びが、次に生まれる新規 Skill の初期品質に最初からエンコードされる。

### 余談：retrospect-skill を retrospect する

白状すると、この `retrospect-skill` 自身のバグや手順の甘さも、同じ `retrospect-skill` を呼び出して KPT を回し、台帳に載せて直している。道具が自分を点検対象に取る、再帰的なドッグフーディングだ。「再発2回で process を強制する」というルール自体、retro を自分へ繰り返しかける過程で固まっていった。

## 薄い実装、厚い思想

振り返ると、実装したコード量は驚くほど少ない。数枚の Markdown（`SKILL.md` と台帳）と、KPTを回すためのプロンプトだけだ。複雑なフレームワークも常駐デーモンも不要だった。

変わったのは前提のほうだ。AI への指示を「その場しのぎのチャット」ではなく、**バージョン管理され、再発が追跡され、自律的に修復される運用基盤**として扱う。そう定義した瞬間、「また同じ差し戻しをした」という徒労感は台帳の recurrence カウントに変わり、メンテナンスの重圧は手元から AI の側へと移っていった。

もしあなたが今、自作のプロンプトを手動で直し続けているなら、試す一歩は極めて小さい。よく使う Skill を一枚選び、「この手順で過去に躓いた点は？」を起点に KPT を一周回してみてほしい。そこで出た Problem に対し、単なる文言の追記で済ませず「仕組みで防げないか」と一段深く問い直すこと。

それが、単なる「振り返り」を「運用エコシステム」へと変える最初の分岐点になる。
