先週、気分がいちばん沈んでいたのは何曜日か。今月に入って新しく増えた関心はどれか。3週間コミットが止まっているプロジェクトはどれか。こうした問いの答えを、私は毎週末にまとめて受け取っています。`/weekly-review` と一言入れると、Claude Code がその週の日記・トレンドのログ・開発の進み具合を横断で読み、繰り返し出たテーマや温めるべきアイデアまで拾って返してくる。

ただ、この記事で書きたいのは、この週次レビューが便利だという話ではありません。書きたいのは、これが**個々のスキルを「集めて役立つもの」に変えている**、その一点です。日記も、自己分析も、トレンド収集も、単体だとその場で完結して終わってしまう。それらを週に一度まとめて畳み直す仕掛けがないと、記録はただ積もるだけで、横断して使われないまま形骸化していきます。

このシリーズでは、Claude Code を1つのリポジトリで動かして、日記・自己分析・投資ツールまでいろいろ運用しています（[全体像はこちら](https://zenn.dev/takotyan00/articles/claude-code-life-repository)）。この記事は、その中で**バラバラなスキルを1つの流れにまとめている「ハブ」の設計**の話です。

## スキルは、その場で完結してしまう

個人用のスキルを書きはじめると、最初はそれぞれが気持ちよく動きます。`/journal` はその日の日記を作って終わり。自己分析はその回の気づきを残して終わり。トレンド収集を足せば、その日のログを吐いて終わり。どれも単体で完結していて、それでいい。

問題は、1週間ぶんが積もったときに起きます。日記は7本、気づきのメモも数本、開発のコミットも溜まっている。でも、**それらを横断して「今週なにが起きていたか」を読む人が、誰もいない**。日記を7本まとめて読み返すのは面倒だし、そもそもやらない。結果、せっかくの記録が「書いたけど使われないログ」になっていきます。

ここで勘違いしやすいのが、スキルを足せば足すほど賢くなる、という感覚です。実際は逆で、単体で完結するスキルを増やすほど、**出力の置き場所が散らばっていくだけ**でした。増えるのは洞察ではなく、バラけた成果物の数です。

足りていなかったのは、新しいスキルではありませんでした。日々の出力を週に一度まとめて読み、次に効く形へ畳み直す\*\*「まとめ役」\*\*——それが1つ、欠けていた。

## 直接は呼び合わず、context.yaml 1枚を共有する

まとめ役を作る前に、そもそもスキル同士がどうやってつながっているかを書いておきます。ここが後の設計を素直にしてくれます。

各スキルは、お互いを直接呼び出しません。代わりに、**1枚の共有ファイル（`context.yaml`）を全員が読み書きする**、という形にしています。

```
# context.yaml — 全スキルが参照・更新する共有状態
interests:            # 関心分野。/journal のタグ付け・関心変化の検出に使う
  - 個人開発
  - 生成AI / Claude Code
active_projects:      # 開発中のもの。/journal が git log を自動収集
  - name: example-project
    focus: MVP を一気通貫で作る
    development_status: active   # 開発が動いているか
    user_facing: none            # 実際に使われているか（別軸）
recurring_themes: []  # 繰り返しテーマ。週次レビューが追記
improvement_backlog: [] # 開発の改善案。週次レビューが追記
```

`/journal` は日記にタグを付けるとき `interests` を見て表記を揃えます。たとえばトレンド収集のスキルを自分で足せば、それも `interests` と `active_projects` を見て「何を集めるか」を決められる。誰も他のスキルの中身を知らないまま、この共有ファイルを介して噛み合う。ソフトウェアで言う疎結合を、設定ファイル1枚で成立させているだけです。

この形にしておくと、後から週次レビューを足すのが楽になります。週次レビューがやるのは、**この共有状態を含めて全部を読み、分析した結果をまた共有状態へ書き戻すこと**だからです。スキルを増やしても配線を増やさずに済む。

## 週次レビューは「まとめ役」— 日々の出力を、次の入力に畳む

週次レビューの仕事を一言でいうと、**バラバラに溜まった1週間ぶんを読んで、3つの成果物に畳む**ことです。

読み込むのは、過去7日の日記・トレンドのログ・共有ファイル・開発中プロジェクトの現状。それらを横断して、気分の推移、繰り返し出たテーマ、新しく芽生えた関心、手作業で繰り返していること……を拾い出します。そして出すのは、次の3つだけに絞っています。

| 出力 | 中身 | どこへ戻るか |
| --- | --- | --- |
| ① アイデア候補 | 日記から見えた「作ると効きそうなもの」 | 対話で絞って issue 化 → 開発へ |
| ② 共有ファイルの更新 | 関心の増減・プロジェクトの進捗・繰り返しテーマ | `context.yaml` へ → 以降の日記タグなどが変わる |
| ③ 改善提案 | ツールへの不満・自動化できる手作業 | `improvement_backlog` へ → 消化用スキルが拾う |

実際に返ってくるのは、たとえばこんな見た目のサマリです（値は例）。

```
## 週次レビュー: 07-01 〜 07-07

### Mood トレンド
平均 3.6 / 5（月 4 → 火 3 → … → 土 4 → 日 2）

### 繰り返しテーマ
- 料理（4回・前週比 →）
- 生活リズムの調整（2回・↑）

### 関心の変化
- 新しく増えた: オープンデータの可視化

### プロジェクトの現状
- example-project:「次はデプロイ」→ 実際は本番稼働中（status を修正）
- another-project: 3週間コミットなし → paused に更新

### アイデア候補
1. 家計の推移を眺める小さなツール（評価: 中）

### 改善提案
- トレンド収集を毎回手で起動している → 定期実行にできないか
```

見ての通り、ただの感想文ではありません。この3つに共通しているのは、**どれも「読み物」ではなく「次の入力」になっている**ことです。週次サマリでよくある失敗は、きれいな振り返りレポートを出して、それ自体が読み返されずに終わること。だからこのハブの出力は、レポートにせず、必ず「次にどこへ効くか」まで紐づけています。

たとえば②で `interests` に新しい関心が1つ足されると、翌週から `/journal` のタグ付けが変わり、トレンド収集が拾ってくる範囲も変わる。今週の出力が、来週の日々のスキルの挙動を静かに書き換えていく。1回ぶんでは地味でも、これが毎週かかると、記録の解像度がじわじわ上がっていきます。

## 放っておくと、context.yaml は実態とズレていく

まとめ役を毎週動かす、もう1つの理由がこれです。**共有ファイルは、こまめに突き合わせないとすぐ実態とずれる**からです。

いちばんずれるのが、開発中プロジェクトの現状でした。手元では機能を1つ実装したのに、共有ファイルの「次にやること」は先週のまま。逆に、新しく始めたプロジェクトを登録し忘れる。この手のズレは気づかないうちに溜まって、ある日「この設定、もう古いな」となって信頼を失います。

なので週次レビューには、**共有ファイルとプロジェクトの実状を毎回、双方向で突き合わせる**手順を必須で入れています。

* 共有ファイルにあるのに、対応する記録が無い／内容が git の実態と食い違っていないか。
* 記録側にあるのに、共有ファイルへ未登録のものがないか。
* 状態は「開発が動いているか」と「実際に使われているか」の2軸で見て、開発停止と利用停止を混同しない。

たとえば、あるプロジェクトの現状に「次はデプロイ」と書いてあるのに、実際はもう本番で動いていた——という食い違いを、この照合が拾って直します。正直に打ち明けると、この手順を仕組みにする前は、プロジェクトの現状が古いまま放置される、というのを5回続けて繰り返していました。毎回のレビューで必ず突き合わせる、と決めてから、その放置が起きにくくなりました。まとめ役は、要約を出すついでに、この共有ファイルを「生きた現状」に保ちつづけている、という側面があります。

## 書き戻すけれど、勝手には書かない

ここまで「週次レビューが共有状態へ書き戻す」と書いてきましたが、実際には**書き換えを自動で確定させていません**。分析結果はいったん提案として全部見せて、私が承認したものだけを反映します。関心を1つ足すのも、プロジェクトの現状を更新するのも、一度必ず目を通す。

全部を自動にしないのは、共有ファイルが「全スキルの土台」だからです。土台が自分の意図しない方向へ勝手に育つのは、それこそ形骸化の入口になります。手作業を減らしたい一方で、**土台に触れる一手だけは人間に残す**——この線引きは崩さないようにしています。

実際のところ、まとめ役の出来は、その週に何が書かれたかにそのまま比例します。日記が薄い週は、レビューも薄い。数値や日記から見えた相関も、あくまで手がかりであって断定ではありません（この「AI が私を平板に読み違えないための補正」は[別記事](https://zenn.dev/takotyan00/articles/claude-code-correction-rules)に書きました）。派手な洞察が毎週出るわけではなく、積もった記録を淡々と次の形に畳み直す、その反復が本体です。

## まとめ役がいないと、記録は死ぬ

最後に、なぜここまでして「まとめ役」を1つ置くのか。

このシリーズで一貫してやっているのは、自分という個を取りこぼしなく AI に理解させ、必要なときにその理解を引き出して使う、というものです。そのために日記や自己分析で入力を溜めている。ただ、入力はただ溜めるだけだと、いつか読めない山になります。**溜めることと、使える状態に保つことは、別の作業**です。

日々のスキルが入力を集める役だとすれば、週次レビューは、その入力を**整理された状態に保ち、次の行動へ返す役**です。関心の変化を追い、古くなった現状を直し、埋もれたアイデアを拾い上げる。人間が握るべきレバーは入力の質だけで、あとは AI に引き出させればいい——そう割り切ったとき、その「質を保つ」部分を毎週ひとりでに回してくれる仕組みが要りました。それがこのハブです。

溜めた記録を、あとから AI に聞いて必要な一本を掘り出す。それは受け身の取り出しで、それ自体は日記が単体でもできます。でも、関心が変わったこと・プロジェクトが進んだことを共有状態に反映しておくのは、こちらから週に一度おこなう能動の作業です。掘り出すのが受け身なら、週次レビューは、掘り出しやすい状態そのものを保つ能動の一手にあたります。

書いても使われない記録は、結局それを整理して次につなぐ工程がないだけなんだと思います。まとめ役を1つ置くと、日記も自己分析もトレンドも、「集めて終わり」から「集めて、畳んで、また効かせる」に変わる。増やしたスキルがバラけずに1つの流れに乗るのは、この一手があるからでした。
