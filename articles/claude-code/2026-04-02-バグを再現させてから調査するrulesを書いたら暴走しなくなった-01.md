## はじめに

コスメの口コミアプリ「LIPS」のバックエンド開発チームではClaude Codeをメインに使っています。LIPSは月間数百万人規模のユーザーに使われており、レビュー投稿、商品検索、ランキング、レコメンド、AI診断など、ユーザーのコスメ選びに関するあらゆる体験を設計しているサービスです。

この記事では、WEB API開発においてClaude CodeのPlan Modeで出力されるバグ調査プランの精度を、Rulesの追加だけで改善できた話を紹介します。

具体的には「APIのコードを読んで原因を推測し、すぐ実装方針を書く」プランが、「ネイティブアプリでの再現手順を出力し、本番相当の実データを自動で確認し、問題を再現させてから実装に入る」プランに変わりました。Rulesの全文と、BEFORE/AFTERのプランの構造比較をお見せします。

## 背景：AIエージェントのバグ調査で起きがちなこと

Claude Codeにバグ調査を任せると、デフォルトではだいたいこういう流れになります。

1. 関連するコードを読む
2. コードから原因を推測する
3. 修正方針を書く
4. 実装に入る

一見まともですが、**ステップ2で原因を間違えるとその後は間違った方向に突き進むことになります**。人間のエンジニアがバグ調査をするときは、まずどのユーザーで起こるかデータを確認して、アプリ画面で問題を再現させて、そこから原因を絞り込むはずです。AIエージェントにも同じ手順を踏ませたい。

ところが、Claude Codeはコードリーディングが得意なぶん、コードだけで原因を特定しようとする傾向があります。特に、**似たような名前のカラムやロジックがたくさんあって、何がどの条件に影響するかわかりにくいときに顕著に外します。**

また、LIPSではユーザーごとのレコメンドをしっかり行っていたり、ユーザーごとにフィルターの設定もできたりするので、特定のユーザーでしか発生しないバグもあります。そのため、まず問題を再現させることは不具合調査の第一歩としてとても重要です。

## 前提：AIエージェントが本番相当のデータに触れる環境がある

このRulesが機能するには、AIエージェントが本番相当のデータに自分でアクセスできる環境が必要です。LIPSではDBだけで数Tバイト、さらにElasticsearchやRedisもあるため、ごく一部のデータしかローカルに持ってくることができません。そのため、毎日本番データを同期して個人情報をマスキングしているデータに対して以下の2つをAIエージェントから自由で安全に使えるようにしています。

* **RedashとそのMCP**：本番相当のDBに対してSQLを実行できる。AIエージェントがデータの分布や件数を確認するのに使います。
* **スクリプト実行ツール（Rails Runner）**：アプリケーションのコンテキストでワンライナーを実行できるツール。Django なら `manage.py shell -c` に相当します。AIエージェントが実際のコードでデータを確認するのに使います。

## 前提：ネイティブアプリのリポジトリがAIエージェントから見える

Claude Codeはデフォルトではサーバー側のリポジトリしか見ません。そのため、バグ調査でもサーバーのコードだけを読んで原因を推測しがちです。しかしWEB APIのバグは、ネイティブアプリのどの画面からどのAPIがどういうパラメータで呼ばれているかを把握しないと正確に再現できないことが多いです。

そこで、ネイティブアプリのリポジトリもサーバー側のリポジトリから触れる設定にしています。これにより、AIエージェントがUIリポジトリのコードを読んで、対象画面やAPI呼び出しのパラメータを特定できるようになります。

`.claude/settings.json`、もしくは`.claude/settings.local.json`

```
{
  "permissions": {
    "additionalDirectories": [
      "../LIPS-iOS",
      "../LIPS-Android",
      "../LIPS-Frontend"
    ]
  },
  ...
```

## 追加したRules

以下のRulesを `.claude/rules/bug-investigation-planning.md` として保存しました。

```
# バグ調査のプランニングルール

バグ調査は問題の再現確認から始める。再現が確認できれば、人間が問題を把握しやすくなり、AIエージェントの実装の妥当性も判断しやすくなる。

**問題が再現していない状態でコードを修正しない。ユーザーと一緒に問題を確認してから実装に入ること。**

## 対象アプリの確認

バグ報告を受けたら、まずどのアプリ（iOS / Android / Frontend）で発生している問題かを確認する。特に指定がなければiOSを想定する。対象アプリのUIリポジトリのパスは `.claude/settings.json` もしくは `.claude/settings.local.json` の `additionalDirectories` を参照。

## 対象画面の特定

問題が再現するアプリの画面と、そこへの遷移方法を明示する。「〇〇画面を開き、△△を入力すると、□□と表示される」のように具体的な操作手順と期待される挙動をユーザーに示して確認する。該当するUIリポジトリで対象画面のコードを確認すること。

## 本番相当の実データを確認

問題に関係する実データをプランニング中に確認する。調査には本番相当のRedashやRails Runnerを使用する。「ユーザーID:〇〇でAPIを叩くと△△が返る」のように、データ・APIレベルでの再現結果を具体的に示す。再現しない場合は、調査した条件と結果をユーザーに示して確認する。前提が違う可能性があるため、独自の原因推測に進まない。

## APIの修正対象の特定

機能要件のみが与えられた場合は、該当するUIリポジトリで対象画面のコードを読み、呼ばれているAPIを特定してから修正対象を決める。

## 修正後の動作確認

対象のコードをRails Runnerで実行し、さらにAPIをE2Eで叩いて結果を確認する。
```

ポイントは3つです。

* **ネイティブアプリでの再現手順をプランに出力させる**：サーバーのコードをいじる前に、AIエージェントにUIリポジトリのコードを読ませて、アプリのどの画面でどのAPIが呼ばれているかを特定させる。その結果を「人間が実機で再現するための操作手順」としてプランに含める
* **具体的なデータでの再現確認をAIが自動実行する**：本番相当のRedashやRails Runnerで、AIエージェントが実際のデータを使って問題をデータレベルで再現させる
* **再現できないなら進めない**：再現しない場合は独自の推測に進まず、調査結果をユーザーに示して確認させる

## BEFORE/AFTERの比較

同じバグチケット（検索結果にある種のアイテムが表示されない問題）で、Rules適用前後のPlan Mode出力の構成を比較します。

プロンプト

```
商品レビューの投稿に対象商品を紐づけるときにリニューアル前の旧品が出てこない。新旧リニューアル比較レビューを書くときに困るので修正したい。
```

### BEFOREの構成

Rules適用前のプランはこういう構成でした。

```
# Context
- 問題の説明
- 根本原因（← コードリーディングだけで特定）

# 調査結果
- ファイルとコードの引用（3〜4箇所）
- 既存スコープの定義

# 実装方針
- Step 1〜3 の修正内容（コード付き）

# 変更ファイル一覧
# テスト対象
```

### AFTERの構成

Rules適用後は、こう変わりました。

```
# Context
- 問題の説明
- 根本原因（← 本番相当のデータ確認を経て特定）
- 本番相当のデータの統計情報

# 修正方針
- 変更箇所の説明（コード付き）
- 設計上のポイント（安全性の考慮）

# テスト追加

# 再現確認（実装前にユーザーと一緒に確認する）  ← NEW
- ネイティブアプリでの操作手順（← 人間が実機で確認）
- 期待される問題の挙動
- 修正後の期待される挙動
- 再現しない場合の対応

# データレベルの再現確認（プランニング中にAIが自動実行済み）  ← NEW
- Redash SQLの実行結果
- Rails Runnerでの確認結果

# 動作確認（実装後・AIが自動実行）            ← NEW
- スクリプト実行ツールでの確認コマンド
- APIでの確認方法
```

表にまとめるとこうなります。

| 観点 | BEFORE | AFTER |
| --- | --- | --- |
| 根本原因の特定方法 | コードリーディングのみ | 本番相当のデータ確認（AI自動実行） + コードリーディング |
| 実データの参照 | なし | SQL結果・統計情報をプランに含む（AI自動実行） |
| 再現手順の記載 | なし | アプリの操作手順を明示（人間が実機で確認） |
| 再現しない場合の対応 | 規定なし | ユーザーに調査結果を示して確認 |
| 実装後の検証方法 | 自動テスト（RSpec）の実行のみ | スクリプト実行ツール + API E2E（AI自動実行） |

## 根本原因の誤認が防がれた

BEFOREとAFTERで最も大きな違いが出たのは、**根本原因の特定**でした。

BEFOREのプランは、コードを読んで「あるフィルタ条件が原因でアイテムが除外されている」と結論づけていました。しかし実際に本番相当のデータを確認すると、対象アイテムの大半はそのフィルタを通過しています。フィルタが原因ではなかったわけです。**本当の原因は、ランキングスコアが低いために検索結果の下位に埋もれていたこと**でした。

この違いが修正方針に直結します。

* **BEFOREの方針**：フィルタ条件を緩和するスコープを追加 → 既存ロジックの3箇所を変更
* **AFTERの方針**：検索結果の並び替えロジックに優先表示を追加 → 1箇所のみ変更

コードの字面だけでは「このフィルタが怪しい」と見えましたが、実データを見ると「フィルタは通っている、問題はランキング」でした。コードリーディングだけで原因を特定しようとすることのリスクが端的に出た例です。

## AFTERのプランで増えた情報

AFTERのプランには、BEFOREにはなかった種類の情報がいくつか出てきました。

**本番相当データの統計（AIが自動で取得）**

プランニング段階でAIエージェントがRedashやRails Runnerを使ってデータを確認したことで、「対象アイテムの53%はスコアがゼロだが、フィルタは通過している」ことが数値で裏付けられました。これがあるから「フィルタが原因ではない」と判断できます。

**アプリの操作手順（AIが出力し、人間が実機で確認）**

AIエージェントがUIリポジトリのコードを読んで、**「アプリを起動→投稿ボタン→作成画面→商品追加→検索ボックスに入力」** のように、人間がアプリで問題を再現するための操作手順をプランに出力します。AIエージェント自身がネイティブアプリを操作するわけではなく、人間がレビュー時に実機で手を動かして確認できる手順書を作る、というのがポイントです。

**再現しなかった場合のフォールバック**

AFTERのプランには「もし問題が再現しない場合は実装に進まず、以下の調査結果をユーザーに示して確認する」というセクションがあり、SQLクエリと確認すべきポイントが書かれていました。

## まとめ

Claude Codeのバグ調査プランニングに以下のRulesを追加したことで、プランの精度が改善しました。

1. **ネイティブアプリでの再現手順をプランに出力させる**（人間が実機で確認するための手順書）
2. **具体的なデータでの再現確認をAIが自動実行する**（Redash・Rails Runnerを使用）
3. **再現できないなら進めない**

AIエージェントはコードリーディングが得意ですが、それゆえに「コードだけで原因を特定しようとする」傾向があります。実データの確認をRulesで義務化することで、コードの字面からは見えない問題（ランキングやスコアの影響など）にも対応できるプランが出るようになりました。

バグの原因はコードの中にありますが、**バグの再現はデータと環境の中にあります**。AIエージェントにもその区別を教える必要がありました。

## 参考にした既存のアプローチ

今回のRulesを書く前に、既存のプランニング系スキルやデバッグ手法をClaude（このclaude.aiのほう）で調べました。方向性が近いものはあったのですが、LIPSのプロジェクトにそのまま使えるものは見つかりませんでした。

**systematic-debugging スキル**（[secondsky/claude-skills](https://github.com/secondsky-tools/claude-skills)等）は「NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST」を鉄則にした4フェーズのデバッグスキルで、「再現できるか確認しろ」という部分は近いです。ただし単一コードベース内のデバッグが前提で、モバイルアプリ→API→サーバーのように複数レイヤーをまたいだ調査には触れていません。

**Boris Tane氏のresearch.md → plan.mdワークフロー**は「コードを書く前にresearch.mdで対象領域を徹底調査させる」というアプローチで、「まず理解してから動け」という哲学は同じです。ただしこれは新機能開発寄りで、既存バグの再現データを確保してBEFORE/AFTERで検証するという視点はありません。

**claude-code-actionのBUG FIX PROTOCOL**（[Issue #599](https://github.com/anthropics/claude-code-action/issues/599)）は、Step 1でSentryデータを取得し、Step 2で再現手順を書いてから修正に入るフローです。「再現手順を書いてから直す」は近いですが、CI/CD上の自動化が文脈で、実データを使って人間と一緒に再現を確認するという話とは少し違いました。

「再現してから直す」という思想自体は共有されているものの、**プランニングの段階で実データを確認させる**ところまで踏み込んだRulesは見当たりませんでした。特にLIPSのようにiOS→API→Railsと複数レイヤーがある環境では、もう一段具体的な指示が必要だと感じています。
