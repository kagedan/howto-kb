ルミナイR&Dチームの宮脇彰梧です。

**💡 結論**  
**「新規プロジェクトの立ち上げ（Scaffolding）は『Sonnet 4.6』の圧勝」**です。SonnetはCLIツールを器用に使いこなし、サーバーの起動確認まで自律的に完遂します。一方のOpus 4.6は、CLIツールがエラーを吐くと**「よし、全部の環境設定ファイルを手書きしよう」という狂気のマニュアル実装**に走り出し、時間とコストを溶かします。

### 1. なぜこのテーマを選んだのか？

「AIに丸投げすれば、全部イイ感じにしてくれるんでしょ？」  
生成AIを使い始めたばかりの企業担当者やエンジニアからよく聞く言葉です。しかし、CLIエージェントの現実はもっと泥臭いものです。

特にClaude Codeのような「ツール呼び出し（Bash実行やファイル書き込み）」ができる自律型エージェントは、モデルの知能によってツールの使い方が全く異なります。一番賢いOpusを指定しておけば安心だと思いきや、思いもよらない「オーバーエンジニアリング」や「力技」で暴走し、我々人間が置いてけぼりになることもしばしば。

この「モデルごとの性格の違い」を正確に把握しておくことこそが、AIエージェントを実務で使いこなすための第一歩だと考え、今回の検証に至りました。

### 2. この技術は採用すべき？

Claude Codeを導入するなら、モデルの使い分けは**絶対に避けて通れない意思決定**です。以下の表とフローチャートを使って、プロジェクトのフェーズに応じた最適なモデルを選択してください。

#### 🆚 Opus 4.6 vs Sonnet 4.6：ゼロイチ開発における特性比較

| 評価軸 | 🏎️ Sonnet 4.6（デフォルト推奨） | 🧠 Opus 4.6（切り札） |
| --- | --- | --- |
| **問題解決スタイル** | 現代のエンジニア型（既存ツールをフル活用） | 頑固な職人型（ツールがダメなら手で一から作る） |
| **CLIツールの扱い** | 巧み（バックグラウンド実行や疎通確認まで行う） | 苦手・力技になりがち（対話プロンプトで詰まる） |
| **自己完結力** | 🙆‍♂️ サーバー起動から動作確認まで自動で完了 | 🙅‍♂️ コードは書くが「あとは自分で起動してね」と投げる |
| **APIコスト感** | 💰 安い | 💸 高い（力技に走るとトークン消費が跳ね上がる） |

#### 🧭 導入判断のフローチャート（ゼロからアプリを作る場合）

1. **タスクの性質は？**
   * 新規プロジェクトの作成（Next.js, Viteなどのテンプレート利用） ➡️ **Sonnet 4.6**
   * 既存の複雑なシステムの根幹に関わるバグ修正・リファクタ ➡️ **2へ**
2. **既存環境の制約は？**
   * 標準的なフレームワークのレールに乗っている ➡️ **Sonnet 4.6**
   * 独自のビルドツールや、ドキュメント化されていない秘伝のタレがある ➡️ **Opus 4.6**

普段の開発は `claude config set model claude-4-6-sonnet` をデフォルトにしておきましょう。Opusは、Sonnetが3回連続でエラーを解決できなかった時の「お助けキャラ」として召喚するのが最もコスパが良いです。

### 3. 技術の仕組み：エージェントの「ツール利用」の違い

Claude Codeの裏側では、モデルが「Bashコマンドの実行」や「ファイルの読み書き」といったツールを呼び出しています。ここで注目すべきは、**エラーに直面した際のリカバリー戦略**です。

* **Sonnet 4.6の戦略（Pragmatic Recovery）：**  
  コマンドが失敗すると、エラーログを読み、コマンドの引数を修正して再実行しようと試みます。非同期処理（バックグラウンド実行）の扱いにも長けており、現代のCLIツールの仕様に非常に適応しています。
* **Opus 4.6の戦略（First-Principles Recovery）：**  
  非常に高い推論能力（第一原理思考）を持つため、ツールが動かないと判断すると「ツールの裏側で何が行われるべきか」を推論し、**その結果を直接手動で生み出そう**とします。

この「賢すぎるが故の力技」が、次の検証で見事なまでに露呈します。

### 4. 実装・検証：スパルタ・ポモドーロタイマー対決

では、実際のClaude Codeの生ログを見てみましょう。

**【お題】**

> Next.js（App Router）とTailwind CSSを使って、ポモドーロタイマーを作って。特徴として、タイマー稼働中にユーザーが別タブに切り替えたり、ウィンドウからフォーカスを外したりしたら、画面いっぱいに『サボるな！』と怒る機能をつけて

全く同じ空のディレクトリで、それぞれのモデルに実行させました。

#### 🏎️ Round 1: Sonnet 4.6 の場合（所要時間: 4分29秒）

**【ハイライトログ】**

1. `Bash(npx create-next-app@latest pomodoro ... --yes)`  
   ➡️ *完璧な引数で対話プロンプトをスキップし、バックグラウンドで実行！*
2. `Write(pomodoro\app\components\PomodoroTimer.tsx)`  
   ➡️ *コンポーネントを適切に分割（204行）。*
3. `Bash(npm run dev 2>&1 &)`  
   ➡️ *なんと、開発サーバーをバックグラウンドで起動！*
4. `Bash(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000)`  
   ➡️ *自らHTTPステータスコード200を確認してタスク完了を宣言。*

**【評価】**  
まさに「できる現代のエンジニア」。バックグラウンド実行を駆使し、最後に `curl` でサーバーの疎通確認まで行う手際の良さには、思わず声が出ました。

#### 🧠 Round 2: Opus 4.6 の場合（所要時間: 4分13秒）

**【ハイライトログ】**

1. `Bash(npx create-next-app@latest pomodoro ...)`  
   ➡️ *引数が甘く `? Would you like to use React Compiler?` の対話プロンプトで引っかかり、タイムアウト（エラー）。*
2. `Bash(mkdir -p pomodoro/app)`  
   ➡️ *（…ん？ディレクトリを直接掘り始めたぞ？）*
3. `Write(pomodoro\package.json)`  
   `Write(pomodoro\tsconfig.json)`  
   `Write(pomodoro\next.config.ts)`  
   ➡️ *なんと `create-next-app` を諦め、**Next.jsの全設定ファイルを「手書き」で作成**し始めました！*
4. `Write(pomodoro\app\page.tsx)`  
   ➡️ *コンポーネント分割もせず、271行の巨大なファイルを1つ生成。*
5. *「起動は cd pomodoro && npm run dev です」と言い残して終了。*

**【評価】**  
**お前、Next.jsの環境構築を手書きでするんか！？** と、画面の前で爆笑してしまいました。CLIツールが使えなかった瞬間に、「じゃあ俺が全部のファイルを書くよ」とゴリ押す狂気。ここで数分間の時間と高額なAPIトークンが溶けました。

### 5. 筆者の考察

今回の実行ログは、ソフトウェアエンジニアリングにおける「抽象化レベルの選択」という観点で、AIエージェントの限界と特性を如実に表しています。

Sonnet 4.6は、「実利主義なモダンエンジニア」です。  
NPMエコシステムやCLIツールの「作法」を深く理解しており、`create-next-app` のようなボイラープレート生成ツールを適切に抽象化のレイヤーとして利用します。エラーが出ても、ツールの使い方の範疇（引数の修正など）でリカバリーを図り、`&` によるバックグラウンド処理や `curl` でのヘルスチェックまで見事にこなしました。

一方のOpus 4.6は、「第一原理思考が強すぎるテックリード」です。  
CLIツールの対話プロンプトでタイムアウトした瞬間、Opusは「ツールを直す」のではなく「ツールの裏側にある本質（必要な設定ファイル群）」を自ら書き出すという、**究極の車輪の再発明**に走りました。  
「俺は `tsconfig.json` も `postcss.config.mjs` も完璧に理解しているから、ツールがコケたなら手で書けばいい」というアプローチです。

これは確かに「賢い（知能が高い）」証拠なのですが、開発現場において設定ファイルを手動で生成させるのは、トークン消費（コスト）と実行時間の観点から完全にアンチパターンです。また、最終的に271行のFat Component（`page.tsx`）を生成してしまった点からも、フレームワークのベストプラクティスよりも「力技で動くものを作る」ことにコンテキストを割いてしまったことが窺えます。

結論として、AIエージェントにタスクを依頼する際、私たちは**そのタスクがどの抽象化レイヤーで解決されるべきか**を人間側でコントロールする必要があります。エコシステムに乗っかったScaffoldingにはSonnetを使い、抽象化が通用しないレガシーコードの解析や、独自の複雑なロジックを読み解く時のみOpusの「第一原理思考」を頼る。この使い分けこそが、現代のAIエンジニアリングにおける最重要スキルと言えるでしょう。

### 6. まとめ

今回のポモドーロタイマー対決から得られた、意思決定の結論を再掲します。

1. **ゼロからの環境構築や定型的なアプリ作成は、「Sonnet 4.6」が最適。CLIエコシステムの恩恵を最大限に活用してくれる。**
2. **「Opus 4.6」は、ツールが失敗すると「手書きでゴリ押す（車輪の再発明）」傾向があり、新規構築には向かない。**
3. **高知能モデル（Opus）は、複雑に絡み合った既存コードの解読や、未知のエラー原因の特定など「本質的な推論」が必要な局面にのみ投入せよ。**

Claude Codeは、ターミナル上でAIとペアプログラミングをする最高の体験を提供してくれます。しかし、彼らの「性格」を理解して適切に指示を出さないと、とんでもない回り道をすることになります。

皆さんもぜひ、自分のプロジェクトで「AIに簡単なモックを作らせる」テストを行い、その挙動の違いを監視してみてはいかがでしょうか？
