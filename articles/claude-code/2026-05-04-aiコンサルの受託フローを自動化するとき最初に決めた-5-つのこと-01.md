## 受注ゼロの段階で、なぜ「フロー設計」から始めたか

最初に正直に言っておく。この記事を書いている時点で、AIコンサルとしての受注はまだゼロだ。

独立して事業を始めると決めたとき、「まず営業して案件を取ってから考えよう」とも思った。でも少し考えると、その順序の危うさが分かった。

相談が来た瞬間に詰まるのだ。

「費用はどのくらいですか?」と聞かれたとき、頭の中に明確な答えがなければ、その場の雰囲気で値段を言ってしまう。「どこまでやってもらえますか?」と聞かれたとき、基準がなければ過剰に請け負う。「いつから始められますか?」と聞かれたとき、稼働上限を決めていなければ「とりあえず受けて後で考える」になる。

AIコンサルに限らず、受託で消耗していく人の多くがここで詰まっていると思う。相談が来てから慌てて条件を考えるから、毎回違う判断をしてしまい、積み重なると判断疲れが起きる。

私がやったことは逆の順序だった。最初の相談が来る前に、「型」を決めた。

このプロセスで気づいたのは、自動化の本当の対象は「業務」ではなく「判断の型」だということだ。Claude Code で受注フローを組んでみると、自動化できるのはタスクの実行部分だけで、判断軸が曖昧なままでは何も自動化できない。フロー設計は自動化の前提条件だった。

以下では、その過程で決めた 5 つのことを書く。

---

## 決めたこと 1: 入金条件 (前払い・着手金 50%・7 日自動検収)

最初に決めたのが入金条件だ。

設定した内容は以下のとおりだ。

* **単発診断 (AIレディネス診断)**: 全額前払い
* **短期実装支援**: 着手金 50% 前払い、残 50% は検収完了後 14 日以内
* **月額伴走**: 初月のみ前払い推奨、翌月以降は月末締め翌月末払い

「検収期間 7 日間、この間に連絡がなければ自動検収とみなす」という条件も、契約書に最初から組み込んだ。

なぜ前払いにこだわったか。後払いにすると、納品後に「ちょっと追加でこれも」が発生したとき、言い出しにくくなるからだ。入金前なら「それは別途見積もりが必要です」と言える。入金後だとなぜか「まあこれくらいは無料でいいか」が始まる。これは性格の問題ではなく、条件設計の問題だ。

自動検収も同様だ。「検収します」という一言を顧客に言ってもらうのを待つと、それが来ない限りずっと「作業が終わっていない状態」が続く。7 日間という期間と「期間内に連絡がなければ完了」という定義を先に合意しておけば、この曖昧さが消える。

向かないケースもある。顧客が「前払いは稟議が通らない」という法人の場合、この条件が壁になる可能性がある。そのときは月末締め翌月末払いに変更するか、最低契約期間 3 ヶ月の月額伴走として組み直すかを、商談の初期段階で確認することにしている。

---

## 決めたこと 2: 並行案件の上限 (月額 2 件 + 単発 1 件)

「何件まで同時に受けるか」を先に決めた。

設定した上限は「月額伴走 2 件 + 単発案件 1 件」だ。短期実装支援が入る場合は単発 1 件に含める形で、同時並行は原則 1 件まで。

この数字は感覚で決めたのではなく、稼働時間から逆算した。

私は「週 30 時間を超えない」という制約を事業設計の前提に置いている。それを崩すと、自分の実験プロジェクト (記事パイプライン等) への時間が消えるし、そもそも自分が AI ツールを実際に使い込む時間がなくなる。コンサルタントとして「使い切ってから語る」スタンスを維持するためには、自分で手を動かす時間は死守する必要がある。

稼働時間の内訳を試算するとこうなる。

| 用途 | 週あたり時間 |
| --- | --- |
| 月額伴走 2 件 (定例・非同期・実装補助) | 10〜12 時間 |
| 単発診断 1 件 (月内に分散) | 2〜3 時間 |
| 営業・提案・社内業務 | 3〜4 時間 |
| 自分の実験プロジェクト (記事パイプライン等) | 5〜7 時間 |
| ツール検証・学習 | 3〜4 時間 |
| **合計** | **23〜30 時間** |

「月額伴走 2 件 + 単発 1 件」でギリギリ週 30 時間に収まる。ここに短期実装支援 (週 8〜10 時間のピーク) が入ると超える。だから短期実装支援を受ける場合、その期間だけは単発を止める、または伴走の 1 件は定例を隔週にしてもらう、という調整を前提にしている。

「断らない、ずらす」というルールも決めた。上限を超えた引き合いが来たとき、その場で断るのではなく「翌月以降なら対応できます」と伝える。これは顧客への誠意でもあるし、チャンスを逃さないためでもある。

---

## 決めたこと 3: サポート範囲の明文化 (月 10 件・30 分上限・有料相談誘導)

「どこまでサポートするか」を事前に決めないと、良心が仇になる。

受託の現場で一番消耗するのは、範囲外の相談に対して「まあいいか」と答え続けることだ。1 件あたりのコストは小さく見えるが、積み重なると本業の作業時間を侵食する。しかも無料で対応しているうちに「それくらいは当然してくれるはず」という期待値になる。

私が決めたのは次の 3 つだ。

**1. 1 件あたりの対応時間は 30 分を上限とする**

30 分を超えそうになったら「現象の再現手順を教えてください」と一度返す。情報収集を依頼することで、自分の調査時間を確保しながら、複雑な問題かどうかを見極める時間を稼ぐ。それでも解決しない場合は「個別対応の範囲を超える」と明示し、有料相談に案内する。

**2. 月 10 件を超えたら有料相談 (30 分 5,000 円) に誘導する**

無料サポートの上限を月 10 件に設定した。10 件を超えた場合は「現在サポートが集中しており対応に時間をいただいています」と返し、有料相談枠を案内する。

この設定をしたのは、サポートが詰まった場合の逃げ道を事前に用意するためだ。「お金を取るのが心苦しい」という感情より、「月 10 件を超えたら有料に移行する」という先に決めたルールの方が、実際の場面では使いやすい。

**3. サポートしない範囲を明示する**

* 購入者の特殊な環境 (社内プロキシ、MDM 設定等) に起因する問題
* パイプラインのカスタマイズ・新ソース追加・プロンプト改造
* API 仕様変更への修正対応 (影響評価のみ、修正はベストエフォート)

「〜はサポート外です」と言えるかどうかは、事前にこれを書いてあるかどうかにかかっている。書いていなければ、断ることが「裏切り」に見える。書いてあれば、最初から合意した条件の確認になる。

## 決めたこと 4: 顧客に求めるもの (BYOK / 環境準備は顧客側)

これは「顧客をお客様扱いしない」という設計判断だ。

私のサービスは、Claude Code を使った AI 実装支援が中心になる。ということは、顧客側に Anthropic の API キーが必要になる。環境準備も顧客側に一定のリテラシーが要る。

最初、「API キーの取得から一緒にやってあげる」方が親切で、受注につながるのではないかと考えた。でも、それを受け入れると、全員のオンボーディングに同じ時間がかかる。しかも、API キーの取得でつまずくような顧客は、その後の実装支援でもつまずく可能性が高く、サポートコストが上がる。

BYOK (Bring Your Own Key: 自分の API キーを用意する) を前提にした理由はもう一つある。顧客データを私のキーで処理すると、その顧客のデータが私のアカウントを通ることになる。顧客情報の取り扱いを最小化するためにも、顧客が自分のキーを持つ方が責任分界点がはっきりする。

「商談の初期段階で、使えるツールと環境を確認する」という項目をヒアリングリストに組み込んだ。確認するのはこういった点だ。

* 現在使っているツール・システムの一覧
* Anthropic / OpenAI 等の API キーを取得できるか
* 意思決定者の関与度 (担当者のみか、経営者も動かせるか)

これで「環境が何もない」状態の顧客を、商談の初期段階でスクリーニングできる。「準備が必要なら一緒に整えましょう」と言いたい気持ちもあるが、それは月額伴走の文脈で、費用をいただきながらやることだと決めた。

---

## 決めたこと 5: 撤退基準 (Day 30 / Day 90 / 6 ヶ月)

最後に決めたのは、「やめる基準」だ。

撤退基準を事前に決めておくことの価値は、感情的にならずに判断できることにある。基準がなければ、うまくいっていないときに「もう少し続ければいけるかもしれない」と希望的観測で引っ張ってしまう。基準があれば、「この数字を下回ったら見直す」という判断を、事前に決めた自分がしてくれる。

私が設定した撤退基準はこうだ。

**Day 30 (1 ヶ月)**

* 記事パイプラインのスターターキット: 売上 10,000 円 (最低 4 件) を下回れば撤退検討
* 問い合わせ: 0 件なら発信戦略を見直す

**Day 90 (3 ヶ月)**

* コンサル受注: 累計 1 件以上なければ商品設計を見直す
* 月次売上: 50,000 円以上になっていなければ単価またはターゲットを変える

**6 ヶ月**

* 月次売上が継続して 100,000 円を下回る場合、事業継続か縮小かを判断する

これらの数字は「達成できそうな感じがするライン」ではなく、「ここを超えなければ意味がない」というラインで設定した。

撤退基準を事前に決めることのもう一つの利点は、「失注しても消耗しない」ことだ。基準に達していない状態でも「まだ撤退基準には達していない、続ける」という判断ができる。逆に、基準を超えていても「撤退基準はクリアした、では次のフェーズへ」という思考に切り替えられる。

「続けることへの執着」と「やめることへの恐れ」を、事前に決めたルールが引き受けてくれる。

## なぜ設計を先に決めるか: 失注しても消耗しないため

5 つの判断を通じて気づいたことがある。

フローを先に設計することの最大の効用は、「失注しても大丈夫」な状態を作ることだ。

相談が来て、断られる。それは必ず起きる。そのとき、「フローがなかったから断られた」と感じるか、「フローはある、次の相談に備えよう」と感じるかは、かなり違う。

もう一つは判断疲れを減らすことだ。相談が来るたびに「この案件、費用どうしようか」「どこまでやろうか」と考えていると、判断のコストが相談ごとにかかる。型があれば、判断のコストは「この型に当てはめられるか」だけになる。

受注ゼロの段階でここまで書くのは、「実績がある人が後から語る話」ではなく「これから試す人が設計段階で何を考えたか」の記録として意味があると思ったからだ。半年後に同じ設計で動いていたら、それが検証になる。動いていなければ、何を変えたかを書く。

## 動かし始めた

この 5 つの判断をベースに、Claude Code で受注フローのテンプレートを組んだ。

ヒアリング議事録のフォーマット、提案書の自動ドラフト生成、AI レディネス診断レポートの出力——これらは Claude Code の Skill ファイルとして定義してある。相談が来たとき、ヒアリングした内容を渡せば提案書の骨格が出てくる状態になっている。

ただ、フロー設計と自動化基盤を整えただけで案件は来ない。次のフェーズは、設計した型で実際に顧客と向き合う実運用の段階だ。Day 30 のレポートは次の記事に書く予定だ。
