## はじめに：流出したのは「ハーネス」だった

2026年3月31日、Anthropic の AI コーディングエージェント「Claude Code」のソースコードが流出しました。npm パッケージにソースマップファイルが誤って同梱されていたことが原因で、約 51 万行・1,900 ファイルが誰でも閲覧できる状態になっていました。

このニュースを見て気になったのは、報道の中で繰り返し使われていた「ハーネス」という言葉です。

> 流出したのはエージェントの「ハーネス」であり、モデルの重みや学習データではない
>
> — [Fortune](https://fortune.com/2026/03/31/anthropic-source-code-claude-code-data-leak-second-security-lapse-days-after-accidentally-revealing-mythos/)

「ハーネス」とは何なのか。なぜモデル本体ではなくハーネスが競争力の源泉として語られるのか。調べてみると、2026年2月から「ハーネス」という言葉が LLM の世界で急速に広まっていたことが分かりました。

![Google Trendsにおける「harness engineering」関連キーワードの検索推移](https://static.zenn.studio/user-upload/95270d7bb728-20260406.png)  
*出典: Google Trends（2024年12月〜2026年4月）*

しかし「ハーネス」には複数の意味があります。評価ハーネス、エージェントハーネス、ハーネスエンジニアリング。これらが混在したまま語られている状況です。

本記事では、語源から最前線の研究まで体系的に整理します。読み終わる頃に「ハーネス」の全体像が頭の中に描けるようになることを目標にします。

## 「ハーネス」の語源と系譜

### 原義：馬具としての harness

「harness」の原義は馬具です。手綱・鞍・くつわなど、馬に装着する装備一式を指します。核心にあるのは「強力だが予測不可能な動物を、有用な方向に導く装備」という意味です。

> 馬は単体では強力だが、ハーネスがなければ農作業には使えない
>
> — [Agentic Harness Engineering: LLMs as the New OS](https://www.decodingai.com/p/agentic-harness-engineering)

強力だが予測不可能なモデルを、人間の目的に沿って動かすための装備。これが LLM 文脈での「ハーネス」の根底にある考え方です。

### 前史：テストハーネス

LLM ハーネスの直接の祖先は、ソフトウェア工学の「テストハーネス」です。

テストハーネスとは、テスト対象のコンポーネントを動かすために用意するスタブやドライバの集合体で、ソフトウェア工学では古くから使われてきた概念です。

共通するのは「対象をそのまま使うのではなく、装備で包んで目的通りに動かす」という発想です。

**馬具 → テストハーネス → LLM ハーネス**。この系譜を知っておくと「ハーネス」という言葉の意味がぶれにくくなります。

![ハーネスの語源の系譜](https://static.zenn.studio/user-upload/b8687920f235-20260406.png)

## ハーネスの全体像

LLM の文脈で「ハーネス」と呼ばれるものは、大きく2つに分かれます。

| 分類 | 一言でいうと | 例 |
| --- | --- | --- |
| 評価ハーネス | LLM を**測る**足場 | lm-evaluation-harness |
| エージェントハーネス | LLM を**動かす**装備一式 | CLAUDE.md、ツール、ループ |

この2つは別系統です。そして、エージェントハーネスを設計・改善し続ける活動が**ハーネスエンジニアリング**と呼ばれています（後述）。

![ハーネスの分類と関係](https://static.zenn.studio/user-upload/52f079f8afc9-20260406.png)

以下、それぞれを詳しく見ていきます。

### 評価ハーネス — LLM を「測る」足場

評価ハーネスはテストハーネスの直系の子孫です。LLM の性能をベンチマークで測定するための標準化フレームワークを指します。

代表例は EleutherAI の [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) です。60 以上の学術ベンチマークを統一インターフェースで実行でき、Hugging Face の Open LLM Leaderboard のバックエンドとしても使われています。

> モデル性能はしばしば実装の細部に左右される。あるコードベースの結果が別のコードベースに直接転用できることを期待するのは非現実的だ
>
> — [Evaluating LLMs — EleutherAI](https://www.eleuther.ai/projects/large-language-model-evaluation)

ここで重要な区別があります。評価ハーネスは「LLM を制御する」ものではありません。「LLM を評価するための標準化された足場」です。次に紹介するエージェントハーネスとは完全に別の文脈です。

### エージェントハーネス — Agent = Model + Harness

エージェントハーネスは、LLM エージェントを取り巻く「モデル以外のすべて」です。

LangChain がこの定義を明確に打ち出しました。

> モデルでなければハーネス。生のモデルはエージェントではない。ハーネスが状態、ツール実行、フィードバックループ、強制可能な制約を与えることで、モデルはエージェントになる
>
> — [The Anatomy of an Agent Harness - LangChain Blog](https://blog.langchain.com/the-anatomy-of-an-agent-harness/)

この「モデルでなければハーネス」という二分法がシンプルで強力です。プロンプトも、ツール呼び出しのコードも、CI も、すべてハーネスに含まれます。

#### 二層構造：プログラムレベルとプロンプトレベル

エージェントハーネスは内部的に2つの層で構成されています。

**プログラムレベルハーネス** — LLM の外側にあるコードやインフラです。決定論的に動作します。

* エージェントループ本体（推論 → ツール呼び出し → 結果フィードバック）
* ミドルウェア / Hooks
* リンター・型チェック・CI
* サンドボックス・権限管理
* ループ検出・タイムアウト管理

**プロンプトレベルハーネス** — LLM のコンテキストウィンドウに入れて自然言語で制御する部分です。

* AGENTS.md / CLAUDE.md
* スキルファイル（必要なときだけ読み込ませる Progressive Disclosure）
* 構造化ドキュメント（docs/）
* システムプロンプト

両者は入れ子構造になっています。プログラムレベルのミドルウェアが「今このコンテキストウィンドウに何を入れるか」を決定し、その結果としてプロンプトレベルのハーネスが形成されます。

![エージェントハーネスの二層構造](https://static.zenn.studio/user-upload/6d1bfff42046-20260406.png)

#### Böckeler の 2×2 マトリックス

Thoughtworks の Birgitta Böckeler は、ハーネスの制御を別の切り口で整理しました。

|  | Computational（計算的） | Inferential（推論的） |
| --- | --- | --- |
| **ガイド**（事前制御） | リンター、型チェック | AI コードレビュー |
| **センサー**（事後制御） | テスト実行、差分検出 | LLM による品質判定 |

**ガイド**はエージェントの行動を事前に操舵します。**センサー**は行動後に観察して自己修正を助けます。それぞれに決定論的な方法と推論的な方法があります。

この分類は、自分のハーネスに何が足りないかを診断するのに便利です。

## ハーネスエンジニアリング

ハーネスエンジニアリングとは、エージェントハーネスを設計・改善し続ける規律です。コンテキスト設計、ガードレール、フィードバックループといった個別テクニックは以前から存在していました。では、なぜ 2026年にわざわざひとまとめの言葉にしたのでしょうか。

### タイムライン：わずか2週間で広まった

概念の断片は 2025年後半から存在していました。しかし用語としてまとまったのは、2026年2月のわずか数週間のことです。

![ハーネスエンジニアリングのタイムライン](https://static.zenn.studio/user-upload/b9bcd44e6926-20260406.png)

2025年11月に Anthropic が「[Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)」を公開したのが萌芽です。そして2026年2月5日、命名者の Mitchell Hashimoto（HashiCorp 共同創業者）がブログで "harness engineering" という言葉を使いました。6日後の2月11日には OpenAI が、2月17日には Böckeler（Martin Fowler のサイト）が記事を公開し、LangChain も同月中に続いています。

Hashimoto はこう述べています。

> 業界で広く受け入れられた用語があるかどうかわからないが、私はこれを "harness engineering" と呼ぶようになった。エージェントがミスをしたら、そのミスを二度と起こさないようにエンジニアリングするというアイデアだ
>
> — [My AI Adoption Journey](https://mitchellh.com/writing/my-ai-adoption-journey)

Böckeler は2月のメモでこの動きを評しています。

> まだ2週間しか経っていない言葉だ。いずれにせよ、AIエージェントを制御するためのツーリングと実践を表す言葉として、私は harness を気に入っている
>
> — [Harness Engineering (memo) - martinfowler.com](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering-memo.html)

### なぜ統合概念が必要になったか

背景にあるのは、モデルの能力向上がボトルネックを移動させたことです。

> ボトルネックは「コードを生成できるか」ではなく「実システムの中で信頼できる振る舞いをさせられるか」に変わった
>
> — [Harness Engineering: The Missing Layer Behind AI Agents](https://www.louisbouchard.ai/harness-engineering/)

個別テクニックの限界も露呈しました。OpenAI の Codex チームは「巨大 AGENTS.md」アプローチの失敗を報告しています。

> コンテキストは稀少なリソースであり、巨大な指示ファイルはタスク、コード、関連ドキュメントを押しのけてしまう。すべてが「重要」だと何も重要でなくなる
>
> — [OpenAI - Harness engineering](https://openai.com/index/harness-engineering/)

プロンプトエンジニアリング（2022〜2024年）、コンテキストエンジニアリング（2025年）、ハーネスエンジニアリング（2026年）。焦点は「リクエストの表現」→「モデルに見せるもの」→「システム全体の設計」へと拡大してきました。個別の工夫だけでは足りず、それらを統合するフレームワークが求められたのです。

### 3つの柱

OpenAI、Anthropic、Martin Fowler の記事から、共通する3つの柱が抽出できます。

**柱1: コンテキスト設計** — モデルに何を見せるかの設計です。

OpenAI は「地図を与えよ、百科事典ではなく」と表現しています。AGENTS.md は約 100 行の目次に留め、詳細は docs/ に分離する構成です。前述の Progressive Disclosure が鍵になります。

**柱2: ガードレール** — 決定論的な制約でモデルの行動を制限します。

カスタムリンター、型チェック、構造テスト、サンドボックスなどです。モデルの出力を「信頼するが検証する」ための仕組みです。

**柱3: フィードバックループ** — 行動後に検証し、自己修正を促します。

テスト自動実行、エージェント間レビュー、ドキュメントの不整合検出などが含まれます。LangChain の報告によると、エージェントの最も一般的な失敗パターンは「自分のコードを再読して『よさそうだ』と判断して止まること」です。元の仕様に対して検証させるループが効果を発揮します。

### 定量的な検証

「ハーネスを変えるだけで本当に差が出るのか」。LangChain がこの問いに数字で答えています。

Terminal Bench 2.0 はコーディングエージェントの総合ベンチマークです。LangChain はこのベンチマークで、**モデル（GPT-5.2-Codex）を固定したままハーネスだけを改善**し、スコアを 52.8% から 66.5% へと 13.7 ポイント引き上げました。

特に効果が大きかったのは「推論サンドイッチ」と呼ばれる工夫です。LLM の API には reasoning budget（モデルにどれだけ深く考えさせるかの上限）を設定するパラメータがあり、xhigh / high などのレベルで指定します。すべての工程を最も深い xhigh で走らせれば精度が上がりそうですが、処理時間が膨らんでタイムアウトが多発し、スコアは 53.9% に留まりました。そこで、計画と検証には xhigh、実装には high という緩急をつけた配分にした結果、66.5% を達成しています。

モデルを一切変えずにハーネスだけで 13.7 ポイント改善できたことは、ハーネスエンジニアリングの実用的な価値を裏付けています。

## ハーネスはどこに向かうか

2026年に公開された Meta-Harness は、ハーネス自体を自動で最適化するアプローチです。

手設計のハーネスをベースに、提案者（Claude Code）がソースコード・スコア・実行トレースを分析して改善を繰り返します。LawBench（215クラスの法律テキスト分類）で +16 ポイント、先行手法の10分の1の評価回数で同等精度を達成しました。

### 自然言語でハーネスを記述する

通常、ハーネスは Python や TypeScript などのコードで実装します。しかしコードで書かれたハーネスは特定のフレームワークに依存するため、別のエージェントに移植したり、異なるハーネス同士を比較したりすることが困難です。

NLAH（Natural Language Agent Harness）は、清華大学・ハルビン工業大学の研究者らが提案したアプローチで、ハーネスの設計をコードではなく自然言語の統一フォーマットで記述します。「このツールをこの順序で使い、結果がこうならやり直す」といった制御ロジックを、フレームワーク非依存の自然言語で表現します。

コードベースのハーネスを NLAH に移行した結果、OS World（デスクトップ操作のベンチマーク）で 30.4 → 47.2 にスコアが向上しました。自然言語で書くことでモデルが指示を理解しやすくなり、かつ異なるエージェント間での再利用が容易になるという利点があります。

### モデルとハーネスの共進化

モデルとハーネスは互いに影響し合う関係です。

Windsurf（AI コーディングエージェント企業、旧 Codeium）は、自社のコーディング特化モデル SWE-1.5 のトレーニングで「モデルとハーネスを共最適化した」と報告しています。自社のモデルを自ら日常的に使い（ドッグフーディング）、ハーネスの問題に気づいてツールとプロンプトを調整し、更新されたハーネスでモデルを再トレーニングするサイクルです。

さらに、各社の最高性能モデル（フロンティアモデル）はハーネス上でポストトレーニング（特定の用途に合わせた追加学習）されています。Claude は Claude Code 上で、GPT-5 Codex は Codex 上で最適化されています。ハーネスはもはやモデルの外部にある「付属品」ではなく、モデルの性能と不可分な存在になりつつあります。

Google DeepMind の Philipp Schmid（元 Hugging Face テクニカルリード）はこう指摘しています。「ハーネスはデータセットである。競争優位は今やハーネスがキャプチャする軌跡（trajectories）にある」。

## まとめ

本記事では「ハーネス」にまつわる3つの概念を整理しました。

* **評価ハーネス**: LLM を測る足場。テストハーネスの直系の子孫
* **エージェントハーネス**: LLM を動かす装備一式。「モデルでなければハーネス」
* **ハーネスエンジニアリング**: ハーネスを設計・改善し続ける規律。2026年2月に結晶化

エージェントハーネスはプログラムレベルとプロンプトレベルの二層構造を持ち、ガイド/センサーの2軸で整理できます。

ハーネスエンジニアリングの3つの柱は、コンテキスト設計・ガードレール・フィードバックループです。LangChain の事例ではモデルを固定してハーネスだけで 13.7 ポイント改善しており、実用的な価値は実証済みです。

最前線では、ハーネスの自動最適化（Meta-Harness）、自然言語での統一記述（NLAH）、モデルとの共進化が進んでいます。ハーネスはモデルの「付属品」から、性能と不可分な存在へと変わりつつあります。

冒頭の Claude Code 流出事件で露出したのは、まさにこのエージェントハーネスでした。モデルの重みではなくハーネスが「流出して困るもの」として扱われたこと自体が、ハーネスの重要性を物語っています。

「エージェントが X をした」と聞いたとき、「モデルがそれをしたのか、ハーネスがそれを可能にしたのか？」と問うこと。この問いが、ハーネスを理解する第一歩です。
