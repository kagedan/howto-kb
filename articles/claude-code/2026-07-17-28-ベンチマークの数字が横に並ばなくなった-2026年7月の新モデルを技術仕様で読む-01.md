## 「で、結局どれを使えばいいのか」に答えられない

2026年7月8日から9日にかけて、OpenAI・SpaceXAI（旧xAI）・Metaがそれぞれ新モデルを出した。タイムラインにはベンチマーク表が流れ、「SOTA更新」「Opus級」「GPT-5.5超え」といった文字が並ぶ。

この記事を書くにあたって、その数字を一次情報で裏取りしようとした。結果として、当初書こうとしていた記事は書けなくなった。**数字が確認できない**のだ。SOTAとされる数値は標準構成での測定ではなかったし、あるモデルはモデルカードそのものが存在せず、あるモデルは公式発表にベンチマーク数値が一行も載っていなかった。

この記事では、2026年7月に登場した新モデル群を技術仕様ベースで整理する。同時に、**なぜ今それらを横に並べて比較できないのか**という構造的な話を扱う。ベンチマーク表を眺めてモデルを選ぶ、という運用がそろそろ機能しなくなってきている。

## TL;DR

* **「フロンティア3社が同日ローンチ」は不正確**。Grok 4.5は7月8日（水）、GPT-5.6 GAとMuse Spark 1.1が7月9日（木）だ（[TechCrunch](https://techcrunch.com/2026/07/08/spacexai-releases-grok-4-5-which-elon-describes-as-an-opus-class-model/)）
* Terminal-Bench 2.1のSOTA「91.9%」は**GPT-5.6 SolのUltra mode**（サブエージェント＋追加計算資源）の数値。max effortのFable 5、xhigh effortのGPT-5.5、構成の記載がないGrok 4.5と、同じ表に並んでいる
* **Grok 4.5はモデルカードが未公開**。Grok 4・4.1では公開されていた。訓練データ汚染を申告したのはxAIではなく**Cursor社**で、該当ベンチは脚注で除外された
* **Meta Muse Spark 1.1は公式ブログにベンチマーク数値が一切ない**。出回っている比較値はすべて第三者測定だ。クローズドウェイト＋Meta初の有料API
* Claude Sonnet 5の新トークナイザーは同じ入力が**1.0〜1.35倍**のトークンになる。Anthropicは導入価格をこれを相殺する「費用中立」として設定したと明記している（[Anthropic公式](https://www.anthropic.com/news/claude-sonnet-5)）
* **Gemini 3.5 Proは7月16日時点で未リリース**。出回っている2Mコンテキスト・$1.25/$10などのスペックは全て報道ベースで、公式発表は存在しない

**用語と方法論の注記**

この記事では情報を2種類に分けて扱う。**一次情報**＝各社公式が自ら出したもの（公式ブログ・公式docs・公式pricingページ・公式X投稿）。**二次情報**＝報道機関・第三者分析・ベンチマーク集約サイトが出したもの。なお「SOTA」は state of the art の略で、そのベンチマークでの最高スコアを指す。

## 2. OpenAI GPT-5.6 — Sol / Terra / Luna の3ティア

GPT-5.6は3つのモデルからなるファミリーとして提供される。

|  | Sol | Terra | Luna |
| --- | --- | --- | --- |
| 入力 | $5.00 | $2.50 | $1.00 |
| 出力 | $30.00 | $15.00 | $6.00 |
| 位置づけ | 最高難度の推論・コーディング・科学 | コスト効率重視 | 高速・大量処理 |

（per 1M tokens。キャッシュ入力はSolで$0.50。出典: [developers.openai.com](https://developers.openai.com/api/docs/models/gpt-5.6-sol)、[Simon Willison](https://simonwillison.net/2026/Jul/9/gpt-5-6/)）

共通スペックは、コンテキスト100万トークン・最大出力128,000トークン・知識カットオフ2026年2月16日（[developers.openai.com](https://developers.openai.com/api/docs/models/gpt-5.6-sol)）。モデルIDは`gpt-5.6-sol`が公式開発者ドキュメントで確認できたが、**TerraとLunaの正式なモデルID文字列は未確認**だ。

API側の新機能として、Programmatic Tool Calling、Multi-agent機能、Prompt cache breakpoints、画像処理の`detail: original`オプション、そして`max` reasoning effortの追加が挙げられている（[Simon Willison](https://simonwillison.net/2026/Jul/9/gpt-5-6/)）。

### 「Spudベース」という説明について

GPT-5.6が「約4兆パラメータのSpudベース」という説明が流通しているが、これは正確ではない。**「Spud」はGPT-5.5のコードネーム**であり、2026年4月23日リリースのGPT-5.5が約4兆パラメータのMoE基盤とされている（[Axios](https://www.axios.com/2026/04/23/openai-releases-spud-gpt-model)）。GPT-5.6がSpud基盤を継承しているのか新基盤なのかは、公式には明言されていない。**GPT-5.6がMoEなのかどうかすら未確認**である。

### Ultra mode — ここが後半の伏線になる

Solには「Ultra mode」と、新しい`max` reasoning effort設定がある。Ultra modeの実体は「**サブエージェントを活用し、より多くの計算資源をタスクに投入することでスコアを引き上げるモード**」だ（[OpenAI Developer Community](https://community.openai.com/t/introducing-gpt-5-6-series-sol-terra-and-luna-coming-july-9-10am-pt/1384931)）。

reasoning effortの効き方は、第三者検証機関であるARC Prize Foundationの測定が分かりやすい。ARC-AGI-3でのSolのスコアは、low effortで0.3%、max effortで7.78%（Semi-Private）・13.33%（Public）だ（[arcprize.org](https://arcprize.org/results/openai-gpt-5-6-sol)）。**同一モデルで25倍以上の差**がつく。「GPT-5.6 Solのスコア」と一言で言っても、設定次第でこれだけ動く。

## 3. SpaceXAI Grok 4.5 — モデルカードのない「Opus級」

まず前提の整理から。xAIは2026年2月にSpaceXが株式交換で買収し、7月6〜7日に正式に「SpaceXAI」へリブランドした（[Dataconomy](https://dataconomy.com/2026/07/07/elon-musk-rebrands-merged-xai-and-spacex-as-spacexai/)）。Grok 4.5は上場後初のモデルリリースにあたる。

| 項目 | 値 | 情報種別 |
| --- | --- | --- |
| リリース | 2026年7月8日 | TechCrunch経由（高信頼） |
| 基盤 | 「V9」1.5兆パラメータ MoE | **二次情報・自己申告** |
| active parameter | 未確認 | — |
| コンテキスト | 500K | 二次情報 |
| 最大出力 | 未確認 | — |
| 入力/出力 | $2 / $6（キャッシュ$0.50） | TechCrunch経由（高信頼） |

出典: [TechCrunch](https://techcrunch.com/2026/07/08/spacexai-releases-grok-4-5-which-elon-describes-as-an-opus-class-model/)、[CometAPI](https://www.cometapi.com/grok-4-5-leak-xai-s-1-5t-v9-model-in-private-beta/)、[Digital Applied](https://www.digitalapplied.com/blog/grok-4-5-cursor-data-flywheel-spacex-private-beta-2026)

価格が目を引く。Opus 4.8が$5/$25、Fable 5が$10/$50であることを考えると、$2/$6は「Opus級」を主張するモデルとしてはかなり安い。

Terminal-Bench 2.1では83.3%という数値が複数の独立ソースで一致している。同じ表に並ぶ他モデルは、GPT-5.5（xhigh）83.4%、Claude Fable 5（max）84.3%、Opus 4.8（max）78.9%だ（[the-decoder.com](https://the-decoder.com/grok-4-5-is-so-cheap-compared-to-fable-5-and-gpt-5-5-that-benchmark-gaps-may-not-matter-much/)、[apidog.com](https://apidog.com/blog/grok-4-5-benchmarks/)）。

### Cursorのデータで学習した、という話

Grok 4.5の特徴として喧伝されているのが、Cursorの実エージェント操作ログによる学習だ。実開発者のワークフロー・デバッガ操作・複数ファイルdiff・ユーザーによる修正といったデータが、V9基盤への補完学習に使われたとされる（[Digital Applied](https://www.digitalapplied.com/blog/grok-4-5-cursor-data-flywheel-spacex-private-beta-2026)）。学習に使われたデータの規模やopt-in/opt-outの条件については、二次記事の記述に幅があり**確定的なことは言えなかった**。

### 「Opus 4.8の25%のトークンで済む」を追いかけた結果

Grok 4.5の紹介でよく見かけるのが、「同等タスクをOpus 4.8の約25%の出力トークンで完了する」というトークン効率の主張だ。この記事でも当初はこの数値を採用するつもりだった。

**出所を特定できなかった。**

根拠としてよく挙げられるMindStudioの比較記事を実際に開いて確認したが、そこにあったのは「Claude Opus 4.8は入力・出力ともGrok 4.5のおよそ5倍高い」という**単価の話**だけで、タスクあたりの出力トークン数の実測値は載っていない。同ページが引用しているベンチマークもSWE-bench ProではなくSWE-bench Verified（Grok 4.5 約63% / Opus 4.8 約60%）だった（[MindStudio](https://www.mindstudio.ai/blog/grok-4-5-vs-claude-opus-4-8-agentic-coding)）。

「25%」がどこかの実測に基づいている可能性は否定しない。だが**広く引用されている出典を当たっても裏が取れず**、xAIはモデルカードを公開していないので公式側で確認する手段もない。したがってこの記事ではトークン効率の数値を採用しない。

これは今回の調査で何度も起きたことの一例だ。**印象的な数字ほど、出所を辿ると誰も測っていない。**

### そして、モデルカードがない

ここが今回の調査で一番引っかかった点だ。**Grok 4.5にはモデルカードが公開されていない**。Grok 4とGrok 4.1では公開されていたにもかかわらず、である。1.5兆パラメータという構成も独立監査は存在せず、自己申告に依存している。

この文脈で、訓練データ汚染の話が出てくる。よく「Grok 4.5は訓練データ汚染を自己申告した」と説明されるが、**これは不正確だ**。実際に開示したのは**Cursor社**であり、しかもブログの脚注においてである。内容は、CursorBench（Cursor自社コードベースを元にした評価）の採点対象コードベースのスナップショットが誤ってGrok 4.5の訓練データに混入し、train/testの重複が生じたため、CursorBenchをローンチ時のベンチマーク表から除外した、というものだ（[paddo.dev](https://paddo.dev/blog/grok-4-5-opus-class-token-bill/)、[kie.ai](https://kie.ai/blog/grok-4-5-canary-traces-cursor-training-1-5t-v9-foundation)）。

公平を期すために書いておくと、汚染の影響範囲はCursorBenchに限定され、DeepSWE・SWE-Bench Pro・Terminal-Benchといった第三者ベンチマークは独自インフラを使うため影響を受けない、と複数のソースが明記している。つまり83.3%という数値自体が汚染されているわけではない。

問題は構造の方だ。**自社ベンチが自社コードで汚染され、その事実が提携先の脚注で開示され、当該ベンチは静かに表から消えた**。そしてモデル本体のカードは存在しない。責められるべきは個別の数字ではなく、検証可能性が担保されていない状態でSOTA主張だけが流通することだと思う。

公式ブログで確認できる一次情報は以下だ（[Meta公式ブログ](https://ai.meta.com/blog/introducing-muse-spark-meta-model-api/)）。

最後の点は地味だが技術的に面白い。毎ステップ1クリックずつ推論する素朴なcomputer useではなく、**スクリプト生成とGUI操作を状況で使い分ける**設計になっている。エージェントのcomputer useが「人間のマネ」から「機械にとって効率的な経路の選択」へ移りつつある。

価格は入力$1.25/出力$4.25（per 1M tokens）、$20分の無料クレジット付き。**ただしこの価格数値は公式ブログ本文には見当たらず**、Bloomberg等の報道での確認だ（[Bloomberg](https://www.bloomberg.com/news/articles/2026-07-09/meta-starts-charging-for-ai-with-muse-spark-1-1-agentic-model)）。

そしてMuse Spark 1.1は**クローズドウェイト**である。ローカル実行もコミュニティによるファインチューニングもできない。オープンウェイト配布を旗印にしてきたLlamaファミリーとは真逆の構造だ。複数の報道が「Metaが自社AIモデルに初めて値段を付けた」と一致して報じている（[Bloomberg](https://www.bloomberg.com/news/articles/2026-07-09/meta-starts-charging-for-ai-with-muse-spark-1-1-agentic-model)、[The New Stack](https://thenewstack.io/meta-muse-spark-api/)）。

### 公式ブログにベンチマーク数値が一行もない

調査していて驚いたのがこれだ。Meta公式ブログには、**GPT-5.5やOpus 4.8との比較数値が一切記載されていない**。流通している比較値はすべて第三者測定である。

| ベンチマーク | Muse Spark 1.1 | Opus 4.8 | GPT-5.5 |
| --- | --- | --- | --- |
| MCP Atlas（スケールしたツール利用） | **88.1** | 70後半〜80前半 | 70後半〜80前半 |
| JobBench（業務系ツール利用） | **54.7** | 48.4 | 38.3 |
| Terminal-Bench 2.1（コーディング） | 80.0 | 82.7 | **83.4** |
| DeepSWE 1.1（長時間の自律コーディング） | 53.3 | 59.0 | **67.0** |

出典: [officechai.com](https://officechai.com/ai/muse-spark-1-1-benchmarks/)、[kingy.ai](https://kingy.ai/blog/muse-spark-1-1-benchmarks-specs-evals/) — **いずれも二次情報**

この表は素直に読むと示唆的だ。Muse Spark 1.1は**ツール利用系で勝ち、コーディング系で負けている**。JobBenchではGPT-5.5に16.4ポイントの差をつけて勝つ一方、DeepSWE 1.1では逆に13.7ポイント差で負ける。Terminal-Bench 2.1の差は3.4ポイントと小さいが、順位としてはやはり最下位だ。

つまり**同じ3モデルの順位が、軸によってきれいに反転する**。JobBenchでは Muse Spark > Opus 4.8 > GPT-5.5、DeepSWE 1.1では GPT-5.5 > Opus 4.8 > Muse Spark。完全な逆順である。この現象は第7章(4)で扱う。

### Sonnet 5のトークナイザーと「費用中立」

技術的に一番押さえておくべきはここだ。Sonnet 5はOpus 4.7で導入された新トークナイザーを採用しており、**同じ入力がおよそ1.0〜1.35倍のトークン数にマップされる**とAnthropicが公式に明記している（[Anthropic公式](https://www.anthropic.com/news/claude-sonnet-5)）。

「約30%多い」という表現が流通しているが、公式の記述は1.0〜1.35倍というレンジであり、厳密には「最大約35%」がより正確だ。しかも増加幅はワークロードの性質で変わる。

そして重要なのが、\*\*Anthropicが導入価格（$2/10）を、このトークン数増加分を相殺する「費用中立」として設定したと明記している\*\*点だ。これは価格表の裏で行われている調整を公式が認めているということで、後述する通り、/tokenという単位そのものの信頼性に関わる話になる。

## 6. Gemini 3.5 Pro — まだ何も確定していない

7月17日リリース、2Mコンテキスト、Deep Think推論レイヤー、$1.25/$10、「2.5 Proのアーキテクチャを捨てて完全に作り直し」——これらの情報が広く流通している。

**7月16日時点で、そのいずれも公式発表ではない。**

blog.google・deepmind.google・gemini.googleを検索した範囲で、Gemini 3.5 Proの正式発表ページは存在しない。報道各社自身が「every spec remains unconfirmed（全スペックが未確認のまま）」と明記している状態だ（[TechTimes](https://www.techtimes.com/articles/320308/20260713/gemini-35-pro-targets-july-17-after-full-rebuild-every-spec-remains-unconfirmed.htm)）。「Gemini 4 Flash」に至っては名称自体が公式ソースに見当たらない。

Google側で7月に確認できた一次情報は別のところにある。**AlphaEvolveの一般提供**（7月10日、[Google Cloud公式ブログ](https://cloud.google.com/blog/products/ai-machine-learning/alphaevolve-is-available-for-everyone)）だ。Gemini Enterprise Agent Platform経由で全Google Cloud顧客に提供される。

技術的な実体は、Geminiを基盤とした進化的アルゴリズム最適化エージェントだ。ユーザーが(1)Seed program（最適化対象のベースコード）と(2)Evaluator（決定論的な採点スクリプト）を渡すと、エージェントがコードの変異体を自動生成・探索し、人間可読な最適化コードを返す。Define→Measure→Optimize→Applyの4段階で、BASF・JetBrains・Kinaxis等が早期採用している。

これは地味だが、\*\*「決定論的な評価関数さえ書ければ、あとはLLMが探索する」\*\*という形式で、ベンチマーク不信の話とは逆方向のアプローチだ。評価を自分で定義する側に回っている。

## 7. 独創的な視点 — なぜ横に並べられないのか

ここからが本題だ。7月の新モデル群の数字を集めていて浮かび上がった、4つの構造的な問題を扱う。

### (1) 「91.9%」は何の数字か — 構成の違うものが同じ表に載っている

Terminal-Bench 2.1のスコアを、流通している通りに並べるとこうなる。

| モデル | スコア | 測定構成 |
| --- | --- | --- |
| GPT-5.6 Sol | 91.9% | **Ultra mode**（サブエージェント＋追加計算資源） |
| Claude Fable 5 | 84.3% | **max** effort |
| GPT-5.5 | 83.4% | **xhigh** effort |
| Grok 4.5 | 83.3% | 記載なし |
| Muse Spark 1.1 | 80.0% | 記載なし |
| Opus 4.8 | 78.9% | **max** effort |

出典: 91.9%は[OpenAI公式X投稿](https://x.com/OpenAI/status/2070555276370169969)。84.3 / 83.4 / 83.3 / 78.9は[the-decoder.com](https://the-decoder.com/grok-4-5-is-so-cheap-compared-to-fable-5-and-gpt-5-5-that-benchmark-gaps-may-not-matter-much/)。80.0は[officechai.com](https://officechai.com/ai/muse-spark-1-1-benchmarks/)

一列に並んでいるが、**測っているものが揃っていない**。SolのSOTA値はサブエージェントを動かして計算資源を追加投入した構成の結果だ。Fable 5とOpus 4.8はmax effort、GPT-5.5はxhigh。Grok 4.5とMuse Spark 1.1に至っては構成の記載がない。

そもそもこの表は**一つのソースから引いたものですらない**。3つの別々の記事から数字を拾って並べている。各社が自社に有利な構成で発表した数値を、第三者が横に並べ直したものが「ベンチマーク比較表」として流通している、というのが実態に近い。

なお、SolのUltra modeでない標準構成のスコアとして88.8%という数値も流通しているが、**この記事では出所を特定できなかった**。確認できたのは「91.9%がUltra modeの値である」という点までだ。

設定差がどれくらい効くかは、第三者検証機関であるARC Prizeの測定が参考になる。ARC-AGI-3ではSolのスコアがlow effortで0.3%、max effortで7.78%（Semi-Private）まで動いた（[arcprize.org](https://arcprize.org/results/openai-gpt-5-6-sol)）。ARC-AGIはパズル系で分散が大きいベンチマークとして知られるので、この効果量がそのままTerminal-Benchに当てはまるとは限らない。だが**同一モデルでも設定次第でスコアが動く**こと自体は、上の表の数ポイント差を読むうえで無視できない。

「91.9% > 83.3%だからSolの方が上」——この読み方が成立するには、両者が同じ構成で測られている必要がある。そうなっていない。

さらに前回の記事で触れた通り、Fable 5のHumanity's Last Exam 53%というスコアは、8〜9%のタスクでセーフガードが発動しOpus 4.8にフォールバックした状態での測定だった。**そもそも全問を同じモデルが解いていない**ケースまである。

ベンチマーク表は「同じ試験を同じ条件で受けた結果」という前提で読まれるが、その前提はすでに崩れている。

### (2) トークナイザーが違うと$/tokenは比較できない — Sonnet 5の「費用中立」が認めていること

価格表も同じ問題を抱えている。$/1M tokensという単位は、**トークンの定義が各社で揃っている場合にのみ**比較可能だ。そして揃っていない。

Anthropicの公式記述がこれを明示的に認めている。Sonnet 5の新トークナイザーでは同じ入力が1.0〜1.35倍のトークンにマップされ、だからこそ導入価格をその増加分を相殺する「費用中立」として設定した、というのが公式の説明だ（[Anthropic公式](https://www.anthropic.com/news/claude-sonnet-5)）。

これは裏を返すと、**単価を据え置いてトークナイザーを変えれば、実質値上げが価格表に現れない**ということでもある。Anthropicはそれを避けるために単価側で調整したと明言したわけで、誠実な対応だと思うが、同時に「$/tokenは額面通りには読めない」ことを公式が認めた形になる。

この視点で7月の価格表を見ると、印象が変わる。

| モデル | 入力 | 出力 |
| --- | --- | --- |
| Claude Fable 5 | $10 | $50 |
| GPT-5.6 Sol | $5 | $30 |
| Claude Opus 4.8 | $5 | $25 |
| GPT-5.6 Terra | $2.50 | $15 |
| Claude Sonnet 5 | $2（〜8/31） | $10 |
| Grok 4.5 | $2 | $6 |
| Meta Muse Spark 1.1 | $1.25 | $4.25 |
| GPT-5.6 Luna | $1 | $6 |

数字の上下は読める。だが「Grok 4.5はOpus 4.8の1/4の出力単価」と言うとき、**同じ仕事に何トークン使うかが揃っていなければ、この比率は意味を持たない**。そして第3章で書いた通り、その「何トークン使うか」の公開データが見つからない。

もっと分かりやすい例を挙げる。Claude Sonnet 4.6とClaude Sonnet 5は、**標準価格がどちらも$3/$15で同額**だ（[モデル仕様一覧](https://platform.claude.com/docs/en/about-claude/models/overview)）。価格表の上では「据え置き」に見える。だがSonnet 5は新トークナイザーを使うので、同じ入力が最大1.35倍のトークンになる。**額面が同じでも、同じ仕事のコストは同じではない**。Anthropicが導入価格を費用中立と説明したのは、まさにこのギャップを埋めるためだ。

つまり単価表は入口の情報にすぎず、**「単価 × そのモデルが実際に消費するトークン数」でしか比較できない**。前回のFable 5の記事でも「単価2倍は高いのか」という論点を扱ったが、7月の状況はそれをさらに厄介にしている。トークナイザーが世代ごとに変わるため、**同じベンダー内ですら世代間の$/tokenが比較できない**からだ。

### (3) モデルカードが消えた週 — 検証可能性という土台

7月の新モデル3本の「検証可能性」を整理すると、こうなる。

| モデル | モデルカード | 公式ベンチマーク数値 | ウェイト |
| --- | --- | --- | --- |
| GPT-5.6 Sol/Terra/Luna | 公式docsあり（本文403で直接確認不可） | 公式X投稿・公式ブログ経由で存在 | クローズド |
| Grok 4.5 | **未公開**（Grok 4/4.1はあった） | 公式ページは403、二次経由のみ | クローズド |
| Muse Spark 1.1 | 公式ブログあり | **数値の記載なし** | **クローズド**（Llamaから転換） |

三者三様に、**外部から数字を検証する経路が細っている**。

Grok 4.5のケースが象徴的だ。モデルカードがない状態で「Opus級」「1.5兆パラメータMoE」と主張され、パラメータ数には独立監査が存在しない。そこにCursorBench汚染の話が乗る。繰り返しになるが、汚染を開示したのは提携先のCursor社で、しかも脚注であり、当該ベンチは静かにローンチ表から消えた。

Metaのケースも別方向で厄介だ。公式ブログはある。だがベンチマーク数値が一行もない。**比較の材料を出さないことで、比較されることを回避している**とも読める。実際、第三者測定を見るとDeepSWE 1.1のようなコーディング系では3モデル中最下位に沈んでいるわけで、都合のいい軸だけを選んで出すくらいなら何も出さない、という判断には一定の合理性がある。

そしてこれらは全てクローズドウェイトだ。Metaがオープンウェイト路線から転換したことで、**7月の新モデルには手元で検証できるものが一つもない**。

前回のFable 5の記事で「セーフガードの透明性デザイン」——能力を落とすこと自体ではなく、落としたことを隠すことが信頼を損なう——という話を書いた。同じ構図が測定の側にも来ている。低いスコアを出すことが問題なのではなく、**どう測ったかを出さないことが問題**だ。

### (4) 「コーディング」と「ツール利用」で順位が反転する

第4章の表を、順位の形に書き直してみる。同じ3モデルだ。

| 軸 | 1位 | 2位 | 3位 |
| --- | --- | --- | --- |
| JobBench（業務系ツール利用） | Muse Spark 1.1（54.7） | Opus 4.8（48.4） | GPT-5.5（38.3） |
| DeepSWE 1.1（自律コーディング） | GPT-5.5（67.0） | Opus 4.8（59.0） | Muse Spark 1.1（53.3） |

**きれいな逆順**になっている。1位と3位が入れ替わり、真ん中のOpus 4.8だけが動かない。しかも僅差ではない。JobBenchでのMuse SparkとGPT-5.5の差は16.4ポイント、DeepSWEでの差は13.7ポイントだ。

これは測定誤差ではなく、**軸が違えば順位が変わる**という話だ。「agentic model」を名乗るモデルが、業務ツールの利用で首位を取りながら自律コーディングでは最下位にいる。Muse Sparkのcomputer useが「スクリプトを書くかクリックするかを使い分ける」設計なのも、この方向性と整合している。ツールを叩く能力とコードを書き切る能力は、別々に最適化できるし、別々に劣化する。

ただし注意も要る。**これは1モデル・2ベンチマークの事例であって、業界全体の構造変化を証明するものではない**。「モデルの能力が一本の軸から分岐しつつある」という読みは、現時点では仮説の域を出ない。しかもこれらのスコアは二次情報だ（Metaは公式に数値を出していない）。確度は高くない。

それでも実務的な含意は拾える。**「総合最強のモデル」を1つ選んで全部に使う運用は、少なくともこの3モデルに関しては最適ではない**。自分のワークロードがツール利用寄りかコーディング寄りかで、選ぶべきモデルが変わる。総合1位が自分のタスクでは3位、というのは実際に起きている。

## 8. 結局、どう選べばいいのか

ここまでの話を実務に落とす。

\*\*ベンチマーク表からモデルを選ぶのは、もう諦めた方がいい。\*\*代わりにやることは3つある。

1. **自分のワークロードで測る。** ベンチマークの代わりに、**自分が先週実際にやったタスク**を10〜20本掘り起こして使う。バグ修正のissue、リファクタリング、テスト追加、ログ調査——普段Claude Codeに投げているもので構わない。これを候補モデルに投げ、成功率・所要時間・**実消費トークン数**を記録する。そして単価ではなく「1タスクあたりのコスト」で比較する。Anthropic APIなら`usage.input_tokens` / `usage.output_tokens`で実測、`count_tokens`エンドポイントで事前見積もりができる。トークナイザーが世代で変わる以上、この実測は避けて通れない
2. **effort設定を固定して比較する。** モデル間の差より設定の差の方が大きいことがある以上、`effort`（Claude）やreasoning effort（OpenAI）を揃えないと何も言えない。逆に言えば、**モデルを変える前にeffortを上げ下げする方が効く**ケースが多い
3. **軸を特定する。** コーディングなのか、ツール利用なのか、GUI操作なのか、長時間の自律実行なのか。軸が決まればベンチマークの取捨選択ができる

そのうえで、2026年7月時点の大雑把な当たりを付けるなら、こうなる。

* **重量級のコーディング・長時間自律実行**: Fable 5 / Opus 4.8 / GPT-5.6 Sol。ただし単価は高い
* **日常的なコーディング・エージェント**: Sonnet 5 / GPT-5.6 Terra。この価格帯が7月時点で最も激戦
* **コスト最優先の大量処理**: GPT-5.6 Luna / Grok 4.5 / Muse Spark 1.1
* **ツール利用・業務システム操作が主**: Muse Spark 1.1 を候補に入れる価値がある（コーディングは期待しない）

この「タスクに応じてモデルを振り分ける」という設計を体系化する話は、以前書いた[Claude Codeで複数モデルを使い分ける──Bloom分類によるモデルルーティング設計](https://zenn.dev/yamada_ai_dev/articles/claude-code-bloom-model-routing)で扱っている。モデルの選択肢が増え、しかも軸ごとに順位が入れ替わる状況では、**「どのタスクにどのモデルか」を先に決めておく設計の価値が上がる**。毎回ベンチマーク表を見て悩むより、タスクの性質で機械的に振り分けた方が速いし、ブレない。

## まとめ

1. 2026年7月8〜9日にGPT-5.6三兄弟・Grok 4.5・Muse Spark 1.1が集中したが、「フロンティア3社が同日ローンチ」は不正確（Grokは7/8）
2. Terminal-Bench 2.1の「91.9%」はUltra mode（サブエージェント＋追加計算資源）の数値で、max effortやxhigh effortの他モデルと同じ表に並んでいる。ARC-AGI-3ではeffort設定だけでSolのスコアが0.3%→7.78%に動く
3. トークナイザーが世代・ベンダーで異なるため$/tokenは額面通り比較できない。Anthropicは公式に「1.0〜1.35倍」を認め、導入価格を費用中立として設定した
4. Grok 4.5はモデルカード未公開、Muse Spark 1.1は公式ベンチ数値なし、全モデルがクローズドウェイト。**7月の新モデルには手元で検証できるものが一つもない**
5. 訓練データ汚染を開示したのはxAIではなくCursor社で、脚注での開示だった。影響範囲はCursorBenchに限定される
6. JobBenchとDeepSWE 1.1では、同じ3モデルの順位がきれいに逆順になる。「最強のモデル1つで全部やる」運用は、少なくともこの3モデルに関しては最適ではない
7. 「Grok 4.5はOpus 4.8の25%のトークンで済む」という広く流通した主張は、**引用元を当たっても出所を特定できなかった**
8. Gemini 3.5 Proは7/16時点で未リリース。流通しているスペックは全て報道ベース

新モデルが出るたびにベンチマーク表が更新され、順位が入れ替わる。その光景は派手だが、**表の各セルが何を測った数字なのかを追いかけると、比較の土台そのものが崩れている**ことが分かる。派手な数字に一喜一憂するより、自分のワークロードで10本測る方が、いまは確実に速い。
