# はじめに

Claude Codeは強力なAIコーディングアシスタントですが、何も制約を設けずに使うと「テストを書かずに実装した」「PRテンプレートを無視した」「設計判断を記録しなかった」といった品質のブレが発生します。

人間の開発者にコーディング規約やレビュープロセスがあるように、AIにも「開発フローを守らせる仕組み」が必要です。

さらに言えば、フローを守るだけでは不十分です。人間の開発者がレビュー指摘から学んで成長するように、**AIにもセッション内の経験を学習として蓄積させる仕組み**があれば、プロジェクトが進むほどAIの出力品質も上がっていきます。

本記事では、Claude Code のSkills機能を使って、プロジェクト固有の開発フローをAIに強制し、さらに**自己改善ループ**を組み込む方法を、実際の運用例とともに紹介します。

## superpowersプラグイン

[superpowers](https://github.com/anthropics/claude-code-plugins)はClaude公式のマーケットプレイスから利用できるサードパーティのプラグインであり、汎用的な開発スキルを提供します。

主なスキルには以下があります。

| スキル | 役割 |
| --- | --- |
| `brainstorming` | 実装前の要件整理・設計検討 |
| `writing-plans` | 実装計画の作成 |
| `test-driven-development` | TDDサイクルの強制 |
| `systematic-debugging` | バグの体系的な調査 |
| `verification-before-completion` | 完了宣言前のテスト・ビルド証拠確認 |
| `requesting-code-review` | サブエージェントによるコードレビュー |

これらは汎用的で、どのプロジェクトでも使えます。しかし、プロジェクト固有の「うちのチームではこうする」というルールはカバーできません。

**そこで、プロジェクト固有のスキルを作成し、superpowersのスキルと組み合わせる**のが本記事のアプローチです。

# 開発フロー全体像

## タスク規模に応じたスキルチェーン

私のプロジェクトでは、タスクの規模に応じて使うスキルの組み合わせを変えています。これをオーケストレーターとなるスキル（後述の`dev-flow`）で制御します。

ポイントは、**タスクが大きくなるほど事前の設計工程が増える**ことです。小規模なら即実装、大規模なら設計→計画→実装という段階を踏みます。

全ての規模で共通して**末尾に学習抽出が入る**のがポイントです。タスク規模に関わらず、毎回の実装が次のセッションのインプットになります。

## タスク規模の判断基準

| 規模 | 基準 | 例 |
| --- | --- | --- |
| **小** | 1-2ファイル変更、影響範囲が限定的 | バグ修正、テキスト変更、軽微な設定修正 |
| **中** | 複数ファイル変更、既存パターンの拡張 | 既存コンポーネントへの機能追加、新APIエンドポイント対応 |
| **大** | 新エントリポイント、アーキテクチャ変更 | 新ページ追加、新コンポーネント体系の導入 |

# プロジェクト固有スキルの紹介

ここからは、実際に作成して運用しているプロジェクト固有のスキルを紹介します。

## 1. dev-flow（オーケストレーター）

**役割:** タスクを受け取ったときに「何から始めるか」を判断するスキル。

開発者が「この機能を実装して」と依頼したとき、Claude Codeがまずこのスキルを参照し、タスク規模を判断して適切なスキルチェーンを選択します。

```
---
name: dev-flow
description: Use when starting any new feature, bugfix, or refactoring task
---

# Development Flow

## Flow

タスク受領 → タスク規模の判断 → 規模に応じたスキルチェーンを選択

## Task Size Guide

| 規模 | 基準 | スキルチェーン |
|------|------|--------------|
| 小 | 1-2ファイル変更 | ADR確認 → TDD → 検証 → レビュー → 学習抽出 |
| 中 | 複数ファイル変更 | ADR確認 → brainstorming → ADR作成 → TDD → 検証 → レビュー → 学習抽出 |
| 大 | 新エントリポイント | ADR確認 → brainstorming → ADR作成 → 計画作成 → 実装 → 検証 → レビュー → 学習抽出 |
```

### ポイント: ADR（Architecture Decision Records）の組み込み

このプロジェクトでは、設計判断をADRとして `docs/adr/` に記録しています。dev-flowスキルに以下のルールを組み込むことで、AIにもADRの確認・作成を強制しています。

* **実装前に必ず既存ADRを確認する** — 過去の設計判断に反する実装を防ぐ
* **中規模・大規模タスクではADRを作成する** — 設計判断を記録として残す
* **小規模タスクでも新たな設計判断がある場合はADRを作成する**

ADRが蓄積されると、Claude Codeが実装前にそれを参照するため、プロジェクト内の一貫性が自然に保たれます。

## 2. code-review（Adversaryモード付きセルフレビュー + 学習抽出）

**役割:** 実装完了後、PR作成前のゲートとして機能するスキル。さらに、レビュー結果から汎用的な学びを抽出してプロジェクトのナレッジとして蓄積する。

このスキルは検証と学習抽出を一括で行います。

### 検証1: テスト・リント・ビルドの実行（証拠が必要）

```
## Verification（証拠が必要）

| 検証項目 | コマンド | 合格基準 |
|---------|---------|---------|
| テスト | npm test | 0 failures |
| リント | npm run lint | 0 errors |
| ビルド | npm run build | exit 0 |

全て合格の証拠を確認してから次のステップへ進む。
1つでも失敗したら修正してから再実行。
```

superpowersの `verification-before-completion` スキルと連携し、**コマンドの出力を実際に確認する**ことを強制します。「通っているはず」での通過を防ぎます。

### 検証2: プロジェクト固有チェックリスト

セキュリティ、命名規則、テスト配置など、プロジェクト固有のルールをチェックリストとして定義しています。

```
### セキュリティ
- [ ] サーバーサイドログにユーザー情報を出力していない
- [ ] Cookieにsecureフラグが付いている

### コンポーネント・命名
- [ ] コンポーネントはプロジェクトのプレフィックスを使っている

### テスト
- [ ] テストファイルはソースと同じディレクトリに配置されている
```

### 検証3: Adversaryモードのコードレビュー

ここが最も特徴的な部分です。superpowersの `requesting-code-review` スキルは、**別コンテキスト（会話履歴ゼロ）でサブエージェントをdispatch**してコードレビューを実行します。

これだけでも実装者のバイアスからは切り離されますが、さらに**Adversary（敵対的）姿勢**をレビュアーに指示しています。

```
## コード品質レビュー（Adversaryモード）

レビュアーには以下の姿勢を指示すること:

> あなたの役割は問題を見つけることです。承認ではなく否定から入ってください。
> 「良さそうに見える」で終わらせず、壊れるケース・見落とされたエッジケース・
> 暗黙の前提を積極的に探してください。
> 問題が見つからなかった場合のみ、その旨を報告してください。
```

**なぜAdversaryモードが有効なのか？**

通常のレビューでは、AIは「問題ないですね」と承認しがちです。これは人間のレビューでも起きる同調バイアスと同じ現象です。サブエージェントは実装時の会話コンテキストを持たないため構造的にバイアスは減りますが、「問題を見つけることが仕事」と明示することでさらに厳しいレビューが得られます。

### 検証4: 学習の抽出（session-learnings）

ここが本記事で新たに追加した部分です。検証1〜3を通過した後、**このセッションで得られた学びをプロジェクトのナレッジとして記録**します。

```
## Session Learnings（学習抽出）

### 抽出対象
- Adversaryレビューで指摘され、修正したパターン
- 実装中に発見した既存コードの暗黙の前提
- テスト・ビルドで発生したエラーとその原因
- 既存ADRに記載されていない設計上の判断

### 判断基準
以下に該当する場合のみ記録する:
- 同じパターンが今後も発生しうる（汎用性がある）
- 既存のdocs/やADRに記載がない（重複しない）
- プロジェクト固有の文脈に依存する知識である

### 出力先
- docs/learnings/ に markdown として記録
- ファイル名: YYYY-MM-DD-<簡潔なタイトル>.md
- 既存ADRへの追記が必要な場合は提案のみ（人間が承認）
```

**なぜcode-review内に組み込むのか？**

学習抽出を別スキルとして独立させることもできますが、code-review内に組み込むことにはいくつかのメリットがあります。

1. **Adversaryレビューの指摘内容がコンテキストに残っている** — 別スキルにすると、レビュー結果を改めて読み直す必要がある
2. **スキルチェーンが増えない** — dev-flowのオーケストレーションがシンプルなまま維持できる
3. **実装→レビュー→学習が1つの流れとして自然** — 人間の開発者が「レビューで指摘されて覚える」のと同じ構造

## 3. create-pr（PR作成）

**役割:** PRテンプレートに沿ったPR本文の自動生成とPR作成。

```
---
name: create-pr
description: Use when ready to create a pull request
---

# PR作成

## Step 1: ブランチ情報の収集
## Step 2: PR本文の生成（テンプレート準拠）
## Step 3: gh pr create で作成
```

このスキルには以下のルールを埋め込んでいます。

* **PRテンプレートの各セクションを必ず埋める** — チケットURL、変更内容、背景・目的、確認方法
* **ブランチ構造を考慮する** — `main` ではなくフェーズブランチへのPRを作成する（ブランチ構造が `main → feature/ph.xx → feature/xxx` の場合）
* **commitメッセージやブランチ名からチケット情報を抽出する**

## 4. pr-review（他人のPRレビュー）

**役割:** 他人のPRをレビュアー視点で評価するスキル。

code-reviewが「自分の実装のセルフチェック」であるのに対し、pr-reviewは「他人のPRをマージしていいか判断する」ためのスキルです。

```
## レビュー結果のまとめ

| 重要度 | 基準 | 対応 |
|--------|------|------|
| Blocking | セキュリティ違反・重大な方針逸脱・CIが通っていない | マージ不可 |
| Important | コード品質・設計上の問題 | マージ前に修正推奨 |
| Minor | スタイル・軽微な改善提案 | 任意対応 |
| LGTM | 問題なし | マージ可 |
```

`gh pr view` や `gh pr diff` を使ってPR情報を取得し、code-reviewと同じプロジェクト固有チェックリスト + Adversaryモードのレビューを実行します。

# 自己改善ループの設計思想

## なぜ「守らせる」だけでは足りないのか

ここまでのスキル設計は、**人間が定義したフローをAIに守らせる（トップダウン）** アプローチです。これだけでも品質の安定化には十分効果がありますが、1つ足りないものがあります。

**AIが経験から学んで、次のセッションで同じミスをしないこと。**

Claude Codeはセッションが切れるとコンテキストを失います。前回のセッションでAdversaryレビューに「この型ガードが甘い」と指摘されて修正しても、次のセッションでは同じミスを繰り返す可能性があります。

## 学びの蓄積先としての docs/learnings/

この問題に対する解決策として、CLAUDE.mdにAIが直接書き込む方法もありますが、採用しませんでした。理由は以下です。

* **CLAUDE.mdはプロジェクトの設定ファイル**であり、AIが自由に書き換えるべきものではない
* **学びの内容は人間にとっても価値がある**はずで、AIだけが読むファイルに閉じ込めるのはもったいない
* **変更の因果関係が追えなくなる** — 何がきっかけでルールが追加されたのかわからなくなる

代わりに、`docs/learnings/` にmarkdownとして記録する方式を採用しています。これは実装の副産物として自然に生成され、次のセッションのClaude Codeが既存ドキュメントとして読める。人間にとっても有用なナレッジベースになります。

## フェーズ間の知識継承

移植プロジェクトのように長期にわたる開発では、1セッションで全てが完了することはありません。フェーズごとにセッションを切りつつ、学びは `docs/learnings/` やADRに残して次のフェーズに引き継ぎます。

1つ目のフェーズで発見した「暗黙の前提」が、2つ目以降のフェーズで自然に活きる構造です。

## なぜシングルスレッドで回すのか

cmuxやomcのようなオーケストレーションツールで複数セッションを並列に動かすアプローチもありますが、自己改善ループとの相性は良くありません。

並列セッションでは、各子セッションが仕様の一部しか見えない状態で動きます。「このコンポーネントはあっちのページのこの挙動と暗黙的に連動してた」のような依存関係を見落とすリスクが高い。

さらに、並列セッションの学びは親に「結果」としてしか返りません。途中経過のコンテキスト — Adversaryレビューで指摘されて修正して「あ、このパターンか」と気づく過程 — は捨てられてしまいます。

自己改善ループは本質的に**前の結果が次の入力になる逐次処理**です。シングルスレッドで `セッション → 学習抽出 → 次のセッション` のサイクルを回す方が、変更の因果関係が追えるし、学びの質も高くなります。

# superpowersスキルとの連携

プロジェクト固有スキルは、superpowersの汎用スキルと組み合わせることで真価を発揮します。

## 連携のイメージ

**プロジェクト固有スキルがオーケストレーター**として、superpowersのスキルを「いつ・どの順序で使うか」を制御します。

superpowersのスキルをラップする形で、プロジェクト固有の文脈を追加しています。たとえばcode-reviewスキルは、superpowersの `requesting-code-review` を呼びつつ、その結果を学習抽出のインプットとして活用します。

```
code-review スキルの構造:

  Step 1-3: 検証（superpowers:verification-before-completion をラップ）
  Step 4:   Adversaryレビュー（superpowers:requesting-code-review をラップ）
  Step 5:   学習抽出（Step 4 の結果をインプットとして使用）← NEW
```

## 実際の開発セッション例

ここでは、小規模タスク（GA計測のバグ修正）を例に、実際のスキル連携を示します。

### 1. タスク受領 → dev-flow が規模を判断

```
開発者: 「GAで /ticketList と /ticketList/ が別ページとして計測されている。
         /ticketList に統一したい」

Claude Code: (dev-flowスキルに従い)
  → タスク規模: 小（1-3ファイルの修正）
  → ADR確認 → 関連ADRなし
  → docs/learnings/ 確認 → 関連する過去の学びなし
  → 実装へ
```

### 2. 実装

```
Claude Code: gtag configファイル（dev/stag/prod）に page_location の
             正規化処理を追加
```

### 3. code-review → 検証 + Adversaryレビュー + 学習抽出

```
開発者: /code-review

Claude Code:
  Step 1: npm test → 171 passed ✓
          npm run lint → 変更ファイルにエラーなし ✓
          npm run build → 成功 ✓
  Step 2: プロジェクト方針チェック → OK
  Step 3: Adversaryモードのサブエージェントをdispatch
          → 「正規表現のエッジケースを確認...問題なし」
  Step 4: 学習抽出
          → 汎用パターンあり:「GA計測の設定は環境別ファイルに分散しているため、
            変更時は全環境ファイルの同期確認が必要」
          → docs/learnings/2026-04-10-ga-config-sync.md に記録
```

### 4. create-pr → PR作成

```
開発者: /create-pr

Claude Code:
  → ブランチ情報を収集
  → PRテンプレートに沿って本文生成
  → gh pr create で作成
```

### 5. 次のセッションで学びが活きる

```
（別のセッション）
開発者: 「GAのイベントトラッキングを追加して」

Claude Code: (dev-flowスキルに従い)
  → docs/learnings/ 確認
  → 「GA計測の設定は環境別ファイルに分散」の学びを参照
  → 最初から全環境ファイルを対象に実装
```

## superpowersのスキルを参照する

プロジェクト固有スキルの中で、superpowersのスキルを `REQUIRED SUB-SKILL` として参照できます。

```
**REQUIRED SUB-SKILL:** `superpowers:verification-before-completion`
```

これにより、汎用的な検証ロジックはsuperpowersに任せつつ、プロジェクト固有のルールだけを自分のスキルに書けます。

# まとめ

Claude Code Skillsでプロジェクト固有の開発フローを定義し、自己改善ループを組み込むことで、以下が実現できます。

* **品質のブレを防ぐ** — テスト・リント・ビルドの検証を毎回強制
* **チームのプロセスをAIに守らせる** — PRテンプレート、ADR、命名規則などを自動チェック
* **同調バイアスを排除する** — Adversaryモードで厳しいセルフレビュー
* **タスク規模に応じた柔軟性** — 小規模は軽く、大規模は設計から丁寧に
* **セッションをまたいだ学習** — レビュー指摘から汎用パターンを抽出し、次のセッションで活用

最初の4つは「AIに開発フローを守らせる」ためのもの。最後の1つが「AIが自分で強くなる」ためのものです。守らせるだけでなく、育てる。この両輪があることで、プロジェクトが進むほどAIの出力品質も上がっていきます。
