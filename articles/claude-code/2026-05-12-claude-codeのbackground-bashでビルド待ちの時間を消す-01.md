こんにちは、エリスです。

Claude Codeに作業を頼んでいて、地味にストレスだったのが **「重いコマンドが終わるのをただ待つ時間」** でした。`npm run build` が3分、`cargo test` が5分、Playwrightのスイートが8分。その間Claudeは止まっていて、次の調査も編集も進まない。タイムアウトを伸ばしても本質的には変わらないやつです。

最近は **`Bash`ツールの `run_in_background`** を使う運用に切り替えていて、これでだいぶ快適になりました。今日はその使い方と、わたしが実際にハマった注意点をまとめておきます。

## `run_in_background` は何をやってくれるのか

普通にBashツールを呼ぶと、コマンドが終わるまで応答が返ってこない。タイムアウトは最長10分まで伸ばせるけど、その間Claudeは他のことができない。

`run_in_background: true` を渡すと、コマンドは別プロセスで起動して、Bashツールは即座に **シェルID** を返してくれます。Claude側は他の調査・編集を続けながら、終わったタイミングで通知を受け取る、という流れになる。

```
Bash({
  command: "pnpm build",
  run_in_background: true,
  description: "Build the app in the background"
})
```

これだけ。後はClaudeが「あ、終わったな」と気づいたら結果を読みに行きます。

## 並列で動かす実例

実際に効くのは **「重い処理 + 別の作業」が並行できるとき** です。わたしのよくあるパターンはこれ:

* バックグラウンドで `pnpm build` を走らせつつ、メインで別ファイルを編集
* バックグラウンドで `cargo test` を走らせつつ、ドキュメントを直す
* バックグラウンドでPlaywrightを走らせつつ、別のテストケースを読む

ポイントは **同じ会話の中で、Claudeが両方の文脈を持ったまま並走できる** こと。タブを切り替える必要がないし、終わったタイミングは通知で来るのでポーリングも要らない。

## `Monitor` でストリーミング監視

`run_in_background` だけだと「終わったかどうか」しかわからないので、進捗を見たいときは `Monitor` ツールを併用します。

例えば長いビルドの出力を逐次見たいなら、シェルIDを `Monitor` に渡してログを追えばいい。条件を満たすまでループで監視するパターンも組めます。

```
# 何かが起きたら通知して欲しい
Monitor({
  shellId: "...",
  pattern: "compiled successfully|Error:",
})
```

これで「成功か失敗か決まる瞬間」だけ通知してくれる。だらだら全行流すよりトークンが節約できます。

## ハマったポイント1: シェルIDを覚えておく

最初にやらかしたのがこれ。`run_in_background` でシェルを起動したのに、シェルIDをメモせずに別の作業に行ってしまって、結果を取りに戻れなくなった。

`BashOutput` でシェルIDを指定して読み出すんですが、IDを失くすと **「動いていることはわかるが結果が取れない」** という不毛な状態に陥ります。

対策: バックグラウンド起動した直後に、用途と一緒に1行メモする。「ビルド: shell\_id=xxx」みたいに。Claude自身も会話文脈に書き残しておくと、後で参照できる。

## ハマったポイント2: 「終わった」と「成功した」は違う

`run_in_background` のシェルが終了したからといって、コマンドが成功したとは限らない。

これも一回やらかしました。バックグラウンドで `pnpm test` を流して、終了通知が来たので「OK次行こう」と進んだら、実はテストが赤かった。exit codeを見ていなかったのが原因です。

対策: 終了通知が来たら **必ず `BashOutput` で結果を読んで、exit codeか結果行を確認する** ところまで1セット。並列化の恩恵を取りに行くと、つい確認をスキップしがちなので注意。

## ハマったポイント3: 何でもバックグラウンドに送らない

慣れてくると全部 `run_in_background` に投げたくなるんですが、これは逆効果でした。

* `ls` とか `git status` みたいな1秒で終わるコマンドをバックグラウンドにすると、ハンドリングのオーバーヘッドのほうが大きい
* `npm install` をバックグラウンド化したら、後続の `npm test` がインストール完了前に走って死ぬ、みたいな依存関係を踏みやすい

わたしの線引きは **「30秒以上かかる + 他の作業と独立している」** の2つを満たすときだけバックグラウンド化、です。

## 実際の使い分けフロー

整理するとこんな感じです。

| ケース | 使う道具 |
| --- | --- |
| 3秒で終わる | 普通のBash |
| 30秒〜10分 + 他作業なし | 普通のBash（タイムアウトだけ伸ばす） |
| 30秒〜10分 + 他作業あり | `run_in_background: true` |
| 進捗を逐次見たい | `run_in_background` + `Monitor` |
| 終わるまで他を完全停止したい | 普通のBash |

「とにかく速くする」ではなく **「並行できるものだけ並行する」** という考え方です。

## まとめ

* `run_in_background: true` で長時間コマンドを別プロセスに逃がす
* シェルIDは必ずメモる。失くすと取り戻せない
* 終了 ≠ 成功。`BashOutput` でexit codeまで確認するワンセット
* 全部バックグラウンド化はNG。30秒以上 + 独立しているときだけ
* 進捗監視は `Monitor` でパターンを指定するとトークン節約

ビルド・テスト待ちで止まっていた時間が、ほぼ全部「次の作業」に置き換わったのが大きいです。1日のフィードバックループが目に見えて速くなる。

それじゃあ、また。
