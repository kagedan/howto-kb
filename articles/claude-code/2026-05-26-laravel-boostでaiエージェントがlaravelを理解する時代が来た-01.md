## はじめに

東京で開催された **Laravel Live Japan 2026** に参加してきました。Laravelの生みの親であるTaylor Otwell氏が登壇し、Laravel Boostについて語っていたのが印象的でした。

Claude CodeやCursorなどのAIコーディングエージェントを使っていて、こんな経験はないでしょうか。

「Laravelでコード書いて」と頼んだのに、**既存構造やベストプラクティスを無視した汎用的なPHPコード**が返ってくる。結局、手動でスキーマを貼り付けたり、CLAUDE.mdにプロジェクト情報を書き連ねたり...。

この「AIにプロジェクトの文脈を伝える」問題を、Laravel公式チームが正面から解決しにきました。それが **Laravel Boost** です。

## Laravel Boost とは

Laravel Boostは、Laravel公式チームが開発した**AIコーディングエージェント向けのMCPサーバー**です。2025年7月のLaracon US 2025でプレビューされ、同年8月にパブリックベータが公開されました。2026年1月にはSkillsシステムを搭載したv2.0もリリースされています。

Composerパッケージとしてプロジェクトにインストールすることで、AIエージェントにLaravelプロジェクトの深い文脈を提供します。

```
composer require laravel/boost --dev
```

完全に**無料・OSS（MITライセンス）** で、現在パブリックベータとして公開されています。

## 3つの柱

Boostは以下の3つの柱で構成されています。

### 1. MCP Server（15以上のツール）

AIエージェントに対して、プロジェクト固有の情報を取得するツールを公開します。

| ツール | できること |
| --- | --- |
| DBスキーマ取得 | テーブル構造・リレーションを把握 |
| Tinker実行 | Laravelのコンテキストでコードを実行・検証 |
| ルート一覧 | 定義済みのルートを全て参照 |
| 設定値取得 | `config/` + `.env` の値をLaravelの仕組み経由で読む |
| Artisanコマンド一覧 | 利用可能なコマンドを把握 |
| ログ・エラー取得 | 直近のエラーやログを参照 |
| ブラウザログ | フロントエンド側のログも取得可能 |

これにより、AIエージェントは「そのプロジェクト固有の状態」を自分で調べられるようになります。

### 2. Documentation API

17,000件以上のバージョン別Laravelドキュメントをエージェントが検索できます。AIの学習データが古くても、**最新のLaravel APIを正確に参照できる**のがポイントです。

### 3. AI Guidelines

Laravel / Livewire / Inertia / Pest / Tailwind CSS / Flux UIなど、Laravelエコシステムのベストプラクティスがエージェントに注入されます。「Laravelらしいコード」を書くための規約を、AIが最初から知っている状態になります。

## 現時点の制限

パブリックベータなので、いくつか注意点があります。

| 項目 | 内容 |
| --- | --- |
| ベータ版 | 挙動の変更や荒削りな部分がある |
| ドキュメント粒度 | メジャーバージョン単位のみ対応。マイナー/パッチの差分は未対応 |
| 対応バージョン | Laravel 10 / 11 / 12 + PHP 8.1以上 |

ちなみに、Laravel **Cloud**（有料のホスティングサービス）とは全くの別物です。BoostはあくまでローカルのComposerパッケージなので、インフラ費用もかかりません。

## MCPサーバーなのにComposerインストール？

ここで素朴な疑問が湧きます。

MCPサーバーといえばGitHub MCP ServerやSlack MCPのように、**外部のスタンドアロンなサービス**をイメージする方が多いのではないでしょうか。なぜBoostはComposerでプロジェクト内にインストールする必要があるのでしょうか。

答えは、**Boostが提供するツールの性質**にあります。

```
外部MCP（例: GitHub MCP Server）
  GitHub API を叩くだけ → プロジェクト外で動ける

Boost（プロジェクト内MCP）
  Laravelアプリの内部構造を読む → プロジェクト内でないと不可能
```

例えば、DBスキーマを取得するには `.env` のDB接続情報とマイグレーション定義が必要です。Tinkerを動かすにはLaravelのbootstrapを経由しないといけません。ルート一覧を出すには `routes/` の定義とServiceProviderの登録情報が揃って初めて可能になります。

つまり、**プロジェクト固有の情報を取得するには、そのプロジェクトの内側にいるしかない**。Composerでインストールするのは、Laravelのオートローダーやサービスコンテナに乗る必要があるからです。

MCPサーバーは「外部サービス」である必要はなく、**プロジェクトのプロセスとしてローカルで起動するもの**でもよい。Boostはこの設計を採用することで、プロジェクトの深い文脈をAIに渡すことに成功しています。

## 活用アイデア：`/lint-laravel` スキル

Boostの3つの柱を組み合わせると、面白いclaudeスキルが作れそうだと感じました。例えば、**Laravelの規約に沿っているかをチェックする `/lint-laravel` スキル**です。

```
/lint-laravel 実行
  │
  ├─ AI Guidelines（ローカル同梱）
  │    → Laravel / Livewire / Pest 等のベストプラクティスを判断基準として参照
  │
  ├─ Documentation API（17,000件）
  │    → 「この書き方は非推奨になっていないか？」を最新ドキュメントで裏取り
  │
  └─ 対象コードを読み込み
       → 上記2つに照らして逸脱箇所を指摘
```

例えばこんな指摘が期待できます：

* `env()` を直接呼んでいるが、`config()` ヘルパー経由にすべき
* FormRequestを使わずControllerでバリデーションしている
* `DB::raw()` を使っているがEloquentで書ける箇所がある

PHPStanやLarastanとの違いは、**静的解析では拾えない設計レベルの規約**をチェックできる点です。Guidelinesはパッケージに同梱されているのでオフラインでも動き、Documentation APIが最新バージョンの仕様変更もカバーしてくれます。

v2.0のSkillsシステムを使えば、こうしたカスタムスキルをBoost内に定義してチーム全体で共有することも可能です。

## 今後の展望

個人的に注目しているのは、**この流れがLaravelに限らず広がっていくだろう**という点です。

AIコーディングエージェントが普及するにつれ、「フレームワーク側がAIをサポートする仕組み」を持つことは自然な進化だと感じます。実際、Next.jsではすでにVercel公式の [next-devtools-mcp](https://github.com/vercel/next-devtools-mcp) が提供されており、開発サーバーの状態をAIエージェントに公開する仕組みが整っています。この流れはさらに広がっていくでしょう。

フレームワークの価値が「開発者の生産性を上げること」にあるなら、その「開発者」にAIエージェントが含まれる時代が来ているのだと思います。LaravelがBoostでその先陣を切ったのは、とても興味深い動きです。

## まとめ

Laravel Boostは、AIエージェントにLaravelプロジェクトの文脈を与えるためのMCPサーバーです。

* 無料・OSS・Composer一発で導入可能
* DBスキーマ、ルート、設定値などをAIが自力で取得できる
* 17,000件以上のバージョン別ドキュメントとベストプラクティスを搭載

AI × フレームワークの新しい関係性に興味がある方は、ぜひ試してみてください。
