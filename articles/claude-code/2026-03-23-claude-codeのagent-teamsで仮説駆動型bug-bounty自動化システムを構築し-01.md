## 免責事項

本記事の内容は個人の研究・検証に基づくものであり、所属組織を代表するものではありません。  
本記事で紹介する手法は、**合法的なBug Bountyプログラムの範囲内、またはやられ環境（DVWA、HackTheBox等）でのみ**使用してください。許可のないシステムへのテストは不正アクセス禁止法に抵触する可能性があります。

---

## はじめに

Bug Bountyの世界では、**多くの人が同じツールチェーン**（subfinder → httpx → nuclei）を回しています。その結果、同じ脆弱性を同時に複数のハンターが発見し、レポートが重複し、報奨金は先着1名にしか支払われません。

ここで疑問が生まれます。

**「AIが人間の専門家のようにアプリの仕組みを理解し、攻撃仮説を立ててからテストしたら、他のハンターとは全く異なる脆弱性を発見できるのではないか？」**

2026年2月にリリースされたClaude CodeのAgent Teams機能は、複数のAIエージェントが並列で協調作業できる機能です。これを使い、「仮説駆動型」のBug Bountyシステムを構築しました。

本システムの有効性は、DVWA（Damn Vulnerable Web Application）やHackTheBoxなどの**やられ環境で検証済み**です。仮説駆動アプローチにより、従来のスキャナーでは検出できないタイプの脆弱性パターンを効率的に特定できることを確認しています。

本記事では、設計思想からClaude Codeでの実装方法までを報告します。

## 核心: bb-hypothesis（仮説推論エンジン）の仕組み

このスキルがシステム全体の差別化です。従来のツールとは根本的に異なるアプローチを取ります。

### 3段階チャンク処理

大量のデータを一度にAIに渡すとコンテキストが溢れるため、3段階に分けて処理します。

```
┌─────────────────────────────────────────────────────────┐
│              Stage 1: グローバルサマリー                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  入力: alive_hosts.json + tech_stack.json のみ (~15KB)  │
│                                                         │
│  Claude への指示:                                        │
│  「全ホストの概要を見て、アプリの種別を判定し、            │
│    どのホストが最も攻撃価値が高いか順位付けせよ」          │
│                                                         │
│  出力: app_summary.json                                  │
│  ├─ アプリ種別: SaaS / EC / フィンテック 等              │
│  ├─ ホスト優先順位リスト                                 │
│  └─ 初期観察メモ                                        │
│                                                         │
├─────────────────────────────────────────────────────────┤
│              Stage 2: ホスト別仮説生成                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ホスト優先順位の上位から1つずつ処理                      │
│  ※ 絶対に複数ホストを同時にコンテキストに入れない         │
│                                                         │
│  入力 (1ホスト分のみ):                                   │
│  ├─ endpoints.json       (~5-10KB)                      │
│  ├─ parameters.json      (~3-5KB)                       │
│  ├─ js_analysis.json     (~10-20KB)                     │
│  └─ app_summary.json     (~2KB)                         │
│                                                         │
│  → 合計 ~40-90KB → コンテキスト内に余裕で収まる          │
│                                                         │
├─────────────────────────────────────────────────────────┤
│              Stage 3: クロスホスト攻撃チェーン              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  入力: 全ホスト仮説のサマリーのみ (~5-10KB)              │
│                                                         │
│  「異なるホストの仮説を横断的に見て、                     │
│    組み合わせ可能な攻撃チェーンを特定せよ」               │
│                                                         │
│  例: 単体ではLowの脆弱性を組み合わせてCriticalに昇格     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 仮説カテゴリ（12種類）

Claudeは各エンドポイントに対して、以下の12の観点から仮説を生成します。重複リスクが低い（＝他のハンターが自動ツールで見つけにくい）カテゴリを優先します。

```
仮説カテゴリ（優先順位順）:
┌────────────────────────────────────────────────────────┐
│ 重複リスク低（＝高価値）                                │
├────────────────────────────────────────────────────────┤
│  a) アクセス制御の欠陥                                  │
│  b) ORM/検索機能経由のデータ漏洩          ← 2025年新   │
│  c) AI/LLM統合機能のセキュリティ          ← 2025年新   │
│  d) ビジネスロジックの欠陥                              │
│  e) 認証プロトコルの実装不備                            │
│  f) パーサー差異 / 文字エンコーディング    ← 2025年新   │
├────────────────────────────────────────────────────────┤
│ 重複リスク中                                            │
├────────────────────────────────────────────────────────┤
│  g) サーバーサイドリクエストフォージェリ                 │
│  h) テンプレートインジェクション           ← 2025年新   │
│  i) サイドチャネル情報漏洩                 ← 2025年新   │
│  j) フレームワーク固有の脆弱性                          │
├────────────────────────────────────────────────────────┤
│ 重複リスク高（後回し）                                  │
├────────────────────────────────────────────────────────┤
│  k) クロスサイトスクリプティング / SQLインジェクション   │
│  l) 既知CVE                                             │
└────────────────────────────────────────────────────────┘
```

### 仮説の優先度計算

各仮説には、過去の精度メトリクスを加味したスコアを付与します。

```
priority_score =
    confidence          × 0.20   (この仮説が正しい確率)
  + severity_estimate   × 0.25   (成功した場合の影響度)
  + bounty_estimate     × 0.15   (過去の報奨金相場)
  + (1 - duplication)   × 0.20   (他ハンターとの重複リスク)
  + accuracy_adjustment × 0.20   (過去の同タイプ仮説の的中率)
```

`accuracy_adjustment`は、システム運用を重ねるごとに蓄積されるメトリクスです。仮説のタイプごとに「予測した確信度」と「実際の的中率」を比較し、予測精度を継続的に改善します。例えば、あるタイプの仮説が繰り返し外れている場合、そのタイプの確信度を自動的に下方修正し、代わりに的中率の高いタイプにリソースを集中する判断を行います。

## 安全基盤: Hooksによるガードレール

Bug Bountyでは「スコープ外のシステムにアクセスしない」ことが絶対です。Claude CodeのHooks機能を使い、**全てのBashコマンド実行前に自動チェック**を行います。

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "./hooks/scope-guard.sh"
        }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "./hooks/checkpoint-save.sh"
        }]
      }
    ]
  }
}
```

`scope-guard.sh`は、Bashコマンド内のURL・ドメインを抽出し、事前定義したスコープファイルと照合します。スコープ外のドメインが含まれていた場合、exit code 2でコマンドの実行をブロックし、Claudeにブロック理由を通知します。これにより、AIエージェントがどんなコマンドを生成しても、スコープ外へのアクセスは物理的に不可能です。

同様に、`checkpoint-save.sh`は全てのツール実行後に実行状態を自動保存し、セッション断絶後も前回の続きから再開可能にします。`rate-limiter.sh`はエラー率に応じてリクエスト速度を動的に調整し、対象サーバーに過負荷をかけない設計です。

## Agent Teamsによる並列実行

Claude Code 2026年2月リリースのAgent Teams機能を使い、フルパイプラインを並列実行します。

```
ユーザー: /hunt <target>

Claude Code (Lead):
│
├── [Agent Team: Phase 1 — 3エージェント並列]
│   ├── recon-agent
│   │   └── サブドメイン列挙 + ポートスキャン + テック検出
│   ├── discovery-agent
│   │   └── クローリング + JS解析 + API仕様取得
│   └── OSINT agent
│       └── 公開レポート・報奨金傾向の調査
│
├── [Phase 2: 仮説構築]
│   └── hypothesis-agent (Opus)
│       └── Stage 1 → 2 → 3 → 仮説生成
│
├── [Phase 3: ユーザーに提示]
│   └── 「仮説N件と攻撃チェーンM件を発見。テストしますか？」
│
├── [Agent Team: Phase 4 — テスト並列]
│   ├── 各種専門テスター (仮説タイプに応じて振り分け)
│   └── 結果をhypothesis-agentにフィードバック
│
└── [結果]
    ├── confirmed → finding記録 + 精度メトリクス更新
    ├── rejected → 防御分析 → バイパス仮説再生成
    └── /report → レポート自動生成
```

tmux環境でClaude Codeを起動すると、Agent Teamsは自動的に画面をペイン分割し、各テammateの作業を視覚的に監視できます。

## 法的注意事項

Bug Bountyに関わる活動を行う際は、以下の点に十分注意してください。

### 不正アクセス禁止法

日本の不正アクセス禁止法は、**目的や意図ではなく「行為」そのもの**で違法性が判断されます。Bug Bountyプログラムで許可されたスコープ内であっても、プログラムの規約を逸脱した行為は法的リスクを伴います。

* テスト対象は必ず**プログラムのスコープ内**に限定すること
* **自分で作成したテストアカウント**のみを使用すること
* 発見した脆弱性は**プログラム経由でのみ**報告すること
* **脆弱性の詳細は、対象企業の修正・許可があるまで公開しない**こと

### 本システムの安全設計

本システムでは`scope-guard.sh`（PreToolUse Hook）により、スコープ外へのアクセスを自動ブロックしています。ただし、技術的な安全装置は法的な免責を保証するものではありません。各自の責任において、適用法令とプログラムの規約を遵守してください。

## 学びと今後の展望

### うまくいったこと

1. **仮説駆動アプローチは有効**。特にアクセス制御とデータ漏洩に関する仮説精度が高い
2. **Agent Teamsの並列実行**により、偵察+発見+OSINTが同時に完了する
3. **精度メトリクスのフィードバック**で、回を重ねるごとに仮説精度が向上

### 課題として残っていること

1. **認証セットアップの手動作業**は避けられない（CAPTCHA、メール認証等）
2. **タイミングに依存するテスト**はスクリプトの信頼性が低い
3. **3セッション以上のOAuthトークン競合**が週2-3回発生

### 今後の計画

* モバイルアプリ解析モジュールの追加
* ブラウザ自動化（Playwright）連携
* パターンファイルの継続的な品質改善

## まとめ

本記事では、Claude CodeのAgent Teams機能を活用した、仮説駆動型Bug Bounty自動化システムの設計・実装方法を報告しました。

* **AIの強みは「スキャン」ではなく「推論」にある**。アプリの仕組みを理解して仮説を立てる能力は、既存のどのツールにも真似できない
* **Claude Codeは6つの拡張ポイント**（Skills, Subagents, Agent Teams, Hooks, Commands, Headless mode）を持つ本格的なエージェントプラットフォーム。全部使うことで真価を発揮する
* **やられ環境での検証**により、仮説駆動アプローチが従来のスキャナーでは検出できないタイプの脆弱性パターンを特定できることを確認した

Bug Bountyに限らず、「AIが仮説を立て、ツールが検証する」というパラダイムは、セキュリティテスト全般に適用可能な考え方です。
