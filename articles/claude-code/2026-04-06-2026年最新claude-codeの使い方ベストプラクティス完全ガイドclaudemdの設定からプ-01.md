## まず知っておきたい：Claude Codeの「拡張機能」の仕組み

Tipsの解説に入る前に、「Claude Codeにはこういう機能がある」という全体像だけ押さえておきます。これを知らないとTipsの意味がわからないので、ここだけお付き合いください。

Claude Codeは、ターミナル（CLI）上で動くAIコーディングアシスタントです。ChatGPTのようなブラウザ画面ではなく、エンジニアが使う黒い画面の中で動きます。普通のAIチャットとの最大の違いは、あなたのPC上のファイルを直接読み書きできること。コードを提案するだけでなく、ファイルの作成・編集・テスト実行まで全部自動でやってくれます。

そして、Claude Codeには以下の6つの強力な拡張機能があります。

1.CLAUDE.md（メモリ）：プロジェクトのルールや規約を書いておく設定ファイルです。Claude Codeはセッション開始時にこれを読み込んでくれるので、毎回「うちのプロジェクトは…」と説明し直す必要がなくなります。いわば「AIへの申し送り書」です。

2.Commands（コマンド）：よく使う指示をテンプレート化する仕組みです。.claude/commands/review.mdというファイルを作っておけば、/reviewと打つだけでコードレビューが走ります。

3.Subagents（サブエージェント）：メインのClaude以外に「専門AI」を立ち上げる仕組みです。「設計担当」「テスト担当」「レビュー担当」をそれぞれ独立した環境で並列に動かせます。

4.Skills（スキル）：再利用できる知識パックです。Commandsが「やることリスト」なら、Skillsは「教科書」。Claudeが状況に応じて勝手に参照してくれます。

5.Hooks（フック）：特定のイベント（ファイル変更やコミットなど）が起きたとき裏側で走る自動処理です。

6.Plugins（プラグイン）：上記をまとめてパッケージ化し、チームやコミュニティで共有する仕組みです。

全部いっぺんに使う必要はありません。この記事のTipsを読みながら「お、これ使えそう」と思ったものから試していけばOKです。

## 1. プロンプトの書き方：「何を言うか」で出力が変わる

Claude Codeの出力品質は、プロンプト（指示文）の書き方に大きく左右されます。リポジトリでは3つのプロンプトTipsが紹介されていますが、どれもすぐ使えるものばかりです。

### Claudeに「自己チェック」させる

一番おすすめなのが「変更内容をClaudeにテストさせてからPR（プルリクエスト）を出す」パターンです。

この変更をグリルして（厳しくチェックして）。テストに合格するまでPRは作らないで。

Claudeに「自分が書いたコードを自分で検証させる」わけです。人間で言えば「提出前に自分で見直して」と頼むようなもの。これだけで、ケアレスミスやバグがかなり減ります。

### 「仕切り直し」のプロンプト

修正を何回か繰り返してもイマイチなとき、こう言います。

ここまでの全情報を踏まえて、今のやり方を捨てて、エレガントな方法で実装し直して。

中途半端なパッチ（応急修正）を重ねるより、一度リセットしたほうが良い結果になることが多いです。「なんか微妙だな」と感じたら、遠慮せず仕切り直してもらいましょう。

### マイクロマネジメントしない

バグを報告するときは「エラー内容を貼って『直して』と言うだけ」でOKです。

「まずこのファイルを開いて、23行目を見て、ここをこう変えて…」みたいに手順を細かく指示する必要はありません。Claudeが自分でコードを読んで、原因を特定して、修正してくれます。手順を指示すると、むしろClaudeの判断力を制限してしまうことがあるんですよね。

## 2. 計画の立て方：「いきなり書かせない」が鉄則

リポジトリの計画系Tipsは6つあって、ここが一番「知ってるかどうかで差がつく」ポイントだと思います。

### プランモードを使う

Claude Codeには「プランモード」があります。shift + tabで切り替えると、Claudeはコードを書かずに「計画だけ」を出力してくれます。

いきなりコードを書かせると見当違いの方向に走ることがあるので、まずここで方針を確認するのが基本です。リポジトリでも繰り返し「まずプランモードから始めろ」と言われています。

### Claudeにインタビューさせる

ざっくりした要件しかないとき、こう頼みます。

以下の要件を読んで、わからないところを質問して。全部答えたら仕様書としてまとめて。

要件: ユーザーがログインできるようにしたい

Claudeが「認証方式はどれにしますか？」「パスワードリセットは必要ですか？」と聞いてきます。それに答えていくと、自然と仕様が固まっていきます。自分の頭の中にあるイメージを引き出してもらう感覚ですね。

### フェーズに分ける

大きな機能を一気に作らせるのではなく、段階を区切ります。

以下のフェーズに分けて実装して。各フェーズ完了時にテストを実行すること。

フェーズ1: データモデルの定義

フェーズ2: APIエンドポイントの実装

フェーズ3: フロントエンドのUI作成

各フェーズにテストを設定しておけば、問題があった場合に早い段階で気づけます。一気に全部作らせて最後にまとめてテスト、だと手戻りが大きくなりがちです。

### もう1人のClaudeにレビューさせる

サブエージェントを使って、経験豊富なエンジニアの視点で計画に穴がないかチェックしてもらう方法です。1人では気づかない問題が見つかることがあります。

### 仕様書は詳しく書く

リポジトリでは「仕様書を詳しく書くほど、Claudeのお守りが減る」と言われています。

「ログイン機能を作って」より「メールアドレスとパスワードでログインする機能。パスワードは8文字以上。ログイン失敗時はエラーメッセージを表示。3回失敗でロック」のほうが、出力の精度が格段に上がります。

### /simplify で整理してから次へ

フェーズが終わるたびに/simplifyコマンドでコードをリファクタリング（整理整頓）しておくと、後半のフェーズで問題が起きにくくなります。

## 3. CLAUDE.mdの育て方：AIを「自分仕様」にする

CLAUDE.mdに関するTipsは6つ。ここは「Claude Codeを長く使う」ほど効いてくるポイントです。

### 何を書けばいいか

プロジェクトの概要、チームのルール、コーディング規約（コードの書き方のルール）、設計方針の決定事項です。最初から完璧に書く必要はありません。使いながら「あ、これも書いておこう」と思ったタイミングで追加していけばOKです。

たとえばこんな感じです。

Markdown

# プロジェクト概要 ECサイトのバックエンド。TypeScript + Express で構築。 # コーディング規約 - 変数名はキャメルケース（例: userName） - APIキーは環境変数から取得。ソースコードに直書き禁止 # よく使うコマンド - テスト: npm test - 開発サーバー: npm run dev

エンジニアでない方は、「このプロジェクトはPythonで書いています。ファイル名は小文字で統一してください」くらいのシンプルな内容でも十分効果があります。

### 毎日更新する

セッション終了時にCLAUDE.mdを更新する習慣をつけると、翌日のClaudeが「昨日の続き」をスムーズに引き継いでくれます。「今日やったこと」「次回の注意点」を数行メモするだけで十分です。

### @記法で外部ファイルを読み込む

設計書やAPI仕様書が別ファイルにある場合、CLAUDE.mdに@docs/api-spec.mdと書いておけば、Claudeがそのファイルも一緒に読んでくれます。

### ルールをファイルで分割する

.claude/rules/というフォルダにルールファイルを分けて置けます。「TypeScriptのルール」「テストの書き方」「Gitの運用ルール」を別々に管理できるので、プロジェクトが大きくなっても破綻しません。

### auto-memoryを使う

Claudeがメモリを勝手に更新してくれるモードもあります。「毎回更新するのは面倒…」という方は、まずはこれを有効にしてみてください。

### lessons learnedを記録する

Claudeが過去に犯したミスのパターンをCLAUDE.mdに書いておくと、同じミスの再発を防げます。「このプロジェクトでは○○の書き方をすると△△のエラーが出る。□□の方法を使うこと」のような形式です。

## 4. デバッグのコツ：バグ修正をうまく任せる

デバッグ（バグ修正）系のTipsは8つ。Claude Codeに修正を任せるコツが詰まっています。

### 情報は全部渡す

エラーのトレース（エラーの発生経路を示す情報）やログは省略せずに全部渡してください。情報が多いほど、Claudeは正確に原因を特定してくれます。

### まず再現、それから修正

いきなり「直して」ではなく、「まずこのバグを再現して、それから修正して」と頼むのが鉄則です。再現できないバグは直しようがないので、ここを飛ばすと的外れな修正になりがちです。

### 根本原因を聞いてから直す

このバグの根本原因を説明してから、修正案を出して。

原因を理解しないまま修正に飛びつくと、別の場所で同じバグが再発することがあります。Claudeに原因分析させることで、表面的な修正ではなく根本的な解決ができます。

### 失敗したら巻き戻す

修正がうまくいかなかったら、チェックポイント機能で巻き戻せます。Escキーを2回押すか、/rewindコマンドで、変更前の状態に戻せます。「やり直しがきく」と思うと、安心して色々試せますよね。

### Webアプリは /chrome でデバッグ

ブラウザを起動して、画面の表示を確認しながらデバッグを進められます。「ボタンを押しても反応しない」みたいなUI（画面表示）のバグには特に有効です。

### エッジケースを確認させる

修正が終わったら、「想定外の入力パターンでもテストして」と頼みましょう。Claudeが見落としがちな境界値（0、空文字、巨大な値など）をカバーできます。
