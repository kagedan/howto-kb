## はじめに

Claude Codeを使って、Salesforceの2GPパッケージを2つ開発した。勤怠管理と工数・人員配置管理。ざっくり見積もって40〜50人月の規模を、1人で1ヶ月、1日数時間で実装からレビュー対応を繰り返し、一通り必要な機能の開発が完了した。

ベテランのSalesforceエンジニアがプロダクトチームでフルコミットした想定での概算。審査準備等は除外。

「40〜50人月」の算出根拠

**勤怠管理パッケージ（Gatcha）**

* カスタムオブジェクト14種（勤務パターン、休暇種別、勤怠記録、打刻変更申請、休暇申請、休日出勤申請、月次勤怠サマリ、会社カレンダー、勤怠変更履歴、勤怠設定、申請理由テンプレート、ユーザー通知設定、通知キュー、休暇残高）
* 各オブジェクトにトリガー・バリデーション・状態遷移
* 労基法対応: 休憩時間バリデーション（34条）、36協定上限管理（年720h / 複数月平均80h / 月100h）、深夜労働時間計算（22:00-05:00）、有給5日取得義務、勤務間インターバル、連続勤務日数制限
* 勤務パターン: 日勤・夜勤・フレックス・半休の組み合わせ対応
* 通知システム: 時間帯抑制・休日ブロック・キュー管理
* バッチ処理: 打刻漏れ検知、残業アラート、有給失効前リマインド、月次集計
* 承認プロセス: 2段階承認、撤回申請、代理打刻
* 管理者ダッシュボード: ページネーション・36協定タブ・CSV出力

**工数管理パッケージ（Gatcha Work）**

* カスタムオブジェクト8種 + Custom Metadata Type
* 画面点数: モーダル込みで約20
* 日報・工数入力（作業内容分割 Phase A〜C、前日コピー、サジェスト、カテゴリマスタ）
* 人員配置管理: キャパシティ計算（供給: 稼働可能時間×人数−バッファ、需要: 案件見積もり工数）
* 予実管理: 月次乖離・累計消化率・消化ペース予測・タイムライン比較
* SES精算幅モニタリング・顧客報告書出力
* 月次工数締め（排他ロック・冪等性確保）

**横断**

* 2GPモノレポ構成（2パッケージの独立ビルド・連携テスト）
* パッケージ間連携: Platform Events経由（コード上のハード依存なし）
* AppExchangeセキュリティレビュー対策: 全`without sharing`クラスにCustom Permissionチェック + CRUD/FLS強制（`stripInaccessible`全適用）
* CI: GitHub Actions + Playwright E2E + Apex全テスト
* Permission Set設計: 3ロール（一般社員 / 承認者 / 配置管理者）

勤怠管理だけで法律対応の調査・設計を含めて25〜35人月、工数管理で20〜30人月、横断基盤で5〜10人月。合計50〜75人月が現実的な見積もりだが、記事では下限の40〜50人月で記載している。

「どうやったか」ではなく、「なぜそのやり方が効いたのか」について触れようと思う。

結論を先に言うと、必要だったのは生成AIの知識ではなく、ソフトウェア開発で昔から言われてきたことだった。

## やったことの概要

開発スタイルは「方針決定以外はClaude Codeにお任せ、基本見守る」

進め方はSPEC駆動開発と呼ばれているもので、ざっくり書くとこうなる。

1. 仕様をざっくり考える
2. Claudeに逆質問してもらい、仕様を詳細化する
3. 考慮漏れや穴がないか確認してもらう
4. フェイズに分解し、さらにタスクに分解する
5. タスク単位で実装・テスト・レビュー
6. 振り返り会
7. 繰り返し

人間がやったのは「方向を決める判断」だけ。

* 労務だから法律が絡む。法律調査をしよう
* ログを残さないといけないから設計をちゃんとしよう
* 打刻がメインだから気持ちよく操作できるようにしよう
* ドメインが違う機能だからパッケージを分離しよう
* 分離するからには疎結合にしよう
* 試行錯誤が多くなったら割り込んで状況を確認しよう

一つひとつは大した判断ではない。だが、これらが間違っていたら、AIがいくら速くコードを書いてもプロダクトにならない。

## なぜ効いたのか

開発で使った考え方を並べてみると……

* コンテクストエンジニアリング
* プロンプトエンジニアリング
* ハーネスエンジニアリング
* スクラムマスター / サーヴァントリーダー
* プロダクトマネジメント
* プロトタイピング
* ナレッジマネジメント
* プロダクトライフサイクル
* シフトレフト
* QA

このうち、生成AI以降に生まれた概念はコンテクストエンジニアリング、プロンプトエンジニアリング、ハーネスエンジニアリングだけだ。残りは全部、生成AI以前から存在する。

つまり、コーディングエージェントとの開発で効いたのは、ソフトウェア開発の基礎体力だった。

## CLAUDE.mdはオンボーディング資料

Claude Codeには `CLAUDE.md` というファイルがある。プロジェクトの文脈をClaudeに伝えるためのドキュメントで、コーディング規約、アーキテクチャの方針、よくあるミスのパターンなどを書く。

これは新メンバーのオンボーディング資料と同じだ。

チームに新しい人が入ったとき、まずプロジェクトの全体像を伝え、コーディング規約を共有し、「ここだけは気をつけて」というポイントを教える。CLAUDE.mdはその全部を1つのファイルにまとめたものにすぎない。

違いがあるとすれば、Claudeはセッションが切れるたびに記憶がリセットされるので、毎回オンボーディングが必要だということ。だからこそ、ドキュメントの質が直接的に開発の質に影響する。

チーム開発でオンボーディング資料が雑だと、新メンバーが同じミスを繰り返す。CLAUDE.mdが雑だと、Claudeが同じミスを繰り返す。構造はまったく同じだ。

### 構造の工夫

プロジェクトが大きくなると、1つのCLAUDE.mdでは収まらなくなる。今回のプロジェクトでは、ディレクトリごとにCLAUDE.mdを配置した。

```
gatcha/
  CLAUDE.md                              # 全体方針・共通ルール
  .claude/
    rules/                               # 自動適用されるルール
    skills/                              # カスタムコマンド
  force-app/
    gatcha/main/default/
      classes/CLAUDE.md                  # Apex固有のルール
      lwc/CLAUDE.md                      # LWC固有のルール
    gatchawork/main/default/
      classes/CLAUDE.md                  # 工数管理Apex固有のルール
      lwc/CLAUDE.md                      # 工数管理LWC固有のルール
  docs/
    SPEC.md                              # 勤怠管理の仕様書
    SPEC_WORK.md                         # 工数管理の仕様書
```

ルートのCLAUDE.mdには全体方針だけを書き、ドメイン固有のルールは各ディレクトリに分散させる。新メンバーに「まず全体像を説明して、詳細は部署ごとに引き継ぐ」のと同じ発想だ。

これはナレッジマネジメントそのもの。知識を適切な粒度で分類し、必要な場所に配置し、メンテナンスし続ける。生成AIの話ではなく、組織の知識管理の話だ。

## 振り返りスキルはスプリントレトロスペクティブである

Claude Codeには「スキル」と呼ばれるカスタムコマンドの仕組みがある。 `/retrospective` というスキルを作って、開発サイクルの終わりに振り返りを実行するようにした。

やっていることは単純で、

1. サイクルの変更差分をgitから分析する
2. よかったこと・伸びしろ・ネクストアクションを抽出する
3. 得られた教訓をCLAUDE.mdに反映する
4. 振り返り結果をRETROSPECTIVE.mdに上書きする

これはスクラムのスプリントレトロスペクティブと同じだ。スプリントの終わりによかったことと伸びしろを出し、次に何をするかを決める。

### 昇格ルール

振り返りで見つかった教訓を、どこに記録するかにルールを設けた。

| 出現回数 | 記録先 |
| --- | --- |
| 1回目 | メモリ（一時的な記録） |
| 2回目 | `.claude/rules/`（自動適用ルール） |
| 3回目以降 | `CLAUDE.md` の Gotchas に昇格 |

チーム開発で「この問題よく起きるね」→「チェックリストに入れよう」→「コーディング規約に載せよう」と段階的に公式化していくのと同じ流れだ。

### 振り返りが回ると何が起きるか

最も重要な効果は、CLAUDE.mdが定期的に更新されることだ。ファイルが更新されると、人間は差分を見たくなる。何が変わったかが気になる。その自然な好奇心が、ドキュメントの棚卸しのトリガーになる。

「ドキュメントを定期的に見直しましょう」と決めても、人間は忘れる。仕組みがドキュメントを更新してくれるから、人間が見に行く理由が自然に生まれる。

## 品質をどこで担保するか

開発中、Claudeは妙なところで失敗した。

* E2Eテストで失敗すると try-catch でくくって「解決」しようとする
* 偽テスト（常にパスするテスト）が潜んでいる
* 修正したつもりが別の場所を壊す

これらは生成AI固有の問題ではなく、人間のエンジニアでもやりがちなミスだ。だからこそ、従来の品質管理の考え方がそのまま適用できる。

### シフトレフト — 不具合を仕様段階で見つける

「TA（Technical Architect）の観点でレビューお願いします」と「敵対的レビューをお願いします」では、返ってくる指摘の分野と深さが全然違う。同じ仕様書でもプロンプト一つで見つかる問題が変わる。

これはシフトレフトの考え方そのもので、不具合をできるだけ上流（仕様・設計）で見つける。コードになってから直すより、仕様の段階で穴を塞ぐ方がコストが低い。AIが高速にコードを書ける時代だからこそ、上流の品質がそのまま最終成果物の品質に直結する。

### QA — バグの出方からパターンを読む

品質フェーズで最も効果があったのは、個別のバグを直すのではなく「このバグとこのバグ、原因は同根では？」と俯瞰する判断だった。

バグの症状はばらばらでも、同じメソッドを経由している。普段は問題ないが、特定の条件（半休×日またぎ×バッファゼロ）で壊れる。こういうパターン認識は、コードを読むだけではなく業務知識がないと出てこない。

Claudeにアバウトに「これ同根では？」と投げると、コードベース全体を走査して本当に根本原因を見つけてくる。人間がパターンを嗅ぎ取り、AIがコードを走査して裏を取る。この分業が効くのは、QAの「根本原因分析」の実践だ。

## 人間の役割はスクラムマスターとプロダクトオーナーの兼任

やっていることを振り返ると、スクラムのPOとスクラムマスターを1人で兼任していた。

* **PO**として：何を作るか決め、バックログの優先順位をつけ、受け入れ判断をする
* **スクラムマスター**として：プロセスの障害を取り除き、開発しやすい環境を整え、振り返りを回す

Claudeが開発チーム。

セッションがスプリントで、SPEC.mdがプロダクトバックログで、BACKLOG.mdが残課題管理で、RETROSPECTIVE.mdがスプリントレトロスペクティブの記録。全部、スクラムの用語で説明がつく。

### サーヴァントリーダーシップ

スクラムマスターの本質はサーヴァントリーダーシップだ。チームが働きやすい環境を整え、障害を取り除き、チーム自身が改善し続ける仕組みを作る。

AIとの開発でも同じで、CLAUDE.mdを整備し、スキルを作り、hooksでPMDを自動実行し、MCPでドキュメント参照を楽にする。全部「Claudeが開発しやすい環境を整える」行為で、これはサーヴァントリーダーの仕事そのものだ。

## プロダクトライフサイクル

技術的負債の返済は大事だが、0にしてからリリースするのでは遅い。今回のプロジェクトでは、出荷ブロッカーとそれ以外を明確に分け、ブロッカーだけ先に潰す判断をした。

勤怠管理パッケージは法律が絡む。開発に1年かけたら、その間に法改正が入るリスクがある。速く出すこと自体がリスク回避になる。

この判断はプロダクトライフサイクルの考え方で、「いつリリースするか」「リリース時にどこまで品質を妥協できるか」を決める。AIの話ではなく、営業と開発の戦略だ。

## 再現性の検証

同じアプローチを、ドメインもスタックも異なる複数のプロジェクトで試した。

| プロジェクト | スタック | セッション数 | 成功率 |
| --- | --- | --- | --- |
| Salesforce 2GP + 小説 + PHP + Hugo | Apex, LWC, PHP, Go | 71 | 97% |
| デスクトップアプリ + 小説 | Rust, TypeScript, SolidJS | 144 | 92% |

スタックが変わっても成功率は90%を超える。後発のSalesforce 2GPのプロジェクトの方が成功率が高いのは、前のプロジェクトで鍛えたスキルやCLAUDE.mdの構造が引き継がれているから。

つまり、このアプローチは特定のドメインやスタックに依存しない。開発プロセスの設計力は応用できるといえそうだ。

## まとめ

コーディングエージェントとの開発で効いたことを振り返ると、大半は生成AI以前からある考え方だった。

| 仕組み | 対応する既存の概念 |
| --- | --- |
| CLAUDE.mdの整備 | オンボーディング、ナレッジマネジメント |
| 振り返りスキル | スクラムのレトロスペクティブ |
| 品質の担保 | シフトレフト、QA、根本原因分析 |
| 人間の役割 | スクラムマスター、サーヴァントリーダーシップ |
| リリース判断 | プロダクトライフサイクル |

新しい技術に見えるが、求められているのは昔からの原則。AIは掛け算の片方で、もう片方は人間の経験と判断力。掛ける側の係数が違えば、同じツールを使っても結果は全然違う。

生成AI時代にエンジニアに求められるものは何か。答えはシンプルで、今までの知識と経験の総力戦だ。
