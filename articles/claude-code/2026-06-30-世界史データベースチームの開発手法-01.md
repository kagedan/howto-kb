## はじめに

こんにちは。株式会社COTENでエンジニアをしている[kotono](https://zenn.dev/helloiamktn)と[mizutaku](https://zenn.dev/takuuum)です。kotonoはプロダクト開発チーム、mizutakuはデータプラットフォーム開発チームに所属しています。本記事は2人の共同執筆で、それぞれが自分のチームの内容を担当しました。  
COTENの2つの開発チームは、どちらも生成AIを前提とした開発スタイルへと大きく舵を切っている最中です。一方で、チームによって扱うシステムの性質が異なるため、AIの取り入れ方やワークフローには違いも生まれています。本記事では、タスク管理・設計・実装のフローやAIツールの活用方法など、両チームそれぞれの開発手法を紹介します。  
※これは2026年6月時点での開発手法であり、日々アップデートを行っています。

## チームの紹介

COTENは「世界史データベース」を開発している会社です。これは、歴史上の人物や出来事を構造化して蓄積し、年表や地図などさまざまな切り口から横断的に見られるようにすることを目指したデータベースで、いずれ誰もが人類の歴史を意思決定や学びに活かせる知的なインフラとなることを目指しています。  
この世界史データベースを実現するために、COTENの開発チームは大きく2つに分かれています。1つは蓄積された歴史データをユーザーに届けるプロダクトを開発する「プロダクト開発チーム」、もう1つはその歴史データを収集・加工・管理する基盤を開発する「データプラットフォーム開発チーム」です。それぞれの体制と開発しているものを紹介します。

### プロダクト開発チーム

#### 開発人数

エンジニア4名、PdM/デザイナー2名で開発を進めています。エンジニアは明確に役割を分けておらず、それぞれの得意領域や関心のある領域を担当しています。

#### 何を開発しているか

2026年2月から、新しいWebアプリケーションを開発しています。世界史データベースのデータを、年表や地図、付箋としてボード上に並べ、自分なりに整理しながら歴史調査ができるプロダクトです。

#### 技術スタック

* フロントエンド：TypeScript / React 19 / Next.js 16（App Router）/ Tailwind CSS v4 / shadcn/ui / Storybook
* バックエンド：Next.js Server Functions / GraphQL / Prisma
* DB：PostgreSQL（PlanetScale）
* 認証認可：NextAuth.js / Firebase Auth
* テスト：Vitest
* インフラ：Vercel / Sentry

### データプラットフォーム開発チーム

#### 開発人数

エンジニア5名で開発を進めています。大まかな役割はあるものの、お互いの役割を補い合いながら、チームとしてシステムとコードに責任を持つような取り組みをしています。

#### 何を開発しているか

データプラットフォーム開発チームが開発しているのは、世界史データベースそのものを支える基盤となるシステム群です。歴史データを集め、整え、品質を保ちながら蓄積していくための仕組みを、以下の3つのシステムとして開発しています。

* データ加工システム（データパイプライン）
  + 2021年7月から始動。収集した歴史データを世界史データベースの形に加工していく中核システム
* データ閲覧・編集・管理システム（社内プロダクト）
  + 2023年8月から始動。エンジニア以外でもデータ収集ができるような仕組み
* データ分析基盤
  + 2023年11月から始動。収集したデータの品質チェックや活用方法の研究を行う

#### 技術スタック

* フロントエンド：TypeScript / React 19 / Tailwind CSS v4 / shadcn/ui
* バックエンド：NestJS 11 / Python3 / Go / Hasura / dbt / Prisma
* DB：PostgreSQL / Snowflake
* 認証認可：Firebase Auth / Hasura
* テスト：Vitest / pytest / dbt-test
* インフラ：AWS / Google Cloud / Grafana / Sentry

## AIに対する考え方

### プロダクト開発チーム

設計や実装時にClaude Codeを使ったり、各種ツールのMCPを連携させたりと、開発フローには積極的にAIを取り入れています。現状は、人間がレビューや意思決定で関わりながらAIを活用するHuman-in-the-loopを基本としています。ただ、少人数でプロダクトの開発速度を上げていく必要があるため、将来的にはAIがタスクを実装し、次のタスクへ自動で着手していくところまで、人間の介入を段階的に減らしていきたいと考えています。

### データプラットフォーム開発チーム

こちらもHuman-in-the-loopを基本としており、「AIが起案・実装・一次レビューを担い、人間が方針決定・最終レビュー・マージを担う」という形をとっています。COTENは、AIが出力する世の中の（最大公約数的な）正解に適さないケースが存在します。Exitを目指さない資金調達、スタートアップのPMFとは違うプロダクト開発、特定のユーザーに対する課題解決をしない思想、データ信頼性の担保を徹底する態度など。その性質上、システム設計において一般的なベストプラクティスをそのまま適用できない場合もあるため、人間の判断を外す「Human-on-the-loop / 自律」の段階へは意図的に進めていません。とはいえAIに悲観的なわけではなく、開発人数も限られているため、慎重に判断しつつも、活用できるところは積極的に使っていく方針です。

## タスク管理

### プロダクト開発チーム / データプラットフォーム開発チーム

タスク管理の方法は両チームで共通しています。どちらもNotionで管理しており、開発初期にチームメンバーで仕様を議論し、作業単位をEpicとして整理しました。仕様変更や追加作業が発生した場合は随時Epicを起票しています。開発は2週間1スプリントのアジャイル形式で進めており、スプリント計画時にEpicの担当者を決めています。

## 設計

どちらのチームも、設計の成果物としてFeature Spec（仕様書）とDesign Doc（設計書）の2つのドキュメントを作成し、Notionで管理しています。一方で、設計の進め方やAIの関わらせ方には違いがあります。

### プロダクト開発チーム

方法を統一しているわけではありませんが、基本的にはEpic担当者がClaude Codeを使って技術調査や設計をすることが多いです。そのアウトプットを元にメンバーで議論することもあります。将来的には、AIがドキュメントを参照しやすくするため、リポジトリ内にマークダウンファイルとして配置することも検討しています。

### データプラットフォーム開発チーム

現段階では「設計に人間の意思を組み込むべき」という思想があるため、人間による積極的な介入を行う方式をとっています。レビューの精度を上げるため、AIに様々な観点から指摘を行ってもらう取り組みも始まっています。

## 実装

### プロダクト開発チーム

#### 実装とルール整備

複数人でClaude Codeを使って開発を進めるにあたり、最低限のコーディングルールは担保したいため、ESLintのルールとAGENTS.mdを充実させるよう意識しています。AGENTS.mdにはパッケージマネージャーやlintの実行方法といった作業ルール、ドキュメントマップ・スキル一覧等を100行程度で記載しており、Claude Codeはこれを起点にプロジェクトのルールと関連ドキュメントを把握します。ルートにあるdocsフォルダにはCODING\_STYLE.mdやARCHITECTURE.mdなどの詳細のルールを置いており、AGENTS.mdのドキュメントマップにそれらを紐付けています。

#### UI実装

デザイナーが用意したデザインデータがFigmaにあるため、Claude CodeとFigma MCPを連携しています。また、UIの統一化を図るために簡易的なデザインシステムも用意しています。Figma MCPとデザインシステムがあることにより、実装時は該当コンポーネントのFigma URLをClaude Codeに渡すだけでFigmaに近いUIの実装が可能になりました。しかし、まだまだ細かなレイアウトやインタラクションの調整が必要なこともあります。そこで、UIフィードバックを楽にするために[Agentation](https://www.agentation.com/)というツールを導入しました。ブラウザ上でUIに直接フィードバックを書き込み、コピーボタンからClaude Codeに貼り付けるだけで対象要素の位置まで特定してくれるので、修正指示の手間が大きく減りました。また、細かいUI調整をデザイナーが行いたい場合にも有用です。

#### 自動化ワークフロー

Issueの起票から実装、PR作成までを一気通貫で自動化するようなワークフローは、現時点ではまだ整備していません。一方で、人がClaude Codeを使って開発を進める中で、任せられる作業はAIに委譲しています。たとえばPRの作成と説明文の記述、テストコードの生成はメンバー全員がClaude Codeに任せており、コミットの分割やブランチ操作なども、明確な共通ルールはないものの各自がAIに任せていることが多いです。まずは人が主導しつつ、任せられる部分から少しずつAIに委譲している段階です。

#### コードレビュー

レビュー用のスキルを用意しており、ローカルでの開発時やPR作成時のCIで使っています。このAIレビューを整備していることもあり、プロダクト開発チームでは他のメンバーによるレビューは任意としており、基本的にはClaude Codeによるレビューのみで進めています。ただ、他の人が何を実装したかや重要な変更を見逃す可能性があるため、実装者が共有する価値があると判断したPRにはshare worthyラベルを付け、ラベルが付いたものを週に1度のコードレビュー会で共有しています。

### データプラットフォーム開発チーム

#### 実装とルール整備

ClaudeやCodexを併用し、AIによる実装を基本としています。AIのコード品質にブレが出ないよう、ルールの明文化とCI/CDの整備を徹底しています。ルールはAGENTS.mdを起点に、各サブプロジェクトのCLAUDE.mdがビルド・テスト・lintコマンドを持つ構成です。静的解析はBiome / ESLint / tscを使い、PRごとのCIではlint・ユニットテスト・実PostgreSQLを使ったDBテスト・E2E・カバレッジ計測を実行します。加えて、RLSのカバレッジチェックやPrismaスキーマのdrift検知も自動で走らせ、これら全チェックが通らないとマージできないようにしています。

#### 自動化ワークフロー

実装の一部は、ワークフローとして自動化を進めています。現在はPoC段階ですが、GitHub Issueに特定のラベルを付与するとAIが設計・実装からPR作成まで行うワークフローや、PRにコメントを入れると修正コミットをpushするワークフローなどがあります。これらはClaude Code GitHub Actionsを中心に実現しています。また、非エンジニアでもAIワークフローを起動できるよう、SlackやNotionからGitHub Issueを作成できる仕組みも計画中です。

#### コードレビュー

最終レビューは人間が行いますが、Claude / Codexによる一次レビューも取り入れています。詳細は後述のコード品質の項で説明しますが、過去レビューの傾向からルール化を行い、そのルールに従ってAIにレビューを行ってもらっています。

## AIの精度を高める工夫（Skills・コード品質）

### プロダクト開発チーム

チームで共有するカスタムスキルを.claude/skills/ にいくつか定義しています。開発環境の起動用スキル、shadcn/uiコンポーネントの追加用スキルといった開発ワークフローの自動化に加え、ローカルコードレビュー用スキル、npm脆弱性の修正スキル、セキュリティスキャン用スキルといった品質・セキュリティ系のスキルも整備しています。

### データプラットフォーム開発チーム

Skillsについては、チームで共有しているものは今のところ特にありません。現状は各自が使いやすいものを作成して使っており、その中で便利だったものをチーム内で共有する、という形をとっています。コード品質については、AIレビューの精度を上げるため、4年間蓄積された約1000件のレビューコメントから、よく頻出するレビュー観点をリスト化し、約60個のルール（アーキテクチャ・命名・エラー処理・テストなど）をAIに与えています。

## おわりに

ここまで、COTENの2つの開発チームの開発手法を紹介してきました。どちらもHuman-in-the-loopを基本に、AIを開発フローへ積極的に取り入れている点は共通しています。一方で、プロダクト開発チームは少人数で開発速度を上げるためにAIへの委譲を進め、データプラットフォーム開発チームは中核システムを支える立場から人間の関与を保つことを重視しており、AIとの距離の取り方には違いが生まれています。ここで紹介した手法は2026年6月時点のもので、両チームとも日々試行錯誤を重ねている最中です。どこかひとつでも、みなさんの開発の参考になれば幸いです。
