AIエージェントのコストを token 数で見ていると、たぶん最初に読み違える。

高いのは一回の返答ではない。勝手に走る仕事である。

短い prompt を書いた。だから安い。軽い model を選んだ。だから安全。そう考えたくなる気持ちは分かる。チャット型の AI なら、ある程度はその感覚で読めた。人間が質問し、AI が返し、人間が次を決める。会話の拍は人間側にある。

でも agentic coding になると、拍が変わる。

AI は読む。探す。直す。テストする。失敗する。別の仮説でやり直す。subagent を呼ぶ。auto-review を走らせる。branch 上でしばらく作業する。人間が画面を見ていない間にも、仕事は進む。

便利だ。だから危ない。

## usage limit の話を、料金の話だけで終わらせない

Codex の usage が想定より速く減る、という話は、表面的には usage limit のニュースに見える。プランごとの上限、消費の見え方、auto-review や retry の扱い。もちろん利用者にとっては大事だ。

ただ、開発者が見るべき論点はもう少し手前にあると思う。

OpenAI の Help でも、Codex の利用は agentic usage limit に含まれ、長い task、大きい codebase、広い編集やテストほど usage を消費しやすいと説明されている。これは「トークンを節約しましょう」というだけの話ではない。

何を agent に渡したのか。

どのくらい探索させたのか。

失敗したとき、何回まで retry させたのか。

このあたりを見ないと、請求や上限だけを眺めても原因が分からない。短い依頼でも、実行範囲が広ければ高い。逆に長い依頼でも、境界が狭く、確認方法が明確なら読みやすい。

agent のコストは文章量ではなく、実行された仕事量に寄っていく。

## 「補完」ではなく「実行ループ」として見る

AI コーディング支援を補完の延長として見ると、設計を間違える。

補完なら、失敗しても比較的軽い。候補が違えば捨てればいい。チャットの回答が外れていれば、次に聞き直せばいい。人間はほぼ常にそこにいる。

エージェントは違う。

Claude Code の hooks を見ると分かりやすい。SessionStart、PreToolUse、PostToolUse、SubagentStart、SubagentStop。実行の入口だけではなく、tool を使う前後、subagent が動く前後、session の単位で止める地点がある。

これは便利な拡張点である前に、境界線である。

どこで止めるのか。どこで記録するのか。どこで人間へ戻すのか。ここを決めずに hook を増やすと、自動化は賢くなるというより、読みにくくなる。処理は増える。ログも増える。だが、責任が戻ってくる場所は曖昧なままだ。

subagent も同じだ。

専門分化できる。tool restriction や permission mode を分けられる。custom prompt を持たせられる。うまく使えば強い。ただし、subagent を「小さくて便利な worker」とだけ見てしまうと、単に見えない作業者が増える。

subagent は分業の単位である前に、権限とログと停止条件の単位である。

## background work は PR の前に膨らむ

GitHub Copilot cloud agent のような仕組みは、この変化をさらに分かりやすくする。

IDE の横で候補を出すのではなく、GitHub Actions powered な環境で repository を調べ、実装計画を作り、branch 上で変更し、test を試し、PR 作成前にも iteration する。ここまで来ると、AI は「答える道具」ではなく「作業場を持つ小さな実行者」になる。

そうなると、レビューの位置も変わる。

PR が出てから見る、では遅いことがある。PR の前に、調査、計画、branch、test、retry が走っているからだ。人間が最後の diff だけを見ると、そこに至るまでの判断や失敗の履歴が抜け落ちる。

しかも、その抜け落ちた部分にコストがある。

大きすぎる issue を渡す。成功条件を曖昧にする。触ってよい範囲を決めない。失敗時の retry 上限を決めない。すると agent は親切に広く走る。親切に何度も試す。親切に branch 上で作業を増やす。

人間にとっては「まだ PR は出ていない」かもしれない。

agent にとっては、もうかなり仕事をした後である。

## 最初に決めるべき最小セット

チームで agent を入れるなら、最初に model 比較表を作るより、作業の上限を決めたほうがいい。

大げさな governance でなくていい。むしろ最初から重くすると使われない。最低限、次のようなものだけでかなり変わる。

* task budget: その依頼で読んでよい範囲、編集してよい範囲、期待する差分の大きさ
* retry cap: test や build が落ちたとき、何回まで自動で直してよいか
* subagent cap: どの種類の subagent を何の目的で呼んでよいか
* hook stop point: tool 実行前、実行後、subagent 開始時など、どこで止めるか
* session timeout: どのくらい走ったら人間へ戻すか
* review queue: 結果を誰が、どの粒度で読むか
* cost owner: usage とレビュー時間を誰の仕事として見るか

これはプロンプト術ではない。作業設計である。

「この画面をいい感じに直して」は短いが、安い依頼ではない。探索範囲が広すぎるからだ。逆に、「この form の validation message だけを既存 component の表示ルールに合わせる。仕様は変えない。対象 directory はここ。確認 command はこれ」と書けば、prompt は長くても仕事は狭い。

エージェントに渡す言葉は、お願いではなく実行境界になる。

## 個人開発でも、放置すれば同じことが起きる

これは大きいチームだけの問題ではない。

個人開発のほうが、むしろ簡単に広がることがある。自分の repository だから approval が速い。token も自分のもの。レビュー担当も自分。便利な CLI や extension を入れて、そのまま private repo まで届かせる。

そして気づくと、AI が触れる範囲が自分の頭の中の想定より広くなっている。

個人なら、まず捨てられる差分の大きさを決めるだけでもいい。失敗したら丸ごと捨てられる task にする。調査だけ、設計だけ、単一 file の修正だけ、test 追加だけ。そういう単位に切る。

全部自動 review させれば安心、という感覚も少し危ない。review も仕事である。何を review させるか。失敗した review をどう扱うか。review 結果を人間がどこまで読むか。それを決めないと、自動 review は安心ではなく、もう一つの background work になる。

## 安く使うとは、短い prompt を書くことではない

AI エージェントは、これからもっと background 側へ寄っていくと思う。

人間が寝ている間に branch を作る。issue を読んで plan を出す。test を直す。PR の前に何度か試す。これはかなり強い。開発の詰まり方を変える可能性がある。

ただし、background work は、見えないだけで無料ではない。

usage を使う。CI を使う。review queue を使う。人間の注意を使う。ときには credential や repository の境界もまたぐ。だから、agent を安く使うというのは、単に安い model を選ぶことではない。

勝手に走る仕事へ名前を付けることだ。

上限を付けることだ。

止まる場所を作ることだ。

AIエージェント時代の開発コストは、料金表の端だけを見ていても読めない。読めるのは、どの仕事を、どこまで、誰の責任で走らせるかを決めたあとである。

短い prompt より、短く止まれる仕事。

たぶん、そちらのほうが長く効く。
