# Claude CodeでLPを作って気づいた。「作る力」より「名前を知っている力」が重要になっている

<https://hallowfield.vercel.app/>  
最近、Claude CodeとGeminiを使ってLPを作っていました。数年前までなら、こうしたサイトを作るにはデザイナーやフロントエンドエンジニアが必要でした。レイアウトを考え、アニメーションを実装し、細かな挙動を調整する。それなりの工数がかかる仕事です。しかし今は、AIに指示を出すだけでかなりリッチなサイトが数分で出力されます。

実際に使ってみると、その進化には驚かされます。コードを書く能力だけで言えば、多くの場面で人間以上です。複雑なレイアウトも組めるし、アニメーションも実装できる。少し前までなら調べながら数時間かかっていた作業が、一瞬で終わることも珍しくありません。

ただ、制作を進める中でひとつ面白いことに気づきました。AIは作れるのですが、こちらが名前を知らないものは作れないのです。

例えば最近のLPでは、スクロールに応じて要素が固定されたり、背景と前景が異なる速度で動いたり、コンテンツが順番に現れたりと様々な演出が使われています。こうした表現には Scroll Jacking、Parallax、Sticky Scroll、Reveal Animation、Marquee、Bento Grid といった名前があります。

人間同士の制作なら「Appleっぽい感じで」「最近のスタートアップっぽく」といった曖昧な会話でも何となく伝わることがあります。しかしAIとの仕事では事情が少し違います。AIは実装力が高い一方で、指示された内容を起点に動きます。そのため、表現の名前を知っているほど指示の解像度が上がり、逆に名前を知らないものは存在しないのとほぼ同じ状態になります。

この変化を見ていて、昔から技術やデザインの世界にいた「やたら詳しい人」の価値が上がっているように感じました。CSSの仕様を追いかけている人、海外のデザインギャラリーを毎日見ている人、新しいライブラリや表現手法を集めている人。以前は「知識はあるけど実務でどれだけ役立つのか分からない」と見られることもありました。しかしAI時代になると、その知識自体がプロンプトになります。

知っている言葉の数だけ、AIに依頼できることが増える。知っているパターンの数だけ、アウトプットの選択肢が広がる。つまり、これまで趣味やオタク知識として扱われていたものが、そのまま生産性に変換されるようになったのです。

もちろん実装力が不要になるわけではありません。出力されたコードを評価したり、問題点を発見したり、最終的な品質を担保したりするには依然として知識と経験が必要です。ただ、価値の重心は少し変わってきているように感じます。これまでは「どうやって作るか」が重要でしたが、これからは「何を知っているか」「どんな名前で呼べるか」の重要性が増していくのではないでしょうか。

Claude CodeでLPを作りながら感じたのは、AIによって実装コストは確実に下がったということです。しかし同時に、知識の価値はむしろ上がっています。AIは作業を代替できますが、「どんな表現が存在するのか」までは教えてくれません。だからこそ、日頃から表現を観察し、名前を覚え、語彙を増やしてきた人が強い。AI時代になって、意外にもオタク的な知識が再評価され始めているように感じています。

おまけ：Claude Codeへの指示例

実際の開発で使った指示を抜粋します。「こう言えばこう動く」という参考にしてください。

---

1. 機能の丸投げ

画面の両サイドに装飾のツタを配置したい。  
スクロールに連動してツタが伸びて、途中で花が蕾から開花するアニメーションを追加して。  
デザインはサイト既存のカラーパレット（--rose, --ember）に合わせて。

Claude Codeがコードを読んで既存の loop() 関数とスクロール変数 p を把握し、CSS / HTML / JS の3箇所を順番に実装してくれた。

---

2. 実装方針の変更

SVGのstrokeDashoffsetでツタを動かしてるけど、  
同じ動画シーク方式（currentTime制御）でできない？  
vine\_left.mp4 を用意したのでそっちを使って。

既存の hallowfield\_seekable.mp4 のシーク制御パターンを流用して SVG → 動画に置き換えてくれた。

---

3. 見た目のバグ報告（説明なしでOK）

ツタの動画が画面の左半分しか表示されてない

原因調査から解決策（hidden video → dual canvas + chroma-key）まで自律的に提案・実装してくれた。

---

4. バグ修正（現象だけ伝える）

スクロールしてもツタが動かない。シーク後にcanvasが更新されないみたい。

currentTime セットが非同期であることを特定し、seeked イベント駆動への書き換えまで一発で直してくれた。

---

5. パフォーマンス改善の指示

chroma-keyのcanvas処理が重そう。最適化して。

Uint32Array を使ったピクセルアクセスと half-resolution canvas（ピクセル数を1/4に削減）を提案・実装してくれた。

---

6. 感覚的なフィードバック

なんか全体的に文字ちっちゃい

html { font-size: 20px; } 1行を追加するだけで全 rem 値を比例スケールアップ。根本原因の特定から修正まで自律で完結。
