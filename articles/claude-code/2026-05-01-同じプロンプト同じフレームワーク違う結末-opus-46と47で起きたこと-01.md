[1本目](https://zenn.dev/belka_dolphin/articles/c885376e01b59d)でQDHEを設計しテスト凍結+FMEAを導入。  
[2本目](https://zenn.dev/belka_dolphin/articles/3702746398a933)で375件GREEN→遊べない問題に直面。  
[3本目](https://zenn.dev/belka_dolphin/articles/d8e32c4036062d)で同条件再実行、[4本目](https://zenn.dev/belka_dolphin/articles/ae2126c1720663)でv2.1導入→さぼり問題。  
[5本目](https://zenn.dev/belka_dolphin/articles/d491dbfafdaa59)でフレームワークなしとの対照実験で品質差を定量化。  
[6本目](https://zenn.dev/belka_dolphin/articles/fa86d28d510a40)でECサイトのL3フルパイプラインを完走させ、QDHEの汎用性を実証。

ここまでの検証はすべてOpus 4.6の時代に積み上げてきたものだ。

そして本記事は、**Opus 4.6から4.7への切り替えで起きた事件の記録**である。

## 事件

2026-04-08、Opus 4.6で**2時間12分**で動く人狼ゲームができた。シリーズ7回目の人狼ゲームで、ブラウザを開けば普通に遊べるものが完成していた。

それから約2週間後、**全く同じプロンプト、全く同じQDHEフレームワーク**でOpus 4.7に投げた。

実働**8日間**かけて、324件のテストを全GREENにした。カバレッジはline 95.69% / branch 87.39%。QDHE内部指標としては「完了」状態に到達した。

**遊べなかった。**

| 項目 | Opus 4.6 | Opus 4.7 |
| --- | --- | --- |
| 期間 | 2時間12分 | 実働8日間 |
| スライス数 | 5 | 12 |
| ソースコード | 2,152行 | 4,092行 |
| テスト数 | 121 | 324 |
| ドキュメント | 1,381行 | 16,000行超 |
| pipeline-state.json | — | 2,691行 |
| unfreeze回数 | 0 | 6 |
| エスカレーション | 0件 | 4件 |
| **ブラウザで遊べたか** | **遊べた** | **遊べなかった** |

変えたのは、Opus 4.6から4.7への切り替えだけ。プロンプトとフレームワークは1文字も変更していない。

## Anthropicが予告していた変化

実は、この事態はAnthropic自身が公式に予告していた変化の延長線上にある。

Anthropicの公式マイグレーションガイドは、4.7の主要な行動変化を以下のように記述している。

> More literal instruction following: Claude Opus 4.7 interprets prompts more literally and explicitly than Claude Opus 4.6, particularly at lower effort levels. It will not silently generalize an instruction from one item to another, and it will not infer requests you didn't make.

要約すると、4.7は4.6より字義通り・明示的にプロンプトを解釈する。ある項目から別の項目への暗黙の一般化はしない。要求されていない事柄を推論しない。

そして同ガイドはこう続ける。

> A prompt and harness review may be especially helpful for migration to Claude Opus 4.7.

Claude Opus 4.7への移行にあたっては、**プロンプトとハーネスのレビュー**が特に役立つ可能性がある、と。

QDHEはまさに「ハーネス」だ。Anthropicが公式に「レビューが必要」と述べている対象そのものである。

Claude CodeのリードであるBoris Chernyも、4.7リリース日に「効果的に使えるようになるのに数日かかった」と発言している。AIモデル開発の最前線にいる本人が「慣れる必要があった」と認めるレベルの変化が、4.6から4.7の間で起きている。

第三者の解説記事の表現を借りれば、変化はこう要約できる。

> 4.6 guessed. 4.7 stopped guessing.

4.6は推測した。4.7は推測をやめた。

本記事は、この公式に予告されていた変化が、実際のフレームワーク運用でどう露呈したかの記録である。

## 何が起きたか

4.7プロジェクトの完了状態はこうだった。

* 全12スライス完了
* 324テスト全GREEN（1件のみskip、これは数学的不可能と判明したケース）
* カバレッジ line 95.69% / branch 87.39%
* 凍結ハッシュ12スライス全て確定
* QDHE内部指標で「完了」を宣言

そして、`vite.config.ts` / `index.html` / `src/client/main.tsx` / `src/client/App.tsx` / HTTPサーバが**全て欠落**していた。

Reactコンポーネントは個別に実装され、jsdom上でレンダリングテストはGREEN。WebSocketサーバも単体では動作する。だがブラウザで開く方法が存在しない。`npm run dev`を叩いてもフロントエンドが起動しない。

要件には「ユーザーが遊べるもの」と明記されていた。テストは全部通っていた。それでも遊べないものができあがっていた。

## プロンプトもフレームワークも完全に同一だった

念のため、与えたプロンプトを確認する。冒頭にはこう書いてある。

> 以下のアプリを作りたい。バックエンドからUIまで一貫して作成し、ユーザーが遊べるものを作って。

そして「フロントエンド」セクションに5画面（ロビー/メインゲーム/夜フェーズUI/投票UI/結果画面）が、「バックエンド」セクションにWebSocket通信が明示されていた。

**フレームワークも同一**。同じQDHEのv2.x、同じ7エージェント体制、同じ10ステップパイプライン、同じテスト凍結ルール、同じFMEA設定。何ひとつ変更していない。

差し替えたのはOrchestrator/Reviewer/AnalystのモデルがOpus 4.6から4.7になったことだけ。Workerは両方ともSonnetのまま。

## 4.6成果物の調査

「なぜ4.6では動いたのか」を解明するために、4.6成果物を1ファイル単位で調べた。

**テストの中身**:

`tests/U1/`配下のテストファイルは2本だけだった。

| ファイル | importしているもの |
| --- | --- |
| `tests/U1/gameReducer.test.ts` | `src/client/state/gameReducer.js`, `types.js`, `gameTypes.js` |
| `tests/U1/useWebSocket.test.ts` | `vitest`のみ（MockWebSocketを自前定義） |

App.tsxもmain.tsxも**どこからもimportされていない**。`vite`も`@vitejs/plugin-react`も参照されていない。`vitest.config.ts`はenvironment:"node"でjsdomは未使用。

**テストリスト**:

`docs/test-list-U1.md`は20件のみ。「ブラウザで起動」「初期表示」系のテストは1件もない。SMOKE/E2Eは別スライス（I1）に集約されていたが、I1のテストもnode環境でWebSocketサーバを立てるだけで、ブラウザは立てない。

**スライス計画**:

`docs/slice-plan.md`のU1スライスの対象ファイルには、`src/client/state/`、`src/client/components/`、`src/client/hooks/`は明記されていた。しかし`App.tsx` / `main.tsx` / `index.html` / `vite.config.ts`は明示列挙されていない。「Build & Wire」のような独立スライスも存在しない。

**判断ログ**:

`.harness/decisions.md`は空のヘッダーのみ。「U1でApp.tsxも作る」「vite.config.tsを追加する」みたいな判断記録はどこにも残っていない。

**にもかかわらず、これらのファイルは実際には存在し、ゲームは動いていた。**

ファイル更新タイムスタンプを並べると、U1スライスのStep 7実装中に8分間で次の順序で連続作成されていた。

```
23:03 gameReducer.ts, types.ts        ← U1 のロジック
23:05 useWebSocket.ts, App.tsx, Lobby.tsx
23:07 main.tsx, ResultScreen.tsx, index.html
23:08 vite.config.ts, package.json    ← Build 設定が最後
```

Coderエージェント（Opus 4.6が担当）は、設計書の`docs/design.md`に書かれたディレクトリ構成図を「全部作るリスト」として解釈し、テスト要求の有無に関わらずファイルを作っていた。さらに、設計書にすら記載のない`vite.config.ts`まで、要件の技術スタック表に「Vite」と書いてあることだけを根拠に補完していた。

## 越境補完という発見

整理するとこうなる。

* `App.tsx` / `main.tsx` / `index.html`：設計書のディレクトリ構成には登場するが、テスト要求はない。それでも作られた
* `vite.config.ts`：設計書にも登場しない。要件の技術スタック表に「Vite」と書かれているだけ。それでも作られた
* `decisions.md`に判断ログなし。Coder自身は「特別なことをしている」という自覚を持っていない

QDHEの本来のルールでは、Coderはスライス計画に明記されたファイルだけを実装する。テスト要求のないファイルを勝手に作るのはルール違反だ。

しかし4.6は、要件と設計書を読んで、**スライス境界・テスト要求の外側に踏み出して、必要なものを補完していた**。

これを**越境補完**と呼ぶことにする。

越境補完は、フレームワークから明示的に指示された範囲を超えて、要件達成のためにモデルが自発的に行う補完行動だ。今回判明したことは、QDHEというフレームワークが、この越境補完に**気づかないまま依存していた**という事実である。

そしてこれは、冒頭で引用したAnthropic公式の言葉と完全に対応している。

> It will not silently generalize an instruction from one item to another, and it will not infer requests you didn't make.

「設計書のディレクトリ図に書かれた他のファイル名から、`vite.config.ts`の必要性を一般化することはしない」「要件で明示されていない起動土台の生成を推論することはしない」。これがまさに4.7で起きたことだ。

4.6が暗黙にやっていた一般化と推論を、4.7はやらない。Anthropicが公式に予告していた変化が、ハーネス（QDHE）の挙動として具体的に現れた。

## 4.7は越境補完をしなかった

同じプロンプト、同じフレームワーク、同じ設計書、同じ要件。それを受け取った4.7は、こう振る舞った。

* テスト要求がないファイルは作らない
* スライス計画に明記されていないファイルは実装範囲外と判断する
* 設計書のディレクトリ構成は「参考情報」であって「実装義務リスト」ではないと解釈する

結果、内部指標は完璧に達成された。

* テスト全件GREEN
* カバレッジ高
* 凍結ハッシュ確定
* チート禁止ルール一切違反なし
* 7エージェントの委任プロトコル全てクリア

しかし起動土台が一切なく、ブラウザで動かせない。

4.7はQDHEのルールに極めて忠実だった。**忠実だったからこそ、フレームワークの欠陥がそのまま結果として露呈した**。

ルール厳守度の高いモデルほど、フレームワークの暗黙前提に苦しむ。これが今回得られた中核的な観察である。

## 8日間で何が起きていたか

2時間で済むはずの作業がなぜ8日間かかったか。記録を追うと、いくつもの摩擦が観察できる。

**unfreeze（凍結解除）が6回発生した**。テスト凍結後に仕様の穴が後発スライスから見えてきて、先行スライスのテストを書き直すケースが連続した。4.6時代は0回。

**`pipeline-state.json`が2,691行に肥大化した**。各エージェントの委任結果、unfreeze履歴、エスカレーション履歴、人間承認履歴が精密に記録された結果、Orchestrator自身がこれを毎回読み込むコストが増大し、context windowを圧迫してセッション分断が複数回発生した。

**1スライス（I1）の修正に11ループを要した**。最終的な修正は2行のコード変更だったが、症状を見て対症療法に飛びつき、根本原因に到達するまで3ループ、修正タイミングの誤りで1ループ無駄、という典型的な局所最適化収束失敗が起きた。

これらは個別には小さな摩擦だが、累積すると2時間が8日間になる。

そして8日間かけてテストを完璧に通した先に待っていたのが、「ブラウザで遊べない」だった。

## フレームワークの暗黙前提

QDHEは、4.6の越境補完力に**暗黙的に依存していた**フレームワークだった。

「テスト要求はないが要件達成のために必要なファイル」を拾うメカニズムが、フレームワーク内に存在しなかった。それを4.6が無自覚に補完していた。フレームワーク設計者（私）も、4.6が補完していることに気づいていなかった。

両者の特性が偶然噛み合っていた間、QDHEは「言語非依存」「ドメイン非依存」「レベル非依存」のフレームワークとして機能しているように見えた。実際は**モデル依存**だった。

これはQDHEだけの問題ではないと考える。以下のような構造を持つAI開発フレームワークは、同じ問題を抱えている可能性がある。

* 「常識的に必要なものは作られる」を前提にしたスコープ定義
* ハードコードされたテスト要求リストに頼った実装範囲の決定
* 「とりあえずやれば動く」を前提にした起動・配線の暗黙化
* スライス・モジュール境界に「誰の責務でもない」領域が存在する設計

特にClaude Codeやcursorのような「カスタム指示書 + ルール集」を作り込んでいるユーザーは、自分のルールセットがモデルの越境補完力に依存していないか、一度棚卸しする価値がある。

## 検出が困難である理由

越境補完依存の厄介さは、**通常のテストでは検出できない**ことにある。

QDHEには内部品質を測定する仕組みが多数組み込まれている。テスト凍結、ミューテーションテスト、カバレッジ計測、FMEAによる故障モード分析、トレーサビリティマトリクス。これらは全て「内部指標」を測る。

しかし「フレームワークの定義する完了状態」と「ユーザーが期待する完了状態」の乖離は、内部指標では一切測れない。テスト全件GREEN・カバレッジ95%という状態は、**両方の完了状態を同時に満たしているように見える**。実際は片方だけしか満たしていない。

この乖離は、モデルが越境補完してくれている間は表に出ない。補完しないモデルに切り替わった瞬間に初めて、フレームワークが何に依存していたのかが露呈する。

## 教訓

今回の発見から得られる教訓は3つある。

**自分のフレームワーク・プロンプト集が「モデルの越境補完力」に依存していないか棚卸しする**。具体的には、過去に動いた成果物のうち、ルールにもテストにも明示されていないが存在しているファイル・コード・設定を全て洗い出す。それらは越境補完で作られた可能性が高い。

**「テスト要求の外側で作られているもの」を明示的にリストアップする**。起動土台、設定ファイル、シェルスクリプト、デプロイ設定、CI/CD定義など、「あって当然」のものほど明示化されにくい。これらが暗黙のままだと、ルール厳守型モデルでは生成されない。

**モデル更新のたびに同じプロンプトで動くか検証する**。越境補完依存は事前検出が困難なため、モデル切り替え時の同条件再現テストが最も確実な検出手段になる。差分が出たら、それは越境補完で埋まっていた何かが消えたサインだ。

QDHEは現在、v2.8でこの問題を構造的に解決する作業に入っている。具体的には**Build & Wireスライスの必須化**と**要件EARS文への観測階層紐付け**を導入する予定だ。「越境補完で埋まっていた領域」をフレームワークに組み込んで明示化する方向である。

結果は次回の記事で報告する。

## 余談 — モデルはサプライヤである

最後に、車載業界での経験から1つだけ補足させてほしい。

車載業界では、サプライヤを変更すると同じ仕様書でも品質が変わるのが常識だ。だから「サプライヤ評価表」で各社の特性を記録し、仕様書には「A社向け」「B社向け」の追加要件を書き足す。前任サプライヤのベテラン工員が暗黙にやっていた工程は、新サプライヤには通用しない。だから明示化する。

AI開発でも同じことをすべき時代になったのだと思う。**モデルは新しいサプライヤである**。同じ仕様書で違う結果が出るのが普通になる。それを前提に、フレームワーク側で暗黙前提を明示化する。

QDHEはこれから、その作業を進める。
