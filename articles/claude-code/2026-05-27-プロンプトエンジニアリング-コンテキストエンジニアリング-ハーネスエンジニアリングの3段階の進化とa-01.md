こんにちは、株式会社エムニでPM兼エンジニアをしている、まさぴょんです🐱  
今回は、**プロンプトエンジニアリング →コンテキストエンジニアリング → ハーネスエンジニアリング**の3段階の進化とAI駆動開発について整理していきます📝

今や、Claude Code / Cursor / Codex といったAIコーディング・エージェントを使った「AI駆動開発」が、もはやスタンダードになっていますね🔥

ただ、「AIにいい感じに頼む」だけだと、なかなかうまくいかないことが多いです🥺  
プロジェクト固有の事情・既存コード・規約・テスト・CI・レビュー導線まで含めて、**AIが失敗しにくい開発環境そのもの**を作っていく必要があります📝  
そして、このAIを正しく導くための環境整備のことこそが、最近流行りの**ハーネスエンジニアリング**になります📝

## TL;DR：3段階の進化について

3段階の進化について、表に整理しました。

| 段階 | 主な問い | 対象 | AI駆動開発での例 |
| --- | --- | --- | --- |
| **1. プロンプトエンジニアリング** | どう指示すればいい出力になるか? | 指示文、ロール、出力形式、例示 | 「あなたはプロのエンジニアです。React + TypeScriptでFrontendを実装をしてください。」「この形式でPR説明を書いて」 |
| **2. コンテキストエンジニアリング** | AIに何を見せれば正しく判断できるか? | 仕様書、既存コード、設計方針、ログ、RAG、履歴・メモリ、ツール結果 | `CLAUDE.md`、設計書、既存API仕様、DBスキーマ、過去のPRやエラーログなど履歴を渡す |
| **3. ハーネスエンジニアリング** | AIが安全・再現性高く作業できる仕組みは何か? | テスト、lint、型チェック、CI、権限、サンドボックス、レビューエージェント、MCP、監視、セルフPDCA | AI実装後に自動でテスト・lint・型チェック・セキュリティチェックを走らせ、失敗したら自己修正させる。etc... |

### 3つの概念は「入れ子」になっている📝

3つの概念の「入れ子」の関係を絵で見ていきます📝

![](https://static.zenn.studio/user-upload/facdedfb80cc-20260520.png)

イメージとしては、マトリョーシカみたいに包まれている感じです🪆

* 一番内側にあるのが **プロンプト (指示文の最適化)**
* それを取り囲む情報環境全体が **コンテキスト (情報環境の設計)**
* さらに、そのコンテキストを生み出すループ・ツール・メモリ・エラー回復までを含めたシステム全体が **ハーネス (システム・環境の設計)**

## 3段階の登場タイムライン📝

それぞれの段階が「支配的になった時期」をざっくりタイムラインで整理すると、こんな感じになります🐱  
![](https://static.zenn.studio/user-upload/f89ea2502677-20260520.png)

この記事では、3つの概念について順番に深掘りしていきます。

## 1. プロンプトエンジニアリング (2022〜2024頃)

![](https://static.zenn.studio/user-upload/871c646c00c2-20260520.png)

最初のステップは、プロンプトエンジニアリングです📝  
ひとことで言うと、**AIに「うまく頼む」段階** ですね。

ChatGPT登場 (2022年11月) 以降、LLMから望ましい出力を得るための「指示文の書き方」を磨く手法として広まりました📝  
当時のモデルは指示の解釈にブレがあって、言い回し一つで出力品質が大きく変わったので、人格設定や、職人芸的なプロンプト設計手法が編み出されていました。

### 主な手法・アプローチ📝

代表的な手法・アプローチは、こんな感じです👀

* Zero-shot / Few-shot prompting（例示を与える）
* Chain of Thought（段階的に考えさせる：「ステップごとに考えて」）
* 役割（ペルソナ）・ロール付与 (「あなたは〜の専門家です」)
* 出力フォーマット指定 (JSONでのOutput指定など)
* プロンプト内でのルールの整備
* 区切り文字でのセクション化

### AI駆動開発における典型的な使い方📝

最初の頃のAI駆動開発は、こんな依頼の仕方が多かったと思います👀

```
あなたは熟練のReactエンジニアです。
以下の要件に従って、Next.js + TypeScriptでログイン画面を実装してください。
出力はファイル単位で示してください。

## 要件
1. 〜〜〜〜〜
2. 〜〜〜〜〜
```

ただ、これだけの情報量だと、落とし穴があります...  
それは、AIはプロジェクト固有の設計・既存コードの癖・DB構造・命名規則・過去の意思決定を知らないので、**「それっぽいけど、プロジェクトには合わないコード」** を出してしまうことです。

### プロンプトエンジニアリングの限界

モデルが学習した過去の知識（事前学習データ）に依存するため、最新情報や社内の非公開データには答えられません。  
また、複雑な業務フローを1つのプロンプトで完結させることには限界があります。  
さらに、モデルが賢くなってくると、こういう小手先の工夫の効果はだんだん薄れてきます。

## 2. コンテキストエンジニアリング (2025年中頃〜) 📝

![](https://static.zenn.studio/user-upload/ba2154e621d6-20260520.png)

コンテキストエンジニアリングは、プロンプトの工夫だけでは解決できない「知識の壁」を突破するための技術です📝  
AIが正確な回答を生成するために必要な「外部情報（コンテキスト）」を収集して、LLMに渡す技術です。

### 主な手法・アプローチ📝

代表的な手法・アプローチは、こんな感じです👀

* RAG（Retrieval-Augmented Generation: 検索拡張生成）
* 過去の会話履歴・ログを渡す。
* Rules (`CLAUDE.md`など)
* Web Srcの参照など

こういったアプローチを通して、**モデルが参照する情報全体(コンテキスト)** をどう構成・圧縮・順序(優先順位)付けするかが問われるようになりました！

### 進化のポイント

AIに「暗記」させるのではなく、AIに「辞書（データベース）」を引かせ、その内容に基づいて回答させる仕組みを作りました。  
これにより、ハルシネーション（もっともらしい嘘）が大幅に削減されました。

!

LLMには **注意リソースの制約** があって、文脈(コンテキスト)が膨らむほど重要情報の取りこぼしや混乱が起きやすくなります😭

これは「ハルシネーション(もっともらしい嘘)」を抑える上でも本質的な観点で、必要な情報だけ渡すことが、効果的な対策になります📝

いいコンテキスト設計は、 **「望ましい結果の確率を最大化する、最小限なコンテキスト情報を渡すこと」** だと思います📝

### AI駆動開発におけるコンテキスト設計の事例としての `CLAUDE.md`

AI駆動開発では、たとえば遵守してもらいたいルールなどの事前コンテキストを `CLAUDE.md` などに整備しておきます📝

```
# CLAUDE.md

## Frontend Rules
- このプロジェクトのNext.js App Routerを使う
- UIはshadcn/ui + Tailwindを使う
- API通信はsrc/lib/api.ts経由に統一する

## Backend Rules
- DBアクセスはPrisma経由にする
- テストはVitestで書く

## Common Rules
- 新機能追加時は必ず既存の類似実装を検索してから実装する
```

### Just-in-Time コンテキスト戦略📝

Anthropicは、エージェントに最初から全情報を詰め込むのではなく、  
**ファイルパス、検索クエリ、リンクなどの軽量な参照** を持たせて、必要に応じてツールで情報を取りに行く "**just in time**" 型のコンテキスト戦略を紹介しています！

Claude Code も、`CLAUDE.md` のような事前コンテキストと、glob / grep などで必要なファイルを実行時に探索する方式を組み合わせていますよね📝

## 3. ハーネスエンジニアリング (2026年初頭〜)

![](https://static.zenn.studio/user-upload/1d3405a048db-20260520.png)

AIを単発の「テキスト生成器」から、現実世界のタスクを実行する「エージェント・システム」へと昇華させる現在の最前線です。  
「ハーネス（Harness）」は元々は馬具を意味し、馬を制御して正しい方向に向かわせる仕組みのこと。  
AIに置き換えると、エージェントの行動を方向づける仕組みを設計することがハーネス・エンジニアリングと呼ばれます📝  
つまり、**AIが「安全に動けるように制御する仕組み・環境」を作る技術** です📝

### 主な手法・アプローチ📝

代表的な手法・アプローチは、こんな感じです👀

* エージェント向けのドキュメント整備: Rulesや、Hooks, Skillsなどのエージェントを制御したり、手続きを整理したドキュメント整備
* ツール利用（Function Calling）: AIにWeb検索、APIの実行、データベースの書き換え、コードの実行などを許可し、自律的に操作させます。
* マルチエージェントシステム: 「コードを書くAI」「レビューするAI」「テストするAI」など、それぞれ違う役割を持ったAI同士を連携させます。
  + Agentic Workflows: AI自身に「計画→実行→反省」のループを回させる。
* LLMOpsとガードレール: AIの出力を監視し、不適切な発言を防ぐフィルター（ガードレール）や、プロンプトのバージョン管理、継続的な精度評価（CI/CD）を行います。
  + Evaluation (評価): AIの回答が正しいかを自動テストする。
  + Guardrails: 不適切な発言や情報の漏洩を機械的に阻止する。

### 進化のポイント

AIが人間の代わりに「思考」するだけでなく、**システムを通じて実際に「行動（実行）」できる**ようになりました。

Claude Code みたいなエージェント型ツールの登場で前景化したこの段階では、  
**「モデルを取り囲む足場・環境 (ハーネス) 全体を設計する」** ことが主題になります🥺

### Feedforward Controls と Feedback Controls🐱

Martin Fowler氏 の記事では、ハーネスを **「モデル以外のエージェントを構成するもの」** と捉えつつ、特にコーディングエージェント利用者にとっては、大きく **2種類の制御** から構成されると整理しています📝

* **Feedforward controls (前方制御)**: AIが動く前に方向づけるガイド
  + 例: `AGENTS.md`、Skills、設計原則、How-toドキュメント
* **Feedback controls (後方制御)**: AIが動いた後に結果を観測して自己修正させるセンサー
  + 例: テスト、lint、型チェック、静的解析、ログ、AIレビュー

「動く前のガイド」と「動いた後のセンサー」、両方揃って初めてハーネスとして機能する、ということですね📝

<https://martinfowler.com/articles/harness-engineering.html>

### AI駆動開発における「自己修正ループ」📝

ポイントは、AIに「ちゃんとやって」と頼むのではなく、こういう **失敗を検知・修正できる環境** を用意することです💪🐱🔥

```
AIに実装依頼
  ↓
既存コードを探索
  ↓
実装
  ↓
npm run lint
npm run typecheck
npm run test
  ↓
失敗したらAIがログを読んで修正
  ↓
PR作成
  ↓
AIレビュー + 人間レビュー
  ↓
CI通過後にマージ
```

Red Hat の記事でも、AI支援開発で安定した結果を得るには、自由文のチケットをAIに投げるのではなく、**リポジトリ影響マップ、実ファイルパス、既存シンボル、受け入れ条件、テスト要件を含む構造化タスク** に落とし込むことが有効、と説明されています🐱

<https://developers.redhat.com/articles/2026/04/07/harness-engineering-structured-workflows-ai-assisted-development>

ここまで来ると、プロンプトやコンテキストは、ハーネスの中の一部品にすぎないということがわかります📝  
**システム全体のアーキテクチャ設計** に近いものになってきている、というのが面白いところです📝

## 3層を6観点で比較する📝

ここまでで各段階の特徴を見てきましたが、改めて並べてみると、こんな比較表になります🐱

| 観点 | プロンプトエンジニアリング | コンテキストエンジニアリング | ハーネスエンジニアリング |
| --- | --- | --- | --- |
| **時期** | 2022〜 | 2025年中頃〜 | 2026年初頭〜 |
| **主対象** | テキスト指示 | コンテキストウィンドウへの入力全体 | LLMを取り囲むシステム全体 |
| **典型タスク** | 文言調整、Few-shot、CoT | RAG、メモリ、トークン予算配分、ツール選定 | エージェントループ、評価、観測性、サブエージェント、リカバリ |
| **失敗時の対処** | プロンプトを書き直す | 入れる情報を再設計する | システム・ハーネスを修正する |
| **求められるスキル** | 言語表現力 | 情報アーキテクチャ | 分散システム / SRE的素養 |
| **包含関係** | 最小 | プロンプトを内包 | コンテキストエンジニアリングを内包 |
| **人間の関与** | 手書きの文字列 | コードが組み立てる情報セット | **モデルが自律的に動く環境そのもの** |

「失敗したときに、どの層を直すか」という発想で見ると、3層の違いがすごく分かりやすいですよね💪🐱🔥

## AI駆動開発における4つの成熟レベル

ここで、もう一歩踏み込んで、AI駆動開発の「成熟度」を4段階に整理してみます📝

最初は「AIにコードを書かせる」ことに見えますが、成熟していくと、エンジニアの仕事は **アーキテクト的な役割** にどんどん変わっていくと思います👀

| レベル | 開発者の役割 | AIの役割 | 成果物 |
| --- | --- | --- | --- |
| **Lv1: プロンプト駆動** | 指示を書く | コード断片を生成する | 単発のコード、記事、SQLなど |
| **Lv2: コンテキスト駆動** | 仕様・設計・既存コードを整理する | プロジェクト文脈を踏まえて実装する | 既存システムに合う変更 |
| **Lv3: ハーネス駆動** | 仕様、制約、検証、CI/CD、レビュー導線を設計する | 実装・修正・テスト・PR作成を反復する | 継続的に品質管理された変更 |
| **Lv4: エージェントチーム駆動** | 複数AIの役割分担と監督をする | 調査・設計・実装・レビュー・テストを分担する | 小さな開発組織のようなワークフロー |

OpenAIの Codex は、コードベースを読み、ファイル編集やコマンド実行、テスト・lint・型チェックの実行、PR提案まで行えるソフトウェアエンジニアリングエージェントとして説明されています📝

<https://openai.com/index/introducing-codex/>

Claude Code でも、サブエージェントは独自のコンテキストウィンドウ・システムプロンプト・ツール権限を持っていて、探索・計画・実装などを分離するために使えます🐱

<https://docs.anthropic.com/ja/docs/claude-code/sub-agents>

つまり、単一の巨大な会話で全部やるのではなく、**タスクごとに文脈を分離して制御する設計** が、すでに普通になってきているということですね💪🐱🔥  
Lv4は、この仕組みを使いこなしている段階に相当します🌟

## 進化のドライバーは「使い方の変化」📝

ここで、ちょっと一歩引いて「なんでこの順番で進化したのか?」を考えてみます🐱

![](https://static.zenn.studio/user-upload/23ea8262928d-20260520.png)

3段階の移行は、LLMの使い方そのものが変化した結果として整理できます👀

1. **単発Q&A** → プロンプトで足りる
2. **RAG・複数ターン対話** → コンテキスト管理が支配的になる
3. **長期実行エージェント・自律タスク** → ハーネス設計が第一級の関心事になる

特に3番目の「長期実行エージェント・自律タスク」がポイントで、エージェントループは非決定的だし、ループ内の「プログラマー」自身がLLMになる、というすごい変化が起きています📝

つまり「**単に強いモデルを追うのではなく、システムアーキテクチャに注力する**」へと、業界の重心が移ってきた、ということですね💪🐱🔥

## 同じ問題を「どの層で吸収するか」を意識する

![](https://static.zenn.studio/user-upload/184952398262-20260520.png)

実務では、**3層のどこで問題を吸収するか** という設計判断が、品質を分ける要素になります💪🐱🔥  
ざっくり、こんな整理になります📝

* **プロンプト層で吸収すべきもの**: 出力形式、トーン、ロール、明示的な制約条件など、入力テキストで一意に表現できる事項
* **コンテキスト層で吸収すべきもの**: ドメイン知識(RAG)、ユーザー固有の状態(メモリ)、直近の会話履歴、ツール選定など、**情報の取捨選択** で解ける事項
* **ハーネス層で吸収すべきもの**: 失敗時のリトライ戦略、サブエージェントへの分業、観測性、永続化、サンドボックス、複数モデルの切り替えなど、**一度直したら恒久的に効く** インフラ的事項

同じ問題でも、どの層で解決するかによって、運用コスト・再現性・スケーラビリティが大きく変わってきます。

たとえば「LLMが時々、間違った形式で返してくる」という問題に対してなら、

* プロンプトを書き直す → 再現性が低い
* Few-shot例を増やす → コンテキスト層で対処
* 出力をパースしてリトライするループを組む → ハーネス層
* 構造化出力APIに切り替える → ハーネス層

こうやって並べてみると、コストも効果の持続性もまったく違うのが、すぐ分かります！

## 比喩で覚える：脳 → 知識 → 手足

3層の役割を次のように考えると、3層の意味がめちゃくちゃ直感的になるかなと思います📝

![](https://static.zenn.studio/user-upload/be5cf64fdedf-20260520.png)

* **プロンプト** は、AIの **脳の動かし方** を最適化する🧠
* **コンテキスト** は、AIに **正しい知識と記憶** を与える📚
* **ハーネス** は、AIに **手足 (ツール) と社会性 (システム連携・安全性)** を与える🤖

この3つが揃うことで、AIは単なるアシスタントを卒業して、自らソフトウェアを構築し、成長していく **AI駆動開発の強力なエンジン** として機能するようになってきました💪🐱🔥

## まとめ🐱

| 段階 | キーワード | 一言で言うと |
| --- | --- | --- |
| プロンプトエンジニアリング | AIの頭の使い方・動き方の設計 | モデルに **何と言うか** |
| コンテキストエンジニアリング | 情報環境の設計 | モデルに **何を見せるか** |
| ハーネスエンジニアリング | システムの設計 | モデルを **どう動かすか** |

ハーネスエンジニアリング時代のAI駆動開発で重要なのは、**「AIがいい仕事をせざるを得ない構造を作る人」になる**ということだと思います。

これからの開発者の価値は、コードを1行ずつ書く力だけじゃなくて、  
仕様・文脈・制約・テスト・レビュー・CI/CD を組み合わせて、**AIエージェントを安全に動かす 設計力** に移っていきます！

最後まで読んでいただき、ありがとうございました🐱  
ぜひ、皆さんもAI駆動開発の3段階を意識して、AI時代の新しい開発体験を作っていきましょう💪🐱🔥
