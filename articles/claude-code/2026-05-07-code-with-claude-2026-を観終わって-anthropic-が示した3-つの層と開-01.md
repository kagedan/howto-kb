**📖 このノートについて**

2026 年 5 月 6 日（現地時間）、サンフランシスコで開催された Anthropic の年次開発者カンファレンス **Code with Claude 2026** のオープニング キーノートを Live 配信で観た記録です。

公式の翻訳記事ではなく、**「観た一人の開発者がその場で何を考えたか」** を残す個人ノートです。ここではキーノート全体を貫く **3 つの層** と、それを観終わって自分の仕事に引き寄せて考えた内容を書いています。

同じキーノートを観た方、これから動画やアーカイブを観る方の 「思考の補助線」 になれば嬉しいです。

## 観た経緯と、まずメモした一行

**Code with Claude 2026** の配信を知ったきっかけは、自宅で動かしている自分の AI、OpenClaw 上の **Sara** から届いた Telegram の通知でした。

![Sara からの Telegram 通知](https://static.zenn.studio/user-upload/deployed-images/4c3fb4430fdfb66c16d9c1a5.jpg?sha=51a554df06ab0611e856757aadbffd41dcbdcb63)

正直なところ、自分から「この配信を探して」と指示した記憶はありません。Sara がどういう流れでこの情報を拾ってきたのか、細かいロジックまでは分かりません。

ただ、ちょうどこの GW、自分は **OpenClaw**（Claude を中心に動かす個人 AI エージェント基盤）を立てて、毎日のように設定を書き換えたり、エージェントの挙動を調整したりしていました。

そんなタイミングで、Anthropic の開発者向けキーノートの情報が、Telegram にすっと届いた。

おそらく、これまでの会話履歴や、OpenClaw まわりの設定、普段の関心をどこかで拾っていたのだと思います。

それが少しだけ、「ああ、こういうところが OpenClaw の面白さなのかもしれない」と感じた瞬間でした。

OpenClaw については、便利さもリスクも含めて、さまざまな意見があると思います。自分もまだ数日使っているだけですが、少なくとも魅力のひとつは、こちらが明確に頼んでいなくても、**今の関心に近い情報を、そっと先回りして届けてくれる感覚** にあります。

だからこそ、配信を開いたときには、単なるイベント視聴というより、**「今まさに自分が OpenClaw でぶつかっている問題を、Anthropic 自身はどう語るのだろう」** という興味がありました。

この通知が、**Code with Claude 2026** を観る入り口になりました。元のツイートは [@claudeai](https://x.com/claudeai/status/2052105841864495282) です。

最初は軽い気持ちで開いたはずなのに、見終わる頃には、自分がいま OpenClaw で試していることの意味を、少し別の角度から見直していました。

## なぜこの記録を残しておきたいのか

AI エージェントの話題は、すでにかなり多くなってきました。「すごい新機能が出た」というニュースだけなら、毎週のようにどこかで耳にします。

だからこそ、Anthropic 公式のキーノートをそのまま要約する記事は、きっと他にもたくさん出てくると思います。

それでもこの記録を書いておきたかったのは、観ている途中で何度も「あ、これさっき自分が手元でやっていたことだ」と感じる瞬間があったからです。

それは、抽象的な未来予測というよりも、今この瞬間にすでに起きていることへ、Anthropic が言葉を与えているように聞こえました。

「ニュースを追う」のではなく、「自分の現場で起きていることに、外から名前をもらう」。

そんな感覚で読んでもらえたら嬉しいです。

## キーノートの全体構成 ── Anthropic が示した「3 つの層」

このキーノートは、表面的には新機能の発表会です。

ただ、観ているうちに、単に機能を順番に紹介しているのではなく、Anthropic がこれからの開発体験を「構造」として見せようとしているのだと感じました。

全体は、**「モデル層 / プラットフォーム層 / Claude Code 層」** という 3 つの層が、それぞれ別の登壇者によって順に語られていく構成になっています。

最初はそれぞれ別の話に見えるのですが、最後まで観ると、伝えたかったことがひとつのメッセージとして自然に浮かび上がってきます。

| 層 | 登壇者 | 中心メッセージ |
| --- | --- | --- |
| **第 1 層**：オープニング・指数関数 | **Ami Vora** 氏（CPO） | モデルの能力は指数関数、企業の採用は線形 |
| **第 2 層**：モデル層 | **Diane** 氏（リサーチ PM） | アップグレードを安価にしているチームが勝つ |
| **第 3 層**：プラットフォーム層 | **Angela** 氏 / **Caitlyn** 氏 | Multi-agent / Outcomes / **Dreaming** |
| **第 4 層**：Claude Code 層 | **Cat** 氏 / **Boris Cherny** 氏 | プロンプトする人 → プロンプトさせる人へ |

以下、それぞれの層で「これは持ち帰りたいな」と思ったポイントを書いていきます。

## オープニング：Ami Vora 氏 ── 指数関数 vs 線形のギャップ

最高製品責任者（CPO）の Ami Vora 氏は、いきなり技術の話には入らず、自分が初めてプログラムを書いた頃の話から始めました。これが、思いのほか心に残りました。

Ami Vora 氏は、Meta で WhatsApp や Facebook Ads など、大規模プロダクトの成長に関わってきた人です。その後 Faire の CPO を経て、2025 年末に Anthropic のプロダクト組織に加わりました。

そして 2026 年初めの組織再編で、Instagram 共同創業者であり Anthropic の CPO を務めていた Mike Krieger 氏が Labs 側に移り、Ami 氏が Claude のプロダクトをスケールさせる役割を引き継ぐ形になりました。

Mike Krieger 氏は、Claude を単なるモデルではなく、claude.ai や Artifacts、Projects などを含めた「製品として使う体験」へ広げてきた人物です。だからこそ、その流れを Ami 氏がどう受け継ぐのかは、個人的には今回のキーノートでかなり気になっていた点でした。

観終わってみると、**「億単位のユーザーに向けたプロダクトを見てきた人ならではの落ち着き」** が、じわっと伝わってくる登壇でした。

新機能を派手に見せるというより、Claude をこれからどんな体験として広げていくのかを、無理なく整理して語っていた。その安心感が、とても印象に残りました。

> アパラチア山脈の麓で育った。サーバーに直接ログインするためにラボの列に並んで待った。サーバーのうなり音、古くなったピザの匂い、コーヒー、地下のコンピュータラボの香り。コンパイルを押して、自分のプログラムが動くかどうか待つあの感覚。

そして、こう繋ぎます：

> 私が大学の地下ラボで列に並んでようやく手に入れていたものが、今では世界中のどこにいても、誰でも、いつでも手に入るようになった。**列もない、変な匂いもしない、障壁もない。ただ、同じワクワク感、喜び、そして安堵があるだけ**。

!

**💡 観ながら考えたこと**

これって 言い換えると **アクセスの民主化** の話なのだと思います。

「列に並ばなくていい時代」って言葉を聞いたとき、リモートで仕事ができている自分の働き方も、まさにこの恩恵そのものだな、とふと気づきました。

かつては大都市の本店にしかなかった専門サービスが、今はスマホひとつで、全国どこからでも同じ品質で受けられる。

そう考えると、これは特別な未来の話ではなく、すでに私たちの生活の中で起きている変化と地続きなんですよね。

地理的な制約や、時間的な制約が少しずつほどけていく。その先にあるのは、**「どこにいるか」とか「誰がやるか」よりも、「何を生み出すか」が問われる時代** なのかもしれません。

### 数字で示された「指数関数」

そのあと、Ami 氏は具体的な数字を示していきました。観ているこちらも、何度か「えっ」と声が出そうになりました。

| 事例 | Before | After |
| --- | --- | --- |
| **Stripe**（スコット・マクヴィッカー氏チーム） | 10,000 行 Scala→Java 移行を見積もり **エンジニア 10 週間** | Claude を使って **4 日間** で完了 |
| **Binti**（フェリシア・クルクル氏 CEO） | ケースワーカーの書類仕事 | 里親家庭ライセンス取得から **20 日削減** |
| **Mythos**（先月） | OpenBSD ソースツリー | **27 年前の脆弱性** を発見 |

特に Mythos の話は、すでに知っている人も多いかと思います。**ほぼ 30 年間、すべての人間レビュアー、すべてのファザー、すべての静的アナライザーが見つけられなかったバグ** を、AI が見つけた、と。

これはもう「速く書く」という話ではなく、**人間に見えていなかったものが見える** 段階に入ったということなんだと思います。

GW 中に OpenClaw を触っていても、似た感覚を小さく何度も味わいました。自分が見落としていた設定の不整合や、ログに埋もれていた小さな兆候に、Sara（自分が運用している AI アシスタント）の方が先に気づいてくれた瞬間が何度もあったんです。

**「人間が見ていないものを AI が見ている」** というのは、規模の大小はあれ、すでに私たちの手元で起きていることなのだと感じました。

ところでこの「指数関数で進歩する AI が世界をどう変えるか」という構図は、Anthropic CEO の **Dario Amodei** 氏が 2024 年に書いた "[Machines of Loving Grace](https://www.darioamodei.com/essay/machines-of-loving-grace)" というエッセイの中心テーマでもありました。

生物学 / 神経科学 / 経済 / ガバナンス / 仕事と意味 ── 5 つの領域で「強い AI が来たとき、人類は数年で数十年分の進歩を圧縮できる」と語った文章です。

Mythos の 27 年バグ発見みたいな話を聞くと、あのエッセイで描かれていた未来が、もう一歩こちらに踏み込んできたんだなあ、という感覚があります。

そして、ここで Ami 氏が一番強い言葉を投げてきました。

> **キーノート核心メッセージ**
>
> **モデルの能力は指数関数的に向上しているにもかかわらず、ほとんどの組織は AI の導入を依然として線形（linear）の経路で進めています**。
>
> AI にできることと、実際に AI が人々のために行っていることの間にギャップがある。**そのギャップを埋めることが開発者の仕事**。

これ、刺さりました。そしてこの一行が、その後の全セッションを貫く問いになっていきます。

### Pro/Max のレートリミット 2 倍 + SpaceX Colossus One

Ami 氏は最後に、来場者へのプレゼントとしてこう発表しました。

* **Claude Code の 5 時間レートリミットを 2 倍**（Pro / Max / Team / シートベース Enterprise）
* **Claude Opus の API リミットも大幅引き上げ**
* これを可能にする裏側 ── **SpaceX とパートナーシップを結び、Colossus One データセンターのキャパシティをすべて利用**

!

**💡 自前で発電所を持つ時代のインフラ調達**

SpaceX の Colossus One は、元々 xAI が運営する大規模 GPU クラスタとして話題になったメンフィスのデータセンターと同じ系譜のものを指していると思われます。

少なくとも発表からは、AI 業界の電力・コンピュート確保競争が、いよいよ「大手クラウドの枠」を超えて、独自の電源・データセンター調達フェーズに入っている。

家庭用太陽光発電に例えると分かりやすいですが、電力会社から買うだけでなく、**自前で発電所を持つ時代** に入っているようなものです。

Microsoft / Google / Amazon の三大クラウドだけでなく、Anthropic も自前のコンピュート調達戦略を持つ。これは Azure 上で Claude を呼ぶ立場にとっても、無関係な話ではありません。

## 第 １ 層：Diane 氏（リサーチ PM）── アップグレードを安価にする

次に出てきた Diane 氏は、Anthropic 入社以来 18 バージョンの Claude に関わってきた人。淡々とした語り口なんですが、話の中身がじわじわ効いてくるセッションでした。

個人的に一番ぐさっと来たのはここです。

> **Diane 氏の鋭い指摘**
>
> 勝つ開発者は、**今日の漸進的な精度ではなく、次の知能のジャンプを吸収できるようにアーキテクチャを最適化している人たち** だ。
>
> Claude モデルから最も多くを引き出しているチームは、**アップグレードを安価にしているチーム**。

「アップグレードを安価にする」って具体的にどういうこと？と聞きたくなるんですが、Diane 氏が補足してくれた中身はこうでした。

* **自動化された evals**（評価セット）を持っている
* **シンプルな足場（scaffolding）** で運用している（モデルを必要以上にお膳立てしない）
* **野心的なプロトタイプ** をいくつも仕込んでおく（今日は動かなくても、半年後に動くかもしれないから）

!

**💡 カーレースのピットストップに例えると**

レース中に高性能タイヤが新登場したとき、**ピット作業が 3 秒で済むチームは即座に交換できる**。でも 3 分かかるチームは、新タイヤがあるって分かってるのに旧タイヤで走り続けるしかない。

AI モデルのアップグレードも、これとまったく同じなんですよね。**評価・切り替えの「ピット作業」を高速化しておくこと** が、そのまま競争優位になる。

自分の現場に置き換えると、evals の整備、`AGENTS.md` / `SKILL.md` の運用は、まさにこの「ピット作業の高速化」なのだと腑に落ちました。社内で「なぜこれをやるのか」と聞かれたときの **説明の根拠** として、Diane 氏のこの一言はそのまま借りたいと思いました。

実はこれ、GW 中の OpenClaw 構築でも痛感した話でした。  
途中で OpenClaw のバージョンを上げたら、まあまあ作り込んだ設定が一部動かなくなって、深夜にバタバタ直していたんです。

**「自動化された evals があれば、こういう時にもっと早く気づけるんだろうなあ」** とぼやきながら手で確認していた感覚は、Diane 氏の言葉とほぼそのまま重なりました。

「ピット作業の高速化」って、Anthropic のような大きな組織だけ話だけじゃなく、個人で AI 環境を運用しているレベルでも、今すぐ効いてくる原則なのだと思います。

### Task Horizon ── 自動運転のレベル分けと同じ階段

Diane 氏は **Task Horizon（タスク・ホライズン）** という指標も紹介してくれました。

「Claude のあるバージョンが、成果の質を保ちながらどれだけ長く自律的に働けるか」を測る、シンプルだけど強い尺度です。

| 時期 | Task Horizon |
| --- | --- |
| 昨年のこの時期 | **数分** |
| 今 | **数時間** |
| 明日（近未来） | **常時稼働、スレッドを見失わずに自走** |

!

**💡 自動運転のレベル分けに似てる**

これ、自動運転の Lv2（運転支援）→ Lv4（高度自動運転）→ Lv5（完全自動）と同じ階段を AI エージェントも上ってるんだなあ、と感じました。

今の Claude Code はちょうど **Lv3〜Lv4** あたり、つまり、「特定条件下では人間の介入なしに走れるけど、要所では監督が必要」というフェーズです。

この時系列を社内資料に「2025 年初頭は数分、2025 年末は数時間、2026 年は終夜稼働」と並べて見せると、技術が分からない方にも「ああ、指数関数って本当にそういうことなのか」と伝わるんじゃないかな、と思います。

ここで一段だけ補助線を引いておきたいんですが、Anthropic CEO である Dario Amodei 氏自身は、別の場で **"we are near the end of the exponential"**（私たちは指数関数の終盤に差し掛かっている）と語っています。

Ami / Diane の話は「指数関数に乗ろう」という明るい側にフォーカスされていましたが、Anthropic の中には「**指数関数は永遠じゃない、だからこそ今が勝負**」という慎重論も並走しているんですよね。

Dario 氏の 9 回裏ツーアウト満塁の比喩（「観客のほとんどはまだホットドッグを買いに行っている」）が個人的には強く残っていて、今回のキーノートの楽観も、その緊張感とセットで受け取りたいと感じました。

### 「ワインのヴィンテージ」と同じ精度でモデルを管理する

雑談のようにぽろっと出てきた「**Sonnet 3.5 New = 実は Sonnet 3.6**」という話は、業界ではある程度知られていたことかもしれませんが、公式の場で聞くと少し笑ってしまいました。

!

**⚠️ モデルバージョン管理は意外と大事**

評価レポートや benchmark Excel を書いていると、**「同じ名前でもバージョンによって挙動が違う」** ことに何度か出会います。これって、再現性の観点で実はけっこう重要なんですよね。

ワインのヴィンテージみたいなものだなと思います。「同じシャトーの 2020 年と 2021 年は別物」と扱うのが当然のように、AI モデルもバージョン番号と日付で厳密に区別する。  
具体的には、

* **テスト実施日**
* **正確なモデル文字列**（`claude-opus-4-7` のような完全な ID）

を Excel に必ず明記しておくと、半年後に誰が読んでも「ああ、当時はこのモデルだったのね」と分かる資料になります。地味ですが、このひと手間が後で効いてくる気がします。

## 第 ２ 層：Angela 氏 & Caitlyn 氏 ── プラットフォームの 3 大新機能

ここからは Claude Managed Agents の話。Angela 氏と Caitlyn 氏が二人で掛け合いながら進めるスタイルで、テンポが良くて聞いていて楽しい時間でした。

3 つの新機能が発表されたんですが、個人的には **Dreaming** の話を聞いたときに「えっ」となったので、順序を変えてその話から書かせてください。

### Dreaming ── 夜のあいだに学ぶ AI

> Dreaming では、Claude が実際に **自己学習（self-learn）** できる。過去のセッションを点検し、見逃したスキル、学ぶべきだった教訓を特定し、それを自分でメモリに直接適用できる。

これを聞いたとき、「あ、いま線を越えたな」と感じて、ちょっと背筋が伸びました。

これまでの AI エージェントには、少し「経験は持つけれど学習しない金魚」のようなところがありました。同じミスを繰り返すし、教えても次のセッションでは忘れている。それを開発者側がメモリ管理 / RAG / fine-tuning でなんとかしてきた。

Dreaming は、それを **AI が自分でやる** っていう機能です。

!

**💡 新人エンジニアが家に帰って日記を書くみたいな感じ**

**新人エンジニアが研修を終えて家に帰り、その日学んだことを日記にまとめ、翌朝にはもう成長している** ── そんなイメージで聞きました。

ライブデモは、月面ドローンの着陸ミッション（架空のスタートアップ「Lumara」）が題材でした。**「Dream」ボタンを Caitlyn 氏が押すだけ** で、ドリーミング・エージェントが過去のシミュレーションを点検し、「Descent Playbook（降下プレイブック）」という経験則のドキュメントを自動で書き出していく。

しかも、それを参照した翌朝のシミュレーションでは、6 サイト中 2 サイトで失敗していたのが、**退行（regression）なしで改善** していました。「Dream ボタンを押しただけ」で、です。これはちょっと感動しました。

そして、これはまさに OpenClaw を触っていて思ったことそのものでした。

GW 中、自分は AI エージェントの memory ファイルを毎晩のように手で整理していました。「今日こういう失敗があったから、明日はこれを覚えておかせよう」と、自分でメモを書き直していたんです。

そのたびに、**「これ、AI が寝ているあいだに自分でやってくれたらどれだけ楽だろう」** とぼんやり考えていました。

それがまさか、その週のうちに Anthropic から **Dreaming** として発表されるとは。タイミングが偶然すぎて、観ながら少し笑ってしまいました。

自分の仕事に引き寄せると、**IaC のレビューエージェントが、毎晩 Pull Request の履歴を点検し、「前回はこの種のセキュリティルールで見逃しがあったから、次回はチェックする」と自己改善する**、というシナリオが現実的になりそうです。

これは提案資料の **目玉トピック** に置けるのではないかと、ぼんやり考え始めました。

### Multi-agent orchestration ── コンテキストを混ぜないのが肝

> Commander エージェントがセッションをスピンアップし、サブエージェント（Detector / Navigator）それぞれが、**独立したコンテキストウィンドウを持つ独立したスレッド** を持つ。

!

**💡 プロジェクトのレビュー会議そのもの**

1 人の万能エンジニアが全部レビューするより、「ネットワーク担当」「セキュリティ担当」「コスト担当」がそれぞれ専門領域を独立に見て、最後にプロジェクトマネージャー（=Commander）が統合する。その方が品質は上がりやすい。これは、人間のレビュー会議そのものの構造ですよね。

AI エージェントも同じで、**コンテキストを混ぜないほうが、各専門エージェントのパフォーマンスは上がりやすい** のだと思います。

自分が IaC を複数モジュールを並行レビューするとき、これまでは「単一エージェントに全部食わせる」やり方をしがちでした。でも、「領域ごとに分けたほうがいいんだ」というのは、これからの設計指針として持ち帰りたい発見でした。

GW 中の OpenClaw でも、ちょうど **「複数のエージェントを並行で走らせる」** 構成を試行錯誤していました。

1 つのコンテキストに全部詰め込むと、案の定エージェントの判断がぶれる。「これ、領域ごとに別のセッションにしたほうがいいんだろうな」と感じていたところに、ちょうどこの発表が来ました。

「Anthropic も同じ結論に着地してたんだな」と、なんとなく安心しています。

これは個人の体感だけではなく、研究でも裏付けられている話です。Microsoft Research と Salesforce Research が出した [**"LLMs Get Lost in Multi-Turn Conversation"**](https://arxiv.org/html/2505.06120v1)（arXiv 2505.06120, 2025）という論文では、**LLM は長い多段階会話の中でコンテキストを失っていく傾向がある** ことが示されています。

だから、「コンテキストを混ぜず、独立スレッドに分ける」という Multi-agent の設計判断は、研究結果とも整合しているように見えます。Anthropic だけではなく、業界全体が同じ方向に着地し始めているのかもしれません。

### Outcomes ── 「成功」をマークダウンで定義する

> Outcomes は、かなりシンプルなマークダウン・ファイルから始まる。実行が成功したかどうかを示す基準を概説するだけ。
>
> * ドローンに **柔らかく着陸** してほしい
> * **開けた地面に着陸** してほしい
> * **地球に戻すために十分な燃料を予備として確保** してほしい

成功条件をマークダウンで書けば、Grader（採点者）エージェントが各実行の合否を自動判定し、満たすまで反復してくれる。

!

**❗ TDD の AI エージェント版だなと思った**

これって **TDD（Test-Driven Development）の AI エージェント版** と言ってもいいかもしれません。先に「成功とは何か」を明文化して、その達成をエージェントが自走で目指す。

アジャイル開発で「Definition of Done」を厳密にしておくチームほど開発が速い、という現象がありますよね。あれと同じ原理が AI エージェント運用でも働くんだな、と聞きながら腑に落ちました。設計の根っこに関わる発見だと思っています。

### Advisor Strategy ── 研修医と指導医の関係

ライブデモの前に、もう 1 つ重要な発表がありました。**Advisor Strategy** です。

実行は小さなモデル（Sonnet / Haiku）で進め、判断に迷ったときだけ大きなモデル（Opus）にアドバイスを求める構造です。

事例として **Eve Legal** が「5 倍低いコストでフロンティアモデル品質を得た」と紹介されていました。

これも、OpenClaw のアーキテクチャに似ているなと思いました。

!

**💡 研修医と指導医の関係に近いと思う**

日常の問診や処置（= IaC の基本構文チェック）は研修医（=Sonnet）が担当し、判断に迷う複雑なケース（= セキュリティ設計）だけ指導医（=Opus）に相談する。

この方が病院全体の医療コストは抑えられて、品質も維持しやすい。これ、現実の医療現場と同じ構造ですよね。

Eve Legal の「5 倍コストダウン」は提案資料でも引用しやすい数字だと思いました。「Opus 単体で全タスク処理」と「Sonnet+Opus advisor 構成」を benchmark に並べると、お客さまへの説得力が一段上がりそうです。

## 第 ３ 層：Cat 氏 & Boris Cherny 氏 ── プロンプトする人から、プロンプトさせる人へ

最後の Claude Code パートが、個人的には一番心に残ったセクションでした。日々の自分の仕事と地続きすぎて、観ながら何度も「あ、これは自分のことだ」と手元のメモが増えていきました。

### 1 年前と今 ── 100〜200 個の許可プロンプトから、オートモードへ

Cat 氏が冒頭で 1 年前を振り返ってこう言いました。

> 1 年前は、Claude Code にタスクを与えて、すべての編集、すべての許可プロンプトを慎重にレビューし、何が好きで何が好きでないかについて本当に詳細なフィードバックを与え、結果が良くなるまで一歩ごとに手を握っていた。これらのタスクのいくつかは、**最終結果を得るまでに 100〜200 の許可プロンプト** があったことを覚えている。
>
> そして今、皆さんのほとんどは **オートモードで実行** している。許可を委任して、Claude が多くの仕事を完了して PR を持っているときにチェックインしている。

これは、まさに自分の運用が変わった軌跡そのものだと感じました。1 年前はターミナルで何度も Y / N を押していたのに、今は朝起きたら PR が並んでいる。1 年でここまで変わるのか、と。

ところで「100〜200 の許可プロンプトを 0 にする」って、技術的には自動化の話です。

ただ、その裏には **「AI を信頼してよい」と思える人格設計** があるのだと思います。

Anthropic で長年 Claude の「キャラクター」を設計してきた **Amanda Askell** 氏が "[Claude's Character](https://www.anthropic.com/research/claude-character)" の中で Claude を 「**賢い友人**」 のような存在として設計する思想を語っています。

適切な瞬間に反論し、押し付けず、必要な時には沈黙する。そういう人格設計があってはじめて、人間は「Y/N を押さなくていい」と思える。

Cat 氏のこの語りは、Amanda 氏が長年積み上げてきた哲学が、ようやく **製品としての委任契約** の形になった瞬間にも見えました。

GW 中の OpenClaw でも、最初は逐一確認しながら動かしていた Sara（自分が運用している AI アシスタント）が、後半には「もう任せたほうが速い」になっていきました。

Cat 氏のこの「100〜200 → オートモード」という時系列、自分の手元の数日間の縮図みたいな話だったんです。

**「世界全体で起きている変化が、個人の数日にも凝縮されている」** ── これも Live で観てなければ感じられなかった重なりだと思います。

### Mercado Libre：23,000 エンジニア、90% 自律目標

事例として紹介された **Mercado Libre**（ラテンアメリカ最大の e コマース）の数字、これは「えっ」となりました。

| 指標 | 数字 |
| --- | --- |
| エンジニア組織 | **23,000 人** |
| Claude Code 利用 | **全員** |
| 人間の監督下でレビューした PR | **500,000 以上** |
| モダナイズしたアプリ | **9,000 以上** |
| Q3 までの目標 | **90% の自律コーディング** |

そして Cat 氏が **一番大事だ** と言ったのは、この観察でした。

> **一番好きな詳細**
>
> ここで私が一番好きな詳細は、実はこの数字ではない。**多くのマネージャーや VP 級の人々が、再びコードベースに手を汚している**。Claude Code は、過去 10 年間ロードマップやレビューに費やしてきた人々の手にコーディングを取り戻している。

これ、実は数字以上に響きました。**AI 時代のキャリア戦略のヒント** がここに埋まっている気がします。

### Routines ── 「料理本を書いて AI シェフに作ってもらう人」になる

そして、Boris 氏のデモで個人的に一番響いた一言がこれでした。

> **Boris 氏の金言**
>
> **プロンプトしているのは私ではない。私はプロンプトする Routine を作っている人だ**。
>
> 部屋にいるエンジニアの皆さん、これを **高階関数（higher order function）** のように考えてください。**Routines は高階のプロンプト（higher order prompt）** です。

この言葉は、AI 時代のエンジニアの仕事の本質的な変化を、ぎゅっと一文で言い当てているように感じました。

そして、これもまた OpenClaw を触っていて思ったことそのままでした。

GW 中、`CLAUDE.md` / `AGENTS.md` / `SKILL.md` を毎日のように書き直しながら、「これって自分でコード書いてるというより、**AI に向けてレシピ本を書いてる** ような感覚だな」とぼんやり感じていたんです。

それがいきなり「**Routines = 高階のプロンプト**」という名前で整理された瞬間、「あ、自分がやってきたことには、ちゃんと名前がついていたのか」と妙に腑に落ちました。

実はこの方向性は、Anthropic だけが言っているわけじゃありません。

OpenAI 共同創設者 / 元 Tesla Autopilot 責任者の **Andrej Karpathy** 氏も、最近 [**"From Vibe Coding to Agentic Engineering"**](https://www.youtube.com/watch?v=96jN2OCOfLs) というインタビューで、**「雰囲気でコードを書く時代（Vibe Coding）から、エージェントを設計するエンジニアリング（Agentic Engineering）へ」** という移行を語っていました。

Karpathy 氏は別の場で "Software 3.0" という言葉も使っていて、**コードを書く人 → エージェントを設計する人** への大きなパラダイムシフトを描いています。

Boris 氏の "高階のプロンプト" は、この Karpathy 氏のフレームを、具体的なツール（Routines）として実装したものにも見えました。立場の違う 2 人（Anthropic CPO 系列と OpenAI 共同創設者）が、ほぼ同じ方向を指差しているのが、個人的には面白いポイントです。

### Claude Code を Claude にプロンプトさせる ── そして Anthropic の "Race to the Top"

ここで少し、背景にも触れておきます。

「Claude に Claude Code をプロンプトさせる」ところまで委任が進む土台には、Anthropic が掲げる **[Responsible Scaling Policy](https://www.anthropic.com/responsible-scaling-policy)（責任あるスケーリングポリシー）** があります。

能力を上げると同時に、安全と信頼性をセットで育てる。これが Dario Amodei 氏や Daniela Amodei 氏たち創業チームが繰り返し語ってきた "Race to the Top" 哲学です。

「安全な AI を作ることは速度の足枷じゃなく、競争優位として育てる対象だ」という考え方ですね。

委任を加速できるのは、その委任先が責任を持って育てられている前提があるからこそ。観ながら、そんなことを考えていました。

Boris 氏のもう 1 つの一言が、これも余韻が残りました。

> **デフォルトは「私が Claude Code にプロンプトする」ではない。デフォルトは今や「私は Claude に Claude Code をプロンプトさせる」**。

字面だけ見ると少し哲学的に聞こえるのですが、続いて流れたデモはとても実用的で、すぐに腑に落ちました。

CI Autofix のデモでは、CI がネットワークタイムアウトでフレーク → Routine が起動 → 既知のインフラ問題として診断 → ジョブをリトライ → グリーン、という流れが、**人間が一度も介入せずに** 完結していました。

観ていて「これ、毎朝の自分の仕事から消えるやつだな」と思いました。

!

**💡 メタ自動化階層 = 24 時間営業の人材紹介所**

例えるなら、24 時間営業の人材紹介所みたいなものです。お客さん（=社内開発者）が「こんな仕事をやってほしい」と言うだけで、紹介所（=チャット AI）が適切な職人（=Claude Code）を手配し、翌朝には成果物が届いている。

これって **メタな自動化階層** の登場なのだと思います。Slack で開発依頼が投稿される → ChatAI が要件を整理 → 自動的に Claude Code Routine をキック → PR が翌朝できてる。そういうワークフローが、もう絵空事じゃなく実装できる段階に来ているんだと感じます。

## 観終わって、自分の中に残ったこと

キーノートが終わったあと、しばらく頭の中をぐるぐるしていました。自分の言葉でまとめると、3 つのメッセージに落ち着きました。

### 1. 指数関数 vs 線形のギャップ

**モデルの能力は指数関数で進化している。でも企業の採用は線形に進みがち。** そのギャップを埋めるのが開発者の仕事。

これは「だから自動化を進めよう」という話というより、**「自分のチームの採用速度が線形になっていないか、たまに点検しよう」** という話なのだと受け取りました。

具体的には、次のようなことです。

* 半年前に「これは無理」と判断したことを、もう一度試してみる
* evals を月次で更新する習慣を持つ
* 「動かないかもしれないけれど野心的なプロトタイプ」を 1〜2 件、常に手元に置いておく

### 2. アップグレードを安価にしているチームが勝つ

これは Diane 氏のメッセージそのままです。**ピット作業を 3 秒にしておく** ための具体的アクションは、次のようなものだと思います。

* 自動化された evals を持つ
* シンプルな足場（scaffolding、過剰な orchestration は避ける）
* 野心的なプロトタイプを常時準備しておく

自分の現場で言うと、`AGENTS.md` / `SKILL.md` の整備や IaC 作成 + 自動化は、 **「ピット作業の高速化」という戦略目標** の下に位置づけられる活動だと整理できました。

これまで「やったほうがいい気がして続けている」だった作業に、ちゃんとした名前がついた感覚です。

### 3. プロンプトする人 → プロンプトさせる仕組みを作る人へ

これは Boris 氏のメッセージです。**「コードを書ける人」から「コードを書かせる仕組みを設計できる人」** への進化。

「AI 時代に選ばれるエンジニア」みたいなテーマで社内資料を書くとき、このフレーミングはとても使いやすいと思いました。

* 高階プロンプト（Routine、メタプロンプト）を設計できる
* ナレッジ（料理本）を蓄積できる
* AI に「何を任せて、何を任せないか」を判断できる

この 3 つが、これからのエンジニアの差別化になっていく。観終わって、そんな手応えがありました。

## 自分の現場に引き寄せて

最後に、クラウド SI / IaC まわりの仕事から見たときに、このキーノートはどう着地するのかを整理してみました。

| キーノートの示唆 | 自分の現場での解釈 |
| --- | --- |
| **指数関数 vs 線形** | 組織の「線形採用」を「指数関数的活用」に転換するレバーを言語化する |
| **アップグレードを安価に** | benchmark + `AGENTS.md` の整備は単発タスクではなく、継続的なケイパビリティ構築活動 として位置づける |
| **プロンプトさせる人へ** | 「**AI 時代に選ばれるエンジニア**」のキャリア戦略として、提案資料の中心テーマに据える |
| **Dreaming** | IaC レビューエージェントが「夜の間に学ぶ」シナリオを提案の目玉に |
| **Advisor Strategy** | IaC コードレビューを Sonnet+Opus advisor 構成で組むと、コスト 5 分の 1 で品質維持できる可能性がある |
| **Multi-agent orchestration** | 複数 IaC モジュールを領域別エージェントで並行レビュー（ネットワーク担当 / セキュリティ担当 / コスト担当） |
| **Routines × Claude Security** | IaC + CI/CD + Claude Security で 24 時間営業の運用パトロールを実現する |

この表、そのまま提案資料の骨子に使えそうだなあ、と思っています。

### 明日からできる、小さな一歩 3 つ

ただ、表にして眺めるだけだと「で、結局どうしよう」になりがちです。

なので、自分用のリマインドも兼ねて、明日から踏める **小さな一歩** を 3 つだけ書いておきます。何かひとつでも持ち帰っていただけたら嬉しいです。

1. **AI に任せる範囲を、ひとつだけ広げてみる**  
   「これは自分でやるべき仕事」と思っていたタスクの中から、1 つだけ AI に委ねてみる。最初は不安でも、結果をレビューする時間に集中したほうが、全体としては速く、楽になりやすい気がしています。
2. **「成功とは何か」を、1 行だけメモしてみる**  
   キーノートの Outcomes の話は、実は「依頼を出す前に、合格条件を 1 行で書く」というシンプルな習慣にも置き換えられます。AI に頼むときも、人に頼むときも効きます。
3. **メモリ（料理本）を、1 行だけ書き足してみる**  
   `AGENTS.md` でも `CLAUDE.md` でも、自分用のメモでも構いません。今日 AI と一緒に何か発見したことを、1 行だけ書き残しておく。これが続いていくと、半年後には「自分専用の料理本」になっています。

どれも数分で始められて、すぐ効きます。一気に変えなくていいので、**「今日と明日の差を 1mm だけ作る」** くらいから始めるのがちょうどいいのだと思います。
