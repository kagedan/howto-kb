# ◆ 猫がタイピングしている画面の正体

僕のMacのデスクトップの隅には、レトロゲーム風の小さなオフィスが常駐している。

木の床のタイルマップ、並んだデスク、観葉植物とコーヒーメーカー。デスクではピクセルのロボ猫たちが座ってタイピングしていて、ときどき席を立ち、家具を避けながら部屋を歩き回る。

これは飾りでもゲームでもない。全部、実際に動いているClaude Code(AIコーディングエージェント)のリアルタイムの稼働状況だ。

・デスクでタイピングしている猫 = いま動いているClaude Codeのセッション

・空席のデスク = 止まっている(または終わった)セッション

・部屋を歩き回っている猫 = 登録済みのAIエージェントたち

・左端の「司令デスク」の猫 = ダッシュボードからの指令を実行する専用Claude

![](https://pbs.twimg.com/media/HNQ4I-XbgAA-XnI.jpg)

つまりこの画面は、自分が雇っているAI社員たちの執務室だ。

## ◆ なぜ作ったか ― AI並列運用の本当のボトルネック

Claude Codeを本気で使い始めると、作業は1セッションでは収まらなくなる。

僕の場合、アプリのポートフォリオ運用、複数サイトのSEO、ツール開発が常時並行していて、Claude Codeのセッションを同時に何本も走らせている。さらにコードレビュー担当、ストア審査対応担当といった専門サブエージェントにも仕事を投げる。

すると、AIの実行能力とは別のところで限界が来る。

・投げたタスクが終わったのか、動いているのか、死んでいるのか分からない

・ターミナルのタブを巡回して生存確認するだけで時間が溶ける

・気づいたら1時間前に止まっていたセッションを放置していた

AIエージェント運用のボトルネックは、実行力ではなく可観測性。「いま何が起きているか」を一目で把握する手段が先に尽きる。

## ◆ 発見: Claude Codeは全部ローカルに記録している

解決の糸口は、Claude Codeの仕様そのものにあった。

Claude Codeは「~/.claude」というフォルダに、プロジェクトごとのセッション履歴(JSONL形式)やタスクの状態を全てローカル保存している。

ということは──このフォルダを数秒おきに読んで画面に描くだけで、公式ツールに一切手を入れずに"監視室"が作れる。

構成は驚くほど小さい。

1. 依存パッケージゼロのNode.jsサーバー1本。~/.claude とgitログを読んでJSONを返すだけ。npm install不要

2. 描画用のHTML数枚。フルボード / デスクトップ常駐のコンパクト版 / ピクセルオフィス版

3. Macの.appラッパー。Dockからワンクリックで起動

「稼働中」の判定もシンプルで、セッションファイルの更新時刻が5分以内なら稼働中、とみなすだけ。Claude Codeは動いていれば必ずファイルに追記するから、中身を複雑に解析するよりファイルの更新時刻(mtime)を見る方が正確だ。

## ◆ 見るだけじゃない ―「発令」で仕事を出す

このボード、可視化だけでは終わらせなかった。ダッシュボードから直接、AI社員に仕事を出せる。

1. ボードの入力欄にプロジェクト宛の指示を書く

2. ヘッドレスのClaude Codeが起動する

3. Claude Codeのhooks機能経由で指令を受け取り、実行する

4. 終わったら最終回答をボードに報告して帰ってくる

もう完全に社長と社員の関係だ。

ただし、この種の「AIに自律的に働かせる仕組み」は、雑に作ると危険な装置になる。だから設計段階で守ったルールがある。

・ユーザー入力をシェルやspawn引数に絶対に混ぜない(起動時に渡すのは固定文言のみ、指令本文はファイル経由)

・権限バイパス系のフラグは使わない(実行の安全確認はClaude Code本来の権限モデルに一元化)

・書き込み系APIはトークン必須、同時実行は1本まで、タイムアウトあり

便利な仕組みほど、境界設計が先。ここは順番を間違えてはいけない。

## ◆ AI社員には「名簿」もある

歩き回っている猫たちの正体は、JSONで管理しているエージェント名簿だ。

planner(実装計画)、code-reviewer(コード品質)、security-reviewer(脆弱性チェック)、ASO担当、ストア審査リジェクト対応係……。名前・役割・稼働状態を書くと、名簿ページとピクセルオフィス
