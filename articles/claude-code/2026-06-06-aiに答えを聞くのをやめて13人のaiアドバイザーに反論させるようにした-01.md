## TL;DR

* 建設会社の社長として、AIには「答え」ではなく「反論」を出してもらうようにした。
* 公開情報から13人分の判断軸を整理し、自分の経営判断をレビューするAIアドバイザーとして使っている。
* 効いたのは有名人っぽさではなく、Human-in-the-loop、役割分担、判断ログだった。

建設会社を経営していると、最後に決めるのはいつも自分になる。

採用するか。撤退するか。人に任せるか。新しい事業に張るか。

相談できる人はいる。でも、毎回すべての背景を説明するのは難しい。

そこで最初は、AIに「どう思う？」と聞いていた。

便利だった。けれど、ある時に限界を感じた。AIはきれいに整理してくれる。でも、自分の考えに寄り添いすぎることがある。

社長に必要なのは、気持ちのいい答えではなく、判断の穴を見つけることだ。

だからAIに答えを聞くのをやめた。

代わりに、13人のAIアドバイザーに反論させるようにした。

---

## 1. ひとり社長の判断は、気づかないうちに偏る

自分は18歳から建設業の現場に入り、20年やってきた。今は建設業、eBay、SaaSの3つを同時に回している。

この状態になると、判断の量が増える。

現場の人員配置、外国人労働者向けの学習アプリ、発信戦略、社内の仕組み化。全部が別の問題に見えるが、最後は同じ問いに戻る。

「今、何に集中するか」

この問いは、意外と危ない。

人間は、自分が見たい方向に理由を集める。忙しい時ほど、都合のいい結論に飛びつく。しかも経営者は、誰かが強く止めてくれるとは限らない。

だから、自分の判断には常に別の視点を入れる必要がある。

最初はChatGPTやClaudeに普通に相談していた。  
「この方針どう思う？」  
「この事業を進めるべき？」  
「この記事は出した方がいい？」

返答はだいたい優秀だった。でも、こちらの前提をきれいに補強して、もっともらしい計画に整えてしまうことがある。

つまり、**AIに相談するだけでは、判断の偏りが残る**。

必要だったのは、答えを出すAIではなく、反論するAIだった。

---

## 2. AIの中に「経営会議」を作った

そこで、自分用のAdvisorスキルを作った。

やっていることはシンプルで、公開情報から人物ごとの思考パターンを整理し、特定の判断軸として使う。

大事なので最初に線を引く。

これは本人の代替ではない。本人公認でもない。誰かの人格を作る話でもない。

公開されている本、講演、インタビュー、動画、記事から「何を重視して判断しているか」を整理し、自分の判断をチェックする道具にしている。

代表的にはこう使う。

| アドバイザー | 役割として見る判断軸 | 聞きたい問い |
| --- | --- | --- |
| 堀江貴文 | スピード、時間、即行動 | 考えすぎて止まっていないか |
| 孫正義 | ビジョン、長期、大きな賭け | 10年後も意味があるか |
| ひろゆき | 不要論、固定費、データ | それ本当にやる必要があるか |
| 木下勝寿 | 利益、無収入寿命、売上最小化 | 増やす前に減らせないか |
| Amanda Askell | 誠実さ、迎合リスク、AI設計 | AIがこちらに寄りすぎていないか |

今は13人分ある。ただし、人数が多いことに価値があるわけではない。大事なのは、視点が重複しないことだ。

全員に同じことを聞くと、会議が長くなるだけで判断は進まない。

## 3. 実際の流れ

AI経営会議の流れはこうしている。

ポイントは、最初に自分の仮説を書くこと。

仮説を書かずにAIへ聞くと、AIは一般論を返す。  
仮説を書くと、AIはそこに反論できる。

自分が使っているプロンプトの型は、だいたいこうだ。

```
# advisor-review

## 相談テーマ
Zennで次に出す記事を、AI経営会議の記事にするか判断したい。

## 自分の仮説
面白いが、有名人名に頼った浅い記事に見えるリスクがある。
一方で、Personal OS / AI Advisory Board の流れには乗っている。

## 見てほしい判断軸
- 堀江視点: 今すぐ出すべきか、考えすぎか
- 孫正義視点: 長期ブランドに効くか
- ひろゆき視点: やらない方がいい理由はあるか
- 木下視点: 事業導線として利益に近いか
- Amanda視点: AIが迎合していないか、誠実な表現か

## 出力してほしいもの
1. 各視点からの反論
2. 見落としているリスク
3. 最終判断ではなく、判断材料

## 制約
最後に決めるのは人間。
本人の代替のような表現は使わない。
```

これだけで、返答が変わる。

「いいと思います」ではなく、「その言い方だと危ない」「この実例がないと薄い」「長期ブランドには効くが、タイトルで遊びすぎると信用を落とす」といった反論が出る。

AIの価値は、答えではなく判断材料を増やすことにある。

## 4. 最初は失敗した

最初からうまくいったわけではない。

「ホリエモンならどう言う？」  
「孫さんならどう考える？」  
「ひろゆきなら？」

こう聞くと、表面的な口調に寄りやすい。

それは経営には使いにくい。

欲しいのはキャラクターではない。判断軸だ。

そこで、人物名よりも先に「何を重視するか」を固定した。堀江さんなら速度。孫さんなら長期ビジョン。ひろゆきさんなら不要論。木下さんなら利益構造。Amandaなら誠実さとAIの迎合リスク。

こう整理してから、会議の質が変わった。

もう1つ失敗したのは、全員に毎回聞いていたことだ。

13人に聞くと、情報量は増える。でも判断は遅くなる。

今は、相談テーマごとに3-5人だけ選ぶ。新規事業なら孫さん、堀江さん、木下さん。コスト削減ならひろゆきさん、木下さん。

人間の会議と同じで、必要な人だけ呼ぶ方がいい。

## 5. 業界ベストプラクティスとして効いたもの

この仕組みは、AI活用の基本にかなり近い。

特に効いたのは3つ。

### Human-in-the-loop

AIが決めるのではなく、人間が最後に決める。

AIに自律性を持たせるほど、どこで人間が止めるかを先に決める必要がある。

経営判断では、ここを絶対に外さない。

AIに任せるのは、論点整理、反論、盲点の発見まで。採用、撤退、投資、公開、送信は人間が決める。

### Maker-checker

1つのAIに作らせて、同じAIに確認させると甘くなる。

だから、作る役と見る役を分ける。

自分の仮説を作るのは自分。反論するのはアドバイザー。事実確認は別のAI。最後に人間が決める。

建設現場でも、施工する人と検査する人は分ける。AIも同じだった。

### Decision trail

判断は、結果だけ残しても学べない。

「なぜその時そう考えたか」が残っていないと、あとから学べない。

だから、自分は重要な判断では次の4点を残すようにしている。

| 残すもの | 理由 |
| --- | --- |
| 自分の最初の仮説 | 後から偏りを見つけるため |
| AIからの反論 | 見落としを記録するため |
| 最終判断 | 何を選んだかを残すため |
| 結果 | 次の判断に学習させるため |

これを残すと、AIは単発の相談相手ではなく、判断ログになる。

---

## 6. この仕組みが向いている人、向いていない人

向いているのは、正解がない判断を何度もする人だ。

経営者、PdM、事業責任者、ひとりで複数事業を見ている人には効く。

逆に、単純な答えが欲しいだけなら向いていない。メール整形、関数修正、文章短縮には、普通にAIを使えばいい。

AI経営会議が効くのは、答えが1つではない問いだ。

たとえば、新規事業を今やるか。採用するか、パートナーに任せるか。発信を強めるか、開発に集中するか。

こういう問いは、1つのAIに聞くだけだと危ない。

違う判断軸をぶつけた方がいい。

---

## おわりに

AIを使い始めた頃、自分は「AIに正解を出してもらう」ことを期待していた。

でも、経営で本当に効いたのは逆だった。

AIに正解を出させるのではなく、AIに反論させる。

自分の判断に、別の視点を常駐させる。

最後に決めるのは、あくまで人間。

この線を守ると、AIは社長の代わりではなく、社長の判断を鍛える道具になる。
