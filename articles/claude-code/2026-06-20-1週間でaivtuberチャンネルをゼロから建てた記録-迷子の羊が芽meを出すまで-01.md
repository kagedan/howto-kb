> このチャンネルのテーマは「過去の失敗から、未来の芽を」。

## はじめに — なぜVTuberを「作る側」で始めたのか

VTuberというと「演者として喋る」イメージが強いのですが、自分が今週やっていたのは、ほぼ全部が**裏側の構築**でした。それはそう、私以外チームはいません。アバター、配信オーバーレイ、台本テンプレ、そして編集を止めないための制作パイプライン。要するに「一人で番組を回すための工場」を、AIツールを総動員して1週間で建てた記録です。

チャンネルのコンセプトは、**はぐれ羊×デジタル生命体のVTuber**。

* **静謐**：歴史・経済の「失敗学」を冷静に語る語り部
* **開発**：「それ作れそうじゃない?」と言って実際に手を動かす開発者

この記事は、後者の開発による「作ってみた」総集編です。

## 1. アバター：VRoidからVRM、そしてVSeeFaceまで

最初の山がアバターでした。VRoid Studioで素体を作り、`.vrm` 形式で書き出す。ここまでは早かったのですが、調整に時間が溶けました。

今週だけで `151Mev2.vrm` → `151Mev3.vrm` → `151Me_sheep.vrm` と何度も作り直しています。羊のツノ（`SheepHorn`、左右で各3,962面）を足したり、髪のメッシュ構成を確認したり。マテリアル単位で中身を棚卸しして、どのスロットがどの面数を持っているかをテキストに書き出しながら詰めました。

```
Horn_L | slot0 | SheepHorn | faces=3962
Horn_R | slot0 | SheepHorn | faces=3962
Hair   | slot2 | HAIR_03   | faces=15270   ← 髪の主役メッシュ
```

地味ですが、**「重い・崩れる」原因はだいたいメッシュとマテリアルの構成を見ればわかる**というのが学びでした。なんとなく作り直すより、面数とスロットを一覧化してから手を入れる方が早い。

動かす側は**VSeeFace**を採用。VRM0でエクスポートすれば、Webカメラなしでも音声ベースのリップシンクが回ります。顔出しもトラッキング機材も不要で立ち上がるのは、一人運用ではかなり効きます。

ライセンスも確認済みです。使ったVRoidモデルは**商用利用Allow／作者本人のみ利用可／`.vrm` ファイル自体の再配布は禁止**。YouTube収益化には問題なし、という線引きを最初に確定させておきました。ここを曖昧にしたまま進めると後で芽が枯れるので。

## 2. 配信オーバーレイ：OBS × わんコメ × HTML/CSS

次に配信画面まわり。OBSのブラウザソース向けに、時計・コメント欄・ロゴ・背景をHTML/CSSで自作しました。コメント欄は **わんコメ（OneComme）様** と連携して、Twitchのコメントをそのまま表示します。

ハマったのは連携方式です。わんコメの旧WebSocket APIは非公開化されていたので、公式テンプレート用SDK（`onesdk.js`）を読み込んで `comments` イベントを購読する方式に変更。さらに、わんコメ未起動などでSDKが読めないときはローカルWebSocket（`ws://127.0.0.1:11180/sub`）へ自動フォールバックするようにしました。

```
# シーン全体（時計＋コメント＋ロゴ＋背景）をまとめて出す場合
151me-scene.html?source=onecomme&service=twitch
# ブラウザソースは 1280 x 720 推奨
```

コメントのアイコン・名前・Twitchバッジ（最大3個）・サービスタグ・本文（Twitch絵文字の `<img>` も）まで描画。\*\*「公式の推奨方式に寄せつつ、壊れたときの退路を一本用意しておく」\*\*のが、配信中に事故らないコツでした。

## 3. コンテンツ設計：２つの顔と30本のネタ帳

工場には設計図が要ります。今週は番組の骨格も固めました。

* **キャラクターバイブル**：性格比（冷静80 / 皮肉15 / 共感5）、口調、NGライン、台詞（「それなんかできそうじゃない?」「記録を開きます／閉じます」）
* **30本のトピックバンク**：A〜Eの5クラスタ（バブルと崩壊／通貨・ハイパーインフレ／企業の崩壊・不正／概念の深堀解説／パニックと連鎖）
* **台本テンプレ**：7つの時間配分ビートで構成
* **公開前バンク「バブルの解剖学」5本**と、約52秒のOP＋約12秒の短縮OP、8カットの絵コンテ

収益ニッチは「演技力で二モードのトーンの落差を作る」ことを差別化要因に設定。最高CPMの保険・法務ジャンルは、信頼ゲート（E-E-A-T）が匿名VTuber×自動化と相性が悪いので**あえて非採用**にしています。CPMの高さだけで飛びつかない、という判断です。

トライアル回として\*\*第1話「チューリップ・バブル（1637）」\*\*の台本を先行で書きました。構造が最もシンプルで、台本テンプレの検証にちょうどいいからです。締めは必ず開発モードへの橋を架けます。

> ——387年前、人は『次の買い手がいる』という前提だけで値段を信じた。  
> ……で、ここで終わるのが普通の歴史チャンネル。  
> でも見てるだけじゃ我慢できない。さぁ、僕たちは何ができるでしょう？

## 4. 制作パイプライン：編集が止まる時間も止まらないために

一人運用の最大の敵は「編集できない時間帯に全部止まること」です。そこで、企画・素材・分析・自動化を週次で回す**パイプライン**を文書化＋リポジトリ化しました。

Claude Code用のリポジトリ `151me-channel`（33ファイル：`CLAUDE.md` / `brand/` / `content/` / `pipeline/` / `assets/` / `publish/` / `docs/`）を用意し、`CLAUDE.md` を永続コンテキストの錨に。キャラ規則・NGライン・コンテンツ比率・制作作法を全部ここに刻んでおけば、次のセッションでも文脈が立ち上がります。

自動化の段階設計はこうしました。

1. **台本〜VO生成**：`pipeline/01_script_to_vo.py`（ElevenLabs連携）
2. **VO〜タイムライン**：台本の `[静]`／`[動]` タグと転換点のタイムスタンプを、VMCプロトコル経由で表情・ポーズの自動切替に変換
3. **以降**：ffmpegで動画組み立て、字幕生成、Shorts切り出し、サムネ生成

重要な線引きとして、**史実検証だけは自動化しません**。量産AI動画との分岐点であり、品質の堀になるので、ここは人手を維持します。アニメ風の動きの質と完全自動化は反比例するので、立ち上げ期は段階1＋2で十分、段階3はスケール期の投資判断としてバックログに置きました。果たしてどこまでできることやら・・・

## 5. 1週間でわかったこと

* **棚卸ししてから手を動かす**：アバターも配信画面も、構成（面数・スロット・連携方式）を一覧化してから直す方が圧倒的に速い。
* **退路を用意する**：わんコメ連携のフォールバックのように、壊れたときの一本を仕込んでおくと本番で死なない。
* **自動化しない場所を決める**：全部自動化するのが偉いのではなく、品質の堀になる工程（史実検証）を人手で守ることが差別化になる。
* **ライセンスは最初に確定**：アバターの商用可否を曖昧にしたまま進めない。
