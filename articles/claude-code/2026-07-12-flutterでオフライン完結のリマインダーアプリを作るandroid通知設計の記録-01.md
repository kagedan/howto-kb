## はじめに

ネットワーク接続なしで完結するリマインダーアプリが自分用に欲しくて、Flutterで作りました。この記事は、その開発過程——特にAndroidの通知まわりの設計判断——の記録です。

完成品の紹介というより、「なぜその設計にしたか」「何を割り切ったか」を中心に書いています。同じように通知系のアプリを作ろうとしている方の参考になれば嬉しいです。

## 背景・作ろうと思った理由

クラウド同期型のリマインダーアプリは便利ですが、通信環境やアカウントに依存します。自分の用途（服薬や水分補給といった日々のルーティンの通知）では同期は不要で、むしろ「端末単体で確実に鳴ること」の方が重要でした。

そこで、通知もデータ保存も完全ローカル（sqflite + ローカル通知）で成立する構成にしました。この「オフライン完結」という要件が、技術選定全体の出発点になっています。

もう一つの動機は、既存の通知アプリでよく聞く不満——「狙った時間に鳴らない」「端末を再起動すると通知が消える」——を、実装レベルで丁寧に潰してみたかったことです。

なお、スコープは服薬に限定せず「毎日繰り返すルーティン全般」にしました。服薬専用にすると医療アプリ的な配慮（診断・治療への言及を避ける設計）が重くなるのと、MVPとしてシンプルに保ちたかったためです。

## 技術選定とその理由

### 採用したパッケージ

```
dependencies:
  flutter_local_notifications: ^18.0.0
  timezone: ^0.9.4
  permission_handler: ^11.3.0
  sqflite: ^2.3.0
```

サーバーは使いません。今回のスコープはAndroidのみです。

### 一番大きな判断：exact alarmを使わない

今回の技術選定で一番考えたのは、「正確なアラーム（exact alarm）を使うかどうか」でした。結論として使わないことにしました。理由は次の通りです。

* `SCHEDULE_EXACT_ALARM` 権限は、Android 13以降をターゲットにした新規インストールアプリではデフォルトで拒否されるようになっています。ユーザーに設定画面まで行ってもらって手動で許可してもらう必要があります
* Google Play側でも、この権限の使い方について警告を受けたという報告が複数あります
* 常時許可される `USE_EXACT_ALARM` という権限もありますが、これは目覚まし時計やカレンダーアプリなど、用途が該当しないと使えない建て付けになっています

ルーティンリマインダーは「目覚まし時計」カテゴリには該当しません。そこで、`AndroidScheduleMode.inexactAllowWhileIdle` による準正確な通知を採用し、**数分のズレは許容する**設計にしました。

「正確さ」を売りにしたかったのに正確なアラームを捨てるのは一見矛盾していますが、権限が拒否されて通知自体が動かないリスクや審査リスクを考えると、「多少ズレても確実に届く」方が実用上の価値は高い、という判断です。個人開発では「技術的に可能か」より「ストアの運用に乗るか」を先に見るべきだと感じた場面でした。

なお、`android_alarm_manager_plus` も検討しましたが、exact alarm前提の設計だったため見送りました。

## アーキテクチャ設計

### 決定論的ロジックとAIの分離

これまで作ってきたアプリと同じ方針で、決定論的なロジックとAI処理をレイヤーで分離しています。

| レイヤー | 役割 | 決定論性 |
| --- | --- | --- |
| `RoutineRepository` | sqfliteでのCRUD | 完全に決定論的 |
| `NotificationScheduler` | 通知の登録・キャンセル・再スケジュール | 完全に決定論的 |
| `RoutineLogRepository` | 実行履歴の記録・継続率計算 | 完全に決定論的 |

### 繰り返しパターンの実装難易度差

ルーティンの繰り返しは「毎日」「曜日指定」「間隔指定（N日ごと）」の3パターンを用意しましたが、実装難易度に大きな差がありました。

| パターン | 難易度 | 方式 |
| --- | --- | --- |
| 毎日 | 簡単 | `matchDateTimeComponents: DateTimeComponents.time` でOS側が自動繰り返し |
| 曜日指定 | 簡単〜普通 | 曜日ごとに `DateTimeComponents.dayOfWeekAndTime` で個別登録 |
| 間隔指定 | 難しい | OS側に「N日ごと」の繰り返し機能がなく、自前設計が必要 |

「毎日」と「曜日指定」はOSのネイティブ繰り返し機能で完結しますが、「N日ごと」だけは対応する仕組みがありません。そこで間隔指定は、**次の1回分だけを登録し、機会があるたびに次回分を登録し直す**方式にしました。

ここで一つ設計上のこだわりがあります。次回分の再登録を、ユーザーのボタン操作（スヌーズや完了のタップ）に紐付けなかったことです。通知をスワイプで消されたり放置されたりすると、再登録が永久に走らなくなるからです。代わりに、アプリ起動時・通知発火時・端末再起動後という複数の経路すべてで「未来の通知が登録済みか」を確認し、なければ補完するようにしました。単一のトリガーに頼らず冗長にチェックする、という考え方です。

次回発火日時は起算日（anchorDate）からの経過日数で毎回計算し直すので、アプリを何日も開かなかった場合でも取りこぼさず次回を算出できます。

### 通知IDの採番ルール

Androidの通知登録には整数IDが必要です。複数ルーティン × 複数曜日 × 1日複数回の時刻、と組み合わせが増えるため、衝突しない採番ルールを最初に決めました。

```
notificationId = routineId * 1000 + weekdayIndex * 10 + timeIndex
```

例えば routineId=5、火曜（index=2）、2回目の時刻（index=1）なら `5021` です。IDから「どのルーティンのどの曜日のどの時刻か」を逆算できるので、キャンセル処理もシンプルになります。

スヌーズで出す一時通知は、これと衝突しないよう別レンジ（`900000 + 元のID`）に分離しました。同じIDで登録すれば前回のスヌーズ通知を自動的に上書きできる、という副次的なメリットもあります。

### 再起動対応

「再起動すると通知が消える」問題への対応は二段構えにしました。

1. `ScheduledNotificationBootReceiver` をManifestに登録し、`BOOT_COMPLETED` でプラグイン側の復元を効かせる
2. それとは別に、アプリ起動時に必ず `NotificationScheduler.scheduleAll()` を実行し、アプリ側のロジックでも通知を再構築できるようにする

プラグインの復元機能だけに任せず、アプリ側でいつでもゼロから再構築できる状態を保つのが信頼性のポイントだと考えています。

## スヌーズ機能の設計

通知には「スヌーズ」「完了」のアクションボタンを付けました。ここで一番気を使ったのがバックグラウンドでの処理です。

通知のボタンが押されるとき、アプリはほぼ確実にフォアグラウンドにいません。`onDidReceiveBackgroundNotificationResponse` にトップレベル関数を登録して処理するのですが、このハンドラは独立したisolateで動くため、Riverpodのproviderには依存できない前提で設計する必要があります。

```
@pragma('vm:entry-point')
void notificationTapBackground(NotificationResponse response) {
  // 独立isolateのため、通常のDIには頼れない
  handleNotificationAction(response.actionId, response.payload);
}
```

処理に必要な情報（routineId、予定時刻、スヌーズ回数）は通知のpayloadにJSONで埋め込んでおき、ハンドラ側ではそれをパースしてDB記録などの最小限の処理だけを行います。

スヌーズには上限回数（3回）を設けました。寝落ちなどで無限にスヌーズされ続けるのを防ぐためです。上限に達したら実行履歴に「missed」として記録します。

また、通知を無視されたりスワイプで消されたりした場合の「missed判定」は、通知側のイベントに頼らない方式にしました。アプリ起動時に実行履歴を走査して、「予定時刻から3時間経過して未完了」のものをまとめてmissedにする、バッチ的な処理です。通知のイベントは取りこぼしがあり得る前提に立つと、起動時の走査で担保する方が確実だと判断しました。

## 実装の流れ（Claude.ai → Claude Code）

自分の定番ワークフローは「Claude.aiで設計 → Markdown仕様書を作成 → Claude Codeに渡して実装」です。今回もその流れですが、いつもと違ったのは**仕様書1枚を一括で渡した**ことです。

段階的に小出しにするのではなく、通知設計・繰り返しパターン・スヌーズまで全部入りの仕様書を最初に渡しました。exact alarm回避の方針、通知IDの採番ルール、interval型の再登録戦略といった設計判断を、実装前にすべて仕様書レベルで決め切っていたためです。

結果として、実装フェーズでの手戻りはほぼありませんでした。今回は「仕様書の完成度を上げることに寄せた回」だったと言えます。

## つまずいたポイント：通知の検証は実時間との戦い

正直に書くと、実装そのもので大きくハマった箇所はありませんでした。設計段階でリスクの高い判断（exact alarm、ID採番、再登録戦略）を潰しておいたことが効いたのだと思います。

ただ、一つ想定していなかったコストがありました。**通知の発火確認に実時間がかかる**ことです。

エミュレータで「この時刻に通知が来るか」を確認するには、基本的にその時刻まで待つしかありません。しかも今回は `inexactAllowWhileIdle` を採用しているので、設定時刻ちょうどではなく数分の幅を持って待つ必要があります。「登録できたはず→待つ→鳴った/鳴らない→修正→また待つ」というサイクルは、コードを書く時間よりも検証の待ち時間の方が長く感じる場面がありました。

今回はこれを**単純に待つ運用で割り切りました**。特別な工夫はしていません。

ただ、次に通知系のアプリを作るなら、最初から検証用の仕組みを入れておきたいと思っています。

* テスト用に「1分後に発火」など直近の時刻で仮登録できるデバッグ機能を最初から用意しておく
* エミュレータのシステム時刻を進める方法（`adb shell date` など）が使えるかの検証

通知系の実装は、コードの正しさとは別に「検証のしやすさ」自体を設計段階で考慮すべきだった、というのが今回の一番の反省点です。

### 追記：実機（Pixel 10 Fold）では通知が約2分遅れて届いた

エミュレータでの検証を終えたあと、実機のPixel 10 Foldで動かしたところ、設定時刻になっても通知が来ず、一瞬「実機では動かないのか」と焦りました。しかし待っていると、**約2分遅れで通知が届きました**。

冷静に考えれば、これは設計時に自分で選んだ `inexactAllowWhileIdle` の想定通りの挙動です。exact alarmを避けて「数分のズレを許容する」と決めたのは自分なのに、いざ実機で設定時刻に鳴らないと不安になる——というのは、設計判断が本当に腹落ちしていたかを試された感覚でした。

同時に、実機で確認できたことには意味がありました。

* 遅延は約2分程度で、ルーティンリマインダーの用途では実用上まったく問題ないレベルだった
* エミュレータではほぼ設定時刻通りに発火していたため、inexactの「ズレ」の実際の幅は実機でないと体感できなかった

「数分のズレを許容する」という判断が、実機でどの程度のズレとして現れるのかを自分の端末で確認できたことで、この設計に対する納得感が上がりました。逆に言えば、もし1分単位の正確さが必須の用途（アラームそのものなど）であれば、この構成は選べないということも体感として分かりました。

## 学び

**技術面**では、「技術的に正確な方法」より「ストアの運用と権限モデルに乗る方法」を優先する判断軸を持てたのが収穫でした。exact alarmを捨てて数分のズレを許容する決断は、スペック上は後退に見えますが、実際に届く通知という意味では前進です。

**ワークフロー面**では、設計フェーズを厚くして仕様書を一括で渡す方式が、少なくとも今回の規模（設計判断が多く、UIは比較的シンプル）のアプリでは有効だと確認できました。手戻りがほぼなかったこと自体が、事前設計の価値を示していると思います。

一方で、「検証に実時間がかかる」という運用コストは設計段階では見えていませんでした。また、`inexactAllowWhileIdle` の「数分のズレ」が実機で実際どの程度なのか（今回は約2分）は、動かしてみて初めて分かる情報でした。設計で決めた許容範囲を、実機の実測で裏付ける——このプロセス自体に価値があると感じています。
