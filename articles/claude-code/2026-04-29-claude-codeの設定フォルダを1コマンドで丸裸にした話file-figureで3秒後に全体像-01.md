【冒頭フック】

Claude Codeを使い始めて数週間、~/.claudeフォルダを開いてみたら、何十個もファイルが並んでいて「これ全部何？」と固まった経験はないだろうか。

settings.json、settings.local.json、CLAUDE.md、history.jsonl、agents/、skills/、sessions/……

自分で作ったファイルなのか、勝手に生成されたものなのか、正直まったく判断できなかった。

そんなとき、file-figureというスキルを試したら、あっという間に全体像が把握できた。今回はその体験を紹介したい。

【そもそも~/.claudeフォルダってどういう構造？】

Claude Codeをインストールして使い始めると、ホームディレクトリに「.claude」というフォルダが自動で作られる。ここはClaude Codeのグローバル設定が全部集まる場所で、設定ファイル・カスタムエージェント定義・スキル・コマンド履歴などが入っている。

問題は、使えば使うほどファイルが増えていくことだ。sessions/、shell-snapshots/、file-history/、backups/……これらは使うたびに自動で生成されるファイル群で、放っておくとフォルダが肥大化していく。

Zennでも「~/.claude/ディレクトリ肥大化問題」として記事になるくらい、多くの人がこの混乱を経験している。私もその一人だった。

自分が手動で作った設定と、Claude Codeが勝手に作ったファイルが入り混じり、「削除して大丈夫なのかな？」と怖くて手をつけられない状態になっていた。

【file-figureスキルとは？】

file-figureは、カレントの作業ディレクトリと~/.claudeフォルダの両方を、ASCII図（ツリー形式）で可視化してくれるClaude Codeのスキルだ。

通常のlsコマンドやファイルエクスプローラーと何が違うかというと、次の3点が大きい。

・各ファイル・ディレクトリに「役割コメント」が自動で付く

・sessions/やbackups/など大量の自動生成ファイルが入るフォルダは「← 自動生成（省略）」と折りたたまれる

・「整理候補」として具体的な改善案を日本語で提案してくれる

つまり、ただファイル一覧を出すだけでなく、「これは何のためのファイルか」「削除・移動していいか」まで教えてくれる。

【実際に試してみた】

claude-workspaceで/file-figureを実行してみたところ、出力はこんな感じだった。

─────────────────────

claude-workspace/

├── .claude/

│ └── settings.local.json # プロジェクト固有設定

├── CLAUDE.md # プロジェクト指示

├── docs/

│ └── memo.md # メモ

├── projects/ # (空) 本番プロジェクト用

├── sandbox/ # (空) 試作・実験用

└── templates/

    ├── CLAUDE\_project.md # CLAUDE.mdテンプレート

    └── python\_script.py # Pythonテンプレート

~/.claude/

├── CLAUDE.md # グローバル指示

├── settings.json # グローバル設定

├── settings.local.json # ローカル上書き設定

├── agents/ # カスタムエージェント定義（10ファイル）

├── commands/

│ └── note-write.md # スラッシュコマンド定義

├── skills/

│ ├── file-figure/

│ │ └── SKILL.md # このスキルの定義

│ └── note-post/ # note/X投稿スキル（15ファイル）

├── history.jsonl # CLIコマンド履歴

├── backups/ ← 自動生成（省略）

├── file-history/ ← 自動生成（省略）

├── sessions/ ← 自動生成（省略）

└── shell-snapshots/ ← 自動生成（省略）

─────────────────────

見た瞬間に「あ、全部わかる」と思った。

それまで得体の知れない存在だったsessions/やshell-snapshots/が「← 自動生成（省略）」と折りたたまれているだけで、「これは自動で作られるから気にしなくていいやつだ」とすぐ判断できた。

逆にagents/やskills/には役割コメントが付いていて、「これは自分でカスタマイズできる場所なんだ」という認識が一発でできた。

【整理候補の提案が地味に助かる】

file-figureが出してくれた整理候補がこちらだった。

・setup-mac.shがある——Windows環境なのにMac用スクリプトが残っている。削除推奨

・auto-pull.logがルート直置き——logs/ディレクトリに移動するとすっきりする

・note-postスキルが肥大化——post\_to\_note.pyとpost\_note.ps1など同目的ファイルが重複気味

・commandsとskillsの使い分けが不明確——スキル定義の置き場所が分散している

正直、「setup-mac.shがWindowsに残ってる」は自分では気づかなかった。言われてみれば確かにそうで、Macを使っていたときの名残がそのまま残っていた。消して問題ないと明示されると、踏み切れる。

このように「何がおかしいか」を具体的に指摘してもらえるのが、ただのls -laと大きく違うところだ。

【どんな場面で使うといいか】

個人的には、次のタイミングで使うのがおすすめだ。

・Claude Codeを使い始めて「設定フォルダ、何が入ってるんだろ？」と思ったとき

・スキルやエージェントを追加しまくって、何があるか把握できなくなってきたとき

・久しぶりにClaude Codeを触るとき（「あれ、このフォルダどんな構成だったっけ？」を解決するのに便利）

・他人のClaude Code設定を参考にしたあと、自分のと比較したいとき

特に「久しぶりに触る」ときの全体像の素早い把握は、個人的に一番恩恵を感じた使い方だ。セッションを中断して数日後に再開したとき、「どこまでやってたっけ」と一緒に「ファイル構成どうなってたっけ」も解消できる。

【使い方は一行だけ】

Claude Codeを起動した状態で、チャット欄に入力するだけ。

/file-figure

追加の引数なしで、作業ディレクトリと~/.claudeの両方を自動でスキャンして出力してくれる。インストールの手順などは必要なく、スキルとして登録されていれば即使える。

スキルの登録方法は別途解説があるが、ざっくり言うと~/.claude/skills/に定義ファイルを置くだけだ。

【まとめ】

file-figureスキルを試してわかったのは、「ファイルが見えていないと整理できない」という当たり前の事実だった。

lsコマンドでファイル一覧を出しても、「これが何か」がわからなければ動けない。file-figureは役割コメント付きのツリーと整理候補の提案によって、「見えている」状態から「判断できる」状態まで一気に引き上げてくれる。

Claude Codeを使い込んでいる人ほど、フォルダ構成が複雑になっていると思う。一度試してみると、思わぬ発見があるかもしれない。

気に入ったら、スキを押してもらえると励みになります。
