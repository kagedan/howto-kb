## 概要

この講演は、2026年4月に Mastra がサンフランシスコで開催したイベント「TypeScript AI Demo Day」で行われたものです。登壇者の Jeff Huber は Chroma の共同創業者兼 CEO で、Chroma は AI エージェントや AI アプリケーション向けに、検索・リトリーバル・コンテキスト注入の基盤を提供している会社です。

オリジナルの動画タイトルは「LOVE IS DEAD」ならぬ「RAG is Dead」。「RAGはもう終わり」というのは「釣りタイトル」で、ちょっといただけないです(笑)。この煽り文句は登壇者の真意ではありません。「RAG」という言葉が人によって意味がばらばらになりすぎていて、建設的な議論の邪魔になっていると指摘します。代わりに据えるべき言葉が **context engineering** であり、さらに将来的には、その仕事のかなりの部分がモデル自身に折り返されていくという見通しを示します。

話の流れとしては、まず Chroma が昨年出した **Context Rot** 研究を踏まえて「大きなコンテキストはそのままでは役に立たない」ことを確認し、次に現在主流のファイルシステム中心アプローチを「それはデータベースの代用品にすぎない」と批判します。そこから、仮想ファイルシステム、ハイブリッド検索、そして Chroma Context One のような検索専用サブエージェントへと、より筋の良い整理に進んでいく構成です。

プロンプトエンジニアリングが騒がれた2023年、コンテキストエンジニアリングが叫ばれた2025年、そして今年2026年はハーネスエンジニアリングが話題です。しかし、まだまだ「コンテキストエンジニアリング」にも考察の余地が多く残っていることに気付かされます。

## 要点

1. **「RAG is dead」は検索や知識注入の否定ではなく、言葉としての RAG が役に立たなくなったという主張**。同じ言葉で人によって別のものを指しているため、議論の土台が崩れている。
2. **今のエージェント失敗の中心は reasoning failure ではなく context failure**。情報が多すぎる、少なすぎる、あるいは矛盾していて優先順位が分からないことが主因だ。
3. **Context Rot の問題は現実であり、大きなコンテキストをそのまま渡しても性能は保てない**。100万トークン級の文脈があっても、それを実務レベルで信頼している開発者はほぼいないという感覚が共有されている。
4. **これから重視すべきは context engineering だが、細かな手作業最適化を人が抱え続けるべきではない**。Jeff は「bitter lesson」に沿って、ツール設計を良くし、最終的にはモデル自身にコンテキスト編集を学ばせる方向を推す。
5. **ファイルシステムはエージェントにとって便利だが、本質的には「よくないデータベース」**。並行更新、インデックス、検索、破損耐性の観点で限界がある。
6. **仮想ファイルシステムは中間解として有力**。エージェントには bash やファイルに見せつつ、裏側はデータベースで支えることで、重いサンドボックスを避けられる。
7. **Jeff のデフォルト推奨は、ハイブリッド検索を下敷きにした agentic search**。単純な grep だけに頼るより、意味検索と語句検索を組み合わせた方が、多くのケースで素直に強い。
8. **検索は必ずしも巨大な推論モデルにやらせるべき仕事ではない**。Chroma Context One のような検索専用サブエージェントを切り出す方が、速く安く、結果も良い可能性がある。
9. **メモリ、コンテキスト圧縮、蒸留は別々の話に見えて、かなり同じ問題を見ている**。大量の履歴から「今、何を思い出すべきか」を選ぶ点で共通している。
10. **「どの手法が優れるか」を語るなら、ベンダーの主張ではなく評価データで見ろ**、という姿勢が一貫している。ここもこの発表の大事なメッセージだった。

## 詳細

### RAG という言葉の曖昧さ

上述の通り、タイトルの「RAG is dead」。Jeff Huber が否定しているのは、リトリーバルそのものではなく、**RAG という言葉の曖昧さ**です。

もともと RAG は Retrieval-Augmented Generation の略で、「外部知識を検索して、その結果を生成時のコンテキストに足す」くらいの意味で使われてきました。ただ、実際には人によって指す範囲が違います。

* ベクターストアから近傍検索する仕組みだけを指す人
* 検索から再ランキング、要約注入まで含めて指す人
* エージェントがツールを使って資料を探す流れまで含める人

このへんの事情は私もこちらの記事で「広義のRAGと狭義のRAG」として述べたことがあります。  
<https://zenn.dev/shiromizuj/articles/9ab0da1496f164>

こうなると、同じ「RAG が効く」「RAG はもう古い」という言葉でも、中身が噛み合いません。Jeff はそこを嫌って、「もうこの単語は捨てて、何をしているのかをもっと具体的に言おう」と提案しています。ここで出てくる代替語が **context engineering** です。なぜなら、いま本当に問題になっているのは「検索を使うかどうか」ではなく、**エージェントに必要な情報を、必要なときに、壊れない形で渡せているか**だからです。

※ 個人的には、RAG（広義のRAG）の代替がコンテキストエンジニアリングなのか、というとどうなんじゃろ、とは思いますが。

---

### エージェントは賢くないのではなく、文脈を渡し損ねている

Jeff は、いまのエージェント失敗の大部分を **context failures** と呼びます。エージェントがうまくいかないとき、「モデルが弱い」「推論が足りない」と考えることがあるかもしれません。しかし、実際には次のような失敗が多いはずです。

* そもそも必要情報が入力されていない
* 履歴や参照文書が多すぎて、重要箇所を見失っている
* 矛盾する情報が混ざっていて、どれを優先すべきか判断できない
* 以前の作業文脈と、今の目的がうまく接続されていない

この見方に立つと、問題の重心は「もっと賢いモデルを選ぶこと」から、「どうやって文脈を整形し、選別し、渡すか」へ移ります。つまり、ここで初めて **context engineering** が本題になるわけです。

---

### Context Rot と「愚かゾーン」

この発表の前提として置かれているのが、Chroma の [**Context Rot レポート**](https://www.trychroma.com/research/context-rot) です。要するに、「コンテキストウィンドウが巨大だからといって、その中の情報を一様な精度で扱えるわけではない」という話です。

理屈の上では、モデルが10万トークンや100万トークン読めるなら、必要な資料を全部押し込めばよさそうに見えます。ですが実務ではそうなりません。情報量が増えると、モデルは重要な箇所に注意を維持しにくくなり、ノイズや矛盾に引っ張られます。Jeff はこれを、開発者が経験的に感じている **dumb zone** と呼んでいます。

この話が大切なのは、単なるトークン上限の問題ではないからです。上限に到達する前でも、十分に性能は崩れます。だから「たくさん入る」こと自体を素朴な価値と見なすのは危険です。

周辺知識がない人向けに言い換えると、これは「分厚い資料を全部机に積めば賢く判断できるわけではない」という話です。人間でも同じで、必要なページに整理されていない資料の山を渡されると、判断はむしろ鈍ります。

---

### context engineering は残るが、手作業最適化は永続しない

ではどうするのか。Jeff の答えは、 context engineering の重要性を認めながら、同時に「人間が手で細かくやり続ける世界」には賭けていません。

ここで持ち出されるのが **bitter lesson** です。これは Rich Sutton の有名なエッセイに由来する考え方で、「人間の巧妙な手作業ルールより、計算量と学習を活かす一般的な方法が長期的に勝つ」という主張です。Jeff はこれを文脈管理にも当てはめています。

* 人間が毎回すべての文脈注入ロジックを細かく決めない
* 代わりに、エージェントへ強いツールと適切なガイダンスを与える
* さらに進むと、モデル自身が必要な文脈を編集・圧縮・選別する

これは「工夫しなくてよい」という意味ではなく、むしろ工夫すべき場所が変わるという話です。プロンプトでコンテキスト注入に関する大量のコツを書くより、検索ツールや文脈編集ツールをどう設計するかの方が重要、という整理です。

---

### なぜ Jeff は「ファイルシステムは悪いデータベースだ」と言うのか

現在のコーディングエージェントやエージェント実装では、bash とファイルシステムを与える構成が非常に多いです。Claude Code も Codex もそうですし、Mastra の ローカル sandbox や remote workspace も然りで、この形はかなり一般的です。

Jeff もそれ自体の有効性は認めています。実際、便利です。ただし彼は、それを長期的な正解とは見ていません。理由はかなり素直です。

* **並行更新に弱い**: 2人または2エージェントが同じ対象を触ると、基本は 「後勝ち」 になる
* **インデックスがない**: 検索や集約のための構造が最初から存在しない
* **検索能力が低い**: だから実質 grep ベースで、文字列一致の世界から出にくい
* **重量が大きい**: 単に読むだけなのに、sandbox 一式を起動するのは過剰なことがある

これは、ある種の逆説です。エージェントにとってファイルシステムは使いやすいのに、システム設計としてはかなり粗い。だから、便利さに引っ張られて使っているうちに、いつのまにか「検索」「整合性」「更新管理」が必要になり、結局データベースっぽいものを書き始めてしまう。そこで Jeff が引用するのが Swyx の「Oops, You Wrote a Database」です。

この指摘は、今のエージェント開発のかなり本質を突いていると思います。コーディングエージェント文脈では、ファイルは「自然な表現」ですが、知識アクセス基盤としては無理が出やすいのです。

---

### 仮想ファイルシステムは、現実的な中間解として筋が良い

ただし Jeff は、「ではファイルシステムを全部捨てよう」とは言いません。代わりに紹介するのが **virtualized file system** です。

これは、エージェントから見ると普通のファイルや bash に見えるが、実際の保存・検索・整合性管理は裏側のデータベースが担う、という構成です。動画中では [Mintlify](https://www.mintlify.com/) がこのアプローチを使った例として紹介されます。

この考え方が面白いのは、人間やエージェントにとって使いやすいインターフェースと、システムとして筋の良い保存層を分離していることです。

* 表側: ls, cat, grep, glob といった「エージェントが使いやすい操作」
* 裏側: インデックス、同期、検索、整合性を持つストレージ

つまり、「見せる顔はファイルシステム、実体は検索基盤」という構図です。個人的には、このアプローチというかアイデアを知らなかったので感心しました。いきなり全エージェントに新しい検索 DSL を叩かせるより、既存の作法に近い形で移行できます。

---

### grep は便利だが、最終回答ではない

Jeff さらに言及したのは grep の扱いです。彼は、grep を完全否定しているわけではありません。小さいコーパスなら、grep 的に探しても最終的に辿り着けることは多いと認めています。

ただし、それはあくまで **何を探しているかを人間がかなり分かっているとき** の話です。grep は本質的にはパターンマッチです。つまり、具体的なキーワードや構文が頭にあるときには強い。でも、意味的に近いものを探したい、曖昧な言い換えや関連概念から当てたい、というときには急に弱くなります。

そこで出てくるのが **hybrid search** です。これは一般に、

* BM25 のような語句一致ベースの検索
* embedding を使う意味検索

を組み合わせたものです。語彙一致の強さと、意味的な広がりの両方を取りにいくわけです。Jeff はこれを、いまのところ多くのユースケースに対する「デフォルトの魔法ボタン」として推しています。

ここでの重要点は、「grep か semantic search か」の二択ではなく、**両方の良いところを使う**という整理です。これは、Mastra の Workspace 機能も、grep と BM25 と Vector検索 をサポートしているのと呼応する考え方です。

---

### Chroma Sync と Context One が示す方向性

発表後半は、Chroma 側で出した2つのプロダクトの紹介です。

#### Chroma Sync

これは、ファイルをバケットに置けば、Chroma がそれを拾って自動でインデックス処理までやってくれる仕組みです。地味に見えますが、現場ではかなり重要です。検索基盤が面倒になるのは、検索アルゴリズムそのものよりも、**取り込みパイプラインの運用**だからです。そこをマネージドに寄せる発想です。

#### Chroma Context One

こちらは検索専用のサブエージェント、あるいは検索専用モデルです。自然言語の問い合わせを受ける点は普通の LLM と似ていますが、出力するのは「最終回答」ではなく「関連ドキュメント」です。

そのドキュメントを、別の大きな推論モデルに渡して最終生成をさせる。つまり、

を分離しているわけです。

この分離はかなり大事です。多くの人は、Opus や GPT-5 のような強いモデルなら検索まで全部上手くやるだろうと期待しがちです。でも Jeff は、検索という仕事は「賢さの総量」で解くより、専用化した方が速く、安く、結果も良いと主張します。

私自身も自分の AIエージェント で「grep と BM25 と Vector検索」の構成にしているのに加え、検索Toolは「ファイルパス」だけを返してエージェントは必ず「ドキュメントを読んで推論する」という風にあえて分離させていることもあって、非常に「我が意を得たり」感がありました。

### Q&Aで見えた本質: memory と compaction も同じ地平にある

質疑応答の最初で出た「これは memory にも使えるのか、それとも context compaction か」という質問に対して、Jeff **その区別は思ったほど大きくない**と回答しています。

大量の履歴や知識があり、その中から「今この瞬間に何を思い出すべきか」を選ぶ。これは、

* memory
* context compaction
* distillation

のどれとしても見える問題です。

この整理は、最近のメモリ議論ともかなり相性が良いです。会話履歴圧縮、ワーキングメモリ、セマンティックリコール、長期メモリなど、名前は違ってもやっていることは「限られた文脈予算の中で、何を残し、何を捨てるか」を決めることだからです。

---

### 「誰がそう言っているか」より「評価データがあるか」を見ろ

最後の Q&A で印象的だったのは、RAG 的な手法と agentic search の境界について聞かれたとき、Jeff がかなり冷静に「ベンダーの言うことは、まずインセンティブを疑え」と返したところです。つまり「ポジショントークを疑え」ということでしょうか。

これは少し刺々しくもありますが、トークンを多く使わせたい側は、重い探索や長い思考を勧める動機を持ちます。だから「全部 agentic search の方が良い」とか「もう grep で十分」といった主張は、そのまま飲み込まず、必ず評価で見るべきだ、というわけです。
