― 個人ゲーム開発に AI を組み込んだ、ぐちゃぐちゃの試行錯誤の全記録 ―  
まずは結論っぽいやつから

僕は本業がエンジニアの会社員で、夜と週末を使って Unity 2D で個人ゲームを作っている。ジャンルはダークファンタジー RPG。世界観も戦闘もシナリオも、ぜんぶ一人で組んでいる。

そして、開発の大半に **Claude Code** を使っている。

「AI でゲーム作りました」みたいな景気のいい話ではない。むしろ、Claude Code を Unity の個人開発に組み込むのは、思ったよりずっと癖があって、最初の数週間は普通に書いたほうが速いんじゃないかと何度も思った。

ただ、ある時期から急に **「AIを"使う"ではなく"育てる"」** という感覚が腑に落ちて、そこから一気に景色が変わった。

この記事では、その「景色が変わるまでに何をやったか」「変わってから何ができるようになったか」を、できるだけ具体に書く。

---

## なぜ Cursor じゃなくて Claude Code なのか

個人ゲーム開発の文脈で、AIコーディング系の話をすると、たいてい Cursor の話になる。実際、Unity × Cursor の組み合わせは記事も動画も豊富にあって、入り口としては優秀だ。

僕も最初は Cursor で始めた。

ただ、ある程度プロジェクトが大きくなってくると、Cursor の「エディタの中で対話する」というモデルが、自分のやり方に合わなくなってきた。

具体的にこういう不満が出てきた。

* **複数ファイルにまたがる変更**を、エディタ上で一個一個レビューするのが疲れる
* \*\*「3つのタスクを順番にやって」\*\*みたいな粒度の指示が出しにくい
* 何度も似たような指示をしていて、**プロジェクト固有の文脈が記憶されない**

Claude Code は、これらの不満をだいたい解消してくれた。

* ターミナルで指示を出して、複数ファイルを一気に書き換えてもらえる
* 「これとこれとこれをやって」と並列のタスクを投げられる
* `CLAUDE.md` にプロジェクトの文脈を書いておけば、毎セッション読み込んでくれる

特に\*\*「プロジェクトのメモリを持たせる」\*\* という発想は、個人開発の文脈で効く。Unity の個人プロジェクトって、開発期間が長くなりがちで、3ヶ月前の自分が考えていたことを忘れる。[CLAUDE.md](http://CLAUDE.md) は、過去の自分が未来の自分（と AI）に書き残す手紙みたいなものだ。

## この記事で書くこと

ここから先で書くのは、こういう話だ。

1. **Windows + WSL2 + Unity + Claude Code の組み合わせ問題**：Unity は Windows ネイティブ、Claude Code は WSL2 上。この二重構造をどう成立させるか
2. **Unity 2D ゲーム開発に特化した** [**CLAUDE.md**](http://CLAUDE.md) **の書き方**：プロジェクト概要、世界観、コード規約、Unity 固有の作法をどう詰め込むか
3. **役割別プロンプト集**：戦闘システム、UI、敵 AI、シナリオ分岐など、ゲーム開発の各場面で使っているプロンプトをコピペできる形で公開
4. **ダークファンタジーの世界観を AI と一緒に作る方法**：用語の一貫性、キャラ台詞のトーン、固有名詞の管理
5. **Unity 特有の「AI に任せてはいけない領域」**：シーン構成、プレハブ、ScriptableObject、エディタ拡張など、AI が苦手な領域とその切り分け
6. **失敗集**：実際にハマった話
7. **平日エンジニアの時間設計**：仕事しながら個人ゲームを完成に近づけるための、AI込みのワークフロー

総ページ数で言うと、ここから先が約 10,000 文字。

正直、ゲーム開発の話って、競合の有料記事を見ても「やってみた感想」「環境構築の手順」みたいな、薄い記事が多い。それは多分、本当に作っている人が、忙しくて記事を書く時間がないからだ。

僕は今、自分のゲーム開発を一段落させて少し時間ができたタイミングなので、覚えているうちに、なるべく具体に書き残しておこうと思う。

それでは、本編へ。完成したら、配布予定
