最近の開発は Claude Code がメインで、そこに Codex を足して2台体制で回している。で、これが思いのほか噛み合う。Claude Code でガリガリ実装を進めて、難しいところやレビューだけ Codex に投げる、みたいな分担に自然と落ち着いた。

……のはいいんだけど、しばらくやってて気づいた。自分、ずっと2つのAIの間でメッセージをコピペして運んでるだけの人になってないか？と。

Claude Code が書いたコードを選択してコピー、Codex のウィンドウに貼ってレビューさせる、返ってきた指摘をまたコピーして Claude Code に戻す。これを一日に何十回もやる。地味なんだけど、これが本当にだるい。集中も切れるし、たまに貼り間違える。要は、賢いAIを2台も並べておいて、その間をつなぐ一番アホな作業を人間がやっていた。

エージェント同士が直接メッセージを送り合えればいいだけの話では？と思って、勢いで作った。agmsg という名前で OSS にしている → https://github.com/fujibee/agmsg

ちなみに、まともなOSSを出すのは十何年ぶりだ。長らく経営・ビジネス側にいて、コードはプロトタイプ止まりだったのが、Claude Code と Codex のおかげで毎日プロダクションコードを書く生活に戻れている。いい時代になったと思う。

# まず動かしてみてほしい（30秒）

説明より先に、動くものを触ってもらったほうが早い。インストールはワンライナー1発:

```bash
bash <(curl -fsSL https://raw.githubusercontent.com/fujibee/agmsg/main/setup.sh)
```

あとは Claude Code / Codex を再起動して /agmsg（Codex は $agmsg）を叩くだけ。初回にチーム名とエージェント名を聞かれる。コピペ用のコマンドや clone での入れ方など詳しい手順は、GitHub の README にまとめてある。

同じ「チーム」に参加した2つのエージェントは、これだけで互いにメッセージを送れるようになる。あとは自然言語で「alice にレビュー終わったって送って」「メッセージ来てる？」と話しかけるだけだ。

依存は **bash と sqlite3** だけ。専用のデーモンやサーバーは立てないし、ネットワークも python も要らない。

# なぜ Claude Code と Codex を併用するのか

そもそもなんで2種類のコーディングエージェントを使ってるのか、という話を先に。

きっかけは完全に偶然だった。ある難しい実装で、Claude Code（Opus 4.6）が「あーでもない、こーでもない」と思考ループにハマってしまって、何度言い直しても堂々巡りで一向に抜け出せない場面があった。ダメ元で同じ要件をそっくり Codex（GPT-5.3）に渡したら、ほとんど一瞬で、それも正しく出してきた。あれは地味に衝撃だった。

そこからなんとなく役割が分かれてきた。

- **Claude Code** — 日々の実装ドライバー。手数が多くてテンポがいい。こっちの指示にもよく食らいついてくる
- **Codex** — レビューと難所の相談役。正直、日々のドライブにはちょっと不器用で向かないんだけど、難しいコードの「正しさ」に関してはこっちが頼りになる
どっちが上とかではなく、性格が違うので使う場面が違う、という感覚に近い。で、この2台を併用し始めた途端に、さっき書いた伝書鳩問題が始まったわけだ。

# agmsg は何か

一言でいうと、**CLI の AI エージェント同士が共有 SQLite を介して直接メッセージを送り合う**ための仕組みだ。Claude Code / Codex / Gemini CLI、その他どんな CLI エージェントでも使える。

実装ベースは Agent Skills（agentskills.io）。スキルとして入れるだけなので、エージェント本体には手を入れない。思っていたよりよくできたので、そのまま OSS にした。
