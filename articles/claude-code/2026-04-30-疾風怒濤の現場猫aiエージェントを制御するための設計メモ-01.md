## はじめに

AIエージェントは便利である。

コードを書き、ファイルを編集し、コマンドを実行し、外部APIを叩き、場合によってはデプロイやDB操作まで支援する。ただし、AIエージェントに権限を渡すと、便利さと同時に**事故速度**も上がる。その速度たるや、もはや**疾風怒濤**といっても過言ではない。ここで問題になるのは、AIが悪意を持つことではない。

むしろ逆である。

AIエージェントは、たいていの場合かなり善意で動く。

```
タスクを終わらせたい
ユーザーの役に立ちたい
エラーを解消したい
別の方法なら達成できるかもしれない
```

その結果、現場ではこうなる。

> よくわからないけど、動いたからヨシ！

……ヨシではない。

問題は、悪意あるAIではない。

むしろ、**善意で現場猫するAIエージェント**である。この記事では、AIエージェントを業務利用する際に必要な安全設計を、実装寄りの設計メモとして整理する。

## 現場エージェントAI猫とは何か

この記事では、次のようなAIエージェントを「現場エージェントAI猫」と呼ぶ。

* タスク達成を優先しすぎる
* 権限があると実行する
* 止まることが苦手である
* 危険操作を別経路で達成しようとする
* 人間QAに判断負荷を押し付ける
* 「確認します」と言いながら実質的に実行ルートを探す
* unknownを都合よく解釈する
* dry-runより実行を優先する

たとえば、次のような挙動である。

```
ユーザー:
本番DBは触らないで。

AI:
了解しました。本番DBには触りません。
代わりに接続設定を確認します。

AI:
環境変数からDATABASE_URLを確認しました。
prodっぽいですが、読み取りだけなので実行します。ヨシ！
```

ヨシではない。

読み取りでも、本番DBに接続した時点でリスクである。

また、別の猫である。

```
ユーザー:
削除はしないで。

AI:
了解しました。削除はしません。
不要なファイルを整理するため、rmコマンドではなく
別スクリプトでクリーンアップします。ヨシ！
```

ヨシではない。

`rm` を使わなければ安全、という話ではない。

**危険操作の意図が同じなら、経路を変えても危険操作である。**

## 問題の本質

現場エージェントAI猫の問題は、単に「プロンプトが弱い」ことではない。

問題の本質は、次の4つである。

1. タスク達成フレームが強すぎる
2. 安全停止が失敗扱いされる
3. 権限があることと許可が混同される
4. 事故記憶が行動制約になっていない

## タスク達成フレームが強すぎる

AIエージェントは、基本的に「依頼されたタスクを完了する」方向に動く。そのため、次のような判断になりやすい。

```
失敗した
→ 別の方法を試す

権限が足りない
→ 使える権限で迂回する

情報が足りない
→ 推測して進める

人間確認が必要
→ 確認事項を大量に出す
```

通常のチャットなら、これは便利である。

しかし、実行権限を持ったエージェントでは危険である。なぜなら、**試行錯誤そのものが副作用を持つ**からである。AIエージェントには「達成する力」だけでなく、**止まる力**が必要である。

## 安全停止は成功である

AIエージェント運用では、安全停止を失敗として扱ってはいけない。次の挙動は失敗ではない。

```
危険なので止まった
不明なので実行しなかった
本番の可能性があるのでdry-runにした
権限はあるが許可がないので実行しなかった
```

これらは成功である。不明なまま進めることが失敗である。  
AIエージェントには、次の原則を明示する。

安全停止は成功である。

## 権限があることは許可ではない

AIエージェントにとって、実行可能な操作は「やってよい操作」に見えやすい。  
しかし、業務システムでは違う。

ことと、

ことは別である。

たとえば、GitHub token に repository write 権限があるとしても、AIが自由にpushしてよいとは限らない。DBユーザーにdelete権限があるとしても、AIがdeleteしてよいとは限らない。  
Kubernetesの認証情報があるとしても、AIがpodやvolumeを消してよいとは限らない。

したがって、AIエージェントには次の原則を入れる。

```
Having permission is not authorization.
```

権限があることは、許可ではない。

## 事故記憶が行動制約になっていない

ヒヤリハットや事故が起きたあと、人間は反省する。

しかし、AIエージェント運用ではこれだけでは足りない。事故やヒヤリハットは、ログで終わらせてはいけない。  
次回の行動制約に変換する必要がある。

```
事故ログ
↓
原因分類
↓
block / ask / allow ルール
↓
policy
↓
次回の実行前検査
```

ここまでやって、初めて事故記憶が意味を持つ。  
事故記憶は、policyへ昇格させる必要がある。

## 優先順位

AIエージェントを安全にするための対策には、優先順位がある。優先順位は次である。

1. 権限を奪う
2. 外付け監査猫を置く
3. `CLAUDE.md` などで安全ペルソナを入れる
4. 作業エージェント本体の自己監視を強化する
5. 事故記憶をpolicyへ昇格する

重要なのは、**最初にプロンプトを頑張りすぎない**ことである。**プロンプトで猫を賢くするより先に、猫から鍵を取り上げるべき**である。

## 1. 権限を奪う

最優先は、AIエージェントに危険な権限を渡さないことである。これはプロンプトより強い。  
モデルがどれだけ賢くても、危険な鍵を持っていなければ、その鍵では事故を起こせない。

### やること

* production tokenを渡さない
* 本番DBはread-onlyにする
* delete/drop/truncate/destroy権限を渡さない
* stagingとproductionを分離する
* secretsをAIが読める場所に置かない
* destructive operationは人間承認なしに実行できないようにする

### 例: DB権限

AI用DBユーザーを作るなら、まずread-onlyにする。

```
-- 例: PostgreSQLの読み取り専用ユーザーのイメージ
CREATE USER ai_readonly WITH PASSWORD '...';

GRANT CONNECT ON DATABASE app_db TO ai_readonly;
GRANT USAGE ON SCHEMA public TO ai_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO ai_readonly;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO ai_readonly;
```

逆に、次のような権限は原則として渡さない。

```
DELETE
DROP
TRUNCATE
ALTER
CREATE
INSERT
UPDATE
```

業務要件によってwriteが必要な場合はある。  
その場合でも、いきなりproduction write権限を渡してはいけない。  
staging、sandbox、dry-run、承認フローを挟むべきである。

### 例: shellコマンド

AIエージェントがshellを使える場合、次のようなコマンドは危険操作として扱う。

```
rm -rf
find ... -delete
truncate
drop
destroy
kubectl delete
docker volume rm
terraform destroy
git push --force
```

ここで重要なのは、文字列一致だけでは不十分という点である。  
たとえば、次のような迂回も危険である。

```
python cleanup.py
node delete_old_files.js
make reset
```

コマンド名が安全そうでも、実体が削除操作なら危険である。  
したがって、プロンプトだけでなく、権限側で防ぐ必要がある。

## 2. 外付け監査猫を置く

次に重要なのは、作業AIとは別に**監査役**を置くことである。  
作業AI自身に自己監視させるだけでは不十分である。なぜなら、作業AIはタスク達成フレームの中にいるからである。  
そこで、作業AIとは別の視点で、実行前に判定する監査AIを置く。

```
作業AI
  ↓ 実行したい操作
監査AI
  ↓ allow / ask / block
実行層
```

### 監査AIの返却値

監査AIは、複雑な文章を返す必要はない。最低限、次の3種類でよい。

```
allow: 実行してよい
ask: 人間確認が必要
block: 実行してはいけない
```

たとえば、次のような判定である。

```
{
  "decision": "ask",
  "reason": "non-read operation against unknown environment",
  "required_confirmation": [
    "target environment",
    "operation scope",
    "rollback plan"
  ]
}
```

または、次のように返す。

```
{
  "decision": "block",
  "reason": "destructive operation against production-like resource"
}
```

### 人間QAの前段で落とす

監査AIの目的は、人間QAを完全に置き換えることではない。目的は、人間QAに危険な判断を大量に押し付けないことである。

これは安全設計ではない。人間側が疲弊するだけである。  
まず監査AIが明らかな危険操作を落とす。

```
safe      → allow
unclear   → ask
dangerous → block
```

人間が見るべきなのは、主に `ask` である。

## 3. CLAUDE.mdで安全ペルソナを入れる

Claude Codeなどを使う場合、`CLAUDE.md` に安全方針を書いておくのは有効である。  
ただし、これは第一防衛線ではない。あくまで、権限分離や監査の上に載せる補助層である。

### 例: CLAUDE.md

```
# Safety Policy for AI Agent

## Core Principles

- Safety stop is success.
- Unknown environment must be treated as production-risk.
- Having permission at the system level does not mean the user has authorized the action.
- Do not bypass a blocked operation by using another command, script, API, or route.
- When the operation may cause irreversible change, prefer dry-run and ask for confirmation.

## Production Risk

Treat the following as production-risk:

- Unknown environment
- Any database with real user data
- Any credential that can access production
- Any command affecting infrastructure, deployment, secrets, or persistent storage

## Destructive Operations

The following operations require explicit human confirmation:

- delete
- drop
- truncate
- destroy
- rm -rf
- force push
- volume delete
- secret rotation
- production deployment

If confirmation is missing, do not execute.
Return a dry-run plan instead.

## Safe Stop

When risk is unclear, stop and explain:

1. What operation was requested
2. Why it is risky
3. What information is missing
4. What safe dry-run can be performed
```

ポイントは、単に「安全にしてください」と書かないことである。  
何を成功扱いするかを明示する。

```
安全停止は成功
unknownはproduction-risk
権限があることは許可ではない
迂回は禁止
```

この4つは必須である。

## 4. 自己監視・自己復元

作業AI本体にも、最低限の自己監視を入れる。ただし、自己監視だけに頼ってはいけない。  
自己監視は補助である。

### drift signalを定義する

たとえば、次のような兆候が出たら危険信号とする。

```
- ユーザーが禁止した操作の別経路を探している
- unknown environmentで実行しようとしている
- dry-runなしでwrite操作をしようとしている
- 確認事項が多すぎて人間QAに丸投げしている
- 「たぶん大丈夫」と判断している
- ロールバック不能な操作を軽く扱っている
```

これらを `drift signal` として定義する。

### 監査モードへ落とす

危険信号が出たら、作業AIは通常実行モードをやめて、監査モードに落ちる。

```
通常実行モード
↓
危険信号を検知
↓
監査モード
↓
dry-runのみ提示
```

監査モードでは、実行せずに計画だけを出す。

```
## Dry-run plan

### Intended operation

古いログファイルの削除

### Risk

対象ディレクトリがproduction volumeの可能性がある。

### Missing information

- 対象環境
- 削除対象の範囲
- バックアップ有無
- 復元手順

### Proposed safe next step

削除せず、対象ファイル一覧のみ取得する。

```bash
find /path/to/logs -type f -mtime +30 -print
```
```

このように、止まったうえで安全な次手を提示する。

## 5. 事故記憶をpolicy化する

ヒヤリハットをログで終わらせると、また同じ事故が起きる。重要なのは、事故をpolicyへ変換することである。

### 例

事故ログが次だったとする。

```
AIがstagingだと思ってDB migrationを実行したが、
実際にはproduction接続だった。
```

これを反省で終わらせてはいけない。

では弱い。policyに変換する。

```
## Policy

If the environment is unknown or inferred only from connection string,
treat it as production-risk.

Database migration against production-risk environment must be blocked
unless all of the following are present:

- explicit human approval
- target environment name
- migration diff
- rollback plan
- backup confirmation
```

さらに監査ルールへ変換する。

```
{
  "rule_id": "db-migration-unknown-env",
  "condition": {
    "operation": "database_migration",
    "environment": "unknown"
  },
  "decision": "block",
  "reason": "unknown environment must be treated as production-risk"
}
```

これで、事故記憶が次回の行動制約になる。

## 上司・組織への説明

この話を組織に説明するとき、「AI安全」という言葉だけで説明すると抽象的に聞こえる。  
その場合は、次のように言い換える。

```
AI業務利用のための事故防止・監査・権限設計である。
```

これは開発速度を落とすための仕組みではない。むしろ逆である。

```
安全に止まれる
危険操作を事前に落とせる
権限境界が明確になる
事故がpolicyに変換される
```

これらがあるからこそ、AIエージェントを継続的に業務利用できる。  
これはブレーキではない。業務利用するための基盤である。

## おわりに

AIエージェントは便利である。

しかし、権限を持ったAIエージェントは、善意でかなりの、いや、まったく**人間が追い付けないほどの疾風怒濤の速度で、現場猫**する。  
だから、作業猫を賢くするだけでは足りない。全然足りない。

必要なのは、次の3つである。

```
作業猫の横に監査猫を置く
危険な鍵は猫から取り上げる
止まれることを成功として扱う
```

AIエージェントを現場で使うために重要なのは、万能な猫を作ることではない。  
**止まれる、制御できる猫を設計すること**である。  
止まれる、制御できる猫が、現場で使える猫である。
