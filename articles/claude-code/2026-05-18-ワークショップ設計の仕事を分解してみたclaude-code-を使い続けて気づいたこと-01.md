## 設計する仕事を、分解してみた

ワークショップを設計する仕事を、あらためて分解してみたことがあります。調べること、判断すること、記録すること——突き詰めると、この3つが仕事の大半を占めているんじゃないかと思うようになりました。

Claude Code を本格的に使いはじめたのは、ちょうどそのことを考えていた頃でした。最初は「設計ツールが増えた」くらいの感覚でしたが、使い込むうちに、設計の進め方だけじゃなく、自分が何を大事にして設計しているかが少しずつ見えてきた気がしています。

うまくいかなかったこともあります。それも正直に含めながら、この数ヶ月で気づいたことを書いてみたいと思います。

以前「ワークショップ設計と Claude Code には同じ構造がある」という記事を書きました。今回はその続きというか、実際に使い込んだうえで「何が変わったか」という観察に近い話です。

  

---

## 視点が広がった：補助線を引いてもらう感覚

毎朝、AI秘書（Claude Code 内に設定した自動収集AIのこと）が収集したトレンド記事のまとめが届くようになってから、「調べる」という行為に少し変化が出てきました。以前は、気になったことを自分でタブを開いて検索して、という流れでしたが、今は朝のまとめの中に「あ、この動向、今設計している案件に関係あるかも」と気づくことが増えてきた気がしています。情報が向こうから来ることで、自分の設計に問いが立ちやすくなった、という感覚に近いかもしれません。

もうひとつ、視点が広がると感じた体験があります。ある体験型手法をドキュメントに整理していたとき、「なぜこの活動を使うのか」を調査担当のサブエージェント（Claude Code 内で動く、調査に特化したAI機能）に掘り下げてもらったことがありました。そこで出てきたのが、認知負荷理論との接続や、よく使う活動の正式名称の確認だったりして、「なんとなく効果がある」と思ってきた手法に、説明できる言葉が少し増えた感じがしました。**「なぜこれを使うのか」が語れるようになる**、という変化は、設計者として地味に大きいと思っています。

「補助線を引いてもらう」という言い方が、僕にはしっくりきています。答えをもらうというより、自分の思考に別の角度からの線が入ってくる、そういう感覚です。

## 実務が加速した：ドキュメントが積み上がるようになった

ある製造業クライアントのワークショップ設計で、タイムライン作成を壁打ちしながら進めたことがありました。まず「この活動の目的・活動目標・学習目標を3層で整理しよう」という目標設定から入って、そこが固まってきたところで「じゃあ、Day1 のブロック構成に落とそう」と進み、さらに「各ブロックの詳細タイムラインを作ろう」というステップへと下りていきました。たとえば「チームの関係性を深めるセッションを午前中に置きたい」と伝えると、「それは関係性の深さを目指すのか、まず安心感をつくるのか、どちらが優先ですか？」と返ってくる。その問いに答えながら、設計の具体性が一段ずつ上がっていく感覚がありました。「ここまで詰められたのか」と少し驚いたことを覚えています。

ドキュメント化についても、変化が出てきました。

以前はやろうと思っても後回しになっていたことが、少しずつ積み上がってきました。**「記録に残す」というコストが下がったことで、設計の知見が蓄積されやすくなった**気がしています。

---

## 2つが交差した瞬間：自分の暗黙知が言語化されていた

壁打ちの中で、こんなやり取りが起きることがあります。

たとえば、タイムライン設計をしていて「この振り返りだけに20分を使いたいんです」と入力すると、「それはなぜですか？」と返ってきます。その瞬間、少し詰まります。なんとなくそうしたい、という気持ちはあるのに、理由をすぐ言葉にできない。

しばらく考えて「参加者が頭を切り替える余白が必要だから」と返すと、「その余白は、次のセッションとつなげるためですか？それとも、今日学んだことをいったん自分の中に落とし込むためですか？」と聞いてきます。「……両方だけど、どちらかというと後者かな」——そう入力しながら、自分が何を大事にしているかが少し鮮明になっていく。そんな感覚です。

意識的に暗黙知を形式化しようとしたこともあります。SECIモデルをフレームにして、自分のファシリテーション知識を体系化しようとしたのですが、いざ言葉にしようとすると表現が抽象的になり、しかも「この場面ではこう、あの文脈ではこう」と状況によって変わるので、対話がどんどん枝分かれして収束しなくなりました。うまくいかなかった体験として記憶しています。

でも後から振り返ると、日々の壁打ちの中で、自然にその行為が起きていたことに気づきました。「なぜそう設計するのか」を問われることで、なんとなくで判断していたことに言葉が貼りつく。**明示的に「形式化しよう」としなくても、設計の対話を続けることで、暗黙知が少しずつ言葉になっていく**んじゃないかと思っています。

## それでも防げなかったこと

Claude Code を使うようになって、設計の精度が上がったと思う場面は確かにありました。でも、ある案件でクライアント担当者から「想定と全然違う」という反応を受けたことがあります。人数想定がずれていたこと、設計の磨き込みに対する考え方の違い、チーム構成のイメージのギャップ——どれも、事前に確認できていれば防げたことでした。

Claude Code を使っていても、それは防げませんでした。当然といえば当然で、クライアントの文脈・担当者の期待・チームの実情は、対話の中でしか見えてきません。AI との壁打ちで設計の精度は上がっても、**人との対話で文脈をつかむ、という部分は代替されていない**んだと、そのとき改めて思いました。

「ツールが増えた分、見えなくなるリスクもある」という感覚は、今でも忘れないようにしています。設計が進んでいる実感があると、「もう大丈夫かな」と確認を省きやすくなる。そのことへの自覚は、むしろ意識的に持ち続けないといけないと思っています。

## 問いの質が、設計者の仕事

「調べること、判断すること、記録すること」という分解に戻ると、Claude Code が得意なのは調べることと記録することだと感じています。判断は一緒に考えることができても、最終的に「これでいく」と決めるのは自分です。

問いを立てるのも同じです。「この振り返りだけに20分を使いたいんです」と最初に入力したのは僕で、そこに問いが返ってきたのがあの体験でした。何を問い、何に焦点を当てるかは、設計者が持ち込む側にある、と思っています。

この数ヶ月の変化を整理すると、次の2つは確かに起きました。

でも「問いを立てる力」は、ツールに取って代わられるものではないと感じています。むしろ、壁打ちを続けることで、自分がどんな問いを持っているか、どんな判断基準を使っているかが、少しずつ見えやすくなってきた気がしています。

この数ヶ月のClaude Codeとの壁打ちを通じて、僕が「Awareness Design」と呼んでいるものの輪郭が、少し鮮明になった気がしています。知覚を言語化し、認知を揺らし、意味を問い直す——ワークショップの場でやろうとしてきたことが、自分自身の設計行為のなかでも起きていた、ということかもしれません。

**設計者の仕事の核心は、問いの質にある**——これはAIを使う前も後も変わらないと思いますし、むしろその実感を強めながら、これからも使い込んでいこうと思っています。もし試してみるなら、まず「なぜこれを選んだのか」を Claude Code に説明してみることから始めると、入りやすいかもしれません。
