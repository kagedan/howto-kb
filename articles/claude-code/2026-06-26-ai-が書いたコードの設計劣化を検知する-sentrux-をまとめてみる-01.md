## はじめに

Claude Code、Cursor、Codex のような AI コーディングエージェントを使うと、コードを書く速度はかなり上がりますが、一方で、AI に何度も修正を任せていると、次のような状態になることがあります。

* 似たような関数が別ファイルに増えまくる
* import の向きがぐちゃぐちゃになる
* 一部のファイルだけが巨大化する
* どこからどこまでが責務なのか分からなくなる
* AI に追加修正を頼んでも、前に書いたコードをうまく使えなくなる

このような状況では、テストが通っていても、型エラーがなくても、コードベース全体の見通しが悪くなります。

今回試す `sentrux` は、この問題に対して「コードベースの設計上の健康状態」を数値化するツールです。

公式には、AI コーディングエージェントのための real-time architectural sensor と説明されています。コードベースをスキャンし、依存関係やファイルの偏りなどから `quality_signal` という 0〜10000 のスコアを出します。

調べてみたところ、日本語でのドキュメントが少なかったため、この記事では、sentrux の概要、インストール、実際の使い方、AI コーディングとの組み合わせ方、注意点をまとめていきたいと思います。

## sentrux とは

[sentrux](https://github.com/sentrux/sentrux) は、コードベースの設計上の状態を測るための OSS ツールです。

特徴としては

* Rust 製
* MIT ライセンス
* 単一バイナリで動く
* macOS / Linux / Windows に対応
* tree-sitter plugin 経由で多言語に対応
* コードベースを依存関係グラフとして解析
* 0〜10000 の `quality_signal` を出力
* MCP 経由で AI エージェントから利用可能

普通の Linter や Formatter は、主にファイル単位・行単位の問題を見ます。たとえば ESLint、Biome などは、構文、スタイル、危険な書き方、未使用変数などを検出します。

一方で sentrux が見ているのは、もう少し大きい単位での「コードベースの形」です。

たとえば、

* ファイル同士の依存関係が複雑になっていないか
* 循環依存が発生していないか
* 特定のファイルだけが責務を抱えすぎていないか
* 似たようなコードが増えていないか
* AI が次に変更するときに迷う状態になっていないか

といった点です。

ここが重要で、sentrux は「コードが正しく動くか」を保証するツールではなく、「コードベースが持続的に変更し続けられる形を保っているか」を見るツールなのです。

## quality\_signal とは

sentrux は、コードベースの状態を `quality_signal` という 0〜10000 のスコアで表します。

公式ドキュメントでは、`quality_signal` は次の 5 つの指標から計算されると説明されています

* modularity
* acyclicity
* depth
* equality
* redundancy

それぞれをざっくり説明すると、以下のようにまとめられます。

| 指標 | 見ているもの |
| --- | --- |
| modularity | モジュールやファイルが適切に分かれているか |
| acyclicity | 循環依存がないか |
| depth | 依存関係の連鎖が深すぎないか |
| equality | 複雑さが一部のファイルに偏っていないか |
| redundancy | 重複や未使用のコードが少ないか |

公式ドキュメントによると、これら 5 つの指標を正規化し、幾何平均を取って 0〜10000 のスコアにしています。

```
quality_signal = (modularity × acyclicity × depth × equality × redundancy) ^ (1/5) × 10000
```

幾何平均を使っている点が面白いですね。要は、どれか 1 つの指標が極端に悪いと、全体のスコアも強く下がります。これで、「重複は少ないけど循環依存だらけ」「ファイルサイズは均等だけど依存関係が深すぎる」といった状態も適切に検知することができます。

使い方は後日まとめようと思います。

## MCP 連携

sentrux は MCP 連携も用意しています。

公式ドキュメントでは、MCP 経由で次のようなツールが提供されると説明されています。

| ツール | 役割 |
| --- | --- |
| scan | ディレクトリをスキャンし、quality signal とファイル一覧を返す |
| health | 詳細な診断を返す |
| session\_start | エージェント作業前の baseline を保存する |
| session\_end | baseline と比較し、劣化を検出する |
| rescan | 変更後に再スキャンする |
| check\_rules | .sentrux/rules.toml に基づいてルールを検証する |
| git\_stats | Git 履歴から hotspot や churn などを見る |
| dsm | 依存関係マトリクスを見る |

MCP 連携を使えば、AI エージェントが自分で sentrux を呼び出して、変更前後の品質差分を見ることができます。

## sentrux の使用にあたって

sentrux は便利ですが、使用にあたり注意しなければいけない点もあります。

まず第一に、sentrux はテストの代替ではありません。  
仕様通りに動くか、バグがないか、ユーザーにとって正しい挙動かは、従来通りテストやレビューで確認する必要があります。

また、type check の代替でもありません。  
TypeScript や Rust の型システムが検出する問題と、sentrux が見る問題は別であることに注意してください。

さらに極論を言えば、、sentrux のスコアが高いからといって、必ずしも良い設計であるとは限らりません。仕様を取り違えたまま、きれいに分割されたコードを書くことはできてしまいます。

sentrux が見ているのは、あくまでコードベースの設計上の形のみです。

そのため、ほかツールと並べた使い方としては次の位置づけになるでしょうか。。

テスト: 挙動が正しいかを見る  
型検査: 型として破綻していないかを見る  
Lint / Formatter: 局所的な書き方をそろえる  
sentrux: コードベース全体の設計劣化を見る

## 試して感じたこと

AI を用いたコーディングでは、コードを書く速度が上がる一方で、レビューする側の負担が増えることが多いです。

特に困るのは、AI が一見それっぽく動くコードを大量に出してくることでしょう。  
その場では動いても、しっかり見ると責務が混ざっていたり、既存の設計方針を無視していたりすることがしばしばです。

sentrux は、「AI が変更した後に、コードベースの形が悪くなっていないか」を確認するセンサーとして機能するでしょう。

特に、以下に当てはまるのような人には試す価値があるでしょう。

* Claude Code や Cursor にまとまった変更を任せている
* AI による変更後のレビューが重くなっている
* コードが動いているのに、だんだん変更しにくくなっている
* 循環依存や巨大ファイルを放置しがち
* CI に設計品質のチェックを入れたい

## まとめ

sentrux は、AI コーディング時代のコードベース品質を測るためのセンサーです。

ポイントは、コードが動くかどうかではなく、コードベースが変更し続けられる形を保っているかを見ることです。

昨今の開発では、AI にコードを書かせるほど、コードベースの劣化も速くなります。  
AI による変更をレビューする際には、テストや型検査だけでなく、設計上の変化も見る必要があります。

sentrux は、そのための選択肢の一つとなるでしょう。
