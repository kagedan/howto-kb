## TL;DR

* Claude Code 組み込みの `/code-review` は、`paths:` でスコープされた `.claude/rules/*.md`（path-scoped rule）を**自動ロードしない**。変更ファイルのパスが glob 判定にかけられておらず、判定対象はレビュー範囲の指定文字列（`{upstream}...HEAD`）そのものだった
* ただし必ず漏れるわけではない。レビュアーが自分の判断で rule を読みに行くことが多く、リスクは「全落ち」ではなく「**不安定な取りこぼし**」
* `/code-review` はサブエージェントではなく **メインセッションで動く**。実測で tool\_use 11回・約 100k トークン。レビュー後に他の工程が続くワークフローでは文脈を圧迫する
* 確実性が要るなら常時ロードの `CLAUDE.md` か Agent 型レビュアー。手で回すだけなら、ドキュメント未記載だが引数に「rule を読め」と自由文を添えれば通る

検証は **2026-07-18**、**Claude Code v2.1.214** 時点の結果（詳細な条件と主張の範囲は末尾）。今後挙動が変わる可能性はあるので、本記事は揮発的な実験レポートである。

## きっかけは「もうコードレビューは公式の `/code-review` でよくない？」

Claude Code には組み込みコマンドとして `/code-review` がある。このコマンドは、任意のセッションで打つとその時点の差分をレビューして、正確性のバグや簡素化の余地を報告してくれる。レビュー範囲はデフォルトで「ブランチのアップストリームより先のコミット＋ワーキングツリーの未コミット変更」で、引数でファイルパスや PR 番号、`main...my-feature` のような ref 範囲も指定できる。`--fix` を付ければ指摘をそのまま作業ツリーに適用してくれる。詳しくは公式ドキュメントの[ローカルで差分をレビューする](https://code.claude.com/docs/ja/code-review#review-a-diff-locally)を参照。

さて、自作の Claude Code ツールキット [claude-kit](https://github.com/tyabu12/claude-kit) に `orchestrate` というワークフロースキルがある。plan → issue → worktree → implement → review → PR を回すよくある流れで、その **review ステップで自作の `code-reviewer` サブエージェントを起動**する形にしている。

ここで素朴な疑問が出た。

> プロジェクト固有のレビュー観点を全部 `.claude/rules/` にまとめて、レビューは公式の `/code-review` に一本化すればよくないか？  
> そうすればワークフローのレビュー指定も `/code-review` で済む。

一見きれいな整理に見える。ただしこれが成立するかは、**`/code-review` がプロジェクトの `.claude/rules/**` を自力で読んで適用するか**にかかっている。ワークフローの一部として自動で回す以上、毎回こちらが「rule を読め」と添えるのでは意味がない。黙って実行して適用されることが要る。

公式ドキュメントからも読み取れない ClaudeCode 側の挙動なので、実験で確かめることにした。

## 前提: `.claude/rules` の「2つのロードモード」

Claude Code のプロジェクトルール（`.claude/rules/*.md`）には2つのロードモードがある。

* **常時ロード**: フロントマターに `paths:` が **無い** ファイル。`CLAUDE.md` と同じ扱いで、セッション開始時にロード
* **path-scoped**: フロントマターに `paths:` が **有る** ファイル。その glob に一致するファイルを**読むとき**にロード

```
---
paths:
  - "MyApp/MyAppTests/**"
---
# Testing Rules
（テスト関連の規約はここに、テストを触ったときだけ載る）
```

path-scoped は、常時ロードのコンテキスト予算を膨らませないための仕組みだ。規約が増えても、関係するファイルを触ったときにだけ載るので、毎ターンのトークンを食わない。

ここで気になるのが「レビュー」との相性だ。レビューは複数パスにまたがる diff を一望するので、適用されるべき rule の *和集合* が要る。トリガーが「読むとき」なら、レビューもファイルを読む以上は発火しそうにも思える。実際はどうなのか。

## 計測方法: `InstructionsLoaded` hook

最初は「規約違反を仕込んで、指摘されるかどうか」で調べようとした。しかしこの方法だと、指摘が出なかったときに**ロードされなかったのか、ロードされた上で報告に値しないと判断されたのか**が区別できない。実際この区別は後述するとおり致命的に重要だった。

Claude Code には `InstructionsLoaded` という hook があり、指示ファイルがコンテキストにロードされるたびに発火する。入力 JSON には次のフィールドが入る。

| フィールド | 内容 |
| --- | --- |
| `file_path` | ロードされた指示ファイル |
| `load_reason` | `session_start` / `path_glob_match` / `nested_traversal` など |
| `trigger_file_path` | 遅延ロードの場合、そのロードを引き起こしたファイル |
| `globs` | `paths:` フロントマターの glob |

`trigger_file_path` があるのが決定的で、「**どのファイルを読んだせいで発火したのか**」まで分かる。推測ではなく観測になる。

`.claude/settings.local.json` にこう書いた。

```
{
  "hooks": {
    "InstructionsLoaded": [
      {
        "matcher": "session_start|nested_traversal|path_glob_match|include|compact",
        "hooks": [
          { "type": "command", "command": "/path/to/log-hook.sh" }
        ]
      }
    ]
  }
}
```

hook スクリプトは標準入力の JSON をログファイルに追記するだけ。

## 観測1: glob 判定の対象がファイルではなかった

対象は個人開発の iOS アプリ（Swift / Swift Testing）。以下、リポジトリ名は `SampleSwiftApp` に伏せている。`paths: ["SampleSwiftApp/SampleSwiftAppTests/**"]` を持つ `testing.md` に「`TaskRunner` を作るスイートは `@Suite(.serialized)` 必須」「全スイートに `.timeLimit(.minutes(1))` 必須」という規約がある。どちらも既存テストが実際に従っている（テスト331ファイル中270ファイルが `.timeLimit` 準拠）。

この2つを破る新規テストファイルを1つだけ置いて `/code-review` を実行した。glob は変更ファイルのパスに完全に一致する。

結果、`testing.md` は**一度もロードされなかった**。そして `/code-review` 実行中に発生した glob 判定イベントは、毎回たった1件だけだった。

```
load_reason      path_glob_match
file_path        .claude/rules/probe-broad.md
trigger_file_path /Users/.../SampleSwiftApp/{upstream}...HEAD
globs            ['**/*']
```

`trigger_file_path` が**ファイルパスではない**。`{upstream}...HEAD` は、レビュー範囲を指す git の ref 範囲の指定そのものだ。これがファイルパスとして glob 判定に流し込まれている。

対照実験も取れた。同じ変更ファイルについて、`/code-review` を使わず普通に「この方針に沿っているかレビューして」と会話で頼んだセッションでは、こうなった。

```
path_glob_match  .claude/rules/testing.md
trigger_file_path  SampleSwiftApp/SampleSwiftAppTests/App/RoundProgressTests.swift
```

**変更ファイルそのものがトリガーになって、`testing.md` が正しくロードされている。** 機構は壊れていない。同じファイル、同じ rule で、`/code-review` の経路だけが違う。

つまり **`/code-review` は変更ファイルのパスを glob 判定にかけていない**。

### 対抗仮説: そもそも `paths:` の書式が効いていないのでは

一つ潰しておきたい可能性がある。過去に「`paths:` を YAML リストで書くとロードされない（未文書の `globs:` なら動く）」という報告があった（[#17204](https://github.com/anthropics/claude-code/issues/17204)、クローズ済み）。これが生きているなら、ロードされなかった原因は `/code-review` ではなく単に書式のせいだ。

これは同じログが排除している。今回使った rule はすべて `paths:` の YAML リスト形式（クォート付き）だが、通常セッションでは実ファイルをトリガーに `path_glob_match` が正しく発火している。書式は v2.1.214 で機能している。差が出るのは `/code-review` の経路だけ。

ただ、この対抗仮説が出てくること自体が示唆的で、rule のロード周辺は既知の不具合が多い領域でもある。

* [#16853](https://github.com/anthropics/claude-code/issues/16853): 通常の編集時でも path-scoped rule がロードされない
* [#16299](https://github.com/anthropics/claude-code/issues/16299): `paths:` を無視して rule が常時ロードされる（逆方向）
* [#76118](https://github.com/anthropics/claude-code/issues/76118): ファイルを1つ読むだけで、glob が一致しない rule まで再ロードされる（同じく逆方向）
* [#63048](https://github.com/anthropics/claude-code/issues/63048): `/code-review` がレビュー範囲を無言で `main...HEAD` にフォールバックする

過少ロードと過剰ロードの両方向に報告が散らばっている。今回の観測もその一つと見るのが自然で、だからこそ「このバージョンではこう動く」以上のことは言えない。

## 観測2: 「広い glob なら載る」ではない

diff には一切登場しない場所にプローブ規約を置き、glob の広さを変えて挙動を見た。

| `paths:` | `{upstream}...HEAD` に一致するか | ロード |
| --- | --- | --- |
| `["**/*"]` | する | **された** |
| `["*"]` | する | **された** |
| `["**/*.swift"]` | しない（拡張子が違う） | されない |
| `["SampleSwiftApp/SampleSwiftAppTests/**"]` | しない | されない |

最初は「広い glob だからロードされた」と読んだが、それは間違いだった。`**/*.swift` は `**/*` とほぼ同じ広さなのにロードされていない。正しくは「**引数文字列に一致する glob だけがロードされる**」。変更ファイルは判定に関与していないので、広さは本質ではない。

`paths: ["**/*"]` を書けば全部ロードされる、という回避策が成立しそうに見えるが、これは引数文字列への偶然のマッチに依存した挙動なので、実装が変われば黙って壊れる。勧められない。

## 観測3: それでも規約は指摘される

ここが当初の予想と最も食い違った部分だ。

`testing.md` は自動ロードされていない。にもかかわらず、レビュー結果には規約違反の指摘が出た。同一の diff に対して3回実行した結果がこれ。

| 回 | `testing.md` の自動ロード | `.serialized` 欠落の指摘 | `.timeLimit` 欠落の指摘 |
| --- | --- | --- | --- |
| 1 | なし | あり | なし |
| 2 | なし | あり | あり |
| 3 | なし | あり | なし |

3回とも、レビュアーは grep やファイル読みを経由して指摘に至り、指摘文の中で `.claude/rules/testing.md` をファイル名で名指ししている。2回目にいたっては節見出しまで正確に引用した。**自動ロードに頼らず、自分の判断で rule を読みに行っている。**

そして同じ rule ファイルでも、拾う項目が回ごとに違う。`.serialized` は3回とも、`.timeLimit` は1回だけ。

言い換えると、リスクは「規約チェックが静かに全落ちする」ではなく「**たまに落ちる**」だ。実務的にはこちらのほうが厄介で、たいてい拾ってくれるぶん、落ちたことに気づきにくい。

## 観測4: ロードされても執行されるとは限らない

もう一つ、設計ミスから偶然分かったことがある。

最初のプローブでは「推測不可能な恣意的な規約」を作った。ドキュメントを読まなければ絶対に知りようがない規約にすれば、指摘されたときの証拠能力が高いと考えたからだ。たとえば「全 Swift ファイルの先頭行に `// reviewed-marker` コメントが必須」といったもの。

これらはロードされた回でも、指摘には一切現れなかった。代わりにレビュアーは、レビュー本体とは別枠でこう報告してきた。要約すると「この3つの規約は Swift 全344ファイル中0件しか準拠しておらず、git 管理下にもなく、`CLAUDE.md` の一覧にも記載がない。実際には運用されていない規約と判断し、従わなかった」。

つまり「推測不可能」にしたことで、同時に「捏造された規約」にもなってしまい、信憑性判定を測る実験に化けていた。実験としては失敗だが、副産物として重要なことが分かる。**ロードは考慮を保証するだけで、執行は保証しない。** レビュアーは規約をコードベースと突き合わせ、形骸化していると判断すれば従わない。

これは欠陥というより美点だと思う。ただし、後述する処方箋が保証できる範囲も、同じ理由で「考慮まで」だということは押さえておきたい。

## 帰結: 確実に効かせたい規約はどこに置くか

冒頭の「`/code-review` に一本化」案は、これで棄却できる。path-scoped rule はレビュー時に自動ロードされないので、**保証がない**。たいてい拾ってくれるが、それは仕組みではなくレビュアーの気まぐれに乗っているだけだ。

選択肢はこうなる。

* **`CLAUDE.md` に移す** → 常時ロードなので確実に載る。ただしコンテキスト予算と衝突する。規約が多いほど毎ターン課金され、path-scoped 化の逆行になる
* **`REVIEW.md` に書く** → このファイルは公式ドキュメント上、GitHub App 経由の Code Review をカスタマイズする手段として説明されていて、ターミナルで実行する `/code-review` の節には登場しない。手元のレビューでは当てにできない
* **`paths: ["**/*"]` にする** → 観測上は動くが、引数文字列への偶然のマッチに依存している。勧めない
* **`/code-review` に自由文で指示する** → 実は効く。後述

### 自由文の指示は通る

公式ドキュメントに載っている `/code-review` の引数は、レビュー対象（ファイルパス、PR 番号、ブランチ、ref 範囲）と努力レベル、`--comment` / `--fix` といったフラグだけだ。それを読んで、自由文は渡せないものと思い込んでいたが、実際に打ってみたら通った。

```
/code-review high この diff に一致する .claude/rules/*.md を読んでから評価して
```

レビュアーは指示を受け取り、レビュー結果の中で「指示どおり、パスにマッチする `.claude/rules/testing.md` を読んだ上で評価した」と明言した。参照した rule のファイル名まで報告している。

ただし、このときも自動ロードは発生していない（hook のログは0件）。指示に従ってツールで読んだだけで、ロード機構は動いていない。挙動としては「レビュアーが自主的に読む」の延長線上にあり、確実性が上がるのは指示があるぶんだけだ。

手で回すぶんにはこれが一番安上がりな回避策になる。逆に、ワークフローに組み込んで自動で回す用途では使えない。毎回同じ文字列を添えるなら、それは結局「指示付きレビュアー」を自前で持つのと同じことで、だったら最初から Agent にしたほうがいい。

### Agent 型の優位はどこにあるのか

残る現実解は、「変更ファイルに一致する `.claude/rules/*.md` を読め」と定義に書いておける Agent 型レビュアーだ。注意したいのは、Agent も path-scoped rule を自動ロードしているわけではないこと。プロンプトにそう書いてあるから読んでいるにすぎない。

つまり優位性の源泉は「指示を注入できる口があること」ではない。口は `/code-review` にもあった。実際の差は3つある。

**1. `/code-review` はメインセッションで動く。** セッション記録を確認したところ、サブエージェントとしては実行されていなかった。grep の結果もファイルの中身も、全部メインの文脈に積まれる。レビューしか実行していない新規セッションで、tool\_use 11回・文脈は約 100k トークンまで膨らんだ。レビューが最終工程なら構わないが、その後に PR 作成などが続くワークフローでは、後段のために残しておきたい文脈を圧迫する。サブエージェントならこれを外に追い出せる。

**2. 出力の形が安定しない。** 今回の実行では、JSON 配列で返ってきた回と箇条書きで返ってきた回があった。「`PASS | FAIL` の行を出せ」といった契約を守らせて、その結果でワークフローを分岐させる用途には向かない。

**3. 指示が毎回同じ内容で入る。** Agent の定義はリポジトリに置けるので、レビューでき、共有でき、変更が履歴に残る。手で打つ引数はそのどれでもない。

そしてこれが保証するのも、観測4のとおり**考慮までで、執行ではない**。実際に運用されている規約なら棄却されないだろうが、そこは Agent 化しても変わらない。

claude-kit ではこの結論を受けて、`orchestrate` の Step 4 に汎用の [`code-reviewer` エージェント](https://github.com/tyabu12/claude-kit/blob/main/agents/code-reviewer.md)を用意している。やっていることは：

1. `git diff --name-only` で変更ファイルを取る
2. 各 `.claude/rules/*.md` の `paths:` を見て、**変更ファイルに一致するもの + `paths:` 無しのものだけ**読む（全部読むと大きなルール集で予算を食う）
3. `CLAUDE.md` ＋読んだ rule で評価し、`PASS | FAIL` の verdict を出す（orchestrate のゲートがパースする契約）

裏を返すと、この設計は「rule を読め」というプロンプト1文に依存している。将来この一文が編集で落ちると、カバレッジが無音で消える。

## 自分の環境で確かめる手順

同じ確認は数分でできる。

1. 上の `InstructionsLoaded` hook を `.claude/settings.local.json` に仕込む。ログが取れれば、指摘の有無から推測する必要がなくなる
2. path-scoped な rule にしか書いていない規約を1つ選ぶ。**コードベースが実際に従っている規約**にすること。恣意的な規約は、観測4のとおり「運用されていない」と判断されて弾かれる
3. その規約を破る変更を作る。新規ファイルの場合、未追跡のままだと `git diff` に出ずレビュー対象から漏れるので、`git add -N` で intent-to-add しておく
4. diff にヒントを残さない。「この規約は◯◯.md にある」といったコメントを書くと、レビュアーがポインタを辿っただけなのか区別できなくなる
5. `/code-review` を実行し、ログの `load_reason` と `trigger_file_path` を見る

## バージョン依存であることの注意

本記事の検証は **2026-07-18**、**Claude Code v2.1.214**、Opus 4.8、effort high のセッションから実行した結果である。今後アップデートで挙動が変わる可能性は全然あり得る。

`trigger_file_path` に ref 範囲の文字列が入るのは、仕様というより実装の癖に見える。修正されれば挙動は変わるし、そのときは `/code-review` に寄せる判断のほうが正しくなる。

主張の範囲も正確にしておく。確認したのは「path-scoped rule は `/code-review` で自動ロードされない」までで、観測回数は glob ログで n=2、指摘の有無で n=3 と少ない。`ultra` を付けたクラウド実行は別物なので対象外。`paths:` 無しの rule は `session_start` でロードされることをログで確認済み。

## おわりに

「公式の `/code-review` があるんだから、それに寄せればいい」。直感的にはそう思うし、メンテを考えると極力そうしたい。ただ path-scoped rule というコンテキスト予算の最適化が、レビュー自動化と噛み合っていなかった。

個人的にいちばんの収穫は、計測方法を変えたことだった。「指摘されるか」で見ていたときは、ロードされなかったのか判断されなかったのかが区別できず、結論を1回の陰性結果から急いでいた。hook でロードを直接観測したら、機序も、当初の解釈の誤りも、一度に見えた。**検出器の出力ではなく、検出器の入力を観測する**。当たり前だけど、やってみるまで気づかなかった。

同じように `.claude/rules/` でレビュー観点を管理している人の参考になれば。挙動が変わったら、ぜひ教えてください。
