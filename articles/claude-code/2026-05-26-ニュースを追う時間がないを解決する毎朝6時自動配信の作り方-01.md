朝、コーヒーを淹れながらニュースサイトを巡回する——日経、業界専門紙、競合のプレスリリース、Twitterのトレンド。気になった記事を保存して、後で読もうと思ったまま夕方になり、結局その日のキャッチアップは断片的に終わる。1日30分〜1時間、月にすると20時間以上が「情報の見落としを怖がって右往左往する時間」に消えています。

これ、**Claude Code Routinesで「毎朝6時にSlackへ要約10本を自動投稿」させると、朝のチェック時間が5分に圧縮できます**。Routinesは、Claude Codeに搭載された「定期実行」機能。指定した時刻に決まった処理を自動で走らせることができます。今日はその設定手順を、ニュース自動要約配信を題材に全部公開します。

## このワークフローで何ができるか

**ビフォー: 毎朝のニュース巡回に30〜60分**（複数サイト/SNS巡回・記事ピックアップ・後で読むへ保存）  
**アフター: 起きたらSlackに要約済み10本が並んでいる状態（巡回0分・確認5分）**

月20時間の情報収集時間が月2時間以下に。空いた時間は、収集した情報をもとに**判断・行動**する時間に回せます。

向いている人

* 業界トレンドや競合動向のキャッチアップが業務に直結するマーケ・経営企画・DX推進担当
* ニュース・プレスリリース・SNSトレンドを毎日複数ソース見ているコンサル・士業
* 情報感度が成果に直結する個人事業主・スタートアップ経営者

向いていない人

## 手順

### ステップ1: 情報ソース（RSS／検索キーワード）を決める

Routinesを組む前に、毎朝拾いたいソースを決めます。下記3パターンの組合せが扱いやすいです。

```
【ソース1: 業界専門サイトのRSSフィード】
- 例: 日経クロステック / TechCrunch Japan / MarkeZine
- RSS URL を3〜5本リストアップ

【ソース2: 検索キーワード】
- 例: 「生成AI 中小企業」「インボイス制度 改正」「自社業界名 補助金」
- Google検索の「過去24時間」フィルタ相当を狙う

【ソース3: 競合・ベンチマーク企業のプレスリリース】
- 例: PR TIMES の特定企業ページ、上場企業のIR適時開示
```

3〜10本に収まる程度で十分です。ソースを欲張ると要約が薄まり「結局全部読まないといけない」状態に戻ります。

### ステップ2: CLAUDE.mdに「要約ルール」と「Slack投稿フォーマット」を書く

新規プロジェクトを作り、CLAUDE.mdに要約方針と出力フォーマットを固定します。

```
あなたは毎朝のニュース要約担当です。
指定された情報ソースから過去24時間以内の記事を収集し、
下記ルールで要約して、Slack Incoming Webhook で送信します。

【要約ルール】
- 1記事あたり3行（120字以内）で要約
- 1行目: 何が起きたか（事実）
- 2行目: なぜ重要か（自社/読者にとっての含意）
- 3行目: 元記事URL

【選別基準】
- 自社業界（中小企業向けDXツール）に直接関係するものを優先
- 同じテーマで複数ソースから出ているものは最も詳細な1本に絞る
- スポンサード/PR記事は除外

【Slack投稿フォーマット】
チャンネル: #morning-news
冒頭ヘッダ: 「📰 朝のニュース要約 YYYY-MM-DD」
本文: 上位10本をリスト形式で投稿
末尾フッタ: 「※自動配信 / 元記事の検証は各自お願いします」

【NG事項】
- 推測で見出しや要約を盛らない
- 出典のない数字や引用を作らない
- 同じ情報源から同じテーマを2本以上拾わない
```

「3行120字」「上位10本」「元記事URL必須」を固定するのがコツです。要約が長くなると朝のスキャンに時間がかかり、結局「自動化前と同じ」状態に戻ります。

### ステップ3: Slack Incoming Webhookを準備する

Slack側で投稿先チャンネルを決め、Incoming Webhook URLを発行します。

```
1. Slackワークスペースの「App」検索で「Incoming Webhooks」をインストール
2. 投稿したいチャンネル（例: #morning-news）を選択
3. 発行されたWebhook URL（https://hooks.slack.com/services/T.../B.../xxx）をコピー
4. プロジェクトの .env または環境変数に SLACK_WEBHOOK_URL として保存
```

Webhook URLは**シークレット情報**なので、CLAUDE.mdに直書きせず必ず環境変数経由で渡してください。GitHubに誤コミットすると第三者がチャンネルに投稿し放題になります。

### ステップ4: Routinesに「毎朝6時実行」を登録する

Claude Code Routinesで、毎朝6時にCLAUDE.mdの処理を自動実行するスケジュールを登録します。

```
Routines設定:
- 名前: morning-news-digest
- スケジュール: 毎日 06:00 JST（cron表記: 0 6 * * *）
- 実行内容: CLAUDE.mdに従って、過去24時間のニュースを収集→要約→Slack投稿
- タイムアウト: 5分
- 失敗時通知: Slackの別チャンネル（#routines-alert）にエラー内容を投稿
```

これだけで、翌朝6時から自動配信が始まります。手動で動作確認したいときは、Routinesの「今すぐ実行」ボタンで即時テストできます。

### ステップ5: 最初の1週間でチューニングする

自動配信が動き始めたら、最初の1週間は毎朝Slackを見て「もっと拾ってほしかった話題」「拾わなくてよかったジャンル」をメモします。

```
チューニング項目:
- 選別基準を絞る/広げる（業界の定義を狭く再定義する等）
- ソースを追加/削除（読まれていないRSSは外す）
- 要約の長さ調整（3行→2行へ短縮 等）
- 投稿時刻の微調整（6:00 → 7:30 等、自分の起床時間に合わせる）
```

CLAUDE.mdを書き換えるだけで翌日から反映されます。1週間で「自分の業務に最適化されたニュース秘書」状態に到達します。

## 応用と注意点

**応用1: 配信先をSlack以外に切り替える**  
Slack Webhookの代わりに、Discord・Microsoft Teams・LINE Notify・自分宛てメール（SMTP/SendGrid）に切り替え可能です。CLAUDE.mdの【投稿フォーマット】の送信先と認証情報だけ差し替えれば、同じワークフローがそのまま動きます。

**応用2: 週次/月次レポートも自動化できる**  
日次の要約を蓄積して、金曜18時に「今週の業界トレンド3本」を別Routinesで配信、月末に「今月のキー動向まとめ」を自動生成、まで一気通貫で組めます。Routinesは複数登録できるので、日次・週次・月次を並行運用するのが王道です。

**注意点1: ニュースの一次確認は人間が行う**  
Claudeが要約した記事は、原文の文脈やニュアンスが落ちている可能性があります。重要な経営判断や顧客提案に使う情報は、必ず元記事URLを開いて一次情報で確認してください。要約は「読むべき記事を選別するための入口」と位置付けるのが安全です。

**注意点2: 著作権とサイトの利用規約**  
ニュースサイトの本文を大量に取得・転載することは、著作権法および利用規約に抵触する可能性があります。要約は「見出し＋3行サマリー＋元記事へのリンク」にとどめ、本文の長文転載は避けてください。RSSフィードが公開されているソースは原則「要約・引用＋リンク」が想定されていますが、各サイトの規約は事前確認をおすすめします。

**注意点3: Webhook URLとAPI鍵の管理**  
Slack Webhook URLは漏洩すると第三者がチャンネルに投稿できます。GitHubへのコミット時は必ず `.gitignore` に `.env` を追加し、誤って公開しないこと。誤公開した場合はSlack管理画面から即時Webhookを再発行してください。

## まとめ

* Claude Code Routinesを使えば「毎朝6時に業界ニュース要約10本をSlack配信」が自動化できる
* 鍵は、CLAUDE.mdに「要約ルール（3行120字）」と「選別基準（自社業界優先）」を固定すること
* 配信先はSlack/Discord/Teams/メール何でも切替可能、日次に加えて週次・月次レポートも同じ仕組みで組める
* 要約は「読むべき記事の入口」、最終的な一次情報の確認は人間が行う

朝のニュース巡回1時間が5分になると、空いた55分は「情報を集める時間」から「集めた情報をもとに考える時間」に変わります。経営判断・企画立案・顧客提案の質が上がる本当の理由は、「考える時間」を取り戻せるかどうかです。
