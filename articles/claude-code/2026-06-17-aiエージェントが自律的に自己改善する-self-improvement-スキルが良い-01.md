私は Claude Code を使って開発しているのですが、「さっき同じこと指摘したのに、また同じミスしてる」というケースがよくあります。

同じセッションなら覚えてくれます。でもセッションをまたぐと忘れる。チームの別メンバーが同じエージェントを動かしたら、また一から同じことを教え直す。**学びがその場限りで蒸発していく**わけです。その度に CLAUDE.md に追記するのも面倒です。

これを解決してくれたのが、今回紹介する `self-improvement` という Agent Skill でした。導入してみて布教したいと思えるものだったので、紹介させていただきます。

## このスキルは何をするのか

ひとことで言うと、**会話の中で生まれた「学び」を構造化してリポジトリに貯めていくスキル**です。

エージェントが以下のような瞬間を検知すると起動します。

* ユーザーがエージェントの誤りを訂正したとき（「いや、実際は…」）
* 存在しない機能を要求されたとき（feature request）
* エージェントの知識が古い・間違っていると判明したとき（knowledge gap）
* 繰り返し作業の中でより良いやり方を見つけたとき（best practice）
* コマンドが失敗し、調査して解決したとき（error）

起動すると、学びの種類を判定して `.learnings/` 配下の Markdown ファイルに、ID・日時・要約・詳細・対応案を構造化して追記します。たとえば訂正系ならこんな形です。

```
## [LRN-20260617-001] correction

**Logged**: 2026-06-17T10:00:00Z
**Priority**: high
**Status**: pending
**Area**: backend

### Summary
バッチ処理のリトライ回数は環境変数で外出しする規約だった

### Details
ハードコードで実装したところ、既存の規約に反すると指摘を受けた。
本プロジェクトではリトライ系の閾値はすべて ENV で管理している。

### Suggested Action
リトライ回数を ENV から読むよう修正し、デフォルト値はコードに残す
```

ポイントは、**これがただのメモではなくリポジトリに track されるファイルだ**ということ。コミットすれば、その瞬間にチーム全員（と全員のエージェント）の共有知識になります。

## なぜおすすめなのか

### 1. 「教えたことが消えない」だけで体験が変わる

エージェントとの開発でいちばん消耗するのは、**同じ説明を繰り返すこと**だと思っています。

`self-improvement` を入れると、一度教えたプロジェクト固有の作法が `.learnings/` に残ります。次のセッションで大きな作業に入る前に、エージェントは自分でこの学びを参照してから動き出す。つまり昨日教えたことを覚えてくれているわけです。

地味ですが、この差は使ってみると本当に大きいです。

### 2. ナレッジ共有が「コミット」だけで完結する

チーム開発でナレッジ共有といえば、Notion に書く・Wiki にまとめる・口頭で伝える…と、だいたい**別フローのコスト**がかかります。そして大抵は風化します。

このスキルの良いところは、**学びの保存先がコードと同じリポジトリ**だという点。

* 学びを生んだ作業の PR に相乗りさせれば、追加の共有作業はゼロ
* レビューで `.learnings/` の差分も一緒に見られる
* 「誰がいつ何を学んだか」が Git 履歴として残る

ドキュメント管理を頑張るのではなく、**ふだんのコミットフローに学習が乗っかってくる**。この設計が秀逸です。

### 3. 溜めた学びが「恒久ルール」に昇格していく

`.learnings/` はあくまで生ログです。同じ種類の学びが何度も記録される（目安：3 回以上／複数タスクにまたがる）と、それは一度きりの記録ではなく**プロジェクトの恒久ルール**にすべきサインです。

このとき、蓄積された生ログを簡潔な「予防ルール」へ蒸留して、`AGENTS.md` や `CLAUDE.md`、コーディング規約へ転記します。

**「記録のハードルは限りなく低く、昇格は繰り返しが証明してから」** という二段構えになっているので、ノイズに埋もれず、かつ拾い漏らさない、よくできた仕組みだと思います。

## どういう人におすすめか

### AI エージェントを毎日活用している人

たまに使うだけなら学びを貯めるメリットは薄いです。でも **日常的にエージェントと開発している人** にとっては、教えたことが積み上がっていく効果が複利で効いてきます。

### チームで同じエージェント設定を共有している人

`AGENTS.md` や `CLAUDE.md` をチームで共有しているなら、相性は抜群です。誰か一人がハマって解決した学びが、コミット一発で全員のエージェントに行き渡る。**個人の経験がチームの資産になる**のは大きいです。

### プロジェクト固有の「暗黙知」が多い現場の人

歴史のあるコードベースほど、「ここはこう書く」「これは触らない」といった、コードを読むだけでは分からない暗黙知が溜まっています。そういう文脈こそ `.learnings/` に向いています。新しく入ったメンバー（人間でもエージェントでも）のオンボーディングコストを下げる効果も期待できます。

## 導入のしかた

Agent Skills CLI で取得できます。

```
npx skills add pskoett/pskoett-ai-skills/skills/self-improvement
```

あとは学びの置き場所を用意するだけ。

```
mkdir -p .learnings
# skill 同梱の assets/ から雛形をコピー
# LEARNINGS.md / ERRORS.md / FEATURE_REQUESTS.md
```

基本はこれだけです。エージェントが訂正や失敗を検知すると、自動でここに学びを追記してくれるようになります。

### サービスごとに学びを分けることもできる

ひとつ便利な点として、**保存先の `.learnings/` は 1 箇所に固定する必要がありません**。

たとえば 1 つのリポジトリ（モノレポなど）の中にフロントエンド・バックエンドのように複数のサービスが同居している場合、**それぞれのサービス配下に `.learnings/` を置いて、触った対象に応じて学びを振り分ける**こともできます。

```
repo/
├─ frontend/.learnings/   # フロントの学び
└─ backend/.learnings/    # バックエンドの学び
```

こうすると、学びがそのサービスのコードと同じ場所に置かれるので、文脈がブレません。振り分けのルール自体はスキル本体ではなく `AGENTS.md` 側に書いておけば、スキルを更新しても自前ルールは消えません。

このあたりの「どこに何を振り分けるか」はプロジェクトの構成しだいです。単一サービスなら素直にルート直下の `.learnings/` ひとつで十分です。

## 一点だけ注意：「身に覚えのない差分」問題

導入したら、チームに **`.learnings/` の存在を周知しておく**のをおすすめします。

この仕組みを知らないメンバーが `.learnings/` の差分を見ると、「誰だこれ書いたの？」「謎の変更だから消そう」となりがちだからです。せっかくの学びが破棄されては元も子もありません。

対策として、

* エージェントが学びを記録したら、必ずユーザーに「記録しました」と通知させる
* 未コミットの `.learnings/` 差分を見つけても勝手に破棄しないルールを `AGENTS.md` に明記する

このあたりをルール化しておくと安心です。また、Prettier などの整形 CI がある場合は、雛形のフォーマットと衝突して CI が落ちることがあるので、`.learnings/` を formatter の対象外にしておくと良いです（地味にハマりポイントでした）。

## 安全性・信頼性について

サードパーティ製のスキルなので、気になるところだと思います。特にスキルは一読してから取り入れないと、プロンプトインジェクションされ得る恐れもあります。私も導入前に中身を確認したうえで使っていますが、このスキルに関しては比較的安心して使える部類だと感じています。

### スキルの実体が Markdown なので、依存がない

そもそも Agent Skill の正体は、エージェントへの指示が書かれた **Markdown ファイル**です（`SKILL.md`）。npm パッケージのように何かをインストールするわけでも、バイナリを実行するわけでもありません。

* `node_modules` も依存ライブラリも増えない
* 中身は全部テキストなので、**読めば何をするスキルか一目で分かる**
* 脆弱性を抱えた依存を間接的に持ち込む、といった心配がない

「導入＝テキストを置くだけ」なので、レビューコストも信頼性の評価コストも非常に低く済みます。なお、自動起動用のフックスクリプト（`activator.sh` など）は同梱されていますが、これらは**任意の opt-in 機能**で、有効化しなくてもスキル自体は問題なく動きます。

### やることが「Markdown を書く」だけに閉じている

このスキルの本質的な動作は、`.learnings/` 配下の Markdown に学びを追記することだけです。

* 本番環境やデプロイには一切触れない
* 学びを外部サービスへ送信したりしない（保存先はあくまでローカルのリポジトリ内）
* 任意コマンドを勝手に実行して何かを壊す、といった性質のものではない

「副作用の範囲が極めて狭い」というのは、信頼性を語るうえで地味に大事なポイントです。

### すべての変更が Git レビューを通る

学びは普通の Markdown 差分としてコミットされるので、**何が記録されたかは PR レビューで人間の目を通せます**。エージェントが裏で何かをこっそり書き換える、という不透明さがありません。記録したらユーザーに通知させるルール（前述）と組み合わせれば、「いつ・何が・なぜ記録されたか」を完全に追跡できます。

### 導入元を固定して管理できる

スキル本体は**無改変で取り込み（vendoring）、取得元とハッシュをロックファイルで管理**するのがおすすめです。こうしておくと、

* いま動いているスキルが「どのバージョンの何か」が常に明確
* アップデートは取り直し＋差分レビューという明示的な操作でしか起きない
* プロジェクト固有の振り分けルールはスキル外（`AGENTS.md` 側）に置くので、更新で勝手に消えない

供給元を握ったうえで使えるので、サプライチェーン的な不安もコントロール下に置けます。

### 一点だけ意識しておきたいこと

`.learnings/` はエージェントが後から読み込む情報源でもあります。つまり、**そこに不適切な内容が混入すると、後続のエージェントの挙動に影響しうる**ということでもあります。

とはいえこれは「Git でレビューしてからコミットする」という当たり前の運用で十分に防げます。記録される学びも結局はコードと同じ差分なので、**いつものレビュー基準をそのまま当てればよい**という話に落ち着きます。むやみに恐れる必要はありませんが、`.learnings/` の差分も他のコードと同じ目線で見る、という意識だけ持っておくと万全です。

## おわりに

AI エージェントは事実をそつなく返してくれますが、**そのプロジェクトの文脈や作法を「学んで覚えていく」のは、放っておくと起きません**。

`self-improvement` は、その学びを貯めて・共有して・恒久ルールへ昇格させる流れを、ふだんのコミットフローに自然に溶け込ませてくれるスキルでした。教えたことが勝手に積み上がるエージェントを作りたい人は、ぜひ一度試してみてください。

エージェントを単なる便利ツールではなく、**プロジェクトの文脈ごと一緒に育っていく開発パートナー**にしていくうえで、こうした「学びを資産化する仕組み」はこれからますます効いてくると感じています。
