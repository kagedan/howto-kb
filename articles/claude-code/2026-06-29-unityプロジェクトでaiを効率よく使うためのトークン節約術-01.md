## はじめに

開発中、または運用中のUnityプロジェクトでAIコーディングエージェントを使うと、調査・実装・レビューの速度はかなり上がります。

一方で、プロジェクトが大きくなるほど次の問題が起きやすくなります。

* 関係ないフォルダまで読みに行く
* `Library/` や `Temp/` などの生成物を読もうとする
* Scene / Prefab / ScriptableObject の巨大YAMLを丸ごと読む
* Addressablesの生成物やログを読み込んでトークンを消費する
* 毎回同じプロジェクト説明をチャットで繰り返す
* `AGENTS.md` / `CLAUDE.md` に書きすぎて、逆に守られにくくなる

AI活用で重要なのは、単に高性能なモデルを使うことではありません。

**AIに何を読ませるか、何を読ませないかを設計すること**です。

この記事ではUnityプロジェクトを基準に、AIエージェントのトークン消費を抑えるための実務寄りの運用を整理します。最後に、Unity / UE5 のどちらでも使いやすい `AGENTS.md` と `CLAUDE.md` のテンプレートも載せます。

## AGENTS.md / CLAUDE.md は英語で書くのがおすすめ

この記事のサンプルでは、`AGENTS.md` や `CLAUDE.md` を英語で書きます。

理由は、主に次の3つです。

1つ目は、コードベース自体が英語中心だからです。クラス名、メソッド名、ファイル名、API名、エラーメッセージ、CLIコマンドは基本的に英語です。指示も英語に寄せると、コード上の名前と自然に対応しやすくなります。

2つ目は、Codex / Claude Code / Gemini CLI / Cursor など複数のAIエージェントで使い回しやすいからです。日本語でも動きますが、チーム共通の機械向けルールとしては英語のほうが無難です。

3つ目は、短く書きやすいからです。`Do not edit generated files.` のようなルールは、日本語で丁寧に書くより短く、命令としても明確です。

4つ目は、結果としてコンテキスト量を少し抑えられる場合があるからです。日本語より英語のほうが常に大きく減る、という話ではありません。モデルやTokenizer、文章の書き方によって変わります。ただ、AI向けの短い命令文に限れば、英語のほうが短くまとまりやすく、余計な説明も削りやすいです。

ただし、英語にしただけで劇的にトークンが減るわけではありません。大きく効くのは、言語そのものよりも、重複を減らすこと、読ませないフォルダを明確にすること、巨大ファイルを丸読みさせないことです。

そのため、すべて英語にする必要はありません。人間向けの補足、社内用語、背景説明は日本語で構いません。  
おすすめは、**AIに必ず守ってほしい短いルールは英語、補足説明は日本語**です。

ただし、チーム全員が英語ルールをレビューできない場合は、無理に英語だけに寄せないほうが安全です。AI向けには英語で短く、人間が確認すべき背景や例外は日本語で補足する、くらいが実務では扱いやすいです。

## まず押さえる前提：指示ファイルは強制設定ではない

`AGENTS.md` や `CLAUDE.md` は便利ですが、最初に誤解を潰しておく必要があります。

これらは、AIエージェントに読ませる**コンテキスト**です。  
強制力のある設定ファイルではありません。

つまり、

と書いても、AIが絶対に `Library/` を読まない保証はありません。

もちろん、書かないよりはかなり良いです。探索の初動を誘導できますし、無駄な読み込みを減らせます。しかし、これは「お願い」や「作業方針」に近いものです。

本当に守らせたい場合は、指示ファイルだけに頼らず、次のレイヤーを併用します。

* `.gitignore`
* `.sembleignore`
* Claude Code の `claudeMdExcludes`
* Claude Code の hook
* Codex の sandbox / approval / config
* CI / lint / test
* 生成物をそもそもGit管理しない運用

特にトークン節約の第一防衛線は `.gitignore` です。  
Unityの `Library/` や `Temp/`、UEの `Intermediate/` や `Saved/` などは、基本的にGit管理しないようにします。

AIエージェントはローカルディスク上の未追跡ファイルを読める場合があるため、`.gitignore` だけで完全に防げるわけではありません。それでも、「検索対象にしない」「インデックスしない」「Git差分に出さない」という意味では非常に重要です。

## AIに全部読ませない

大きなプロジェクトで一番やってはいけないのは、AIに対して雑にこう頼むことです。

```
このUnityプロジェクトを見て、問題を直してください。
```

これは人間に対して「この会社の全ファイルを読んで、なんか改善して」と言うようなものです。  
AIは律儀に広く探そうとして、関係ないファイルまで読みに行きます。

良い頼み方は、最初から探索範囲を絞ります。

```
ログイン画面の入力バリデーションだけを調査してください。
まず Semble で関連ファイルを探し、候補ファイルと理由を出してください。
その後、必要最小限のファイルだけ読んでください。
```

重要なのは、最初に **調査対象** と **探索手順** を指定することです。

* どの機能か
* どのフォルダを見るべきか
* どのフォルダを見ないべきか
* まず何で検索するか
* いきなり修正してよいか
* 先に調査結果を出すべきか

このあたりを明示するだけで、無駄な読み込みはかなり減ります。

## Unityでトークンを消費しやすいもの

Unityプロジェクトでは、AIに読ませると重いものが多くあります。

代表例は次の通りです。

| 対象 | 問題 |
| --- | --- |
| `Library/` | 自動生成物が大量にある |
| `Temp/` | 一時ファイルで読む価値が低い |
| `obj/` | C#ビルド成果物 |
| `Logs/` | 大きくなりやすく重複も多い |
| `*.csproj` / `*.sln` | Unity生成物なので通常は直接編集しない。必要時だけ読む |
| Scene / Prefab | YAMLが巨大になりやすい |
| ScriptableObject | データ量次第で巨大化する |
| Addressables catalog / bundle | 調査対象としては重い |
| StreamingAssets配下の生成物 | バイナリや生成データが混ざりやすい |
| 大量CSV / JSON / YAML | 丸読みするとすぐトークンを消費する |

逆に、まず見てほしい場所はだいたい決まっています。

| 対象 | 理由 |
| --- | --- |
| `Assets/Scripts/` | ゲームロジック本体 |
| `Assets/Editor/` | 独自Editor拡張 |
| `Assets/Plugins/<自社名>/` | 自社ライブラリやランタイム |
| `Packages/manifest.json` | 使用パッケージ確認 |
| `ProjectSettings/` | Unity設定。読む価値は高いが、変更は慎重に扱う |
| `*.asmdef` | アセンブリ分割や依存関係 |
| `Assets/AddressableAssetsData/` | Addressables設定確認 |

`AddressableAssetsData` 配下にもバージョンや設定によって生成物が混ざるため、重い `.bin` やカタログは丸読みせず抽出を優先します。

## まず検索にSembleを使う

大きなコードベースでは、`grep` や `rg` で文字列検索してから複数ファイルを読むだけでも、かなりのトークンを使います。

そこで、ローカルのセマンティックコード検索を使うと効率が上がります。例として、この記事では **Semble** を使う前提にします。

Sembleの詳しい導入方法や実測事例は、この記事の最後に[参考](#%E5%8F%82%E8%80%83)としてまとめています。Sembleを初めて見る場合は、先にそこへ飛んで概要だけ確認しても大丈夫です。

Sembleは、AIエージェント向けのコード検索ライブラリです。自然言語クエリで関連するコード片を探し、ファイル全体ではなく必要なスニペットだけを返します。MCP、CLI、`AGENTS.md` 経由の指示などで利用できます。

公式では `grep + read` と比べたトークン削減が示されています。また、Zennにも実測記事があり、Claude Code のコード検索で `semble` を使って読み取り量を削減した一例が紹介されています。

こうした数値は、対象リポジトリ、インデックス対象、検索クエリ、AIエージェント側の読み方に大きく依存します。本文では「常に同じ削減率が出る」という意味ではなく、検索先行の考え方を実現する手段の一つとして扱います。

ただし、効果はプロジェクト構成、インデックス対象、検索クエリの質に依存します。`.gitignore` や `.sembleignore` を整えずに生成物まで検索対象にすると、検索結果自体がノイズになります。

ただし、Sembleも万能ではありません。

* 検索クエリが悪いと関係ない結果が返る
* 生成物や巨大データまでインデックスするとノイズになる
* 最終確認には元ファイルを読む必要がある
* API名や正確な識別子検索では `rg` のほうが速いこともある

実務では、次の使い分けが良いです。

```
意味で探す: Semble
正確な文字列で探す: rg / grep
巨大ファイルを見る: スクリプトで抽出
最終確認する: 必要な元ファイルだけ読む
```

`AGENTS.md` にはこう書いておくと扱いやすいです。

```
## Search policy

Use Semble first for semantic code search when available and suitable.
Use rg/grep only when exact string matching is needed or Semble results are insufficient.
Do not scan the entire repository without explaining why.
```

## フォルダ構造を事前に書く

AIに毎回プロジェクト構造を推測させるのは無駄です。

`AGENTS.md` に、最低限のフォルダ構造を書いておきます。

```
## Project layout

- `Assets/Scripts/`
  - Runtime gameplay code.
- `Assets/Editor/`
  - Unity Editor extensions.
- `Assets/Plugins/CompanyName/`
  - Company-owned runtime/editor packages.
- `Assets/Scripts/Generated/`
  - Generated code. Do not edit directly.
- `Assets/AddressableAssetsData/`
  - Addressables settings. Avoid generated binary files.
- `Packages/`
  - Unity package manifest and local packages.
- `ProjectSettings/`
  - Unity project settings.
```

この程度で十分です。  
全フォルダを細かく書く必要はありません。

重要なのは、AIが最初に見るべき場所と、見なくてよい場所を判断できることです。

## AGENTS.md / CLAUDE.md に書くこと

`AGENTS.md` / `CLAUDE.md` は仕様書ではありません。  
AIエージェント用の作業導線です。

書くべきことは、だいたい次の範囲です。

* プロジェクト概要
* 使用技術
* 重要なフォルダ
* 読まないフォルダ
* 検索方針
* 生成物の扱い
* コーディング規約
* 作業手順
* 検証方法
* 変更禁止事項

逆に、次のものは書きすぎ注意です。

* 技術スタックの一般論
* 長い設計思想
* 古い仕様
* たまにしか使わない手順
* 1機能専用の詳細ルール
* READMEと重複する大量説明

AIに常に読ませるファイルは、短いほど扱いやすいです。

Claude Codeのドキュメントでも、`CLAUDE.md` が長すぎるとコンテキストを消費し、追従性が落ちる可能性があると説明されています。Codexも `AGENTS.md` の連結サイズに上限があります。

目安としては、ルートの `AGENTS.md` は100〜200行程度に抑え、詳細手順は別ファイルに逃がすのが扱いやすいです。

```
## Playbooks

Read these only when relevant:

- `docs/ai/playbooks/addressables.md`
  - Addressables investigation workflow.
- `docs/ai/playbooks/prefab-audit.md`
  - Scene/Prefab extraction workflow.
- `docs/ai/playbooks/release-check.md`
  - Release verification checklist.
```

ただし、Claude Code の `@path` import は、読み込まれるとコンテキストに展開されます。  
**分割しただけでトークンが減るわけではありません。**

分割の目的は、管理しやすくすることと、必要なときだけ読ませる導線を作ることです。

## Codex と Claude Code の違い

Codexでは、基本的に `AGENTS.md` を使います。  
グローバルの `~/.codex/AGENTS.md` と、リポジトリ内の `AGENTS.md` を組み合わせる運用ができます。

一方、Claude Codeが読む標準ファイル名は `CLAUDE.md` です。  
既に `AGENTS.md` を使っている場合は、`CLAUDE.md` から import できます。

```
@AGENTS.md

## Claude Code specific rules

Use plan mode before large refactors.
Do not run Unity Editor unless explicitly requested.
```

これで、共通ルールは `AGENTS.md` に集約し、Claude Code 固有のルールだけ `CLAUDE.md` に書けます。

注意点として、Codexに `CLAUDE.local.md` 相当の標準ファイル名はありません。  
`.codex-local.md` のような名前を作っても、自動では読まれません。

Codexのinstruction discoveryは、基本的に**1ディレクトリにつき最大1ファイル**です。探索順は `AGENTS.override.md`、`AGENTS.md`、`project_doc_fallback_filenames` に登録した名前です。同じディレクトリに `AGENTS.md` がある場合、fallback名に追加した `.codex-local.md` が `CLAUDE.local.md` のように追記されるわけではありません。

また、`AGENTS.override.md` は名前の通りoverrideです。リポジトリルートで使うと、同階層のチーム共有 `AGENTS.md` を補足するのではなく、置き換える可能性があります。

Codexで個人用ルールを使いたい場合は、まず次の順で考えるのが安全です。

* `~/.codex/AGENTS.md` に、全リポジトリ共通の個人ルールを書く
* 作業用プロファイルを分けたい場合は `CODEX_HOME` を使う
* リポジトリ内の `AGENTS.override.md` は、チームで意味を合意した場合だけ使う
* 一時的な個人メモは、セッション開始時に手動で読ませる

チーム共有のルールと個人用ルールは分けたほうが安全です。Claude Codeなら `CLAUDE.local.md` がその用途に向いています。

```
# personal AI agent notes
CLAUDE.local.md
```

Codexで独自名を使う場合も、それは「同階層の `AGENTS.md` への追記」ではなく、`AGENTS.md` がない場合などの代替候補として扱います。

## 大容量データは読ませずに抽出させる

Scene、Prefab、ScriptableObject、ログ、CSV、JSON、YAMLを丸ごとAIに読ませると、すぐにトークンを消費します。

こういう場合は、AIにファイルを読ませるのではなく、**AIに抽出スクリプトを書かせる**ほうが効率的です。

例えば、Prefab調査なら丸ごとYAMLを読むのではなく、Editor script で次の情報だけ出します。

| 調査対象 | 抽出する情報 |
| --- | --- |
| Prefab / Scene | GameObject階層、Component一覧、Missing参照 |
| ScriptableObject | 型名、主要フィールド、参照GUID |
| Addressables | Address、Label、Group、Path |
| ログ | エラー行、前後N行、重複件数 |
| CSV / JSON | 行数、キー、問題箇所、サンプル数件 |

`AGENTS.md` には、次のように書いておくとよいです。

```
## Large file policy

Do not read large YAML, JSON, CSV, log, scene, prefab, or ScriptableObject files directly.

For large files:
1. Write a small read-only script to extract only relevant fields.
2. Show the script before running it when it may touch Unity/Unreal assets.
3. Do not call SaveAssets, ForceReserializeAssets, asset delete/move APIs, or import-triggering operations unless explicitly requested.
4. Write extraction output to a dedicated gitignored report directory such as `.ai-tmp/ai-extract/`.
5. Show the extraction summary.
6. Read the original file only if the summary is insufficient.
```

ポイントは、最初から「全部読め」と言わないことです。

AIに「読む」のではなく「絞る」作業をさせるのが重要です。ログならエラー種別と前後数行、PrefabならGameObjectパス・Component名・Missing参照だけ、というように最初から抽出条件を指定します。

抽出結果の置き場所は、Unityの `Temp/` ではなく、プロジェクト側で管理できる `.ai-tmp/ai-extract/` のような専用ディレクトリにして、`.gitignore` に入れておくと扱いやすいです。`Temp/` はUnity Editor側の一時領域なので、AI用の作業場所としては避けたほうが安定します。

特にUnity Editor scriptを実行させる場合は、**読み取り専用**を前提にします。`AssetDatabase.SaveAssets()`、`ForceReserializeAssets()`、アセット削除・移動・再インポートを伴う処理は、調査用スクリプトでは原則使わせないほうが安全です。実行前にGit差分がクリーンであることも確認しておくと、万が一の差分にも気づきやすくなります。

## 生成コードは直接直させない

APIクライアント、マスターデータ定義、ローカライズ定義、Protocol Buffers、独自ツールのC#出力などは、次回生成時に上書きされます。

```
## Generated files

Do not edit generated files directly.

Examples:
- `Assets/Scripts/Generated/`
- `Assets/Plugins/CompanyName/Generated/`
- `*.g.cs`
- `*.generated.cs`

If a generated file has a bug, fix the generator or template instead.
```

「直すなら生成元」と明記しておくだけで、かなり事故を減らせます。

## 作業前に調査結果を出させる

AIにいきなり修正させると、関係ないファイルを触ることがあります。

大きめの作業では、次の流れを固定したほうが安全です。

```
## Work process

For non-trivial tasks:

1. Search relevant files first.
2. Report the files you plan to inspect.
3. Summarize the current behavior.
4. Propose the smallest safe change.
5. Modify only the necessary files.
6. Run the relevant verification command if available.
```

これにより、AIの探索範囲を人間が確認できます。

特にUnityでは、1つの機能に対して Runtime / Editor / Addressables / ScriptableObject / Prefab が絡むことがあります。  
先に「どこを見るか」を出させるだけで、無駄読みと誤修正を減らせます。

## 変更範囲を小さくする

AIは、放っておくと「ついでに綺麗にする」ことがあります。  
これは実務では困ることがあります。

特に運用中プロジェクトでは、目的外のリファクタリングはレビューコストを増やします。

```
## Change policy

Keep changes minimal.
Do not perform broad refactors unless explicitly requested.
Do not change public APIs unless required.
Do not reformat unrelated files.
```

「最小変更」をルールにしておくと、レビューしやすくなります。

## 検証コマンドを書く

AIに「確認して」と頼んでも、何を実行すべきか分からなければ曖昧になります。

プロジェクトごとに検証方法を書いておくと便利です。

```
## Verification

Prefer the smallest relevant verification.

Examples:
- Run EditMode tests from Unity Test Runner.
- Run PlayMode tests only when the change affects runtime behavior.
- Run Unity batchmode tests when available.
- Run .NET tool tests with `dotnet test` only for standalone tool projects.

If verification cannot be run, explain what should be checked manually.
```

Unity Test RunnerをCLIで回しているプロジェクトなら、コマンド例も書きます。

```
Unity batchmode example:

`Unity.exe -runTests -batchmode -projectPath . -testPlatform EditMode -testResults TestResults.xml`
```

Unity Test Framework の `-runTests` 例では `-quit` を付けない形が示されています。通常のビルド用 batchmode では `-quit` を使うことがありますが、テスト実行例では混ぜないほうが安全です。

ただし、UnityのCLI実行は環境差が出やすいため、チーム内で実際に使っているコマンドを書いたほうがよいです。`Unity.exe` はWindows例であり、実際にはUnity Hubのインストールパス、OS、CIイメージ、ライセンス、`-logFile` の扱いに依存します。記事やテンプレートの例をそのまま標準コマンドとして扱わず、プロジェクトの実コマンドに置き換えます。

## 個人用メモはGitに入れない

チーム共有の `AGENTS.md` / `CLAUDE.md` と、個人環境だけのメモは分けます。

Claude Codeでは `CLAUDE.local.md` をgitignoreして使えます。Unity / Unreal のローカルパス、個人用コマンド、一時的な調査メモなどを書く場所です。

Codexには `CLAUDE.local.md` 相当の標準ローカル追記ファイルはありません。個人ルールは `~/.codex/AGENTS.md` を第一候補にします。`AGENTS.override.md` は同階層の `AGENTS.md` を補足するファイルではなく、置き換える可能性があるため、チームで合意した場合だけ使います。

## Unreal Engineの場合

UE5でも考え方は同じです。  
AIに読ませる場所と読ませない場所を分けます。

優先して見る場所:

```
Source/
Config/
Plugins/*/Source/
*.Build.cs
*.Target.cs
*.uproject
*.uplugin
```

基本的に読ませない場所:

```
Intermediate/
Saved/
DerivedDataCache/
Binaries/
```

`Content/` 配下の `.uasset` や `.umap` は、テキストとして直接読んでも意味が薄いです。  
BlueprintやMapの調査は、Asset Registry、Editor Utility、Python Script、C++の補助コマンドなどで必要情報を抽出してからAIに渡すほうが効率的です。

UEはUnity以上にバイナリAssetが多いため、\*\*「ファイルを読ませる」より「Editor経由で情報を抽出する」\*\*意識が重要です。

## チャット側の頼み方も重要

`AGENTS.md` を整えても、チャットでの頼み方が雑だと効果は落ちます。

```
Unityの起動時に出ているNullReferenceExceptionを調査してください。
まず Semble で関連しそうな初期化処理を探してください。
ログ全文は読まず、該当エラーと前後5行だけ確認してください。
修正前に、原因候補と変更予定ファイルを出してください。
```

良い依頼には、対象機能、最初の検索方法、読まないもの、変更前の報告要求が入っています。プロンプトの美しさより、**探索範囲の指定**が効きます。

## 最低限のチェックリスト

公開前に最低限確認したいのは、`.gitignore`、読む場所 / 読まない場所、生成コードの扱い、大容量ファイルの抽出方針、検証方法、個人用メモの分離です。これだけでも、AIの無駄な読み込みはかなり減ります。

## まとめ

AIエージェントのトークン消費を減らすには、モデル選びだけでは不十分です。

重要なのは、AIが読む情報を設計することです。

* `.gitignore` で生成物を管理外にする
* `AGENTS.md` / `CLAUDE.md` で探索方針を書く
* ただし、指示ファイルは強制設定ではないと理解する
* Sembleなどで検索を先行する
* 巨大ファイルはスクリプトで抽出する
* 作業前に調査結果を出させる
* 変更範囲を小さくする
* 検証方法を書く

Unityプロジェクトは、AIに読ませると重いファイルが多いです。  
だからこそ、**全部読ませない設計**が効きます。

最後に、Unity / UE5 のどちらでも使いやすい汎用テンプレートを載せます。

ここから先は、本文の内容をコピペしやすい形にまとめ直したものです。記事としては少し重複しますが、実際のプロジェクトに貼って使うために、意図的に独立したテンプレートとして残しています。

このまま全部使うと長いので、不要な項目は削ってから使ってください。Unity専用・UE専用の項目は `AGENTS.md` の一番下に分けてあります。UnityだけのプロジェクトならUE側を、UEだけのプロジェクトならUnity側を削って使う想定です。

## コピペ用 AGENTS.md

```
# AGENTS.md

## Language
- Prefer short English for machine-facing rules.
- Use a language the team can review and maintain.
- Reply to users in the language they use.

## Project overview
- This is a game development project using Unity and/or Unreal Engine.
- Prioritize small, safe, reviewable changes.
- Do not perform broad refactors unless explicitly requested.

## Core rules
- Do not read the entire repository by default.
- Search first, inspect only relevant files, then propose the smallest safe change.
- Instruction files are guidance, not enforcement.
- Use `.gitignore`, tool settings, hooks, tests, and CI for hard guarantees.

## Search policy
- Use Semble first for semantic code search when available and suitable.
- Use `rg` / `grep` for exact symbols, APIs, errors, and paths.
- Do not scan the entire repository without explaining why.

## Workflow for non-trivial changes
1. Search for relevant files.
2. Report candidate files and why they matter.
3. Inspect only necessary files.
4. Summarize current behavior.
5. Propose the smallest safe change.

## Avoid by default
Do not inspect these unless explicitly requested:
- `.git/`, `.vs/`, `.idea/`, `.gradle/`
- `node_modules/`, `dist/`, `build/`, `Build/`, `Builds/`
- `logs/`, `MemoryCaptures/`
- `.ai-tmp/`, except report files created for the current task

## Large files
Do not read large YAML, JSON, CSV, logs, scenes, prefabs, ScriptableObjects, `.uasset`, or `.umap` files directly.
- Create a small read-only extraction script or command first.
- Show the script before running it when it may touch assets.
- Do not save, reserialize, delete, move, reimport, or modify assets unless explicitly requested.
- Write reports to `.ai-tmp/ai-extract/`.
- Read only the generated summary first.
- Read raw files only if the summary is insufficient.

## Generated files
- Do not edit generated files directly.
- Examples: `**/Generated/**`, `*.g.cs`, `*.generated.cs`, generated API clients, data loaders, localization files, protobuf/message files.
- If generated output has a bug, fix the generator, template, or source data instead.

## Coding policy
- Follow existing style.
- Keep changes minimal.
- Do not reformat unrelated files.
- Do not change public APIs unless required.
- Do not introduce new packages without approval.
- Prefer readable code over clever code.

## Verification
- Run the smallest relevant verification when possible.
- Use the project's actual commands rather than generic examples.
- If verification cannot be run, explain why and what should be checked manually.

## Reporting
When making changes, report:
- files inspected
- files changed
- reason for each change
- verification performed
- verification not performed
- remaining risks

## Platform-specific sections
Unity and Unreal Engine rules are intentionally placed at the end.
Delete the section that does not apply to your project.

## Unity policy
Prefer reading:
- `Assets/Scripts/`
- `Assets/Editor/`
- `Assets/Plugins/<CompanyName>/`
  - Treat third-party plugins and purchased assets as read-only unless explicitly requested.
- `Packages/manifest.json`, `Packages/packages-lock.json`
- `ProjectSettings/`
  - Read when relevant, but change only with explicit intent and explain the diff.
- `Assets/**/*.asmdef`

Avoid unless explicitly requested:
- `Library/`, `Temp/`, `temp/`, `obj/`, `Obj/`, `Logs/`, `UserSettings/`
- `*.csproj`, `*.sln`
  - Usually generated by Unity. Read only when needed; do not edit directly.
- `Assets/StreamingAssets/aa/`
- large generated Addressables files, bundles, catalogs, and binary outputs

For serialized assets, extract path, GUID, GameObject path, component type, serialized fields, missing references, and Addressables address/group/labels.
Verification examples: EditMode tests, PlayMode tests, Unity batchmode tests without adding `-quit` to `-runTests`, project-specific validation scripts.

## Unreal Engine policy
Prefer reading:
- `Source/`
- `Config/`
- `Plugins/*/Source/`
- `*.Build.cs`, `*.Target.cs`, `*.uproject`, `*.uplugin`

Avoid unless explicitly requested:
- `Binaries/`, `Intermediate/`, `Saved/`, `DerivedDataCache/`

Do not treat `.uasset` or `.umap` files as readable source.
For Blueprint, Map, Asset, or Content analysis, extract metadata through Unreal Editor, Asset Registry, Editor Utility, Python, or C++ helper scripts.
Verification examples: build the affected target, run relevant automation tests, run project-specific validation scripts.
```
