この記事は Claude（Anthropic）との対話を通じて執筆しました。

## はじめに

最近「ハーネス」とか「ハーネスエンジニアリング」という言葉をよく見かけるようになりました。ざっくり言うと、AI モデルそのものではなく、モデルの周りに組む足場（ツール、環境の制約、検証ループ）が実際の性能を決める、だからモデルよりも足場の設計に投資しよう、という考え方です。Claude Code みたいなコーディングエージェント自体がハーネスの一種で、同じモデルでもハーネスの出来で成果が全然違う、という文脈で使われています。

で、この言葉を見るたびにずっと引っかかっていたことがあります。

これ、今年の1月に私が「レール」って呼んで記事に書いたやつじゃないか?

<https://zenn.dev/tai_kimura/articles/0a2b213c569ad2>

当時はハーネスなんて言葉は（少なくとも自分の観測範囲では）まだ聞きませんでした。この記事では、当時書いたことといまのハーネス論がどう対応するのかを整理しつつ、半年経って気づいた「自分の記事に足りていなかった半分」の話をします。

## 1月に書いたこと

元記事の主張を要約すると、AI 駆動開発には車と電車の2つのアプローチがある、という話でした。

プロンプトで自由に走らせる開発は車です。速いかどうかは運転手（プロンプトを書く人間）のスキルに依存するし、道にも迷うし、走り続けると疲れる。一方、制約を敷いた上で走らせる開発は電車です。行き先はレールが決めているので迷わないし、誰が乗っても同じところに着く。そして一番言いたかったのはここで、**車両（AI モデル）が進化するたびに、同じレールの上で勝手に速くなる**。各駅停車が新幹線になり、リニアになる。レールを敷いた投資はモデルの世代が変わるたびに回収され続ける、という話です。

そのうえで、レールの中身を3つに分けました。

1. 構造のレール。アーキテクチャとディレクトリ構成を統一して、CLI で自動生成する
2. 言語のレール。自然言語ではなく形式言語で指示して、AI の解釈のブレを排除する
3. 責務のレール。1体の万能 AI ではなく、レイアウト・ロジック・データ定義・テストといった責務ごとの専門エージェントに分ける

自分はこの思想を JsonUI という自作フレームワークで実装して、実際にこのやり方で開発しています。

## ハーネス論との対応

いまのハーネス論の語彙と並べてみると、こうなります。

| 1月の記事（レール） | いまのハーネス論 |
| --- | --- |
| 構造のレール（統一アーキテクチャ・CLI 生成） | 環境の制約設計。エージェントが動く空間を狭くして無駄な探索を減らす |
| 言語のレール（形式言語で解釈のブレを排除） | 構造化された入出力、スキーマ契約。自然言語を境界から追い出す |
| 責務のレール（専門エージェント分離） | サブエージェントへの責務分割、マルチエージェントオーケストレーション |
| 車両が進化するたびに同じレールで速くなる | モデルはエンジン、実現される能力はハーネスが決める。モデルが上がるほどハーネスの差が開く |

4行目が一番本質的だと思っています。「制約は足枷ではなく加速装置で、モデルが進化するたびに無料で恩恵を受けられる」という主張は、ハーネス論者が言う「良いハーネスはモデル更新のたびに勝手に強くなる」とほとんど同じことを言っています。

## 半年経って気づいた、記事に足りなかった半分

ただ、答え合わせをして気持ちよくなって終わりにはできない部分もありました。いまのハーネス論には、1月の記事が前面に出していなかった要素がひとつあります。

**検証ループ（verifier）です。**

ハーネス論は「制約で縛る」ことと同じくらい、「機械的な合否信号で出力を検証して、失敗を自動で突き返す」ことを重視します。レールが静的な制約だとすると、こちらは動的なフィードバックループで、エージェントに自走させるならこっちが無いと品質が閉じません。

悔しいのは、これも半分は実践していたのに、記事では言語化できていなかったことです。JsonUI には記事を書いた時点から「ビルドは警告ゼロ」「仕様と実装の差分ゼロ」という機械チェックがゲートとして入っていて、AI に書かせたコードもこのゲートを通らない限り完成扱いになりません。ただ当時の自分はこれを「品質を守る門番」としか捉えていなくて、「検証ループを厚くするほどエージェントは自走できる」という加速装置側の意味には気づいていませんでした。

その後、この半年で検証側をかなり厚くしました。そしてつい先日、AI が正規化処理の戻り値の扱いを間違えた（タプルをそのまま JSON に書いた）のを自動チェックが即座に検出して、人間の私がレビューする前に修正まで終わる、という場面がありました。人間のミスを想定して作った検証装置が、AI のミスも同じように捕まえた瞬間です。

つまり正確に言うと、私のシステムは「レール + 検証ループ」で徐々にできあがっていて、1月の記事はその半分しか言語化していなかった。ハーネスという言葉が広まったおかげで、自分が作っていたものの全体像を説明する語彙が揃った、というのがフェアな総括だと思います。

## 自分の思いつきに、後から名前がついた

ここからは技術の話ではないので、余談として。

自分がエンジニアになったのは20代後半で、遅いほうだと思います。それまでプログラムの知識は完全にゼロ。パソコンなんてネットサーフィンくらいにしか使っていませんでした。

そういうキャリアなので、技術の世界ではずっと「後から追いかける側」でした。すでに名前がついていて、すでに体系化されていて、すでに先人が Qiita や Zenn に書いてくれているものを、後から学んで追いつく。たまに「これは自分で考えた仕組みだ」と思えるものを実装しても、よく調べたらとっくに名前がついて世の中に存在していて、新しくもなんともない。そんなことばかりでした。

そこに AI が来ました。業界全体の仕組みごと変えてしまうレベルの変化で、しかも新しすぎて誰も正解を持っていない。その流れの中で、自分の頭で考えて、こうやるのが正しいはずだと決めて、フレームワークを拡張して仕組み化しました。当時は呼び名が無かったので、記事の中では「レール」としました。それが半年経ったら、世界の側が同じものに「ハーネス」という名前をつけて、みんながその重要性を語り始めていた。

自分が考えて形にしたものに、後から名前がつく。ずっと追いかける側だった人間からすると、この体験が初めてで、単純に楽しいんですよね。大げさに言えば、自分の理論が世間に評価されたような気持ちになって、素直にとても嬉しいです。

もちろん、私が最初に考えたと言いたいわけではありません。同じ時期に同じ結論に達した人は世界中にいくらでもいたはずで、だからこそ言葉が生まれて広まったわけです。ただ、誰かの答えを写したのではなく、自分で考えて出した答えが世の中の答え合わせと一致した。エンジニアとして積んできたこの十数年のキャリアの、答え合わせにもなった気がしています。

## おわりに

新しい技術が出てきた直後の期間は、まだ名前がついていないことだらけの期間でもあります。つまり、キャリアの長さや積み上げに関係なく、自分の頭で考えた人から順に正解にたどり着ける、めずらしくフェアな期間です。

私みたいにスタートの遅かった人間でも、考えて、作って、動かしていれば、半年後に世間の側から答え合わせが来ることがある。これは AI 時代の数少ない、そして最高におすすめできる楽しみ方だと思います。

私も、また半年後の答え合わせになるようなアイディアを思いつけるように、毎日試行錯誤していこうと思います。
