## はじめに

こんにちは。Finatextでクレジットビジネスプラットフォーム「Crest」の開発に携わっている[@chinaka](https://zenn.dev/chinaka)です。

ゼニックスというチームの一員として、社内のAIコンテストに参加しました。この記事は、チームでの取り組みを、私がまとめたものです。

既存の業務システムに、AIオペレーターをどう安全に組み込むか。在籍確認という審査業務を題材に、MCPとSkillで組んだ設計の話です。

## なぜ「AIオペレーター」なのか

### まだ人手が要る、時間のかかる仕事

Crestは、クレジット事業を立ち上げ・運営するためのSaaS型クレジットビジネスプラットフォームです。その裏側では、審査や債権管理まわりのオペレーション業務が日々動いています。なかには、まだ人手に頼るしかない、手間と時間のかかる仕事も少なくありません。ざっと挙げるだけでも、こんな業務があります。

* 収入証明書の受付・チェック
* 在籍確認
* 名寄せ（同一人物の重複判定）
* 交渉記録の登録

どれも、判断材料が複数の書類やシステムに散らばっています。人が横断して突き合わせ、一件ずつ判断する。1件は小さくても、件数が多いぶん、積み重なると大きな負担になります。

### 全自動化ではなく、「AIオペレーター」

こうした業務をルールベースで全部自動化するのは、簡単ではありません。表記の揺れや例外が多く、最後は人の判断や責任が要る場面も多いからです。

そこで考えたのが、業務をまるごとAIに置き換えるのではなく、AIを「もう一人のオペレーター」としてチームに入れるアプローチです。AIが担当するのは、対象を探し、根拠を集め、突き合わせ、判断の案を作るところまで。最終的に確定させるのは人間です。「担当者が申請し、別の人が承認する」という既存のワークフローに、AIが担当者として入るイメージですね。

このしくみを支えるのが、MCPとSkillの2つ。次の章から、在籍確認を例に実際の動きを追っていきます。

## どうやって動かしているか

### 在籍確認を動かしてみる

まずは、在籍確認をAIオペレーターに任せると何が起きるか。最初から最後まで追ってみます。

在籍確認は、新規の貸付申込で「申込者が、申告した勤務先に本当に在籍しているか」を確かめる業務です。手がかりは主に3つあります。

* 申込時に申告された勤務先（会社名・電話番号など）
* 収入証明書の発行元（源泉徴収票などに記載された会社名）
* 信用情報機関（CIC）に登録された勤務先

AIオペレーターは、まず未処理の対象を一覧で取得し、その1件について申込内容・収入証明書・CICの情報を集めます。Crestのデータには、業務操作の口（後述するMCP）を通してアクセスします。

ここで最初の難所です。3つの勤務先名が、同じ会社を指しているのに一致しない——そんなケースが珍しくありません。

| 申告された勤務先 | 収入証明書の発行元 | 実際の関係 |
| --- | --- | --- |
| スマートプラスクレジット | Finatext | サービス名と運営会社 |
| 〇〇小学校 | 〇〇市教育委員会 | 公立学校と所管する教育委員会 |

（いずれも説明のための例です）

店舗・ブランド名と運営会社、子会社と親会社……文字面は違っても実体は同じ、というパターンはいくらでもあります。単純な文字列一致では不一致と判定されますが、人間の担当者なら背景知識やWeb検索で「同じ事業体だ」と判断できます。この曖昧な照合（ファジーマッチング）が、AIの得意分野です。必要なら、AIが自分でWeb検索して裏を取ります。

集めた根拠と突き合わせの結果から、AIは「在籍確認できた／できない」の判断案を、理由・根拠コメントつきで提出します。

ただ、この提出はまだ確定ではありません。AIはハルシネーションを起こす可能性があります。その誤りをそのまま通さないため、提出と同時にCrest側で承認依頼が自動で作られ、別の人間（承認者）が再鑑します。承認者がチェックするのは、結果と理由が整合しているか、「同じ事業体だ」と判断した根拠がコメントに残っているかです。申請者本人は承認できません。誰が何を根拠にどう判断したかは、監査ログに残ります。

AIが下調べと判断案まで一気にこなし、人間が最後の確定を担う。これが一連の流れです。

### いま動いた裏側を、3層で捉える

いまの流れは、責務を3つの層に分けた設計で動いています。

切り分けの軸はシンプルです。固定機能はCrest本体、業務ごとの差分や横断的な判断はAI（MCP経由）、複雑な業務ロジックやカスタムはSkill。ざっくり言うと、MCPが「何にアクセスして何を実行できるか」、Skillが「どう確認してどう判断するか」を持ちます。

### 守りは本体に、操作だけをMCPで

Crest本体は、これまで通りです。具体的には、業務データ、認証・認可、承認ワークフロー、監査ログなどの機能です。AIを足してもここは変えません。自動架電や書類OCRのような、定型的でUIに密着した処理も本体側の仕事です。

その本体に対して、AIがアクセスする口がMCPです。とはいえ、MCPは「AI用の裏口」ではありません。AIに自由なDB操作をさせるのではなく、業務として意味のある操作だけを、許可した範囲で渡します。在籍確認でAIが使っていたのも、こんな業務単位の操作でした。

* 対象を探す：未処理や承認待ちなど、業務状態から対象を取得する
* 根拠を見る：申込情報・書類・信用情報など、判断材料を取得する
* 結果を申請する：確認結果やコメントをCrestのワークフローへ流す
* ログに残す：誰が何を根拠に申請したかを追える状態にする

業務操作の単位で権限を渡すため、AIに任せても画面操作の自動化とは違って監査しやすく、許可した範囲を超えません。

### 業務の手順は「Skill」に書く

Skillには、業務の手順・確認順序・判断基準を書きます。在籍確認のSkillを抜粋すると、全体はこんな構成です。

```
---
# 以下は employment-verification/SKILL.md の抜粋です
name: employment-verification
description: Crest で在籍確認を担当者として実施・申請するためのスキル。
---

# 在籍確認（担当者）

新規貸付申込における勤務先在籍確認を担当者として実施する。
目的は ①申告内容の真偽確認 ②返済能力の把握 ③なりすまし検知。

## 業務フロー

### 1. 対象確認

未処理の在籍確認対象を取得し、申告された勤務先情報を確認する。
...

### 2. 確認の実施

在籍確認の手段は3つ。収入証明書 → CIC → 架電 の順に試す。
...

#### 2-1. 収入証明書類の発行元との突合

収入証明書の発行元を取得し、申告された勤務先名と突き合わせる。

- 完全一致：在籍確認完了
- 不一致でも、合理的に同一の事業体と確認できる場合：在籍確認完了として扱う
  - 例：勤務先名が「〇〇小学校」、発行元が「〇〇市教育委員会」
  - 迷う場合はWeb検索などで関連を確認し、根拠をコメントに残す

#### 2-2. CIC情報との突合

2-1 と同様に、CIC 登録の勤務先名・電話番号と突き合わせる。
...

#### 2-3. 架電による在籍確認

収入証明書・CIC で確認できなければ、架電に回す。
...

### 3. 結果登録

確認結果（手段・結果・理由・コメント）を提出する。
...

## 判断時の原則

- 3つの確認手段がすべて揃っている必要はない。1つで一致が取れれば完了
- 不一致なのに「確認完了」にしない。確証がなければ保留にして架電へ回す
- コメントは承認者への引き継ぎを意識して、何をどう確認したかを残す
- ...
```

ポイントは、コードが一行も出てこないこと。「収入証明書 → CIC → 架電 の順に試す」「合理的に同一の事業体なら完了扱い」「迷ったらWeb検索して根拠を残す」——どれも人間向けの業務マニュアルそのものです。エンジニアがコードに翻訳しなくても、ドメインの担当者が読んで直せます。

Skillの強みは、この手順書を事業者ごとに重ねられること。Crestは複数のクレジット事業者に提供するので、在籍確認の大枠は共通でも、事業者ごとの固有ルールがいくつも出てきます。たとえばある事業者（仮にA社）には、「自社グループの社員からの申込は特別扱いする」という特則があります。これを標準Skillに直接書き込むと、事業者が増えるほど分岐が膨らんで手に負えません。

そこで、標準Skillの上に事業者ごとのSkillを重ねます。A社なら、「標準の在籍確認に入る前に、申込者がA社本体かグループ会社の社員かを確認し、該当すれば社員申込として扱う」という特則をA社向けSkillとして用意します。判断を支えるのは、A社グループの関連法人をまとめた参照ファイルです。一覧にない会社は社員申込と即断せず、公式情報をWeb検索で確認し、裏が取れなければ標準フローへ戻す。こうしたガードレールごとSkillで配布できます。

ファイル構成は、Claude Codeが読み込む `.claude/skills/` の下に標準スキルと事業者向けスキルを別名で並べ、どちらを先に適用するかを `CLAUDE.md` / `AGENTS.md` で指示します。

標準の手順と事業者ごとの差分が、ファイルとして分かれています。改善スピードは、この分離で決まります。

本体に実装すればいいのでは、という案も最初はありました。ただ、収入証明書のOCR精度をチューニングした経験から、業務ロジックを本体に埋め込む重さは見えていました。本体に持たせると、判断基準を少し直したいだけでも、精度を調べて、改修して、QAを通して、本番リリースという工程が発生します。1回の改善で2週間かかることもあります。

|  | 本体に埋め込む場合 | MCP + Skill の場合 |
| --- | --- | --- |
| 精度調査・改善 | 開発者がデータ基盤で調査 | 開発者以外でも、業務の中で自然言語で確認・改善できる |
| 業務知識の変更 | フィードバック → チケット化 → 改修 | Skillに直接反映できる |
| リリース | 改修 → QA → 本番リリース | Skillを配布するだけ |
| 事業者差分・本番基準 | コードに混ざる | コードから分離して保てる |

業務知識の変更を、開発チームの手を借りずにSkillへ直接反映できる。改善のサイクルがぐっと短くなりました。

## 現場で使い、広げていく

### Finstage Cowork：ブラウザで対話するだけ

ここまでのMCPとSkillを、オペレーターが日常的に使えるようにする入り口が [Finstage Cowork](https://finatext.com/finstage/) です。ブラウザ上のAIエージェント実行環境で、チャットで対話するだけで業務が進みます。

![](https://static.zenn.studio/user-upload/fff498fdc475-20260629.png)

CoworkはSkill・MCP接続・プラグインを束ねます。どの業務SkillとどのMCPを使えるかを管理し、誰がいつ何を実行したかを監査ログに残し、事業者ごとにテナントを分けます。オペレーターはシンクライアントのように、安全な環境からCowork越しにCrestの業務を進められます。MCPassと組み合わせれば、ドメイン外の担当者や社外ユーザーにも、認証や監査ログを保ったままCrestをAI経由で触ってもらえます。

### 在籍確認で作った型を、周辺業務へ

新しい業務を増やすときにやるのは、MCPに業務操作を1つ足して、Skillを1つ書くだけ。最初に挙げた収入証明書の受付・チェック、名寄せ、審査決裁、交渉記録の登録なども、在籍確認と同じ「情報を集める → 根拠を整理する → 判断案を作る → 人間が確定する」型で扱えます。本体側のしくみはそのままです。

### 24時間・365日、止めずに回す

Claude Agent SDKのような実行基盤をスケジューラから定期起動すれば、夜間や休日も対象を巡回できます。未処理を検知して、確認できるものは結果案を作ります。人間に戻すのは、架電が必要なもの・判断できないもの・期限が近いものだけ。審査オペレーションは件数ベースの業務なので、件あたりの工数をAIで減らせれば、その分だけ運用コストが下がります。在籍確認は、その入口にすぎません。

## 現状の課題

正直なところ、このしくみにはまだ粗があります。

* 現在は標準スキルを事業者ごとのプロジェクトにコピーして置いているので、標準を直すたびに各プロジェクトへ手で反映しないといけません。Skill自体に base / overlay みたいな仕組みがまだないんですよね。ゆくゆくは、標準は1か所だけ管理して、各プロジェクトには差分だけ置く。ビルド時に標準をテンプレートのように差し込んで生成する、という形にしたいです。
* スキルの改善も、現在は再鑑の結果を見ながら手で直しています。ここは将来的には、SkillClaw のような「スキルが自分で育っていく」しくみに寄せていきたいです。

## まとめ

設計のキモを4つに絞ると、こうです。

* MCPで「業務操作の境界」をAIに渡す。裏口ではなく、許可した操作だけを正規の口から
* Skillで業務の手順と事業者ごとの差分を分け、配る。改善はSkillを直すだけ
* AIは提案、人間が確定。曖昧な照合はAIに任せ、最後の判断と責任は人間が持つ
* Finstage Coworkを入り口に、監査ログとテナント分離で守る

本体の守りを崩さずに、AIを業務へ乗せる。これがいちばんやりたかったことです。ドメインのスペシャリストからエンジニアまで同じチームで動けたから、現場の業務知識をそのままSkillに落とせました。今回いちばん効いたのは、ここでした。
