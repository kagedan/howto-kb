## AI強者がよく言う「構造化」を、初心者でもClaude Codeで再現する方法

---

**「こんなアプリを作りたい。設計、コード、レビューの流れで全部お願い」**という指示をしたことありませんか？

一見便利そうに見えますが、**最もよくある失敗パターン**です。  
この依頼には4つの役割が同時に混ざっています。

* **PM役：** 何を作るか決める
* **設計者役：** どう作るか決める
* **開発者役：** 実際に作る
* **レビュアー役：** 問題点を見つける

構造化とは、これらを別々に指示することです。

「**そうなんだぁ、毎回？不便だね**」って思いますよね。

**有料部分では、全く同じ  
「こんなアプリを作りたい。設計、コード、レビューの流れで全部お願い」  
という指示だけで、構造化された指示に自動変換して実行する具体的な方法を提供します。**

---

## この記事について

**無料部分**では、なぜ役割を分けると回答が整理されるのか、どんなファイルを用意すれば運用しやすくなるのか、実際の使い方を解説します。

**有料部分**では、すぐ使えるテンプレートを付録として用意しています。

## この記事でできるようになること

* Claude Codeの回答がブレる理由を説明できる
* Claude Codeに渡す情報を役割ごとに分けられる
* **PM役・設計者役・開発者役・レビュアー役を分けて依頼できる**
* 過去の失敗を残して、次回以降に活かせる
* 回答がズレたときに、修正プロンプトで方向を戻せる

---

## なぜClaude Codeの回答はブレるのか

回答がブレる原因はいくつかあります。モデル側の状態やClaude Code側の仕様変更が影響することもありますが、ユーザー側で改善しやすいのは**情報の整理不足**です。よくある原因を3つに絞ります。

### 原因1：役割が混ざっている

冒頭でも触れた通り、「設計もコードもレビューも全部お願い」という依頼は役割が混ざっています。人間のチームでも、企画・設計・実装・レビューを1人が同時にやろうとすると判断基準がバラバラになって成果物が散らかります。Claude Codeも同じです。

**役割を混ぜると具体的に何が起きるか**を見ると、要件がまだ曖昧なのにコードを書き始める、設計の話をしているのに実装の提案が混ざる、レビューを頼んだのに新機能の提案が出てくる、といった現象が起きます。どれも「頼んだこと」ではあるのに、「今やってほしいこと」ではない状態です。

解決策はシンプルで、**1回の依頼で役割を1つに絞る**だけです。まず「今はPM役として要件だけ整理して」と頼み、終わったら「次は設計者役として構成を考えて」と切り替えます。この順番で進めると、Claude Codeは各役割に集中できるので回答が整理されやすくなります。

依頼の重要なポイントは「**まだコードは書かないでください**」という一文です。これを入れないと、Claude Codeは親切心でコードも出そうとします。役割を宣言するだけでなく、**やらないことも明示する**のがコツです。

### 原因2：判断基準がない

AIは、何を優先すればいいかを知らないまま回答することがあります。「最小構成で作りたい」と思っていても、それを伝えていないと大きな構成を提案されるかもしれません。判断基準は先に渡します。

* **初心者でも実行できること**
* **手順が少ないこと**
* **まずは最小構成にすること**
* **後から修正しやすいこと**

### 原因3：前提と失敗ログが残っていない

毎回その場で前提を説明していると、説明の粒度が変わります。前回の失敗や禁止事項を書き忘れることもあります。その結果、Claude Codeは同じ失敗を繰り返しやすくなります。

* いきなりコードを書き始める
* 成功確認方法を書かない
* 設計と実装が混ざる
* 今回やらないことまで提案する

## 構造化で用意するもの

Claude Codeの構造化では、次の4つを用意します。

* **CLAUDE.md：** 全体方針
* **役割別設定ファイル：** 役割ごとの作業ルール
* **プロンプトセット：** よく使う依頼文
* **学習ログ：** 過去の失敗と改善点

### 1. CLAUDE.md

CLAUDE.mdは、Claude Codeに毎回読ませたい**基本方針**を書くファイルです。プロジェクト全体で守ってほしいルールや回答方針を書いておきます。

* Claude Codeの役割
* 回答方針
* 禁止事項
* 優先順位
* 参照してほしいファイル

毎回「初心者向けに説明して」「成功確認方法も書いて」「いきなりコードを書かないで」と伝えているなら、その内容はCLAUDE.mdに入れておくと便利です。置き場所は作業フォルダ直下の ./CLAUDE.md で大丈夫です。公式ドキュメントでも、CLAUDE.md は永続的な指示として説明されています。ただし強制設定ではなく、Claudeが参照するコンテキストです。

### 2. 役割別設定ファイル

役割ごとに専用の設定ファイルを用意します。PM役・設計者役・開発者役・レビュアー役の4つです。これを使うと、**初回の指示だけ**で要件整理→設計→実装→レビューが**役割が混ざることなく**実行されます。

仕組みと完成版テンプレートは有料部分に用意しています。

### 3. プロンプトセット

役割別設定ファイルでカバーできない作業（デバッグなど）は、prompts/ フォルダに依頼文として保存します。

### 4. 学習ログ

学習ログは、過去の失敗や改善点を残すファイルです。置き場所は docs/LESSONS\_LEARNED.md です。失敗した依頼文・Claude Codeが誤解したポイント・次回から入れるべき条件などを書き残します。

このファイルを作るだけでは、Claude Codeが毎回自動で読んでくれるとは限りません。使いたいときは、作業開始時に「docs/LESSONS\_LEARNED.md を確認してください」と伝えます。学習ログは、人間があとで見返して、必要なルールをCLAUDE.mdや設定ファイルに反映するためのメモです。

## 事前準備

作業用のフォルダを1つ決めてください。名前は何でも構いません。この記事では、次の構成を使います。

```
project/
├── CLAUDE.md
├── .claude/
│   ├── skills/
│   │   ├── requirements.md
│   │   ├── design-review.md
│   │   ├── implementation.md
│   │   └── review.md
│   └── settings.json
├── prompts/
│   ├── initial-instruction.md
│   └── debug.md
└── docs/
    └── LESSONS_LEARNED.md
```

各ファイルの役割は次の通りです。

* **CLAUDE.md：** プロジェクト全体の基本方針
* **.claude/skills/requirements.md：** PM役（要件整理）
* **.claude/skills/design-review.md：** 設計者役（構成整理）
* **.claude/skills/implementation.md：** 開発者役（実装）
* **.claude/skills/review.md：** レビュアー役（問題点の指摘）
* **prompts/initial-instruction.md：** 作業開始時の初期指示
* **prompts/debug.md：** デバッグ用の依頼文
* **docs/LESSONS\_LEARNED.md：** 失敗と改善の学習ログ

## 初期設定

作る順番は次の通りです。

1. CLAUDE.md
2. 役割別設定ファイル（4つ）
3. prompts/initial-instruction.md
4. prompts/debug.md
5. docs/LESSONS\_LEARNED.md

### STEP1：CLAUDE.mdを作る

作業フォルダ直下に CLAUDE.md を作り、Claude Codeの役割・回答方針・禁止事項・参照ルールを書きます。長ければ長いほど良いわけではありません。毎回必ず守ってほしいルールだけを入れます。長い手順や特定作業だけで使う内容は、役割別設定ファイルやprompts/に分けます。

すぐ使える完成版テンプレートは有料部分に用意しています。

### STEP2：役割別設定ファイルを作る

.claude/skills/ の下に requirements.md・design-review.md・implementation.md・review.md の4ファイルを直接置きます。各ファイルはPM役・設計者役・開発者役・レビュアー役に対応しています。

すぐ使える完成版テンプレートは有料部分に用意しています。

### STEP3：初期指示テンプレートを作る

prompts/initial-instruction.md を作ります。これは、作業開始時に手動で読ませるテンプレートです。目的の確認・役割の明確化・やること/やらないことの分離・手順・成功確認・よくある失敗・次の一手の7点を方針として書きます。

すぐ使える完成版テンプレートは有料部分に用意しています。

### STEP4：デバッグ用プロンプトを作る

prompts/debug.md を作ります。エラー文・直前にやったこと・期待していた動作・実際に起きたことを渡す形式にします。

すぐ使える完成版テンプレートは有料部分に用意しています。

### STEP5：学習ログを作る

docs/LESSONS\_LEARNED.md を作ります。日付・作業内容・うまくいったこと・失敗したこと・原因・次回の改善・設定ファイルへの追記候補を書く構成にします。

すぐ使える完成版テンプレートは有料部分に用意しています。

## 実行手順

### STEP1：最初に読ませるファイルを指定する

Claude Codeに最初に送る文はこれです。

```
まず CLAUDE.md と prompts/initial-instruction.md を読んでください。

以後の回答では、これらの内容を優先してください。

今回の目的は、Claude Codeの回答がブレないように、作業を構造化して進めることです。
```

次のような返答が来れば、読み込み成功です。

```
確認しました。以後は CLAUDE.md と prompts/initial-instruction.md の方針に従います。
目的・前提・やること・やらないこと・手順・成功確認・よくある失敗を整理して進めます。
```

どのファイルを読んだか曖昧な場合は、こう確認します。

```
今読んだファイル名を列挙してください。
それぞれのファイルから反映したルールを簡単にまとめてください。
```

### STEP2：PM役から始める

作業は必ずPM役から始めます。まだ何も作らせません。「今回作るもの・作らないもの・優先順位」を整理させるだけです。

```
今回はPM役として動いてください。まだコードは書かないでください。

目的：（作りたいものを書く）
困っていること：（現状の問題を書く）

必要な機能・今回やらないこと・優先順位を出してください。
```

PM役が終わったら、同じ会話の中で「次は設計者役として〜」と宣言するだけで切り替えられます。

### STEP3：設計者役に切り替える

PM役で要件が固まったら、設計者役に切り替えます。構成・ファイル構造・処理の流れを整理させます。まだ実装はさせません。

```
ここからは設計者役として動いてください。
先ほどの要件をもとに、最小構成のファイル構成と処理の流れを整理してください。
まだコードは書かず、構成と実装順だけを出してください。
```

### STEP4：開発者役に切り替えて実装する

設計が固まったら、開発者役に切り替えて実装します。

```
ここからは開発者役として動いてください。
先ほどの設計をもとに実装してください。

条件：
- 変更するファイル名を明記する
- コードの前に実装方針を書く
- 実行手順を書く
- 成功確認方法を書く
- よくあるエラーと対処法も書く
```

特に大事なのは**成功確認方法**です。成功したときに何が表示されるのか、どのコマンドで確認するのかまで書かせます。

### 役割の切り替えがうまくいくと何が変わるか

PM役→設計者役→開発者役の順に進めると、次のことが起きます。

**ただし、毎回手動で役割を切り替えるのは正直面倒です。**「役割を分けることの価値はわかった。でも毎回3〜4回プロンプトを貼るのは手間が増えた感覚になる」という感覚は正しいです。

有料部分では、「こんなアプリを作りたい。設計、コード、レビューの流れで全部お願い」という依頼だけで、PM役→設計者役→開発者役→レビュアー役が順番に自動実行される設定ファイル一式を提供しています。

### STEP5：結果を学習ログに残す

作業が終わったら、うまくいった点と失敗した点を docs/LESSONS\_LEARNED.md に書き残します。次回以降も使いたい場合は、作業開始時にこう伝えます。

```
作業前に docs/LESSONS_LEARNED.md を確認してください。
今回の作業に関係する失敗や改善ルールがあれば反映してください。
```

## 動作確認

構造化がうまくいっているかは、次の4つで確認します。

**確認1：回答の順番が安定しているか**

毎回、目的→前提→やること→やらないこと→手順→成功確認→エラー対処→次の作業の順番で返ってくるなら成功です。

**確認2：毎回同じ説明をしなくて済むか**

次のような指示が毎回不要になっていれば、構造化が効いています。

* 初心者向けにしてください
* 成功確認方法も書いてください
* いきなりコードを書かないでください
* やることと、やらないことを分けてください

**確認3：役割が混ざっていないか**

PM役として依頼したのにいきなりコードを書き始める場合は、役割指定が弱い可能性があります。その場合はこう伝えます。

```
今回はPM役です。まだコードは書かないでください。要件整理だけをしてください。
```

**確認4：過去の失敗が反映されているか**

反映されていない場合はこう伝えます。

```
docs/LESSONS_LEARNED.md を確認してください。
今回の作業に関係する失敗や改善ルールを反映して、回答を作り直してください。
```

## よくあるエラーと対処法

**エラー1：いきなりコードを書き始める**

原因は「まだコードは書かないで」と伝えていないことです。次のプロンプトで修正します。

```
まだコードは書かないでください。
まず実装方針・作成するファイル・作業順だけを整理してください。
コードは「実装してください」と言ってから書いてください。
```

**エラー2：回答が長すぎて読みにくい**

出力形式を指定していないことが原因です。次のプロンプトで修正します。

```
回答を短く整理してください。
1. 今やること / 2. 実行手順 / 3. 成功確認 / 4. 次にやること
補足説明は必要なものだけにしてください。
```

**エラー3：前回の失敗をまた繰り返す**

docs/LESSONS\_LEARNED.md を作っただけで、Claude Codeに参照させていないことが原因です。次のプロンプトで修正します。

```
docs/LESSONS_LEARNED.md を確認してください。
今回の作業に関係する失敗や改善ルールがあれば回答に反映してください。
```

**エラー4：違う方向に進む**

目的・やらないこと・優先順位が曖昧なことが原因です。次のプロンプトで修正します。

```
今回やることと、今回やらないことを分けて整理してください。
この範囲から外れた提案はしないでください。
```

**エラー5：/requirements を呼び出しても動かない**

配置場所やファイル名が間違っている可能性があります。次の場所に置かれているか確認します。

```
.claude/skills/requirements.md
.claude/skills/design-review.md
.claude/skills/implementation.md
.claude/skills/review.md
```

確認方法はこれです。

```
.claude/skills/requirements.md を確認してください。
このファイルから反映するルールを列挙してから、回答を作り直してください。
```

**エラー6：initial-instruction.mdが反映されない**

このファイルは自動で読まれません。作業開始時に手動で読ませます。

```
prompts/initial-instruction.md を読んでください。
反映するルールを箇条書きでまとめてから、以後の回答をしてください。
```

## 注意点

Claude Codeの構造化で一番効果があるのは、プロンプトを長くすることではありません。**毎回の依頼文を短くするために前提やルールを外に出すこと**です。

構造化ファイルを作りすぎると管理が面倒になります。最初から完璧な構成を作る必要はありません。まずはこの3つだけで十分です。

一発投げで全フェーズ実行したい場合は、役割別の設定ファイルを4つ追加します。
