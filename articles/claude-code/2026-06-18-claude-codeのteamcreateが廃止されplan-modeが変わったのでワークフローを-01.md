# Claude Codeの更新とワークフローの変化

最近の更新で以下の変化があったので、それに合わせてスキルの手順を変える必要があった。

* Plan Modeの変化
* TeamCreateの廃止

TeamCreateが v2.1.178 でなくなったが、Agent Teamsは残っているようだ。それに伴い、以前に書いた[エージェントチームの記事](https://zenn.dev/n314/articles/e06f189f093902)のスキルが使えなくなったので、現在やっていることを書いておく。

また、前からやっていたが、セッションを振り返って改善するための仕組みを強化したのでそれも記述しておく。

# Plan Modeの変化

いつから変わったのか分からないが、TeamCreateが使えなくなったのと同じぐらいの時期にPlan Modeの動きが大きく変わった。  
Plan Modeでは**This supercedes any other instructions**という最優先のメッセージと共に、Exploreを使ってコードを調べたりレビューをしたりという各Phaseに分かれたワークフローのメッセージが差し込まれる。

スキル中にPlan Modeを切り替えたりすると、Plan Modeのワークフローとスキルのワークフローが混ざってスキルのワークフローを無視するような動きになる。

Plan Modeはそれ自体が細かなワークフローを含むので、単に「Read Onlyで進める」という目的では使わない方が良さそうだった。

以前はPlan Modeのワークフローを優先するような動きになっていなかったので、どこかで更新されたのだと思われる。

# TeamCreate廃止に伴う依存スキルの変更

TeamCreateが廃止されたので「チーム」という概念がなくなったのかもしれない。ただし、公式サイトにはTeamCreate/TeamDeleteがなくなっただけでAgent Teamsの説明は残っている。これをチームと言っていいのか分からないが、現状は過去に作ったスキルを大きく変えずに進めることにした。

## AskUserQuestion の使い方

AskUserQuestion は **決定取得ツール** であり、議論・説明ツールではない。役割分担:

- **インラインテキストで担当**: 複数案の比較、コード例、トレードオフ説明、論点の列挙
- **AskUserQuestion で担当**: 説明済みの選択肢から1つ選んでもらう確定アクション

理由: AskUserQuestion の選択肢説明だけでは情報不足になりがち。比較は必ずインラインのテキストで先出しする。

### 使う前のチェックリスト

以下を **全て満たす** 場合のみ AskUserQuestion を使う:

1. 選択肢は **相互排他** で **4個以下** に収まる
2. 各選択肢の意味が **label + 短い description** だけで伝わる
3. 自由記述（数値・パス・命名）ではなく **既定肢からの選択** である
4. ユーザーが必要な比較情報を **すでにインラインテキストで見ている**

1つでも No なら、先にテキストで補完するかテキスト質問に切り替える。

### 回答後の対応

AskUserQuestion の回答に「詳しく」「詳細を教えて」等の情報要求が含まれていたら、すぐに次の質問やPhase 4に進まず、まずテキストで該当論点の詳細を説明する。詳細出力後に改めて簡潔な決定質問を行い、Phase 4 に進む。

まとめに含める内容:

- 調査まとめ（フロー図、調査結果表）
- 編集対象ファイル一覧と、各ファイルの変更概要
- 実装の難易度と工数

まとめにコード例を含める場合は、提示前に以下をセルフチェックする:
- 調査結果（DB仕様、ドメイン事実）を反映した最短実装になっているか
- 不要な一時変数・冗長なループになっていないか（例: 1SKU=1行集約と判明しているのに累積用配列を作っていないか、2回ループで済むものを1回にできないか）
- 「将来の拡張」「万一の安全策」を名目にした不要な抽象が入っていないか

- 既存パターンとの一貫性が気になるなら 調査サブエージェント に類似実装との比較を依頼

レビュー結果はメインセッションで集約し、修正が必要なら修正する。

# セッション振り返りのためのスキル

## 同じ課題を異なる方法で実装したブランチを比較する

スキルや指示を改善するために複数のブランチを比較するためのスキルを作った。  
Claude Codeで異なる方式で複数実行したり（上記team-issueを使わない、など）、Codexで実装したり、複数の実装があるときにこのスキルで比較してフィードバックを受ける。

内容が長くプロジェクト固有の記述も多くあるので、以下の文章はかなり省略している。

.claude/skills/compare-impl/SKILL.md

```
---
name: compare-impl
description: 同一課題の複数実装（ブランチ/worktree）を詳細比較し、復元用パッチを保存する
argument-hint: 課題(issue.mdパス等) 比較対象1 比較対象2 [比較対象3...]
disable-model-invocation: true
---

同一課題に対する複数の実装を詳細比較し、ブランチ削除後でも復元できるように
**分岐元ハッシュ + git format-patch** を保存する。

入力: `$ARGUMENTS`
- 先頭: **課題**（issue.md のパス、または課題名）
- 以降: **比較対象**（ブランチ名 または worktree/clone のディレクトリパス）を2つ以上

下記 Phase を Todo リストで管理しながら順に実行する。

---

# 作業ログ規約（process.md）— 全 Phase 共通・最優先

このスキルは各 Phase の作業を **`<保存ルート>/impl-compare/process.md` に逐次追記**しながら進める。
これは「やったつもり／黙ってスキップ」を構造的に塞ぐための背骨であり、comparison.md（結論）に対する**根拠と作業記録**。
過去にフロー図検証やチェックリストを**サイレントにスキップ**して誤結論を出した事故があり、その再発防止が目的。

- **各 Phase / 各ゲートは、process.md に「証拠つき done」か「スキップ：対象＋理由」のどちらかを必ず残す。**
  空欄・bare な「✓」「確認した」だけは**禁止**（散文の主張は偽れる）。**サイレントなスキップ・代替を禁止**し、
  逸脱するなら必ず「スキップ：X を Y の理由で代替／省略」と**明示申告**する。
- **証拠＝ユーザーが監査できる実物**にする:
  - Phase 0 → 要件述語リスト本体／確定した対象・BASE
  - Phase 2 → チェックリスト×全案のマトリクス（差／差なしを各セル明記）／変更関数の呼び出し index（functioncall.md）
  - Phase 3 → **フロー図**（呼び出し元→分岐/AND・ORゲート→下流消費先を file:line で、観測出力 or マスクまで辿る）
  - DB照合 → 実クエリと出力
  - 各ゲート → 通過/不通過の判定と根拠
- 「潜在bug／regression／scope creep／観測差」と書く主張は、**process.md 内の図/証拠を必ず指す**。
  図/証拠で示せない主張は comparison.md に載せない（読んで確信、は不可）。
- 粒度は **Phase/ゲート単位**（マイクロステップを全部書かない＝ノイズ化を避ける）。
- comparison.md の各セル検証列 ⇄ process.md の証拠を**相互参照**（comparison=結論、process=根拠・作業記録）。
- process.md は Phase 4 で comparison.md と並べて保存する（成果物）。

# Phase 0. 入力の確定

`$ARGUMENTS` を解析し、不足は **AskUserQuestion** で質問する。推測で進めない。

1. **課題の特定 ＋ 要件の述語分解**（比較全体の背骨。ここで手を抜くと後半でコード品質に重心が流れる）
2. **比較対象の特定**（2つ以上必須）
3. **表示ラベルの決定**
4. **保存先の決定**
5. **process.md の作成**（作業ログ規約の起点）

確定した「課題・各対象（ブランチ/ディレクトリ・ラベル・スラッグ）・保存先・process.md パス」を一覧表示してから次へ進む。

# Phase 3. フロー図によるレビュー検証

Phase 2 の比較表は **diff の読みから組んだ草案**。実コードの挙動と食い違う（追えていない分岐・呼び出し先・DB操作）ことがあるため、比較表の**各行（観点）を1行ずつ**、`flow-diagram-generator` の描くフロー図で裏取りして確定する。

**分担を厳守する:**
- **flow-diagram-generator はフロー図を描くだけ。** 各対象の実コードを追跡し、その観点に関わる実行フロー（呼び出し階層・分岐・DB操作・file:line）を**事実として**図示させる。**評価・対象間の比較・優劣判断・修正提案はさせない**（評価軸を持ち込むと事実が歪む）。
- **検討はメインセッションが行う。** 返ってきたフロー図を突き合わせ、比較表の記述が実コードと合うか検証し、必要なら表を直す。

**手順（比較表を上から1行ずつ。1行終えてから次の行へ。並列は同一行内の対象間のみ）:**

1. **その行の追跡対象を特定**: 観点が触れるコードパスを1点に絞る（例:「未ログインガード」→ 入口〜ガード判定、「ラベル保存SQL」→ クエリ組み立て〜実行）。
2. **対象ごとに並列でフロー図エージェント起動**: その行に登場する**全対象**について `flow-diagram-generator` を**同時に**起動する。
   **挙動が分かれる対象を1つも飛ばさない**（「他案と同じだろう」で省略しない）。各エージェントに渡す:

# Phase 5. 完了報告

**あっさり済ませない。** 保存先の表示だけで終わらず、比較の結論まで含めた詳細な報告をターミナルに出す。

**推奨確定前ゲート（最初に必ず実行）**: 蓄積したコード印象でなく **Phase 0 の要件述語と issue.md を読み直し**、結論を**再導出**する（design-judgment-rules.md 原則0(d)）。確認:
- 各推奨判断が**どの述語/failure-mode に紐づくか**言えるか（言えない・`[純コード品質]` 偏り＝原則0赤信号→要件軸へ引き戻す）。

# 注意

- 本プロジェクトのルール上、**書き込み系 git 操作（commit/push/branch削除/am等）は手動**。このスキルは read-only な `diff`・`format-patch`・`log` のみ実行し、復元適用コマンドは manifest に記載するだけに留める。
- 比較対象が3つ以上でも同じ流れで対応する。
```

チェック項目が多すぎてまだ整理しきれていない。AIに任せるとどんどん増えるので手動で書き直した方がいいかも。  
比較したあとに改善点を適用し、再度スキルを実行すると再び異なる指摘が出てくることも多いので（しかも少しズレている）、AIに優劣を判定してもらうのは難しいと感じている。

会話のやり取りの中で指摘した内容をスキルに全部盛り込んだとしても、その会話セッションとスキル実行セッションでは異なる結果になってしまう。もうこれは半分諦めて、AIに相違点を出してもらうだけにして評価は自分でやった方がいいかもしれない。それでも、単に「ブランチA、B、Cを比較して」と言うよりはだいぶ良い結果になっているので改善しながら試しているところ。
