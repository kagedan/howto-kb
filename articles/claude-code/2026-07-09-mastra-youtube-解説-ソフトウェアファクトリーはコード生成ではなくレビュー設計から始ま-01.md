## 詳細

### コードは安くなったが、レビューは安くなっていない

「コードを書くこと自体は安くなった。でも、レビューは安くなっていない。みんな PR の洪水に溺れていると言うけれど、実際に溺れているのは PR そのものではなく、質の悪い PR なんだ」

そんな強い一言から、この回は始まります。番組では HumanLayer の Dex Horthy を迎え、まずは Dex の紹介が入ります。サンフランシスコでのイベント運営や、AI まわりの論点を少し早めに言い当てる人として以前から注目していた、とホストは話します。そこで Dex が最近公開した動画を流し、その内容を起点に話を広げていこう、という流れになります。

動画の中で Dex は、今の開発現場を覆っている `AI code slop` という言葉を何度も繰り返します（訳註：`slop` にはきれいな定訳がありませんが、ここでは「**雑なコード**」くらいにとらえればよいと思います）。AI でコードを書く量は増えた。コミットも増えた。コードベースが壊れる速度も速くなった。コードをよく読まず、理解せず、ただ生成量だけを増やしていくと、いずれ大きな行き止まりにぶつかる。だから「少し落ち着け、全部壊れているぞ」と言うわけです。

続いて HumanLayer の短い紹介動画が流れます。Dex は自分を HumanLayer の CEO 兼共同創業者だと名乗り、この 1 年、AI コーディングにおける slop の問題をずっと観察してきたと説明します。今回公開した HumanLayer は、AI IDE、コラボレーションプラットフォーム、そしてソフトウェアファクトリーのための building blocks をまとめたものだと言います。狙っているのは、複雑なコードベースで難しい問題を解くエンジニアが、SDLC （Software Development Life Cycle：ソフトウェア開発ライフサイクル）全体で 2 倍から 3 倍速く動けるようにすることです。

そこで鍵になるのが、「エージェントを走らせる前に、まずシステムについて考える」という発想です。優れたエンジニアリングチームは昔から design review をやってきた。しかし多くのチームでは、そのプロセスが断片的で、手作業で、AI 以前の世界に取り残されている。HumanLayer は、その失われていた高レベル設計の議論を、チームの日常に戻そうとしているのだと語ります。

動画が終わると、Dex 本人がその問題意識を補足します。自分が一貫してやってきたのは hype （誇大宣伝）を切り裂くことだ、と。AI の周囲には「これをやらないと乗り遅れる」というノイズが多すぎる。そこには不安商法も、単なる engagement bait （「いいね！」集め）も、善意の誤解も混ざっている。だからこそ、自分は実際に顧客やユーザーの現場で役に立った、手触りのあるやり方だけをできる限り共有したいのだと言います。

その延長で彼は、「使える言葉を守ること」の大切さにも触れます。`agent` は人によって百通りの意味を持つ言葉になってしまった。`spec-driven dev` もそうだ。`harness engineering` も、もともと定義が曖昧なまま広がってしまった。研究所や agent builder が作る inner harness と、現場側で feedback loop を含めて設計する outer harness は別のものなのに、ひとつの流行語で雑にまとめられてしまう。だから Dex は、言葉が壊れる前に、現場で本当に役に立つ意味をなんとか守りたいのだと話します。

さらに彼は、モデルの学習や強化学習の中身を AI 研究者レベルで理解していなくても、コーディングモデルがどう鍛えられているかを少し掴むだけで、日々の使い方はかなり変わると強調します。ここ数週間、自分でもかなり深く調べ、分からない数式は Claude に噛み砕いて説明させながら理解してきた。その結果として見えてきたことを、今日はソフトウェアファクトリーの話とつなげて語りたい、というのが前半の導入です。

### ホワイトボードで描く、AI 時代のソフトウェアファクトリー

ここでホストが、「Mastra のユーザーや、コードを書く人たちから最近よく聞く言葉が 2 つある。`software factory` と `SDLC の自動化` だ。ほぼ同じ意味で使われているけれど、Dex から見るとどうか」と話を振ります。すると Dex は、「その言葉も次に壊れる言葉だ」と即答します。もう手遅れに近い、でも一度ホワイトボードで整理しよう、と画面共有に入ります。（訳註：私なりに日本語訳したホワイトボードを作ってその画像を貼っていきます）

![](https://static.zenn.studio/user-upload/d3b5934cecca-20260708.png)

Dex はまず、AI 以前から存在していた最も抽象的なソフトウェアファクトリーを描きます。最初にあるのは、やるべき仕事の一覧です。Linear でも Jira でも issue tracker でも構いません。そこにはエンジニア、PM、CEO のビジョン、顧客要望など、さまざまな入力が流れ込みます。そのあとに architecture planning や sprint planning のような人間のプロセスがあり、誰かが実際にものを作る。ローカルでテストをする。ブラウザで手動確認をする。PR を作る。人間がレビューする。PM や QA やエンジニアがテストし、CI/CD が unit test や静的解析やセキュリティチェックを回し、通ったら production に出す。ユーザーが使い、support ticket や feature request や monitoring の alert が戻ってきて、それがまた次の仕事になる。

つまり、AI が入る前から、ソフトウェア開発にはすでに外側の大きなループがあるし、レビューで差し戻されるような内側のループもあるわけです。Dex は、まずこの全体像を見失わないことが重要だと言います。

そのうえで、もっとも素朴な AI ソフトウェアファクトリーを描き直します。違いはひとつで、「誰かが作る」の部分が「agent が作る」に置き換わるだけです。しかし、ここには実際にはかなり多くの部品が入ります。作業を起動する orchestration、コーディングエージェントを閉じ込める sandbox、Claude Code や Codex のような inner harness、実際に推論する LLM、必要ならブラウザや VM や各種 feedback tool も必要になる。

![](https://static.zenn.studio/user-upload/5174c0e82b11-20260708.png)

そして最初に足されるものが `agentic code review` だと Dex は言います。なぜなら、agent がコードを書く段になると、「ものを作る」工程が 2 時間から 2 分にまで縮むからです。すると backlog は「やるべきことの山」から「レビュー待ちの山」に変わってしまう。ボトルネックが別の場所へ移るのです。だから次に欲しくなるのが、agent によるコードレビューや、PR に向けた agentic testing になります。サンドボックス内でブラウザを動かし、外から見れば manual test に近いことを機械にやらせるイメージです。

![](https://static.zenn.studio/user-upload/7e6d14ca330e-20260708.png)

それでも人間のレビューや人間のテストは残ります。ただし、すでに低い位置の不具合を agent が拾ってくれているので、人間は以前より自信を持って確認できる。レビューの床が上がる、という説明です。

### planning を抜いた瞬間に、slop は増殖を始める

Dex はさらに、その次に追加されがちなものとして `agentic feedback loop` を挙げます。support や feedback、あるいは Sentry のような monitoring から上がってきたエラーを、いったん人間に投げずにそのまま tracker に流し込み、agent に拾わせる。たとえそのうち 30% しか一発で直せなかったとしても、その 30% は人間が最初から考えずに済む。人間は出てきた PR を見て判断するだけでいい。こうして、より外側のループまで agent が回し始めると、だんだん「agentic software factory」が形になってくるわけです。

![](https://static.zenn.studio/user-upload/c926741dde70-20260708.png)

ただし、ここで多くの人が planning を落とし始める、と Dex は指摘します。エージェントが速くなったのだから、どう作るかを 2 時間も考える必要はない、細かい分解も不要だ、全部さっさと走らせればいい、と。

![](https://static.zenn.studio/user-upload/cbdec652fcd2-20260708.png)

Mastra 側のメンバー達は、そこをまだ捨てていないと答えます。sprint planning のような儀式は薄くなったが、goal planning はやっている。各人が agent と組んだ `army of one` のような形で、「金曜までにこれを達成する」といった人間レベルの目標を持ち、そのうえで factory に入る。architecture planning も儀式としては弱くなったが、やらないと何かがおかしくなるので、暗黙の前提として残しているという話です。

ここで出てくる表現が印象的です。planning を外すと、毒が漏れ始める。slop が発酵し始める。ある領域を誰もちゃんと理解しなくなり、そのわずかな乱れが次の乱れを呼び、問題が増殖していく。ceremony は減らせても、planning そのものは消せない、という認識が共有されます。

さらにホストは、「人間は今でもちゃんとコードレビューしているのか」と聞きます。答えは yes です。しかもかなり頻繁に、プログラム設計そのものを変えるレビューが入る。ただしそれは、人間が素手でコメントを書くというより、自分の持っている文脈を agent に渡して specific な review をさせる形に近い。CodeRabbit のような自動レビューとは別に、人間とその agent が一緒にレビューしている、という整理です。

そのレビューは時にかなり手厳しい。相手本人に話しているというより、相手の agent に言っている感覚になるので、あまり遠慮もしない。けれど、そのくらい強く設計に介入しないと、最初に踏み外したものは戻ってこない。ここもかなり実務的な温度感です。

![](https://static.zenn.studio/user-upload/3142e884e69b-20260708.png)

そして Dex は、次の図として `lights-off software factory` を出します。これは人間のレビューをほぼ外し、CI/CD、agentic testing、静的解析、セキュリティチェックだけを頼りに、最後は「人間は何を作るかだけ決め、あとは閉じたループで回る」という理想像です。ユーザーが不満を言えば直る。誰も深く考えなくても直る。確かに夢のような構図です。

しかし Dex は、自分たちも 2025 年 9 月ごろにほぼそれを作ってみた結果、3 か月から 6 か月ほどでコードベースが目に見えて壊れていったと言います。レビューを抜いたり、レビューを雑にしたりして、「多少 sloppy だけど agent に任せておこう」とやり始めると、コードベースは時間とともにボロボロになる。その背景にある仕組みを説明するために、彼はコーディングベンチマークの話に移ります。

### いまのコーディングベンチマークは、bad code design を罰せない

Dex が持ち出すのは、たとえば `SWE-bench Multilingual` のような評価の形です（訳註：GitHubの実際の Issue をAIに渡して、コードを修正させ、テストが通ったら「解決」とカウントする評価手法）。そこには problem statement があり、正解となる golden patch があり、テストに対する patch があります。agent がコードを書き換えたとき、もしテストまで触っていたらその変更は捨てて、代わりに正解側のテスト patch を当てたうえで、agent の変更がテストを通るかを見ます。

この方式は、小さく閉じた one-shot 問題を大量に比較するには有効です。1000 問解けるモデルと 1100 問解けるモデルがあれば、後者のほうが coding 能力は少し高そうだという signal にはなる。しかし Dex が問題にしているのは、その verifier が見ているのはあくまで「テストが通ったかどうか」だという点です。そこでは bad code design を罰せない。

強化学習の流れも、彼はかなり噛み砕いて説明します。まず verifier が「テストに通ったか」を返す。モデルは rollout を走らせ、長い agentic trace の中で推論し、ツールを呼び、何度か試行錯誤する。最後に、正しかったか、簡潔だったか、テストに通ったか、時間をかけすぎていないか、といった指標で評価し、その結果を使って重みを少しずつ更新していく。これを何か月も、何千何万という問題で回し続けると、より coding が得意なモデルができる。

ただ、そのループのどこにも、「6 か月後に別のエンジニアがこのコードベースで変更を加えたとき、めちゃくちゃやりにくくなっていないか」は入っていない。こちらを直したら向こうが壊れる、という種類の悪い設計に対して、モデルは十分な罰を受けていない。Dex にとっての bad code quality とはまさにそこですが、その次元をモデルに学習させるのはとても難しいというのが彼の見立てです。

だから結論はかなりはっきりしています。architecture planning を捨てるな。sprint planning を捨てるな。code review を捨てるな。この 3 つを保ったまま leverage を取りにいくべきだ、と。6 時間かけてひどい PR をレビューするくらいなら、最初にどう作るかをモデルと握っておいて、レビューに来たときには 20 分や 5 分で済む状態まで精度を上げたい。大事なのは、モデルが sandbox で書き始め、テストし、あちこちに依存を張り巡らせてから方向修正するのではなく、もっと早い段階で re-steer することです。

![](https://static.zenn.studio/user-upload/5c58dbbf4de5-20260708.png)

HumanLayer が作ろうとしているのも、まさにそこだと Dex は言います。チームで協調しながら、スピードを上げつつ、品質も落とさず、それでも最終的な outcome の ownership は人間が持ち続ける。そのための仕組みをどう作るか、という話です。

ホスト側もこの点に強く共感します。PR review がボトルネックなら、レビューの速度そのものよりも、「レビューに届く時点で 95% から 99% 正しいコードになっている確率」を上げるべきだ、という Dex の言葉を all hands でも引用した、と話します。

### いま必要なのは、長い時間軸でコード品質を測る benchmark

最後に、「では結局、より良い planning や architecture が唯一の道なのか」と問われると、Dex は少しバランスを取って答えます。deterministic な linter はもちろん有効だし、agentic code review も役に立つ。ただ、それだけを唯一の対策にすると、コードベースはかなりの確率で時間とともに劣化する。

ここで彼は、冒頭の言葉をもう一度なぞります。コードは安くなった。でもレビューは安くなっていない。人は PR に溺れているのではなく、bad PR に溺れている。YOLO プロンプトで適当に AI に投げると、PR は平均で半分くらい手戻りが必要になるかもしれない。逆に言えば、5% だけ手直しすれば済む PR は本当にレビューしやすいし、むしろ楽しい。目指すべきなのは、そういう PR を factory の入口で増やすことだ、という話です。

そのうえで、今の benchmark 自体にも改善が必要だと話が進みます。単発タスクを解いたかではなく、5 個目、6 個目のタスクになったときも速く正しく進めるかを見る、より長い horizon の benchmark が必要ではないか。最初の 4 つが一応通っていても、その時点でコードベースがぐちゃぐちゃなら、5 個目で急に遅くなるはずだからです。

Dex は、自分たちも Harbor ベースの benchmark harness を作っていると明かします。ポイントは、モデルに事前に roadmap を教えず、少し変わったプロダクトに対して 20 個くらいの feature request を連続で積んでいくことです。各 feature を失敗したら罰し、さらにたとえば issue 20 まで Opus 級モデルで進めたあと、issue 21 を Sonnet や Haiku に渡してみる。もし小さくてあまり賢くないモデルでも次の issue を解けるなら、そのコードベースはまだきれいだという signal になる。逆に、そこで急に解けなくなるなら、前段で書かれたコードの質に問題があることが増幅されて見える。

こうした長期的な benchmark がなければ、モデルの重みは「技術的負債を増やさない」という方向にはなかなか進化しないだろう、と Dex は言います。今の最適化目標は、「テストに通ったか」「タスクを達成したか」に寄りすぎていて、「将来のチームや将来の agent にとって扱いやすいコードだったか」は入っていないからです。

最後は採用とプロダクトの案内で締めくくられます。RLVR や code quality に効くモデル改善に興味があるなら連絡してほしい。HumanLayer は 3 人まで無料で試せる。小さなチームがまず触ってみるにはちょうどいい。そう言って回は終わります。
