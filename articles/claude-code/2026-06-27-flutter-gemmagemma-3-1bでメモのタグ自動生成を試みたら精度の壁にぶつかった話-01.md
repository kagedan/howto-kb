## はじめに

「メモアプリにAIを組み込みたい」と思ったとき、最初に頭に浮かんだのはクラウドAPIでした。でも、すぐに引っかかりが生まれました。

* メモの内容はプライベートな情報が多い
* オフライン環境でも動いてほしい
* API費用をユーザーに負担させたくない

この3つを解決できるのが、**オンデバイスLLM**です。端末の中でモデルが動くため、データがサーバーに送られることがなく、ネットワーク不要で、追加コストもかかりません。

そこで `flutter_gemma` パッケージを使い、Gemma 3 1B をオンデバイスで動作させるAIメモアプリをClaude Codeで開発しました。機能のひとつとして、**メモ内容から自動でタグを生成する機能**を実装したのですが——ここで壁にぶつかりました。

この記事は「うまくいった話」ではありません。**タグ生成の精度が思ったより低く、その原因を技術的に考察した記録**です。同じ課題にぶつかっている方の参考になれば幸いです。

## アプリの概要

* **フレームワーク**: Flutter
* **使用パッケージ**: `flutter_gemma`
* **モデル**: Gemma 3 1B（int4量子化）
* **動作**: 完全オフライン・オンデバイス
* **対象OS**: Android（iOS対応予定）

メモを書いた後にAIボタンをタップすると、本文を読んでタグを自動生成するシンプルな設計です。

## タグ生成の実装

プロンプトはざっくり以下のような形にしています。

```
final prompt = '''
以下のメモの内容を読んで、適切なタグを3〜5個生成してください。
タグはカンマ区切りで出力してください。余計な説明は不要です。

メモ:
$memoContent

タグ:
''';
```

シンプルな指示追従タスクです。「読んで → タグを → カンマ区切りで出す」だけなので、最初は「これで十分動くだろう」と思っていました。

## 実際の挙動：期待と現実

### 期待していた動作

たとえばこんなメモがあるとします。

```
来週の月曜日に上司との1on1がある。
アジェンダは①最近の業務報告②副業申請の相談③来期の目標設定。
事前に資料を準備しておく。
```

期待するタグ：`仕事`, `1on1`, `ミーティング`, `準備`, `目標設定`

!

### 実際の出力（よくある例）

```
タグ: 来週, 1on1, 会議, アジェンダ, 準備
```

……あまり正確なタグは出てきませんでした。

他にも：

* タグが1個しか出ない
* 英語と日本語が混在する（`work`, `仕事`, `meeting` が同時に出る）
* カンマ区切りのはずが改行区切りで出る
* 関係のないところがタグになっている
* タグではなく「このメモは〇〇についての記録です」という要約文が出る

一言で言うと、**出力フォーマットと内容の両方が不安定**でした。

## 考察：なぜ精度が低いのか？

ここからが本題です。いくつかの仮説を立てて整理してみます。

### 仮説1：モデルサイズの根本的な限界

Gemma 3 1B はパラメータ数が約10億。スマートフォンで動かすためのモデルとしては妥当なサイズですが、**指示追従（Instruction Following）の安定性**という観点では、7B以上のモデルと比べて明らかに苦手なことがあります。

「タグを生成する」というタスクは一見シンプルですが、実は複合的な能力を要求します。

* テキストの意味を理解する
* 適切な粒度でキーワードを抽出する
* 指定されたフォーマットで出力する
* 余計なことを書かない（=ネガティブな指示を守る）

特に最後の「余計なことを書かない」は、小型モデルが最も苦手とする部分です。大規模モデルは膨大なRLHF（人間フィードバックによる強化学習）でこの能力を磨いていますが、1Bクラスのモデルでは限界があります。

### 仮説2：量子化による品質劣化

flutter\_gemmaで使うモデルは **int4量子化**されています。量子化とは、モデルの重みを32bit浮動小数点数から4bit整数に圧縮する技術で、ファイルサイズと計算コストを大幅に下げられます。

ただし、品質とのトレードオフがあります。特に**細かい指示の解釈や、出力の一貫性**に影響が出やすいとされています。

「カンマ区切りで出力してください」という単純な形式指定が守られないのは、量子化による精度劣化の影響も考えられます。

### 仮説3：プロンプト設計の問題

これは自分の責任でもあります。現状のプロンプトはシンプルすぎる可能性があります。

大規模モデルであればゼロショットでも動きますが、小型モデルでは **few-shot（出力例の提示）** が効果的とされています。

```
# 悪い例（現状）
タグを3〜5個、カンマ区切りで出力してください。

# 改善候補
以下の例を参考に、タグを出力してください。

例）
メモ: 今日はジムで30分走った。体重が少し減った気がする。
タグ: 運動, ジム, ランニング, 健康

メモ: Flutterのstate managementについて調べた。RiverpodとBlocの違いが気になる。
タグ: Flutter, 技術調査, Riverpod, Bloc

メモ: $memoContent
タグ:
```

few-shotを加えることで、「タグとはこういう粒度・形式のものだ」をモデルに伝えられます。これはまだ試せていないので、次のステップとして試してみます。

### 仮説4：タスクとモデルの相性

そもそも「タグ生成」は生成タスクです。モデルが自由に文字を出力できるため、指定外の文字列が入り込みやすい。

一方、**分類タスク**（「このメモは仕事・プライベート・学習のどれ？」）であれば、選択肢を限定できるため小型モデルでも安定しやすいです。

タグをあらかじめ定義しておき、「以下のタグから当てはまるものを選んでください」という設計に変えると、精度が大幅に改善する可能性があります。ただし、ユーザーが自由にタグを作れるというメモアプリの柔軟性は失われます。

## 整理：仮説と対策のまとめ

| 仮説 | 可能性 | 対策 |
| --- | --- | --- |
| モデルサイズの限界 | 高 | 3B/7Bモデルへの変更、またはハイブリッド構成 |
| 量子化による劣化 | 中 | int8量子化モデルとの比較検証 |
| プロンプト設計の問題 | 高 | few-shot追加、出力フォーマットの厳密化 |
| タスクとモデルの相性 | 中 | 定義済みタグからの選択方式への変更 |

## 次のステップ

優先度順に試してみようと思っています。

**1. プロンプト改善（コスト：低）**  
few-shotの追加と、出力フォーマットの厳密化。モデルを変えずに改善できる可能性があるため、まず最初に試す。

**2. モデルのアップグレード（コスト：中〜高・現実的には困難）**  
より高性能なモデル（3B、7B）を試せば精度が改善する可能性は高い。ただし、スマートフォンで動かすという前提がある以上、モデルサイズには物理的な上限がある。Gemma 3 の3Bモデルでも量子化後で2GB前後のファイルサイズになり、推論時のRAM使用量も相応に増える。ハイエンド端末なら動作するかもしれないが、一般的なユーザーの端末で快適に動かせるかは別問題だ。「高性能なモデルを使いたい」という欲求と「オンデバイスで動かす」という制約は、根本的にトレードオフの関係にある。

**3. ハイブリッド構成（コスト：高）**  
オフライン時はオンデバイスLLMで簡易処理、オンライン時はGemini APIにフォールバックする構成。以前Gemini APIを使った経験があるので、実装自体は難しくない。ただしプライバシーポリシーとUX（「いまネットに送りました」という告知）の設計が必要になる。

## おわりに

「オンデバイスLLMで完結するAIメモアプリ」というコンセプト自体は今でも正しいと思っています。ただ、1Bという小さなモデルでどこまでできるかという、その現実の壁にぶつかりました。

精度が低いことを記事にするのは少し勇気が要りましたが、同じ壁にぶつかっている人が「あ、自分だけじゃないんだ」と思えるなら、それで十分だと思って書きました。

オンデバイスLLMは「プライバシー・オフライン・コスト」という問題を一気に解決してくれる魅力的な選択肢です。ただ、その恩恵を受けるためにモデルを端末に閉じ込めた瞬間、性能の天井も一緒に持ち込むことになります。この制約とどう折り合いをつけるか、それがオンデバイスAI開発のリアルな課題だと実感しています。
