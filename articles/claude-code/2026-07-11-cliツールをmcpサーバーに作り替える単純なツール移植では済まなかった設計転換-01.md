## はじめに

「MCP対応させる」と聞くと、多くの人はこう考えるのではないだろうか。「CLIのコマンドを1つずつMCPツールに置き換えれば終わりだろう」と。`fetch` は `fetch_feeds` に、`generate` は `generate_article` に――そんな1対1の対応表を頭の中で描いてから着手する人も少なくないはずだ。筆者自身、最初はまさにその計画で作業を始めた。

しかし実際に自作のRSSキュレーションCLI「feed-curator-cli」をMCPサーバー「feed-curator-mcp」に作り替えてみると、コマンドをツールに置き換える作業だけでは全く済まなかった。出力の形、エラーの扱い方、そして何より「AIをどこに置くか」という設計の前提そのものをひっくり返す必要があった。

本稿は筆者が自作CLIをMCPサーバーに作り替えた実録である。両リポジトリはすでに公開しているので、気になる箇所は実物のコードと突き合わせながら読んでもらえる。

<https://github.com/yunbow/feed-curator-cli>  
<https://github.com/yunbow/feed-curator-mcp>

あくまで筆者の実例に基づく記録であり、MCP設計の一般論として全てのケースに当てはまると主張するものではない点は先に断っておく。

5行まとめ（クリックで展開）

**TL;DR（5行まとめ）**

1. CLIのコマンドをMCPツールに1対1で置き換える計画は、実装を進めるうちに破綻した
2. 最大の転換はAIの位置――CLI版はAIを子プロセスとして呼んでいたが、MCP版ではAI（ホストAI）が呼び手側に回る「prepare/save」パターンに再設計した
3. 出力は `console.log` による人間向け表示から、「人間可読なMarkdown＋機械可読なJSON」の2層レスポンスに変わった
4. エラー処理は「プロセス終了コード＋人間がログを読む」前提から、「AIが読んで自己修正できる構造化エラー」に変わった
5. 呼び手がAIになったことで信頼境界が変化し、CLI版には存在しなかったSSRF対策・パストラバーサル対策が新たに必要になった

---

題材となる feed-curator-cli は、RSSフィードを収集し、生成AIで記事を採点・要約し、著作権に配慮した事実ベースの記事ドラフトを作るCLIツールである。処理は5ステップに分かれており、全データはSQLite（`data/curator.db`）に一元管理されている。設定ファイルは存在しない。

| ステップ | コマンド | 内容 | 保存先 |
| --- | --- | --- | --- |
| 1 | `fetch` | 登録フィードからURLを取得 | `articles` テーブル |
| 2 | `score` | 重複・類似チェック後、生成AIで9観点をスコアリング（0-10×9 → 0-100に正規化） | `articles` テーブル |
| 3 | `extract-facts` | スコア閾値以上・非重複の記事から、著作権に配慮した事実ベースの素材を抽出 | `facts` テーブル |
| 4 | `list` | おすすめ事実の一覧を表示 | - |
| 5 | `generate <factId>` | 選択した事実から記事ドラフトを生成 | ファイル出力＋DB更新 |

`pipeline` コマンドは1→2→3を一括実行するショートカットだ。生成AIの呼び出しは外部プロセス `claude -p` を `spawnSync` で呼ぶ形（`src/claude.ts`）で実装されており、AnthropicのSDKには依存していない。

Node.js 20+、TypeScript、ESM。依存は `commander ^13.1.0`、`better-sqlite3 ^11.8.1`、`rss-parser ^3.13.0` のみとシンプルだ。

src構成もシンプルで、`cli.ts`（Commanderのエントリ）、`claude.ts`（AI呼び出し）、`db.ts`（DB操作の一極集中）、`dedup.ts`、`rss.ts`、`util.ts`、そして `commands/` 以下に `fetch.ts` / `score.ts` / `facts.ts` / `list.ts` / `generate.ts` / `feed.ts` / `config.ts` の各コマンドハンドラが並ぶ。`buildScoringPrompt` のようなプロンプト構築関数も、対応するコマンドハンドラファイルに同居している。「1ファイルにコマンドの表示・DB操作・プロンプト組み立てが全部入る」という、CLIとしては自然な構成だ。

## 2. 素朴な計画と3つの壁——なぜ「コマンド→ツール変換」で済まないのか

最初に立てた計画は単純だった。`fetch` → `fetch_feeds`、`generate` → `generate_article` のように、既存コマンドをMCPツールへ機械的に対応させる。コマンドの引数をツールのinputSchemaに変換し、`commands/*.ts` の中身をほぼそのまま移植すればよい――そう考えていた。

しかし着手してすぐに3つの壁にぶつかった。

**壁1: 出力の壁**

CLI版の `commands/*.ts` は、`console.log` による人間向け表示とドメイン処理が一体化しており、戻り値を返さない設計になっている。例えば `cli.ts` のコマンド定義を見ると分かる。

src/cli.ts (CLI版)

```
program
  .command("generate")
  .description("【5】選択した事実(Fact ID)から生成AIで記事を作成し、記事作成済みに更新")
  .argument("<factId>", "生成対象の Fact ID", (v) => parseInt(v, 10))
  .option("-f, --force", "生成済みでも再生成する")
  .option("-o, --out <dir>", "記事の出力ディレクトリ", "output")
  .action((factId, opts) => {
    runGenerate(factId, opts, dbPath());
  });
```

`runGenerate` は内部で `console.log` を叩きながら処理を進め、値を返さない。だがstdio transportで動くMCPサーバーでは、**stdoutはJSON-RPCの通信専用**になる。stdoutに通常ログを出すとプロトコルを壊してしまう。つまり「表示」と「処理」を分離しないまま移植することはそもそも不可能だった。

**壁2: エラーの壁**

CLI版のエラー処理は `console.error` + `process.exitCode = 1` というシンプルなものだ（`cli.ts` 末尾）。

src/cli.ts (CLI版)

```
program.parseAsync(process.argv).catch((err) => {
  console.error(`エラー: ${err.message}`);
  process.exitCode = 1;
});
```

これは「プロセスが終了し、人間がターミナルのログを読んで対処する」ことを前提にした設計だ。しかしMCPサーバーの呼び手はAIであり、プロセスを終了させるわけにもいかない。AIがエラーの内容を読んで次の一手を判断できる形が必要になる。

**壁3: AI呼び出しの壁**

そして最大の疑問はここだった。CLI版のコアである `score` / `extract-facts` / `generate` は、いずれも `claude -p` サブプロセスに依存している（`src/claude.ts`）。

src/claude.ts (CLI版)

```
export function callClaude(prompt: string, timeoutMs = DEFAULT_TIMEOUT_MS): ClaudeResult {
  const result = spawnSync("claude", ["-p", "--output-format", "text"], {
    input: prompt,
    encoding: "utf-8",
    timeout: timeoutMs,
    maxBuffer: 50 * 1024 * 1024,
    // Windows では claude が .cmd / .ps1 ラッパーのことがあるため shell 経由で解決する
    shell: process.platform === "win32",
  });
  ...
```

MCPサーバーはホストAI（すでに会話しているAI）から呼び出される。その中でさらに `claude -p` という**別のAIプロセス**を呼ぶのは、二重にAIを起動することになる。推論コスト、レイテンシ、監査対象が増えるし、そもそも「ホストAIがすでに存在するのに、なぜサーバー内からもう一体呼ぶのか」という設計上の疑問が拭えなかった。

この3つの壁は、個別に潰せるようなバグではなかった。突き詰めると根っこは1つで、「AIをどこに置くか」という前提が違っていたのだ。

なお、すべてのコマンドがこの壁にぶつかったわけではない。`feed add` / `feed list` / `config set` のようにAIを介さない純粋なCRUD操作は、`add_feed` / `list_feeds` / `update_settings` へほぼそのまま1対1で変換できた。壁にぶつかったのは、あくまで生成AIが絡む `score` / `extract-facts` / `generate` の3ステップだけだ。裏を返せば、「AIが絡む処理」と「絡まない処理」を切り分けて考える必要がある、ということでもある。

## 3. 最大の設計転換——AIの居場所が逆転する

ここが本記事の核心である。

CLI版の構造を一言で言えば「**プログラムが主、AIが従（子プロセス）**」だった。`commands/*.ts` が処理の主導権を握り、必要なタイミングで `callClaude()` を呼んでAIに仕事をさせ、結果を受け取ってDBに保存する。AIは道具として使われる側だった。

MCP版ではこれが逆転する。「**AIが主（呼び手）、サーバーが従**」という構造になる。サーバー自身は外部LLMを一切呼ばない。推論（採点・事実抽出・記事執筆）はすべてホストAI――つまりMCPクライアントの向こうで会話しているAI自身――が担う。

この逆転を実装したのが「**prepare\_\* / save\_\*** 」という2段構成のパターンだ。例えば採点ステップは次のように分かれる。

1. `prepare_scoring` を呼ぶ → サーバーは重複除外済みの記事バッチと、採点の指示文（instruction）、そして次に呼ぶべきツールの案内（next\_step）を返すだけ
2. ホストAIが自分の推論（会話のターン内）で9観点を採点する
3. `save_scores` を呼ぶ → 採点結果を型付き引数として渡し、サーバーはDBに保存する

`prepare_scoring` の実装（`server.ts`）を見ると、この構造がそのまま表れている。

src/server.ts (MCP版、prepare\_scoring)

```
return toolResult(
  {
    duplicates_marked: duplicates.length,
    remaining_unscored: Math.max(0, keepIds.length - articles.length),
    batch_size: articles.length,
    articles,
    instruction: buildScoringInstruction(articles, settings.target_country),
    next_step: {
      tool: "save_scores",
      description: "9観点(各0-10)で採点した結果を保存する",
      arguments: { scores: "[{ id, novelty, ... empathy, reasoning }]" },
    },
  },
  `🧮 重複 ${duplicates.length}件除外 / 採点対象 ${articles.length}件。指示に従い採点して save_scores を呼んでください。`,
);
```

サーバーがやっているのは「重複除外」「素材の整形」「指示文の構築」「次の一手の提示」であって、**採点そのものは一切行っていない**。採点はこのレスポンスを受け取ったホストAIが、会話の続きとして自分の頭でやる。

副産物として、CLI版で複雑だった仕組みがまるごと不要になった。CLI版は `callClaude()` → `extractJson()`（コードフェンス優先＋バランス括弧探索のフォールバック）→ `isValidFactItem` 等の手書き形式検証 → 失敗時は `score_attempts` / `fact_attempts` カウンタで最大3回リトライ、という一連の防御機構を持っていた。AIがテキストで返してくる出力を、パースして検証するための仕組みだ。

MCP版では、少なくともこの「AIの自由形式テキストからJSONを抜き出して構文検証する」問題は消える。`save_scores` の引数は最初からzodスキーマで型検証される。

❌ CLI版（AIのテキスト出力を正規表現とバランス括弧探索でパースし、手書きの形式検証にかける）

src/claude.ts (CLI版)

```
export function extractJson<T = unknown>(text: string): T | null {
  // 1. markdown コードブロックを優先
  const fence = text.match(/```(?:json)?\s*([\s\S]*?)\s*```/);
  if (fence) {
    const parsed = tryParse<T>(fence[1]);
    if (parsed !== null) return parsed;
  }
  // 2. 配列 or オブジェクトを直接探し、対応する閉じ括弧までをバランス対応で切り出す。
  ...
```

✅ MCP版（zodスキーマで宣言するだけで、パースと検証をSDKに任せる）

src/server.ts (MCP版)

```
const SCORE_FIELDS = {
  id: z.number().int().describe("対象記事の id（prepare_scoring が返した id）"),
  novelty: z.number().min(0).max(10),
  utility: z.number().min(0).max(10),
  gap: z.number().min(0).max(10),
  emotion: z.number().min(0).max(10),
  authority: z.number().min(0).max(10),
  trend: z.number().min(0).max(10),
  surprise: z.number().min(0).max(10),
  numbers: z.number().min(0).max(10),
  empathy: z.number().min(0).max(10),
  reasoning: z.string().default("").describe("判定理由（40文字以内）"),
};
```

「AIのテキスト出力をパースして検証する」問題が、「zodスキーマで型検証されたツール引数を受け取る」問題に変わったのだ。もちろん、`id` が `prepare_scoring` で返した記事に含まれているか、採点理由が空でないか、スコアが運用上妥当かといった**意味検証**は別途残る。それでも `extractJson` のようなフォールバック処理、構文パース用のリトライカウンタ、そしてMCPサーバー内で `claude -p` を呼ぶための追加認証・実行環境管理は不要になった。

> CLIはAIを子プロセスとして呼んだ。MCPサーバーは自分がAIに呼ばれる側になった。この居場所の逆転こそが、CLI→MCP設計転換の本質だった。

5ステップとMCPツールの対応は次のようになった。

| CLIステップ | MCPツール（prepare/save 2段） | 備考 |
| --- | --- | --- |
| 1. fetch | `fetch_feeds` | 単一ツールのまま（推論不要） |
| 2. score | `prepare_scoring` → `save_scores` | 採点はホストAIが担当 |
| 3. extract-facts | `prepare_facts` → `save_facts` | 抽出はホストAIが担当 |
| 4. list | `list_facts` | 単一ツールのまま |
| 5. generate | `prepare_article` → `save_article` | 執筆はホストAIが担当 |
| pipeline | （意図的に作らず） | オーケストレーションはホストAIの仕事になった |

CLI版にあった `pipeline`（1→2→3の一括実行）に対応するツールは、MCP版には意図的に作らなかった。オーケストレーション――「次に何を呼ぶか」を決める役割自体が、ホストAI側に移ったからだ。単一のツールにまとめる必要そのものが消えた、という判断である。

## 4. 読み手がAIになる——レスポンスとエラーの再設計

呼び手がAIになったことで、レスポンスの設計も作り直す必要があった。

CLI版は `console.log` で人間が読む前提の文字列を出すだけでよかった。MCP版では、13個のツール（`add_feed` / `list_feeds` / `remove_feed` / `get_settings` / `update_settings` / `fetch_feeds` / `prepare_scoring` / `save_scores` / `prepare_facts` / `save_facts` / `list_facts` / `prepare_article` / `save_article`）すべてで共通のレスポンス整形関数を用意した。

src/responses.ts (MCP版)

```
/** 成功レスポンス（Markdown表示 + 構造化JSON） */
export function toolResult(structured: Record<string, unknown>, markdown: string): ToolResponse {
  return {
    content: [{ type: "text", text: markdown }],
    structuredContent: structured,
  };
}
```

`content` は人間可読なMarkdown、`structuredContent` は `outputSchema` と対応させられる構造化JSONという「2層レスポンス」だ。人間がログを見ても分かるし、AIがプログラム的に処理する値としても使える。多くの成功レスポンスには `next_step`（次に呼ぶべきツール名と引数のヒント）が含まれており、ホストAIが迷わず次の一手に進めるようにしている。

各ツールには次の4種の `annotations` も付与し、ツールの性質をホスト/クライアント側に事前に伝えている。

* `readOnlyHint`: 状態を変更しない読み取り専用ツールかどうか
* `destructiveHint`: 破壊的な操作を伴うかどうか
* `idempotentHint`: 同じ引数で複数回呼んでも結果が変わらないかどうか
* `openWorldHint`: 外部の未知のシステム（インターネット上のフィードなど）とやり取りするかどうか

`prepare_scoring` の場合は `readOnlyHint: false`（重複マーキングでDBを更新するため）、`idempotentHint: false` が設定されている。こうした宣言は、信頼済みサーバーの中では「このツールを呼んでよいか」「再試行してよいか」を判断する材料になる。ただしMCP仕様上、annotationsはあくまでヒントであり、信頼境界や権限確認の代替にはならない。信頼できないサーバー由来のannotationsをそのまま信用すべきでもない。

エラーも同様に構造化した。

src/responses.ts (MCP版)

```
export function toolError(payload: ToolErrorPayload): ToolResponse {
  const tail = payload.suggestion ? `\n→ ${payload.suggestion}` : "";
  return {
    content: [{ type: "text", text: `❌ ${payload.message}${tail}` }],
    structuredContent: payload,
    isError: true,
  };
}
```

`error_code` / `message` / `suggestion` / `related_tools` / `retry_allowed` を持つ構造化エラーであり、`isError: true` を立てることでプロセスは終了しない。ホストAIが `suggestion` を読んで自己修正し、そのまま会話の続きでリトライできる。想定外の例外は `handler.ts` の `runTool()` が一括で捕捉し、`internal_error` として同じ構造に変換する。

CLI版の「エラーメッセージは人間が読んで対処する」から、MCP版の「エラーはAIが読んで自己修正する」への転換――これも、AIの居場所が呼ばれる側から呼ぶ側に変わったことの直接の帰結だ。

なお、MCPには `resources` や `prompts` といった機能もSDKに用意されているが、feed-curator-mcp ではどちらも使っていない。すべて `tools` として実装している。instructionの組み立てを担う `prompts.ts` というファイル名はあるが、これはMCPの `prompts` 機能とは無関係で、あくまでサーバー内部のヘルパーモジュールの名前である。

## 5. 信頼境界の変化——CLIには不要だったセキュリティ層

CLI版は「自分のPCで自分がコマンドを打つ」道具だった。フィードURLも出力ディレクトリも、実行する本人が指定する値であり、信頼できる前提で成り立っていた。

MCP版では、これらの引数を渡してくるのはホストAIである。ただし脅威モデルは「AIそのものが悪意を持つ」というより、ユーザー入力、会話履歴、取得したRSS本文、過去のツール結果などに誘導されて、AIが想定外の引数を組み立ててしまう可能性を考える必要がある、というものだ。CLIからMCPへの変換で、機能は同じでも**置かれる環境が変わればセキュリティ要件も変わる**ということだ。

そのため、CLI版には存在しなかったセキュリティ層を新たに追加した。

* `net.ts`（336行）: フィード取得時のSSRF対策。プライベートIP・ループバックIP・リンクローカルIPへのアクセス拒否、DNS rebinding対策、リダイレクト先の再検証を行う
* `paths.ts`: `save_article` の `out_dir` 引数に対するパストラバーサル対策
* 全ツールの入力に対するzodスキーマによる型検証（フィードURLは `http(s)` スキームのみ許可するなど）

src/server.ts (MCP版)

```
/** フィードURLは http(s) のみ許可する（file: などによる SSRF/ローカル読み取りを防ぐ）。 */
const httpUrl = z
  .string()
  .url()
  .refine((u) => /^https?:\/\//i.test(u), {
    message: "フィードURLは http(s) スキームで指定してください。",
  });
```

これはCLI版のコードが甘かったという話ではない。「自分のPCで自分が叩く」という前提そのものが、MCPサーバー化によって崩れた。同じ機能でも、置かれる場所が変われば求められる防御の水準が変わる――そのことを痛感した部分だ。

## 6. 何が残り、何が消え、何が生まれたか——移行の全体対応表

最後に、モジュール単位でCLI版とMCP版を対応させておく。

| CLI版 | MCP版 | 変化 |
| --- | --- | --- |
| `cli.ts`（Commanderエントリ） | `index.ts` + `server.ts` | 書き直し（Commander→MCP SDK + zod） |
| `claude.ts` | （なし） | 消滅（AI呼び出しをホストAIに委譲） |
| `db.ts`（DB操作を一極集中） | `store.ts` / `feeds.ts` / `settings.ts` | 分割（articles/facts は store.ts、feed管理とsettings管理を分離） |
| `score.ts` 内にインラインの9観点定義 | `scoring.ts` | 切り出し（9観点の定義と正規化ロジック） |
| `dedup.ts`（単語分割＋CJK文字bi-gramのハイブリッド） | `dedup.ts`（全言語文字bi-gramに統一） | アルゴリズムごと書き直し |
| `rss.ts`（`parser.parseURL` を直接呼ぶ） | `rss.ts`（`net.ts` の `safeFetchText` 経由で取得） | 書き直し（SSRF対策を通す構造に変更） |
| （なし） | `handler.ts` | 新規（`runTool()` による例外の一括捕捉とエラー変換） |
| （なし） | `responses.ts` | 新規（2層レスポンス・構造化エラーの整形） |
| （なし） | `prompts.ts` | 新規（instructionの組み立て。MCPの prompts 機能とは別物） |
| （なし） | `net.ts` | 新規（SSRF対策） |
| （なし） | `paths.ts` | 新規（パストラバーサル対策） |
| `commands/*.ts`（表示とロジックが一体） | `server.ts`（895行、13ツール登録） | 書き直し（表示とロジックを分離） |

`commander` は依存から消え、`@modelcontextprotocol/sdk` と `zod` が新たに加わった。`better-sqlite3` と `rss-parser` はそのまま残っている。

`db.ts` の分割は、単に見通しをよくするためだけの整理ではない。CLI版はDB操作を1ファイルに集約していても、呼び出し元は `commands/*.ts` という信頼できるコードだけだった。MCP版では `store.ts`（articles/facts）・`feeds.ts`・`settings.ts` に分けたことで、どのツールがどのテーブルに触れるのかがファイル単位で追いやすくなっている。`dedup.ts` の重複判定アルゴリズムも、CLI版の単語分割＋CJK文字bi-gramのハイブリッド方式から、MCP版では全言語文字bi-gramに統一する形で書き直した。

登録はシンプルだ。Claude Codeなら次のコマンドで追加できる。

```
claude mcp add feed-curator -- node /absolute/path/to/feed-curator-mcp/dist/index.js
```

Claude Desktopの場合は `claude_desktop_config.json` の `mcpServers` に `command` / `args` / `env`（`CURATOR_DB_PATH` でDBパスを指定）を記載する形になる。動作確認には MCP Inspector（`npm run inspect`）を使い、各ツールを単体で叩いて挙動を見ることができた。

## まとめ

CLIをMCPサーバーに作り替える作業は、「移植」ではなく「責務の再分配」だった。プログラム側に残ったのは、フィード取得・重複判定・型検証・永続化といった決定的な処理。AI側に移ったのは、採点・事実抽出・記事執筆という推論と、5ステップをどう繋ぐかというオーケストレーションそのものだ。

> CLIはAIを道具として呼んだ。MCPサーバーは自分がAIの道具になる。この視点の反転がCLI→MCP設計の本質だ。

これからCLIツールをMCP化しようとしている人に向けて、筆者がつまずいた3つの壁をチェックリストにしておく。

「コマンドをツールに置き換える」という発想から入ると、この再設計の必要性には気づきにくい。筆者自身、実際に手を動かしてようやくAIの居場所の逆転に気づいた。CLIとMCPサーバーは見た目こそ「コマンドラインで叩くか、AIから呼ばれるか」の違いに見えるが、内実は「誰が考え、誰が手を動かすか」という役割分担そのものの設計問題だった。
