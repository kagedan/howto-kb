## はじめに

「コードを書かずにアプリが作れる」──2025年から急速に広まったこの開発スタイルが、2026年現在、個人開発者の間で完全に主流になっています。

これが\*\*Vibe Coding（バイブコーディング）\*\*です。

2025年2月にOpenAI共同創設者のAndrej Karpathy氏がX（旧Twitter）で命名したこの手法は、自然言語でのやり取りを繰り返しながらアプリを完成させる開発スタイルです。

本記事では、Vibe Codingの概念・ツール選び・実際の進め方・限界まで整理します。

---

## Vibe Codingとは

一言でいうと、\*\*「AIに日本語で指示を出し、AIが書いたコードを動かしながらアプリを作る開発スタイル」\*\*です。

従来の開発フロー：

```
要件定義 → 設計 → コーディング → デバッグ → テスト → デプロイ
（全部自分でコードを書く）
```

Vibe Codingのフロー：

```
やりたいことをAIに伝える
→ AIがコード生成（UI・バックエンド・DB構造を一括生成）
→ 「ここをこう変えて」と対話しながら改善
→ テスト・デプロイ
```

プロトタイプや新規プロジェクトにおいて3〜5倍の効率改善が報告されており、ルーティンタスクでも25〜50%の改善が見込まれています。

---

## 2種類のVibe Codingツール

Vibe Codingツールは大きく2つのグループに分かれます。

### グループ1：AIアプリビルダー

プロジェクトの説明文から完全なアプリを生成するツールです。ホスティングまで管理してくれるので、**コーディング経験がなくても使えます。**

代表的なツール：

* **Lovable**：UIの自動生成が得意。デザインを言葉で指定できる
* **Bolt.new**：フルスタックアプリをブラウザ上で即生成
* **Replit**：ブラウザベースでインストール不要。Agent 4が自律的にコーディング

### グループ2：AIコーディングアシスタント

開発環境（コードエディタ・プラグイン・ターミナルエージェント）に組み込み、開発者がコードを書くスピードを上げるツールです。既存のコードベースを持つプロジェクトや、本番品質を求めるときに使います。

代表的なツール：

* **Claude Code**：ターミナルベースのAIエージェント。複雑なタスクに強い
* **Cursor**：VS Codeベース。エディタ上でAIと対話しながらコードを書ける
* **GitHub Copilot**：コード補完・生成の定番

---

## ツール別の使い分け

2026年の標準的な使い方は「Cursor or CopilotでデイリーのコーディングをしながらClaudeCodeで複雑なタスクをこなす」というハイブリッド運用です。

| シチュエーション | おすすめツール |
| --- | --- |
| ゼロからプロトタイプを素早く作りたい | Lovable / Bolt.new / Replit |
| 既存コードベースに機能追加したい | Cursor / Claude Code |
| 複雑なロジックや長時間タスク | Claude Code |
| コスト重視のデイリーコーディング | GitHub Copilot / Cursor |
| 複数機能を並行して進めたい | Claude Code |

Claude Code、Cursor、GitHub Copilotなどのツールを組み合わせて使うのが実務的で、用途によって使い分けるのが2026年のデファクトです。

## 実際の進め方：Claude Codeを使ったVibe Coding

Claude Codeはすでに使ったことがある方も多いと思いますが、Vibe Codingとして活用する際のポイントを整理します。

### セットアップ

```
npm install -g @anthropic-ai/claude-code
claude
```

### 基本の流れ

**1. やりたいことを具体的に伝える**

```
「Notionのようなシンプルなメモアプリを作りたい。
機能はメモの作成・編集・削除・タグ付けのみ。
技術スタックはNext.js + Supabase。
まず全体構成と必要なファイルを提案して。」
```

**2. 一気に作らせずステップを刻む**

```
「まずDBスキーマとSupabaseの設定ファイルだけ作って。
UIはまだ作らなくていい。」
```

**3. 動かしながら修正を重ねる**

```
「タグのフィルタリングが動かない。
エラーログはこれ：[ログを貼り付け]
原因を特定して修正して。」
```

**4. CLAUDE.mdで指示を固定する**

プロジェクトルートに `CLAUDE.md` を置くと、毎回説明しなくてよくなります。

```
# このプロジェクトのルール

## 技術スタック
- Frontend: Next.js 15 (App Router)
- Backend: Supabase
- Styling: Tailwind CSS

## コーディングルール
- TypeScriptを使う
- コメントは日本語で書く
- エラーハンドリングを必ず入れる

## やってはいけないこと
- .envファイルをコミットしない
- console.logを本番コードに残さない
```

## Vibe Codingの本質：「何を作るか」を言語化する力

Vibe Codingの本質は「ドメイン知識の言語化」であり、AIに何をさせたいかを正確に伝える力が最重要スキルです。

「いい感じにして」ではAIは動けません。「誰が・何のために・どんな操作をするアプリか」を具体的に伝えるほど、AIの出力が使えるものになります。

これはContext Engineeringの考え方とも重なります。AIへの情報設計そのものが、Vibe Codingの腕前を決めます。

---

## 限界と注意点

Vibe Codingは万能ではありません。正直に書きます。

**プロトタイプは作れるが本番品質は別**  
基本的なアプリ構築はプログラミング知識なしでも可能ですが、品質の高いプロダクトを作るにはコードを読んで検証できる力が必要です。AIが生成したコードにバグやセキュリティ上の問題がないかを判断するには、プログラミングの基礎知識があると大きな差が出ます。

**複雑なロジックは苦手**  
Devinは仕様を投げれば30分放置できるが、複雑なロジックでは失敗することが報告されており、Claude Codeでも同様です。アーキテクチャの設計判断は人間が担う必要があります。

**コストがかかる**  
長時間のVibe Codingセッションはトークン消費が多く、APIコストが積み上がります。プロトタイピングフェーズはOllamaのローカルLLMで代替する、スコープを絞るなどのコスト管理が現実的です。

**「動いているように見えるだけ」問題**  
AIが生成したコードは一見動いていても、エッジケースやセキュリティ上の問題を含むことがあります。本番投入前のレビューは必須です。

## 個人開発者におすすめの始め方

```
Step 1: Bolt.new か Lovable で小さなアプリを1つ作ってみる
        （コード不要で「作った感」をつかむ）

Step 2: Claude Code をインストールし、
        既存プロジェクトの改修タスクをやらせてみる

Step 3: CLAUDE.md を書いてプロジェクトのルールを固定する

Step 4: CursorとClaude Codeを用途で使い分けるハイブリッド運用へ

Step 5: 本番品質が必要な部分は自分でコードレビューする習慣をつける
```
