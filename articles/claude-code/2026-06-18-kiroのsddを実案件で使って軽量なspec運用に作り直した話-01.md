## はじめに

この記事では、cc-sddを実案件に入れたあと、日々の開発で重くなったSpec運用をどう軽量化したかを書きます。

* 対象読者: Kiro風のSDDやAIエージェント向けのSpec運用を、実案件に入れようとしている人
* 扱う問題: Requirements、Design、Tasksをすべての変更で維持すると、文書更新の負担が大きくなる
* 持ち帰れるもの: `DIRECT` / `UPDATE_SPEC` / `NEW_SPEC` の分類、`documentation-guide.md`を入口にした文書構成、軽量な`spec.md`の骨格
* TL;DR: すべての変更をSpec化するのではなく、契約として残す必要がある変更だけをSpecに残す

しばらく触っていない機能を直すときは、設計した本人でも、まずどこを変えればいいのかを調べ直すところから始まります。チームで開発していると、機能が増えるほど、この調べる時間が、変更そのものより長くなっていきました。

「この処理はどこにあるのか」「このモジュールを変更すると、どこに影響するのか」。コードをたどれば答えは見つかりますが、毎回そこに時間がかかります。新しいメンバーが入れば、今度は機能の背景や分け方から説明することになります。

最近は、AIエージェントがリポジトリを読み、関連するコードや呼び出し関係をかなり速く探してくれます。以前より調査は楽になりました。

ただ、コードを見つけられることと、設計意図を理解できることは同じではありません。なぜその単位で機能を分けたのか。どこまでを担当し、何を担当しないのか。別の機能がどの契約に依存しているのか。こうした背景は、コードだけから毎回同じように読み取れるとは限りません。

もう一つ、AIエージェントを使った実装を、もう少し安定させたいという狙いもありました。同じリポジトリを読ませても、出てくる実装はセッションや依頼の仕方で少しずつ変わります。目の前の変更だけを見て手を動かすと、既存の責務を越えたり、別の機能が前提にしている契約を見落としたりする。毎回ではありません。でも、起こります。

そこで、次に長く育てていくプロジェクトでは、機能ごとの目的、責務、依存関係を早い段階から残しておきたいと考えていました。実装の細部を文書へ書き写すのではなく、後からコードを読む人とAIエージェントが、最初に知りたいことを残すイメージです。

ちょうど新しいプロジェクトをPoCから始めることになり、最初からこの考え方を取り入れました。導入したのが、Kiroの仕様駆動開発（SDD）をClaude Codeなどで利用できるようにした [gotalab/cc-sdd](https://github.com/gotalab/cc-sdd) です。

導入当初は、Requirements、Design、Tasksを順番に整理してから実装に入る流れに、かなり手応えがありました。ところが、機能を追加し、日々の修正にも使うようになると、少しずつ違和感が出てきました。

## ハマった点: SDDをそのまま日常運用に持ち込むと重い

最初の大きな機能では、この進め方がうまく機能しました。実装前に要求を整理し、設計を決め、作業を分割しておくと、AIエージェントが途中で目的を見失いにくくなります。特に、複数のモジュールにまたがる機能では、最初に境界を決めておく効果を感じました。

一方、この流れをすべての変更にそのまま続けていると、今度は維持する側の負担が気になり始めました。引っかかったのは、大きく三つです。

### 1. 小さな変更にも同じ工程が発生する

表示ラベルの調整や、仕様どおりに動いていない箇所の修正でも、Requirements、Design、Tasksの更新を検討することになります。

変更そのものより、どの文書を直すべきか考える時間の方が長くなることもありました。このあたりから、文書を整えるために開発しているような感覚が出てきました。

もちろん、すべての変更でSpecが不要なわけではありません。ただ、変更の大きさやリスクに関係なく、同じ工程を適用する必要もないはずです。

### 2. 同じ内容が複数の文書に現れる

Requirementsに書いた内容をDesignで説明し直し、さらにTasksへ実装手順として書き直すと、似た情報が複数の場所に残ります。

実装中に判断が変わった場合は、コードだけでなく複数の文書を更新しなければなりません。どれか一つを直し忘れると、AIエージェントは古い説明と新しいコードを同時に読むことになります。

文書の量よりも、「同じ事実を複数の場所が所有していること」が問題でした。

### 3. 完了済みのTasksが現在の仕様に見える

Tasksは実装中には便利ですが、完了後も更新を続けると、現在の仕様と過去の作業記録が混ざります。

後から参加したメンバーやAIエージェントは、完了済みのチェックリストを現在の要件として読むべきなのか、単なる履歴として読むべきなのか判断しにくくなります。

そこで、Tasksは一回の実装サイクルだけに使い、完了後は凍結することにしました。現在の仕様はSpec、正確な実装はコードとテスト、変更履歴はGitとPRに任せます。

## 捨てたのはSpecではなく、すべての変更を同じように扱うこと

ここまでの三つは、別々の不満に見えて、使っているうちに同じ一点へ集まっていきました。

軽量化を考え始めたとき、最初に手が伸びたのは文書を減らすことでした。でも、ファイルの数を削るだけでは、必要だった判断まで一緒に捨てかねません。減らせばいい、という話ではなかったのです。

見直すべきだったのはSpecそのものではなく、すべての変更を同じ重さで扱っていたことでした。

そこで、変更を次の三つに分類することにしました。

### DIRECT

既存の契約を変えない変更は、Specを更新せずに直接実装します。

明確なバグ修正、小さなUI調整、テストの追加、内部リファクタリングなどが該当します。実装と検証は必要ですが、将来参照する新しい契約は生まれません。

### UPDATE\_SPEC

利用者や下流の処理から見える振る舞いが変わり、既存の領域がその変更を所有できる場合は、担当するSpecだけを更新します。

上流の契約が変わった場合は、それを利用する機能も再検証します。ただし、関連するすべてのSpecへ同じ説明をコピーするのではなく、定義する場所は一つに絞ります。

### NEW\_SPEC

独立した目的、ライフサイクル、データ、外部連携などを持つ新しい能力には、新しいSpecを作ります。

この分類によって、「Kiroを使うか、使わないか」という二択ではなく、変更のリスクに合わせて残す情報量を変えられるようになりました。

実際には、次のような判断をしています。

| 分類 | 例 | Specの扱い |
| --- | --- | --- |
| DIRECT | 表示ラベルの修正、テスト追加、既存契約を変えない内部リファクタリング | 更新しない |
| UPDATE\_SPEC | 既存の出力項目を増やす、下流が参照する判定ルールを変える | その振る舞いを所有するSpecだけを更新する |
| NEW\_SPEC | 評価ツール、外部連携、独立したレビュー画面などを新しく追加する | 新しいSpecを作り、依存関係と再検証条件を書く |

この分類は、プロジェクトの文書ルールをまとめた`documentation-guide.md`に置いています。個別のSpecに散らすのではなく、文書をどう使うかの入口に置くことで、人もAIエージェントも同じ判断基準を参照できるようにしています。

判断の流れだけを抜き出すと、次のようになります。

## 文書ごとの責務を決める

次に見直したのは、各文書が何を所有するかです。

RequirementsとDesignを常に別ファイルにすることはやめ、長期間参照する振る舞い、所有権、契約、重要な判断を、一つの`spec.md`にまとめました。

ただし、`spec.md`だけを見直したわけではありません。プロジェクト全体では、文書の入口として`docs/documentation-guide.md`を置き、README、docs、`.kiro/`の役割をそこで管理するようにしました。

```
docs/documentation-guide.md
├── README.md の役割を定義する
├── docs/ の役割を定義する
└── .kiro/ の役割を定義する
```

`documentation-guide.md`は、個別機能の仕様を書く文書ではありません。どの情報をどこに置くか、どの変更でどの文書を更新するかを決めるための入口です。

| 文書 | 記録する内容 | 記録しない内容 |
| --- | --- | --- |
| documentation-guide | 文書全体の使い分け、変更レベル、更新ルール | 個別機能の仕様 |
| Spec | 振る舞い、責務、境界、依存関係、重要な判断 | 詳細な実装手順や作業履歴 |
| Tasks | 今回の実装範囲、完了条件、検証方法 | Specの説明の繰り返し |
| README | セットアップと利用方法 | 設計の経緯や内部契約 |
| docs | 人が読む運用ガイドや補足資料 | コードと同期が必要な詳細定義 |
| ADR | 複数領域に影響し、後から変更しにくい判断 | 日常的な小さな変更 |

変更されやすく、コードを見れば確認できる情報まで文書へ複製しません。

Specに残すのは、コードだけでは読み取りにくい目的や境界です。

この全体ルールのうち、AIエージェントが主に参照する領域が`.kiro/`です。現在の`.kiro/`は、実際には次のような規模です。以下は実プロジェクトの構成をもとに、領域名や作業名を置き換えた例です。

```
.kiro/
├── steering/
│   ├── product.md
│   ├── architecture.md
│   └── development.md
├── specs/
│   ├── core-domain/
│   │   ├── spec.md
│   │   └── tasks-YYYYMMDD-feature-change.md
│   ├── review-ui/
│   │   └── spec.md
│   └── evaluation-tooling/
│       └── spec.md
├── decisions/
│   └── README.md
└── templates/
    ├── spec.md
    ├── tasks.md
    └── adr.md
```

`steering/`はプロジェクト全体の前提、`specs/`は機能ごとの契約、`tasks-*.md`は一回の実装サイクルの作業メモとして分けています。

`spec.md`の骨格は、だいたい次の形にしています。

```
---
status: active
owner: <domain-or-team>
depends_on: []
consumers: []
---

# <Feature>

## Purpose

この領域が所有する問題と、利用者やシステムへ提供する結果を書く。

## Current Capabilities

- 現在実装されている主要な振る舞いを書く。

## Scope

### Owns

- このSpecが責任を持つ振る舞い、データ、契約を書く。

### Does Not Own

- 似ているが、このSpecでは扱わない責務を書く。

## Contracts

下流が依存する安定した契約を書く。
正確なフィールド定義は、必要に応じてコード側を参照する。

## Decisions

あとから読んだときに必要な、非自明な判断と理由だけを書く。

## Dependencies

- `<upstream-spec>`: 依存する契約と、その理由を書く。

## Revalidation Triggers

- どの変更が起きたら、このSpecや下流を再検証するかを書く。

## Verification

- 自動テストのコマンド
- 必要な手動確認や実行時確認
```

ポイントは、Specを「実装手順」ではなく「現在有効な契約」として読むことです。実装の細部をここに写しすぎると、コードとの差分を維持する作業が増えてしまいます。

## Specを英語に変えた

運用中に、もう一つ気になったのが文書の言語でした。

導入当初は、cc-sddの日本語設定を使い、Specも日本語で生成していました。人が読むことを考えると、日本語で統一する方が自然に思えたからです。

しかし、生成と修正を繰り返すうちに、文書の中で日本語と英語が不規則に混ざるようになりました。技術用語やコード上の識別子は英語のまま残り、その周辺だけ翻訳調の日本語になります。モデルによって表現も変わり、同じ概念に異なる言葉が使われることもありました。

実際に、`deploy`の意味で「部署対象」という表現が生成され、そのままプロジェクト構造の説明に残っていたことがあります。意味を推測できないわけではありませんが、人が読む文書としても、AIエージェントが再利用する固定語彙としても不安定でした。

ここで、Specは誰のための文書なのかを考え直しました。

私たちのプロジェクトでSpecを最も頻繁に読むのは、実装やレビューを行うAIエージェントです。一方、READMEや運用ガイドを読むのは、プロジェクトを利用する人です。両方を同じ言語にそろえる必要はありませんでした。

現在は、次のように分けています。

* `.kiro/`配下のSpec、Steering、Tasks、ADRは英語で書く
* `README.md`と`docs/`配下の人向け文書は自然な日本語で書く
* コード上の識別子、ファイル名、コメントは英語で書く

英語の方が常に優れていると考えているわけではありません。日本語で安定したSpecを生成し、チームで維持できるなら、それも選択肢だと思います。

今回の判断は、文書の主な読み手と目的に合わせて言語を分けた結果です。AIエージェント向けの文書では固定語彙と構造の安定性を優先し、人向けの文書では自然さと読みやすさを優先しています。

## 現在の運用ルール

重さに気づいてから、変更の分類、文書ごとの責務、言語の使い分けまで、ひとつずつ決めてきました。それを今の形にまとめると、運用ルールは次のようになります。

> Specは作業手順ではなく、あとから参照する契約を置く場所にする。

| 観点 | ルール |
| --- | --- |
| 変更の分類 | 契約を変えない変更は`DIRECT`、既存契約を変える変更は`UPDATE_SPEC`、独立した能力は`NEW_SPEC`として扱う |
| 所有権 | 一つの振る舞いや契約を所有するSpecは一つだけにする |
| ライフサイクル | Specは現在有効な契約を記録し、Tasksは一回の実装サイクルだけに使う |
| 情報の正 | 正確で変更されやすい情報はコードとテストを正とする |
| 言語 | AIエージェント向けの文書は英語、人向けの文書は日本語で書く |

これはKiroやcc-sddの一般的な正解ではありません。現在のチーム規模、開発速度、AIエージェントの使い方に合わせて、cc-sddの考え方を日常運用へ寄せた形です。

## シンプルにして残ったもの

最終的に、日常運用で常に意識するものは次の三つになりました。

1. 変更の分類
2. 機能ごとの`spec.md`
3. 一回の実装サイクルで使う`tasks-*.md`

Requirements、Design、Tasksを毎回そろえるのではなく、変更の扱いを決める入口、現在の契約を持つSpec、今回だけ使うTasksに分けています。この三つに絞ると、AIエージェントが変更の扱いを判断しやすくなります。

依頼のたびに人間が必ず`DIRECT`、`UPDATE_SPEC`、`NEW_SPEC`を指定するわけではありません。むしろ、文書の入口である`documentation-guide.md`にこの分類を置いておくことで、AIエージェント側が「今回はSpecを更新しない変更なのか」「既存Specを更新する変更なのか」「新しいSpecとして分ける変更なのか」を判断しやすくなります。

毎回すべての文書を作るのではなく、変更の種類に合わせて、AIエージェントが読む文書と更新する文書を選べるようにする。私たちの運用では、ここが一番効いています。

## おわりに

cc-sddを導入したことで、実装前に要求と設計を整理する習慣ができました。一方、すべての変更に同じ量の文書を求めると、その維持自体が目的になりやすいことも分かりました。

私たちが軽量化で目指したのは、Specを減らすことではありません。

変更のリスクに見合った量だけ、将来も参照する価値のある情報を残すことです。

効くと感じているのは文書の数ではありません。その文書を最後に読むのは誰なのか。どの情報を、その文書が責任を持って所有するのか。そして、いつまで更新し続ける文書なのか。気にしているのは、結局このあたりです。

こうして並べるときれいに見えますが、どれも最初から分かっていたわけではなく、運用しながら決め直してきたものです。

この三つさえ決まっていれば、テンプレートはプロジェクトに合わせて変えてよいと思います。逆に、ここが曖昧なまま文書だけを増やしても、人やAIエージェントが参照できる安定した情報にはなりません。

cc-sddを使って、Requirements、Design、Tasksを整理する価値を知りました。そのうえで、実案件の日々の変更に合わせて、必要な部分だけを残しました。

今の形は、cc-sddのよかった部分を残しつつ、日常の変更に耐えるくらいまで軽くした運用です。
