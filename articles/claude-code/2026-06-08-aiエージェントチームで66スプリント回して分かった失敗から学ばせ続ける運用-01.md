## はじめに

私は**Claude Code のエージェントを SM・DEV・PO・レビュアーとして構成し、スクラムを回しながら個人開発のプロダクトを作っています**。実施したスプリント数は、執筆時点で **66** です。

この話をすると、だいたい「バイブコーディング」というワードが相手から出てくるのですが、やっている本人としては「ちょっと違うんだよな」という感覚がずっとありました。`Planning始めて` と言えば、Planning から Sprint Review までエージェントが一気通貫で進めてくれる。たしかに自律的ではあるけれど、コードを"雰囲気"で受け入れているわけではありません。むしろ逆で、**レビューも規約もプロセスもしっかり用意して、その上で自律させています**。

この記事は、その違和感の正体を言語化したものです。中心にあるのは\*\*「一発で100点を取るのではなく、失敗をその都度チームの進め方に反映して賢くしていく」という改善ループの考え方\*\*です。

## こんな「AIによる開発のモヤモヤ」を抱える人へ

AI を使って開発していて、こんな経験はありませんか。

* **「やるな」と言ったことを、AI が平気でやる**。たとえば「区分値をマジックストリングで直書きするな（生成済みの enum を使え）」と規約に明記してあるのに、AI は平然と `'0'` / `'1'` を直書きしてくる。
* **AI にレビューさせているのに、毎回同じ指摘が発生する**。レビュアーはちゃんと検知して差し戻すし、直る。レビュー自体は機能している。でも次のスプリントでまた同じことをやる。それも何スプリントも。「いや、学習してくれよ……」と何度思ったことか。
* **一発で完璧な指示を出そうとして、消耗する**。

この記事は、そういう「AI に同じ失敗を繰り返される」モヤモヤを感じたことがある方に、参考にしていただきたく書きました。

## 何を作っているか

題材は **HwHub** という、家事を家庭内で共有・管理する自作アプリです。あくまで趣味で開発しているもので、リリース予定はありません（なぜ作っているのかは、末尾の「おまけ」で触れます）。

機能の詳細は本筋ではないので省略しますが、構成だけ共有しておくと **7つのリポジトリ**にまたがっています。詳しくは各リポジトリの README.md をご参照ください。

ポイントは、フロント・バック・モバイル・バッチ・インフラが一通り揃っていること。Acceptance Criteria を満たし、レビューを通し、PR を出してマージする、という開発の現場で当たり前に行われているサイクルを AI チームが回している、という点だけ押さえてもらえれば十分です。

ちなみにこの記事は「Java ならこのディレクトリ構成」のような**スタック固有のベストプラクティス集ではありません**。スタックが Vue / Spring Boot / Flutter / Terraform とバラバラでも、同じチーム・同じループで回せている。これは「何を作るか」ではなく「どう改善し続けるか」に着目しているからです。スタックが変わっても応用が効きます。

## AIチームにおける「スクラム」の定義

**スクラム**というワードを使ってはいますが、教科書どおりではありません。スクラムに興味を持って読んでくださる方に先に断っておくと、セレモニーをすべて行っているわけではありませんし、参加者や実施内容が異なったり、期間が固定されていなかったり、スクラムマスターがタスクを回しすぎていたりします。

それでも私がこれを「スクラム」と呼んでいるのは、スクラムを支える\*\*3つの柱（Pillars）\*\*が機能しているからです。

* **Transparency（透明性）**：現在の正しい情報が一箇所に整理されており、次の行動が誘発されている状態
* **Inspection（検査）**：目標・目的が明確で、それに対する現状が明確で、その差分が把握され続けている状態
* **Adaptation（適応）**：目標・目的が明確で、現状が明確で、その差分を埋め続けている状態

特に **Adaptation ＝改善ループ**こそがこの取り組みの心臓部です。

## チームの全体像とアーキテクチャ

### 全体像

エージェントチームの基盤には、Claude Code の **Agent Teams（experimental）** を採用しています。  
![AIエージェントチームの全体像](https://static.zenn.studio/user-upload/7374713d6116-20260608.png)

**ユーザ(私)が起点**になって PBI(プロダクトバックログアイテム)を起票し、Planning を指示する。**Agent Teams** の中では SM がリードし、DEV・Reviewerが動く。コードは **Git リポジトリ**へ流れ、進捗ややりとりは **Discord** に残り、バックログは **GitHub Projects** で管理されます。

なお、**GitHub Projects** と **Discord** の具体例は、末尾の「おまけ」をご参照ください。

### ロールの紹介

チームの中身はこの6ロール（全員 Claude Code のエージェント）です。これらは **Agent Teams** で動かしていますが、同じような運用は **subagent** でも可能です（subagent との違いや Agent Teams を選んだ理由は、末尾の「おまけ」をご参照ください）。

| ロール | 役割 | モデル |
| --- | --- | --- |
| **SM**（Scrum Master） | チームリード。イベント進行・調整・レビュー集約・PR 作成・Retro | Opus |
| **DEV**（Developer） | 実装担当。TDD で AC を満たす | 計画=Opus / 実装=Sonnet |
| **PO**（Product Owner） | バックログ管理・Issue 起票・AC の詳細化 | Sonnet |
| **規約レビュアー** | コーディング規約の観点でレビュー | Haiku |
| **セキュリティレビュアー** | セキュリティ観点でレビュー | Sonnet |
| **パフォーマンスレビュアー** | パフォーマンス観点でレビュー | Haiku |

なお、ユーザ(私)がやることは **バックログを整える・実装方針を承認する・Sprint Review で動作確認する** の3つくらい。残りはチームが進めます。

### ディレクトリ構成

Agent Teams を動かしているディレクトリの構成です（[scrum-agent-base](https://github.com/ryokkon624/scrum-agent-base)）。

```
scrum-agent-base/
├── CLAUDE.md                  # 起動時に毎回読む全体設定
├── .claude/
│   ├── agents/                # 各ロールの定義（使えるツール・モデル）
│   │   ├── scrum-master.md
│   │   ├── developer.md
│   │   ├── product-owner.md
│   │   └── *-reviewer.md       # 規約 / セキュリティ / パフォーマンス
│   ├── skills/                # ロール別の手順・規約・チェックリスト
│   │   ├── scrum-master-workflow/
│   │   ├── developer-workflow/
│   │   └── {frontend,backend,mobile}-conventions/ …
│   ├── rules/                 # トピック別ルール（git・DB など）
│   └── hooks/                 # コミット前チェックなどの自動処理
├── memory/                    # チームの記憶
│   ├── sm/{short_term,long_term}.md
│   ├── dev/{short_term,long_term}.md
│   └── po/{short_term,long_term}.md
└── backlog/
    └── sprint_XX/
        ├── sprint_backlog.md        # SM が Planning で作成
        ├── implementation-notes.md  # DEV の仕様外メモ
        └── review-#N.html           # Sprint Review 用 HTML
```

各種ファイルは、何をどこに書くかを運用しながら整理しました。

| 置き場所 | 役割 | 判断ポイント |
| --- | --- | --- |
| `CLAUDE.md` | チーム全体に毎回必要な短い事実・ルール | 200行以内。長い手順は書かない |
| `rules/` | トピック別・パス別の条件付きルール | パス条件があるか、トピックを分離したいか |
| `skills/` | ロール固有の手順・参照ガイド | 手順書・チェックリスト・レビュー観点 |
| `agents/` | エージェントのツール・モデル定義 | 独立コンテキストで動く専門ロール |
| `hooks/` | Claudeの判断に依存せず確実に実行したい処理 | 何かをする時に必ず実行するか |
| `memory/` | 各ロールの記憶 | 今Sprint限り→short\_term、永続→long\_term |

`agents/` と `skills/` は、どちらも **Frontmatter ＋ Markdown 本文**という同じ形式です。**冒頭の `---` で囲った部分が Frontmatter ＝ Claude Code が読み込むメタデータ宣言**で、`name`・`tools`・`skills`・`model` などを書きます。違うのは Frontmatter のフィールド。たとえば DEV のAgent定義（`agents/developer.md`）はこうなっています。

```
---
name: developer
description: スプリントバックログのアイテムを実装し、ACを満たす。SMからの指示で起動する。
tools: Read, Glob, Grep, Edit, Write, Bash, TaskCreate, TaskUpdate, SendMessage, ...
skills: [developer-workflow, backend-conventions, ...]
---
あなたは Developer です。起動したらすぐに `memory/dev/short_term.md` と `memory/dev/long_term.md` を読んで行動してください。
```

`tools` で使えるツールを絞り、`skills` で参照する手順書を紐づけ、本文で役割と起動時の動作を書きます。

**モデルの使い分けも、ここでやっています。** たとえばレビュアーのAgent定義（`agents/convention-reviewer.md`）は `model:` をHaikuで固定しています。

```
---
name: convention-reviewer
description: コーディング規約観点のレビュアー。DEVの実装完了後にSMから起動される。
model: claude-haiku-4-5-20251001
tools: Read, Glob, Grep, ...
---
```

一方、DEV の定義には `model:` がありません。**DEV は SMが spawn するときにモデルを指定する**ことにしており、計画フェーズは Opus、実装フェーズは Sonnet、と起動のたびに切り替えています。「固定でいいロールは定義に書いておく、フェーズで変えたいロールは起動時に指定する」という使い分けにしています。

#### memory ── 短期記憶と長期記憶 × 各ロール

memory は SM / DEV / PO のロールごとに、`short_term.md`（今Sprintの作業メモ）と `long_term.md`（過去Sprintの教訓）に分けています。

`short_term.md`があると、たとえば **Sprint の途中でトークンのリミットに達して止まっても、作業状態が書いてあるため、仮に別セッションを開始しても続きから再開できます**。AI の記憶は揮発しますが、ファイルに書いておけば引き継がれます。

`long_term.md` に何を書くかは、ロールごとに少し違います（更新はおもに Retro）。

* **DEV の `long_term.md`**：繰り返し受けた指摘パターン／技術的なハマりどころ／習得した設計パターン
* **SM の `long_term.md`**：スプリント進行の判断パターン／DEV のレビュー指摘傾向／Skills の更新履歴  
  いずれも "次の自分が同じところで迷わない・つまずかないための手掛かり" を残す、という方針です。

このmemoryの読み書きは自動では行われません。**Agent 定義に「起動したら memory を読め」、スキルに「いつ何を更新するか」を書いて**、エージェントに実行させています。たとえば実装方針は計画フェーズで `short_term.md` に書く、経験をもとに得た教訓は Retro で `long_term.md` に書くなどを スキルで指示しています。Claude Code 標準のメモリ機能ではなく、ファイルベースの自作の仕組みです。標準のメモリ機能も Claude が自律的に覚えてくれますが、ここでは "いつ・何を・どのロールが書くか" まで自分の手でコントロールしたかったこともあり、memoryを自作しています。

### 作業フロー

これらの作業は、`scrum-master-workflow` や `developer-workflow` などのスキル、`*-reviewer.md` などの Agent 定義で設計しています。**入口は専用コマンドではなく、自然言語**です。`scrum-agent-base` で `claude` を起動して「SMモードで動いて。Sprint XX の Planning を開始して」と打ちこむ。`CLAUDE.md` に「『SMモードで動いて』と言われたら `agents/scrum-master.md` を参照」というマッピングが書いてあり、それに応じて読み込む Agent 定義が切り替わります。Sprintは、エージェント同士が SendMessage でやりとりしながら、次のように進みます。

各ステップはこのようなことをしています。  
最後のRetrospectiveでAIエージェントが自ら振り返りをし、スキル、memory、rulesなどを更新していく仕組みにしています。

| # | フェーズ | 誰が | 作業 | 作業内容 | 主なOUTPUT | 出力先 |
| --- | --- | --- | --- | --- | --- | --- |

| 2 | Sprint Planning | SM | Sprint Backlog作成 | GitHub Projects から対象Issueを特定し、ゴール・リスク・チャレンジを整理して Sprint Backlog を作成 | スプリントバックログ / Planning報告 | `backlog/sprint_XX/sprint_backlog.md` / Discord #10-planning |

## 品質をどう担保するか

自律させるほど「品質をどう守るか」が問われます。私はこの3点の工夫をしています。

### TDD ＋ 3観点レビュー ＋ memory

* **TDD**：DEV は RED→GREEN→REFACTOR を必須にしています。このスタイルは、[superpowers](https://github.com/obra/superpowers)（Jesse Vincent 氏作）の TDD スキルから着想を得ています。
* **3観点レビュー**：規約・セキュリティ・パフォーマンスのレビュアーが、DEV の実装を並列でレビューします。指摘があれば DEV に差し戻し（ラリー）、全員「指摘なし」になるまで Sprint Review に進めません。
* **memory**：過去の指摘傾向を long\_term に溜め、次の実装時に「前回までこういう指摘が多かった」と認識させる。

TDD には**副次的な効果**もあります。テストを先に書くので、テストコードがおのずと積み上がっていきます。今後 [Claude Mythos](https://red.anthropic.com/2026/mythos-preview/)のようなものが広く使えるようになれば、脆弱性の発見も、その修正のスピードも格段に上がるはずです。そのとき、直す対象にテストコードが一つも無かったら……考えたくないですよね。テストの蓄積は、そんな未来への備えにもなります。

### Sprint Review は HTML で

最初は Sprint Review も Markdown とPRの突合せで対応していたのですが、**Sprintが進むほど人間（私）の確認がボトルネックになりました**。エージェントの手は速いのですが、私のレビューに時間がかかる。

そこで、**AC とコードの変更内容を、人が見やすい HTML にまとめて出力する**ようにしました。レビューポイントを絞って提示させているので、私は「ここだけ見ればいい」状態で確認できます。feature/refactor と bug でHTMLのテンプレートを分けています。

この「レビューを HTML で出力する」発想は、ちょうど Markdown 確認がつらくなっていた頃に見かけた、Anthropic の Thariq Shihipar 氏の記事「[The Unreasonable Effectiveness of HTML in Claude Code](https://x.com/trq212/status/2052809885763747935)」から取り入れたものです。

### スコープクリープ対策 ── 防ぐより"残させる"

エージェントは、AC に書いていない"気を利かせた実装"を勝手に足すことがあります。いわゆるスコープクリープ。

「仕様にないことはするな」と指示しても、気を利かせて足してしまう。完全には抑えきれません。そこで、**禁じるのではなく"残させる"** ことにしました。DEV は実装中、**「ACに書かれていなかった判断・変更・妥協点」をそのつど `implementation-notes.md` に書き出します**。たとえば「ACにnullの記述がなかったので空リストを返すことにした」のように。

この「仕様外の判断を `implementation-notes` に残す」手法は、私のオリジナルではありません。Anthropic 社員が使っているプロンプトとして X で共有されていたもの（[@ClaudeCode\_love の投稿](https://x.com/ClaudeCode_love/status/2056625826091975050)で知りました）を取り入れています。原文のプロンプトはこちらです：

> implement `<SPEC>` and while you do keep a running `implementation-notes.html` file (or markdown) with decisions you had to make weren't in the spec, things you had to change, tradeoffs you had to make or anything else I should know

このメモは Sprint Review の HTML に取り込まれ、私が「勝手にこう判断したのね」と確認できるようにしています。**防ぎきれない分を、可視化して検知に回す**という発想です。

## 失敗からどう学ばせるか

ここがこの記事の山場、改善ループの設計です。

### 大前提：一発で100点は取れない

繰り返しになりますが、一発で AI に完璧な指示を出すことはできません（できるかもしれませんが、完璧なプロンプト・ハーネスを作る労力は相当大きいと思います）。だから重要なのは、**失敗したときに「なぜ自分の指示が誤って伝わったのか」を観察し、設定に反映すること**です。Sprintを回すこと自体より、このループを回し続けることが大事だと考えています。

### 「してはいけない」は通じない → 予防と検知の二段構え

スコープクリープが「書いていないことを勝手にやる」問題なら、こちらは逆に「**規約に明記してあるのに破る**」問題です。「マジックストリングを使うな」「例外を握りつぶすな」とコーディング規約に書いても、エージェントは破るときは破ります。**禁止事項をいくら書いても守りきれない**ので、「予防で減らし、検知で捕まえる」、この二段構えに落ち着きました。

* **予防**
  + skill に「やってはいけないこと」を明記する … **これだけではあまり効きません**
  + レビュー指摘を memory に記憶させ、作業時に前回までの指摘傾向を認識させる
  + レビュー依頼前のチェックリストを用意する（これはエージェント自身が作ってくれました）
* **検知**
  + レビュアーの規約レビューで、予防からすり抜けたものを捕まえる

具体例で、どう"二段構え"に行き着いたかを紹介します。冒頭で挙げたマジックストリングと同じで、たとえばモバイルのテストで「日本語テキストを直接検証する」（`find.text('パスワード変更')` のような書き方。ARB を変えると壊れるので禁止）という指摘も、なかなか止まりませんでした。

1. **規約に書く**（mobile-conventions）→ それでも破られる
2. **レビュアーが検知**して差し戻す → 品質は守られるが、**Sprint 50・53 と同じ指摘が再発し続ける**。
3. **long\_term に「繰り返し指摘」として記録** → 傾向は意識されるが、完全には止まらない
4. **Sprint 59、レビュー依頼前の"チェックリスト"に昇格**（規約にあっても見落とすので、提出前に自分で潰す項目にした）→ **これが効いて、以降は再発しなくなった**

※ただし、4の対策を導入してからまだ十分なSprintを回せていないため、たまたまの可能性はあります。

これまで人がやってきた開発の現場でも、レビュー依頼前のチェックリストとか作ってましたよね。Agent も人と同じで、逆に言えば、人がやってきたことは Agent にも有効、ということだと思います。

別エージェントにレビューさせること自体はかなり有効です。最終的な品質はこれだけでも担保されると思います。しかし、毎回同じ指摘が発生するのはトークン使用効率が悪いため、**レビューが品質の安全網、memory やチェックリストが再発の頻度を下げていく** ── この合わせ技としました。

### 適したファイルに書く

設定ファイルは運用とともにどんどん育ちます。ところが育つと今度は **「ルールはあるのに守られない事件」** が起きます。原因のひとつが「書く場所が適切でない」こと。毎回必要な短い事実が長い手順書に埋もれていたり、ロール固有の手順が全体設定に混ざっていたり。前述の「どこに書くか」の判断テーブルは、この事故を経て整理し直したものです。

特に気にせず CLAUDE.md に全部書いても、それっぽく動いてしまうのが怖いところです。でも、規律やトークン使用量を考え始めると、適材適所に書くに越したことはありません。

memory 自体も、同じように"育てて整理"が要ります。最初は long\_term にSprintごとの出来事を逐一書いていたところ、ごみで膨らみました。そこで Sprint 35 のとき、**溜まった memory を一度ぜんぶ分析して**、「long\_term には"次の自分が判断に迷わない手掛かり"（判断パターン・レビュー傾向・更新履歴）だけを残す」とルールを決め直し、**214行 → 86行に圧縮**しました。

こうして一つずつ潰してきた結果、いまでは **全レビュー・Sprint Review ともに指摘ゼロで終わるSprint**も出るようになりました。

## やってみて学んだこと・有効だったこと

仕組みの話とは別に、運用してみて「これは有効だ」と実感したことを書いておきます。

### 自作の memory

Claude Code には組み込みの記憶機能もありますが、私はこのチーム用に \*\*自前の memory（ロール別の short\_term / long\_term）\*\*を作りました。これがよく効きます。

* **品質が底上げされる**：TDD ＋ レビュー ＋ memory の3点セットで、過去の指摘が次に活きる。
* **セッションを跨いで再開できる**：トークンのリミットで Sprint 中に止まっても、short\_term に状態が残っているので別セッションで続きから再開できる。
* **"裏でカバーしている問題"に気づける**：これが一番大きい学びでした。

最後の点を補足します。たまに memory を覗いてみると、**エージェントが問題を裏で回避したまま、ゴールだけ達成している**ことがあります。代案で帳尻を合わせてしまうので、表面上は成功して見えますが、でもそれは根本原因が残ったまま。見つけたら根本解決してあげるべきサインです。実例を2つ。

* **レビュアーが Discord に投稿できず、SM が代理投稿していた**。一見「投稿はできている」。でも根本原因は、agent 定義の **Tools 記載漏れ**でした（Sprint 57 で `discord_send` を追加して解決）。
* **Discord ログインが毎回"失敗"から始まっていた**。原因は、Discord の TOKEN を `settings.local.json` に書いていたこと（本来は `.mcp.json` が正しく、キー名も間違っていた）。そのため自動認識されず、エージェントは「TOKEN は別の場所にあるのでは」と探し回り、見つけた TOKEN でログインを再試行して切り抜けていた。気づくまで、この無駄が続いていました。

memory は、こういう"見えない回避"を暴き出すのにも役立ちます。

### `/insights` で改善する

もうひとつ有効だったのが、Claude Code 標準の **`/insights`** コマンドです。これは**直近のセッション履歴（`~/.claude/` のログ）をローカル解析して、自分の"つまずきパターン"や改善提案を HTML レポートにまとめてくれる**ものです。`CLAUDE.md` への追記案まで出してくれるので、それを設定に取り込みます。  
![Claude Code の  コマンド](https://static.zenn.studio/user-upload/04d1d845f214-20260608.png)  
*`/` を打つと候補に出てくる、標準コマンド（[公式リファレンス](https://code.claude.com/docs/en/commands)）*

たとえば前述の「Discord ログインが毎回失敗から始まる」問題も、こうした振り返りで「毎回同じ無駄が起きている」と気づけました。**自分では気づけない点を、ログを根拠に外から指摘してもらう**。これが改善ループの精度を上げてくれます。

ちなみに `/insights` が見ているのは私の入力だけではありません。**セッションログ全体 ── Claude の応答やツールのエラーまで**解析するので、さっきの「同じところで毎回つまずいている」も拾えたようです。ただし対象は直近の限られたセッションで、teammate の別セッションまでは深掘りしません。チーム内部で繰り返される失敗そのものは、`/insights` より **レビュー＋memory** が主担当 ── という役割分担になっています。

## 課題とこれから

うまく回ってきた一方で、まだ宿題もあります。

* **スキルの肥大化**：失敗を刻み続けると、当然ファイルが膨らみます。「過度な詳細がノイズになる」状態が起き始めています。対策として、SKILL.md は1行要約にして、詳細は `patterns/` 配下に逃がす方向を考えています。
* **昇格はあるが降格・削除がない**：失敗を short\_term → long\_term → Skill と"昇格"させる仕組みはあるのに、**古くなったルールを降格・削除する仕組みがない**。今は足し算だけ。昇格を「2回ルール」にする（1回コケただけなら long\_term 止まり、2回目で初めて Skill へ）、Retro に「棚卸し」を足す、あたりを考えています。具体的には、**Retro でエージェント自身に「直近 N Sprintで一度も指摘に登場しなかったルールを挙げて」と棚卸しさせ、"モデルがもう自力でできる／そもそも起きなくなった"ものを削除候補にする**──という"引き算の自動化"を考えています。
* **DEV の実装方針確認をなくしたい**：Sprint が進むと、DEV が無理やり確認事項をひねり出してくることがある（「ブランチはこれでいいですか？」など）。ここまで安定したら、Planning〜Sprint Review まで完全に一気通貫でいい気もしています。
* **並列 DEV / 複数人同時実行**：**プロダクトが大きくなるほど、DEV 1人ずつでは追いつかなくなります**。実装方針を立てる時点で並列数を決めて、git worktree で並列実装させたい。さらに大規模なプロダクトを見据え、複数の人間で同時に回せるよう、各自が育てた **memory や skills をマージする**ことを考えています。
* **PO の改善ループが回っていない**：PO はあえてDEVに質問されたら「バックログ見て」「いま風にして」くらいしか返さない設計にしています。そして、聞かれた内容をもとに今後聞かれないように新たに起票するPBIの記載を充実させようとしていました。しかし、Sprint 1 では DEV が PO に質問していたのに、以降は質問しなくなったこともあり改善できていません。**DEV からユーザへの確認事項と回答を PO に覚えさせる**ことを考えています。これはやればすぐ改善ループを回せたのにやらなかったのが反省点です。

## おわりに

この記事で一番伝えたかったのは、個別の設定そのものではなく、**「失敗を観察して、設定に反映し、賢くしていく」という改善ループ**の考え方です。

そしてこの考え方は、特定の言語やアーキテクチャに縛られません。あなたのスタックでも、たぶん同じように応用できます。

**まずは1つのリポジトリで、AI が同じミスを2回したら、`CLAUDE.md` に「次からはこうして」と1行だけ書き足してみてください。** `CLAUDE.md` は毎回読み込まれるので、それだけで次から効くようになります。"一発で100点を狙わず、失敗を刻んで賢くする"改善ループの、最小の一歩です。そこから、効いたら増やしていく、増えてきたら、本文で触れたように置き場所を適材適所へ分ければOK。

66スプリントやってきて、まだ宿題は山ほどあります。けれど、**失敗を一つずつ設定に反映するたびにチームが賢くなっていく**感覚は、なかなか面白いものです。

### Agent Teams を選んだ理由（subagent との違い）

チームの実体は、Claude Code の **Agent Teams**（experimental）です。とはいえ、この運用の多くは **subagent** でも実現できます。Agent Teams を選んでいる理由は主にこの2点です。

* **エージェント同士・teammate と直接やりとりできる**：subagent は親に一度結果を返すだけで、エージェント同士は会話しません。Agent Teams なら SM が DEV やレビュアーと往復し、指摘対応のラリーを続けられます。
* **タスクを共有・可視化できる**：teammate が共有のタスクリストを持つので、`claude-task-viewer` で進捗を眺められます。

逆に言えば、1回のレビューでいいなら subagent でも可能です。私は「SM がチームを率いて何度もやりとりする」形にしたかったので Agent Teams にしています。

余談ですが、最初は Agent Teams ですらなく、**Discord だけで完結させようとしていました**。Claudeの`--channels` 機能を利用し、Discord上で「SM が DEV をメンションしてDEV起動」「DEV が SM をメンションして完了報告しSM起動」と、各エージェントが Discord 上で自律的に呼び合う形です。ところが **Discord は bot から bot へメンションできない**仕様であったため諦めました。その代わりに行き着いたのが Agent Teams、という経緯です。なので「どうしても Agent Teams でなければ」という理由は、あまりありません。

結果として、Discord は"通知ログ"としてエージェントが投稿し、人間が読む（必要なら指示も投げる）場所です。一方で、**エージェント同士の会話は Discord を介さず、ローカルの Agent Teams に閉じています**。

Agent Teamsの使い方は`~/.claude/settings.json` の `env` に `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"` を足して有効化し（要 Claude Code v2.1.32 以降）、あとは通常どおり `claude` を起動して、自然言語で「チームを作って」と頼むだけです。このプロジェクトでは SM がリードになり、DEV やレビュアーの teammate を spawn します。teammate の役割には、本文「ディレクトリ構成」で触れた `agents/*.md` の定義を使っています。

### Sprint進捗の可視化（claude-task-viewer）

Agent Teams はタスクを JSON に残すため、[claude-task-viewer](https://github.com/L1AD/claude-task-viewer) という OSS を別ターミナルで動かすと、ブラウザで Kanban としてリアルタイムに眺められます。エージェントが今どのタスクをやっているかが見えると安心感があります。しかし、正直これを見ていたのは最初の数Sprintです。ロボット掃除機に掃除させて、それをずっと見ているような感覚になり、起動することすらなくなりました。

### なぜ、リリースしないアプリにここまでやるのか

なぜリリースもしない趣味アプリに、本格的なインフラと規律あるスクラムを持ち込むのかというと、**目的が"アプリを作ること"ではなく、"AI エージェントでの開発のやり方そのものを作ること"だから**です。AI エージェントで開発するのが当たり前になる時代に向けて、**実際の開発現場でも「作るよね」という機能を一通り備えたアプリ**を題材に、エージェント運用を試したかったためです。

### どこまでを「この取り組み」で書いたか

**7つのリポジトリのすべてを、この記事で紹介したスクラムだけで書いたわけではありません**。少し補足しておきます。

* **mobile 以外の6リポジトリ**：大部分は人と AI が会話しながら書き、そのあとのバグ修正・機能追加をスクラムで回しています。
* **mobile**：基盤部分(デザインシステム、認証、ルーティングなど)を人と AI の会話で書き、機能面をスクラムで実装しています。

ちなみに、mobileについては**人間が直接書いたコミットは、おそらく1つもありません**（コードはすべて AI が書いています）。とはいえ、その全部が "この取り組み" によるものではない ── そこだけは正確にしておきたく、補足します。
