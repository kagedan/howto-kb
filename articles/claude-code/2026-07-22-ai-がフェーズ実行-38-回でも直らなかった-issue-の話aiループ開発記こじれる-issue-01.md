[前回の記事](https://zenn.dev/m2dw/articles/n8n-ai-cli-loop-overview)で、GitHub Issue を書くと AI が実装・レビュー・修正して PR を返してくる自動開発ループを紹介しました。実測値も出しました。「約 7 割の Issue は無人で完走する」「マージまで至った Issue はループ回数の中央値 2、こじれた Issue は中央値 8.5」。今回はその続き — **なぜ、どうやって Issue はこじれるのか**を、記録とデータから掘ります。

タイトルの「38 回」は、実在した 1 つの Issue で記録されたフェーズ実行回数の合計です。ただしこれは、後述する「ループ」の定義どおりの往復回数ではありません。何を数えた数字なのかは、ケーススタディ 1 で解剖します。

## 前提: 修正ループとキャップの仕組み

前回のおさらいを 1 段落だけ。ループの内実はこうです。AI(Claude Code)が実装し、別の AI(Codex)がレビューする。blocking な指摘が出れば、自動で修正モードに差し戻される。この「実装 → レビュー」の往復を 1 ループと数えます。歯止めは**連続 5 回**— 到達するとキャップに達し、人間にエスカレーションされます。人間側はカウンターをリセットして再開させることもできますし、手動で介入することもできます。

人間によるリセットは、キャップ到達 → 人間がリセット → また到達、を何度も繰り返すことがあります。タイトルの「38」は、その繰り返しの中で記録に残った実行回数の合計です。実装 → レビューの往復数そのものではない点を含め、内訳はケーススタディ 1 で説明します。

## キャップに何回到達したら諦めるべきか

まず全体のデータから。クローズ済み Issue について、キャップ到達回数ごとの PR マージ率を集計しました。

| キャップ到達回数 | マージ率 |
| --- | --- |
| 0 回 | 95% (183/193) |
| 1 回 | 88% (28/32) |
| 2 回 | 78% (14/18) |
| 3 回以上 | 55% (12/22) |

集計する前の仮説は「2 回目のキャップに到達したらほぼ回収不能だろう」でした。この仮説は外れました。実際は**崖ではなく勾配**で、キャップ 3 回以上でも、人間がリセットしながら粘れば半分強はマージまで到達しています。

とはいえ 95% → 55% まで落ちるのも事実です。この推移から、「1〜2 回のリセットは正常範囲。3 回目に入ったら、粘るより Issue の切り方を疑う」が実用的な目安だと言えます。

## ケーススタディ 1: 「38」の解剖(#498)

こじれ方の実例として、最多記録の Issue を開いてみます。内容は「**semantic conflict の文脈を conflict-resolution プロンプトに渡す**」— つまり、**マージコンフリクト対応を賢くするための Issue** でした。後述の通り、この Issue 自体が最終的にコンフリクト絡みで難航することになります。

イベントログを要約するとこうなります(時刻は一部丸めています):

```
7/3 15:16  初回実装 → PR 作成
    15:31〜 レビュー指摘 → 修正 → 再レビュー、を約 10 分間隔でピンポン
    16:50  キャップ 1 回目(5/5)→ 人間へ
    19:06  人間がリセット、ループ再開
    20:41  キャップ 2 回目
    23:21  ワークツリーが dirty で実装が 2 連続中断(別作業の残骸)
7/4        リセット → ピンポン → キャップ、を繰り返し
7/5 02:16  実装エージェントが許可外コマンド(npm install)を要求 → 人間へ
    07:05  人間が対応してループ再開
    07:10  「claude exited 0 but produced no file changes」(修正の空振り)
    07:23  キャップ(通算 5 回目)
7/6 05:20  レビュー通過(No blocking findings)
    06:20  直後のコンフリクト解決で検証(テスト)が失敗 — マージできず
```

最終的にキャップ通算 6 回。この間に走ったフェーズの実行回数は、実装 18 回・レビュー 19 回・コンフリクト解決 1 回の合計 38 回でした(前提で定義した「実装 → レビューの往復」1 ループとは数え方が異なります)。コンフリクト対応を改善するための Issue そのものが、コンフリクト解決時の検証失敗によってマージされないままクローズされました。

この 38 回の中身を見ると、「AI の実装力不足で同じ場所を直せない」わけではないと分かります。**ゴールの方が動き続けていた**のです。レビューのたびに指摘の角度が変わる。修正中に main が進んで前提がズレる。途中では環境要因(dirty tree、許可外コマンド)の事故まで挟まる。近づいては弾かれる、を繰り返した末に、最後は本題そのもの(コンフリクト)に足を取られました。

エージェントループのパターン集 [loop-engineering の Failure Mode Catalog](https://github.com/cobusgreyling/loop-engineering/blob/main/docs/failure-modes.md) では、同じ PR に自動修正が 5 回以上走っても収束しない症状を **Infinite Fix Loop** として挙げています。

## ケーススタディ 2: 4 連敗した Issue 群を、刻み直したら 9 勝した

Issue ごとに独立した git worktree で実装を走らせる — そんな基盤変更を 4 つの Issue(#441〜#445)で進めようとしたことがあります。結果は 4 連敗でした。それぞれ 5〜12 ループを消費して、どれもマージに至らなかった。実装フェーズ・レビューフェーズ・クリーンアップ・権限操作が絡み合う変更で、1 つの Issue が制御フロー全体に手を入れる構図になっていたのが原因です。

そこで方針を変えて、失敗した 4 Issue を **10 個の小さい Issue に刻み直しました**。最初の 1 つはタイトルからして「Split #441: minimal implementation worktree execution」— 元 Issue の最小スライスです。1 Issue が変えるのは 1 つの振る舞いだけ、既存の挙動は変えない、前の Issue の上に積む、という切り方です。

結果は **10 本中 9 本マージ**。同じ機能、同じ AI、同じループで、Issue の切り方だけを変えたら通った。前回の記事で「Issue の質が実装の質をほぼ決める」と書きましたが、質とは書きぶりだけでなく**サイズ**のことでもある。こじれた Issue への対応は、分割してスコープを縮小することでした。

## こじれた Issue を分類すると 4 パターンだった

マージされずに終わった Issue(24 件)を全部見返して分類すると、きれいに 4 つに分かれました。

| 類型 | 件数 | 何が起きるか |
| --- | --- | --- |
| 判定キャリブレーション系 | 6 | 正解が「コードが動く」ではなく「判定が妥当」。AI レビューが粗探し化して収束しない(全件 10〜15 ループ) |
| スコープ過大系 | 4 | 制御フロー全体に跨る変更。→ ケーススタディ 2 の通り、分割したら通った |
| semantic conflict 系 | 2 | 進行中の他変更と意味レベルで衝突し続ける。→ ケーススタディ 1 |
| 設計判断系 | 3 | 「どの挙動が正しいか」が実装中に揺れる。人間の判断が必要だった |

残りは単発の事故(環境要因など)です。今回確認した事例に共通していたのは、**どの類型も「AI の実装力」の問題ではなかった**ことです。ゴールの定義が動く・曖昧・大きすぎる、という **Issue 側の性質**がループを空転させていました。逆に、ゴールが固定されていて検証がテストで書ける Issue は、前回書いたとおり中央値 2 ループで淡々と通っていきます。

## Issue を書く時点でできること

このデータから、私が実際に運用を変えた点です。

1. **基盤・制御フローを跨ぐ変更は、最初から分割する。** 「1 Issue = 1 つの振る舞いの変更」まで刻む。刻み方に迷うレベルで大きいなら、まず「仕様を確定させる Issue」を先に立てる
2. **「判定の妥当性」が成果物のタスクは、自動レビューのループに乗せない。** AI レビューは基準のない対象に対して無限に改善点を生成できてしまう。この手のタスクは人間のレビューゲートに直行させる
3. **キャップは勾配として扱う。** 1〜2 回のリセットは正常運転の範囲。3 回目に入ったら、リセット連打ではなく Issue の切り直しを検討する

「Issue に書かれなかった意図は、AI の自信満々な推測で埋められる」— この問題を Addy Osmani は [Intent Debt(意図の負債)](https://addyosmani.com/blog/loop-engineering/)と呼んでいます。今回の 4 類型のうち、少なくとも設計判断系とキャリブレーション系は、この負債が表面化したものと言えそうです。

## おわりに

こじれの主因が Issue 側にあるということは、裏を返せば**ループ本体をいじらなくても成績は上げられる**ということでもあります。一方でループ側の改善余地もはっきりしていて、レビューに Issue 本文を渡して「要求を満たしたか」を見させる、キャップ到達時に分割を提案させる、あたりが次の一手だと思っています。このへんはまた別の記事で書くかもしれません。
