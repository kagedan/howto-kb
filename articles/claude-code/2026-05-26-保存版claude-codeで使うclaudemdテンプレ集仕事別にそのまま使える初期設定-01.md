Claude Codeを使い始めたら、最初に整えるべきファイルがあります。

**`CLAUDE.md`**です。

`CLAUDE.md`は、Claude Codeへ毎回伝えている作業ルールを残す場所です。

**出力のトーン、禁止事項、保存先、確認ルール**を書いておくと、毎回同じ説明をする必要が減ります。

ただ、初心者が最初に詰まるのもここです。

何を書けばいいか分からない。長く書きすぎる。  
仕事ごとの違いが分からない。

結果として、毎回チャット欄に同じ指示を貼り続けます。

この記事では、仕事別にそのまま使える**`CLAUDE.md`テンプレ**をまとめます。

## CLAUDE.mdは全作業共通の作業ルール

![](https://assets.st-note.com/img/1779755663-vJXj0I2CcxeUaNuL385i6O1k.png?width=1200)

`CLAUDE.md`は、プロンプト集ではありません。

全作業に共通するルールを書く場所です。

たとえば、次のような内容です。

* **日本語で出力**
* **結論から書く**
* **勝手に削除しない**
* **保存先が不明なら質問**
* **変更後に報告**
* **根拠不明の数字を書かない**

公式ドキュメントでも、**`CLAUDE.md`はプロジェクトのmemory**として説明されています。

プロジェクト構造、コーディング標準、共通ワークフローなどを置く用途です。

初心者向けに言うと、毎回言い直している注意事項を置く場所です。

## まず入れる共通テンプレ

![](https://assets.st-note.com/img/1779755687-lUFjR35tbCxmEWqK7MrsodSv.png?width=1200)

最初は短くて十分です。

```
# 作業ルール

## 目的

- このフォルダはClaude Code作業用
- 文章作成、資料整理、調査、企画に使う
- 機密情報と本番データは扱わない

## 出力ルール

- 日本語で出力
- 結論から書く
- 箇条書きを優先
- 長文にしない
- 不明点がある場合は作業前に質問

## 安全ルール

- 勝手にファイル削除しない
- 既存ファイルを上書きする前に確認
- コマンド実行前に目的を説明
- APIキー、パスワード、顧客情報を扱わない

## 保存先

- 素材: inbox/
- 参考資料: reference/
- 下書き: draft/
- 完成物: output/

## 報告形式

- 変更したファイル
- 変更内容
- 確認したこと
- 次にやること
```

まずはこれで十分です。

重要なのは、長くすることではありません。

毎回の言い直しを減らすことです。

## note記事作成用テンプレ

![](https://assets.st-note.com/img/1779755696-5NzvWdUg9CLKt0yAhjVw3JfX.png?width=1200)

noteを書く人は、文章のトーンと構成ルールを入れます。

```
# note記事作成ルール

## 目的

- note記事の企画、構成、本文作成に使う
- 読者が今日使える具体策を渡す

## 文体

- です・ます調
- 一文は短く
- 冒頭で結論か痛点を書く
- 抽象論だけで終わらない

## 構成

- 読者の悩み
- なぜ起きるか
- 解決策
- 具体手順
- 失敗パターン
- 今日やること

## 禁止

- 根拠のない数字
- 誇張した成果
- タイトルだけ煽ること
- 機能名だけの見出し
```

記事用では「どう書くか」より「**読者が何をできるようになるか**」を入れます。

## 議事録整理用テンプレ

![](https://assets.st-note.com/img/1779755711-Zr5z8V9HYf7AWEoNi2USgkGM.png?width=1200)

議事録では、正確性が最優先です。

```
# 議事録整理ルール

## 目的

- 会議メモを実行できる形に整理する
- 決定事項、TODO、未決事項を分ける

## 出力形式

- 会議概要
- 決定事項
- TODO
- 未決事項
- 次回確認事項

## 禁止

- 発言にない内容を決定事項にしない
- 担当者を勝手に補完しない
- 期限を勝手に作らない
- 不明点は未確認と書く
```

議事録では、AIの気の利いた補完が事故になります。

不明なものは不明なまま出すルールが必要です。

## リサーチ用テンプレ

![](https://assets.st-note.com/img/1779755720-fZOyAobQ12CUL97MgPuSTJsH.png?width=1200)

調査では、事実と解釈を分けます。

```
# リサーチ整理ルール

## 目的

- 調査情報を意思決定に使える形へ整理する
- 事実、解釈、次アクションを分ける

## 出力形式

- 結論
- 確認済み事実
- 解釈
- 出典
- 実務影響
- 次アクション

## 禁止

- 出典がない数字を断定しない
- 事実と意見を混ぜない
- 公式情報を確認せず仕様を断定しない
```

AIニュースや競合調査を扱う人は、このテンプレが効きます。

## 開発作業用テンプレ

![](https://assets.st-note.com/img/1779755729-gWl8xp6tR9dFSiZI12eLBVvo.png?width=1200)

開発では、変更前確認と差分確認を入れます。

```
# 開発作業ルール

## 作業前

- 変更前に対象ファイルを読む
- 既存の実装パターンを確認する
- 影響範囲を短く説明する

## 作業中

- 関係ないリファクタをしない
- 既存の命名規則に合わせる
- 不明点があれば質問する

## 作業後

- 変更ファイルを報告する
- 確認したコマンドを報告する
- テストできない場合は理由を書く
```

初心者は、いきなり本番リポジトリで使わないでください。

練習用フォルダで、差分確認の習慣を先に作ります。

## 事業企画用テンプレ

![](https://assets.st-note.com/img/1779755739-DhU3BpwnCA9cNbXqIzWskFJ0.png?width=1200)

企画では、目的と顧客を先に固定します。

```
# 事業企画ルール

## 進行順

1. 目的
2. 制約
3. 顧客
4. 提供価値
5. 商品
6. 導線
7. 実行計画
8. 計測

## 禁止

- 顧客未定でコピーを書かない
- 目的未定でLP構成を作らない
- 価格未定で販売導線を断定しない

## 出力形式

- 結論
- 想定顧客
- 提供価値
- 商品案
- 導線
- 次アクション
```

企画系のClaude Code運用では、順番を固定するだけで品質が上がります。

## 書きすぎを防ぐルール

![](https://assets.st-note.com/img/1779755749-aoblj67LOHFtD2XnQ05vzA3i.png?width=1200)

**`CLAUDE.md`は長ければ良いわけではありません。**

初心者がやりがちな失敗は、全部を`CLAUDE.md`へ入れることです。

分け方はこうです。

* **全作業に関係すること: `CLAUDE.md`**
* **特定作業だけの手順: skills**
* **一度だけの依頼: 通常プロンプト**

記事企画の手順を毎回使うなら、`article-planner` skillへ入れます。

議事録整理の手順なら、`meeting-summary` skillへ入れます。

`CLAUDE.md`は、薄く保つ方が使いやすいです。

## 使いながら直す方法

![](https://assets.st-note.com/img/1779755767-XUCdWHMuTsF4L1zObfk0S5aj.png?width=1200)

最初から完璧に作る必要はありません。

3回使って、直します。

見直すポイントは、次の5つです。

* **毎回同じ注意をしていないか**
* **出力が長すぎないか**
* **保存先を間違えていないか**
* **禁止事項が守られているか**
* **自分の仕事に合っているか**

直し方は簡単です。

毎回言い直した文を、`CLAUDE.md`へ1行追加します。

たとえば、出力が長いならこうです。

```
- 通常回答は1200字以内
- 詳細が必要な場合だけ長く書く
```

勝手な提案が多いなら、こうです。

```
- 依頼外の提案は最後に「追加提案」として分ける
```

Claude Codeは、使うほど勝手に育つわけではありません。

自分が直した指示を残すほど、仕事に合うようになります。
