### ⚠️ その前に：閉じ込めの「穴」を1つ塞ぎました

会議を始める前に、前回のコードに**閉じ込めの穴**を見つけたので直しました。

main.py

```
# 修正前：プロンプトの先頭50文字がCloud Loggingに残ってしまう
logger.info(f"Sending prompt to Gemini: {request.prompt[:50]}...")

# 修正後：文字数だけログする（本文は一切残さない）
logger.info(f"Sending prompt to Gemini (length: {len(request.prompt)} chars)")
```

VPCで**通信経路**を閉じても、ログに本文を書いたら**GCPのログ基盤に秘密の冒頭50字が残ります**。公開記事の校正なら無害ですが、本物の秘密を流すなら致命傷。「経路の閉じ込め」と「ログの漏れ」は別の穴なんです。

**要するに「セキュリティは通信だけじゃなくログも見ろ」が今回最初の学びでした！**

## 2. 密室AI会議（/council）の設計 🏛️

穴を塞いだところで、本題の会議システムです。

### AI Town と何が違うの？

「AI同士が勝手に暮らす」といえば、少し前に話題になった **AI Town / Smallville**（AIたちが村で生活するシミュレーション）があります。最初「あれに似てるな〜」と思ったんですが、比べてみると**別物**でした。

|  | /council（今回作るやつ） | AI Town / Smallville |
| --- | --- | --- |
| きっかけ | 私がテーマを渡して開始 | 勝手に始まる・ずっと動いてる |
| 進行 | **ラウンド制で順番に発言**、司会が仕切る | 各自が勝手に行動・移動・雑談 |
| 終わり | 決めた回数で終了、**議事録が出る** | 終わらない（生活が続く） |
| 記憶 | その会議だけ（持ち越さない） | 永続的な記憶を持って翌日も覚えてる |
| 目的 | お題を詰めて**成果物（議事録）を出す** | 社会が創発的に生まれるのを観察 |

そして最大の違いがここ👇

> AI Townは「賑やかな村」。  
> こっちは **restricted VIPで外界から隔離された「秘密が漏れない密室都市」**。  
> 住人（AI）同士は喋れるけど、会話も秘密も、街から一歩も外に出ない。

### 実装の中身

役割（ペルソナ）の違う参加者を用意して、ラウンドごとに順番に発言させます。ポイントは **それまでの全発言（transcript）を毎回プロンプトに渡す**こと。これで「前の人の発言に反論する」が成立します。

📝 /council の実装（main.py 抜粋）（クリックで開閉）

main.py

```
# デフォルトの参加者。persona=役割、brain=担当AI
DEFAULT_PARTICIPANTS = [
    {"persona": "楽観的な発案者", "brain": "gemini"},
    {"persona": "慎重な批判者", "brain": "claude"},   # ←この席が悲劇を生む
    {"persona": "現実的なまとめ役", "brain": "gemini"},
]

def _build_council_prompt(persona, theme, transcript, others):
    # これまでの議事録を丸ごと渡すのがキモ
    history = transcript if transcript else "(まだ誰も発言していません。あなたが口火を切ります。)"
    return f"""あなたは「{persona}」という役割でこの会議に参加しています。
【テーマ】
{theme}
【これまでの議事録】
{history}
【指示】
・「{persona}」の立場から短く意見を述べてください（300文字以内）。
・他の参加者の発言に賛成・反論するのも歓迎です。
"""

@app.post("/council")
def council(request: CouncilRequest):
    # 暴走・課金対策でラウンド数を1〜5に制限
    rounds = max(1, min(request.rounds, 5))
    transcript = ""
    for r in range(1, rounds + 1):
        for p in participants:
            reply = _council_speak(p.brain, p.persona, request.theme, transcript, others)
            transcript += f"\n### {p.persona}（{p.brain}）\n{reply}\n"
    # 最後にMarkdownの議事録を返す
```

地味に大事なのが `rounds = max(1, min(request.rounds, 5))`。**AIたちが延々喋り続けるとトークン代が溶ける**ので、上限は必ず入れましょう。

### 動かしてみた

お題は何でもOKです。たとえば **「遠足のおかし代はいくらまで？」** というゆるいテーマで、Gemini 3人（発案者・批判者・まとめ役）に議論させた結果がこちら——

![密室AI会議ビューア。遠足のおかし代についてAIたちが熱く議論している様子](https://static.zenn.studio/user-upload/b87e30f6849e-20260713.png)  
*おかし代（500円）を巡って、AIたちがめちゃくちゃ真面目に「思い出は値段じゃない」と議論しています😂*

テーマ「個人開発者が最初に固めるべきセキュリティは何か」で、Gemini 3人（発案者・批判者・まとめ役）に議論させた結果——

![密室AI会議ビューア。楽観的な発案者と慎重な批判者がチャット形式で議論している](https://static.zenn.studio/user-upload/0078351aa140-20260713.png)  
*議事録JSONをチャット風に描画するビューア（※この時点ではまだClaudeが使えなかったため、批判者役もGeminiが担当しています）。発案者の主張に批判者がちゃんと反論してる…！*

これ、ただ順番に喋ってるんじゃなくて **"議論"になってる** んです：

* **発案者**：「認証＋入力検証こそ最高の投資だ！」
* **批判者**：「"最小限の手間で最高"は楽観的すぎ。ツールは**設定ミスで逆に穴になる**。認可やビジネスロジックの見落としも危ない」←ちゃんと反論してる
* **まとめ役**：「両者を汲んで、まずインフラ基礎＋既知脆弱性対策→**枯れたライブラリをデフォルトで使う**方針に」←合意形成してる

後の人が前の発言を読んで、引用して、反論して、まとめてる。ラウンド制・transcript蓄積が機能してる証拠です。

しかもこの会話、**全部restricted VIPの内側＝密室の中**で起きています。外に一文字も漏れてません🔒

**要するにこのステップでは「transcriptを毎回全部渡す、だけで議論は成立する」が大事！**

## 3. Devinのサンドボックスとどう違うの？ 🔬

前章の「密室」を見て、「それってDevin（AIエンジニア）のサンドボックスと同じでは？」と思った方、鋭いです。

両方「隔離（サンドボックス）」なんですが、**隔離してる"対象"が真逆**なんです。

|  | VPC閉じ込め（今回） | Devinのサンドボックス |
| --- | --- | --- |
| 何を隔離してる | **データの通り道（通信）** | **AIの作業場（実行環境）** |
| 何から守る | 秘密データが外に漏れること | AIがホスト/本番を壊すこと |
| 脅威モデル | 「情報が流出したら困る」 | 「AIが暴走したら困る」 |
| たとえ | **盗聴されない防音個室**で相談する | **何を壊してもいい実験室**にAIを入れる |

* **VPC** = 「AIは信頼する。でも**データは1ミリも外に出さない**」→ 守る対象は**データ**
* **Devinのサンドボックス** = 「AIは信頼しきれない。だから**作業場を隔離する**」→ 守る対象は**環境（ホスト）**

そしてこの構造にはトレードオフがあります：

```
閉じ込め   ＝ セキュリティ↑ ／ 外の最新情報を取りに行く力↓
開けておく ＝ 最新情報に強い ／ でもデータが外に出る
```

密室のAIたちは最新ニュースを検索できません。**安全と引き換えに、外の世界を知らない**んです。ちょっと切ない設計です。

## 4. Claudeの引っ越し手続きが大変すぎた 🚚

さて、会議は動きました。次は住人を増やします。**Vertex AIのModel GardenにはClaudeもいる**んです。

Geminiが住んでる街に、Claudeが引っ越してくる。「慎重な批判者」の席はClaudeのために用意しました。

……と思ったら、**APIキーでAIを呼ぶのとはわけが違いました**。

### 手続きの全体像

| ステップ | 内容 | 大変さ |
| --- | --- | --- |
| ① Model Gardenで有効化 | Claudeのページで利用規約に同意＋**利用申請フォーム** | 😅 会社情報を聞かれる |
| ② リージョン選び | `global` を使う（東京に全モデル揃ってないため） | 🙂 コード1行 |
| ③ 認証 | Cloud Run上なら**キー不要で自動**（ADC認証） | 😊 何もしなくていい |
| ④ クォータ | **初期値0**。増枠申請が必要 | 😭 ここで地獄 |

🔰 なぜAPIキーが「存在しない」の？（クリックで開閉）

Vertex版Claudeは、Anthropic社のAPIキーではなく **GCPのADC認証（Application Default Credentials）** で動きます。

Cloud Runは「サービスアカウント」という身分証を持って動いているので、コードに何も足さなくても認証が通ります。

**キーという文字列がそもそも存在しない**ので、キーの漏洩事故が原理的に起きません。めちゃくちゃ安全な設計です。……ただしそのぶん、**手続きの段階を踏まないと呼べない**ので泣きそうでしたw

💡 リージョン `global` でも閉じ込めは効くの？（クリックで開閉）

効きます！ここが今回一番安心したポイント。

`global` エンドポイントの宛先は `aiplatform.googleapis.com`——これは前回 `199.36.153.5`（restricted VIP）に解決されることを**証明済みの、まさにあのホスト**です。

つまりClaudeを `global` で呼んでも、通信は密室の中。閉じ込めはそのまま保たれます🔒

### そして429エラー

申請フォームも正直に書いて（個人開発でも、実名・実サイト・本当の用途で埋めればOKです。ウソの会社をでっち上げるのはダメ）、有効化して、いざ会議に参加させたら——

```
（claude）は発言できませんでした: Error code: 429 -
'Quota exceeded for aiplatform.googleapis.com/global_online_prediction_requests_per_base_model
 with base model: anthropic-claude-haiku-4-5. Please submit a quota increase request.'
```

**新規プロジェクトはClaude系のクォータが初期値0**。つまり「使えるけど、1分あたり0回まで」😂

### クォータ増枠申請

GCPコンソールの「割り当てとシステムの上限」から、`anthropic-claude-haiku-4-5`（global）の行を探して、控えめに「10リクエスト/分」で申請しました。

![GCPのクォータ編集画面。anthropic-claude-haiku-4-5の割り当てが0で、新しい値10を申請している](https://static.zenn.studio/user-upload/5ada5d886eeb-20260713.png)  
*現在の値：0。せめて10リクエスト/分ください……という控えめな申請*

すると、メールが来ました。

![メール画面](https://static.zenn.studio/user-upload/afd7f5ffae30-20260713.png)

> **メール①**「クォータの増量リクエストの処理には、通常2営業日かかります」  
> **メール②**「残念ながら、現時点では追加の割り当て枠をご提供できません。**請求アカウントに履歴が追加されるまで48時間お待ちください**」

え〜48時間待つの！？　……待ちました。再申請しました。**また同じ返事が来ました。**

### なぜこんなに大変なのか

ClaudeはGoogle製ではなく「**他社（Anthropic）の製品をGoogle経由で借りる**」形だから。他社モデルにはGoogleが**二重の関所**を置いています：

1. **利用申請フォーム**（会社名とか聞かれたやつ）
2. **クォータ初期値0**（使う前に増枠申請が必要）

これ全部、**企業ユーザー想定の作り**なんです。だから個人だと関所でいちいち引っかかる。

私のGoogleアカウントも請求アカウントもプロジェクトも昔からあるものなので、「履歴が貯まるまで待って」が何度も来るのは変なんですよね……（この結末は最終章で🥲）

## 5. Grok、来襲 🛸

Claudeの審査を待つ間、Model Gardenを眺めていたら——**Grokがいました**。

しかもこちらは**申請すらいらず、コードを書くだけで呼び出せる**とのこと。早速やってみたら、すんなり動いた上に、**Geminiとのやりとりに爆笑しました**😂

![Grokは有効化のボタンすら無し](https://static.zenn.studio/user-upload/9da0ceb5533d-20260713.png)  
*Grokは有効化のボタンすら無し！*

### 事件：Grokが真逆のことを言い出す

![ぶっ飛んでるGeminiに引いてるGrok😂](https://static.zenn.studio/user-upload/9ce5afa91ae3-20260713.png)  
*ぶっ飛んでるGeminiに引いてるGrok😂*

最初は役割（ペルソナ）を付けていたんですが、なぜかGeminiのほうがぶっ飛んでいたので、いったん役割を外して素の状態で雑談させてみました。テーマは「こたつから出るべきか」というゆるいお題。すると——

![テーマは「こたつから出るべきか」](https://static.zenn.studio/user-upload/e1a8b28981d5-20260713.png)

> **Grok**：出るべき！春だよ、もう。  
> **Gemini**：春だからって？油断できないですよね、春の寒さ！  
> **Grok**：春の寒さなんて**こたつで溶かせばいいじゃん！出ない！**

**え！真逆のこと言ってるw**

さっき「出るべき」って言ったよね！？　ほんとに面白いです🤭

### これ、バグじゃなくて「モデルの特性」です

最初は「文脈が渡ってないのかな？」と疑いました。でも違うんです。会議のプロンプトには毎回**それまでの全会話（transcript）が丸ごと**入っています。Grokは過去の発言を全部渡された状態で喋ってる。

じゃあなぜ矛盾したのか👇

**正体：私が選んだのが「非Reasoning（考えない）版」だったから**

選んだモデルは Grok 4.1 Fast (**Non-Reasoning**) ＝ "喋る前に考えない"最速・最安タイプ。この子の特徴：

* **直前の相手の発言に反射的にツッコむ**のは得意（テンポは最高）
* でも **「自分がさっき何て言ったか」を保つ一貫性は弱い**

つまり：

1. R1「出るべき！春だよ」
2. （Geminiが「春でも寒いよ」と返す）
3. R2「春の寒さはこたつで溶かせばいい！**出ない！**」← **Geminiに反論したくて、自分の立場を裏切った**

忘れてるんじゃなく、**"直前に反応"を優先して自分の意見をコロコロ変えてる**。非Reasoning版あるあるです。

## 6. まとめ：この密室、どこまで安心？ 🔒

ここまでの構成を整理すると、この会議（？）は：

* ✅ 通信はインターネットに出ない（restricted VIP）
* ✅ Vertex AIなので**入力が学習にも使われない**（規約）
* ✅ ログにプロンプト本文を残さない（第1章で修正）

だから、**普通の無料チャットよりずっと安心して相談できます**。

ただし、**"絶対に漏れない金庫"ではありません**。最後の壁 VPC SC は組織アカウント必須で、個人では未構築。なので **自分ごとの相談は気軽に、他人の個人情報やガチの機密は慎重に**。ここが「個人でできる範囲」と「企業の本番」の境目です。

🏢 「企業がそのまま本番投入」に足りないもの（クリックで開閉）

* 🧱 **VPC SC**（組織アカウント必須の最終防壁）← 前回ぶつかった壁
* 📋 監査ログ・アクセス制御（誰が呼べるか）・DLP
* 🏗️ Terraform等のIaC・CI/CD・テスト・監視/アラート
* 📜 コンプラ（SOC2/ISO）・データガバナンス・契約(DPA)

それでも、**"会社のあれこれ"をフランクに相談できる相手が手元にある**のは大きいと思うんです。Claude CodeやDevinやCodexに、そういうざっくばらんな相談ってしないですよね。しかもコードをいじれば**RAGとして自分専用の資料も読ませられます**！

## 7. 残念な結果 🥺

……そして、Claudeの件の結末です。

何度か時間をあけて再申請しましたが、**毎回同じ返信**。支払い実績もプロジェクトも昔からあるのに「48時間待ってね」が繰り返されるのは、どう考えてもおかしい。

そして決定的な証拠を見つけてしまいました。

![「受け付けました」メールと「付与できません」メールが、まったく同じ時刻19:13に届いている](https://static.zenn.studio/user-upload/da312d3110aa-20260713.png)  
*「リクエストを受け付けました」と「残念ながら付与できません」が、**まったく同じ時刻（19:13）** に届いている*

却下メールには「処理には通常**2営業日**かかります」と書いてあるのに、受付と却下が**同時刻**。つまり**人間は一切見ておらず、自動で弾かれている**のは間違いなさそうです。

なぜ個人だと自動で弾かれるのか——**ここからは私の推測ですが**、GoogleにとってClaude（Anthropic社）は他社モデルで、提供できるキャパシティに限りがある。そのため「**企業の専任営業担当（Sales Rep）がついている大口顧客**」が優先され、営業担当のいない個人アカウントの増枠申請は自動で後回しにされているのでは、と考えています（却下メールも「Sales Repに相談してね」と案内してきます）。

とても悔しいです。

実は「きっと引っ越してこられるはず」と思って、お祝いの漫画まで作っていました。めちゃ可愛くできたので、**現実には来られなかったけど、漫画の中だけでも住まわせてあげてください**🥲

![Claude（黒いロボット）がアーチの街に引っ越してきて、手続きを終え、GeminiとGrokに歓迎されてケーキでお祝いされる4コマ漫画](https://static.zenn.studio/user-upload/5a18c7e4cb32-20260713.png)  
*幻の「Claudeくん引っ越し完了」漫画。手続き完了！3人でアーチの街を彩っていこう……（現実：クォータ0で門前払い）*
