Claude Codeを使い込んでくると、必ずこうなる。

「保存したら自動でフォーマットして」「ファイル変えたらコミット作って」「作業終わったらSlackに通知して」——同じことを毎回頼むのが面倒くさくなってくる。

そこで出番なのが **Hooks**。Claude Codeが特定のイベントを起こした瞬間に、自動で別の処理を走らせる仕組みだ。

---

**自分**：「ファイル保存したら毎回フォーマットしてって、何回言わせる気？」

**Claude**：「それ、Hookに書けば一回で終わるよ」

**自分**：「Hookって何？」

**Claude**：「『ファイル保存』というイベントに、『フォーマット実行』をくっつける仕組み。一度書いたら、以降は自動で動く」

---

## Hooksとは——「イベント」と「処理」を繋ぐ仕組み

レストランの厨房で例えるとわかりやすい。

注文が入る（イベント）→ シェフに伝票が回る（自動処理）→ 料理ができたらホールに知らせる（自動処理）

Claude Codeも同じ。「ツールを使った」「作業が終わった」といったイベントに、自分で決めた処理（シェルコマンド）を紐づけておく。それだけだ。

## Hookの主な3種類

Claude Codeにはいくつかのフックポイントがあるが、ここでは初心者がまず押さえるべき3つを紹介する。

**PreToolUse** ——ツールを使う前  
Claudeがファイル編集や `git push` などのツールを呼ぼうとした瞬間、その**直前**に走る。「本当にやっていい？」のチェックや、ログ記録に使える。

**PostToolUse** ——ツールを使った後  
ツール実行が完了した**直後**に走る。「ファイルを編集したら自動でフォーマットを掛ける」「ファイル作成後にlintを通す」など、後始末に使うのが定番。

**Stop** ——Claudeが一区切りつけた後  
Claudeが返答を終えてユーザーに会話の番が戻った瞬間に走る。「終わったらSlackに通知」「終わったらBeep音」など、完了通知に使う。

## 実用例3つ

### 例1：Stop × 完了通知（最初に試すならこれ）

Claudeの返答が終わったら、macOSの通知センターにメッセージを出すだけのシンプルなHook。

```
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claudeの作業が終わりました\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
```

別の作業をしている間も、Claudeの完了タイミングが目に入る。SlackやDiscordのWebhookに飛ばすパターンもよくあるが、まずは手元の通知から始めるのが安全だ。

### 例2：PostToolUse × 自動Gitコミット

Claudeがファイルを編集・新規作成したら、自動でその差分をコミットに残す。後で「これ、いつ・なぜ変えたんだっけ？」を辿れるようになる。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "git add -A && git commit -m 'auto: claude edit' --allow-empty || true"
          }
        ]
      }
    ]
  }
}
```

`matcher` は「どのツールに反応するか」の指定。ここでは `Edit`（既存ファイルの編集）と `Write`（新規作成）の2つに反応させている。末尾の `|| true` は、コミットが失敗したときに全体を止めないためのおまじない。

### 例3：PostToolUse × 自動フォーマット

Pythonファイルを編集したら、自動で `black`（Pythonのフォーマッター）を実行する。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "black . --quiet 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
```

JavaScriptなら `prettier`、Goなら `gofmt`、Rustなら `cargo fmt`。自分の言語のフォーマッターに置き換えて使える。コードの見た目が常に整うから、レビューが楽になる。

## 実際に設定する手順

設定ファイルは `~/.claude/settings.json`。エディタで開いて、`hooks` キーを追加する。

```
{
  "model": "claude-sonnet-4-6",
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"完了\" with title \"Claude\"'"
          }
        ]
      }
    ]
  }
}
```

保存したら、Claude Codeを再起動する。再起動しないと新しいHookは効かない。

次回からClaudeが返答を終えるたびに、通知が飛んでくる。試して動いたら、PostToolUseに進んでいく。

## 最初に設定すべきは Stop フック一択

Hookは便利だが、いきなり複数仕込むと事故る。**初手はStopフック1つだけ**にすべきだ。理由は3つ。

1. **副作用が小さい** ——会話の区切りで一度しか走らないので、デバッグが楽
2. **体感しやすい** ——通知が来るから「お、動いた」と即わかる
3. **失敗してもダメージが小さい** ——通知が出ないだけで作業自体は止まらない

逆にPostToolUseは「ツールを使うたびに毎回」走るから、設定ミスがあるとClaudeの動作が全部詰まる。慣れてから進むのが安全だ。

## Hookに重い処理を書かない

ここが一番のハマりポイントだ。Hookは**毎回実行される**ので、重い処理を入れると、Claude Codeの動作自体が遅くなる。

* ✅ `echo` や `osascript` で通知 → 0.1秒
* ✅ `git commit` → 1秒以内
* ✅ `prettier` や `black` のフォーマッター → 2〜3秒
* ❌ `npm test` をフルで走らせる → 30秒以上、毎回これだとストレス
* ❌ 全ファイルをスキャンしてレポート生成 → 重すぎ

目安は**3秒以内で終わる処理**。それを超えるものは、Hookではなく手動コマンドかcronに任せたほうがいい。

## まとめ——「Claudeに頼む」のを止めると楽になる

| 毎回頼んでいたこと | Hookで自動化 |
| --- | --- |
| 「ファイル保存したらフォーマットして」 | PostToolUse × フォーマッター |
| 「変更したらコミット作って」 | PostToolUse × git commit |
| 「終わったら教えて」 | Stop × 通知 |

Hookは小さい仕組みだが、効果はじわじわ効いてくる。「Claudeに毎回お願いしていたこと」を、「Claudeが勝手にやること」に格上げできる。
