## TL;DR

* Claude Code の人気スキル [`grilling`](https://github.com/mattpocock/skills)（実装前に計画を容赦なく詰めるインタビュー）を、鵜呑みにせず **A/B検証** した。
* 素のスキルは優秀（抽出した技術主張 **15/15 が正しい**）。**ただし「検証ステップ」が無く**、記憶頼みで自信満々に**誤った機序**を出すことがあった（実例：「Looker Studio はシートを `gid` で参照する」→ 実際は **タブ名**）。
* 「当たっていた」は **運であって過程ではない**。そこで5つのガードレールを足し、用途で使い分けられるよう `/grill`（軽）/ `/grill-deep`（重）に分けて公開した：  
  → <https://github.com/0x000x7f/claude-code-skills>

## `grilling` とは

Matt Pocock 氏の [skills リポジトリ](https://github.com/mattpocock/skills) にある生産性スキルで、**実装前に計画・設計を1問ずつ容赦なく詰める**インタビュー。各質問に「なぜ重要か・潰すリスク・推奨回答」を添え、設計ツリーを辿りながら曖昧な前提や破綻点を着手前に洗い出す。コード生成前の“やけど防止”に効く良いパターンだ。

## なぜ自作したか

最初の動機は単純で、**軽量・自前制御**。上流を丸ごと入れず、最小限を `~/.claude/skills/` にコピーして使い始めた。この時点では「ミスを見つけたから」ではない。**使ってみてから**問題に気づいた。

## 使ってみて発覚：検証ステップが無い

ある計画（汎用例：Google Colab で CSV を読み、gspread で Google Sheet に書き、Looker Studio のダッシュボードが参照する）を grill にかけ、出力を**独立にファクトチェック**してみた。方法は単純な A/B：

1. 素の `grilling` で計画を stress-test
2. 強化版で同じ計画を stress-test
3. 別エージェントが、素の出力の**技術的主張を1つずつ公式ドキュメントと照合**

結果は意外だった。**素のスキルは技術主張 15/15 が正しく**、前提（「そもそも Colab はスケジューラではない」等）も自力で否定できていた。つまり「盲目」ではない。**優秀**だ。

問題は出力の正しさではなく**構造**にあった：

* **検証ステップが無い。** すべて記憶からの断定で、参照も裏取りも無い。今回は全部当たっていたが、その過程は「正しい」と「自信のある推測」を**区別できない**。
* 実際、別の実行では **`grilling` が自信満々に誤った機序を出した**：

  > 「Looker Studio はワークシートを **`gid`** で参照するので、タブを作り直すと壊れる」

  公式ドキュメントを確認すると、Looker Studio の Google Sheets コネクタは **タブ名**で参照する（`gid` ではない）。結論（in-place 更新が安全）は同じでも、**理由が間違っている**。そして**それを止める仕組みがスキルに無い**。

「当たっていた」は運であって、再現性のある過程ではない——これが core の気づきだった。

## 5つのガードレールで強化

そこでスキル末尾に5つのルールを足した。最重要は1つ目：

1. **検証するか、未検証と明示せよ** — 決定を左右する技術的主張は、ドキュメント/コードで**検証する**か、できなければ `[UNVERIFIED]` と**タグ付け**する。確信ある推測を“確定した事実”に化けさせない。
2. **見た目でなく成果物を開け** — ファイル/データの**実体**（エンコーディング・ヘッダ・サンプル値）を見てから処方する。バグは図ではなくバイトに潜む。
3. **前提を疑え** — 設計ツリーを辿る前に「そもそもこの木が正しいか」を問う。
4. **横断リスクを掃引せよ** — privacy・冪等性・原子性・レート/クォータ・依存バージョン・秘密衛生・可観測性を一周。
5. **完全性パスで締めよ** — 最後に「何が漏れた?」を1周し、残る `[UNVERIFIED]` を棚卸し。

## 使い分け：`/grill`（軽）と `/grill-deep`（重）

ここで正直なトレードオフがある。**5ルールは重い**。小さなタスクには過剰で、テンポを殺す。A/Bが示したとおり素でも多くは当たるので、限界効用は場面次第だ。

だから1本にせず**2段**にした：

| コマンド | 中身 | 用途 |
| --- | --- | --- |
| `/grill` | ベース＋**検証ルール1つだけ**（常時ON） | 小〜中タスク |
| `/grill-deep` | ベース＋**5ルール全部** | 高stakes（インフラ/セキュリティ/不可逆/データ/ポートフォリオ） |

両方とも `disable-model-invocation: true` の**手動起動**。勝手には発火させず、「この計画は詰める価値がある」と自分が判断したときだけ撃つ。

## 結論：スキルの最適点は「誰の何のため」で変わる

面白いのは、\*\*同じ5ルールが「汎用スキルには過剰／個人の高stakes運用には適正」\*\*になることだ。汎用配布なら簡潔さが正義で、上流が簡潔を保つ判断は正しい。一方、インフラ変更やセキュリティ作業を日常的にやるなら、検証・横断リスクのガードレールは確実にペイする。

採用される・されないより、**自分の判断軸を持ち、検証で裏取りする**こと自体が資産になる——という小さな実例だった。
