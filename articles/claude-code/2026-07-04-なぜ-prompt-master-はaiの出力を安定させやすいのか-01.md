## この記事で見ること

AIに何かを頼むとき、私たちはついこう書いてしまいます。

> いい感じに直して  
> もっと分かりやすくして  
> このコードを改善して  
> プロっぽいデザインにして

チャットAIに軽く相談するだけなら、これでも返事は返ってきます。

ただ、Claude Code、Cursor、Devin のように、AIがファイルを編集したり、コマンドを実行したりする場面では、この曖昧さが手戻りにつながります。

* 関係ないファイルまで触る
* 必要か分からない依存関係を追加する
* 頼んでいないリファクタまで始める
* 何をもって完了なのか分からず、やり取りが長引く

`prompt-master` がやっていることは、この曖昧さを減らすことです。

<https://github.com/nidhinjs/prompt-master>

この記事では、`prompt-master` の仕組みを読みつつ、小さな実機検証で「雑なプロンプト」と「prompt-master型のプロンプト」を比べます。

![曖昧なお願いから作業指示へ変わる流れ](https://static.zenn.studio/user-upload/a949c91ce550-20260704.png)  
*曖昧な依頼ではAIが推測で補う。制約と成功条件を渡すと、判断範囲が狭まる。*

## prompt-master は長いプロンプトを作るツールではない

prompt-master の README には、次のような趣旨の説明があります。

> The best prompt is not the longest. It's the one where every word is load-bearing.

<https://github.com/nidhinjs/prompt-master/blob/main/README.md>

良いプロンプトは、単に長いプロンプトではありません。

結果に影響する情報が、必要な場所に入っているプロンプトです。

雑な依頼に説明を足せば、プロンプトはすぐ長くなります。しかし、長くなるほど重要な制約は見つけにくくなります。AIにとっても、何が必須で何が補足なのか判断しづらくなります。

prompt-master は、不要な語を削り、必要な制約を前に出す方針を取っています。

たとえば「プロっぽくして」という依頼を、そのまま投げません。

* どのAIツールに渡すのか
* 何を作るのか
* どの形式で出すのか
* 何をしてはいけないのか
* いつ完了なのか
* どこまで自律的に作業してよいのか

こうした情報に分解してから、プロンプトを組み直します。

## 9つの意図を先に固定する

prompt-master の `SKILL.md` では、プロンプトを書く前に9つの観点を抽出する設計になっています。

<https://github.com/nidhinjs/prompt-master/blob/main/SKILL.md>

![prompt-masterが固定する9つの意図](https://static.zenn.studio/user-upload/00d86e6e7f3a-20260704.png)  
*粗い依頼を左から右へ、4つの棚に分解してから貼れるプロンプトへ組み直す。*

* Task
* Target tool
* Output format
* Constraints
* Input
* Context
* Audience
* Success criteria
* Examples

この中でも、実務で効きやすいのは **target tool** と **success criteria** です。

同じ依頼でも、Claude に投げるのか、Cursor に投げるのか、Midjourney に投げるのかで、書き方は変わります。

* Midjourney なら、散文よりも視覚的な記述子が重要
* Cursor なら、対象ファイルや関数名が重要
* Claude Code なら、作業範囲、禁止事項、停止条件が重要

それなのに、人間はよく「AIに聞く」という一言でまとめてしまいます。

prompt-master はここを分けます。AI一般に向けたプロンプトではなく、そのツールに合わせたプロンプトに変換するわけです。

## 失敗しやすいプロンプトには型がある

prompt-master の `references/patterns.md` には、再プロンプトやトークン浪費につながるパターンが整理されています。

<https://github.com/nidhinjs/prompt-master/blob/main/references/patterns.md>

たとえば、次のようなものです。

* vague task verb: `help me`、`make it better`
* no success criteria: 完了条件がない
* missing output format: 出力形式がない
* no scope boundary: 作業範囲がない
* no stop condition for agents: エージェントの停止条件がない
* hallucination invite: 根拠なしに答えさせる聞き方

プロンプトが期待通りに動かない原因は、モデルの能力だけでは説明できません。

こちらの依頼が、作業としてまだ十分に定義されていないこともあります。

たとえば、次の依頼ではAIがどこまで触ってよいのか分かりません。

少なくとも、次のような点が曖昧です。

* `Login.tsx` だけか
* API route も触るのか
* DB schema は変更してよいのか
* テストは追加するのか
* UI文言も変えるのか
* 既存仕様はどこまで維持するのか

この曖昧さを残したままAIに渡すと、AIは推測で埋めます。その推測が外れると、手戻りになります。

prompt-master は、この推測部分を先に減らします。

## AIエージェント時代には停止条件が必要になる

チャットAIに質問するだけなら、出力が気に入らなければ読み捨てれば済みます。

しかし、AIエージェントは少し違います。

Claude Code や Devin のようなツールは、ファイルを読み、編集し、テストを実行し、場合によっては依存関係も追加します。

このとき重要になるのが、次のような指定です。

![AIエージェントに渡す作業境界](https://static.zenn.studio/user-upload/137a0f55b2b2-20260704.png)  
*実行権限を持つAIには、触ってよい範囲、禁止事項、停止条件、人間に戻す条件を渡す。*

```
src/auth/ 配下のファイルだけを編集してください。
データベーススキーマは変更しないでください。
新しい依存関係は追加しないでください。
ファイルを削除する前に、必ず確認してください。
既存テストが通り、user が null の場合でも login が例外を投げなくなったら完了です。
```

これは、文章術というより **作業境界の設計** です。

prompt-master が agentic tool 向けに重視しているのもここです。

* starting state
* target state
* allowed actions
* forbidden actions
* stop conditions
* human review triggers

この情報があると、AIは「何をすればよいか」だけでなく、「何をしてはいけないか」も判断しやすくなります。

AIの自律性が高くなるほど、プロンプトには自由度だけでなく境界も必要になります。

## 実機で小さく試した

ここまでの話が机上の説明だけにならないように、同じテーマで2つのプロンプトを実際に投げて比べました。

* 雑なプロンプト: `prompt-masterがなぜ良い結果になるのか、技術記事向けに説明して。`
* prompt-master型プロンプト: 対象ツール、入力事実、制約、出力形式、成功条件を明示したプロンプト
* 実行環境: `codex exec --sandbox read-only`
* 検証スクリプト: `experiments/prompt-master-validation/run_validation_codex.py`
* 生出力: `experiments/prompt-master-validation/outputs_codex/`

結果は次の通りです。

![雑なプロンプトとprompt-master型プロンプトの小さな比較](https://static.zenn.studio/user-upload/d81b6c6c3c3d-20260704.png)  
*同じテーマを生成し、同じルーブリックで採点した今回の検証フロー。*

| 条件 | プロンプト文字数 | 出力文字数 | ルーブリック通過 | 実行時間 |
| --- | --- | --- | --- | --- |
| 雑なプロンプト | 38 | 1778 | 4/6 | 28.26秒 |
| prompt-master型プロンプト | 954 | 1023 | 6/6 | 16.21秒 |

採点では、次の6項目を確認しました。

* 対象ツールごとの振り分けに触れているか
* 成功条件または停止条件に触れているか
* ベンチマーク証拠がないことを書いているか
* 価値を「曖昧さを減らすこと」として説明しているか
* 誇張語を避けているか
* 指定したMarkdown見出しを守っているか

この1回の検証では、prompt-master型のプロンプトのほうが、指定した構成と注意点を守りました。雑なプロンプトでも内容は大きく外れていませんでしたが、指定見出しを満たさず、ベンチマーク証拠への言及も抜けました。

検証で使ったprompt-master型プロンプト

```
あなたは、AI支援開発の実務を日本語のソフトウェアエンジニア向けに説明する技術ライターです。

タスク:
`prompt-master` を使うと、なぜAIの出力が期待に近づきやすいのかを説明する、日本語の技術記事セクションを書いてください。

前提:
以下の事実だけを使ってください。
- prompt-master は、プロンプトを書く前に対象AIツールを検出する。
- prompt-master は、task、target tool、output format、constraints、input、context、audience、success criteria、examples の9項目を抽出する。
- 重要な情報が欠けている場合、最大3つまで確認質問をする。
- Claude、Cursor、Claude Code、Midjourney、Stable Diffusion など、ツールごとの書き方へ振り分ける。
- token-efficiency audit により、出力に効かない文言を削る。
- agentic tool 向けには、作業範囲、禁止事項、停止条件、人間レビュー条件を追加する。
- リポジトリには、性能向上を独立に証明するベンチマークは提供されていない。

制約:
- prompt-master による性能向上が実証済みだとは書かない。
- ソース上の事実と、自分の解釈を分けて書く。
- 「革新的」「圧倒的」「魔法」「必ず」のような誇張語を避ける。
- 実務的で落ち着いた文体にする。

出力形式:
日本語Markdownで、見出しは必ず次の5つだけにしてください。
1. `## 要点`
2. `## 効きやすい理由`
3. `## エージェント利用で特に効く点`
4. `## 注意点`
5. `## 自分で応用するなら`

成功条件:
- 対象ツールごとの振り分けに触れている。
- 成功条件または停止条件に触れている。
- ベンチマーク証拠がないことを書いている。
- 価値を「モデルを魔法のように改善すること」ではなく、「曖昧さを減らすこと」として説明している。
- 900字以内に収める。
```

## ただし、性能向上が実証されているわけではない

注意点もあります。

GitHub リポジトリを見る限り、prompt-master の効果を独立したベンチマークで検証した結果は見当たりません。

そのため、次のようには言えません。

> prompt-master を使えば性能が上がる

より正確には、次のように見るのがよさそうです。

> prompt-master は、AI出力が失敗しやすい原因を減らす設計になっている

入力情報が薄ければ、prompt-master を使っても限界があります。対象ツールの仕様が変われば、合う書き方も変わります。また、AIが苦手なタスクを、プロンプトだけで解決できるとは限りません。

それでも、プロンプトを「なんとなくの依頼」から「確認できる作業指示」に近づける効果はあります。

## 自分で書くときの最小テンプレート

prompt-master を使わない場合でも、次の項目を入れるだけでプロンプトの質は変わります。

```
対象ツール:
タスク:
背景:
制約:
出力形式:
成功条件:
禁止事項:
停止条件:
確認が必要な操作:
```

特に AIエージェントに作業させるなら、`禁止事項`、`停止条件`、`確認が必要な操作` は重要です。

AIに何をさせるかだけでなく、何をさせないか。どこで止まるか。どこから人間に戻すか。

ここまで書くと、プロンプトは依頼文というより、作業指示に近づきます。

## まとめ

prompt-master が期待に近い結果を出しやすい理由は、派手なプロンプト技法にあるわけではありません。

やっていることは、曖昧な依頼を分解し、対象ツールに合わせ、制約を置き、成功条件を決め、不要な語を削ることです。これによって、AIが推測で補う余地を減らしています。

AIの性能が上がるほど、雑な指示でも何かしらの答えは返ってきます。ただ、仕事で使うなら「何か返ってくる」だけでは足りません。

必要なのは、同じ意図を安定して伝えられることです。そのためには、プロンプトを単なる文章ではなく、作業指示として扱う必要があります。

prompt-master は、その考え方を実務で使いやすくするためのエージェントスキルです。
