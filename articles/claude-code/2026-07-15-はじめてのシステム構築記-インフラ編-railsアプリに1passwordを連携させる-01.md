トリビューでWebエンジニアとして、コーポレートエンジニアを兼務しているHirokazu  
（[@hirokazu\_narrgi](https://x.com/hirokazu_narrgi?s=11&t=sOCKBGZAYg5z9JfYaJYriA)）です。

これまで主にバックエンドを触ってきましたが、徐々に領域を広げ、1人でサービス全てを扱えるエンジニアになるため勉強中です。まずはインフラ分野を触りたいと思っていましたが、個人でやるには初期環境から作るのが大変でどうしようか悩んでいました。ちょうどその時、今回のテーマとなる課題が社内で議論されていたので、挑戦したいと手を上げて担当しました。こうして、はじめてのインフラ構築に取り組むことになりました。

トリビューには、自社サービスの運用業務のために共有アカウントを使用することがあります。ただ、そのパスワードを変更する手段は手動しかなく、組織としてローテーションを回す仕組みもありませんでした。今回、こうした共有アカウントのパスワードを1Passwordに集約し、発行からローテーション・削除までを社内システムから自動で行える仕組みづくりの道のりを記事として書かせていただきます。

対応してみると要件の言語化、Kubernetes上でのConnect Serverのセルフホスト、RailsアプリからのAPI連携と、想定以上に幅のあるテーマで、初心者あるあるだと思いますが、何から進めればいいかわからないという状態からスタートとなりました。わからないことが何かを明確にすることから始め、吐き出し、チームメンバーやAI（Claude）と壁打ちを繰り返し、一つずつ丁寧に確認していくことで、要件を固め、技術を選び、実装までやり切ることができました。

「経験が浅い分野の場合、どう進めれば迷わずにやり切れるのか」。同じ悩みを持つ方の、一つの実例になればうれしいです。

## 要件の洗い出し

いきなり製品や実装方法を選ぶのではなく、まず「ありたい状態」を要件として言語化するところから始めました。

トリビューには、設計の判断根拠をDesign Docとして残す文化があります。今回もその形に沿って、AIに壁打ちしながら「何がしたいのか」「何が決まれば前に進めるのか」を一つずつ文章にしていきました。決まったことだけでなく、“まだ決めきれていないこと”も未決事項として列挙し、順に潰しながら進めます。たとえば今回は、こんな疑問をそのまま未決事項として書き出しました。

* パスワードマネージャーはどれにするか
* セルフホストの基盤はどこに立てるか
* 権限はどう設計するか
* ローテーションはどの頻度で回すか
* Vaultはいくつ必要か
* Google Workspaceと連携して、利用者の管理まで自動化できないか

それぞれの着地の詳細を書くと長くなりますので割愛しますが、わからないことを「わからない」と書ける場所があるのは、とても心強いものでした。疑問が浮かんだらすぐに積み、一つひとつチームや関係者に確認しながら、今回のスコープに含めるかどうかまで決めていく。答えは、AIを用いて一次情報から裏をとりながら埋めていき、未決事項が片づくほど、仕様は抜け漏れのない強固なものに変わっていきました。この「先に言語化する」工程は、振り返って一番「やっておいてよかった」と思う部分です。実装が始まってから「これはどうするの？」「ここは考えた？」が出てくることがなく、決めきれていない部分が実装に紛れ込んでプルリクエストが膨らむこともない。どの判断も、理由まで説明できる状態で実装に取り組むことができました。

「ありたい状態」の要件を次の3つとしました。

* 共有アカウントのパスワードは、**パスワードマネージャーに常に最新の状態で同期されている**
* 社内システムから、**APIでアカウントの作成・変更・ローテーション・削除**ができる
* **社内メンバーがパスワードそのものを知らなくてもログインできる**

3つ目が今回の肝です。パスワードを「安全に配る」のではなく、そもそも**人が知らなくていい状態**にする。パスワードは仕組みが自動生成してパスワードマネージャーに登録し、使うときも人手を介さずに参照される。ここまでできて初めて、ローテーションも削除も安心して自動化できます。

![](https://static.zenn.studio/user-upload/7ee096affe8c-20260714.png)

* **API連携ができること。** 社内システムからアカウントの登録・更新・参照をプログラムで実行できることが必須です。ここができないと、そもそも発行やローテーションの自動化が成立しません
* **セキュリティの実績が信頼できること。** 認証情報を預けるサービスである以上、ベンダー自身の過去のインシデントや現在の暗号化アーキテクチャは無視できません
* **運用の変更コストが小さいこと。** 既存の資産や運用フローからの乗り換え負荷は、小さいほど望ましいです

要件、ゴール、前提を全て揃えてから、具体的な製品や実装方法の検討に進みました。

## 実装方法の選定 — Ruby SDKがないためConnect Serverをセルフホスト

製品は、タイトルにもある通り1Passwordです。  
認証情報をアプリから書き込める公式APIを持つパスワードマネージャーは、調べてみると意外と少なく、要件に照らすと候補はほとんど絞られました。また、セキュリティの実績にも懸念がなく、トリビューでの利用実績もあったためです。

次は、Railsアプリからどうやって1Passwordを操作するか。ここでも候補を並べて、要件に照らして一つずつ確認しました。1Passwordが提供する連携手段は、大きく3つあります。

| 連携手段 | 概要 | 今回の評価 |
| --- | --- | --- |
| op CLI | 実行環境にCLIを配置してシークレットを注入 | 常時起動アプリの継続的な読み書きには運用が重い |
| 公式SDK | Go / JavaScript / Python | 当時Ruby版がない（アプリはRails） |
| **Connect Server** | 自社インフラに配置するREST APIサーバー | ◎ 採用。API通信を社内ネットワークに閉じられる |

Connect Serverは、自社インフラに配置するREST APIサーバーです。アプリケーションからはHTTPで叩くだけなので、言語を問いません。「Ruby SDKがない」「CLIの常駐は避けたい」「API通信は社内ネットワークに閉じたい」という条件を並べると、**Connect ServerをセルフホストしてREST APIを叩く**構成が一番シンプルでした。アプリからは社内ネットワーク内のエンドポイントに対して、Connect TokenをBearerトークンとして付けてアクセスします。

次に、Connect Serverをどこに置くかです。インフラ初心者がいきなり挑戦すると良いことがありません。そのため新しい基盤を一から作るのではなく、**すでに運用しているEKS on Fargateのクラスタに載せる**ことにしました。監視・シークレット注入・リリースフローといった運用の仕組みがすでに揃っているからです。外部SaaSの公式コンテナを自社ネットワーク内で常駐させるのはトリビューでは初めてでしたが、既存のパターンに徹底的に寄せることで、新しく面倒を見るものを最小限にできます。新規にECSクラスタを立てる案、EC2にdocker-composeで載せる案、App Runnerを使う案との比較もDesign Docに残したうえで、既存の運用パターンから外れることや、Connect Server公式の2コンテナ（connect-api / connect-sync）＋共有ボリュームという構成との相性から見送っています。

こうして「既存に倣う」進め方ができた背景には、トリビューのインフラがほぼすべて**Terraformでコード化されている**（いわゆるIaC）ことがあります。サーバーやネットワークの構成が、管理画面での操作の積み重ねではなくコードとして残っているので、既存の構成はコードを読めば理解できますし、新しく足すものも既存の書き方に倣えます。インフラ経験の浅い私でも内容を理解しやすかったのは、この土台のおかげです。そしてこれはインフラに限った話ではなく、構成や決まりごとを“コードと文書で残す”方針がトリビューの各所で徹底されているので、どのレイヤーにも読める手本があります。

シークレットの扱いでは、**秘密の値をGitにもTerraform stateにも残さない**ことを徹底しました。AWS Secrets Managerにはあらかじめシークレットだけを用意しておき、認証情報の値は管理者が手動で投入します。Terraformが扱うのはシークレットへの参照だけなので、リポジトリにもIaCの状態ファイルにも秘密値が現れません。

こうして決まった構成の全体像がこちらです。

![](https://static.zenn.studio/user-upload/0d03bae13149-20260714.png)

| コンポーネント | 役割 |
| --- | --- |
| AWS Secrets Manager | Connect Serverの認証情報ファイルの置き場。値は手動で投入 |
| External Secrets Operator（ESO） | Secrets Managerの値をKubernetes Secretへ定期同期 |
| connect-sync | 1Password本体（クラウド）とアイテムを同期 |
| connect-api | クラスタ内からのREST APIリクエストを処理 |
| Railsアプリ（社内システム） | Connect Tokenを使ってアイテムを作成・更新・削除 |

シークレットは次のように流れます。Connect Serverが1Password本体に認証するための認証情報ファイルはAWS Secrets Managerに置き、External Secrets Operator（ESO）が定期的にKubernetesのSecretへ同期します。Connect ServerのPodは公式が提供するconnect-apiとconnect-syncの2コンテナ構成で、このSecretをファイルとしてマウントして起動します。connect-syncが1Password本体とアイテムを同期し、connect-apiがクラスタ内からのリクエストを受けます。Railsアプリも同じクラスタ内で別のPodとして動いており、クラスタ内向けのServiceを通じてconnect-apiのAPIを叩きます。

なお、トークンの権限や保管・ローテーションの方針、可用性の設計といったセキュリティ・運用に関わる詳細は、本記事では触れません。

## それでもハマったところ

方針が決まれば、あとは作るだけです。実装はAI（Claude Code）とのペアプログラミングで進めました。プルリクエストは、クライアント基盤・業務ロジック・環境ガード（1Password操作を本番環境だけに絞る仕組み）の3つに分けて小さく出しています。

クライアントの実装も、ほかの外部サービス連携で使っている社内の“型”に倣うだけで、迷うところはほとんどありませんでした。

トリビューではプルリクエストへのAIレビューが標準で、人間のレビューと二段構えです。それでも、事前の要件や設計だけでは拾いきれない課題は出てきます。幸い、いずれも本番環境へのリリース前に気づくことができました。ここからは、そのハマりどころをインフラとアプリに分けて紹介します。

### 🔷インフラでハマったところ

**ESOで「JSONのファイル」をSecret化するときは、extractしない。**  
ESOの`dataFrom.extract`は、JSONをキーごとに分解してSecretにする機能です。今回の認証情報ファイルはJSON形式の「1ファイル」なので、extractを使うとバラバラのキーに分解され、元のファイルとして復元できなくなってしまいます。`data` + `remoteRef`で「1キー＝1ファイル」として丸ごと格納する必要がありました。

**リソース名は英字始まり（DNS-1035）。**  
KubernetesのService名などは「英字で始まる」必要があり、「1password-…」という名前は数字始まりでエラーになります。検証環境に適用した時点で気づいて付け直しました。製品名を冠した名前を付けたくなる場面では、意外と踏みやすい罠だと思います。

### 🔷アプリでハマったところ

**リトライは「一時的なネットワークエラー」だけに絞る。**  
このクライアントは社内システムのリクエストの中から同期的に呼ばれます。レート制限（429）などを受けてクライアントの中で長く待つと、呼び出し元のほうが先にタイムアウトしてしまいます。そこで自動リトライは接続・読み取りタイムアウトのような一時的なネットワークエラーだけとし、429を含むHTTPエラーはその場で例外化して、待つかどうかの判断は呼び出し元に委ねる方針にしました。

**「マスクする」より「出さない」。**  
トークンや生成したパスワードはログのパラメータフィルタに追加していますが、それだけには頼らず、APIレスポンスの本文そのものをログに出さない設計にしました。レスポンスにはパスワードを含むフィールドがあり得るため、マスクで守るより出さないほうがリスクを根本から断つことができます。ログに残すのはステータスコードなど必要最小限だけです。

## まとめ

共有アカウントの認証情報は1Passwordに集約し、アカウントの発行・ローテーション・削除は社内システムから1Passwordと連動して回っています。パスワードは仕組みが自動生成して登録し、メンバーは1Passwordの自動入力でログインします。発行から利用まで、人がパスワードを目にする必要はなくなりました。

それぞれの章での学びをまとめると:

* **わからないことはわからないままでも出しきってしまう**（要件の洗い出し）
* **目指す状態を明確にして、判断ができるまでの材料を集める**（実装方法の選定）
* **決めて作っても課題は出る。動かして確認し、理解して穴をなくす**（それでもハマったところ）

一番の学びは、**わからないことを一つずつ丁寧に確認しながら進めれば、インフラがほぼ初めてでも、要件を固め、技術を選び、実装までやり切れる**ということです。いきなり公式ドキュメントを読むとわからないことが多いので、まずは疑問点の洗い出しをします。そもそもこの用語の意味は？どうしてここから始めるの？今は全体のどこを扱ってる？これは社内の既存構成の何に影響するの？この繰り返しだけで、わからない部分がどんどん出てきます。わからない部分が明確になればあとは調べるだけです。調べた結果、判断に迷うことが出てきます。そんな時はすぐに相談する。そうすれば止まる場面はほとんどありませんでした。要件の壁打ち、製品の下調べ、設計の整理、実装、コードレビュー。各段階で「確認の相手」としてメンバーやAIがいたことが、足を止めずに進められた理由です。

また、とりあえず作るというのも重要でした。私の場合はイメージがつかないものは理解しづらく、とりあえず作業をするということを徹底しました。完成品を作ってからではなく、パーツごとにレビューをしています。そうするとデータの流れがわかり、それぞれが何をしているのかよくわかります。特に今回は、これまでバックエンド開発では“最初からあるもの”として使っていた認証情報が、どこで生まれていつ必要になるのかが見えた気がします。

ちょうど来期の目標を話す機会があったので、今度はモバイルアプリもやってみたいと相談してみました。モバイルアプリは最も多くのユーザーが触れ、価値提供に直接関われる領域です。これまでお願いする側だった改善を、自分の手でも形にしてユーザーに届けられるようになりたいと思っています。書けそうであれば「モバイルアプリ編」もお届けします。目指せフルスタックエンジニア！
