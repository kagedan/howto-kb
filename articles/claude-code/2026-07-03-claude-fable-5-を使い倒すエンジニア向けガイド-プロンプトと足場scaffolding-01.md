Claude Fable 5 の話題は「ベンチマークが高い」ばかりが先行しがちですが、実際に日々のエージェント開発で効いてくるのは**プロンプトと足場（scaffolding：モデルを動かす周辺の仕組み）の作法が変わった**という点です。旧モデル（Opus 4.8 など）の感覚のまま投げると、むしろ品質を落とすケースすらあります。

この記事は、Anthropic 公式の [Prompting Claude Fable 5](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5) を軸に、**エンジニアが今日から効かせられる勘所**だけを抜き出して整理したものです。入門記事ではないので、Claude Code や API でエージェントを組んでいる前提で進めます。

![Claude Fable 5 エンジニア向けガイド](https://static.zenn.studio/user-upload/deployed-images/adad8e348c43a2cbfc11fd5b.png?sha=6fa846b4b186e34760f29e68b2e35085e0a65334)

## 1. 効きを回すのは `budget` じゃなく `effort`

まず、ネット上に出回っている「Fable 5 は `budget_tokens` で思考量を 4000〜6000 くらいに…」という類の解説は**そのまま真に受けないほうがいい**です。公式ドキュメントによると、Fable 5 は **adaptive thinking（適応的思考）のみ**で、**拡張思考の予算指定（budget\_tokens）は廃止**されています。

代わりに、知能・レイテンシ（応答の遅さ）・コストのトレードオフを回す主役は **`effort`（努力度）パラメータ**です。

![effort の段階と使い分け](https://static.zenn.studio/user-upload/deployed-images/5f89fc62a26b82dd5b551003.png?sha=cb88011a46137420e213a6d1e6c37537ff87754f)

* **`high` を既定**にしておくのが無難。迷ったらこれ。
* 最も精度が要る仕事だけ **`xhigh`**。
* 定型作業やインタラクティブに速く回したいときは **`medium` / `low`**。

面白いのは、**Fable 5 の低 effort でも、旧モデルの xhigh を上回ることが多い**という点。むやみに上げる必要はありません。「タスクは完了するけど遅い」と感じたら、まず effort を下げるのが正解です。

なお高 effort では、頼んでいない整形やリファクタまで踏み込みがち。抑えたいなら、挙動を一つずつ禁止するのではなく、短くこう添えるだけで効きます（公式の例より要約）。

> バグ修正に周辺の掃除は要らない。ワンショットの処理にヘルパーは要らない。仮の将来要件のために設計しない。動く最小限をやれ。

## 2. 旧モデル向けの指示は、むしろ「消す」

ここがいちばん見落とされる移行ポイントです。Fable 5 は**指示追従（instruction following）が強化**されているため、挙動をひとつずつ列挙しなくても、短い一言でだいたい steer（誘導）できます。

裏を返すと、**旧モデル向けに盛りに盛った長大なプロンプトや Skill は、Fable 5 ではむしろ出力品質を下げることがある**。公式も「古い指示は見直して、既定挙動で足りるなら削れ」とはっきり書いています。

![旧モデル向けの過剰な指示はむしろ品質を下げる](https://static.zenn.studio/user-upload/deployed-images/ad6d331b94476a7f64122a87.png?sha=a208f8ec1053a751d09ee776e9cf5d4b1f94f202)

たとえば「冗長に説明しない・箇条書きにしすぎない・PR説明を盛りすぎない…」と個別に並べる代わりに、こう一文入れるだけで同等の効果が出ます。

> まず結論から。最初の一文は「何が起きたか／何が分かったか」に答える。詳細と理由はその後。読みやすさと簡潔さは別物で、読みやすさを優先する。

移行作業としては、**既存の system プロンプトや Skill を「引き算」の目で監査する**のがおすすめ。足すより削るほうが効くフェーズに入りました。

## 3. 「思考を本文に書かせる」指示は地雷

これは知らないとハマります。Fable 5 には\*\*安全分類器（危険な出力を判定するフィルタ）\*\*が入っていて、そのひとつが `reasoning_extraction`（モデルの内部推論の抽出）カテゴリ。

つまり、**「考えた過程を本文に全部書き出して」「reasoning をレスポンスに転写して」といった show-your-thinking 系の指示は、この拒否カテゴリを踏み、結果として Opus 4.8 への自動フォールバックが増えます**。

![推論を本文に書かせる指示は拒否カテゴリを踏む](https://static.zenn.studio/user-upload/deployed-images/c27b7f816622e688c9a90e7e.png?sha=7046744f57e05ead74f03d32072454a1a3531eb4)

推論の可視化が必要なら、本文に書かせるのではなく **adaptive thinking の `thinking` ブロックを読む**のが正しい作法。移行時は、既存の Skill / system プロンプトに「自分の思考を説明しろ」系の指示が紛れていないかを一度監査しておきましょう。地味に効きます。

## 4. 長時間ランを前提に足場を組み替える

Fable 5 は 1 リクエストが**数分〜数時間**走ることがあります（長丁場のエージェント運用）。ここでコード側の足場が旧来のままだと事故ります。

* **クライアントの timeout・streaming・進捗表示を先に直す。** ブロッキングで待つのをやめて、スケジュールジョブなどで**非同期にチェック**する構成に寄せる。
* **進捗の「捏造」対策**。長時間ランだと、モデルが実際のツール結果と食い違う進捗を報告することがあります。公式はこう指示すると激減したとのこと。

> 進捗を報告する前に、各主張をこのセッションのツール結果に照らして監査せよ。証拠を指し示せる仕事だけ報告し、未検証なら未検証と明言せよ。

* **勝手なアクションの抑制**。頼んでいないメール下書きや、防御的な git ブランチ作成をやることがあるので、境界を明示します。

> ユーザーが問題を説明している・質問している・考えを口に出しているだけのときは、成果物は「あなたの所見」だ。所見を報告して止まれ。頼まれるまで修正を当てるな。

## 5. 並列サブエージェント & メモリを積極的に

Fable 5 は**サブエージェントの派遣（delegation）が旧モデルより得意**で、投げやすくなっています。

* 独立したサブタスクは**どんどんサブエージェントに委譲**し、走らせたまま本体は次を進める。オーケストレータとサブは**非同期通信**で。長寿命のサブはコンテキストを持ち越せるので、キャッシュ読みでコストと時間を削れます。
* **検証は「自己批判」より「別のサブエージェント」が強い**。fresh context（まっさらな文脈）の検証役サブに、仕様に照らして自分の成果をチェックさせるのが公式推奨。
* **メモリシステムを持たせる**。1 教訓 = 1 ファイル、先頭に要約 1 行。過去ランの学びを Markdown に貯めて参照させると、Fable 5 は特に伸びます。

## 6. 使いどころ：Sonnet と Fable 5 の線引き

全部 Fable 5 で殴るのはコスト的に下手です。ざっくりの判断基準はこう。

![Sonnet と Fable 5 の使い分け](https://static.zenn.studio/user-upload/deployed-images/e7a05d2578664e7a3688e5c9.png?sha=015066b7620a7bd2d5bb67b77edfdf37696acd8a)

* **単一ターンで終わる／5 ステップ以下**の作業は、軽いモデル（Sonnet など）で十分。
* **10 ステップ以上を自律で回す／失敗の代償が大きい**仕事は Fable 5。
* **model mixing**（推論の重い所だけ Fable 5、実行・整形は軽いモデル）でコストを大きく落とせます。

そして評価するときの鉄則が **「難易度レンジの“上”から試す」**。簡単なタスクだけで測ると Fable 5 を過小評価します。人間なら数時間〜数日かかる仕事（大規模移行、複数段の分析、複雑なエージェントフロー）でこそ地力が出ます。

## 7. 安全分類器を踏むことがある

最後に運用上の注意。Fable 5 の安全分類器は、**攻撃的サイバーセキュリティ・生物/生命科学・思考抽出**を対象にしています。厄介なのは、**良性の作業（普通のセキュリティ実装や有益なライフサイエンス処理）でも稀に引っかかる**こと。

対策はシンプルで、**サーバ側 / クライアント側で Opus 4.8 へのフォールバックを設定**し、`stop_reason: "refusal"` を握って自動で流す。ここを組んでおけば、たまの拒否で運用が止まりません。

## まとめ

Fable 5 に乗り換えるときの勘所を一枚で言うと、**「賢くなった分、足場と指示を“引き算”しろ」** です。

* 思考量は `budget` じゃなく **`effort`**（budget\_tokens は廃止）
* 旧モデル向けの過剰な指示・Skill は**むしろ削る**
* **show-your-thinking 系の指示は拒否を踏む**ので外す
* 長時間ラン前提で **timeout / 非同期 / 進捗監査**を組み直す
* **並列サブエージェントと外部メモリ**を積極活用
* **Sonnet と Fable 5 を使い分け**、評価は難易度の上から
* **Opus 4.8 フォールバック**を必ず用意

「新モデルだから今までのプロンプトを盛ろう」ではなく、「賢くなったから減らせる」方向に発想を切り替えるのが、Fable 5 を使い倒す近道です。
