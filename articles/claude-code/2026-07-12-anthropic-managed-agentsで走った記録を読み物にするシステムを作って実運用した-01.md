# Anthropic Managed Agentsで「走った記録を読み物にする」システムを作って実運用した話

## 何を作ったか

自転車で走った記録(GPX)を投げると、AIエージェントがライドレポート——数値の羅列ではなく、その日の走りを綴った読み物——を生成するシステムを個人で作り、実運用している。

まず現物を見てもらうのが早い。

動機は単純で、「走った記録を、ただの数字じゃなく読み物として残せたら面白いのに」と思ったからだ。Stravaに残るのは距離・獲得標高・速度と地図。正確だが、半年後に読み返して面白いものではない。その日の走りを、文脈と物語のあるレポートとして残したかった。

この記事では、システムの構成と、Anthropic Managed Agents を実運用して得た知見——特に **Memory Store が3回のライドをまたいで「育った」体験**と、**実測コスト(1ライドあたり$1.24)**——を書く。

## システム全体像

バックエンドは FastAPI を GCP Cloud Run に載せた構成。GPX を受け取るとジョブとして登録し(状態は Firestore)、エージェントによるレポート生成は非同期で走る。生成されたダッシュボード(HTML)は Cloud Storage に置かれ、完成すると Discord に距離・獲得標高・走行時間つきの通知が届く。失敗したときも、エラー概要つきの通知が飛ぶ。通知はベストエフォートで、万一 Discord 側が落ちていてもレポート生成自体は巻き込まれない設計にしている。

スマホで走行を記録して、帰宅後に GPX を投げ、風呂から上がる頃には Discord にレポートの URL が届いている——という運用になっている。

## Managed Agents の構成と、役割分担の設計判断

Managed Agents は、エージェント本体(system prompt)に加えて、Skills(ドキュメントとして与える知識・規約)、Memory Store(会話をまたいで持続する記憶)、custom tools(バックエンドの API をエージェントに使わせる)を組み合わせられる。

このシステムでは、試行錯誤の末に次の役割分担に落ち着いた。

**処理ロジックは system prompt、デザイン規約は Skill。** レポート生成の手順(GPX の解析結果をどう解釈し、どの順で何を生成するか)は system prompt に書く。一方、ダッシュボードの見た目に関する規約(配色、レイアウト、フォントなどのデザイントークン)は Skill に切り出す。ロジックと見た目を別の場所に置くことで、デザインだけ直したいときに system prompt を触らずに済む。

**指示は「番号付きの必須手順」で書く。** これは実運用で一番効いた学びかもしれない。当初、system prompt に独立したセクション見出しで指示を書いていたが、エージェントに無視されることがあった。**番号付きの MANDATORY 手順(「以下を必ずこの順で実行する: 1. …、2. …」)に書き換えたところ、安定して守られるようになった。** エージェントへの指示は、散文的な規約より、手順書の形が通る。

custom tools としては、地名解決(geocode)などのバックエンド API に加えて、**render\_screenshot** というツールを持たせている。仕組みは分業になっていて、**撮るのはホスト側**(バックエンドが Playwright + Chromium でエージェントの生成した HTML をレンダリングして PNG 化)、**見て判定するのはエージェント自身**(返された画像を見て、Skill に定義したチェックリスト——地図マーカーの重なり、テキストの見切れ、異常な縦伸び、ポリラインの妥当性——を適用する)。生成物の検証を人間ではなくエージェント自身にやらせる、という発想だ。

ここで効いたのが**回数のハードキャップ**だ。自己チェックは「1回撮って確認し、NG なら HTML を修正して1回だけ再確認。合計2回まで、3回目は禁止」と system prompt に明記した。エージェントに自己チェックをさせると、放っておけば「念のためもう一度」を繰り返してターン数(=コスト)が膨らむ。**自己修正は有界にする**——実運用でエージェントを回すなら、これは効く。

## Memory が育って、エージェントが「相棒」になった

このシステムで一番面白かったのは、技術構成でもコストでもなく、**Memory Store が3回のライドをまたいで育っていく過程**だった。

舞台は鍋谷峠。大阪側から和歌山へ抜ける、関西のサイクリストにはおなじみのヒルクライムだ。私はここに3回挑んだ。

**1回目(R1)は、時間切れだった。** 蔵王峠を和歌山側へ降りた時点で午後5時を回っていて、鍋谷峠の登りにかかる頃には道が暗くなり始めた。これ以上は無理だと判断して和歌山側へ引き返し、和歌山市まで走って輪行で帰った。標高712m地点での撤退。脚ではなく、時間のマネジメントの失敗だった。

ライド後、GPX を投げると、エージェントはこの顛末をレポートにまとめ、Memory Store に「ライダーは鍋谷峠に挑戦して712m地点で引き返した」という趣旨の記録を自分で書き込んだ。私が書けと指示したわけではない。Memory Store を有効にしたエージェントは、レポート生成のついでに、ライダーについて覚えておくべきことを自分で選んで書く。

**2回目(R2)は、もっと悔しい敗退だった。** 再挑戦のつもりが、手持ちのサイコン(Bryton)が提示したルートの道が実在せず、山中で行き場を失った。そのまま和泉葛城山を下り、また和歌山側から輪行で帰った。この失敗も、エージェントは覚えた。

(余談だが、このときの「サイコンのルート案内に裏切られた」経験は、のちに自作サイクルコンピュータを作る動機の一つになる。それはまた[別のリポジトリ](https://github.com/FujiiNoritsugu/cycling_computer)の話。)

**3回目(R3)で、ようやく攻略した。** そしてこのときのレポートに、指示していない一節が現れた。

> 初回の反省を活かしてエコペース

エージェントは、R1の敗因(時間切れ)を記憶していて、R3の走行データ(序盤を抑えたペース配分)をそれと結びつけ、「今回は反省を活かした走りだった」という**3回の挑戦をまたいだ物語**として、レポートを書いたのだ。レポートには、R1の712m撤退、R2のルート消失も踏まえて「3回目での攻略成功」であることが、自然な文章で織り込まれていた。

単発の GPX 解析なら、R3 は「獲得標高◯m、平均速度◯km/h の良いライドでした」で終わる。Memory があることで、同じデータが「三度目の正直の物語」になった。これこそが、このシステムで作りたかった「記録が読み物になる」の実現形だった。

ちなみに、Memory Store の中身は覗ける。エージェントが自分で書いた rider\_profile を読むと、そこには「エージェントから見た私」が文字になっている。自分の走りが、自分ではない視点から記述されているのを読むのは、少し不思議で、悪くない体験だ。

## 実運用コスト: 1ライドあたり $1.24

「作ってみた」で終わらせず運用しているので、コストの実測値を書いておく。**1ライドのレポート生成にかかるコストは、実測でおよそ $1.24** だった。

内訳で支配的だったのは、生成(出力)ではなく**キャッシュ読込トークン**だ。長い system prompt、Skills、Memory の内容がコンテキストとしてエージェントの各ターンに乗るため、入力側のトークンがコストの主役になる。逆に言えば、prompt caching が効いているからこの金額で収まっている(キャッシュ読込は通常入力よりはるかに安い)。長いコンテキストを持つエージェントを実運用するなら、コスト構造は「何を生成するか」より「毎ターン何を読ませるか」で決まる——というのが実測から得た感覚だ。

モデルは Sonnet を使っている。レポート生成という用途には十分な品質で、この用途に最上位モデルは要らないと判断した。月に数回のライドなら月あたり数ドル。趣味の実運用としては完全に許容範囲に収まっている。

## 運用して得た細かい知見

* **SDK の属性名を推測で書かない。** Managed Agents の SDK はまだ情報が少なく、レスポンスの属性名などを「たぶんこうだろう」で書くと外れる。ソースコードを読んで確認するのが結局一番速かった。
* **生成物の検証をエージェント自身にやらせる。** 前述の render\_screenshot がそれで、「HTML を生成して終わり」ではなく「自分の生成物を見て確認する」ところまでをエージェントの仕事にした。
* **エージェントの自己チェックは有界にする。** render\_screenshot は最大2回のハードキャップ付き。無限の「念のため確認」はターン数=コストに直結する。
* **開発は Claude Code との分業で進めた。** 各ステップで停止して報告させ、人間が承認してから次へ進む運用。エージェント(実行系)とチャット(戦略系)で役割を分けると、破壊的な操作の事故を防ぎながら速度が出る。

## まとめ

このレポート生成システムは、単体ではなく3つのリポジトリからなるシステム群の一部として作っている。

Memory を持ったエージェントは、記録を蓄積するほど、その人の文脈で語れるようになる。「三度目の正直」を勝手に物語にしてくれた瞬間、このシステムは道具から相棒に半歩近づいた気がした。
