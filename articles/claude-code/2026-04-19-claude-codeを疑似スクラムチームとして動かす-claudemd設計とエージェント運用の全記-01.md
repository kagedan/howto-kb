## はじめに

Claude Codeは「1人のAIアシスタント」として使うだけでなく、**チームのメンバー全員として使う**ことができます。

本記事では、1つのリポジトリに対して**Product Owner・Scrum Master・Developerx2・QA**の5ロールをClaude Codeに担わせ、実際にPyQt5製AIエディタを3スプリントかけて開発した経験をまとめます。

> 「CLAUDE.md に何を書けば、Claude Code はチームっぽく動くのか？」
>
> その答えを、設計思想・実際のファイル構成・運用の流れとともに解説します。

---

## なぜ疑似スクラムなのか

### 通常の使い方の限界

Claude Codeを普通に使うと、1会話の中で「設計→実装→テスト→レビュー」を全部やってしまい、**誰も止めてくれない**という問題が起きます。

* バックログを勝手に変える
* レビューなしでmainにマージする
* 別タスクのコードを副作用で壊す

これをロール制約で防ぐのが、疑似スクラム構成の狙いです。

### ロール＝権限分離＝安全ネット

```
PO       →  backlog.md のみ書き込み可
SM       →  sprint-current.md, impediments.md のみ
Dev      →  src/ のみ（バックログ変更禁止）
QA       →  sprint-current.md のレビュー列のみ（実装コード修正禁止）
```

Claude Codeは指示されれば何でもやろうとしますが、**CLAUDE.md に禁止事項を明記しておくと、自分でブレーキをかけるようになります**。

## ディレクトリ構成

```
project-root/
├── CLAUDE.md              ← チーム憲法（全エージェントが毎回読む）
├── memory/                ← 全エージェント共有メモリ
│   ├── backlog.md         ← PO管理（プロダクトバックログ）
│   ├── sprint-current.md  ← SM・Dev管理（進行中スプリント）
│   ├── impediments.md     ← SM管理（障害リスト）
│   ├── standup-log.md     ← SM管理（スタンドアップ記録）
│   ├── sprint-history/    ← 完了スプリントのアーカイブ
│   └── retrospectives/    ← レトロスペクティブ記録
└── src/                   ← Dev管理（実装コード）
```

`memory/` 配下が**全エージェントの共有メモリ**です。Claude Codeが会話をまたいでも状態を保てるのは、このファイル群があるからです。

## CLAUDE.md の設計

これが疑似スクラムの核心です。

```
# 疑似スクラムチーム — Claude Code エージェント構成

## プロジェクト概要
このリポジトリはClaude Codeエージェントによって運営される疑似スクラムチームです。
各会話の冒頭で自分のロールを確認し、そのロールの責務と権限に従って行動してください。
```

冒頭の「**各会話の冒頭で自分のロールを確認し**」が重要です。Claude Codeは会話ごとにCLAUDE.mdを読むので、毎回ロールを自己認識させます。

### ロール定義

```
### Product Owner (PO)
- **責務**: `memory/backlog.md` の管理、ユーザーストーリーの受付と優先度付け
- **書き込み可能ファイル**: `memory/backlog.md`
- **読み取り可能ファイル**: すべて
- **禁止事項**: コードの実装、`src/` 配下の編集

### Scrum Master (SM)
- **責務**: スプリント進捗監視、障害除去、デイリースタンドアップ司会
- **書き込み可能ファイル**: `memory/sprint-current.md`, `memory/impediments.md`, `memory/standup-log.md`
- **禁止事項**: バックログの優先度変更、`src/` 配下の編集

### Developer Agent (Dev-1 / Dev-2)
- **責務**: スプリントバックログのタスク実装、テストコード作成
- **書き込み可能ファイル**: `src/` 配下、`memory/sprint-current.md`（担当タスクのステータスのみ）
- **禁止事項**: バックログの変更、他エージェントのタスクへの干渉

### QA/Reviewer Agent
- **責務**: コードレビュー、テスト実行、品質基準の維持
- **書き込み可能ファイル**: `memory/sprint-current.md`（レビュー列のみ）
- **禁止事項**: 実装コードの直接修正（修正指示のみ許可）
```

**設計のポイント：禁止事項を必ず書く**

「できること」だけ書いても Claude Code はあいまいな境界を超えてきます。「**できないこと**」を明示することで、越権行為を自分で検知して止まるようになります。

### スプリントルール

```
## スプリントルール
- スプリント期間: 1週間（7日）
- ストーリーポイント上限: 20pt/スプリント
- Definition of Done: コードレビュー通過 + テスト通過 + sprint-current.md のステータスが DONE

## タスクステータス遷移
TODO → IN_PROGRESS → REVIEW_PENDING → DONE / NEEDS_REVISION
```

ステータス遷移を明記しておくことで、Claude Codeが自分でステータスを適切に更新します。

### ファイル更新規約

```
## ファイル規約
- `memory/` 配下のファイルは全エージェントが読む「共有メモリ」
- 更新時は末尾にタイムスタンプとロール名を記録すること
- 作業前に必ず対象ファイルを読み込むこと
```

「**作業前に必ず対象ファイルを読み込むこと**」は特に重要です。これにより Claude Code は他のエージェントが書いた最新状態を確認してから動きます。

### Git規約

```
## Git規約
- ブランチ命名: `feature/sprint{N}-{task-id}`
- コミットメッセージ: `[Dev-{N}] {タスクID}: {タスク説明}`
- PRはQA Agentのレビュー完了（APPROVED）後にmainへマージ
```

### カスタムコマンド（スキル）

```
## カスタムコマンド一覧
| コマンド           | 実行タイミング     | 主なロール |
|------------------|----------------|---------|
| /sprint-planning | スプリント開始時   | PO → SM |
| /standup         | 毎朝（Cron自動化）| SM      |
| /review          | 実装完了後        | QA      |
| /retro           | スプリント終了時   | SM      |
```

これらは Claude Code の\*\*スキル（`/` コマンド）\*\*として `.claude/commands/` に定義しておきます。呼び出すだけで、そのロールとして振る舞い始めます。

## 共有メモリの設計

### backlog.md（PO管理）

```
# Product Backlog
最終更新: 2026-04-18 by Product Owner

| ID     | タイトル              | 優先度 | ポイント | ステータス  |
|--------|--------------------|--------|----------|------------|
| US-101 | UI基盤・画像キャンバス | P1     | 5        | IN_SPRINT  |
| US-102 | ツールバー           | P1     | 2        | IN_SPRINT  |
| US-103 | 背景除去             | P1     | 4        | IN_SPRINT  |
| US-104 | 生成塗りつぶし        | P2     | 11       | IN_SPRINT  |
```

**タイムスタンプとロール名**を末尾に残すことで、「誰がいつ変更したか」がファイルだけで追えます。

### sprint-current.md（SM・Dev共同管理）

```
# Sprint 3

- **開始日**: 2026-04-18
- **終了日**: 2026-04-25
- **スプリントゴール**: 生成塗りつぶし・高解像度化・初回DLウィザードを完成させる
- **総ポイント**: 18pt

## スプリントバックログ

| タスクID | タスク説明          | 担当  | ステータス | レビュー           |
|---------|-------------------|-------|------------|------------------|
| T-301   | マスク描画ブラシUI  | Dev-1 | DONE       | ✅ APPROVED       |
| T-302   | プロンプトダイアログ | Dev-1 | DONE       | ✅ APPROVED       |
| T-303   | SD Inpaint推論     | Dev-2 | DONE       | ✅ APPROVED       |
```

このファイルがチームの「ホワイトボード」です。Devはステータスを `IN_PROGRESS` → `REVIEW_PENDING` → `DONE` と更新し、QAはレビュー列だけを書きます。

### impediments.md（SM管理）

```
| ID      | 発見日       | 内容                              | 影響タスク    | ステータス |
|---------|------------|----------------------------------|------------|----------|
| IMP-001 | 2026-04-18 | PyQt5未インストール、テスト実行不可 | T-001〜T-006 | 解決中    |
```

実際にSprint 1でこの障害が発生し、SMロールのClaude Codeがパッケージインストール方法を調べて即日解消しました。

## 実際のスプリントの流れ

### Sprint Planning（/sprint-planning）

月曜の朝に `/sprint-planning` を呼び出すと：

1. **POとして** `memory/backlog.md` を読み、優先度の高いストーリーをピックアップ
2. **SMとして** ポイント合計が20pt以下になるよう調整し、`memory/sprint-current.md` を作成
3. スプリントゴールを宣言して終了

```
[PO] Sprint 3のバックログを確認します...
     US-104（11pt）・US-105（5pt）・US-106（2pt）= 18pt、上限内です。

[SM] sprint-current.md を作成しました。
     スプリントゴール: 生成塗りつぶし・高解像度化・初回DLウィザードの完成
```

### デイリースタンドアップ（/standup）

毎朝 `/standup` を呼ぶと、SMとして3点を確認します：

1. `memory/sprint-current.md` のステータスを読んで昨日の進捗を報告
2. 今日の予定タスクを確認
3. `memory/impediments.md` に未解決の障害があれば共有

### 実装（Dev-1 / Dev-2として）

ユーザーが「UIをAdobe Premiere風に改善してほしい」と依頼すると：

1. `memory/sprint-current.md` を読んで担当タスクを確認
2. `src/` 配下の対象ファイルを読み込む
3. 実装してテストを書く
4. ステータスを `REVIEW_PENDING` に更新

```
# Dev-1が実装した例：ブラシ硬さ（ソフトエッジ）
def _draw_brush(self, scene_pt, erase):
    ...
    if self._brush_hardness >= 1.0:
        # ハードブラシ：円内を一律255
        self._mask_arr[y0:y1, x0:x1][dist_sq <= r*r] = 0 if erase else 255
    else:
        # ソフトブラシ：ガウス減衰でエッジをなめらかに
        sigma = max(r * (1.0 - self._brush_hardness * 0.85), 0.5)
        alpha = np.exp(-dist_sq / (2.0 * sigma**2))
        alpha[dist_sq > r*r] = 0.0
        stroke = (alpha * 255).astype(np.uint8)
        self._mask_arr[y0:y1, x0:x1] = np.maximum(
            self._mask_arr[y0:y1, x0:x1], stroke
        )
```

### レビュー（/review）

`/review` を呼ぶとQA Agentとして：

1. `REVIEW_PENDING` のタスクを `sprint-current.md` で確認
2. 実装コードを読む（`src/` を読み取り専用で参照）
3. `APPROVED` または `NEEDS_REVISION` をレビュー列に記録
4. `NEEDS_REVISION` の場合は修正指示のみ残す（自分では直さない）

Sprint 1では `_build_ui()` が55行を超えていたため、QAが `NEEDS_REVISION` を出し、Devが3メソッドに分割して即日解消しました。

### レトロスペクティブ（/retro）

```
# Sprint 1 レトロスペクティブ

## Keep ✅
- タスク分解が適切でスムーズに実装できた
- QAレビューで関数サイズ違反を検知し即座に修正できた
- 13テスト全パスで品質を担保できた

## Problem ❌
- PyQt5が事前にインストールされておらず、テスト実行でブロックされた
- _build_ui()が50行超過、レビュー基準を事前に意識できていなかった

## Try 🔄
- Sprint開始前にrequirements.txtをインストールするチェックリストを設ける
- 実装中にリアルタイムで関数サイズを意識する（1メソッド = 1責務）
```

レトロの内容を `memory/retrospectives/` に保存しておくと、次スプリントの `/sprint-planning` 時にSMが読み込んで改善アクションを反映します。

## 運用して気づいた設計のコツ

### 1. 禁止事項は「書く」より「具体的に書く」

❌ 悪い例：「他のロールの仕事をしないこと」

✅ 良い例：「バックログの変更禁止、`src/` 配下の編集禁止」

Claude Code は曖昧な禁止を無視することがあります。ファイルパスや操作名で具体的に書くと守られます。

### 2. 共有メモリはファイルに書く、会話に残さない

会話のコンテキストはスプリントをまたぐと失われます。`memory/` ファイルに書いておくことで、どの会話でも同じ状態から始められます。

### 3. タイムスタンプとロール名を残す

```
更新: 2026-04-18 by Scrum Master
```

これにより「なぜこのデータになったか」が後から追えます。git log と組み合わせると特に有効です。

### 4. Definition of Done を定量的に書く

```
Definition of Done:
コードレビュー通過 + テスト通過 + sprint-current.md のステータスが DONE
```

「コードレビュー通過」だけでは Claude Code が自己承認することがあります。**3条件すべて**を明示することで、1エージェントが全部やってしまう問題を防げます。

### 5. カスタムコマンドでロール切替をスムーズに

`/standup`・`/review`・`/retro` をスキルとして定義しておくと、ユーザーは呼び出すだけで適切なロールに切り替わります。毎回「SMとして振る舞ってください」と書く必要がありません。

## まとめ

Claude Codeを疑似スクラムチームとして動かすための要素をまとめます。

| 要素 | 内容 |
| --- | --- |
| **CLAUDE.md** | ロール定義・権限・禁止事項・ルールの「チーム憲法」 |
| **memory/** | 会話をまたいで状態を共有する「チームの記憶」 |
| **カスタムコマンド** | ロール切替を自動化する「儀式のスクリプト」 |
| **禁止事項の明記** | 越権行為を防ぐ「ガードレール」 |
| **タイムスタンプ記録** | 変更の追跡を可能にする「監査ログ」 |

「1人で全部やる万能AI」より「役割が決まったチームAI」のほうが、実際の開発では安定・安全です。ぜひ `CLAUDE.md` のロール設計を試してみてください。

## リポジトリ構成テンプレート

この構成はそのままテンプレートとして使えます。

```
.
├── CLAUDE.md
├── memory/
│   ├── backlog.md          # PO: ユーザーストーリー管理
│   ├── sprint-current.md   # SM+Dev: 進行中スプリント
│   ├── impediments.md      # SM: 障害リスト
│   ├── standup-log.md      # SM: スタンドアップ記録
│   ├── sprint-history/     # アーカイブ
│   └── retrospectives/     # レトロ記録
└── src/                    # Dev: 実装コード
```

`CLAUDE.md` にロール定義を書き、`memory/backlog.md` にストーリーを1件追加するだけで、疑似スクラムチームとしてのClaude Codeが動き始めます。
