それでも毎週、AWSアップデートの解説動画をYouTubeに配信しています。

!

**この記事の結論**

* 動画編集ができなくても、動画は**コードで**作れる（Remotion + ffmpeg + Python）
* LLMをバッチに組み込むコツは「**非決定性を最小の範囲に閉じ込める・境界で多段検証する（機械チェック＋別LLMによるファクトチェック）・公開判断は人間に残す**」の3点

今回は「**動画生成をAWSの上でどう動かすか**」に絞ります。キャラクター素材の作り方・リップシンク合成の実装詳細・Remotionコンポーネントの書き方は別記事に譲ります。

毎週日曜18時にバッチが起動し、「今週のAWSアップデート」をキャラクターが解説する動画ができあがります。字幕・アップデートカード・BGM/効果音・口パクまで、すべてコードで合成しており、動画編集ソフトは一切登場しません。**動画編集ができないなら、動画をコードで書けばいい。** Reactコンポーネントとして動画を宣言的に書けるRemotionというフレームワークのおかげで、これが本当にできてしまいます。

## 全体アーキテクチャ：LLMは5フェーズ中1つだけ

![](https://static.zenn.studio/user-upload/2d763deb7507-20260705.png)

| フェーズ | 処理 | LLM関与 |
| --- | --- | --- |
| phase0 | `claude -p` スモークテスト（ハング検知・fail fast） | 疎通確認のみ |
| phase1 | What's New RSS を前週日〜土の7日分取得・マージ | なし |
| phase2 | **台本生成 → 構造バリデーション → ファクトチェック → S3保存** | **★ここだけ** |
| phase3 | 音声合成 → props生成 → Remotionレンダー＋リップシンク合成 | なし |
| phase4 | final.mp4 をS3保存 → SNS成功通知 | なし |

**唯一LLMを呼ぶのはphase2**です。RSSで取得したアップデート一覧から`claude -p`に台本（episode JSON）を生成させ、構造バリデーションと後述のファクトチェックの両方を通ったものだけS3に保存して次のフェーズへ進みます。それ以外のフェーズ（RSS取得・音声合成・レンダリング・保存・通知）は全部決定的なPythonスクリプトです。

## なぜLambdaではなくAWS Batchなのか

![](https://static.zenn.studio/user-upload/d5645b0eaed1-20260705.png)  
音声合成〜レンダー〜口パク合成等の処理が入り１動画30分コースとなり、Lambdaの15分制限では門前払いです。レンダリングでCPU/メモリをガッツリ使うため、ジョブ定義で**16vCPU/32GB**と柔軟に指定でき、週1回しか動かないのでコンテナ常駐の無駄もありません。

ジョブのタイムアウトは3時間で、インフラ起因のエラーだけ自動リトライし、台本のバリデーション失敗のようなアプリ側の失敗は即座に`EXIT`させます。

## 各コンポーネントの構成

必要な環境（Node/Python/ffmpeg/Remotion/Claude Code CLI＋キャラクター素材一式）を詰め込んだイメージを**ECR**にプッシュし、コンテナ起動時は`entrypoint.sh`が持ち物検査だけして本体の`run_pipeline.py`にバトンを渡します（ロジックは全部Python側）

```
ScheduleExpression: cron(0 18 ? * SUN *)
ScheduleExpressionTimezone: Asia/Tokyo
```

このスケジュールがBatchジョブを1本起動します。認証情報は**Aivis**のAPIキーと**Claude Code**の長期OAuthトークンの2つで、どちらも**Secrets Manager**に格納。Claude CodeもAivisもサブスクプラン（定額）の長期トークン\*\*で運用しており、音声生成も台本生成もファクトチェックも呼び出すたびの追加費用はかかりません。

## 設計の勘所：非決定性を1フェーズに閉じ込める

![](https://static.zenn.studio/user-upload/fd881dc3c6b5-20260705.png)  
このパイプラインの最大の特徴は、途中に**非決定的なLLM**が挟まっていることです。同じRSSを入力しても台本の切り口は毎回変わりますが、これはむしろ面白い——問題は**制御不能なガチャにしないこと**です。**LLMという非決定的なワーカーを台本生成フェーズ（phase2）だけに閉じ込め、その前後は全部決定的なコードで固める**こと。

アップデート内容の採用基準・使用できるアイコンやモーションのwhitelist・口調や件数といった原稿ルールなど、ブレてほしくない部分を明文化して固定しています。  
ただし、どれだけ丁寧に指示しても本当に守られている保証はありません。  
出てきた結果を境界で検証する仕組みが必要です。それが次の章の話です。

## 検証は3段構え：機械バリデーション、LLMファクトチェック、そして人間

![](https://static.zenn.studio/user-upload/d7762a5d8621-20260705.png)  
LLMは平気で間違えます。RSSのタイトルだけから内容を"推測"で膨らませるケースは特に危ない。このパイプラインでは、検証を3段構えにしています。

### 1段目：機械バリデーション（fail fast）

台本が生成されたら、まず機械的に検査します。JSONとして壊れていないか、アイコンやモーションがwhitelistに含まれているか、件数が多すぎないか（12件超で即エラー）など。ここで引っかかれば次の段には進まず、後述の自動修正ループに回します。

### 2段目：LLMファクトチェック——書くLLMとチェックするLLMを分ける

構造が正しくても、内容が正しいとは限りません。そこで台本生成とは**別のclaude -p呼び出し**（台本生成とは別のモデル）でファクトチェックする仕組みにしています。

台本生成・バリデーション・ファクトチェックを一つの試行として扱い、指摘が残れば**最大3回**まで自動修正ループ（最近流行ってるアレ）を回します。  
リトライ時は前回のepisode JSON全文と指摘リストをまるごと同梱し「指摘箇所だけ直して全文を再出力せよ」と依頼する方式です。  
最終試行でerrorが残れば失敗、warningだけなら無人運用なので止めずに進めます（どうせ公開前に人間の目が入るので）。

### 3段目：人間（私）による最終チェック

そのうえで、**公開前の最終確認は必ず自分でやります**。「機械チェックを通ったからOK」ではなく、自分がアップデート内容を理解した上でチェックする。  
このパイプラインは**S3にfinal.mp4を保存するところまでがスコープ**で、YouTubeへのアップロードは意図的に含めていません。  
公開前には必ず人間のチェックが挟まる構造で、**自動化するのは作業であって、理解と責任ではない**——これが運用してみての結論です。

## 気になるランニングコスト

ざっくり概算です（2026年7月時点の東京リージョン料金からの試算。単価は変わるので、正確には[Fargateの料金ページ](https://aws.amazon.com/jp/fargate/pricing/)を参照してください）。

| 項目 | 概算 |
| --- | --- |
| Fargate（16vCPU / 32GB・ARM64） | 動いている間だけ、約120円/時 |
| ↑ 週1回・1回1時間として月額換算 | **月500円ちょい** |
| Claude Code（台本生成＋ファクトチェック） | サブスクプラン内（追加費用ゼロ） |
| Aivis（音声合成） | サブスクプラン内（追加費用ゼロ） |

**月500円で、毎週休まず納品してくる動画クリエイターが雇える**と考えると破格です。LLMを台本生成とファクトチェックの2回呼んでも、**定額サブスクの範囲内でLLM代はゼロ**なのがありがたいところです。
