Claude Codeのカスタマイズ手段、意外と多いです。CLAUDE.md、Skills、Hooks、MCP、Subagents、Plugins。6種類ある。

とりあえず全部CLAUDE.mdに書いていませんか。公式ドキュメントによると、200行を超えると指示の遵守率が下がります。適材適所に振り分けるだけで、レスポンスの質が変わる。

## 6種の早見表

| 機能 | やること | コンテキストコスト | 使い所 |
| --- | --- | --- | --- |
| **CLAUDE.md** | 毎セッション読まれるルール | **高い**（毎リクエスト全文） | 「常にこうしろ」系の短い指示 |
| **Skills** | 必要時に読み込まれる知識・手順書 | 低い（呼ぶまでは説明文だけ） | 参照ドキュメント、ワークフロー |
| **MCP** | 外部サービス接続 | 低い（Tool Searchで遅延読み込み） | DB、Slack、ブラウザ操作 |
| **Subagents** | 隔離コンテキストで動く別働隊 | ゼロ（メインと完全に独立） | 重い調査、並列作業 |
| **Hooks** | イベント駆動のシェルスクリプト | ゼロ（外部実行） | lint、通知、ファイル保護 |
| **Plugins** | 上記をまとめて配布 | 中身次第 | 複数リポジトリでの共通化 |

「コンテキストコスト」列だけ見れば使い分けの8割は決まります。CLAUDE.mdだけが毎リクエストで全文消費される。残り5種はオンデマンドかゼロです。

## いつ何を追加するか

公式ドキュメントが推奨している導入順がそのまま使えます。

| きっかけ | 追加する機能 |
| --- | --- |
| Claudeが規約を2回間違えた | CLAUDE.md |
| 同じプロンプトを何度も打っている | Skill（`/名前` で呼べるようにする） |
| ブラウザのデータを毎回コピペしている | MCP |
| サイドタスクの出力でコンテキストが埋まる | Subagent |
| 毎回手動で何かを実行している | Hook |
| 別リポジトリでも同じ設定を使いたい | Plugin |

上から順に検討すれば過不足なく揃います。

## CLAUDE.md vs Skills、一番迷う2択を整理する

6種の中で判断に迷うのはたいていこの2つです。どちらも「Claudeへの指示」を書く場所なので、境界が曖昧に感じる。

まずコードで見たほうが早いです。

```
# CLAUDE.mdに書くもの
- インデントは2スペース
- テストは pnpm test
- コミット前にlintを通す

# Skillに切り出すもの
- APIエンドポイント一覧
- デプロイ手順書（/deploy で呼ぶ）
- コードレビューチェックリスト
```

判断基準は「**全セッションで毎回必要か**」の一点。

|  | CLAUDE.md | Skill |
| --- | --- | --- |
| 読み込み | 毎セッション自動 | オンデマンド |
| 向いている内容 | 短い「常にこうしろ」ルール | 長い参照ドキュメント、手順書 |
| コンテキスト消費 | 毎リクエスト全文 | 呼ぶまでゼロにできる |
| 目安行数 | 200行以内 | 制限なし |

CLAUDE.mdが膨れてきたら、参照系のコンテンツからSkillに移す。逆に、Skillに書いたルールが毎セッション必要なら、CLAUDE.mdに昇格させる。この往復で適正な分量を保てます。

もうひとつ大事な設定があります。

```
---
disable-model-invocation: true
---
```

Skillのフロントマターにこれを付けると、`/名前` で手動呼び出しするまでコンテキストに一切入りません。デプロイや本番操作のような副作用のあるSkillには必須です。付けないと、Claudeが「関連ありそう」と判断した時点で勝手に読み込まれます。

CLAUDE.mdの200行制限は「指示の遵守率が落ちる」という実測に基づく公式の推奨値です。行数が気になったら `wc -l CLAUDE.md` で確認してみてください。

## Hookは「判断不要で毎回やること」専用

LLMを介さず、決定論的に動きます。ファイル編集後にPrettierを走らせる、`.env` への書き込みをブロックする、compaction後にコンテキストを補充する。こういう処理はHook一択。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}
```

settings.jsonに書くだけ。コンテキスト消費ゼロ。

対になる概念として、`type: "prompt"` や `type: "agent"` のHookもあります。こちらはLLMに判断を委ねるタイプで、「タスクが本当に完了しているか確認してから止まれ」のような条件付き停止に使います。

## MCP・Subagents・Plugins

| 機能 | 一言で | 導入タイミング |
| --- | --- | --- |
| MCP | 外部サービスへの橋渡し | Claude単体では届かないデータが必要になったとき |
| Subagents | 隔離コンテキストのワーカー | 調査タスクでメインのコンテキストが埋まり始めたとき |
| Plugins | 上記全部のパッケージ | 2つ目のリポジトリで同じ設定が欲しくなったとき |

MCPとSubagentsは組み合わせると効きます。MCP経由でDBに接続し、Subagentにクエリを投げさせる。メインのコンテキストにはSubagentの要約だけが返る。コンテキストを汚さずに外部データを取得するパターンです。

Subagentsは親セッションのSkillsを引き継ぎません。必要なSkillはSubagent定義の `skills:` フィールドで明示的に渡す必要があります。

## まとめ

CLAUDE.mdに全部書くのは楽ですが、200行を超えた時点で逆効果です。コンテキストコストを意識して6種を使い分けるだけで、応答精度は目に見えて変わります。迷ったら冒頭の早見表に戻ってください。
