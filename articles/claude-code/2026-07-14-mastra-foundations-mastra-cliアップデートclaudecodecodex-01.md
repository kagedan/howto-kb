# [Mastra Foundations] Upgraded Mastra CLI をどう読むべきか - Mastra の新機能ではなく、Mastra 開発を AI に委ねやすくする発表

[Mastra](https://mastra.ai/) の公式発表を読み解くとき、しばしば混同しやすいのが「Mastra 自体の新機能」なのか、「Mastra で作ったエージェントの新機能」なのか、それとも「Mastra を使った開発作業の改善」なのか、という3つの層です。

2026年5月20日に公開された [Close the Loop With the Upgraded Mastra CLI](https://mastra.ai/blog/upgraded-mastra-cli) は、見出しだけだと単なる CLI 強化にも見えます。しかし本質は、**Mastra フレームワーク上で動くアプリやエージェントを、Claude Code や Codex のような外部コーディングエージェントで開発しやすくする**ところにあります。

先に公開した速報寄りの記事はこちらです。  
<https://zenn.dev/shiromizuj/articles/60c5564a579f49>

今回はその続編として、**この発表の主語は何なのか**、そして **なぜ Mastra の発表群の中では少し珍しいのか** を整理します。

---

## 先に結論

この発表は、次のどちらでもありません。

* Mastra 本体を開発する人向けに、Mastra CLI の内部実装を改善した話
* Mastra で作った業務エージェントに、新しいユーザー向け機能が追加された話

そうではなく、**Mastra というエージェントフレームワークを使ってアプリを作る開発者が、Claude Code / Codex / GitHub Copilot のような外部コーディングエージェントに、その開発作業を委ねやすくなる**という発表です。

言い換えると、今回よくなったのは「Mastra 製エージェントの能力」ではなく、**Mastra プロジェクトを作る・試す・観測する・評価する・デプロイするまでの作業導線**です。

---

## この発表で主語になっているのは誰か

私が書くような Mastra の記事は、だいたい次のどれかに分類できます。

1. Mastra フレームワーク自体に新しい能力が入る
2. Mastra 上で動くエージェントに新しい runtime 機能が入る
3. Mastra を使った開発を、周辺ツール込みでやりやすくする

今回の CLI 強化は、明らかに 3 です。

たとえば元記事では、CLI から次のようなことができると説明されています。

* agent や workflow を起動する
* tool を実行する
* memory や thread を調べる
* trace / log / metric を引く
* dataset / experiment を回す
* deploy や env 管理を行う

この一覧だけ見ると、Mastra のランタイム機能が増えたようにも見えます。しかし実際には、これらの多くはすでに Studio や Platform、あるいは既存 API の世界にあった操作です。今回の価値は、それらを **CLI という、コーディングエージェントが扱いやすい操作面** にそろえたことにあります。

つまり発表の中心は、「Mastra の業務エージェントが何をできるようになったか」ではありません。**Mastra でアプリを作っている最中の開発者と、その横で働くコーディングエージェントが、どこまで一続きに作業できるか**が主題です。

---

## 「Mastra の新機能」に見えてしまう理由

この発表が少し分かりにくいのは、CLI から触れる対象が runtime、memory、observability、eval、deploy まで広いからです。

たとえば次のような説明は、表面だけ読むと「Mastra が業務エージェント向けに新しい機能を増やした」と見えます。

* Memory を検索できる
* Trace や log を調査できる
* Evals を回せる
* デプロイできる

ですが、ここで増えたのは機能そのものというより、**外部コーディングエージェントがその操作を自分の作業ループに取り込めること**です。

たとえば Claude Code や Codex のようなツールは、もともと次のような仕事が得意です。

* リポジトリを読む
* ファイルを編集する
* テストやコマンドを回す
* 失敗ログを見て修正する

今回の CLI 強化は、その「読む・書く・実行する・直す」という作業に対して、Mastra 固有の操作面を足したものだと読むと分かりやすいです。

つまり、

* 一般的なコーディングエージェントの能力: コードベースを触る
* 今回 Mastra が追加したもの: Mastra プロジェクト特有の実行・観測・評価・配備の入口

という関係です。

## Skills や MCP Docs Server の延長線上にある

この発表を理解するうえで大事なのは、Mastra が 2026 年前半から進めてきた [Build with AI](https://mastra.ai/docs/getting-started/build-with-ai) の流れです。

2月には、Mastra は次の2つを強く打ち出していました。

* Mastra Skills
* MCP Docs Server

この2つが解決していたのは、主に **知識アクセス** の問題です。つまり、コーディングエージェントが Mastra の作法やドキュメントに届きやすくなる、という話でした。

それに対して今回の CLI 強化が広げたのは、**操作アクセス** です。

* Skills / MCP Docs Server: Mastra をどう書くかに届きやすくする
* Upgraded Mastra CLI: 書いたあとにどう実行し、どう観測し、どう改善するかに届きやすくする

この2つは役割がきれいに分かれています。前者だけでは、コーディングエージェントは「正しいコードを書きやすくなる」止まりです。後者まで入ると、**そのコードを動かし、挙動を見て、必要なら修正するところまで一続きに進められる**ようになります。

だから元記事の `Close the Loop` という言い方は、Mastra 製業務エージェントのループというより、**Mastra 開発を手伝うコーディングエージェントの作業ループ** を閉じる話だと読むのが自然です。

---

## 何が楽になるのか

実務上の変化としては、次の流れをコーディングエージェントが担いやすくなります。

1. Mastra の agent / workflow / tool を含むコードや設定を書く
2. CLI でその場で実行して、期待通りか確かめる
3. memory や trace や log を見て、何が起きたか調べる
4. dataset / experiment で変更の妥当性を評価する
5. 問題なければ deploy や env 管理まで進める

ここで重要なのは、この 1 から 5 が **Mastra アプリの開発フロー** だということです。

エンドユーザーは CLI を直接触らないかもしれませんし、完成した業務エージェントの能力がこの発表だけで急に増えるわけでもありません。変わるのは、**その業務エージェントを作っている最中の開発体験**です。

この意味では、今回の発表は「Mastra の DX 改善」と言ってもよいのですが、普通の開発者体験改善より一歩踏み込んでいます。人間が使いやすいだけでなく、**外部コーディングエージェントが扱いやすい形に、Mastra 固有の操作を整形した**からです。

---

## Mastra の発表としては、なぜ少し珍しいのか

Mastra の発表を追っていると、多くは次のどちらかです。

* runtime や memory や workflow の新機能
* platform や observability や deployment の新機能

これらは最終的に、Mastra 上で動くエージェントやアプリが何をできるかへ返っていきます。

一方で今回の CLI 強化は、もう少しメタなレイヤーにあります。直接の対象は、Mastra アプリの利用者ではありません。**Mastra アプリの開発者と、その開発者が使うコーディングエージェント**です。

その意味で、この発表はかなり珍しいです。Mastra が「どんなエージェントを作れるか」だけでなく、**そのエージェントを AI と一緒にどう作るか** まで自分たちの責務として取り始めたことを示しているからです。

これは単なる CLI 拡張以上のメッセージを含んでいます。Mastra はフレームワークとして、ランタイムだけでなく **AI 支援付きの開発導線** まで設計対象に入れ始めた、ということです。

## 速報記事と今回の記事の違い

先に公開した速報記事では、発表の全体像を追いながら、Runtime / Memory / Evals / Observability / Platform / Project という6つのまとまりを広く整理しました。

あの記事の役割は、「Upgraded Mastra CLI で何ができるようになったのか」を把握することでした。

それに対して今回の補足記事の役割は、「その機能群を **どの主語で読むべきか**」を明確にすることです。

要するに、前回は **何が増えたか**、今回は **それは誰のための増加なのか** を整理しています。

この違いを押さえておくと、CLI 強化を「Mastra の新機能」とだけ雑に読むよりも、Mastra が 2026 年にどこへ向かっているかが見えやすくなります。

## まとめ

Upgraded Mastra CLI は、Mastra 製業務エージェントに新しいユーザー向け能力を足した発表ではありません。

本質は、**Mastra というエージェントフレームワーク上で動くアプリを、Claude Code や Codex のような外部コーディングエージェントで作るときに、実行・観測・評価・配備まで含めて委ねやすくした**ことにあります。

だからこの発表を一文で言い換えるなら、次の表現がいちばんぶれません。

**Mastra を使ったアプリ開発を、外部コーディングエージェントに委ねやすくするための操作面拡張**。

Mastra の発表群の中では少し珍しいですが、実はかなり重要な一歩です。なぜならこれは、「どんなエージェントを作れるか」だけでなく、**AI と一緒にそのエージェントをどう開発するか** を Mastra が本格的に設計し始めたサインだからです。
