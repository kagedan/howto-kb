## 📝 Summary

* LLMは学習時点で重みが固定される静的なモデルで、本来「今がいつか」を内在的に知る手段を持たない。多くのAIプロダクトは会話冒頭に現在日時をシステムプロンプトへ注入することでこれを補っている。
* しかし「時刻を渡す」実装の仕方次第で、プロンプトキャッシュのヒット率・コスト・そしてAIの時間感覚の精度が大きく変わる。
* 本記事では、システムプロンプトに`datetime.now()`的な値を毎リクエスト埋め込むとキャッシュが壊れる仕組み、Claude/Geminiでの挙動の違い、実際にAIエージェントの時間誤認が報告された事例、そして「時刻＝コンテキストのシャーディングキー」として日誌ファイルを日次パーティション化する応用まで、時刻付与を「取得の粒度」という設計変数として整理する。

## 🖋 Content

### 1. 導入：AIには本来「今」がない

LLMは学習データのある時点で重みが固定される、いわばスナップショットである。会話が今日行われているのか、1年前に行われたログの再生なのか、モデル自身には区別する手段がない。これは「知識のカットオフ」とは別の話で、カットオフはあくまで知識の限界であって、現在時刻の情報ではない。

だからこそ多くのAIプロダクトは、会話が始まるタイミングでシステムプロンプトに現在日付を注入している。実際、Anthropicの公式ドキュメントでもclaude.aiやモバイルアプリは会話冒頭に現在日付を含むシステムプロンプトを自動的にロードすると明記されている([Claude System Prompts](https://docs.anthropic.com/en/release-notes/system-prompts))。この仕組み自体はよく知られているが、「どう渡すか」の設計判断まで踏み込んで語られることは少ない。ここから先がこの記事の本題になる。

### 2. 時刻を与えるメリット（簡潔に）

* **相対日付表現の解決**：「来週」「先週の続き」といった表現を絶対日付に変換できる
* **知識カットオフとの切り分け**：今が学習後どれくらい経っているかを自覚できるため、「自分の知識は古いかもしれないから検索すべきか」を判断する材料になる

このあたりは既に広く語られている話なので深追いはしない。本題はここから先、「渡し方」によって生じる副作用のほうだ。

### 3. 副作用①：プロンプトキャッシュの破壊

Claude APIのプロンプトキャッシュは「同じ内容の再利用」ではなく、**リクエストの先頭からのバイト列が前回と完全に一致している区間**をキャッシュとして再利用する仕組みである。リクエストは`tools → system → messages`の順でレンダリングされ、`cache_control`で指定したブレークポイントまでの区間がキャッシュ対象になる。

ここでシステムプロンプトに

```
system=f"You are an assistant. Current time: {datetime.now()}"
```

のように現在時刻を差し込むと、この文字列は**リクエストのたびに1文字も同じにならない**。キャッシュは先頭からの完全一致を見ているため、時刻部分より後ろは毎回「前回と違う内容」として扱われ、キャッシュヒットが起きなくなる。

具体的な挙動は次の通り。

* `usage.cache_read_input_tokens`が常に0になる（キャッシュから読めた分がゼロ）
* 逆に`cache_creation_input_tokens`だけが毎回発生し続ける。これは新規書き込み扱いなので、通常の入力トークンより**約1.25倍**（5分TTLの場合）割高になる
* つまり「キャッシュを使っているつもり」で、実際は毎回フルコスト＋書き込みプレミアムを払い続けている状態になる

エラーにはならず、レスポンスも一見正常に返ってくるため、コストや速度をちゃんと見ていないと気づきにくい。この「システムプロンプトに`datetime.now()`を埋め込む」パターンは、Anthropic自身のドキュメントでもキャッシュを無効化する典型例として名指しされている。

#### Geminiでも同じ原理で壊れる

Gemini 2.5以降のモデルにはデフォルトで「暗黙キャッシュ（implicit caching）」が有効になっており、これもリクエスト間の共通プレフィックスを見て自動でキャッシュヒットを判定する仕組みである。Google自身が「リクエストの先頭は固定にして、変わる部分は末尾に置け」と案内しており([Gemini API context caching](https://developers.googleblog.com/en/gemini-2-5-models-now-support-implicit-caching/))、原理はAnthropicとほぼ同じだ。したがってシステムプロンプトの先頭に時刻を埋め込めば、同じ理由でヒット率が落ちる。

ただし2点、Anthropicとの違いがある。

1. **ヒットの保証がない**。Googleの公式ブログでも暗黙キャッシュは"automatically enabled... no cost saving guarantee"と明言されている([Gemini 2.5 implicit caching](https://developers.googleblog.com/gemini-2-5-models-now-support-implicit-caching/))。Anthropicのように`cache_control`で明示的に指定し、`usage.cache_read_input_tokens`で確実に検証できる方式とは対照的だ。
2. 実際、共通プレフィックスが長い（9,000トークン超）ケースでヒット率が40〜60%程度でばらつくという不具合報告もある([python-genai issue #1880](https://github.com/googleapis/python-genai/issues/1880))。つまりGeminiは「時刻を埋め込んで壊す」以前に、素の状態でもキャッシュヒットが安定しないことがある。

なおGeminiには別に「明示キャッシュ（explicit caching）」という、コンテンツを一度アップロードしてキャッシュオブジェクトを作り、以降はそのハンドルを参照する方式もある。こちらはプロンプト文字列の一致ではなくオブジェクトの使い回しなので、この記事で扱う"時刻埋め込み問題"とは別の話になる。ただしこちらのTTL設定でタイムゾーン情報のないnaiveなdatetimeオブジェクトを渡すと期待通りに動かない、という別種の罠がある点だけ触れておく。

### 4. 副作用②：取得の粒度という設計変数

ここまでの話を踏まえると、「時刻を渡すか渡さないか」は二択ではなく、**「いつ取得し直すか」という粒度の設計問題**として捉えたほうが正確だ。

| 粒度 | キャッシュへの影響 | 得られる精度 |
| --- | --- | --- |
| リクエスト毎に再取得 | 毎回プレフィックス破壊、常にフルコスト | 秒単位の精度だが、ほぼどのタスクにも過剰品質 |
| セッション開始時に1回 | セッション中はプレフィックス固定＝キャッシュが効く | 「今日は何日か」程度の粒度には十分 |
| 渡さない | キャッシュには無関係 | 相対的な日付表現が解決できない |

「セッション開始時に1回」は一見バランスの良い落とし所に見えるが、これにも別の罠がある。

**日付だけでは足りないケースがある。** Claude Codeでは会話冒頭に日付（`currentDate`）は注入されるが、正確な時刻やタイムゾーンまでは含まれない。この設計に対して、実際にGitHub上で「日付だけで時刻がないと、AIが時間帯の判断を誤る」という報告が上がっている。たとえば21時なのに21時半だと勘違いしたり、カレンダーの予定がすでに終わっているかどうかを誤判定したりする、というものだ([anthropics/claude-code Issue #34530](https://github.com/anthropics/claude-code/issues/34530))。日付という粗い粒度が、エージェント的なタスク（「そろそろ寝る時間か」「この予定はもう過ぎているか」）には足りていない、という具体例になっている。

**長時間セッションでは鮮度が劣化する。** セッション開始時に固定した時刻は、会話が長く続くほど「嘘」になっていく。実際、この記事を書いている最中のセッションでも、現在日付はセッション冒頭に一度注入されたきりで、以降のターンで更新される仕組みにはなっていない。数日にまたがるような長い会話であれば、モデルはいつまでも古い日付を「今日」だと思い込んだまま話し続けることになる。

つまり、**キャッシュ効率と時刻の鮮度はトレードオフの関係にある**。粒度を粗くすればキャッシュは守れるが鮮度は落ち、粒度を細かくすれば鮮度は保てるがキャッシュが壊れてコストが上がる。どちらかに倒すのではなく、タスクの性質（キャッシュの再利用頻度が高いバッチ処理的な用途か、鮮度が命に関わるエージェント的な用途か）に応じて選ぶべき設計変数だ、というのがここでの結論になる。

### 5. 応用：時刻をコンテキストのシャーディングキーとして使う

粒度の話を踏まえると、もう一段実践的な使い方が見えてくる。時刻を「相対日付を解決するための情報」としてだけでなく、**「どのファイル（どの文脈の塊）を読み書きすべきか」を決めるためのキー**として使う、という発想だ。

具体的には、日々の作業ログやAIとのやり取りの要約を、日付をファイル名に含めた形で1日1ファイルに分けて記録しておく。すると：

* 「昨日の続き」を頼まれたら、昨日の日付のログファイルを読めば文脈が復元できる
* 「今日の続き」なら、その日のログファイルがすでに存在するはずなので、そこから引き継げる

これは会話履歴を丸ごと保持し続ける、あるいは圧縮（compaction）に頼るのとは別の、軽量なコンテキスト管理の方法になる。時刻がわかっているからこそ「今どのパーティションにいるか」が機械的に決まり、必要な文脈だけをピンポイントで読み込める。

そしてこの運用は、3節で挙げた「長時間セッションでの鮮度劣化」という問題への部分的な解決策にもなっている。一つの会話をだらだらと続けて時刻情報を古びさせるのではなく、日ごとに区切って毎回正確な現在日付を起点にし直すほうが、鮮度の面でも健全になるからだ。時刻を渡すという行為が、単なる相対日付の解決だけでなく、エージェントの記憶構造そのものの設計に関わってくる、という一例だと思う。

### 6. まとめ

* AIには本来「今」の感覚がなく、多くのプロダクトはシステムプロンプトへの日時注入でこれを補っている
* ただし注入の粒度を誤ると、プロンプトキャッシュが静かに壊れ、コストが上がる。これはClaude・Gemini双方に共通する原理
* 逆に粒度を粗くしすぎると、日付だけでは時間帯の判断を誤ったり、長時間セッションで鮮度が劣化したりする
* 時刻は「渡すか渡さないか」の二択ではなく、「いつ・どの粒度で取得し直すか」という設計変数として扱うべきで、日次パーティション化のような文脈管理の応用にもつながる
