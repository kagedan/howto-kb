## はじめに

無料の天気APIを個人ツールに組み込もうとしたとき、規約をちゃんと読んだら「リクエストが多すぎるとIPアドレスでBANする場合があります」と書いてありました。

そこまで大量に叩く想定はなかったのですが、自分だけでなく同じIPを使う人にも迷惑がかかる可能性があると思うと、ビクビクしながら開発するのは嫌だなと。それでキャッシュを入れることにしました。

もうひとつの動機がAIコーディングです。Claude CodeなどのAIエージェントに作業を任せると、テストやデバッグの過程でAPIを含むスクリプトを何度も自動実行することがあります。「気づいたらAPIを50回叩いていた」という状況を防ぐためにも、キャッシュで最小限のリクエストに抑えたかった。

どちらも「まあ普通にキャッシュを入れれば済む話」なのですが、Redis や SQLite を持ち出すほどでもなく。`requests-cache` のようなライブラリも検討しましたが、HTTPクライアントに `httpx` を使っていたので合わず。結局、標準ライブラリだけで、40行ほどのモジュールとして自作することにしました。

## 設計方針

シンプルに保つため、次の方針にしました。

| 要素 | 方針 |
| --- | --- |
| キャッシュキー | リクエストパラメータの辞書をそのまま使う |
| ファイル名 | パラメータを `sort_keys=True` でJSONシリアライズ → SHA256ハッシュの先頭16文字 |
| TTL管理 | ファイル内に `fetched_at` を埋め込んで取得時に判定 |
| ストレージ形式 | `{"fetched_at": "...", "data": ...}` のシンプルなJSON |

URLクエリ文字列ではなく辞書を正として扱うことで、パラメータが増えても自動的にキーが変わります。設計変更に強く、ハッシュ衝突も起きにくいです。

## 実装

```
# cache.py
import hashlib
import json
from datetime import datetime, timedelta, timezone
from pathlib import Path
from typing import Any

CACHE_DIR = Path(".cache")

def _cache_path(params: dict) -> Path:
    # sort_keys=True が重要：辞書のキー順序が違っても同じファイルにする
    key = json.dumps(params, sort_keys=True, ensure_ascii=False)
    digest = hashlib.sha256(key.encode()).hexdigest()[:16]
    return CACHE_DIR / f"{digest}.json"

def load(params: dict, ttl_seconds: int) -> Any | None:
    path = _cache_path(params)
    try:
        cached = json.loads(path.read_text(encoding="utf-8"))
        fetched_at = datetime.fromisoformat(cached["fetched_at"])
        if datetime.now(timezone.utc) - fetched_at < timedelta(seconds=ttl_seconds):
            return cached["data"]
    except Exception:
        # ファイルがない・壊れている場合はキャッシュミスとして扱う
        pass
    return None

def save(params: dict, data: Any) -> None:
    path = _cache_path(params)
    path.parent.mkdir(parents=True, exist_ok=True)
    payload = {
        "fetched_at": datetime.now(timezone.utc).isoformat(),
        "data": data,
    }
    path.write_text(json.dumps(payload, ensure_ascii=False, indent=2), encoding="utf-8")
```

### 3つのポイント

**① `_cache_path` の `sort_keys=True`**

Pythonの辞書はキーの挿入順を保持するため、`{"lat": 35.6, "lon": 139.7}` と `{"lon": 139.7, "lat": 35.6}` は `json.dumps` で別の文字列になります。`sort_keys=True` を指定することで、同じ内容の辞書は必ず同じハッシュになります。

なお、`params` の値には `str`・`int`・`float`・`bool`・`None` など JSON シリアライズ可能な型のみを使ってください。`datetime` やカスタムクラスを渡すと `TypeError` で落ちます。

**② `load` の例外の飲み込み**

ファイルが存在しない、途中で破損している、フォーマットが古い——いずれの場合も `None` を返してキャッシュミスとして処理します。キャッシュはあくまで補助なので、失敗したら素直にAPIを叩けばいい。

**③ `save` の `mkdir(parents=True, exist_ok=True)`**

`.cache/` ディレクトリがなくても自動作成されます。初回実行時のセットアップ手順が不要になります。

## 使い方：APIラッパーへの組み込みパターン

天気APIを叩く関数への組み込み例です。

```
import httpx
import cache

TTL = 60 * 30  # 30分

def fetch_weather(lat: float, lon: float) -> dict:
    params = {"type": "weather", "lat": lat, "lon": lon}

    # 1. キャッシュを確認
    if cached := cache.load(params, ttl_seconds=TTL):
        return cached

    # 2. キャッシュがなければAPIを叩く
    resp = httpx.get(
        "https://api.example.com/weather",
        params={"lat": lat, "lon": lon},
    )
    resp.raise_for_status()
    data = resp.json()

    # 3. 結果を保存して返す
    cache.save(params, data)
    return data
```

「load → 早期 return → fetch → save」の4ステップが基本パターンです。どのAPI関数も同じ構造になるので、慣れると迷わず書けます。

## キャッシュキーの設計が肝

`params` には `"type": "weather"` という識別子を含めています。同じ座標で「現在の天気」と「週間予報」を取得する場合、座標だけをキーにすると衝突するためです。

```
# 衝突する例
params_weather  = {"lat": 35.6, "lon": 139.7}  # ← 同じキー
params_forecast = {"lat": 35.6, "lon": 139.7}  # ← 同じキー

# 衝突しない例
params_weather  = {"type": "weather",  "lat": 35.6, "lon": 139.7}
params_forecast = {"type": "forecast", "lat": 35.6, "lon": 139.7}
```

`type` フィールドを混ぜる、エンドポイント名を含めるなど、**何を取得したかわかる識別子をキーに入れる**のがコツです。

## 注意点と割り切り

**古いキャッシュファイルは自動削除しない**

TTL切れのファイルは次回アクセス時に上書きされますが、アクセスされないファイルは残り続けます。気になるなら手動でクリアしてください。

```
# 7日以上前のキャッシュを削除
find .cache -name "*.json" -mtime +7 -delete
```

**並行書き込みは考慮していない**

複数プロセスが同時に同じファイルを書くと壊れる可能性があります。個人CLIツールで直列実行する前提なので割り切っています。並行処理については別途検討必要です。

**データ量が増えてきたら移行のサイン**

ファイル数が数千を超えたり、辞書をバリューとして検索したくなったりしたら、そもそもデータベースや何らかのライブラリを使うべきですね。`SQLite` などへの移行を検討するタイミングだと思います。

## まとめ

(根本解決ではないですが)このキャッシュを入れてから、BANを気にしながら開発することも、AIに好き勝手APIを叩かれることもなくなりました。特殊なライブラリを `pip install` せずに利用できるのもいいんじゃないでしょうか。

「load → 早期 return → fetch → save」のパターンはどのAPIラッパーにも使い回せるので、40行ごとコピーして次のプロジェクトでも使っています。
