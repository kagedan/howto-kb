## 1. 品質担保の最終地点はテスト、なのに仕様書は文章だった

チームで開発工程について話していたときに、「品質担保の最終的な拠り所は、結局テストケースに落とし込まれている」という話になりました。  
それなのに入口には文章の仕様書があり、人の頭の中で翻訳が行われる。そこに解釈の余地が生まれてしまっていることが課題の1つだと感じていました。

## 2. 仕様 ≒ テストケースの集合

上記の課題は、AIを活用するようになってからさらに鮮明になりました。人間は文章の曖昧さを経験で補えますが、AIは書かれた通りに受け取ります。  
検証可能な形が終着点なら、入口からその形で書くべきだと考えました。

仕様書に書かれていることの多くは、突き詰めると状態遷移・振る舞い・条件分岐です。  
これらはどれも検証可能な単位に分解でき、そのままテストケースとして表現できます。  
そこで、仕様を「理解を固定する文書」ではなく、「検証可能な振る舞いの集合」として捉え直すことにしました。

そうすることで、3 つの効果が同時に働くと期待できます。

* AI が解釈しやすくなる — Given / When / Then の欠落が穴として見える
* 実装・テストとの認識ずれを減らせる — 受入基準をそのまま実装とテストに写せる
* 変更影響を定量化できる — テストケース単位で差分を数えられる

この考え方自体は BDD（Behavior-Driven Development）や "Specification by Example" として以前から体系化されています。この記事では、それをチームの日常的な仕様作成フローに組み込んで、AI と一緒に回した内容を紹介します。

## 3. 具体例

上記の考え方を、具体的な書き方に落としてみます。題材は同僚評価の平均スコア表示です。360度評価システムの一機能で、同僚が入力した評価スコアの平均を算出して表示する画面を想定しています。

### Before — 文章中心の仕様

```
# 同僚評価の平均スコア表示

同僚評価の画面では、対象メンバーの同僚からの評価平均を表示する。
評価項目ごとに、同僚が回答した評価値の平均を算出する。
回答していない同僚は除外する。
全員が未回答の場合は平均を出せないのでダッシュ（—）を表示する。
自己評価は含めない。
```

読めば意図は伝わりますが、実装に渡すと解釈の余地がいくつも残ります。

* 平均の丸め方は？小数第何位まで？
* 同僚 1 人だけ回答の場合も平均を出す？
* 自己評価は「完全除外」？それとも別枠表示？
* ダッシュは半角ハイフン？em ダッシュ？

### After — 画面イメージ + テストケース

テストケースは要件 ID を参照するため、先に要件定義を置きます。

requirements.md

```
## 要件定義

| ID      | 要件                                                                               |
| ------- | ---------------------------------------------------------------------------------- |
| REQ-001 | 同僚が 1 人以上回答した場合、小数第 1 位（四捨五入）の平均スコアを表示する         |
| REQ-002 | 未回答の同僚は分母から除外する。全員未回答の場合は em ダッシュ（U+2014）を表示する |
| REQ-003 | 自己評価は分子・分母どちらにも含めない                                             |
```

spec.md

```
## 画面イメージ

![評価項目ごとに平均スコアが表示される一覧](./score.png)

## テストケース

### TC-001: 同僚全員が回答した場合は平均を出す

- Given: 同僚 3 人の評価が { A:4, B:5, C:3 }
- When: 平均スコアを算出する
- Then: 4.0 を表示する（小数第 1 位、四捨五入）
- Covers: REQ-001

### TC-002: 未回答の同僚は分母から除外する

- Given: 同僚 3 人の評価が { A:4, B:未回答, C:2 }
- When: 平均スコアを算出する
- Then: 3.0 を表示する（(4+2)/2、未回答 B は分母に含めない）
- Covers: REQ-002

### TC-003: 同僚が全員未回答なら em ダッシュを表示する

- Given: 同僚 3 人全員が未回答
- When: 平均スコアを算出する
- Then: 「—」（U+2014 em ダッシュ）を表示する
- Covers: REQ-002

### TC-004: 自己評価は分子・分母どちらにも含めない

- Given: 本人が自身に 5、同僚 A/B が 3 を回答
- When: 平均スコアを算出する
- Then: 3.0 を表示する（(3+3)/2、本人の 5 は集計対象外）
- Covers: REQ-003
```

Before で曖昧だった箇所は、After では Then の中に具体値として固定されています。  
仕様の読み手が解釈する余地を減らし、検証可能な形に寄せられているのがポイントです。

## 4. Skills として組み立てる

上記の仕様作成の流れを Claude Code の Skills として組み立てました。

公開入口は 2 つだけに絞っています。

* `/spec` — 新規作成や続きからの生成を行う入口。要件定義だけ、画面仕様だけ、要件から画面仕様まで一気通貫、のどれを行うかは内部で判定する
* `/spec-edit` — 既存仕様の編集入口。変更対象の仕様書を特定し、要件・画面仕様のどちらに反映するかを内部で振り分ける。変更率も REQ / TC 単位で記録する

ユーザーは自然言語で依頼するだけで、どこを呼び出すかは Skills 側で決めます。

実際の操作はこんな感じです。

```
> /spec 同僚評価の平均スコア表示機能の仕様を作って
```

これだけで、要件定義の生成 → テストケースの生成 → ファイル保存が走ります。  
既存の仕様を変更するときは `/spec-edit` に変更内容を自然言語で伝えます。

```
> /spec-edit 未回答の場合の表示を「N/A」から「-」に変更する
```

Skills は影響する REQ ID と TC を特定したうえで変更し、変更率を記録します。

レビューや変更の中で決まった判断は別途ファイルに蓄積しており、後から意思決定の歴史を振り返れるようにしています。

## 5. 気づきと、課題

いくつかの気づきがありました。

* **ドキュメントは最終的に捨てられる方が良い**  
  テストケースがそのままテストコードに流用できれば、仕様書は役割を終えて消えていけます。「作って残す」から「作って流用されて消える」に向かうと、保守負荷が本質的に下がります
* **仕様の変化を数字で追えるようになる**  
  変更率やカバレッジを数値で取れるので、どこに変更が集中しているか、どの要件のカバレッジが薄いかを議論の対象にできるようになりました
* **書き方を変えると、レビューの観点も変わる**  
  文章だと「意図が伝わるか」「漏れがないか」が中心でしたが、テストケースだと「Given や Then に穴はないか」「TC は REQをカバーしているか」と、観点が構造的になります
* **テキストの曖昧性は、再現可能な仕組みに寄せる**  
  仕様をテストケースで書くのは、「文章の曖昧さを減らし、静的解析や自動テストで担保できる範囲を広げる」という流れの一例です。仕様以外でも、判断が文章に委ねられている箇所は、同じ考え方で置き換えていけそうです

一方で、課題も残っているため、今後取り組んでいく予定です。
