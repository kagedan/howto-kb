*GitHub Copilot Custom Agents / Agent Mode を Hook で制御・監査する試み  
──  
執筆協力: Tech Lead BOSS*

![](https://static.zenn.studio/user-upload/81418c39de7b-20260421.png)

これまでの記事では、GitHub Copilot 上に「AIの相棒」を作るために、判断基準、責務分離、止まれる構造をどう設計したかを書いた。

今回は、その続きとして **Hook** の話を書く。

最初に結論を書く。

**プロンプトで「止まれ」と書くだけでは足りなかった。だから、Copilot の実行ライフサイクルの外側に Hook を置いて、入力・ツール実行・サブエージェント委譲・セッション終了を監査できるようにした。**

この記事は、Hook の設定方法だけを説明するものではない。  
GitHub Copilot Custom Agents を実務で使うときに、

* どこで止めるか
* 何をログとして残すか
* サブエージェントが本当に動いたかをどう見るか
* 失敗やハングを次の改善にどう戻すか

を、Hook でどう制御層にしたかの記録である。

なお、GitHub Docs では Hooks は Copilot agent の実行中の重要なタイミングで shell command を実行し、ツール実行の承認・拒否、監査ログ、セキュリティ検査などに使える仕組みとして説明されている。  
公式リファレンスは次を参照してほしい。

---

## Hookを入れる前に起きていたこと

相棒を運用していると、危ない場面がいくつか見えてきた。

代表的にはこの3つだった。

| 起きていたこと | 何が危なかったか |
| --- | --- |
| 存在しないファイルを前提に進み始めた | 前提確認がプロンプト側の善意に寄っていた |
| AI が自律的に PR を出そうとした | 承認境界が実行時に強制されていなかった |
| サブエージェント処理が途中で止まって見えた | 委譲の開始・終了が観測できなかった |

ここで分かったのは、エージェント定義にどれだけ「止まる条件」を書いても、それは基本的に **エージェントが守ってくれることを期待するルール** だということだった。

もちろんプロンプトのルールは必要だ。  
各カスタムエージェントの責務分離も必要だ。

ただし、それだけでは足りない。

エージェントが実際にツールを呼ぶ瞬間、危険な入力が渡った瞬間、サブエージェントが起動した瞬間、セッションが異常終了した瞬間。  
そこはプロンプトではなく、実行基盤側で見たい。

**判断基準はエージェントに持たせる。実行境界は Hook で見る。**

この分離が必要だった。

## 失敗に基づく改修: Hookは後から生えた防御だった

この Hook 構成は、最初からきれいにあったわけではない。

最初は、エージェント定義に止まる条件を書けば足りると思っていた。  
しかし実際に運用すると、止まる条件を書いたことと、実行時に止まれることは別だった。

失敗というより、正確には「プロンプトだけでは拾い切れなかった運用上の痛み」が先にあった。

* サブエージェントが止まって見えた
* terminal 実行がハングしそうになった
* `gh` 操作が監査しづらかった
* transcript が巨大になり、後から追えなくなった
* セッション終了時に何が起きたか残したくなった

ここでやってはいけないのは、全部をプロンプトの注意書きに戻すことだった。

「サブエージェントが止まったら気をつける」  
「terminal 実行ではハングに注意する」  
「GitHub 操作は慎重にやる」

こう書くことはできる。  
だが、これでは結局、エージェントと人間の注意力に依存する。

そこで、この辛さを Hook のどこで拾うかに翻訳した。

| 起きたこと | Hook側に移した制御 | 効いたこと |
| --- | --- | --- |
| 危険な依頼がそのまま渡りそうになる | `userPromptSubmitted` で入口監査する | 危険入力の傾向を先に記録できる |
| terminal 実行がハングしそうになる | `preToolUse` で無期限実行を禁止する | 無期限実行を実行前に止められる |
| GitHub 操作の単位が粗い | `preToolUse` で実行単位を制限する | 監査しやすい単位に分けられる |
| 実行失敗が後から追えない | `postToolUse` で失敗・deny・duration を残す | 失敗を次の制御条件へ戻せる |
| 委譲がどこで止まったか見えない | 独自の委譲ログを記録する | 起動したのか、終わったのかを追える |
| セッション終了時の状態が残らない | `sessionEnd` で集計・通知・incident 化する | セッション単位で振り返れる |

この順番で見ると、Hook は「便利そうだから足した機能」ではない。

**プロンプトで守れなかった境界を、実行ライフサイクルの外側へ逃がしたもの**だった。

## Hookは「便利な拡張」ではなく制御層だった

公式の Hook 設定として読むなら、まず GitHub Docs の形式に合わせる必要がある。  
例えば、監査ログのための最小構成は次のような形になる。

```
{
  "version": 1,
  "hooks": {
    "sessionStart": [{ "type": "command", "bash": "./audit/log-session-start.sh", "timeoutSec": 30 }],
    "userPromptSubmitted": [{ "type": "command", "bash": "./audit/log-prompt.sh", "timeoutSec": 20 }],
    "preToolUse": [{ "type": "command", "bash": "./audit/pre-tool-security.sh", "timeoutSec": 20 }],
    "postToolUse": [{ "type": "command", "bash": "./audit/log-tool-result.sh", "timeoutSec": 20 }],
    "sessionEnd": [{ "type": "command", "bash": "./audit/log-session-end.sh", "timeoutSec": 30 }]
  }
}
```

自分の運用では、この公式 Hook に加えて、サブエージェント委譲や圧縮前保全に相当する独自ログも置いている。  
これらを並べると、Copilot の実行をだいたい次のように囲える。

![Copilot の実行ライフサイクルを Hook が囲む流れ](https://static.zenn.studio/user-upload/deployed-images/dc704d1544b048e9cc4cd6b6.png?sha=88f49dd8a211df5f015892e179e0ac259eb1acb7)

※図中の Signal Keeper / agent-server は GitHub Copilot の公式機能ではなく、自分の環境で Hook ログやセッション状態を集約するために置いた独自のログ基盤・制御プレーンである。

ここで大事なのは、Hook がエージェントの代わりに設計判断をしているわけではないことだ。

設計判断は統括AIが持つ。  
委譲と分割は審査ゲートAIが持つ。  
実装や調査は実務実行AIが持つ。

Hook はその外側で、入力監査、ツール実行ガード、メトリクス、サブエージェント起動記録、セッション終端処理を持つ。

つまり Hook は「もう1体のエージェント」ではない。  
**エージェントたちが動く床に敷いた制御層**である。

## 1. Promptを受けた瞬間に監査する

最初に効いたのは `userPromptSubmitted` 相当の入口監査だった。

ここでは、ユーザーのプロンプトが統括AIや審査ゲートAIに渡る前に危険な文字列を検査する。

例えば次のようなものを検出する。

* `rm -rf /`
* `git reset --hard`
* `git clean -fd`
* `DROP DATABASE`
* `--no-verify`
* AWS / GitHub / Anthropic / Google API Key らしき文字列

ただし、公式の `userPromptSubmitted` は出力が無視され、prompt modification も現在は処理されない。  
そのため、ここは「強制的に止める場所」ではなく、監査・記録・警告の入口として扱うのが安全だ。

強制的に止める主戦場は、次に出てくる `preToolUse` に寄せる。  
これは「エージェントに危険だから止めてね」と頼むのとは違い、実際に危険な操作へ進む直前のツール境界で止めるという考え方だ。

もちろん、これで全ての危険を防げるわけではない。  
ただ、壊れ方が変わる。

プロンプト側の注意書きだけだと、疲れているときや文脈が重いときに抜ける。  
Hook で入口に置くと、少なくとも明確な危険パターンを機械的に検知できる。

ただし、ここで注意したいのは、Hook が強すぎるとエージェントの柔軟性を奪うし、自律的処理継続のチャンスも奪うということだ。（実際に何度か止まったことがある）  
独自実装で入口ブロックを試す場合、たまにあるのが、統括AIや審査ゲートAIからの指示プロンプトに禁句文字が入っていて、Hook が止めてしまうパターン。  
これは、徐々に調整しながらやっていくしかない。（涙）

次に効いたのが `preToolUse` だった。

ここはかなり強い。  
公式ドキュメントでも、pre-tool use hook はツール実行を承認または拒否できる強力な Hook として説明されている。  
出力としては `permissionDecision` に `"allow"` / `"deny"` / `"ask"` を返せるが、現在処理されるのは `"deny"` とされているため、この記事では「危険なものを deny する境界」として扱う。

自分の実装では、主に次を見ている。

| 検査 | 意図 |
| --- | --- |
| Secret パターン検出 | ツール入力に認証情報が混ざったらブロック |
| `run_in_terminal` の `timeout=0` | 無期限ハングを防ぐ |
| 1回の terminal 実行に複数の `gh` コマンド | GitHub 操作の監査単位を分ける |
| `gh pr create` の必須引数不足 | 対話待ちで止まる状態を避ける |
| 破壊的コマンドパターン | 警告ログを残す |

ここで面白かったのは、Hook を足したことで「ルールを守らせる」だけでなく、**エージェントへの依頼の形そのものが変わった**ことだった。

例えば、以前なら「PR を作って」になりがちな依頼でも、Hook 側で `gh pr create` に `--title` / `--body` / `--base` / `--head` を要求する。  
すると、エージェント側も自然に非対話・監査可能な実行形式を意識するようになる。

プロンプトで注意するより、実行境界で要求するほうが強い。

`postToolUse` では、ツール実行後の結果を記録する。

やっていることは地味だ。

* tool 名
* 成功 / 失敗 / deny
* 実行時間
* 出力サイズ
* session id
* trace span
* 失敗時の incident 化

これを `.github/hooks/logs/tool-metrics.jsonl` や trace JSONL に残すようにしている。

最初は「ログを残すだけなら後で見ないのでは」と思っていた。  
でも実際には、後からチャット履歴や Hook ログを見返したときにかなり効いた。

自分のとある環境では、VS Code Copilot Chat の transcript を解析したところ、22セッション、54,525イベント、約43.9MB の履歴があった。  
その中で審査ゲートAIまわりの言及は 951 件拾えた。

この量になると、人間が「たぶんこの辺で詰まった」と思い出すのは無理がある。  
Hook 側に tool failure、deny、timeout、session end を残しておくと、あとから「どの種類の失敗が多いか」を見られる。

**失敗を、その場の反省で終わらせず、次の制御条件に戻せる。**

これが `postToolUse` の価値だった。

## 4. Sub-Agentの開始と終了を観測する

マルチエージェント構成で一番困ったのは、サブエージェントが「動いているのか、止まったのか、終わったのか」が見えにくいことだった。

統括AI → 審査ゲートAI → 実務実行AI という構造にすると、ユーザーと直接話しているのは統括AIだけになる。  
その裏で審査ゲートAIが実務実行AIに投げている。

このとき、委譲が詰まると外からはこう見える。

**何かが止まっている気がするが、どこで止まっているか分からない。**

そこで公式 Hook とは別に、サブエージェントの起動・終了に相当する独自の委譲ログを `subagent.jsonl` に残すようにした。

これは品質を直接上げる機能ではない。  
でも、マルチエージェント運用ではかなり重要だった。

| 観測できること | 効き方 |
| --- | --- |
| どの agent が起動したか | 審査ゲートAIの委譲が意図通りか確認できる |
| 終了ログが来たか | ハングか完了かを分けられる |
| 起動だけあって終了がない | 詰まりの調査起点になる |

「賢く委譲する」だけではなく、「委譲の痕跡が残る」ことが必要だった。

## 5. Context圧縮前に、消えてほしくない方針を残す

圧縮前の保全は、まだ慎重に扱っている。

自分の実装では、圧縮前保全に相当する独自の観測点をデフォルトでは no-op にしている。  
`HOOK_ENABLE_PRECOMPACT_STEERING=1` のときだけ、圧縮後にも残したい方針を `systemMessage` として返ようにしている。

例えば次のようなものだ。

* 実装変更は適切な担当に委譲する
* インフラ変更はインフラ担当に委譲する
* コード変更にはテストを書く
* Secrets をハードコードしない
* 既存ファイルの削除から新規作成で置き換えない

ここを常時有効にしなかったのは、Hook が強すぎるからだ。

圧縮前に system message を注入できるということは、エージェントの振る舞いにかなり深く影響できる。  
だから、まずはログと観測を主軸にして、steering は明示フラグで有効化する形にした。

Hook は制御層だが、制御層が強くなりすぎると、それ自体が新しい複雑さになる。

## 6. sessionEndでセッションを閉じ、次の改善に戻す

最後に `sessionEnd`。

ここでは、セッション終了時に次を行っている。

* tool metrics の集計
* timeout / abort の incident 化
* trace の session span end
* `sessions.jsonl` への要約追記
* knowledge abstraction / fitness / pattern lifecycle / generation cycle の起動
* localhost の agent-server へ completion notify

ここで大事なのは、Hook の通知先を外部 webhook に広げなかったことだ。

今の構成では、Hook から agent-server への通知は `http://127.0.0.1:8080/hook-events/ingest` に固定している。  
外部通知や OS toast はやめて、ローカル保存を正本に寄せた。

理由は単純で、制御層が外部依存で壊れると本末転倒だからだ。

Hook は失敗してもエージェントの本体処理を過剰に壊さない。  
ただし、監査ログは残す。  
このバランスにした。

## Hookによる制御構成という考え方

ここまで実装して、Hook は単発の便利機能ではなくなった。  
自分の中ではこれを **Hook Control Mesh** と呼んでいる。  
日本語で言えば、**Hookによる制御構成** である。

図解にすると、全体像はこうなる。

![Hookによる制御構成の全体図](https://static.zenn.studio/user-upload/deployed-images/b62e89bf1dcc911ebea750d9.png?sha=d7689ceccbbfb95c8e991e75ece002953ab7f52f)

※図中の Hook Control Mesh は、この節でいう「Hookによる制御構成」の旧呼称として読んでほしい。

この図で見てほしいのは、Hook が統括AI・審査ゲートAI・実務実行AIの中に入っていないことだ。

Hook は判断役ではない。  
実装役でもない。

でも、全員の実行境界に効く。

| レイヤ | 主責務 | Hookとの関係 |
| --- | --- | --- |
| 統括AI | 戦略判断、ユーザー対話、最終承認 | 入力監査・ツール実行ガードの対象 |
| 審査ゲートAI | 三観点審査、Pulse分割、実行役への委譲 | サブエージェント起動/終了が観測される |
| 実務実行AI | 実装・調査・検証 | ツール実行ガードが共通で適用される |
| Hook | 入力監査、deny、ログ、trace、終端処理 | 判断はしないが境界を守る |

この分離がかなり効いた。

エージェント定義に責務を書く。  
Hook に実行境界を書く。  
ログに実際の挙動を残す。  
独自のログ基盤・制御プレーン側で次の制御へ戻す。

この循環ができると、「うまくいかなかった」がただの感想ではなくなる。

## プロンプトで守るもの、Hookで守るもの

最終的に、役割分担はこうなった。

| 守るもの | 主な置き場所 |
| --- | --- |
| 価値判断、責任境界、委譲方針 | Custom Agent の instructions |
| 入口の危険入力 | `userPromptSubmitted` で監査・記録 |
| ツール実行前の危険操作 | `preToolUse` で deny |
| 実行結果、失敗、deny | `postToolUse` |
| サブエージェント委譲の痕跡 | 独自の委譲ログ |
| 圧縮後に残すべき最小方針 | 独自の圧縮前保全 |
| セッション終端、学習ループ | `sessionEnd` |

プロンプトに全部書くと、文脈が重くなる。  
Hook に全部持たせると、制御層が硬すぎる。

だから、分ける。

**エージェントには判断の型を持たせる。Hook には実行境界と観測を持たせる。**

この分離が、Custom Agents を実務で使う上でかなり重要だと思っている。

## 小さく真似するなら

ここまでの構成をいきなり全部入れる必要はない。

最小構成なら、まずこの3つでよいと思う。

### 1. `userPromptSubmitted`

危険な破壊操作や secret らしき文字列を入口で監査する。

最初から完璧な検出にしなくていい。  
まずは「明らかに危ない依頼が来ている」ことを機械的に記録するだけでも効果がある。

ツール実行前に、破壊的操作、無期限実行、secret 混入を検査する。

特に terminal 実行と GitHub 操作は、後から監査しやすい単位にしておくと効く。  
強制的に止めたいものは、ここで `permissionDecision: "deny"` を返す。

### 3. `postToolUse`

成功 / 失敗 / deny / duration を JSONL に残す。

最初は分析ダッシュボードを作らなくていい。  
JSONL で残しておくだけでも、後から「どこで詰まったか」を追える。

## まとめ

相棒を作り始めた頃は、エージェント定義を強くすれば制御できると思っていた。

でも、運用してみると違った。

エージェント定義は、判断基準を揃えるには効く。  
しかし、実行時の境界、ログ、失敗、ハング、サブエージェント委譲の痕跡までは、それだけでは足りない。

そこで Hook を入れた。

Hook を入れたことで、相棒は「賢く答えるもの」から少し変わった。

**どこで止まり、何を実行し、何に失敗し、次に何を直すべきかが残るもの**になった。

Custom Agents を本気で使うなら、エージェントの中身だけでなく、その外側の制御層を見る必要がある。

自分にとって Hook は、そのための実装だった。
