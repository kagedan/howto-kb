[2026 Agentic Coding Trends Report](https://resources.anthropic.com/hubfs/2026%20Agentic%20Coding%20Trends%20Report.pdf)では、プロンプトエンジニアリングからコンテキストエンジニアリングへの移行というパラダイムシフトが示されています。

コンテキストエンジニアリングの具体的な手法として、Every.toで紹介されている[Compound Engineering](https://every.to/guides/compound-engineering)（複利的エンジニアリング）を考えた場合、Claude Codeの各種機能の捉え方も変わってくるように感じました。

複利的エンジニアリングと、そのためのコンパウンド工程及びそれらのClaude Codeでの実践方法についてまとめました。

---

## 複利的エンジニアリングについて

Every.toの[Compound Engineering](https://every.to/guides/compound-engineering)で以下のように紹介されています。

> *"The core philosophy of compound engineering is that each unit of engineering work should make subsequent units easier—not harder."*

> 複利的エンジニアリングのコアとなる考え方は、エンジニアリングの各作業単位が、次の作業をより難しくするのではなく、より簡単にしていくべきだということです。

開発作業の一つひとつが、開発フローのコンテキストと相互に作用しながらコンテキストをスムーズにしていく——そうした積み重ねを意図的に行うアプローチが、複利的エンジニアリングということだと解釈できそうです。

そして、この記事において、具体的には従来の開発プロセスに **コンパウンド** という工程を追加することが提案されています。

```
Plan → Work → Review → コンパウンド → Repeat
```

---

## コンパウンド

**コンパウンド** とは、作業から得た知見・手順を、次の作業でAIエージェントが参照できるコンテキストとして記録・整理するステップです。  
この工程こそが、複利的エンジニアリングのコアコンセプトと言えそうです。

**コンパウンド** 工程で起きることを整理すると以下のとおりです。

* **コンテキストが明確になる** → 「何をすべきか」の境界が見える
* **タスクが明確になる** → 曖昧さがなくなり、AIエージェントに委譲できるようになる
* **AIエージェントに委譲できるタスクが増える** → 人間はより抽象度の高い判断に集中できる

さらに、コンテキスト内のタスクは関連し合っていることも多く、あるタスクが明確になると、それに関連するタスクとの境界も自然と見えてくることとなります。  
そのようにして、コンテキスト全体がエンジニアリングされていきます。

コンパウンドとは、部分的にでもコンテキストを明確にしていくプロセスと解釈できそうに思います。

振り返りによって未来の活動を行いやすくするという点では、従来のナレッジマネジメントと同様だと言えるかもしれません。  
エージェンティックコーディングが身近となった現在、コンテキストを整備・構造化することが、AIエージェントとの協働の質を高めることにつながります。  
そして、そのコンテキスト整備もAIエージェントに任せられるようになってきています。  
それが **コンパウンド** 工程の現代的な意味と言ってもいいかもしれません。

## Claude Codeでの実践：コンパウンド工程に関わるファイルと機能

Claude Code は機能を継続的に拡張しているため、あくまでコンパウンド工程に関わるしくみの一例として紹介します。

| 機能 | コンパウンドにおける役割 |
| --- | --- |
| `CLAUDE.md` | 常時読み込まれるコンテキスト |
| `.claude/rules/` | 関心事ごとに分割されたルール。パス単位で動的に読み込まれる |
| `memory/MEMORY.md` | セッションをまたいで蓄積する発見・判断履歴 |
| `.claude/skills/` | `/スキル名` で呼び出せる、結晶化されたワークフロー |
| Hooks（`SessionEnd` など） | セッション終了などのタイミングで、コンパウンド処理を自動起動できる |
| Agents（サブエージェント） | コンパウンド処理を並列実行できる。 |

Every.toのガイドでは Review の後にコンパウンド工程が置かれていますが、コンパウンドは作業の区切りならいつでも実行することができます。  
Every.toのガイドでは `/workflows:compound` コマンドを使い、コンテキストアナライザー（会話内容を解析するエージェント）・ソリューション抽出など複数のサブエージェントが並列動作して学びを自動でドキュメント化することが紹介されています。

## skill-creator

[skill-creator](https://github.com/anthropics/skills/tree/main/skills/skill-creator) は、Anthropic から公開されているスキルです。

skill-creator をインストールしておくことで、コンパウンド工程において、Claude が「この知見はスキルにしたほうがいいな」と思った場合に skill-creator を自動で呼び出してスキルを作成してくれるかと思います。

## コンパウンド工程の実践

コンパウンド工程で行うことはプロジェクトによって異なると思いますが、2つの概念的なステップとして共通していると思います。

**ログ（Log）** — 気づきを外部化する

作業から得た学びを言語化します。何がうまくいったか、繰り返し直面した問題、次回も使えそうなパターンなどです。  
蓄積するかどうかは後で判断できます。まず元情報とすることを目的としています。

**適用（Apply）** — コンテキストへ変換する

気づき情報をコンテキストへ変換します。  
Claude がコンテキストへの適用方法を決め、必要に応じて skill-creator を呼び出してスキルの作成などが行われます。

every.to のガイドでは、このフローを実装した **compound-engineering プラグイン** が公開されています。実際の実装の一例として参考になるかと思います。

## Context Rot：コンテキストの鮮度を保つ

コンテキストは量が多ければいいというものでもなく、古い情報・矛盾した情報が蓄積すると、AIの精度を低下させる原因にもなってきます。  
英語では **Context Rot** と表現されています。

Bojie Li（[Claude's Context Engineering Secrets](https://01.me/en/2025/12/context-engineering-from-claude/)）は4パターンに整理しています。

| パターン | 内容 |
| --- | --- |
| Poisoning | 古い・誤った情報が残存し、正確性を阻害する |
| Distraction | 無関係な情報がフォーカスを散漫にする |
| Confusion | 類似情報の混在が判断を曇らせる |
| Clash | 矛盾する記述が動作を不安定にする |

> *"For each line, ask: 'Would removing this cause Claude to make mistakes?' If not, cut it."*

> 各行について、「これを削除すると Claude がミスをするか？」と問いかけます。そうでなければ、削除します。

削ることもコンテキストを明確にする行為と言えそうです。

## まとめ

* **コンパウンド** 工程を加えることで、複利的エンジニアリングとエージェンティックコーディングの解像度が上がる
* Claude Codeの各機能（CLAUDE.md、rules/、スキル、Hooks）は、コンパウンド工程を実践するための仕組みとして捉えられる
* **Context Rot** に注意しながら、削ることも含めてコンテキストをエンジニアリングし続けることが重要

コンパウンドを工程として捉え、プロジェクトごとの開発フローをコンテキストとしてエンジニアリングしていくことがエージェンティックコーディング時代において重要な意味を持ってくると感じています。
