## はじめに: 「覚えておいて」が機能しない問題

Claude Codeを数ヶ月使い込んでいると、こんな経験はないでしょうか：

* 「覚えておいて」と伝えたのに、次のセッションで全く活用されていない
* 長いセッションの後半で、「さっき伝えた指示」が忘れられている
* スキルで得られた知見が、次の実行で全く反映されない

私も同じ問題に悩まされていました。「メモリがただ溜まっていくだけ」の状態です。

そこで、nogataka氏のQiita記事シリーズをきっかけに、2セッションかけて徹底調査しました。公式ドキュメント3件、技術記事12件を読み、Codex(GPT-5.4)・Gemini(3.1 Pro)・独自レビューエージェントの3者レビューも経て、**検証済みの知見だけ**をこの記事にまとめました。

## 壁①: MEMORY.mdは「先頭200行」しか読まれない

### 見落としていた事実

Claude Codeのメモリシステムには、**ほとんどのユーザーが知らない重大な制約**があります。

**MEMORY.mdの先頭200行だけ**がセッション開始時に自動ロードされます。200行を超えた部分は**完全に切り捨てられます**。圧縮ではなく、切り捨てです。

そして、MEMORY.mdからリンクされたトピック別ファイル（`feedback_*.md` や `project_*.md`）は**自動ロードされません**。Claudeが「このファイルが必要だ」と判断した時に初めてReadツールで取得しに行きます。

この判断はMEMORY.md内の各エントリの「説明文（description）」に大きく依存します。つまり、**MEMORY.mdのインデックス品質が、メモリ活用率を直接的に決定する**のです。

### 対策: L0インデックス化

各エントリに **「いつ参照すべきか」のコンテキストヒントタグ** を追加します。

**Before:**

```
- [feedback_hook.md](feedback_hook.md) — Hook設計のフィードバック
```

**After:**

```
- [feedback_hook.md](feedback_hook.md) — Hook設計: tmpファイル❌ タグベース⭕ [Hook作成・修正時] [Windows互換性]
```

これはOpenVikingのL0/L1/L2（3層コンテキスト構造）の考え方を、Claude Codeの標準機能で実現したものです。L0（メタデータ）で全体を俯瞰し、必要な部分だけL2（詳細）に掘り下げる設計です。

## 壁②: 会話中の指示が「消える」

### なぜ後半で品質が落ちるのか

コンテキストウィンドウの\*\*約83.5%\*\*が埋まると、自動的に「コンテキスト圧縮（Compaction）」が発動します。

この時、**CLAUDE.mdとrulesはディスクから再読込されて完全に生存**します。しかし、**会話中にだけ伝えた指示は全て消えます**。

なお、「Smart Zone (~40%)」という概念が一部の記事で紹介されていますが、**公式裏付けはありません**。公式に確認できているのは「約83.5%で圧縮発動」のみです。

### 対策: context-management.md + `/compact`

`~/.claude/rules/context-management.md` を作成し、「重要な指示はCLAUDE.mdかrulesに書く」というルールを明文化。また、長時間タスク時は `/compact 計画書の進捗と次ステップを保持` で圧縮後も核心情報を維持できます。

当初は「計画書をCLAUDE.mdに一時追記」する案もありましたが、CodexとGeminiの両方が「エラー中断時に削除されずCLAUDE.mdが汚染されるリスク」を指摘。`/compact` で対応する方針に修正されました。

## 壁③: スキル実行の知見が「揮発」する

### サブエージェントのメモリ障壁

スキル（`skills/*.md`）には**memoryフィールドがありません**。エージェント定義（`agents/*.md`）のみが `memory: user/project/local` を持てます。

つまり、レビュースキルや調査スキルが実行中に得た知見は、**そのセッションが終わると消えます**。

### 提案Eの経緯: スキル→エージェント移行を検討し、破棄した

スキルをエージェント定義に移行してmemoryを活用する案も検討しましたが、**3者全員が破棄を推奨**しました。理由は:

1. **Progressive Disclosure（必要な時だけ詳細を読み込む仕組み）が失われる** — references/フォルダ（スキルの補助資料置き場）による段階的開示が使えなくなる
2. **自動トリガーが失われる** — description検索によるスキル自動マッチングが使えなくなる
3. **同期ルールの大幅改修が必要** — agents/はskills/と別ディレクトリ

### 対策: ルールで知見キャプチャ

`~/.claude/rules/memory-hygiene.md` を作成し、「スキル実行後に非自明な知見があればメモリ保存を提案」するワークフローを定義。スキルの構造を変えずに、**ルールで知見の揮発を防止**します。

## 3者レビューで得られた重要な学び

今回の提案をCodex(GPT-5.4 xhigh)・Gemini(3.1 Pro Preview)・独自レビューエージェントの3者でレビューしたところ、**予想外の指摘が多数**ありました。特に重要だった3つ:

### 1. 検証不能なルールは形骸化する（3者一致）

「焦ったら止まれ」「検証を省略しない」のような定性的規範をrulesに書いても、**hookで自動検証しない限り形骸化**します。筆者環境の品質チェックルール（"anti-satisficing"と命名）が有効なのは、stop hookが出力フォーマットを自動検証しているからです。同じ検証メカニズムがないルールは「書いてあるだけ」になります。

### 2. AIの自律性に過度な期待をしない（Gemini）

「設定ファイルの構造いじりより、タグ付けとガードレールで制御すべき」。CLAUDE.mdを動的に編集させる提案は、エラー中断時に汚染されるリスクが指摘されました。

### 3. L0/L1/L2は標準機能で実現可能（Codex）

OpenVikingの3層コンテキスト構造は、Claude Codeの既存機能で実現できます。MEMORY.mdのタグベースインデックスがL0、paths付きrulesとサブディレクトリCLAUDE.mdの遅延ロードがL1、トピック別ファイルのオンデマンド取得がL2に相当します。

## まとめ: 実装した4つの改善

| # | 施策 | 解決する問題 | 工数 |
| --- | --- | --- | --- |
| A | MEMORY.mdのL0インデックス化 | メモリが使われない | 5分 |
| B | context-management.md | 指示が消える | 15分 |
| C | memory-hygiene.md | 知見が揮発する | 15分 |
| D | グローバルCLAUDE.md | 毎回同じ指示を出す | 20分 |

メモリは「溜める」だけでは意味がありません。**「いつ・どう呼び出されるか」を設計**することで、初めて活きた知識になります。
