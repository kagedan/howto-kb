## はじめに

生成AIアプリケーションを作っていると、最初はモデルの性能やプロンプト設計に目が行きがちです。

しかし、本番運用に入ると、別の問題が見えてきます。

それが **トークンコスト** です。

PoCや個人開発では気にならなかったAPI利用料も、以下のようなケースでは急に大きくなります。

* AI Agent が1回の処理で何度もLLMを呼び出す
* RAGで長いコンテキストを毎回渡す
* チャットボットやCopilotの利用者が増える
* バッチ処理で大量の文章を要約・分類・翻訳する
* GPT / Claude など複数の主流モデルを使い分ける

AIプロダクトでは、トークンコストがそのまま粗利率に影響します。

この記事では、AI APIコストを下げる方法のひとつとして、OpenAI / Anthropic互換APIの `base_url` を切り替える考え方を整理します。

## トークンコストはなぜ増えやすいのか

AI APIのコストは、基本的に利用量に比例します。

特に注意したいのは、ユーザーから見える1回の操作と、裏側のLLM呼び出し回数が一致しないことです。

例えば、Agent系のアプリケーションでは、1つのリクエストに対して以下のような処理が走ることがあります。

* タスク分解
* 検索クエリ生成
* 外部ツール呼び出し
* 中間結果の要約
* 最終回答の生成

ユーザーにとっては1回の操作でも、内部では複数回モデルを呼び出していることがあります。

また、RAGや長文処理では、入力トークンが大きくなりがちです。出力トークンだけでなく、入力トークンもコストに効いてきます。

## よくあるコスト削減方法

AI APIコストを下げる方法はいくつかあります。

### 1. 軽量モデルを使う

すべての処理に高性能モデルを使うのではなく、分類・整形・短い要約などには軽量モデルを使う方法です。

これは有効ですが、品質とのトレードオフがあります。

### 2. プロンプトを短くする

不要なシステムプロンプトや会話履歴を削ることで、入力トークンを減らせます。

ただし、出力品質や挙動が変わる可能性があります。

### 3. キャッシュする

同じ入力に対して同じ出力でよい場合は、キャッシュが有効です。

一方で、ユーザーごとに入力が変わる用途では使いづらい場合もあります。

### 4. APIの接続先を切り替える

もうひとつの選択肢が、OpenAI / Anthropic互換APIを使い、既存のSDKやリクエスト形式を大きく変えずに、`base_url` だけを切り替える方法です。

この記事ではこの方法を扱います。

## base\_urlを切り替えるとは

OpenAI SDKを使っている場合、通常は公式APIにリクエストを送ります。

一方、OpenAI互換APIを提供しているゲートウェイやプロバイダーを使う場合、接続先の `base_url` を変更することで、同じようなリクエスト形式のまま別のAPIエンドポイントへ送ることができます。

アプリケーション側で主に扱う項目は以下です。

* `api_key`
* `base_url`
* `model`
* `messages`
* `temperature`
* `stream`

整理すると、以下のような流れです。

```
1. アプリケーションから互換API形式でリクエストを送る
2. base_url に指定したAPIゲートウェイへ届く
3. ゲートウェイ側で利用可能なモデルルートに振り分ける
4. GPT / Claude などの主流モデルからレスポンスが返る
5. アプリケーション側では通常のAPIレスポンスとして扱う
```

つまり、アプリケーション側の変更点を小さくしながら、接続先だけを切り替えて検証できるのが特徴です。

## Pythonでの例

通常のOpenAI SDK利用例です。

```
from openai import OpenAI

client = OpenAI(
    api_key="YOUR_OPENAI_API_KEY"
)

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "user", "content": "日本語で短い自己紹介を書いてください。"}
    ],
)

print(response.choices[0].message.content)
```

互換APIゲートウェイを使う場合は、`base_url` を指定します。

```
from openai import OpenAI

client = OpenAI(
    api_key="YOUR_API_KEY",
    base_url="https://YOUR_API_BASE_URL/v1"
)

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "user", "content": "日本語で短い自己紹介を書いてください。"}
    ],
)

print(response.choices[0].message.content)
```

実際の `base_url`、APIキー、利用可能なモデル名は、利用するサービスのドキュメントに従ってください。

## Node.jsでの例

Node.jsでも考え方は同じです。

```
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.AI_API_KEY,
  baseURL: process.env.AI_API_BASE_URL,
});

const response = await client.chat.completions.create({
  model: process.env.AI_MODEL,
  messages: [
    {
      role: "user",
      content: "要点を3つにまとめてください。",
    },
  ],
});

console.log(response.choices[0].message.content);
```

環境変数で分けておくと、接続先の切り替えがしやすくなります。

```
AI_API_KEY=your_api_key
AI_API_BASE_URL=https://YOUR_API_BASE_URL/v1
AI_MODEL=gpt-4o-mini
```

## コストを比較するときの考え方

AI APIのコストは、ざっくり以下で考えられます。

```
APIコスト = 入力トークン数 × 入力単価 + 出力トークン数 × 出力単価
```

例えば、月間で以下のような利用があるとします。

```
入力トークン: 100,000,000
出力トークン: 20,000,000
```

この場合、モデルごとの入力単価・出力単価によって、月額コストは大きく変わります。

比較するときは、単純な平均単価だけでなく、以下を分けて見るのがおすすめです。

* 入力トークン
* 出力トークン
* モデル別利用量
* 用途別利用量
* 成功リクエスト
* リトライや失敗リクエスト
* バッチ処理とリアルタイム処理

以下は、仕組みを説明するための仮の単価による計算例です。実際の単価は利用するモデルやプロバイダーによって異なります。

```
input_tokens = 100_000_000
output_tokens = 20_000_000

input_price_per_1m = 0.15
output_price_per_1m = 0.60

cost = (
    input_tokens / 1_000_000 * input_price_per_1m
    + output_tokens / 1_000_000 * output_price_per_1m
)

print(f"monthly cost: ${cost:.2f}")
```

出力例です。

## 切り替える前に確認したいこと

`base_url` を変更するだけで接続先を切り替えられる場合でも、本番投入前には確認が必要です。

### モデル名の対応

互換APIであっても、利用できるモデル名が完全に同じとは限りません。

確認すべき項目は以下です。

* 使いたいモデルが利用可能か
* モデル名は同じか
* 代替モデルを使う必要があるか
* 入力トークン上限は同じか
* 出力トークン上限は同じか

### パラメータの互換性

利用するゲートウェイによって、対応しているパラメータの範囲は異なります。

例えば、以下のような項目です。

* `temperature`
* `top_p`
* `max_tokens`
* `stream`
* `tools`
* `tool_choice`
* `response_format`

function calling / tool calling など公式API固有の機能を使っている場合は、利用するゲートウェイ側で対応しているか、実際のレスポンス形式まで確認した方が安全です。

### レイテンシと安定性

コストが下がっても、レスポンスが遅すぎるとユーザー体験に影響します。

特にリアルタイムチャットではレイテンシが重要です。一方、バッチ処理ではレイテンシより単価の影響が大きい場合もあります。

確認すべき項目は以下です。

* レスポンス時間
* 5xxエラー率
* タイムアウト率
* レート制限
* リトライ時の挙動
* 利用量ログの確認方法
* 障害時の切り戻し手順

### データの扱い

企業利用では、送信データの扱いも確認が必要です。

* 入力内容がログ保存されるか
* 保存期間はどれくらいか
* 個人情報や機密情報を送れるか
* 利用規約上の制限はあるか
* セキュリティレビューに必要な情報があるか

## 小さく検証する手順

いきなり全トラフィックを切り替えるのではなく、小さく検証するのがおすすめです。

### Step 1: 開発環境で疎通確認

まずは開発環境で `base_url` を変更します。

```
AI_API_BASE_URL=https://YOUR_API_BASE_URL/v1
AI_API_KEY=your_test_key
```

確認することは以下です。

* APIが正常に返るか
* レスポンス形式が期待通りか
* 既存コードで例外が出ないか

### Step 2: 低リスクな処理で試す

最初は、ユーザー体験への影響が小さい処理で試すのが安全です。

例えば、以下のような処理です。

* 社内用の要約
* バッチ分類
* 開発用テスト
* 非同期のコンテンツ生成
* ログ分析やデータ整形

### Step 3: 結果を比較する

以下を比較します。

| 項目 | 確認内容 |
| --- | --- |
| 品質 | 出力内容が実用レベルか |
| コスト | 同じトークン量でどれくらい変わるか |
| レイテンシ | 許容範囲か |
| エラー率 | 本番利用できる水準か |
| 運用性 | ログや利用量を確認しやすいか |

### Step 4: 一部トラフィックで試す

問題がなければ、一部の本番トラフィックで試します。

例えば、以下のように段階的に切り替えます。

* 5%だけ切り替える
* 特定機能だけ切り替える
* 特定ユーザーグループだけ切り替える
* バッチ処理だけ切り替える

この段階で、品質・コスト・安定性を見ながら広げていきます。

## どんなケースに向いているか

`base_url` の切り替えによる検証は、以下のようなケースに向いています。

### AI Agent

Agentは内部で複数回モデルを呼び出すため、トークンコストが増えやすいです。

### AI SaaS

ユーザー利用量に応じてAPIコストが増えるため、粗利率に直結します。

### コンテンツ生成

文章生成、要約、翻訳、分類などを大量に実行する場合、単価差の影響が大きくなります。

### 社内AIツール

複数部署で生成AIを使う場合、利用量管理とコスト管理が重要になります。

### 開発・検証用途

Claude CodeやGPTなどを使いながら開発・検証する場合、検証コストを抑えやすくなります。

## 向いていない可能性があるケース

一方で、以下のようなケースでは慎重に検証した方がよいです。

* 超低レイテンシが必要なリアルタイム処理
* 特定モデルの細かい挙動に強く依存している
* function callingの互換性が重要
* セキュリティ要件が非常に厳しい
* 公式API以外を使えない契約・社内ルールがある

コストだけで判断せず、品質・安定性・運用要件も含めて比較する必要があります。

## まとめ

AI APIのコストは、本番運用に入ると大きな課題になります。

特に、AI Agent、AI SaaS、バッチ処理、コンテンツ生成のようにトークン消費が大きい用途では、モデル単価の差がそのまま利益率に影響します。

OpenAI / Anthropic互換APIを使って `base_url` を変更する方法は、既存コードを大きく変えずに接続先を切り替えられるため、小さく検証しやすいアプローチです。

ポイントは以下です。

* トークンコストは本番運用で急に大きくなる
* 互換APIなら既存SDKを活かしやすい
* `base_url` の変更だけで接続先を切り替えられる
* コストだけでなく、品質・レイテンシ・互換性・可用性を見る
* まずは開発環境や低リスクな処理から試す
