Claude Code Maxプランの5時間枠が15分で消える。

3月末からこの現象に悩まされている人は多い。GitHub Issueには91件以上の報告が集まり、関連Issue [#40524](https://github.com/anthropics/claude-code/issues/40524) には220リアクション、[#41930](https://github.com/anthropics/claude-code/issues/41930) には28リアクションと20コメントがついた。The Register、TechRadar、DevOps.comまで取り上げた。

Anthropicの公式見解は4月2日に出た。プロダクトリードのLydia Hallieが「ピーク時間の制限が厳しくなった」「セッションの1Mコンテキストが大きくなった」と説明し、対策として「Sonnetをデフォルトに」「Extended Thinkingを切る」「--resumeを避ける」を推奨した。

だが、コミュニティの実測データはその説明では足りないことを示している。

この記事では、独立研究者[ArkNill氏のclaude-code-hidden-problem-analysis](https://github.com/ArkNill/claude-code-hidden-problem-analysis)（48スター、プロキシ実測に基づく体系的分析）と、12名のコミュニティメンバーの検証を元に、7つのバグを体系的に整理する。修正済み3件、未修正4件。v2.1.92時点の情報だ。

> この記事は2026年4月時点の情報に基づいている。Claude Codeは頻繁にアップデートされるため、最新バージョンでは状況が変わっている可能性がある。

## 7バグの全体像

まず一覧を見てほしい。

| # | バグ名 | 内容 | 影響 | 状態 |
| --- | --- | --- | --- | --- |
| B1 | Sentinel | スタンドアロンバイナリのキャッシュプレフィックス破壊 | キャッシュ読み率4-17%に低下 | ✅ 修正済み |
| B2 | Resume | `--resume`がフルコンテキストを未キャッシュで再送 | 最大20倍のコスト | ✅ 修正済み |
| B3 | Cache invalidation | キャッシュ無効化で偽レート制限 | cache\_read=0でフルcache\_write | ✅ 修正済み |
| B4 | Microcompact | ツール結果がセッション中にサイレント消去 | コンテキスト品質の劣化 | ❌ 未修正 |
| B5 | Budget cap | ツール結果の200K集約上限 | 古い結果が1〜41文字に切り詰め | ❌ 未修正 |

| B7 | Log inflation | Extended ThinkingのJSONL重複記録 | ローカルで最大2.87倍の水増し表示 | ❌ 未修正 |

修正済みが3件。未修正が4件。ここからは順番に見ていく。

## 修正済み3件——アップデートで解消されたもの

### B1: Sentinel（キャッシュプレフィックス破壊）

Anthropicのカスタムフォーク内で、APIリクエストごとに課金用のsentinel文字列を置換する処理が走っていた。問題は、会話履歴に課金関連のキーワードが含まれると、置換が意図しない箇所にヒットすること。

キャッシュプレフィックスが壊れると、それ以降の全ターンが「初回リクエスト」として扱われる。キャッシュ読み率が4〜17%まで落ちる。通常は80%以上あるべき値だ。

v2.1.86からv2.1.88の間で修正された。最新版を使っていれば影響は受けない。

### B2: Resume（セッション再開のキャッシュ破壊）

`--resume`や`--continue`フラグでセッションを再開すると、ツール結果の挿入位置が新規セッションと異なる。キャッシュプレフィックスが丸ごと無効化され、会話全体が再処理される。

長いセッションほど影響が大きい。セッションが100ターンを超えていると、再開1回で通常の20倍のトークンが消費されるケースが報告されていた。

v2.1.89以降で修正された。ただし、v2.1.89以前のセッションをv2.1.90以降で再開すると、同様の問題が発生する旨が[#42749](https://github.com/anthropics/claude-code/issues/42749)で報告されている。古いセッションの再開は避けた方がいい。

### B3: Cache invalidation（キャッシュ無効化と偽レート制限）

会話履歴がターンごとに意図せず変更され、キャッシュが丸ごと無効化される問題。[#40524](https://github.com/anthropics/claude-code/issues/40524)で220リアクション。7バグの中で最も影響範囲が広かった。

症状は明確だ。`cache_read`が0になり、全てが`cache_write`として処理される。毎ターンが初回リクエスト扱い。

4月4日に[#40524がCLOSED](https://github.com/anthropics/claude-code/issues/40524)された。[@qing-ant](/qing-ant "qing-ant")による修正。ただし、修正が含まれる正確なバージョンは未確認のため、最新版へのアップデートを推奨する。

## 未修正4件——今も影響を受けているバグ

ここからが本題だ。4件とも2026年4月5日時点でv2.1.92において未修正。Anthropicからの公式応答もない。

### B4: Microcompact（サイレントなツール結果消去）

セッション中、Claude Codeは3つのコンパクション機構をサイレントに実行している。コンテキストウィンドウを管理するための仕組みだが、問題はユーザーに通知なく動くこと。

ArkNill氏の測定によれば、1セッションで158回以上のclearing event（ツール結果の消去）が発生していた。GrowthBookのA/Bテストフラグ（`tengu_hawthorn_window`、`tengu_pewter_kestrel`、`tengu_summarize_tool_results`）で制御されており、フラグが「disabled」に設定されていても発動するケースが確認されている。

ユーザーSn3thは「impossible to work with right now（今の状態では作業にならない）」とコメントしている。

`DISABLE_COMPACT=true`を設定しても、このサイレント消去は防げない。`DISABLE_COMPACT`はユーザー向けのcompactコマンドを無効化するだけで、内部のmicrocompactは別の経路で動く。

**確認方法**: セッションのJSONLログ（`~/.claude/projects/`配下）で、ツール結果の内容が途中から空や極端に短い文字列に変わっていないか確認する。

**対処法**: セッションを長引かせない。30ターンを超えたら新しいセッションを開始する。[#42542](https://github.com/anthropics/claude-code/issues/42542)で報告されているが、Anthropicからの応答はない。

### B5: Budget cap（200K集約上限）

ツール結果には200K文字の集約上限（`tengu_hawthorn_window`）がある。この上限を超えると、古いツール結果から順番に1〜41文字まで切り詰められる。

ArkNill氏の測定では、1セッションで**261回のbudget event**が発生。つまり261回、過去のツール結果が切り詰められた。Readで読んだファイルの内容、Bashの実行結果、Grepの検索結果——全てが対象だ。

v2.1.91で`maxResultSizeChars`の修正が入ったが、これはMCPツール専用。**Read、Bash、Grepなどの内蔵ツールには適用されない**。

Claude Codeが「さっき読んだはずのファイルの内容を忘れている」現象に覚えがあるなら、このバグが原因の可能性が高い。

**確認方法**: セッションJSONLで、ツール結果の`content`フィールドが数文字まで短縮されていないか確認する。

**対処法**: 1回のツール呼び出しで大量のデータを読まない。ファイルは必要な部分だけ`offset`と`limit`で指定して読む。[#42943](https://github.com/anthropics/claude-code/issues/42943)で専用報告がある。

### B6: False rate limit（偽レート制限）

ArkNill氏の65セッション分析で、151個の`<synthetic>`エントリが発見された。`model:"<synthetic>"`、`input_tokens:0`——つまりAPIを実際に呼び出していないのに、レート制限のカウントに計上されるエントリだ。

ユーザーrwp65は2.9MBの比較的小さなセッション（通常の1/25サイズ）でも再現を報告。セッションサイズではなく、並行サブエージェントの実行が原因である可能性が高い。

症状は「まだ全然使っていないのにレート制限に引っかかる」。/usageで確認すると使用率は低いのに、Claude Codeが「Usage limit reached」を表示する。

**確認方法**: セッションJSONLで`"model":"<synthetic>"`を検索する。

**対処法**: 現時点で有効な回避策はない。サブエージェントの並行実行を減らすことで発生頻度を下げられる可能性はある。[#40584](https://github.com/anthropics/claude-code/issues/40584)で報告中。

### B7: Log inflation（ログのトークン水増し）

セッションのJSONLログにExtended Thinkingのトークン数が重複して記録される。ArkNill氏の測定結果:

* サブエージェント: 1.71倍〜2.67倍の膨張
* メインエージェント: 2.87倍の膨張

`/cost`コマンドで表示されるトークン消費量が実際の1.97倍に水増しされる。これはサーバー側の実課金には影響しない（ローカルログの問題）が、ユーザーが消費量を正確に把握できなくなる。

v2.1.92のwhitespace修正はこの問題とは無関係。v2.1.92で`/cost`にキャッシュヒット内訳が追加されたが、水増し問題自体は修正されていない。

**確認方法**: セッションJSONLで同一メッセージIDに対してトークン数が複数回カウントされていないか確認する。

**対処法**: `/cost`の表示値を鵜呑みにしない。実際の消費量は表示の約半分と見積もる。[#41346](https://github.com/anthropics/claude-code/issues/41346)で報告中。

## 今すぐできる対処法

7つのバグを把握した上で、今すぐできることをまとめる。

### 1. 最新版にアップデートする

B1〜B3は修正済み。古いバージョンを使い続ける理由はない。

### 2. --resumeを避ける

B2は修正済みだが、古いセッションの再開は依然としてリスクがある。新しいセッションを開始する方が安全だ。

### 3. セッションを短く保つ

B4（microcompact）とB5（budget cap）は長いセッションほど影響が大きい。30ターンを目安に新しいセッションに切り替える。重要な情報はCLAUDE.mdやタスクファイルに外部化し、ツール結果に依存しない運用にすると、B5の影響を軽減できる。

### 4. ツール結果のサイズを意識する

B5の200K上限を意識して、1回のツール呼び出しで取得するデータ量を抑える。Readの`offset`/`limit`、Grepの`head_limit`を活用する。

### 5. /costの値を疑う

B7のlog inflationにより、`/cost`の表示は最大2.87倍に水増しされている可能性がある。「思ったより消費が激しい」と感じたら、まずこのバグを疑う。

v2.1.92で`/cost`にモデル別・キャッシュヒット別の内訳が追加された。これを使えば`cache_read`と`cache_creation`の比率を直接確認できる。ただしB7の水増し問題自体は修正されていないため、表示される合計値は依然として実際より大きい。

### 6. JSONLログで自分のセッションを診断する

各バグの確認方法を書いたが、JSONLログの場所と具体的な確認コマンドをまとめておく。

```
# セッションログの場所（最新のJSONLを探す）
ls -lt ~/.claude/projects/*/*.jsonl | head -5

# B3: キャッシュが効いているか（cache_readが0なら異常）
grep -o '"cache_read_input_tokens":[0-9]*' SESSION_FILE.jsonl | tail -5

# B4/B5: ツール結果が消去・切り詰めされていないか
# contentフィールドが極端に短い（10文字以下）エントリを検出
python3 -c "
import json, sys
for line in open(sys.argv[1]):
    try:
        d = json.loads(line)
        msg = d.get('snapshot', d.get('message', {}))
        if not isinstance(msg, dict): continue
        for block in msg.get('content', []):
            if isinstance(block, dict) and block.get('type') == 'tool_result':
                c = block.get('content', '')
                if isinstance(c, str) and 0 < len(c) < 10:
                    print(f'Truncated: {c!r}')
    except: pass
" SESSION_FILE.jsonl

# B6: <synthetic>エントリの検出
grep -c '"<synthetic>"' SESSION_FILE.jsonl

# B7: /costとの比較（重複カウントの兆候）
grep -o '"output_tokens":[0-9]*' SESSION_FILE.jsonl | \
  awk -F: '{s+=$2} END {print "JSONL合計:", s, "tokens"}'
```

### 7. キャッシュ比率を自分で測定する

本当にキャッシュが効いているか確認したい場合、[ArkNill氏のcc-relay](https://github.com/ArkNill/claude-code-hidden-problem-analysis)を使ってプロキシ経由でAPIリクエストを監視できる。健全なセッションなら`cache_read`が`cache_creation`の4倍以上ある。逆転していたら何かがおかしい。

## Anthropicの公式見解との乖離

整理しておく。

Lydia Hallieの4月2日の説明は「ピーク時間の制限強化」と「1Mコンテキストの肥大化」の2点。これは間違いではないが、**7バグのうちB4〜B7には一切触れていない**。

ArkNill氏の分析が示しているのは、「制限が厳しくなった」だけでは説明できない構造的な問題の存在だ。GrowthBookフラグの矛盾（B4）、内蔵ツールに適用されないサイズ制限の修正（B5）、APIを呼んでいないのにカウントされるエントリ（B6）。これらはピーク時間やコンテキストサイズとは独立した問題だ。

## 追記（2026-04-17）: 8つ目 — Opus 4.7新トークナイザー

4月16日にOpus 4.7がデフォルトモデルになった。APIの単価は据え置きだが、**新しいトークナイザーが同じテキストで最大35%多いトークンを生成する**（[Finout.io分析](https://www.finout.io/blog/claude-opus-4.7-pricing-the-real-cost-story-behind-the-unchanged-price-tag)）。コード・構造化データ・非英語テキストで特に顕著。

コーディングエージェントの試算: **$10/day → $13.50/day（+35%）**。

さらに深刻なのは、auto modeの安全分類器がOpus 4.6にハードコードされていた問題（[#49618](https://github.com/anthropics/claude-code/issues/49618)）。3日間で20件以上のデータ損失が報告されている。

対策はPreToolUseフックの導入。モデルバージョンに依存しない安全装置になる。

7つのバグ。修正済み3件、未修正4件。

修正済みの3件（B1 Sentinel、B2 Resume、B3 Cache invalidation）は最新版へのアップデートで解消する。特にB3は最大の影響範囲を持っていたバグだ。（参考: Maxプラン$100/月でOpus利用の場合、キャッシュ無効化1回で通常の10〜20倍のトークンが消費される計算になる。$100の枠が実質$5〜10相当まで縮む。）
