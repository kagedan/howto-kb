# AI駆動開発におけるチーム開発の問題点

「AIのおかげで、開発楽になったー。やったー」って思うかもしれないですが、

1.***指示の出し方がメンバーによって違う***  
これをやるとコードがぐちゃぐちゃになってしまい、バグや手戻りなどを起こしてしまうリスクが高くなります。💦

2.***設計思想、コーディング規約が守られない***  
品質担保できなくなってしまい、長期運用するのが、難しくなってきてしまいます。😱

3.***レビューコスト高い***  
品質がぐちゃぐちゃになるので、リーダーなどのレビューに負担が大きくなります。💦  
AIにレビューをさせるにせよAIのレビューの負担も大きくなってきます。

4.***毎回AIに説明する必要がある***  
実装のたびに、規約やルール、プロジェクト概要などを伝えるってかなり面倒です。  
これめんどくさくないですか💦

5.***AIが書いたから大丈夫と思い込む(セキュリティや品質は保証がない)***  
AIは普通に間違えますし、こちらの意図を全部汲み取ってくれないです。😢  
あくまで作業スピードがめちゃくちゃ速いだけなんです。

# 環境を整備しよう

***上記の問題を解消するために、AIと人間が一緒に安全に開発できる道標が必要***ってことです。そのためには、まずは***環境整備をしていく必要***があります。  
これは個人の技術力とか経験値の問題ではなく、AIが開発しやすくするための  
環境整備ができていないってことなんです。

# ハーネスエンジニアリング

そこで登場するのが、***ハーネスエンジニアリング***です！  
***AIが暴走しないで、安全に開発できるようにすること***です。  
***AIが安心して開発できるようにするための安全装置***みたいなものですね。  
ハーネスって元々「馬具(手綱)」って意味なんですね。  
ロッククライミングなどでつける命綱や  
ジェットコースターなどでつける安全ベルトです。

# CLAUDE.md

***1番重要なファイルで、これはプロジェクトの説明書***です。  
***Claude Codeは、このファイルがルートにあると自動で毎回読み込んで***くれます。

ex)  
プロジェクトの概要、規約、設計、コマンドなど

今回はQiitaクローンを想定してやります。

```

## プロジェクト概要

Qiitaのような技術記事の投稿・共有プラットフォームのクローンです。

本プロジェクトはハーネスエンジニアリング講座のサンプルアプリとして作成します。

- 記事へのコメント

- PostgreSQL

- RuboCop（Lint / フォーマッタ）

### フロントエンド（`frontend/`）

- コミットは Conventional Commits 形式で書く

### CLAUDE.mdの注意点

***200行以上書いたりして長すぎると精度が下がり、トークンを無駄に使ってしまうのが難点***です。情報が多すぎると「本当に守ってほしいルール」を守ってくれないです。💦

公式も 「200行以内に収める」「オーナーを決めて、コードと同じようにレビューする」 ことを推奨しています。  
CLAUDE.mdは「全部書く場所」ではなく、「目次または概要」くらいの気持ちで、  
詳しい規約や手順は、別ファイルに分けていきましょう💪

# rules

細かいコーディング規約などは、 `.claude/rules/`に置きます。  
CLAUDE.mdに全部書くと太りすぎるので、規約はこっちに分けるのがいいです。

```
.claude/rules/
├── backend.md
├── frontend.md
└── testing.md
```

rulesの凄いところは 「***パススコープ（path-scoped）***」 が使えることです！  
ファイルの先頭に`paths:`を書くと、そのパスのファイルを触ったときだけ、  
ルールが読み込まれます。  
関係ないときはコンテキストに乗らないので、トークンの無駄がありません。  
ここでは、api.mdを扱います！

```
---
paths:
  - "backend/app/controllers/**"
  - "backend/app/serializers/**"
---

# API実装ルール

Qiitaクローンの API エンドポイントを実装・変更するときのルール。
`backend/app/controllers/` または `backend/app/serializers/` を編集するときだけ読み込まれる。

## レスポンス

- モデルを直接 `render json: model` しない。**必ず serializer を経由する**
- 一覧系のエンドポイントはページネーションを返す（`page` / `per` パラメータに対応）
- 日時は ISO 8601 形式で返す

```ruby
# ❌ NG: モデルを直接返す（不要なカラムまで漏れる）
render json: @article

# ✅ OK: serializer 経由で返すフィールドを制御
render json: ArticleSerializer.new(@article)

## HTTPステータスコード

正しいステータスコードを返すこと。

- 作成成功: `201 Created`
- 更新・取得成功: `200 OK`
- バリデーションエラー: `422 Unprocessable Entity`
- 未認証: `401 Unauthorized`
- 権限なし: `403 Forbidden`
- リソースが存在しない: `404 Not Found`

## エラーフォーマット

エラーレスポンスは必ず以下の形に統一する。フロントエンドはこの形を前提にする。

``json
{
  "error": {
    "code": "validation_failed",
    "message": "タイトルは必須です"
  }
}
``
## コントローラ

- コントローラは薄く保つ。複雑なロジックはモデル / サービスオブジェクトに置く
- パラメータは Strong Parameters で許可したものだけ受け取る
- 認証が必要なエンドポイントは `before_action :authenticate_user!` で保護する
- LGTM・ストック・コメントなど、本人以外が操作できてはいけない処理は所有者チェックを入れる

## パフォーマンス

- 関連を伴う一覧は `includes` で N+1 を防ぐ
  （例: 記事一覧 → 著者・タグ・LGTM数をまとめて取得）
```

# skills

***定型タスクをスラッシュコマンド化***します！  
`.claude/skills/` には、***繰り返しやる定型タスクの手順書を置きます***。  
各スキルは`SKILL.md`というファイルに、名前・説明・手順を書きます。

ポイントは 「***名前と説明だけ常時読み込み、本文は呼ばれたときだけ読み込む」 仕組み***になっていることなので、手順が長くても、普段はコンテキストを圧迫しません。

ex)

向いているのは、チームでやり方が決まっている手続き的な作業です。

```
ex)

- デプロイ手順

「毎回同じ説明をしてた作業」を、一度スキルにしておけば、次からは名前を呼ぶだけになります！  
***例の問題4「毎回説明するの面倒」が、これでかなり解消されます***🙌

# PR説明文の生成

git diff を読み、変更内容を下記フォーマットでまとめてください。

## 手順

1. 現在のブランチの差分（`git diff main...HEAD`）を読む
2. 変更されたファイルから、影響するドメインを判定する

   - フロントエンド: `frontend/src/features/` 配下（articles / tags / likes / comments / auth）、`frontend/src/shared/`
3. 下記テンプレートの各項目を埋める
4. PRタイトルは Conventional Commits 形式にする（例: `feat: 記事へのLGTM機能を追加`）

## 出力フォーマット

```markdown

## 変更したドメイン
（例: backend/記事, frontend/features/likes, frontend/shared/ui）

## 影響範囲

## チェックリスト
- [ ] テストを追加 / 更新した（`pnpm test` / `bundle exec rspec`）
- [ ] Lint が通る（`pnpm lint` / `bundle exec rubocop`）
- [ ] 型チェックが通る（`pnpm type-check`）
```

## ④ Hooks

ここが **ハーネスの“安全ベルト”** にあたる、超重要パートです💪

今までの**CLAUDE.md・rules・skills は、ぜんぶ「お願い」** なんです。  
Claudeはだいたい守ってくれますが、会話が長くなったりすると、  
AIが忘れてお願いを守ってくれないことがあります。  
例えば、本番DBを消すような操作は「だいたい」じゃダメで、**100%止めたい**ですよね。

そこで使うのが **Hooks（フック）** です。Hooksは、**ファイル編集・コマンド実行・セッション開始といった“イベント”をきっかけに、コードを自動実行する仕組み**で、`settings.json` などに登録します。最大の特徴は、AIの気分やコンテキストに左右されず、**決定論的に必ず動く**こと。

たとえばこんな使い方ができます。

```
- **PostToolUse**: ファイルを編集したあとに、自動で formatter / linter をかける
- **PreToolUse**: 危険なコマンドを実行する**前に**検査して、ダメなら止める
- セッション開始時に環境変数を読み込む、完了時にSlackへ通知 など
```

特に **`PreToolUse` フックは、検査スクリプトが終了コード `2` を返すとその操作をブロックできます。** 「お願い」ではなく「物理的に止める」ができるわけですね。

フックは2つの部品でできています。

* **スクリプト**（何をするか）
* **settings.json**（いつ動かすか）

### 検査スクリプト: guard.sh

実行しようとしているコマンドを読んで、**危険なら止める**スクリプトです。

**`.claude/hooks/guard.sh`**

```
#!/bin/bash
# .claude/hooks/guard.sh
# PreToolUse フック: Bashコマンドを実行前に検査し、危険なものをブロックする。
# 終了コード 2 で操作を中断し、理由(stderr)を Claude に返す。

INPUT=$(cat)

# Bashツールに渡されたコマンド文字列を取り出す（jq優先、無ければ python3）
if command -v jq >/dev/null 2>&1; then
  COMMAND=$(printf '%s' "$INPUT" | jq -r '.tool_input.command // empty' 2>/dev/null) || exit 2
elif command -v python3 >/dev/null 2>&1; then
  COMMAND=$(printf '%s' "$INPUT" | python3 -c 'import json,sys; print(json.load(sys.stdin).get("tool_input", {}).get("command", ""))') || exit 2
else
  echo "guard.sh: コマンドの解析ツール(jq / python3)が見つかりません" >&2
  exit 2
fi

# コマンドが空なら何もしない
if [ -z "$COMMAND" ]; then
  exit 0
fi

# --- 安全と判断したものは許可（早期 exit 0）---

# git commit は除外
if echo "$COMMAND" | grep -q "^git commit"; then
  exit 0
fi

# --- ここから下はブロック対象（exit 2）---

# 本番環境への直接操作をブロック
if echo "$COMMAND" | grep -q "production"; then
  echo "本番環境への直接操作は禁止されています" >&2
  exit 2
fi

# DBを破壊する操作をブロック
if echo "$COMMAND" | grep -Eq "db:(drop|reset|purge|schema:load|migrate:reset)"; then
  echo "DBを破壊する操作(db:drop / db:reset など)は禁止されています" >&2
  exit 2
fi

# ファイルの再帰削除をブロック
if echo "$COMMAND" | grep -Eq "rm +-[rf][rf]|rm +-[rf] +-[rf]"; then
  echo "rm -rf によるファイル削除は禁止されています" >&2
  exit 2
fi

# force push をブロック
if echo "$COMMAND" | grep -Eq "git push .*(-f\b|--force)"; then
  echo "force push は禁止されています" >&2
  exit 2
fi

exit 0
```

* **`exit 2` でブロック**：終了コード2を返すと、Claude Codeはその操作を中断し、`>&2`（標準エラー）のメッセージを「なぜダメだったか」としてClaudeに伝えます。
* **`production` / `db:drop` / `rm -rf` / `force push` を禁止**：CLAUDE.mdの「やらないこと」に書いた項目を、**文章の“お願い”から“物理的に止める”に格上げ**しています。
* **`git commit` は許可**：スクリプトなら「これは通す」という例外や、わかりやすいエラーメッセージを自由に書けます。例外やロジックが要るからこそ、一行の設定ではなくスクリプトにするわけです。

> 💡 ざっくり指針:  
> 「**毎回かならずYしてほしい**」→ Hooks  
> 「**こういう方針で書いてね**」→ CLAUDE.md / rules

### settings.json

スクリプトは書いただけでは発火しません。**いつ・どのイベントで動かすか**を `settings.json` で紐づけます。

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/guard.sh"
          }
        ]
      }
    ]
  }
}
```

覚えるキーはこれだけ。`PreToolUse` はツールを**実行する前**に発火するイベント、`matcher: "Bash"` はBashのときだけ反応、`type: "command"` はシェルコマンドを実行するフック、`$CLAUDE_PROJECT_DIR` はプロジェクトのルートを指す変数です。

そして忘れがちなのが**実行権限**。これを付けないと動きません⚠️

```
chmod +x .claude/hooks/guard.sh
```

> 💡 `PostToolUse`（実行“後”）のフックを使えば、ファイルを編集するたびに自動でフォーマッタをかけることもできます。`PreToolUse` が「危険を止める番人」なら、`PostToolUse` は「後片付け係」です。

なお `settings.json` には、フック登録のほかに **「何を許可し、何を禁止するか」を宣言する `permissions`** という役割もあります。これは後ほど「⑥ settings.json」で扱います。

## ⑤ Agents（サブエージェント）── 専門特化の別働隊

AIエージェントは一般に **計画（Planner）→ 実装（Generator）→ 評価（Evaluator）** のループで動きます（Claude Code固有の内部仕様ではなく、エージェント設計の考え方です）。この“標準のループ”に対して、**自分で足すカスタムの専門家**が、これから説明するサブエージェントです。

![ChatGPT Image 2026年6月26日 19_54_08.png](https://res.cloudinary.com/zenn/image/fetch/s--_evZ81x1--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/1748789/fe09b74f-327b-488f-be67-fc19e2e2fb34.png?_a=BACMTiAE)

`.claude/agents/`には、***特定の仕事を任せる“専門スタッフ***を定義できます。  
これがサブエージェントです。  
たとえば「このコード全体を調べて」とメインのClaudeに頼むと、  
調べた大量のログが会話に流れ込んで、画面がぐちゃぐちゃになりますよね。  
サブエージェントに任せると、***別の机（自分専用の作業スペース）で作業して***、  
終わったら***結果のまとめだけを持ってきてくれます***。  
途中の散らかりはあっちの机に置いてくるので、メインの会話はスッキリしたまま、  
こういう「丸投げして結果だけ受け取りたい」タスクに向いています。

```
コードベースの深掘り調査
ログの解析
依存パッケージの監査
コードレビュー専任
```

例えば、`.claude/rules/`に沿ってレビューする専任エージェントを作ってみます。

**`.claude/agents/code-reviewer.md`**

```
---
name: code-reviewer
description: 変更差分をレビューして、規約違反やバグの可能性を報告する
tools: Read, Grep, Bash
---
あなたはQiitaクローンのコードレビュー専門エージェントです。
変更差分を読み、以下を指摘してください（修正はせず、指摘のみ行うこと）。

- `.claude/rules/` の規約違反（例: serializer を通さず render している、N+1 を放置している）
- 潜在的なバグ・セキュリティ上の懸念（所有者チェック漏れ、Strong Parameters の不足など）
- テストの不足（新しいエンドポイントに RSpec が無い、など）
```

> 🔍 skillsとの使い分け:  
> いつもの作業を、この会話の中でサッとやらせたい → skills  
> 重い調査などを別の場所に丸投げして、結果だけ受け取りたい → サブエージェント

### サブエージェントの呼び出し方

使い方はかんたんで、**ターミナル（Claude Code のセッション）でふつうに会話で頼むだけ**です。専用コマンドは要りません。

これで、`code-reviewer` の **`description`（説明文）にマッチすれば、Claudeが自動でそのエージェントに振り分け**てくれます。

「確実にこのエージェントで」というときは、**名前を出すか `@` で指名**します。

```
code-reviewer で差分をレビューして     ← 名前で指定
@code-reviewer 差分をレビューして       ← @メンションで確定（いちばん確実）
```

> ⚠️ `.claude/agents/` は**起動時にしか読み込まれません**。Claude Code を立ち上げたまま新しくエージェントを作ったら、**いったん再起動**しないと反応しません。「作ったのに動かない」はだいたいこれです。

## TDD(テストで品質を保つ)

AI駆動開発こそ **TDD（テスト駆動開発）** が効きます。

理由はシンプルで、**テストが「仕様書」かつ「ガードレール」になる**からです。  
***先にテストを書いておけば、AIが書いたコードが正しいかどうかを、人間がいちいち確認しなくても自動で判定***できます。AIのおかげでテストコードを書きやすくなったので、TDDがやりやすくなりました💪。

* **Red**: まず失敗するテストを書く（＝仕様を先に決める）
* **Green**: テストが通る最小のコードをAIに書かせる
* **Refactor**: テストが通る状態を保ったまま整える

***AIが書いたコードよりも、テストが通ってからコードを書いた方が品質は保つことができます***！Ruby on RailsとかならRSpecがありますよね！  
注意点として、***見た目（UI）の頻繁に変わる部分はTDDしにくい***です。一方で、***バリデーションや状態管理などのロジックはフロントでもTDDが効きます。***（Vitestなど）。  
見た目の確認は、TDDより Playwright のE2Eやスナップショットの担当ですね。

## CIガードレール

Hooksは便利ですが、あくまで**ローカル（手元）の安全装置**です。  
最後の砦として、**CI（GitHub Actions など）にもガードレールを敷いておきましょう。**

CIには、性質の違う2種類を置くのがおすすめです。

### ① 必ず通す関門（lint / 型チェック / テスト）

これが本流を守る確実なゲートです。**通らなければマージできない**ようにします（ブランチ保護）。

* **バックエンド**: `bundle exec rubocop` / `bundle exec rspec`
* **フロントエンド**: `pnpm lint` / `pnpm type-check` / `pnpm test`

これで、**AIが書いたコードであっても、CIを通らない限りマージできない**という防御ラインができます。

### ② AIにもレビューさせる(Claude Code ActionsとCodeRabbit)

`Claude Code Actions`を使うと、PRを出すたびに、**Claudeが差分をレビューして、レビュー**をしてくれます。***AIで書く時代に、AIにも見張らせる***、という一段です。

うれしいのは、**CIで動くClaudeも `CLAUDE.md` を読む**こと。つまり、**ローカルと同じ規約でレビューされる**ので、チーム全員のPRに同じ基準が一貫して当たります。

> ⚠️ ただし、AIレビューは **「マージをするかしないかを決めるゲート」ではなく「最初のチェック」** として扱ってください。AIは見落とします（認証・決済まわりは人間レビュー必須）。**①の決定論ゲートが“強制”、②のAIレビューは“賢い補助”** ── 記事前半の「お願い vs 強制」と同じ役割分担です。

**ローカル（Hooks）× リモート（CI）** の二層に、さらに **決定論ゲート × AI補助** を重ねる。こうやって品質を“システムとして”担保するのが、ハーネスの考え方です。

> 💡 このほか、シークレットスキャンや依存監査をCIに足すと、さらに守りが固くなるみたいです。

ただClaude Code Actionsは設定が難しかったりするので、CodeRabbitとかを採用していいかもしれないです。

[知らないとヤバい😱 コードレビューしてもらえる最強ツール! CodeRabbitの導入法](https://qiita.com/Hashimoto-Noriaki/items/f1eab565027d84bc9b0b)

## MCPは使いすぎない

***Claude Codeと外部ツールを連携する仕組み***を`MCP(Model Context Protocol)`は、とても便利なんですが、**繋ぎすぎ注意⚠️** です。

ex)  
外部サービスは、FigmaやSlackなど

MCPを大量に使うと、**トークンを大量に消費して**\*しまいます。

```
- 本当に必要なものだけ繋ぐ
- 「これMCPじゃなくてSKILL.mdで足りるな」というケースも多い
```

基本的に、***CLIコマンドやSkillsがいい***です。

# AI駆動開発に向いている技術選定

AI駆動開発向きの選定方法もあったりします。ただこれはPJの状況などによって変わるので、ご注意ください。

軸は **「書き方が決まっているか」**。決まっているほど、AIの出力がばらけません。

### Ruby on Rails（Ruby）

**AIの出力がばらけないフレームワークは Ruby on Rails です。設計思想が固定化されているので、AIの出力がばらけづらい**特徴があります。スピード重視だったり、小、中規模なら  
Javaより向いています。

[AI駆動開発では、Ruby on Railsが最強な理由(Rubyはオワコンではない💪)](https://zenn.dev/norihashimo/articles/2de4da097201b8)

### Spring Boot（Java）

**Javaは静的型付けなので、AIが型の合わないコードを書いても、コンパイル（人間が書いたコードを、コンピュータが実行できる形に“翻訳”する作業）で弾けます。** さらに**Spring Bootは書き方が定型化されていて、AIの出力がばらけにくい**です。大規模案件とかなら採用するといいと思います。

**海外の大手もJavaが主力**で、Netflix は Spring Boot に標準化、Amazon・Google・Uber なども採用しています。

[How Netflix Really Uses Java（YouTube）](https://www.youtube.com/watch?v=ucJTPda_zx0)

### Go

**Goは仕様がシンプルで書き方がほぼ一通りなので、AIが書いてもブレません。** これまでは定型コードを自分で書く手間がありましたが、**AIでそこが一気に楽になりました**。軽量で起動も速いので、マイクロサービスとも相性がいいです。

### PHP / Python（乗せ方しだい）

自由な言語でも、**規約や型を後付けすれば**AI駆動向きにできます。PHPは **Laravel**、Pythonは **型ヒント＋ Django / FastAPI** に乗せれば締まります。とくに **Python は AI・データ領域では事実上の標準**で、その用途なら第一候補です。

### TypeScriptは流行っているけど

いま **Next.js の App Router や Hono** など、**TypeScriptだけでフロントもバックも開発できる**ようになりました。言語がバックエンドとフロントエンドで同じだから開発スピードが上がる、と思いますよね。

ただ1つ弱点があります。それは**いろんな設計思想があるので、AIの出力がばらける**ということです。

なので、TypeScript はむしろ **ハーネスをしっかり設計する必要がある**言語です。型という土台はあるので、そこに CLAUDE.md・rules・ディレクトリ構成を“決め切る”ことで、ぐっと安定します。

## まとめ

| 仕組み | 役割 | 強制力 |
| --- | --- | --- |
| **CLAUDE.md** | プロジェクトの説明書（毎回自動ロード） | お願い |
| **rules** | コーディング規約（パス単位で自動適用） | お願い |
| **skills** | 定型タスクをスラッシュコマンド化 | お願い |
| **Hooks** | イベントで自動実行・危険操作をブロック | **強制** |
| **Agents** | 専門特化の別働隊（結果だけ返す） | お願い |
| **settings.json** | 許可/禁止コマンドの制御 | **強制** |
| **TDD / CI** | テストで品質を縛る最後の砦 | **強制** |

ポイントは、**「お願い系」と「強制系」を使い分けること**。方針はCLAUDE.md/rules/skillsで伝え、絶対に外せない一線はHooks/permissions/CIで物理的に止める。これがハーネスエンジニアリングの肝です。
