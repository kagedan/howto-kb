## Claude Codeを使っていて感じていたモヤモヤ

Claude Codeを使っていると、こういうことがあると思います。

* 頼んだことの実装自体はできているものの、既存コードのルールや構造には従っていない（ので、結局自分がさらに指示を出す or 面倒になって自分で直す）
* 精度を上げようとするとプロンプトを頑張って書かないといけない、もしくは何往復も質問をしてClaude側に深掘りさせないといけない
* Plan Modeも試してみたけど、詳細な実装計画は出してくれるものの、自分のユースケースではもう一歩物足りなさを感じていた

Claude Codeに頼んでいるはずなのに、結局自分が頑張っている、という本末転倒感があります。

自分の場合、実際にあったのはこんなケースです。

* 既存のディレクトリ構造を無視して、本来は複数のファイルに分けるべき処理を1つのファイルにすべて書いてきた
* モノレポで他のパッケージにまたがる共通関数があるのに、その存在に気づかず使ってくれなかった or 自作することがあった（車輪の再発明）

たしかに動くは動くんですが、プロジェクトのルールには従ってほしかった...

プロジェクトのルールをCLAUDE.mdなどでコンテキスト情報としてある程度整備していたとしても、こういった「同じディレクトリの隣のファイルを見ればわかる」レベルの細かい知識まですべてドキュメント化するのはコスパが悪く、現実的ではありません。  
ゆえに問題の根本は、**AIコーディングエージェントがコードベースを十分に把握しないまま実装に入ってしまう**ことにあると思っています。

## feature-devプラグインとは

feature-devは、[anthropics/claude-plugins-official](https://github.com/anthropics/claude-plugins-official)（＝Claude Code向けの高品質なプラグインを集めたリポジトリ）に入っているClaude Code用のプラグインです。  
`/feature-dev:feature-dev`というコマンドで起動でき、以下の7つのフェーズからなる構造化されたワークフローで機能開発を進めてくれます。

| フェーズ | 内容 |
| --- | --- |
| 1. Discovery | 要件の明確化・制約の洗い出し |
| 2. Codebase探索 | 既存コードのパターンや類似機能を調査 |
| 3. 質問 | 曖昧な点をすべて質問し、回答を待ってから次へ |
| 4. アーキテクチャ設計 | 複数の設計案を提示し、トレードオフを比較 |
| 5. 実装 | 承認後にコーディング開始 |
| 6. コードレビュー | 品質・バグ・規約の観点でレビュー |
| 7. サマリー | 変更内容と次のステップをまとめ |

内部では3つの専門エージェントが動いています。

* **code-explorer**: 既存のコードベースを深く調査する
* **code-architect**: 調査結果をもとに設計を行う
* **code-reviewer**: 実装後の品質レビューを行う

詳細は公式のREADME.mdをご覧ください。

<https://github.com/anthropics/claude-plugins-official/tree/main/plugins/feature-dev>

### Plan Modeとの違い

似たような機能としてClaude Code組み込みのPlan Modeがありますが、両者には明確な違いがあります。

|  | Plan Mode | feature-dev |
| --- | --- | --- |
| 用途 | 読み取り専用でコードベースを分析し、実装前に計画を立てる | 計画から実装・コードレビューまで一気通貫でエージェントに委譲 |
| 動作モデル | 単一モデルが計画・実行 | 複数の専門エージェントが分業 |
| トークン消費 | 少ない傾向 | 多い傾向 |

Plan Modeもコードベースを読み取り専用で調査した上で計画を出してくれますが、単一のClaudeが動いているだけで、専門エージェントによる分業はありません。feature-devはコードベース調査・設計・レビューをそれぞれ専門エージェントが担当するため、各フェーズの精度が高く、かつ曖昧な点を実装前にしっかり整理してくれる、というのが大きな違いだと思っています。

## 何が変わったのか

使ってみて一番実感したのは、**まず既存コードをしっかり調べてから動く**という点です。

素のClaude Code（プラグインなしのデフォルト状態）だと、プロンプトの情報だけをもとに実装に入ってしまうことがあります。具体的な例を挙げると、モノレポ構成のプロジェクトで他のパッケージにまたがる共通関数がある場合、素のClaude Codeはその存在に気づかず使ってくれないことがあります。自分のプロジェクトで実際にあったのは以下のようなケースです。

* TypeScriptのdiscriminated unionのswitch文でパターン漏れを型エラーとして検知するための`assertNever`という共通関数をプロジェクト内で用意しているのに、使ってくれずdefaultのケースが素通りになっていた
* 共通のloggerモジュールがあるのに、それを使わず独自のログ出力を書いてきた

feature-devを使った場合は、コードベースを先に読み込んでいるおかげで、既存の類似実装からこういった共通関数の存在を読み取って正しく使ってくれます。

また、最初のフェーズで関連するコードを深く読み込んだ上で、設計・実装の判断に迷う箇所についてだけ質問をしてきます。しかもその質問が適切な情報量で、こちらが答えるだけである程度の精度が担保されます。

![feature-devの質問フェーズの様子](https://static.zenn.studio/user-upload/deployed-images/c26e29c1a1dbe7a5ac1b5e71.png?sha=b32d42733572b4ef225f1e0dd775a91163039448)  
*コードベースを調査した上で、設計の判断が必要な箇所だけ質問してくれる*

今まではその精度を出すために自分でプロンプトを頑張って書くか、何往復かやり取りをしてClaude側に深掘りさせる必要がありました。feature-devはその手間をワークフロー側が引き受けてくれているイメージです。

さらに、実装完了後にcode-reviewerエージェントによるレビューが入るため、型エラー・リンターエラー・ユニットテストの失敗などはだいたいチェック済みのものが上がってきます。素のClaude Codeだと自分が何度もお世話しないといけなかったところが、かなり楽になりました。

あくまで私個人の主観的な印象ですが、必要最小限のプロンプトを与える状況下において、「動くは動くけど...」という状態から「ほぼそのまま使える」状態に近づいた感覚があります。

## 使い分けの基準

feature-devが万能かというとそうではなく、タスクの規模感によって使い分けるのがよいと思っています。

* **小さいバグ修正・ちょっとした改善 → 不要**  
  ワークフローを回すほどでもない変更に使うと、かえって時間がかかります。素のClaude Codeで十分です。
* **中規模の実装タスク → 最適**  
  新機能の追加や、既存機能への一定規模の変更など。コードベースの理解が精度に直結するタスクで効果を発揮します。
* **大規模タスク → 分解してから使う**  
  大規模なタスクをそのままfeature-devに渡すのはおすすめしません。自分の場合は、まずDesign Docを書いて設計とざっくりとした実装計画をまとめておき、タスクを中規模程度に分解してからfeature-devに渡すようにしています。Design DocはリポジトリにMarkdownで置いておき、コマンド実行時に参照させます。

  ```
  /feature-dev:feature-dev docs/design/feature-name.md に記載されている〇〇の部分を実装して
  ```

  こうすることで、feature-devが設計の意図を理解した上でコードを書いてくれるので、アウトプットの質がさらに上がります。

## 注意点

**トークン消費と待ち時間**  
内部でサブエージェントを複数動かしているため、トークンの消費量は素のClaude Codeより多くなります。また、各フェーズを順番に進めていく都合上、アウトプットが返ってくるまでの時間も長めです。小さいタスクには使わない、という使い分けの理由はここにもあります。

**auto compact（未検証）**  
なお、各フェーズを順番に進めていく都合上、コンテキストも多く使う可能性があります。私はまだ遭遇していませんが、途中でauto compactが走った場合にいつも通りの精度が出るのかは不明です。コンテキストウィンドウが圧迫されてきた場合は念のため注意してください。

## インストール方法

[公式README](https://github.com/anthropics/claude-plugins-official/tree/main?tab=readme-ov-file#installation)に記載されている以下のコマンドでインストールできます。（2026年3月時点）

```
/plugin install feature-dev@claude-plugin-directory
```

※ もしmarketplaceが見つからないというエラーが出たら、先に以下を実行してください。

```
/plugin marketplace add anthropics/claude-plugins-official
```

インストール後、`/reload-plugins`を実行すると使えるようになります。

あとは以下のように実行するだけです。

```
/feature-dev:feature-dev 〇〇を実装して
```

※ 公式READMEや他の記事でも`/feature-dev`で呼び出せると書いてあるのですが、命名規則が変わったのか少し長くなっているようです。

## おわりに

feature-devを使い始めてから、「Claude Codeに頼んでいるのに結局自分が頑張っている」という感覚がだいぶ薄れました。コードベースを先に深く読んでくれる、というシンプルな仕組みが、実装の精度に大きく効いているんだと思います。

Claude Codeは使っているけどfeature-devはまだ、という方はぜひ試してみてください！
