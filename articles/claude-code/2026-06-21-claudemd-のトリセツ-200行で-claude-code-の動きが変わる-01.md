## はじめに：200行がなぜ重要なのか

CLAUDE.md にルールを書いたのに、Claude がそのとおりに動いてくれない。セッションが長くなるほど、書いたはずの指示が無視される——Claude Code を使い込んだ人なら経験したことはあるのではないでしょうか。

公式ドキュメントは CLAUDE.md を **200 行以内**に収めることを推奨しています。ただし 200 行で切り捨てられるわけではありません。CLAUDE.md は長さに関わらず**全文ロード**されます。問題は「切られる」ことではなく「薄まる」ことです。

CLAUDE.md の内容はシステムプロンプトの一部ではなく、システムプロンプトの**後ろに user メッセージとして注入**されます。Claude はそれを読んで従おうとしますが、強制力はありません。セッション開始時にロードされた後は毎ターンのコンテキストとして消費され続けるため、長くなるほど会話の文脈と競合し、指示の遵守率が下がります。

しかし 200 行は思ったより少ない。プロジェクトの規約、ビルドコマンド、ディレクトリ構造、禁止事項——書きたいことはいくらでもあります。この記事では、200 行という制約の中で CLAUDE.md の効果を最大化する書き方、そして 200 行に収まらない情報をどこに逃がすかを整理します。

## CLAUDE.md の仕組み — いつ・どこから読まれるか

CLAUDE.md を効果的に使うには、まず読み込みの仕組みを押さえる必要があります。

### 配置場所とスコープ

CLAUDE.md は配置場所によってスコープが決まります。読み込み順は上から下で、すべて累積的に結合されます。

| スコープ | 配置場所 | 用途 | 共有範囲 |
| --- | --- | --- | --- |
| マネージド | `/Library/Application Support/ClaudeCode/CLAUDE.md` 等 | 組織全体のポリシー | 全ユーザー |
| ユーザー | `~/.claude/CLAUDE.md` | 個人の全プロジェクト共通設定 | 自分だけ |
| プロジェクト | `./CLAUDE.md` or `./.claude/CLAUDE.md` | チーム共有のプロジェクト規約 | チーム全員 |
| ローカル | `./CLAUDE.local.md` | 個人のプロジェクト固有設定 | 自分だけ |

### 読み込みの挙動

* 作業ディレクトリから**ルートに向かって**ディレクトリツリーを遡り、各階層の CLAUDE.md を発見・結合する
* 同じ階層では `CLAUDE.md` → `CLAUDE.local.md` の順で読まれる（ローカルが後、つまり優先度が高い）
* **サブディレクトリ**の CLAUDE.md は起動時には読まれず、Claude がそのディレクトリのファイルにアクセスしたときにオンデマンドで読み込まれる
* HTML コメント（`<!-- -->`）はコンテキスト注入前に除去される。人間向けのメモに使える（ただしコードブロック内のコメントは保持される）

## よくあるアンチパターン — 200行を超える原因

CLAUDE.md が 200 行を超えてしまう、あるいは 200 行以内でも効果が出ない。その原因はほぼ以下のパターンに集約されます。

### 何でも CLAUDE.md に詰め込む

API のエンドポイント一覧、データベースのスキーマ定義、デプロイ手順書——これらが混入すると 200 行はあっという間に超えます。CLAUDE.md は「毎セッション Claude が見る情報」です。毎回必要でない情報は skills や `.claude/rules/` に移動してください。

### 曖昧な指示

検証できない指示は従いようがありません。

| 曖昧な指示 | 具体的な指示 |
| --- | --- |
| コードをきれいに書いて | インデントは 2 スペースで統一する |
| 適切にテストして | コミット前に `npm test` を実行する |
| ファイルを整理して | API ハンドラは `src/api/handlers/` に配置する |

「きれいに」「適切に」「整理して」は人間同士の会話では通じますが、CLAUDE.md の指示としては機能しません。

### 矛盾する指示の散在

複数の CLAUDE.md や `.claude/rules/` に相反するルールがあると、Claude はどちらか一方を任意に選びます。定期的に `/memory` で読み込まれているファイルを確認し、矛盾がないか見直してください。

### お願いで強制しようとする

「絶対に `.env` を編集しないでください」と CLAUDE.md に書いても、それは**お願い**であって強制ではありません。Claude の判断で上書きされる可能性があります。

確実にブロックしたい操作は、CLAUDE.md ではなく別の手段を使ってください。

.claude/settings.json

```
{
  "permissions": {
    "deny": ["Edit(.env)", "Edit(**/.env.*)"]
  }
}
```

PreToolUse フックでブロックすれば、Claude の判断に関係なく阻止できます。

### 放置して腐る

削除したディレクトリのパス、変更済みのビルドコマンド、もう使わないライブラリの名前——古い情報が残っていると、Claude が存在しないファイルを参照したり、動かないコマンドを実行したりします。CLAUDE.md もコードと同様にメンテナンスが必要です。

## 200 行以内に収める技術

200 行に収めるには、CLAUDE.md に残すものと外に出すものを意識的に分けることが必要です。

### 情報の逃がし先

| 情報の種類 | 移動先 | 理由 |
| --- | --- | --- |
| 特定ファイル向けルール | `.claude/rules/`（paths 付き） | 該当ファイルを扱うときだけ読み込まれる |
| 手順書・ワークフロー | skills | `/deploy` のように必要時に呼び出せる |
| リファレンス情報 | skills | オンデマンドで読み込まれる |
| 個人設定 | `CLAUDE.local.md` | チーム共有の CLAUDE.md から分離 |
| 全プロジェクト共通の個人ルール | `~/.claude/rules/` | 全プロジェクトで適用される個人ルール |

`~/.claude/rules/` はユーザーレベルの rules です。「コメントは日本語で書く」「PR タイトルは英語」のような個人の好みを、チーム共有の `.claude/rules/` に混ぜずに管理できます。ユーザー rules はプロジェクト rules より先にロードされ、プロジェクト側が後勝ちになります。

この分離を行ったうえで、CLAUDE.md に残す指示の書き方にも原則があります。

### 具体的に書く

CLAUDE.md の指示は**検証可能**であるべきです。Claude が指示に従えたかどうかを客観的に判定できる粒度で書いてください。

CLAUDE.md

```
## ビルドとテスト

- ビルド: `pnpm build`
- テスト: `pnpm test`（コミット前に必ず実行）
- リント: `pnpm lint`

## コーディング規約

- インデント: 2 スペース
- import 順序: Node.js 標準 → 外部パッケージ → プロジェクト内
- コンポーネントは `src/components/` に配置
- API ハンドラは `src/api/handlers/` に配置
```

### 構造化する

マークダウンの見出しと箇条書きで整理してください。Claude は構造を読み取って情報を検索するため、密な段落よりも箇条書きのほうが遵守率が高くなります。

セクション例:

* `## ビルドとテスト` — コマンド一覧
* `## コーディング規約` — スタイルルール
* `## プロジェクト構造` — ディレクトリの役割
* `## やってはいけないこと` — 禁止事項

## CLAUDE.md に書くもの・書かないもの

CLAUDE.md に入れるかどうかの判断基準は「**毎セッションで Claude が知っておくべきか**」です。

### 書くべきもの

* ビルド・テスト・リントのコマンド
* コーディング規約とスタイルルール
* プロジェクトのディレクトリ構造と各ディレクトリの役割
* 「常にXする」「絶対にYしない」のルール
* 使っているフレームワークやツールの注意事項

### 書くべきでないもの

これらは CLAUDE.md 以外の仕組みに移してください。

| 情報 | 移動先 | 理由 |
| --- | --- | --- |
| API ドキュメント、スキーマ定義 | skills | 毎セッション不要。必要時にオンデマンドで読む |
| デプロイ手順、リリースチェックリスト | skills | `/deploy` で呼び出すワークフロー |
| 特定言語・ディレクトリ向けルール | `.claude/rules/`（paths 付き） | 該当ファイルを扱うときだけ読み込む |
| 強制すべき制約 | hooks / `permissions.deny` | CLAUDE.md は強制力がない |
| 個人の好み（エディタ設定等） | `~/.claude/CLAUDE.md` | 全プロジェクト共通のユーザー設定 |

### CLAUDE.md / rules / skills の使い分け

| 観点 | CLAUDE.md | `.claude/rules/` | skills |
| --- | --- | --- | --- |
| 読み込みタイミング | 毎セッション | 毎セッション or ファイルアクセス時 | オンデマンド |
| スコープ | プロジェクト全体 | ファイルパスで絞れる | タスク単位 |
| 適した内容 | ビルドコマンド、コーディング規約 | 言語別・ディレクトリ別ガイドライン | 手順書、リファレンス |
| コンテキストコスト | 毎リクエスト消費 | paths 付きなら条件付き | 使用時のみ |

## `.claude/rules/` でルールを整理する

CLAUDE.md が長くなってきたら、`.claude/rules/` にルールを分割するのが有効です。

### 基本構成

```
.claude/
├── CLAUDE.md            # コアの指示（200行以内）
└── rules/
    ├── code-style.md    # コードスタイル
    ├── testing.md       # テスト規約
    └── security.md      # セキュリティルール
```

`paths` フロントマターなしのルールは、CLAUDE.md と同様に毎セッション読み込まれます。

### paths でスコープを絞る

`paths` フロントマターを付けると、Claude がマッチするファイルを扱ったときだけ読み込まれます。

.claude/rules/frontend.md

```
---
paths:
  - "src/components/**/*.tsx"
  - "src/hooks/**/*.ts"
---

- コンポーネントは関数コンポーネントで書く
- カスタムフックは `use` プレフィックスを付ける
- CSS は Tailwind のユーティリティクラスを使う
```

これにより、バックエンドのコードを扱っているときにフロントエンドのルールがコンテキストを消費することがなくなります。

よく使う glob パターン:

| パターン | マッチ対象 |
| --- | --- |
| `**/*.ts` | 全ディレクトリの TypeScript ファイル |
| `src/**/*` | src 配下の全ファイル |
| `*.md` | プロジェクトルートの Markdown ファイル |
| `src/components/*.tsx` | 特定ディレクトリの React コンポーネント |

## `@import` と `/init` を活用する

### @import で外部ファイルを取り込む

CLAUDE.md 内で `@path/to/file` と書くと、そのファイルの内容がコンテキストに展開されます。

CLAUDE.md

```
プロジェクトの概要は @README.md を参照。
利用可能なコマンドは @package.json を参照。
```

@import の注意点:

* 取り込んだファイルもコンテキストを消費する。@import は整理手段であり、コンテキスト削減にはならない
* 再帰 import は**最大 4 ホップ**まで
* 相対パスは**作業ディレクトリではなく import 元ファイル基準**で解決される
* パスを文字列として書きたいだけの場合はバッククォートで囲む（`` `@README` `` はリテラル、`@README` は import）

Claude Code は **CLAUDE.md を読むのであって AGENTS.md は読みません**。AGENTS.md を使っている場合は、CLAUDE.md から import するか symlink で対応します。

CLAUDE.md

```
@AGENTS.md

## Claude Code 固有の指示

plan mode で `src/billing/` 配下を変更する場合は事前承認を求める。
```

### /init で CLAUDE.md を自動生成する

既存プロジェクトに CLAUDE.md がない場合は、`/init` でスタートできます。Claude がコードベースを分析し、ビルドコマンド、テスト手順、プロジェクト規約を自動で検出して CLAUDE.md を生成します。

既に CLAUDE.md がある場合は、上書きではなく改善提案として機能します。`/init` は既存の `AGENTS.md`、`.cursorrules`、`.devin/rules/`、`.windsurfrules` も読み取って取り込むため、他ツールからの移行にも使えます。

## auto memory との使い分け

Claude Code には CLAUDE.md とは別に **auto memory** という仕組みがあります。

| 観点 | CLAUDE.md | auto memory |
| --- | --- | --- |
| 書く人 | あなた | Claude |
| 内容 | 指示とルール | 学んだパターンや知見 |
| 保存場所 | プロジェクト内 | `~/.claude/projects/<project>/memory/` |
| 読み込み | 毎セッション（全文） | 毎セッション（MEMORY.md の先頭 200 行 or 25KB） |

auto memory は Claude が自分でメモを取る仕組みです。「ビルドに `pnpm` を使う」「テストは `vitest` で実行する」といった情報を、あなたの操作から学習して記録します。

使い分けの目安:

* **Claude に守らせたいルール** → CLAUDE.md に書く
* **Claude が自分で気づく事実** → auto memory に任せる

`/memory` コマンドで読み込まれている CLAUDE.md 一覧と auto memory の内容を確認・編集できます。

## よくある問題と対処法

### CLAUDE.md に書いたのに従わない

Step 1: `/memory` で CLAUDE.md が読み込まれているか確認する。一覧に表示されなければ、配置場所が間違っています。

Step 2: 指示が十分に具体的か見直す。「テストを書いて」より「`src/__tests__/` に vitest のテストを作成する」のほうが従いやすい。

Step 3: 矛盾する指示がないか確認する。複数の CLAUDE.md や rules で相反するルールがあると、Claude が任意に選びます。

### `/compact` 後に指示が消えた

プロジェクトルートの CLAUDE.md はコンテキスト圧縮後に再注入されます。しかしサブディレクトリの CLAUDE.md は再注入されず、次にそのディレクトリのファイルにアクセスしたときに再読み込みされます。

会話中だけの指示はコンパクション時に失われます。繰り返し伝えている指示があれば、CLAUDE.md に書いてください。

### モノレポで他チームの CLAUDE.md が読まれる

`claudeMdExcludes` で除外できます。

.claude/settings.local.json

```
{
  "claudeMdExcludes": [
    "**/other-team/CLAUDE.md",
    "**/other-team/.claude/rules/**"
  ]
}
```

## まとめ

CLAUDE.md を効果的に使うためのポイントを整理します。

* **200 行以内**に収める。肥大化したら rules・skills に分離する
* **具体的に書く**。検証可能な指示は遵守率が高い
* **強制したい制約は CLAUDE.md に書かない**。hooks や `permissions.deny` を使う
* **`.claude/rules/`** で paths スコープを活用し、関係ないルールのコンテキスト消費を避ける
* **auto memory** との役割分担を意識する。指示は CLAUDE.md、学習は auto memory
* **定期的にメンテナンス**する。古い情報は Claude を混乱させる

まずは `/init` で CLAUDE.md を生成し、200 行以内を意識しながら育てていくのがおすすめです。「同じ修正を 2 回指摘したら CLAUDE.md に書く。200 行を超えたら rules か skills に逃がす」——このサイクルを回すだけで、Claude Code の動きは確実に変わります。
