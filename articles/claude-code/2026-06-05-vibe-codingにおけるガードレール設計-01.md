# AIエージェントにガードレールを「一枚」ではなく「層」で張る ― 5つの仕組みを噛み合わせて初めて効く

> これまで、AIコーディングエージェントを実務で長期運用するための仕組みを4本に分けて書いてきました。
>
> この記事は、その**総集編**です。一つひとつは独立した工夫に見えますが、本当に効くのは「組み合わさったとき」だという話をしたいと思います。

## はじめに：一枚のガードレールは、必ず抜けられます

AIエージェントにルールを守らせる方法を考えると、人はたいてい**一点豪華主義**に陥ります。「`CLAUDE.md`に全部書けばいい」「pre-commitフックで弾けばいい」「テストを必須にすればいい」。

しかし、どれも単体では破られます。

* **ルールを書くだけ**では、指示は「お願い」にすぎません。エージェントは長い作業の終盤でルールを忘れ、「テストを追加した」と報告しながら実際には追加していない、ということが起きます。
* **フックで弾くだけ**では、何を弾くべきかの判断基準（仕様）がブレます。古いドキュメントを根拠に「正しい実装」をした結果、正しくフックを通過してしまいます。
* **テストを必須にするだけ**では、形だけのテストが量産されます。そもそも「何が正解か」を知らなければ、テストは意図を固定できません。

つまり、**ルール・仕様・テスト・監査・実行制御は、それぞれが互いの前提を補完し合っています**。一枚ずつ見ると地味ですが、層にすると抜け道が塞がります。これが本題です。

生成量が人間のレビュー能力を超えた今、「全diffを人間が見る」という前提はすでに破綻しています。ですから人間のレビューを増やすのではなく、**機械が落ちる場所を構造的に減らす**しかありません。

## 5つの柱と、それぞれが守る「別々の失敗」

噛み合わせを語る前に、各層が**どの失敗を担当しているか**を整理します。重要なのは、5つが**別々の失敗モード**を見ているということです。

### ① CLAUDE.md ―「結論」だけを、短く、揺るがず

ルールは**短いほど守られます**。ですから`CLAUDE.md`には「必ずこうする」という規範と、どのルールがどこにあるかの**ルーティング索引**だけを置きます。背景・例外・理由は一切書きません。

肝は\*\*「理由」を物理的に分離する\*\*ことです。「なぜそうなのか」「過去にどの案を却下したか」は、すべてObsidianのような別レイヤーに追い出します。こうすると指示は圧倒的に短くなり、エージェントは迷いません。ルールを疑いたくなったときだけ、別の場所で理由をたどればよいのです。

### ② docs ―「補助資料」ではなく「一次情報」

AIエージェントにとって、ドキュメントは参考書ではなく**判断の根拠そのもの**です。曖昧・古い・場所が不明な記述は、そのまま誤実装になります。

ですから docs は次のように設計します。

* **ルールと仕様を混ぜない**（混ざると矛盾記述に挟まれて誤動作します）
* 用語は**glossaryを唯一の正典**にし、表記レイヤー（DB=snake\_case / 概念=PascalCase / 散文=日本語）を分けます
* 大きな文書は**全文読み前提をやめ、TOC＋部分読み**で設計します。「全文Read禁止リスト」を指示書に明記し、横断調査は探索サブエージェントに委譲します
* URL・エンドポイント・画面と**コード起点の対応表**を置き、grep/findの舐め回しをなくします
* 廃止記述は削除するか`SUPERSEDED`を明記し、**定期的に蒸留**します

ポイントは、ドキュメントを「人が読む文書」ではなく\*\*「エージェントが最短で正解に到達するための索引付きデータベース」\*\*として設計する点にあります。

### ③ test ―「品質保証」ではなく「変化検知センサー」

テストの位置づけを変えます。テストは「通すべき制約」ではなく、**意図せぬ変化を捕まえるセンサー**です。そして`Test First`を**例外なしの義務**にします。

実行コストの問題は、階層化で割り切ります。

完全な一発検査を目指しません。**性質の違う不完全な検査を重ねます**。これは⑤の思想とも一致します。

### ④ commit時の監査 ―「自己申告」を物理的に検証する

ここが、ルールを「お願い」から「強制」に変える要です。設計は**二層**になっています。

**層2の静的チェック**は、追加行（diffの`+`）だけを対象に、機械的に走ります。

* ハードコードIP / 秘密情報（環境変数や`${...}`は許容）
* **例外の握り潰し**（`except: pass`、空except、シェルの`|| true`）
* 設定ファイルの出どころ逸脱（正規の env 以外の読み込み）
* 関数定義の重複（＝共有コード再利用の促し）
* bashの`set -e`罠（`local x=$(cmd)`は終了コードを飲み込みます）
* プロジェクト固有のアーキ制約（例：APIネームスペース分離）
* 構文・lint・型・複雑度（ツール不在なら黙ってスキップ＝graceful degradation）

そして**接合点がdiffハッシュ契約**です。層1が完走したときのハッシュと、いまコミットしようとしているdiffのハッシュが一致しなければ、「監査したつもり（＝層1未実施）」と判定してブロックします。これで\*\*「やったつもり」が物理的に成立しなくなります\*\*。

誤検出への逃げ道はあります。ただし全て**理由必須**です（`# audit-ignore: <理由>`、`audit-ignore-file: <理由>`、一度きりの`.audit-skip`：読まれた瞬間に消えます）。「理由なし」は抑制と認めません。**回避してもログに痕跡が残る**設計になっています。

### ⑤ deploy / 並行作業の制限 ―「不可逆」と「奪い合い」を止める

これが、これまで単体記事にしていなかった5本目の柱です。①〜④が「**間違ったコードを止める**」のに対し、⑤は「**間違った"操作"と"競合"を止める**」役割を持ちます。

* **deploy承認ゲート**：`deploy.sh`の実行、サーバーへのinstall、本番アーカイブのアップロードを、`PreToolUse`フックが検出して**deny**します。承認は「理由を書いて30分だけ開く窓（mtimeで自動失効）」でしか与えられません。読み取りやログtailは誤検知しません。
* **main直編集の禁止**：mainブランチへの直接Edit/Writeをフックがblockし、worktreeでの作業を強制します。
* **セッションロック**：同一ディレクトリで複数のエージェントが同時に走るのを検出し、working treeのstaged変更を奪い合う事故を防ぎます。
* **worktree残骸の検出**：放置されたworktreeをセッション開始時にflagします。

①〜④が「コードの中身」を守るのに対し、⑤は「**いつ・どこで・誰が**手を動かすか」という実行の文脈を守ります。AIは並行で速く動ける分、ここを縛らないと不可逆な操作と状態破壊で簡単に事故ります。

## 本題：なぜ「組み合わせ」が効くのか

ここからが言いたかったことです。5つは独立した便利機能ではなく、**互いの欠陥を埋め合う一つの系**になっています。

### 連鎖1：①と②がなければ、④は何も判断できません

監査フック（④）は「正しさ」を判定しますが、その正しさの基準は①のルールと②の仕様から来ています。

> 「APIネームスペースを跨ぐな」というフックのチェックは、`CLAUDE.md`のルールと`docs/`の設計判断があって初めて意味を持ちます。逆に、ルールを書いても④がなければ守られません。

**ルールは④に「何を弾くか」を与え、④はルールに「強制力」を返します。** 双方向です。

### 連鎖2：③と④は同じ思想の表裏です

テスト（③）も監査（④）も、根っこは同じです ―「**完全解を一個作るのではなく、性質の違う不完全な検査を重ねる**」。テストは挙動の変化を、監査はパターンの混入を捕まえます。守備範囲が違うので、両方あって穴が塞がります。片方だけだと、もう片方が見ている失敗がすり抜けます。

### 連鎖3：②の「蒸留」と知識住み分けが、系の劣化を防ぎます

ガードレールは作って終わりではありません。仕様が古びれば④の判定根拠も腐ります。ですから②の**ドキュメント蒸留**と、Obsidianの\*\*status管理（active / superseded / obsolete）\*\*が、系全体の「賞味期限」を管理しています。**ガードレールを生かし続けるためのガードレール**です。

### 連鎖4：⑤がなければ、①〜④を真面目にやる意味が薄れます

どれだけコードを丁寧に検査しても、誰かが承認なしにdeployしたり、並行セッションが互いの変更を上書きしたりすれば、検査結果ごと吹き飛びます。⑤は①〜④の成果を**保全する外枠**になっています。

### 全体像：時間軸に沿って多段で止める

噛み合わせを時間軸で見ると、こうなります。

**一点で完璧に止めようとせず、別々の失敗を別々のタイミングで、それぞれ得意な層に担当させます。** これが組み合わせの正体です。一枚の壁は乗り越えられますが、向きの違う網を何枚も重ねれば、すり抜けられる確率は層の数だけ掛け算で下がります。

## 設計を貫く3つの共通原則

5つの層は、実は同じ設計思想を共有しています。

1. **結論と理由を分離する** ― ルール・仕様は短く揺るがず、理由は別レイヤーへ。短いほど守られ、深掘りしたいときだけ理由をたどります。
2. **記憶ではなく仕組みに埋め込む** ― エージェントの「覚えておく」に依存しません。フック・ポリシー・証跡ファイルという**外部の構造**に強制力を持たせます。AIは毎回初対面ですから、記憶は当てにしません。
3. **抜け道は塞がず、痕跡を残させる** ― 緊急回避は許しますが、必ず理由を要求し、ログに残します。「やむを得ず破った」が**可視化される**ことが、乱用を抑えます。

---

## おわりに：ガードレールは「足し算」ではなく「噛み合わせ」です

個々の工夫 ― ルールを書く、ドキュメントを整える、テストを必須にする、監査を入れる、deployを縛る ― は、どれも単体でも価値があります。ですが本当に効くのは、それらが**互いの前提と成果を補完し合う一つの系**になったときです。

* ①が②に支えられ、
* ④が①の基準で動き、
* ③と④が違う失敗を分担し、
* ②の蒸留が系を腐らせず、
* ⑤がその全部を保全する。

AIエージェントは速いです。だからこそ、間違いも速く・大量に・並行して起きます。それを「人間が頑張ってレビューする」で受け止めるのは、もう無理があります。

**ルールは書くだけでなく強制する。検査は一枚でなく重ねる。そして、それぞれを噛み合わせる。** 一枚の完璧な壁を探すより、向きの違う不完全な網を重ねるほうが、現実には強いのです。これが、数ヶ月スケールでAIエージェントと開発して得た、いまのところの結論です。
