# はじめに

2026年4月14日、AnthropicがClaude Codeに[**Routines**機能をリサーチプレビューとして発表](https://claude.com/blog/introducing-routines-in-claude-code)しました。

これは、Claude Codeの立ち位置が変わるレベルのアップデートです。

内容は、**Claude Codeが「起動してもらう道具」から「自分で動くエージェント」に変わった**ということ。プロンプトを設定し、リポジトリと外部連携を接続する。これだけで、Anthropicのクラウド上でClaude Codeが自律的に起動・実行・完了します。

* PCつけっぱなし → **不要**
* Mac Miniなどの常時稼働マシン → **不要**
* ハック的な構成 → **不要**

Claude Codeが完全に「自前インフラ不要」なAIエージェント運用に到達しました。

本記事では、Routinesの仕組み・トリガー設計・既存ツールとの違いを整理し、このアップデートが何をどう変えるのかを解説します。

設定URL：<https://claude.ai/code/routines>  
対象プラン：Pro / Max / Team / Enterprise（全有料プラン）

# これまでの課題は何だったのか

Claude Codeは強力なAIコーディングツールですが、これまではあくまで「**道具**」でした。

開発者がPCを開き、ターミナルを起動し、指示を出す。結果を受け取ったら、次の指示を出す。あくまで**人間が主語**のワークフローでした。

この構造には根本的な制約があります。

| 課題 | 具体的な問題 |
| --- | --- |
| PCの起動が前提 | マシンが落ちていれば何も実行されない |
| 常時稼働マシンのコスト | Mac Miniやクラウドインスタンスを別途用意すると維持費と管理の手間が発生 |
| GitHub Actionsでの代替 | CIとしては動くが、Claude Codeの文脈理解やMCP連携が使えない |
| 設定数の制限 | 以前の自動化機能では3つまでしか設定できなかった |

要するに、Claude Codeの能力は十分なのに、「**人間がトリガーを引かないと動けない**」ことがボトルネックでした。

# Routinesが変えた「常駐しないエージェント」という設計思想

Routinesは、この前提を根本から覆しました。

トリガーが発火すれば、Claude Codeがクラウド上で勝手に起動し、コンテキストを読み、タスクを実行し、結果を返す。**Claude Codeが主語**のワークフローへの転換です。

そしてここで重要なのは、Routinesは常時起動しているわけではないということです。  
「Claude Codeが自分でタスクを拾い、処理し、完了報告だけ残す」

これはいわゆるサーバーレスのAIエージェント版です。常駐プロセスを維持するコストもリスクもなく、**トリガーに応答してオンデマンドで動く**。「常駐しないエージェント」という新しい設計思想がRoutinesの本質です。

# 3種類のトリガーとその破壊力

Routinesのトリガーは3種類。**1つのルーティンに複数トリガーを同時設定**することも可能です。

### スケジュールトリガー「寝てる間にAIが働く」

毎時・毎夜・毎週のサイクルで定期実行。CLIの `/schedule` コマンドとも連動します。

ここで重要なのは、従来のcronとは決定的に違う点があります。cronは「決められたスクリプトを決められた通りに実行する」だけですが、Routinesは**AIがリポジトリの現状を読んだうえで動的に判断**して実行します。

たとえば「毎朝、昨日マージされたPRの内容を要約して」というプロンプトを設定した場合、マージされたPRが0件なら何もしない、3件あればそれぞれの変更内容を読んで要約する、といった判断をAI自身が行います。条件処理などを自らの手で書く必要はありません。

**ユースケース例**

* 毎朝、前日のPR一覧をサマリーして通知
* 毎週月曜に依存パッケージの更新チェックおよびPRの自動作成
* 毎夜、テストカバレッジの低下を検知してIssueを立てる

### APIトリガー「Claude Codeを呼び出せるAPIにする」

ルーティンごとに**専用エンドポイント**が発行され、POSTリクエストで即時起動できます。

これはかなり強力で、Claude Codeの文脈理解能力を、任意の外部システムからHTTPリクエストから起動できるということです。

GitHub ActionsでClaude APIを呼び出す構成では、環境構築・認証・コンテキスト受け渡しにかなりの手間がかかりますが、APIトリガーならPOSTを1本投げるだけで起動できます。

**ユースケース例**

* Slackのスラッシュコマンドから起動
* 社内管理画面のボタンに紐づけてワンクリック起動
* Zapier / Make / n8n 等のノーコードツールからClaude Codeを呼び出す

### GitHubイベントトリガー「PRが来たら即座にAIが読む」

GitHub Webhookと連携し、PRオープン・push・merge等のイベントをリアルタイムに検知して起動します。

* PRが開いたら → 動く
* Issueが立ったら → 動く
* Webhookが飛んだら → 動く

従来、PRオープン時のAIレビューをやろうとすると、GitHub Actionsでワークフローを組み、Claude Code（またはAPI）を呼ぶスクリプトを書き、認証を通し、実行環境を整えるという工程が必要でした。

Routinesなら設定画面でリポジトリとトリガーを選び、プロンプトに「PRの変更内容をレビューして、問題があればコメントして」と書くだけです。

**ユースケース例**

* PRオープン時に自動コードレビュー
* mainブランチへのマージ後にリリースノート自動生成
* 特定ディレクトリへのpush時にドキュメントの自動更新

## GitHub Actions / cron と何が違うのか

「結局GitHub Actionsでよくない？」という疑問は当然出ると思うので、明確に整理しておきます。

| 観点 | GitHub Actions / cron | Claude Code Routines |
| --- | --- | --- |
| 実行内容 | 事前定義したスクリプト | **AIが状況を読んで動的に判断** |
| 設定方法 | YAML / crontab | **自然言語のプロンプト** |
| 条件分岐 | if文で事前にすべて定義が必要 | **AIが文脈から判断**（記述不要） |
| MCP連携 | なし | **あり**（Notion, Slack, DB等と直接連携） |
| コードレビュー | lint等の静的解析 | **変更意図を理解したレビュー** |
| 環境構築 | Workflow定義 + Docker等 | **設定画面でプロンプトを書くだけ** |

GitHub Actionsが「何をどう実行するか」をすべて人間が定義するのに対し、Routinesは「何を達成したいか」だけ書けば、実行方法はAIが判断します。この差は、「自動化できる人」の範囲を一気に広げます。

もちろん、まだ複雑なCI/CDパイプラインやデプロイフローはGitHub Actionsの方が適しています。Routinesが真価を発揮するのは、**「判断」が入るタスクの自動化**です。

今後の差は「どのツールを使うか」ではなく、「**どんなトリガーでAIを動かすか**」の設計力に移っていくのではないでしょうか。

## セットアップ

<https://claude.ai/code/routines> にアクセスし、以下を設定するだけです。

* ルーティン名を入力
* プロンプト（何を達成したいか）を自然言語で記述
* リポジトリを選択
* トリガーを1つ以上設定（複数設定可）
* 必要に応じてMCP設定を追加

CLIからは `/schedule` コマンドでスケジュールトリガー付きのルーティンを直接作成することもできます。

## 今後どうなるのか

Anthropicは、GitHub以外のWebhookソースへの拡張を予定していると報告しています（時期は未定）。GitLab・Bitbucketへの対応が進めば、GitHub以外をメインで使っているチームにも選択肢が広がります。  
もう一つ注目しているのは、MCP連携との組み合わせです。現時点でもNotionやSlack等との接続が可能ですが、対応サービスが増えれば「GitHubのPRをトリガーに、Notionのドキュメントを更新し、Slackに通知する」といった、複数サービスをまたいだワークフローが自然言語のプロンプト1つで完結する世界が見えてきます。  
Routinesは、Claude Codeを「対話型のコーディングツール」から「開発ワークフローのバックボーン」に押し上げる第一歩だと捉えています。

## 所感

以前からClaude Codeの自動化機能は使っていましたが、「設定は3つまで」「PCが起動していないと動かない」という2つの制約が常にありました。Routinesはその両方を同時に取り払うアップデートです。

個人的に最も価値を感じるのは、「プロンプトで自動化を定義できる」という点です。GitHub ActionsでAIレビューを組もうとすれば、YAMLを書き、スクリプトを整備し、認証を通し、エラーハンドリングを入れるという工程が必要です。Routinesなら「PRが来たらレビューして」の一文で済むようになります。この設定コストの圧倒的な低さが今後のエンジニアリング業務を大きく変えていくと思います。

まだリサーチプレビュー段階なので実行回数の上限や安定性には注意が必要ですが、これはClaude Codeの使い方が「道具として使う」から「エージェントとして任せる」に変わる転換点だと思います。
