僕は一人で、ブログ（400記事）、note、栄養特化サイト、音楽、動画、それと個人開発のアプリ8本を回しています。完全にキャパオーバーでした。

そこで、Claude Code を「賢いチャットツール」として使うのをやめました。**コンテンツ制作のパイプラインそのものを Claude Code の中に作り込む**ことにしたんです。

今、その中身はこうなっています。

* スラッシュコマンド: **60個**（15カテゴリ）
* チームのロール定義: **50個**（12カテゴリ）
* 単一エージェント定義: **26個**

この記事では、この「工場」をどう設計したか、運用して分かった効くパターン、そして踏んだ落とし穴を共有します。コードを書く話ではなく、**エージェントを組み立てる話**です。

## 設計の中心：コマンドは「誰に振るか」の指示書

全部の起点は `.claude/commands/` 以下のMarkdownファイルとカスタムスラッシュコマンドです。コマンドファイルの中で `@` 記法を使い、処理を誰に振るかを宣言します。

| 記法 | 意味 |
| --- | --- |
| `@teams/xxx.md` | Agent Teams のテームメイト（並列・相互通信あり） |
| `@agents/xxx.md` | 単一サブエージェント（Task tool で逐次実行） |
| `@docs/xxx.md` | 参照ドキュメント（読むだけ） |
| `@templates/xxx.md` | 出力フォーマット |

ポイントは、**コマンド＝オーケストレーションの宣言**にして、実際の仕事はロール定義ファイルに分離したことです。コマンドを読めば「何が、どの順で、誰に振られるか」だけが分かります。中身（プロンプト）は別ファイルなので、片方を直してももう片方が壊れません。

## 2つの実行モデルを使い分ける

同じロール定義を、2つの方法で動かしています。

* **Task tool（逐次）**: ロールをサブエージェントとして順番に起動します。安定して速い。これがデフォルトです。
* **Agent Teams（並列）**: TeamCreate でチームを作り、テームメイトを並列で走らせます。テームメイト同士が SendMessage で直接やり取りできます。大きいタスク向けです。

最初は全部 Agent Teams で組みましたが、ほとんどの記事制作は逐次で十分だと分かりました。並列の旨味が出るのは「SEO戦略」と「ビジュアル企画」のように依存関係のない工程を同時に走らせるときだけです。**並列化はコストとデバッグ難度を上げる**ので、依存グラフを描いて「本当に同時に走るのはどこか」を見てから決めた方がいいです。

## 実例：`/blog:new` は7役の組み立てライン

一番作り込んだのがブログ記事の生成です。`/blog:new` を打つと、7つのロールがバケツリレーで記事を仕上げます。

```
Interviewer    → meta/interview.json      ヒアリング結果
SEO Strategist → meta/seo-strategy.json   タイトル・KW・見出し構成
Researcher     → meta/research.json       競合・一次ソース
Ghostwriter    → meta/draft-raw.md        本文ドラフト
Visual Planner → meta/visual-plan.json    アイキャッチ・図のプロンプト
Editor         → draft.md                 校正・内部リンク
Link Builder   → 既存記事を更新           被リンクを張ってインデックス促進
```

ここで一番大事な設計判断があります。**エージェント間のデータの受け渡しを、メモリ（会話）ではなくファイル（`meta/` 以下のJSON）でやっている**ことです。理由は次のセクションで書きます。

## 運用して効いた設計パターン

### 1. ステートはファイルで渡す

エージェントの出力を会話の中で次のエージェントに口頭で渡すと、長くなるほど崩れます。だから各ロールは成果物を `meta/seo-strategy.json` のようにファイルへ書き、次のロールはそれを読みます。

これには3つ効能があります。

* **耐久性**: 途中で落ちても成果物が残る
* **検査可能**: JSONを開けば何が決まったか一目で分かる
* **再開可能**: 次のパターンに直結する

### 2. コンテキストが飛んでも途中から再開する

長い制作だと途中でコンテキストウィンドウが溢れます。そこで「どの `meta/` ファイルが既に存在するか」を見て、再開地点を決めるロジックを入れました。

```
interview.json ある？      → あれば Interviewer スキップ
seo-strategy.json ある？   → あれば SEO Strategist スキップ
research.json ある？       → あれば Researcher スキップ
draft-raw.md ある？        → あれば Ghostwriter スキップ
...
```

ファイルにステートを外出ししたおかげで、`/compact` が走ろうがセッションが切れようが、続きから走れます。**会話履歴を信頼しない設計**にしておくと、長尺タスクが安定します。

### 3. 順序のバグは「下流の確定」で直す

最初、Visual Planner（図のプロンプト生成）を SEO Strategist の直後に置いていました。SEO段階の「仮の見出し」で図を作らせていたわけです。

ところが Ghostwriter が執筆中に見出しを変えると、図と本文がズレます。そこで **Visual Planner を Ghostwriter の後ろに移しました**。確定した見出し（draft-raw.md）を入力にすれば、ズレません。

エージェントの順序は「論理的に正しい順」ではなく、\*\*「下流が必要とする入力が確定する順」\*\*で組むべき、という学びでした。

### 4. 破壊的操作の前に承認ゲート

Link Builder は WordPress の既存記事を実際に書き換えます（被リンクを足す）。ここは自動で走らせず、**プランを提示してユーザー承認を取ってから**実行します。AIに外部の本番データを触らせる工程には、必ず人間のゲートを1枚挟みます。

### 5. 品質ゲートを工程に埋める

note の制作は Writer → Editor → **Checker** の3段にして、最後の Checker に「AI臭」を0〜10でスコアリングさせています。emダッシュの多用、均一な文長、定型構成、こうした生成文っぽさを検出して弾きます。出力の質を祈るのではなく、**検査する工程をパイプラインに組み込む**わけです。

## うまくいかなかったこと

正直に書きます。

* **バックグラウンド実行で権限が死ぬ**: `run_in_background: true` と `bypassPermissions` を併用すると、エージェント内の Write が拒否されることがありました（体感5回中4回）。ファイル保存が要る工程はフォアグラウンドで回しています。
* **トークンが溶ける**: サブエージェントを8個並列で回して20万トークン消し飛ばした日があります。並列＝偉い、ではありません。
* **小タスクには過剰**: 一段落直すだけならコマンドを呼ぶより手で書いた方が速いです。工場は量産にしか効きません。

## コマンド自体を、ログから育てる

最後に、一番気に入っている仕組みです。コマンドやロールは作って終わりではなく、実行ログから改善する自己改善ループに乗せています。

```
log（実行を記録）→ review（繰り返す問題を検出）→ amend（定義を修正）→ eval（効果を検証）
```

「Ghostwriter が だ/である調を出しがち」のような繰り返しの不具合をログから拾い、ロール定義を直します。**プロンプトを静的なファイルでなく、フィードバックで進化するコンポーネントとして扱う**と、使うほど精度が上がります。
