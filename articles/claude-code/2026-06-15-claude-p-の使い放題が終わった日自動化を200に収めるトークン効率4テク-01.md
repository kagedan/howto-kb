私は今、27本のcronスクリプトを `claude -p` で回しています。自動リファクタ、コンテキストの改善フィードバック、品質チェック、プロダクトのLPやマーケティング結果の定期分析。寝ている間に自律的に動く、小さなエージェント群です。

2026年6月15日、その前提が静かに変わりました。

これまで `claude -p` の実行は、私が払っているClaude Max 20x（$200/月）の「使い放題」の内側にありました。寝ているうちに分析してもらっても、追加の請求は来ない。ところが6月15日から、`claude -p` とAgent SDKの利用は対話用のサブスク枠から切り離され、**別の月次クレジット**を消費するようになりました。Max 20xなら月$200分、API標準レートで。

つまり私の自律エージェントには、今月から「天井」ができたわけです。そして私はまだ、自分のcronが月いくら使っているのか正確に把握していません。月末にいきなり止まってから気づく、という未来がうっすら見えています。

## 何が変わったのか -- 「対話」と「自動化」が別財布になった

公式サポート記事の表現を借りると、変更点は一文です。

> Claude Agent SDK and `claude -p` usage no longer counts toward your Claude plan's usage limits.  
> （出典: [Use the Claude Agent SDK with your Claude plan -- Anthropic Support](https://support.claude.com/en/articles/15036540)）

`claude -p`（ヘッドレス実行）とAgent SDKの利用が、サブスクの対話利用枠から外れた。代わりに、プランごとの月次クレジットを消費します。

| 項目 | 6/15まで | 6/15から |
| --- | --- | --- |
| 対話（ターミナル/IDE） | サブスク枠 | **変更なし** |
| `claude -p` / Agent SDK | サブスク枠の内側（実質使い放題） | **別の月次クレジット** |
| Max 20xのクレジット額 | -- | 月$200分（API標準レート） |
| 繰り越し | -- | **なし（使い切り型）** |
| 枯渇後 | -- | usage credits有効→従量課金 / 無効→**停止** |

注意したいのは、このクレジットが**毎月リセットされる使い切り型**だということです。今月$50しか使わなくても、来月が$350になるわけではない。毎月$200に戻ります。貯金はできません。

そして枯渇したとき、私のようにAPIキーを設定していない環境では、追加課金の受け皿がない。つまりその月の自動化は、クレジットを使い切った時点で止まります。翌月リセットされるまで、cronは沈黙します。

## 効率化が「美徳」から「コスト管理」に変わった

ここが一番伝えたい点です。

6/15まで、私のcronがトークンを無駄遣いしていても、痛みはありませんでした。サブスクの内側では、無駄もタダだったからです。冗長なプロンプト、必要以上に賢いモデル、毎日律儀に走る重い判断。どれも「動いているからヨシ」で済んでいた。

6/15から、その無駄は一つずつ値札を持ちます。同じ自動化でも、トークン効率が悪ければ月の早い段階で$200を食い潰し、効率が良ければ月末まで走り続ける。**自動化の良し悪しが、機能ではなく原価で決まるようになった**わけです。

身も蓋もない言い方をすると、私がこれまで「丁寧な設計」と呼んでいたものの何割かは、ただの課金前提の浪費でした。今月、請求書がそれを採点してくれます。

以下、私が27本のcronを$200に収めるために効くと考えている4つのテクを、効き目の大きい順に並べます。

## テク1: 工程ごとにモデルを使い分ける（効き目: 大）

一番効くのはこれです。すべての工程を最上位モデルで回す必要はありません。

私の運用基盤は、ざっくり3つの工程に分かれています。

* **Observer（観測）**: アクセス数などのメトリクスのスナップショットを読んで集計する。判断はほぼ不要。
* **Strategist（方針）**: 観測結果から次にどう動くかを決める。ここは判断が重い。
* **Executor（実行）**: 決めた方針に沿って重い生成タスクを処理する。一番重い。

実行ログを見ると、工程ごとの重さは実行時間にもはっきり出ています。Observerは実行0〜1秒（スナップショットを読むだけ）、Executorは約3,000秒（7件の重いタスクをまとめて処理する）。実行時間はトークン消費と完全に比例するわけではありませんが、どの工程が重いかの当たりはつきます。

であれば、Observerに最上位のOpusを使う理由はありません。安いHaikuで十分です。判断が要るStrategistはSonnet、出力の質が成果に直結するExecutorだけ上位モデル。こうして、工程の重さとモデル単価を一致させる。

モデル間には明確な価格差があります（Haiku ≪ Sonnet ≪ Opus）。読むだけの軽い工程を一段安いモデルに落とすだけで、月の消費はまとまった割合で下がります。

```
# 工程ごとにモデルを指定して claude -p を呼ぶ
claude -p "$OBSERVER_PROMPT"   --model claude-haiku-4-5   # 観測は安いモデルで
claude -p "$STRATEGIST_PROMPT" --model claude-sonnet-4-6  # 判断はSonnet
claude -p "$EXECUTOR_PROMPT"   --model claude-opus-4-8    # 重い実行だけ最上位
```

「とりあえず全部賢いモデルで」は、6/15までは贅沢で済みました。今月からは、ただのコスト超過です。

## テク2: LLMに判断させない工程を剥がす（効き目: 大）

トークン効率の話をしていると忘れがちですが、**一番安いトークンは「呼ばないトークン」です**。

LLMでなくてもできる処理を、LLMにやらせない。これだけで消費はごっそり減ります。

私の例で言うと、定期ジョブのスケジューリングがそうでした。以前は `claude -p` に「何日の何時に実行して」と時刻計算までさせていたのですが、既存予約との衝突回避も時刻のずらしも、本来ただのシェルスクリプトで足りる処理です。今はスケジュール登録を全部シェル側に移し、`claude -p` には重い生成タスクだけをさせています。

つい先日も、予約済みのジョブを別の対象に差し替えるツールを作りましたが、これも中身はpure shellです。「どの予約を消して、どれを登録し直すか」の判断にLLMは要らない。`at` コマンドの操作と日付計算だけなので、シェルのほうが速くて、確実で、そして無料です。

```
# Before: claude -p に予約時刻の計算までさせる（LLMトークンを消費）
# After:  空きスロット計算も at 登録もシェルで完結。claude -p は重い生成だけ
available-slots.py "$START" "$COUNT" "$TIME" ...           # 空き枠をシェルで算出
echo "run-job.sh $SLUG" | at "$PUB_TIME" "$PUB_DATE"       # スケジュールもシェルで
```

判断の余地がない決定的（deterministic）な処理は、LLMから剥がしてコードに戻す。地味ですが、これが一番リターンの大きい片付けでした。賢いモデルに電卓の代わりをさせていたのだと気づくと、少し恥ずかしくもあります。

## テク3: コンテキストを軽くする（効き目: 中）

`claude -p` は、呼ばれるたびに前提となるコンテキストを読み込みます。プロジェクトの `CLAUDE.md`、記憶ファイル、ロードされるスキルの説明文。これらは出力の前段でかかる**入力トークン**として、毎回課金対象になります。

ここで効くのは3つです。

1. **常時ロードされるものを削る**。`CLAUDE.md` や記憶インデックスが肥大化すると、全cronの全実行にその分が乗ります。私の記憶ファイルは「インデックス1行 + 詳細は別ファイル」に分割していて、毎回フルロードされるのは索引だけにしています。
2. **スキルを必要なときだけ重くする**。私の日本語文章チェックは、検出だけの軽い `detect` 版と、違反が出たときだけ読む `rewrite` 版に分けてあります。毎回フルの巨大スキルを読ませない。違反ゼロのときは軽い版だけで終わる。
3. **プロンプトキャッシュを効かせる**。1回の実行の中で同じ前提を何度も使うなら（例: 1セッションで7件の重いタスクを続けて処理する）、共通部分はキャッシュが効いて入力分が割り引かれます。バッチを細切れに呼び直すより、まとめて1セッションで回すほうがキャッシュの恩恵を受けやすい。

入力トークンは「出力ほど派手ではないが、毎回必ずかかる固定費」です。固定費を削るのは、家計と同じで地味に効きます。

## テク4: 実行頻度とバッチ化 -- 週1で1週間分を決める（効き目: 中）

最後は、そもそも何回呼ぶかという話です。呼び出し回数は、消費に直結します。

私のある定期運用は、毎日「今日は何をするか」を判断させるのをやめて、**週1回のStrategistで1週間分の計画をまとめて決める**形にしています。判断という重い工程を週1に集約し、あとはその計画に沿って実行するだけ。毎日判断を走らせるのに比べて、重い判断工程の呼び出し回数が7分の1になります（実行そのものは案件ごとに走るので、減るのは判断の回数です）。

一方でObserver（観測）のような軽い工程は、毎日走らせても痛くありません。安いモデルで数秒なので。**重い工程はまとめて少なく、軽い工程はこまめに**。この配分が、トークン予算の効きどころです。

## 私の$200予算の組み方

4つのテクを、効き目の大きい順にもう一度並べます。

| テク | 何を削るか | 効き目 |
| --- | --- | --- |
| 工程ごとにモデル使い分け | 重い工程以外を安いモデルに | 🟢 大 |
| LLMから処理を剥がす | 決定的な処理をコードに戻す | 🟢 大 |
| コンテキスト軽量化 | 毎回かかる固定の入力費 | 🟡 中 |
| 頻度・バッチ化 | 重い工程の呼び出し回数 | 🟡 中 |

ただ、正直に書くと、私はまだ自分のcronが月いくら使うのか計測できていません。4つのテクはどれも効きますが、その効き目を確かめるにも、土台として**計測**が要ります。最初は粗くて十分で、私が見ようとしているのはこの3つです。

* どの工程（観測・判断・実行）がクレジットを食っているか
* 入力トークン（常時ロードされるコンテキスト）と出力、どちらが効いているか
* 1回あたりの消費 × 実行頻度 ＝ 月次の見込み

ここが見えれば、4つのテクのどれを優先すべきかが決まります。最適化はそこからです。

効率化のテクを4つ並べておいて結局「まず計測しろ」というのは締まりませんが、原価が見えていないものは管理できません。これは自動化に限った話でもないでしょう。

## まとめ

6月15日、`claude -p` は「使えなくなった」わけではありません。サブスクの内側で無料だった自動化が、原価の見える従量制に変わっただけです。

その瞬間から、自動化の上手さの定義が一つ増えました。動くこと、ではなく、**いくらで動くか**。トークン効率は、もう美徳ではなく原価管理です。

私はこれから、自分の27本のcronを一本ずつ採点し直すつもりです。まずは計測から。

あなたのcronは、今月いくら使っていますか。
