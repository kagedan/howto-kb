**Claude Code、まだ素のまま使ってませんか？**

2026年6月、Anthropicは怒涛のリリースラッシュを敢行しました。`v2.1.160`〜`v2.1.169`までのわずか1週間で、**地味だけど効く「隠し機能」**が大量に追加されています。

しかも、これらの多くは公式ブログのトップでは大きく取り上げられず、**Changelogの片隅にひっそり書かれているだけ**。知っている人だけが得をする状態です。

## 結論から言うと

- `--safe-mode` で**カスタマイズ全無効の安全起動**ができる（トラブル切り分けが一瞬に）
- `/cd` で**プロンプトキャッシュを壊さずに作業ディレクトリを移動**できる
- `fallbackModel` で**モデル過負荷時に自動フェイルオーバー**できる（落ちない）
- `MAX_THINKING_TOKENS=0` で**thinkingを完全オフ**にしてレスポンスを爆速化できる
- ダイナミックワークフローのトリガーが `workflow` → **`ultracode`** に変わった（要注意）

:::note info
本記事は2026年5月末〜6月初旬のChangelog（`v2.1.160`〜`v2.1.169`）に基づいています。Claude Codeはターミナル・VS Code・JetBrains・デスクトップアプリ・Webで動作し、すべての環境が対象です。
:::

## ① `--safe-mode`：カスタマイズ全無効で「犯人」を一発特定

`v2.1.169`（6月9日）で追加された、トラブルシュート専用の起動フラグです。

### これは何？

`--safe-mode` を付けて起動すると、**設定ファイル・カスタムスキル・フック・MCPサーバー・カスタムスラッシュコマンドなど、あらゆるカスタマイズを無効化**した「素の状態」でClaude Codeが立ち上がります。

### なぜ重要？

「急にClaude Codeが変な動きをする」「フックが原因なのか、MCPが原因なのか分からない」——こういうとき、従来は設定ファイルを1つずつコメントアウトして潰していく地獄の切り分け作業が必要でした。

`--safe-mode` なら、**素の状態で再現するか試すだけ**で「カスタマイズ起因かどうか」を1コマンドで判定できます。再現しなければ、犯人はあなたの設定の中にいます。

### 使い方

```bash
# すべてのカスタマイズを無効化して起動
claude --safe-mode

# 「safe-modeでは正常」なら、原因は自分の設定ファイル側にある
```

:::note info
切り分けの鉄則：「素で再現しないなら、自分の設定を疑え」。`--safe-mode` はこの鉄則をワンコマンドにしてくれます。
:::

## ② `/cd`：キャッシュを壊さずに作業ディレクトリを移動

同じく `v2.1.169` で追加された、地味だけど超実用的なセッション内コマンドです。

### これは何？

`/cd <path>` で、**セッションを終了せずに作業ディレクトリを切り替える**ことができます。ポイントは「**プロンプトキャッシュを壊さない**」こと。

### なぜ重要？

これまで作業ディレクトリを変えたいときは、いったんClaude Codeを終了して別ディレクトリで起動し直す必要がありました。すると、**それまで積み上げたプロンプトキャッシュが全部吹き飛ぶ**。長いセッションでは、これがコスト・レイテンシの両面で無視できない損失でした。

`/cd` を使えば、モノレポ内のサブプロジェクト間を行き来したり、メイン作業の途中で隣のリポジトリをちょっと覗いたり——という操作が、**キャッシュを温存したまま**できます。

### 使い方

```bash
# セッション中にそのまま打つ
/cd ../another-service

# モノレポのパッケージ間移動も一瞬
/cd packages/api
```

## ③ `fallbackModel`：モデル過負荷でも止まらない自動フェイルオーバー

`v2.1.166`（6月6日）で追加された設定。**可用性を底上げする**重要機能です。

### これは何？

`settings.json` に `fallbackModel` を指定しておくと、**プライマリモデルが過負荷（overloaded）のときに、自動で別モデルへ切り替えて処理を継続**してくれます。

### なぜ重要？

Opus 4.8はデフォルトでhigh effort（高負荷思考）で動くため、ピーク時間帯には過負荷エラーに当たることがあります。長時間のエージェント作業の途中で `overloaded` で止まると、それまでの文脈が無駄になりかねません。

`fallbackModel` を設定しておけば、**「メインはOpus、混んでたらSonnetで継続」**といった構成を自動化でき、ジョブが途中で死ぬのを防げます。

### 使い方（settings.json）

```json
{
  "model": "claude-opus-4-8",
  "fallbackModel": "claude-sonnet-4-6"
}
```

:::note info
無人で回すスケジュール実行やCIパイプラインでは、`fallbackModel` の有無が「完走率」に直結します。自動化勢は必ず設定しましょう。
:::

## ④ `MAX_THINKING_TOKENS=0`：thinkingを切ってレスポンス爆速化

これも `v2.1.166` の追加。**速度とコストをコントロールする**ためのノブです。

### これは何？

環境変数 `MAX_THINKING_TOKENS=0` を指定すると、**拡張思考（extended thinking）を完全に無効化**できます。さらに、**モデルごとにthinkingのオン/オフを切り替える**設定もサポートされました。

### なぜ重要？

thinkingは難問では強力ですが、**定型作業・短い編集・整形タスクではオーバーキル**です。思考トークンを消費する分だけ遅く、高くつきます。

「簡単なリファクタを大量に回したい」「とにかくレスポンスを速くしたい」という場面では、thinkingを切ることで体感速度が大きく上がります。

### 使い方

```bash
# thinkingを完全オフにして起動
MAX_THINKING_TOKENS=0 claude
```

```json
// settings.json でモデルごとに制御する例
{
  "env": {
    "MAX_THINKING_TOKENS": "0"
  }
}
```

:::note warn
難しい設計・デバッグ・大規模リファクタの「計画」フェーズではthinkingが効きます。常時オフにするのではなく、**タスクの性質で使い分ける**のが正解です。
:::

## ⑤ トリガーキーワードが `workflow` → `ultracode` に改名

`v2.1.160`（6月4日）での変更。**知らないと「動かない！」と混乱する**地雷ポイントです。

### これは何？

数十〜数百体のエージェントをオーケストレーションする**ダイナミックワークフロー**の起動キーワードが、`workflow` から **`ultracode`** に変更されました。

### なぜ重要？

旧来の記事やメモを見て `workflow` と打っても、もう発火しません。**「前は動いたのに」とハマる人が続出する**変更です。大規模なコードベースマイグレーションや一括処理を仕掛けるときは、新キーワードを使ってください。

### 使い方

```bash
# 旧: workflow ...
# 新: ultracode で起動
ultracode このモノレポ全体をTypeScript化して、テストも通して
```

## おまけ：地味に効く安全＆軽量化の改善

5機能以外にも、見逃せない改善が入っています。

### `disableBundledSkills`：同梱スキルをモデルから隠す（v2.1.169）

バンドルされた標準スキルを**モデルの選択肢から除外**できる設定。自前のスキルだけを使わせたいときに、ノイズを減らせます。

```json
{
  "disableBundledSkills": true
}
```

### 設定ファイル書き込み前のプロンプト（v2.1.160）

**シェルの起動ファイル（`.zshrc` 等）や設定ディレクトリへ書き込む前に確認プロンプト**を出すようになりました。意図しない環境改変を防ぐ安全策です。

### deny ルールの glob 強化＆クロスセッション通信の堅牢化（v2.1.166）

deny ルールで**より広いglobパターン**が使えるようになり、権限制御が柔軟に。加えてセッション間メッセージングのセキュリティも強化されています。

```json
{
  "permissions": {
    "deny": ["Read(./secrets/**)", "Read(**/.env*)"]
  }
}
```

### ストリーミング/アニメーション中のCPU負荷低減（v2.1.169）

地味ですが、**ストリーミング描画やアニメーション時のCPU使用率が削減**されました。ノートPCでファンが唸る問題が和らぎます。

## まとめ

- **`--safe-mode`** はトラブル時の最初の一手。「素で再現するか」を1コマンドで判定。
- **`/cd`** でキャッシュを温存したままディレクトリ移動。モノレポ勢は必須。
- **`fallbackModel`** で無人実行・CIの完走率を底上げ。Opus×Sonnetの二段構えが定番。
- **`MAX_THINKING_TOKENS=0`** は定型作業を爆速化。タスクの性質で使い分けを。
- **トリガーは `ultracode`**。`workflow` はもう発火しないので要注意。

たった1週間のアップデートで、ここまで使い勝手が変わります。**素のまま使うのは、もったいない**です。まずは `settings.json` に `fallbackModel` を1行足すところから始めてみてください。
