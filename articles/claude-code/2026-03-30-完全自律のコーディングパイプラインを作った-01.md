## 各社が自律型コーディングエージェントを作り始めている

[Ramp](https://builders.ramp.com/post/why-we-built-our-background-agent)はPRの約30%がエージェント経由で作られている。[Stripe](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents)は週に1000件以上のPRを完全自動で生成・マージしている。[Uber](https://www.youtube.com/watch?v=i1tZN41VKcE)もエンジニアリング組織全体でエージェントへのシフトを進めている。

各社とも数億行規模のコードベースや社内インフラとの統合が必要なので内製しているが、やっていること自体はシンプルで、エージェントにコードを書かせて、プログラムで検証して、通るまでループを回す、という構造になっている。これは既存のAIコーディングツールを組み合わせればローカルで再現できる。

この記事では、Claude Code、Codex CLI、Cursor CLIだけで構成した完全自律のコーディングパイプラインのアーキテクチャを書く。[前回の記事](https://zenn.dev/theaktky/articles/1c6c3b9333117c)ではコードレビューの自動化について書いたが、今回はその先で、UXデザインから実装、レビュー、リファクタまでを1本のパイプラインとして繋げた話になる。

## やっていること

エージェントとプログラムを組み合わせた小さなワークフローをYAMLで定義し、それをClaude Code（親エージェント）が順に実行・判断・ループする、という2層構成で、UXデザインから実装・レビュー・リファクタまでを自律で回している。

## 2つのレイヤー

このシステムは**小さなワークフロー**と**親エージェント**の2層でできている。

### ワークフロー層

各ステージはYAMLで定義された小さなワークフローになっている。1つのワークフローの中に、エージェント（Claude、Codex、Cursor）のステップと、shellでformat・test・typecheckを実行するステップが並んでいる。

たとえば実装ステージなら、エージェントがコードを書いた直後にshellステップがtypecheckを実行する、という流れがYAML上で直列に定義されている。エージェントが「typecheckはたぶん通るだろう」と判断して先に進む余地がない。書いたら必ず検証が走る。

レビューステージも同様で、エージェントがdiffをレビューした後に、別のエージェントがそのfindingsをトリアージして過剰な指摘を削る、という流れがワークフローとして固定されている。

ローカルにインストールされたClaude Code、Codex CLI、Cursor CLI、OpenCodeといったエージェントCLIをオーケストレーションするために[Rigg](https://github.com/tryrigg/rigg)というCLIを作ったので、興味があれば見てみてほしい。

### 親エージェント層

親エージェントはターミナルで動くClaude Code。各ワークフローを順番に呼び出して、結果を読んで、次のステージに何を渡すか判断して、必要なタイミングで人間に確認を求める。

ワークフロー自体は「与えられた入力に対してステップを順に実行する」だけで、パイプライン全体の流れを知らない。「UXデザインが承認されたから次は計画に進む」「レビューのfindingsがまだ残っているからもう1周回す」といった判断は、すべて親エージェントがやっている。**ワークフローは手順を持つが、意図は持たない。意図を持つのは親エージェントだけ。**

具体的には、親エージェントの振る舞いをClaude Codeのカスタムスラッシュコマンド（SKILL.md）として定義している。どのワークフローをどの順で呼ぶか、ワークフロー間で何をするか（結果をユーザーに要約する、承認を待つ、レビューをfindingsゼロまでループする、など）を書いておく。ユーザーは`/develop <要件>`と打つだけで、親エージェントがこの指示に従ってパイプライン全体を駆動する。

エージェントを含むワークフローは実行に数分から数十分かかることがあるので、SKILL.mdではワークフローをバックグラウンドで起動し、ログをポーリングして進捗を監視するように指示している。こうすることでClaude Codeがワークフローの完了をブロッキングで待たずに済む。

実際のSKILL.mdの例はこちら。ワークフローの実行結果を見てループを制御する親エージェントの定義になっている。

<https://github.com/aktky/codex-review-loop/blob/main/.claude/skills/codex-review/SKILL.md>

親エージェントにClaude Codeを使っているので、Claude Code自体の機能もそのまま活きる。たとえばRemote Controlと組み合わせると、パイプラインの承認ステップがスマホに通知として届く。パイプラインを起動してPCから離れて、UXデザインができた時点でスマホから確認・承認する、という使い方ができる。

## なぜ「小さなワークフロー」なのか

すべてをワークフローにするわけではない。**「必ずこうあるべき」と言い切れる部分だけをワークフローにする。** 判断が要る部分は親エージェントに残す。

### 自然言語の指示は守られない

AGENTS.mdやCLAUDE.mdに「typecheckを必ず通すこと」と書いても、エージェントはそれを無視することがある。エラーが複雑だと途中で諦めることもある。自然言語の指示はあくまで提案であって、実行を保証する仕組みではない。

ワークフローにすればこれが保証に変わる。「typecheckがパスするまで最大5回ループする」というのはエージェントに対するお願いではなく、ワークフローランナーが実行する制御フローだ。エージェントが「もういいだろう」と判断してループを抜けることはできない。

同じことがレビューにも言える。「findingsがゼロになるまで修正を繰り返す」をCLAUDE.mdに書いても、エージェントは2回くらいで満足して止まることがある。ワークフローのループとして定義すれば、findingsがゼロにならない限り次のイテレーションが走る。

要は、エージェントにはコードを書く・レビューする・修正するといった判断の要る仕事をやらせて、「書いたら必ずtypecheckする」「findingsがゼロになるまで止めない」といった機械的なルールはワークフロー側で強制する、という分担になっている。

### コンテキスト分離

各ワークフローはクリーンなセッションで起動する。実装ステージのコンテキストがレビューステージに漏れることはない。

これはコンテキストウィンドウの節約という実用的な理由もあるが、それ以上に、ステージ間の依存を明示的にするという設計上の理由が大きい。親エージェントが次のワークフローに何を渡すかを明示的に決めるので、「前のステージで何を話していたか」に暗黙的に依存する、という不安定な状態にならない。ワークフロー内のステップ間も同様で、各ステップは自分が依存する出力だけを受け取る。

## パイプライン

僕が実際に運用しているパイプラインは5ステージ構成。全ステージに共通する設計として、コーディング規約ドキュメントをほぼすべてのエージェントステップのプロンプトに注入している。ワークフロー定義の中でプロンプトに組み込んでいるので、エージェントが自分の判断で読み飛ばすことがない。また、各ステップで都度注入するので、コンテキストが長くなっても規約が埋もれない。

### 1. UXデザイン

まずコードベースの構造と、関連する外部ツールのドキュメントをリサーチする。その結果を元に、2つのモデルが独立にUXを設計する。この2つは並列で走るので、互いの出力に影響されない。最後に第3のモデルが、2つの設計案を読み比べて良いところを取り込み、1つのデザインドキュメントに統合する。

1つのモデルに任せると視野が狭くなりがちなので、独立した2案を出させてから統合する、という構成にしている。

<https://github.com/tryrigg/rigg/blob/main/.rigg/ux-design.yaml>

### 2. 計画

コードベースを探索して、実装に必要なコンテキスト（既存の型定義、関連モジュールの構造、使っているパターンなど）を集める。そのコンテキストとUXデザインを元に実装計画をドラフトする。

ドラフトができたら、別のモデルが計画を監査する。計画の中で参照しているファイルパスは実在するか、使おうとしている型やインターフェースは実際のコードと整合しているか、ステップの実行順序は依存関係を満たしているか、といった点をチェックする。指摘があれば修正し、問題がなくなるまで繰り返す。

計画の段階で「存在しない関数を呼ぼうとしている」「依存順序が逆」といった問題を潰しておくと、実装ステージでの手戻りが大幅に減る。

<https://github.com/tryrigg/rigg/blob/main/.rigg/plan.yaml>

### 3. 実装

計画に沿ってコードを書く。書き終わったらshellステップでformat、test、typecheckを実行する。全部通ればそのまま次へ進む。失敗したら、エラー出力をエージェントに渡して修正させ、再度shellで検証する。このループを最大5回回す。

前述の通り、このループはエージェントの判断で回っているのではなく、ワークフローの制御フローとして定義されている。エージェントが「このテスト失敗は無視してよい」と判断してスキップすることはできない。

<https://github.com/tryrigg/rigg/blob/main/.rigg/impl.yaml>

### 4. レビュー

mainとのdiffに対してコードレビューを実行する。レビューで出たfindingsを別のモデルがトリアージして、severityの低い指摘や偽陽性を除去し、修正が振り子になっている箇所にはbinding directiveを発行して片方に固定する。トリアージを通過した指摘に対してエージェントが修正し、さらに別のモデルが「その修正は根本原因に対処しているか、表面的なband-aid fixになっていないか」を検証する。コミットして、findingsがゼロになるまで最大6回ループする。

このレビューフローの詳細は[前回の記事](https://zenn.dev/theaktky/articles/1c6c3b9333117c)に書いた。

<https://github.com/tryrigg/rigg/blob/main/.rigg/review.yaml>

### 5. リファクタ

実装とレビューを通過したコードに対して、可読性や構造の改善を目的としたリファクタリングを行う。リファクタした結果に対してレビュー・トリアージ・修正のループを回し、リファクタが新たな問題を生んでいないか確認する。その後コードの簡素化を行い、もう一度レビューループを回す。

<https://github.com/tryrigg/rigg/blob/main/.rigg/refactor.yaml>

親エージェントはステージ1（UXデザイン）と2（計画）の後にユーザーへ結果を見せて、承認を得てから先に進む。ステージ3以降は自律で走る。人間が介入するのは2回の承認だけ。

またこれの良いところは、ワークフロー間で親Agentと議論をしながら調整していけるという点にある。単なるワークフローだとこれができない。

## これを実現するためにRiggというCLIを作った

上のワークフローは[Rigg](https://github.com/tryrigg/rigg)で定義している。AIコーディングエージェント向けのオープンソースのワークフローランナーで、エージェントとshellコマンドを組み合わせたマルチステップワークフローをYAMLで定義して実行できる。ループ、分岐、並列実行、ステップ間のデータの受け渡しに対応している。

headlessモードでの起動にも対応していて、親エージェントからUIなしで呼び出せる。各ステップの実行ログやステータスはローカルのSQLiteに保存されるので、親エージェントは`rigg logs`や`rigg history`といったCLIコマンドから、構造化された形でワークフローの進捗や結果を取得できる。

アーキテクチャ自体はシンプルで、小さなワークフローが「必ずこうあるべき」という制約を強制して、親エージェントがパイプライン全体の意図と判断を担う。この2つをどこで分けるかが、全体を安定させるための設計上の要点になっている。

```
curl -fsSL https://tryrigg.com/install | bash
rigg init
rigg run review
```

<https://github.com/tryrigg/rigg>

## 一番難しいのはワークフローを作るところ

ツールやアーキテクチャは用意できても、結局一番難しいのは自分たちのコードベースや開発フローに合ったワークフローを組むところだと思う。どのステップでどのモデルを使うか、トリアージの基準をどうするか、何回ループさせるか。こういった判断は実際に回してみないとわからないし、回した結果を見て常に調整し続ける必要がある。

僕自身もまだ実験の途中で、どこまでこのアプローチがスケールするのかは正直わかっていない。もしこのアーキテクチャで自分たちの開発を自動化してみたいという人がいれば、ぜひ声をかけてほしい。一緒に試行錯誤できると嬉しい。
