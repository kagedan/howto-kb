こんにちは！[株式会社ムクイル](https://www.mukuil.com/)のiwaです。

普段は VSCode 上で AI を使いながら開発を進めているのですが、今回はその中でも「そもそも VSCode で AI を使って開発するなら、何を入れればいいの？」というテーマで書いてみます。初めての記事なので拙い部分もあると思いますが、同じように迷っている方の参考になれば嬉しいです。

いま VSCode 上で動く AI コーディングツールは GitHub Copilot、Cursor、Windsurf、Cline、Continue、Claude Code……と選択肢が多く、しかも進化が速いので「とりあえず有名なやつを入れる」で止まっている人も多い印象です。

この記事では、

* VSCode で AI を使う主な選択肢とその違い
* 自分がどういう軸で比較して、何を選んだか
* 選んだあと、実際にどう開発を進めているか

を、あくまで自分の現時点（2026年7月）の理解として書きます。ツール名や仕様は変化が速い分野なので、「今はこう」というスナップショットとして読んでもらえればと思います。

## VSCode で AI を使う選択肢を整理する

大きく分けると、こんな系統があると理解しています。

**エディタ一体型（フォーク系）**

* **Cursor**：VSCode をフォークした AI ネイティブエディタ。補完からエージェントによるマルチファイル編集まで一気通貫。
* **Windsurf**：同じくフォーク系。Cascade というエージェント機能でタスクを自律的に進める設計。

**VSCode 拡張として動く汎用エージェント**

* **GitHub Copilot**：補完機能から始まり、今は Chat / Agent モードでファイル編集やコマンド実行までこなす。VSCode との統合が最も歴史的に深い。
* **Cline / Roo Code**：OSS の VSCode 拡張。BYOK（Bring Your Own Key）で好きなモデルを差し込める自由度が特徴。
* **Continue**：これも OSS で、モデルを自由に選べる路線。

**CLI 発のエージェント（VSCode 拡張・IDE 連携あり）**

* **Claude Code**：もともとターミナルの CLI ツールで、そこに VSCode / JetBrains の拡張が乗る形。CLI で完結する自律性の高さが強み。

ざっくり言うと、「エディタごと変える」か「今のエディタに拡張として足す」か、「モデルを固定にする（一体型）」か「好きなモデルを選べるようにする（BYOK 系）」か、という軸で整理できます。

## 何を軸に選ぶべきか

自分が比較したときに重視したのは、以下の点でした。

* **エージェントの自律性**：補完だけで十分なのか、複数ファイルにまたがる変更やコマンド実行までまかせたいのか。
* **コンテキスト管理の仕組み**：プロジェクトのルールや仕様をどう読み込ませられるか。前回の記事で書いた「常時ロード」と「必要なときだけロード」を切り分けられるかどうかは、開発が進むほど効いてきます。
* **拡張性**：MCP（外部ツール連携）、hooks（イベント時の自動実行）、サブエージェント、再利用可能なワークフロー（スキル的なもの）が用意されているか。
* **既存ワークフローとの共存**：普段使っている VSCode の拡張機能・デバッガ・Git 周りの操作を崩さずに使えるか。エディタごと乗り換えるコストは意外と大きいです。
* **チームでの共有しやすさ**：設定やルールを Git で管理して、チームメンバー間で再現できるか。個人の手元にしか残らない設定は、結局属人化します。
* **料金体系**：サブスクリプション型か、BYOK で従量課金か。チームで使うときのコスト予測のしやすさにも関わります。

エディタ一体型（Cursor / Windsurf）は導入すればすぐ強力ですが、「今使っている VSCode をそのまま使いたい」という自分たちの状況には少し重い乗り換えでした。逆に BYOK 系（Cline / Continue）はモデル選択の自由度は魅力ですが、その分「どう設定すればチームで再現できるか」を自分たちで組む必要が出てきます。

## 自分が選んだもの：Claude Code（VSCode 拡張）

結論から言うと、自分は Claude Code を VSCode 拡張として使う形に落ち着きました。理由は主に3つです。

**1. VSCode をそのまま使える**  
拡張として動くので、普段のデバッガ・拡張機能・ショートカットを一切変えずに済みます。エディタで選択した範囲をそのままエージェントの文脈に渡せたり、変更差分をエディタの diff 表示で確認できたりと、乗り換えコストがほぼゼロでした。

**2. コンテキスト管理の仕組みが体系立っている**  
前回の記事で書いた `CLAUDE.md` のフルロードと `.claude/rules/` の path-scoped ロードの使い分けは、実際に運用してみてかなり効きます。ルールをただ「たくさん読ませる」のではなく、「必要なときだけ読ませる」設計にできるのは、プロジェクトが大きくなるほど効いてくる部分でした。

**3. 周辺のエコシステムが自律性を後押ししてくれる**

* **サブエージェント**：調査や大きめの探索を別のコンテキストに逃がして、メインの会話を汚さない。
* **hooks**：コミット前のフォーマット確認など、イベント駆動で自動化できる。
* **skills（スキル）**：チームのワークフローを再利用可能な形でパッケージ化できる。
* **auto memory**：セッションをまたいでも「このプロジェクトではこういう方針だった」という学びを引き継げる。

これらは全部 Git で管理できるものが多く、チームで共有しやすいのも大きなポイントでした。

## 実際どう開発を進めているか

選んだあとの運用も書いておきます。

### Plan → 確認 → 実装の2段階にする

いきなり実装させず、まず設計・変更方針を提示してもらい、内容を確認してから実装に進む、という流れを徹底しています。特に URL やパラメータなど、エンドユーザー（発注者）の意図に直結する変更は、合意なしに進めると手戻りが発生しやすいので、ここは明確にルール化しています。

### 調査とレビューはサブエージェントに逃がす

「このコードはどう動いているか」「このバグの原因はどこか」といった調査は、専用のサブエージェントに投げて結論だけをメインの会話に持ってくるようにしています。長い探索の過程でメインのコンテキストが埋まってしまうのを防げます。

### ルールは path-scoped、資料は docs/ のまま

これは前回の記事の内容そのままですが、「言語・レイヤーごとの規約」は `.claude/rules/` に paths 付きで、「仕様書・調査ログ・実装計画」のような都度参照する資料は `docs/` に、という住み分けを続けています。

### 学びは auto memory に、繰り返し使う手順は skills に

「このプロジェクトではこの単語を使わない」「実装前に必ず確認を取る」といった、コード上には現れない運用ルールは memory に残しています。逆に「毎回同じ手順で行うレビュー」「毎回同じ形式で作る資料」のような再現性のある作業は skills 化して、都度説明しなくても同じ品質で実行できるようにしています。

## まだ迷っているところ

正直に書くと、まだ答えが出ていない部分もあります。

* **権限設計のちょうどよさ**：エージェントに何をどこまで自動許可するかは、チームの慣れとリスク許容度によって変わりそうで、まだ「これが正解」という設定に辿り着いていません。
* **チーム展開時の教育コスト**：自分が慣れてきた運用ルール（CLAUDE.md の書き方、rules の切り方など）を、他のメンバーにどう伝えるとスムーズに立ち上がるか、模索中です。
* **エディタ一体型との比較の再検証**：Cursor や Windsurf も進化が速いので、「今の結論」が半年後も同じかは分かりません。定期的に見直すつもりです。

## まとめ

* VSCode で AI を使う選択肢は、エディタ一体型（Cursor / Windsurf）、汎用拡張（Copilot / Cline / Continue）、CLI 発の拡張（Claude Code）に大別できる
* 選ぶときは「エージェントの自律性」「コンテキスト管理の仕組み」「拡張性」「既存ワークフローとの共存」「チームでの共有しやすさ」を軸に比較するとよさそう
* 自分たちは「VSCode をそのまま使える」「コンテキスト管理が体系立っている」「周辺エコシステムが自律性を後押しする」という理由で Claude Code を選んだ
* 実際の運用では、Plan→確認→実装の2段階、サブエージェントへの調査の切り出し、path-scoped なルール管理、memory と skills での知見の蓄積、を組み合わせている
