プロダクトのコード量が増えてくると、AIエージェントに調査を頼んでも「関係しそうなファイルを片っ端から読み漁った結果、時間もトークンも溶ける」という場面が増えてました。  
たとえば実際の現場ではこういう調査依頼が日常的に発生します。

> クーポンの登録API・更新APIで、ある項目が **必須か任意か** を調べてほしい。もし必須なら、DBのカラムを **NOT NULL から NULL 許可に変えたい** という要望があるので、その **影響範囲** も同時に報告してほしい。

この手の質問は「Zodスキーマ → DTO → ユースケース → Prismaモデル → 物理カラム → 設計書」と **層をまたいで** 追わないと正しく答えられません。

しかも対象は小さくありません。今回のモノレポはおよそ次の規模です。

| 項目 | 規模（およそ） |
| --- | --- |
| ソースコード | 約 47.5 KLOC（フロント 約26,900行 + バックエンド 約20,600行） |
| API 数 | 約 57 本 |
| 画面数 | 約 32 画面 |

たった1項目の要否を追うだけでも、複数ディレクトリ・複数レイヤーを横断します。だからエージェントは関連ファイルを大量に読み込みます。

そんな折、次の記事を読みました。

@[Claudeに「オントロジー」を持たせたら、コスト半分・3倍速になるかも](https://zenn.dev/takupeso/articles/claude-ontology-knowledge-structuring)

**「コードベースの概念と“関係”を、型付きの知識グラフ（オントロジー）として事前に外部化しておけば、Claudeは推測でファイルを読み漁らずに関係を辿れるので、コストが半分・速度が3倍になる」** というもの。  
著者の実測では、同一リポジトリ・同一質問で **コスト $1.26→$0.62（約50%減）、応答 2分7秒→44秒（約3倍）、キャッシュ読取トークン 764.7k→118.1k（約85%減）** ということでした。

自分の手元のコードベースで検証してみたいと思いました。  
実在するモノレポ（Nx + Hono + Prisma + Next.js 構成）を題材に **A/B検証** をしました。

## そもそも「オントロジーを持たせる」とは

**MCPの公式 `memory` サーバ**（`@modelcontextprotocol/server-memory`）を利用して、AIにオントロジーを持たせます。  
次の5要素で知識グラフを記録することを指します。

| 要素 | 意味 | 例（一般化） |
| --- | --- | --- |
| クラス | 概念の「型」 | 「サービス」「テーブル」「API項目」 |
| インスタンス | クラスの具体例 | 「注文サービス」「決済サービス」 |
| プロパティ | インスタンスの属性 | 「外部APIに依存する」「NOT NULL である」 |
| **関係（型付き）** | インスタンス同士の繋がり | `calls`（呼ぶ）／`depends-on`（依存する）／`persisted-to`（永続化される） |
| 制約 | 成り立つべき条件 | 「サービスはDBを直接共有しない」 |

そして **関係には種類（型）** があります。代表的なものだけでも次のように、「どう繋がっているか」を型として持てます。

* `is-a`（〜の一種）／`part-of`（〜の一部）
* `calls`（〜を呼び出す）／`depends-on`（〜に依存する）
* `persisted-to`（〜に永続化される）／`validated-by`（〜で検証される）

一方、`CLAUDE.md` の箇条書きや `@` でのファイル指定だけでは、「モノがどこにあるか」は示せても「どう関係しているか」までは表現できません。  
だからエージェントは毎回その関係を推測することになり、その推測のブレが、無駄なファイル読み込みとハルシネーションの温床になる——というのが参考記事の主張です。

### なぜ節約・高速になるのか

* **推測の排除**: 「この2つはたぶん呼び出し関係」を毎回推論せず、`calls` / `persisted-to` のように **確定情報** として辿れるから
* **必要ノードだけを濃く渡す**: グラフから関連サブグラフだけを引けば、無関係ファイルを読まずに済むから
* **多ホップ推論**: 「フィールド → カラム → 別テーブルの同名カラム」のように関係を連鎖して辿れるから

ここまでを1枚にまとめると、こういうイメージです。

## 検証セットアップ

### 1. クーポンドメインのオントロジーを作る

登録API・更新APIのリクエスト項目（約30項目）について、**「フィールド → 要否 → 永続先テーブル.カラム → DBのNULL制約」** と、層をまたぐアーキテクチャ関係を型付きで記録した知識グラフを用意しました。  
`memory` サーバのネイティブ形式（`memory.jsonl`）はこんな行の集まりです。

```
{"type":"entity","name":"Field:serviceTicketCode","entityType":"api-field","observations":["登録での要否: 必須","更新での要否: 必須"]}
{"type":"entity","name":"Col:coupon.service_ticket_code","entityType":"db-column","observations":["物理: coupon.service_ticket_code","NULL制約: NULL許可"]}
{"type":"entity","name":"Col:coupon_image.service_ticket_code","entityType":"db-column","observations":["物理: coupon_image.service_ticket_code","NULL制約: NOT NULL","一意キー(tenant_id+service_ticket_code)"]}
{"type":"relation","from":"Field:serviceTicketCode","to":"Col:coupon.service_ticket_code","relationType":"persisted-to"}
{"type":"relation","from":"Field:serviceTicketCode","to":"Col:coupon_image.service_ticket_code","relationType":"also-persisted-to"}
{"type":"relation","from":"Rule:showImage→serviceTicketCode必須","to":"Col:coupon_image.service_ticket_code","relationType":"depends-on"}
```

この6行を図にすると、こういう関係になっています。

同じ `serviceTicketCode` が2つのカラムへ `persisted-to` / `also-persisted-to` で関連付いていて、**片方（赤）だけが NOT NULL** という関係が一目で分かります。  
この「型付きの繋がり」を辿れることが、後述の罠を回避する鍵になります。

このようにして最終的に **80エンティティ / 145関係** がグラフになりました。

### 2. 本家 `memory` サーバで動くことを確認

記事どおりのセットアップは1コマンドです（ローカル実行・クラウド接続なし）。

```
claude mcp add --transport stdio memory -- npx -y @modelcontextprotocol/server-memory
```

実際に本家サーバ（`memory-server 0.6.3`）へ上記 `memory.jsonl` を読み込ませて `read_graph` を叩けば、グラフ全体が返ります。

```
initialize OK: memory-server 0.6.3
tools(9): create_entities, create_relations, add_observations,
          delete_entities, delete_observations, delete_relations,
          read_graph, search_nodes, open_nodes
read_graph OK -> entities: 80  relations: 145
```

**「関連ノードだけを濃く渡す」** が確認できます。  
`search_nodes` は全体ではなく関連サブグラフだけを返します。

```
search_nodes('serviceTicketCode') -> entities: 4  relations: 9
  E: Field:serviceTicketCode | api-field
  E: coupon.controller.ts     | controller
  E: Rule:showImage→serviceTicketCode必須 | domain-rule
  R: Field:serviceTicketCode - persisted-to      -> Col:coupon.service_ticket_code
  R: Field:serviceTicketCode - also-persisted-to -> Col:coupon_image.service_ticket_code
  R: Rule:showImage→serviceTicketCode必須 - depends-on -> Col:coupon_image.service_ticket_code
```

80/145 の全グラフではなく **4/9** の小さなサブグラフだけが返る。  
これが「無関係ファイルを読まない」＝トークン節約の正体です。

### 3. A/Bの組み方

同一の設問を、同一モデル（Claude Opus 4.8）・同一のサブエージェントで、条件だけ変えて走らせました。

* **条件A（オントロジーなし）**: リポジトリのパスだけ渡し、生コードを読んで回答
* **条件B（オントロジーあり）**: 上記の知識グラフ（可読ビュー）を先に渡し、関係を辿って回答。不足時のみソース参照可

各条件2試行、合計4エージェント。トークン数・ツール呼び出し回数・所要時間・正答性を、ハーネスの実測値とトランスクリプトのusage集計で取得しました。

設問は冒頭で記載した内容です。

> `serviceTicketCode` は登録・更新で必須か。必須なら、DBカラムを NOT NULL→NULL 許可に変えたい要望があるので影響範囲を報告せよ。

## この設問が「良問」な理由（横断調査の罠）

先に種明かしすると、この設問には **層をまたいで確認しないと踏む罠** が仕込まれています。

1. **API層では必須**。登録・更新とも `serviceTicketCode: z.string().trim().min(1).max(10)`（optional/nullableなし、空文字も弾く）。設計書でも「全カテゴリ常時必須」。
2. **ところがDBの `coupon.service_ticket_code` は、Prismaスキーマ上すでに `String?`＝NULL許可**。つまり「NOT NULL→NULL許可にする」DDL変更は、このカラムには **そもそも不要**。
3. **本当に NOT NULL なのは、同名の別カラム `coupon_image.service_ticket_code`**（画像テーブルの一意キー）。だから `showImage=true` のときだけ、コントローラが `serviceTicketCode` を追加で必須化している。
4. したがって要望の実体は「DBのDDL変更」ではなく、「APIの必須要件を緩めたいのか」「対象は `coupon` テーブルか `coupon_image` テーブルか」を確認する必要がある。

Zodスキーマだけを見て「必須です。DBもNOT NULLだからマイグレーションが必要」と答えると **二重に間違えます**。横断調査でしか正解にたどり着けない、検証にうってつけの設問です。

## 結果

4エージェントすべて、**同名カラムが2つあり片方は既にNULL許可**という罠まで含めて正解に到達しました。その上でコスト・速度に差が出ました。

### 読み取れること

* **トークンは効く**: 総トークンで約4.7倍、キャッシュ読取で約4.8倍の削減。記事の「キャッシュ読取85%減」に近い水準（今回は79%減）。ツール呼び出し（＝ファイルを読む回数）が18.5→7.5に減っており、これがトークン削減の実体です。
* **コストも効く**: 課金トークン相当で59%減。記事の「コスト約半分」と同じレンジ。
* **速度は“それなり”**: 1.76倍速。参考記事の3倍には届きませんでした。エージェントの思考時間やツールのオーバーヘッドが残るため、トークン削減率ほどには速くならない、というのが今回の体感です。
* **正答性は変わらなかった**: A/Bとも2/2。今回のベースライン（条件A）は丁寧に読み切ったため、記事が挙げる「幻覚抑制」効果は今回は観測されませんでした。**差はコストと速度に出た** という結論です。

それでも効果はあった。バックエンド2万行規模での検証だが、システムが育って層・ファイル・APIが増えるほど、この差は相対的に効いてくる。

## 正直な留保（ここが大事）

数字を鵜呑みにしないよう、限界も一応挙げておきます。

* **試行数が少ない（n=2）**。分散は大きめ。
* **正答性の差は出ていない**。今回はコスト・速度に効いた。
* **オントロジーの構築コストは別途**。多数の調査で償却される前提の投資。

なので今回の結論は、あくまで「この規模・この設問での現時点の話」。試行を増やしたり、コードベースがさらに育てば、もっと効くかもしれないし、別の結論になるかもしれない。そこは引き続き見守っていきたい。

## どういうときに導入すべきか

記事の判断基準は、今回の体感とも一致します。

**効く場面**

* 数百〜数千ファイルの、そこそこ育ったコードベース
* 「層をまたぐ」「呼び出し階層をまたぐ」構造的な質問が繰り返し発生する
* マルチターンの長期作業・チーム運用

**効きにくい場面**

* 小規模プロジェクト（グラフ自体がノイズになる）
* 単一ファイルで完結する事実確認（`@`指定やgrepで十分）

### 起点探しはどうする？（今回はベクトル検索を使わない）

オントロジーの担当は「関係を辿る」ことです。  
**その起点（どのファイルから見始めるか）** がポイントとなります。  
方法としては主に3つあります。

| 起点の見つけ方 | 手段 | 位置づけ |
| --- | --- | --- |
| ファイルを名指し | Claude Code の `@ファイル指定` | 当たりが付いているとき最速 |
| キーワード検索 | 標準の `grep`/`ripgrep`（Claude Code既定） | 固有名で当てられるとき十分 |
| 意味で検索 | ベクトル検索（RAG） | 表記ゆれに強いが、今回は不採用 |

今回のA/Bでは、起点探しに **`@`指定と標準grep** を使い、そこからの関係辿りをオントロジーに任せる構成にしました。  
**ベクトル検索は入っていません**。

* **Claude Code の既定はベクトル検索ではない**。標準は grep 系のキーワード検索で、`serviceTicketCode` のような固有名は名前で十分当てられます。
* **公式のベクトル検索MCPが現時点で存在しない**。グラフ側の `memory` は公式で用意されている一方、ベクトル検索は Chroma / Qdrant などの第三者MCPを足すか自作が必要。  
  今回はオントロジー(公式で完結する構成)ので効果を測りたかったからです。

## 再現手順（最小）

```
# 1. memory サーバを登録（ローカル実行）
claude mcp add --transport stdio memory -- npx -y @modelcontextprotocol/server-memory

# 2. 新規セッションでコードベースをグラフ化するよう依頼
#   「このリポジトリの主要なモジュールと依存・呼び出し関係を調べて、
#    memory に entities(種類つき) と relations(呼ぶ/依存する 等) として記録して。
#    各要素の役割は observations に短く添えて。」

# 3. 別セッションで比較
#   A: 「memory は使わず、コードを読んで答えて。<設問>」→ /usage を記録
#   B: 「memory のグラフを使って答えて。<設問>」      → /usage を記録
```

グラフは手元の `memory.jsonl` に永続化され、`read_graph` / `search_nodes` で確認できます。

## まとめ

* 「オントロジーを持たせる」＝**コードの関係を型付き知識グラフとして外部化し、エージェントが推測でファイルを読み漁らないようにする** こと。
* 実在コードベースのA/Bで、**トークン約79%減・課金相当59%減・1.76倍速** を確認。方向と規模は元記事とおおむね一致（速度は3倍には届かず）。
* 一方で **正答性の改善は今回は出なかった**。差はコストと速度に出る、というのが偏りのない結論です。
* **横断調査が多い・育ったコードベースほど投資対効果が高い**。「必須か調べて、DBのNULL制約変更の影響も」のような設問は、まさにグラフが刺さる典型でした。

### 今後の進め方（PDCAの一案）

まずは小さく始めるのが良さそうです。コード量が増えて調査が重くなってきたチームは、「よく聞かれる横断質問」の周辺だけ小さくグラフ化し、`/usage` で before/after を測る。効いたら対象ドメインを広げ、効かない質問はグラフの粒度や関係の型を見直す——という PDCA で回すのが現実的です。

さらにチーム運用に広げるなら、**グラフを個々人が持つのではなく中央で一元管理する**構成も考えられます。たとえば中央のオーケストレーター（自分の場合は Hermes Agent）に知識グラフの構築・更新・最適化を担わせ、各メンバーからの問い合わせを **グラフを使って一次トリアージ** する、という運用です。  
「グラフの鮮度を誰が保つか」問題を集約でき、個々のセッションは軽い問い合わせだけで済む——というのが次に試したい方向性です。ここはまだ構想段階のため、検証できたら追って記事にします。
