朝の作業を始めるとき、「今日何をやるか」を毎回ゼロから考え直していた時期がありました。手動起動の仕組みを書いてはいたのですが、起動を忘れればブリーフィングも出ない。

Claude CodeのRoutinesを使うと、毎朝7時に自動でブリーフィングが届く設定を5分程度で組めます。設定自体はシンプルでした。ただ、動かしてみて気づいたのは「自分のカレンダー運用が止まっていた」という事実のほうで、そちらのほうが記録する価値があると感じました。

---

## Routinesとは何か

RoutinesはClaude Codeのクラウド実行機能です。指定した時刻・APIイベント・GitHubイベントをトリガーに、プロンプトを自動実行します。

利用可能プランはPro・Max・Team・Enterprise（無料プランは対象外）。ステータスはresearch previewのため、仕様変更の可能性は排除できません。スケジュールの最小間隔は1時間で、実行環境はAnthropic管理のクラウドインフラ。ローカルファイルへの直接アクセスはできません。

管理画面のURLは <https://claude.ai/code/routines> です。今回はブラウザで進めましたが、Desktopアプリのサイドバー「Code」→「Routines」からも同じ画面を開けます。

---

## 5分で動かすまでの手順

![](https://assets.st-note.com/img/1778918498-RtjJohkd6YEmVIr8By34wfn7.png?width=1200)

Routinesのページには8種類のテンプレートがあります。今回は「☀️ Briefing」（"Summary of your calendar, emails, and messages"）を選びました。

設定画面で変更したのは実質1箇所だけです。

**トリガーの変更:**

曜日を「平日」から「毎日」に変え、時刻を7:00に設定しました。個別の「保存」ボタンはなく、画面下の「作成」ボタンが保存兼用です。作成後は詳細ページに「次回実行: 明日の7:00」と表示されます。「▶ 今すぐ実行」ボタンで翌朝まで待たずテストできます。

## 個人予定が混入したので、プロンプトでカレンダーを絞った

「今すぐ実行」を押した初回の出力に、こんな予定が含まれていました。

```
5/15 13:15 用事A
5/18 10:00 用事B
```

BriefingテンプレートはデフォルトでGoogle Calendar全体を参照します。私のアカウントには仕事や個人用のカレンダーも繋がっているため、すべての予定が混入してきました。

コネクター側で「特定カレンダーのみ」と絞り込む設定は確認できませんでした。現状の現実解は**プロンプトで指定する**方法です。指示欄の冒頭に1行追加しました。

```
Only consider events in the calendar named "◯◯◯". Ignore all other calendars (personal, family, other businesses).
```

カレンダー名が日本語でも、英語の指示文で問題なく動作しました。

## 絞り込んだら「予定が空っぽ」だった

![](https://assets.st-note.com/img/1779062793-U8VuDTi6NY4MQfKyICSXpJZH.png?width=1200)

カレンダーを絞った後の再実行で、Routinesがとった自律的な動作を実行ログから確認しました。

```
1. ToolSearch 起動
2. "Let me fetch the calendar list first to find the right calendar ID."
3. Google-Calendar: list calendars 実行
4. "Found it. Now fetching upcoming events from ◯◯◯."
5. Google-Calendar: list events 実行
6. "No events returned for the next two weeks. Let me also check recent past events for context."
7. もう1度 list events で過去14日分取得
8. ブリーフィング生成
```

「カレンダー名からIDを自動で調べる」「予定ゼロだったので過去14日も確認する」という判断は指示文に書いていません。プロンプトで意図を伝えると、手順は自律的に組み立てます。

そして出力された結果が、想定とは違うものでした。

> Schedule（今後2週間）: **なし**

「予定: なし」はブリーフィング側の不具合ではなく、カレンダーに何も登録されていないという事実の反映でした。今回見たカレンダーは、5/2以降14日間イベントゼロ。毎朝Routinesが動いても「今日の予定: なし」を返し続けることになります。前提となるカレンダー運用そのものが止まっていた、というのが今回の発見です。

---

## 繰り返しルールで予定の枠を先に置いた

Routinesが「予定がない」と正直に報告したことで、カレンダー整備を先に進めました。対象カレンダーに繰り返しルール（RRULE）を使い、定例の枠だけをまとめて登録しています。種類別の内訳はこのくらいです。

種類件数朝の定例（複数曜日 × 4週）12週次の定例（月曜 × 4週）4月次の単発（月末）1**合計17**

具体的な時刻や曜日は省きますが、「定例の枠を先に置く」「中身は当日決める」という運用に切り替えました。これでRoutinesが毎朝拾える予定が揃います。

補足として、今回作成したBriefingはリモート実行です。Claude Desktopの管理画面には「ローカル」と「リモート」のラベルがあり、ローカルは「コンピューターが起動している間のみ実行」と注釈されています。リモートはPCがスリープや電源オフでも動きます。朝7時の自動実行はリモート前提で組みました。

---

## 翌朝の自動実行 — 実際に動いた記録

設定の翌朝、結果を確認しました。PCはスリープ状態でしたが、リモート実行のため問題なく動いていました。

実行は7:10 GMT+9に「予定済み」ステータスで完了していました。公式仕様の「サーバー負荷を分散するため、実行は数分ずつずらして行われます」がそのまま反映されています。トリガーが7:00で、実際の実行は7:10。10分の遅延は事前に予告されていた通りでした。

出力されたブリーフィングは以下の構造でした。

```
1. Schedule
| 日時      | イベント   |
| 当日 朝   | 記事A 公開 |
| 2日後 朝  | 記事B 公開 |
| 2日後 午前 | 週次の集計 |
| 4日後 朝  | 記事C 公開 |

準備メモ：
- 2日後の週次集計に備え、先週分のデータを事前にまとめておくこと
- 記事B 公開に合わせて、X告知文も事前に準備が必要

注：メール・DMのデータには接続していないため、Section 2〜4はカレンダー以外の
ソースが必要です。それらのツールを接続いただければ追加でブリーフィングできます。
```

前日に登録した5パターンの繰り返しイベントが、想定通り抽出されていました。さらに「準備メモ」として、登録イベントの内容から逆算した具体的なアクションが自動で添えられていた点が予想外でした。指示文に「準備メモを書いて」とは入れていません。Scheduleを並べた延長で、Claude側が自律的に推測して追加した項目です。

未接続のコネクター（メール・DM）には、「接続していないので扱えない」と正直な注記が入っていた点も確認できました。推測で埋めずに済ませている挙動は、ハルシネーションを警戒する身としては今回いちばん安心した部分です。

想定どおりに動いた記録です。

## まとめ

設定そのものは数分で終わります。Routinesのページを開き、Briefingテンプレートを選んで時刻を変更し、「作成」を押す。それだけです。

動かして気づいたのは機能の話より「自分の運用の穴」の話でした。カレンダーに何も入っていなければブリーフィングは毎日「予定なし」を返し続ける。自動化は、止まっていた習慣を浮き彫りにする使い方もできます。
