# Claude Code で最小再現環境の構築を自動化する — minimal-repro skillの紹介

## TL;DR

バグの原因切り分けに必要な「最小再現環境（minimal reproduction）」の構築を Claude Code のスキルとして自動化しました。[`/minimal-repro`](https://github.com/t6adev/claude-code-tools/blob/main/tools/skills/code/minimal-repro/SKILL.md) と打つだけで、複数パターンの再現環境を一括生成し、実行・比較・レポート出力まで一気通貫でやってくれます。実際に Vite 8 アップグレード時の問題でこのスキルを試して、原因の特定と解決に至ったので、使える事例として紹介してみようと思います。  
(※こういうskillが既に世にあるとは思いつつ自分の意図したものとは確実にズレると思い作りました。調べてもいないのでもしあれば教えてください！)

## モチベーション

ライブラリのバグを踏んだとき、Issue を報告するにも、自分で原因を追うにも、まず必要になるのが最小再現環境です。ただ、これが地味に手間です。

* 元プロジェクトから関連コードだけを抜き出す
* `package.json` を最小構成で書く
* バージョン違いで何パターンも環境を作って比較する
* 必要があれば結果をまとめてレポートにする

1 パターンならまだしも、「このバージョンでは？」「このプラグインを外したら？」と条件を変えながら複数パターン試すとなると、かなりの時間を取られます。

この手の作業は構造がはっきりしている割に手を動かす量が多くて、なんとか AI にやらせられないかなとずっと思っていました。

## きっかけ：Vite 8 での問題

そんなことを考えていたところ、ちょうど Vite 8 へのアップグレード時に実際の問題が発生しました。これはいい機会だなと思って、スキルを作って試してみることにしました。

結果として、複数バージョンでの比較環境が自動生成されて、原因の特定から解決まで進めることができました。Vite 8 というホットな話題でもあったので、スキルの実用性を示すにはちょうどいい事例になったのではないかと思います。

## スキルの仕組み

### 使い方

一番シンプルな使い方はこんな感じです。

```
/minimal-repro Vite 8 でビルドすると plugin-react でエラーが出る
```

実際には、プロジェクト内部の文脈（どの PR で問題が起きたか、関連するエラーログなど）をもっと添えて使いました。このあたりは問題の性質に応じて自由に書けるので、上記はあくまで一例として捉えていただければと思います。

引数なしで `/minimal-repro` とだけ打てば、対話的に問題の詳細を聞いてくれるモードもあります。

### ワークフロー

スキルを実行すると、以下の一連の作業を Claude Code が自動で進めてくれます。

1. **問題の把握** — エラーメッセージや関連パッケージを整理。PR URL やコミットハッシュを渡せば、変更内容から自動で情報を抽出してくれます
2. **環境分析** — 元プロジェクトの `package.json`、`tsconfig.json`、Node.js バージョンなどを収集
3. **切り分けパターンの設計** — 問題に応じたパターンを選定
4. **再現環境の一括生成** — シェルスクリプトでディレクトリ構造を自動生成
5. **依存インストールと動作確認** — 各パターンで `pnpm install && pnpm run repro` を実行
6. **結果の整理とレポート出力** — 内部レポートと外部レポートの 2 種類を生成

### 工夫その1：複数パターンの一括生成

このスキルで一番こだわったのは、**環境を複数パターン用意する**ようにした点です。  
（こだわったと言っても[skill-development](https://github.com/anthropics/claude-code/blob/main/plugins/plugin-dev/skills/skill-development/SKILL.md)に作らせただけですが・・・）

「最小再現環境を 1 つ作る」だけだと、原因の切り分けとしては足りない事も多いです。「バージョン A では動くのに B では壊れる」「このプラグインを外すと正常になる」みたいな比較ができて初めて、原因に迫れると思っています。

スキルでは 4 種類の切り分けパターンを定義しています。

| パターン | 目的 | 例 |
| --- | --- | --- |
| **baseline** | 現環境の最小再現（必須） | 現在の構成をそのまま最小化 |
| **version-matrix** | バージョン違いの比較 | Vite 7 vs Vite 8 |
| **isolation** | 特定パッケージの除外・置換 | plugin-react を外して試す |
| **code-variant** | コードの書き方を変える | ESM vs CommonJS |

生成されるディレクトリ構造はこのようになります。

```
repro-vite-build-error/
├── REPORT.md              # 内部レポート
├── REPORT-EXTERNAL.md     # 外部レポート
├── baseline/              # 現環境の最小再現
│   ├── package.json
│   ├── repro.ts
│   └── README.md
├── version-vite-7/        # Vite 7 で試す
│   └── ...
└── no-plugin-react/       # プラグインなしで試す
    └── ...
```

結果を見て追加の切り分けが必要になったら、既存パターンから派生して新しいパターンを作ることもできます。横展開スクリプトを使えば、あるパターンに追加したテストファイルを他の全パターンに一括コピーすることも可能です。

### 工夫その2：内部レポートと外部レポートの 2 種類を出力

せっかく再現環境を作って調査したなら、その結果を自分だけで閉じるのはもったいないです。そこで、**レポートを内部用と外部用の 2 種類**出すようにしました。

* **内部レポート（`REPORT.md`）** — 自分やチームが読むためのもの。プロジェクト固有の情報（PR URL、ブランチ名、ファイルパスなど）をそのまま含みます。「次に何をすべきか」の推奨アクション付きです
* **外部レポート（`REPORT-EXTERNAL.md`）** — パッケージ作者やコミュニティが読むためのもの。プロジェクト固有の情報を排除して、問題を一般化して記述します。`package.json` と再現コードの全文を含んでいて、コピー&ペーストで誰でも再現できる品質を目指しています

外部レポートはそのまま GitHub Issue の報告に使えますし、再現リポジトリの公開フローも組み込んであるので、調査結果を他の開発者に共有しやすくなっています。内部レポートで自分の問題を解決しつつ、外部レポートで同じ問題に困っている人の助けにもなる、この二段構えにしたのは、個人的にはけっこう気に入っているポイントです。

## AI slop と AI 活用のバランス

最近「AI slop」、特にOSSにおけるAI Slop問題を目にします。

AI が生成した低品質な Issue や PR が OSS プロジェクトに大量に投げ込まれる問題が話題になっています。これは今に始まったことではなく、生成AIが登場した当初からよく耳に入ってきていました。

メンテナの負担を増やすだけの雑な貢献(?)は、AIの悪い使い方の典型例です。

一方で、このスキルがやっていることは逆方向かなと感じています。

* **再現環境を丁寧に作る** — 「動きません」ではなく、コピー&ペーストで再現できる環境を提供します
* **複数条件で検証する** — どのバージョンで壊れるか、何が原因かを切り分けた上で報告できます
* **メンテナが読みやすい形式で書く** — プロジェクト固有の情報を排除して、一般化された問題として記述します
* **既知の Issue を事前に調査する** — 重複報告を避けるために、関連リポジトリの既存 Issue を検索してから報告します

AI に「Issue を書かせる」のではなく、AI に「質の高い再現環境を構築させる」。手間のかかる構造化作業は自動化して、人間が判断すべき部分（問題の解釈、報告するかの判断、追加調査の方針）は人間が行う。この形での AI 活用であれば、OSS エコシステムにとってもプラスになるんじゃないでしょうか。（なったらいいな。。。）

もちろん、出力した内容は改めて自身でチェックする必要があります。issueとして報告できればそれも良し、いまいち判断がつかなければ迷惑にならない範囲でネットに放流しておくだけでも良いかなと思います。（↑で紹介した記事も結果は良かったものの、中身は未検証なので時間があればチェックしてvite/viteestあたりにissueを立てようと思いますが、というところです）

## まとめ

minimal-repro スキルは、バグの原因切り分けにおける「構造化された作業」を AI にやってもらうアプローチです。複数パターンの環境構築、実行、結果比較、レポート生成という一連の定型作業を自動化しつつ、問題の解釈や判断は人間に委ねます。

AI にコードを書かせるのでも、Issue を書かせるのでもなく、**質の高い再現環境を構築させる**、です。地味ですが確実に手間は減りますし、結果としてバグ報告に貢献できる道も作れます。  
こういう使い方が、もっと広がってもいいんじゃないかなと思っています。

## 注意事項

このスキルは出来たてほやほやで、まだ上記のような事例でしか実戦投入していません。改善の余地は大いにありますし、汎用性についてはまだまだでしょう。「こういうケースではうまくいかなかった」というフィードバックがあれば、ぜひ教えていただけると嬉しいです。
