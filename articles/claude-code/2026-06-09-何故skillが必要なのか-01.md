# はじめに

最初に Skill の存在を知ったとき、システムプロンプトや Agents as Tools 等のマルチエージェントと比べて｢本当に必要なの?｣と思ってしまいました。  
ですが調べてみると、LLM の性質を非常に上手に利用しており、コンテキスト効率・保守性・テスト容易性の全てを改善する仕組みであることに気づきました。

この記事では、Skill がどのように優れているのかを体系的にまとめていきます。

## この記事で得られること

1. Skill がどのように動作し、LLM の推論にどう影響するかが分かる
2. コンテキスト効率・保守性・テスト容易性がどう改善されるか理解できる
3. 単一プロンプトとの比較から、Skill を使うべき場面を判断できる

# Skill の構成と仕組み

まずは　Skill の構成と仕組みについて見てみましょう。

## 最小構成の Skill の要素

Skillの最小構成は下記の通りです。

| セクション | 用途 |
| --- | --- |
| name | Skill の名称(システムプロンプトに埋め込まれる) |
| description | Skill の説明(システムプロンプトに埋め込まれる) |
| 本文 | Skill で読み込まれるコンテキスト |

イメージとしては手順書の表紙と概要が name と description でその中身が Skill 本文のコンテキストでしょうか。

## Skill の仕組み

次にAIエージェントがどういった仕組みで　Skill を利用しているかを見てみましょう。

システムプロンプトに埋め込まれた name と description をヒントに AIエージェント が自律的に考えて Skill が必要と判断とすると Tool Calling の仕組みを利用して Skill の本文を取得します。  
その後、AIエージェントはチャット履歴の末尾に挿入された Skill の本文を見てユーザーへの応答を考えます。

### Skill の実行フロー

* Skill 読み込み前に「Skill が必要かどうかの判断」が入る
* Skill 読み込み後に「Skill の内容を踏まえた再推論」が入る
* そのため Skill の読み込みは1回の推論では完結しない

# Skill の効果

次に　Skill を利用するとどういった効果があるかを見てみましょう。

## 1. コンテキスト効率の改善

Skill は必要な時にのみコンテキストとして挿入されるためコンテキスト効率が良くなります。また、コンテキストの挿入箇所が LLM にとって注目されやすい位置となります。

### LLMの性質

* Lost in the Middle
  + 中間コンテキストは無視される傾向にある
  + 末尾 > 先頭 > 中間 の順に注目されやすい
    - **末尾にあるコンテキストが注目**されることが Skill における重要なポイント

### Skill を使ったチャット履歴のイメージ

| No. | チャット履歴の内容 |
| --- | --- |
| 1 | システムプロンプト（固定） |
| 2 | ユーザーチャット |
| 3 | Tool Calling 実行(Skillの読み込み) |
| 4 | Tool Calling 応答(Skill 本文) |
| 5 | Skillを使った回答 |

※ No.5の回答を生成するためにTool Callingのループ等が発生するケースもあります。

No.4のように Skill のコンテキストが末尾に挿入された状態でAIエージェントが回答を考えるため、それを意識した回答が高確率で生成されます。  
この仕組みはチャット履歴が長くなった場合も同様です。

> ただし、Skillの利用は読み込みに時間がかかるので「ボタン押下→要約」といった単一のタスクは不向きです。そういったケースは素直にシステムプロンプトを調整しましょう。

## 2. 管理コストの最適化

Skill を利用することでコンテキストの管理が楽になるため、保守性が向上します。

### 単一のシステムプロンプトの問題

| 問題 | 説明 |
| --- | --- |
| 修正影響の不透明性 | 巨大な単一プロンプトを変更すると、どの応答に影響が出るか追いにくい。 |
| 肥大化 | 全ルールを一箇所に詰め込むと可読性と保守性が低下する。 |

### Skill の管理

| メリット | 説明 |
| --- | --- |
| 分割管理 | 機能ごとに Skill を分けることで影響範囲を限定できる。 |
| 差分更新が容易 | 個別 Skill の更新だけで振る舞いを変えられる。 |
| 再利用性 | 共通の手順やルールを複数のAIエージェントで使い回せる。 |

人にとって分かりやすい単位で責務が分離されるため、管理が行いやすくなります。

## 3. 検証パターンの単純化

システムプロンプト単体では「いつ、そのコンテキストが参照されるか」が曖昧でしたが、Skill は 「読み込みが発生したか」 と 「読み込まれた後に想定通り動作するか」 の二段階に分けて検証できます。

* Skill が読み込まれるか
  + 質問に対して 特定 Skillの Tool Calling が実行されることを確認するテスト
  + Skill の name と description が対象になります
* Skill が読み込まれた後に想定通りに動作するか
  + 挿入された Skill に従って応答が変化するかを検証するテスト
  + Skill の本文が対象となります

読み込みという概念が追加されたことで複雑になったように見えますが実際にはコンテキストが意識されているかどうかを区別できるようになったという点は面白いですね。

## 比較表

| **観点** | **単一システムプロンプト** | **Skill** |
| --- | --- | --- |
| **可読性** | 低い | 高い |
| **修正影響の追跡** | 困難 | 容易 |
| **コンテキスト効率** | 低（埋もれやすい） | 高（必要時に最適位置へ挿入） |
| **テストの単純さ** | 複雑 | 単純化される |
| **運用コスト** | 高 | 低〜中 |
| **再利用性** | 低 | 高 |
| **適正** | 単純なタスク | 複雑なタスク |

# まとめ

Skill は「Tool Calling でコンテキストを追加するだけ」のシンプルな仕組みですが、実際には下記のメリットを得られるAIエージェント開発において非常に強力な手段でした。

* LLM の性質（末尾が強い）を最大限に活かす
* コンテキスト管理の保守性が劇的に向上する
* テスト容易性が上がる
* 再利用性が高い

特に、Skill 読み込み → 再推論 の流れは、従来の「巨大なシステムプロンプト一本勝負」では得られなかった柔軟性をもたらします。ぜひ、AIエージェント開発に有効活用していきましょう。
