## はじめに

少し前に [「推測」から「計測」へ ― AIネイティブ時代における意思決定スタンスのアンラーニング](https://zenn.dev/fixu/articles/measure-dont-guess-ai-native) という記事を書きました。AI を業務に組み込んだことで計測コストの桁が変わり、エンジニアの意思決定スタンスそのものを書き換える必要が出てきた、という話です。

あれから少し時間が経ち、計測コストの低下が、エンジニアの意思決定だけでなく、**CS（カスタマーサポート）業務の一次調査** にまで侵食してきました。本記事は、CS担当者から Slack で受け取った問い合わせを、Claude Code (Opus + デスクトップアプリ版) に渡して **コード調査 → 本番DB SELECT → 本番ログ grep → Slack返信** までを30分弱で駆け抜けた現場記録です。

前回と外形は似ていますが、波及している層が違います。

| 観点 | 前回（推測 vs 計測） | 今回（CS問い合わせ調査） |
| --- | --- | --- |
| AIが侵食した領域 | エンジニアの意思決定（方針判断 / キャパプラ） | CS問い合わせ調査（コード〜本番ログ横断の一次調査） |
| 計測コストが下がった対象 | 機能利用量 / コード品質 | お客様一人の行動時系列 / 切り分け |
| 組織への波及 | 個人の意思決定スタンスの更新 | 職能境界の再設計（CS ↔ エンジニア） |
| 残る人間負荷 | 戦略層の判断 | 委譲経路の業務設計（権限・鍵管理） |

そしてもう一つ、この日の体験で一番おもしろかったのは、調査のスピードそのものよりも、**「これ自分でもできるんですか？」** という同僚たちのリアクションでした。CSとエンジニアのあいだに引かれていた職能の線が、一段薄くなった瞬間の話としても読んでいただければ嬉しいです。

## 1. お客様の声を起点に — CS現場から飛んできた問い合わせ

ある朝、CS担当者から Slack の社内チャネルに、こんな趣旨の問い合わせが投げ込まれました（個別情報は伏せています）。

> あるお客様が利用申請をされたのですが、店舗側の管理画面の「利用申請管理」一覧に該当者が表示されないようです。利用申請の通知メールは届いているとのこと。原因を教えていただけますか？

合わせて、店舗側に届いた通知メールのスクリーンショットも添えられていました。対象店舗・申請ユーザー名・プラン・期間・決済金額（このケースは0円）が一通り写っていて、現場で必要な情報はきれいに揃っている状態です。

ここで一点、強調しておきたいことがあります。CS担当者がこの状態で渡してくれた情報は、**自然言語の問い合わせとして十分明瞭**でした。メール本文から抽出された対象データが箇条書きで提示されていて、エンジニア側が「で、どの店舗ですか？」「申請日いつですか？」と聞き返す必要がなかった。これが後の30分調査を成立させた最初のピースです。

CS担当者は「通知だけ届いて画面に出ないなんて、そんなことあるのか？」という仮説を一緒に添えてくれていました。これは、運用を熟知している現場ならではの良い問いです。

## 2. ワークフロー全体像 — 30分のタイムライン

実際にやった作業を時系列で並べると、次のようになります。

| 経過時間 | 作業 | 主な道具 |
| --- | --- | --- |
| 0:00 | Slack スレッドを読んで状況把握 | Claude Code (Slack MCP) |
| 0:02 | コードベースで「利用申請管理」一覧の表示条件を調査開始 | Claude Code (subagent: Explore) |
| 0:08 | コード上の最有力仮説「審査不要プランで自動承認されたケース」を特定 | (同上) |
| 0:10 | 本番DBに参照クエリを投げて該当レコードを確認 | Claude Code (社内 fixu-aws-mysql スキル) |
| 0:13 | 仮説を反証するデータを発見（実は審査ありプランだった）→ 別仮説へ |  |
| 0:15 | 「論理削除フラグが立つコードパス」を網羅調査 | Claude Code (subagent: Explore) |
| 0:20 | 本番 nginx access log を該当時刻でgrep | Claude Code (社内 fixu-aws-log スキル) |
| 0:24 | お客様の行動を秒単位で時系列再構成 → 真の原因を特定 |  |
| 0:27 | 中間報告（私がレビュー） | チャット |
| 0:30 | CS担当者への報告 + エンジニア同僚へのダブルチェック依頼を Slack スレッドに投稿 | Claude Code (Slack MCP) |

**従来の感覚で見積もると、半日コースの内容です。** コードを開き、ローカルでgrepし、本番サーバに SSH して、ログを掘り、DBに繋いで、ステークホルダーに丁寧な報告文を書く。これを一人のエンジニアが手作業でやれば、午前中いっぱい消えます。

それが、自然言語で「この問い合わせに返答したい。1. コードベースで使用を把握して 2. ログとデータを確認して 3. 内容をまとめて私のレビューを受けて 4. Slackに回答して」と4ステップ書いて Claude Code に渡すだけで、30分で帰結しました。

## 3. 各ステップの中身 — Claude Code が何をしたか

ここからは技術記事として、各ステップで Claude Code が具体的に何をしたかを残しておきます。再現性が伝わるように書きますが、社内固有の情報は抽象化しています。

### 3.1 コードベース調査（subagent 委譲）

Claude Code には、メインセッションから派生させる **subagent** という仕組みがあります。コンテキストを汚さずに探索系のタスクをまかせられるので、「リポジトリ内で利用申請一覧の表示条件を網羅的に特定して」というような、grepと読解の往復が多い作業を投げるのに向いています。

今回は subagent に対して、

* 管理画面の「利用申請管理」一覧画面のコントローラ・モデル・ビュー・ルーティングを特定
* 表示条件（WHERE句、scope、店舗との紐付き、状態フィルタ）の抽出
* 「通知メールは届くが画面には出ない」を引き起こしうる条件分岐の仮説出し

を依頼しました。返ってきたレポートには、コントローラのファイルパス＋行番号、Eloquent クエリの抜粋、論理削除フラグや審査結果フラグの値の意味、そして「審査不要プランで自動承認されたケースなら、通知メールは送信されるが審査待ち一覧には出ない」という最有力仮説が綺麗にまとまっていました。

この時点でかかった時間はおよそ8分。手作業で同等の網羅性を担保しようとすると、ファイル全文検索を繰り返すだけで30分は溶ける範囲です。

### 3.2 本番DB SELECT

最有力仮説を反証するためのクエリを、社内スキル（本番DBに参照クエリだけを安全に投げる薄いラッパー）経由で実行しました。

* 対象店舗の特定
* 対象プランの審査要否フラグ確認
* 対象ユーザーの申請レコード取得

結果、**最有力仮説は外れ** ました。対象プランは審査が必要な設定で、申請レコードも審査待ち状態で作成されている。にもかかわらず、論理削除フラグが立っていて、しかも申請レコード作成から15分後に立っている。否認操作なら別フラグも一緒に変わるはずなのに、それは変わっていない。

「否認以外のなにかが、申請の15分後に発火している」ところまで切り分けられました。ここまで15分弱。

### 3.3 削除パスの網羅調査（subagent 再委譲）

「論理削除フラグだけが立ち、審査結果フラグはそのまま」という状態を生みうるコードパスを、subagent にもう一度網羅させます。

返ってきた候補は数本ありましたが、その中に「ユーザー本人がアプリから退会した場合、その人の全申請レコードに論理削除フラグを立てる」というパスがありました。これが今回の真犯人候補として急浮上します。

### 3.4 本番ログ grep

仮説検証の最後は、本番 nginx access log を該当時刻でgrepすることです。社内スキルで該当時間帯のログを取得し、お客様の API トークンに紐づくリクエストだけを抽出しました。

抽出されたリクエスト系列を時系列で並べると、こうなります（時刻と作業の対応関係に置き換えて記載しています）。

* T+0:00 — アプリから利用申請＋クレジットカード登録 API が成功
* T+0:00 〜 T+13:00 — 同じ店舗を再検索したり閲覧したりしている
* T+13:30 — 退会 API がタイムアウト
* T+13:45 — 退会 API がもう一度タイムアウト
* T+14:00 — 退会 API が成功（HTTP 200）
* T+14:15 — DB上で「ユーザー本人」と「申請レコード」が同じ秒に論理削除される
* T+15:00 〜 — 退会後に再ログイン試行を3回（おそらく退会したご自覚なくログインできない状態）

つまり、**お客様ご本人が利用申請の約15分後に退会操作を完了されたため、退会処理のカスケードで申請レコードも論理削除され、店舗側の管理画面から消えた** というのが真相でした。通知メールは申請成立時点で発信されているので、店舗側には「通知だけ届いて画面に出ない」状態が残ります。

これは仕様通りの挙動で、システム不具合ではありません。店舗側としても、お客様が退会されている以上、特に承認操作は不要、というクローズの仕方ができます。

### 3.5 Slack返信ドラフト → 私のレビュー → 投稿

最後は Claude Code がそのまま Slack スレッドに返信する直前まで持っていってくれました。書きあがった返信ドラフトには、

* 結論（仕様通りの挙動である旨）
* 時系列（本番ログ＋DB実測ベース）
* 仕組みの解説（一覧の表示条件と退会処理のカスケード）
* 店舗側の対応指針（承認操作は不要）
* 改善余地（将来「申請取り消し」を店舗側にも通知する等）

が一通り含まれていました。私はそれを読み、CS担当者宛に「報告」、エンジニア同僚宛に「ダブルチェック依頼」という体裁を指示しただけです。あとはメンション込みで Slack スレッドに投稿してくれました。

## 4. クローズまで持っていけたことの意味

地味ですが、ここが一番大事だと思っています。

CS問い合わせ調査の価値は、原因特定ではなく **「現場が次に何をすればいいか」を返せること** にあります。今回の場合、

* 店舗側 → 承認操作は不要
* CS担当者 → 「ユーザー側で退会されたためご対応不要です」とお返しいただければOK
* エンジニア → 仕様通りで対応工数なし、ただし将来の改善余地は別チケット候補

という形で、3者全員の「次の一手」が確定した状態で Slack スレッドに置きました。原因究明だけして「あとはよしなに」と渡されると、CS担当者はもう一段階かみ砕き直す必要があります。クローズ可能な形まで持っていってくれるなら、CS現場にとってAIは初めて「使いやすい同僚」になります。

これは Claude Code の能力というより、**指示を出す側がクローズまでを射程に入れていたかどうか** の差です。今回は「Slackに回答して」とまで指示した結果、自然とクローズドラフトまで作ってくれました。

## 5. 同僚のリアクション — 職能境界が溶ける感覚

返信を Slack に投稿したあと、CS担当者と、ダブルチェックを依頼したエンジニア同僚から、ほぼ同じトーンの反応が返ってきました。

CS担当者から：

> 何と連携したらここまで Claude で調査できるんでしょうか？Claude 使えば自己解決できそうなことを質問してしまい申し訳ございません！

PdMから：

> 僕もこの情報でここまで調査できるものなのかとびっくりしてる

このリアクションが、本記事の核です。

CS担当者の側からすると、「自分が困ったとき、エンジニアに渡さずに自分で解像度を上げられるなら、それが一番いい」というのは自然な発想です。エンジニアの側からすると、「コード・本番DB・本番ログ を横断して30分で結論まで持っていく動きが、AI で再現可能」というのは、自分の職能の輪郭が変わる感覚を伴います。

両側から「この境界、もしかして溶けるのでは？」という気配が同時に出てきた、というのが、この日いちばん印象的な瞬間でした。

組織のなかで誰がどの作業を担当するかは、**「その作業に必要なスキル × アクセス権 × 時間」** の組み合わせで決まってきました。AI はこの式の「スキル」項を大きく圧縮します。残るのは「アクセス権」と「時間」と、最終判断に必要な「文脈理解」です。文脈は CS 担当者のほうがよほど豊かに持っていたりするので、アクセス権の問題さえ解ければ、職能の組み替えは思っていたより早く来るかもしれません。

## 6. このワークフローを成立させているもの

参考までに、この日の30分を支えていた構成要素を列挙しておきます。

* **Claude（有料プラン MAX）** — 並列コンテキストとモデル品質の両方が必要だったので MAX を使用。Pro でも回りそうですが、subagent の同時並列度や本番ログサイズを思うと、安心感が違います
* **Claude Code デスクトップアプリ版** — エディタを開いていない状況でも、Slack を起点にそのままワークフローを走らせられる
* **社内スキル（本番AWS操作の薄いラッパー）**
  + `fixu-aws-log` — 本番ログ取得を type/server で抽象化
  + `fixu-aws-mysql` — 本番DBへのSELECTだけを安全に通す
* **Slack MCP** — スレッド読み取り・メンション付き返信が直接できる
* **GitHub 上の自社リポジトリ全体** — ローカルに clone 済み。AI はここを grep / 読解する

そして、いま現在この構成を使えるのは、**SSH秘密鍵を信頼してもらえている人** に限定されます。これが次の節のテーマです。

## 7. 非エンジニアへの委譲ロードマップ — 渡せない理由と、渡すための条件

CS担当者から「自分でもやれるなら是非」という反応をもらった一方で、私は次のように返しました。

> この作業を CS 担当者にもやってもらえるようにしたいんですが、いまはサーバーに入っていない人でも何でもできちゃう強い権限を渡すことになるので NG です。インフラ刷新完了まであと数日お待ちください。

具体的な制約はこうです。

* 現状の社内スキル群は **SSH秘密鍵を手元に持っていること** を前提にしている
* SSH鍵を全員に配布すると、本来なら参照しか必要のない作業者が、サーバー上で任意のコマンドを実行できてしまう
* これは権限分離の観点で許容できないし、本人に何かあったとき誰が何をしたかの追跡も困難になる

裏を返せば、このSSH鍵依存を取り除けば、CSメンバーへの委譲は技術的にはほぼ準備が整っています。

将来像として描いているのは、

* **AWS IAM ベースの権限管理** — 「誰に何を許可するか」を IAM ユーザー / ロール側で一元定義し、ローカルに鍵を持たないとできないことを潰していく
* **Session Manager 経由のアクセス** — SSH 秘密鍵を配らずに、IAM 認証で必要なコマンドだけを通す経路に置き換える
* **本番ログ取得は CloudWatch Logs に集約** — サーバーに入らなくても、IAM で許可された範囲のログを参照できる
* **本番データは Reader インスタンス + SELECT 権限のみを持った専用 DB アカウント** — Writer インスタンスにも、更新権限のあるアカウントにも、調査用途では一切触らせない
* **誰がいつ何を見たか** の監査ログを残す

この移行の作業は別途進めていて、新しい AWS インフラ側で土台を作っているところです。土台が整い次第、CSメンバー向けに「Claude Code でこれだけは自己解決してみよう」というオンボーディング枠を作る予定です。

## 8. おわりに — 計測コスト激減が組織構造に効く層

[前回の記事](https://zenn.dev/fixu/articles/measure-dont-guess-ai-native) では、計測コストの激減が **エンジニア個人の意思決定スタンス** を書き換える、という話をしました。今回の体験は、その流れがもう一段先に進んで、**組織内の職能境界そのもの** に効きはじめている、という実感を与えてくれました。

今回の時系列の末尾に、私はこんなメモを残していました。

> （おそらく退会したご自覚なくログインできない状態）

これは仕様通りの挙動として処理できる事象ですが、お客様の側には「退会した記憶がないのにログインできない」という小さな戸惑いが残っています。AI が一次調査を巻き取ってくれることの最大の価値は、おそらく調査スピードそのものではなく、**こうしたお客様の戸惑いに、CS や PdM がプロダクト改善やサービス運営の議論としてもっと深く向き合える余白が生まれる** ことだと思っています。エンジニアも「一次調査」から解放され、「真に意思決定が必要な層」「未来の構造を作る層」にもっと時間を割ける。それぞれが、お客様にとって本当に効く仕事に時間を回せる未来です。

そのためには、AI の能力だけでなく、**鍵 / 権限 / 監査経路といった地味な業務設計** を整える必要があります。AIネイティブ化は、モデルの強さだけで進むわけではなく、**「誰にどこまでの権限を渡せば安全に走れるか」を業務設計として一段ずつ詰める仕事** とセットで進みます。

この日の最後に CS 担当者からもらった一言が、私のなかで一番効きました。

> 自分自身でのインプットも大事だと思うので、いい感じに Claude の力借りつつがんばります！

職能境界が溶けるとは、誰かを置いていくことではなく、両側がお互いの領域に半歩ずつ越境できる状態を作ることだと思っています。インフラ刷新が終わったら、次は CS 担当者の半歩を支える仕組み作りに取りかかります。
