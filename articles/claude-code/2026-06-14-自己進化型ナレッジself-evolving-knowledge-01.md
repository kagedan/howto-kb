はじめまして。私はAIエンジニアではありません。機械学習の専門教育も受けていません。ソフトウェア開発チームのマネジメントに10年以上携わってきたプロジェクトマネージャーで、以前はウェブ開発者としての経歴もあります。本格的なAIエージェントが登場してから、週末を使って個人プロジェクトで実験を重ねるようになりました。

そのうちの一つが、語彙学習用のモバイル単語帳アプリです。私は日本語を学んでいるのですが、新しい単語を辞書に追加する作業がストレスにならないサービスが見当たらず、それなら自分で作ってしまおうと考えました。GPUクラスターも開発チームもありません。あるのはMacBook、空いている日曜日、そして解決したい具体的な課題だけでした。

以下では、一介のPMとしての視点から気づいたことと、そこから生まれたアイデアおよびコンセプトについてお伝えします。

## wikiLLM

少し前、Andrej Karpathyが提唱した [**wikiLLM**](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) というアイデアに出会いました。知識を構造化されたwikiとして管理し、エージェントがそれを読み込み、生データで更新しながら回答に活用するという手法です。ロジックはシンプルで美しい - 整理された知識ベースを与えれば、エージェントはより精度高く動作する。そこで試してみました。少なくとも、私なりの解釈で。最初はうまくいっていました。メインの.mdインストラクションファイルを解体し、wiki/フォルダ配下に整理し直したのです。そして、興味深い気づきが訪れました…

PMとして、「余分なもの」と「足りないもの」を見つけることが習慣になっています。このケースも例外ではありませんでした。wikiは整備したものの、それは成長せず、エージェントは「賢く」なりませんでした。なぜか？ 後から振り返れば明白です。初期セットアップ以降、生データをエージェントに与えていなかったため、知識が増えなかったのです。ウェブ検索で少し補完されることはあっても、それだけでした。

なぜそうなったのでしょうか？

これはwikiLLMが悪いアプローチだという話ではありません - Andrej Karpathyは間違いなく私より優秀です =)。単純に、私のユースケースがこの手法に完全には合致していなかった（少なくとも、私がこの手法を理解した範囲では）ということです。私には、wikiの記事として分解できるような生データのストリームがありません。私の生データとは、仮説、コンセプト、機能要望、バグです。必要なのは司書ではなく、プロダクトの課題を解決する**開発者**でした。「このボタンを追加して」「ここにバグがある」と伝えれば、プロジェクト固有のコンテキストをもとに解決してほしい。そして理想的には、類似のケースで同じミスを繰り返さないでほしい。

wikiLLMアプローチのもとで、重要だと思うものを手動でどんどん追加していきました。ルールはwikiへ。得た教訓はlessons-learned.mdへ。重要事項はCLAUDE.mdに直接重複して記載、「確実に認識させるために」。3ヶ月後には7,000トークンのCLAUDE.mdが出来上がり、それでもエージェントはすでに解決したはずのミスを繰り返していました。

問題はコンテンツではありませんでした。問題は、**誰がそれを作成すべきか**でした。

繰り返しますが、これはwikiLLMが悪いアプローチだという話ではありません。私が使い方を誤っていた - そしてそれを理解して初めて、正しい方向へ進めるようになったという話です。

## あの瞬間

長年、開発者と一緒に仕事をしてきました。ジュニアがチームに加わったとき、「知っておくべきことすべて」を200ページのドキュメントとして渡すことはしません。実際の作業を通じて学ぶのです - そして学習の核心となる瞬間は、ほぼ常に同じものです：**予想外の出来事**。

コードを書いて、想定外の挙動に遭遇する。バグ。エッジケース。ドキュメント化されていないAPIの仕様。そこで本物の知識が生まれます - ドキュメントからではなく、実体験から。1年後には、そうしたケースが頭の中に何十個も蓄積されている。それが、その人を価値ある存在にするのです。

そこで答えが見えてきました：**もしエージェント自身が、予想外の出来事の瞬間に自分のコンテキストを生成するとしたら？**

私が知識を記録するのではなく、エージェントが期待と現実のギャップに気づき、自らドラフトエントリーとして記録する。後で同じ状況が再発したとき、それがパターンであると認識し、ルールへと昇格させる。

これがまさに、ジュニアがシニアへと成長するプロセスです。ドキュメントを読むことによってではなく、実際の課題から積み上げた経験によって。答えは見えました。あとはそのプロセスをモデル化するだけです。

## SEKのコンセプト

こうして生まれたのが、私が **Self-Evolving Knowledge（SEK）** と名付けたコンセプトです。

PMとしての経験が大きく役立ちました。「誰が何に責任を持つか」「手動管理なしで機能するプロセスをどう設計するか」という思考が染みついています。ジュニア→シニアのフレームワークは比喩ではありません。文字通り、新しいチームメンバーのオンボーディングをモデル化しようとしました。

「エージェント-司書」ではなく「エージェント-開発者」のシナリオでwikiLLMアプローチを適用しようとした際に、いくつかの具体的な問題が生じました。

**main\_instruction.mdの肥大化。** エージェントが確実にルールに従うよう、インストラクションに直接貼り付けることになります。6ヶ月後には、wikiの半分が「念のため」そこに移住しています。タスクの内容を問わず、セッション開始時に7,000トークン消費。

**知識の重複。** 「得た教訓」がlessons-learned.mdに存在し、「ルール」はwiki/rules/にある。同じ事実が2か所に。エージェントはどちらが最新か混乱します。

**トークンコストの線形増加。** 「必須」リストに追加するページごとに起動コストが上がります。1年後には、最初のメッセージを入力する前にコンテキストウィンドウの30-40%が消費されています。

**ロードトリガーが散文で書かれている。** 「フロントエンドのタスクの前にこのファイルを読め」はエージェント自身が解釈しなければなりません。機械可読なレイヤーがなく、保証がありません。

しかし、これら4つの問題の根底にある、より深刻な問題があります：**ユーザーが依然として唯一の知識源であること。** エージェントは何も追加しない。手動で入力されたものを消費するだけでした。

SEKはまさにそれを変えます。

## アーキテクチャ

SEKは知識を4つのコンテキストレベルに整理します。指針となる原則は一つ：**今すぐ必要なものだけをロードする**。  
![](https://static.zenn.studio/user-upload/bc678b96779c-20260614.jpg)

## L0 - ブートストラップ（常にコンテキストに）

これがメインインストラクション - CLAUDE.md、AGENTS.md、GEMINI.md、あるいは何であれ。根本的に削ぎ落とされたもの。

目標：起動に必要なすべてを最小限のトークン数に収める：

1. 私は誰で、どの言語で応答するか
2. どのカテゴリの知識を持っているか
3. 何をロードし、そのトリガーは何か

余分なものは一切不要。詳細なルール説明も、例示も、「～をするな、なぜなら…」も不要。骨格だけでよい。

main\_instruction.mdから消えるもの：「やってはいけないこと」のセクション、wikiページの詳細説明、インラインルール - これらはすべてwiki/rules/に移動し、トリガーでロードされます。

## L1 - ルーティングとインデックス（必要時）

**`wiki/_routing.md`** - キーワードとファイルをマッピングするYAML形式のフルルーティングテーブル。L0のルーティングでカバーされない場合にのみ読み込まれます。

```
triggers:
  - id: frontend
    keywords: [react, frontend, ui, component, tsx, jsx]
    load:
      - wiki/design/design-system.md
  - id: llm-cost
    keywords: [pricing, calc_cost, rate, cost]
    load:
      - wiki/references/llm-pricing.md
```

YAMLを選んだのは偶然ではありません - 「キーワード→ファイル」を機械的にパースするための最も低コストかつ正確なフォーマットでありながら、人間にも読みやすいからです。

**`wiki/_index.md`** - 目次：ファイルパスと一行の説明。約200-300トークン。ルーティングで結果が得られない場合のフォールバックレイヤーです。

大きなファイル（design-system、edge-cases）は消えるわけではありません - 「念のため」ロードされなくなるだけです。それらのトピックについての回答品質は低下しません：必要なときにロードされます。

## L2 - 検証済み知識（トリガー時）

プロジェクトの全専門知識：`wiki/rules/`、`wiki/references/`、`wiki/architecture/`、`wiki/workflows/`。タスクがそのトピックに明確に関連する場合にのみロードされます。

各ルールには一意のIDタグ（`[XXX-001]`、`[YYY-ASYNC]`）が付与されます。新しいルールを追加する前に、エージェントはgrepを実行し、同じ内容を二度書きません。

## L3 - エフェメラル状態（新しい知識が生まれる場所）

ここに本当のメカニズムがあります - 「自己学習」の部分です。

`memory/inbox.md` はレッスンのアーカイブではありません。パイプラインです。

```
予想外の出来事（バグ、ワークアラウンド、ドキュメント化されていないAPIの挙動）
        ↓
REGISTER → memory/inbox.md（1エントリー3行）
        ↓
PROMOTE → wiki/rules/ または wiki/references/
        ↓
inboxから削除
```

エージェントは私の関与なしに自らinboxに書き込みます。記録のトリガーは4つ：予想外のシステム挙動、発見したワークアラウンド、ドキュメント化されていないAPIの挙動、私からの明示的な修正。まさにジュニア開発者が「覚えておこう」とメモする状況と同じです。

ハードリミットは20エントリーに設定しています。確認なしにN日を超えたエントリーは自動削除されます。inboxは膨らまない。設計によってエフェメラルであり続けます。

## draft → validated → core

すべての知識は3つの成熟段階を経ます。  
![](https://static.zenn.studio/user-upload/14eccedb7b9e-20260614.jpg)

| レベル | 保存場所 | 記録者 | 信頼度 |
| --- | --- | --- | --- |
| `draft` | `memory/inbox.md` | 事後にエージェントが記録 | 仮説、検証が必要 |
| `validated` | `wiki/rules/`、`wiki/references/` | 繰り返しまたはユーザーによる確認 | 有効なルール |
| `core` | main\_instruction.md（Critical Rules） | ユーザーの明示的な決定 | 違反 = リグレッション |

**draft → validatedへの昇格**は自律的に行われます。同じトピックが再び登場するか、同一トピックに3件以上のエントリーがある場合、昇格候補となります。書き込み前にエージェントは既存ルールをgrep - 重複は書きません。昇格後は応答内に一行：`📚 Promoted: [TAG] → wiki/rules/Y.md`。

**validated → coreへの昇格**には私の明示的な「はい」が必要です。エージェントは提案できます（「ルール[TAG]が今月5回以上登場しました。Critical Rulesに昇格しますか？」）が、main\_instruction.mdを自律的に編集することはありません。これが非対称な保護設計です：書き込みの閾値は低く（積極的に記録）、coreへの閾値は高い（許可が必要）。

**main\_instruction.mdは憲法です。エージェントはそれを書き換えません。**

**エラー保護。** ルールが昇格から7日未満で初めて適用される場合、エージェントはフラグを立てます：`🆕 Using recently promoted: [TAG]. 挙動が一致しない場合は /revoke [TAG]`。`/revoke` コマンドはwiki/からエントリーを削除し、ロールバックを記録します。何も痕跡なく消えることはありません。

## ジュニアからシニアへ

同じプロジェクト上でのエージェントの成長を大まかに示すと：

| ステージ | wiki/rules | memory/inbox | 挙動 |
| --- | --- | --- | --- |
| **ジュニア**（初日） | 5-10件のコアルール | 空 | 質問が多い、webドキュメントに頼る、内部ルールをほとんど参照しない |
| **ミドル** | 20-40件の検証済みルール | 5-10件の新しい観察 | 自信を持ってルールを参照し、パターンを認識し、web検索が減少 |
| **シニア** | 60件以上のルール＋クロスリファレンス | inboxはほぼ空 | 「これは[TAG]のケースです、ルールXを参照」、アーキテクチャ的な解決策を提案 |

![](https://static.zenn.studio/user-upload/1c150ff8cb85-20260614.jpg)

この移行は手動でトリガーする必要がありません。エージェントがプロジェクトに取り組み、REGISTER → PROMOTEサイクルを実行するにつれて自然に起こります。モデルは同じまま。知識ベースが成長し続けるだけです。

PMとして、このパターンが人間で何十回も繰り返されるのを見てきました。ジュニアとシニアの違いは、知性やツールではありません。実際のタスクから積み上げた、構造化された経験です。SEKはその同じメカニクスをエージェントに適用します。

ハードウェアをアップグレードするのではない。人を採用し、成長する時間を与えるのです。

## 何が変わるのか

**SEK**はゼロからの書き直しではありません。

**変わらないもの：** フォルダ構造（wiki/、memory/）、カテゴリロジック（rules、references、services）、既存ファイルの内容、git履歴、すべてのランタイム。

**変わるもの：** エージェントのコンテキストに何がロードされるか、どの順序で、そして最も重要なのは - 誰が新しい知識を作成するか。あなただけではなく、エージェントもです。

## まとめ

SEKは新しいモデルでも、ファインチューニングでも、高価なツールでもありません。誰がエージェントのために知識を作り、どのように蓄積されるかについての、別の見方です。

エージェントはより賢くなる必要はありません。あなたの特定のプロジェクトについて、より多くを知る必要があるのです。その知識はドキュメントからではなく、実際の経験から生まれます：予想外の出来事 → ドラフト → ルール。どんな専門家も成長するのと同じように。

モデルは同じままです。変わるのは、その作業リファレンスがどれだけ豊かになるか、です。
