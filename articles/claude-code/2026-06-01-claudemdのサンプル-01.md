とはいえ、具体的なイメージをつけるのは困難だと思いますので、サンプルとしてプロジェクトから固有の情報をなくしたCLAUDE.mdを下に書いておきます  
PoCレベルではなく10万行を越えるプロジェクトだと思ってください

```
# <PROJECT> プロジェクト指示

> このテンプレートは横断ルールの雛形。`<PROJECT>` / `<project>` / `<host>` などの `<...>` プレースホルダはプロジェクトに合わせて置換する。不要な節は削除し、プロジェクト固有の前提があれば該当箇所に追記する。常時適用ルールは本ファイル (root) に置き、書式・テーブル・手順の詳細はサブ CLAUDE.md か detail store に逃がす。

このファイルはプロジェクト全体に常に適用される横断ルール。**領域固有 (バックエンド / フロントエンド / deploy / ログ署名 など) のルールはサブディレクトリの `CLAUDE.md` に分散** されており、該当領域のファイルを編集するときだけ自動 load される。下記の「ルーティング索引」参照。

## コミット前監査 (必須)

- **コード変更には必ずテストを同時に追加・更新する**
- **commit 前に `/precommit` を実行**して以下を監査:
  1. test の存在確認 (無ければ追加)
  2. fall-through / hard-code / 本番 docker 配慮 / patchwork / 共有コード再利用
- Stop hook `.claude/hooks/pre-commit-audit.ps1` が静的 red-flag を自動検出し、commit を block する
- 詳細は `.claude/commands/precommit.md` と `docs/rules.md`「コード監査チェックリスト」

## ブランチ / worktree ルール (必須)

### 作業サイクル

1. 並行作業は `git worktree` で行ってよい (`git worktree add C:/tmp/<project>-worktrees/<topic> -b <branch>`)
2. 作業完了後、その turn の中で **main に merge** し、**worktree と branch を削除**する
   - `git checkout main && git merge <branch>` → `git worktree remove <path>` → `git branch -d <branch>`
3. 「あとで消す」「session 終了時に消す」は禁止 — 残置は構造的混乱の元

### worktree path は main repo の外 (必須)

- 正規 base: `C:/tmp/<project>-worktrees/<topic>/` (Windows / WSL 共通)
- main repo 内ネストは **禁止** — file watcher / 別 Claude session / linter / auto-formatter が「main repo の一部」として走査・干渉する事故の元
- main repo 内に worktree の残骸を見つけたら `git worktree remove <path>` または `rm -rf <path>` で清掃

### dir と branch の対応

- **一つの dir = 一つの branch** (main dir は main のみ、worktree dir は対応する 1 branch のみ)
- **同一 dir で複数の Claude process を同時に動かさない** (working tree の staged 変更を奪い合う事故原因)
- **複数 worktree を別 process が同時編集する場合も、各 worktree は別々の `C:/tmp/<project>-worktrees/<topic>/` に置く** (path が衝突しなければ干渉しない)
- 新しい作業を始めるときは `git worktree list` と worktree base 配下の残骸を確認

### merge / conflict

- merge 失敗・conflict 解消ミスは通常の bug として fix する (ルール違反ではない)
- 作業ディレクトリ / worktree の残置は構造的混乱を生むのでルール違反扱い

## 開発ルール (絶対不変)

- 配布 archive の名前は常に `<project>.tar.gz`、直下は `<project>/`
- 環境変数は `/etc/<project>/global.env` (非秘密、644 root:root) と `/etc/<project>/secrets.env` (秘密、600 root:root) から読み込む。**これ以外の場所から環境変数を読むのはすべてバグ**。systemd の `EnvironmentFile` は両方を指定 (`secrets.env` は `-` prefix で optional)
- `<project>/config/global.env` は配布用テンプレート。実行時に読み込むものではない
- **破壊的変更は不可**
- **勝手な改良・改善は不可** (ユーザーの指示による変更のみ)
- **新規 API を作成するときは提案と合意の上でのみ**
- 一度成功・動作した内容は、仕様変更や別の問題が発生しない限り変更しない
- README は特別な指示がない限り作成しない
- (プロジェクト固有の前提があればここに記載。例: 個別環境を構築・運用しているのではなく、それを行うプロダクト自体を開発している、など)
- **用語の正典は [docs/glossary.md](docs/glossary.md)**: 新規 doc/code を書くとき、または既存 doc を編集するときに `user`/`key`/`session`/`unit`/`server`/`review` 等の多義語を使う場合は glossary に従う。表記レイヤー (DB snake_case / 概念 PascalCase / prose 日本語) の区別も glossary §1 に従う。既存記述の機械的一括置換はしない

## ルーティング索引 (どの CLAUDE.md が auto-load されるか)

| 編集対象 | auto-load される CLAUDE.md | 主な内容 |
|---|---|---|
| `scripts/` 配下 | [scripts/CLAUDE.md](scripts/CLAUDE.md) | deploy、SSH/bastion、ビルド、archive ルール |
| `<frontend-dir>/` 配下 | [<frontend-dir>/CLAUDE.md](<frontend-dir>/CLAUDE.md) | フロントエンド basePath/port、URL→コード起点、API namespace |
| `<backend-dir>/` 配下 | [<backend-dir>/CLAUDE.md](<backend-dir>/CLAUDE.md) | バックエンド framework 規約、API namespace、DB アクセス |
| `<logging-dir>/` 配下 | [<logging-dir>/CLAUDE.md](<logging-dir>/CLAUDE.md) | ログ署名の絶対原則、ログ収集パイプライン |
| `docs/` 配下 | [docs/CLAUDE.md](docs/CLAUDE.md) | doc 編集 (新規・既存)、TOC、SYMBOLS、glossary 連携 |

**新規領域のルールを追加するとき**: 該当 dir に `CLAUDE.md` を作成 → 本表に1行追加 → root CLAUDE.md からは該当ルールを削除 (重複させない)。

「キーワード → 該当 CLAUDE.md/doc」の細粒度索引は [docs/SYMBOLS.md](docs/SYMBOLS.md) を Grep。

## アーキテクチャ概要 (最小)

- **API namespace 分離**: クライアントは `<client-namespace>/api/*` で **client 向け API** とだけ通信、管理画面 web は `<admin-namespace>/api/*` で **admin 向け API** とだけ通信する。新規 endpoint はこのルールに従う。詳細・移管手順は [docs/architecture/api-namespace-split.md](docs/architecture/api-namespace-split.md)
- **リレーショナル DB** = 構造化データ (エンティティ・関連)、**ログ・統計 DB** = ログ・メトリクス
- 主要コンポーネント: `<component-1>` / `<component-2>` / `<frontend>` など (プロジェクトの構成に置換)
- 領域別詳細はサブ CLAUDE.md (上表) を参照

## 現在の環境

- 環境: `<env-1>`, `<env-2>` (テスト環境など)
- サーバー: `<bastion-host>` (踏み台), `<app-host>`, `<db-host>`, `<proxy-host>` ほか (役割に応じて記載)
- デプロイは **`./scripts/deploy.sh`** が唯一の正規入口 (詳細は [scripts/CLAUDE.md](scripts/CLAUDE.md) と [docs/operations/deploy-canonical-reference.md](docs/operations/deploy-canonical-reference.md))
- SSH: WSL 経由 (`wsl bash -c "ssh <host-alias> ..."`)
- プロセスマネージャ (PM2 等) の稼働ホスト・実行ユーザーをここに明記する

## 詳細仕様

`docs/` ディレクトリに詳細ドキュメントがあります。

| カテゴリ | ディレクトリ | 内容 |
|---------|-------------|------|
| アーキテクチャ | `docs/architecture/` | 認証フロー、ディレクトリ構造、ログシステム など |
| サーバー構成 | `docs/servers/` | 各サーバーの構成 |
| 運用手順 | `docs/operations/` | インストール後設定、セットアップ、監査ログ、自動対応 |
| 開発 | `docs/development/` | API、コンテナ、テスト、ログ収集 |
| クライアント | `docs/client/` | クライアント (デスクトップ / スクリプト) |
| トラブルシューティング | `docs/troubleshooting/` | 過去の障害対応記録 |

詳細な索引は `docs/README.md`。シンボル → 正典 doc 索引は [docs/SYMBOLS.md](docs/SYMBOLS.md)。

## 調査・探索の委譲 (token 削減, 必須)

- **3 箇所以上のファイルを横断する調査、または結果量が事前に予測できない広い探索は Explore subagent に委譲する** (`Agent` tool, `subagent_type=Explore`)。該当ファイルが特定済みのときのみ直接 `Grep`/`Read` を使う
- 巨大ファイル (>1000 行) を読むときは `Read` の `offset`/`limit` で必要部分のみ取得。全体を main 文脈に載せない
- 生成物 (`.next/`, `dist/`, `node_modules/`, lockfile, `<project>.tar.gz`) は `Read` しない。必要なら `Grep` までにとどめる

### 全文 Read 禁止リスト (partial-Read 必須)

以下は **全文 Read 禁止**。各ファイル先頭の `## TOC` を `Grep` で拾い、`Read` の `offset`/`limit` で必要セクションだけ読む。(プロジェクトの大型 doc を実体に合わせて列挙する)

| ファイル | 行数 | 内容 |
|---|---|---|
| `<large-index-doc>.md` | ~XXX (index) | 索引のみ。実体は `<dir>/<area>.md` に分割。各 sub-file 冒頭の TOC から必要セクションだけ partial Read |
| `<large-design-doc>.md` | ~XXXX | (設計内容の要約)。関連する一覧・別観点は別ファイルに分離 |
| `<large-flow-doc>.md` | ~XXXX | (フロー内容の要約)。複数版が併存する場合は冒頭に「どこから読むか」を置き、正典版を明示する |

理由: `Read` で main context に載ったファイルは以降の全 turn で累積課金される。Explore は sub-context で読んで要約だけ返すので、長い session の総 token を大きく下げられる。

### URL を起点とした調査 (必須)

URL を示されたら **その URL を起点として直接ファイル + ログを開く**。grep / find は最後の手段。「念のため」探索は禁止。

URL → コード起点 + ログ場所の対応表は領域別 CLAUDE.md に分散している:

- `<admin-namespace>/api/*` → [<backend-dir>/CLAUDE.md](<backend-dir>/CLAUDE.md) `### URL → コード起点`
- `<client-namespace>/api/*` `<frontend-path>/*` → [<frontend-dir>/CLAUDE.md](<frontend-dir>/CLAUDE.md) `## URL → コード起点`

HTTP レベルの URL hit / 4xx・5xx / proxy エラーは proxy: `/var/log/nginx/access.log` + `error.log`。

手順: (1) URL → 該当ファイルを直接 Read → (2) 該当ログを tail (時刻があれば `--since`、なければ末尾 50 行) → (3) エラー内容と該当コードを突き合わせて対応。特定できない場合のみ grep に切り替える。

### Bash output 削減 (PreToolUse hook, opt-in)

`.claude/hooks/bash-compactor.ps1` が冗長 bash コマンド (`pytest` / `git log` / `git diff` / `docker compose logs` / `docker ps` / `npm install` / `pip install`) を自動でコンパクト版に書き換える。**環境変数 `<PROJECT>_BASH_COMPACT=1` を設定したときのみ動作** (デフォルト OFF)。verbose 系フラグ (`-v`, `-p`, `--patch`, `--tail` 等) があれば書き換えない (意図を尊重)。個別 opt-out はコマンド内に `# no-compact` を含める。書き換えルールと activity log の見方は [.claude/hooks/bash-compactor.ps1](.claude/hooks/bash-compactor.ps1) と [docs/operations/observability.md](docs/operations/observability.md)。

## 知識管理 (Obsidian)

個人ノート (調査・決定・試行錯誤・教訓・rejected 仮説) は Obsidian vault `<PROJECT>/` 配下に蓄積。**ルール本体は書かない** — CLAUDE.md / docs/ が SoT (Obsidian は過去の試行錯誤・rejected 仮説の置き場所。混在は drift 事故の温床)。役割分担・自律記録の置き場所表・成果確認テンプレート・status/archived 蒸留・検索除外クエリ・採用しなかった案の **全詳細は正典 [[<PROJECT>/META/distillation-rules]]** (下記 §番号)。役割分担の前提は `<PROJECT>/knowledge-hierarchy.md`。

常時適用 (毎ターン context に乗せる最小トリガのみ):

- **作業開始時 (必須)**: `obsidian_simple_search` で過去の調査・決定・教訓・rejected 仮説を検索 (現行ルール lookup ではなく「過去に同じ罠を踏んだか / 却下案は何か」)。**結果から `<PROJECT>/archived/` を含む path を post-hoc 除外** (superseded/obsolete が top hit に出る事故防止)。残り hit を読んでから着手。クエリ詳細は distillation-rules §4
- **作業中・終了時 (自律記録)**: 次回も役立つ事実・複数案からの選択・30 分以上詰まった罠を、**ユーザー確認なしに** 記録 (調査=`investigations/`、決定=`decisions/`、つまずき=日次)。空振りも記録。追記後は回答末尾に「Obsidian に記録: `<path>`」と 1 行事後報告。置き場所・タイミングは distillation-rules §8-A、inline 蒸留 trigger は §3
- **決定ノートには「成果確認」セクション必須** (仮説・観察 signal・振り返り日 default 1 週間・結果欄)。テンプレートと書き方は distillation-rules §8-B。overdue な結果欄は `scripts/list_pending_outcomes.py --overdue` が抽出し `/precommit` 冒頭で warning
- **蒸留 (必須)**: ノート冒頭 frontmatter に `status:` (`active` / `superseded by [[..]]` / `obsolete (理由)`)。現実と乖離したノートは **許可確認なしに** obsolete 化 + `archived/<種別>/` へ移動し事後報告。手順は distillation-rules §1-3
- **迷ったら記録する**。**書かない**: SoT 違反 (絶対ルール→CLAUDE.md / 公式設計→docs/ / 正式仕様→docs/architecture/)、本プロジェクトに無関係な話題 (Claude Code の使い方・汎用プログラミング・他プロジェクト・雑談)。一般化・安定化した知見は `docs/` に昇格
```
