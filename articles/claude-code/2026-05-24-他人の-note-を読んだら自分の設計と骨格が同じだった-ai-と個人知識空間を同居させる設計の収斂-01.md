## TL;DR

* ある日 note.com で [長津氏の記事](https://note.com/kadobeya2002/n/na518f56802b2) を読んだ。 故・松岡正剛の著作を Obsidian に取り込み、 AI が自動でノートを生成・接続する仕組みを作っている、 という話
* 私は少し前から `hippocampus-mcp` という、 自分の会話ログと個人ライブラリを PostgreSQL + pgvector に入れて Claude Code から MCP 経由で叩く仕組みを動かしている
* 媒体 (Obsidian の md file vs DB)、 動機 (故人の思想の内在化 vs 自分の continuity)、 方向 (死者を再構築 vs 自分を保存) — どれも対照的
* にもかかわらず両者は **同じ三層構造** (= 人が書く層 / AI が読み書きする層 / ルール層) に独立に到達していた
* これは生物学でいう **収斂進化** に近い。 「LLM が長期記憶を持たない」 + 「人の領域を AI から保全したい」 という共通の選択圧があるとき、 異なる門から似た形質が独立に獲得される
* この記事では両者を flat に並べ、 共通形質・分岐形質・選択圧を整理する。 第三の独立実装があれば見つけたい

## 1. きっかけ

note.com で偶然見かけた [長津氏の記事](https://note.com/kadobeya2002/n/na518f56802b2) を読んでいた。 故・松岡正剛の著作群を Obsidian に取り込んで AI がノートを自動生成する話で、 個人の知識管理 (PKM) と LLM を組み合わせた一実例として読み始めた。

途中で違和感が来た。 設計の骨格が、 自分が動かしている `hippocampus-mcp` と同じだった。 媒体も、 動機も、 対象も違うのに、 「層の切り方」 が一致していた。

`hippocampus-mcp` は 「自分の Claude Code セッション履歴 + ChatGPT export + 個人ライブラリ (蔵書・アニメ字幕・青空文庫など) を PostgreSQL に集めて、 任意の AI セッションから MCP 経由で検索する」 という工学寄りの仕組みで、 一方 長津氏のは 「故人の著作を AI が読み解いてノート空間を構築する」 という詩的な側面を持つ。 にもかかわらず、 出来上がった schema が双子だった。

この一致は偶然ではなく、 同じ選択圧に対する独立解だろう、 というのがこの記事の主張である。

## 2. 標本 A: Obsidian + 松岡正剛 wiki (長津氏)

長津氏の構成を、 私が記事から読み取った範囲でまとめる (元記事も合わせて読んでほしい)。

**対象**: 故・松岡正剛の主要著作 (『千夜千冊』 約 1,857 記事、 『情報の歴史』 224 ページ)。 そこから人物・概念・書籍を抽出し、 相互リンクされたノート空間を作る。

**媒体**: Obsidian (= ローカル markdown file + graph view)。 vault 内に三層を切る:

* **raw 層**: ユーザー (= 長津氏) が書いた一次メモ。 AI は読むが書かない
* **wiki 層**: AI が生成・編集するノート群。 raw 層の取り込みから自動でリンク構造を構築する
* **schema 層**: 命名規則・バックリンクの張り方・タグ体系などのルール定義。 AI が wiki 層を生成するときの制約として参照される

**技術スタック**: NotebookLM で著作からの抽出 (= ハルシネーション抑制のため選定、 と記事内にあった)、 Defuddle CLI で Web 記事を markdown 化、 Claude Code でエージェント操作、 Claude Routines で夜間バッチ。

**動機**: 故人の思想を 「疑似的に内在化」 する体験。 著者は 「AI による編集行為が松岡の手法と同一か」 という問いを意図的に保留し、 主観的な知識体系の構築として受け入れる、 と記事は述べている。

詩的なプロジェクトであり、 同時に運用が地に足のついた具体的な実装でもある。 という二面性がある。

## 3. 標本 B: `hippocampus-mcp` (筆者)

こちらは私が動かしている方。

**対象**: 自分の Claude Code セッション履歴 (= `~/.claude/projects/` 配下の jsonl)、 ChatGPT / claude.ai の export ZIP、 個人ライブラリ (自分が過去に書いたblogや雑誌の原稿・蔵書・青空文庫・アニメ字幕・OCR したマンガなど)、 そして AI エージェント自身が書く memory file 群。

**媒体**: PostgreSQL + pgvector (BGE-M3 で 1024 次元埋め込み)。 MCP server を一本立て、 Claude Code / Codex CLI / その他のエージェントから `search_personal_memory` / `search_library` / `search_ghost_memory` 等のツールで叩く。

**層分け**: 三つに切れている。 後付けで気付いたが、 結果として 長津氏のと同形になっていた:

* **`personal` schema**: 自分の会話ログ・ライブラリ・個人メモ。 人が一次的に書く / 取り込む層
* **`agent.ghost_memories` schema**: AI エージェント (Claude Code / Codex) が各プロジェクトの作業中に書いた memory file を横断ベクトル検索可能にした層。 AI が書き、 別セッションの AI が読む
* **frontmatter scope + allowlist**: どの memory file を ghost 層に dub するかを 「memory file 側の `scope: shared` 宣言 + リポジトリ側の allowlist」 の二重署名で制御する規約層

**enforcement**: 規約だけでなく PostgreSQL role で物理化されている。 `agent_dub` (write 専用)、 `agent_read_mcp` (read 専用 view 経由)、 `agent_acl_admin`、 `agent_purge_admin` の 4 役を分け、 MCP server が握る接続文字列は read-only に絞る。

**動機**: 自分の cross-session continuity。 Claude Code のセッションは閉じれば履歴が context から消えるが、 「あの議論あったよな」 を別セッションから引きたい。 ついでに、 私が読んだ本・観たアニメ・書いた scratch メモも同じ store に置けば、 「あの作品で似た主題を見た」 が AI 側から想起できる。

工学寄りで、 詩的ではない。

## 4. 共通形質

二つを並べると、 構造的に同じ箇所が四つある。

| 形質 | 長津氏 (Obsidian) | `hippocampus-mcp` (PG) |
| --- | --- | --- |
| **三層分離** | raw / wiki / schema | personal / agent.ghost / frontmatter+allowlist |
| **権限の二段** | 人が書く層 / AI が書く層 | 人が書く層 / AI が書く層 |
| **opt-in gating** | schema 層のルールが wiki 生成範囲を縛る | `scope: shared` 宣言 + allowlist の dual-signal |
| **AI-人 混在 store** | 同一 vault に raw と wiki が共存 | 同一 DB に personal と agent.ghost が同居 |

特に三層分離が一致したのは、 私には意外だった。 私は層を切ったとき 「DB schema を分けるのは自然」 程度の理由しか持っていなかった。 長津氏が同じ切り方に到達しているのを見ると、 これは設計者の好みではなく、 何らかの構造的要請が働いている疑いが強くなる。

## 5. 共通の選択圧

両者が同じ形質に到達した理由を、 環境圧として三つ挙げられる。

**(a) LLM が長期記憶を持たない**。 セッションを跨いだ参照ができないので、 外部 store に置いて検索させる必要がある。 これが 「AI が読める個人知識空間」 という niche を成立させる。

**(b) 人が書く領域を AI から保全したい**。 AI に全権を渡して raw メモまで書き換えられると、 人間の一次資料が侵食されて復元不能になる。 だから 「人が書く層」 と 「AI が書く層」 を分ける圧力が働く。

**(c) AI に検索・想起させたい**。 単に保存するだけなら ZIP で済む。 AI から能動的に叩ける形 (= MCP / NotebookLM / agent loop) で構造化する圧力がある。

この三つが同時に働くと、 必然的に 「人の領域」 「AI の領域」 「両者の境界を定義するルール」 の三層に分かれる。 媒体や実装技術に関係なく、 構造としてそうならざるを得ない、 というのが収斂進化として読み解ける部分である。

## 6. 分岐形質

一方、 環境差で別形質を取った部分も四つある。

| 軸 | 長津氏 | `hippocampus-mcp` |
| --- | --- | --- |
| **媒体** | file system (Obsidian markdown + graph) | DB (PostgreSQL + pgvector) |
| **enforcement** | 規約 (= convention) | DB role + SQL GRANT で物理化 |
| **目的** | 構造化 (= ノート間リンクとしての wiki) | 想起 (= ベクトル検索による recall) |
| **方向** | 故人 → 自分 (外部知識の内在化) | 自分 → 自分 (自己の保存) |

媒体の違いは見た目には大きいが、 私の関心では二点目の **enforcement の硬さ** の差が決定的に思える。 Obsidian で raw/wiki を分けても、 AI に vault 全体への write 権限を与えれば規約は破られる。 これは convention 依存の脆さで、 ルールを書いた本人 (= 設計者) がエージェントの行動を全て把握していないと崩れる。

`hippocampus-mcp` では DB role を分けてある。 MCP server が握る接続文字列は read-only の `agent_read_mcp` で、 そもそも書く SQL を発行できない。 dub (= 書き込み) は別プロセス・別 role で、 cron 経由でしか動かない。 規約ではなく権限境界として物理化されている。

ただし、 目的が違うので enforcement の必要量も違う。 長津氏のプロジェクトは故人の著作という固定的な一次資料を扱っており、 raw 層が AI に書き換えられても元の本に戻れる。 `hippocampus-mcp` の場合、 自分の過去会話が改ざんされると元に戻せない (= 私の脳の代替なので)。 後者の方が enforcement を硬くする必要があった、 とも言える。

方向の違いも面白い。 長津氏は 「外部の故人の思想を取り込む」 (= 入力方向)、 私は 「自分の過去を外部に保存する」 (= 出力方向)。 同じ schema を、 矢印を逆向きにして使っている。

## 7. 含意 — local optimum 仮説と募集

両極で同じ三層構造に独立到達したことから、 暫定的な仮説を置く:

> **AI と個人知識空間を同居させる設計の現時点での local optimum は、 「人が書く層 / AI が書く層 / 境界を定義するルール層」 の三層分離である。**

local optimum と書いたのは、 これが唯一解だと主張するには標本が二つでは足りないからである。 反例 (= 三層を持たないが運用が成立している実装) がもしあれば、 この仮説は棄却または修正される。

そこで募集したい。 もし同じ niche で動いている **第三の独立実装** を知っている人がいたら教えてほしい。 条件は以下:

* 個人の知識・会話・記録を扱う store であること
* AI が能動的に読む・書く設計であること
* 設計者が 長津氏や筆者と無関係に到達したものであること (= 同じ記事を読んで真似したものは反例にならない)

三層構造を持っていれば仮説の補強になり、 持っていなければ仮説の修正材料になる。 どちらでも面白い。

---

## 8. 余談

長津氏の記事を読んでいたとき、 「同じことを違う言語で言っている人を見つけた」 という感覚があった。 PKM 界隈と AI エージェント界隈は普段あまり交わらないが、 LLM が両方の niche に同じ選択圧をかけ始めた結果、 設計が収斂し始めている可能性がある。

詩的なアプローチと工学的なアプローチが同じ schema に辿り着くのを見るのは、 単純に楽しい。
