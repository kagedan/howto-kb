## やったこと(要約)

組織内でのAIエージェントの利用拡大と生産性の向上を目的に、設計思想や業務フローに合う形のskillを作って配布しました。また、これが機能するようにCLAUDE.mdやClaude向けのドキュメントも合わせて整備しています。  
Claudeの使用方法が個人任せになっている状況から、活用を一歩進ませた紹介です。

## ⚠ 注意点

* 進歩が早いのでもっと良い方法はすぐに登場します。
* ここでのskillは短期間の利用を想定しています。モデルのバージョンアップなどで環境が変わればskillも大幅に更新するか破棄し、新たなものに乗り換える予定です。
* Claude Codeに対象を絞っています。複数モデルの共用は考慮していません。
* 組織といっても、小さい組織またはチームという括りで捉えてください。大規模な組織全体ではまた異なったアプローチが必要です。

## Claude Code を渡しただけでは組織の生産性は上がらない

利用実態に個人差があるというのは、現時点でのAIエージェント利用で言われる課題点かと思います。ある人は生産性が跳ね上がっても、それがチーム全体で見ると思ったよりも効果が見えない、という現象は私の周りだけではないはずです。

### 課題1: コンテキストの欠如

プロジェクトや組織によって進め方は大きく違います。チーム特有の事情をAIは汲み取ってくれません。ドメイン知識や実装内にある暗黙の了解、GitHubなどのコード管理の仕方だけでなく、顧客などのステイクホルダーとのコミュニケーションのとり方なども影響します。こういったものが積み重なると、Claudeに指示しにくい状況が生まれます。

### 課題2: レビュー負荷

課題1のような状況で生成されたコードが良いものかどうか、レビュー負荷がものすごいです。実装は短縮したが、その時間がレビューや差し戻しに入れ替わっただけというパターンが散見していました。  
（ここでの"良いもの"とはコードの一貫性であったり、システム全体で見たときに問題かないかなどで、修正点だけ見ればAIのほうがずっといいコードな場合がほとんど）

### 課題3: すべての人がAIを使いこなせるわけではない

Zennを始めとする技術記事を見ていると、エージェントを使いこなせている人は、モデルがどのような挙動をする特徴があるのか、どのようなコンテキストを渡せばうまく動くのかをよく理解しているように感じます。しかし、人類全員にそれができるわけではありません。

## skillを作って配布する

### 前段階

大前提として、Claudeが問題なく動作できる環境を整える必要があります。考えることは多くありますが、ここでは組織やプロジェクトのコンテキストに観点を絞ります。

`/init`でCLAUDE.mdを生成するだけではあまりに不十分です。デフォルトの出力には、肌感覚ではCLAUDE.mdに書くべきことではなく、README.mdに記述されるべきことが多いと感じました。  
私はCLAUDE.md(特にルートのもの)には、システム全体の簡潔な要約と、"目次"のようなものを主に用意しました。開発の進め方、フレームワークやライブラリの利用方針、コード規約のように実装上で気をつけること、gitの運用方法などは、例えば`doc/agents`のようなディレクトリを切って個別に配置しています。目次とは、一般的なディレクトリ構成の説明だけでなく、このようなエージェント向けの参照情報を指しています。

### skillを作る

claudeに「こういうことがしたいから、そのためのskillを作って。」と指示するだけです。CLAUDE.mdに組織のコンテキストに当たるものがあるか参照させることも忘れてはいけません。あとは結果を確認しながら修正や追記をしていきます。  
どのプロジェクトでも使えるものは、GitHubに`agent-skills`のようにリポジトリを用意し、`gh skill`のコマンドを用いて各プロジェクトに導入しています。一方で、プロジェクト固有のものや、短期間のみ使用する目的のはっきりした使い捨てのskillはリポジトリに含めています。

### 組織内でのコミュニケーション

残念ながら、作っただけでは組織的には機能しません。こういうものを作ったから使ってほしいとデモンストレーションしてみたり、今している作業を聞いてskillにできないか考えたりする必要があります。聞いてみると、うまくClaudeが使えなかった事例が思っているよりも出てきます。  
そうして見えた課題をskillに落とし込んでいくことを繰り返します。

### skillの例

* 顧客とのやり取りと実装を参照させ、伝えた対応方針と合わせてGitHub Issueをたてる
* Issueと既存コードを参照して実装計画を作り、Issueに追記する
* 計画と承認を終えたIssueの実装とレビューをしPull Requestにする

非常にざっくり書きましたが、例えば"実装"の中身はTDDをサブエージェントを使ってさせるなど、組織としてやらせたいことが含まれていると捉えてください。  
結果、これらはコンテキストを参照するように作っているので、最終的な成果物が実装前の想定と違うという事故が格段に減ります。また、副次的な効果として、人間によるコーディングだけでなくAIへの指示の差でできていたコードの品質が均一化します。

## 見えた課題

1. skillの管理自体が煩雑になりがちです。1つのskillで多くのことをさせると良い結果が得られないために、skillの数が増え、それらが場合によっては密結合になります。エージェントの進化もあるので、私達は使わなくなったskillはすぐに捨てる方針にしています。
2. 既存の古いプロジェクトこそ導入したいのに整備が追いつかない。skillを作ることよりも、エージェントを確かに動かすための環境を用意することに時間と労力がかかります。古いプロジェクトほど、まずは人間による情報整理が必要になるというジレンマがあります。
3. もちろんskillだけで解決しない問題も多くあります。

## まとめ

エージェント向けのドキュメントを整える。これを読んで動くskillを作る。組織的に使えるようにして配布する。それだけではありますが効果をかなり感じます。小さく始めるひとつとしていかがでしょうか。
