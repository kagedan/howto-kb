## 概要

この記事では、Scalar が開発しているシステムアーキテクチャ・ツールキット **nexus-architect** の **product プラグイン**について解説します。

product プラグインは、Claude Code 上で動作する「プロダクトの方向づけ（Product Direction）」のためのエージェントです。プロダクトのビジョンから始めて、対話を通じて\*\*検証駆動（validation-driven）\*\*で UX・仕様・ドメイン・API・SLA/NFR までを段階的に導出し、最後にシステム実装設計を担う architect プラグインへ受け渡します。

「AI に企画から設計までの一連の流れを、抜け漏れなく・根拠つきで支援させる」とはどういうことかを、実際のパイプライン構成とともにご紹介します。

### 対象読者

* AI 駆動開発で企画〜設計フェーズの自動化に取り組んでいる方
* Claude Code のプラグインやスキルで複数ステップの業務を組み立てたい方
* プロダクトマネジメントやドメイン駆動設計（DDD）の進め方に興味があるアーキテクト・エンジニア

## 背景・目的

私たちはこれまで、レガシー刷新やグリーンフィールド設計を支援する architect プラグイン、ScalarDB アプリ開発を支援する scalardb プラグインを nexus-architect として整備してきました。一方で、いざシステム設計を始めようとすると、その**手前**にある「そもそも何を作るべきか」「どの仮説に賭けているのか」という方向づけが、属人的で抜け漏れの多い領域として残っていました。

特に課題だったのは次の2点です。

* **検証されていない整合性**: ドキュメント同士の辻褄は合っているが、「一番リスクの高い仮説が検証されないまま」前に進んでしまう
* **トレーサビリティの断絶**: ビジョンと機能、機能とデータモデル、SLO と NFR のつながりが追えず、後工程で根拠を再構築する手戻りが発生する

product プラグインは、この「方向づけ」のプロセスそのものをパイプライン化し、**最もリスクの高い仮説を早期に Go/No-Go で検証してから深い設計に進む**ことを強制する設計になっています。狙いは、企画から実装設計までを途切れさせず、AI エージェントが根拠つきで橋渡しすることです。

弊社では、こうした AI 駆動開発の考え方を実プロジェクトにも適用しています。たとえば常石造船の基幹システム刷新では、AI エージェントと ScalarDB・Kong を組み合わせ、マイクロサービス化を支援しました。

## 解説：検証駆動パイプラインの全体像

product プラグインは全 21 のスキルから構成され、`/product:start` がオーケストレーターとしてそれらを依存関係順に実行します。各スキルは `/product:<skill-name>` として個別にも呼び出せます。

パイプラインは大きく6つのフェーズと、その間に置かれた**検証ゲート**で構成されます。

### Phase 1：プロダクトコアを言語化する

最初に、対話を通じてプロダクトの核を固めます。

```
# 対話的に方向づけを開始（プロファイルと言語を指定可能）
/product:start --profile=full --lang=ja
```

* `define-vision` — Vision / Mission / Values を Product Vision Board と Amazon 流の PR-FAQ として定義
* `define-success-metrics` — 1つの North Star Metric と 3〜5個の入力指標
* `research-landscape` — 市場規模（TAM/SAM/SOM）や競合・トレンドの調査（任意）
* `design-revenue` — 収益モデルと、再計算可能な便益評価テンプレート
* `define-scope` — 制約を正規化し、スコープの In / Out を決定

### 検証ゲート：最もリスクの高い仮説を先に潰す

product プラグインの中核がこの **validation-assumptions** ゲートです。Phase 1 の成果物から「一番外れたら痛い仮説」を抽出し、**最も安価に検証できるテスト**を割り当てて Go/No-Go を判定します。

* `no-go` の場合は前進を止め、Phase 1 の成果物を修正してから再実行します（失敗ではなく**前向きなイテレーション**として扱います）
* `go` の場合のみ、UX 以降の深い設計に進みます

このゲートは何度でも再実行可能で、ウォーターフォールではなく「検証しながら前に進む」ことを担保します。

### Phase 2〜5：UX から品質要件まで一気通貫で導出

ゲートを通過すると、UX 基盤 → 仕様 → ドメイン/API → 品質要件へと、前工程の成果物を入力にしながら設計を積み上げていきます。

* **UX 基盤**: Jobs-to-be-Done に基づくペルソナ、ステージ×レイヤーのジャーニーマップ、Dunford の5要素ポジショニング
* **仕様化**: Lo-fi な UI モック → 各画面アクションを Command/機能として抽出 → モックと機能からデータモデルを2パスで導出（明示エンティティ → 暗黙エンティティ）
* **ドメイン/API**: 機能とエンティティを境界づけられたコンテキスト（DDD 戦略的設計）へ抽象化し、System / Process / Experience の3層（API-Led）で論理 API を設計
* **品質**: 顧客の期待値から SLI/SLO/SLA とエラーバジェットを設定し、SLO を測定可能な NFR（可用性・p95/p99 レイテンシ等）へ落とし込み

最後に `design-architecture` が、ここまでの成果物を統合してアーキテクチャ案と**技術フィットネス評価**（`tech-stack-fitness.md`）を出力します。

### architect プラグインへの受け渡し

`define-nfr` / `map-domains` / `design-api` までの成果物が揃うと、それらをそのまま architect プラグインの入力として渡せます。

```
# product の成果物をシステム実装設計へ受け渡す
/architect:define-requirements --input=reports/03_domain/
```

技術フィットネス評価で ScalarDB / ScalarDL が **Adopt** と判定された場合は、`/architect:select-scalardb-edition` → `/architect:design-scalardb` へ直接つながります。論理（product）と物理（architect）を分離しつつ、企画から実装設計まで一本の線でつなぐ構成です。

## まとめ

nexus-architect の product プラグインは、「何を作るべきか」という方向づけのプロセスを検証駆動のパイプラインとして構造化し、企画から実装設計までを AI エージェントで橋渡しする試みです。

* **成果:** ビジョンから SLA/NFR までを依存関係順に導出し、最もリスクの高い仮説を早期に Go/No-Go で検証する一連の流れを、21スキルのパイプラインとして整備できました。architect プラグインへの受け渡しまでを含め、企画〜実装設計が途切れません。
* **課題:** 各フェーズの成果物の品質は対話の質に左右されるため、ヒアリングの設計や検証テストの妥当性をさらに磨き込む余地があります。実プロジェクトでの適用知見を反映しながら改善を続けていきます。
