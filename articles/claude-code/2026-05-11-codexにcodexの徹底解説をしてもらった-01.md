## Codex CLI の基本

`codex` を単体で実行すると、対話型の TUI が起動します。公式 docs では、TUI からプロンプト、コード片、スクリーンショットを送れること、Codex の計画や差分を見ながら進められること、会話の resume ができることが説明されています。

よく使う起動例は次のとおりです。

```
codex
codex "Explain this codebase to me"
codex --cd apps/frontend
codex --cd apps/frontend --add-dir ../backend --add-dir ../shared
codex -m gpt-5.5
codex --search "最新の公式仕様を確認しながら、この依存ライブラリの移行手順を調べて"
```

`--cd` は作業ルートを明示します。`--add-dir` は複数ディレクトリにまたがる変更で便利です。たとえば frontend、backend、shared package を同時に触る monorepo では、作業範囲を明示したほうが安全です。

### session の再開と分岐

Codex はローカルに transcript を保存し、以前の会話を再開できます。

```
codex resume
codex resume --last
codex resume --all
codex resume <SESSION_ID>
```

また、現在の会話から別案を試したい場合は `/fork` が使えます。実装方針 A と B を比較したいとき、元の文脈を捨てずに分岐できるのは実務でかなり便利です。

## `codex exec`: 自動化の本命

`codex exec` は非対話モードです。CI、定期ジョブ、レビュー補助、レポート生成など、Codex をスクリプトから呼びたいときに使います。

基本形:

```
codex exec "summarize the repository structure and list the top 5 risky areas"
```

公式 docs では、`codex exec` は進捗を `stderr` に流し、最終メッセージだけを `stdout` に出すと説明されています。つまり、シェルパイプやリダイレクトに組み込みやすい設計です。

```
codex exec "generate release notes for the last 10 commits" | tee release-notes.md
```

stdin と組み合わせることもできます。

```
git diff --staged | codex exec "この差分をレビューして。重大度順に指摘して"
```

機械可読にしたい場合は `--json` を使います。

```
codex exec --json "summarize the repo structure" | jq
```

`--json` を使うと JSON Lines で event が出ます。公式 docs では `thread.started`、`turn.started`、`turn.completed`、`item.*`、`error` などの event type が例示されています。agent message、reasoning、command execution、file change、MCP tool call、web search、plan update も item として扱われます。

非対話実行は権限設定が特に重要です。公式 docs では、`codex exec` の default は read-only sandbox と説明されています。編集が必要な場合だけ明示的に許可します。

```
codex exec --sandbox workspace-write "README のセットアップ手順を実際の scripts に合わせて更新して"
```

`danger-full-access` は、隔離された CI runner や container のように外側で十分に制御されている場合以外は避けるべきです。

## `codex review`: レビュー専用 agent として使う

Codex は実装だけでなくレビューにも使えます。CLI では `/review`、非対話では `codex review` が使えます。

手元の `codex-cli 0.130.0` では、次のようなオプションを確認できました。

```
codex review --uncommitted
codex review --base main
codex review --commit <SHA>
codex review --title "任意のレビュータイトル"

公式 docs では、CLI の `/review` は専用 reviewer を起動し、選択した diff を読み、作業ツリーを触らずに優先度付きの actionable finding を返すと説明されています。

使い方としては、PR を出す前に次のような流れが実用的です。

```
codex review --uncommitted "バグ、セキュリティ、テスト不足を優先してレビューして"

## `codex cloud`: クラウドタスクを CLI から扱う

古い記事や手元の記憶では、CLI からの cloud delegation を「未対応」または「coming soon」と説明している場合があります。しかし 2026-05-11 に確認した公式 CLI features では、`codex cloud` によって terminal から Codex cloud tasks を triage / launch できると説明されています。ここは特に情報が変わりやすいので、手元の CLI version と公式 docs の両方を確認してください。

公式 docs と手元 CLI help を合わせた基本例:

```
codex cloud
codex cloud exec --env ENV_ID "Summarize open bugs"
codex cloud exec --env ENV_ID --attempts 3 "Summarize open bugs"
codex cloud exec --env ENV_ID --branch feature/foo "このブランチで flaky test を調査して"
```

手元の `codex-cli 0.130.0` でも `cloud` サブコマンドが存在し、次を確認できました。

```
codex cloud exec
codex cloud status
codex cloud list
codex cloud apply
codex cloud diff
```

`--attempts` は best-of-N の複数試行を依頼したいとき、`--branch` は cloud task の対象 branch を明示したいときに使います。

使い分けとしては、ローカル CLI は「今の作業ディレクトリで細かく伴走」、Codex cloud は「バックグラウンドで長めのタスクを走らせる」と考えると分かりやすいです。

## Codex web と cloud environment

Codex web は、`chatgpt.com/codex` からクラウド上の Codex にタスクを委譲する入口です。公式 docs では、GitHub account を接続し、Codex が repository を読んで変更や PR 作成まで進められることが説明されています。

Codex cloud の特徴は次のとおりです。

* バックグラウンドで実行できる
* 複数タスクを並列に走らせられる
* 各 task が isolated cloud environment で動く
* GitHub 連携で PR 作成につなげられる
* CLI、IDE、GitHub など複数の入口から扱える

ローカル CLI と違い、Codex cloud は自分の端末そのものを触りません。OpenAI 管理の cloud environment で動くため、依存関係の setup、secret の扱い、internet access の扱いを別途設計する必要があります。

## approval と sandbox: 最重要の安全設計

Codex 運用の核心は、approval と sandbox です。公式 docs では、Codex の安全制御は 2 層で説明されています。

* sandbox mode: 技術的に何ができるか
* approval policy: いつ人間に確認するか

手元の CLI で確認できた sandbox mode は次の 3 つです。

| sandbox mode | 意味 | 使いどころ |
| --- | --- | --- |
| `read-only` | 読み取り中心 | 調査、設計相談、レビュー |
| `workspace-write` | workspace 内の編集を許可 | 普段の実装作業 |
| `danger-full-access` | sandbox を外す | 外部で隔離済みの環境だけ |

approval policy は次のとおりです。

| approval policy | 意味 | 使いどころ |
| --- | --- | --- |
| `untrusted` | 信頼済みの read-only 系 command 以外は確認 | 慎重に始めたいリポジトリ |
| `on-request` | Codex が必要に応じて確認を出す | 普段の対話作業 |
| `never` | 確認を出さない | 自動化や read-only 実行 |
| `on-failure` | 失敗時のみ確認、deprecated | 新規利用では避ける |

公式 docs では、ローカルの default は network access off、write 権限は通常 workspace に制限されると説明されています。`workspace-write + on-request` は実装作業の標準的な落とし所です。

```
codex --sandbox workspace-write --ask-for-approval on-request
```

相談だけしたいときは read-only に寄せます。

```
codex --sandbox read-only "この修正方針を比較して"
```

`--dangerously-bypass-approvals-and-sandbox` は、名前の通り非常に危険です。ローカル端末で普段使いするものではありません。

## internet access と web search を混同しない

Codex では「コマンドがネットワークに出られること」と「Codex が web search tool を使えること」は別物です。

ローカル CLI / IDE extension では、`workspace-write` sandbox の default は network access off です。必要なら設定で明示します。

```
[sandbox_workspace_write]
network_access = true
```

一方で、web search は `web_search` で制御します。公式 docs では、local tasks の web search は default で cached mode になり、OpenAI-maintained index から結果を返すと説明されています。最新情報が必要なときは `--search` または `web_search = "live"` を使います。

```
web_search = "cached"
# web_search = "live"
# web_search = "disabled"
```

Codex cloud では、setup script は internet access を使えますが、agent phase は default で internet access が block されます。必要な場合だけ、environment ごとに domain allowlist と HTTP method を制限して有効化します。

## Plan mode は何のためにあるか

Plan mode は、Codex に「すぐ編集させる」のではなく、まず実行計画を出させるためのモードです。CLI の slash commands docs では、`/plan` が active conversation を plan mode に切り替え、任意で prompt も渡せると説明されています。

```
/plan 認証周りのリファクタ方針を、影響範囲・リスク・テスト戦略つきで整理して
```

Plan mode が効く場面:

* 変更範囲が広い
* 既存設計を壊したくない
* 複数案を比較したい
* セキュリティやデータ移行など失敗コストが高い
* 先にテスト戦略を決めたい
* subagents に分担させる前に作業分解したい

設定では `plan_mode_reasoning_effort` を使って、Plan mode 時だけ reasoning effort を上げられます。

```
model_reasoning_effort = "medium"
plan_mode_reasoning_effort = "high"
```

実務では、複雑な変更ほど「まず `/plan`、方針確認後に実装」という流れが安定します。

## `AGENTS.md`: Codex の品質を底上げする repo instruction

`AGENTS.md` は Codex 運用の土台です。公式 docs では、Codex は作業前に `AGENTS.md` を読み、global guidance と project-specific guidance を重ねると説明されています。

探索順序は重要です。

1. Global scope: `~/.codex/AGENTS.override.md` または `~/.codex/AGENTS.md`
2. Project scope: project root から current working directory までの各階層
3. Merge order: root 側から current directory 側へ連結され、近い階層の指示が後ろに来る

Codex は default で combined size 32 KiB まで読みます。この上限は `project_doc_max_bytes` で調整できます。

`AGENTS.md` に書くべきこと:

* このリポジトリの目的
* セットアップ手順
* テスト、lint、format の実行コマンド
* 触ってよい領域、触ってはいけない領域
* 設計上の制約
* dependency 追加時のルール
* PR 前の確認項目
* レビュー観点
* 出力言語や文体

書かないほうがよいこと:

* secret、token、credential
* すぐ古くなる個人メモ
* 実在しないコマンド
* 「いい感じに」など検証不能な指示
* 長大な設計思想だけで具体的な実行手順がない文章

サンプル:

```
# AGENTS.md

## Repository Expectations

- 日本語で簡潔に回答する。
- 変更前に既存の設計と周辺コードを確認する。
- 新しい依存関係を追加する前に理由を説明する。
- フォーマットだけの大きな差分を混ぜない。

## Commands

- Install: `pnpm install`
- Lint: `pnpm lint`
- Test: `pnpm test`
- Typecheck: `pnpm typecheck`

## Review Checklist

- 既存 API の互換性を壊していないか確認する。
- エラー処理と空状態を確認する。
- UI 変更では mobile 幅も確認する。
- テストできない場合は、理由と残リスクを明記する。
```

## `config.toml`: Codex の運用設定

Codex の user-level configuration は `~/.codex/config.toml` に置きます。project scoped config は `.codex/config.toml` に置けますが、trusted project の場合だけ読み込まれます。

公式 docs の precedence は次の順です。

1. CLI flags と `--config` overrides
2. `--profile <name>` の profile values
3. project config files
4. user config
5. system config
6. built-in defaults

最小構成の例:

```
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

model_reasoning_effort = "medium"
plan_mode_reasoning_effort = "high"

[history]
persistence = "save-all"
```

用途別 profile の例:

```
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

[profiles.readonly]
sandbox_mode = "read-only"
approval_policy = "never"

[profiles.deep-review]
model_reasoning_effort = "high"
approval_policy = "never"
sandbox_mode = "read-only"

[profiles.automation]
approval_policy = "never"
sandbox_mode = "workspace-write"
```

使い方:

```
codex --profile readonly "この変更の実装計画だけ立てて"
codex --profile deep-review "このブランチを main と比較してレビューして"
codex exec --profile automation "docs の表記ゆれを修正して"
```

Profiles は公式 docs 上で experimental と説明されています。安定運用では、チーム全体に強制するというより、自分の作業モードを切り替える用途から始めるのが良いです。

## Skills: 再利用できる作業手順

Skills は、Codex に task-specific な capability を追加する仕組みです。公式 docs では、skill は instructions、resources、optional scripts を package 化したものと説明されています。

重要なのは、Skills が単なる prompt 保存ではないことです。特定の作業を安定して再現するための workflow package です。

Skill の基本構成:

```
my-skill/
  SKILL.md
  scripts/
  references/
  assets/
  agents/
    openai.yaml
```

`SKILL.md` には `name` と `description` が必要です。Codex は最初から全 skill の全文を読むのではなく、まず name / description / file path を見て、必要になった skill の `SKILL.md` を後から読みます。公式 docs ではこれを progressive disclosure と説明しています。

簡単な skill 例:

```
---
name: pr-review-checklist
description: Use when reviewing a pull request for bugs, regressions, missing tests, and security risks.
---

# PR Review Checklist

1. 差分全体を確認する。
2. 重大度順に finding を並べる。
3. file path と line を明記する。
4. 推測で断定しない。
5. テスト不足がある場合は残リスクを書く。
```

保存場所は repository、user、admin、system scope があります。repository では `.agents/skills` が使われ、Codex は current working directory から repository root まで探索します。

## MCP: 外部ツールと文脈をつなぐ

MCP は Model Context Protocol のことです。Codex docs では、MCP は model を tools と context につなぐ仕組みとして説明されています。Codex は CLI と IDE extension の両方で MCP servers を support します。

MCP でつなげるものの例:

* ドキュメント検索
* GitHub や issue tracker
* Figma
* ブラウザ操作
* 社内 API
* database や observability tool
* 独自の社内知識ベース

CLI で MCP server を追加する例:

```
codex mcp add context7 -- npx -y @upstash/context7-mcp
```

設定ファイルで書く場合:

```
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
startup_timeout_sec = 10
tool_timeout_sec = 60
```

HTTP MCP server の場合:

```
[mcp_servers.github]
url = "https://github-mcp.example.com/mcp"
bearer_token_env_var = "GITHUB_TOKEN"
enabled_tools = ["list_issues", "get_pull_request"]
disabled_tools = ["delete_repository"]
```

TUI では `/mcp` で active MCP servers を確認できます。OAuth を持つ MCP server では `codex mcp login <server-name>` が使えます。

## Subagents: 並列化と分業

Subagents は、Codex が specialized agents を並列に spawn し、その結果を集約する仕組みです。公式 docs では、codebase exploration や multi-step feature plan のような並列性の高い task に有効と説明されています。

重要な制約もあります。Codex は明示的に依頼されたときだけ subagent を spawn します。また、subagents はそれぞれ model と tool work を行うため、単一 agent より token を消費します。

Codex には built-in agents があります。

| agent | 役割 |
| --- | --- |
| `default` | 汎用 |
| `worker` | 実装や修正 |
| `explorer` | 読み取り中心のコード探索 |

依頼例:

```
この PR を main と比較してレビューして。以下の観点ごとに 1 つずつ agent を spawn し、全員の結果を待って重大度順に統合して。
1. セキュリティ
2. バグ
3. テスト不足
4. 保守性
5. パフォーマンス
```

設計相談の依頼例:

```
認証処理のリファクタ案を比較したい。explorer agent に現行設計の調査、別の explorer agent にテスト構造の調査を任せ、親 agent は結果を統合して実装計画を出して。
```

Subagents は sandbox policy を継承します。interactive CLI では inactive thread から approval request が出る場合もあり、overlay に source thread label が表示されます。

## Codex app-server: rich client を支えるプロトコル

Codex の仕組みを理解するうえで、app-server は重要です。公式 docs では、`codex app-server` は VS Code extension のような rich client を支える interface と説明されています。

app-server は JSON-RPC 2.0 ベースの双方向プロトコルを提供します。transport は default の stdio に加えて、experimental な websocket もあります。

中心概念は次の 3 つです。

| primitive | 意味 |
| --- | --- |
| Thread | Codex agent との会話 |
| Turn | 1 回の user request と、それに続く agent work |
| Item | user message、agent message、command run、file change、tool call など |

代表的な API:

* `thread/start`
* `thread/resume`
* `thread/fork`
* `turn/start`
* `turn/steer`
* `turn/interrupt`
* `command/exec`
* `fs/readFile`
* `fs/writeFile`
* `mcpServer/tool/call`
* `config/read`

つまり、Codex の UI は単なる chat box ではありません。thread / turn / item / approval / tool call / file change を event stream として扱う agent runtime です。

参照:

Codex は MCP client として外部 tool を呼べるだけでなく、`codex mcp-server` により Codex 自身を MCP server として起動できます。

GitHub の `codex_mcp_interface.md` では、これは experimental interface であり、MCP transport over stdio で local Codex engine を制御する JSON-RPC API だと説明されています。

扱う対象:

* threads
* turns
* accounts
* config
* approvals

代表的な method:

* `thread/start`
* `thread/resume`
* `thread/fork`
* `thread/read`
* `thread/list`
* `turn/start`
* `turn/steer`
* `turn/interrupt`
* `account/read`
* `config/read`
* `model/list`

approval も protocol に含まれます。`applyPatchApproval` や `execCommandApproval` を client に要求する設計です。

これは、Codex を別の agent や orchestration system の中に組み込むための入口です。ただし experimental なので、長期互換性を前提にしすぎないほうが安全です。

* 目的
* 変更してよい範囲
* 実行してほしい検証
* 触ってほしくないもの
* 出力形式
* 不明点がある場合の扱い

## チーム運用のおすすめ構成

個人利用なら `~/.codex/config.toml` と `~/.codex/AGENTS.md` だけでも十分です。チームで使うなら、次の構成が現実的です。

```
repo/
  AGENTS.md
  .codex/
    config.toml
  .agents/
    skills/
      pr-review/
        SKILL.md
      release-note/
        SKILL.md
```

root の `AGENTS.md` には repository 全体のルールを書きます。module ごとのルールがある場合は、対象 directory に `AGENTS.md` または `AGENTS.override.md` を置きます。繰り返し行う作業は `.agents/skills` に切り出します。

おすすめの導入順:

1. まず個人が CLI で read-only 調査に使う
2. repo root に最小の `AGENTS.md` を置く
3. PR 前に `codex review` を使う
4. 繰り返しレビュー観点を skill 化する
5. MCP で GitHub や docs 検索をつなぐ
6. 安定したら `codex exec` を CI や定期ジョブに組み込む
7. 長いタスクだけ Codex cloud に委譲する

## よくある落とし穴

### 1. 権限を広げすぎる

最初から full access にしないほうが良いです。多くの作業は `workspace-write + on-request` で足ります。調査だけなら read-only で十分です。

### 2. internet access と web search を混同する

web search を許可しても、shell command の network access を許可したことにはなりません。逆に network access を許可すると、依存取得や外部送信のリスクが増えます。

### 3. `AGENTS.md` に古いコマンドを書く

存在しない test command や古い setup 手順を書くと、Codex はそれを信じます。`AGENTS.md` は短くてもいいので、実在するコマンドだけを書きます。

### 4. 大きすぎる依頼を 1 agent に丸投げする

大きな task は Plan mode で分解し、必要なら subagents に調査を分担させたほうが良いです。

### 5. review を実装後の飾りにする

Codex のレビューは、PR 直前だけでなく、設計時、実装中、差分作成後に使うと効果が出ます。特に `codex review --uncommitted` は習慣化しやすいです。

### 6. 公式 docs と CLI version の差分を見ない

Codex は更新が速いです。記事や古いブログより、まず手元の version と help を確認します。

```
codex --version
codex --help
codex exec --help
codex cloud --help
codex mcp --help
```

## 安全に使うためのチェックリスト

* 変更前に `git status` を確認する
* 初見リポジトリでは read-only から始める
* 編集を許可する場合は `workspace-write` を基本にする
* full access は隔離環境だけで使う
* network access は必要な domain と method に絞る
* `AGENTS.md` に secret を書かない
* MCP server の tool は allow / deny list を検討する
* destructive tool は approval を必須にする
* Codex が実行した command と diff を必ず確認する
* 生成コードはテストとレビューを通す

## まとめ

Codex は、単なるコード生成 UI ではありません。ローカルでのペア作業、クラウドへの委譲、非対話自動化、外部 tool 連携、subagent による分業まで含むコーディングエージェント基盤です。

特に重要なのは次の 7 点です。

1. 日常作業は CLI / IDE、長い作業は Codex cloud に分ける
2. approval と sandbox を理解して、権限を段階的に開く
3. 複雑な変更は `/plan` から始める
4. `AGENTS.md` で repository 固有ルールを永続化する
5. Skills で繰り返し作業を workflow 化する
6. MCP で外部ツールと文脈をつなぐ
7. `codex exec` と app-server / MCP server により、自動化や組み込みにも広げられる

Codex の価値は「AI がコードを書いてくれる」だけではありません。人間が普段やっている調査、方針決定、実装、検証、レビュー、引き継ぎの流れを、権限管理つきで一つの作業体験にまとめられる点にあります。
