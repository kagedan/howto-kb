API開発のたびに「OpenAPI（Swagger）の定義書、また手で書かなきゃ……」とため息をついていませんか？エンドポイントが増えるほどyamlの記述量も膨らみ、コードとの整合性を保つのも一苦労です。この記事では、Claude Codeを使ってOpenAPI定義を効率的に作る方法を、具体的なプロンプト例と検証手順つきで解説します。読み終える頃には、手書きの何倍ものスピードでドキュメントを整備できるようになるはずです。

## 結論：Claude Codeは「既存コードからの逆生成」と「ゼロからの設計」どちらにも対応できます

結論から言うと、Claude CodeはOpenAPI定義の作成において強力な相棒になります。すでにあるコントローラーやルーティングのコードを読み込ませて仕様書を逆生成することもできますし、逆にAPIをまだ作っていない段階で「こんなAPIを作りたい」と会話しながらゼロからyamlを設計することも可能です。手書きで悩んでいませんか？その悩み、プロンプト一つで解決できる場合が多いです。

### なぜClaude CodeがOpenAPI作成に向いているのか

理由は大きく2つあります。1つは、Claude Codeがプロジェクト全体のコードを読み取れるため、コントローラーのメソッド名・引数・戻り値の型からエンドポイントの仕様を推測できる点です。もう1つは、CLAUDE.mdに出力ルール（バージョン、命名規則、タグ付け方針など）を書いておけば、毎回同じフォーマットでyamlを生成してくれる点です。私はこの2つを組み合わせることで、仕様書のばらつきがかなり減ったと感じました。

## Claude CodeでOpenAPI（Swagger）を作る3つのアプローチ

実際にOpenAPIを作る場面は、プロジェクトの状況によって変わります。代表的な3パターンを表にまとめました。

| アプローチ | どんなときに使うか | Claude Codeへの指示の方向性 |
|---|---|---|
| ①既存コードから逆生成 | すでにAPIが動いていてドキュメントだけ足りない | 「このディレクトリのコントローラーを読んでOpenAPI 3.1のyamlを生成して」 |
| ②会話しながらゼロから設計 | これからAPIを実装する／仕様を固めたい | 「ユーザー管理APIを作りたい。エンドポイント一覧から一緒に考えて」 |
| ③スキーマファースト | チームでAPI仕様を先に決めて実装はあとで | 「このyamlに沿って、まずスキーマだけ厳密に固めて」 |

❌ いきなり「OpenAPI作って」とだけ指示する
✅ 「対象ディレクトリ」「OpenAPIのバージョン」「認証方式（Bearerなど）」を最初に伝える
◎ さらに「レスポンス例（example）も含めて」と明示する

この3行だけでも、生成されるyamlの精度はかなり変わってきます。

## 手順解説：Claude CodeでOpenAPI定義を作る実践フロー

### 1. プロジェクトのコンテキストを読み込ませる

最初に「このリポジトリのAPIルーティング部分を把握して」と伝え、エンドポイントの一覧をざっくり認識させます。Express・FastAPI・Rails・Laravelなど、フレームワークが何であっても、ルーティング定義やコントローラーのファイルを起点に読み込ませるのがコツです。

### 2. CLAUDE.mdに出力ルールを書いておく

プロジェクトルートのCLAUDE.mdに、以下のようなルールを書いておくと、毎回似たフォーマットでOpenAPIを出力してくれるようになります。
### 3. 生成を依頼する

「`src/api/`配下のコントローラーを読んで、openapi.yamlを生成してください。CLAUDE.mdのルールに従ってください」というように、対象範囲とルール参照を明示して依頼します。

### 4. swagger-cliやRedocly CLIで検証させる

生成されたyamlは、必ず構文チェックをかけましょう。Claude Codeに「`npx @redocly/cli lint openapi.yaml`を実行して、エラーがあれば修正して」と続けて依頼すれば、検証〜修正までを一気にやってくれます。

### 5. 必要であればHookで自動Lintをかける

繰り返しOpenAPIを更新するプロジェクトなら、ファイル編集後に自動でLintを走らせるHookを組んでおくのもおすすめです。手動チェックを忘れるリスクを減らせます。

## さらに深掘り：精度を上げるプロンプト設計とよくある失敗

ここまでの手順を踏んでも、最初から完璧なOpenAPIが出てくるとは限りません。私が実際に試して気づいた、精度を上げるためのポイントを深掘りします。

### よくある失敗パターン

| 失敗パターン | 原因 | 対策 |
|---|---|---|
| エンドポイントの抜け漏れ | 対象ディレクトリの指定が曖昧 | 「このフォルダ配下を全部見て」と範囲を明確にする |
| レスポンス例が空 | exampleを書く指示をしていない | CLAUDE.mdやプロンプトで明示的に依頼する |
| enumの粒度が粗い | コード側にコメントや型情報が乏しい | コード側に型定義やコメントを補強してから依頼する |
| 認証方式の記述ミス | securitySchemesの理解が浅いまま生成 | 「Bearer認証であることを明記して」と一文加える |

### 一気に作らず、エンドポイント単位で確認する

私は最初、API全体を一度にOpenAPI化しようとして、結局見直しに時間がかかってしまいました。今は「まずusers関連だけ生成→確認→ordersを追加」というように、リソース単位で少しずつ進める方法に変えています。この方が結果的に手戻りが少なく、納得感のある仕様書に仕上がると感じています。

### 既存コードとの整合性チェックも忘れずに

OpenAPI定義とコードの実装がズレていないか、Claude Codeに「このyamlと実装コードを比較して、矛盾点があれば教えて」と聞くのも有効です。ドキュメントだけが古くなる、というQiitaでもよく見かける悩みをここで予防できます。

## まとめ

- ✅ Claude CodeはOpenAPIの「逆生成」「ゼロから設計」「スキーマファースト」すべてに対応できる
- ✅ CLAUDE.mdに出力ルールを書いておくと、生成のブレが減る
- ◎ 生成後は必ずswagger-cliやRedocly CLIで構文検証をかける
- ✅ 一気に全体を作らず、リソース単位で確認しながら進めるのが安全
- ◎ コードとOpenAPIの整合性チェックも、Claude Codeに任せられる
