## はじめに

こんにちは、jinjerで SRE をしている与儀さねです。  
jinjer沖縄オフィスで活動しています。

jinjerでは HR 領域の SaaS を複数プロダクト展開しており、各プロダクトのアプリケーションは New Relic APM で監視しています。これまで SLO（Service Level Objective）は New Relic の UI 画面から設定していましたが、運用のスケールや変更管理の観点から、より効率的かつ確実な管理手法への移行を模索していました。

そこで今回、**Claude Code + New Relic MCP Server** を組み合わせて、以下の 2 つのフェーズを AI アシスト付きで進める仕組みを作ってみました。本記事ではその流れと、AI 活用で得られた実感値を共有します。

!

1. **CUJ の特定** ―― アプリケーションのソースコードと New Relic の実トラフィックを統合的に判断し、計測すべき CUJ（Critical User Journey、重要なユーザー動線）と対象トランザクションを特定する
2. **SLO 設定の Terraform 化** ―― 特定した CUJ の SLO 定義を Claude Code で Terraform 化し、PR レビュー可能な状態にする

**SLO設定の方法にはさまざまなアプローチがあるかと思いますが、本記事の内容はあくまでその一例として捉えていただけますと幸いです。**

## なぜ SLO設定を Terraform 化したか

SLO は「定義して終わり」では機能せず、運用で育てるものです。そのためには以下が必要でした。

* **PR レビュー可能にする**：閾値変更・追加を Git 上で議論できるように
* **根拠を残す**：例えば「P95 が 5.63 秒だから閾値を 6 秒にした」といった意思決定を記録
* **横展開を容易にする**：あるプロダクトで作ったパターンをほかのプロダクトへも横展開しやすく

UI画面を介した手動設定のみでは、設定のコード化による議論や、多プロダクトへの迅速な横展開を担保することが難しいという側面がありました。また、SREチームではAWSのインフラ管理にもTerraformを利用しています。

## フェーズ1：ソースコード × New Relic MCP Server で CUJ を特定する

SLO を作る前段として、まず「何を CUJ として測るか」「どのトランザクションを対象にするか」を決める必要があります。ここで効いたのが、**Claude Code にアプリケーションのソースコードを読ませつつ、New Relic MCP Server で実トラフィックも参照させて、両者を統合的に判断させる**というアプローチでした。

### コード視点：ソースコードから CUJ 候補を洗い出す

対象のプロダクトは中〜大規模のソースコードを持つアプリケーションで、コントローラだけでも数百ファイル規模にのぼります。これらをすべて目視で確認し、重要な機能を網羅的に抽出するには多大な工数がかかり、見落としのリスクも排除できませんでした。そこで Claude Code に以下のような流れで依頼しました。

1. ルーティング定義を読んで、外部公開されているエンドポイント一覧を把握してもらう
2. コントローラのアクションとあわせて、業務上の意味（だれがいつ何のために使うか）を推定してもらう
3. 権限（一般利用者 / 管理者 など）と業務フェーズ（入力・申請・承認・出力 など）でグルーピング
4. 「業務影響度」などで CUJ 候補をランク付け

ここで重要だったのは、**Claude がコードを読みながら「業務文脈」に翻訳してくれる**ことでした。たとえば「この配下のコントローラは月次の締め処理に使われ、失敗すると後続業務に連鎖影響する」といった**業務影響の言語化**まで一気に出してくれます。これは grep や IDE では得られない価値です。

### トラフィック視点：New Relic MCP で実データと突きあわせる

ソースコードから引いた候補は、あくまで「コード上の重要度」です。実際にどれだけ使われているか・どのトランザクションが対象になるかは、本番の実トラフィックを見ないとわかりません。そこで活躍したのが New Relic MCP Server でした。

MCP（Model Context Protocol）は Anthropic が公開している「AI と外部ツールをつなぐ共通プロトコル」で、Claude Code から MCP Server を呼び出すことで、**Claude がそのまま New Relic にクエリを投げられる**ようになります。今回特によく使った New Relic MCP Server のツールがこちらです。

| MCP ツール | 用途 |
| --- | --- |
| `execute_nrql_query` | NRQL を直接実行して結果取得 |
| `natural_language_to_nrql_query` | 自然言語から NRQL を生成 |
| `analyze_transactions` | トランザクションの件数・P95 などを算出 |
| `list_entity_error_groups` | エラー傾向の確認 |
| `get_entity` | APM アプリの GUID 解決 |

これにより、Claude Code がコードから挙げた候補を、その場で実トラフィックと突きあわせて判断できます。たとえば `analyze_transactions` で**トランザクション件数の多い処理 トップ**を取らせたところ、コード上は地味だが実は超高頻度な API（たとえばログイン）が見つかり、当初の候補に追加できました。

**コード上の重要度（コード視点）× 実際の利用量（トラフィック視点）** を 1 つの対話の中で統合的に判断できたことで、CUJ と対象トランザクションを精度よく特定できました。これがフェーズ1 の成果です。

## フェーズ2：特定した CUJ の SLO を Terraform 化する

CUJ と対象トランザクションが定まれば、あとは SLO（目標値・計算期間・SLI の NRQL）に落とし込むだけです。この Terraform 化も Claude Code に任せました。

具体例として、対象トランザクションの SLI 閾値を決めるために、Claude Code に MCP 経由で P95 を取得させ、その実測値を見ながら閾値を決定 ➔ そのまま Terraform リソースに反映、という流れです。

!

**Before（手動による設定）**

1. New Relic UI で NRQL を書く
2. トランザクション名・属性名を目視確認
3. P95 を取って閾値を決める
4. Terraform に書き写す

1 つの SLO 定義をテンプレートとして示せば、残りの定型部分は Claude Code が機械的に生成してくれるため、人間はレビューに集中できます。ブラウザとエディタを往復しなくていい、というのが体験として一番大きな変化でした。

## AI 活用で得られた具体的なメリット

実際に進めてみて、定性的ではありますが以下の 3 点が大きかったです。

### 1. CUJ 特定がコードとデータの両面で精度よくできる

「何を測るか」の叩き台を、**ソースコード解析で半自動的に出しつつ、その場で実トラフィックと突きあわせられる**ようになりました。コード視点だけでもトラフィック視点だけでも漏れが出ますが、Claude Code が両者を 1 つの対話で統合してくれるため、SRE 単独でも精度の高い一次設計まで踏み込めます。

### 2. 検証〜量産のサイクルがバクソクになる

NRQL を書いた瞬間に MCP 経由で実行できるので、トランザクション名や属性名の指定ミスがあっても**件数 0 件という形で即座にわかります**。Terraform の定型生成も含め、「書く➔投げる➔確認」がバクソクで回ります。

### 3. 意思決定の根拠をそのまま description に残せる

Claude Code は P95 の実測値を一緒に出してくれるので、`description` に「ある参照系 API の P95 が 5.63 秒のため閾値を 6 秒に設定」という形で**定量的な根拠**を残せます。半年後の自分が「なぜ 6 秒なんだっけ？」とならない SLO になりました。

## まとめ

* **CUJ の特定は、ソースコード（コード視点）と New Relic MCP の実トラフィック（トラフィック視点）を Claude Code に統合的に判断させる**のが効果的だった
* 特定した CUJ の **SLO 定義は Claude Code でそのまま Terraform 化**でき、実データを確認しながら閾値を決められる
* SLO を Terraform でコード管理すると、PR レビュー・横展開・根拠の記録が段違いに楽になる

今回、SLO設定におけるAIの活用方法をご紹介しました。想像以上に簡単に設定が進められたことに私自身驚きが大きかったです。しかし、この設定した内容を活かすも殺すのも、このあとの実運用の進め方次第です。運用の中でも大いにAI活用ができると考えています。同じような悩みを持つ SRE の方の参考になれば幸いです。
