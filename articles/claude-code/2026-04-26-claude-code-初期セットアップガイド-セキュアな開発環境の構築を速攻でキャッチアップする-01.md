## 背景と目的
Claude Codeが出現し1年、エージェンティックコーディングが世の中で浸透する中、転職しエンジニアとしてキャリアをスタートする中でカスタマイズエージェントAIに触れられず（AWS Kiroは触っていたがClaudeCodeほど拡張性はない）、非常に強い危機感を感じており、早急にキャッチアップしたく実施しました。また自身への定着のためアウトプットまで実施したく計画。Claudeの公式・コミュニティソースを参照しながら初期設定を構築

Claude Codeはターミナルで動作するAIコーディングエージェントで、ファイルの読み書き・コマンド実行・Webアクセスまで自律的に行う。そのため**何も設定しないまま使い始めると、ローカルマシン全体へのアクセスを許可した状態で動作する**ことになる。

本記事では以下の課題を解決する初期設定を解説する。

- `.env`やSSH鍵などの機密情報へのアクセスを機械的にブロックする
- Claudeが実行できるコマンドの範囲を制限する
- プロジェクト作業をDockerコンテナ内に隔離し、ホストのファイルシステムを守る
- Claude Opus 4.7の設計思想に沿ったワークフローを設定する

## セキュリティ設計：3層の防御

### 層1：`settings.json` のdeny（機械的ブロック）

Claude Codeの権限設定。**CLAUDE.mdの指示より強制力が高く**、Claudeが逆らえない物理的な制限。

Read/Edit/Writeの3ツールすべてに対して`.env`ファイルへのアクセスをブロックする点が重要。Bashコマンド経由だけでなく、ReadツールやEditツール経由のアクセスも防ぐ。

```json
{
  "permissions": {
    "deny": [
      "Read(.env)", "Read(.env.*)", "Read(**/.env)", "Read(**/.env.*)",
      "Edit(.env*)", "Write(.env*)",
      "Read(~/.ssh/*)", "Read(~/.aws/*)",
      "Bash(rm -rf /)", "Bash(rm -rf ~)",
      "Bash(curl*|*bash*)", "Bash(wget*|*sh*)",
      "Bash(--dangerously-skip-permissions*)",
      "Bash(printenv)", "Bash(env)"
    ]
  }
}
```

### 層2：`security-precheck.sh`（PreToolUse hook）

`settings.json`のdenyを**さらに強化する二重チェック**。Bashコマンドが実行される直前に起動し、パターンマッチングでブロックする。

ブロック対象：
- pipe-to-shell（`curl | bash`・`wget | sh`）
- 認証情報ディレクトリへのアクセス（`~/.ssh/`・`~/.aws/`）
- 環境変数の外部送信（curl + API_KEY など）
- world-writable chmod（`chmod 777`・`chmod a+rwx`）

```json
"hooks": {
  "PreToolUse": [{
    "matcher": "Bash",
    "hooks": [{"type": "command", "command": "bash ~/.claude/hooks/security-precheck.sh"}]
  }]
}
```

### 層3：devcontainer（OS レベルの隔離）

最も根本的な防御。プロジェクト作業をDockerコンテナ内に閉じ込めることで、**Claudeからホストのファイルシステム全体へのアクセスを物理的に遮断**する。

```
ホストMac（安全）
├── ~/Documents/    ← コンテナから見えない
├── ~/.ssh/         ← コンテナから見えない
└── ~/myproject/    ↔ /workspace/ としてマウント（ここだけ）
```

加えて`init-firewall.sh`でネットワークをホワイトリスト制限：
- `api.anthropic.com` ✅
- `github.com` ✅
- `registry.npmjs.org` ✅
- それ以外 ❌ DROP

## CLAUDE.md の設計方針

### スリム化と動的ロード

CLAUDE.mdはClaudeが毎回読み込む指示書だが、**200行を超えると指示が無視され始める**ことが知られている。そのため詳細ルールを`~/.claude/rules/common/`配下に分離し、CLAUDE.mdは参照のみに留める設計を採用した。

```markdown
## Rules
Detailed rules are dynamically loaded from ~/.claude/rules/common/:
- security.md — シークレット・危険コマンド・AgentShield
- file-management.md — ~/.claude/はユーザー管理領域
- workflow.md — モデル選択・SubAgent・コーディング原則
```

### `~/.claude/`はユーザーが管理する領域

Claudeが勝手に`~/.claude/`配下を変更しないよう明示的に禁止している。プロジェクト固有のファイル（plan・memo・スキル）はプロジェクトディレクトリ内に作成させる。

## ワークフロー設計：Claude Opus 4.7 準拠

### モデルの使い分け

| タスク | モデル |
|---|---|
| アーキテクチャ設計・実装計画・深い推論 | **Opus** |
| 通常のコーディング・レビュー・定型作業 | **Sonnet**（デフォルト） |

```bash
/model opus    # 複雑なタスク
/model sonnet  # 通常作業
/effort xhigh  # Opus 4.7推奨デフォルト（maxではない）
```

### SubAgent の委譲判断

4.6まではSubAgentを積極的に呼び出すことが推奨されていたが、**4.7では逆効果になる**ことが公式から明示された。

> "More judicious about when to delegate work to subagents."（Anthropic公式）

4.7では以下の場合のみ明示的に委譲する：
- 複数ファイルへの**並列作業**が必要な場合
- **独立した複数タスク**を同時に処理する場合

それ以外はClaude自身が自己完結する。グローバルエージェント（planner・code-reviewer・build-error-resolver）は定義しているが、毎回呼び出す運用はしない。

### コーディング原則 4つ

`rules/common/workflow.md`に以下を定義している。

**1. Think Before Coding**：曖昧な指示は実装前に確認。解釈が複数あれば選択肢を提示。

**2. Simplicity First**：要求されていない抽象化・パターンを導入しない。200行→50行にできるなら書き直す。

**3. Surgical Changes**：依頼箇所だけを正確に修正。動作している箇所をリファクタしない。

**4. Goal-Driven Execution**：タスクを「検証可能なゴール」に変換してから実装。テストを先に定義してClaude自身が自動検証できる構造にする。

### 推奨プロンプト形式（4.7準拠）

```
Goal: <達成したいこと>
Constraints: <守るべき制約>
Acceptance criteria: <完了の判定基準>
```

初回プロンプトに全要件を記載し、途中介入を減らすことで4.7の自律実行能力が最大化される。

## カスタムステータスライン

`statusline.sh`でターミナル下部にリアルタイム情報を表示する。

```
Sonnet 4.6  |  myproject  |  🌿 main  |  💰 0.0042  +156 -23
 current  ● ● ● ○ ○ ○ ○ ○ ○ ○  30%
  weekly  ● ● ● ● ○ ○ ○ ○ ○ ○  38%  resets 14:00
   extra  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○  0%
```

- **ライン1**：モデル・ディレクトリ・Gitブランチ・セッションコスト・編集行数
- **ライン2〜4**：コンテキスト使用率・5時間レートリミット・7日レートリミット（ドットバー、使用率で色変化）

`settings.json`に以下を追記するだけで有効化される：

```json
"statusLine": {
  "type": "command",
  "command": "bash ~/.claude/statusline.sh",
  "padding": 0
}
```

## `claudecontainer`コマンド

VS Codeなしでdevcontainerを起動してClaude Codeを実行するカスタムコマンド。

```bash
# 新規プロジェクト
cd ~/myproject
claudecontainer init   # .devcontainerを自動生成
claudecontainer up     # コンテナ起動 → Claude Code起動

# その他
claudecontainer status # 状態確認
claudecontainer stop   # コンテナ停止
```

`claudecontainer init`は以下を自動で行う：
1. AnthropicのGitHubから公式テンプレートをダウンロード
2. `init-firewall.sh`を冪等化（繰り返し実行可能に）
3. `devcontainer.json`に`--privileged`と`containerEnv`を追加
4. コンテナ起動時にホストの`~/.claude/`設定を自動同期

### サブスクリプション認証について

APIキー（`ANTHROPIC_API_KEY`）が未設定の場合、Pro/Maxサブスクリプションのブラウザ認証が使われる。初回コンテナ起動時にブラウザ認証が必要だが、以降はボリュームにトークンが保存され再認証不要。

## インストール手順

### 前提条件

- macOS（Apple Silicon）
- Claude Code v2.1.111以上
- Docker Desktop インストール済み・起動済み
- `devcontainer CLI`（`npm install -g @devcontainers/cli`）

### セットアップフォルダ構成

```
setup/
├── install.sh
├── CLAUDE.md
├── settings.json
├── statusline.sh
├── security-precheck.sh
├── session-monitor.sh
├── claudecontainer
├── rules/common/
│   ├── security.md
│   ├── file-management.md
│   └── workflow.md
├── agents/
│   ├── planner.md
│   ├── code-reviewer.md
│   └── build-error-resolver.md
└── session-copy/
    ├── SKILL.md
    └── session-copy.sh
```

### 実行

```bash
# 1. ~/.claudeを完全削除（認証情報は~/.claude.jsonに別管理されているので安全）
rm -rf ~/.claude/

# 2. install.shを実行
cd ~/Downloads/setup
bash install.sh

# 3. PATHを反映
source ~/.zshrc

# 4. 動作確認
claude --version
claudecontainer --help
```

### プロジェクトごとの初期化

```bash
cd ~/myproject

# .devcontainerを生成してコンテナを起動
claudecontainer init
claudecontainer up

# プロジェクトルートに.claudeignoreを追加
echo ".env\n.env.*\n*.pem\n*.key\nsecrets/" > .claudeignore
```

## セッション管理

### セッションコピー

調査用ディレクトリで得た知見を別のプロジェクトに引き継げるカスタムスキル。

```bash
/session-copy /path/to/source-project
```

セッション履歴は`~/.claude/projects/<encoded-path>/<uuid>.jsonl`に保存される。コピー後は`/resume`または`claude --resume <uuid>`で再開。

### セッション履歴の自動削除防止

```json
"history": {
  "clean_period_days": 99999
}
```

デフォルトの30〜60日自動削除を無効化。

## まとめ

本設定で実現している防御レイヤーを整理する。

| レイヤー | 手段 | 防御対象 |
|---|---|---|
| ファイルアクセス制限 | `settings.json` deny | `.env`・SSH鍵・認証情報への直接アクセス |
| コマンド実行制限 | `security-precheck.sh` | pipe-to-shell・危険なrm・権限バイパス |
| ネットワーク制限 | `init-firewall.sh` | ホワイトリスト外への通信 |
| OS分離 | devcontainer | ホストのファイルシステム全体へのアクセス |
| 指示レベル制限 | `CLAUDE.md`・`rules/` | Claudeの行動指針・ファイル管理ルール |

完全な防御は存在しないが、**devcontainerを使用することで被害をワークスペース内に限定できる**点が最も重要な設計判断である。センシティブなプロジェクトでは必ず`claudecontainer up`で作業することを推奨する。
