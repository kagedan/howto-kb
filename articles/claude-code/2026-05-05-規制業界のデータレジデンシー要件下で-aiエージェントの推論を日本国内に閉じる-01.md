## サマリ

* 2025/10 に登場した **Amazon Bedrock JP Geo Cross-Region Inference（JP CRIS）** は、AWS 公式が「financial institutions, healthcare providers, and government agencies」を想定用途に挙げた、規制業界向けのデータ境界機能である
* JP CRIS 対応モデルは **Opus 4.7 / Sonnet 4.6 / Sonnet 4.5** の 3 系統（執筆時点の AWS モデルカードで `jp.anthropic.*` プロファイル ID を確認できるもの）。ルーティングは **ap-northeast-1 (Tokyo) ⇔ ap-northeast-3 (Osaka) の 2 リージョン内で完結**し、AWS Global Network 上で日本国外に出ない。Haiku 4.5 は 2025/10 のアナウンス時点では Sonnet 4.5 と並ぶ JP CRIS 対象モデルとして紹介されたが、執筆時点（2026/5）の AWS モデルカードには JP Geo プロファイルが掲載されていないため、本記事では扱いを Tokyo In-Region に寄せている
* **Claude Code on AWS** は inference profile ID / ARN を `ANTHROPIC_MODEL` や `modelOverrides` で指定でき、JP CRIS をファーストパーティでサポートする
* 一方、**Kiro CLI には現時点で Bedrock inference profile を明示指定する公開手段が提供されていない**（モデル切替は `/model` コマンドによる alias ベース、2026/5 時点）。結果として、規制業界のデータレジデンシー要件下では **Kiro CLI / Kiro IDE は構造的に選択肢から外れ、Claude Code on AWS が唯一の現実解** となる
* 運用上の落とし穴は 2 つ。(1) Claude Code のデフォルト primary model は `us.*` プレフィックスで US にルーティングされる、(2) ap-northeast-1 は Global CRIS のソースリージョンにも含まれる。SCP で `us.*` / `global.*` を明示ブロックする境界強制が必須

## はじめに — 規制業界で AI コーディングを日本国内に閉じる

シリーズ 6 本目「Eclipse 利用企業の Q Developer EOS 対応」で、Eclipse を全社標準とする組織にとっては「Kiro 一択ではなく、Claude Code on AWS も対等な選択肢」と論じた。マトリクスとして 4 シナリオを提示し、機能面では両者がほぼ対等であることを示した。

しかし、**規制業界（金融・公共・医療）のデータレジデンシー要件** というフィルターをかけると、この対等性は崩れる。

結論を先に置くと、以下のようになる。

* **規制業界で AI コーディングツールを導入する場合、Kiro CLI / Kiro IDE は構造的に選択肢から外れる**
* 理由: Kiro CLI は Bedrock の推論プロファイル（inference profile）指定をサポートしない
* 一方、**Claude Code on AWS は JP CRIS（Bedrock Japan Geo Cross-Region Inference）をファーストパーティでサポート**し、プロンプトと応答を ap-northeast-1 (Tokyo) と ap-northeast-3 (Osaka) の 2 リージョン内に閉じ込められる
* これにより、**規制業界でも最新モデル（Opus 4.7）を日本国内に閉じて使うことができる**

!

**本記事のスコープ**

本記事は、金融・公共・医療の規制業界の IT 部門、CCoE（Cloud Center of Excellence）、セキュリティ・コンプラ担当を想定読者とする。AI コーディングツールをデータレジデンシー要件下で導入する際に、JP CRIS を前提とした Claude Code on AWS の設計パターンを体系化する。

法解釈や AWS 契約の具体条件には踏み込まず、AWS Support（Business プラン以上）・AWS Partner・AWS アカウントチーム（SA / TAM のアサインがある場合）・AWS re:Post のいずれかの経路で AWS に照会すべき論点として提示する。SA のアサインがない組織でも、Support ケースや Partner 経由で同等の相談が可能な前提で書く。

本記事は以下の流れで進める。

1. 規制要件の再整理（FISC 第 11 版 / ISMAP / 3 省 2 ガイドライン / 個情法 28 条）
2. JP CRIS の技術解説（対応モデル、ルーティング、Geo 種別の違い）
3. Claude Code on AWS での実装パターン（環境変数 / IAM / SCP / modelOverrides / Guardrails / 監査証跡）
4. 運用上の落とし穴 2 点
5. Opus 4.7 の 2 つのモード（Tokyo In-Region vs JP CRIS）
6. Kiro CLI が規制業界で選べない理由
7. 3 業界ケーススタディ（金融 / 公共 / 医療）
8. 監査証跡の設計、運用上の注意点

最後にシリーズ 6 本との接続を整理し、CCoE 運用モデルとしてツール非依存の抽象層と、ツール固有の実装層をどう切り分けるかを提示する。

## 1. データレジデンシー要件の再整理

AI コーディングツールは、ソースコード・プロンプト・応答をモデルに送り、処理する。このデータ処理の所在地が、各業界の規制要件と交差する。

ここでは 4 つの規制枠組みを簡潔に整理する。細部の法解釈や自社の適用範囲は、必ず AWS Support・AWS Partner・法務部門・監査法人に確認してほしい。本節は「AI 推論処理がどの規制対象になりうるか」の見取り図を示すにとどめる。

### 1.1 FISC 安全対策基準 第 11 版（金融）

FISC（金融情報システムセンター）の「金融機関等コンピュータシステムの安全対策基準・解説書」第 11 版は、金融機関のクラウド利用におけるデータ保管・処理の統制を求める実質的な業界標準である。

AI コーディングツール導入の観点で関連する論点:

* クラウド事業者のデータセンター所在地の把握と管理
* 業務データ（顧客情報を含むソースコード、データベーススキーマ、業務ロジック等）の国外流出防止
* 委託先管理としての AI ベンダー / クラウド事業者の評価
* 障害発生時・規制当局調査時の証跡確保

推論プロファイルを国内に固定する構成は、「データ処理を国内に閉じる」という統制要件を満たす具体的な実装手段になる。ただし、FISC 第 11 版の適用範囲は組織ごと・業務ごとに異なるため、自社の内規と監査法人の判断を優先する。

### 1.2 ISMAP（公共）

ISMAP（政府情報システムのためのセキュリティ評価制度）は、政府機関・独立行政法人・地方公共団体が利用するクラウドサービスの安全性を評価・登録する制度である。

AI コーディングツール導入の観点で関連する論点:

* ISMAP 登録クラウドサービスの利用原則（「ISMAP 管理基準」の遵守）
* 機密区分別のリージョン・データ所在地要件
* 調達仕様書でのデータ所在地指定の慣行
* ISMAP-LIU（Low-Impact Use）での簡易利用との区別

Amazon Bedrock は ISMAP 登録サービスに含まれている。JP CRIS を使って ap-northeast-1 / ap-northeast-3 に閉じる構成は、「国内リージョン完結」という調達仕様書の要求と整合しやすい。ISMAP 上で特定の機能（JP CRIS）がどう評価されているかは、AWS Support や公共担当 Partner の最新情報を確認する。

### 1.3 3 省 2 ガイドライン（医療）

「3 省 2 ガイドライン」は、医療情報の取扱いに関する以下の 2 つのガイドラインの総称である。

* 厚生労働省「医療情報システムの安全管理に関するガイドライン」
* 経済産業省・総務省「医療情報を取り扱う情報システム・サービスの提供事業者における安全管理ガイドライン」

医療機関・電子カルテベンダ・医療 SaaS 事業者がクラウドを利用する際の統制を定めている。AI コーディングツール導入の観点で関連する論点:

* 医療情報（PHI: Protected Health Information）の外部保管要件
* 医療情報を含むコード・テストデータのクラウド送信時の暗号化・匿名化
* 医療情報処理者としての監督義務
* 2024/4 改定 Q&A での「第三者提供」該当性の明確化

AI コーディング作業中のソースコードに PHI が含まれる可能性は、業務によって異なる。電子カルテ本体の開発では PHI を含むテストデータや参照データを扱う場面が想定され、それを AI モデルに送る行為がガイドラインの「外部保管」「第三者提供」に該当するかは組織ごとに判断が必要になる。JP CRIS で国内完結させた上で、PHI を含まないダミーデータで開発する運用設計が現実解になる。

### 1.4 個情法 28 条と AI 推論の位置づけ

個人情報保護法（以下、個情法）第 28 条は、個人データの**外国にある第三者への提供**を原則として本人同意を要する旨を定める。

AI コーディングツールの文脈で関連する論点:

* プロンプトに個人情報（開発者自身のメタ情報、テストデータ中の個人情報等）が含まれる場合の取扱い
* クラウド事業者が「第三者」に該当するかどうか（委託先として整理できるか）
* リージョン外（特に国外）へのルーティングが「外国にある第三者への提供」に該当するか
* 個人情報保護委員会のガイドライン・ガイドライン Q&A での解釈

「AWS Japan の管理下にあるリージョン内で処理が完結すれば、個情法 28 条の越境提供には該当しない」という整理が実務上有力視される場合がある。一方、AWS 自体が米国法人であることをどう評価するかは別論点として残る。いずれも最終的には法務部門と個情法・委託ガイドラインに基づく判断が必要である。本記事は、判断の前提となる\*\*「データ処理を国内に閉じる技術手段」\*\*を提供する立場を取る。

## 2. JP CRIS とは何か

JP CRIS（Japan Geo Cross-Region Inference）は、Amazon Bedrock のクロスリージョン推論プロファイルの一種で、**推論リクエストのルーティング先を日本の 2 リージョン（東京 + 大阪）に限定する** 機能である。2025/10 に Claude Sonnet 4.5 / Haiku 4.5 で提供が開始され、その後 Sonnet 4.6 / Opus 4.7 も順次対応した。

### 2.1 AWS 公式の位置づけ — 規制業界を明確なターゲットに

AWS 公式ブログは、JP CRIS の提供開始に際して、**想定する業界を明記**している。

> This new feature is recommended for sectors such as **financial institutions, healthcare providers, and government agencies** handling sensitive data.
>
> — AWS Machine Learning Blog, 2025/10/31

つまり、AWS 自身が金融・医療・政府を JP CRIS の主要ユースケースとして位置づけている。これは単なる throughput 向上機能ではなく、**データレジデンシー要件への対応手段** として設計されている。

加えて、公式ドキュメントは GEO CRIS と Global CRIS の使い分けについても明確にしている。

> **For customers who need to process data locally within specific geographical boundaries**, GEO CRIS is the recommended option, as it makes sure inference processing stays within the geography boundaries of the specified GEO.

「データを特定地理境界内で処理する必要がある顧客」に対する推奨オプションが GEO CRIS（JP / AU / US / EU のいずれか）である、という位置づけ。

### 2.2 対応モデルラインナップ

2026/5 時点で、AWS モデルカード上で JP Geo プロファイル ID を確認できる Anthropic Claude モデルは以下の 3 系統である。

| モデル | Inference Profile ID | Bedrock 提供開始日 | JP CRIS 対応時期 | 備考 |
| --- | --- | --- | --- | --- |
| Claude Opus 4.7 | `jp.anthropic.claude-opus-4-7` | 2026/4/16 | 2026/4 以降 | Tokyo In-Region も可、1M context |
| Claude Sonnet 4.6 | `jp.anthropic.claude-sonnet-4-6` | 2026/2/17 | 2026/2 以降 | 1M context |
| Claude Sonnet 4.5 | `jp.anthropic.claude-sonnet-4-5-20250929-v1:0` | 2025/9/30 | 2025/10/31（JP CRIS 初期リリース） | — |

Haiku 4.5 は 2025/10/31 の AWS 公式ブログ時点では Sonnet 4.5 と並ぶ JP CRIS 対象として案内されたが、**執筆時点（2026/5）の Haiku 4.5 モデルカードには JP Geo テーブルが掲載されていない**。Tokyo からは In-Region 呼び出しのみ可能な状態のため、規制業界で Haiku 4.5 を採用する場合は Tokyo In-Region 前提で設計するか、最新状況を AWS Support / Partner に確認してほしい。

注目すべきは、**最新フラッグシップ（Opus 4.7）まで JP CRIS に対応している**ことである。規制業界だからといって、最新モデル性能を諦める必要はない。

### 2.3 ルーティング構造（Tokyo ⇔ Osaka）

JP CRIS は、ap-northeast-1（東京）と ap-northeast-3（大阪）の 2 リージョン間で推論リクエストをルーティングする。どちらをソースリージョンとして呼び出しても、ルーティング先は同じ 2 リージョン内に留まる。

| Source Region | Destination Regions |
| --- | --- |
| ap-northeast-1 (Tokyo) | ap-northeast-1 (Tokyo), ap-northeast-3 (Osaka) |
| ap-northeast-3 (Osaka) | ap-northeast-1 (Tokyo), ap-northeast-3 (Osaka) |

AWS 公式ブログは、このトラフィックが AWS Global Network 上で転送されることを明記している。

> The traffic flows exclusively over the AWS Global Network without traversing the public internet between Regions listed as destination for your source Region.

つまり、Tokyo ⇔ Osaka 間の通信はパブリックインターネットを経由しない。規制業界のデータ処理要件と整合する設計である。

### 2.4 Global / US / EU / AU / JP の違い

Bedrock の Cross-Region Inference Profile は、地理的スコープで 5 種類に分類される。Opus 4.7 を例にすると、利用可能なプロファイルは以下の通り。

| Profile 種別 | Inference Profile ID 例 | ルーティング範囲 | 規制業界での推奨 |
| --- | --- | --- | --- |
| **Global** | `global.anthropic.claude-opus-4-7` | 全世界の商用リージョン | ✗（国外ルーティングあり） |
| **US Geo** | `us.anthropic.claude-opus-4-7` | US リージョン群 | ✗（米国内完結） |
| **EU Geo** | `eu.anthropic.claude-opus-4-7` | EU リージョン群 | ✗（EU 内完結、日本の要件と不整合） |
| **JP Geo** | `jp.anthropic.claude-opus-4-7` | 東京 + 大阪 | **◯（日本国内完結）** |

規制業界で国内データレジデンシーを満たす唯一の Geo プロファイルが `jp.*` である。Global プロファイルは throughput を優先する構成で、データが全世界の任意リージョンにルーティングされる可能性があるため、規制業界では使わない。

なお、Opus 4.7 は Tokyo で **In-Region 呼び出し** もサポートする（後述 §5）。単一リージョン完結が必要な厳格要件では、こちらの選択肢も有効になる。

## 3. Claude Code on AWS で JP CRIS を使う実装

Claude Code 公式ドキュメントは、Bedrock 経由での認証・モデル指定の仕組みを詳しく説明しており、inference profile ID / ARN の指定もファーストパーティでサポートする。ここでは規制業界向けの標準構成を示す。

### 3.1 環境変数設定

Claude Code で JP CRIS を使う最小構成は以下の通り。

```
# Bedrock バックエンドを有効化
export CLAUDE_CODE_USE_BEDROCK=1

# ソースリージョンは東京を指定（大阪 = ap-northeast-3 でも可）
export AWS_REGION=ap-northeast-1

# モデルを JP CRIS プロファイルにピン留め
export ANTHROPIC_DEFAULT_OPUS_MODEL='jp.anthropic.claude-opus-4-7'
export ANTHROPIC_DEFAULT_SONNET_MODEL='jp.anthropic.claude-sonnet-4-6'
# Haiku 4.5 は執筆時点で JP Geo プロファイル未掲載。Tokyo In-Region で運用する例:
export ANTHROPIC_DEFAULT_HAIKU_MODEL='anthropic.claude-haiku-4-5-20251001-v1:0'
```

ポイント:

* `CLAUDE_CODE_USE_BEDROCK=1` が Bedrock 経路のトリガー
* `AWS_REGION` は Claude Code が `.aws/config` を読まないため、明示的に設定する
* `ANTHROPIC_DEFAULT_*_MODEL` で各 alias（opus / sonnet / haiku）を JP CRIS プロファイルに固定
* これらを設定しないと、Claude Code のデフォルトは `us.*` プレフィックスになる（§4.1 参照）

### 3.2 IAM 権限設計

JP CRIS の inference profile を呼び出すには、以下 2 つの権限が必要になる。

1. inference profile リソースへの `bedrock:InvokeModel*`
2. profile のルーティング先 foundation model への `bedrock:InvokeModel*`（条件付き）

AWS 公式ブログが示したサンプルをベースにした、Opus 4.7 用 IAM ポリシー:

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowJPCRISInferenceProfile",
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream",
        "bedrock:ListInferenceProfiles",
        "bedrock:GetInferenceProfile"
      ],
      "Resource": [
        "arn:aws:bedrock:ap-northeast-1:<account-id>:inference-profile/jp.anthropic.claude-opus-4-7",
        "arn:aws:bedrock:ap-northeast-1:<account-id>:inference-profile/jp.anthropic.claude-sonnet-4-6"
      ]
    },
    {
      "Sid": "AllowBackingFoundationModels",
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream"
      ],
      "Resource": [
        "arn:aws:bedrock:ap-northeast-1::foundation-model/anthropic.claude-opus-4-7",
        "arn:aws:bedrock:ap-northeast-3::foundation-model/anthropic.claude-opus-4-7",
        "arn:aws:bedrock:ap-northeast-1::foundation-model/anthropic.claude-sonnet-4-6",
        "arn:aws:bedrock:ap-northeast-3::foundation-model/anthropic.claude-sonnet-4-6"
      ],
      "Condition": {
        "StringLike": {
          "bedrock:InferenceProfileArn": "arn:aws:bedrock:ap-northeast-1:<account-id>:inference-profile/jp.anthropic.*"
        }
      }
    },
    {
      "Sid": "AllowHaikuTokyoInRegion",
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream"
      ],
      "Resource": [
        "arn:aws:bedrock:ap-northeast-1::foundation-model/anthropic.claude-haiku-4-5-20251001-v1:0"
      ]
    }
  ]
}
```

ポイント:

* `bedrock:InferenceProfileArn` の Condition で、Opus 4.7 / Sonnet 4.6 の foundation model への直接呼び出しを禁止しつつ、JP CRIS 経由の呼び出しのみ許可する
* これにより「開発者が誤って `anthropic.claude-opus-4-7` を直接指定する」ケースでも、JP CRIS 以外の経路では動作しなくなる
* Haiku 4.5 は JP Geo プロファイルが現時点で掲載されていないため、`ap-northeast-1` の foundation model を直接許可する形で Tokyo In-Region 呼び出しのみ開く。モデルカードで JP Geo が追加され次第、他モデルと同じ形（inference profile + Condition）に揃える

### 3.3 SCP による境界強制

IAM だけでは、ユーザーが誤って `us.*` や `global.*` の inference profile を指定するリスクを防ぎきれない。**組織レベルの SCP（Service Control Policy）で、JP 以外のリージョンへの Bedrock 呼び出しを明示的に禁止する** のが規制業界での標準設計になる。

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyBedrockOutsideJapan",
      "Effect": "Deny",
      "Action": [
        "bedrock:InvokeModel*",
        "bedrock:Converse*",
        "bedrock:CreateModelInvocationJob"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "ap-northeast-1",
            "ap-northeast-3"
          ]
        }
      }
    },
    {
      "Sid": "DenyGlobalAndUSInferenceProfiles",
      "Effect": "Deny",
      "Action": [
        "bedrock:InvokeModel*",
        "bedrock:Converse*"
      ],
      "Resource": [
        "arn:aws:bedrock:*:*:inference-profile/global.*",
        "arn:aws:bedrock:*:*:inference-profile/us.*",
        "arn:aws:bedrock:*:*:inference-profile/eu.*",
        "arn:aws:bedrock:*:*:inference-profile/apac.*",
        "arn:aws:bedrock:*:*:inference-profile/au.*"
      ]
    }
  ]
}
```

第 1 ステートメントで「東京 / 大阪以外のリージョンでの Bedrock API 呼び出し」を一律禁止。第 2 ステートメントで「`jp.*` 以外の inference profile ARN」を明示的に禁止。この 2 段構えで、意図しない国外ルーティングのリスクを構造的に排除する。

### 3.4 modelOverrides による組織配布

Claude Code は `modelOverrides` 設定を通じて、組織ポリシーとして特定のモデル ID を application inference profile ARN にマッピングできる。CCoE が組織標準の設定ファイルを配布する際の主力機能になる。

```
{
  "modelOverrides": {
    "claude-opus-4-7": "arn:aws:bedrock:ap-northeast-1:<account-id>:application-inference-profile/opus-jp-prod",
    "claude-sonnet-4-6": "arn:aws:bedrock:ap-northeast-1:<account-id>:application-inference-profile/sonnet-jp-prod",
    "claude-haiku-4-5-20251001": "arn:aws:bedrock:ap-northeast-1:<account-id>:application-inference-profile/haiku-tokyo-inregion-prod"
  },
  "availableModels": ["opus", "sonnet", "haiku"]
}
```

この設定を組織標準（例: `.claude/settings.json` の enterprise 版、または managed settings）として配布すれば、**開発者が `/model` ピッカーで選べるモデルは、組織が承認した国内プロファイル経由のみ** に制限できる（Opus 4.7 / Sonnet 4.6 は JP CRIS、Haiku 4.5 は Tokyo In-Region）。

Application inference profile を使うメリット:

* コストタグ付けができる（部門別コスト配賦）
* Guardrails 設定を profile レベルで紐付けられる
* CloudTrail ログで用途が識別しやすい

### 3.5 Guardrails 統合

Bedrock Guardrails は、プロンプト入力・モデル応答に対するコンテンツフィルタリング・PII 検出・センシティブ情報マスクを提供する。規制業界では、PHI・個人情報・機密情報の誤投入を防ぐ防波堤として活用する。

Claude Code から Guardrails を有効化する方法:

```
{
  "env": {
    "ANTHROPIC_CUSTOM_HEADERS": "X-Amzn-Bedrock-GuardrailIdentifier: jp-prod-guardrail\nX-Amzn-Bedrock-GuardrailVersion: 1"
  }
}
```

Guardrails 側では、cross-region inference 対応を有効にしておく必要がある。PII 検出・denied topics（禁止トピック）・sensitive information filter を組み合わせて、業界固有のリスクに対応する設定を行う。

### 3.6 監査証跡

監査証跡は §8 で詳しく扱うが、ここでは概要のみ示す。

* **CloudTrail management events（デフォルト有効）**: Bedrock の control plane 操作に加え、`InvokeModel` / `InvokeModelWithResponseStream` / `Converse` / `ConverseStream` の呼び出しメタデータ（誰が・いつ・どの inference profile を呼んだか）がデフォルトで記録される
* **Model invocation logging（デフォルト無効、アカウント単位で明示有効化）**: プロンプト・応答本体を S3 / CloudWatch Logs に記録。入出力 JSON が 100 KB を超える場合や画像・動画などバイナリは S3 側に格納される

JP CRIS を使うと、CloudTrail management event の `requestParameters.modelId` に `arn:aws:bedrock:ap-northeast-1:xxx:inference-profile/jp.anthropic.claude-opus-4-7` が記録されるため、**「日本国内に閉じたルーティングで処理された」ことが事後に証明できる**。

## 4. 2 つの落とし穴

JP CRIS を使った規制業界向け構成には、公式ドキュメントに書かれているにも関わらず、見落とされやすい 2 つの罠がある。両方とも設計段階で対策しないと、「意図せず国外に出ていた」という事態を招く。

### 4.1 デフォルトは us.\*

Claude Code の Bedrock 経由での**デフォルト primary model** は、公式ドキュメントで以下のように定義されている。

| Model type | Default value |
| --- | --- |
| Primary model | `us.anthropic.claude-sonnet-4-5-20250929-v1:0` |
| Small/fast model | `us.anthropic.claude-haiku-4-5-20251001-v1:0` |

つまり、`CLAUDE_CODE_USE_BEDROCK=1` だけ設定して `ANTHROPIC_DEFAULT_*_MODEL` を明示していない場合、**Claude Code は `us.*` プレフィックスの inference profile を選択し、US リージョンにルーティングされる**。

これは Claude Code 単独の設計判断だが、規制業界では致命的な挙動になる。対策は以下 3 層で行う。

1. **`ANTHROPIC_DEFAULT_*_MODEL` を明示設定**（§3.1）
2. **`modelOverrides` で組織ポリシーとして強制**（§3.4）
3. **SCP で `us.*` プロファイルへのアクセスを明示的に Deny**（§3.3）

### 4.2 東京は Global CRIS のソース

もう 1 つの罠は、ap-northeast-1 (Tokyo) が Global CRIS のソースリージョンにも含まれていることである。

Bedrock 公式ドキュメントより:

> The Global cross-region inference profile is currently only supported on Anthropic Claude Sonnet 4 model for the following source Regions: US West (Oregon), US East (N. Virginia), US East (Ohio), Europe (Ireland), and **Asia Pacific (Tokyo)**. The destination Regions for Global inference profile include all commercial AWS Regions.

つまり、東京リージョンから `global.anthropic.claude-sonnet-4-6` のような Global プロファイルを呼び出すと、**全世界の任意の商用リージョンにルーティングされる**。これは JP CRIS の明確なアンチパターンである。

発生シナリオ:

* 開発者が「最新モデルでとにかく throughput を上げたい」と考え、`global.*` プロファイルを試す
* 設定ファイルのコピペミスで、他プロジェクトの `global.*` 設定が混入する
* CI / CD パイプラインや MCP サーバー経由で、意図せず Global プロファイルが呼ばれる

対策は §3.3 の SCP で示した通り、以下の Resource を明示的に Deny する。

```
arn:aws:bedrock:*:*:inference-profile/global.*
arn:aws:bedrock:*:*:inference-profile/us.*
arn:aws:bedrock:*:*:inference-profile/eu.*
arn:aws:bedrock:*:*:inference-profile/apac.*
arn:aws:bedrock:*:*:inference-profile/au.*
```

!

**SCP は組織ルートに設定する**

SCP は AWS Organizations の機能であり、組織ルートまたは OU レベルで設定する。各 AWS アカウントの IAM ポリシーだけでは、開発者権限のあるユーザーが自身の IAM ポリシーを緩和して回避するリスクが残る。**JP CRIS 境界の強制は、必ず SCP レイヤーで行う**こと。

あわせて、`aws:RequestedRegion` による東京 / 大阪以外のリージョンの Bedrock 呼び出し禁止も SCP で設定する。2 重の Deny が規制業界の標準構成。

## 5. Opus 4.7 の 2 つのモード

Claude Opus 4.7 は、JP CRIS 対応に加えて **Tokyo In-Region 呼び出し** もサポートする。JP CRIS と In-Region の両方が Tokyo で利用できる Claude モデルは、2026/5 時点では Opus 4.7 のみ（Sonnet 4.6 / Sonnet 4.5 は Tokyo In-Region 非対応。Haiku 4.5 は Tokyo In-Region のみで JP Geo 非掲載）。両モードを併用できる柔軟性は、規制業界の選択肢として重要な意味を持つ。

Opus 4.7 モデルカードの Regional Availability テーブル（抜粋）:

| Region | In-Region | Geo | Global |
| --- | --- | --- | --- |
| `ap-northeast-1` (Tokyo) | **Yes** | Yes | Yes |
| `ap-northeast-3` (Osaka) | No | Yes | Yes |
| `us-east-1` (N. Virginia) | Yes | Yes | Yes |
| `us-east-2` (Ohio) | Yes | Yes | Yes |
| `eu-north-1` (Stockholm) | Yes | Yes | Yes |
| `eu-west-1` (Ireland) | Yes | Yes | Yes |

Tokyo が In-Region / Geo / Global すべて Yes という稀少構成。Sonnet 4.6 は Tokyo が Geo のみで In-Region 不可（Sonnet 4.6 で In-Region が Yes なのは eu-west-2 の London のみ）なのと対照的である。Haiku 4.5 は Tokyo が In-Region のみ Yes で Geo 非掲載と、さらに非対称な配置になっている。

### 5.1 Tokyo In-Region（単一リージョン完結）

Tokyo In-Region モードは、**推論リクエストを ap-northeast-1 の 1 リージョン内で完結させる**。大阪にもルーティングされない。

指定方法:

```
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=ap-northeast-1
export ANTHROPIC_DEFAULT_OPUS_MODEL='anthropic.claude-opus-4-7'
```

モデル ID は Geo プレフィックス（`jp.`）なしの foundation model ID を直接指定する。

特徴:

* **最厳格コンプラ要件に対応**: 単一リージョン完結が明示的に求められるケース
* **ルーティング先が 1 箇所に固定**: 証跡の単純化、障害解析の容易化
* **東京単体の quota を使用**: 大阪のキャパシティは使えない
* **可用性は東京リージョンの SLA に依存**: 東京障害時のフェイルオーバー手段が別途必要

### 5.2 JP CRIS（Tokyo + Osaka）

JP CRIS モードは、ap-northeast-1 / ap-northeast-3 の 2 リージョン間でルーティングする。

指定方法（§3.1 と同じ）:

```
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=ap-northeast-1
export ANTHROPIC_DEFAULT_OPUS_MODEL='jp.anthropic.claude-opus-4-7'
```

特徴:

* **日本国内の 2 リージョン完結**: 国外に出ない保証は維持される
* **Tokyo + Osaka のキャパシティを利用**: スループット向上、東京障害時の自動フェイルオーバー
* **ルーティング先が動的**: 個別リクエストがどちらのリージョンで処理されたかは CloudTrail で確認する
* **高可用性を得やすい**: 通常運用での第一選択になりうる

### 5.3 コンプラ要件レベルによる選び分け

業界・案件ごとの要件レベルに応じて、In-Region と JP CRIS を使い分ける。

| 要件レベル | 典型業界 / 案件 | 推奨モード | トレードオフ |
| --- | --- | --- | --- |
| **最厳格**（単一リージョン完結が必須） | 金融基幹系、防衛系、個人情報大量保有 | **Tokyo In-Region** | 可用性・スループットが東京 SLA 依存 |
| **厳格**（国内完結が必須、複数リージョンは許容） | 金融周辺系、公共 ISMAP、医療 | **JP CRIS** | 東京 + 大阪のルーティング先が動的 |
| **標準**（個情法配慮レベル） | 一般的な SaaS、業務系 | **JP CRIS**（または Global） | 性能優先なら Global も選択肢になるが本記事では非推奨 |

Opus 4.7 の 2 モードが揃うことで、規制業界は**業務ごとにきめ細かくデータ所在地を設計できる**ようになった。この「Tokyo In-Region と JP CRIS を両取りできる」柔軟性は、他の Claude モデル（Sonnet 4.6 / Sonnet 4.5 は Tokyo In-Region 不可、Haiku 4.5 は JP Geo 非掲載）では現時点で得られない、Opus 4.7 固有のメリットである。

## 6. Kiro CLI が規制業界で選べない理由

ここまで Claude Code on AWS 側の実装を見てきた。では同じことが Kiro CLI でできるかというと、現時点では **構造的にできない**。

### 6.1 Kiro CLI には inference profile を明示指定する手段が公開されていない

Kiro CLI は、モデル選択として Auto ルーターを中心に据えている。`kiro-cli` は内部で Kiro サービス経由で各モデルバックエンド（Claude / DeepSeek / GLM / Qwen / MiniMax）にルーティングする構造で、**ユーザーがモデルを切り替える公開手段は `/model` コマンドによる alias 選択に限られる**（2026/5 時点、公式ドキュメント確認）。

Claude Code 側が `ANTHROPIC_MODEL='jp.anthropic.claude-opus-4-7'` のような環境変数で JP CRIS inference profile を明示指定できるのに対し、Kiro CLI には等価な設定項目が公開されていない。結果として、**Bedrock のどの inference profile が使われるかは Kiro サービス側の実装に委ねられる**ことになる。

### 6.2 規制業界では最新モデル Opus 4.7 を享受できない

Kiro CLI で inference profile を明示指定する手段が公開されていない現状の、規制業界にとっての実務的な帰結は 2 つある。

**第 1 に、データレジデンシー保証が得られない。**  
JP CRIS を明示指定できない以上、「このリクエストは日本国内に閉じた」という統制・証跡が成立しない。SCP で Bedrock API のリージョンを限定する方法も、Kiro CLI が Kiro サービス経由でどのように Bedrock を呼ぶかによっては機能しない可能性がある。

**第 2 に、最新モデル（Opus 4.7）を規制要件下で使うパスがない。**  
§5 で見たように、Opus 4.7 の Tokyo In-Region も JP CRIS も、どちらも inference profile 指定（または foundation model ID 指定）を前提とする。Kiro CLI の Auto ルーターは、ユーザーがコンプラ要件を満たすかどうかを意識せずにモデル選択する設計であり、規制業界の「明示的に JP プロファイルに pin する」要件と相性が悪い。

!

**現時点の Kiro CLI は規制業界で採用候補から外れる**

規制業界のデータレジデンシー要件下では、inference profile を明示指定できないツールは採用候補から外れる。これは Kiro CLI / Kiro IDE の品質を否定するものではなく、**現時点の設計思想が規制業界のユースケースと合わない** という構造的な制約である。Kiro CLI は一般業務・新規事業開発・スタートアップ向けに最適化されたツールであり、規制業界はそのターゲット外と整理するのが実務上は正確である。将来の機能追加については [kirodotdev/Kiro Issues](https://github.com/kirodotdev/Kiro/issues) と [Kiro CLI Changelog](https://kiro.dev/changelog/) を定期的に確認するとよい。

### 6.3 シリーズ 6 本目マトリクスの規制業界フィルター

シリーズ 6 本目で提示した「Eclipse 利用企業の 4 つの現実的シナリオ」マトリクスに、**規制業界フィルター** を適用すると、選択肢は以下のように絞り込まれる。

|  | Kiro 系（後継ライン） | Claude Code 系（Bedrock） |
| --- | --- | --- |
| **Eclipse 継続パス** | シナリオ B Eclipse + Kiro CLI **✗ 規制業界で不可** | **シナリオ A** Eclipse + Claude Code on AWS **◯ JP CRIS / In-Region 対応** |
| **IDE 刷新パス** | シナリオ C Kiro IDE 全社移行 **✗ 規制業界で不可** | **シナリオ D** VS Code + Claude Code on AWS **◯ JP CRIS / In-Region 対応** |

規制業界では **シナリオ A と D のみが現実解** となる。IDE を Eclipse で続けるか VS Code に刷新するかは組織の IDE 戦略次第だが、**AI ラインは Claude Code on AWS 一択** である。

シリーズ 6 本目のマトリクスは「機能面では Kiro 系と Claude Code 系は対等」と論じたが、**規制業界という特定の条件下では対等ではない**。この非対称性は、AI コーディングツールの選定における重要な判断材料になる。

## 7. 業界別採用パターン（金融・公共・医療）

ここまで Claude Code on AWS × JP CRIS の実装と落とし穴を見てきた。これを規制業界別にどう組み合わせるかは、業界固有の規制要件・IT 環境・コンプラ文化で変わる。本章では金融・公共・医療の 3 業界について、仮想事例ベースで採用パターンを整理する（実企業名は出さない）。

### 7.1 横断比較表

| 項目 | 金融（地銀の内製開発） | 公共（中央省庁 + SIer） | 医療（電子カルテベンダ） |
| --- | --- | --- | --- |
| **規模感** | システム部 200 名、内製 40 名、Q Dev Pro 40 席 | 省庁 100 名 + SIer 300 名 | 開発 150 名、施設 500 導入 |
| **規制主軸** | FISC 第 11 版、単一リージョン志向 | ISMAP 管理基準、調達仕様書 | 3 省 2 ガイドライン、PHI 非露出 |
| **シリーズ #6 シナリオ** | シナリオ A（Eclipse + Claude Code on AWS） | シナリオ D（VS Code + Claude Code on AWS） | シナリオ D に段階移行 |
| **AI モード** | **Opus 4.7 Tokyo In-Region**（§5.1） | **Opus 4.7 JP CRIS**（§5.2） | **Opus 4.7 JP CRIS**（§5.2） |
| **AWS Support 契約** | Enterprise Support、SA アサイン | Partner 経由（SA 別途協議） | Business Support、Support + Partner |

### 7.2 金融（FISC 第 11 版）の設計ポイント

Opus 4.7 Tokyo In-Region を選ぶ理由は、FISC 第 11 版準拠を志向する内規において「単一リージョン完結」が最も議論しやすいこと。throughput より厳格性を優先する規模感に合う。

* SCP で **ap-northeast-1 のみ許可**（大阪も含めない単一リージョン構成）
* IAM は `anthropic.claude-opus-4-7` foundation model を直接許可（inference profile 経由でない）
* 東京リージョン障害時の業務継続は、JP CRIS への一時切替許可 or 業務停止のいずれかを運用設計で明文化
* `modelOverrides` で Opus 4.7 / Sonnet 4.6 / Haiku 4.5 の ARN を組織配布
* AWS Control Tower のガードレールで開発・検証・本番アカウント構成を強制

**AWS Support / Partner への確認論点**:

* FISC 第 11 版（最新追補版）での Bedrock の評価状況
* 「AI 推論処理を委託先管理の枠組みで整理する」リファレンスの有無
* 東京リージョン障害時の事業継続方針と JP CRIS 併用の許容範囲

### 7.3 公共（ISMAP）の設計ポイント

JP CRIS を選ぶ理由は、調達仕様書の「国内リージョン完結」要件は Tokyo + Osaka の 2 リージョン完結でも満たせること、行政システムの可用性要件を自動フェイルオーバーで担保できること。

ISMAP は機密区分（極秘 / 秘 / 部外秘 / 公表可）によって利用可能な構成が変わるため、**AI コーディングツールで扱うコード・プロンプトがどの区分に相当するかを案件ごとに整理する**必要がある。

| 機密区分 | 一般論（自組織方針に従うこと） |
| --- | --- |
| 公表可 / 部外秘 | JP CRIS 構成で議論可能 |
| 秘 | JP CRIS + Tokyo In-Region の組合せ、PrivateLink 経由のみ許可など追加分離を検討 |
| 極秘 | 原則として外部クラウド AI 利用を避ける方向で整理、または分離環境での特別承認プロセス |

* AWS Organizations OU を「ISMAP 対象」「ISMAP 対象外」に分離、対象 OU のみ §3.3 の SCP を適用
* Bedrock への **PrivateLink**（Interface VPC Endpoint）設定を検討し、`aws:SourceVpce` Condition で特定 VPC 経由のみ許可

**AWS Support / Partner への確認論点**:

* ISMAP 最新情報での Bedrock / JP CRIS の位置づけ（services-in-scope ページを定期確認。Bedrock の登録状況は、採用検討時点で ISMAP クラウドサービスリストを必ず一次確認）
* 調達仕様書に JP CRIS を明記する場合のベストプラクティス
* 政府共通プラットフォーム（GSS）との接続構成、分離環境（閉域網）での Bedrock 利用可能性

### 7.4 医療（3 省 2 ガイドライン）の設計ポイント

JP CRIS を選ぶ理由は、Opus 4.7 の 1M context と最新性能を活用しつつ、国内 2 リージョン完結で 3 省 2 ガイドラインの「外部保管」要件の議論基盤とすること。

医療の場合は **PHI（Protected Health Information）を AI に渡さない運用原則** が、技術構成と同じくらい重要になる。

* **原則 1**: ソースコード本体・テストデータから PHI を排除（完全匿名化・合成データのみ）
* **原則 2**: AI に参照させるサンプルに PHI を含めない（事前レビュー）
* **原則 3**: Guardrails で氏名・生年月日・患者 ID・保険証番号パターンの検出を有効化
* **原則 4**: 開発者教育で「AI プロンプト貼付け前のサニタイズ」を習慣化

## 8. 監査証跡の設計

シリーズ 5 本目「規制業界での Kiro 運用」で論じた監査証跡の基本構造は、Claude Code on AWS でもそのまま適用できる。ここでは Claude Code × JP CRIS 固有の論点を補足する。

### 基本構造（#5 と共通）

規制業界の監査証跡は、以下 3 つのレイヤーで構成される。

| レイヤー | 目的 | 実装 |
| --- | --- | --- |
| **開発活動の証跡** | 誰がいつ何を書いたか | Git commit、GitHub/GitLab/CodeCommit の Audit Log |
| **CI/CD の証跡** | どの変更がいつどう検証されたか | GitHub Actions / CodeBuild のログと成果物、Object Lock 付き S3 |
| **AI 推論の証跡** | どのモデルがどのリージョンで何回呼ばれたか | CloudTrail management events + Bedrock Model invocation logging |

この 3 層が揃ってはじめて、「AI を使った開発活動が、どこで・誰に・何のモデルによって処理されたか」を事後に証明できる。開発 / CI/CD 側の詳細はシリーズ 5 本目「規制業界での Kiro 運用」を参照のこと。

### CloudTrail と Model invocation logging の役割分担

Claude Code on AWS × JP CRIS の AI 推論証跡は、**2 つの異なる仕組みを組み合わせる**。混同しやすいので整理しておく。

| 仕組み | 記録内容 | デフォルト | 記録先 |
| --- | --- | --- | --- |
| **CloudTrail management events** | `InvokeModel` / `InvokeModelWithResponseStream` / `Converse` / `ConverseStream` の呼び出しメタデータ（誰が、いつ、どの inference profile を、どのリージョンから呼んだか） | **有効** | CloudTrail trail（S3） |
| **Bedrock Model invocation logging** | プロンプト・応答の**本体**（テキスト、画像、動画、埋め込み） | **無効**（アカウント単位で明示有効化） | S3 または CloudWatch Logs |

### JP CRIS 使用の証跡（CloudTrail management event）

JP CRIS を経由した `InvokeModel` 呼び出しは、CloudTrail ログに以下の情報が記録される。

* `eventName`: `InvokeModel` / `InvokeModelWithResponseStream` / `Converse` / `ConverseStream`
* `eventSource`: `bedrock.amazonaws.com`
* `requestParameters.modelId`: 呼び出された inference profile ID または ARN
* `awsRegion`: ソースリージョン（ap-northeast-1 または ap-northeast-3）
* `sourceIPAddress`: 呼び出し元 IP
* `userIdentity`: IAM Principal 情報
* `managementEvent`: `true`

重要なのは `requestParameters.modelId` に **`jp.anthropic.claude-opus-4-7`** や **application inference profile ARN** が記録されること。これにより、**事後的に「このリクエストは JP CRIS 経由で処理された」ことを証明できる**。

逆に言えば、ログ上に `us.*` / `global.*` / `eu.*` / `apac.*` / `au.*` の文字列が出てきたら、それは境界違反の証拠になる。CloudTrail Lake や Athena で定期的にスキャンする運用が望ましい。

CloudTrail trail の出力先は、運用アカウントとは別の監査アカウントの S3 バケットに設定し、Object Lock（Compliance Mode）で改ざん不可化する。これは #5 で論じた CI/CD 成果物保全と同じ考え方である。

### プロンプト / 応答の秘匿・保全（Model invocation logging）

CloudTrail management events は、プロンプトと応答の**本体**は記録しない（メタデータのみ）。業務によっては、プロンプト・応答の本体自体を監査対象にしたい場合がある。

Bedrock の **Model invocation logging** を有効化すると、S3 または CloudWatch Logs にプロンプト・応答の本体を記録できる。特徴は以下:

* アカウント単位・リージョン単位の設定（デフォルト無効）
* 対象オペレーション: `InvokeModel` / `InvokeModelWithResponseStream` / `Converse` / `ConverseStream`
* 入出力 JSON が 100 KB 以下なら CloudWatch Logs イベント / S3 の JSON 行に直接格納
* 100 KB 超や画像・動画などバイナリは、S3 バケット側に個別オブジェクトとしてオフロード
* 記録先は設定アカウント・同一リージョン内の S3 / CloudWatch Logs のみ

ただし運用上の検討事項がある:

* 記録対象に PHI や個人情報が含まれる可能性があり、取扱いに注意
* S3 保存時は KMS 暗号化（SSE-KMS）、アクセス権限の厳格化、保持期間の明確化が必要
* 記録内容にアクセスできるユーザーを限定（監査担当のみ）
* ログ自体が個情法・業界規制の対象物になる可能性がある

Model invocation logging を有効化するかは、監査法人・法務部門・情報セキュリティ担当と相談のうえ、「**記録することによるリスク** vs **記録しないことによるリスク**」を天秤にかけて判断する。AWS Support / Partner に設定時のベストプラクティスを確認するのもよい。

### 監査証跡の中期保全と検索

CloudTrail ログ・S3 オブジェクトは、業界規制の保持期間（金融 7-10 年、医療 5-10 年等）に合わせて保管する。S3 Lifecycle Policy で Glacier Deep Archive にオフロードすることで、長期保管コストを圧縮できる。

検索性を確保するため:

* CloudTrail Lake での SQL 検索
* CloudWatch Logs Insights
* Athena + Glue Crawler での S3 直接検索

規制当局の照会・監査法人の抜き取りチェック時に、**短期間で該当ログを抽出できる体制** を運用手順として文書化しておく。

## 9. 運用上の注意点

本番運用開始前に押さえておきたい項目を整理する。

### 9.1 quota 管理

JP CRIS の quota は、**ソースリージョン単位** で管理される。AWS 公式ブログより:

> For geographic-specific CRIS, quota management is performed at the source Region level. This means that quota increases requested from the source Region will only apply to requests originating from that Region.

つまり、東京（ap-northeast-1）から呼び出すリクエストの quota と、大阪（ap-northeast-3）から呼び出すリクエストの quota は別々にカウントされる。

実務的なポイント:

* デフォルトの quota は保守的な値で始まる（TPM: Tokens Per Minute, RPM: Requests Per Minute）
* 高利用が予想される場合、**AWS Service Quotas コンソールから事前に増枠申請** する
* 開発環境・検証環境・本番環境で別アカウントに分けている場合、アカウントごとに申請が必要
* Service Quotas の利用状況は CloudWatch メトリクスで監視可能。閾値アラートを設定

Opus 4.7 のデフォルト quota は `bedrock-runtime` で 15M TPM（東京 / 大阪それぞれ）と公式ドキュメントに記載あり。規模感として参考にできる。

### 9.2 レイテンシとフェイルオーバー

JP CRIS のルーティングは Tokyo ⇔ Osaka 間で行われる。ネットワークレイテンシの実測値は公表されていないが、物理距離（約 400km）と AWS Global Network 上の伝送を考慮すると、追加で 10-20ms 程度が見込まれる（実測推奨）。

フェイルオーバー設計:

* **JP CRIS モード**: 東京障害時、Bedrock が自動的に大阪にルーティングする（透過的）
* **Tokyo In-Region モード**: 東京障害時は、業務側で明示的な切替が必要
  + 切替先候補: JP CRIS モード（一時的に大阪ルーティングを許可）、または業務停止
  + どちらを選ぶかは BCP（事業継続計画）で事前に合意しておく

開発チームが PoC を実施する段階で、以下を実測しておくと本番設計の精度が上がる。

* 通常運用時のレイテンシ分布
* Tokyo ソース時と Osaka ソース時のレイテンシ差
* 1M context 利用時の応答時間（Opus 4.7 / Sonnet 4.6）

### Opus 4.7 のモデル制約

Opus 4.7 は、過去モデルと比べていくつか挙動が変わっている。公式モデルカードに記載のある制約を押さえておく。

* **Thinking モード**: `thinking.type: "adaptive"` のみサポート。`thinking.type: "enabled"` は 400 エラー（Opus 4.6 からの変更）
* **サンプリングパラメータ非対応**: `temperature`, `top_p`, `top_k` が非対応になった。推論時のチューニングは原則プロンプトで行う方針
* **Extended thinking の仕様変更**: budget\_tokens の明示指定ではなく、adaptive モードでモデル側が判断

Claude Code は通常利用ではこれらを自動的に吸収するが、カスタム呼び出しや MCP サーバー経由での呼び出しでは挙動差分を検証しておく。

### PoC で検証しておくべき項目

本番展開の前に、PoC 段階で以下を確認すると安全に移行できる。

* JP CRIS プロファイルへの pin が効いている（CloudTrail ログで確認）
* SCP で `us.*` / `global.*` への呼び出しが確実に Deny される（意図的にテスト）
* Guardrails が想定通りに作動する（PII パターンの検出テスト）
* Model invocation logging の記録内容（有効化する場合）
* レイテンシと throughput の実測値が業務要件を満たす
* 東京障害シミュレーション下でのフェイルオーバー挙動（JP CRIS 採用時）
* quota 超過時のエラーハンドリング

これらのチェックリストを事前にドキュメント化し、PoC 完了条件として明示しておく。

### ツール依存レイヤーと抽象レイヤーの切り分け

シリーズ 7 本を通じて見えてきたのは、AI コーディングツール導入の議論は以下のように層別できることである。

| レイヤー | 内容 | ツール依存 |
| --- | --- | --- |
| **上位（ガバナンス）** | CCoE 運用モデル、組織標準、監査証跡の基本構造 | 非依存 |
| **中位（設計パターン）** | 組織標準の配布メカニズム、開発者体験、セキュリティ境界 | 抽象化可能 |
| **下位（ツール固有）** | 設定ファイル・環境変数・CLI の具体、inference profile 指定 | 強く依存 |

CCoE は上位レイヤーの設計に最も労力を割き、中位レイヤーは抽象化で複数ツール対応を見据え、下位レイヤーは現在採用するツールで合わせる、という整理が実務的である。

規制業界という特定の条件下で Claude Code on AWS が選ばれるのは、下位レイヤー（inference profile 指定可否）の制約が上位レイヤー（データレジデンシー）の要件を通すかどうかで決まるからである。この構造を理解しておくと、今後 AI コーディングツールが新たに登場したときの評価軸も明確になる。

## まとめ

本記事の論点を 5 点で再確認する。

1. **規制業界のデータレジデンシー要件** は、FISC 第 11 版・ISMAP・3 省 2 ガイドライン・個情法 28 条のいずれにおいても、AI コーディングツールのデータ処理所在地を問題にしうる
2. **JP CRIS** は AWS 公式が規制業界向けと明言した、Bedrock の地理境界機能。執筆時点のモデルカード上で JP Geo プロファイルが確認できるのは Opus 4.7 / Sonnet 4.6 / Sonnet 4.5 の 3 系統で、ap-northeast-1 ⇔ ap-northeast-3 の 2 リージョンで完結する（Haiku 4.5 は 2025/10 のアナウンス時点の対象から現在掲載なし。Tokyo In-Region は利用可）
3. **Claude Code on AWS** は inference profile ID / ARN の明示指定をファーストパーティでサポートし、SCP + IAM + modelOverrides + Guardrails の組み合わせで規制業界の標準構成を組める
4. **2 つの落とし穴**（デフォルトは `us.*` / 東京は Global CRIS のソース）を SCP レイヤーでブロックする境界強制が必須。環境変数だけに頼る構成は不十分
5. **Kiro CLI には現時点で inference profile を明示指定する公開手段が提供されていない** ため、規制業界のデータレジデンシー要件下では選択肢から外れる。Claude Code on AWS（シナリオ A / D）が現実解となる。将来の機能追加は [kirodotdev/Kiro Issues](https://github.com/kirodotdev/Kiro/issues) と Kiro CLI Changelog を継続確認する

AI コーディングツールを規制業界に導入する検討は、組織の法務・コンプラ・監査法人・AWS Support・Partner・（SA のアサインがあれば）AWS アカウントチームと連携して進める。本記事の技術構成は、そうした議論の前提を整理するための技術リファレンスとして活用いただきたい。
