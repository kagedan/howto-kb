## はじめに
みなさんは Claude Code を使うときに、**サブエージェント（SubAgent）** と **スキル（Skills）** をうまく使い分けられていますか？

いまいち自分はサブエージェントを使う場面が分からないことが多く、「これってサブエージェントに投げるべき？それともスキルとして切り出すべき？」と毎回ちょっと迷っていました。
調べてもサブエージェント側の解説／スキル側の解説はそれぞれあるものの、**「結局どっちをどう使い分けるの？」** という比較記事があまり無かったので、自分の整理も兼ねてまとめてみます。

## SubAgentとは
>サブエージェントは、特定の種類のタスクを処理する特化した AI アシスタントです。サイドタスクがメイン会話に検索結果、ログ、または再度参照しないファイルコンテンツで溢れかえる場合に使用します。サブエージェントはそのタスクを独自のコンテキストで実行し、概要のみを返します。同じ種類のワーカーを同じ指示で繰り返し生成する場合は、カスタムサブエージェントを定義します。
**公式Docより**

サブエージェント（SubAgent）は、ざっくり言うと **「Claude が別の Claude を呼び出して、特定のタスクをまるごと任せる仕組み」** です。
メインのエージェント（あなたが対話しているClaude）が、必要に応じて専門のサブエージェントを起動し、そのサブエージェントが独立したコンテキストでタスクを処理して**結果だけ**を返してくれる、というイメージです。

## スキルとは
>スキルは、Claudeが動的に読み込んで特定のタスクのパフォーマンスを向上させる命令、スクリプト、リソースのフォルダです。スキルは、会社のブランドガイドラインに従ったドキュメント作成、組織固有のワークフローを使用したデータ分析、または個人的なタスクの自動化など、特定のタスクを反復可能な方法で完了する方法をClaudeに教えます。
**公式Docより**

ぶっちゃけ初見だと「うーん、サブエージェントと何が違うん？」となるんですが、ポイントは **「動的に読み込む手順書（プレイブック）」** だという点です。

スキルの特徴を整理するとこんな感じ。

- **実体はフォルダ + `SKILL.md`**
    - `SKILL.md` の中にトリガー条件・手順・参考リソースを書いておく
- **段階的に読み込まれる（progressive disclosure）**
  - スキルの **名前と `description`（frontmatter）は常にコンテキストに載っている**ので Claude は「どんなスキルが存在するか」を把握できる
  - 一方で **本体（`SKILL.md` の中身や同梱スクリプト）は、関連すると Claude が判断したときに初めて読み込まれる**
- **スクリプトや補助ファイルも同梱できる**
    - 単なるプロンプトではなく、決まった処理を呼ぶ手順や、社内ルールの YAML なんかも置ける
- **再利用性が高い**
- 「PR レビューの社内ルール」「Qiita スタイルでの記事作成」など、**チームで共有できるノウハウの単位**

つまり、サブエージェントが「人」だとすると、スキルは **「マニュアル」** に近いイメージです。



## 使い分けについて

ここまでくると、なんとなく違いが見えてきたかと思います。整理するとこんな表になります。

| 観点 | SubAgent | Skills |
|---|---|---|
| 役割 | 別Claudeにタスクを丸投げ | 特定タスクの手順書を読み込ませる |
| コンテキスト | **独立** している（結果のサマリだけ返る） | 現在のセッションに **追加** される |
| 主な使い所 | 大量調査・長時間タスク・並列処理 | 反復作業・社内ルール・定型ワークフロー |
| 共有 | プロジェクト／ユーザー／組織レベルで定義可能 | プロジェクト／ユーザー／組織レベルで定義可能 |

私なりの使い分け基準は次の通りです。

- **「結果だけ欲しい / 途中過程はメインの履歴に残したくない」 → SubAgent**
  - 例: 「このコードベース全体から〇〇の使用箇所を洗い出して」
  - 例: 「並列でファイル A・B・C の影響範囲をそれぞれ調べて」
- **「決まった手順で毎回同じように動いて欲しい」 → Skill**
  - 例: 「PR を作るときは必ずこのテンプレで」
  - 例: 「ブログを書くときはこの文体ガイドに従って」
- **コンテキストウィンドウの節約**
    - 全ての情報をメインの会話に詰め込むと、トークン数が膨大になってしまう
- **両方使うパターンもアリ**
  - スキルの手順書の中で「ここの調査はサブエージェントに任せる」とすると、コンテキスト節約しつつ手順も再現でき、両方の強みを活かすことができます

迷ったときは **「再利用するか？」** で考えるとスッキリします。1回しか使わない探索は SubAgent、何度もやることは Skill、です。

## 最近SubAgentを使う人が減ってきた...?

ここからは完全に私の主観なんですが、**最近サブエージェントを使う人がちょっと減ってきた気がしませんか？**

X(旧Twitter)とか個人ブログを眺めていても、半年前くらいは「サブエージェント分割が正義」みたいな流れが強かった気がするんですが、最近は「メインに全部やらせる派」が増えてきている印象です。

理由は何となく察しが付いていて、

- **Claude Sonnet / Opus が普通に賢くなった**
  - 単体でも複雑なタスクを最後までやり切れるケースが増えた
- **コンテキストウィンドウが広くなった**
  - わざわざ別エージェントに切り出して節約しなくても、メインで全部読める
 - **Skills の登場で役割の棲み分けが進んだ**
    - 「定型処理を任せる」用途は Skills 側に寄せ、「独立コンテキストで重い調査を回す」は SubAgent、という具合に**競合ではなく補完関係**として整理されてきた

とはいえ、SubAgent が要らなくなったわけでは全然なくて、**「並列でガッと走らせたい」「巨大な探索でメインの履歴を汚したくない」** ようなケースでは今でも圧倒的に強いです。

要は **適材適所** ということですね。「とりあえずサブエージェント分割しとけ」みたいな思考停止だけは避けたいところ。

## 終わりに

いかがでしたでしょうか？

今回は Claude のサブエージェントとスキルの違いと使い分けについてまとめてみました。

ざっくり覚えておくと良いのは、

- **SubAgent = 別Claudeに丸投げ（コンテキスト独立・サマリだけ返る）**
- **Skills = 動的に読み込む手順書（再利用・チーム共有向き）**

の2点です。あとは「再利用するか？」で判断すれば、だいたい困らないはずです。

Claude Code、本当に進化が早くて毎月のように使い方のベストプラクティスが変わっていくので、引き続きキャッチアップしながら自分の開発フローに馴染ませていきたいなと思います。
