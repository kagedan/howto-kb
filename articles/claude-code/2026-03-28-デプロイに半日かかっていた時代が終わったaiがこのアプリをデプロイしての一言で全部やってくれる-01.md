## この記事でわかること（ミニ結論）

* MicrosoftがClaude CodeとGitHub Copilotに**Azureへの自動デプロイ機能を追加**する  
  「Azure Skills Plugin」を公開した
* AIエージェントに\*\*「このアプリをデプロイして」と指示するだけ\*\*で、インフラ選定から  
  Dockerfile生成・ビルド・デプロイまで全自動で実行される
* AWSも同様の\*\*「Agent Plugins for AWS」\*\*を同時期にリリースしており、  
  クラウド各社でAIによるデプロイ自動化が加速している
* **従来のデプロイ作業は多くの手動工程とヒューマンエラーの温床**だった
* デプロイという「エンジニアの最後の聖域」がAIに委ねられる時代が来た

## デプロイはなぜこんなに大変だったのか

まずAzure Skills Pluginの話をする前に、**従来のデプロイがいかに複雑だったか**を  
振り返っておく必要がある。

### 従来のデプロイ工程（手動時代）

```
① コードのビルド
   └ ビルドコマンドの実行
   └ 依存関係の確認
   └ テストの実行・確認

② Dockerイメージの作成
   └ Dockerfileの手動記述
   └ イメージのビルド
   └ タグ付け

③ インフラの準備
   └ クラウドサービスの選定（EC2？ECS？Lambda？）
   └ セキュリティグループの設定
   └ IAMロール・権限の設定
   └ 環境変数の設定

④ デプロイの実行
   └ イメージのプッシュ
   └ デプロイコマンドの実行
   └ ヘルスチェックの確認

⑤ 動作確認
   └ ログの確認
   └ エラーの修正
   └ ロールバックの判断
```

これを**手動でやると半日〜1日かかることも珍しくなかった。**

### 従来のデプロイで起きがちだったミス

```
# ❌ よくあるミス集

# 1. 環境変数の設定忘れ
docker run myapp
# → 本番環境でDB_PASSWORDが未設定でアプリがクラッシュ

# 2. 間違ったタグでデプロイ
docker push myapp:latest  # latestタグは危険
# → 本番に古いバージョンがデプロイされていた

# 3. ポート番号の設定ミス
EXPOSE 8080  # Dockerfileに書いた
# → 実際のアプリは3000番で動いていた...

# 4. ステージング環境と本番環境の設定を混同
export DB_HOST=staging-db.internal  # ← 本番なのにステージングを向いてた

# 5. リソース制限の設定忘れ
# メモリ制限なし → OOMでコンテナが突然死
```

| よくあるミス | 原因 | 影響 |
| --- | --- | --- |
| 環境変数の設定忘れ | 手動コピペ・チェックリスト漏れ | 本番障害・データ損失 |
| 間違ったバージョンのデプロイ | タグ管理の属人化 | 機能退行・予期しない動作 |
| ポート番号のミス | Dockerfile手動記述のミス | サービス無応答 |
| 権限設定のミス | IAMポリシーの複雑さ | セキュリティリスク・アクセス不能 |
| ロールバック忘れ | 障害対応の混乱 | ダウンタイムの長期化 |

**こうした手動工程とヒューマンエラーの積み重ねが、デプロイを「恐怖のイベント」にしていた。**

## 最新動向：Azure Skills Pluginの登場

2026年3月25日、MicrosoftはClaude CodeとGitHub Copilotに  
\*\*Azureへの自動デプロイ機能を追加する「Azure Skills Plugin」\*\*を公開した。

AWSが同様の「Agent Plugins for AWS」をリリースした翌日のことだ。  
クラウド大手2社が同時期にAIデプロイ自動化をリリースしたことで、  
この流れは**業界標準になりつつある**ことが明確になった。

### Azure Skills Pluginの3つのコンポーネント

| コンポーネント | 役割 |
| --- | --- |
| **Azure Skills** | ビルド・デプロイ・監視・トラブルシューティング・インフラ設計・データ/AIアクセスなど**20種類のスキルファイル** |
| **Azure MCP Server** | **40以上のAzureサービス**にリアルタイム接続。コスト情報やログの取得・サービス操作を実行 |
| **Foundry MCP** | AI関連サービスに特化。Microsoft Foundryのモデルカタログ・エージェント・デプロイを操作 |

### 実際の動作フロー

```
ユーザー：「このアプリをAzureにデプロイして」
    ↓
【Decision Layer（判断層）】
    ↓
AI がコードを解析
    ↓
┌─────────────────────────────────────┐
│ ・アプリの種類を判定                  │
│ ・最適なAzureサービスを選定           │
│   (App Service? Container Apps?     │
│    Azure Functions? AKS?)           │
│ ・必要なリソースを見積もり            │
└─────────────────────────────────────┘
    ↓
Azure MCP Server経由で実行
    ↓
┌─────────────────────────────────────┐
│ ・Dockerfileを自動生成               │
│ ・構成ファイルを自動生成              │
│ ・ビルドを実行                       │
│ ・Azureへデプロイ                    │
│ ・ヘルスチェックを確認               │
└─────────────────────────────────────┘
    ↓
デプロイ完了を報告
```

**人間がやることは「デプロイして」と言うだけ。**

## 過去との比較：デプロイの進化

| 時代 | 方法 | 所要時間 | 主なリスク |
| --- | --- | --- | --- |
| 〜2010年代前半 | FTP手動アップロード・サーバーSSH直操作 | 数時間〜1日 | ファイル上書きミス・本番直接編集 |
| 2010年代後半 | CI/CD（Jenkins・CircleCI）+ スクリプト | 30分〜2時間 | 設定ファイルのミス・環境差異 |
| 2020年代前半 | IaC（Terraform・CloudFormation）+ GitOps | 15分〜1時間 | 宣言的設定の複雑化・ドリフト |
| **2026年〜** | **AIエージェントへの自然言語指示** | **数分** | **AIの判断ミス・過剰なリソース確保** |

デプロイにかかる時間が**数時間から数分へ**。  
ただしリスクがゼロになったわけではなく、**「人間のミス」から「AIの判断ミス」へ**リスクの性質が変わった。

## 現場エンジニア目線の考察

半導体の現場でも、設備の制御ソフトやデータ収集システムの  
デプロイは長らく「職人技」だった。

特定の担当者しか手順を知らない・環境ごとに設定が微妙に違う・  
本番デプロイは深夜に行うのが慣例──そういった現場を見てきた身としては、  
今回の発表は\*\*「デプロイの民主化」\*\*として非常に大きな意味を持つと感じる。

ただし手放しで喜べない点もある。

* **AIが選んだインフラが本当に最適かどうか**をレビューできる人間が必要
* **コスト管理**：AIが自動でリソースを確保すると、気づかないうちにコストが  
  膨らむリスクがある
* **セキュリティ設定**：自動生成された権限設定が過剰になっていないか確認が必要
* **製造・半導体系のOT環境**ではクラウドデプロイ自体がまだハードルが高く、  
  この恩恵を受けるにはインフラの整備が先決

「AIに任せる」と「AIを使いこなす」は全く違う。  
**デプロイの手順を知らないエンジニアがAIに任せるのは危険**で、  
逆に手順を熟知したエンジニアがAIを使うと圧倒的な生産性向上になる。

## まとめ・次のアクション

**結論：デプロイは「恐怖のイベント」から「会話のひとこと」に変わりつつある。  
ただしAIの判断を検証できる知識は依然として必要だ。**

今すぐ取り組むべきことは以下だ。

1. **Azure Skills Plugin / Agent Plugins for AWSを試す**  
   （Claude CodeまたはGitHub Copilot環境があれば今すぐ使える）
2. **AIが生成したDockerfile・構成ファイルを必ずレビューする習慣をつける**
3. **コスト上限（Budget Alert）を設定してからAIデプロイを使う**
4. **従来の手動デプロイ手順を一度ドキュメント化しておく**  
   （AIが何をやっているか理解するための基礎知識として）

デプロイを知っているエンジニアが、AIを使って10倍速く動く時代が来た。  
デプロイを知らないエンジニアが、AIに任せて10倍のリスクを抱える時代も来た。
