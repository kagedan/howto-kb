## はじめに

Claude Code でタスクを自動オーケストレーションするハーネスを自作している。その中に **fugu-router** という小さな Rust 製プラグインがある。名前から想像がつく通り、Sakana AI が2026年6月に発表した **fugu** から着想を得ている。

ただし fugu-router は fugu の劣化コピーではない。「fugu が学習済みコーディネータでやっていることを、Claude の重みを一切いじれない制約の中で、どう近似するか」という設計判断の産物で、実装には\*\*多腕バンディット（Thompson sampling）\*\*を使っている。

この記事では、

* Sakana の fugu が何をやっているか
* なぜそのまま真似できないのか
* fugu-router がどう「学習」を代替設計したか（k-NN検索 + Thompson sampling の実装詳細）
* 開発中に踏んだバグ、かかったコスト
* 実際に本番で役に立っているのか

を、実装コードと開発ログの両方から書く。

## Sakana AIのfuguとは何か

[Sakana AI の fugu](https://sakana.ai/fugu-release/) は、複数の LLM を動的に統合するマルチエージェントシステムだ。単一の API 経由で呼び出せて、「フロンティア級の性能を出しながら単一ベンダーへの依存を避ける」ことを謳っている。

ポイントは、fugu 自体が **1つの言語モデルであり、「様々な LLM を呼び出すこと自体を学習している」** という点にある。タスクを受け取ると、

1. 自分で直接解くか
2. Thinker / Worker / Verifier のような役割ごとに専門モデルのチームを組成するか

を自動判断し、モデルの選択・委譲・検証・統合をすべて内部で完結させる。ユーザー側にオーケストレーションのコードは出てこない。

この「いつ委譲すべきか」「エージェント同士がどう対話すべきか」を学習する部分は、ICLR 2026 に採択された TRINITY と Conductor という2本の研究がベースになっている。つまり **コーディネータそのものが訓練対象** で、進化戦略や強化学習によって「どのモデルにどう振るとうまくいくか」を重みとして獲得している。新しいモデルが世に出ると、それをプールに組み込んでコーディネータ自身も進化していく設計だ。

## そのまま真似できない理由

ここで詰まる。fugu の核心は「訓練済みコーディネータ」であり、その訓練には勾配更新ができる基盤モデルと評価環境が要る。一方 Claude Code から使えるのは Claude の重み固定の API 呼び出しだけで、コーディネータを勾配で学習させる手段がない。

「じゃあ fugu の真似は無理」と諦めるのが自然な結論に見えるが、ここで立ち止まって「fugu の本質は何か」を分解した。

> 学習かどうかは重要でなく、結果の積み重ねで賢い選択ができればよい

これが設計時にユーザーと交わした対話の結論だった。fugu の本質は「①コーディネータをオーケストレーション対象から切り離された独立コンポーネントとして持つこと」「②観測の積み重ねで判断が改善すること」の2つであって、「③勾配更新である」ことは手段の一つにすぎない。①と②さえ再現できれば、③をパラメトリックな学習ではなく**検索(retrieval)とオンラインバンディット**で置き換えても目的は満たせる、という判断をした。

## fugu-routerの設計 —— retrieval + online bandit

fugu-router は condukt（このハーネスのオーケストレーション背骨）が生成したタスク分解 JSON を受け取り、各タスクの `suggested_model` を「過去の実績」で上書きするだけの、決定論的なコマンドラインツールだ。

```
record で実績を蓄積 ── episodes.jsonl ──▶ 似た過去タスクを k-NN で検索
(model, pass?, cost)                       │
                                           ▼
                            しきい値を歴史的に満たす
                            最安ティア  →  suggested_model
```

学習を意図的に2系統に分けている。

| 系統 | 手法 | 強み |
| --- | --- | --- |
| 非パラメトリック | 記録済みエピソードへの k-NN 検索 | 1件の追加で即座に挙動が変わる／完全に解釈可能／悪いエピソードを消すだけで訂正できる |
| パラメトリック/オンライン | ティアごとの Beta 事後分布への Thompson sampling | 実績の乏しい安いティアも不確実性の下で試され、結果からオンラインで方策が更新される |

似ているかどうかの判定（k-NN の距離）は、タイトル + ファイルパスの Jaccard 類似度に加えて、接尾辞ステミングとドメイン概念辞書（`semantic.rs`）を組み合わせている。埋め込みサービスなしで `login ↔ auth ↔ session` のような同義語を橋渡しするための工夫だ。

限界は正直に書いておく。ルーティングの粒度はトークン単位でも、ターン単位でもなく**タスク単位**。fugu にあるような隠れ状態ヘッドによるルーティングや、ニューラルな重み更新は存在しない。バンディットが学習するのは「どのティアが報酬(pass)に見合うか」という1次元の統計量だけで、深い表現学習ではない。意味的な橋渡しも埋め込みではなく辞書ベースだ。粗いが、その代わり API キーも埋め込みサービスも要らない — Claude のサブスクリプションの中だけで完結する。

## 本題: 多腕バンディットとしてのモデル選択

### 解いている問題

haiku / sonnet / opus という3本の「腕」があり、それぞれ「検証を通る(pass)確率」という未知の報酬分布を持つ。各タスクで1本を選び、結果(pass/fail)を観測する。目標は総報酬の最大化、つまりリグレットの最小化。

```
リグレット(T) = T・μ*  −  Σ 実際に得た報酬      (μ* = 最良腕の期待報酬)
```

核心は **探索(exploration)と活用(exploitation)のトレードオフ**。今まで一番調子のいい腕を引き続ければ安定するが、本当はもっと良い（もっと安い）腕を見逃しているかもしれない。逆に未知の腕ばかり試していては、劣った腕に報酬を浪費する。

### Thompson sampling の考え方

Thompson sampling のアイデアは一言で言うと、**「各腕の当たり率を “確信度つき” で予想し、その予想からくじを引いて一番高い腕を選ぶ」**。当たり率を1つの数字ではなく分布(山)で持つのがポイントで、

* 試行が少ない腕 → 山が広い(自信なし) → たまに高い値が出る → 探索される
* 試行が多い腕 → 山が狭い(確信あり) → 安定した値が出る → 活用される

この「山」はベルヌーイ報酬の共役事前分布である Beta 分布で表現できる。成功・失敗を数えるだけで更新できるのが利点だ。

```
山 = Beta(1 + 当たった回数, 1 + 外れた回数)
  0勝0敗 → Beta(1,1) = 真っ平ら(0〜100%まったく不明)
  7勝3敗 → Beta(8,4) = 70%付近に寄った山。試行が増えるほど尖る
```

毎ラウンドの手順は「①各腕の山からくじを1枚引く → ②引いた値が最大の腕を選ぶ → ③当たり/外れでその腕の山を更新」の3ステップだけ。自信のない腕はたまに高いくじが出て試され、情報が増えて山が狭くなる。本当に良い腕は高い位置で狭くなり安定して選ばれる。探索と活用のバランスが自動で取れる。

ε-greedy（確率εでランダム探索）や UCB（不確実さをボーナス加点する決定論的手法）と比べても、Thompson sampling は「不確実な腕を確率的に、かつ不確実さに応じた頻度で試す」ため実務で強いとされる。手元で3本腕(真の当たり率 A=0.30, B=0.55, C=0.70)を2000ラウンド回す最小構成の検証をすると、Thompson sampling はほぼ全ラウンドを最良腕Cに収束させ、リグレットはε-greedyの半分以下に収まった(実測: Thompson regret=18 vs ε-greedy regret=50)。ダメな腕を早く見切って、良い腕にだけ試行を集中させる挙動が数字にはっきり出る。

### fugu-routerでの実装

fugu-router の実装はこのデモと2点だけ違う。

| デモ | fugu-router |
| --- | --- |
| Beta 分布から直接サンプリング | **正規分布で近似**してサンプリング(軽量・依存ゼロ) |
| 「くじが最大の腕」を選ぶ | 「閾値 `pass_threshold` を超えた中で一番**安い**ティア」を選ぶ(コスト最適化) |

実装はこう(`src/policy.rs::decide_bandit`、抜粋)。

```
let posterior = |tier: &str| -> (f64, f64, usize) {
    let (count, passes) = stats.get(tier).map(|s| (s.count, s.passes)).unwrap_or((0, 0));
    let a = 1.0 + passes as f64;
    let b = 1.0 + (count - passes) as f64;
    let mean = a / (a + b);
    let var = (a * b) / ((a + b) * (a + b) * (a + b + 1.0));
    (mean, var.sqrt(), count)
};

for (idx, tier) in TIERS.iter().enumerate() {
    let (mean, sd, count) = posterior(tier);
    // Beta(a, b) の平均・分散を正規分布に近似してサンプリング
    let sample = (rng.normal(mean, sd).clamp(0.0, 1.0) + cheap_bonus(idx)).min(1.0);
    if sample < pass_threshold {
        continue; // このティアはこのラウンドでは「くじが外れた」
    }
    // ...閾値を超えたティアの中から最安(=最も安い/最も効率が良い)を採用
}
```

`Beta(1+passes, 1+fails)` の平均と分散を計算し、正規分布からサンプリングする軽量近似にしている（`betavariate` のような専用実装を持たず、標準ライブラリの乱数だけで済ませたかったため）。

もう一つの工夫が `cheap_bonus`。安いティア(haiku)ほど大きく、最も高いティア(opus)ではゼロになる小さなボーナスをサンプル値に加算する。

```
/// haiku (idx 0) は満額のボーナス、opus (最終 idx) はゼロ。
/// 安いティアへ「だけ」バイアスがかかり、opus に有利になることは絶対にない設計。
fn cheap_bonus(idx: usize) -> f64 {
    let last = TIERS.len() - 1;
    EXPLORE_CHEAP_BONUS * (last - idx) as f64 / last as f64
}
```

これは「実績の少ない安いティアを、実績十分の高いティアより優先して試す」ためのバイアスだ。安いティアで pass すれば恒久的なコスト削減になるので、探索する価値がそもそも非対称に高い。ただしボーナスは小さく抑えてあり、本当に実績が悪いティアはこのボーナスだけでは救われない —— 僅差のときだけ安い方に転ぶ設計になっている。

さらに開発途中で「pass率が同じでも、コストが桁違いに違うティアがある」ことに気づき、`decide_bandit` にコスト効率調整を後から追加した(`cost-adjusted Thompson sampling`)。

```
// コストデータがある場合: サンプルされた合格率を cost-per-pass で調整して
// 「効率」で比較する。cpp が min_cpp の4倍なら、4倍高くサンプルされないと勝てない。
let efficiency = if cpp > 0.0 { sample / (cpp / min_cpp) } else { sample };
```

haiku が「5/5 pass, $0.10/pass」、sonnet が「5/5 pass, $1.00/pass」なら、合格率は同じでも haiku の効率スコアは10倍になり、ほぼ確実に haiku が選ばれる。コストデータが一切無ければ `min_cpp` が発散せず、従来通りの「cheapest-first」に自然劣化する後方互換設計にしてある。

乱数生成器 (`rng.rs`) も自前で書いた。標準ライブラリだけで完結させたかったのと、テストで決定論的な再現性が欲しかったためだ。

```
pub fn new(seed: u64) -> Self {
    // splitmix64 で最初の一発から系列をばらけさせる。
    // 連番シード(1, 2, 3, …)を素朴な xorshift に渡すと最初の draw が強く相関するため。
    let mut z = seed.wrapping_add(0x9E37_79B9_7F4A_7C15);
    z = (z ^ (z >> 30)).wrapping_mul(0xBF58_476D_1CE4_E5B9);
    z = (z ^ (z >> 27)).wrapping_mul(0x94D0_49BB_1331_11EB);
    z ^= z >> 31;
    Self { state: z.max(1) }
}
```

コメントに書いてある通り、この splitmix64 のスクランブルは本番で踏んだバグの修正の跡である(後述)。

### コールドスタートの設計

ストアが空の状態(履歴ゼロ)では k-NN も Thompson sampling も機能しない。そこでキーワードベースの prior にフォールバックするが、これも**安い方向に意図的にバイアス**してある。

* `design` / `architecture` / `security` などのハイステークスなキーワードがあれば即 opus
* `rename` / `typo` / `format` などの機械的作業は haiku(ファイル数が多い一括作業でも sonnet 止まり — 「作業の性質」で見て、ファイル数だけで高いティアに引き上げない)
* それ以外は変更ファイル数で haiku / sonnet / opus をスケール

「履歴が無ければとりあえず安全に高いティアを使う」ではなく「履歴が無ければ床(haiku)から始め、検証落ちのカスケードエスカレーション(haiku→sonnet→opus)が本当に必要なタスクだけを買い上げる」という設計判断にしている。安全側に倒して opus を使い続けると、それだけで恒久的にコストが高止まりし、「安く済むはずだったタスクが安く済んだ実績」がいつまで経っても貯まらないからだ。

verifier(検証者)も worker とは独立に、かつ worker より一段安いか高いかを状況に応じて選ぶ。opus で書いたコードは(コストを抑えるため)独立した sonnet で検証し、haiku で書いたコードは(haiku で haiku を検証しても意味がないため)一段上の sonnet で検証する。

## condukt との統合(実際に動くパイプライン)

```
condukt validate --file decomp.json
fugu-router route --file decomp.json --report /tmp/route.json > decomp.routed.json
condukt schedule --file decomp.routed.json
```

`route` サブコマンドが `suggested_model` を書き換えた JSON を吐き出し、そのまま condukt に渡す。タスクの検証が終わったら:

```
fugu-router record --title "<task title>" --files "<touched_files>" \
  --class parallel --model sonnet --status verified --cost 0.09
```

で結果を書き戻す。condukt と併用する場合は `record` の発火すら手書きが要らず、condukt の Stop hook が `condukt state record-run --all` で決定論的・冪等に行う。

大事な設計判断は、この結合を**ソフト**にしたこと。`fugu-router` バイナリが存在しなければ、condukt は interpreter 自身の `suggested_model` にそのままフォールバックし、何も壊れない。学習コンポーネントを外付けにして、無くても動く土台の上に載せている。

## 開発中に踏んだバグとコスト

fugu-router を最初に実装したセッション(2026-06-23)は、他の下回り実装や 18 個のプラグインを marketplace 経由で実機に導入する作業と合わせて **総コスト $32.21、総トークン約4600万(そのほとんどはキャッシュ読込)** かかっている。fugu-router 単体の重さではなく、そのセッションで一気にハーネス全体を本番投入したための数字だ。

実装中に見つかった実バグは主に2つ。

1. **ステミングの過剰除去**: `semantic.rs` の接尾辞ステミングが `renders` → `rend` のように削りすぎる不具合があった。`login ↔ auth` のような同義語橋渡しのための処理が、逆に無関係な単語を同一視してしまう副作用を生んでいた。
2. **シードの相関**: 当初、秒粒度のタイムスタンプをシードにしていたため、短時間に連続で `decide_bandit` を呼ぶと同じ乱数列になってしまう問題があった。さらに `Rng::new(1), Rng::new(2), Rng::new(3), ...` のような連番シードを素朴な xorshift64 に渡すと最初の draw が相関することも判明した。ナノ秒粒度への変更と、上で挙げた splitmix64 スクランブルの追加で解決している。

「学習ロジックの正しさ」よりも「乱数と文字列処理の地味なバグ」の方が圧倒的に踏みやすい、というのは実装してみて初めて分かった実感だった。

## 実際に役に立っているのか

作って終わりでは意味がないので、実運用のログを追ってみる。

* 導入した翌々日、ローカルに貯まった **29件のエピソード**を `fugu-router import` で record-repo にマージし、`fugu-router sync` で GitHub リポジトリ(`yukineko/fugu-router-record`)へ push。以後は Stop hook が自動で sync するようになっている。
* 実際の condukt run で、`ad-sync-create-user-ldap68-idempotent` というタスクが `worker=sonnet / verifier=opus` にルーティングされた実例がログに残っている(いずれも `create-user-ldap68` 系タスク)。design/security キーワードを含まない通常のタスクだが、ファイル数と過去実績から自動的にこの組み合わせが選ばれた。
* gauge(コスト計測プラグイン)が実測コストを記録し、それが fugu-router の cost-per-pass 学習信号として使われる —— 「HOTL 構成で両方入れる意味がある」という運用上の相乗効果も実際に確認できている。
* 副産物として、当初スコープ外だった `procedures`(旧 `playbook`)機能 —— 「似た検証済みタスクがどう解かれたか」を k-NN で検索して interpreter に注入する機能 —— も、開発が進む過程で「fugu-router のソースは実は既にこのリポジトリ内にある」と判明したことをきっかけに追加実装された。当初は「調査担当エージェントの `open_question`(未確認)」を鵜呑みにして見送っていたが、ユーザーに「やらない理由は?」と聞かれて確認し直したら実装可能だと分かった、という一幕もあった。**「できない理由」は自分で確認してから採用すべき**という教訓が、fugu-router の機能追加そのものにも表れている。

一方で、正直に言うと以下は未検証・未着手のまま残っている。

* **Recency 減衰は未実装**。エピソードには `ts`(タイムスタンプ)は保存されているが、近傍の重みを `exp(-age/halflife)` のように時間減衰させる仕組みはまだ入っていない。モデルの実力自体が時間とともに変わる(非定常)ことへの対策としては手つかず。
* 効果測定は「ルーティングが実際に行われ、結果が記録され、次のルーティングに反映されている」という**パイプラインが生きている**ことの確認が中心で、「fugu-router を導入した結果、導入前と比べて実コストが何%下がったか」というA/B比較のような定量評価はまだやっていない。episodes.jsonl の蓄積が浅く、統計的に意味のある比較にはまだ届いていない。

## まとめ

fugu-router は、Sakana の fugu が持つ「訓練済みコーディネータが役割ごとにモデルを割り当てる」という**構造**だけを借り、Claude の重みを学習できないという制約の中で、その**判断**を k-NN 検索と Thompson sampling によるオンラインバンディットに置き換えた。

「学習」という言葉を勾配更新に限定せず、「観測の積み重ねで方策が改善するなら目的は満たす」と割り切ったのが最初の設計判断で、実装してみて分かったのは、ルーティングという低次元・少データな意思決定には、むしろ retrieval + bandit の方が **即時適応・解釈可能性・訂正の容易さ** で分がある場面がある、ということだった。実運用でも実際にルーティング判断がタスクごとに下され、その結果が記録され、次の判断に反映されるループが回っている。

限界は限界としてそのまま書く。ターン単位のルーティングでも埋め込みベースの類似検索でもなく、深い表現学習でもない。それでも、コストとタスクの重さのトレードオフをある程度定量的に、かつ再現可能な形で扱えるようにはなった。
