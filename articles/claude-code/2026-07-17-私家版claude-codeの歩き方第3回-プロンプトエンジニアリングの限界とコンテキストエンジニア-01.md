――Karpathyの投稿が可視化した、「命令の技術」から「環境の設計」への転換

前回、コンテキストウィンドウを「AIの作業机」に例えました。机の広さには限りがあり、そこに何を載せるかで、AIが出せるアウトプットの質は大きく変わります。

では、その机の上に何を、どう並べるべきなのでしょうか。

この問いに答えようとしたとき、私たちはしばらくの間、間違った場所を探していたように思います。探していたのは「命令文の書き方」でした。しかし本当に必要だったのは、「机全体の設計」でした。

## プロンプトエンジニアリングという時代

ChatGPTが世に広まった当初、AIを使いこなす技術は「プロンプトエンジニアリング」と呼ばれていました。

どんな言葉遣いをすれば良い回答が返ってくるか。役割を与える（「あなたは優秀な編集者です」）、例を示す（Few-shotプロンプティング）、思考の手順を促す（「ステップバイステップで考えてください」）――こうしたテクニックが、SNSやブログで盛んに共有されていました。

これらの工夫は、今でも無意味ではありません。しかし、そこには一つの前提がありました。それは、「AIに渡す情報は基本的に一回分の指示文である」という前提です。

一問一答であれば、この前提は成り立ちます。良い聞き方をすれば、良い答えが返ってくる。プロンプトエンジニアリングは、いわば「話しかけ方の技術」でした。

![](https://assets.st-note.com/img/1784058227-2NMVHRlEWUCTYe3oO4LIjZX7.jpg?width=1200)

## エージェントの登場が、前提を壊した

ところが、AIが複数のツールを使い、複数の手順を踏み、時には数十分から数時間かけて一つのタスクをこなす「エージェント」として動きはじめると、この前提は崩れます。

エージェントは、一度の指示文で完結しません。タスクの進行に応じて、ファイルを読み、検索結果を取り込み、過去のやり取りを参照し、外部ツールの実行結果を受け取りながら、何度もモデルを呼び出します。そのたびに、コンテキストウィンドウの中身は組み替えられていきます。

このとき問われているのは、「どんな言葉で頼むか」ではありません。「その瞬間、机の上に何を置いておくべきか」です。過去のどの会話を残すか。どのファイルの、どの部分を見せるか。ツールの実行結果をどこまで要約するか。ルールや制約をどう提示するか――これらすべてを、タスクの各段階に応じて動的に設計する必要が出てきたのです。

一回分の指示文の工夫では、もはや対応しきれない領域に入っていました。

## Karpathyの投稿が言葉を与えた

この変化に、一つの名前を与えたのが、著名なAI研究者による2025年半ばの投稿だったと記憶しています。

投稿の趣旨は、おおよそこうでした。AIを使いこなす本質的な技術は「賢い一文をひねり出すこと」ではなく、「次の一手のために、コンテキストウィンドウへ過不足なく情報を詰め込む、繊細な技芸と科学である」と。この投稿をきっかけに、「コンテキストエンジニアリング」という言葉が急速に業界へ広がっていきました。

![](https://assets.st-note.com/img/1784058263-h9DpHq5lPe8GaMfvkOJmzSgi.jpg?width=1200)

私自身、この投稿を読んだとき、腑に落ちる感覚がありました。自分が「とあるAIプラットフォーム」で立ち尽くした経験（第1回で書いた話です）は、まさに「話しかけ方が下手だった」のではなく、「机の上に何を並べるべきか、そもそも設計していなかった」ということだったのだと。

## 二つの技術の違いを整理する

プロンプトエンジニアリングとコンテキストエンジニアリングの違いは、次のように整理できると思います。

プロンプトエンジニアリングが扱うのは、一回の指示文そのものです。言葉選び、構成、例示の入れ方――対象は「文章」です。

コンテキストエンジニアリングが扱うのは、指示文を含む、机の上全体です。会話履歴、参照ファイル、ツールの実行結果、ルール、記憶、外部データ――対象は「情報の建築」です。

前者は、優れた一文を書く才能に近いものでした。後者は、情報をどこに置き、いつ持ち出し、いつ手放すかを決める、設計の仕事です。

## なぜ経営者がこれを知る必要があるのか

技術者ではない立場からすると、これは些末な用語の違いに見えるかもしれません。しかし私は、ここに経営上の含意があると考えています。

「良いプロンプトを書ける人材」を育てることと、「自社の情報をAIに渡せる形に整理する仕組み」を作ることは、まったく別の投資です。前者は個人のスキルに依存しますが、後者は組織の資産になります。

![](https://assets.st-note.com/img/1784058317-xNy67vaSjIGPJnHs925CF3qX.jpg?width=1200)

自社のどの情報を、どのタイミングで、どういう形式でAIに渡すか。これを設計できる組織と、できない組織の差は、今後ますます開いていくように思います。そしてこの設計こそが、単発の指示文の工夫では代替できない、本当の意味での「AI活用の技術」なのだと思います。

次回は、この「机の上」に情報を詰め込みすぎることの弊害――コンテキストロットについて書きます。「全部読ませておけば安心」という発想が、なぜ危険なのかを整理していきます。
