# はじめに

最近、AIエージェントをどう使うかだけでなく、AIエージェントに任せられる形で業務をどう切り出すかを考えることが増えました。

特にソフトウェア開発の領域では、すでに「どこまで置き換えられるのか」という検証は進んでいるように見えます。コードを書く、レビューする、テストする、調査する、ドキュメントを書く、といった作業の多くは、もうエージェントに任せる前提で考え始めてもよい段階に来ていると思います。

そしてこれは、ソフトウェア開発に限った話ではありません。機械設計では生成設計やText-to-CADのような取り組みがあり、電子回路設計でも生成AIやマルチエージェントを使った設計支援の研究・実装が出てきています。万能な自動設計というよりも、設計意図を入力し、候補を作り、人間が検証するための支援が少しずつ現実味を帯びてきている、という見方です。

そんな中で、今さらではありますが、エージェントスキルを作ることの大切さを整理しておきたいと思いました。

# エージェントスキルとは何か

ここでいうエージェントスキルとは、AIエージェントに特定の作業を任せるための、再利用可能な能力パッケージです。

たとえば、以下のようなものをまとめたものだと考えると分かりやすいです。

* 作業手順
* 判断基準
* チェックリスト
* テンプレート
* 参照資料
* 実行スクリプト
* ツールやAPIの使い方

Cursor Agent Skills や Claude Skills では、こうした情報をMarkdownや関連ファイルとしてまとめ、必要なときにエージェントが読み込めるようにできます。

これは「長いプロンプトを毎回貼る」こととは少し違います。むしろ、新人向けの業務手順書やオンボーディング資料を、AIエージェントが読める形にしたものに近いと感じています。

たとえば、最初の1つは以下のような小さな `SKILL.md` からでも十分です。

Codexでリポジトリに置く場合のファイルパス例: `.agents/skills/weekly-report/SKILL.md`

> 注: `SKILL.md` の考え方や基本構造は Agent Skills の標準に沿っていますが、配置場所はツールごとに異なります。たとえば Codex ではリポジトリ用に `.agents/skills/<skill-name>/SKILL.md`、ユーザー共通用に `$HOME/.agents/skills/<skill-name>/SKILL.md` を使います。一方、Claude Code ではプロジェクト用に `.claude/skills/<skill-name>/SKILL.md`、個人用に `~/.claude/skills/<skill-name>/SKILL.md` を使います。実際に作るときは、利用するエージェントのドキュメントで配置場所を確認してください。

```
---
name: weekly-report
description: 週次レポートを作成するときに使う。売上、問い合わせ数、障害件数を確認し、先週との差分を整理する。
---

# 週次レポート作成スキル

## 入力

- 売上データ: `reports/sales.csv`
- 問い合わせ一覧: `reports/support.csv`
- 障害メモ: `docs/incidents.md`

## 手順

1. 今週と先週の主要指標を比較する
2. 20%以上変化した指標を理由付きで取り上げる
3. 未確認の値は推測せず、確認事項として残す
4. テンプレートに沿って本文を作る

## 完了条件

- 数値の出典が分かる
- 異常値の扱いが明記されている
- 次のアクションが3つ以内に整理されている
```

この程度でも、毎回の説明をかなり減らせます。大事なのは、最初から完璧なスキルを作ることではなく、「入力」「手順」「判断基準」「完了条件」を分けて書き始めることだと思います。

# 置き換わるのはソフトウェア開発だけではない

AIエージェントの適用範囲は、いわゆるエンジニアリング業務だけに閉じていません。

ホワイトカラー業務の多くも、情報を集める、整理する、判断材料を並べる、資料に落とす、関係者に伝える、といった作業の組み合わせです。これまでは自動化しづらかったロングテール業務にも、エージェントを適用する方向性が見えてきています。

いわゆる仕事の多くは、突き詰めると情報の整理なのだと思います。そして今、その情報整理の媒体のほとんどはPC上にあります。

だからこそ、AIに任せる前提で、業務の前提・判断基準・成果物をどう書き出しておくかが重要になってきていると感じます。

# それでも進まない理由

一方で、現実の業務では「技術的にはできそうなのに、なぜか進まない」ことも多いです。

その理由のひとつは、AIエージェントに任せるための形で、自分の業務を言語化できていないケースが多いからではないかと思っています。

これは単にプロンプトを書くのが苦手、という話ではありません。自分の業務を分解して、何を入力にして、何を確認して、どんな判断基準で、どんな出力を作っているのかを言葉にできない、という問題です。

本当に業務に立ち返ることができるなら、ほとんどの場合、それはエージェントスキルにできるはずです。少なくとも、PC上で行っている作業の多くはスキル化できると思います。

# 業務に立ち返れば、だいたいスキルにできる

スキル化できるものは、特殊な技術作業だけではありません。

たとえば、毎週のレポート作成、問い合わせ対応、議事録整理、調査資料の作成、請求や契約まわりの確認、データの突き合わせ、社内ドキュメントの更新なども対象になります。

たとえば「週次レポート作成スキル」であれば、参照するデータの場所、確認すべき指標、異常値の判断基準、レポートのテンプレート、提出前のチェックリストをまとめておく、という形になります。

PCに入っている情報であれば、ファイル、ブラウザ、メール、チャット、スプレッドシート、PDFなどをエージェントが扱えるようになってきています。

さらに、PCの中に最初から入っていない紙の資料や音声データであっても、今のAIエージェントは画像認識や音声認識を通じて扱えるようになっています。

つまり、焦点は「AIにできるか」だけではなく、「人間側が入力、判断、完了条件をどこまで明確に渡せるか」に移ってきているのだと思います。

# 公開されているスキルを使うだけでは足りない

もちろん、誰かが公開しているエージェントスキルを使うこと自体は良いことだと思います。

セキュリティリスクを確認したうえで、信頼できるスキルを取り入れることは、有効なショートカットになります。ゼロから作るよりも、既存の良い型を見たほうが、自分たちのスキルを設計しやすくなることもあります。

ただし、それはあくまで「他の誰かに最適化されたスキル」であることを前提にしたほうがよいです。

多くの業務は、現場ごとの決まり、やり方、使っているツール、承認フロー、判断基準に依存しています。外から持ってきたスキルが、そのまま自分たちの業務にぴったり合うことはあまり多くないはずです。

つまり、「誰かのスキルを使えば、自分では用意しなくてよい」という話ではありません。公開されているスキルは出発点にはなりますが、最終的には自分たちの業務に合わせて育てる必要があります。

# スキルを作ると何が変わるのか

エージェントスキルを作ると、毎回同じ説明をプロンプトに書く必要が少なくなります。

それ以上に大きいのは、判断基準や作業手順がチームで共有できる形になることです。これまで属人的だった作業を、レビューできるもの、改善できるもの、引き継げるものとして扱えるようになります。

スキルはAIエージェントのためだけのものではありません。新人教育や業務引き継ぎ、手順の見直しにも使えるはずです。AIに渡すために業務を整理することが、そのまま人間にとっても分かりやすい業務設計につながります。

# これから重要になるのはスキルを外出しする力

これからのエンジニアリングにおいて重要になるのは、自分の中にあるスキルや、誰かの中にあるスキルを言語化し、外出しできる力だと考えています。

個人の経験則、チームの暗黙知、現場の判断基準、毎回なんとなくやっている確認作業。こうしたものを、AIエージェントが読める形に変換できるかどうかで、業務の再現性や自動化のしやすさは大きく変わります。

エージェントスキルを作ることは、単なる効率化ではありません。自分の仕事を分解し、再利用可能な形にし、他者やAIに渡せるようにする行為です。

そう考えると、エージェントスキルを作ることは、これからの時代の重要なエンジニアリング活動なのではないかと思います。

# おわりに

AIエージェントによって、置き換わる業務はこれからさらに増えていくはずです。

そのときに大切なのは、AIに仕事を奪われるかどうかだけを考えることではなく、普段の仕事から切り出せる小さな手順をどれだけ増やせるかだと思います。

エージェントスキルを作ることは、そのための実践的な第一歩です。
