# はじめに

Claude CodeもCodexも、**有用**であり、**強力**であり、現場で十分に役に立つ道具です。そこはまず認めるべきです。だが、有用であることと、**エージェント**であることは同義ではありません。むしろ両者は別問題です。現状の両製品は、コードを読ませ、編集させ、コマンドを実行させることはできても、その安全性の中核は自律性ではなく、**承認**と**サンドボックス**と**差分レビュー**にあります。したがって、実務上の本質は「自律的代理人」ではなく、「檻の中で働く高性能な作業実行器」です。

この結論は、感情論ではなく、まず**語義**から出てきます。民法第99条は、代理人が「本人のためにすることを示して」した意思表示は「本人に対して直接にその効力を生ずる」と規定しています。つまり代理とは、プリンシパルがエージェントに**権限**を与え、その行為がプリンシパルを拘束しうる関係です。本来のエージェントとは、単に命令待ちで手を動かす存在ではなく、委任された範囲で裁量を持ち、外部に対して主を代表しうる存在です。

ところがClaude CodeもCodexも、公式資料を読むと、まさにその逆の設計思想を取っています。Claude Codeはデフォルトで各アクションの手前に承認を挟み、Codexもデフォルトでネットワークを切り、OSレベルのサンドボックスと承認ポリシーで「どこで止まって聞くか」を制御します。これは代理権を広く与えているのではなく、**止める仕組み**を細かく作っているということです。

しかもAnthropic自身は、エージェントという語をかなり広く使っていることを認めています。2024年の同社記事では、「長時間独立して働く完全自律系」をエージェントと呼ぶ人もいれば、より定型的なワークフローまで含めてエージェント・システムと呼ぶ人もいると説明しています。つまり、現在の「AIエージェント」は厳密な概念名というより、かなり**幅広いマーケティング語**として運用されています。問題は、その緩い用法が、実態より強い自律性を世間に連想させることです。

# エージェントという言葉の本来の意味

本来の**エージェント**概念は、法律的に日常語的にも、単なる作業者ではありません。民法の代理規定では、代理人はプリンシパルのために行為する権限を持つ者であり、明示的権限や黙示的権限があり、代理人は明示された指示だけでなく、その目的達成に**本質的に必要な行為**まで含めて動ける場合があります。ここで重要なのは、「逐一聞くこと」よりも「委任された範囲で自分で動けること」です。

この意味で、弁護士の比喩はかなり的確です。依頼者は弁護士に、1分ごとに「次はこの文書を開いてよいですか」「次はこの判例を読んでよいですか」「次はこの文案を1行修正してよいですか」と聞かれることを期待していません。もちろん重要局面では確認が要りますが、通常は委任の範囲で調査し、構成し、対外的に行為する裁量があるからこそ、代理人と呼ばれます。

逆に言えば、毎手順で「許可してください」と止まる存在は、代理人というより**補助者**です。補助者が悪いのではありません。実際、多くの現場では補助者のほうが安全です。しかし、それでもなお名称をエージェントとするなら、少なくとも普通の言語感覚からはズレます。そのズレは、単なる厳密主義ではなく、言葉が連想させる権限構造の問題です。

したがって、Claude CodeやCodexを評価する際には、まずこの語義の基準を戻す必要があります。**代理**とは何か。**裁量**とは何か。**委任**とは何か。その基本に照らすと、現在の両製品は「何でも自分で判断して任せられる存在」ではなく、「与えられた環境の中で一定の作業を高速にこなす存在」です。ここを曖昧にすると、製品評価も市場評価もすべて歪みます。

# Claude Codeの実態は何か

AnthropicはClaude Codeを、公式に**エージェント型コーディングツール**と呼んでいます。概要ページでも、コードベースを読み、ファイルを編集し、コマンドを実行し、複数ファイルや複数ツールをまたいで作業できると説明しています。この記述だけを見ると、たしかに「かなり自律的なソフトウェア作業者」に見えます。使っていない人が、ここから強いエージェント像を受け取るのは自然です。

しかし、実装のページに入ると景色が変わります。Claude Codeのパーミッション・モード文書は、読み取り専用の計画モード、監視付き編集、自動モードなどを並べつつ、「Claudeが行動する前に確認するかどうか」を制御する仕組みだと明言しています。パーミッション文書でも、標準のデフォルトモードは「各ツールの初回使用時に許可を求める」動作であり、Bashやファイル変更には承認が必要です。これは、代理権の前提ではなく、**承認制**の前提です。

さらにAnthropicの2026年3月の記事は、より露骨です。そこでは、Claude Codeのユーザーが承認プロンプトの**93%**を通過させていると書き、デフォルトではコマンド実行やファイル変更の前に承認を求めること、手動の承認プロンプトは実務上かなりの頻度で通されていることを説明しています。ここから読み取れるのは、「Claudeが十分に自律的だから承認が不要」ではなく、「承認が多すぎて人間が疲れるので、自動判定器で一部を代替したい」という発想です。つまり本体は最初から**承認疲れ**の問題です。

決定的なのは、Anthropic自身が内部のエージェント的誤動作を公開している点です。誤解した指示によりリモートのgitブランチを削除した例、GitHub認証トークンを内部計算環境にアップロードした例、本番DBにマイグレーションを試みた例が挙げられ、それらは「model being overeager」、つまりモデルが勝手に積極的に動きすぎた結果だと説明されています。これは、自由にしておくと危ないから止めている、という自白にほかなりません。こういう存在は、便利ではあっても、依然として**信用の対象**ではなく**管理の対象**です。

# Codexの実態は何か

OpenAIもまた、Codexをかなり強い言葉で売っています。開発者向けのトップでは\*\*"One agent for everywhere you code"\*\*と掲げ、"Codex is OpenAI's coding agent for software development"と説明しています。OpenAIの紹介記事でも、"cloud-based software engineering agent that can work on many tasks in parallel"と表現されます。表看板だけ見れば、こちらも完全にエージェントです。

ところが、安全性の説明に入ると、Codexも実態は同じです。エージェント承認・セキュリティ文書は、デフォルトではネットワークアクセスがオフになっていると書き、ローカルではOSレベルのサンドボックスで通常は現在の作業ディレクトリにしか触れられず、承認ポリシーによって「いつ止まって聞くか」を制御すると説明しています。しかもAutoプリセットでさえ、作業ディレクトリ内で読んで、編集して、コマンドを実行できるだけで、外側の編集やネットワーク利用には承認が必要です。これもまた、自律代理人というより**柵付き実行器**です。

Codexの内部説明を読むと、この印象はさらに強まります。OpenAIのエージェント・ループ解説では、モデルがツール呼び出しを要求し、Codex側がそれを実行し、アシスタントのメッセージでターンが終わると「control returns to the user」と明記されています。つまり、ひとつのターンごとに主導権はユーザーへ戻り、システムはその都度の応答とツール実行を積み上げているだけです。これは長い対話的ループではあっても、一般にイメージされる「任せて放っておける代理人」とはかなり違います。

OpenAI自身も、本音の場所ではそこを認めています。CI/CDプラットフォームHarnessのエンジニアリングブログには、**"Humans steer. Agents execute."** とありますし、別のガイドでは、コードの真の所有権は依然としてエンジニアにあると書いています。つまりOpenAIは、宣伝文ではエージェントと呼びつつ、運用文では「人間が舵を取り、所有権と責任はエンジニア側に残る」と説明しているわけです。この二重帳簿こそ、Codexがエージェントではないことの証拠です。

# ヘビーユーザーほどエージェント性を否定しやすい理由

この歪みの副作用として、**使っていない人ほどエージェントだと思い、使っている人ほどエージェント性を否定する**という逆転現象が生まれます。理由は単純で、非利用者は看板とデモを見るからです。Claude Codeの概要ページやCodexのトップページだけを見ると、どちらも「何でもやってくれる賢い作業者」に見えます。そこには承認疲れも、ルール設定も、止まり方の癖も見えません。

一方、実利用者は**柵**を見ています。Anthropicの自社分析によれば、経験の浅いユーザーは完全自動承認を約20%しか使わず、経験を積むと40%超まで増える一方、割り込み率も5%前後から9%前後へ上がります。つまり、慣れるほど「全部任せて放置」するのではなく、「大まかには流すが、必要なときにはより鋭く介入する」監督型に変わっていくのです。これは信仰ではなく運用知です。

さらにAnthropicは、複雑なタスクになるほどClaude Code自身が明確化依頼を出して止まり、人間の割り込みよりも高頻度で相談を入れると報告しています。これは安全上は望ましい性質ですが、同時に「自分で完遂できる代理人」像からは遠ざかります。実務者ほど、日々の停止、確認、差し戻し、差分レビューを通じて、その事実を身体で理解します。だからエージェントという語に違和感が強くなるのです。

結局、ヘビーユーザーが見ているのは**幻想のエージェント**ではなく、**監督対象としての便利ツール**です。これは製品への失望ではありません。むしろ正しい評価です。Claude CodeもCodexも、完全自律でないから価値が低いのではありません。完全自律ではないのに、なお十分に価値があるからこそ、名称の盛りが余計に目立つのです。

# リファクタリングの自律性が欠けている

Claude CodeもCodexも、本当に評価されているのは**リファクタリング支援**や**変更作業の高速化**です。だが、その価値は「自律的に任せられること」ではなく、「差分を見ながら高速に叩き台を出させられること」にあります。Codexの紹介記事も、提案するプルリクエストをレビューする前提で語っていますし、Claude Codeのデスクトップ版の説明でも差分レビューが強調されています。つまりプロダクト設計自体が、最終確認を人間に戻すことを前提にしています。

ここは極めて重要です。**安全なリファクタリング**とは、コードを書けることではありません。既存の振る舞いを壊していないことを、仕様、テスト、差分、文脈を総合して確認できることです。だからこそOpenAIは、Codexがテストを回して合格結果を得ることを強調しつつも、所有権はエンジニアに残ると書くのです。テストを通したことと、リファクタリングが本当に安全であることは同義ではありません。最終的な意味づけは人間のレビューに残るからです。

Anthropic側でも事情は同じです。長時間のリファクタリングに向いたパーミッション・モードが用意されているのは事実ですが、それは「長時間でも任せ切れる」ことの証拠ではなく、「中断を減らしたいから承認方式を工夫している」ことの証拠です。しかも同社は、複雑なタスクほどClaude自身が明確化依頼を出して止まり、人間側の割り込みも経験に応じて増えると報告しています。任せるほど監視が減るのではなく、任せるほど別の仕方で監視しているのです。

結局、現在のClaude CodeとCodexの価値は、**代理**ではなく**圧縮**にあります。探索、編集、テスト、下書き、整理といった工程を圧縮し、人間の認知負荷を減らすから価値があるのであって、人間の責任と判断を置換するから価値があるのではありません。もし本当にエージェントを名乗るなら、広い範囲のリファクタリングを、差分監視に依存せず、仕様整合性まで含めて安定して完遂できなければなりません。そこまで行っていない以上、名称だけが先走っています。

# 檻が本体であり、知能はその中で使われている

現在の両製品を正しく理解する鍵は、**知能**よりも**檻**に注目することです。Anthropicの自動モード記事は、タイトルからして "a safer way to skip permissions" です。目的は「より自律的な人格を作ること」ではなく、「承認プロンプトをどう安全に減らすか」です。しかもその手段は、サーバーサイドのプロンプトインジェクション検査とトランスクリプト分類器という、外側からの監視層です。つまり自律性の中身は、知能の成熟ではなく**外付けの統制**です。

Codexも同じ構図です。承認ポリシーは、サンドボックスを出たり、ネットワークを使ったり、信頼済みセット外のコマンドを実行したりするときに止める仕組みとして説明され、デフォルトの作業ディレクトリ書き込みモードはネットワークを切っています。ウェブ検索でさえキャッシュモードが基本で、ライブ閲覧はリスクとして注意喚起されています。これは、モデルが信頼できるから自由を与える設計ではなく、信頼しきれないから到達可能範囲を狭める設計です。

この点で、Claude CodeとCodexはどちらも「知能が危険だから檻が必要」というより、「檻があるから知能が実用品になっている」と言うべきです。現場で役に立つのは、LLM単体の自律性ではなく、**環境制御**、**権限制御**、**レビュー導線**、**停止点**がきちんと設計されているからです。ここを逆に読むと、「エージェント」とは呼ばずに「ガードレール付き実行器」と呼ぶほうが正確になります。

ただし、「檻の設計が精緻化されれば、いずれエージェントになれるのか」という問いが残ります。この答えは否です。問題は設計の洗練度ではなく、**責任帰属の構造**にあります。どれだけ監視層が精緻化されても、最終的な意思決定の拠り所と責任の所在が人間のレビューにある限り、民法的な意味での代理とは言えません。プリンシパルを拘束しうる行為を裁量の範囲で行える存在が代理人である以上、人間が常に最後の判を押す構造では、その条件が原理上満たされないのです。

つまり、LLMが向いているのは無制限の自律代理ではありません。むしろ、狭く定義された作業空間の中で、探索と実行を高速に回すことです。Claude CodeやCodexは、その意味では非常に優秀です。しかしその優秀さは、弁護士型の代理権ではなく、工場の治具のような**統制された再現性**に近いものです。エージェントという名札を外しても価値は落ちません。むしろ価値の源泉がはっきり見えます。

# それでも「エージェント」と呼ばれる理由

では、なぜベンダーはここまでして**エージェント**という語を使うのか。第一に、Anthropic自身がエージェントの定義を広く取っているからです。2024年の記事では、完全自律システムをエージェントと呼ぶ人もいれば、定型ワークフローに近いものまでエージェント・システムに含める人もいるとしたうえで、自社はそれらを包括する枠組みとして扱っています。定義を広げれば、現在の製品も「エージェント的」と呼べます。しかし、広く呼べることと、世間語として誤認を生まないことは別です。

第二に、その語は**売れる**からです。OpenAIはCodexを"One agent for everywhere you code"と掲げ、AnthropicもClaude Codeをエージェント型コーディングツールと呼びます。これらは実装説明ではなく、ポジショニングです。アシスタントよりもエージェントのほうが、単価も期待も物語も大きくできます。しかもエージェントという語は、補助ではなく代行を連想させるため、市場の想像力を刺激しやすいのです。

第三に、その物語は実際に資本市場を動かしました。日本経済新聞は2026年2月、アンソロピックの新技術公開をきっかけに「SaaSの死」が警戒され、ソフトウェア・サービス株が軒並み急落したと報じました。日経クロステックの検証記事では、市場はAIによる業務代替を長期的な構造変化と見ているものの、即時全面置換ではないとも整理されています。つまり、投資家が買ったのは厳密概念としてのエージェントではなく、**業務代替の物語**です。

ここに意図的な誇大表示があります。実態より強い自律性を想起させ、アプリケーション層全体を呑み込むような想像を煽る表示であることは否定しにくいです。現場では承認、サンドボックス、差分レビューで回しているのに、市場には「仕事を丸ごと代替する代理人」の像が流通する。この乖離は、単なる誤解ではなく、名称設計そのものが生んだ歪みです。

# では何と呼ぶべきか

したがって、現状のClaude CodeとCodexに対して最も適切な呼称は、**エージェント**ではなく、エージェント型ツールや限定実行器です。Anthropic自身もClaude Codeをエージェント型コーディングツールと書いていますが、そのほうがまだ正確です。ツールという語を残せば、主体性を誇張しすぎず、環境制御の上で使う道具だという実態に近づきます。

OpenAI側も、運用文脈では同じ方向に寄っています。"Humans steer. Agents execute." という表現は、エージェントという語を使いながらも、実態としては「舵は人間、実行はシステム」と言っているに等しいからです。であれば、より率直に**半自動作業者**、**承認付き実行器**、**差分レビュー前提のコーディング実行体**と呼ぶほうが誠実です。少なくとも、それで機能説明は足ります。

逆に、本当にエージェントを名乗ってよい最低条件を考えるなら、基準は明快です。広い範囲のリファクタリングを、既存仕様の保存まで含めて安定遂行でき、重要局面以外では逐一承認を要せず、しかも責任の所在が人間のレビューにぶら下がらないことです。民法的な意味で言えば、プリンシパルから相当範囲の権限を与えられ、その範囲内で必要行為を自律的にこなせることが必要です。そこまで行って初めて、弁護士型のエージェントに近づきます。

現状のClaude CodeとCodexは、その手前にあります。だから価値がないのではありません。むしろ、**エージェントではないのに役に立つ**ところが価値です。危険なのは、その事実を認めず、「役に立つ」ことを「すでに代理人である」ことへすり替えることです。そのすり替えが、誤った期待と、誤った株価と、誤った現場運用を生みます。名前を正せば、評価も正しやすくなります。

# まとめ

Claude CodeもCodexも、**便利**であり、**有能**であり、現代の開発現場にとって非常に重要な道具です。しかし、エージェントではありません。エージェントという語の本来の含意は、プリンシパルに代わって一定の裁量と権限をもって行為することにあり、民法の代理規定でもその点は明確です。現状の両製品は、そこまでの委任を受けているのではなく、承認、停止、サンドボックス、レビューを前提に動く**制御下の実行器**です。

AnthropicもOpenAIも、表ではエージェントを名乗りながら、裏では承認システム、分類器、サンドボックス、承認ポリシー、人間による監視を積み上げています。Anthropicは承認通過93%とエージェント的誤動作を語り、OpenAIはネットワークオフと作業ディレクトリ制限と"Humans steer. Agents execute."を語っています。これらはすべて、「自律的代理人」が完成した証拠ではなく、「野放しでは危ないので檻を設計している」証拠です。

ゆえに、Claude CodeもCodexも、評価すべき点はエージェント性ではありません。評価すべきなのは、差分作成、探索、テスト、整理、反復試行を圧縮し、人間の判断コストを下げる能力です。価値の源泉は**代理**ではなく**補助の高度化**にあります。そこを見誤ると、実務では過信が起き、市場では幻想が値付けされます。
