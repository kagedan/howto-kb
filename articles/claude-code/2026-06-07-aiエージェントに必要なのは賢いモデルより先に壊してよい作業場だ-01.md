AIエージェントの話になると、すぐモデル比較になる。

どのモデルがコードを読めるのか。どのモデルが長いコンテキストを扱えるのか。どのモデルが修正を一発で通せるのか。その比較自体は避けられない。だが、実務で胃が痛くなるのは「モデルが少し鈍い」瞬間ではない。

本当に怖いのは、そこそこ賢いエージェントが、壊してはいけない場所を自信満々に触ることだ。

人間がチャット欄で相談しているだけなら、間違いは文章で済む。ところが、いまのエージェントはファイルを読み、コマンドを実行し、git を触り、外部ツールにつながる。もう返答するAIというより、作業場を持った実行主体に近い。

だから、2026年の agentic development で差がつく抽象化は、プロンプト術ではなく実行環境の設計だと思う。

## MCPは「手」を増やすが、作業場までは作らない

MCP は面白い。AIアプリケーションが外部データやツール、ワークフローへ接続するための共通口になる。モデルに閉じた賢さではなく、モデルが現実のシステムへ触れるためのインターフェイスを整える発想だ。

これはたぶん、かなり正しい方向である。

エージェントが仕事をするには、リポジトリ、issue、DB、ドキュメント、ブラウザ、社内ツールのような環境に触れなければならない。テキストだけで世界を理解しているAIは、どれだけ賢く見えても、最後の一歩で手が届かない。

ここで勘違いすると危ない。

接続できることと、安全に作業できることは違う。

MCP の transport ひとつ取っても、stdio だけでなく HTTP 系の接続が出てくる。localhost へ bind するのか、Origin を検証するのか、認証をどう扱うのか。これは「AI便利ツール」の話ではなく、普通にプロトコルとセキュリティの話である。

MCP はエージェントに手を増やす。だが、その手で何を触ってよいか、失敗したときにどこで止まるか、作業後に人間がどう確認するかまでは決めてくれない。

手だけ増やしたエージェントは、器用なだけで怖い。

## サンドボックスは檻ではなく、作業台である

AIに自由を与えるには、まず閉じた場所を作る必要がある。

この言い方は一見矛盾している。だが開発者なら直感的にわかるはずだ。新しいライブラリを試すとき、いきなり本番リポジトリの main でやらない。ブランチを切る。検証用ディレクトリを作る。Docker や VM や一時環境を使う。壊してよい場所を作るから、大胆に試せる。

エージェントでも同じだ。

サンドボックスは、AIを閉じ込めるためだけの仕切りではない。人間が失敗を回収できる単位を作るための作業台である。

ここを「制限」とだけ見ると、導入が妙に窮屈になる。権限を絞り、ファイルアクセスを止め、コマンド実行を全部承認制にする。必要な場面はあるが、それだけではAIは仕事をしにくい。

大事なのは、自由に試してよい範囲を明確にすることだ。

たとえば、ある機能の修正をエージェントに任せるなら、作業ディレクトリ、実行してよいコマンド、変更してよいファイル種別、使ってよい認証情報、ネットワークの可否を先に決める。失敗しても捨てられる branch にする。作業ログとテスト結果を残す。

この環境なら、エージェントは遠慮なく動ける。人間も安心して読める。

逆に、境界のない自由は速そうに見えて遅い。エージェントがどこまで読んだのか、なぜそのファイルを変えたのか、どの副作用があり得るのかを、あとから人間が全部追うことになるからだ。

## gitは成果物ではなく観測面になる

エージェントがリポジトリを触る時代になると、git の意味も少し変わる。

branch、commit、diff、pull request はこれまでも大事だった。だが人間が手で作った差分では、暗黙の文脈が残っていた。「この人がこの issue を読んで、この設計を知っていて、このファイルを触った」という前提でレビューできた。

エージェントの作業では、その前提が薄くなる。

だから git は単なる成果物置き場ではなく、観測面になる。どの branch で作業したのか。commit の粒度はどう分かれているのか。どのコマンドを走らせたのか。テストは何を保証したのか。どこに不確実性が残っているのか。

この情報が差分の近くにないと、レビューは苦しくなる。

最終的なコードが正しそうに見えても、その途中でエージェントが何を見落としたのかはわからない。テストが通っていても、必要なテストを走らせたとは限らない。説明が流暢でも、判断の根拠があるとは限らない。

AI時代の git 運用で必要なのは、立派な履歴ではなく、回収可能な履歴である。

一つのタスクが一つの作業単位に閉じている。差分が大きくなりすぎたら止まる。検証コマンドが残る。失敗したコマンドも隠さない。人間が「これは見れば判断できる」と思える形にする。

エージェントの能力を上げるほど、この地味な観測面が効いてくる。

## repo instructionsはプロンプトの代替ではない

AGENTS.md、CLAUDE.md、hooks、skills のような仕組みを、単なる便利メモとして扱うのはもったいない。

あれはチームの作法を、エージェントが読める形に固定するための層である。

毎回チャット欄に「このプロジェクトではこうしてください」と書くのは弱い。人間が忘れるし、表現が揺れるし、チームで揃わない。さらに悪いことに、うまくいった指示が個人のローカルなプロンプト資産になってしまう。

残すべきなのは、プロンプトの名文ではない。

このリポジトリではどのコマンドで検証するのか。どのディレクトリ境界を守るのか。どのファイルは触ってはいけないのか。どんな変更は事前に相談するのか。UI変更では何を確認するのか。コミット前に何を見るのか。

こういう作法は、コードと同じ場所に置いたほうがよい。

エージェントは毎回違うモデルかもしれない。ツールも変わる。だが、リポジトリ側に作業文化が書かれていれば、少なくとも入口は揃う。これは「いい返事を引き出す」ための文章というより、作業の再現性を上げるための設計図だ。

プロンプトは一回の会話を強くする。repo instructions はチームの作業場を強くする。

## 導入は強いモデルから始めないほうがいい

AIエージェントを導入するとき、つい強いモデルを選びたくなる。

わかる。精度が高いほうがよい。長い文脈を読めるほうがよい。ツール利用がうまいほうがよい。それでも、最初に見るべき場所はそこではない。

まず、小さな sandbox を作る。

次に、限定されたタスクを渡す。

そして、検証コマンドを固定する。

たとえば「全体を見て改善して」ではなく、「このテストが落ちているので、対象ディレクトリ内で原因を探し、最小差分で直し、実行結果を残す」と頼む。最初の成功体験は、AIが巨大な機能を作ったことではなく、人間が安心してレビューできる差分が出たことに置く。

この順番を間違えると、AI導入は妙な博打になる。

うまくいった日はすごく速い。失敗した日は、なぜ失敗したのかわからない。誰かが「やっぱりAIは信用できない」と言い、別の誰かが「使い方が悪い」と言う。そこで議論がモデルの賢さに戻ってしまう。

ここで問うべきは環境だ。

そのエージェントは、壊してよい場所で作業していたか。人間が回収できる単位で止まれたか。検証結果を残したか。不確実性を隠さなかったか。チームの作法を読んでいたか。

この条件が揃っていないなら、強いモデルを入れても危ない。むしろ危ないものが速くなる。

## エージェントを同僚にする前に、机を用意する

AIエージェントを同僚のように扱う、という言い方がある。

悪くない比喩だと思う。ただ、同僚に仕事を頼むとき、人間は作業環境も渡している。リポジトリへの権限、issue の文脈、ローカルの立ち上げ方、レビューの基準、やってはいけないこと。新しく入った人が最初に苦労するのも、たいていコードそのものより、この作業場の理解である。

AIだけが例外になるはずがない。

賢いモデルを選ぶことは必要だ。だが、モデルの知能は作業場の上で初めて効く。MCP は接続の標準になる。sandbox は失敗の境界になる。git、logs、test outputs は観測面になる。repo instructions はチームの作法を固定する。

この四つをまとめて設計して初めて、エージェントは実務の中でまともに働ける。

AIにコードを書かせる時代の最初の仕事は、AIに命令することではない。AIが失敗できる机を作ることだ。

その机がないまま賢さだけを求めると、開発は速くならない。ただ、回収不能な差分が速く届くだけである。
