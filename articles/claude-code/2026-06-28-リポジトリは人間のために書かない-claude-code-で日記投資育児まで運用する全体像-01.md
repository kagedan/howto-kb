私は自分の人生を、1つの Git リポジトリと、Claude Code による複数の AI エージェントで運用しています。日記、自己分析、投資、趣味、育児。どれも、その困りごとを解くために個人開発したアプリです。育休でまとまった時間ができたので、それらを別々にではなく1か所にまとめて動かすようにしました。そして、**このリポジトリの第一読者は、私ではなく AI です。**

人間のために書くのをやめた瞬間、日記も投資も育児の記録も、AI が横断して読める1つの知識ベースに変わりました。バラバラだったアプリが、ここで初めてつながります。

この記事はその全体像の地図です。個別の作り方は別記事に分けるので、ここでは「なぜ1つに束ねるのか」と「どう動いているのか」だけを書きます。

## 動機：私はいつも「自分を含む家庭の困りごと」から作る

私がものを作る起点は、だいたい自分か家族の困りごとです。

* 日記が三日坊主になる
* 投資の判断に自信が持てない
* 見るアニメが尽きる
* 育児の情報がノイズだらけで結論にたどり着けない

最初はこれらを、それぞれ独立した個人開発アプリとして作っていました。けれど数を作るうちに、別々のアプリだと「自分」という一貫した文脈が毎回分断されることに気づきました。投資ツールは私の価値観を知らないし、日記アプリは私の思考の癖を知らない。同じ私が使っているのに、文脈が共有されない。

ここで私の考え方の癖が出ます。私は問題を前にすると、**まず条件を分解して、いちばん筋の良いルートを探します。**

「便利ツールをたくさん並べる」のか「1つの土台にまとめて載せる」のか。後者なら文脈を1か所に集約できて、作るほど基盤が強くなる。複利が効くのはどちらか、と考えて、1つのリポジトリに人生を入れることにしました。

## 土台：ドキュメントは「人間のため」でなく AI のために書く

ここが一番の前提で、たぶんいちばん人と違うところです。

チーム開発なら、ドキュメントは人間同士の共有のために書きます。でも個人開発に、その読み手はいません。私のリポジトリの第一読者は AI です。だから私は、ドキュメントを人間が読みやすいように設計していません。AI が読みやすく・探しやすい構造で書いています。

具体的には、3つのことをしています。

* **1ファイル1事実**に分ける
* frontmatter に検索可能なメタデータを持たせる
* ファイル名は「あとで AI が探せるか」を最優先にする

散文や装飾、線形に読み下す前提の文章は削ります。人間が中身を読みたくなったら、ファイルを開くのではなく AI に聞きます。

ドキュメントは AI の知識ベースで、人間用の UI は対話、という分担です。

リポジトリの骨格はこうなっています。

```
life/
├── journal/      # 対話式の日記（年/月/日.md）
├── persona/      # 自己分析（dimensions/=横断・chapters/=時系列・insights/=自動抽出）
├── projects/     # 各開発プロジェクトのメタ情報
├── ideas/        # アイデアの種
└── .claude/
    ├── skills/            # 用途別エージェント（/journal, /weekly-review …）
    ├── rules/             # エージェントに注入する振る舞いのルール
    └── config/context.yaml  # 全スキルが共有するコンテキスト
```

ファイルの先頭には、AI がクエリできるメタデータを必ず付けます。たとえば日記なら、こうです。

```
---
date: 2026-03-12
mood: 4
tags: [開発, アイデア]
highlights:
  - 新しいプロジェクトを構想した
---
```

この発想を人生に広げると、日記・自己分析・価値観・プロジェクトの記録が、すべて AI が読める構造化された知識として1か所に積み上がります。これが、私が人生を運用するときの土台です。いわゆるセカンドブレインやライフログを、人間ではなく AI が読む前提で組み直したもの、と言ってもいいかもしれません。

## 何を出して、何を出さないか

先に線引きを書いておきます。**「どう作ったか」は共有できて、「私の個人データそのもの」は伏せる。** この記事もこの線引きの内側で書いています。

| 出すもの | 出さないもの |
| --- | --- |
| アーキテクチャ・設計思想 | 日記の中身 |
| 考え方・思考様式 | 家族の情報 |
| 汎用化したスキル / CLAUDE.md | 実際の保有資産 |
| ディレクトリ構成 | 自己分析の本文 |

ひとつ補足すると、公開するスキルや CLAUDE.md は、リポジトリから切り出して汎用化したものです。普段の記録は AI のために書きますが、人に渡すものは人間が読んで使える形に直します。両者は別物として扱っています。

## 動かし方：Claude Code のスキルを多エージェントとして動かす

この基盤の上で、用途ごとのエージェント（スキル）が動きます。日記を書くエージェント、自己分析を深めるエージェント、週次で全体を振り返るエージェント、トレンドを集めるエージェント。それぞれが同じ知識ベースを参照します。

たとえば日記を書くエージェントは、その日の出来事を聞く前に、私の思考の癖をまとめたルール（「私の自己評価は実際の量より控えめに出る」など）を読み込んでから質問を組み立てます。週次の振り返りエージェントは、日記とプロジェクトの進捗と体調データを横断して見ます。**1つのエージェントの出力が、別のエージェントの入力になる。** これが「多エージェントで運用する」の中身です。

そして全部に共通して、私の「条件を分解して最適ルートを探す」やり方が、エージェントへの指示やルールとして埋め込まれています。私という人間の思考の癖が、システムの挙動として再現される。ここが、汎用の AI ツールをそのまま使うのとの違いだと思っています。

## 4つのモジュール（自己理解・お金・趣味・育児）

この基盤の上に、いまのところ4つの領域で個人開発したアプリが載っています。それぞれ独立した記事にするので、ここでは地図として置きます。

* **自己理解**：対話式の日記と、自分のペルソナ（価値観・思考の癖・感情のパターン）を分析する仕組み。これは差別化の核なので、次の節でもう少し書きます。
* **お金（投資）**：投資判断を支えるために作った、市場データの取得基盤＋「攻め」（構造変化を早く見抜く）と「守り」（高配当を淡々と積む）の三層構成。
* **趣味**：アニメの評価データから、自分でも言葉にできていない好みの傾向をどう引き出すか。その推薦の仕組み。
* **育児**：エビデンスベースの育児情報を、ノイズを削って結論に最短で着く形に整えるアプリ。

4領域がバラバラに見えて1つにまとまるのは、AI が読む同じ知識ベースと、条件分解という思考様式を共通にしているからです。たとえば自己分析でためた私の傾向は、ほかの領域からも同じ知識ベースとして参照できます。ただ4つのアプリが並んでいるのとは、そこが違います。

## ペルソナ：AI に自分自身を分析させる

4つの中でいちばん力を入れているのが、この自己理解です。ここがたぶん、世の中の「AI 秘書」と最も違うところでもあります。

多くの AI 活用は、予定やタスクといった「自分の外側」を扱います。私が個人開発で自動化しているのは、その手前にある\*\*「自分自身を AI に正しく理解させること」\*\*です。対話式の日記を入り口に、自分の価値観・思考の癖・感情のパターンを、日記の引用や日付といったエビデンス付きで少しずつ構造化していきます。これが私のペルソナです。

ここで効いてくるのが補正ルールです。AI は放っておくと、人間を都合よく平板に要約します。私が「いつものこと」と言っただけで「軽い一日」と決めつけたり、条件を分解して決めたつもりでも、その裏で言語化していない基準を並行で見ていたことを見落としたり。だから私は、自分の癖を補正ルールとして書き出し、AI の読み違いを先回りで直しています。

このペルソナと補正ルールは、自己理解だけのものではありません。日記の深掘りの質を上げ、ほかのエージェントが私の文脈を踏まえて動くための足場にもなります。

作り方——ペルソナをどう構造化し、どんな補正ルールで AI の読み違いを直すか——は、今後の記事で書く予定です。

## 正直なところ

私のやり方は「実用に必要な分だけ掴んで、横断して束ねる」です。完全に理解してから作るのではなく、動かすのに要る分で前に進む。だから各領域の理論的な深さ——深層学習や金融工学の最深部——では勝負しません。掴んだのはここまで、という線を正直に引いて書きます。私が出せる価値は、深さではなく横断の設計の方にあります。

それと、全部が常にうまく動いているわけでもありません。私の開発には波があります。新しい仕組みを作っている時は楽しくて進みますが、出来上がったものを整える「整地」の時期は退屈で、手が止まります。上の4領域にも、いま動いているものと、作ったまま止まっているものがあります。**止まっている理由は「飽きたから」で、それ以上の物語はありません。**
