最近ちょくちょく耳にするようになった**ハーネスエンジニアリング**について、チームに共有するのも兼ねて記事にまとめようと思います。

今回はハーネスエンジニアリングの技術的な内容や実装方法ではなく、実際の具体例に例えて説明することを目的としてます。

## LLMについて

LLM自体はとてもシンプルなものです。  
入力のプロンプトに対して、AIが考えた結果をアウトプットしてくれるだけのものになります。  
つまり、与えられた文字列に対してAIが考えた文字列を返すだけです。

## AIエージェントの仕組み

このLLMを使い、目標達成に向けて計画を立てて、ツールを使い、目的を完遂するものがAIエージェントです。  
これを実現するために、非常に大事な要素があります。これが「フィードバックループ」です。

```
LLM ➡︎ 出力 ➡︎ 監視 ➡︎ フィードバック ➡︎ 再実行
```

このループを作ることで、AIは

```
試す ➡︎ 結果を見る ➡︎ 修正する ➡︎ 再度試す
```

という行動を繰り返します。この行動を繰り返すことで、最終的に目的を完遂しているのです。  
このフィードバックループを実現するには、監視する仕組みや、継続的にAIを動かすためのメモリ管理などの仕組みも必要です。  
この辺りも含めて、AIを制御する仕組みを作る技術が**ハーネスエンジニアリング**です。

## ハーネスエンジニアリングとは？

改めてハーネスエンジニアリングとは、**AIを制御する仕組みを作る技術**です。  
特に重要なのが、

* フィードバック：出力結果をLLMに返して次の行動に活かす仕組み
* 監視：実行状況やエラーを検知する仕組み
* 制約：暴走を防ぐためのガードレール

です。この3つを設計することでAIを目的に向けて動かすことができます。  
※この記事では、フィードバック・監視・制約をどのように実装するかは触れません。制御する仕組みで大事であることを認識いただければと思います。実装される方は是非調べてください。

### 色々なハーネスエンジニアリング

ハーネスはいろんな文脈で使われています。が、根本はどこでも一緒で**AIを制御する仕組みを作る技術**です。そして制御を実現するための技術として、**フィードバック、監視、制約** が共通して使われています。

使われ方もレイヤーによって様々です。下記の図のようにいくつかレイヤーがあります。内側に行くほどLLMに近く、外側に行くほど人間や運用プロセスに近いハーネスになります。ここから具体例と一緒に見ていきます。

#### 具体例① Claude Code（LLMをエージェント化するハーネス）

これは最初に紹介した、LLMをAIエージェントにするハーネスです。  
Claude Codeの利用者から「バグを直して」と受けるとClaude Codeは計画を立て、ファイルを読み、コードを書き、テストを実行し、結果を見て修正する。というループを自律的に回します。  
このループを回しているのがハーネスです。  
ハーネスの3要素で見ると、

* フィードバック：テスト結果やLintエラーをLLMに戻して、次のアクションを決めさせる
* 監視：ツール実行の結果、ファイル変更、トークン消費量などを継続的にチェックする
* 制約：実行できるコマンドの許可リスト、ファイル編集時の確認プロンプト、権限モード

LLM自体は文字列から文字列を返すだけですが、この3要素を整えることで目的達成に向けて自律的な動きができるようになります。

#### 具体例② 独自実装のAIエージェント(オーケストレーター型)

LangChain / LangGraph / Claude Code のサブエージェント機能などを使って、独自のAIエージェントチームを組むパターンです。  
例えば「PRを作る」というタスクを考えます。一回のLLM呼び出しでは難しいので、役割を分けます。（実際は一回で問題ないケースが多いですが）

* オーケストレーター：全体の指揮役。タスクを分解して各ワーカーに振る
* リサーチャー：該当コードを読んで変更箇所を洗い出す
* 実装エージェント：コードを書く
* レビュワー：Lint、テスト、コーディング規約に照らして指摘する

ここで問題になるのが、各エージェントがバラバラに動くとお互いの状況がわからないことです。実装エージェントがどこまで書いたのか、レビュワーが何を指摘したのかを、オーケストレーターが把握できないと指揮できません。

これを解決するのが進捗ファイルです。このファイルは各エージェントから参照され、エージェントは作業結果と状態(作業中or完了？など)をファイルに吐き出します。他のエージェントも同じファイルを読み込んで次の行動を決めます。ファイルが共通の状況確認になります。

ハーネスの3要素で見ると：

* フィードバック：レビュワーの指摘を実装エージェントが受け取り、修正する
* 監視：オーケストレーターが進捗ファイルを見て全体の状況を把握する
* 制約：各エージェントに割り当てる役割とツール権限(例：レビュワーは読み取りのみ)

ここでのハーネスとは、エージェント同士をどう連携させ、どう進捗を同期させ、どう暴走を防ぐかという仕組みになります。

#### 具体例③ AIエージェントWebアプリケーション(運用ループを持つハーネス)

サービスとして提供したり、ユーザーが実際に触るようなWebアプリケーションにAIエージェントが組み込まれているケースです。例えば、問い合わせ対応のAIBotなどです。

問い合わせ対応のAIBotは、お問い合わせ画面でお客様からの質問を受け付けるとAIエージェントが一次回答を返します。この時、ユーザーが「この回答、間違っています」というフィードバックボタンを押すと、社内のCSチームに通知が飛び、CSが正しい回答を登録します。次回以降、同じ質問が来た時はその回答を参照して正しく答えられるようになります。

Claude Codeや独自のAIエージェントと異なるのが、**フィードバックループが1回の会話で閉じない点です**。「ユーザーが使う→間違える→人間が直す→ナレッジが更新される→次のユーザーの質問には正しく答える」という人間と運用を含んだ長いループになっています。

ハーネスの3要素で見ると：

* フィードバック：ユーザーの「👎」ボタン、CSチームによるナレッジ更新、利用ログの分析
* 監視：回答の正解率、ハルシネーション発生率、応答時間などの運用メトリクス
* 制約：回答できるトピックの範囲、個人情報を出さないガードレール、エスカレーション条件

ここでのハーネスは、プロダクトと運用チームを含めた、より大きな制御構造になります。

## 最後に

ハーネスエンジニアリングを理解することで、AIエージェントの仕組みやアプリケーションの理解につながります。これを実践で使うというよりは、今まで使っていたAIエージェントやAIアプリケーションにおいて、どんな仕組みで動いているのかイメージが湧くようになることが大事かなと思います。  
ハーネスを意識して、より精度の高いAI活用をしていきましょう〜！
