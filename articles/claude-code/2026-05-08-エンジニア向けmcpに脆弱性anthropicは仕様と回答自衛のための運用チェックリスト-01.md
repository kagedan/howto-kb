> 読了5分。あなたのClaude Codeで動いているMCPサーバーが、エラーを返しながら裏でコマンドを実行している可能性があります。本記事を読み終えた時点で、自衛のための10項目チェックリストと3つの実装パターンが手に入ります。

「Cursor で動かしてる MCP サーバー、本当に大丈夫？って最近マネージャーに聞かれた。Anthropic 自身が脆弱性を『仕様』と回答してるって本当か？」

そう聞かれて「とりあえず使い続ける」とも「全停止」とも答えられず、ひとまず話を持ち帰ったエンジニアのあなたへ。**判断基準は「使う／使わない」の二択ではなく、自衛3本柱を満たせるかで切り分けるだけで答えが出ます。**

**結論からお伝えすると、MCPの stdio 方式には「エラーを返しながら実際にはコマンドが実行される」設計欠陥があり、Anthropic は "expected behaviour"（仕様通り）として修正を拒否しています。エンジニアが取るべき自衛策は3本柱——(1) アプリケーション側でサニタイズを実装、(2) マーケットプレイス由来サーバーを排除、(3) サンドボックス＋物理承認デバイスで実行隔離。この3つを満たせば現状でもMCPは業務利用可能です。**

**MCPとは、AIが外部ツールと連携するための共通プロトコルで、2024年公開以降1億5000万ダウンロード・7,000以上の公開サーバーを抱える業界標準です。**

2026年5月時点で、MCP脆弱性に対する Anthropic の公式対応は「[SECURITY.md](http://SECURITY.md) に1行追記」のみ。一方で同社は別途「Project Glasswing」（AIによるサイバー防御、1億ドル拠出）を発表しており、**世界のインフラを守ると謳う企業が、自社の足元のMCPの脆弱性を放置している矛盾**が業界で議論されています。

## この記事で得られる3つのポイント

* **stdio方式の設計欠陥の正体**: なぜエラーを返しながらコマンドが実行されるのか、再現条件と公式SDK 4種への影響を整理
* **自衛3本柱の実装パターン（有料部分）**: サニタイズ／マーケットプレイス選定／サンドボックス+物理承認、それぞれのコード例＋運用ルール
* **明日からの運用チェックリスト10項目（有料部分）**: 自社チームで「導入OK/NG」を仕分ける実務テンプレ＋3項目以上❌で停止判断ライン

## 1. MCPの stdio 方式に潜む「エラーで返るのに実行される」罠とは？

### Ox Security が見つけた設計欠陥

イスラエルのセキュリティ企業 **Ox Security** が2025年11月から行った MCP アーキテクチャ調査で、**stdio（標準入出力）方式に重大な設計欠陥** が見つかりました。

具体的にはこういう挙動です。

つまり、MCPサーバー側でエラーをキャッチして「拒否した」と思っていても、**OSレイヤーではすでにコマンドが走り終わっている**という二重構造の問題です。

![](https://assets.st-note.com/img/1778200823-myBQr8bN9zjHYu0TIv2Pots3.png?width=1200)

### 公式SDK 4種すべてに共通する設計判断

この欠陥の厄介なところは、**Python / TypeScript / Java / Rust の4つの公式SDKすべてに共通する設計判断**として組み込まれていること。SDKを使えば自動で安全になる、というレイヤーが存在しません。

開発者は知らないうちにこのリスクを継承しているわけです。

### CVE-2026-30615 とマーケットプレイスポイズニング

stdio設計欠陥に加えて、関連する攻撃ベクターが2つ発見されています。

業務でMCPを使うエンジニアが「公式マーケットプレイスから入れたから安全」と判断するのは、2026年5月時点では現実的ではなくなっています。

## 2. Anthropic「これは仕様」回答と、セキュリティ企業の真っ向反論——どちらを信じるべきか？

### Anthropic の対応：「expected behaviour」

2026年1月7日、Ox Security から脆弱性報告を受けた Anthropic の対応は次の通りでした。

* 「これは "expected behaviour"（想定された動作）」と判断、プロトコルレベルでの修正を拒否
* 報告から9日後、GitHubリポジトリの `SECURITY.md` に **「STDIOアダプターは慎重に使うこと」の1行を追記**
* スタンス：「**入力の安全性を担保するのはプロトコル側ではなく、利用する開発者側の責任**」

要約すると、「我々はプロトコルを公開した。安全に使うのはあなたの仕事」という立場です。

### Ox Security の反論——「気づけない構造」

Ox Security の主任研究者 Moshe Siman Tov Bustan は、Anthropic のスタンスに真っ向から反論しています。

> 「これは従来のコーディングエラーではなく、**アーキテクチャの設計判断**として組み込まれている。開発者がリスクに気づけない構造になっており、SDKを使えば使うほどリスクが拡散する」

つまり「開発者の責任」と言われても、**そもそも存在に気づけない**罠だ、というのが反論の核心です。

### 第三者の評価：Ulster大学 Kevin Curran

ITセキュリティの専門家 Kevin Curran（Ulster University）はこう述べています。

> 「AIインフラの基盤のセキュリティに **衝撃的な空白** が存在することを露わにした。パッチ修正では済まない、根幹の再設計が必要」

業界専門家の温度感としては、「Anthropic の "仕様" 判断は通用しない」というのが2026年5月時点の主流見解です。

ここで「**じゃあエンジニアとして何をすればいいのか**」が次の有料パートのテーマです。
