## 概要

この動画の主題は、**1体のエージェントに全部やらせようとするのではなく、適切な相手に仕事を委譲できるようにしよう**、という話です。ただし、その委譲先は一種類ではありません。別チームが運用している remote agent に任せたいときもあれば、Claude Code や Gemini CLI のような既存の coding harness に一仕事だけ投げたいときもあります。Mastra はその 2 つを、それぞれ A2A と ACP という別プロトコルで受け止めようとしています。

前半の A2A パートでは、Python 製の CrewAI エージェントや別プロセスの Mastra エージェントを、Mastra の supervisor agent 配下のサブエージェントとして扱う考え方が説明されます。ここで重視されているのは、単に別言語と話せることではありません。**別チーム、別デプロイ、別ライフサイクルのエージェント同士を、HTTP 越しに疎結合でつなげる**ことです。agent card による discoverability、非同期タスク、push notifications といった仕組みも、この運用前提に沿って紹介されます。

A2Aについての解説記事はこちらをご覧ください  
<https://zenn.dev/shiromizuj/articles/d8123a18c899a4>

後半の ACP パートでは、委譲先が「リモートの別エージェント」ではなく、「ローカルまたは同一マシン上で起動する specialized harness」である場合が扱われます。Ward の整理では、Claude Code のような環境は、ファイル探索や bash ツール、コード生成のループがすでに非常に洗練されています。ならば Mastra 側で同じものを再発明するより、**必要なタスクだけハーネス側へ handoff したほうが速い**。この動画は、A2A と ACP を単なる新機能の列挙ではなく、「Mastra がどの境界を自前で抱え、どの境界をプロトコルで外に開くのか」を示す設計回として見ると非常に分かりやすいです。

ACPについての解説記事はこちらをご覧ください  
<https://zenn.dev/shiromizuj/articles/c94b17a3a3b704>

## 要点

1. A2A は remote agent 向け、ACP は CLI ベースの coding harness 向けで、Mastra はこの 2 種類の委譲先を分けて扱おうとしています。
2. A2A の価値は「別言語と話せる」こと以上に、別チーム・別デプロイのエージェントを疎結合に接続できることにあります。
3. A2A では agent card が重要で、説明・能力・ストリーミング対応可否などを名刺のように公開し、呼び出し側エージェントが委譲先を選びます。
4. Mastra では A2A agent を sub agent として渡すだけで、supervisor agent や workflow から比較的自然に remote agent を扱えます。
5. A2A は長時間ジョブを前提にしており、結果待ちのための polling だけでなく push notifications も視野に入れた設計になっています。
6. ただし A2A の streaming は、現在のデモ時点では細かい進捗というより heartbeat に近く、内部で何をしているかまで詳しく返すわけではありません。
7. ACP は Claude Code や Gemini CLI のような既存ハーネスを Mastra から利用するための道具で、強力な既成の作業環境を再利用できる点が核心です。
8. ACP は現状では同一マシン寄りの用途が中心で、ローカルワークフローや専用サーバー上の作業委譲に向いています。
9. MCP は主にツールやデータソース向けの同期的な接続であり、A2A のような長時間・自律的な agent-to-agent 連携とは役割が異なります。
10. 動画全体を通じて見えてくるのは、Mastra Studio を将来的に「Mastra 純正以外も含めたエージェントの統合ハブ」へ広げたいという方向性です。

## 詳細

### まず整理したいのは「A2AとACPは似ているが、解きたい問題が違う」という点

このワークショップでは、A2A と ACP をまとめて紹介しながら違いを説明しています。どちらも相互運用性を高めるためのプロトコルですが、接続先の性質が違います。

Ward の整理をそのまま借りると、A2A は remote agent のためのものです。別プロセス、別コードベース、別言語、場合によっては別チームが管理しているエージェントへ、HTTP 越しにタスクを送るための仕組みです。対して ACP は、Claude Code や Gemini CLI のような既存ハーネスを Mastra の流れに取り込むためのものです。こちらは「他チームの業務エージェントに聞く」というより、**既成のコーディング環境や作業環境に ひと仕事任せる**ニュアンスが強いです。

ここを分けて考えると、なぜ Mastra が 2 つの規格を同時に押さえようとしているのかが見えてきます。Mastra が解きたいのは「自前のエージェントだけで完結させること」ではなく、**どこに委譲すべきかを選べる土台を用意すること**だからです。

### A2A は「エージェントをAPIとして公開し、別のエージェントから呼ぶ」ための仕組み

A2A の説明で最も重要なのは、Ward がこれを単なる protocol trivia として話していない点です。彼が強調するのは、企業の中には部署ごとに別々のエージェントがあり、それぞれ別々に開発・デプロイ・更新したいという現実です。

もし sales team が data team の知見を使いたいなら、以前は専用 API を切って「この endpoint を呼んでください」という会話になりがちでした。A2A が目指しているのは、その境界をエージェント単位に引き直すことです。sales 側のエージェントが data 側のエージェントに対して、「直近 30 日のページビューを集計して」「離脱ユーザーを分析して」と自然言語ベースの仕事を投げられるようにするわけです。

この構図は、マイクロサービスにかなり近いです。ただし返ってくるものは同期的な JSON だけではなく、相手エージェントが内部で複数ステップの思考やツール実行をしてもよい。そのため A2A では、すぐ返らない仕事、長時間ジョブ、あとで回収する結果、という考え方が前面に出てきます。

### agent card は「何ができるか」をエージェント同士で共有する名刺になる

A2A の中で印象的なのが agent card の説明です。Ward はこれを business card にたとえています。どんな名前で、何ができて、どんな説明があり、ストリーミングや push notifications をサポートしているのか。呼び出し側はまずこれを見て、「この相手なら自分のタスクを助けられそうか」を判断します。

MCP のサーバーが「ここに使えるツールがあります」という粒度で能力を公開するのに対し、A2A のカードはもう少しエージェント寄りです。相手の中で何が起きるかをすべて知る必要はないが、少なくともどの仕事を任せられるかは知りたい。動画の表現を借りれば、A2A は「この MCP server にはこういう tools があります」よりも、「この agent はこういう役割と skill を持っています」に寄っています。

この差は実務で効きます。ツールの粒度ではなく、任せたい責務の粒度で委譲先を選べるからです。

### Mastra の supervisor パターンと A2A の相性はかなりよい

デモでは、CrewAI 製の release changelog agent や Mastra 側の別エージェントを sub agents として並べ、親の supervisor agent がどこへ委譲するかを判断します。この構成は、Mastra が以前から持っていた supervisor の考え方とかなり噛み合っています。

Mastra の スーパーバイザー・パターンについてはこちらの解説記事をご覧ください。  
<https://zenn.dev/shiromizuj/articles/50513de990ca0b>

ポイントは、remote agent であっても、Mastra 側からは比較的自然に sub agent として見せられることです。description を与え、agents 配下に渡し、必要なら client 側から `getAgentCard` や `sendMessageStream` を呼ぶ。Alex も途中で「同じことを tool call でやることもできるが、なぜわざわざ難しくするのか」と言っていますが、まさにその通りで、**agent として扱ったほうが supervisor の推論対象にしやすい**のがこの構成の利点です。

さらに、Mastra のストリーム表示では、ツールよりサブエージェントのほうが「今どの責務に委譲しているのか」を人間が追いやすい、という話も出てきます。実装の都合というより、観測性と運用性の都合で agent として持つ価値があるわけです。

### A2A の非同期性は、UX より先に運用要件から理解したほうがよい

Studio 上のデモを見ると、A2A は少しもっさり見える場面があります。Ward 自身も、現状の streaming は内部の細かい進捗というより heartbeat に近いと認めています。ですが、この仕様を UI だけで評価すると本質を外します。

本当の論点は、**長時間タスクの結果を取るために、呼び出し元が接続を保持し続けなくてよいか**です。push notifications はまさにそこを狙っています。タスク ID と callback 先を持っておき、終わったら知らせてもらう。これは serverless や durable workflow と相性が良い発想ですし、企業内のバッチ的なエージェント処理にも向いています。

Ward も、Mastra 側では durable workflow がまだ完全ではないため、そこまで end-to-end で美しく見せ切れているわけではないと率直に話しています。ただ、動画の価値はその未完成さを隠さない点にもあります。A2A を「今日から何でも滑らかになる魔法」としてではなく、**長時間・疎結合・非同期な agent interaction に向けた土台**として説明しているのは誠実です。

### A2A の本当のユースケースは「別言語対応」より「別組織対応」にある

この回を見ていて特に重要だと感じるのは、Ward が Python 連携そのものを主役にしていないことです。もちろん CrewAI や LangChain のような別言語・別フレームワークと話せるのは大きいです。しかし彼が何度も出す例は、部署横断やチーム横断です。

support team のエージェント、open source 側の GitHub issue 分析エージェント、data team の分析エージェント、sales team の活用エージェント。これらが独立してデプロイされ、独立して更新され、それでも protocol でつながる。もしどちらかが壊れても、相手のコードベースに直接組み込まれていないから blast radius を抑えやすい。この発想は、単なるインターオペラビリティというより、**エージェント時代の組織設計**に近い話です。

動画の中で Alex が「agent as a service みたいな未来もありそうだ」と口にする場面がありますが、まさにそこです。A2A を理解するうえで大切なのは、1 対 1 の会話 API を増やす話ではなく、エージェントを組織内のサービス境界として扱う話だと捉えることです。

### ACP は「Claude Codeのような強い作業環境を部品として使う」ための現実的な解

後半の ACP は、A2A よりも手触りが具体的です。たとえば PowerPoint を作る、Excel を生成する、GitHub issue を読んで整理する、リポジトリを深く検索する。こうした作業は、専用の coding harness がすでに非常に得意です。

Ward が言っているのは、Mastra がその能力を一から再実装する必要はない、ということです。Claude Code には bash ツールもあれば、ファイル探索もあり、既存の skill やフックもあります。ならば ACP で handoff したほうが速い。Mastra は orchestration と supervisor に集中し、重い作業は specialized harness に委譲する。この役割分担はかなり合理的です。

動画のデモでも、Claude 系の ACP エージェントにスライド生成を任せようとしています。ライブデモなので最終的な PowerPoint 生成は崩れますが、重要なのは成功・失敗そのものではなく、**Mastra から ACP サブエージェントを普通のサブエージェントのように扱える**ことを示した点です。

### ACP は今のところ「同一マシン寄り」と理解しておくのが実務的

ACP については、Ward がかなり現実的な制約も説明しています。今の主用途はローカルマシンや専用サーバー上で、同一マシンに入っている harness を起動して使うことです。serverless 上で何でも透過的に遠隔利用する、というよりは、ワークフローの一部で CLI ベースの強いエージェントに一仕事任せるイメージです。

この前提を押さえると、A2A と ACP の違いはますますはっきりします。A2A は「別の場所にいる別のエージェント」と話すためのもの。ACP は「この環境にある強いハーネスを呼び出す」ためのもの。両者は競合ではなく補完関係です。

さらに ACP では workspace の閉じ込めや permission hook の話も出ます。ファイル読み書きをどこまで許すか、どの workspace だけに限定するか、といった制御は、特に coding harness を組み込むなら必須です。Mastra が今後この系統を強化していくのは自然な流れだと思います。

### MCP は依然として重要だが、A2A や ACP の代用品ではない

この動画は A2A と ACP が主役ですが、MCP をどう位置づけるかも地味に重要です。Ward は MCP を否定していません。むしろ、ツールやデータソースに接続する標準として依然重要だと扱っています。

ただし、MCP は基本的に「ツールを呼ぶ」発想です。裏でエージェントが動いていてもよいものの、表現としてはツール寄りです。これに対し A2A は、相手をより主体的なエージェントとして扱い、長時間タスクや discoverability、push notifications を含む文脈で捉えています。ACP はさらに別で、CLI ハーネスに仕事を渡すための接続です。

つまり、MCP ひとつで全部代替するのではなく、**ツールか、remote agent か、local harness か**で接続モデルを分けているのが今回の整理です。これは開発者にとってかなり有用です。何でも同じ abstraction に押し込めないほうが、かえって設計判断がしやすくなるからです。

### 「Mastra Studioをエージェント統合ハブにしていく構想」でもある

後半でさらっと触れられるのですが、かなり重要なのが今後の Mastra Studio の位置づけです。Ward は、将来的には Mastra 純正エージェントだけでなく、Cursor、Claude 系、Codex 系のような外部の agent SDK や harness も Studio から扱えるようにしたいと話しています。

これは単に対応先が増えるという話ではありません。Mastra を「エージェントを自前実装するフレームワーク」から、「複数のエージェント系 runtime や harness を束ねる control plane」へ寄せる発想です。A2A と ACP の両方を取り込む意味も、ここまで来るとかなり明瞭になります。

自分で作ったエージェントも、他チームが持つ remote agent も、Claude Code のような CLI harness も、最終的には一つの supervisor や workflow の中で役割分担して動く。動画が実際に見せているのはその最初の形です。

### 主題は「全部を一つに詰め込まない」こと

タイトルは Agent-to-Agent ですが、内容としてはもっと広い話です。どこまでを一つのエージェントに抱え込ませ、どこから先を他へ委譲するのか。その境界を protocol で表現しようとしているのが Mastra の今の動きです。

A2A は、組織や言語やデプロイ境界をまたいで remote agent とつながるための仕組みです。ACP は、Claude Code のような強力な作業環境を自前実装せずに取り込むための仕組みです。そして MCP は、依然として外部ツールやデータソースをつなぐための土台です。

この 3 つを切り分けたうえで、Mastra の supervisor、workflow、Studio に落としていく。今回のワークショップは、個別機能の紹介というより、**Mastra がエージェント時代の接続レイヤーをどう設計しようとしているかをかなり率直に見せた回**だったと言ってよいと思います。
