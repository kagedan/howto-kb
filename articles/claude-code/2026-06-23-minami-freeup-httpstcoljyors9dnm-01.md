## そもそもMCP連携って何

MCP（Model Context Protocol）は、AIとObsidianの間に専用の橋を架ける仕組み。
Obsidian側にプラグインを入れて、AIがObsidianの内部機能を直接叩けるようにします。

**MCP連携でできること：**

- Obsidianの検索インデックスをAIから直接使える（タグ検索・全文検索がObsidianの精度で動く）
- Dataviewクエリの結果をそのままAIに渡せる
- バックリンクやグラフのつながりをAIが読める
- Canvas・テンプレート展開などプラグイン機能をAIから操作できる
**ただし条件がある：**

- Obsidianが起動してないと動かない
- プラグインの導入・設定・認証が要る
- AI側にもMCPクライアントの設定が必要
- 設定が壊れたら繋がらなくなる
「できることは増えるけど、管理する部品も増える」ってことなんです。

## フォルダ指定だけで3ヶ月やってること

一方でフォルダ指定は、VaultのフォルダパスをAIに教えるだけ。
設定もプラグインも要らない。

![](https://pbs.twimg.com/media/HLdYjqXa4AEHemA.jpg)

MCP連携なしで私がやってること：

- 毎日のX記事作成（ルール・スキル・過去記事を参照して書いてもらう）
- note記事・音声台本の作成
- Inbox（メモ置き場）の整理・分類
- 海外記事クリップの翻訳・要約
- 週報・月報の自動生成
- 発信振り返り（Xアナリティクスと過去記事を照合）
- ルールファイルの点検・更新
- スキル（AIへの作業手順書）の新規作成・修正
これ全部、AIがVaultのファイルを直接読んで直接書いてるだけ。

「えっ、MCP連携なしでそこまでできるの？」って思いませんか？
できるんですよ。フォルダの中身を直接読めるなら、大体のことはできます。

## フォルダ指定で使い倒す方法

ここから具体的な「やること」を出します。
フォルダを開くだけだと、AIは何していいかわからない。
「ここで何をどう動くか」を伝える仕組みが要ります。

やることは3つだけ。

やること1：Vaultのフォルダをそのまま開く

ツールごとに開き方が違うだけで、やることは同じ。
VaultのフォルダパスをAIに教えるだけです。

- **Claude Code**：ターミナルでVaultフォルダに移動してClaude Codeを起動する。Vaultが作業フォルダになる
- **Codex**：設定ファイル（codex.json）にVaultのパスを書く。Codexがそこを読み書きする
- **Cursor**：VaultフォルダをCursorで「フォルダを開く」から開く。ワークスペースになる
どのツールでも、AIはフォルダ内の全ファイルを読み書きできるようになります。
Obsidianが起動してなくても動く。ここがMCP連携と違うところ。

やること2：AIへのルールファイルを置く

フォルダを開いただけだと、AIは「ここにどんなルールがあるか」を知らない。
Vaultの一番上に、2つのルールファイルを置きます。

**CLAUDE.md / AGENTS.md（指示書）**

AIがフォルダを開いたときに最初に読むファイル。
「このVaultのルール」「やっていいこと・ダメなこと」「迷ったらどこを見るか」を書く
