昨日「Claude Code、6月15日で使い放題終了」という記事を公開した後、何人かから同じ質問をもらった。

「モデル切り替えは分かった。でも、subagentの**種類**で何が変わるの？」

正直に言う。私もこれを意識して使い分け始めたのは3ヶ月前だ。それまでは全タスクにgeneral-purposeを投げていた。その時期のトークン消費記録を後から集計したら、**1タスクあたりの平均が現在の2.3倍だった。**

この記事では、49本のlaunchdエージェントを5ヶ月運用して確立した「SubAgent type 使い分け設計」を全公開する。6月15日の課金変更まであと10日。今すぐ設計を見直せば、追加コストをほぼゼロにできる。

## なぜSubAgentの「種類」がコストに直結するのか

Claude Codeのエージェント機能には、目的別に異なる4つのタイプが存在する。これを知らずに使っていると、全タスクで最も重いタイプが動き続ける状態になる。

問題の本質は「ツール初期化コスト」にある。general-purposeタイプは起動時に、Edit・Write・Bash・WebSearch・WebFetch・Glob・Grep・TaskCreate...といった全ツールのスキーマをコンテキストにロードする。これだけで数百トークンのオーバーヘッドが発生する。

**調査だけのタスクに全ツールを持ったエージェントを使う必要はない。**

## Claude Code 4種SubAgent：それぞれの特性と得意領域

### Exploreタイプ：読み取り専用の高速エージェント

最も重要な発見はこのタイプだった。Exploreは**すべての書き込み系ツールが無効化**されている。Edit・Write・Bashが使えない。

だからこそ3つのメリットがある。

**トークン効率が高い。** 書き込みツールのスキーマをロードしないため、タスクによっては消費トークンが35〜45%少ない。うちの計測では、コードベース探索タスクでgeneral-purposeの58%のトークンで同品質の出力が得られた。

**実行が速い。** 確認待ち・検証ステップが発生しない。コードベース探索タスクの体感時間が約40%短縮した。

**並列起動が安全。** 書き込みが発生しないため、複数のExploreエージェントを同時に走らせてもファイル競合が起きない。うちのシステムでは検索系タスクを最大8本同時並列で走らせている。

### Planタイプ：設計専門の分析エージェント

Planタイプは、ROI評価・設計判断・優先順位付けに特化している。

重要な特性は**「実行判断を自分でしない」**という設計思想だ。計画・分析・トレードオフの整理を行い、実行は呼び出し元に委ねる。

うちのプロジェクトでは月に平均23回このタイプを呼ぶ。主な用途はこれだ。

Planエージェントを正しく使うと、「自分で迷う時間」がほぼゼロになる。

### general-purposeタイプ：万能だが最もコストが高い

全ツールが有効。コード編集、ファイル作成、Bash実行、Web検索まで何でもできる。

**だからこそ、使い過ぎが最大の落とし穴だ。**

私が後から集計した結果、うちのシステムでgeneral-purposeを使っていたタスクの**約61%は、実はExploreかPlanで完結できた。**

つまり、2025年末まで毎日「余分なオーバーヘッド」を全タスクに乗せていたことになる。月換算でざっと170万トークンの無駄が発生していた。

### 専門エージェント：特定業務に最適化された高精度AI

bp-auditor（Anthropicベストプラクティス準拠チェック）、issue-resolver-worker（バグ修正）、issue-resolver-reviewer（修正内容の独立レビュー）といったタイプがある。

**汎用エージェントに同じことをやらせると、品質スコアが平均12〜18点下がる。**（3ヶ月間の独立レビューログを集計）

専門エージェントはタスク領域に特化したシステムプロンプトを持つため、同じ基盤モデルでも成果物の品質が明確に異なる。コスト最適化だけでなく、品質向上の観点でも使い分けは有効だ。

## 実際の使い分けパターン：5ヶ月で確立したルール

タスクを「何をするか」ではなく**「何を返すか」**で分類するとシンプルになった。

**「情報・分析・判断を返すだけ」→ ExploreまたはPlan**

コードベースの調査、依存関係の把握、ファイル検索、関連issue一覧取得など。全てExploreで完結する。判断・優先順位付け・ROI評価はPlanへ。

**「ファイルを書く・コードを実行する」→ general-purpose**

記事の下書き保存、issue修正、スクリプト作成など。書き込みが発生するタスクのみgeneral-purposeを使う。

**「特定品質基準を評価する」→ 専門エージェント**

コードレビュー、コンテンツ品質審査、BP準拠チェックなど。専門エージェントを呼ぶ。

この3分類に移行した結果、月間のSubAgent起動コストが**導入前比で55%削減**した。（2025年12月〜2026年5月、49本launchdエージェント5ヶ月計測）

## 踏んだ失敗3選と対処法

### 失敗1：ExploreにBashを実行させようとした

Exploreタイプはbashコマンド実行ができない。**git statusやnode scriptの実行が必要なタスクをExploreに投げると、そのステップでサイレントにスキップされる。**

症状として現れたのは「なぜか空の結果が返ってくる」だった。エラーが出ないため原因に気づくのが遅れた。2週間、原因不明の品質低下に悩んだ。

対処：「このタスクはファイル読み取りだけで完結するか？」を着手前に必ず確認する。1つでも「実行」があればgeneral-purposeへ。

### 失敗2：Planエージェントへのcontext過多投入

Planエージェントに「過去の会話ログ全部 + 関連ファイル10本 + レビュー結果」を一度に渡していた時期があった。**判断の精度が落ち、コンテキスト超過エラーが頻発した。**

Planエージェントは「整理されたインプット」で最も機能する設計になっている。情報が多すぎると、本来すべき「判断」より「整理」に処理が向かってしまう。

対処：Planエージェントへの入力は「①判断に必要な事実 ②選択肢と根拠 ③制約条件」の3点に絞る。詳細ファイルはパスだけ渡して自分でReadさせる。

### 失敗3：専門エージェントの多重起動でレビュー独立性を失った

「レビュアエージェント同士の指摘を共有すれば精度が上がる」と思って、reviewer-Aの結果をreviewer-Bに渡したことがある。結果、**アンカリング（anchoring）が発生して2体が同じ誤りを犯した。**

具体的には、reviewer-Aが「このコードのパフォーマンスは問題ない」と判断した。その判断をreceiveしたreviewer-Bは、本来なら発見できていたボトルネックを見過ごした。

対処：レビュアエージェントへの入力は「レビュー対象のパス + 評価基準のパス + ルール強制Read指示」のみ。他のレビュアの出力は絶対に渡さない。独立性の担保が複数エージェント起動の前提条件だ。

## 6月15日に向けて今すぐやること

課金変更まであと10日。SubAgent最適化は、モデル切り替えと並んで**即効性が高い施策**だ。

**ステップ1：現在のSubAgent呼び出しログを集計する（所要1時間）**

```
grep -rn "subagent_type" .claude/ logs/ | grep -E "(Explore|Plan|general-purpose)" | awk -F: '{print $NF}' | sort | uniq -c | sort -rn
```

**ステップ2：general-purposeのタスクリストを洗い出す（所要2時間）**

洗い出したタスクを「読み取りのみ」「書き込みあり」に分類する。「読み取りのみ」は全てExploreへ移行する。私の経験では最初の洗い出しで全タスクの半数以上がExploreに移行できた。

**ステップ3：Explore移行後の実測値を1週間比較する**

移行前後のトークン消費をログで比較する。2〜3日で効果が数値として見えてくる。

6月15日以降、--printモードでのClaude Code実行は別枠クレジットから課金される。Exploreタイプへの切り替えは、特にAPIコストが発生する場面で効く。

**副業規模の自動化システムなら、SubAgent最適化だけで月額追加コストをほぼゼロに近づけられる。**

## まとめ

SubAgent type使い分けの要点をまとめる。

* **Explore**：コードベース探索・ファイル検索・情報収集 → 35〜45%のトークン削減・並列起動可
* **Plan**：ROI評価・優先順位決定・設計判断 → 迷い時間ゼロ
* **general-purpose**：書き込みが必要なタスクのみに限定 → 乱用が最大のコスト要因
* **専門エージェント**：品質審査・特定業務判断 → 汎用比で12〜18点スコア向上

全タスクにgeneral-purposeを投げることは、全社員に取締役の給与を払いながら全員に雑用をやらせているようなものだ。

6月15日の課金変更は、自律AI運用の「設計力」を問う転換点だ。ツールを使うコストより、ツールを正しく設計するコストの方がはるかに安い。

自律AIエージェントシステムの設計を体系的に進めたい人には、タスク分解・プロンプト設計・自動化テンプレートを一括でまとめたセット商品が役立つ。
