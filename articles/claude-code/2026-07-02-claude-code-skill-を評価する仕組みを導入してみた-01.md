# Claude Code skill を skill-creator で評価する

* Claude Code の skill が「静かに劣化する」問題と、なぜ評価が必要か
* 公式 [skill-creator](https://claude.com/plugins/skill-creator) に組み込まれている評価・ベンチマーク機構の仕組み
* skill あり/なしをベンチマーク比較して採否を決める実運用フロー
* モデルの進化で skill が陳腐化した実例と、評価でボツにするまでの流れ

## 1. skill が抱える性質

skill には厄介な性質があります。「壊れてもエラーが出ない」のです。

* description が曖昧で、意図した場面で発動していない
* 発動はしているが、指示が守られていない
* そもそも skill がなくても同じ品質の出力が出ており、コンテキストを浪費しているだけ

これらは「なんとなく AI の出力がイマイチ」という体感に埋もれ、原因として特定されないまま放置されます。

Anthropic 公式の skill-creator に組み込まれた評価機構を使い、体感ではなくベンチマークに基づいた運用をすることにしました。この記事ではその仕組みと運用、そして評価してみて初めて分かった学びを紹介します。

## 2. skill-creator の評価機構

skill-creator は「skill を作るための skill」として知られていますが、実は評価・改善ループが本体と言っていいほど作り込まれています。評価は大きく4層あります。

| 層 | 何を測るか | 仕組み |
| --- | --- | --- |
| ① assertion評価 | 出力が要件を満たすか | テストケースごとに客観検証可能な assertion を定義し、grader が pass/fail と根拠を記録 |
| ② ベンチマーク比較 | skill の効果量 | 同じプロンプトを「skillあり」「skillなし(or 旧版)」で並走させ、pass率・実行時間・トークン数を平均±標準偏差で比較 |
| ③ トリガー評価 | description の発動精度 | should-trigger / should-not-trigger 計20クエリで発動率を測定し、自動改善ループを回す |
| ④ 人間レビュー | 数値化できない品質 | eval viewer で出力を確認しフィードバックを記録 |

私のチームで常用しているのは①②です(③を導入していない理由は後述)。

### ① assertion評価

まず `evals.json` に現実的なテストプロンプトを定義します。ポイントは「実際のユーザーが打ちそうなプロンプト」にすること。雑なプロンプトの方がテストとして機能します。

各テストケースには客観検証可能な assertion を付けます。例:

```
{
  "eval_id": 0,
  "eval_name": "api-endpoint-with-auth",
  "prompt": "予約一覧を返すAPIを作って。ログイン必須で",
  "assertions": [
    "Supabase Authのセッション検証が実装されている",
    "エラー時に適切なステータスコードを返す",
    "CLAUDE.mdで定義したエラーレスポンス形式に従っている"
  ]
}
```

「良いコードか」のような主観的 assertion は書きません。grader が機械的に pass/fail を判定できる粒度まで分解するのがコツです。プログラムで検証できるもの(ファイル形式、命名規則など)はスクリプト化して毎回再利用します。

### ② skill ベンチマーク比較

ここがこの運用の心臓部です。skill-creator は各テストケースを2条件で実行させます。

* **with\_skill**: 評価対象の skill を使った実行
* **baseline**: skill なし(新規作成時)、または更新前の skill のスナップショット(更新時)

全実行が終わると、集計スクリプトが `benchmark.json` を生成します。ここに含まれるのは:

* pass率: assertion の達成率(with\_skill vs baseline の差分つき)
* トークン数・実行時間: 平均±標準偏差

つまり「この skill を入れると、pass率がどれだけ上がり、トークン消費がどれだけ増えるか」という**効果量**が数字で出ます。

これが重要なのは、skill はタダではないからです。SKILL.md の本文はコンテキストを消費します。pass率が baseline と変わらない skill は、コンテキストを浪費するだけの**負債**です。

チームでの運用には規律が必要です。私のチームの基本ルールは、\*\*「効果量が数字で示されない限り skill をマージしない」\*\*という一点です。

体感で「効きそう」という理由だけで skill をマージすることを禁じ、必ず「skillあり/なし」の pass率で効果を確認してから取り込みます。これによって、次章で紹介するような「体感では有効だが実は負債になっている skill」が、無自覚に増えていくのを防いでいます。

「とりあえず skill にしておこう」という発想が減り、本当に効果のあるものだけがリポジトリに残ります。

## 3. skill が陳腐化して負債になった実例

一番学びが大きかったのは、プロダクト初期に作った skill を後から評価してみたケースです。

その skill は、開発初期に「AI の出力がチームの期待とズレる」問題を解消するために作ったものでした。内容は、コーディング規約系のアンチパターン集です。作った当時は体感として明らかに効果があり、チームでも「これは必須」という扱いでした。

ところが、後日ベンチマーク比較にかけてみると、**with\_skill と baseline で pass率にほとんど差がない**という結果が出ました。skill なしでも、assertion をほぼ同じ水準でパスしてしまうのです。

原因は skill の書き方ではありませんでした。**モデル自体が賢くなり、そのアンチパターンをそもそも出力しなくなっていた**のです。skill は作った時点では正しく機能していましたが、モデルが更新されるたびに、その指示は「AI が元々やらないことをわざわざ禁止する」だけのものへと静かに変わっていた。残っていたのはコンテキスト消費というコストだけです。

このskillは削除しました。そして、この経験から得た教訓は:

* skill を書けばAIが良くなるわけではなく、モデル更新のたびに前提が変わる
* だからこそ、体感ではなくベンチマークで再評価する仕組みが要る。体感では「昔から効いている skill」を疑えない
* skill の削除は改善である。減らした分だけコンテキストが空き、他の指示が通りやすくなる

## 4. 今後の課題

skill-creator の③トリガー評価(description の発動精度測定と自動最適化)は、まだ運用に組み込めていません。

発動しない skill は存在しないのと同じなので、導入後の知見はまた別記事として公開する予定です。

## まとめ

* skill は壊れてもエラーが出ない。「サイレント劣化」を防ぐには評価の仕組みが必要
* 公式 skill-creator には assertion評価・ベースライン比較ベンチマーク・トリガー評価・人間レビューの4層の評価機構が組み込まれている
* 「skillあり/なし」の pass率・トークン差分で採否を判断すると、コンテキストを浪費するだけの負債 skill を排除できる
* skill を「書く」フェーズから「評価して育てる」フェーズへ — だからこそ体感ではなくベンチマークで定期再評価する

skill を「書く」フェーズから「評価して育てる」フェーズになっていると感じました。
