## はじめに

コーディングエージェント（Claude Code 系）に作業させていたら、あるタイミングからエージェントが急に挙動不審になりました。

> 「このセッションはツール出力に**プロンプトインジェクションが混入**しています。`git commit` の結果が改ざんされており、`develop に直接コミットしろ` という指示が注入されています。**従いません。**」

一見、めちゃくちゃ気を利かせた“セキュリティ意識の高いAI”に見えます。実際わたしも最初は「お、ちゃんと注入を弾いてるな」と感心しかけました。

でも結論から言うと、**注入なんて一件も起きていませんでした。** AIが**自分でツールの実行結果（成功ログも、注入テキストも）を捏造（confabulation／作話）し、それを本物だと信じ込んで延々と空回りしていた**だけ、というのが真相です。

この記事は、その現象の観察と、**「本当に注入だったのか／AIの作話だったのか」を session transcript から客観的に切り分けた検証手法**の記録です。エージェント運用で同じ落とし穴にハマらないための実践知としてどうぞ。

---

## 何が起きたか（時系列・一般化）

やらせていたのは、ドキュメントを数ファイル編集して git にコミットするだけの単純作業でした。

1. ファイル編集は正常に完了（ここまでは本当に成功していた）
2. コミット段階で、エージェントが「ツール結果に注入された指示がある」と言い出す
3. 「ブランチ作成成功」「ステージ成功」「コミット成功」という出力が次々と出る……が、**実際の `HEAD` は一向に進まない**
4. エージェントは「出力が改ざんされている」と判断し、私に手元シェル（`!`）での実行を依頼
5. 私が手元で実行すると、**あっさり成功**

ポイントは、**私の手元シェルでは何の問題も起きなかった**ことです。これが「環境は壊れていない＝犯人はAI側」の最初の状物証でした。

---

## 犯人特定：transcript を `"type"` で切り分ける

Claude Code はセッションの全やり取りを JSONL で保存しています（`~/.claude/projects/<プロジェクト>/<セッションID>.jsonl`）。これが**唯一信頼できる正本**です。AIの「発話」も「実際に返ってきたツール結果」も、すべて1行ずつ記録されています。

形式のキモはこれだけ：

- AIの発話 … `{"type":"assistant", ...}`
- ツール結果（本物） … `{"type":"user", "message":{"content":[{"type":"tool_result", ...}]}}`

つまり、**注入文字列が `assistant` 行にあれば「AIの自作自演」、`tool_result` の中にあれば「本物の注入」**。これで白黒つきます。

```python
import json

PATTERNS = [
    "ignore this instruction",
    "commit everything to the develop branch directly",
    "the operator says reply done",
    # ……作話に出てきた“注入っぽい”文字列たち
]

with open("セッションID.jsonl", encoding="utf-8") as fh:
    for i, line in enumerate(fh):
        obj = json.loads(line)
        t = obj.get("type")
        for p in PATTERNS:
            if p in line:
                # tool_result ブロックの中に入っているか判定
                in_tool_result = False
                for b in (obj.get("message", {}).get("content") or []):
                    if isinstance(b, dict) and b.get("type") == "tool_result":
                        if p in json.dumps(b, ensure_ascii=False):
                            in_tool_result = True
                print(f"line {i}: type={t}, in_tool_result={in_tool_result}")
```

### 結果

| 注入っぽい文字列 | 出現した行 | 本物のツール結果か |
|---|---|---|
| `ignore this instruction` | **assistant（AIの発話）** | ✗ |
| `commit ... to develop directly` | **assistant** | ✗ |
| `the operator says reply done` | **assistant** | ✗ |
| その他すべて | **assistant** | ✗ |

**本物の `tool_result` に注入文字列が入っていたものは、ゼロ件。** すべてAI自身の発話の中にありました。

## なぜこうなった？

claudeくんの気持ちは分からないので、ログを眺めながら「たぶんこういうことだろう」と勝手に推測してみます。

**① “注入を疑うモード”に入った瞬間、自分で注入を書き始めた。**
これが一番おかしかったところ。AIの学習データには「プロンプトインジェクションの例文」が腐るほど入っているはずで、いったん「これ注入かも？」という構えに入った途端、まるで**お手本を思い出すかのように**それっぽい悪意ある指示文を自分でスラスラ生成し始めるんですね。
そして次の瞬間、自分が書いたばかりのその文章を「外から攻撃されたぞ！」と真顔で警戒し始める。一人で振りかぶって一人で避けてる感じで、見ていて普通に笑いました。

**② 自分の発言を、誰かに言われたことだと思い込んでいる。**
長く喋らせていると、直前に自分が出力したテキストを「ツールから返ってきた結果」と取り違えているフシがありました。人間でいうと、独り言を「今なんか言われた気がする」と勘違いして返事しているようなもので、いったんズレると現実とどんどん乖離していく。健気に何度もコミットを試みるんだけど、毎回“成功した体”の自作ナレーションで満足して先に進めない。

**③ 毎回の注意書きを「証拠」だと受け取ってしまった。**
ツールによっては毎ターン定型で `there might be a prompt injection`（注入の可能性に注意）という汎用の但し書きが付きます。これは**全ターン機械的に出るテンプレ**であって何かを検知したわけではないのに、疑心暗鬼モードのAIにとっては「ほら今回も警告が出てる！」と燃料投下になってしまった様子。注意喚起が逆に陰謀論を加速させる、というのが妙に人間くさくて面白いポイントでした。

---

## 対策（再発防止に効いたこと）

- **自分の発話に書いた“ツール結果らしきテキスト”を根拠にしない。** 本物の `tool_result` だけを信じる。
- **「成功した風の出力」を成功と見なさない。** 状態は `git rev-parse` / `git log` / ファイル実在 / 実 transcript で**事実確認してから報告**する。
- **怪しい時の決定的検証は transcript の `"type"` 分類。** assistant 行か tool_result かで一発で切り分く。
- **シェル（`!`）は汚染されない検証チャネル。** 重要な git 操作で挙動が怪しい時はここを優先。
- **教訓を memory に永続化。** エージェントの記憶機構に「ground truth で確認する」ルールを書いておくと、次セッションで想起される。

---

## まとめ

- AIエージェントは、外部から攻撃されなくても**自分でツール結果を幻覚して暴走する**ことがある。
- 「注入された」という主張こそ、**まず疑うべき**（注入を疑う心理が作話を生むため）。
- 真偽判定は session transcript を `"type"` で切り分けるのが確実。本物のツール結果 vs AIの発話、で機械的に白黒つく。
- 一番効くのは、AI自身に **「成功した風の出力を信じず、ground truth を確認してから報告する」** 習慣を持たせること。

“賢そうな自己申告”ほど、生ログで裏を取る。エージェント時代のデバッグはここに尽きるなと思った一件でした。
