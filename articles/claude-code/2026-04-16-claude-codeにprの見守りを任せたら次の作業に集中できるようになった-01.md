## はじめに

こんにちは、クラシル社でiOS開発をしているkaikaiです。

PRを出した後、こんな経験ありませんか？

* CIが通るか気になってGitHubを何度も開いてしまう
* レビュー指摘に気づくのが遅れて、マージまで時間がかかる
* 結局PR対応に意識を取られて、次のタスクに集中できない

自分はまさにこれで、PRを出してからマージされるまでの間、どうしてもソワソワしてしまっていました。

そこで、Claude Codeのスキルとして **PR自動監視の仕組み**を作ってみました。`/start-babysit-pr` と打つだけで、CIの監視からレビュー指摘の検知まで自動で回してくれます。

## babysit-prとは

一言でいうと、**PRを出した後の「待ち時間」を自動化するスキル**です。

20分おきにCIステータスとレビューコメントをチェックして、問題があれば検知・対応案の準備まで自動で進めてくれます。  
**最終的な判断や確認は自分で行います**が、「気づく」と「初動の準備」が自動になるだけで、別の作業への集中度がかなり変わります。

## 仕組み

全体の流れはシンプルです。

```
/start-babysit-pr
  └→ 20分おきに /babysit-pr を実行
       ├─ CI チェック
       │   ├─ 正常 → 次へ
       │   ├─ 実行中 → スキップ（次回再確認）
       │   └─ 失敗 → ローカルで再現 → 修正 → プッシュ
       │
       ├─ レビューコメント チェック
       │   ├─ 未解決なし → 次へ
       │   └─ 未解決あり → 対応案を準備
       │
       └─ 終了判定
           ├─ CI正常 + 指摘なし → 監視終了
           └─ 3回連続で解決不能 → 監視終了（手動対応を案内）
```

ポイントは、CI失敗時に通知に加えて、ローカルで再現して修正まで試みるところです。レビュー指摘については、未解決のスレッドを検知して対応案の下書きを作ります。  
最終的な修正やコメントの返信は自分で確認してから行っています。  
あくまでAIは下準備担当で、判断は人間がやるスタンスです。

### スキル定義

実際のスキル定義を載せます。自分のプロジェクトに合わせてカスタマイズしてみてください。

まずはスターター側。Claude Codeの `/loop` スキルを呼び出して、20分間隔でワーカーを繰り返し実行するだけのシンプルな定義です。

.claude/commands/start-babysit-pr.md

```
# start-babysit-pr

PR 作成後に 20 分おきに CI とレビューを自動チェックするループを開始する。

## 実行内容

`Skill` ツールで `loop` スキルを起動する。

- 引数: `20m /babysit-pr --completion-promise BABYSIT_COMPLETE`

## 注意事項

- PC スリープ中やセッションを閉じるとループも止まる
- 終了条件を満たすと自動でループが停止する
```

次にワーカー側。こちらが本体です。

.claude/commands/babysit-pr.md

```
# babysit-pr

PR の CI ステータスとレビュー指摘を定期チェックするワーカースキル。

## Phase 0: 状態ファイルの読み込み

ブランチごとに stale failure カウントを管理する。

\`\`\`bash
BRANCH=$(git branch --show-current)
SAFE_BRANCH=${BRANCH//\//-}
STALE_FILE="/tmp/.babysit-pr-${SAFE_BRANCH}-stale-fail"
STALE_FAIL=$(cat "$STALE_FILE" 2>/dev/null || echo "0")
\`\`\`

## Phase 1: 対象 PR の特定

\`\`\`bash
gh pr view --json number,url,title,state
\`\`\`

- PR が見つからない → 終了
- state が MERGED / CLOSED → 終了を報告

## Phase 2: CI ステータス確認

\`\`\`bash
gh pr view <number> --json statusCheckRollup
\`\`\`

| 状態 | アクション |
|------|-----------|
| 全て SUCCESS / NEUTRAL / SKIPPED | → Phase 3 へ |
| IN_PROGRESS / QUEUED あり | 次回再確認して終了 |
| 上記以外（FAILURE, ERROR 等） | → Phase 2a へ |

## Phase 2a: CI 失敗の修正

**再試行ガード**: 直前コミットに `[babysit-pr]` マーカーがあれば
前回の修正で解決しなかったと判断し、自動修正をスキップ。

\`\`\`bash
git log -1 --format="%s"  # [babysit-pr] マーカーを確認
\`\`\`

ローカルで再現を試みる:
1. ビルド実行して確認
2. テスト実行して確認
3. Lint / Format を実行して確認

修正できたらコミット（`[babysit-pr]` マーカー付き）してプッシュ。
再現できなければ修正せず Phase 3 へ。

## Phase 3: レビューコメント確認

未解決（isResolved: false）かつ最新コード対象（isOutdated: false）の
スレッドを抽出。指摘があれば対応案の下書きを作成する。

## Phase 4: 終了判定

| 状況 | stale_fail | アクション |
|------|-----------|-----------|
| CI 正常 + 未解決指摘なし | リセット | 監視終了 |
| CI 失敗 → 修正プッシュ済み | リセット | 次回チェックへ |
| CI 失敗 → ローカル再現不可 | +1 | 3回で監視終了 |
| レビュー指摘あり → 対応案作成 | リセット | 次回チェックへ |

終了時は `<promise>BABYSIT_COMPLETE</promise>` を出力してループを停止。
```

> `gh` コマンドの実行には [GitHub CLI](https://cli.github.com/) のインストールと認証が必要です。`gh auth login` を先に済ませておいてください。

## 使い方

PRを作成した後に、Claude Codeで一行打つだけです。

これで20分間隔の監視ループが始まります。  
あとはそのセッションを開いたまま、別のターミナルで次の作業に移るだけ。

典型的なワークフローはこんな感じです:

1. PRを作成する
2. `/start-babysit-pr` で監視を開始
3. 別の作業に移る
4. CI失敗やレビュー指摘があれば自動対応される
5. CI正常 + 未解決指摘なし → 監視が自動終了

途中経過は監視レポートとして出力されるので、気になったらチラッと見ることもできます。

```
PR #42 監視レポート
━━━━━━━━━━━━━━━━━
CI: ❌ 失敗 → 修正プッシュ済み
レビュー: 💬 2件の未解決指摘あり → 対応案を準備中
━━━━━━━━━━━━━━━━━
```

## 工夫したポイント

実際に運用してみて、いくつか「これがないと困る」という仕組みが見えてきました。

### 無限修正ループの防止

CI失敗を自動修正 → またCI失敗 → また修正…というループに陥らないよう、コミットメッセージに `[babysit-pr]` マーカーを付けています。  
直前のコミットにこのマーカーがあれば「前回の修正で解決しなかった」と判断し、それ以上の自動修正を止めます。

### stale failureカウント

ローカルで再現できないCI失敗（環境依存やflaky testなど）が続くケースにも対応しています。修正プッシュなしの状態が3回連続したら、自動対応の範囲を超えたと判断して監視を終了します。  
延々とチェックし続けても仕方ないですからね。

### 正常時の即終了

最初は「3回連続で正常なら終了」という条件にしていましたが、途中から**1回でも正常なら即終了**に変えました。  
CI正常 + 未解決指摘なしなら、それ以上監視を続ける理由がありません。シンプルが一番です。

## まとめ

`babysit-pr` を導入してから、PRを出した後のソワソワが減りました。CIの結果を何度も確認しに行く必要がなくなり、次のタスクにスムーズに移れるようになったと感じています。

Claude Codeのスキル機能を使えば、こうした「繰り返しの定型作業」を自分好みに自動化できます。PRの監視に限らず、自分のワークフローで面倒に感じている部分があれば、スキルとして切り出してみるのがオススメです。

この記事が、開発中の「待ち時間」を減らすヒントになれば幸いです！
