会社を離れて、フリーランスとして食っていくことにしました。まず名刺の代わりになる文章が要るのですが、完璧主義のせいか、どうも一文字も書けない。頭の中の完成形に最初の一行が届かないと分かると、そこで止まってしまう。

それならAIに書いてもらえばいい。そう思ってClaudeに下書きしてもらったのですが、返ってきた文章は、なんというか、私の文章ではない気がしました。断定的で、妙に自信に満ちていて、私を深掘りした後に書かれた文章には思えない。一般論を、それらしく整えているだけのように読めるのです。

そこで発想を変えました。AIに「書いてもらう」のをやめて、AIに「取材してもらう」。その仕組みを Claude Code の Skill / Agent で組んだので、技術メモとして残しておきます。以降は手順の話なので、少し平坦に書きます。

## 全体構成

執筆を4つの役割に分け、それぞれを Skill / Agent にしました。

```
zenn-interview → zenn-write(下書き) → koetsu(校閲) → editor(編集) → preview → 公開
  取材            本人の言葉で組む      表記/誤字       構成/熱量               人間承認
```

配置は Claude Code の標準パスです。

* スキル: `.claude/skills/<name>/SKILL.md`
* エージェント: `.claude/agents/<name>.md`

## 1. AIに本文は書いてもらわない — zenn-interview

核はここです。**AIは質問と選択肢だけを出し、本文（prose）は書かない。** 中身の言葉は必ず本人から取ります。

理由は2つあります。ひとつは、AIが冷たく下書きすると、本人を知らないまま平均的な一般論を書くから。もうひとつは、完璧主義の人間はゼロから創る（0→1）のは苦手でも、出されたものを選んだり直したり（1→10）は即答できるから。この非対称を突きます。

.claude/skills/zenn-interview/SKILL.md

```
## 中核原則
1. 選択肢・構造で攻める。AI は本文（prose）を書かない。
   中身の言葉は必ず本人から取る。
2. 技法: 1問ずつ／相手の言葉を拾って返す／尋問しない／
   脱線を止めない／2:8 で本人が多く喋る。
```

取材で引き出す観点は、技術記事に寄せています。

| 観点 | 引き出し方 |
| --- | --- |
| 解決した課題 | 「これが無いと何が困る?」 |
| つまずき・エラー | 「エラー文、そのままコピペできる?」（★原文が命） |
| 意外な発見 | 「やってみて "え、そうなの" だったことは?」 |
| 選択の理由 | 「なんで X にした? Y じゃないのはどこが効いた?」 |

## 2. 校閲を“感覚”でなく“ルール”に — koetsu

AIに「校閲して」と頼むと、指摘が毎回ぶれます。感覚で見ているからです。そこで [textlint-ja](https://github.com/textlint-ja) のルールプリセットを土台にしました。

`ja-technical-writing` プリセットには、こういう具体ルールが揃っています。

* 一文は100文字以下 / 読点は一文に3つまで
* ら抜き言葉、二重否定、同一助詞の連続の禁止
* 「ですます」と「である」の混在検出
* NFD濁点・制御文字などコピペ由来の異物

`textlint` があれば併用し、無ければ同じ観点で目視します。

```
npx textlint --preset ja-technical-writing <file>
```

出力は「箇所・種別・指摘・修正案・重要度」の赤入れ表に統一しました。

## 3. 編集は3層に分ける — editor

編集は校閲と違い、構成や熱量に踏み込む主観的な作業です。出版の編集フローに倣って3層に分け、`editor` エージェントは ①② を担当、③ は `koetsu` に委譲します。

1. **構造編集（developmental）** — この原稿は本当は何の話か / どこで熱量を失うか
2. **line編集** — 段落の接続・トーン・リズム
3. **校正（copy editing）** — 表記・誤字（→ koetsu）

いきなり直さず、まず診断を言語化してもらうのがコツでした。

## 4. 執筆と公開 — zenn-write

最後に記事本体を書く導線です。frontmatter、構成テンプレ（はじめに→前提→手順→ハマりどころ→まとめ）、Zenn独自Markdown をまとめてあります。

公開ステップには人間承認ゲートを入れました。`git push` による公開は後戻りしにくいライブ操作なので、AIが独断で `published: true` にはしません。

## この記事自体の作られ方

種を明かすと、この記事も `zenn-interview` で取材されて書いています。導入は取材で引き出した私の言葉、本体はその素材を技術メモに整えたものです。私は「執筆」らしいことをほとんどしていません。

## まとめ

やったことは、ひとつの考え方に集約できる気がします。**AIに書いてもらうのではなく、本人の偏りをAIに渡して、AIを本人に寄せる。** 断定的で一般論になりがちなAIの文章を、取材で自分の声に引き戻す、という発想です。

これは独立の、たぶん序章にあたる記録です。仕組みの背景——完璧主義で書けなかったことや、独立を決めた経緯といった、個人的な話のほうは、note に書きました。

最後に。もし、この記事のどこかが役に立ちそうだったら、いいねやフォローをしてもらえると嬉しいです。読んでくれる人がいる、ということ自体が、次を書くときの締め切りになる——というのは、note のほうに書いたとおりで。同じように「書けない」で止まっている人がいたら、そっと渡してもらえたら。
