Claude Codeで何かカスタマイズを作ろうとしたとき、最初に迷うのが「スタンドアロン設定にするべきか、プラグインにするべきか」という選択肢ではないでしょうか。

ドキュメントを読み始めると、さらに「スキル」「エージェント」「フック」「MCP」という言葉が出てきて、余計に判断が難しくなります。「とりあえず `.claude/` に置いておけばいいか」で止まっている方も多いと思います。

この記事では、**プラグイン作成の3つのアプローチを比較しながら、どの方法を選ぶべきか**の判断基準を整理します。

## まず「3つの方法」を整理する

Claude Codeの拡張機能を作る方法は、大きく分けると以下の3つです。

| 方法 | 置き場所 | スキル名の例 | 主なユースケース |
| --- | --- | --- | --- |
| スタンドアロン設定 | `.claude/` ディレクトリ | `/hello` | 個人・単一プロジェクト向け |
| ローカルプラグイン | 任意のディレクトリ + `.claude-plugin/plugin.json` | `/my-plugin:hello` | チーム共有・複数プロジェクト展開 |
| マーケットプレイスプラグイン | Git リポジトリ + 上記と同構造 | `/my-plugin:hello` | コミュニティ配布・バージョン管理 |

「プラグイン」と一言で言っても、開発中のローカルプラグインと、配布用のマーケットプレイスプラグインは別物として考えると整理しやすいです。

## 方法1: スタンドアロン設定

`.claude/` ディレクトリにファイルを置くだけで動く、最もシンプルな方法です。

```
my-project/
└── .claude/
    ├── commands/
    │   └── deploy.md       # /deploy として呼び出せる
    ├── agents/
    └── skills/
```

`/deploy` のように短いスキル名をそのまま使えるのが最大のメリットです。プロジェクトのリポジトリに一緒にコミットしておけば、チームメンバーもすぐに使えます。

スタンドアロン設定が向いているケース:

* 特定プロジェクト専用のカスタマイズ（デプロイコマンド、コード規約チェックなど）
* まずアイデアを試したい段階
* 個人の作業フローの効率化

**ハマりやすいポイント**は、複数プロジェクトで同じスキルを使いたくなったときです。その場合は各プロジェクトの `.claude/` にコピーしなければならず、メンテナンスが面倒になります。そのタイミングがプラグイン化の合図です。

## 方法2: ローカルプラグイン（開発・チーム共有向け）

プラグインは専用のディレクトリ構造を持ちます。キモは `.claude-plugin/plugin.json` というマニフェストファイルです。

```
my-plugin/
├── .claude-plugin/
│   └── plugin.json          # プラグインの定義ファイル
├── skills/
│   └── code-review/
│       └── SKILL.md
├── hooks/
│   └── hooks.json
└── settings.json
```

`plugin.json` の中身はシンプルです。

```
{
  "name": "my-plugin",
  "description": "チーム共通の開発支援ツール",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}
```

起動時に `--plugin-dir` フラグで読み込みます。

```
claude --plugin-dir ./my-plugin
```

複数プラグインを同時に読み込むこともできます。

```
claude --plugin-dir ./plugin-one --plugin-dir ./plugin-two
```

よくある間違い: `skills/` や `hooks/` などのディレクトリを `.claude-plugin/` の中に入れてしまうケースがあります。`.claude-plugin/` の中は `plugin.json` だけです。その他のディレクトリはすべてプラグインルートに置きます。

スタンドアロンと比べたときのメリットは**ネームスペース**です。`/my-plugin:code-review` のように、プラグイン名がプレフィックスになるため、複数のプラグインで同名のスキルが存在しても衝突しません。

### スタンドアロン vs ローカルプラグイン の比較

| 観点 | スタンドアロン | ローカルプラグイン |
| --- | --- | --- |
| セットアップの手間 | ほぼゼロ | `plugin.json` 作成が必要 |
| スキル名 | 短い (`/hello`) | 長い (`/plugin:hello`) |
| 複数プロジェクトで使う | ファイルコピーが必要 | フラグで読み込むだけ |
| チームへの配布 | リポジトリに含めるか手動共有 | リポジトリURLを共有するだけ |
| バージョン管理 | なし | `version` フィールドで管理 |

## 方法3: マーケットプレイスプラグイン（配布向け）

ローカルプラグインの構造は同じですが、**Gitリポジトリとして公開し、マーケットプレイス経由でインストールできる形にする**のがマーケットプレイスプラグインです。

利用者側は `/plugin install` コマンドでインストールでき、アップデートも管理されます。自分でメンテナンスしているプラグインをAnthropicの公式マーケットプレイスに申請することも可能です。

```
# 利用者側の操作
/plugin install https://github.com/your-name/my-plugin
```

開発者としてやることは主に2つです。

1. `README.md` にインストール方法・使い方を書く
2. `plugin.json` のバージョンをセマンティックバージョニングで管理する

マーケットプレイスプラグインが向いているケース:

* オープンソースとして公開したい
* 社内で複数チームが使う共通ツール
* 定期的なアップデートが必要な機能

## 結局どれを選ぶべきか

判断のフローはシンプルです。

```
├─ 今いるプロジェクト専用？ → スタンドアロン設定
│
├─ 複数プロジェクトやチームで使う？
│   ├─ まだ開発中・テスト中 → ローカルプラグイン (--plugin-dir)
│   └─ 安定して配布したい → マーケットプレイスプラグイン
```

公式ドキュメントでも推奨されているアプローチは「まずスタンドアロンで試して、共有が必要になったらプラグイン化する」です。

### 既存のスタンドアロン設定をプラグインに移行する

`.claude/` に溜まったカスタマイズをプラグイン化するのは難しくありません。

```
# プラグイン用のディレクトリを作成
mkdir -p my-plugin/.claude-plugin

# マニフェストを作成
cat > my-plugin/.claude-plugin/plugin.json << 'EOF'
{
  "name": "my-plugin",
  "description": "Migrated from standalone configuration",
  "version": "1.0.0"
}
EOF

# 既存ファイルをコピー
cp -r .claude/commands my-plugin/
cp -r .claude/skills my-plugin/
cp -r .claude/agents my-plugin/
```

フックだけは移行時に注意が必要です。`settings.json` の `hooks` オブジェクトをそのまま `hooks/hooks.json` に移します。

```
// my-plugin/hooks/hooks.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command", "command": "jq -r '.tool_input.file_path' | xargs npm run lint:fix" }]
      }
    ]
  }
}
```

移行後の変化をまとめると:

| 移行前 (スタンドアロン) | 移行後 (プラグイン) |
| --- | --- |
| `.claude/commands/` | `my-plugin/commands/` |
| `settings.json` 内の `hooks` | `my-plugin/hooks/hooks.json` |
| プロジェクトに紐づく | `--plugin-dir` でどこでも使える |
| 手動コピーで共有 | `/plugin install` でインストール |

## まとめ

3つの方法の使い分けを一言で表すと、**「スコープの大きさ」で選ぶ**です。

* 1プロジェクト完結 → スタンドアロン
* チーム・複数プロジェクト → ローカルプラグイン
* コミュニティ・組織全体 → マーケットプレイスプラグイン

スタンドアロンからプラグインへの移行は数分でできます。まずは `.claude/` に気軽にスキルを作り始めて、「これ他のプロジェクトでも使いたいな」と感じたタイミングでプラグイン化するのが現実的な進め方です。
