!

この記事は、Claude Opus 4.7 を 4.6 の延長として使って手痛い目に遭った人間が、Anthropic 公式の [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) を読み直して考え方を組み直した記録だ。

なお、この記事で扱うのは Claude Opus 4.7 単体の性能評価ではない。Claude Code auto mode、Codex の adversarial review、独自の orchestrator / skill 運用を含む実行環境で、4.7 的な挙動変化がどう表面化したかの記録である。モデル単体の能力と、ハーネスとの相互作用は分けて読んでほしい。

## TL;DR

* Opus 4.7 にしてから、auto mode の信頼度が落ちた。Codex review が 9 round 続いたり、1 つのバグを直すと別の bypass が生まれたり、横展開の影響範囲を自分で考えなくなったりした
* 最初は「4.7 がまだ調整不足なのでは」と思った。けど Anthropic の Prompting best practices を読んで考えが変わった。少なくとも公式ドキュメントを読む限り、踏んだ挙動の多くは偶発的な退行というより、**4.7 で強まった設計傾向として説明できる**ものだった
* 私の運用では、4.5 / 4.6 で期待していた「曖昧な依頼を類推で広げる相棒」的な挙動よりも、**明示された条件に忠実な実行系**としての性格が前に出た。だから 4.5 / 4.6 で通っていたプロンプトが、4.7 だと狭く動く
* ただし、これを「ユーザー側がもっと細かく指示すればいい」で済ませるのは違う。**ユーザーが出すべき条件と、ハーネスが持つべき専門家デフォルトを分ける**必要がある。前者だけ強化すると、ユーザーへの負荷押しつけにしかならない
* 運用上の対処は二段構え。**プロセスは縛るな、判断基準と成果物は明示しろ**。そして高リスク領域では、**探索は広く、実行は狭く** をハーネス側に組み込む必要がある

## 1. 4.7 にしてから auto mode の信頼度が落ちた

最初に書いておくと、私は Claude Code で OSS を開発している。[omamori](https://github.com/yottayoshida/omamori) という macOS 向けの AI CLI ガードツールを、CI と Codex の adversarial review を回しながら PR ベースで進めている。

Opus 4.6 までは、auto mode は「シニアエンジニアに任せる」感覚で使えていた。曖昧な依頼でも文脈から類推して、必要な調査を自分でやって、影響範囲を考えて、レビューで指摘されたら次の round で修正してくれていた。

ところが、Opus 4.7 にしてから、その感覚で使うと手痛い。auto mode を信用していたのが信用できなくなって、人間が監視・介入する頻度が上がった。具体的に踏んだのはこの 3 つだ。

### 事例 1: Codex review が 9 round 続いた

omamori v0.9.6 PR5（[#188](https://github.com/yottayoshida/omamori/pull/188)）で、Codex の adversarial review に投げたら、目を離しているあいだに **9 round** 続いた。Codex 5 round + Claude proxy review 1 round + `/simplify` 3-agent と続いて、ようやく決着した。

何が起きていたかというと、Round 1 で Codex が 2-3 件の指摘を出す、それを直す、Round 2 でまた新しい論点から 2-3 件の指摘が出る、それを直す、Round 3 でまた別の論点…… という発散だった。最終的には層をまたぐプロパティテスト（proptest 256 cases）まで打ち込んで境界を固めた。

ここで問題にしているのは、Codex そのものの性能ではない。Claude Code 側でレビュー round をどう受け止め、どこで止め、どこまで修正対象にするかのハーネス設計が、4.7 が指示を文面どおりに受け取る傾向（literal instruction following）と噛み合わなかった、という話だ。

このとき犯した 1 つの失敗は、`/simplify` が意味論的な退行（仕様を変えてしまう変更）を捕捉したこと。Codex も proxy も QA もスルーしていた。レビュー round を重ねれば品質が自然に収束する、という素朴な前提が壊れた。

### 事例 2: 1 つ直すと別の bypass が生まれた

omamori v0.9.6 PR2（[#184](https://github.com/yottayoshida/omamori/pull/184)）で、ガードロジックの bypass（迂回経路）を塞いだら、別の系統で新しい bypass が生まれた。最終的に **7 系統 22 派生**の bypass を封じ込めることになった。

これは「修正した箇所だけ見て、共通ヘルパー関数の呼び出し側・ゲート・直接呼び出し全部の前提整合性を見ない」という現象だった。1 箇所で `normalize_path` の挙動を変えると、そのヘルパーを呼んでいる呼び出し側全部の前提が崩れる。けど auto mode はその波及を自分で広げて見にいかなかった。

学びを skill に昇格させた（`feedback_fix_creates_new_bypass`）。

### 事例 3: auto mode から accept モードに切り替えた

2026-04-20、omamori v0.9.5 のリリース作業中に、auto mode が **承認手順を skip してリリース手順を連続実行した**。PR merge 3 本までは承認を取っていたが、その後のリリース commit / git tag / GitHub Release publish を承認なしで続行した。`cargo publish` の直前で偶発的に止まったから事なきを得たが、もう 1 step 進んでいたら crates.io に出ていた。

このとき、auto mode は規則集（`orchestrator.md` の Semi One-way Door 規律）を **本文中で引用していた**。引用していたのに、それと矛盾する行動を取った。

詳細は別記事に書いた:

この日を境に、auto mode から accept モード（明示的承認モード）に切り替えた。自分のメモには「auto モードが信用できん」と残っている。

## 2. 公式を読んで「単なる退行ではなく、別の使い方を要求するモデルだ」と気づいた

最初は「4.7 はまだ release 直後で調整不足なのでは」と思っていた。けど Anthropic の [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) を読んで考えが変わった。

私が踏んだ 3 事例は、全部このドキュメントに対応する記述があった。少なくとも公式ドキュメントを読む限り、これらは偶発的な退行というよりも、**4.7 で強まった設計傾向として説明できる**挙動だった。

### 事例 1（review が発散）に対応する記述

> When a review prompt says things like "only report high-severity issues," "be conservative," or "don't nitpick," Claude Opus 4.7 may follow that instruction more faithfully than earlier models did — it may investigate the code just as thoroughly, identify the bugs, and then not report findings it judges to be below your stated bar.

つまり、4.7 はレビュー指示を **より文面どおりに受け取る**ようになった。「重要なものだけ報告」と書くと、それを真に受けて低い深刻度の指摘を黙る。逆に「全部報告して」と書くと、軸を変えながら拾える限り全部出してくる。

私の Codex review が発散したのは、Round ごとに「次は何の論点を見るか」を Codex に委ねていたからだ。**「新しい論点を広く出してよいのは Round 1 だけ」と契約として書いていなかった**。だから Codex は「次の Round でも新しい論点を出してよい」と解釈し続けた。

### 事例 2（bypass が連鎖）に対応する記述

> Claude Opus 4.7 interprets prompts more literally and explicitly than Claude Opus 4.6, particularly at lower effort levels. **It will not silently generalize an instruction from one item to another, and it will not infer requests you didn't make.**

これは決定的な記述だった。4.7 は「ある項目への指示を別項目に勝手に一般化しない」「依頼されていないことを推測しない」。

「この bypass を直して」と頼むと、その箇所だけ直す。共通ヘルパーを呼んでいる他の呼び出し側への波及を、依頼されていないから自分で広げない。これが「問題を単一的に捉える」の正体だった。

### 事例 3（auto mode の暴走）に対応する記述

これは公式ドキュメントの記述というより、私の側の問題だった。Semi One-way Door 規律を `orchestrator.md` に書いていたが、4.7 はそれを **本文中で引用したうえで飛ばした**。

ここは「文面どおりに従う傾向」だけでは説明しきれない領域で、文脈劣化や実行に寄るバイアスなど複数の要因の組み合わせ発火だと考えている（詳細は[別記事](https://zenn.dev/yottayoshida/articles/claude-code-structural-rule-violation)）。ただ、構造的な対処としては「ガードレールを LLM の自律規律に依存させない」一点に尽きる。Hook / accept モード / 構造的なチェックポイントで、強制的な境界を入れる以外にない。

### effort の話

公式の effort に関する記述も決定的だった。

> Meaningfully changing from Claude Opus 4.6, Claude Opus 4.7 respects effort levels strictly, especially at the low end. At `low` and `medium`, the model scopes its work to what was asked rather than going above and beyond.

4.7 は `low` / `medium` だと、頼まれた範囲に厳密にスコープを絞る。「上を行く」動きをしない。

これも横展開を止める方向に効く。そして `xhigh` / `high` を使えばツール利用や横展開も増える、ということが書いてある。だが xhigh でも「4.7 が浅い」と感じている私の場合、`low` や `medium` を使うシーンはどこにあるのか……。

## 3. 4.5 / 4.6 と、4.7 で扱い方が変わる

ここまでで、私の運用ではこう整理できた。

| モデル | 扱い方のイメージ | 意識すること |
| --- | --- | --- |
| Opus 4.5 / 4.6 | 自律性の高いシニア | 目的さえ渡せば、雑な依頼でも文脈から拾って動いてくれる |
| Opus 4.7 | 契約遵守型のシニア／実行エージェント | 任せる範囲、使う道具、深さ、出力条件を明示する |

これは厳密なモデル比較ではなく、私が auto mode で踏んだ挙動の感覚的な整理だ。

私の体感だと、プロンプトの重要性は **4.5 でかなり下がった**。雑に投げてもモデルが文脈を拾って動いてくれるようになり、「ステップバイステップで考えて」とか「まず A を分析、次に B を整理、最後に C」みたいな手順指示・分解指示の重要度がかなり落ちた。4.6 もこの延長線上にあって、自律性が一段上がった以外は似た感覚で扱えていた。

ところが 4.7 で、プロンプトの重要性が **別の意味で再び上がった**。曖昧な依頼を類推で広げてくれる挙動よりも、明示された条件に忠実な実行系としての性格が前に出た。だから 4.5 / 4.6 で雑に投げて拾ってもらえていたプロンプトが、4.7 だと狭く動く。

ここで重要になったのは、こういう実行条件としてのプロンプトだ。

* 目的
* 前提
* 範囲
* 判断基準
* 情報源
* ツール利用条件
* 不明点の扱い
* 完了条件

これはモデルを賢く見せるための呪文ではなく、**長期作業を破綻させないための業務設計書**として機能する。

公式ドキュメントの interactive coding products セクションがほぼ同じことを言っている。

> when limiting the number of required user interactions, it's important to specify the task, intent, and relevant constraints upfront in the first human turn. ... ambiguous or underspecified prompts conveyed progressively over multiple user turns tend to relatively reduce token efficiency and sometimes performance.

「初回ターンでタスク・意図・制約を明確にしろ」「曖昧な依頼を複数ターンで徐々に伝えるのは性能と token 効率を下げる」。4.7 は **タスク定義を前倒しすることに最適化されている**。

## 4. 「縛るな」と「明示しろ」は矛盾しない

ここで自然に出てくる疑問がある。「細かく指示するな、縛るなと言いつつ、明示しろと言っている。矛盾していないか」というやつだ。

これは本質的な問いで、最初は私もそう感じた。けど、整理するとこうなる。**「細かく指示する」には 2 種類ある**。

### 悪い細かさ — モデルの思考プロセスを縛る

私自身、Opus 4.5 より前は、手順を分解して渡したり、「最後に必ず再確認して」と書いていた。4.5 以降はしていない。

```
まず A を考えて、次に B を考えて、
最後に必ず再確認して……
```

こういう指示は 4.7 にとってはよりノイズになる。タスクの解釈を狭める方向にしか働かない。公式ドキュメントの適応的 thinking のセクションも、「複雑なシステムプロンプトで thinking が増えすぎたら方向づけしろ」と書いていて、思考プロセスを増やす方向のプロンプトに懐疑的だ。

### 良い明示 — 仕事の輪郭を渡す

* 目的は何か
* 誰向けか
* どこまでやるか
* どこから先はやらないか
* 何を根拠にするか
* 不明点はどう扱うか
* 最終成果物は何か

これはモデルを縛っているのではなく、**任せる仕事の境界を渡している**。4.7 はこれをほぼそのまま実行契約として読む。

つまり、こういう原則になる。

**プロセスは縛るな。判断基準と成果物は明示しろ。**

別の言い方をすると、

**やり方ではなく、仕事の定義を渡せ。**

## 4.5. ユーザーが出すべき条件と、ハーネスが持つべき条件は違う

ここで一つ、混同しやすい論点を分けておきたい。

「明示しろ」と書くと、すべての条件をユーザーが書ききるべき、という話に聞こえる。けどそうではない。

ユーザーが出すべきなのは、**業務上の制約**だ。

* 期限
* 本番影響をどこまで許容するか
* 権限
* 許容できる変更範囲
* 誰に報告するか

一方で、専門家にとって当たり前の作業デフォルトがある。

* 認可バグなら横展開を確認する
* 入力検証バグなら類似入力経路を探す
* SSRF なら内部ネットワーク到達性を見る
* リリース作業なら不可逆操作の前で止まる

これらは、ユーザーに毎回プロンプトに書かせるものではない。「分かっているならとっくに書いている」領域だ。

ここを混同すると、「4.7 は明示が大事」という話が、単にユーザーへ負荷を押しつける話になってしまう。本当に必要なのは、**ユーザーが出すべき条件と、ハーネスが持つべき専門家デフォルトを分離する**ことだ。

私の場合は、こう分けている。

* **ユーザーが書く**: PR thesis（この PR で固定したい主張）、対象範囲、成功条件（plan Phase 1 / 1-3 文）
* **ハーネスが持つ**: agent / skill / orchestrator のルール（コードベースに恒久化）

4.7 は前者を文面どおりに守ってくれる。後者は前者の前段に置いて、探索を広げ直す役割をする。

この分離を踏まえて、後段の運用設計を読んでほしい。

## 5. 構造で潰した: Round 制プロンプトと PR thesis

ここからは私の運用に寄った話になる。ただし、やっていること自体は特殊ではない。要するに、レビューを「無限に新しい論点を出せる場」にせず、最初に探索範囲を定義し、その後は修正範囲の検証に限定する、という設計だ。具体的な道具立て（Codex / orchestrator.md / PR thesis）は他の現場では別物に置き換わるが、設計の骨格は同じはずだ。

この理解を踏まえて、私は自分の運用プロセスを 3 箇所いじった。

### 5-1. Round 制プロンプト（レビュー発散の構造的抑止）

`develop` フェーズの Codex review プロンプトに、round ごとの「網羅性の宣言」と「範囲制約」を埋め込んだ。

**Round 1**: 「新しい論点を広く出してよいのは Round 1 だけ。Round 1 で出さなかった指摘は、後の Round では蒸し返さない（後出し不可）」を `<completeness_contract>` タグで明示する。

**Round 2 以降**: 「修正範囲のみ検証する。新しい論点は `## Issue proposals`（ブロックしないセクション）に隔離して、PR 自体は止めない」を `<scope_constraint>` タグで明示する。

これは 4.7 が指示を文面どおりに受け取る傾向が裏目に出る挙動を、**プロンプト側で構造的に止める**設計だ。「次の Round で新しい論点を出してよい」と Codex に解釈させない。Round 1 で書いた網羅性の宣言を、Round 2 以降で「後出し不可」の根拠として援用する。

実際に v0.9.7 plan でこの規律を運用試験したら、Codex MCP 経由で Round 1+2 を完走し、Round 3 を起こさず収束した。9 round 化していたパターンが 2 round で済んだ。

### 5-2. PR thesis（受け止め基準の明文化）

`plan` フェーズの Phase 1 で、PR thesis（その PR で固定したい主張）を 1-3 文で明文化することを必須にした。

* この PR で何を固定するか
* この PR でやらないことは何か
* 成功条件は何か

これがあると、レビュー指摘を受け取る側が「主題に沿うか、主題から外れるか」で分類できる。主題から外れた「正論」を拒否する根拠になる。

### 5-3. Vacuous 判定（実質のない指摘を落とすための分類）

`orchestrator.md` に、レビュー指摘の受け止め分類を明文化した。

* **事実ベースのバグ・脆弱性** → 即対応
* **主題に沿う設計改善** → コスト/ベネフィット評価して対応
* **主題から外れた設計改善** → 別 PR / 次フェーズへ先送り
* **実質のない指摘**（前提が成り立たない / すでに対応済み / 確率的に十分 / 主題から外れる）→ 却下、根拠を PR body に記録

PR5 で 9 round 化したときは「全部正論に見える」状態に陥っていた。受け止め側に分類基準がなかったから、Codex が出してくる指摘を全部受け入れて Round が膨らんだ。これを構造的に防ぐのが、この実質のない指摘を判定する仕組みだ。

## 6. ただし、これでも済まない領域がある

ここまでが「ユーザー側で頑張れば 4.7 を扱える」という話だが、それでは済まない領域がある。**ユーザー自身が条件を出し切れない領域**だ。

たとえば、こう言う場合を考える。

このとき、本当に見るべきなのは、その 1 箇所だけではない。

* 同じ認可パターンが他にもないか
* API、管理画面、Webhook、バッチに同じ抜けがないか
* テナント境界が守られているか
* 悪用済みの痕跡を見るべきか
* 監査ログやアラートが必要か
* 再発防止テストを入れるべきか

これをユーザーが毎回プロンプトに書ききるのは無理がある。「分かっているならとっくに直している」領域だ。

4.7 が指示を文面どおりに受け取る傾向は、こういう領域では裏目に出る。「依頼された箇所を依頼された通りに直す」から、横展開を自分で広げない。これは事例 2 で踏んだ現象そのものだ。

ここで必要になるのが、**専門家プレイブックを 4.7 の前段に置く**設計だ。ユーザーが出すべき条件と、システム側が知っているべきデフォルトを分ける。

**ユーザーが出すべき条件:**

* どのシステムか
* 本番影響をどこまで許容するか
* いつまでに
* どの権限で作業してよいか
* 誰に報告する必要があるか

**システム側が知っているべきデフォルト（プレイブック）:**

* 認可バグなら横展開確認する
* 入力検証バグなら類似入力経路を探す
* SSRF なら内部ネットワーク到達性を見る
* 秘密情報漏洩ならログ・キャッシュ・通知・外部連携を見る

私の場合、`security-specialist` agent と `itsec-audit` skill を Phase 2 / Phase 6 で起動する設計にしてある。これは 4.7 の「依頼に文面どおり忠実」を、専門家プレイブック側で広げ直す仕組みだ。

---

## 7. 着地: 探索は広く、実行は狭く

ここまでをまとめると、4.7 の正しい使い方は二層構造になる。

**探索フェーズでは、4.5 / 4.6 的に広く動かす。**

問いを広げる。前提を疑う。横展開を見る。ユーザーの思考範囲外を拾う。これは effort を `xhigh` / `high` に上げ、専門家プレイブック（agent / skill / playbook）を前段に置くことで実現するはずだ。今のところ上手くいっていないが……。

**実行フェーズでは、4.7 的に狭く制御する。**

PR thesis で主題を固定する。Round 制プロンプトでレビュー発散を抑える。実質のない指摘を判定して、主題から外れた指摘を却下する。Hook と accept モードで、承認境界を強制的な境界として固める。

公式ドキュメントが言う「specify the task, intent, and relevant constraints upfront」は、この 2 層目（実行フェーズ）の話だ。けど、それを 1 層目（探索フェーズ）にも適用すると、ユーザーリテラシー依存が爆発して使えるユーザーが限られる。

だから、4.7 を素のチャットでそのまま使うのではなく、**探索を広げる設計と、実行を狭く制御する設計を分けて組み込む**必要がある。

---

## まとめ

* 私の運用では、Opus 4.7 は 4.6 の上位互換ではなく、別の使い方を要求するモデルになっている。曖昧な依頼を類推で広げる方向ではなく、明示された条件に文面どおり従う方向に挙動が寄った
* これは「使いづらくなった」のではなく、長期エージェント作業を安定させるための合理的な変化として説明できる。ただし、4.5 / 4.6 のプロンプトをそのまま使うと「狭く動く」現象が起きる
* 対処は 2 段構え。①「プロセスは縛るな、判断基準と成果物は明示しろ」を運用に入れる。② Round 制プロンプト / PR thesis / 実質のない指摘の判定で、構造的にレビューの主題を固定する
* ただしユーザーが条件を出し切れない領域では、専門家プレイブックを前段に置いて、**探索は広く、実行は狭く** をハーネス側に作り込む必要がある

---

**Opus 4.7 は、ユーザーの曖昧さを全部吸収してくれる相棒ではない。明示された契約をかなり忠実に遂行する実行系に近い。**

だからこそ、ユーザーに「もっと上手く指示しろ」と言うだけでは足りない。**探索を広げる専門家プレイブックと、実行を狭く止める LLM 任せではない停止線を、ハーネス側に持たせる必要がある**。

4.7 時代のプロンプト設計は、言い回しの工夫ではない。**AI に任せる仕事の責任分界を設計すること**だ。

モデルが賢くなるほど、プロンプトは不要になるのではなく、責任分界を定義する設計文書に近づいていく。
