# ループエンジニアリング指南書 — AIエージェントがループで成果を出すための手順書

この文書は、あるテーマを**ループエンジニアリング**で達成するときに、AIエージェントが **何を・どの順番で・どういう思考で** 行うかを規定します。

## 1. 定義 — ループエンジニアリングとは何か

> "Loop engineering is replacing yourself as the person who prompts the agent.  
> You design the system that does it instead." — Addy Osmani

**自分が「AIにプロンプトを打つ人」であることをやめ、「AIに指示するシステム（ループ）」を設計する実践。**  
プロンプトを上手く書く話でも、コンテキストを整える話でもなく、**あなたという人間をその場所から“どける”話**である。  
エンジニアは「作業者の一員」から「作業フローを設計する人」へ立ち位置を移す。

比喩は組織のリーダー: 逐一マイクロマネジメントしても回らないが、丸投げでも回らない。  
良いリーダーは「目標・レビュー体制・引き継ぎの仕組み」を整えて、自分が口を出さなくても回る状態を作る。

### 1.1 積み上げの系譜 — どの層の話をしているか

各層は競合ではなく入れ子で積み上がる。上の層ほど面倒を見る範囲が広い。

| 層 | 何を設計する | 中心の問い |
| --- | --- | --- |
| Prompt Engineering | 1ターンの指示文 | モデルに何を伝えるか |
| Context Engineering | その瞬間のコンテキストウィンドウ | 何を検索し、要約し、捨てるか |
| Harness Engineering | 単発実行の環境・制約・装備 | どのツールを許し、**何を以て完了とするか** |
| **Loop Engineering** | ハーネスの上で自動で回すシステム | **どうやって人間抜きで何度も回らせるか** |

| 観点 | ハーネス | ループ |
| --- | --- | --- |
| 設計対象 | エージェントが動く「環境」 | エージェントに「指示するシステム」 |
| トリガー | 人間が指示を出す | スケジュール・イベントが自動起動 |
| 稼働イメージ | 人間がいるときに動く | 人間がいなくても動き続ける |

ループはプロンプトで構成される。**ループ内のずさんなプロンプトは「ずさんな作業を速く生産する」だけ**になるため、  
下位層（プロンプト・コンテキスト・ハーネス）の品質は前提であって省略できない。

### 1.2 ループの内側 — 修正サイクル

システムの最小単位は次の修正サイクルである（ループの1周の中で回る内側のループ）:

```
        ┌──────────────────────────────┐
        │                              ▼
   生成（LLM） ──→ 検証（コード・決定的） ──→ 合格 → 次の工程へ
        ▲                │
        │   エラーリスト   │ 不合格
        └────────────────┘（上限つき。超過で人間の Inbox へ）
```

「速く動くループ」より\*\*「正しく止まるループ」\*\*を設計する方が難しく、重要である。

## 2. ループの5つの構成動作

完全なループは次の5動作を備える。設計時はこの5つを1つずつ埋めていく（§4）。

| 動作 | やること | 例 |
| --- | --- | --- |
| **Trigger（起動）** | 条件が満たされたら動き出す | 毎朝9時 / ファイル追加・CI失敗などのイベント / 人間の一度きりの指示 |
| **Discovery & Triage（発見と選別）** | ループが自分で仕事を見つけ、優先度を付け、状態ファイルに書き出す | 未処理ファイルの走査、新規 issue の収集 |
| **Execution（実行）** | 発見した仕事をエージェントが実行。並列時は作業空間を物理分離（Worktree 等） | 文字起こし・抽出の実行 |
| **Verification（検証）** | 結果を機械的に検証する。**ループの心臓部**（§3） | スキーマ検証・テスト・照合 |
| **Memory / Persistence（記憶）** | 状態を会話の外（ファイル等）に永続化し、次サイクルへ引き継ぐ | manifest / STATE.md / loop\_state.json |

道具立て（プリミティブ）との対応: Automations/cron（起動）、Skills（発見・手順の固定化）、  
Connectors/MCP（外部システムへの手: チケット・DB・シート。**ファイルしか見えないループは小さなループ**）、  
Worktrees（実行の隔離）、Sub-agents（検証: 生成役と評価役の分離）、Memory/状態ファイル（記憶）。

## 3. 心臓部 — Verification（検証）と停止条件

### 3.1 検証可能なゴールだけをループに渡す

* ループの「完了」は検証が True を返したときだけ。**「完了した」は主張であって証明ではない。**
* 良いゴール: 「対象配下の全テストがパスし lint がクリーン」「29列すべてがスキーマ検証を通る」— 機械的に確認でき、ループは真の完了に収束する。
* 悪いゴール: 「品質を改善する」「資料をより良くする」— 完了が判定できず、エージェントは“もっともらしく見える出力”に収束する。  
  これは\*\*スロップファクトリー（眠っている間に動くゴミの量産工場）\*\*になる。
* **原則: 機械的に確認できないゴールはループに渡さない。** 検証できない品質は人間レビューの領分として明示的に残す。

### 3.2 検証の実装原則

1. **決定的にする**: 検証器の中核は LLM を使わないコード（スキーマ・テスト・照合）。同じ入力→同じ結果でなければ停止条件にならない。
2. **Maker-Checker 分離**: 生成役（Maker）と検証役（Checker）を分ける。書いた本人に採点させると自己評価バイアスでエラーを見逃す  
   （自分の文章の誤字に気づけないのと同じ）。コードで書けない検証を LLM に頼む場合は、**別の指示・できれば別モデル**で、  
   姿勢は「承認する理由」ではなく\*\*「拒否する理由を探す」\*\*に設定する。
3. **エラーは修正指示になる形で返す**: フィールド単位・機械可読・修正ヒント付き。
   * 悪い例: `"validation failed"`
   * 良い例: `{"path": "knowledge_rows[0].ナレッジ（一言）", "rule": "grounding", "message": "引用が文字起こしに存在しません。最も近い実発言: 「…」", "hint": "引用を実発言に合わせて修正するか、行を削除してください"}`
4. **創作されやすい箇所に照合を置く**: 引用→原文との一致検査（グラウンディング検証）。メタデータ（ID・日付）→ LLM 出力を採用せず信頼ソースで上書き。属性→「不明なら空欄」を許し、値には根拠を要求。

### 3.3 サーキットブレーカー（必須）

止まらないループはコスト爆発を招く（高頻度・高自律のループほどトークンコストは跳ね上がる）。次の3つを必ず入れる:

* **試行回数上限**（例 `MAX_ATTEMPTS=3`）: 超過は「失敗」ではなく「人間の判断が必要」という正常な出口。
* **タイムアウト**（例 30分/周）: 進捗のないループを自動で止める。
* **コスト記録と予算**: 周回ごとのトークン消費を状態に記録し、月次予算と突き合わせる。

## 4. 設計フェーズ — テーマを受けたら、この順番で考える

いきなり実装やプロンプト作成に入ってはならない。以下を順に行う。**各 Step には「次に進んでよい条件」がある。**

### Step 0: 自律度（Autonomy Tier）を決める

最初に「このループをどこまで無人にするか」を人間と合意する。段階は3つ:

| ティア | 内容 | 人間の位置 |
| --- | --- | --- |
| **L1 Report** | 分析・検出まで。自動修正なし。コスト最小 | 全件を人間がレビュー |
| **L2 Assisted** | 修正案まで自動生成。**外部反映の直前に人間ゲート**（承認） | 承認ボタンだけ押す |
| **L3 Unattended** | 許可リスト/拒否リストの制約内で完全自律。危険な変更は人間キューへ昇格 | 例外・昇格分のみ |

**必ず L1 か L2 から始める**（§7「小さく始める」）。信頼できる結果を出している運用で、いきなり L3 に丸投げしている例はない。  
ループが得意なのは「**繰り返し発生する・検証可能な・境界が明確な**」タスクであることを確認する。この3条件を欠くテーマはループ化しない。

### Step 1: 正解の実物を入手する（最重要・最初にやる）

* ゴール成果物の**実例**（既存の完成品・過去の人手作業の出力）を必ず入手して読む。仕様書や口頭説明だけで進めない。
* 実物から読み取る: フィールドの完全な一覧と順序 / 値の語彙（enum）/ 例外ケースの扱い / 暗黙の粒度。
* 例（本リポジトリ）: 実スプレッドシートを読んだことで「1通話から複数行が生まれる」「ナレッジが無い通話も理由付きで1行残す」という、指示に無かった運用ルールを発見した。知らずに設計していれば検証器もプロンプトも作り直しだった。
* **進んでよい条件**: 成果物のサンプルを最低1つ、自分で開いて確認した。

### Step 2: 「合格」をコードで書ける形に翻訳する（停止条件を先に決める）

* 実物から合格条件を機械検証可能な表現に落とす。優先順位: ①構造（スキーマ・型・enum）→ ②事実整合（入力との照合）→ ③業務整合（フィールド間ルール）。
* §3.1 の通り、翻訳できない品質は検証器に入れず、人間レビューの領分として文書化する。
* **進んでよい条件**: 「この成果物を渡されたら合否を返せる関数」の仕様が書ける。

### Step 3: パイプラインをステージに分解する

* 変換の流れを「入力→出力」が明確なステージ列に切る。切る基準: 使う道具が変わる / 失敗の種類が変わる / 人間ゲートが入る、ところ。
* 各ステージに4点を定義する。**1つでも書けないなら分割が間違っている**: ①入力 ②出力 ③完了条件（Step 2 のどの検証か）④失敗時の遷移先。
* **進んでよい条件**: ステージ表（本リポジトリ `docs/architecture.md` §2.1 の形式）が埋まる。

### Step 4: 状態・Discovery・Trigger を設計する

* **状態の外部化**: 処理単位ごとに状態を持たせ、追記型ログ（JSONL 等）に永続化。コンテキストがリセットされても状態だけで再開できること。  
  状態レコードには最低限 `status` / `attempts` / `last_error` / `requires_human`（blocked理由）/ `updated_at` を持たせる。
* 状態名は「ステージ完了形」（`transcribed`）にする。曖昧な `processing` を作らない。  
  必須の状態: 正常系各ステージ + `needs_review`（人間の Inbox）+ `failed`（恒久エラー）。全状態から出口があること（袋小路禁止）。
* **Discovery**: ループが自分で仕事を見つける手順を決める（どこを走査し、何を「未処理」と判定し、どう優先度を付けるか）。
* **Trigger**: 起動方式を決める。最初は「人間の一度きりの指示」（コマンド実行）でよい。cron/イベント化は運用が安定してから（Step 7）。
* **進んでよい条件**: 「途中で強制終了しても、再実行すれば状態ファイルだけを頼りに続きから回る」と言い切れる。

### Step 5: 検証器を最初に実装する（Validator-First）

* 実装順は **検証器 → ループランナー → 生成プロンプト**。検証器が先にあれば、プロンプト調整を「印象」でなく「合格率」で評価できる。
* 検証器の出力仕様は §3.2-3 に従う。
* **進んでよい条件**: 手で壊したサンプルに対し、検証器が期待どおりのフィールド単位エラー+修正ヒントを返す。

### Step 6: 修正ループとサーキットブレーカーを組む

擬似コード（他の AI はこれをそのまま移植すればよい）:

```
def run_stage_with_repair(item, generate, validate, max_attempts, timeout):
    feedback = None
    with deadline(timeout):                        # サーキットブレーカー②
        for attempt in range(1, max_attempts + 1): # サーキットブレーカー①
            output = generate(item, feedback)      # Maker。feedback は前回のエラーリスト原文
            errors = validate(output, item)        # Checker。決定的・生成役と分離
            record_cost(item, attempt)             # サーキットブレーカー③
            if not errors:
                save(output); mark(item, "validated", attempts=attempt)
                return "ok"
            feedback = format_feedback(errors)     # §5.2 の形式。要約・省略しない
    mark(item, "needs_review", errors=errors, requires_human=True)  # 人間の Inbox へ
    return "needs_review"

def run_batch(items):
    for item in items:                             # 件単位で独立。1件の失敗で止めない
        try:
            run_pipeline(item)
        except PermanentError as e:
            mark(item, "failed", error=str(e))
    report_summary(items)                          # 成功/要レビュー/失敗、コスト、次アクション
```

* **進んでよい条件**: モック生成器（わざと1回目に壊れた出力を返す）で「2回目に合格」「上限超過→needs\_review」の両経路が通る。

### Step 7: 小さく始めて自律度を上げる

* まず1件を手動で通す → 検証器 → ループ → 10件バッチ → 本番、の順。いきなり全部を任せない。
* 収束率・needs\_review 率・コストが安定してから、Trigger の自動化（cron/イベント）と Tier の引き上げ（L2→L3）を検討する。
* Tier を上げるときは、自動適用してよい変更の**許可リスト/拒否リスト**を明文化してからにする。

## 5. 実行フェーズ — ループランナーとしての振る舞い

設計が済んだら、エージェントは**設計者から実行者にモードを切り替える**。

### 5.1 反復ごとの規律

1. **判定は検証器に聞く**。自分の出力を読み返して「良さそうだから合格」としない。逆に、合格が出たら過剰な自己懐疑で再生成しない。
2. **エラーは全文をそのまま戻す**。自然文に要約し直さない（情報が劣化し、同じ誤りを繰り返す）。
3. **修正は指摘された箇所だけ**。エラーと無関係な箇所を「ついでに改善」しない（合格済みを壊しループが振動する典型原因）。
4. **引用のグラウンディングNGは「探す」で直す。「作る」で直さない**。原文から該当発言を探して置き換える。無ければその行を取り下げる。それらしい文の作文は検証を通すための偽装であり、最悪の失敗。
5. **上限に達したら即座に手を止める**。「あと1回だけ」をやらない。needs\_review に送り、エラーリストと収束しなかった理由の仮説を添えて次の item へ。

### 5.2 修正フィードバックの標準形式

```
あなたの前回の出力は検証に不合格でした。以下のエラーをすべて修正し、完全な JSON を再出力してください。
エラー以外の箇所は変更しないでください。

## 検証エラー（N件）
1. [knowledge_rows[0].ナレッジ（一言）] grounding: 引用が文字起こしに存在しません。
   最も近い実発言: 「住宅価格って、すごくどんどん金額が上がってきてたんですけど…」
2. [knowledge_rows[0].年齢] no_evidence: 値「30代」に対応する根拠が文字起こしにありません。空文字にしてください。

## 前回の出力
<出力全文>
```

### 5.3 バッチ実行の規律

* 件単位で独立処理。1件のエラーで全体を止めない。
* 最後に必ずサマリ: 処理数 / 合格 / needs\_review / failed、消費コスト、次のアクション。
* 外部への副作用（シート書き込み・送信・公開）は Tier に従う。L2 では **dry-run 提示 → 人間の明示承認 → 実行 → 事後照合** の4手順を崩さない。
* 進捗の唯一の正は状態ファイル。「さっきどこまでやったか」を会話履歴から思い出そうとしない。状態ファイルを読む。

## 6. 人間の Inbox — 「半自動」の設計

すべてをループに解決させない。**対処できないもの・判断が必要なものを人間に届ける経路を最初から組み込む。**

| 関与点 | 内容 | 設計上の要件 |
| --- | --- | --- |
| **Inbox（レビュー）** | needs\_review 件の判断 | エージェントは「エラー・原文該当箇所・修正案」をセットで提示し、人間は選ぶだけにする |
| **Gate（承認）** | 外部反映前の最終確認（L2） | dry-run で差分を人間が読める形式（表）で提示する |

* 線引きは「**機械検証で担保できない品質だけを人間に回す**」。人間を毎回の生成確認に置くと自動化の意味がなく、置かないと信頼を失う。
* Inbox で人間が下した判断（例:「この種の通話はナレッジ化しない」）は、その場限りにせず**プロンプトまたは検証器へ還元する**（§7）。これで needs\_review 率が下がり続ける。

---

## 7. 改善フェーズ — ループ自体を改善するループ（メタループ）

運用開始後は収束率・needs\_review率・コストを計測し、ボトルネックを**正しい層**に還元する:

| 症状 | 直す場所 | やること |
| --- | --- | --- |
| 同じフィールドで繰り返し検証NG | プロンプト | ルールの明文化・良い例/悪い例の追加 |
| 検証は通るが人間レビューで差し戻される | 検証器 | 人間の指摘を新しい検証ルールに翻訳して追加 |
| 修正ループが振動する | フィードバック形式 | 「エラー以外変更禁止」の強調、エラー粒度を細かく |
| 特定の入力タイプだけ失敗する | ステージ分解 | 前処理ステージ追加・入力の分類（例: 録音不良の早期検知） |
| コストが想定を超える | Trigger/Tier | cadence を落とす、Tier を下げる、対象を絞る |
| そもそも要求が揺れている | スキーマ（合格の定義） | 人間と合意して schema 更新 → プロンプト → 検証器の順に同期 |

**合格の定義（スキーマ）・生成（プロンプト）・検証（コード）の3点は常に同期して変更する。**  
1つだけ変えるとループは静かに壊れる（永遠に不合格 or 誤った合格）。

## 8. 3つのリスク — ループを回す者が常に自覚すべきこと

1. **Verification の甘さ**: 無人で動くループは、ミスも無人でし続ける。検証を厚くする以外の対策はない。
2. **Comprehension Debt（理解の負債）**: 自分が書いていない成果物が速く出荷されるほど、「存在するもの」と「理解しているもの」のギャップが広がる。ループが速いほど「何が起きているか分からない資産」の膨張も速い。→ 人間レビューの間引きすぎに注意。サマリ・差分は人間が読める形で出し続ける。
3. **Cognitive Surrender（認知の降伏）**: ループの出力を何でも受け取ってしまう誘惑。ループを「思考を避けるツール」にした瞬間、失敗の加速装置になる。

> 「ループを設計することは、判断を持ってやれば解決策になる。思考を避けるためにやれば加速装置になる。  
> 同じ行為、正反対の結果だ」 — Addy Osmani

## 9. アンチパターン集

| アンチパターン | なぜダメか | 正しい形 |
| --- | --- | --- |
| 検証不可能なゴールをループに渡す | スロップファクトリー化する | 機械検証可能なゴールに翻訳してから渡す（§3.1） |
| LLM に「合格ですか?」と聞く / Maker に自己採点させる | 自己評価バイアスで見逃す | 検証はコード、LLM検証が必要なら別役割・別モデルで「拒否する理由を探す」（§3.2） |
| 検証NGを通すために引用・数値を作文する | 検証の偽装。最悪の失敗 | 原文から探す。無ければ取り下げる |
| 無限リトライ / 「あと1回だけ」 | コスト爆発・品質は上がらない | 上限・タイムアウト・コスト記録の3点セット（§3.3） |
| エラーを要約してから再生成に渡す | 情報が劣化し同じ誤りを繰り返す | エラーリスト原文を渡す（§5.2） |
| 進捗をエージェントの記憶・会話履歴で管理 | 中断・コンテキスト喪失で全損 | 状態ファイルに外部化（Step 4） |
| いきなり L3（無人）に丸投げする | 切り分け不能・信頼を失う | L1/L2から。1件→検証器→ループ→バッチの順（Step 0, 7） |
| 完成品サンプルを見ずにスキーマを書く | 暗黙の運用ルールを取りこぼす | Step 1 を最初にやる |
| 「ついで」の改善を修正ループ内で行う | 合格済みを壊しループが振動する | 修正は指摘箇所のみ。改善は別サイクルで |
| 人間の Inbox を設計しない | 例外がループ内で腐るか、無理に自動処理される | needs\_review 経路を最初から組み込む（§6） |

## 10. 本リポジトリへの対応表（適用例）

本プロジェクトの現在地は **L2 Assisted**（抽出まで自動、シート書き込み前に人間ゲート）。Trigger は当面「人間の一度きりの指示」（`/process-audio`）で、Phase 5 で cron 化（L3方向）を検討する。

| 指南書の項目 | 本リポジトリでの実装 |
| --- | --- |
| Step 0 自律度 | L2。書き込みゲート = `/write-sheet` の dry-run→承認 |
| Step 1 実物の入手 | 実スプレッドシートから29列/6列と運用ルールを採取（`docs/architecture.md` §5） |
| Step 2 停止条件の翻訳 | `config/knowledge_db_schema.json` / `refusal_schema.json` + グラウンディング検証 |
| Step 3 ステージ分解 | scan→transcribe→extract→validate→export→write（§2.1） |
| Step 4 状態/Discovery/Trigger | `state/manifest.jsonl`（attempts/error/状態遷移）/ scan\_audio.py が Discovery / Trigger は `/process-audio` |
| Step 5 Validator-First | Phase 2 で `validate.py` を extract より先に完成させる（`docs/implementation-plan.md`） |
| Step 6 修正ループ+ブレーカー | `extract.py` 内ループ + `max_repair_attempts: 3` + `state/usage.jsonl`（コスト記録） |
