## TL;DR

* Claude Codeはセッションごとに新しいコンテキストで起動するが、**2つのメモリ機構**でセッション間に知識を保持する
* どちらも**強制設定ではなくコンテキスト**として扱われる。具体的・簡潔・構造化された指示ほど遵守率が高い
* 大規模プロジェクトでは `.claude/rules/` でトピック分割し、`paths` frontmatterでファイル種別ごとにスコープすると効率的

## 2つのメモリシステムの比較

![](https://assets.st-note.com/img/1777679545-R0l2WGgAsSLJjnpKrviaEkTh.png?width=1200)

**使い分けの原則**:

* 動作をガイドしたい→CLAUDE.md
* 修正の積み重ねから自動学習したい→自動メモリ

## CLAUDE.md ファイル

### いつ追加するか

「同じ説明をもう一度するくらいなら書く」が基本姿勢。具体的には:

* Claudeが2回目に同じ間違いを犯したとき
* コードレビューで「これは知っておくべき」と指摘されたとき
* 前回のセッションでも同じ修正をチャットに書いたとき
* 新メンバーが立ち上がるのに必要なコンテキストが発生したとき

ただし、**全セッションで保持すべき事実に限定**する。複数ステップの手順や特定箇所だけに関係するルールは、skillsまたはパススコープルールに移すべき。

### 配置場所とスコープ

より**具体的な場所が広い場所より優先**される。具体的な優先順は以下。

* ローカル指示（./CLAUDE.local.md）
* プロジェクト指示（./CLAUDE.md）
* ユーザー指示（~/.claude/CLAUDE.md）
* 管理ポリシー（/Library/ApplicationSupport/ClaudeCode/CLAUDE.md

**注意点**

* ファイルは上書きではなく連結される。全部読み込まれた上で、矛盾時に「より具体的＝後に読まれたもの」が勝つ
* 同じディレクトリ内では CLAUDE.local.md がCLAUDE.mdの後ろに追加されるため、個人メモが最後勝ち
* ワーキングディレクトリから上の階層を辿るので、深い階層のCLAUDE.mdほど優先される（モノレポでサブパッケージ独自のルールが効くのはこのため）
* 管理ポリシーCLAUDE.mdは claudeMdExcludesでも除外不可 —個別設定で外せないので、組織方針は常に適用される（優先度は低いが「必ず適用」される点は別軸）
* ブロックレベルのHTMLコメント `<!-- maintainer notes -->` はコンテキストに注入される前に**削除**される。トークンを使わずメンテナーへのメモを残せる。コードブロック内のコメントは保持される
* ワーキングディレクトリより上の階層のCLAUDE.mdは起動時に全部読み込まれ、サブディレクトリのものはそのファイルを読むときにオンデマンドで読み込まれる

**効果的な指示の書き方**

**サイズ**: 1ファイル**200行以下**を目標。長いほどコンテキストを食い、遵守率が下がる。

**構造**: マークダウンの見出し・箇条書きで関連項目をグループ化。Claudeも人間と同じように構造をスキャンする。

**具体性**: 検証可能なレベルまで具体的に。

![](https://assets.st-note.com/img/1777679773-Zj46VgKvbX2umFW8J39xBL7z.png?width=1200)

**一貫性**: 矛盾するルールはClaudeが任意に選んでしまうので、定期的に整理する。

### `/init` でのブートストラップ

`/init` を実行するとClaudeがコードベースを分析し、ビルドコマンド・テスト指示・規約を含むCLAUDE.mdを自動生成する。既存ファイルがあれば上書きせず改善を提案。

### 追加ファイルのインポート

`@path/to/import` 構文で別ファイルを取り込める。

```
プロジェクト概要については @README を参照し、利用可能なnpmコマンドは @package.json を参照。

# 追加の指示
- git ワークフロー @docs/git-instructions.md
```

> 初回の外部インポート検出時に承認ダイアログが出る。**拒否するとインポート機能はそのまま無効化**され、ダイアログは再表示されない点に注意。

### AGENTS.md との併用

Claude Codeは `AGENTS.md` を直接は読まない。他のコーディングエージェントと共存するなら:

```
@AGENTS.md

## Claude Code

`src/billing/` 配下の変更にはPlan Modeを使う。
```

このようにCLAUDE.mdから `@AGENTS.md` をインポートして、Claude固有指示を下に追記する。

### モノレポ向けの除外設定

他チームのCLAUDE.mdを巻き込みたくないときは `.claude/settings.local.json` で除外:

```
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/user/monorepo/other-team/.claude/rules/**"
  ]
}
```

ただし**管理ポリシーCLAUDE.mdは除外できない**（組織全体の方針を強制するため）。

## `.claude/rules/` でルールを整理する

大規模プロジェクトでは指示をトピックごとに分割する。

```
your-project/
├── .claude/
│   ├── CLAUDE.md           # メイン指示
│   └── rules/
│       ├── code-style.md   # コードスタイル
│       ├── testing.md      # テスト規約
│       └── security.md     # セキュリティ
```

`paths` frontmatterのないルールは `.claude/CLAUDE.md` と同じ優先度で起動時にロードされるため、frontmatterの記載は忘れずに。

### パススコープルール

特定ファイルで作業するときだけ読み込まれるルールを書ける。

```
---
paths:
  - "src/api/**/*.ts"
---

# API開発ルール
- すべてのAPIエンドポイントは入力検証を含める
- 標準エラー応答形式を使う
- OpenAPIドキュメンテーションコメントを含める
```

ブレース展開で複数拡張子もまとめられる: `"src/\*\*/\*.{ts,tsx}"`。

### シンボリックリンクで共有

`.claude/rules/` はシンボリックリンクをサポート。複数プロジェクトで共通ルールを共有できる。

```
ln -s ~/shared-claude-rules .claude/rules/shared
ln -s ~/company-standards/security.md .claude/rules/security.md
```

### ユーザーレベルのルール

`~/.claude/rules/` に置くと全プロジェクトに適用される。プロジェクトルールより**先に**ロードされ、プロジェクトルールが優先される。

## 自動メモリ

Claude Code v2.1.59以降の機能。**Claudeが自分のために**学習・パターンを書き溜める。

### 有効化と保存場所

デフォルト有効。切り替えは `/memory` コマンドのトグル、または:

```
{ "autoMemoryEnabled": false }
```

環境変数なら `CLAUDE\_CODE\_DISABLE\_AUTO\_MEMORY=1`。

保存場所はプロジェクトごとに `~/.claude/projects/<project>/memory/`。同じgitリポジトリ内のworktree・サブディレクトリは**1つの自動メモリディレクトリを共有**する。

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 簡潔なインデックス、毎セッション読み込み
├── debugging.md       # デバッグパターン詳細
├── api-conventions.md # API設計の決定
└── ...
```

別の場所に変えたいときは `autoMemoryDirectory` 設定（プロジェクト設定からは受け付けない＝共有プロジェクトが個人の機密領域に書き込まないようにするため）。

### 仕組み

* `MEMORY.md` の**最初の200行 or 25KB**（先に到達した方）が毎会話開始時にロードされる
* それを超える内容は**起動時には読まれない**
* `debugging.md` などのトピックファイルはオンデマンド読み込み（必要なときだけ）
* Claudeは `MEMORY.md` を簡潔に保つため、詳細はトピックファイルに切り出す
* セッション中に「Writing memory」「Recalled memory」と表示されたら、Claudeが自動メモリにアクセスしている

> **CLAUDE.mdは長さに関係なく完全に読み込まれる**。この200行/25KB制限は自動メモリの `MEMORY.md` のみに適用される。

### 監査と編集

ファイルはプレーンマークダウンなので普通に編集・削除できる。`/memory` コマンドで一覧表示と直接オープンが可能。

### 自然言語での記憶指示

Claudeに「常にnpmではなくpnpmを使って」「APIテストはローカルRedisが必要、覚えておいて」のように頼むと、自動メモリに保存される。

CLAUDE.mdに書きたいなら明示的に「これをCLAUDE.mdに追加して」と頼むか、`/memory` でファイルを開いて手動編集する。

## `/memory` コマンド

現在のセッションに読み込まれた全ての CLAUDE.md / CLAUDE.local.md / ルールファイルをリスト表示する。自動メモリのオン/オフ切り替えと、自動メモリフォルダを開くリンクも提供。

## トラブルシューティング

### Claude が CLAUDE.md に従わない

CLAUDE.mdの内容は**システムプロンプトそのものではなく、システムプロンプト後のユーザーメッセージとして配信される**。読んで従おうとはするが厳密な保証はない。

デバッグ手順:

1. `/memory` で読み込まれているか確認（一覧になければClaudeから見えていない）
2. 配置場所が正しいか再確認
3. 指示をより具体的に書き直す
4. ファイル間で矛盾していないか探す

スクリプト/自動化で**システムプロンプトレベルの指示**が必要なら `--append-system-prompt` を使う。

> Tip: `InstructionsLoaded` hookでどの指示ファイルがいつ・なぜロードされたかを正確にログ化できる。パス固有ルールやレイジーロードのデバッグに有効。

### 自動メモリが何を保存したか分からない

`/memory` で自動メモリフォルダを開いて中身を見る。プレーンマークダウンなので読み・編集・削除が自由。

### CLAUDE.md が肥大化した

200行を超えると遵守率が下がる。対策:

### `/compact` 後に指示が消えた

* **プロジェクトルートのCLAUDE.mdは圧縮を生き残る**（圧縮後にディスクから再注入される）
* サブディレクトリのネストCLAUDE.mdは自動再注入されない（次にそのディレクトリを読むタイミングで再ロード）
* 会話のみで与えた指示はセッションを跨がないので、永続化したいならCLAUDE.mdに追記する

## 設計ヒント

* **CLAUDE.mdに書くべきもの**: 全セッションで必要な事実（コーディング標準、ビルドコマンド、アーキテクチャ）
* **`.claude/rules/` に書くべきもの**: 特定ファイルタイプ・サブディレクトリ限定の指示
* **skillsに書くべきもの**: タスク固有で「呼び出されたとき」だけ必要な手順
* **自動メモリに任せるもの**: 修正の積み重ねから自然に学ばせたい好み・パターン

設定ルール（`permissions.deny`、`sandbox.enabled` など）は**技術的強制**、CLAUDE.mdは**行動ガイダンス**。レイヤーを混同しない。
