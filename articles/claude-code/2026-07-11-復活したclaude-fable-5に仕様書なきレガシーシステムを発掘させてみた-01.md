2026年7月1日、[Claude Fable 5へのアクセスが復活しました](https://www.anthropic.com/news/redeploying-fable-5)。6月9日のリリースからわずか3日で米国の輸出規制により全面停止、19日間の空白を経ての再開です。

SNSはコーディング能力の話題で持ちきりですが、私の関心は最初から一点だけでした。

**このモデルは、仕様書なきレガシーシステムの「診断」に使えるのか。**

私は普段、SIerでレガシーシステムのモダナイゼーション支援をしています。Struts 1.x、Java 7、設計書は初版のまま、書いた人はもう社内にいない。そんな現場です。この記事では、実在するOSSのレガシーコードベースを対象に、Claude Code上でFable 5に診断をやらせ、Opus 4.8との差分を検証しました。診断に使ったプロンプトは全文載せるので、そのまま自分のリポジトリで追試できます。

## Fable 5の位置づけ

Fable 5は、Anthropicが新設した「Mythos」クラスのモデルです。従来の最上位だったOpusの上に置かれた新ティアで、一般提供モデルとしては同社史上最高性能です。上位版のMythos 5と同一のモデルですが、サイバーセキュリティなどの高リスク領域にセーフガードを加えることで一般提供を可能にした、という建て付けです。

レガシー診断に効くスペックは3つです。

* **コンテキストウィンドウ**：標準100万トークン(出力は最大12.8万トークン)。中規模システムなら主要ソースが一度に視野に入る
* **料金**：入力$10/出力$50(100万トークンあたり)。Opusより高いが、診断の代替コストは人月なので比較の土俵が違う
* **ベンチマーク**：項目によってはOpus 4.8を10%以上上回る

(スペックの詳細は[公式ドキュメント](https://platform.claude.com/docs/en/about-claude/models/introducing-claude-fable-5-and-claude-mythos-5)を参照)

そして[Anthropicの発表](https://www.anthropic.com/news/claude-fable-5-mythos-5)には、今回の検証の動機になった一文があります。

> タスクが長く複雑であるほど、Fable 5と他モデルの差は大きくなる

事例も添えられています。早期テストに参加したStripeは、5,000万行のRubyコードベースで、通常ならチームで2ヶ月以上かかるコードベース全体のマイグレーションを1日で完了したと報告しています。

レガシー診断は、この「長く複雑なタスク」の典型です。巨大なコードベースを読み、文脈を保持したまま、離れたファイル間の矛盾を突き合わせる作業だからです。Opusまでのモデルとどれだけ変わるのか、試す価値があります。

なお、Pro/Max/Team(および一部Enterprise)プランでは、7月12日まで週間利用上限の50%枠でFable 5を利用できます(当初の期限7月7日から延長)。以降は利用クレジット経由になるので、追試するなら期限内が安く済みます。

## なぜ「診断」なのか

レガシー刷新プロジェクトの失敗は、多くの場合、実装フェーズではなく「わからないまま見積もる」段階で始まっています。見積もりの前提が、そもそもこうだからです。

* 設計書は初版のまま10年以上更新されていない
* 有識者は退職済み。あるいは「あの人しか知らない」が3人に分散している
* コードだけが唯一の真実。しかし誰も全体を読めない

この状態で立てた移行計画は、走り出してから「聞いていなかった挙動」に次々ぶつかります。中でも厄介なのが、私が**仕様化された不具合**と呼んでいるものです。実装ミスとして生まれたにもかかわらず、長年の運用で業務側がその挙動に依存してしまい、直すと業務が壊れるようになったコードを指します。端数処理の不可解な切り捨て、握りつぶされた例外、コメントと矛盾する条件分岐などです。ドキュメントにはまず書かれておらず、コードの中から発掘するしかありません。そして、刷新の失敗要因の常連です。

「コードから真実を掘り起こす」という営み自体には、国際的な系譜があります。私はこれを日本のSIerの現場文脈に再構成し、**レガシー考古学**と呼んでいます。仕様書なきシステムをAIで発掘し、安全に刷新するための方法論です。今回の検証は、この発掘作業の主力道具が世代交代するのかどうかの確認です。

## 検証の設計

検証対象は、実案件の顧客コードではなくOSSのStruts 1.xアプリケーションにしました。理由は二つあります。守秘義務の制約なしに結果をそのまま見せられることと、読者が同じ手順で追試できることです。Struts 1.xは、公式サポート終了から10年以上が経った今も日本の基幹系に現役で残る、古いフレームワークの代表格でもあります。

* **対象**：Mifos(旧版、1.12-SNAPSHOT)。マイクロファイナンス機関向けの基幹業務システム(融資、貯蓄、返済、会計)。リポジトリは `github.com/mifos/head`(現在はApache Fineractに移行済みで、旧コードはHISTORICALとして保存)。規模はJava約3,300ファイル、約47万行(実効約29万行)、JSP 310枚、Struts Action 76ファイル。技術スタックはStruts 1.2.7 + Spring 3.1.0 + Hibernate 3.6.8 + MySQL、Java 1.6
* **選定理由**：①金融ドメインは仕様化された不具合が業務に焼き付きやすい(利息、端数、返済スケジュール) ②45の金融機関、110万顧客で実運用された「玩具ではない」規模と実績 ③cloneひとつで誰でも追試できる
* **環境**：Claude Code(CLI)。メインは `claude-fable-5`、走査ワーカーはSonnetのサブエージェント
* **比較対象**：発掘フェーズと同一のプロンプトを、Claude Opus 4.8で別のクローンに実行

評価軸は4つに絞りました。

1. **構造把握の精度**：アーキテクチャと「地層」(開発時期の異なるコード層)をどこまで正確に読むか
2. **発掘力**：仕様化された不具合の候補を、どれだけ具体的に、根拠付きで掘り当てるか
3. **対Opus 4.8差分**：同じプロンプトでの深さ、見落とし、幻覚の差
4. **時間**：所要時間と、人手で同等の調査をした場合の工数との対比

なお、診断プロンプトは脆弱性の悪用ではなく挙動の理解を求めるものなので、Fable 5のセーフガード(高リスク要求を拒否する分類器)に抵触する場面は設計上ほぼ想定されません。実際、今回の全プロンプトを通してrefusal(拒否)は一度も発生しませんでした。

## 診断プロンプト

実案件で使っている診断フローの入口部分を、OSS検証用に整えたものです。Claude Codeで対象リポジトリを開き、順に投入してください。

### プロンプト1：地層調査(全体像の把握)

```
あなたはレガシーシステムの調査を専門とするアーキテクトです。
このリポジトリを調査し、以下を報告してください。コードの変更は行わず、調査のみです。

1. 技術スタック(フレームワーク、主要ライブラリ、Javaバージョン、ビルド方式)と、それぞれのバージョンの根拠となるファイル
2. アーキテクチャ構造(レイヤー、主要コンポーネント、依存の向き)
3. 規模感(クラス数、実効コード行数、Action/JSP/DAO数)
4. 「地層」の分析: コードスタイル・命名規則・設計パターンの変遷から、開発時期の異なる層を推定し、各層の特徴と推定根拠を示す
```

### プロンプト2：動脈調査(データフローと外部接続)

```
このシステムの「動脈」を調査してください。

1. 主要な業務データの流れ: 画面入力からDB永続化までの代表的な経路を3つ、クラス名・メソッド名レベルで追跡
2. 外部接続点の全列挙: DB接続、ファイルI/O、外部API/ソケット、バッチ連携、メール送信など。設定ファイルとコードの両方から拾うこと
3. トランザクション境界: どこで開始・コミット・ロールバックされるか。境界が曖昧な箇所や、二重コミット・コミット漏れの疑いがある箇所を指摘
4. 刷新時に切り離しが難しい「癒着」箇所トップ3とその理由
```

### プロンプト3：遺物発掘(仕様化された不具合)

```
このコードベースで「バグに見えるが、既存の挙動として業務が依存している可能性のある箇所」を発掘してください。

観点:
- 例外を握りつぶしている箇所(catchして何もしない/ログのみ)
- コメントと実装が矛盾している条件分岐
- 丸め・切り捨て・日付・文字コード処理で、一般的な仕様と異なる実装
- コピペで増殖したロジックのうち、1箇所だけ微妙に異なるもの
- マジックナンバーやハードコードされた例外的分岐(特定コード値だけ特別扱い等)

各箇所について以下の形式で報告:
1. ファイルと行番号
2. 何が「おかしい」のか(技術的説明)
3. これが「仕様」である可能性(業務がこの挙動に依存しているとしたら、どんなシナリオか)
4. 刷新時の扱い(現行踏襲すべきか、修正すべきか、誰に何を確認すべきか)

確信度の高い順に並べ、根拠の薄い推測はその旨を明記してください。
```

### プロンプト4：発掘報告書(刷新優先度マップ)

```
ここまでの調査結果を統合し、刷新プロジェクトの計画者(PM・情報システム部門)向けの発掘報告書を作成してください。

1. システム全体の健全性サマリ(3行)
2. 刷新優先度マップ: 主要コンポーネントを「業務価値」×「技術的リスク」の2軸で分類し、着手順序を提案
3. 「仕様化された不具合」一覧と、それぞれの確認先(業務部門に聞くべきこと)
4. 見積もりの不確実性が高い領域トップ3と、不確実性を下げるために追加で必要な調査
5. このシステムを人手だけで同レベルに調査した場合の工数感(あなたの推定)
```

## 検証結果

### 構造をどこまで読めたか

地層調査(プロンプト1)は約4分で完了しました。技術スタックの特定は当然として、目を引いたのは、「このシステムは刷新の途中で凍結されている」という診断を数字付きで立ててきたことです。

Web層を数えると、Strutsをimportする旧世代のファイル174に対し、新世代のSpring @Controllerは124ファイルで、約4割が新方式へ移っています。ところが永続化層では、旧世代のHibernate XMLマッピング(.hbm.xml)176ファイルに対し、新世代のJPA @Entityは34ファイルにとどまります。UIの刷新は進んだのに、データ層の84%は旧方式のまま取り残されている、という非対称です。Fable 5はこれを「UIの刷新が先行し、データ層の刷新が追いつかないまま開発が停止した『上だけ新しい地層』」と読み、傍証として、Copyright表記を持つ1,920ファイルのうち1,905ファイルが「2005-2011」の範囲表記で一斉に止まっている事実を挙げてきました。移行の途中で凍結された遺跡、という診断です。

リポジトリ全体だけでなく、ファイル一つの内部の地層も読んでいます。通貨クラス `Money.java` では、日割り計算のメソッドだけがインデントも異質でチケット番号のコメント付きであることから、「後年、別の手が継ぎ足したパッチ」だと指摘してきました。全体をgrepしても見えない、精読しないと拾えない層です。

動脈調査(プロンプト2)では、返済処理が「HTTPリクエスト全体を1トランザクションとする」というStruts時代の暗黙前提に依存していることを突き止めました。しかも同じ返済業務に `makeEarlyRepayment`(リクエスト境界でコミット)と `makeEarlyRepaymentWithCommit`(自前コミットだがロールバックなし)という**二重玄関**があり、どちらから呼ばれるかでトランザクションの安全性が変わります。刷新の最大の障壁がどこかを、着手前に名指しできたことになります。

### 「仕様化された不具合」を掘り当てたか

発掘フェーズ(プロンプト3)は約15分かかりました。Fable 5は確信度「高」の候補だけで10件、確信度が中低のものや、開発者自身がコメントで認めているFIXME群(いわば自白付き)を含めると20件超を、いずれもファイルと行番号、根拠コード付きで掘り出しました。走査ワーカーのSonnetが例外握りつぶし49件とFIXME系231件を広く拾い、メインのFable 5が上位候補を精読して裏取りする、という分業がそのまま機能したかたちです。

主役はこれです。**会計仕訳の勘定科目が、「ランダムに」選ばれていました。**

`BaseAccountingEntry.java` の `getGLcode()` は、仕訳の借方/貸方に使う勘定科目コードを決めるメソッドです。ところがその実装は、候補集合(`HashSet`)を全件ループして、最後に出てきた要素を返すだけです。`HashSet` の反復順序は仕様上不定なので、勘定科目のマッピングが下層に複数ぶら下がっている場合、どの科目に仕訳が切られるかは実行環境次第で変わることになります。

驚いたのはここからです。このメソッドのJavadoc自身が、「返されるアカウントコードはランダムに選ばれる(集合のイテレータ実装によって決まる)」と書いていました。**バグではなく、開発者が認識し、文書化までされた挙動**だったのです。Fable 5はコードの挙動とJavadocの記述を突き合わせて、これを仕様化された不具合の教科書的な事例と診断しました。

なぜこれが「仕様」たりえたのか。総勘定元帳の合計は科目がどれに転んでも一致するため、実害が「科目内訳のブレ」に留まるからです。Fable 5の分析では、導入先の金融機関は下層科目を1対1でマッピングすることでこの挙動を回避してきたと推定されます。しかし既存帳簿の科目内訳は、このランダムな挙動の産物です。決定論的なロジックに直すと、過去データと帳尻が合わなくなります。知らずに直すと壊れる、というタイプのコードでした。

他にも、残額が0.09以下なら「完済」とみなす端数免除ルール、手数料IDが未採番のときに `hashCode()` をIDの代わりにして支払いを配分する応急処置(コメントに "special workaround for MIFOS-4517" と明記)、コピペで複製された双子クラスの間で初回利息の日割り計算の発動条件が食い違う(片方は返済2回で発動、もう片方は3回)といった、金額に直結する遺物が並びました。

外した例も書いておきます。走査ワーカーは当初、丸め処理を詳述した104行のコメントブロックを「埋葬された旧コード(コメントアウトの塊)」の候補として拾いました。しかしFable 5が現物を精読した結果、これは埋葬コードではなく、丸め仕様を解説した優良なJavadocで、むしろ刷新時には仕様書として使える最重要文書だと判定し、候補から除外しました。機械的スキャンの偽陽性を、精読が打ち消したわけです。「広く拾う層」と「精読して裏取りする層」の分業は、AIパイプラインの内側でも機能していました。

### Opus 4.8との差分

発掘フェーズ(プロンプト3)を、別のクローンでOpus 4.8でも実行しました。結論から言うと、勝ち負けというより「発見が相補的だった」というのが実態です。

| 評価軸 | Fable 5 | Opus 4.8 |
| --- | --- | --- |
| 地層推定の精度 | 新旧比率を定量化し「凍結された遺跡」と診断 | (発掘フェーズのみ実行のため対象外) |
| 発掘件数(確信度高) | 10件+自白FIXME群 | 10件+埋葬遺構13件を列挙 |
| 根拠提示の具体性 | コードとJavadocを突き合わせて裏取り | 該当行の実コード引用で全件裏取り |
| 幻覚と誤指摘 | 偽陽性1件を精読で自己除外 | 偽陽性の除外を明記 |
| 所要時間 | 発掘 約15分(全4工程で約27分) | 発掘 約21分 |
| 検出の重複 | 共通の発見は約5件 | 残りは各モデル固有の発見 |

Fable 5だけが掘り当てたのは、仕訳科目のランダム選択、PARバッチのトランザクション迷宮(commitなしの生SQL直叩き)、センター実績集計の "THIS METHOD DOES NOT WORK" という自白コメントです。一方Opus 4.8は、最終丸めのデフォルトだけが四捨五入ではなく常に切り上げ(CEILING)になっている会計ルール、別々の権限が同一ID(142)を共有している権限マスタ、利息計算の年間日数が360/365に固定され閏年を弾く設計を掘り当てました。どちらも単独では見つけきれなかった遺物で、**2モデルを併用すると発掘の網羅性が上がる**という、実務上の価値がある副産物が得られました。

つまり、発掘だけならOpus 4.8でも十分に戦えます。では、Fable 5ならではの強みはどこに見えたのか。地層調査で見せた、フレームワーク混在比率の定量化やファイル内の地層の読み取り、つまり長く複雑な文脈の保持です。もっとも、地層調査はOpus 4.8では走らせていないので、ここは直接比較ではありません。47万行を貫いて構造の来歴を組み立てる作業でこそ「タスクが長く複雑であるほど差が大きくなる」という設計思想が効いてくる、というのが今回の手応えで、次回の検証で確かめたい点でもあります。

### 所要時間と人手工数の対比

正直に言うと、今回はトークン消費と料金の計測を取り損ねました(セッション終了後に気づいたため)。ここは次の検証で `/cost` を記録して追記します。ただ、時間と工数の対比は今回の結果だけでも語れます。

Fable 5は全4フェーズ(地層、動脈、発掘、報告書)を通しで約27分、Opus 4.8は発掘フェーズを約21分で完了しました。一方、発掘報告書(プロンプト4)自身が見積もった「未経験のシニアエンジニアが人手で同等の調査(層序の定量化、代表経路のクラスレベル追跡、遺物の裏取り、統合報告書まで)をする場合」の工数は、**約18〜28人日**(1人で4〜6週間)です。AI自身の推定なので割り引いて読む必要はありますが、内訳には根拠があります。grep的な機械走査は人手でも可能で、支配項になるのは候補の裏取り読解(1件30分〜2時間が数十件)のほうだ、という分析です。

AIが圧縮しているのは、この「読んで裏取りする」部分です。数週間かかっていた一次調査が数十分で終わる。これが、料金ではなく時間の軸で見た変化の実像です。

## PMと情シスにとって何が変わるか

レガシー刷新に関わる意思決定者にとっての含意を3点挙げます。

第一に、診断フェーズの経済構造が変わります。従来、着手前にシステムの実態を知る方法は「有識者ヒアリング+ドキュメント精査+サンプリング的なコードリーディング」で、数週間から数ヶ月の人件費がかかる割に、サンプリングである以上、網羅性はありませんでした。AIによる発掘はこの一次調査を桁で圧縮し(今回の検証では数十分)、人間の役割を「調査」から「検証と業務突き合わせ」に移します。有識者の時間は、AIが立てた仮説を業務と突き合わせる作業にだけ使えばよくなります。

第二に、「読めないから現行踏襲」という意思決定の質が変わります。現行踏襲は本来、中身を理解した上で選ぶ戦略のはずですが、実際には「調べるコストが高すぎるから仕方なく」選ばれてきました。調査コストが桁で下がるなら、踏襲する箇所と直す箇所を根拠を持って仕分けできます。仕様化された不具合を「知らずに引き継ぐ」のと「知った上で引き継ぐ」のとでは、プロジェクトのリスクとして全くの別物です。

第三に、それでもAIは仮説製造機であって、裁定者ではありません。発掘された「仕様化された不具合」候補が本当に業務依存なのかを決めるのは、業務部門への確認という人間の仕事です。AIの網羅性と人間の裁定を分業させる設計は、道具がFable 5に世代交代しても変わりません。むしろ仮説の量と質が上がるほど、「何を誰に確認するか」を設計する力が成果を分けます。

## まとめ

仕様書なきレガシーシステムの診断は、網羅的な一次発掘を数十分で終えられる段階に入りました。変わったのは一次診断の速度と深さです。Fable 5は金融システムの奥に眠る仕様化された不具合を根拠付きで掘り出し、Opus 4.8と併用すれば取りこぼしはさらに減ります。変わらなかったのは、掘り出した仮説の裁定が人間の仕事だという構造です。道具が世代交代しても、業務部門と仮説をどう裁定するかが人間の主戦場です。

本記事の診断プロンプトは、実案件で使っている手法の入口部分です。「自社のあのシステムでやるとどうなるのか」に関心のある方向けに、このような記事を連載します。
