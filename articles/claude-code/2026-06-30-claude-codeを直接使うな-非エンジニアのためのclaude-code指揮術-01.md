Claude Codeに何を頼めばいいか、毎回手が止まる。  
怖いのは、使えないことではなく、雑に頼むと勝手に作業が進むこと。  
非エンジニアが最初に覚えるべきなのは、Claude Codeの操作ではなく「指揮の出し方」です。

Claude Codeは、普通のチャットAIではありません。

質問に答えるだけではなく、ファイルを読み、編集し、コマンドを実行できる「作業するAI」です。だから、設計相談も、方針決めも、実装指示の整理も、全部Claude Codeの同じ会話でやると混乱しやすくなります。

ここで言う「直接使うな」は、Claude Codeを使うなという意味ではありません。

Claude Codeに、考える仕事まで全部やらせるな、という意味です。

小さな文言修正なら、直接頼んでもいい。でも、繰り返し作業、複数ファイルの修正、サイト運用、記事制作、業務フロー化のような作業では、先に「Claude Codeへのプロンプトを作るAI」を挟んだ方が安定します。

僕はこれを「指揮官AI」と呼んでいます。

この記事では、Anthropic公式サイトのClaude Code関連ドキュメントを中心に、「指揮官AI」の作り方と、Claude Code用プロンプトを作らせるためのコピペ用プロンプト10個を用意しました。

## Claude Code初心者が詰まる理由

Claude Code初心者が詰まる原因は、プロンプトを知らないことだけではありません。

本当の問題は、

何を頼むべきか  
どこまで任せていいか  
何を禁止すべきか  
最後に何を確認すべきか

この判断を、毎回自分でやろうとしていることです。

非エンジニアにとって、これはかなり難しいです。

たとえば「問い合わせフォームの文言を変えたい」と思ったとします。  
でも、その作業がどのファイルに関係するのか、設定を触る必要があるのか、確認コマンドを走らせるべきなのか、公開前に何を見るべきなのかは、すぐには判断できません。

そこでClaude Codeに直接、

「この文言を変えて」

と頼む。

Claude Codeは作業できます。  
でも、作業範囲が曖昧なまま進むので、意図しない変更が入る可能性があります。

だから初心者が最初に作るべきなのは、10個のプロンプト暗記ではありません。

Claude Codeに渡すプロンプトを作ってくれる「指揮官AI」です。

## 指揮官AIとは何か

指揮官AIは、Claude Codeの代わりに作業するAIではありません。

役割は、Claude Codeへ渡す前のプロンプトを整理することです。

指揮官AIは、Claude Codeの中に作りません。  
作る場所は、通常のClaudeチャット側です。

できればClaudeのプロジェクトに専用の場所を作り、そこに指揮官AIのプロンプトを入れます。

そこで設計相談をする。  
Claude Codeに渡すプロンプトを作る。  
Claude Codeには、その完成したプロンプトだけを貼る。

この順番にすると、Claude Codeの会話を設計議論で汚さずに済みます。

あなたが指揮官AIに伝えるのは、仕事の言葉でOKです。

「LPの文言を直したい」  
「記事制作フローを整理したい」  
「この作業をClaude Codeに頼んでいいか判断したい」  
「毎回同じミスが出るからルール化したい」

こう伝えると、指揮官AIがClaude Code用のプロンプトに変換します。

Claude Code用のプロンプトには、次の要素が必要です。

目的。  
作業範囲。  
変更していい場所。  
変更してはいけない場所。  
禁止事項。  
実行前の確認。  
完了条件。  
検証方法。  
人間が最後に確認する点。

これを毎回、自分で考えるのは大変です。

でも指揮官AIを挟めば、あなたは「やりたいこと」を話すだけで、Claude Codeに渡せる形まで整えてもらえます。指揮官AIを挟むと何が変わるか

指揮官AIを挟むと、あなたが毎回Claude Code用プロンプトを考える必要がなくなります。

あなたは、仕事の言葉で話す。  
指揮官AIが、作業範囲・禁止事項・確認条件を整理する。  
Claude Codeは、整理されたプロンプトだけを受け取って作業する。

この分離があるだけで、かなり安定します。

特に非エンジニアにとって大きいのは、危ない操作を事前に止められることです。

削除していいのか。  
本番に触っていいのか。  
外部送信していいのか。  
設定ファイルを変えていいのか。  
検証せずに終わっていないか。

こういう判断を、Claude Codeに作業させながら考えるのは危険です。

先に指揮官AIと整理してから、Claude Codeへ渡す。  
これが、非エンジニア向けのClaude Code運用です。

## なぜClaude Codeに直接考えさせない方がいいのか

Claude Codeの会話の中で、

「どう作ればいい？」  
「こっちの方がいい？」  
「やっぱりこの条件も入れて」  
「さっきの案を少し変えて」

と議論を続けると、その議論の履歴が同じ会話に残ります。

そして実装に入る頃には、Claude Codeの中に、検討途中の案、ボツにした案、途中で変えた条件、雑談に近い確認が混ざっています。

これは、作業机の上に企画書、議事録、メモ、修正前の図面を全部広げたまま、細かい作業をしようとしている状態です。

作業しにくいに決まっています。

だから、考える場所と作る場所を分けます。

考える場所は、指揮官AI。  
作る場所は、Claude Code。  
残す場所は、CLAUDE.mdやSKILL.mdなどのルールファイル。

この分離ができると、Claude Codeには「議論の過程」ではなく「決まったプロンプト」だけを渡せます。

## 指揮官AIに入れる基本プロンプト

まずは、Claudeのプロジェクトなど、継続して使える場所に指揮官AIを作ります。

以下を指示欄に入れます。

```
あなたは「Claude Code指揮官AI」です。

役割は、ユーザーのやりたいことを整理し、Claude Codeに渡せるプロンプトへ変換することです。
あなた自身は実装しません。
設計、確認、分解、Claude Code用プロンプトの作成を担当します。

目的:
非エンジニアでもClaude Codeに安全に作業を依頼できるようにする。

基本方針:
- Claude Codeは作業するAIとして扱う
- 設計相談と実装作業を同じ場所で混ぜない
- 曖昧な依頼を、目的・範囲・禁止事項・完了条件・検証方法に分解する
- 削除、本番操作、外部投稿、課金、秘密情報へのアクセスは必ず人間確認にする
- 「いい感じに」「適当に」「全部やって」のような曖昧な依頼を具体化する
- 実在しないURL、コマンド、仕様、数値は作らない
- 不明点は断定せず、確認事項として出す

出力するもの:
1. 作業の整理
2. Claude Codeで進めてよいか
3. 危険な点
4. 先に確認すべきこと
5. Claude Codeに渡すコピペ用プロンプト
6. 人間が最後に確認すべき点

Claude Codeに渡すプロンプトには、必ず以下を含める:
- 目的
- 作業範囲
- 変更してよい範囲
- 変更してはいけない範囲
- 禁止事項
- 進め方
- 完了条件
- 検証方法
- 最後の報告形式

出力ルール:
- Claude Codeに渡すプロンプトは、コピペできるコードブロックで出す
- 補足説明や記入例はコードブロックに入れない
- コードブロックに入れるのは、実際にClaude Codeへ貼る本文だけにする
- まず安全性を優先し、危険な依頼は「それはNGです」と止める
```

これで、Claude Codeに直接投げる前に、指揮官AIへ相談できるようになります。

## 指揮官AIに渡すプロンプト10個

ここから紹介するのは、Claude Codeに直接貼るプロンプトではありません。

指揮官AIに渡して、Claude Code用の安全なプロンプトを作らせるためのプロンプトです。

つまり流れはこうです。

あなた  
→ 指揮官AIに相談する

指揮官AI  
→ Claude Code用プロンプトを作る

Claude Code  
→ 整理されたプロンプトを受け取って作業する

この順番にすることで、Claude Codeの中で設計議論を長々とやらずに済みます。

## 1. 質問だけしたい時

使う場面:

まだ作業させず、意味や方針だけを知りたい時。

指揮官AIに渡すプロンプト:

```
Claude Codeに質問だけしたいです。
作業は禁止したまま、回答だけをもらうためのClaude Code用プロンプトを作ってください。

知りたいこと:
ここに確認したい内容を入れてください。

条件:
- ファイル編集は禁止
- コマンド実行は禁止
- 削除、外部送信、コミット、PR作成は禁止
- 結論、理由、注意点、作業するなら先に確認すべきことを出させる
```

補足:

「知りたいこと」には、今確認したい内容を普通の言葉で入れます。  
エラーの意味を知りたい、修正方針が危険か見たい、触っていいファイルか確認したい、などです。

## 2. Claude Codeでやっていい作業か判断したい時

使う場面:

削除、公開、外部送信、複数ファイル変更など、少し怖い作業の前。

指揮官AIに渡すプロンプト:

```
この作業をClaude Codeに任せてよいか判断したいです。
危険な操作、先に確認すべきこと、人間が承認すべきことを分けたうえで、Claude Code用プロンプトを作ってください。

やりたいこと:
ここにやりたい作業を入れてください。

条件:
- まだ作業させない
- 作業可否の判断を先にさせる
- 危険な操作は人間承認にする
- Claude Codeに任せない方がいい操作も出させる
```

補足:

Claude Codeは便利ですが、何でも任せていいわけではありません。  
危ない作業ほど、先に「任せてよいか」を判断させるのが安全です。

## 3. どこを触る作業なのか調べたい時

使う場面:

やりたいことは決まっているけれど、どのファイルが関係するかわからない時。

指揮官AIに渡すプロンプト:

```
この作業で、どのファイルが関係するかをClaude Codeに調査させたいです。
編集はまださせず、関連箇所の調査だけを行うClaude Code用プロンプトを作ってください。

やりたいこと:
ここに実現したいことを入れてください。

条件:
- まだ編集しない
- 関係しそうなファイルを出させる
- それぞれのファイルが関係しそうな理由を出させる
- 最初に確認すべき場所を出させる
- 触らない方がよさそうな場所も出させる
```

補足:

見た目は小さな文言変更でも、画面、設定、翻訳、テスト、ドキュメントが関係することがあります。  
非エンジニアほど、まず関連箇所の調査だけを挟む方が安全です。

## 4. 実装前に計画だけ出してほしい時

使う場面:

複数ファイルにまたがる作業、大きめの修正、失敗した時の影響が大きい作業。

指揮官AIに渡すプロンプト:

```
Claude Codeに実装前の計画だけを出させたいです。
まだ編集させず、調査と変更計画だけを行うClaude Code用プロンプトを作ってください。

目的:
ここに目的を入れてください。

条件:
- 関連ファイルを調査させる
- 変更計画を出させる
- リスクを出させる
- 検証方法を出させる
- 編集は承認後にする
- 削除、コミット、PR作成はまだ禁止する
```

補足:

大きな作業は、調査、計画、実装、検証に分けた方が安定します。  
いきなり実装させるのではなく、まず計画だけ出させます。

## 5. 小さな修正を安全に頼みたい時

使う場面:

文言変更、README修正、軽い表示修正など。

指揮官AIに渡すプロンプト:

```
小さな修正だけをClaude Codeに安全に頼みたいです。
余計な変更を防ぐため、変更範囲と禁止事項を明確にしたClaude Code用プロンプトを作ってください。

修正内容:
ここに修正内容を入れてください。

変更してよい範囲:
ここに変更してよい範囲を入れてください。

条件:
- 直接関係ないファイルは触らせない
- 設定ファイルやlockファイルは原則触らせない
- 削除操作は禁止する
- 変更内容を最後に要約させる
- 未確認点があれば明記させる
```

補足:

「軽く直して」は危険です。  
軽いの基準は、人間とAIでズレます。  
変更していい範囲と、触ってはいけない範囲を先に決めます。

## 6. バグ修正を頼みたい時

使う場面:

エラー、表示崩れ、想定通りに動かない機能を直したい時。

指揮官AIに渡すプロンプト:

```
Claude Codeにバグ修正を頼みたいです。
原因調査、修正方針、必要最小限の変更、検証までを含めたClaude Code用プロンプトを作ってください。

症状:
ここに起きている問題を入れてください。

再現手順:
ここに再現手順を入れてください。

期待する動作:
ここに本来どうなってほしいかを入れてください。

条件:
- まず原因を調査させる
- 原因候補を出させる
- 修正方針を出させる
- エラーを握りつぶす修正は禁止する
- 修正後に確認させる
- 最後に原因、変更内容、検証結果、残るリスクを報告させる
```

補足:

バグ修正では、エラー表示だけ消す修正が一番危険です。  
症状、再現手順、期待する動作を分けると、Claude Codeが目的を誤解しにくくなります。

## 7. 既存の書き方に合わせてほしい時

使う場面:

新しい実装や修正を、今あるファイルの書き方に合わせたい時。

指揮官AIに渡すプロンプト:

```
Claude Codeに既存の書き方に合わせて修正させたいです。
新しい設計を勝手に足さず、既存パターンに寄せるClaude Code用プロンプトを作ってください。

やりたいこと:
ここにやりたいことを入れてください。

参考にしてほしい既存ファイル:
ここに参考ファイルがあれば入れてください。

条件:
- 既存の命名規則に合わせる
- 既存のフォルダ構成に合わせる
- 既存のテスト形式に合わせる
- 新しい設計パターンを勝手に追加しない
- 直接関係ないリファクタはしない
- 実装前に参考にした既存パターンを説明させる
```

補足:

初心者は、正しい書き方をゼロから考える必要はありません。  
すでに動いている近いファイルに合わせる方が安全です。

## 8. 修正後に確認してほしい時

使う場面:

修正後に、壊れていないか確認したい時。

指揮官AIに渡すプロンプト:

```
Claude Codeに修正後の確認をさせたいです。
使える確認方法を先に調べ、安全な範囲で検証するClaude Code用プロンプトを作ってください。

確認したいこと:
ここに確認したい内容を入れてください。

条件:
- 使える確認コマンドを先に調べさせる
- 実行前に、実行するコマンドを提示させる
- 安全なコマンドだけ実行させる
- 失敗した場合は原因を調査させる
- 勝手に大きな修正へ進ませない
- 最後に実行した確認、結果、人間が確認すべき点を出させる
```

補足:

テストは自動確認、ビルドは動く形に組み立てる確認、lintはルール違反チェックです。  
ただし、実在しないコマンドを作らせないよう、先に確認させるのが安全です。

## 9. 変更内容を要約してほしい時

使う場面:

作業後に、何を変えたか自分や他の人に説明したい時。

指揮官AIに渡すプロンプト:

```
Claude Codeに今回の変更内容を要約させたいです。
実際の差分に基づき、やっていない検証や存在しない情報を混ぜないClaude Code用プロンプトを作ってください。

条件:
- 実際の差分に基づいて書かせる
- 背景、変更内容、影響範囲、検証結果、残るリスクを出させる
- やっていない検証は書かせない
- 実在しないURLやチケット番号は作らせない
- PR作成やコミットはまださせない
```

補足:

PRは、変更内容をレビューしてもらうための仕組みです。  
非エンジニアは、まず「変更説明文」くらいに考えればOKです。

## 10. 次回のために作業ログを残したい時

使う場面:

作業が長くなった時、途中で止める時、次回また続きから始めたい時。

指揮官AIに渡すプロンプト:

```
Claude Codeに作業ログと次回引き継ぎを作らせたいです。
事実と推測を分け、次回そのまま再開できるClaude Code用プロンプトを作ってください。

条件:
- 目的を出させる
- 完了したことを出させる
- 変更したファイルを出させる
- 実行した検証を出させる
- 未完了のことを出させる
- 次にやることを出させる
- 注意点を出させる
- 人間が確認すべきことを出させる
- まだ保存せず、まず本文案だけ出させる
```

補足:

長い作業ほど、途中の判断が会話の中に埋もれます。  
最後に作業ログを出しておくと、次回の再開がかなり楽になります。

## まとめ

Claude Codeを使いこなすコツは、プロンプトを10個暗記することではありません。

大事なのは、Claude Codeに直接考えさせすぎないことです。

考える場所を分ける。  
作る場所を分ける。  
判断基準を残す。  
作業範囲と禁止事項を明確にする。  
最後は人間が確認する。

Claude Codeは作業するAIです。  
だからこそ、非エンジニアは「作業を直接頼む人」ではなく、「作業プロンプトを設計して渡す人」になる方が強いです。

Claude Codeを直接使うな。  
これは、Claude Codeを避けろという意味ではありません。

Claude Codeを、作業員として正しく指揮しようという意味です。

AI活用やClaude Codeの業務導入を、自社の作業に合わせて整理したい方は、DMでご相談ください。
