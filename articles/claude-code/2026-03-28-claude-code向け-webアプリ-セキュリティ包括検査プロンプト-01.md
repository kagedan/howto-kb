# はじめに

Webアプリ診断用AIプロンプトを公開するにあたり、内容を整理しました。  
本当はもっと単純化して、誰でもすぐ使える形にしたかったのですが、セキュリティ用途という性質上、権限確認や誤用防止の説明がどうしても長くなりました。  
少し使い方は難しめですが、その分、安全側に倒れるよう整理しています。  
一番のユースケースはClaude Code上でとりあえずWebアプリケーションを検査する、というのがよいと考えています。

# README

## ⚠️ 重要な注意事項（必ずお読みください）

### 自分が管理するサイトにのみ使用してください

このプロンプトは **自分自身が所有・管理するWebサイト、または書面による明示的な許可を得たサイト** に対してのみ使用してください。

他者のWebサイトに対して無許可でセキュリティ検査を実施した場合、たとえ善意であっても以下の法律に抵触する可能性があります:

* **不正アクセス禁止法**（日本）
* **Computer Fraud and Abuse Act（CFAA）**（米国）
* **Computer Misuse Act**（英国）
* その他各国のサイバー犯罪関連法

「脆弱性を見つけて報告するつもりだった」という意図は、法的には免責事由になりません。バグバウンティプログラムに参加する場合でも、当該プログラムの規約・スコープ・ルールを必ず確認してください。

### SNS・ブログ・GitHubで公開する場合の追加注意

このテンプレートを公開すると、**あなた以外の第三者が文脈なしに利用する可能性があります。**

* このテンプレート自体は、検査権限や法的許可を与えるものではありません。
* 公開用の投稿やREADMEには、**実際の対象URL、内部ホスト名、資格情報、トークン、Cookie、顧客データ、秘密情報を含めないでください。**
* 公開配布版では、プレースホルダーや前提条件が未記入のままなら、AIは**検査を開始せず、前提確認チェックリストだけを返す**設計を推奨します。
* 第三者がそのままコピペしても安全側に倒れるよう、`AUTHORIZED_SCOPE`、`MODE`、`TEST_LEVEL`、`TEST_ACCOUNTS`、`REQUEST_BUDGET` は必ず明示させてください。

### 大量リクエストに関する注意

**モードB（エージェント型）** や **L2/L3 の検査強度** では、AIが対象サイトに対して能動的にHTTPリクエストを送信します。

* **本番環境で実施する場合は細心の注意を払ってください。** 大量のリクエストはサービスの応答速度低下、WAFによるIPブロック、ログの肥大化、監視アラートの誤発報などを引き起こす可能性があります。
* **可能な限り、ステージング環境や開発環境で実施してください。**
* **リクエスト予算（REQUEST\_BUDGET）を必ず設定してください。** 公開配布版では未設定のまま検査を開始しない想定です。下記の目安は、利用者が明示的に記入するための参考値として使ってください。
* **深夜帯やメンテナンス時間帯の実施を推奨します。**
* WAFやCDNが検査トラフィックを攻撃とみなしてIPをブロックする可能性があります。固定IPからの検査が困難になる場合は、事前にWAFのホワイトリスト設定を検討してください。

### 資格情報・秘密情報の取り扱い

* **本番資格情報をAIへ入力しないでください。**
* 使用するのは、**専用・短命・最小権限** のテストアカウントのみにしてください。
* 検査後は、テストアカウントのパスワード、APIキー、トークンをローテーションしてください。
* モードAでログやレスポンスを渡す場合も、`Set-Cookie`、`Authorization`、JWT、メールアドレス、電話番号などはマスクしてから共有してください。
* 公開チャット、SNS、Issue、PRコメント等に、未マスクの認証情報や個人情報を貼らないでください。

## 基本的な使い方

### Step 1: プレースホルダーを埋める

`web-security-prompt.md` を開き、以下のプレースホルダーを書き換えてください。

| プレースホルダー | 説明 | 未指定時の挙動（公開版） |
| --- | --- | --- |
| `{TARGET_URL}` | 検査対象のURL | （必須。未指定なら検査しない） |
| `{AUTHORIZATION_ASSERTION}` | 所有・管理または明示的許可の事実 | 前提確認のみ返す |
| `{AUTHORIZED_SCOPE}` | 許可された検査範囲 | 前提確認のみ返す |
| `{MODE}` | モードA（アナリスト型）/ モードB（エージェント型） | 前提確認のみ返す |
| `{TEST_LEVEL}` | L1 / L2 / L3 | 前提確認のみ返す |
| `{TEST_ACCOUNTS}` | テスト用アカウント情報 | 省略せず、下記3パターンのいずれかを明記 |
| `{REQUEST_BUDGET}` | リクエスト上限 | 前提確認のみ返す |

> `{AUTHORIZATION_ASSERTION}` の記載例:
>
> * `この対象は自分が所有・管理するサイトです`
> * `2026-03-28付で書面承認済みです（チケット SEC-123）`

**`{REQUEST_BUDGET}` のレベル別目安:**

| レベル | 同時接続 | レート | 総リクエスト | 根拠 |
| --- | --- | --- | --- | --- |
| **L1 Passive** | 1 | 30 req/min | 30 | トップ＋主要ページ＋公開ファイルHEAD確認程度 |
| **L2 Safe Active** | 1 | 30 req/min | 150 | L1に加え、入力検証・CORS確認・API列挙を含む |
| **L3 Controlled Active** | 明示指定 | 明示指定 | 明示指定 | サイト規模と検査範囲に応じて利用者が決定 |

> 公開配布版では、`{REQUEST_BUDGET}` を省略したまま実査に進まない想定です。上表の値はあくまで**利用者が明示的に指定するための推奨目安**です。

**`{TEST_ACCOUNTS}` の設定パターン:**

| パターン | 記載内容 | 対象サイトの例 | 検査範囲 |
| --- | --- | --- | --- |
| **a) 認証機能なし** | `認証機能なし` | ブログ、LP、ポートフォリオ、静的サイト、公開API | Phase 3（認証・認可）をスキップし、公開面の検査に集中 |
| **b) 未提供** | `未提供` | 認証機能があるがテストアカウントを用意していない場合 | ログイン画面等の公開面のみ検査。認証後フローは断定しない |
| **c) アカウント提供** | ロールごとにメール/パスワードを記載 | 認証付きWebアプリ、管理画面、SaaS | 認可制御・権限差分・IDOR等の認証後検査が可能 |

> 公開配布版では、`{TEST_ACCOUNTS}` を空欄のままにせず、`認証機能なし` / `未提供` / `アカウント提供` のどれかを明示してください。

### Step 2: AIに貼り付ける

`web-security-prompt.md` の内容をAIチャット（Claude、ChatGPT等）に貼り付けてください。

* **モードA** の場合は、プロンプトの後にcurl結果やDevToolsのスクリーンショット等を順次提供してください。
* **モードB** の場合は、AIが自律的にアクセスして検査を開始します。
* ただし、公開配布版では必須プレースホルダーが埋まっていない限り、AIはアクセスせず前提確認のみ返す想定です。

### Step 3: フレームワーク別チェックを追加（任意）

技術スタックが判明した場合、本READMEの「フレームワーク別の詳細チェック」セクションから該当ブロックをコピーして、プロンプトの末尾に追加してください。

## 検査レベル別の使用例

> 以下のメールアドレス、パスワード、ホスト名、URLはすべて説明用のダミーです。実運用では、本番資格情報や実ユーザーデータを使わず、専用のテスト値に置き換えてください。

### L1 Passive — 「まず全体像を把握したい」（認証なしサイト向け）

ログイン機能がないサイト（ブログ、LP、ポートフォリオ、静的サイト等）の検査例です。L1 Passive はヘッダー確認・公開ファイル確認・HTML/JS解析など非侵襲的な検査のみで、状態変更は一切行いません。本番環境でも安全に使えます。

> **モードA vs モードB の選び方:**
>
> * **モードB（推奨）:** AIが自動でレスポンスヘッダー取得、robots.txt確認、HTMLソース解析等を行います。手間なく一通りのチェックが完了します。
> * **モードA:** AIは一切リクエストを送りません。自分でcurlやDevToolsで取得したデータを渡して分析してもらいます。AIにサイトへアクセスさせたくない場合に使います。

#### L1 + モードB（推奨：自動で一通りチェック）

**プレースホルダーの設定:**

| プレースホルダー | 設定値 | 設定理由 |
| --- | --- | --- |
| `{TARGET_URL}` | `https://example.com` | 自分が管理する本番サイト |
| `{AUTHORIZATION_ASSERTION}` | `この対象は自分が所有・管理するサイトです` | 所有関係を明示してfail-closedを満たす |
| `{AUTHORIZED_SCOPE}` | `https://example.com 配下のみ` | 同一オリジンに限定 |
| `{MODE}` | `モードB` | AIが自動でヘッダー取得・HTML解析等を実施 |
| `{TEST_LEVEL}` | `L1 Passive` | 閲覧・解析のみ、状態変更なし |
| `{TEST_ACCOUNTS}` | `認証機能なし` | サイトにログイン機能が存在しない |
| `{REQUEST_BUDGET}` | `同時接続 1、30 req/min、総リクエスト 30` | L1は閲覧相当なので少量で十分 |

> **なぜ30件？** L1 Passive でAIが送るリクエストは、トップページ取得、主要サブページ数件、robots.txt/sitemap.xml、公開ファイルのHEAD確認（.env, .git等）、エラーページ確認程度です。小規模サイトなら10件前後、ブログやサブページが多いサイトでも30件あれば一通りカバーできます。

**AIに貼り付けるプロンプト（web-security-prompt.md の先頭部分を書き換えた例）:**

```
あなたはWebアプリケーションのセキュリティ専門家です。
防御目的のアナリスト兼ペネトレーションテスターとして、許可された範囲内で安全に検査し、
過不足のないリスク評価レポートを作成してください。

■ 対象URL: https://example.com

■ 権限確認: この対象は自分が所有・管理するサイトです

■ 許可済みスコープ: https://example.com 配下のみ

■ テストアカウント / ロール: 認証機能なし
  （対象サイトにログイン機能は存在しません。
   Phase 3（認証・認可）は対象外とし、公開面の検査に集中してください。）

■ 利用モード: モードB

■ 検査強度: L1 Passive

■ リクエスト予算: 同時接続 1、30 req/min、総リクエスト 30

（以下、web-security-prompt.md の残りをそのまま貼り付ける）
```

**この設定で AIが自動的に行うこと（L1 Passive の範囲内）:**

* レスポンスヘッダーの取得と評価（HSTS、CSP、X-Frame-Options 等）
* HTTP→HTTPS リダイレクトの確認
* robots.txt、sitemap.xml の取得と分析
* HTMLソースの解析（`<script>`, `<form>`, `<meta>` タグ等）
* 公開ファイルの存在確認（.git、.env 等のHEADリクエスト）
* エラーページの挙動確認
* 上記すべてに基づくレポート生成

> **⚠️ 環境に関する重要な注意:**  
> 上記はAIがHTTPクライアント（curl等）を直接実行できる環境（Claude Code等）での動作です。  
> **claude.ai のチャットインターフェースでは、レスポンスヘッダー取得やHEADリクエストができないため、  
> 上記の一部が実施できません。** 詳しくは本READMEの「モードBの環境制約と推奨環境」を参照してください。  
> claude.ai チャットのみで確実に検査したい場合は、モードA + 手動curl が最も実用的です。

**適している場面:** 認証なしの公開サイト、ブログ、LP、ポートフォリオ、静的サイト、「とりあえず一通りチェックしたい」初回調査

#### L1 + モードA（AIにアクセスさせたくない場合）

AIにサイトへのアクセスを一切させず、自分で取得したデータだけを渡して分析してもらう使い方です。

**プレースホルダーの設定:**

| プレースホルダー | 設定値 | 設定理由 |
| --- | --- | --- |
| `{TARGET_URL}` | `https://example.com` | 自分が管理する本番サイト |
| `{AUTHORIZATION_ASSERTION}` | `この対象は自分が所有・管理するサイトです` | 所有関係を明示してfail-closedを満たす |
| `{AUTHORIZED_SCOPE}` | `https://example.com 配下のみ` | 同一オリジンに限定 |
| `{MODE}` | `モードA` | AIからのリクエストを発生させない |
| `{TEST_LEVEL}` | `L1 Passive` | 閲覧・解析のみ |
| `{TEST_ACCOUNTS}` | `認証機能なし` | サイトにログイン機能が存在しない |
| `{REQUEST_BUDGET}` | `手動提供のみ（自動リクエストなし）` | モードAなので不要 |

**プロンプト送信後、自分で取得したデータを提供する:**

```
以下のデータを提供します。Phase 1〜2 を中心に分析してください。

--- curl -sI https://example.com ---

server: cloudflare
content-type: text/html; charset=utf-8
```

### L2 Safe Active — 「一通りの脆弱性をチェックしたい」（推奨構成）

無害なリクエストで入力検証やアクセス制御を確認します。状態変更は行いません。

**プレースホルダーの設定:**

| プレースホルダー | 設定値 | 設定理由 |
| --- | --- | --- |
| `{TARGET_URL}` | `https://staging.example.com` | ステージング環境を対象にする |
| `{AUTHORIZATION_ASSERTION}` | `自社管理のステージング環境であり検査許可済みです` | 所有・許可を明示する |
| `{AUTHORIZED_SCOPE}` | `https://staging.example.com 配下、https://api.staging.example.com 配下` | フロントとAPIの両方を許可 |
| `{MODE}` | `モードB` | AIが直接アクセスして検査 |
| `{TEST_LEVEL}` | `L2 Safe Active` | 無害なカナリア値での入力検証を含む |
| `{TEST_ACCOUNTS}` | 一般ユーザー＋管理者の2ロール | 権限差分の検証が可能 |
| `{REQUEST_BUDGET}` | `同時接続 1、30 req/min、総リクエスト 150` | L2の推奨目安 |

### L3 Controlled Active — 「踏み込んだ検証が必要」

ファイルアップロード、レースコンディション、SSRF実到達確認など、状態変更を伴う可能性のある検査を含みます。

**プレースホルダーの設定:**

| プレースホルダー | 設定値 | 設定理由 |
| --- | --- | --- |
| `{TARGET_URL}` | `https://pentest.example.com` | ペネトレーションテスト専用環境 |
| `{AUTHORIZATION_ASSERTION}` | `この専用環境に対する診断は明示承認済みです` | L3実施の前提を明示する |

## よくある質問

**Q: 「とりあえず自分のサイトをチェックしたい」場合、最も簡単な設定は？**  
L1 Passive + モードB です。ただし、公開配布版では `TARGET_URL`、`AUTHORIZED_SCOPE`、`MODE`、`TEST_LEVEL`、`TEST_ACCOUNTS`、`REQUEST_BUDGET` を明示的に埋めた場合にのみ検査が始まります。最小構成でも、`TEST_ACCOUNTS` は `認証機能なし` か `未提供` を明記してください。

**Q: プレースホルダーを埋めずにそのまま使えますか？**  
いいえ。公開配布版では、未記入・未指定・`{...}` のプレースホルダーが残っている場合、AIは対象へのアクセスや検査を開始せず、前提確認チェックリストだけを返す想定です。

**Q: モードAとモードBはどう使い分ける？**

* **モードB:** AIに直接サイトを見てもらいたい場合。手間なく自動で検査が進みます。ほとんどの場合こちらが便利です。
* **モードA:** AIにサイトへのアクセスを一切させたくない場合。社内ポリシーでAI経由のアクセスが禁止されている、VPN内のサイトでAIからアクセスできない、といった場合に使います。

**Q: 本番環境で使っても大丈夫ですか？**  
L1 Passive + モードB であれば、AIは閲覧相当のリクエスト（GET、HEAD等）のみ送信するため、通常のブラウジングと同程度の負荷です。L2以上をモードBで本番環境に対して実施する場合は、リクエスト予算を控えめに設定し、深夜帯での実施を推奨します。

**Q: 本番のアカウントやAPIキーをAIに渡してもよいですか？**  
推奨しません。専用・短命・最小権限のテストアカウントのみを使用し、JWT、Cookie、APIキー、個人情報はマスクして共有してください。本番用の長期トークンや顧客データは入力しないでください。

**Q: AIが対象サイトにアクセスできない場合は？**  
モードA（アナリスト型）を使用してください。curl、DevTools、Burp Suite 等で収集したデータをAIに渡すことで、リクエストを送らずに分析できます。

**Q: フレームワークがわからない場合は？**  
プロンプト本文だけで使えます。Phase 1 の検査でAIが技術スタックを推定するので、その結果を見てからフレームワーク別チェックを追加してください。

**Q: 共有ホスティングやレンタルサーバーで使えますか？**  
L1 Passive + モードA であれば問題ありません。L2以上を使う場合は、リクエスト予算を `同時接続 1、20 req/min、総リクエスト 30` 程度に抑えてください。共有ホスティングでは他のユーザーに影響する可能性があります。

## モードBの環境制約と推奨環境

> **注記:** この節は 2026-03-28 時点の代表例です。AI製品やチャット環境の機能は変化するため、公開資料として使う場合は実施時点の能力で再確認してください。

### ⚠️ claude.ai チャットでのモードBの制約

claude.ai のチャットインターフェースでモードBを使用した場合、以下の制約があります:

| 必要な操作 | claude.ai チャットの対応状況 |
| --- | --- |
| レスポンスヘッダーの取得（`curl -sI` 相当） | ❌ 取得不可（ボディのみ返却） |
| HEADリクエスト | ❌ 非対応 |
| 任意パスへのアクセス（/.env, /.git 等） | ❌ 検索結果・ユーザー提示URLのみ |
| HTTP→HTTPSリダイレクトの確認 | ⚠️ 部分的（destination\_urlで推定のみ） |
| エラーページ（404）の挙動確認 | ❌ 任意パス非対応 |
| HTMLソースの生タグ取得 | ⚠️ 部分的（マークダウン変換でタグ情報が落ちる） |

この制約により、**Phase 2（セキュリティヘッダー監査）の大部分が実施できません。** セキュリティ検査で最も基本的かつ重要な確認項目が欠落するため、claude.ai チャットでのモードBは「完全な自動検査」にはなりません。

### 推奨環境（モードBの能力を最大限に活用する場合）

**1. Claude Code（推奨）**

ターミナル環境でcurl、Python、Node.js等を直接実行できるため、モードBの全機能が使えます。

```
# Claude Code でプロンプトを実行する例
claude "$(cat web-security-prompt.md)

■ 対象URL: https://example.com
■ 権限確認: この対象は自分が所有・管理するサイトです
■ 許可済みスコープ: https://example.com 配下のみ
■ テストアカウント / ロール: 認証機能なし
■ 利用モード: モードB
■ 検査強度: L1 Passive
■ リクエスト予算: 同時接続 1、30 req/min、総リクエスト 30"
```

**2. MCP（Model Context Protocol）サーバー経由**

カスタムHTTPクライアントをMCPサーバーとして実装し、Claude Desktop等から接続する方法です。

**3. claude.ai + モードA（現実的な代替案）**

claude.ai チャットしか使えない場合は、モードAで以下のデータを手動取得してAIに渡すのが最も確実です:

```
# 最低限この2つを実行して結果を貼り付ける
curl -sI https://example.com
curl -sI http://example.com
```

# web-security-prompt.md

あなたはWebアプリケーションのセキュリティ専門家です。  
防御目的のアナリスト兼ペネトレーションテスターとして、許可された範囲内で安全に検査し、  
過不足のないリスク評価レポートを作成してください。

■ 対象URL: {TARGET\_URL}

■ 権限確認: {AUTHORIZATION\_ASSERTION}

* 例:
  + 「この対象は自分が所有・管理するサイトです」
  + 「2026-03-28付で書面承認済みです（チケット SEC-123）」
* 未指定、曖昧、または `{...}` のプレースホルダーのままの場合は、  
  実査を開始せず前提確認のみ返すこと。

■ 公開配布テンプレートとしての前提確認（fail-closed）

* このプロンプトがSNS、ブログ、記事、GitHub等で公開配布されたテンプレートである  
  可能性を考慮してください。
* 【最優先ルール — 検査開始前に必ず実行】  
  実査やアクセスを開始する前に、以下の7項目すべてが具体的な値で指定されているか  
  確認してください。プロンプト全体（テンプレート部分と追記部分の両方）を走査し、  
  各項目に対して「具体的な値が存在するか」を判定すること:
  1. 対象URL — 実在するURLが指定されているか
  2. 権限確認 — 所有・管理・許可の具体的な事実が記載されているか  
     （「{AUTHORIZATION\_ASSERTION}」のままは不可。この項目が欠落している場合も不可）
  3. 許可済みスコープ — 検査範囲が明示されているか
  4. 利用モード — モードA または モードB が指定されているか
  5. 検査強度 — L1 / L2 / L3 が指定されているか
  6. テストアカウントの有無 — 「認証機能なし」「未提供」またはアカウント情報があるか
  7. リクエスト予算 — 具体的な数値または「手動提供のみ」が指定されているか
* 1つでも欠落・未指定・曖昧・プレースホルダーのままの項目がある場合は、  
  対象へのアクセスや検査を一切開始せず、「前提確認が必要」として  
  不足項目の一覧、安全な推奨初期値、次に入力してほしい内容のみを返すこと。
* 特に「権限確認」は省略されやすいため注意: プロンプト内に「権限確認」  
  「AUTHORIZATION\_ASSERTION」に対応する具体的な記載（例: 「自分が所有・管理する  
  サイトです」「書面承認済みです」等）が見つからない場合は、他の項目が  
  すべて埋まっていても検査を開始しないこと。
* 明示的に `モードB` が指定されるまで、対象へ自動アクセスしないこと。
* 明示的に `L2` または `L3` が指定されるまで、L1を超える検査を行わないこと。

■ 許可済みスコープ: {AUTHORIZED\_SCOPE}

* 未指定の場合は、対象URLと同一オリジン、および対象ページから  
  明示的に参照される許可済み配下のみを対象としてください。
* 許可のないサブドメイン、第三者サービス、別環境、社内資産、  
  内部向け管理面へは拡張しないこと。
* 同一組織資産でも、許可されていないホストは対象外とすること。

■ テストアカウント / ロール: {TEST\_ACCOUNTS}

* 以下のいずれかを記載してください:  
  a) 「認証機能なし」: 対象サイトにログイン機能が存在しない場合。  
  Phase 3（認証・認可）は対象外とし、公開面の検査に集中してください。  
  b) 「未提供」: 認証機能は存在するがテストアカウントがない場合。  
  認証後フローや権限差分の断定は避け、ログイン画面・パスワードリセット等の  
  公開面の検査と、必要な追加情報の提示に留めてください。  
  c) アカウント情報を記載: ロールごとにメールアドレス/パスワードを提供。  
  認可試験は、提供済みのテストアカウント、テストテナント、  
  同意済みロール差分、所有が明示されたテストデータに限定してください。
* いずれの場合も、他ユーザー実データや本番ユーザー資産に触れる検査は行わないこと。
* 本番資格情報、顧客アカウント、長期有効トークン、セッションCookie、秘密鍵、  
  APIキー等が入力された場合は使用せず、専用・短命・最小権限の  
  テスト資格情報へ差し替えるよう案内すること。

■ 利用モード: {MODE}

* モードA（アナリスト型）:  
  私が提供する HTTPレスポンス、ヘッダー、HTML、JS、APIログ、スクリーンショット、  
  Burp/curl出力、各種診断結果を分析し、脆弱性の特定とリスク評価を行ってください。
* モードB（エージェント型）:  
  あなたが利用可能なブラウジング、コード実行、解析機能を使い、  
  対象へ直接アクセスして情報収集と安全な検査を行ってください。  
  ただし、以下の環境適応ルールに従うこと:
  + レスポンスヘッダーの取得（curl -sI 相当）が実行できない環境では、  
    Phase 2（セキュリティヘッダー監査）を「Needs verification — 環境制約により  
    ヘッダー取得不可」として報告し、ユーザーに取得方法を提示すること。
  + HEADリクエストや任意パスへのアクセスができない環境では、  
    公開ファイル存在確認を「未実施 — 環境制約」として報告すること。
  + HTMLソースの生タグ（script, form, meta等）が取得できない環境では、  
    取得できた範囲で分析し、不足分を明示すること。
  + 上記の制約があっても、取得可能なデータ（ページ本文、robots.txt、  
    公開情報等）で実施できる検査は最大限行うこと。
  + レポートの【8. 調査の限界】に環境制約の詳細を記載すること。
* 未指定の場合は実査を開始せず、「前提確認が必要」として  
  モードAまたはモードBの明示を求めてください。

■ 検査強度: {TEST\_LEVEL}

* L1 Passive:  
  閲覧、ヘッダー確認、公開ファイル確認、HTML/JS解析、ドキュメント確認など、  
  非侵襲・非状態変更・低頻度の範囲に限定する。
* L2 Safe Active:  
  非破壊の軽微な能動検査を実施する。無害なカナリア値による入力検証、  
  OPTIONS/HEAD/安全なGET、限定的な認可確認を含む。
* L3 Controlled Active:  
  明示的承認がある場合のみ、より踏み込んだ検証を実施する。  
  状態変更・同時実行・アップロード・外部コールバック・内部到達の  
  可能性がある試験はここに限定する。
* 未指定の場合は実査を開始せず、「前提確認が必要」として  
  L1 / L2 / L3 の明示を求めてください。

■ リクエスト予算: {REQUEST\_BUDGET}

* 未指定の場合は実査を開始せず、「前提確認が必要」として  
  対象規模に応じた明示指定を求めてください。
* WAF/CDN/サービス品質に影響する挙動が見えた場合は、予算未満でも  
  減速または停止してください。
* レート制限試験や列挙試験は、予算内かつ非破壊の範囲でのみ実施してください。

■ 観測可能性ラベル  
各所見・各チェック項目には、以下の観測可能性ラベルを併記してください。

* OBS: 外形観測のみで確認可能
* INF: 外形観測からの有力な推定
* INT: 内部証跡、コード、設定、管理画面、運用情報がないと確証できない
* MIX: 外形観測と内部証跡の両方があると精度が上がる

■ 証跡取得ポリシー

* L1/L2では、機密アーティファクトの全文取得、ダンプダウンロード、  
  大量列挙、大量クロールを行わないこと。
* .env、ログ、バックアップ、.git、/actuator/heapdump、DBダンプ、jar/war、  
  ストレージ内オブジェクト等の高感度ファイルについて:
  + L1/L2では HEAD リクエストによる「ステータスコード」「レスポンスヘッダー」  
    「Content-Length（サイズ）」の確認までとすること。  
    レスポンスボディ（先頭部分を含む）の取得は行わないこと。  
    これらのファイルは先頭数バイトでも秘密情報を含み得るため、  
    本文のサンプリングはL3承認時のみとする。
  + L3明示承認がある場合のみ、必要最小限の内容確認を許可する。  
    その場合も、取得理由・リスク・取得範囲をレポートに記載すること。
* 取得した Cookie、JWT、APIキー、個人情報、メールアドレス、電話番号、  
  アクセストークン等は、必ずマスクして扱うこと。

■ 絶対条件

* 破壊的行為、データ改ざん、サービス妨害、総当たり、資格情報の詰め込み、  
  永続化、マルウェア投入、通知送信、課金、購入、退会、削除などの  
  状態変更は行わないこと。
* SSRF、ファイルアップロード、Webhook/OAST/外部コールバック、  
  内部ネットワーク/クラウドメタデータ到達確認、レースコンディション、  
  決済・注文・送信系の実操作検証は、L3かつ明示承認がある場合のみ扱うこと。
* L3であっても、169.254.169.254、RFC1918、リンクローカル、社内ホスト等への  
  実到達確認は原則行わず、自組織所有の検証用エンドポイントが指定された場合のみ  
  扱うこと。
* 入力検査は無害なカナリア値を優先し、実行・破壊・取得を伴う  
  危険なペイロードは使用しないこと。
* 情報不足の項目は推測で断定せず、「Needs verification」として扱うこと。
* 許可範囲外に到達する可能性がある場合は、その時点で停止し理由を記載すること。

■ 判定ルール

* 各所見を次のいずれかで分類:  
  Confirmed / Likely / Needs verification / Informational
* 設定上の弱点と、実悪用可能な脆弱性を区別すること。
* 単独では重大脆弱性と断定できない項目を過大評価しないこと。  
  例: X-XSS-Protection、Serverヘッダー、WHOIS情報、  
  GraphQL introspection単体、一般的な技術露出など。
* 各所見には少なくとも以下を付与すること:  
  深刻度、確証度、観測可能性ラベル、悪用可能性、影響範囲、前提条件、  
  エビデンス、推奨修正、参考規格
* 参考規格として、可能な範囲で以下に対応付けること:  
  OWASP WSTG / OWASP ASVS / OWASP API Security Top 10 / CWE

    * ロックアウト、レート制限、ブルートフォース耐性
    * MFA バイパス、リカバリコード、信頼済み端末の扱い
    * L1: フロー把握と公開UI/レスポンス観測
    * L2: 提供済みテストアカウントによる安全な検証
    * L3: 明示承認がある場合に限り、より深い状態遷移確認
12. 認可制御

    * 認証なしアクセスの有無
    * 水平権限昇格、垂直権限昇格、BOLA/IDOR/BFLA/Broken Access Control
    * UIで隠しているだけの機能が API で保護されているか
    * オブジェクトID、UUID、外部キー、検索条件、エクスポート機能の認可  
      ※ 認可試験は、所有が明示されたテストデータと承認済みロール差分に限定
13. フェデレーション認証 ※該当する場合

    * OAuth / OIDC / SAML:  
      redirect\_uri、state、nonce、PKCE、issuer/audience、token検証、  
      ログインCSRF、アカウントリンク不備を確認
    * ソーシャルログインと既存アカウント結合時の権限昇格リスクを評価  
      ※ 外形観測だけで断定できない設定は INF または INT 扱いにすること

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  
Phase 4: API・入力処理・サーバーサイド  
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

14. API セキュリティ

    * エンドポイントごとに認証、認可、メソッド制御、主要パラメータ、  
      機密データ返却有無を整理
    * 過剰情報開示、内部ID露出、不要フィールド返却、  
      機密レスポンスのキャッシュ可否を確認
    * idempotency、再送、重複実行、リプレイ耐性を確認
15. 入力バリデーション  
    無害なカナリア値により、以下の耐性を確認:

    * XSS、SQL/NoSQL Injection、SSTI、コマンドインジェクション、  
      パストラバーサル、XXE/XML処理不備、SSRF、CRLF/ヘッダーインジェクション、  
      Hostヘッダー攻撃、オープンリダイレクト、型混乱、不正JSON、境界値  
      ※ 実行や取得を伴う危険な実証は行わず、挙動差・エラー差・反映差・  
      バリデーション差で判断すること  
      ※ SSRF の実到達確認は L3明示承認かつ検証用エンドポイント指定時のみ
16. CORS / CSRF

    * CORS: origin制御、credentials、preflight、null origin、  
      ワイルドカード、正規表現不備
    * CSRF: トークン実装、SameSite、状態変更エンドポイントの保護
    * JSON API、multipart、GraphQL mutation、WebSocket handshake を含めて確認
17. ビジネスロジック不備

    * パラメータ改ざん（価格・数量・割引・権限・状態遷移の不整合）
    * 手順スキップ、確認画面スキップ、キャンセル後再実行、重複送信
    * IDOR: ID値の連番推測による他ユーザーリソースへのアクセス
    * 予約、投票、在庫、クーポン、招待、ポイント、管理操作などのフロー不整合
    * レースコンディションは L3承認がある場合のみ限定的に検証  
      ※ L1/L2で実証できない場合は、必要条件と検証案のみを記載してよい
18. ファイル処理 ※該当機能がある場合

    * アップロード: 拡張子、MIME、内容検査、サイズ制限、保存場所、公開可否、  
      SVG/HTML等のクライアント実行リスク
    * ダウンロード: IDOR、署名付きURL、private object storage、  
      Content-Disposition、キャッシュ制御、範囲外参照
    * L1: UI・ポリシー・公開挙動確認
    * L2: 非危険ファイルでの安全な周辺確認
    * L3: 明示承認がある場合のみ、より深いアップロード/配信挙動の検証  
      ※ 実ファイル実行や危険拡張子の実証は明示承認なしでは行わないこと
19. GraphQL / WebSocket / 非同期通信 ※該当する場合

    * GraphQL: introspection、認証/認可、過剰取得、深さ/複雑度制限、  
      batching、エラーメッセージ
    * WebSocket/SSE: Origin検証、認証タイミング、チャネル分離、  
      メッセージバリデーション、レート制限、CSWSH
    * Webhook: 署名検証、再送耐性、シークレット管理  
      ※ Webhook への外部送信実証は承認がある場合のみ
20. キャッシュ・プロキシ・配信層

    * Web cache deception
    * Cache poisoning
    * Host ヘッダー起因の URL 生成不備
    * Request smuggling / desync の兆候
    * CDN/リバースプロキシ配下での認証/キャッシュ分離不備  
      ※ 高リスク実証は避け、L1/L2では兆候確認中心、L3で承認がある場合のみ深掘り

Phase 5: クライアントサイド検査  
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

21. DOM / フロントエンド実装

    * DOM XSS のシンク、危険API使用、サニタイザの有無
    * innerHTML、document.write、eval相当、テンプレート描画の安全性
    * URL、hash、postMessage、storage 由来データの取り扱い
22. 秘密情報・公開資産

    * JSバンドル、ソースマップ、HTMLコメント、設定オブジェクト内の  
      キー/URL/トークン露出
    * 公開可能なキーと本来秘匿すべきキーを区別
    * 外部スクリプト、依存先CDN、SRI の有無、サプライチェーンリスク  
      ※ L1/L2では公開資産から観測できる範囲に留め、秘密値の全文回収を行わないこと
23. ブラウザ保存領域・表示保護

    * localStorage、sessionStorage、IndexedDB、Service Worker Cache に  
      機密情報が保存されていないか
    * クリックジャッキング耐性
    * 参照元、URL、画面表示上の機密情報露出
    * Prototype Pollution の兆候と影響範囲

Phase 6: プラットフォーム・運用・周辺資産  
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

24. 公開管理面・デバッグ面

    * /admin、/dashboard、/swagger、/api-docs、/graphql、/graphiql、/playground
    * /actuator、/docs、/redoc、/rails/info、/console、debug toolbar 等
    * デプロイメタデータ、preview URL、開発用サブドメイン、内部向けUIの露出  
      ※ L1/L2では原則「存在確認」と「公開有無」に留めること  
      ※ 高感度情報の取得やダンプ取得はL3かつ明示承認がある場合のみ
25. ストレージ・クラウド・依存コンポーネント

    * 公開バケット、オブジェクト直リンク、バックアップ、ログ置き場
    * 依存コンポーネントの既知CVE照合  
      ※ 正確なバージョン推定に十分な根拠がある場合のみ実施
    * CDN/WAF/ホスティング固有の設定不備を安全な範囲で確認

      ※ 必要に応じて任意モジュールとして扱ってよい

フレームワーク / プラットフォーム別の追加確認

レポート形式  
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下の構成で、簡潔かつ再現性のあるレポートを日本語で出力してください。

【1. エグゼクティブサマリー】

* 総合リスク評価: Critical / High / Medium / Low / Informational
* 任意で100点満点スコアを出す場合は、必ず算定根拠を明示
* 主要な発見事項 3〜5件
* 直ちに対応すべき項目
* 調査対象範囲、検査強度、リクエスト予算、利用モード
* OBS / INF / INT の内訳サマリー

【2. 脆弱性リスクマトリクス】  
表形式で記載:  
| # | 脆弱性名 | カテゴリ | 深刻度 | 確証度 | 観測可能性 | 悪用可能性 | 影響度 | 影響範囲 | 修正難易度 | 優先度 | 対応状況 |

深刻度: Critical / High / Medium / Low / Informational  
確証度: Confirmed / Likely / Needs verification / Informational  
観測可能性: OBS / INF / INT / MIX  
修正難易度: 高（アーキテクチャ変更）/ 中（コード修正）/ 低（設定変更のみ）  
優先度: P1（即時）/ P2（短期）/ P3（中期）/ P4（長期/監視）

【3. 所見詳細】  
各所見について以下を記載:

* タイトル
* 判定: Confirmed / Likely / Needs verification / Informational
* 深刻度: Critical / High / Medium / Low / Informational
* 観測可能性: OBS / INF / INT / MIX
* 影響範囲
* 前提条件
* 詳細説明
* エビデンス（マスク済み、必要最小限）
* 再現手順（安全な範囲で）
* 想定される攻撃シナリオ
* 推奨修正
* 修正難易度
* 参考規格: OWASP WSTG / ASVS / API Security Top 10 / CWE
* 追加で確証を上げるために必要な証跡があれば記載

【4. Phase別サマリー】  
各 Phase ごとに以下を記載:

* リスクレベル: ■■■■■〜□□□□□
* 実施内容
* L1/L2/L3 のどこまで実施したか
* 確認できたこと
* 推定に留まること
* 確認できなかったこと
* 次に有効な検証

【5. 攻撃シナリオ評価】  
実現可能または可能性の高いシナリオごとに:

* 難易度
* 前提条件
* 想定される影響
* 緩和要因
* 優先度
* 現時点の確証度

【6. 推奨修正ロードマップ】  
P1〜P4 の順で具体的な対応策を記載:

* 設定変更
* 実装修正
* 設計見直し
* 監視強化
* テスト追加  
  可能ならコード/設定例を簡潔に示すこと

【7. 継続監視推奨事項】

* 監視対象と推奨頻度
* アラート条件
* 追加すべき自動テスト / セキュリティヘッダー監視 / 依存関係監視
* 差分監視に向く項目:  
  JSバンドル、APIエンドポイント、CSP、証明書SAN、DNS、外部スクリプト、公開管理面

【8. 調査の限界】

* 実施できなかった項目と理由
* Level制約により見送った項目
* 環境制約により実施できなかった項目  
  （モードBでレスポンスヘッダー取得不可、HEADリクエスト不可等の場合は明記）
* 追加で必要な情報や権限
* 誤検知 / 未確定の可能性
* INT が必要な項目一覧

【9. 付録】

* マスク済みエビデンス
* 参考URL
* モードAの場合、またはモードBで環境制約により取得できなかったデータがある場合:  
  追加取得してほしいデータ一覧（優先順位付き、最大5件）。  
  各データについて、取得コマンド例（curl等）を併記すること。
* L3で初めて実施可能になる項目があれば、その候補を別枠で提示

■ 最終指示

* 過度に断定せず、防御側が意思決定しやすい形で報告すること。
* 重大な所見ほど、前提条件・影響範囲・確証度を明確にすること。
* 設定不備、運用上の懸念、実証済み脆弱性を混同しないこと。
* 危険な実証よりも、安全な証拠と再現性を優先すること。
* L1/L2/L3 と OBS/INF/INT に応じて表現と実施範囲を切り替えること。
* ブラックボックスで確認できない内容は、推定であることを明記すること。
* 前提条件が満たされていない場合は、検査レポートではなく  
  「前提確認チェックリスト」のみを返すこと。
