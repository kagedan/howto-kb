### 4月23日、深夜2時、エラーが消えない

4月23日の夜、僕は note 記事を自動投稿する Playwright スクリプトを書いていました。**目標は1つ**。markdown を読み込んで、[note.com](http://note.com) の本文エリアに正しく貼り付けること。

ところが、`.ProseMirror` セレクターに `evaluate()` で innerHTML を入れても、**改行が全部消える**！ 何度試しても、本文がペシャンと潰れて1行になる。

正直、**もう辞めようかと思いました**！「IT会社員で、コード書くなんて夢を見てたんだ」と、本気で挫折しかけた瞬間です。

### 翌朝、Codexに投げてみた

翌朝、ふとした思いつきで、Codex（GPT-5系列）に同じ問題を投げてみました。

> 「Playwrightで`.ProseMirror`にmarkdown相当のHTMLを貼り付けたいけど、改行が消える。なぜ？」

Codex の答え：

> **「`.ProseMirror`は ProseMirror エディタの実体なので、innerHTMLを直接書き換えても内部状態が更新されません。クリップボード経由で Ctrl+V で貼り付けるか、ProseMirror の API を直接叩いてください」**

**3分で解決**！

### でも、Codex単体じゃ完結しなかった

ここで重要なのは、**「じゃあCodexの方が優れてる」という結論じゃない**こと。

Codex の答えを元に「クリップボード経由でCtrl+V」する実装を書こうとしたら、今度は**Windows環境特有のクリップボード文字化け問題**にぶつかりました。Codex はそこからの解決策を、なぜか英語Mac前提でしか出してくれない。

ここで Claude Code に戻って、

> 「Codex から『クリップボード経由でCtrl+V』のアドバイスをもらったけど、Windows でクリップボードに日本語を入れると文字化けする。どう実装する？」

Claude Code の答え：

> **「`pyperclip` ではなく `navigator.clipboard.writeText` を使ってブラウザ側で書き込み、`page.keyboard.press('Control+v')` で貼り付けてください。Windows のCP932問題を回避できます」**

**完璧な答え**！

両方を順番に使うことで、**1つのAIだけでは絶対に辿り着けなかった解決策**に到達しました。

## 第1章｜Claude Code と Codex の強み・弱みを正直に比較

ここで、3週間使い倒した僕の**忖度なしの比較**を出します。

### Claude Code（Anthropic）の特徴

**強み**

* **長文理解が圧倒的**：3万行のリポジトリ全体を読ませても、文脈を保ったままリファクタしてくれる
* **日本語の自然さがダントツ**：コミットメッセージ、コメント、ドキュメントを日本語で書かせると、そのまま使えるレベル
* **対話的な問題解決が得意**：「ここをこうしたい、けどこの制約があって…」みたいな曖昧な要件を、対話で詰めながら実装に落とせる
* **エージェント的な動きが秀逸**：ファイル探索→読み込み→修正→テスト実行を、自分で計画して実行
* **「考えてから書く」傾向が強い**：いきなりコードを書かず、設計をまず提示してくる

**弱み**

* **コード生成のスピードが遅め**：Codex に比べて、同じタスクで体感1.5倍くらい時間がかかる
* **最新ライブラリの知識がやや古い**：2025年後半に出た新APIを知らないことがある
* **Pythonエコシステムの細部で勘違いをすることがある**：特定ライブラリのバージョン特有の挙動など

### Codex（OpenAI / GPT-5系）の特徴

**強み**

* **コード生成の速度が爆速**：「Pythonでこれ書いて」みたいな短いタスクを、3秒で出してくる
* **最新ライブラリの知識が豊富**：2026年1月以降のAPIにも追従している
* **OpenAIエコシステムとの統合が強い**：ChatGPT のWeb検索、コード実行サンドボックス、画像生成と連携
* **「とりあえず動くコード」を出す力が強い**：細かい設計議論より、まず手を動かしたい場面で速い
* **エラーメッセージの解釈が早い**：スタックトレースを投げると、瞬時に原因と修正案を返す

**弱み**

* **長文の文脈保持が苦手**：5,000行を超えるコードベースを丸ごと読ませると、後半を忘れる
* **対話の途中で前提を忘れる**：「さっき決めた方針」を覚えていないことがある
* **日本語の自然さは Claude より一段落ちる**：コメントを日本語で書かせると、やや翻訳調になる
* **エージェント的な動きは弱め**：自分でファイル探索しながら計画を立てるより、人間が指示を出してから動く

## 第2章｜なぜ"連携"が最強なのか（思想編）

ここで、僕がこの3週間で辿り着いた**連携の思想**を、3つの原則にまとめます。

### 原則①：AIは"道具"じゃなく"チームメンバー"として扱う

人間の優秀なエンジニアチームを想像してみてください。**1人で全部やる人**より、**設計が得意な人と実装が爆速な人がペアで動く**チームの方が、圧倒的に速い。

Claude Code と Codex も同じ。「片方を使う」じゃなく、「2人のAIエンジニアを雇って、得意分野で分担させる」発想にシフトする。これが連携の出発点です。

### 原則②：プロンプトは"バトン"だと考える

連携の核は、**片方の出力をもう片方の入力にする**こと。

例えば：

1. Claude Code に「設計を書かせる」
2. その設計を**そのままコピペ**して Codex に投げる
3. Codex が爆速で実装する
4. その実装を**そのままコピペ**して Claude Code に渡す
5. Claude Code がレビュー・リファクタする

**プロンプトをバトンとして渡す**。これが連携の基本動作です。

### 原則③：「どっちが正しい」じゃなく「どこが違うか」を見る

同じタスクを2つに投げて、答えが違ったとき。**「どっちが正解か」を決めるんじゃない**。**「なぜ違うのか」を観察する**。

たいていの場合、両方とも部分的に正解で、部分的に間違っている。**その差分から、自分が見落としていた視点が見えてくる**。これが、人間1人＋AI1台では絶対に得られない学びの源泉です。
