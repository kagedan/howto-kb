Claude Codeでエージェントを使っている人ほど、Workflowsは見た方がいいです。

正直、最初は「また新しい自動化機能かな」くらいに見ていました。

でも、公式ブログとdocsを読み直すと、かなり意味が違いました。

これは、AIに賢く答えさせる機能ではありません。

AIに、毎回同じ段取りで仕事を回させる仕組みです。

Claude CodeチームのThariq氏は、Workflowsをskills・subagentsに次ぐ大きなアップグレードとして紹介していました。

ユーザー提供の文脈では、元ポストは約100万ビュー、いいね4,696。

伸びた理由はシンプルです。

Claude Codeを使っている人の多くが、同じ限界を感じているから。

- 毎回プロンプトで手順を説明し直す
- 複数ステップの作業を任せると、途中で段取りが変わる
- 並列でエージェントを走らせたいが、制御が難しい
- 探索は得意なのに、決まった手順の反復が苦手
- 非エンジニア業務に落とし込むと、まだ使いどころがぼやける
先に結論を言うと、WorkflowsはClaude Codeを「相談相手」から「段取りを持った仕事係」に変える機能です。

この記事を読むと、次の5つが分かります。

① Workflowsが何をしているのか
② skills/subagentsと何が違うのか
③ なぜ非技術タスクにも効くのか
④ どんな作業をワークフロー化すべきか
⑤ 逆に、ワークフロー化しない方がいい作業

これが分かると、毎回5分かけて説明していた手順を、1回組んで何度も回す発想に変わります。

週5回やる作業なら、5分差でも月100分。

映画1本分くらいの時間が戻ってきます。

## Workflowsは、AIに「段取り」を持たせる機能

Claude Codeは、すでにかなり強いです。

調査もできます。

コードも読めます。

ファイルも直せます。

サブエージェントも動かせます。

ただ、仕事で使っていると、だんだん別の問題が出てきます。

「この順番でやって」と毎回説明している。

「調査して、分類して、下書きして、確認して、最後に保存して」と言ったのに、途中で順番が変わる。

「A案とB案を別々に検討して」と言ったのに、1つの文脈に混ざる。

これ、AIが弱いというより、段取りをチャットの中だけで持たせているのが原因です。

公式docsでは、Dynamic Workflowsを「Claudeが書くJavaScript scriptがsubagentsを大規模にオーケストレーションする仕組み」と説明しています。

ポイントは、計画をClaudeの会話ターンではなく、スクリプト側に持たせること。

つまり、

調査する
↓
複数の観点に分ける
↓
それぞれ別エージェントに渡す
↓
結果を検証する
↓
最後に1つの答えへまとめる

この段取り自体を、実行できる形にする。

これがWorkflowsの大きな違いです。

実践用素材として、まず棚卸し用の型を置いておきます。

次の作業をWorkflows化できるか判断してください。

【作業名】
例: セミナー後の参加者アンケート要約

【毎回やっている手順】
1. 【手順1】
2. 【手順2】
3. 【手順3】
4. 【手順4】

【途中で分岐する条件】
- 【条件A】なら【対応A】
- 【条件B】なら【対応B】

【並列で見たい観点】
- 【観点1】
- 【観点2】
- 【観点3】

【最後に欲しい成果物】
- 【Markdown / CSV / スプレッドシート / PRコメント / レポート】

この作業は、会話で1回だけ頼むべきか、Workflowsとして段取り化すべきかを判定してください。

すべての仕事をWorkflowsにする必要はありません。

ただ、毎回同じ順番を説明している自覚があるなら、試す価値はあります。

## skillsは知識、subagentsは別働隊、Workflowsは司令塔

ここが一番混乱しやすいところです。

Claude Codeには、すでにskillsやsubagentsがあります。

だからWorkflowsを見たときに、

「これ、何が新しいの？」

と思う人も多いはずです。

ざっくり分けると、こうです。

skills    = 専門知識や手順書
subagents = 作業を任せる別働隊
Workflows = 別働隊をどの順番・並列・反復で動かすかを決める段取り

skillsは、AIに渡す説明書です。
