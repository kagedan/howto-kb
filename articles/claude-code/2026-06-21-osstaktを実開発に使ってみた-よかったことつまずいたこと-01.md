## はじめに

AIエージェントのオーケストレーションツール（OSS）「**TAKT**」を、アプリ開発の実開発に使ってみた体験を記載します。本記事は連載の第2部・実体験編です。

第1部（解説編）では、TAKTが「AIエージェントを外側から制御する」OSSであり、ワークフロー（workflow）全体をYAMLで宣言し、遷移ルール・ループ監視・実行記録が最初から揃っていることを解説しました。連載全体の結論は **「オーケストレーションは『人間の都度確認』を『ワークフロー定義』に置き換える」** というものです。  
第2部では、その置き換えを実際の開発で試して見えてきた、**よかったこと**と**つまずいたこと**を正直に共有します。  
<https://zenn.dev/alpha_omega/articles/56ca3136bc5cd8>

開発で利用したアプリの構成としては、フロントエンドとバックエンドが1リポジトリに同居するモノレポ構成で、ここに「既存デザイン → UI/UX実装 → API修正」までを一気通貫で依頼しました。  
なお、このモノレポ構成が、後半の課題①（FE/BE横断タスクの失敗）の伏線になります。

## 実際に使ってみた

特筆したいのは実行記録のしくみです。TAKTはランごとに`.takt/runs/<タイムスタンプ-スラッグ>/`というディレクトリを作り、何を指示し・何が返り・何が書き出されたかをすべて残します。手元のプロジェクトに実際に残っているランの構造はこうです。

```
.takt/runs/<タイムスタンプ-スラッグ>/
├── meta.json                 # ランのメタ情報（ワークフロー名・状態・現在ステップ）
├── logs/*.jsonl              # 実行ログ（各ステップに注入された指示の全文を含む）
├── context/
│   ├── knowledge/            # ステップに渡された知識ファイル
│   ├── policy/               # ステップに渡されたポリシー
│   └── previous_responses/   # 前ステップの応答
└── reports/                  # ステップが書き出したレポート
```

手元のプロジェクトには15ラン分の記録が蓄積しており、「あのとき何を指示されて何を返したか」を後から完全に追跡できます。本記事の引用も、この実行記録から復元したものです。

## TAKTで実際に作ったもの・動かし方

前述の実行記録の話とは別に、TAKTで「何を・どう作ったか」を簡潔に共有します。UI実装での使い勝手と、筆者の操作手順をまとめます。

### 何を作ったか

題材は、既存PWAに **PC版の画面（表示層）がまだ無かった**ので、それをデザイン基点で新規実装するタスクです。デザイン自体は存在していたので、それを唯一の正としてPC専用のレイアウト・各画面コンポーネント・PC専用ルートを新規に作り、ロジック・API・状態管理 といった既存資産は再利用する、という切り分けで進めました。バックエンド側は「Figmaを設計図に新規実装」ではなく、既存実装への接続・再利用と、別途の不具合修正が中心です。

デザインの取り込みには **Figma MCP** を使いました。TAKTが駆動する実行エージェント（第1部の`provider`）にClaudeを使っているので、Claude側に設定済みのMCP（Figma）をワークフロー内からそのまま呼び出せたのは便利でした（`FIGMA_API_KEY`の設定が前提）。

ただし常時うまくいったわけではありません。  
トークン失効・レート制限・権限未付与でFigma MCPが取得不能になることもあり、その際は書き出し済みのデザイン画像（PNG）があれば、それを正としてAIに読ませ、実装との構図照合まで回せました。一方で画像が手元に無い場面では「推測で埋めず、止めて次のランへ繰り越す」判断もしており、画像で必ず代替できたわけではない、というのが正直なところです。

### どう使ったか（操作フロー）

**筆者の操作手順**です。  
第1部の「ほぼ2コマンド」に対応づけると、`takt`（対話的にタスク定義 → キューへ）の内側が手順1〜4（要件議論 → `/go`で指示書生成 → `/play`でキュー投入）に、`takt run`（キューの自律実行）が手順5（実装ワークフローからの`/play`実行）にあたります。

1. `takt`を起動し、目的のワークフローを呼び出す
2. アシスタント（Planner）と要件を議論し、不明点を潰す
3. `/go`で、それまでのやり取りを基に指示書（`order.md`）を生成する
4. 「タスクにつむ」でタスクをキューに積む
5. 改めて`takt`から実装用ワークフローを呼び出し、`/play`で積んだタスクを実行させる
6. あとは完了まで人手を入れずに待つ（放置）

要件議論で指示書を固め、あとはキューに積んで実行を任せる、という流れです。投入・実行の実体はキュー定義（`tasks.yaml`）と各タスクの`order.md`、runs配下の実行記録として裏取りできます（指示書を固める手間そのものは、後述の課題3で触れます）。

## よかったこと

### 丸投げして放置できる、品質も上がる

一気通貫のタスクをまるっと投げて、完了まで放置できます。  
Claude単体利用で都度発生していた人間確認の手間がなくなりました。

コード品質もかなり上がったと感じています。実感だけでなく裏付けもあって、プロジェクト用ワークフローの実行記録にはアーキテクチャ / セキュリティ / QA / テスト / pure / コーディング / AIアンチパターンの**7観点が並列でレビューした記録**が毎回残っています。人間が依頼しなくても、この多観点レビュー⇄修正がワークフロー定義によって毎回強制されるわけです。

### ワークフロー定義で細かく制御できる

目的・ゴールをワークフローで定義することで、細かな制御ができます。実例を2つ紹介します。

**実例1: 禁止ルールによる暴走の抑止。**  
以前、ghコマンドによるIssue作成→PR作成の自動化を試したところ、暴走してissueを勝手に起票をしまうことがありました。  
そこでプロジェクト用ワークフローでは「`git add`（ステージング）までは実行し、それ以降は一切実行しない」という設計にしました。ステップ指示の原文がこれです。

```
## 禁止事項（厳守）

- `git commit` の実行 → 禁止
- `git push` の実行 → 禁止
- PR の作成（`gh pr create` / GitHub API）→ 禁止
```

この禁止はワークフロー定義・ステップ指示・persona定義の**3ファイルに多重で明文化**されています。

**実例2: Why / What / Howログの自動出力。**  
実装が終わると、最終ステップが`docs/<日付>/`に「改修サマリ」と「コミット粒度別の内訳」の2本のMarkdownを定型フォーマットで自動出力するようにしました。実際に自動生成されたドキュメントのWhy節がこちらです。

```
## Why（なぜ）
- 本番環境では id/password を使わず、特定の Service Account による
  IAM 認証・Private IP 環境で Cloud SQL に接続したい（セキュリティ・運用要件）。
- Prisma（スキーマ・クエリ・モデル）は廃止せず流用したまま、
  接続経路だけを切り替えたい。
```

コミット粒度別の内訳も、1論理コミット単位ごとにWhy / What / Howが書かれます。

```
## コミット 2: feat(db): DB_CONNECTION_MODE による Cloud SQL/IAM 接続切替を実装
- **Why**: 本改修の中核。環境変数で従来接続と Cloud SQL/IAM 接続を
  切り替えられるようにする。
- **What**: 接続切替ロジックと env バリデータ
- **How**: ローカル経路から Cloud SQL コードへ到達させないため
  動的 `await import` を採用。
```

このフォーマットには「diffに存在しない変更は記述しない」というルールも添えられており、推測で盛った文書になりません。なぜその実装をしたのかを後から追跡でき、人間の判断がかなり楽になりました。

## つまずいたこと

よかったことばかりではないので、課題と解決方法を4点セットで書きます。

| 課題 | 解決・現状 |
| --- | --- |
| 1. FE/BE横断タスクが高確率で失敗 | 両対応のプロジェクト用ワークフローを自作 |
| 2. トークン消費が大きい（※体感値） | タスクを細かく切れば抑えられるが、ジレンマあり |
| 3. 曖昧な指示書でエラー停止 | 指示書の作成段階で人間判断が必要な箇所を潰す |
| 4. どこまで任せるか（スコープ）の判断が難しい | 今はドメイン機能ごとに分割依頼。全機能の一括丸投げは今後検証 |

### 課題1: FE/BE横断タスクが高確率で失敗した

デフォルトのワークフローはフロントエンド用・バックエンド用が分かれており、それを使い分ける運用では、「デザイン → UI/UX実装 → API修正」のようにフロントエンドとバックエンドを跨ぐタスクが高確率でエラーで失敗しました。

**解決: フロント＋バック両対応のプロジェクト用ワークフローを自作して整理しました。**  
実行ログから復元した構成では、手元には両対応ワークフローが2本あります。汎用版の`dual`（plan → テスト作成 → 実装 → AIアンチパターン自己レビュー⇄修正 → 2段階レビュー → 修正 → 監督の10ステップ）と、プロジェクト特化版（6ステップ＋サブワークフロー2本）です。プロジェクト特化版の流れを図にするとこうなります。

実装＋自己レビューの`draft`と7観点並列レビューの`peer-review`は`workflow_call`で部品として呼び出し、最後に文書化ステップ（前述のWhy / What / How出力）を追加しています。

面白かったのは、両対応の実現方法が「ステップをFE用 / BE用に分岐させる」のではなかったことです。実行ログを見ると、次の**4つの知識文書を連結して、同一ステップのコンテキストに注入する**方式（いわばナレッジ合成）になっていました。

* **フロントエンド専門知識**: 層構造 / 状態管理 / 型安全性 / アクセシビリティ など
* **React知識**: effectの再実行 / Context / custom hookの責務 など
* **バックエンド専門知識**: ヘキサゴナルアーキテクチャ / バリデーション戦略 / リポジトリパターン など
* **アーキテクチャ知識**: 境界での解決 / 品質特性 / 変更スコープの評価 など

ステップ構造はシンプルなまま、知識の側で両対応させる発想です。これらの知識文書は実行記録の`context/knowledge/`に残っており、第1部で述べたFaceted PromptingでいうKnowledge facetがここで効いていることが分かります。

**「とはいえ、ワークフロー自作ってハードルが高いのでは？」** と思うかもしれません。  
実は、この自作自体もTAKTに任せられます。やることは普段のタスクと同じで、`takt`を起動し、ワークフロー開発（TAKT開発）を選択し、`default`ワークフローを選び、「こういうワークフローを作りたい」をアシスタントと壁打ちするだけ。あとは前述の`/go`（指示書生成）→ `/play`（実行）と同じ流れです。出来上がりがエラー続きだったり、思ったものと違っても、同じ修正サイクルをもう一周回せばよい。ワークフローを作る作業そのものも、結局はTAKTへの丸投げの範囲に収まりました。

### 課題2: トークン消費が大きい

レビュー⇄修正ループや並列レビューが何度も回るぶん、トークン消費は大きいです。  
この辺りについては第3部の方でどれだけ消費するか等の具体的な内容を記載する予定です。

**解決（というよりジレンマ）:** タスクを細かい粒度で切れば消費は抑えられます。ただし粒度を小さくすると、「丸投げして放置できる」というオーケストレーションの意義自体が薄れてしまいます。ここは根本対策が見つかっておらず、トレードオフとして付き合っています。

### 課題3: 指示書が曖昧だとエラー落ちする

指示書（タスクの依頼文）が曖昧だと、ワークフロー内のレビューで「人間の判断が必要」と判定され、そこでエラーとして停止します。数時間実行した終盤でこれが起きると、大きなストレスです。

**解決: 指示書の作成時点で、人間判断が必要になりそうな箇所を潰しておきます。** 具体的には、計画を担当するPlannerと密にやり取りし、「不明点はないか → 指示書を修正」のサイクルを多く回してから実行に入ります。

### 課題4: どこまで任せるか（スコープ）の判断が難しい

「どこまでをひと息に任せるか」という粒度の見極めが、地味に難しいです。今回はアプリ全体をまるっと渡すのではなく、**特定のドメイン機能ごとに区切って依頼**しました。本音は全機能を一括で丸投げしたいのですが、要件を詰める段階でアシスタント側から「分割して進めることを推奨します」と提案されることが多く、自然と機能単位に落ち着きました。

粒度を大きくすれば「丸投げして放置」の旨味は増える一方、失敗時の手戻りやトークン消費（課題2）は膨らみます。逆に小さく切れば制御は効きますが、オーケストレーションの意義は薄れます。**次は思い切って全機能を一括で投げるタスクも検証**し、どこまでなら任せきれるのかの肌感を掴みたいと考えています。

## まとめ

第2部では、TAKTを実開発に使ってみた体験を共有しました。

* 一気通貫の開発タスクを丸投げして完了まで放置でき、Claude単体で都度発生していた人間確認の手間がなくなりました
* レビュー⇄修正の強制によりコード品質は上がり、禁止ルールやWhy / What / Howログのような細かな制御も定義で実現できました
* 一方で、FE/BE横断ワークフローの整備・トークン消費・指示書の精度という新しい課題も生まれました

今後は、エラー制御の方法（すべてのエラーが解決できているわけではありません）、トークン消費の改善、そして想定外に負担が大きくなっている指示書の作成・確認工数の軽減を検討していきます。設計に投資した分だけ、安心して任せられる範囲が広がる――丸投げして放置できる代わりに、指示書とワークフローの設計が人間の新しい仕事になる、という感覚です。

### 次回予告

第2部は、あくまで筆者ひとりの実体験であり、「品質が上がった」「放置できた」も主観的な実感が中心です。第3部（検証編）では、これを客観的な数字で裏取りします。TAKTとClaude Codeに**まったく同じ指示書**（Python（FastAPI）でクリーンアーキテクチャのREST APIをSTEP1→STEP5で積み上げる）を渡し、所要時間・トークン・コスト・テスト件数といったメトリクスと、生成されたコードの中身を突き合わせて比較します。「機能品質は同点でも、そこに至る過程と対価が違う」という結果を、コードと数字で確認していきます。
