## この記事について

2026年4月16日、AnthropicがClaude Opus 4.7を一般提供開始したで。Opus 4.6から何が変わったんか、新機能・ベンチマーク・Claude Codeでの使い方のベストプラクティスまで、まとめて紹介していくで。

この記事は以下の構成で進めるで。

* **Opus 4.6から何が変わったんか** — アップデート概要、ラインナップの位置づけ、Mythosという話題のモデル
* **Chapter 1: 主要な新機能** — 高解像度画像、xhigh effort、Claude Codeの新機能
* **Chapter 2: 能力改善とベンチマーク** — SWE-bench、CursorBench、GDPval-AA、Vision
* **Chapter 3: Claude Codeプラクティス** — 検証ループ、プロンプト設計、トークン挙動
* **まとめ** — 今日から変えるべき運用ポイント

## Opus 4.6から何が変わったんか

### 知能の底上げ（3点）

まずは知能面のアップデートから見ていくで。大きく3つあるんや。

1つ目が、**コーディング・エージェント能力の大幅向上**や。代表的なベンチマークSWE-bench Proで\*\*64.3%**を記録した。Opus 4.6比でプラス10.9ptの改善や。さらにCursorBenchでは**70%\*\*と、エディタ統合の領域でも強い数字を出してる。

2つ目が、**指示遵守能力の大幅強化**。ここかなり大事なポイントやで。

| モデル | 指示の解釈 |
| --- | --- |
| Opus 4.6 | 緩く解釈する傾向 |
| **Opus 4.7** | **文字通りに解釈** |

つまり「これはしないで」と書いたらその通りに動く。そのぶん、プロンプトの書き方を見直す必要があって、**肯定的な望ましい例を中心に再点検**するのがおすすめやで。

3つ目が、**画像認識と読み取り能力の強化**。長辺2576px / 3.75MPまでの高解像度画像をサポートした。これはClaudeシリーズで初の対応や。

基本スペックはこれ。

| 項目 | 値 |
| --- | --- |
| API ID | `claude-opus-4-7` |
| 発表 | 2026年4月16日 |
| 入力単価 | $5 / 100万トークン |
| 出力単価 | $25 / 100万トークン |

Opus 4.6と同じ価格帯やで。

### 運用・プランの拡張（2点）

1つ目が、**新しいエフォートレベル `xhigh` の追加**や。これまではlow / medium / high / maxの4段階やったけど、そこに`xhigh`が加わった。**コーディング・エージェント用途ではこの`xhigh`がデフォルト推奨**になる。Claude CodeでもAPIでも共通で使えるで。

併せて `task_budgets` という機能もベータで追加。これはエージェントループ全体のソフト上限を指定できるもので、Messages API専用や。

2つ目が、**Auto modeのMaxプランへの提供開始**や。これまでClaude CodeのAuto modeは、Team・Enterprise・API経由でのみ使えたんやけど、今回のOpus 4.7リリースに合わせて**Maxプランにも展開**される。

ただし**Proプランは引き続き対象外**。この点は注意してな。

### ラインナップの位置づけとMythos

Claudeのモデルラインナップを整理するで。

| モデル | 位置づけ | Context | 入力/出力 |
| --- | --- | --- | --- |
| **Claude Opus 4.7** | 最も高性能なGAモデル | 1M | $5 / $25 |
| Claude Sonnet 4.6 | 速度と知性のバランス | 1M | $3 / $15 |
| Claude Haiku 4.5 | 最速・低コスト | 200k | $1 / $5 |

そして気になるのが **Claude Mythos**。これはProject Glasswingの中核として位置づけられてる非公開のフロンティアモデルで、Apple・Google・AWS・Microsoft・NVIDIAなど**わずか12組織のみに配布**されてる。

数字がすごくて、SWE-bench Proで **77.8%**。Opus 4.7を13pt以上上回る性能を持ってるんや。なんで広く配布しないかというと、**ゼロデイ脆弱性の発見レベル**の能力を持ってるから。安全性の観点から意図的に配布範囲を絞ってるということやな。

So Whatとしては、**一般の開発者が触れるGA最上位はOpus 4.7**。Mythosは「存在するけどまだ届かない上限値」として把握しておく、くらいの温度感でええと思うで。

## Chapter 1: 主要な新機能

### 高解像度画像サポート（2576px / 3.75MP）

入力画像の上限が大きく引き上げられたで。

| 項目 | Before | After (Opus 4.7) |
| --- | --- | --- |
| 長辺 | 1568px | **2576px** |
| 画素数 | 1.15MP | **3.75MP** |

面積比でいうと**約3.3倍の情報量**や。さらに、画像座標が実ピクセルと1:1で対応するようになったから、これまで必要やったbounding boxの座標変換が不要になるで。

ディスプレイで例えると、Full HD未満から2K・QHD相当への進化に近い。スマホのスクリーンショットの細部まで読める解像度になった、というイメージや。

**実務でのメリット3つ:**

* **ブラウザ自動化** — Playwrightなどでスクリーンショットを介した操作判断や、要素特定の精度が向上
* **Webフロントエンド開発** — 画面キャプチャを渡しての実装レビューや、デザイン差分チェックの精度が上がる
* **プレゼンスライド作成・校正** — レイアウトや図表の読み取り精度が上がり、生成品質が底上げされる

**コストの注意点:** 2000×1500pxの画像で約4,000トークン。従来の最大1,568トークンと比べると**最大約3倍のトークン消費**や。不要な高解像度の場合は事前にダウンサンプリングしとくのがおすすめやで。

### 新しい `xhigh` effortレベル

効果レベルごとの推奨用途を整理するで。

| レベル | 推奨用途 |
| --- | --- |
| `low` / `medium` | コストとレイテンシ重視の短い作業 |
| `high` | 知能とコストのバランス。並行セッション向け |
| `xhigh` | **ほとんどのコーディング・エージェントの開始点（デフォルト推奨）** |
| `max` | 本当に難しい問題だけに使う |

`max` は**過剰思考**（考えすぎ）に陥りやすい傾向がある。使い分けの基本方針は「**ほとんどのタスクは`xhigh`、最も難しいタスクだけ`max`**」や。

1つ覚えておきたい挙動として、`max`は**現在のセッションにだけ適用**される一方、それ以外のエフォートレベルは次のセッションでも維持される。`max`は一時的な切り替え、`xhigh`などは永続的な設定、というイメージやで。

### Opus 4.7と同時リリースのClaude Code機能

Opus 4.7と同時にリリースされたClaude Codeの新機能は4つあるで。

* **1. Auto modeのMaxプラン拡張** — `Shift + Tab`でトグルできる権限モード。低リスクな判断をClaudeに委任できるので、事前にコンテキストを与えた上での長時間タスクに最適
* **2. `/ultrareview`** — シニアエンジニア相当の多パスのコードレビューを実行するコマンド。Pro / Maxユーザーには無料で3回使える枠が付与されてる
* **3. `/fewer-permission-prompts`** — セッションの履歴から、頻繁に使っていて安全なbashやMCPコマンドを洗い出して、allowlistへの追加候補を提案してくれる。許可プロンプトを減らして作業効率を上げるための機能や
* **4. Recap** — エージェントが「これまでやったこと」と「次にやること」を短く要約してくれる機能。長時間タスクから復帰したとき、状況把握がすぐできて便利やで

So Whatとしては、**MaxプランでもAuto modeが解禁されたのが大きい**。長時間タスクの委任を前提に、運用を再設計する好機やと捉えてええで。

## Chapter 2: 能力改善とベンチマーク

### 能力改善の3つの柱

Opus 4.7の能力改善は3つの柱で整理できるで。

**1つ目: Coding**

* Anthropic内部タスク93本で、Opus 4.6比 **13%改善**
* Rakuten-SWE-Benchで解決できるタスク数が**約3倍**
* **self-verification（自己検証）を組み込んだ最初のモデル** — 自分で出力した結果を自分で検証する仕組み

**2つ目: Knowledge Work**

* Wordの`.docx`のredlining（赤入れ校正）や、PowerPointの`.pptx`編集について自己検証が行われる
* PIL（Python画像ライブラリ）を経由して、図表のピクセル単位の転写もできる

運用上の注意として、これまでプロンプトに入れてた\*\*「レイアウトを確認せよ」といった足場指示は、Opus 4.7では除去してベースラインを取り直すのが推奨\*\*やで。self-verificationがモデル側に内蔵されたから、プロンプトで強制する必要がなくなったってことや。

**3つ目: Vision**

* 視覚鋭度ベンチマークの**XBOWで98.5%**（Opus 4.6は54.5%でほぼ倍近い改善）
* 図表推論の**CharXiv Reasoningで82.1%**（Opus 4.6は69.1%で劇的改善）

### 14指標中12で首位

Opus 4.7は**14指標のうち12で首位**という結果を出してる。主要数字はこれ。

| ベンチマーク | Opus 4.7 | Opus 4.6 | GPT-5.4 | Gemini 3.1 |
| --- | --- | --- | --- | --- |
| SWE-bench Pro | **64.3%** | 53.4% | 57.7% | 54.2% |
| SWE-bench Verified | **87.6%** | 80.8% | — | 80.6% |
| CursorBench | **70%** | 58% | — | — |
| Terminal-Bench 2.0 | 69.4% | 65.4% | **75.1%** | 68.5% |
| OfficeQA Pro | **80.6%** | 57.1% | 51.1% | 42.9% |
| GDPval-AA (Elo) | **1,753** | — | 1,674 | 1,314 |
| BrowseComp | 79.3% | 83.7% | **89.3%** | 85.9% |

**強い領域:** SWE-bench Pro、CursorBench、OfficeQA、GDPval-AA — 他社を大きく引き離してる

**弱点として残る領域:**

* **Terminal-Bench 2.0** — ターミナル操作はGPT-5.4に負けてる
* **BrowseComp** — Webブラウジングは**Opus 4.6にも負けてる**（83.7% vs 79.3%）

用途に応じて使い分けるのが賢い選択やで。

## Chapter 3: Claude Codeプラクティス

ここが一番実務で効く部分や。4つのプラクティスを紹介するで。

### プラクティス1: Claudeに検証手段を与えるだけで成果が2〜3倍

これはAnthropicのBoris Cherny氏が強調してるポイントや。**Opus 4.7ではこれまで以上に重要になる**と語ってるで。

> 「Claudeに自分の仕事を検証する方法を与えること。これがClaudeから2倍から3倍の成果を引き出す唯一の方法であり、4.7ではこれまで以上に重要だ」

具体例を4つ挙げるで。

* **Backend** — 自動テスト、リンター、サーバ起動、APIレスポンスの検証などを渡す
* **Frontend** — Chromiumの拡張機能やPlaywrightを使ってブラウザの動作確認をさせる
* **Desktop・GUI** — Computer useを使って実際の画面操作を検証させる
* **ドキュメント** — スキーマや仕様との突合チェック

要は、**Claude自身が「正しくできたか」を確認できる仕組みを必ず渡す**、これだけで成果が数倍変わるってことや。

### プラクティス2: プロンプト設計の見直し（Opus 4.7の性格に合わせる）

Opus 4.7は指示を文字通りに解釈するで。この性格を踏まえた調整が必要や。

**4つの影響と対応:**

1. **否定形より肯定例を使う** — 「これはしないで」より「こうあってほしい」という望ましい例を示すほうが効果的
2. **ツール使用の明示指定** — Opus 4.7はデフォルトでツール呼び出しが減って推論重視に振れる傾向がある。検索やファイル読込を積極的に使わせたい場合は、その旨をガイダンスに書く
3. **サブエージェントも明示指定** — サブエージェントの生成判断が4.6より慎重になってる。並列で動かしたい場合は「同じターンで複数のサブエージェントを生成してほしい」と明記
4. **応答長もタスクの複雑度に追従** — 4.6ほど冗長には書かれへん。期待するスタイルや長さは明示的に指定する必要がある

### プラクティス3: Opus 4.7はプログラマーではなくエンジニア

これは接し方の大きな転換や。

**Claudeを「1行ずつ指示するペアプログラマー」ではなく「仕事を委任する有能なエンジニア」として扱うほうが効果的**になるんや。これはAnthropicが公式のベストプラクティス記事で強調してる考え方や。

**3つのポイント:**

* **最初のターンで全部渡す** — 意図、制約、受入基準、関連ファイル、これらを1ターン目でできる限り具体化して渡す。曖昧な指示を複数ターンに分けて段階的に与えると、トークン効率も品質も下がる
* **対話回数を減らす** — ユーザー操作が入るたびに推論のオーバーヘッドが増える。質問はまとめて渡し、モデルが処理を継続できるだけのコンテキストを先に渡すのがコツ
* **長期タスクで真価を発揮** — Opus 4.7はセッション間のコンテキスト引き継ぎが強化されてる。複雑な複数ファイル変更、曖昧な状況でのデバッグ、サービス全体のレビューなど、これまで人間の監視がボトルネックになってたタスクに特に向いてる

### プラクティス4: トークン消費と思考の挙動変化

ここは見落とすとコストが合わなくなるから、注意して読んでな。

**1. 同じ入力でもトークンが増える**

Opus 4.7では新しいトークナイザーが導入されてて、コンテンツの種類によって**約1.0〜1.35倍のトークン増**が見込まれる。Opus 4.6時代のコスト試算はそのままでは当てにならへんから、必ず再ベンチマークしてな。

**2. 長時間セッションで思考が深まる挙動**

特にエージェント設定の後半ターンで、**エフォートレベルを自分で引き上げる傾向**がある。難タスクの信頼性は向上するけど、そのぶん出力トークンも増えるで。

**3. 適応型思考という仕組み**

拡張思考の固定思考予算（「この予算の範囲で考えてね」という指定）には対応していへん。**Claudeがステップごとに思考の必要性を判断して、単純なクエリでは思考をスキップ**する。そのおかげで考えすぎにも陥りにくい設計や。

**4. 思考の深度を明示的に制御するテクニック**

プロンプトに1行入れるだけで挙動が変わるで。

* **深く考えさせたいとき**: 「答える前に、慎重に段階的に考えてください」
* **速く応答させたいとき**: 「深く考えるより素早く反応を優先してください。迷ったら直接反応してください」

覚えておくと便利や。

## ほな、まとめていこか

今日のポイントを4つに整理するで。

### 1. Auto modeがMaxプランへ拡張

`Shift + Tab` でトグルでき、低リスク判断をClaudeに委任できるモードや。併せて、`/ultrareview`、Recap、API側では高解像度画像の2576px対応、新エフォートレベル`xhigh`もリリースされてる。

### 2. Opus 4.7は「エージェント運用を前提とした」アップデート

self-verification（自己検証）を内蔵した最初のモデルで、指示を文字通りに解釈する。**長時間タスクでも深く考え続けられる「委任できるエンジニア」**、これがOpus 4.7のキャラクターや。

### 3. ベンチマークは広範に首位、ただし万能ではない

* SWE-bench Pro: 64.3%（Opus 4.6比 +10.9pt）
* CursorBench: 70%と強い
* Vision XBOW: 98.5%と激変

一方で**BrowseCompなどOpus 4.6のほうが優るケースもある**。用途に応じて使い分けるのが賢い選択やで。

### 4. 今日から変えるべき運用ポイント

* **肯定例ベースのプロンプトへ見直す**
* **最初のターンで意図・制約・受入基準を渡す**（ペアプロから委任へのシフト）
* **検証手段を必ず与える** — これだけで成果が2〜3倍になる
