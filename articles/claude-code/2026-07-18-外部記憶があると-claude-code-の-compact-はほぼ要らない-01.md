## 同じことを 2 回やっていないか

Claude Code でいくつかのセッションを回していて、ふと気づいた。

作業を終えるとき、筆者の環境では重要な決定を外部記憶に保存し、セッション全体も要約してから終了する。その直後、次の作業に備えて `/compact` を実行した。

そこで「あれ、今まったく同じ情報を 2 回要約していないか？」となった。

先に結論を書くと、自然な区切りで外部記憶へ退避できるなら、`/compact` の代わりに次の流れを使える。

```
終了処理 → /clear → 次のセッション冒頭で recall
```

これなら `/compact` の利点である文脈の継続性を残しながら、コンテキストウィンドウ (以下、コンテキスト窓) は `/clear` でまっさらにできる。本稿は、この気づきと実測した収支を軽くまとめたものである。

## /compact が実際にやっていること

`/compact` は、会話履歴をその場で短く畳んで終わり、という軽い処理ではない。

実体は、その時点のコンテキスト全量を入力にしたフラッグシップモデルの要約呼び出しである。200K のコンテキスト窓なら、auto-compact が発火する頃には約 150〜180k トークンが入力される。そして生成された要約ブロブは、次の窓の先頭に前払いで固定注入される。

次の窓も埋まれば、また同じ処理が起きる。長いセッションを続けるほど、この「窓の大部分を読ませて要約し、次の窓へ持ち越す」という処理を繰り返すことになる。

もちろん、作業途中の細かな状態まで引き継げるのは強い。一方で、生成物には次の性質がある。

* そのセッションを跨いで再利用できない
* 後から検索できない
* 過去の要約として蓄積されない
* 圧縮後の窓では、要約ブロブの分だけ最初から場所を使う

つまり `/compact` は、現在の窓にある記憶を、次の窓でも使える形へ一時的に詰め直す処理だと捉えられる。

## 外部記憶がある場合の終了処理

筆者は自作の外部記憶 MCP server「CPersona」を使っている。ただし、ここで重要なのは製品固有の機能ではなく、記憶をコンテキスト窓の外へ保存して検索できるというパターンである。同じ役割を持つメモリ系 (memory) MCP server なら、基本的な構造は変わらない。

筆者の環境では、次の 2 種類を外部 DB に残している。

* 重要な決定や再利用したい知識は、作業中に `store` で逐次保存する
* セッション終了時には、会話の要点を `archive_episode` でエピソードとして格納する

`archive_episode` に渡す summary と keywords は、より安価なサブエージェントの Sonnet に前計算させている。そのため、終了処理でメインモデルによるコンテキスト全量の要約呼び出しは発生しない。

この状態でさらに `/compact` すると、すでに永続化した情報をもう一度まとめ直すことになる。しかも 2 回目の生成物は、揮発性で、検索できず、蓄積もされない。

言い換えると、**同じ情報の 2 回目の要約を、より劣化した保存形式で作っている**。終了境界を作れる状況なら、これはほぼ無意味ではないか、というのが今回の気づきである。

## 置き換えの収支

終了境界での可変部分だけを並べると、次のようになる。

| 方法 | 圧縮・退避時の入力 | 次の窓に乗る量 | 永続性 | 後から検索 |
| --- | --- | --- | --- | --- |
| `/compact` | 約 150〜180k トークン | 要約ブロブを固定注入 | なし | できない |
| `store` + `archive_episode` + 次回 `recall` | 合計約 10k トークン以下 | `recall(limit=10)` で実測約 4.5k トークン | あり | できる |

外部記憶へ置き換える一式は、`store` 数件、`archive_episode`、次回の `recall` を合わせても約 10k トークン以下だった。`/compact` 1 回の約 150〜180k トークンに対して **15〜20 分の 1** であり、オーダーが 1 つ違う。小手先のトークン削減というより、支出構造の置き換えである。

### 4 ヶ月分の記憶と compact 6〜7 回が同じ規模

以下は推測値ではなく、筆者が約 4 ヶ月運用した外部記憶 DB を read-only で集計した実測値である。対象は agent 1 つ分である。

* memory: 1,251 件、平均 1,100 字、最大 2,472 字。1 件あたり約 1k トークン級
* episode summary: 316 件、平均 1,447 字、最大 6,577 字
* 全 memory コーパス: 1,376,423 字、約 100 万トークン前後

一番おもしろいのは最後の数字である。4 ヶ月かけて蓄積した全 memory コーパスが、入力トークン量で見ると `/compact` をわずか 6〜7 回実行する規模と同等だった。

同程度のトークン量を使うなら、一度きりで揮発する要約ブロブではなく、検索可能で永続し、セッションを横断できる記憶が 4 ヶ月分残る。外部記憶のほうが、トークンの使い道としてかなり筋がよい。

なお、生のトランスクリプトである `.jsonl` は、実測した 94 セッションでは典型的に 5〜11MB、最大 64MB だった。ただし `/compact` の実入力はコンテキスト窓の上限でキャップされる。64MB という見栄えのする数字をそのまま比較に使うと過大広告になるため、本稿では窓ベースの約 150〜180k トークンを使っている。

## `/compact` を置き換えられる成立条件

ここまで読むと、常に `/clear` でよいように見える。しかし、この置き換えには明確な成立条件がある。

### 自然な区切りが必要

置き換えられるのは、終了処理を走らせられる境界があるときである。

タスクのど真ん中で auto-compact が発火した場合、直前に読んだファイル、作業中の中間結果、まだ確定していない推論などは外部記憶へ落ちていない。ここで `/clear` すると作業状態を失うため、単純には置き換えられない。

したがって、長いタスク中の緊急退避としての `/compact` には今も意味がある。狙うべきは、タスク完了後や方針が固まった後など、人間が意図的に区切れる場面である。

### recall は確率的である

`/compact` の要約には、劣化しながらも元の文脈全体が材料として乗っている。一方の `recall` は、クエリに関連すると判定された記憶だけを取得する。必要な情報が毎回確実に返るとは限らない。

そのため、ベクトル検索、全文検索、キーワードなどを含む外部記憶側の検索品質が成立条件になる。保存して終わりではなく、次のセッションで適切に引き出せる必要がある。

### 固定費は差し引いて考える

CLAUDE.md や MCP schema などの固定情報は、`/compact` の後でも `/clear` の後でも再注入される。どちらを選んでも払い直す部分なので、比較上の差にはならない。

差が出るのは、過去の会話を丸ごと要約するか、必要な記憶だけを検索して戻すかという可変部分である。先ほどの比較表も、この可変部分の収支として整理している。

## 試すには — SKILL に任せる

本稿のワークフローを自分の環境で試すには、外部記憶 MCP server と、自然な区切りで終了処理を走らせる運用規律の両方が必要である。server を導入しただけでは、重要な情報を保存し忘れたり、`/clear` の前にセッションを退避し損ねたりする。仕組みと運用をセットで用意して、初めて「終了処理 → `/clear` → `recall`」が成立する。

筆者の構成をそのまま試す場合、CPersona には Agent Skill の `cpersona-memory` が同梱されている。この SKILL はインストーラであると同時に運用規範である。CPersona 本体のインストール、埋め込みサーバーである CEmbedding の起動、MCP 登録、動作確認までを Claude 自身が実行する。

それだけでなく、導入後の Claude には「重要な決定は即座に `store` する」「セッション終了フレーズを検知したら `archive_episode` する」という発火条件も入る。これは、本稿で述べてきた終了処理の規律そのものである。つまり、本稿が前提にした「自然な区切りで終了処理が走る」状態が、SKILL 導入だけで再現される。

前提条件は Python 3.11+ と uv 程度である。導入は、次のコマンドで SKILL を配置するところから始める。

```
# SKILL をユーザーレベルの skills ディレクトリに配置
git clone --depth 1 https://github.com/Cloto-dev/cpersona.git /tmp/cpersona
mkdir -p ~/.claude/skills
cp -r /tmp/cpersona/skills/cpersona-memory ~/.claude/skills/
```

あとは Claude Code のセッションで「CPersona をセットアップして。永続記憶が欲しい。」と言うだけ。

手動構成の詳細や内部の仕組みは、[『Claudeは明日もあなたを忘れる — MCP Memory Server cpersona 設計と実践』](https://zenn.dev/cloto/books/claude-memory-mcp-server)に譲る。

## まとめ

外部記憶がないなら、`/compact` は長い作業をつなぐためのよくできた仕組みである。しかし重要事項とセッション要約がすでに外部 DB にあり、次のセッションから検索できるなら、終了直後の `/compact` は同じ情報をもう一度、再利用しにくい形へ圧縮する処理になる。

自然な区切りでは、次の流れで十分である。

```
store / archive_episode → /clear → recall
```

**`/compact` は、記憶がコンテキスト窓の中にしかない世界の道具である。外部記憶を持って区切りさえ作れば、`/clear` の完全下位互換になる。**

筆者が使っている外部記憶 server の設計と実践は、Zenn 本の[『Claudeは明日もあなたを忘れる — MCP Memory Server cpersona 設計と実践』](https://zenn.dev/cloto/books/claude-memory-mcp-server)にまとめている。外部記憶そのものの構成に興味があれば、参考になるはずである。
