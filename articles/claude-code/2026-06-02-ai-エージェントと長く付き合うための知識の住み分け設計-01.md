現行プロジェクトではLLM Wiki(Obsidian)を使って知識管理をしている。その理由を以下に記述した

— ルール・仕様・試行錯誤を、3つの場所に分けて置くお話

## きっかけ:同じ罠を二度踏んでしまう

AI コーディングエージェントと数ヶ月単位で同じプロジェクトを進めていると、ある問題に気づきます。**エージェントは毎回「初対面」なのです。** 先週半日かけて「この方法は動かない」と結論づけた検証も、昨日「A ではなく B を採用する」と決めた判断も、次のセッションでは消えています。

素朴な対策は「全部どこかにメモする」ことです。ところが、これをやると今度は逆の事故が起きます。

* ルールブックに「試したけれどダメだった案」が混ざり、後から読んだエージェントがそれを**正規仕様**だと誤読してしまう
* 「現在の仕様」と「過去の検討メモ」が同じ場所に同居し、どちらが今の正解か分からなくなる(drift)

つまり問題は「記録するかどうか」ではなく、**「何を、どこに置くか」** だったのです。私たちのプロジェクトでは、知識の置き場所を性質によって3つに分けています。

## 3つの置き場所と、その境界線

| 置き場所 | 役割 | 一言で言うと |
| --- | --- | --- |
| **CLAUDE.md** | 絶対ルール・横断ルール | 「**必ずこうしてください**」 |
| **docs/** | 公式設計・正式仕様 | 「**現在こうなっています**」 |
| **Obsidian vault** | 試行錯誤・決定の経緯・却下案 | 「**過去にこう考えました**」 |

判断は一言ルールで足ります。

> **「ルール・仕様」なら CLAUDE.md / docs、「過程・選択・罠・却下案」なら Obsidian。**

ポイントは、この3つが**単一の真実の源泉(Single Source of Truth, SoT)として互いに排他的である**ことです。同じ事実を2箇所に書きません。重複は必ずいつか食い違い、食い違いは事故になるからです。

## いちばん大事な観点:「理由」はどこにあるのか

この3分割を、別の角度から眺めると本質が見えてきます。**「理由(なぜ)」がどこに宿っているか** で並べ替えてみましょう。

| 置き場所 | 書いてあること | 理由(なぜ)は? |
| --- | --- | --- |
| **CLAUDE.md** | ルール(「こうしろ」) | **無い** |
| **docs/** | 現在の形(「こうなっている」) | **無い** |
| **Obsidian** | やったこと・やらなかったこと | **両方に理由がある** |

つまり、こういうことです。

* **CLAUDE.md にはルールはありますが、理由はありません。** 「必ずこうしろ」とだけ書いてあり、なぜそのルールに至ったかは書きません(書くと肥大化し、例外探しの余地を生み、ルールが揺らぎます)。
* **docs/ には現在の形はありますが、理由はありません。** 「今こうなっている」を書きますが、なぜ別の設計ではなくこの設計になったのかは書きません。
* **Obsidian に収めるのは、やったことと、やらなかったこと、そして "それぞれの理由" です。** 採用した案にも理由があり、却下した案にも理由がある。その**理由の集積**こそが Obsidian の正体です。

ルールと仕様は「結論」です。結論は短く、揺るがず、理由を語りません。理由を知りたくなったとき(=ルールを疑いたくなったとき、設計を変えたくなったとき)に開くのが Obsidian です。**結論と理由を別の場所に置く**——これがこの設計の核心です。

## Obsidian に入れるもの

Obsidian に書くのは、SoT には決して載らない種類の情報に限ります。

* **調査の過程**(`investigations/`)— 原因究明でどの signal を見て、何と何を突き合わせたか
* **決定の経緯**(`decisions/`)— なぜ A を選び、**B を却下したのか**
* **つまずき**(日次ノート)— 30分以上ハマった罠、踏んでしまった落とし穴
* **却下した仮説** — 「これは試したけれどダメだった」

却下案こそが Obsidian の主役です。CLAUDE.md は「結論」しか書きません。なぜその結論に至り、何を捨てたのかは Obsidian にしか残らないのです。これが「同じ罠を二度踏む」のを防いでくれます。

## Obsidian に入れないもの、とその理由

入れるものと同じくらい大事なのが、**入れないものを決めておく**ことです。ここが曖昧だと、Obsidian はやがて「何でも倉庫」になり、検索が効かなくなって、過去の罠を引き当てる本来の役目を失います。

| 入れないもの | 正しい行き先 | なぜ入れないか(理由) |
| --- | --- | --- |
| **絶対ルール・横断ルール** | root / サブの CLAUDE.md | SoT が二重化する。ルールが2箇所にあると、片方の更新漏れで必ず食い違い、エージェントがどちらを信じるか分からなくなる(drift) |
| **公式設計・正式仕様** | docs/ | 仕様の正典は docs/ ただ一つ。Obsidian に写すと「古い仕様のコピー」が残り、誤読の温床になる |
| **コードを読めば分かること**(構造・過去の修正・git 履歴) | コード本体 / git | コードと git が SoT。重複させても腐るだけで、価値を生まない |
| **既存ノートと重複する内容** | 既存ノートを更新 | 同じ事実の二重記録は食い違いの種。新規作成ではなく追記・更新する |
| **AEGIS と無関係な話題**(Claude Code の使い方一般・汎用プログラミング・他プロジェクト・雑談) | どこにも書かない | 検索の信号対雑音比(S/N)を下げる。ノイズが増えるほど「過去に同じ罠を踏んだか」の検索が当たらなくなる |
| **一般化・安定化して "もう試行錯誤ではない" 知見** | docs/ へ昇格 | 定着した事実は仕様。Obsidian に置き続けると「現役の試行錯誤」と「確定事実」が混ざる |

判断に迷ったときの軸はこうです。

> **「これは "結論(ルール・仕様)" か? それとも "過程・理由" か?」**  
> 結論なら CLAUDE.md / docs。過程・理由なら Obsidian。**どちらでもない無関係なものは、どこにも書かない。**

「記録のための記録」はしません。**残すべきは、次に誰かが同じ場所に立ったとき役に立つ情報だけ**です。

## 新しいことを始めるとき:まず「すでにありました」を疑う

この設計が本当に効くのは、**何か新しいことを始めようとした瞬間**です。

過去の作業が Obsidian に蓄積されていると、新しいタスクに着手する前に\*\*「これは過去にやったことではないか?」を検出できます。\*\* だから私たちは、着手前に必ずこのルーティンを回します。

> **着手する前に検索し、ヒットすれば「すでにありました」と確認・報告してから動く。**

いきなり手を動かさず、まず過去の自分に問い合わせるのです。検索で見つかれば、「これは過去に扱っています。当時はこうしました」と確認したうえで作業に入れます。最良の場合、**そもそも作り直す必要がなかった**ことが分かります。検出 → 確認 →「すでにありました」という報告までを、一つのルーティンとして固定しています。

このとき投げる問いは3つです。

1. **これは過去にやったことか?**  
   まず検索します。同じ問題、似た問題に過去の自分が触れていないかを確かめます。ここで `archived/`(廃止・置き換え済み)を含む結果は除外します——古い結論をつかんで、また同じ誤読を始めないためです。
2. **そのとき何をしたか?**  
   ヒットしたら、当時「何を試し、何を採用し、何を却下したか」を読みます。CLAUDE.md と docs/ は「今の結論」しか教えてくれませんが、Obsidian は「そこに至る道筋」を教えてくれます。採用案の理由と却下案の理由——この両方が、今回の判断材料になります。
3. **何が足りなかったか?**  
   過去にやって**うまくいかなかった**のなら、何が欠けていたのかを見ます。前提が変わったのか、見落とした signal があったのか、単に時期尚早だったのか。決定ノートの「成果確認」欄(後述)を読めば、当時の仮説が実際どう転んだかが分かります。**「足りなかったもの」が、今回最初に埋めるべきものです。**

この3問を通すと、新しい作業は「ゼロからの思いつき」ではなく「過去の続き」になります。空振り(調べて何も無かった)だったとしても、それ自体を記録します。「ここは未踏だ」という事実も、次の誰かにとっては立派な手がかりだからです。

## 「決定ノート」は、振り返りのためのドキュメントでもある

Obsidian は「過去に何をしたか」を残す場所ですが、それだけではありません。**やったことを振り返り、「効果があったのか/なかったのか」「そもそも正しかったのか」、さらには「そもそも確認されているのか」を知るためのドキュメント**でもあります。

だから決定ノートには、書いて終わりにせず\*\*「成果確認」セクションを必須\*\*にしています。

* **仮説** — この決定で何が良くなると考えたか
* **観察する signal** — 効果をどう測るか
* **振り返り日**(デフォルトは1週間後)
* **結果欄** — 実際どうだったか

この4点は、振り返りの3つの問いにそのまま対応します。

| 振り返りの問い | どこを見るか |
| --- | --- |
| **効果があったか/なかったか** | 結果欄 ×(仮説・観察 signal) |
| **そもそも正しかったか** | 結果欄が仮説を支持したか、裏切ったか |
| **そもそも確認されているか** | 結果欄が**埋まっているか、空のままか** |

3つ目が、いちばん見落とされがちで、いちばん大事です。

私たちは決定を下すのは得意ですが、**「その決定が効いたかどうかを確かめる」のはたいてい忘れます。** 効果があったかどうか以前に、**そもそも誰も確認していない**——これが現場で最も多い状態です。「やった」「決めた」だけが残り、「で、どうだったの?」が永遠に空欄のまま積み上がっていきます。

そこで結果欄が空のまま振り返り日を過ぎたノートは、`scripts/list_pending_outcomes.py --overdue` が抽出し、**precommit の冒頭で警告**します。「あなたは2週間前にこれを決めましたが、結果を確認していません」と突きつけてくるわけです。これによって、

* **効果の有無**(結果欄の中身)だけでなく、
* **確認そのものの有無**(結果欄が空かどうか)

の両方を、仕組みで可視化しています。**「決めっぱなし」「やりっぱなし」を防ぐ**のが狙いです。

そしてこの結果欄こそが、未来の自分が「過去に何を試し、それは正しかったのか、何が足りなかったのか」を読み解く一次資料になります。新しいことを始めるときの3つ目の問い——「何が足りなかったか?」——に答えられるのは、過去の決定がきちんと振り返られ、結果が記録されているからなのです。

## 蒸留:ノートは腐る前提で運用する

記録を貯め続けると、いつか現実と乖離します。ですから全ノートの冒頭に状態フラグ(`status:`)を持たせています。

* `active` — 現役
* `superseded by [[...]]` — 別ノートに置き換わった
* `obsolete (理由)` — 廃止

現実と乖離したノートは、**許可を待たずに** obsolete 化して `archived/` へ退避し、事後報告します。さらに検索時には `archived/` を含む結果を除外します。これをやらないと、廃止済みのノートが検索トップに出てきて、また誤読が始まってしまいます。

そして重要なのは逆方向の動きです。**一般化・安定化した知見は `docs/` に昇格させます。** Obsidian で「これはもう試行錯誤ではなく定着した事実だ」となったものは、SoT である docs/ に引き上げて、Obsidian からは消します。知識は下(試行錯誤)から上(仕様)へと流れていきます——ただし、**理由は Obsidian に置いたまま**にします。docs/ に上がるのは「形」だけ、「なぜ」は依然として Obsidian の担当です。

## ルールブック自体も、肥大化させない

実は CLAUDE.md にも同じ「住み分け」の発想が効いています。プロジェクト横断の絶対ルールだけを root に置き、**領域固有のルール(Flask / Next.js / deploy など)はサブディレクトリの CLAUDE.md に分散**させて、その領域のファイルを触るときだけ読み込まれるようにしています。ルーティング索引が1枚あって、どのルールがどこにあるかを示してくれます。

ここでも原則は同じです。**重複させません。** 領域ルールをサブに書いたら root からは消します。SoT は常に一箇所です。

## まとめ:住所が決まっていれば、迷わない

| 書きたいこと | 行き先 | 理由は? |
| --- | --- | --- |
| 「必ずこうしてください」(絶対・横断) | root CLAUDE.md | 無い |
| 「必ずこうしてください」(領域固有) | サブディレクトリの CLAUDE.md | 無い |
| 「現在こうなっています」(設計・仕様) | docs/ | 無い |
| 「やったこと・やらなかったこと・それぞれの理由」 | Obsidian | **ここに集約** |
| コード構造・git 履歴で分かること | コード / git | 重複させない |
| AEGIS と無関係 | どこにも書かない | ノイズ排除 |

AI エージェントは毎回初対面ですが、**住所の決まった知識**を渡せば、初対面でも正しく動けます。ルールと仕様で「今どうすべきか」を即座に知り、迷ったら Obsidian を開いて「なぜそうなのか」「過去に何を試し、何が足りなかったか」をたどる。

そして新しいことを始めるときは、いきなり手を動かす前に必ずこう問います。**「これは過去にやったことか? そのとき何をしたか? 何が足りなかったか?」** 過去の作業は Obsidian が検出してくれるので、「すでにありました」と確認・報告してから動けます。その答えは、結論(CLAUDE.md / docs)ではなく、理由の置き場所(Obsidian)にあります。

「記録するかどうか」で悩むのをやめて、「これはルールか、仕様か、それとも"理由"か。それとも、どこにも書かないノイズか」を問いましょう。**迷ったら記録する。ただし、正しい住所に。**
