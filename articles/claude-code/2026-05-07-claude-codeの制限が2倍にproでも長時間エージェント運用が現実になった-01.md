2026年5月6日、AnthropicがClaude Codeまわりで大きめの発表をした。要点はシンプルで、

* Claude Codeの5時間レート制限を2倍にする
* Pro / Maxのピークタイム制限を撤廃する

の2つ。

普段からClaude Codeを触っている人なら、これがどれくらい効くかは想像がつくと思う。リファクタの途中で制限に当たる、複数ファイルを読ませて設計から実装まで任せたら息切れする──このあたりが日常的に起きていた人にとって、地味に大きい改定になっている。

公式によると、5時間レート制限はPro / Max / Team / seat-based Enterpriseで2倍。Pro / Maxではピークタイム時の制限低下も撤廃された。Maxだけでなく、Proユーザーにもがっつり効くアップデートだ。

## 何が変わったか

### 1. 5時間制限が2倍

Claude Codeは普通のチャットと違って、コードベース全体を読ませたり、複数ファイルを編集させたり、テストを回したりするので、使用量の食い方が桁違いに大きい。体感としては相談できるAIというより開発環境に住みついているエージェントに近い。

ここの制限が緩むと、扱いやすさが一段変わる。日常開発で途中で止まるがだいぶ減るはず。

### 2. Pro / Maxのピークタイム制限が撤廃

これも嬉しい。AIサービスは混雑時間帯に体感の使用量が落ちることがあって、特に日本時間の夜や、海外のピークと重なる時間帯に「今やりたいのに」となりがちだった。

上限が増えただけでなく使いたい時間に使えるようになったのが大きい。

### 3. 背景にあるSpaceXとのcompute契約

今回の緩和は気合いで捻出したわけではなく、ちゃんと裏付けがある。AnthropicはSpaceXのColossus 1データセンターのcompute capacityを使う契約を結んでおり、1か月以内に300MW超、22万基超のNVIDIA GPU相当の新規capacityにアクセスできるとしている。

Claude Codeは便利だが、裏側のcomputeをとにかく食う。AIコーディングの競争はもうモデル性能だけではなく、どれだけ長く、安定して、実務に投入できるかのフェーズに入ってきている。

## Proでも戦えるか

Claude Proは月払いなら$20/月で、Claude Codeも含まれる。

余裕という意味ではもちろんMaxが上だが、今回の改定でProでもかなり現実的に運用できるラインに来た。Proで狙いたい使い方はざっくりこのあたり。

* 朝に設計を相談する
* 昼に小さめの実装を任せる
* 夕方にテストとレビューを回す
* 夜にリファクタや記事化を手伝わせる

このくらいの「開発の相棒」運用なら、十分に乗ってきた。

逆に、複数リポジトリを同時に走らせる、巨大PRを何本も並行レビューする、丸一日エージェントを並列稼働させる、といった運用ならMaxを検討した方がいい。

## まずやること：Claude Codeを最新化

```
claude update
claude --version
```

公式ドキュメントではv2.1.105〜v2.1.113あたりで、Opus 4.7、xhigh effort、Routines、/ultrareview、/usageの改善などがまとめて入っている。古いバージョンのまま使うのはもったいない。

## おすすめ設定：Opus + xhigh

今いちばん試したい組み合わせは、Opus 4.7とxhigh effort。

公式によると、Opus 4.7はMaxとTeam Premiumのデフォルトになり、`/model`から利用可能。xhighはhighとmaxの間に位置するeffortレベルで、coding / agentic taskに向いているとされる。

```
/model opus
/effort xhigh
```

maxは強いがそのぶん重い。highだと複雑なリファクタや調査ではやや物足りない。xhighはちょうど中間で、賢さは欲しいけど毎回maxはやり過ぎという現場の温度感に合う。

ざっくり使い分けるなら、

* 軽い修正：Sonnet / medium
* 普通の実装：Opus / high
* 重めの設計・調査・リファクタ：Opus / xhigh
* 絶対に外したくない大仕事：Opus / max

くらいの感覚。

## /usageで使用量の内訳を見る

制限が2倍になった分、つい雑に使いがちだが、Claude Codeは使い方次第で今でも普通に使用量を食う。

公式ドキュメントによると、`/usage`では並列セッション、サブエージェント、キャッシュミス、長いコンテキストなど、制限に効いている要因を確認できる。

制限に当たりやすい人は、だいたい次のどれかをやっている。

* 巨大ファイルを毎回読ませている
* 不要なコンテキストを抱えたまま作業している
* 同じ説明を何度もさせている
* 複数セッションを雑に並列化している
* テスト失敗のログを丸ごと貼り続けている

余裕ができた今こそ、自分の使い方を可視化しておく価値がある。

## /ultrareviewでPR前レビューを任せる

PR番号を指定するなら、

公式ドキュメントによると、`/ultrareview`はブランチをClaude Code Web上の並列レビュアーに展開し、検出結果に対して批評パスを実行、検証済みのレポートを返す機能。

これがかなり実務向き。PRを出す前に、

* この差分にバグはないか
* 境界値の考慮漏れはないか
* 命名や責務分離は妥当か
* テストは足りているか

を一通り聞ける。制限緩和のおかげで、こういう重めのレビューを日常的に差し込みやすくなった。

いきなり本番コードを大改造させるより、まずレビュー用途から入る方が安全だと思っている。AIの書いたコードが正しい保証はないが、既存の差分を読ませて怪しいところを探させるのはかなり相性がいい。

## Routinesで定期実行されるClaude Codeへ

Claude Code WebのRoutinesも面白い。スケジュール、GitHubイベント、API呼び出しをトリガーにして、テンプレート化されたクラウドエージェントを動かせる機能だ。

たとえばこんな使い方ができる。

* 毎朝9時にPRレビュー
* releaseブランチ作成時に変更点を要約
* Issue作成時に再現手順の不足をチェック
* mainへのmerge前にテスト観点を洗い出す
* 週1で依存パッケージ更新の影響を調べる

ターミナルからスケジュールを作る例もある。

```
/schedule daily PR review at 9am
```

ここまで来ると、Claude Codeは質問に答えるAIというより、開発チームの横で勝手に見張ってくれる補助エージェントに近い。

## Proでやるなら、このワークフロー

Proで運用するなら、自分はこの流れに落ち着いた。

### 1. 作業範囲を絞る

いきなり「このリポジトリ全体を改善して」は危険。対象を絞って依頼する。

> このPRの差分だけ見てください。認証まわりには触らないでください。テストが落ちた場合は、修正前に原因を説明してください。

エージェントは自由にさせるほど暴れる。自由研究させるのではなく、業務委託する感覚で頼む方が安定する。

### 2. 実装前に計画を書かせる

いきなり編集ではなく、まず計画。

> まず変更計画だけ出してください。ファイルはまだ編集しないでください。影響範囲、変更ファイル、テスト方針を箇条書きで。

ここで変なことを言っていたら止める。まともなら次に進む。

### 3. 小さく実装させる

> では、まずバリデーション部分だけ修正してください。UIには触らないでください。

一気にやらせるとレビューがしんどい。AIが速くなった分、人間の確認能力がボトルネックになる。

### 4. テストとレビューを別タスクにする

実装が終わったら、別の観点で確認。

> 今回の変更に対して、足りないテスト観点を洗い出してください。

さらにPR前に`/ultrareview`をかける。

この流れだと、Claude Codeを作るAIだけでなく見るAIとしても使える。

## 注意：2倍でも無限ではない

ここは強調しておきたい。

長いコンテキスト、巨大ログ、並列セッション、サブエージェント、キャッシュミスは今でも普通に使用量を食う。`/usage`で内訳が見えるようになったので、まずは自分の使い方を可視化するのが先。

あと、AIの生成コードは普通に間違える。怖いのはコンパイルエラーではなく、動くけど仕様がズレているコード。Claude Codeは強いが、最終責任は人間。任せる範囲と自分で見る範囲は分けた方がいい。

## まとめ

今回のClaude Code制限2倍化は、地味にデカい。Proユーザーにとってはちょっと試すツールから日常開発に組み込めるツールに位置づけが変わる規模の改定だ。

要点だけ:

* 5時間レート制限がPro / Max / Team / seat-based Enterpriseで2倍
* Pro / Maxではピークタイム時の制限低下が撤廃
* 背景にSpaceX Colossus 1を含む大規模なcompute増強
* `/effort xhigh`、`/usage`、`/ultrareview`、Routinesとの相性が良い
* 2倍でも無限ではないので、使用量と作業範囲の管理は引き続き必要

Claude Codeはここに来てだいぶ開発OSっぽくなってきた。コードを書く、レビューする、テスト観点を出す、定期的にPRを見る、設計の壁打ちをする──ここまで一つの開発フローに組み込めるなら、Proでも元は取れる。

AIコーディングはもう使うかどうかではなくどこまで任せるかのフェーズに入ってきた。今回の緩和で、任せられる時間がぐっと伸びた感じがある。
