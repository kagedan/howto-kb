## Opus 4.8は何が変わったのか

Anthropicの発表によると、Claude Opus 4.8は Opus 4.7をベースに、coding、agentic task、reasoning、professional workの領域で改善されています。

価格は Opus 4.7から据え置きです。Claude APIの価格表では、Opus 4.8も Opus 4.7と同じく、**入力 $5 / MTok、出力 $25 / MTok**です。

今回、特に印象に残ったポイントは以下です。

* コードの欠陥を「指摘せず通す」確率が、前モデルの **約1/4に低下**
* **Legal Agent Benchmark**で、初めて **10%を突破**
* **Online-Mind2Web**（browser agent系の評価）で **84%**
* **Super-Agent benchmark**では全 caseを end-to-endで完走し、GPT-5.5と同等コスト条件で勝利
* Opus 4.7から Opus 4.8への移行は **breaking changeなし**
* Claude Managed Agents利用者は、基本的に model名を変えるだけ
* **Fast modeでは 2.5倍速く、前モデル向け fast modeより 3分の1のコスト**で利用可能

もちろん、ベンチマークはあくまでベンチマークです。実際の業務で使う場合は、自分たちのユースケースで evalを回す必要があります。

ただし、今回の発表で重要なのは、個々の数字そのものよりも、**Anthropicがどこを伸ばしているのか**だと思います。

## 一番大きいのは honestyの改善

個人的に一番大きいと感じたのは、honestyの改善です。

Anthropicは、Opus 4.8について「不確かなことを不確かだと示し、根拠の薄い進捗主張をしにくくなった」と説明しています。

さらに、評価では「自分が書いたコードの欠陥を、そのまま見逃して通す」確率が、前モデルに比べて **約4分の1** になったとされています。

これは、Agentベースの開発ではかなり効く改善です。

AIエージェントを使っていると、よくあるのが、

という問題です。

コードは生成できる。テストも一部通る。説明もそれらしく書ける。

しかし、実際には仕様の読み違いがあったり、境界条件を落としていたり、実装方針に無理があったりすることがあります。

このとき、人間側のレビュー負担はかなり大きくなります。AIが「できました」と言ったものを、結局人間が疑いながら確認しなければならないからです。

Opus 4.8の honesty改善は、このレビュー負担を少し下げる方向に効くのではないかと感じています。

つまり、モデルが単に「答えを出す」のではなく、

> 自分の出力に対して、どこまで不確実性や欠陥を検出できるか

という方向に進んでいる。これは、Agentの実用性に直結します。

## Agentic taskの評価も一段上がっている

Opus 4.8では、agentic task系の評価も目立ちます。

たとえば、browser agent系の評価である **Online-Mind2Web** では \*\*84%\*\*とされ、Opus 4.7や GPT-5.5から大きく前進したと紹介されています。

また、**Super-Agent benchmark** では、全 caseを end-to-endで完走した唯一のモデルとして紹介され、GPT-5.5と同等コスト条件で上回ったとされています。

**Legal Agent Benchmark** でも、初めて **10%を突破** したという点が紹介されています。

これらは、それぞれ単独で見ると限定的なベンチマークです。

しかし、共通しているのは、**単発の Q&Aではなく、複数ステップの作業を継続し、判断し、最後までやり切る能力**が問われている点です。

Agentベースのシステムでは、この「最後までやり切る」能力が非常に重要です。

* 途中で方針を見失わない
* 必要なツールを呼び出す
* 間違えたら戻る
* 検証する
* 必要なら別の観点から再確認する

このような一連の動きが、今後のAI活用ではますます重要になります。

## Dynamic Workflowsとの組み合わせが大きい

同じ日に、Anthropicは Claude Codeの **Dynamic Workflows** も発表しました。

これは、Claude Codeが1つの session内で、**同時に最大16・累計で最大1,000の subagent**を並列に動かし、大きなタスクを分解・実行・検証する仕組みです。

Dynamic Workflowsの起動方法は2通り用意されています。

1つは `/effort` メニューで `ultracode` を選ぶ方法です。これは **xhigh effort + workflow 自動判断** がセットになった設定で、Claude側がタスクごとに workflow が必要かを判断してくれます。

![Claude Code v2.1.156 の /effort メニュー。最右に ultracode（xhigh + workflows）が並ぶ](https://static.zenn.studio/user-upload/deployed-images/5388970cf786813528992847.jpg?sha=0e98a35a5bfa0b4c5a38b445a72215853f6507f9)  
*Figure 2: `/effort` メニューの最右に追加された `ultracode`。Faster ↔ Smarter のスライダーで Smarter側の端に位置する。*

もう1つは、プロンプトに `workflow` という単語を入れる方法です。入れた瞬間、その単語が虹色でハイライトされ、入力欄の下に `Dynamic workflow requested for this turn · meta+w to ignore` という通知が出ます。

![プロンプトに dynamic Workflows と入力した瞬間、Workflows の文字が虹色でハイライトされる](https://static.zenn.studio/user-upload/deployed-images/51e9267a3f80af1c1b01ba8f.jpg?sha=74e2cb2ad03f1264693f7f0423d91ce7bfe1ff67)  
*Figure 3: `workflow` キーワードを認識した瞬間の Claude Code。意図せず workflow を起動してしまわないよう、明示的な確認 UI が入る。*

どちらの起動方法でも、Claude側に「タスク全体の orchestration を任せる」という意思決定が、ユーザーから明示的に渡される設計になっています。

これまでの Claude Codeでも subagentは重要でした。しかし Dynamic Workflowsでは、より大きな単位の作業を Claude自身が計画し、subagentに分解し、並列に実行し、結果を検証して統合する方向に進んでいます。

発表では、コードベース全体の bug hunt、optimization audit、security audit、大規模 migration、modernizationなどがユースケースとして挙げられています。

これは、かなり大きな変化です。

これまでのAIコーディングは、どちらかというと、

* 1ファイルを直す
* 1機能を追加する
* 1つのエラーを解消する
* 1つの pull requestをレビューする

という単位が中心でした。

しかし Dynamic Workflowsの方向性は、

* コードベース全体を見る
* 複数の観点で並列に調査する
* 独立した subagentに検証させる
* 結果を統合する
* 必要なら修正ループを回す

というものです。

つまり、AIが単なる補助ツールではなく、**小さな開発チームのように振る舞う方向**へ進んでいるように見えます。

## Dynamic Workflowsは推論パラダイムの実装第一弾かもしれない

Dynamic Workflowsを「Claude Codeの便利な新機能」として読むことは、もちろんできます。

しかし、研究側の動きと並べて見ると、もう少し大きな話に見えてくる気がしています。

MIT CSAILの Alex Zhangら3名から、**[Recursive Language Models (RLM)](https://github.com/alexzhang13/rlm)** という論文が出ています。LLMに任意の長さのプロンプトを扱わせるための、新しい推論パラダイムを提案したものです。

RLMの発想はシンプルです。

普通のLLMは、長いプロンプトをそのまま context windowに詰め込みます。RLMは違います。

長いプロンプトをLLMの「外部環境」として扱い、LLMはその環境をプログラム的に分解し、必要なら自分自身を再帰的に呼び出します。

つまり、プロンプトを **「読ませる」のではなく、「読むためのコードを書かせる」** 発想です。

![RLM論文 Figure 1: GPT-5（左）と RLM(GPT-5)（右）の比較](https://static.zenn.studio/user-upload/deployed-images/173b7a681ff643ea55c0ad01.jpg?sha=02d7a589aabbb3496f9423dd82db842f1d9bc238)  
*Figure 1: 同じ long-contextタスク（S-NIAH / OOLONG / OOLONG-Pairs）で、GPT-5単体（左）は context windowの限界である 262K以降で機能停止するのに対し、RLM(GPT-5)（右）は 1Mまで性能を維持する。出典: Zhang et al., [Recursive Language Models](https://github.com/alexzhang13/rlm), MIT CSAIL.*

論文の数字を抜粋すると、

* GPT-5を RLM化したものは、4つの long-contextタスクで、compactionより中央値 **+26%**、CodeAct with sub-callsより **+130%**、Claude Codeより **+13%** の改善を、近いコストで達成
* 小規模では Qwen3-8Bを post-trainした **RLM-Qwen3-8B** が、元の Qwen3-8Bより中央値 **+28%**、3タスクで vanilla GPT-5の品質に近づく

とされています。

ここで興味深いのは、著者の Alex Zhang本人が、Dynamic Workflowsの発表を受けて X でこう書いていることです。

> もしダイナミックワークフローがなぜそんなに強力で未来のものなのか気になったら、RLM論文を読んでみてください。Claude Codeでの Opus 4.8 + Dynamic Workflowsは、おそらくフロンティアモデルがRLMとして本気で訓練された最初の事例だと思います。1年以内に、これらがほぼすべてのコーディングエージェントのやり取りの標準になると私は予想しています。

これが正しければ、Dynamic Workflowsは単なる便利機能ではないことになります。

モデル自体が「**自分を再帰的に呼び、サブ問題に分解する**」前提で訓練されている、ということになるからです。

言い換えると、Claudeが JavaScriptで orchestrationスクリプトを書ける能力は、prompt engineeringの工夫ではなく、Opus 4.8という model側に組み込まれた性質である、という主張です。

正直、この点はまだ完全には裏取りできていません。Anthropic側も、ここまで踏み込んで語ってはいません。

ただ、もし本当だとすれば、Agent設計の前提が一段ずれることになると感じています。

これまでは、context windowをどう節約するか、どこを要約するか、どこを切り捨てるかが、Agent設計の中心的な悩みでした。

RLMのパラダイムでは、**context windowは「全部を渡す場所」ではなく、「必要な部分を呼び出す入口」になります。**

扱える長さの上限は、モデル単体の context windowではなく、再帰呼び出しの深さとコストで決まります。

「どこまで読ませるか」ではなく、「どこまで再帰的に分解させるか」に近い設計思想です。

このパラダイムが本当にフロンティアモデルに焼き込まれ始めたのだとすると、Agent設計の前提も、ゆっくり変わっていくはずです。

1年以内に標準になるかは分かりません。

それでも、Dynamic Workflowsを「Claude Codeの便利な新機能」として片付けるだけでは、見落とすものがある気がしています。

## Opus 4.7から 4.8への移行は比較的軽い

開発者視点でありがたいのは、Opus 4.7から Opus 4.8への移行が比較的軽いことです。

Anthropicの migration guideでは、Opus 4.7で動いているコードは、Opus 4.8でもそのまま動くと説明されています。

基本的には、model名を `claude-opus-4-7` から `claude-opus-4-8` に変えるだけです。

もちろん、確認すべき点はあります。たとえば、

* effortの設定
* 1M context window
* mid-conversation system messages
* prompt cachingの minimum
* `stop_details`
* cost / latencyの再ベースライン

などです。

ただ、これは「破壊的変更に対応する」というより、**よりよく使うための再調整** に近いと思います。

Claude Managed Agentsを利用している場合も、Messages APIのように細かい harnessを自前で持っていないため、基本的には model名の更新が中心になります。

このあたりは、Agentを本番利用する立場ではかなり重要です。モデルの性能が上がっても、移行負荷が高すぎると採用しづらい。しかし、今回の Opus 4.8は、少なくとも Opus 4.7からの移行に関しては、かなり現実的な更新に見えます。

## ROIの計算をやり直すタイミングかもしれない

今回の発表で、もう一つ重要だと思ったのは価格です。

Opus 4.8は、Opus 4.7と **同じ価格** で提供されています。つまり、同じ単価で agentic taskの判断力や honestyが上がることになります。

これは、AI活用のROI計算に直接影響します。

たとえば、AIエージェントを業務で使う場合、単純な token costだけで判断するのは不十分です。本当に見るべきなのは、

* 人間のレビュー時間がどれだけ減るか
* 手戻りがどれだけ減るか
* 誤った実装や判断をどれだけ早く検出できるか
* 何時間・何日かかっていた調査をどこまで短縮できるか
* 成果物の品質を維持したまま、どこまで自動化できるか

という点です。

もし honestyの改善によって、AI自身が欠陥や不確実性をより正直に出せるようになるなら、人間側のレビュー設計も変わります。

すべてを人間が疑って確認するのではなく、

> AIにも一定の自己検証を担わせたうえで、人間はより重要な意思決定や最終レビューに集中する

この方向に進むと、AIの価値は「作業時間の短縮」だけではなく、**レビュー負荷の削減** や **判断品質の向上** にも広がっていきます。

## Fast modeの使い分けも重要になる

Opus 4.8では Fast modeも提供されています。発表では、Opus 4.8の Fast modeは **2.5倍速く、前モデル向け fast modeより 3分の1のコスト** で利用できるとされています。

ここも、実運用ではかなり効きそうです。

すべてのタスクに最高品質・最高 effortを使う必要はありません。たとえば、

* 仕様の初期整理
* 軽いリファクタ案の作成
* ログの一次分類
* 既知パターンの問い合わせ対応
* 下書き生成
* 単純な調査タスク

のようなものは、Fast modeや低めの effortで十分なケースがあります。

一方で、

* 大規模 migration
* セキュリティレビュー
* 複数システムにまたがる設計判断
* 本番影響のあるコード変更
* 法務・金融・医療など高リスク領域の分析

では、通常の Opus 4.8や高い effort設定を使うべき場面が増えるはずです。

つまり今後は、単に「どのモデルを使うか」ではなく、

> どのタスクに、どのモデルを、どの effortで、どの harnessと組み合わせるか

が重要になります。

## 最近の Anthropicの動きは「点」ではなく「線」で見るべき

ここ数日の Anthropicの動きを見ると、かなり一段上がった印象があります。

* Claude Managed Agentsのような公式 harnessが出ている
* CodeRabbitのような orchestration事例が公式 blogに並んでいる
* Claude Codeでは Dynamic Workflowsが機能化された
* そして、モデル自体も Opus 4.8に進化した

これらは、別々の発表に見えます。

しかし、Agentベースのシステムを作っている立場から見ると、かなり一貫した流れがあります。

それは、

> モデル単体の性能向上から、**Agentを長く・安全に・検証しながら動かすための実行基盤** へ

という流れです。

LLMは、単発のチャットから、作業を実行する存在へ変わりつつあります。

そのときに必要なのは、モデルの賢さだけではありません。必要なのは、

* どの文脈を渡すか
* どの toolを使わせるか
* どこで subagentに分けるか
* どこで検証させるか
* どこまで自動実行させるか
* どこで人間が止めるか
* 失敗したときにどう戻るか

という設計です。

Opus 4.8は、その中の「モデル側の判断力」と「自己検証力」を一段押し上げたアップデートだと思います。

## まとめ

Claude Opus 4.8は、派手な名前の新世代モデルというより、**Opus 4.7を実用面でさらに磨いたアップデート** に見えます。

しかし、Agentベースの開発という観点では、かなり意味があります。

特に大きいのは、**honestyの改善** です。

AIが「できました」と言うだけではなく、

> 「ここは怪しい」「ここは確認が必要」「この実装には欠陥があるかもしれない」と言える方向に進む

これは、エージェントを本番業務で使ううえで非常に重要です。

そして Dynamic Workflowsと組み合わせることで、AIは単なるコード補助ではなく、複数の subagentを使って調査・実装・検証を進める存在になっていく。

その背景には、RLMのような「プロンプトを読むのではなく、読むためのコードを書く」推論パラダイムの研究があり、Opus 4.8はそこを本気で訓練し始めたモデルかもしれない、という観測もあります。

今回の Opus 4.8は、単なるモデル更新ではなく、

> エージェント主体の開発の天井が、また少し上がったアップデート

だと感じています。

これからAIエージェントを設計するうえでは、モデルの性能だけでなく、**harness、workflow、tool、eval、cost、そして honestyを含めた信頼性** をセットで見ていく必要がある。

その意味で、Claude Opus 4.8は、今後のAgent設計を考えるうえで重要な節目になると思います。
