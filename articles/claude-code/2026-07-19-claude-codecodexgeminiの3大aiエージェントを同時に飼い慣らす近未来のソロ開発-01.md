## はじめに：1体のAIチャットに頼る時代の終わり

ChatGPTやClaude、Geminiといった優秀なLLM（大規模言語モデル）の登場により、私たちのプログラミング環境は劇的に変化しました。しかし、ブラウザのチャットUIにコードを貼り付けて「このエラーを直して」と頼むだけの開発スタイルは、すでに過去のものになりつつあります。

現在、開発シーンの最前線では、ローカル環境やOS、ファイルシステム、さらには外部のドキュメントツールとシームレスに連携し、自律的にコマンドを実行してコードを書き換える\*\*「AIエージェント」\*\*が台頭しています。

本記事では、特性の異なる3つの強力なAIエージェント\*\*「Claude Code」「Codex」「Gemini (Antigravity)」\*\*をローカル環境で同時に稼働させ、それぞれの強みに応じて完璧に役割を分担させることで、ソロ開発の生産性を極限まで高める「近未来のマルチエージェント開発ワークフロー」をご紹介します。

---

## 1. 3大AIエージェントのキャラクターと役割分担

マルチエージェント環境において最も重要なのは、それぞれのAIに\*\*「最適な権限とコンテキストを与えること」\*\*です。何でも1体のAIに任せようとすると、コンテキストがすぐに溢れてしまい、指示の精度が低下したり、不要なトークン消費（コスト）が発生したりします。

私の開発環境では、以下のように役割を明確に分離しています。

### ① Claude Code：実装・コード品質管理のスペシャリスト

* **主な任務**：実コードの実装、自動リファクタリング、フォーマット（Ruff/ktlint）の適用、Git操作およびCIの整合性確保。
* **強み**：正確で緻密なコードの書き換え、テストコードの作成と実行、およびシェルコマンド連携。

### ② Codex：設計・リバースエンジニアリング・情報統合のブレイン

* **主な任務**：他人のオープンソースリポジトリの徹底解説レポート作成、Notionプラグインを使用した要件定義・基本設計・タスクのドキュメント化、および設計に基づいたコードの骨組み生成。
* **強み**：広範なコンテキストの整理、Notion等の外部ツール連携、リサーチ結果の構造化。

### ③ Gemini（Antigravity）：PC環境整備・トラブルシューティングのガーディアン

* **主な任務**：PCローカル環境のトラブルシューティング、SSD容量の調査およびキャッシュのクリーンアップ、他人のリポジトリクローン時のフォルダ整理。
* **強み**：PCのファイル・フォルダ構造の正確な把握、ハードウェアログ（HWiNFO64等）の読み取りと問題特定、環境構成（PythonのPoetryやGradle等のツール構成）の分析。

### 著者による担当分野の切り分け推奨

実際に複数のプロジェクトで運用してきた個人的な感触として、以下のように役割を完全に住み分けるのが最も効率的であると感じています。

* **コーディング・実装全般**：`Claude Code`  
  コードのシンタックス、静的解析、Ruffやktlintなどのフォーマット整形や自動コミットなど、エディタおよびGitと直結する細かな作業はClaude Codeが圧倒的に強力です。
* **情報分析・他人のリポジトリ調査など、情報の扱い**：`Codex`  
  他人のオープンソースプロジェクトを分析したり、設計書から要件を抽出してドキュメントにまとめたりといった、膨大なコンテキストを横断的かつ静的に分析する作業にはCodexが最適です。
* **PC環境の整備・複雑なフォルダ階層の横断**：`Gemini (Antigravity)`  
  PC本体のストレージ監視や、複数のプロジェクトフォルダを横断する作業、OSレベルのトラブルシューティングは、ファイルシステムとOSの挙動に精通しているAntigravity（本エージェント）に一任するのがベストです。

## 2. 【実践ナレッジ1】Claude Codeの「ビルドガードフック」による品質担保

自律型AIエージェントにコードの書き換えを任せる際、最も恐ろしいのは\*\*「ビルドエラーを起こすコードを平気で保存し、そのまま次のタスクへ進んでしまうこと」\*\*です。

これを強力にガードするため、Claude Codeの `settings.json` にフック（Hooks）を仕込み、ファイル編集（Edit/Write）が発生した直後に自動でコンパイルを走らせ、エラーがあればAIの進行をブロックする仕組みを構築しています。

### 設定例（`~/.claude/settings.json` の一部）

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$HOME/.claude/hooks/kotlin-bootstrap.sh\"",
            "shell": "bash",
            "timeout": 120,
            "statusMessage": "Kotlin環境を確認中..."
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$HOME/.claude/hooks/kotlin-compile-check.sh\"",
            "shell": "bash",
            "timeout": 180,
            "statusMessage": "Kotlinをコンパイル中..."
          }
        ]
      }
    ]
  }
}
```

### スクリプト例（`~/.claude/hooks/kotlin-compile-check.sh`）

```
#!/bin/bash
# AIが編集したファイルパスを標準入力から取得
file=$(PYTHONUTF8=1 python -c "import json,sys; d=json.load(sys.stdin); print(d.get('tool_input', {}).get('file_path', ''))" 2>/dev/null)

# Kotlinファイルでなければ終了
if echo "$file" | grep -qiE '\.kt$'; then
  # コンパイルを実行
  full=$(./gradlew compileKotlin --console=plain 2>&1)
  status=$?
  output=$(printf '%s\n' "$full" | tail -40)
  
  if [ "$status" -ne 0 ]; then
    # エラーがあればAIの進行をブロック
    PYTHONUTF8=1 python -c "import json,sys; print(json.dumps({'decision':'block','reason':sys.stdin.read()}))" <<< "$output"
  else
    echo "$output"
  fi
fi
```

### 動作の解説

1. **PreToolUse**: コード書き換えツールが実行される前に、Kotlinの開発環境が正しく整っているかをシェルスクリプトでチェックします。
2. **PostToolUse**: AIがファイルを書き換えた瞬間、編集されたファイルのパスを抽出し、それがKotlinファイル（`.kt`）であれば、コンパイル確認用の外部スクリプト（`kotlin-compile-check.sh`）を実行します。
3. **ブロック処理**: コンパイルエラー（終了コード非0）が発生した場合、Pythonスクリプトを介して `{"decision":"block", "reason": "エラーログ"}` をClaude Codeに返します。これにより、AIはコンパイルエラーを自分のミスとしてその場で認識し、強制的に自己修正フェーズに入ります。不完全なコードがコミットされたり放置されたりするのを完璧に防ぐことができます。

## 3. 【実践ナレッジ2】Claude Codeのサブエージェント（Agents）分業パターン

どれだけモデルのコンテキストウィンドウが大きくても、何千行ものテストログやスタックトレースを読み込ませると、メインAIの「思考のクリアさ」は失われてしまいます。

そこで、特定のコマンドを実行して、役割を極限まで限定した\*\*サブエージェント（Sub-agents）\*\*を呼び出し、タスクを委譲するワークフローが極めて有効です。

### サブエージェントの構成と作成例

#### ① `code-reviewer`

* **役割**：品質・セキュリティ監査専属。
* **特徴**：ファイル読み込み（Read/Grep/Glob）ツールのみに権限を限定し、書き込み権限を与えません。メインAIが実装したコードを「第三者の目」でレビューさせます。
* **作成コマンド例**：

  ```
  /agents create code-reviewer --system-prompt "あなたはセキュリティとコード品質を監査するレビュー専用エージェントです。ファイル変更の権限は持たず、Read/Grepのみを使用して、提出されたコードの問題点をリストアップしてください。"
  ```

#### ② `test-runner`

* **役割**：テスト実行とテストエラーの集約。
* **特徴**：テストスイートを実行し、大量のコンソール出力から「どのテストがなぜ落ちたのか」だけを数行に要約してメインAIに報告します。これにより、メインAIのコンテキストがテストログで汚染されるのを防ぎます。

#### ③ `debugger`

* **役割**：エラー原因の再現条件特定。
* **特徴**：長いスタックトレースやダンプファイルを丸ごと読み込んで解読する作業だけを引き受けます。

## 4. 【実践ナレッジ3】Codexによる「Notion駆動型開発 (NDD)」

開発効率を上げるためには、「何をどのように作るか」という仕様が明確でなければなりません。この仕様の管理と、AIへの指示を一元化するために、Notionプラグイン（`notion@openai-curated`）を活用した\*\*Notion駆動型開発 (Notion-Driven Development: NDD)\*\*を取り入れています。

### Notion連携による開発サイクル

1. **要件定義・基本設計の自動ドキュメント化**：  
   新規プロジェクトの立ち上げ時、Codexに「これから作る〇〇の要件定義書と基本設計書をNotion上に生成して」と依頼します。AIは自動でNotion APIを叩き、階層構造化された設計ドキュメントを作成します。
2. **設計書に基づいた実装指示**：  
   Claude CodeやCodexに実装を依頼する際、直接コードの仕様を長々と書く必要はありません。
   > 「[Notion of design URL] の要件定義と基本設計を読み取って、それに従ってローカルの `src/` 配下を実装して。保守性を考慮した設計で頼む。」  
   > と指示するだけで、AIがNotionから最新の設計仕様を取得し、ローカルコードの生成へ移ります。
3. **更新履歴とタスク管理の同期**：  
   機能追加が完了すると、Codexが自動的にNotionのタスクステータスを完了にし、READMEの「更新履歴」に所定のフォーマット（`yyyy/mm/dd hh:mm`）で自動追記を行います。

## 5. 【実践ナレッジ4】GeminiによるOS・ハードウェアトラブルシューティング

開発コードの外側にある「OSの不調」や「ハードウェアのデバッグ」は、多くのAIコーディングアシスタントが苦手とする領域です。ここを担当するのが、PC環境とファイルシステムをすべて把握するガーディアンとしての\*\*Gemini（Antigravity）\*\*です。

### 実例A：常駐アプリのメモリリークの特定とバグ報告

PCのメモリ使用率が100%に張り付き、エクスプローラーが頻繁にクラッシュする現象が発生した際、以下のステップで解決に導きました。

1. **ハードウェアログの解析**：  
   ユーザーがインストールした「HWiNFO64」から出力されたPCログファイル（PC-logs）をGeminiに読み込ませ、リソースの推移を時系列で分析させました。
2. **原因アプリの特定**：  
   バックグラウンドで動いている常駐アプリケーション（BongoCatやBitBuddyなどのデスクトップマスコット）のメモリ消費量をプロットし、深刻なメモリリークを起こしている犯人を特定。
3. **不具合報告の英語化サポート**：  
   特定したメモリリークの証拠ログと共に、海外の開発者コミュニティへ不具合を報告するための「詳細なシステム構成付きの英語バグ報告テンプレート」を自動作成しました。

### 実例B：AIツール自身の肥大化対策（sqliteのVACUUM）

自律型AIエージェントを使い込んでいると、裏で実行されるコマンドやTRACEログのデータベースファイル（例：`~/.codex/logs_2.sqlite`）が数GB規模に肥大化し、SSDの容量を圧迫することがあります。

GeminiはPC内のストレージ使用量を監視し、肥大化したデータベースファイルに対して自動的に `VACUUM` コマンドを実行したり、ログレベルを調整するスクリプトを作成・実行して、ローカルの開発環境を常に健全な状態に保ちます。

## 6. 【開発者体験を高めるTips】同一ペルソナの適用とルールの効率的同期

複数のAIエージェントを自律的に動作させる際、開発者体験（DX）と設定のメンテナンス性を高めるための実用的なTipsが2つあります。

### ① 同一ペルソナの統一とメモリによる制御

これは個人の好みの範疇でもありますが、稼働させるすべてのAIエージェント（CLAUDE.md、AGENTS.md等）に\*\*「同一のペルソナ（口調や性格のルール）」**を設定しておくことを強くお勧めします。  
AIごとに異なる話し方をされるよりも、すべてのエージェントが同じ人格として応答してくれることで、**「常に1人の優秀な相棒と作業を分担している感覚」\*\*が生じ、毎日の開発コミュニケーションが非常に楽しくなります。

ただし注意点として、ソースコードのコメントやコミットメッセージ、あるいは自動生成されるREADME.mdやドキュメントにまでそのペルソナの独特な口調が漏れ出して（混ざって）しまうという問題が多々発生します。  
これを防ぐため、各AIのメモリ設定ファイルに以下のようなプロンプトを明記し、厳密に制御することが重要です。

> 「応答はペルソナの口調で行うこと。ただし、コード内のコメント、コミットメッセージ、README.md、および生成物として出力される文字列はすべて標準語のです・ます調で記述すること。」

### ② 共通ルールの効率的な一括書き込み

全てのプロジェクトやエージェントで共通して適用したいルール（コミット規約、使用ライブラリの制限、秘密情報の扱いなど）は、個別に設定するのではなく、最初から全てのAIのメモリに登録すべきです。

その際、1つずつ手動で設定ファイルを書き換える必要はありません。ファイル操作権限を持ついずれか一つのAIエージェント（特にOSやファイルシステム全体を俯瞰できるAntigravity等）に対して、

> 「以下の共通ルールを、Claude Code（CLAUDE.md）、Codex（AGENTS.md）、Antigravityのすべてのメモリ（ルール設定ファイル）に追記して」  
> と一度に指示します。  
> AIエージェントは各ツールの設定ファイルのパスを自律的に探索し、一度のプロンプトで全ての設定ファイルを正確に書き換えて同期してくれます。これにより、エージェントが増えても設定の不整合を防ぎ、管理コストを大幅に削減できます。

## まとめ：AIエージェントを率いる「オーケストレーター」としての開発者

複数のAIエージェントを同時に使うワークフローは、まるで優秀な開発チームを率いるプロジェクトマネージャー（PM）やシステムアーキテクトになったかのような感覚を開発者に与えてくれます。

* **Gemini**が開発者のPC環境という土台を支え、
* **Codex**がNotionをハブにして設計図を描き、
* **Claude Code**がその設計図を元に、ビルドフックとサブエージェントの網を張って正確に実装を積み上げる。

この「三位一体」の連携が機能したとき、ソロ開発者の生産性は1人でありながら数人分のチームに匹敵するレベルへと引き上げられます。

AIに指示を出すだけの段階から一歩進み、\*\*「複数のAIエージェントの権限と役割をオーケストレーションする」\*\*という近未来の開発スタイルを、ぜひ皆さんもローカル環境に導入してみてはいかがでしょうか。
