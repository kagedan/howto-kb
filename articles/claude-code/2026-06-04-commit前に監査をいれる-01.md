# AIエージェント時代のコミット前監査 ―「やったつもり」を機械的に検証する

## なぜコミット前監査が要るのか

AIコーディングエージェント（Claude Code のようなもの）にプロジェクトのルールを守らせたいとき、多くの人はまず `CLAUDE.md` / `AGENTS.md` のような**指示ファイル**にルールを書きます。「コードを変更したら必ずテストも書く」「秘密情報をハードコードしない」「例外を握り潰さない」……。

ところが、これだけでは足りません。理由は単純で、**指示ファイルはあくまで「お願い」であり、強制力がない**からです。エージェントは（人間と同じく）次のように振る舞うことがあります。

* 「テストも追加しました」と報告するが、実際には追加していない
* ルールを途中まで守り、長い作業の終盤で忘れる
* 握り潰し（`except: pass`）や暫定パッチを、動くからという理由で滑り込ませる

つまり問題は **「やった」という自己申告と、実際にやったことの乖離** にあります。そこで必要になるのが、コミットの直前に次の3つを**機械的に**検証する仕組みです。

1. **指示ファイルで定義した処理が、実際に行われているか**
2. **テストは存在するか**
3. **テストはクリアされているか**（そして静的に明らかな地雷を踏んでいないか）

本記事では、この「コミット前監査」を**二層アーキテクチャ**として実装したパターンと、実際に走らせている監査ルールの全量を紹介します。特定プロジェクトに依存しない、汎用的に持ち込める設計として書きます。

---

## 二層アーキテクチャ ― プレイブックと強制フック

監査を「エージェント自身に回させる」だけでは、結局それも自己申告になってしまいます。逆に「機械チェックだけ」にすると、設計レベルの問題（重複ロジック、暫定パッチ）を拾えません。

そこで役割を2層に分けます。

| 層 | 実体 | 担い手 | 役割 | 失敗時 |
| --- | --- | --- | --- | --- |
| **層1: プレイブック** | スラッシュコマンド（例 `/precommit`） | エージェント | テスト確認＋コード監査＋客観レビューの**手順を回す** | 指摘がゼロになるまで修正ループ |
| **層2: 強制フック** | シェル/PowerShell スクリプト | ランタイム（Stop hook / pre-commit framework） | 静的な red-flag を**機械検出してコミットを止める** | exit 非ゼロでブロック |

ポイントは、**層2が層1の実施を検証する**ことです。後述の「diff ハッシュ契約」によって、エージェントが本当にプレイブックを完走したかをスクリプト側が確認します。これにより「監査したつもり」も潰せます。

```
 エージェントがコードを書く
        │
        ▼
 ┌─────────────────────────┐
 │ 層1: /precommit          │  ← エージェントが自分で回す
 │  テスト追加 / コード監査 │
 │  / subagent レビュー      │
 └─────────────────────────┘
        │ 完了の証跡 (diff hash + verdict) を書き出す
        ▼
 ┌─────────────────────────┐
 │ 層2: pre-commit フック    │  ← ランタイムが自動で発火
 │  静的検出 + 証跡の検証    │
 └─────────────────────────┘
        │ 指摘ゼロ            │ 指摘あり
        ▼                     ▼
      コミット成立          コミットをブロック
```

---

## 層1：プレイブック（エージェントが回す監査手順）

プレイブックは、エージェントが「順次 → 並列 → 順次」の3フェーズで実行します。要点だけ抜き出すと次の通りです。

### Phase A（順次）：対象の確定

* 変更ファイルを列挙し、**監査対象が1件もなければ早期終了**します
* 対象：ソース拡張子（`.py / .ts / .tsx / .js / .jsx / .sh / .ps1 / .yml / .toml` …）
* 除外：生成物（`node_modules/`, `dist/`, `__pycache__/`）、純粋なドキュメント

### Phase B（並列）：テスト確認・静的監査・客観レビュー

この3つを**1メッセージで同時に発火**してパイプライン化するのがコツです。静的検査（grep / diff）は数秒で終わりますが、subagent レビューはLLM呼び出しで数十秒かかります。直列に並べると待ち時間が積み上がるため、並列にして「最も遅い1本（＝レビュー）」に実時間を寄せます。

1. **テストの存在確認と追加**
   * 変更したソースに対応するテストがあるか確認します（命名揺れ `test_foo` / `foo_test` / `*.test.ts` を吸収）
   * 無ければ最低限の smoke test を追加します（import が通る / 主要関数が期待値を返す / 異常系で例外を投げる）
   * インフラ依存等で書けない場合は、理由付きの `TODO: test (reason: ...)` を残してコミットメッセージにも明記します
2. **静的なコード監査**（後述の red-flag パターン）
3. **subagent による客観レビュー**
   * 静的検査では拾えない設計レベルの問題（暫定パッチ、重複ロジック、テスト網羅の穴）を、別エージェントに diff を読ませて指摘させます
   * レビュー用エージェントが使えない環境では**スキップし、「人手レビューが必要」と明示してコミットはブロックしません**（graceful degradation）

### Phase C（順次）：統合・テスト実行・報告

4. Phase B の指摘を統合 → 修正 → 再監査を **指摘がゼロになるまで反復**します
5. **テストを実行**します（プロジェクト標準の仮想環境・ランナーで揃える。ローカルのバラついた処理系を避ける）
6. 監査結果をまとめてコミット可否を判断・報告します

報告フォーマット例：

```
## 監査結果
- 変更ファイル: N件
- test追加: N件
- 監査指摘と対応:
  - fall-through: <件数, 対応>
  - hard-code:    <件数, 対応>
  - 暫定パッチ:   <件数, 対応>
  - 共通コード再利用: <件数, 対応>
- smoke test: PASS / FAIL
## コミット可否: 可 / 不可（不可なら理由）
```

完走すると、エージェントは `last-review.json` のような証跡ファイルに **「現在の diff のハッシュ」と「verdict: pass」** を書き出します。これが層2の検証材料になります。

## 層2：強制フック（コミットをブロックする機械検査）

層2は、ランタイムが自動で発火するスクリプトです。2つのモードで同じロジックを共有します。

* **stop モード**：エージェントが作業を終えて自動コミットしようとする瞬間に発火します。証跡（subagent レビュー）の検証も行います。指摘を見つけたら exit 2 で**エージェントに差し戻します**（指摘内容がそのままエージェントへのフィードバックになります）
* **precommit モード**：人間が `git commit` したときに pre-commit framework 経由で発火します。ループ内にエージェントが居ないので証跡検証はスキップし、exit 1 でコミットを止めます

以下、実際に走らせている検査の全量を挙げます。**すべて「diff の追加行（`+` 行）」だけを対象にする**のがコツで、既存の負債を一括で蒸し返さず、今回の変更分だけを審査します。

### Check 1：ソース変更にテストが伴っているか

最重要かつ最も単純です。**ソースファイルが変更されているのにテストファイルの変更・追加が1件も無ければブロック**します。「テストを書いたつもり」を物理的に検出する forcing function です。

```
ソース変更あり AND テスト変更ゼロ  →  "TEST MISSING" でブロック
```

### Check 2：red-flag パターンの静的検出

追加行に対して正規表現で地雷を探します。

#### 2a. ハードコードされたIPアドレス

ドット区切り4オクテットの数値列を検出します。ただし誤検出を避けるため、明らかに無害な値は除外します。

* `127.0.0.1`, `0.0.0.0`, `255.255.255.x`, パブリックDNS（`8.8.8.8`, `1.1.1.1`）
* **RFC 5737** のドキュメント用予約レンジ（`192.0.2.0/24`, `198.51.100.0/24`, `203.0.113.0/24`）→ テスト/モックで安全に使えます
* コメント行（`#`, `//`, `--`, `*` で始まる行）も除外します

検出されたら「設定ファイル経由（環境変数）にせよ」と促します。

#### 2b. ハードコードされた秘密情報

`password` / `passwd` / `secret` / `api_key` / `token` / `private_key` などへの**文字列リテラル代入**を、大文字小文字を無視して検出します。ただし環境変数取得（`os.environ`, `getenv`, `process.env`）やプレースホルダ（`${...}`, `<REDACTED>`, `EXAMPLE`, `CHANGEME`）は許容します。

#### 2c. 例外の握り潰し / fall-through

`except: pass`、空の `except Exception:`、`|| true` での失敗握り潰しを検出します。意図的なら理由付きの抑制マーカーを要求します。

#### 2d. 設定の出どころ逸脱

プロジェクトで**正典と決めた設定ファイル以外**から環境変数を読むコード（`load_dotenv`, `EnvironmentFile`, `source xxx.env` 等）を検出します。設定の集中管理を崩す変更を早期に止めます。

#### 2e. 関数定義の重複（共通コード再利用の促し）

新規追加された関数定義名を拾い、**リポジトリの他ファイルに同名定義があれば**「既存実装を流用できないか」と促します。ただし誤検出が多いので、**汎用的に各モジュールで再定義されがちな名前は allowlist** で除外します。

* ライフサイクル：`__init__`, `setUp`, `main`, `init`, `setup`, `start`, `stop`, `run`, `close`, `connect` …
* CRUD系：`get`, `post`, `put`, `delete`, `list`, `create`, `update`, `load`, `save` …
* 汎用ヘルパ：`handle`, `process`, `validate`, `parse`, `format`, `encode`, `decode`, `serialize` …

（この allowlist は運用で育てます。実際、監査ログを分析したら重複検出の2割強がこれら汎用名の誤検出だった、という調整履歴があります。）

#### 2f. `set -e` スクリプトの `local VAR=$(...)` 罠

これは bash の有名な落とし穴です。`local` / `declare` / `readonly` / `export` は**ビルトイン**なので、右辺のコマンド置換が失敗しても**ビルトイン自身の終了ステータスは0**になり、`set -e` が発火しません。結果、ゴミの入った変数で処理が続き、数行後の見当違いな場所で死にます。

```
# BAD ― set -e が効かない（silent failure）
local x=$(some_cmd)

# GOOD ― 代入の rc が set -e に届く
local x
x=$(some_cmd)
```

`set -e`（および `set -eu` / `set -euo pipefail` / `set -o errexit`）を使っているスクリプトに限って、宣言と代入が一体化したコマンド置換を検出します。

#### 2g. プロジェクト固有のアーキテクチャ制約

「このディレクトリのコードはこのAPIだけを呼んでよい」といった**プロジェクト固有の構造ルール**も、同じ追加行スキャンの仕組みで強制できます。アーキテクチャ憲章を人間の記憶ではなくフックに焼き込むことで、移行期のなし崩しを防ぎます。

### Check 3：構文・lint・型・複雑度

「テストはクリアされているか」の前段として、**そもそもコードが壊れていないか**を軽量にチェックします。各ツールは**存在すれば走り、無ければ黙ってスキップ**します（環境差で監査が暴発しません）。

| 対象 | チェック |
| --- | --- |
| Python | `py_compile`（構文）、`ruff`（lint） |
| シェル | `bash -n`（構文） |
| JS/TS | `eslint`（該当プロジェクト配下のみ、ローカルインストール時のみ） |
| TypeScript | `tsc --noEmit --incremental`（プロジェクトごとに1回、ビルドキャッシュ利用、25秒で打ち切り） |
| Markdown | リンク切れ検出（リポジトリ相対パスのリンク先が解決するか。外部URL/アンカー/mailto は除外） |
| Python | `radon` 循環的複雑度（rank D以下＝McCabe > 20 をフラグ。AI生成コードは過剰ネストになりがち） |
| 多言語 | `semgrep`（SQLi/XSS/path traversal 等、正規表現では拾えない意味解析。バイナリがあれば） |
| 全ファイル | `gitleaks`（2b の正規表現が漏らす秘密情報。バイナリがあれば） |

### Check 3g：改行コード

`.ps1` / `.cmd` / `.bat` は CRLF、`.sh` は LF を強制します（LF混入したシェバング行で死ぬ事故を防ぎます）。`.gitattributes` でチェックアウト時に矯正していても、エディタやエージェントの書き込みで混入しうるので二重で守ります。

### Check 4：subagent レビュー契約（層1の実施を検証）

ここが**二層を結ぶ要**です。stop モードでソース変更がある場合、証跡ファイルを検証します。

```
last-review.json の diff_hash == 現在の diff から計算したハッシュ
AND  verdict == "pass"
```

一致しなければ「**プレイブックを走らせ、レビューを完走してから証跡を更新せよ**」とブロックします。これにより「監査したつもりで実はしていない」を機械的に潰します。

> 実装上の地雷：ハッシュ計算は**LF区切りで統一**しないと、OSやロケール（日本語環境のCP932等）で文字化けが起き、自分で書いた証跡を「古い」と誤判定します。フック全体で標準出力を UTF-8 にそろえる必要があります。

### Check 5：ルールドキュメントの drift（警告）

ルールの「物語版ドキュメント」と「機械可読なポリシー登録簿」が食い違ったら警告します（ブロックではありません）。ルールを片方だけ直す事故を検出します。

### Check 6：ソースファイルの肥大化（警告のみ）

変更/追加されたソースが一定行数（例：500行）を超えたら**警告だけ**出します。AI支援編集でファイルが膨らみがちなのを早期に気づかせます。ただし既存の巨大ファイルが大量にあるとブロックは過剰なので、**終了コードは変えません**（助言に留めます）。行数カウントは1行ずつストリーミングし、巨大ファイルを丸ごとメモリに載せません。

## 抑制機構 ― 「理由必須」のエスケープハッチ

監査は完璧ではありません。誤検出も、意図的にルールを外れる正当なケースもあります。そこで抑制マーカーを用意しますが、**理由のないマーカーは無効**にするのが肝です。

* **行単位**：問題行に `# audit-ignore: <理由>`（JS/TSは `//`）を付けます
* **ファイル単位**：先頭20行以内に `audit-ignore-file: <理由>` を書きます（赤旗スキャンをファイルごと無効化。テストフィクスチャ等に限定して使います）
* **複雑度**：`# audit-ignore: <理由>` を関数定義行か直前行に置きます
* **一度きりの緊急バイパス**：`.audit-skip` ファイルを置きます。**読まれた瞬間に削除される**ので、バイパスが居座りません

理由を必須にすることで、「とりあえず黙らせた」を防ぎ、後からレビューできる痕跡を残します。

## 監査ログ ― 効きめを測る

毎回の監査結果を JSON Lines で追記します。

```
{"ts":"...","event":"pre-commit-audit","mode":"stop","verdict":"block","issueCount":3,"issueCategories":["TEST MISSING","HARD-CODED IP"],"warningCount":1,"changedCount":7}
```

これがあると、**どのルールが何回発火し、どれが誤検出だったか**を後から分析できます。前述の「重複検出の2割が誤検出だったので allowlist を拡張した」という調整は、このログがあってこそです。監査ルールは作って終わりではなく、ログで効きめを測りながら閾値と allowlist を育てるものです。

---

## 設計上の細かい工夫

実運用で効いてくる、地味だが重要なポイントです。

* **再帰防止**：Stop hook が自分の発火でループしないよう、フラグで二度目を抑止します
* **並列発火**：層1の Phase B は静的検査と subagent レビューを1メッセージで同時発火し、実時間を最遅の1本に寄せます
* **graceful degradation**：lint/型/semgrep/gitleaks 等の外部ツールは**無ければ黙ってスキップ**します。環境差で監査自体が壊れません
* **追加行のみ審査**：`+` 行だけを見て、既存負債を蒸し返しません
* **文字コード統一**：ハッシュ計算・正規表現一致が文字コードでブレないようにします
* **スキップ条件**：ドキュメントのみ・設定ディレクトリのみの変更は監査を省略します

## まとめ ― ルールは「書く」だけでなく「強制する」

AIエージェントにルールを守らせる鍵は、**指示ファイルに書くこと**ではなく、**その遵守を機械的に検証してコミットを止めること**にあります。本記事のパターンを一言でまとめると、次のようになります。

> 「テストを添える＋複数観点のコード監査＋客観レビュー」を**エージェント自身に回させ（層1）**、それを**静的検査と diff ハッシュ契約で機械的に強制する（層2）**。

検証している問いは終始シンプルです。

1. **指示通りの処理が実際に行われたか** → subagent レビュー契約（diff ハッシュ）
2. **テストは存在するか** → Check 1（ソース変更にテストが伴うか）
3. **テストはクリアされているか** → 構文・lint・型・複雑度・テスト実行

そして、誤検出には**理由必須の抑制マーカー**で逃げ道を作り、効きめは**監査ログ**で測って育てます。

「やったつもり」と「実際にやった」の差を埋める forcing function ―― これが、AIエージェントと共同開発する時代のコミット前監査の本質です。
