ここ最近、[Claude Code](https://claude.com/product/claude-code)や[Codex](https://openai.com/codex/)、あるいは[Cursor](https://cursor.com/)のComposerなどの強力なコーディングエージェント（Coding Agents）を日常のワークフローに組み込む開発者が急増している。LLMの推論能力向上とコンテキストウィンドウの拡大により、彼らは単なるオートコンプリートの枠を超え、自律的にリポジトリを読み、計画を立て、コードを修正できるようになった。

しかし、これらのツールを数週間、数ヶ月と長期間使い続けていると、ある深刻なフラストレーションに直面することになる。

「なぜ、昨日決めたアーキテクチャの前提を今日には忘れているのか？」  
「なぜ、このプロジェクト特有のコーディング規約を毎回プロンプトで教え直さなければならないのか？」

多くの開発者は、この問題を解決するためにRAG（Retrieval-Augmented Generation）を導入したり、ベクターストアにドキュメントを突っ込んだりする。しかし、RAGを入れてもAgentは依然として「本当に大事なこと」を忘れる。

本記事では、長期間稼働するAgentシステムにおいて、なぜRAGだけでは不十分なのかを紐解き、LLMシステムにおける「Memoryアーキテクチャ」の設計要件と、その自然な解の一つとして注目されるインフラストラクチャ「MemoryLake」のアプローチについて考察する。

---

## 優秀だが毎日“記憶喪失”になる同僚：長期間利用で見えてくるペイン

Claude Codeのようなエージェントを実際のプロダクト開発で長期間使っていると、「モデルが賢くない」ことよりも「記憶の連続性（Continuity）がない」ことへのペインが顕在化してくる。

具体的な開発シーンを想像してほしい。  
昨日、あなたはAgentと共に複雑なバグを修正した。その過程で「このコンポーネントでは特定のライブラリの旧APIは使わず、新しいラッパー関数を経由する」というプロジェクト内の暗黙のルールに気づき、Agentと合意したとしよう。

しかし翌日、別のタスクで新しいセッション（ターミナルやチャットウィンドウ）を立ち上げると、Agentは再び旧APIを使ってコードを書き始める。セッションが切れた瞬間、Agentの記憶はリセットされ、また「初日に入社してきた新入社員」に戻ってしまうのだ。

これを防ぐために、開発者は以下のようなワークアラウンドを強いられる。

* **Context Stuffing（コンテキストの詰め込み）**: リポジトリの背景、アーキテクチャの概要、チームの規約などをまとめた巨大な `.cursorrules` や `prompt.txt` を用意し、毎回LLMに読み込ませる。
* **手動での引き継ぎ**: 「前回はこういうアプローチを取ったので、今日もそれを踏襲して」と、人間がAgentの記憶を代行してあげる。

時間が経てば経つほど、コンテキストウィンドウに詰め込むべきルールや経緯は肥大化する。トークンコストは無駄に跳ね上がり、何より「どのコンテキストを今日のAgentに渡すべきか」を人間が管理する認知負荷が限界を迎える。

## なぜRAGを実装してもAgentはすぐ忘れるのか？

「記憶の問題なら、RAG（ベクトル検索）を実装すれば解決するのでは？」と考えるかもしれない。実際、現在の多くのLLMアプリケーションは、過去のチャット履歴やドキュメントをチャンク化し、Vector DBに保存して類似度検索で引っ張ってくる構成をとっている。

しかし、**RAGはRetrieval（検索・召喚）のメカニズムであって、Memory（記憶）のアーキテクチャではない**。

ここには決定的な役割の違いがある。

### 1. RAGは「状態の更新と衝突」を解決できない

RAGが得意なのは、静的なドキュメントから事実の断片（Snippet）を見つけ出すことだ。しかし、Agentの記憶は動的である。「先週まではAという設計方針だったが、昨日Bに変更した」という場合、単純なベクトル検索ではAとBの両方がヒットしてしまい、Agentはどちらが現在の正なのか判断できない（あるいは古い情報を優先してハルシネーションを起こす）。

### 2. 「見つけられること」と「連続性を持っていること」は違う

RAGは、ユーザーのプロンプトに似た過去の文脈の「破片」をプロンプトに差し込むだけだ。しかし、真の「記憶」とは、過去の対話や決定事項から抽象的なルールを学習し、現在のタスクにバックグラウンドとして適応させる能力を指す。RAGは必要な断片を見つけることはできても、それをどうやって持続的な「経験」としてAgentの行動原理に組み込むかは定義していない。

### 3. スキルやワークフローの記憶が抜け落ちる

開発における記憶とは、単なるテキストのナレッジベースではない。「このエラーが出た時は、このログを見て、このスクリプトを叩く」という一連の手順（Skill / Workflow memory）も記憶の一部だ。単なるベクトル検索では、こうした「行動の記憶」を適切にロードすることは難しい。

## 「記憶」をRetrievalではなくArchitectureとして考える

Agentが真の意味で「記憶」を持ち、長期的なパートナーとして機能するためには、チャット履歴をDBに保存したりRAGを導入したりするだけでは不十分だ。LLMシステムは、より包括的な「Memory Architecture」を必要としている。

システム設計の観点から見ると、Agentの記憶は以下のような階層（Layer）として分離・整理されるべきである。

1. **Context Window (Working Memory)**: 現在のタスクを実行するための短期記憶。LLMのコンテキストウィンドウの限界（例えば200Kトークンなど）に依存する。
2. **Retrieval / RAG Layer**: 外部の静的ドキュメントやコードベースから、必要な事実をオンデマンドで引き出すためのメカニズム。
3. **Long-term Memory Layer**: エージェント自身が対話を通じて得た経験、ユーザーの好み、プロジェクトの決定事項などを抽出し、構造化して保存する層。
4. **Memory Governance & Versioning**: 記憶の衝突を解決し、古い記憶を非推奨にし、記憶の出所（Provenance）を追跡する管理層。
5. **Cross-Session / Cross-Agent Continuity**: セッションが途切れたり、背後で動くモデル（ClaudeからGPTへ等）が変わったり、あるいは別の特化型Agentにタスクを引き継いだりする際にも、コンテキストを透過的に持ち運べる仕組み。

このように考えると、「記憶」はLLMモデルそのものに依存するものでも、単一のVector DBに押し込めるものでもなく、**独立した永続的なインフラストラクチャ**として設計されるべきであることがわかる。

## アーキテクチャ上の自然な解としての「MemoryLake」

こうした「Agentの長期記憶問題」に対するシステムレベルでの解答として注目されるのが、\*\*[Memorylake](https://www.memorylake.ai/)\*\*のようなMemory Infrastructure（記憶のインフラストラクチャ）のアプローチである。

MemoryLakeを、「ただのRAGツール」や「別のVector DB」として捉えると本質を見誤る。これは、AgentやLLMシステムのための**Persistent（永続的）でPortable（持ち運び可能）な、Governed（ガバナンスの効いた）Memory Layer**である。

アーキテクチャの観点から、なぜMemoryLakeのようなレイヤーが必要になるのか、その位置づけを整理してみよう。

### AIのための「Second Brain」としての独立性

MemoryLakeは、特定のLLM（例えばAnthropicのAPIやOpenAIのAPI）や特定のセッションに依存しない。Agentがタスクを実行する過程で得た知識、ユーザーの意図、コードの背景といったコンテキストを抽象化し、継続的に蓄積・更新していく。  
これにより、今日のClaude Codeでのセッションの続きを、明日の別のワークフローAgentに引き継ぐといった「クロスエージェントでのコンテキスト共有」が可能になる。いわば、Agentが持ち歩ける**Memory Passport**として機能する。

### Memory Governance と Versioning

RAG最大の弱点であった「記憶の衝突」に対して、MemoryLakeは記憶のガバナンスを提供する。単にベクトルを突っ込むのではなく、「いつ、どのセッションで、何の目的で抽出された記憶か（Provenance）」を追跡し、状態の更新（Update/Deprecate）を管理する。これにより、Agentは「過去の古いルール」に引きずられることなく、常に最新の合意事項に基づいた行動がとれるようになる。

### 毎回コンテキストを再構築するコストの削減

MemoryLakeがシステムに組み込まれることで、開発者は毎回長大な `.cursorrules` を貼り付けたり、過去の経緯をプロンプトで説明したりする作業から解放される。Agentはセッション開始時に自身のLong-term Memoryにアクセスし、現在のユーザーやリポジトリに関する重要なコンテキストを自動的にロード（あるいはバックグラウンドで同期）した状態で起動できる。

## まとめ：長期運用を前提にするなら、Memoryは独立した設計対象になる

単発のタスクや短いスクリプトを書かせるだけの「DemoレベルのAI」であれば、RAGやシンプルなプロンプトエンジニアリングで十分かもしれない。

しかし、Claude CodeやCodexを本格的な開発現場に投入し、長期的に運用しようとした瞬間に、私たちは「記憶の断絶」という大きな壁にぶつかる。モデルがどれだけ賢くなっても、昨日までの背景（コンテキスト）を持たないAgentは、自律的に深いタスクをこなすことはできない。

* RAGは強力な検索エンジンだが、Memoryそのものではない。
* 「覚えている」とは、単にテキストを検索できることではなく、状態の更新と継続性を持って知識を再利用できることである。

もしあなたが、Agentを活用した自社システムのアーキテクチャを設計していたり、現在のコーディングエージェントの「記憶喪失」に限界を感じているなら、RAGの延長線上で解決しようとするのを一度立ち止まってみてほしい。

LLMを推論エンジンとして扱うのであれば、それに伴う「記憶」もまた、[Memorylake](https://www.memorylake.ai/)のような独立したInfrastructure層（AIのためのSecond Brain）として切り出し、設計・評価していくフェーズに入っているのだ。
