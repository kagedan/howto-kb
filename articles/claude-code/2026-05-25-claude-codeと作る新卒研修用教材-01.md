## はじめに

こんにちは、セーフィーのたかぎです。  
新卒エンジニア向けの研修プログラムをより充実させたいと思いつつも、新しい講座を通常業務の傍らで一から用意するのはハードルが高いものです。  
今回、Claude Codeを活用することで、これまで手が出せなかった新しい教材の作成に踏み切ることができました。

この記事では、Claude Codeを使って新卒研修の教材を実際に作った過程と、そこで得た知見を共有します。

## なぜ今回できたのか

結論から言うと、**生成AIの性能向上により教材のハードルが大幅に下がった**からです。

単純に教材を1から作っていくとなると記述量が膨大になります。  
教材を自前で用意したいなと思っていましたが、本業の合間にこれをやるのは現実的ではなく、先送りにしていたのが正直なところです。

Claude Codeを使えば膨大な文章を一瞬で生成してくれるため、記述が大変という問題が解消されました。

## 作った教材

今回作成したのは以下の3テーマです。それぞれ**座学パート**と**ハンズオンパート**で構成しています。

### リーダブルコード

この講座は『リーダブルコード ―より良いコードを書くためのシンプルで実践的なテクニック』（Dustin Boswell、Trevor Foucher 著／角 征典 訳、オライリー・ジャパン）の内容をベースに演習を作成しました。

<https://www.oreilly.co.jp/books/9784873115658/>

書籍の内容をそのまま教材に盛り込むと著作権に触れる懸念があるため、説明は書籍に委ねてほぼ演習という構成にしてあります。

書籍の内容に沿った演習問題を用意しており、「読みにくいコードを読む → 自分で改善する → チームでディスカッション」というサイクルで進めます。  
中間と最後に総合演習を1問ずつ設けて、学んだ内容を横断的に実践できるようにしています。

ちなみにセーフィーはPythonで作っているプロダクトが多いので演習はPythonで作っています。

```
# 【やること】
# どのような名前にすれば意図が伝わるか考えてみましょう。

def calc(data):
    """注文データから合計金額を計算する"""
    tmp = 0
    for d in data:
        tmp += d['price'] * d['qty']
    return tmp
```

### Git

Git講座では、バージョン管理がなぜ必要なのか？からはじまり、コマンド操作、ブランチ運用、チームでのGit運用とステップアップしていく形にしました。

Git操作そのものに集中できるように最終章以外は自己紹介ファイルの作成をテーマにしてあります。

### Docker

Docker講座では掲示板アプリ（React + Python + MySQL）を題材に、Docker Composeを使った開発環境の構築をハンズオンで学ぶ構成にしました。

フロントエンド・バックエンド・データベースの3コンテナで構成される開発環境を自分で構築する流れで進めます。  
手を動かしながらDockerfileの書き方やDocker Composeの設定を理解できるようにしています。

以下のような穴埋め形式になっています。

```
# 1. ベースイメージを指定
#    考え方: 必要なツールが揃っていて、かつ軽量なイメージを選ぶ
#    判断基準:
#      - 言語のバージョンは固定する（再現性のため）
#      - slim/alpine は軽量だがビルド時にトラブルが起きやすい
#      - 公式イメージを優先する（セキュリティ・メンテナンス）
#      - 必要なツールが最初から入っているイメージがあればそれを使う
FROM _____

# 2. 作業ディレクトリを設定
#    考え方: コンテナ内のどこで作業するかを決める
#    判断基準:
#      - /app, /src など分かりやすい名前を使う
#      - ルート直下は避ける（既存ディレクトリと衝突の可能性）
WORKDIR _____

# 3. 依存関係ファイルをコピー
#    考え方: 変更頻度の低いものから順にコピーする（レイヤーキャッシュ活用）
#    判断基準:
#      - 依存関係 → 変更頻度低い → イメージにベイク
#      - ソースコード → 変更頻度高い → ボリュームマウント
#      - この順序でCOPYすることでビルド時間を短縮できる
COPY _____ .
```

## Claude Codeをどう使ったか

### 構成を決める

最初にやったのは、Claude Codeと対話しながら教材の構成を決めることです。

「新卒向けのGit研修を作りたい。座学とハンズオンで構成したい」といった大枠を伝え、章立てや各セクションで扱うトピックをやりとりの中で詰めていきました。一人で白紙から考えるよりも、壁打ち相手がいることで構成が素早く固まります。

### 説明・コード生成する

構成が決まったら、各セクションのテキストやハンズオン用のコード・手順書をClaude Codeに生成させました。

ここがもっとも工数削減に効いた部分です。たたき台を人間がゼロから書くのと、生成されたものをレビュー・修正するのとでは、かかる時間がまったく違います。

### レビュー

作成されたものはClaude Codeにもレビューしてもらいました。わかりにくいものは表現を変えたり、もっと簡潔にできないか検討してもらいました。  
修正はClaude Codeを使うことが多かったですが、簡単なものは直接修正しました。  
最終的な仕上がりは人の目で確認していたので、品質を完全に任せるというのは今回やりませんでした。

## うまくいかなかったこと・注意点

便利な一方で、AI任せにできない部分や注意が必要な点もありました。

### 概念の説明は苦手

「Gitとは何か」「なぜDockerを使うのか」といった基礎の説明は、Claude Codeに任せるだけでは生成されず、そもそも含まれないことがほとんどでした。  
明示的に「Dockerについて初心者向けに解説して」と説明を求めても、How寄りの内容になりがちで、基礎部分はこちらで組み立てて指示する必要がありました。

### 情報量が多い

Claude Codeは一つの説明に大量の情報を詰め込んでくる傾向がありました。  
しかし、情報量が多ければいいわけではなく、むしろインプットの負荷が上がって理解の妨げになります。  
研修教材では「今の段階で必要なこと」だけを教えることが重要だと考えています。  
そのため、生成された内容から本質的な情報とそれに必要な具体例だけを残し、それ以外を削ぎ落とす編集作業が必要でした。

### 情報が古い場合がある

Claude Codeは学習データの範囲を超える情報を持たないため、出力が最新のベストプラクティスを反映していない可能性があります。  
単に「最新の内容に更新して」と指示すると、バージョン番号を調べて差し替える程度にとどまることが多く、ベストプラクティスそのものの変化には対応しきれませんでした。  
そのため、自分で現時点のベストプラクティスを調べたうえで、それを根拠として渡し「これをもとに作って」と依頼するやり方が有効でした。

## まとめ

大量の情報を整理してまとめる作業をClaude Codeに任せられたのは大きな利点でした。  
ただし、どの情報を使い、どう構成するかは人間が組み立てる必要がありました。  
一方で、その組み立てや「何を教えればいいか」自体をClaude Codeと相談しながら進められるため、試行錯誤の時間は大幅に短縮できました。  
何より、工数的にあきらめていたことに取り組めるようになったのが一番大きな変化でした。
