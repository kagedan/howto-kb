24時間動き続けるAIエージェントの設計で、一番の敵は「賢さ不足」ではありません。敵は記憶喪失です。

長時間タスクをやらせると、エージェントは数時間ごとにコンテキストウィンドウを使い切ります。新しいセッションは、前の自分が何をしたか一切覚えていません。毎朝、記憶を消されて出社する新人を想像してください。能力は高いのに、昨日の続きができない。これが長時間実行エージェントの正体です。

Anthropicはこの問題に正面から向き合い、2本の公式ガイドで設計思想を公開しました。私はこの2本を読み込み、自分でも長時間タスクを回してみました。この記事では、そこで見えた「ハーネス設計の急所」を実装思想のレベルまで掘り下げます。単なる解説では終わらせません。

なお本記事は、各社のハーネス設計を横断比較した既存記事とは別の角度です。今回はAnthropic単独の長時間実行設計だけを詳しく見ます。

## ハーネスとは何か:モデルとシステムの境界線

最初に言葉を揃えます。ハーネスとは、モデルの周りを取り囲む足場のことです。

モデル本体は推論するだけです。ファイルを読む、テストを走らせる、進捗を記録する、コンテキストを管理する。こうした作業はすべてモデルの外側、つまりハーネスが担います。馬具という語源どおり、暴れがちな能力に手綱をつける仕組みだと考えると分かりやすいです。

Anthropicの設計思想で一番好きなのが、次の一文です。

> ハーネスのあらゆる構成要素は、モデルが単独ではできないことについての前提を符号化している。

引用元は2026年3月のガイド「Harness design for long-running application development」です。つまりハーネスとは **モデルの弱点リスト** を、コードの形で書き出したものなのです。逆に言えば、モデルが賢くなるたびに前提は陳腐化します。この視点が後半の伏線になります。

## なぜコンパクションだけでは足りないのか

長時間タスクのコンテキスト管理には、大きく2つの手があります。

1つはコンパクション。会話履歴を要約して圧縮し、ウィンドウに収め続ける方法です。もう1つはコンテキストリセット。ウィンドウを丸ごと空にして、構造化された引き継ぎだけを次のセッションに渡す方法です。

直感的にはコンパクションの方が楽に見えます。要約して続ければいいのですから。ところがAnthropicの検証では、Claude Sonnet 4.5に対してコンパクションだけでは不十分でした。リセットが不可欠だったのです。

その理由が、このガイドで最も独自性のある概念につながります。

## Context Anxiety:モデルが自分の残量に怯える

Context Anxiety、私は「コンテキスト不安」と訳しています。

これはコンテキストウィンドウが埋まってくると、モデルの出力品質が落ちる現象です。人間に例えるなら、月曜朝に未読メールが300件たまっている、あの感覚に近いです。情報が溢れると、人は判断力が鈍ります。モデルも同じでした。

さらに厄介なことがあります。Claude Sonnet 4.5は、自分のコンテキストウィンドウを意識できる初めてのモデルだと報告されています。Cognitionが自社エージェントを4.5向けに再設計したとき、この性質に気づきました。モデルは残量が少ないと感じると、タスクを早めに切り上げようとするのです。

しかも自己評価が当てになりません。残りトークンを過小に見積もり、見積もりは妙に精密なのに値は間違っている。締め切りが近いと焦って手を抜く新人と、構図がそっくりです。

ここで前半の伏線が効いてきます。コンテキストリセットというハーネス部品は、4.5の不安という前提を符号化したものでした。モデルが落ち着けば、その部品は要らなくなります。

## claude-progress.txt:現在地を渡す引き継ぎノート

リセットを選ぶと、新しい問題が生まれます。記憶を消した次のセッションに、どうやって状況を引き継ぐのか。

Anthropicの答えが `claude-progress.txt` というファイルです。発想の元は、優秀なソフトウェアエンジニアが毎日やっていることでした。

役割分担はシンプルです。

* **git history** は「何が変わったか」のログです
* **claude-progress.txt** は「今どこにいるか」の現在地です

この2つをセットで渡すと、新しいセッションは即座に状況を把握できます。私たちが朝、コミット履歴をざっと眺めてから作業メモを開き、昨日の自分が残した付箋を読むのと同じ動きです。

擬似コードで書くと、各セッションのループはこうなります。

```
# 1セッションの基本ループ(Anthropicの設計を簡略化)
def coding_session(repo):
    git_log = read_git_history(repo)          # 何が変わったか
    progress = read_file("claude-progress.txt")  # 今どこにいるか
    state = restore_state(git_log, progress)     # 状況を復元

    feature = pick_one_failing_feature(state)    # 機能を1つだけ選ぶ
    implement(feature)                            # 実装する
    run_e2e_tests(feature)                        # ブラウザで実際に検証
    write_progress_note("claude-progress.txt")    # 次の自分への付箋
    git_commit(message=describe(feature))         # 説明付きでコミット
```

ポイントは「機能を1つだけ選ぶ」の行です。エージェントを放っておくと、アプリ全体を一撃で作ろうとして破綻します。1セッション1機能に絞ると、各セッションが小さく確実に進みます。布石を1手ずつ打つ感覚に近いです。

## 初回コンテキストは特別扱いする

私が地味に効くと思うのが、この原則です。

> 最初のコンテキストウィンドウには、異なるプロンプトを使う。

Anthropicはセッションを2種類に分けます。最初に1回だけ走る初期化エージェントと、その後で何度も起こされるコーディングエージェントです。

初期化エージェントの仕事は、環境を整えることです。具体的には3つの成果物を残します。

| 成果物 | 役割 |
| --- | --- |
| feature-list.json | 200件超の機能を構造化。全部「未達」で開始 |
| init.sh | 起動時に開発サーバを自動で立ち上げる |
| claude-progress.txt | 以降のセッションが読む現在地ノート |

feature-list.jsonを最初に全部「未達」で埋めるのが利いています。やることリストが先に確定するので、エージェントが途中で勝手に「完成しました」と宣言できなくなります。

初めて会う人と10年来の友人に、同じ挨拶はしませんよね。初対面では自己紹介から入り、旧友には用件から入ります。初回セッションは環境構築という自己紹介、2回目以降は続きから入る旧友。プロンプトを分けるのは、ごく自然な発想です。

コンテキストリセットの全体像を図にすると、こうなります。

*初回だけ走る初期化エージェントと、何度も起こされるコーディングエージェントの分業*

## Generator-Evaluator:GANから借りた自己批判の構造

もう1つの柱が、生成と評価を分ける設計です。

これはGAN(敵対的生成ネットワーク)に着想を得た構造だと説明されています。生成エージェントが作り、評価エージェントが壊しにかかる。役割を分けるのです。

なぜ分けるのか。LLMに自分の出力を評価させると、ほぼ確実に自画自賛するからです。明らかに凡庸な成果でも、自信たっぷりに褒めます。自分で書いたコードのレビューが甘くなる、あの現象と同じです。私も身に覚えがあります。

そこで構造で解決します。評価エージェントは、生成の過程を一切見ていません。だから埋没コストの心理が働きません。容赦なく粗を指摘できます。

```
# Generator-Evaluatorの基本ループ(Anthropic設計の概念図)

  [Generator] ──作る──▶ 成果物
       ▲                  │
       │                  ▼
   フィードバック    [Evaluator]
       │            (生成過程を見ていない)
       └──壊しにかかる──┘

  Evaluatorの指摘が、次のGeneratorの入力になる
```

この敵対的なテンションが、品質を引き上げます。設計のような主観的なタスクほど、自己評価より他者評価が効くというのは納得感があります。

## マルチエージェントからシングルエージェントへ

ここで前半の伏線がもう一度効きます。

Anthropicは当初、計画・生成・評価を別々のエージェントに分けたマルチエージェント構成で設計していました。ところがClaude Opus 4.6で性能が上がると、構成を簡素化しました。スプリント単位でタスクを分解する仕組みを外し、新しいモデルなら、より長く一貫した作業を1人でこなせると判断したのです。

これが「あらゆる構成要素はモデルの弱点を符号化している」という原則の実践です。

* コンテキストリセットは、Sonnet 4.5の不安を補う部品でした
* マルチエージェント分割は、モデルの一貫性不足を補う部品でした
* どちらもモデルが賢くなると、前提が陳腐化します

OpenAIとの違いも、この観点で整理できます。

| 観点 | OpenAI | Anthropic |
| --- | --- | --- |
| 出発点 | 大規模プロジェクトの全自動化 | 長時間実行の安定性 |
| 力点 | 宣言的プロンプト、スケーリング | コンテキスト管理、品質保証 |
| 構成 | 並列実行でスケール重視 | シングルへ簡素化、シンプル重視 |
| 独自概念 | Agent-first world | Context Anxiety / progress file |
| モデルへの姿勢 | エージェントに全面委任 | 信頼せずガードレールを置く |

Anthropicの一貫した姿勢は「モデルを信頼しすぎない」です。ただし不信は固定ではありません。モデルが育てば、ガードレールを1本ずつ外していく。子どもの自転車から補助輪を外すのに似ています。

## 自分で回して気づいたこと

私も短い長時間タスクを回してみました。失敗から学んだことを2つ共有します。

1つ目。progress fileを甘く書くと、次のセッションが盛大に道を間違えます。「APIを実装中」とだけ書いて寝かせたら、翌セッションは別のAPIを作り始めました。付箋には、次の一手まで具体的に書く。これが効きます。

2つ目。テストをハーネス側で強制しないと、エージェントは平気で「完成」と言います。feature-list.jsonを全部「未達」で開始する設計の意味が、ここで腹落ちしました。検証されない自己申告は、信用してはいけないのです。

設計の急所は、賢いプロンプトより地味な引き継ぎ規律でした。これは正直、意外でした。

## 自分のハーネスを設計に落とすには

ここまでの思想を、自分の長時間タスクに落とす手順をまとめます。私が実際に意識している順番です。

まず、セッションをまたぐ状態を1か所に固定します。git履歴は自動で残るので、足りないのは「次の一手」を書く現在地ノートだけです。ファイル名は何でも構いません。大事なのは、次のセッションが最初に必ず読む場所を決めることです。

次に、完了条件を自己申告に任せないことです。やることリストを最初に固定し、各項目はテストが通って初めて「達成」にします。エージェントは検証がなければ平気で完成を宣言します。ここを人間が握っておくと、暴走の被害が小さく済みます。投資で言えば、最悪でも守れる最低ラインを先に引いておく感覚に近いです。

最後に、入れた部品を定期的に疑うことです。コンテキストリセットもマルチエージェントも、特定のモデルの弱点に合わせた足場でした。半年後、モデルが世代を更新したら、その足場はもう要らないかもしれません。一度組んだ設計を聖域にしないこと。これがAnthropicの思想の核だと私は受け取りました。

| 設計の問い | 確認すること |
| --- | --- |
| 状態をどこに置くか | 次セッションが最初に読む現在地ノートを決めたか |
| 完了をどう判定するか | テストで検証し、自己申告に任せていないか |
| その部品はまだ要るか | モデル更新で前提が陳腐化していないか |

## まとめ

Anthropicの長時間実行エージェント設計を、要点で振り返ります。

* 長時間タスクの敵は記憶喪失です。新セッションは前の記憶を持ちません
* Sonnet 4.5はContext Anxietyを抱えるため、コンパクションだけでなくコンテキストリセットが要りました
* リセットの引き継ぎは git history と claude-progress.txt の2点セットで行います
* 初回セッションは初期化、2回目以降は続行と、プロンプトを分けます
* 生成と評価を分けるGenerator-Evaluator構造で、自画自賛を構造的に防ぎます
* モデルが賢くなると前提が陳腐化し、ハーネスは簡素化されます

最後に1つ。ハーネスとはモデルの弱点リストです。モデルが育てば、補助輪は外していい。今のあなたのエージェント設計に、もう要らない補助輪は付いていませんか。一度棚卸しすると、面白い発見があるはずです。面白くいきましょう。
