## はじめに — コピーが効かない2つの理由

Claude Code に「勝手に進めないで」「削除前に確認して」と CLAUDE.md へ書いたのに、守られなかった経験はないでしょうか。では、よくできた他人の CLAUDE.md をコピーすれば解決するかというと——今度は自分の作業の流れに合いません。

この2つは別の失敗です。前者は**ルールが守られる構造になっていない**問題、後者は**何を承認制にしたいかというリスク感覚が人ごとに違う**問題。そして他人のルール本体のコピーは、どちらも解決しません。

転用できるのは中身ではなく**設計の型**です。具体的には、

* 「いつ」で引くトリガー構造（守られる形にする）
* リスクに比例させる承認粒度（自分の感覚に合わせる次元）
* ルールを腐らせない運用手続き（作る→検証する→記録する）

の3点。この記事では、チェックリスト型のルールで一度失敗した筆者が、作り直しで辿り着いたこの型を共有します。ルールの中身は筆者の環境の例として引用しますが、持ち帰ってほしいのは型のほうです。

なお本記事はシリーズの1本目（ルール設計編）です。続編として②トークン削減編（遅延ロードと hook 注入）・③レビュー委任編（サブエージェントレビュー）を予定していますが、執筆はこれからです。

## 1. 失敗した旧方式 — チェックリスト型の全ルール常駐

最初に作ったルールセットは「アクション前チェックリスト」方式でした。守ってほしい事項を全部書き出し、「行動する前にこのチェックリストを確認すること」と CLAUDE.md に常駐させる形です。

結果は部分的にしか機能しませんでした。ある場面では確認してくれるのに、別の場面では素通りする。ルールの内容を充実させても改善せず、むしろ長くなるほど個々の項目の存在感が薄れていきました。

振り返ると、原因は内容ではなく構造にありました。チェックリストは「**いつ適用するか**」を読み手の判断に丸投げしています。モデルは作業の真っ最中に、いま直面している状況がリスト内のどの項目に対応するかを毎回自力で照合しなければなりません。この照合が起きるかどうかが運任せになっていたのです。

人間の新人に渡す作業マニュアルでも同じことが起きます。「注意事項100連発」は読まれず、「削除する前にこのページを開け」は読まれる。ルールに必要なのは内容の網羅性より先に、**発火条件の明確さ**でした。

## 2. 守られる形にする — 「いつ」で章を立てる

そこで作り直したルールセットは、**トリガー（〜する前／〜した時）をそのまま章節の見出しにする**構造にしました。現行の章構成はこうなっています。

```
0. ルールを定義・変更しようとした時
1. 作業の宣言・準備
   1.1 複数ステップの作業を始める前
   1.2 調査・分析を始める前
   1.3 作業フォルダが git 未管理だった時
   1.4 中断した作業を再開する時
   1.5 レビューを始める前
2. 作業中の判断
   2.1 仮定を置こうとした時
   2.2 次ステップの選択肢を提示する前
   2.3 宣言済みのスコープを変更したくなった時
   2.4 ユーザーに複数の質問をする時
3. 操作前の承認
   3.1 ファイルを操作する前
   3.2 コマンドを実行する前
   3.3 外部に送信・公開しようとした時
   3.4 一括・反復操作を行う前
   3.5 機密情報を扱おうとした時
   3.6 統制機構そのものを変更しようとした時
   3.7 サブエージェントに委任しようとした時
   3.8 外部の情報・コードを取り込もうとした時
   3.9 セッションをまたぐ記憶を保存しようとした時
   3.10 バックグラウンド・定期実行を仕掛けようとした時
4. 問題・発見時
   4.1 エラーが発生した時
   4.2 挙動の誤りを指摘された時
   4.3 作業中にスコープ外の問題・気になる点を発見した時
   4.4 作業が中断・未完了で終わった時
   4.5 作業対象がセッション外で変更されていた時
5. 完了を報告しようとした時
```

見出しを眺めるだけで「どの瞬間に何のルールが効くか」が分かります。作業のライフサイクル（準備→判断→操作→問題→完了）に沿って並んでいるので、モデルにとっても人間にとっても、状況とルールの対応付けが照合ではなく**参照**になります。

もう1つの要素が**二層構造**です。CLAUDE.md 本体には各節の「要点（トリガー時の行動と承認の重さ）」を2〜4行だけ書き、境界判断・例外・出力フォーマットといった詳細は `rules/X.Y-xxx.md` という個別ファイルに分けて、**トリガーに該当したときに Read してから行動する**規約にしています。たとえば CLAUDE.md 側の1節はこの分量です。

> ### 4.3 作業中にスコープ外の問題・気になる点を発見した時
>
> 「発見・報告するが修正しない」を基本とする。重大度に応じて報告タイミングを変える（重大：即中断・報告、中程度：発見時点で報告、軽微：完了後にまとめて報告。迷う場合は中程度）。  
> → rules/4.3-out-of-scope.md

常駐するのは見出しと要点だけなのでコンテキストを圧迫せず、詳細は必要な瞬間に必要な1ファイルだけ読まれます（この遅延ロード化とトークン削減の顛末、および「Read してくれること」自体を hook で補強する仕組みは、シリーズ②で扱う予定です）。

## 3. 自分に合わせる次元 — 承認の粒度3種

「いつ」を固定したら、次は「どのくらい止めるか」です。ここが冒頭で述べた**自分のリスク感覚に合わせる次元**です。

全ルールで共通の語彙として、承認の重さを3種類だけ定義しています。

| 粒度 | 意味 | 使う場面 |
| --- | --- | --- |
| **明示承認** | 「ok」「進めて」等の肯定的応答を得るまで進まない | 破壊的操作・外部送信・仮定など、誤りの影響が大きい場面 |
| **方向修正待ち** | 提示後、否定・修正指示がなければ進んでよい | スコープ宣言・中間判断など、影響が限定的な場面 |
| **選択** | 提示した選択肢から選ぶ回答を得るまで進まない | 出力先の確認・中断後の次の行動など、複数の正解があり得る場面 |

各ルールは要点の中で「この場面は明示承認」「これは方向修正待ち」と粒度を指定します。ポイントは**リスクと承認の重さを比例させる**ことです。すべてを明示承認にすると確認地獄になって作業が進まず、ユーザー側の承認も雑になって形骸化します。そこで各ルールに例外・簡略化規定を分散して埋め込んでいます。

* 手戻りが軽微な仮定（目安: 1ファイル以内かつ10行以内）は承認不要、ただし仮定の明示はする
* 読み取り専用コマンドの事前説明は1文に簡略化してよい
* スコープの縮小方向は方向修正待ちでよい（拡大方向は明示承認）
* 同一ファイル内の複数編集は、全件列挙した「編集セット」の一括提示で承認1回にまとめられる

どの操作を明示承認に置くか、「軽微」の目安をどこに引くかは、まさに人によって違う部分です。粒度の語彙を3種に固定してあると、この調整が「このルールの承認を明示承認から方向修正待ちに下げる」という**1語の置き換え**で済みます。他人のルールセットが合わない正体はたいていここの感覚差なので、語彙だけ持ち帰って自分の値を入れるのが早道です。

## 4. ルール1本の実例

詳細ファイル側の書き方も1本だけ実例を見せます。全ルールは共通の雛形（**目的 → 適用範囲 → 詳細ルール → 例 → 関連ルール**）に従います。先ほどの 4.3（スコープ外発見）の詳細ファイルの骨格はこうです。

* **目的**: 作業範囲を勝手に広げないことと、発見した問題を黙殺しないことの両立。「ついでに直す」も「黙ってスルー」も防ぐ
* **適用範囲**: スコープ外のバグ・改善余地・不整合・セキュリティリスクの発見時。typo 等の1行修正は対象外（直して完了報告で一言触れる）
* **詳細ルール**: 「発見・報告するが修正しない」を原則に、重大度で報告タイミングを変える（軽微=完了後にまとめて／中程度=発見時点／重大=即中断。迷ったら中程度）。報告フォーマットも固定:

```
【スコープ外の発見】
- 場所: <ファイルパス:行番号>
- 内容: <何が問題か>
- 重大度: 軽微 / 中程度 / 重大
- 推奨対応: <どう直すべきか>
```

* **例**: 良い例（完了報告にまとめて列挙）と悪い例（「ついでに null チェックも追加しておきました」／黙殺）を対で載せる
* **関連ルール**: 報告後にユーザーが修正を指示したら、それはスコープ変更なので 2.3 へ接続する——という**ルール間の受け渡し**を明記

「勝手に直すな」という一言で済みそうな規範でも、報告のタイミング・フォーマット・境界（どこからが「軽微」か）・次のルールへの接続まで落とすと、挙動が安定して再現されるようになります。例（良い例・悪い例）は特に効きます。

## 5. 運用編 — ルールを腐らせない

ここまでが「作る」の型です。しかしルールセットは作って終わりではなく、運用の中で増え、変わり、放置すると矛盾と陳腐化で腐ります。筆者の環境では「**作る→検証する→記録する**」のサイクルを型にしています。自分でルールを策定・運用し続けるための部分であり、コピーでは決して手に入らないのはむしろこちらです。

### 5.1 ルールのルール（メタルール）

章構成の先頭に「0. ルールを定義・変更しようとした時」というメタルールを置いています。新規ルールは目的・トリガー・詳細・統制観点の明記と雛形への準拠を必須にし、追加前に既存ルールとの重複・競合・優先順位の曖昧さ・相互参照の漏れをチェックします。

もう1つ重要なのが、**ルール・hook・permissions といった統制機構そのものの変更に、通常より重い承認を課す**ルール（3.6）です。変更を緩和方向（許可の追加・チェックの削除など）と強化方向に分類し、緩和は必ず明示承認、迷ったら緩和扱い。AI 自身が「作業を通しやすくするために自分の縛りを緩める」提案をする場合は、その動機の明示まで義務付けています。ルールセットの実効性は、この自己言及的な一段で担保されます。

### 5.2 穴の見つけ方 — 統制観点8軸

ルールを足していくと「このセット、どこかに穴がないか」が気になり始めます。網羅性を思いつきではなく検証するために、評価軸を導出して使っています。AI への委任の構成要素は「行為」「情報」「統制系（ルールや記録という仕組み自体）」の3つに閉じる、という分解から8観点を出します。

| 対象 | 観点 | 問い |
| --- | --- | --- |
| 行為（事前） | 決定権 | 判断の節目をユーザーが握れているか |
| 行為（事中） | 可観測性 | 作業の過程が見えているか |
| 行為（事後） | 正確性 | 事実と推測が区別され、検証されているか |
| 行為（事後） | 可逆性 | 失敗から元に戻れるか |
| 情報 | 情報境界 | 情報が越えてよい境界が管理されているか |
| 統制系 | 強制力 | ルールは仕組みで担保されているか |
| 統制系 | 追跡性 | 経緯と現状を記録で追えるか |
| 統制系 | 比例性 | 統制の重さがリスクに比例しているか |

検証は、全ルールを観点にマッピングして「どの観点にも載らないルール」と「ルールが1つもない観点」を探す双方向マッピング、実際に起きたヒヤリを「どの観点の欠如か」で説明するインシデント駆動、STRIDE 等の既存分類との照合、の3本立てです。8観点で説明できない失敗が出たら、観点自体を見直します。

軸は自作でなくても構いません。大事なのは「ルールを足す根拠」と「足りていることの確認方法」を、その都度の思いつきから切り離すことです。

### 5.3 台帳で記録する

最後が記録です。ルールの実体（`~/.claude/` 配下）とは別に、記録用のリポジトリで3つの台帳を管理しています。

* **Catalog** — いま何がある（ルール・hook・スキル等の現状一覧）
* **CoverageMatrix** — 8観点と実装のマッピング（穴の検証結果）
* **WorkLog** — 何を・なぜ変えたか（経緯）

ルール変更のたびに台帳を同期する義務も、それ自体がルールになっています。数ヶ月後の自分が「なぜこのルールはこうなっているのか」を辿れるかどうかは、この経緯の記録で決まります。なお、グローバル設定側から特定プロジェクトを参照しない（参照は常に記録側→設定側の一方向）という原則を守ると、設定の可搬性が保てます。

この3つの台帳とレビュー記録の実物（執筆時点のスナップショット）は、[付録リポジトリ](https://github.com/RS-Dev-Git/claude-code-rules-appendix)で公開しています。

## おわりに — 持ち帰るのは型

この構造にしてから、「承認なしで破壊的操作が走る」「聞いてほしい判断を素通りされる」場面は目に見えて減りました。一方で、テキストの規約だけでは「詳細ファイルを Read せずに動く」「長いセッションでルールの存在感が薄れる」リスクが原理的に残ります。そこは hook による仕組み側の補強（シリーズ②）と、成果物を別の目で検証するサブエージェントレビュー（シリーズ③）で埋めています。

繰り返しになりますが、この記事のルール本文はコピーしないでください。持ち帰るのは3つの型です。

1. **トリガーで章を立てる** — 「いつ」を見出しにし、要点常駐＋詳細遅延の二層にする
2. **承認粒度を3語に固定する** — リスクとの比例は自分の感覚で調整する
3. **運用の手続きを持つ** — メタルールで増改築に手続きを課し（作る）、穴は検証軸で探し（検証する）、経緯は台帳で未来の自分が辿れるように残す（記録する）

なお、公開しているルール・hook 一式（[設定スナップショット](https://github.com/RS-Dev-Git/claude-code-config-snapshot)）は執筆時点のスナップショットです（筆者の現行環境はその後も更新されており、公開物には追随しません）。あくまで型を検討するためのベースラインであり、セキュリティに関わる部分（hook の検知パターン・permissions）は、各自の環境とプロジェクトのリスクに合わせて詰めてください。

最初の1本は、自分が実際にヒヤリとした場面——「あのとき勝手に消された」「あの確認が欲しかった」——から書き始めるのが最短です。それが冒頭の「自分に合ったルール」の出発点になります。
