複数の制作を並行していると、**非公開の作業メモ**と**公開用の実績一覧**が別ツールに分かれ、同じ URL や日付を二度入力することになりがちです。

今回公開した **my-atelier-vault**（Obsidian Wiki）と **Gallery**（GitHub Pages SPA）は、Wiki を正本にしてポートフォリオへ流すための MIT テンプレートです。

作品としてのコンセプトや推しポイントは[Qiita](https://qiita.com/egaruk/items/06aaccbc671930604fb7)に書きました。こっち見た方が何やりたいかわかりやすいかも。ZENNの記事は裏方向けです。

## 解決したいこと

* 音楽・イラスト・開発・同人誌など **領域が違うプロジェクト**を同時進行している
* タスクと経緯は **非公開**に残したいが、作品・SNS・コミュニティは **公開**したい
* ポートフォリオ用に **サーバーを契約したくない**

この 3 つを同時に満たすために、次の 2 層に分けました。

## my-atelier-vault — Backroom を正本にする

[my-atelier-vault](https://github.com/YUzushio/my-atelier-vault) は Obsidian で開く Wiki テンプレです。

### プロジェクトの分類

各プロジェクトは `Backroom/{slug}/_index.md` に集約し、frontmatter で方針を明示します。

| フィールド | 意味 |
| --- | --- |
| `visibility: open` | 公開中の制作・コミュニティ |
| `visibility: private` | 非公開で継続（副業・社内など） |
| `visibility: closed` | 終了プロジェクト |
| `gallery: true` | Gallery に載せる意思 |

`gallery: true` は visibility と独立です。private プロジェクトでも「法人設立・公式 HP だけ Gallery に出す」といった使い分けができます。

### Agent Skills でセットアップと横断 TODO

fork 直後は Cursor / Claude Code で `@wiki-setup`、VS Code Copilot では `.cursor/skills/wiki-setup/SKILL.md` を参照します。

| Skill | 用途 |
| --- | --- |
| `@wiki-setup` | 初回セットアップ · `vault.config.json` 生成 |
| `@wiki-todo-query` | Backroom 横断 TODO · ローカルダッシュボード |
| `@morning-briefing` | 朝の着手提案 |
| `@gallery-vault-sender` | Gallery 掲載用メタの整備 |

スクリプトのみでも動きます。

```
node .cursor/skills/wiki-setup/scripts/setup.mjs
node .cursor/skills/wiki-todo-query/scripts/open-dashboard.mjs
```

## Gallery — JSON だけのポートフォリオ SPA

[Gallery テンプレ](https://github.com/YUzushio/yuzushio.github.io) は React 19 + Vite + TypeScript です。データは `public/data/gallery.json` のみ。

### GitHub Pages で無料公開

User Pages（`{ユーザー名}.github.io`）想定です。fork 後にリポジトリ名をリネームし、`main` へ push すれば GitHub Actions がビルド・デプロイします。VPS・ドメイン契約は必須ではありません。

初回セットアップは `@gallery-setup`（または `node .cursor/skills/gallery-setup/scripts/setup.mjs`）が案内します。

### データモデルの要点

```
{
  "id": "my-work",
  "kind": "work",
  "category": "project",
  "genres": ["web"],
  "title": "表示タイトル",
  "summary": "一行説明",
  "year": 2026,
  "site": "https://example.com"
}
```

* `kind: hub` — SNS チャンネルなどの**入口**（Feed 非表示がデフォルト）
* `kind: work` — 動画・楽曲・同人誌などの**個別成果**
* `parentId` — hub 配下の子（例: YouTube チャンネル → 各動画）

Feed / Drill / Timeline / Category の 4 ビューは、同じ `items[]` から自動的に切り替わります。

## Vault → Gallery の受け渡し

Wiki 側で `## Gallery 掲載` 表を書いておくと、receiver Skill が子パネル（`work` + `parentId`）に展開します。

**Vault（送信）**

```
## Gallery 掲載

| タイトル | 投稿日 | URL | メモ |
|----------|--------|-----|------|
| 初版リリース | 2026-06-14 | https://github.com/... | |
```

**Gallery（受信）**

> `@gallery-vault-receiver` Backroom の {slug} を gallery.json に反映して

`export-from-vault.mjs` だけでは `kind` やサムネが足りないため、受信専用 Skill があると安全です。

## 使い始める最短手順

1. [my-atelier-vault を fork](https://github.com/YUzushio/my-atelier-vault/fork) → clone → `@wiki-setup`
2. Obsidian で vault として開く
3. [Gallery を fork](https://github.com/YUzushio/yuzushio.github.io/fork) → リポ名を `{ユーザー名}.github.io` に変更 → `@gallery-setup`
4. Wiki でプロジェクトを書き、`gallery: true` を付ける
5. Gallery 側で `@gallery-vault-receiver`

ローカルでは sibling 配置（`~/my-atelier-vault` と `~/gallery`）にすると Skill のパス参照が楽です。

## 作者の運用例

作者（YUzushio）の公開 Gallery <https://yuzushio.github.io/> には、YouTube・X・同人誌・コミュニティ（OV2L など）を Wiki から載せています。  
テンプレ自体も今後ここに掲載予定です（本ノート: `Backroom/atelier-open-toolkit/`）。

## まとめ

* **Wiki** = 非公開ログも含む正本 · Obsidian + Git
* **Gallery** = 公開ビュー · GitHub Pages 無料
* **Skills** = fork 直後からエージェントでセットアップ・同期

MIT ライセンスなので、社内 Wiki の土台や個人サイトのベースとして自由に使えます。
