「Claude Codeに何回同じ前提を説明したら、自分のことを覚えてくれるんだろう」。こんな経験がある人は、今夜30分だけ時間を取ってください！

私も会話のたびに「私はXX担当で、判断軸はこれで……」と説明し直すストレスを感じてました。

解決策を探す中で、**Claude Code × Obsidian の活用**にたどり着きました。

巷には Obsidian 関連の記事が溢れております。**この記事では、10を超える国内外の記事・事例を私が徹底分析・凝縮しました！**

この記事を読んで手順をまねるだけで、Claude Code は**あなたを完全に理解した専属 AI エージェントへと進化します。**

背景には、2026年4月の2つの動きがあります。元OpenAI共同創業者の Andrej Karpathy が gist「LLM Wiki」を公開し、同月『Building a Second Brain』著者の Tiago Forte が「AI時代のPARA」を宣言しました。

**第二の脳というコンセプトが、人間が読むためのノート群から、AIが直接読むためのノート群へと再定義**され始めています。

この記事では、私が実際に運用している Claude Code × Obsidian で「第二の脳」を作る手順を解説します。**30分で構築できて、一度構築したら Claude Code が会話間での記憶を永続記憶し、専属 AI エージェントとして機能します。**AI が毎セッション最初に `Memory.md` を読み、あなたの役割・文体・判断基準を前提として共有した状態で会話が始まります。さらにそれだけではなく、毎朝「おはよう」と打つだけで AI が前日からの繋がり・最近のパターン・次に問うべき問いを返してくれます。週末には過去の自分との矛盾を炙り出してくれます。  
**今夜の30分は、その入口です！**

## 第二の脳の設計図 — Karpathy と Tiago Forte が提示した最新コンセプト

まず、なぜ今わざわざ Obsidian なのかを押さえます。設定を進める前にこの設計図を共有しておくと、後で迷ったときに自分で判断できるようになります。

### 1. Karpathy の「LLM Wiki」3層 3操作

Karpathy の gist は、自分のリサーチノートを LLM に直接書かせる仕組みを公開したものです。原文の核心はこの一文に詰まっています。

> Obsidian is the IDE; the LLM is the programmer; the wiki is the codebase.

日本語に意訳すると、**Obsidian は作業環境、AI は書き手、Wiki（蓄積されたノート群）はその成果物**、という分業宣言です。

この分業を支えるのは、Obsidian の Vault フォルダ配下のシンプルな3層構造です。

![](https://assets.st-note.com/img/1779289517-qNkv2wjEG4Cr16SPQARtsFxW.png?width=1200)

Obsidian Vaulyフォルダ構造

* `raw/` — 取り込んだ生の素材（記事・PDF・会議メモ・チャットログ）を放り込む箱
* `wiki/` — AI が `raw/` を読んで書き起こす、自分の言葉でまとまったノート群
* `CLAUDE.md` — 「自分の Wiki をどう運用するか」のルールブック。AI が毎回最初に読む取扱説明書

そのうえで AI が走らせる3操作は次の通りです。

1. Ingest（インジェスト・取り込む）— 素材を `raw/` に入れる
2. Query（クエリ・聞く）— AI が `wiki/` 全体を読んで質問に答える
3. Lint（リント・整える）— 古い情報・矛盾・孤立ノートを定期的に掃除する

Karpathy 本人のリサーチ Wiki はすでに100記事・40万語まで成長しており、その大半を LLM が自動執筆しているとのことです。原典はこちらです。

<https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f>

### 2. Tiago Forte の「AI時代のPARA」

もう一方、第二の脳構築法（BASB）の本家である Forte は、Master Prompt Method という考え方を打ち出しました。  
これは「**自分の役割・価値観・判断基準を凝縮した、AI向けの取扱説明書を Vault（金庫＝ノート保管庫）の中に置く**」という発想です。

つまり、AI に毎回ゼロから自分を説明するのをやめて、自分の取扱説明書を一度書いて、ファイルとして手元に持つということです。これが後で出てくる `Memory.md` の正体になります。Karpathy の `CLAUDE.md` と Forte の Master Prompt は別ルートから出てきた発想ですが、結論はほぼ同じところに着地しています。

### 3. なぜ Obsidian なのか

ここまで読んで「Notion ではダメなのか」と思った方もいるはずです。  
結論を先に言うと、**2026年時点で AI エージェントが直接読み書きできる主要 PKM は Obsidian だけ**、というのが英語圏の複数ガイド共通見解です。

> **PKM とは**  
> Personal Knowledge Management の略で、個人の知識を整理して保管するツール全般のことです。Notion・Apple Notes・Google Docs・Evernote・Logseq 等が PKM に分類されます。

* Notion — API 経由しかアクセスできず、エージェントが直接ファイルを grep（中身を検索）できない
* Apple Notes — ファイルシステムからアクセスする手段がない
* Google Docs — 連携の認証フローが複雑
* Logseq — ブロックベースの構造で AI 連携の主流ではない

Obsidian は中身がただの Markdown ファイルの集まりで、フォルダごとパソコンの中に保存されています。だから AI は認証トークンも OAuth（オーオース・他サービスにログインさせる仕組み）も使わず、ファイルを読む・書く・検索するだけで動けます。

**これが「AI ネイティブな第二の脳」の物理的な根拠です。**

## 今夜30分のハンズオン — STEP 1〜5

ここから手を動かします。所要時間は環境差はありますが、合計30分程度を見込んでください。Mac / Windows どちらでも進められます。

用意するものは次の3つです。

### STEP 1: Vault フォルダを作って git init する（所要 5 分）

まず Obsidian の Vault（ボールト・金庫の意味）になるフォルダを1つ作ります。場所はどこでも構いませんが、`Documents/second-brain` のような分かりやすい場所をお勧めします。

次に、そのフォルダの中で `git init` というコマンドを1回だけ打ちます。  
Git（ギット）は変更履歴を管理してくれる仕組みで、万一の操作ミスでもファイルを巻き戻せる安心装置になります。Mac はターミナル、Windows はコマンドプロンプトか PowerShell を開いて、次の2行を順に打ってください。

```
cd ~/Documents/second-brain
git init
```

`cd` は change directory（フォルダ移動）の略で、自分の作業場所を変える命令です。`git init` はこの場所に履歴管理を始める命令です。これで Vault のベースラインが取れました。

![](https://assets.st-note.com/img/1779289730-pPb0dQsmg31yhalUGiYOtVv4.png?width=1200)

git initでバージョン管理

### STEP 2: Obsidian をインストールして Vault を開く（所要 5 分）

Obsidian の公式サイトからアプリをダウンロードしてインストールします。

起動したら「Open folder as vault（フォルダを Vault として開く）」を選び、STEP 1 で作った `second-brain` フォルダを指定します。これで Obsidian がそのフォルダの中の Markdown ファイルを管理するモードになります。最初は空っぽの画面が出ますが、それで正常です。

### STEP 3: obsidian-skills を 1 行コマンドでインストールする（所要 3 分）

ここが2026年の最新ポイントです。Obsidian CEO の Steph Ango が、Claude Code 用の専用 Skill 集を公開しています。

<https://github.com/kepano/obsidian-skills>

これは**2026年5月時点でGitHub スター約 31,500 件と、注目度が極めて高いリポジトリ**です。同梱されている Skill は5つあります。

* `obsidian-markdown` — Obsidian 独自の記法（wikilinks・embeds・callouts・properties）を Claude に理解させる
* `obsidian-bases` — Bases ファイル（データベース機能）の文法
* `json-canvas` — Obsidian Canvas の操作
* `obsidian-cli` — Vault をコマンドラインから操作
* `defuddle` — Web ページからきれいな Markdown を抽出

Vault フォルダのトップで以下のコマンドを1回打つだけで、5つすべてが Claude Code に組み込まれます。

```
cd ~/Documents/second-brain
mkdir -p .claude
cd .claude
git clone https://github.com/kepano/obsidian-skills.git skills
```

これを入れずに Claude Code を使うと、Obsidian の独自記法（特に `[[wikilink]]` のような二重角括弧リンク）を Claude が破壊してしまう可能性があります。**これは飛ばさないでください。**

### STEP 4: [Memory.md](http://Memory.md)（あなたの取扱説明書）を5分で書く（所要 8 分）

Vault のトップに `Memory.md` というファイルを新規作成し、以下のテンプレをコピペしてから、各項目を自分の言葉で30秒ずつ埋めてください。これが Forte の言う Master Prompt の正体で、AI が毎セッション最初に読む「あなたの取扱説明書」になります。

```
# Memory.md — 私の取扱説明書

## 私は誰か
- 役割: （例）事業会社のマーケティング担当 / 26歳 / マネジメント経験なし
- 専門領域: （例）BtoB SaaS の獲得チャネル / コンテンツマーケ
- 開発スキル: コード未経験。コマンドのコピペはできる

## 現在のフォーカス（YYYY-MM-DD 時点）
- 今週特に追っているテーマ: （例）AI を業務に組み込む / Claude Code 導入 / Obsidian 第二の脳構築
- 直近の判断ポイント: （例）有料 SaaS の選定基準を整理したい

## 文体・出力ルール
- 結論先出し。理由は後
- 敬体（〜です・〜ます）を維持
- 過剰な絵文字や「！」連打は使わない
- 専門用語は初出時に1行で平易に説明する

## 判断基準
- シンプル第一。変更は最小限
- 削減優先。新規ファイル追加前に既存への統合を検討
- 根本解決。応急処置で済ませない

## AI に期待する仕事
- 私の見落としを浮かび上がらせる
- 私の前提に同意する前に挑戦する
- 過去の発言と矛盾していたら指摘する
```

**この5セクション・全部で約50行のテンプレが、第二の脳の起動装置になります。**「自分はどう動きたい人間か」を文字にして手元に持つだけで、  
毎セッション「結論先出しで」「絵文字は要らない」と打ち直す手間が消えます。

### STEP 5: [CLAUDE.md](http://CLAUDE.md)（Vault 運用ルール）を5分で書く（所要 5 分）

Vault のトップにもう1つ、`CLAUDE.md` というファイルを作ります。これは Karpathy の言う「Wiki の運用ルール」で、フォルダ構造・命名規則・AI への指示の置き場所を1ファイルにまとめたものです。

以下のテンプレをコピペすれば、最低限のルールが揃います。

```
# CLAUDE.md — この Vault の運用ルール

## あなたの役割
私の第二の脳の伴走者として、私が放り込んだ素材を読み解き、私の言葉で再構成する。

## フォルダ構造
- raw/ — 取り込んだ生素材(記事・PDF・会議メモ・チャット)
- wiki/ — 自分の言葉でまとまった主張ノート
- daily/ — 毎日の Daily Note
- inbox/ — 仕分けに迷ったときの一時置き場

## ノート作成ルール
- 1ノート1主張。タイトルは領域名ではなく完結した主張にする
- 例: NG「X 戦略」 / OK「無料コンテンツで具体手順を見せることが有料会員への最短導線になる」
- 私が AI に「振り分けて」と言うまで、フォルダ移動は勝手にしない

## 私の取扱説明書
Memory.md を毎セッション最初に読むこと

## 禁止事項
- 私が指示していない範囲のファイルを書き換えない
- 出典のない数値を本文に置かない
```

公式 Claude Code ドキュメントは `CLAUDE.md` の長さを「200行以下を目標に」と推奨しています。**Karpathy はさらに厳しく80行を提案**しています。理由は単純で、全行が AI のコンテキストウィンドウを消費するからです。最初はこの30行版で十分なので、運用しながら育ててください。

参考: Claude Code 公式 Memory 仕様

### おまけ STEP: Obsidian Web Clipper で素材の入口を作る（所要 3 分）

最後にもう1つ、素材の取り込み口を確保しておきます。Obsidian Web Clipper は公式の無料ブラウザ拡張で、Web ページを Markdown 形式で Vault に保存できます。

中に Interpreter という LLM 連携機能が組み込まれていて、ページをクリップする瞬間に「日本語で要約して」「YAML frontmatter（ファイル冒頭のメタ情報領域）を抽出して」のような自然言語指示を Gemini / Claude に投げられます。  
これで**気になる記事を見つけたら、3秒で `raw/` に放り込める導線が完成**します。

![](https://assets.st-note.com/img/1779289903-sfvgYZdmuDz48T0J1rIXcLx2.png?width=1200)

完成形

ここまでで30分。今夜の第二の脳1日目は終了です。

## 翌日からの育て方 — Daily Brief と Weekly Synthesis

ここからは2日目以降の運用です。第二の脳は「作って終わり」ではなく「毎日5分・週に15分」だけ AI と対話することで成長します。具体的には、毎朝の Daily Brief と毎週日曜の Weekly Synthesis という2つのルーティンを設定します。両方ともコピペできるプロンプトを置いておきます。

### Daily Brief（毎朝5分）

朝、Claude Code を起動して以下のプロンプトを送ります。これだけで AI が Vault 全体を読んで、3要素を返してくれます。

```
私の Vault を読んで、Daily Brief を返してください。

出力フォーマット:
- CONNECTIONS: 昨日のノートと過去のノートで見つかった意外な繋がりを3点
- PATTERN: 直近1週間で繰り返し出てきているテーマや感情を1つ
- QUESTION: 今日、私が次に問うべき問いを1つ

出典として、参照したノートのファイル名を必ず添えてください。
```

これをやると、自分でも気づいていなかった頭の中の偏りや、放置していた問いが浮き上がってきます。

### Weekly Synthesis（毎週日曜15分）

日曜の夜、こちらのプロンプトを送って週次の振り返りをします。

```
私の Vault の直近7日間のノートを読んで、Weekly Synthesis を返してください。

出力フォーマット:
- EMERGING THESIS: 今週の私のノートから形を取り始めている主張を1つ
- CONTRADICTIONS: 過去の私の発言と矛盾している箇所があれば指摘
- KNOWLEDGE GAPS: このまま行くと判断を誤りそうな知識の空白を1つ
- ONE ACTION: 来週、私が取るべき具体行動を1つ

出典として、根拠にしたノートのファイル名を必ず添えてください。
```

この2つのプロンプトをコピーしておけば、明日の朝から第二の脳が本当に「対話相手」として動き始めます。週末の Weekly Synthesis は、自分一人ではまず気づけない「過去の自分との矛盾」を炙り出してくれる点で、特に強力です。

### 初心者を殺す3大失敗パターン — 先に知っているかで半年後の差が出る

ここで運用の落とし穴を3つ共有します。多くの人が最初の2週間で挫折するという統計があり、価値が見え始める変曲点は80〜150ノート蓄積後と言われています。そこに辿り着く前にやめてしまう人がどこで詰まるかを、先に知っておくと打率が上がります。

1. 旧アプリから全ノートを一括移行する — 過去の Notion / Evernote から数百ノートを移してしまうと、Vault が他人の言葉だらけになり、自分の言葉で書く動機が消えます。最初の1週間は新規ノートだけで運用してください
2. コミュニティプラグインを早期に山積みする — Dataview などは強力ですが、プラグイン依存のロックインを作ります。最初の1ヶ月はプラグインゼロで運用してください
3. Obsidian を「なんでも屋」にする — タスク管理・カレンダー・家計簿まで詰め込むと、AI が読む対象が肥大化し、検索精度が落ちます。Obsidian は「考えるための場所」に絞ってください

### 取り込みの摩擦をゼロにする2つのルール

運用継続には、入力の心理的コストを下げる仕組みも欠かせません。次の2つのルールが英語圏ガイドの共通解です。

つまり、入力時には絶対に整理しない、ということです。整理は AI と週次でやれば十分です。

### 1 ノート 1 主張・タイトル = 主張

もう1つ、ノートの書き方の原則を1つだけ覚えてください。**タイトルを領域名ではなく、完結した主張にする**ことです。

AI は領域名で検索しているのではなく、意味で検索しています。タイトルが主張になっていると、現在のクエリと意味的に一番近い1ファイルが一発で当たる。検索精度が物理的に上がります。

## 3 ヶ月後に来る壁と、次のステップ

この章は、しばらく運用したあとに「ここで詰まる人が多い」という長期視点の話です。今夜の30分には影響しませんが、半年後に効きます。

### ノート数の三層整理 — 重くなる前に知っておく

「Obsidian は1万ノートで重くなるのでは」という不安をよく見ますが、定量的な目安があります。

* 約500ノート — AI 検索精度に最初の手当てが必要になるライン（キーワード衝突が出始める）
* 約5,000ノート — Vector Hybrid 検索（意味検索とキーワード検索の組合せ）が価値を持ち始める
* 約7,000ノート — Obsidian アプリ自体の動作遅延が出始める（HDD 環境）
* 約12,000ノート — SSD 環境でも日常操作にもたつきが出始める

つまり SSD のパソコンであれば、当面1万ノートまでは現実的に到達しません。最初の1〜2年は「重くなる心配」より「続ける仕組み」を優先してください。

### Vault 資産と OS 資産の責務分離

もう1つ、長期で資産を守るコツがあります。それは Vault 資産（Markdown のデータ層）と OS 資産（`.claude/` 配下の Skill / Hook / Command / `CLAUDE.md` の運用ルール）を頭の中で別物として扱うことです。

月初に10分だけ取って、追加した Skill や Command を A〜D 等級で点検すると、ハーネス固有（D等級）に過剰投資して「乗り換え不能なほど深く依存する」状態を避けられます。

### 自作する前に既製コマンド集を探す

第二の脳の運用に慣れてくると「こういうコマンドが欲しい」と思う場面が増えます。そのとき、いきなり自作する前に既製のコマンド集を探してください。代表的なものに `eugeniughelbur/obsidian-second-brain` という GitHub リポジトリがあります（7週間で 1,000+ stars を獲得、Claude Code / Codex CLI / Gemini CLI / OpenCode の4 CLI 共通で動く 30 を超えるコマンド体系）。

<https://github.com/eugeniughelbur/obsidian-second-brain>

注目コマンドだけ抜粋すると次の通りです。

* `/obsidian-connect` — 2つの離れたドメインの間に橋渡しになる概念を探す

### 複数デバイスで使う場合の Sync 選定

複数のパソコン・スマホで同じ Vault を見たい場合は、同期方法の選び方が長期安定性を決めます。長期での安定性順は次の通りです。

* 1位: Obsidian Sync（公式・Standard $5/月 〜 Plus $10/月）— Obsidian の挙動を最もよく知っているため安定
* 2位: Dropbox — 開発者間で信頼性評価が高い
* 3位: iCloud — Windows 環境では破損・重複報告がある（プレースホルダー化挙動が Obsidian と相性が悪い）
* 4位: Syncthing — 無料でプライバシー特化だが技術知識が必要

Windows + Mac 併用なら、最初から Obsidian Sync を選ぶのが将来の事故を一番減らせます。Mac 単独なら iCloud でも当面は大丈夫ですが、ノート数が増えてきたら（前述の 500 ノートを目安に）Obsidian Sync に乗り換えるのが現実的なラインです。

## ぶっちゃけ、まだ第二の脳には限界がある

手放しで勧めるのも違うので、運用してみないと見えない弱点を3つだけ正直に書きます。

* 弱点1: 最初の2週間は「これ続けて意味あるのか」と何度か思う — 80〜150ノートに届くまで繋がりが見えにくく、AI の出力もまだ薄い。ここは耐えるしかありません。Daily Brief を機械的に毎朝5分続けるだけで突破できます
* 弱点2: 検索精度はノート数500を超えたあたりで明確に落ちる — このタイミングで Vector 検索系プラグインの導入を検討する必要が出ます。最初から入れる必要はありません
* 弱点3: 機密情報の扱いは慎重に — 業務の機密情報を Vault に入れる場合、Claude API の利用規約と社内ルールの両方を必ず確認してください。Anthropic はデフォルトでは学習に使わない方針ですが、組織として明文化されたガイドラインが必要です

この3つを先に知っていれば、半年後に「なんで続けられなかったんだろう」と振り返らずに済みます。
