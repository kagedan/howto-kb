## はじめに

Claude Code は強力なCLIコーディングエージェントですが、「気がついたらAPI代がとんでもないことになっていた」という経験はありませんか？

その原因は、Claude Code の **ステートレスな設計** にあります。やり取りを重ねるほど、会話の履歴全体が毎回再送信されるため、セッション後半になると「たった1文字直すためのお願い」で数万トークンを消費する事態に陥ります。

本記事では、日常的にClaude Codeを使い倒すエンジニア向けに、**生産性を保ちながらトークン消費を最小化する実践的なコスト最適化術** を解説します。

---

## 💸 なぜClaude Codeはトークンを食うのか？

まず、Claude Code のトークン消費の仕組みを図解します。

Claude Codeはステートレスであるため、コンテキスト（記憶）を維持するために「直近のファイル内容」や「これまでの会話」を常に抱えたままAPIを叩きます。これが「トークン破産」の根本原因です。

---

## 📊 Claudeの4つの課金要素とそれぞれの対策

Claude 4.x ファミリ（Sonnet 4.6 や Haiku 4.5、及びClaude Code）のコストを把握するには、Anthropic APIの **Prompt Caching（プロンプトキャッシュ）** の仕組みを含めた「4つの課金要素」を正しく理解する必要があります。

| 課金要素 | コスト感 | 発生するタイミング | いつ増大してしまうのか？ | 一般的な削減手法 |
| --- | --- | --- | --- | --- |
| **① インプット** *(Input)* | 標準（基準ベース） 例: **$3.00**/1M | キャッシュ対象外の短いテキストや、新たな質問文を送信したとき | 毎回異なる大量のファイル群をコンテキストに含めたり、不要なログ出力を読ませたとき | 作業に関係ないディレクトリを `.claudeignore` で除外し、`@`メンションで必要なファイルだけを指定する |
| **② アウトプット** *(Output)* | 高い（約5倍） 例: **$15.00**/1M | Claudeが回答のテキストやコードを生成したとき | 一度にファイル全体の巨大な書き換えや、複数機能の同時実装を要求したとき | まず「計画（Plan）」だけを出力させ、無駄な試行錯誤を減らす。一度に小さなタスクだけを依頼する |
| **③ キャッシュ生成** *(Cache Write)* | やや高い（1.25倍） 例: **$3.75**/1M | 新しいプロジェクトや履歴データ、長大な `CLAUDE.md` を「そのセッションで初めて」評価したとき | コンテキストの先頭部分（システムプロンプト等）がコロコロ変わったり、頻繁にセッションをリセットしすぎたとき | `CLAUDE.md` のような重い不変データは固定化し、作業タスクを小さく保つ |
| **④ キャッシュ読込** *(Cache Read)* | **激安（0.1倍）** 例: **$0.30**/1M | 制限時間（通常5分）内に、③で生成した内容と「完全に一致する部分」を再利用できたとき | （※これは増大するほど得をする、コスト削減の要となる要素です） | 一つのタスクを連続してやり取りし、「キャッシュを温めた状態」を維持する（Hit率を上げる） |

> [!IMPORTANT]  
> **Prompt Cachingの極意**  
> Claude Codeは自動的に履歴やワークスペースのコンテキストをキャッシュします。  
> 最初の1回は「キャッシュ生成（Cache Write）」として通常の+25%のコスト増になりますが、2回目以降のやり取りは「キャッシュ読込（Cache Read）」として処理され、なんと**標準インプットの1/10（10%）のコスト**に劇下がりします。  
> つまり、**「1回の重いコンテキストを持たせたら、連続してやり取りしてキャッシュの恩恵をしゃぶり尽くす」** のが最もコストパフォーマンスが良い状態となります。ただし、肥大化しすぎるとアウトプット等に影響するため、ここでも `/compact` が重要になります。

## 🛠️ コストの可視化とトラッキング：`ccusage` の活用

コスト最適化の第一歩は、**現在の利用額を正確に把握すること**です。  
Claude Codeの利用状況や消費コストを簡単に確認したい場合は、便利なCLIツールである `ccusage` を活用しましょう。ターミナルで以下のコマンドを実行するだけで、すぐに利用状態を確認できます。

このコマンドを叩くことで、「今日どれくらいトークンを消費したのか」「直近のコストはいくらか」がコマンドライン上でサクッと可視化されます。作業の区切りやセッションの終わりにこまめに `ccusage` を実行して自分のトークン消費ペースを把握しておけば、「請求書を見て破産・絶望する」という事態を未然に防げるはずです。

### 🚨 数値に異常値を見つけた場合の「4つの対処法」

`ccusage` 等で確認した際、特定の課金要素が異常に跳ね上がっている場合の具体的なトラブルシューティングと対処フローは以下の通りです。

#### 1. インプット（Input）が多い場合の対処法

**原因**：毎回異なる大量のファイル群をコンテキストに読み込ませていたり、エラーログの全出力を読ませている可能性があります。  
**対策**：

* 不要なディレクトリや巨大なビルド出力先は `.claudeignore` で確実に除外します。
* `@` メンションで巨大なファイルを丸ごと読み込ませず、「機能Xの実装部分だけツールで検索して」と指示して部分読み取りさせます。

#### 2. アウトプット（Output）が多い場合の対処法

**原因**：ファイル全体の巨大な書き換えや、複数機能の同時実装をClaudeに依頼している可能性が高いです（アウトプットは単価が最高額です）。  
**対策**：

* **「コードを書く前に、まずマークダウンで実装計画（Plan）だけを箇条書きで出力して」** と指示し、手戻りによる無駄な巨大コード生成を減らします。
* 一度に1つの小さなコンポーネントだけを修正・生成するよう、タスクを極小化します。

#### 3. キャッシュ生成（Cache Write）が多い場合の対処法

**原因**：コンテキストの基本となるデータ（ルールファイル等）が頻繁に変更されたり、少しのやり取りですぐにセッションをリセット（ `/clear` ）しすぎている証拠です。  
**対策**：

* `CLAUDE.md` などのシステム系のファイルは一度書いたら不用意に編集せず、静的な「不変のルール」として置いておきます。
* 一度 `/clear` して新しいタスクを始めたら、その実装が完了するまではセッションを継続し、キャッシュを温め続けます。

#### 4. キャッシュ読込（Cache Read）が多い場合

**原因**：二つのパターンがあります。一つは「キャッシュ機能が正常に働き、通常インプットの1/10の価格で再利用できている（健全な状態）」です。もう一つは、**「不必要に巨大なコンテキストを含めたまま対話を続けることで、1回あたりのキャッシュ読込量が異常に肥大化している（危険な状態）」** パターンです。いくら単価が激安とはいえ、無駄な数十万トークンのキャッシュをターンごとに読み込んでいれば、総コストは確実に膨れ上がります。  
**対策**：

* **「現在のコンテキストに、今のタスクには不要な巨大ファイルが含まれていないか」** を疑います（安いからといって無条件にファイルを `@` しすぎていないか注意します）。
* 読込量が肥大化してコストが嵩んでいる場合は、思い切って `/clear` を実行し、現在直面している課題に必要なファイルだけに絞ってセッションを再構築します。

## 🗑️ セッション管理の鉄則（Session Hygiene）

### 1. タスクが変わったら `/clear` を徹底する

最も簡単で効果的な節約術です。認証機能の実装が終わったら、UIの修正に移る前に必ず `/clear` を叩いてください。無関係のコード履歴を引きずることは、トークンをただのゴミ箱に捨てているのと同じです。

### 2. コンテキストを定期圧縮する `/compact`

`/clear` をしたくない進行中の複雑なタスクには `/compact` コマンドが有効です。

> [!TIP]  
> 使い方： `/compact`  
> これを実行すると、Claudeが今までの長い文脈を要約し、不要なファイル履歴を破棄した上で、スリムな状態にコンテキストを再構築してくれます。

**目安:** 30〜40往復程度のやり取りが続いたら、一度 `/compact` することで後続のトークン消費を数分の一に抑えられます。

## 🛡️ コンテキストの防衛線

### `.claudeignore` の導入

Claude Codeは必要に応じてディレクトリを探索しますが、無駄なファイルを読むと探索だけで膨張します。プロジェクトのルートに `.claudeignore` を置きましょう。

```
node_modules/
dist/
build/
coverage/
.next/
*.log
# DBダンプや巨大な画像など
data/**/*.sql
```

### `CLAUDE.md` は最小限に

`/clear` をしても、プロジェクトのルートにある `CLAUDE.md` は毎セッション自動で読み込まれます。  
ここに長大なコーディング規約を書くのはアンチパターンです。

* **❌ 悪い例**: CLAUDE.md に「Reactの規約」「APIの規約」「CSSの規約」をすべて記述してしまう（毎ターン常に5,000トークンを消費します）。
* **⭕️ 良い例**: CLAUDE.md には「それぞれの規約は `docs/rules/react.md` などを参照してください」とインデックスのみを記述します。Claudeが必要な時だけ個別に読みに行きます。

## 🧠 モデル・ルーティング戦略

Claude Codeでは `/model` コマンドで対話の途中に動的にモデルを切り替えることができます。すべての作業を最高スペックモデルに任せる必要はありません。

| モデル | ユースケース | 単価目安 (Input / Output) 【Cache Write / Read】※1Mあたり |
| --- | --- | --- |
| **Claude Haiku 4.5** | ファイル探索、簡単なLint修正、正規表現の変換 | **$1.00 / $5.00** 【$1.25 / $0.10】 |
| **Claude Sonnet 4.6** | 日常の機能実装、大半のコーディング | **$3.00 / $15.00** 【$3.75 / $0.30】 |
| **Claude Opus 4.6** | 複雑なアーキテクチャ設計、原因不明のバグ調査 | **$5.00 / $25.00** 【$6.25 / $0.50】 |

### ⚠️ モデル切り替え時の「キャッシュ断絶の罠」

`/model` コマンドによる切り替えは強力ですが、**1つのセッション内で数ターンごとに頻繁にモデルを切り替えるのは絶対にNG（アンチパターン）です。**

Anthropic APIのプロンプトキャッシュは「モデルごと」に独立しています。そのため、Sonnetでキャッシュが温まった状態からHaikuに切り替えた瞬間、Haikuにとっては「初見の履歴」となり、これまでの全テキストに対して**丸ごとHaiku用のキャッシュ生成（Cache Write）コストが発生**します。細かく切り替えすぎると、「ずっとSonnetのキャッシュ読込（Cache Read）のまま維持していた方が圧倒的に安かった」という本末転倒な事態に陥ります。

> [!IMPORTANT]  
> **正しいモデル切り替えフロー**  
> 「安いから」といって細かく切り替えるのではなく、タスクのフェーズごとに切り替えます。
>
> 1. まずは安価な `Haiku` でプロジェクト全体を探索させ、影響範囲の調査や大まかな計画（Plan）を立てさせます（Haikuのキャッシュを育てます）。
> 2. 計画が完了し、いざコードを書く段階になったら `/model` で `Sonnet` に切り替えます。
> 3. **以降、該当機能の実装が完了するまでは Sonnet のままセッションを継続**し、Sonnetのキャッシュ読込（Cache Read）の恩恵をしゃぶり尽くします。

## 🎯 プロンプトとワークフローの最適化

### ファイル指定の罠：巨大ファイルに `@` は禁手

「認証の機能を直して」と大雑把に頼むと、Claudeはファイル探索ツールを使って無関係なファイルを何度も探り、ツール実行のアウトプットトークンを浪費します。これを防ぐために `@src/auth/validate.ts` と明示的に指定するのは節約の定石です。

しかし、**`@` はファイル全体を無条件でコンテキストにフルロード（全読み込み）してしまう**という罠があります。

* **数百行の小さなファイル**：`@` ディレクティブによる明示指定で問題ありません。探索コストをゼロにする恩恵が上回ります。
* **数千行の巨大なファイル**：`@` すると数万トークンが毎回消費され本末転倒です。この場合は `@` を使わず、**「`src/auth/validate.ts` の中から `validateToken` 関数だけをツールで探して読んで」** と文字列でパスだけを教え、Claudeのツール経由で部分読み取りさせるのが真のテクニックです。

### 先に「Plan」を求めよ

AIにいきなり実装させると、方針が間違っていた場合に何度も修正のやり取り（＝ループ）が発生し、驚くべき速度でトークンが溶けます。

```
# 悪い例
> この機能を作って

# 良い例
> この機能を作りたい。まずは実装計画を Markdown で出力して。まだコードは書かないで。
```

計画が合意できた状態（Plan）でコードを書かせる（Execute）ことで、無駄な試行錯誤コストをカットできます（これは superpowers 等のAIハーネスが取り入れている「brainstorm → plan → execute」の規約と同じ思想です）。

### 英語と日本語のハイブリッド運用（現実的なトークン節約）

日本語よりも**英語を用いたほうがトークン消費・コスト面で圧倒的に有利**になるというトークナイザーの構造的な事実があります。  
英語は「1単語≒1〜1.3トークン」として効率よく圧縮されますが、日本語は「1文字≒1〜3トークン」を消費しやすく、同じ意味でも日本語は英語の**2〜3倍のトークン**を要求してしまいます。

しかし、「プロンプトの入力も、AIの出力もすべて英語で行う」というのは日本人エンジニアにとっては認知負荷が高く、本来の目的である生産性向上に逆行しかねません。そこで **「裏側のコンテキストは英語、日常の会話は日本語」というハイブリッド運用** を推奨します。

* **❌ すべて英語でやり取りする**：英作文のストレスや、AIからの長文英語の仕様解説を読む手間がかかり、人間の生産性が落ちてしまいます。
* **⭕️ ルール（`CLAUDE.md` や `rules/` のドキュメント等）だけ英語にする**：毎セッション必ず自動送信され続ける「重いファイル群」のみを英語化し、日常のチャット入力や出力結果は日本語に設定します。

人間がダイレクトにやり取りする部分は日本語で快適さを保ちつつ、AIが裏で何度も評価する「不変のシステムファイル」を英語化しておくだけで、長期的なAPI利用において無理のない大幅なコストダウンを実現できます。

## 📝 まとめ

Claude Codeのコスト最適化は、**「AIにいかに無駄なテキストを読ませないか」** という **Context Engineering（文脈構築）** の技術そのものです。

1. **`/clear` と `/compact` で履歴を掃除する**
2. **`.claudeignore` で探索範囲を絞る**
3. **適材適所でモデル (`/model`) を切り替える**
4. **ファイルは `@` で明示的に指定する**

これらを意識するだけで、トークン消費のペースは劇的に下がります。便利さを損なうことなく、快適な「破産しないAIコーディング環境」を手に入れましょう！
