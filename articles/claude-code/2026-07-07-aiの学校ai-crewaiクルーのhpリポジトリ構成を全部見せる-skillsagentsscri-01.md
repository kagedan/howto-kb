私は非エンジニアの一人社長で、AIの学校「[AI Crew（AIクルー）](https://www.ai-crew-school.jp)」を運営しています。そのオウンドメディア（HP＋ブログ＋SNS展開）は1つのgitリポジトリの中でClaude Codeと一緒に運営しています。記事のネタ収集から執筆・公開・計測・リライトまでが、このリポジトリの上で回っています。

「AIで業務を回す」という話は抽象的になりがちなので、この記事では逆に **ディレクトリ構成という一番具体的なレイヤー** から、何をどこに置いているかを全部見せます。

## 全体像

```
ai_crew_hp/                     # Astro製のHP＋ブログ本体と同じリポジトリ
├── CLAUDE.md                   # 常駐ルール（表現ルール・git運用・SSOTの所在）
├── docs/
│   └── 運用方針.md              # ブログ運用のSSOT（運用ループ・コンテンツ戦略）
├── knowledge/                  # ネタ帳。1ファイル=1ノウハウ
│   └── _README.md              # ネタ帳の書式定義
├── src/content/blog/           # 記事本体（Markdown）
├── social/                     # note/Zenn/Qiita/Instagram向けの派生版（gitignore）
├── .claude/
│   ├── skills/                 # 定型作業の手順書（12個）
│   └── agents/                 # レビュー担当のサブエージェント（3体）
└── scripts/                    # 計測・通知のNodeスクリプト（GSC/GA4/Bing/IndexNow/PSI…）
```

ポイントは、**コンテンツもルールも手順書もレビュー体制も計測コードも、全部同じリポジトリに同居している**ことです。AIに「編集部の一員」として働いてもらうには、編集部の道具と規則が全部同じ場所にあるのがいちばん効きます。

## 各ディレクトリの役割

### knowledge/ ― ネタ帳（状態を持つ）

1ファイル=1ノウハウで、frontmatterに `status: ネタ / 記事化済み` を持たせています。ルールは「一次情報だけを書く」。どこかで読んだ一般論を貯めると、記事が競合と似通う原因になるからです。Discordの独り言チャンネルからの取り込みもここに合流します。

状態管理をDBやタスク管理ツールでなく **Markdownのfrontmatter** でやるのがミソで、AIがGrepで「未記事化のネタ一覧」を一瞬で出せます。

### docs/運用方針.md ― 編集方針のSSOT

「運用ループは①貯める→②書く→③出す→④測る→⑤磨くの5段階」「記事末尾のCTAはこう」「キーワード起点＋ターゲットずらし＋一次情報」といった編集方針を1ファイルに集約し、**実態が変わったらその場で書き換える**運用です。AIが迷ったらここに戻る、が徹底できると出力が安定します。

### .claude/skills/ ― 作業手順書

「ブログを書く」「公開する」「Instagramカルーセルに変換する」「週次レビューする」など、**繰り返す作業はすべてスキル（手順書）化**しています。現在12個。手順書には手順だけでなく判断基準（例: リライト候補の条件は「表示多×CTR低×10位以内」）まで書きます。

### .claude/agents/ ― レビュー担当（人格を分ける）

公開前レビューは、書いた本人（メインのClaude）にやらせず、**別コンテキストのサブエージェント**に任せます。ファクトチェック係・公開安全チェック係・画像レビュー係の3体。書いた本人のセルフレビューは人間同様甘くなるので、ツール権限も絞った「別人格」を通します。

### scripts/ ― 決定的な処理はコードに固定する

Search Console・GA4・Bing・IndexNow・PageSpeedへのアクセスは全部Nodeスクリプトです。原則はこうです。

> **毎回同じ結果になるべき処理はスクリプトに固定し、判断が要る処理だけAIにやらせる**

API集計をAIに毎回アドホックに書かせると、微妙に集計条件が変わって週次比較が壊れます。スクリプトはAIに書かせますが、一度動いたら**スキルから呼ばれる固定の道具**になります。

## この構成に落ち着いた設計原則

1. **状態はファイルに持たせる**（ネタのstatus、公開済み管理の表）。AIの記憶やチャット履歴に状態を置かない
2. **ルールはSSOTを1か所に決めて、他からは参照させる**。同じ事実を2か所に書かない
3. **決定的な処理はscripts、判断が要る作業はskills、品質チェックはagents**。この3層に分けると「AIがやるたびに結果が違う」問題がほぼ消える
