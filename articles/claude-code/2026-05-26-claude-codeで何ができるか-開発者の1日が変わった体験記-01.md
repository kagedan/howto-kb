おはようございます。  
SE歴15年。小さな印刷会社の社内SE、40歳。"いえのした"です。  
今回はClaude Codeで開発の1日がどう変わるか紹介します。

## 概要：「コードを書くAI」ではなく「一緒に開発するAI」

Claude Codeはターミナルで動くAIエージェントです。  
プロジェクト全体を理解してファイルの作成・編集・テスト実行・git操作・PR作成まで一貫して行います。

RPGでいえば、コード補完ツールは「便利な装備」、Claude Codeは「パーティーメンバー」。  
装備は開発者の能力を底上げするが、戦うのは自分。  
パーティーメンバーは自分で判断して動き、作業を分担してくれる。

ただ、「すごいらしい」と聞いても具体的に何が変わるのかイメージしにくい。この記事では、実際の開発の1日をBefore/Afterで比較し、Claude Codeが開発フローのどこに効くのかを解説します。

## Before: Claude Code導入前の1日

ごく普通の開発フローです。

```
09:00  Slackで依頼を確認。「ユーザー一覧にソート機能を追加してほしい」
09:15  コードベースを読んで現状を把握。該当ファイルを探す
09:45  実装方針を決める。既存のソート機能があるか調査
10:00  コーディング開始
11:00  テストを書く
11:30  手動で動作確認
12:00  PRを作成。タイトルと説明文を書く
12:15  レビュー待ち
       ─── ここまでで約3時間 ───
```

この間、開発者は「コードを書く」以外にかなりの時間を使っています:

* コードベースの把握（30分）
* 実装方針の調査（15分）
* PRの説明文作成（15分）

これらは「考える作業」ではなく「探す作業」です。

## After: Claude Codeがある1日

同じタスクを、Claude Codeと一緒に進めます。

```
09:00  ターミナルでClaude Codeを起動
       「ユーザー一覧にソート機能を追加したい。
        現状のコードを確認して、実装方針を提案して」

       → Claude Codeが自分でファイルを探し、
         既存のコードを読み、方針を提案する（約2分）

09:05  方針を確認。「その方針でいい。テストから書いて」

       → TDDで実装。テストを先に書き、実装し、
         リファクタリングまで自動で進む（約15分）

09:20  実装結果を確認。動作をチェック

09:30  「PRを作成して」

       → 差分を分析し、タイトルと説明文を自動生成。
         レビューエージェントがコード品質をチェックした上で
         PRを作成する（約3分）

09:35  完了
       ─── ここまでで約35分 ───
```

3時間の作業が35分になった——というのは少し極端な例ですが、「探す作業」の大部分がなくなるのは事実です。

## 変化：何が変わったのか

### 1. 「探す時間」がほぼゼロになる

Claude Codeはプロジェクト内のファイルを自分で読みに行きます。「このファイルはどこ？」「この関数の呼び出し元は？」という調査を、人間の代わりにやります。

巨大なコードベースほど効果が大きい。何万行もあるプロジェクトでも、Claude Codeは必要なファイルを瞬時に特定します。

### 2. コンテキストの切り替えが減る

通常の開発では、エディタ・ターミナル・ブラウザ・ドキュメントを行き来します。  
Claude Codeでは、ターミナル1つで完結します。

```
「このAPIのレスポンス形式を確認して」  → ファイルを読んで回答
「テストを実行して」                  → テストを実行して結果を表示
「PRを作って」                       → git操作からPR作成まで
```

ウィンドウを切り替える回数が減ると、集中力の途切れも減ります。

### 3. 「定型作業」を任せられる

PRの説明文を書く、テストのボイラープレートを書く、gitの操作をする——これらは創造的な仕事ではありません。Claude Codeに任せることで、設計判断や仕様検討など「人間がやるべき仕事」に時間を使えます。

## 変わらないこと：何が変わらないのか

万能ではありません。Claude Codeでも変わらないこと:

* **設計判断は人間がする**: Claude Codeは方針を提案しますが、最終判断は開発者が下します。「この方針でいい」と言うのは人間の仕事です
* **ドメイン知識（業務知識）は教える必要がある**: 業務ロジックの「なぜ」はコードに書いてありません。Claude Codeに伝える必要があります
* **レビューは省略できない**: Claude Codeが書いたコードも、人間のレビューは必要です。AIも普通に間違えます。**完ぺきではありません！**

## 発展：「道具」と「相棒」の違い

Claude Codeを「コードを生成するツール」として使うこともできます。  
でも、それだけでは本来の力を引き出せません。

違いは**カスタマイズ**にあります。

「道具」のままだと毎回同じ説明をする必要があり、品質もセッションによってばらつきます。  
「相棒」にすると、プロジェクトの文脈を理解した上で動くようになる。ゲームでいえば、レベル1の仲間をそのまま連れ回すか、スキルを覚えさせて装備を整えるかの違いです。

さらに踏み込むと、公式のカスタマイズ機構の上に独自の仕組みを積み上げることもできる。  
セッションを超えて知見を蓄積するシステム、実装前に方針の妥当性を検証するガード、寝ている間にissueを消化する自律パイプライン——  
こうした仕組みは公式機能を組み合わせて自作したもの。

このシリーズでは、公式機能の基本からこうした独自設計まで段階的に紹介していきます。

**準備：始めるのに必要なもの**

Claude Codeを試すのに必要なのは以下だけです:

1. **Anthropicアカウント**: [claude.ai](http://claude.ai) でサインアップ
2. **Pro/Maxプラン**: Claude Codeの利用にはサブスクリプションが必要（月額$20/$100/$200）
3. **ターミナル**: macOS / Linux / Windows（WSL）対応
4. **インストール**: `npm install -g @anthropic-ai/claude-code`

起動は `claude` と打つだけ。プロジェクトディレクトリで実行すれば、そのプロジェクトのコンテキストで動きます。

## まとめ

Claude Codeは「コードを書くAI」ではなく「一緒に開発するAI」です。  
ファイルの探索、テスト実行、PR作成まで一貫して行い、開発者は設計判断と最終確認に集中できます。

ただしデフォルト設定のままでは「便利な道具」止まり。  
カスタマイズすることで、プロジェクトの文脈を理解した「開発の相棒」になります。
