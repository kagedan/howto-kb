## 「とりあえず入れた」で止まっていないか

WIREDが「Claude Codeを"追いかける立場"となったOpenAI」と報じたように、2026年5月時点でのAIコーディングエージェント競争において、Claude Codeはすでに業界標準の一角を占めています。Uravationが開催したClaude Code実践編では受講者133名・満足度96%を記録し、「全体像が見えた」と答えた受講者は6割に上りました。

裏を返せば、**残り4割は「まだ全体像が見えていない」**ということです。

MITテクノロジーレビューは「たった1年でソフトウェア開発が激変した」とClaudeの影響力を総括していますが、現場でClaude Codeを使い込んでいる側からすると、「インストールはできたけど、何となく使っている」という状態で止まっているケースが非常に多いと感じます。

この記事では、Claude Codeをインストールした後に**9割の人が躓くポイント**と、実際の開発現場で積み上げてきた**7つの実践パターン**を、具体的なコマンドや設定例とともに解説します。

---

## まず押さえる：Claude Codeとは何者か（2026年5月時点）

Claude Codeは、Anthropicが提供するCLIベースのAIコーディングエージェントです。ターミナルから起動し、ファイルの読み書き・コード生成・テスト実行・Gitコミットまでを自律的にこなします。

2026年5月現在、内部で使用されるモデルは**Claude Sonnet 4**（claude-sonnet-4-20250514）がデフォルトです。タスクの複雑度に応じてClaude Opus 4への切り替えも可能で、モデル指定は以下のように行います。

```
claude --model claude-opus-4-20250514
```

インストール自体はnpmで1行です。

```
npm install -g @anthropic-ai/claude-code
```

ただし、「インストールできた」と「使える」の間には、思った以上に大きな溝があります。

## 実践パターン①：CLAUDE.mdを「設計書」として書く

Claude Codeがプロジェクトを読み込む際の最初の参照先が、リポジトリルートに置くCLAUDE.mdです。ここが貧弱だと、AIは毎回同じ文脈説明をされるまで動けない状態になります。

**落とし穴：** 多くの人がCLAUDE.mdに「このプロジェクトはNext.jsです」程度しか書いていない。

**ベストプラクティス：** 以下の4項目を必ず含めること。

```
# プロジェクト概要
受発注管理SaaS。Next.js 15 App Router + Supabase + TypeScript 5.4構成。

# コーディング規約
- コンポーネントはsrc/components/配下、命名はPascalCase
- APIルートはsrc/app/api/配下
- エラーハンドリングはResult型パターンで統一

# 禁止事項
- any型の使用禁止
- console.logをコミットに含めない

# よく使うコマンド
- ローカル起動: npm run dev
- テスト: npm run test:watch
- 型チェック: npx tsc --noEmit
```

このCLAUDE.mdの質が、そのままClaudeの出力品質に直結します。

## 実践パターン②：タスクの粒度を「30分で終わるサイズ」に分解する

Claude Codeに「ECサイトの注文機能を作って」と投げると、途中で迷走します。これはClaudeの問題ではなく、**タスク設計の問題**です。

AWSが報告したキヤノンITソリューションズの事例（3ヶ月で開発スピード3倍）でも、AI Coding Agentの導入効果を最大化したのは「タスク分解の仕組みづくり」だったとされています。

**効果的な指示の粒度：**

```
# NG（大きすぎる）
「ユーザー管理機能を実装して」

# OK（適切なサイズ）
「src/app/api/users/route.tsにGETエンドポイントを追加する。
Supabaseのusersテーブルから全件取得し、
{ id, email, createdAt }の形で返す。
既存のauthMiddlewareを使って認証チェックを入れること。」
```

さらに実践的なのは、**大きなタスクをClaudeに分解させる**アプローチです。

```
claude "この要件をステップごとのタスクリストに分解して。
実装は始めずにリストだけ出力して。" < requirements.md
```

分解されたリストを確認してから、1タスクずつ実行させます。

## 実践パターン③：--continueフラグで文脈を引き継ぐ

Claude Codeのデフォルトは毎回新しいセッション開始です。連続した作業ではーcontinueを使い、直前のセッションの文脈を引き継ぎます。

```
# 前回の続きから作業を再開
claude --continue

# または特定の会話IDを指定
claude --resume <conversation-id>
```

**実務でのユースケース：** 「昨日途中まで進めたリファクタリングの続き」「エラーが出て止まったので、前回の状態から別のアプローチで再試行」

会話IDは~/.claude/conversations/配下に保存されています。長期プロジェクトでは週次でIDをメモする習慣をつけると、文脈の断絶を防げます。

## 実践パターン④：MCPサーバーを接続して「知識の壁」を越える

Claude Codeは単体でも強力ですが、**MCPサーバー（Model Context Protocol）** と組み合わせることで別次元の使い方ができます。

設定は~/.claude/config.jsonに記述します。

```
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxxxxxxxxxx"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "postgresql://..."
      }
    }
  }
}
```

GitHubのMCPサーバーを繋ぐと、「先週マージされたPRの中で、認証関連の変更を要約して」という自然言語での指示が可能になります。PostgreSQLのMCPを繋げれば、「usersテーブルのスキーマを見て、適切なTypeScript型定義を生成して」が一発で動きます。

## 実践パターン⑤：/doctorコマンドで環境を定期チェック

Claude Codeが思ったように動かないとき、まず疑うべきは**環境の不整合**です。

```
claude /doctor
```

このコマンドで、APIキーの有効性・モデルへのアクセス権・ネットワーク疎通・CLIのバージョンが一括チェックされます。

よくある詰まりパターン：

* ANTHROPIC\_API\_KEYが古いキーのまま（更新後にシェルを再起動していない）
* npmのキャッシュが古いバージョンを参照している → npm cache clean --force後に再インストール
* 社内プロキシ環境でのSSL証明書エラー → NODE\_EXTRA\_CA\_CERTSの設定が必要

## 実践パターン⑥：Cursorと「役割分担」して使う

Claude CodeとCursorは競合ではなく**補完関係**です。現場での使い分けはシンプルです。

```
| シーン | 推奨ツール |
| 既存ファイルの部分修正・インライン補完 | Cursor |
| 複数ファイルにまたがるリファクタリング | Claude Code |
| 新機能の0→1実装(ファイル新規作成含む) | Claude Code |
| コードレビューの補助・説明生成 | Cursor(CMD+K) |
| テストコードの一括生成 | Claude Code |
| ペアプロ的な対話 | Cursor(Chat) |
```

CursorのエージェントモードにもClaude Sonnet 4が使えるため混乱しがちですが、**ファイルシステムへの自律的な書き込みが多いタスクはClaude Code、IDE上のインタラクティブな作業はCursor**と覚えておくと判断が速くなります。

## 実践パターン⑦：コストを可視化して「使いすぎ」を防ぐ

Claude Codeはトークン消費が想定より速い。意識しないとAPIコストが月単位で跳ね上がります。

```
# 今月のトークン使用量を確認
claude /usage

# セッション単位のコスト確認
claude /cost
```

**コスト最適化のTips：**

* 定型タスク(テストコード生成・型定義生成)は**Claude Haiku 3.5**を指定して単価を下げる
* ファイル丸ごとの読み込みではなく、関連関数だけを--add-fileで渡す
* CLAUDE.mdを充実させることで「確認の往復回数」が減り、結果的にトークンが節約される

大量のファイルを一気に読ませる「全部渡してみよう」アプローチは、コストと精度の両面でマイナスになることが多いです。

## まとめ：「インストールした日」より「設計した日」が起点

Claude Codeの本当の立ち上がりは、インストールした日ではなく、**CLAUDE.mdを書いてMCPサーバーを繋いだ日**です。

日経クロストレンドが「GeminiやChatGPT圧倒」と評したClaudeの強さは、指示の精度に対する忠実さにあります。その強さを引き出すのは、ツール側の機能ではなく、**使う側の設計力**です。

まず今日やること：

1. npm install -g @anthropic-ai/claude-codeでインストール
2. プロジェクトルートにCLAUDE.mdを作成(上記テンプレートを参考に)
3. 一番小さい「30分タスク」を1つ投げてみる

ここから始めてみてください。
