お疲れ様です、都内でエンジニアをしているshingoです。

今回は、AIエージェント時代の開発のベストプラクティスとも言える「スペック（仕様）駆動開発」について、その考え方と具体的な進め方を解説します。

## この記事でわかること

* AI（Claude Code / Codex等）を使った開発で「手戻り」が発生する根本的な原因と対策
* AIと齟齬なく合意を形成するための「What」と「How」の2軸フレームワーク
* 今すぐあなたのプロジェクトに導入できる `CLAUDE.md` の基本テンプレート
* AIのコンテキストを節約し、開発をコントロールする「永続ドキュメント」と「ステアリング（作業）ドキュメント」の分類運用

## なぜこのテーマが重要なのか

近年、Claude CodeやCodex、Cursor、Geminiといった強力なAI開発ツールが登場し、ソフトウェアの開発スピードは劇的に向上しました。

僕の働いている会社でも、AIを使うことでPRの数がこれまでの2倍以上になっているチームは当たり前です。

しかし、同時に多くのエンジニアが以下のような悩みに直面していると思います。

* AIに『〜の機能を作って』と頼んだら、既存のコードを破壊された
* 何度も修正を繰り返しているうちに、AIのコンテキストが肥大化して動きが遅くなった
* リポジトリの設計思想を無視した、おかしなアーキテクチャのコードが書かれてしまった

これらが発生する原因はすべて同じです。「AIにいきなりコードを書かせているから」です。

どれだけ優秀なAIであっても、仕様やアーキテクチャの合意がないまま実装を丸投げすれば、手戻りが発生するのは当然です。

人間同士の開発でも、仕様書なしでいきなりコードを書くことはありません。AI相手でも同じことです。

AI時代の開発現場だからこそ、コードを書く前に「仕様」と「設計」のスペックを明文化し、AIと合意を形成する「スペック（仕様）駆動開発」が重要になります。

## まず結論

AI時代の開発における標準作法は、「コードを書く前に、AIとドキュメントで合意を形成する」ことです。

具体的には、プロジェクト全体の指針となる「永続的ドキュメント（`docs/`）」と、そのタスク限りの舵取りをする「ステアリングドキュメント（`.steering/`）」を明確に切り分けます。そのうえで、段階的にAIと合意を形成しながら実装を進めていきます。

この仕組みをリポジトリ内に定義し、AIに開発ルールを叩き込むために、「AIへの行動規範ファイル」としてお馴染みの`CLAUDE.md`（Anthropic Claude Code）や `AGENTS.md`（OpenAI Codex）を活用します。

## 基本の考え方

AIとスムーズに開発を進めるために、僕たちが理解しておくべき基本コンセプトは大きく3つあります。

### 1. 合意形成の「2つの軸」

AIに指示を出すとき、僕たちは以下の2つの軸「What と How」で合意を形成する必要があります。

### 2. 永続的ドキュメントとステアリングドキュメント

AIとの合意内容を記録するドキュメントは、以下の2種類に分けて管理します。

* **永続的ドキュメント（`docs/`）**：プロジェクト全体の仕様。プロダクト要求定義、機能設計、アーキテクチャ、用語集など、すべてのタスクで参照され続ける基本仕様です。
* **ステアリングドキュメント（`.steering/`）**：特定タスク限りの「舵取り」。タスクごとの要件・設計・ToDoを管理し、作業完了後は履歴として残します。

この分類によって、AIが参照すべきコンテキストが整理され、長期的な設計意図と短期的なタスク指示が混在するのを防ぎます。

実際のリポジトリでは、以下のようなディレクトリ構成になります。あくまで例です。

```
my-project/
├── CLAUDE.md                          # AIの行動規範（開発ルール定義）
├── docs/                              # 永続的ドキュメント（プロジェクトの憲法）
│   ├── product-requirements.md        #   プロダクト要求定義書
│   ├── functional-design.md           #   機能設計書
│   ├── architecture.md                #   技術仕様・インフラ構成
│   ├── repository-structure.md        #   リポジトリ構造定義
│   ├── development-guidelines.md      #   コーディング規約・開発標準
│   └── glossary.md                    #   用語集（ユビキタス言語）
├── .steering/                         # ステアリングドキュメント（タスク単位の舵取り）
│   ├── 20260601-ユーザー認証/           #   完了済みタスク（履歴として残す）
│   │   ├── requirements.md
│   │   ├── design.md
│   │   └── tasklist.md
│   └── 20260615-決済機能/              #   現在進行中のタスク
│       ├── requirements.md
│       ├── design.md
│       └── tasklist.md
└── src/                               # ソースコード
    └── ...
```

### 3. 段階的な合意形成

ドキュメントの作成は、必ず1ファイルずつ行います。1つのドキュメントが完成するたびに人間が内容を確認・承認し、認識をすり合わせてから次に進みます。

AIに一度にすべての設計を書かせると、前提がズレたまま大量のドキュメントが生成され、修正コストが跳ね上がります。段階的に合意を積み上げるこのアプローチが、手戻りの発生源を根本から絶つ鍵です。

## `CLAUDE.md` の基本テンプレート

開発プロジェクトで今すぐスペック駆動開発を始められるよう、`CLAUDE.md`の基本テンプレートを用意しました。

リポジトリのルートに配置する `CLAUDE.md` は、AI（特にClaude Code）が読み込む「行動規範」です。Anthropicの公式ドキュメントでも、`CLAUDE.md` はプロジェクト固有の指示をAIに与えるための主要な仕組みとして位置付けられています。ここにドキュメント管理ルールと開発プロセスを明文化しておくことで、AIは自動的にこのプロセスを遵守するようになります。

```
# CLAUDE.md

## 概要
本プロジェクトにおける開発標準ルール、ドキュメント管理、および「スペック駆動開発（SDD）」のプロセスを定義します。

## プロジェクト構造
本リポジトリは、仕様変更やコード追加の前に必ず設計仕様を作成し、合意を形成する「スペック駆動開発（SDD）」を採用しています。

### ドキュメントの分類
#### 1. 永続的ドキュメント (`docs/`)
- `product-requirements.md` - プロダクト全体としての要求定義書
  - プロダクトビジョンと目的
  - ターゲットユーザーと課題・ニーズ
  - 主要な機能一覧
  - 成功の定義
  - ビジネス要件
  - ユーザーストーリー
  - 受け入れ条件
  - 機能要件
  - 非機能要件

- `functional-design.md` - 画面遷移やデータモデル等の機能設計書
  - 機能ごとのアーキテクチャ
  - システム構成図
  - コンポーネント設計
  - ユースケース図、画面遷移図、ワイヤフレーム
  - API設計

- `architecture.md` - インフラ構成や全体技術仕様書
  - 技術スタック
  - 開発ツールと手法
  - 技術的制約と要件
  - パフォーマンス要件

- `repository-structure.md` - リポジトリ構造定義書
  - フォルダ・ファイル構成
  - フォルダの役割
  - ファイル配置ルール

- `development-guidelines.md` - コーディング規約や型定義、開発標準ライン
  - コーディング規約
  - 命名規則
  - スタイリング規約
  - テスト規約
  - Git 規約

- `glossary.md` - チーム内の共通言語（ユビキタス言語）定義
  - ドメイン用語の定義
  - ビジネス用語の定義
  - コード上の命名規則

#### 2. 作業単位のドキュメント (`.steering/[YYYYMMDD]-[開発タイトル]/`)
特定の開発タスクにおける設計とタスク管理を行うための、一時的なステアリングドキュメント。
作業完了後は履歴として残し、新しい開発では新しいディレクトリを作成します。
- `requirements.md` - 今回のタスクの要求（目的、受け入れ条件）
  - 変更・追加する機能の説明
  - ユーザーストーリー
  - 受け入れ条件
  - 制約事項

- `design.md` - 今回の変更箇所の設計（DB、API、UIへの影響）
  - 実装アプローチ
  - 変更するコンポーネント
  - データ構造の変更
  - 影響範囲の分析

- `tasklist.md` - 細分化された実装タスクのリスト（ToDo）
  - 具体的な実装タスク
  - タスクの進捗状況
  - 完了条件

## 開発プロセス
開発を開始する際、AIおよび開発者は必ず以下のプロセスを遵守してください。

1. **影響分析**
   - 既存の `docs/` 配下のドキュメントを確認し、基本設計に影響するか確認する。

2. **ステアリングディレクトリの作成**
   - `.steering/[YYYYMMDD]-[開発タイトル]` ディレクトリを作成する。

3. **作業ドキュメントの段階的作成**
   - 以下の順にファイルを作成し、各段階でAIと開発者の間で合意（承認）を得てから次のファイルへ進む。
     1. `.steering/[YYYYMMDD]-[開発タイトル]/requirements.md` の作成と合意
      2. `.steering/[YYYYMMDD]-[開発タイトル]/design.md` の作成と合意
      3. `.steering/[YYYYMMDD]-[開発タイトル]/tasklist.md` の作成と合意

4. **永続的ドキュメントの更新**
   - 今回の開発によってプロダクトの基本仕様に変更があった場合、対応する `docs/` 配下のファイルを更新する。

5. **段階的実装**
   - 合意した `tasklist.md` に基づき、タスクを1つずつステップバイステップで実装する。
   - 1つのタスクが完了するごとに動作確認とテストを行い、`tasklist.md` を更新する。

6. **品質チェック**
   - 実装完了後、リント (`npm run lint`)、型チェック (`npm run typecheck`)、およびテスト (`npm run test`) を必ず実行する。
```

あくまで基本の型なので、これをカスタマイズしていくイメージです。

## スペック駆動開発の進め方

具体的にスペック駆動開発の進め方について説明します。

スペック駆動開発は、大きく分けて**プロジェクトの土台を作るフェーズ（初回のみ）** と **個別タスクを実装するフェーズ（毎回のサイクル）** の2段階で構成されています。

### フェーズ 1：プロジェクトの土台を作る（初回のみ）

新しいプロジェクトを始める際、もしくは既存プロジェクトにスペック駆動開発を初めて導入する際に、一度だけ行う準備ステップです。

#### 1. 何を作るか、アイデア出し

まずは「何を作るか」のブレインストーミング（アイデア壁打ち）です。ここはAIの一番の得意分野なので、良い壁打ち相手になってくれると思います。

頭の中にあるモヤモヤしたアイデアや課題感をAIに語りかけることで、対話を重ねながら作るべき機能の解像度を上げていきます。

AIから「それならこういうユースケースも考えられますね」「こういった技術的な懸念もあります」とアイデアを引き出してもらい、言葉を具体化しながら輪郭をシャープに絞り込んでいきます。

#### 2. 永続ドキュメント（`docs/`）を段階的に作成

アイデアが固まったら、仕様の「What」を定義する `docs/product-requirements.md` から作成を始めます。

ここでも人間がゼロからMarkdownを書く必要はありません。壁打ちしたスレッドの文脈をそのまま使い、AIに「これまでの議論をベースに、docs/product-requirements.md を作成して」と依頼するだけです。すでにドメインが決まっている場合は、資料をインプットとしてAIにMarkdownに落とし込んでもらうと良いです。

AIは `CLAUDE.md` で定義された「永続的ドキュメントの分類や役割」をすでに頭に入れているため、指示されるだけでビジョンや主要機能一覧、受け入れ条件などを網羅した綺麗なドキュメントを一瞬で組み立ててくれます。人間は出来上がったドキュメントに過不足がないかを最終レビューするだけで完了です。

要求定義書が固まったら、機能設計（`functional-design.md`）や用語集（`glossary.md`）など、残りの永続的ドキュメントも同様に **1ファイルずつ** 作成していきます。

#### 3. `CLAUDE.md` にルールを明文化

永続ドキュメントが揃ったら、プロジェクトのルートに `CLAUDE.md` を配置し、ドキュメント管理ルールと開発プロセスをAIの行動規範として明文化します。先ほど紹介したテンプレートをベースにカスタマイズしてください。

### フェーズ 2：個別タスクの実装サイクル（毎回）

土台ができたら、いよいよ個別の開発タスクを進めていきます。以下の6ステップを毎回繰り返します。

すでに `CLAUDE.md` に「スペック駆動開発のプロセス」が明文化されているため、AIに「〜の機能を実装して」と指示を出すだけで、AIはいきなりコードを書くのではなく、このプロセスに沿って動き始めます。

#### Step 1：影響分析

`docs/` 配下の永続的ドキュメントを確認し、今回の変更がプロジェクト全体のアーキテクチャやデータベース設計に影響を与えるかを分析します。もし影響がある場合は、まず永続的ドキュメントの更新からAIと議論します。

#### Step 2：ステアリングディレクトリの作成

今回のタスク専用のステアリングディレクトリを作成します。

```
mkdir -p .steering/$(date +%Y%m%d)-[開発タイトル]
```

#### Step 3：作業ドキュメント（スペック）の作成と段階承認

AIと一緒に、以下の3つのファイルを順番に作成していきます。ここが一番重要です。「1ファイル作るごとにAIに内容を確認させ、合意（承認）を得てから次へ進む」という段階的アプローチを徹底します。

1. **`requirements.md`（要求内容）**
   * 今回作るものの「What（目的、受け入れ条件、今回やらないこと）」を明文化します。
2. **`design.md`（設計仕様）**
   * 今回の変更の「How（画面・API・DBへの具体的な影響、エッジケース）」を記述します。
3. **`tasklist.md`（タスクリスト）**
   * 実装手順をステップバイステップ（1つのタスクが数分〜10分程度で終えられる粒度）に分解したToDoリスト。

AIから「タスクリストを作成しました。問題ないか確認をお願いします」と提案されるので、人間が内容を確認・レビューし、問題がなければ「これで実装を進めて」と承認します。

#### Step 4：永続ドキュメントの更新

今回追加した仕様のうち「今後も永続的に参照すべき基本仕様」があれば、`docs/` 配下の永続的ドキュメントを更新します。

明示的にルール化しているので、負債あるある「人間がドキュメントを更新し忘れる」という心配がありません。

#### Step 5：実装開始

合意した `tasklist.md` に基づき、AIに指示を出して実装を進めます。

「タスク1を実装し、テストを書いて動作確認してください。」と指示すれば、タスクの進捗管理もAIが自動で行ってくれます。

#### Step 6：品質チェック

実装がすべて完了したら、リント、型チェック、テストを必ず実行して品質を確認します。

問題なければ、ステアリングファイルの内容をマージしてクローズします。

## まとめ

AI開発における最大の手戻りは、人間とAIとの間の「前提のズレ」から生まれます。

「スペック駆動開発」は、そのズレをコードが書かれる前にドキュメントによって検知し、安全に最短ルートでゴールに到達するための羅針盤です。

高品質な製品を作るためには、バイブコーディングのようにAIに操られるのではなく、僕たち人間がドキュメントを通じてAIを強力にリードし、クリーンな開発フローを確立する必要があります。

まずは、次回の小さなタスクから、`.steering/` フォルダを1つ作って始めてみるのはいかがでしょうか。

本当に差がつくのは、コードの書き方ではなく、AIとの合意の作り方です。
