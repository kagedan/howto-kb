# ハーネスエンジニアリングって何よ！

ハーネスエンジニアリングとは、AIの出力を改善するためにドキュメントなどの周辺環境の整備を行うことです。

具体的にどんなことをすればいいんだい？ってことで  
Claude Code 用に作った `.claude/` 配下にいろんなリポジトリで使える汎用的なドキュメントを作ったので紹介します

# 作ったもの

```
claude-rules/
├── CLAUDE.md              # プロジェクトコンテキストの雛形
├── settings.json          # permissions / hooks
├── skills/                # skills（init / draft / debug / refine / review / commit）
├── rules/                 # coding.md / git.md
├── hooks/                 # session-context.sh / format.sh
└── scripts/setup.sh       # 任意プロジェクトの .claude/ に配布する
```

`scripts/setup.sh` を叩けば、対象プロジェクトの `.claude/` に一式が展開され、`CLAUDE.md` のテンプレ変数（プロジェクト名・スタック・主要ディレクトリ・実行コマンド）は `package.json` などから自動で埋まるような仕組みにしています

## フォルダ構成の意味

| 場所 | 役割 |
| --- | --- |
| `CLAUDE.md` | プロジェクトの常時コンテキスト（雛形） |
| `settings.json` | permissions（許可コマンド）と hooks の設定 |
| `skills/` | スラッシュコマンド（`/init` `/draft` `/debug` `/refine` `/review` `/commit`） |
| `rules/` | 詳細な規約。`coding.md`（コーディング規約）と `git.md`（Git・PR 規約） |
| `hooks/` | 自動実行スクリプト。`session-context.sh`（起動時に git 状況を注入）/ `format.sh`（編集後フォーマット） |
| `scripts/setup.sh` | この `.claude/` 一式を他プロジェクトへ配布するスクリプト |

## 4層で出力を安定化させる

```
1層目　CLAUDE.md   → 常に読む「前提知識」（誰が・何を・どう作る）
2層目　rules/      → 細かい規約の本体（CLAUDE.md から参照される）
3層目　skills/     → 定型作業を1コマンドで呼び出す（/debug など）
4層目　hooks/      → 人が呼ばなくても自動で走る処理
```

`CLAUDE.md` が「概要と入口」、`rules/` が「規約の詳細」、`skills/` が「作業の型」、`hooks/` が「自動化」——という4層にすることで、Claudeちゃんの出力の精度を安定化させます

プロンプトアーキテクチャとも言われるらしいです

# どうやって作ればいいんだ！

## 1. 普段AIに頑張ってもらってることを考える

自分が Claudeちゃんにお願いしていることを棚卸しします

| 普段やらせていること | 毎回つけている注文 |
| --- | --- |
| 機能開発 | 計画 → 承認 → 実装の順で頑張れ。実装は承認だしてからね |
| バグ修正 | 対症療法禁止。根本原因まで特定して |
| コミット | 命令形・WHY 中心・push はするな |
| レビュー | 実装したコードでやばいところない？ |

これらのタスクをskillにしました。それらのskillの共通項がrulesやCLAUDE.mdになるイメージです。

claudeにskillにしてって言えば叩き台を作成してくれます。ありがたい！

## 2. 改善する作業も skill にする

手作業で毎回skillを更新するのは大変です

その改善作業もまた繰り返しなので、Claudeとのsessionを振り返って、そこから自動でskillの修正案を提案してくれるskillを作りました（`/refine`）

AIがAIを育てるといった感じですね！

# 所感
