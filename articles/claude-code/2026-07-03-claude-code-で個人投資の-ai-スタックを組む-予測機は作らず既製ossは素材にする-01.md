Claude Code で個人投資のツールを組むとき、私は「株価を当てる AI」を作りませんでした。金融のマルチエージェントは、いまや clone すれば動く時代です。数万スター級の OSS が公開されていて、私も中身を読んでみました。でも、そのままは使いませんでした。

前の3本では自己理解の話（日記・ペルソナ・補正ルール）を書きました。この記事は領域を変えて、お金——個人投資のツールを、どう組んで自分用に仕立てているかの全体像です。やっていることは自己理解のときと同じで、**引っ張ってきて、自分の要件に合わせて仕立て直す**、それだけです。

私が作ったのは、市場データの取得基盤を1つ置き、その上に「攻め」と「守り」を別々のアプリとして載せる三層構成です。この記事では、既製ツールをそのまま使わずになぜこう組んだのか、を書きます。

> 前提となる全体像はこちら：[リポジトリは人間のために書かない — Claude Code で日記・投資・育児まで運用する【全体像】](https://zenn.dev/takotyan00/articles/claude-code-life-repository)

## 「clone すれば動く」時代のマルチエージェント

まず前提として、投資向けのマルチエージェントは、もう思想の発表段階ではありません。実物の OSS が公開されていて、clone すれば手元で動きます。私が中身を読んだのは、たとえばこのあたりです。

どれも `clone` して依存を入れれば動きます。**ゼロから車輪を再開発する必要は、もうありません。** これは前提として認めるところから始めます。

ただ、中を読むと共通の但し書きがありました。**いずれも「研究・教育用途」と明記されていて、そのままでは私の実際の保有とは連動しません。** つまり「よくできたデモ」であって、「私の投資判断そのもの」ではない。ここが出発点でした。

## でも「そのまま使う」のも、たぶん違う

車輪の再開発が不要なら、既製をそのまま使えばいい——とはなりませんでした。汎用ツールは、当たり前ですが**私の要件を知らない**からです。

私が投資に対して最初に決めた前提——設計上ゆずらない「憲法」——は、2つあります。

1. **株価の予測機は作らない**。当てにいくのではなく、「なぜこの構造変化に賭けるのか」という確信のロジックと、その確信が崩れたら気づける反証の仕組みを作る。
2. **判断は最後まで人間（私）が持つ**。AI は材料を出す係で、仮説を自動確定させない。

役割エージェントを並べて売買シグナルを吐く汎用ツールは、この憲法とそもそも向いている方向が違います。だから発想を、こう整理しました。**車輪の再開発はもう要らない**——動く OSS がある。でも、そのまま飲み込むのも凡庸で、汎用ツールは私の実保有も目標も税も制度も、私の憲法も知りません。だから最適解は、引っ張ってきて選んで削って、自分の憲法に通すことでした。

この「選んで、自分用に仕立てる」は、実は自己理解クラスタ（日記・ペルソナ・補正ルール）でやっていたことと同じ骨格です。汎用の仕組みを、自分という個体の要件に通す。お金はその2例目にすぎません。

## 私の三層設計（素材は1か所・判断は各アプリ）

そのうえで組んだのが、次の三層です。真ん中に思想を持たない**素材の基盤**を1つ置き、その上に思想の違う2つの判断アプリを載せます。

設計の軸は一言でいうと、**「素材は1か所、判断は各アプリ」** です。

株価・財務・配当・適時開示といった生データの取得は、1つの基盤に集約しました（データ元は J-Quants・TDnet・yfinance・SEC EDGAR といった無料のソースです）。この層は**スクリーニングの閾値も仮説も買い場の判断も一切持ちません**。取得・正規化・保持・配信だけをやる中立レイヤーです。取得に失敗しても本流を止めない（失敗を例外でなく値として扱う）設計にして、無料データソースの不安定さを基盤の中で吸収します。

その上に、思想の異なる2つのアプリを **read-only** で載せます。素材を共有しつつ、判断ロジックはお互いに独立している。この分離があるおかげで、攻めと守りで正反対の思想を、矛盾なく同居させられます。

| 層 | 役割 | 中で持つもの / 持たないもの |
| --- | --- | --- |
| 市場データ基盤 | 素材の一元取得 | 生データを持つ。判断は持たない |
| 攻め | 構造変化に賭ける確信づくり | 仮説・影響マップ・反証トラッキング |
| 守り | 高配当を淡々と積む | 割安判定・減配リスク・買い場検出 |

## 攻め：株価の予測機を作らない

攻めのアプリは、NVIDIA のような「構造変化に乗った勝者」を、周りが気づく前に見つけて、数年単位で握り続けることを狙っています。

ここで憲法が効きます。作ったのは**株価予測機ではありません**。やっているのは2つです。

* **確信を作るロジック**：政治・経済・技術・地政学といった構造変化から、「なぜこの会社が勝つのか」という仮説を組み立てる。粗利率・顧客の質・再投資の効き・オーナー性といった、株価とは独立した決定的な指標で候補を絞る。
* **握力を支える反証トラッキング**：一度賭けた仮説が崩れていないかを継続的に監視する。「増収率が前年同月比で2四半期続けて失速したら、成長が続くという前提が崩れた候補として通知する」——このように、崩れる条件を具体的なルールに落として見張ります。

面白いのは、この2つ目です。たとえば私が中を読んだ TradingAgents は、判断のあとの反省を「結果（株価など）が当たったか」に紐づけています。でも私のアプリは、反省の教師信号を**株価ではなく構造の推移**に置いています。株価は当てにいかないと決めたので、当たり外れで学習させると憲法に反するからです。ここは、既製ツールから移せなかった（移してはいけなかった）ところでした。

そして、仮説を最終的に確定させるのは私です。AI は材料と反証候補を出しますが、賭けるかどうかの判断は人間が持つ。いわゆる me-in-the-loop の構成にしています。

## 守り：あえて LLM を使わない

守りのアプリは、攻めとは正反対です。高配当株を淡々と積み増して、配当収入を目標額まで育てる。派手さはなく、退屈なくらい機械的に回すのが正解の領域です。

だからこのアプリでは、**あえて LLM を一切使っていません。** 全部、確定的な財務ルールで動かしています。

守りで一番こわいのは「高配当の罠」——利回りは高いのに、実は業績が悪化していて、いずれ減配する銘柄です。これを見分けるために、配当性向・連続非減配・ROE・売上トレンド・キャッシュフローのカバー率・同業比の利回りといった複数の軸で罠を検出します。ここに LLM の曖昧さを入れると、**同じ入力から日によって違う答えが出て、淡々と積むという目的そのものが壊れます。** 決定性が要る領域では、賢いモデルより、ぶれないルールのほうが正しい。

「AI スタック」と言いながら守りに LLM を入れないのは、矛盾ではありません。**どこに知能を効かせて、どこをあえて機械的に固めるか**を、領域の性質で分けている。これも「自分の要件に合わせて仕立てる」の一部です。

## 外から借りると決めたのは、たった1つ

では、数万スター級の既製マルチエージェントから、私は結局何も借りなかったのか。そうではありません。込み入った比較をしたうえで、**借りる価値があると判断したものが1つだけ**ありました。

それは **bull（強気）と bear（弱気）を対決させる敵対ステップ**です。単発の LLM に銘柄を分析させると、強気材料に弱気材料をなんとなく溶接して、全体としてポジティブな結論に寄せがちです。そこに「弱気側だけを担当する役」を強制的に立てると、都合の悪いケースが表に出てくる。私の憲法にある「反証」と、この敵対ステップは相性がいい。

ただし借りるのは**構造だけ**です。既製ツールが持っている「株価の方向を当てにいく」思想（トレーダー層や、株価で学習する反省ループ）は、私の憲法に反するので捨てました。敵対ステップも、売買シグナルやポジションサイズは出させず、**反証候補を人間に渡す任意入力**に位置づけ直す設計にしています。

正直に書くと、これはまだ設計メモ（提案）の段階で、実装はこれからです。開発のタイミングで入れる予定で、いまは「借りるならこの1つだけ、しかも思想は捨てて構造だけ」という判断が固まっている状態です。

この「1つだけ・構造だけ・自分の憲法に通してから」が、**そのまま使うのではなく素材として使う**、ということの具体です。

## Claude Code が「仕立てる」相棒になる

この一連——既製を評価し、借りるものを1つに絞り、思想を捨てて構造だけ抜き、自分の憲法に通してコードに落とす——を、私は Claude Code とやっています。

とくに効いたのは、既製 OSS と自分のアプリを**思想・計算・出力まで込みで比較する**ときでした。表面の説明だけで「似ている / 違う」と判断すると、AI はそれらしい対応表を作ってしまいます。そこで実際のコードを読ませて裏を取り、憶測のマッピングは却下する。この「一次情報で確かめる」進め方ができる相棒だと、選別の精度が上がります。

引っ張ってくる → 自分の要件で組み替える → 自分の憲法に通す。この反復の相棒が Claude Code だ、というのが、傘テーマ「Claude Code で人生を運用する」のお金版です。

## 正直なところ

いくつか、線を引いておきます。

まず、私は金融工学の最深部では勝負していません。私が掴んでいるのは「実用に必要な分」までで、価値は理論の深さではなく、攻め・守り・素材基盤を横断して束ねる設計のほうにあります。ここは前の記事群と同じ立場です。

それから、この記事は**設計と選び方の話**であって、コピペで動く実装を配る話ではありません。攻めと守りのアプリは、実際の保有・税区分・制度といった私に固有の泥臭い部分と結びついているので、コードそのものは公開しません（実保有データは公開しない、という線引きは全体像の記事で書いたとおりです）。配れるのは、三層の分け方と、「そのまま使わず素材にする」という選び方だけです。

最後に、借りると決めた敵対ステップはまだ実装していません。設計として「これだけは借りる」と決まっているだけで、動いているものとして書いているわけではない、という点は正直に。

## なぜ、こう組むのか

理由は、自己理解の記事に書いたのと同じです。

いまは、汎用のツールも思想も、大量に用意されて手に入る時代です。だからこそ、**そこから自分に合うものを選び、自分用に削って仕立て直せる人**が果実を取れる。ゼロから作る必要はないけれど、出来合いをそのまま飲み込むのも凡庸で、その中間——選んで、仕立てる——にいちばん価値がある。

お金でやったのは、この方法の2例目です。既製の投資マルチエージェントを素材として評価し、借りるものを1つに絞り、自分の憲法（予測機を作らない・判断は人間が持つ）に通して、攻め・守り・素材基盤の三層に組み直した。自己理解でやったこと（汎用スキルを自分の癖に通す）と、まったく同じ骨格です。

領域が変わっても、Claude Code を相棒に「選んで仕立てる」という一本の背骨は変わりません。次は、この攻めのアプリの中身——構造変化から勝者を見抜くエンジンの設計——を、単独で掘り下げて書く予定です。
