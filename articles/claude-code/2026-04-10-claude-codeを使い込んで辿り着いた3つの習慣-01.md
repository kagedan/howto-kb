Claude Codeを日常的に使い始めてしばらく経ち、自分なりの「型」のようなものができてきました。この記事では、使い込む中で自然と身についた3つの習慣を紹介します。

## `/grill-me` — 実装前に仕様を徹底的に詰める

`/grill-me` は、Matt Pocock 氏が公開したスキルのひとつで、計画や設計に対して徹底的にインタビューし、決定ツリーの全分岐を解決して共通理解に到達するためのスキルです。

[こちらの記事](https://www.aihero.dev/my-grill-me-skill-has-gone-viral)でスキルが紹介されています。純正の`/skill-creator`スキルを利用して以下のようにClaudeに入力することで簡単に導入できます。

```
/skill-creator https://www.aihero.dev/my-grill-me-skill-has-gone-viral このスキル追加して。
```

### スキル化する前

以前は、実装に入る前に「不明点が一切なくなるまで質問し続けてください」といった指示を毎回手動で出していました。これは音声入力で行っていたので、自分の気分やコンディション次第でやったりやらなかったり。仕様確認の質にムラがある状態でした。

### スキル化した後

`/grill-me` としてスキル化したことで、**仕様を明確にしてから実装に入る**というフローが習慣として定着しました。

特に変わったのは、すでに仕様が明確に書かれているように見えるGitHub Issueに対しても、`/grill-me` を毎回実行するようになったことです。「もう十分書いてあるだろう」と思っていたIssueでも、グリルしてみると意外な検討漏れが見つかります。

さらに、`/grill-me` で出てきた疑問点をミーティングで事前に取り上げることで、実装後の出戻りが大幅に減りました。実装前の数分の投資で、手戻りによる数時間のロスを防げるようになったのは大きな変化です。

## `/simplify` — 実装段階でコードをスマートにする

`/simplify` は、変更したコードの再利用性・品質・効率をレビューし、問題があればその場で修正してくれるスキルです。

このスキルは[Claude Codeの純正スキル](https://code.claude.com/docs/en/skills#bundled-skills)なので設定なしで利用することが可能です。

### 以前のやり方

もともとPRレビュー用のスキルを自作していて、5人のレビュアー体制でレビューするようなものを使っていました。ただ、これはPRが出た後にレビューする前提だったので、指摘を受けてから直すという流れになりがちでした。

### `/simplify` を使うようになってから

`/simplify` を実装段階で使うようにしたことで、**PRを出す前にコードの質を上げる**習慣がつきました。結果として、その後のコードレビューがスムーズに進むようになっています。

気に入っているのは、`/simplify` がレビューのスコープをしっかり意識してくれる点です。「今回のPRでやらないこと」を区別して動いてくれるので、過度なリファクタリングに走ることがありません。あくまで今回の変更をスマートにすることに集中してくれる、その塩梅が絶妙です。

## 「チームを構成して」 — 多角的な視点で開発する

Claude Codeに「チームを構成して」というキーワードで指示を出すと、`/simplify` のように複数の視点を持ったチームを構成して多角的に動作させることができます。

これをレビューや実装に取り入れるようになってから、**先入観の少ないフラットな視点**でClaude Codeが動作するようになりました。一つの視点に偏らず、セキュリティ・パフォーマンス・保守性など複数の観点から検討してくれるので、レビューも実装も精度が上がったと感じています。

### トークン消費には注意

ただし、チームを構成して動かすとトークンの消費量はかなり激しくなります。MAXプランであれば気にせず使えますが、Proプランの方は使いどころを見極める必要があるかもしれません。重要な設計判断やリリース前のレビューなど、ここぞという場面で使うのがおすすめです。

## おわりに

3つの習慣に共通しているのは、**Claude Codeの動作を仕組み化・習慣化する**ことで、開発の質が安定したという点です。

* `/grill-me` で実装前の仕様確認を仕組み化
* `/simplify` で実装中の品質担保を仕組み化
* 「チームを構成して」で多角的レビューを仕組み化

毎回手動で指示を工夫するのではなく、スキルやキーワードとして型にしておくことで、気分やコンディションに左右されない安定した開発フローが実現できるようになりました。
