## はじめに

AIコーディングエージェントを日常的に使っていると、ある壁にぶつかります。「1つのエージェントが作業している間、自分も同じリポジトリで別タスクを進めたい」「複数のエージェントを同時に走らせたいけど、ブランチやファイルが衝突してしまう」——こうした並列開発の悩みです。

Emdash は、まさにこの課題に向き合ったツールです。Claude Code や Codex といったCLIエージェントを、それぞれ独立した Git worktree で隔離し、並列に起動・管理できるデスクトップアプリケーションになっています。

この記事では、Emdash の設計思想・主要機能・実際の使用感を紹介していきます。

## Emdash とは何か

**Emdash** は、複数のAIコーディングエージェントを並列に起動・隔離・管理するための実行基盤（Agentic Development Environment）です。MIT ライセンスのオープンソースで、Y Combinator W26 のバッチにも採択されています。

ここで押さえておきたいのは、Emdash は「自律型オーケストレーター」ではないという点です。タスクの分解や自律的な実行判断は行いません。あくまで人間が指示を出し、エージェントがそれぞれの worktree で作業し、人間がレビューしてマージする——という、人間中心のワークフローを支える基盤になっています。

## 並列開発の現状と、Emdash が解決すること

### 手動 worktree 管理の限界

AIコーディングエージェントの活用が広まるにつれて、Git worktree を使った並列開発は一般的なプラクティスになってきています。Claude Code を worktree 内で動かす、Codex を別ブランチの worktree で起動する——こうした運用は、すでに特別な使い方ではなく「エージェントを本格的に使うならこうなる」という自然な流れです。

ただ、worktree を自分で管理する運用にはすぐに限界が見えてきます。1つ2つなら `git worktree add` と `cd` で十分ですが、タスクが3つ、4つと増えてくると管理が追いつかなくなります。「この worktree はどのチケットに対応していたか」「エージェントは終わったのか」「差分はどうなっているか」——こうした状態の把握だけでも負荷になりますし、worktree の作成・削除・ブランチ管理・.env ファイルの同期といった定型作業も積み重なってきます。

さらに、チケットとの紐付け、差分レビュー、PR作成、CI確認まで含めると、エージェントの「外側」にかかる作業負荷は無視できません。

### この課題に向き合うツールが続々登場している

こうした背景もあり、worktree ベースの並列実行を前提としたツールが続々と登場しています。

* **claude-squad**（★6,900+）: tmux + worktree で複数エージェントを管理するターミナルアプリ。Go製、AGPL-3.0
* **Agent Orchestrator**（★6,100+、ComposioHQ）: worktree 隔離 + ダッシュボード監視。CI failure の自動ルーティングにも対応
* **Emdash**（★3,700+）: Electron デスクトップアプリ。23種のエージェント対応、SSH リモート、ローカルファースト

いずれも「エージェントごとに worktree を分離し、安全に並列開発する」という基本設計を共有しています。ツールごとに思想や得意領域は異なりますが、この記事では Emdash に焦点を当てて、その特徴を掘り下げていきます。

### Emdash が引き受けてくれること

Emdash は、先ほど挙げた「エージェントの外側の管理コスト」をまとめて引き受けてくれるツールです。

worktree の自動生成と削除、.env 等の設定ファイルの自動コピー（preserve patterns）、ライフサイクルスクリプト（setup / run / teardown）の実行、さらには worktree のプール（事前予約生成）による起動高速化まで——並列開発にまつわる定型作業をまとめて面倒を見てくれます。加えて、チケット連携・diff レビュー・PR作成・CI確認といった開発フロー全体をひとつの画面で扱えるのが、単なる worktree 管理ツールとの違いです。

## 主要機能の詳細

### 1. Git worktree による完全隔離

Emdash の中核にあるのは、タスクごとに独立した Git worktree を自動生成し、各エージェントをその中で実行する仕組みです。

worktree はリポジトリのサイドディレクトリとして作られるため、互いの変更が干渉しません。あるエージェントがファイルを大胆に書き換えても、別のエージェントの作業には一切影響しない。この隔離は物理的なディレクトリレベルで行われるので、ブランチの切り替え忘れといった事故も起きません。

### 2. 23種のCLIエージェントに対応

Emdash は特定のエージェントに依存しない、プロバイダー非依存の設計をとっています。2026年4月時点で以下の23種に対応しています。

Claude Code / Codex / Gemini CLI / GitHub Copilot / Cursor / Continue / Cline / Amp / Auggie / Autohand Code / Charm / Codebuff / Droid / Goose / Hermes Agent / Kilocode / Kimi / Kiro（AWS） / Mistral Vibe / OpenCode / Pi / Qwen Code / Rovo Dev

インストール済みのCLIを自動検出してくれるので、セットアップは基本的にエージェント側のCLIをインストールしておくだけで済みます。

### 3. チケット連携（Linear / GitHub Issues / Jira）

GitHub Issues、Linear、Jira のチケットをタスクに紐付けて、エージェントのコンテキストとして渡すことができます。

ただし、ここでも「自律」ではなく「パススルー」のモデルである点は意識しておくとよさそうです。Emdash がチケットを自動で拾ってタスク分解するわけではありません。人間がチケットを選び、エージェントに渡す。エージェントはそのチケットの内容をコンテキストとして受け取り、作業を行います。

この設計は、エージェントの暴走を防ぎつつ、チケットのコンテキストをコピペなしで渡せるという実用的なバランスをうまくとっていると思います。

### 4. diff レビューとPR作成

各エージェントの作業結果は、Emdash 上で差分を確認できます。Side-by-Side の diff ビューで変更内容をレビューし、問題なければそのまま PR を作成・マージまで進められます。

CI/CDのチェック状況も確認できるので、PRを出す前にテストが通っているかどうかを Emdash の画面上で把握できるのも便利なポイントです。

### 5. SSH によるリモート開発

Emdash ならではの機能の一つが、SSH/SFTP によるリモートサーバー上での開発サポートです。ローカルマシンの Emdash から、リモートサーバー上のリポジトリに対してエージェントを起動・管理できます。

SSH Agent、秘密鍵、パスワード認証の3方式に対応しており、クレデンシャルはOSのキーチェーンに保存されます。ホスト鍵の検証も `~/.ssh/known_hosts` に基づいて行われるため、セキュリティ面もきちんと考慮されています。

大規模なコードベースや、ローカルでは動かしづらい環境での開発に重宝する機能です。

### 6. ローカルファーストのデータ管理

アプリケーションの状態はすべてローカルの SQLite に保存されます。クラウドへのデータ送信は各エージェントプロバイダーへのAPI通信のみで、Emdash 自体がコードやファイルパスを外部に送ることはありません。

テレメトリもデフォルトで匿名化されており、`TELEMETRY_ENABLED=false` で完全に無効化できます。

### 7. その他の機能

上記以外にも、実用的な機能がいくつか用意されています。

* **Best-of-N モード**: 同じタスクを複数のエージェントに実行させ、結果を比較できます（[ドキュメント](https://github.com/generalaction/emdash)参照）
* **Plan モード**: エージェントを読み取り専用で動かし、実装前の調査・計画に活用できます
* **Automations（β）**: 定期実行タスクの設定が可能です（Emdash 起動中のみ動作）
* **MCP対応**: 複数のMCPサーバーとの接続に対応しています
* **Skills連携**: skills.sh カタログからスキルをインストールできます

## インストールと基本セットアップ

### インストール

macOS の場合、Homebrew でかんたんにインストールできます。

```
brew install --cask emdash
```

Windows は MSI インストーラー、Linux は AppImage または deb パッケージが用意されています。

### 基本的なワークフロー

1. Emdash を起動し、プロジェクト（リポジトリ）を追加
2. 使用するエージェントを選択（CLIがインストール済みであれば自動検出されます）
3. 新しいタスクを作成し、必要に応じてチケットを紐付け
4. エージェントが worktree 内で作業を開始
5. 完了後、diff をレビューし、PR作成またはマージ

チケット連携を使う場合は、Settings から各サービスの認証情報を設定しておきます（GitHub CLI の `gh auth login`、Linear の API キー、Jira のサイトURL + APIトークン）。

## 良かった点

### 並列実行の安心感が大きい

複数エージェントを同時に走らせても「変なところを壊していないか」という不安がありません。これは単に worktree で分かれているだけでなく、PTY（疑似端末）レベルでもプロセスが独立していたり、worktree のパス検証で誤削除を防いでいたりと、隔離が多層的に設計されているからだと思います。手動で worktree を管理していた頃と比べると、心理的な負荷がかなり下がりました。

### 人間がコントロールを持てる設計

自律型オーケストレーターのように「全部おまかせ」ではなく、人間がタスクを割り当て、レビューし、マージするフローが前提になっています。Plan モードで実装前にエージェントの計画を確認し、diff ビューで変更をレビューし、マージ前にCI結果を確認する——という多段階のチェックポイントが設計に組み込まれているので、エージェントの出力をきちんと確認したい場面では安心感があります。

### チケット連携で指示の手間が省ける

GitHub Issues や Linear、Jira のチケットをタスクに紐付けられるので、エージェントへの指示をいちいち手入力しなくて済みます。チケットの内容がそのままコンテキストとして渡されるため、「このIssueの内容をコピペして……」という手作業がなくなるのは地味にありがたいポイントでした。

### エージェントの切り替えが容易

Claude Code で試してみていまいちだったタスクを Codex でやり直す、といった使い方がスムーズにできます。23種のエージェントに対応していて、プロバイダーごとの設定差異（auto-approve、session-id、resume 等）を内部で吸収してくれるので、エージェントの違いを意識せず統一的なワークフローで使えるのは実運用で大きなメリットです。

### タスクの再開がスムーズ

CLIで直接エージェントを使っていた頃は、中断したセッションを再開するのに `--resume` や `--continue` のようなオプションを探して指定する必要がありました。Emdash ではタスクを選択するだけで前回の続きから再開できるので、この手間がなくなったのは地味にストレスが減ったポイントです。

### セットアップが楽

Homebrew で一発インストール、起動後はUIに従うだけでプロジェクト追加からエージェント実行まで進められます。設定ファイルの手書きが不要で、CLIに慣れていない方でも始めやすいのは強みです。

### 外部ツールとの連携が柔軟

Emdash 内蔵のエディタは最低限の機能ですが、Settings の Tools からよく使うエディタ（VS Code、Cursor、Xcode 等）を登録しておけば、Emdash からワンクリックで開けます。Finder や Terminal も同様に連携できるので、Emdash の中で完結させようとせず、使い慣れたツールと組み合わせて使う前提の設計になっています。この割り切りは好印象でした。

### ショートカットキーが充実している

タスクの切り替え（`⌘]` / `⌘[`）、新規タスク作成（`⌘N`）、Kanbanの表示切り替え（`⌘P`）、エディタの切り替え（`⌘E`）など、よく使う操作にはひと通りショートカットが割り当てられています。並列で複数タスクを扱っているとキーボードで素早く切り替えたい場面が多いので、大体のやりたいことがショートカットでできるのは助かりました。

### セキュリティ面の配慮

クレデンシャルがOSのキーチェーンに保存される点、SSH接続時のシェルコマンドがallowlist方式で制御されている点、環境変数のサニタイズなど、このあたりは裏側をきちんと調べてみて好印象でした。ローカルファースト設計で、コードやファイルパスがEmdash経由で外部に送られない点も安心材料です。

## 気になった点・今後に期待したいこと

### 完全自動化を求める方には物足りないかも

タスクの自動分解や自律的な実行判断を期待すると、Emdash の設計とは方向性が異なります。これは欠点というよりも設計上の選択ですが、期待値のずれが生じやすいポイントではあると思います。

### Windows環境の安定性にはまだ課題がある

GitHub Issues を見ると、Windows上でClaude Codeの実行が失敗する問題（[#1703](https://github.com/generalaction/emdash/issues/1703)）や、ライフサイクルスクリプトがWindows環境で動作しない問題（[#1688](https://github.com/generalaction/emdash/issues/1688)）など、Windows固有の課題が複数報告されています。macOSでの利用は安定していますが、Windows環境をメインで使う方は現時点では注意が必要です。

### 「前のターンからの差分」が見たい

複数ターンにわたるタスクで、セッション全体の差分ではなく最新のエージェントのレスポンスによる変更だけを確認したい場面があります。GitHub Issues でも要望として挙がっていて（[#1635](https://github.com/generalaction/emdash/issues/1635)）、これが入るとレビュー体験がかなり良くなりそうです。

### Agent Presets がほしい

タスクの性質に応じてモデルやフラグの組み合わせ（例: 軽いタスクには Sonnet + 低推論、重いタスクには Opus + 高推論）をテンプレートとして保存しておきたい、という要望（[#1472](https://github.com/generalaction/emdash/issues/1472)）には共感します。現状はタスク作成後にエージェントを中断して手動設定する必要があり、ここがスムーズになるとうれしいです。

### CI failure の自動対応に期待

GitHub Actions の失敗を検知してエージェントに自動で修正させるワークフロー（[#1375](https://github.com/generalaction/emdash/issues/1375)）は、実現すれば開発フローの割り込みを大きく減らせそうです。「人間主体」の設計と両立させながらどう組み込むのか、今後の方向性が気になります。

### タスク状態の管理がもう少し柔軟だとうれしい

現在は「表示 / 非表示」の二値で、アーカイブすると一覧から消えてしまいます。実際のプロジェクトでは「保留中」「待機中」のタスクもあるので、Active / On Hold / Archived のような中間状態（[#1336](https://github.com/generalaction/emdash/issues/1336)）があるとタスクの見通しが良くなると思います。

### ブランチ名のプレフィックスがプロジェクト全体で固定

Settings の Repository でブランチ名のプレフィックス（例: `emdash/`）を設定できますが、これがプロジェクト全体で一律です。タスクの種類によって `feature/` や `fix/` のように使い分けたい場面では少し不便に感じました。

### 内蔵エディタは最低限

Emdash 上でファイルを確認することはできますが、本格的な編集には向いていません。ただ前述の通り、VS Code 等の外部エディタをすぐ開ける設計になっているので、実用上は大きな問題にはなりませんでした。

## どんな人におすすめか

### 複数のタスクを並列で回したい方

たとえば、feature A をエージェントに任せつつ、自分は feature B をレビューし、同時に bugfix C も別エージェントに走らせる——といった運用が、worktree 隔離のおかげで安全にできます。worktree のプール機能もあるため、タスクの起動待ちも最小限です。

### エージェントの出力を自分でレビュー・判断したい方

Plan モード → 実行 → diff レビュー → CI確認 → PR作成という多段階の承認フローが設計に組み込まれているので、エージェントの出力を確認しながら進めるワークフローとよく合います。

### 複数のエージェントを比較・使い分けたい方

23種のエージェントに対応しており、プロバイダー固有の設定差異を内部で吸収してくれます。タスクの性質に応じてエージェントを使い分けたり、Best-of-N で比較したりといった柔軟な運用が可能です。

### リモート環境で開発している方

SSH サポートがアーキテクチャレベルで組み込まれており、接続プールや60秒間隔のkeepalive等、実用的な実装がされています。ローカルでの開発と同じ感覚でリモートでも使えます。

### セキュリティやデータ管理を気にする方

ローカルファースト設計で、コードやファイルパスがEmdash経由で外部に送られることはありません。クレデンシャルのキーチェーン管理やテレメトリの無効化オプションも含め、データの取り扱いに配慮されています。

## まとめ

Emdash は、AIコーディングエージェントの「実行基盤」という、まだ新しいけれど確実にニーズのあるレイヤーを提供してくれるツールだと思います。

自律型のオーケストレーターとは異なり、人間が指示・レビュー・判断の主体であり続ける設計は、組織の体制や現時点でのAIエージェントの信頼性を考えると合理的な選択になる感じています。worktree 隔離による並列実行の安心感と、23種のエージェント対応による柔軟性は、日常的にエージェントを使う開発者にとって実用的な価値があります。

まだ使い込みの途中ですが、並列開発の管理コストを確実に下げてくれるツールだと実感しています。新たな知見が得られ次第、この記事を更新していく予定です。
