▼作ったもの  
CCIndex — Claude Codeに関するノウハウ記事を、7つの情報ソースから自動収集してAIスコアリングで表示する検索エンジン。  
<https://ccindex.unravel-workspace.com/>  
Zenn、Qiita、GitHub Issues、YouTube、note、はてなブックマークから「Claude Code」に関する記事を横断的にクロールし、各記事をHaiku 4.5が5軸で評価。スコア順にProduct Hunt風のUIで表示する。  
現在497件の記事がインデックスされている。  
▼なぜ作ったか  
Claude Codeの情報は爆発的に増えている。だが散在している。  
Zennには体系的な実践記事がある。Qiitaには実装寄りのTipsがある。GitHubのIssuesには最も技術的深度の高い情報がある。YouTubeには動画チュートリアルがある。noteには非エンジニアの体験記がある。  
どのプラットフォームにも質の高いノウハウがあるのに、横断的に「今いちばん役に立つ記事」を探す手段がなかった。  
Google検索では「Claude Code 使い方」で検索しても、SEOが強いだけで中身が薄い記事が上位に来る。本当に実践的なノウハウはZennの深い記事やGitHub Issuesの中に埋もれている。  
だから作った。  
▼技術スタック  
FastAPI + Jinja2 (SSR) + SQLite WAL  
Python 3.12 / httpx / anthropic SDK  
Cloudflare Tunnel（外部公開）  
Claude Code（実装全般）  
Reactは検討したが、SEOを優先してJinja2のSSRに統一した。検索ページだけJS fetchで「もっと見る」を実装。

▼スコアリングの仕組み  
各記事に対して4つの指標を重み付けして最終スコアを算出する。  
final\_score = usefulness × 0.50  
+ freshness × 0.25  
+ authority × 0.15  
+ engagement × 0.10  
usefulness（有用性） が最大の重み。Haiku 4.5が記事のタイトルと本文冒頭を読み、以下の5軸で各20点満点（合計100点）で評価する。  
軸  
評価基準  
practicality  
読者がすぐに実践できるか  
depth  
表面的でなく深い理解が得られるか  
accuracy  
技術的に正確か  
rarity  
他では得られない情報か  
structure  
構造化されて読みやすいか  
freshness（鮮度） はカテゴリごとに半減期を設定。「アップデート速報」は14日で半減、「ワークフロー」は60日で半減。Claude Codeは進化が速いので、古い情報は自動的に順位が下がる。  
authority（権威性） はソースごとの基準値。GitHub Issues（公式リポジトリ）= 100点、Zenn = 85点、X = 55点。さらに著者ごとの過去実績で±15点の補正。  
クローラーの実装  
7つのクローラーはそれぞれ異なるアプローチで実装した。  
API系（安定）： Zenn API、Qiita API v2、GitHub REST API。ページネーション対応で最大100件ずつ取得。  
HTMLパース系（不安定）： YouTube（ytInitialData JSON抽出）、note（API v3のフィールド名がsnake\_caseとcamelCaseで揺れる問題を修正）、はてなブックマーク（RDF/RSS 1.0形式のXMLパース）。  
停止中： X/Twitter。Nitterが全エンドポイント停止中のため0件。$100/月のAPI Basic Planは検証後に判断。  
いちばん苦労したのはnoteとはてな。両方ともHTMLパーサーが最初は0件を返し、実際のHTTPレスポンスを叩いてデバッグするところからやり直した。noteはAPI v3のpublish\_at（snake\_case）をパーサーがpublishAt（camelCase）で参照していた、という1文字の問題だった。  
▼5時間の開発タイムライン  
この開発はClaude Code（CC）とClaude.ai（CAI）の二層構造で進めた。  
CAI（戦略レイヤー） = この会話。コンセプト設計、仕様決定、タスク分解、品質チェック。  
CC（実装レイヤー） = bridge-broker経由で自動実行。コード生成、テスト、デプロイ。

bridge-brokerによる並列タスク実行  
今回の開発で特徴的なのは、CAIからbridge-broker経由でCCタスクを投入し、結果をポーリングで確認するというワークフロー。  
CAI側からタスク投入  
bridge-broker:submit\_task(  
goal="Product Hunt風デザインに全テンプレート刷新",  
model="opus",  
project="ccindex"  
)  
→ タスクID返却、CCが自動実行  
→ 6分後にstatus=done、feedback=A  
1タスクあたり平均3-6分で完了。DAG依存（depends\_on\_task\_id）を使えば、「クローラー完了後に検索機能を実装」のような直列実行も可能。  
今回は合計15タスクを投入。うち13タスクがフィードバックA（最高評価）で完了した。  
▼苦労したポイント  
Cloudflare Tunnel（30分）  
既存のCloudflare TunnelがリモートConfigモードで動いていることに気づかず、ローカルのconfig.ymlをsedで編集し続けて時間を浪費した。YAMLをsedで編集してインデントを壊すという古典的なミスも。  
最終的にCloudflare Zero TrustダッシュボードのPublished Application Routesから追加して解決。Cloudflare Tunnelを使っている人は、自分のトンネルがリモート管理かローカル管理かを最初に確認すべき。  
note/はてなのHTMLパース  
外部サイトのHTMLパースは本質的に不安定。noteは内部API v3が使えたが、フィールド名のケースが揺れるという罠があった。はてなはRSS（RDF/RSS 1.0形式）が最も安定した取得方法だった。  
教訓：HTMLパースよりAPI > RSS > JSON-LD の優先順位で攻める。
