## はじめに

Codex や Claude Code を使うと、実装や記事執筆の速度は上がります。短い作業なら、会話の中だけで完結してもあまり困りません。

一方で、数日後に同じテーマへ戻ると、別の問題が出てきました。

前回なぜその設計にしたのか。どの代替案を捨てたのか。その判断は、次の記事やポートフォリオ説明に使えるのか。こうした情報が、会話ログの中に埋もれていきました。

そこで、ローカルの Obsidian Vault で運用している個人用ナレッジベース `My-Skill-Graph` に、AI との作業で生まれた設計判断を残すようにしました。

この記事では、Zenn 記事管理リポジトリ [harness17/zenn-articles](https://github.com/harness17/zenn-articles) での運用を例に、会話ログをそのまま保存するのではなく、**後で再利用できる判断記録に圧縮する**方法を書きます。

## 会話ログから判断だけを探すのは重かった

AI との会話ログには、かなり多くの情報が残ります。

試した案、捨てた案、コマンドの出力、途中の文章、レビューで出た指摘。作業の過程を振り返るには便利です。ただ、後から使いたい情報は会話全体ではありません。

たとえば、記事候補を考えたあとに見返したいのは、次のような情報です。

* 何を記事候補にしたか
* なぜその候補を優先したか
* 何と重複するので避けたか
* 次にどの切り口で書くか

設計でも同じです。

「Entity を直接 View に渡すか、ViewModel を分けるか」「外部 API を毎回叩くか、RSS とキャッシュを組み合わせるか」のような判断は、作業時点では会話の流れで決まります。しかし後日、同じ種類の判断に戻ったとき、会話ログから理由を探し直すのは時間がかかりました。

そのため、会話ログを保存すること自体ではなく、判断を再利用できる単位に切り出すことにしました。

## decisions には「何を選んだか」ではなく「なぜ選んだか」を置く

`My-Skill-Graph` では、設計判断を `decisions/` に残しています。

ここで意識しているのは、タイトルを名詞にしないことです。

たとえば、`ViewModelについて` というタイトルでは、後で検索結果に出てきても中身が分かりません。代わりに、次のように命題文にします。

```
---
description: "画面専用の入力状態をEntityから分離し、ControllerとViewの責務を読みやすくする判断。"
alternatives: "Entityを直接Viewへ渡す / 入力用DTOを別に作る"
rationale: "画面都合の項目がEntityへ混ざると、永続化モデルと入力フォームの変更理由が分かれなくなるため。"
project: "[[DevNext-index]]"
status: active
type: decision
created: 2026-05-14
---

# 入力フォームをViewModelに寄せたのは画面都合をEntityへ混ぜないため
```

この形にすると、本文を開く前に判断の方向が分かります。

重要なのは、採用した技術名だけを残さないことです。`ViewModelを使った` だけでは、次の判断に使いにくいです。`画面都合をEntityへ混ぜないため` まで入れておくと、別の画面を作るときにも比較軸として使えます。

AI に実装を頼むときも、この粒度の判断が残っていると再利用しやすくなります。

「このプロジェクトでは入力フォームの画面都合は ViewModel に寄せる」という判断を前提として渡せるからです。毎回、会話の中で同じ設計方針を説明し直さなくて済みます。

## strategies には技術判断がどう使えるかを置く

設計判断は、そのままだとコードの文脈に閉じます。

ただ、個人開発では、設計判断が記事、OSS、職務経歴書、面接での説明材料につながることがあります。そこで、技術判断から外に出せる価値は `strategies/` に残しています。

たとえば、この記事自体も最初は「AI との作業ログをどう残すか」という話でした。しかし、同シリーズで近接する「AI 2 台のクロスレビュー運用」記事と重複しやすい切り口でした。

そこで、次のように切り直しました。

```
---
description: "AIとの会話に埋もれた設計判断を再利用できる形にすると、記事テーマとポートフォリオ説明の両方に使える。"
opportunity_type: product
status: active
skills_required: ["[[設計判断を命題文で残す]]"]
next_action: "クロスレビュー記事とは分け、判断の再利用に絞って本文を書く"
created: 2026-05-14
---

# 設計判断メモは記事テーマとポートフォリオ説明に再利用できるため
```

このメモで残しているのは、記事本文ではありません。

「この記事は何のために書くのか」「どの隣接テーマとは分けるのか」という判断です。これを残しておくと、別の日に続きを書くときも、AI に依頼するときも、切り口がぶれにくくなります。

特に Zenn 記事では、似たテーマが増えやすいです。

今回も、`AI同士のクロスレビュー運用` と `設計判断を再利用する運用` は近い場所にあります。前者はレビュー品質、後者は判断の保存と再利用です。この境界を戦略メモとして残すことで、記事同士の役割を分けられました。

## 記録する条件を決めないと続かない

最初から全部を記録しようとすると、運用は重くなります。

一方で、気が向いたときだけ記録すると、重要な判断ほど忙しいタイミングで抜けます。そこで、記録する条件を絞りました。

今の運用では、主に次のときだけ残します。

* 代替案を比較して設計判断をした
* 新しいライブラリ、パターン、連携方法を採用した
* 技術作業から記事、OSS、仕事探しに使える示唆が出た
* 次にやることや作業スレッドが変わった

反対に、単純なファイル読み取り、軽い文言修正、フォーマットだけの変更、git 操作だけでは残しません。

この線引きで、`My-Skill-Graph` を作業日誌にしすぎないようにしています。残したいのは、すべての行動ではなく、後で説明したい判断です。

Zenn 記事リポジトリでも同じです。誤字を直しただけなら残しません。記事の切り口を「ツール紹介」から「判断の再利用」に変えたなら残します。後者は、次の記事構成にも、公開前レビューにも使う判断だからです。

## 記事候補を探すときに判断メモが効いた

この運用を始めてから、記事候補の探し方が少し変わりました。

以前は、記事を書こうとしたときに「何を書けるか」をその場で考えていました。今は、過去の判断記録から探します。

見ているのは、次のようなメモです。

* 何かに詰まった記録
* 代替案を比較した記録
* 後から後悔した設計
* 別の用途にも使えそうな運用改善

この探し方にすると、体験記事にしやすい候補が見つかります。

たとえば、YouTube Data API のクォータ対策は、「API を使った」という話ではなく、「毎回 API を叩く設計では日次クォータに耐えられなかったので、RSS とキャッシュを組み合わせた」という判断として記事にできます。

ASP.NET Core MVC の ViewModel 分割も、「ViewModel の説明」ではなく、「画面都合を Entity に混ぜないために分けた」という判断として書けます。

AI との作業ログも同じです。単に「Obsidian に保存しています」では、単体では体験記事の固有性が出にくいです。今回のように、「会話ログから判断だけを切り出して、次の記事や実装に再利用する」と置くと、体験記事として書きやすくなります。

## クロスレビュー記事とは役割を分ける

近いテーマとして、Codex と Claude Code をクロスレビューさせる運用があります。

そちらは、作成者とレビュアーを分けることで、事実誤認、守秘リスク、AI 的な締め文を検出する話です。中心はレビュー品質です。

この記事の中心は、レビューではありません。

レビュー前後に生まれた判断を、会話の外に出して再利用することです。

たとえば、クロスレビューで「この表現は守秘リスクがある」と分かったとします。その場で直すだけならレビュー結果です。しかし、「今後、業務ドメイン由来の enum 名は記事用の汎用名に置き換える」と残せば、次の記事にも使える判断になります。

この違いを意識すると、2つの記事は重複しません。

* クロスレビュー運用: 別 AI で何を検出できたか
* 設計判断メモ運用: 検出や実装から生まれた判断をどう再利用するか

今回の記事では、後者だけを扱います。

## まとめ

AI との作業では、会話ログを残すだけでは後から使いにくいと感じました。

そこで、`My-Skill-Graph` に `decisions/` と `strategies/` を作り、設計判断と記事・ポートフォリオへの接続を分けて残すようにしました。

効いたのは、次の3点です。

* タイトルを命題文にして、判断の中身が検索結果だけで分かるようにする
* 技術判断と、それを記事や仕事探しに使う戦略を分ける
* すべてを記録せず、後で説明したい判断だけを残す

AI を使って作業を速く進めるだけなら、会話ログだけでも足ります。数日後、別の作業や記事に判断を渡したいなら、会話の外に判断を残す仕組みが必要でした。
