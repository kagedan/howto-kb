## はじめに

これまで「AIエージェント検証用PCを作るメモ」として、設計思想の整理からDocker、WSL、Git、Node.js、Codex CLIの導入までを記録してきた。

しかしシリーズを振り返ると、そもそも「なぜ検証専用のPCを用意したのか」という出発点を、十分に書けていなかった。

この検証用PCを作った本来の目的は、Codex CLIを動かすことだけではない。

試してみたかったのは、ブラウザ上の情報を読み取り、複数のタブを扱い、場合によってはWeb上で操作まで行うAIエージェントだった。

しかし、普段使いのPCやブラウザには、ログイン情報、Cookie、閲覧履歴、クラウドサービス、個人ファイルなどが集まっている。

便利そうだからといって、いきなり普段使いの環境で試すのは不安だった。

そこで、古いノートPCを初期化し、Microsoftアカウント、Googleアカウント、AIサービスのアカウントも検証用として分けることにした。

---

## 本来試したかったのはブラウザAIエージェントだった

現在このシリーズでは、以下のような構築を進めている。

```
Docker Desktop
WSL2 / Ubuntu
Git
Node.js / npm
Codex CLI
AIエージェント用sandbox
安全運用ルール
```

ここまでの流れだけを見ると、Codex CLIや開発環境を作ることが最初の目的だったように見える。

しかし、出発点は別のところにあった。

本来試してみたかったのは、以下のようなAIエージェントだった。

* ブラウザ上の複数ページを読み取る
* Web上の情報を比較・整理する
* ブラウザ内で作業を補助する
* ローカルファイルを読んだり編集したりする
* 複数のツールを横断して作業する

従来のチャットAIよりも、PCやブラウザに触れられる範囲が広い。

だからこそ、普段使いの環境とは分けて試したいと考えた。

---

## 普段使いのPCで試すのが怖かった理由

普段使いのブラウザには、想像以上に多くの情報が集まっている。

```
ログイン中のWebサービス
Cookieやセッション情報
検索履歴・閲覧履歴
自動入力情報
クラウドストレージ
メールやカレンダー
ダウンロードしたファイル
仕事や個人用のブックマーク
```

ブラウザAIエージェントがどこまでアクセスできるかは、製品や設定によって異なる。

ただし、アクセスできる範囲が広いツールを試すなら、最初から本物のデータが入った環境を渡さない方が安心できる。

そこで、普段使いのPCに追加するのではなく、古いノートPCを検証専用機として使うことにした。

---

## 最初に端末を初期化した

検証用に使うノートPCは、一度初期化した。

目的は、単に動作を軽くするためではない。

以前の利用データやアカウント、普段使いのファイルが残った状態でAIエージェントを試さないためだ。

```
メインPC
→ 普段の仕事、制作、個人データを扱う

検証用PC
→ AIエージェントの実験だけに使う
```

これはネットワークから完全に切り離す「エアギャップ」ではない。

同じ家庭内ネットワークへ接続する場合もあるため、端末を分けただけですべてのリスクが消えるわけではない。

それでも、個人データや仕事環境と物理的に端末を分けることで、誤操作時の影響範囲を小さくできると考えた。

---

## アカウントも検証用に分けた

端末だけでなく、検証に使うアカウントも分けた。

セキュリティ上の理由だけではない。

普段使いのAIサービスには、単なる作業履歴ではなく、長い時間をかけて積み重ねてきた大切な対話や設定が残っている。

検証中の誤操作や予期しない連携によって、それらへ影響が及ぶ可能性をできる限り避けたかった。

そのため、多少手間や費用が増えても、Microsoft、Google、AIサービスのアカウントを検証用に分けることにした。

今回、新しく用意したのは以下のようなアカウント。

* Windowsで使うMicrosoftアカウント
* Google関連サービスの検証用アカウント
* ChatGPTなどAIサービスの検証用アカウント
* 今後使う可能性がある開発・クラウドサービスの検証用アカウント

実際の記事では、メールアドレス、アカウント名、電話番号などは公開しない。

大事なのは、普段使いのアカウントをそのままAIエージェントへ渡さないことだった。

検証用アカウントには、最初から以下を持ち込まない。

```
個人メール
仕事のメール
本物の顧客データ
普段使いのクラウドストレージ
個人写真
決済情報
本番用APIキー
仕事用の認証情報
```

まずは公開情報、ダミーデータ、失っても困らないテストファイルだけで試す。

---

## 実は、この時点で小さな実験は始められた

端末を初期化し、検証用アカウントを用意し、個人データを持ち込まない状態にした。

公開Webページの要約や、ダミーアカウントを使った小さなブラウザAI実験であれば、この時点から始めることもできた。

しかし当時は、Dockerを使えばブラウザAIエージェントも含め、さらに強く隔離できるのではないかと考えていた。

そこで、先にDockerやWSL2の環境構築へ進んだ。

---

## Dockerなら全部隔離できると思っていた

当時は、Dockerに対して「アプリや実行環境を、安全な箱の中へ閉じ込められる仕組み」というイメージを持っていた。

```
AIエージェント
↓
Dockerへ入れる
↓
PC本体から隔離できる
↓
安全に試せる
```

Dockerは、アプリケーションや依存関係をコンテナ内へ分離するために便利な仕組みだ。

Pythonなどの実行環境をWindows本体へ直接入れず、コンテナ内だけで動かす用途にも向いている。

実際に、以下のような検証はできた。

```
Windows側
C:\AI-LAB\workspace

Dockerコンテナ側
/workspace
```

Windows側の検証用フォルダだけをコンテナへ渡し、コンテナ内のPythonを動かすところまでは成功した。

しかし、DockerはPC上のあらゆるAIエージェントを、そのまま簡単に閉じ込められる万能な箱ではなかった。

---

## ブラウザ統合型AIをDockerへ入れるのは別の話だった

ブラウザAIエージェントの中には、普段使うデスクトップブラウザへ統合されるものがある。

この場合、Dockerコンテナ内へ単純にインストールすれば終わり、とはならない。

GUIブラウザをコンテナ内で動かす場合、用途によっては以下のような仕組みが必要になる。

```
ヘッドレスブラウザ
PlaywrightやSelenium
画面表示用のVNC
ブラウザプロファイルの永続化
Cookieやセッションの管理
ダウンロード先のマウント
ホスト側ファイルとの受け渡し
```

さらに、一般向けのブラウザ統合機能やデスクトップアプリは、そもそもDocker内で動かすことを前提としていない場合もある。

Docker内でChromiumを自動操作する仕組みを作ることと、普段使うブラウザのAI機能をDockerへ閉じ込めることは別の問題だった。

実際に調べて触れていく中で、同じAIエージェントでも、ブラウザ、ローカルファイル、ターミナル、コンテナなど、アクセスできる範囲が大きく異なると分かった。

そのため、一つの隔離方法ですべてを守ろうとするのではなく、種類ごとに「檻」を分けて考えることにした。

---

## AIエージェントごとに違う「檻」

現在は、エージェントを大きく3種類に分けて考えている。

### ブラウザ系AIエージェント

ブラウザ上のページやタブ、Webサービスを扱うエージェント。

```
専用PC
検証用ブラウザプロファイル
検証用アカウント
個人データを入れない
連携するサービスを最小限にする
ダウンロード先を検証用フォルダへ限定する
```

ブラウザ系では、Dockerだけに頼るよりも、端末・アカウント・ブラウザプロファイルを分けることが現実的な隔離方法になる。

### CLI・ローカルファイル操作系

Codex CLIやClaude Codeなど、ターミナルからファイルを操作するエージェント。

```
WSL / Ubuntu
~/agent-sandbox
Gitによる差分確認
SECURITY_POLICY.md
AGENTS.md / CLAUDE.md
秘密情報を置かない
```

こちらは、操作対象となる作業ディレクトリを限定し、Gitで変更を確認できる状態を作る。

### Dockerで動かすアプリやサービス

Python環境、Webアプリ、Difyなど、コンテナとして動かす対象。

```
Dockerコンテナ
マウントするフォルダを限定
不要なポートを公開しない
秘密情報の渡し方を別途設計
ボリュームやログの保存先を確認
```

Dockerは、この種類の隔離に強い。

---

## CLIエージェントの構築は脱線ではなかった

本来の目的がブラウザAIエージェントだったなら、CLIエージェントの構築は遠回りだったようにも見える。

しかし、最初の検証対象としてCodex CLIを試したことは、結果的に重要な学びになった。

Codex CLIを試す過程で、以下を実際に確認できた。

* AIエージェントに触らせる作業場所を限定する
* 作業前にGitの状態を確認する
* 作業後に `git status` と `git diff` を見る
* 変更内容を確認してからコミットする
* 秘密情報を作業場所へ持ち込まない
* AI向けの安全ルールをファイルで渡す
* 破壊的な操作を承認なしに行わせない

これらは、今後Claude Codeなど別のCLIエージェントを試す場合にも共通して使える考え方だ。

Docker、WSL、Git、Codex CLIの構築は、本来の目的から外れたのではない。

CLIエージェントへどこまで権限を渡し、何を自分で確認するのかを理解するための、別レイヤーの準備だった。

---

## 現在の検証環境

現在の構成は、以下のようになっている。

```
検証専用ノートPC
├─ 検証用Microsoftアカウント
├─ 検証用Googleアカウント
├─ 検証用AIサービスアカウント
│
├─ Windows側
│  └─ C:\AI-LAB\
│     ├─ DESIGN_POLICY.md
│     └─ workspace\
│
├─ WSL / Ubuntu側
│  └─ ~/agent-sandbox\
│     ├─ README.md
│     ├─ SECURITY_POLICY.md
│     ├─ AGENTS.md
│     ├─ CLAUDE.md
│     └─ .gitignore
│
└─ Docker
   └─ 必要な検証用コンテナ
```

外側には、専用PCと検証用アカウントがある。

内側には、Windows側の `AI-LAB`、Ubuntu側の `agent-sandbox`、Dockerコンテナがある。

一つの仕組みですべてを守るのではなく、複数の小さな隔離を重ねる構成になった。

---

## これから試したいこと

今後は、エージェントごとに小さな実験から始める。

### ブラウザ系

* 公開Webページの要約
* 複数タブの情報比較
* ダミーアカウントを使った操作
* 連携サービスを最小限にした検証
* ダウンロードや外部送信の確認

### CLI・ローカルファイル系

* `agent-sandbox` 内の小さなファイル編集
* Gitで差分確認
* CodexとClaude Codeの挙動比較
* AI向けルールファイルの検証

### Docker系

* 検証用アプリのコンテナ実行
* マウント範囲の確認
* ポート公開範囲の確認
* 秘密情報を扱う前の設計
* Docker secretsなどの検討

いきなり本番データや普段使いのアカウントは渡さない。

まずは「失敗しても困らない実験」を繰り返す。

---

## 初心者だったからこそ残しておきたいこと

当初は、Dockerを使えばすべてのAIエージェントを安全に隔離できるのではないかと考えていた。

実際には、エージェントの種類によって、必要な隔離方法は異なっていた。

これは単なる失敗というより、最初の仮説を実際の構築を通して更新した結果だ。

```
最初の仮説
Dockerでまとめて隔離する

構築後の理解
ブラウザ系・CLI系・Docker系で隔離方法を分ける
```

最初から正解を知っていたわけではない。

分からない状態から小さく試し、違いに気づき、設計を修正していった。

同じように「AIエージェントを試したいが、普段使いのPCで動かすのは怖い」と感じている人にとって、一つの検討材料になればよいと思う。

---

## まとめ

ブラウザAIエージェントを安全に試すため、古いノートPCを初期化し、検証用のアカウントも分けた。

当初はDockerがすべてを隔離してくれると考えていた。

しかし、実際にはAIエージェントの種類ごとに、異なる隔離方法が必要だった。

```
ブラウザ系
→ 専用PC・検証用アカウント・ブラウザプロファイル

CLI・ファイル操作系
→ WSL・専用sandbox・Git・安全ルール

Dockerで動かすサービス
→ コンテナ・限定マウント・ポート・秘密情報管理
```

専用PCを用意した判断は、最終目的から外れていなかった。

むしろ、ブラウザAIエージェントを試すために必要な、一番外側の隔離を最初に作っていた。

Docker、WSL、Git、CLIエージェント環境の構築は、その内側へ安全な作業場所を追加していく工程だった。

本編では十分に触れられなかった「検証環境の外側をどう分けたか」を整理するため、今回の記事を設計補遺としてまとめた。

これからは、最初に試したかったブラウザ系AIエージェントについても、検証用環境の中で小さく試していく。
