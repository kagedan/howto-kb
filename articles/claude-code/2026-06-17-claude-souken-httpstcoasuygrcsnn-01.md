これらは全て拡張機能（公式の言葉では extensions）によって解決できます。「エンジニア向けの難しい話だ」と引いてしまう人もいるかもしれませんが、大丈夫です。その本体は「Claude Code に何を覚えさせ、何を任せ、どこに繋ぐか」の設計であって、業務をわかっている人ほど向いています。(コードを書かない人にも橋は架かります)

本記事ではこれらを「8種の神器」（CLAUDE.md / Skills / Code intelligence / MCP / Subagents / Agent teams / Hooks / Plugins）と称して、それらを「全体地図」として一枚にまとめています。**最後まで読むと、これらが使うタイミング別に4つ+αに分類でき、自分の使いたいタイミングで、どれを最初に触ればいいかを判断できる状態になります。**

# 1. そもそも「神器」は、いつ必要になるのか

# 

神器は、手に入れること自体が目的ではありません。Claude Code で困り始めた瞬間が入り口です。逆に言えば、困っていなければ手に入れなくていい。これが大前提です。

冒頭で挙げたような壁を整理すると、困り方は大きく次の4つに分かれます。

- 同じ前提・同じ規約を毎回説明している（覚えさせたい）
- 決まった手順を毎回呼び出している（手順を部品にしたい）
- 手元の外にあるツールやデータに繋ぎたい（接続したい）
- 会話が長くなると本筋がぼやける、または毎回同じ後始末を頼んでいる（任せたい・自動化したい）

まずそのまま使い、不便を感じた点に対して機能を足していく順番がお勧めです。公式も「Build your setup over time（設定は時間をかけて育てる）」として、最初から全部を盛る設計ではないことを明示しています。

神器は「困り方」と1対1で対応しています。この記事の残りは、4つの困り方に対して「どの神器が効くか」を地図にしていく作業です。地図さえ手元にあれば、神器が増えても置き場所に迷いません。

# 2. 8つの神器は「いつ使うか」で分けられる

# 

ここがこの記事の背骨です。8つを丸暗記するのではなく「いつ使うか」で束ねると、一気に分かりやすくなります。

なぜ「いつ使うか」で分けるのか。Claude Code には一度に扱え報量（コンテキスト、AIがその時に同時に見ていられる作業机の広さ）に上限があり、机に「いつ載せるか」で、賢さも速さもコストも変わるからです。この観点で並べ直すと、下図の通り8つのうち7つが4つの層に収まり、8つ目の Plugins だけは「その4層をまとめて配る箱」という別格の位置になります。

※冒頭で「4つ＋α」と書いたのは、この4層とPlugins のことです。

## **8種の神器の全体マップ**

![](https://pbs.twimg.com/media/HLCU44cbEAAq70z.jpg)

この4層は、第1章の「4つの困り方」とそのまま対応します。

- 覚えさせたい→常時ロード層
- 部品にしたい→オンデマンド層
- 接続したい→接続層
- 任せたい・自動化したい→分離・自動化層
「こんな時どうすれば…」と感じたタイミングに応じて、触る層が一意に決まります（Plugins は「固まった設定を配りたくなったら」の一段あとの話）
