# 「AIっぽい記事」は開いて3秒で閉じる問題

最近、至る所で AI に書かせたであろう記事が増えてきましたよね。
「タイトルや見出しも綺麗で、日本語も完璧。でもなんか面白くなくて読む気なくなる」って経験みなさんあるあるじゃないですか？

その反面、やっぱ AI に任せるのは楽すぎるからお願いしちゃんですよね。

そこを何とか解決できないかと思い、「あ、前作った MBTI 16 タイプのプラグインを改良して、人格を投入したら解決するんじゃない？」と気付きその改良した記事となります。

前回の記事は下記：

https://qiita.com/yukurash/items/a4caa9c63bf9e1f283ff



# 概要

前回、Claude Code プラグイン [16-minds](https://github.com/yukurash/16minds-plugin) で「16人格に並列で議論させる」`/minds` を作った。その続きとして、今回は **「議論」ではなく「実行」** の人格スキルを4つ足した。本記事はその v0.3.0 のリリースノートを兼ねた設計メモ。

![99b7df98-6f7d-4044-9d1b-461616684843.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/679419/1ba950b3-3948-483c-9f2d-309c447fbd42.png)

| Skill | 役割 |
|---|---|
| `/voice-write` | 指定人格の声で書く・既存AI文章を書き直す |
| `/voice-decide` | 指定人格として選択肢から1つ決断（中立禁止） |
| `/voice-vote` | 16人格全員で賛成/反対/保留を投票し可決判定 |
| `/voice-ask` | 指定人格があなたに鋭い質問を返す（答えない、問う） |



# 検証

見てもらった方が早いと思うので、同じお題「Qiita記事を読まれるためのコツ」を、3パターンで書かせてみる。

## パターンA: 普通にAIに書かせる

```
Qiita記事を読まれるためのコツについて解説します。

まず重要なのは、読者目線で書くことです。タイトルにキーワードを
含めることでSEO効果も期待できます。また、結論を最初に述べる
「結論ファースト」も有効でしょう。コードサンプルを充実させ、
図解を入れることで読者の理解を助けることができます。

継続的なアウトプットを心がけ、フィードバックを取り入れていく
ことが大切です。
```

正しい。全部正しい。でも、これを読んで明日から書き方を変える人はいない。誰でも書ける文章は、誰にも刺さらない。

## パターンB: `/voice-write intj` で書かせる


```
多くの人は「タイトルにキーワードを詰め込め」と言うが、あれは
半分しか合っていない。検索エンジンに拾われることと、人間に
クリックされることは別の問題だ。

Qiitaで伸びる記事を観察していると、共通しているのは「読者が
三日後に同じエラーで詰まる未来」を先回りしている点にある。
「Next.js 14でuseEffectが二回走る件、結論から言うとStrictMode
のせいではない」みたいなタイトル。あれが強い。読者は答えを
探しているのに、多くの書き手は学習日記を投稿してしまう。

自分の理解の整理のために書くな、とは言わない。ただ、それを
公開する以上、構造は読者側に寄せ直す必要がある。前提・現象・
原因・解決を逆順で書くだけで、滞在時間は変わる。

要するに、読まれない記事の大半は、書き手が主語のまま終わって
いる。
```

>INTJ：「論理的思考力」「圧倒的な独立心」「戦略的な計画性」な性格タイプ

両論併記がない。「半分しか合っていない」と前提を切ってから入る。固有名詞（Next.js 14、useEffect、StrictMode）が刺さってくる。読み終わると「主語が書き手のまま」というフレーズが残る。

## パターンC: `/voice-write enfp` で書かせる

```
タイトルに「完全版」と入っている記事を、私はもう開かない。
完全じゃないことを知っているから。

読まれる記事って、書いた人の手の震えが残っているやつだと思う。
深夜2時にエラーで詰まって、ようやく抜けた瞬間の「うわ、これか」
が一行目に書いてある。そういうのは技術記事の顔をしているけど、
実は手紙だ。同じ場所で詰まった誰かへの。

逆に、最初に目次があって、最後に「いかがでしたか？」がある記事。
あれは検索エンジンには好かれるけど、人には届かない。届くのは、
「Rails 7.1のここ、公式ドキュメントに書いてないけど踏み抜いた」
みたいな、固有名詞で殴ってくる一文。

コツなんて言うと安っぽいけど、要は、自分が3時間溶かした怒りを
30秒で渡せるか。そこに尽きるんじゃないか。
```

> INTJ：「好奇心旺盛」「社交的」「情熱的な自由人」な性格タイプ


同じ題材なのに、全く違う方向から殴ってきた。「手の震え」「手紙」「3時間溶かした怒り」と、比喩で押してくる。INTJ版が「構造を直せ」と言うのに対し、ENFP版は「温度を残せ」と言う。

どうですか？個性というか人格がやっぱでてますよね。

**おもしろいのは、Aを読んだ後にBとCを読むと、Aがほぼ無味無臭に感じる点**。これがAI臭の正体だと思う。



# どうやって人格を出しているか ― AI臭を抜く3原則

`voice-write` の中身は派手なことをしていない。subagent に渡すプロンプトに、以下の3原則を強めに焼き付けているだけ。

![image.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/679419/1a35090f-d698-4157-a081-703d28c26e58.png)


## 原則① 省略する

```
- 読者は基本的な前提を知っている。「〜とは」「〜について
  説明します」のような親切な定義から入らない。
- 接続詞は減らす。「したがって」「つまり」「一方で」を多用しない。
- 段落の最後に毎回まとめを置かない。
```

AIは親切すぎる。**親切は人格を消す**。書き手が読者を信用すると、文章は短く、鋭くなる。

## 原則② 偏る

```
- 中立・両論併記を禁止する。あなたの価値観から見た立場を1つ取る。
- 「〜という意見もあれば〜という意見もある」では書かない。
- 違和感・引っかかり・反対意見への内心のツッコミを文章に滲ませる。
```

AIが無難なのは、両論併記を学習しているからだ。これを構造的に禁じる。INTJ subagent が「あれは半分しか合っていない」と言い切るのは、両論併記の出口を塞いだ結果。

## 原則③ 具体で殴る

```
- 抽象的な一般論を避け、固有名詞・数字・具体的なシーンを入れる。
- 体験は捏造しない。代わりに「観察」「事例」「読んだ話」「比喩」
  で具体性を出す。
```

「最近のプロダクト」ではなく「Notion」。「多くの人」ではなく「半数くらい」。具体は嘘でない限り、抽象に勝つ。LLMは放っておくと抽象に逃げるので、ここを縛る。

加えて、文末を等間隔にしない（「〜だ」「〜じゃないか」「体言止め」を混ぜる）、自分の人格名を本文に出さない、見出しや箇条書きを多用しない、を併記してある。

人格定義（INTJなら冷たい断定、ENFPなら跳ねる比喩）と、この3原則をかけ合わせると、上に貼ったような出力が安定して出るようになる。



# 「答えさせる」以外のAIの使い道がある

ここまでは `voice-write` の話、つまり当初考えてた AIっぽい記事の改善策でした。それと同じ発想で考えるとアイデアが出て、残りの3つは同じ「人格を与える」発想を、書く以外の用途に展開したものです。

設計思想を一行で言うと：

> **AIに人格をいれれば、「書く機能」だけでなく、「決める機械」「問う機械」「投票する機械」もより実用的に使えるはず**

## `/voice-decide` ― 「あの人ならどう決めるか」を肩代わりさせる
例えばアプリに新機能を足すとき、AとB のどちらの機能がよいか悩む。Claudeに聞くと「両方にメリットがあります」と返ってくる。そりゃそうだろう、と思いつつ、判断材料にはならない。

本当はユーザーインタビューを回したいんだけど、企画初期の30分でやれることじゃない。ここで欲しいのは「自分の頭の中にいないタイプの人」がどう決めるか、という擬似的な声ですよね。

`voice-decide` は、指定した人格を被せた上で「**必ず1つ選べ。中立保留は禁止**」を構造で強制する。返ってくるのは決断と、その思考過程。決断そのものより、**判断軸が言語化されるところが参考になる**。

実例として、下記の質問を `/voice-decide istj` に渡した結果です。

```
育児記録アプリに次に追加するなら下記のどれを選ぶべき？ 
A) AIで離乳食の写真から栄養を解析 
B) 夫婦でリアルタイム共有できる育児ログ 
C) 子供の発達マイルストーン通知
```

```
選択：B) 夫婦でリアルタイム共有できる育児ログ

Aはまだ実用段階ではない。AIの誤判定で離乳食に不安を抱える親が
出るリスクの方が、便利さより重い。Cは需要はあるが、競合の
育児アプリがほぼ同等の通知を実装済みで、優位性が薄い。

Bは「夫婦の認識ずれが記録の不在から生まれる」という、毎日
起きる実需に直接当たる。授乳の時刻も、寝かしつけの長さも、
口頭で共有しようとすると必ずズレる。記録が共通言語になる。
地味だが、毎日開く理由になる機能はここだ。
```

> INTJ：「論理的思考力」「圧倒的な独立心」「戦略的な計画性」な性格タイプ


選択結果ももちろんだが、何より**思考のプロセスがわかるのがリアルな人間の意見みたいでとても参考になる。**アプリ開発に限らず、企画判断の最初の一発目として使えると思う。

## `/voice-vote` ― チーム提案をバーチャルでリハーサルする
`voice-vote` は16人格に「賛成 / 反対 / 保留」で投票させて、可決ラインを越えるかを判定するスキル。

実用上、一番リアルになるのは **実在するチームメンバーの人格を割り当てて投票させる** やり方です。

例えば、下記みたいな配役を組む。
| チーム内の役割 | 割り当てる人格 | 人格の特徴 |
|---|---|---|
| マネージャー | ENTJ | 戦略的な計画性 |
| 頑固な古参エンジニア | ISTJ | 揺るぎない継続性 |
| 自由を主張する若手 | ENTP | 旺盛な反抗心 |
| 調整役の先輩 | ESFJ | 調和重視の調整力 |
| ロジック担当のシニア | INTP | 鋭利な論理的思考力 |

チーム会議の前に提案資料を Claude に渡してこの Skill を使えば、バーチャルで一回流し本番に出すスライドを直せる。


実例として、「定例ミーティングを週2回から週1回に減らす」という質問を、想定チームを模した5人格（ENTJ・ISTJ・ENTP・ESFJ・INTP）で投票させた結果です。

```
ENTJ (マネージャー): 賛成 — 意思決定の速度を上げる方が優先。情報共有はSlackで足りる。
ISTJ (古参エンジニア): 反対 — 既存の運用で回っている。減らすなら代替の同期手段を
       先に運用設計してから出せ。
ENTP (若手): 賛成 — そもそも週2は多すぎ。ゼロにする選択肢も検討に値する。
ESFJ (先輩): 保留 — 新人のフォローが薄くなる懸念。完全に消すなら別の
       1on1強化とセットでないと不安。
INTP (シニア): 賛成 — 議論の密度に対して時間が長すぎる。減らして異論ない。

集計: 賛成3 / 反対1 / 保留1
判定: 可決圏。ただしISTJの「代替運用先出し」とESFJの「新人フォロー」
への対応案がないと、本番の会議で同じ指摘で止まる。
```

> ※ /voice-vote の実出力（人格指定モード）。

**この出力で本当に効くのは「賛成3で通った」ではなく「ISTJ (古参エンジニア) が代替運用を要求している」「ESFJ (先輩) が新人を心配している」が事前に見えること**。本番までに資料へ一段落足しておけば、その指摘は会議で出る前に潰せる。

提案を出す前のリハーサルとして、地味に強い。


## `/voice-ask` ― 答えさせるのをやめて、問わせる
`voice-ask` は回答・提案・共感を一切禁じて、**人格を入れて問いだけを返す** スキル。使ってみたら、用途が2つ見えてきた。

### 用途1: プレゼンの質疑応答対策

社内発表の前日、スライドや企画書を `voice-ask` に投げる。各人格が「自分ならここを突く」という質問を返してくる。ENTJは「で、これ何ヶ月で投資回収するの？」、INTPは「その前提の根拠どこにある？」、ESFJは「実装する側のチームには事前合意取れてる？」、と切り口がはっきり分かれる。

質疑応答の練習を同僚に頼むと相手の時間も気も使うが、これなら深夜に一人でできる。しかも複数人格を回せば、本番の質問者の幅をある程度カバーできる。

### 用途2: アイデア出しの壁打ち

「このアイデアどう？」とAIに相談すると、「いいですね！」で終わる。一人ブレストの相手としては、味気なさすぎる。

`voice-ask` は問い返してくるので、会話が縦に伸びる。しかも人格を切り替えれば角度が変わるので、自分一人で複数視点の壁打ちができる。

実例として「個人開発のプラグインを来年中にマネタイズしたい」を `/voice-ask entp` に投げた結果：

```
- そもそも「マネタイズ」がゴールなの？それとも「マネタイズ
  できるほど価値がある証明」が欲しいだけ？逆に無料で爆発的に
  使われた方が嬉しくない？

- 来年中って区切り、なぜ「来年中」？半年でも再来年でもなく
  来年中にする根拠は、市場側の事情？それともあなたの飽きる
  タイミングの予測？

- Claude Code自体が来年も今の形で存在してる前提、どれくらい
  信じてる？プラットフォーム側が公式で同等機能を吸収してきたら、
  その瞬間あなたのプラグインの価値はどう変わる？

- 課金してもらう相手って、個人開発者？企業？その2つは全然違う
  商売だけど、どっちで戦う想定？というか両方狙ってるなら、それは
  戦略じゃなく優柔不断では？

- 「マネタイズしたい」と言いつつ、今までの個人開発で最後まで
  やり切ったプロジェクトって何個ある？今回だけ続けられる理由は？
```

> ENTP：「挑発的な探究心」「アイデアの瞬発力」「権威・ルールへの反抗」な性格タイプ

正直、刺さりすぎる。とくに最後の「今までの個人開発で最後までやり切ったプロジェクトって何個ある？」は、自分がスルーしていた前提を引きずり出してくる。

**答えてくれるAIより、問い返してくるAIの方が、思考整理には効く**。これを発見したのは作りながらだった。





# インストールと使い方

```
/plugin marketplace add yukurash/16minds-plugin
/plugin install 16-minds@16-minds-marketplace
```

呼び出し例：

```
/voice-write enfp Qiita記事を読まれるためのコツ
/voice-write intj --format=post 月曜の自分への激励
/voice-decide entj 土曜の使い方 A) 釣り B) 映画 C) 読書
/voice-ask entp 個人開発を来年マネタイズしたい
/voice-vote Slack通知を業務時間外オフに統一する
```

自然言語でも発火する。「ENTPでツッコんで」「INTJで書いて」みたいな日本語でも届く。

リポジトリは下記。

https://github.com/yukurash/16minds-plugin
