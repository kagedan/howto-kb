「まめに `/clear` したらトークン節約できる」というプラクティスは、完全に常識と化してるので私も深く考えずに実践していました。  
実際に、これは効果覿面で日次や週次で見ても明らかに消費されるトークンが減りました。

しかし、具体的にどのタイミングでコンテキストをクリアすべきなのかわからなかったり、コンテキストとコンテキストウィンドウなど術語の意味を混同していたりしたので、勉強しなおしました。  
この記事はそのまとめです。

## コンテキストとは何か

生成AIは入力をもとに次のトークンを予測するという処理を繰り返します。  
このときモデルに与えられる入力全体のことをコンテキストと呼びます。

### 初回のプロンプトで考えるコンテキストの内訳

このコンテキストの具体的な中身を理解するために、まず初回のプロンプト入力時を例に考えてみます。

例えば、`claude` コマンドなどで既存の会話履歴を引き継がない状態でセッションを開始する場面を想像してみましょう。  
ここで、初回のプロンプトを入力するとします。

このセッションではまだ初回のプロンプトしか入力してないので、ここ「コンテキスト = プロンプト」ではないかと思われるかもしれません。  
しかし、そうではありません。  
実際には、システムプロンプトというAIの挙動をあらかじめ規定する情報も入力に含まれています。

このシステムプロンプトとユーザーが入力するテキストを区別して、後者はユーザープロンプトと呼ばれることがあります。  
明確に区別した方が齟齬が防げると思うので、以降、ユーザーが入力するテキストのことは「ユーザープロンプト」と表記することにします。

また、システムプロンプト以外にも、AIが利用できるツール（関数）の定義や、検索結果、あるいはユーザーの現在地といったメタ情報が含まれている場合があります。  
このことから、初回であってもコンテキストはユーザープロンプト単体ではないことがわかります。

Claude Code であれば `/context` コマンドでコンテキストの中身を実際に見ることも可能です。

## コンテキストウィンドウとは何か

### コンテキスト肥大の過程

AIは推論の結果を返しますが、それをもとにユーザーが何か新しい指示を出すというのが一般的です。  
つまり、ユーザープロンプトを繰り返し入力していくというのが一般的な用途です。

これを繰り返すとコンテキストが徐々に肥大していきます。  
これは、過去の自分の質問とAIからの回答もすべてコンテキストとして次の入力に引き継がれるためです。

### コンテキストウィンドウとコンテキストの違い

生成AIは会話履歴を「内部メモリとして保持」しているわけではありません。  
過去のやり取りを毎回プロンプトの一部として送り直しています。  
そのため、セッションが長くなればなるほど、一回の推論で処理（消費）されるデータ量も増えていきます。

生成AIとのセッションが長くなった場合に、Auto-compact という処理が行われることがあります。  
これはコンテキストウィンドウの制限に近づいたときに、ツール側が会話履歴を要約してコンテキストを縮小する処理です。

コンテキストウィンドウとは、モデルが一度の推論で入力として扱えるトークンの最大範囲のことです。  
例えば Claude 系モデルでは、一般的に 200k tokens 程度のコンテキストウィンドウが提供されています。

つまり、コンテキストウィンドウは「モデルが扱える最大サイズ」であり、その中に実際に入力されているテキストがコンテキストです。  
両者は似た言葉ですが、同じ意味ではありません。

## なぜコンテキストを節約すべきか

以下の理由からコンテキストは節約すべきです。

* 無駄なトークン消費を避ける
* 精度の低下を予防する
* 意図しない情報損失を避ける

### 無駄なトークン消費を避ける

これは自明です。

例えば、課題が2つあるとしてそれらは依存関係がなく、並行して進められるものだとします。  
課題Aを解決するためのセッションで、コンテキストを維持したまま課題Bも進めてしまうと無駄なトークン消費が発生します。

なぜなら、課題Aの履歴が課題Bのために入力するユーザープロンプトといっしょにコンテキストにのってしまうからです。  
課題Aの履歴は課題Bの問題解決に寄与しないのですから、存在しているだけで常に構造に負担をかける死荷重 (dead load)となってしまいます。

このことを避けるために、タスクを可分な単位で分割して終わったらまめにコンテキストはクリアするという手法は有効です。

### 精度の低下を予防する

AIはコンテキスト内の全ての情報を等しく重視できるわけではありません。  
情報量が増えるほど、本来集中すべき「現在の指示」に関連する重要な情報が、過去の不要なやり取りの中に埋もれてしまうことがあります。

つまり、コンテキストがウィンドウ内に収まっていても、情報の密度が薄まれば精度が落ちる可能性も高くなるため、これを防止する目的でのコンテキストクリアは有効な手法です。

### 意図しない情報損失を避ける

コンテキストウィンドウから溢れた情報は、AIの記憶から完全に消え去ることになります。

コンテキストが膨らむと Auto-compact が発生する事例は先に触れていますが、ツールがコンテキストを要約する過程で重要な情報が欠落してしまう可能性があります。

後で必要になるはずだった細かい仕様やニュアンスが「不要なもの」として切り捨てられるという事態を避けるために、Auto-compact が実行される前にユーザー主導で必要な情報は保全するというアクションを行った方が得策である場合があります。

## いつコンテキストをクリアすべきか

あくまで目安ですが、前章の内容を踏まえて以下が一般的です。

* タスクの完了時
* 議論が停滞・堂々巡りになった時
* 前提条件の大きな変更
* Auto-compact (自動要約)の直前

### タスクの完了時

理由が明確なので割愛

### 議論が停滞・堂々巡りになった時

これは、AIが同じ誤りを繰り返したり、指示が意図通りに伝わらなくなった状況のことを指します。  
蓄積された失敗のパターンがAIの推論をバイアスさせている可能性があるため、一度リセットして前提を再提示すべきタイミングと考えることができます。

### 前提条件の大きな変更

これは、仕様の抜本的な見直しや技術スタックの切り替えなど、過去のやり取りが現在の最適解と矛盾するようになった時点です。  
古い情報に基づいた推論を排除するために仕切り直しとしていいタイミングです。

### Auto-compact (自動要約)の直前

ツールによる自動要約より、自ら重要な結論や未解決事項を抽出して整理したうえで、クリーンな環境で再開することが有効であるケースもあります。

## 個人的にやっている節約

以下は既出の事柄を踏まえた上で個人的に実践しているプラクティスです。  
※ 有効性の程は不明なので参考まで

### 単純なユーザープロンプトが本当に必要か考える

例えば、「ok」や「commitしてください」といった短いユーザープロンプトを送るケースを考えます。

一見すると数トークンしか消費していない軽いリクエストに見えますが、実際にはそのリクエストには過去の会話履歴がすべて含まれます。

前述の通り、ユーザープロンプトは毎回、履歴全体とセットで送信されることになります。  
つまり、一言単位のユーザープロンプトでも、その時点までに蓄積されたコンテキストを丸ごと入力として再評価させることになります。  
これはイメージとしては、差分ではなく「フルスナップショットを毎回リクエストに含めて再送している」ような挙動です。

"ok commit" のようなわずかなトークンの命令でも、場合によっては10万トークン規模の履歴とセットで送信されることになります。

そのため、単なる確認やリアクションのようなユーザープロンプトは極力省き、本当にそのターンが必要かを考えた上でAIに指示を出すようにしています。

### 無駄な反復を避ける

前節のような「ok commit」といったユーザープロンプトを送ると、大抵AIは「次は何をしますか？」といった応答を返してきます。  
このように質問が予測できる場合はあらかじめ「コミットしたら、次は〜をやって」というふうにまとめて1回で指示をすると、不要なラウンドトリップを減らせます。

追加の情報や意思決定を含まないやり取りで、その時点のコンテキスト長分のコストを丸ごと発生させてしないように注意しています。
