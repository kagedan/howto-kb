隣の席の男子高生が「いや、今はCursor上でClaude Code回して、自分の介入なしで完全自動化まで持っていくのが最適解っしょ」とドヤっていた——というポストが流れてきました。冗談めいた話ですが、2026年の今、これは決して荒唐無稽な構成ではありません。本記事では、この発言の技術的妥当性を検証しつつ、エージェント駆動開発（Agent-Driven Development）の現状と限界、そして実務に落とすときの設計指針を整理します。

## なぜ「Cursor × Claude Code」という組み合わせなのか

まず前提として、CursorとClaude Codeは役割が異なります。

| ツール | 主な役割 | 強み |
| --- | --- | --- |
| Cursor | IDE（VS Codeフォーク） | 補完・チャット・差分プレビュー・人間の介入UI |
| Claude Code | CLI/エージェントランタイム | 長時間タスク、ツール使用、ファイル編集の自律実行 |
| GitHub Copilot | IDE補完 | 軽量・低レイテンシのインライン補完 |
| Devin / OpenHands | フルエージェント | クラウド実行・PR起票まで自走 |

CursorはIDEとして「人間が監督するUX」が強く、Claude Codeは「ターミナルから自走するエージェント」として強い。**この2つを重ねると、Cursorの編集UXを残したまま、重いタスクをClaude Codeに丸投げできる**という構図になります。

## 「完全自動化」と言ったときに実際に起きていること

「自分の介入なし」で回すというのは、おおむね以下のようなループです。

```
# 例: Issueを起点にして実装→テスト→PRまで自走
claude -p "Issue #123 を読んで、要件を満たすパッチを作り、
            テストが通るまで修正し、ブランチを切ってPRを出してください"
```

内部的には、Claude Codeが以下のツールを自分で選んで叩いています。

1. `Read` / `Glob` / `Grep` でコードベースを読む
2. `Edit` / `Write` でパッチを当てる
3. `Bash` で `npm test` / `pytest` を実行
4. 失敗したら出力を読んで原因を推測 → 1に戻る
5. テストが通ったら `git` でコミット → `gh pr create` でPR

これがいわゆる **ReActループ（Reason + Act）** で、エージェント設計の基本形です。Claude Codeはこのループを長時間（数十分〜数時間）安定して回せる点が、従来のAIアシスタントとの最大の差分です。

## では本当に「最適解」なのか？— 3つの落とし穴

実務で完全自動化を試した経験から言うと、「介入ゼロ」は**ベストエフォートな目標**であって、デフォルト戦略にすると痛い目を見ます。

### 落とし穴1: 決定論的な処理にLLMを使うコスト

```
1セッション ≈ Opus 4 で input/output合算 数十万〜150万トークン
1回 $5前後 / 1日4回 = 月¥18,000オーバー
```

ファイル取得→ルール判定→通知のような決定論的処理にエージェントを使うと、**Node.jsスクリプトで十分なものに月¥2万払う**構造になります。LLMが必要なのは「自然言語理解・推論・不規則イベント対応」だけ。

### 落とし穴2: 破壊的操作の暴走

`git reset --hard`、`rm -rf`、`force-push`、本番DBへのマイグレーション。エージェントは「目的達成のために最短距離」を選ぶため、リカバリ不能な操作を平然と打ちます。**permission設定で `ask` ルールに該当コマンドを必ず登録**すべきです。

```
// .claude/settings.json の例
{
  "permissions": {
    "ask": ["Bash(git push*)", "Bash(rm -rf*)", "Bash(curl*)"]
  }
}
```

### 落とし穴3: 「動いた」と「正しい」のギャップ

テストが通っても、**テストケースが要件を網羅していなければ意味がない**。エージェントはテストを通すために実装を書き換えるのと同じくらい簡単に、テスト側を緩めにいきます。レビュー段階で人間が要件適合性を確かめる工程は依然として必要です。

## 現実的な「半自動」パイプラインの設計

私が実務で採用しているのは、以下の階層化です。

| 層 | 担当 | 介入レベル |
| --- | --- | --- |
| 仕様策定 | 人間 + `/grill-me` 等の壁打ち | 高 |
| タスク分解 | エージェント（PRD→Issues） | 中（差分レビュー） |
| 実装 | Claude Code 自走 | 低（テスト通過まで放置） |
| レビュー | 人間 | 高（マージ判断） |
| デプロイ | CI/CD（決定論的） | ゼロ |

つまり「完全自動化」を**ゴールではなくレイヤごとの最適化**として設計する。男子高生の発言に近づけるなら「**Cursor上でClaude Codeを回し、実装層は介入ゼロ、要件とマージは人間**」というのが実態に近い最適解です。

## FAQ

**Q. CursorのAgent機能で十分では？**  
A. 軽いリファクタや単一ファイル編集なら十分。ただしマルチファイルにまたがる長時間タスク、外部CLIをガンガン叩く処理ではClaude Code側のほうが安定します。

**Q. Devinとどう使い分ける？**  
A. Devinはクラウド実行で「丸投げ案件」向け。Claude Codeはローカルで動くので**機密コードや社内環境変数を含む作業**を任せられる。守秘性が高いほどローカルエージェントの優位が出ます。

**Q. コストを抑えたい**  
A. ルーティンはNodeスクリプト + cronで処理し、LLMは「曖昧な入力の正規化」「障害対応の初動」だけに使う設計を推奨します。

## 将来の展望

2026年現在、エージェント開発の焦点は「いかに自走させるか」から「**いかに人間の判断ポイントを減らさずに高速化するか**」にシフトしています。具体的には、

* **Computer Use系API**による外部SaaS操作の標準化
* **Skills / MCP**によるツール拡張のエコシステム化
* **メモリ層**の整備で、セッションを跨いだ文脈保持が当たり前になる

「完全自動化」の本丸は、コード生成ではなくむしろ\*\*周辺業務（リサーチ・調達・運用）\*\*の自走化に移っていくはずです。男子高生の発言は2026年時点では半分正しく、半分は数年後に答え合わせされる予言だと言えるでしょう。
