> I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops.

Claude Code を作った Boris Cherny の言葉だ。これが意味するのは、エージェントに指示を出す主体が人間からコードに移った、ということになる。今週、同じことを別々の角度から言う文章がほぼ同時に出た。Armin Ronacher の「The Coming Loop」、Addy Osmani の「Loop Engineering」、そして日本語圏でも「個人のプロンプト術をやめて、チームで回るAI駆動開発ループを作った話」が Zenn のトレンドに上がっている。

通説はこうだった。エージェントの成果はプロンプトの質で決まる、だからプロンプトを磨け。今週の観測はその通説を裏返している。磨く対象はもうプロンプトではなく、エージェントを回すループそのものに移った。本記事では、このループを「制御構造」として設計するとき実装上なにが問題になるのかを、アーキテクチャの目線で整理する。先に結論を置いておく。ループは長く回すほど賢くなるわけではない。賢さを決めるのは回す回数ではなく、止め方と検証だ。

## ループの正体は二重構造

まず用語を分けておきたい。いま多くの人が「エージェントが自律的に動く」と呼んでいるものは、実際には入れ子になった2つのループからできている。

内側は agent loop だ。モデルがツールを呼び、ファイルを読み、テストを走らせ、1回のセッションのなかで反復して問題を解く。これは LLM 自身が回している。

外側は harness loop と呼ぶ。セッションの外側にいる仕組みが、仕事が終わったかどうかを判定し、文脈を差し替えて再投入し、タスクを待ち行列に積み、必要なら別マシンに逃がす。Ronacher の表現を借りるなら、harness はモデルが本来「I am done（終わりました）」と言うはずだった地点を超えて、タスクを生かし続ける。

![agent loop と harness loop の二重構造を示す状態遷移図。外側の harness が検証で完了を判定し、状態を永続化して文脈を差し替えながら再投入する](https://static.zenn.studio/user-upload/1be5b94b5ad4-20260624.png)

つまり「ループを書く仕事」とは、内側を賢くする作業ではない。外側の harness を設計する作業だ。モデルの賢さは内側で頭打ちになる。外側をどう組むかで、同じモデルが使い物になるかどうかが分かれる。

ここで内側と外側の責務を取り違えると、設計を間違える。内側に「終わったか」を判断させてはいけない。終了判定は外側の責務だ。

## 難しさは消えず、移動した

「もうプロンプトしない」と聞くと、自動化が進んで楽になった話に聞こえる。実際は逆だ。難しさは消えていない。ボトルネックがモデルの推論品質から、ループの制御ロジックへ移っただけだ。

外側を設計するとき、最低でも次の3点を自分で持つ必要がある。停止条件、検証ゲート、状態の永続化。それぞれ素朴に書くと事故る。

1つめの停止条件。モデルの「終わりました」を信じてはいけない。終了は外側が検証で判定する。素朴な疑似コードで書くとこうなる。

```
# アンチパターン: モデルの自己申告で止める
while True:
    result = agent.run(task)
    if result.says_done:   # モデルが「終わった」と言ったら止める
        break              # → 自信を持って間違えたまま止まる
```

```
# 制御点を入れた harness loop
state = load_state(task_id)          # 反復をまたいで状態を持つ
for attempt in range(MAX_ATTEMPTS):  # 無限ループにしない
    result = agent.run(task, context=state.context)
    verdict = verifier.check(result) # 実装者とは別の主体が採点する
    state.record(result, verdict)    # 進捗を永続化する
    if verdict.passed:
        break
    if budget.exceeded():            # コスト上限で必ず抜ける
        escalate_to_human(task, state)
        break
    state.context = revise_context(state, verdict)  # 文脈を差し替えて再投入
```

2つめの検証ゲート。ここが Osmani の指摘でいちばん効く所だ。実装したエージェントに自分の成果を採点させてはいけない。彼は sub-agent を役割で分け、explorer・implementer・verifier を別々に立てることを挙げている。自分の答えを自分で丸つけする構造は、自信のある誤りをそのまま通してしまう。

3つめの状態の永続化。エージェントは実行と実行のあいだで記憶を失う。Osmani は完了した作業と次の手順を markdown や Linear のボードに書き出して持たせる primitive を挙げている。状態を外に出さないループは、毎回ゼロから同じことを再発見しはじめる。

![自律ループの3つの制御点（停止条件・検証ゲート・状態の永続化）をまとめたインフォグラフィック](https://static.zenn.studio/user-upload/71d2a33eca81-20260624.png)

|  | プロンプトを書く時代 | ループを書く時代 |
| --- | --- | --- |
| 磨く対象 | 指示文の言い回し | 停止条件・検証・状態 |
| 終了判定 | 人間が見て止める | harness が検証で判定 |
| 失敗の主因 | 指示が曖昧 | 制御点が無い |
| 必要なスキル | 言語化 | 制御構造の設計 |

## ループを長く回すほど、コードは悪くなりうる

ここからが本題の2段目だ。「回せば回すほど良くなる」という直感は、長寿命のコードでは裏切られる。

Ronacher が鋭いのは、いまのモデルが書くコードの癖を具体的に指摘している点だ。モデルは不正な状態を作れなくする代わりに、不要なフォールバックを足した防御的で複雑なコードを書く。強い不変条件を持たず、きれいな抽象を作らずにコードを複製する。彼の言葉では、モデルは「例外を死ぬほど恐れている（mortally terrified of exceptions）」。だからローカルな防御を反復のたびに足していく。

問題は、この癖がループと掛け算になることだ。1回ごとに局所的な防御が積まれ、複雑さが反復ごとに増幅する。検証ゲートが「テストが通るか」しか見ていないと、テストは通るが誰も読めないコードが量産される。回した回数が、品質ではなく負債の量に化ける。

もう1つ、見えにくい因果がある。理解の負債（comprehension debt）だ。Osmani も Ronacher も同じ警告を出している。自分の注意を向けずに生成させたコードは、自分で説明できなくなる。Ronacher はさらに踏み込んで、最悪の場合「人間にとって保守が難しいだけでなく、機械の参加を保守モデルの前提に組み込んだコードベース」を作ってしまう、と書く。モデルへのアクセスが規制やコストで断たれた瞬間に、誰も触れないコードベースが残る。これはループの設計ミスというより、ループに何を任せるかを間違えた帰結だ。

## ループが効くかは「検証関数が書けるか」で決まる

では、どこにループを使い、どこに使わないのか。Ronacher はループがうまくいく領域として、コードの移植（Bun の Zig から Rust への移植を挙げている）、性能探索とベンチマーク、セキュリティ走査と調査、PoC 生成を並べる。彼自身はこれを「寿命が長くない成果物」や「機械的な変換」と説明している。

ここで私の見立てを1つ出す。この4領域に共通するのは「寿命が短い」ことではない。検証関数が書けることだ。移植は元の挙動と照合できる。ベンチマークは数値で測れる。セキュリティ走査は再現する脆弱性で確かめられる。PoC は動くか動かないかだ。どれも、成果が正しいかを機械が判定できる。

![ループを任せていい仕事の見分け方。判断軸は検証する関数を書けるかどうか、という左右対比図](https://static.zenn.studio/user-upload/2cf6f959e276-20260624.png)

逆に、長寿命の設計判断、きれいな抽象の選択、ドメイン特有の正しさは、検証関数が書きにくい。だからここはループを長く回しても収束せず、人間のレビューがボトルネックのまま残る。タスクの種類で線を引くより、「このタスクに自動の検証関数を書けるか」で線を引いたほうが、ループを任せる範囲を間違えにくい。

この基準は実装にそのまま落ちる。新しい仕事にループを当てる前に、verifier.check() の中身に何を書けるかをまず考える。書けないなら、それはまだ人間が見るべき仕事だ。

## まとめ

今週の3本が同じ方向を指していたのは偶然ではない。AI 開発の主戦場が、プロンプトという入力の言い回しから、ループという制御構造へ移ったということだ。そして制御構造の設計には、停止条件・検証ゲート・状態の永続化という、ソフトウェア工学が昔から持っていた道具がそのまま要る。

実務に効く持ち帰りは1つに絞れる。ループに新しい仕事を任せる前に、その成果を機械が採点する関数を書けるかを確かめる。書けるならループは強力に効くし、書けないならループは負債を増幅する装置になる。プロンプトエンジニアリングが過渡的なスキルだったように、次に過渡的になるのは「とりあえず長く回す」運用だと私は見ている。残るのは、止め方と検証を設計できる人の手だ。
