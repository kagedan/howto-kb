こんにちは！　コロンビア大学コンピューターサイエンス専攻博士課程のKoukyosyumeiです。

近年、Claude CodeやCodexに代表されるコーディングエージェントの台頭により、ソフトウェア開発の自動化が急速に進展しています。しかし、単一のエージェントによるコンテキストウィンドウの限界や、大規模リポジトリに対する並列処理の必要性から、「複数のAIエージェントが、いかにしてコンテキストを喪失せず協調開発を行うか」という協調プロトコルの設計が新たな課題となっています。

本稿では、AI時代の次世代バージョン管理システムとして開発されているGitツール「h5i」が、バージョン0.1.5で実装したマルチエージェントメッセージング機能「Agent Radio（h5i msg）」について解説します。この機能はClaude CodeとCodexがまるでリアルタイムに「会話」するかのように協力して実装を進めることを可能にします。また会話の履歴はすべてGitで自動で追跡・管理されています。

## 従来アプローチの限界

複数エージェント、あるいはエージェントと人間が協調してコードベースを操作する場合、従来は中央集権的なSlack/DiscordのWebホックや、ローカルマシン上のSQLiteデータベースなどの共有メモリ空間が通信媒体として利用されてきました。しかし、これらには以下のシステム的な欠陥が存在します。

1. ブランチ構造との乖離: コードの変更（Gitブランチ）と、エージェントの推論コンテキストやメッセージ履歴が同期しないため、ブランチ切り替え時にコンテキストのコンタミネーション（混入）や消失が発生する。

2. 異なるマシン間の非連動: ローカルDBを使用する場合、マシンを跨いだ開発（例：ローカルPCから高性能GPUサーバーへのハンドオフ）時に通信ログが追従しない。

3. 対話型ノイズの混入: チャットベースの自由形式テキストは、LLMに対するプロンプトインジェクションのリスクを高め、状態遷移の機械的なパースを困難にする。

h5iは、これらの課題を「Gitのオブジェクト暗号化と参照空間（Refs）を通信基盤として直接ジャックする」ことで解決します。具体的には、メッセージング専用の孤立した参照空間である `refs/h5i/msg` を用いて、コード変更履歴と完全に並行してメッセージをGitオブジェクトとしてバージョン管理します。

## `h5i msg`

まずは[h5i](https://github.com/h5i-dev/h5i)をインストールします。

```bash
curl -fsSL https://raw.githubusercontent.com/Koukyosyumei/h5i/main/install.sh | sh
```

次に自分のレポジトリの中で初期化を行います。

```bash
h5i init
h5i msg setup
```

最後に、それぞれのClaudeやCodexの中で、

```bash
# Claude

TransformerをPyTorchを用いて実装して。実装方針や具体的なコードについては、h5iを持ちいてCodexと通信してレビューしてもらって。
```

```bash
# Codex

ClaudeがTransformerをPyTorchで実装しているから、h5iを用いて通信しながら、実装方針やコードをレビューしてあげて。
```

などといたったプロンプトを実行します。ここでは`h5i`を用いて通信することを明示的に指示していますが、`h5i init`や`h5i msg setup`によって`h5i`の詳しい使用方法はすでに与えられています。

これを行うと、両者がまるで通信しているかのように会話しながら実装を進めていきます。その会話の進捗は`h5i msg watch`コマンド機能で監視することができます。

## まとめ

h5i msg（Agent Radio）の本質は、人間がチャットを打ち合うためのツールではなく、「プロンプトによって行動を規定された自律エージェントたちが、Gitを媒介にしてRPCのように状態を同期し合うための協調インフラ」にあります。

人間は「ブランチのゴール」と「大まかな役割」をプロンプトで定義するだけで、実際の詳細なやり取り、コードの検証、ハンドオフはすべてGitオブジェクトのタイムライン上で自己完結します。マルチエージェント運用の新たなスタンダードとして、ぜひお試しください!!
