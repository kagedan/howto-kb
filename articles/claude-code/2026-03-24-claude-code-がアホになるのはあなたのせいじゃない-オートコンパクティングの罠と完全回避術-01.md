## はじめに

Claude Code を使っていると、こんな経験はないでしょうか？

> 「30分前に決めたアーキテクチャ方針を Claude が忘れて、全然違うコードを生成してきた……」

これは **オートコンパクティング** という仕組みが動作した結果です。  
本記事では、Claude Code のメモリ・コンテキスト管理の仕組みを初学者向けに解説し、情報損失を最小化する実践的な対策を紹介します。

---

## 📌 3行まとめ

* Claude Code の記憶は有限で（モデルにより **200K〜1M トークン**）、**いっぱいになると古い会話が自動的に要約・圧縮される**（＝オートコンパクティング）
* 要約は完璧ではなく、**序盤の詳細な指示・設計の意図ほど消えやすい**ため、前提崩壊が起きる
* 対策の本質は「**消えてほしくない情報は事前に CLAUDE.md に書き、Claude 自身に記憶させるなら Memory MCP を使う**」こと

---

## 🔍 オートコンパクティングとは？ ― まず 1 分で理解する

Claude Code のオートコンパクティングは、**コンテキストウィンドウ（会話の作業領域）が約 80〜83% 埋まったときに自動発動する、会話履歴の要約・圧縮機能**です。

```
会話が積み重なる
      ↓
コンテキスト使用率が約 80〜83% に到達
      ↓
【オートコンパクティング発動】
  古いメッセージ → 要約テキスト 1 つに圧縮
      ↓
空き領域を確保して会話を継続
```

![](https://static.zenn.studio/user-upload/1b3db4ccf224-20260324.png)  
*オートコンパクティングが発動した際*

**ポイントは 2 つです。**

1. **作業フローがぶった切られる可能性がある** ― 自動発動のタイミングは選べないため、実装の途中でいきなり走ることがある
2. **手動で `/compact` を実行すれば防げる** ― 区切りのよいタイミングで自分から圧縮すると、引き継ぎたい情報を Claude に明示できるため要約の質が上がる

```
# 手動コンパクション（観点を指定するとより良い）
/compact focus on the API design decisions and architecture
```

この 2 点を押さえた上で、以降では仕組みの詳細と対策を解説します。

## 1. Claude Code の「記憶」とは何か

### コンテキストウィンドウ ＝ 作業机の広さ

Claude Code（および LLM 全般）の記憶は、 **コンテキストウィンドウ** と呼ばれる有限のメモリ空間に格納されています。

![](https://static.zenn.studio/user-upload/aaeddab06c91-20260324.png)

* ファイルを読む・コードを書く・テストを実行するたびに、この机のスペースが埋まっていく
* 机が満杯になると、古い内容を「要約」して空きを作る ＝ **オートコンパクティング**

### モデル・プランによるコンテキストサイズの違い

コンテキストウィンドウのサイズは**使用するモデルとプランによって大きく異なります**。2026年3月時点の情報です。

| モデル | コンテキストサイズ | Claude Code での適用条件 |
| --- | --- | --- |
| **Sonnet 4.6** | **1M トークン（約75万字）** | モデル ID に `[1m]` サフィックスが必要（未指定時は 200K で動作） |
| **Opus 4.6** | **1M トークン（約75万字）** | Max / Team / Enterprise で自動適用。Pro は `/extra-usage` で有効化 |
| Sonnet 4.5 / Sonnet 4 | 1M（ベータ） | ベータ扱い |
| Haiku 4.5 など旧世代 | **200K トークン（約15万字）** | 変更なし |

!

**Sonnet 4.6 で 167K 付近でコンパクティングが発動する場合、Sonnet 4.6 が 200K モードで動いています。これは正常な動作です。**

* **Opus 4.6**：Max/Team/Enterprise プランで 1M が自動適用
* **Sonnet 4.6**：1M の自動適用は保証されておらず、プラン・バージョンによって動作が異なる

Sonnet 4.6 で 1M を試みる場合、公式ドキュメントでは `/model` コマンドでの指定や `[1m]` サフィックスの使用が案内されています。ただし「Sonnet 4.6 with 1M context is not available for your account」と表示されるケースも報告されており、アカウント・プラン条件によって利用可否が変わります。

```
# /model コマンドで指定する場合（利用可能な場合のみ）
/model sonnet[1m]
 
# 環境変数で指定する場合
export ANTHROPIC_DEFAULT_SONNET_MODEL=claude-sonnet-4-6[1m]
```

確実に 1M コンテキストを使いたい場合は **Opus 4.6 への切り替え**が現時点では最も安定した方法です。

![](https://static.zenn.studio/user-upload/b8451969b2a3-20260324.png)  
*Visual Studio Codeの拡張機能でモデルを指定する場合は1Mモデルを指定する。(/modelコマンド)*

## 2. オートコンパクティングとは何か

### ガベージコレクション（GC）とは違う

オートコンパクティングはプログラミングのガベージコレクションとは異なります。GC は「不要なメモリを解放」しますが、オートコンパクティングは **「古い会話を要約テキストで置き換える」** 処理です。

### 処理の流れ

### 発動タイミング

| 設定 | タイミング |
| --- | --- |
| デフォルト | コンテキスト使用率 約80〜83%（1M ウィンドウなら約 800K〜830K、200K ウィンドウなら約 160K〜166K トークン時点） |
| 環境変数で調整可 | `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75` 等 |

残り約33Kトークンは、要約処理そのものに使われるバッファです。

## 3. 何が失われ、何が残るのか

### 消えやすい情報

* 具体的な変数名・関数名の由来
* 「なぜその設計にしたか」という理由・背景
* セッション序盤に決めたアーキテクチャ上の細かい制約
* 「〇〇は絶対にやらないで」という否定的な指示

### 残りやすい情報

* 使用言語・フレームワーク（大まかな技術スタック程度の粒度）
* 直近数ターンのやり取り
* 主要なファイル名・ディレクトリ構成

### 結論

> **要約の精度 ＝ コンパクティング後の Claude の「記憶」の精度**
>
> 要約が粗いと、誤った前提でコードを生成し続ける。

公式ドキュメントも明言しています。

> 「ユーザーのリクエストと主要なコードスニペットは保持されますが、会話の初期段階にあった詳細な指示は失われる可能性があります。」  
> — [Claude Code 公式ドキュメント](https://code.claude.com/docs/en/how-claude-code-works)

## 4. 解決策：情報をコンテキスト外に逃がす

コンテキストウィンドウが揮発性の RAM だとすると、解決策は「重要な情報を不揮発ストレージに逃がすこと」です。

```
【揮発】コンテキストウィンドウ（会話履歴）
         ↑ オートコンパクティングで圧縮・消滅
 
【不揮発】 ← ここに逃がす
  ├── CLAUDE.md（毎ターン注入 ※肥大化注意）
  ├── .claude/rules/（分割ルール・必要時に参照）
  ├── Memory MCP（Claude が自律的に読み書き）
  └── docs/・architecture.md 等（明示参照）
```

---

## 5. 対策① CLAUDE.md ― 最も重要な一手

### CLAUDE.md とは

プロジェクトルートに置く Markdown ファイルで、**セッション開始時に必ずコンテキストへ注入される**特別なファイルです。オートコンパクティングが走っても、CLAUDE.md の内容は次のターンで再読み込みされます。

ただし**注入される分だけトークンを消費する**ため、書きすぎは禁物です。変更頻度が低い詳細ルールは `.claude/rules/` に分割し、CLAUDE.md には「毎ターン必要な最小限」だけを書くのが理想です。

### 書くべき内容

```
# CLAUDE.md（プロジェクトルート）
# ここには「毎ターン必要な最小限」だけ書く
 
## プロジェクト概要
- 目的：〇〇を実現するWebアプリ
- 技術スタック：（使用言語・主要フレームワーク）
- デプロイ先：（本番環境）
 
## 絶対に守るアーキテクチャ方針
- （変えてほしくない設計の制約を箇条書き）
 
## 現在の作業状態
- 完了：〇〇機能の実装
- 次のステップ：〇〇
 
## Compact Instructions
コンパクティング時は以下を保持すること：
- 各ファイルの役割と実装済み機能
- アーキテクチャ上の決定とその理由
- 未解決の課題・次のステップ
 
## 詳細ルールの参照先
コーディング規約 → .claude/rules/coding.md
テスト方針 → .claude/rules/testing.md
```

### ポイント

* CLAUDE.md は**毎ターン自動的にコンテキストへ注入される**。書きすぎると圧迫するため「毎回必要な最小限」だけを書く
* 変更頻度が低い詳細ルール（コーディング規約・テスト方針等）は `.claude/rules/` に分割し、必要時に参照させる
* `## Compact Instructions` セクションでコンパクティング時の要約品質を制御できる
* 「やらないこと」も明示する（例：`直接 DB アクセスは〇〇経由のみ`）
* こまめに更新する（Claude 自身に更新させることも可能）

## 6. 対策② 手動 `/compact` ― 自動より品質が高い

オートコンパクティングは「いつ来るか分からない」のが問題です。**区切りのよいタイミングで手動実行すると、要約の質が上がります**。なぜなら、何が重要かを明示できるからです。

```
# 観点を指定して要約させる（focus に続けて引き継ぎたい内容を英語で記述）
/compact focus on the API design decisions and tool definitions

# 単純に圧縮する
/compact
```

> **補足：** 観点指定は英語で書くのが確実です。日本語でも動作しますが、要約品質は英語のほうが安定する傾向があります。

**推奨タイミング：**

* 1つの機能実装が完了したとき
* バグ修正が完了したとき
* 作業の区切り（ランチ前・退勤前など）

> コミットの前に `/compact` を実行するのが習慣になると良いです。

## 7. 対策③ Memory MCP ― コンテキスト外への完全永続化

CLAUDE.md は「人間が書いて管理する」ファイルです。これに加えて、**Claude 自身が記憶を読み書きできる外部ストレージ**として MCP Memory Server が使えます。

### MCP Memory Server とは

Claude 自身がツールとして呼び出せる外部記憶サーバーです。コンテキストウィンドウが圧縮・消滅しても、データは外部ストレージに残ります。実装によってバックエンドが異なり、用途に応じて選択できます。

| 実装 | ストレージ | 特徴 |
| --- | --- | --- |
| `@modelcontextprotocol/server-memory` | JSON ファイル | 公式リファレンス実装。セットアップが最もシンプル |
| `mcp-memory-keeper` | SQLite | チェックポイント機能あり。複数セッション間で共有可能 |
| `basic-memory` | Markdown + SQLite | セマンティック検索対応。人間も読みやすい形式 |
| `mem0` | クラウド or ローカル | ベクトル検索対応。月1万件まで無料プランあり |
| `mcp-memory-service` | SQLite + ベクトルDB | 複数エージェント間での共有・長期記憶に強い |

本記事では最もシンプルな公式リファレンス実装（JSON ファイル）を紹介します。

```
Claude Code
    ↓ ツール呼び出し（stdio トランスポート）
Memory MCP Server（npx プロセス）
    ↓ 読み書き
外部ストレージ（JSON / SQLite / ベクトルDB など実装による）
```

### セットアップ

設定ファイルに追記する方法が最も確実です（パスは絶対パスで指定）。

```
// ~/.claude.json （グローバル設定）
{
  "mcpServers": {
    "memory": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-memory",
        "--memory-path",
        "/Users/yourname/.claude-memory"
      ]
    }
  }
}
```

CLI コマンドで追加する場合（stdio サーバーは `--` でコマンドを区切る）：

```
# stdio サーバーの正しい構文: claude mcp add <name> -- <command> [args...]
claude mcp add memory -- npx -y @modelcontextprotocol/server-memory \
  --memory-path ~/.claude-memory
 
# 追加後に動作確認
claude mcp list
```

### 使い方

セッション開始時に Claude に依頼します：

```
「このプロジェクトの現在の状態を memory から読み込んで」
```

区切りのよいタイミングで：

```
「今決めたアーキテクチャ方針を memory に保存して」
「authentication_system エンティティを作成して、
 JWT採用・24時間TTL・auth/ディレクトリに実装済み
 という情報を記録して」
```

### 保存される情報のイメージ

```
{
  "entities": [
    {
      "name": "agent_loop",
      "entityType": "component",
      "observations": [
        "FastAPI + SSE でストリーミング実装済み",
        "stop_reason で終了を自己判断する設計",
        "MAX_TURNS=10 で無限ループ防止",
        "tools.py でツール定義を分離"
      ]
    },
    {
      "name": "architecture_decision",
      "entityType": "decision",
      "observations": [
        "LangGraph は採用しない（Claude Code の動的ループで十分）",
        "サブエージェントは dispatch_agent ツールで実装",
        "決定日: 2026-03-24"
      ]
    }
  ]
}
```

## 8. 対策④ 発動タイミングを早める

デフォルト（約83%）では、発動時点でコンテキストがすでに汚染され始めていることがあります。**早めに発動させると、要約に使えるトークンが多く残るため、品質が上がります。**

```
# ~/.bashrc または ~/.zshrc に追記
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75
```

| 値 | 説明 |
| --- | --- |
| `75`（推奨） | 品質劣化ゾーン到達前に発動。多くの作業に最適 |
| `60〜70` | 短いタスクや積極的な圧縮を好む場合 |
| `83`（実質上限） | できるだけ遅らせたい場合（これ以上は内部クランプで無効） |

---

## 9. 対策⑤ 重いツール出力をファイルに逃がす

`bash` コマンドの実行結果や大きなファイルの内容は、コンテキストを一気に消費します。

```
# ❌ 大きな出力がそのままコンテキストに入る
find . -name "*.py" | xargs wc -l

# ✅ ファイルに書き出してコンテキストを節約
find . -name "*.py" | xargs wc -l > /tmp/line_counts.txt
# → Claude に「/tmp/line_counts.txt を読んで」と指示する
```

また、テスト結果のような長大なログも同様です。

```
pytest --tb=short 2>&1 | tail -50 > /tmp/test_result.txt
```

## 10. 対策まとめ ― 優先順位付き

| 優先度 | 対策 | 効果 | コスト |
| --- | --- | --- | --- |
| ⭐⭐⭐ | **CLAUDE.md に方針・規約を書く** | 毎セッション確実に注入 | 低（一度書けばOK） |
| ⭐⭐⭐ | **手動 `/compact` を区切りで実行** | 要約品質が上がる | 低（習慣化） |
| ⭐⭐⭐ | **Auto Memory を活用する** | Claude が自動でセッション間記憶を蓄積 | 低（v2.1.59+ でデフォルト有効） |
| ⭐⭐ | **Memory MCP（JSON）で永続化** | セッション跨ぎで明示的な記憶継続 | 中（初期設定が必要） |
| ⭐⭐ | **Memory MCP（SQLite / ベクトルDB）** | 長期記憶・セマンティック検索・複数エージェント共有 | 高（実装選定が必要） |
| ⭐⭐ | **`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75`** | 早期発動で要約品質向上（遅らせる方向には効果なし） | 低（設定一行） |
| ⭐ | **大きなツール出力をファイルに逃がす** | コンテキスト節約 | 中（意識が必要） |

---

## 11. まとめ

Claude Code の「記憶」は、コンテキストウィンドウという有限な空間に存在します。

* **オートコンパクティング** = 古い会話を要約で置き換えること（GCとは違う）
* **情報は完全消失ではなく「劣化」する**（詳細なほど消えやすい）
* **解決の本質** = 重要な情報をコンテキスト外（CLAUDE.md / Memory MCP）に逃がすこと

一番最初にやるべきことは **CLAUDE.md にプロジェクトの方針・規約・作業状態を書くこと**です。これだけで、30分前の決定が忘れられる問題の多くは解消されます。
