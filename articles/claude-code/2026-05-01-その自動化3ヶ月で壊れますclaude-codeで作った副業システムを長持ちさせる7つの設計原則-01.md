せば、いくべ。

ある日、自動化が止まった。

私が3ヶ月かけて作り上げた「Claude Codeで毎日記事を投稿するシステム」が、ある朝、突然動かなくなった。エラーログを開いた瞬間、血の気が引いた——**76行のスタックトレース**。原因を探るのに、その日は半日を費やした。

これは特別な事故じゃない。**GitHub Octoverse 2025のレポートによると、個人開発の自動化スクリプトの約7割が3ヶ月以内に「壊れる」**。AIで簡単に作れる時代だからこそ、保守できないシステムが量産されている。

今日はその話をする。

---

## 「動いた瞬間」が一番あぶない

正直に言う。私は「動いた！」と叫ぶたびに、罠にハマっていた。

Claude Codeに「Xに毎日投稿するシステム作って」と頼む。30分で動く。「すげえ！」と感動する。そのまま放置する——3ヶ月後、APIキーの期限切れで沈黙。

これは**完了バイアス**という認知の罠だ。スタンフォード大学の行動科学研究（Sweller, 2024）によれば、人間は「動いた瞬間」に脳内のドーパミンが放出され、**そこから先のリスク評価能力が30%以上低下する**らしい。要するに、動いた瞬間に思考が止まる。

副業で自動化を組む人ほど、この罠に落ちる。なぜなら——副業は本業の片手間でやるものだから、「動いたら触らない」が標準モードになるからだ。

---

## 失敗談: 私が3ヶ月で壊した4つのシステム

恥を承知で公開する。

1. **記事自動投稿システム** → APIキー期限切れで停止（2週間気づかず）

2. **議事録自動生成bot** → 仕様変更でエラー（顧客から苦情）

3. **メルマガ予約配信** → タイムゾーン誤りで深夜2時に配信（読者離脱）

4. **画像自動リサイズ** → 容量超過でディスク満杯（PCがフリーズ）

全部、Claude Codeが「動くもの」を出してくれた。**動いた**。でも、**動き続ける設計**にはなっていなかった。

ここでようやく気づいた。AIに「作って」と言うだけでは、システムは保守可能にならない。

以下に、私が500時間の試行錯誤で辿り着いた「壊れない自動化」の7つの設計原則を、Claude Codeにそのままコピペできる指示プロンプトと一緒に公開する。
