現在、私が所属するプロジェクトではReact・Next.jsを用いたリアーキテクチャを進めています。

## Claude Codeの設定

Claude Codeでは`settings.json`に設定を記述しますが、今回はその中でも私が推している設定として`model`・`env`・`hooks`・`statusline`を紹介できたらと思います。

<https://code.claude.com/docs/ja/settings>

### model

Claude Codeを使っているとき、Plan モードと実行モードでモデルを使い分けたい場面がかなりあると思います。  
そんなときにいちいち`/model`で切り替えるのは手間です。

そこで紹介するのが`opusplan`という設定です。  
やり方は簡単です。

Claude Codeを起動し、`/model opusplan`とコマンドを実行するだけで、自動でモデルの使い分けの設定ができます。

<https://x.com/so_ainsight/status/2033702652127940728>

### env

私は、`env`には以下の2つの設定をしています。

settings.json

```
{
    "env": {
        "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1",
        "ENABLE_TOOL_SEARCH": "true"
    }
}
```

1つ目の、`"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"`は**agent teams**を有効にする設定です。

<https://code.claude.com/docs/en/agent-teams>

agent teamsとは、役割の異なる複数のエージェントが、共通の目的に向かって自律的にタスクを分担・協力して進める仕組みのことです。

<https://x.com/lydiahallie/status/2019469032844587505>

2つ目の`"ENABLE_TOOL_SEARCH": "true"`はMCPツールを動的に使用するための設定です。  
この設定をすることで、使用しないMCPツールの定義がコンテキストを占有してしまうのを防ぐことができます。

実際に、設定を有効にするだけで、MCPツールのトークン使用量をかなり抑えることができます。  
私自身体感として、`/compact`されるまでのやり取りがかなり多くできるようになると感じています。

<https://x.com/azukiazusa9/status/2005887536070279249>

### hooks

`hooks`は以下のような設定を使っています。  
※ `osascript`などmacOSを前提とした設定となっております。

hooks

settings.json

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "tool == \"Bash\" && tool_input.command matches \"git push\"",
        "hooks": [
          {
            "type": "command",
            "command": "#!/bin/bash\n# Open editor for review before pushing\necho '[Hook] Review changes before push...' >&2\n# Uncomment your preferred editor:\n# zed . 2>/dev/null\n# code . 2>/dev/null\n# cursor . 2>/dev/null\necho '[Hook] Press Enter to continue with push or Ctrl+C to abort...' >&2\nread -r"
          }
        ]
      },
      {
        "matcher": "tool == \"Write\" && tool_input.file_path matches \"\\\\.(md|txt)$\" && !(tool_input.file_path matches \"README\\\\.md|CLAUDE\\\\.md|AGENTS\\\\.md|CONTRIBUTING\\\\.md\")",
        "hooks": [
          {
            "type": "command",
            "command": "#!/bin/bash\n# Block creation of unnecessary documentation files\ninput=$(cat)\nfile_path=$(echo \"$input\" | jq -r '.tool_input.file_path // \"\"')\n\nif [[ \"$file_path\" =~ \\.(md|txt)$ ]] && [[ ! \"$file_path\" =~ (README|CLAUDE|AGENTS|CONTRIBUTING)\\.md$ ]]; then\n  echo \"[Hook] BLOCKED: Unnecessary documentation file creation\" >&2\n  echo \"[Hook] File: $file_path\" >&2\n  echo \"[Hook] Use README.md for documentation instead\" >&2\n  exit 1\nfi\n\necho \"$input\""
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\\\.(ts|tsx)$\"",
        "hooks": [
          {
            "type": "command",
            "command": "#!/bin/bash\n# Run TypeScript check after editing TS files\ninput=$(cat)\nfile_path=$(echo \"$input\" | jq -r '.tool_input.file_path // \"\"')\n\nif [ -n \"$file_path\" ] && [ -f \"$file_path\" ]; then\n  dir=$(dirname \"$file_path\")\n  project_root=\"$dir\"\n  while [ \"$project_root\" != \"/\" ] && [ ! -f \"$project_root/package.json\" ]; do\n    project_root=$(dirname \"$project_root\")\n  done\n  \n  if [ -f \"$project_root/tsconfig.json\" ]; then\n    cd \"$project_root\" && npx tsc --noEmit --pretty false 2>&1 | grep \"$file_path\" | head -10 >&2 || true\n  fi\nfi\n\necho \"$input\""
          }
        ]
      },
      {
        "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\\\.(ts|tsx|js|jsx)$\"",
        "hooks": [
          {
            "type": "command",
            "command": "#!/bin/bash\n# Warn about console.log in edited files\ninput=$(cat)\nfile_path=$(echo \"$input\" | jq -r '.tool_input.file_path // \"\"')\n\nif [ -n \"$file_path\" ] && [ -f \"$file_path\" ]; then\n  console_logs=$(grep -n \"console\\\\.log\" \"$file_path\" 2>/dev/null || true)\n  \n  if [ -n \"$console_logs\" ]; then\n    echo \"[Hook] WARNING: console.log found in $file_path\" >&2\n    echo \"$console_logs\" | head -5 >&2\n    echo \"[Hook] Remove console.log before committing\" >&2\n  fi\nfi\n\necho \"$input\""
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display alert \"タスクが完了しました。\" buttons {\"OK\"} default button \"OK\"'"
          }
        ]
      },
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "#!/bin/bash\n# Final check for console.logs in modified files\ninput=$(cat)\n\nif git rev-parse --git-dir > /dev/null 2>&1; then\n  modified_files=$(git diff --name-only HEAD 2>/dev/null | grep -E '\\.(ts|tsx|js|jsx)$' || true)\n  \n  if [ -n \"$modified_files\" ]; then\n    has_console=false\n    while IFS= read -r file; do\n      if [ -f \"$file\" ]; then\n        if grep -q \"console\\.log\" \"$file\" 2>/dev/null; then\n          echo \"[Hook] WARNING: console.log found in $file\" >&2\n          has_console=true\n        fi\n      fi\n    done <<< \"$modified_files\"\n    \n    if [ \"$has_console\" = true ]; then\n      echo \"[Hook] Remove console.log statements before committing\" >&2\n    fi\n  fi\nfi\n\necho \"$input\""
          }
        ]
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display alert \"Claude Codeが許可を求めています。\" buttons {\"OK\"} default button \"OK\"'"
          }
        ]
      }
    ]
  }
}
```

それぞれ何をしているのかは以下の表のとおりです。  
特に、`ts`・`tsx`ファイルの型チェック、`console.log`のチェックはフロントエンド開発の効率にかなり役立ちました。

また、今回プロジェクトでは使用していませんが、仕様書駆動開発では大量にドキュメントをAIが作ることが予想されるため、`md`・`txt`ファイルの自己判断による生成も役に立つことが考えられます。

| イベント | 概要 | 理由 |
| --- | --- | --- |
| `PreToolUse` | `git push` 前に一時停止してレビューを強制 | AI が意図しない変更を push する事故を防ぐため。Ctrl+C で中断できる |
| `PreToolUse` | `README.md` 等以外の `.md` / `.txt` 作成をブロック | AI がタスクと無関係なドキュメントを自己判断で量産するのを防ぐため |
| `PostToolUse` | `.ts` / `.tsx` 編集後に `tsc --noEmit` で型チェック | ファイル編集のたびに型エラーをその場で検知し、後から気づくコストを減らすため |
| `PostToolUse` | `.ts` / `.tsx` / `.js` / `.jsx` 編集後に `console.log` を警告 | AI がデバッグ用に仕込んだ `console.log` をコミット前に気づけるようにするため |
| `Stop` | タスク完了を Mac アラートで通知 | 離席中や別作業中でもタスクの完了に気づけるようにするため |
| `Stop` | 変更された全ファイルの `console.log` を最終チェック | PostToolUse の警告をすり抜けた `console.log` を検知するため |
| `Notification` | 許可要求を Mac アラートで通知 | Claude Code が許可を求めて止まっていることに気づけるようにするため |

### statusline

私のstatuslineの設定ですが、以下のように`ccusage`というツールを使っています。

```
{
    "statusLine": {
        "type": "command",
        "command": "bun x ccusage statusline",
        "padding": 0
    }
}
```

<https://ccusage.com/>

`ccusage`を使うことで、statuslineにClaude Codeの使用状況を表示することができます。

![statusline](https://static.zenn.studio/user-upload/b735a6925608-20260319.png)

左から順に以下の情報を表示できます。

* 使用中のモデル
* 現在のセッションの累計コスト
* 今日1日の累計コスト
* 現在の5時間ブロック内で使用したコスト（画像では`1h 52m left`とあるので、残り1時間52分でこのブロックが終了するということ）
* 直近の消費速度
* コンテキストウィンドウの使用量

特に、コンテキストウィンドウの使用量は重要な指標なので、これを機に設定してみてください。

## ターミナル・ランチャー

ここからはツール系の紹介となります。  
まずは、ターミナルです。

### cmux

私が使用しているターミナルは`cmux`というターミナルです。

<https://cmux.com/ja>

見た目は以下のポストの通りです。

<https://x.com/catnose99/status/2025129675262251026>

メリットは以下のとおりです。

* 垂直タブ
* 入力待ちのタブが青く囲まれる
* 通知機能でClaude Codeのセッションを開ける
* ブラウザ内蔵
* Ghosttyベースのため、Ghosttyの機能を使用でき、Ghosttyからの移行がスムーズ
* サイドバー編集のAPIがあるため、コンテキストの管理がしやすい
* 音声入力ができる

導入すると自動で通知機能をClaude Codeに連携してくれるため、こちら側でhookの設定をする必要がありません。

通知を開くと以下のようなポップアップが表示され、各セッションにジャンプすることができます。

#### サイドバーにコンテキストウィンドウの使用率を出す方法

以下の手順でサイドバーにプログレスバーを出すことが可能です。

1. シェルスクリプトを作成
2. Claude Codeのhooksでスクリプトを実行

まず、スクリプトですが、以下を適当なローカルフォルダにて作成します。

mux-context-watcher.sh

mux-context-watcher.sh

```
JSONL_PATH="${1:-}"
SOCKET_PATH="${2:-}"
TAB_ID="${3:-}"
POLL_INTERVAL=2  # 秒

LOG_FILE="/tmp/cmux-watcher-${TAB_ID:-unknown}.log"

# ログ関数
_log() {
  echo "[$(date '+%H:%M:%S')] $*" >> "$LOG_FILE"
}

_log "起動: JSONL=${JSONL_PATH} SOCKET=${SOCKET_PATH} TAB=${TAB_ID}"

# 引数チェック
if [[ -z "$JSONL_PATH" || -z "$SOCKET_PATH" || -z "$TAB_ID" ]]; then
  _log "エラー: 引数が不足しています"
  exit 1
fi

# PID ファイルに書き込む（SessionEnd フックから kill するため）
PID_FILE="/tmp/cmux-watcher-${TAB_ID}.pid"
echo $$ > "$PID_FILE"
_log "PID=$$ を $PID_FILE に書き込み"

# クリーンアップ関数
_cleanup() {
  _log "終了: クリーンアップ実行"
  rm -f "$PID_FILE"
}
trap _cleanup EXIT

# ソケットへ送信
_send() {
  local payload="$1"
  _log "送信: $payload"
  python3 -c "
import socket, sys
s = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM)
try:
    s.connect(sys.argv[1])
    s.sendall((sys.argv[2] + '\n').encode())
    s.close()
    sys.exit(0)
except Exception as e:
    print(f'socket error: {e}', file=sys.stderr)
    sys.exit(1)
" "$SOCKET_PATH" "$payload" 2>> "$LOG_FILE" || {
    _log "送信失敗（無視して継続）"
    return 0
  }
}

# JSONL から最新の usage を取得して ratio と label を出力
_parse_usage() {
  python3 - "$1" 2>> "$LOG_FILE" <<'PYEOF'
import sys, json

jsonl_path = sys.argv[1]
last_usage = None

try:
    with open(jsonl_path, 'r') as f:
        for line in f:
            line = line.strip()
            if not line:
                continue
            try:
                entry = json.loads(line)
                if entry.get('type') == 'assistant':
                    msg = entry.get('message', {})
                    usage = msg.get('usage')
                    if usage and usage.get('output_tokens', 0) > 1:
                        last_usage = usage
            except json.JSONDecodeError:
                pass
except Exception as e:
    print(f'parse error: {e}', file=sys.stderr)
    sys.exit(0)

if last_usage is None:
    sys.exit(0)

i  = last_usage.get('input_tokens', 0)
cc = last_usage.get('cache_creation_input_tokens', 0)
cr = last_usage.get('cache_read_input_tokens', 0)
o  = last_usage.get('output_tokens', 0)

total   = i + cc + cr + o
ratio   = total / 200_000
percent = ratio * 100

if percent < 50:
    icon = "🟢"
elif percent < 80:
    icon = "🟡"
else:
    icon = "🔴"

# ラベルにスペースを含めると cmux がソケット受信時にトークン分割してしまうため除く
label = f"{icon}{percent:.0f}%"
print(f"{ratio:.4f} {label}")
PYEOF
}

# 初期表示: 0%
_send "set_progress 0.0 --label=🟢0% --tab=${TAB_ID}" || true

last_ratio=""

_log "ポーリング開始 socket=${SOCKET_PATH}"

# ソケットが存在する間ループ（cmux が終了するとソケットが消えて自然終了）
while [[ -S "$SOCKET_PATH" ]]; do
  if [[ -f "$JSONL_PATH" ]]; then
    result=$(_parse_usage "$JSONL_PATH" 2>/dev/null) || result=""

    if [[ -n "$result" ]]; then
      ratio=$(echo "$result" | awk '{print $1}')
      label=$(echo "$result" | cut -d' ' -f2-)

      # 値が変わった時だけ送信（不要な通信を減らす）
      if [[ "$ratio" != "$last_ratio" ]]; then
        _log "更新: ratio=$ratio label=$label"
        _send "set_progress ${ratio} --label=${label} --tab=${TAB_ID}" || true
        last_ratio="$ratio"
      fi
    fi
  else
    _log "JSONL未作成: $JSONL_PATH"
  fi

  sleep "$POLL_INTERVAL" || true
done

_log "ソケットが消えたため終了: $SOCKET_PATH"
```

## CLIツール

次に私がよく使うCLIツールを紹介しようと思います。

ツールと言えばMCPがLLMの話では真っ先に上がりますが、私はCLIツールがあれば、CLIツールを使用するというのが最近の基本方針になっています。  
そこで、ツールの紹介に入る前に、前提として「なぜCLIツールを使用するのか」を記載しておきます。

### MCPでなくCLIツールを使用する理由

理由は**トークン消費**にあります。

MCPはツール定義のフルスキーマが会話コンテキストに残り続けるため、長い会話ほどコンテキストウィンドウを圧迫します。  
Claude Codeの`Tool Search`も、定義の読み込みを遅らせるだけで根本的な解決にはなりません。  
（もちろん、`Tool Search`を使わなければ、最初からコンテキストウィンドウを圧迫するので、そこもネックになります）

一方、CLIツールはSkillsとの相性も良く、必要なときだけ実行されるオンデマンド方式のため、無駄な常時消費がほとんどありません。スクリプト化や自動化も比較的容易です。

そのため、CLIツールを積極的に活用するというのが最近の私の基本方針となっています。

以下は、MCPとCLIの比較をまとめたものになります。

| 項目 | MCP | CLI |
| --- | --- | --- |
| **起動時のオーバーヘッド** | セッション開始時点でツール定義が全てコンテキストに読み込まれる。複数サーバーで55,000〜143,000トークン消費の報告あり | ゼロ。シェル実行指示のみ |
| **1ツールあたりのコスト** | 名前・説明・JSONスキーマ・パラメータ等で550〜1,400トークン/ツール | CLIの出力テキストのみ（数十〜数百トークン） |
| **同一タスクのトークン比較** | CLI比で4〜32倍のトークン消費 | ベースライン。最小タスクで1,365トークン程度 |
| **コンテキスト残存** | 使わないツール定義も会話を通じて残り続ける | 実行結果のみ。不要な定義は残らない |
| **Skills連携** | ツール定義が重複しやすく非効率 | オンデマンド実行で相性が良い |
| **自動化・スクリプト化** | MCP設定が必要で環境依存しやすい | シェルスクリプトで比較的容易 |
| **セキュリティモデル** | システムプロンプト指示ベースで脆弱になりやすい | 権限モデルをCLIバイナリに直接組み込める |
| **実測例** | 3サーバーで200kトークン中143k（72%）消費の報告 | 極めて軽量 |

### context7 CLI

[context7](https://github.com/upstash/context7)はLLMのためのドキュメント参照ツールで、`context7 MCP`としてよく知られています。

AIを使っていると、最新の情報じゃなく古い情報を元に開発を進めることがありますが、context7を使用することで最新のドキュメントとコード例を取得することができるため、調査・開発には必須のツールとなっています。

<https://context7.com/docs/clients/cli>

そんなcontext7のCLIには次の3つの機能があります。

1. ライブラリのドキュメントを取得する  
   a. 任意のライブラリを名前で解決し、ブラウザを開かずにターミナルで最新のドキュメントを直接照会する。
2. AIコーディングスキルを管理する  
   a. context7レジストリからスキルをインストール、検索、生成、削除する

最も使用するのが1のドキュメント取得ですが、これはcontext7側が[find-docs](https://context7.com/skills/upstash/context7/find-docs)というskillsを提供してくれているので、こちらを基本的に使用することになります。

### agent-browser

次に紹介するのが、agent-browserです。

<https://github.com/vercel-labs/agent-browser>

これはAI AgentがCLIでブラウザを操作するためのツールです。  
CLIでブラウザを操作することで主に2つの課題を解決することができます。

1. トークン消費の増大
2. 複雑な要素指定

具体的に見てみましょう。  
従来のPlaywrightなどを使用する方法では、以下のAIは手順を踏んで、特定の要素を取得します。

1. ページのHTML構造を解析
2. 目的の要素を見つける
3. その要素を一意に特定するためのCSSセレクタ（`button#submit-form.css-1abcde`など）を生成

この方法だと、正確なセレクタを生成することが難しいことです。  
これは現代のWebサイトのHTML構造が非常に複雑になっており、動的に変化するクラス名、ライブラリ側でで生成するハッシュ付きクラス（`css-1abd5c`など）なども多く含まれていることが要因です。

また、AIがセレクタを生成するために多くの「推論」を行う必要があり、その分だけ処理時間がかかり、トークン数がかさみます。さらに、生成されたセレクタがサイトの更新によって無効になり、動作しなくなるリスクも常にあります。

agent-browserを使用するブラウザ操作では、`Refシステム`という仕組みを使用して、これらの問題を解決します。

Refシステムでは、まずスナップショットを取得し、現在の「アクティブツリー」を解析し、各要素に`@e1`のような一意のIDを付与します。その後はそのIDを使って操作します。

そのため、AI Agentはスナップショットの結果を見て`@e1`が操作対象であることを判断することができます。  
つまり、セレクタを文字列操作で生成する必要がないのです。

また、トークンコストについては従来の手法と比較して、LLMのコンテキストウィンドウの消費量を最大93%削減することができるようです。

具体的には以下のようなコマンドでブラウザ操作をしていきます。

```
agent-browser snapshot -i
# → @e1 [input type="email"], @e2 [input type="password"], @e3 [button] "ログイン"

agent-browser fill @e1 "user@example.com"
agent-browser fill @e2 "password123"
agent-browser click @e3
```

このように、agent-browserを使用することで、フロントエンドのAI Agent開発は格段に効率的になります。

特定要素の不具合修正・改修、ネットワークエラーの調査からE2Eテストの実行までを軽快に行うことができるためです。

さらに、開発だけでなく、簡単なスクレイピングや定型業務の自動化にも使えるなど活用の幅も広いのも良いポイントかなと思います。

agent-browser公式にも[skills](https://github.com/vercel-labs/agent-browser/blob/main/skills/agent-browser/SKILL.md)があるので、これを使用すれば、agent-browserを自然言語で介して簡単に活用することができます。

余談ですが、agent-browser以外に最近、Claude Codeで[Browser Use CLI](https://docs.browser-use.com/open-source/browser-use-cli)がskillsとして使用できるようになったようです。

<https://x.com/browser_use/status/2035081807209931153>

簡易的ではありますが、agent-browserとの比較も記載しておこうと思います。

| 項目 | agent-browser | Browser Use CLI |
| --- | --- | --- |
| **製作元** | Vercel Labs | Browser Use |
| **技術基盤** | Rust + Node.js | Python + CDP |
| **起動速度** | 超軽量・50ms未満 | デーモン常駐型（一度起動すれば以降は高速） |
| **操作方式** | スナップショットで要素に`@e1`などのIDを付与し`click @e1`で操作。サイト構造が変わっても壊れにくい | 要素に連番を振り`click 5`で操作 |
| **トークン効率** | 優秀（Ref方式により大幅削減） | やや多め（2.0で最適化済み） |
| **複数タブの同時操作** | ○ | ○ |
| **セッション永続化** | ○ | ○ |
| **JSON出力** | ○ | ○（`--output-format json`） |
| **MCP連携** | ✗ CLIのみ | ○ MCPサーバーとしても動作 |

操作方法が大きな違いですが、Browser Use CLIは要素に連番を振り、その番号で操作することが特徴です。

```
browser-use state
# → [0] input "メールアドレス", [1] input "パスワード", [2] button "ログイン"

browser-use click 2
browser-use type "user@example.com"
```

agent-browserの方式はサイト自体のHTML構造が変わっても安定して動作しやすいという特徴があります。一方、Browser Use CLIの連番はページを開くたびに振り直されるため、操作前に毎回stateコマンドで番号を確認する必要があります。

そのため、手順が毎回変わるなど柔軟な操作が必要な場合やUIが頻繁に変わるサイト、Python APIと組み合わせて細かく制御したいという場合には、Browser Use CLIを使用し、それ以外は基本的にagent-browserを使用するといった使い分けができそうです。

### React Doctor

1つのコマンドでコードベースをスキャンし、セキュリティ、パフォーマンス、正確性、アーキテクチャ上の問題点を検出し、0～100のスコアと具体的な診断結果を出力してくれます。

プロジェクトのディレクトリで、以下のコマンドを実行することで、診断が可能です。

```
npx -y react-doctor@latest .
```

AI Agentを使った開発だと、コードの量が膨大になるため、こうした診断ツールがあると修正などが自動化しやすいため、Reactの開発で積極的に活用すると良さそうです。

[Skills](https://github.com/millionco/react-doctor/blob/main/skills/react-doctor/SKILL.md)も提供されており、自動修正・改善ワークフローがこれでできるみたいです。

## おわりに

本記事では、私が実際に使用しているClaude Code周辺の設定・ツールを紹介しました。

改めて振り返ると、共通するテーマは「**コンテキストウィンドウをいかに効率よく使うか**」だと感じています。

MCPではなくCLIを選ぶのも、Skillsと組み合わせてオンデマンドで動かすのも、CLAUDE.mdにドキュメントインデックスを静的に持たせるのも、すべて同じ方向を向いています。コンテキストを無駄に消費しないことが、長いセッションでも精度を落とさずに開発を続けるための戦略です。

AI開発のエコシステムは現在も急速に進化しており、今回紹介したツールも数ヶ月後には大きく変わっている可能性があります。ただ、「コンテキストを節約する」「壊れにくい操作をする」といった考え方は、ツールが変わっても有効であり続けると思っています。

この記事が、AI開発をもう一歩使いこなしたい方の参考になれば幸いです。
