いまさらではありますが、Claude Opus 4.7 が出てから、X や Reddit では「Claude がバカになった」「Opus 4.6 のほうがよかった」という反応をよく見かけます。

ただ、私の使い方では、正直あまりその体感がありませんでした。

もちろん、私の感度が低いだけという可能性はあります。普段から全モデルを横並びで厳密に比較しているわけではないので、細かい変化を全部拾えている自信はありません。

それでも、少なくとも日常的に AI エージェントへ実装を任せている中で、「急にアホになったな」と強く感じる場面はあまりありませんでした。

最初は単に運がよかったのか、私が鈍かったのかと思っていたのですが、使い方を振り返ると別の理由がありそうでした。

私は AI エージェントにいきなり「この機能をいい感じに実装して」と短く投げるのではなく、まず `requirements-definition` という自作 Skill で要件定義書を作り、その仕様書をもとに実装させていました。

実装フェーズでは、Codex では自作の `implement-from-spec` Skill を使い、Claude Code では公式 plugin の `feature-dev` を使う、という形です。ツールは違いますが、どちらも「仕様書を渡して実装させる」運用に寄せています。

この運用が、Opus 4.7 の体感劣化をかなり避けていたのではないか、という話です。

## 何が話題になっていたか

Opus 4.7 は 2026 年 4 月 16 日に公開されました。

Anthropic の公式発表では、Opus 4.7 は Opus 4.6 から特に次の領域が改善されたと説明されています。

* 高度なソフトウェアエンジニアリング
* 長時間の agentic task
* 高解像度の画像理解
* より厳密な instruction following
* 出力前の自己検証

一方で、公開直後から「ミスが増えた」「反抗的になった」「トークンを食いすぎる」「4.6 に戻したい」という声も出ました。

Business Insider も、X / Reddit 上の反発として、誤答、履歴書の書き換えミス、トークン消費、adaptive reasoning への不満などをまとめています。

ただし、トークン消費については、私の環境ではきちんと比較できていません。Opus 4.6 と 4.7 で同じタスクを同じ条件で何本も走らせて、使用量を記録したわけではないからです。

なのでこの記事では、トークン効率の良し悪しについては踏み込みません。あくまで「出力や実装の体感として、なぜ私は大きな劣化を感じにくかったのか」に絞ります。

つまり、公式ベンチでは伸びている一方で、日常の体感では悪化したと感じた人もいる。

このギャップが気になりました。

## Opus 4.7 は「情報が少ない依頼」で印象が変わりやすい

Opus 4.7 の変化で重要なのは、Anthropic が Claude Code 向けベストプラクティスで書いている次の方針です。

> Treat Claude more like a capable engineer you're delegating to than a pair programmer you're guiding line by line.

つまり、一行ずつ指示するペアプロ相手というより、任せられるエンジニアに委任するように扱え、ということです。

同じ記事では、最初のターンで intent、constraints、acceptance criteria、関連ファイルなどを含む well-specified task description を与えることが推奨されています。

これは、かなり大きな使い方の変化です。

Opus 4.6 までは、曖昧な依頼でもモデル側がかなり補ってくれました。

```
この機能追加して
レビューして
いい感じに直して
前と同じ感じで
必要なら適当に判断して
```

こういう依頼でも、4.6 は背景や意図を補って、それなりに形にしてくれることが多かった。

一方で 4.7 は、より明示された指示に忠実です。Claude の Opus 4.7 チュートリアルでも、以前のモデルより instruction を literal に受け取りやすいので、必要なことを明確に書くべきだと説明されています。

なので、少なくとも私の観測では、最初に渡す情報が少なく、会話の中で少しずつ補っていく使い方ほど、こう感じやすいのではないかと思っています。

* 空気を読まなくなった
* 出力が薄い
* 余計に確認してくる
* 気の利いた提案が減った
* 前より冷たくなった

これは「知能が落ちた」と断定するより、暗黙の補完を前提にしたワークフローとの相性が悪くなった可能性として見るほうが、私にはしっくりきました。

## 私の運用: 先に要件定義書を作る

私は、実装前に `requirements-definition` という Skill を使っています。

これは、まだ粗いアイデアをいきなり実装に投げず、対話型ヒアリングで実装可能な要件定義書に変換するための Skill です。

流れはだいたいこうです。

1. プロジェクトの背景・目的を確認する
2. ターゲットユーザーや利用主体を整理する
3. 機能を洗い出して Must / Should / Could に分ける
4. Must 機能だけ詳細に掘る
5. I/O、バリデーション、エラー、画面/API/バッチ要件を定義する
6. 非機能要件と技術要件を確認する
7. Markdown の要件定義書にする

この Skill では、曖昧な回答は具体化するまで掘り下げるようにしています。

たとえば「適切にエラーを出す」では止めません。

* どの条件をエラーにするのか
* どんなメッセージを出すのか
* ユーザーは次に何ができるのか
* 再試行できるのか
* データは保存するのか

こういう粒度まで落とします。

さらに、Must 機能については、要求仕様 ID、受け入れ条件、入力・出力、処理フロー、バリデーション、エラーハンドリング、データ永続化、画面 / API / バッチごとの要件まで仕様書に落とします。

これを作ってから、実装用のワークフローに渡します。

私の場合、Codex では `implement-from-spec` Skill、Claude Code では公式 plugin の `feature-dev` を使っています。どちらも細部は違いますが、いきなり実装を始めるのではなく、仕様書を読み、実装範囲を決め、タスクに分解して進めるための入口として使っています。

Codex 側の `implement-from-spec` では、デフォルトで Must 項目だけを対象にします。Should / Could は詳細な受け入れ条件がなければ実装対象から外します。曖昧な要件も実装しません。

さらに、Must 要件をタスクに対応付ける Traceability Matrix を作り、各タスクに acceptance criteria を持たせます。Claude Code 側でも `feature-dev` に仕様書を渡すことで、少なくとも「短い一文から実装を始める」状態にはしないようにしています。

つまり、AI エージェントに渡しているのは「いい感じに作って」ではありません。

```
この仕様の、この Must 要件を、この受け入れ条件を満たすように実装して。
制約はこれ。
対象ファイルはこれ。
完了条件はこれ。
実装後に仕様準拠レビューもする。
```

という入力です。

## これは Opus 4.7 の推奨利用にかなり近い

Anthropic の Claude Code 向けベストプラクティスでは、Opus 4.7 では次のような使い方が推奨されています。

* 最初のターンでタスクを包括的に指定する
* intent、constraints、acceptance criteria、関連ファイルを含める
* ユーザーとの往復を減らす
* 長めのタスクでは、必要な文脈を先に渡して任せる

これは、私がやっていた要件定義書駆動の実装とかなり近い考え方でした。

私の運用では、Opus 4.7 に渡る前にすでに次の変換が済んでいます。

| 情報が少ない入力 | 仕様書駆動での入力 |
| --- | --- |
| いい感じに実装して | Must 要件と受け入れ条件 |
| エラーも適切に | エラー条件・メッセージ・振る舞い |
| 必要な画面を作って | 画面一覧・遷移・UI 要素・インタラクション |
| API も用意して | エンドポイント・リクエスト・レスポンス・認証 |
| 落ちないようにして | リトライ・冪等性・監視・ログ |
| たぶんこれで | 未確定事項一覧 |

Opus 4.7 への不満として語られがちな「暗黙の意図を補う」作業を、少なくとも私の運用では、実装前の Skill がかなり肩代わりしていたように見えます。

だから、私は Opus 4.7 の体感劣化を感じにくかったのではないかと思います。

## 私には入力形式の影響が大きく見えた

私の見立てでは、Opus 4.7 の体感は入力形式でかなり変わる可能性があります。

曖昧な会話を積み重ねる使い方では、4.6 のほうが気持ちよく感じる場面もあるのではないかと思います。

4.6 は、短い依頼を受け取って、ユーザーが言っていない意図までかなり補ってくれるモデルでした。これは便利ですが、裏を返すと、仕様として明示されていないことまで補って実行する可能性もあります。

一方で 4.7 は、私の使い方では、明確な委任タスクに寄せたときのほうが扱いやすく見えました。

少なくとも私の使い方では、仕様、制約、受け入れ条件、対象範囲が明確なときに扱いやすく感じました。

逆に、それらが曖昧なままだと、私の印象では冷たく見えやすい。

これは人間のエンジニアでも似ています。

短い依頼から意図を汲んで動ける同僚も、仕様と受け入れ条件を渡したときに安定する同僚も、それぞれ別の良さがあります。

Opus 4.7 は、私の使い方では後者に寄ったモデルとして見えました。

## 仕様書駆動で良くなること

私の運用では、要件定義書を挟むことで、Opus 4.7 との相性以前に、AI 実装全体の事故を減らしやすくなりました。

### 実装範囲がぶれにくい

Must / Should / Could を分けておくと、少なくとも私の運用では、AI が意図せず「ついで機能」まで実装することが減りました。

特に Codex の `implement-from-spec` 側で、詳細化されていない Should / Could を除外するようにしているのは、私の運用では効いているように感じます。

AI は気を利かせて広げがちですが、仕様書駆動にすると「今回は Must だけ」と線を引きやすくなります。

### レビューしやすい

受け入れ条件があると、レビューが感想ではなく照合になります。

ではなく、

```
F-001 の受け入れ条件を満たしているか？
異常系 V-003 の振る舞いは実装されているか？
未確定事項 TBD-002 を仕様外で実装していないか？
```

と確認できます。

これは AI にレビューさせるときにも、人間が読むときにも扱いやすくなります。

### コンテキストを圧縮できる

要件定義書があると、会話ログ全体を読ませなくても、実装に必要な情報を再構成しやすくなります。

Opus 4.7 はトークン使用量の変化も話題になりました。Anthropic の公式発表でも、新しいトークナイザーにより同じ入力が 1.0〜1.35 倍のトークンになる場合があり、高 effort では特に後半ターンで出力トークンが増えやすいと説明されています。

長い雑談ログを引きずるより、仕様書とタスク計画に落としてから渡すほうが、私には扱いやすく感じます。

## 注意点

もちろん、要件定義書を作れば何でも解決するわけではありません。

### 仕様書の品質が悪いと、そのまま悪い実装になる

Opus 4.7 は指示に忠実になった分、仕様書が間違っていると、その間違いにも忠実になりやすいはずです。

なので、未確定事項を本文に紛れ込ませず、明示的に `TBD` として分離するのは重要です。

私の Skill でも、推測で補完した箇所や未確定項目は「未確定事項一覧」に集約するようにしています。

### 仕様を作るコストはある

小さな修正なら、毎回フルの要件定義書を作るのは重すぎます。

私がこの運用を使いたいのは、次のようなタスクです。

* 新機能
* 複数ファイルにまたがる実装
* UI / API / バッチがつながるもの
* 失敗時の振る舞いが重要なもの
* 後で保守する予定があるもの
* AI にある程度任せたいもの

逆に、単純な typo 修正や小さなリファクタなら、普通に短く頼むほうがよいと思っています。

また、私の運用は業務寄り・実装寄りのタスクが中心です。創造的な探索、ラフな試作、文章のトーン出しのような用途では、仕様書で縛りすぎないほうがよい場面もあると思います。

### 仕様書はプロンプトではなく境界

仕様書を「長いプロンプト」として見ると、ただのトークン増に見えます。

でも私の運用では、仕様書は実装範囲の境界として機能しています。

何をやるかだけでなく、何をやらないかも決める。  
曖昧なものを実装から外す。  
レビュー時に照合する。

ここまで含めると、仕様書は単なるコストではなく、制御面として見やすくなります。

## まとめ

少なくとも私のケースでは、Opus 4.7 が「バカになった」と感じなかった理由を、モデル単体だけで説明するのは難しそうでした。

入力をどのくらい構造化していたかが、かなり影響していそうです。

私が劣化をあまり感じなかった理由は、おそらく Opus 4.7 が賢いかどうか以前に、次の運用をしていたことが大きそうです。

* 情報が少ない依頼をそのまま実装に渡さない
* 先に要件定義書に変換する
* Must / Should / Could を分ける
* Must だけ詳細化する
* I/O、バリデーション、エラー、非機能要件を明示する
* acceptance criteria を持たせる
* 曖昧な要件は実装対象から外す
* 仕様準拠レビューを回す

Opus 4.7 は、曖昧なプロンプトを空気で補う相棒としては 4.6 より冷たく見えるかもしれません。

でも、少なくとも私の運用では、仕様書を渡して委任する実装担当としてはかなり扱いやすかった。

なので、Opus 4.7 時代の実装フローはこう考えるのがよさそうです。

```
私の場合は、プロンプトを頑張るより、先に仕様を作る。
```

私にとっては、モデルを賢く使うというより、モデルが迷いにくい入力を作るという話でした。
