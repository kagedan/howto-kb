## まず結論から書くと、今回かなり好きです

この更新、見た目は地味です。新機能が何十個も増えたわけでもないし、派手なデモ映えもしない。

でも、**Claude Codeを毎日使う人ほど効くアップデート**だと思う。今回の芯は2つで、**長文を最後まで吐き切れる力**と、**読ませてよい場所を細かく絞れる権限制御**が、同じタイミングで強くなったことです。 Xで話題になっていたのは、 Code 2.1.77で

**Opus 4.6のデフォルト最大出力が64kに上がったこと**、そして**Opus 4.6とSonnet 4.6の上限が128kまで広がったこと**。さらに、サンドボックス設定に **allowRead** が追加され、**denyReadで塞いだ領域の中でも、必要な読み取りだけ再許可できる**ようになりました。 > これ、つまりどういうことか。  
AIが途中で息切れしにくくなって、しかも秘密ファイルの扱いを現場向けに細かく設計しやすくなった、ということです。 ---

## 64kと128kが効くのは、派手な作文より実務です

最大出力トークンというのは、AIが一回の応答でどこまで長く返せるか、の上限です。ここが小さいと、設計書の途中で切れる、長いリファクタ案が分割される、複数ファイルの変更案が中途半端になる、みたいなことが起きやすい。

逆にここが伸びると、**大きめの機能追加、広い範囲のリファクタ、実装＋テスト＋移行メモの一括生成**みたいな流れがかなりやりやすくなります。今回の変更は、まさにそこを押し上げています。 しかも土台のモデル側も強い。Opus 4.6とSonnet 4.6はどちらも**1Mトークンの長文コンテキスト**を持っていて、大量のコードや資料を読んだうえで作業しやすい設計になっています。Sonnet 4.6は、コード、長文推論、エージェント計画、コンピュータ操作まで広く底上げされていて、実務のバランスがかなりいい。 ここで大事なのは、入力だけデカくても微妙だということです。

コードベース全体を読めても、出力が短くて分割まみれなら、結局こちらが何度も継ぎ足すことになる。**読む力と、吐き切る力の両方が揃って初めて、エージェントっぽい開発体験になる**。だから今回の64k/128k拡張は、見た目以上に重いです。 ---

## allowReadが良いのは、企業でも個人でも「ちょうどいい制限」が作れるから

もうひとつの更新、allowReadもかなり実務的です。 Codeの設定では、ファイルシステムに対して allowWrite、denyWrite、denyRead、allowRead のような制御ができます。今回の追加で、**基本は読ませないけれど、このパスだけは読んでいい**という例外設計がしやすくなりました。 たとえば、`.env` や秘密鍵のある領域、顧客データのあるディレクトリは広く塞ぎたい。でも、そこに隣接する設定ファイルやテンプレートだけは読ませたい、みたいな場面は普通にあります。

いままでは制御が粗いと、全部閉じるか、やや広めに開けるかの二択になりがちでした。**allowReadの追加で、セキュリティと開発速度の妥協点をかなり細かく探れる**ようになったわけです。 > 正直、こういう機能はバズりにくいです。  
でも、会社でAIコーディングを広げるときに最後まで残る論点は、性能より権限設計だったりします。

## AI駆動開発の目線で見ると、Sonnet 4.6の立ち位置がさらに強くなった

個人的には、今回の話はClaude Code単体の小ネタではなく、**AI駆動開発のツール選定そのもの**に効いてくると思っています。Sonnet 4.6は、 上でFree/Proのデフォルトモデルになっていて、コーディングや長いタスクでの評価もかなり高い。CursorやGitHubのような開発ツール側からも、長いコードベース探索や複雑な修正での改善が語られています。  
だから今後は、

「軽い補完は別ツール、重い改修はClaude Code」みたいな住み分けが、さらに進むんじゃないかなと思う。

特に**大きなコードベースを読ませて、複数ファイルを一気に直す**用途では、単発補完よりエージェント型の価値がはっきり出る。今回の出力拡張は、その流れを後押ししています。 さらに今の価格体系だと、Opus 4.6とSonnet 4.6は**1Mコンテキストを標準料金で扱える**形になっています。ここは地味に効きます。長文を扱うたびに特別料金を気にする世界だと運用が濁るけど、そこがシンプルだと「大きい文脈を前提に設計する」判断がしやすい。スタートアップや個人開発には特にありがたいです。 API Docs

## たぶん今後、エンジニアの仕事は「書く人」より「切り分ける人」に寄っていく

この更新を見ていて思うのは、エンジニアの価値がまた少し動いたことです。

AIが長く読めて、長く書けて、しかも安全に扱える領域を細かく決められるなら、人間がずっと握るべきなのはキーボード速度じゃなくて、**どこまで任せるか、どこで止めるか、どう分解するか**の設計です。  
Cursor、 Code、GitHub Copilot、どれが勝つかを一発で決める話ではないです。

でも、**長文コンテキストを読み切る力**と**大きな出力を返し切る力**、さらに**権限制御の細かさ**まで含めて選ぶ時代に入ったのは、かなりはっきりしてきた。ここを見ずに「補完が速いから」で決めると、半年後にしんどくなる気がします。 GitHub+3Claude  
僕はこういうアップデート、かなり好きです。

派手な革命というより、**毎日の開発を本当に前へ進めるためのアップデート**だから。使い込むほど差が出るタイプです。 GitHub
