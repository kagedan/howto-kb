ログラス プロダクト基盤チームの小林です。

昨今、AI コーディングエージェントによりサービスの成長やチーム分割が加速していく中で、新しいリポジトリが次々と生まれる状況かもしれません。一方、「マルチリポジトリ環境」（以下、マルチレポと略）に特化した運用プラクティスとなると意外と事例が少ないように見えます。

私たちは役割柄、複数のプロダクトや基盤をまたいで開発・運用を行う機会が多くあります。日常的に触れるリポジトリの数は、両手で数えきれない規模になっています。

そういったマルチレポ環境は、以下のような問題にぶつかりがちです。

* 自チームの開発であっても、他リポジトリの理解がないと正確なコードが書けない
* 他チーム所有のリポジトリの改修が必要な場面で、そちらの規約に沿った PR を出せない
* API 改修において、利用側のシステムへの波及を踏まえない PR が出てくる

従来であればエンジニア自身が文脈を補完していましたが、 **AI エージェントには明示的に情報を与えなければ全体の繋がりを把握しません** 。本記事は、そんな状況に対応すべくマルチレポ環境でおこなってきた工夫をまとめたものです。チームの取り組みをベースにしつつ、筆者個人の解釈や強調点も多分に含まれている点はご了承ください。

## 横断的な情報をひとつのリポジトリに集める

まず、この記事のプラクティスはすべて、 **横断的な情報をひとつのセントラルリポジトリに集める** ことを共通の前提にしています。

マルチレポ環境では、 **「価値の境界」（ユーザーに価値を届ける機能の単位）と「実装の境界」（リポジトリの単位）が一致しません** 。ひとつの中規模機能を追加するために、API 定義・アプリケーション・プラットフォームの 3 つのリポジトリを順に触る、といったことが日常的に起こります。

その結果、作業はリポジトリを行き来しながら進み、成果物の側でも特定のリポジトリに収まらない情報——複数リポジトリをまたぐ Epic の要件定義や設計、リポジトリ間の依存関係を整理したカタログ、横断的な ADR——が次々に生まれます。どれかひとつのリポジトリの `docs/` に置けば収まりが悪く、複数リポジトリに重複させれば整合が崩れます。

そこで **特定のリポジトリには収まらない横断的な情報のための一段上の場所** として、セントラルリポジトリを用意します。AI からのアクセス容易さの観点でも、Confluence や Notion などに置いていたドキュメントを GitHub に寄せたい事情とも噛み合います。

### 7 つのプラクティス

以降の 7 つのプラクティスは、いずれもこの方針を異なる角度から支える型です。

| プラクティス | 概要 |
| --- | --- |
| P1. すべての起点はセントラルリポジトリ | 物理配置と動的フェッチ |
| P2. リポジトリカタログで関係を可視化 | リポジトリ間の関係の中央整理 |
| P3. スキルをマトリックスで整理 | リポジトリ別専門知識の中央整理 |
| P4. ルールを多層化する | 共通ハーネスと個別ハーネスの中央配備 |
| P5. 異なるフェーズのドキュメントを並べる | 弱整合での並列保管 |
| P6. as-built（竣工図） | 実態の知識メモリ |
| P7. 緩さを残す | 標準化を急がず、変化への追従余地を残す |

## P1. すべての起点はセントラルリポジトリ

マルチレポをまたぐ作業のトポロジーには、いくつかのパターンがあります。

* **相互参照型**: 各リポジトリが互いに他リポジトリを参照する。中心はなく、関係が分散している。
* **ハブ&スポーク型**: 中央リポジトリが他リポジトリを参照し、他リポジトリは中央リポジトリのみを参照する。中央が情報の集積点になるが、作業はそれぞれのリポジトリで行う。
* **セントラル起点型**: 中央リポジトリを起点に、全リポジトリへアクセスする。

![](https://static.zenn.studio/user-upload/b2025320fa8e-20260506.png)

最後が、本記事のパターンになります。

### 読む場所と開発する場所を分けない

ドキュメントだけをセントラルにまとめておき、 **開発の起点（Claude Code を立ち上げるディレクトリ）は各リポジトリにする** 、というハブ&スポーク型の選択肢もあり得ます。しかし、エージェントが優秀になるにつれて、一度のセッションで複数リポジトリを一気に触りたくなる場面が増えますし、漸進的に（最小の価値の単位で）リリースしたい場合でも、結局は複数リポジトリを薄くそれぞれ編集することになります。毎回ディレクトリを移動してセッションを立ち上げ直す運用は、人間にも AI にも煩雑です。

そこで、私たちは **セントラルリポジトリを開発の起点そのものにしてしまう** セントラル起点型に倒しています。作業のあいだ、この場所から移動しません。サブエージェントの切り替え単位としても自然です。

### 動的読み込み: オンデマンドで必要なコンテキストを読む

セントラル起点で開発するとはいえ、他チーム所有のリポジトリやその規約など、自分たち側でフルにメンテし続けるのが現実的でない情報については、 **それらを対象リポジトリ側に置いたまま、必要なときに参照しに行く** ことになります（次章 P2 で改めて扱います）。

そこで、 **コマンドで、対象リポジトリのコンテキストを動的に読み込む** 仕組みを置きます。たとえば `/product/<repo-alias>/load-context` という命名規約のコマンドで、対象リポジトリの `CLAUDE.md` や `.claude/*`、コーディング規約等をまとめて読み込む、といった設計にします。

### 隣接配置: ローカルではすべてのリポジトリを隣に並べる

複数のリポジトリの配置にはいくつかの選択肢がありますが、 **隣接配置** を採っています。

ホームディレクトリ配下の特定のワークスペースに、複数リポジトリを横並びにクローンして置きます。AI エージェントは常にセントラルの位置から起動し、隣接する他リポジトリへは相対パス（`../<repo-name>/...`）で参照しに行きます。物理的に近くに置いておくことで、相対パス参照やファイルアクセスのコストを安く抑えられます。

配置イメージは次のとおりです。

```
~/workspace/
├── repo-central/                       # セントラル本体（開発の起点）
├── repo-home/                          # 共通 API 基盤
├── repo-home_88_new-feature-api/       # repo-home の worktree
├── repo-home_91_auth-fix/              # repo-home のもう一つの worktree
├── repo-auth/                          # 認証
├── repo-catalog/                       # 蔵書管理
├── repo-catalog_104_grpc-follow/       # repo-catalog の worktree
├── repo-discovery/                     # 推薦
└── repo-manifests/                     # K8s マニフェスト
```

セントラルの本体と各リポジトリの本体（main 相当のクローン）を並べたうえで、 **作業中の worktree もすべて同じワークスペース直下に並べる** 形に揃えています。

!

**隣接配置か submodule か**

この運用に切り替えると、 **git worktree の数が一気に増える** ことになりました。同じリポジトリ内でも並列作業のために worktree が増え、リポジトリが異なれば当然 worktree も別々になります。先ほど「隣接配置」と書きましたが、実際にはワークスペース直下に **複数の worktree が常時同居する** ような状態です。

複数リポジトリをまとめて扱う手段としては git submodule という選択肢もあり得ますが、現時点では採用していません。submodule のリビジョン参照を整合させ続けるコストが、隣接配置のシンプルさを上回ってしまうと判断したためです。隣接配置 + 相対パス参照という素朴な仕組みが、もっとも扱いやすい形だと感じています。

## P2. リポジトリカタログで関係を可視化する

現実の開発体制を踏まえたとき、すべてのリポジトリを自由に改修できるとは限りません。 **リポジトリのオーナーシップと影響の波及方向** を考えたとき、変更は次のパターンに整理できます。

* **(1) 所有するリポジトリの修正**: もっとも日常的な型。自チーム所有のリポジトリ内の開発が中心です。
* **(2) 非所有のリポジトリへの修正**: インフラ系のリポジトリや基盤系の共通パッケージなど、相手チームの規約に従って追記・提案 PR を出す型です。
* **(3) 所有リポジトリに依存する非所有リポジトリへの変更の波及**: 自分たちが提供する境界（API 定義など）を変えると、それを利用しているクライアント群に追従が要求される型です。

3 類型はそれぞれ AI 運用上の壁の質が違います。(1) は自チーム他リポジトリも含めた横断参照の経路設計、(2) は相手チームの規約理解と PR の作法、(3) は波及先クライアントの取りこぼしない把握が、それぞれ壁の中心になります。

![](https://static.zenn.studio/user-upload/299b33fd4759-20260506.png)

### リポジトリカタログによる整理

3 類型のどれを扱うにも、 **「いまから触ろうとしている変更が、どのリポジトリに、どのチーム所有で、どう影響するのか」** を最初に確認できる目録が必要です。これがリポジトリカタログです。

リポジトリカタログは、 **関係する全リポジトリの位置づけ・所有チーム・依存関係などを 1 ファイルに集約** した Markdown 文書です。所有チームごとに、次のような表を並べる形でシンプルに作っています。

```
## 〇〇チーム管轄

| リポジトリ | 役割 | 言語/技術 | 主な依存・利用 |
|---|---|---|---|
| repo-central | ドキュメント・スキル管理 | Markdown | 全リポジトリの参照起点 |
| repo-home | 共通 API 基盤 | Go / gRPC | repo-auth を利用 / 各 product レポから利用 |
| repo-auth | 認証 (iDaaS 連携) | Go | 各 product レポ・repo-home から利用 |
```

特殊な仕組みはありません。新しいリポジトリの追加や責務変更があったタイミングで AI と壁打ちしながら手で更新する程度の運用です。

これから始める変更がどの類型に属するのか、AI エージェントが「どのリポジトリにどう影響するか」を判断する起点になります。

### 3 類型に対応した日常運用

3 類型への対応は、それぞれ次のような形で日常運用に落ちています。

**(1) 自チーム所有のサービスの修正**

日々の開発の中心です。通常は 2〜3 個の自チーム所有リポジトリへの修正と、ドキュメントリポジトリでもあるセントラルへの修正（ADR、設計ドキュメント、要件定義などの追記）が並行して走ります。後述のマトリックススキルにおける縦軸スキルも最も充実させています。

**(2) 非所有のリポジトリへの修正**

インフラ変更などで、他チーム所有のリポジトリへ手を入れる必要があるケースです。相手チーム側にスキルが整備されていれば、それを `load-context` 経由で読み込み、相手チームの規約に従った形で PR を整えます。更に自分たちのサービス固有の制約をアドオンする場合には、ラップした固有のスキルとしてセントラルに配置します。

**(3) クライアント側に波及する変更**

自分たちが提供しているサービスを利用するクライアント群に影響が出る変更（API 契約変更、認証フロー変更など）は、表面的に API 定義が破壊的変更でなくとも、考慮が必要なケースは往々にしてあります。 **どのリポジトリがクライアントなのかはすべてカタログに定義してある** ため、そこを起点に一気に影響範囲の調査が可能であり、たとえ人間が忘れていても、AI が気を利かせて調査してくれます。

## P3. 増えるスキルをマトリックスで整理する

セントラルにドキュメントとスキルを集約していくと、 **スキル数の増加によって AI が適切なスキルを選びづらくなる** という、認知負荷の問題が露見します。「どのスキルを呼ぶか」が AI の判断力を強く要求する設計になってしまうと、品質がブレがちです。

### マトリックスの 2 軸

そこでスキルを 「いま対象リポジトリは X、いまフェーズは Y」という 2 軸の **マトリックスで整理する** ことで増えるスキルを整理します。いわば、 **モジュラーモノリス** のように捉える構図です。

モジュラーモノリスの世界では、 **レイヤー** と **ドメイン** の 2 軸でモジュールを配置しますが、スキルマトリックスもこの構造からのアナロジーで、レイヤーに **開発プロセスのステップ** を、ドメインに **リポジトリ** を対応させます。

* **横軸 = レイヤー（開発プロセスのステップ）**: 調査 → 要件 → 設計 → 実装 → レビュー → デプロイ
* **縦軸 = ドメイン（リポジトリ別の専門知識）**: 各リポジトリ固有のスキル群（リポジトリごとの規約・コマンド・ドメイン知識）

横軸のフェーズスキルは **リポジトリ非依存または起点** として設計します。たとえば、同じ「レビュー」スキルを、どのリポジトリを対象にしても同じ段階を踏んで進められるように作りつつ、必要な専門レビュースキルをチョイスできるエントリーポイントにもしています。レビュー単体の実施が必要なときも何も考えずに `/review` と打てるようになります。一方、縦軸のリポジトリ別スキルは **対象リポジトリ固有の作法や知識** を持ちます。コンテキスト読み込みの対象も変わったりします。

![](https://static.zenn.studio/user-upload/4c6ac41f7215-20260507.png)

### プロセスを軽量にも重厚にもアレンジしやすい

スキル群の利用を厳格な手順にすると、簡易な修正にまで重厚なプロセスが発動し、時間もトークンも浪費します。マトリックスで整理することで、 **「重厚なプロセスにも、簡易な修正にも使える」柔軟性** を持たせやすくなります。

* スキルの組み合わせで重さを調整できるようにする
* スキル内部で軽量モードと重厚モードを切り替えられるようにする

たとえば、同じ「開発」フェーズでも、Issue 1 本規模の小さな不具合修正と、複数リポジトリをまたぐ Epic 規模の新機能開発では、実質必要なプロセスが異なります。前者には軽量フローのスキルの組み合わせを、後者には PRD から完了レビューまでを通す重厚フローのスキルの組み合わせを、別々に用意して呼び分けるアプローチが取れます。

### スキルのマトリックス配置イメージ

セントラルには多数のスキル・コマンドが配置されています。横軸と縦軸を分けて設計しており、（この枠組みに収まらないものも多いですが）整理すると次のようになります。

横軸（汎用フェーズ）の例:

* 調査・分析（複数段階に分けた承認フロー内包）
* 要件定義（ユースケース洗い出し）
* 設計（Design Doc、ADR、図の作成）
* レビュー（公式レビュースキル、または専門観点を指定）

縦軸（リポジトリ別／対象別の専門知識）の例:

* コンテキスト読み込み系（対象を触る前に、関係する情報だけを読み込み）
* コードレビュー系（対象固有の規約や品質観点に基づくコードレビュー）
* 専門ツール系（対象でのみ意味を持つ専門コマンド。画面設計・動作確認など）

## P4. ルールを多層化する — 共通ハーネスと個別ハーネス

リポジトリをまたぐと守るべきルールが異なることが多いと思いますが、マルチレポ環境では、 **人間の注意力にも AI の記憶にも頼らない強制機構** が、単一リポジトリ環境よりずっと重く効いてきます。

いわゆるハーネスエンジニアリングの出番ですが、ここでいうハーネスは、ルール（何を守るか）と検証スクリプト（機械的に確認する手段）をペアで配置し、 **AI 自身の判断に委ねず、外部スクリプトによる客観的検証で守らせる** 仕組みです。

### 共通ハーネスと個別ハーネス

マルチレポにおけるハーネスは、多層的です。

* **共通ハーネス**: 全リポジトリに共通する観点（セキュリティ、CI/CD のベストプラクティス、Git・GitHub の取り扱いなど）。セントラルに集約し、対象リポジトリを問わず呼び出せるようにする。
* **個別ハーネス**: 特定のリポジトリに固有の観点（そのリポジトリの実装規約、ドメイン知識に依存する検証など）。対象リポジトリ内に直接配置する場合と、セントラルに対象別にディレクトリを切って置く場合とがある。

### リポジトリをまたぐ整合性の確保

個別ハーネスのなかで手強いのは、 **複数リポジトリをまたぐ整合性検証** です。たとえば、「gRPC 定義リポジトリと実装リポジトリでスキーマがズレていないか」など、 **個々のリポジトリからは他リポジトリを参照しづらい** ため、セントラルからしか実装できません。マルチレポ環境のハーネスは、自然と「中央集約 + 横断検証」という構造を取ります。

## P5. 異なるフェーズのドキュメントもすべて置くが、強整合にはしない

ここまで開発の起点 (P1)、関係の可視化 (P2)、スキル (P3)、ルールの多層化 (P4) と、いずれも開発フェーズに直結するプラクティスを見てきました。次はもう一段引いて、 **設計プロセスのドキュメントをセントラルでどう扱うか** に進みます。

### すべて置く: フェーズ別のドキュメントが個別の価値を持つ

中規模以上の機能を作るとき、たとえば次のような段階を経ていきます。

ロードマップ → 概要要件定義 → 調査と分析 → 詳細要件定義・設計・開発

各段階の中間成果物（ロードマップ、概要要件、調査メモ、詳細要件、設計ドキュメント、実装計画などのセット）はそれぞれ別のドキュメントとして残ります。これらは **「その時点の判断材料」として個別の価値を持つ** ので、捨てずにすべてセントラルに集めて並べておきます。

たとえば私たちのプロダクト基盤チームの場合、後から組成された経緯もあり、既存コードの実態 (GitHub)、運用の現状 (Notion)、 ADR にも残っていない「歴史的経緯」(git log, Slack) までを掘り起こす必要があり、調査と分析の成果物が日常的に多くを占めます。これらの成果物を残しておくことは、将来のコンテキスト把握や意思決定の基礎になります。

### 強整合は求めない: 時間軸もオーナーも違うから

ただし、これらの中間成果物の整合を取り続けようとすると破綻します。 **時間軸とオーナーが違う** からです。

ロードマップは数年単位、要件定義は数か月単位、設計や調査は週単位、実装計画は日単位、というように、扱う時間スケールが層ごとに異なります。担当者も、関わるフェーズによって入れ替わります。書かれた瞬間から、それぞれの中間成果物の前提や鮮度はずれていき、後から横並びで強く整合させ続けるコストは見合いません。

そのため、 **「弱整合」** で運用しています。各ドキュメントは「その時点の判断材料としては正しい」「後から見返したときに『なぜこの判断をしたか』が追える」ところまでを担保し、最新コードや最新仕様と整合させ続けることはしません。

![](https://static.zenn.studio/user-upload/0e1d9eef82c3-20260506.png)

### フェーズとディレクトリのイメージ

設計プロセスの中間成果物の保管庫を、時間軸と性質で複数のディレクトリに切り分けています。

* ロードマップ用（半期から数年）
* 要求整理・概要要件定義用
* 実験的コード・プロトタイプ用
* 開発中の Epic ごとの詳細要件定義・設計・タスク管理用
* ADR と Design Doc 用
* as-built 用（後述）

各ディレクトリは独立したプロセスとして運用するイメージです。たとえば、新しい Epic を始めるときは、過去の調査分析時の仮設計を記録したディレクトリをコピーしてから書き換えたりします。中間ドキュメントは「その時点の判断材料」としては正しく、後から見返しても「なぜこの判断をしたか」の文脈を追えるだけの粒度では残しています。

## P6. 最終ドキュメントはリバース生成 — as-built（竣工図）

対象範囲が広いほど、 **人間も AI も「実態の把握」自体に時間がかかる** ようになります。コードや運用の現状を毎回ゼロから読み解くのは重い作業で、特にマルチレポではコンテキストも時間も大きく削っていきます。

### ドキュメントをリバース生成する

これを緩和する仕組みが、「実装後の現状を逆向きに記述する」ドキュメンテーション、すなわち **as-built（竣工図）** です。建築や土木の世界では、設計図と竣工後の実態の差分をすべて反映した最終図面を「竣工図」と呼びます。同じ思想をソフトウェアの世界に持ち込んだものと位置づけられます。

私には馴染みが薄い用語だったのですが、広く定着している一般用語のようで、 **「現状はこうなっている」と言い切られた構造化ドキュメント** という位置づけを、AI エージェントも理解しやすいようです。

`as-built` はコードの変化に追随する必要がありますが、 Epic 開発プロセスのラップアップステップで「変更内容に応じて関連竣工図の更新を提案する」スキルを発火させる仕組みで運用しており、追随の手間は比較的軽く保てています。

### as-built に書くこと

`as-built` には、コードを読む前に把握しておきたい構造的な事項を書きます。

* 主要なエンティティとそのライフサイクル
* 認証・データフロー
* コンポーネント境界
* 主要 API の使われ方

なお、コードや git の履歴にも表れない **「意図」** は `as-built` には書きません。詳細は別の機会に譲りますが、意思決定の履歴は ADR として残し、 **as-built（現状）と ADR（意図）を補完的に使う** という分担です。

Javadoc がコードから生成される API 仕様であるように、 `as-built` も実装からリバース生成される「実態の構造図」と位置づけており、人間の読みやすさと AI のコンテキスト把握の両取りを狙っています。まだ最適化の余地がある現在進行形の施策です。

![](https://static.zenn.studio/user-upload/fc7a26d028f0-20260506.png)

## P7. 緩さを残す

ここまで 6 つのプラクティスを並べてきましたが、最後にもうひとつ、これら全体に対する運用スタンスとして **「緩さを残す」** を加えます。

### 個人レベルの開発プロセス最適化を許容する

セントラルにスキルやプロセスを集めていても、 **個々のメンバーが自分の開発プロセスを最適化することは許容しています** 。たとえば [GitHub Spec Kit](https://github.com/github/spec-kit) のような仕様駆動開発ツールを好んで使うメンバーもいます。この記事自体も筆者個人の意見がだいぶ反映されたものとなっています。

### 整合させるのは「ストック情報の一部」だけ

P5 では中間成果物のレベルで整合を諦めましたが、 **チーム間・メンバー間でも同じ発想** で動いています。すべての情報の整合を取りに行こうとするとまず人間の頭が破綻するので、整合を取りに行くのは **ストック情報のごく一部** （リポジトリカタログ、横断的な方針書、主要な ADR など）に絞っています。それ以外は各個人がそれぞれのやり方で動いていて構わない、という構えです。

### 半年後には不要になっているかもしれない

社内の他のチームを見渡しても複数の進化が並行して進んでいます。モックから UI 開発まで自動化するチーム、性能検証を自動改善するチーム、Discovery すら AI と対話で詰めるチームなど、それぞれが自分たちの文脈における最適解を試行錯誤しているからこそ、組織全体として時代の変化のスピードに追いついていけると感じます。

本記事で紹介したプラクティスの多くも、 **半年後には不要になっている可能性があります** 。AI エージェントの能力が上がり、社内外のプラクティスも溜まれば、より包括的により抽象的に開発全体の最適化が進んでいくはずです。

「中央が標準を画一的に決める」ようなトップダウン型・プッシュ型ではなく、 **ボトムアップ型・プル型で互いに取り入れる** 文化が軸になっています。「あのチームのあのスキルを、ここの文脈に合うように取り入れて」と AI に指示する形で、必要なものを必要なだけ持ち込むことも可能であり、緩さを残すことは現時点では理にかなっていそうです。

## まとめ

マルチリポジトリ環境で AI コーディングエージェントを運用する際の最大の課題は、 **「リポジトリ間の関係を知っている存在が AI 側に不在になる」** ことです。本記事では、その解としてセントラルリポジトリと、それを支える 7 つのプラクティスを提示しました。

紹介したプラクティスは唯一の正解というよりは、各組織が自分たちの文脈で組み替えていく **叩き台** として使っていただければと思います。前提となる AI エージェントの能力も社内外のプラクティスも変わり続けていく中で、いま手探りでマルチレポ × AI と向き合っている方の引き出しに、何かひとつでも残れば幸いです。

AI を利用した開発の変化の流れは非常に速く、半年前ですら想像していなかった形に開発プロセスは進化しています。不安と興奮が高速で入り乱れる時代になったとつくづく感じます。今後想像より早く、単なるマルチリポジトリという視点を超えて、AI が開発プロセス全体をより包括的にサポートしてくれるような発展に到達しそうですね。
