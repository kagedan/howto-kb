[前回の記事](https://zenn.dev/m2dw/articles/n8n-ai-cli-loop-overview)で、GitHub Issue を書くと AI が実装・レビュー・修正して PR を返してくる自動開発ループの骨格(intake → 実装 → レビュー → outbox)を紹介しました。

ただ、実際に 1 ヶ月回し続けると、骨格だけでは済まないことが次々に起きます。この記事では、\*\*「何に困って、どんな工夫を足してきたか」\*\*の順で、現在のループを支えている仕組みを紹介します。なお、この仕組みの大半も Issue を書いてループ自身に実装させたものです。

## そもそも n8n はどう見えているのか

ここで使っている n8n は、ローカルマシンで動かしているセルフホスト版です(この使い方なら無料)。実物はこんな見た目です。

![n8n のワークフローキャンバス](https://static.zenn.studio/user-upload/deployed-images/493901da8cf9be578df88f0c.png?sha=983692af625cff39fea9fbde87d58992e57d7449)  
*トリガー → 実行コンテキスト作成 → リポジトリロック取得 → フェーズ実行(子ワークフローが CLI を呼ぶ)→ ロック解放、という並び*

ワークフロー自体は、トリガーとロックの前後処理、CLI の呼び出しだけの素朴なもので、頭脳は全部 TypeScript の CLI 側にあります。それでも 5 分おきに 24 時間回り続けてくれるのが仕事です。

![n8n の実行履歴](https://static.zenn.studio/user-upload/deployed-images/5aa6f5c81a3659b5f2cec410.png?sha=973fb1c39945d5b0db55ea55b09c9ecef31afe84)  
*5 分おきの実行履歴。かかる時間はその回の仕事量次第で、数十秒から十数分までまちまち*

Claude Code や Codex を対話で使っていると出てくる「このコマンドを実行しますか?」の確認、あれの無人ループ版です。対話ならその場で Yes/No を押せますが、このループには押す人がいません。

そこで、エージェントが許可外のコマンド(`npm install` など)を必要とすると、ループは実行せずに「このコマンドを実行したい」という申請(**Tool Request**)として Issue を人間にエスカレーションします。人間はリクエストを見て、自ら実行するか、拒否するかを決めます。実行した場合はその結果をループに通知し、ループが再開します。おなじみの許可プロンプトを、**非同期の申請ワークフローに置き換えた**形です。

## 工夫 2: 依存関係は守る。ただし人間のマージは待たない

Issue 同士の順序は GitHub の Issue Relationship(`blocked by`)で表現します。intake は着手前に依存をチェックし、依存先が終わっていない Issue はキューに入れません(依存の状態が確認できないときも、安全側に倒して着手しません)。

ポイントは「終わっている」の判定に **PR のマージを含めない**ことです。依存先の実装がレビューを通って完了ラベルが付けば、マージ前でも後続 Issue は進行可能になります。このとき後続の作業ブランチは、main ではなく**依存先のブランチから切ります**。マージまで待つ設計にすると、人間がマージするまで後続がすべて止まる — つまり**寝ている間にループが止まってしまう**ためです。

*#101 はレビュー通過・人間のマージ待ち。#102 は main ではなく #101 のブランチから切って先に進む*

## 工夫 3: Issue ごとに git worktree を分ける

初期は、リポジトリの作業ディレクトリ 1 つを全フェーズで共有していました。これは工夫 1 の Tool Request と相性が悪く、人間はリクエストされたコマンドを実行したいだけなのに、作業ディレクトリは別の Issue の作業中かもしれず、**ループを止めてブランチを切り替えてから実行する**羽目になります。

もうひとつ厄介なのが、実行の失敗などで**コミットされていない変更が作業ツリーに残ったままになる**ケースです(いわゆる dirty な状態)。無関係な変更を巻き込まないよう、dirty な状態を解消しないと次の処理には入れません。作業ディレクトリが 1 つだと、**1 つの Issue の残骸がループ全体を止めてしまう**わけです。

そこで **Issue ごとに独立した git worktree** を切り、各フェーズも人間の作業も、その Issue の worktree の中で行うようにしました。事故が起きても被害はその Issue の worktree に閉じ、他の Issue は素通りできます。同じ Issue が同時に 2 回走らないよう、**Issue 単位のロック**も入っています。

*作業ディレクトリが 1 つの世界*

*Issue ごとに worktree を分けた世界*

なお、worktree を分けたことで処理の並列化もできるはずですが、共有リソースのロック処理がまだ不十分なため、そこまでは至っていません。今は 1 回の実行で 1 フェーズずつ進めています。

worktree は使い捨てではなく Issue の生存期間中は保持され、不要になったものは admin コマンドで棚卸し・掃除します(dirty なものや push されていないコミットを持つものは削除を拒否する、といった安全側の作りです)。

## 工夫 4: コンフリクト解決を専用フェーズにする

複数の Issue が並行で走り、しかも工夫 2 のとおりマージを待たずにブランチを積んでいくので、**この設計ではコンフリクトは例外ではなく日常**です。そこで、実装・レビューと並ぶ独立したフェーズとして「コンフリクト解決」を用意しています。コンフリクトが検出された Issue はこのフェーズに送られ、PR ブランチを worktree に取り出して AI に解決を試みさせ、テストが通った場合だけ push します。解決できなければ人間へエスカレーションします。

(ちなみに、意味レベルの競合 — テキストは綺麗にマージできるのに挙動として矛盾するケース — は今も難敵です。この話は次回に詳しく書きます)

## まとめ

エージェント本体は最初から Claude Code と Codex のままで、この 1 ヶ月で増えたのはほぼ全部その周辺 — 並行実行、権限の関所、コンフリクト処理、依存の交通整理 — でした。

紹介しきれなかったものもまだあります。約 40 コマンドに膨れ上がった運用 CLI という伏魔殿、調査・計画のフェーズ、複数リポジトリの同時運用、そして作ったのに一度も動いていない機能たち。このへんは次回以降に書きます。

直近の次回は、これだけ機構を揃えてもなおこじれる Issue の話 — AI が 38 回修正しても直らなかった Issue の解剖と、こじれる Issue の 4 類型を、データ付きで書く予定です。
