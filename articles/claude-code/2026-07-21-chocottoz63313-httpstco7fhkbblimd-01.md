# サマリー

Claude Codeを使い始めると、まずは「何ができるのか」に目が向きます。

- 文章を書く。
- コードを修正する。
- ファイルを整理する。
- 資料を要約する。
- 調査や発信の準備を進める。

たしかに、Claude Codeは幅広い作業に対応できます。

しかし、しばらく使っていると、別の問題が見えてきます。

- 毎回、同じ前提を説明している。
- 同じ指示でも、出力の方向性が少しずつ変わる。
- ファイル操作を任せるのが不安で、結局は自分で細かく確認している。
- 便利なはずなのに、思ったほど作業が楽にならない。

この原因は、プロンプトの短さや長さだけではありません。

> Claude Codeへ作業を頼む前に、前提、手順、安全策、確認方法が整理されていないことが大きく影響します。

重要なのは、次の五つです。

- 前提を共有する。
- 実行前に計画を確認する。
- 危険な操作を仕組みで止める。
- 失敗しても戻せる状態を作る。
- 最後に検品する。

この記事では、Claude Codeを単発の便利なツールで終わらせず、日々の作業へ安定して組み込むために、最初に整えたいことを解説します。

# 1．最初に整えるべきなのは、機能ではなく作業の渡し方

# 

Claude Codeを使い始めると、多くの人は機能やコマンドを覚えようとします。

- 何ができるのか。
- どこまで自動化できるのか。
- どの設定を使えば便利になるのか。
- どのような指示を出せば、期待した結果が返ってくるのか。

機能を理解することは必要です。

> ただし、利用できる機能を増やすだけでは、作業が安定するとは限りません。

Claude Codeは、依頼のたびに多くの前提を読み取りながら作業を進めます。

- 誰に向けた文章なのか。
- どのような文体で書くのか。
- どのファイルを参照してよいのか。
- どこまで変更してよいのか。
- 何をしてはいけないのか。
- 成果物をどこへ保存するのか。
- 完成と判断する基準は何か。

これらが曖昧なままだと、Claude Codeは、その場の情報から推測するしかありません。

> 推測に任せる部分が多いほど、出力も不安定になります。

これは、人に仕事を頼む場合と同じです。

> 目的、判断基準、禁止事項、完成イメージを伝えずに「いい感じに進めて」と頼めば、依頼する側と作業する側の認識はずれます。

Claude Codeでも、同じことが起こります。

だからこそ、最初に整えるべきなのは機能の数ではありません。

> 先に整えるべきなのは、作業を渡すための前提と進め方です。

> Claude Codeが能力を発揮できるかどうかは、依頼の巧さだけでなく、迷わず作業できる環境が用意されているかによって変わります。

## 

## この章の要点

Claude Codeの成果を安定させる第一歩は、便利な機能を増やすことではありません。

> 作業の前提、範囲、禁止事項、完成基準を整理し、迷わず受け取れる形にすることです。

# 2．CLAUDE.mdで毎回の説明を減らす

# 

Claude Codeを継続的に使うなら、最初に整えたいのがCLAUDE.mdです。

> CLAUDE.mdには、毎回変わらない作業上の前提を記載します。

たとえば、次のような内容です。

- 取り組んでいる活動やプロジェクトの概要
- 想定する読者や利用者
- 文体や言葉遣い
- 使用を避けたい表現
- ファイルやフォルダの構成
- 保存場所やファイル名のルール
- 変更してよい範囲
- 成果物の品質基準
- 作業後に確認すべき項目
