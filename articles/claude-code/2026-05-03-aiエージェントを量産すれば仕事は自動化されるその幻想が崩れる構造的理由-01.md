## はじめに：「課金してエージェントを並べれば勝てる」という幻想

「Claude や GPT に課金して、エージェントを 10 体、20 体と並列に走らせれば、自分は寝ていても仕事が進む」——AI 業界に身を置く人なら、一度はこの夢を見たことがあるはずです。

実際、Claude Code、Cursor、Devin、Manus、OpenAI Swarm、LangGraph など、マルチエージェントを組むためのフレームワークは 2025 年に入って爆発的に増えました。「AI 組織を作る」という言葉も、スタートアップ界隈では半ば当たり前のように飛び交っています。

しかし、実際にこれを業務に投入してみた人の多くが、ある共通した壁にぶつかっています。\*\*「動いているように見えて、何も終わっていない」\*\*という壁です。

本記事では、X で話題になっている @Kohaku\_NFT さんのポストを起点に、**AI 組織が崩壊する構造的な理由**と、それに対する**現実的な対策**を、技術者視点で言語化していきます。

## なぜ AI 組織は崩壊するのか：4 つの構造的失敗

### 1. コンテキストの分断（Context Fragmentation）

最も根深い問題が、**エージェント間でコンテキストが共有されない**ことです。

人間の組織では、暗黙知・空気・過去の意思決定の記憶が、メンバー間で自然に補完されます。しかし AI エージェントは、明示的に渡された情報しか知りません。

```
[エージェント A] タスク分解担当
   ↓ JSON でタスク渡し
[エージェント B] 実装担当
   ↓ 実装結果のみ返却
[エージェント C] レビュー担当 ← なぜその実装になったか知らない
```

C は「なぜこの設計になったのか」「過去にどんな選択肢が却下されたのか」を知らないまま、表面的なレビューしか行えません。結果、**全員が真面目に動いているのに、出力の質は人間ひとりの作業を下回る**という現象が起きます。

### 2. 責任の希薄化（Diffusion of Responsibility）

人間の組織でも起きる「みんなの仕事は誰の仕事でもない」現象は、AI 組織でも同様に発生します。

| 状況 | 人間組織 | AI 組織 |
| --- | --- | --- |
| エラーの検知 | 誰かが気づく | 誰も気づかない（気づくロールが定義されていない） |
| 例外処理 | 暗黙にハンドリング | プロンプトに書いていなければスルー |
| 品質基準 | 経験で判断 | 評価基準が曖昧だと「OK」を返す |

エージェントは**与えられたロールを愚直に演じる**だけなので、「自分の責任範囲外」と判断したものは平然と無視します。

### 3. 評価ループの欠如（No Feedback Loop）

エージェント同士の出力を、**人間に近い厳しさで評価する仕組み**を持たない構成は、ほぼ確実に破綻します。

LLM は本質的に「もっともらしい出力」を返す装置です。Critic 役を別エージェントに任せても、結局 Critic も「もっともらしい合格判定」を返してしまう。

これを防ぐには、**外部の決定論的な評価軸**が必要です。

```
# 悪い例：LLM 同士で評価し合う
critic_response = llm.evaluate(worker_output)  # 主観の応酬

# 良い例：決定論的なテストで評価
test_result = run_tests(worker_output)  # 真偽が一意に決まる
if not test_result.passed:
    feedback = test_result.error_log
    worker.retry(feedback)
```

### 4. オーケストレーションコストの過小評価

「エージェントを増やせば速くなる」という直感は、ほぼ間違いです。

エージェント間通信のオーバーヘッド、ステート管理、競合解決、エラーリカバリー——これらは**エージェント数の二乗に比例して複雑化**します。

3 体までは管理可能でも、5 体を超えるとオーケストレーター側のロジックが破綻し始めます。

## ツール別：マルチエージェント運用の現実

| ツール | 強み | 崩壊しやすいポイント |
| --- | --- | --- |
| Claude Code | 単一セッションでの深い文脈保持 | 並列化するとコンテキスト共有がほぼ不可能 |
| Cursor | エディタ統合・差分ベース修正 | 複数タブで並走するとファイル競合 |
| Devin | エンドツーエンド自走 | タスク粒度を間違えると無限ループ |
| LangGraph | 明示的なステート管理 | グラフが複雑化するとデバッグ不能 |
| OpenAI Swarm | 軽量・教育用途に最適 | 本番運用には機能不足 |

## 崩壊を防ぐ対策：5 つの実践原則

### 原則 1：エージェント数は「3 以下」から始める

最初から 10 体並べる構成は禁忌です。**1 体で完結するか？** を毎回問うてください。1 体で十分なら 1 体にする。

### 原則 2：コンテキストは「ファイルシステム」に集約する

メモリやセッションに依存せず、**全エージェントが読み書きする共通のファイル**を Single Source of Truth にします。

```
project/
├── CONTEXT.md       # 全エージェント必読
├── DECISIONS.md     # 過去の意思決定ログ
├── tasks/active.json # 進行中タスク
└── outputs/         # 各エージェントの成果物
```

### 原則 3：評価は「決定論的なテスト」で行う

LLM 同士の相互評価ではなく、**型チェック・ユニットテスト・E2E テスト**など、真偽が一意に決まる仕組みを必ず差し込みます。

### 原則 4：「人間の承認ゲート」を残す

完全自動化を急ぐと品質が崩壊します。重要な意思決定ポイントには、**Discord や Slack 経由の承認フロー**を挟むのが現実解です。

### 原則 5：「LLM が必要な箇所」を厳密に切り分ける

決定論的に処理できるもの（データ取得・ルール判定・通知）は、**Node スクリプトや Python で書く**。LLM は自然言語理解・推論・文章生成・不規則イベント対応にだけ使う。

これを守らないと、月数万円〜数十万円のトークン課金が積み上がります。

## FAQ

**Q. マルチエージェントが「全く意味がない」ということですか？**  
A. いいえ。**役割が明確に分離でき、評価軸が決定論的に作れる場面**では非常に強力です。問題は「とりあえず並べる」発想です。

**Q. エージェント数を増やすべき場面はありますか？**  
A. **完全に並列で、互いに依存しないタスク**（例：100 件のドキュメント要約）には有効です。ただしこれは「組織」ではなく「並列実行」です。

**Q. Claude Code のサブエージェント機能は使うべき？**  
A. 親エージェントのコンテキストを汚さず、独立した調査タスクを並列化したい場合は有効。意思決定や設計判断を委譲するのは危険です。

## 将来の展望：AI 組織はどこに向かうのか

2026 年現在、業界は「マルチエージェント幻想」のピークを越え、\*\*「シングルエージェント + 強力なツール群」\*\*へと回帰しつつあります。

Anthropic が公開した [Building effective agents](https://www.anthropic.com/research/building-effective-agents) でも、「エージェントは最後の手段」と明言されており、まずワークフローで解決できないかを検討すべきと述べられています。

今後 2〜3 年で起きるであろう変化：

* **「AI 組織」という言葉の死語化**：代わりに「エージェント駆動ワークフロー」が主流に
* **オーケストレーションレイヤーの標準化**：MCP（Model Context Protocol）のような共通プロトコルが普及
* **評価基盤の重要性向上**：LLM-as-a-Judge ではなく、決定論的評価を組み込んだフレームワークが台頭

## まとめ

「AI に課金してエージェントを大量生産すれば仕事が自動化される」——この幻想は、**コンテキスト分断・責任希薄化・評価ループの欠如・オーケストレーションコスト**という 4 つの構造的問題によって、ほぼ必ず崩壊します。

対策は逆説的ですが、\*\*「エージェントを減らす」「ファイルシステムにコンテキストを集約する」「決定論的評価を挟む」「人間の承認ゲートを残す」「LLM が必要な箇所だけに絞る」\*\*という 5 原則に集約されます。

@Kohaku\_NFT さんのポストには、これらの「生々しい現場の言語化」が詰まっています。AI 組織を本気で運用しようとしている方は、夢から覚める前に必ず一読することをおすすめします。
