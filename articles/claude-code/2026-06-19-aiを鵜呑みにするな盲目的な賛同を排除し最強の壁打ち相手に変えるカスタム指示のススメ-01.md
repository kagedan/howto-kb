## はじめに

ChatGPTやClaudeなどのLLMと開発のディスカッション（壁打ち）をしているとき、以下のような枕詞にモヤモヤしたことはありませんか？

> 「とても素晴らしい視点ですね！」
>
> 「ご提示いただいたアイデアは非常に画期的です！」

一見気持ちよく会話できますが、エンジニアが求めているのはお世辞ではなく、ロジックのバグやエッジケースの指摘です。AIの盲目的な賛同や、それっぽい回答を鵜呑みにしていると、重大な設計ミスや技術選定の失敗を見落とすリスクがあります。

この記事では、AIの「YESマン化」を防ぎ、客観的事実と批判的思考（クリティカル・シンキング）に基づいた「優秀な議論のパートナー」に変貌させるための「カスタム指示」の定義方法を紹介します。

## AIを鵜呑みにしてはいけない理由

AIは構造上、「ユーザーが喜びそうな、滑らかな文章」を出力する傾向（プロンプトへの過剰適合や、ハルシネーション）があります。

:::note warn
- **思考の解像度が下がる**：自分のアイデアの欠点が見えなくなる。
- **技術的な嘘を見逃す**：間違った前提にAIが「その通りです！」と同調し、そのまま破綻したコードや設計が生成される。
:::

これを防ぐには、AIに対して「私に忖度（そんたく）するな」とあらかじめ釘を刺しておく必要があります。

## AIを「最強の壁打ち相手」にするカスタム指示

ChatGPTの「カスタム指示」や、各AIツールの「システムプロンプト」に以下のテキストを登録してください。これにより、不要な褒め言葉が消え、ロジックの検証に特化した回答が出力されるようになります。

### コピペ用プロンプト

```markdown
# 振る舞いのアプローチ
- ユーザーの前提や意見に盲目的に賛同しないでください。常に客観的な視点を持って判断し、論理的な誤りや矛盾があれば正確に指摘してください。
- 回答の冒頭にあるような、不要な褒め言葉やお世辞（例：「とても良い質問です」「素晴らしいアイデアですね」など）は一切排除してください。

# 提案・アイデアの検証
- ユーザーからアイデアが提示された際は、最新の情報を調査・考慮したうえで、客観的な事実や反証（デメリット、リスク、代替案）を列挙してください。
- 単に否定するだけでなく、ユーザーの真の意図（ゴール）を確認し、それに合致しているかを検証してください。

# 思考スタンス
- 同意や共感よりも、「客観的な事実」「正確性」「批判的思考（クリティカル・シンキング）」を最優先してください。
- ユーザーの機嫌を取るアシスタントではなく、建設的な議論を行う対等なパートナーとして振る舞ってください。
```

## この指示を入れることで何が変わるか？

### 1. 「お世辞」のノイズが消え、トークンと時間の節約になる

スクロールや読む時間を奪う「素晴らしい提案ですね！」という定型文が消滅し、最初の一行目から結論や指摘が飛んでくるようになります。

### 2. 自分の認知バイアスを破壊してくれる

「この技術選定（または設計）でいこうと思う」と投げた際、これまでは「最適です！」と返してきたAIが、「〇〇のケースにおいてパフォーマンス劣化のリスクがあります。代替案として××を検討しましたか？」と、ブレーキをかけてくれるようになります。

### 3. 事実ベースのファクトチェックが走る

最新のドキュメントや仕様と照らし合わせ、「そのAPIは非推奨になっている可能性があります」といった客観的事実をベースに議論を展開してくれます。

## 【実例】Before / After で回答はこう変わる

抽象的な説明だけではイメージしづらいので、実際の壁打ちを比べてみます。お題は技術選定の相談です。

> **質問（ユーザー）**
> 「個人開発のSNSアプリ、バックエンドはFirebaseでいこうと思う。どうかな？」

### Before（カスタム指示なし＝YESマンAI）

> 素晴らしい選択です！🎉 
Firebaseは個人開発に最適なBaaSで、認証・DB・ホスティングがオールインワン。
サーバー管理も不要で、すぐにMVPを立ち上げられます。きっと良いアプリになりますよ！

一見親切ですが、リスクに一切触れていません。読み手は「お墨付きをもらえた」と安心して、そのまま突き進んでしまいます。

### After（カスタム指示あり＝壁打ち相手AI）

> 要件次第ですが、SNSという特性を踏まえると以下のリスクがあります。採用前に確認してください。
>
> 1. **複雑なクエリに弱い**：Firestoreは「フォロー中ユーザーのタイムラインを新着順かつ絞り込み」のような複合クエリが苦手です。フィードが主機能のSNSとは相性が悪い可能性があります。
> 2. **コストが読みにくい**：課金はread/write/delete回数ベース。タイムラインの再読み込みが多いSNSは、ユーザー増加に対してコストが非線形に跳ね上がりがちです。
> 3. **ベンダーロックイン**：Firebase独自APIに依存するため、後からの移行コストが高くなります。
>
> **確認したい点**：「個人開発で素早く出したい」のがゴールなら、初期はFirebaseで割り切るのも合理的です。一方で「将来的にスケールさせたい」のであれば、Supabase（PostgreSQLベースで複合クエリに強い）も比較検討する価値があります。どちらを優先しますか？

否定で終わらず、リスクの根拠・代替案・ゴールの再確認までセットで返ってくるのがポイントです。これは「冷たい否定」ではなく、レビューしてくれる同僚のフィードバックに近い感覚になります。

### CLI / IDE のコーディングエージェントの場合

Claude CodeやCursorのような、ターミナルやエディタで動くコーディングエージェントにも同じ考え方が使えます。これらは**プロジェクト直下に置いた指示ファイルを自動で読み込む**ため、Web版のカスタム指示に相当するルールをファイルとして管理できます。

| ツール | 指示ファイル | スコープ |
|---|---|---|
| Claude Code | `CLAUDE.md`（プロジェクト直下）／ `~/.claude/CLAUDE.md`（全体） | リポジトリ単位／ユーザー全体 |
| Cursor | `.cursor/rules/*.mdc`（旧 `.cursorrules`） | プロジェクト単位 |
| GitHub Copilot | `.github/copilot-instructions.md` | リポジトリ単位 |
| Gemini CLI | `GEMINI.md` | プロジェクト／ユーザー全体 |
| Codex ほか汎用 | `AGENTS.md` | リポジトリ単位 |

この方式には、Web版にはないメリットがあります。

:::note info
- **一度置けば毎回読まれる**：会話のたびに貼り付ける必要がありません。リポジトリにコミットすれば、**チーム全員のエージェントに同じ「忖度するな」ルール**が効きます。
- **プロンプトはほぼ流用可能**：今回の「振る舞い／検証／思考スタンス」はそのまま `CLAUDE.md` などに貼っても機能します。
- コーディング文脈では「**テストの前提を確認する**」「**既存コードのパターンに合わせる**」といったコード向けの観点を足すと、さらに精度が上がります。
:::

## 注意点：この指示は「万能」ではない

ここまで効果を強調してきましたが、この手法を過信するのもまた「鵜呑み」です。導入前に以下の限界を理解しておいてください。

:::note warn

- **ハルシネーション自体は消えない**：批判的に振る舞わせても、AIが事実と異なる内容を「もっともらしく」生成するリスクは残ります。「反証」として挙げられたデメリットが、そもそも間違っていることもあります。**指摘の根拠も自分でファクトチェックする**前提は変わりません。

- **「批判のための批判」になることがある**：強く釘を刺すと、本来妥当なアイデアにまで無理やり難癖をつけ、議論が空回りすることがあります。リスクの指摘を受けたら「それは自分のケースで本当に問題になるか？」と一段引いて判断してください。

- **指示が無視・希薄化されることがある**：会話が長くなる、複雑なタスクが重なるなどで、システムプロンプトの効きが弱まる場合があります。「褒め言葉が戻ってきたな」と感じたら、指示を再掲したり会話を仕切り直すのが有効です。

- **最終判断は人間の責任**：AIはあくまで論点を増やす壁打ち相手であって、意思決定の代行者ではありません。出てきた賛否を踏まえて決めるのは、最後まで自分です。

:::

要するに、このカスタム指示は「AIを信じる」ためのものではなく、**「AIを疑いやすくする」ための道具**だと捉えるのが正解です。

## まとめ：AIは「褒めてもらう相手」ではなく「使い倒す相手」

AIの回答を鵜呑みにせず、むしろ「AIに自分のロジックを論破させてみる」くらいのスタンスで挑むのが、現代のエンジニアリングにおいて最も安全かつ効率的です。

「AIの回答がちょっと浅いな」「YESマンになっていて議論の手応えがないな」と感じている方は、ぜひこのカスタム指示を設定して、AIを鋭い意見をくれる「冷徹で優秀なテックリード」に変身させてみてください。
