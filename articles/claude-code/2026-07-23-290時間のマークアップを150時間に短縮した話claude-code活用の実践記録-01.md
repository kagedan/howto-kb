# 290時間のマークアップを150時間に短縮した話——Claude Code活用の実践記録

## はじめに

本記事は、2026年6月に進行しているフロントエンドコーディング案件において、Claude Codeを実務投入した際の記録です。案件名および発注元は非公開としますが、規模・工数・作業内訳については実数に基づいて記述します。

なお、本記事が対象とするのはマークアップ工程のみです。デザイン（Figma制作）およびCMS・バックエンドへの組み込みは一切含まれていません。

### 成果物の規模

| 種別 | 数量 |
| --- | --- |
| HTMLページ数 | 17ページ |
| HTML総行数 | 約19,000行 |
| SCSS総行数 | 約25,000行（ソースファイルベース） |

| 画像ファイル数 | 218ファイル以上 |

### ページ構成

ページの内訳は以下のとおりです。

* トップページ（1ページ）：メインビジュアル、複数の案内セクション、ニュース、ライブ配信枠を含む
* カテゴリ詳細ページ（4ページ）：共通の構造を持ちながら、カテゴリごとに配色・素材・一部レイアウトが異なる。SCSSは各ページ3,000〜3,700行規模
* シーズン詳細ページ（1ページ）：上記カテゴリの派生。HTML約1,000行、SCSS約100行
* ガイドページ（1ページ）：スライダー、リーグ比較表を含む
* 年間スケジュールページ（1ページ）：タブ切り替えによる表示制御を含む。専用SCSSとJSを持つ
* 参加方法ページ（1ページ）：ステップ表示、装飾素材多用
* 用語集ページ（1ページ）
* アーカイブページ（1ページ）：カードグリッドレイアウト
* ニュース詳細ページ（5ページ）：テンプレートを基に、配信対象カテゴリ別のバリアント
* ニューステンプレート（1ページ）

### SCSS構成

SCSSはPartials構成で管理しました。

ビルドパイプラインは `sass` （expanded出力）+ `autoprefixer` であり、VSCodeのLive Sass Compileと同等の出力を再現する構成としています。

## Claude Codeが効いた場面

### 量産ページへの差分適用

本案件の最大の特徴は、共通構造を持つカテゴリページが4枚存在したことです。各ページはSCSSが3,000行以上あり、単純なコピーではなく、配色・画像パス・一部レイアウトの変更が必要でした。

1枚目を完成させた後、「このページを○○カテゴリ向けに差分を当てて」という指示を与えることで、HTMLおよびSCSSの書き換え箇所をClaude Codeが特定・置換しました。結果として、人間が行う作業は最終的な確認と細部調整のみに絞ることができました。

### SCSS大量記述の肩代わり

25,000行のSCSSのうち、大部分は構造的な繰り返しで構成されています。具体的には以下のようなパターンです。

* PCとSPで異なる `font-size` や `padding` 、レイアウトを持つメディアクエリ対応
* 親要素のネストに従った詳細セレクタの展開
* 変数（色、サイズ）を使いながらのバリアント生成

これらはFigmaの設計意図を言語で伝えると、Claude Codeが実装コードとして出力します。確認コストはかかりますが、タイピングにかかる物理的な時間はほぼゼロになりました。

### JavaScript初稿の生成

`main.js` （705行）の主要な実装は、ハンバーガーメニュー、ページトップボタン、アコーディオン、タブ切り替え、ヘッダー固定判定などです。これらの挙動をテキストで仕様として与えることで、動作する初稿を生成させました。デバッグや挙動の微調整は人間が行いましたが、ゼロからの記述に要する時間は不要となっています。

### 納品前チェック

HTMLのコーディング品質チェック（閉じタグ漏れ、alt属性の抜け、重複属性、不要なコメントの残存など）をClaude Codeに実行させました。人間のQAを「AIが発見した候補を確認する」作業に変換できたことで、見落とし率の低下と時間削減を両立しています。実際に、`</span>` の閉じ忘れや重複した `</a>` タグなど、目視では見落としやすいバグが検出されました。

## Claude Codeが苦手だった場面

### Figmaと実装の差異判断

Figmaのデザインデータと実際のブラウザレンダリングの間には、常に解釈の余地があります。たとえば「点線の太さと密度がFigmaと微妙に違う」「影の広がり方が意図と異なる」といった問題は、視覚的な感覚に基づく判断が必要であり、Claude Codeへの指示に落とし込む前に人間が正解を確定させなければなりません。

結果として、このカテゴリの問題はClaude Codeへの投入前後に人間の判断工程が必ず挟まるため、工数削減の恩恵を受けにくい領域でした。

### ページ構成・ディレクション判断

「このセクションをページのどこに置くか」「このカテゴリページとトップページの導線をどう整合させるか」といった構成レベルの判断は、成果物全体の意図を把握した人間が行う必要があります。Claude Codeに実装を任せる前提として、この判断は常に人間側にあります。

### レンダリング微調整

ブラウザ間の表示差異や、特定の組み合わせで起きるSCSSのスタック問題など、「なぜそうなるか」の原因特定が必要な局面では、Claude Codeの出力がそのまま正解になることは多くありませんでした。問題の切り分けは人間が行い、修正候補の生成にClaudeを活用するという役割分担になっています。

## 工数の内訳と削減分析

実績147時間の内訳を、削減要因ごとに推定します。

各項目の詳細な分析

### 1. 量産ページの差分生成（推定削減：-15〜20h）

カテゴリページ4枚のうち、2〜4枚目は「1枚目からの差分」として実装しました。手動の場合、1枚のページをコピーして全体を確認・修正するサイクルを3回繰り返す必要があります。SCSSが各3,000行以上ある規模では、この確認コストが1枚あたり5〜7時間に達すると推定されます。  
Claude Codeに差分の検出と適用を任せることで、確認と細部調整のみに作業を絞り込めました。

### 2. SCSS大量記述の代替（推定削減：-10〜15h）

25,000行のSCSSのうち、構造的な繰り返しと見なせる部分の割合は体感で50〜60%程度です。メディアクエリの複写、ネストに沿った展開、変数適用のパターン記述などがこれにあたります。  
タイピングを省略できた行数×平均記述速度で換算すると、10〜15時間に相当する削減が生じたと推定しています。ただし確認と修正のコストは残るため、これは純粋な削減の上限値として捉えるべきです。

### 3. HTMLマークアップの骨格生成（推定削減：-8〜10h）

19,000行のHTMLのうち、繰り返しパターンが多いセクション（定義リスト、テーブル、複雑なネスト構造を持つカードグリッド等）の骨格生成をClaude Codeに任せました。  
特にニュース詳細の5バリアント（合計約2,400行）は、1枚のテンプレートから差分で生成したため、実質的に4枚分の記述コストが削減されています。

### 4. JavaScript初稿生成（推定削減：-5〜8h）

`main.js`・`schedule.js` の合計約1,100行について、実装仕様をテキストで与えて初稿を生成しました。アコーディオン、タブ切り替え、ハンバーガーメニュー、スクロール連動ボタン表示などの標準的なUIインタラクションは、初稿の品質が比較的高く、人間の修正コストが少なくて済みました。  
ゼロから手書きした場合と比較して、5〜8時間の削減と推定しています。

### 5. 納品前チェック・バグ出しの補助（推定削減：-3〜5h）

HTMLのバグ検出をClaude Codeが担い、人間は確認と修正のみを行いました。検出されたバグには、閉じタグの漏れ、重複タグ、不正なalt属性、不要なTODOコメントの残存などが含まれました。  
目視での網羅的チェックに要する時間と比較して、3〜5時間の削減と推定しています。

### 削減工数の積み上げと実績との照合

| 削減要因 | 推定削減幅（h） |
| --- | --- |
| 量産ページの差分生成 | 15〜20 |
| SCSS大量記述の代替 | 10〜15 |
| HTMLマークアップの骨格生成 | 8〜10 |
| JavaScript初稿生成 | 5〜8 |
| 納品前チェック・バグ出し | 3〜5 |
| 合計 | 41〜58 |

上記の積み上げ値は41〜58時間です。見積もり上限290時間から実績147時間を引いた差分は143時間であり、積み上げ値との差分（約85〜100時間）は、見積もり自体の余裕分、およびClaude Codeが効果を発揮した箇所がこれら5項目以外にも分散していたことで説明できます。

換言すれば、特定の工程だけが劇的に短縮されたわけではなく、作業全体を通じてClaude Codeが低レベルな記述タスクを継続的に肩代わりし続けた結果として現れた数字であると言えます。

## まとめ・所感

### 再現可能な条件

本記事の成果が再現可能かどうかは、以下の条件に依存します。

!

効果が高い条件

* ページ数が多く、かつ共通構造を持つページが複数ある
* デザインが確定した状態でコーディングが始まる（Figmaが渡された状態）
* SCSS設計の方針をあらかじめ明確に持っている
* 実装者がClaude Codeの出力を判断・修正できる技術力を持っている

!

効果が低い・注意が必要な条件

* デザインが流動的で仕様が変わり続ける案件
* ブラウザ間の微妙な表示差異が品質基準になる案件
* 出力コードを確認せず採用し続ける運用（品質が著しく低下する）

### 「確認コスト」を正しく見積もること

Claude Codeを活用した場合でも、生成されたコードの確認と修正は人間が行う必要があります。「生成した＝完成した」ではなく、「生成したものを確認・採用するかどうかを判断する工程が必ず存在する」という認識が前提になります。

この確認コストを無視して工数見積もりを行うと、実際には想定より時間がかかります。本案件では、確認コストを含めた上での実績が147時間です。

### 最終的な所見

Claude Codeは「コーディングを自動化するツール」ではなく、「低レベルな記述を代替することで、人間が構成・判断・確認に集中できる環境を作るツール」として機能しました。

人間が担う判断の総量は変わりません。変わったのは、その判断を実装コードに変換するコストが大幅に下がったということです。
