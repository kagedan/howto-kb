## Claude Codeのソースコードの流出

2025年3月31日、Anthropicが`@anthropic-ai/claude-code`をnpmに公開した際、ソースマップファイルが同梱されてしまい、TypeScript約1,906ファイル・51万2,000行超の完全なソースコードが流出した。原因は`.npmignore`に`*.map`を書き忘れたという、たった1行の抜け漏れだった。

ただ、この記事で書きたいのは事件の経緯ではない。流出したコードや、それを分析した他の記事を読んでわかった**AIエージェントの設計思想**のほうが、私にとってはるかに価値があった。

私は`Mastra` + `AI SDK`でドキュメントQ&Aアプリを作っている。Claude Codeの設計を知ったとき、エージェント開発で「何に時間を使うべきか」の考え方が変わったので、その内容を整理することにした。

## エージェントのコアは20行のwhileループだった

流出コードを分析した記事を読んで最初に驚いたのは、エージェントループ自体がとても単純だったこと。

```
// Claude Codeのエージェントループを簡略化した擬似コード
// ※ 実際の流出コードそのものではなく、構造を再現したもの
async function agentLoop(messages: Message[]): Promise<string> {
  while (true) {
    const systemPrompt = buildSystemPrompt({ /* 動的に組み立て */ });
    const response = await anthropic.messages.create({
      system: systemPrompt,
      messages,
      tools: toolDefinitions,
      stream: true,
    });

    if (!response.hasToolCall()) return response.content;

    const toolResults = await Promise.all(
      response.toolCalls.map(async (call) => {
        await checkPermission(call);
        return executeTool(call);
      })
    );
    messages.push(response.toMessage(), ...toolResults);
  }
}
```

複雑なDAG（有向非巡回グラフ）も状態機械もない。「ツール呼び出しがあるか？」のwhileループが全てで、1回のレスポンスに複数ツール呼び出しがあれば`Promise.all`で並行実行する。これだけ。

では51万行は何かというと、このループを支える**ハーネス**に費やされている。コンテキスト管理、パーミッション、ツールスキーマ、エラー回復、メモリ、圧縮。Sebastian Raschkaはこれを分析して「現在の主要LLMのバニラ版は性能が近い。差を生み出しているのは**ハーネスの質**」と結論づけている。

つまり、ユーザーが体感する「AIの賢さ」の大部分は、モデルではなくハーネスが決めている。

ここからは、流出コードから読み取れる設計原則を5つに分けて整理し、私のアプリのコード例も交えながら、実際にどう使えるかを書いていく。

## 設計原則1：ツールは専用・型付き・制約埋め込み

Claude Codeは約52個の専用ツールを持っている。汎用のBashツールもあるが、そのツール定義自体に「`find`・`grep`・`cat`の実行にこのツールを使うな」と書かれている。

なぜ汎用シェルではなく専用ツールなのか。

| 理由 | 説明 |
| --- | --- |
| 監査のしやすさ | 型付きパラメータは解析が簡単。`bash -c "grep -rn..."`は検査できない |
| ガード強制 | Editツールは事前にReadを前提にして盲目的な上書きを防ぐ |
| 認知負荷の削減 | LLMが「どのコマンドを使うか」を推論する必要がなくなる |

もう一つ重要なのは、**安全制約をツール定義の中に埋め込む**パターンだ。  
ツールのdescriptionに「いつ使うか」「いつ使わないか」を明記して、LLMが必ず参照する場所に制約を置く。

私のアプリでもこのパターンを採用している。例えば、ベクトル類似度検索のツール定義はこうなっている：

```
export const semanticSearchTool = tool({
  description: `意味的類似度検索。ベクトル検索でドキュメントチャンクを検索する。

USE WHEN:
- 概念的な質問・言い換え・抽象的な質問のとき
- キーワード検索では取りこぼす可能性がある質問のとき

DO NOT USE:
- 固有名詞・品番・日付など完全一致が必要なとき（keywordSearch を使うこと）
- 既に十分な検索結果がある場合`,
  inputSchema: z.object({
    query: z.string().describe("検索クエリ（自然言語）"),
  }),
  execute: async ({ query }) => { /* ... */ },
});
```

`USE WHEN`と`DO NOT USE`を明記することで、LLMにツールの選択判断を委ねつつも、誤使用を防いでいる。これはClaude Codeのツール設計から直接学んだものだ。

ツールの数は多ければいいわけではなく、「5〜8個の特化ツール > 万能な1つ」が原則。私のアプリでは3つのツールに分けている：

| ツール | 役割 | 使い分け |
| --- | --- | --- |
| `semanticSearch` | ベクトル類似度検索 | 概念的・抽象的な質問 |
| `keywordSearch` | tsvector全文検索 | 固有名詞・品番・完全一致 |
| `chunkDeepRead` | 前後チャンク取得 | 検索結果の深掘り |

## 設計原則2：クエリ分類で処理を分岐する

Claude Codeのエージェントループは毎回同じ処理を通すわけではない。入力の性質に応じて適切なパスに振り分けている。

私のアプリでは、これをクエリ複雑度の3段階分類で実装した。ユーザーの質問をLLMで`simple`・`moderate`・`complex`に分類して、それぞれ別のパスで処理する。

```
ユーザーの質問
    ↓
[クエリ分類]
    ├── simple → LLM直答（検索なし）
    ├── moderate → ハイブリッド検索1回 → LLM
    └── complex → Agentic RAG（LLMが自律的に検索）
```

分類プロンプトはこんな感じ：

```
"simple": 挨拶・雑談・一般常識で答えられる質問
"moderate": 1つのドキュメントで答えられる単一トピックの質問
"complex": 複数ドキュメントの横断・比較・マルチホップ推論
```

`complex`と判定されたクエリだけがエージェントループに入る設計にしている。全てのクエリをエージェントに通すと、簡単な質問でもツール呼び出しが走って応答が遅くなるし、APIコストも無駄になる。Claude Codeも同様の考え方で設計されている。

この「全部を同じパスに通さない」という判断は地味だけど、体感のレスポンス速度とコストに直結する。

## 設計原則3：コンテキストを管理する（肥大化防止）

Claude Codeの流出コードで特に面白かったのが、コンテキスト管理の設計だ。Anthropicはシステムプロンプトを「静的部分」と「動的部分」に分離して、静的部分をキャッシュしている。

```
┌─── 静的部分（キャッシュ対象）───┐
│  エージェントの行動ルール
│  全ツール定義とスキーマ
│  安全制約・コーディングルール
├─── DYNAMIC_BOUNDARY ────────┤
│  現在のディレクトリ
│  Gitブランチ・変更差分
│  CLAUDE.md
│  セッションメモリ
└─────────────────────────────┘
```

これだけでAPIコストが72%削減できるという試算が出ている。

もう一つ、コンテキストの肥大化対策として**3段階圧縮**がある：

1. **クリッピング**：個別のツール出力が長すぎたら、先頭60%と末尾30%だけ残す
2. **インクリメンタル圧縮**：古いメッセージをローリング圧縮（直近30%はリッチに保持）
3. **要約**：コンテキスト上限が近づいたら軽量モデルで会話サマリーを生成

自動コンパクション機能にバグがあったとき、1日約25万回の無駄なAPIコールが発生していたという実例もある。コンテキスト管理の設計ミスがどれだけ大きなコストになるか、よくわかるエピソードだと思う。

ここは私のアプリではまだ十分に実装できていない部分で、会話が長くなったときの圧縮処理は今後の課題になっている。ただ、ツール出力のクリッピングは意識していて、検索結果を返すときにtopK件に絞っているのも、同じ発想だ。

## 設計原則4：メモリは階層構造にする

Claude Codeのメモリは3層構造になっている。

| 層 | 内容 | ロード戦略 |
| --- | --- | --- |
| 第1層 | MEMORY.md（軽量インデックス） | 常時ロード |
| 第2層 | トピック別ファイル | 必要時にオンデマンドロード |
| 第3層 | 過去の会話ログ | grep検索時のみ参照 |

ポイントは「常時ロードするものを最小化する」こと。第1層はプロジェクト概要と重要ファイルパス、トピックファイルへのポインタだけ。500トークン程度に抑えている。全てをコンテキストに詰め込むのではなく、必要な情報に到達するための「目次」だけを持たせる設計だ。

さらに面白いのが**autoDream**という仕組みで、アイドル状態のときにサブエージェントが起動して、過去の観察結果をマージ・重複除去・矛盾解消する。人間が寝ている間に記憶を整理する脳の仕組みに似ている。

この階層構造の考え方は、RAGアプリのドキュメント管理にもそのまま適用できる。私のアプリでも、ドキュメントをチャンクに分割してベクトルDBに格納している。ファイル単位のメタデータ（ファイル名・種類・登録日）はPostgreSQLのテーブルに、チャンクの詳細はpgvectorのインデックスに、という2層構造だ。全チャンクの中身を常時参照するのではなく、検索で関連チャンクを絞り込んでからテキストを取得する。Claude Codeの階層メモリと同じ発想だと思う。

## 設計原則5：セキュリティは「使う場所」に埋め込む

Claude Codeのセキュリティ設計は多層防御で、5つのレイヤーを通過してからツールが実行される。

```
リクエスト → トラスト確立 → モデル判断 → パーミッションチェック
 → Bashセキュリティ（23種） → ユーザー確認 → 実行
```

ここで学べる原則は2つある。

1. **制約をツール定義内に書く**：セキュリティルールをどこか別の設定ファイルに書くのではなく、ツールの`description`に直接埋め込む。LLMは毎回ツール定義を読むので、制約を見落とすリスクが下がる。
2. **判断と実行を分離する**：未公開機能のKAIROS（常時稼働バックグラウンドエージェント）では、「何をすべきか」の判断と「実際にやる」実行を明確に分けて、実行前にパーミッションチェックを挟んでいる。エージェントが自律的に動くほど、この分離は重要になる。

実際に`CVE-2025-59828`という脆弱性では、以下のように順番を間違えるだけでセキュリティホールになっている。これはClaude Codeの起動時に、ユーザーの信頼性を確認する前にフック経由でコードが実行できてしまうという問題だった。

```
✅ 正しい順序：トラスト確立 → チェック → 実行
❌ CVE-2025-59828：コード実行 → トラスト確立（確認する前に動いてしまう）
```

エージェントの自律性が上がるほど、こうした設計の細部が致命的な差になるのだと感じた。

## エージェント設計のまとめ

Claude Codeの流出コードから抽出した設計原則を、表として整理する。

|  | 原則 | 実践 |
| --- | --- | --- |
| 前提 | ループはシンプルに、ハーネスに投資 | whileループ + メッセージ履歴で十分 |
| 1 | ツールは専用・型付き | 5〜8個の特化ツール。`USE WHEN`、`DO NOT USE`を明記 |
| 2 | クエリ性質で処理を分岐 | 全クエリを同じパスに通さない |
| 3 | コンテキストの肥大化と戦う | 静的・動的分離、クリッピング、圧縮 |
| 4 | メモリは階層構造 | 常時ロードは最小限。必要な情報への「目次」だけ持つ |
| 5 | セキュリティは使う場所に | ツール定義にルール埋め込み。判断と実行の分離 |
| 結論 | ハーネスの質 > モデルの質 | 同じLLMでも体感品質はハーネス次第 |

Raschkaの主張を一言で言えば、AIエージェントの品質を決めるのは「地味な配管工事」。コアループをどう磨くかではなく、コンテキスト管理・ツール設計・メモリ・セキュリティといったハーネスにどれだけ投資できるかが、プロダクション品質のAIアプリを作るうえで一番大事だと思います。
