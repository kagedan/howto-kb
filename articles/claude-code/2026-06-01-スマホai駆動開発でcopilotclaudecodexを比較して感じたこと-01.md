## はじめに

**※この記事の内容は筆者が使用した当時の仕様に準拠しています。  
現在とは各ツールの仕様が異なる可能性があります。**

この記事は、GitHub Copilot、Claude、Codexの一般的な性能比較ではありません。

あくまで、自分がスマホからAI駆動開発を進めた範囲での所感です。

「どのAIが一番賢いか」を評価したいわけではありません。評価軸は、\*\*「スマホから隙間時間にタスクを投げ、個人開発を前に進めやすかったか」\*\*です。

スマホAI駆動開発という前提での使い勝手について、プロジェクトごとの利用経験をもとに整理します。

開発フロー全体やスマホでうまくいったこと・限界については、別記事「スマホからAI駆動開発で個人開発を進めた実践録」で整理しています。

---

## 比較軸

今回の使用感を評価するにあたって意識した軸は以下のとおりです。

* スマホから使いやすいか
* タスクを投げて放置しやすいか
* 進行状況が見えやすいか
* コンテキストを制御しやすいか
* GitHub連携や認証が自然か
* 隙間時間開発に向くか

モデルの賢さや精度の絶対値より、「スマホから短い時間で操作し、そのまま別の作業に移れるか」という体験面を重視しています。

これは私のニーズが **通勤時間や休憩時間といった隙間時間を使って開発を前に進めること** だったからです。

## GitHub Copilotの所感

### 使った場面

* **利用環境**：GitHubアプリ（スマホ）
* **利用プロジェクト**：家計簿アプリ（プロトタイプ作成、機能追加の一部）
* AI Assisted Editorでは未使用

### 良かった点

**GitHubアプリへの統合が自然でした。**

GitHub上でリポジトリを見ている流れでそのままCopilotを呼び出せます。追加の認証フローが少なく、コードレビューや実装相談を依頼するまでの導線がスムーズでした。

複数モデルを使い分けられる点も、試しながら使う際には良かったです。

### 気になった点

**タスクの進行状況が見えにくかったです。**

タスクを投げた後、エージェントセッションでは現在の進捗状況を確認できません。  
実はAIエージェントのタスクはGitHub Actionsで動いているのでそちらを見れば状況確認は可能なのですが、それに気づくまでは実行に失敗したと勘違いして同じタスクを2回実行してしまったことがありました。

スマホから隙間時間に投げるという使い方では、「投げたら安心して離れられる」かどうかが重要です。その点で、進行状況の見え方がもう少し明確だとよかったです。

**autoモードの不透明さ。**

GitHub Copilotは複数モデルを使用可能なだけに、autoモードではどのモデルが選ばれるかが分かりません。モデルによってプロンプトへの反応や挙動が変わるため、「今回どのモデルが動いているか」が分からないと、反応を読みにくい場面がありました。

また、他のツールと共通する話ですが、規約を与えていないと見当違いの方向へ実装が進むことがあります。これはCopilotに固有の問題ではありませんが、タスク設計の重要性を改めて感じた点です。

### スマホ開発との相性

GitHub Copilotは、GitHubアプリに統合されていることによる導線の自然さが強かったです。一方で、タスク進行状況の見え方やautoモードの不透明さがあり、スマホから隙間時間に安心して投げる用途では少し不安もありました。

悪くはないですが、UXに粗さを感じる場面があったというのが正直な所感です。

## Claudeの所感

### 使った場面

* **利用環境**：Claude CLI（PC）、Claudeスマホアプリ
* **利用プロジェクト**：
  + 家計簿アプリ（機能追加）
  + AI Assisted Editor（コンポーネント調整、リファクタリング、複雑な作業）

### 良かった点

**PC環境ではガードレールやコンテキスト制御が豊富で、頼りになります。**

Claude CLIをPC上で使う場合、コンテキスト制御の仕組みや設定の柔軟さが強かったです。複雑な作業、コンポーネント調整、リファクタリングなど、精度が求められる場面では安心感がありました。

### 気になった点

**Claudeスマホアプリ経由での利用は、スマホから放置する用途に向きませんでした。**

Claude CLI自体はPC環境では有効です。しかし、Claudeスマホアプリ経由でコマンドを実行させる場合、操作のたびに許諾を求められる確認フローがあります。

「スマホからタスクを指示して、あとは放置する」というやり方には合いませんでした。隙間時間でのスマホ開発という文脈では、確認フローの多さがネックになりました。

### スマホ開発との相性

Claude CLIはPC環境では頼りになります。ただし、Claudeスマホアプリ経由で使う場合は、コマンド許諾の確認が多く、「スマホから隙間時間でタスクを投げて放置する」というニーズには合いにくかったです。

「Claudeは使いづらい」という話ではありません。**Claude CLIはPCでは有効。ただしスマホアプリ経由の隙間時間開発には合いにくい**、というのが正確な所感です。

## Codexの所感

### 使った場面

* **利用環境**：Web
* **利用プロジェクト**：AI Assisted Editorの主要開発

### 良かった点

**並列実行とサンドボックス環境が、スマホからの放置スタイルに合っていました。**

今回の使い方で最もフィットしていたのがCodexです。

複数のタスクを並列で走らせられる点が特に良かったです。スマホから複数の実装タスクを投げ、結果をまとめて確認するという流れが作りやすかったです。

サンドボックス環境でタスクが実行されるため、「投げたまま別の作業に移れる」安心感がありました。スマホから隙間時間に使う場合、「投げた後に何かしなければならない」回数が少ないほど使いやすいです。

スマホからでも使える機能が多く、UI/UX面も良かったです。

### 気になった点

**計画モードとSkillsには癖があります。**

計画モードは、実装を進める前に計画だけ確認できる仕組みですが、指示の中に「実装する」などの単語が入ると、計画モードでもファイルが変更されることがありました。機械的なガードレールというよりは、プロンプトベースで挙動が変わる印象です。

Skillsについては、明示的に使うよう指示しないと使われないことが多かったです。  
Claude Codeではスラッシュコマンドで明示的に読み込ませる事もできますが、  
Codexにそういった機能は見つからなかったため、プロンプトで指示する必要があるようです。

全体的に、プロンプトで補完しなければならない細かい要素が多く、  
タスク設計に気を使う必要がありました。  
このあたりはOpenAIの思想に由来する部分かもしれません。

### スマホ開発との相性

Codexは、今回の「スマホから隙間時間にAI駆動開発を進める」という目的にはかなり合っていました。特に、並列実行できること、サンドボックス環境でタスクを任せられること、スマホから使える機能が多いことは大きかったです。

一方で、計画モードやSkillsの扱いには癖があります。プロンプトで明示しなければ期待通りに動かないこともあり、タスク設計には注意が必要でした。使いやすいですが、癖を理解したうえで使う必要があります。

## まとめ

今回の用途では、単純な性能比較というより、「スマホからタスクを投げて、隙間時間で開発を前に進められるか」が重要でした。

その観点では、**Codexが最も目的に合っていました。** 並列実行やサンドボックス環境により、スマホからタスクを投げ、そのまま別の作業に移れる体験が強かったです。

**Claude CLIはPC環境では安心感があり、複雑な作業やコンテキスト制御には向いています。** ただし、Claudeスマホアプリ経由で扱う場合はコマンド許諾が多く、スマホから放置前提で使うには確認フローが重かったです。

**GitHub CopilotはGitHubアプリとの統合が自然で導線は良いです。** しかし、タスク進行状況の見え方やautoモードの不透明さにより、今回のようなスマホ中心の開発では少し不安が残りました。

結局、重要なのは「どのAIが一番賢いか」ではなく、自分の開発フローと合っているかでした。

スマホから隙間時間で開発を進めるという目的のもとで使うなら、各ツールのUXや操作感の違いが体験に直結します。モデルの性能だけでなく、「投げて放置できるか」「進行状況が見えるか」「スマホで操作しやすいか」という観点で選ぶことが、今回のような使い方では重要でした。
