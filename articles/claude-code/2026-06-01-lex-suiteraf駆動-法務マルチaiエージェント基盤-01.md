## はじめに

**Lex Suite** は、Microsoft Agent Hackathon 2026 への応募作品として構築した、法務業務向けのマルチエージェント AI システムです。

筆者は**現役の組織内弁護士で、現在、国内大手金融グループの証券会社の法務部管理職**を務めており、日々の法務実務と組織のリーガルリスクマネジメントの双方を背負う当事者です。

### 既存のリーガルテックは「帯に短し襷に長し」

現在、市場には多くの「AI契約書レビューサービス」や「AIリサーチツール」などのいわゆるリーガルテックが存在します。それらを実務に導入しようとする際、法務の現場では決まって次のような壁にぶつかります。

第一に、**契約書レビューツール**の壁です。  
これらは契約書の一般的な問題点を指摘し、標準的な修正案を提示してくれますが、当事者である現場の評価はシビアです。

> 「一般的な指摘はしてくれるが、当社の**固有のリスク許容度、すなわち、リスクアペタイト**に合っていないため、結局人間が採否を一つ一つ判断し、指摘内容を全文コピペし直さなければならない」

第二に、**AIリサーチツール**の壁です。  
官公庁の新着情報や法律文献・判例情報などを網羅的に集めてくることは得意ですが、膨大なヒット件数から「当社の業務遂行のために本当に必要な情報」を、人間が改めてスクリーニングし直す必要があります。

第三に、**契約書自動作成ツール**の壁です。  
自社固有の契約書作成を自動化するには、下準備として可変項目の洗い出し、Wordファイル内へのコーディング、システム入力などの膨大な手間がかかります。せっかく仕組みをセットアップしても、年間10〜20件しか利用実績がないといったケースも多く、初期コストを上回るメリットを感じにくいという問題があります。

これら既存の汎用リーガルテックは、どれも「平均的・一般的な機能」を提示するに留まるため、現場の実務にとっては「帯に短し襷に長し」となり、結局は法務担当者の手作業による「二度手間」を発生させているのが実態です。

### 法務実務の「本質的な難しさ」と RAF という概念

なぜ既存ツールではうまくいかないのか。

その理由は、法務実務の「本質的な難しさ」にあります。

法務判断の多くは、白黒はっきりした「適法か違法か」の判定ではありません。

その間にある広大な**グレーゾーンにおける許容性判断**こそが仕事の核心です。また、大半の業務が「定型的ながら機械的・画一的とまではいえない」という点も、自動化・AI化を拒む鬼門となっています。

本当に難しいのは、「グレーな領域において、当社固有の様々な事情を勘案し、どこまでリスクを許容してビジネスを前に進めるか」という**組織固有の意思決定**に他なりません。この「どこまでのリスクを許容するのか」の基準を、法務・ガバナンスの用語で **RAF（Risk Appetite Framework：リスク許容枠組み）** と呼びます。

### 解決策としての「RAF駆動」

この法務実務の特徴を踏まえ、汎用ツールの平均的な判断を後から人間が手作業で補正するのではなく、最初から自社固有の基準である RAF に基づいて AI を駆動する——これが、本システムが提唱する**RAF駆動**という考え方です。

会社固有の上位規範である `RAF.md` を AI の憲法（Constitution）として全エージェントに貫流させ、状況に応じて「当社の基準」で自律動作する**セミオーダーの法務エージェント**を構築する。

これこそが、既存ツールに依拠せず本システムを「自作」しようと思った本質的な動機です。

### Lex Suite の全体像

本システムは、Slack を入口に、4 つの AI エージェント（Researcher / Reviewer / Composer / Orchestrator）が協働します。

その中核を貫くのが **RAF.md** という共通の上位規範です。`RAF.md`は Blob Storage に置かれ、実務を担う 3 エージェント（Researcher / Reviewer / Composer）が直接参照し、Orchestrator はその貫流を統括します。

エージェントは Anthropic Claude Managed Agents (Beta) と Azure AI Foundry の二系統に分散配置し、MCP（Model Context Protocol）サーバ群を Azure Container Apps 上で稼働させることで、法務に不可欠な「厳格なセキュリティ」と「最新情報の取得」を両立しています。

本記事は、現役弁護士としての「法務の本質的なペイン」と、それをマルチエージェント×MCPの最新スタックでどう解いたかという「泥臭い実装プロセス」の双方から、以下の構成で解説します。

* **設計理念**: なぜそう設計したか（弁護士・法務マネジメント実務の視点）
* **法務業務の実態**: 現場の制約と仕事の中身
* **技術スタック**: 実装・依存・経路・実戦で突破した3大トラップ

## Lex Suite とは

### 4 Bot 構成

| # | エージェント | 動作 | 経路 | モデル |
| --- | --- | --- | --- | --- |
| ① | **LexResearcher** | 調べる | Anthropic Managed Agents (Beta) 直接 | claude-sonnet-4-6 |
| ② | **LexReviewer** | レビューする | Azure AI Foundry Direct Endpoint | claude-sonnet-4-6 |
| ③ | **LexComposer** | 書く | Azure AI Foundry Direct Endpoint | claude-sonnet-4-6 |
| ④ | **LexOrchestrator** | 統括 | Anthropic Managed Agents (Beta) 直接 | claude-sonnet-4-6 |

### アーキテクチャの全景

ユーザーは Slack 上で Manager に依頼を投下します（例: 「個人情報保護法（個情法）の改正について調べて、秘密保持契約（NDA）への影響もまとめて」）。

Manager がそれを「調べる → レビューする → 書く」の組合せに分解し、各 sub-agent を非同期 spawn で起動、進捗を監視して結果を統合し、Slack に最終応答 + 添付 .docx を返します。

## 設計理念

Lex Suite の設計判断は、法務業務の本質と現場制約から導いています。

### 判断 1: RAF を Constitution として全エージェントに貫流させる

法務業務の半分以上は「グレーゾーンにおけるリスク判断」です。

そもそも、適法か違法かが一見して明白な事案は、現場から法務に相談として上がってきません。

法務の手元に来るのは「個人情報をどこまで取って良いか」「この契約条項のリスクは飲めるか」「この新規ビジネスを止めるべきか」といった、法律に明確な答えのない案件です。

こうしたグレーな領域で、組織として「どこまでリスクを取るか」を判断するのが、法務部門の本質的な責任の一つです。

「はじめに」で触れた RAF とは、単なる心構えやスローガンではなく、**リスク類型ごとに「どこまでが許容（Accept）か、どこからが要エスカレーション／不許容（Not Accept）か」を成文化した社内文書**です。

本システムの `RAF.md` でも、たとえば個人情報に関わるリスクを「Not Accept」と位置づける条を置いており、これは後述の dry-run 出力（🔴 判定で「RAF 第 11 条（個人情報リスク = Not Accept）」が引かれる箇所）で各エージェントが実際に参照します。

このRAFを、実務を担う各 LLM エージェント（Researcher / Reviewer / Composer）の上位規範（Constitution）として共通参照させ、とりわけ契約書のレビューや法律文書作成の場面においては、個別の下位規範（契約書ひな型・ガイドライン・プレイブック等）をより優先度の高いものとして参照させ、AIを「RAF駆動」させるというのが、LexSuiteの根本的なアイディアです。

`RAF.md` を改訂すれば全エージェントの動きに反映される構造です。

```
人間：RAF を策定する。最終判断も留保する。
AI ：RAF に縛られた範囲内で、判定の候補を機械的に出す。
```

「AI 丸投げ」でも「人間が全部判断」でもない、第三の道として RAF を中心に置いた設計です。

### 判断 2: 「分野」ではなく「動作」で分ける

企業法務の課組織は分野で分かれています（金融規制 / 取引法務 / 知的財産 / 会社法務等）が、これは AI エージェントの分割軸としてはなかなかうまくいきません。

実際の法務案件は分野横断（企業合併・買収（M&A）は会社法 × 取引法務 × 規制対応）で、分野で分けると「どの係に振るか」で組合せが爆発してしまいます。

そこで、このLex Suiteでは、法務業務をAIエージェント化する際に、「調べる」「レビューする」「書く」という「動作」で役割を分担することにしました。

法務の固有業務の本質を解きほぐすと、「リサーチ」・「レビュー」・「ライティング」という3種類の「動作」に行きつきます。

```
調べる：法令・制度趣旨・判例等、関連する法律文献等を調べる (例: 法令改正の経緯・趣旨・内容等の調査)
レビューする：一定の規範に照らして対象をレビューする (例: 新規取引のリーガルチェック)
書く：法的なコンテンツを一定の書式に従って文書化する (例: 法令制度調査レポート)
```

動作軸（Research / Review / Compose）で 3 エージェントに分け、Orchestrator が動作の組合せフローを管理します。

分野は各動作の内部で横断扱いします（知財案件であれ金融規制案件であれ、Researcher が関連 DB を呼ぶだけで対応）。

### 判断 3: 幅のある現場で「予測可能 × 低学習コスト」に動かす

法務 AI を実際に使うのは、ベテラン弁護士だけではありません。

法務アシスタント、契約担当、新規ビジネス担当、コンプライアンス担当など、**AI リテラシーに大きな幅がある現場**です。

加えて、日本の会社では定期的な人事異動があり、法務部所属の職員でも必ずしも法務専門性が高いとは限りません（3〜5 年で異動する組織では、配属直後の担当者は法務知識自体に幅があります）。

**AI リテラシーの幅 × 法務専門性の幅** が現場の実態です。

リテラシーや専門性の低いユーザーに「AI の出力を補正する」「プロンプトを工夫する」を求めることはできません。

AI 側の挙動が予測可能でないと、出力を盲信して誤った業務判断につながるリスクがあります。

「契約書を読む」「最新法令を調べる」「.docx を起案する」を 1 個の万能 LLM に詰めると、プロンプトが長大化し優先順位判断が状況依存になり、同じ依頼でも出力形がブレます。

**責任範囲を明確に切ったマルチエージェント構成**で、各エージェントは決まった動作だけを行い、入出力のフォーマットも決まっている形にしました。

挙動が予測可能になり、ユーザーは「このエージェントはこれをする」を理解した上で使えます。

そして、この予測可能なエージェント群を、**現場が既に日常的に使っている業務チャット（Slack）** の上に載せました。Microsoft Teamsを導入している企業であれば、Teamsのアプリという形で導入でも構いません。

新しい専用ツールを一から導入すると、その操作自体を覚えるコストが発生し、リテラシーの幅がそのまま導入の壁になります。

これに対し、誰もが毎日触れているチャットに「依頼を書いて投げるだけ」でエージェントが応答する形にすれば、ユーザーの学習コストはほぼゼロになります。

挙動がある程度予測できるエージェントを、既存の業務導線にそのまま組み込む——この組合せによって現場での実利用が促進され、結果として組織全体の生産性が底上げされます。UI をゼロから作るのではなく、すでにそこにある業務基盤を入口にする、というのがここでの判断です。

### 判断 4: 判断は人間が留保する — LLM は「羈束裁量」

法務判断の責任は最終的に人間（弁護士・法務マネジメント）が負います。

誤判定が訴訟リスクや行政処分に直結するため、AI に「判断者」の地位を与えることはできません。

AI の役割は **判定の候補を、根拠とともに、機械的に提示する** ことです。最終判断は人間が留保します。

法律用語に **羈束裁量（きそくさいりょう）** という概念があります。

行政庁の判断が、法令の要件に縛られた範囲内で僅かな選択の余地のみ認められる、という意味です。「自由裁量」（全面的に任せる）でも「完全にハードコード」（裁量ゼロ）でもない、**枠付きの裁量** です。

Lex Suite の各エージェントの LLM はこの羈束裁量に近い動き方をします。  
① RAF（会社の上位規範）  
② プレイブック（契約類型別ガイドライン）  
③ MCP tool の input\_schema と description  
④ agent instructions  
という 4 種類の **枠** に縛られた範囲内で、LLM が選択肢を出します。

判定の候補は LLM が出しますが、その候補は上記 4 種類の枠の中に限定され、最終判断は人間に渡します。

RAF Triage（🔴/🟡/🟢）は候補の提示で、エスカレーションされた事項に対する判断（実際にどう動くか）は人間が決定します。

### 判断 5: python-docx 一本、LLM に Word を直接編集させない

法務文書は「Wordの修正履歴で契約交渉をする」文化です。

レビュー結果は Word の「変更履歴の確認」UI で開ける形（w:ins / w:del XML エレメント、コメントの w:comment）で渡さないと業務に乗りません。

Markdown のままでは社内共有 → Print → 顧客送付 のフローに通せませんし、「ここを直した」をテキストで説明するのも非効率です。

Word の XML 構造は仕様が緻密で、LLM に直接吐かせると壊れる確率が高い（出力タイミング・トークン制限・コール失敗の影響）です。

そこで、Lex Composerで採用したアプローチは、「LLMにJSON構造化したレビュー内容を出力させ、それをpython-docxで決定論的にWordファイルを調製する」というものです。

Word の構造編集は python-docx + lxml で確定的に行い、LLM には Word を直接吐かせません。

LLM の出力は構造化 JSON（sections / values / 修正提案）までで、JSON → .docx の変換は python-docx で固めています。

LLM のコール失敗・再試行の影響を Word の構造編集に持ち込みません（フォーマットと内容で異常系を分離）。

### 判断 6: 死ぬ UI と残る UI を分ける

法務業務の最終成果物は .docx ファイルです。

今後は、AI・AIエージェントを前提とした業務改革の影響により、この実務慣習が変わっていく可能性はありますが、少なくとも、現時点における大半の業務が、.docxのやり取りによって行われています。

メール添付で顧客送付、社内共有フォルダで版管理、Print で社内会議・・・

Lex Suite の最終出力は .docx に着地させます。

Slack の AI チャット履歴は時間が経つと参照されにくくなります。AI が出した「丁寧な説明テキスト」も、業務に乗るのは結局 .docx に着地した形です。

そこで、Slack 上の AI 応答は「流れる UI」、添付の .docx が「残る UI」として業務にそのまま乗る形を取りました（Slack を入口として採用した理由は判断 3 を参照。本判断は、そこで生まれた出力をどこに着地させるか、という別軸の話です）。

例えば、Lex Reviewerは修正履歴付きの正規の .docx を返すので、人間レビュアーはそのまま「承認 / 拒否」を Word 上で操作できます（後続フローへの引継ぎが完結します）。

## 法務業務の実態

設計判断の根拠となる、現場の制約と仕事の中身です。

### 法務リサーチの実態

法務リサーチは、1 論点について以下のような複数経路を横断する仕事です。

```
1. 法令本文（条文・改正経緯・施行日）
2. 判例・命令・審決
3. パブリックコメント（パブコメ）本文 — 行政の制度趣旨資料
4. 監督官庁の最新発信（Q&A・ガイドライン）
5. 国会・帝国議会の議事録 — 立法経緯
6. 学術論文 — 学説・解釈
7. 国立国会図書館のレファレンス — 関連資料探索
8. Web の業界動向・専門家コメント
```

これを人間がやると膨大な時間を要します。論点 1 件で 10-30 件の一次情報を集め、矛盾なく統合してレポートにする必要があります。AI に任せたいのはまさに、**様々なソースからの情報の取得と統合作業**です。

「最新を常時追う」必要がある領域（法改正・監督指針・最新ニュース）と、「過去蓄積に変化がない」領域（パブコメ・判例）は、検索方式を分けるのが効率的です。

前者はクローラで取りに行く、後者は RAG で意味的に拾う、という切り分けです。

例えば、日々官公庁のウェブサイトなどで公表される法令制度改正の情報については、新着情報のURLをあらかじめ特定しておき、クローラを作成します。官公庁のウェブサイトは、省庁によって構造が異なるため、最初に多数のクローラを作成するのは大変ですが、一度作ってしまえば、アドホックに情報を取得することも可能ですし、エージェントのスケジューラの機能を用いて、自動的かつ定期的に情報を取得させることも可能になります。

これに対して、パブリックコメントの結果（国民からの質問や意見に対して、監督官庁が回答をしたものの記録）については、日々改正されるものではなく、あくまで過去の回答結果を参考資料として用いるものです。そして、「パブリックからのコメント = 一般用語」と「行政の回答 = 専門用語」が同じレコード内に混在するため、語彙ベースの完全一致型の検索では、どうしても精度が落ちてしまいます。

そこで、パブコメ結果をベクトルデータベース化し、セマンティック検索（embedding）をできるようにすることが効果的です。

リサーチ結果は、必ず RAF の観点から、要対応判定を行います。

要対応判定は、「緊急性 × 重要性」 の 2 軸に分解しバイナリ判定を行うことで、人間が「赤・黄・緑のどれか」を直接決めるのではなく、各軸の ◯/× を判定した結果として色が機械的に決まる形にしています。

「中間判断」は法務マネジメント実務でも最もブレやすい領域ですが、このアプローチ（「判断」をバイナリな判定に細分化し、比較的基準の明確な「白か黒か」の判定の集積とすること）によって、「知識と経験を有する人間の職人技的な判断」を「AIによる決定論的な判定」に近づけられる点がポイントです。

### 契約レビューの実態

契約レビューは「うちのひな型」と「相手方ドラフト」の差分を取り、差分各々について「この変更を飲めるか」を判断する仕事です。判断基準は二段構成です。

```
Layer 1: RAF (Risk Appetite Framework)
   会社が定めた「どこまでリスクを取るか」の上位規範

Layer 2: 契約類型別プレイブック (Guidelines、交渉ガイドライン)
   秘密保持契約（NDA）/ 業務委託（外部委託）/ ファイナンシャル・アドバイザー（FA）等
   レッドライン（譲歩不可ライン）/ 標準条項 / 典型ケース
```

修正提案を本文に組み込み、コメントで議論を残し、修正履歴付きの .docx を相手方に返送できる形まで含めて 1 つの仕事になります。指摘の根拠を遡れること（「RAF 第○条」「プレイブック §X 該当」）が、社内で AI 判定を採用するための前提条件です。

「ひな型改訂の緊急度」判定は、「全社的なリスク観点」とは別軸の意思決定軸です。「個情法改正で会社全体の方針見直しが要る」と「うちの NDA ひな型をいつまでに改訂するか」は対象が違います。Lex Suite は両軸の判定を **並置** で出します。

### 法務文書化の実態

法務業務の最終成果物は .docx ファイルです。

表紙・章番号・フッタで社内ブランディング、章構造で論理階層、修正履歴で版管理、コメントで議論履歴。Markdown のままでは業務フロー（社内共有 → Print → 顧客送付）に通せません。

法務文書の作成パターンは大きく 2 つあります。

* **ひな型 + 案件情報 → ドラフト**：契約書ドラフト、社内承認文書
* **リサーチ結果 → 構造化レポート**：制度動向レポート、社内ナレッジ発信

法務文書自動生成の一般的なパターンは「タグ埋込ひな型 + Questionnaire プリセット」ですが、ひな型ごとにタグを埋め込み Questionnaire を整備・メンテナンスするコストを考えると、年間 100 件以上の案件規模でないとなかなかペイしません。中規模法務（法務 1-5 名規模、年間 20〜数十件、毎回パターンが微妙に違う）では、この初期導入コストを負担するより、**その場でひな型を読み込み、可変項目を抽出し、ドラフトする** アプローチが効果的です。

社内規定スタイル（表紙の意匠 / 章番号の形式 / フッタの版表記）が決まっている法務文書では、LLM の出力をそのまま貼るのではなく、社内規定スタイルの .docx に落とすことで「業務にそのまま乗る」形になります。

### オーケストレーションの実態

ユーザーの依頼は単機能呼出ではなく、複合タスクが大半を占めます。

```
「個人情報保護法の改正について調べて、NDA への影響も整理して、
  必要ならうちのひな型改訂案も出して」
   ↓
   調べる (Researcher) → 書く (Composer) → レビュー (Reviewer)
```

Manager が動作の組合せフローを管理することで、ユーザーは「次は何を呼ぶか」を意識せずに依頼を投げるだけで複合タスクが完走します。リテラシーの低いユーザーに「フロー設計」を要求しない構造です。

## 技術スタック

技術選択は、LLM の確率的な振る舞いを安全な範囲に収め、法務実務に必要な再現性とセキュリティを満たすことを基準に行いました。以下、その構成と、なぜその選択をしたかを述べます。

### 全体像：4 Bot と 3 MCP サーバ

本システムは、利用者接点となる **4 つの Slack Bot** と、システム操作・情報アクセスを決定論的に担う **3 つの MCP サーバ** からなる、Python ベースのマルチエージェント構成です。

* **Slack Bot 4体（利用者接点）**:  
  `lex-orchestrator-slack-bot`, `lex-researcher-slack-bot`, `lex-reviewer-slack-bot`, `lex-composer-slack-bot`
* **MCP サーバ 3体（決定論的道具箱）**:  
  リサーチを担う `lex_researcher`、テンプレート・ガイドラインを司る `lex_reviewer`、およびジョブの統括を担う `lex_orchestrator`

エージェント実行基盤は二系統です。自律ループを回す **Anthropic Managed Agents**（Researcher / Orchestrator）と、ループを回さず単発で呼び出す **Azure AI Foundry Direct Endpoint**（Reviewer / Composer）。この使い分けの理由は後述します。

### 言語・ランタイム

主要言語は Python です。MCP サーバ群・Slack Bot 群はいずれも `requires-python = ">=3.10"` を要件とし、基本コンテナは軽量な `python:3.11-slim` です。

例外はリサーチ用の `lex-researcher-mcp` です。動的ロードを伴う官公庁サイトのクローリングとヘッドレスブラウザ（Chromium）の同梱が必要なため、Playwright 公式の Python イメージ `mcr.microsoft.com/playwright/python:v1.45.0-jammy` を採用しています。一次情報を直接取りに行くために、このコンテナだけは重くなっています。

### UI / 利用者接点

フロントエンドを自作せず、`slack-bolt` の Slack Socket Mode（`SocketModeHandler`）を採用しました。

現場の導入の壁（新しいツールの学習コスト）を下げるためです。

Slack Web API によるファイルの双方向受け渡し、`LexReviewer` の Slack Block Kit、長文分割投稿ロジックなどで、チャット導線にエージェントを載せています。

Socket Mode のため、外部からの HTTP Ingress（公開ポート）を開ける必要がありません。ただし Orchestrator が専門 Bot へ処理を引き継ぐ（Handoff）ために、各 Bot は `aiohttp` による内部 `/handoff` エンドポイントを保持しています。

### MCP レイヤ

MCP（Model Context Protocol）サーバは `fastmcp` で実装し、Transport は Streamable HTTP です。LLM に自由なシステム操作を許さず、明示的に定義したツールだけを与える境界として機能します。

#### 1. LexResearcher MCP

主要な依存は、ベクトル検索用の `sqlite-vec` / `openai` / `numpy`、スクレイピング用の `requests` / `beautifulsoup4` / `lxml` / `playwright`、Blob 連携の `azure-identity` / `azure-storage-blob` です。  
公開されている 17 ツールが、法務リサーチの複数経路をカバーします。

* **NDL / J-STAGE**: `search_ndl`（国会図書館） / `get_diet_record`（国会議事録） / `get_imperial_diet_record`（帝国議会議事録） / `search_crd_reference`（レファレンス） / `search_jstage_articles` / `get_jstage_abstract`
* **官公庁クロール・e-Gov**: `list_regulatory_sites` / `fetch_regulatory_latest` / `diff_regulatory_crawl` / `egov_search_laws` / `egov_fetch_article` / `egov_keyword_search`
* **判例・パブコメ・RAF**: `search_caselaw` / `search_fsa_pubcom` / `search_egov_pubcom` / `get_egov_pubcom_detail` / `get_raf`  
  *(※ `websearch` と `deep_research` は、ハルシネーションと情報の空振りを徹底的に防ぐため、コード上から意図的に登録除外しています)*

#### 2. LexReviewer MCP

ひな形・ガイドライン・RAF の安全な取得と、変更履歴付きWordの生成を担当します。主要依存は `fastmcp` / `httpx` に加え、Blob 正本アクセスの `azure-identity` / `azure-storage-blob`、Word調製の `python-docx` / `lxml` です。

* **公開 8 ツール**: `list_templates` / `get_template` / `register_template` / `list_guidelines` / `get_guideline` / `register_guideline` / `generate_counter_draft` / `get_raf`

#### 3. LexOrchestrator MCP

Orchestrator Manager から呼び出される sub-agent の自律的ハンドリングを担います。主要依存は `fastmcp` / `httpx` / `anthropic` / `azure-identity` です。

* **非同期制御 9 ツール**: `spawn_researcher` / `check_research_status` / `fetch_research_result` / `spawn_reviewer` / `check_reviewer_status` / `fetch_reviewer_result` / `spawn_composer` / `check_composer_status` / `fetch_composer_result`
* **引継ぎ・終結 4 ツール**: `handoff_to_researcher` / `handoff_to_reviewer` / `handoff_to_composer` / `respond_to_user`  
  同期呼び出しのタイムアウト制限を破壊し、ジョブ ID を介した非同期状態ポーリングへと切り替えるための頭脳部です。

### Agent / LLM 実行基盤：自律ループを回すか、単発で呼ぶか

4 体とも同じ `claude-sonnet-4-6` を使いますが、**呼び出し方**が二系統に分かれます。これはモデルの選択ではなく、アーキテクチャの選択です。

#### Anthropic Managed Agents（自律ループ）

Orchestrator Manager と Researcher は、`anthropic` Python SDK 経由で Managed Agent session を駆動します。`ANTHROPIC_API_KEY` と固有のエージェント ID（`LEX_DEEP_RESEARCHER_AGENT_ID`, `LEX_ORCHESTRATOR_MANAGER_AGENT_ID`）を用い、MCP ツールセットをバインド。エージェント自身が多ツールの連鎖・反復・終了判断を行う、開いた自律ループです。Researcher の探索も、Orchestrator の spawn/check/fetch 統括も、この自律ループを必要とします。

#### Azure AI Foundry Direct Endpoint（単発・確定変換）

一方 Reviewer と Composer は、与えられた材料を 1 パスで構造化出力（JSON）に変換すればよく、自律ループを必要としません。そこで Foundry Direct Endpoint（`/openai/responses`）を `httpx` で呼び出します。`DefaultAzureCredential` で認証し、システム指示（デフォルト: `claude-sonnet-46-orchestrator`）を注入する構成です。

同じ Sonnet 4.6 でも、自律ループを回すか単発で呼ぶかで、振る舞いの確率性は大きく変わります。LLM の精度が上がった現在、このループの有無は「LLM か Python か（確率か決定か）」の対比に匹敵するインパクトを持ちます。境界の定まった仕事（Review / Compose）を単発側に、開いた仕事（Research / Orchestrate）を自律ループ側に置く——これは、本記事の確率 vs 決定の軸を、LLM 層の内部にまで延長した判断です。

### Slack Bot 層

* **Orchestrator Slack Bot**: 入力を受け、Managed Agent の統括ループへ回します。成果物の Markdown/JSON を Word へ変換・添付送信するための OS 依存として `pandoc` を内包します。
* **Researcher Slack Bot**: Anthropic のリアルタイムなセッションストリームを Slack の進捗ボードに流し込み、`/handoff` エンドポイントをホストします。
* **Reviewer Slack Bot**: アップロードされた `.docx` を `markitdown[docx]` で抽出し、Blob へアップロード。Foundry 経路でレビュー概要を生成し、Reviewer MCP の `generate_counter_draft` を連鎖させて成果物を Slack へ再添付します。
* **Composer Slack Bot**: Foundry Direct 経路でドラフトを生成し、レビューと連携した修正履歴を適用。意図判定器 `intent_classifier` を備え、PDF/Word 書式変換のための OS 依存として `pandoc` / `libreoffice` と日本語ロケール (`ja_JP.UTF-8`) を内包します。

### データ取得・検索

* **官公庁 crawler (55サイト)**: 日銀、金融庁、法務省、財務省、国税庁、消費者庁、取引所（JPX/JSCC）、米国規制当局（SEC/CFTC）等の 55 サイトを `requests` / `BeautifulSoup` および一部 Playwright でクローリング。
* **NDL / J-STAGE / e-Gov API**: 国立国会図書館、国会・帝国議会議事録、J-STAGE 論文情報、デジタル庁の e-Gov 法令 API を繋ぎ込み、法令本文や条文を一次情報から直接取得。
* **FSA パブコメ検索**: `text-embedding-3-small` による Azure OpenAI embeddings を用い、Azure Files mount（`/mnt/fsa-pubcom/index.db`）した `sqlite-vec` インデックスをセマンティック検索。検索インデックスはコンテナイメージに含めず、マウントした外部ストレージから読みます。
* **判例・命令・審決検索**: 裁判所判例、情報公開・個人情報保護答申、中央労働委員会命令、公取委審決などの 6 つの DB を横断検索。

### 文書処理：決定論的な Python で固める OpenXML 操作

Word（.docx）まわりの処理は、すべて決定論的な Python コードで固めています（「LLM に Word を直接触らせない」判断 5）。

* **抽出・生成**: `python-docx` / `markitdown[docx]` による可変項目・本文の読み書きと、JSON/Markdown からの正確な Word の調製。
* **修正履歴・コメント**: `lxml` ＋ `zipfile` を用い、Word の OpenXML 構造である `w:ins` / `w:del` 要素およびコメント `w:comment` を直接書き込み、相手方との交渉にそのまま乗るカウンタードラフトを生成します。Word の構造を組むのは決定論的なコードであり、LLM はこの XML に一切触れません。

### Storage / Secret / Identity

#### Azure Blob Storage

Blob を「正本（source of truth）」として扱い、2 種類を保持します。

* **規範層**: `RAF.md`、ひな形・ガイドライン・ドラフトの各レジストリ
* **成果物層**: アップロードされた契約書の一時保管と、カウンタードラフトの出力  
  認証は `DefaultAzureCredential` に統一し、Container Apps 上では Managed Identity（RBAC）、ローカルでは Azure CLI ログインを自動識別します。アクセスキーを持ち回らない構成です。

#### Secret / env

API キー（`ANTHROPIC_API_KEY`）、Slack トークン（`SLACK_*_BOT_TOKEN` 等）、Handoff 用トークン（`HANDOFF_AUTH_TOKEN`）などのシークレットは、Container Apps のランタイムシークレットから環境変数として供給します。コンテナイメージに `.env` ファイルを焼き込まない構成です。

### 技術的な境界：LLM は「知覚」、コードは「精密な運動」

本システムの設計思想を貫く、最も重要な境界線は以下の通りです。

1. **Slack Bot ＝ 流れる界面**: 利用者接点に特化し、重い処理は一切抱え込まない。
2. **MCP ＝ 制限された道具箱**: LLM が実行できるシステム操作を、明示的なツールだけに厳格に制限。
3. **文書処理 ＝ 決定論的コード**: 負荷の高い Word（.docx）処理は、LLM ではなく Bot や MCP のプログラムコード側で実行。
4. **長時間ジョブ ＝ 非同期の分離**: 時間のかかるリサーチやドラフト処理は、Orchestrator MCP の非同期ワークフロー（`spawn / check / fetch`）へ移送。
5. **認証 ＝ 透過的アイデンティティ**: Blob や Foundry へのアクセスは、キーを使わず Managed Identity（RBAC）で透過的に行う。

LLM の「確率的に揺れる判断」と、システム操作・文書処理という「壊れてはならない決定論的な処理」を、アーキテクチャの境界線で切り分ける。この分離が、Lex Suite を実務で安定して動かすための核です。

## 実戦で得られた工学的ブレイクスルー

E2E の実機検証を重ねる中で、単なるプロンプト調整ではない、現実の業務フローで動かすための設計上の山を 3 つ越えました。

前半の 2 つは、**確率的に揺れてよいもの（LLM の判断）と、決定論的に揺れてはいけないもの（基盤・文書）を、構造で切り分ける**という、本記事の設計思想そのものを工学レベルで体現するもの。最後の 1 つは、それらを実戦の E2E ループとして繋ぎ切るための、泥臭い対処です。

### 1. オーケストレーションの本質は「spawn / check / fetch」だった

当初、Orchestrator は素朴に「sub-agent を呼び、応答を待ち、返す」**同期 RPC** で実装していました。

しかしながら、これはことごとく壊れました。Foundry の Direct Endpoint（Responses API）には同期呼び出しで約 120 秒のサーバ側タイムアウトがあり、MCP のトランスポート層にも約 5 分の HTTP タイムアウトがある。長文契約書のレビューや数百件の一次情報を統合するリサーチは、平気でこの壁を超えます。

ここで多くの実装は「タイムアウトを延ばす」「リトライを足す」で済ませようとします。が、それでは直りません。**そもそも、長時間タスクの間、同期コネクションを開いたまま保持することができない**——これが本質でした。

そこで、全 sub-agent の呼び出しを次の 4 動作に分解しました。

```
spawn  : ジョブを起動し、即座に job_id を返す（待たない）
check  : job_id の状態を確認する（runningなら再度 check）
fetch  : completed になったら結果本体を取得する
respond: 統合した最終応答をユーザーに返す
```

Researcher だけでなく、Reviewer も Composer も **対称に** この形へ揃えました。すると、トランスポート層のタイムアウトという制約から構造的に解放され、処理時間の長短に依存しない頑健な調整が可能になります。

なお、Foundry 側に残る同期 120 秒の生成上限そのものは、Composer / Reviewer の呼び出しを Foundry の **background モード**（`background:true` でジョブを投入し、`response id` をポーリングして取得する非同期方式）へ切り替えて回避しています。

spawn / check / fetch が MCP 層の長時間保持を、background モードが Foundry 層の生成時間を——それぞれ別レイヤーのタイムアウトを別々に吸収する形です。

この形は **AWS Step Functions / Apache Airflow / Temporal / Kubernetes Jobs** といった、世界中の長時間ワークフロー基盤が一様に採用しているパターンと一致しています。

**オーケストレーションの本質は、突き詰めると spawn → check → fetch → respond の 4 動作に収斂する。** この事実を身をもって実感しました。

### 2. Word を「壊さない」ために、LLM に Word を触らせない

法務の世界では、契約交渉は **Word の修正履歴（変更履歴）** でやり取りします（今後どうなるかはともかく、少なくとも現状では。）。

レビュー結果は、Word の「校閲」タブで修正履歴——OOXML の `w:ins` / `w:del` 要素とコメント（`w:comment`）——を加えるのが実務です。

素朴な発想は「LLM に修正済みの .docx を吐かせる」です。

しかしながら、これはなかなかうまくいきません。

Word の XML 構造は非常に緻密であり、フォント指定、段落スタイル、リスト定義、そして変更履歴のメタデータなど、LLM が何十 KB もの XML をハルシネーションなしに一撃で吐き出させることはほぼ不可能です。

たとえ一度うまくいっても、中長期的に安定した実務運用をすることが難しくなります。

そこで、Lex Suite のとった選択は**LLM には Word（.docx）を絶対に触らせない**という境界線でした。

* **LLM の領分 ＝ 柔らかい知覚**: 「どこをどう直すべきか」というレビュー結果を、構造化 JSON（対象箇所、現在の文言、修正案、RAF条項、コメント）として出力することに徹する
* **コードの領分 ＝ 硬い運動**: Bot または MCP の Python プログラム側が、その JSON を決定論的にパースし、`python-docx` と `lxml` を使って元の Word ファイルの `w:ins`（挿入要素）や `w:del`（削除要素）、コメント（`w:comment`）の XML ノードへと「物理的に」変換・埋め込む

この切り分けにより、LLM がどんなにクリエイティブに発想しても、出力されるドキュメントの構造が壊れないことを構造的に担保しました。

### 3. Bot 自身のアップロードは、自分のイベントに返ってこない

Orchestrator が単発タスクを専門 Bot（Composer / Reviewer）へ直接引き継ぐ「ハンドオフ（Handoff）」では、Slack の製品仕様の罠を踏みました。

専門 Bot 側の `/handoff` エンドポイントが指示とファイルを受け取り、`files_upload_v2` でユーザーの DM に投稿する——ここまでは動きます。

ところが、**Slack は「Bot 自身がアップロードしたファイル」を、その Bot の `on_message` イベントへは配送しません**（無限ループを防ぐための仕様）。

結果、通常のイベント駆動ルーティングが発火せず、せっかく投稿したファイルが処理フローに乗らない、という事態に陥りました。

突破策は 2 つです。

第一に、イベントの到着を待たず **ルーティング処理（`_route_message`）を直接呼び出す**。

第二に、ファイル取得処理（`download_slack_file`）に **monkey-patch（モンキーパッチ）** を当て、アップロード時に手元にある Raw Bytes をメモリに退避しておき、後続の処理が同じファイルを読むときは Slack へ取りに行かず **キャッシュからオンメモリで返す** ようにバイパスしました。

製品仕様を言い訳にせず、欠けたイベント経路をコード側で塞いで自律ループを閉じる。この一手が、Lex Suite の E2E（End-to-End）を最後に接続した鍵となりました。

## これからの拡張 — 「動作」で分けたから、エコシステムを迷わず挿せる

Lex Suite を法律の「分野（金商法 / 会社法 / 知的財産…）」ではなく、人間の「動作（Research / Review / Compose）」で分けた設計判断（判断 2）は、単なる実装の都合を超え、将来の拡張性において決定的な意味を持ってきます。

法務向けの外部ツールやデータベース（有料の判例リサーチDB、契約書レビューSaaS、起案支援AI）は、その本質を解きほぐすと、すべて「動作」の性質に帰結します。

リサーチDBは「調べる（Research）」道具であり、レビューSaaSは「評価する（Review）」道具であり、起案支援は「書く（Compose）」道具です。

そして、これらのツールは今後、MCP（Model Context Protocol）コネクタという形で世界中から大量に供給されていくことになります。

現に、私たちが本システムを構築していた最中の **2026 年 5 月 12 日、Anthropic は「Claude for Legal」を発表し、20以上のリーガルMCPコネクタ（判例リサーチDB、契約ライフサイクル管理、e-discovery等）と、Microsoft Word・Outlook等との統合**を世界に向けて公開しました。

動作で分け、MCPを統合レイヤーに置き、最終成果物をWord（.docx）に着地させる——私たちが泥臭く実装を進めていたアーキテクチャの構造は、奇しくも業界の巨人が目指す方向に一致しているように思われます。

動作軸でエージェントを分割していると、こうして増えていく強力な外部コネクタの「接続先」が一意に決定されます。

* **有料判例リサーチ DB** ＝ **Researcher** に挿し込む
* **契約レビュー LegalTech** ＝ **Reviewer** に挿し込む
* **ドラフト・テンプレート支援ツール** ＝ **Composer** に挿し込む

もしも法律の分野軸（金商法 Bot / 会社法 Bot / 知財 Bot…）でエージェントを分けていたら、判例リサーチDBを一つ追加するだけで、すべての分野別Botに対して配線をやり直す必要があり、その接続数は「コネクタ数 × 分野数」で爆発していました。

動作軸であれば、**1コネクタ ＝ 1接続先**で接続が完結します。

有料リサーチDBや専門SaaSの機能的威力は本物です。Lex Suite はそれらと無謀に競合するのではなく、それら強力な「動作」の入口をいつでも迎え入れられる\*\*「正しいソケット」\*\*をあらかじめ用意しています。内製の官公庁クローラや e-Gov 法令 API との連携は、そのソケットが既に実戦で機能していることの証明に他なりません。

LLMのモデルやエージェント実行基盤、そして汎用コネクタ群はいずれコモディティ化していきます。しかし、最後に残るのは、**(1) 実務に即して動作で美しく分かれた接続構造**と、**(2) 組織固有の RAF という上位ガバナンス規範**です。これら二つだけは、どのような最新のコネクタをもってしても「外部から買ってきて挿す」ことができない、組織固有の強靭な資産です。

### 運用面の発展

このアーキテクチャの上で、実務での運用をさらに前進させるためのロードマップを敷いています。

* **定期ウォッチの自律化**: スケジューラ機能を用いて55の官公庁サイトとe-Govを常時巡回。Researcher が `RAF.md` に照らして「🔴（対応必須）」とトリアージした重要差分だけを Slack に Push通知し、人間をノイズの海から解放する。
* **RAF 更新フローの Slack ワークフロー化**: 現在は手動の Blob アップロードで行っている `RAF.md` の改訂を、Slack 上の承認プロセスと連動して自動デプロイする仕組みの構築。
* **司法水準の監査可能性の担保**: すべてのツール呼び出し履歴と、LLM がどの RAF 条項を参照して要対応判定を行ったかのプロセス（思考ログ）を Azure Cosmos DB に永続化。「誰が、いつ、何を根拠にその法的リスクを判断したか」を遡れる監査ログ層の確立。
* **組織横断的な横展開**: `RAF.md` を切り替えるだけで、業界別・会社別のリスク基準に対応。さらにはコンプライアンス、個人情報保護、知的財産管理など、法務以外のガバナンス部門へ適用範囲を拡張する。
* **多言語・クロスボーダー対応**: 英文契約書の審査や海外規制のウォッチに向けて、英語表記の RAF と英文契約ひな型を独立してホストする別コンテナのデプロイ。
* **人間レビュアーとの協働UI**: AIが生成した判定結果と修正案（Word）を、人間がワンクリックで検証・承認して相手方への送付フローへと流す、Slack Socket を起点としたシームレスなハンドオフワークフローの構築。

## おわりに

Lex Suite は、法務業務の動作（Research / Review / Compose）を 3 つの専門エージェントに割り当て、それらが人間の策定した `RAF.md` という上位規範を憲法として共通参照する構成のマルチエージェント AI システムです。

法務実務の「本質的な難しさ（グレーゾーンでの許容性判断）」と現場の制約に向き合いつつ、自然な形でAIエージェントの設計に落とし込み、泥臭く実装したものです。

奇しくも、本システムを構築していた最中の 2026 年 5 月 12 日、Anthropic が「Claude for Legal」（20+ の MCP コネクタ + Microsoft Office 統合）を発表しました。動作で分け、MCP を共通インターフェースとし、最終的に Word に着地させる——市場が向かう方向と、現場から積み上げた私たちの設計は、近い地点を向いていました。

重要なのは、**業務の本質に従って素直に切り分けた結果として、こうした最新サービスとの親和性まで自然に確保できていた**ことです。

外部機能を MCP ツールとして接続する構成のため、新しいコネクタの追加が容易であること。そして、エージェントを「分野」ではなく「動作」で分けているため、コネクタ数 × 分野数で接続が膨張する MCP 接続の N×M 問題を、ここでも回避できていることです。

ただし、自社が定めた `RAF.md` というガバナンス基準は、汎用コネクタとして外部から買うことができません。自分たちの手で作り、磨き続けるほかないものです。

これからも、この自律性と安定性を両立させた強靭なリーガルエージェントを、実務のリアルな変化とともに育てていきます。

人間は規範の策定とガバナンスに集中し、AIは安全な枠内で高速な実務を回す。  
人間とAIが融和する、法務の新しい未来へ。
