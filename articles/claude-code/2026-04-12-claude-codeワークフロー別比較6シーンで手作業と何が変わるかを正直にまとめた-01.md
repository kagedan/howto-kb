最近のQiitaでは「Claude Codeを入れてみた」系の入門記事は増えてきましたが、「どの作業でどれだけ変わるか」をシーンごとに比較した記事はあまり見かけません。入門記事の多くは「こんなことができる」で止まっていて、実際どのシーンに組み込めばいいかが見えにくい。

この記事では日常的な開発作業をシーンごとに切り出し、従来の手作業とClaude Codeを使った場合を比べます。「ここは効く」「ここはそうでもない」も含めて整理するので、自分の開発フローに組み込む判断材料になると思います。

## Claude Codeが実際にやること

まず前提として仕組みを一言で。Claude Codeは「エージェントループ」というサイクルで動いています。

1. コンテキスト収集（ファイルを読む・コマンド結果を確認する）
2. アクション実行（ファイル編集・コマンド実行）
3. 結果検証（テスト実行・エラー確認）

このサイクルを、あなたが承認しながら何度も回します。要は「ターミナルにいる同僚」です。

組み込みツールはカテゴリに分けると5種類あります。

| カテゴリ | できること |
| --- | --- |
| ファイル操作 | 読み込み・編集・新規作成・リネーム |
| 検索 | glob/regex でファイル・コンテンツを探す |
| 実行 | シェルコマンド・テスト・git |
| Web | ドキュメント参照・エラーメッセージ検索 |
| コードインテリジェンス | 型エラー確認・定義ジャンプ（要プラグイン） |

このツール群を使いながらループするので、単なる補完ツールとは動き方が根本的に違います。

## シーン別比較

### 1. コードベースを把握する

**手作業の場合**: README と package.json を読み、ファイルツリーをざっと眺め、動きそうなところから grep する。新規参画のプロジェクトだと数時間かかることもあります。

**Claude Codeの場合**:

```
cd /path/to/project && claude
> give me an overview of this codebase
> what are the key data models?
> how is authentication handled?
```

Claude がファイルを横断して読み、アーキテクチャを説明してくれます。「具体的には src/auth/ のここで処理しています」と場所まで特定してくれるのが助かります。

Claude Code は起動したディレクトリのファイルツリー・git ステータス・CLAUDE.md を自動でコンテキストに含めます。最初の質問から全体像を把握できる理由はここにあります。

**変わる点**: コードを読む順番を自分で決めなくていい。「この関数どこで呼ばれている?」という追跡作業が会話で完結します。

---

### 2. バグを修正する

**手作業の場合**: エラーを読む → 関連ファイルを特定 → コードを追う → 修正 → テスト実行。同じスタックトレースを何度も見返す手間があります。

**Claude Codeの場合**:

```
> I'm seeing an error when I run npm test
[エラーメッセージをペースト]
> suggest a few ways to fix the @ts-ignore in user.ts
> update user.ts to add the null check you suggested
```

手順は同じですが、ファイルを手で開く・コマンドを打つ・結果を解釈するという反復が半分以下になります。

**変わる点**: エラーを説明すると関連ファイルを自分で探してくれます。修正案を複数出してから適用するフローが自然に組めます。

---

### 3. リファクタリングする

**手作業の場合**: 対象コードを把握 → 修正方針を決める → 変更 → テスト → 壊れた箇所を直す。広範囲の変更は変更漏れが起きやすい。

**Claude Codeの場合**:

```
> find deprecated API usage in our codebase
> suggest how to refactor utils.js to use modern JavaScript features
> refactor utils.js to use ES2024 features while maintaining the same behavior
> run tests for the refactored code
```

テスト実行まで一連で指示できます。途中で「後方互換性を保ってほしい」と伝えれば調整してくれます。

特に便利なのが **Plan Mode** です。`Shift+Tab` で切り替えると、ファイルの読み取り専用操作だけを行い「計画」を出してくれます。コードは一切変えません。

```
# Plan Mode で起動
claude --permission-mode plan
> Analyze how we handle sessions and suggest a plan for adding OAuth support
```

計画を確認・修正してから実行に移れるので、大規模なリファクタリングの前に「何を変えるか」だけ合意したいときに重宝します。`Ctrl+G` で計画をエディタに開いて直接編集することもできます。

**変わる点**: 変更の影響範囲を把握してから手を動かすフローが自然に組めます。

---

### 4. テストを書く

**手作業の場合**: カバレッジを計測 → テスト対象を特定 → テストを書く → 通るまで直す。既存のテストスタイルに合わせる確認作業も地味に手間がかかります。

**Claude Codeの場合**:

```
> find functions in NotificationsService.swift that are not covered by tests
> add tests for the notification service
> add test cases for edge conditions in the notification service
> run the new tests and fix any failures
```

既存テストのスタイル（フレームワーク・アサーション形式）を自動で読み取って合わせてくれます。エッジケースの提案もしてくれるので、見落としが減ります。

**変わる点**: 「テストを書くのが面倒な関数」を後回しにしなくなります。エッジケースの洗い出しも会話で完結します。

---

### 5. PR を作成する

**手作業の場合**: 変更を整理 → コミットメッセージを書く → PR を作る → 説明文を書く。説明文を何を書くか迷って時間を使いがちです。

**Claude Codeの場合**:

これだけで、差分を読んで適切なタイトル・説明文を生成してから `gh pr create` を実行してくれます。追加指示も可能です。

```
> enhance the PR description with more context about the security improvements
```

PR 作成時のセッションは自動的にその PR に紐づくので、後から `claude --from-pr <number>` で再開できます。

**変わる点**: PR 説明文を書く時間がほぼゼロになります。特にリリースが多い時期に効果が出ます。

---

### 6. 複数タスクを並列で進める

ここが Claude Code のかなり独自の強みです。他のツールではなかなかできません。

**手作業の場合**: ブランチを切り替えながら作業する。スタッシュや切り替えのたびに文脈を思い出す手間があり、実質的な並列進行は難しい。

**Claude Code + Git worktrees の場合**:

```
# 別ディレクトリでそれぞれ Claude を起動する
claude --worktree feature-auth   # feature-auth ブランチで作業
claude --worktree bugfix-123     # bugfix-123 ブランチで作業
```

Git worktrees によって、別々のディレクトリに別々のブランチが展開されます。それぞれで Claude が独立したセッションとして動くので、一方の作業が他方に干渉しません。

worktree で作業を終えて exit すると、変更がなければ自動でクリーンアップされます。変更がある場合は「残す/削除する」を確認されます。

`.gitignore` に含まれる `.env` などのファイルは worktree にコピーされません。`.worktreeinclude` ファイルを使えば指定したファイルを自動コピーできます。

```
.env
.env.local
config/secrets.json
```

リポジトリルートに置くだけで OK です。

サブエージェントに worktree を使わせることもできます。カスタムサブエージェントの frontmatter に `isolation: worktree` を書いておくと、各エージェントが別 worktree で独立して作業します。

```
> フロントエンドのテストとバックエンドの API 修正を並列でやってほしい
```

**変わる点**: ブランチ切り替えによる文脈喪失がなくなる。サブエージェントを使えばタスク自体を並列化できます。

## 6シーン比較まとめ

| シーン | 手作業からの変化 | 特に効くポイント |
| --- | --- | --- |
| コードベース把握 | 大 | 全体像を会話で把握できる |
| バグ修正 | 中 | 関連ファイルの追跡が速い |
| リファクタリング | 大 | Plan Mode で事前確認できる |
| テスト作成 | 中 | 既存スタイルへの追従が自動 |
| PR 作成 | 大 | 説明文の生成が特に楽 |
| 並列作業 | 大（独自） | worktrees + 並列セッションで干渉なし |

バグ修正やテスト作成が「中」なのは、Claude が正しい修正を出すかの判断はまだ人間が必要だからです。「大」のシーンは作業そのものの構造が変わります。

## まとめ

Claude Code は個々の作業を速くするというより、「一連の作業の繋ぎ目を埋める」使い方で真価が出ます。ファイルを探して読んでコマンドを打って結果を確認する、その往復が減るのが体感として大きいです。並列作業の worktrees は他のツールにはない独自の使い勝手なので、まだ試したことがなければ一度触ってみてください。
