## はじめに

こんにちは、クラシルでクラシルリテールネットワークの開発をしている akawai11 です。

ALBのアクセスログを定期的に分析する必要があるのですが、これが毎回結構な手間になっていました。

ALBのアクセスログ機能を有効化し、S3に保存しています。  
実作業としてはS3からログをダウンロードして、解凍して、集計する。  
やること自体は毎回ほぼ同じなのに、都度CLIでコマンドを組み立てていました。期間指定を変えてダウンロードして、パスのフィルタを調整して、集計軸を変えたりといった作業が定常化していたんです。

今回、この分析作業をClaude Codeのスキルとしてまとめました。  
スキルにしておけば、チーム内で分析を共有しやすくなります。  
S3のパス構造等を知らなくても使えるので、「特定の人しかログを見られない」「ログ調査をやりたいけど何から手をつければいいかわからない」という状態を解消したかったのが動機です。

## Claude Codeスキルにしようと思った理由

### 集計結果をレポートにまとめたい

自分が欲しかったのは集計結果そのものではなく、**集計結果をまとめたレポート**でした。

たとえば「5xxエラーが14時台に集中していて、特定のパスに偏っている」という情報は、コマンドでも出せます。

でも複数の集計結果を突き合わせて「14時台にXXで4xxが急増、レスポンスタイムも同時間帯にp99がN秒超え」とまとめるのは、大変です。

Claude Codeスキルなら、コマンド実行で一次データを集めた上で、レポートとして整理してくれます。

### 追加指示で深掘りできる

スクリプトの場合は決まった処理しかできませんが、スキルなら「今回は5xxだけ深掘りして」「このパスに絞って調べて」のような追加指示にもその場で対応できます。  
その場で追加指示を出せるのがClaude Codeスキルの使い勝手のよいところです。

### 使うたびに改善される

スキルにはStep 6として「振り返りと改善提案」を組み込んでいます。  
実行のたびにスキル自体の改善点を洗い出して、SKILL.mdにフィードバックしていく仕組みです。  
スクリプトは書いた人が手動でメンテする必要がありますが、スキルなら運用しながら改善のヒントが蓄積されていきます。

## スキルの全体像

SKILL.mdに定義した処理フローは6ステップで構成しています。

### Step 1〜3: ログの取得

```
1. 日時入力
2. 作業ディレクトリ作成
3. S3からログダウンロード（入力された時間単位でフィルタリング）
```

ALBログのS3キーは時間単位のprefixを含むため、`aws s3 cp`で対象時間帯のファイルだけを取得しています。  
1日分のログを全量ダウンロードすると数GBになることがあるため、必要な時間帯に絞ることで処理時間を短縮しています。

### Step 4: bashコマンドで分析

分析はすべてbashコマンドで実行します。Claude Codeに分析処理を任せるのではなく、再現性のあるコマンドをSKILL.mdに定義しておく設計にしました。

5つの切り口で「いつ・どこで・どれだけ」が一度にわかるようにしています。

| 分析項目 | 出力ファイル | 内容 |
| --- | --- | --- |
| ステータスコード分布 | `status_category.txt` | ステータスコード別集計 |
| パス別エラー集計 | `path_errors.txt` | 400以上のエラーをパス別に集約 |
| 時系列推移 | `timeseries_requests.txt` | 1時間単位のリクエスト数・エラー数 |
| レスポンスタイム分析 | `response_times.txt` | p50/p90/p95/p99/maxの統計 |
| ステータスコード別パス内訳 | （動的） | エラーコードごとの詳細分析 |

パス中のIDやUUIDは`:id`、`:uuid`に正規化してグルーピングしています。  
たとえば`/hoge/12345`と`/hoge/67890`は同じ`/hoge/:id`として集計されます。

### Step 5: Markdownレポート生成

ここがClaude Codeスキルにした最大の理由です。  
Step 4で出力した各ファイルの集計結果を、Claude Codeが1つのMarkdownレポートにまとめます。

レポートには概要、ステータスコード別分析、時系列推移、レスポンスタイム分析、注目ポイントが含まれます。  
エラーが集中しているパスや、レスポンスタイムの異常値があれば、考察とアクション候補を提示してくれます。

このレポートをベースにチームに共有しやすくなりました。後日確認する際も再度ログをダウンロードする必要がなく、レポートを見るだけで状況を把握できます。

さらに、レポートの内容をもとに「このパスについてもう少し掘り下げて」といった追加指示もできるので、レポートが次の分析の起点にもなります。

### Step 6: 振り返りと改善提案

毎回の実行後に、スキル自体の振り返りを行うステップを設けています。  
コマンドのエラー、データの品質、分析の網羅性などの観点で振り返り、結果をファイルに保存します。

ユーザーが内容を確認し、適用するかを意思決定することで、オシレーション（修正を繰り返すうちに定義が揺れて品質が下がる現象）を防ぎつつ、必要な改善だけを取り入れる運用にしています。  
このステップがあることで、改善のヒントを蓄積しやすくなります。

## 使ってみてどうだったか

結論から言うと、ログ分析にかかる時間は大幅に減りました。

以前はコマンドを組み立てて実行して、結果を見て、別の切り口で再集計して——というサイクルを30分〜1時間かけていました。

スキル化してからは、期間を指定して実行するだけで、レポートまで出てきます。  
その間に別の作業を進めることができます。

**良かった点:**

* **コマンドの組み立てが不要**。コマンドを組み立てる手間も、都度正しさを確認するコストもなくなりました
* **レポート形式で出力**。チームへの共有がしやすくなり、過去の分析結果を振り返る際もレポートを見るだけで済みます
* **振り返りの改善ヒントが蓄積される**。実行するたびにスキル定義の改善点が洗い出されていきます

**改善点:**

* **振り返りの改善サイクルが手動**。Step 6で改善提案は出てきますが、それを読んでSKILL.mdに反映するかどうかはユーザーの判断です。自動適用するとスキル定義が意図せず書き換わるリスクがあるため、意図的に確認ステップを残していますが、それにより改善サイクルが止まってしまうことがあります。
* **レポートの深堀りは人の判断が必要**。レポートは出力されますが、「このエラーは何が原因か」「対応が必要か」の判断は人がやる必要があります。注目ポイントは提示してくれますが、次のアクションまでは自動化できていません。

## まとめ

ALBログ分析をClaude Codeスキルにしたことで、定常的なログ調査の手間が減りました。  
スクリプトとの違いは、集計結果をレポートとしてまとめてくれる点と、使うたびにスキル自体を改善していく仕組みを取り入れやすい点にあります。

まだ作って間もないので、定常的に運用しながら改善していくフェーズです。  
振り返りの改善サイクルは、オシレーションを防ぎつつ自動化できる仕組みを模索していきたいです。  
また、レポートから次のアクションまでつなげられるようになると、分析の価値がもう一段上がると感じています。
