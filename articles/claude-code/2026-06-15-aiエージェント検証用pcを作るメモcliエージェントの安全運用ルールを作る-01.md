## はじめに

前回は、Codex CLIをUbuntu側へ導入し、`~/agent-sandbox` 内の `README.md` に1行だけ追記させた。

変更後は、`git status` と `git diff` で内容を確認し、自分でコミットした。

今回は、Codex CLIや今後導入予定のClaude Codeを安全に使うため、AIエージェント向けの運用ルールを作る。

今回のゴールは以下。

* 共通の安全方針を `SECURITY_POLICY.md` にまとめる
* Codex向けに `AGENTS.md` を作る
* Claude Code向けに `CLAUDE.md` を作る
* 秘密情報をGitへ登録しないための `.gitignore` を作る
* Codexにファイルを作成・修正させる
* `git diff --cached` で人間が最終確認する
* 問題がないことを確認してからコミットする

今回も、本物のAPIキーや秘密情報は扱わない。

---

## 前提

環境は以下。

* OS: Windows 11
* WSL環境: Ubuntu
* 作業場所: `~/agent-sandbox`
* Git: 導入・初期コミット済み
* Codex CLI: 導入・動作確認済み
* Claude Code: 未導入
* 本物のAPIキーや秘密情報: 未配置

基本方針は、これまでと同じ。

今回作るルールは、完全な防壁ではない。

AI向けの指示ファイル、Gitによる差分確認、秘密情報を持ち込まない運用を組み合わせて、安全性を高める。

```

役割はそれぞれ異なる。

```

今後、Claude Codeなど別のCLIエージェントも、同じ安全方針のもとで試す予定。

## なぜ安全ルールをファイルにするのか

AIエージェントは、自然文で指示を出すだけでファイルを読み、編集し、コマンドを実行できる。

便利な反面、その都度口頭で注意事項を伝えるだけでは、指示が抜けたり、作業ごとに方針が変わったりする可能性がある。

そこで、安全方針をファイルとして残すことにした。

```
共通方針
→ SECURITY_POLICY.md

Codex向けの要約
→ AGENTS.md

Claude Code向けの要約
→ CLAUDE.md
```

共通の基準文書を一つにし、各エージェント向けファイルは短い運用要約として扱う。

ルール同士が食い違った場合は、`SECURITY_POLICY.md` を優先する。

---

## 最初は手入力しようとした

当初は、`SECURITY_POLICY.md` をUbuntu上で手入力しようとした。

しかし、長い英語のルール文を別画面で見ながら入力したため、以下のようなミスが発生した。

* `SECURITY_POLICY.md` のファイル名を間違えた
* `Do not` を `Do` と入力した
* `.env` の途中に空白が入った
* Markdownの箇条書きや改行が崩れた
* 英単語のスペルを間違えた
* ルールを入れる見出しの位置が混ざった

安全ルールを作っている最中に、入力ミスによって意味が逆になる可能性があることに気づいた。

たとえば、以下は意味が大きく異なる。

```
Do not write secrets ...
→ 秘密情報を書かない

Do write secrets ...
→ 秘密情報を書く
```

安全ルールほど、一文字の違いが重大になる。

そのため、長い文章を自分で手入力する方針をやめ、Codexに作成・修正させたうえで、人間がGit差分を確認する方法へ切り替えた。

## Codexに安全ルールを作成させる

Codexには、既存の `SECURITY_POLICY.md` を基準にしながら、以下のファイルを作成・修正するよう依頼した。

* `.gitignore`
* `AGENTS.md`
* `CLAUDE.md`

主な条件は以下。

```
APIキーや秘密情報は書かない
.env は作らない
作業対象をリポジトリ内に限定する
破壊的操作は明示的な承認なしに行わない
変更は小さく、レビューしやすくする
コミットは行わない
```

Codexには、作成だけでなく、スペルミス、Markdownの崩れ、ルール間の矛盾もレビューさせた。

ただし、Codex自身の完了報告だけでは確定しない。

最後は自分でGit差分を確認する。

## SECURITY\_POLICY.md

`SECURITY_POLICY.md` は、CLIエージェント共通の基準文書。

最終的な内容は以下。

```
# SECURITY_POLICY.md

## Purpose

This repository is a small sandbox for testing AI agent behavior.

## Core Rules

- Work only inside this repository.
- Do not read, modify, or use files outside this repository.
- Do not ask for, store, print, or expose API keys, passwords, tokens, cookies, credentials, secrets, or personal information.
- Do not create, edit, print, or expose `.env` files.
- Do not perform destructive actions without explicit user approval.
- Keep changes small and easy to review.
- Before making changes, explain the plan briefly.
- After making changes, summarize what changed.

## Git Workflow

- The user checks changes with `git status` and `git diff`.
- Do not commit changes unless the user explicitly asks.
- If a change looks risky, stop and ask the user before continuing.

## Secrets Policy

- Do not use `.env` files in this repository.
- Do not use real credentials or secrets in this repository.
- Do not store real secrets in this repository.
- Do not write secrets into README files, logs, examples, or source files.
- Docker secrets may be considered later when Docker Compose experiments begin.
- Until then, this sandbox should contain no real secrets.
```

このファイルでは、以下を明示した。

* リポジトリ外へアクセスしない
* 秘密情報を要求・保存・表示・露出しない
* `.env` を作成・利用しない
* 明示的な承認なしに破壊的操作を行わない
* 勝手にコミットしない
* 本物の秘密情報をsandboxへ持ち込まない

## .gitignore

`.gitignore` には、秘密情報が入りやすいファイルや、Git管理する必要がないディレクトリを追加した。

```
.env
.env.*
secrets/
*.key
*.pem
*.token
node_modules/
.envrc
.npmrc
.pypirc
.netrc
id_rsa
id_rsa.*
id_ed25519
id_ed25519.*
```

`.gitignore` は、秘密情報を安全に保管する仕組みそのものではない。

Gitへの誤登録を防ぐための保険。

そのため、以下の方針は別途必要になる。

```
秘密情報をsandboxへ持ち込まない
APIキーをコードへ直書きしない
本物の認証情報をテストファイルへ書かない
Gitへ登録されていないから安全だと思い込まない
```

---

## 未追跡ファイルは通常のgit diffに出ない

今回、4ファイルは新規の未追跡ファイルだった。

そのため、通常の以下のコマンドでは差分が表示されなかった。

そこで、一度ステージングした。

```
git add SECURITY_POLICY.md .gitignore AGENTS.md CLAUDE.md
```

ステージングはコミットではない。

その状態で、以下を実行する。

これにより、新規ファイルの内容をGit差分として確認できた。

ファイルごとに確認する場合は、以下のように指定できる。

```
git diff --cached -- .gitignore
git diff --cached -- SECURITY_POLICY.md
git diff --cached -- AGENTS.md
git diff --cached -- CLAUDE.md
```

画面下に `END` が表示された場合は、`q` を押すと終了できる。

---

## Codexにもレビューさせた

Codexへ、作成した4ファイルをレビューするよう依頼した。

確認項目は以下。

* スペルミスやMarkdownの崩れ
* ルール同士の矛盾や重複
* リポジトリ外へのアクセスを許す表現がないか
* 秘密情報の扱いに抜けがないか
* 破壊的操作を無断で許していないか
* `AGENTS.md` と `CLAUDE.md` が `SECURITY_POLICY.md` に従っているか

レビューでは、以下の改善案が出た。

```
ask for だけでなく store / print / expose も禁止する
user approval を explicit user approval にする
SECURITY_POLICY.md を最優先にする
.gitignore に認証系ファイルを追加する
```

これらを反映したあと、再度Git差分を確認した。

AIに作らせ、AIにレビューさせたが、最終判断は自分が行った。

---

## コミット前の最終確認

最終版を再びステージングする。

```
git add SECURITY_POLICY.md .gitignore AGENTS.md CLAUDE.md
```

空白や差分上の問題を確認する。

```
git diff --cached --check
```

問題がなければ、何も表示されない。

その後、コミットした。

```
git commit -m "Add AI agent safety rules"
```

履歴を確認する。

今回は、以下の3つのコミットが確認できた。

```
889da35 Add AI agent safety rules
8aa4577 Add Hello AI Agent
c6c9f9b Initial commit
```

これで、安全運用ルールをGitへ記録できた。

## 今回ハマった点

### 長い安全ルールを手入力しない

今回一番大きな学びだった。

長い英語のMarkdownを手入力すると、スペル、記号、改行、否定語を間違える可能性がある。

特に `Do not` のような否定表現は、抜けると意味が逆になる。

今後は以下の流れにする。

```
AIに作成させる
↓
別AIや同じAIにレビューさせる
↓
Git差分を人間が確認する
↓
問題なければコミットする
```

### Codexの完了報告だけで確定しない

Codexが「完了しました」と返しても、それだけでコミットしない。

必ず以下を見る。

```
git status
git diff
git diff --cached
```

### 未追跡ファイルはgit diffに出ない

新規ファイルは、通常の `git diff` だけでは中身が表示されない場合がある。

ステージングして `git diff --cached` で確認した。

### .gitignoreは秘密管理機能ではない

`.gitignore` に入っていても、ファイルはPC上に存在する。

AIエージェントから読める可能性がなくなるわけでもない。

秘密情報は、最初からsandboxへ置かない。

## まとめ

この記事では、Codex CLIや今後導入予定のClaude Codeを安全に試すため、以下を作成した。

```
SECURITY_POLICY.md
AGENTS.md
CLAUDE.md
.gitignore
```

今回のポイントは以下。

最初は安全ルールを手入力しようとして、ファイル名や英語、Markdownを何度も間違えた。

その経験から、長い文章を人間が手入力するよりも、AIに作成・修正させ、Gitで確認する方が安全な場合があると分かった。

AIエージェントを安全に使うためのルールを、AIエージェント自身に作らせる。

ただし、最後に確定するのは人間。

この流れを、今後のCLIエージェント運用の基本にする。
