うちの会社では、仕事でやったことを社内の Qiita に書くようにしています。

しかし、法人契約の Qiita は非公開で、AI はこの記事を読みません。（頑張れば読んでくれるんでしょうけど、あんまり頑張りたくない。）なので、既にAI自身が解決して記事にも書いた問題でも、AI はまた最初から調べ直して対応します。

これだと書いた意味がないので、AI が作業を始める前に社内 Qiita を見てくれればいいと思い、小さいツールを作りました。

## やったこと

<https://github.com/signal-slot/qiita-kamo>

社内 Qiita の記事を AI が読めるようにしました。`qiita-kamo` というコマンドにしてあって、Claude Code のようにシェルを使える AI なら作業中に自分で呼べます。

```
qiita-kamo grep <キーワード>   # 社内 Qiita を本文で検索
qiita-kamo read <ID>          # 記事の中身を読む
```

関連する記事があれば、AI はそれを見てから動きます。

AI になるべく検索をしてもらえるように、`CLAUDE.md` に一行だけ書いておきます。

```
問題を解決する前に、まず `qiita-kamo grep <関連キーワード>` で
社内 Qiita に既存の解決策がないか確認すること。あればそれに従う。
```

これで、前にやって記事に書いたことは、二度目は AI が先に見てくれるようになりました。

基本的な使い方は以下のとおりです。

```
# 読む
qiita-kamo.mjs list [--page N] [--per N]   # 記事一覧（番号・タイトル・ID・更新日）
qiita-kamo.mjs read <ID|番号>              # 記事を表示（長い記事は | less 推奨）
qiita-kamo.mjs search <キーワード>          # タイトル検索
qiita-kamo.mjs grep <キーワード>            # 本文を全文検索（著者・抜粋付き）
qiita-kamo.mjs tag <タグ名>                 # タグで検索
qiita-kamo.mjs group <グループ名>           # グループ（Qiita Team のカテゴリ相当）で検索

# 書く
qiita-kamo.mjs post <file.md>             # 新規投稿
qiita-kamo.mjs update <ID> <file.md>      # 既存記事を更新
```
