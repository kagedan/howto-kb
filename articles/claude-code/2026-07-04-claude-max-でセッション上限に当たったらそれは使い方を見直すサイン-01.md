## はじめに

Claude Code を Max プランでガシガシ使っている人に向けた記事です。

僕は Max に入ってからほぼ毎日 Claude Code を回していますが、**一度もセッション上限（5時間）に当たったことがありませんでした**。それが昨日、初めて上限に達して使えなくなりました。

「課金してるのに使えないの？」と一瞬イラッとしたんですが、調べてみると原因は完全に**自分の使い方**でした。同じように「最近やたら上限に当たる」「ウィークリーの枠がすぐ溶ける」と感じている人は、たぶん同じ落とし穴にハマっています。

この記事は、僕が**何を調べて・何が原因で・どう対策したか**の実体験ログです。

## 何が起きたか

いつも通り作業していたら、突然これ。

> You've reached your usage limit. Resets at ...

セッション（5時間枠）もウィークリーも、今までは余裕で収まっていた。コードの規模が急に増えたわけでもない。**使い方は「いつも通り」のつもりだった**のに上限に当たった。ここが引っかかりました。「いつも通り」のはずなのに結果が変わった＝どこかで消費構造が変わっている。

## 原因調査：使用統計が全部教えてくれた

Claude Code には、自分の使用量の内訳を出してくれる画面があります（`/usage` 系の表示）。直近 24h の内訳を見たら、犯人がはっきり出ていました。

* **94%** が「subagent を多用したセッション」由来
* **94%** が「8時間以上アクティブなセッション」由来
* **86%** が「context 150k 超の状態」での消費
* `deep-research` という subagent 群だけで **11%**
* ブラウザ操作の MCP（Playwright）が **5%**

数字を見て自分の仮説が半分外れていたことに気づきました。最初は「**subagent が最近デフォルト ON になったのが原因では？**」と疑っていたんですが、そうではなかった。**subagent が勝手に起動していたのではなく、自分が subagent を大量に立てる使い方をしていた**んです。

### 消費が跳ねる「三重がけ」

整理すると、原因は単独ではなく掛け算でした。

| 要因 | 何が起きていたか | なぜ重い |
| --- | --- | --- |
| ① 高 effort 設定 | `effortLevel: high` をグローバル設定にしていた | main も**全 subagent も**毎ターンの思考量が底上げされ、出力トークンが一律に増える |
| ② subagent / workflow の多用 | deep-research や並列ワークフローを気軽に回していた | 子エージェントは**それぞれ独立のリクエスト列**。立てた数だけ消費が線形〜掛け算で増える |
| ③ 長寿命・大 context | `/clear` せず 8 時間級のセッションを continue し続けた | キャッシュが効いても、毎ターンの入力は context 量に比例して課金される。長いほど線形に高い |

特に効いていたのが **①の `effortLevel: high`** でした。これは使用統計の内訳には項目として出てこない「土台のかさ上げ」で、**全セッション・全 subagent に一律で乗る**。気づきにくいのに一番広く効いている、いやらしいやつでした。

そこに②（subagent 乱発）と③（長セッション）が重なって、「いつも通りのつもりが消費だけ倍増」が完成していた、というのが結論です。

## やった対策

派手なことはしていません。**一番広く効く1行**＋**運用習慣のルール化**です。

### 1. effort をグローバルで `medium` に下げた（最優先・即効）

`~/.claude/settings.json`:

```
- "effortLevel": "high",
+ "effortLevel": "medium",
```

これだけで全セッション・全 subagent の毎ターンの思考量が下がります。**難しいタスクのときだけ** `/effort high` で都度上げる運用に切り替えました。常時 high にしておく必要は、振り返ってみると全然なかった。

### 2. subagent / workflow を「乱発しない」と決めた

* 既定は「ツールを持った**単一エージェント**」。
* deep-research や並列ワークフローは、**価値がコストを明確に上回るときだけ**使う。
* 単純な subagent には**安いモデルを明示**（`model: "sonnet"` / `"haiku"`）。Opus 一律をやめる。

これは設定というより**判断のルール**なので、自分のグローバル設定ファイル（全プロジェクトで読まれる `CLAUDE.md`）に明文化しました。94% の塊がここなので、ここに効かせるのが一番大きい。

### 3. セッションを長生きさせない

* タスクが切り替わったら `/clear`。
* 続行でも context が肥大したら `/compact`。
* 8 時間級の continue セッションを作らない。

86% が「150k 超」由来だったので、ここは効きます。

## さらにやるなら（今回はやらない）

もっと踏み込むなら、**設定で自動強制する**手があります。たとえば、

* `permissions` の `ask` に `Task` / `Workflow` を入れて、subagent / workflow が立つたびに確認プロンプトを挟む（＝承認制にする）
* PreToolUse の hook で subagent 起動を検知して警告／ブロックする

hook を使った強制系の話は、前の連載[「AI に『やるな』と書いても無駄だった — 機械で止める 3 層ガードレール」](https://zenn.dev/yuhtaihara/articles/ai-guardrails-machine-level)でも書きました（プロンプト送信フックや `exit 2` でのブロックなど）。でも今回はそこまではやりません。理由は、

* そもそも `effortLevel: medium` だけでかなり効くはずで、まずはその効果を見たい
* 承認ゲートは「毎回 y/n を聞かれる」摩擦が大きく、deep-research を1回回すだけで内部で何度も止まる
* Claude Code には「session 長/context 量のハード上限」を設定する仕組みが**そもそも無い**ので、hook を足しても近似の警告止まり

なので今回は **effortLevel を下げて様子見**、というスタンスです。強制で縛るより、まず一番効く1行を変えて、足りなければ次の手、の順で十分。

## まとめ

* Max でセッション上限に当たったら、プランの問題というより**使い方の構造が変わったサイン**と捉えるのが正解だった。
* 自分のケースの主因は「**高 effort 設定 × subagent 乱発 × 長寿命大 context**」の三重がけ。
* 一番 ROI が高い対策は **`effortLevel` を high → medium に下げる1行**。次点が subagent/workflow を絞ること、長セッションを切ること。
* いきなり hook で縛らず、**まず一番効く設定を変えて様子を見る**。

同じく「最近やたら上限に当たる」人は、まず `/usage` の内訳を見て、`effortLevel` が `high` になっていないか確認してみてください。たぶんそこが効いています。
