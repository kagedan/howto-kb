## Claude Codeをどう使い倒しているか（本題）

ここからが本題。Claude Codeは「コードを書く道具」じゃなく、**「副業のオペレーション全部を回す相棒」**として使っています。

### 役割分担：ぼく30%・Claude 70%（あなたの業務にどう当てはめるか）

ぼくの中で、Claude Codeとの分業ルールは1行で言えます。

> **ぼくは0→1の判断と最終OKだけ。1→100の手数はClaudeに全部やらせる。**

具体的には：

役割 担当 比率 企画・テーマ決定・最終判断・公開 ぼく 30% 下書き・実装・リサーチ・レポート・ドラフト Claude 70%

![](https://assets.st-note.com/img/1778113764-2Hk3oiEvdXh7S0yWIOZmY54c.png?width=1200)

図2｜ぼく30%・Claude 70%の役割分担マトリクス

これを読みながら、**「自分の日常業務でルーティン化しているけど、頭はそんなに使っていないタスク」**を頭の中で1つ思い浮かべてみてください。会議の議事録まとめ、定例レポートの整形、メール返信の下書き、月次の数字集計——**それこそがClaudeに任せるべき70%です。**

ぼくの場合は「Note記事の構成案出し」「アプリの定型コード」「Notionへの起票」「収支の集計スクリプト」が、その70%でした。これに割り切れたから、1ヶ月で記事9本・アプリ3本まで回せました。

### ① 記事量産を可能にする：コンテンツ生成の3段フロー

Note記事は全部このフローで作っています。

```
Step 1: Claude が体験ベースで初稿を生成（v1.md）
Step 2: Gemini API が客観レビュー（読みやすさ・SEO・正確性・第三者視点）
Step 3: Claude がレビューを反映して修正版生成（v2.md）
Step 4: SVG図解 → PNG変換
Step 5: アイキャッチ生成
Step 6: HTMLでプレビュー → ブラウザで確認
```

ポイントは**Claude → Gemini → Claude のクロスレビュー**。なぜGeminiを挟むかというと、**Claudeはぼくの体験を全部知っているから「身内目線」になりがち**だからです。Geminiに第三者の目でぶった切らせて、Claudeに直させる。これで「身内ノリ」がだいぶ抜けます。

ハマったポイント：最初のころ、GeminiのレビューでmaxOutputTokensが2048で出力が10行で切れて「Geminiやる気ないな…」と諦めかけたことがありました。8192に上げたら3,700字のガッツリレビューが返ってきて反省。**LLMのデフォ値は疑え**、これ大事です。

ちなみにこの記事も、いま読んでくださっているv2はGeminiの辛口レビュー（★★★必須2件・★★☆推奨1件・★☆☆任意1件）を全部反映した修正版です。

ぼくのやることは「こういうアプリ作りたい」「この機能を追加して」「ここバグってる」を投げるだけ。**コードレビューと挙動確認はぼくがやる**けど、書くのはClaude。

### ③ 育休パパの時間捻出術：自動化スクリプトを「副業の裏方」に常駐させる

ここが一番効いてます。1ヶ月で3本の自動化スクリプトをClaude Codeに作らせました。

スクリプト 役割 note\_publisher.py Markdown→Note入稿用にSVG→PNG変換、HTMLプレビュー、タイトル・本文のクリップボードコピー fetch\_expenses.ps1 Claude/Gemini/AWS等の利用額を自動取得して家計簿JSに追記 gemini\_review.py 記事ドラフトをGemini APIに渡してレビュー取得

**この3本がぼくの代わりに毎日働いている**感覚があります。育休パパで時間がないから、毎日5分の手作業を消すことが、月で見ると3時間になる。これがストック化の本質だと思います。

### ④ AWS構築の調べ物時間を10時間削る：Claudeに伴走させる

ShiseiKeeperは S3 + CloudFront で公開していますが、AWS構築の手順もClaude Codeに伴走してもらいました。

「OACってどう設定するの？」「CloudFrontのキャッシュ無効化、毎回Invalidationでいいの？」「Billingアラーム¥500で仕掛けたい」みたいな質問を投げると、**コマンドと注意点をセットで返してくれる**。

ぼくはAWS基礎知識はあるので、Claudeの回答が「合ってるか怪しいか」は判別できる。**完全初心者には危ないけど、基礎がある人にとってはAWSドキュメントを読み倒すより速い。** これは正直、ぼくの作業時間を1ヶ月で10時間は削っています。

### ⑤ 毎セッションの自己紹介をゼロにする：MEMORY.mdで「ぼくの取扱説明書」を渡す

これは地味だけど効いた話。Claude Codeには MEMORY.md という個人設定ファイルがあって、そこに**ぼくの属性・好み・運用ルールを全部書いておける**んです。

ぼくは以下を書いてます：

これがあるから、新しいセッションで作業を始めても**ぼくのキャラがブレない**。「育休パパなのに独身者みたいな投稿が出てきた」みたいな事故が減りました。

### ⑥ 事業計画を「育てる」運用にする：Claudeに追補を生やさせる

ぼくの事業計画書は HTML で書いていて、現在 v3.8追補⑧ まで来ています。1ヶ月で**追補が8本**生えてます（笑）

なぜ追補方式かというと、**現実が動くたびにClaudeに「ここをこう変えたい」と投げて、追補を生やしてもらうほうが早い**から。ゼロから書き直すより、変更点だけ追補で積み上げて、必要に応じて統合する。

これも「ぼくがゼロから書かない」原則の派生です。

## 1ヶ月で一番身に染みた学び：「作れること」と「売れること」は別物だった

これが今月、いちばん突きつけられた学びです。

Claude Codeを相棒にすると、**ぼく一人では絶対に到達できない速度**で開発・アプリ製作・事業計画・運営が回ります。これは断言できる。1ヶ月でアプリ3本・記事9本・自動化スクリプト3本という数字がそれを証明しています。

**でも、収益を上げるのはまた別の話でした。**

アプリやコンテンツを「作れる」ことと、それが「お金になる」ことは、つながっているようで全然別の領域だった。手数で殴れば売上が立つかというと、そんなことは1ミリもなかったです（売上¥0が答え）。

### 売れるものを作るには、「人が魂を込める」工程が必要

じゃあ何が足りないのか。1ヶ月やって思ったのは、**「頭を使って、人に売れるものを考え、作る」**——この一段がClaudeでは肩代わりできない、ということです。

* 誰の・どんな痛みを解決するか
* なぜそれにお金を払う気になるのか
* 競合と何が違って、なぜ自分が作るべきなのか

ここを考え抜いて、出力に**自分の魂を込める作業**は、絶対に人間しかできない。AIに任せた瞬間に「無難で、誰の心も動かさないもの」が出来上がる。今月の衛星×ESG記事（ビュー13・スキ0）が、それを身をもって教えてくれました。

「もしかしたらAIを使いこなせていないだけかも」とは思います。プロンプトの設計をもっと突き詰めれば、もっと刺さるものを出せるのかもしれない。でも、**少なくとも今のぼくの腕では、魂のある「核」は自分で握るしかない**というのが、1ヶ月の正直な結論です。

### 来月以降の方針：手数を絞って、収益が見える1本に魂を入れる

1ヶ月でClaude Codeの「広さ」は把握できました。コンテンツも、アプリも、自動化も、AWSも、Notion運用も、なんでも回せる。**広く試したからこそ、次は絞れる**。

来月（5月）からはこの軸で進めます。

* **FitForm AIのβ公開に、可処分時間の大半を投下**（9月有料化が一番現実的な収益源）
* **Note記事は本数より「読者が金を払いたくなる切り口」を優先**
* **新しいアプリは原則やらない**（ShiseiKeeperで姿勢監視は十分仕込んだ）

「広く試した1ヶ月」から「狭く深く彫る1ヶ月」へ。Claude Codeで何ができるか分かったから、**そのうち収益が見えるものに賭け直す**フェーズに入ります。
