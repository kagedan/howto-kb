## 1. はじめに

「いいと思います」と言っていた企画が、実装が形になった途端「思ってたのと違う」とひっくり返る——PMなら誰もが覚えのあるこの瞬間を、**企画の段階で先回りして潰す** 話をします。

具体的には、Claude Codeに頼んで作る、ブラウザで開くだけで動く1枚のHTML画面。ボタンを押すとタブが切り替わる、ダミーデータの入ったダッシュボード——その程度の「触れるもの」で、ステークホルダーと「何を作るか」の合意を取ってしまう、というやり方です。

これまでの記事では、Claude CodeでPM業務がどう変わるかを **「時間を取り戻す」** という軸で書いてきました。

> [PMが本来やるべき仕事の時間を取り戻す — Claude Codeで変わった3つのPM業務](https://zenn.dev/medirom_tech/articles/4a61491a6b5059)  
> [優秀なAI専門チームと爆速で高品質に仕上げる要件定義 — PMのためのClaude Code × AIエージェント実践ガイド](https://zenn.dev/medirom_tech/articles/c8d422b940fa6d)

今回はその続きではなく、**別の軸の話**です。  
テーマは、**PM自身がAIエージェントでプロトタイプを作り、ステークホルダーとの合意形成を前倒しする** こと。要件定義よりもさらに手前、「何を作るかをみんなで握る」フェーズを変える話です。

ただし、ここには落とし穴もあります。**AIで誰でも"それっぽい"画面が一瞬で作れる今**、作り方・見せ方を間違えると、プロトが完成品だと誤解されたり、議論や後工程のデザインが見た目に引っ張られたりする——むしろ逆効果になりかねません。だから本記事は「作ろう」で終わらせず、**どう作り、どう見せ、どう渡すか**まで踏み込みます。

この記事のゴールは3つです：

* **ワークフローの考え方**：合意形成のスタート地点をどう変えるか
* **"それっぽさ"の罠の避け方**：完成品誤解とデザインの固定化（アンカリング）をどう防ぐか
* **今日始める道筋**：明日から何を始めればいいか

> **先に大事なことを1つ。**  
> これは **「PMがデザインをやろう」という話ではありません。** ここで作るプロトタイプは、デザインの完成品でも、そのまま実装するものでもありません。**「その機能ができたら、こういう体験になる」というイメージを、関係者全員で触って確かめるための叩き台** です。デザインの良し悪しを問う場ではない——この前提だけ、最初に握らせてください。

## 2. 「言葉だけの合意形成」のつらさ

新しい機能を企画したとき、PMなら誰でも覚えがあるはずです。

* **口頭やパワポで説明しても、人によって浮かぶ絵が違う**。同じ言葉を聞いて、営業と開発と上司がまったく別のものを想像している
* **「いいと思います」が、後でひっくり返る**。企画段階では合意したはずなのに、実装が形になってから「あれ、思ってたのと違う」と言われる
* **手戻りが一番痛いのは"作った後"**。デザインも実装も終わってから方向性がズレていたと分かる。早く形を見せられればよかった、といつも思う

このつらさの正体は何でしょうか。

合意がズレるのは、関係者の理解力が足りないからではありません。  
**言葉や資料と、完成した実物との間に、"触れるもの"が何もなかったから** です。

人は、言葉では合意できているつもりでも、実際に触れるものを前にすると初めて「あ、ここはこうじゃない」と気づきます。

逆に言えば、言葉やパワポでも、人はすでに"自分の頭の中の絵"に引っ張られています。ただし、バラバラに、見えないところで。だから合意がズレる。プロトタイプは、その引っ張られる対象を**全員で見える1つの形に揃える**ものだ、とも言えます。引っ張られること自体が悪いのではなく、揃っていないことが問題なのです。

だったら——**その"触れるもの"を、企画の段階で先に出してしまえばいい。** それが本記事のテーマです。

## 3. 「決めてから作る」から「作って、見せて、決める」へ

これまでの開発フローは、だいたいこうでした。

```
従来：企画 → 要件定義 → デザイン → 実装 →〔ここで初めて形が見える〕→「思ってたのと違う」→ 手戻り
```

形が見えるのは、いつも一番最後。だから合意のズレが発覚するのも一番最後で、手戻りのコストが最大化します。

ここにAIエージェントを挟むと、フローはこう変わります。

```
新 ：企画 → PM×AIでプロトタイプ → 見せて合意 → 要件定義 → デザイナー/エンジニアへバトン → リリース
```

PMがAIエージェントと一緒に、**触れるプロトタイプを企画の段階で作ってしまう。** それをステークホルダー（チーム・営業・上司・顧客）に見せて、フィードバックをもらい、合意を取る。**合意が取れてから** 要件定義・詳細仕様に進み、デザイナーやエンジニアにバトンを渡す。

ポイントは、合意形成のスタート地点が **「言葉」から「触れるもの」に変わる** ことです。

> **ここでのPMの立場をはっきりさせておきます。**  
> PMはデザインやエンジニアリングの専門家ではありません。だから **「何を作るか」を決めるまでがPMの主戦場** で、それが固まったら、作り込みはデザイナーやエンジニアにバトンを渡すのが基本です。  
> （もちろん、自分とAIで十分に良いアウトプットが出せる場面なら、PM自身がリリースまで持っていっても構いません。線引きは固定ではなく、その案件で一番速く価値が出る形を選べばいいだけです。ただしこれは、社内限定・閲覧専用など影響範囲が閉じているものに限ります。ユーザーデータを扱う画面、本番システムに乗るもの、社外ユーザーやブランドに触れるものは、規模に関わらずエンジニアとデザイナーのレビューを通してください。）

なお、正直に書いておくと、この手法が一番効くのは **UI・画面が議論の中心になる機能** で、PMがプロトを直接見せられる関係性がある場合です。バックエンドやAPIが主体の機能、規制上プロトを社外に見せにくい業界、デザインプロセスが厳密に定められた組織では、そのままは使えません。その場合も「言葉より先に、形で揃える」という考え方自体は、図解やサンプルデータなど別の形で応用できるはずです。

## 4. PMでもプロトタイプは作れる — HTMLという選択

「プロトタイプを作る」と聞くと、専用ツールの習得や、デザインスキルが必要に思えるかもしれません。  
でも、私が使っているのは **ただのHTML** です。

なぜHTMLなのか。理由はシンプルです。

* **フレームワークもビルドも要らない**。ファイルが1つあればいい
* **ブラウザで開くだけで動く**。環境構築ゼロ
* **ファイルを送るか、画面共有するだけで、誰でも触れる**。相手にツールのインストールを強いない

営業も、上司も、顧客も、ファイルを開けばその場で触れる。この「誰でもすぐ触れる」というハードルの低さが、合意形成では何より効きます。

そして肝心なのは、**PMは"頼むだけ"** だということです。  
Claude Codeに「こういう画面を作って」と伝えれば、動くHTMLが返ってきます。CSSの書き方を知っている必要はありません。

さらに質を上げるコツが2つあります。

* **既存画面のスクリーンショットを読ませる**：Claude Codeなら画像ファイルのパスを伝えれば読み込んでくれます。「このアプリの世界観に合わせて」とスクショを渡すと、ブランドのトーンに沿ったプロトタイプが出てきます。なお、本番ユーザーの情報が映ったスクショはそのまま渡さず、ダミーデータに差し替えてから渡してください。社内のデータ持ち出しルールも事前に確認しておくと安心です
* **仕様メモや要望を一緒に渡す**：箇条書きのメモでも、それを踏まえた画面を組み立ててくれます

デザインの専門家でなくていいのは、**ここで作るのが合意のための叩き台だから** です。色やフォントの精度ではなく、「この機能、こういう体験になりますよね？」が伝わればいい。それならPMにも十分作れます。

## 5. 実際のワークフロー — 作って、見せて、直す

では、具体的にどう進めるのか。プロトタイプ駆動の合意形成は、大きく3ステップのループです。

### Step 1：作る — イメージをClaudeに渡す

頭の中にある「こういう機能」を、Claude Codeにそのまま言葉で伝えます。難しい指示は要りません。

```
社内向けの勤怠ダッシュボードに、メンバー全員の「今日の出社ステータス（在席／外出／休暇）」を
一覧で見られるカードを追加したい。
触って雰囲気が伝わる簡単なHTMLプロトタイプを作って。
このツールの世界観に合わせたいので、既存画面のスクショ（添付）を参考にして。
```

これだけで、タブやカード、サンプルデータの入った、**触れる画面** が返ってきます。  
PMがやるのは「何を作りたいか」を言葉で伝えることだけ。組み立てはAIが引き受けます。

### Step 2：見せる — 機能のイメージを共有する

できたプロトタイプを、ステークホルダーに触ってもらいます。  
このとき、**見せ方に一言だけ前置き** を入れるのが大事です。

> 「これはデザインの完成版ではなく、機能のイメージを掴んでもらうための叩き台です。色やレイアウトの細部ではなく、**"この機能があったら、こういう体験になる"が想像通りか** を見てください」

この前置きがあるだけで、「ボタンの色が」「フォントが」といったデザイン議論に逸れず、**本質的な合意——何を作るか——に集中** できます。  
言葉で説明していた頃と違い、相手は実際に画面を触りながら反応します。「ここに前回の数値も並べたい」「この順番は逆のほうがいい」——具体的なフィードバックが、その場で出てきます。

### Step 3：直す — フィードバックを反映して、もう一度見せる

もらったフィードバックは、Claudeに直してもらいます。

```
「今週の出社率の推移」も同じ画面に並べたい。
あと、今日まだ未打刻の人を上に表示する並び順にして。
```

直すのは数分で済む作業です。**会議のその場で直す必要はありません。** 「いったん持ち帰って直してきます」で十分。大事なのは、フィードバックをもらってから次に見せるまでが速いことです。  
**この「見せる→直す→また見せる」のループが速いほど、合意は早く、深くなります。** 2〜3回まわす頃には、関係者全員が同じ絵を見ている状態になります。

ループを閉じるとき、もう一つだけ。**「合意した点」と「まだ決めていない点」を、最後に言葉にして残してください。** プロトの説得力は強いので、放っておくと「全部OKした」ことになりがちです。「今日決まったのは◯◯。△△はまだ決めていません」と一言残し、あわせて「実装に入る前なら撤回のコストはほぼゼロなので、あとから違和感が出たら遠慮なく言ってください」と伝えておく。合意を前倒しすることと、異論を言えなくすることは違います。早く固めた合意ほど、開けておく窓も明示するのが安全です。

## 6. 合意が取れたら、バトンを渡す — 要件定義へ

プロトタイプで「何を作るか」が固まったら、PMの次の仕事は **要件定義・詳細仕様化** です。  
そして、ここで作った叩き台は、そのまま **次工程の共通言語** になります。

私はプロトタイプで合意を取ったあと、要件定義の文書（Design Doc）を作るのですが、**そのプロトタイプを文書と紐づけて参照できるようにしています。** 文書に書く数値や画面例は、プロトタイプと食い違わないように揃える。こうすると、「文書を読んでもピンとこない」という関係者も、プロトタイプを触れば一発で理解できます。  
（私の場合は、Design Docのフォルダの中に `prototype/` を切ってHTMLを置き、文書から相対リンクで参照しています。完全な同期を手作業で維持するのは難しいので、「画面と仕様の食い違いに気づいた人が、片方を直したら、もう片方にもメモを残す」くらいの軽い運用にしておくと続きます。）

ここから先の要件定義の進め方は、前回の記事で詳しく書きました。

> [優秀なAI専門チームと爆速で高品質に仕上げる要件定義 — PMのためのClaude Code × AIエージェント実践ガイド](https://zenn.dev/medirom_tech/articles/c8d422b940fa6d)

うれしいのは、**プロトタイプが要件定義の最高のインプットになる** ことです。  
画面のイメージも、操作の流れも、もう形になっている。「白紙から要件を書き起こす」のではなく、「合意済みのプロトタイプを言語化する」作業になるので、ここでも負担がまるで違います。言葉で書いた仕様と、触れる画面。この2つが揃っていると、後工程の「思っていたのと違う」が、目に見えて減ります。

そして要件定義が固まったら、**デザイナーとエンジニアにバトンを渡します。**  
プロトタイプはあくまで叩き台なので、本物のデザインはデザイナーが、実装はエンジニアが担当する。PMは「何を作るか」を握り切った状態で、専門家に引き継ぐ。これが基本の形です。

このときエンジニアに渡すのは「何が合意されたか」であって、**プロトのコードではありません**。プロトのHTMLは仕様を伝えるための使い捨てです。あわせて、画面の要素を「合意済み／未決／AIが補完しただけで要求ではない」の3つに分けたメモを添える——これがないと、エンジニアは画面の全ピクセルを要求として扱うしかなくなります。

PMの役割は「決めること」。専門家の役割は「最高の形で実現すること」。**プロトタイプは、その2つをつなぐバトン** です。

## 7. 失敗から学んだ、プロトタイプ合意形成で気をつけること

正直に言うと、最初からうまくいったわけではありません。やってみて初めて分かった落とし穴が、5つありました。

### 1.「完成品」と誤解される罠

きれいに動くプロトタイプを見せると、相手はつい **「もう完成してる」** と受け取ってしまいます。「じゃあこれで進めて」と話が飛んで、デザインや実装の検討がすっ飛ばされかけたことがありました。  
だから、**見せるたびに「これは叩き台です」と宣言する。** 毎回しつこいくらいでちょうどいい、というのが学びです。

### 2. 作り込みすぎる罠

時間をかければ、いくらでもリッチなプロトタイプが作れてしまいます。でも作り込むほど、相手の関心は「機能のイメージ」から「デザインの細部」に逸れていく。しかも自分の工数も食う。  
**合意に必要な分だけ作る。** 荒くても、伝わればいい。これが鉄則です。

### 3.「"それっぽいUI"に引っ張られる」罠

少し前に「言葉でも人は"頭の中の絵"に引っ張られている」と書きました。プロトタイプはそれを1つの形に揃える道具ですが、今度は揃えた先のプロトの**見た目のレベルまで**引っ張られてしまう。**議論が「この機能を作るべきか」より「ボタンの位置や色」に吸い寄せられ、後のデザインまでプロトの見た目に縛られる**——いわゆるアンカリングです。さきほどの「完成品と誤解される」罠の、もう一段先にある静かな罠だと思ってください。

これはUXの世界で古くから知られています。Nielsen Norman Group の Kara Pernice は、完成度が高いほど（特に意思決定層が）「もう稼働させよう」と言いやすくなること、時間と労力をかけたぶん、うまくいかなくても引き返しにくくなること、逆に大ざっぱなものは「明日出荷されるもの」とは受け取られないこと、を指摘しています（[低忠実度か高忠実度か（U-Siteによる日本語訳）](https://u-site.jp/alertbox/ux-prototype-hi-lo-fidelity)）。

しかも私の体感では、**この罠はAIで起きやすくなっています**。昔は「作り込みすぎ」に"手作業"という自然なブレーキがかかっていました。今はAIが、ブレーキなしに一瞬で"それっぽい"高完成度の画面を出してしまう。便利さの裏で、デザインを固定化させる力も強くなっているのです。

当時のこの知見の含意は「だから低忠実度から始めよう」でした。AI時代の含意は少し違うと私は考えています。低忠実度も高忠実度も、作るコストがほぼ同じになった今、新しい指針は「**論点に合わせて、忠実度を選び直せること**」です。

対策はシンプルです。

* **忠実度を「決めたい論点」と「相手」に合わせる**：「この機能を出すか」をチーム内で決める段階なら、色も整えない灰色の箱で十分。一方、営業や顧客など"完成イメージで判断する相手"に灰色の箱は伝わらないので、世界観を合わせた画面が要ります。目安は「議論したいレイヤーだけ、忠実度を上げる」。きれいに作るほど、議論は表面に吸い寄せられます
* **アンカーするレイヤーを選ぶ**：揃えたいのは「何を見せるか（機能）」であって、レイアウトや配色ではない。そこは流動的なまま渡す
* **論点が大きいときだけ、複数のラフ案を並べる**：機能の方向性そのものを決めたい局面では、1枚の完成度を上げるより2〜3案を雑に並べたほうが、概念レベルの議論が保てます（方向性が固まっているなら、1枚で十分です）
* **デザイナーには"意図"を渡す**：プロトそのものを「この画面の通りに」と渡すのではなく、「《確定》画面の目的・必要な要素・優先順位」「《未確定》レイアウト・配色・遷移の細部」の2行に分けたメモを添えて渡す。デザイナーが配置を変えたときの、ステークホルダーへの再説明はPMが引き受ける

### 4.「動くからOK」の罠

プロトタイプが動くと、つい「技術的にもいけそう」と思い込んでしまいます。でもHTMLで動くことと、実際のシステムで実装できることは別物です。プロトタイプは"その瞬間だけ動いている見た目"であって、本番データの取り回し・権限制御・パフォーマンス・エラー時の挙動・状態遷移は、一切考慮されていません。  
プロトタイプはあくまで **合意の道具**。「動いてるから来週からβ公開できますよね？」のような話が出たら、いったん止める。実装可能性やデータの裏側は、**必ずエンジニアと握ってから** 約束する。ここを飛ばすと、後で「それはできません」となって信頼を失います。

特に社外（顧客）に見せるときは、二段の備えをおすすめします。まず見せる前に、エンジニアに30分だけプロトを眺めてもらう。聞くことは2つだけ——「この画面に映っているデータ、うちのシステムに存在しますか？」「致命的に無理な要素はありますか？」。設計レビューではなく、**顧客の前で約束できない絵を見せないための安全確認**です。そして見せるときは、画面の隅に「Prototype（検討用）」と入れておく。「いつできますか？」と聞かれたら、「画面の動きはご覧の通りですが、裏側のデータ連携や権限はこれから設計します。御社の運用に合わせて作り込むので、別途お時間をください」と返す。キャプチャやファイルの二次拡散も起きやすいので、配布はその場の操作までに留めるのが安全です。

### 5.「見せる前に挫折する」罠

実は、一番多いのがここかもしれません。最初の出力が思った絵と違うと、つい「やっぱり自分には向いていない」と諦めてしまう。でも、**最初の1発で完璧な出力が出ることは、まずありません。** 「ここはこう変えて」を3〜5回重ねた先で、ようやく人に見せられるものになります。1発目を完成品の基準で評価しないこと——続けられるかどうかの分かれ目は、案外ここにあります。

## 8. まず試してみる — 今日できる最初の一手

理屈は分かった。では何から始めるか。今日できる3ステップです。

### Step 0：Claude Codeを動かせる状態にする

まだ触ったことがなければ、ここだけ先に済ませてください。

1. **インストールする**：[公式クイックスタート（日本語）](https://code.claude.com/docs/ja/quickstart) の手順どおりに進めます。初回はインストール＋ログインで30分ほど見ておくと安心です（2回目以降の起動は1分かかりません）
2. **フォルダを作る**：デスクトップに `proto` などの空フォルダを1つ作ります
3. **ターミナルで移動する**：ターミナルを開き、`cd` （cdと半角スペース）と打ってから、作ったフォルダをターミナルに**ドラッグ＆ドロップ**してEnter
4. **起動する**：`claude` と打ってEnter。もし「command not found」と出たら、インストールが終わっていないだけです（何も壊れていないので、手順1に戻ればOK）
5. 初回はログインや英語の確認メッセージが出ますが、案内に沿って進めれば大丈夫です

### Step 1：まず1枚、作ってみる

Claude Codeを起動して、こう頼んでみてください。慣れないうちは、抽象的に頼むより、条件を少し添えたほうがいい結果が返ってきます。

```
[業界]の[ユーザー]向けに、[作りたい機能]の画面イメージが伝わる1枚のHTMLを作って。
・index.html の1ファイルで完結（CSS・JSも中に入れる）
・サンプルデータを5件くらい入れる
・色味は落ち着いたトーンで
・ブラウザで開いたら、その場で触れる状態に
```

出てきたHTMLファイルをブラウザで開く。**「あ、ちゃんと触れる」** という体験が、最初の「おお」になるはずです。

### Step 2：既存画面のスクショを読ませる

慣れてきたら、自社プロダクトの画面のスクリーンショットを渡して「この世界観に合わせて」と頼んでみましょう。出力がぐっと"それっぽく"なり、ステークホルダーにも見せやすくなります。このとき、本番のユーザー情報が映った画像は、ダミーに差し替えてから渡すのを忘れずに。

### Step 3：誰か一人に見せて、直す

完璧を目指す前に、**誰か一人に見せてみる。** もらった一言をClaudeに直してもらい、もう一度見せる。  
この「見せる→直す」のループを一度体験すると、合意形成の景色が変わります。最初は社内の身近な相手で十分です。

## 9. おわりに — 合意形成が変わると、チームが変わる

> **「言葉だけで分かってもらう」は、もう終わりにできます。**

プロトタイプ駆動に変えると、合意形成のスタート地点が **「言葉」から「触れるもの」へ** 変わります。

関係者は、実物を触りながら反応する。だからフィードバックが具体的になり、合意が深くなる。「思ってたのと違う」が、企画の段階で——つまり一番安いタイミングで——潰せる。後工程の手戻りが減り、ステークホルダーの納得感が積み上がっていきます。

PMの仕事の本質は変わりません。**何を作るかを決め、判断し、責任を持つ。**

言葉で説明して、想像してもらって、すれ違う。  
そんな合意形成は、もう手放していいのかもしれません。
