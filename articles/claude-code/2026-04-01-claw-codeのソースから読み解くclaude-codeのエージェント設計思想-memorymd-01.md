## はじめに：ソースマップ流出が明かしたもの

2026年3月31日、Anthropicの「Claude Code」のnpmパッケージ（v2.1.88）に、本来含まれるべきではないソースマップファイルが同梱されていたことが発覚しました。

約51万行・1,900以上のTypeScriptファイルからなるClaude Codeの全ソースコードにアクセス可能な状態になっていたのです。Anthropicは「リリースパッケージングのヒューマンエラー」と声明を出しています。

本記事では、流出コードそのものではなく、それを基にPythonで再実装された**claw-code**プロジェクトの構造を手がかりに、Claude Codeに組み込まれたエージェント設計思想を読み解いていきます。

本記事では流出したオリジナルコードの直接引用は行いません。

## claw-codeとは何か

[claw-code](https://github.com/instructkr/claw-code)は、韓国の開発者Sigrid Jin氏が公開した、Claude Codeの**クリーンルーム・リライト**プロジェクトです。

クリーンルーム・リライトとは、オリジナルのソースコードを直接コピーするのではなく、公開された情報や動作仕様を基にゼロから独自に再実装する手法です。著作権法上、オリジナルコードの複製には該当しないとされる一般的なアプローチです。

**主な特徴：**

* GitHubで**史上最速となる2時間で5万スター**を獲得（現在6.6万超）
* Pythonによる再実装で、Rustへの移植も進行中
* 207コマンド・184ツールの表面仕様をJSON形式で保持
* `coordinator/`、`memdir/`、`hooks/`、`skills/`等のディレクトリ構成を忠実に再現

## 注目すべき設計ポイント3選

### 1. MEMORY.md方式 — 「懐疑的メモリ」の設計哲学

Claude Codeのメモリシステムは4層構造です。

| レイヤー | 役割 | 永続性 |
| --- | --- | --- |
| CLAUDE.md | ユーザーが明示的に書いたプロジェクト指示 | ファイルとして永続 |
| Auto Memory | AIが会話中に学んだ補正・パターン | ファイルとして永続 |
| Session Memory | 通常のコンテキストウィンドウ | セッション内のみ |
| AutoDream | セッション間のメモリ統合 | バックグラウンド処理 |

最も興味深いのは「**懐疑的メモリ（skeptical memory）**」という設計方針です。

MEMORY.mdはインデックスファイルであり、メモリそのものではありません。各記憶は個別のMarkdownファイルに保存され、MEMORY.mdにはポインタだけが記載されます。

そして、メモリに記録された情報は**過去のスナップショット**に過ぎないと明確に位置づけられています。「メモリにXが存在すると書いてある」と「今Xが存在する」は同義ではない。メモリを参照したら、必ず現在の状態を確認してから行動する設計です。

これは長期稼働エージェントにおける「陳腐化した記憶に基づく誤動作」を防ぐ実践的なアプローチです。

メモリの型も明確に分類されています：

* **user** — ユーザーの役割、好み、知識レベル
* **feedback** — 修正指示（「なぜ」と「今後の適用方法」をセットで記録）
* **project** — 進行中のプロジェクト状況
* **reference** — 外部システムへのポインタ

特にfeedback型は、修正内容だけでなく**Why**と**How to apply**をセットで記録します。これにより、エッジケースでも文脈を理解した判断が可能になります。

### 2. AutoDream — アイドル時のメモリ統合

AutoDreamはClaude Codeの中でも最も先進的な未リリース機能の一つです。UC Berkeleyの論文「Sleep-time Compute」（arXiv:2504.13171）にインスパイアされた設計で、推論時のコンピュートを約5倍削減しながら同等精度を達成できるという研究に基づいています。

**起動条件は3つのゲート：**

1. **時間ゲート** — 前回の統合から24時間以上経過
2. **セッションゲート** — 5セッション以上経過
3. **ロックゲート** — 統合ロックの取得（並行実行防止）

**4フェーズの統合サイクル（約8〜10分）：**

| フェーズ | 処理内容 |
| --- | --- |
| Orient | MEMORY.mdと既存トピックファイルのスキャン |
| Gather Signal | 修正指示、意思決定、繰り返しパターンの抽出 |
| Consolidate | 重複排除、矛盾解消、**相対時間の絶対化**（「昨日」→「2026-03-15」） |
| Prune and Index | MEMORY.mdを**200行・25KB以内**に維持、詳細はトピックファイルへ分散 |

この設計の本質は「**後ろ向きの統合**」です。AutoDreamは過去の記憶を整理するだけで、未来を予測しない。予測が外れるリスクを完全に排除し、次のセッション開始時により効率的なメモリ状態を用意する、という割り切りです。

### 3. マルチエージェントコーディネーター — プロンプトで制御する

Claude Codeのコーディネーターモードで最も衝撃的なのは、**オーケストレーションアルゴリズムがコードではなくプロンプトで実装されている**点です。

一般的なフレームワーク（LangGraph、CrewAI等）ではエージェント間の調整はコードとして実装されます。Claude Codeのアプローチは根本的に異なり、コーディネーターにプロンプトで指示を与えています：

> * 「弱い成果物を安易に承認するな」
> * 「調査結果を理解してから次の作業を指示しろ」
> * 「並列処理がお前の超能力だ。ワーカーは非同期で動く」

**4フェーズ構成：**

```
Research（並列調査）→ Synthesis（統合・仕様策定）→ Implementation（実装）→ Verification（検証）
```

ワーカー間の通信は`<task-notification>`というXML形式のメッセージと共有スクラッチパッドで行われます。

LLMの推論能力に調整を委ねることで、ハードコードされたワークフローよりも柔軟に新しいタスクパターンに適応できるという利点があります。

## 自律エージェント開発への応用

これらの設計思想は自律エージェント開発に直接応用できます。

### エージェント設計への5つの教訓

**1. メモリは「信頼しない」前提で設計する**  
MEMORY.mdの懐疑的メモリ設計は、長期稼働エージェントの必須パターンです。メモリはキャッシュと同じで、「有効期限切れかもしれない」前提で扱うべきです。

**2. メモリ統合は非同期・バックグラウンドで**  
AutoDreamが示すように、メモリの整理はアイドル時間に行うべきです。

**3. オーケストレーションの「コード vs プロンプト」は二択ではない**  
基本構造はコードで定義し、調整ロジックはプロンプトに委ねるハイブリッドが実践的です。

**4. ハードリミットを設ける**  
200行キャップのように物理的制約を設けることで「何を残すか」の優先順位判断が強制されます。

**5. 高速な判定にはLLMを使わない**  
Claude Codeではフラストレーション検知に正規表現を使っています。適材適所でツールを選ぶ姿勢は重要です。

## まとめ

claw-codeプロジェクトから読み解いたClaude Codeの設計思想：

* **懐疑的メモリ** — 過去の記憶を「スナップショット」として扱う
* **AutoDream** — アイドル時間を活用した非同期メモリ統合
* **プロンプト駆動のオーケストレーション** — LLMの推論能力に調整を委ねる

これらは自律エージェント開発全般に適用可能な知見です。claw-codeのようなクリーンルーム・リライトを通じて、設計思想を自分のプロジェクトに取り込んでいくことが、今できる最も生産的なアプローチでしょう。
