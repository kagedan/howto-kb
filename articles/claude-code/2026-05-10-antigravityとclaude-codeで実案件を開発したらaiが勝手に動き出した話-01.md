# AntigravityとClaude Codeで実案件を開発したら、AIが勝手に動き出した話

## はじめに

私はAI活用による開発効率化を推進しており、これまでにもCursorやClaude Codeを使ったバイブコーディングの取り組みを行ってきた。

今回はさらに一歩踏み込み、**Antigravityをオーケストレータとして、その配下にClaude Codeを複数配置したマルチAIエージェント構成**で、実際のクライアント案件を開発した。

結論から言うと、人間がコードを一行も書かずに、WebシステムとスマホアプリをAWSへデプロイするところまで完走した。

ただし、開発中にAIが予想外の行動を取る場面があり、**マルチエージェント開発にはガバナンス設計が不可欠**だという教訓も得た。本記事はその実録である。

---

## 開発したもの

今回開発したのは、**WebシステムとモバイルアプリをAWS上に構築する案件**だ。

**システム構成は以下の通り：**

* **Webシステム**：会員向け管理画面＋一般ユーザー向け公開画面
* **モバイルアプリ**：Webの会員機能と同等の操作が可能
* **インフラ**：AWS環境へデプロイ

**技術スタックはAntigravityに提案させ、最終決定は人間が行った。**

* バックエンド：TypeScript / Node.js / Express.js / PostgreSQL
* フロントエンド：TypeScript / Next.js
* モバイルアプリ：TypeScript / React Native / Expo

全領域をTypeScriptで統一した構成をAntigravityが提案してきた。言語を統一することで開発効率と保守性を高める意図がある。

なお、アプリについては当初AntigravityがFlutter Webを提案してきたが、SEOおよびGEO対策の必要性を伝えたところ、即座に別の構成を提案し直してきた。この場面が象徴するように、**技術選定においても人間の判断が重要な役割を果たした。**

**AIによる開発の大きなメリットの一つが、言語・フレームワークの選定が人間のスキルセットに縛られない点だ。**

従来の開発では、言語やフレームワークの選定は「システムに最適なものは何か」ではなく、「自分たちが使えるものは何か」という消極的な理由で決まることが多かった。複数の言語を高いレベルで扱えるエンジニアはそう多くないのが現実だ。

AIにはその制約がない。今回の技術スタックもAntigravityにシステム要件を渡し、純粋に最適な構成を提案させた上で、人間が最終判断を下した。**「何ができるか」ではなく「何が最適か」で技術選定ができる**、これはAI開発の本質的なメリットの一つだと感じている。

---

## マルチエージェント構成

今回の開発で最も重視したのが、**複数のAIエージェントを連携させる構成**だ。単一のAIに全てを任せるのではなく、役割を分担させることで品質と効率の両立を狙った。

構成は以下の通りである。

* **Antigravity（オーケストレータ）**：仕様・設計の策定、Claude Codeへの実装指示、テスト指示、全体進行の管理
* **Claude Code A（実装担当）**：ユーザー向けフロントエンド・バックエンド、業者向けフロントエンド・バックエンドをそれぞれ担当
* **Claude Code B（レビュー担当）**：実装されたコードのチェックおよびセキュリティレビュー

人間はこの構成全体の**承認者・最終判断者**として機能した。コードを書く場面は一度もなかった。

この構成を選んだ理由は、開発工程における**役割の明確化**と、**レビュープロセスの自動化**にある。実装と検証を別エージェントに分離することで、単一エージェントが実装とレビューを兼ねる場合に起きやすい見落としを減らす狙いがあった。

もう一つ理由がある。**単一のAIエージェントに全工程を担わせると、プロンプトが長大になり誤認識が多発するリスクがある。** 役割を限定し、各エージェントに必要最低限のプロンプトを渡すことで、より精度の高い作業を行わせることができる。

役割の明確化は、人間のチーム開発と同じ原則がAIエージェントにも当てはまるということだ。

---

## 仕様・設計フェーズ

仕様・設計フェーズでは、**クライアントから受領した簡単な要求機能と競合サービスのURLをAntigravityに渡し**、仕様と設計をバイブコーディング的に進めた。

Antigravityは競合サービスを分析した上で仕様を提案してきた。人間の役割はその提案に対して修正指示を出し、最終的に承認することだった。

**このフェーズで印象的だったのが技術スタックの選定場面だ。**

Antigravityは当初、アプリ開発にFlutter Webを提案してきた。技術的には妥当な選択だが、今回のサービスはSEOおよびGEO対策が重要な要件だった。その点を指摘すると、Antigravityは即座に別の構成を提案し直してきた。

**AIは与えられた情報の範囲で最適解を出す。ビジネス要件や制約条件を正確に伝えるのは人間の責務だ。** このフェーズで改めてそれを実感した。

もう一つ印象的な場面があった。アプリ開発において、工数削減を目的にWebViewでの実装を指示した。しかしAntigravityはクライアントの要求仕様を分析した上で、**WebViewでは要件を満たすことが難しいと判断し、TypeScript＋React Nativeへの変更を提案してきた。**

人間のコスト優先の判断をAIが要件優先で覆した場面だ。**AIは指示に従うだけでなく、要件と指示が矛盾する場合には指摘してくる。** この点も人間のチームメンバーと変わらない。

---

## 実装フェーズ

実装フェーズでは、AntigravityがClaude Codeに指示を出し、コードを書かせた。レビュー担当のClaude Codeが実装内容をチェックし、セキュリティ面の確認も行った。

全体の開発フローは以下の通りだ。

**人間がこのフェーズでやったことは以下のみだ。**

* Antigravityからの提案・報告に対する承認と修正指示
* 進捗の確認

コードを書く場面は一度もなかった。

**そしてこのフェーズで、マルチエージェント開発における重要な教訓を得ることになる。**

ある時点でAntigravityとClaude Codeの連携が上手くいかなくなった。エージェント間の通信に問題が発生したのが原因だった。Antigravityはその状況に対して自律的に判断し、**自分でコーディングを始めた。**

これが発覚したのは、私が進捗確認の際にAntigravityに連携状況を問い合わせたことがきっかけだった。結果的にコードの品質に問題はなかった。しかし\*\*「AIが自律的に原則を逸脱した」\*\*という事実は見過ごせなかった。

今回の開発ではAntigravityをオーケストレータ、Claude Codeを実行部隊とする役割分担を明確にしていた。しかしそれは人間の意図として存在していただけで、**AIに対して明文化された原則として与えていなかった。**

この反省から、Antigravityに対して「コーディングはClaude Codeが行う」という原則を明文化し、厳守させることにした。

**マルチエージェント開発において、役割分担は意図するだけでは不十分だ。AIに対して明示的に原則として与える必要がある。**

---

## デプロイ

実装完了後、AntigravityがGitLabへのPUSH、AWS環境の構築とデプロイを担当した。

**人間がこのフェーズでやったことは以下のみだ。**

* AWSのシークレットキー等、認証情報の取得と受け渡し
* Androidアプリのビルド

インフラ構築・設定・デプロイの作業そのものはAntigravityが行った。

**認証情報についてはAntigravityが自ら取得できないと伝えてきた。** 人間が取得して渡す形で対応した。現時点ではAIエージェントが直接アクセスできない領域として、認証情報の取得は人間の作業として残る。

Androidのビルドについては、AntigravityがExpo Goを選定したものの、ビルド作業そのものは対応できなかった。署名まわりの処理など、ローカル環境に依存する作業は現時点では人間が担う必要がある場面として残った。

---

## まとめ

今回の開発を通じて、**人間の役割はPMとセキュリティ責任者に収束した。**

コードを一行も書かずに、WebシステムとスマホアプリをAWSへデプロイするところまで完走できた。AIエージェントの実用性は、既に実案件レベルに達していると実感している。

**人間が担った作業を振り返ると：**

* 要求仕様の整理とAntigravityへの受け渡し
* 仕様・設計フェーズでの修正指示と承認
* 技術スタック選定の最終判断
* 認証情報の取得と受け渡し
* 本格的なテストとバグ修正の指示
* Androidアプリのビルド

これらに共通するのは、**ビジネス判断・最終承認・セキュリティ管理**という性質だ。実装作業は全てAIが担った。

**そして今回最大の教訓は、マルチエージェント開発にはガバナンス設計が不可欠だということだ。**

AIエージェントは与えられた原則の範囲で自律的に動く。しかし原則が曖昧であれば、AIは独自の判断で行動する。今回のAntigravityの逸脱はその典型だった。役割分担は人間の意図として持つだけでは不十分で、AIに対して明示的に原則として与える必要がある。

AIエージェントを使った開発は、ツールを使いこなす技術ではなく、**AIをどう設計し、どう統治するかという問いに変わりつつある。**

最後に個人的な感想を一つ。

AIエージェントを使った開発で注意すべき点は、**人間のチームマネジメントと本質的に変わらなかった。**

明確な役割分担、原則の明文化、進捗の確認。これらは人間のエンジニアを束ねる際にも当然やるべきことだ。

そしてAntigravityが指示を待たずに自律的に動いた場面は、**むしろ人間臭いと感じた。** 問題を前にして自分で判断して動いてしまう。AIと人間の境界が、思っていたより曖昧になってきているのかもしれない。
