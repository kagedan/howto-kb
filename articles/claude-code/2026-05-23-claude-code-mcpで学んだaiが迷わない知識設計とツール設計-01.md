MCPサーバーを使った検証で、設計として大事だと感じたのは次の2つでした。

* `CLAUDE.md` に知識を集めすぎない
* MCPで公開する関数を増やしすぎない

## `CLAUDE.md` は薄いほうがよかった

`CLAUDE.md` は便利ですが、何でも書き始めるとすぐ大きくなります。  
Claude Code では `CLAUDE.md` がセッション開始時に読み込まれるので、肥大化すると毎回最初に関係ない情報まで載りやすくなります。

そのため、`CLAUDE.md` には以下だけを書くほうがよさそうでした。

詳細は別ファイルに分けておいて、必要なときだけ読ませる形です。

```
.
├─ CLAUDE.md
└─ docs/
   └─ knowledge/
      ├─ viewer-app-overview.md
      ├─ mcp-tool-contracts.md
      ├─ screen-a-notes.md
      ├─ auth-flow.md
      └─ failure-patterns.md
```

大事なのは、`CLAUDE.md` を情報の本体にしないことでした。ナビゲーションと絶対ルールの置き場にしておくほうが扱いやすいです。

## 詳細知識は分割して置いたほうがよかった

viewerアプリの画面仕様や、MCPツールごとの入出力前提のような情報は、どうしても増えていきます。  
それを全部 `CLAUDE.md` に書くより、用途ごとに分けたほうが読みやすく、更新もしやすかったです。

特に重要だと感じたのはファイル名でした。  
**名前を見れば何が書いてあるか分かる**状態にしておくと、AIも人間も迷いにくいです。

なお、ここで言っているのは Claude Code の skills のような自動読込機能そのものではなく、通常のプロジェクトファイルを必要時に読みやすく整理する、という意味です。

## MCPの公開関数も増やしすぎないほうがよかった

もう1つ大きかったのが、MCPで公開する関数の数です。

今のところ10個前後で、しかも名前から役割が分かる状態だと、AIも迷いにくい印象がありました。  
ただ、似た関数を増やしすぎると、今度は「どれを使うべきか」が曖昧になります。

ここで言いたいのは、ツールが増えると即コンテキストが破綻する、という話ではありません。  
Claude Code には MCP Tool Search があり、公式ドキュメントでも、起動時にはツール名だけが読み込まれ、完全なスキーマは必要になるまで遅延されると説明されています。

問題になりやすいのは、むしろこちらです。

* 名前が似ている
* 役割の差が小さい
* 入出力の違いが分かりにくい

こうなると、AIにとっては「選択肢が多い」のではなく「どれを使えばいいか分かりにくい」状態になります。

## 新しい関数を足す前に、まず既存関数を広げる

そのため、新しいことをやりたくなったときに、すぐ別関数を増やすのではなく、まずは既存関数の引数追加で吸収できないかを考えるほうがよさそうでした。

たとえば、

```
fetch_page
fetch_page_with_retry
fetch_page_with_headers
fetch_page_with_proxy
```

のように横に増やすより、`fetch_page` に必要なオプションを足したほうが、AIには選びやすい場合があります。

もちろん、1つの関数を何でも屋にしすぎるのもよくありません。  
ただ、少なくともPoC段階では、**似た責務の関数を増やしすぎない**ことのほうが重要だと感じました。

## おわりに

今回の学びを一言でまとめると、AI向けの設計では「たくさん与える」より「迷わないように絞る」ほうが大事、ということでした。

* 知識は `CLAUDE.md` に集めすぎない
* 公開関数は増やしすぎない

この2つを意識するだけでも、だいぶ扱いやすさが変わりそうです。
