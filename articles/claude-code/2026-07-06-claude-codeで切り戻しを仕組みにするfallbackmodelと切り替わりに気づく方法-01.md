先に設定から書きます。Claude Codeには、主モデルが落ちたとき自動で別モデルに切り替えるチェーンを、settings.jsonに書けます。

```
// ~/.claude/settings.json（またはプロジェクトの .claude/settings.json）
{
  "model": "claude-fable-5",
  "fallbackModel": ["claude-opus-4-8", "claude-sonnet-4-6"]
}
```

主モデルが過負荷や提供停止で応答しないとき、配列の順に試して、最初に通ったモデルで続行します。エントリは最大3つ、モデルIDでもエイリアス（opus など）でも書けます。v2.1.166以降の機能なので、入っていなければ `claude update` してください。

前回、「Fable 5が3日で消えたので、特定モデル前提の組み方をやめた」という話を書きました。あれは考え方の話で、今回はその実装です。切り戻しは、根性や当日の機転ではなく、設定で持てます。

## availability の切り戻しと、content の切り戻し

自分で組んでみて最初に整理が要ったのは、Claude Codeの「切り替わり」が一種類ではないことでした。

ひとつは、いま書いたfallbackModelチェーン。モデルが過負荷（529）や停止で「応答できない」ときの、可用性ベースの切り替えです。

もうひとつが、Fable 5固有の、内容ベースの切り替え。Fable 5はサイバーセキュリティや生物系の内容に安全分類器が走っていて、リクエストが引っかかると、そのリクエストを既定のOpus（Anthropic APIならOpus 4.8）で再実行し、以後のセッションはOpus側で続きます。トランスクリプトに通知が出て、戻したければ `/model fable` を打つ。

自分のようにセキュリティ記事やプラグインの脆弱性対応を扱う人間には、後者が意外と身近です。面白いのは、セッションの最初のリクエストで発火することがある点。最初のリクエストにはCLAUDE.mdやgit statusといった作業場の文脈が乗るので、こちらが何も変なことを送っていなくても、その中身で分類器が反応しうる。「初手からOpusになっている」があり得ると知っているだけで、混乱が減ります。

## 切り替わったことに、気づけるか

前回書いたとおり、モデルの入れ替わりでいちばん怖いのは「静かに質が変わる」ことです。チェーンを組むなら、切り替わりに気づく仕掛けとセットでないと、半分しか意味がありません。

自分がやっているのは3つです。

トランスクリプトの通知を読み飛ばさない。切り替え時には通知が出ます。長いセッションだと流れがちなので、「おや、指摘の粒度が変わったな」と感じた瞬間に遡って探す癖をつけました。

`/status` で現在のモデルを確かめる。感覚がズレたと思ったら、まずこれ。思っていたモデルと違うことが、実際にありました。

ステータスラインにモデル名を常時出す。StatusLineの設定でモデルを表示しておくと、切り替わりが視界に入ります。確認しにいくのではなく、目に入る側に置く。地味ですが、これがいちばん効いています。

## 切り替わった日に、品質を確かめる

気づいたあとの話です。前回の記事で「代表的な入力と期待する出力のペアを数件、手元に持っておく」と書きました。fallbackが発火した日は、その見本ペアを切り替わり先のモデルに流して、出力を見比べます。

大げさな評価基盤は要りません。自分の場合、プラグインのコード生成と記事レビュー、それぞれ数件の「これが通れば大丈夫」を置いてあるだけです。チェーンで自動的に助かるのは稼働で、品質は自動では確認されない。そこだけ手で締める、という分担にしています。

## チェーンが救ってくれない場面

正直に限界も書いておきます。fallbackModelが動くのは、モデルが応答できないときです。認証やAPIキーの問題で落ちているときは、どのモデルに切り替えても同じ理由で落ちるだけなので、チェーンは助けになりません。エラーの種類を見て、モデルの問題か、自分の環境の問題かを切り分けるのは、変わらず人間の仕事です。

また、管理された環境でavailableModelsの許可リストが敷かれている場合、チェーンの各要素もそのリストで検査されます。リスト外のモデルを書いても、そこはスキップされる。会社の環境で組む人は、チェーンと許可リストを一緒に見てください。

## 次の自分に渡すメモ

* 切り戻しは設定で持てる。`fallbackModel` に最大3つ、順に試される（v2.1.166〜）
* Fable 5には内容ベースの切り替えが別にある。初回リクエストで発火することもある
* 切り替わりに気づく仕掛けをセットで。通知・`/status`・ステータスラインの常時表示
* 発火した日は、見本ペアを流して品質を手で確かめる
* 認証系の失敗はチェーンでは救えない。エラーの切り分けは人間の仕事

3日で消えるのを見た直後は、「次に消えたらどうしよう」とぼんやり不安でいました。設定を書いて、気づく仕掛けを置いて、確かめる手順を決めたら、その不安はだいたい片づきました。不安は、備えに変換できるうちに変換しておくのが安上がりです。
