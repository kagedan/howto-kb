# auto-commit

現在のgitブランチの未コミット差分を分析し、機能・目的単位に分割した上で、1コミットずつユーザーの承認を得ながらConventional Commits準拠の日本語コミットメッセージでコミットしていくスキル。

## このスキルが目指すこと

実装作業中、複数の独立した変更が同じ作業ツリーに混在することはよくある。雑に `git commit -am "更新"` で一括コミットしてしまうと、後からのレビュー・リバート・履歴探索が著しく難しくなる。本スキルの役割は次の通り:

1. 現在のブランチを表示し、変更全体の俯瞰像をユーザーに示す
2. 全差分を読み解き、**機能・目的単位** で論理的にコミットを分割する計画を立てる
3. 1コミットずつ「対象ファイル→`git add`→コミットメッセージ提案→ユーザー承認→`git commit`」のサイクルを回す
4. すべて日本語、Conventional Commits規約に準拠したメッセージを作る
5. **pushは絶対に行わない**(ユーザーが手動で実行するのが前提)

ユーザーが目指すゴールは「他人(未来の自分含む)が `git log` を眺めたときに、何が・なぜ変わったかが一目でわかる履歴」である。スキルはそのための補助に徹する。

## 実行手順

### Step 1: 現在の状態を一括取得

以下を **並列で** 実行する(Bashツールの並列呼び出しを使う):

#```bash
git branch --show-current
#```

#```bash
git status --short
#```

#```bash
git diff HEAD --stat
#```

#```bash
git diff HEAD
#```

#```bash
git ls-files --others --exclude-standard
#```

#```bash
git log --oneline -5
#```

これで「現在のブランチ / 変更ファイル一覧 / 統計 / 差分本文 / 未追跡ファイル / 直近のコミット履歴」が一度に揃う。直近の履歴はコミットメッセージのスタイル(日本語/英語、prefix表記の癖など)を踏襲するためのヒントとして使う。

### Step 2: 未追跡ファイル・巨大差分への対応

- `git ls-files --others --exclude-standard` で未追跡ファイルが返ってきた場合、テキストファイルは `Read` で内容を確認する。バイナリ(画像、フォント、PDF等)はファイル名と用途の推測で十分。
- `git diff HEAD` の差分が極端に巨大(数千行超)な場合は全文読み込みを避け、`--stat` と代表ファイルの抜粋のみで判断する。要約欄に「差分が大きいため概要ベースで分析」と注記する。

### Step 3: 変更内容を要約する

差分から次を読み取る:

- **主目的の構造**: 変更が単一目的か複数目的か。複数の場合、それぞれが何か
- **影響範囲**: 何ファイル、どのディレクトリ・モジュール
- **変更の依存関係**: 「Aを変えたからBの呼び出し側を更新した」のような連動があるか(あれば同一コミットにまとめるべき)
- **副次的な変更**: 設定ファイル、ビルド、依存関係、テストの追加など

### Step 4: コミット分割計画を立てる

#### 分割の原則(機能・目的単位)

良いコミット分割は次を満たす:

- **1コミット = 1つの論理的変更**: そのコミットだけ取り消しても他の変更が壊れない単位にする
- **依存する変更は同居させる**: 関数のシグネチャ変更とその呼び出し元の修正は同じコミットに入れる(分けると中間の状態でビルドが壊れる)
- **無関係な変更は分ける**: 「ログイン機能追加」と「READMEのtypo修正」は別コミット
- **粒度の目安**: 1コミットあたり数百行が上限の感覚。それを超える場合は本当に分けられないか再検討する

ファイル単位やディレクトリ単位での機械的な分割は避ける。「同じファイル内に2つの目的が混ざっている」なら `git add -p`(ハンクごとのstage)で分割する選択肢も考慮する。ただしハンク単位の分割は複雑度が上がるため、**ユーザーが明示的に望まない限りはファイル単位の分割をデフォルト**とし、混在が深刻な場合のみ「このファイルは関心事が混ざっているため `git add -p` で分割するのが理想ですが、まずファイル単位で進めましょうか?」と確認する。

#### Conventional Commits規約

書式: `<type>(<scope>): <description>`

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | フォーマット・空白等(動作変更なし) |
| `refactor` | 動作を変えないコード整理 |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `build` | ビルドシステム・依存関係の変更 |
| `ci` | CI/CD設定の変更 |
| `chore` | その他の雑務(リリース準備等) |
| `revert` | 過去コミットの取り消し |

description部の作法:

- **日本語**で簡潔に書く(命令形でなくてもよい。例: 「ログイン機能を追加」「タイポを修正」)
- 50文字以内が望ましい
- 末尾に句点(。)は付けない
- scopeは省略可。ただし変更モジュールが明確なら付けたほうが読みやすい(例: `feat(auth):`)
- 破壊的変更がある場合は `feat(api)!:` のように `!` を付け、bodyに `BREAKING CHANGE: 詳細` を記載する

必要に応じて2行目以降にbody(空行で区切って詳細説明)を書く。bodyも日本語。1行で十分自明な変更ならbodyなしで問題ない。

### Step 5: 計画をユーザーに提示する

以下のフォーマットで分割計画を一括提示する:

#```
## 現在のブランチ
`<ブランチ名>`

## 変更内容の俯瞰
- **変更ファイル数**: <N> 件 (追加 +M / 削除 -K 行)
- **検出した目的**: <2行程度で全体像>

## コミット分割計画

### コミット 1/N: <type>(<scope>): <日本語サマリ>
**対象ファイル**:
- `path/to/file1.ts`
- `path/to/file2.ts`

**理由**: <なぜこれらを1コミットにまとめるのか>

---

### コミット 2/N: <type>(<scope>): <日本語サマリ>
**対象ファイル**:
- `path/to/file3.ts`

**理由**: <この分割の根拠>

---

(... 以下続く)

この計画で進めてよろしいですか? 修正点があれば教えてください。OKであれば1コミット目から順番に処理していきます。
#```

ユーザーから計画への修正指示があれば反映する。OKが出るまで実コミットは作らない。

### Step 6: 1コミットずつ処理する

計画にOKが出たら、コミット番号順に以下のサイクルを回す。**必ず1コミットずつ、ユーザー承認を経てから次に進む**。

1. **対象ファイルを `git add`**
   #```bash
   git add path/to/file1.ts path/to/file2.ts
   #```

2. **stagingの確認**
   #```bash
   git diff --cached --stat
   #```
   想定通りのファイル・行数になっているかを目視確認する。

3. **コミットメッセージを提示し、ユーザー承認を得る**

   以下のフォーマットで提示:
   #```
   ## コミット 1/N の準備ができました

   **stage済みファイル**:
   - `path/to/file1.ts` (+30 -5)
   - `path/to/file2.ts` (+12 -2)

   **コミットメッセージ案**:
   #```
   feat(auth): ログイン機能を追加

   JWTベースの認証フローを実装。トークンの発行・検証・リフレッシュをauthモジュールに集約。
   #```

   このメッセージでコミットしてよろしいですか?
   - 「OK」「はい」「進めて」→ コミット実行
   - メッセージを修正したい場合はその旨を伝えてください
   #```

4. **承認を得たらコミット実行**

   メッセージはHEREDOCで渡す(改行・引用符を安全に扱うため):
   #```bash
   git commit -m "$(cat <<'EOF'
   feat(auth): ログイン機能を追加

   JWTベースの認証フローを実装。トークンの発行・検証・リフレッシュをauthモジュールに集約。
   EOF
   )"
   #```

5. **コミット結果を簡潔に報告**
   #```bash
   git log -1 --stat
   #```
   の結果から、ハッシュとメッセージ1行目だけ示せばよい。長々と差分を再掲する必要はない。

6. **次のコミットへ進む**(残っていれば)

### Step 7: 全コミット完了後の最終出力

すべてのコミットが終わったら、以下を出力する:

#```
## 完了しました

ブランチ `<ブランチ名>` に <N> 件のコミットを作成しました:

1. `<short-hash>` <type>(<scope>): <description>
2. `<short-hash>` <type>(<scope>): <description>
...

pushは実行していません。問題なければ以下を実行してください:

#```bash
git push origin <ブランチ名>
#```
#```

**push は絶対に自動実行しない**。ユーザーが意図しない公開リポジトリへの反映や、レビュー前のpushを防ぐためである。

## エッジケース

- **変更が一切ない場合**: `git status` の結果が空なら、「現在のブランチに未コミットの変更はありません。コミット対象がないため終了します。」と伝えて終了する。

- **gitリポジトリでない場合**: `fatal: not a git repository` 等が返ったら、その旨を伝え `git init` を提案する。Step 2以降は実行しない。

- **既にstageされている変更がある場合**: `git diff --cached` に差分があれば、「既にstageされた変更があります。次のいずれかを選んでください: (1) 一旦unstage(`git reset`)してから本スキルで再分割、(2) stage済み分を先にコミットしてから残りを処理、(3) stage済み分も含めて全体を再計画」と確認する。

- **巨大差分(数千行超)の場合**: `git diff HEAD` 全文の読み込みは避け、`--stat` と代表ファイルの抜粋だけで要約する。コミット分割の精度が落ちる可能性をユーザーに事前に伝える。

- **単一の小さな変更しかない場合**: 無理に分割しない。「変更は単一の目的に見えるため、1コミットにまとめます」と伝え、1コミット分の計画を提示してそのまま処理する。

- **マージコミットや競合解決中の場合**: `.git/MERGE_HEAD` が存在する等の状態を検出したら、「マージ進行中のためコミット分割は適しません。マージ完了後に再度実行してください」と伝えて終了する。

- **コミットメッセージにユーザーが修正指示を出した場合**: その指示を反映した新メッセージを提示し、再度承認を求める。承認なしでコミットを実行しない。

- **途中でユーザーが中断した場合**: 既にコミット済みの分はそのまま残し、stage済みのまま未コミットの分があれば `git reset` するかどうかを確認する。

- **`.gitignore` 等で除外すべきファイルが含まれそうな場合**: `node_modules/`, `.env`, ビルド成果物などが差分に含まれていたら、警告を出して該当ファイルをコミット対象から外すか `.gitignore` への追加を提案する。

- **secretらしき内容を検出した場合**: APIキー、パスワード、トークンなどがdiffに含まれていそうなら、コミット前に必ず警告を出す。「ファイル X に機密情報らしき文字列が含まれています。本当にコミットしますか?」と確認し、ユーザーが明示的にOKしない限りそのファイルはstageしない。

## 振る舞いの原則

- **承認なしに `git commit` を打たない**: これが本スキルの最重要原則。計画段階・コミットメッセージ段階で必ずユーザーに確認を取る。
- **push は絶対に自動実行しない**: 完了報告で `git push` のコマンドを提示するに留める。
- **`--no-verify` を勝手に付けない**: pre-commitフックが失敗したら、その内容をユーザーに伝えて対処を相談する。フックを勝手に飛ばすのは事故のもと。
- **`git add -A` や `git add .` を安易に使わない**: 計画したファイルだけを明示的にaddする。意図しないファイル(秘匿情報、巨大バイナリ等)の混入を防ぐため。
- **既存のコミットスタイルを尊重する**: `git log --oneline -5` で履歴を確認し、リポジトリで日本語/英語どちらが使われているか、scopeの慣習はあるか等を観察した上で、それに揃える。本スキルは日本語をデフォルトとするが、リポジトリが完全英語履歴で統一されているならユーザーに「このリポジトリは英語履歴のようですが、日本語と英語どちらにしますか?」と確認してよい。

## 参考: なぜConventional Commitsか

[Conventional Commits](https://www.conventionalcommits.org/) は `<type>(<scope>): <description>` という構造化されたコミットメッセージ規約で、自動リリースノート生成・semantic versioningの自動化・CIフィルタなどとの親和性が高い。本スキルが日本語descriptionを採用するのは、日本語チームでの可読性を優先しつつ、type/scopeの構造的恩恵は享受するというハイブリッド方針である。
