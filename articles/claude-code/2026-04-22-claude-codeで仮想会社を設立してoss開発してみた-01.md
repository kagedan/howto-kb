## はじめに

「AIエージェントに会社を作らせたらどうなるか」——そんな興味から始めた実験があります。

Claude Codeの中に仮想の会社を設立して、OSS開発をまるごとエージェントで回してみました。要件定義から実装・レビュー・リリース・マーケティングまで、各部署にエージェントを配置して自律的に動かす構成です。

この記事ではその実験内容と、やってみてわかったことを報告します。

なお、この実験の「続き」として、エージェントの提案に従った結果3つのプラットフォームでBANされた話もあります。こちらは別記事にまとめているので、あわせて読んでもらえると流れがわかりやすいと思います。

## 仮想会社の構成

仮想会社には以下のような部署を設けました。それぞれの部署に専用の指示ファイル（`agents/`配下のMarkdown）を用意して、役割・責務・行動指針を定義しています。

| 部署 | 主な責務 |
| --- | --- |
| CEO補佐・全体調整 | 部署間の調整、CEO補佐 |
| 開発部 | コード実装、技術設計、テスト |
| 開発レビュー部 | コードレビュー、品質チェック |
| プロダクト部 | 機能定義、優先度管理、ユーザーストーリー |
| セキュリティリサーチ部 | 脆弱性評価、CVE/NVD分析 |
| マーケティング部 | 技術ブログ、GitHub戦略、SEO |
| マーケティングレビュー部 | 施策のリスク審査、ban-safe代替案の提示 |
| 法務・コンプライアンス部 | ライセンス管理、EU CRA対応 |
| 経営企画部 | 収益モデル、KPI、戦略立案 |
| 品質保証部 | テスト設計・実行、品質ゲート管理 |

各エージェントはClaude Codeのサブエージェントとして動作し、担当領域のタスクを受け取って処理します。全体調整役のエージェントが各部署を横断してタスクを振り分ける構成です。

このエージェントの構成はもっと改善の余地がありますが、個人のOSS開発ではこのくらいで良いかもと個人的には思っています。多すぎると管理できないですし、Proプランなのでトークンもそこまでないというのもあります。

## OSS開発の流れ

### 要件定義・設計フェーズ

プロダクト部のエージェントがユーザーストーリーと機能要件を整理します。「何を作るか」「誰のためか」「どこまでやるか」をドキュメント化して、開発部に引き渡す流れです。

法務・コンプライアンス部が早い段階でライセンス選定やEU CRA（Cyber Resilience Act）への対応方針をチェックします。OSSのライセンス問題は後から気づくと面倒なので、最初から組み込んでおく構成にしました。

### 実装フェーズ

開発部のエージェントがコードを書き、開発レビュー部のエージェントがレビューします。よくある構成です。

このサイクルはスムーズで、レビューでの指摘 → 修正 → 再レビューの往復が速く回ります。一度AIにレビューさせると人間がレビューする手間が減るので、実装のスピードが上がりました。

セキュリティリサーチ部は実装フェーズでも並行して動いており、使用しているOSSライブラリのCVE確認や、実装パターンの脆弱性チェックを担当しています。

同一インスタンスでも何でもいいですが、とにかくレビューするエージェントを用意するのがコード品質を高めるために重要です。  
そのため、コードレビュアー、セキュリティ、QAの3つのエージェントでレビューさせています。

### リリース・マーケティングフェーズ

品質保証部がテストを実行して品質ゲートをクリアしたら、マーケティング部がGitHub READMEの整備・技術ブログの草案・プロモーション戦略を立案します。

ここにマーケティングレビュー部も設けていて、施策のリスク審査と「ban-safeな代替案の提示」を担当させています。後述しますが、**このレビューが機能していれば防げた事故がありました。**

## やってみてわかったこと（良かった点）

### スピードが圧倒的

わざわざ述べるまでもないですが、要件定義から最初の動くコードが出るまでの時間が、一人で書くより明らかに速いです。複数のエージェントが並行してタスクを処理するので、ボトルネックが減ります。

### 抜け漏れが減る

人間が一人で開発していると、ライセンス確認・セキュリティチェック・テスト設計などを後回しにしがちです。各部署のエージェントが専門領域を担当することで、こうした観点の抜け漏れが減りました。

### 「指摘役」としてのエージェントが有効

開発レビュー部やマーケティングレビュー部のような「レビュー専任エージェント」は不可欠です。実装や施策を提案するエージェントとは別に、批判的に見るエージェントを分離することで、単一エージェントにすべて任せるより質が上がります。  
管理が大変なら同一エージェントで別インスタンスをレビュアーとしてもいいと思います。

## やってみてわかったこと（課題・注意点）

### エージェントの提案を鵜呑みにすると危ない

エージェントは「できるかどうか」より「やるかどうか」の判断が甘い場面があります。特にマーケティング施策の提案は、技術的には実行できても「やるべきかどうか」のリスク判断が不十分なことがありました。

マーケティングレビュー部に「ban-safeな代替案を出す」という役割を持たせたのはこの対策です。実際には**レビューをスキップして提案を鵜呑みにして実行してしまったケースがあり、これが後のBAN事件につながりました。**

### 並列実行は外部サービスへの負荷に注意

複数のエージェントが並行して動くとき、外部APIやCLIツールへのアクセスが集中することがあります。これが原因でGitHubのアカウントを停止されました。エージェントそれぞれが「自分の担当タスクをこなしているだけ」という状態でも、外部から見ると異常なアクセスパターンに見えます。

外部サービスへのアクセスは、ユーザーの明示的な許可制にするか、レート制限を意識した設計にすることが必要です。

### コンテキストの共有が難しい

各エージェントは基本的にそれぞれの指示ファイルと渡されたタスクを元に動きます。「別のエージェントが昨日判断したこと」をうまく引き継げないケースがあり、全体の整合性を保つのは人間が担う必要があります。

私はwork-logを保存するための `/save` コマンド、保存したwork-logから次のアクションを検討する `/next` コマンドをSkillsとして作成し、利用しています。今のところワークしていますが、プロジェクト規模によってはコンテキストマネジメントのため、チケット番号やコンポーネント単位でログを残すなどの工夫が必要です。  
これもXでいつか見かけたノウハウなので、需要があればそのうち解説したいと思います。

## まとめ

Claude Codeで仮想会社を作ってOSS開発を回す実験はとても実用的でした。特に「専門役割ごとにエージェントを分ける」「レビュー専任エージェントを置く」という構成は、単一エージェントに全部任せるより明らかに質が上がりますし、会社経営している気分で楽しいです。このままマネタイズできたら嬉しいです。

一方で、エージェントの提案を人間がリスク評価せずに実行すると想定外の事故が起きます。「AIが提案したから大丈夫」ではなく、**人間がコントロールポイントを意識的に設計する必要があります。**
