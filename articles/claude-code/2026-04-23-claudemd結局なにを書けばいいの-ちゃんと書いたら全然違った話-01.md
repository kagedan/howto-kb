## はじめに

Claude Code の体験記、最近ほんとに増えましたよね。

「アプリが1日でできた」「寝てる間にリファクタが終わってた」。どれもワクワクするんですが、いざ自分で触り始めると、

> 同じ指示をしたのに、昨日と違う結果が返ってくる。  
> 頼んでないファイルまで勝手に直す。  
> 何度注意しても、同じミスを繰り返す。

こうなって「あれ？思ってたんと違う」となりがちです。

結論から言うと、**原因のほとんどは `CLAUDE.md` の書き方**でした。ちゃんと書き直したら、体感が全く変わったので、その知見を共有します。

## 「アプリが作れる」と「安定して動く」は別の話

Claude Code は確かに強力で、ちょっとしたアプリなら初期実装はあっという間に終わります。でも**長く付き合うほど「ブレ」のコストがきいてくる**。

* 昨日通った指示が今日通らない
* 指示に無い機能が勝手に足される
* 同じ注意を何度もチャットで繰り返す

これらは全部、ルールをチャットで毎回投げ直しているから起きます。**永続化すべきルールは `CLAUDE.md` に書く**。それだけで再現性が一気に上がります。

## ちゃんと効く CLAUDE.md の3つのポイント

### 1. とにかく短く保つ

これが一番大事です。

フロンティアモデルが確実に守れる指示は **だいたい 150〜200 個が上限** と言われています。そして Claude Code は、自分自身のシステムプロンプトで **すでに 50 個くらい使っている** 状態でスタートします。

つまり、こちらが CLAUDE.md にびっしり書くと、**古いルールから順に "うっすら無視される"** ような挙動になります。

判断基準はシンプルで、

> この1行を消したら、Claude は本当にミスをするか？

に Yes と答えられない行は消す。「あったら安心」は罠です。指示は多ければ多いほど、**全体の遵守率が一様に下がる**。

### 2. 否定形ではなく、肯定形で書く

悪い例：

```
- console.log は使わないで
- CommonJS は使わないで
- any 型は使わないで
```

良い例：

```
- ログは logger.info / logger.error を使ってください
- モジュールは ESM (import / export) で書いてください
- 型は明示的に指定してください（unknown を許容し、any は禁止）
```

LLM は「しないで」と言われても、**"何をすべきか" の代替が無いと別のところで暴走しがち**。禁止と代替はセットで書くと事故が減ります。

### 3. 冒頭に「役割と境界線」を書く

CLAUDE.md の一番上は、**「あなたは誰で、どこまでやっていいのか」** を明確に書くのがおすすめです。

```
## あなたの役割

- コードの「整理・提案・下書き」を担当します
- 最終的な採否は人間（開発者）が判断します
- 要件が曖昧なときは、実装に入る前に必ず確認してください
- スコープ外のファイル変更は行いません
```

これを書いておくだけで「頼んでないファイルまで勝手に直す問題」がかなり減ります。エージェントの暴走は、**権限と責任を定義していないから** 起きることがほとんどです。

## おすすめの構成テンプレ

僕が今つかっている CLAUDE.md の骨組みを共有します。コピペして各プロジェクトに合わせて削ってください。

CLAUDE.md

```
# プロジェクト名

## あなたの役割
- 何をする人か
- 何はしない人か
- 迷ったときのふるまい

## 技術スタック
- 言語 / フレームワーク / DB / ホスティング
- バージョンも明記（pnpm@9, Node 20 など）

## プロジェクト構造
- ディレクトリの意味を1行ずつ
- どこに何を置くか

## コーディングルール
- 命名規則、import のルール、エラーハンドリング方針
- "肯定形" で書く

## やりとりのルール
- 作業前に必ず計画を出す
- 変更範囲が広がるときは一度止めて確認
- コミットメッセージの形式

## ファイル命名規則
- コンポーネント / ユーティリティ / テストの拡張子と置き場
```

ポイントは、**プロジェクト固有の "クセ" を書く場所** にすること。汎用的なベストプラクティスをコピペしてくるだけだと、さっきの "150〜200 の上限問題" にぶつかります。

具体例：モノレポでの CLAUDE.md 抜粋

```
## プロジェクト構造
- apps/* : 各サイト（1 domain = 1 Pages project）
- packages/ui : 共通 UI コンポーネント
- content/* : Keystatic で管理するコンテンツ

## コーディングルール
- 常にイミュータブル。既存オブジェクトは変更せず、新しいオブジェクトを返す
- 1ファイル 200〜400 行、最大 800 行
- 入力バリデーションは Zod で行う
```

具体的な制約を書くほど、Claude の出力がブレなくなります。

## CLAUDE.md は "育てる" もの

最初から完璧にしようとすると、書き始めるハードルが上がります。おすすめの育成サイクルは以下です。

1. **最小限で始める**（5〜10 行。役割 + 技術スタックだけでいい）
2. **使いながら "意図と違う挙動" をメモする**（Slack の自分宛 DM でもいい）
3. **同じ注意を2回以上した内容だけ CLAUDE.md に昇格させる**
4. **月1で棚卸しして、効いていない行を消す**

特に **3** が効きます。「1回しか起きていない事故」にまでルールを足すと、あっという間に 200 行超えの CLAUDE.md になり、結果的に全体の遵守率が下がる。

> CLAUDE.md は "禁止リスト" ではなく "契約書"。  
> 書けば書くほど強くなるわけではなく、**合意が取れてる最低限** が一番強い。

## まとめ

* Claude Code のブレの大半は CLAUDE.md で潰せる
* **短く・肯定形で・役割から書く**
* プロジェクト固有の "クセ" だけを書き、汎用ベストプラクティスは書かない
* 最初は 5〜10 行でいい。運用しながら育てる

ちゃんと書いた CLAUDE.md がある状態とない状態は、同じ Claude Code でも**別の道具**くらい違います。まだ適当に書いてる人は、今日の夕方に 30 分だけ時間を取って書き直す価値アリです。
