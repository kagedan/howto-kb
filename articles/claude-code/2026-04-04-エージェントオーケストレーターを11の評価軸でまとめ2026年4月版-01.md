## この記事について

Claude Code、Codex、Gemini CLIなどのコーディングエージェントを**複数同時に走らせる・タスクごとに隔離する・issueからPRまで自動化する**ためのツール群——「エージェントオーケストレーター（Parallel Agent Runner）」が急増しています。

この記事では [awesome-agent-orchestrators](https://github.com/andyrewlee/awesome-agent-orchestrators) 等から収集した**36以上のツール**を、以下の11の評価軸で網羅的に比較しました。

| # | 評価軸 | 見ていること |
| --- | --- | --- |
| 1 | 実行トリガー | 何を起点にエージェントが動くか |
| 2 | タスク管理連携 | GitHub Issues / Linear / Jira との連携 |
| 3 | 利用可能エージェント | 対応エージェントの種類と幅 |
| 4 | エージェント切り替え | タスクに応じた動的な使い分け |
| 5 | 並列実行・分離 | worktree隔離、同時実行、コンフリクト防止 |
| 6 | PR・レビュー連携 | 自動PR作成、レビューコメント追従、CI連携 |
| 7 | 監視・可観測性 | ダッシュボード、ログ、コスト追跡 |
| 8 | 実行中の介入・制御 | 一時停止、方針修正、追加指示 |
| 9 | 安全性・ガードレール | 承認フロー、変更範囲制限、ロールバック |
| 10 | セットアップ・運用コスト | ライセンス、ホスティング形態、導入の手間 |
| 11 | 拡張性・プロトコル対応 | MCP / A2A対応、プラグイン機構 |

### この記事の読み方について

この記事は「どのツールが優れているか」のランキングではなく、**各ツールがどの機能をカバーしているかの対応表**です。実際の品質・安定性・使い勝手を評価したものではありません。

ツール選定においては、対応機能の広さよりも**自分のワークフローに必要な機能が揃っているか**のほうが重要だと考えています。各セクションの「注目ポイント」を参考に、自分のチームの条件（使っているタスク管理ツール、主力エージェント、チーム規模など）と照らし合わせてみてください。

## 1. 実行トリガー

何を起点にエージェントが動き出すか。オーケストレーターの運用スタイルを最も大きく左右する項目です。

手動起動で十分なのか、Linearのチケットが動いたら自動で走ってほしいのか、PRへのレビューコメントをきっかけに修正を回したいのか——この違いでツールの選択肢が大きく絞り込まれます。

!

**ポイント:**  
大半のツールは「手動でタスクを作って起動する」方式です。外部タスク管理と連動して自動起動できるのは **dorothy, bernstein, 1code, lalph, symphony, tutti** の6つ。  
特に dorothy は GitHub polling / Jira / cron / Telegram / Kanban と最もトリガーが多様です。チケット駆動の開発フローに組み込むなら、この6つが候補になります。

## 補足

### 個別ツールに関する注意

* **crystal** は非推奨（Nimbalystに移行済み）
* **ghast** は厳密にはagent runnerではなくterminal multiplexer
* **multi-agent-shogun** は封建制メタファー（将軍/家老/足軽/軍師）の4階層構造が特徴。Git worktree非対応だがYAMLベースのファイル通信でAPI呼び出しゼロのコーディネーションを実現。定額制CLIサブスクリプション前提のコストモデル
* **symphony** (OpenAI) は早期プレビュー段階で情報が限定的
