Claude Code を私の業務脳として常駐させ、その Claude Code から複数の Codex に作業を振る体制を作ってみました。正直言ってめちゃくちゃ良かったので、興味ある人は是非真似してみてください！

人間が毎回すべてを説明し、AI にひとつずつ作業を頼む形ではありません。私の事業文脈、判断基準、過去の失敗、投稿ルール、ナレッジ管理の型を Claude Code に読ませておきます。そのうえで、Claude Code が「これは分けられる作業だ」と判断したものを Codex に配り、Codex が各自の作業を終えて戻ってきます。

今日のセッションでは、この構造で既存の有料記事をインデックス化し、たまっていた未処理素材の一部を並列でナレッジ化しました。直列でやると重い作業が、同時進行でかなり前に進みました。

## 全体像

この構造で大事なのは、Claude Code と Codex に別々の役割を持たせることです。

Claude Code は、私の脳に近い場所を担当します。AI駆動塾が何を扱っているのか、どんな読者に向けて発信しているのか、どの投稿フォーマットが過去に機能したのか、どの表現を避けるべきか。そういう前提を持ったうえで、今やるべき作業を判断します。

一方で Codex は、作業員として動きます。ひとつの素材を読む。記事を分類する。タグを付ける。ナレッジノート候補を抽出する。完了したら決められた形式で報告する。こうした、独立して進められる作業に強いです。

そこで Claude Code を間に置きます。人間が直接 Codex に毎回長い指示を書くと、どの素材を渡すか、どのルールを読ませるか、終わった結果をどう検証するかを、毎回人間が考えることになります。

Claude Code には、私の業務ルールと [SkillGraphs](https://note.com/l_mrk/n/ne638c6da16f3) （詳しくはこれ読んで。）読ませます。SkillGraphs は、Obsidian のウィキリンクで知識ノードを結ぶナレッジネットワークです。単なるメモ置き場ではなく、AI が必要な知識を探索できる外部記憶として使っています。仕組みの詳しい解説は上記リンクの記事にまとめています。

Claude Code はこの前提を使って、Codex 用の作業指示書を作ります。Codex はその指示書だけを読んで、自分の担当分を処理します。戻ってきた結果を Claude Code が検査し、問題があれば弾き、通るものだけを採用します。

この形にすると、人間の仕事はかなり絞られます。私は「この方向で進める」「ここは違う」「この結果は採用する」といった判断に集中できます。素材を開いて、読み、メモし、整形し、リンクを確認するような手作業は、作業員側に渡せます。

ポイントは、AI をひとつの相談相手として使うのではなく、役割を分けた小さな組織として使うことです。私は事業オーナーとして、方針を渡し、成果物を確認します。

## 用意するもの

道具は4つです。

- **Claude Code** - Anthropic のターミナル型エージェント。指揮者役
- **Codex CLI** - OpenAI のターミナル型エージェント。作業員役
- **Obsidian** - SkillGraphs を構築する Markdown ボールト
- **Google スプレッドシート** - 数値データ（X インプや note 売上等）の記録場所（これは任意。）
Claude Code は、フォルダ内のファイルを読んだり、コマンドを実行したり、複数ステップの作業を進めたりできます。私の場合、事業の記憶を読ませる中枢として使っています。

Codex CLI もファイルを読み書きできますが、今回の使い方では「独立した作業員」として起動します。Claude Code からブリーフ
