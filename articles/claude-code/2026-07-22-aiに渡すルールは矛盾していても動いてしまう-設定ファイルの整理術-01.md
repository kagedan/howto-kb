前回、[CLAUDE.md の `@` で読み込んでいたファイルが、全部セッション開始時に載っていたという話](https://zenn.dev/tamaki2026/articles/a000b6db8ab649)を書きました。

その続きです。全部載っていたせいで、**古いルールと新しいルールが同時に生きていました**。しかもエラーは一切出ません。気づいたきっかけと、整理した結果を残します。

## jQueryのルールが2か所にあった

私の環境には、jQueryの書き方ルールが2つありました。

* `skills/jquery.md` — CLAUDE.md から `@` で参照。常に全文が載る
* `jquery-responsive` — 公式のSkills形式。必要時だけ読まれる

同じ領域を扱っているので、単なる重複だと思っていました。並べて比べたら、そうではありませんでした。

|  | `skills/jquery.md`（常に載る） | `jquery-responsive`（Skill） |
| --- | --- | --- |
| セレクタ | data属性を優先 | `js-` プレフィックスクラス |
| スムーススクロール | `.animate()` を使う実装例を掲載 | `.animate()` は使わない |
| アニメーション | 記述なし | CSS transition + クラス切替 |

古いほうのファイルには、推奨パターンとしてこう書いてありました。

```
// スムーススクロール
$("[data-scroll]").on("click", function(e){
  e.preventDefault();
  const target = $($(this).attr("href"));
  $("html,body").animate({ scrollTop: target.offset().top - 80 }, 400);
});
```

新しいほうでは `.animate()` を禁止しています。つまり自分で書いた推奨コードが、自分で決めた禁止事項を踏んでいる状態でした。

**厄介なのは、常に載っているほうが古いルールだという点です。** Skillが起動しなかった場面では古い流儀で書かれ、起動した場面では新しい流儀になる。同じ案件のコードの中で書き方が揺れます。しかもどちらもエラーにはならないので、レビューで気づかない限り混ざったまま納品されます。

ルールを更新したとき、私は新しいSkillのほうだけを直して、古い `@` 参照ファイルを消し忘れていました。**消し忘れたファイルのほうが確実に読まれる。** これが一番怖いところです。

## フォームのほうは、もっとまずかった

同じ構図がフォームにもありました。

* `skills/form-builder.md` — `@` 参照。常に載る
* `contact-form` — Skills形式

こちらは矛盾より、**欠落**のほうが問題でした。

|  | `form-builder.md` | `contact-form` |
| --- | --- | --- |
| 送信後 | ページ遷移なしでサンキュー表示 | thanks.html へリダイレクト（二重送信防止） |
| バリデーション | HTML5 + jQuery の両方 | サーバー側が本体。クライアント側はUX補助 |

`form-builder.md` には、**サーバー側バリデーションの記述が一言もありませんでした**。書いてある通りに実装すると、クライアント側の検証だけで終わります。フォームとしては論外です。

`contact-form` のほうには、ヘッダインジェクション対策もハニーポットも時間チェックも書いてあります。後から整備したものです。ただ古いファイルを消していなかったので、両方が同じ重みでClaudeに渡っていました。

人間なら「古いほうは無視して」で済みます。**Claudeにはどちらが新しいか分かりません。** ファイルの更新日時を見て判断してくれるわけでもない。同時に渡された2つの指示として扱われます。

幸い実害が出る前に気づきましたが、Skillが起動しない場面でフォームを実装させていたら、サーバー側検証の抜けたものが出てきた可能性は十分あります。

## 整理の手順

やったことは単純です。

**1. 突き合わせて、どちらが正か決める**

矛盾している項目を表にして並べました。理由が明確なほうを残します。フォームの送信後の挙動なら、`contact-form` のリダイレクトには「二重送信防止」という理由があるので、こちらが正。

**2. 役割が違う部分は移す**

`form-builder.md` には、フォーム項目リスト（お名前・会社名・メール…）も書いてありました。これは実装ルールではなく**案件の仕様**です。案件ごとに変わるので、DESIGN.md へ移しました。

アクセシビリティ要件（label紐づけ、`aria-invalid`、`role="alert"`）は `accessibility.md` に統合。もともと半分は書いてあったので、足りない分を追記しただけです。

**3. 古いファイルは行ごと消す**

CLAUDE.md の `@` 参照を削除し、ファイル本体も消しました。`skills/jquery.md` と `skills/form-builder.md` の2つです。

整理後に `/context all` で確認したところ、Memory files が9ファイルから7ファイルに減り、`accessibility.md` が247→319、`DESIGN.md` が1.3k→1.4k に増えていました。移した分がちゃんと反映されています。**この確認まで含めて作業です。**

## 矛盾はゼロにできない

ここまでで一段落かと思ったら、まだ残っていました。案件のCLAUDE.md／DESIGN.md と、汎用のSkillの間です。

|  | 値 |
| --- | --- |
| CLAUDE.md（案件） | ブレイクポイント 768px / **1360px** |
| `jquery-responsive`（汎用） | ブレイクポイント 768px / **1200px**（増やさない） |
| DESIGN.md（案件） | フォント **Noto Sans JP** |
| `jquery-responsive`（汎用） | フォントは **Noto Sans JP 一択にしない** |

これは前の2つとは性質が違います。Skillは他の案件でも使う汎用ルールなので、**片方を消して解決する話ではありません**。1200pxも「Noto一択にしない」も、汎用ルールとしては正しい。

なので、優先順位を明示する方向にしました。

**CLAUDE.md 側**（常に載るので、どの場面でも効く）

```
## ルールの優先順位
案件固有の値（ブレイクポイント・フォント・カラー等）は
DESIGN.md と CLAUDE.md の記述を優先する。
Skills 側に異なる値の記載があっても、本案件では適用しない。
```

**Skill 側**（他案件でも使うので、逃げ道を書いておく）

```
- ブレイクポイントは **768px / 1200px** の2つ（増やさない）
  （※案件の DESIGN.md / CLAUDE.md に指定がある場合はそちらを優先する）
```

**両方に書くのがポイントです。** CLAUDE.md にしか書かないと、Skillを読んだClaudeが素直に1200pxで計算する余地が残る。Skillにしか書かないと、Skillが起動しない場面で効きません。

## 教訓

* **同じ領域のルールを2か所に置かない。** 置くなら、どちらが正か明示する
* **古いファイルは消す。** 残しておくと、更新したつもりの側と忘れた側が同時に生きる
* **仕様と実装ルールを混ぜない。** 案件ごとに変わるものはプロジェクト文書へ
* **汎用ルールと案件ルールの衝突は、消すのではなく優先順位で解く**
* **整理したら `/context` で確認する。** 数字が動いていれば反映されている

前回の記事と共通しているのは、**エラーが出ないから気づけない**という点です。矛盾した指示を渡しても、Claudeは止まりません。どちらかを選んで、何事もなかったように動きます。片方が古いことも、片方に重大な抜けがあることも教えてくれません。

コードならリンタやテストが拾ってくれる部分を、設定ファイルは自分で見に行くしかない。定期的に並べて読む時間を取るのが、いまのところ唯一の対策だと思っています。
