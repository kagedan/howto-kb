## どうしても気になってしまう問題

Claude Codeを長時間使い続けると、おそらくこんなことを感じたことがあるでしょう。作業を続けるほど動作が遅くなり、精度も低下し、使用量が想定以上に早く消費されてしまうのです。私はずっとモデルのせいにしていたのですが、それは間違いでした。

本当の原因は技術的なものです。長時間のセッション中、Claude Codeは同じファイルを何度も読み返し、詳細なツール出力（npm installのログ、長いgitログ、大量のgrep結果など）をコンテキストウィンドウに直接書き出します。モデルは、そのすべてに対して、毎回コストを支払っているのです。私が実際に計測した4時間にわたるリファクタリングセッションでは、ファイル読み込み全体の38%が重複していました。すべての読み込み操作の3分の1以上が、モデルがすでに確認済みのバイトを再確認する作業だったのです。

これはモデルの不具合ではありません。これはメモリアーキテクチャ上のギャップです。エージェントには「これはすでに読み込んだ」と記憶する低コストな手段がないため、再取得を行い、トークンコストを再度支払ってしまうのです。

## 私が開発したもの

私は、Claude Code向けのオープンソース（MITライセンス）CLIおよびPreToolUseフックである「Sipcode」を開発しました。これは、2つの単調で機械的な処理を行います：

1. 詳細なツール出力（grep、npm install、git log、tsc）がモデルに到達する前に、その出力を制限します。
2. 同じセッション内で変更されていないファイルが再読み込みされるのを防ぎます。

また、セッション中にClaudeが自身のコンテキスト衛生状態の統計情報を確認できるようにする15のMCPツールと、再現可能なベンチマークも同梱されています。

インストール：

```
npm i -g sipcode && sipcode init
```

リポジトリ内の固定された20タスクのベンチマークでは、ツール出力の削減率の中央値は62%です。ベンチマークはリポジトリにチェックインされているため、`sipcode benchmark` を実行して独自の数値を確認できます。

## リリース直前で危うく間違ったまま出荷しかけた部分（そしてそこから得た教訓）

これは私が最も誇りに思っている部分なので、正直な経緯をお伝えします。

私は自分の仕事で Sipcode を実際に使っていました。ドリフトレポートによると、1 つのセッションで重複読み取りにより 624,940 トークンが無駄になっていたとのことです。まったく同じセッションに対して私が実行した `proxy --stats` では、節約されたトークンはわずか 7,553 と表示されました。自分のツールが、自分のセッションについて、83倍も過小評価していたのです。

原因はこうです。セッションの途中で Sipcode をインストールすると、重複排除キャッシュは空の状態から始まるため、インストール前に Claude が参照したファイルを再読み込みしても、比較対象がありません。セッションの前半は、ツールからは認識されていなかったのです。

修正（24時間後にリリース）では、フックが最初に発火した際にアクティブなClaude Codeのトランスクリプトを走査し、トランスクリプトに記録されたファイルの内容からキャッシュを遡及的に埋めます。重要な点として、候補が受け入れられるのは、改行コード（CRLF/CRをLFに正規化）の正規化とBOMの削除を行った後でも、トランスクリプトのバイト内容が現在のディスク上の内容と一致する場合に限られます。ファイルが変更されていた場合、読み取りは変更されずに通過します。誤った重複排除は、テストカバレッジではなく、設計上排除されています。

私が得た教訓は、あらゆるメトリクスには、独立した経路を通じて計算された対立メトリクスが必要だということです。一致すべき2つの数値が一致しない場合、その乖離こそがバグ検出器となります。私のドリフトレポートと統計カウンターの数値には83倍の乖離があり、その不一致こそがバグを発見したのです。もし両者が同じ実装を共有していたなら、どちらもバグを発見できなかったでしょう。

## なぜこれがトークン数の削減よりも重要なのか

トークンの節約は「証拠」であって、「目的」ではありません。真の目標は、よりクリーンなコンテキストウィンドウを実現することです。Anthropic自身の研究報告によると、コンテキストがクリーンな状態を維持できると、品質が約29％向上し、エージェントのエラーが約40％減少します。冗長なトークンが少なくなれば、モデルは実際に重要な事柄に注意を向けることができます。コストの削減は、信頼性の向上に伴う副次的な効果に過ぎません。

これで幻覚が完全に止まるという主張はしません。止まりません。しかし、長時間のセッションを静かに劣化させる機械的な無駄の一種を取り除くことはできます。

## できないこと（正直な範囲）

* 現時点では Claude Code のみ対応（Cursor や aider は未対応）。
* セッション間の永続化は行いません。それにはメモリ管理ツールがあり、Sipcode はそれを補完するものです：セッション内の整理整頓 vs セッション間の永続化。
* 実際にファイルを再読み込みしている場合のみ、トークンを保存します。そうでない場合は、ほとんど効果はありません。
* 通常の使用ではネットワーク呼び出しはゼロです。これは、ビルド失敗を引き起こすプライバシーテストによって強制されています。すべてローカルで行われます。
