こんにちは、Webエンジニアのまさぴょんです！

Claude Codeを実案件(AI開発のPoC)で毎日使い込んでいると、遭遇する問題は大きく2種類に分かれます。

1. **事故**: 「え、いまそのファイル消した……？」系のヒヤリハット
2. **手戻り**: commitがでかすぎる、PRの説明が古い、報告が英語、発見が会話ログに埋もれて消える……系の地味なロス

この2つをまとめて解決してくれるのが **CLAUDE.mdのRule整備** です。  
そして今回のポイントは、そのRuleを**人間が書くのではなく、Claude自身に書かせる**というメタ手法🐱

## 1. なぜ「Claude自身に書かせる」のか

CLAUDE.mdは、いわばClaude Codeの**職務規程**です。ここに書いたルールは毎回のセッションで読み込まれ、Claudeの振る舞いを縛ります。

で、この職務規程、人間がゼロから書くと大体うまくいきません。理由は3つあります。

1. **抜け漏れが出る**: 危険コマンドを人力で列挙しても、`git clean -fd` みたいな盲点が必ず残る
2. **Claudeに伝わる表現がわからない**: どう書けば安定して守られるかは、正直Claude自身のほうが知っている
3. **書くのが面倒で後回しになる**: 事故ってから「ルール書いておけば……」と後悔するのがお決まりのパターン

そこで発想を変えて、**「あなたが安全に働くためのルールを、あなた自身が書いて」と依頼**します。人間の仕事は、生成されたRuleをレビューして微調整するだけ。

![Ruleを書かせるメタ手法の3ステップ](https://static.zenn.studio/user-upload/deployed-images/a0c7d5addb99d1a91669cba6.png?sha=8810a9c1b4b493dd0594a94f97518c58bc4c6a9c)

この方式のいいところは、Claudeが「自分がやらかしそうなこと」を自覚しているところです。実際に生成させると、人間が思いつかない事故パターンまで先回りして盛り込んできます。

## 2. 第1部: 安全系Rule(ガードレール)

### Rule 1: 危険コマンドのガードレール

**目的**: `rm -rf` や `git push --force` のような破壊的コマンドを、Claudeが「よかれと思って」実行してしまう事故を防ぐ。

実際に使った追加プロンプトがこちらです👇

ガードレールRule追加プロンプト

```
ultrathink
あなたが作業中に破壊的なコマンドを実行して事故が起きるのを防ぎたいです。
CLAUDE.mdに「危険コマンドのガードレールRule」を追加してください。

## 要件
- あなた自身が「これを勝手に実行されたら困る」と思うコマンドを洗い出して、
  危険度別(実行禁止 / 実行前に必ず確認)に分類すること
- 検知したときの振る舞い(停止して、目的と影響範囲を説明して、
  ユーザーの確認を取る)を定義すること
- より安全な代替手段があるなら、あわせて提案するRuleにすること
```

生成されたRuleの例(抜粋)がこちら。**人間が列挙するより網羅的**なのがわかると思います。

CLAUDE.md(生成されたRule例・抜粋)

```
## 危険コマンドのガードレール(最優先ルール)

### 実行禁止(代替手段を提案する)
- 広範囲の削除: `rm -rf /`, `rm -rf ~`, `rm -rf .`
- 履歴の破壊: main/masterへの `git push --force`
- 未追跡ファイルの一括削除: `git clean -fd`
- DBの破壊的操作: `DROP TABLE`, WHERE句なしの `DELETE FROM`

### 実行前に必ずユーザー確認を取る
- 特定ディレクトリの削除: `rm -rf <dir>`
- 作業内容の破棄: `git reset --hard`, `git checkout .`
- Docker資源の一括削除: `docker system prune`, volume削除
- 本番環境に影響する操作(deploy・インフラ変更)

### 検知時の振る舞い
1. コマンドを実行せずに停止する
2. 「何のために・何に影響するか」を日本語で説明する
3. より安全な代替手段(退避 / stash / dry-run)があれば提案する
```

![危険コマンドガードレールの仕組み](https://static.zenn.studio/user-upload/deployed-images/ef712e8d57787feacb6902e6.png?sha=9b0d5863506c96d7aa1a4b5b91bde93ff2bc5318)

**効果**: 導入後、「消しちゃった」系の事故はゼロになりました。地味に優秀なのが3番の代替提案で、「`git checkout .` の代わりに、いったん `git stash` に退避しますか？」のように、**破壊せずに目的を達成する手段**を出してくれます。

### 補足: settings.jsonとの「二段構え」

ここで大事な注意点をひとつ。**CLAUDE.mdのRuleは「ソフトガード」** です。Claudeは基本的に守ってくれますが、プロンプトの読み込まれ方次第で、すり抜ける可能性はゼロではありません。

なので本気で止めたいコマンドは、`settings.json` の `permissions.deny` にも登録して**ハードガード**にします。こちらはClaudeの判断を経由せず、ツール実行の手前で機械的にブロックされます。

.claude/settings.json

```
{
  "permissions": {
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push --force*)",
      "Bash(git reset --hard*)",
      "Bash(git clean*)"
    ]
  }
}
```

![CLAUDE.md×settings.jsonの二段構え](https://static.zenn.studio/user-upload/deployed-images/e1742534fee1aa9ac58f3817.png?sha=bf821f3847c5d2b0df5b33a55fad1902a7c8b94a)

役割分担はこうです。

| レイヤー | 役割 | 得意なこと |
| --- | --- | --- |
| `settings.json` の deny | ハードガード(機械的ブロック) | 「絶対に実行させない」の担保 |
| CLAUDE.md の Rule | ソフトガード(判断+説明) | 影響範囲の説明・代替手段の提案・グレーゾーンの確認 |

denyだけだと「ブロックされました」で終わってしまい、**なぜ危険か・どうすればいいか**が出てきません。二段構えにすることで、安全性と作業の継続性が両立します。

## 3. 第2部: 効率系Rule(自動化)

### Rule 2: 適切な単位での自動commit & push

**目的**: 気づいたら変更ファイル30個・巨大な1commitになっている問題を防ぎ、レビュー可能な粒度でcommitを積ませる。

自動commit&push Rule追加プロンプト

```
CLAUDE.mdに「自動commit & pushのRule」を追加してください。

## 要件
- 意味のある作業単位(1機能・1修正・1リファクタ)が完了するたびに
  commitすること
- commitメッセージの規約: prefix(feat/fix/docs/refactor/test/chore)+
  日本語の要約
- lint・テストなどの確認が通ってからcommitすること
- commit後は作業ブランチへpushすること
- main/masterへの直接commitは禁止(必ずブランチを切る)
```

生成されたRuleの例(抜粋)👇

CLAUDE.md(生成されたRule例・抜粋)

```
## commit & push の運用ルール

- 1つの作業単位(機能追加・バグ修正・リファクタ)が完了したら、
  ユーザーの指示を待たずにcommitする
- commit前チェック: lintエラーがないこと・関連テストが通ること
- メッセージ形式: `<prefix>: <日本語の要約>`(例: `fix: 検索結果の重複を排除`)
- commit後は現在の作業ブランチにpushする
- main/masterに直接commitしない。作業開始時にブランチを確認し、
  mainにいる場合はブランチ作成を提案する
```

**効果**: 長時間の作業を任せても、進捗がcommit履歴として残るので、**途中経過の把握と巻き戻しが簡単**になりました。「あの変更どこまで進んだ？」がgit logを見るだけでわかります。

### Rule 3: PR自動作成・説明文の自動更新

**目的**: 「pushはされてるのにPRがない」「PRの説明文が初回のまま化石化している」問題をなくす。

PR自動化Rule追加プロンプト

```
CLAUDE.mdに「PR自動作成・更新のRule」を追加してください。

## 要件
- 作業ブランチの最初のpush時に、Draft PRを自動作成すること
- PR説明文の構成: 変更概要 / 変更内容の一覧 / テスト・確認内容 / 残課題
- 以降commitを積むたびに、PR説明文を最新の内容に更新すること
- PR作成・更新には gh コマンドを使うこと
```

![自動commit&push→PR自動作成・更新のフロー](https://static.zenn.studio/user-upload/deployed-images/707e48340f743e8b1036dc62.png?sha=c4b4563c4cc9f5f8bd66104ec93df40ef2e600d2)

**効果**: このRuleとRule 2の合わせ技で、\*\*「作業を依頼したら、レビュー可能なPRが自動で育っていく」\*\*状態になります。急に別件が入って作業を止めても、PRを見れば現状と残課題がわかるので、引き継ぎ資料としてもそのまま機能します。

### Rule 4: 発見のドキュメント化

**目的**: デバッグ中に判明した外部APIの仕様、環境構築の罠、設計判断の理由……こういう「再利用可能な発見」が会話ログに埋もれて消えるのを防ぐ。

発見のドキュメント化Rule追加プロンプト

```
CLAUDE.mdに「発見のドキュメント化Rule」を追加してください。

## 要件
- 作業中に得た再利用可能な発見(外部システムの仕様・環境の罠・
  設計判断の理由・ハマりどころと解決策)は、docs/ 配下に記録すること
- 「次に同じ作業をする人(またはClaude自身)が同じ調査を繰り返さない」
  ことを判断基準にすること
- 記録したら、作業報告の中で「今回ドキュメント化した発見」として
  一覧すること
```

**効果**: これが**ドキュメントが資産になる感覚**をいちばん実感できるRuleです。数週間後に同じ罠を踏みそうになったとき、Claudeが自分で書いたdocsを参照して「この件は docs/domain/xxx.md に記録があります」と回避してくれたときは感動しました🥹

### Rule 5: 日本語報告の強制と報告フォーマット

**目的**: 報告が突然英語になる・報告の粒度がバラバラで読むのに時間がかかる問題を解消する。

日本語報告Rule追加プロンプト

```
CLAUDE.mdに「報告のRule」を追加してください。

## 要件
- ユーザーへの報告・説明・質問は常に日本語で行うこと
- 作業完了報告のフォーマット:
  結論(何がどうなったか) → 変更点 → 確認方法 → 残課題
- 結論を最初に書くこと(経緯から始めない)
```

**効果**: 報告の**読む時間が体感で半分以下**になりました。フォーマットが固定されると、報告をそのままSlackに貼って非エンジニアに共有する、みたいな使い方もしやすくなります。

## 4. 運用Tips: Ruleが守られないときに見直す3つのこと

CLAUDE.mdにRuleを書いても、守られないことはあります。そのときにルール文を増やす前に見直すポイントが3つあります。

![Rule運用の3つのTips](https://static.zenn.studio/user-upload/deployed-images/ecddebd34e9ca7fe9fcfd865.png?sha=bb59c1401bf5fb9736a3ac6f56629eea272aaf97)

### Tips 1: 配置 — 大事なRuleは冒頭に置く

長いCLAUDE.mdの後半に書いたRuleは、参照が安定しません。**ガードレール系は冒頭に「最優先ルール」という見出しで置く**のが鉄板です。逆に、CLAUDE.mdが肥大化してきたら「Ruleを重要度順に並べ替えて、重複を整理して」とClaude自身にリファクタさせます。

### Tips 2: 監査 — 守れているかをClaude自身に点検させる

たまにこう依頼します。

Rule監査プロンプト

```
今日の作業を振り返って、CLAUDE.mdのRuleに違反した行動が
なかったか自己点検してください。
違反があった場合は、原因と「Ruleの表現をどう直せば守りやすくなるか」を
あわせて報告してください。
```

面白いのは、違反の原因が「Ruleの表現が曖昧だから」と特定されるケースが多いこと。**Ruleの品質改善までClaudeにやらせる**のがこの運用のキモです。

### Tips 3: 育成 — 事故が起きたら、その場でRule化する

ヒヤリハットや手戻りが起きたら、その場でこう依頼します。

再発防止Rule追加プロンプト

```
いまの事象の再発を防ぐRuleをCLAUDE.mdに追記してください。
原因の分析と、Ruleの文面の両方をお願いします。
```

これで**失敗がそのままCLAUDE.mdという資産に変換**されます。skill-mapの記事で紹介した「地図を育てる運用」と同じ思想ですね。

## まとめ

* CLAUDE.mdはClaude Codeの職務規程。**Rule自体をClaude自身に書かせる**と、網羅的で守られやすいRuleが数分で手に入る
* 安全系: 危険コマンドのガードレールは **CLAUDE.md(ソフト)× settings.jsonのdeny(ハード)の二段構え**にする
* 効率系: **自動commit&push + PR自動作成・説明文更新**で「PRが自動で育つ」状態を作る
* 資産化: **発見のドキュメント化 + 日本語報告フォーマット**で、作業の副産物を資産に変える
* 運用: 守られないときは「配置・監査・育成」。事故が起きたらその場でRule化して、CLAUDE.mdを育てる

Rule整備は一度やれば毎日のセッション全部に効いてくる、**費用対効果が最強クラスの投資**です。まずは危険コマンドのガードレールからぜひ試してみてください🛡️
