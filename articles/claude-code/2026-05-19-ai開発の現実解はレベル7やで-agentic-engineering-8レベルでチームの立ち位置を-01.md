## はじめに ― レベル分けは組織のAI体力テストや

「ウチもAI活用してるで！」って言う人と話してて、よくよく聞いたら **タブ補完しか使ってへんかった** ってこと、結構ある。逆に、「サブエージェントを役割で分けてレビューも別エージェントに任せてる」みたいに、運用が一段上のチームもある。同じ「AI活用」でも、中身は段違いやねん。

これを **段階で揃える** ええフレームに、最近出会った。

> [The 8 Levels of Agentic Engineering — Bassim Eledath](https://www.bassimeledath.com/blog/levels-of-agentic-engineering)

元記事の主張をざっくり言うと **「現実解はレベル7、レベル8はまだ早い」**。これを自分の運用で **実費を払って検証した結論** も、ほぼ同じやった。さらに自分の独自視点として **「レベル5こそ一番危ない」** っていう警告も付け足したい。

この記事のゴールはこの3つや。

* **L7（サブエージェント＋責務分割）** を現実解として理解してもらう
* **L5の罠**（MCP整備でアウトプット速度だけ上がって、レビュアーが沈むやつ）を可視化する
* **自分の立ち位置を測る共通言語** として、8レベルを使えるようになる

## 3行サマリ ― ここだけ持って帰ってくれてもええ

* **現実解はレベル7** ― サブエージェント＋責務分割＋AI同士レビュー。人間は上流の監督に回る
* **レベル5が最危険ゾーン** ― MCP/Skillsだけ整って、品質保証が人間頼みのまま
* **レベル8はまだ早い** ― Agent Teams は将来性高いけど、現状はコスパで噛み合わへん

## 8レベル早見表 ― スキャナーはここだけ見ても帰れる

| Lv | フェーズ | 主な道具・キーワード | 効果範囲 | 自分の評価 |
| --- | --- | --- | --- | --- |
| 1 | タブ補完 | GitHub Copilot | 個人 | 入口 |
| 2 | AI統合エディター | Cursor / Windsurf / Antigravity | 個人 | 入口の延長 |
| 3-4 | エージェント＋コンテキスト整備 | `CLAUDE.md` / システムプロンプト / ルールファイル | チーム展開可 | ここで初めて組織が動く |
| 5 | MCP＋Agent Skills | MCP / Skills / 業務システム連携 | 業務全般 | **一番ヤバい所** |
| 6 | ハーネスエンジニアリング | テスト・lint・型・自動フィードバックループ | 組織 | 壁は分厚いが必須 |
| 7 | エージェント・オーケストレーター | サブエージェント／責務分割／AI同士レビュー | 組織 | **現実解** |
| 8 | 自律型エージェントチーム | Agent Teams / 横協調 | 組織 | まだ早い |

「ウチはレベルいくつ？」って会話を、上司にも部下にも投げてみてほしい。それだけでも気付きが結構あるはずや。

## なんで段階で見るんがそんなに大事なんや？

DevOpsをやってきた人にはなじみの考え方やけど、組織の能力を **段階で可視化する** ってのは Capability Maturity Model（CMM）の系譜で、リーンとDevOpsの世界でさんざんやってきた話や。それのAI版がいま整理されようとしてる。

なんでそんなに大事かって言うと、

* 「ウチはAI使ってます！」が、実は **個人のタブ補完止まり** やったりする
* 隣のチームと比べて、なにが足りてへんのか **言語化できない**
* 経営層に「次に何へ投資すべきか」を説明する **共通言語** が無い

このへんを一発で揃えられるからや。「ウチはレベル5、半年でレベル6に上げよう」って会話が成立した瞬間、AI投資の議論は別次元になるで。

## レベル1-2: タブ補完とAIエディター ― ここで止まってたら何も積み上がらん

GitHub Copilot がタブ補完で世の中をひっくり返した。続いて Cursor / Windsurf / Antigravity みたいな **AI統合エディター** が出てきて、エディター内蔵のチャットから複数ファイルにまたがる変更を提案してくれるようになった。

レベル1-2 はあくまで **個人単位** の話や。組織能力としてはまだ何も積み上がってへん。「うちのチーム、AI使ってます」って言いがちなフェーズやけど、ここで止まってる組織はスタートラインに立ってへんと思った方がええ。

## レベル3-4: エージェント化＋コンテキスト整備 ― ここから組織の話になる

ここから AI が **エージェント** として動き出す。自分でファイルを読んで、編集して、ターミナルを叩いて、テストを走らせる。ただ、生で使うとノイズだらけになるんで、

* **システムプロンプト** の整備
* **ルールファイル**（`CLAUDE.md` / `.cursorrules` 系）の整備
* **コンテキストエンジニアリング**（必要なものだけ渡す設計）

このへんが盛り上がった。**チームで配布・共有できる** ってのがミソやで。レベル1-2 は個人プレーやったけど、ここから初めて「複数人で底上げ」ができる。

ただし、精度は上がっても **AIができることの幅は広がらん**。これがレベル5への伏線になる。

## レベル5: MCP＋Agent Skills ― 主戦場やけど一番ヤバい

**MCP（Model Context Protocol）** と **Agent Skills** が、AIの **能力そのもの** を拡張する。レベル3-4 までで「どう使うか」は整理できたけど、企業が業務に組み込むには結局 **既存システムと繋がへん** とどうしようもない。

* 勤怠管理サービス
* 給与・人事系のバックオフィス
* 社内DB・ドキュメントストア
* 採用ワークフロー

こういう壁を MCP が抜いた。Agent Skills は企業のドメイン知識をAIに渡す仕組みとして整備が進んでる。

結果、

* AIができる仕事の **幅** が一気に広がった
* **エンジニア以外のビジネス職** にも恩恵が回り始めた
* 「AI、ウチの業務でも使えるやん」と気付く非エンジニアが急増

2026年5月の今、肌感では **多くの企業がここに居る**。レベル5は、いまの主戦場や。

…ただな、ここに罠があるんや。次のセクションで殴り書きするで。

## ここが一番ヤバい ― レベル5の罠を、もうちょい解像度上げて

元記事は「現実解はレベル7」って結論で、それ自体は同意。でも自分は、それより手前で **レベル5でドツボにハマる組織が一番多くなる** と踏んでる。

ポイントは「AIキャッチアップ度合いがメンバーごとに段違い」っていう事実や。

* 習熟度が高いメンバー：プロンプトを磨いて、自分で出力検証してからPRを上げる
* 習熟度が低いメンバー：AIの出力をほぼノーチェックでPRに積む

MCP/Skills をワークフローに **ガッツリ組み込んだ後** に、このバラつきを人間レビューだけで吸収しようとすると、**質の悪いものが量産されてチェック側がパンク** する。レベル5で「ウチもAI活用進んでるで！」って満足してしまわず、**ハーネス整備（L6）とサブエージェント設計（L7）をセットで仕込む** 意識がリーダー層に無いと、ここでカオスになる。

「結構、危ういのかな、と思ってる」が、自分の素直な感覚や。

## レベル6: ハーネスエンジニアリング ― AIに自分を直させる仕組み

「ハーネスエンジニアリング」って言葉、初耳の人もおると思う。簡単に言うと **モデル以外の周辺仕組みの整備全般** や。馬具のハーネスのイメージで、モデル本体（＝馬）を制御するための装具一式やと思てくれたらええ。

レベル6 でやるのは、

* AIに **成功基準** を与える（テストが通る・型検査が通る・lint が通る等）
* AIに **検証手順** を与える（自分で動作確認できるようにする）
* 失敗したら自分で原因を分析して **自律的にループを回す** 設計にする

つまり **自動フィードバックループ** が回る仕組みを作るんが、レベル6 の核心や。レベル5 までは「AIにいろいろやらせる」だけやったけど、ここから **AIが自分で改善する** ようになる。これが人間のレビュー負荷を直接的に削る一番効く打ち手や。

## レベル5→6の壁が、ほかの段差と比べて分厚すぎる件

レベル3-4 から レベル5 へのジャンプは、ぶっちゃけ **ツール導入で何とかなる**。MCP対応ツールを入れて、Skills を書いてみるところまでは、AIキャッチアップが進んでる個人なら一人で突破できる。

ところが、**レベル5 → レベル6 は別次元** や。これは独自検証で実感した。

| 移行 | 主な障壁 | 個人で突破可？ | 組織コミット必要？ |
| --- | --- | --- | --- |
| L1-2 → L3-4 | プロンプト・ルールファイル整備 | ◎ | △ |
| L3-4 → L5 | MCP・Skills 導入 | ◎ | △ |
| **L5 → L6** | **ハーネス整備・継続ブラッシュアップ** | **✕** | **◎（必須）** |
| L6 → L7 | サブエージェント設計・責務分割 | △ | ◎ |
| L7 → L8 | 自律協調・横エージェント運用 | ✕ | ◎ + コスト判断 |

レベル6 のハーネスは、一発で作って終わりやない。**プロジェクトが変わればハーネスも変わるし、新しいツールが出れば組み込み直す**。継続ブラッシュアップが要る世界や。これを個人投資でやると回収できへん。一方で、大人数で使うほど **効果が乗算で効く** から、組織として投資判断が下りないと前に進めない。

リーダー層・経営層を巻き込まんことには、レベル5 で永遠に足踏みする。

## レベル7: エージェント・オーケストレーター ― **ここが現実解や**

ここからが本記事の主役、レベル7 や。

レベル7 の核心は **エージェントのオーケストレーション**。Claude Code みたいなツールが持ってる **サブエージェント** 機能を、役割ごとに分けて使うイメージや。

### 責務を分けるってどういうことか ― 3エージェント体制の具体例

抽象的に「責務を分ける」言うても伝わりにくいんで、自分が実運用してる3エージェント体制を書いとく。

**実装エージェント**

* 役割: コードを書く・テストを通す
* 持たせる Skill: 実装観点（コーディング規約、フレームワーク作法、テストの書き方）
* 与える権限: ファイル編集（Edit/Write）、テスト実行（Bash）
* 与えへん権限: PRマージ、本番デプロイ

**レビューエージェント**

* 役割: 実装エージェントの成果物を別視点でチェック
* 持たせる Skill: コードレビュー観点（読みやすさ、保守性、セキュリティ、エッジケース）
* 与える権限: 読み取り専用（Read/Grep）
* 与えへん権限: **編集権限は意図的に外す**。レビュー専任にすることで「直しに行ってしまう」誘惑を断つ

**オーケストレーター（人間）**

* 役割: 仕様を渡す → 実装エージェントに書かせる → レビューエージェントに通す → 必要なら差し戻す → マージ判断
* 持たせるもの: 仕様書、要件、判断軸
* 一番大事な仕事: **「AIが揃ってOKと言ってるけど、ほんまにこれでええんか？」を最後に問う**

この3つを回すだけで、レベル5 で起きてた「ノーチェックPRの氾濫」がほぼ消える。レビュー作業の **第1段** を別AIが受け持つから、人間は **判断にだけ集中** できるようになる。

### 設計思想の肝 ― 「AIの自己評価の甘さ」を前提にする

ここが一番大事やから、独立して書いとく。

実装エージェントには「実装観点の最適化」だけをさせる。レビューエージェントには「批判的視点での粗探し」だけをさせる。**評価者と被評価者を物理的に分ける** から、自己評価のバイアスがかからん。

このトリックは人間社会のレビュー制度（コードレビュー、論文査読、人事評価の評価者分離）と同じや。AIにも同じ規律を当てはめるだけ。

### チームへの組み込みパス ― 個人運用から組織運用へ

L7 は「個人がローカルで使う段階」と「組織のワークフローに組み込む段階」の2段ある。

1. **個人運用フェーズ**：自分のローカル環境でサブエージェントを定義して、自分の作業フローで回す。Claude Code の `.claude/agents/` 配下に YAML/Markdown でエージェントを書くやつ。ここは1人で始められる
2. **チーム共有フェーズ**：自分で回して効いたエージェント定義を、リポジトリにコミットしてチームで共有する。`.claude/agents/code-reviewer.md` みたいなのを team-wide に展開
3. **CI/CD組み込みフェーズ**：PR作成時に自動でレビューエージェントが走る、コミット時にハーネス（L6）と連携してテスト・型・lint まで含めた検証が回る、ここまで来ると組織として完全にL7

国内外のAI先進企業のテックブログを見ると、すでに3番目のフェーズに到達してる事例が出始めてる。エンジニアは **より上流の監督ポジション** に移っていく、っていうメッセージは、いろんなとこで強く言われてる。

AIで競合に勝つには、レベル7 ぐらいの仕組み作りを **全社的に** やる段階に来てる、と感じる。

## レベル8: 自律型エージェントチーム ― まだ早いってことだけ書いとく

レベル8 は **自律型エージェントチーム**。Claude Code の **Agent Teams** 機能をイメージしてもろたらええ。複数AIエージェントが横で協調して、人間の手をほぼ介さずに一つのタスクを完遂する世界や。理論的には完成形に近い。

ただ、自分も Agent Teams を業務タスクで一通り試したけど、**現状はコスパで噛み合わへん** っていうのが結論や。複数セッションが協調するための調停・コンテキスト共有コストが、いまのモデル価格と釣り合うてへん。同じタスクを L7 のサブエージェント方式でやった方が、いまは安く・速く済む。

最終的に **すべてのタスクを L7（サブエージェント方式）に戻した**。詳しい数字や経緯は別記事で書く予定やけど、結論だけここに置いとく ―― **L8 は将来性は高いが、今この瞬間に全タスクを置き換えるのはコスパの罠** や。

将来モデルが安くなるか、Agent Teams 側の効率が上がったら一気に現実解になる可能性はある。なので **PoC・機能評価としては触り続ける価値はある** けど、本番運用の主軸に据えるのは時期尚早、というのが2026年5月時点の自分の判断や。

## ほな、次に何をすべきかを立場別に書いとく

レベル感が見えたら、次は「自分はどこに投資するか」やな。

### 個人エンジニアの場合

* まずレベル5（MCP + Skills）を確実にこなせる状態にする
* 自分が再利用したい手順を **Skill 1個** から切り出してみる
* **L7 のサブエージェント運用を、自分のローカルから始める**（`.claude/agents/` に1ファイル書くだけや）
* L6 以上はチームで動くフェーズや。一人で全部背負うな

### チームリーダーの場合

* レベル5で **止めない**。ここで止まると、半年後にレビュー破綻する
* **L7 のサブエージェント運用をチームに展開する** ― 実装/レビュー/オーケストレーターの3役分担を試す
* ハーネス整備（L6）の投資判断を **リーダー層・経営層に上げる**
* 「L5の危うさ」を可視化して、投資判断の材料にする
* AI習熟度のメンバー差を、評価ではなくフォロー体制で吸収する設計を作る

### 組織として

* **レベル7を全社目標** に据える
* レベル8 はまだ **機能評価・PoC** の段階に留める
* ハーネス整備とサブエージェント設計を **エンジニアリングの正式な投資領域** に組み込む
* 「エンジニアは上流の監督ポジションに移る」っていうキャリアシフトを、評価制度・採用要件にも反映していく

## ほな、まとめていこか

8レベルをもう一回おさらいや。

1. **L1** タブ補完
2. **L2** AI統合エディター
3. **L3-4** エージェント＋コンテキスト整備
4. **L5** MCP + Agent Skills ― **今ここが一番多くて、一番危ない**
5. **L6** ハーネスエンジニアリング ― 自動フィードバックループ
6. **L7** エージェント・オーケストレーター ― **現実解**
7. **L8** 自律型エージェントチーム ― まだ早い

最後に持って帰ってほしい3点や。

* **L7が現実解** ― サブエージェント＋責務分割＋AI同士レビュー、人間は上流の監督に回る
* **L5は危険** ― アウトプット速度だけ上がって、レビュアーが沈む
* **L8は時期尚早** ― 将来性高いけど、本番運用の主軸にはまだ早い

組織のAI成熟度を上げる優先順位は、**L7のサブエージェント運用を仕込んで、レビュー破綻を防ぐこと** から始めるのが結局一番効率がええ。「うちはAI使ってます！」止まりやなくて、**人間が監督ポジションに移れる仕組みづくり** に投資するフェーズに来てるってことやな。
