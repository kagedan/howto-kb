業務で使っている方、プライベートで相談や調べ物に使っている方など、さまざまだと思います。

私は、`Raspberry Pi` で `Claude Code`を動かし、自動でニュースやトレンドをまとめてもらっています。  
毎朝起きると、自分用の記事ができている☕  
—— そんな生活、けっこう気に入っていました。

手書きノートのOCRまで深夜に勝手に走っています。  
全部 Claude にやらせて、私は寝てるだけ。最高ですね☺️

ところが、**2026年6月15日**。  
Anthropic がこの日を境にサブスクの料金体系を変えるとの発表がありました😥

ざっくり言うと、

……え？？私の寝てる間の自動化、これ直撃では？

というわけでこの記事は、

**「自動化に Claude を使い倒している人間が、6/15改定で実際いくら払うことになるのか」を自分の環境で実測してみた**

話です。

最後に、調べる途中で思いついてしまった「対話モードで回避できるのでは？」という発想についても触れますが、

その理由も含めて注意喚起として書きます。

## 対象読者とゴール

* 対象
  + Claude Pro / Max を契約して、`claude -p`・Agent SDK・GitHub Actions などで**自動化**している人
* ゴール
  + 6/15に何が変わるのかを正確に理解する
  + 「自分の自動化、月いくらかかるの？」を見積もる物差しを持つ
  + 取れる選択肢と、その損得、そして禁じ手を知る

## 先に結論

* **対話利用は無傷**
  + チャット、ターミナル/IDEの対話的な Claude Code、Cowork は今までどおりサブスク枠内
* **痛いのは自動化勢**
  + `claude -p`・Agent SDK・GitHub Actions・SDK経由のサードパーティが、サブスクとは別枠の**月次クレジット**へ移行する
* **クレジットは Pro=$20 / Max 5x=$100 / Max 20x=$200**
  + 消費は API標準レート
  + 繰り越し不可・共有不可・ユーザー単位
* **私の自動化は実測で月$30前後**
  + 実行回数 × 実測単価で算出
  + Pro の $20 枠には収まらず、月末を待たずに枯渇する水準
* **続けるには追加コストが要る**
  + overflow（超過分を従量課金）を有効にするか、$100 枠の Max 5x に上げるか
  + $20で使い放題だった自動化が、実質有料化される
* **結論：全自動は「コストを意識する」時代へ**
  + 「サブスクで使い放題」感覚の全自動は、6/15以降は明確に見積もりが必要になる

では順番に見ていきます。

## 6/15 で何が変わるのか

公式の言い方はいろいろありますが、エンジニア目線で一番大事なのは **「対話利用」と「プログラム利用」が別の財布になる** という一点です。

| 区分 | 具体例 | 6/15以降の扱い |
| --- | --- | --- |
| 🟢 対話利用（影響なし） | claude.ai チャット、ターミナル/IDEの**対話的**な Claude Code、Cowork | これまでどおりサブスク枠内 |
| 🔴 プログラム利用（分離） | **`claude -p`**、Agent SDK、Claude Code GitHub Actions、SDK経由の3rdパーティ | **別枠の月次クレジット**を消費（APIの標準レート課金） |
| 🟡 Developer Platform API キー | 自分でAPIキーを発行して叩く分 | これまでどおり従量課金（このクレジットの**対象外**） |

新クレジットの仕様も押さえておきましょう。

* 付与額: （月次）**Pro $20 / Max 5x $100 / Max 20x $200**
* **繰り越しなし**（つまり、月末に余っても翌月へ繰越不可）
* **共有なし・ユーザー単位**（つまり、チームでプールできない）
* 使い切ったら、overflow（usage credits）を有効化していなければ **自動化はそこで停止**

正直、「サブスク払ってるのに自動化はまた別料金なの…？」という気持ち、めちゃくちゃわかります。私も最初そう思いました。

でも裏を返せば、**重い自動化を回している一部のユーザーが、定額サブスクの裏でAPI数百ドル分を消費していた**わけで、まあ分離されるよなあ、という納得感もあります。  
たぶんあなたもその「一部のユーザー」です。私もです。

## 私の自動化、これからいくらかかるのか！？【実測】

ここがこの記事の本体です。  
**実際のセッションログからトークンを集計して算出**しました。

### 自動化の規模感

中身の詳細は伏せますが、ざっくり言うと **Raspberry Pi の cron で、毎日いくつかの「情報収集して要約・整理する」系のタスクを Claude に自動で回している**——という使い方です。

Webから情報を集めて構造化したり、画像を読ませたり、それなりに重めの処理が毎日複数。モデルは `Sonnet` を使っています。

「自分の自動化が重いか軽いか」は人それぞれなので、**金額そのものより「どう測るか」と「規模に対していくらになるか」の感覚**を持ち帰ってもらえればと思います。

### 測り方

`usage` のトークン数（入力・出力・キャッシュ書込・キャッシュ読込）を集計し、**API標準レート**で金額換算します。集計には [`ccusage`](https://github.com/ryoppippi/ccusage) を使いました。

＊ Sonnet のレート（/1Mトークン、公式の料金表より）

| 種類 | 単価 |
| --- | --- |
| 入力 | $3.00 |
| 出力 | $15.00 |
| キャッシュ書込 | $3.75 |
| キャッシュ読込 | $0.30 |

### 結果：種類ごと・月ごとの実消費

まず、「手動の対話利用＋一部の自動実行」の実消費をAPI換算で、種類別・月別に出すとこうなりました。

| 種類 | 5月トークン | 5月コスト | 6月トークン（5日時点） | 6月コスト |
| --- | --- | --- | --- | --- |
| 入力 | 15,524 | $0.07 | 61,531 | $0.30 |
| 出力 | 255,911 | $4.48 | 333,799 | $5.80 |
| キャッシュ書込 | 2,954,922 | $12.03 | 1,463,263 | $5.70 |
| キャッシュ読込 | 41,876,694 | $12.40 | 44,626,667 | $13.73 |
| **合計** | **約45M** | **$28.98** | **約46M** | **$25.54** |

5月でだいたい **$29**。  
今回は次に示す自動化分の方がメインなので、本記事の最終的な計算には含みません。

### メインの自動化（cron）分だけだといくらか

私の cron は実行のたびにセッションログを消す作りにしていたので、集計から漏れていました。そこで、メインの自動化分は **「実際の実行回数 × 1回あたりの実測コスト」** で見積もります。

毎日 cron で回しているのは3本。実行回数はログから数えました。

| スキル | 1回あたり（実測） | 5月の実行回数 | 6月（5日） |
| --- | --- | --- | --- |
| 情報収集タスクA | 約 $0.50 | 31 | 5 |
| 情報収集タスクB | 約 $0.46 | 31 | 5 |
| OCRタスク | 約 $0.52 | 2（下旬開始） | 5 |

* **5月の自動化コスト ≈ $31**（31×0.50 + 31×0.46 + 2×0.52）
* **6月は3本とも毎日化したので、月換算 ≈ $44**

正直に言うと、最初は「月$200超溶けてるのでは」と思い込んでいました。が、ちゃんと重複を除いて計算し直したら **$60前後**。内、自動化分はドラマチックな金額を期待していた自分には肩透かしでしたが、**実数はこれです**。

サブスクの定額の裏で、API換算このくらいが動いていた、ということ。

### クレジット枠に収まるのか？

6/15以降、この**自動化の $31〜44/月だけ**が新しいクレジット枠から引かれます（手動の対話利用は従来どおりサブスク枠で無傷）。各プランのクレジットと並べるとこうです。

![chart-1-vs-credit.png](https://static.zenn.studio/user-upload/deployed-images/361ba947d3721faa54e699f7.png?sha=12c5eea2d8005c393ec51d3d0835741cdfa91858)  
*私の自動化（推定 約$31/月）を、各プランのクレジット枠と並べたところ（当然の図だが。）*

| プラン | クレジット | 私の自動化（$31〜44）は収まる？ |
| --- | --- | --- |
| Pro ($20) | $20 | ❌ **月末を待たず枯渇**（私は今ここ） |
| Max 5x ($100) | $100 | ✅ **十分収まる** |
| Max 20x ($200) | $200 | ✅ 余裕（むしろオーバースペック） |

私は今 **Pro** なので、6/15以降は自動化が月末まで持ちません。続けるには、

* **overflow（超過分を従量で払う）を有効にする**
* **Max 5x に上げる**

の二択。  
いずれにせよ、

**$20で“使い放題”だった自動化に、これからは追加のコストがかかる**

——これが実測から出た答えです。

### 学び：トークン量とコストは一致しない

ログを集計していて、ひとつ面白い発見がありました。

**トークン数で見ると、その9割以上（5月は93%）は「キャッシュ読込」です。**

Claude Code は会話が進むほどコンテキストが膨らみ、毎ターンそれを読み直すので、キャッシュ読込トークンが大量に積み上がります。

ところが——**コスト（金額）の内訳は、トークン数とは比率が違います。**

![chart-2-token-vs-cost.png](https://static.zenn.studio/user-upload/deployed-images/9a44f587ad0b2a21e9572786.png?sha=f4ee3a573bd8910586a4ebc0d9e30e85ce3f1ac2)  
*同じ消費を「トークン量」と「コスト」で見ると、内訳が変わる。*

* キャッシュ読込は単価が安い（入力の0.1倍）ので、**トークンの93%を占めるのにコストでは43%**
* 逆にキャッシュ書込は**トークンでは7%なのにコストでは42%**
* 出力に至っては**トークンの1%未満なのにコストの15%**

> **「量が多い＝高い」ではない。**「少ないけど単価が高いもの（出力・キャッシュ書込）」が、地味にコストを作っています。

じゃあ最適化はどうするか。  
結論はシンプルで、**会話（セッション）を短く保つ**こと。  
セッションが伸びるほど、キャッシュの読み書きも出力も全部積み上がるからです。  
具体的には、

* **1タスク1セッションで完結させる**
* **不要なMCPサーバー・巨大なコンテキストを最初から減らす**
* **LLMでやる必要のない処理（単純なスクレイピング等）は、Claudeに渡す前にスクリプト側で済ませる**

「出力だけ削ればいい」と思いがちですが、効くのは**セッション全体を軽く・短くする**ことでした。

## 取れる選択肢を比べてみた

破滅的な額ではないとはいえ、Pro の $20 枠は超える。現実的な打ち手を並べて比較しました。

| 選択肢 | 概要 | 向くケース | コスト感 |
| --- | --- | --- | --- |
| A. クレジット内に最適化 | 実行頻度・モデル・出力・セッション長を絞る | 自動化が軽め | 追加 $0 |
| B. APIキー従量へ逃がす | 自動化だけ Developer Platform API に移行 | 重い自動化／コストを可視化したい | 従量（青天井注意） |
| C. overflow を有効化 | クレジット超過分を従量で許可 | 月によって波がある | 変動 |
| D. ローカル/他モデル併用 | 軽いタスクはローカルLLM等へ | コスト最優先 | ほぼ $0 |
| E. 対話運用へ戻す | 自動化をやめ、手で Claude Code を叩く | 自動化が趣味の範囲 | 追加 $0 |

私の方針は **A（最適化）をベースに、重い処理だけ D（LLM不要な部分の切り出し）** です。具体的には、

* 情報の「収集」部分（単純なスクレイピングなど）はそもそもLLM不要なので、Claudeに渡す前にスクリプト側で済ませる
* セッションを毎回新規で立てて短命にする（コンテキストの肥大を防ぐ）
* そもそも「毎日AIを回す必要があるタスクか？」を棚卸しして、頻度を落とせるものは落とす

## 【やってはいけない】「対話の皮をかぶせれば回避できるのでは？」という発想と、その結論

……ここまで真面目に計算してきましたが、調べている途中で、私は良くない考えを思いついてしまいました。**何度も書いていますが、これは公式の利用規約違反です。絶対にやらないでください。**

なぜダメなのかも含めて、注意喚起として共有します。

---

...  
さて、私はこれまで Pro プラン（月$20）で生活してきました。  
自動化も対話利用もまとめてサブスクの利用枠から消費されていたので、「使い放題」の感覚でいました。

ところが6/15以降は、その自動化が別枠の月次クレジットに移ります。

Pro のクレジットはたったの $20。私の自動化は月$31〜44なので、**月末を待たずに枠を使い切って、自動化が止まる**計算です。このまま上に記載した対策をしないのも方法の一つです。ただ、その場合、続けるには overflow（超過分を従量で払う）を有効にするか、$100 クレジットの Max 5x に上げるしかありません。

**$20で“使い放題”だったものに追加で金を払う**

——この「これまでタダだったのに」感が、なんとも惜しい。  
しがないサラリーマンエンジニアとしては、できれば1円も増やしたくない。なんとかならないでしょうか？

ここで、6/15のルールを思い出すと、

> 🟢 **ターミナルの対話的な Claude Code は影響なし**

とあります。  
これを見て、悪い考えがよぎりました👿👿👿👿👿👿。

「**`claude -p` を使わず、対話モードを自動で叩けば、文面上はクレジット対象外で回避できるのでは？**」

と。

そう、**`claude -p`（ヘッドレス）でも Agent SDK でもなく、tmux の中で「対話モードの Claude Code」を起動して、`tmux send-keys` でスラッシュコマンドを流し込むという裏技**です。

```
# ざっくりこういうことをやっている
tmux new-session -d -s alternative
tmux send-keys -t alternative "claude --model sonnet --dangerously-skip-permissions" Enter
sleep 2
tmux send-keys -t alternative "/my-skill" Enter   # ← 人間が打つ代わりに送り込む
...
...
tmux kill-session -t alternative # ← いい感じのタイミングで落とす
```

**でも、これはダメです。やらないでください。**

調べてみたら、これは解釈の余地なく **Anthropic の利用規約（Consumer Terms）違反**でした。規約の第3条に、こう書かれています。

> Except when you are accessing our Services via an Anthropic API Key or where we otherwise explicitly permit it, [you must not] access the Services through automated or non-human means, whether through a **bot, script, or otherwise**.  
> （訳：APIキー経由など明示的に許可された場合を除き、ボットやスクリプトなど自動的・非人間的な手段でサービスにアクセスしてはならない）

tmux で `send-keys` を流し込むこの方式は、**まさに「スクリプトによる自動アクセス」そのもの**。しかも私が使っているのは API キーではなく**サブスク認証**なので、「APIキー経由」の例外にも当てはまりません。グレーではなく、**ど真ん中でアウト**です。

なので、私の結論はこうです。

私自身も、**素直に正攻法へ寄せます**。追加の数十ドル（または overflow の従量分）は確かに惜しいですが、規約違反でアカウントを飛ばすリスクと天秤にかければ、比べるまでもありません。ルールを守って使うのが、結局いちばん安いです。

> ※利用規約の解釈は時期・地域・プランで変わり得ます。最終的には[Anthropicの公式ポリシー](https://www.anthropic.com/legal/aup)を自分で確認してください。いずれにせよ、課金回避を狙った自動化は推奨されません。

## ハマりどころ・注意点

最後に、6/15前後で踏みやすい落とし穴を共有します。

* **クレジットは繰り越し不可。** 月末に余らせると損、月初に使い切ると自動化が止まる。利用ペースは月内で平準化を。
* **ユーザー単位でチーム共有不可。** 「チームで $200 をプール」はできない。組織運用の人は要注意。
* **overflow（usage credits）を有効化し忘れると、ある日突然 cron が無言で死ぬ。** クレジットが尽きた瞬間、自動化は黙って止まります。アラートを仕込んでおくと安全。
* **「対話 vs ヘッドレス」の境界を勘違いしやすい。** IDE上で手で動かすぶんは対話、スクリプトから叩くぶんはヘッドレス。ここの線引きを間違えると見積もりがズレます。

## まとめ：6/15までにやることチェックリスト

「使い放題だから」と何でも自動化していた時代から、**「これは自動化する価値があるか？」をコストで問い直す時代**になった、とも言えます。個人的にはこれ、悪い変化だとは思っていません。寝てる間にいくら使っているかも分からないまま回し続けるより、**自分の消費を把握して使う**ほうが、よっぽど健全です。

それでは、良い自動化ライフを。
