## 1. はじめに

最近は.claudeとか.agentsとかリポジトリ内にエージェント用のフォルダを用意するのが当たり前になってきました。  
そして色々と課題が出てきていると思います。

* 複数リポジトリにまたがってskillsを管理するのがめんどくさい
* 外部ソース(skills.shなど)からskillsをinstallし、日々メンテナンスする手間
* エージェントの提供ベンダー独自のフォルダ構成、ファイル命名規則への対応

#### そんなときに役立つのが`apm`です。

## 2. apmとは

**Agent Package Manager**の略称であり、エージェント関連ファイル群をパッケージとして定義し、単一コマンドでインストール・管理できるツール。

* `npm install`や`pip install`のような感覚でエージェント設定ファイルを扱える
* 外部リポジトリやURLからパッケージを取得
* 複数リソース（ローカル・GitHub・レジストリ等）をまたいでファイルを一括配置

## 3. さっそく使い方の説明

下記の手順通りに進めれば最速で`apm install`を体験できます。

### Step2. エージェント関連ファイルを管理するためのリポジトリを作成（ここではshared-agent-packageというリポジトリを作成）

#### shared-agent-packageリポジトリに下記のようなフォルダ構成でファイルを作成する

リポジトリ内のディレクトリ構造：

```
.
├── .apm
│   ├── agents
│   │   └── sample-agent.agent.md
│   ├── instructions
│   │   └── sample.instructions.md
│   └── skills
│       ├── git-commit
│       │   └── SKILL.md
│       └── grill-me
│           └── SKILL.md
└── apm.yml
```

apm.yml：

```
name: shared-agent-package                       # 必須　：パッケージ識別子
version: 1.0.0                                   # 必須：パッケージバージョン
description: Shared skills for spike projects    # 任意　：パッケージ情報
```

* `agents`
  + `.apm/agents/${subagent_name}.agent.md`の形式でサブエージェントファイルを保存しておく
  + 公式ドキュメントは[こちら](https://microsoft.github.io/apm/producer/author-primitives/instructions-and-agents/#agents)
* `instructions`
  + `.apm/instructions/${instruction_name}.instructions.md`の形式でプロンプトやコマンドファイルを保存しておく
  + `.apm/prompts/${prompt_name}.prompt.md`にプロンプトファイルを保存しておくというやり方もあるが、今回は配布先のプロジェクトでclaudeを使う想定のため割愛
  + 公式ドキュメントは[こちら](https://microsoft.github.io/apm/producer/author-primitives/instructions-and-agents/#instructions)
* `skills`
  + `.apm/skills/${skill_name}/SKILL.md`の形式でスキルファイルを保存しておく
  + 公式ドキュメントは[こちら](https://microsoft.github.io/apm/producer/author-primitives/skills/)
* `apm.yml`
  + .apmフォルダをひとつのパッケージとして認識させるための設定ファイル

### Step3. エージェント関連ファイルのinstall先リポジトリを用意する（今回はspike-apm-projectというリポジトリを準備する）

#### spike-apm-projectリポジトリにapm.ymlを作成する：

```
name: spike-apm-project                  # 必須　：パッケージ識別子
version: 1.0.0                           # 必須　：パッケージバージョン
description: spike apm project           # 任意　：パッケージ情報
targets:                 # 必須：展開先エージェントフォルダ（今回は.claudeに配布）
  - claude
dependencies:
  apm:
    - kub-otake/shared-agent-package             # 必須　：パッケージ識別子
    - anthropics/skills/skills/frontend-design   # 必須　：パッケージ 
    - microsoft/apm-sample-package#v1.0.0        # version pinも可
includes: auto                    # 任意　：パッケージのインクルード設定
```

## 5. 付録

筆者が検証中に躓いた話を付録として書いておきたいと思います。  
躓いたポイントはズバリ、YAMLの拡張子です。  
`apm.yaml`という拡張子で検証し始めたせいで1日溶かしました。  
どうやら、`apm.yml`のように`yml`形式じゃないとだめなようです。  
だったらyamlも対応してくれよと思ってコミットを試みましたが、すでに`yml`形式のみにするという整理になっているようでした。  
無念。。。

* `.yml` and `.yaml` are equally valid YAML extensions. GitHub Actions uses `.yml`, Docker Compose uses `.yaml`. There's no industry convention to align with.
* The lockfile uses `.lock.yaml` for IDE syntax-highlighting reasons specific to lockfiles -- that rationale does not transfer to the manifest.
* `apm.yml` is now the canonical, documented manifest filename in this project and will not change. This is recorded in `docs/src/content/docs/reference/manifest-schema.md` (and will be reinforced if the doc is unclear).
