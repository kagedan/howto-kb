カスタムGPTで調査させた結果を、別のチャットにコピペして渡したことはないか。  
エージェントが出した設計書を、自分でまとめ直して、次のエージェントに貼り付けた。  
そのAI同士の橋渡しを、人間がやっている。

それ、2026年の作業じゃない。

## エージェントは互いを知らない

なぜそうなるか。各エージェントは「自分が何をできるか」を互いに知らないからだ。誰に何を渡すべきか判断できない。だから人間が間に入って中継する。

専門家を何人も雇って、その人たちの仕事を自分でコーディネートし続けているようなものだ。専門家を使う意味が半減している。

だから、エージェントを「業務をリレーできるシステム」として設計する必要がある。各エージェントが自分の役割を宣言し、スコープ外のタスクが来たら適切なエージェントに渡す。人間は中継ではなく承認者になる。これがマルチエージェントシステムの本来の姿だ。

そのために必要なのは2つだ。**やること/やらないことの定義（境界）** と、**誰に/どう委託するかの定義（ルーティング）**。この両方が揃って初めて、エージェントは「知りません」で止まらず、業務プロセスを自律的に執行できる。

## 繋げようとすると、今度は別の問題が起きる

では、エージェントを繋いで自律的に動かせばいい。ただし、それをすると別の問題が起きる。

「このエージェントは調査だけ」と決めたつもりでも、実際はそうならない。「調査してくれ」という指示は制約ではなく、依頼だ。エージェントは渡されたコンテキストで自然な補完をするため、調査の途中で「実装まで書いた方が親切だ」と判断してやる。止める仕組みがない。

調査エージェントが「余計に」書いた設計書を、実装エージェントが「正しい前提」として読み込む。エージェントAの判断ミスがエージェントBの前提になり、コードが積まれる。人間が中継をやめると、今度は誰も頼んでいない判断が積み上がる。

人間がルーターになるか、エージェントが暴走するか。どちらに転んでも詰まる。

## ロールプロンプトでは解決しない

「あなたは調査専門エージェントです」というプロンプトを渡しても、タスクの途中で状況が変われば、エージェントはその宣言より目の前のコンテキストを優先する。これはバグではなく、LLMが自然な補完として動いている結果だ。

各エージェントは互いの行動を見ていない。全体の分業マップを持っているのは人間だけで、エージェント自身は「自分がどの範囲を担当しているか」も「他のエージェントが何をできるか」も知らない。

「エージェントのロールプロンプトが効かない」という経験は、ロールプロンプトの問題ではなく、**エージェントに機械可読な自己紹介がない**という構造的な問題だ。

## AgentCardで「自己紹介」させる

A2A Protocol（Google発、Linux Foundation傘下のオープンソース仕様）の`AgentCard`を個人エージェント環境に転用する。

`AgentCard`はエージェントの能力・スキル範囲・入出力の型を機械可読な形で宣言するJSONファイルだ。プロジェクトルートの`.well-known/agent.json`に置き、`CLAUDE.md`から参照させる。

実際に使っているTARSエージェント（Telegram経由でユーザーと対話する意図構造化エージェント）の`agent.json`:

```
{
  "name": "TARS",
  "version": "0.1.0",
  "description": "Human Intent → Agent Network 変換レイヤー。ユーザーの曖昧な意図を構造化し、適切なエージェントに接続する。",
  "skills": [
    {
      "id": "intent-structuring",
      "name": "Intent Structuring",
      "description": "曖昧なアイデア・概念・問いを、命名・分類・設計フレームワークに変換する。",
      "inputModes": ["text"],
      "outputModes": ["text/markdown"],
      "examples": ["URLを渡すと技術仕様を構造化したサマリーを返す", "タスクの意図を整理して委譲ドキュメントに変換する"]
    },
    {
      "id": "human-to-agent-routing",
      "name": "Human to Agent Routing",
      "description": "ユーザーの生の意図をAgent対応の構造化タスクに変換し、適切なエージェントへのルーティング・委譲を行う。",
      "inputModes": ["text"],
      "outputModes": ["application/json"]
    }
  ]
}
```

`CLAUDE.md`に1行追加する:

```
あなたのスキルは .well-known/agent.json に定義されている。それ以外のことはやらない。
```

最小テンプレート:

```
{
  "name": "エージェント名",
  "skills": [
    {
      "id": "skill-id",
      "description": "何をするか。何をしないかも含めて書く。",
      "inputModes": ["text"],
      "outputModes": ["text/markdown"]
    }
  ]
}
```

## 何が変わるか

スキル宣言後のTARSは、実装リクエストに対してこう返すようになった:

> 「これは自分のスキル外だ。実装担当エージェントBに渡す。目的: 〇〇の機能実装。前提条件: この設計書。委譲ドキュメントを /docs/handoff.md に置いた。」

「できません」ではなく「**誰に・何のために・どこへ**」を返す。

人間のオーバーヘッドは承認ボタン1つになった。コピペ中継から解放された。

そしてもう一つ重要なことがある。`agent.json`に宣言されたスキルは機械可読だ。**別のエージェントがそれを読める**。エージェントAが「このタスクは誰に渡すべきか」を判断するとき、エージェントBの`agent.json`を参照できる。記憶の引き継ぎ、セッション間の情報受け渡し、エージェント間プロトコル——その土台が、この宣言ファイル1つから始まる。

まだエージェント同士で情報をやり取りしていないなら、まずここから始めてほしい。各エージェントに`agent.json`を持たせること。それがAI同士の中継を人間がやらなくて済む、最初の一歩だ。
