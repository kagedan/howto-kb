## ある夜、私は「何かがおかしい」と気づいた

その日は朝からClaude Codeを開いていました。ちょっとチャットして、数行コードを直して。お昼になっても同じターミナルの中で、ちょっと厄介なバグと格闘していました。ファイルをもう少し読んで、こっちの方向で試して、ダメで、戻って、別の方向で試して。夕方にはほぼ片付いて。夜になってふと `/cost` を走らせた瞬間、私は呆然としました — Max $200のクォータがもう尽きかけていたのです。*「あれ？そんなに使った覚えないんだけど…」*

**先に結論:** Claude Codeがトークンを食うのは、Enterを押すたびに会話履歴すべてが毎回頭から送り直されているからです。コストは会話の長さに比例ではなく、指数関数的に膨らみます。この記事ではその仕組みを説明したうえで、具体的な6つの習慣（タスクごとにターミナルを分ける、能動的な `/compact`、必要な部分だけ渡す、CLAUDE.mdを圧縮する、`.claudeignore` を使う、`/cost` を確認する）を紹介します。追加ツールを入れなくても、1日に数ドルは削れるはずです。

もし同じような感覚を味わったことがあるなら、この記事はあなたのためのものです。「使い方が悪い」わけじゃありません。問題は、Claude Codeが見た目からは想像もつかないような仕組みで動いている、という点にあります。

多くの方が「送ったメッセージ1通ごとに独立して課金される」と思っているみたいです。LINEのメッセージみたいに、100通なら50通の倍かかる、みたいな感じで。でも実はそうじゃありません。Claudeがセッション中のやり取りを「覚えている」ためには、**Enterを押すたびに、会話履歴まるごとが再送されている**のです。読み込ませたファイルの中身、走らせたツールの出力、貼り付けた長いログ — それ全部が毎回パックで送られています。

だから会話が長くなるほど、毎ターン送られる「荷物」はどんどん膨らんでいきます。Claudeは送られるデータ量（トークン、まあ文字数みたいなものですね）で課金する仕組みなので、50ラウンド続くデバッグセッションだと、*Enterを押すたびに* 5万〜10万トークン送っていることも珍しくありません。それがターンごとに積み重なる。だから「小さな」タスクのつもりが、気づかないうちに数十ドル飛んでいる、なんてことが起こるわけです。

安心してください — この6つはどれも、追加コストなしで今日から始められます。新しいツールを買う必要もないし、新しいワークフローを覚える必要もありません。セットアップに30分もあれば十分。でも積み上げれば、1日数ドル、1年で軽く1000ドル単位の節約になります。簡単なものから順に紹介していきますね。

## 習慣1 — タスクごとにターミナルを分けて、context rotを避ける

私、こういう状況によく遭遇するんです。朝、認証バグを直すためにターミナルを開いて、無事に片付ける。で、午後も同じターミナルを使い続けて、今度は課金機能の実装をやっている。2つのタスクは何の関係もありません。でもClaude Codeからすると、朝の会話はそっくりそのまま残っていて、午後のメッセージにくっついて毎回送られています。とっくに終わった認証関連の話を、Claudeに「覚えさせる」ためだけに、お金を払い続けているんです。

さらに悪いことに、古いコンテキストはClaudeを変な方向に引っ張ります。履歴に認証ミドルウェアの話が出ているせいで、課金機能について聞いたのに、認証を絡めた提案を出してきたりする。コンテキストが「汚染」される — 私はよくこう呼んでいます。

解決策は驚くほどシンプルで、**新しいタスクは新しいターミナルで**。終わったら閉じる。前のタスクを引きずらない。もし「修正→エラー→修正→エラー」のループに5回以上ハマっていると気づいたら、それは合図です。ターミナルを閉じて、新しく開きましょう。壊れかけたセッションにトークンを注ぎ込み続けるほうが、クリーンな状態でやり直すより高くつくことがほとんどです。

コンテキストが腐っていないか、手軽に確認する方法もあります。セッション開始時と終了時に同じ質問をしてみるんです。たとえば *「PostgreSQLとSQLite、どっちを使うと決めたんでしたっけ？」* とか。回答が違っていたら、それはClaudeが忘れ始めて混乱している証拠 — 界隈では *context rot*（コンテキストの腐敗）と呼ばれるやつです。さらに進むと、Claudeは数時間前にわざわざ削除した変更を、自信満々にもう一度提案してきたりします。ここまで来ると `/compact` でも救えません。新しいセッションを開いて進むのが正解です。

## 習慣2 — `/compact` を早めに使ってClaude Codeのトークンを減らす

Claude Codeにはauto-compactという自動機能があって、コンテキストが残り20%になると自動でこれまでの履歴を要約してスペースを空けてくれます。便利そうに聞こえますよね？でもその時点ではもう手遅れなんです。

なぜ手遅れなのか。残り20%に到達した瞬間、3つのことが同時に起こっています。**1つ目**、1ターンあたりのコストはとっくに跳ね上がっています — コンテキストが80%まで膨らんでいるということは、Enterを押すたびに巨大なデータ塊の料金を払っているということ。auto-compactしたところで、払った分は戻ってきません。**2つ目**、auto-compactはClaude Codeの圧縮パイプラインのなかで最も重い層で、裏で別のAIを呼び出して会話全体を要約させます。それ自体がそれなりに高い。自分で早めに `/compact` を叩けば、もっと軽くて安い層で済みます。**3つ目**、これが一番厄介なんですが、Claudeの回答品質はコンテキスト60〜70%あたりから落ち始めます。20%まで待ってはくれません。そのあたりからClaudeはもうちょっとぼんやりしていて、古いファイル構造を参照したり、最初に置いた制約を忘れたりします。20%に到達した頃には、「ちゃんと働けるゾーン」をとうに過ぎてるんですね。

というわけで私のルールは、**60%で赤信号**。60%に触れたら一度立ち止まって、`/compact` か `/clear` を検討する。いまの位置は `/context` コマンドで確認できます。

ここで、よく混同される3つを整理しておきます。`/compact`、`/clear`、そして新しいセッションを開くこと。似ているようで目的が違います。`/compact` はデバッグの途中で、積み上げた知識を残したいとき — 決定事項や進捗を保持したまま要約してくれます。`/clear`（あるいはターミナルを閉じて開き直す）は、あるタスクが完全に終わって、まっさらから始めたいとき。`--fork-session` はもうちょっと複雑なので、第2回まで取っておきます。

もう1つ、あまり知られていない小技があります。`/compact` には「これは残してね」という指示を添えられます。何も言わないとClaudeが勝手に判断するんですが、残念ながら*最新のもの*を優先して*重要なもの*を優先しない傾向があります。なので私はよくこんな風に書きます。

```
/compact Keep: キャッシュにはRedisを使っている（Memcachedではない）、
/api/orders の500エラーはまだ未修正、
src/auth/* を編集中、src/billing/* はそのまま触らない方針。
```

compactが終わったら、\*「いまどこまで進んでいて、次は何をするんでしたっけ？」\*と一言確認して、大事なものが抜け落ちていないかを確認するのがおすすめです。

## 習慣3 — 必要な部分だけ渡す。ファイル全体をコンテキストに貼るのはやめる

Claudeにコードを聞くとき、とりあえずファイル丸ごとコピペしたくなりますよね。気持ちはよくわかります — 私も昔はそうでした。でも、もしそのファイルが2,000行あったら、あなたは2,000行をコンテキストに突っ込んで、以降毎ターン2,000行を送り続けることになります。実際には、注目している関数まわりの30行があれば十分なのに。

このテクニックは地味ですが、多くのタスクで50〜80%のトークンを削れることが多いです。やり方は簡単で、`Read src/components/Form.tsx` で2,000行を一気に取ってくるのではなく、`Read src/components/Form.tsx offset=120 limit=50` で関数まわりの50行だけ取ってくる。私の定番ワークフローは、先にGrepで対象関数の行番号を特定して、そのあとピンポイントのoffsetでReadする、というものです。

もっと単純な方法もあります。**ファイルの中身を貼らずに、パスだけ渡す**。`config.py` まるごと貼るのではなく、\*「`src/config.py` を見て構造を把握してから…」\*と言うだけ。Claudeは必要なときに必要な分だけ自分で読んでくれます。あなたは節約できるし、Claudeもちゃんと仕事ができる。

これはエラーログにも当てはまります。バグを踏んだとき、保険で500行のスタックトレースを丸ごと貼りたくなりますよね。でもデバッグのヒントはだいたい最初の10行 — 例外とその直後の数行 — にあります。残りの490行は大半がライブラリ内部へのトレースで、ほぼ役に立ちません。私はいつも、例外本体と直後5〜10行だけ貼って、\*「必要なら `/tmp/error.log` にフルログがあります」\*と添えるようにしています。本当に必要ならClaudeが自分で読みに行ってくれますが、8割のケースでそれは不要です。結果、トークンは95%削れても、バグはちゃんと直せます。

何度も貼りすぎて学んだ原則はこれです。**最初のフィルタはあなた自身**。自分のトークンを使って、ゴミの選別をClaudeにやらせないこと。

## 習慣4 — CLAUDE.mdを最適化する: 短く圧縮して200行以内に収める

CLAUDE.mdというファイルはちょっと特別で、Claude Codeはこれを毎セッション例外なく読み込みます。つまり、ファイルが2,800トークンあって、1日に20セッション開くとしたら、実際の仕事を1文字もしないうちに56,000トークンが「燃え尽きている」わけです。これはClaude Codeを開くたびに毎回上乗せされる固定費です。

幸いなのは、固定費だからこそ一度最適化すればずっと効くということ。履歴書と同じで、1回きちんと作れば何度も使い回せます。

構造としては、私は3層にはっきり分けています。*グローバル*層は `~/.claude/CLAUDE.md`、これは私の仕事の仕方全般 — コミュニケーションの好み、よく使うツール、全プロジェクト共通のもの。*プロジェクト*層は `<project>/CLAUDE.md`、こっちはそのプロジェクト固有の情報 — ディレクトリ構成、ワークフロー、スキルのトリガーなど。そして狭い文脈でしか効かない条件付きルールは、別途 `.claude/rules/` に置きます。いちばん大事なのは、**プロジェクトAの情報をグローバルに書かない**こと。そうしないとB、C、Dにも漏れ出します。

内容については、何度か見直して削ってきた結果、いくつか原則がまとまりました。まず、箇条書きは散文よりほぼ常に安上がりです。同じ内容でも `Stack: React 18 + TS 5` のほうが *「私たちはコンポーネントモデルが良いのでReact 18を使っていて…」* よりずっと軽い。次に、**Claudeが学習データですでに知っていることは書かない**。TypeScriptが型安全なこと、Gitがバージョン管理ツールなこと、Dockerが何か — これは書くだけトークンの無駄です。Claudeが**自力で推測できない**ものだけ残す — プロジェクト固有の決定、制約、独自の慣習、そういうやつです。3つ目、機械でチェックできるルール — 命名規則やコミットフォーマット — はlinterやCI/CDに任せて、CLAUDE.mdに詰め込まない。最後に、ファイルは200行以内にして、*必ずやること*と*やってはいけないこと*は最初の50行に集約する。長すぎるとClaudeは流し読みになって、重要なルールを見落とします。

小さな例を。以前私が書いていた実際のCLAUDE.mdの一部です。

```
Our project uses TypeScript because it provides type safety which helps
catch bugs early. We use React 18 for our frontend framework because of
its component model. For testing, we prefer Jest because it integrates
well with our CI pipeline...
```

圧縮後。

```
Stack: React 18 + TS 5 + Vite + Jest
Commits: Conventional Commits
Tests: co-locate with component, suffix .test.tsx
```

2,800トークンから950トークンへ — 66%減。1日20セッションなら、どれだけ節約になるか想像がつくと思います。

ついでに、非常に安定した情報（使っているPythonのバージョン、ツールチェーン、APIのbase URLなど）を何度も繰り返したくない場合は、`/memory` コマンドが置き場所として便利です。一度書けばClaudeはずっと覚えてくれます。ただしCLAUDE.mdと同じキャッシュのルールが効くので、頻繁に書き換えないほうがいいです — なぜそうなのかは第2回で説明します。

## 習慣5 — `.claudeignore` で不要ファイルをClaude Codeのコンテキストから外す

リポジトリには、作業のためにClaudeが読む必要のないファイルがたくさんあるはずです。ビルド成果物、lockファイル、自動生成コード、画像、スナップショットなど。これらは推論に何も貢献しないのに、トークンだけ消費します。`.claudeignore` は「これらは無視してね」と事前にClaudeに伝える仕組みです。

文法は `.gitignore` と同じなので、すぐ書けます。新しいプロジェクトで私がよくコピーする初期テンプレートがこちら。

```
# Build artifacts
dist/
.next/
build/

# Lock files
package-lock.json
yarn.lock
pnpm-lock.yaml

# Generated code
*.generated.ts
prisma/migrations/

# Media
*.png
*.jpg
*.svg
*.pdf

# Test snapshots
__snapshots__/
*.snap

# Logs
*.log
logs/
```

中規模のTypeScriptプロジェクトだと、これくらい無視するだけで1セッションあたりClaudeが読むファイル数が45から18まで減り、コードベース探索フェーズで62%のトークン削減になります。

あまり知られていないヒントもいくつか。`prisma/migrations/` はスキーマ把握のために1ファイルずつ読ませる必要はなく、`schema.prisma` だけで十分です。長大な `CHANGELOG.md` も日々のコーディングには要らない。`__snapshots__/` は自動生成ファイルばかりで、直接レビューすることはほぼないでしょう。

1つ注意。`.claudeignore` と `.gitignore` は、文法こそ同じでも別物です。`.gitignore` は *「gitで追跡しなくていい」*、`.claudeignore` は *「Claudeに読ませなくていい」*。gitでは追跡したいけどClaudeに読ませる必要はないファイルだってあります — `CHANGELOG.md` なんてその典型です。

最後に、プロジェクトが成長したら `.claudeignore` も更新しましょう。ある日Storybookを入れたら、`storybook-static/` を足しておく。ゴミが溜まっていかないようにしたいですね。

## 習慣6 — `/cost` と `ccusage` で毎日コストを見張る

Claude Codeには1つ大きな弱点があって、UI上でコストがリアルタイム表示されないんです。1日中ガチャガチャ作業しても、自分で確認するまでは高いのか安いのかわからない。見えないから、そもそも気にする理由がない。

対策は、能動的に可視化してしまうこと。私は2つ使っています。1つ目は `ccusage`。`npx ccusage@latest` を走らせるだけで、インストール不要。使ったトークン数と、もっと重要な*バーンレート* — 消費スピードを表示してくれます。「このペースだと、あと2時間でクォータが尽きます」みたいに。異常にトークンを食ってるセッションを発見するのに重宝します。2つ目は、Claude Code内の `/cost` コマンド。これはシンプルで速い。私は1日2回走らせる習慣にしています — 朝は前夜と朝でいくら使ったかチェック、夕方は切り上げる前にチェック。

面倒に聞こえるかもしれませんが、こんな話があります。

2025年3月、Anthropicは Claude Code 上で2つのキャッシュ不具合が重なって起きていたことを認めました。1つ目は、新しいデータをキャッシュに書き込めず黙って失敗していた。2つ目は、古いキャッシュも読み出せなかった。結果として、すべてのリクエストがキャッシュ割引を受けられず、通常料金でまるごと処理されていた、ということです。Max $200/月 — 通常なら2〜3ヶ月もつクォータ — のユーザーの多くが、普段通りの作業で47時間できれいに底をつきました。コストが8〜15倍。しかも静かに、です。システムはエラーも警告も出さなかった。ツールは「いつも通り」動いていた。

早く気づいて他のツールへ切り替えられた人たちは、共通して `/cost` を確認する習慣を持っていました。そういう習慣のなかった人たちは…クォータが尽きて初めて気づいたわけです。

ここで得るべき教訓は「Claude Codeは信頼できない」ではありません。**感覚だけでコストを把握しようとしても、それは無理**、ということです。まず可視化が先。可視化から気づきが生まれ、気づきから行動が変わります。あらゆるコスト最適化の、最初の一段です。

## 1日の流れに落とし込んでみる

理屈だけだと退屈なので、この6つを取り入れた典型的な1日がどんな感じになるか、ちょっと書いてみますね。

*朝*、新しいターミナルを開いて、`/cost` で昨日と今朝の消費を確認。最初のタスクに取りかかるとき、最初のメッセージはきちんと書きます — ゴールは何か、制約は何か、関連するファイルはどれか。最初のターンでしっかりコンテキストを渡しておけば、Claudeも初手からいい動きをしてくれます。

*午前中盤*、タスクが片付いた — `/compact` を保持指示つきで走らせて、同じタスクの次のフェーズへ移ります。あるいは別のタスクに切り替えるなら、ターミナルを閉じて新しく開き直す。

*お昼*、休憩前にもう一度 `/cost`。異常に高いと感じたら、直前のセッションを振り返る — たいていログを貼りすぎたか、大きなファイルを余計に読ませたかです。

*午後*、横のターミナルに `ccusage` を開きっぱなしにしておきます。ときどきチラ見して、バーンレートが急に上がっていたら一度手を止めて確認。

*週末*、15分ほどかけて `CLAUDE.md` と `.claudeignore` を見直します。不要になったルールはないか、新しく生まれたファイル種別で追加すべきignoreはないか。

これだけです。新しいツールも複雑なワークフローもありません。要するに、**お金が見える → お金を気にする → 行動が変わる**、それだけ。

## よくある質問

**Claude Codeは1日にどれくらいトークンを使いますか？**  
決まった数字はありません — 会話の長さと、読み込ませるファイル数次第です。ただし毎ターン履歴全体が再送されるので、50ラウンドのデバッグセッションだと *Enterを押すたびに* 5〜10万トークン送ることもあります。Max $200/月のクォータは通常2〜3ヶ月もちますが、2025年3月のキャッシュ不具合のときは、普通に作業していて47時間で尽きた人が続出しました。

**`/compact` はトークンを消費しますか？**  
します。でもauto-compactよりはるかに軽いです。auto-compact（残り20%で発動）は圧縮パイプラインの最重量級で、裏で別のAIを呼んで全会話を要約させます。60%の段階で能動的に `/compact` を叩けば、もっと軽くて安い層で済みますし、60〜70%から始まる回答品質の低下ゾーンも回避できます。

**CLAUDE.mdはどのくらいの長さが理想ですか？**  
200行以内に収めて、*必ずやる/やらない*系のルールは最初の50行にまとめること。長すぎるとClaudeは流し読みになって重要なルールを見落とします。CLAUDE.mdは毎セッション読み込まれるので、2,800トークン×20セッション/日 = 56,000トークンが、あなたが1文字打つ前に消えている計算になります。

**`/clear` と `/compact` はいつ使い分ければ？**  
デバッグ途中で、積み上げた知識（決定事項、進捗）を残したいなら `/compact`。タスクが完全に片付いてまっさらから始めたいなら `/clear`（またはターミナル閉じて開き直し）。「修正→エラー→修正→エラー」ループに5回以上ハマっているなら、ターミナルを閉じて開き直すほうが、壊れたセッションにトークンを注ぎ足すより安上がりです。

## まとめ

Claude Codeは Enter を押すたびにコンテキストを丸ごと再送するので、コストは会話の長さに比例ではなく指数関数的に増えます。これが諸悪の根源です。そして今回紹介した6つの習慣 — タスクごとにターミナルを分ける、60%ラインで能動的に `/compact`、ファイル全体ではなく必要な部分だけ渡す、CLAUDE.mdを200行以下に圧縮、`.claudeignore` でゴミを落とす、1日2回 `/cost` を確認 — は、最初の土台です。

全部やっても30分で済みます。今日から毎日数ドルが浮きます。新しいツールは何もいりません。

でもこれはまだ基礎レイヤー。最大のテコ — コストを**10分の1**にできる可能性がある仕組み — は、意外と正しく理解している人が少ないメカニズムにあります。それが **prompt cache**。通常の入力より10倍安く使えますが、同時にとても壊れやすくて、見た目なんでもないような操作でうっかり吹き飛ばしてしまえるんです。その話は第2回で。
