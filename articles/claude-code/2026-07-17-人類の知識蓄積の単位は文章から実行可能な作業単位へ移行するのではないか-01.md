# AIエージェントと skills がもたらす、実行可能な集合知の可能性

AIエージェントと skills の活用について考えていたとき、ひとつの仮説に思い至りました。

それは、人類の知識蓄積の単位が、これまでの「文章」から、これからは **実行可能な作業単位** へと移行していくのではないか、という仮説です。

もう少し丁寧に言えば、それは単なる実行可能な手順ではなく、**失敗履歴を持ち、それを通じて改善され、どの条件で破綻するのかが明示された作業単位** であって、これがこれからの知識継承の重要な「形」になっていくのではないか、という仮説です。

少し大げさに聞こえるかもしれません。  
しかし、人類の知性の発展を振り返ると、知識は常に何らかの「単位」に圧縮され、共有され、再利用されることで蓄積されてきました。

* 言葉
* 概念
* 書籍
* 論文
* 業務手順
* マニュアル
* 設計思想
* ソースコード

これらは、個人の経験や思考を、他者が再利用できる形に変換した媒体でした。  
そして近代以降、その中心にあったのは、広い意味での「文章」だったと思います。

* 本を読む。
* 論文を読む。
* マニュアルを読む。
* 仕様書を読む。
* 事例を読む。

人間は、そうした文章を読み、理解し、自分の状況に合わせて解釈し、行動へとつなげてきました。  
この意味で、**文章は人類の集合知を支える重要な形式** でした。

しかし、AIエージェントが自然言語で書かれた手順を読み、それをもとに実際の作業を遂行するようになると、この構造に少し変化が生まれます。  
知識は、単に読まれるものではなくなり始める。  
それは、選択され、実行され、時に失敗し、修正され、再利用されるものになっていくかもしれません。

# 文章として蓄積されてきた集合知

人間社会における知識の発展を振り返ると、文章は圧倒的に重要な役割を果たしてきました。

書籍、論文、マニュアル、法律、設計書、などなど。

これらはすべて、個人や組織の経験を、他者が参照し、理解し、再利用できる形に変換するための仕組みです。

たとえば、ある熟練者が、長年の経験から「このような場合はこう判断すべきだ」という知見を得たとします。  
その知見が本人の頭の中にあるだけなら、それは個人の知恵に留まります。  
しかし、それが文章化され、共有され、他者に読まれ、批判され、改善され、別の文脈で再利用されるようになると、それは集合知の一部になります。

この意味で、文章は人類の知的進化における重要な圧縮形式でした。

* 複雑な経験を圧縮する
* 言葉として固定する
* 他者に共有する
* 別の文脈で再利用する
* さらに改善する

このサイクルを通じて、人間社会は知識を蓄積してきました。

ただし、文章には今まではひとつの大きな前提がありました。  
それは、**文章を読んで実行する主体が、基本的には人間** ということです。

マニュアルは、人間が読んで作業するためのものでした。  
論文は、人間が読んで理解し、研究や設計に活かすためのものでした。  
設計書は、人間が読み、議論し、実装へと変換するためのものでした。

つまり、文章は知識の保存媒体ではあっても、それ自体が直接作業を行うわけではありませんでした。  
**知識と行動の間には、人間の理解と判断が挟まっていた** のです。

では、人類の知識の共有は、すべて「知識とその遂行の間に人間の理解と判断が挟まっていた」ものだったかというと、近年、そうとも言えない形態が出現してきました。

# ソフトウェアはすでに「実行可能な知識」だった

ある見方をすると、人類はすでに、「文章としての知識」から「実行可能な知識」への大きな移行を経験しているとも言えます。

それがソフトウェアです。

ソフトウェアは、単なる説明ではありません。  
実際に動きます。  
ある処理の手順、判断条件、例外処理、データ構造、インターフェース、テスト、運用上の前提が、コードという形で記述され、計算機によって実行されます。  
ライブラリといった小さな再利用可能な部品も流通しています。

その意味で、**ソフトウェアは非常に強力な「実行可能な知識」** でした。

特にオープンソースソフトウェアは、単なるプログラムの共有ではありませんでした。  
そこには、コード、issue、pull request、レビュー、テスト、バグ報告、リリース履歴、ドキュメント、設計判断が結びついていました。

そのプロセスを通じて、ソフトウェアは失敗しながら改善され、成熟していきます。

その意味で、GitHub のような仕組みは、**実行可能な知識を世界中の人々が共同で育てるための社会的インフラ** だとも言えます。

* コードが動く
* バグは記録される
* 修正は履歴に残る
* 議論は issue や pull request に残る
* 改善はバージョンとして蓄積される

これは、かなり高度な「実行可能な集合知」の形だったのではないかと思います。

しかし、ソフトウェアには大きな制約がありました。  
それは、その読み書きに参加できる人が、基本的にはプログラマーに限られていたということです。

ソフトウェアは実行可能ですが、計算機のために書かれています。  
特定のプログラミング言語を理解し、依存ライブラリや実行環境を整え、ビルドし、テストし、デバッグしなければなりません。  
場合によっては、プログラマーであっても、巨大なコードベースを正確に読み解き、改善することは簡単ではありません。

つまり、オープンソースソフトウェアは強力な実行可能な集合知でしたが、その入口はまだかなり狭かった。

実行可能な知識ではあった。  
しかし、その知識を読み、書き、改善するには、高い専門性が必要だったのです。

# skills は「実行可能な作業単位」を民主化するのかもしれない

AIエージェントにおける skills は、この構造をさらに変える可能性があります。

ここでいう skill とは、AIエージェントに「この種類の作業をするときは、こう考え、こう進め、こう確認する」と伝えるための、再利用可能な作業手順のようなものです。

単なる一回限りのプロンプトではありません。  
ある作業を行うための手順、適用条件、判断基準、制約、参照資料、場合によっては scripts や tools までを含んだ、まとまりのある知識単位です。  
たとえば、次のような skill がありえます。

* 技術調査を行う skill
* 障害対応の初期切り分けを行う skill
* 文章の構成をレビューする skill
* 契約書のリスク項目を確認する skill
* 顧客対応の方針を整理する skill

これらは、人間が読むマニュアルにも似ています。  
しかし決定的に違うのは、AIエージェントがそれを直接読み、作業に使えるという点です。

つまり skill は、知識であると同時に、行動のための単位でもあります。  
そしてソフトウェアと違って、skills は自然言語を中心に記述できます。

もちろん、良い skill を書くにはリテラシーが必要です。  
AIエージェントの挙動、tool の使い方、権限、実行環境、失敗パターンを理解する必要もあります。

それでも、プログラミング言語でソフトウェアを書くよりは、多くの人に開かれている可能性があります。

* 業務の熟練者
* 編集者
* 研究者
* デザイナー
* 法務担当者
* カスタマーサポート担当者
* 医療や教育の専門家

現場で何度も失敗と改善を繰り返してきた人々

そうした人たちが、自分たちの判断手順や確認観点や失敗パターンを、自然言語を中心にした skill として記述し、AIエージェントに使わせることができるようになるかもしれません。

もしそうだとすれば、**skills は「実行可能な作業単位の民主化」をさらに一段進める可能性** があります。

ソフトウェアは、実行可能な知識を作りました。  
しかし、その読み書きは主にプログラマーの領域でした。  
skills は、自然言語を通じて、実行可能な作業単位の読み書きを、より広い人々に開くかもしれません。

これは、単なるノーコードや自動化ツールの延長ではないと直感します。  
むしろ、**人間の経験や判断や失敗から得られた実務知を、AIエージェントが利用可能な形で蓄積していくための、新しい知識媒体** なのではないかと思えてきます。

# しかし、共有されるだけでは集合知ではない

ただし、ここでひとつ重要な問題があります。  
skill は、共有された瞬間に集合知になるわけではありません。  
ましてや、信頼できる実行可能な集合知になるわけではありません。

単に「良さそうな手順」が書かれ、GitHub や marketplace に置かれているだけでは、それはまだ本当の意味で成熟した知識ではないと思います。

これは、ソフトウェアにおいても同じでした。  
コードが公開されているだけでは、信頼できるソフトウェアとは限りません。

実際に使われ、バグが見つかり、issue が立ち、pull request が作られ、レビューされ、テストが追加され、リリースを重ねることで、少しずつ信頼できるものになっていきます。

skill も同じではないでしょうか。

信頼できる実行可能な集合知の単位となるには、skill は十分な失敗履歴を持ち、その履歴を通じて改善され、さらに自らがどの条件で破綻するのかを明示していなければならない。

ここでいう失敗履歴とは、単なるエラーログではありません。それは、

* その skill がどのような状況で誤用されたのか
* どの前提を取り違えたのか
* どの場面で人間に確認すべきだったのか
* どの tool の結果を過信したのか
* どの環境ではうまく動かなかったのか
* どの修正によって再発が防がれたのか

そうした経験の蓄積です。  
こうした失敗履歴と改善履歴を持って初めて、skill は単なる「共有された手順」から、「信頼できる実行可能な知識」へと近づいていくのではないかと思います。

# 失敗の知識は、どこに蓄積されてきたのか

ここで重要なのは、熟練者の知識の多くは、成功手順だけでできているわけではないということです。

むしろ、実務における判断のかなりの部分は、過去の失敗の記憶によって支えられているように思います。

たとえば、プログラマーとして何かアーキテクチャ上の判断をしようとするとき、突然、過去の痛い記憶がよみがえることがあります。

「あのとき、これと似たDBスキーマにして、あとで拡張性がなくて大変な目にあった」  
「あの設計は最初はきれいに見えたけれど、運用が始まると例外ケースに耐えられなかった」  
「あの抽象化は便利そうだったけれど、半年後には誰も理解できなくなった」

こうした記憶は、必ずしも明示的なルールとして言語化されているわけではありません。  
しかし、判断の瞬間には確かに働いています。

ある選択肢に対して、理由を完全には説明できないまま「これは危ない」と感じる。

ある設計に対して、過去の失敗と似た匂いを感じる。  
ある手順に対して、「このまま進めると後で詰まる」と直感する。  
このような知識は、成功事例の知識というより、失敗によって身体に刻まれた知識です。

熟練工や職人の世界でも、同じことがあるはずです。  
道具の扱い方、力の入れ方、素材の癖、危ない兆候、やってはいけない手順。

それらの多くは、教科書に完全な形で書かれているというより、現場での経験、失敗、叱責、観察、模倣を通じて伝わってきたものです。  
つまり、人間社会では、失敗から得られた知識のかなりの部分が、師弟制度、職場での実習、レビュー文化、現場の会話、暗黙知として伝承されてきました。

一方で、そのような失敗の知識は、公共の知識としては蓄積されにくい傾向があります。

成功事例は公開されやすい。  
うまくいった設計、きれいなアーキテクチャ、成功したプロジェクト、洗練されたベストプラクティスは、記事や本やカンファレンスで共有されやすい。

しかし、失敗事例は公開されにくい。  
そこには、ある種のバイアスがかかるからです。

* 失敗は恥ずかしい
* 会社の信用に関わる
* 顧客情報や社内事情に結びついている
* 特定の環境に依存していて一般化しにくい
* 責任の所在が問題になる

そのため、実際には組織の内部で多くの失敗が共有され、改善に活かされていたとしても、それが公共の知識として外に出てくることは限られています。

ここが、現在のAIエージェントの弱さの一部の根源として、関係しているのかもしれません。

AIは大量の公開テキストから学習できます。  
しかし、公開されている知識は、どうしても成功事例や整えられた説明に偏りがちです。

実務の中で蓄積されてきた「痛い失敗の記憶」や「この条件では壊れる」という負の知識は、十分には言語化されておらず、公開もされていない。

だとすれば、**AIエージェントが実世界の複雑な作業で弱さを見せる理由のひとつは、単に推論能力が足りないからではなく、失敗の経験が足りないから** なのかもしれません。

# 多段の作業では、失敗は連鎖する

この問題は、AIエージェントが多段の作業を行うときに特に大きくなります。

単発の質問に答えるだけなら、多少の不確実性があっても、ある程度うまく見えるかもしれません。  
しかし、実世界の作業は、多くの場合、単発では終わりません。

* 調べる
* 判断する
* 変更する
* 確認する
* 例外に対応する
* 再判断する
* 必要なら人間に戻す

このような多段の処理が連続します。

ここで、仮に1つのステップの成功確率が 95%であったとしても、それを10段重ねると、全体の成功確率はおよそ 60%まで下がります。  
つまり、各ステップが一見高い精度で動いていても、長い作業全体として見ると、失敗の可能性は急速に大きくなります。

実世界の作業では、単に「各ステップの成功確率が高い」だけでは不十分です。  
重要なのは、どこで失敗しやすいかを知っていることです。

* 失敗の兆候を検知できること
* 危ないときに止まれること
* 人間に確認すべき条件を持っていること
* リカバリープランを持っていること
* そして、その失敗が次の改善に反映されること

この意味で、信頼できる skill に必要なのは、成功手順だけではありません。  
むしろ、失敗の履歴、破綻条件、回復手順、人間にエスカレーションすべき条件こそが、skill を実用的な知識単位へと成熟させるのだと考えます。

そして、ここで次の問題が生まれます。  
**そもそも、AIエージェントがある skill をいつ読み、いつ使い、どの手順を適用し、どこで失敗したのかを、私たちは知ることができるのでしょうか。**

失敗から学ぶためには、まず失敗を観測できなければならない。  
ここに、AIエージェント時代の新しい observability の問題があるのではないかと考えました。

# skill の observability とは何か

失敗から学ぶためには、まず失敗を観測できなければなりません。  
これはソフトウェアでは、ある程度当たり前のこととして扱われています。

* ログがある
* テストがある
* スタックトレースがある
* CI の結果がある
* issue がある
* pull request がある
* diff がある
* リリース履歴がある

もちろん、それでも原因分析は簡単ではありません。  
しかし、少なくとも「何が実行されたのか」「どこで失敗したのか」「どの変更が影響したのか」を追跡するための仕組みは、それなりに整備されています。

一方で、AIエージェントにおける skills は少し厄介です。

たとえば、MCP などの tool call は比較的観測しやすいです。  
なぜなら、tool call は LLM の外側に出るからです。

どの tool が呼ばれたのか、どんな引数が渡されたのか、どんな結果が返ってきたのか、エラーが起きたのか、どれくらい時間がかかったのか。  
こうしたことは、ログとして残しやすい。

しかし **skill の実行は、LLM の実行文脈の中に溶け込みやすい。**

* AIエージェントが SKILL.md を読む
* その内容をコンテキストに取り込む
* 手順や判断基準を自然言語的に解釈する
* そして、その影響が出力や tool call や判断に反映される

このとき、外側から見ているだけでは、どの skill が実際に効いたのかが分かりにくい。  
最終的な出力が skill に沿って見えたとしても、それが本当にその skill によるものかは分かりません。

ここに、AIエージェント時代の新しい observability の問題があります。

知りたいのは、単に「どの tool が呼ばれたか」だけではありません。

* どの skill が読まれたのか
* どの skill を使うと宣言したのか
* どの reference file を参照したのか
* どの手順に従ったつもりだったのか
* どこを省略したのか
* どこに不確実性が残ったのか
* 最終的に、どの skill や file が作業に影響したとエージェント自身が考えているのか

こうした情報がなければ、skill の失敗を後から分析することは難しい。  
そして分析できなければ、改善もできません。

つまり、信頼できる実行可能な集合知を作るためには、skill そのものだけでなく、skill が使われたときの痕跡が必要になります。  
実行可能な知識には、実行の証拠が必要なのです。

# SkillTrace という小さな実験

この問題意識から、私は SkillTrace という小さなツールを作り始めました。  
[**SkillTrace は、AIエージェントの skill usage を観測するための実験的なツールです。**](https://zenn.dev/h1deya/articles/skilltrace-introduction)

最初から大きなことをしようとしているわけではありません。  
一番最初の小さな、しかし不可避なな問いに注力しています。

それは、AIエージェントが、ある run の中で、

* どの skill file を読んだのか
* どの skill を使うと宣言したのか
* 最終的に、どの skill や reference file が作業に影響したと振り返ったのか

これらを見えるようにすることです。

SkillTrace では、大きく3種類の証拠を比較します。  
1つ目は、file access です。  
skill file や reference file が実際にアクセスされたのかを、OS の機能を使って観察します。

2つ目は、MCP を通じた宣言です。  
エージェントに「この skill を使い始めた」「この reference を読んだ」といった情報を明示的に記録するための MCP tool を呼び出すことの指示を、大元の skill file（AGENTS.md / CLAUDE.md）の冒頭に、一時的に挿入します。

3つ目は、run の最後の reflection です。  
エージェント自身に、どの skill や reference が作業に影響したと思うかを振り返らせる指示も、一時的に挿入します。

もちろん、file access は、読まれたことを示すだけで、使われたことを証明しません。  
MCP を通じた使用の報告は、エージェントの自己申告です。  
reflection もまた、後からの自己申告です。

しかし、これら３つの情報ソースを突き合わせることで、理解の確度が上がります。また情報ソース間の不一致は、失敗分析の入り口になります。

SkillTrace は、skill 利用の真実を証明するツールではありません。  
むしろ、複数の痕跡を並べ、そのズレを観察するための小さな実験です。  
詳しい使い方や技術的な仕組みについては、[**別の記事**](https://zenn.dev/h1deya/articles/skilltrace-introduction) で紹介しようと思います。

ここでお伝えしたかったのは、こうした観測の仕組みが、将来的には skill ecosystem の基礎になるかもしれないという気づきです。

# 実行ログは、説明責任のためだけではない

AIエージェントの実行ログというと、しばしば説明責任や監査の文脈で語られます。

* なぜその判断をしたのか
* 誰が責任を持つのか
* 後から検証できるのか
* 不正や逸脱はなかったのか

もちろん、それらは重要です。  
しかし、私がここでより強く関心を持っているのは、少し違った側面です。

実行ログは、責任を問うためだけのものではありません。  
失敗を学習可能な経験へ変換するための素材でもあります。  
ある skill が失敗したとき、

* その失敗を後から再構成できる
* どの前提が間違っていたのかを考えられる
* どの手順が足りなかったのかを見つけられる
* どの条件では人間に戻すべきだったのかを追加できる
* 同じ失敗を防ぐための regression case を作れる
* skill の説明や適用条件や reference を改善できる

そうして初めて、失敗は単なる一回限りの失敗ではなくなります。  
それは、skill を成熟させるための材料になります。

人間の組織でも、失敗から学ぶ文化を持つ組織と、失敗を隠す組織では、長期的な学習能力が大きく変わります。

事故調査、postmortem、ヒヤリハット報告、品質管理、SRE の incident review。

これらはすべて、失敗を責任追及で終わらせず、再発防止と組織学習につなげるための仕組みです。

AIエージェントの skills にも、同じような仕組みが必要になるのではないでしょうか。

ただし、そのためにはまず、失敗が観測可能でなければならない。

* どの skill が読まれたのか
* どの skill が使われたと宣言されたのか
* どの reference が影響したのか
* どの手順が実行されたのか
* どこで不確実性が残ったのか
* どの条件で破綻したのか

これらが見えなければ、postmortem は書けません。  
postmortem が書けなければ、skill は失敗から成熟できません。  
そして、失敗から成熟できなければ、それは信頼できる実行可能な集合知にはなりにくい。

この意味で、skill observability は、skill ecosystem のかなり基礎的なレイヤーになるのではないかと考えています。

# 失敗履歴を持つ skill ecosystem へ

もし skills が本当に「実行可能な作業単位の民主化」を進めるのだとすれば、次に必要になるのは、skill を単に共有する場所ではないと思います。

必要なのは、skill が失敗し、修正され、評価され、再利用され、また失敗し、さらに改善されていくための ecosystem です。

そこでは、skill は単なる Markdown ファイル本体だけではないでしょう。

* skill 本体
* 適用条件
* 使ってはいけない条件
* 必要な tool や権限
* reference file
* known failure modes
* incident reports
* postmortems
* regression cases
* version history
* trust information

こうしたものが結びついた、履歴を持つ知識単位になっていくはずです。

成熟した skill とは、単に美しく書かれた skill ではないと思います。  
むしろ、十分に使われ、十分に失敗し、その失敗が記録され、改善に反映され、どの条件で壊れるのかが明示されている skill です。

人間の熟練者が、成功の知識だけでなく、痛い失敗の記憶によって判断しているように。  
信頼できる skill もまた、成功手順だけでなく、失敗履歴によって成熟していくのではないでしょうか。

このとき、AIエージェントの知識基盤は、単なる prompt sharing や skill marketplace とは違うものになります。  
それは、実行可能な知識に、実行証拠と失敗履歴と改善履歴を結びつける仕組みです。

書籍が、文章としての集合知を蓄積してきた。  
ソフトウェアが、実行可能な知識を世界中で共同開発する道を開いた。  
そして skills は、自然言語を通じて、より広い人々が実行可能な作業単位を記述し、共有し、改善する道を開くかもしれない。

しかし、そのためには、失敗から学べる仕組みが必要です。  
そして、失敗から学ぶには、まず観測できなければならない。  
SkillTrace は、そのための小さな第一歩です。

# まだ始まったばかりの変化

もちろん、これはまだ始まったばかりの話です。

現在の AIエージェントは、まだ不安定です。  
skills も成熟しているとは言えません。  
skill の書き方も、評価方法も、共有方法も、失敗履歴の扱い方も、まだほとんど手探りです。

さらに、失敗履歴をどう共有するかには難しい問題があります。  
失敗には、機密情報が含まれることがあります。  
会社や顧客の信用に関わることがあります。  
特定の環境や文脈に強く依存することがあります。  
そのまま公開できないことも多い。

したがって、すべての失敗をそのまま公共の知識にすることはできません。  
しかし、抽象化できる失敗もあります。  
匿名化できるパターンもあります。  
一般化できる破綻条件もあります。  
共有できる recovery pattern もあります。

今後重要になるのは、失敗そのものをただ晒すことではなく、失敗から得られた学びを、再利用可能な形に変換することだと思います。

人類の知識蓄積は、これまで文章を中心に発展してきました。  
文章は、経験を圧縮し、共有し、再利用するための強力な形式でした。

その一部はソフトウェアとなり、実行可能になりました。  
ソフトウェアは、知識を計算機の上で動かし、issue や pull request やテストを通じて、失敗しながら改善される知識単位を生み出しました。

そして今、AIエージェントと skills によって、自然言語で書かれた知識が、より直接的に実行へ接続されようとしています。

もしこの流れが続くなら、これから重要になるのは、単に「良い文章を書くこと」や「良い skill を共有すること」だけではないのかもしれません。  
むしろ、

* どのように使われたのか
* どこで失敗したのか
* どう改善されたのか
* どの条件で信頼できるのか
* どの条件では人間に戻すべきなのか

そうした履歴を持った、実行可能な作業単位を蓄積していくこと。  
それが、人類の知識蓄積の次の単位になっていく可能性があります。

もちろん、文章がなくなるわけではありません。  
むしろ文章は、これからも重要であり続けるはずです。  
新しい概念を社会的に固定化するという役割は、より高度な skill を実現するために不可欠ですし、  
文芸作品は引き続き人々を魅了し続けるでしょう。

ただし、文章はそれ単独で読まれるというだけではなく、skill の一部となり、実行の文脈に埋め込まれ、失敗履歴や改善履歴と結びついていくかもしれない。

次の時代の知識は、ただ書かれるだけではなく、実行される。  
実行されるだけではなく、失敗する。  
失敗するだけではなく、その失敗を通じて改善される。  
そして、その履歴ごと再利用される。

もしそうだとすれば、私たちはいま、人類の知識蓄積の形式が少しずつ変わり始める、かなり面白い変革期にいるのかもしれません。

そう考えると、少しわくわくしてきませんか？
