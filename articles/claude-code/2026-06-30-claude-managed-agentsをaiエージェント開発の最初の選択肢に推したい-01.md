Claude Managed Agentsが便利で、最近のAIエージェント開発では最初の選択肢に挙げているので、それについて書きます。

<https://platform.claude.com/docs/ja/managed-agents/overview>

<https://www.anthropic.com/engineering/managed-agents>

Claude Managed Agentsは、Anthropicが提供するインフラやSandbox環境で、AIエージェントを作成・実行できるマネージドサービスです。モデルプロバイダーであるAnthropic自体が、本番レベルなAIエージェントを構築するために必要とされる環境を全て用意しています。MastraやGoogle ADKなどのAIエージェントフレームワークとは異なります。

開発者がやることは、エージェントのプロンプト、スキル、ツール、MCP、マルチエージェント構成など、エージェント自体の設計だけで済み、インフラや環境を考える必要がなくなります。

直近1年間で、いくつかのAIエージェントアプリケーションの開発に携わり、AIエージェントフレームワークは、Mastra、Google ADKを使ってきました。これらを使えば、シンプルなAIエージェントなら本当に手軽に開発できます。

しかし、本番レベルのAIエージェントを開発したいとなると、結構大変です。エージェント実行基盤を自前で作る必要がありました。

## 本番レベルのAIエージェントとは

本番レベルのAIエージェントを実現するには、以下が不可欠です。

* エージェントに依頼したタスクが非同期実行できる（ブラウザを閉じてもタスクは継続される）
* タスクは長時間（数十分〜数時間）実行できる
* 仮にタスクが失敗しても自動で再実行できる
* タスクを隔離コンテナ環境（Claude Codeみたいにbashコマンド使ったり、ファイルをread/writeしたり、ライブラリを勝手にインストールしたりできる）で実行できる

そのためには、非同期実行のためのJob/Worker、リトライ/チェックポイントの仕組み、隔離コンテナ環境の構築を、開発者がやる必要がありました。この基盤構築がエージェント開発の面倒な作業でした。

**Claude Managed Agentsは、上記の面倒なエージェント基盤を丸ごと管理してくれます。** エージェントループもAnthropic側で実行されるので、Claudeでこの用途のエージェントを作る場合は、MastraやGoogle ADKなどのエージェントフレームワークが不要になるケースも多いです。開発者はエージェント自体の設計と、タスク管理やUI等のビジネスロジックに集中でき、圧倒的に短い開発期間で、本番レベルのAIエージェントを開発できるようになります。

誤解を恐れずにいうと、Claude CodeやManusみたいなAIエージェントを、めちゃくちゃ手軽にアプリに組み込めるようになった的なイメージです。

GAOGAOのAIエージェント開発提案でも、Claude Managed Agentsは最初に提案すべき選択肢になってきています。

非常に便利なので、私が関わっている3つのAIエージェント開発では、全てClaude Managed Agentsへと置き換えました。新規プロジェクトも、まずはこれで開発をスタートしています。

## Mastra/Google ADK → Claude Managed Agents移行の実体験

Claude Managed Agentsが出る前は、MastraやGoogle ADKを使って、AIエージェントを開発していました。業務効率化AIエージェント基盤を開発している際に、本番レベルのAIエージェントにしていくために、以下の3つの基盤を自前で構築する必要があると考えて以下の通り設計し、今後の開発タスクとして積んでいました。

**ワーカー基盤**  
Cloud Run + SSEで自前運用していた構成では、ADKのエージェント実行はHTTPリクエストのライフサイクルに縛られており、ブラウザを閉じれば実行が止まり、SSE接続が切れれば途中経過ごと消えます。これを解決するために、ユーザー入力をCloud Tasksキューに投入し、detached workerで非同期実行する構成を設計しました。

**リトライ基盤**  
タスクが途中で失敗しても、最初からやり直すしかありませんでした。これを解決するために、タスクの実行状態をDBに永続化し、失敗時に途中地点から再開できる仕組みを設計しました。タイムアウト管理、キャンセル、二重実行防止も含めると、TemporalやRestateといったワークフローエンジンの導入も検討が必要な規模感でした。

**Sandbox環境**  
エージェントにコード実行能力を持たせるための隔離環境です。まずはGemini BuiltInCodeExecutionで最小限の計算能力を確保し、段階的にDocker / gVisor、最終的にはFirecracker microVMへ拡張する計画でした（この辺は候補に挙げただけで、詳しく調査できていません）。

このうちワーカー基盤だけでも、Cloud Tasks統合 + 状態管理 + リトライの仕組みを一気通貫で構築する必要があり、これだけでもかなり大変でした。エージェント自体の開発ではなく、エージェントを動かすための基盤づくりに時間を取られる状況でした。

結局、実装に移る前に Claude Managed Agents がリリースされ、設計していた基盤のほぼ全てがプラットフォーム側の標準機能として提供されていました。工数を大きく取られるはずだった基盤の構築をスキップし、エージェント自体の設計に集中できるようになりました。

## 課題や懸念点

**Claude依存になる**  
モデルと実行基盤の両方を Claude に依存することで、ベンダーロックインのリスクがあります。インフラをAnthropicではなく、GCPやAWSに制限したいケースはあると思います。その場合は、Claude以外でもManaged Agentsサービス（AWS AgentCoreやGoogleのManaged Agentsなど）が盛り上がってきているので、今ならそちらを使う選択肢もあると思います。

**コストが結構かさむ**  
料金はトークン料金＋$0.08/セッション時間（アイドル無料）です。Claude APIを直接使うのと違うのは、$0.08/セッション時間が追加されるだけです。それよりも、長時間×多数ツール呼び出しでトークン料金が効いてきます。Sandbox環境でエージェントを動かす際、タスクによっては何度もループが回るので、コストがかさみます。

**Advisor tool**  
これは機能追加の要望ですが、Advisor toolが公式ツールとして提供されたら嬉しいなと思っています。上述した通り、コストが結構かさむので、Claude Managed Agents上に構築するAIエージェントのモデルをSonnetにしてコストを抑えたい、とはいえ精度は保ちたいケースで、Advisor toolでOpusをセットできたら便利そうです。ただ、Advisor tool自体はClaude Platform / Claude Codeに存在しますが、Claude Managed Agents内ではまだ提供されていません。欲しかったので、自前でカスタムツールとしてAdvisor toolを作ってしまいました。作ったものの、すぐに公式ツールで提供される気はしています。
