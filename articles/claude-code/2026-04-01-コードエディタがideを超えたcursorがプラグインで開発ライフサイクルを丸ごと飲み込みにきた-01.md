## この記事でわかること（ミニ結論）

* AIコードエディタ「Cursor」が**プラグインによる他ツール連携に対応**した
* Figma・Linear・Stripe・AWS・GitLabなど**30以上のプラグインが  
  Cursor Marketplaceで公開**された
* プラグインで**設計・実装・決済・デプロイ・監視をエディタ1つで完結**できる
* **MCPサーバー・Skills・Subagents・Hooks・Rules**の5要素で  
  プラグインが構成される
* 開発者・チームが**独自プラグインを作成してコミュニティ共有**することも可能

---

## なぜエンジニアは毎日ツールを行き来するのか

これまでのエンジニアの1日はこんな感じだった。

```
LinearでIssueを確認 → ブラウザ
Figmaでデザインを確認 → 別アプリ
Cursorでコードを書く → エディタ
Stripeのドキュメントを調べる → ブラウザ
Vercelにデプロイ → ターミナル
Datadogでログを確認 → ブラウザ
```

**コンテキストスイッチのたびに集中が途切れ、生産性が落ちる。**

CursorはAIコードエディタとして急速にシェアを拡大してきたが、  
今回のプラグイン対応で一線を越えた。

## 最新動向：Cursor Marketplaceの全容

2026年2月17日、CursorのメーカーAnysphereがプラグインによる  
他ツール連携対応を発表。3月11日にはパートナー企業による  
**30以上の新プラグイン**が「Cursor Marketplace」に追加された。

### カテゴリ別プラグイン一覧

| カテゴリ | 対応ツール | できること |
| --- | --- | --- |
| **設計・プロジェクト管理** | Figma、Linear | デザインをコードに変換、課題・ドキュメントにアクセス |
| **決済・サブスクリプション** | Stripe | 決済連携コードをエディタ上で生成 |

| **データ・アナリティクス** | Databricks、Snowflake、Amplitude、Hex | 本番データを直接クエリしてインサイトを抽出 |
| **開発・監視管理** | GitLab、Atlassian、Datadog | コード管理・監視・課題追跡を一元化 |

## プラグインの5つの構成要素

Cursorのプラグインは以下5要素の組み合わせで構成される。

```
┌──────────────────────────────────────────┐
│             Cursor Plugin                │
│                                          │
│  ① Skills      AIエージェントが実行できる   │
│                ドメイン固有のプロンプト・    │
│                コード                     │
│                                          │
│  ② Subagents   タスク実行に使う専用        │
│                エージェント               │
│                                         │
│  ③ MCP Servers 外部ツール・データソースへの │
│                接続サービス               │
│                                         │
│  ④ Hooks       エージェントの動作を変更・   │
│                拡張するカスタムスクリプト    │
│                                         │
│  ⑤ Rules       コーディング規約・設定・     │
│                好みを反映するシステムレベル  │
│                の指示                    │
└──────────────────────────────────────────┘
```

この5要素の組み合わせで、エージェントが**タスクを自律的に  
実行するための完全な文脈**が生まれる。

## Before / Afterで見る変化

### Before（プラグインなし）

```
6つのツールを行き来しながら作業
→ コンテキストスイッチが多発
→ 集中力が途切れる
→ 作業ログが分散する
```

### After（プラグインあり）

```
「LinearのIssue #123を実装して、
Figmaのデザインに合わせて、
Vercelにデプロイして」

→ AIがすべてのツールを横断して実行
→ エディタから離れない
→ 作業の文脈がCursorに集約される
```

## 独自プラグインの作成・共有も可能

開発者・チームが独自プラグインを作成して  
Cursor Marketplaceにコミュニティ共有できる。

社内ツールや独自ワークフローをプラグイン化すれば、  
チーム全員が同じAIの文脈で開発できるようになる。  
これは\*\*「チームのナレッジをAIに組み込む」\*\*という新しい発想だ。

## 現場エンジニア目線の考察

半導体の現場でも、設計ツール・シミュレーター・  
検査システムなど、複数ツールを行き来しながら作業することは日常だ。

Cursorのプラグイン対応は、ソフトウェア開発だけでなく  
**「複数ツールの連携が必要な現場全般」への示唆**を含んでいる。

特に注目したいのが**MCPサーバーの役割**だ。  
CursorのプラグインがMCPで外部ツールに接続するように、  
製造現場でも各設備・ツールをMCPで繋いだAIエージェントが  
横断的に作業する未来は現実的になってきた。

一方で懸念もある。

* **セキュリティ管理の複雑化**：30以上のツールが接続されると  
  権限管理とデータの流れの把握が難しくなる
* **ベンダーロックイン**：Cursor Marketplaceへの依存が高まると  
  他エディタへの移行コストが上がる
* **AIの判断の追跡**：ツールを横断してAIが自律実行する場合、  
  何がどこで実行されたかの記録が重要になる  
  （これはEntire CLIのCheckpoints機能が活きる場面でもある）

## まとめ・次のアクション

**結論：Cursorはコードエディタではなくなった。  
開発ライフサイクル全体を管理するAIネイティブプラットフォームだ。**

今すぐ取り組むべきことは以下だ。

1. **Cursor Marketplaceを確認する**  
   （自分のプロジェクトで使えるプラグインがすでにあるかもしれない）
2. **Figmaプラグインを試す**  
   （デザイン→コード変換は即効性が高い）
3. **チームの開発ツールをリストアップして  
   プラグイン化できるものを検討する**
4. **MCPの仕組みを理解する**  
   （Cursorプラグインの根幹技術として今後必須の知識になる）

「コードを書く場所」と「開発を管理する場所」が統合される時代が来た。  
エディタの選択がそのまま**開発体験の設計**になる。
