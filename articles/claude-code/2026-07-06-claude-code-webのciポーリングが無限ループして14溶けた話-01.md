## はじめに

Claude Code Webでリサーチ作業をしていたら、CIポーリングのスケジュールが夜通し回り続けて、APIクレジットが約$14消えました。

今回設定されたスケジュールに「変化がなければ1時間後にチェックを再設定する」という内容が含まれていて、**CIがグリーンでもポーリング自体が再設定され続ける**状態になっていました。これに気づかずパーミッションを承認してしまったのが原因です。

同じような体験をする方を少しでも減らせればと思い、備忘録としてまとめます。

## やっていたこと

Claude Code Webで、リサーチ内容をMarkdownにまとめてディスカッションする作業をしていました。モデルはClaude Fableです。作業が終わってGitHubにプッシュしたあと、CIの結果を確認するためのポーリングが設定されました。

## CIポーリングが止まらなくなった

今回設定されたスケジュールの内容がこちらです。

![CIポーリングのスケジュール設定](https://static.zenn.studio/user-upload/deployed-images/7ed21a107b44b111a27dacce.png?sha=f83e30be9db509f031dff73787627f035ecef0a6)  
*実際に設定されたスケジュールの内容*

CI状態・レビューコメント・マージコンフリクトを確認し、変化がなければ1時間後に再チェックを設定する、という内容でした。

ポイントは、**「変化がなければポーリング自体を再設定する」ところまでスケジュールに含まれていた**ことです。CIはグリーンなのに「変化なし」と判定されて、1時間ごとのチェックが終了条件なく延々と繰り返されていました。

![定期チェックが繰り返されるログ](https://static.zenn.studio/user-upload/deployed-images/66cc3c6f638df0ea27b0a71d.png?sha=44a44c5326348bb286c8d21150a5ebbc6aeff2d9)  
*「変化なし、次回チェックを1時間後に再設定」が繰り返される*

## パーミッション承認の落とし穴

Claude Code Webはスケジュール設定のたびにパーミッションを求めてきていました。

1回目は普通にOKしました。2回目は「さっき許可したはずなのに」と思いつつOK。3回目は「毎回同じのが来るな、バグかな」と思って\*\*「許可なしで実行」\*\*にしてしまいました。

スマホやPC、通知欄など、同じ通知が重複して来ることがあるイメージだったので、バグ的な重複だと思い込んでいました。実際はすべて別々のスケジュールに対する許可要求で、「許可なしで実行」にしたことで夜通しポーリングが回り続けました。

## Fableのレート制限の仕組み

被害を大きくしたのがFableを使っていたことです。FableはAnthropicで最も高性能で料金も高いモデルですが、**他のモデルとは別に専用のレート制限が設定されています**。

今回の流れはおそらくこうです。

1. cronが夜通し回り、まずFable専用のレート制限を使い切った
2. Fableの枠がなくなり、超過分がAPIクレジットから消費されるようになった
3. APIクレジットも月間上限の$20に達して、ようやく止まった

もともと$6ほど使っていたので、この一晩で約$14が消費されたことになります。

## 被害のまとめ

![週間制限の使用状況](https://static.zenn.studio/user-upload/deployed-images/5c53e45ba03d0fb595e87d01.png?sha=bd76746e577436d419fb7d32037af0d1960fc976)  
*Fableのレート制限が100%に到達*

![利用クレジットの使用状況](https://static.zenn.studio/user-upload/deployed-images/82512640812c8463440109f3.png?sha=5373a76576ad2dcad850a7e08f0cfe73065bed59)  
*月間上限$20を超過し、残高がマイナスに*

* **APIクレジット**: 約$14を消費（月間上限$20に対して$20.27使用）
* **週間レート制限（すべてのモデル）**: 約40%を消費されて合計53%に
* **週間レート制限（Fable）**: 100%使い切り

月間クレジットの上限を$20に設定していたことと、自動チャージをオフにしていたことで、ここで止まりました。虎の子のFable枠がなくなったのはとても悲しい、、、。

ただ、もし自動チャージをオンにしていて、月間上限も設定していなかったら、無限にクレジットが消費され続けていた可能性があります。それを思うとかなりゾッとしますね、、第3者の攻撃でもないし、返金対応してくれたかというと怪しい気がする、、、。

## 教訓

* **スケジュール設定のパーミッションは内容を確認する** — 「ポーリングの再設定」までスケジュールに含まれていたのが今回の核心です。この一文に気づいていれば防げました
* **同じに見える通知を連打しない** — 重複だと思い込んで「許可なしで実行」にしたのが一番の原因です
* **クレジットの月間上限は必ず設定する** — この設定のおかげで被害が$20程度に収まりました
* **自動チャージの設定は慎重に** — オフだったことも歯止めになりました

ちなみに、Fableを使っていたからこそ専用のレート制限を早々に使い切り、クレジット消費が始まって異変に気づけたという面もあります。Fable以外のモデルだったらクレジットの被害は出なかったかもしれませんが、気づかないまま回り続けていた可能性もあります。どちらがマシだったのかは正直なんとも言えないですね、、

## おわりに

$14の出費も痛かったですが、それ以上にFableのレート制限を使い切られたのが悲しいです。Fableはサブスクリプション期間内での利用枠が限られていて、一度使い切ったら戻ってきません。貴重な枠をポーリングのループに消費されたのは、金額とは別の意味でダメージが大きかったです。

スケジュール機能は便利ですが、内容を理解した上で許可する必要があるなと痛感しました。この記事が参考になれば幸いです。以上です。ありがとうございました。
