アメリカ現地時間の2026年6月9日、二つの大きな発表が同じ日に重なりました。

ニューヨークで開催されたDatadogの年次イベントDASH 2026のキーノートで、Bits AIファミリーが大幅に拡張されました。Detection、Investigation、Remediation、Infrastructure、Code、Release、Testing、Data Analysis、Chat、Memories、Evals。エージェント単位で数えると10種類を超え、合計100以上の新機能がまとめて発表されています。発表の全体像はキーノートのまとめ記事に整理されています。

<https://www.datadoghq.com/blog/dash-2026-new-feature-roundup-keynote/>

同じ日、AWSはFinOps Agentをプレビューとして発表しました。Cost Explorer、Cost Anomaly Detection、Cost Optimization Hub、Compute Optimizerの四つのデータソースを束ね、コスト異常の自動調査、自然言語でのコスト質問、定期コストレポート、最適化機会の集約までを、SlackとJiraに直接届ける構成です。詳細はAWSのブログにまとまっています。

<https://aws.amazon.com/blogs/aws-cloud-financial-management/aws-finops-agent-is-now-public-preview/>

AWS DevOps Agentはすでに3月にGAしており、こちらはインシデント対応を担います。コスト管理を担うFinOps Agentが加わったことで、運用の主要なドメインに「AWS製の標準エージェント」が並んできたことになります。ただし、DevOps AgentはAWSだけでなくマルチクラウドやオンプレミス環境も対象にする点で、AWSのコストデータを対象とするFinOps Agentとは守備範囲が異なります。GA時の詳細は次のブログにまとまっています。

<https://aws.amazon.com/jp/blogs/news/announcing-general-availability-of-aws-devops-agent/>

表面上は、監視プラットフォームのDatadogと、クラウド基盤のAWSとで、別の領域の話に見えます。ただ、両者の発表を並べて読むと、Opsという同じ陣地を別の入口から取りに来ているのがわかります。機能を並べると大部分が重なっていて、表面的なスペック比較では違いが見えません。本記事では、この同日リリースを両者の立ち位置の違いから整理し、よく似たエージェント群が実際には何を奪い合っているのか、その見極めの軸と、そこから読み取れる予測まで踏み込みます。

対象読者は、SRE・FinOps・Platform Engineeringに関わる方を想定しています。

## なぜ「Opsエージェント」というカテゴリが今、立ち上がるのか

FinOps、DevOps、SREは別の名前を持っていますが、同じ構造的な課題を抱えていました。異常に気づいた運用者と、実際に直す開発者。この二者のあいだの往復コストです。

具体的には、こういう流れが日常的に発生するでしょう。たとえばコスト管理。Cost Anomaly Detectionがコストの異常を知らせるアラートを上げる。誰かがダッシュボードを開き、サービス別の内訳を見る。スパイクの原因がEC2なのかRDSなのかを切り分け、CloudTrailを開いて誰が何をデプロイしたかを照らし合わせる。Slackで関係者に確認を取り、リポジトリを開いて該当のIaCコードを探す。ここまでに、いくつもの画面とツールを行き来することになります。修正案を書いてPRを出すのはその後です。

インシデント対応も同じ流れです。アラートが鳴り、ダッシュボードを開き、ログを見て、APMのトレースを追い、関係者をSlackに呼び、修正を入れる。コンテキストスイッチの連続で、対応が深夜にかかればなおさら消耗します。

LLMの精度がこの解釈と往復作業を引き受けられる水準に達したのが、ここ1年ほどの変化と言えます。監視データを読み、原因を解釈し、修正を提案するという一連の作業を、人間の代わりに走らせられる。監視と操作を一気通貫にしたいという欲求は前からありましたが、ようやくそれを自律エージェントという形で世に出せるようになった、というタイミングの問題です。実際、AWSが運用向けの一連のエージェントをそろえ始めたのも、昨年末のre:Invent 2025でDevOps AgentやSecurity Agentをプレビュー公開したあたりからです。

<https://aws.amazon.com/about-aws/whats-new/2025/12/devops-agent-preview-frontier-agent-operational-excellence/>

DASH 2026とAWS FinOps Agentが同じ日に重なったのは、おそらく偶然ではないでしょう。別の領域にいた両社が、同じ方向へ同時に進化している。それを示す一日だったと言えます。

## DatadogとAWSの戦略の違い

両者の戦略を、データ源、守備範囲、エージェントの位置づけ、という三つの軸で比較してみます。

### データ源

AWSは、コストデータとCloudTrailを一次データとして所有しています。これはAWSだけが完全に持っているデータです。

AWS FinOps Agentの異常調査は、この特性を直接活かす設計になっています。Cost Anomaly Detectionの検知イベントを起点にし、その前後のCloudTrailイベントを相関させる。CloudTrailは「誰が」「いつ」「どのAPIを」呼んだかを記録しているため、コスト変動を引き起こした操作と、それを実行したIAMユーザーやロール、つまり責任を持つオーナーを特定するところまで自動化されます。出力は、事業要因まで踏み込んだ解釈ではなく、責任を持つエンジニアに直接届くJiraチケットかSlack通知です。

これは、コストが発生する場所とリソースが操作される場所を同時に持っていなければ作れないフローと言えます。サードパーティのFinOps SaaSはCUR（Cost and Usage Report）を取り込めますが、CloudTrailとの突合まで同じ精度で行うのは、データの遅延や権限の面でAWS自身ほど容易ではないはずです。AWSは自社内のAPI呼び出しをほぼリアルタイムで把握できる立場にあり、ここに構造的な優位があります。

Datadogは、APM、ログ、トレース、RUM、プロファイラのテレメトリを一次データとして所有しています。Bits Detectionは、リアルタイムのメトリクスやログに加えて、過去のベースライン、サービストポロジ、オーナーシップ情報、ソースコードのコンテキストまでを参照しながら、いまの状態が健全かを継続的に判断します。根本原因の調査はBits Investigationが辿り、その修正はBits Codeが引き受けて、GitHub上に本番向けのPRまで作ります。

源泉となるデータの種類が違うため、得意な異常も違ってきます。AWSは「お金の異常」、Datadogは「性能と挙動の異常」に強い。両者とも汎用のOpsエージェントを名乗っていますが、内側のデータ源が違う以上、得意領域の重心は自ずと違ってきます。

### 守備範囲

AWSは、自社クラウド内で完結します。マルチクラウドやSaaSはカバーしません。

これは欠点であると同時に、強みでもあります。AWSだけで完結している組織なら、コストから実行ログまでが、すべて同じクラウドの中で揃います。逆に、Snowflake、Databricks、Google Cloud、Vercelなど複数のサービスが混ざっている組織では、AWS FinOps Agentだけでは組織全体のクラウドコストの異常は見えません。

Datadogは、マルチクラウドやSaaSをまたぐうえに、自社インフラに置いたログ基盤まで取り込みます。ログを自身のクラウド環境に置いたまま検索できるBYOC Logsに加え、DASH 2026では、DatabricksやClickHouseといった外部データストアをLog Explorerから同じ構文で横断検索できるFederated Logsも打ち出しました。ログがどこに散らばっていても、Datadogから検索できるという方向に広げています。

<https://www.datadoghq.com/blog/introducing-datadog-byoc-logs/>

組織のオペレーションがどこまで広がっているかで、選ぶ側が変わります。AWS 100%の組織なら前者で十分ですが、複数クラウドやSaaSが混ざっている組織では後者の範囲が必要になる、という分かれ方です。両方使う組織でも、コスト最適化はAWS FinOps Agent、性能と可用性はDatadog Bits、という分業はあり得る構成と言えるでしょう。

### エージェントの位置づけ

この違いこそ、本記事で最も伝えたい部分です。

AWSは、AIエージェントを自社プラットフォーム上で動かすための構成要素として揃えようとしています。Bedrock AgentCoreがその基盤で、Memory（記憶）、Identity（権限管理）、Observability（動作の可視化）、Gateway（外部ツール連携）、Runtime（実行環境）、そしてBrowser ToolやCode Interpreterといった組み込みツールが揃ってきました。DevOps AgentやFinOps Agentは、このレイヤーの上に乗るAWS製の標準エージェントとして位置づけられます。

つまりAWSは、エージェントの実行環境そのものを自社で握り、その上で動く具体のエージェントを順次出してくる戦略です。AgentCoreはLangGraphやCrewAIなど他社のフレームワークにも対応しており、サードパーティのエージェントも同じ基盤に乗ることが期待されています。

Datadogは、AIエージェントを監視対象として外から見ようとしています。同日発表されたDatadog Agent Consoleは、組織で使われているコーディングエージェント（Claude Code、Cursor、GitHub Copilotなど）と、Datadog自身のBits AIを横断で可視化する仕掛けです。誰がどのエージェントをどれだけ使っているか、それがAIを使わない場合と比べて効いているか、コストがどこで無駄になっていて何を直せるか。こうした問いに、ひとつのUIで答えられるようにします。

ここで重要なのは、監視対象にDatadog自身のBits AIも入っているという点です。自社のエージェントも、競合のエージェントも、同じUIで横並びに比較できます。うちのエージェントを使ってくれと訴求するのではなく、あなたの組織のエージェント運用そのものを我々が可視化してあげる、という立ち位置です。

AI Guardも同じ思想の延長にあります。これはAIエージェントをプロンプトインジェクション、ツールの悪用、データの持ち出しといった攻撃から守る仕掛けですが、保護対象はDatadog製エージェントに限りません。環境内の保護されていないエージェントを洗い出し、外部の自作エージェントも対象に含めます。Datadogは、エージェントを監視し、保護し、統治する立場を取りに来ています。

エージェントを所有しようとするAWSと、エージェントを監視しようとするDatadog。この違いが、両者の戦略の核と言えるでしょう。

## エージェントが担う領域の対応表

運用から開発まで、AIエージェントが担う領域を、両者がそれぞれ何で実現しているのかを並べてみます。クラウドサービス全体の比較ではありません。上半分が運用と開発の実務（検知・調査・修復から、コード変更・リリース・テストまで）、下半分がそれを支える基盤（記憶・評価・監視・保護・実行環境）です。この表では、スラッシュは複数の製品が候補になること、プラスは複数を組み合わせて実現することを表します。

| 役割 | Datadog | AWS |
| --- | --- | --- |
| 異常検知 | Bits Detection | Cost Anomaly Detection / DevOps Agent |
| 異常調査 | Bits Investigation | FinOps Agent + CloudTrail / DevOps Agent |
| 自動修復 | Bits Infrastructure（インフラ操作・修復） | DevOps Agent |
| コード変更 | Bits Code | Kiro（仕様駆動の開発環境） |
| リリース検証 | Bits Release | CodePipeline / CodeBuild |
| テスト自動化 | Bits Testing | Kiro |
| 自然言語での質問 | Bits Chat / Bits Data Analysis | FinOps Agent（コストに限定） |
| 知識・記憶 | Bits Memories | AgentCore Memory |
| エージェント評価 | Agent Evals | Bedrockの評価機能 |
| エージェント監視 | Datadog Agent Console（自社外も含む） | AgentCore Observability（自社内） |
| エージェント保護 | AI Guard | AgentCore Identity / Bedrock Guardrails |
| エージェント実行基盤 | Datadogプラットフォーム本体 | AgentCore Runtime |

役割としては大部分が重なります。重なっていない領域や、明らかに強さが違う領域に、それぞれの強みが出ています。

Datadogは、「自社外のエージェントまで監視対象に入れる」拡張性と、Bits CodeやBits Releaseという「検知から修正、検証までのループをDatadog内で閉じる」設計を持っています。

AWSは、CloudTrailという所有データを源泉にした「コスト異常の根本原因特定」と、AgentCore Runtimeという「他社エージェントを自社プラットフォームに引き込む土台」を持っています。後者は、サードパーティの開発者に「AWSの上でエージェントを動かしてほしい」と誘うための装置です。

## 主導権の分岐点は「誰がコードを変えるか」

機能対応表を眺めると、ある争点が浮かびます。検知から調査、コード変更、リリース検証までの流れを、どこまで自社のUIで完結させるか。ここが両社の陣取り合戦になっています。

DatadogのBits Codeは、この観点で象徴的なプロダクトです。公式の説明によれば、Bits CodeはError Tracking、APM Recommendations、Continuous Profiler、Test Optimization、Code Security、そしてBits Investigationからのシグナルを受け取り、エンジニアが普段手でやる一連の作業を肩代わりします。問題のトリアージ、該当コードの特定、修正の記述、テストの実行、そしてPRの起票。ここまでを一手に引き受けます。

つまり、これまで「監視ツールはアラートを出すところまで、修正はIDEとGitHubの仕事」だった分業を、Datadog側に取り込もうとしています。リポジトリ自体はGitHubやGitLab側にありますが、PRを生成する主体はDatadogです。

AWS DevOps Agentも、根本原因特定から修正提案までは行います。ただし、修正対象がアプリケーションコードに踏み込む場合、コードの主舞台がAWS自社サービス上にあるとは限らないため、ここの完結性は弱めです。AWSのコーディングの本命は、仕様駆動を得意とするKiroやBedrock経由のサードパーティエージェントになりやすく、DevOps Agentはそれらと連携する側に回ります。AWSの戦略は、エージェント実行環境としてのAgentCoreを握り、その上で動くサードパーティを含めて自社プラットフォームに乗せる、という異なる道筋です。

ここでDatadog Agent Consoleが効いてきます。組織が使うClaude CodeやCursor、GitHub Copilotを監視対象に入れることで、Datadogの外で動くコーディングエージェントの活動まで可視化の範囲に入ります。Bits Code自身も監視対象になるため、自社製と他社製を同じUIで並べて比較できます。

つまりDatadogは、コーディングエージェント自体を所有していなくても、それを監視する立場を取れば主導権を握れる、というポジションを取りに来ています。一方のAWSは、AgentCoreの上で完結させれば、監視まで自社内で済む、という立場です。

どちらの戦略が成立するかは、エンジニアが普段どこに居るかに左右されます。AWSコンソールに長く滞在するチームならAWS側、DatadogのUIとSlackを主舞台にしているチームならDatadog側、というのが現時点での自然な分かれ目でしょう。IAMやVPC、Cost ManagementのようにAWSコンソール経由でしか直接さわれないリソースが多い組織では、AWS側の優位が効きやすい。Slack通知とPRレビューを軸にオペレーションが回っている組織では、Datadog側がカバーできる範囲が広く伸びていきます。

## 事実から見える予測

ここまで整理してきた事実から、いくつかの予測を引き出してみます。

### 予測1: AgentCoreは、Bedrockの戦略をエージェントで繰り返す

AWSの戦略は、過去のBedrock戦略の再現に見えます。Bedrockは、Anthropic、OpenAI、Metaといった各社のモデルをAWSでホストし、自社のNovaモデルと同じ土俵に並べて、すべてAWSの請求書に乗せる基盤として広がりました。どのモデルが勝つかを当てにいくのではなく、モデルが動く実行環境と課金を握ることで、AI市場の取り分を確保する戦略です。

AgentCore Runtimeはこれと同じ構造を、エージェント市場で再現しようとしています。エージェントそのものを大量に自社で作るのではなく、サードパーティのエージェント開発者に「AWSの上で動かしてほしい」と誘導する装置です。DevOps AgentやFinOps Agentは、その基盤の上で動く標準実装の見本という位置づけになります。AgentCore Memory、Identity、Observability、Gatewayといったコンポーネントは、エージェント開発者が自前で作ると面倒な共通部品を、AWSが肩代わりする仕掛けに見えます。

この戦略が成立するかどうかは、エージェント開発者がAgentCoreの制約を許容するかにかかっています。Bedrockがモデル提供者の取り込みに成功したように、AgentCoreがエージェント提供者の取り込みに成功すれば、AWSは監視・実行・課金のすべてを自社内に抱え込めます。

### 予測2: Bits CodeとBits Releaseの両立が示すDatadogの完結度戦略

DASH 2026で、Datadogはコード変更を担うBits Codeと、リリース検証を担うBits Releaseを同じファミリーにそろえました。Bits Releaseは、コード変更の意図した影響を分析し、検証計画を立て、ステージングでチェックを走らせ、ロールアウトを見届けるエージェントです。この二つが並んだこと自体が、戦略的な意図の表れと読めます。

検知から調査、コード変更、リリース検証までのループを自社内で完結させるには、まずコード変更（修正PR）を握るのが起点になります。コード変更を握れば、その後のリリース検証は「Datadogが出したPRを、Datadogがリリース時に見届ける」という自然な延長になります。逆に、コード変更を他社のIDE側エージェントやGitHub Copilotに握られると、リリース検証だけ自社で持っても、閉じたループにはなりません。

Bits Codeでループの起点を押さえ、Bits Releaseでその出口まで自社に取り込む。この二つがそろったことで、監視から修正、リリースまでをDatadogのUI内で完結させる道筋が見えてきました。まだプレビュー段階の機能も含みますが、ここを通しで仕上げてくる可能性は高いと見えます。

### 予測3: チャットとチケットへの統合は「プッシュ型UX」への大きなシフトの一部

両社とも、エージェントの出力先として、チャットとチケット管理ツールを重視しています。AWS FinOps Agentは出力先をSlackかJiraで指定でき、Datadog Bitsも調査結果や通知を同種のツールに届けます。

これは「ダッシュボードを見に行くUX」から「エンジニアがすでにいる場所に届くUX」への、大きなシフトを示しています。これまで監視ツールもFinOpsツールも、ユーザーがダッシュボードを見に行く前提で作られていました。アラートはメールで届き、詳細はダッシュボードで確認する、という流れです。

エージェントの時代になると、これが反転します。問題が起きたら、解釈と推奨アクションがチャットに直接届く。ダッシュボードを開くのは、深掘りしたいときと、エージェントの判断に納得できないときだけ。ダッシュボードは、エージェントの判断を裏で支える参照資料という位置に下がっていきます。

この変化は、課金モデルにも及び始めています。従来の閲覧数やseat単位の課金に加えて、Datadogはすでにエージェントの作業量に応じたAIクレジットや、調査一件あたりの従量課金を取り入れています。ダッシュボードをどれだけ見るかではなく、エージェントにどれだけ働かせるか。監視の課金の軸は、そちらへ動き始めています。

### 予測4: コーディングエージェント運用は「個人ツール」から「組織ツール」へ

Datadog Agent Consoleが立ち上がったこと自体が、コーディングエージェント運用がジャンルとして成立しつつあることを示しています。Claude CodeやCursor、GitHub Copilotを野良で導入していて、誰がどれだけ使っているかが見えないまま、という状況も起こりがちです。

各ツールも、GitHub Copilotのエンタープライズ向け管理機能やCursorのチーム機能のように、組織での利用を束ねる方向に動いています。ただし個別ツールの管理画面は、基本的にその製品の中で閉じています。Datadog Agent Consoleは、その上に横串の管理レイヤーを乗せようとしています。GitHub Copilotが他社のコーディングエージェントを自分のプラットフォームに取り込み始めているのと、向きは違うが狙いは近い動きと言えます。

「組織のコーディングエージェント運用ダッシュボード」というカテゴリは、すでに立ち上がりつつあります。まだ見えていないのはその先で、コスト配賦のための業界標準のタグ規約や、エージェント単位の生産性指標（PRの採択率、エラー率、採択されたPRあたりのコストなど）の整備が、次に来るところでしょう。

### 予測5: 主導権争いは三つ巴に発展する

ここまではDatadogとAWSの二者で書いてきましたが、実際の主導権争いは三つ巴になりそうです。ソースコードの置き場を持つGitHubやGitLab、実行プラットフォームを持つAWSやGoogle Cloud、Azure、監視プラットフォームを持つDatadogやGrafana Labs、New Relic。それぞれが、エージェント時代の主導権を狙っています。

GitHubは、CopilotとActionsの組み合わせで、コード変更からリリース、モニタリングまでのループをGitHub内で完結させる方向に動いています。GitHub Advanced SecurityにはCopilot Autofixが含まれ、CodeQLが検知した脆弱性に対して修正案のPRを起票するところまで、すでに作り込まれています。

リポジトリ側、実行基盤側、監視側の三者が、それぞれ自分の所有領域から検知・修正・リリースのループを取りに来ています。三者のうちどれかが完全に勝つというより、利用する組織が、自分たちの規模や業種、既存のスタックに合わせて主役を選ぶ。勝者は組織ごとに分かれていく、という決着になりそうです。

### 予測6: FinOpsの組織論的シフト

AWS FinOps Agentがいちばん大きく変えるのは、技術ではなく組織のあり方かもしれません。

FinOpsはこれまで、中央のFinOpsチームが主導する形が主流でした。専門家集団がコストレポートを作り、月次のレビューで開発チームと対話する、という形です。近年は、中央チームが標準やガードレールを整えつつ、各チームに責任を委ねる方向へ移りつつありました。ただ、その担い手はまだ人でした。AWS FinOps Agentは、個々のエンジニアが自然言語でコストを質問でき、異常が起きればエンジニアに直接Jiraチケットが届く設計になっています。

これは、セルフサービスの側に寄せたい組織にとって、その移行を一段進める装置と言えます。これまで人が担っていた分散化を、エージェントが引き受けられるからです。この道を選ぶ場合、FinOpsチームの仕事は、みんなに教えることから、アカウントとオーナーの対応づけやタグ規約を整え、エージェントに渡すコンテキストを設計することへと移っていきます。FinOpsの専門家が消えるわけではなく、活動の中身が、より土台を整える作業に変わっていく。そういう選択肢が現実的になった、という見方です。

似た構図は、ほかの領域でも見られます。SREでは、監視を直接担うことと、監視の仕組みを整えること。Platform Engineeringでは、各チームが自分でインフラを扱えるよう、ガードレールを設けた標準の道筋を用意すること。セキュリティでは、専門チームが一件ずつレビューすることと、ポリシーをプラットフォームのテンプレートに埋め込むこと。専門チームが現場を直接さばくのか、現場が自分で動ける仕組みを整える側に回るのか、という選択がそれぞれにあります。

どちらが正解というわけではありません。完全な中央集権が向く組織もあれば、現場に大きく委ねる組織もあります。多くは、中央が標準とガードレールを用意し、その範囲で現場が自分で動く、という中間に落ち着くでしょう。AIエージェントは、この「現場が自分で動く」側を、人手より低いコストで実現できる選択肢として加わってきました。FinOpsのセルフサービス化もその一例です。AIエージェントの導入は、監視や操作のやり方だけでなく、組織がどこに役割の重心を置くか、その選択肢そのものを広げています。

## 考えられる反論

ここまでの整理に対して、いくつか反論が考えられます。先に挙げて、それぞれに答えておきます。

### 反論1: Datadogのマルチクラウド優位

> マルチクラウドもSaaSも見られるDatadogのほうが、守備範囲で結局は勝つのでは？

たしかにマルチクラウドという守備範囲の広さは、Datadogの強みです。ただしAWSには、コストデータとCloudTrailという、AWS自身しか一次データを持てない領域があります。コスト変動の根拠を最も高い精度で辿れるのはAWS自身だ、という優位は残ります。マルチクラウド組織でも、AWSのコストガバナンスだけはAWS FinOps Agentに寄せる、という分業はあり得るでしょう。監視の一元化と、データ源の精度。両立の難しいこの二つの軸が、ここで問われます。

### 反論2: 自社に閉じることの不利

> AWSは自社クラウドの中に閉じてしまう。守備範囲が狭いぶん不利なのでは？

守備範囲の狭さは、たしかに不利な要素です。ただし、AWSを主力に使う開発者は、すでにAWSコンソールに長く滞在しています。エンジニアが普段触っている画面にそのまま入り込めることは、行動変容のコストを下げる強い武器です。新しいUIに移ってもらう必要がないというのは、ツール採用の現場では大きく効きます。

### 反論3: 両方併用という選択

> どちらかに絞らず、両方導入すればいいだけでは？

実際、これは現実的な選択肢です。コーディングの現場ではすでに、Claude Codeで設計を詰めながら、実装をCodexに任せ、その出力をもう一方にレビューさせる、といった複数エージェントの併用が広がっています。得意分野の違うエージェントを使い分け、片方の結果を片方に検証させる。Opsエージェントでも、同じ世界観は十分に成り立ちます。

ただし、それが成立しているのは、人間が「どれをどの目的で使い、どの出力を信じるか」の判断軸を持っているからです。Opsエージェントは機能の重なりが大きく、異常調査も自動修復も自然言語での質問も、両者でできてしまいます。判断軸のないまま両方を入れると、同じ体験を二か所で抱えるコストだけが残り、エンジニアはどちらに聞けばいいかで迷います。かつての監視ツール乱立と同じ構図です。併用を活かすも殺すも、使い分けの設計次第と言えます。

### 反論4: 結局Datadogから全部見える可能性

> AgentCoreに乗ったエージェントもDatadogから見えるなら、最後はDatadogが押さえるのでは？

これは十分に考えられる未来です。AgentCore ObservabilityはOpenTelemetry互換で、すでにDatadogを含む外部の監視ツールにテレメトリを送り出せます。AWSが実行基盤と課金を握っても、監視の主舞台はDatadog側に残る、という分担はあり得ます。実行は握るが、監視は外に開く。この濃淡が、長期的な主導権の分かれ目になりそうです。

### 反論5: GitHubの参入

> リポジトリを持つGitHubが、同じループを先に作ってしまうのでは？

これはむしろ、「予測5」で触れた三つ巴の話に直結します。GitHubは実際にこの方向へ動いていて、Copilot AutofixとActionsの組み合わせで、コード起点のループを作りつつあります。DatadogのBits Codeは監視起点、GitHub Copilot Autofixはコード起点、AWS DevOps Agentは実行基盤起点。同じループを、それぞれ異なる入り口から押さえに来ている構図と整理できます。

## どこから手をつけるか

ここまでで見極めの軸は揃いました。データ源、守備範囲、エージェントの位置づけ。この三つで各エージェントの素性が読めるようになったところで、実際に何から動くか、という話に移ります。AWSを実行基盤、Datadogを監視、GitHubをコード管理に使っている、という構成を前提に整理してみます。

### ステップ1: 自組織のオペレーションが「何起点」で動いているかを言語化する

最初にやるべきは、ツール選定でも導入検証でもなく、現状の言語化です。インシデントやコスト異常が起きたとき、最初にどこを見て、どこで原因を特定し、どこで修正しているか。この往復経路を一度書き出してみると、自組織がどのベンダーと最も相性がよいかが見えてきます。

判断軸は次のように整理できます。

| 中心になっている課題 | 軸になるエージェント | 判定の根拠 |
| --- | --- | --- |
| コスト最適化・コスト異常の調査 | AWS FinOps Agent | コストデータとCloudTrailの突合精度は、サードパーティでは出しにくい |
| AWS内のインフラ障害・リソース異常 | AWS DevOps Agent | AWS環境に閉じるなら、実行ログまで自社のAPIで取れる |
| マルチクラウドやSaaS横断の性能・可用性 | Datadog Bits | テレメトリの所有量と守備範囲の広さが効く |
| コーディングエージェント運用の可視化 | Datadog Agent Console | 自社外のエージェントを横串で見られるのは現状ほぼ唯一 |
| コード起点のセキュリティ修正 | GitHub Copilot Autofix | リポジトリに密着した修正フローが回しやすい |

多くの組織では、これらが複数同時に当てはまるでしょう。AWSを実行基盤、Datadogを監視、GitHubをコード管理に使っているなら、表の行のほとんどに該当するのではないでしょうか。だからこそ、全部を同時に入れるのではなく、いま最も往復コストが高い経路はどれか、で優先順位をつける必要があります。

### ステップ2: 最も往復コストが高い一経路だけを、まずエージェント化する

すべてを同時にエージェント化しようとすると、評価軸が定まらないまま、いくつものエージェントを一度に試すだけで終わります。現実的なのは、「ステップ1」で書き出した経路のうち、人手の往復が最も発生している一本に絞ることです。

たとえば、コスト異常の調査に毎月まとまった工数を必要としている組織なら、AWS FinOps AgentのPublic Previewを、Slack通知と閾値フィルタ（一定額以上の異常だけ調査させる）の構成で試すのが入口になります。インシデント対応の初動調査が深夜の負担になっているなら、AWS DevOps AgentかDatadog Bits Investigationのどちらかを、自組織の障害がAWS内で完結するか外部のSaaSに及ぶかで選びます。一本に絞れば、人手でやっていたときと比べて何が速くなり、何が信用できないかを、具体的に評価できます。

### ステップ3: 試験導入のうちにコンテキストとタグ規約を整える

エージェントの精度は、渡すコンテキストで大きく変わります。AWS FinOps Agentは、アカウントとオーナーの対応づけ、チーム定義、タグ規約、レビュー周期といったコンテキストファイルを受け取れる設計です。「あるチームのコストは」と聞いて正しいアカウント群に解決できるかどうかは、このコンテキスト整備にかかっています。

ここが大事なのは、コンテキストとタグ規約の整備が、どのベンダーを選んでも無駄にならない投資だからです。アカウントとオーナーの対応表、コスト配賦のためのタグ規約、サービスとチームの紐づけが含まれます。これらはAWS FinOps AgentでもDatadogでも、エージェントに渡す前提情報として効きます。試している段階のうちにここを整えておけば、本格導入に進んだ段階でも、別のベンダーに乗り換える場合でも、そのまま使えます。エージェント本体より先に足元のメタデータを整えるのが、遠回りに見えて近道です。

### ステップ4: コーディングエージェント運用の可視化を、別の流れとして早めに始める

ここはOpsやFinOpsとは少し別の論点ですが、組織のAI活用を健全に運用していくうえで、並行して動かす価値があります。

Claude CodeやCursor、GitHub Copilotを組織で使っているなら、誰がどれだけ使い、いくらコストがかかり、PRがどれだけ採択されているか。こうした数字を取ろうとして、苦労している組織も多いのではないでしょうか。個人の試用から始まった利用も、いずれチームや組織全体へ広がっていきます。その段階で慌てないために、最初からチーム・エンタープライズでの利用を見越しておきたいところです。Datadog Agent Consoleのようなエージェント運用の可視化レイヤーが出てきたいま、たとえ導入は先送りでも、自組織のコーディングエージェント利用をどう可視化するかの議論だけは、早めに始めておくのが安全です。

把握しておきたい指標は、利用者数と利用頻度、エージェント別の支出、PRの採択率、修正の手戻り率あたりです。これらが見えていれば、どのエージェントに投資を寄せるか、どこで無駄が出ているかを、数字で判断しやすくなります。

### 過渡期だからこそ、乗り換えやすさを設計に織り込む

ここまでのステップに共通しているのは、いまはまだ全面的に乗り切らない、という姿勢です。Opsエージェントはプレビュー段階のものが大半で、機能も課金体系も今後変わります。

だからこそ、特定のベンダーの作法に深く依存する作り込みは避け、「ステップ3」で触れたコンテキストやタグ規約のような、どこに乗っても使える資産を厚くしておくのが、過渡期の現実的な構えと言えます。データ源と往復経路を言語化し、メタデータを整え、一経路ずつ検証する。この順番で進めれば、試験導入から本格導入に進んだ段階でも、主導権の構図が変わった段階でも、慌てずに動けます。

## いま、何が起きているのか

両社の発表が同じ日に重なったのは、Opsエージェントというカテゴリが、複数のベンダーが同時に同じ陣地を狙う段階に入った合図と言えるでしょう。監視を強みとするDatadogと、クラウド基盤を持つAWSが、それぞれ自身の強みを源泉にして主導権を取りに来ている。データを多く持つ者がその先のアクションも握る、という構造は、AIエージェント全般に当てはまる動きです。

AWSの戦略は、エージェントの実行環境という下のレイヤーを握り、その上に乗るエージェントを順次自社製でそろえ、外部の開発者も同じ土台に呼び込む、というプラットフォーム型です。Datadogの戦略は、監視対象という外側のレイヤーを握り、自社製も他社製も同じUIで見える化する、というメタプラットフォーム型です。そして三つ巴の三者目として、ソースコードの置き場を握るGitHubやGitLabが、同じ陣地に入ってきています。

どの戦略が前に出るかは、組織が何を起点にオペレーションを回しているかによって変わり、一律の答えは出ません。ただ、この競争の構造そのものは、エージェント時代における主導権争いの縮図と言えます。それぞれが自分の領域を足がかりに、その先まで取りに来ています。インフラを所有する者は、その上で動くソフトウェアの実行環境まで。監視する者は、監視対象を広げた先のアクションまで。コードを預かる者は、コード生成とリリースのループまで。

奪い合われているのは、検知から修正、リリースまでのループを、どこのUIで完結させるかという主導権です。2026年6月9日の同日リリースは、その争いがはっきり表に出た日だったと言えるでしょう。
