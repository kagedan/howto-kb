少し前までの自分は、AIに指示を出すたびに800字くらいの前提説明をコピペしていました。

今は3行です。

変わったのはAIの性能ではなく、\*\*こちらの「渡し方」\*\*です。自分の知識・手順・ルールをファイルにしてAIに常時搭載したら、指示はどんどん短くなり、出力はどんどん安定しました。経費精算SaaSを3日で作れたのも、自分のプラットフォームで販売しているKitを短いプロンプト一発で量産できているのも、土台はぜんぶこの仕組みです。

この記事では、その構築手順を6 Stepに分けて、**使っているプロンプトを全部そのまま公開**します。読みながらコピペしていけば、今日中に「自分仕様のAI」が組み上がる構成です。

---

## なぜ「汎用のまま」だと損なのか

AIの出力がズレる原因の9割は、モデルの性能ではなく**文脈不足**です。あなたの業界用語も、顧客の特徴も、社内のNGワードも、AIは知りません。だから毎回説明する。毎回説明するから、説明の仕方が微妙に変わって、出力の品質もブレる。

ここで大事な事実がひとつあります。

**AIに毎回説明していることは、必ず次の3つのどれかに分類できます。**

分類 中身 置き場所 知識（事実） 商材・顧客・用語・社内事情 ナレッジMD 手順（プロセス） 毎回同じ流れの定型業務 手順書MD → Skills ルール（制約） 文体・禁止事項・毎回守ること CLAUDE.md／プロジェクト指示

この対応表が、この記事の背骨です。「毎回説明していること」を3つに仕分けて、それぞれの置き場所に入れる。やることはこれだけ。ここから6 Stepで順番に組んでいきます。

---

## Step 1：AIに毎回説明していることを洗い出す【棚卸し】

カスタマイズの素材は、新しく考えるものではありません。**自分がAIに毎回コピペ・説明していることの中に、全部あります。**

まずはそれを洗い出します。下のプロンプトを貼ると、AIがインタビュアーになって業務をヒアリングし、毎回説明している内容を「知識／手順／ルール」の3分類で整理してくれます。所要10分。このStepだけでも「自分のAI利用のムダ」が可視化されるので、まずはここから試してみてください。

```
あなたは「AIカスタマイズの設計コンサルタント」です。
私がAIに毎回説明している内容を洗い出し、ドキュメント化すべき素材を特定してください。

# 進め方
1. まず以下を1問ずつ質問してください（まとめて聞かない）：
   - 私の職種・業務内容
   - AIを日常的に使っている作業トップ3
   - その作業でAIに毎回コピペ・説明している情報
   - 「AIの出力がズレる」と感じる瞬間
2. 回答が出揃ったら、私が毎回説明している内容を次の3分類で整理してください：
   - 【知識】業界用語・商材・顧客・社内事情など「事実」
   - 【手順】毎回同じ流れで行う「プロセス」
   - 【ルール】文体・禁止事項・優先順位など「常に守る制約」
3. 最後に、それぞれをどこに書くべきか対応表で出してください：
   知識 → ナレッジMD ／ 手順 → 手順書MD（将来スキル化） ／ ルール → CLAUDE.mdまたはプロジェクト指示

# 出力形式
- 3分類は表で出す
- 各項目に「ドキュメント化の優先度（高・中・低）」を付ける
- 優先度は「説明する頻度 × 説明の長さ」で判断する

私の状況：〔職種や業務をひとことで。例：フリーランスのWeb制作、花き流通のDX支援、経理〕
```

〔　〕の中だけ自分の状況に書き換えて送信してください。以降のプロンプトもすべて同じ方式です。

## Step 2：自分の知識を"AIが読める形"にする【ナレッジMD】

Step 1で「知識」に分類されたものを、MDファイル（Markdownファイル）にします。

ポイントは、人間向けの資料をそのまま渡すのではなく、**AIが誤読しない構造で書き直す**こと。具体的には「1行1情報」「用語は表で定義」「曖昧な形容詞は数字と固有名詞に置き換える」の3つです。

下のプロンプトはヒアリング形式なので、質問に答えるだけでナレッジMDが完成します。

```
あなたは「AIが読みやすいドキュメント（ナレッジMD）の設計者」です。
私の業務知識をヒアリングし、AIに渡すだけで文脈が伝わるMarkdownファイルを作ってください。

# ヒアリング（1問ずつ）
1. このMDを使う場面（例：提案書作成、SNS投稿、顧客対応メール）
2. 事業・商材・サービスの概要
3. 顧客・ターゲットの特徴
4. 業界用語・社内用語（AIが誤解しがちな言葉）
5. 絶対に間違えてほしくない事実・数字
6. やってはいけないこと（NG表現・出してはいけない情報）

# 出力ルール
- ファイル名の候補と一緒に、コードブロックでMD全文を出力する
- 構成：# 概要 ／ ## 前提となる事実 ／ ## 用語集（表） ／ ## よくある文脈 ／ ## NG事項
- 1行1情報。曖昧な形容詞は数字か固有名詞に置き換える
- 「AIへの指示」ではなく「事実の記述」で書く（指示はCLAUDE.md側の役割）
- 全体で100行以内。入りきらない詳細は「別MDに分ける」提案をする

対象の業務：〔例：花き流通業界向けのDX提案資料づくり〕
```

できあがったMDは、会話に添付するか、claude.aiのProjectのナレッジに入れておきます。以後は「このMD前提で」と一言添えるだけ。**800字の前提説明が、ここで消えます。**

## Step 3：定型業務を"手順書MD"にする【SOP化】

次は「手順」です。週1回以上やっている定型業務をひとつ選んで、AIが再現できる手順書に変換します。

手順書MDの威力は**再現性**です。「手順書MD＋今回の実データ」を渡すだけで、誰がいつ実行しても同じ品質の出力が出る。自分はこの型でKit量産の仕組みを組んでいて、今では短いプロンプト一発で同じ構造・同じ品質のアウトプットが出ています。

```
あなたは「業務手順のドキュメント化専門家」です。
私が繰り返しやっている作業を、AIが再現できる手順書MDに変換してください。

# ヒアリング（1問ずつ）
1. 対象の作業（週1回以上やっているもの）
2. その作業の入力（何を受け取って始まる？）
3. 完成形のサンプル（過去の成果物があれば次のメッセージで貼ってもらう）
4. 途中の判断ポイント（「〜の場合は〜する」の分岐）
5. 提出前に必ずチェックしていること

# 出力ルール
- コードブロックでMD全文を出力。構成は次の4部：
  ## 入力（この手順を始めるのに必要なもの）
  ## 手順（番号付き。判断分岐は「もし〜なら」で明記）
  ## 出力形式（完成形のテンプレ or サンプル）
  ## 品質チェックリスト（提出前チェック）
- 手順は「AIが実行する」前提で書く（人間にしかできない箇所は【人間】と明記）
- 1手順＝1アクション。「いい感じに」「適切に」等の曖昧語は禁止

完成後、次のメッセージで私が実データを渡すので、この手順書どおりに実行してテストしてください。

対象の作業：〔例：議事録の清書、週次レポート作成、求人原稿の作成〕
```

プロンプトの最後にある通り、**生成したら必ず実データでテスト**してください。ズレた箇所を手順書側に反映する——この改善ループはStep 6で仕組み化します。

## Step 4：手順書を"呼び出すだけ"のスキルにする【Skills】

Step 3の手順書MD、実は毎回貼るのも手間です。ここで登場するのがClaudeの**Skills**という仕組み。

手順書をスキル化しておくと、/スキル名 と打つだけで呼び出せる上、**関連する依頼をすると自動で発動します**。もう手順書を貼る作業すらなくなります。

仕組みはシンプルで、「フォルダ＋SKILL.md 1枚」を置くだけです。

claude.ai側でも、設定からスキルの追加・管理ができます。

ここで覚えておくべき公式の使い分けがひとつ。\*\*「事実・ルールはCLAUDE.md、手順はSkills」\*\*です。CLAUDE.mdは毎回読み込まれますが、Skillsは使うときだけ読み込まれる。だから長い手順書を置いてもコストがかかりません。

下のプロンプトにStep 3の手順書MDを貼れば、発動条件（description）まで最適化されたSKILL.mdに変換してくれます。

```
あなたは「Claude Skills（Agent Skills）の設計者」です。
以下の手順書MDを、Claudeのスキル（SKILL.md）に変換してください。

# 変換ルール
- 冒頭にYAML frontmatterを付ける：
  ---
  description: （このスキルが何をするか＋どんな依頼のときに使うかを1〜2文で。ユーザーが実際に言いそうな言葉を含める）
  ---
- description は自動発動のトリガーになるため、「〜のとき使う」を必ず含める
- 本文は手順書の内容を簡潔に再構成する（目安100行以内、最大でも500行以内）
- 毎回変わる情報（日付・案件名・今回のデータなど）は本文に書かず「ユーザーから受け取る」と書く
- スキル名（フォルダ名）は英小文字とハイフンのみで提案する
- 最後に、保存場所と使い方を3行で案内する：
  - このプロジェクトだけ：.claude/skills/〔skill-name〕/SKILL.md
  - 全プロジェクト共通：~/.claude/skills/〔skill-name〕/SKILL.md
  - 呼び出し：/〔skill-name〕 と入力、または関連する依頼をすると自動で使われる

# 手順書MD
〔ここにStep 3で作った手順書MDを貼り付け〕
```

コツはdescriptionです。**「〜のとき使う」を、ユーザーが実際に言いそうな言葉で書く**——これが自動発動の精度を決めます。

## Step 5：プロジェクトの"憲法"を書く【CLAUDE.md】

最後の層が「ルール」です。置き場所は**CLAUDE.md**。

CLAUDE.mdはプロジェクト直下に置くと**毎セッション自動で読み込まれる**設定ファイルです。つまりここに書いたルールは「毎回、確実に」効く。だからこそ鉄則はひとつだけ。

**書いてよいのは「毎回真であること」だけ。**

正確なコマンド（テスト・ビルド・起動）、ディレクトリ構成の要点、やってはいけないこと。逆に、長い手順はSkillsへ。リンターで強制できるスタイルルールは書かない。目標は80行以内です。CLAUDE.mdは薄いほど効きます。

自分がSaaSを3日で作れたのも、初日にこの土台を固めてClaude Codeが迷わない状態にしたのが大きいです。Claude Codeには /init という自動生成コマンドもあるので、「/initで叩き台 → 下のプロンプトで削って研ぐ」の順が最速です。

```
あなたは「CLAUDE.mdの設計コンサルタント」です。
私のプロジェクト用に、リーンで効くCLAUDE.mdを作ってください。

# あなたが守る設計思想
- CLAUDE.mdは毎セッション自動で読み込まれる。書いてよいのは「毎回真であること」だけ
- 手順・プロセスが長くなる内容はCLAUDE.mdに書かず、Skills化を提案する
- リンター等のツールで強制できるルールは書かない
- 全体で80行以内を目標にする

# ヒアリング（1問ずつ）
1. プロジェクトの一言説明（何を作っている？）
2. 技術スタック（言語・フレームワーク・DB・デプロイ先）
3. よく使うコマンド（テスト・ビルド・起動。正確なコマンド文字列で）
4. ディレクトリ構成の要点（どこに何がある？）
5. 過去にAIがやらかしたこと・絶対にやってほしくないこと
6. コード規約のうち、ツールで強制できていないもの

# 出力
- コードブロックでCLAUDE.md全文を出す
- 構成：# プロジェクト概要 ／ ## コマンド ／ ## 構成 ／ ## ルール（Do / Don't）
- ルールは禁止事項を優先。理由が必要なものだけ1行で理由を添える
- 最後に「Skillsに切り出すべき候補」があれば指摘する

※エンジニア以外の場合：3は「よく頼む作業」、4は「参照してほしい資料の場所」に読み替えて質問してください。
出力はclaude.aiのProjectの「カスタム指示」に貼る前提の文章にしてください。

プロジェクト：〔一言で。例：Next.js製の経費精算SaaS ／ 個人のSNS運用〕
```

エンジニアじゃない方も大丈夫です。claude.aiのProjectにある\*\*「カスタム指示」がCLAUDE.mdと同じ役割\*\*なので、プロンプト内の注記どおりに作った内容をそこに貼ればOKです。

## Step 6：ズレをフィードバックして"育てる"【運用】

ここまでで環境は完成です。ただし、本当の価値はここからです。

出力がズレたとき、多くの人はその場で指示を書き足します。でも正解は、**指示ではなく「ファイル側」を直す**こと。これを繰り返すと、ドキュメントが使うほど精度の上がる資産になります。これが「AIを育てる」の正体です。

下のプロンプトは、ズレた出力を貼ると「原因はどのファイルのどの行か」を特定し、最小の差分で修正案を出してくれます。

```
あなたは「AIカスタマイズの改善コンサルタント」です。
AIの出力が期待とズレたので、設定ファイル側の修正案を出してください。

# 私が渡すもの
1. 使用した設定ファイル（CLAUDE.md／ナレッジMD／SKILL.mdのいずれか）
2. 実際の指示文
3. 実際の出力（ズレた部分）
4. 期待していた出力

# あなたの仕事
1. ズレの原因を次の3つに分類する：
   A. 情報不足（ファイルに書かれていない）
   B. 記述が曖昧（複数の解釈ができる）
   C. 置き場所が不適切（別のファイル・層に書くべき内容）
2. 修正案を「差分形式」で出す（変更前 → 変更後を並べる。追記なら追記行だけ）
3. その修正が他の用途に悪影響を出さないかを1行でチェックする

# ルール
- ファイル全文の書き直しはしない。最小の差分で直す
- 「その場しのぎの追記」より「原因の1行を直す」を優先する
- 同じ種類のズレが再発しない書き方（数字・固有名詞・具体例）に直す

--- 以下、材料 ---
設定ファイル：〔貼り付け〕
指示文：〔貼り付け〕
実際の出力：〔ズレた部分を貼り付け〕
期待していた出力：〔貼り付け〕
```

その場しのぎの追記でファイルを太らせないことが、長く効かせるコツです。週1回、ズレのログをまとめて反映するだけで十分回ります。

## まとめ：「AIを育てる」の正体はドキュメント運用

最後に全体をおさらいします。

1. **棚卸し**——毎回説明していることを「知識・手順・ルール」に3分類する
2. **ナレッジMD**——知識をAIが誤読しない構造で書く
3. **手順書MD**——定型業務を再現可能な手順にする
4. **Skills**——手順書を"呼び出すだけ"にする（事実はCLAUDE.md、手順はSkills）
5. **CLAUDE.md**——毎回真であるルールだけを80行以内で書く
6. **育成ループ**——ズレたらファイル側を直す

AIを「汎用のまま使う」のと「自分の仕事に合わせて育てる」のとでは、半年後の生産性がまったく別物になります。しかも育てた資産はぜんぶただのテキストファイルなので、モデルが進化しても乗り換え先でもそのまま使えます。
