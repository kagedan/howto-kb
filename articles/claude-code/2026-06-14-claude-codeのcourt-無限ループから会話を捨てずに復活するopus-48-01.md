**Opus 4.8 にしてから、Claude Code が `court` とか `count` みたいな謎の単語を出して、ツール呼び出しが止まる ―― そんなこと、増えていませんか？**

しかもタチが悪いのは、対処法として返ってくるのが大体「**セッションを変えろ**」なこと。そんなことは分かっている。でも、こっちは長く積み上げた会話をどうにか引き継ぎたいのに、その引き継ぎコマンドを打っても court で止まるから困っているんです。新しいセッションにしたら、ここまでの文脈は全部忘れられる。引き継ぎが大事なのは分かっている、なのにまたそこで `court`。仕方なく諦めてセッションを変えて、同じことをもう一度命令して、そこでまた `court`……。

正直、ムカつきます。Claude を使うのを辞めてやろうかと思うほどに。そのうえ、こっちは会話を「進めたい」のに世の中の対処法は「会話を“やめろ”、“セッションを変えろ”」ばっかりなんです。Opus 4.7 に下げろ、履歴を忘れさせろ、新セッションを開け ―― いや分かるけども、それは全部「今までの会話を捨てろ」と言っているのと同じ。

こちらがやりたいのは「Opus 4.8 を使っての相談」です。セッションが壊れた、文脈がいっぱいになった、というなら、それは仕方ない。でもそうなら、せめてしっかり「引き継ぎを完了させ」たい。なんのために課金して Opus 4.8 を使っているのか、という話ですよね。そこで私が欲しいと思ったのは、つきつめると次の2つだけなんです。

1. そもそも court を出したくない、出ても止まらないようにしたい（とにかく進めたい）
2. 出て止まっても、ループに載せずせめて「最後の1個の命令」だけは通したい（会話を捨てずに）

同じことを思っている人がいるはずだと思い、この2つ ―― **会話を捨てずに切り抜ける** ―― を主軸にまとめました。「4.7 に下げる」「履歴を消す」といった“会話を捨てる系”の回避策は、どうしようもないときの最終手段として、いちばん下に置いています。

## まず、court で止まるのはこういう状態です

具体的に何が起きているかというと、ファイルを読んだりコマンドを実行したりする「ツール呼び出し」が、こんなふうに崩れて出力されます。

```
court
<invoke name="Read">
<parameter name="file_path">C:\Users\...\CLAUDE.md</parameter>
</invoke>
```

本来は内部の命令としてそのまま実行されるはずのものに、先頭へ `court`（や `count`、`call`）という余計な単語がくっつき、囲みのタグも崩れています。これをハーネスが「parse できない（malformed）」と判断して、何も実行されないまま止まる、というわけです。

腹立たしいのは、ここで「なんで止まるんだよ、動かしてよ」と急かしても、  
「さっきこういう命令をしたんですが、こんな感じで止まっています。ちゃんと動かしてください」とキャプチャ付きで丁寧に説明しても、その返信がまた

```
court
<invoke name="Read">
<parameter name="file_path">C:\Users\...\CLAUDE.md</parameter>
</invoke>
```

「おいまたかよ！」と軽く机を叩きたくなります。嘘です叩きました何度も。

しかもこれは一度きりでは終わりません。崩れた出力が会話の履歴に残ると、モデルはそれを「正しい書き方」だと勘違いして真似し続けます。これが、何度やり直しても同じところで止まる「court 無限ループ」の正体です。

ただ、実は court が壊すのはあくまで「ツール呼び出し」だけで、普通の文章はちゃんと返ってくるんです。どういうことか？これが、会話を捨てずに切り抜けるための逃げ道になるので順を追って説明します。

## これはデスクトップアプリもCLIも全環境で起きます、Opus 4.8 である限り

最初は「デスクトップアプリ（Code タブ）だけの問題かな」と思っていましたが、違いました。これはモデルがツール呼び出しを生成する瞬間に起きるので、使っている環境を選びません。デスクトップアプリでも、ターミナルの CLI（`claude`）でも、WSL 上の Claude Code でも、条件が揃えば出ます（私自身もWSLなどで同じことが起こるのを確認しています）。

そういう事情なので、この記事の対処法は、まず環境を問わない方法を中心に紹介します。ターミナルでしか使わないコマンド（`--continue` など）が出てくるところは、そのつど「CLI の場合」と断っておくので、デスクトップアプリしか触らない方も「自分には関係ないコマンドばかりだ」と感じずに読めるはずです。

## 会話を捨てずに切り抜ける、3つの段階

ここからが本題です。考え方はシンプルで、「court を消す」ことではなく「会話を残したまま、どうにか前に進める」ことを最優先にします。別に court が出たって動けばいいんです。上から順に試して、できるだけ上の段階で粘ってください。下に行くほど、会話を手放す度合いが大きくなります。

| やりたいこと | 方法 | 会話の残り方 |
| --- | --- | --- |
| 出ても最後の1個を通す | 謝罪させず、ツール呼び出しだけ再送 | そのまま残る |
| そもそも出させない | UserPromptSubmit hook で前置きを禁止 | そのまま残る |
| 壊れた履歴だけ捨てる | `/compact` で要約して圧縮 | 要約として残る |
| どうしても無理なとき | 引き継ぎを作ってから新セッション | いったん手放す |

### ① 出てしまっても、最後の1個だけは通す

一番効くのに、あまり知られていない方法です。しかも会話を1ミリも捨てません。

よく「一度壊れたらリトライしても無駄」と言われますが、これは半分だけ本当です。「説明の文章を書いてからやり直す」とたしかに直りませんが、「文章をいっさい書かず、止まったツール呼び出しだけをそのまま送り直す」と、かなりの確率で通ります。

理由ははっきりしていて、崩れる引き金が **ツール呼び出しの直前に置かれた文章** だからです。「すみません、もう一度実行します」と前置きを書くと、その直後の開始タグがずれて、また court に化けます。前置きさえなければ、開始タグから素直に出てくれるんです。

なので、あなたがやることは「丁寧な説明をやめる」ことです。止まったら長い指示を足さず、「court」「止まってる」「もう一回」くらいの短い一言だけ返してください。  
「止まってます。なんで？」と聞いたり「またとまってるんですがどういうことですか？」「いい加減にしろよ」とかいうとClaudeは「すみません」から入ろうとして「court」になるわけです。  
ブチ切れたくなるのはわかります、私も何度も切れました。でも命令上は「止まってる」これだけです。ここで抑えるんです。

後で紹介する hook や CLAUDE.md を入れておけば、Claude は謝罪の文章を書かずにツール呼び出しだけを出し直します。よかれと思って状況を細かく説明してあげるのが、この症状にかぎっては逆効果なんです。

### ② 連鎖して通らなくなったら、壊れた履歴だけ捨てる

①で粘れなくなったら、崩れた出力が履歴にいくつも溜まっている状態です。それでも、いきなり会話を全部捨てる必要はありません。`/compact` を使えば、会話の中身は要約として残したまま、無限ループの燃料になっている崩れた履歴だけを落とせます。`/clear` が全部消すのに対して、`/compact` は要点を残す中間策だと思ってください。これは CLI でもデスクトップアプリでも、`/compact` と打てば使えます。

とはいえ、スラッシュコマンドに馴染みがない方もいると思います。その場合は無理にコマンドを使わなくて大丈夫です。日本語で「ここまでの会話を要約して、壊れている部分は省いてまとめ直してください」とお願いするだけでも、要点を残したまま立て直すきっかけになります。崩れているのが直近の数ターンだけなら、そこまで戻してしまうのも手です。ターミナルなら Esc を2回押して巻き戻し、デスクトップアプリなら巻き戻せる操作があればそれで戻し、なければ次の③に進みます。

### ③ どうしても戻らないなら、捨てる前に引き継ぎを作る

①も②もダメで、無限ループから抜けられない。そのときだけ、新しいセッションに移ります。正直に言えば、ここからは「会話を諦める」側です。4.7 に下げる、履歴を消す、新しいセッションを開く ―― どれも court は消えますが、積み上げた文脈も一緒に消えます。やりたいことではなく、最後の妥協だと割り切ってください。

だからこそ、捨てる前にひと手間かけます。ここで「court が壊すのはツール呼び出しだけで、文章は壊れない」という性質が効いてきます。止まったセッションでも、文章でならちゃんと答えが返ってくるからです。

注意したいのは、ひとことの要約では足りない、ということです。長い会話だと、要点だけ書いた短いメモを新セッションに渡しても、肝心の「やってきたこと」が抜けて、うまく引き継げません。なので、頼むときは具体的に指定します。たとえば「この作業の目的、前提、これまでに決めたこと、触ったファイルと変更内容、まだ残っている作業、つまずいた点を、新しいチャットに引き継ぐ前提で、できるだけ詳しくまとめてください」と書けば、引き継ぎに耐える記録が文章で返ってきます。それをコピーして、新しいセッションのいちばん最初に貼ってから再開します。

「そんなものはファイルに保存しておけばいいだけでは」と思うかもしれません。そのとおりなのですが、やっかいなのは、その保存（これもツール呼び出しです）が court で止まってしまうことです。つまり「ファイルに書いておけば安全」という当たり前の対策が、肝心の書き込みの瞬間に壊れる。だから順番は逆で、まず文章として吐き出させてから、それを自分でファイルに貼るか、新しいセッション側で保存させる、というのが現実的な手順になります。

新しいセッションに移るときの注意は一つだけ、壊れたセッションを「続けない」ことです。ターミナルなら `/clear` でリセットするか `claude` を起動し直しますが、このとき `--continue`（`-c`）や `--resume` は付けないでください。これらは前回の壊れた履歴を読み戻すフラグなので、せっかく仕切り直しても、また同じところで止まります。デスクトップアプリの方は `--continue` のようなコマンドを打つ機会はありません。壊れたチャットを開き直さず、引き継ぎの文章を持って新しいチャットを始める、とだけ覚えておけば大丈夫です。

..ちなみに私は一応履歴として残しつつもタイトルを「【壊れた】」とかにしています。これが良いのかどうかはおいておいて。

## そもそも court を出させない

ここまでは「出てしまったあと」の話でした。でも毎回これを手でやるのは消耗します。理想は、そもそも止まらないことです。①で見たとおり、止まる引き金は「ツール呼び出しの前に文章を書くこと」でした。だったら、Claude が毎回その前置きを書かないように仕向けてしまえばいい。それを自動でやるのが UserPromptSubmit hook です。

最初は、この回避策を CLAUDE.md やメモリに書いていました。「ツールを使うときは前置きの文章を書かず、ツール呼び出しから始める」「1回の返事でツールは1つだけにする」「重いツールは分けて呼ぶ」といった内容です。書いていることは正しいのですが、セッションをまたぐと再発しました。CLAUDE.md やメモリは会話の開始時に一度読まれるだけなので、やり取りが進むうちに意識が薄れて、また前置きを書いてしまうんです。指示は「お願い」であって「強制」ではない、ということですね。

そこで考え方を変えました。「読んで覚えてもらう」のをやめて、「毎ターン、機械的に差し込む」ことにしたんです。Claude Code の UserPromptSubmit hook は、こちらが入力するたびに発火して、返した内容を Claude のコンテキストに注入してくれます。つまり、注意書きを毎ターン強制的に目の前へ置ける。会話の開始時に一度きりの CLAUDE.md と違って、これなら薄れません。

設定は、フォルダにシェルスクリプトを一つ置いて、それを呼ぶように登録するだけです。`~/.claude/settings.json` をいじれる環境なら、CLI でも WSL でもデスクトップアプリでも効きます（私はデスクトップアプリで効いているのも確認しました）。シェルに馴染みがなければ、この章は飛ばして、前半の対処法だけでも十分役に立ちます。

まず `~/.claude/hooks/court-guard.sh` を作ります。

```
#!/bin/bash
# UserPromptSubmit hook: ツール呼び出し malformed（court/count/call 混入＋プレフィックス欠落）対策を毎ターン注入する
cat <<'JSON'
{"hookSpecificOutput":{"hookEventName":"UserPromptSubmit","additionalContext":"[ツール呼び出し malformed 対策] ツールを使う応答は必ず守る: (1)本文テキストを一切前置きせず、ツール呼び出しだけで応答を開始する (2)1応答1ツール（連続させない）(3)ネストJSON引数の重いツール（browser_batch等）は単発に分解 (4)説明・報告はツール成功後の別メッセージで出す (5)malformedで止まったら謝罪文を書かず直前のツールだけ即再出力。同一ターンで2回以上続いたら新セッションへの切り替えを提案する"}}
JSON
```

実行できるようにしておきます。

```
chmod +x ~/.claude/hooks/court-guard.sh
```

最後に `~/.claude/settings.json` へ登録します。

```
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          { "type": "command", "command": "$HOME/.claude/hooks/court-guard.sh" }
        ]
      }
    ]
  }
}
```

これで毎ターン、Claude が答えを書き始める直前に「前置きを書くな、ツールは1つ、止まったらツールだけ出し直せ」という注意を見るようになります。体感では、止まる回数がはっきり減りました。

一つだけ、誤解されやすい点を書いておきます。これは hook で malformed を「ブロックする」しくみではありません。この崩れ方はパースの前の段階で起きるので、PreToolUse hook は発火すらしないんです（崩れた呼び出しは、そもそもツールとして認識されないため）。UserPromptSubmit hook がやっているのは、生成のたびに「前置きを書かせない」よう仕向けて、引き金を踏ませないこと。直接の防御ではなく、起きる確率を下げるための運用だと理解しておくのが正確です。

ほかに効いたのは、タスクを小さく分けてやり取りを短く保つことと、`/compact` をこまめに打って履歴を膨らませないことです。Opus 4.7 に下げるという手もありますが、それは「会話を進める」よりも「court を消す」ための選択なので、私はまず hook と短いやり取りと①で粘っています。

## どんなときに踏みやすいか

私の観測と各所の報告を合わせると、踏みやすい条件はだいたい決まっています。長く続けて文脈が大きく膨らんだセッション、とくに常時立ち上げているような長い会話。ツール呼び出しの引数に日本語が混ざるとき。ツールを連続で呼んだり、重い引数のツールを呼んだりするとき。このあたりが重なると、ぐっと出やすくなります。

念のため確認しましたが、これは自分の環境だけの不運ではありません。GitHub の claude-code の issue にも同じ現象が複数上がっていて、「文章を前置きすると壊れて、ツール呼び出しから書き始めると起きない」「前のバージョンでは出ない」「長い文脈で多発する」と、再現条件まで一致しています。つまり、すでに知られている挙動です。ただ、モデルの内部で実際に何が起きているかを外から断定はできないので、ここでは「自分の環境ではこう踏んで、こう直った」という話と、「こう報告されている」という話を分けて書いています。

## おわりに

大事なのは、目的を取り違えないことだと思っています。やりたいのは「court を消すこと」ではなく、「積み上げた会話を捨てずに、前に進めること」です。だから順番はいつも同じになります。まず、止まっても文章を書かずにツール呼び出しだけを送り直して、最後の1個を通す。それでもダメなら `/compact` で壊れた履歴だけ落とす。どうしても無理なときだけ、引き継ぎの文章をしっかり作ってから新しいセッションに移る。そして次からは、UserPromptSubmit hook で前置きを毎ターン禁止して、そもそも止まらないようにしておく。

「リトライは無駄」というのは半分だけ本当で、無駄なのは文章を付けたリトライだけです。文章を外してツール呼び出しだけを送れば通る。そして court が壊すのはツール呼び出しだけで、文章は無事だからこそ、最悪でも引き継ぎは取り出せる。この二つを覚えておくだけで、止まっても会話を手放さずに済みます。同じところで消耗している方の役に立てば、うれしいです。

## 最後にもう一言

ここまで書いて思うのは、「なんでこんなことユーザーが気にしなきゃいけないんだよ」ということ。しっかりして欲しいですが、すっかり依存しているので抜け出せないというのはAI時代ならではの悩みです。  
この前 Fable が出てすぐ禁止されてしまいましたが、モデルによって振り回される、AIによって振り回される、これはもう仕方ないことなんでしょうね。
