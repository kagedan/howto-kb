# [備忘録]Fableを何に使った？

米国太平洋時間で2026年7月1日~2026年7月7日まで、限られた時間、限られた利用量のFableをどう使ったのかを備忘録として記事にします。

## おすすめの使いどころ

**過去の作業履歴や設定ファイルを読み込んで、次の一手を自分で洗い出し、地固めする系のタスク**に使うべきだと判断した。具体的には次のような場面で使いました。

* 溜まったルール・設定ファイルの棚卸しと整理
* 使用状況データ（insights等）を踏まえた改善の洗い出しと実装
* 過去の実装履歴・セッション履歴を横断して次の開発計画を立てる

体感ですが Opus4.8 に比べて、ルールの追記先や既存記述との整合など仕様が曖昧な分岐点で「どちらの方針にするか」を都度質問してくる頻度が高く、複数の改善案を出したうえで採用理由まで揃えてから実装に進む傾向があるように感じました。  
サブスクリプション利用になる場合でも、難易度の高い**新機能の要件定義や設計**で使うとよさそうに思いました。(どれくらい料金がかかるか次第)

## 実際にやった改善

今回、エージェント運用のルールやプロジェクトの一部をFableで手直しした。

1. **ローカルのルールファイルの見直し**

   * **目的**Sonnet5やOpusでFable5の性能(行動原則の忠実性)を発揮できるように、既存の `CLAUDE.md` の構成を見直す
   * **与えた情報**: 既存の `~/.claude/rules/` 配下のルール群（自律実行系の基準が存在しないことが分かった）と、CLAUDE.mdの構成
   * **Fableの提案**: 「常時ロードされる rules/ 方式にする」「hooksでの機械的強制はせずガイダンスに留める」という設計方針とあわせて、タスク分解・自己検証・次アクション判断・報告の4章構成とRed Flags表を持つ新規ルールファイル `rules/autonomy.md` の全文を提案された
   * **提案の所管**: 既存のsuperpowersスキル群やplan強制hooksを置き換えず、その上に乗る「毎ターンの判断アルゴリズム」に絞られており、既存ルールと矛盾しない設計だった
2. **insightsの結果を踏まえた改善**

   * **与えた情報**: 使用状況の分析結果（insights）の改善提案と、それを「Rules / Skills / Hooks」に落とし込むという指示、既存の `rules/` `skills/` `hooks/` `settings.json` の現状
   * **Fableの提案**: 壊れていた並列作業ルールの再作成、既存ルールへの調査ファースト規則の追記、専用スキル3本の新設、運用実態に合わせたルール修正、レビュー用スキルの新規作成、設定の日次バックアップhookの追加、後片付け依頼と横断検証まで含む7タスクの実装計画を提示された
   * **提案の所管**: 壊れていたルールなどはコンテキスト不足で意図しないplanを作った(それ以外は特に不満点はない)
3. **開発エージェントの列挙とplan作成**

   * **与えた情報**: 過去の実装セッション履歴と、現行のルール・設定ファイル一式を読み込ませた。
   * **Fableの提案**: フロントエンド開発を役割ごとに分担する、複数の開発エージェント構成案を提示された
   * **提案の所管**: 提案された役割分担が自分の開発フローに近く、まず導入してみて不満が出れば直す前提で受け入れられそうだった
4. **ループ型エンジニアリング基盤の構築**

   * **与えた情報**: 特定のタスクハンドリングツールと、タスク内の役割分担を担うツール群を組み合わせたいという要件、既存のルール・スキル構成。
   * **Fableの提案**: 前者でタスクをハンドリングし、タスク内の計画・実装は後者の役割分担で回す構成案を提示された。
   * **提案の所管**: 役割分担が明確で、チケット起点で「計画 → 実装(TDD) → 検証」のループを自動で回す運用に合いそうなため採用した

## まとめ

Fableは、地味だが効果の大きい環境改善を、履歴や設定を読み込んで自分で拾い上げてくれる点が強みだった。次にFableを使うなら、コードを書かせるより先に、**溜まった運用ルールや過去の作業ログを読ませて改善点を洗い出させる**使い方が良さそうでした。
