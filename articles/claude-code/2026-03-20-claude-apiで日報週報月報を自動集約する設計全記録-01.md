## はじめに — 20年ぶりにコードに触れたおじさんが、Claude Codeとアプリを設計した話

20年ぶりにコードに触れました。

IT企業に勤めてはいるものの、プログラミングからは長く離れていた46歳のおじさんです。最近Claude Codeを使い始めて、また開発を試しています。

きっかけは、日報の集約作業でした。毎日書いている日報を週報にまとめ、さらに月報にまとめる——この作業を手動でやるたびに「もう少し楽にならないか」と思っていました。5日分の日報を読み返して週報にまとめるだけで30分以上かかります。月報となると、4〜5週分の週報をさらに抽象化するので、もっと時間がかかります。

「Claude APIに日報を渡したら、自動で週報や月報にまとめてくれるアプリが作れるのでは？」

そう思い立って、Claude Codeと一緒に設計から始めました。この記事では、そのアプリの設計を全記録として共有します。

!

この記事で得られること：

* Claude APIを「集約エンジン」として活用するアーキテクチャパターン
* ADR（Architecture Decision Record）を使った技術選定の意思決定プロセス
* 月内週方式という独自の週区切りロジックの設計と実装
* Claude Codeと一緒に設計するAI駆動開発の実践知

## 全体アーキテクチャ — 3層集約パイプライン

### システム構成図

このアプリは、Next.js App Router + SQLite/Prisma + Claude API のフルスタック構成です。入力から集約まで、1つのNext.jsプロジェクト内で完結します。

```
┌─────────────────────────────────────────────────┐
│                   クライアント                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ WebUI    │ │ ファイル  │ │ マイク音声入力   │ │
│  │ 直接入力 │ │ アップ   │ │ (Web Speech API) │ │
│  └────┬─────┘ └────┬─────┘ └───────┬──────────┘ │
└───────┼────────────┼───────────────┼────────────┘
        │            │               │
        ▼            ▼               ▼
┌─────────────────────────────────────────────────┐
│              Next.js API Routes                  │
│  ┌──────────────────────────────────────────┐   │
│  │           入力パーサー層                  │   │
│  │  MD │ TXT │ PDF │ Excel │ テキスト│音声  │   │
│  └──────────────────┬───────────────────────┘   │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐   │
│  │       RawInput（元データ保存）            │   │
│  └──────────────────┬───────────────────────┘   │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐   │
│  │         Claude API（集約エンジン）         │   │
│  │  日報生成 │ 週報集約 │ 月報集約           │   │
│  └──────────────────┬───────────────────────┘   │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐   │
│  │         Prisma ORM + SQLite              │   │
│  │  raw_inputs → daily → weekly → monthly   │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### データフローの全体像

データは以下の流れで処理されます。

```
入力ファイル/テキスト/音声
    │
    ▼
[パーサー] → 統一テキスト形式に変換
    │
    ▼
[RawInput保存] → 元データをDBに保存（再生成に備える）
    │
    ▼
[日報生成] → Claude APIで構造化された日報を生成
    │
    ▼
[DailyReport保存] → 日報をDBに保存
    │
    ▼
[週報集約] → 月内週の日報をClaude APIで集約（粒度↑）
    │
    ▼
[月報集約] → その月の週報をClaude APIで集約（粒度↑↑）
```

ここで重要な設計思想は\*\*「元データを残して再生成可能にする」\*\*ことです。アップロードされたファイルのパース結果をRawInputとしてDBに保存しておくことで、プロンプトを改善した後に日報を再生成できます。AIの出力品質はプロンプト次第で大きく変わるので、この「再生成可能」な設計は後々とても助かります。

## 技術選定をADRで記録した — 4つの意思決定

このプロジェクトでは、技術選定の判断をADR（Architecture Decision Record）として記録しました。ADRとは「なぜその技術を選んだのか」を構造化して残すドキュメントです。Claude Codeと一緒に設計する中で、選択肢の比較や理由の言語化を自然にやってくれたので、ADRという形に落とし込みやすかったです。

### ADR-001: Next.js App Routerの採用

**決定**: Next.js の App Router を採用する。

フルスタック一体型を選んだ最大の理由は、フロントエンドとAPIを1つのプロジェクトで管理できることです。

| 観点 | 選定理由 |
| --- | --- |
| フルスタック | フロントエンドとAPIを一つのプロジェクトで管理可能 |
| Server Components | 大量のレポートデータ表示時にクライアント負荷を軽減 |
| API Routes | ファイルアップロードやClaude API呼び出しのバックエンドを統合 |
| TypeScript対応 | 型安全な開発が可能 |
| デプロイ | Vercel等へのデプロイが容易 |

個人開発で「フロントエンド用のサーバー」と「API用のサーバー」を別々に立てるのは管理コストが高すぎます。Next.jsならApp Router内でAPI Routesも書けるため、1プロジェクトで完結します。

### ADR-002: SQLite + Prisma ORM

**決定**: SQLite を Prisma ORM 経由で使用する。

これは開発フェーズと公開フェーズの2段構えを見据えた選定です。

| フェーズ | DB | 理由 |
| --- | --- | --- |
| 開発（現在） | SQLite | サーバーレス、設定不要、個人利用に最適 |
| 公開（将来） | PostgreSQL | 同時接続対応、スケーラビリティ |

Prisma ORMを使う最大のメリットは、DBの切り替えが**設定1行の変更で済む**ことです。`schema.prisma`の`provider`を`"sqlite"`から`"postgresql"`に変えるだけで移行できます。Prismaが差異を吸収してくれるため移行のハードルは低いですが、データ型の違いやマイグレーション履歴の扱いなど、いくつか注意点はあります。

```
datasource db {
  // 開発中は SQLite を使用
  // 公開時に PostgreSQL に切り替え（provider = "postgresql"）
  provider = "sqlite"
}
```

さらに、テーブル設計は入力データ別テーブル方式を採用しました。Claude Codeと一緒に4つのパターンを比較検討した結果です。

| パターン | 概要 | 不採用の理由 |
| --- | --- | --- |
| 全データ1テーブル | 日報・週報・月報を1テーブルで管理 | typeカラムでの分岐が複雑になり、保守性が低い |
| レポートと入力を同一テーブル | RawInputとレポートを分離しない | 再生成時に元データの取り回しが困難 |
| **入力データ別テーブル方式（採用）** | 元データ→日報→週報→月報を独立テーブルで管理 | — |
| JSON埋め込み方式 | 構造化データをJSONカラムに格納 | SQLiteでのJSON検索性能に懸念 |

採用した入力データ別テーブル方式の構成は以下の通りです。

```
raw_inputs（元データ）
    │ N:1
    ▼
daily_reports（日報）
    │ N:1
    ▼
weekly_reports（週報）← 月内週方式
    │ N:1
    ▼
monthly_reports（月報）
```

元データ → 日報 → 週報 → 月報の4層がそれぞれ独立したテーブルとなり、集約の各段階でデータを参照・再生成できます。

### ADR-003: Claude APIによる集約処理

**決定**: Claude API（Anthropic SDK）を使用して集約処理を行う。

日報→週報→月報の集約には、テキストの**意味理解**と**粒度変換**が必要です。ルールベースの要約（正規表現やキーワード抽出）では品質が不十分でした。

| 比較軸 | ルールベース要約 | Claude API集約 |
| --- | --- | --- |
| 日本語理解 | パターンマッチのみ | 文脈を理解した要約 |
| 粒度制御 | 固定ルール | プロンプトで柔軟に指定 |
| 出力形式 | テンプレート固定 | 構造化Markdown出力 |
| コスト | 無料 | API従量課金 |

コストはかかりますが、集約の品質を考えるとClaude APIを使う価値は十分にあります。特に「日報レベルの細かい記述を週報レベルの粒度に変換する」処理は、AIでなければ実現が難しいです。

### ADR-004: 月内週方式

**決定**: 月内週方式を採用する。

これが4つのADRの中で最も議論に時間をかけた選択肢です。「週報を月報に集約する際、週が月をまたいだらどうする？」という問題に対して、4つの選択肢を比較検討しました。

| 方式 | 概要 | 問題点 |
| --- | --- | --- |
| A. 月〜金で区切り、月報は日報ベース | 月報が週報を経由しない | 3層構造が崩れる |
| B. 月またぎ週報を両月に含める | 1つの週報が2つの月報に重複 | データ整合性が複雑 |
| C. 週の月曜日が属する月に所属 | シンプル | 月初・月末の日が所属しないケースが発生 |
| **D. 月内週方式（採用）** | 週は必ず1つの月に収まる | 第1週・最終週が短い場合がある |

月内週方式のルールはシンプルです。

* **第1週** = 月の1日 〜 その月最初の日曜日
* **第2週以降** = 月曜 〜 日曜（7日単位）
* **最終週** = 最後の月曜 〜 月末日

この方式の最大の利点は「**月報 = その月の全週報**」ときれいに集約できることです。週が月をまたがないため、1つの週報は必ず1つの月報にのみ所属します。

```
例: 2026年4月
  4月1週: 4/1（水）〜 4/5（日） — 5日間
  4月2週: 4/6（月）〜 4/12（日）— 7日間
  4月3週: 4/13（月）〜 4/19（日）— 7日間
  4月4週: 4/20（月）〜 4/26（日）— 7日間
  4月5週: 4/27（月）〜 4/30（木）— 4日間
```

実際の企業（派遣契約の勤務報告等）でも使われている方式なので、ユーザーにとっても馴染みがあります。

## DB設計 — 入力データ別テーブル方式と月内週方式のスキーマ

### テーブル構成と主要カラム

Prisma schemaから主要モデルを見てみます。

**RawInput（元データ）**

```
model RawInput {
  id               String   @id @default(cuid())
  inputType        String   // FILE_UPLOAD / WEB_INPUT / VOICE_INPUT
  fileType         String?  // MARKDOWN / TEXT / PDF / EXCEL
  originalFileName String?  // 元のファイル名
  rawContent       String   // パース後のテキスト内容
  fileSize         Int?     // ファイルサイズ（バイト）

  dailyReport   DailyReport? @relation(fields: [dailyReportId], references: [id])
  dailyReportId String?

  createdAt DateTime @default(now())

  @@map("raw_inputs")
}
```

`rawContent`にパース済みのテキストを保存します。元ファイルがExcelでもPDFでも、ここにはプレーンテキストが入ります。

**DailyReport（日報）**

```
model DailyReport {
  id         String   @id @default(cuid())
  reportDate DateTime // 対象日
  title      String   // 日報タイトル
  content    String   // Markdown形式の完全な日報

  // 構造化された個別項目（一覧表示・検索用）
  highlights  String? // 本日のハイライト
  issues      String? // 課題・問題点
  nextActions String? // 明日の予定
  notes       String? // 所感・気づき
  workHours   Float?  // 作業時間（概算）

  prompt String? // 生成に使ったプロンプト

  rawInputs      RawInput[]
  weeklyReport   WeeklyReport?  @relation(fields: [weeklyReportId], references: [id])
  weeklyReportId String?

  @@unique([reportDate]) // 同じ日の日報は1つだけ
  @@map("daily_reports")
}
```

### 構造化カラムの設計意図

ここで注目してほしいのは、`content`（全文）と個別項目（`highlights`, `issues`等）の**二重保存パターン**です。

`content`にはClaude APIが生成した完全なMarkdown形式の日報を保存します。一方、`highlights`や`issues`などの個別項目にも同じ内容を分割して保存します。

なぜ二重に保存するかというと、一覧表示や検索では個別項目を使い、詳細表示では`content`全文を使うためです。これにより、一覧画面で「今日のハイライトだけ見たい」「課題がある日だけ抽出したい」といった操作が高速にできます。

もう1つ重要な設計判断は、`prompt`カラムの存在です。日報・週報・月報の各モデルに`prompt`カラムがあり、**生成に使ったプロンプトをDBに保存**しています。これにより、後からプロンプトを振り返って改善サイクルを回すことができます。

**WeeklyReport（週報）の月内週方式カラム**

```
model WeeklyReport {
  id          String   @id @default(cuid())
  year        Int      // 年（2026）
  month       Int      // 月（1〜12）
  weekInMonth Int      // 月内の第何週か（1〜6）
  weekStart   DateTime // 週の開始日
  weekEnd     DateTime // 週の終了日

  // KPT形式の振り返り
  kptKeep      String? // Keep（続けること）
  kptProblem   String? // Problem（問題点）
  kptTry       String? // Try（挑戦すること）

  @@unique([year, month, weekInMonth]) // 同じ月内の同じ週番号は1つだけ
  @@map("weekly_reports")
}
```

`@@unique([year, month, weekInMonth])`の複合ユニーク制約により、「2026年4月第2週」のような組み合わせが一意に決まります。月内週方式だからこそ、この制約がきれいに機能します。

## 集約ロジックの設計 — 粒度をどう変えるか

### 日報→週報→月報の粒度変換ルール

集約の核心は「粒度の変換」です。日報では細かく書かれた内容を、段階的に抽象度を上げていきます。

| レポート | 粒度 | 内容例 |
| --- | --- | --- |
| 日報 | 細かい | 「ログイン画面のCSSを修正した。ボタン配置のズレが原因だった」 |
| 週報 | 中程度 | 「認証機能の改善: ログイン画面のUI修正とパスワードリセット機能を実装完了」 |
| 月報 | 粗い | 「ユーザー認証基盤の整備を完了。問い合わせ削減が見込まれる」 |

各レベルの出力構成も明確に定義しています。

* **日報**: 本日の実績 / 課題・問題点 / 明日の予定 / 所感・気づき / 作業時間の概算
* **週報**: 今週の主な成果 / プロジェクト別進捗 / 課題と対応状況 / 来週の計画 / KPT（Keep / Problem / Try）振り返り（KPTとは、「続けること」「問題点」「挑戦すること」を整理する振り返りのフレームワークです）
* **月報**: 今月のハイライト / 目標に対する達成状況 / 月次サマリー / 改善アクション / 来月の重点項目

### Claude API呼び出しの実装例

実際のClaude API呼び出し部分のコードを見てみます。

まず、Anthropic SDKのクライアント初期化です。

```
import Anthropic from "@anthropic-ai/sdk";

const globalForAnthropic = globalThis as unknown as {
  anthropic: Anthropic | undefined;
};

export const anthropic =
  globalForAnthropic.anthropic ??
  new Anthropic({
    apiKey: process.env.ANTHROPIC_API_KEY,
  });

if (process.env.NODE_ENV !== "production") {
  globalForAnthropic.anthropic = anthropic;
}

// 使用するモデル（コスト抑制のため Sonnet を使用）
export const MODEL_ID = "claude-sonnet-4-20250514";
```

`globalThis`を使ったシングルトンパターンで、開発中のホットリロード時にクライアントが複数生成されることを防いでいます。ホットリロードのたびに`new`されると接続数が増え続けてメモリを圧迫するため、`globalThis`に保持して再利用しています。これはNext.js公式ドキュメントでも推奨されているパターンです。モデルはコスト抑制のためSonnetを使用しています。

次に、週報生成プロンプトの組み立て関数です。

```
export function buildWeeklyReportPrompt(
  year: number,
  month: number,
  weekInMonth: number,
  weekStart: string,
  weekEnd: string,
  dailyReports: { reportDate: string; content: string }[]
): { system: string; user: string } {
  const system = `あなたは業務報告の集約を担当するアシスタントです。
複数の日報を読み、1週間の週報として集約してください。

## 出力ルール
- Markdown形式で出力してください
- 個別タスクの詳細ではなく、プロジェクト単位の進捗にまとめてください
- 日単位の課題は、週を通じた傾向・パターンに昇華してください
- 細かい作業ログではなく、成果・アウトプットに焦点を当ててください
- 数値データ（作業時間、完了タスク数等）があれば定量化してください

## 出力構成（必ずこの順序・見出しで出力）
1. ## 今週の主な成果
2. ## プロジェクト別進捗
3. ## 課題と対応状況
4. ## 来週の計画
5. ## 振り返り（KPT）
   - ### Keep（続けること）
   - ### Problem（問題点）
   - ### Try（挑戦すること）

上記以外の見出しやセクションは追加しないでください。`;

  const dailyContents = dailyReports
    .map((r) => `### ${r.reportDate}\n${r.content}`)
    .join("\n\n---\n\n");

  const user = `以下は ${year}年${month}月 第${weekInMonth}週（${weekStart}〜${weekEnd}）の日報です。
これらを集約して週報を作成してください。

---

${dailyContents}`;

  return { system, user };
}
```

ここでのポイントは3つです。

1. **system promptでロールと出力構成を厳密に指定**しています。「業務報告の集約を担当するアシスタント」というロールを与え、出力するセクションの順序まで指定します
2. **粒度変換のルール**をsystem promptに明記しています。「個別タスクの詳細ではなく、プロジェクト単位の進捗にまとめる」「日単位の課題は週の傾向に昇華する」といった具体的な指示です
3. **user promptに日報を連結して渡す**構成です。日報間を`---`で区切り、日付ごとのヘッダーを付けて構造を明確にしています

### エラーハンドリングとリトライ設計

Claude API呼び出しは外部サービス依存のため、エラーハンドリングが重要です。設計方針は以下の通りです。

* **リトライ**: APIエラー（5xx系）の場合は指数バックオフで最大3回リトライ
* **レート制限**: 429エラーの場合はRetry-Afterヘッダーに従って待機
* **パースエラー**: 構造化Markdown出力が崩れた場合は、再リクエストまたは`content`全文をそのまま保存

特にパースエラーへの対応は重要です。Claude APIが指示通りのMarkdown構成で返さない場合もあるため、各セクション（`## 今週の主な成果`等）の見出しが見つからなければ`content`フィールドに全文を入れ、個別項目は`null`にするフォールバック処理を設けています。

### Claude APIプロンプト設計方針

集約レベルに応じたロール指定がプロンプト設計の要です。

| 集約レベル | ロール指定 | 視点 |
| --- | --- | --- |
| 日報生成 | 日報アシスタント | 作業者目線で細かく記録 |
| 週報集約 | マネージャー視点 | プロジェクト単位で進捗をまとめる |
| 月報集約 | 経営報告視点 | 中長期目標に対する達成度 |

出力は構造化Markdownで返すよう指示しています。Markdownなら人間も読みやすく、パースも比較的容易です。また、生成に使ったプロンプトをDBの`prompt`カラムに保存し、「このプロンプトで生成したらこの品質の出力が得られた」という記録を残すことで、プロンプト改善サイクルを回せるようにしています。

## 月内週方式の実装 — week-utils.ts を読み解く

### ルールの定義

月内週方式のルールを改めて整理します。

* **第1週** = 月の1日 〜 その月最初の日曜日
  + 1日が月曜なら → 1日〜7日（7日間）
  + 1日が水曜なら → 1日〜5日（5日間）
  + 1日が日曜なら → 1日のみ（1日間）
* **第2週以降** = 月曜 〜 日曜（7日単位）
* **最終週** = 最後の月曜 〜 月末日

この方式はISO 8601の週番号（W01〜W53）とは異なる独自の体系ですが、「月報 = その月の全週報」という集約の整合性を最優先した結果です。

### TypeScript実装のポイント

`getWeeksInMonth()` の実装を見てみます。

```
export interface MonthWeek {
  weekInMonth: number;
  weekStart: Date;
  weekEnd: Date;
}

function getNextSunday(date: Date): Date {
  const day = date.getUTCDay(); // 0=日, 1=月, ..., 6=土
  if (day === 0) return new Date(date);
  const daysUntilSunday = 7 - day;
  const result = new Date(date);
  result.setUTCDate(result.getUTCDate() + daysUntilSunday);
  return result;
}

export function getWeeksInMonth(year: number, month: number): MonthWeek[] {
  const firstDay = new Date(Date.UTC(year, month - 1, 1));
  const lastDay = new Date(Date.UTC(year, month, 0));

  const weeks: MonthWeek[] = [];
  let weekNumber = 1;

  // 第1週: 1日〜最初の日曜日
  const firstSunday = getNextSunday(firstDay);
  if (firstSunday <= lastDay) {
    weeks.push({
      weekInMonth: weekNumber++,
      weekStart: new Date(firstDay),
      weekEnd: new Date(firstSunday),
    });

    // 第2週以降: 月曜〜日曜（7日単位）
    let currentStart = new Date(firstSunday);
    currentStart.setUTCDate(currentStart.getUTCDate() + 1);

    while (currentStart <= lastDay) {
      const weekEnd = new Date(currentStart);
      weekEnd.setUTCDate(weekEnd.getUTCDate() + 6);

      weeks.push({
        weekInMonth: weekNumber++,
        weekStart: new Date(currentStart),
        weekEnd: weekEnd <= lastDay ? new Date(weekEnd) : new Date(lastDay),
      });

      currentStart = new Date(weekEnd);
      currentStart.setUTCDate(currentStart.getUTCDate() + 1);
    }
  }

  return weeks;
}
```

実装上の重要なポイントは**UTC基準で日付処理している**ことです。`Date.UTC()`と`getUTCDay()`/`setUTCDate()`を使っています。

なぜUTC基準にするかというと、JavaScriptのDateオブジェクトはローカルタイムゾーンの影響を受けるためです。サーバーとクライアントのタイムゾーンが異なると、同じ日付でも「何曜日か」の判定が変わってしまい、週の区切り位置がずれます。UTC基準にすることで、どの環境でも同じ結果を返すことが保証されます。

## 6種類の入力形式をどう統一するか

### 対応形式と処理フロー

このアプリでは6種類の入力形式に対応しています。

| 形式 | 拡張子 | 処理ライブラリ | 最大サイズ |
| --- | --- | --- | --- |
| Markdown | .md | 組み込み（文字列処理） | 5MB |
| テキスト | .txt | 組み込み（文字列処理） | 5MB |
| PDF | .pdf | pdf-parse | 10MB |
| Excel | .xlsx, .xls | xlsx | 10MB |
| WebUI直接入力 | - | - | - |
| 音声入力 | - | Web Speech API | - |

すべての入力は最終的に「統一テキスト」に変換されてからClaude APIに渡されます。

```
入力（各形式）
    │
    ▼
[形式判定] → 拡張子・MIMEタイプで判定
    │
    ▼
[パーサー選択] → 形式に応じたパーサーを呼び出し
    │
    ▼
[テキスト抽出] → プレーンテキストに変換
    │
    ▼
[文字コード正規化] → UTF-8に統一
    │
    ▼
[統一テキスト] → Claude APIへ渡す形式
```

### 各形式の実装上の注意点

形式ごとに考慮すべき点があります。

* **Excel**: 複数シートがある場合は全シートを処理します。ヘッダー行を自動検出し、空行・空列は除去します。数式は計算結果の値を取得します
* **PDF**: テキストベースのPDFのみ対応です。スキャン画像PDFは非対応ですが、将来のOCR対応を検討しています
* **テキスト**: BOM付きUTF-8に対応し、Shift-JIS等の日本語エンコーディングも検出・変換します
* **音声入力**: Web Speech APIを使用し、ブラウザ側で音声認識を行います。音声データそのものはサーバーに送信しません（プライバシー保護）

### パーサー層の設計パターン

各パーサーは共通インターフェースに従って実装しています。新しい入力形式（例えばWord文書への対応）を追加する場合も、同じインターフェースに従ったパーサーを追加すれば、既存のフローに自然に組み込めます。

バリデーションでは、ファイルサイズの上限チェック（MD/TXTは5MB、PDF/Excelは10MB）と、MIMEタイプの検証を行います。変換エラーが発生した場合は、エラー内容をユーザーに返し、手動でのテキスト入力を案内するフォールバックフローを用意しています。

## セキュリティと公開戦略 — 開発→本番の移行パス

セキュリティは開発フェーズと公開フェーズの2段構えで設計しています。

| 項目 | 開発フェーズ（現在） | 公開フェーズ（将来） |
| --- | --- | --- |
| 認証 | なし（ローカル開発） | NextAuth.js（Auth.js） |
| DB | SQLite | PostgreSQL |
| APIキー | 環境変数 | 環境変数（変更なし） |
| データ分離 | 不要（個人利用） | 全テーブルにuserIdカラム追加 |
| APIルート | 認証チェックなし | 全ルートに認証チェック追加 |

公開時の移行手順は以下の通りです。

1. Prismaの`provider`を`"postgresql"`に変更（設定1行）
2. 全テーブルに`userId`カラムを追加
3. APIルートにNextAuth.jsの認証チェックを追加
4. DBアクセスはすべてAPI Routes経由（DB接続情報はサーバー側のみ）

Prisma ORMのおかげで、DB切り替えが設定1行で済む点は非常に心強いです。スキーマファーストの開発により、マイグレーションも管理しやすくなっています。

## Claude Codeと一緒に設計して分かったこと

### Vibe codingの驚き

最近「Vibe coding」という言葉を知りました。AIと対話しながらコードを書いていくスタイルのことだそうです。私もまさにそれをやっていたわけですが、驚いたのは**コードを書くだけではなく、設計まで一緒にやってくれる**ということです。

20年ぶりにコードに触れた自分にとって、Claude Codeは単なるコード生成ツールではありませんでした。「日報を週報にまとめたい」という曖昧な要件を伝えたら、テーブル設計の選択肢を4パターン提示してくれて、それぞれのメリット・デメリットを比較してくれたのです。それがADR-002の入力データ別テーブル方式の選定に繋がりました。

### アジェンダファースト方式

このプロジェクトでは「アジェンダファースト」という開発方式を採用しました。**コードを1行も書く前に、必ず作業計画（アジェンダ）を作成して承認を得る**というルールです。

```
① 指示を受ける
    ↓
② アジェンダを作成して提示する
    ↓
③ 3視点の事前レビューを自動実施
    ↓
④ 承認 → コード実装へ
```

「3視点レビュー」とは、UI/デザイン視点、シニアプログラマ視点、セキュリティ視点の3つの角度から事前にチェックすることです。これをClaude Codeが自動で実施してくれます。

### ADRをClaude Codeに書かせた体験

ADR（Architecture Decision Record）の作成は、Claude Codeとの設計で最も手応えを感じた部分です。

例えば月内週方式（ADR-004）では、「週報を月報に集約するとき、週が月をまたいだらどうする？」という問いかけに対して、Claude Codeが4つの選択肢を提示してくれました。それぞれの長所・短所を比較した上で、データの整合性を最優先するなら月内週方式が良いという提案を受け、採用しました。

自分の言葉では「週が月をまたぐと面倒」くらいしか説明できなかった設計判断が、Claude Codeとの対話を通じてADRとして構造化されました。「自分では説明しにくいけれど、確かにそうだと納得できる設計」を言語化してくれる——これがAI駆動設計の大きなメリットだと感じています。

### AI駆動設計で感じたメリットと注意点

メリットとして強く感じたのは以下の3点です。

1. **設計の言語化が強制される**: Claude Codeに伝えるために、自分の頭の中の曖昧な構想を言葉にする必要があります。これが結果的に設計品質を上げます
2. **選択肢の網羅性**: 自分だけでは思いつかない選択肢（例えばテーブル設計の入力データ別テーブル方式）を提示してくれます
3. **壁打ち相手としての有用性**: 「これで大丈夫かな？」と聞くと、懸念点を指摘してくれます。一人で開発していても、レビューを受けている感覚があります

一方で注意点もあります。AIの提案を鵜呑みにしない姿勢は重要です。最終的な判断は自分がしなければなりません。Claude Codeが提案した4つのテーブル設計パターンのうち、入力データ別テーブル方式を選んだのは自分の判断です。

> **AIは選択肢と比較材料を提供してくれますが、決めるのは人間の役割です。**

「この業務にはどれが合うか」を判断するには、現場の文脈や運用の制約を知っている人間の視点が不可欠です。

## まとめ

この記事では、Claude APIで日報→週報→月報を自動集約するアプリの設計を全記録として共有しました。

1. **集約パイプラインは「元データ保持 + 3層テーブル」で設計する** — 元データを残すことで、プロンプト改善後の再生成が可能になります
