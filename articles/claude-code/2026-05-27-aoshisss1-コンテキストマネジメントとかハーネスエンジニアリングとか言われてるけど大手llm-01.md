コンテキストマネジメントとかハーネスエンジニアリングとか言われてるけど、大手LLMは内部ロジックとして処理できるからいい感じに過去メモリを参照・編集してくれるのはいいけど、自分でメモリ編集するときは向いてないよね。Hermes Agentは逆に管理過剰になりがちで、中間がほしい

これindex作るのがいいようだけど、多分index自体を自動生成するのがいいよね。メモリはすでにこの機能あったと思うが、やってくれないのかな。明示的にyaml, jsonあたりで確認できたらそんなに問題なさそう。多分週1、月1ぐらいでauditするぐらいのバランスでいいと思う。

結構やること多い

- Skill Pack: プロジェクト単位で整理
- Skill Cart規約: メタ情報分類
- shadowing lint: Skill検出用スクリプト
- Skill Router, Index: SQLiteから候補選定

メモリ、ドキュメントは人力整理すると散らかりやすいのは自明だから、やっぱり整理用のスキルが必要になるだろうな。今のところ正常動作してるけど、分類が下手くそ化すると、この方法も破綻することになるし。
llm-wiki的文脈にまとめあげるほうがいいか

コンテクスト対話で考えるとCodexは遅いからChatGPT寄りのレスポンススピードが早いから、コンテクストを自動読み込みした上で、API挙動のチャットUXが最高なんだよね。
「共通基盤作れ」とはチャッピー言うてる
