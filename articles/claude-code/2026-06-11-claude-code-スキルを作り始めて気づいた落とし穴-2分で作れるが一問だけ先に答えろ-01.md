スキル（Skills）を最初にいくつか作ると、手応えがあります。毎回の締め作業が一発で揃う。セッション開始時の現在地が一瞬で戻る。感動して次々と作りたくなり、気づくと似たスキルが乱立して「どれを使うか」で迷い始めます。

役割の異なる4人の使用者にヒアリングしたところ、全員が同じ一線で失敗していました。

---

**その問いとは:**

YESなら → 2分で作れます（後述）  
NOなら → 作っても壊れます（その理由を先に読んでください）

---

## NOの理由：判断を型に入れると壊れる

先に「NO」の話をします。壊れる理由を知ってから作る方が、早い。

失敗例1: 複数フェーズを統括するスキルを作ろうとして過剰設計に陥り、退却。総括や優先度の判断はそのエージェントの核心価値そのものであって、手順に落とせるものではありませんでした。

失敗例2: ブレストの発散プロセスをスキルに固めた。毎回同じステップで考えさせたら、出てくるアイデアが同じパターンに収束して発散の意味が消えました。「角度が死んだ」という表現がぴったりでした。

4人全員が踏んだ失敗の芯は同じ。**「判断」を型に入れた瞬間に壊れます。**

スキルが効くのは「配管」だけです。毎回同じ順序で動く段取り（締め手順・記録・定型チェック）を固めます。「何を考えるか」は固めません。

---

## YESの場合：2分で作れる

作業が終わった直後に、そのまま貼って使えます。

```
今やった一連の手順を、次から一発で呼べるスキルにして。
description（何をするスキルか）は「デプロイ前チェックを実行する」で。
```

AIが手順をまとめて `.claude/skills/<名前>/SKILL.md` を作ります。手で書く必要はありません。

作ったあとは、Claude Codeのセッションで「レビューして」のように普通に頼むと自動で使われます。確実に呼びたいときは `/デプロイ前チェック` のようにスキル名を直接指定できます。

descriptionだけ丁寧に書きます。起動するかどうかはここで決まります。

```
# 良い
description: デプロイ前チェックを実行する

# 悪い
description: デプロイのやつ
```

あいまいだと、必要な場面で呼ばれず死蔵されます。

---

## まとめ

着手前に一問。

**「その作業は、次回も同じ手順で再現できるか？」**

YES → 今すぐ「これ、スキルにして」と頼んでください。2分で終わります。  
NO → 固めません。その判断は自分で握り続けます。
