## **時点**: 2026-05 時点での主要 SaaS / OSS を統合

## 短答

> **ハーネスは薄く、契約面（標準プロトコル）で囲む。モデルが賢くなったら剪定し、新サービスは標準準拠の差し替えで吸収する。**

Claude の知性を活かすエージェント設計の3パターン が示した3原則を中核に、**10 層のコンポーネント** を各層独立の標準で接続する。

---

## 全体像

タスクが投入されてから完了するまでの **データフロー** と、各層を横断する2つの **クロスカット層**（観測 / 供給契約）を1枚にまとめると下記になる。読み進める前にこの図でメンタルモデルを作っておくと、後段の各層詳細が「どこの話か」を見失わずに読める。

### ざっくり読み方

| ステージ | 図中の流れ | 何が決まる層か |
| --- | --- | --- |
| **仕事を受け取る** | L10 → L9 | どの起点 (Linear / GitHub) からエージェントが動き出すか |
| **長時間実行に耐える** | L9 内部 | プロセスが落ちても進捗を失わない（journal/replay） |
| **複数エージェントで分担** | L9 → L8 | 役割分担・並列化・品質ゲートのパターン選択 |
| **1 エージェントを動かす** | L8 → L7 | 思考→ツール→観察のループ実装 |
| **能力を呼ぶ** | L7 → L6 → L4 | ツール（MCP）・スキル・コード実行（サンドボックス） |
| **記憶する** | L7 → L5 | 長時間・クロスセッション知識の永続化 |
| **モデルに問い合わせ** | L7 → L2 → L1 | フォールバック・キャッシュを経て LLM へ |
| **常に守る（横断）** | L3 → L6/L4 | ツール呼び出し前にポリシー判定 |
| **常に観る（横断 X）** | Obs → 全層 | OTel GenAI semconv で span / metric を流す |
| **常に契約（横断 Y）** | Repo → L7/L6 | AGENTS.md がリポジトリ指示、APM が依存管理 |

実線（→）は**実行時のデータフロー**、点線（- ->）は**横断的・条件的な参照**を表す。

各層の候補・判断軸は §2 以降で詳述する。

## 1. 設計哲学（3原則）

Claude の知性を活かすエージェント設計の3パターン は Anthropic Platform チームのハーネス設計指針として、以下の3原則を提示している：

| 原則 | 含意 |
| --- | --- |
| **Claude が知るツールを使う** | `bash + text editor` 基盤、MCP（Model Context Protocol） で拡張。新規ツールの乱立を避ける |
| **止められることを問う** | コンテキスト管理・サブエージェント分岐・要約の判断をモデル側に寄せる |
| **境界は慎重に** | キャッシュヒット最大化、不可逆操作のみ専用ツールでゲート、観測性のため型付きツール |

ハーネスは「Claude がひとりではできないこと」に関する仮定をコードに埋め込んでいる。モデル能力が向上するたび、その仮定は陳腐化する。**定期的に仮定を再検証し、剪定する**運用が前提となる（Claude の知性を活かすエージェント設計の3パターン）。

加えて OpenAI Symphony が示した **「SPEC.md を canonical、実装は参考」** が次の階段。`WORKFLOW.md` のような**リポジトリ所有の契約**を中心に据えれば、ハーネスをコーディングエージェント自身に書き換えさせる仕様駆動の運用に到達できる。

## 2. 10 層アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│ L10  Task Source                                         │
├─────────────────────────────────────────────────────────┤
│ L9   Durable Execution / Orchestration                   │
├─────────────────────────────────────────────────────────┤
│ L8   Multi-Agent Coordination                            │
├─────────────────────────────────────────────────────────┤
│ L7   Agent Runtime / Loop SDK                            │
├─────────────────────────────────────────────────────────┤
│ L6   Capability Layer（Tools / Skills）                  │
├─────────────────────────────────────────────────────────┤
│ L5   Memory & State                                      │
├─────────────────────────────────────────────────────────┤
│ L4   Execution Sandbox                                   │
├─────────────────────────────────────────────────────────┤
│ L3   Governance / Policy（実行前判定）                   │
├─────────────────────────────────────────────────────────┤
│ L2   LLM Gateway / Router                                │
├─────────────────────────────────────────────────────────┤
│ L1   Model                                               │
├─────────────────────────────────────────────────────────┤
│  X   Observability（OTel GenAI semconv 横断）            │
├─────────────────────────────────────────────────────────┤
│  Y   Supply Chain & Repo Contract                        │
└─────────────────────────────────────────────────────────┘
```

各層を **目的 → 契約面 → 候補 → 判断軸** の構造で整理する。

---

### L1 Model

| 項目 | 内容 |
| --- | --- |
| **目的** | 推論・計画・対話 |
| **契約面** | OpenAI 互換 Chat Completions / Anthropic Messages API |
| **候補** | Opus 4.7（Claude Opus 4.7 ）/ GPT-5.5（GPT-5.5 ）/ Qwen3.6（Qwen3.6-35B-A3B / Qwen3.6-27B ） |
| **判断軸** | フロンティア性能 vs コスト vs オンプレ可否 |

**サービス vs 自前**：**SaaS API が原則**。自前学習は能力面・コスト面で非現実的（Claude Opus 4.7 や GPT-5.5 のようなフロンティアモデルの再現コストを考えれば自明）。オンプレ要件があるなら Qwen3.6-35B-A3B のような OSS モデルをセルフホストする選択肢が現実的。

---

### L2 LLM Gateway / Router

| 項目 | 内容 |
| --- | --- |
| **目的** | 複数モデルへの統一インターフェース・ルーティング・フォールバック・キャッシュ・予算管理 |
| **契約面** | OpenAI 互換 API（事実上の標準） |
| **候補** | OpenRouter（SaaS、200+ モデル）/ LiteLLM（MIT OSS）/ Portkey（managed enterprise、セマンティックキャッシュ） |
| **判断軸** | プロトタイプ→OpenRouter、データ主権→LiteLLM、$10k 超/月でセマンティックキャッシュが欲しい→Portkey |

**サービス vs 自前**：3 候補の比較と決定木は LLM Gateway / Router 比較（OpenRouter / LiteLLM / Portkey） にまとまっており、**OpenAI 互換 API を提供するもの（OpenRouter / LiteLLM）に寄せる**と Gateway 自身の置き換えコストが下がる。Portkey は独自 SDK 依存のため移行コストが残る。Advisor Strategy（アドバイザー戦略） のモデル切り分けを実行するには Gateway が事実上必須。

---

### L3 Governance / Policy

| 項目 | 内容 |
| --- | --- |
| **目的** | ツール呼び出し・リソースアクセス・エージェント間通信の **実行前** 決定論的判定 |
| **契約面** | YAML / OPA-Rego / Cedar（Microsoft Agent Governance Toolkit (AGT) ） |
| **候補** | Microsoft Agent Governance Toolkit (AGT)（AGT、OWASP 10/10 カバー、p50 0.012ms） |

**サービス vs 自前**：**常に自前 OSS（AGT）を挟む**。プロンプトベースの守備が red-team 試験で 26.67% の違反率を出すのに対し、AGT のアプリ層強制は 0.00%（Microsoft Agent Governance Toolkit (AGT)）。Microsoft 自身が「アプリ層ガバナンスは OS 分離ではない」と明示し、**コンテナ分離との層状防御**を推奨している。マネージドサービス側ポリシーだけに任せると OWASP Top 10 for Agentic Applications (2026) の ASI-01〜10 を網羅できない。

---

### L4 Execution Sandbox

| 項目 | 内容 |
| --- | --- |
| **目的** | FS / ネットワーク / プロセス分離。プロンプトインジェクション・サプライチェーン汚染対策 |
| **契約面** | 統一規格なし（POSIX FS + ネットワーク allowlist） |
| **候補（ローカル / OSS）** | コーディングエージェントのサンドボックス技術 の3層：OS native（Seatbelt / bubblewrap）/ コンテナ（Docker / gVisor）/ MicroVM（Apple VZ / Docker Sandbox）+ Rivet agentOS （Wasm + V8、コールドスタート 6.1ms） |
| **候補（SaaS）** | クラウドコードサンドボックス比較（E2B / Daytona / Modal / Vercel） ：E2B（Firecracker microVM、~150ms cold start）/ Daytona（Docker、27〜90ms、無期限永続、HIPAA/SOC2）/ Modal（gVisor、GPU 強い）/ Vercel Sandbox / Claude Code on the Web（Anthropic 公式クラウド実行面） （Anthropic 公式 VM、4 vCPU / 16GB、追加課金なし） |

**サービス vs 自前**：**最低 1 層のサンドボックスは必須**（コーディングエージェントのサンドボックス技術）。アプリ層除外設定は突破される前提で設計する（OWASP Top 10 for Agentic Applications (2026) ASI04）。ローカル開発なら OS ネイティブ、長時間自律実行なら MicroVM、マルチテナント大量並列なら Rivet agentOS（32× 安価）。本番 SaaS は クラウドコードサンドボックス比較（E2B / Daytona / Modal / Vercel） の決定木で選ぶ。Claude Code を使うなら Claude Code on the Web（Anthropic 公式クラウド実行面） が**追加課金なし**で Anthropic マネージド VM + 専用 GitHub proxy + Auto-fix PR を一括提供するため第一候補。

---

### L5 Memory & State

| 項目 | 内容 |
| --- | --- |
| **目的** | 長時間タスク・クロスセッション知識・チーム共有知識 |
| **2系統** | (a) **ファイル系**（memory folder） / (b) **構造化メモリ系**（Facts / Events / Instructions / Tasks） |
| **候補（ファイル系）** | ローカル FS / Amazon S3 Files |
| **候補（構造化系）** | Cloudflare Agent Memory （Llama 4 Scout 抽出 + Nemotron 3 合成 + 5 並行検索 + RRF） |

**サービス vs 自前**：両系統は補完関係。**ファイル系**（Amazon S3 Files）はエージェント自身に管理を任せる用途で、Claude の知性を活かすエージェント設計の3パターン のパターン2c「memory folder」と直接対応する。**構造化系**（Cloudflare Agent Memory）は自動抽出 / 分類 / 検索が必要なときに使う。後者はパイプライン再実装が困難なため、ほぼ SaaS 一択。

---

### L6 Capability Layer

| 項目 | 内容 |
| --- | --- |
| **目的** | 外部システム接続（ツール）+ 手続き知識の段階開示（スキル） |
| **契約面** | MCP（Model Context Protocol）（JSON-RPC 2.0、Tools / Resources / Prompts） + Agent Skills （agentskills.io オープン標準） |
| **候補** | MCP サーバー（公式・サードパーティ）/ SKILL.md パッケージ |

**サービス vs 自前**：**プロトコルは標準にコミット**するのが鉄則（MCP（Model Context Protocol） / Agent Skills とも中立組織主導）。サーバ実装は OSS or 自前。標準化レイヤを受け入れることで個別サーバの差し替えが容易になる。Microsoft Agent Governance Toolkit (AGT) の MCP Security Scanner で tool poisoning / typosquatting / 隠し命令を検出することで、ここの信頼性を底上げできる。

---

### L7 Agent Runtime / Loop SDK

| 項目 | 内容 |
| --- | --- |
| **目的** | 単体エージェントのループ実装（思考 → ツール呼出 → 結果取込 → 継続） |
| **候補（マネージド）** | Claude Managed Agents （$0.08 / session-h）/ Claude Code on the Web（Anthropic 公式クラウド実行面）（追加課金なし、Claude.ai アカウント認証） |
| **候補（OSS SDK）** | Claude Code Agent SDK / OpenAI Agents SDK（次世代進化版） （Agent / Runner / Handoffs / Guardrails）/ OpenHands Software Agent SDK / Hermes Agent |

**サービス vs 自前**：プロトタイプ → マネージド、本番カスタマイズ → OSS SDK、データ主権 → OSS セルフホスト。Anthropic 公式の運用ループを使うなら Claude Code on the Web（Anthropic 公式クラウド実行面） が**追加課金なし** で、Claude Code /ultrareview 等の上位機能ともシームレスに繋がる。

---

### L8 Multi-Agent Coordination

| 項目 | 内容 |
| --- | --- |
| **目的** | 役割分担・並列化・品質ゲート |
| **パターン（ 6 類型）** | (a) **アドバイザー戦略**（Advisor Strategy（アドバイザー戦略））/ (b) **並列独立検証**（Claude Code /ultrareview）/ (c) **Manager-Workers**（HiClaw 、Matrix プロトコル）/ (d) **会話駆動**（AutoGen）/ (e) **グラフ状態マシン**（LangGraph）/ (f) **役割ベースのチーム**（CrewAI） |
| **候補（フレームワーク）** | マルチエージェントフレームワーク比較（AutoGen / LangGraph / CrewAI） ：AutoGen（Microsoft、maintenance mode）/ LangGraph（LangChain、durable execution ビルトイン、エンタープライズ採用 34% 一位）/ CrewAI（最速プロトタイプ 2〜3 日、Fortune 500 の 60%） |

**サービス vs 自前**：用途で分かれる：

* **コンプライアンス重視**（金融・医療・規制業界）→ **LangGraph**（durable execution + 監査証跡、マルチエージェントフレームワーク比較（AutoGen / LangGraph / CrewAI））
* **業務ドメイン役割分担**（営業 / マーケ / サポート）→ **CrewAI**（マルチエージェントフレームワーク比較（AutoGen / LangGraph / CrewAI））
* **コスト最適化**（フロンティア知性 × 大量実行）→ Advisor Strategy（アドバイザー戦略）（Sonnet 実行 + Opus 相談、Haiku + Opus アドバイザーで Sonnet 比 85% コスト削減）
* **品質ゲート**（マージ前レビュー）→ Claude Code /ultrareview パターン（複数レビュアーが独立に検証 → 再現可能な指摘のみ集約）

Microsoft Agent Governance Toolkit (AGT) が3フレームワーク全てを統合先として明示しているため、L3 governance とは直交して組み合わせ可能。

---

### L9 Durable Execution / Orchestration

| 項目 | 内容 |
| --- | --- |
| **目的** | 長時間実行・多段ステップ・失敗許容・人間承認待機 |
| **契約面** | ワークフロー定義ファイル（OpenAI Symphony の `WORKFLOW.md` がリファレンス的） |
| **候補** | Temporal（耐久実行エンジン） （journal / replay、Series D で評価額 $5B、Serverless Workers on AWS Lambda、Workflow Streams、External Payload Storage with S3、OpenAI Agents SDK（次世代進化版） サンドボックス GA）/ Cloudflare Dynamic Workflows / 耐久実行 for AI Agents （per-tenant コード、V8 アイソレート）/ Open Agents（Vercel Labs） Workflow SDK / OpenAI Symphony（issue → PR 上位オーケストレータ） |

**サービス vs 自前**：Temporal（耐久実行エンジン） と Cloudflare Dynamic Workflows / 耐久実行 for AI Agents の使い分けが重要：

* **事前定義ワークフロー + AWS Lambda 中心** → Temporal（耐久実行エンジン） Serverless Workers
* **テナント毎に異なるエージェント計画** → Cloudflare Dynamic Workflows / 耐久実行 for AI Agents（per-tenant コード読み込み + V8 アイソレートでアイドルコストほぼゼロ）
* **issue → PR の上位オーケストレータ** → OpenAI Symphony を Temporal（耐久実行エンジン） / Cloudflare Dynamic Workflows / 耐久実行 for AI Agents の上に重ねる構成

Temporal（耐久実行エンジン） の journal / replay は自作の差が出にくいためマネージド or OSS セルフホストでよく、自前実装は推奨しない。データ主権要件があれば Temporal OSS セルフホスト。

---

### L10 Task Source

| 項目 | 内容 |
| --- | --- |
| **目的** | エージェントを動かす「仕事」の在処 |
| **候補** | Linear / Jira / GitHub Issues / Slack（Hermes Agent ゲートウェイ）/ Cron |

**設計原則**：OpenAI Symphony §11.5 が示すとおり、**オーケストレータは tracker reader に徹し、書き込みはコーディングエージェント側のツールに任せる**。これにより責務が分離され、tracker の置き換え（Linear → Jira 等）が容易になる。

---

### X 横断：Observability

| 項目 | 内容 |
| --- | --- |
| **契約面** | OpenTelemetry GenAI Agent Observability （GenAI semantic conventions、Agent Application 規約は確定版へ移行済） |
| **候補（可視化）** | Datadog / Honeycomb / New Relic（OpenTelemetry GenAI Agent Observability でベンダ対応を明示） |
| **候補（計装）** | Langfuse / Traceloop / Langtrace、フレームワークベイクイン（CrewAI / AutoGen / LangGraph / Semantic Kernel / PydanticAI、OpenTelemetry GenAI Agent Observability および マルチエージェントフレームワーク比較（AutoGen / LangGraph / CrewAI）） |

**理想**：**計装層は OTel GenAI semconv にコミット**、可視化基盤は差し替え可能に保つ。OpenTelemetry GenAI Agent Observability は MCP（Model Context Protocol） / Agent Skills と同列の「層と層の契約面」として標準化が収束中。

---

### Y 横断：Supply Chain & Repo Contract

| 項目 | 内容 |
| --- | --- |
| **契約面（リポジトリ指示書）** | AGENTS.md（コーディングエージェント向け統一指示書フォーマット） （Linux Foundation 管理、20+ ツール対応、60,000+ プロジェクト） |
| **契約面（依存管理）** | Agent Package Manager (APM) （`apm.yml` で Agent Skills / MCP（Model Context Protocol） サーバー / Hooks を宣言） |

**サービス vs 自前**：**AGENTS.md（コーディングエージェント向け統一指示書フォーマット） を共通指示の在処に**、ツール固有差分は CLAUDE.md / `.cursor/rules/` / `.github/copilot-instructions.md` に分離する（AGENTS.md（コーディングエージェント向け統一指示書フォーマット））。Agent Package Manager (APM) は OWASP Top 10 for Agentic Applications (2026) の ASI04（サプライチェーン汚染）対策としてインストール前のセキュリティスキャンを提供する。

## 3. 「サービス vs 自前」決定木

```
データ主権 / GDPR / ZDR 要件?
├─ YES → セルフホスト OSS 優位
│        L2: LiteLLM
│        L4: Rivet agentOS / Docker / Daytona BYOC
│        L7: OpenHands Software Agent SDK / Hermes Agent
│        L9: Temporal（耐久実行エンジン） selfhost
│        L5: ローカル FS / 自社 S3
│
└─ NO → スケール段階で判断
    ├─ プロトタイプ（〜$2k / 月）
    │   → 全層 SaaS / マネージド
    │     L2: OpenRouter / L4: Daytona 無料$200 or Claude Code on the Web（Anthropic 公式クラウド実行面）
    │     L7: Claude Managed Agents
    │     L9: Cloudflare Dynamic Workflows / 耐久実行 for AI Agents or Temporal（耐久実行エンジン） Cloud
    │     L5: Cloudflare Agent Memory
    │
    ├─ 成長期（$2k〜$10k / 月）
    │   → ハイブリッド。**L3 Governance だけは自前 OSS（Microsoft Agent Governance Toolkit (AGT)）を必ず挟む**
    │
    └─ 規模期（$10k+ / 月）
        → 選択的セルフホスト + マネージド
          L2: LiteLLM or Portkey
          L4: Rivet agentOS（32× 安価）/ E2B Pro（$150 / 月 + microVM 分離）
          L9: Temporal（耐久実行エンジン） Cloud or Cloudflare Dynamic Workflows / 耐久実行 for AI Agents
```

### 「絶対に自前を挟む」べき層

| 層 | 理由 |
| --- | --- |
| **L3 Governance** | プロンプト層守備 26.67% 違反 vs アプリ層強制 0.00%。Microsoft 自身が層状防御を推奨（Microsoft Agent Governance Toolkit (AGT)） |
| **L4 Sandbox（最低 1 層）** | アプリ層除外設定は突破される前提（コーディングエージェントのサンドボックス技術） |
| **X 観測（OTel 計装）** | ベンダーロック回避のため OTel semconv 準拠（OpenTelemetry GenAI Agent Observability） |
| **Y AGENTS.md** | リポジトリにコミットされる中立フォーマット（AGENTS.md（コーディングエージェント向け統一指示書フォーマット）） |

### 「ほぼ SaaS でよい」層

| 層 | 理由 |
| --- | --- |
| **L1 Model** | 自前学習は能力・コストで非現実的 |
| **L5 Memory（構造化系）** | Cloudflare Agent Memory のパイプライン再実装はコスト過大 |

| **L4 Sandbox（補助）** | クラウドコードサンドボックス比較（E2B / Daytona / Modal / Vercel） のいずれかを SaaS で使うと運用負荷激減 |

## 5. アーキテクチャ設計の3つのメタ原則

### 5.1 中立組織主導の標準にコミット

各層で **「ロックインを避けるロックイン」** に賭ける。技術的にはベンダーへの依存だが、その先が**中立組織**である選択は将来の差し替え可能性を最大化する：

| 層 | 標準 | 中立組織 |
| --- | --- | --- |
| L2 | OpenAI 互換 API | OpenAI（実質的に） |
| L6 ツール | MCP（Model Context Protocol） | Anthropic 発、modelcontextprotocol.io |
| L6 スキル | Agent Skills | Anthropic 発、agentskills.io |
| L9 | Temporal の SDK / Workflow definition | Temporal Inc.（Temporal（耐久実行エンジン）） |
| X | OTel GenAI semconv | CNCF（OpenTelemetry GenAI Agent Observability） |
| Y | AGENTS.md（コーディングエージェント向け統一指示書フォーマット） | Linux Foundation（Agentic AI Foundation） |

中立性が担保されることで、ベンダーが個別実装を変えても**契約面（プロトコル）は安定**する。これは OpenTelemetry GenAI Agent Observability が「ベンダーやフレームワークごとに異なるテレメトリ形状を採用するとロックインが発生する」と論じた構図と同じパターンを各層で適用する戦略となる。

### 5.2 層は薄く保つ

Claude の知性を活かすエージェント設計の3パターン の **「ハーネスを継続的に剪定する」** はモデル進化への対応戦略。L8 で AutoGen が maintenance mode に入った（マルチエージェントフレームワーク比較（AutoGen / LangGraph / CrewAI））ように、フレームワーク自体も陳腐化する。**層内の実装差し替えコストを最小化**するため、契約面（標準）を信じて実装は薄くする。

### 5.3 仕様駆動への移行

OpenAI Symphony の `SPEC.md` canonical / 実装は参考、`WORKFLOW.md` がリポジトリ所有のワークフロー契約、というパターンは新しい標準。**仕様があればコーディングエージェント自身がハーネスを書ける**。これは「実装ではなく仕様を中心に据える」開発スタイルへの移行を示唆する。
