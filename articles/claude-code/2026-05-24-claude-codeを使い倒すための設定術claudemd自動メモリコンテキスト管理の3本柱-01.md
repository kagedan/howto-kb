## はじめに

「毎回同じことを説明している」。Claude Code を使い始めてしばらくたった頃、私はこの感覚に気づいた。セッションを開くたびにプロジェクトの構成を教え、コーディングスタイルを伝え、使ってはいけないライブラリを注意する。AIが有能であることはわかっていても、文脈がリセットされるたびに準備コストが発生していた。

この問題には構造的な解法がある。Claude Code が提供する3つの仕組み、**CLAUDE.md**・**自動メモリ**・**コンテキスト管理**、を正しく組み合わせることで、「毎回同じ説明をしなくて済む」状態を作れる。

本記事では、それぞれの仕組みが何をするものか、どう設定すべきか、そして3つをどう連携させるかを整理する。

## CLAUDE.md ── AIに「このプロジェクトの常識」を教える

### 何者か

CLAUDE.md はリポジトリのルートに置く Markdown ファイルで、**セッション開始のたびに自動で読み込まれる**。Claude Code はこのファイルを「プロジェクトの常識集」として扱い、以降の会話に反映させる。

### 書くべきこと

| カテゴリ | 具体例 |
| --- | --- |
| **プロジェクト構成** | ディレクトリの役割、エントリポイント、ビルド成果物の場所 |
| **コマンド** | ビルド・テスト・lint の実行方法 |
| **命名規則・制約** | スラッグのフォーマット、使用禁止のライブラリ |
| **暗黙のルール** | なぜそのフレームワークを選んだか、技術選定の背景 |
| **公開フロー** | デプロイ手順、環境変数の扱い |

CLAUDE.md の核心は「**コードを読んでもわからないことを書く**」という点にある。ディレクトリ構造はファイルを見ればわかる。しかし「なぜ Redux ではなく Zustand を使っているか」「この API エンドポイントはパートナー企業との契約で変更禁止になっている」といった文脈は、コードのどこにも書いていない。

### 書くべきでないこと

* **コードパターンや関数名** ── `git blame` や `grep` で引ける情報に CLAUDE.md を使うのは冗長だ
* **頻繁に変わる情報** ── チケット番号、担当者名、進行中のスプリント内容などは陳腐化が早い
* **長い説明文** ── Claude は読んでいるが、トークンは有限だ（後述）

### チーム共有のベストプラクティス

CLAUDE.md の最大のメリットは、**リポジトリに commit してチーム全員が同じ文脈を共有できる**ことだ。新しいメンバーがリポジトリを clone した瞬間から、Claude Code は正しいプロジェクト文脈の中で動く。

関心ごとが増えてきたら、`.claude/rules/` サブディレクトリに分割するのも有効だ。このリポジトリでは `articles.md` という執筆ガイドラインを別ファイルにしているが、これにより CLAUDE.md 本体をコンパクトに保ちつつ、領域固有のルールを管理できる。

チームで運用するなら、PR のレビューチェックリストに「この変更は CLAUDE.md に追記すべきか？」という項目を加えることを勧める。設計決定の背景がコードに埋もれる前に、明示的に記録する習慣が定着する。

```
# .claude/rules/ を使ってルールを分割する例
.claude/
  rules/
    articles.md      # 記事の執筆ガイドライン
    api.md           # API設計規約
    testing.md       # テスト戦略
```

## 自動メモリ ── Claudeが作業から学習する仕組み

### 何者か

自動メモリは、作業を通じて Claude が学んだこと、コマンドの使い方、好みのスタイル、ユーザーへのフィードバック、を**セッション間を超えて保持する仕組み**だ。

保存先は `~/.claude/projects/<プロジェクトパス>/memory/` 以下のファイル群で、Markdown で書かれている。Claude は会話の中で「これは次回以降も覚えておくべき情報だ」と判断したとき、このディレクトリに書き込む。

### 何が保存されるか

自動メモリには4つの種類がある。

| 種類 | 内容 | 保存タイミング |
| --- | --- | --- |
| **user** | ユーザーの役割・知識・好み | 会話からプロファイルを推定したとき |
| **feedback** | 指摘・修正・確認された判断 | 「それは違う」「まさにそれ」などの反応 |
| **project** | 進行中の作業・目標・期限 | プロジェクト固有の文脈を把握したとき |
| **reference** | 外部システムの場所 | Linear プロジェクト、Grafana ダッシュボードなど |

CLAUDE.md との違いは明確だ。CLAUDE.md はチーム全員に共通の「プロジェクトの常識」であり、自動メモリは**特定のユーザーとの作業を通じて積み上がる個人的な文脈**だ。

### 監査と編集の方法

自動メモリは Claude が書き込むが、**最終的な管理権はユーザーにある**。ファイルは通常の Markdown なので直接編集できる。

```
# メモリファイルの場所を確認する
ls ~/.claude/projects/<プロジェクトパス>/memory/
```

特に `feedback` メモリは、過去の「それは違う」を蓄積するため、内容が古くなったり間違いが固定化されたりする可能性がある。定期的に内容を確認し、不要なものは削除する習慣を持ちたい。

また、間違った学習をリセットしたい場合は、該当ファイルを削除するか、「〜を忘れてください」と明示的に伝えることで対応できる。

### 保存すべきでないこと

コードパターンや関数名、git 履歴から引ける情報は、メモリには書かない。コードベースは常に変化するが、メモリは変化を追いかけない。「〇〇関数が src/utils にある」という記録は、リファクタリングで場所が変わった瞬間に誤情報に変わる。

## コンテキストウィンドウの節約術

### トークン制限が何を引き起こすか

Claude Code のセッションには、1度に扱える情報量（コンテキストウィンドウ）に上限がある。この上限に近づくと、Claude は会話の古い部分を「忘れ」始める。症状としては、序盤の指示が無視される、矛盾した回答が増える、といった形で現れる。

長時間のコーディングセッションや、多数のファイルを開いたセッションでこの問題は顕在化しやすい。

### `/compact` ── 圧縮して継続する

`/compact` は現在の会話を要約してコンテキストを圧縮するコマンドだ。**作業を中断せずに続けたい**とき、ウィンドウの使用量が増えてきたと感じたときに使う。

要約は Claude が自動生成するが、重要な情報が失われることもある。使うタイミングの目安は、大きなタスクの区切り（実装が一段落したとき、コードレビューが終わったときなど）だ。作業の中間ではなく、キリのいい場所で使うほうが品質劣化を抑えられる。

### `/clear` ── 完全リセットが適切な場面

`/clear` は会話を完全にリセットする。前のセッションの文脈を一切引き継ぎたくないとき、まったく別のタスクに切り替えるときに使う。

```
# コンテキスト管理の使い分け
/compact  ← 作業を継続しながら圧縮したいとき
/clear    ← 全く新しいタスクに切り替えるとき
```

### CLAUDE.md を簡潔に保つことがコンテキスト節約にも直結する

ここで CLAUDE.md の話に戻る。CLAUDE.md は**毎回全文が読み込まれる**。長大な CLAUDE.md はそれだけでコンテキストを消費する。

「書くべきでないことを書かない」という原則は、品質の問題であると同時に、コスト効率の問題でもある。過去の設計ノート、更新されていない担当者リスト、長い説明文が積み重なった CLAUDE.md は、毎セッションで無用なトークンを消費し続ける。

また、`@ファイル指定` で必要なときだけファイルを読み込む使い方も有効だ。全ファイルを最初から渡すのではなく、「今このタスクに必要なファイル」だけを Claude に渡す意識がコンテキストを長持ちさせる。

### 3本柱の連携

3つの仕組みの役割を整理すると、こうなる。

| 仕組み | 役割 | スコープ |
| --- | --- | --- |
| **CLAUDE.md** | プロジェクトの常識を設定する | チーム共有・永続 |
| **自動メモリ** | 個人の作業文脈を蓄積する | ユーザー固有・永続 |
| **コンテキスト管理** | 1セッションの品質を維持する | セッション内・一時 |

CLAUDE.md でベースの文脈を確立し、自動メモリで個人の好みと蓄積した知識を補い、コンテキスト管理でセッションを健全な状態に保つ。この3層で「毎回同じ説明をしなくて済む」状態が作られる。

## まとめ

Claude Code を効率よく使うための3本柱を整理した。

1. **CLAUDE.md は「コードに書けない文脈」を書く** ── チームの暗黙ルール、技術選定の背景、禁止事項を明文化してリポジトリに commit する。長くするほどコンテキストを消費するので、書くべきでない情報は積極的に削る
2. **自動メモリは定期的に監査する** ── `~/.claude/projects/.../memory/` を確認し、古くなった情報や誤った学習を修正する。コードパターンや関数名は保存しない
3. **コンテキストは設計する** ── `/compact` と `/clear` を使い分け、大きなタスクの区切りでコンテキストを圧縮する。`@ファイル指定` で必要なファイルだけを渡す習慣を持つ

最初の30分でこの設定を整えておくだけで、その後の全セッションの品質が変わる。AIに毎回同じ説明をするコストは、設定という形で一度だけ払えばよい。
