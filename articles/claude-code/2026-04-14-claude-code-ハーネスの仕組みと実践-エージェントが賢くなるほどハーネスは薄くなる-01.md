## Claude Code ハーネスの仕組みと実践

この記事でわかること

* エージェントハーネスというものが、Claude 本体と何を分担しているのか
* Claude に任せるために、こちらが削っていくべきもの
* 小さなお試しプロジェクトで試せる、三つの実践パターン

## はじめに

ハーネスという言葉を初めて意識したのは、自分で書いた [CLAUDE.md](http://CLAUDE.md) が 200 行を超え、しかも守ってもらえていないと気づいた日のことでした。規約を足せば足すほど Claude の動きがぎこちなくなり、削ったほうがいいのは直感で分かるのに、何を削っていいかは分からない。そんな戸惑いのなかで「ハーネス」という視点に出会いました。

ハーネスとは、エージェントの周囲にわたしたちが置いている一切の構造のことです。[CLAUDE.md](http://CLAUDE.md)、Skill、Hook、ツール定義、権限設定、プロンプトの並び。Anthropic が最近公開した記事が、この設計の勘所を三つのパターンに整理しています。本記事では、そのパターンを小さなお試しプロジェクトで手を動かしながら辿ります。URL は末尾に置きます。

## 第1章　ハーネスとは何か

最初にこの章を置くのは、言葉の共通理解がないと、以降のハンズオンが単なる手順書に化けてしまうからです。

エージェントの動きは、次のようなループで考えると整理がつきます。

```
       [ユーザーの依頼]
              │
              ▼
        ┌──────────┐   推論・判断
        │  Claude  │ ─────────────┐
        └──────────┘              │
              ▲                   ▼
              │             ツール呼び出し
           観測結果               │
              │                   ▼
              └────── [ファイル / Bash / Web / …]
```

この図のなかで、Claude 本体が担うのは推論と判断だけです。残りのすべて、つまりどんなツールを渡すか、観測結果をどう要約して戻すか、どこで止めるか、何を覚えさせるか、それらがハーネスの領分です。

ハーネスが担うべきもの担うべきでないものの線引きが、この記事を通じて問い直される基準になります。キャッシュ戦略、ツール定義、UX の境界、危険な操作のゲートは担うべきものです。一方、個別の判断やコードの組み立ては Claude が本来できることなので、わたしたちが先回りして手順書を足すほど、Claude の自由度が削られていきます。ハーネスは育てるものではなく、むしろ Claude が賢くなるたびに削っていくものだ、という感覚をここで持ってもらえたら十分です。

## 第2章　ハンズオン準備

手を動かす準備をしておきましょう。この章を置くのは、以降の実演をそのまま再現できるようにするためです。

Claude Code がインストール済みであることを前提にします。まだの方は `claude --version` で確認してください。お試し用のディレクトリを切ります。

```
mkdir harness-lab && cd harness-lab
claude
```

`harness-lab` に `CLAUDE.md` を置くと、起動時に自動で読み込まれます。このファイルはプロジェクト固有の記憶で、新しく来たメンバーに最初に渡す環境設定ガイドのようなものだと思ってください。以降の章では、この一枚を書き換えたり、脇に Skill を足したりしながら、三つのパターンを試していきます。作るのは小さなファイル整理エージェントです。大げさな実装はしません。

## 第3章　パターン1　Claude がすでに知っていることを使う

この章が重要なのは、ここを誤ると、使わなくていい車輪を自分で再発明してしまうからです。

Claude にはすでに熟知しているツールがあります。bash とテキストエディタです。モデルは大量のシェルスクリプト、Unix コマンド、ファイルの読み書き操作を学んできました。SWE-bench で 49% のスコアが報告されたときの構成も、派手な専用ツールの束ではなく、bash とファイル編集というシンプルな組み合わせでした。これは Claude にとっての母語です。

カスタムツールを新しく作るほど、Claude はそれを学び直す必要が出てきます。既知のものを使わせるほど、こちらが書く説明は減ります。パターン1の核心はそこにあります。

やってみましょう。`harness-lab` に次のファイルとディレクトリを置きます。

```
mkdir -p inbox sorted
cat > inbox/note01.txt <<'EOF'
明日の定例会議の議題メモ
EOF
cat > inbox/note02.txt <<'EOF'
買い物リスト: 牛乳、卵、パン
EOF
cat > inbox/note03.txt <<'EOF'
バグ再現手順: ログイン直後にリロードするとセッションが切れる
EOF
```

`CLAUDE.md` は次の一枚だけにします。

```
## タスク

./inbox/ 内の .txt ファイルを読み込み、内容のカテゴリを判断して
./sorted/{カテゴリ名}/ に移動してください。
カテゴリはあなたが決めてください。
```

Claude を起動し「[CLAUDE.md](http://CLAUDE.md) のタスクを進めて」と頼みます。すると Claude は `ls` で一覧を見て、`cat` で中身を読み、`mkdir -p sorted/meeting` のようにカテゴリ名のフォルダを作り、`mv` で振り分ける、という流れを自分で組み立てます。こちらは手順を一つも書いていません。カテゴリ分けツールも書いていません。それでも仕事は終わります。

気づきはここにあります。専用ツールを書きたくなった瞬間、一度立ち止まって「これ、bash の組み合わせで Claude に任せられないか」と問うてみる。その問いを挟むだけで、ハーネスに積み上がるコードが目に見えて減っていきます。

## 第4章　パターン2　何をやめられるか？

この章が重要なのは、コンテキストは有限の資源だからです。詰め込むことより、何を通さないかを設計するほうが、長い目で効いてきます。

### 4-1　自身のアクションを調整させる

Claude が大量のデータを扱うとき、そのすべてをコンテキストに戻す必要はありません。bash のパイプを使えば、中間結果を本体に見せずに次のコマンドへ流せます。ハーネス側ですべきことは、パイプという概念を邪魔しないことです。

試してみます。大きめのログを用意しましょう。

```
mkdir -p logs
for i in $(seq 1 10000); do
  echo "2026-04-14 10:00:$((i % 60)) level=info msg=ok id=$i"
done > logs/app.log
echo "2026-04-14 10:05:00 level=error msg=boom id=42" >> logs/app.log
```

Claude に頼みます。「logs/app.log から level=error の行だけ抽出して、件数と一緒にまとめて」。Claude はおそらく `grep level=error logs/app.log | wc -l` のような組み立てで、必要な行と件数だけを取り出します。1 万行の本文はコンテキストに入ってきません。

気づきはここです。ハーネスが何も足さなくても、Claude はパイプで自分の視野を守れます。こちらが「途中経過を見せて」と余計な指示を足した瞬間、この防波堤は崩れます。黙っている勇気もハーネスの一部です。

### 4-2　コンテキストを自分で管理させる

次は、コンテキストを段階的に開示するという話です。Skill という仕組みは、見出しだけを常に見せておき、本文は呼ばれたときにだけ読ませるという控えめな設計になっています。

`.claude/skills/file-organizer/SKILL.md` を作ります。

```
---
name: file-organizer
description: ./inbox/ のテキストをカテゴリ別に ./sorted/ へ振り分ける手順。ファイル整理の依頼があったときだけ読むこと。
---

## 手順

1. ls ./inbox/ で対象を確認する
2. 各ファイルの先頭 30 行を cat で読み、カテゴリを決める
3. mkdir -p ./sorted/{カテゴリ} を作ってから mv で移動する
4. 最後に ls ./sorted/ で結果を報告する
```

起動時、Claude が読むのは description の一文だけです。ファイル整理の依頼が来てはじめて、本文が読み込まれます。全部を先に読ませないこと、これがコンテキストの注意経済を守るコツです。

気づきはここです。長いルールを書きたくなったら、それは常時見える [CLAUDE.md](http://CLAUDE.md) ではなく、Skill に置く候補かもしれない。そう自問するだけで、[CLAUDE.md](http://CLAUDE.md) が勝手に痩せていきます。

### 4-3　コンテキストを自分で永続化させる

最後は記憶の外出しです。長いセッションで得た知見を、毎回伝え直すのはもったいない。`memory/` というフォルダを Claude に使わせると、自分で「何を覚えておくべきか」を判断して書き出すようになります。

[CLAUDE.md](http://CLAUDE.md) に一行だけ足してみます。

```
## メモリ

重要な判断や、次回のセッションでも参照したい事実は
./memory/ に Markdown で書き出してください。ファイル名は自由です。
```

この一行だけで、Claude はタスクの節目に `memory/decisions.md` のようなファイルを作り始めます。こちらが指示したのは置き場所だけで、何を書くかは言っていません。

気づきはそこです。ハーネスで決めるのは置き場所と方針、内容の選別は Claude に委ねる。これがパターン2の設計感覚です。

## 第5章　パターン3　境界を慎重に設定する

この章が重要なのは、賢いエージェントほど暴れたときの被害が大きく、境界の設計だけは人間が責任を持つべき領域だからです。

### 5-1　キャッシュ設計

ハーネスの書き方はキャッシュ効率にも響きます。プロンプトキャッシュは先頭から静的な順に積むほど効くので、[CLAUDE.md](http://CLAUDE.md) のように常時読まれる内容は先頭、セッションごとに変わる内容は後ろ、という並びが基本になります。モデルを頻繁に切り替えたり、ツール定義をセッション中に書き換えたりすると、それだけでキャッシュが壊れます。

気づきは一つだけ挙げておきます。[CLAUDE.md](http://CLAUDE.md) に日付や可変な情報を混ぜない。それだけでキャッシュの寿命が目に見えて伸びます。

### 5-2　専用ツール vs bash ツール

パターン1で bash に任せる話をしましたが、何でも bash に寄せていいわけではありません。判断基準は三つです。取り消せるか。監査が必要か。ユーザーへの提示が必要か。このどれかに当たるなら、bash の一撃ではなく確認ステップを挟むべきです。

比較してみましょう。

```
## パターン A（CLAUDE.md に書く）

./output/ 内のファイルを全削除してください。
```

```
## パターン B（CLAUDE.md に書く）

./output/ 内のファイルを削除する前に、
削除対象のファイル一覧を表示し、続行してよいか確認を求めてください。
```

少し読者に考えてもらいたい部分です。パターン A と B、どちらが「よい」ハーネスでしょうか。

答えを書きます。パターン A は速いですが、意図しないファイルが紛れていたときに取り返しがつきません。パターン B は一拍遅れますが、削除は不可逆な操作なので、この一拍が保険になります。ハーネスで境界を引くとは、こうした小さな確認ステップをどこに挟むかを決める作業です。

気づきはここにあります。自動化の価値は速さではなく、疲れていても正しい判断に戻れる足場を持てることです。不可逆な操作の前で一瞬止まる癖をハーネスに組み込むのは、速度を捨てているのではなく、寝不足の自分を守っているということです。

## 第6章　ハーネスは育てるものではなく、削るもの

Anthropic の記事のなかに、印象に残った話があります。以前のハーネスには、コンテキストが荒れてきたときに一度リセットしてやり直すための仕組みが組み込まれていた。それが Opus 4.5 の代になると、Claude 自身が自分のコンテキストを律することができるようになり、そのリセット機構が丸ごと不要になった、というものです。

ここに設計思想の核があります。エージェントが賢くなるたびに、ハーネスには削れる箇所が増えていきます。去年必要だった補助輪が、今年は邪魔になっている。そういう逆転が、静かに、しかし確実に起きています。

わたしたちがやるべきなのは、ハーネスに機能を足していく競争ではなく、今日の Claude に任せられる範囲を見直し、先週書いた規約を一行でも削れないかと問い続けることです。この記事を閉じる前に、一つだけ自問してみてください。あなたのエージェントハーネスに、今日削れるものはありますか。
