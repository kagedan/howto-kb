# Claude Code / Codex では、repoそのものが“記憶”になるのではないか

## はじめに：第1部で考えたこと 🧠

前回の記事では、ChatGPTやClaudeを長く使うなら、  
「普段使いAI」と「検証用AI」を分けた方がいいかもしれない、という話を書きました。

普段使いのAIは、自分の文体、考え方、過去の相談、Project設定、メモリなどを踏まえて、かなり使いやすい相棒になっていきます。

これは便利です。  
かなり便利です。

一方で、研究、調査、記事化、事業構想のように、  
「いつもの見方から少し外れること」も必要な場面では、  
AIが自分に寄り添いすぎることが、探索範囲を狭める可能性もあります。

そこで前回は、ざっくり次のような運用を考えました。

* 普段使いAI
* 検証用AI
* 白紙確認用AI
* 反証用AI
* 外部ログ

今回は、この考え方をもう少し広げます。

テーマは、Claude Code / Codex のようなAIコーディングエージェントです。

この記事で考えたいのは、次の問いです。

**AIコーディングエージェントでは、repoそのものが“記憶”のように働くのではないか。**

ただし、ここでいう「記憶」は、モデルが学習して覚えるという意味ではありません。

あくまで、AIが作業時に参照する文脈、環境、設定、既存コード、ログの話です。

この記事は、Claude CodeやCodexを否定するものではありません。  
むしろ、長く使うなら、repoや指示ファイルやログをどう扱うかを設計した方がよいのではないか、という問題提起です。

---

## 通常チャットAIの文脈はどこにあるのか 💬

まず、通常のチャットAIについて考えます。

ChatGPTやClaudeを普通に使う場合、AIが参照する文脈は主に次のようなものです。

* チャット履歴
* メモリ
* Project設定
* カスタム指示
* 添付ファイル
* その場で入力したプロンプト

もちろん、具体的に何が参照されるかは、サービスや設定によって変わります。  
そのため、公開前には公式仕様を再確認する必要があります。

ただ、大きく見ると、通常チャットAIの文脈は「会話」と「設定」に寄っています。

つまり、AIがこちらに合わせてくれる理由は、主に会話履歴や指示、メモリ、Projectの設計にあります。

この場合の注意点は、前回の記事で扱ったように、  
AIが自分にとって使いやすくなるほど、いつもの考え方に戻りやすくなる可能性がある、という点です。

---

## コードエージェントでは、repoそのものが文脈になる 🧩

では、Claude Code / Codex のようなAIコーディングエージェントではどうでしょうか。

ここでは、通常チャットAIよりも、参照される文脈が広がります。

たとえば、次のようなものです。

* README
* CLAUDE.md / AGENTS.md
* docs
* tests
* 既存コード
* branch
* permission
* 実行ログ
* 過去のAI生成コード
* lint結果
* CI結果
* error log
* commit履歴
* issueやPRの文脈

もちろん、ツールや設定によって参照範囲は違います。  
また、常にすべてを読んでいるという意味でもありません。

ただ、通常チャットAIと比べると、コードエージェントは「会話」だけではなく、repoや開発環境そのものを手がかりにして作業します。

![](https://static.zenn.studio/user-upload/a66198552224-20260506.png)  
*通常チャットAIとコードエージェントの文脈比較図*

ここで大事なのは、repoが単なるファイル置き場ではなくなる可能性です。

AIにとっては、READMEも、docsも、testsも、過去の実装も、コメントも、仮置きコードも、全部が「このプロジェクトではこういう前提で進めるらしい」という材料になり得ます。

つまり、コードエージェントにとってrepoは、  
**作業時に参照される文脈の集合体**になります。

この意味で、repoそのものが“記憶”のように働く可能性があります。

---

## ここでいう「記憶」は、モデル学習の意味ではない ⚠️

ここは誤解されやすいので、はっきり分けておきます。

この記事でいう「repoが記憶になる」は、  
モデルがrepoを学習して、次のモデルに覚え込む、という意味ではありません。

そうではなく、次のような意味です。

* AIが作業時にrepoを参照する
* repo内のREADMEやdocsを判断材料にする
* testsや既存コードを前提に実装方針を立てる
* 過去の実装やログに引っ張られる可能性がある
* 指示ファイルが作業方針に影響する可能性がある

つまり、これは**学習としての記憶**ではなく、  
**作業文脈としての記憶っぽさ**の話です。

この区別をしないと、話が大きくなりすぎます。

この記事では、AIの学習データやモデル更新の話ではなく、  
日々の開発作業でAIが何を参照し、どの前提で実装判断するのか、という運用上の話として扱います。

---

## 便利さとリスクは同じ場所にある 🔧

repoを文脈として使えることは、AIコーディングエージェントの大きな強みです。

既存コードを読んでくれる。  
testsを見てくれる。  
READMEを踏まえてくれる。  
設計方針に沿った実装を提案してくれる。  
エラーを見て修正案を出してくれる。

これは、通常チャットAIだけでは難しかった部分です。

ただ、便利さとリスクは同じ場所にあるかもしれません。

たとえば、repoの中に次のようなものがある場合です。

* 古い設計方針
* 一時的な仮置きコード
* もう使っていないdocs
* 現在の実装とズレたREADME
* とりあえず通したtests
* 過去のAI生成コード
* 削除予定のbranch
* 例外的に許可したpermission
* 本番では使わないdebug log

人間なら、「これは古い」「これは仮」「これは後で消す」と判断できることもあります。

しかし、AIにとっては、それが明示されていなければ、  
repo内の情報を現在の前提として扱う可能性があります。

もちろん、AIが必ず間違えるという話ではありません。  
むしろ、文脈が多いほど正確になる場面もあります。

ただ、長く使うなら、repo内の文脈がAIの実装判断にどう影響するかは、運用として見た方がよいのではないでしょうか。

これはまだ仮説です。

## AI運用の4レーンを、コードエージェントにも広げる 🚦

前回の記事では、AIを長く使うなら、普段使いと検証用を分ける考え方を出しました。

コードエージェントでも、同じような運用が必要かもしれません。

![](https://static.zenn.studio/user-upload/5c4fe184c9aa-20260506.png)  
*AI運用の4レーン構造図*

ここでは、ざっくり4つのレーンで考えてみます。

---

### 1. 普段開発レーン 🛠️

これは、日常的に使うAIコーディング環境です。

* いつものrepo
* いつものbranch
* いつものCLAUDE.md / AGENTS.md
* いつものdocs
* いつものtests
* いつもの権限設定

普段の開発では、このレーンが一番便利です。

プロジェクト固有の事情を踏まえてくれるため、  
細かい説明を毎回しなくても、ある程度スムーズに作業できます。

ただし、このレーンは「便利な文脈」が多い分、  
古い前提や過去の設計にも引っ張られやすい可能性があります。

---

### 2. 白紙レビュー・レーン 🧼

これは、できるだけ余計な文脈を減らして見るレーンです。

目的は、普段のrepo文脈に引っ張られず、  
「そもそもこの設計でよいのか」を見ることです。

たとえば、次のような使い方が考えられます。

* 対象ファイルだけを渡す
* READMEやdocsをあえて渡さない
* 最小限の仕様だけでレビューさせる
* 別AIに同じ設計を見せる
* 新規チャットで前提をリセットする

もちろん、白紙レビューが常に正しいわけではありません。

文脈が少ないため、現場制約を無視した提案が出る可能性もあります。

それでも、普段のAIが見落としている前提を見つけるには、役立つ場面があるかもしれません。

---

### 3. 逆張りレビュー・レーン 🔍

これは、あえて反対側から見てもらうレーンです。

目的は、実装案の粗探しではなく、  
「このrepoの前提が間違っているとしたら、どこか」を見ることです。

たとえば、次のように聞きます。

* この設計で将来詰まりそうな点は？
* testsが通っても危ない点は？
* READMEと実装がズレている可能性は？
* AI生成コードっぽい弱さはどこにある？
* このbranchの変更を本番に入れる前に疑うべき点は？

普段開発レーンでは、AIはプロジェクトに協力的に動いてくれます。

それは強みです。

ただ、協力的であるほど、前提そのものを疑う力が弱くなる可能性もあります。

だからこそ、逆張りレビュー・レーンを別に置く意味があります。

---

### 4. 外部ログ・レーン 📜

最後は、外部ログ・レーンです。

これは、AIの回答そのものではなく、  
人間があとで検証できるように、判断の材料を残すレーンです。

たとえば、次のようなものです。

* 重要なAI提案
* 採用した理由
* 採用しなかった理由
* 実行したコマンド
* 失敗したログ
* tests結果
* CI結果
* 人間が最終判断した箇所

AIコーディングでは、気づくと「AIが提案して、人間がなんとなくOKした」状態になりがちです。

しかし、あとで不具合が出たときに、  
なぜその実装になったのかが追えないと、責任や改善が曖昧になります。

外部ログ・レーンは、AIを疑うためだけではありません。

AIと人間の共同作業を、あとで振り返れる形にするためのレーンです。

---

## 実務で使うなら、まず何を分けるか 🧭

いきなり完璧な運用を作る必要はないと思います。

まずは、次の3つだけでも分けるとよさそうです。

### 1. 開発用とレビュー用を分ける

普段の実装を助けるAIと、  
設計や前提を疑うAIを同じ役割にしない。

同じAIを使う場合でも、チャットやbranchや入力情報を分けるだけで、多少は役割を変えられるかもしれません。

### 2. 指示ファイルに「事実」と「方針」と「仮説」を混ぜすぎない

CLAUDE.mdやAGENTS.mdに何でも入れると、便利ではあります。

ただ、そこに古い知識、仮説、暫定ルール、好み、禁止事項が混ざりすぎると、AIが何をどの強さで扱うべきか分かりにくくなる可能性があります。

### 3. 外部ログを残す

AIが何を提案したか。  
人間がなぜ採用したか。  
どのtestsを見たか。  
どのログを根拠にしたか。

これを残しておくと、あとで見直しやすくなります。

AIコーディングが進むほど、実装結果だけでなく、  
判断過程のログが重要になるかもしれません。

---

## まとめ：repoを育てるなら、AIの文脈も設計したい 🌱

Claude Code / Codex のようなAIコーディングエージェントは、かなり強力です。

通常チャットAIと違って、repo、docs、tests、既存コード、ログ、branch、permissionなど、開発環境そのものを手がかりにできます。

これは大きな強みです。

一方で、repoに古い前提や仮置きコードや過去のAI生成コードが残っていると、  
それもまたAIの判断材料になる可能性があります。

だから、AIコーディングエージェントを長く使うなら、  
「AIに何を読ませるか」だけではなく、  
「どのレーンで読ませるか」を考える必要があるのかもしれません。

この記事の仮説は、次の一文にまとめられます。

**AIコーディング時代には、repoそのものを“AIが参照する文脈”として運用設計する必要があるのではないか。**

これはまだ仮説です。

文脈が多い方がよい場面もあります。  
白紙レビューが役に立たない場面もあります。  
クリーン環境より、現場の制約を知っているAIの方が正しい場面もあります。

それでも、AIコーディングが日常化するほど、  
repo、指示ファイル、logs、tests、branch、permissionを含めた「AIの作業文脈」をどう扱うかは、整理する価値があると思います。
