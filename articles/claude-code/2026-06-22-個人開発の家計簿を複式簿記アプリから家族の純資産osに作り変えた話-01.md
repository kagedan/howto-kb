## TL;DR

* 「複式簿記の家計簿」を個人開発していたが、**それ自体は差別化にならない**と評価でハッキリ言われた。
* 競合（Firefly III / Actual Budget / ezBookkeeping / dLedger / MoneyBalance / Dualy）が既にいる。
* 作り直す代わりに、**すでに実装済みだった「純資産」「ゼロ知識E2E」「一行入力」を“武器”として束ね直し**、「**家族の純資産OS × 運営者にも見えないプライバシー × 摩擦ゼロ入力**」へ再ポジショニングした。
* 実装はほぼそのまま。変えたのは **言葉・見た目・SEO・機能の重点**。Claude Design → Claude Code でアプリとLPを全面刷新した。

## 作っていたもの

[kurofukubo（黒福簿）](https://kurofukubo.com/) という、複式簿記ベースの個人向け家計簿SaaS。React + AWS（Lambda/DynamoDB/Cognito/CloudFront）で、貸借対照表（BS）・損益計算書（PL）・キャッシュフロー計算書まで出せる——という、わりと“ガチ”なものです。

自分では「個人向けでBS/PLが出せるのは珍しい」と思っていました。

## 評価で突きつけられた現実

第三者にコンセプトを評価してもらったところ、結論はシンプルでした。

> **「複式簿記家計簿を作る」こと自体が、もう差別化にならない。**

2026年時点で、複式簿記・自己ホスト・AI仕分け・PWA……といった要素は、OSSや新興アプリが既に満たしています。

| セグメント | プレイヤー |
| --- | --- |
| 複式簿記ガチ勢 | Beancount, Ledger, Firefly III |
| 予算特化 | Actual Budget, YNAB系 |
| 新興・複式家計簿 | MoneyBalance, Dualy |
| OSS個人開発 | ezBookkeeping, dLedger |

そして個人開発者が一番ハマる罠も指摘されました。

> **「会計好きが作る会計ソフト」になること。**

総勘定元帳・補助科目・多通貨……を実装して自分は満足するが、ユーザーは増えない。ユーザーが欲しいのは「正しい管理」ではなく「**面倒でない・続く・資産が見える**」だと。

刺さりました。

## 気づき：武器はもう持っていた

落ち込みつつ自分の実装を棚卸しすると、実は**差別化の素材はすでにあった**ことに気づきます。

1. **純資産（家族のBS）** … 現金・証券・NISA・iDeCo・クレカ負債・住宅ローンを1画面に束ねられる。
2. **ゼロ知識（E2E）暗号化** … 運営者にも中身が読めない設計。これは前に実装済み（[別記事](https://zenn.dev/pakupaku_x/articles/801c47459db3d5)参照）。競合の MoneyBalance/Dualy/ezBookkeeping は持っていない。
3. **摩擦ゼロ入力** … 「食費 1200 現金」の一行入力 → 裏で複式仕訳を自動生成 → BS化。

つまり、**バラバラに実装していた機能を“1つのコンセプト”に束ね直せば**、模倣されにくいポジションになる。

## 再ポジショニング

捨てた看板：**「複式簿記の家計簿」**  
掲げた看板：**「家族の純資産OS」**

```
家族の“純資産”が見える家計簿。
NISA・iDeCo・ローンまで一つに。
しかも運営者にも中身は見えない。
```

複式簿記は“裏方”に隠し、表では一切使わない。差別化の核は

> **純資産統合 × ゼロ知識プライバシー × 摩擦ゼロ入力**

の掛け算に置きました。

## やったこと（実装はほぼ据え置き）

重要なのは、**コードの大半は変えていない**こと。変えたのは見せ方です。

### 1. 全面リデザイン（Claude Design → Claude Code）

Claude Design でアプリ／LP のデザイン案を作り、その出力（コンポーネント構造・配色・レイアウト）を Claude Code で既存の React アプリへ反映。

* 配色：ゴールド系 → **ティール（データフォワード×ソフトカード）**
* ダッシュボードに**純資産ヒーロー**を新設（純資産・収入・支出・収支を一目で）
* フォント／角丸カード／ピル型ナビ／ライト・ダーク両対応
* 「**表示/非表示マスク**」（人前でも開ける `¥****`）を追加
* 巨大な金額やモバイルでの**レイアウト崩れを地道に潰す**（`minmax(0,1fr)` 大事）

### 2. SEOの付け替え

ここが地味に効く。狙うキーワードを\*\*“複式簿記”から“純資産・資産管理・プライバシー”へ全面的に振り替え\*\*ました。

* 捨て：`複式簿記 家計簿`（競合飽和・検索少）
* 主軸：`純資産 管理 アプリ` / `資産 見える化` / `家族 資産 管理`
* 投資系：`NISA iDeCo まとめて 管理`
* プライバシー：`家計簿 プライバシー` / `連携なし 家計簿`

そのうえで、ロングテール記事（「家族のバランスシートの作り方」「NISA/iDeCoを含めた純資産の出し方」「家計簿のデータは誰が見られる？」）を用意し、LP・メタ・OGP・構造化データ（FAQPage）も純資産＋プライバシー基調に。

## 学び

* **個人開発で難しいのは「作れるか」ではなく「どこで戦うか」**。技術的に作れることと、選ばれることは別。
* **差別化は“新機能”より“既存機能の束ね方”で生まれることがある**。自分の手札を、競合のいない交点に並べ直す。
* 「連携しない（銀行APIを使わない）」という**弱点すら、“認証情報を預けない安全性”という強みに転換**できる。
