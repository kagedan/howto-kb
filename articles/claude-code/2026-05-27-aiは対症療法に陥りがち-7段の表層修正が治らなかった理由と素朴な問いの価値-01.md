## はじめに

少し前に [LLM agent に誤前提が 17 連鎖した話 — typo 1 文字が生む指摘の連鎖と、その断ち方](https://zenn.dev/fixu/articles/ai-design-premise-verify-chain) という記事を書きました。あれは「**上流の誤字が下流まで通り抜けて、設計時の誤前提が連鎖した話**」でした。

今回はその姉妹編です。同じ AI エージェントと同じ運用で、別機能の **実装中の動作確認** を進めていたところ、今度は **症状を順次つぶす局所的な修正が 7 段にわたって積み上がり、それでも根本問題が解消しなかった** という事象に遭遇しました。前作が「設計時の誤前提連鎖」だとすれば、今回は「**実装時の対症療法連鎖**」です。

先に手触りを書いてしまうと、

> **AI は目の前のエラーに対する局所的な修正を積み重ねがちで、「そもそもこのチェック自体が必要か?」という一段抽象化した問いは、ユーザーからの素朴な問いかけなしには出てきにくい。**

7 段の修正をフェーズ A〜G として時系列で並べたあと、なぜ AI は症状修正に陥りやすいのか、構造的な 5 つの理由と効く対策を書き残します。AI に耳の痛い話ですが、自分自身もユーザーとして 7 段の修正を黙認した側なので、自虐込みの振り返りでもあります。

---

## 1. 何が起きたか — フェーズ A〜G、7 段の対症療法

舞台は店舗管理機能 (= 親店舗が複数の子店舗を束ねる多拠点向けの機能) の開発環境での動作確認でした。「親店舗から子店舗を選択する → 子店舗のサービスに遷移する」導線を作っていたところ、子店舗側のトップ画面に「**この経路ではアクセスできません**」というエラーが出る症状が見つかりました。

ここから AI と私の協働で修正を積み重ねたのですが、**7 段デプロイしても既存の行き詰まったユーザーの症状は解消しませんでした**。順を追って何を直したかを並べます。

**フェーズ A (設計違反の撤去):** 親店舗側のコントローラーが子店舗へのリダイレクトを握っていました。遷移はプラットフォーム側の URL スキームで完結すべきで、撤去しました。

**フェーズ B (URL 組み立てバグ):** 管理画面側で、親店舗のメニュー URL に子店舗用の識別子を埋めていたバグを修正しました。

**フェーズ C (潜在バグ):** 「許可パスリスト」に該当画面のパスが含まれておらず、内部転送が弾かれていました。追加しました。

**フェーズ D (潜在バグ):** 内部転送時に Cookie 情報を保持しておらず、転送先でセッションが空になる挙動を修正しました。

**フェーズ E (誤字バグ — これが象徴的):** 認証関連のコントローラーにこんなコードがありました。

```
// 「セッションのユーザーIDを消す」のつもりで書かれていたが…
Session::flash('USER_ID');
```

Laravel の `Session::flash($key)` は引数 1 つだと「`$key` に `true` を書き込み、次のリクエスト終了時に削除」という挙動です。**削除どころか書き込み** になっていた古典的な誤字バグでした。

**フェーズ F (メッセージ改善):** トークン期限切れ時の不親切なエラー文を「セッションが切れています」に改善しました。

**フェーズ G (応急処置):** 行き詰まったユーザーへの対応として、認証情報からセッションのユーザー ID を自動復元する処理を入れました。

---

A〜G のどれも「目の前のエラーログから辿れる症状」を直しているだけで、**「そもそも何故ここでセッション内のユーザー ID に依存しているのか?」という上位の問いは、7 段を通じて一度も立ちませんでした。**

## 2. 素朴な問いがアーキテクチャ再設計を引き出すまで

7 段が終わっても改善しなかった事実を見て、私はようやく一段抽象化した疑問を投げました。

> 「子店舗選択画面を開くときには、他の画面 (例: アカウント画面) と同様に、**ログインさえしていればページを開けることが期待値ではありませんか?** 子店舗を選択した *後* にセッション情報を書き込む、というのが全体フローなのでは? アーキテクチャを整理してください。」

整理するとこういうことです。

| 画面 | 期待される認可 | 実装上の実態 |
| --- | --- | --- |
| アカウント画面・各種設定 | 認証 (ログイン済みか) | 認証に依存 |
| **子店舗選択画面** | 認証で十分なはず | **セッション内のユーザー ID に依存** |

子店舗選択画面だけが、なぜか認証ではなく **セッション内のその場限りの状態** に依存して認可を行っていたのです。フェーズ A〜G は全て「このユーザー ID が消えないよう各経路で保持・復元する」応急処置でした。本来やるべきは「**他画面と同じく認証ベースで認可する。ユーザー ID は子店舗選択後のセッション書き込み用のパラメータに過ぎない**」というアーキテクチャ整理でした。

ここからフェーズ H として、認証ベースの根本リファクタリングに着手することになりました。

## 3. なぜ AI は症状修正に陥りやすいのか — 構造的な 5 つの理由

ここから本論です。「AI を慎重に使えば良いだけでは?」で済ませず、なぜ症状修正に陥りやすいかを構造的に書きます。心当たりがあれば各エージェント型ツール (Claude Code / Cursor / Cline / Aider 等) でも同じ罠は発動しているはずです。

### 理由 1: フィードバックループの即時性バイアス

目の前にエラーログがあると、「このログを消す修正」までの経路は最も短く、最も自然に書けます。  
「そもそもこのチェック自体が必要か?」「この値をセッションで持ち運ぶ設計が妥当か?」のような一段抽象化された問いは、認知的コストが高い上に、答えが出ても **即時の検証手段がありません** (= 大規模リファクタリングになるので 1 ターンで動作確認できない)。

AI の挙動はこの即時性バイアスに強く影響されます。「次の動作確認で動くかどうか」をゴールにすると、ゴールに最短で届く経路が選ばれる。それが局所的な修正の積み重ねです。

### 理由 2: 訓練データの偏り

OSS のプルリクエストは「バグ報告 → パッチ → マージ」のサイクルが大量に学習データとして存在します。プルリクエスト 1 件 1 件は症状単位の小さな修正であり、「アーキテクチャを整理しなおす」種類の変更はプルリクエストではなく **設計ドキュメント + 大規模リファクタリングのプルリクエスト (めったに無い)** という形でしかリポジトリに残りません。

つまり統計的に、AI が学習している「修正とはこういうもの」のモデルは、本質的に **症状単位の局所的なパッチ** に偏っています。アーキテクチャを整理する変更は訓練データ上では稀少種です。これは AI 開発元側でもデータの取捨選択で補正しようとしているはずですが、自然発生する修正の絶対量に押されます。

### 理由 3: 「当事者性 (skin in the game)」がない

人間のエンジニアは半年後の自分のために綺麗に書こうとします。負債を残せば自分が刺されるからです。

AI には半年後がありません。現在のセッションでタスクを解決すれば「完了」と認識して終わります。フェーズ A から G まで積み上がった応急処置が将来どれだけの負債になるか、AI には体感がありません。これは AI を責める話ではなく、構造としてそうなっているという話です。

人間側でこれを補わなければ、AI エージェントは構造的に「目先の動作」最適化を続けます。

### 理由 4: 確認バイアスの増幅

これが最も厄介な性質です。フェーズ A で「親店舗側コントローラーのリダイレクトが根本原因だ」と一旦認識すると、次の動作確認で別のエラーが出ても、AI は「あ、別のバグが出てきた」と認識します。**「先程の根本原因の認識自体が間違っていたのでは?」とは認識しません。**

各フェーズで「根本原因を特定 → 修正 → 次のバグが出る → 次の根本原因を特定」を繰り返しているうちに、「7 つも根本原因があった」という奇妙な結論が積み上がります。冷静に考えれば「根本原因がそんなにたくさんあるはずがない、共通の上位構造があるのでは?」と自己批判すべきところを、各フェーズの局所成功体験が自己批判を抑制します。

ユーザーが介入して「**7 個も根本原因が見つかること自体が変ですよね?**」と問うまで、自己修正が起こらない。

### 理由 5: 「正しい抽象」のモデル不在

「プラットフォーム由来のユーザー ID は認証とは別レイヤーの関心事 (= 外部環境固有のアイデンティティ情報) なので、認可とは分離すべき」のようなアーキテクチャ上の気づきは、業務的に「**ユーザーのアイデンティティとは何か**」「**認証と認可の責務分離**」を理解していないと出てきません。

AI はパターン認識は強いのですが、**パターン自体が誤っている時にそれを検出する** のは苦手です。なぜなら「正しい抽象」のモデルがそもそも明示的に与えられていないからです。「認証と認可は別レイヤー」と教科書で読んだ知識は持っていても、目の前のコードが「セッション内のユーザー ID で認可している」と気づいて違和感を覚える、というところまでは行きにくい。

これは 1 ターンの設計レビューなら反応するのですが、**7 段の修正を順次積み上げる流れの中では「違和感センサー」がオフになっている** という点が問題です。

## 4. 効く対策

5 つの理由を整理した上で、現場で効いた / 効きそうな対策を 5 つ書きます。

| 対策 | 何をやるか | 発動条件 | コスト |
| --- | --- | --- | --- |
| 素朴な疑問の投げ込み | 「他の画面と同様にログインしてれば開けるのが期待値ですよね?」 | 修正が 3 段連続で当たらなかった時 | 低 (= 1 発言) |
| 追い打ち質問で逃げ道封じ | 「認証ベースではなくセッション内のユーザー ID に依存している理由を教えて」 | AI が「別の根本原因がありました」と言い始めた時 | 低 |
| 修正回数の上限 | 3 段で一旦止めて「待て、何かおかしい」と聞く | ルールとして事前合意 | 中 (= 流れを止める認知的コスト) |
| メモリ / スキルでのルール化 | 設計前提の実機動作確認必須ルール、3 段連続修正後の自己批判ルール | 永続化 | 中 (= ルール管理コスト) |
| 事前検死 (Pre-mortem) の強制 | 「この修正をデプロイした後、何が次に壊れる?」を AI に答えさせる | 各修正のデプロイ前 | 中 |

### 素朴な疑問の威力

実は今回のケースで効いたのは、技術的に高度な指摘ではなく **超素朴な「他の画面と同じ作りでよくないですか?」** という疑問でした。

これが効く理由は理由 1〜5 全てに同時に効くからです。即時性バイアスを破り (= 即時の検証に直結しない問い)、訓練データのパターン外の問いを投げ (= 「プルリクエストを書く」ではなく「設計を疑う」)、半年後の視点を持ち込み、確認バイアスを解除し (= 既存の根本原因認識を全否定する)、「正しい抽象とは何か」の議論に強制的に引き上げます。

専門用語を使わない素朴な問いの方が、これら全てに同時に効きます。

### 事前検死は事前と事後の両方に要る

事前検死 (Pre-mortem) の概念は、過去記事 [AI は知っているのに使わない — 設計タスクの "task-kickoff" プロトコルで潜在能力を引き出す](https://zenn.dev/fixu/articles/task-kickoff-ai-design-protocol) で「**設計時** のフェーズ 0-3」として組み込み済みです。

今回得た追加知見は、**実装中の各修正の直後にも小さな事前検死が要る** ということでした。「この修正をデプロイした後、何が次に壊れる?」「この修正が間違っている可能性はあるか?」「根本原因が別にある可能性はあるか?」の 3 問を、各修正のコミット前に強制すれば、フェーズ B あたりで「あれ、同じ画面で似たバグが連続している」と AI 自身が気づける可能性があります。

### 修正回数上限 ≒ 連続失敗時の警告灯

「3 段連続で修正を打ったのに動作確認が通らなかったら、一旦止まってアーキテクチャを疑う」というルールは、5 つの理由のうち **理由 4 (確認バイアスの増幅)** に直接効きます。各フェーズの局所成功で麻痺しがちな自己批判を、ルールベースで強制的に発動させる仕組みです。

具体的にはスキル / `CLAUDE.md` に書き込んでおけば、AI 側からも「ルール上 3 段連続失敗のため、一段抽象化した整理を提案します」と発話するようになります。

## 5. まとめ — AI 時代に「素朴な問い」の価値は上がっている

7 段の対症療法を積んだあと、私が投げた問いはこの一文だけでした。

> 「子店舗選択画面を開くときには、他の画面と同様に、ログインさえしていればページを開けることが期待値ではありませんか?」

技術的にはどうということのない問いです。専門知識を要するわけでもなく、「他の画面と一貫していないのは変では?」というだけの問い。それでも、この問いがフェーズ A〜G の全ての応急処置を一段上から見直すフェーズ H の再設計に直結しました。

ここから引き出したい結論は、

> **AI 時代に「専門知識を持つ人間が AI に何をやらせるか」よりも、「専門知識を持つ人間が AI の判断に何を問いかけるか」の方が成果を決めるようになりつつある。**

ということです。AI は実装と症状修正を高速に量産します。それ自体は素晴らしい能力です。が、「**この修正の積み上げ方は変では?**」「**そもそもこの設計の前提は正しい?**」という一段抽象化された問いは、AI からは構造的に出にくい。

前作 [LLM agent に誤前提が 17 連鎖した話](https://zenn.dev/fixu/articles/ai-design-premise-verify-chain) は「上流の誤字が下流まで通り抜けた」設計時の誤前提連鎖でした。本記事は「下流の局所的な修正が層を成して根本リファクタリングを遅らせた」実装時の対症療法連鎖でした。両方とも真因は同じで、**「一段抽象化した問いかけが、人間からの介入なしには起こらなかった**」ことに尽きます。

AI エージェントツールが進化するほど、人間側の役割は「コードを書く」ことから「**AI の判断軸そのものを揺さぶる素朴な問いを投げ込む**」ことへとシフトしていく気がしています。コードを書ける人より、素朴な問いを投げ込める人の方が、AI 時代には希少資源かもしれません。

自分を含めて、3 段で止まれなかった全エンジニア仲間に、この記事を捧げます。
