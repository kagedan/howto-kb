## 作ったもの

Claude Code の **5時間レートリミットが特定の % に達したら Discord に通知**してくれる仕組みを作りました！

![](https://static.zenn.studio/user-upload/a4068e521946-20260712.png)

リポジトリはこちら。

<https://github.com/NKMAK/claude-code-limit-alert-notice>

チェックの仕組みは2種類あり、併用もできます。

* **定期チェック**: macOS 標準のスケジューラで5分おきに使用量を確認
* **hooks 方式**: Claude Code の応答が返ってきたタイミングで確認

hooks 方式はClaude Code の応答が返り終わった瞬間に発火するので即時性が高いのですが、**1ターンが終わるまで一切チェックされない**という欠点もあります！  
サブエージェントを多用するような長いターンだと、1ターンの間に一気に % が進み、フックが発火したときには 80% を飛び越えて 100%…ということがありえます。

個人的には定期チェックが本体だと思っていて、5分おきに必ず見にいくので「気づかないうちに 100%」はまず起きないです！  
両方 ON にもできるので、迷ったら併用がおすすめです！

## 作ろうと思った背景

Claude Code で作業していたら、気づいたら使用量が **100% に到達**してしまっており、これ以上作業ができないよ😭という状態によくあっていました。

80% くらいの段階で気づけていれば、引き継ぎプロンプトを作って Codex に委任する、といった逃げ道も取れたのに…という経験があり、「閾値を超えたら勝手に通知してくれる仕組み」が欲しくなったのが きっかけです。

## できること

* **5hレートリミットの閾値ごとにdiscord通知**: 設定された閾値に達したらdiscordにメンション付きで通知してくれます（重複通知なし）
* **通知内容が正確**: Claude Code 本体の `/usage` と同じサーバー側の値を使うので、表示とズレない。通知には現在の使用率・リセット時刻・週間使用率が載る
* **メンション対応**: `DISCORD_MENTION="<@自分のID>"` を設定すると通知にメンションが付き、Mac のデスクトップ通知（バナー）で確実に気づける
* **トリガーの切り替え**: 定期と hooksチェックを `.env` のフラグでON/OFF。両方ONにすると取りこぼしが最も少ない

| 方式 | 即時性 | 長いターンの途中で閾値通過を捕捉 |
| --- | --- | --- |
| 定期チェックのみ | △（最大5分遅れ） | ◎ |
| hooks のみ | ◎（応答直後） | ✕（長いターンで飛び越えることがある） |
| 両方ON | ◎ | ◎ |

## セットアップ

### 必要なもの

| 項目 | 要件 |
| --- | --- |
| OS | macOS |
| Claude | Claude Code |
| 通知先 | Discord（Webhook を作成できるチャンネル） |

### Claude Code にお任せセットアップ（推奨）

このリポジトリには対話セットアップ用のスキル `/setup` を同梱しています。clone して Claude Code を起動し `/setup` と打つだけで、閾値・メンション・トリガーのヒアリング → `.env` 生成 → インストール → 実通知テストまで対話形式で完了します。

```
git clone https://github.com/NKMAK/claude-code-limit-alert-notice.git
cd claude-code-limit-alert-notice
claude   # 起動して「/setup」を実行
```

Discord Webhook の作り方やフルディスクアクセスの付与手順も、その場で Claude が案内してくれます。

### 手動セットアップ

```
git clone https://github.com/NKMAK/claude-code-limit-alert-notice.git
cd claude-code-limit-alert-notice
sh install.sh
# .env に Webhook URL を設定
```

`install.sh` が launchd への登録・Stop フックの登録・ラッパーのビルドまで自動でやってくれます。詳細はリポジトリの README を参照してください。

## 技術構成

全体の流れはシンプルで、**POSIX シェルスクリプト1本**が本体です。

```
launchd（5分おき）─┐
                    ├→ usage-alert.sh → claude -p "/usage" で使用率取得
Stopフック ────────┘        ↓
                      閾値判定（通知済みならスキップ）
                            ↓
                      Discord Webhook に POST（curl）
```

### 使用量の取得: `claude -p "/usage"`

使用率は `claude -p "/usage" --output-format json` で取得しています。これは Claude Code 本体の `/usage` コマンドと同じ**サーバー側の正確な値**（セッション使用率・リセット時刻・週間使用率）が返ってくるうえ、ローカル処理なので**課金トークンを消費しません**。

最初は ccusage でローカルのログからトークン数を集計し、上限を推定して % を近似する方式でした。しかし実際の上限はトークン数に正比例せず、校正してもすぐズレるため、`/usage` の公式値に全面移行しました。

### 重複通知の防止: 状態ファイル

「50% の通知が5分おきに何度も飛んでくる」と困るので、通知済みの閾値を状態ファイル（`~/.claude/.usage-alert-state`）に記録しています。

* 判定単位（ウィンドウ）は **リセット時刻**。リセット時刻が変わったら新しい5hウィンドウとみなして発火履歴をクリア
* launchd と Stop フックの両方から同時に呼ばれても、`mkdir` によるロックで多重起動を防止
* 状態ファイルは毎回1行を上書きするだけなので、使い続けても肥大化しない

### 許可モーダル対策

launchd からスクリプトを定期実行すると「ほかのアプリからのデータへのアクセス」の許可モーダルが出ます。

![](https://static.zenn.studio/user-upload/3955bcd86e74-20260712.png)  
![](https://static.zenn.studio/user-upload/0cf09b0c6502-20260712.png)  
厄介なのは、macOS がこの許可を「実行者のファイル」単位で記録することでした。シェルスクリプトを直接起動する構成だと、実行者は `claude` 本体とみなされます。ところが `claude` の実行ファイルは**バージョンごとに別ファイル**なので、自動更新されるたびに知らない実行者扱いになり、**更新のたびにモーダルが再出現**してしまいます。

そこで、launchd の起点にスクリプトを起動するだけの小さな自作バイナリを置き、**実行者をこのバイナリに固定**しました。実行者が変わらないので、システム設定でこのバイナリにフルディスクアクセスを1回付与すれば、`claude` が何回更新されようがモーダルは二度と出ません。ビルドは `install.sh` が自動でやるので、利用者は特段意識する必要はないです！

## まとめ

* Claude Code の使用量が閾値に達したら Discord に通知してくれるツールを作った
* 「気づいたら 100%」になる前に、80% の段階で引き継ぎ準備などの手が打てるようになった
* セットアップは `/setup` スキルで Claude Code にお任せできる
