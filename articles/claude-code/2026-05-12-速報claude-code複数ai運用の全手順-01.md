## まず結論

![](https://assets.st-note.com/img/1778545175-w6lhBYHGe8gJZLpU9rA1TtRE.png?width=1200)

Agent viewは、Claude Codeを「**複数AIを回す作業管理画面**」に変える機能です。

Anthropicは2026年5月11日、Claude Code向けに**Agent view**をResearch Previewとして公開しました。

公式ブログでは、複数のClaude Codeセッションを1画面で管理し、状態確認やインライン返信ができる機能として説明されています。

覚える入口は3つです。

* **`claude agents`: Agent viewを開く**
* **`/bg`: 現在のセッションをバックグラウンドへ送る**
* **`claude --bg`: 新しいバックグラウンドセッションを始める**

重要なのは、コマンドそのものではありません。  
Claude Codeを「相談相手」ではなく「複数の作業者」として扱う設計が必要になったことです。

## Agent viewで何が変わったか

これまでは、Claude Codeを1つの作業窓として使う人が多かったはずです。

全部を1つの流れで進めると、待ち時間が増えます。

さらに、調査ログやエラー出力が同じ文脈に混ざり、後半の判断も重くなります。

Agent viewは、この使い方を変えます。

複数のセッションを並べて見られるため、作業を分けやすくなります。

たとえば、メインでは実装を進めながら、別セッションに関連仕様の確認を任せる。

さらに別セッションにテスト失敗の原因調査を任せる。

人間は、全部の作業を1列で待つ必要がなくなります。

代わりに、どの作業をどのAIへ渡すかを決める役になります。

## 3つの入口だけ覚える

最初に覚える操作は3つだけです。

---

### 1つ目は、`claude agents`です。

Agent viewを開くための入口です。  
複数のセッションを確認し、必要に応じて返信できます。

---

### 2つ目は、`/bg`です。

いま開いているセッションをバックグラウンドへ送る操作です。  
長めの調査や待ち時間がある作業を、表の作業から切り離せます。

---

### 3つ目は、`claude --bg`です。

新しいバックグラウンドセッションを開始する操作です。  
最初から裏で走らせたい調査や確認に向いています。

この3つを覚えるだけで、Claude Codeの見え方が変わります。

1つのAIに順番待ちするのではなく、複数の作業を同時に走らせる前提で考えられます。

## 実務では何を任せるべきか

Agent viewで最初に分けるべき作業は、成果物が明確なものです。

向いている作業は5つあります。

* **仕様調査**
* **既存コードの影響範囲確認**
* **テスト失敗の原因調査**
* **実装案の比較**
* **PRレビュー前のセルフチェック**

逆に、最初から丸投げしない方がいい作業もあります。

* **要件が曖昧な新機能**
* **判断基準がないレビュー**
* **セキュリティ影響が大きい変更**
* **ユーザー体験の最終判断**
* **事業判断を含む仕様決定**

複数AI運用で強いのは、作業を増やすことではありません。

判断しなくていい作業を外に出し、人間が見るべき判断に集中することです。

## 失敗する使い方

Agent viewで失敗する人は、複数セッションを増やすだけで終わります。

セッションが増えると、一見すると仕事が進んでいるように見えます。

ただし、出口が決まっていない作業を増やすと、確認コストも増えます。

失敗パターンは3つです。

* **何を返してほしいか書かない**
* **完了条件を書かない**
* **最終判断をどこで人間が見るか決めない**

たとえば「このコードを見て」とだけ投げると、戻ってくる情報が広がりすぎます。

代わりに、こう書く方が実務向きです。

```
この変更の影響範囲だけ調査してください。
出力は3項目です。
1. 影響を受けるファイル
2. 壊れそうなテスト
3. 人間が確認すべき判断
実装変更はしないでください。
```

AIを増やすほど、指示は短くても構造化する必要があります。

## 今日から試す運用テンプレ

最初は、3役だけで十分です。

```
Main:
実装または本文作成を進める

Background A:
関連仕様、既存コード、公式ドキュメントを調べる

Background B:
テスト失敗、矛盾、抜け漏れを確認する
```

Claude Codeで試すなら、流れはこうです。

```
claude agents
```

Agent viewを開きます。

次に、いまの作業を裏へ回します。

```
/bg
```

新しい裏作業を始める場合は、こうです。

```
claude --bg
```

この時、各セッションに渡す指示は「役割」と「完了条件」をセットにします。

```
あなたの役割は調査担当です。
実装はしないでください。
この機能に関係する公式情報だけ確認してください。
出力は、根拠URL、重要ポイント、未確認点の3つにしてください。
```

これだけで、Claude Codeの使い方はかなり変わります。

## 次にやること

Agent viewを使う前に、まず自分の仕事を3つに分けてください。

* **自分が判断する作業**
* **AIに調べさせる作業**
* **AIに確認させる作業**

ここが曖昧なまま複数セッションを増やしても、確認する情報が増えるだけです。

Claude Codeは、プロンプトが上手い人だけの道具ではなくなっています。

これから差が出るのは、AIに渡す作業の分け方です。

Claude Codeを入れたけれど、何を任せるべきか分からない人は、最初に業務棚卸しをしてください。

次に、調査、実装、レビュー、テストをどこまでAIに渡すか決める。

それができると、Claude Codeは「便利なチャット」ではなく、自分の作業を進める小さなチームになります。
