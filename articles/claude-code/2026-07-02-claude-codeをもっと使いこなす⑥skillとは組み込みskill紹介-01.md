# Claude Codeをもっと使いこなす⑥｜Skillとは？組み込みSkill紹介

こんにちは、開発日和です。

スラッシュコマンドの中に `/review` や `/simplify` のような便利なコマンドがありましたよね。実はあれも「Skill」という仕組みの上に作られているらしいです。

今回はSkillの概念と、最初から使えるビルトインSkillを紹介します。

---

## Skillとは何か

Skillは、**繰り返し使う操作をスラッシュコマンドとして登録する仕組み**です。

例えば、「このプロジェクトのコードをレビューして、修正も適用して」という一連の操作を、毎回長いプロンプトで書く代わりに `/review --fix` の一言で呼び出せるようにしたもの、というイメージです。

### ↓Skillがない状態で、毎回これを書いていたとします。

```
> このブランチの変更をセキュリティの観点でレビューして、
  問題点を優先度順にリストアップし、修正できるものは直接修正して
```

### ↓Skillとして登録すれば、これだけで指示できるようになります。

---

## ビルトインSkillの一覧

Claude Codeには最初から使えるSkillが複数あります。

### `/code-review` ― コードレビュー

```
> /code-review
> /code-review --effort high    # 徹底的に
> /code-review --fix            # 指摘箇所を直接修正
> /code-review --comment        # PRコメントとして投稿
```

現在のブランチの差分を対象にレビューします。`--fix` をつけると指摘だけでなく修正まで行います。

### `/security-review` ― セキュリティ特化型レビュー

セキュリティの観点に特化したレビュー。SQLインジェクション・XSS・認証の不備などを重点的にチェックします。

### `/simplify` ― シンプルにリファクタリング

コードの重複・不必要な複雑さ・改善できる書き方を見つけて修正します。バグ探しではなく、きれいさを上げるためのSkillです。

### `/init` ― CLAUDE.md生成

プロジェクトを解析してCLAUDE.mdを生成します。Part3で詳しく解説しました。

### `/run` ― アプリを起動して確認

プロジェクトのアプリを実際に起動して動作確認します。フロントエンドがあるプロジェクトで特に便利です。  
バックエンドでも、コードから読み取ってdockerを起動してくれたりします。

### `/verify` ― 変更が意図通りか確認

コードの変更が期待通りに動いているかを実際に動かして確認するSkillです。  
git logやgit diffなどが使用され、変更点をチェックするところから動きます。  
「どのように変更点を把握しているのか」も、claudecodeが実行しているコマンドから読み取れます。

## Skillとスラッシュコマンドの違い

ちょっと整理しておきます。

| 区分 | 例 | 特徴 |
| --- | --- | --- |
| スラッシュコマンド | `/clear`, `/model`, `/usage` | Claude Codeの基本操作 |
| ビルトインSkill | `/review`, `/simplify` | 複雑な処理をワンコマンド化されているもの |
| カスタムSkill | 自分で作るもの | プロジェクト・個人の用途に合わせて作る |

コマンドはツールの設定操作系、Skillは「作業をやってもらう」系、というざっくりした分け方で覚えるとわかりやすいです。

### スラッシュコマンド

`/help`で確認できるものがスラッシュコマンドです。  
これらは Claude Code のバイナリに内蔵されていて、Skill ファイルとは別物です。  
![](https://static.zenn.studio/user-upload/5b0a76841ea5-20260702.png)

### ビルトインSkill

プロンプトとして定義されたマクロです。  
実体は Markdown ファイル（プロンプト）であり、実際には、claudeはそのプロンプトに従って動くだけ。

---

## つまずいたポイント

**`/review` と `/code-review` の違いが分かりにくい**

実は `/review` は `/code-review` のエイリアス（別名）です。どちらを使っても同じです。公式ドキュメントやリリースノートで表記が揺れることがあるので、動かなかったら別の表記を試してみてください。

**Skillがあることを知らずに長いプロンプトを書き続けていた**

最初の頃、レビューをお願いするたびに「このブランチの変更を〇〇の観点で...〇〇の部分を重点的に...」と長文を書いていました。`/review` を知ってから劇的に楽になりました。PRの番号指定でもレビュー依頼できるので、かなり便利です。まずビルトインSkillを一通り試してみるのをおすすめします。

---

## まとめ

| Skill | 使い所 |
| --- | --- |
| `/review` | コミット前のレビュー、PRのレビュー |
| `/security-review` | 認証・決済・個人情報を扱う実装のセキュリティチェック |
| `/simplify` | 動いてるけどもっときれいにしたいとき |
| `/init` | 新プロジェクト・CLAUDE.mdを作りたいとき |
| `/run` | 実際に動かすコマンドや作業が面倒なとき |
| `/verify` | 変更が意図通りになっているかを確かめたいとき |

次回のPart7では、自分でSkillを作る方法（カスタムSkill）を解説します。
