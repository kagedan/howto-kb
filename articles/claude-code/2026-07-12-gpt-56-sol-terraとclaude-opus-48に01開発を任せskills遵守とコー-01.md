## 本記事の前提

2026年7月12日に、以下のモデルへ同じ0→1開発を任せた。

* GPT-5.6 Terra Medium
* GPT-5.6 Terra High
* GPT-5.6 Sol Medium
* GPT-5.6 Sol High
* Claude Opus 4.8 Medium
* Claude Opus 4.8 High

各モデル・effortで試したのは1回ずつ。

厳密なベンチマークではなく、自分の環境における一度の実装結果と所感になる。

また、GPTはCodex CLI、ClaudeはClaude Code CLIを利用している。Skillsは同等の内容になるよう用意したが、ツールまで含めた完全な同一条件ではない。

## 免責事項

本記事内の比較結果は、厳密なベンチマークではない。

検証当初は記事化することを想定しておらず、各モデルの所要時間や消費トークンなど、具体的な数値をすべて記録していなかった。  
そのため、時間とトークンについては、一部を除いて相対的な比較になっている。  
また、Skillsの理解度、ロジック品質、UT品質については、生成されたコードを確認したうえでの自分の評価であり、定量的な基準に基づくものではない。  
各モデル・effortで試した回数も1回ずつなので、モデル全体の傾向として断定できる結果ではない。

## 何を確認したかったか

Codex CLIとClaude Code CLIのユーザーレベル設定に、主に以下を定めたSkillsを配置している。

* オニオンアーキテクチャで実装する
* UTは権威的なドキュメントを参考に観点を洗い出し、整理してから実装する

実際のSkillsには、レイヤーごとの責務や依存方向、テスト対象、モック範囲などをさらに細かく書いている。

今回確認したかったのは、単純に動くコードを書けるかではない。

**実装内容から必要なSkillsを自動で選び、Skillsの内容とモデル自身の能力を組み合わせて実装できるか**を確認したかった。

## 実装させたもの

コードレビューで発生した指摘コメントと、そこから汎用化したレビュールールを管理するDjangoアプリケーションを実装させた。

主な要件は以下。（実際の内容は[こちら](https://gist.github.com/koki-nakamura22/15e24efd703298eeae2721fd669ee2ca)。）  
この内容は、ChatGPTに作らせたものである。

* Django adminからルールや指摘コメントを登録・編集する
* 通常画面からルールや指摘コメントを検索・閲覧する
* 複数条件で絞り込む
* 公開済みルールをMarkdownとして出力する
* JSON APIからルールと指摘コメントを取得する
* 状態遷移、重要度、適用範囲、匿名化ポリシーを扱う
* Docker ComposeとPostgreSQLを使用する
* pytestでUTを書く

環境はWSL2 Ubuntu。

要件定義ファイルだけが存在するディレクトリから、Djangoプロジェクト全体を作らせた。

モデルへ渡した指示は以下だけ。

```
@docs/requirements.md を元に実装して。
```

プロンプトからSkillsの利用は明示していない。

モデル・effortごとに新しいセッションを開始し、前のモデルが生成したコードや会話内容は見えない状態にした。

## 結果

| モデル | Skills遵守 | ロジック | UT | 時間・トークン |
| --- | --- | --- | --- | --- |
| GPT-5.6 Terra Medium | domain層のみ | 詳細評価せず | 詳細評価せず | 未整理 |
| GPT-5.6 Terra High | domain層のみ | Mediumより少し良い | 詳細評価せず | 未整理 |
| GPT-5.6 Sol Medium | 遵守 | 概ね満足 | ケース不足 | 未整理 |
| GPT-5.6 Sol High | 遵守 | 満足 | Mediumより良いが不足あり | 比較基準 |
| Claude Opus 4.8 Medium | 概ね遵守 | 細部に粗さあり | GPTより良い | 約2倍 |
| Claude Opus 4.8 High | 遵守 | 明確に良い | 明確に良い | 約3倍 |

時間とトークンは、GPT-5.6 Sol Highを基準にした概算。

## GPT-5.6 Terra Medium / High

結論、今回の実装では採用できるレベルとは思えなかった。

domain層はSkillsに沿っていたが、それ以外の層は独自の構成で実装していた。

Terra Highへ理由を確認すると、以下の趣旨の回答が返ってきた。

> まずは初期実装を一通り揃えることを優先した。

Skillsを読み取れなかったわけではなく、認識したうえで、Skillsに沿うことよりも、一通り実装することを勝手に優先していた。（何のためのSkillsなんだ？となった。）

Terra HighはMediumよりも少しコードの解像度が高かったが、同様にSkillsを無視したため、ロジックやUTの詳細な評価までは行っていない。  
違反を指摘した後に修正できるかも試していない。

## GPT-5.6 Sol Medium / High

Sol Mediumは、オニオンアーキテクチャを含むSkillsに沿って実装した。  
レイヤーごとの責務や依存方向も意図した構成になっており、ロジック品質には概ね満足している。  
一方で、UTはケース数が少なかった。  
最低限のテストは存在するが、業務ルールや分岐を十分に網羅できていない。人間による追加が必要という印象。

Sol HighもSkillsに沿って実装した。  
MediumよりもSkillsの内容を理解し、個別の責務分割や依存関係へ反映できていた。  
ロジック品質には満足している。  
UTもMediumより良かったが、網羅性にはまだ不足があった。  
今回の結果では、**GPTを使い、Skillsとモデル自身の能力を組み合わせた実装を期待するなら、Sol Highからが候補**になった。

## Claude Opus 4.8 Medium / High

Opus Mediumは、パッと見では良さそうなコードを生成した。  
UTはGPTより筋が良かった。  
特に、境界値と業務ルールまでテスト対象に含めていた点が良い。  
一方で、プロダクションコードには以下の粗さがあった。

要件は満たしているが、そのまま採用するには少し気になるという評価。  
時間とトークンは、Sol Highのおおよそ2倍だった。

Opus Highは、ロジックとUTの両方が明確に良かった。  
UTでは、状態遷移、境界値、保存ポリシーと匿名化状態の組み合わせ、不正な永続化状態など、要件に書かれた業務ルールをテストケースへ変換できていた。  
Mediumで気になった責務配置、命名、型の粗さも大きく減っていた。  
時間とトークンは、Sol Highのおおよそ3倍。  
品質は良いが、毎回使うには重い。

## 実際の運用をどうするか

現時点では、以下の2モデルが候補。

### GPT-5.6 Sol High

* Skillsを守る
* ロジック品質に満足できる
* Opus Highより速く、消費トークンも少ない
* UTは人間による確認と追加が必要

コストや速度とのバランスを考えると、基本候補はこちらになりそう。

### Claude Opus 4.8 High

* ロジックとUTの初回品質が高い
* 境界値や業務ルールまでUTへ落とし込める
* Sol Highの約3倍の時間とトークンを使う

初回からロジックとUTの両方を高い水準で揃えたい場合はこちら。

別案として、以下も考えている。

1. Opus Mediumで初期実装とUTを作る
2. Sol HighにSkills適合とコードの細部を修正させる

Opus MediumのUT観点と、Sol HighのSkills遵守・ロジック品質を組み合わせられるかもしれない。  
ただし、まだ試していない。  
二重にコードを読ませることで、Opus High以上に時間やトークンを使う可能性もある。  
が、gptのコード全体像を把握しての編集に対する評価の高さから、期待はできる。

## おまけ：GPT-5.5 Highとの使い分け

今回の同一条件比較とは別だが、少し触った感じ、明示した内容をそのまま実行させるだけならGPT-5.5 Highでも良いと感じている。

ただ、指示された作業の実行以外の痒いところに手を伸ばしてほしい場合は、GPT-5.6 Sol Highを使った方が良いかなという所感。  
要は使い分け。（要はケースバイケースおじさん）

## 結論

今回の結果では、  
**Skillsとモデル自身の能力を組み合わせた実装を期待するなら、GPT-5.6 Sol HighまたはClaude Opus 4.8 Highからが良さそう**  
だった。

コストと速度を含めた基本候補はGPT-5.6 Sol High。  
ロジックとUTの初回品質を優先するならClaude Opus 4.8 High。

ただし、各モデルを試したのは1回ずつ。  
モデル全体の傾向として断定できる結果ではない。

次は、Opus Mediumで実装したコードをSol Highへ修正させる構成を試したい。  
モデルの時間やトークンだけではなく、人間がレビューと修正に使った時間も含めて比較しないとな。。。  
あと、書いたコードの修正を、今回使ったモデルにそれぞれ指示した結果も試してみたい。。。
