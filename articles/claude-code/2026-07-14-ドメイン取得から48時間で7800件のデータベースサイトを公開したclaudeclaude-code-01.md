## この記事の対象読者

* Claude Code（またはCursorなどのAIコーディングツール）を使っているが、「指示→実装」の単純な往復で止まっている方
* AIに任せる範囲と人間が握るべき範囲の線引きに悩んでいる方
* 個人でWebサービスやメディアを運営していて、立ち上げを高速化したい方

逆に、プログラミング入門やClaude Codeのセットアップ方法は扱いません。

## 結論：AIを2つの役割に分けたら、立ち上げが2日で終わった

先日、行政の給付金情報を検索できるデータベースサイト（約7,800件収録）を立ち上げました。ドメイン取得からデータパイプライン構築、SEO実装、ファクトチェック済みブログ記事10本の公開、週次の自動運用体制の構築まで、**かかった期間は実質2日**です。

従来の個人開発では、この規模のサイトはリリースまで数ヶ月かかるのが相場でした。AIを使う開発者の間でも「2週間〜1ヶ月」がよく言われるラインです。それが2日で済んだ最大の要因は、AIの性能ではなく**体制の組み方**にありました。

具体的には、こういう三者分業です。

| 役割 | 担当 | やること |
| --- | --- | --- |
| 設計レビュー係 | Claude（チャット） | 方針の壁打ち、成果物のレビュー、次の指示書の作成 |
| 実装係 | Claude Code | 調査、コーディング、デプロイ、検証 |
| 承認係 | 人間（私） | チェックポイントでの承認、アカウント操作、最終判断 |

ポイントは、**Claude Codeの完了報告を、私ではなくもう1つのClaudeにレビューさせる**ことです。私は2つのAIの間で成果物と指示書を運び、要所で判断するだけ。この記事では、この体制の具体的な回し方と、実際に起きた問題、そして正直な限界をお話しします。

このサイトを作ろうと思ったきっかけは、給付金や手当の情報が行政機関ごとに分散していて、自分に関係する制度を探すのが思った以上に大変だと感じたことです。せっかく公開されている情報なのに、必要な人に届きにくいのはもったいない。そこで、まずは検索しやすい形にまとめたらどこまで役に立つものになるのか、個人開発として試してみることにしました。

## なぜ「指示→実装」の単純往復では速くならないのか

Claude Codeは優秀ですが、ふわっとした指示を渡すと、ふわっとした方向に全力疾走します。私も以前は「〇〇なサイトを作って」と指示して、出てきたものを見て修正指示を重ねる使い方をしていました。これだと手戻りが多く、結局人間がボトルネックになります。

試行錯誤の末にたどり着いたのが、**フェーズ分割とチェックポイント駆動**です。

```
フェーズ1: 調査と設計提案（実装禁止）
   ↓ 人間が承認
フェーズ2: 基盤構築（デプロイ禁止）
   ↓ 人間が承認
フェーズ3: 公開準備とデプロイ
   ↓ 人間が承認
フェーズ4: コンテンツ公開とデザイン
```

各フェーズのプロンプトには、必ず「今回やること」と同じ分量の **「やってはいけないこと」** を書きます。たとえばフェーズ1ならこんな制約です。

```
## 制約・注意事項
- 実装・ファイル作成・デプロイはまだ行わないこと。調査と提案レポートのみ
- 提案は「推奨案 + 代替案」の形式で、判断材料を示すこと
- 不明点や私の判断が必要な箇所は、最後にまとめて質問リストとして出すこと
```

「推奨案＋代替案」の形式を強制するのが地味に効きます。AIは放っておくと1つの案を確信度高く出してきますが、代替案とトレードオフを併記させると、人間が判断できる材料になるからです。

## 「レビュー係のClaude」が何をしているのか

この体制の肝は、Claude Codeの完了報告を**別のClaude（チャット側）に読ませてレビューさせる**工程です。

実例を挙げます。フェーズ2の完了報告で、Claude Codeは「実データ約7,800件を取り込み、ビルド成功」と報告してきました。一見完璧です。しかし報告の末尾の申し送り事項に「データファイル（52MB）のデプロイ時の扱いが未決定」という1行がありました。

レビュー係のClaudeはここに反応し、「Vercelのサーバーレス環境で52MBのJSONを毎回読む構成はコールドスタートが悪化する。ISR再検証時の挙動もローカルと変わる可能性がある。**次のフェーズより先にこれを潰すべき**」と指摘し、解決策を3案（軽量インデックス抽出／1件1ファイル分割／外部ストレージ）比較した上で修正指示のプロンプトを作ってくれました。

結果、「一覧用の軽量インデックス2.45MB＋詳細は1制度1ファイル」というハイブリッド構成に落ち着き、本番で問題が起きる前に構造ごと解決できました。

人間の私がこの報告を読んでいたら、「ビルド成功」の文字に安心して申し送りを流し読みしていたと思います。**AIの報告書はAIに読ませる方が、見落としが減る**というのが実感です。

もう一つレビュー係に助けられたのが、ブログ記事のファクトチェックの設計です。当初は「記事を書いた後、そのままAIに内容を確認・修正させればよい」と考えていましたが、レビュー係から、検証と修正を一度に任せると、どこが間違っていて何を直したのか人間が追いにくくなると指摘されました。

そこで、「まず不一致を報告するだけ」「修正は人間が確認してから別工程で行う」という流れに変更しました。この分離が実際に何を防いだかは、次の章でお話しします。

## YMYL領域だったので、あえて「全自動にしない」設計にした

このサイトは給付金・手当というお金の情報を扱います。いわゆるYMYL（Your Money or Your Life）領域で、金額を1桁間違えれば読者に実害が出ます。

そこで、AI活用の記事なのに逆説的ですが、**「AIが自動でできてしまうこと」に人間のゲートを意図的に挟みました**。

### ファクトチェックを「報告と修正」に分離する

前章で触れた「検証と修正の分離」の具体的な形です。ブログ記事はClaude Codeが下書きしますが、公開前に全記事の事実主張（金額・対象者・期限）をWeb検索で一次情報源と突き合わせさせ、結果を表形式のドキュメントに記録させます。プロンプトではこう表現しています。

```
各記事について、本文中のすべての事実主張を抽出し、
一次情報源と突き合わせて検証すること。
不一致・確認不能の項目は記事を修正せず報告のみ
（修正判断は私が行う）
```

実際にこの仕組みで、10記事中2件の不一致が見つかりました。1つは制度の開始経緯の説明のズレ、もう1つは申請窓口となる実施主体の記述の誤りです。どちらも「検出→人間が修正要否を判断→修正→再検証」の流れで潰しました。検出と修正を分けるのは一手間ですが、AIが「検証のついでに良かれと思って書き換える」事故を構造的に防げます。

### 週次の自動運用も「公開だけは絶対にさせない」

サイトの運用はClaude Codeのスケジュール実行に任せていますが、その指示書（SKILL.md）にはこう書いてあります。

```
**公開判断（draft: falseへの変更）は必ずユーザーが行う。
このルーチンが自動で公開することは絶対にない。**
```

さらに、公開済み記事の内容修正はmainブランチへの直接コミットを禁止し、必ずレビュー用ブランチに隔離。ビルド失敗時のリトライは2回まで、本番確認に失敗したら自動でrevert、無人で触ってよいコード修正は3種類に限定列挙——という具合に、**「してはいけないこと」を「できること」より丁寧に書いています**。

無人実行のAIに渡す指示書は、性善説で書くと事故ります。「迷ったら書かない・公開しない・ログに残して人間に聞く」という安全側の原則を明文化しておくのがおすすめです。

## 正直な話：圧縮できなかったもの

いいことばかり書くとフェアではないので、限界も書いておきます。

**Googleの評価の時計は圧縮できません。** サイトは2日で作れても、検索エンジンにインデックスされ、順位がつき、収益が立つまでの数ヶ月は今まで通りかかります。だからこそ「早く公開してクロールの時計を回し始める」ことに価値があった、とも言えます。

**アカウント操作は結局人間です。** DNS設定、アナリティクスのプロパティ作成、ドメイン購入。この辺りはAIに認証情報を渡すリスクの方が大きいので、私が手を動かしました。2日のうち、体感で2〜3割はこの手の作業です。

**過去の失敗資産がないと、この速度は出ません。** 私は同型のサイトを2つ運用しており、そこで踏んだ地雷（内容の薄いページでの広告審査落ち、構造化データへの意図しないURL混入など）の対策をプロンプトに埋め込めました。1サイト目からこの速度が出るわけではない、というのは正直に書いておきます。

## まとめ

* Claude（設計レビュー）× Claude Code（実装）× 人間（承認）の三者分業で、データベースサイトの立ち上げが実質2日で完了した
* 速さの源泉は「フェーズ分割＋チェックポイント」と「AIの報告をAIにレビューさせる」体制
* YMYL領域では、あえて自動化を止める人間ゲート（公開承認・ファクトチェックの検出と修正の分離）を仕込む
* ただしSEOの時間とアカウント操作は圧縮できない

同じような体制を組みたい方の参考になれば嬉しいです。
