2026年5月19日にAnthropicが公開した「自社サンドボックス（Self-Hosted Sandboxes）」と「MCPトンネル」は、Claude Managed Agentsの**ツール実行をどこで動かすか**を選べるようにした機能だ。

この2つの機能を正確に理解するには、まず Managed Agents の構造を分解する必要がある。

## TL;DR（チェックリスト）

---

## エージェントループとツール実行の分離

Managed Agentsの内部は大きく2レイヤーに分かれる。

| レイヤー | 内容 | 配置 |
| --- | --- | --- |
| **エージェントループ** | オーケストレーション・コンテキスト管理・エラーリカバリ | **常にAnthropicインフラ** |
| **ツール実行** | Bash・ファイル操作・MCP呼び出し | 自社サンドボックスに移せる |

今回の変更で「ツール実行レイヤーだけを自社インフラ（またはマネージドプロバイダ）に置ける」ようになった。コードは社内ネットワークを出ない。ただしセッション状態の永続化はAnthropicのサーバサイドで行われるため、ゼロデータリテンション（ZDR）やHIPAA BAAの対象外である点は変わらない。

アーキテクチャを図にすると、こうなる。

「エージェントが考える部分（ループ）はAnthropicに残り、エージェントが手を動かす部分（ツール実行）だけを自社に引き寄せる」——これが設計の核心だ。

---

## どちらを使うかの判断基準

自社サンドボックスが**必要な条件**は3つだ。

1. **データが自社ネットワーク外に出せない**：社内Git・プライベートレジストリ・機密ファイルをエージェントが直接触る用途
2. **特定のコンピューティングリソースが必要**：大規模ビルド、画像生成、長時間バッチなどデフォルトサンドボックスのリソース制限に引っかかるジョブ
3. **既存の監査ログ基盤・ネットワークポリシーに組み込みたい**：社内セキュリティツールとの統合

上記に該当しない「公開APIを叩くだけ」「外部パッケージをインストールするだけ」のジョブは、Anthropicのクラウドサンドボックスで十分だ。サンドボックスプロバイダの管理コストを増やす前に、本当に必要かを確認する。

---

## サンドボックスプロバイダの選び方

Anthropicが動作確認済みのプロバイダは現時点で4つある。

* **Cloudflare**：microVM＋isolate。ゼロトラスト秘密情報注入とエグレスプロキシ設定が使える。「エージェントの外向き通信を細かく制御したい」場合の第一候補
* **Daytona**：長時間ステートフル。SSH接続・一時停止＆復元が可能。「数時間動き続けてファイルシステム状態を保ちたい」コーディングエージェントに向く
* **Modal**：サブ秒起動・CPU/GPUオンデマンド・大量並列。AI推論ワークロードの並列スケールに強い
* **Vercel**：VPCピアリング＋ネットワーク境界でのクレデンシャル注入。金融・フィンテックのような厳格なセキュリティ要件向け

選択の軸は「**長時間ステートフルか、コールドスタートで大量並列か**」で大半が決まる。

## データ境界の整理：何がどこに残るか

自社サンドボックスを使うからといって、すべてのデータが自社に閉じるわけではない。コンプライアンス判断の前に、どのデータがどこに留まるかを整理しておく。

| データの種類 | Anthropicインフラ | 自社サンドボックス |
| --- | --- | --- |
| モデルへのプロンプト / レスポンス | ✅ 経由する | — |
| セッション状態・会話履歴 | ✅ 保存される | — |
| ツール実行の入力・出力（結果） | ✅ ループが受け取る | ✅ 実行はここで |
| ツールが触るファイル・データ | — | ✅ 自社のみ |
| 社内DBのレコード（MCPトンネル経由） | ❌ 通過しない | ✅ 自社のみ |

要点は「**ツール実行の結果はエージェントループ（Anthropicインフラ）に返る**」という点だ。「社内DBを読んだ結果」はいったんツール出力としてAnthropicのサーバに渡る。その結果をモデルが処理し、次のツール呼び出しを決定する——このサイクルがAnthropicのインフラ上で完結する。

ZDR（ゼロデータリテンション）が必要なら、Managed Agentsではなく **Messages APIで自前のエージェントループを実装する** 構成を選ぶ。これはZDR対応の追加コストと天秤にかける設計判断だ。

---

## MCPトンネルの構造

MCPトンネルは「社内ネットワークにあるMCPサーバをエージェントから呼ぶ」ための経路だ。

仕組みはシンプルで、自社側に軽量ゲートウェイをデプロイし、そこから**外向き1接続だけ**Anthropicのインフラに張る。インバウンドのファイアウォール解放は不要で、通信はE2E暗号化される。エージェントから見ると、社内DB・プライベートAPI・ナレッジベースが「呼び出せるツール」として見える。

現状リサーチプレビューのため、利用には Claude Console から事前申請が必要だ。

## 実装：最小限のAPIコード

Managed Agentsを使うには、通常のMessages APIリクエストにBetaヘッダを1行追加するだけだ。セッションを起動して、ツール呼び出しを非同期でポーリングする流れを示す。

```
import anthropic, time

client = anthropic.Anthropic()

# 1. セッション作成（自社サンドボックスを使う場合は sandbox_config を追加）
session = client.beta.managed_agents.sessions.create(
    model="claude-opus-4-8-20260301",
    system="あなたは社内コードレビューエージェントです。",
    betas=["managed-agents-2026-04-01"],
    # sandbox_config={"provider": "cloudflare", ...}  # 自社サンドボックスを使う場合
)
session_id = session.id

# 2. タスクを投入
task = client.beta.managed_agents.sessions.submit(
    session_id=session_id,
    messages=[{"role": "user", "content": "src/ 以下のPR差分をレビューしてください。"}],
    betas=["managed-agents-2026-04-01"],
)

# 3. 完了までポーリング
while task.status not in ("completed", "failed", "cancelled"):
    time.sleep(2)
    task = client.beta.managed_agents.sessions.get(
        session_id=session_id,
        task_id=task.id,
        betas=["managed-agents-2026-04-01"],
    )

# 4. 結果を取得
if task.status == "completed":
    print(task.result)

# 5. セッションを明示的に削除（履歴を残さない運用）
client.beta.managed_agents.sessions.delete(
    session_id=session_id,
    betas=["managed-agents-2026-04-01"],
)
```

`betas=["managed-agents-2026-04-01"]` がBetaヘッダの実体だ。SDK経由では配列で渡し、HTTPで直接叩く場合は `anthropic-beta: managed-agents-2026-04-01` ヘッダを付与する。

自社サンドボックスを使う場合は `sandbox_config` を追加する（プロバイダのドキュメントに設定値が記載されている）。MCPトンネルを使う場合は `mcp_servers` にトンネルのエンドポイントを渡す形になる（リサーチプレビュー段階のため仕様は変動しうる）。

## 失敗パターン

**「ツール実行を自社に移せばコンプライアンスが完全にクリアできる」と考える**  
セッション履歴・状態はAnthropicのサーバサイドに残る。ZDR・HIPAA BAAが必要な場合は Messages API で自前のエージェントループを実装する構成が必要だ。

**MCPトンネルを汎用HTTPプロキシとして使おうとする**  
MCPサーバへの接続専用だ。社内DBを直接叩きたい場合は、DBの前にMCPサーバを立てる構成にする。

**全部自社サンドボックスに移そうとする**  
エージェントループは常にAnthropicのインフラに留まる。「クラウドに出せないデータのツール実行だけを自社に置く」のが設計の基本で、全移行しようとするとインフラ管理コストが跳ね上がる。
