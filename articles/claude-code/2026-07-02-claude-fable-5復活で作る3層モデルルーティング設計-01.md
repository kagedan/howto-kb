# Claude Fable 5復活で作る3層モデルルーティング設計

「Fable 5が戻った」で読むと、ただの復旧ニュースに見える。実際に面白いのは逆で、復旧したからこそ、強いモデルを本番に出すときの境界線がかなり生々しく見えるようになった。

Anthropicは6月9日に Claude Fable 5 と Mythos 5 を公開し、6月12日に米政府の輸出管理指令を受けて両モデルを全ユーザー向けに止めた。外国籍ユーザーだけを即時に判定する信頼できる仕組みがなく、全停止するしかなかったためだ。6月30日に輸出管理が解除され、7月1日から Fable 5 はグローバルに戻った。出典は公式だけで足りる。

この記事で見たいのは、性能比較ではない。Fable 5の再提供版が「どこを締め直して戻ってきたか」だ。

## 1. 安全策はフィルタではなく、余白を持った境界線になった

Fable 5は Mythos 5 と同じ基盤モデルで、一般向けに強いセーフガードをかけた版として出ている。classifierがサイバーセキュリティ、生物・化学、蒸留のような領域を検知したとき、Fable 5自身ではなく Claude Opus 4.8 に処理を渡す。

今回の差分は、6月12日の停止につながった報告を受けて、その挙動を締め直した点にある。Amazon研究者の報告に含まれていた回避パターンを対象に、新しい安全classifierを訓練した、とAnthropicは説明している。手口には触れないが、脆弱性の特定と一部デモ生成が問題視された。

対象パターンは「99%超」でブロックするようになった。ただし、通常のコーディングやデバッグでも無害なリクエストをより多く拾ってしまう、とも明記している。ここが実装目線ではかなり大事なんだよね。安全classifierは「危険なら止める」部品ではなく、「たぶん安全だけど少し怪しい」領域まで広めに取る境界線になっている。

## 2. 「別モデルへ逃がす」は成功ではなく、観測すべき状態

今日、この記事を書く前に手元でイベント名を書き出していて、地味に引っかかったのがここだった。fallbackを「応答が返ったから成功」と扱うと、品質の変化をログで拾えない。

Fable 5では、classifierが反応したときにユーザーへ通知し、リクエストは Opus 4.8 に送られる。完全拒否より体験はましだ。でも開発側のログでは、これは成功ではなく「経路変更」として別に数えるべきだと思う。

最低限、`requestedModel`、`servedModel`、`routeReason`、`taskType` は残したい。この粒度でも、見るべきものが変わる。精度の平均ではなく、taskTypeごとの fallback率を見る。普通の debugging で safety\_classifier に寄り始めたら、プロンプトの書き方か、タスク分解を見直す。

やってはいけないのは、Fable 5とOpus 4.8の結果を同じeval行に混ぜること。結果が通っても、通したモデルが違えば意味が違う。長いコード修正では、途中の1ステップだけ別モデルに逃げても最終出力はそれっぽく整う。

## 3. 可用性はAPIの稼働率だけでは測れなくなった

もう一つ大きいのは、停止の理由が普通の障害ではなかったこと。6月12日の声明によると、政府指令は17時21分ETに届き、外国籍ユーザーのアクセス停止が求められた。国籍を即時検証できず、全顧客からFable 5とMythos 5を外した。

アプリ側から見ると、嫌な種類の停止だ。ステータスページが緑でも、契約や地域やユーザー属性でモデルが急に使えなくなる。

7月1日の再提供でも、Pro、Max、Team、一部Enterpriseでは7月7日まで週次上限の最大50%まで含まれ、その後は usage credits になる。つまり「Fable 5が使えるか」は、モデル名だけでは決まらない。

自分なら、本番のモデル選定を3層に分ける。入口では plan、region、surface を見る。実行中は requestedModel、servedModel、routeReason を残す。出口では evalScore と retryCount を見る。

入口で落ちたものと、実行中に逃げたものと、出口で品質が落ちたものを混ぜると、対策を間違える。全部「Fable 5が不安定」でまとめると、何も直せない。

## 復活後に見るべき指標

Fable 5が戻ったことで、強いモデルを使える場面は増える。ただ、今回の復活版は「強いモデルがそのまま戻った」というより、「制御面が一段前に出てきた」と見た方が実装には効く。

最低限、アプリ側では3つを別々に見たい。普通のcodingやdebuggingで Opus 4.8 へ流れる割合。fallback後の品質差。API、Claude Code、Claude.ai、Enterpriseごとの availability差。

この3つを分けると、Fable 5復活の見え方が少し変わる。復活はゴールではなく、モデル本体、classifier、提供条件をまとめて運用するフェーズの始まりだと思う。これから高性能モデルを本番に入れるなら、モデル名を設定ファイルに書くだけでは足りない。どの層で別経路に入ったのかをログに残して、evalもその経路ごとに切る。そこまでやって初めて、強いモデルを「使っている」のではなく「運用している」と言えるはず。
