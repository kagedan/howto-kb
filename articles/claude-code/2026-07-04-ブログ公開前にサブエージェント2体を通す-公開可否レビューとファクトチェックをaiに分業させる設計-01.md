AIで記事を量産できるようになると、次のボトルネックは執筆ではなく**公開前チェック**に移ります。「これ、出していい情報か？」「この仕様の記述、合ってるか？」を毎回人力で全文チェックするのは重い。そこで私は、Claude Codeのサブエージェント2体を「コンテンツのCI」としてドラフトと公開の間に挟んでいます。運用して分かった設計のコツを書きます。

## なぜ2体に分けるのか

最初は「公開前チェックエージェント」1体で全部見させようとしましたが、分けたほうが明確に精度が上がりました。理由は2つあります。

**1. 関心の分離。** 「出してはいけない情報の検出」と「事実の正誤判定」は、必要な思考がまったく違います。前者は疑い深く網羅的に読む仕事、後者は主張を抽出して一次情報と突き合わせる仕事。1体に混ぜると、どちらも中途半端になります。

**2. ツール権限を役割ごとに最小化できる。** サブエージェントは定義ファイルで使えるツールを制限できます。

```
---
name: publish-safety-review
description: ブログ記事ドラフトの公開可否レビュー。個人の特定につながる情報・内部数字・認証情報などを洗い出す。ドラフト完成後、ユーザーレビューの前に必ず使う。
tools: Read, Grep, Glob        # ← ローカル読み取りのみ。Web接続なし
---
```

```
---
name: ai-fact-check
description: 記事内のAI関連の事実記述（モデル名・価格・機能・仕様）を公式ドキュメントと照合して正誤判定する。
tools: Read, Grep, Glob, WebFetch, WebSearch   # ← こちらは公式docの照合にWebが必要
---
```

公開可否レビューは社外秘を扱う可能性があるのでWebアクセスを持たせない。ファクトチェックは公式ドキュメントへのアクセスが本質なのでWebFetch/WebSearchを持たせる。**役割ごとに権限を絞る**のは、人間の組織設計と同じです。

## 設計のコツ1: 判定基準を「観点リスト」で固定する

「問題がないかチェックして」では毎回違う観点で見てしまいます。公開可否レビューには5観点を明示しています。

1. 個人の特定（実名・ハンドルネーム・職業+地域+エピソードの組み合わせ）
2. 内部の数字（売上・成約率など未公表の事業数字）
3. 認証情報・内部URL（APIキー・トークン・社内ツールのURL）
4. 第三者の権利・信頼（無断引用・根拠のない批判）
5. 未公開の予定（リリース前の価格・キャンペーン）

ネタ元がDiscordの走り書き（社外秘混在が前提）なので、この観点リストが実質的な「公開ポリシーの実行形」になっています。

## 設計のコツ2: 出力フォーマットを契約にする

最終メッセージの形式を定義ファイル側で固定します。

```
判定: 公開OK / 要修正
指摘:
- [深刻度: 高/中/低] 該当箇所（行番号と引用）→ 問題の理由 → 修正案
```

さらに「問題ゼロでも、なぜ問題ないと判断したかを観点ごとに1行で示す」を義務化しています。これがないと「問題ありません」の一言で仕事をした気になるからです。空振りの説明責任まで含めて、はじめてレビューとして信頼できます。

## 設計のコツ3: 迷ったら安全側に倒す指示を明文化する

公開可否レビューには「**迷ったら要修正に倒す**（公開後に消しても魚拓・検索キャッシュに残るため）」、ファクトチェックには「**公式情報が見つからない主張は『未確認』とする（誤りと断定しない）**」「**ブログ・まとめ記事は根拠にしない。一次情報のみ**」と書いています。デフォルトの倒し方を決めておかないと、AIは無難に「OK」寄りの判定をしがちです。

## それでも最後は人間

この2体はあくまで一次フィルタで、公開の最終判断は人間がします。実際、ファクトチェックが公式ドキュメントの表示価格を根拠に「誤り」と指摘してきた記述が、実は税込価格で正しかった、というケースがありました。AIレビューは網羅性で人間を超えますが、**文脈の正しさの最終責任は人間側に残す**。この線引きがある限り、量産と品質は両立できます。

## まとめ

* 公開前チェックは「安全性」と「事実性」で分業させ、ツール権限も役割ごとに最小化する
* 観点リスト・出力フォーマット・迷ったときの倒し方を定義ファイルに明文化する
* AIレビューは一次フィルタ。最終判断は人間に残す
