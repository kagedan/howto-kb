* AI 応答に「完遂判定」を組み込む DSL プロトコル **PPPC executor** を、v3.0 → v3.1 → v3.1.1 → v3.2 と4段階で育てた。
* 各段階で **別種の問題** が出た: 判定ルートの欠落(Critical)、論理反転(Bug)、役割分担の曖昧さ(Update)。
* 特に **BUG-01(yes/no 論理反転)** は、人間が読むと違和感なく通るが、LLM に食わせると事故るタイプ。同種のバグを踏んでいる仕様は多い。
* 「仕様書を育てる」作業は、レビュー → バグチェック → Update反映 の3フェーズで粒度が違うタスクに分かれる。混ぜると落ちる。

## 前提: PPPC executor とは

PPPC (Promise Purpose Process Complete Protocol) は、AI 応答の終わり方を厳密化するためのプロトコルです。

通常の AI 応答は「処理が終わったら終わる」になりがちですが、PPPC はこれを拒否します。**目的に対する必要十分な到達が確認された状態** を完遂と定義し、以下の5条件を全て満たした場合のみ完遂と判定します。

```
completion_condition:
  - Purpose_Achieved      # 目的達成
  - MustFixLine_Met       # Must-Fix Line 達成
  - User_Ready_Next       # 次行動可能
  - Handover_Ready        # 引き継ぎ可能
  - Critical_Remaining = 0  # 致命点ゼロ
```

この5条件に加えて、残課題を `Deferred / Blocked / Rejected` に分類して引き継ぎ、次ターンの起動文まで出力する—という構造を YAML ライクな DSL で書いたのが PPPC executor です。

v3.0 はおおむね機能していました。ただ、仕様として自立して動くかを検証したところ、予想外の穴が次々に出てきました。

## v3.0 → v3.1: 仕様レビューで出た4つの Critical

v3.0 を通しで読んで、**仕様本体の判定ルートが機能するか** という観点で4つの Critical を検出しました。

### C1: `must_fix_line` が `completion_gate` の判定に接続されていない

`priority_order` では `must_fix_line` が `critical_check` より上位に置かれていました。しかし、実際の判定を行う `completion_gate.checks` は4項目のみで、Must-Fix Line の達成確認がゲート条件に入っていませんでした。

> 優先度上位のものが判定式に出てこない

これは典型的な「宣言と実装の乖離」です。優先度を宣言するだけでは動かない。判定フィールドに組み込む必要がある。

**修正:** `completion_gate.checks` に `MustFixLine_Met` を追加し、`note` で `Critical_Remaining=0` だけでは代替できない旨(Must-Fix は目的起点、Critical は品質起点で粒度が異なる)を明記。

### C2: `PPPC-NODE.line` フィールドが未定義

引き継ぎコード `PPPC-NODE` の末尾フィールド `line` が、どこにも定義されていませんでした。Must-Fix Line なのか、Final Improvement Line なのか、両方なのか不明。

**修正:** `field_definitions` セクションを新設し、`line: 今ターンで採用した完遂ラインの記述。形式は "must_fix=○○ / final_improvement=○○" を推奨` と定義。

### C3: `turn_execution_flow` に Must-Fix Line 設定ステップが無い

step\_1 「Purpose Lock」の action が「今回達成したいこと・十分条件・暫定目的を固定する」のみで、Must-Fix Line の設定タイミングが抜けていました。出力テンプレートには Must-Fix Line が必須なのに、実行フロー側で設定するポイントが定義されていない。

**修正:** step\_1 の action に Must-Fix Line 固定を追加し、`Must-Fix Line が step_1 で確定していない場合は step_2 に進まない` という note を付与。

### C4: `output_rule` と `default_response_structure` の食い違い

短縮ルール(`output_rule`)と詳細構造(`default_response_structure`)で、Must-Fix Line と Completion Certificate が前者にだけ欠落していました。

LLM は短い指示を優先参照しがちなので、短い方で必須項目が落ちるリスクがあります。

**修正:** `output_rule` を拡張し、`output_rule_note` で「どちらかを更新した場合は他方も更新すること」という同期ルールを追加。

## v3.1 → v3.1.1: バグチェックで出た7つのバグ

v3.1 を出した後、もう一度 **逐行で論理的整合性** をチェックしました。ここで新種の問題が7件出ました。最も致命的だったのが BUG-01 です。

### BUG-01: `critical_check.decision_rule` の yes/no 方向が逆転

これが一番ヤバかったバグです。

```
/critical_check
mandatory_questions:
  - 目的を取り違えていないか
  - 実行不能な手順が含まれていないか
  - 必須条件が欠落していないか
  - 誤解を生む曖昧さが残っていないか
  - 次ターンの品質を下げる欠陥が残っていないか
decision_rule: 1つでも yes があれば complete=false  # ← ここ
```

質問文はすべて「〜していないか」形式です。つまり:

* **yes** = 「取り違えていない」= **健全**
* **no** = 「取り違えている」= **問題あり**

しかし `decision_rule` は「1つでも yes があれば complete=false」となっていました。

> **健全だと答えたのに未完判定が出る**

人間が読むと「まあそういう意味でしょ」と脳内補正されて通ってしまう。しかし LLM に文字通り実装させると、健全な応答が毎回「未完」になります。

**修正:**

```
answer_semantics: yes = 健全(問題なし) / no = 問題あり(修正必要)
decision_rule: 1つでも no があれば complete=false
```

`answer_semantics` を追加して yes/no の意味を明示した上で、判定ルールを反転。これで文字通り実装しても事故らない状態になりました。

### BUG-02〜04: 上位定義が v3.1 の変更を反映していない

* `core_definition.completion_condition` が「4条件」のまま(v3.1 で5条件化したのに)
* `turn_execution_flow.step_4` の action が5項目全判定になっていない
* `strict_judgement.rule_3` に Must-Fix 達成と Critical Remaining=0 が抜けている

いずれも **「下位を変更したら上位もチェックする」** の徹底不足。差分を当てた後の整合性確認が甘かった。

### BUG-05〜07: 未定義と暗黙前提

* `priority_order` が何の優先度か未定義(実行順? 衝突時? 判定?) → 「衝突時の判定順」と明示
* `purpose_lock.rule_3` と `meeting_turn_behavior` の記述が部分一致で情報欠損 → 参照に一元化
* `final_improvement_line.set_condition` に Must-Fix 達成が未記載 → 暗黙だが明示化

## v3.1.1 → v3.2: Update 群の反映と役割分担の明示

v3.1.1 で論理バグが全て消えた時点で、Update Queue に溜まっていた 11項目(U1〜U8, E1〜E3)の反映に移りました。このフェーズは「動くけど読みにくい / 誤用しやすい」を解消するのが主目的です。

### 特に効いた変更: `role:` の全判定ルート付与

`critical_check` / `completion_gate` / `strict_judgement` / `handover_protocol` / `completion_certificate` に `role:` フィールドを追加しました。

```
/completion_gate
role: step_4 で使う完遂判定フィールド定義。判定時の補助ルールは /strict_judgement を参照

/strict_judgement
role: completion_gate の判定時に使う補助ルール集。フィールド定義は /completion_gate を参照
```

判定系が3つ(`critical_check` / `completion_gate` / `strict_judgement`)あり、役割が分かれているのに並列に置かれていたのが v3.1.1 までの課題でした。role を付与することで、**どこで何を参照するか** が自明になりました。

### 新設した3セクション

1. **`/absolute_rules`** — 絶対に破れないルールを集約(AR\_01〜06)。元は `core_definition.absolute_rule` に1件だけあったのを昇格。
2. **`/critical_fix_definition.escalation`** — critical がその場で解消できない場合のエスカレーションパスを定義(blocked\_issue への変換 + 解除条件必須)。v3.1.1 までは「critical は必ずその場で解消」としか書かれておらず、詰むケースの逃げ道がなかった。
3. **`/meeting_turn_behavior.completion_mapping`** — meeting\_turn(目的不明ターン)での completion\_gate.checks 読み替え。これがないと meeting\_turn 中に厳密判定が暗黙解釈になる。

### 細かいが重要: `naming_convention` の明文化

PPPC-NODE のフィールドは英語、template\_output の見出しは日本語、という使い分けが暗黙になっていました。`expression_layer.naming_convention` で以下のように明文化:

```
naming_convention:
  english_keys: 構造コード・機械処理用 (PPPC-NODE フィールド、completion_gate.checks 等)
  japanese_keys: ユーザー可読表示用 (template_output の見出し、各セクションの説明文等)
  rationale: 構造は他システム連携のため英語固定、表示は読み手負荷を下げるため日本語を優先
```

v3.2 を読んだ人が「なぜここは英語でここは日本語?」と迷わなくなります。

## 気づき: 仕様書を育てる3つの原則

4段階の進化を経て見えてきた原則を3つに整理します。

### 原則1: レビューとバグチェックは別タスク

* **レビュー** = 仕様の意図に対して判定ルートが機能するか(設計視点)
* **バグチェック** = 書かれた通り実装したときに事故が出ないか(実装視点)

v3.0 → v3.1 はレビュー、v3.1 → v3.1.1 はバグチェック。両者を混ぜると見える粒度が変わってしまい、どちらも中途半端になります。**一度に一つの視点** で通す方が精度が高い。

### 原則2: LLM 実装を前提にすると「人間の脳内補正」に頼れない

BUG-01 の yes/no 反転は、人間の読み手なら「そういう意味でしょ」で通ります。しかし LLM は書かれた通りに動くため、文字通りのロジックで事故が出ます。

**脳内補正に頼っている箇所を全部炙り出す** のが、LLM 向け DSL を書くときの本質的なデバッグ作業です。

### 原則3: `role:` と `note:` は無料の保険

各セクションに「このセクションは何のためにあるか(role)」と「運用上の注意(note)」を書いておくと、後から仕様を触る人(半年後の自分を含む)の誤用率が激減します。

コード量はほぼ増えないのに、解釈のブレが減る。**書いておいて損のない追加情報** は書く方が良い。

## まとめ

PPPC executor を v3.0 → v3.2 まで育てた過程で、3種類の別問題が出ました。

| 段階 | 問題の種類 | 件数 | 代表例 |
| --- | --- | --- | --- |
| v3.0 → v3.1 | Critical(判定ルート欠落) | 4 | Must-Fix Line が判定式に入ってない |
| v3.1 → v3.1.1 | Bug(論理反転・未定義) | 7 | yes/no 方向が逆 |
| v3.1.1 → v3.2 | Update(役割分担・読みやすさ) | 11 | judgement 系の役割分担が暗黙 |

特に BUG-01 は、同種のバグを踏んでいるプロンプトや仕様書が世の中にかなり多いはずです。**「〜していないか」という質問形式と、「yes なら NG」という判定ルール** の組み合わせを見たら、一度疑った方がいいです。
