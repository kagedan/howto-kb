コンテキストの分岐（セッションフォーク）
LLMの文脈管理にも「git branch」が使えます。巨大な設計書やAPI仕様を読み込ませたマスターセッションを作り、タスクごとにフォーク。毎回トークンと時間を消費して前提条件を読み込ませなくてもできます！

スキル（Skills）やProjectsは、
「このシステムの設計書はこれ」という初期設定を保存するもの。毎回新しいチャットを立ち上げる際、ゼロからではなく「知識を持った状態」からスタートできる。（＝ゲーム開始時から最強装備を持っている状態）

一方セッションフォークの場合

これはRPGのセーブデータ分岐と同じ。
「複雑なバグの原因を特定した直後」の状態でセッションをフォーク（分岐）し、アプローチAとBを並行して試すことができます。失敗したら前の分岐点に戻るだけ。トライ＆エラーの時間が劇的に減ります！

ちなみに「セッションフォーク」の具体的な活用例はこんな感じです👇

1️⃣ マスターセッションに「要件定義やAPI仕様書」を全部読ませる
2️⃣ フロントエンド実装用にフォーク（分岐）して作業
3️⃣ バックエンド実装用にマスターから再度フォーク

毎回長文プロンプトを読ませるトークン代と時間を節約できる上、AIが文脈を見失うのを防ぐ最強の小技です🔥
