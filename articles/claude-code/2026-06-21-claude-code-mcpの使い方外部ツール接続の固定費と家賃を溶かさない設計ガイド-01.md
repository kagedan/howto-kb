「MCPを入れると便利になるらしい」

——そう聞いて入れてみたものの、何が変わったのか実感がない。

あるいは、おすすめされた通りに5個も6個も入れたら、なんだか動きが重くなった気がするし、課金も増えた気がする。そんなモヤモヤを抱えていませんか。

実は、その不安はあなたのせいではありません。

多くの解説記事が  
「便利だから入れよう」

「入れたら何にいくらかかるのか」  
「どれを入れて、どれを入れないのか」

という肝心な判断基準を教えてくれないからです。

しかも2026年に入って、状況は大きく変わりました。  
「MCPは入れすぎるとコンテキスト（AIの作業机）を圧迫するから注意」という、これまでの常識が半分くずれたのです。  
にもかかわらず、その事実を踏まえていない古い記事がまだたくさん残っています。

この記事は、ただの設定手順書ではありません。

連載でずっと使ってきた「固定費」と「家賃」という考え方で、MCPを「不動産契約」のように捉え直します。

読み終える頃には、繋ぎすぎず・溶かさず、本当に効くMCPだけを選んで残せるようになっているはずです。エンジニアでない方も、置いていきません。

## ■ MCPとは？―「AIに“手と目”を与える共通の差し込み口」

まず一言で。MCP（エムシーピー／Model Context Protocol）は、AIと外部ツールをつなぐ“共通の差し込み口”です。

よく「USB-Cみたいなもの」と説明されます。スマホでも家電でも、USB-Cという同じ形の差し込み口があれば、ケーブル1本で繋がりますよね。あれと同じで、GitHubでもデータベースでもSlackでも、MCPという共通規格があれば、AIから同じやり方で繋げるのです。

なぜこれが大事か。

AIはとても賢い「頭脳」ですが、そのままではあなたのファイルも、社内のツールも、ネットの最新情報も見られません。  
頭はいいのに、目も手もない状態です。MCPは、その頭脳に「目（情報を読む）」と「手（操作する）」を後付けする仕組みだと思ってください。

▼ MCPの「仕組み」や「なぜ生まれたのか」「ビジネスにどう効くのか」をもっと根本から知りたい方は、別記事**「知らないと損するMCP——AI×外部ツール接続が変えるビジネスの未来」**で、非エンジニア向けにじっくり解説しています。  
この記事では概念は深追いせず、Claude Codeで実際に使うところに振り切ります。

### ◇ この連載の地図：「常駐・強制・分業・協働」、その先の【接続】

この記事はClaude Code連載の第6弾です。これまでの5本を一枚の地図にすると、こうなります。

・第5弾

・第6弾

ここまでは「Claudeをどう動かすか（内側の話）」でした。今回はついに、Claudeを“外の世界”に繋ぐ（外側の話）に進みます。そして後で詳しく書きますが、この「接続」にも、連載でおなじみの**固定費と家賃が発生**します。

## ■ まず動かす：Claude CodeにMCPを追加する3ステップ

理屈より先に、繋がる体験をしましょう。難しい操作はありません。コマンドを1行打つだけです。

### ◇ ステップ1：「claude mcp add」で繋ぐ（最短ルート）

Claude Codeを起動した状態で、ターミナル（コマンドを打つ画面）に次のような形で打ちます。これは公式が推奨しているやり方で、設定ファイルを手書きするより安全です。

　claude mcp add ＜名前＞ -- ＜起動コマンド＞

＜名前＞は自分が分かれば何でもOK、＜起動コマンド＞は各MCPの公式ページに必ず書いてあるので、それをコピペするだけです。ゼロから自分で書く必要はありません。

### ◇ ステップ2：スコープを選ぶ（どこまでの範囲で有効にするか）

ここが初心者がいちばん事故るポイントなので、丁寧にいきます。  
MCPを追加するとき、「どこまでの範囲で有効にするか（スコープ）」を選べます。日本語にすると「置き場所」です。

・local … 今の自分・このプロジェクトだけ（自分の引き出し）  
・project … このプロジェクトの「全員」に共有される（チームの共有棚）  
・user … 自分の「全プロジェクト」で有効（自分専用の道具箱）

何が怖いかというと、「project」にすると、その設定がチーム全員に配られることです。自分だけの実験のつもりが、知らないうちにメンバー全員の環境に影響する、という事故が起きます。

👉 迷ったら、まず「local」か「user」。チームに配るのは「これは全員に要る」と確信できてからで十分です。

### ◇ ステップ3：繋がったか確認する

Claude Codeの中で「/mcp」と打つと、今つながっているMCPの一覧と状態が見られます。ここに名前が出て、緑のサイン（接続済み）になっていれば成功です。出てこない・赤い場合は、後半の「繋がらないときの対処」を見てください。

たった3ステップ。繋ぐだけなら、本当にこれで終わりです。問題はこの先、「どれを繋ぐか」「繋いだら何にいくらかかるか」です。

## ■ 最初に入れるべき定番MCPと、「その家賃」

ネットで「Claude Code MCP おすすめ」と調べると、判で押したように同じ顔ぶれが並びます。代表的なものを、非エンジニアにも分かる言葉で整理します。

・GitHub … プログラムの保管庫を読み書きする（開発・共同作業をする人向け）

・Context7 … 部品（ライブラリ）の最新の使い方を取ってくる（古い情報でAIが間違えるのを防ぎたい人向け）

・Serena … 大きなコードを賢く検索・把握する（大規模プロジェクトを扱う人向け）

・Playwright … ブラウザを自動で操作する（画面テストや操作の自動化をしたい人向け）

これらは確かに強力です。でも、ここで連載の出番です。「定番だから全部入れる」は、いちばんやってはいけない判断です。

なぜなら——

### ◇ 「定番◯選」を鵜呑みにしてはいけない

考えてみてください。

ブラウザ操作のMCP（Playwright）は、ブラウザを操作しない人には一度も出番がありません。それでも入れておくと、後述する「家賃」だけが毎回かかります。

使わないMCPは、便利な道具ではなく、ただの負債です。

判断はシンプルです。「自分の作業の中で、それが実際に火を噴く場面があるか？」——イエスなら入れる、ピンとこないなら入れない。それだけです。

## ■【最重要・独自】MCPの「固定費」と「家賃」——2026年に常識が変わった話

**ここがこの記事の核心であり、他のどの記事にも書いていない部分です。**連載で育ててきた考え方を、MCPに当てはめます。

### ◇ かつての常識：「MCPは入れすぎるとコンテキストを圧迫する」

少し前まで、MCPには明確な弱点がありました。MCPを繋ぐと、「このMCPはこういう道具を持っていますよ」という説明書きが、AIの“作業机（コンテキスト）”の上に常に広げられたのです。

連載第2弾で、CLAUDE.mdを「毎回読まれる固定費」と呼んだのを覚えているでしょうか。

MCPも同じでした。繋いだ瞬間から、使っても使わなくても、説明書きが机のスペースを占領する。だから「MCPは入れすぎ注意」が当時の常識だったのです。

### ◇ ところが——Tool Searchで常識が半分くずれた

2026年1月、Tool Search（ツールサーチ）という仕組みが入りました。これが効きます。

仕組みはこうです。  
MCPの説明書きが机の1割以上を占めそうになると、Claudeは自動で「全部広げっぱなし」をやめ、「必要になったときだけ、その道具の説明を取りに行く」方式に切り替えるのです。

たとえるなら、机の上に全部の道具を出しっぱなしにするのをやめて、使うときだけ棚から取り出す“優秀な棚卸し係”を雇ったようなもの。  
おかげで、MCPをいくつ繋いでも、机が説明書きで埋まる心配がかなり減りました。

これは大きな変化です。「入れすぎ注意」という古い前提だけで書かれた記事は、もうアップデートが必要なのです。

### ◇ それでも残る「家賃」——タダにはならない

では入れ放題か、というとそうではありません。

ここを勘違いすると、別のところでコストを払うことになります。Tool Searchが消してくれたのは「机のスペース（コンテキスト）」だけ。  
次の3つの“家賃”は、依然として毎月かかり続けます。

1. 認証・接続の手間賃：MCPを繋ぐにはログイン（認証）が要るものが多く、期限が切れれば繋ぎ直しが発生します。実は、いちばん地味に重いのがこの「メンテの時間」です。
2. セキュリティの家賃：MCPを1つ繋ぐ＝あなたの環境に“鍵”を1本渡すこと。鍵が増えるほど、悪用されたときの被害範囲も広がります（後述）。
3. 判断のノイズ：道具が多いほど、AIは「どれを使うか」の判断を迫られます。指示が曖昧だと、道具を取り違える確率が上がります。

つまり結論はこうです。「Tool Searchのおかげで“机代”はほぼ無料になった。でも“鍵代”と“メンテ代”は残る」。だから、入れる前に「この家賃を払ってでも欲しい接続か？」と一呼吸おくのが、溶かさないコツです。

### ◇ alwaysLoadの罠——便利さと引き換えに固定費へ逆戻り

2026年4月に「alwaysLoad（オールウェイズロード＝常に読み込む）」という設定が増えました。「このMCPだけは毎回すぐ使えるように、最初から机に出しておく」という指定です。

便利そうに見えますが、これを多用すると、せっかくのTool Searchを自分でオフにして、昔の「固定費まみれ」状態に戻してしまいます。本当に毎回・即・使うMCP（1〜2個まで）に絞るのが鉄則です。  
「念のため全部alwaysLoad」は、自分で机を埋める自爆行為だと覚えてください。

## ■【独自】4レイヤーの仕分け——MCPはどこに効かせるか

連載を読んできた方への、ご褒美のような一節です。これまでの登場人物を役割地図にすると、迷いが消えます。

・[CLAUDE.md](http://CLAUDE.md) →「常駐」毎回読ませる前提・ルール  
・hooks →「強制」決めた操作を機械的に必ず実行  
・サブエージェント →「分業」別部屋に作業を出して結論だけ受け取る  
・MCP →「接続」外部ツールに手を伸ばす

この4つは競合しません。組み合わせるものです。特に相性がいいのが——

### ◇ サブエージェントに「専門の道具だけ」を渡す

連載第4弾のサブエージェント（別部屋の専門家）に、MCPという“専門の道具”をセットで持たせると強力です。  
たとえば「ブラウザ調査担当」の部屋にだけブラウザ操作MCPを渡し、他の部屋には渡さない。

こうすると、それぞれの担当が必要な道具だけを持ち、余計な鍵を全員に配らずに済みます。「専門の部屋に、専門の道具だけを置く」——整理された仕事場のイメージです。エンジニアでなくても、この“役割と道具を一致させる”発想は、そのまま日常の仕事の整理術にも通じます。

## ■ 繋がらない・認証で詰まるときの対処

「3ステップでやったのに繋がらない」  
——大丈夫です。原因はだいたい決まっています。

### ◇ つまずきの8割は「スコープ（置き場所）」

「昨日は動いたのに今日は出てこない」  
「あの人の環境では動くのに自分では動かない」

——こういうとき、機能の不具合ではなく、スコープ（local／project／user）の取り違えがほぼ犯人です。  
「/mcp」で一覧を確認し、自分が今いるプロジェクトに対して、正しい置き場所で追加されているかを見直してください。

### ◇ OAuthで「繋がるが、権限が広すぎる」問題

ログイン連携（OAuth＝オーオース、サービス同士を安全に繋ぐ仕組み）で繋ぐMCPでは、新しい注意点があります。  
「繋がること」と「安全なこと」は別だということ。

繋ぐとき、たいてい「これらの操作を許可しますか？」と権限を聞かれます。ここで何でもかんでも許可すると、必要以上に広い鍵を渡してしまう。「読むだけでいいのに、書き換え・削除まで許可していた」というのはよくある事故です。許可は、最小限に。  
これは非エンジニアこそ気をつけたいポイントです。

## ■【独自】安全に使うための注意点——鍵を増やす前に

MCPの一番のリスクは、「便利だから」と知らない人が作ったMCPを気軽に繋いでしまうことです。繰り返しますが、MCPを繋ぐ＝自分の環境に鍵を渡すこと。入れる前に、最低限これだけは確認しましょう。

【第三者製MCPの審査チェックリスト】  
✅ 作者・提供元は明確か（公式／信頼できる組織か）  
✅ どんな権限を要求しているか（読むだけ？書き換える？削除する？）  
✅ 利用者が多く、評判が確認できるか  
✅ 自分のこの作業に、本当に必要か（「念のため」は入れない）  
✅ 後で消せるか（棚卸しできる前提で入れる）

### ◇ Windowsユーザーの落とし穴

連載で一貫してお伝えしている通り、Windowsは少し作法が違う場面があります。MCPの起動コマンドの書き方や、パス（ファイルの住所）の指定でつまずきやすいので、各MCPの公式ページで「Windowsの場合」の記述を必ず確認してください。Mac前提のコマンドをそのまま打って動かないは、Windowsあるあるです。

## ■ 入れた後の話：MCPの「棚卸し」と“消し方”

世の中の記事は「入れ方」ばかりで、「消し方」を書きません。でも本当に上手な人は、出口（消すこと）から逆算して入れます。

月に一度でいいので、「/mcp」で一覧を眺めて、「最近一度も使っていないMCP」は外す。使わないMCPは、机代こそTool Searchが消してくれても、鍵代（セキュリティ）とメンテ代（認証更新）は払い続けているのでしたよね。

使わない接続は、静かにあなたのコストを食う負債です。増やしたら、定期的に減らす。これだけで運用がぐっと軽くなります。

## ■ MCPの実務・副業・市場へのインパクト

最後に、「で、自分にどう効くのか」を持ち帰っていただきます。

実務への影響：MCPによって、AIは「賢い相談相手」から「実際に手を動かす担当者」へ変わります。資料を読みに行く、データを取ってくる、画面を操作する——これまで人がやっていた“つなぎの作業”を肩代わりさせられます。

非エンジニア・副業への影響：ここが朗報です。  
MCPはエンジニア専用ではありません。むしろ、ファイル整理・情報収集・定型レポート作成といった非エンジニアの日常業務こそ、相性のいい用途があります。「自分は技術者じゃないから関係ない」と思っていた人が、最初に体感価値を得やすい領域です。

市場への影響と将来性：MCPはもはやClaude（Anthropic）だけの話ではありません。OpenAIやGoogle、Microsoftも採用する“業界共通規格”になり、2025年末には中立的な団体へ寄贈されました。USB-Cが全機種共通になったように、「AIに外部ツールを繋ぐ作法」はMCPで一本化されつつあるのです。

将来を少しだけ。

3年後には、MCPは「自分で設定するもの」から「最初から入っている前提」へ。

5年後には、どの接続を許すかの“目利き”が、CLAUDE.md設計と並ぶ生産性スキルとして評価されるでしょう。

10年後には、人間が選ぶのではなく、AIが必要な接続を自分で契約・解約する——MCPの設定は完全に裏側へ消えていくはずです。

だからこそ今、**「繋ぐ・選ぶ・外す」の感覚を手に入れておく**ことが、いちばんの先行投資になります。

## ■ よくある質問（FAQ）

Q. MCPは無料で使えますか？  
A. MCPの仕組み自体に追加課金はありません（Claude Codeの全プランで利用可）。ただし、繋いだ先のサービス側に料金がかかる場合や、使用量に応じたトークン消費は別です。

Q. MCPは結局どこに設定すればいいですか？  
A. 迷ったら、まず自分だけの範囲（localまたはuser）。チーム全員に配られるprojectは、「全員に必要」と確信してからにしましょう。

Q. MCPを入れすぎるとどうなりますか？  
A. 2026年のTool Searchで「机（コンテキスト）を圧迫する」問題はかなり解消されました。ただし、認証のメンテ・セキュリティ・判断ノイズという“家賃”は残るので、使わないものは外すのが正解です。

Q. Claude CodeでMCPが繋がりません。  
A. 8割はスコープ（置き場所）の取り違えです。「/mcp」で状態を確認し、正しい範囲で追加されているか見直してください。認証切れ・Windows固有のコマンド差も定番の原因です。

Q. MCPとプラグイン（Plugins）の違いは？  
A. MCPは「接続」の一要素。プラグインは、MCP・スキル・コマンド・hooksなどをまとめて束ねて配る入れ物です。連載の最終回で詳しく扱います。
