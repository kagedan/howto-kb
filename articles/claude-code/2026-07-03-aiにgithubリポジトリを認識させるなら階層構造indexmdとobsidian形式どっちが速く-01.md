## はじめに

「ドキュメントやナレッジをAIに読ませたい」というニーズが一気に増えました。Claude Code のようなエージェントにリポジトリを丸ごと渡す、RAG の知識ベースとして食わせる、社内 Wiki をそのまま参照させる——用途はさまざまですが、そこで必ず出てくる問いがあります。

> ディレクトリがきれいに整理されていて `index.md` もあるリポジトリと、Obsidian のようにタグ付けされたフラットな Markdown 群。**AIにとって、どっちが「速く・安く・正確に」contextを認識できるのか？**

結論から先に書きます。

* **AIエージェントに「探索」させるなら → 階層構造＋index.md が有利**（速い・安い・迷いにくい）
* **RAG（ベクトル検索）で「引っ張る」なら → タグ（メタデータ）は強力な武器になる**
* **ベストは両取り**：ディレクトリで構造を、frontmatterのタグでメタデータを持たせる

この記事では、なぜそうなるのかを「AIがどうやってリポジトリを読むか」という仕組みから解説し、実例と比較表、そして今日から使える設計指針までまとめます。

---

## そもそもAIは2通りの「読み方」をする

比較の前提として、AIがリポジトリを認識する方法は大きく2つに分かれます。ここを混同すると議論がかみ合いません。

### 1. エージェント型（探索して読む）

Claude Code、Cursor、Devin のようなコーディングエージェントは、人間のエンジニアと同じように**ファイルツリーを歩きます**。

```
ls / glob でファイル一覧を見る
  ↓
ディレクトリ名・ファイル名から「どこを見るべきか」を推測
  ↓
必要なファイルだけ read する
  ↓
足りなければ grep で検索
```

このとき消費されるのは「読んだファイルのトークン」です。つまり **無駄なファイルを開かせないほど安く・速くなる**。

### 2. RAG型（事前にEmbeddingして引く）

ドキュメントをチャンクに分割し、Embedding してベクトルDBに入れておき、質問時に類似チャンクを引っ張る方式です。

```
全ドキュメントをチャンク分割 → Embedding → Vector DB
  ↓
質問をEmbedding → 類似検索 → ヒットしたチャンクだけLLMに渡す
```

こちらでは、ディレクトリ構造そのものよりも**チャンクの意味的なまとまり**と**メタデータ（タグ）でのフィルタリング**が効いてきます。

この「2通り」を頭に置くと、どちらの形式が有利かは用途で変わることが見えてきます。

## 「階層構造＋index.md」形式の強み

Next.js や Kubernetes の公式ドキュメント、あるいは Diátaxis に沿った docs ディレクトリを思い浮かべてください。

```
docs/
├── index.md              # 全体の地図（目次＋概要）
├── getting-started/
│   ├── index.md          # このセクションの概要
│   ├── installation.md
│   └── quickstart.md
├── guides/
│   ├── index.md
│   ├── authentication.md
│   └── deployment.md
└── reference/
    ├── index.md
    └── api.md
```

### なぜエージェントに強いのか

* **ディレクトリ名が最強のメタデータ**：`guides/authentication.md` は開く前から中身が推測できる。エージェントは「無駄なファイルを開かない」判断ができる。
* **`index.md` が地図になる**：トップの `index.md` を1枚読むだけで全体像が掴める。これは**数百トークンで全体マップを得る**ということで、コストが劇的に安い。
* **見出し階層がそのままcontextになる**：`##`／`###` の構造は、RAGでチャンク分割したときに「パンくず（このチャンクはどの章の話か）」として付与しやすい。

### コストのイメージ

100ファイルのリポジトリを探索させる場合:

* **index.md あり**：`index.md`（〜500トークン）→ 目的の2〜3ファイルだけ read。合計 **数千トークン**。
* **index.md なし・フラット**：まず全ファイルを `ls`、当たりをつけるため関係ありそうなファイルを片っ端から read。**数万トークン**に膨れることも。

エージェント運用では、この差が**そのまま料金と待ち時間**になります。

## 「Obsidian形式（タグ＋リンク）」の強みと弱み

Obsidian や Foam のスタイルは、フラットに並んだ `.md` に **frontmatterのタグ** と **`[[wikilink]]`** で意味的なつながりを持たせます。

```
---
tags: [auth, security, backend]
aliases: ["認証", "ログイン"]
---

# 認証設計メモ

詳しくは [[JWT実装ガイド]] と [[セッション管理]] を参照。
```

### 強いところ

* **タグはRAGのメタデータフィルタと相性抜群**：`tags: [auth]` は、そのままベクトルDBのメタデータになり「認証カテゴリだけに絞って検索」といったハイブリッド検索が実現できる。これは階層構造だけでは持ちにくい情報。
* **リンクが知識グラフになる**：`[[...]]` の関連は Graph RAG（関連ノードをたどって文脈を補強する手法）の素材になりうる。
* **1トピック1ファイルの粒度**：Zettelkasten的に粒度が揃っていると、チャンク分割がきれいで、Embeddingの精度が出やすい。

### 弱いところ（エージェントには不利になりがち）

* **`[[wikilink]]` はAIにとってただのテキスト**：Obsidianアプリはリンクを解決してグラフを描くが、Claude Code などは**リンク先を自力で grep して探す**しかない。人間向けの便利機能が、エージェントには余計な検索コストになる。
* **フラット構造は「地図」がない**：全ファイルが1階層に並ぶと、`ls` の結果が巨大なリストになり、どれを開くべきか推測しづらい。ファイル名が `2024-03-15.md` のような日付だと最悪。
* **タグはファイルを開かないと読めない**：ディレクトリ名は `ls` だけで見えるが、frontmatterのタグは各ファイルを read しないと分からない。探索型では割高。

つまり Obsidian形式は **RAGの前処理をきちんと組む前提** なら光るが、**「そのままエージェントに渡す」用途では階層構造に一歩譲る**、というのが実態です。

## 比較表

| 観点 | 階層構造＋index.md | Obsidian形式（タグ＋リンク） |
| --- | --- | --- |
| context正確性（エージェント） | ◎ ディレクトリ名で意図が明確 | △ フラットで迷いやすい |
| context正確性（RAG） | ○ 見出しでパンくず付与 | ◎ タグでメタデータ絞り込み |
| 認識速度（エージェント） | ◎ index.mdで一発把握 | △ 全ファイル走査になりがち |
| コスト（トークン） | ◎ 開くファイルが最小 | △ 探索が広がりやすい |
| チャンク分割のしやすさ | ○ 見出し階層が効く | ◎ 1トピック1ファイル |
| メタデータ活用（ハイブリッド検索） | △ ディレクトリ名頼み | ◎ タグ・aliasが強力 |
| 人間の編集体験 | ○ | ◎ |
| そのままAIに渡す手軽さ | ◎ | △ 前処理推奨 |

ざっくり言えば、**探索型はディレクトリ、検索型はタグ**が効く、という住み分けです。

## じゃあどう作ればいい？——両取りの実践レシピ

現実の最適解は「どちらか」ではなく「重ねる」ことです。

### 1. トップに「AI用の地図」を1枚置く

`README.md` や `index.md`、そして AI エージェント向けには **`CLAUDE.md`**（Claude Code）や **`llms.txt`**（Jeremy Howard／Answer.AI提唱の、サイト構造をLLMに伝える提案フォーマット）を用意します。ここに「何がどこにあるか」を数百トークンで書いておくだけで、エージェントの初動コストが激減します。

```
# このリポジトリの歩き方
- `guides/` : 実装ガイド
- `reference/` : APIリファレンス
- `adr/` : 設計判断の記録
迷ったらまず各ディレクトリの index.md を読むこと。
```

### 2. ディレクトリで「構造」を持たせる

意味のあるディレクトリ名を付け、各ディレクトリに `index.md`（概要＋子ファイルへのリンク）を置く。これがエージェントの探索を安く速くします。

### 3. frontmatterで「メタデータ」を持たせる

Obsidianの良さを取り込みます。ディレクトリ構造を壊さずに、各ファイルの先頭でタグを付ける。

```
---
title: "認証設計"
tags: [auth, security, backend]
updated: 2026-07-03
---
```

これでRAGに載せるときにメタデータフィルタが効く。**構造（ディレクトリ）とメタデータ（タグ）は排他ではなく、両立できる**のがポイントです。

### 4. リンクはAIに優しい形で

`[[wikilink]]` より、エージェントが辿りやすい **相対パスの通常リンク**（`[認証設計](./auth.md)`）を併用すると、人間にもAIにも親切です。ObsidianはWikilinkと標準Markdownリンクの両対応なので、設定で標準リンク寄りにしておくと潰しが効きます。

## 実際の事例・ツールで確認する

「どっちが正しいか」は、既存の成功例を見ると答え合わせになります。

* **llms.txt**：まさに「AIにサイト/リポジトリ構造を伝える」ための階層的な地図フォーマット。フラットなタグではなく、**構造化された目次**を推している点が示唆的です。
* **Dendron**：Obsidian的なMarkdownナレッジツールですが、`project.backend.auth` のような**階層的な命名**を採用している。フラットな知識管理ツールですら、AI/検索性を意識すると階層に寄っていく好例。
* **Repomix / gitingest**：リポジトリを1つのテキストにまとめてLLMに渡すツール。ここでも**ディレクトリツリーを冒頭に出力**します。構造こそがcontextの土台だと分かります。
* **PARA / Diátaxis**：人間向けの情報整理術ですが、いずれも「役割ごとにディレクトリを分ける」発想。AIにとっても、この区分けはそのままcontextのヒントになります。

一方でタグの価値も明確です。\*\*ハイブリッド検索（ベクトル＋メタデータフィルタ）\*\*は近年のRAGの定番で、`tags` はそのフィルタ条件として直接効きます。タグを捨てる必要はまったくありません。

## まとめ

| もし目的が… | おすすめ |
| --- | --- |
| Claude Code等にリポジトリを探索させたい | **階層構造＋index.md（＋CLAUDE.md / llms.txt）** |
| RAGの知識ベースにしたい | **階層構造＋frontmatterタグ**（ハイブリッド検索前提） |
| 人間もAIも使う社内ナレッジ | **両取り**（ディレクトリ構造＋タグ＋標準リンク） |

* エージェントは「人間と同じようにツリーを歩く」ので、**ディレクトリ名と index.md が最強のcontextヒント**になり、速く・安く・正確。
* Obsidianのタグやリンクは**RAGのメタデータとしては超優秀**だが、フラット構造・Wikilinkは**そのままのエージェント探索では割高**。
* だから対立させず、**構造はディレクトリ、意味はタグ**で重ねるのが2026年時点のベストプラクティス。

「AIに優しいリポジトリ」は、結局のところ「新人エンジニアに優しいリポジトリ」とほぼ同じです。地図があり、名前で中身が分かり、迷わない。

あなたのリポジトリは今、**AIエージェントが最初に開く1枚の地図**を持っているでしょうか？
