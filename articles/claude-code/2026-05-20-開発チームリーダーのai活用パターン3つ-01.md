## はじめに

こんにちは！開発部の根岸です。

弊社の開発部はいくつかのチームに分かれており、私が所属するチームでは **2週間スプリントのアジャイル開発** を回しています。私はそのチームの開発リーダーを担当しており、スプリント計画・チケットのアサイン・進捗管理・リリース調整といった「開発を進めるための周辺業務」を、メンバーの実装と並行して進めています。

スプリント・チケット・アサインなどの管理はすべて **Notion** に集約しており、AIは主に **Claude（Claude Code）** を使っています。

この記事では、開発リーダーという立場で実施しているAI活用パターンを3つ紹介します。

## 1. コードレビュー サブエージェント

### 特徴

コードレビューは Claude Code で実施しています。多くのチームでやっている定番の使い方かと思います。`code-reviewer` という[サブエージェント](https://code.claude.com/docs/ja/sub-agents)に切り出しています。

サブエージェントは Claude Code の機能で、特定のタスク専用に独立したシステムプロンプトを持つエージェントを定義できる仕組みです。`code-reviewer` には以下の責務を持たせています：

* PRモード（PR番号やURLが与えられた時）と、ローカル差分モード（PR作成前の最終チェック）の両対応
* レビュー観点を **コード品質 / セキュリティ / パフォーマンス / テスト** の4軸で固定
* 指摘を 🔴 Critical / 🟡 Warning / 🟢 Suggestion の3段階で出力

## 2. 要件チェックスキル — Notionの要件定義の不足を洗い出す

### 特徴

新規機能のチケットを切る時、Notionに要件定義を書きます。ただ、要件定義を書いた本人は前提知識を持っているので、 **「言われなくても分かるよね」というところが抜け落ちがち** です。これが後で「これって仕様どうなってるんだっけ？」とメンバーから質問が飛んでくる原因になります。

そこで `requirements-checker` という[スキル](https://code.claude.com/docs/ja/skills)を作りました。Notion の要件定義 URL を渡すと、以下を実行します：

1. プロダクト・技術スタックの前提知識 (`knowledge/` ディレクトリ) をロード
2. 指定された要件定義を Notion MCP 経由で取得（子ページ・コメントも追跡）
3. 要件に含まれるキーワードから関連するknowledgeをピンポイントで追加読み込み
4. **不足・曖昧・矛盾・考慮漏れ** を 🔴 重大 / 🟡 要確認 / 🔵 調査推奨 の3区分で指摘

スキルの原則として「**確実に間違っているものだけでなく、グレーも指摘する**」ようにしています。見落とすより過剰指摘の方がマシ、という判断です。

### 効果

* 要件定義のレビュー前に1回スキルを通すだけで、 **確認すべき点が事前に洗い出される** 状態になり、レビューの質が向上
* 「リリース計画書いてない」「エラーハンドリングの仕様が抜けてる」みたいな定型的な指摘が機械的に出るので、人間は本質的な議論に集中
* 過去の意思決定（`decisions/`）やインシデント（`incidents/`）を knowledge に蓄積していけば、 **「あの時の判断と矛盾してませんか？」** という指摘も可能に

### ポイント

#### ポイント① コードから既存仕様を読み解く

我々のプロダクトは歴史的経緯から **仕様書がNotionに整備されておらず、最新の仕様は実装コードが正** という状態です。Notionの要件定義だけ読ませても「既存仕様との整合性チェック」ができません。

そこで `knowledge/` ディレクトリに **「コードを読んで分かったこと」** を蓄積するアプローチを取りました。各リポジトリの責務、サービス間の連携、状態遷移の主要パターン、過去の意思決定の理由などをMarkdownで書き溜めていく。これを「チームの共有脳」として育てていく構成です。

#### ポイント② 常に読むナレッジと読まないナレッジを分ける

knowledge を全部読ませるとコンテキスト量が多くなってしまいます。プロダクト全体のドメイン知識を全部読んでから個別要件をレビュー、なんてやってると、肝心の要件チェックに使えるコンテキストが残らないということが起こり得ます。

そこで `INDEX.md` というメタデータ索引を導入し、各knowledgeファイルに front matter で `priority: always`（常時読む）か `priority: on-demand`（必要時のみ）かをタグ付けする運用にしています。

* **always**: プロダクト全体像・用語集・アーキテクチャ・リポジトリ一覧（プロダクト全体に効くので必読）
* **on-demand**: 個別機能の仕様（予約・決済・診療記録など）。要件定義のキーワードと `domain` タグを突き合わせて関連するものだけ読む

「全部読ませる」から「索引から必要な分だけ引く」に切り替えたことで、コンテキスト消費を抑えることができました。

## 3. 毎日のチームの健康状態確認スキル — 定常業務の自動化とフィードバックループ

### 特徴

定常業務として、毎朝バックログとリリース状況をチェックする時間がありました。具体的には：

* 直近リリースに対して未アサインのチケットが無いか
* 計画から遅れているチケットが無いか
* リリース手順書の準備に抜けが無いか
* 特定メンバーに負荷が偏っていないか
* 滞留している不具合チケットが無いか

これを5つのNotion DB（リリース・バックログ・サブタスク・メンバー・バグ）を横断して目視確認していました。

これを `backlog-management` というスキルとして自動化しました。毎朝 launchd で claude をプロンプト付きで起動し、その中で `backlog-management` を呼び出し、観点別の健康状態を HTML レポートと macOS 通知で受け取れるようにしています。

### 効果

* **毎朝の作業を削減** — レポートを眺めて、赤いアラートだけ深掘りすれば良い
* **見落としが減った** — 人間では定期作業を忘れてしまう日があったりするが、毎日定期作業として必ず実施することで抜け漏れが減り業務の質が上がった

### ポイント

#### ポイント① フィードバックループの導入

この発想は、チームメンバーが社内勉強会で共有してくれたAI活用の取り組みから学びました。**AIスキルは作って終わりではなく、運用しながら育てるものだ** 、という考え方です。具体的には：

* スキル運用中に出てきた **誤検知 / 見落とし / ユーザー訂正** を `feedback/` ディレクトリに記録
* 同種のフィードバックが3件以上溜まる、または構造的な問題を含むフィードバックが1件でもあれば、 `knowledge/` や `SKILL.md` へ **昇格**
* 昇格基準・運用ルールも `references/PROMOTION_RULES.md` として明文化

一発で完璧なスキルを作ろうとするのではなく、 **「とりあえず動くものを出して、フィードバックを溜めて、定常化したら昇格させる」** のサイクルを回します。

#### ポイント② state（動的事実）と knowledge（恒久知見）の分離

メンバーが今週は他チームへ応援に出ている、別チームから応援メンバーが一時的に加入している、みたいな **「今この瞬間の動的な事実」** を knowledge に書くと寿命の長い知見と混ざってしまいます。そこで `state/` ディレクトリを別に切って、こちらは **INDEXを介さず毎回全読み** する設計にしました。

* `knowledge/` — プロダクト・技術の前提知識（更新頻度: 低）
* `state/` — 今この瞬間のチーム状態（更新頻度: 高）
* `feedback/` — 運用中の気づきログ（昇格待ち）

データの寿命と更新頻度で読み込みポリシーを分けることで、AIへのインプットが常に新鮮かつ軽量に保たれます。

## まとめ

開発マネジメントにおけるAI活用パターンを3つ紹介しました。

3つを通して個人的に感じているのは、AIに「考えさせる」よりも **AIが考えやすい環境を整える** ことに時間を使った方がリターンが大きい、ということです。  
具体的には、役割を限定したサブエージェントを用意したり、ナレッジを索引化したり、運用しながら育てるフィードバックループを回したり——どのパターンも「AIの出力」より「AIに与えるインプットと設計」に注力したものばかりでした。

今後もAIを活用して楽して楽しく仕事をしていけたらと思います。
