## はじめに「LLMへの同じ指摘にげんなりしているこの頃」

こんにちは！  
[株式会社アドバンテッジリスクマネジメント](https://www.armg.jp/)（ARM）でフロントエンド開発担当のもろです。プリンは固めが好きです。

社内ではClaude Codeが導入されており、開発作業の効率化を推奨されています。私自身自社サービスの開発はもちろん、社内向けのツール開発など、複数のプロジェクトを担当しています。

そのなかで、プロジェクトごとの業務フローや参照ファイルなど、**Claude Code に毎回共有しなければならない情報**が増えていきました。

さらに、そうしたルールや前提を整理・管理する手間にも不満を感じていました。

今回は、そういった**プロジェクトごとの特徴やルールを整理する作業**について、自分なりに考えたので共有です。ぜひ同じ課題を感じている方の参考になれば嬉しいです！

### CLAUDE.mdの限界

Claude Code には、プロジェクトごとのルールや前提を書いておける `CLAUDE.md` があります。

そのため、「Claude Code に毎回同じ前提を伝えるのが面倒」という課題だけを見ると、まずはプロジェクトごとの `CLAUDE.md` を作ればよさそうです。私もこれまでは`CLAUDE.md`を使った管理をしていました。

ただ、`CLAUDE.md` は作業リポジトリに含まれることが前提になるため、基本的にはチームメンバーにも共有される形になります。**私にとって最適な仕事の進め方を、チーム共有の `CLAUDE.md` に書くのは違う**と感じ、今回は別の運用を考えることにしました。

つまり欲しかったのは、

* チーム共有ではなく、**自分専用**で管理できること
* **プロジェクトごとに整理できる**こと
* Markdown ベースで**気軽に更新できる**こと

この3つを満たせる管理場所でした。

そのため、プロジェクトごとに分けた Markdown ファイルを、自分が見やすい形（リンク・タグ・検索）で運用できる **Obsidian に、管理の中心を寄せる形**にしました。

### Obsidianならば、「個人専用・プロジェクト横断・書き込みしやすい」

Obsidianであれば以下のメリットがあります。

* 個人専用なので、**率直な失敗ログや改善メモを書ける**。
* 複数プロジェクトを持っていても、一つのVaultにまとめることで**フォルダ・ファイルを横断して整理**しやすい。
* Markdown中心で、AIに頼らず**自分自身でも書き込みができる**し、運用がしやすい。  
  もともとObsidianについてはタスク管理や学びの記録で使用していたため、利用のハードルが低かったですね。

### Obsidianで管理する情報の全体像

そんなわけで、Obsidianに各プロジェクトのルールをまとめ、Claude が記録・読み込みできるように整備することにしました。

![](https://static.zenn.studio/user-upload/494826ff08ec-20260516.png)  
**目標としては、以下の要望を満たすこととしました。**

* 各プロジェクトの自分用ルールをObsidianで管理する
* 同じ指摘の再発防止をする仕組みを作る
* Claude Codeが自動で読み込み・書き込みができるよう整備する

## フォルダ構成

![](https://static.zenn.studio/user-upload/e666c298f22f-20260516.png)

実際の構成はこんな感じです。Obsidian の Vault 直下に `00_AI-OS/` を切って、その中に「共通」と「プロジェクト固有」を分けて置いています。

```
00_AI-OS/
├── 00_Guide.md         ← 人間向けの運用ガイド（入口）
├── Global.md           ← 全プロジェクト共通のルール
└── Projects/
    ├── _Template/      ← 新規プロジェクト用ひな形
    │   ├── Rules.md
    │   └── Feedback.md
    └── 【プロジェクト名】/
        ├── Rules.md
        └── Feedback.md
```

考え方はシンプルで、「**全プロジェクトで共通して守りたいこと**」は `Global.md` に、「**そのプロジェクトでだけ通用すること**」は `Projects/【プロジェクト名】/` に分けて置く、それだけです。新しいプロジェクトを始めるときは `_Template/` を複製して名前を変えるだけで運用に乗せられるようにしています。

ファイル数も意図的に最小限にしています。最初はフィードバックを「種類ごとに別ファイル」と細かく分けていたのですが、ファイルが増えるほど書き込み先で迷ってしまい、結局更新が止まりました。今は1プロジェクトあたり `Rules.md` と `Feedback.md` の2ファイルに絞っています。

ちなみに、00という頭文字は、Obsidianフォルダ内の一番上に表示させるためにつけました。小さなライフハックですね。

### 00\_Guide.md

私自身が読み返すためのガイドブック的な役割です。運用の思想や各ファイルの役割、昇格ルールの判断基準など、「迷ったときに開く」入口を一箇所にまとめています。

ポイントは、ここにルール本文そのものを書かないことです。書いてしまうと `Global.md` と二重管理になり、どちらが正なのかわからなくなります。Guide には「これは `Global.md` に書く」「これは `Rules.md` に書く」といった参照先だけを書いて、本文は各ファイルに任せる方針にしました。

### Global.md

全プロジェクトに共通するルールだけを書く場所です。私の場合は次の4つを置いています。

* 期待品質（要件を外さない、不確実な前提を断定しない など）
* 絶対やらないこと（明示承認なしの外部書き込み、推測で URL を変える など）
* 確定ルール（説明を求めた時の書き方、コード実装記法など）
* 自分の特性（「確認して」は確認・報告のみ、など指示語の癖）

実際にはこんな感じで書いています。

```
## 期待品質
- 要件を外さない
- 不確実な前提を断定しない

## 絶対やらない（Do Not Repeat）
- 明示承認なしに外部システム（GitHub issue/PR・push）へ書き込む
- 推測で文言・トースト・URL を変える

## 確定ルール
- 実装前に変更対象ファイルと影響範囲を1回明示する
- 外部URL・ヘルプリンクは定数ファイルに集約し、view の data にハードコードしない
```

ここに書くかどうかの基準は「**どのプロジェクトでも同じ判断になるか**」だけです。

1プロジェクトでしか起きない事情は、後述の `Rules.md` 側に書きます。混ぜると `Global.md` がどんどん肥大化して読まれなくなるので、ここは意識して薄く保つようにしています。

### Projects/【プロジェクト名】/Rules.md

そのプロジェクトでだけ通用する確定情報を置くファイルです。私が担当しているサービスの例だと、こんな内容を入れています。

* 概要（リポジトリパス、主要技術、参照ドキュメント）
* 重視すること（既存コンポーネントの踏襲、定数やカラー変数の集約 など）
* 絶対やらないこと（同じコードを重複させて書くこと　など）
* 実装ルール / Issue 運用 / PR・コミット運用

実際にはこんな感じで書いています。

```
## 概要
- リポジトリ: 【org】/【プロジェクト名】
- フロントエンド: Vue 3 + Vuex + Vue Router
- 主要領域: 管理画面 / 従業員画面

## 絶対やらない（プロジェクト固有）
- view の data に外部URLをハードコードする
- 共通コンポーネントを既存使用例を確認せずに使う

## 実装ルール
- 外部URLは定数ファイルから参照し、view の data に直書きしない
- デモ値・モック・暫定処理には `// TODO: API完成後 → ◯◯` を付ける
```

ここは「**今後このプロジェクトでずっと守ってほしいルール**」だけを置く場所と決めています。試行錯誤中のメモや一度きりの気づきは入れません。そういった生の指摘は次の `Feedback.md` に書いて、繰り返し発生したものだけを `Rules.md` に昇格させる、という流れにしています。

### Projects/【プロジェクト名】/Feedback.md

会話のなかで**自分が受けた指摘・違和感・気づきを、原文に近い形で残していくログ**です。1指摘=1ブロックで、新しいものほど上に追加します。

```
2026-05-14 [コード]
- 指摘(原文): 「TODO部分をコメントアウトで書かれていません。APIがまだ実装されていない部分については、TODO コメントの記載をお願いします。」
- 再発防止: API 未実装の箇所にモック値や暫定処理を入れる場合は、宣言行と使用行の両方に `// TODO: API完成後に◯◯へ置き換え` 形式のコメントを残す。コードを見ただけで「どこが暫定実装なのか」「何に置き換えるのか」がわかる状態を保つ。
- 関連ファイル: src/.../script.js
```

意識しているのは次の3点です。

* **原文をそのまま残す**：自分で意訳すると、後で読んだときに「何が嫌だったのか」のニュアンスが消える
* **5つのタグで分類する**：`[コード]` `[UI]` `[テスト]` `[チケット]` `[コミュ]`。あとで再発傾向を集計しやすい
* **同じ指摘が3回出たら `Rules.md` に昇格させる**：感覚ではなく回数で判断する

なお、ここへの記録については、自分で書く場合もありますが、後述するスキルを使って自動で入れるようにしていますね。

以上のファイル構成を、各プロジェクトごとに用意しています。  
3ファイルの責務を整理すると、以下のようになります。

![](https://static.zenn.studio/user-upload/3782a805b571-20260518.png)  
このように管理することで、「**個人専用・プロジェクト横断・書き込みしやすい**」環境になったかと思います！

ただ、ファイル構成を整えただけでは、継続して更新されなければ意味がありません。そこで次に考えたのが、**skills による自動化**です。

## skills構成「ミスを指摘するごとに、勝手に賢くなるようにする」

![](https://static.zenn.studio/user-upload/3b06b483f188-20260516.png)

Obsidian の Markdown ファイルを **Claude Code 側から読み書きする形で半自動化する**ことにしました。  
私が用意したのは3つのスキルで、それぞれ「**読む**」「**即時書く**」「**まとめて整理する**」の役割を分担しています。

* 着手前：`read-project-memory`（Obsidianを読み込む）
* 会話中：`log-feedback`（指摘されたその場で書き込む）
* 会話末尾：`thread-memory`（仕分けてまとめて書き込む）

私がやるのは「**指摘する**」と「**最終的に昇格を承認する**」の2つだけです。記録と読み込みはスキルに任せて、自分は本来の作業に集中できる状態を作っています。

それぞれのスキルについて解説します！

#### read-project-memory「Obsidianの情報を読み取るskill」

作業に取りかかる直前に、`Global.md` と該当プロジェクトの `Rules.md`、それから `Feedback.md` の直近5件を機械的に読み込んでくれるスキルです。

実行されると、会話の冒頭にこんなブロックが1回だけ出てきます。

![](https://static.zenn.studio/user-upload/59aaca05142f-20260518.png)

起動するのは、実装・設計・デバッグなど実作業に入る**最初の1回だけ**です。挨拶や1往復で終わる質問では動かないようにしています。

**導入してから「前回も同じ指摘したのに」と感じる場面がかなり減りました**。毎回ルールを口頭で伝える必要がなくなったのが、地味に一番ありがたいですね。

#### log-feedback「指摘を自動的に記録するskill」

会話の最中に私が「違う」「そうじゃなくて」「〜しないで」「ダメ」と言った瞬間、`Feedback.md` の最上部へ自動で1ブロック追記してくれるスキルです。

書き込まれる内容は次のような形です。

* 日付
* 5タグから1つ（`[コード]` `[UI]` `[テスト]` `[チケット]` `[コミュ]`）
* 指摘の原文（意訳しない）
* 再発防止1文
* 関連ファイル

このスキルが入ってから、**「あとで書こう」が完全になくなりました**。指摘した瞬間に Obsidian 側へ記録が残るので、心理的なハードルがゼロです。

#### thread-memory「会話の学びをまとめて記録するskill」

会話の終わりに、**そのセッションで出た学び・ルール候補・気づきを仕分けして Obsidian に書き込んでくれるスキル**です。

log-feedback と役割が似ているので、自分のなかでは次のように使い分けています。

| スキル | タイミング | 書く内容 |
| --- | --- | --- |
| log-feedback | 会話中（即時） | 訂正発話を1ブロックずつ |
| thread-memory | 会話末尾（一括） | feedback / Rules候補 / 学び をバケット分け |

起動するのは、「整理して記録して」「次回に活かしたい」「AIメモリに書いて」などと私が明示したときです。

log-feedback だけだと、「訂正未満だけど次に活かしたい気づき」や「うまくいったやり方」が拾えません。**そういう粒度を補ってくれるのが thread-memory です。**

また、このタイミングで同じフィードバックを複数回実施していないかもチェックしています。もし3回以上似た内容のものがあれば、`Rules.md` に昇格させるか提案するようにしました。  
![](https://static.zenn.studio/user-upload/fb4c4f44d997-20260518.png)

以上のフォルダ構成とスキルを活用し、日々のプロジェクトをこなしています。構築には少し手間がかかりましたが、**課題が解決できた手応え**があります！

## 実際に運用して感じたメリット・デメリット

![](https://static.zenn.studio/user-upload/9d791d5cbb48-20260518.png)

#### メリット①. 個人専用で率直に改善ログを残せる

日々の指摘ログや失敗原因を他者の目を気にすることなく、整理しておくことができますね。  
やはり自分専用のAIは自分自身の性格、こだわり、仕事の進め方に起因するものなので、**指示の記録から学習してもらうことは重要**だと感じました。

結果として、同じ指摘をする回数が減り、満足しています。

#### メリット②. 複数プロジェクトを横断管理できる

プロジェクトごとに情報が分散していると、該当ファイルを一つずつ開かないと編集できませんでした。

今回、Obsidianに各プロジェクトの情報をまとめたことで**複数プロジェクトが簡単に確認できますし、編集もしやすく運用しやすくなりました。**

#### メリット③. AIツールを乗り換えても知識資産が残る

ObsidianにまとめているのはMarkdownファイルであるため、Claude Code以外であっても、簡単に読み込むことが可能です。

将来的に**Codex等にツールが変わっても簡単に再利用できる**ことは大きなメリットだと思います。

### デメリットはチームにノウハウが共有されにくいこと

最後に現状感じているデメリットも紹介しておきます。  
それは個人専用になったため、チーム全体に展開したほうがよい情報があっても共有されないことです。  
プロジェクト内で`CLAUDE.md`を持ち、積極的に更新する方針になったときに、**二重管理になることは問題**になるかと思います。

定期的に連携し、必要であれば更新する仕組みを今後考えていきたいですね。
