こんにちは！RemitAidでソフトウェアエンジニアをしているnos0611です。

## はじめに

RemitAidではDBスキーマを `schema.sql` で管理しているのですが、テーブルの増加に伴い関連性の把握がしんどくなってきました。  
そこで「`schema.sql` を読み取ってER図にする」ための Claude Code の**スキル** を作ってみたため紹介します。

成果物は公開しているため、もしよければご覧ください。  
<https://github.com/nos0611/schema-to-er>

## なぜ作ったのか

ER図の生成ツールはすでに色々あります。手軽なところだと [Mermaid](https://mermaid.js.org/) の `erDiagram` などです。  
これまでいくつかツールは使ってきましたがどうもしっくりくるものがなく、とりあえずClaude Codeで作ってみるかと思いついたのが発端です。  
私としては以下のような辛さを感じており、似たような悩みを持つ方の参考になれば幸いです（過去の記憶なので「最近はそんなことない」ということがあればすみません🙇）。  
逆に「この辛さを解消するんだったらこのツールがおすすめ！」などあれば教えていただけると嬉しいです！

* Mermaid
  + 自動レイアウトしか選べないため、テーブルが増えてくるととんでもなく見づらい
  + GitHubでもプレビューできるのはGoodだが、テーブルが多いとあまりプレビューも役に立たない
* PlantUML
  + やはり自動レイアウトがきつい。一定調整はできるが、思った通りに配置できないこともしばしば
* DBML
* ツール共通
  + DDLをSource of Truthとして使いづらい
    - 専用の言語で記述する必要があるため、本物のschemaと同期が面倒（CIなどでやりようはあると思いますが気持ちとして）
    - 変換ツールはあることが多いですが、変換でエラーになったりするケースも
  + RemitAidでは金融系プロダクトを扱っていることもあり、変なツールにDB情報を食わせたくない

## 作ったもの

ひとことで言うと **`schema.sql` → 単一HTMLのインタラクティブER図** を吐くツールです。  
Claude Code のSkillとして作成したため、「このスキーマをER図にして」と頼めば作成してくれます。

## 特徴

### 配置に関して

初めの方はなんとかして自動配置だけでベストを狙えないかと考えていたのですが、どうしてもテーブルがダマになったり、近くにいてほしいテーブルが散らばったりと問題点がありました。  
全自動で完璧な配置を求めることを諦め、ある程度自動配置したものを手動で整えてそのレイアウトを保存する方向にしました。

#### 階層レイアウトで自動配置

初手の自動レイアウトについてはMermaid が内部で使っている [dagre](https://github.com/dagrejs/dagre) と同じ系統の階層レイアウトを採用しました。  
なるべく線の交差が少なくなるように、テーブルの下を通らないようになっています。  
自動配置では上から下にリレーションが流れる形で配置されます。  
![](https://static.zenn.studio/user-upload/451e90f7bee6-20260611.png)

#### 意味的クラスタリング（AI支援）

Skillにした主な理由はここです。  
後から手動で並び替えるという前提に立った場合に、テーブルが**意味**（認証・カタログ・決済…）ごとにまとまっていた方が再配置が楽だと考えました。  
Claude がグループ分けと「隠すべきノイズ（全テーブルから参照されるハブなど）」を提案し、こちらが承認・修正すると、ヘルパースクリプトがそれをレイアウトに反映します。  
クラスタリングする場合はテーブルの流れを上→下or左→右で選択でき、以下の画像では左→右かつ同じようなグループが縦に並んでいます。  
![](https://static.zenn.studio/user-upload/24440508cfb6-20260611.png)

### 手で並べた配置を保存

自動配置されたものが気に入らなければ手動で配置修正が可能です。  
また、手動配置した上でExportすることでそのlayoutを記録しておくことが可能です。  
Exportしたlayoutを適用する場合はImportを実行するか、そのlayoutファイルと一緒に再生成するようSkillに指示を出します。  
![](https://static.zenn.studio/user-upload/7c00033a1a38-20260611.png)

### リレーションに関して

#### 明示的なFKと推測FKを区別

`REFERENCES` で定義されたFKは**実線**、`xxx_id` という命名から推測したFKは**破線＋別色**で描き分けます。  
推測は保守的にしていて、対応するテーブルが実在するときだけ線を引きます（`ghost_id` に対応する `ghosts` が無ければ何も描かない）。  
「定義済みの確実な関係」と「命名からの推測」が一目で区別できます。

#### リレーションの非表示機能

例えばマルチテナントのアプリケーションの場合、tenantsとのリレーションを全テーブルが持っているようなケースがあるかと思います。  
この場合にtenantsとのリレーションが表示されるとノイズになるため、選択して非表示にすることができます。

#### リレーションの集約機能

例えばuserAがuserBにタスクをアサインするような場合、tasksテーブルにはusersとのFKが2本貼られるようなケースがあります。  
この場合にもノイズになるため、同一テーブル同士のリレーションを1本に集約することができます。

## Claude Code のスキルとして作った話

今回はツール本体（Python＋HTMLテンプレート）を、Claude Code の `skill-creator` を使ってスキル化しました。  
スキルは `SKILL.md`（いつ発火するか＝説明文と使い方）に加えて、`scripts/` や `assets/` といった**スクリプトやテンプレートを同梱**できるのが便利で、まさに今回のような「決まった処理をするスクリプト＋資産」と相性が良かったです。  
レイアウトや線の描画は一発で決まったわけではなく、Claude Code と対話しながら「もっと拡散して」「線がテーブルに被るのを減らして」「ドラッグ中も避けて」…と何度も調整して詰めていきました。

## まとめ

* `schema.sql` から単一HTMLのインタラクティブER図を生成するツールを作りました
* 自動レイアウト＋「手で並べた配置を保存して使い回す」を重視しています
* Claude Code のスキルにすると、スクリプトと資産を同梱しつつ「自然言語で発火」できて便利でした
