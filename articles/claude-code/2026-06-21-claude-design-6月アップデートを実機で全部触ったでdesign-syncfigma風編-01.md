## なんでこの記事書いたんか

2026-06-17 に **Claude Design** がけっこうデカいアップデートを受けてな。`/design-sync` でデザインシステムとコードベースが双方向に同期できるようになったり、Figma みたいに要素を手でドラッグして直せるようになったり、使用量の枠が変わったり……と、けっこう盛りだくさんやってん。

ただ、このアップデート、ネットの解説記事を読んでると **事実と違うこと書いてる記事がけっこうある**んよ。「キャンバスの直接編集が6月の新機能や！」とか「Canva 連携が今回追加された！」みたいな話、ぜんぶ間違いやねん（理由は後で説明するわ）。

そこでこの記事では、ワイが実際に手元のアカウントとプロジェクトで触って確かめた範囲で、

* そもそも Claude Design って何やっけ（仕組み）
* 6月アップデートで **ほんまに変わったとこ** はどこか
* ネットでよう誤解されてる「6月の新機能じゃないもの」
* `/design-sync` で取り込んだデザインシステムの中身を実測したらどうなったか
* 結局 Figma や Canva とどう使い分けるんか

を順番に話していくで。公式が言うてることと、二次情報どまりの話と、ワイが実機で確かめた話は、ちゃんと分けて書くようにするわ。

## 3行サマリ ― ここだけ持って帰ってもええ

* **`/design-sync`** が今回の主役。GitHub やローカルのコードベースからデザインシステムを取り込んで、Claude Code 側にも Claude Design 側にも **双方向で同期**できる。出力が `#333` みたいなハードコード値やなくて **デザイントークン名** で出るようになるから、デザインレビューの手戻りが減る
* **要素を `drag / resize / align` で手直しできるようになった**のが6月の目玉。Claude に「作り直して」って頼まんでも自分でちょこっと直せる＝その分 **トークン（使用量）を食わへん**。初期に批判された "トークン浪費" 問題への直接的な回答や
* **使用量が chat / Claude Code と統一された**。設計をやりすぎるとコーディングに使える週次枠が直接減る（逆もまた然り）。ここは知らんと損する運用ポイントや

ほな、いってみよか。

## そもそも Claude Design って何やっけ

ざっくり言うと、**Claude のチャットからスライド・プロトタイプ・LP・1ページャー・マーケティング素材みたいなビジュアル成果物を作る Anthropic Labs の製品**や。アクセスは `claude.ai/design`、もしくはデスクトップアプリのサイドバーから。2026-04-17 に Opus 4.7 駆動でローンチされた。Pro / Max / Team / Enterprise が対象で、Free は対象外やで。

ここで一個だけ押さえといてほしい技術的な性質があってな。

> Claude Design は **ベクター編集ちゃう**。フロントエンドのコード（HTML / CSS / JS）を書いて、それをライブレンダリングする方式やねん。

これが全部の長所・短所の根っこになっとる。コードを書く方式やから「速い」「インタラクティブな成果物が作れる」「Git で管理できるテキスト資産になる」っていう強みがある一方で、「SNS の規定寸法ピッタリの画像書き出し」とか「精密なベクター編集」みたいなのは苦手や。ここ、あとでまた出てくるから頭の隅に置いといてな。

## 6月アップデートで変わったとこ、ちゃんと整理するで

ここからが本題や。6/17 アップデートの中身を、信頼度ごとに分けて見ていくわ。

### ① 使用量が「別枠」から「統一」になった

これ、地味やけど **運用にいちばん効く変化**やと思う。

ローンチ当初、Claude Design の使用量は chat や Claude Code とは **別建ての専用枠**やってん。Anthropic 自身も "alongside (not inside)"（既存の枠の "中" やなくて "横" に置く別枠）って説明しとった。

それが 2026-05末ごろから「Claude Design now shares usage limits with Claude.ai and Claude Code」っていうアプリ内通知が出はじめて、6/17 のアップデートで Anthropic が改めて **「Claude Design は chat / Cowork / Claude Code と使用量上限を共有する」** と言及した。

実務的にどう効くかというと、

> 統一後は **Claude Design を重く使うと、coding や chat に使える週次容量がそのまま減る**（逆も同じ）。

設計作業が本体チャットの枠を食うようになった、っちゅうことやな。「最近コーディングの枠の減りが早い気がする」と思たら、Design 使いすぎが原因かもしれへんで。

### ② `/design-sync` でデザインシステムとコードベースが双方向同期する

ここが今回いちばんおもろいとこや。

`/design-sync` は **Anthropic 公式のスラッシュコマンド**（Claude Code の skill）で、GitHub リポジトリ・デザインファイル・生アップロード・ローカルコードベースから、1つ以上のデザインシステムを取り込める。

ポイントは **双方向（two-way sync）** なところや。

* `/design-sync` で **デザインシステムをリポジトリに pull** して、実コンポーネントに対してビルドする
* 逆に **Claude Code で作ったものを Claude Design に push** して戻す

この双方向同期が 2026-06-17 に公式告知された分やね。

ほんで地味に効くのが **ブランド遵守チェック**や。Claude が「自分のデザインシステムにある実コンポーネント」でビルドして、出力をそれに照合して自動修正してくれる。結果、出力が `#333` みたいなハードコード値やなくて、ちゃんと **デザイントークン名で出る**ようになる。これ、デザインレビューで「色が定義と違う」「このフォント勝手に変えた？」みたいな手戻りが減るっちゅうことや。エンジニアならこの嬉しさ、伝わると思う。

### ③ `drag / resize / align` で手直しできる＝トークンを食わへん

これが6月の **目玉**やと思う。

新しくなったレイアウト操作で、要素を `drag`（移動）・`resize`（リサイズ）・`align`（整列）できるようになった。公式ブログも「New, rich layout controls let you **drag, resize, and align** elements」って表現しとる。

何が嬉しいかというと、

> 小さな手直しを **Claude に「作り直して（regenerate）」って頼まんでも、自分で直接できる**。

ローンチ初期の Claude Design って、ちょっとした修正でもいちいち Claude にプロンプト投げてアセット全体を再生成させとったから、**使用量（トークン）をめちゃくちゃ食う**って批判されとったんよ。「Pro の週次枠を25分で8割消費した」みたいなレビューもあったくらいや。

それが今回、手で直せるようになったことで、**小さな修正でモデル推論（＝トークン消費）を走らせんで済む**ようになった。公式も "the average turn now uses **fewer tokens**"（1ターンあたりの消費トークンが減った）って言うとる。①の使用量統一でキツくなった分を、こっちの効率化で補う狙いやろな。

## 実機で確かめた：Edit の Pro モードはガチで Figma 級やった

ここからはワイが実機（`claude.ai/design`）で触って確かめた話や。ネットの調査だけやと「Claude Design にはレイヤーパネルもインスペクタも無い」って書いてる記事があるんやけど、**実機で触ったら完全に否定された**で。

左パネルはモードで中身が変わる。**通常はチャット欄**やねんけど、「Edit」に切り替えると **編集パネル**になって、さらにその中に **Simple / Pro / Code / Tweaks** のタブがある。

で、**Pro モードがガチで Figma 同等の本格編集UI**やった。具体的にはこんな感じや。

| 要素 | 中身 |
| --- | --- |
| **レイヤーツリー** | Section / group / Heading / Text みたいな階層がツリーで見える |
| **インスペクタ** | 要素を選ぶと Sizing（Width / Height ＋ Hug / Fixed / Fill）、Position（Inline / Absolute ＋ offset）、Typography（Font / Size(px) / Color / Weight / Leading）が編集できる |
| **選択ハンドル** | キャンバス上で要素を選ぶとハンドルが出て、そのまま掴んで動かせる |

「Figma みたいに手で直せる」っちゅう体感は **正しかった**。しかもこれらの手動編集は **プロンプトを投げへん＝モデル推論を走らせへん操作**やから、トークン（使用量）を消費せえへん。さっきの③の話とつながるな。

## ここ誤解されがちやで：6月の「新機能じゃない」もの

冒頭で言うた「ネットの記事が間違ってる」話、ここで回収するわ。**6月の新機能と思われがちやけど、実は4月のローンチ時点から既にあったもの**を3つ挙げとく。

### 1. キャンバスの直接編集（WYSIWYG）は元からあった

「4月はチャットだけ → 6月でドラッグ＆ドロップ編集が追加された」って書いてる記事があるけど、これは不正確や。

**ローンチ時（4/17）から「テキストの直接編集・インラインコメント・調整スライダー」はキャンバス上に存在しとった**。公式も "without re-prompting"（再プロンプト不要）ってちゃんと明記しとったんよ。

ほな6月で新しくなったのは何かというと、**要素の `drag / resize / align`（リッチなレイアウト操作）に限られる**。直接編集そのものは既存機能や。ここの線引き、間違えんようにな。

### 2. Canva 連携・PDF / PPTX / HTML エクスポートも4月から

「Canva 連携が6月の新機能！」も間違い。**Canva（と PDF / PPTX / スタンドアロン HTML）は4/17ローンチ時点から既存のエクスポート先**や。

6月で純粋に新規追加された外部連携は **Adobe / Base44 / Gamma / Lovable / Miro / Replit / Vercel / Wix の8つ**やねん。公式ブログの "the list of connectors now includes ..." っていう文言は「現時点の全一覧」であって、「全部が新規」っちゅう意味やないんや。

### 3. 「正式版（GA）になった」わけでもない

これも紛らわしい。4/17 ローンチ時は Anthropic Labs の **"research preview"**（実験的プロダクト）やった。6/17 以降は公式も各報道も **"beta"** 表記になっとる（公式ブログ "in beta on Claude Pro, Max, Team, and Enterprise"、課金ヘルプ "beta-period rate limits"）。

ただし、

> **「preview から beta へ昇格させた」と明示宣言したソースは無い**し、**GA（正式版）化した証拠も皆無**や。

Anthropic Labs の実験枠は維持されたまま、提供フェーズが一段進んで表記が変わった、くらいの理解が正確やで。「正式版になりました！」とは書かんといてな。あと **Enterprise はデフォルト off**（管理者が組織設定で有効化する）やから、会社で使えへん人はそこ確認や。

## デザインシステムは結局ただのテキストや（ここがエンジニアに刺さる）

さて、ここがワイがいちばん伝えたかったとこや。

`/design-sync` でローカルに取り込んだデザインシステムの中身を、実際にワイのプロジェクトで開いて数えてみた。そしたらこんな感じやった。

| 種類 | 実測値 | 中身 |
| --- | --- | --- |
| **デザイントークン** | 126個 | `tokens/colors.css` `typography.css` `spacing.css` `effects.css` `fonts.css` に CSS 変数として展開 |
| **コンポーネント** | 10個 | 各 `.jsx` ＋ `.d.ts`（props 定義）＋ `.prompt.md`（用法）。バンドルは `_ds_bundle.js` |
| **specimen カード** | 41枚 | `guidelines/*.card.html` 等。先頭行の `<!-- @dsCard ... -->` マーカーで自動インデックス化されとる |
| **スライド** | 25枚 | `slides/` 配下 |
| **その他** | — | LP の UI kit、スライドデッキ／LP のテンプレートも込み |

何が言いたいかというと、

> Claude Design のデザインシステムは、\*\*ぜんぶ Markdown / CSS / トークンっていう「ただのテキスト」\*\*でできとる。

これ、エンジニアからしたらめちゃくちゃデカいで。テキストやから **Git で管理できる**し、差分も追える、壊れへんし使い回せる。デザインツールにありがちな「謎のバイナリ形式で誰かのアカウントに人質に取られてる」状態にならへんのや。

しかも構成が賢くてな。デザインシステムには **単一の入口 `styles.css`** があって、「Consuming projects link THIS file only（使う側はこのファイルだけ link せえ）」って明記されとる。つまり、スライドやろうが LP やろうが UI モックやろうが、**この1ファイルを link するだけで全ブランド定義を参照できる**。色やフォントを毎回手で合わせる消耗から解放されるわけや。

「ブランド定義を1回書いたら、誰が・どの成果物を作っても同じ見た目に揃う」――これが Claude Design の中核的な価値やと、実物を触って改めて思たわ。

## 何が作れるんか：スライド・LP・図解…ただし画像生成は無理

活用例も整理しとくわ。作れるもんと、苦手なもんをハッキリさせとく。

| 成果物 | 確度 | できること |
| --- | --- | --- |
| **スライド（16:9）** | ✅ | 「Q1結果の10枚デッキを」みたいな指示で完成デッキを即生成。インタラクティブHTMLでリアルタイム編集 |
| **LP / UIモック / プロトタイプ** | ✅ | ワイヤーフレーム〜コード駆動プロトタイプ。テキスト / DOCX / PPTX / XLSX / コードベース / Webキャプチャを入力に取れる |
| **図解・チャート** | ✅ | スライド内にデータ可視化を直接生成。HTML出力やからアニメ表現もいける |
| **ブランド資産（LP・メール・SNSプレビュー）** | 🟡 | 同一トークンの1ファミリーとして媒体横断で一貫出力。ただし後述の画像問題あり |

公式が押してるキャッチは **"a week of back-and-forth … in a single conversation"**（1週間の往復作業が、たった1回の会話で終わる）や。デザイン知識ゼロでも会話するだけでそれなりのもんが出てくる、っちゅうのが売りやね。

ただし、**苦手なとこ・できへんことも正直に書いとく**で。

* ⚠️ **AI画像生成（ラスター画像）は非対応**。扱えるのはチャート・ベクターグラフィック・レイアウトで、写真風の JPG / PNG を生成する機能は無い。SNS 投稿用のピクセル画像を狙った寸法でカチッと書き出す、みたいな用途は別ツールが必要や
* ⚠️ **PPTX エクスポートにはバグが残っとる**。グラデーション欠落とか画像クロップのズレとか。PDF はほぼ問題ないんやけどな
* ⚠️ さっきも言うたとおり、精密なベクター編集や開発ハンドオフ、ガチのチーム協業はまだ弱い

## 従来ツール（Figma / Canva）とどう使い分けるか

ほな、既存のツールとの使い分けや。比較表でいくで。

| 観点 | Claude Design | 従来（Figma / Canva / PowerPoint / 素の Marp） |
| --- | --- | --- |
| プロンプト→動く成果物の速度 | ✅ 圧倒的に速い | 手作業・テンプレ習熟が必要 |
| ブランド一貫性 | ✅ 定義後は全成果物に自動適用 | 都度手で合わせる・揺れやすい |
| 反復編集の細かさ | 🟡 Figma に一歩劣る | ✅ Figma が優位 |
| ベクター編集 / 開発ハンドオフ / チーム協業 | ❌ まだ弱い | ✅ Figma が必要 |
| 最終ポリッシュ・SNS規定寸法 | 🟡 苦手 | ✅ Canva 等が優位 |
| コスト / レート制限 | ⚠️ 使用量の消費が速い | 月額固定が多い |

ワイの結論はシンプルで、

> **初稿の高速生成とブランド一貫性は Claude Design、最終ポリッシュや正確な寸法は Canva・Marp で仕上げる** ――このハイブリッドが現実解や。

「全部 Claude Design でやろう」とすると、画像書き出しや細かい寸法でハマる。逆に「速い初稿づくり」と「ブランド定義の一元管理」に絞って使うと、めっちゃ強い。役割分担が大事やね。

## ほな、まとめていこか

最後に要点だけもう一回まとめとくわ。

* **`/design-sync` でデザインシステムとコードベースが双方向同期できる**ようになったのが今回の本丸。出力がデザイントークン名で出るから、ブランド遵守の手戻りが減る
* **`drag / resize / align` の手動レイアウト操作**で、作り直しを頼まず自分で直せる＝トークン（使用量）を食わへん。初期の "トークン浪費" 批判への直接的な回答や
* **使用量が chat / Claude Code と統一**された。設計しすぎるとコーディング枠が減るから、運用では意識しといたほうがええ
* 実機で触ったら **Edit の Pro モードは Figma 級**（レイヤーツリー・インスペクタ・選択ハンドル完備）やった。「レイヤーパネルが無い」は誤情報や
* 一方で **「直接編集」「Canva 連携」「GA化」は6月の新機能ちゃう**。ここは誤解せんように
* デザインシステムは **ただの Markdown / CSS / トークン**。Git 管理できて壊れへん、使い回せる。エンジニア視点ではここがいちばん刺さるとこや

Claude Design、初期は「速いけどトークン食いすぎ」「凝った編集は無理」って評価やったけど、今回のアップデートで **「手で直せる」「コードベースと同期できる」「使用量が現実的になった」** と、実務で使える方向にちゃんと寄ってきた印象や。

特に **デザインシステムをテキストで一元管理して、スライドも LP も UI もそこから生やす** っていう使い方は、エンジニアにこそハマると思う。気になった人は、まず手元のプロジェクトで `/design-sync` を一回回してみるとええで。ほな、また！
