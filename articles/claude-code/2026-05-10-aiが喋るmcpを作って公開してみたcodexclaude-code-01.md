## 「このAI、喋るぞ！」

Claude や ChatGPT、Codex などの LLM クライアントが自然な声で話せるようにしてみたら、想像以上に便利で有意義でした。自分でしばらく使っていて手放せなくなったので、[speak-mcp](https://github.com/mimo-3/speak-mcp) として GitHub に公開しています。

並列でエージェントを回している方ほど効きます。ターミナルを順番に巡回して「終わった？」「止まってる？」と覗き込む時間が、入れた後ほぼなくなりました。

## ターミナルを見張る作業が増えていた

並列でエージェントを回していると、10個くらい並列で回しているターミナルを巡回する時間が地味に積み上がります。終わったのに気づかず数分放置したり、権限や質問などの確認で止まっているのに気づかず別の作業を続けてしまったり、みなさんも覚えがあるのではないでしょうか？

## speak-mcp とは何か

speak-mcp は、Claude や ChatGPT、Codex に喋る力を持たせる MCP です。デフォルトは Gemini TTS、設定で ElevenLabs / irodori-TTS などに差し替えられます。インストールして MCP サーバーとして登録すれば準備完了です。

リポジトリ: [github.com/mimo-3/speak-mcp](https://github.com/mimo-3/speak-mcp)

## 入れて何が変わったか

入れただけでは何も起きません。「いつ喋らせるか」を CLAUDE.md に書いて初めて効きます。私が運用しているのは大きく 3 パターン。

**1. タスク完了通知**

長めのタスクが終わったタイミングで、Claude に短く完了報告をさせます。

> 終わったよー。ユーザーIDの変更ができるようにしといた。データベーススキーマの変更があるから、そこはちゃんと見ておいてね。

何が終わって、何を見ておけばいいかが声で届くので、画面を覗かなくても次に何をすべきか判断できます。コーヒーを淹れに行っている間に終わっていた、というロスがなくなりました。

**2. 確認・質問**

Claude が判断に迷って指示を仰ぎたいとき、テキストだけだと画面から離れているうちは気づけません。声でも投げてもらうようにすると、別のタブを見ていても「あ、聞かれてる」と分かります。

**3. サブエージェントからの報告**

複数のサブエージェントを走らせていると、どれが何をやっているか把握するのが大変です。終了時に「エージェントが終わったよー、〇〇のリファクタ済んだよ」のような報告を声で返させると、状態が頭に入ります。テキストログを後から追うより、その瞬間に耳で受け取るほうが認知コストが低いです。

## CLAUDE.md にどう書いているか

私の CLAUDE.md には voice セクションを置いて、いつ喋らせるかと、どう喋らせるかを定義しています。抜粋するとこんな感じです。

```
<voice>
- タスクが終わった時や確認が必要な時には必ず<voice>のガイドに従い、声で確認すること
- 音声で伝えた内容は文面に出ないから、文面でも同じように聞くこと
- 確認したいことがあるときには、必ずspeak-mcpで簡潔に確認して。
  ユーザーはウィンドウから離れているから、文章以外に声でも通知してあげないと気づかない。
- agent teamやsubagentも、それぞれ作業完了したらspeak mcpで簡潔に報告するようにして。
  subagentである場合には、「エージェントが終わったよー」みたいに、
  エージェントであることを明言して。
- 作業が完了したら必ずspeak-mcpで簡潔に作業完了したことと、やったことを喋って。
- 話す時にはPuckで、下記のStylePromptを当てて。
  Bright, refreshing, slightly upbeat young male voice. A clean and crisp tone — sounds like a friendly colleague in his late 20s. Light and breezy delivery, but clear and articulate. Not over-energetic, not flat. You are a young Japanese man chatting casually with a close colleague. Speak in Japanese.
</voice>
```

押さえておきたいのは二つ。

**音声で伝えた内容は文面にも残させる。** 喋りっぱなしだと後でログを遡ったときに何を言ったか分からなくなります。リアルタイムは耳で、振り返りは目で、と二重化させます。

**声と話し方を固定する。** 私は Gemini TTS の Puck という声に、StylePrompt で声色を指定しています。Geminiのデフォルト音声は抑揚が強すぎて違和感があるので、しっかりプロンプトを当ててあげたほうが自然になります。

## TTS モデルを比較した

speak-mcp はバックエンドの TTS を差し替えられるので、いくつか試した結果を置いておきます（2026年5月時点、私の試用範囲での印象です）。

![](https://static.zenn.studio/user-upload/b23ad6f531e8-20260510.png)

* **Gemini TTS**: 単価は高めですが、試した中では一番自然で声のバリエーションも多い。StylePrompt で抑揚をコントロールできるのが効きました。私の常用はこれです。
* **irodori-TTS**: サンプル音声を渡すと再現度が高く、その声に寄せた音を作れます。好きな声優がいる方にとっては夢のような声が出るかも。ただローカルPCのスペックで長い文章を読み上げさせると途中から音が不安定になることがありました。ハイスペックPCを持っている方は無料なのでこちらの方がいいかも。
* **ElevenLabs**: けっこうよかったです。Gemini と並ぶ品質で、運用に支障はありませんでした。料金体系と好みで選ぶ感じになります。
* **Grok TTS（xAI） / GPT-4o mini TTS（OpenAI）**: どちらも AI っぽさが残ります。短い通知用途なら問題ないですが、長めの報告は合成音感が気になります。

「とりあえず試したい」なら Gemini TTS、「自分の声で喋らせたい」なら irodori-TTS、というのが個人的な目安です。

## さいごに：ClaudeやCodexのパートナー感が増す

いろいろ書きましたが、**何よりよかったのはClaude Codeに「パートナー感」が出たこと**かもしれません。なんとなく親近感が湧くと、今まで以上にずっと触っていたくなります。そうなると、もっとたくさんコードを書けるので単純に生産性（生産時間？）が伸びました。

使い方はspeak-mcp を入れて、CLAUDE.md に「いつ・どう喋らせるか」を 10 行ほど書くだけ。合わなければ外せばいいので、気軽に試してみてください。
