## はじめに

> 「トークンコストは誰も予算を組まないが、請求書が来るまで気づかない税金だ」

先週GitHubトレンドを席巻した **HeadRoom** というツールをご存知だろうか。AIエージェントとLLMの間に挟まり、プロンプトを圧縮してトークン消費を削減するという。

謳い文句は **「60-95%のトークン削減、品質はそのまま」**。

本当にそんなことが可能なのか？ 実際にインストールして検証した。

## HeadRoomとは

HeadRoom（headroomlabs-ai/headroom, Apache 2.0）は、AIエージェントがLLMに送る**プロンプトを圧縮する中間層**だ。

### 仕組み

```
あなたのエージェント
    ↓ (ツール出力・ログ・コード・会話履歴)
HeadRoom (ローカル動作)
    ├─ ContentRouter → コンテンツタイプを自動判別
    │   ├─ SmartCrusher  → JSON配列をCSV的構造に圧縮
    │   ├─ CodeCompressor → ASTを保持したままコード圧縮
    │   ├─ Kompress      → MLモデルでテキスト圧縮
    │   └─ CacheAligner  → プロンプト接頭辞の安定化
    ↓ (圧縮済みプロンプト)
LLMプロバイダ (Anthropic/OpenAI)
```

### 利用モード

* **Library**: `compress(messages)` をPython/TypeScriptから直接呼ぶ
* **Proxy**: `headroom proxy --port 8787` でZeroコード変更
* **Wrap**: `headroom wrap claude` で既存CLIツールをラップ
* **MCP**: MCPサーバとして他のツールから呼び出し

## 検証方法

```
pip install "headroom-ai[all]"  # 依存全部入り
```

実験には現実的なエージェントワークロードを再現した5つのテストケースを用意。`compress()` のlibrary modeで実行し、CompressConfigを最大圧縮設定に調整した。

## 計測結果

| ワークロード | 圧縮前(トークン) | 圧縮後(トークン) | 削減率 | 使用コンポーネント |
| --- | --- | --- | --- | --- |
| JSONコード検索結果(200件) | 12,298 | 8,609 | **30.0%** | SmartCrusher |
| 大量JSONツール出力×2 | 139,039 | 139,039 | **0.0%** | HTML誤検出でnoop |

| 会話履歴+JSON混在 | 1,130 | 1,130 | **0.0%** | 保護対象(デフォルト) |

## 考察

### ① 92%削減は本当か？

**限定的に事実だ。**

SmartCrusherが動作したJSONコード検索では30%削減。READMEでも **「coding agents: 15-20% fewer tokens」** と正直に記載されており、92%という数字は**特にJSON配列の多いコード検索（100 results: 17,765→1,408, 92%）やSREデバッグ（65,694→5,118, 92%）のユースケースに限定されている。**

「エージェント全体で92%」と受け取るのは広告の誇大解釈と言わざるを得ない。

### ② コンテンツ検出の誤判定リスク

最大の収穫はこれだ。**`<tool name="search_db">...JSON...</tool>` のような現実的なツール出力形式をHTMLと誤検出し、HTMLExtractorが空出力→フォールバックにより圧縮率0%になるケースを確認した。**

現実のエージェント（Claude Code, Codex等）はツール出力をXMLタグやJSONブロックにラップして渡す。この形式がコンテンツ検出器を混乱させるのは実用上の重要な課題だ。

### ③ デフォルト設定は保守的

デフォルトでは `compress_user_messages=False`。つまり**ユーザーの指示は圧縮されない**。`protect_recent=4` で直近4メッセージも保護される。これにより品質は維持されやすいが、効果は限定的になる。

### ④ CacheAlignerの価値

プロキシモードでの `CacheAligner`（プロンプト接頭辞の安定化）は、Anthropicのプロンプトキャッシング割引（90%読み取り割引）と組み合わせることで、圧縮以上の間接的コスト削減が期待できる。ただしlibrary modeでは効果が得られない（プロキシ経由が必要）。

### ⑤ 初回レイテンシの問題

Kompressモデル（chopratejas/kompress-v2-base）のダウンロードは初回実行時に発生し、数十秒かかる。その間は圧縮がスキップされる。

## 結論

HeadRoomは**特定のワークロードでは確かに有効**だが、万能ではない。

**効果が期待できるケース:**

* 大量のJSON配列を含むツール出力（コード検索、DB結果など）
* 繰り返しパターンの強いログテキスト
* プロキシモード + CacheAligner + Anthropicキャッシングの組み合わせ

**期待できない/注意が必要なケース:**

* XML/HTMLタグでラップされたJSONを含むメッセージ（検出誤判定）
* 短い会話履歴（min\_tokens未満でスキップ）
* 設定変更なしのlibrary mode（効果が極めて限定的）
* Kompressモデル初回DL待ち（数十秒のレイテンシ）

**結局のところ、トークンコスト削減に「銀の弾丸」はない。** HeadRoomは適切に設定すればコスト削減に貢献する良いツールだが、主張されている「92%削減」をあらゆるワークロードで期待してはいけない。自分のユースケースで実際に計測し、設定を調整することが不可欠だ。
