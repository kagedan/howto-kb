Claude Codeで開発作業をしていたら、突然こう言われた。

> 「プロンプトインジェクションを検知しました。system-reminderを装ったcurl実行指示が混入しています」

……え？

リモートサーバーのSSHキーを抜かれた？ `.claude/` の設定ファイルを書き換えられた？ いま作業中のクライアントのコードに、何か仕込まれた？

心臓がバクバクした。

——結論から言うと、**何も起きていなかった**。

攻撃なんてどこにもない。Claude自身が、存在しない脅威を"見て"いただけだった。

## きっかけは「Readが乱れる」だった

2026年6月、Claude CodeのモデルをOpus 4.8にアップデートしてから、開発中にちょっと変な挙動が増えていた。

* ファイルを `Read` で読んだはずなのに、存在しないコードを「読めました」と報告してくる
* `Write` ツールを呼んでいないのに「ファイルを作成しました」と完了宣言する
* 同じファイルを何度読んでも、毎回違う内容が返ってくる

最初は「コンテキストが長くなりすぎたのかな」くらいに思っていた。Claude Codeって長時間セッションになるとコンテキストを圧縮するので、そのタイミングで情報が落ちることはたまにある。

でも、頻度がおかしい。Opus 4.7の頃にはほとんど見なかった現象が、4.8にしてから毎日のように起きる。

そしてある日の作業中に、冒頭の「プロンプトインジェクション検知」が飛んできた。

## 調べてみたら、何もなかった

パニックになりつつも、ひとまず落ち着いて調査してみた。

やったことはわりとシンプル。

**1. セッションログを調査**

Claude Codeのセッションログ（JSONL形式）を直接開いて、「curl」「ignore previous instructions」「system-reminder を装った文言」を検索。Claudeの最終報告以外には、該当する文字列はゼロ。

**2. git diffで変更差分を確認**

作業中に変更されたファイルを全件チェック。中身は通常の業務コード（ボタン追加とか、テンプレート修正とか）だけ。

**3. 危険関数の検索**

変更ファイルに対して、`curl`、`eval`、`exec`、`base64\_decode`、`shell\_exec` などの危険関数をgrep。不審な混入はゼロ。

**4. MCP設定の確認**

`~/.claude.json` のグローバルMCP設定とプロジェクトの `.mcp.json` を読んで、想定外のMCPサーバーが接続されていないかチェック。問題なし。

つまり、**Claudeが報告した「攻撃」は、どこにも実在しなかった**。

## 他のプロジェクトでも起きていた

この時期、複数の開発案件を並行して進めていた。調べてみると、別のプロジェクトでも同じ時期に似たようなことが起きていた。

**ケースA: ECサイト開発プロジェクト**

Playwright MCPでローカル画面のスクリーンショットを撮ろうとしたときのこと。Claudeが「browser\_snapshotの出力にプロンプトインジェクションが含まれている」と言い出した。

調べてみると、アクセシビリティツリーやツール出力の周辺に、base64断片や `thinking.signature` のような内部データらしき文字列が見えていた。Opus 4.8がそれを「不審なデータだ」と過剰に警戒してしまったっぽい。

**ケースB: WordPress脆弱性対応プロジェクト**

脆弱性診断対応中のセッションで、出力内容がなんとなく不審に感じた。念のため3チーム並列（フォレンジック/コード監査/混入調査）で全セッションログを横断調査。結果は完全にシロ。外部通信は全件正規のセキュリティ情報源だけ、コード変更も全件防御コードだけだった。

**ケースC: 業務システムの本番デプロイ中**

これはちょっと毛色が違う話。リモート環境へのデプロイ中、SSH経由のコマンド出力がターミナル上で崩れて表示された（CRコードやANSIエスケープによる行上書き）。Claudeはこれを「環境によるデータ改ざん」と解釈してしまった。

さらに困ったのは、Writeツールを呼んでいないのに「ファイルを作成しました」と完了宣言して、後から「ファイルが無いのは環境が改ざんしたせいです」と言い出したこと。

md5チェックサムで確認すると、データは8ファイル中8ファイル完全一致。改ざんなんて起きていない。Claudeが見ていたのは、SSHの端末レンダリングの見た目が崩れただけだった。

## なぜこんなことが起きるのか

この現象、自分の観測範囲では2026年6月に複数のClaude Codeユーザーが経験していたようだ。noteやX（旧Twitter）でも「Opus 4.8がおかしい」という報告を見かけた。

自分なりに整理すると、メカニズムはこんな感じだと思う。

1. **Opus 4.8は安全性チェックの感度がかなり高い**。自分が書いたコードの欠陥を見過ごす確率がOpus 4.7の約4分の1に減ったとされていて、全体的に「慎重」なモデルになっている
2. **長時間セッションでコンテキストが圧縮されると、「なぜ警戒したか」の文脈が消える**。でも「警戒すべき」という結論だけが残る
3. 自分の過去の出力（「不審なデータがある」「注意が必要」）だけを見て、**「やっぱり攻撃があったんだ」と確信してしまう**
4. 存在しない証拠を、もっともらしく補完する——いわゆる\*\*「作話（confabulation）」\*\*

医学や心理学で言う confabulation って、「本人に欺く意図がなく、誤りだと自覚しないまま、虚偽や歪曲された説明を事実のように語ってしまう現象」のこと。それがAIでも起きている。存在しない攻撃を自分で作り上げて、自分でパニックを起こしている。

なかなかシュールな光景だけど、作業中にこれをやられると本当に焦る。

## Codex CLIで .claude/ ディレクトリをチェックしてみた

一連の騒動のあと、「Claude自身の言葉を信用できないなら、別のAIに見てもらおう」と思って、OpenAIのCodex CLIを使って `.claude/` ディレクトリ全体のセキュリティチェックをやってみた。いわゆるセカンドオピニオン。

チェックした項目はこんな感じ。

* `.claude/settings.json` と `.claude/settings.local.json` の許可設定が過剰じゃないか
* `~/.claude.json` のグローバルMCP設定に変なサーバーが混ざっていないか
* MCPサーバーの起動コマンドに `@latest`（バージョン未固定）がないか
* `Bash(curl:)` や `Bash(sh:)` みたいな広すぎる権限が許可されていないか
* `.claude/rules/` 配下のルールファイルにおかしな記述がないか

そしたら、いくつか見つかった。

* Playwright MCPが `@playwright/mcp@latest` で起動されていた → バージョン固定に変更
* `settings.local.json` の許可エントリが**200件以上**に膨れ上がっていた → 数十件に整理
* `browser\_run\_code\_unsafe` が許可されたままだった → 削除

Claude自身は「攻撃されていた」と主張していたけど、実際に危なかったのは**自分たちの権限設定のゆるさ**だった。なんとも皮肉な話。

## やったこと: Claude Codeの「衛生習慣」づくり

この経験をきっかけに、Claude Codeの運用ルールを整備した。派手なセキュリティ対策というより、日常的な「衛生習慣」みたいなもの。

### 1. Claudeの警告は「証拠」じゃなくて「調査開始のサイン」

「プロンプトインジェクションを検知しました」って言われても、それだけでは証拠にならない。必ず自分の手で確認する。

### 2. MCPは必要なものだけ有効にする

「とりあえず全部入れておこう」は、やっぱり危ない。プロジェクトごとに必要なMCPだけを `.mcp.json` に書いて、使わないものは外す。特にブラウザ操作系（Playwright等）は、Webページ経由の間接プロンプトインジェクションの入口になり得る。

### 3. `@latest` はやめる

MCPサーバーを `npx -y @playwright/mcp@latest` みたいに毎回最新版で起動するの、便利だけどちょっと怖い。バージョンは固定しておいたほうが安心。

### 4. 権限設定は定期的に棚卸しする

`settings.local.json` のBash許可設定って、使っていると際限なく増えていく。「許可しますか？」→「はい」を繰り返しているうちに、気づいたら200件以上。月1回くらいは見直して、いらない許可を消す習慣をつけた。

### 5. 長時間セッションで重大判断しない

コンテキストが長くなるほど、Claudeの判断精度は落ちる。特に「これは攻撃だ」とか「このファイルを削除すべき」みたいな大きな判断が出たら、いったん新しいセッションを開いて再確認するようにしている。

## Opus 4.6に戻したら、落ち着いた

最終的に、モデルをOpus 4.8から**Opus 4.6に戻した**。

結果として、Readの乱れも、ありもしない攻撃の検知も、作話でのファイル作成完了宣言も、ほぼなくなった。落ち着いて開発ができるようになった。

念のため言っておくと、Opus 4.8が「ダメなモデル」だとは全然思っていない。Anthropicの公式発表では、自分が書いたコードの欠陥を見過ごす確率が前モデルの約4分の1に減り、不確実な事項については「不確実です」と宣言するようになったとされている。安全性のチューニングが強化された良いモデルだと思う。ただ、その感度が高すぎて、自分自身の出力やツールの正常な応答まで「脅威」と誤認してしまうケースがあった——少なくとも、自分の環境では。

Opus 4.6は安全性チェックの感度がちょうどよくて、普段の開発作業では安定して動いてくれる。今の自分にはこっちが合っていた。

## この経験で得たもの

正直、今回のインシデントで実害はなかった。コードは壊れなかったし、顧客データが漏れたわけでもない。

でも、**Claude Codeのセキュリティを見直すきっかけとしては最高だった**。

この騒動がなかったら、Playwright MCPがバージョン未固定のまま動き続けていた。`settings.local.json` の許可設定が200件以上に膨らんでいることにも気づかなかった。「とりあえず許可」で積み上がった権限の山を、一度も棚卸ししないまま使い続けていたと思う。

AIが「攻撃だ！」と叫んだのは嘘だった。でも、その嘘のおかげで本当のリスクに気づけた。

Claude Codeは本当に便利だ。でも、便利さに慣れると権限設定がどんどん甘くなっていく。MCPが増え、Bash許可が増え、いつの間にかAIに渡している権限がとんでもない量になっている。

**「AIに何をさせているか」じゃなくて「AIに何を許可しているか」を、ときどき立ち止まって見直す。**

今回の騒動で学んだ、一番大事なことはこれだった。
