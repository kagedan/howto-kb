最近、Claude Desktopを称賛する記事をよく見かけます。

たしかにClaude Desktopは便利です。  
MCPサーバーを接続すれば、ローカルファイル、Notion、GitHub、Google Analytics、WordPressなど、さまざまな外部サービスとClaudeをつなげることができます。

「これからはClaude Desktopに全部寄せればいいのでは？」

そう思う人も多いのではないでしょうか。

私もそう思い、実際にCLI版のClaude Code中心の運用から、Claude Desktop中心の運用に一度切り替えてみました。

しかし結論からいうと、**3日でClaude Code中心の運用に戻しました。**

理由はシンプルです。

**重いMCPをガンガン使うエンジニアにとって、Claude Desktopはまだ向いていない場面がある**と感じたからです。

この記事では、Claude Desktopそのものを否定するのではなく、実際に使ってみて感じた「思わぬ落とし穴」について書きます。

---

## 前提：Claude for Desktopは便利

まず前提として、Claude for Desktopはとても便利です。

特に次のような用途では、かなり相性がいいと感じました。

* MCP経由でデータを取得する
* 取得した情報を分析する
* グラフや表で可視化する
* ドキュメントを作成する
* NotionやGoogle Analyticsなどの情報を横断的に整理する
* 検索系、参照系のMCPを使う

いわゆる **GET系のMCP** とはかなり相性が良いです。

たとえば、GA4やSearch Consoleのデータを取ってきて、自然言語で分析させる。  
Notionやドキュメントを読ませて、要約や構成案を作らせる。  
ローカルファイルを横断検索させて、調査結果をまとめさせる。

こういった用途では、Claude Desktopはかなり使いやすいです。

GUIで会話できるため、ターミナルに慣れていない人にも使いやすい。  
ビジネス側の人と一緒に使う場合にも向いています。

ただし、問題はそこではありません。

私がつまずいたのは、**重い更新系MCPを実行する場面**でした。

---

## 落とし穴1：MCPツール実行のタイムアウト

一番大きかった問題は、MCPツール実行時のタイムアウトです。

私の環境では、データInsert系の重い処理や、WordPress記事をMCP経由で作成・更新するような処理を実行すると、Timeoutエラーが頻発しました。

単純な参照ではなく、複数のデータを登録したり、外部サービスに対して更新処理を行ったりするような処理です。

特に、次のような処理で顕著でした。

* データ登録系の重いMCP
* WordPress記事をMCP経由で執筆・更新する処理
* 複数件のInsertを伴う処理
* 外部APIをまたぐ更新処理
* 同時に複数の重いMCPを走らせる処理

体感としては、**同時に4つくらい重い登録系MCPを動かすと、かなり厳しい**です。

タイムアウトが頻発し、安定して作業を進められませんでした。

---

## 「MCPサーバー側の処理が失敗している」とは限らない

ここで重要なのは、MCPサーバー側の処理が必ず失敗しているとは限らない点です。

Claude Desktop側ではタイムアウトになっていても、MCPサーバー側では処理が完了している。  
しかしClaude Desktop側が先に待ち受けをやめてしまい、結果を受け取れない。

このような挙動が起きることがあります。

実際、Claude Desktopに関するGitHub issueでも、MCPサーバーはレスポンスを返しているにもかかわらず、Claude Desktop側が結果を受け取らず、`No result received from client-side tool execution` になるという報告があります。報告では、約60秒を超えると失敗しやすいとされています。

また別の報告では、Claude Desktopアプリがstdio MCPツール呼び出しを約60秒でキャンセルし、同じ処理をClaude Code CLIで実行すると完了する、という再現例も出ています。

さらに、remote / hosted MCP connectorでは「4分ちょうど」でタイムアウトするという報告もあります。

つまり、現時点では少なくとも次のように整理するのが安全だと思います。

```
Claude DesktopのMCPタイムアウト挙動は、
OS、Claude Desktopのバージョン、MCP transport、MCPサーバー実装、
local MCPかremote/hosted MCPかによって変わる可能性がある。

ただし、重いMCPツール実行で
クライアント側タイムアウトに引っかかる報告は複数ある。
```

そのため、「Claude Desktopは必ず2分で切れる」といった断定は避けたほうがよいです。

一方で、**重いMCPツールをClaude Desktopから長時間実行する場合、クライアント側タイムアウトに注意が必要**という主張は、実体験としても公開報告としてもかなり妥当だと思います。

---

## Claude Codeではタイムアウト調整の余地がある

Claude Code側では、MCPツール実行タイムアウトを調整する仕組みがあります。

公式ドキュメントでは、`.mcp.json` のサーバー設定に `timeout` フィールドを追加することで、サーバー単位のツール実行タイムアウトを設定できると説明されています。たとえば `"timeout": 600000` とすれば10分です。

また、環境変数 `MCP_TOOL_TIMEOUT` はMCPツール実行のタイムアウトを制御するものとして記載されています。

もちろん、Claude Codeでも何でも無限に実行できるわけではありません。  
処理設計が悪ければ遅くなりますし、外部API側のタイムアウトもあります。

ただ、少なくとも私は上記パラメータを延長してもタイムアウトの発生は抑えられませんでした。

Claude for DesktopとMCPサーバをつなぐときにはその間に中継サーバがあって、そこでタイムアウトするから、という話もありましたが事実かは不明です。

---

## 落とし穴2：同じMCPなのに実行時間が遅く感じる

もう一つ気になったのが、実行速度です。

同じMCPサーバーを使っているにもかかわらず、Claude Codeから実行する場合と比べて、Claude Desktop経由のほうがかなり遅く感じる場面がありました。

厳密なベンチマークを取ったわけではありません。  
そのため、ここはあくまで私の環境での体感です。

ただ、次のような処理では、Claude Desktop経由のほうが明らかに待ち時間が長く感じました。

* 外部APIをまたぐ更新処理
* WordPress記事の生成・更新
* データ登録を伴う処理
* 複数のMCPを並列的に使う処理

原因は断定できません。

Claude Desktopの内部実装、MCP transport、Claude側の処理経路、UI側の待ち受け制御、ツール実行時の制御など、いろいろな要素が絡んでいる可能性があります。

Claude DesktopとMCPサーバーの間に何らかの中継的な仕組みがあり、それが原因ではないかとも考えられますが憶測の域を出ません。

そのため、この記事ではそこは断定しません。

言えるのは、**私の環境ではClaude CodeよりClaude Desktop経由のほうが、重いMCP実行で遅く感じる場面が多々あった**ということです。

## CUIでMCPを使い倒す人にはつらい

私の場合、MCPを単なる便利ツールとしてではなく、実作業の中心に置いています。

たとえば、次のような使い方です。

* WordPress記事をMCP経由で作成・更新する
* DBに対して登録系の処理を行う
* 外部APIを使ってデータを更新する
* 複数のMCPを並列で動かす
* CUI上で重い処理を連続的に流す

このような使い方をする場合、Claude Desktopは少し厳しいと感じました。

特に、Claude Codeであれば普通に流せていた処理が、Claude Desktopではタイムアウトする。  
あるいは、動くには動くがかなり遅い。  
この差は、日常的に使うにはかなり大きいです。

もちろん、1回だけ実行する軽い処理なら問題ありません。

しかし、開発作業の中でMCPを連続的に使う場合、タイムアウトや遅延はかなりストレスになります。

---

## 重い更新系MCPはそもそも設計にも注意が必要

ここで補足しておくと、これはClaude Desktopだけの問題ではありません。

MCPで重い更新処理を実行する場合、MCPサーバー側の設計にも注意が必要です。

たとえば、次のような設計は危険です。

```
1回のMCPツール呼び出しで、
大量データの取得、加工、Insert、外部API連携、記事生成、更新まで全部やる
```

これは、どのMCPクライアントを使っても重くなります。

本来は、次のように分けたほうが安定します。

```
1. 対象データを取得する
2. 差分を確認する
3. 更新案を生成する
4. ユーザーに確認する
5. 小さな単位で更新する
6. 結果を検証する
```

特に更新系MCPでは、タイムアウトだけでなく、途中失敗時の整合性も問題になります。

たとえばWordPress記事を更新する場合、Claude Desktop側ではタイムアウトしているが、WordPress側では一部更新が完了している、という状態もありえます。

これはかなり怖いです。

そのため、重い更新系MCPでは次のような工夫が必要です。

* 1ツール1責務にする
* 長時間処理を1回のMCP呼び出しに詰め込まない
* Insert / Updateは小さな単位に分ける
* dry-runを用意する
* 実行前に差分を表示する
* 実行後に検証ツールを用意する
* 冪等性を意識する
* タイムアウトしても途中状態を確認できるようにする

Claude for Desktopで使ってほしいMCPサーバを作る場合、レスポンスタイムには特にシビアになったほうがいいと思います。

## Claude Desktopが向いている用途

一方で、Claude Desktopが向いている用途も明確にあります。

それは、**参照・分析・可視化・ドキュメント作成**です。

たとえば、次のような用途です。

* GA4やSearch Consoleのデータ分析
* NotionやGoogle Docsの要約
* GitHub IssueやPRの整理
* ローカルファイルの検索
* データを取得してグラフ化する
* 議事録や仕様書の作成
* 調査結果の整理

このような用途では、Claude Desktopはかなり便利です。

GUIで見やすく、会話もしやすい。  
ビジネス側の人と一緒に使う場合にも向いています。

特に、MCPを「情報取得の窓口」として使う場合には、非常に良い体験になります。

---

## Claude Codeが向いている用途

逆に、Claude Codeが向いているのは、よりエンジニアリング寄りの作業です。

特に、次のような用途ではClaude Codeのほうが向いていると感じます。

* 重いMCPを実行する
* 更新系MCPを使う
* 複数のMCPを連続的に動かす
* 長時間処理を実行する
* CUI中心で作業する
* コード変更とMCP操作を一体で進める
* ターミナル上で実行結果を確認しながら作業する
* タイムアウト設定を調整したい

つまり、MCPを「作業実行エンジン」として使うなら、Claude Codeのほうが向いている可能性が高いです。

## 使い分けの結論

私の中では、次のような整理になりました。

```
Claude Desktop
  → 参照・分析・可視化・ドキュメント作成に向いている

Claude Code
  → 重い更新処理・CUI作業・エンジニアリング実務に向いている
```

Claude Desktopは便利です。  
これは間違いありません。

ただし、**重いMCPを使い倒すエンジニアにとっては、必ずしも最適解ではない**と感じました。

特に、次のような人は注意したほうがよいです。

* MCPで重い更新処理を実行している
* WordPressなどの外部サービスをMCP経由で更新している
* DBへのInsertやUpdateをMCPで行っている
* 複数のMCPを並列または連続で動かしている
* Claude CodeでCUI中心に作業している
* タイムアウトに敏感な処理を扱っている

一方で、次のような用途ならClaude Desktopは非常に向いています。

* GET系MCPで情報を取得する
* 分析結果を会話形式で確認する
* グラフや表で可視化する
* ドキュメントを作る
* 非エンジニアも含めて使う
* 調査や検索を効率化する

---

## まとめ：Claude Desktopは万能ではない

Claude Desktopは素晴らしいツールです。

ただし、MCPをどのように使うかによって、向き不向きはかなり変わります。

少なくとも、**重いMCPを複数並列でバリバリ動かすような使い方**をしている人は、Claude Desktopに全面移行する前に、一度慎重に試したほうがよいと思います。

私自身は、Claude Desktopを完全に使わないわけではありません。

分析やドキュメント作成には使います。  
GET系MCPで情報を取得して、結果を整理したり可視化したりする用途では、とても便利です。

ただし、重い更新系MCPや開発作業の中心は、Claude Codeに戻しました。

結局のところ、重要なのは「どちらが優れているか」ではありません。

**用途に応じて使い分けること**です。

Claude Desktopを称賛する記事は多いですが、実際に重いMCPを使い倒してみると、見えてくる落とし穴もあります。

この記事が、Claude Desktopへの移行を検討しているエンジニアの参考になれば幸いです。

## 補足：この記事の前提

本記事は、筆者環境での実体験をもとにしています。

Claude DesktopのMCPタイムアウト挙動は、OS、Claude Desktopのバージョン、MCP transport、MCPサーバー実装、local MCPかremote / hosted MCPかによって変わる可能性があります。

公開されている報告でも、約60秒で切られるケースと、4分で切られるケースの両方が確認されています。

そのため、本記事では「Claude Desktopは必ず2分で切れる」といった断定は避け、**重いMCPツール実行ではクライアント側タイムアウトに注意が必要**という趣旨で整理しています。
