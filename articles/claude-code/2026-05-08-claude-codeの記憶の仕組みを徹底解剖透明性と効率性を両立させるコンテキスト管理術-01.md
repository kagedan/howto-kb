「クロードコード（Claude Code）が、以前話した内容を覚えているのはなぜ？」「記憶の仕組みはどうなっているの？」

そんな疑問を持ったことはありませんか？実は、クロードコードの記憶構造は、人間の脳と非常に似た、巧妙な3層構造になっています。そして、その実体は意外なほどシンプルで、透明性の高いものです。

この記事では、クロードコードの記憶の仕組みを徹底的に掘り下げ、それを踏まえた効率的な使いこなしのコツを解説します。

  

---

## 第1章：クロードコードを支える3つの記憶構造

クロードコードには、人間の脳の機能に対応する3つの記憶があります。1つではなく、役割の異なる3つの記憶が連携することで、高度な文脈理解と知識の蓄積を実現しています。

![](https://assets.st-note.com/img/1778197791-yaFEu9fOXSIT1GCnzMNY05Rw.png?width=1200)

【画像1：Claude Codeの記憶構造の全体像】

人間の脳（左）とClaude Code（右）の記憶構造の対応図。コンテキスト（作業メモリ）、セッション（短期記憶）、メモリー（長期記憶）の3つが、それぞれ異なる役割を担っています。

1. **コンテキスト（Context）：人間の「ワーキングメモリー」**
2. **セッション（Session）：人間の「短期記憶」**
3. **メモリー（Memory）：人間の「長期記憶」**

**衝撃の事実：記憶はすべて「ただのファイル」**

クロードコードの記憶は、データベースや暗号化されたブラックボックスではなく、すべてローカルのプレーンテキストファイル（Markdown、JSONL）です。テキストエディタで開けば、中身をすべて見ることができ、編集や削除も自在です。この「オープンブック」な透明性が、クロードコードの大きな特徴です。

## 第2章：長期記憶「メモリー」の実態と活用法

長期記憶である「メモリー」は、プロジェクトの効率を左右する重要な要素です。その実体は、主に2つのマークダウンファイルと、6層のヒエラルキー構造で管理されています。

### 2つの核心ファイル：claude.md と memory.md

メモリーは、役割の異なる2つのファイルに分かれています。

### メモリーの6層ヒエラルキー

メモリーは2ファイルだけでなく、組織全体のポリシーからセッションごとの要約まで、6つの層で構成されています。

![](https://assets.st-note.com/img/1778198003-uDn6URhQzdN7sSjivptM1OVB.png?width=1200)

【画像2：メモリー構造の深掘り】

6層の階層構造（左）と、中心となる2つのファイル（右）の関係。上層ほど具体的で優先度が高く、下層のルールをオーバーライドする。

1. **Managed Policy（組織ポリシー）：** 会社全体のルールなど、最も広範。  
   フォルダパス：アンソロピックが提供するシステム設定
2. **User Memory (Global)（ユーザー設定）：** 個人ごとのグローバル設定。  
   フォルダパス：~/.claude/config.json 等 (グローバル設定)
3. **Project Memory（プロジェクト設定）：** claude.md。プロジェクト共通ルール。  
   フォルダパス：(プロジェクトTOP)/claude.md
4. **Project Local（プロジェクトローカル）：** 個人×プロジェクト固有の、コミットしないルール。  
   フォルダパス：(プロジェクトTOP)/.claude/claude.md (ローカル用)
5. **Auto Memory（オートメモリー）：** memory.md。クロードが自動で書くメモ。フォルダパス：(プロジェクトTOP)/memory.md
6. **Session Memory（セッションメモリー）：** 会話中の自動要約（summary.md）。  
   フォルダパス：(プロジェクトTOP)/.claude/sessions/(UID)/summary.md

同じ内容で矛盾がある場合、より**具体的（上層）なルールが勝つ**仕組みです。

## 第3章：短期記憶「セッション」の仕組みと管理

セッションは、1回のCLI起動から終了までの会話全体を指します。各セッションには一意のUIDが振られ、JSONLファイルに記録されます。

### セッションの起動と終了

起動には4つのパターンがあります。Claudeに過去の記憶を呼び起こさせたいときに過去のセッションに切り替えることができます。

1. **claude：** 真っさらな新規セッション。
2. **claude continue：** 最新のセッションを再開。
3. **claude resume：** 過去のセッションを一覧から選んで再開。
4. **claude resume <UID>：** UIDを指定して再開。

終了は、exitコマンド、Ctrl+C、ターミナル自体のクローズ、または5時間の放置（タイムアウト）のいずれかで実行されます。

### 裏で走る「自動要約」

セッション中、会話が約1万トークンに達すると、バックグラウンドで「自動要約（Session Memory）」が走り始めます。summary.mdというファイルに、現在の状態、タスクの仕様、触ったファイル、エラーと修正内容などが、構造化された形式で書き込まれます。

過去の会話を再開（resume）する際、すでに圧縮（要約）されている部分は、このサマリーと未圧縮のメッセージだけを読み込むため、高速に復元できます。

**注意点：リバインドと保持期間**

## 第4章：作業メモリ「コンテキスト」の制約と自動圧縮

最もテクニカルで、かつ注意が必要なのが、人間のワーキングメモリーにあたる「コンテキスト」です。

### 200Kトークンの内訳：実質的な作業領域は？

クロードコードが使用するモデル（Opus 4.6やSonnet 4.5）のコンテキストウィンドウは200Kトークンですが、**そのすべてを会話に使えるわけではありません**。

![](https://assets.st-note.com/img/1778199565-CZJ5eipIHUwu1MjdcVRPro9L.png?width=1200)

【画像3：コンテキスト管理と自動圧縮】

200Kトークンの内訳（左）。固定消費と予約領域により、実質的な会話・作業領域は117Kトークン程度になります。右のゲージは使用率に応じた色の変化と、90%での自動圧縮（AUTO-COMPACT）の発動、要約への置き換えイメージを示します。

1. **固定消費（約50K）：**
2. **予約（33K）：**
3. **実質的な作業領域（約117K）：**

### コンテキストが一杯になったら？「自動圧縮（AUTO-COMPACT）」

コンテキストウィンドウが一杯になり、会話ができなくなるのを防ぐため、使用率が90%に達すると「自動圧縮（AUTO-COMPACT）」が発動します。

圧縮プロセスでは、**古いメッセージの入力と出力を、会話の「要約」に置き換えます**。これによりコンテキストを空けますが、ユーザーのリクエストの骨子や重要なコードスニペットは保持されるものの、**会話初期の細かい指示やツール実行の中間結果は消失する可能性があります**。「30分前に言ったことを忘れている」と感じる時は、この圧縮が原因であることが多いです。

## 第5章：クロードコードを使いこなすためのコンテキスト管理術

クロードコードの記憶構造を理解したところで、明日から実践できる、効率的な使いこなしのコツをまとめます。

1. **claude.md は150行以内に整理する**
2. **memory.md を「目次（インデックス）」として使う**

   * memory.md（Auto Memory）にも**200行制限があり、それを超えた分はファイルにあってもセッション開始時に読み込まれません。**ここも詳細を書くのではなく、「プロジェクト構造：Next.js 15 + TypeScript」と書き、詳細は別ファイルに分けて参照させる、インデックス的な使い方を徹底しましょう。
   * Claudeはプロジェクト内のファイルをスキャンできるため、基本的にはどのフォルダでも認識しますが、プロジェクトのルート（一番上の階層）にドキュメント専用のフォルダを作成するのが最も確実です。  
     おすすめの構成は以下の通りです。  
     my-nextjs-app/  
     ├── .cursorrules (または [memory.md](http://memory.md)) <-- ここには概要のみ記載  
     ├── docs/ <-- 詳細ドキュメント用フォルダ  
     │ ├── [tech-stack.md](http://tech-stack.md) <-- 技術スタックの詳細  
     │ ├── [coding-standards.md](http://coding-standards.md) <-- 命名規則や設計指針  
     │ └── [architecture.md](http://architecture.md) <-- ディレクトリ構造の詳細  
     ├── src/  
     ├── next.config.ts  
     └── package.json  
       
     **別ファイル（参照先）の書き方**  
     別ファイルには、Claudeが「何についてのファイルか」を即座に理解できるよう、冒頭に**ファイルの役割**を明記します。
3. **タスクの区切りごとに compact コマンドを活用する**
4. **大規模な調査はサブエージェントに投げる**

サブエージェントについては別の記事でまとめて紹介します。

## まとめ

クロードコードの記憶は、人間の脳を模倣した3層構造（コンテキスト、セッション、メモリー）で、すべてローカルのプレーンテキストファイルとして管理されています。この高い透明性により、開発者はクロードが何を覚えているかを完全に把握し、コントロールすることができます。

明日から、claude.md と memory.md の整理、そしてタスクごとの compact 活用を試してみてください。コンテキスト管理が劇的に楽になり、クロードコードとの開発がさらにスムーズになるはずです。
