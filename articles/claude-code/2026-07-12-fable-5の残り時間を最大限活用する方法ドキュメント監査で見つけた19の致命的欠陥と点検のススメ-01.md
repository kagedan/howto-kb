Claude Code Fable 5の締切が迫っています。日本時間7月13日（月）15:59まで。よく「7/12まで」と誤解されがちですが、実は丸1日余裕があります。この残り時間をどう使うかで、得られる価値が大きく変わります。

私はこの2日間を「生成系」ではなく「点検系」のタスクに充てることを強くおすすめします。その理由を、実際に経験したドキュメント監査の事例を交えて詳しく解説します。

## ドキュメント監査で見つけた19件の指摘

先日、私は自身の環境で運用しているドキュメント62ファイルを、Fable 5を使って丸ごと監査しました。対象はスキル定義、コマンド一覧、エージェント定義、ガイドラインなど、システム全体を支えるあらゆるドキュメントです。

結果、19件の重大な指摘が見つかりました。その中でも特に衝撃的だったのは、「本番の実装と真逆のことを教えるドキュメント」が4本も存在していたことです。

### 具体例：自動投稿システムのドキュメント問題

最も危険だった事例を紹介します。自動投稿システムの手順書には、こう書かれていました：

```
【正しい手順】
1. 投稿内容を生成
2. ユーザーに承認を求める（必須）
3. 承認が得られたら公開
```

しかし、実際の実装コードを確認すると、6週間前の改修で承認ゲートは撤去され、承認なしで自動公開される仕様に変わっていました。さらに、リライトエンジンも別のAPIに換装済み。ドキュメントだけが古い世界に取り残されていたのです。

このドキュメントは、障害が起きた深夜に読まれることを想定した緊急手順書でした。もし事故が発生し、この古いドキュメントを信じて操作していたら、混乱はさらに拡大していたでしょう。まさに時限爆弾です。

### なぜチャットでは発見できないのか

ここで重要なのは、この種の問題は「チャットで質問」しても絶対に発見できないという点です。例えば、Claudeに「自動投稿の手順を教えて」と尋ねても、ドキュメントに書かれている内容（古い手順）をそのまま返すだけです。62ファイルの記述と実装スクリプトを突き合わせて、初めて矛盾が見つかるのです。

SWE-bench Proで前世代+11ptという数字の意味は、まさにこの「長く深く読み続ける仕事」で体感できます。単発の質問応答ではなく、システム全体を俯瞰し、複数のドキュメントを横断的に比較検証する能力が、Fable 5の真価です。

## 残り2日でやるべき3つの点検タスク

では、具体的に何をすれば良いのか。おすすめの点検系タスクを3つ紹介します。

### 1. ドキュメント vs 実装のズレ監査

最も効果が高いのがこれです。手順は以下の通り：

```
# ドキュメントファイルの一覧を取得
find ./docs -name "*.md" -type f > doc_list.txt

# ドキュメントと実装を突き合わせて監査
claude "docs/ 以下の全ドキュメントを読み、対応する実装コード（src/ 以下）と比較して、矛盾点や古い記述をリストアップしてください。特に手順の変更やAPIの置き換えに注目"
```

この監査で見つかる典型的な問題：

* 削除された機能の説明が残っている
* 引数や戻り値の仕様が実装と異なる
* エラーハンドリングの手順が古い
* 依存関係が更新されていない

### 2. ナレッジベースの矛盾検出

複数のドキュメント間で矛盾がないかチェックします。例えば：

* 同じ用語が異なる定義で使われていないか
* 推奨プラクティスが場所によって異なっていないか
* バージョン番号や日付の表記揺れ

```
# 用語の一貫性チェック
claude "knowledge_base/ 以下の全ファイルを読み、以下の項目について矛盾を検出してください：
1. 'デプロイ'の定義が統一されているか
2. 推奨されるPythonのバージョン
3. エラーメッセージのフォーマット
4. 日付の表記方法（YYYY-MM-DD vs YYYY/MM/DD）"
```

### 3. 溜まった教訓メモの「確定ルール集」への蒸留

開発中にメモした断片的な教訓を、体系的なルール集にまとめ直します。これは単なる編集作業ではなく、知識の質を高めるプロセスです。

```
# 教訓メモをルール集に変換
claude "lessons/ 以下の未整理メモを読み、以下の観点で整理してください：
1. 現在も有効なルールと、時代遅れになったものの分別
2. ルール同士の優先順位付け
3. カテゴリ分類（セキュリティ、パフォーマンス、可読性など）
4. 具体例の追加
出力は rules/ 以下にMarkdownファイルとして保存"
```

## 生成系 vs 点検系：比較表

| 観点 | 生成系（記事量産など） | 点検系（ドキュメント監査など） |
| --- | --- | --- |
| 即時的な成果 | 記事が増える | 問題が見つかる |
| 期限後も残る価値 | 低い（記事は消費される） | 高い（改善が資産になる） |
| 必要な労力 | 低い（プロンプト次第） | 高い（深い読み込みが必要） |
| 発見できる問題 | 表面的な誤字脱字 | 構造的な矛盾や欠陥 |
| チームへの影響 | 一時的な情報提供 | 長期的な品質向上 |
| リスク低減効果 | ほぼなし | 障害防止に直結 |

## なぜ点検系が重要なのか：将来の展望

AIエージェントの性能が向上するにつれ、「生成」のコモディティ化はさらに進みます。2024年時点で、GPT-4やClaude 3.5は一般的な記事生成を人間と同等以上に行えます。差別化要因は、むしろ「点検・監査・検証」の領域にシフトしています。

特に以下の分野では、点検系のスキルが今後ますます重要になります：

* **コードレビュー**：AIが生成したコードの品質担保
* **セキュリティ監査**：脆弱性の早期発見
* **コンプライアンスチェック**：規制要件への適合確認
* **知識ベースのメンテナンス**：組織知の鮮度維持

Fable 5のような高性能エージェントは、単なる「高速な生成マシン」ではなく、「徹底した検証パートナー」として活用するときに真価を発揮します。

## まとめ：今日から始める点検ライフ
