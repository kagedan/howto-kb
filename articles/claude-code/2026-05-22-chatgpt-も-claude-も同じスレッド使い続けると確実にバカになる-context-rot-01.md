同じスレッドで ChatGPT や Claude を 1 日中使い続けているなら、それはシートベルトなしで高速道路に乗っているのに近い状態です。AI の精度がじわじわ落ちる感覚は気のせいではなく、Context Rot を含む 4 つの別現象が同時に効いた結果として説明できます。

この記事では、その 4 現象を一次情報で裏取りしながら分解し、明日から変えられるスレッド運用までを並べます。

## なぜ「同セッション使い続け問題」が今ホットなのか

長文コンテキストの限界は、2023 年に Stanford の Liu らが "Lost in the Middle" 論文で学術的に指摘した時点で一度話題になりました。それから 2 年経った 2025 年 7 月、Chroma 社が 18 モデルにまたがる横断検証を「Context Rot」というレポートで公開し、再び議論が燃え上がっています。

この間にモデル側は劇的に強くなりました。コンテキストウィンドウは数年前の数千トークンから、Claude Opus 4.7・Claude Sonnet 4.6・GPT-4.1・Gemini 2.5 Pro の各 1M トークン級まで広がっています。一方でユーザー側の使い方は ChatGPT 登場時とほぼ変わっていません。同じスレッドを延々と使い回し、話題が飛んでもそのまま続ける。

問題はむしろ悪化しています。「ウィンドウがデカいから大丈夫」という油断が、長いセッションを許容する方向に働いているからです。

## 同セッション劣化の正体は 4 つの別現象(TL;DR)

「なんとなく精度が落ちる」を 4 つに分解すると、対策は驚くほどシンプルになります。混ぜて理解しているうちは、的外れな対処をしがちです。

![](https://assets.st-note.com/img/1779411016-SgrCGNFeKVW0luTvjOR5zDn7.png?width=1200)

ここに加えてユーザー側の問題として、話題が変わってもスレッドを切らない習慣があります。レシピ相談していた同じスレッドで仕事のメール文を頼む、みたいな飛びは、上の 4 現象を全部同時に悪化させる起爆剤になります。

以下、現象 1 から順に中身を見ていきます。

## 現象 1 — コンテキストウィンドウの圧迫(物理の話)

LLM は内部に「会話を覚えている」のではなく、毎ターン会話履歴を最初から全部読み直しています。1 ターン目に投げた文章は、20 ターン目や 100 ターン目になっても、その都度モデルに再投入されます。比喩で言えば、毎朝出社するたびに昨日までの全議事録を読み直してから席につく新人のような状態です。

この再投入量には物理的な上限があります。2026 年時点の主要モデルは、Claude Opus 4.7 と Claude Sonnet 4.6 と GPT-4.1 と Gemini 2.5 Pro がそれぞれ 1M トークンのクラスにいます。

数字だけ見ると「全然余裕」に見えます。実際、ChatGPT や <Claude.ai> の UI で普通に会話している限り、上限超過のエラーに遭遇することは稀です。ただし、ここから先で話す現象 2・3 は、この物理上限のはるか手前で発動します。「ウィンドウが大きい = いくらでも入れていい」という直感は、次のセクションから順番に裏切られていきます。

## 現象 2 — Lost in the Middle(中央の情報が抜け落ちる)

Stanford の Liu らが 2023 年に発表した論文 "Lost in the Middle" は、長文コンテキストにおけるモデルの欠陥を体系的に示した最初の有名研究です。論文番号は arXiv:2307.03172、査読版は TACL 2024 に掲載されています。

実験は 2 種類のタスクで行われました。マルチドキュメント QA(複数の参考文書を渡して質問に答えさせる)と key-value 検索(キーから値を引かせる)です。論文の核心は、関連情報を「先頭」「中央」「末尾」と位置を変えて配置し、それぞれで正答率を測った点にあります。

結論はきれいな U 字カーブでした。先頭と末尾に関連情報がある場合は正答率が高く、中央に置かれると顕著に落ちる。しかも、明示的にロングコンテキスト対応を謳ったモデルでもこの傾向は変わらない、と論文は結んでいます。

イメージとしては、1 時間の会議で冒頭の議題と最後の決定事項は覚えているものの、中盤 40 分の議論を片っ端から忘れる中間管理職です。長い会話の冒頭で「日本語で回答して」「コードブロックは bash で」と指示したのに、中盤の質問になると平然と英語で返してくる、あれが典型的な Lost in the Middle の発露です。

## 現象 3 — Context Rot(長くなるほど雑になる)

この記事の主役です。Chroma 社が 2025 年 7 月 14 日に公開した "Context Rot: How Increasing Input Tokens Impacts LLM Performance" レポートが起点になります。このレポートは 18 モデルを横断的に検証し、「入力長が増えるほど性能が劣化する」ことを定量的に示しました。

対象モデルの内訳は次のとおりです。合計 18 モデル。

* **Anthropic(5)**: Claude Opus 4 / Sonnet 4 / Sonnet 3.7 / Sonnet 3.5 / Haiku 3.5
* **OpenAI(7):** o3 / GPT-4.1 / 4.1 mini / 4.1 nano / GPT-4o / GPT-4 Turbo / GPT-3.5 Turbo
* **Google(3)**: Gemini 2.5 Pro / 2.5 Flash / 2.0 Flash
* **Alibaba(3)**: Qwen3-235B-A22B / Qwen3-32B / Qwen3-8B

検証手法は 3 種類が組み合わされました。

* **NIAH 拡張**: 古典的な Needle in a Haystack に手を入れ、意味的に近い針と紛らわしいダミー文を混ぜて検出難度を上げた版
* **LongMemEval**: 平均約 11.3 万トークンの会話 QA を 306 プロンプトで評価し、「関連部分だけ」と「全文込み」の差を測る
* **Repeated Words Task**: 25 語から 10,000 語まで 1,090 通りの長さで、繰り返し語列の中の異物検出をさせる

レポートの結論は、英文のまま引用するのが正確です。

> Models do not use their context uniformly; instead, their performance grows increasingly unreliable as input length grows.

> (モデルは自身のコンテキストを一様に扱っていない。むしろ入力長が伸びるほど性能の信頼性は下がっていく)

注目すべきは、古典的なベンチマークの罠です。レポートは「Needle in a Haystack のような古典ベンチマークではどのモデルもほぼ満点を取れる」と認めています。しかし、より現実的な条件で測ると軒並み崩れる、とも指摘しています。具体例として挙がっているのは、Claude Opus 4 が Repeated Words タスクで 2,500 語あたりから「やる前に観察を述べてから挑戦する/挑戦しない」という不審な振る舞いを始めるケースです。「公称ウィンドウまでは安全」は古典ベンチマークでしか成り立たない、と読むのが正確です。箱の大きさと、中身を最後まで丁寧に扱えるかは別問題です。

## 現象 4 — 自己強化ループ(履歴に残った誤りが強化される)

最後はメタ問題です。LLM が一度誤った前提を出すと、その発言は履歴に残ります。次のターンでモデルは履歴を再投入されるので、「自分の過去発言」を参照する形で同じ前提に乗ったまま回答します。ユーザーが明示的にツッコまない限り、この誤りは延々と再生産されます。

たとえば序盤で「あなたのプロジェクトは Python 3.10 ですね」と AI が勝手に決めつけ、ユーザーが訂正しなかったとします。10 ターン後にコードを書かせると、当たり前のように Python 3.10 系の構文で返してきます。本当は Python 3.12 だったとしても、です。

これを「自己強化ループ」と呼ぶのは便宜上の整理で、確立された学術用語ではありません(類似の議論は "hallucination cascade" などの名前で散見されます)。ただ、現場の体感としては別の名前があってよい現象です。

厄介なのは、現象 1〜3 と相互作用して悪化する点です。会話が長くなる(現象 1)→ Lost in the Middle で初期の正しい前提を取りこぼす(現象 2)→ Context Rot で全体精度も落ちる(現象 3)→ 残った誤りを次ターンの参照に取り込む(現象 4)。同じスレッドで長く話すほど、4 つが手を組んで雪だるま式に劣化します。

## なぜ「メモリ機能」「要約」「広いウィンドウ」では防げないか

ここまでで「対策はどれもあるじゃん」という反応が出るはずです。実は、よく使われる 4 つの対策はどれも「セッションを切る」の代替にはなりません。順番に見ていきます。

**ChatGPT や Claude のメモリ機能**は、ユーザーの嗜好や定常的な前提を保存する仕組みです。誤った前提の伝播を止めるためのものではなく、現象 4 には効きません。むしろ間違った前提を記憶されると永続化される副作用があります。

**自動要約や自動コンパクション**(Claude Code の `/compact` など)は、長い履歴を圧縮して再投入量を減らす仕組みです。物理的な圧迫(現象 1)には効きますが、要約のプロセス自体が情報を歪めます。誤りが要約の中に紛れ込めば、それはもう本物として固定化されます。

**ウィンドウが広いモデルへの乗り換え**は、Context Rot の知見からすると一番裏切られやすい選択肢です。Chroma レポートが示したのは、古典ベンチマークが満点でも、現実的なタスクではモデル・条件ごとに違うタイミングで劣化が立ち上がること。1M に乗り換えれば全部解決、という直感は通らないと読むのが安全です。

**RAG**(検索で関連文書だけを入力に入れる仕組み)は、入力側のノイズを減らす意味では強力です。ただし、出力履歴の汚染、つまりモデル自身の過去発言が次ターンに乗ってくる部分は手付かずのままです。

結論として、技術的な工夫はどれも「同じスレッドを長く使い続けても大丈夫」を担保してくれません。

## 自衛策 — 明日から変えられる「スレッドの切り方」5 つ

ここまでが現象の話、ここからが行動の話です。優先度順に並べます。

* **(1) 話題が変わったら新規スレッド**: レシピ相談の続きで仕事のメール文を頼まない。話題の切り替えは現象 1〜4 を全部同時にリセットできる唯一の操作です
* **(2) 1 タスク 1 スレッド原則**: 「とりあえずさっきと同じ場所で」が一番の毒。タスクが変わったら新規、を初期動作にする
* **(3) 長い試行錯誤の後はリセット**: 失敗を含む履歴は次の質問の毒。デバッグや探索が一段落したら、要点だけメモして新スレッドに移る
* **(4) 重要な前提は毎ターン冒頭か末尾に再掲**: Lost in the Middle への直接対策。コーディング依頼なら言語・バージョン・スタイルを毎回 1 行で添える
* **(5) AI に「以前の自分の発言を疑え」と明示**: 自己強化ループへの直接対策。「これまでの会話で前提を間違えていないか、まず洗い出してから回答して」と一言入れる

Claude Code を使うなら、`/clear` の使いどころも明確になります。コーディング 1 タスクが終わったタイミングで毎回流す、くらいで丁度よいです。「セッションを切ると文脈ロストする」と感じても、要点メモを次セッションに渡せば十分です。

## 独自視点 — 「1 セッション 8 時間引きずって最後ズレた」失敗談

ここからは個人開発で実際に踏んだ失敗の話です。

Claude Code でとあるリファクタを丸 1 日かけて回したとき、朝に決めた「既存 API の互換は壊さない」という方針を、夕方の段階では AI 自身が忘れていました。

最後にコミット直前のレビューをさせたところ、AI から「既存 API の引数順を変更しているので、呼び出し側を更新しました」というコメントがサラッと返ってきました。これは初期方針に真っ向から矛盾しています。途中の試行錯誤で「やっぱり引数順は変えた方が綺麗ですね」みたいな自分の発言が履歴に残り、それが現象 4 の自己強化ループで「正」として固定化されたのが原因です。

それ以来、コーディング 1 タスクは 1 セッションで完結させる、長くなりそうなら途中で要点だけ別ファイルに書き出して `/clear`、というルールに変えました。体感ですが、初期方針からのズレが目に見えて減りました。

業務での ChatGPT スレッド運用も同じで、相談用のスレッドは週 1 で棚卸ししてアーカイブに回しています。コンテキストを使い切ってから切るより、定期的に切る方が結果的に質が高い、というのが個人開発・業務両方での結論です。

## まとめ

同じスレッドを使い続けたときに起きる精度劣化は、コンテキストウィンドウの圧迫・Lost in the Middle・Context Rot・自己強化ループという 4 つの別現象が合成された結果です。どれもユーザー側からは「なんとなく頭が悪くなった」としか見えませんが、対策は現象ごとに違います。

モデルのウィンドウは年々広がっています。それでも Context Rot は公称値の手前で発動する以上、行動の側を変えるしかありません。スレッドの切り方は、AI を使うすべての人にとってのシートベルトです。
