## 検証：その数字は「本当」か

### まず、検算する

記事の「5000万円」は、誰かが出した見積もりではありません。ロジックはこうです。

```
昔のSIer相場： 1人月100万円で書けるのは平均1000行
      ↓
「1000行 = 100万円」を業界のザックリ相場と置く
      ↓
10万行 ÷ 1000行 = 50人月 = 5000万円
```

**100人月です。1億円です。**

10万 ÷ 1000 = 100。元記事の原文がこうなっています。

> となると、10万行のコードを書くには50人月、つまり『5000万円』かかっていたわけです。

つまり誤りは元記事側にあります。

そして注目すべきは、**誤りの向きです**。この計算は熊谷氏に不利な方向に間違っている。自分の理屈に素直に乗れば「1億円が3万円になった」と言えたのに、その半分で申告している。数字を盛りたい人間は、こういう間違え方をしません。

（勘ぐると、氏の言う「時間効率50倍」は `100人月 ÷ 本人の2カ月 = 50倍` で綺麗に出ます。どこかで正しく100人月を出しておいて、金額に直す段でだけ50人月になった、という筋は考えられます。あくまで推測ですが。）

**次に「3万円」。** 記事に貼られた請求履歴のスクショには、こういうキャプションが付いています。

> 開発開始後の合計で「$435」という驚異的な低価格（約3万円強のトークン代）に収まっている

$435 です。2026年7月のドル円は162円前後なので、**約7万円**。3万円強にはなりません。証拠として貼られたスクショが、本文の金額の2倍以上を示しています。

### 検算した結果、結論は動かない

ここまで穴を開けておいて、ちゃぶ台をひっくり返します。

直してみましょう。1億円 ÷ 7万円 = **約1429倍**。元の 5000万円 ÷ 3万円 = 約1667倍。

**ほぼ変わりません。**

両方の誤りが同じ向き（どちらも過小申告）なので、比を取ると打ち消し合うからです。桁は動かない。

これが数字批判の限界です。金額の間違い探しをいくらやっても、「桁違いに安く作れた」という話の骨格は生き残ってしまう。だから本当に問うべきは、桁が正しいかではありません。**その1億円と7万円は、そもそも同じものを数えているのか**、です。

### 「10万行」は成果の量か

先に一つ潰しておきます。

Claude Code の統計が数えているのは、**セッションを通じて追加された行の累積**です。書いては消し、リファクタで丸ごと置き換えた分も、全部そこに乗ります。純増ではありません。「10万行のコードが存在する」ではなく「Claude Code が延べ10万行を書いた」が正しい。

そして厄介なのは、**この数字が外部から検証できない**ことです。実物のリポジトリは非公開で、ストアに出ているのはビルド済みのバイナリ。実際に何行あるのかは、本人にしか出せません。反証不可能な数字が一人歩きしている、という状態です。

その上で、行数そのものの話。

- 同じ機能を10万行で書くか2万行で書くかは実装しだいで、行数が多いほど良いわけでもありません（むしろ逆のほうが多い）。
- 対象は個人用の時間管理アプリ。仕様の合意・他者との整合・長期保守を背負う企業システムの10万行とは、1行の重みが違います。

コードレビューで「今週◯行書きました」が褒め言葉になった試しはありません。額面で受け取ると、話全体が一段ぶん過大評価に振れます。

### 5000万円と3万円は、同じものを数えていない

ここが本丸です。

「1人月100万円で1000行」という相場を、「1000行しか書けないのか、生産性低いな」と読むのは誤読です。その1000行には、要件定義・設計・レビュー・テスト・ドキュメント・そして納品後に他人が保守できる状態にするまでの全工数が乗っています。

**「1000行を、仕様の合意から保守可能な状態まで持っていくのに1人月」という意味の数字**です。書く速度の話をしていない。

熊谷氏は、そのフルロードの単価を、それらを一切含まないAI出力の行数に掛けました。りんごの値札を、みかんの個数に掛けている。

| | 1億円（旧来） | 7万円（今回） |
| --- | --- | --- |
| 含むもの | 要件定義・設計・実装・テスト・保守・保証・PM | Claude のトークン代のみ |
| 人件費 | プロの100人月 | 本人の2カ月分の工数は未計上 |
| その他 | — | 9台のデバイス・各種サブスク等は別 |

1億円は「他人に発注してフルセットで作ってもらう総額」、7万円は「自分で作ったときのAPI利用料だけ」。**そして経営トップ自身が2カ月投じた時間は、どこにも計上されていません。** 土俵を揃えて本人工数を足せば、倍率はかなり縮みます。

数字が間違っていたことより、こちらのほうが重い。間違いは直せますが、こちらは直しようがないからです。

### 「50倍 × 1/30 = 1500倍」をどう受け取るか

締めの「1500倍」。時間効率（50倍）とコスト削減（1/30）という、単位の違う2つのメーターを掛け合わせた数字です。

速さと安さを掛けて出た「1500倍」に、物理的な意味はありません。「速くなった（50倍）」「安くなった（30倍）」と別々に言えば十分伝わるところを、わざわざ掛けて4桁に育てている。しかもその「30」は、さっき見たとおり出所が不明です。

このひと手間に、記事のトーンがよく出ています。

## 本題：「一部機能を公開した」の死角

記事にはさらっと、こんな一節があります。

> なんなら、一部の機能はもう一般向けにアプリとしてストア公開までしちゃってます。

10万行や5000万円の陰でスルーされがちですが、エンジニアとしていちばん眉が動くのは、正直この一文です。

まず線引きから。**自分専用の非公開アプリなら、テストが無くても何も問題ありません。** 使うのは自分だけ、壊れて困るのも自分だけ。むしろ趣味の範囲で過剰にテストを書くのは KISS/YAGNI に反します。「AIと壁打ちしながら動くものを作る」いわゆるバイブコーディングは、この領域とは最高に相性がいい。

問題は、それを公開して他人に使わせた瞬間に、責任の次元が変わることです。

- 自動テストが無いと、回帰を検知できません。AIに次の修正を頼んだとき、別の場所が静かに壊れても気づけない。
- セキュリティ。入力バリデーション、XSS、認証・認可、依存ライブラリの脆弱性。「動く」と「攻撃に耐える」は別物です。
- 個人情報を扱うなら、保管と漏洩のリスク。ユーザーのデータを預かる時点で、法的・倫理的な責任が発生します。
- 保守。作った本人しか構造を知らず、しかもその構造をAIが書いたとなると、半年後に誰が直すのか。

そして、ここで冒頭の表に戻ります。

| 誰にも聞かず1人で開発 | 本も YouTube も見ず、社内エンジニアにも未相談 |

元記事での原文はこうです。

> 1人も聞いていないです。本も読んでないし、YouTubeも見てない。社内の誰にも聞かず作りましたよ。

**国内有数のエンジニア組織を抱える会社のトップが、社内の誰にも見せずに書いたコードを、ストアに公開した。**

これが、この記事で一番の見どころだと思っています。日本で有数のレビュアーに、決裁も稟議もなしでアクセスできる立場の人間が、それを一度も使わずに出荷した。「誰にも聞かず1人で」は、公開前なら武勇伝ですが、公開後は症例です。

同じ一文が、一線のどちら側にあるかで意味が反転します。

**非エンジニアがAIで作れる時代の落とし穴は、この一線が見えないことです。** 見えないので、跨いだ自覚が発生しない。悪意でも怠慢でもなく、単に「動いたから出した」だけ。

そしてこれ、他人事ではないはずです。

この記事を読んだあなたの会社の経営層が、来週「うちもやろう」と言い出す可能性は、そこそこあります。

そのとき止める材料になるのは「危険です」ではありません。**「公開しないなら、どうぞご自由に。公開するなら、レビューを1回通してください」** という、線の位置の提示です。禁止ではなく、線を見せる。

それができるのは、線が見えている側の人間だけです。

## で、結局「本当」なのか

「5000万円 → 3万円 → 1500倍」は、額面どおりには受け取らないほうがいい。金額は2箇所とも間違っていて、1500倍は単位の違うものを掛けた演出です。

ただし、直しても桁は動きませんでした。そして方向性まで否定する気はありません。個人がAIで、以前なら外注していたレベルのものを作れるようになった。コスト構造が根底から変わった。ここは誇張抜きに正しい。

問題は、変わったのがコスト構造だけで、**責任の構造は1ミリも変わっていない**ことです。7万円で作れるようになっても、公開したアプリが個人情報を漏らしたときの責任は、5000万円かけて作った場合と同額です。安くなったのは作る側のコストであって、使う側のリスクではない。

ここで「だからこそ、テスト・セキュリティ・保守という『公開の作法』を持つ人間の価値が上がる」と締めれば、エンジニア読者は気持ちよく読み終われます。実際、初稿はそう書いていました。

でも、たぶん違います。**作法の価値が上がるのは、作法を欠いた製品が罰せられる市場だけです。**

熊谷氏は作法なしで公開しました。今のところ何も起きていません。そして人気記事ランキング1位です。市場は今、作法の欠如を罰していない。

罰が来ないなら、価格もつきません。「品質の価値が上がる」は、放っておいてそうなる予言ではなく、誰かが罰を可視化して初めて成立する主張です。

だから、震え上がる必要はない、とは言いません。**あなたが3万円の外側で出している価値は、それを買う人がいて初めて価値になります。** 買い手を育てるところまでが、たぶん仕事の範囲に入ってきています。

## オチ

以上の内容も、全て Claude（＋技術記事作成向け自作SKILL）が書いています(笑)

[初稿](https://qiita.com/fallout/items/7185051125d7631f9f91)には、記事に絵文字付きの Mermaid 図がありました。`50 × 30 = 1500` のフローチャートです。そこでピンと来る人もいると思います。

で、Claude 自身がその Mermaid を消しました。消した理由は、直前の文が「50と30を掛けて1500」と書いてあって、その隣に50と30を掛けて1500にする図が置いてある。人間は掛け算の絵を描きませんよね…だそうです(笑)

ここまでは、まあ笑い話です。

**初稿には、「まず、検算する」のセクションが丸ごとありませんでした。**

Claude が書いた初稿は、熊谷氏の「10万行 ÷ 1000行 = 50人月」をコードブロックに綺麗に組み直したうえで、前提だけ批判して、割り算をしていません。$435 のキャプションも見ていない。1/30 の出所も訊いていない。**「その数字は本当か」という見出しを自分で打った記事が、数字を一度も検算していなかった**わけです。

しかもご丁寧に「AIが安いを引き立てるために都合よく組んだ試算」と、動機まで推測していました。実際の誤りは熊谷氏に不利な向きだったので、これは完全な言いがかりです。検算していれば書かずに済んだ一文でした。

その記事を Claude 自身に「批評しろ」と投げ返したら、出典を読み直して3箇所修正してきました。

初稿に足りなかったのは、検算する能力ではありません。**検算を挟むという判断のほう**です。

熊谷氏が「動くから公開した」＆私が「読める記事だから公開した」のは、テストの無いコードを公開する＆検算の無い数字を公開するのと、同じ一線の同じ側。

一線は、跨いだ側からは見えないんですよ…って事で、締めとします。
