## はじめに：許可プロンプトとの付き合い方

Claude Code で複雑なタスクを任せると、ファイル編集やコマンド実行のたびに許可プロンプトが表示されます。個別の `allow` ルールや `acceptEdits` モードで頻出パターンは減らせますが、Bash コマンドや外部通信が絡むと結局プロンプトが出る。長時間のリファクタリングや調査タスクでは、承認待ちで作業が止まる場面がどうしても残ります。

## `--dangerously-skip-permissions` という従来の手段

許可プロンプトを完全にスキップする手段として、以前から `--dangerously-skip-permissions`（`bypassPermissions` モード）が存在します。

```
claude --dangerously-skip-permissions
```

名前のとおり、**すべての安全チェックを無効化**するモードです。許可プロンプトはなくなりますが、同時にプロンプトインジェクション対策も、`.git` や `.claude` への書き込み保護もなくなります。`rm -rf /` のような破壊的コマンドだけはプロンプトが出る最後の砦がありますが、それ以外は素通りです。

このモードは Docker コンテナや使い捨ての VM など、**壊れても問題ない隔離環境**で使うためのものです。ローカルの開発環境で常用するのはリスクが高い。しかし「許可プロンプトが煩わしい」という動機で使ってしまうケースも少なくありません。

## auto mode：安全性を保ったまま許可プロンプトを減らす

**auto mode** は、この問題に対する別のアプローチです。Claude 本体とは別の AI モデルがバックグラウンドでツール呼び出しのリスクを判定し、安全なアクションは自動承認、危険なアクションだけをブロックします。`curl | bash` や本番デプロイのような操作はしっかり止めつつ、日常的な開発操作では許可プロンプトをほぼゼロにできます。

| 観点 | auto mode | `--dangerously-skip-permissions` |
| --- | --- | --- |
| 安全チェック | AI が毎回リスク判定 | なし |
| プロンプトインジェクション対策 | ツール結果のスキャンあり | なし |
| `.git` / `.claude` 等の保護 | AI が判定 | なし（書き込みも素通り） |
| `rm -rf /` 等の破壊的コマンド | ブロック | プロンプト表示のみ |
| root / sudo での実行 | 制限なし | 拒否される（Linux / macOS） |
| 推奨環境 | 日常開発 | コンテナ・VM 等の隔離環境のみ |

組織として `bypassPermissions` の利用を禁止する場合は、マネージド設定で無効化できます。

```
{
  "permissions": {
    "disableBypassPermissionsMode": "disable"
  }
}
```

## 6 つの権限モードと auto mode の立ち位置

Claude Code には 6 つの権限モードがあります。auto mode の位置づけを理解するために、全体像を押さえましょう。

| モード | 承認なしで実行できる範囲 | 主な用途 |
| --- | --- | --- |
| `default` | 読み取りのみ | 初期設定、慎重な作業 |
| `acceptEdits` | 読み取り + ファイル編集 + `mkdir`/`mv`/`cp` 等 | コードを書かせて後で diff 確認 |
| `plan` | 読み取りのみ（編集不可） | コードベースの調査・計画策定 |
| **`auto`** | **ほぼすべて（分類器の安全チェック付き）** | **長時間タスク、プロンプト疲れの解消** |
| `dontAsk` | 事前許可されたツールのみ | CI/CD、スクリプト実行 |
| `bypassPermissions` | すべて（チェックなし） | コンテナ・VM 等の隔離環境専用 |

auto mode は **「安全性を維持しつつ、作業の流れを最大化する」** ゾーンに位置します。`bypassPermissions` のように全チェックを無効化するのではなく、分類器という安全網を張ったうえで許可プロンプトを省略するのが特徴です。

モードの切り替えは **`Shift+Tab`** でサイクルできます。CLI 起動時に指定する場合は `--permission-mode auto` です。

## auto mode の仕組み — 分類器の判定フロー

auto mode の核心は**分類器**です。Claude 本体とは別のモデルが、すべてのツール呼び出しを 4 段階で評価します。

### 判定フロー

```
1. deny / ask ルール → マッチしたら即ブロック or プロンプト
2. 読み取り専用 or WD 内のファイル編集 → 自動承認（protected paths 除く）
3. 分類器が評価 → 安全なら実行、危険ならブロック
4. ブロック時 → Claude に理由を返し、代替手段を試みる
```

重要なのは、`permissions.deny` や明示的な `ask` ルールは**分類器より前に評価される**点です。つまり、deny ルールでブロックした操作は auto mode でも絶対に通りません。

### デフォルトでブロックされるもの

| カテゴリ | 具体例 |
| --- | --- |
| 外部コードの取得・実行 | `curl | bash`、ダウンロードしたスクリプトの実行 |
| 機密データの外部送信 | API キーや環境変数を外部エンドポイントに送る |
| 本番環境への操作 | デプロイ、マイグレーション |
| クラウドストレージの大量削除 | S3 バケットの一括削除 |
| IAM / リポジトリ権限の変更 | 権限付与、アクセス制御の変更 |
| 不可逆なファイル破壊 | セッション開始前から存在するファイルの削除 |
| Git の危険操作 | `git push --force`、`main` への直接 push |

### デフォルトで許可されるもの

| カテゴリ | 具体例 |
| --- | --- |
| ローカルファイル操作 | 作業ディレクトリ内の読み書き |
| 依存パッケージのインストール | lock ファイルやマニフェストに基づくもの |
| 認証情報の正当な利用 | `.env` の読み取りと対応 API への送信 |
| 読み取り専用の HTTP リクエスト | GET リクエスト |
| ブランチへの push | 開始時のブランチ or Claude が作成したブランチ |

### サブエージェントへの適用

auto mode はサブエージェントにも 3 段階のチェックを適用します。

1. **起動前** — タスク内容を評価し、危険なタスクはスポーン自体をブロック
2. **実行中** — 各アクションを親セッションと同じルールでチェック
3. **完了時** — 全アクション履歴をレビューし、問題があればセキュリティ警告を付与

サブエージェントの frontmatter で `permissionMode` を指定しても、auto mode では無視されます。

## セットアップ

### 利用要件

auto mode を使うには以下の条件を満たす必要があります。

| 要件 | 詳細 |
| --- | --- |
| バージョン | Claude Code v2.1.83 以降 |
| プラン | 全プランで利用可能 |
| モデル | Anthropic API: Opus 4.6 以降、Sonnet 4.6 / Bedrock・Vertex AI・Foundry: Opus 4.7 以降のみ |
| 管理者設定 | Team・Enterprise プランでは管理者による有効化が必要 |

### Step 1: auto mode の有効化

**CLI から起動する場合:**

```
claude --permission-mode auto
```

**セッション中に切り替える場合:**

`Shift+Tab` を数回押すと auto mode に到達します。初回は確認プロンプトが表示されます。

**デフォルトにする場合:**

~/.claude/settings.json

```
{
  "permissions": {
    "defaultMode": "auto"
  }
}
```

### Step 2: Bedrock / Vertex AI / Foundry の場合

これらのプロバイダでは、追加の環境変数が必要です。

~/.claude/settings.json

```
{
  "env": {
    "CLAUDE_CODE_ENABLE_AUTO_MODE": "1"
  }
}
```

設定後、`Shift+Tab` のサイクルに auto mode が追加されます。

## 自組織に合わせた分類器設定

デフォルトの分類器は**作業ディレクトリとリポジトリの設定済みリモート**のみを信頼します。つまり、社内の GitHub org への push や社内 API への通信も、設定なしではブロックされる可能性があります。

### `autoMode.environment` で信頼インフラを宣言する

`autoMode.environment` に信頼するインフラを自然言語で記述します。正規表現やツールパターンではなく、**新しいエンジニアにインフラを説明するように書く**のがコツです。

~/.claude/settings.json

```
{
  "autoMode": {
    "environment": [
      "$defaults",
      "ソース管理: github.com/our-company 配下の全リポジトリ",
      "信頼するクラウドバケット: s3://our-build-artifacts",
      "信頼する内部ドメイン: *.internal.our-company.com",
      "主要サービス: Jenkins (ci.our-company.com), Artifactory (artifacts.our-company.com)"
    ]
  }
}
```

### ルール階層：`hard_deny` / `soft_deny` / `allow`

分類器内部では 4 段階の優先順位でルールが評価されます。

| 優先度 | ルール | 動作 | ユーザーの明示的意図で上書き |
| --- | --- | --- | --- |
| 1 | `hard_deny` | 無条件ブロック | 不可 |
| 2 | `soft_deny` | ブロック | 可能（`allow` ルールまたはユーザーの明示的指示） |
| 3 | `allow` | `soft_deny` の例外として許可 | — |
| 4 | ユーザーの明示的意図 | 残りの soft block を上書き | — |

「明示的意図」とは、ユーザーが**具体的なアクションを直接指示した場合**です。「リポジトリを整理して」のような漠然とした指示では force push は許可されませんが、「このブランチを force push して」と明示すれば `soft_deny` を上書きできます。

設定例:

~/.claude/settings.json

```
{
  "autoMode": {
    "environment": [
      "$defaults",
      "ソース管理: github.com/our-company 配下の全リポジトリ"
    ],
    "allow": [
      "$defaults",
      "ステージング環境へのデプロイは許可: 本番から隔離されており毎晩リセットされる"
    ],
    "soft_deny": [
      "$defaults",
      "マイグレーション CLI 以外でのDB マイグレーション実行を禁止",
      "infra/terraform/prod/ 配下のファイル変更を禁止: 本番インフラの変更はレビューワークフロー経由"
    ],
    "hard_deny": [
      "$defaults",
      "リポジトリの内容をサードパーティのコードレビュー API に送信しない"
    ]
  }
}
```

### CLI で設定を検査する

設定が意図通りか確認するための 3 つのコマンドがあります。

| コマンド | 用途 |
| --- | --- |
| `claude auto-mode defaults` | 組み込みのデフォルトルールを JSON で表示 |
| `claude auto-mode config` | `$defaults` を展開した実効設定を JSON で表示 |
| `claude auto-mode critique` | カスタムルールに対する AI フィードバックを取得 |

設定を変更したら `claude auto-mode config` で実効ルールを確認する習慣をつけましょう。

## ブロックされたときの対処

### `/permissions` で確認とリトライ

auto mode がアクションをブロックすると、`/permissions` の **Recently denied** タブに記録されます。`r` キーを押すとリトライを指示でき、Claude Code が会話を再開して再試行します。

### フォールバック条件

以下の条件で auto mode は一時停止し、通常の許可プロンプトに戻ります。

| 条件 | 動作 |
| --- | --- |
| **3 回連続**ブロック | 許可プロンプトに切り替え（承認すると auto mode 再開） |
| **累計 20 回**ブロック | 許可プロンプトに切り替え |

これらの閾値は変更できません。連続カウンターは許可されたアクションでリセットされますが、累計カウンターはセッション中保持されます。

繰り返しブロックされる場合は、分類器にコンテキストが不足しているサインです。`autoMode.environment` に対象のインフラ情報を追加してください。

### 会話中の境界宣言

auto mode は会話中の制約も認識します。「push しないで」「デプロイ前にレビューさせて」と伝えると、分類器がそのアクションをブロックするようになります。

ただし注意点が 2 つあります。

* 境界は**ルールとして保存されるわけではない**。分類器がトランスクリプトから毎回読み取る
* コンテキスト圧縮で境界を記述したメッセージが消える可能性がある。確実にブロックしたい場合は `permissions.deny` ルールを使う

## auto mode vs 他のアプローチ：使い分けガイド

| 比較軸 | auto mode | acceptEdits | sandbox | bypassPermissions |
| --- | --- | --- | --- | --- |
| ファイル編集 | 自動承認 | 自動承認 | 制限可能 | 自動承認 |
| Bash コマンド | 分類器が判定 | 毎回プロンプト | サンドボックス内で自動実行 | 自動承認 |
| 外部通信 | 分類器が判定 | 毎回プロンプト | ドメイン制限 | 自動承認 |
| 安全性の担保 | 分類器 + deny ルール | ユーザー判断 | OS レベルの隔離 | なし |
| 推奨環境 | 日常開発全般 | コード編集中心の作業 | 信頼できないコンテンツ | コンテナ・VM のみ |

**推奨シナリオ:**

* **個人開発で集中したい** → auto mode を `$defaults` のまま使い始める
* **チームでの利用** → マネージド設定で `autoMode.environment` を配布し、`hard_deny` で組織ポリシーを強制
* **CI/CD パイプライン** → `dontAsk` で許可リストを明示するか、コンテナ内で `bypassPermissions`
* **コードを書かせつつレビューしたい** → `acceptEdits` で diff 確認

## まとめ

auto mode は「許可プロンプトの壁」と「安全性」の間にある**最適バランス**を提供します。

* 分類器が**安全なアクションを自動承認**し、**危険な操作だけをブロック**する
* `autoMode.environment` で**信頼するインフラを宣言**すれば、社内リソースへのアクセスもスムーズに
* `$defaults` を含めることで**組み込みの安全ルールを維持**しつつカスタマイズ可能
* ブロック時は `/permissions` で確認し、必要なら `environment` に追加する**段階的アプローチ**がおすすめ

まずは `claude --permission-mode auto` で試してみてください。許可プロンプトの数が激減し、Claude Code との作業が別物に感じられるはずです。
