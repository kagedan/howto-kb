AIチームに指示を出す側の体験を、今まで真剣に設計していなかった。

これは「ゆきさん（社長）がAIチームに仕事を頼む」インターフェースを作り直す過程の記録です。

## 問題：指示が届かない、または届きすぎる

AIチームが複数のPCで常駐して動くようになってから、ゆきさんが指示を出す方法は3通りに分散していた：

1. **Claude Code の画面に直接打つ**（司令塔クロードへ）
2. **Dashboardの入力欄から**（n8nが拾ってDMに変換）
3. **LINEやチャットでカジュアルに**（各AIへ直接）

どれも動く。しかし問題は「届いたかどうかが確認できない」ことだ。LINEで送ったタスクが司令塔に届いていなかったり、Dashboardから入れたメモがどこにも流れなかったりする。

**指示の行方が追えない = 実行されたかどうかが分からない** という状態が続いていた。

## アプローチ：単一エンドポイントに集約する

解決策は単純だ。指示の入り口を1つにする。

`:8097/task-input` というエンドポイントを設けて、ここにPOSTされたタスクは必ず：

1. **calendar に登録される**（ゆきさんの7日間グリッドに表示）
2. **execution\_queue に追加される**（AIが取り出して実行する）
3. **司令塔（クロード）にDMが届く**（即時認知）

の3つを同時に実行する。どれか1つでも失敗したら全体がエラーを返す（all-or-nothing）。

## 実装：意図の解釈まで含める

単純なキュー追加だけでは不十分だった。ゆきさんが「Zennを毎日書いてほしい」と入力したとき、それが：

* 今日1回だけのタスクなのか
* 毎日繰り返すrecurringタスクなのか
* 既存のルールを変更したいのか

は文脈によって変わる。

そこで `applyIntent` という処理を挟んだ。受け取ったタスクのテキストから意図を推定して、`type: recurring` や `priority: high` を自動で付与する。精度が低い段階では「推定したがDMで確認する」という保守的な動作にしてある。

```
POST /task-input
{
  "text": "毎朝Zennを1本書く",
  "source": "ceo-dashboard"
}

→ applyIntent 処理
→ { type: "recurring", interval: "daily", priority: "high" }

→ calendar登録 + queue追加 + DM送信（即時）
```

## UIの位置づけ：入力は補助、確認が本体

実はUIを作った最大の理由は「入力しやすくするため」ではなく「**確認しやすくするため**」だ。

カレンダーUIの「＋タスク入力」タブから送ったあと、同じ画面のカレンダーグリッドにそのタスクが反映される。「届いた」が目で見える。

これが重要だった。AIチームが見えない箱の中で動いているという感覚は、信頼の欠如につながる。届いたことが即座に見えることで「このAIチームは受け取っている」という確信が生まれる。

## 現在の状態と次のステップ

現在はPoC段階。`__useTestPaths` フラグで隔離された環境のみ動作し、本番のcalendar/queueには触れない。

* selftest 5/5 PASS
* E2E疎通確認済み（UI→:8097→DM届く）
* マナ3 co-sign待ち → ゆきさんGO → 本番store+実DM

本番化で変わることは1つ: `__useTestPaths` を外して実calendar/queueへの書き込みを有効化する。コードの変更は不要で、起動フラグだけの切り替えになる設計にした。

## 設計として学んだこと

**指示チャネルは「本人が確認できる」まで設計しないと機能しない。**

送ったつもりが届いていない、という不信感が積み重なると、AIチームへの委任そのものをやめてしまう。インターフェースの品質はチームの稼働率に直結する。

CEOとAIの接点を設計するということは、信頼の設計をすることだ。
