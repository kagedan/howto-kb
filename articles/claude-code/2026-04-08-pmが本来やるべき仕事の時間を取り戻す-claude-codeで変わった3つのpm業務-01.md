## 1. はじめに

**Claude Code、使っていますか？**

「エンジニア向けのツールでしょ？」と思っていた時期が、私にもありました。  
でも実際に触ってみて気づいたんです。**私にとってのClaude Codeは、エンジニア向けツールというより、PM業務の前後にある“重い作業”を減らすための道具** だったと。

半信半疑で使い始めてみたら、PM（プロダクトマネージャー）の仕事が **質も速度も** 変わりました。

この記事では、PMの私がClaude Codeを使いこなして変わった **3つの仕事** を紹介します。  
目指すのは「速いだけ」のAI活用ではなく、**質と速度を両立しながら、PM本来の仕事に時間を取り戻す** ワークフローです。

---

## 2. Claude Codeとは

**Claude Code**は、Anthropic（Claude開発元）公式のAIアシスタントツールです。  
ターミナルで動きますが、**自分でコードを書かなくても、自然言語で指示するだけで Claude がコードや文書を読み書きしてくれます**。

ChatGPTやGeminiにも Projects やワークスペース連携など、似た機能はあります。  
そのうえで Claude Code の特徴をひとことで言うなら、**「ローカルの作業フォルダを起点に、ファイル・設定・ツール連携をまとめて扱いやすい」** こと。  
私の感覚では、**「プロジェクトの情報を把握した、記憶力のある優秀なチーム」** をAIで構築できるツール、というのが一番近い表現です。

**より詳しい概要**: [Claude Code 公式ドキュメント（日本語）](https://code.claude.com/docs/ja/overview)

## 3. PMが時間を奪われる「3つの仕事」

PMの時間を奪う仕事は、突き詰めると以下の3つに集約されます。  
**この記事では、この3つを軸にClaude Codeでどう変えたかを紹介します。**

| 仕事 | 時間泥棒の正体 |
| --- | --- |
| **① 要件定義・仕様化** | 白紙からの執筆、漏れチェック、用語統一 |
| **② 情報整理・調査** | システム横断の検索、過去経緯の再構築 |
| **③ 報告・共有資料作り** | 構成作り、初稿の執筆、見栄え整え |

**原則：「Why と判断」はPM、「How と作業」はAI。**

PMの仕事は変わりません。変わるのは「**前後の作業時間**」と「**情報の集まり方**」です。

## 4. PMはWhyを伝えるだけ — Claudeが自動でチームを編成する

Claude Codeには、PMが自分専用のチームを組むための機能が大きく3つあります。  
会社組織にたとえると、こんなイメージです：

| 機能 | 会社にたとえると | 何をするもの | 具体例 |
| --- | --- | --- | --- |
| **Skills**（旧 Custom Commands を含む） | 業務マニュアル | 「この種の仕事はこう進める」という手順書をAIに渡しておくもの。`/skill-name` で呼び出せる形で型化できる。Claudeが必要な時に自動で読み込んで参照する | 「Design Docを書くスキル」「議事録要約スキル」、`/weekly-review` |
| **Sub Agents** | 専門職の部下 | 特定領域に特化したAIエージェント。それぞれが独立した文脈で動き、専門の仕事を任せられる | 調査担当の `research-analyst`、レビュー担当の `critical-reviewer` |
| **Plugins / 外部連携** | 外部ベンダー／既製パッケージ | 既製の業務セットやツール連携を取り込みやすくする仕組み | 公式・コミュニティの各種プラグイン |

これらは **自分で作ることもできるし、公式・サードパーティが提供している既製のものを入れることもできます**。  
業務マニュアルを社内で書くか、市販のマニュアルを買うか——という発想に近いです。

そして一番驚いたのは、**適切に設定しておけば、Claudeが依頼内容に応じて関連する Skill や Sub Agent を使い分けてくれる** ことです。

たとえば「この機能のDesign Docを書いて」と頼むと、ClaudeはDesign Doc作成スキルを呼び出し、必要に応じて research-analyst エージェントに論文調査を依頼し、最後に critical-reviewer エージェントにレビューさせる——という流れを、Claude側で組み立ててくれます（もちろん完全自律ではなく、設定・権限・文脈整備が前提です）。

PMはあくまで **「何をしたいか（Why）」を伝えるだけ**。  
誰に何をやらせるかは、Claude がだいぶ引き受けてくれます。これがPMにとってのClaude Codeの一番の魅力かもしれません。

> 各機能の詳細は、登場するセクション内で都度補足します。まずは「Skills や Agents を登録しておくと、Claudeがそれを呼び出して使ってくれる」とだけ覚えておけばOKです。

**Skillの作り方を知りたい方へ**: [Claude Code Skills の作り方（Zenn / tmasuyama1114）](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/skills-creation) — SKILL.md の書き方や references フォルダの活用まで分かりやすくまとまっています。

## 5. 仕事①：要件定義・仕様化

PMの一丁目一番地です。業界でよく言われる経験則として「品質は要件定義で大きく決まる」と語られるほど、ここをサボると後工程すべてが歪みます。

私は自作の **Design Doc 作成スキル** を使っています。  
このスキルを呼び出すと、AIが以下を自動実行します：

1. **ヒアリング**：背景・課題・目的・スコープ・非機能要件を対話的に確認
2. **テンプレート適用**：Design Doc の定型フォーマットに沿って叩き台を生成
3. **セルフレビュー**：要件の漏れ・矛盾、記述の一貫性を自動チェック

「白紙から書く」のと「8割完成した叩き台を編集する」のでは、負担が全く違います。

**Design Doc そのものについては**: [プロダクトマネージャーの必須スキル：デザインドックの書き方（note / kosuke mori）](https://note.com/kosukemori/n/n968cd16c53eb) が PM 視点でよくまとまっています。

### 論文のエビデンス調査もAIに下調べさせる

IoT領域では、要件の裏付けに **論文や業界レポート** を当たりたい場面が頻繁にあります。  
「この方式は本当に効果があるのか」「業界の最新研究ではどう扱われているのか」——こうした問いに対して、私は **research-analyst エージェント** に下調べを依頼しています。

関連論文の候補を洗い出し、要約し、Design Doc に組み込みやすい形に整理してくれる。  
ただし、**最終的な採用判断と一次情報の裏取りはPM側で行う** のが鉄則です。AIは下調べを爆速にしてくれますが、出てきた情報をそのまま信じるのは危険。**PMが一次情報にアクセスしやすくなり、要件の説得力が一段上がる** ——これがDesign Docの品質に直結しました。

### Before / After（私のケース）

以下は私の手元で実感している変化です。**チームのテンプレートと前提情報が揃っていれば** ここまで縮みますが、ゼロから整える場合は当然もっとかかります。

| 観点 | Before | After |
| --- | --- | --- |
| 初稿作成 | 白紙から書く（半日〜1日） | テンプレート＋AI生成（1〜2時間） |
| 網羅性チェック | 自分で漏れを確認 | AIが観点別に自動チェック |
| エビデンス調査 | 論文・規格を自分で当たる | research-analyst エージェントが要約 |

> Claude Code には公式・コミュニティ双方のプラグインやスキルがあり、PRD作成系の既製品も活用できます。**自作スキルと組み合わせて使う** こともでき、たとえば既製品で初期PRDを作り、自作スキルで社内テンプレに整形する、といった使い方も可能です。

## 6. 仕事②：情報整理・調査

PMが時間を奪われている2つ目の仕事です。

「先月の決定事項ってどこに書いてあったっけ？」  
「あのお客さんの要望、Slackのどのチャンネルだっけ？」  
「先週の数値どうだった？Amplitudeで見れば分かるけど、開くのが面倒…」

**情報を集めるだけで半日が溶けていく** ——これがPMの現実です。

### 6-1. システム連携で「探す手間」が消える

Claude Code は **MCP（Model Context Protocol：Claudeが外部システムと連携するための標準プロトコル）** やプラグインで、外部システムと直接連携できます。  
**MCPの仕組みを詳しく知りたい方は**: [Model Context Protocol（MCP）とは？（Zenn / Cloud Ace）](https://zenn.dev/cloud_ace/articles/model-context-protocol)  
私が連携しているのは以下の4つです：

| システム | 用途 | MCPサーバー |
| --- | --- | --- |
| **Slack** | チャンネルのトーク履歴、決定事項抽出 | [Slack公式](https://slack.com/intl/ja-jp/help/articles/48855576908307-Slack-MCP-%E3%82%B5%E3%83%BC%E3%83%90%E3%83%BC%E3%81%AE%E3%82%AC%E3%82%A4%E3%83%89) |
| **Google Drive** | 過去の議事録・仕様書を横断検索 | サードパーティ／コミュニティ実装を利用 |
| **Amplitude** | プロダクト数値の取得 | [Amplitude公式](https://amplitude.com/docs/amplitude-ai/amplitude-mcp) |
| **GitHub** | Issue / PR / バックログの進捗確認 | [GitHub公式](https://github.com/github/github-mcp-server) |

これらに対して、Claude Codeにこう聞くだけです：

```
先月の#product-xxxチャンネルで決まった仕様変更を一覧化して。
関連するGitHub Issueと、Amplitudeの該当指標もまとめて。
```

条件が揃えば、**4つのシステムから集めた情報が、数分〜十数分で1つのレポートに整理されて返ってきます**（環境やデータ量によって差はあります）。  
人間が手で集めれば半日かかる作業です。

> なお、サードパーティ製のMCPサーバーは便利な一方で、セキュリティや prompt injection のリスクもあります。公式ドキュメントでも注意喚起されているので、**信頼できるものだけを導入する** のが前提です。

**セキュリティを深く理解したい方へ**:

### 6-2. スケジューラーで「毎朝勝手にまとまっている」状態を作る

さらに強力なのが **スケジューラーによる定期実行** です。  
定期実行には複数の選択肢があります。CLI内の軽量な繰り返し実行は7日で自動失効しますが、**Cloud scheduled tasks** や **Desktop scheduled tasks** を使えば、PCがオフでも継続するような運用も可能です（詳細は [Run prompts on a schedule（公式）](https://code.claude.com/docs/en/scheduled-tasks)）。私はローカルファイルや既存運用との親和性を重視して、macOS標準の **launchd** を使っています（Linuxならcron、Windowsならタスクスケジューラが同等の選択肢）。

走らせている例：

* **議事録の取り込み**：定例MTGの議事録を自動で要約・決定事項抽出
* **Slackの会話ダイジェスト**：主要チャンネルの議論を定期的に要約
* **バックログ進捗のレビュー**：GitHub/Linear のタスクから停滞・遅延を検出

## 7. 仕事③：報告・共有資料作り

要件定義書、顧客向け提案書、月次MTG資料、ステークホルダー報告——どれも構成を考え、初稿を書き、見栄えを整える作業に時間が溶けていきます。

ここで効いてくるのが **仕事①と②で蓄積されたコンテキスト** です。  
要件と背景がドキュメント化され、関連する議論・数値・進捗が日々まとまっている状態で「○○の資料を作って」と指示すると、**目的を明確に伝えるだけで、質の良い叩き台が高速で出てきます**。

私が一番効果を感じているのは、**社内報告資料** と **顧客向け提案書** です。  
特に社内向け報告資料では、Slack/Drive の議論履歴、Amplitude の利用データ、GitHub のバックログ進捗といった **②で集めた情報** を踏まえた上で目的を伝えると、現場の実態を反映した叩き台が数十分で出てきます。

PMは「**メッセージを磨き、伝える順番を決め、固有の意思決定材料を加える**」ことに集中できる。**日常的な情報整理が、報告資料の質に直結する** ——これが3つの仕事をつなげて回す効果です。

### Before / After

| 観点 | Before | After |
| --- | --- | --- |
| 構成検討 | 白紙から構成を考える（数時間） | コンテキストを渡して目的を伝えるだけ（数分） |
| 情報集め | 関係者ヒアリング・過去資料探し（数時間） | ②で蓄積済みの情報を流用（即時） |
| 初稿執筆 | 半日〜1日 | 数十分で叩き台 |
| PMの集中先 | 構成・情報集め・執筆 | **メッセージの磨き込みと意思決定** |

## 8. 失敗から学んだ、PMがやるべき3つのこと

最初のAI生成ドキュメントは、形だけ整って中身スカスカでした。  
セクションは綺麗に並んでいるのに、中身は **どのプロジェクトでも通用しそうな一般論** ばかり。プロジェクト固有の制約も、過去のステークホルダー合意も反映されておれないことも多々ありました。さらに、引用された数値が **裏取りなしの“もっともらしい数字”** だった、という危ない場面もありました。

**AIは「それっぽい文書」を作るのは得意。でも「本当に必要な要件」を判断するのはPMの仕事。** この失敗から学んだのが次の3つです。

1. **「Why」と目的を明確にする** — 誰のどんな課題か、成功基準は何か、やらないことは何か。**PMの意思決定の質 ＝ AIの成果物の質。**
2. **コンテキストを集約する** — フォルダに集める＋外部システムと連携する。まずは議事録と仕様書だけでも十分。
3. **Skills と Sub Agents を磨き続ける** — 指摘された観点をテンプレに追加。**PMの暗黙知が形式知に変換され、組織の知見がツールに蓄積されていきます。**

## 9. 始め方 — まずは手を動かして触ってみよう

claude codeの始め方に関しては、公式ドキュメントを参照ください。  
<https://code.claude.com/docs/ja/quickstart>

### まず質問してみる

いきなりDesign Docを作らなくてOK。  
プロジェクトフォルダで起動し、**「このフォルダの議事録から先月の決定事項をまとめて」** と聞いてみてください。  
**フォルダ内のファイルを読んで回答してくれる体験** が、最初の「おお」になるはずです。

慣れてきたら、よく使うテンプレートを Skills に登録し、専門エージェントを追加し、必要ならプラグインも試していく。**段階的にチームを育てていくイメージ** です。

> **セキュリティについて**: 機密情報を扱う場合は、Claude Codeの **権限設定や組織ポリシー** を確認したうえで利用してください。特に **外部ツール連携やサードパーティ製MCPの利用時は注意が必要** です。詳細は公式ドキュメントを参照してください。

## 10. おわりに — その先の仕事に時間を取り戻す

> **「速いだけ」のAI活用は終わり。PMの仕事は、質と速度を両立できる時代に入った。**

PMの仕事は変わりません。**「Why」を決め、「判断」し、「責任を持つ」。**  
変わったのは、その前後を任せられる **AIエージェントのチーム** が手に入ったことです。

### そして、その先にある仕事

要件定義・情報整理・報告資料は、紛れもなくPMの中核業務です。  
ただ、これらのスピードと質が上がると、**「その先の仕事」に使える時間が一気に増えてきます**。

* **価値ある機能を見極めること**
* **プロダクトのビジョンを描き、磨き続けること**
* **顧客と深く対話し、まだ言葉になっていない課題を発見すること**
* **チームに「なぜこれを作るのか」を語り、納得させること**

これらは **AIには代替できない、PMにしかできない仕事** です。  
そしてこれまでのPMは、ドキュメント作りや情報集めに追われ、**この「その先の仕事」にかける時間が常に足りませんでした**。

雑務の質と速度を上げ、空いた時間で顧客と話し、プロダクトの未来を考える——Claude Code で取り戻せるのは、単なる「時間」ではなく、**PMがPMにしかできない仕事に集中できる状態** だと思います。

今後3つの具体的な3つの業務を削減した詳細解説記事なども書いていく予定です。
