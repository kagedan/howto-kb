## 概要

MastraのCo-FounderでCPOのShane Thomasと、Developer ExperienceのAlexが登壇したロードマップウェビナーの記録です。「ロードマップを公開しているフレームワークはそんなに多くない」とShane自身も認めながら、それでもオープンソースコミュニティに対して透明性を持って向き合うために開催したこのセッションは、単なる機能紹介にとどまらず、Mastraが「エージェントとは何か」「どう作るべきか」という思想をどう深化させているかが伝わってくる内容でした。

前半は2026年2〜4月に出荷されてきた機能の振り返り。後半は現在開発中・近日リリース予定の機能群、そして最後に「Mastra Agent Builder」という新しいプロダクトの発表という構成になっています。参加者からのチャット質問も随所に挟まれ、ライブ感のある60分でした。

## 詳細

### 2〜4月の怒涛のリリース振り返り

ウェビナーの前半は、この2〜3ヶ月で出荷してきたものをひとつひとつ振り返る時間でした。

**Observational Memory**（2月）は、LongMemEvalで95%というスコアを記録した人間インスパイアのメモリーシステムです。オブザーバーエージェントとリフレクターエージェントがメッセージ履歴を監視し、観察と内省を行いながら記憶を凝縮・整理していく。安定したコンテキストウィンドウを持つためプロンプトキャッシュとの親和性が高く、「数行追加するだけでベストインクラスのメモリーが手に入る」というのが売り文句です。

**Mastra Code**（2月）は、もともとObservational Memoryのテストベッドとして作られたコーディングエージェントです。これが気づけばMastraチーム全員の日常ツールになり、今では数百人の外部ユーザーも使っているオープンソースプロジェクトになっています。Claude Codeに不満がある人はぜひ試してほしい、という軽めの売り込みもありました。

**Workspaces**（2月）は、エージェントにファイルシステムやシェル環境を持たせるためのもの。スキルをダウンロードして付与することもでき、「エージェントが環境を持つ」というコンセプトの重要な基盤です。

**Experiments**（3月）は、データセットをエクスペリメントとして実行してスコアをトラッキングできる機能。エージェントが「動く」ことと「品質が良い」ことは全然別問題であり、後者に掛ける時間は前者の数倍になることが多い——エージェント開発者なら身に沁みる話ではないでしょうか。

**Studio Auth**（3月）・**Metrics & Logs**（4月）・**Agent Editor**（4月）・**Browser Support**（4月）と続きます。特に**Agent Editor**の背景にある思想は面白い。医療エージェントを作っているチームで「エンジニアは医師ではない」、航空エージェントのチームで「エンジニアはパイロットではない」——主題領域の専門家が自分でシステムプロンプトを書き換え、ツール説明を調整し、モデルを変えて試行できる環境を作ったのがこの機能です。コードを書かないドメインエキスパートが開発ループに入れるようにする、という発想は今後のエージェント開発の方向性をよく表しています。

そして**Mastra Platform**——Studio・Server・Memory Gatewayの3点セット。Memory Gatewayは「Mastraを使っていなくても、LLM呼び出しをMemory Gatewayに向けるだけでObservational Memoryを使える」という設計で、Python勢にとってはかなり嬉しい選択肢になります。

### 長時間稼働するエージェント（Long-Running Agents）

ここからが「近い将来リリースするもの」の話です。Shaneはまず大前提を語りました。「モデルは改善し続ける。その前提に立つと、エージェントはより長い時間軸で動作できるようになる。だから今からそれに備えておく必要がある。」

**Background Agents**（5月上旬リリース予定）。エージェントがバックグラウンドでサブタスクを実行し、完了したら返ってくる仕組みです。ディープリサーチやコーディングエージェントなど、「長くて重いタスクを並列で流したい」場面にフィットします。

**Durable Agents**。「1時間動かしているエージェントのウィンドウを閉じたら、続きが見られなくなる」という問題を解決します。ストリームを一時停止・再開できるようにし、切断を乗り越えて続きから確認できる。当たり前のようでまだほとんどのフレームワークが対応していない機能です。

**Dynamic Sub-Agents**。スーパーバイザーパターンとの違いをShaneはこう説明しました——スーパーバイザーパターンでは人間がエージェントの構成を決める。ダイナミックサブエージェントでは、エージェント自身が新しいエージェントを定義し、ツールを選び、命令を書き、モデルを選択する自律性を持つ。「現時点では少しニッチなユースケースだが、モデルが進化すればここに面白い可能性がある」という言い方でした。ざっくりとしたガードレールだけ設定して、あとはエージェントに任せる——そういう使い方が現実的になる日が来るかもしれません。

**Temporal Workflows**サポートも数週間以内。IngestはすでにサポートされていてTemporalも多くのリクエストがあったため対応。PRは開いてあってテスト・検証中とのこと。

### エージェント同士が話す時代へ（A2A & Agents Love Mastra）

大規模なチームからは「複数チームが別々のエージェントを作っていて、そのうち連携が必要になる」という声が増えています。**A2A（Agent-to-Agent）サポートの強化**がその答えです。仕様が変わったためゼロから見直して改善中。本番で使っている人はまだ少ないが関心は高く、Mastraで作ったエージェントが他ツールのエージェントと通信できる体制を整えます。

もう一つ面白いのが**Mastra Studio用CLI**の話。「エンジニアのためではなく、コーディングエージェントのために作るCLI」という発想がポイントです。Mastra ServerのAPIに対応したCLIを使えば、コーディングエージェントがエンドツーエンドのライフサイクルを自律的にテストできる。エージェントが自分のエージェントをテストする、という「エージェントが自己開発に参加する」絵図が見えてきます。

---

### OpenClawの哲学をエンタープライズへ（File System Providers / Channels / Heartbeats）

OpenClawが多くの人に「エージェントの可能性」を見せましたが、大企業がそのまま導入するのはなかなか難しい。そこでShaneが言ったのは「OpenClawのアイデアから学びつつ、エンタープライズが使いやすい形に落とし込む」ということです。

**File System Providers**では、S3・GCS・Azure・AgentFS・Google Drive（近日）を使ってエージェントがリモートファイルシステムにアクセスできます。「エージェントにS3のバケットを指定すれば、あなたが使っているファイルと同じものをエージェントも使える」というシンプルな発想です。

**Channels**は、Slack・Discord・Telegramなどどこでもエージェントと会話・操作できるようにします。技術的にはすでに一部実装されていて、まもなく正式アナウンス予定。

**Heartbeats**は「スケジュールタスク＋コンテキスト認識」。クロンジョブのようなものですが、前回の実行内容を覚えていて、プロンプトと記憶を持って自律的に動く。これが揃うと、エージェントが人間に頼まれなくても自分から世界に働きかけるような動作が現実的になります。

---

### Observational Memory Retrievalという設計の妙

Shaneは自分で作ったメールエージェントの話をしました。約1万5千通のメールを1つのスレッドで処理するエージェントで、メール分類の好みや文体のスタイルは覚えているが、長期的な会話の中で細かいディテールが徐々に失われていく——これはObservational Memoryの限界でもありました。

その解決策として開発中なのが**Observational Memory Retrieval**です。これはエージェントが必要なときにだけ「過去のメッセージの全詳細」をデータベースから引き出せるツールです。

ここで重要なのが「ツールとして提供する」という設計思想です。多くのメモリーシステムでは「常時セマンティック検索で似たものを引いてくる」アプローチを取りますが、それをすると毎回コンテキストウィンドウが変わってプロンプトキャッシュが壊れてしまいます。Observational Memory Retrievalはエージェント自身が「今これが必要だ」と判断した時だけ取得するため、コンテキストの大部分をキャッシュ可能な状態に保てる。実質的な「無限のメモリー」をコスト効率良く実現する設計です。

### 可観測性の深化——専門家をループに引き込む

**アノテーション＆レビュー**は、専門家がトレースに注釈をつけてレビューキューを作れる機能です。「エンジニアは医師ではないし、パイロットでもない」というAgent Editorの文脈の延長で、主題領域の専門家がエージェントの挙動を評価・フィードバックできる仕組みを整えます。フラグ付きトレース→専門家レビュー→データセット追加→エクスペリメント実行、というフィードバックループが完成すれば、エージェントの継続的な品質改善が仕組みとして回ります。

**アラーティングと通知**（6月初旬予定）。エバリュエーションスコアの低下、トークン消費の急増、モデル性能の劣化——こういった異常を検知して通知を送れるようにします。本番で動かしているエージェントが「いつの間にかおかしくなっていた」という事態を防ぐために不可欠な機能です。

### Mastra Agent Builder——最大の発表

セッション最後のパートが、ある意味最もインパクトのある発表でした。

Shaneが繰り返し聞いてきた声として紹介したのは、「エンジニアリングチームが社内の他部署から『私たちにもエージェントを使わせてくれ』と圧力をかけられている」という状況です。OpenClawが話題になるたびに「私たちにもあれがほしい」と言われる。でもコードは書けない。では誰が作るのか？

**Mastra Agent Builder**は、エンジニアが「コントロールされた環境」を用意し、非エンジニアが「その中で自分のエージェントを作れる」プラットフォームです。

* エンジニアがツール・MCP接続・インテグレーション・ワークスペース・スキルを設定する
* RBACで誰が何にアクセスできるかを制御し、監査ログも確保する
* エンドユーザーは自然言語でエージェントを構築できる
* 自分のエージェントやスキルを社内の共有ライブラリに公開し、チームで発見・再利用できる
* Slack連携でワンクリックインストール——作ったエージェントをチャンネルに即投入

「エンジニアが王国を守りながら、他の人が遊び場で楽しめるようにする」とShaneは表現しました。Composio・Arcadeなどのインテグレーションプロバイダーとの連携も計画中。

デモ動画も少し公開されましたが、自然言語でエージェントを設定していく様子が確認できました。チャット欄には「最後の2ヶ月やってたことが全部あるじゃないか」というコメントが飛んでいたほど。

現時点はデザインパートナーと早期アクセスユーザー向けの段階ですが、興味があればコンタクトフォームから連絡すれば話を聞いてもらえるとのことです。

### UIインテグレーション（AGUI / CopilotKit）

「エンドユーザー向けUIはどう考えているか」という質問へのShaneの回答は明確でした。「Mastraは意図的にUIコンポーネントを提供しない選択をした。代わりにAPIとストリーミングを扱いやすくして、CopilotKit・Assistant UIなどのライブラリとの統合を容易にする」。

特にCopilotKitとはAGUI（Application GUI）標準への対応強化を密に進めており、「よりダイナミックなジェネレーティブUI」を作りやすくする方向で議論が進んでいます。標準への準拠と高い開発速度を両立させる、というスタンスです。

---

### Mastraの競合との比較について

「Langfuse・BrainTrust・Arizeと何が違うのか」という直球の質問に対してShaneが語ったことは印象的でした。

「機能的にはほぼ同等だが、Mastraはエージェントループ全体にアクセスできる。スタジオ上でエージェントを直接テストでき、トレースが実行環境と同じ場所にある。つまり、データを見てすぐエージェントを変更できる。二つの独立したシステムを行き来して知見を転記するのではなく、縦断的に統合されているのがMastraの強みだ。」
