# はじめに

AIエージェント周りでは、最近いろいろな言葉を見かけるようになりました。

* Prompt Engineering
* Context Engineering
* Harness Engineering
* Loop Engineering

どれも重要そうなのですが、言葉だけを追っていると、それぞれが何を指しているのか分かりにくくなります。

特に、実際にエージェントを使っていて失敗に出会ったときに迷います。

「思った通りに動いてくれない」  
「必要なファイルを読まずに判断している」  
「テストせずに完了したことにしている」  
「何度も同じ失敗を繰り返している」

こうしたとき、直すべき場所は本当にプロンプトなのでしょうか。

もしかすると、指示の書き方ではなく、見せている情報、実行環境、反復の流れに問題があるかもしれません。

この記事では、Prompt / Context / Harness / Loop という言葉をまず分けたうえで、AIエージェントの失敗を分類するための見方として整理します。

# まず用語を分ける

最初に、4つの用語を分けておきます。

どれもAIエージェントをうまく動かすための考え方ですが、見ている場所が違います。

## Prompt Engineering: 指示を設計する

Prompt Engineering は、モデルへの指示を設計することです。

何をしてほしいのか、どのような形式で返してほしいのか、何を優先してほしいのかを、依頼文として整えます。

たとえば、単に「レビューして」と頼むのではなく、次のように書くのは Prompt の改善です。

```
この変更をコードレビューしてください。
優先して見てほしいのは、バグ、仕様漏れ、テスト不足です。
スタイルの好みだけの指摘は不要です。
指摘は重要度順に並べ、ファイル名と行番号を含めてください。
```

モデルに見えている情報は足りているが、依頼の解釈がズレる。

その場合は、まず Prompt を疑います。

## Context Engineering: 見せる情報を設計する

Context Engineering は、モデルに何を見せるかを設計することです。

AIエージェントは、見えていない仕様やファイルを前提に判断できません。

プロジェクトのルール、関連ファイル、設計方針、過去のIssue、ADR、PRの議論などを、必要なタイミングで参照できるようにします。

たとえば、記事を書くときに「です・ます調で書く」「はじめにで始める」「おわりにで終える」というルールがあるなら、毎回プロンプトに書くより `AGENTS.md` などに置いた方が扱いやすくなります。

Codex の場合、`AGENTS.md` は「作業前に読む永続的な指示」として扱われます。リポジトリ全体のルールだけでなく、ディレクトリごとのルールも置けるため、プロジェクト固有の前提を渡す場所として使いやすいです。

指示は悪くなさそうなのに、前提を知らないせいで間違える。

その場合は、Context を疑います。

## Harness Engineering: 実行環境を設計する

Harness Engineering は、モデルの外側にある実行環境を設計することです。

ここでいう Harness は、エージェントを動かすための周辺システムです。

たとえば、次のようなものが含まれます。

* ツール
* ファイルシステム
* 権限
* サンドボックス
* テスト
* lint
* typecheck
* レビュー観点
* Agent Skills
* MCP
* ログやトレース

プロンプトで「ちゃんとテストして」と書くだけでは、実行環境としては弱いです。

テストを実行できること、どのコマンドを実行すべきか分かること、失敗した結果を読んで修正できること、危険な操作を止められることが必要になります。

Codex の Agent Skills も、この Harness を作るための部品として考えやすいです。

OpenAI のドキュメントでは、Agent Skills は Codex にタスク固有の能力を追加する仕組みで、手順を書いた `SKILL.md` に加えて、必要に応じて参考資料やスクリプトを同梱できるものとして説明されています。

たとえば、毎回「Zenn記事を書くときは frontmatter を確認し、見出しを整え、参考リンクを最後に置く」と指示しているなら、それはプロンプトに書き続けるより Agent Skills にした方が再利用しやすくなります。

エージェントが実行や検証をうまく扱えない。

その場合は、Harness を疑います。

## Loop Engineering: 反復の流れを設計する

Loop Engineering は、エージェントをどのように反復させるかを設計することです。

単発の指示ではなく、目的を達成するまで「実行する、観測する、修正する、再実行する」という流れを作ります。

たとえば、次のような設計です。

```
1. Issue を確認する
2. 修正方針を作る
3. 実装する
4. テストを実行する
5. 失敗したら最大3回まで修正する
6. 3回失敗したら、人間に状況を報告して止まる
7. 成功したら reviewer agent に確認させる
```

ここで重要なのは、ただ何度も実行することではありません。

どこで始めるか、何をもって成功とするか、どこで止めるか、どの時点で人間に戻すかまで含めて設計することです。

Loop Engineering をもう少し分解すると、単に「同じ処理を繰り返す」だけではありません。

Addy Osmani の記事では、Loop を作る部品として、Automations、Worktrees、Skills、Plugins / connectors、Sub-agents、State / memory が挙げられています。

ここでいう Plugins / connectors は、エージェントを外部ツールや業務システムにつなぐ部品としての意味です。

LangChain の記事では、Loop を次のような層として整理しています。

* Agent loop: モデルがツールを呼びながらタスクを進める
* Verification loop: 出力をテストやrubricで確認し、失敗したら戻す
* Event driven loop: cronやwebhookなどをきっかけに自動で起動する
* Hill climbing loop: 実行ログやtraceを見て、Harness自体を改善する

こう見ると、Loop は「何度も実行する仕組み」というより、実行、検証、起動、状態管理、改善をどうつなぐかの設計だと考えやすくなります。

エージェントが毎回人間の次の指示を待っている、または逆に止まらず走り続ける。

その場合は、Loop を疑います。

# 4つの違いを表で見る

4つの違いをまとめると、次のようになります。

| 用語 | ざっくり言うと | 設計するもの | 典型的な改善 |
| --- | --- | --- | --- |
| Prompt Engineering | 指示の設計 | どう依頼するか | 依頼文、制約、出力形式を書く |
| Context Engineering | 入力情報の設計 | 何を見せるか | 関連ファイル、仕様、ルールを渡す |
| Harness Engineering | 実行環境の設計 | どう実行・検証・制御するか | テスト、lint、権限、ツールを整える |
| Loop Engineering | 反復運用の設計 | いつ、何を、どこまで回すか | 起動条件、完了条件、停止条件を決める |

もう少し実務寄りに言うと、次のように分けられます。

| 問い | 対応するもの |
| --- | --- |
| どう言えばよいか | Prompt |
| 何を見せればよいか | Context |
| どう動かせば安全か | Harness |
| どう回し続ければよいか | Loop |

この切り分けができると、「AIエージェントが失敗した = プロンプトが悪い」と決めつけずに済みます。

ここからは、Codex を例にして考えてみます。

ただし、Codex の各部品が Prompt / Context / Harness / Loop のどれか1つにきれいに対応するわけではありません。

実際には複数の層にまたがります。

ここでは「主にどの問題を解くために使うことが多いか」という観点で整理します。

| 部品 | 主に効く場所 | 役割 |
| --- | --- | --- |
| `AGENTS.md` | Context | リポジトリのルール、コマンド、レビュー観点などを作業前に読ませる |
| Agent Skills | Context / Harness | 繰り返し使うワークフロー、参考資料、スクリプトをまとめる |
| MCP | Context / Harness | 外部ツールや共有システムの情報・操作に接続する |
| Plugins | 配布単位 | Agent Skills、MCP、アプリ連携などをまとめて配布・インストールできる形にする |
| Subagents | 役割分担 / Loopで使うことがある | 作業を専門エージェントに分け、並列に調査・検証させる |

ここで注意したいのは、1つの部品が1つの分類だけに閉じるわけではないことです。

たとえば Agent Skills は、`SKILL.md` の説明文や参考資料という意味では Context でもあります。

一方で、手順、検証コマンド、補助スクリプトをまとめて「毎回同じやり方で作業する」ための仕組みでもあるので、Harness の改善にも効きます。

この記事では、Agent Skills を「Context と Harness の間にある、再利用可能な作業手順」として扱います。

# 失敗を分類する

まず、失敗の症状から見ると次のように分類できます。

| 失敗の症状 | 疑う層 | 直すもの |
| --- | --- | --- |
| 指示の解釈がズレる | Prompt | 依頼文、制約、出力形式 |
| 必要な情報を知らない | Context | 参照ファイル、仕様、プロジェクトルール |
| 実行や検証が弱い | Harness | テスト、lint、権限、ツール、Agent Skills |
| 反復の回し方が弱い | Loop | 起動条件、完了条件、停止条件、役割分担 |

ここからは、失敗の症状ごとにもう少し詳しく見ていきます。

## 指示の解釈がズレる: Prompt の問題

次のような失敗は、Prompt の問題として扱いやすいです。

* 出力形式が期待と違う
* 作業の粒度が大きすぎる、または小さすぎる
* 優先順位を誤解している
* 「やらないでほしいこと」をやってしまう
* 結論だけほしいのに、長い説明を返してくる

この場合、モデルに必要な情報は見えているものの、依頼の形が曖昧な可能性があります。

改善するなら、次のような方向です。

* ゴールを明確にする
* 制約を書く
* 出力形式を書く
* 判断基準を書く
* やらないことを書く
* 良い例、悪い例を入れる

たとえば「調べて」ではなく、「日本語記事を優先して、タイトル、URL、公開日、切り口を表にしてください」と書くと、期待する作業に近づきます。

## 必要な情報を知らない: Context の問題

次のような失敗は、Context の問題として考えやすいです。

* プロジェクト固有のルールを知らない
* 古い仕様を前提にしている
* 関連ファイルを読まずに判断している
* 似た名前のファイルを取り違えている
* 過去の決定事項を無視している
* チーム内の暗黙知を知らない

これは指示文の問題というより、エージェントに渡っている前提情報の問題です。

改善するなら、次のような方向です。

* `AGENTS.md` にプロジェクトルールを書く
* 設計方針をドキュメント化する
* 関連ファイルを明示する
* 仕様、ADR、Issue、PR、過去の議論を参照させる
* 長い情報を必要な単位に分けて渡す
* 毎回使う前提を永続的な指示にする

毎回プロンプトに同じ説明を書いているなら、それは Context として外に出せる可能性があります。

## 実行や検証が弱い: Harness の問題

次のような失敗は、Harness の問題として考えやすいです。

* テストせずに完了したことにする
* lint や typecheck を無視する
* 危険なコマンドを実行しそうになる
* 同じミスを何度も繰り返す
* レビュー観点が毎回ブレる
* 外部ツールにアクセスできず、手作業が残る
* 失敗したときの回復手順がない

この場合、プロンプトで注意するだけでは弱いです。

「実行できる」「検証できる」「止められる」「観測できる」仕組みが必要です。

改善するなら、次のような方向です。

* `npm test` や `npm run lint` などの検証コマンドを明示する
* CI で落ちる条件を作る
* pre-commit hook を入れる
* サンドボックスや権限を設計する
* レビュー用のチェックリストを用意する
* Agent Skills に定型作業をまとめる
* MCP などで必要な外部ツールに接続する
* ログやトレースを見やすくする

同じ失敗が繰り返されるなら、プロンプトで注意し続けるより、Harness に寄せた方が安定します。

Agent Skills に向いているのは、単なる一文の好みではなく「手順がある作業」です。

たとえば、次のようなものです。

* PRレビューの観点
* リリース作業
* ドキュメント更新
* 調査レポート作成
* UI確認の手順
* データ移行の検証手順

逆に、「短く答えてほしい」「です・ます調で書いてほしい」のような全体方針は、Agent Skills より `AGENTS.md` や通常の Prompt に置く方が自然です。

## 反復の回し方が弱い: Loop の問題

次のような失敗は、Loop の問題として考えやすいです。

* 毎回人間が次の指示を出している
* いつ終わるべきか分からない
* 失敗しても同じ試行を繰り返す
* 途中経過が残らない
* コストだけ増えて成果が出ない
* 書く役と検証する役が同じで、チェックが甘くなる
* 長時間走らせた結果、目的からズレていく

Loop の問題は、エージェントを「動かす」だけでなく「回し続ける」段階で見えてきます。

改善するなら、次のような方向です。

* 起動条件を決める
* 完了条件を決める
* 停止条件を決める
* 再試行の上限を決める
* 途中で人間に確認する条件を決める
* writer と reviewer を分ける
* sub-agent に任せる範囲を決める
* automation で定期実行する範囲を決める

特に停止条件は大事です。

「成功するまで頑張る」ではなく、「3回失敗したら止める」「テストが通ったら止める」「差分が大きくなりすぎたら人間に戻す」のように、終わり方を先に決めておく方が扱いやすくなります。

# Harness と Loop はどこが違うのか

Harness と Loop は近いので、混ざりやすいです。

私は次のように分けて考えています。

| 観点 | Harness | Loop |
| --- | --- | --- |
| 主な関心 | 動く環境 | 回し方 |
| 例 | テストコマンド、権限、ツール、Agent Skills、MCP | 起動条件、完了条件、停止条件、再試行 |
| 問い | エージェントは安全に実行・検証できるか | エージェントはいつ、何を、どこまで続けるか |

たとえば、テストコマンドを用意するのは Harness です。

テストが落ちたら修正して再実行し、3回失敗したら止める、という設計は Loop です。

レビュー観点をチェックリストにするのは Harness です。

実装担当とは別の reviewer agent に確認させる流れを作るのは Loop です。

つまり、Harness は「エージェントが安全に働ける作業場」を作る話で、Loop は「その作業場でどのように仕事を回すか」の話だと考えると分かりやすいです。

# 小さく始めるなら何から直すか

最初から複雑なループを作る必要はないと思っています。

個人開発や小さなリポジトリなら、次の順番で十分です。

## 1. 毎回説明していることを外に出す

まずは `AGENTS.md` に、毎回説明していることを書きます。

Codex では、`AGENTS.md` はリポジトリに置ける永続的な指示です。ビルド・テストコマンド、レビュー観点、リポジトリ固有の慣習、ディレクトリごとの注意点などを書くのに向いています。

```
## 作業ルール

- 変更後は npm test を実行する
- UI変更時はスクリーンショットで確認する
- 既存の設計パターンを優先する
- 不要なリファクタリングはしない
```

これは Context の入り口になります。

ただし、`AGENTS.md` に書くだけでは実行が保証されるわけではありません。テストやlintを本当に強制したい場合は、CI、pre-commit hook、型チェックなどの Harness 側の仕組みも合わせて用意します。

## 2. 繰り返す手順を Agent Skills にする

次に、毎回同じ流れで実行している作業を Agent Skills にします。

Codex の Agent Skills は、`SKILL.md` を中心にしたディレクトリです。`SKILL.md` には、どのタイミングで使うのか、どんな手順で進めるのかを書きます。必要なら `scripts/` や `references/` も置けます。

```
---
name: zenn-article
description: Zenn記事を作成・編集するときに使う。frontmatter、構成、文体、参考リンクを確認する。
---

1. frontmatter に title, emoji, type, topics, published があるか確認する。
2. はじめに、本文、おわりに、参考の流れを確認する。
3. コードブロックには言語名を付ける。
4. 参考リンクがある場合は最後にまとめる。
```

これは Prompt を長くするためのものではなく、作業の型を再利用するためのものです。

Codex は Agent Skills の `name` や `description` を見て、必要なときに明示的または暗黙的に使えます。詳細な手順は必要になったときに読むので、毎回すべての手順をプロンプトに詰め込むより扱いやすくなります。

## 3. 検証コマンドを明示する

エージェントが迷わないように、検証コマンドを書きます。

```
## 検証

- npm run lint
- npm test
- npm run build
```

「何をもって完了とするか」を決めやすくなります。

## 4. 同じ失敗をルールやチェックにする

同じ失敗が2回起きたら、プロンプトで注意するのではなく、ルールやチェックにします。

* ルールに書く
* テストを追加する
* lint にする
* CI で落とす
* Agent Skills に手順をまとめる

このあたりから Harness Engineering っぽくなります。

## 5. 繰り返し作業だけループ化する

最後に、繰り返しやすい作業だけ Loop にします。

* 依存関係更新
* CI失敗の調査
* リンク切れ確認
* issue triage
* リリースノート作成

逆に、成功条件が曖昧な作業をいきなりループ化すると、コストだけ増えやすいです。

# おわりに

今回は、いくつかの情報リソースを見ながら、LLM と一緒に Prompt / Context / Harness / Loop という言葉を整理してみました。

調べてみると、まったく新しい考え方というより、これまでソフトウェアエンジニアリングでやってきたことに名前が付いている面も大きいと感じました。

たとえば、プロジェクトの前提をドキュメントに残す、テストやlintで検証する、手順を再利用できる形にする、失敗したら止まる条件を決める、といったことです。

そう考えると、用語そのものに大きな価値があるというより、用語をきっかけに「今、自分はどの層を直そうとしているのか」を見直せるところに意味がありそうです。

プロンプトを直すのか、見せる情報を直すのか、実行環境を整えるのか、反復の流れを設計するのか。

整理してみると、次にやると良さそうなことも少し見えてきます。

まずは、毎回プロンプトで注意していることを1つ、`AGENTS.md`、Agent Skills、チェックのどれかに移すところから始めてみようと思います。

次はどんな用語が出てくるのでしょうか。
