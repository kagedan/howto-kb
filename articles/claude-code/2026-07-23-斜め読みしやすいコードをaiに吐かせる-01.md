AIが生成したコードをいかにレビューするか。いやはや、もはや従来的な人間がAIのコードをいちいち読むという行為がナンセンスで、別の仕組みで品質を担保するべきだ、といったような話を連日目にする。

そのような大きな価値転換のテーマも面白いが、もう少しライトな、今まで通りコードに目を通す前提で、ちょっと工夫を入れることで読みやすくしているという話をしてみたい。

直近の私の仕事的に、アイデアを元に単発のCLIツールをTypeScriptで一からAIに作らせるようなことが多く、既存の巨大なプロダクト開発などではちょっと当てはまらないこともあるかもしれないが、エッセンスは共通なはずである。

個人的に、3つの観点の指示をコンパクトにCLAUDE.mdに書くようにしている。  
（CLAUDE.mdだけだと守らないこともあるので、SKILLとか終了時のHookとかでやってもいいはず。この辺の細かいテクニックには触れない）

# 順序

```
# 並び順

ファイル・interface・定数いずれにおいても、**理解に必要なものが先、参照用のものが後**。
上から読めば概要がわかり、下に進むほど詳細になるように構成する。
この原則は再帰的に働き、export グループ内・private グループ内でも同じく適用する。

## ファイル内

1. imports
2. export する型・定数（API の形。関数シグネチャ理解の前提）
3. export 関数（公開 API。ファイルの目的が一目でわかる）
4. private 関数（export 関数内での出現順に並べ、呼び出し先を直後に置く）
5. private データ（マップテーブル等。参照されるだけの末端データ）

## interface のフィールド

ドメインの自然な順序に従って並べる。

識別子 → コア属性 → 条件・コンテキスト → 詳細・メタデータ

## enum 的な定数・マップテーブル

ドメイン固有の序列に従う（階層の高低、強度の軽重、公式の番号順など）。
マップテーブルのキー順は、対応する const 定義の順序と一致させる。
```

AIが書いたコードを読む際には、一行一行同じ濃淡の脳みその使い方で読むのではなく、できる限り斜め読みをしたい、と多くの人が感じているだろう。

AIの台頭以前でもそうだったとは思うが、いかんせん今ではコードが生成されるスピードが早すぎて、そうでもしないとレビュー自体を諦めてしまう、と言った方が正確か。

ということで、上から目を動かしていくだけでコードの概要が無意識に一番入ってきやすいようにしておくといいだろう、とこのような指示をしている。

1つのファイル内で重要な、exportしている型や関数のインターフェース／流れだけをひとまとめにチェックできるし、瑣末なロジックの内容はスキップしたり、速度を上げて斜め読みするなど、濃淡をつけやすい。

大方今のAIは賢いし、テストを自律的に回したりするので、もちろん場合には依るがprivate関数の短いロジックは大体合っていることが多く、人間がツッコミを入れたいのは、インターフェースや一段上の設計、などだったりする。

またAIは過剰にオプションを追加しがちなので、JSでいうところのobject内のkeyなども重要なのものから並べると、無駄に気づき削ぎやすい。

これらは、実際に何も指定しなかった場合に私個人的に読みづらかった観点を元に書いてある。

今のところ明示的に指示をしないとこうはならない上に、可読性向上のコスパがいいので、結構気に入っている方法である。

# コメント

```
# コメント

**コードは What を、コメントは Why だけを語る。** 名前から読み取れることは繰り返さない。
コードだけでは理由がわからない場合にのみ書く:

- 意味が自明でないマジックバリュー
- 外部のバグや非自明な仕様に合わせている箇所（例: ベンダー側の typo に追従）
- 汎用的な型で中身がわからない場合の具体例（例: `Record<string, string>` が実際は何と何の対応か）
- 複数の選択肢がある中でなぜその方法を選んだか
- 非自明な変換（例: `year % 100`）
```

WhatではなくWhyをコメントに書け、というのは耳にタコな話だが、ご存知の通りAIはWhatを大量に書くので口を酸っぱくして言っておきたい。

その他、先ほどのコードの順序の話と同様に、できる限り斜め読みしやすいよう、非自明なことにコメントを残すように補足している。

AIの書くコードは、なかなか自分のこれまでの経験や発想の外にある「スマート」な実装をしてくることが多いので、コメントによって助かることも多い。

なお、コメントは英語で書かないと気持ち悪いという方も多いと思うし、OSSや仕事で多国籍の環境とかだと、より英語MUSTとなると思うが、自分しか読まないコードであれば、思い切ってコメントを日本語（母国語）で強制してしまうのもアリかもしれない。

# レイヤー

```
# レイヤー設計

依存は一方向のみ: cli → usecases → domain / ports。adapters は ports を実装する形で外側から注入する。

- cli: エントリーポイント。adapters を組み立てて usecases に注入する配線だけを書き、ロジックを持たない
- usecases: 処理の流れ（オーケストレーション）。domain と ports にのみ依存し、adapters は引数で注入される
- domain: ビジネスロジックの本体。plain な型と純粋関数のみで構成し、外部サービス・他レイヤーに依存しない
- ports: usecases が必要とする外部サービスの抽象。interface のみを置き、実装を持たない
- adapters: ports の実装。SDK・API 呼び出し等の技術詳細はすべてここに閉じる
```

私はヘキサゴナル的な思想でディレクトリを切って書くことが多いが、可読性の観点でも嬉しい点が多いことに気づいた。

例えば、外部サービスと連携するところのロジックは往々にして泥臭く、その前提知識が求められることが多い。

VibeCodingが光る、自分のカバー領域外の実装においては、初めて出会う知識も多く、ここで目が止まってしまうこともある。

そういった際に、一旦インターフェースだけに着目しつつロジックは後で読む、とスキップするか、逆に重要／キャッチアップしたいところなので重点的に読みつつ、わからないことをAIに積極的に聞く、という風にも使い分けられる。

何も指示しないでコードを書かせると、基本こういったレイヤー分離はしないはずなので、全レイヤーが、AIが考えた適当なセマンティックで分けられたファイルに分割され、斜め読みする際にレイヤーを横断させられまくって、こういった読み方をしづらい。

usecases（上記に書いた通り、処理の流れの層と思ってもらって構わない）なども、流れ自体が重要なときは読み込めばいいし、グルーコード的な際には目を滑らせればいい。

といったように、各々、自分の感覚で「何も指示をしなかった際に斜め読みするときに読みにくい」と思う障害を取り除くように、ルールを書いていくといいかもしれない。

人間が読みやすいようにするためのものなので、重要なのは、このルール自体をAIだけに任せないことだろうか。
