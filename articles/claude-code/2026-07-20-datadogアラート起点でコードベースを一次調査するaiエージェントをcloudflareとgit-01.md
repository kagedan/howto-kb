## はじめに

プロダクトを運用していると、Datadog のエラーアラートが Slack に流れてきて、誰かが「これ何のエラー？」とスレッドで一次調査を始める、という光景が日常的にあります。ログを見て、トレースを辿って、該当コードを開いて、原因の当たりをつける。この最初の 30 分は定型的な作業が多いです。

この記事は、この 30 分の定型作業を Cloudflare Workers + GitHub Actions + Claude Code で自動化する AI エージェントを構築した話です。アーキテクチャとスタック選定の理由、約 1 ヶ月運用した実測の費用感をまとめます。

## 何を作ったか：全体アーキテクチャ

コンポーネントは 4 つだけです。

| コンポーネント | 役割 |
| --- | --- |
| Datadog | エラーモニターが発火し、Slack のアラートチャンネルに通知を投稿する |
| Cloudflare Workers（Hono + D1） | Slack イベントを受けてアラートを検知し、GitHub Actions を起動する「薄い受け口」 |
| GitHub Actions | `repository_dispatch` で起動し、Claude Code（headless）が調査を実行する環境 |
| Slack | アラートスレッドに調査結果が返る |

正常系のシーケンスはこれだけです。

ポイントは、**Worker はあくまで薄い受け口に徹して、重い調査は GitHub Actions 側に寄せている**ことです。Slack の Events API は 3 秒以内の応答を求めるので、Worker は署名検証とアラート判定だけをやって、すぐに `repository_dispatch` を投げます。調査本体は Actions 上で Claude Code CLI をヘッドレス実行（`claude -p --model sonnet`）し、対象プロダクトのリポジトリを shallow checkout したものと、Datadog MCP サーバー（read-only のツールのみ許可）を材料に一次調査させています。

Datadog の Webhook で Worker を直接呼ぶこともできますが、あえて Slack を経由しています。アラート・調査結果・その後の議論を同じスレッドに残すことで、Slack のアラートチャンネルがそのままエラー対応のストックになるからです。トリガーを Slack のアラート投稿にしておけば、そのメッセージへのスレッド返信だけでこの形が成立します。

調査の結果「コード修正が必要」と判断された場合は、スレッドの返信に Issue 作成ボタンが付きます。本格的に対応すると決めたらこのボタンで GitHub Issue を作成し、あとはローカルの Claude Code にその Issue を読み込ませて修正のプルリクエストを上げる、という運用にしています。エージェントの仕事はあくまで一次調査と Issue の下書きまでで、修正は人間のレビューが効くフローに乗せる、という線引きです。

エージェントは Slack に直接投稿しません。調査結果を Markdown ファイルとして書き出すところまでが仕事で、投稿は後続の「信頼された」ステップが行います。アラート本文はユーザー入力が混ざり得る攻撃可能面なので、ペイロードは常にデータとして扱う・スキルディレクトリを read-only にする・出力にシークレットらしき文字列があれば投稿を拒否する、といったプロンプトインジェクション対策も入れています。

## なぜこのスタックにしたのか

### 1. 自社のコードベースまで踏み込んだ調査ができる

Datadog 標準の AI 機能はテレメトリ（ログ・メトリクス・トレース）の世界で完結します。しかし一次調査で本当に欲しいのは「どのファイルのどの行が原因で、どう直せばよさそうか」までの当たりです。自前のエージェントなら、対象リポジトリを checkout して `file:line` まで特定させ、修正の方向性まで書かせることができます。ここが既製品との一番の差分でした。

### 2. 費用が完全に従量で、1 回あたりのコストが見える

このエージェントの構成要素は、すべて「動いたぶんだけ課金」です。アイドル時のコストはゼロで、アラートが鳴らなければ 1 円もかかりません。SaaS の AI 機能はシート課金や上位 SKU に紐づくことが多く、「このアラート 1 件の調査にいくらかかったか」は見えません。自前だと API キーを機能ごとに分けるだけで、後述のとおり実測の 1 回あたり費用が出せます。

### 3. Cloudflare なら受け口を安く・簡単に運用できる

受け口を Cloudflare Workers にしたのは、yusukebe さんの[「AIエージェントはCloudflareに賭けろ」](https://zenn.dev/yusukebe/articles/ccb1f953e48ee1)の影響が大きいです。V8 Isolate ベースの Workers は立ち上がりが速くメモリ効率がよいので、「Slack イベントを 3 秒以内に捌いて非同期処理に逃がす」という今回の受け口の要件にぴったりはまります。状態管理も D1（SQLite）が Worker から直接使えて、重複チェックのテーブルを 1 つ作るだけで済みました。Hono で書けば Worker 自体は小さな Web アプリでしかなく、認知負荷が低いのもよい点です。

一方で、調査本体を Workers ではなく GitHub Actions に置いたのは、**リポジトリ全体を checkout して Claude Code を走らせる**という要件のためです。数分〜十数分かかるファイルシステム前提のワークロードは、CPU 時間とファイルシステムの制約がある Workers よりも、`repository_dispatch` で起動する Actions のジョブが素直です。Actions はコードのすぐ隣にあり、シークレット管理も揃っていて、しかもイベント駆動なので待機コストがありません。

## 費用面の工夫：同じエラーは二度調査しない

運用してみて一番効いたのがこの機構です。

エラーアラートの現実として、**同じエラーは何度も鳴ります**。しかも「実害は小さいが直っていないエラー」ほど、対応が後回しになったまま鳴り続けます。放置された同種アラートは通知疲れを生み、そのたびに AI が同じ調査を繰り返せば API 費用も無駄になります。実運用でありがちな「知ってるエラーがまた鳴ってるだけ」に、毎回フルの調査を走らせる意味はありません。

そこで、Datadog の Error Tracking が発行する `issue.id` をフィンガープリントとして、D1 に調査済みテーブルを持たせました。

```
CREATE TABLE incident_issues (
  issue_id    TEXT PRIMARY KEY,   -- Datadog Error Tracking の issue.id
  channel     TEXT NOT NULL,
  thread_ts   TEXT NOT NULL,      -- 最初の調査スレッド
  triaged_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

同じ `issue.id` のアラートが 14 日以内に再度来たら、Actions は起動せず（= トークン消費ゼロ）、新しいアラートのスレッドに「このエラーは対応中です。最初の対応スレッドはこちら」と最初の調査スレッドへのリンクだけを返します。

判定は D1 への `INSERT ... ON CONFLICT DO NOTHING` で行うので、同時に複数アラートが来ても調査が二重に走ることはありません。

この機構の副次効果として、**調査結果とその後の議論が最初のスレッドに集約される**のも地味に効いています。同じエラーのスレッドが乱立して「どこで話してたっけ」となる問題が構造的に起きません。14 日を過ぎて再発した場合は「まだ直っていない・状況が変わったかもしれない」とみなして再調査を走らせます。

細かい設計判断として、重複判定に失敗したとき（D1 障害や `issue.id` が取れないアラート）は**調査を実行する側に倒す**ようにしています。まれに重複調査が走るコストより、必要な調査が漏れるコストのほうが大きいからです。

このほかにも、費用と暴走を抑えるガードをいくつか入れています。

* モデルは Sonnet に固定し、ジョブに `timeout-minutes: 20` の上限
* エージェントに許可するツールは read-only の Datadog MCP ツールとローカルのファイル操作のみ（Bash 不許可）
* API キーを機能ごとに分離し、コストを機能単位で計測できるように

## 実測の費用感

2026/06/21〜07/19 の約 1 ヶ月、実運用した実測値です。

前提として、調査対象のコードベースは TypeScript 中心のモノレポで、ソースコードだけで約 4,600 ファイル・ディレクトリ約 1,800・合計 100 万行超という規模です。対象ユーザーが千規模の業務アプリケーションで、おもちゃのサンプルではなく、この規模のプロダクションコードに対して以下の費用感で運用できています。

### アラート発報数と調査実行数

まず前提となる母数です。この期間、対象プロダクトのエラー系モニターの発報（Triggered）は本番環境だけで **265 件**ありました。一方、実際に調査（GitHub Actions）が走ったのは **121 回**です。Recovered / Re-notified の除外と重複エラーのスキップ機構によって、発報の半分以下まで実行を絞れていることになります。スキップされたぶんは丸ごとゼロ円です。

### AI（Anthropic API）

この機能専用の API キーの消費が **$93.90** でした（立ち上げ期間のテスト実行も含む）。調査実行 121 回で割ると、**1 回の一次調査あたり約 $0.78** です。一次調査 1 回に人が 30 分かけると考えれば、1 回 $1 弱は十分安いと感じています。

### Datadog 標準の AI と比べて安いのか

Datadog にも自律調査の [Bits Investigation](https://www.datadoghq.com/product/ai/bits-investigation/) があり、こちらは [AI Credits](https://www.datadoghq.com/pricing/?product=bits-ai-sre) で課金されます。執筆時点で **$500/500 クレジット・月（年間契約、オンデマンドは $1.30/クレジット）**、自律調査 1 回あたり **約 6.5 クレジット**なので、**1 調査あたり約 $6.5〜8.5** です。

自前の約 $0.78/回はその **1/8〜1/10** で、月 121 回を Bits でまかなうと単純計算で $790〜1,000 相当になります。しかもクレジットのコミットは月 $500 から始まるのに対し、自前構成は固定費ゼロです。もちろん Bits はマネージドでセットアップ不要という別の価値があるので単純な優劣ではありませんが、「アラート駆動の一次調査」という用途に限れば、自前のほうが 1 桁安く運用できています。

### GitHub Actions

`ubuntu-latest` のジョブは 1 回あたり平均 3 分で、月 121 回の合計は約 360 分でした。GitHub の有料プランに含まれる Actions の無料枠（Team プランで 3,000 分/月、Enterprise で 50,000 分/月）に収まるため、**追加費用はゼロ**でした。仮に無料枠を使い切って全額課金されたとしても、Linux ランナーは $0.008/分なので月 $3 弱です。各ジョブには `timeout-minutes: 20` の上限を設けているので、すべての実行が上限まで走る最悪ケースでも月 2,420 分 ≒ $19 程度に収まります。

### Cloudflare

Workers はアラートのイベントを受けるだけなので、リクエスト数は無料枠（10 万リクエスト/日）に遠く及びません。D1 も読み書きは 1 アラートあたり数行で、無料枠の誤差レベルです。**実質 $0**（Workers Paid を契約していても $5/月の基本料のみ）で運用できています。

### まとめると

| 項目 | 実測（約 1 ヶ月） |
| --- | --- |
| Anthropic API（調査本体、121 回） | $93.90（約 $0.78/回） |
| GitHub Actions | $0（無料枠内） |
| Cloudflare Workers + D1 | $0 |
| **合計** | **約 $94**（参考: Bits Investigation 換算で $790〜1,000 相当） |

月 $94 で、アラートのたびに一次調査が自動で走り、結果がスレッドに集約される。オンコールの一次対応を人が毎回やるコストと比べれば、個人的には完全にペイしていると感じます。また、全額がほぼ Anthropic API 費用＝純粋な調査の実行コストであり、基盤側の固定費がゼロというのが、このスタックのいちばんの成果です。

## おわりに

Datadog アラート起点の一次調査エージェントを、Cloudflare Workers（受け口）+ GitHub Actions（調査実行）+ Claude Code（調査本体）で構成しました。振り返ると、このスタックの本質は「**イベント駆動の部品だけで組んだので、固定費が構造的に発生しない**」ことに尽きます。

運用面では、派手な調査能力よりも「同じエラーを二度調査しない」という地味な機構がいちばん効きました。エラーアラートの運用には「直っていないが実害の小さいエラーが鳴り続ける」という現実があり、AI エージェントを素朴に接続すると、その現実がそのまま API 費用に変換されてしまいます。デデュープは費用対策であると同時に、通知疲れ対策であり、議論をスレッドに集約する運用改善でもありました。

今後の拡張として考えているのは、一次調査の先です。まずは現実的なところで、Issue 作成から先の「修正ブランチを切って draft PR を作るところまで」を Actions 上のエージェントに任せる形（マージ判断は必ず人間に残す）。さらにその先として、Cloudflare が Sandboxes や Dynamic Workers といったエージェント向けの実行環境を拡充しているので、いまは GitHub Actions に置いている調査・修正の実行系を Cloudflare 側に寄せて、受け口から修正プルリクの作成までをエッジ上で完結させる構成も試したいと考えています。
