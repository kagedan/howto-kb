## 私の自動化、これからいくらかかるのか！？【実測】

**実際のセッションログからトークンを集計して算出**しました。

### 自動化の規模感

中身の詳細は伏せますが、ざっくり言うと **Raspberry Pi の cron で、毎日いくつかの「情報収集して要約・整理する」系のタスクを Claude に自動で回している**——という使い方です。

Webから情報を集めて構造化したり、画像を読ませたり、それなりに重めの処理が毎日複数。モデルは `Sonnet` を使っています。

「自分の自動化が重いか軽いか」は人それぞれなので、**金額そのものより「どう測るか」と「規模に対していくらになるか」の感覚**を持ち帰ってもらえればと思います。

### 測り方

`usage` のトークン数（入力・出力・キャッシュ書込・キャッシュ読込）を集計し、**API標準レート**で金額換算します。集計には [`ccusage`](https://github.com/ryoppippi/ccusage) を使いました。

＊ Sonnet のレート（/1Mトークン、公式の料金表より）

| 種類           | 単価     |
| -------------- | -------- |
| 入力           | \\$3.00  |
| 出力           | \\$15.00 |
| キャッシュ書込 | \\$3.75  |
| キャッシュ読込 | \\$0.30  |

### 結果：種類ごと・月ごとの実消費

まず、「手動の対話利用＋一部の自動実行」の実消費をAPI換算で、種類別・月別に出すとこうなりました。

| 種類           | 5月トークン |  5月コスト | 6月トークン（5日時点） |  6月コスト |
| -------------- | ----------: | ---------: | ---------------------: | ---------: |
| 入力           |      15,524 |      $0.07 |                 61,531 |      $0.30 |
| 出力           |     255,911 |      $4.48 |                333,799 |      $5.80 |
| キャッシュ書込 |   2,954,922 |     $12.03 |              1,463,263 |      $5.70 |
| キャッシュ読込 |  41,876,694 |     $12.40 |             44,626,667 |     $13.73 |
| **合計**       |   **約45M** | **$28.98** |              **約46M** | **$25.54** |

5月でだいたい **$29**。
今回は次に示す自動化分の方がメインなので、本記事の最終的な計算には含みません。

### メインの自動化（cron）分だけだといくらか

私の cron は実行のたびにセッションログを消す作りにしていたので、集計から漏れていました。そこで、メインの自動化分は**「実際の実行回数 × 1回あたりの実測コスト」**で見積もります。

毎日 cron で回しているのは3本。実行回数はログから数えました。

| スキル          | 1回あたり（実測） | 5月の実行回数 | 6月（5日） |
| --------------- | ----------------: | ------------: | ---------: |
| 情報収集タスクA |          約 $0.50 |            31 |          5 |
| 情報収集タスクB |          約 $0.46 |            31 |          5 |
| OCRタスク       |          約 $0.52 | 2（下旬開始） |          5 |

- **5月の自動化コスト ≈ $31**（31×0.50 + 31×0.46 + 2×0.52）
- **6月は3本とも毎日化したので、月換算 ≈ $44**

正直に言うと、最初は「月$200超溶けてるのでは」と思い込んでいました。が、ちゃんと重複を除いて計算し直したら **$60前後**。内、自動化分はドラマチックな金額を期待していた自分には肩透かしでしたが、**実数はこれです**。

サブスクの定額の裏で、API換算このくらいが動いていた、ということ。

:::note memo
ちなみに今回触れている `Raspberry Pi` 以外のマシンでも小さいタスクを動かしているので、本当はもっと使っています。
:::

### クレジット枠に収まるのか？

6/15以降、この**自動化の $31〜44/月だけ**が新しいクレジット枠から引かれます（手動の対話利用は従来どおりサブスク枠で無傷）。各プランのクレジットと並べるとこうです。

![chart-1-vs-credit.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/1655179/50090d2d-71f5-4932-921a-3478d9020108.png)

*私の自動化（推定 約$31/月）を、各プランのクレジット枠と並べたところ（当然の図だが。）*

| プラン         | クレジット | 私の自動化（$31〜44）は収まる？      |
| -------------- | ---------- | ------------------------------------ |
| Pro ($20)      | $20        | ❌ **月末を待たず枯渇**（私は今ここ） |
| Max 5x ($100)  | $100       | ✅ **十分収まる**                     |
| Max 20x ($200) | $200       | ✅ 余裕（むしろオーバースペック）     |

私は今 **Pro** なので、6/15以降は自動化が月末まで持ちません。続けるには、
- **overflow（超過分を従量で払う）を有効にする**
- **Max 5x に上げる**

の二択。
いずれにせよ、

**$20で“使い放題”だった自動化に、これからは追加のコストがかかる**

——これが実測から出た答えです。

### 学び：トークン量とコストは一致しない

ログを集計していて、ひとつ面白い発見がありました。

**トークン数で見ると、その9割以上（5月は93%）は「キャッシュ読込」です。** 

Claude Code は会話が進むほどコンテキストが膨らみ、毎ターンそれを読み直すので、キャッシュ読込トークンが大量に積み上がります。

ところが——**コスト（金額）の内訳は、トークン数とは比率が違います。**

![chart-2-token-vs-cost.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/1655179/88c696e2-1893-42fb-bc44-1714358b1c4d.png)
*同じ消費を「トークン量」と「コスト」で見ると、内訳が変わる。*

- キャッシュ読込は単価が安い（入力の0.1倍）ので、**トークンの93%を占めるのにコストでは43%**
- 逆にキャッシュ書込は**トークンでは7%なのにコストでは42%**
- 出力に至っては**トークンの1%未満なのにコストの15%**

> **「量が多い＝高い」ではない。**「少ないけど単価が高いもの（出力・キャッシュ書込）」が、地味にコストを作っています。

じゃあ最適化はどうするか。
結論はシンプルで、<font color="red">**会話（セッション）を短く保つ**</font>こと。
セッションが伸びるほど、キャッシュの読み書きも出力も全部積み上がるからです。
具体的には、

- **1タスク1セッションで完結させる**
- **不要なMCPサーバー・巨大なコンテキストを最初から減らす**
- **LLMでやる必要のない処理（単純なスクレイピング等）は、Claudeに渡す前にスクリプト側で済ませる**

「出力だけ削ればいい」と思いがちですが、効くのは**セッション全体を軽く・短くする**ことでした。

## 取れる選択肢を比べてみた

破滅的な額ではないとはいえ、Pro の $20 枠は超える。現実的な打ち手を並べて比較しました。

| 選択肢                   | 概要                                       | 向くケース                       | コスト感           |
| ------------------------ | ------------------------------------------ | -------------------------------- | ------------------ |
| A. クレジット内に最適化  | 実行頻度・モデル・出力・セッション長を絞る | 自動化が軽め                     | 追加 $0            |
| B. APIキー従量へ逃がす   | 自動化だけ Developer Platform API に移行   | 重い自動化／コストを可視化したい | 従量（青天井注意） |
| C. overflow を有効化     | クレジット超過分を従量で許可               | 月によって波がある               | 変動               |
| D. ローカル/他モデル併用 | 軽いタスクはローカルLLM等へ                | コスト最優先                     | ほぼ $0            |
| E. 対話運用へ戻す        | 自動化をやめ、手で Claude Code を叩く      | 自動化が趣味の範囲               | 追加 $0            |

私の方針は **A（最適化）をベースに、重い処理だけ D（LLM不要な部分の切り出し）** です。具体的には、

- 情報の「収集」部分（単純なスクレイピングなど）はそもそもLLM不要なので、Claudeに渡す前にスクリプト側で済ませる
- セッションを毎回新規で立てて短命にする（コンテキストの肥大を防ぐ）
- そもそも「毎日AIを回す必要があるタスクか？」を棚卸しして、頻度を落とせるものは落とす

## まとめ：6/15までにやることチェックリスト

- [ ] 自分の自動化が `claude -p` / Agent SDK / GitHub Actions を使っているか棚卸しする
- [ ] セッションログ等から **1回あたりのトークン・コスト**を実測する
- [ ] 月換算して、自分のプランのクレジット（\\$20/\\$100/\\$200）に収まるか確認する
- [ ] 収まらないなら、A〜E の打ち手を選ぶ（まずは**セッションを短く保つ**最適化が効く）
- [ ] overflow を有効化するか決め、**クレジット枯渇のアラート**を仕込む

「使い放題だから」と何でも自動化していた時代から、**「これは自動化する価値があるか？」をコストで問い直す時代**になった、とも言えます。個人的にはこれ、悪い変化だとは思っていません。寝てる間にいくら使っているかも分からないまま回し続けるより、**自分の消費を把握して使う**ほうが、よっぽど健全です。

それでは、良い自動化ライフを。
