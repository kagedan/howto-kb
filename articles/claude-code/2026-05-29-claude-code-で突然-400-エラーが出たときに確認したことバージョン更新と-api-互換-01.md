今朝、いつものようにターミナルで Claude Codeを起動したところ、突然400エラーが出るようになりました。

昨日までは問題なく使えていたなのに、なぜ今日はだめになってしまいましたか。最初は API Key やローカル設定の問題だと思っていましたが、いくつか確認していくと、ローカル環境そのものではなく、Claude Code側のバージョン更新と DeepSeek API側の互換性だと分かりました。

この記事では、実際に確認したこと、試した対処、最終的に効果があった回避策をメモとして残します。

同じように費用を節約するため、Claude CodeをDeepSeek API経由で使っていて、急に400エラーが出るようになった方の参考になれば幸いです。

## 発生した現象

朝、いつも通りClaude Codeを起動したところ、リクエスト時に400エラーが返るようになりました。

昨日までは正常に動作していたため、最初は以下のような原因を疑いました。

* API Key が無効になった
* ローカルの設定ファイルが壊れた
* ネットワーク環境に問題がある
* DeepSeek API側で一時的な障害が起きている

ただし、API Key や設定を確認しても特に問題は見つかりませんでした。

そして、Claude Codeを使っている知人に確認したところ、同じタイミングで問題が起きた人もいませんでした。そのため、Claude Code全体の障害より、利用環境やAPIの組み合わせによって発生している問題のように見えました。

## 試したこと

まずはローカル側の問題か特定したいですから、以下を試しました。

`npm uninstall -g @anthropic-ai/claude-code`

その後、Claude Codeを再インストールしてみました。しかし、エラーは解消しませんでした。  
次に、設定ファイルを見直し、API Keyや接続先の設定も確認しましたが、状況も変わりませんでした。  
さらに、API Keyの再設定やネットワーク環境の変更までも試しましたが、同じく400エラーが頻繁に返ってきました。

この時点で、少なくとも「API Keyが間違っている」「ローカル設定が壊れている」といった問題ではなさそうだと判断しました。

## 原因として考えられたこと

調べていくと、Claude Codeの CLI が更新され、新しいバージョンでリクエストの形式に変更が入った可能性がありました。

一方で、DeepSeek API側はその新しい形式にまだ完全には対応していない可能性があります。

つまり、今回の問題は次のような構図だと考えられます。

Claude Codeが更新される  
↓  
送信されるメッセージ形式やリクエスト形式が変わる  
↓  
DeepSeek API側がその形式を受け付けられない  
↓  
400 エラーになる

Claude Codeを公式の Claude APIで使っている場合は問題が出ない一方で、DeepSeek APIやルーティングツールを経由している環境では、API 互換性の差分が表面化した可能性があります。

実際、DeepSeek 公式ドキュメントでもClaude Codeとの連携方法は案内されていますが、Claude Code側の更新タイミングによっては、互換性の問題が起きる可能性があります。

## なぜ人によって発生状況が違うのか

同じClaude Codeを使っていても、チーム内全員に同じ問題が出るとは限りません。

今回のようなケースでは、以下の違いによって挙動が変わる可能性があります。

* Claude 公式APIを使っているか、DeepSeek APIを使っているか
* Claude Codeのバージョンが更新済みか
* 自動更新が有効になっているか
* claude-code-routerなどのルーティングツールを挟んでいるか
* ローカル環境ごとに設定ファイルの内容が異なるか

そのため、知人の環境では問題なく動いているのに、自分の環境だけエラーになる、という状況も十分あり得ます。

## 対処方法

今回、ターミナルではClaude Codeを前の安定版に戻すことで回避できました。

以下のように、一度インストール済みのClaude Codeを削除し、問題が起きる前のバージョンを指定してインストールします。

`npm uninstall -g @anthropic-ai/claude-code`  
`npm install -g @anthropic-ai/claude-code@2.1.153`

![](https://static.zenn.studio/user-upload/dfe40d2fdf12-20260529.png)

2.1.153は、現時点の最新バージョンより一つ前のバージョンにあたります。

インストール後、コードでバージョンを確認します。

`claude --version`

その後、再度Claude Codeを起動して、400エラーが解消されているか確認します。

## 自動更新を止める

バージョンを戻しても、自動更新が有効なままだと、しばらくして再び同じ問題が起きる可能性があります。

そのため、必要に応じてClaude Codeの自動更新を無効化しておきます。

~/.claude/settings.json がない場合は、手動で作成します。

`mkdir -p ~/.claude`  
`touch ~/.claude/settings.json`

~/.claude/settings.jsonファイルがすでにある場合、次のコードを利用してください。

`open ~/.claude/settings.json`

そのうえで、settings.jsonファイルに以下のコードをコピーぺして設定します。

```
  "DISABLE_AUTOUPDATER": "1",
  "DISABLE_UPDATES": "1"
```

保存後、ターミナルを開き直してClaude Codeを再起動します。

環境によっては、自動更新の無効化方法や設定キーの扱いが変わる可能性があります。設定してから、うまく反映されない場合は、Claude Codeの公式Issueや現在使っているバージョンの仕様もあわせて確認したほうがよさそうです。

## 確認ポイント

今回のようにClaude Codeで突然APIエラーが出た場合は、いきなり再インストールを繰り返す前に、次の順番で確認すると切り分けやすいです。

`claude --version`

まずClaude Codeのバージョンを確認します。

次に、使用しているAPIがClaude公式APIなのか、DeepSeek APIなのか、または claude-code-routerなどを経由しているのかを確認します。

そのうえで、以下を見ます。

* 昨日から今日にかけてClaude Codeのバージョンが変わっていないか
* API Key は有効か
* 接続先の base URL は正しいか
* ルーティングツール側でエラーが出ていないか
* GitHub Issues や Reddit などで同様の報告がないか

特に、Claude Codeは CLI ツールなので、ローカル環境の問題に見えやすいです。しかし、実際には「CLI 側の更新」と「接続先APIの互換性」の組み合わせで起きることもあります。

## まとめ

今回の400エラーは、ローカル設定やAPI Keyの単純なミスではなく、Claude Codeのバージョン更新とDeepSeek API側の互換性によって発生していた可能性が高そうです。

自分の環境では、以下の対応で回避できました。

* Claude Codeを前のバージョンに戻す
* 自動更新を無効化する
* Claude CodeのバージョンとAPI接続先を確認する

Claude Codeや周辺ツールは更新が速いため、昨日まで動いていたものが今日突然動かなくなることがあります。

特に、Claude 公式API以外のAPIやルーティングツールを使っている場合は、Claude Code本体の更新によって互換性の問題が出る可能性があります。

同じようなエラーに遭遇した場合は、まずバージョン差分とAPIの組み合わせを確認してみるのがよさそうです。
