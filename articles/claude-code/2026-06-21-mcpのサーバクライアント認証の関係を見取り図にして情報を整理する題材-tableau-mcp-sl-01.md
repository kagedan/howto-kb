直近、Tableau MCP界隈のあれやこれやについて、個人的に全容把握と実践のための調査をZennのエントリとして何本か重ねて来ました。Tableau MCP全体寄りの記事がこちら。

そして、Tableau MCPからクライアント接続するケースの代表としてSlackを起点にした場合、どうなるか(なりそうか)でまとめた記事群がこちら。

これらの記事を書く上で色々調べてみた結果、またこの分野に於いて同僚とも会話を重ねてきた結果、率直な感想として

**🤔.oO ( MCPの「接続まわり」、どうにもカオスだな... )**

という思いが浮かびました。

諸々整理してみると、引っかかっているポイントは大きく3つに集約されそうでした。

* 接続を提供する**サーバ側**の出し方：トランスポート(stdio / HTTP)を単独で出すのか、両対応なのか
* 受け取る**クライアント側**の対応状況：コネクター接続できるのか、できないのか
* そのあいだに挟まる**認証**：OAuthなのか、PATなのか、そもそも要るのか要らないのか

先に挙げたTableau MCP×Slack連携にしても、実に様々なつなぎ方・連携方法が乱立していて、**『よく分からん』** という思いも無くは無い感じです。

というのもあり、当エントリでは「個別のつなぎ方を覚える」のではなく、カオスを切り分けるための”見方・考え方”そのもの を整理しておこうと思います。具体例としては、これまで追ってきたTableau MCP × Slackを引き続き題材にします。

## はじめに：なぜ「カオス」に見えるのか

MCPの接続でつまずくとき、たいていは **「サーバ／認証／クライアント」のどこが噛み合っていないのか** が切り分けられていない状態です。逆に言えば、この3軸さえ意識すれば「なぜ繋がらないのか」「どうつなげばよいのか」が一気に見通せるようになります。

![MCP接続が成立する3つの軸の見取り図](https://static.zenn.studio/user-upload/8089c9b8f31e-20260620.png)

そして、これがカオスに"見える"最大の理由はシンプルで、**この3軸が、仕様の急速な進化の途中でバラバラに実装されているから** です。MCP仕様は約1年で4バージョン重ねていて(うちトランスポートは複数回見直され)、その都度クライアント実装の対応がずれ込みました。順番に分解していきます。

## 軸その1：トランスポート — サーバが出す「接続口」

### stdio(標準入出力)とは何か

`stdio` トランスポートは、クライアントがMCPサーバを **子プロセスとして起動** し、標準入力(stdin)・標準出力(stdout)経由でJSON-RPCメッセージをやり取りする方式です。ネットワークもポートも認証ハンドシェイクもなく、OSのパイプを通じて2つのプロセスが会話するだけ、という素直な作りになっています。

ローカルのファイル操作、CLIツール、IDEの隣で動くDBコネクタなど「サーバとクライアントが同じマシンにいる」場面に最適で、新規にMCPサーバを作るならまずここから始めるのが定石です。ネットワークを介さないので速い反面、単一クライアント・ローカル限定で、多人数の同時利用にはスケールしません。

### HTTPベースのトランスポートは「2回」変わっている

リモート接続が初めて必要になった2024-11-05仕様では、**HTTP+SSE(Server-Sent Events)** が採用されていました。`/sse` への持続的GET接続でストリームを受け取り、送信は別の `/messages` へPOSTする、という2エンドポイントの非対称な構造です。これは状態管理が煩雑で、長命な接続を維持し続ける必要がありスケールしにくく、再接続もできない、という弱点を抱えていました。

これを置き換えたのが、2025-03-26仕様で導入された **Streamable HTTP** です。単一エンドポイント(慣習的に `/mcp`)がPOSTとGETの両方を受け、単純な応答なら通常のJSONを、ストリームするなら `text/event-stream`(SSE)を返します。**SSEは「独立したトランスポート」ではなく「Streamable HTTPの中の一モード」に格下げされた**、と捉えると分かりやすいです。

この「約1年で4回」の改訂を時系列にするとこうなります。

![MCP仕様のトランスポート変遷タイムライン](https://static.zenn.studio/user-upload/8a29c1991cfe-20260620.png)

* **2024-11-05**：初版。stdio と HTTP+SSE を定義
* **2025-03-26**：Streamable HTTP を導入し、HTTP+SSE を**非推奨化**。OAuth 2.1ベースの認可枠組みも追加
* **2025-06-18**：MCPサーバを「OAuth 2.1リソースサーバ」と明確化。構造化ツール出力やElicitationも追加
* **2025-11-25**：最新安定版。stdio と Streamable HTTP の2本が標準、SSEは後方互換のみ。認可は CIMD(Client ID Metadata Documents)が推奨デフォルトに

### 結局 stdio と Streamable HTTP は何が違うのか

ここを絵で押さえておくと、次の「認証」の話が一気に分かりやすくなります。両者は **「サーバがどこで動き、どう会話するか」** がまったく違います。

![stdioとStreamable HTTPの構成比較](https://static.zenn.studio/user-upload/15d6ee4dac20-20260620.png)

### どちらを選ぶか

判断基準そのものはシンプルで、前掲のおさらい記事でも触れた指針がそのまま効きます。

> 個人用ツール・ローカルファイル操作・プロトタイプなら **stdio**。  
> チーム共有・本番・多人数アクセスなら **Streamable HTTP**。

新規実装でわざわざSSEを選ぶ理由は、もうありません。

## 軸その2：認証 — サーバが要求する「鍵」

トランスポートが決まると、次は「鍵」の話になります。ここがまた、`stdio` と `リモートHTTP` でくっきり分かれます。

### ローカル(stdio)は「認証なし」が基本

MCPの仕様は「認可はオプション」としていて、**stdioの実装は認可フローに従うべきではなく、代わりに環境変数から資格情報を取得すべき** と明記しています。ローカルのstdioサーバは自分で起動するぶん暗黙に信頼されるので、APIキーやトークンは環境変数でポンと渡す、というのが基本形です。

### リモート(HTTP)は OAuth 2.1 が標準

一方リモートHTTPは、誰でも到達できてしまうので、そうはいきません。**OAuth 2.1ベースの認可** が標準になりました。仕様の歩みはこんな具合です。

* **2025-03-26**：OAuth 2.1、認可サーバメタデータ、DCR(動的クライアント登録)、PKCE などの中核方針を確立
* **2025-06-18**：MCPサーバを「OAuth 2.1リソースサーバ」と明確化。`Protected Resource Metadata`(RFC 9728)や `Resource Indicators`(RFC 8707)を要求し、トークンが「そのサーバ向けに発行されたものか」を検証させるように
* **2025-11-25**：CIMD が推奨デフォルトに昇格、DCRは SHOULD→MAY へ。PKCEは全クライアント必須

### 認可フローを分解する

この OAuth のやり取りが、初見だと「いま何が起きてるの？」となりがちな部分です。流れを分解しておきます。

![リモートMCPのOAuth 2.1認可フロー](https://static.zenn.studio/user-upload/09f8e25b1f93-20260620.png)

要するに、**「トークン無しで叩く → 401で『鍵はあっちで取ってきて』と教えられる → 取ってきて出し直す」** という素直なやり取りです。

### DCR・PAT・APIキーの立ち位置

* **DCR(動的クライアント登録 / RFC 7591)**：クライアントIDを人手を介さず自動取得する仕組み。便利な反面、企業IdPが対応していない・なりすましの懸念があるなどの事情から、最新仕様では CIMD に主役を譲りました
* **PAT(個人アクセストークン)/ APIキー / Bearerトークン**：内部ツールや個人用なら、ヘッダにトークンを載せる簡易方式も許容されていて、むしろこちらが好まれる場面も多いです

## 軸その3：クライアント — 手元のアプリが「対応しているか」

3軸目は、項目が多くて対応もまちまちなので、表で見るのが早いです。2026年6月時点でのまとめは以下の通り。

| クライアント | トランスポート | OAuth対応 | リモート接続の追加方法 |
| --- | --- | --- | --- |
| Claude(Web / デスクトップ / Cowork / モバイル) | stdio(デスクトップのローカル)・SSE・Streamable HTTP | 対応(authless も可) | カスタムコネクターとしてURL追加(モバイルは新規追加不可) |
| Claude Code | stdio / HTTP / WebSocket | 対応(`/mcp` で認証) | `claude mcp add` で設定 |
| Cursor | stdio・HTTP/SSE | 対応(v1.0〜, 2025-06-04) | ワンクリックMCPセットアップ |
| VS Code(GitHub Copilot) | stdio・Streamable HTTP(SSEフォールバック) | OAuth 2.1/2.0 対応(v1.102 でGA, 2025-07-09) | `mcp.json`(ルートキーは `servers`) |

これこそがカオスの実態で、**「同じMCPサーバなのに、このクライアントでは繋がる／繋がらない」** が普通に起こります。設定ファイルのキー名すら `mcpServers`(Cursor・Claude Desktop)と `servers`(VS Code)で食い違っていて、初心者がつまずく地味なポイントになっています。

### Slackを「起点」にする場合のパターン

今回の題材であるSlack起点だと、主に次の3経路が考えられます。

1. **Claude in Slack 経由**：Claudeに追加したコネクター(リモートMCP)を、Slack上のClaudeから使う
2. **Slack公式MCPサーバ経由**：SlackがOAuth付きのリモートMCPサーバを提供しており、Claude等からSlackのメッセージ・ファイルにアクセスできる
3. **Salesforce/SlackのホストMCP・Agentforce経由**：次章のTableau例で詳述

このうち3本目を、Tableau MCPという具体的なサーバに当てはめて分解したのが次章です。1と2はSlack上でMCPを使う一般的な経路なので、本記事では深追いしません。

## 具体例：Tableau MCP × Slack を3軸で見る

`tableau/tableau-mcp`(GitHub)は Tableau / Salesforce 公式のMCPサーバで、VizQL Data Service や Metadata API 経由でTableau Cloud/Serverのコンテンツにアクセスさせます。`TRANSPORT` 環境変数で `stdio`(既定)と `http`(＝Streamable HTTP)を切り替えられる、というのは前掲記事でも触れたとおりです。

### 認証だけで「4モード」ある

Tableau MCPは、今回のカオスの縮図でもあり、収束の好例でもあります。なにせ認証だけで4モードあるのですから...

| 認証モード(`AUTH`) | 仕組み | 主な用途・前提 |
| --- | --- | --- |
| `pat`(個人アクセストークン) | `PAT_NAME` / `PAT_VALUE` を渡すだけ。既定 | 最も簡単。ただし同時並行利用が不可で、検証・プロトタイプ向き |
| `direct-trust` | Connected App 情報からJWTを生成しREST APIに認証 | サーバ間の信頼。複数ユーザー利用に対応 |
| `uat`(Unified Access Tokens) | Tableau Cloud 専用、2025年12月追加の新モード | Tableau Cloud専用の統合アクセストークン方式。Cloudでサーバ間連携する際の選択肢 |
| `oauth`(Tableau OAuth) | ユーザー本人としてREST APIを呼ぶ | **ユーザーごとの権限・行レベルセキュリティ(RLS)を維持できる**。Tableau Server 2025.3以降が前提(自己ホストOAuthはTableau Server 2025.3以降。Tableau CloudはHosted Tableau MCPのOAuthを利用) |

ここがポイントで、PATやDirect Trustは事実上「単一の固定ユーザー」として振る舞うのに対し、**OAuthを使うと各ユーザーの権限・RLSがエンドツーエンドで尊重されます。** 「誰がClaude/Slack経由で聞いても、その人が見られるデータしか返ってこない」を担保したいなら、OAuth一択ということです。

### Hosted Tableau MCP で何が変わったか

当初のTableau MCPは、自分でNode.jsやDockerコンテナとして **自己ホスト** する必要があり、`stdio + PAT` のローカル実行が中心でした(このあたりは #2 で扱った世界観です)。

そこから Tableau 2025.3 で HTTPトランスポート＋OAuth に対応し、そして **Tableau 2026.2** で、自己ホスト不要・OAuth 2.1内蔵の **Hosted Tableau MCP for Tableau Cloud** がGA予定(2026年6月下旬)。これが #3 で追った「Hosted対応」の核心で、**SlackやClaudeからTableau分析へつなぐ敷居が大きく下がる見込みです。**

### Slack → Tableau MCP の接続構成(難易度順)

| 構成 | 概要 | 難易度 |
| --- | --- | --- |
| ④ Agentforce Tableau in Slack | Slackの会話に分析が直接出てくるネイティブ体験(最も製品化済み) | 低 |
| ③ Hosted Tableau MCP をリモートコネクター接続 | OAuth 2.1内蔵のホスト型に、リモートMCP対応クライアントから接続 | 低〜中 |
| ② 自己ホスト + Streamable HTTP + OAuth | 自前で多人数提供。公開到達性とOAuth実装が必要 | 中〜高 |
| ① 自己ホスト + stdio/PAT | 自分でサーバを立ててローカル検証 | 高(学習・検証向き) |

「とにかくまず触りたい」なら④、「ちゃんと権限まで効かせたい」なら③、「仕組みを自分で握りたい」なら①の自己ホスト、という住み分けになりそうです。

## 3軸を1枚に畳む：Tableau MCP × Slack 状況総まとめ

ここまで3つの軸を分けて見てきましたが、「Tableau MCPに、Slackを起点としてつなぐ」という一点に絞ると、取り得る構成は実質4つに収束します。3軸(トランスポート / 認証 / クライアント)を縦に畳み込むと、こう整理できます。

| 構成 | トランスポート | 認証(AUTH) | 繋げるクライアント | Slack起点の適性 | 位置づけ |
| --- | --- | --- | --- | --- | --- |
| ① 自己ホスト：stdio + PAT | stdio | PAT | stdio対応のみ(Claude Desktop/Code・Cursor・VS Code 等) | ✗ 直接不可(リモート専用クライアントは繋げない) | 学習・個人検証の最短路。共有・同時利用は不可 |
| ② 自己ホスト：Streamable HTTP + OAuth | Streamable HTTP | OAuth(製品の既定はPAT。Direct Trust/UATも選択可) | リモート対応クライアント全般(Claude Web・Slack・ChatGPT 等) | ○ 可(公開到達性とOAuth実装が前提) | 自前で多人数提供。ホスティング/認証/公開の手間は中〜高 |
| ③ Hosted Tableau MCP(Tableau Cloud) | Streamable HTTP(マネージド) | OAuth 2.1 内蔵 | リモート対応クライアント全般 | ◎ 運用負荷なしで繋げる | 2026.2 GA。サーバ運用不要。Tableau Cloud 前提 |
| ④ Agentforce Tableau in Slack | (Salesforce側が抽象化) | (同上・OAuth系) | Slack(ネイティブ体験) | ◎ 最も製品化された体験 | 自分でMCPを意識しない最短ルート |

この表の読み解きはシンプルです。

* **「Slackを起点にしたい」と決めた時点で、① は実質脱落します。** Slack(やWeb版Claude・ChatGPT)はリモート専用で、stdioを起動できないからです。残る選択肢は ②③④。
* **手間を最小に、今すぐ繋ぎたい** なら ④(Agentforce)か ③(Hosted Tableau MCP)。
* **各ユーザーの権限・RLSをきちんと効かせたい** なら、OAuthが乗る ②③④(とりわけ②③は明示的に設計できる)。
* **仕組みを自分で握りたい／Tableau Cloud以外で動かしたい** なら ② の自己ホスト。

つまり結論はこうです。

> **「Slack × Tableau MCP」は、3軸で言えば〔Streamable HTTP × OAuth × リモート対応クライアント〕の組み合わせに収束する。**  
> あとは、それを **自分でホストする(②)／Tableauに任せる(③)／Salesforceのネイティブに乗る(④)** のどれを選ぶか、という問いに変わります。

3軸を別々に眺めると「カオス」でも、具体のゴール(Slack起点)を1つ決めるだけで、選択肢はこれだけに絞り込める——というのが、この記事全体の一番の収穫でした。

## 総括：接続を見極めるチェックリスト

ここまでの3軸を、ひとつのフローにまとめておきます。新しいMCPサーバに出会ったら、この順に問えば「繋がるか・どうつなぐか」が見えるはずです。

![MCP接続の見極めチェックリスト](https://static.zenn.studio/user-upload/6329e744b49c-20260620.png)

実務寄りの指針として、現時点での結論を残しておきます。

* **学習・検証は `stdio + PAT` から。** Tableau MCPなら `npx -y @tableau/mcp-server@latest` ＋ Claude Desktop ＋ PAT が最短路
* **本番・多人数・Slack連携は、OAuth付きのリモート/ホスト型へ。** Tableau Cloudなら Hosted Tableau MCP(2026.2 GA、OAuth 2.1内蔵)を第一候補に
* **クライアントがOAuth非対応なら `mcp-remote` 等でブリッジ。** ただし `mcp-remote` には脆弱性 `CVE-2025-6514`(CVSS 9.6、v0.1.16で修正)の事例があったので、**バージョン固定は徹底** したいところ
* **判断を変える閾値：** 同時接続が要るようになったらPATを卒業してOAuth/Connected Appへ。SSE専用サーバを保守しているなら次に触る時にStreamable HTTPへ

## まとめ

という訳で、今回はMCP周りの情報に関して、「個別のつなぎ方」ではなく「カオスを切り分ける見方」を整理してみました。

調べる前は本当に **『よく分からん』** だったのですが、**「サーバが何を出していて × クライアントが何に対応していて × 認証がどう要求されるか」** の掛け合わせなのだ、と幾分整理出来てきた気がします。Tableau MCP × Slackのような「乱立して見える」構成も、結局はこの3軸の組み合わせの一つにすぎない、というのが今回の収穫でした。
