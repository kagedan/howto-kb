## はじめに

最近、Claude Code や Cursor を使って開発することが増えてきて、「AI との開発って結局どうやるのが正解なんだろう？」とずっと考えてました

Vibe Coding みたいに全部 AI に任せるのも楽だけど、プロダクションで動くものを作ろうとすると品質が担保できない。かといって従来通りに全部自分で書くなら AI 使う意味が薄い

試行錯誤した結果、**フェーズによって開発手法を使い分ける**のが一番しっくりきました

## なぜ2つに分けるのか

0→1 と 1→N では、リスクの種類がまったく違う

| 観点 | 0→1（MVP） | 1→N（機能追加） |
| --- | --- | --- |
| 既存コード | ない | ある（壊すリスク） |
| 既存データ | ない | ある（不整合リスク） |
| ドキュメント | これから作る | 既にあるべき |
| 影響範囲 | 全部新規だから気にしない | 既存への波及が怖い |
| 優先すべきこと | スピード + 方向性の検証 | 品質 + 安定性 |

0→1 で品質ガチガチにやると遅すぎるし、1→N で Vibe Coding すると既存機能が壊れる。だから分ける

## 2つの手法の概要

### エージェント駆動開発（0→1）

PM / Dev / QA の3つの AI エージェントがサイクルを回して、**保守性のある MVP** を高速に作る手法

ただ速く作るだけじゃなくて、**最初に品質基盤を決めて、タスクごとにドキュメントも更新する**のが特徴。MVP が完成した時点でテストもドキュメントも揃っている状態を作る

世界的には **Multi-Agent Development** と呼ばれている方向性で、Devin や Factory AI がこの領域に取り組んでいる

→ 詳細は[エージェント駆動開発編](https://zenn.dev/m_525/articles/098c7fcfc80471)で

### ドキュメントファースト開発（1→N）

既存プロダクトへの機能追加・保守で使う手法。**ドキュメントを Single Source of Truth にして、実装前にドキュメントを更新してから開発する**のが特徴

Stripe がやっている Spec-Driven Development や、Google の AI-Augmented Engineering と同じ方向性

→ 詳細は[ドキュメントファースト開発編](https://zenn.dev/m_525/articles/90d4981995b8b0)で
