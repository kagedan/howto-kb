広告運用において、A/Bテストは「やったほうがいい施策」ではなく「やらなければ改善が止まる前提条件」です。クリエイティブ、配信ターゲット、LP、入札戦略——Meta・Google・Apple Search Ads・TikTokのいずれを回していても、私たちは日々無数の比較検証を行っています。

ところが、そのテスト結果の「分析」と「レポート化」の工程は、いまだに多くの現場で手作業のまま残っています。本記事では、Anthropic社のコマンドラインツール「Claude Code」を使い、A/Bテスト結果の統計的な分析からスプレッドシートへの出力までを自動化する実践的な方法を、具体的なプロンプトとともに解説します。

  

## なぜ広告A/Bテストの「分析」は属人化・形骸化しやすいのか

現場で起きがちな問題は、大きく3つあります。

**1つ目は、媒体をまたいだ集計の手間です。** 各プラットフォームの管理画面からCSVをダウンロードし、列を揃え、CVRやCPAを計算し直す。この転記作業は地味に時間を食い、列ずれや日付の取り違えといったヒューマンエラーの温床にもなります。

**2つ目は、「見た目の差」を結果と取り違えてしまう罠です。** 「BパターンのCVRがAより高いから、Bの勝ち」——この判断は、実は危険です。サンプルサイズが小さければ、その差は単なる偶然のブレ（誤差）かもしれないからです。統計の世界では、観測された差が偶然では説明しづらいと判断できる状態を「統計的に有意」と呼びます。一般的には、有意水準5%（p値が0.05未満）を基準に判定し、これを満たさない差は「効果があったとは言えない」と解釈するのが正しい姿勢です。この検定を省略したまま勝敗を決めてしまうと、誤った施策に予算を投下し続けることになりかねません。

**3つ目は、属人化です。** 「あの人にしか分析の手順がわからない」「レポートのたびに同じ計算をゼロからやり直している」——分析ロジックが個人の頭の中やExcelの数式に閉じていると、再現性も引き継ぎ性も失われます。

Claude Codeは、この3つの問題に対して、それぞれ明確な解を持っています。

## なぜA/Bテスト分析にClaude Codeが向いているのか

ChatGPTやClaudeのチャット版でもデータ分析は可能ですが、A/Bテストの定常運用という文脈では、ターミナル上で動くエージェントであるClaude Codeに固有の強みがあります。

* **ローカルファイルを直接読み込める**：管理画面からエクスポートしたCSVを、フォルダごと「このフォルダのファイル全部読んで」で渡せます。コピペが不要です。
* **統計検定をコードとして実行する**：CVRやCTRの差が有意かどうかを、Python（scipyやstatsmodelsなどのライブラリ）を使ってZ検定やカイ二乗検定で厳密に計算させられます。「なんとなく多い」ではなく、p値と信頼区間で判断できます。
* **分析ロジックがスクリプトとして残る**：一度作った分析処理はPythonスクリプトとして保存できるため、翌週・翌月も同じ処理を再利用できます。属人化が解消され、誰が回しても同じ結果になります。
* **そのままスプレッドシートに出力できる**：結果をExcelファイルとして生成したり、Googleスプレッドシートに直接書き込んだりできます。
* **監査証跡が残る**：どのファイルを、どんなロジックで、どう計算したか——会話ログとスクリプトに記録が残るため、後から検証できます。クライアントワークでは特に重要なポイントです。

要するにClaude Codeは、「統計の専門知識を持ったアナリストが、毎回同じ品質で、再現可能な形で分析してくれる」状態を、コマンドラインの中に作れるツールだと言えます。

## 実践ワークフロー：CSVから統計分析まで

ここからは具体的な手順です。Claude Codeはターミナルで動くツールで、Node.jsがインストールされた環境に以下のコマンドで導入できます。

```
npm install -g @anthropic-ai/claude-code
```

導入後は claude コマンドで起動します。（インストールの詳細手順については、別記事の非エンジニア向けガイドも参考にしてください）

### Step 1：データを用意する

各媒体の管理画面から、A/Bテストの結果をCSVでエクスポートします。最低限、以下の列があれば分析できます。

```
パターン,インプレッション,クリック,コンバージョン,コスト
A,52000,1040,52,156000
B,51500,1133,78,158000
```

「パターン」「試行回数（インプレッションやクリック）」「成功回数（クリックやコンバージョン）」の3要素が揃っていれば、検定にかけられます。

### Step 2：統計的有意差まで含めて分析させる

ここがこの記事の核心です。単に「分析して」と頼むのではなく、**統計的有意性の判定まで明示的に指示する**のがプロのプロンプトです。以下は実際に使えるプロンプト例です。

```
このフォルダの ab_test_creative.csv を読み込んで、以下を分析してください。

1. 各パターンのCTRとCVRを算出（表形式）
2. AパターンとBパターンのCVRの差について、Z検定で
   統計的有意差を判定（有意水準5%）。p値と95%信頼区間を明記
3. 現在のサンプルサイズで、相対改善率10%を検出するのに
   十分な検出力（80%）があるか評価。不足する場合は
   必要なサンプルサイズも提示
4. 上記を踏まえた、運用上の推奨アクション（B採用 / 継続観測 / 判断保留）

計算ロジックはPythonスクリプトとして保存し、来週も同じ
CSVフォーマットで再利用できる形にしてください。
前提条件や仮定はすべて冒頭に明記してください。
```

このプロンプトのポイントは3つです。**第一に、検定手法・有意水準・信頼区間を指定している**こと。AIに「有意かどうかは雰囲気で決めないで」と釘を刺しています。**第二に、サンプルサイズの妥当性まで評価させている**こと。差が出なかった場合に「効果がない」のか「まだデータが足りない」のかを切り分けられます。**第三に、ロジックをスクリプトとして残させている**こと。これで翌週からはCSVを差し替えるだけになります。

Claude Codeは、計算結果のサマリーと、再利用可能なスクリプトの両方を返してくれます。

### Step 3：複数媒体・複数テストをまとめて回す

運用が複雑になってきたら、媒体ごとのCSVをフォルダにまとめて投入し、横断的に処理させることもできます。

```
media_data フォルダ内の各CSV（meta.csv, google.csv, asa.csv）を
それぞれ読み込み、媒体ごとにA/Bの有意差を判定してください。
有意差が出ているテストだけを抽出し、媒体・テスト名・勝ちパターン・
改善率・p値を一覧にまとめてください。
```

「有意なものだけ抽出」という指示が効くと、レビュー会議で「で、結局どれが本物の勝ちなの？」という問いに即答できるアウトプットになります。

## スプレッドシートへの出力：3つの方法

分析が終わったら、結果をスプレッドシートに落とし込みます。目的に応じて3つの方法があります。

### 方法A：Excelファイルを生成させる（最も手軽）

Claudeは、数式・条件付き書式・複数シートを備えたExcelファイルをゼロから生成できます。「分析結果を、サマリーシートと生データシートに分けたxlsxファイルにして」と頼めば、そのまま配布・提出できるファイルが手に入ります。導入のハードルが最も低く、まず試すならこの方法です。

### 方法B：Googleスプレッドシート連携（コネクタ / MCP）

チームで共有しているGoogleスプレッドシートに直接書き込みたい場合は、連携の仕組みを使います。読み取り・要約が中心ならGoogleコネクタが手軽ですが、毎週の繰り返し処理や安定した書き込み運用を考えるなら、MCP（Model Context Protocol）経由の接続が向いています。たとえば「共有スプレッドシートの『週次A/B結果』シートに、今週の有意差判定結果を追記して。書き込む前にサマリーを見せて確認を取って」といった運用が可能になります。

### 方法C：Google Workspace CLI（gws）で直接操作する

2026年に入ってから注目されているのが、Googleが公式リポジトリで公開しているコマンドラインツール「Google Workspace CLI（gws）」との組み合わせです。Claude Codeから gws コマンドを実行させることで、スプレッドシートの構造を直接読み取らせ、それに合ったコードを生成させられます。手動でフォーマットを説明するより精度が高く、効率的です。

ただし、執筆時点ではv1.0前の開発段階のツールであり、Google公式サポート製品ではない点に注意が必要です。破壊的変更の可能性があるため、いきなり重要なファイルで使うのではなく、まずはテスト用のシートで挙動を確認してから導入することをおすすめします。

## 運用で押さえるべき3つの注意点

最後に、実務で使う上での落とし穴を共有します。

**1. 統計の落とし穴を理解しておく。** A/Bテストには、結果を途中で何度も覗いて有意になった瞬間に止めてしまう「覗き見問題（peeking）」や、たくさんのパターンを同時に比較すると偶然の有意差が紛れ込む「多重比較」といった落とし穴があります。Claude Codeに検定を任せても、テスト設計そのものが甘ければ正しい結論は出ません。「いつ止めるか」「いくつ比較するか」は人間が事前に決めるべき領域です。

**2. AIの計算は検算する。** AIが生成した数式やロジックは、条件の解釈を誤ることがあります。特に重要な意思決定に関わる分析では、p値や信頼区間が手元のA/Bテスト計算ツールの結果と一致するか、最初の一回は必ず突き合わせて検証してください。スクリプト化しておけば、検算は一度で済みます。

**3. 機密データの取り扱い。** クライアントの広告データやコンバージョンデータは機密情報です。クラウドベースのAIを使う以上、自社およびクライアントの情報取り扱いポリシーに沿った運用になっているか、導入前に必ず確認しましょう。
