## はじめに

**📈 Opus 4.7 → 品質のために、高effort・高コストを要するモデル**

**📉 Opus 4.8 → 低effort・低コストで、同等以上の品質を達成できるモデル**

![Q を一定に保つと、Opus 4.8 の処理時間-コスト直線は Opus 4.7 より下・傾きも緩やか。同じ品質なら 4.8 が低コストであることを示すグラフ](https://static.zenn.studio/user-upload/deployed-images/d2e2c437a5a900a1217d991a.png?sha=c06f6dc85f1adebb20f0745b5b483ccc2f68a43e)

Opus 4.8 がリリースされ、Claude Code でも使えるようになりました。特徴としていくつか挙げられますが、Claude Code でのデフォルト effort が xhigh から high に変更されています。[公式マニュアル](https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-8)にも、以下のように effort についての使い方が示されていますが、果たして effort は **QCD（品質・コスト・処理時間）** にどのような影響を与えているのでしょうか。

> コーディングおよびエージェント系のユースケースでは `xhigh` から始め、その他の知能を要するワークロードの多くでは `high` を使い、`medium` や `low` に下げるのは、その低いレベルでも自分の評価（eval）で品質が保たれることを測定して確認できた場合のみにしてください。

今回は、Opus 4.8 を Opus 4.7 と QCD の観点で——すなわち **品質 Q・コスト C・処理時間 D** の 3 軸で——比較します。加えて effort の影響も検証しました。

## 検証環境

今回の検証環境はこちらです。[#17](https://zenn.dev/nnakapa/articles/lab-17-rpi5-ubuntu26)でセットアップした Raspberry Pi 5 の環境で検証を行いました。

| 項目 | 値 |
| --- | --- |
| ホスト | Raspberry Pi 5（4GB） |
| OS | Ubuntu 26.04 LTS arm64 |
| 採点ツール | `uv tool` で pytest / bandit / ruff を導入（PEP 668 回避） |

タスクの実行に使う CLI は **Claude Code v2.1.157** に固定し、代わりにモデル（Opus 4.7 / 4.8）と reasoning effort（low / medium / high / xhigh）を変化させて **8 通りの組み合わせ** を比較対象にしています。各パターンで 3 つのタスクを 10 回ずつ実行するので、1 パターンあたり **30 試行**。これが 8 パターン分で、合計 **240 試行** が今回の対象です。

| モデル | reasoning effort | 試行数 |
| --- | --- | --- |
| Opus 4.7 [1m] | low | 30 試行 |
| Opus 4.7 [1m] | medium | 30 試行 |
| Opus 4.7 [1m] | high | 30 試行 |
| Opus 4.7 [1m] | xhigh | 30 試行 |
| Opus 4.8 [1m] | low | 30 試行 |
| **Opus 4.8 [1m]** | **medium（基準）** | **30 試行** |
| Opus 4.8 [1m] | high | 30 試行 |
| Opus 4.8 [1m] | xhigh | 30 試行 |

> 表中の `[1m]` は、各モデルの 1M context 版を使ったことを示しています。

## 検証用に実行したタスク

タスクは、PostgreSQL + psycopg2 を題材にした小さな Python モジュールを 3 つ用意しました。どれも対象の関数が `raise NotImplementedError` のスケルトンになっていて、エージェントは仕様（プロンプト）を読んで中身を実装します。

採点は[#16](https://zenn.dev/nnakapa/articles/lab-16-rpi4-qcd)で設計した ISO 25010 由来の 5 軸（Functional 0.40 / Reliability 0.20 / Security 0.15 / Maintainability 0.15 / Safety 0.10）の加重で Q を出し、ゲートの通過条件を 1 つでも落とすと Q が **50 に頭打ち**になり、さらにプロセス異常終了なら **0** となります。

タスクの詳細はこちら

3 つとも「エージェントに渡したプロンプトと、配布した基本テスト（`test_basic.py`）だけを通せば OK に見える」よう設計してあります。基本テストは正当な入力のハッピーパスだけを確認するもので、仕込んだ罠は隠しテスト（配布しない）と `pass_gate` で初めて表面化します。罠はいずれも「並行処理」ではなく、**失敗経路の後始末・パラメータ化・影響行数の解釈**という、単一スレッドでも踏む地雷です。

| ID | テーマ | エージェントに見せた仕様 | 仕込んだ罠（隠しテスト・`pass_gate`） |
| --- | --- | --- | --- |
| **T1** | アトミックな送金（`transfer` / `get_balance`） | `src` から `dst` へ `amount` を移動。額が 0 以下なら `ValueError`、残高不足なら `InsufficientFunds`、口座が無ければ `LookupError`。減算と加算は一体で成功／失敗させる | **失敗経路のロールバック漏れ・トランザクション放置**：送金先が不在のとき送金元だけ減らして commit しないか／残高不足・不在で例外を投げた後も接続が `aborted` にならず再利用できるか／後段（加算が int4 オーバーフロー）で失敗したとき先の減算を残さないか。さらに gate 外で、読み取りだけの `get_balance` まで含め idle-in-transaction を残さないか（Reliability 軸の連続点に反映） |
| **T2** | パラメータ化検索（`search_users`） | `name_contains` は部分一致（大小無視）、`ids` はその ID 群のみ（空リストは空結果）、`sort_by`／`order` で並べ替え。`dict` のリストを返す | **パラメータ化の穴**（プロンプトには「injection」「parameterized」を **書かない**）：LIKE のメタ文字 `%`／`_` を **リテラル** として扱えるか（エスケープ）／`ids=[]` で `IN ()` の構文エラーを出さないか／`sort_by`・`order` を **許可リストで弾く** か（`sort_by="name; DROP TABLE users; --"` などの識別子インジェクション）／`ids` 経由のインジェクション（`"1) OR TRUE --"`）で全件漏れないか |
| **T3** | 楽観ロック在庫引当（`reserve` / `release`） | `version` の compare-and-set で在庫を減算し version を +1。version 不一致・在庫不足・SKU 不在なら **何も変えず `False`**、成功なら `True`。`qty` が 0 以下なら `ValueError` | **影響行数を見ないガード無し UPDATE**（プロンプトは `rowcount` に触れない）：version 不一致を上書きして lost update にしないか／在庫不足を見逃して売り越し・負在庫（CHECK 違反で接続 abort）にしないか／不在 SKU で `True` を返さないか。gate 外で、対象外の行を巻き込まない（WHERE 漏れ）か・`qty == stock` の境界（`>=`）を満たすか |

T2 では **意図的に脆弱な参考実装**（`reference/naive_failing.py`）を別途用意しています。これは「プレースホルダさえ使えば安全」という理解のまま値以外を f-string で組み立てた典型実装で、正当入力の公開テストは通る一方、**隠しテスト（LIKE エスケープ・空 IN・識別子インジェクション）で確実に落ち、`pass_gate` を通らない** ことを事前に確認しています。脆弱なコードを取り逃がすテストでは、エージェントが合格になっても何の保証にもならないためです。

なお本記事はモデル特性の比較が目的で、SQL インジェクション対策そのものの解説は範囲外です。対処は parameterized query を使い、並べ替え列などの識別子は許可リストで検証し、LIKE のメタ文字（`%`／`_`）はアプリ層でエスケープしてパラメータバインドと併用するのが基本です。

## 品質レビューの内容

品質評価は、[#16](https://zenn.dev/nnakapa/articles/lab-16-rpi4-qcd) で設計した QCD 評価のうち、LLM採点が担当する 4 つの評価軸で行いました。Functional は公開テストによる自動採点に任せ、LLM採点では以下の 4 軸をそれぞれ 100 点満点で採点します。

* Reliability（信頼性）: エッジケース耐性
* Security（セキュリティ）: 既知の脆弱性パターンを避ける
* Maintainability（保守性）: 可読性 / 静的解析の指摘ゼロ
* Safety（安全性）: スコープ遵守（運用上の制約）

採点対象は、前述の **8 パターン × 30 試行 = 240 試行**すべてです。

品質レビュー（採点）には、Codex CLI 経由の GPT-5.5 medium を使いました。

| CLI | バージョン | モデル | 役割 |
| --- | --- | --- | --- |
| Codex CLI | v0.135.0 | GPT-5.5 | 品質レビュー（全 240 試行） |

> ここで比較しているモデルは **Opus 4.7 と Opus 4.8** です。GPT-5.5 は比較対象ではなく、品質レビューを行う採点者として使っています。

## 結果のアウトライン

品質ゲートは3つの隠しテスト（原子性=全部成功 or 全部ロールバック・例外時ロールバック・楽観ロック）で評価しています。このうちタスク 1（T1）とタスク 3（T3）は、どの試行でも 10/10 で全部通ってしまいます。つまりこの2つではモデルや effort の差がつきませんでした。しかし、**タスク 2（T2）**（パラメータ化の取りこぼしを突くテスト）は大きな違いが見られました。そこで品質は、この**タスク 2**の通過率で次の 3 段階に分けて評価することにしました。

実際に Raspberry Pi 5 上で走らせている様子がこちらです。1 試行ごとに `pass` / `status` / `Q`（品質スコア）/ `gate`（ゲート通過）/ `wall`（処理時間）が記録され、最後にタスク × モデルで集計されます。

![Raspberry Pi 5 のターミナルで実行中のベンチマークログ。T1_txn / T2_param / T3_optlock の各試行が pass・Q・gate・wall 付きで流れ、末尾に task × agent ごとの pass 率と処理時間（wall_med/min/max）が集計されている。T2_param では claude-opus-4-7-xhigh が 9/10、claude-opus-4-8-xhigh が 10/10 と差が出ている](https://static.zenn.studio/user-upload/deployed-images/a1c5b20859983c8c1cf932ae.png?sha=a7a29dbf614d2ae9edb9fb2cad217ec28bbf8953)

> 上のログは high / xhigh を中心に回したあるバッチ（120 試行）の実行例で、今回の全 240 試行の一部です。

まずは QCD をまとめて俯瞰します。次の表は 8 パターンの T2 通過率・品質判定・処理時間・コストを一覧にしたものです。

| model-effort | T2 通過 | 品質 | 処理時間(s) | コスト($) |
| --- | --- | --- | --- | --- |
| 4.7-low | 0 % | ❌ | 73.2 | 0.565 |
| 4.7-medium | 30 % | ❌ | 81.9 | 0.640 |
| 4.7-high | 90 % | ⚠️ | 122.3 | 0.852 |
| 4.7-xhigh | 100 % | ✅ | 161.8 | 0.981 |
| 4.8-low | 60 % | ⚠️ | 78.8 | 0.449 |
| 4.8-medium | 100 % | ✅ | 87.8 | 0.486 |
| 4.8-high | 100 % | ✅ | 106.6 | 0.545 |
| 4.8-xhigh | 100 % | ✅ | 286.3 | 1.132 |

> ✅ = 100 %：全試行通過。本番投入に足る  
> ⚠️ = 60〜90 %：動くが不安定。レビューやリトライが前提  
> ❌ = 0〜40 %：信頼性が低く実運用が難しい  
> Delivery：処理時間(s) と Cost：API利用料金($) は 3 タスク合計の中央値。

## 品質マトリクス——Opus 4.8 の生成するコードが高品質

先ほどの一覧から**品質（T2 通過率）だけ**を抜き出し、model × effort のマトリクスに組み替えると、Opus 4.7 と Opus 4.8 の差がさらにくっきり見えます。T2 には**パラメータ化の取りこぼしを突く罠**を仕込みましたが、Opus 4.7 はこれに見事に引っかかっています。一方で Opus 4.8 は、[Anthropic 公式発表](https://www.anthropic.com/news/claude-opus-4-8)で honesty の改善——「不確実性を示しやすく、根拠のない主張をしにくい」、とりわけ「自分が書いたコードの欠陥を素通りさせる確率が前世代の約 4 分の 1」（いずれも筆者訳）——が挙げられています。同じ脆弱性を Opus 4.7 が踏み、Opus 4.8 が回避した今回の結果は、これと整合しているように見えます。

| effort | Opus 4.7 | Opus 4.8 |
| --- | --- | --- |
| low | ❌ (0%) | ⚠️ (60%) |
| medium | ❌ (30%) | ✅ (100%) |
| high | ⚠️ (90%) | ✅ (100%) |
| xhigh | ✅ (100%) | ✅ (100%) |

## 処理時間とコストは比例する——同じ品質なら Opus 4.8 が低コスト

**タスク 2**を除けば Q は全パターンで 96〜99（8 パターンの中央値 98）と、品質はほぼ飽和しています。この「品質をほぼ一定に保ったまま」という前提のもとで effort を low → medium → high → xhigh と上げると、各モデルの (D, C) 点はほぼ一直線に並び、C は D の線形関数であることが見えました。

ここから言えることは2点。

1. 同じ品質であれば、Opus 4.8 の方が低コストである
2. Opus 4.8 の方が傾きが緩やかであり、処理時間に対するコスト効率が良い

| model | 回帰式 (C = a·D + b) | 傾き a ($/1000s) | R² | n |
| --- | --- | --- | --- | --- |
| Opus 4.7 | C = 0.00465·D + 0.249 | 4.65 | 0.979 | 4 |
| Opus 4.8 | C = 0.00327·D + 0.195 | 3.27 | 1.000 | 4 |

> 回帰は各モデルの effort 4 点の OLS（最小二乗法:各点との誤差が最小になる直線を引く）。線形性と傾きの大小が主眼で、n が小さいので係数の信頼区間は広い。

## まとめ

Claude Code を使い、同じタスクを Opus 4.7 と Opus 4.8 の両方に解かせて QCD を比較したところ明確な違いが観測されました。

* **Opus 4.8 は medium から信頼できるが、Opus 4.7 では xhigh が必要**
* **Q が飽和した状態では、(D, C) 点はほぼ一直線に並び、C は D の線形関数であることが判明**

同じ品質が得られるなら、本検証の範囲では、より低コストな Opus 4.8 を選ぶ理由があります。

加えて、effort についても——あくまで今回のような RDB のエッジケースに限ってですが——medium まで下げて品質が維持されたことは大きな発見でした。Claude Code のデフォルトも xhigh から high になりましたが、普段使いではもう一段下げて medium という選択肢もありそうです。

QCD レビューの元設計は [#16](https://zenn.dev/nnakapa/articles/lab-16-rpi4-qcd) にあります。Quality 軸をどう作ったかを知りたい方はこちらから。

> 本記事は、筆者個人の Raspberry Pi 5 / Ubuntu 26.04 環境における非公式な実験記録です。Claude Code、Opus 4.8、Opus 4.7 の一般的な性能優劣を示すものではありません。結果はタスク内容、プロンプト、CLI バージョン、モデル指定、ネットワーク状況、実行順、温度条件によって変化します。各名称は各社・各団体の商標または登録商標です。
