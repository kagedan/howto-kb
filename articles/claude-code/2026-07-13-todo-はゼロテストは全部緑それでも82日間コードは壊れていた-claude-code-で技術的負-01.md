放置マーカー（TODO / FIXME）は0件。テストは91ファイル・1,076関数あって、CI は緑。技術的負債の指標として私が見ていたものは、全部きれいでした。

それでもこのプロジェクトは、判断ロジックの一本が **82日間、数式的に無意味な結果を返し続けていました**。今日 Claude Code に監査させて、初めて気づきました。

対象は個人開発の競馬予想アプリ（複数の分析エージェントが議論して買い目を出すもの）です。この記事は、その監査で何が出てきたか、そしてなぜ「気づいたら直す」を仕組みに置き換えたのかの話です。

## 静かに壊れていたのは、行数の多いファイルではなかった

監査で最も重い指摘は、肥大化でも複雑度でもありませんでした。

時系列で書くとこうなります。4月21日、古い予測モデルを廃棄しました。このとき、そのモデルのスコアを受け取っていたエッジ計算モジュールは、2つある呼び出し元の両方から空の辞書を渡されるようになります。5月26日、そのモジュールの docstring に「このロジックは今、縮退している」と自分で書き込みました。そしてその2日後、**呼び出し元が2つあるうちの片方だけ**が、新しいモデルの信号を渡すよう修正されます。

もう片方——公式の分析フローのほう——は、取り残されました。空の辞書を受け取り続け、計算は常に同じパターンへ縮退し、出力は意味を失ったまま。それが4月21日から監査当日まで、82日間続いていたわけです。

たちが悪いのは、この状態が**どの計器にも引っかからない**ことです。例外は出ません。テストは通ります。lint も通ります。画面には数字が出るので、動いているようにしか見えない。壊れ方として、これがいちばん静かです。

正直に補足しておくと、金銭的な実害はありませんでした。この経路を本番で回して実際に賭けていたわけではないからです。ただ、実害が無かったのは運用形態のおかげであって、気づけたからではない。そこは分けて考えないといけないと思っています。

## 「きれい」は「測っていない」と見分けがつかない

もうひとつの指摘は、私の思い込みを正面から折るものでした。

このリポジトリは lint 抑制コメント（`# noqa` や `@ts-ignore` の類）が極端に少なく、私はそれを「行儀よく書けている証拠」だと思っていました。監査の結論は逆です。

* `pyproject.toml` に ruff の設定セクションが存在しない
* CI で回しているのは backend のテストだけで、lint も型チェックも実行していない
* frontend にいたっては CI 自体が無い

試しに複雑度の閾値を10に設定して測り直すと、違反が66件出ました。最悪の関数は複雑度48です。抑制コメントが少なかったのは、行儀が良かったからではなく、**誰も何もチェックしていなかったから**でした。

きれいに見えることと、測っていないことは、外からは区別がつきません。

## 計器そのものが汚れていた

さらに笑えないことに、私が普段眺めていた計器も壊れていました。

無人の週次スキャナが「400行超のファイル＝57本」と報告していたのですが、監査によれば、そのうち24本はテストコード（16本）、廃棄して倉庫に放り込んだ古いコード（7本）、OpenAPI から自動生成された型定義（1本）でした。人が手で保守するコードだけ数えると **33本**です。

除外ルールを入れて測り直したら、別のプロジェクトでも閾値超えの警報が1件消えました。危険水域にいたのは実体ではなく、数え方のほうだったわけです。負債を測る道具が、それ自体、負債になっていました。

| 監査で出たもの | 私の計器に映るか |
| --- | --- |
| 判断ロジックの片系統が縮退（82日） | **映らない**（例外なし・テスト緑） |
| lint が CI で強制されていない | **映らない**（抑制コメントは逆に少なく見える） |
| 予測を束ねる中枢モジュールがテストから参照ゼロ | **映らない**（テスト総数は1,076件ある） |
| 廃棄コード 70MB・13,834行が tracked のまま82日 | 映る（が、放置される） |
| 依存112件すべてが outdated・脆弱性 critical 1件 | 映る（が、放置される） |
| 1,745行の肥大ファイル・複雑度48の関数 | 映る |

上の3行が、この監査でいちばん価値のあった部分です。3行目は冒頭のエッジ計算とは別のモジュールで、各アナリストを束ねる本体（563行）が、テストから import すらされていませんでした。テストは1,076件ある。ただ、当たっていない場所には1件も当たっていない。

## 警告バナーは、もう誰も読んでいない

以前は、編集のたびに「差分が大きいので整理しませんか」と促すバナーを出していました。これは完全に死にました。

理由は単純で、**受け手がいなくなった**からです。少なくとも私のように auto mode 中心で回している運用では、私はコードを手打ちしないし、Claude も止まらず先へ進む。人間向けに書かれた警告は、人間がその場にいないので誰にも読まれない。ナッジは、読む人がいて初めてナッジとして機能します。

そこで、品質の維持を2つのカデンスに分けて、どちらも歯を持たせることにしました。

|  | 何を守るか | いつ発火するか | どう効かせるか |
| --- | --- | --- | --- |
| **micro** | 挙動を変えない整理（重複・足場コード・過剰設計） | コミット境界 | 一定行数を超える差分の初回コミットを一度だけ拒否し、整理を挟ませる |
| **macro** | 負債の棚卸し（肥大化・死蔵・依存・テスト空白） | 週次 or 閾値超過 | 無人スキャン → セッション開始時に催促 → 監査コマンド |

micro は「一度だけ拒否する」のが肝です。拒否のときにクールダウンの印を残すので、整理してから再実行すれば必ず通ります。無限ループもしないし、「整理不要」と判断すればそのまま通せる。それでもコミット境界で必ず一度は俎上に載るという点で、無視できるバナーとは別物になりました。

macro のほうは、週に一度 launchd が無人でスキャンします。やるのはローカルでの軽い集計だけで、ネットワークにも出ないし、トークンも消費しません。閾値を超えたリポジトリは記録され、次にそのプロジェクトを開いた瞬間、セッション開始時に催促が出ます。

```
🩺 [dev-health] 負の遺産の定期監査が来ています（負債スコア6/12・god:57 TODO:0）。
   /dev-health で 肥大化・重複・死蔵・依存陳腐化 を検知→提案（issue化）できます
   （適用は承認＋テスト緑をゲート）。不要ならスルーでOK。
```

この god:57 が、さっきの汚れた計器です。催促に応じて監査コマンドを叩くと、複数のサブエージェントが観点ごとに並列で走ります。今回は「死蔵・重複」「肥大化・複雑度・構造」「依存・テスト・lint」の3体で、それぞれ実際にコマンドを叩いて実測値を取ってくる。冒頭の縮退したロジックは、そのうち2体が独立に同じ箇所を指摘してきたものです。

## 見つけさせるが、片付けさせない

ここが設計上いちばん悩んだところです。

監査に issue を書かせるところまでは全自動にしました。今回は6本立ちました。一方で、修正を勝手に適用させることはしません。負債の解消はしばしば挙動を変えます。それを承認なしでAIに走らせるのは、私が最も見たくない事故です。

なので解消フェーズには関門を置きました。承認された項目だけ、feature ブランチで作業し、フルテストが緑になるまでコミットせず、PR にして事後レビューする。AIに任せるのは**発見と提案**まで、判断と適用は人間の手前に残す。この線引きは、「守り」の投資ツールで生成AIを判断から外したのと同じ理屈です。

そしてこの割り切りには、地味な副作用があります。今日立った6本の issue は、まだ1本も解決していない。負債は依然そこにあります。仕組みが変えたのは、負債が消えたかどうかではなく、それが見えているかどうかのほうです。

## 計器の外側で、負債は育つ

82日のあいだ、私はこのプロジェクトを何度も触っていました。機能も足したし、テストも増やした。それでも、静かに壊れた一本の経路には気づかなかった。見ていた計器——TODOの数、テストの本数、CIの色——が、全部そこを映していなかったからです。

負債は、自分が測っている指標の外側で育ちます。だから、たまに指標の外を覗きにいく仕組みが要る。しかもそれは、私の善意や勤勉さに依存しない形でなければ、いずれ形骸化したバナーと同じ運命をたどる。

そう考えて「片付けろ」と促す仕組みを自分で作り、そして今日、その仕組みに促されて監査を回したら、82日壊れていたコードが出てきました。次の82日で何が育つのかは、まだ誰も知りません。だから来週も、あの催促は出ていいと思っています。
