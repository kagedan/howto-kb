## TL;DR

* Node.jsのJSONLパース+集計処理をRust CLIに切り出したら**4.9倍高速化**(105ms→21ms)
* 「バックエンド全体をRust(Axum)に書き換える」までやったら**運用が複雑になって最終的に削除**した
* そして高速化した集計は、速いだけで**2.16倍過大計上していた**。Claude CodeのJSONLは同じエントリを複数回書き込むため、`message.id + requestId` での重複排除が必須
* 残った構成は「ホットパスだけRust CLI + フォールバック + **Node/Rust両実装の出力一致を検証**」。これが費用対効果の最適解だった

## 背景

Claude Codeの使用量を可視化するローカルダッシュボード(Express + React)を作っています。データソースは `~/.claude/projects/` 配下のJSONLファイル群で、リクエストのたびに全ファイルをパースして日別・月別・モデル別に集計します。

データが1ヶ月分溜まった時点で、この集計がNode.jsで**約105ms**。ページを開くたびに全エンドポイントが叩かれるので、体感でもっさりし始めました。ログは今後も増える一方なので、Rustで書き換えてみることにしました。

## まずベンチマークを取る

いきなり書き換えず、処理を分解して測りました。

| 処理 | Node.js | Rust | 高速化倍率 |
| --- | --- | --- | --- |
| File I/O | 22.4ms | 11.0ms | 2.0倍 |
| JSONパース | 24.2ms | 5.2ms | **4.7倍** |
| データ集計 | 42.4ms | 5.3ms | **8.0倍** |
| 合計 | **104.9ms** | **21.4ms** | **4.9倍** |

わかったこと:

* 最大のボトルネックはI/Oではなく**集計ループ**(8倍差)。V8のオブジェクト生成・GCコストがモロに効く部分
* JSONパースも`serde`が圧倒的
* File I/Oは2倍しか変わらない(結局ディスクが律速)

「I/Oバウンドだからどうせ変わらないでしょ」と思っていたら、実際はCPUバウンドだったパターンです。**測ってよかった**。

## Phase 1: Rust CLIプロセッサ(これが正解だった)

最初のステップは、集計処理だけをRustのCLIバイナリに切り出す構成です。`clap`で引数を受けて、`serde`でJSONLを行パースして、集計結果をJSONでstdoutに吐くだけの素朴なCLI。Node.js側は`execFileSync`で呼び出します。

```
// Node.js側: フォールバック付きで呼ぶ
function processProjectDataWithRust() {
  try {
    const result = execFileSync(RUST_PROCESSOR_PATH,
      ['--projects-path', CLAUDE_PATHS.projects],
      { encoding: 'utf8', maxBuffer: 50 * 1024 * 1024, timeout: 30000 });
    return JSON.parse(result);
  } catch (error) {
    return null; // → Node.js実装にフォールバック
  }
}
```

ポイントは**バイナリが無ければ(またはコケたら)黙ってNode.js実装にフォールバック**すること。`cargo build --release` した人だけ速くなり、していない人・CIでも壊れない。プロセス起動+JSON往復のオーバーヘッドを含めても46ms、集計だけなら21msで、体感は明確に変わりました。

## Phase 2: Axumでバックエンド全体をRust化(やりすぎた)

気を良くして、次はAPIサーバーごとRustにする実験をしました。Axumで主要4エンドポイントを実装し、Expressからプロキシする構成です。ベンチ上はこれも速く(プロキシ経由でもNode比2.8倍)、distrolessイメージで15MBになり、数字だけ見れば大成功でした。

### 何が問題だったか

* **Docker構成が壊れた**: 開発用composeにrust-backendコンテナを足したら起動が安定せず、結局`USE_RUST_BACKEND=false`で無効化して塩漬けに
* **二重実装の維持コスト**: エンドポイントを増やすたびにNode.jsとRustの両方を触ることになる。個人開発でこれは重い
* **効果が薄い**: ボトルネックは集計処理であって、HTTPハンドリングではない。Phase 1で美味しいところは食べ終わっていた

半年後、休眠したままのAxumバックエンド・専用Dockerfile群・プロキシ分岐を**全部削除**しました(git履歴には残っています)。リポジトリから3,600行が消えて、見通しは劇的に良くなりました。

## そして最大の問題: 速くなった集計は、間違っていた

ここからがこの記事の本題です。公開前のレビューで「**JSONLの重複エントリをdedupしていますか?**」という指摘を受けました。していませんでした。NodeもRustも。

### Claude CodeのJSONLは同じ行を何度も書く

Claude Codeはストリーミング中に、**同じAPI応答(同じ `message.id` と `requestId`)を、`output_tokens` が漸増する形で複数回JSONLに書き込みます**。さらにセッションのfork / resume / compact時には、同じターンが複数のトランスクリプトファイルに複製されます。素朴に全行を合算すると、盛大に重複計上します。

これは私の環境固有の話ではなく、[claude-codeリポジトリにissueが立っている既知の挙動](https://github.com/anthropics/claude-code/issues/5034)で、[ccusageでもdedup戦略が議論されています](https://github.com/ryoppippi/ccusage/issues/888)。

手元の1ヶ月分を実測した結果:

```
assistant行(usage付き): 9,799行
ユニークな (message.id, requestId): 4,389キー
重複度: 2回や3回の重複が全体の半分超 / 最大23回=1キー

素朴合算:   3,804,727,942 トークン (約38.0億)
重複排除後: 1,762,902,617 トークン (約17.6億)
→ 2.16倍の過大計上
```

ダッシュボードは「1ヶ月で36億トークン」と表示していましたが、実態は17.6億でした。**4.9倍速く、2.16倍間違った数字を出していた**わけです。

### Rustでの重複排除の実装

対策は、キーごとに**トークン合計が最大のエントリ(=ストリーミングの最終値)だけを採用する**こと。fork複製は同一の完成値なのでどれを取っても同じ、ストリーミング途中の部分値は最終値より必ず小さいので、largest-winsで両方のケースを一度に潰せます。

なお「最初に見たエントリを採用(first-wins)」だと今度は**過少計上**になります。最初のエントリはストリーミング途中の部分値のことが多いからで、実際に[ccusageのissue #888](https://github.com/ryoppippi/ccusage/issues/888)では first-wins 実装が output\_tokens を大幅に取りこぼすと報告されています。採用すべきは last-wins か largest-wins です。

```
let key = match (&message_content.id, &msg.request_id) {
    (None, None) => {
        fallback_key += 1;
        format!("__nokey__:{}", fallback_key) // キー無し行はそのまま採用
    }
    (mid, rid) => format!("{}:{}",
        mid.as_deref().unwrap_or(""), rid.as_deref().unwrap_or("")),
};

match dedup.get(&key) {
    Some(existing) if existing.usage_total > candidate.usage_total => {}
    _ => { dedup.insert(key, candidate); } // 大きい方=最終値が勝つ
}
// 集計は dedup.values() に対する第2フェーズで行う
```

集計を「行を読みながらインクリメンタルに加算」から「**収集+重複排除 → 集計**の2フェーズ」に変える必要があるのが実装上のポイントです。

### ついでに見つかった第2の間違い: モデル単価

修正のためにコードを精読して、もうひとつ発見しました。Rust移植版のコスト計算にこう書いてあったのです。

```
// 簡易的なコスト計算（モデル別の価格は省略）
let cost = (input_tokens as f64 * 0.003 / 1000.0) + ...
```

**全モデルをSonnet相当の固定単価で計算していました**。Opusの実単価はこの1.67倍です。移植時に「後で直す」つもりの簡易実装が、フォールバック設計のせいで誰にも気づかれず本番の数字を出し続けていた——高速化の皮肉な副作用です。Node側にも別の罠があり、価格表にない新モデルが古いOpus単価($15/$75)にフォールバックして3倍過大になっていました。

### 検証: 二重実装を逆手に取る

修正後の検証には、皮肉にも「二重実装の維持コスト」が役立ちました。**NodeとRustが独立に同じ答えを出せば、実装ミスの可能性はかなり潰せる**からです。

```
Node : days=29 totalTokens=1,762,902,617 cost=$1560.94
Rust : days=29 totalTokens=1,762,902,617 cost=$1560.94
```

1トークン・1セントまで完全一致。さらに集計コードと無関係な使い捨てPythonスクリプトでも独立に計算し、同一スナップショットで3実装すべて 1,762,902,617 トークンで一致しました(なお、この記事の執筆作業自体がClaude Code上で行われてカウントされ続けるため、端数は計測のたびに少しずつ増えます)。

## 学んだこと

**1. 書き換える前に処理を分解して測る。** 「I/Oバウンドに見えてCPUバウンド」は普通にある。

**2. 「全部Rust化」より「ホットパスをCLIに切り出す」。** `execFileSync` + JSON往復で十分速い。フォールバック付きなら実験も怖くない。

**3. 高速化の前に、数字が正しいかを疑う。** 間違った集計を8倍速くしても、間違った答えが8倍速く出るだけ。データソースの仕様(今回ならJSONLの重複書き込み)は、パフォーマンスより先に検証すべきでした。

**4. 移植は、元実装の罠も一緒に移植する。** dedup漏れはNode実装から忠実に移植されていた。「簡易実装(TODO)」はフォールバックの影に隠れて生き残る。

**5. 二重実装は検証資産になる。** 維持コストと引き換えに、独立した実装同士の突き合わせという最強の回帰テストが手に入る。片方を消すなら、消す前に一致を確認してから。

**6. 動かなくなった実験は消す。** git履歴が全部覚えていてくれる。

## 現在の構成

```
リクエスト → Express → rust-processor(あれば) → JSON
                    ↘ Node.js実装(フォールバック)
              ※ 両実装とも message.id + requestId で重複排除
```

Phase 1のCLIだけが生き残り、今日も静かに21msで——今度は正しい数字を——集計しています。
