MarkdownとHTML。どちらも決して新しい技術ではありません。それなのに生成AIやAIエージェントが広がった今、この二つの形式を目にする機会が再び増えています。

ChatGPTやClaudeに文章を頼むと、見出しやリストを含むMarkdownが返ってきます。コーディングエージェントは`SKILL.md`から作業手順を読みます。その一方で、Webページを表示するブラウザやコンテンツを公開するCMSが最終的に扱うのはHTMLです。

なぜ、AI時代の中心に古い二つの形式が戻ってきたのでしょうか。

この記事では、Markdownを「書くための形式」、HTMLを「届けるための形式」として捉え直します。さらに、AIエージェントのスキル、トークン消費、そしてこの記事をZennへ公開するまでの実際の作業を通して、両者の関係を考えます。

## Markdownは「書く」ための形式

Markdownの強みは、内容と最低限の構造を、少ない記号で同時に書けることです。見出し、箇条書き、リンク、表、コードブロックがありながら、元のテキストをそのまま読んでも意味が分かります。

```
## 製品の特徴

- 導入が簡単
- オフラインでも利用可能
- [ドキュメントを見る](https://example.com)
```

タグの開閉や大量の属性を意識せずに済むため、人間は文章へ集中できます。生成AIにとっても、Markdownは文章と構造をまとめて出力しやすい形式です。だからREADME、技術文書、チャットの回答、仕様書など、AIが関わる多くの場所で使われています。

### SKILL.mdに見るAIエージェントとの相性

MarkdownがAI時代に選ばれている分かりやすい例が、エージェントの「スキル」です。[ChatGPT Codex](https://developers.openai.com/codex/skills)や[Claude Code](https://code.claude.com/docs/en/skills)では、繰り返し使う作業手順をスキルとしてまとめ、`SKILL.md`を入口にできます。

```
html-to-markdown/
├── SKILL.md
├── scripts/
└── references/
```

`SKILL.md`の先頭には、スキル名や用途を示すYAML front matterを書き、その後にエージェントが従う手順をMarkdownで記述します。

```
---
name: html-to-markdown
description: Webページの本文をMarkdownへ整理する
---

# 手順

1. `main`または`article`を抽出する
2. ナビゲーション、スクリプト、スタイルを除外する
3. 見出し、リスト、表、リンクをMarkdownへ変換する
4. 変換結果を確認する
```

この形式なら、人間が手順を読んで修正でき、Gitで変更履歴も確認できます。さらに、エージェントは必要なスキルだけを選び、詳細な参考資料も必要になった段階で読み込めます。すべての知識を毎回プロンプトへ貼り付ける必要がありません。

## HTMLは「届ける」ための形式

Markdownが書き手に近い形式だとすれば、HTMLは受け取り側のシステムに近い形式です。ブラウザはHTMLの要素と階層を解釈し、見出し、段落、リンク、画像、表、フォームとして画面に表示します。

先ほどのMarkdownをHTMLへ変換すると、次のようになります。

```
<h2>製品の特徴</h2>
<ul>
  <li>導入が簡単</li>
  <li>オフラインでも利用可能</li>
  <li><a href="https://example.com">ドキュメントを見る</a></li>
</ul>
```

人間がHTMLのソースを読むという意味ではありません。読者がブラウザで読みやすいページを見るために、ブラウザやCMSがHTMLを必要とする、という意味です。

### AIエージェントがDOMを読む場面

HTMLは、ブラウザを操作するAIエージェントにとっても重要です。エージェントは画面の見た目だけでなく、DOMを使って「これはボタン」「これは見出し」「これは入力欄」と判断できます。

`main`や`article`から本文を取得し、リンクの`href`を確認し、フォームのラベルと入力欄の関係を調べる。このような精密な操作は、Markdownだけではできません。HTMLは内容を届ける形式であると同時に、Web上の対象を機械が識別するための構造でもあります。

## AI時代にMarkdownからHTMLへの変換が増える理由

以前から、Hugo、Jekyll、Astro、Docusaurusなどの静的サイトやドキュメント生成ツールは、Markdownで書いた原稿をHTMLへ変換して公開してきました。MarkdownからHTMLへの変換自体は、決して新しい処理ではありません。

変わったのは、Markdownを書く人が増えたことです。現在は開発者だけでなく、ChatGPT、Claude、GeminiなどのAIもMarkdownを生成します。その出力をWebサイト、CMS、社内システム、レポート画面へ移すとき、最終的にHTMLが必要になります。

ユーザーが求めているのは、単に`<h2>`や`<p>`を生成することではありません。本当の目的は、手元にあるMarkdownを、見出し、表、リンク、コードブロックなどの構造を保ったまま、別のシステムへ渡すことです。

### 公開前のプレビューにも変換が必要

変換は、HTMLをコピーするためだけのものではありません。タイトルの階層は正しいか、表は崩れていないか、コードブロックやリンクは期待どおりに表示されるか。公開前のプレビューも、MarkdownをHTMLとして表示して初めて確認できます。

そのため、Markdown to HTMLツールは、変換器であると同時に、Markdownエディタ、プレビュー、HTMLソース生成器として使われることがあります。

## Markdownで書いた内容を公開するまで

筆者も、普段使わない言語でいくつかのプラットフォームへ投稿するときは、まずローカルのMarkdownで構成と内容を整理し、表現を確認しながら投稿用の文章に仕上げています。

その後、本文を投稿先のエディタへ移します。編集画面では`##`やコードフェンスがそのまま見える場合もありますが、プレビューへ切り替えると見出し、リスト、表、コードブロックとして表示されます。この二つの画面を行き来することで、「書きやすさ」と「届けられる状態」の違いを確認できます。

Markdownを直接扱えるプラットフォームもありますが、すべてのCMSや編集画面がMarkdownへ対応しているわけではありません。対応していないシステムへ移す場合は、自分でHTMLへ変換する必要があります。

### 必要な変換ツールを自分で作った

筆者自身もMarkdownとHTMLを変換する必要があり、いくつかのツールを試しました。変換そのものはできても、結果を確認しながら用途に合わせて調整するには、もう少し選択肢がほしいと感じることがありました。

そこで、自分の作業に合わせて使える無料の変換ツールとして[mdhenkan](https://mdhenkan.com/)を作りました。[MarkdownからHTML](https://mdhenkan.com/markdown-to-html)への変換と、[HTMLからMarkdown](https://mdhenkan.com/html-to-markdown)への変換を利用できます。入力内容と変換結果はブラウザ内で処理され、mdhenkanのサーバーには送信・保存されません。

特にこだわったのは、Markdownの原文を変更せずに、変換後のHTMLの表示だけを調整できるようにしたことです。再利用しやすいMarkdownを原稿として残しながら、公開先や用途に合わせてHTMLの見た目を整えられます。

さらに、調整した内容は変更履歴として残るようにしました。どこをどのように変更したのかを後から確認できるため、自分が加えた調整を忘れず、必要なときに同じ考え方を再利用できます。

今後も、Markdownの書きやすさとHTMLの表現力をどちらか一方に寄せるのではなく、それぞれの長所を活かせる機能を増やしていく予定です。

同じように、変換するだけでは少し物足りないと感じている方がいれば、興味のあるときに試してみてください。

## トークンと情報量の違い

同じ可視内容を表すなら、Markdownは一般にHTMLより短くなります。`## 見出し`と書く代わりに、HTMLでは開始タグ、終了タグ、場合によっては`id`や`class`も必要です。文章量が増えるほど、この差はAIへ渡すコンテキストにも積み重なります。

実際のWebページには、本文以外にもナビゲーション、フッター、長いutility class、インラインSVG、スクリプト、非表示要素などが含まれます。AIに本文だけを読ませたいなら、`main`や`article`を抽出し、整理したMarkdownへ変換するほうが効率的です。

一方で、短ければ常に良いわけではありません。ボタンの状態、リンク先、フォームの属性、ARIA、SEOメタデータなどを確認するタスクでは、HTMLの情報を残す必要があります。

また、トークン数は単純な文字数ではなく、モデルのトークナイザーや言語によって変わります。Markdownなら必ず一定割合を削減できる、とは言えません。重要なのは、最短の入力を作ることではなく、タスクに必要な情報だけを渡すことです。

### 変換は常に無損失ではない

MarkdownはHTMLのすべてを表現できません。細かなレイアウト、`data-*`属性、ARIA、インラインスタイルなどは、HTMLからMarkdownへ変換すると失われることがあります。反対に、Markdownから生成されるHTMLも、使用するレンダラーによって見出しIDやコードハイライトの構造が変わります。

したがって、変換の目的を先に決める必要があります。文章をAIへ読ませるなら情報を整理し、Webページを検証するなら必要なHTML構造を残す。同じ変換でも、正解は用途によって変わります。

## まとめ：書く形式と届ける形式を分ける

MarkdownとHTMLがAI時代に再び目立つようになったのは、どちらか一方が万能だからではありません。人間とAIが内容を書く場面ではMarkdownが扱いやすく、ブラウザやCMSが読者へ届ける場面ではHTMLが必要だからです。

`SKILL.md`は、Markdownが文章だけでなく、AIエージェントの手順にも使えることを示しています。HTMLとDOMは、エージェントがWebを理解し、正確に操作するための構造になります。

Markdownは「書く」、HTMLは「届ける」。両者を競わせるのではなく、役割の境界を理解して使い分けることが、AI時代のコンテンツ制作とエージェント活用をシンプルにしてくれます。
