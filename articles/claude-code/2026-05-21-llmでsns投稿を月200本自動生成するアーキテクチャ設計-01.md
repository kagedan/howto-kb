# LLMでSNS投稿を月200本自動生成するアーキテクチャ設計

## はじめに：コンテンツ生成の「量」問題

マーケターやコンテンツ担当者から「週に何本投稿すれば効果が出ますか？」と聞かれたとき、SNSのアルゴリズムが出す答えはだいたい同じです。**「もっとたくさん」**。

X（旧Twitter）なら1日3〜5投稿、LinkedInなら週5本、メルマガは週1本以上——数字だけ見れば現実的でも、それを毎週、毎月、ネタ切れなく続けるのは人間には限界があります。

筆者が関わったあるSaaS企業では、マーケチームが「コンテンツ疲弊」を理由に半年で3人離職しました。原因を掘り下げると、**一人あたり月150〜200本のSNS投稿・メルマガ・ブログを半手動で作り続けていた**ことが判明しました。

この課題を解決するために、Next.js + Claude APIを使ったコンテンツ自動生成SaaS「ContentAI」のアーキテクチャを設計・実装した経験をもとに、本記事では以下を解説します。

* ストリーミングレスポンスによるUX改善
* 再利用可能なプロンプトテンプレート管理
* Stripeと連動した使用量課金
* 月200本規模のコンテンツを安定稼働させるための設計判断

実際に動くコードスニペットを交えながら、実装レベルで解説します。

---

## 全体アーキテクチャの概要

システム全体は以下のレイヤーで構成されます。

```
[Next.js App Router]
    ↓ API Routes (Edge Runtime)
[Claude API / Anthropic SDK]
    ↓ Streaming Response
[PostgreSQL (Supabase)]
    ↓ 生成ログ・テンプレート管理
[Stripe]
    ↓ Webhookで使用量カウント
[Redis (Upstash)]
    ↓ レート制限・キャッシュ
```

フロントエンドはNext.js 14のApp Routerで構築し、APIはEdge Runtimeで動かします。Claude APIのストリーミングをそのままクライアントに流すことでTime-to-First-Tokenを最小化し、体感的な「速さ」を実現しています。

## ストリーミングレスポンスの実装

LLMを使ったサービスで最初に躓くのが\*\*「生成完了まで画面が固まる」問題\*\*です。Claude APIはServer-Sent Eventsベースのストリーミングをサポートしているため、Next.jsのRoute Handlerと組み合わせることで、生成中のテキストをリアルタイムで表示できます。

```
// app/api/generate/route.ts
import Anthropic from "@anthropic-ai/sdk";
import { NextRequest } from "next/server";

export const runtime = "edge";

const client = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY!,
});

export async function POST(req: NextRequest) {
  const { templateId, variables, userId } = await req.json();

  // プロンプトをテンプレートから組み立て（後述）
  const prompt = await buildPrompt(templateId, variables);

  const stream = await client.messages.stream({
    model: "claude-opus-4-5",
    max_tokens: 1024,
    messages: [{ role: "user", content: prompt }],
  });

  // ReadableStreamに変換してレスポンス
  const readableStream = new ReadableStream({
    async start(controller) {
      for await (const chunk of stream) {
        if (
          chunk.type === "content_block_delta" &&
          chunk.delta.type === "text_delta"
        ) {
          controller.enqueue(new TextEncoder().encode(chunk.delta.text));
        }
      }
      // 生成完了後にStripeの使用量を記録
      await recordUsage(userId, stream.finalUsage());
      controller.close();
    },
  });

  return new Response(readableStream, {
    headers: {
      "Content-Type": "text/plain; charset=utf-8",
      "X-Content-Type-Options": "nosniff",
    },
  });
}
```

クライアント側はReactの`useState`とカスタムフックで受け取ります。

```
// hooks/useContentStream.ts
import { useState, useCallback } from "react";

export function useContentStream() {
  const [content, setContent] = useState("");
  const [isStreaming, setIsStreaming] = useState(false);

  const generate = useCallback(
    async (templateId: string, variables: Record<string, string>) => {
      setContent("");
      setIsStreaming(true);

      const res = await fetch("/api/generate", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ templateId, variables }),
      });

      const reader = res.body!.getReader();
      const decoder = new TextDecoder();

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        setContent((prev) => prev + decoder.decode(value));
      }

      setIsStreaming(false);
    },
    []
  );

  return { content, isStreaming, generate };
}
```

このパターンにより、ユーザーはボタンを押した瞬間から文字が流れてくる体験を得られます。

## プロンプトテンプレート管理システム

月200本のコンテンツを品質を保って生成するには、**プロンプトをコードに埋め込まず、データとして管理する**設計が必須です。

テンプレートはPostgreSQLに保存し、変数プレースホルダーを`{{変数名}}`形式で管理します。

```
-- テンプレートテーブル
CREATE TABLE prompt_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  content_type TEXT NOT NULL, -- 'tweet' | 'blog' | 'newsletter'
  system_prompt TEXT NOT NULL,
  user_prompt_template TEXT NOT NULL,
  variables JSONB NOT NULL DEFAULT '[]',
  tone TEXT NOT NULL DEFAULT 'professional',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- サンプルデータ
INSERT INTO prompt_templates (name, content_type, system_prompt, user_prompt_template, variables)
VALUES (
  'X投稿_教育系',
  'tweet',
  'あなたはSNSマーケティングの専門家です。エンゲージメント率を最大化する投稿を作成してください。',
  '以下のトピックについて、{{tone}}なトーンで280文字以内のX投稿を{{count}}本作成してください。
  トピック: {{topic}}
  ターゲット: {{target_audience}}
  含めるキーワード: {{keywords}}
  絵文字の使用: {{use_emoji}}',
  '["tone", "count", "topic", "target_audience", "keywords", "use_emoji"]'
);
```

`buildPrompt`関数はテンプレートを取得して変数を展開します。

```
// lib/prompt-builder.ts
import { createClient } from "@/lib/supabase/server";

export async function buildPrompt(
  templateId: string,
  variables: Record<string, string>
): Promise<{ system: string; user: string }> {
  const supabase = createClient();
  const { data: template } = await supabase
    .from("prompt_templates")
    .select("*")
    .eq("id", templateId)
    .single();

  if (!template) throw new Error("Template not found");

  // 変数の検証
  const requiredVars: string[] = template.variables;
  const missing = requiredVars.filter((v) => !(v in variables));
  if (missing.length > 0) {
    throw new Error(`Missing variables: ${missing.join(", ")}`);
  }

  // プレースホルダーを置換
  let userPrompt = template.user_prompt_template;
  for (const [key, value] of Object.entries(variables)) {
    userPrompt = userPrompt.replaceAll(`{{${key}}}`, value);
  }

  return {
    system: template.system_prompt,
    user: userPrompt,
  };
}
```

このアーキテクチャの利点は、**プロンプトの改善をコードデプロイなしに行える**点です。A/Bテストもテンプレート単位で実施できます。

## Stripe課金連携：使用量ベースの課金設計

コンテンツ生成SaaSでは「月○○本まで」のような**使用量ベース課金**が自然なモデルです。Stripeの`Meter`機能を使って、生成トークン数に応じた課金を実装します。

```
// lib/stripe-usage.ts
import Stripe from "stripe";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);

export async function recordUsage(
  userId: string,
  usage: { input_tokens: number; output_tokens: number }
) {
  // ユーザーのStripe Customer IDをDBから取得
  const customerId = await getStripeCustomerId(userId);

  // 生成トークンをイベントとして記録
  const totalTokens = usage.input_tokens + usage.output_tokens;

  await stripe.billing.meterEvents.create({
    event_name: "content_generation_tokens",
    payload: {
      stripe_customer_id: customerId,
      value: String(totalTokens),
    },
  });

  // 生成ログをDBにも保存（分析用）
  await saveGenerationLog(userId, usage);
}

// Webhook: Stripe → 利用量超過の通知処理
export async function handleStripeWebhook(event: Stripe.Event) {
  switch (event.type) {
    case "billing.meter_error_report_triggered": {
      const report = event.data.object;
      // 利用上限に達したユーザーの生成をブロック
      await disableGenerationForUser(report.reason.error_count);
      break;
    }
    case "customer.subscription.updated": {
      const subscription = event.data.object as Stripe.Subscription;
      // プランアップグレード時に制限を即時解除
      await updateUserPlanLimits(subscription.customer as string, subscription);
      break;
    }
  }
}
```

プラン管理はシンプルなテーブルで管理します。

| プラン | 月間投稿数上限 | メルマガ | ブログ |
| --- | --- | --- | --- |
| Starter | 50本 | × | × |
| Growth | 200本 | ○ | ○ |
| Enterprise | 無制限 | ○ | ○ |

## レート制限とキューイングの設計

月200本を「一気に生成」しようとするユーザーへの対策も必要です。Claude APIには1分あたりのリクエスト上限があるため、**Redis + Upstashでレート制限**をかけます。

```
// middleware/rate-limit.ts
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, "1 m"), // 1分間に10リクエスト
  analytics: true,
});

export async function checkRateLimit(userId: string) {
  const { success, limit, reset, remaining } = await ratelimit.limit(userId);

  if (!success) {
    throw new RateLimitError({
      message: "生成リクエストが多すぎます。しばらく待ってから再試行してください。",
      retryAfter: Math.ceil((reset - Date.now()) / 1000),
      remaining,
      limit,
    });
  }

  return { remaining, limit };
}
```

大量生成が必要な場合は、**バックグラウンドジョブとして非同期処理**に切り替えます。Supabase Edge FunctionsとPg-cronを組み合わせて、夜間バッチ生成を実装することで、日中のAPIレート制限を回避できます。

## 生成品質を担保するバリデーション層

自動生成コンテンツをそのまま公開するリスクへの対策も重要です。ContentAIでは**二段階の品質チェック**を実装しています。

1. **Claude自身によるセルフチェック**：生成後に「この投稿に不適切な表現や誤情報が含まれますか？」と問い合わせる
2. **ルールベースフィルター**：NGワードリスト、文字数チェック、URL含有チェック

```
// lib/content-validator.ts
interface ValidationResult {
  passed: boolean;
  issues: string[];
  score: number; // 0-100
}

export async function validateContent(
  content: string,
  contentType: "tweet" | "blog" | "newsletter"
): Promise<ValidationResult> {
  const issues: string[] = [];

  // ルールベースチェック
  if (contentType === "tweet" && content.length > 280) {
    issues.push(`文字数超過: ${content.length}文字（上限280文字）`);
  }

  const ngWords = await fetchNGWordList();
  const foundNG = ngWords.filter((word) => content.includes(word));
  if (foundNG.length > 0) {
    issues.push(`NGワード検出: ${foundNG.join(", ")}`);
  }

  // LLMによるセマンティックチェック
  const checkResult = await client.messages.create({
    model: "claude-haiku-4-5", // 高速・低コストなモデルで十分
    max_tokens: 256,
    messages: [
      {
        role: "user",
        content: `以下のSNS投稿を評価してください。
        投稿: "${content}"
        
        JSON形式で回答してください:
        {"quality_score": 0-100, "has_issues": true/false, "issue_description": "問題があれば説明"}`,
      },
    ],
  });

  const llmCheck = JSON.parse(checkResult.content[0].text);
  if (llmCheck.has_issues) {
    issues.push(llmCheck.issue_description);
  }

  return {
    passed: issues.length === 0,
    issues,
    score: llmCheck.quality_score,
  };
}
```

## 本番運用で見えてきた改善ポイント

実際に数ヶ月運用して気づいた設計上の知見をまとめます。

**1. プロンプトのバージョン管理**  
テンプレートを更新すると過去の生成結果と品質が変わります。`prompt_templates`テーブルにバージョンカラムを追加し、生成ログには使用したテンプレートのバージョンを記録するのが必須でした。

**2. エラーリトライの設計**  
Claude APIは稀にタイムアウトします。Edge RuntimeではNode.jsの`retry`ライブラリが使えないため、シンプルな指数バックオフを自前実装しています。最大3回リトライ、初回待機500msで運用上の問題はほぼなくなりました。

**3. コスト管理の可視化**  
月200本生成すると思ったよりトークンを消費します。`generation_logs`テーブルにトークン数とコスト（概算）を記録し、Supabaseのダッシュボードで日次コストを監視する仕組みを早期に作ることをお勧めします。Claude Opus使用時と比べ、Haikuに切り替えることでバリデーション処理のコストを**約80%削減**できました。
