日本時間2026年04月17日午前、Anthropic社のYouTubeチャンネルにて **『Mastering Claude Code in 30 minutes(30分でClaude Codeをマスターする)』** というイベントセッション動画が公開されていました。

登壇者はAnthropic社のテクニカルスタッフメンバー、**Boris Cherny**氏。AnthropicでClaude Codeのクリエイター兼責任者を務めるソフトウェアエンジニアです。

MetaでInstagramのPrincipal Software Engineerとして約6年間勤務した後、2024年に入社し、わずか入社1ヶ月でサイドプロジェクトとしてClaude Codeを立ち上げ、現在は同社の主力AIコーディングツールに育て上げました。『Programming TypeScript』の著者で、正式なCS学位を持たない独学エンジニア(self-taught)としても知られています。

そんな、『Claude Codeの生みの親』であるBoris氏のセッションと聞いたら観ないわけにはいかないでしょう！ということで、視聴レポートの形でその内容を紹介します。

## 概要：Claude Codeとは何か

Claude Codeは、これまでの「1行ずつ補完するコーディングアシスタント」とは根本的に異なる、完全エージェント型のAIアシスタントです。ユーザーを特定のワークフローへ誘導することはせず、エンジニアが自分のやり方で自由に使えることを重視しています。

* 機能全体の実装、関数・ファイルの丸ごと生成、バグの完全修正など、大きな単位での作業を担う
* VS Code、Xcode、JetBrains、Vim、EmacsなどあらゆるIDE・ターミナルで動作し、既存ワークフローをそのまま維持できる
* ローカル・リモートSSH・Tmux などあらゆる環境で動作する
* Claude Code自体が持つツールはシンプルな3種類(ファイル編集・bash実行・ファイル検索)だけで、それらを自律的に組み合わせて複雑な作業を完遂

## ステップ1：まずは「コードベースQ&A」から始めよう

Claude Codeを初めて使う人・チームに導入する際に最初にやるべきこととして『コード編集や高度なツール活用より前に、まずコードベースへの質問から始めること』を挙げています。

![](https://static.zenn.studio/user-upload/2422b7825a1b-20260417.png)

* なぜQ&Aから始めるのか
  + プロンプトの感覚が身に付く： Claude Codeに何ができるか、何が一発で解決でき、何は対話が必要かの境界線を体感できる
  + チームへの浸透が早い： Anthropicではこのアプローチによりエンジニアのオンボーディングが 2〜3週間 → 2〜3日 に短縮された
  + コードはローカルから出ない： リモートDBへのアップロードなし・インデックス作成なし・モデル学習への利用なし。ダウンロードしたらすぐ使える

### Q&Aで使える実践的プロンプト例

| ケース | 質問例 | 補足 |
| --- | --- | --- |
| コードの使われ方を調べる | **『このクラスはどのようにインスタンス化されていますか？実際の使用例を教えてください』** | 単なるテキスト検索ではなく、コードベース全体を探索してWikiやドキュメントレベルの詳細な回答を生成 |
| Gitの歴史から「なぜ」を調べる | **『なぜこの関数には15個の引数があるのか、Gitの履歴を調べて教えてください』** | 「誰がいつ追加したか」「どんな状況だったか」「関連するIssueは何か」を自動でまとめてくれる。システムプロンプトで指示しているわけではなく、モデルが自律的にGitを使うと判断 |
| GitHub Issueの文脈を調べる | **『Issue #1234 の背景と現在の状況を調べてください』** | WebFetch機能でIssueを直接取得可能 |
| 週次レポートの自動生成 | **『今週私がリリースしたものをGitログから教えてください』** | 登壇者(Boris)が毎週月曜のスタンドアップで実際に使っている方法。ユーザー名を自動で認識し、出荷物一覧を生成してくれる。そのままドキュメントにコピペするだけで良い |

## ステップ2：コード編集とワークフロー

『コード編集』及び『ワークフロー』については以下ステップに分ける形で進めると良い、と解説。

![](https://static.zenn.studio/user-upload/4839a924d1b7-20260417.png)

### ワークフロー #1. 計画してから書かせる(最重要パターン)

* 大きな実装を一発で頼むと、意図と全然違うものができあがることがある
* まず計画を確認させてから実装に入るのが鉄則

```
コードを書く前に、まずアプローチの計画を立てて私に見せてください。
承認してから実装に進んでください
```

* プランモードなどの特別なツールは不要。自然な言葉でお願いするだけ
* 「3000行の機能実装」のような大きなタスクほど、この手順が重要になる
* ブレインストーミングだけをさせて人間が判断する使い方も有効

### ワークフロー #2. コミット・プッシュ・PRを一発で

この短い指示だけで、以下一連の作業を自動で完遂

* Gitログを調べてプロジェクトのコミットフォーマットを把握
* 適切なコミットメッセージを生成
* 新しいブランチを作成してプッシュ
* GitHubでプルリクエストを作成

### ワークフロー #3. フィードバックを与えて自律反復させる(最強パターン)

```
このモック画像を参照して、WebUIを実装してください。
開発サーバーを起動し、Puppeteerでスクリーンショットを撮りながら
モックに近づくまで反復してください
```

* 1回目でも十分良い出来になるが、2〜3回反復するとほぼ完璧になることが多い

## ステップ3：チームのツールを組み込む

ステップ3に続ける形で、『チームのツールを組み込む』部分についても説明。  
![](https://static.zenn.studio/user-upload/9d73b085bea6-20260417.png)

### bashツールの活用

* チーム内で使っている任意のCLIツールをそのまま使わせる事も可能

```
barley CLIを使ってデータベースのマイグレーションを実行してください。
使い方がわからなければ --help で調べてください
```

* CLIの使い方を事前に教える必要はなく、--help を使って自分で調べさせる
* 良く使うCLIはCLAUDE.mdに記載しておけば、セッションをまたいで記憶される

### MCPツールの活用

* Claude CodeはMCPツールも扱える。既存のチームツールをそのままClaude Codeに渡せる点が大きい。
* mcp.json をリポジトリにチェックインするだけでチーム全員が Claude Code 起動時に自動でMCPを使える状態になる
* Anthropic社内の実例：Puppeteer MCP
  + Appsリポジトリ(Webとアプリのコードがあるリポジトリ)にmcp.jsonをチェックイン済み
  + そのリポジトリで作業するエンジニアは全員、自動でPuppeteerが使える状態になる
  + エンドツーエンドテストの自動操作・スクリーンショット撮影・反復改善が全エンジニアに自動提供される
  + 各自がインストールする必要がない

## ステップ4：CLAUDE.mdでコンテキストを整える

Claude Codeにおいては、コンテキストをうまく与えるほど、Claude Codeの判断精度が上がります。「チームの頭の中にある暗黙知」を文書化してClaude Codeに渡すイメージです。このステップではその辺りのポイントについて解説。

### CLAUDE.md の種類と使い分け

![](https://static.zenn.studio/user-upload/3f9388ac1c15-20260417.png)

| 種類 | 場所 | 読み込みタイミング | 用途 |
| --- | --- | --- | --- |
| エンタープライズ | エンタープライズルート (`/<enterprise_root>/CLAUDE.md`) | 自動(全社員に適用) | 全社員に強制適用するポリシー・ルール |
| ユーザー共通 | ユーザールート (`~/.claude/CLAUDE.md`) | 自動(ユーザー配下の全プロジェクトに適用) | 自分だけの設定 |
| プロジェクト共有 | プロジェクトルート (`project_root/CLAUDE.md`) | セッション開始時に自動 | チーム全体で共有する知識 (ソース管理にチェックイン) |
| 個人設定 | プロジェクト配下 (`project_root/.claude.local.md`) | セッション開始時に自動 | 自分だけの設定(チェックインしない) |
| ディレクトリ別 | 各サブディレクトリ配下 (`subdir/CLAUDE.md`) | そのディレクトリ作業時に自動 | 特定機能・モジュール固有の知識 |

### CLAUDE.md に書くべき内容

Anthropicのコードベースの実例を参考にすると、以下のようなことを書くのが良い、とされている。ポイントは『記述を短く保つこと』。長くなりすぎるとコンテキストを大量消費して逆効果になる。「このコードベースで作業するために最低限知っておくこと」に絞るべき

* よく使うbashコマンド(テスト実行・ビルド・デプロイなど)
* スタイルガイド(コードの書き方のルール)
* 重要なコアファイルの場所と役割
* 設計上の重要な決定事項(なぜこのアーキテクチャになっているか等)
* MCPツールの使い方

### スラッシュコマンドの活用

* `~/.claude/commands/`(ホームディレクトリ) またはプロジェクト内の`.claude/commands/`にMarkdownファイルを置くことで、カスタムスラッシュコマンドを作成できる。
* Anthropic社内での活用例：
  + Claude CodeのリポジトリにGitHub IssueへのラベリングをするスラッシュコマンドをGitHub Actionとして登録
  + Claude Codeが自動でIssueを読んで適切なラベルを付ける
  + 人間がラベリング作業をする必要がなくなった

### /memory コマンドによる管理

現在読み込まれているすべてのメモリファイルを一覧・確認・編集できる。エンタープライズポリシー・ユーザーメモリ・プロジェクトCLAUDE.md・ディレクトリ別CLAUDE.mdがすべて表示される。

### #（ポンド記号）でリアルタイム記憶

```
# このプロジェクトのテストは npm run test:unit で実行する
```

入力すると、その内容をどのメモリファイルに保存するかを選んで書き込める。Claude Codeがツールの使い方を間違えたときなど、その場で修正を記憶させる用途に最適。

## 設定の階層構造：エンタープライズ活用

Claude Codeの設定はすべて階層構造で管理されており、個人・プロジェクト・全社という単位で使い分けられる。

```
エンタープライズポリシー(全社員に強制適用)
  └── グローバル設定(全プロジェクト共通)
        └── プロジェクト設定(リポジトリ固有)
              └── 個人設定(個人の好み)
```

* この階層構造は以下すべてに適用される
  + CLAUDE.md(コンテキスト)
  + スラッシュコマンド
  + パーミッション設定(許可・ブロックリスト)
  + MCPサーバー設定

![](https://static.zenn.studio/user-upload/0c64fba0bb2c-20260417.png)

### 企業での活用例

* コマンドの自動承認：
  + 全社員が使う共通のテストコマンドをエンタープライズポリシーに登録 → 社員が実行するたびに自動承認される
* URLのブロック：
  + フェッチしてはいけないURLをポリシーに追加 → 社員が上書きできない形で永続的にブロックされる
* 導入のファーストステップとして最適なのは「共有プロジェクトコンテキスト」。一人が書けばチーム全員が恩恵を受けるネットワーク効果が生まれる

## キーバインド：知らないと損する操作一覧

ターミナルアプリの性質上、これらのショートカットは発見されにくい。

![](https://static.zenn.studio/user-upload/59d1c065438b-20260417.png)

| キー操作 | 動作 | 使いどころ |
| --- | --- | --- |
| `Shift + Tab` | 編集の自動承認モードへ切り替え | Claude Codeが正しい方向で進んでいるとき。ユニットテスト生成中の反復など |
| `Escape` | 動作を安全に停止 | いつでも・何をしていても安全に止められる。セッションは壊れない |
| `Escape × 2` | 履歴を遡る | 前の状態に戻りたいとき |
| `#` + テキスト | CLAUDE.mdへ記憶 | ツールの使い方を修正させたいとき、設定を永続化したいとき |
| `!` + コマンド | bash直接実行 (コンテキストにも入る) | 確実に実行したいコマンドをClaude Codeにも見せたいとき |
| `@` + ファイル名 | ファイル・フォルダーをコンテキストに追加 | 参照させたいファイルを明示的に指定するとき |
| `Ctrl + R` | コンテキストウィンドウの全出力を表示 | Claude Codeが何を見ているか確認したいとき |
| `claude --resume` | 前回のセッションを再開 | セッション終了後に続きをやりたいとき |

### Escapeの賢い使い方

単純に止めるだけでなく、細かい修正を指示するツールとしても使える。

> 例：20行の編集が提案されたが、19行は完璧で1行だけ変えたい  
> → Escapeで止める  
> → 「この1行だけこう変えて、残りはそのまま」と指示  
> → やり直してもらう

### `!` コマンドの便利な使い方

* ローカルで実行されるが、コマンドと出力がClaude Codeのコンテキストウィンドウに入る
* 次のターンでClaude Codeがそのコマンドと結果を見た状態で作業を継続できる
* 長時間かかるコマンドや、確実に自分でコントロールしたいコマンドに有用

## マルチモーダル活用：画像を渡す

Claude Codeは最初から完全にマルチモーダル対応しているが、ターミナルアプリの性質上、この機能を知らない人が多い。

### 画像の渡し方（3種類）

1. ドラッグ＆ドロップ：ターミナル上に直接ドロップする
2. ファイルパス指定：`@/path/to/mockup.png` のように指定
3. コピー＆ペースト：クリップボードの画像をそのまま貼り付ける

### 実践的な活用例

モックを渡して実装させ、開発サーバーと組み合わせて完全自動で反復改善させるのがポイント。

```
このデザインモック（添付画像）を参照して、Reactコンポーネントとして
WebUIを実装してください。
localhost:3000 を開発サーバーとして起動し、
Puppeteerでスクリーンショットを撮りながらモックに近づくまで反復してください。
```

## Claude Code SDK（`-p` フラグ）：パイプライン活用

`claude -p` は「超高度なインテリジェントなUnixユーティリティ」として機能する。プロンプトを渡せばJSON(またはストリーミングJSON)が返ってくる。

### 基本的な使い方

```
claude -p "このコードのバグを修正して" \
  --allowed-tools "Bash,Edit" \
  --output-format json
```

### パイプとの組み合わせ例

```
# git statusの出力をClaudeに渡して解析
git status | claude -p "この変更内容をまとめて" | jq '.result'

# GCPのログをClaudeに渡して異常を検出
gcloud logging read "..." | claude -p "このログで注目すべき点は何か"

# SentryのCLIから取得したエラーをClaudeに処理させる
sentry-cli events list | claude -p "直近のエラーの傾向を分析して"
```

### Anthropicでの活用領域

* CI/CDパイプライン：コードレビュー・テスト分析の自動化
* インシデント対応：ログ分析・原因特定の自動化
* 各種自動化パイプライン：定期バッチ処理など

この使い方の可能性はまだほとんど開拓されていないと登壇者自身も「この使い方の可能性はまだほとんど開拓されていない」と述べており、現時点では発展途上の領域。

## 上級者向け：並列実行

パワーユーザーが実践している並列実行パターン。

### 方法1：複数ターミナルタブ（最もシンプル）

別々のリポジトリを異なるターミナルタブで開き、それぞれでClaude Codeを動かす。

### 方法2：tmuxを使ったマルチセッション

tmuxでセッションを分割し、複数のClaude Codeを同時に監視・操作。

### 方法3：同一リポジトリの複数チェックアウト

同じリポジトリを複数の場所にcloneし、それぞれで独立したClaude Codeを動かす。ブランチ間で干渉しない。

### 方法4：git worktreeによる分離（推奨）

```
git worktree add ../feature-branch feature/new-ui
git worktree add ../bugfix-branch fix/critical-bug
```

各worktreeで独立したClaude Codeセッションを実行することで、同一リポジトリを複数チェックアウトせずに並列作業が可能。

登壇者曰く『Anthropicはこのあたりの体験をさらに改善すべく開発中』とのこと。  
現時点ではこれらの手動の手法が最善。

## Q&A

セッション最終盤では、イベント参加者との質疑応答時間が設けられていました。

### Q：実装で一番難しかった部分は？

A. bashコマンドの安全管理。bashは本質的に危険で、システム状態を予期せぬ方法で変更できる。しかし「すべてのコマンドを手動承認させる」と生産性が壊滅的に下がる。この相反する課題に対して最終的に以下の解決策を構築した：

1. 読み取り専用コマンドの定義：副作用がないコマンドを自動承認対象に
2. 静的解析：コマンドの組み合わせが安全かどうかを解析
3. 段階的パーミッションシステム：許可リスト・ブロックリストを個人・プロジェクト・エンタープライズの各レベルで管理

Dockerコンテナで開発していないユーザーも多くいるため、環境に依存しない安全性の担保が特に難しかった。

### Q：なぜIDEではなくCLIにしたのか？

A.  
理由1：ターミナルが最大公約数だから。AnthropicではVS Code、Zed、Xcode、Vim、Emacsなど多様なIDEが使われており、全員に対応するものを作るのが困難だった。ターミナルは唯一の共通基盤。

理由2：IDEが不要になる未来への備え。Anthropicでは日々モデルの急速な進化を目の当たりにしており、「年内に多くの人がIDEを使わなくなる可能性がある」と考えている。UIレイヤーへの過剰投資が無駄になるリスクを避けた。

### Q：機械学習・研究用途では使っているか？

A. Anthropicの技術系社員の約80%が毎日Claude Codeを使用しており、エンジニアだけでなく研究者も活用している。研究者はノートブックツールを使って Jupyter Notebookの編集・実行もClaude Code上で行っている。

## まとめ

という訳で、Claude Codeの生みの親、Boris Cherny氏によるセッション『Mastering Claude Code in 30minutes(30分でClaude Codeをマスターする)』の聴講レポートでした。初歩的なものから中級レベル、そしてエンタープライズレベルにまで広がるトピックについて、とても分かりやすい解説内容だったと思います。自分自身も個人利用からチーム規模、そして組織全体に関わるような取り組み(及びそこに関わる設定やルールなど)を眼の前にして『どういう風に設計し、全体に広めていくべきなのだろうか』というところを思案していたところもあったので丁度良い指針、参考となりました。全部が全部、一挙に取り込めるわけではないですが、1つ1つ吟味の上良いものは積極的に取り込んでいこうと思います。
