## はじめに

会社のClaude Enterprise（API経由・社内プロキシ経由）で1日あたり$40前後を消費していた開発環境が、MAX x20への切り替え後は週間使用量が極めて低い水準に収まりました。同じOpus 4.6モデル、同じツールチェーン（rtk、kizami、同一プラグイン群）、さらに同じ社内プロキシ設定（ANTHROPIC\_BASE\_URL）のままです。

この結果から、トークン消費の差は社内プロキシの有無ではなく、EnterpriseとMAXの課金モデルの構造差が主因だとわかりました。本記事ではこの構造差を分析したうえで、プランに関係なく効果を発揮するトークン節約手法を紹介します。

本記事の対象読者は、Claude CodeをOpus 4.6で日常的に使っている開発者です。Enterprise環境でコストに悩んでいる方、MAXプランでも使用量メーターの消費が気になる方の双方に向けて書いています。

## なぜプラン変更だけでコストが劇的に変わったのか

筆者は会社PCで社内プロキシの設定（ANTHROPIC\_BASE\_URL）をそのままにして、EnterpriseからMAX x20に切り替えただけでトークン消費が劇的に改善しました。プロキシが原因ではないことが確認できたため、課金モデルの構造差が主因です。

### 課金モデルの構造差

まず前提として、この従量課金の構造はEnterprise固有の問題ではありません。Claude APIを直接利用する場合（API keyによるConsole経由の利用）も同じく全トークンがAPI標準レートで従量課金されます。Enterpriseプランは、この従量課金モデルの上に、SSO/SCIM、監査ログ、カスタムデータ保持、コンプライアンスAPI、組織単位の利用上限管理、HIPAA対応といった企業のガバナンスに必要な機能を提供するプランです（[Enterpriseプランの概要](https://support.claude.com/en/articles/9797531-what-is-the-enterprise-plan)）。これらの機能はMAXプランでは利用できないため、組織のセキュリティやコンプライアンス要件によってはEnterpriseが唯一の選択肢になります。

そのうえで、課金の仕組みは以下のように異なります。

Enterpriseプラン（およびAPI直接利用）では、シート料金とは別に全トークンがAPI単価で従量課金されます（[Enterprise課金の仕組み](https://support.claude.com/en/articles/11526368-how-am-i-billed-for-my-enterprise-plan)）。割引枠は存在せず、1トークンごとに実費が発生します。

MAX x20（月額$200）は定額制で、5時間のローリングウィンドウと週間キャップの範囲内であれば追加課金はありません（[Max plan](https://claude.com/pricing/max)）。使用量は「token consumption」に基づいて計算されますが、メーター内部でキャッシュ済みトークンがどう扱われるかはAnthropicから公式に開示されていません。

| 項目 | Enterprise / API | MAX x20 |
| --- | --- | --- |
| 基本料金 | シート料金（年額契約）またはなし | 月額$200の定額制です |
| トークン課金 | 全トークンをAPI標準レートで従量課金します | 定額内に含まれます |
| 超過時 | 上限なしで課金が続きます | ウィンドウのリセットを待ちます |
| ガバナンス機能 | SSO、監査ログ、コンプライアンスAPIなどを利用できます | 利用できません |

一言でまとめると、EnterpriseやAPI直接利用では呼び出すたびにコストが積み上がりますが、MAXは月額固定なので5時間/7日のウィンドウ制限を守っている限り追加課金されません。[公式ドキュメント](https://code.claude.com/docs/en/costs)でも「Claude Max and Pro subscribers have usage included in their subscription, so `/cost` data isn't relevant for billing purposes」と明記されています。

[同ドキュメント](https://code.claude.com/docs/en/costs)によると、Enterprise環境でのClaude Code利用コストは「平均$13/開発者/稼働日、90%のユーザーが$30/稼働日以下」とされており、筆者の$40/日は上位10%に入る高い消費量でした。このような高消費パターンのユーザーほど、定額制への移行で恩恵が大きくなります。ただし、組織のガバナンス要件でEnterpriseが必要な場合は、後述するトークン節約手法でAPI従量課金自体を削減するアプローチが有効です。

### Prompt Cachingの役割

Claude Codeは内部的にPrompt Cachingを活用しており、システムプロンプトやCLAUDE.mdなどの繰り返しコンテンツのコストを削減しています（[公式ドキュメント](https://code.claude.com/docs/en/costs)）。API経由のEnterprise環境では、cache readトークンは通常の10%で課金されるため、キャッシュが効いているほど従量課金の総額が下がります（[Prompt caching - Claude API Docs](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)）。

ただし、MAX x20の使用量メーターでPrompt Cachingがどのように考慮されるかはAnthropicから公式に開示されていません。筆者の環境では、社内プロキシの設定を変えずにプランだけをMAX x20に切り替えた結果、Token消費が劇的に改善しました。この改善が「定額制の枠内に収まった」ことによるものなのか、「メーター内部でキャッシュが割引計算されている」ことによるものなのかは、現時点では切り分けられていません。

なお、プロキシ経由でPrompt Cachingが無効化される問題（OpenClaw [issue #37325](https://github.com/openclaw/openclaw/issues/37325)）やClaude Codeのキャッシュ関連バグ（[#41930](https://github.com/anthropics/claude-code/issues/41930)）も報告されています。Enterprise環境でAPIコストが高い場合は、`/cost`コマンドでcache hit率を確認してください。

## プランに関係なく効く3つのトークン節約手法

ここからはEnterpriseでもMAXでも導入できる手法を紹介します。3つの手法はそれぞれ異なる層に作用するため、効果が重複せず組み合わせで相乗効果を生みます。

| 手法 | 作用する層 | 効果の仕組み |
| --- | --- | --- |
| RTK | セッション内のI/O | Bashコマンドの出力トークンをリアルタイムに圧縮します |
| メモリツール | セッション間の知識 | 前回の文脈を次のセッションに引き継ぎ、毎回の再構築を防ぎます |
| プラグイン/SKILL | ワークフロー全体 | 非効率な作業手順を排除し、手戻りによるトークン浪費を減らします |

RTKは「1回のセッション内で発生するI/Oトークンを減らす」手法です。メモリツールは「セッションをまたいで蓄積した知識を再利用し、毎回のコンテキスト構築コストをなくす」手法です。この違いを意識すると、それぞれの役割が明確になります。

### 手法1 — RTKによるCLI出力の92%圧縮

[RTK（Rust Token Killer）](https://github.com/rtk-ai/rtk)は、CLIコマンドの出力をLLMが理解しやすい形に圧縮するプロキシです。

Claude Codeの`PreToolUse`フックがBashコマンドを透過的にRTK経由に書き換えます。開発者もClaude Codeも、通常どおりコマンドを発行するだけです。

```
git status → rtk git status
cat file.rb → rtk read file.rb  
grep -n pattern → rtk grep pattern
```

筆者の環境での直近30日間の実測結果です。

```
Total commands:    998
Tokens saved:      2.2M (92.3%)
Total exec time:   23m22s (avg 1.4s)
```

コマンド別に見ると、特に構造化データの圧縮効果が顕著です。

| コマンド | 実行回数 | 節約トークン | 削減率 |
| --- | --- | --- | --- |
| `rtk read` | 135 | 907.6K | 16.7% |
| `rtk find` | 29 | 790.4K | 53.9% |
| `rtk grep` | 66 | 117.6K | 15.8% |
| `rtk curl` | 5 | 140.6K | 98.1% |
| `rtk ps aux` | 1 | 48.9K | 98.3% |

`curl`や`ps aux`が98%超の削減率を示しているのは、APIレスポンスやプロセスリストはLLMが全行を読む必要がないためです。RTKはこうした構造化データを要約して返します。

導入は`cargo install rtk && rtk init -g`の2コマンドで完了します。`rtk discover`を実行すると、まだ最適化できていないコマンドも確認できます。

筆者の環境では追加241Kトークンの節約余地がまだ残っており、RTKは継続的に改善が進んでいます。

### 手法2 — セッション間メモリで再構築コストをゼロに

セッション間の記憶を永続化するツールとして、代表的なものに[claude-mem](https://github.com/thedotmack/claude-mem)があります。セッション中の作業をAIで要約・圧縮して保存し、次のセッションで関連コンテキストを注入する仕組みです。

ただし、claude-memはMCPサーバーとして動作するため、ツール定義のトークンオーバーヘッドが発生します。MCPサーバーは接続しているだけで毎メッセージにツールスキーマが注入され、5サーバー接続時に約55Kトークンが会話開始前に消費されるケースが報告されています（[Claude Code MCP Servers and Token Overhead](https://www.mindstudio.ai/blog/claude-code-mcp-server-token-overhead)、[MCP Server Token Costs](https://www.jdhodges.com/blog/claude-code-mcp-server-token-costs/)）。

どちらのツールを選んでも、セッション間メモリがもたらす効果は共通しています。

メモリがない場合、毎セッション冒頭で「このプロジェクトの構造を教えて」「前回どこまで進んだ？」といったやりとりが発生し、ファイル読み込みやコマンド実行で推定5,000から20,000トークンが消費されます。メモリがあれば、この再構築コストがほぼゼロになります。

kizamiの場合、`kizami setup`を実行するだけでSessionEndとUserPromptSubmitのフックが自動登録されます。Claude Codeのauto memory（`~/.claude/projects/*/memory/`）と組み合わせることで、プロジェクト固有の設計判断やフィードバックも永続化されます。

ツール選択の判断基準は以下のとおりです。

* MCPサーバーの数が少なく、オーバーヘッドが許容範囲であればclaude-memが手軽です
* MCPサーバーを多数接続している環境や、トークン消費を極限まで抑えたい場合はhookベースのkizamiが適しています

### 手法3 — プラグインとHooksによるワークフロー最適化

筆者の環境では20個のプラグインと39個のフックが稼働しています。この構成に至った背景には「Claude Codeは放っておくと非効率な探索パターンに陥りやすい」という経験則があります。

たとえばfeature-dev SKILLがない状態で「この機能を実装して」と指示すると、Claude Codeはリポジトリ内のファイルを手当たり次第に読み始めることがあります。SKILLがあれば、コードベースの探索からアーキテクチャ設計、実装へという手順が強制され、不要なファイル読み込みが排除されます。

#### 主要プラグイン

| プラグイン | 提供元 | インストール |
| --- | --- | --- |
| superpowers | Anthropic公式マーケットプレイス | `claude plugin install superpowers` |
| feature-dev | Anthropic公式マーケットプレイス | `claude plugin install feature-dev` |
| code-review | Anthropic公式マーケットプレイス | `claude plugin install code-review` |
| security-guidance | Anthropic公式マーケットプレイス | `claude plugin install security-guidance` |
| figma | Anthropic公式マーケットプレイス | `claude plugin install figma` |
| playwright | Anthropic公式マーケットプレイス | `claude plugin install playwright` |
| 各種LSP | Anthropic公式マーケットプレイス | `claude plugin install <name>-lsp` |
| [ecc](https://github.com/affaan-m/everything-claude-code) | サードパーティ | `claude plugin add-registry ecc https://github.com/affaan-m/everything-claude-code.git` の後に `claude plugin install ecc` |
| [claude-mem](https://github.com/thedotmack/claude-mem) | サードパーティ | `claude plugin add-registry thedotmack https://github.com/thedotmack/claude-mem` の後に `claude plugin install claude-mem` |
| [coderabbit](https://github.com/coderabbitai/coderabbit) | Anthropic公式マーケットプレイス | `claude plugin install coderabbit` |
| [claude-hud](https://github.com/jarrodwatts/claude-hud) | サードパーティ | `claude plugin add-registry claude-hud https://github.com/jarrodwatts/claude-hud` の後に `claude plugin install claude-hud` |

サードパーティのプラグインは、レジストリの事前登録（`add-registry`）が必要です。

#### Hooks構成（合計39フック）

| フェーズ | フック数 | 役割 |
| --- | --- | --- |
| PreToolUse | 12 | RTK書き換え、no-verify防止、commit品質チェック、コンパクト提案を行います |
| PostToolUse | 11 | フォーマット、型チェック、lintを実行します |
| UserPromptSubmit | 2 | kizami recallとナレッジ収穫を実行します |
| SessionEnd | 2 | kizami saveとセッション記録を行います |
| Stop | 6 | 型チェック、console.log検出、コスト追跡、デスクトップ通知を実行します |
| その他 | 4 | SessionStart初期化、Compact前後処理、エラー処理を行います |

全部を一度に導入する必要はありません。最初はRTKとメモリツールだけ導入し、効果を確認してからプラグインとフックを段階的に追加するのが現実的です。

## どこから始めるか

導入の優先順位は、環境に応じて以下のように判断できます。

Enterprise環境で$40/日前後を消費している場合は、課金モデル自体が従量制であることが最大のコスト要因です。MAXプランへの切り替えが可能であれば、それだけで劇的に改善する可能性があります。筆者の場合、社内プロキシの設定を変えずにMAX x20に切り替えただけで解決しました。

MAXプランで使用量メーターの消費が気になる場合は、RTK導入が最も即効性があります。`cargo install rtk && rtk init -g`の2コマンドで導入でき、92%の出力圧縮効果がすぐに確認できます。

いずれの環境でも、最終的にはRTK（I/O層）、メモリツール（記憶層）、SKILL（ワークフロー層）の3層で最適化するのが効果的です。それぞれ独立して導入可能ですが、3つが揃うと相乗効果で「同じ作業がより少ないトークンで、より高い品質で完了する」状態になります。

```
# 導入の全手順
cargo install rtk && rtk init -g                    # RTK
git clone https://github.com/okamyuji/kizami.git    # kizami
cd kizami && pnpm install && pnpm build && npm link
kizami setup
claude plugin install superpowers                    # プラグイン
claude plugin install feature-dev
claude plugin install code-review
```

効果の確認には以下のコマンドを使います。

```
rtk gain              # RTKのトークン節約量を確認します
rtk discover          # まだ最適化できるコマンドを発見します
kizami stats          # メモリの統計情報を表示します
```
