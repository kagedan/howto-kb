## はじめに

Claude Code を本格的に使い込んでいくと、ある時点で必ず「コンテキストが重い」問題にぶつかります。プロジェクトが大きくなるほど `CLAUDE.md` も肥大化し、セッション開始時に大量のトークンを食い、しかも指示の遵守率まで下がっていく――この悪循環です。

公式ドキュメント（[How Claude remembers your project](https://code.claude.com/docs/en/memory)）を読み込むと、実はこの問題を解決するための仕組みがいくつも用意されています。この記事では、それらを「**トークン節約**」という一本の軸で整理し、実際のプロジェクト構成に落とし込む方法をまとめます。

対象読者は、`CLAUDE.md` を1ファイルで運用していて、そろそろ限界を感じ始めた人です。

## 大前提：CLAUDE.md は「全量ロード」される

最初に一番重要な事実を押さえます。

> ディレクトリ階層の「上方」にある `CLAUDE.md` / `CLAUDE.local.md` は、セッション開始時に**全文がコンテキストに読み込まれる**。

ここでよく誤解されるのが「200行の上限」です。正確には次のとおりです。

* **CLAUDE.md の 200 行は「目標値（target）」** ― 長さに関係なく全量ロードされる。長いほどトークンを食い、指示の遵守率が下がるので「200行以内を目指せ」という推奨にすぎない。
* **ハードな 200 行 / 25KB の上限は auto memory の `MEMORY.md` 側** ― こちらは先頭 200 行または 25KB を超えた分が起動時にロードされない。

つまり「CLAUDE.md は200行で切られる」というのは誤りで、**切られないからこそ自分で短く保つ必要がある**、というのが正しい理解です。ここを混同したまま記事を書くと突っ込まれるので注意。

## ロードの方向性を理解する（節約の核心）

トークン節約のカギは、公式の次の挙動にあります。

> 作業ディレクトリより**上の階層**の CLAUDE.md は起動時に全量ロードされるが、**サブディレクトリ**の CLAUDE.md は、Claude がそのディレクトリ内のファイルを読んだときに**オンデマンドでロード**される。

これを図にするとこうです。

```
your-project/          ← ここで `claude` を起動
├── CLAUDE.md          ← 起動時に全量ロード（プロジェクト共通ルールだけ）
├── frontend/
│   └── CLAUDE.md      ← frontend のファイルに触れたときだけロード
└── backend/
    └── CLAUDE.md      ← backend のファイルに触れたときだけロード
```

フロントエンドの作業をしているとき、`backend/CLAUDE.md` はコンテキストに入りません。逆もしかり。**作業中の領域に関係するルールだけがロードされる**ので、1ファイルに全部詰め込む場合と比べて起動時のトークンを大幅に削れます。

### 落とし穴：「上か下か」は作業ディレクトリ基準

ここが地味にハマるポイントです。「上方／下方」はリポジトリのルートではなく、**`claude` を起動したディレクトリ**を基準に決まります。

* **リポジトリルートで `claude` を起動** → `frontend/` も `backend/` も「下方サブディレクトリ」扱い → 両方オンデマンド ✅（理想）
* **`cd frontend && claude`** → `frontend/CLAUDE.md` は「作業ディレクトリそのもの」なので全量ロードされ、ルートの CLAUDE.md も「上方」として全量ロードされる。`backend/` は探索パスに入らない。

節約効果を最大化したいなら、**リポジトリルートから起動する**のが一番素直です。

## さらに上：`.claude/rules/` でパススコープ化

サブディレクトリに `CLAUDE.md` を散らす方式も有効ですが、公式がより推奨しているのは `.claude/rules/` を使った**パススコープ・ルール**です。効果は似ていますが、ルールが一箇所に集まるぶん保守性が高い。

```
your-project/
├── .claude/
│   ├── CLAUDE.md           # プロジェクト共通の指示
│   └── rules/
│       ├── code-style.md   # paths なし → 常時ロード
│       ├── frontend.md     # paths 指定 → 該当ファイル時のみ
│       └── go-backend.md   # paths 指定 → 該当ファイル時のみ
```

YAML フロントマターの `paths` に glob を書くと、そのパターンに一致するファイルを Claude が読んだときだけルールが効きます。

```
---
paths:
  - "backend/**/*.go"
---

# Go バックエンドのルール

- API ハンドラは `internal/handler/` に置く
- エラーは標準のラッパー経由で返す
```

`paths` を書かないルールは無条件ロードで、優先度は `.claude/CLAUDE.md` と同じ。複数パターンやブレース展開も使えます。

```
---
paths:
  - "src/**/*.{ts,tsx}"
  - "tests/**/*.test.ts"
---
```

**使い分けの目安**：ルールが少ないうちはサブディレクトリ CLAUDE.md が直感的。種類が増えてきたら `.claude/rules/` に集約したほうが管理が楽になります。

## `@import` はトークンを節約「しない」

ありがちな誤解その2。`@path/to/file` の import 構文は、ファイルを整理するのには便利ですが、**トークン節約にはならない**点に注意。

> import されたファイルは起動時に展開され、参照元の CLAUDE.md と一緒にコンテキストへロードされる。

つまり import は「見た目の整理」であって「遅延ロード」ではありません。トークンを減らしたいなら import ではなく、前述の**パススコープ・ルール**か**サブディレクトリ CLAUDE.md**を使うのが正解です。

なお相対パスは「import を書いたファイル基準」で解決され、再帰 import は最大4ホップまで。

## 「いつもロードしなくていいもの」は Skill へ

CLAUDE.md と rules は「毎セッション（または該当ファイル時）に必ず入る」ものです。一方で、**特定のタスクのときだけ必要な手順**は Skill にするのが筋です。

Skill は `.claude/skills/<skill-name>/SKILL.md` に置きます（必須ファイルは `SKILL.md` のみ）。Skill の本体は、Claude がそれを選択したときに初めてロードされる――いわゆる progressive disclosure です。

* フロントマターの `description` だけは起動時に全 Skill 分が読まれる（だから description は超重要）
* 本体の手順・例・参照ドキュメントは、選択されるまでロードされない

「マルチステップの手順」や「コードベースの一部でしか使わない知識」を CLAUDE.md から Skill へ追い出すと、起動時コンテキストがぐっと軽くなります。

## `/compact` 後の挙動も知っておく

長いセッションで `/compact` を打ったあとの挙動：

* **ルートの CLAUDE.md は生き残る** ― 圧縮後にディスクから再読込・再注入される。
* **サブディレクトリのネスト CLAUDE.md は自動では戻らない** ― 次にそのディレクトリのファイルを読んだとき再ロードされる。

「圧縮したらフロントのルールが消えた気がする」ときは、たいてい再ロード待ちなだけ。frontend のファイルを1つ開けば戻ってきます。

## まとめ：節約のための設計フロー

最後に、プロジェクトが育ってきたときの判断フローとして整理します。

1. **ルート CLAUDE.md** … 全領域に効く最小限の共通ルールだけ。200行以内を目標に。
2. **領域別ルール** … `frontend/` `backend/` などはサブディレクトリ CLAUDE.md、または `.claude/rules/` のパススコープ・ルールへ。該当ファイル時のみロードされる。
3. **タスク手順** … 毎回は要らない多段手順は `.claude/skills/` の Skill へ。
4. **import は整理用** … トークン削減目的では使わない。
5. **起動はリポジトリルートから** … 遅延ロードを最大限効かせる。
6. **auto memory** … 自分で書かなくても Claude が学習を蓄積（`MEMORY.md` は先頭200行/25KBが上限）。CLAUDE.md とは別物として併用する。

「1つの巨大な CLAUDE.md」から「必要なものが必要なときだけ入る階層構成」へ移すのが、トークン節約の本質です。プロジェクトが大きくなるほど効いてきます。
