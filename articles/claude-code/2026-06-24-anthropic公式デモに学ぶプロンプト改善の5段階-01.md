## 10か月前の動画が、いまになって見直されている

きっかけは、X(旧 Twitter)で流れてきた一本の投稿でした。「知らなかった Claude Code の機能を 24 分で全部暴露」「大多数の人がスルーする内容」。知らない機能があるなら見ておきたい、という気持ちで再生ボタンを押した人は多かったはずです。

実際に開いてみると、それは Claude Code の新機能紹介ではありませんでした。Anthropic 社(対話 AI「Claude」の開発元)の応用 AI チームに所属する Hannah Moran さんと Christian Ryan さんが、2025 年 5 月のイベント「Code w/ Claude」で行った、プロンプトの書き方を題材にしたライブデモです。公開は 2025 年 7 月 31 日。今からおよそ 10 か月前の動画が、いま再び注目を集めました。

<https://youtu.be/ysPbXH0LpIE>

タイトルの付け方が巧みだった、という見方はそのとおりだと思います。「知らなかった機能」で好奇心を、「大多数が見逃す」で希少性を刺激する。Xで伸びやすい型にきれいに乗っています。

ただ、タイトルだけでこれだけ再生されたわけではありません。中身が、流行り廃りの影響を受けにくい「基礎」だったからです。Claude Code のような新しいツールを使う人ほど、その下にあるプロンプトの組み立て方が効いてきます。土台の知識なので、10 か月たっても需要が消えません。

この記事では、その 24 分のデモの中身を追いかけます。とくに、ひとつのプロンプトを v1 から v5 まで実際に改善していく過程を、何を足すと出力がどう変わるのかという対応関係で整理します。Claude Code を使い始めた人にも、すでに AI コーディングツールを日常的に使っている人にも、明日から効く形で持ち帰ってもらうのが狙いです。

## 題材：スウェーデンの事故報告書を Claude に読ませる

デモで使われるのは、実在の顧客から着想を得たという具体的なタスクです。あなたはスウェーデンの保険会社で働いていて、毎日、自動車保険の請求を処理している。手元には 2 つの情報があります。

1. 自動車事故報告書のフォーム(17 個のチェックボックスと、車両 A・車両 B の 2 列で構成される)
2. 事故がどう起きたかを当事者が手で描いたスケッチ

この 2 枚を Claude に渡し、「何が起きたのか」「誰に過失があるのか」を判断させたい。最終的には、その判定をシステムに保存して、人間の査定担当者を支援する形につなげたい。これがゴールです。

抽象論ではなく、こういう「読みづらい手書きの帳票を読み取らせる」という現場で起きがちなタスクを題材にしているところが、この動画が実用的だと言われる理由のひとつです。では、ここからプロンプトを育てていきます。

## v1：いきなり投げると「スキー事故」になる

最初のプロンプトは、極端にシンプルです。舞台設定は何もしません。

prompt-v1.txt

```
あなたは事故報告書を精査し、最終的に事故で何が起きたのか、
誰に過失があるのかを判断します。

[フォームの画像]
[スケッチの画像]
```

温度(temperature)は 0 に設定されています。温度は出力のランダムさを決めるパラメータで、0 にすると最も確定的な(ぶれない)応答になります。保険査定のように「同じ入力なら同じ判断であってほしい」タスクでは、ここを 0 にするのが理にかなっています。

このプロンプトを実行すると、Claude はこう答えます。これは「チャパンガン通りで起きたスキー事故」に関する報告書です、と。自動車事故のはずが、スキー事故になってしまいました。

笑ってしまう誤りですが、登壇者はこれを「ある程度は理解できる無邪気な間違い」と表現します。プロンプト側が、ここで何が起きているのかという舞台設定を一切渡していないからです。モデルには手書きの帳票と落書きのような図しか見えておらず、文脈がない。誤読の責任は、半分はこちら側にあります。

## v2：役割とトーンを与える

v1 の問題は、Claude が「自分が何者で、何をする場面なのか」を知らないことでした。そこで最初に足すのが、プロンプト構造の先頭 2 要素である **タスクコンテキスト** と **トーンコンテキスト** です。

* タスクコンテキスト: Claude の役割と、どんなシナリオで働いているか
* トーンコンテキスト: どう振る舞ってほしいか(ここでは「事実に基づき、確信があるときだけ判断する」)

prompt-v2.txt

```
あなたは、スウェーデン語で書かれた自動車事故報告書を審査する、
人間の保険査定担当者を支援する AI アシスタントです。
渡されるのは、当事者が手で描いた事故のスケッチです。

完全に確信が持てない場合は、評価を下さないでください。
事実に基づいて述べ、自信があるときだけ判断してください。

[フォームの画像]
[スケッチの画像]
```

v2 では、入力データも図示して「いま注目しているのはここ」を明確にします。実行すると、出力が変わります。今度はちゃんと自動車事故だと認識し、車両 A はチェックボックス 1、車両 B は 12 にチェックがある、と読み取ります。

そして注目したいのが次です。下にスクロールすると、Claude は「誰に過失があるかを完全な確信を持って判断するには、情報が足りない」と答えます。

これは失敗ではありません。むしろ成功です。「自信があるときだけ判断する」というトーンコンテキストが効いて、わからないことをわからないと正直に申告した。あいまいなまま断定して人を惑わせるより、ずっと安全な振る舞いです。

## v3：変わらない知識はシステムプロンプトへ

v2 で足りなかったのは、フォームそのものについての知識でした。この帳票は毎回まったく同じ書式です。列の数も、行の意味も、クエリごとに変わりません。変わるのは「記入された内容」だけです。

毎回変わらない情報は、システムプロンプト(モデルの役割や前提知識を置く領域)に移すのが定石です。v3 では、フォームの構造を背景情報としてシステムプロンプト側に書き、範囲を XML タグ(`<tag>...</tag>` で囲んで意味づけする記法)で区切ります。Claude は構造化された入力を好むため、XML タグや Markdown は理解の助けになります。

system-prompt-v3.xml

```
<form_structure>
これはスウェーデンの自動車事故報告書です。

- スウェーデン語で記入されます
- 2 列構成で、各列が別々の車両(車両 A / 車両 B)を表します
- 全 17 行あり、各行の意味は次のとおりです: ...(1行ずつ説明)
- 人間が手で記入するため、完璧ではありません。
  丸で囲んだり、走り書きしたり、×印を付けないこともあります
</form_structure>
```

これを渡すと、出力に 2 つの変化が出ます。

ひとつは速度です。Claude はもうフォームの書式を推測する必要がないので、毎回ゼロから「これは何の欄だろう」と考える時間が減り、処理が速くなります。

もうひとつは自信です。今度は Claude が、図とスケッチに基づいて「このケースでは車両 B に過失がある」と踏み込んで判断します。図とリストを見れば、おそらく多くの人が同じ結論に同意するであろう内容です。

## v4：読む順番を指定する

ここまでで Claude はかなり正確になりました。v4 では、分析の質をもう一段上げるために **読む順番** を指定します。デモの中で「顧客事例に取り組む中で見つけた、非常に重要な点」として紹介される部分です。

なぜ順番が重要なのか。登壇者は人間の読み方にたとえます。あなたがいきなり手描きのスケッチだけを渡されても、箱と線の集まりにしか見えず、意味がわかりません。でも先にフォームを読んで「これは自動車事故で、どの車両がいつ何をしたかのチェックがある」と理解できていれば、同じスケッチがぐっと読めるようになります。

そこで、Claude にも同じ順番で読ませます。

task-steps-v4.txt

```
次の順序で分析してください。

1. まずフォームを非常に注意深く見て、どのボックスにチェックが
   入っているかを確認する
2. 見落としが無いか確かめ、フォームに書かれている内容を
   すべてリストアップする
3. それからスケッチに進む
4. フォームから自信を持って得た事実を踏まえ、
   そのスケッチから何が読み取れるかを考える
5. フォームの理解とスケッチを照らし合わせ、最終的な評価に到達する
```

実行すると、Claude は作業過程を段階的に示すようになります。各チェックボックスについて「これはチェックされているか、いないか」を一つずつ確認していく。複雑な図やあいまいなフォームでも、この段階的な思考が正しい評価にたどり着く力になります。

## v5：出力の形を固定する

最後の v5 で足すのは、**ガイドライン(リマインダー)** と **出力フォーマット** です。プロンプトの末尾で、守ってほしいことを改めて念押しし、出力の形を確定させます。

末尾での念押しには、幻覚(プロンプトに無い情報をモデルが捏造すること)を防ぐ狙いがあります。「どのボックスにチェックがあるか判別できないなら、勝手に推測してチェック済みと決めつけないでほしい」「スケッチが理解不能なら、人間にも読めないほど下手な絵だと正直に言ってほしい」。こうした注意を最後にもう一度置きます。

出力フォーマットも重要です。このデータを扱うエンジニアからすると、凝った前置きはありがたくても、最終的に欲しいのは保存できる判定結果です。SQL データベースなどに入れたいのは、判定そのものであって、そこに至る思考の全文ではありません。そこで、最終判定を専用の XML タグで囲わせます。

prompt-v5-tail.txt

```
<guidelines>
要約は明確・簡潔・正確にしてください。
判断の妨げになる前置きは書かないでください。
車両 B が右折したと述べるなら、
「2 番目のチェックボックスが明らかにチェックされている」
という事実を根拠として示してください。
</guidelines>

最終判定は、必ず次のタグで囲んでください。
<final_verdict>
...
</final_verdict>
```

「事実を根拠として示せ」という指示が、ここでも効きます。主張のたびにフォーム内の根拠を参照させることで、幻覚をさらに抑えられます。

実行すると、出力は以前より簡潔になり、最後は `<final_verdict>` タグできれいに囲まれます。タグの中だけをアプリケーション側で取り出せば、そのまま保存・後続処理に回せます。

ここまでの 5 段階を振り返ると、出力は次のように変化しました。

| Ver | 足した要素 | 出力の変化 |
| --- | --- | --- |
| v1 | (素のプロンプト) | スキー事故と誤読 |
| v2 | 役割 + トーン | 自動車事故と認識。情報不足を正直に申告 |
| v3 | 背景知識をシステムプロンプトへ | 高速化し、車両 B の過失を判断 |
| v4 | 分析の順序 | 段階的に思考し、複雑なケースにも対応 |
| v5 | ガイドライン + 出力フォーマット | 簡潔で、そのまま保存できる確定出力 |

スキー事故という珍回答から、やや自信なげな回答を経て、厳密にフォーマットされた自信ある判定へ。同じタスクでも、プロンプトの組み立て方だけでここまで変わります。

## プロンプトを構成する10の要素

デモでは、この改善の裏側にある「良いプロンプトの構造」が 10 個の要素として整理されています。v1 から v5 で足してきたものは、この一覧の一部です。順番もこのとおりが推奨です。

| # | 要素 | 役割 |
| --- | --- | --- |
| 1 | タスクコンテキスト | Claude の役割と、働くシナリオ |
| 2 | トーンコンテキスト | 事実主義など、振る舞いの指定 |
| 3 | 背景データ・文書・画像 | 毎回変わらない知識。システムプロンプトへ |
| 4 | 詳細なルール | 守るべき制約 |
| 5 | 例(few-shot) | 難しい・グレーゾーンの事例を見せる |
| 6 | 会話履歴 | ユーザー向けアプリならコンテキストとして注入 |
| 7 | 直近タスクの再確認 | いま何をしてほしいか |
| 8 | 段階的な思考・分析順序 | 人間の認知順を模倣させる |
| 9 | 出力フォーマット | XML タグなどで確定出力を囲う |
| 10 | 事前入力(prefill) | 応答の冒頭を指定して出力を誘導する |

全部を毎回使う必要はありません。タスクに応じて、効きそうな要素から積み上げます。デモのタスクでは 6 番(会話履歴)は使われませんでしたが、ユーザーと長くやり取りするアプリなら、ここがコンテキストの宝庫になります。

## デモには出てこなかった、効きの大きい手札

時間の都合で実演されなかったものの、登壇者が「強調しておきたい」として挙げた手法がいくつかあります。実務で効きやすいので触れておきます。

### 例(few-shot)を渡す

例を見せることは、Claude を制御する強力な手段だと繰り返し強調されます。Claude が苦手とするグレーゾーンの事例を、人間が正しくラベル付けした形で渡しておくと、次に似たケースが来たときの判断の拠りどころになります。テキストだけでなく、画像を Base64 エンコード(画像を文字列に変換する方式)して視覚的な例として渡し、その読み解き方を添えることもできます。

保険会社向けに本番運用するなら、難しい事例が数十、数百と出てきます。それらを例としてシステムプロンプトに蓄積し、フィードバックループを回して育てていく。これもまた経験科学です。

### 事前入力(prefill)で出力を誘導する

Claude の応答の冒頭を、こちらであらかじめ書いておく手法です。たとえば応答を `{` で始めさせれば JSON を、`<final_verdict>` で始めさせれば XML を、それぞれ出力フォーマットとして強制できます。後続の処理で構造化データが必要なときに便利です。

prefill の使いどころ

出力を解析しやすい形に固定したいとき、prefill は確実です。XML タグの解析でも十分ですが、JSON としてそのまま次の API 呼び出しに渡したい場合などは、応答の頭を `{` で始めさせると安定します。前置きが不要なら前置きを省く誘導にも、望んだ手順で推論させる誘導にも使えます。

### 拡張思考(extended thinking)を観察に使う

Claude 3.7 と、とくに Claude 4 は、拡張思考を使えるハイブリッド推論モデルです。これを有効にすると、思考タグやスクラッチパッド(モデルが考えを書き出す作業領域)が応答に付きます。

登壇者が面白い使い方として挙げるのが、この思考の記録を「観察対象」にすることです。私たちの直感を持たないモデルが、渡したデータをどう処理しているかが見えるので、その記録を読めば「システムプロンプトのどこを直せば良いか」のヒントが得られます。プロンプト改善の道具として拡張思考を使う、という発想です。

## なぜこの「基礎」は古びないのか

24 分のデモを通して語られたのは、特定のモデルや特定のツールのテクニックではありません。役割を与える、変わらない知識をシステムプロンプトに固定する、読む順番を指示する、出力の形を決める、根拠を述べさせる。どれもモデルが世代交代しても効き続ける原則です。

冒頭の話に戻ります。この動画が「知らなかった Claude Code の機能」というタイトルで再注目されたのは、framing の巧みさもありますが、本質は中身が土台の知識だったからです。Claude Code のようなツールは、結局こうしたプロンプトの組み立て方の上で動きます。エージェントに長い指示を渡すときも、役割・背景・順序・出力形式を整える発想はそのまま使えます。

新しい機能を追いかける前に、まずプロンプトの v1 を v2 に直せるか。スキー事故をスキー事故のまま放置しないか。そこを押さえておくと、どんな新ツールが来ても土台が効きます。
