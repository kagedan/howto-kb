Claude CodeやCodexを使っていると、ある日突然「Skill」という言葉が出てきます。

説明を読んでも、こんな疑問が残りませんか。

先に結論を言います。

> Skillは、必要な場面でAIが読み込んで使う「再利用できる仕事の手順書＋道具箱」です。

AIモデルを再学習させる仕組みではありません。毎回チャットで説明していた手順、判断基準、参考資料などを、次回も使える形でまとめておく仕組みです。

この記事では、高度なSkillの作り方までは扱いません。まずはClaude CodeとCodexに共通する考え方をつかみ、自分の作業がSkill向きか判断できるところを目指します。

## Skillは「必要なときに開く仕事の手順書」

たとえば、AIにコードレビューを頼むたび、次のような説明をしているとします。

```
変更差分を確認してください。
バグの可能性を優先し、重大度を付けてください。
問題がなければ「問題なし」と明記してください。
最後に不足しているテストを挙げてください。
```

1回だけなら、そのままプロンプトとして送れば十分です。

しかし、別のプロジェクトでも翌週でも同じ説明を使うなら、毎回入力するより「レビューするときは、この順番と基準で進める」と保存しておくほうが扱いやすくなります。

これがSkillの出番です。

![](https://assets.st-note.com/img/1784577983-ZaUl8YqwemCKkxEHO3WAosBI.png?width=1200)

図1：Skillは、毎回の説明を再利用できる手順と素材へまとめるイメージ。

図1は、Skillを使う前後の変化を4段階で表しています。

つまりSkillは、AIへ渡す説明を省略するだけではなく、「何を、どの順番で、どこまで行うか」を次回も使える形に整える役割を持ちます。

普通のプロンプトが「今回のお願い」だとすれば、Skillは「この種類の仕事をするときに使う、保存された進め方」です。

ただし、長いプロンプトを保存すれば何でも良いわけではありません。Skillに向いているのは、複数回使う手順や、毎回参照したい判断基準がある仕事です。

## Skillの中身はどうなっている？

Claude CodeとCodexのSkillは、どちらも`SKILL.md`を中心にしたフォルダです。

最小構成なら、必要なのはSkill用のフォルダと`SKILL.md`です。`SKILL.md`には、主に次の内容を書きます。

必要に応じて、補助ファイルも一緒に置けます。

* `scripts/`: 定型処理や検証を実行するスクリプト
* `references/`: 詳しい仕様、社内ルール、参考資料
* `assets/`: テンプレート、見本、画像など

![](https://assets.st-note.com/img/1784578069-ADQGIpPtnskR9Ozli0MeaBx8.png?width=1200)

図2 Skillの基本構造

図2の上側にある`SKILL.md`が、Skillの中心です。ここにSkillの名前、使う場面、AIが従う手順を書きます。

下側の3つは、必要な場合だけ追加する補助要素です。

* `scripts`: 計算、変換、検証など、決まった処理を実行するためのファイル
* `references`: 詳しい仕様や判断基準など、作業中に参照する資料
* `assets`: 文書テンプレート、見本、画像など、成果物づくりに使う素材

最初から4種類すべてを用意する必要はありません。まず`SKILL.md`だけで始め、文章だけでは安定しない処理や、何度も参照する資料が出てきたときに補助ファイルを追加します。

共通仕様の最小例は、次のような形です。

```
---
name: review-check
description: 変更差分を確認し、バグの可能性と不足テストを整理する。コードレビューを頼まれたときに使う。
---

1. 変更差分を確認する。
2. バグの可能性を重大度順に整理する。
3. 不足しているテストを挙げる。
4. 問題がなければ「問題なし」と明記する。
```

これは説明用の最小例です。特定の実績や、すべての環境で同じ結果になることを示すものではありません。

Agent Skillsの共通仕様では、`name`と`description`を含む`SKILL.md`が中心で、スクリプト、参照資料、素材などを追加できます。詳細は[Agent Skillsの仕様](https://agentskills.io/specification)で確認できます。

## なぜ全部の説明を最初から読み込まないの？

Skillでは、「段階的に読み込む」という考え方が使われています。

大まかな流れは次の3段階です。

1. 最初はSkillの名前と説明を見て、使えそうなSkillがあるか判断する。
2. 依頼に合うSkillが選ばれたら、`SKILL.md`の詳しい手順を読む。
3. さらに必要なら、参照資料を読んだり、同梱スクリプトを使ったりする。

すべての手順書を最初から広げておくのではなく、仕事に合うものだけを開くイメージです。

このため、`description`は単なる紹介文ではありません。「何をするか」「どんな依頼で使うか」を判断する重要な入口です。曖昧すぎると選ばれにくく、広すぎると関係のない場面で選ばれることがあります。

CodexとClaude Codeの公式資料でも、名前や説明を手掛かりにSkillを見つけ、必要になったときに詳しい内容を読み込む流れが説明されています。出典は[CodexのSkill公式資料](https://learn.chatgpt.com/docs/build-skills)と[Claude CodeのSkill公式資料](https://code.claude.com/docs/en/skills)です。

## Claude CodeとCodex、何が同じで何が違う？

共通しているのは、`SKILL.md`を中心に、再利用したい知識や手順をまとめる考え方です。

一方、明示的な呼び出し方や保存場所には違いがあります。

### 

![](https://assets.st-note.com/img/1784578443-Lu5rvdlSaGeE7ykC8j9iFstm.png?width=1200)

ここで大切なのは、「共通仕様を使っている」ことと「すべてのSkillが無調整でまったく同じように動く」ことを分けて考えることです。

基本的な指示や資料は共有しやすい一方、それぞれの製品には独自の設定や機能があります。呼び出し方、利用できるツール、権限、スクリプトの実行環境が違えば、同じ`SKILL.md`でも結果が変わる可能性があります。

最初は「共通する芯は同じ。置き場所と製品固有機能は別」と覚えれば十分です。

Skillが分かりにくい理由の1つは、似た言葉が同時に登場することです。

言い換えると、MCPは「外部とつながる手段」、Skillは「つないだ道具をどう使って仕事を進めるか」という手順にも使えます。両者は競合するものではなく、組み合わせることもできます。

## どんな作業がSkill向き？

たとえば、次のような作業はSkillの候補になります。

### 1. コードレビューの観点をそろえる

確認する順番、重大度の付け方、出力形式が毎回ほぼ同じなら、再利用する価値があります。

### 2. リリース前の確認を抜けにくくする

テスト、ビルド、変更履歴、公開後の確認など、順番と完了条件が決まっている作業はSkillへまとめやすい例です。

### 3. 決まった形式の文書を作る

必要な入力、見出し、チェック項目、保存形式が決まっているなら、テンプレートや見本もSkillへ含められます。

反対に、次のような依頼は、無理にSkillへしなくても構いません。

最後のような処理は、製品や目的に応じてHookや通常の自動化が適する場合があります。この記事では、詳細には踏み込みません。

## 5分でできる「Skillにする？」3問チェック

ここまで読んだら、実際に自分の作業を1つ思い浮かべてください。

次の3問に答えます。

### 質問1：同じ説明を、すでに何度か繰り返している？

毎回ほぼ同じ前提、チェック項目、出力形式を伝えているなら、Skill候補です。

### 質問2：共通する入力、手順、完了条件を言葉にできる？

「何を渡し、どの順で進め、何ができたら終わりか」を書ける作業ほど、Skillにまとめやすくなります。

### 質問3：「常に守るルール」ではなく、「必要なときに使う仕事の進め方」？

毎回必ず守る短い決まりなら、`CLAUDE.md`や`AGENTS.md`のほうが自然です。特定の仕事をするときだけ必要になる手順なら、Skillが候補になります。

判定の目安は次のとおりです。

* 3つとも「はい」: Skill候補として、手順を書き出してみる。
* 1〜2つが「はい」: まずはプロンプトとして保存し、共通部分が固まるか観察する。
* すべて「いいえ」: 今はSkillにしなくてよい。

これは簡易な目安で、効果や再現性を保証するものではありません。Skillにした後も、実際の依頼で試し、必要に応じて説明や手順を直していく必要があります。

## 今日やることは、Skillを完成させることではない

この記事を読んだ直後に、フォルダを作ったり設定を覚えたりしなくても大丈夫です。

まずは、何度も入力している指示を1つだけメモしてください。

```
作業名:
毎回渡すもの:
毎回行う手順:
終わったと判断する条件:
```

4つを無理なく書けたら、それはSkill候補です。書けなければ、もう少し普段の使い方を観察してからでも遅くありません。

Skillは、AIに魔法の能力を追加するものではありません。

毎回説明していた仕事の進め方を、必要なときに取り出せる形へ整える仕組みです。

まずは1つ、自分が繰り返している作業を見つける。そこが、Skillを使い始める最初の一歩です。
