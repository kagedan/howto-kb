## まず、このポストが言っていること

正直、今回の話はかなり大きいです。 がチャットで答えるだけじゃなく、**実際にMac上でアプリを開き、画面を見て、クリックして、入力して、作業を進める**ところまで来た、という話だからです。 これまでのAIは、文章生成やコード補助は得意でも、最後は人がブラウザを開いて、フォームを埋めて、表計算ソフトを触って、確認作業をしていました。

今回の機能は、その「最後の人手」をかなり削りにきています。しかも対象はClaude CoworkとClaude Codeで、研究プレビューとして提供されていて、対象はPro/Max、そしてmacOSです。 > ここでの本質は、AIが「答える存在」から、画面の向こうで仕事を進める存在に変わったことだと思う。

---

## 何がそんなに違うのか

面白いのは、 がいきなり何でもGUI操作するわけじゃないことです。

まずはSlackやカレンダーみたいな接続済みの手段を優先し、それで足りない時に、画面を見てマウスやキーボードで補完する。つまり、**コネクタ優先、足りない部分だけ画面操作**という流れなんです。 これがかなり重要で、単なるデモっぽい自動操作ではなく、実務フローに寄せた設計になっています。

ブラウザ操作、ファイルを開く、開発ツールを動かす、といった作業を「設定なし」で進められる一方、新しいアプリに触る前には許可を求める設計で、止めたい時は人が止められる。安全面もかなり意識されています。 正直、ここまで来るとRPA（定型操作の自動化）とAIエージェントの境目がかなり曖昧です。

違いがあるとすれば、RPAは手順を固定してなぞるのが得意で、今回の方向は**画面を見ながら状況に合わせて進める**ことに強い、というところじゃないかなと思います。 API Docs

## 背景にあるのは、モデル性能の上振れ

この機能が急に出てきたわけではなくて、土台にはSonnet 4.6の強化があります。

Sonnet 4.6は、コーディング、computer use、長文文脈、エージェント計画、知識労働全般で底上げされたモデルとして出ていて、computer useの性能向上もかなり前面に出されています。 ベンチマークでも、実際のPC操作に近いOSWorld-VerifiedでSonnet 4.6は72.5%まで伸びています。

2024年後半の世代から見るとかなり上がっていて、**「たまに当たるおもしろ機能」から、「普通に仕事を振れるかも」と思える水準**に近づいてきたのが大きいです。 > 個人的には、今回すごいのは新しい概念そのものより、実用になる精度に寄せて製品へ載せてきたことです。

## AI駆動開発の現場では何が変わるか

ここは開発者目線だとかなり想像しやすいです。

たとえばClaude Codeがコードを書くだけでなく、**IDEを触る、ローカルでテストを回す、ブラウザで確認する、必要ならPR作成の流れまで持っていく**、という一連の仕事に近づいていきます。 CursorやGitHub Copilot系が強かったのは、主にエディタの中です。

でも今後は、エディタの外にある確認作業、データ入力、ドキュメント更新、ブラウザ検証まで含めてどこまで繋げられるかが勝負になるはずです。\*\*「コード補完」より「作業全体の委任」\*\*の比重が上がっていくと思います。 その時に必要になるスキルも少し変わります。

実装力そのものはもちろん必要だけど、それ以上に、どこまでAIに任せるか、どこで確認を挟むか、失敗しやすい箇所をどう監督するか、そういう**監督設計のセンス**が効いてきます。

## 個人開発者や小さいチームほど恩恵が大きい

これ、大企業だけの話ではないです。むしろ少人数チームのほうが効きます。

朝にスマホからタスクを投げておいて、移動中にClaudeが作業を進め、戻ったら下書きや変更案が揃っている。そういう流れが現実味を帯びてきました。Dispatchと組み合わせると、その色がかなり濃いです。 ただ、万能感はまだ危ない。

公式にも、computer useはまだ早い段階で、複雑な作業は再試行が必要だったり、信頼できるアプリから始めて、機微データは避けたほうがいいとされています。**便利さはかなり本物だけど、完全放置で任せる段階ではない**です。 個人的には、ここから先の技術スタック選定は少し変わると思っています。

モデル性能だけじゃなく、ローカル実行、権限設計、接続先、監査性、失敗時の復旧まで含めて、「どのAIがどの仕事をどこまで代行できるか」でツールを選ぶ時代に入った感じがあります。 がMacを触り始めた、というのは、その入口としてかなり象徴的です。
