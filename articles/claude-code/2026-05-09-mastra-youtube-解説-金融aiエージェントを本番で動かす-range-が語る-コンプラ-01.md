## 概要

Range の Chief AI Architect である David Cusatis が、金融アドバイスという規制の強い領域で、Mastra ベースの AI エージェントをどのように本番運用してきたかを語るセッションです。最初に「Range はどんな会社で、なぜエージェントに本気なのか」を短く共有したあと、論点を **Compliance（コンプライアンス）、Reliability（信頼性）、Observability（可観測性）** の 3 つに絞り込みます。

単なる成功談ではなく、具体的な“実装上の学び”が紹介されています。たとえば、初期には Workflow と Agent のどちらを選ぶべきかで悩み、障害耐性のために Resilient Language Model （敵対的堅牢性の高い言語モデル）を作り、品質評価のために post-generation analysis （生成後の分析） を導入し、最終的には「コンプライアンスとは explainability （説明可能性）を作ることでもある」と整理しています。

後半の Q&A では、Range がすでに 15 を超えるエージェントを本番に出していること、チーム全員を「AI エンジニア」とみなしていること、そしてモデルの進化が以前の scaffolding を食い潰していったことなどが語られます。

## 要点

1. Range は資産運用、税金、キャッシュフロー、退職設計などを統合した AI ベースの wealth management プロダクトで、Mastra 上に複数の本番エージェントを構築している
2. 登壇の中心メッセージは、金融 AI エージェントを本番で動かすには **Compliance、Reliability、Observability** の 3 本柱が必要だ、ということ
3. Workflow と Agent は対立概念ではなく、タスクの性質に応じて使い分けるべき抽象化である
4. 初期に人間向けに作っていたツールを Agent に接続しただけでも、最新モデルではかなり強く機能する
5. 障害対策として、Anthropic API から Bedrock、あるいは別モデル群へ切り替えられる Resilient Language Model を構築している
6. 品質評価では LLM-as-a-judge を使った post-generation analysis を非同期に回し、grounding、感情分析、欠落データ、訂正、各種ポリシー遵守を評価している
7. 金融領域ならではの強いコンプライアンス要件は、裏を返せば「なぜその答えになったのか」を追跡する explainability の要請でもある
8. Range ではすでに 15 以上のエージェントが本番稼働しており、PM やデザイナーもコードを push するなど、組織全体が AI ネイティブな開発に寄っている
9. モデル性能の向上によって、以前は必要だった OCR 前処理や細かい学習済みモデルによる補助の一部は不要になりつつある
10. マルチエージェント的な構成も使っているが、完全に自由な supervisor ではなく、より guided なサブエージェント実行に寄せている

## 詳細

### Range社 と その主力商品 Rai の紹介

David はまず Range がどんな会社かを説明します。単なるチャットボット会社ではなく、投資、税金、退職設計、キャッシュフローまで含む **オールインワンの wealth management プロダクト** であること。しかも SEC （米証券取引委員会）登録済みの投資顧問業者で、対面アドバイザーの代わりに AI 主導のプロダクト体験を提供していること。「これは“少し便利な AI アシスタント”の話ではなく、金融助言を含む重い業務の話だ」と。

そのうえで David は、主力エージェントの Rai を紹介します。社内の投資哲学を学習し、社内で作り込んだツール群を呼び出しながら、高品質な金融アドバイスを行う。しかも税金関連では、連邦税だけでなく全 50 州の税法をコード化した税額試算システムまで使う。さらに、オンボーディング、書類解析、社内アドバイザー支援、サポート診断といった別系統のエージェントもある。つまり一つの flagship agent だけではなく、**エージェント群がプロダクト運用の中に埋め込まれている**わけです。

この説明のあとに、David は「私たちが本番でスケールするために集中したのは 3 つだけだった」と言います。それは、Compliance、Reliability、Observability の 3 つです。

### Compliance は金融特有の話でありながら、実は explainability の話でもある

金融プロダクトなので、Range は SEC 規制に従う必要があります。ログ保持、監査可能性、各種トレースの保存などは optional ではありません。David はこの点をかなり率直に話していて、すべての LLM 呼び出し、ツール呼び出し、思考トレースにまたがるメタデータを記録する必要があると言います。

ここで面白いのは、彼が途中で「結局コンプライアンスが解いている問題は explainability なのではないか」と言い直すところです。これは金融業界以外の読者にも効くポイントです。

金融領域では、たとえば次のようなことを確認しなければなりません。

* fiduciary standard （信認義務：自身の利益よりも顧客の利益を優先し、専門家として誠実に業務を行う基準）を満たしているか
* duty of care （注意義務）や duty of loyalty （忠実義務）に反していないか
* 必要な disclosure （開示）が入っているか
* 投資助言として危険な表現をしていないか

しかし、これらは単に「法令上のチェック項目」というだけではありません。なぜその回答を返したのか、どのデータを根拠にしたのか、なぜその表現になったのかを追えなければ、そもそも規制対応はできません。つまりコンプライアンス要件が厳しいほど、システムは explainability に寄らざるを得ない。

この整理は非常に重要です。一般的な AI プロダクトでも、ユーザーから「なぜそうなったのか」を問われる場面は増えています。金融はその圧力が最も強い領域の一つで、そこで鍛えられた設計は他分野にも応用できます。

### Reliability は「モデルが賢いか」ではなく「落ちても返せるか」の話

David の Reliability の話もかなり現実的です。金融のような複雑な話を AI に相談していて、途中で動かない、返ってこない、突然品質が崩れる。これが最悪だと彼は言います。そして、その不快さをなくすために、かなり意識的にフェイルオーバー戦略を組んでいます。

ここで出てくるのが **Resilient Language Model** です。発想自体はシンプルで、障害時に別のプロバイダーや別モデルへ切り替えられるようにするものですが、David が強調するのは「単純に Bedrock へ逃がせばよいわけではない」という点です。なぜなら、みんな同じことを考えるからです。

この指摘は本番運用のリアリティがあります。個別のアプリから見ると、Anthropic API が不安定なら Bedrock へ逃がせば良さそうに見えます。しかし大規模障害の局面では、他社も同じ経路へ殺到し、結局連鎖的に詰まる。つまり、フェイルオーバーがあるだけでは足りず、**どう制御するか**が本質になるわけです。

これは最近の LLM アプリ全般に共通する話でもあります。モデルの能力比較やベンチマークの議論は盛り上がりますが、実際の運用では「どのモデルが一番賢いか」以上に「どのように degradation （劣化）し、それをどう吸収するか」が重要になる。David の話はそこをよく表しています。

### Workflow と Agent の使い分けを、かなり冷静に整理している

初期の Mastra には、Workflow か Agent か、かなりはっきりした二択の感覚があったと David は振り返ります。そして Range はかなり Workflow に寄せて使っていたと言います。理由はレイテンシや処理の決定性を重く見ていたからです。

このあたりの話が良いのは、「Agent がすごいから全部 Agent にする」という雑な方向に流れていないことです。David は明確に、**正しい抽象化を選べ**と言います。

彼が挙げるドキュメント解析の例は分かりやすいです。

1. 前処理する
2. 分類する
3. 情報を抽出する
4. 正しくできたか検証する

こういう処理は Workflow の方が向いている。一方で、オープンエンドなタスクや、多少レイテンシに余裕があるものは Agent が向いている。そして最新モデルは、多くのツールを与えたとき非常にうまく働く。

この整理は、今後エージェントシステムを作る人にとってかなり実務的です。Mastra のように Agent と Workflow が composable な（組み合わせ可能な）フレームワークを使うなら、どちらか一方に思想的に寄るより、**タスクの性質で分ける方が強い**。Range のケースはその実例になっています。

### 人間向けに作ったツールを Agent に接続したら、そのまま戦力になった

個人的にかなり印象に残ったのは、「これらのツールは人間向けに先に作っていたもので、Agent に配線したらそのまま動いた」という話です。

これはエージェント開発を特別なものにしすぎない、良い視点です。AI のためにゼロから全部設計し直すのではなく、既存のドメインロジックや社内システム、既存ツール群を“使える形で expose する”ことの価値が大きい。しかも最新モデルは、そのようなツール群をかなり上手に扱える。

この話は、エージェント時代のアーキテクチャの方向性も示しています。企業がすでに持っている強みは、たいてい UI ではなく、内部の処理ロジック、ルールエンジン、データパイプライン、検証手続き、業務ワークフローにあります。Range では税額試算システムや人間アドバイザー向けツール群がそれにあたる。エージェントを足したから価値が出たのではなく、**もともとの深い業務資産を Agent interface 化したから強かった**、という構図です。

### Observability は“見える化”ではなく、エージェントを理解するための前提

David は observability をかなり重く扱っていますが、その理由は単に「グラフが見たいから」ではありません。サブエージェント、各ステップ、ツール呼び出し、ネストした実行、その全体を追えなければ、何が起きているか分からないからです。

Mastra と AI SDK の execution lifecycle hooks がここで効いています。David が挙げているのは、たとえば次のような観測点です。

* サブエージェント実行
* 各ステップの開始と完了
* ツール呼び出しの開始と終了
* レイテンシ
* 成功率と失敗率
* トークン使用量
* アーカイブと監査ログ

この発表を見ていると、observability は「あとで見る補助情報」ではなく、**本番 Agent の構成要素そのもの**だと分かります。観測できないものは改善もできませんし、コンプライアンスも満たせませんし、品質悪化の兆候も見えません。

Mastra Studio が最近メトリクスやロギングを強化している流れとも、非常に相性の良い話です。実際、David も「Mastra がそのあたりを出し始めたので試すとよい」と言及しています。

### Post-generation analysis がこの発表の中心技術

このセッションの技術的なハイライトは、やはり **post-generation analysis** です。David 自身、「この 1 年で自分の起きている時間の多くを、AI 金融アドバイスの品質をどう測るかに使った」と言っています。かなり重い言葉です。

やっていることを単純化すると、生成された応答を、その後段で別の評価パイプラインにかけ、LLM-as-a-judge を使って複数指標で採点する、というものです。そして、その結果をライブの脈拍のように監視し、必要ならエスカレーションや修正に使う。

見ている項目も非常に実務的です。

* grounding rate （根拠データ活用率）
* user sentiment （ユーザー感情）
* missing data （欠落データ）
* corrections （訂正・修正指摘）
* compliance and policy checks （コンプライアンスおよびポリシーチェック）

この設計が面白いのは、Eval を開発時の一括評価に閉じ込めていないことです。多くのチームにとって Eval は「実験時にまとめて回すもの」ですが、Range はそれを **運用時の live pulse** に近い位置まで持ち込んでいます。金融という高リスク領域では、これはかなり自然な発想です。モデルが 1 回良かったからといって安心できない。だから本番の各応答に対して、ポストホックにでも評価をかけ続ける必要がある。

Mastra で非同期ジョブやバックグラウンド処理を絡める価値が、ここで非常によく見えます。回答生成そのものとは別系統で、品質監査とコンプライアンス監査を走らせる。これは今後いろいろな業界で必要になるパターンだと思います。

### Q&A で見えるのは、Range がすでに「組織ごと AI 化」していること

後半の質疑応答も面白いです。短い Q&A ですが、発表本編では見えにくかった組織の実態が出てきます。

まず驚くのは、本番で 15 以上のエージェントが動いていることです。しかも、それぞれで「良い」の定義が異なる。だから post-generation analysis の基準も一枚岩ではなく、各エージェントごとに違う。これは当然と言えば当然ですが、実際にそれを運用の前提にしているのが重要です。

さらに David は「技術職 25 人くらいの全員が AI エンジニアだと思っている」と言います。しかも PM やデザイナーまでコードを push している。これは単に AI ツールを使っている、という話ではなく、**組織のソフトウェア開発の境界が崩れ始めている**ことを示しています。

もちろん、誰もが高度な LLM システムをゼロから設計しているわけではありません。しかし、社内に AI platform 的な土台があり、その上に多様な役割の人が貢献している。これは今後かなり一般化しそうな組織像です。

### モデルが良くなると、以前の工夫が不要になる

Q&A でとても良い話がもう一つあります。それは、昔は必要だった工夫が、モデルの改善で不要になっていくという話です。

Range のオンボーディングでは、銀行データ接続が 80% くらいはうまくいくが、残り 20% が痛い。その補完として、書類や明細をアップロードしてもらい、そこから情報抽出する仕組みが必要だった。以前は OCR して、その結果を整えて、Structured Outputs に落とすという多段処理をしていた。でも最近は、多くのケースで OCR ステップを飛ばせるようになった。

これは “agents are eating OCR” という軽い冗談で流されていますが、実際にはかなり本質的です。以前なら周辺で支えていた前処理や補助モデルや細かな最適化が、モデルの改善によって不要になっていく。David も、昔は独自モデルを訓練してワークフローの一部に差し込み、レイテンシ最適化をしていたと言いますが、今はその多くが agentic な workflow に置き換わっていると話します。

いわゆる bitter lesson 的な話ですが、ここで面白いのは、彼がそれを「残念なこと」ではなく「良いこと」として受け止めている点です。下の層の複雑さより、上の層で扱えることが増える方が、プロダクトチームとしては圧倒的に回しやすい。

### マルチエージェントは使っているが、かなり guided

最後の質問では、Workflow と Agent の比較から一歩進んで、multi-agent systems や supervisor pattern の話が出ます。David の答えはかなり慎重です。サブエージェント的な実行はあるけれど、完全に自由な、オープンエンドな supervisor ではない。もっと guided な構成だと言います。

ここにも Range らしさが出ています。金融助言という領域では、自由度より制御の方が価値が大きい。Mastra の composability は活かしつつも、実行の開放度は意図的に抑えている。つまり、**マルチエージェントだから高度、ではなく、必要十分な自律性だけを与える**という姿勢です。

このあたりは、エージェントを現実の業務へ入れるときの大事な感覚だと思います。研究っぽい open-ended な自律性は面白いですが、本番ではまず責任分界と観測可能性が必要になる。Range の選択はかなり納得感があります。

## まとめ

このセッションは、金融という厳しいドメインの事例でありながら、実はかなり普遍的です。David Cusatis が示していたのは、「AI エージェントを本番に出す」とは、賢いモデルを選ぶことではなく、**Compliance、Reliability、Observability を一体の設計として持つこと**だ、ということでした。

今後 Mastra で本番運用を考える人にとって、この動画はかなり良いケーススタディです。特に「Agent を作ること」と「Agent を運用すること」の違いを実感したい人には興味深いと思います。
