## Claude Codeで業務自動化ツールを作った時の失敗から学んだこと

私がClaude Codeを使って中小企業向けの業務自動化ツールを開発する中で、何度も失敗してきました。その過程で得た学びを今回は共有したいと思います。

「Claude Codeなら簡単に自動化ツールが作れるのでは？」そう考える人は多いです。実際、私もはじめはそう思っていました。でも現実はそう甘くなかったんです。

  

## 失敗から生まれた、本当に使える自動化

### 失敗① エラーハンドリングを甘く見ていた

Claude Codeを初めて使ったとき、私は **自動生成されたコードが完璧に動くと思い込んでいた** んです。実際に開発した最初の案件は、スプレッドシートから顧客データを自動抽出するツールでした。

実装から2日後、クライアントから連絡が。データフォーマットが想定と異なると、ツール全体が止まってしまうという問題が発生していました。

その時に気づいたんです。Claude Codeが生成するコードは、基本的には「happy path」（すべてが正常に動く想定）で作られていることが多いということを。

現在は、実装時に必ず以下の点をチェックリストに入れています：

* 不正なデータ形式が入った場合の処理
* APIレスポンスが遅延した場合のタイムアウト処理
* 部分的なネットワーク障害への対応
* ログ記録機能の実装

これを徹底するだけで、本番環境での問題が大幅に減りました。

### 失敗② 複雑な業務ロジックを一度にAIに任せようとした

別の案件では、複雑な在庫管理システムの自動化をClaude Codeに任せました。業務フローは以下の通り：

1. 複数のサプライヤーからの発注データを集約
2. 既存在庫との照合
3. 需要予測に基づく自動発注判定
4. 承認フロー
5. 外部APIへの自動送信

**最初は一つのプロンプトで全てを要求しました。** その結果、生成されたコードは確かに長いものの、複数の部分で判断の曖昧さが残っていました。

この失敗から学んだのは、業務自動化は「分割統治」が重要ということです。

Claude Codeを使うときは、大きなタスクを以下のように細分化するようになりました：

* **ステップ1：データ抽出と整形機能**（Supabaseを使ったDB操作）
* **ステップ2：ロジック検証機能**（検証ルールを明確に定義）
* **ステップ3：外部連携機能**（API呼び出しと例外処理）
* **ステップ4：ログと監査機能**（実行履歴の記録）

このアプローチで、各ステップが独立してテスト可能になり、バグの原因特定も簡単になりました。実装時間は確かに長くなりますが、**保守性と信頼性は大幅に向上** します。

### 失敗③ ユーザーインターフェースを過小評価していた

3ヶ月目の失敗は、正直ショックでした。

Claude Codeで自動化ロジックは完璧に実装されていたのに、クライアントから「社員が使わない」という報告をもらったんです。原因は、実行結果が単純なテキストログで表示されていただけだったこと。

技術者の私たちにはそれで十分でも、実際に使う現場の人たちにはわからない。ダッシュボードが欲しい、結果を見やすく表示してほしい、エラーが起きたときは警告を目立たせてほしい、という要望が次々と出てきました。

これ以降、私は **Claude Codeの使い方を拡張** するようにしました。

Claude Codeは単なるロジック実装ツールではなく、フロントエンドの実装にも使えるんです。Vercelにホストされた簡単なWebUIを、Claude Codeで生成させることで、非技術者でも直感的に使える自動化ツールに仕上げることができました。

現在は、自動化ツール開発の流れとして：

1. Claude Codeで業務ロジックの実装
2. Claude CodeでシンプルなReactコンポーネント生成
3. Vercel経由で簡単にデプロイ

このワークフローが標準になっています。

## 実践的な改善策

### テストファースト思考の導入

失敗を重ねた結果、私が採用したのは **テストケースの事前準備** です。

Claude Codeにコードを生成してもらう前に、こちらから与えるテストケースを用意しておくんです。例えば：

```
入力：不正なメールアドレス形式
期待される出力：エラーメッセージ「メールアドレスの形式が正しくありません」

入力：空のデータセット
期待される出力：「処理対象がありません」というログ出力
```

このようなテストケースをプロンプトに含めておくと、Claude Codeが生成するコードの品質が劇的に向上します。

### プロンプト設計の工夫

もう一つ重要な学びは、**プロンプトの質がコード品質を左右する** ということです。

曖昧なプロンプト：  
「ユーザーデータを処理するコードを作ってください」

明確なプロンプト：  
「Supabaseから取得したユーザーデータを処理します。以下の条件を満たしてください：(1)メールアドレスのバリデーション、(2)重複排除、(3)結果をCSVで出力、(4)エラーログの記録」

後者の方が、期待通りのコードが生成される確率が圧倒的に高いです。

## 中小企業での実装経験から見えたこと

これまで10社以上の中小企業でAIエージェントの導入をサポートしてきた中で感じるのは、**技術的な完璧さより、実用性が優先される** ということです。

完璧なコードより、「今その企業が必要としている部分から始める」というアプローチが重要なんです。

例えば、手作業で月10時間かかっている業務があれば、その部分だけをまずClaude Codeで自動化する。そして3ヶ月使ってもらって、フィードバックをもらった上で拡張する。このような **段階的な実装** が現実的で、成功率も高いです。

## 学んだベストプラクティス

### 1. エラーハンドリングを最優先

フレームワークやライブラリ選択より先に、エラーハンドリング戦略を決める。

### 2. 小さく始めて段階的に拡張

完全な自動化より、まず部分的な自動化から。ユーザーフィードバックを基に改善。

### 3. ログとモニタリング機能を必須に

何が起きたかを追跡できないツールは、本番環境では使い物になりません。

### 4. UIまで込みで考える

技術者向けのツールとして設計していませんか？実際のユーザーを想定した設計を。

### 5. Claude Codeは万能ではないことを理解する

Claude Codeは強力ですが、複雑なシステム設計や大規模な最適化が必要な場合は、人間の判断が必要です。

## 次のステップ

Claude Codeを使った業務自動化は、確かに開発スピードを大幅に短縮できます。ただ、失敗を避けるには、**段階的アプローチ、十分なエラーハンドリング、ユーザーを想定した設計** が不可欠です。

今、業務自動化を検討されている方や、Claude Codeの導入を考えている組織の方へ。失敗から学ぶより、他人の失敗から学ぶ方が効率的です。
