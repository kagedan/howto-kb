# はじめに

Claude Codeで日々コードを書いたり調査をしていると、会話の中で生まれる知識が想像以上に多いことに気づきます。

* バグの根本原因と対処法
* 画面操作の手順
* 「この機能は実はこういう仕様だった」という発見
* ドメイン知識

でもこれらは、会話を閉じた瞬間に消えます。メモリーに残るものもありますが、限定的です。かといってAIに要約させて、コピペしてWikiに貼るのは面倒で、結局続きません。

そこで、会話を止めずに「〇〇に保存して」「あの件のドキュメントある？」と言えばClaudeが勝手にやってくれる仕組みを、Claude CodeのSkill機能で作りました。この記事はその設計と、真似して自分で作るためのポイントの解説です。

本記事ではドキュメント管理ツールを「hedgedoc」で説明してますが、ツール自体はAPIで管理できれば、なんでも問題ないです。

# 概要

保存と検索2つのSkillを作りました。

### `/hedgedoc-save` （保存）

```
ユーザー:「/hedgedoc-save」
Claude: 「hedgedoc-saveにバックグラウンドで保存中です。」
　　　　（数秒後）「保存完了しました: https://.../xxxxx」
```

会話の流れを止めずに、直前までのやり取りからMarkdownを作成 → ドキュメント管理ツールに投稿してくれます。「hedgedoc-save 〇〇」とすることで、会話の一部を切り取ってドキュメント化することも可能です。

### `/hedgedoc-search` （検索）

```
ユーザー:「/hedgedoc-search XXXシステムの申込の際、選択可能な決済方法教えて」
Claude:
| # | title | URL |
| 1 | XXXシステム 申込フロー・決済方法一覧 | ... |
| 2 | XXXシステム 決済手段ごとの注意事項 | ... |
| 3 | XXXシステム 申込画面の仕様書 | ... |
読み込むドキュメントの番号を指定してください。

ユーザー:「1」
Claude: （1のドキュメントのURLを返し、ドキュメントの本文を取り込んでコンテキストに含める）
```

キーワードで既存ドキュメントを絞り込み、選んだドキュメントの本文をコンテキストに取り込んで続きの会話ができます。

# アーキテクチャ全体像

| レイヤ | 役割 | 実体 |
| --- | --- | --- |
| 認証 | Cookieの取得と使い回し | shellスクリプト1本 |
| 保存/検索ロジック | 会話→Markdown化、履歴→検索 | SKILL.md（Claudeへの指示書） |
| 実通信 | HedgeDoc APIを叩く | サブエージェント |

Claudeは「司令塔」で、APIを叩くのはサブエージェント、という分業にしているのが肝です。

# 設計上の意識したこと

## 検索ロジック

検索の際、いかにトークンを節約してユーザが求めている結果を返せるかが重要です。  
例えばドキュメントの本文すべてをAIに読ませて、ユーザが求めているものを返すとすれば、結果が素晴らしいものが返ってきますが、トークンの使用量がすさまじいものになります。一方で、タグ検索やタイトルのLIKEにすればトークンはほぼ使用しませんが、ユーザの検索体験はあまりよくありません。

なので、SKILLSでも使われている「title」「description」とメタデータをドキュメントに持たせて、以下のような検索ロジックにしました。

1. メタデータの一覧を取得
2. メタデータだけでフィルタし候補をテーブルで提示
3. ユーザーが番号指定
4. 選ばれたドキュメントの本文だけ取得

これにより、トークンの使用量をある程度抑えつつ、より良い検索体験を提供します。

## 実通信はサブエージェントに投げる

### 投稿

HedgeDocへの投稿はAPIを複数回叩きます（本文POST → 履歴更新POST）。ネット越しに数十秒かかります。  
これをメインセッションで同期的にやると、ユーザーは次の入力までずっと待たされます。会話の流れが切れるのは最悪です。  
また、投稿は単に決まった作業やAPIを叩くだけなので、メインセッションで使う高性能なモデルを利用する意味がありません。

そこで、

* メインセッション: 会話からMarkdownを組み立てる
* サブエージェント（Haiku）: 投稿作業を丸投げ

という分担にします。投稿が終わるとサブエージェントが完了通知を返してくれます。

### 検索

検索はもっと切実な理由があります。ドキュメント一覧の全メタデータを取得すると、ドキュメントが増えれば増えるほどメインセッションのコンテキストを荒らします。

そこでサブエージェント（Sonnet）に「一覧取得→キーワードフィルタ→結果テーブルだけ返す」を任せます。メインセッションには絞り込まれた結果表だけが戻ってきます。

「重い・汚い仕事はサブエージェントに隔離する」というのは、Claude Codeでの実装全般に効く設計パターンです。

# 同じ設計で他のツールにも応用できる

今回はHedgeDocですが、この骨組みはツール非依存です。以下のどれでも同じ構造で作れます。

| ツール | 認証 | 保存API | 検索API |
| --- | --- | --- | --- |
| HedgeDoc | Cookie | POST /new + history | GET /history + /{id}/info |
| Notion | Bearer token | POST /pages | POST /search |
| Confluence | Basic/Bearer | POST /content | GET /content/search |

変わるのはAPIの殻だけで、

* Skill × 2（保存・検索）
* 認証スクリプト × 1
* サブエージェントに実通信を委譲

という構成はそのまま使えます。
