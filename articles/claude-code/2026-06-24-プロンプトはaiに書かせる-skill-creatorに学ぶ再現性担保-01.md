## はじめに

AIに何か仕事をさせたいとき、どういうプロンプトを書けばいいかを考える時間が発生する。  
どうすれば脳内で描いてる通りに動いてくれるか、もしくは自分のイメージ以上のことを遂行してくれるか。

そもそもシンプルなタスクなら、AIにプログラムを作らせてそれを使えばいい。  
プログラムではなくAIそのものに仕事をさせるなら、ゴールに到達するまである程度の柔軟性は欲しかったりする。  
柔軟性があるゆえに、分岐は多岐に渡るし、想定外のこともきっと多い。  
それらを考慮したプロンプトを自分の力で作るとなると、相当骨が折れるし頭が痛くなる。

そもそもそんな苦労を要することはなく、  
**仕事をさせるためのプロンプトはAIに書かせるべきだ、と思い至った。**  
というのがこの記事の出発点。

人間がやるべきなのは、**ゴールと前提**をAIに伝えてプロンプトの作成を依頼するだけでよく、その過程でプロンプトの**評価軸**(何が成功で、何が失敗か) もAIに提案してもらう。

このとき、**評価軸**こそがプロンプトの品質を担保する。  
プロンプトの品質とは、結果の再現性だ。

そうなると、**評価軸**の正当性はどうやって担保するの? と堂々巡りになるわけだが、  
Anthropic公式の`skill-creator`というskillに「プロンプトの品質担保」のイロハが盛り込まれている。  
今回は`skill-creator`がどうやって「品質担保」を行なっているかを紐解いてみたい。

なお、本記事はClaude CodeやClaude.aiを触ったことがあり、AIに定型作業を任せていきたい人を想定読者とする。

## まずはskillsを押さえる

`skill-creator`とは「skillを作るためのskill」である。  
本題に入る前に、その「skill」とは何かをさらっと押さえておく。

skillsはAnthropicがClaude向けに導入した機能 (2025年10月リリース)。  
その後Anthropicは[agentskills.io](https://agentskills.io)でオープン標準として公開している。  
本記事はClaudeのskillsを扱う。

skillsとは1ディレクトリ単位の、AIに向けて書かれた指示プロンプトの名称。  
最低限`SKILL.md`という1ファイルだけあればよく、構造はYAMLのfrontmatter + Markdown本文の2つだけで構成されている。

```
---
name: skill-name
description: 何をするか + いつ使うか
---

# 本文

具体的な手順をここに書く
```

`description`に書いた内容にマッチする状況をAIが察すると、自動でこのskillを読み込んでくれる。  
つまりskillは「いつ使うか」をAI側に判断させる仕組みを持っている。  
本文 (Markdown部分) は具体的な手順・ルール・例を書く欄になる。

ここまでが機能としてのskillをサクッとおさらいした。  
ここから先は本記事の主題、「プロンプトの品質」の話に入っていこうと思う。

## プロンプトの品質が大事

skillの中身はAIに向けた指示プロンプトだと書いたが、適当に書いて放り込んでもうまく動くとは限らない。理由は、

**LLMは純粋関数ではないからだ。**

普通の関数は`f(1, 1) = 2`なら何度呼んでも2を返す。LLMは違う。  
複雑なタスクや曖昧な指示には、呼ぶたびに「揺れた」結果が返ってくる。

skillはLLMに対する指示なので、この「揺れ」と向き合う必要がある。  
同じ状況なら同じように振る舞ってほしい、つまり**結果の再現性**を担保した「設計されたプロンプト」が品質の高いskillだ。

### アンチパターン: 手順が曖昧 → 推論に頼って結果が揺れる

設計が無いとどうなるかの例を1つ。

```
# my-skill

良い感じにPRの説明を書いてください。
```

これは極端な例だが、似たような「ふんわり指示」のskillsは実際にある。「良い感じ」が何かはAIが推論で埋めるしかない。  
推論で埋まる部分は呼び出しごとに違う出力になる。  
揺れると再現性が失われていくのは明白だ。

具体的に何をどうするかが書かれていないと、呼ぶたびに違う結果になるskillができあがりかねない。

## Anthropic公式の`skill-creator`

冒頭で触れたAnthropic公式の「skillsを作るためのskill」[`skill-creator`](https://github.com/anthropics/claude-plugins-official/blob/main/plugins/skill-creator/skills/skill-creator/SKILL.md)について、どういうものなのか、どのように品質を担保しているのかを見ていく。

### `skill-creator`が踏ませる6つの工程

以下では、いちばんフルに動くClaude Codeの挙動を基準に紹介する (Claude.aiやCoworkでの挙動の違いは末尾の補足を参照)。`skill-creator`はskill作成者に次の工程を踏ませる。

1. 意図の把握
2. SKILL.mdの作成
3. テストケースを2-3個提案
4. 評価ループ
5. 改善ループ
6. descriptionの最適化

順に見ていく。

### 1. 意図の把握

ここが冒頭で書いた「人間がゴールと前提を伝える」工程にあたる。最初にユーザーに4つの質問をする。

1. このskillでAIに何をさせたいのか
2. いつ発火させたいのか (どんなユーザーの発話・状況で)
3. 期待する出力形式は何か
4. テストケースを用意するか

### 2. SKILL.mdの作成

構成はfrontmatterの`name` (skill識別子) と`description` (発火条件 + 機能)、それに本文の組み合わせになる。  
特徴的なのは`description`の書き方である。

`description`は発火条件としての役割を持つので、

* **何をするskillか**(機能)
* **どんな状況で使うべきか**(発火条件)

の両方を書く。`skill-creator`のドキュメントには「Claudeはskillsを、本当は使うべき状況でもスルーしがちなので少し押しつけがましいくらいに書け」とも書かれている。たとえば「会議の話が出たら、ユーザーが明示的に『議事録』と言わなくても、必ずこのskillを使え」のように。

### 3. テストケースを2-3個提案

ここは`skill-creator` (= AI側) がドラフトを作り、ユーザー (= 人間) がレビューしてOKを出す協業フローになる。  
「現実のユーザーが実際に言いそうな」プロンプトを2-3個用意する。

ポイントは「**抽象的でなく、具体的に書く**」ことだと考えている。

* 悪い例: 「データをフォーマットして」
* 良い例: 「先週の議事録 (Downloadsの `議事録_最終確定_v2.docx`) から、決定事項だけ箇条書きで抜き出してほしい」

なぜかというと、現実の状況には文脈・固有名詞・タイポ・口語表現がたくさんあるからだ。  
クリーンすぎるテストケースは「テストにはパスするのに本番では動かない」状況を生みやすくなる。

### 4. 評価ループ

ここが`skill-creator`の中核となる部分ではないだろうか。冒頭で書いた**評価軸の正当性**を担保する仕組みにあたる。

skillを書いたら、各テストケースについて\*\*「skillありで実行するsubagent」と「skillなしで実行するsubagent」を同じターンで並列に起動\*\*する。そして両者の出力を比べる。

* skillありの方が良い結果になっているか
* そもそもskillなしでも十分なのか (skillいらない?)
* どの検証項目が通った / 通らなかったか

評価結果はUIで開いて、ユーザーがレビューする。

なぜskillなしと比較するのか?

これは重要な視点の1つではないかと思う。「skillありで良い結果が出た」だけだと、

> 「元からAIができることをやっただけかもしれない」

という疑念が残る。**skillが実際に効いているかどうかは、skillなしと比較してはじめて見えてくる**と考えている。

なお、ここで述べているのは新規作成時の挙動だ。既存skillの改善時は、比較対象が「旧バージョン」になる。

### 5. 改善ループ

レビューで指摘が出たら直して、また同じテストケースで評価する。これを納得いくまで繰り返す。

ここで`skill-creator`が強く言ってる注意点がある。

> **特定のテストケースを直すんじゃなく、一般化して直せ。**

たとえば「テストケース3でこの部分が変な出力になった」というフィードバックに対して、「テストケース3のときだけXすること」と書き加えてしまうのは避けたい。それでは過適合に寄りやすく、結果としてテストケース3でしか効きにくいskillになりかねない。

そうではなく「この種のケースではXしがちなので、Yという方針で考える」のように、一段抽象化してから直す。

### 6. descriptionの最適化

skill本体ができたら、最後にdescriptionを最適化する。

発火条件用の評価ケースを2グループに分けて準備する。

特に発火すべきでない側で、\*\*「キーワードは似ているが文脈的にはこのskillじゃない」\*\*という紛らわしいケースを入れることが推奨されている。たとえばPDF操作skillに対して「フィボナッチ関数を書いて」みたいな自明な発火すべきでないケースはテストとしては弱い、というわけだ。

最適化スクリプトがdescriptionの候補をいくつか提案 → 各候補で評価ケース全体を実行 → 一番スコアが高いものを選ぶ、というループを回す。

このとき、評価データを学習用と検証用に分けて、**検証用スコアでベストを選ぶ**のがポイントだ。学習用と検証用の両方を見ることは、過適合を防ぐ常套手段でもある。

## まとめ: `skill-creator` から学べるプロンプト品質の原則

`skill-creator` から、プロンプト品質を担保するための原則が見える。

### 原則1: 書き手は自分のプロンプトを評価できない

書き手は「読めば分かるよね」と思ってしまうが、第三者には伝わらない。対策は**別のエージェント (subagent) に実行させる**こと。書き手の意図を持たずに読んだとき、どう動くかを観察する。

### 原則2: 複数の、現実的なテストケースで測る

具体的な状況・固有名詞・口語表現を含む「現実的な」テストのほうが、本番での挙動を予測しやすい。

### 原則3: 比較対象を置く

「良くなった気がする」を「実際に良くなった」に変えるには比較対象が要る。新規作成時と改善時では、比較対象も変わる。

### 原則4: 定量 + 定性の両輪で評価する

アサーション (定量) + ユーザーレビュー (定性) を併用する。数字だけ見ると「数字は上がったが何かおかしい」を取り逃しやすい。

### 原則5: 一発で完成しない、ループする

評価 → 改善 → 評価を回す。そして**過適合に気をつける**(特定例ではなく一般化して直す、description最適化は学習用 / 検証用を分けて検証用で選ぶ)。

---

skillとは要するに「設計されたプロンプト」だ。  
この設計の核心は、はじめに書いたとおり、品質を担保する「**評価軸**」にある。  
プロンプト本文は、つまるところ評価軸に向かって最適化された結果である。

## skill化のメリットと第一歩

skill化することには、プロンプト品質を担保できる以外にもメリットがある。

* **業務内容が明確になる**: 自分の作業をskillとして言語化することで、なんとなくやっていた手順が言葉に落ちる
* **属人化を避けられる**: skillとして共有可能な形にしておけば、他者にも同じやり方で実行してもらえる
* **ナレッジが貯まる**: skillが10個20個と溜まってくると、それ自体が組織の財産になる

最初に作るなら、自分が繰り返しやっている定型作業を1つ選んでみるとよい (議事録の要約、日報・週報の整形、問い合わせメールの返信ドラフトなど)。逆に、毎回大きく文脈が変わるタスクや判断基準が一定しないタスクは、skill化しても効きにくい。

LLMにタスクを実行させたいなら、まずは`skill-creator`でskillを作ってみる。  
作成されたskillの内容を読んでみたり、実際に使ってみて改善点を見つけブラッシュアップしたりする。そうしてプロンプトの品質を少しずつ理解していくことが、AIに仕事をさせるうえでの入り口になるのではないかと思う。

## 補足: 利用環境

`skill-creator`はClaude Codeだけでなく、Claude.ai (chat / Cowork) やClaudeのデスクトップアプリでも使える。  
ただし環境ごとに使える機能が違うので、評価ループの動き方が少し変わる。

| 環境 | subagent | ブラウザ | 評価ループの動き方 |
| --- | --- | --- | --- |
| Claude Code | ○ | ○ | skillあり / skillなしのsubagentを並列に実行して比較するフルバージョンが動く |
| Claude.ai (chat) | × | ○ | 並列比較は省略。skillを読んでテストプロンプトを順次実行し、結果を会話の中で見せてユーザーがレビュー |
| Cowork | ○ | × | 評価結果は静的HTMLとして書き出し、ユーザーが手元で開く |
