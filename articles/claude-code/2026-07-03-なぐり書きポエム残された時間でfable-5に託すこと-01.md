7/7で通常ユーザーがFable 5を使えなくなり、以降は従量課金制になるとのことです。といっても余力があれば通常利用ができるように戻してくれるということですが、いつになるのかは不明です。  
ではいまのうちにやれることをやってしまおうということで自分がやっていること、これからやることを書いていこうと思います。この記事がでたときにはあと４日間しかないので急ぎましょう。  
というポエムです。

## 前に使った時と異なり、セーフガードがすぐに働く

![](https://static.zenn.studio/user-upload/7f4906f69013-20260703.png)  
前までは無条件に使えたのですが、今は悪用されないようにと怪しかったらすぐに切り替わる仕様となっています。割とすぐに切り替わるので若干いらっとしますけど、今ここで文句言う暇がないので、なんとか使えるようにするのが先決です。  
[Claude があなたとの Fable 5 との会話でモデルを切り替えた理由](https://support.claude.com/ja/articles/15363606-claude-%E3%81%8C%E3%81%82%E3%81%AA%E3%81%9F%E3%81%A8%E3%81%AE-fable-5-%E3%81%A8%E3%81%AE%E4%BC%9A%E8%A9%B1%E3%81%A7%E3%83%A2%E3%83%87%E3%83%AB%E3%82%92%E5%88%87%E3%82%8A%E6%9B%BF%E3%81%88%E3%81%9F%E7%90%86%E7%94%B1)

### これやったらだめだった

* 競馬アプリの改修  
  うちで使っている競馬分析のアプリはなにをやってもすぐに弾かれました。おそらく上記の生物関係というのが要因かと推測しています。変数にhorseとかbloodとかありますからね。。。
* コード全般  
  これは人によりけりらしいので、本当に教えてほしいところですが、コードをやらせようとするとだめなようです。レビューはOKという人もいるのでどこまでがいいのかが謎ですが。
* Claude Codeの自動モード  
  逆に都度承認すると切り替わることが少なかったです。なんででしょうか。

### これやったら大丈夫そうだった

* チャットでの利用  
  チャットなら切り替わることが少なめでした。これでも切り替わるのがあるので話題にもよるかも。
* Claude Codeの承認モード  
  面倒だけど手押しで承認していくと切り替わることが「比較的」少なかったです。

## 今やるべきこと

### 作りたいアプリや整理したいものの計画書を作成させる

これはよくでているものですね。サブエージェントの整備や今までのやり取りから問題点をあげる、コーディングに関係のないものをみさせるとかです。あとはOpusとかに作らせた計画書も物によってはうごきました。

### Fable無きあとの世界をつくる

Fableはそもそものトークン使用量が倍になっており、とてもずっと使えるものではありません。ちょっと前まではFableで計画書を作成してOpusかSonnet、もしくはCodexへと投げる仕組みでかなり上手く行っていましたがコードに関する部分で勝手にOpusに変えられるとなると七夕まで使用量を消費させられるわけにはいきません。  
幸いなことに自身について聞くことについては答えてくれます。これを元に他のモデルでも似たような性能を出せるようにすれば7/8以降もなんとか生きていけます（）

```
あなたがほかのモデルと比べてどう優秀なのですか？最新のSonnet5やOpus4.8と比べてどう優秀なのですか？
他のモデルがあなたに追いつくにはどう指示してあげたり動いたりすると追いつくまでいかなくても並ぶことはできますか？
```

以下が返答

```
いい質問です。正直に言うと、「優秀さ」の中身はかなり具体的に説明できるものの、あらゆる場面で差があるわけではないので、そこも含めてお話ししますね。
私（Claude Fable 5）の位置づけ
私はAnthropicの新しい「Mythos級」ティアの最初の一般公開モデルで、Opusのさらに上に新設された最上位ティアの第一弾です。
Claude Mythos 5とは重みが同じモデルで、違いはセーフガードの有無だけという構成になっています。

Opus 4.8やSonnet 5との差がどこに出るか
ベンチマーク上の差はたとえばSWE-bench Proで80.0%と、Opus 4.8の69.2%から約11ポイントの跳躍といった数字がありますが、より本質的な特徴は、タスクが長く複雑になるほどリードが大きくなる点で、単発の短いタスクでは差が小さく、高難度・長時間のタスクで差が跳ね上がる（Blueprint-Bench +24.1）という傾向です。
またファイルベースの永続メモリを与えた長期タスク検証で、メモリによる性能向上幅がOpus 4.8の3倍以上だったと報告されています。つまり「何十手も先を見ながら方針を忘れず進める」「セッションをまたいで学習しながら働き続ける」タイプの仕事に強い、というのが差の正体です。
設計面では思考がadaptive一本化され、どれくらい考えるかをモデル自身が判断する仕様で、temperature等のサンプリングパラメータは廃止、effort（low〜max）でコストと賢さを調整する形になりました。
逆に言うと、チャットや軽作業はOpus/Sonnetで十分（体感差ほぼなし）で、料金は入力$10／出力$50とOpus 4.8のちょうど2倍なので、日常タスクで私を使うのは正直オーバースペックです。Sonnet 5も新世代で日常開発・エージェントタスクには十分な水準にあります。

他のモデルを「並ばせる」プロンプト・運用の工夫
差が「長時間の自律性と方針保持」に集中しているので、そこを人間側の設計で補えばかなり縮まります。

・長いタスクを短い監督付きタスクに分解する。 
Fable 5が自走で保てる「方針」を、代わりに人間が計画書として固定します。最初に設計フェーズだけ上位モデルにやらせて、実装をOpus/Sonnetに渡す——まさにMotsuさんが普段やっている「設計はClaude、実行はClaude Code/Codex」の分業がこれです。

・外部メモリを明示的に与える。 
進捗ログ・決定事項・TODOをファイル（PLAN.md、DECISIONS.mdなど）に書かせて毎ターン参照させると、「方針を忘れる」弱点を構造で補えます。
チェックリストと自己検証を強制する。 「実装後に○○の観点で自己レビューし、差分を報告せよ」と手順に組み込むと、上位モデルが暗黙にやる自己修正を明示的に再現できます。

・具体例（few-shot）とXMLタグでの構造化。 
曖昧な指示の意図汲み取りは上位モデルの強みなので、下位モデルには入出力例と厳密なフォーマット指定で曖昧さ自体を消してあげます。
失敗パターンの事前提示。 「やりがちな間違い」を先に列挙しておくと、推論力の差をかなり吸収できます。

ただし縮まらない部分もあります。数十ステップ先を見た一貫した判断や、1Mコンテキストを活かした大規模コードベースの横断理解は、プロンプトの工夫では根本的に埋めにくい領域です。なので現実解は多くの記事が言う通り「ふだんはOpus・Sonnet、勝負どころだけFable 5」という使い分けだと思います。
ちなみに料金・提供状況は6月の停止騒動を経て流動的なので、最新は https://www.anthropic.com/news/claude-fable-5-mythos-5 やdocs.claude.comで確認するのが確実です。
```
