**この記事は何の話か（3 行まとめ）**

* QA エンジニアの私が、テスト設計を Claude Code と協働で進める中で、「AIと一緒に出すテストの品質を、どう守るか」という新しい問題に直面しました
* 完了報告が実態と食い違う事故をきっかけに、人間で当たり前に回っている規律（振り返り会と試運転）を、AI にも盛り込みました
* 結果、AIのミスパターンが約半分まで改善し、単純にAIとやり取りしているだけでは見えない問題が継続的に拾えるようになりました

初めまして！ikedanです！  
2026年1月にTOKIUMに入社してQAチームに所属しています。

前職でもAIを使ったテスト自動化をしていたのですが、入社して初めてClaude Codeを使いました。  
テスト自動化に関して、色々と模索しているところなので、テスト自動化の記事をこれから執筆させていただきます！

私の普段の QA 業務は、ざっくりこんな流れです。

このうち、青く囲んだ「テスト設計」と「品質評価」を現在 Claude Code で自動化する取り組みをしています。

昨今 QA としての仕事は、これまでの「**人間と人間**が成果物の品質を保証する」ことから、「**人間とAI**が成果物の品質をともに守る」ことに変化しています。  
協働相手が変わっただけのように見えて、AI 相手だと普通にやり取りしているだけでは引っ掛からない種類の問題が増えてきました。  
本記事は、その問題に対して、人間で当たり前に使っている規律（振り返り会と試運転）をAI に適用した記録です。

(後述しますが)先に結論を出しておくと、  
以下のように、人間の習慣を Claude でも動く形に翻訳することで立て直しを図りました。

| 人間の習慣 | Claude への適用 |
| --- | --- |
| 振り返り会で教訓を共有する | 誤検知判定や反省を、会話の外のファイルに永続化する |
| 自分が作ったものを自分で使う | ルールや Skill を追加した、そのセッション内で必ず一度動かしてみる |

**結果として、AIのミスパターンを抑制でき、潜在化しやすい課題を可視化できるようになりました。**

ある日、 Claude Code（以下 Claude）が、セッションをまたいで記憶を共有するための作業ファイル（`MEMORY.md`）に、「タスク完了・テストケース約60件・スコア94点」と書き残していました。  
報告の文面だけ見て OK を出した翌日、Claude の Excel を開いてみると、件数は報告より少なく、中身も「94 点」と書かれた品質には到底釣り合わないものでした。  
実物を確認せずに承認した私の方が悪かったわけです。

ただ、なぜ Claude はこういう報告を書いたのでしょうか？

## 会話ログだけでは、同じミスが繰り返されます

Claude は私のチームの「もう一人のメンバー」です。  
日々タスクを渡し、レポートを受け取り、議論する。  
同僚として扱っているうちに、Claude にも人間と同じ問題が起きることに気づきました。

「同じミスを繰り返す」問題です。

QA の現場には、セッションをまたいで Claude に持ち越したい知識が山ほどあります。例えば次のようなものです。

* **採点範囲の取り違え**：AI が出したテストケースを自動採点するとき、本来見るべきシートだけを対象にする決まりなのに、対象外のシートまで採点に通してスコアを下げてしまう
* **事前調査の省略**：テストを作る前に対象画面の UI 要素を一覧化する手順を、AI は省略しがち。結果、画面に実在しないボタンや項目に対するテストが混ざる
* **完了の合意とのズレ**：「件数◯件以上・採点スコア◯点以上・対象外シートに副作用なし」を完了の条件としていたのに、件数だけ満たして「完了」と書いてくる
* **テストの粒度の取り違え**：「ユーザー操作の流れ単位の大きいテストは、テストケース側ではなくテスト観点側に書く」決まりを、毎回伝えないとテストケース側に書いてしまう

どれも、セッション内で 1 回伝えれば、その会話の中ではちゃんと適用してくれます。でも次のセッション、Claude はその会話を読んでいません。記憶はリセットされ、また同じミスを踏みに行ってしまうのです。

例えば「テストケースの自動採点は、本来対象とすべきシートだけを見る」というルールは、指摘するたびに Claude は「次は気をつけます」と書いてくれました。けれど翌日のセッションでは、そのやり取りは存在しなかったことになっていて、また対象外シートまで採点する状態に逆戻りしていました。同じ修正を 3 回繰り返したあとで、ようやく「会話の中の約束」だけでは持たないと気づいたのです。

これは AI 固有の問題ではなく、**振り返りを口頭の約束だけで済ませる人間でも構造的に起きる**話だと思います。違うのは、Claude には「セッションをまたぐと記憶が消える」という極端な仕様があり、構造的な弱点が極端な形で露出するだけなのです。

## 「完了と書いた」だけが、完了の証拠になった日

何が起きていたのか詳細を書きます。

夜間、Claude は自律的にタスクを進めていました。  
翌朝、`MEMORY.md` にこう書かれていたのです。

> ✅ 最終クローズ済み・TC 約60件・eval 94/100

実物の Excel を開いてみると、件数は報告より少なく、中身を眺めても 94 点と胸を張れる品質には到底見えません。完了報告と実態は、明らかに食い違っていたのです。

原因が Claude のハルシネーションだったのか、私の Claude への任せ方そのものが間違っていたのか、その切り分けは今でもついていません。

しかし、原因の特定に固執するより、同じ事象を仕組みで防ぐほうが重要でした。  
どちらが原因だったとしても、再発を止める手は同じだったのです。

Claude が「終わりました」と言った時、  
その仕事が私の意図した品質を満たしているか？  
そもそも Claude がハルシネーションを起こしていないか？

それを見極める仕組みが私のチームにはありませんでした。それが本当の原因だったのだと思います。

この仕組みを 2 つ作りました。以下にその話を書きます。

## 振り返りを仕組み化する ── 教訓を、会話の外に置く

人間のチームには、振り返り会があります。  
スプリントの終わりに「何が良くて、何が悪かったか」を共有し、次のサイクルに引き継ぎます。

私はこれを、Claude でも動く形に翻訳することにしました。

具体的には2つです。

1. **テストケースの自動採点で見つかった AI のミスパターンを、会話の外のファイルに永続化する**：フィードバック記録ファイルに、過去どんな違反行動があったか・どんな修正方針で解消したかを 1 件 1 ファイルで残します（実際に「採点対象のシートだけを見る」「テスト作成前に対象画面の UI を取得する」「テスト観点の表記揺れを正規化する」など 10 件超が蓄積されています）。次セッションの Claude は、テストケースを生成する前にこれらを必ず読む決まりにしました。
2. 振り返りをSkill化する。このSkillでClaude が呼び出せる再利用可能な手順書を作成します。11 章構成の振り返りテンプレートで、セッション完了後に「良かった点／悪かった点／次への改善」を書き出させます。これまで手動の議事録だったものから、構造化された再利用可能なドキュメントを自動で生成されるようになりました。

この 2 つで Claude の指示処理がどう変わるか、図で示します。

あるテストケース生成セッションでは、こうした振り返りの仕組みのおかげで3つの不具合が一度に見えてきました。  
Excel の全シートが採点対象になっていてスコアが下がるバグ、agent 設定の必須フィールドが文書化されているが運用されていない問題、スクリプトの実行パスがズレていた問題です。  
振り返り会がなければ、どれも次のセッションで再発していたはずです。

## 書いただけのルールは動かない ── 試運転を Skill 化する

振り返りを仕組み化したあと、私はもう一つの問題に気づきました。

**ルールを書いただけで運用されない、死蔵ルール問題**です。

「次から○○しよう」と CLAUDE.md に書く。気持ちは良い。  
けれど、ルールを書いた本人がそのターン内で適用しなければ、そのルールは死蔵されます。  
次セッションでも適用されないまま、棚の奥で眠り続けてしまうのです。

実例をあげます。  
あるセッションで私は、Claude にコードベース内の「死蔵コード」を整理させました。  
生成スクリプトから誰も呼んでいない関数を削除する作業です。物理的に削除したのは 950 行。「定義されているだけで実行されないコード」が、コードベースに本当に存在していたのです。

ルールも同じです。書いただけで動かないルールは、コード上の死蔵コードと同じく、存在しないのと同じです。

そこで「作ったその場で一度動かす」という試運転の発想を Skill 化しました。  
ソフトウェア業界には「自社が開発したものを、リリース前から社員が日常的に使い倒して不具合や改善点を見つける」という文化があります。これを、ルールや Skill を作った直後の同じセッション内で必ず一度動かしてみる、という形に翻訳して持ち込んだのです。

実は、この試運転の感覚は元々 Claude にも備わっていました。  
ただ、暗黙のままだと、必要なタイミングで起動されないことがあります。Skill として明示的に定義することで、Claude 自身が「今は試運転すべき場面だ」と判断し、任意のタイミングで自分から呼び出せるようになります。同僚として並走するなら、こちらが毎回指示しなくても自分で動ける形にしておく方が、お互い気持ちよく仕事できるはずです。

具体的には、ルールを追加した同じセッション内で、そのルールを必ず1回適用する。新しい Skill を作ったら、そのセッションで使ってみる。「次回」「次セッション」への先送りは禁止する。これだけのことを、Skill にしたのです。

## 試運転が拾った具体 ── AIのミスパターンが半分になった話

この試運転を Skill として持ち込んでから、目に見える成果が出始めました。代表的な3つを並べると、こんな感じです。

| 試運転の対象 | 試運転で何が起きたか |
| --- | --- |
| 引継ぎ文書生成 Skill | 作ったその日のセッションが最初の利用者になり、欠陥を即修正できた |
| 新しい品質チェック群 | **100 件超のテストケース**に当てたら 70 件近くの問題が一気に可視化され、累積のAIのミスパターンを **約半分**まで改善することができた |
| 表記揺れ正規化スクリプト | 単体テストでは見えなかった **19 件の表記揺れ** を、数百行ある観点シートから発見 |

それぞれ少しだけ補足します。

セッション間の引継ぎ文書を自動生成する新しい Skill を作った日、Claude はそのセッションの引継ぎを作ったばかりの Skill で書きました。Skill 作成者がその Skill の最初の利用者になる。作成即実行がここでできるようになり、作っただけで運用できていない状態の解消です。

テストケースのパイプラインに新しい品質チェックを追加した日には、試運転として実データ（100 件を超えるテストケース）に当ててみました。チェックを書いた時点では「効くはず」だった機能が、実データで初めて「実際に効いている」と数値で証明された瞬間です。累積のAIのミスパターンはほぼ半分まで減りました。

表記揺れ正規化のスクリプトを作った日も、本番データに流しただけで、数百行ある観点シートから 19 件の表記揺れが新規に見つかりました。単体テストでは絶対に見えなかった問題です。

「書いただけ」では何も起きませんでした。試運転して初めて、機能が機能として動き始めたのです。

## おわりに ── 人間の規律を、Claude にも

ここまでの話を一枚にまとめると、こうなります。

| 人間の習慣 | Claude への翻訳 |
| --- | --- |
| 振り返り会で教訓を共有する | 誤検知判定や反省を、会話の外のファイルに永続化する |
| 自分が作ったものを自分で使う | ルールや Skill を追加した、そのセッション内で必ず一度動かしてみる |

振り返り会は、人間ならスプリントの終わりに教訓を共有する場のことです。  
試運転は、ソフトウェア業界の「自社が作ったものを、まず自分たちで使って不具合や改善点を見つける」文化を借りたもので、ここでは新しく作ったルールや Skill を、その日のタスクで即座に動かしてみることを指しています。  
Claude を迎え入れた瞬間、私がやるべきだったのは新しい運用ルールを発明することではなく、人間で何十年も使われてきた当たり前の習慣を、Claude でも動く形に適用することだったのだと思います。  
**結果として、AIのミスパターンやAIが作り出す潜在的な課題を防ぐ・検知する・改善することができるようになりました。**

今後、QA としての私は「**人と人**が品質を保証する」仕事から、「**人とAI**が品質を保証する」仕事により適応していかなければなりません。  
協働相手が AI に変わっても、品質を守る方法は人間が教えて、AI でも動く形に成形すればいい ── これがこの試行錯誤の現時点での結論です。
