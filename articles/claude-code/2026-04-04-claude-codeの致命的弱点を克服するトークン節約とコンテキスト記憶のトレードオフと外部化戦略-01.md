## ⚖️ はじめに：AIコーディングにおける「節約」と「知能」のジレンマ

Claude Code は強力なCLIコーディングエージェントですが、そのAPI利用料（コスト）は開発者にとって悩みの種です。前回の記事では、コスト最適化のために頻繁に `/clear` を実行し、コンテキストを常にクリーン（ステートレス）に保つ重要性を解説しました。

しかし、この最適化術には一つだけ**致命的なトレードオフ**が存在します。それは「AIが過去の文脈やプロセス（学び）を忘却してしまう」ということです。

本記事では、コストを削ることによって失われる「記憶」の代償に焦点を当て、公式ドキュメントが推奨する記憶の外部化戦略と、それをインフラとして解決する2026年最新のOSSエコシステムについて解説します。

---

## 🧠 Claude Codeが持つ情報保持の仕組みとその限界

まず、前提として Claude Code がどのように情報を引き継いでいるのか、公式ドキュメントの仕様をおさらいしましょう。

### `CLAUDE.md` による静的記憶

プロジェクトのルートに置かれる `CLAUDE.md` は、Claude Code における「不変のシステムプロンプト」として機能します。コーディング規約やプロジェクトの全体アーキテクチャなど、**「絶対に忘れてはならない静的なルール」** を記述する場所であり、セッション開始時に自動的に注入されます。

### Auto Memory による動的記憶

Anthropicによると、Claude Code はバックグラウンドで環境制約や普段のビルドコマンド、ユーザーの好むコーディングスタイルを自動的に学習・保存する仕組み（Automated Memory）を備えています。明示的に指示しなくても、「あなた」の環境に合わせた些細な設定を自動で保持しようと試みます。

!

**⚠️ 記憶領域の情報量制限と「言語（トークン）の壁」**  
Auto Memoryや `CLAUDE.md` などの記憶メカニズムは「無限のストレージ」ではありません。これらはシステム上、**毎ターンのコンテキストウィンドウの先頭にテキストとして自動注入される**仕組みになっています。つまり、記憶が増えるほど、AIの思考枠（通常20万トークン）と毎回のコストを圧迫し続けます。

ここで非常に重要になるのが**記録する言語**です。Claudeのトークナイザーの構造上、日本語は英語に比べて同じ意味でも約2〜3倍のトークンを消費します。つまり、AIが **「日本語で記憶を保存（Auto Memory）したり書き出したり」** すると、英語で記録した場合に比べて**実際に保持できる事実（ファクト）の情報量が1/2〜1/3に激減してしまう**のです。「普段のやり取りは日本語でも、記憶（ドキュメントやMemory）の記録は英語にするよう定義する」という戦略は、AIの記憶容量を実質的に拡張する最強のハックとなります。

## ☠️ 最大の弱点：Context Rot（コンテキスト浸食）

これら標準の記憶機能が備わっているにもかかわらず、コスト節約のために `/clear`（セッションの切断）を多用すると、実プロジェクトでは以下のような深刻な問題が発生します。

* **「なぜ直したのか」という Why の喪失**：セッション内で踏んだエラーや、回り道をして選定したライブラリの**意思決定プロセス**が消滅します。
* **同じバグの再発**：別のセッションで機能を拡張しようとした際、AIが過去の経緯を知らないため、以前直したはずの全く同じボトルネック（バグ）を再び踏み抜きます。
* **前提の再入力コスト**：重要なコンテキストを一から説明し直すハメになり、結局膨大なトークンと時間を浪費します。

このように、古い意思決定の断絶によって推論精度が劣化していく現象を **Context Rot（コンテキスト浸食）** と呼びます。

---

## 🛡️ 公式推奨の解決策「記憶の隔離と外部化」

このトレードオフを克服するために、Anthropicの公式仕様を活かした2つのベストプラクティスが存在します。

### アプローチA：暗黙知のドキュメント外部化（Export）

AIの脳内（セッション履歴）にある「暗黙知」を、人間が意図的に静的ファイルへと「外部化」します。

```
# 作業終了時、/clear する前に必ず指示する
> 今日の作業で決定したデータモデルの変更理由と、踏んだエラーの回避策を `docs/decisions/001-user-model.md` に追記してから終了して。
```

知識をMarkdownに書き出させることで、次回のセッションでピンポイントにそのファイル（`@docs/decisions/...`）を参照させ、コストを抑えつつ文脈を復元させることができます。

### アプローチB：サブエージェント（Sub-agents）によるコンテキスト隔離

本流のタスク（例：複雑なReactコンポーネントの実装）を行っている最中に、「ついでにパッケージだけアップデートしたい」と思い立つことがあります。  
このとき、無関係な出力をメインセッションに入れてコンテキストを汚染したくないからといって `/clear` するのは悪手です。

```
# 隔離されたサブエージェントに雑用を任せる
> Can you create a subagent to update the testing libraries and run tests?
```

公式機能である「サブエージェント」を活用することで、メインセッションの貴重で高価な「文脈（コンテキスト）」を無傷のまま保護しつつ、別のタスクを並行・委譲することが可能です。

## 🌐 OSSを活用した「文脈記憶インフラ」のトレンド

手動でのドキュメント外部化や `CLAUDE.md` だけでは、開発チーム間で文脈を共有したり、何十ものセッションにまたがる長期間の記憶を維持するのには限界があります。  
そのため2026年現在は、**Claude Codeの外部に記憶専用のレイヤー（ラッパー）を構築するOSS**の導入がトレンドとなっています。

### 時間的記憶（セッション圧縮）系インフラ

* **[claude-mem](https://github.com/thedotmack/claude-mem)** などの自動キャプチャ系ツール。ツールの実行結果やAPIの返り値を裏側のデータベース（Chroma等）に自動記録し、AIが次期セッションで自己要約して読み込みます。
* **[engram](https://github.com/Gentleman-Programming/engram)** のようなSQLiteベースのローカル記憶ツール。手動で記憶をFTS（全文検索）データベースに放り込み、必要な時だけコンテキストへ引き戻します。

### プロセス規律強制系AIハーネス

* **[superpowers](https://github.com/obra/superpowers)** などのAIハーネス（ワークフロー制御ツール）。たとえば「セッション終了時に必ず `memory.json` にアクセスし、今回の学び（Learned）を構造化保存しなければ終了とみなさない」といった規律（ルール）をプロセスとしてAIに強制します（Iron Law）。

## 📝 まとめ

Claude Codeにおける**トークン節約（忘却）とコンテキスト保持（知能）は、本質的にトレードオフの関係**にあります。

* `/clear` でステートレスを保てばコストは下がるが、**Context Rot（文脈の喪失・バグの再発）** という大きな代償を払う。
* 代償を防ぐためには、公式推奨の**ドキュメント外部化**や**サブエージェント**を活用する。
* 本格的な長期開発においては、`claude-mem` や `superpowers` といった**記憶レイヤーOSS**の導入を検討する。

本記事で解説したアプローチを組み合わせることで、「破産しないコスト管理」と「AIの継続的な知能向上」を両立した、強靭な Context Engineering（文脈構築）を実現してください。

## 🐍 蛇足 (Appendix)：AIエージェント開発における「言語エコノミクス」とCJKモデルの逆襲

本記事を通じて「言語ごとのトークン消費効率の差」がコストと記憶容量に直結することを解説しました。ここでは少し視点を広げ、現在のAIエージェント開発の最前線における **「米国製AI vs 中国製AI」の言語（トークン）エコノミクス**について触れておきます。

現在、世界をリードする強力な生成AI・エージェント基盤を開発しているのは、実質的にアメリカ（OpenAI, Anthropic, Google等）と中国（DeepSeek, Alibaba/Qwen等）です。

### 🇺🇸 米国製AI（Claude, GPT等）の中核は「英語」

Claudeをはじめとする米国発のAIモデルのトークナイザー（文章をトークン単位に分割・圧縮する辞書）は、英語を第一言語として最適化されています。そのため、日本語のような言語を入力すると「1文字ごとに1〜2トークン」という非効率な分割が行われやすく、深刻なトークン枠の圧迫（トークン・インフレーション）を引き起こします。

### 🇨🇳 中国製AI（Qwen, DeepSeek等）と「日本語」の意外な親和性

一方で、近年猛威を振るっている中国製モデル（**Qwen** や **DeepSeek** ファミリ）は、CJK言語（Chinese, Japanese, Korean）のデータセットを重視して学習されており、独自のトークナイザーにも膨大な漢字の語彙が登録されています。

この **「CJKへの最適化」** により、中国製モデルの多くは、日本語の文章を解析した際の **Fertility（テキストをエンコードするのに必要な平均消費トークン数）が非常に低い** という決定的な優位性を持ちます。  
つまり、全く同じ日本語の指示書やログ（Memory）を読み込ませた場合でも、米国製モデルに比べて、**QwenやDeepSeekは極めて少ないトークン消費量（＝低コスト・省スペース）で処理できる**ケースが多いのです。

### メモリバックエンドとしての中国製AIの可能性

もしあなたが、膨大な日本語の社内ドキュメントやチャット履歴（文脈記憶）を保持する自律型AIプロセスを構築しようと考えた場合、言語の構造上、中国製AIの活用は圧倒的なコストメリットを生み出します。  
**「複雑な設計・推論のエースとしてClaudeを起用しつつ、過去の記憶の要約や大量の日本語ログの抽出などの『メモリ層』の処理には、日本語を安価に圧縮して読めるQwenやDeepSeekなどの中国製AIに委譲する」** というマルチモデル戦略は、極めて合理的です。

「日本語ユーザーにとって、根底のトークナイザー仕様上、中国製AIは非常にコスト親和性が高い（安価で大容量の日本語を保持できる）」という事実は、AIのコンテキスト空間を設計する上でぜひ知っておきたい知識です。
