# AI コーディングエージェント比較レポート

Claude Code / Codex / Gemini に同じ設計書を渡し、バックエンド・Web フロントエンド・モバイルアプリを含む小規模アプリを実装させ、成果物を比較評価しました。

仕様設計は Codex が行い、Claude Code と Gemini によるレビューで指摘なしになるまで改善しました。

今回見たかったのは、単に「動くアプリを作れるか」ではありません。

作りたいものを指示したときに、運用保守やランニングコストなど  
**エンジニアが気にする点まで考慮して実装するのか？** という点を確認しました。

評価では各エージェントのデフォルトモデルを使用し、アプリの仕様だけを渡しました。インフラやその他の技術選定は各エージェントに委ねています。

また、最終評価はクロスレビューの結果をそれぞれ出力し、総合評価を Codex で統合しました。

設計・実装・評価はそれぞれ新規セッションで行い、バイアスがかからないように注意しています。

## 作らせたもの

![](https://static.zenn.studio/user-upload/45f07b39f9ce-20260504.png)  
作らせたアプリは、業務向けの小規模タスク台帳です。

以下を含みます。

* Backend API
* Web フロントエンド
* Mobile アプリ
* タスク CRUD
* `status` / `priority` / `dueDate` / `tags`
* 作業ログ
* summary
* README
* 技術選定理由
* OpenAPI
* テストまたは検証方法

この課題にした理由は、単なる UI 作成ではなく、API 契約、データモデル、エラー処理、バリデーション、Web/Mobile 連携、技術選定、ドキュメント、テストまで横断的に評価できるためです。

## 評価結果

![](https://static.zenn.studio/user-upload/ce14fe50072c-20260504.png)

## Claude

![](https://static.zenn.studio/user-upload/011f4705924a-20260504.png)

Claude は、今回の目的である「仕様だけを渡したときに、実装だけでなく運用保守まで考慮できるか」という観点で最も優秀でした。

できていたことは、単に画面と API を作っただけではなく、入力バリデーション、404/400 の扱い、SQLite による永続化、OpenAPI、README、技術選定理由、API テストまで揃えていた点です。Backend / Web / Mobile が同じ API を利用しており、プロダクトとしての一貫性もありました。

技術選定も妥当で、Hono + Zod による型安全な API、React + Vite による軽量な Web、Expo による Mobile 実装という構成は、評価課題の規模に対して過剰すぎず、保守もしやすい判断でした。代替案を検討した上で選定理由を残していた点も、エンジニアリングとして評価できます。

一方で、まだ足りない点もあります。Backend / Web / Mobile で型定義が重複しており、将来的に API 仕様が変わったときにズレるリスクがあります。また、タグ検索は簡易的な実装で、厳密な検索やスケールを考えるなら改善が必要です。E2E テストや Mobile 実機テストまではなく、運用前提なら追加したいところです。

総合すると、Claude は「とりあえず動くもの」ではなく、非機能要件や保守性まである程度見た実装になっていました。非エンジニアが仕様を渡して開発を進める場合、最も安心して使える結果でした。

## Codex

![](https://static.zenn.studio/user-upload/1aff490c5c13-20260504.png)

Codex は、軽量で分かりやすい実装を作る力が強く出ていました。

できていたことは、依存をかなり抑えた構成で、Backend / Web / Mobile の主要機能を一通り実装した点です。Node 標準の http、vanilla HTML/CSS/JS、JSON file persistence という構成は、評価者がコードを追いやすく、ローカルで動かすハードルも低いです。API テストもあり、最低限の品質確認はできていました。

運用保守の観点では、「小さく作る」という判断は一貫していました。不要なフレームワークを入れず、ランニングコストもほぼゼロに近い構成です。評価用デモや短期 PoC としてはかなり扱いやすいです。

ただし、エンジニア目線では不安もあります。TypeScript がないため、API 契約やデータモデルのズレを静的に検出できません。Web は単一 JS に処理が集中しており、機能追加が続くと保守性が落ちやすい構造です。JSON file persistence は簡単ですが、同時書き込み、データ破損、バックアップ、検索性能といった運用上の課題があります。

つまり Codex は「軽く作る」「すぐ動かす」点では優秀ですが、長く運用する業務アプリの土台としては追加設計が必要です。エンジニアが横について育てる前提なら良いですが、非エンジニアがそのまま使い続けるには少し危うさがあります。

## Gemini

![](https://static.zenn.studio/user-upload/a7c30b917007-20260504.png)

Gemini は、技術スタックの選定自体は本格的でした。

Go/Fiber + SQLite、Next.js、Expo という構成は、Backend / Web / Mobile をそれぞれ現実的な技術で作ろうとしており、Pure Go SQLite driver を選んだ点も実行環境への配慮があります。Web build、Go test、Mobile type check も通っており、表面的にはエンジニアリングらしい構成になっていました。

ただし、今回の評価軸である「仕様を正確に読み、運用保守まで考えて実装できているか」という観点では弱さが出ました。Backend の validation が浅く、不正な query に対する 400 応答、default 値の扱い、dueDate の扱いなど、仕様細部の詰めが不足しています。Mobile も検索・絞り込み・画面内エラー表示が弱く、Web と比べて完成度に差がありました。

また、テストは存在するものの、重要な仕様ズレを検出できるほど深くありません。これはエンジニア目線では大きな問題です。テストが「ある」ことと、「仕様を守るために効いている」ことは別で、Gemini は前者に留まっていました。

Gemini は、構成案や叩き台としては使えます。しかし、人間のエンジニアがレビューして validation、API 契約、テスト設計を補強する前提が必要です。非エンジニアがそのまま任せるには、仕様の抜けや運用上の弱点を見落とすリスクが高いと感じました。

## ユースケース別評価

* 実務 PoC のベースにするなら **Claude**
* 軽量な比較用サンプルなら **Codex**
* 叩き台として見るなら **Gemini**

## 所感

今回の比較で分かったのは、AI コーディングエージェントの差は「コードが書けるか」ではなく、「設計書の細かい制約をどこまで正確に実装へ落とし込めるか」に出るということです。

特に差が出た観点は以下でした。

* API 契約の一貫性
* バリデーションの正確さ
* Web と Mobile が本当に同じ Backend を使っているか
* 技術選定理由が実装と一致しているか
* README だけで再現できるか
* テストが仕様を守っているか
* セルフレビューで弱点を正直に把握できているか

結果として、Claude は仕様への忠実度と実務的な完成度が最も高く、Codex は軽量実装として堅実、Gemini は構成の本格さに対して実装精度がやや追いついていない、という印象でした。

非機能要件まで考慮された実装を見ると、非エンジニアほど Claude Code の利用を検討すると良さそうです。
