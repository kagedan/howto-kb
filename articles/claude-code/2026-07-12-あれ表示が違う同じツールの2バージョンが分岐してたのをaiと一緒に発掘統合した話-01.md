## はじめに

前回、Tailscaleネットワークを監視するTUIツール「[Tail-Pulse](https://github.com/kuroiko0429/tail-pulse)」を作った記事を書いた。

<https://zenn.dev/kuroiko0429/articles/tailscale-tui>

その続き。GitHubに公開してGoReleaser的な体裁を整えて`go install`できるようにしよう、というだけの作業のはずが、途中で「同じツールの2つのバージョンが別々のディレクトリでずっと分岐してた」ことが発覚して、地味に大掛かりな統合作業になった話。

## 発端: リリース作業のつもりが

Tail-PulseをGitHubで公開実績にできるレベルまで整えようと思い立ち、Claude Codeと一緒に着手した。

1. リポジトリ名を`Tail-Pulse` → `tail-pulse`に統一
2. `go.mod`のモジュール名が`main.go`という事故(`go mod init main.go`のやりがちなやつ)になっていたので修正
3. パッケージ分割したリファクタ版のコードをコミット
4. `v0.1.0`としてタグ付け・バイナリビルド・GitHub Release作成

ここまでは順調だった。ビルドも通るし、READMEも整えたし、v0.1.0を出して満足していた。

## 「あれ、表示が違う…」

ところがしばらくして、自分が実際に使っているツールと見比べてみたら、様子がおかしいことに気づいた。

* 自分が普段使っているのは `~/.local/bin/tailpuls`
* 今回リリースした`tail-pulse`は見た目もキーバインドも別物

タイトルバーの文字列からして違う。

```
# 普段使ってる方
CTOS // TAILNET_MONITOR // v4.0.0

# 今回リリースした方
CTOS // TAILNET_MONITOR_ADVANCED // v2.0.0
```

さらに調べると、`Enter`キーを押したときの挙動も違った。普段使っている方は詳細画面が開いて、そこから`s`でSSH・`w`でWake-on-LAN・`g`でファイル受信、というふうにアクションを選べる。リリースした方は`Enter`を押すといきなりSSH接続が始まる。

## 原因調査: プロジェクトが2つに分岐していた

GitHubのコミット履歴とローカルのソースを突き合わせて分かったのは、次のような状態だった。

* `~/Documents/tail-pluse/`（ディレクトリ名がタイポってる）: パッケージ分割されたリファクタ版。gitの最初のコミットから1つも進んでいなかった
* `~/Documents/go-lang/tailpuls/`（これも名前がちょっと違う）: 機能てんこ盛りのフラット構成版。実際に`~/.local/bin/tailpuls`としてビルドして日常使いしていたのはこっち

つまり、リファクタ作業を始めた時点のコードをベースに機能追加を続けた「本命」と、パッケージ構成に整理し直した「整理版」が、**どちらもGitHubにpushされないまま別々のディレクトリで生き続けていた**。

git logで確認すると、GitHub上の最初のコミットはどちらの系譜にも共通する1個だけで、そこから先はローカルで2方向に分かれたまま誰にも気づかれていなかった。

同じコミットから2方向に分かれたきり、どちらもリモートに戻ってこないまま育っていた。

## 機能の突き合わせ・統合

パッケージ分割版（見た目がきれい、ポートスキャン機能あり）をベースに、フラット版にしかない機能を移植していくことにした。

洗い出した差分:

* Exit Nodes / Serve / Logs / Daemon タブ
* Wake-on-LAN（ローカル送信 + SSH経由のプロキシ送信）
* Taildropでのファイル送受信
* ノードのオンライン/オフライン切り替わりのデスクトップ通知
* `journalctl -u tailscaled -f`のライブログストリーミング
* SSHポート自動検出（`ssh -G`）
* xclipクリップボードフォールバック

一通り移植してビルドが通った時点で満足しかけたが、今度は**Enterキーの挙動そのものが違う**ことに気づいた。パッケージ分割版は`d`キーでインライン詳細パネルを開く方式、フラット版（＝普段使っている方）は`Enter`でフルスクリーンの詳細画面に切り替わってそこから`s`/`f`/`g`/`a`/`w`のアクションを選ぶ方式。機能は移植できていても、操作感が別物のままだった。

これはスクリーンショットを3枚並べて見比べて初めて気づけた。

```
# 移植前(パッケージ分割版)
[Enter] → 即SSH接続 + ハックアニメーション演出

# 普段使っている方
[Enter] → 詳細画面(DNSName / Key Expiry / 帯域 / WoL MAC / タグ)
        → [s]SSH [f]送信 [g]受信 [a]ルート許可 [w]WoL [Esc]戻る
```

操作フローそのものを普段使っている方に合わせて作り直した。

## さらに出てきた地味な差分

「他に旧版にあって新版にないものは？」と聞き直して、ソースを1行ずつ突き合わせたら、まだいくつか漏れていた。

* `PgUp` / `PgDn`でカーソルを10行ジャンプ
* `o`キーでオンラインのみ表示に絞り込み
* `r`キーで手動リフレッシュ
* PING列のDERPリレー / Direct接続経路の表示（`tailscale ping`の出力をパースして判定していた）

最後のDERP経路表示は、OSの`ping`コマンドではなく`tailscale ping`を使わないと出てこない情報だったので、レイテンシ計測のコマンド自体を差し替える必要があった。

## 実害のあったバグも1個見つかった

機能の突き合わせ中に、実際に動作を壊すバグも見つかった。

```
// 修正前: SSHターゲットの文字列をまるごと1つの引数として渡してしまう
cmd := exec.Command("ssh", target) // target = "100.x.x.x -p 2222"
```

`target`にカスタムポート付きの文字列（`"100.x.x.x -p 2222"`）が入っている場合、これを1つの引数としてsshに渡すと、sshは`"100.x.x.x -p 2222"`という空白入りの1つのホスト名として解釈しようとして接続に失敗する。

```
// 修正後: 空白で分割して複数の引数として渡す
cmd := exec.Command("ssh", strings.Fields(target)...)
```

カスタムSSHポートを設定していたノードへの接続が、リファクタの過程でずっと壊れたままになっていた。README上は「SSHポート自動検出対応」と胸を張っていたのに、実際には特定条件で動いていなかったということになる。

## 教訓: ファイル管理はちゃんとしよう

一連の作業を振り返って一番の反省点は、技術的なことより単純なファイル管理の話だった。

* `tail-pluse`と`tailpuls`という似た名前のディレクトリを放置していた
* どちらもgit管理されていなかった、あるいは`.git`だけ壊れて履歴が追えない状態だった
* 「リファクタ版」と「本番で使ってる版」が別ディレクトリに分かれていることに、しばらく気づかなかった

似たようなディレクトリを複数の場所に作ってしまったら、

1. 名前を紛らわしくしない（タイポ違いの名前は絶対避ける）
2. 作業を始めたらすぐ`git init`してGitHubにpushする
3. 「どっちが最新か」を後から調べなくて済むように、こまめにコミットする

という当たり前のことを当たり前にやるだけで、今回のような「2つのバージョンが誰にも気づかれず分岐する」事故は防げたはず。AIと一緒に作業していると「とりあえず動くコードを量産する」スピードが上がる分、こういう地味な整理をサボるとすぐにツケが回ってくる。

## 今は`go install`で使える

一通りの統合が終わって、GitHub Releaseとして`v0.4.0`まで出した。

```
go install github.com/kuroiko0429/tail-pulse@latest
```

または[Releases](https://github.com/kuroiko0429/tail-pulse/releases)ページからビルド済みバイナリをダウンロードしてもいい。

設定ファイル（`~/.config/tail-pulse/config.yaml`）はGruvboxテーマがデフォルトで、Wake-on-LAN用のMACアドレスやSSHポートのオーバーライドもここで設定できる。

## おわりに

技術的には地味な作業の積み重ねだったけど、「AIと一緒に作業してるとレビューが甘くなる」という自戒を込めて書いた。実際にスクリーンショットを並べて見比べる、旧バージョンのソースを1行ずつ読み直す、という手を動かす確認をやらないと、こういう分岐や退行バグには気づけない。
