「Claude Code、使い放題終わるの?」という声が流れてきて、一瞬どきっとしました。6月15日から課金が変わる、という話です。

先に言ってしまうと、ターミナルで対話的に Claude Code を使うのが中心なら、個人開発者の大半は影響を受けません。効くのは、自動でClaudeを呼んでいる人です。この記事は、自分がどちら側かを切り分けて、6月15日までに何を確認すればいいかをチェックリストにしたものです。

:::note info
立ち位置です。一次情報は Anthropic の公式ヘルプ。発表は2026年5月13〜14日（@ClaudeDevs）、適用は6月15日。金額や対象は The New Stack・InfoWorld・Zed の解説でも同内容を確認しました。数字や仕様は揺れるので、最新は公式ヘルプと自分の請求画面で確認してください。出典は末尾です。
:::

## 何が変わるか

Claudeの使い方が、対話とプログラムの2つに分けて課金されます。

対話で使うぶんは、これまでどおりです。

- Claude.ai のチャット
- ターミナルで直接動かす Claude Code
- Claude Cowork

別建ての月次クレジットに移るのは、プログラムから自動で呼ぶぶんです。

- Claude Agent SDK
- `claude -p`（画面を介さないヘッドレス実行）
- Claude Code の GitHub Actions
- ACP 経由でつながる第三者エージェント（Zed など）

値上げというより、自動化のぶんだけ別会計にする整理です。サブスクの上限そのものは変わりません。

## まず、自分は影響を受けるか

ここを先に切り分けます。

影響を受けない側。

- ターミナルで Claude Code を手で動かしている
- Claude.ai でチャットしている
- Cowork を使っている

要するに、自分が画面の前にいて対話している使い方は、何も変わりません。

影響を受ける側。

- CI のパイプラインで Claude を呼んでいる
- cron などの定期実行で `claude -p` を回している
- GitHub Actions に Claude Code を組み込んでいる
- Zed のような第三者ツールから ACP 経由で使っている

自分が見ていないところでスクリプトやジョブがClaudeを叩く使い方が、別建てに移ります。

調べるなら、リポジトリやCI設定、cron を横断して grep するのが早いです。

```bash
# 自動でClaudeを呼ぶ経路がないか、横断して探す
grep -rn -E "claude -p|claude_agent_sdk|anthropic" \
  ~/projects .github/workflows ~/.config 2>/dev/null

# cron に登録がないかも確認
crontab -l 2>/dev/null | grep -i claude
```

ヒットしなければ、自分は対話だけの側だとほぼ確信が持てます。ヒットしたら、それが今回別会計に移る経路です。

:::note warn
別建てクレジットの中身です。プランごとに月次で Pro $20 / Max 5x $100 / Max 20x $200。消費はAPIの通常レート、繰り越しなし、ユーザー単位、毎月リセット。使い切ると標準では自動化が止まります。止めたくなければ、超過をAPIレートで課金する設定（公式名「usage credits」）を有効にしておく。止まるか、課金して続けるかを自分で選ぶ形です。標準のEnterprise席にはクレジットが付かない、ともされています。
:::

## 6/15までにやること

影響を受ける側だった場合に、やっておくことです。

1. 直近30日のプログラム利用を棚卸しする。`claude -p`・Actions・定期実行・第三者ツールが今いくつ動いて、どれくらいClaudeを叩いているか把握する
2. クレジットを受け取る。Anthropic から受け取り用メールが届くとされるので、6月15日前に一度受け取っておく
3. 超過設定を決める。止めたくない自動化は「usage credits」を有効に。止まってよい用途は入れずにクレジット内に収める
4. 恒常的にクレジットを超える自動化は、直接のAPIキー（従量課金）へ寄せる。毎月いくらか読めて、追跡もきれいになる
5. プロンプトキャッシュを使う。キャッシュが効くと入力ぶんが大きく安くなる（おおむね1割の単価）ので、繰り返し系の自動化はクレジットの持ちが変わる

## サブスククレジットか、直接APIか

寄せ先は、自分の自動化の量で決まります。

- 軽い自動化（月に数回程度）なら、付いてくるクレジットで足りることが多い。APIキーを別管理する手間のほうが重いくらい
- 重い自動化（毎日動く、量が安定して多い）なら、直接API従量のほうが読めて安い。公式も、共有の本番自動化はこちらを勧めている

サブスクのクレジットは、超えたら止まるか超過課金になるかのどちらか。量が読めているなら、最初から従量にしておくほうが心臓に優しいです。

:::note info
なぜこうなったか。対話する人間は1日に数十回プロンプトを送る程度ですが、自律エージェントは数千回のリクエストを生成し、テストを連続で回し、自分を再帰的に呼ぶこともあります。Zed の見積もりでは、サブスクはエージェント利用をAPI価格の15〜30倍ぶん補助していたとのこと。その差を別会計で埋める動きです。2026年に入って、1月のOAuthブロック（数日で撤回）、2月の規約での制限、4月の追加制限に続く、3度目の調整でもあります。
:::

## 次の自分に渡すメモ

腑に落ちたのは、これが値上げではなく線引きの話だ、ということでした。対話という「自分が見ている使い方」はサブスクのまま、自動化という「見ていない間に流れる使い方」は別会計へ。AIの道具が、お得なサブスクから、コストとして管理するインフラへ片足を移した瞬間だと思います。

まずやることは、慌てて乗り換えるより先に、自分がどちら側かを見ること。対話だけなら何も変わらない。自動化があるなら量を棚卸しして、クレジットで足りるか直接APIに寄せるかを決める。トークン代はどこかの財布に着地する、という話の続きでした。自分も small な自動化を持っているので、6月15日の前に grep で棚卸ししておきます。
