この記事は、Claude Codeをある程度使っているが、まだ次のような状態にある人に向けています。

* 作りたいものを自然言語で依頼し、出てきたものを修正して使っている
* 要件定義や設計を飛ばして、いきなり実装依頼をしてしまう
* manus、Genspark、ChatGPTなどに任せている業務を、もっと自分仕様に寄せたい
* GAS、リサーチ、営業追客、資料作成、業務改善ツールづくりをClaude Codeで回したい
* [CLAUDE.md](http://CLAUDE.md)、Skills、MCP、Git、APIの役割がまだ曖昧
* 外部記事で紹介されている設定やSkillsを安全に取り入れてよいか不安

**結論から言うと、Claude Codeを本格的に使う鍵は「プロンプト力」ではありません。**

大事なのは、Claude Codeが働ける環境を作ることです。

## Claude Codeはチャットではなく作業環境である

Claude Codeは、単に質問に答えるチャットではありません。

ローカルのファイルを読み、検索し、編集し、コマンドを実行し、テストし、Gitの状態を見ながら作業できます。

つまり、人間が毎回説明しなくても、正本ファイル、検証方法、履歴、ルールがそろっていれば、自律的に仕事を進めやすくなります。

逆に言えば、次の状態では本来の力が出ません。

* **作業ファイルが散らばっている**
* **仕様がチャット内だけにある**
* **何をもって完了かが曖昧**
* **テストや確認方法がない**
* **失敗しても戻せる履歴がない**
* **毎回同じ前提を手入力している**

この状態で「いい感じに作って」と頼むと、Claude Codeは推測で動くしかありません。

## 最初に整えるべき5つの武器

Claude Code運用で最初に理解すべき武器は5つです。

![](https://assets.st-note.com/img/1776903868-1BtSI8lHrEYfGWOnViJzs7pX.png?width=1200)

順番も重要です。

まずフォルダ構造とCLAUDE.mdを整える。次にGitで戻れる状態にする。繰り返す業務が見えてからSkills化する。外部サービスを直接扱う必要が出てからMCPを接続する。

最初からSkillsやMCPを大量に入れる必要はありません。

## CLAUDE.mdに書くべきこと

![](https://assets.st-note.com/img/1776903882-53uH2PkFJhxgAdXoaMnyKOG1.png?width=1200)

CLAUDE.mdは、Claude Codeが毎回読むプロジェクトの前提です。

書くべきなのは、Claudeがコードを読んでも推測しにくい事実です。

例:

* **このプロジェクトの目的**
* **よく使うコマンド**
* **テスト方法**
* **保存先のルール**
* **命名規則**
* **扱ってはいけないファイル**
* **APIキーや個人情報の扱い**
* **完了時に必ず確認すること**

逆に、長いマニュアル、詳細なAPI仕様、毎回は使わない業務手順はCLAUDE.mdに入れすぎない方がよいです。

CLAUDE.mdは毎回読み込まれるため、長すぎると重要な指示が埋もれます。

Claude Code公式ドキュメントでも、CLAUDE.mdは短く具体的に書くことが推奨されています。

目安としては、1ファイル200行以内に収めます。

## Skillsは「何度も貼っているプロンプト」を保存する場所

![](https://assets.st-note.com/img/1776903926-un9R2B1wXLeK3plEd6M0Ujot.png?width=1200)

Skillsは、Claude Codeに追加できる再利用可能な手順書です。

たとえば、毎回次のような依頼をしているならSkills化の候補です。

* **営業追客リストを作る**
* **リサーチ記事を作る**
* **GASを修正してテストする**
* **スライド構成を作る**
* **議事録からタスク一覧を作る**
* **セキュリティ観点でレビューする**

Skillsに向いているのは、次の条件を満たす業務です。

* **手順がある**
* **入力と出力がある**
* **毎回同じ注意点がある**
* **期待する成果物の型がある**
* **検証方法がある**

個人で使うSkillsは **`~/.claude/skills/<skill-name>/SKILL.md`** に置きます。

プロジェクト内で共有するSkillsは **`.claude/skills/<skill-name>/SKILL.md`** に置きます。

## MCPは「外部ツールを直接触らせたい時」だけ使う

![](https://assets.st-note.com/img/1776904178-2DAqwlbmaV4CLB7jHiF1oE5G.png?width=1200)

MCPは、Claude Codeを外部ツールやデータソースにつなぐ仕組みです。

たとえば、次のようなことをしたい場合に使います。

* **Gmailから未返信メールを取得する**
* **Slackの未返信スレッドを調べる**
* **Google Drive上の資料を読む**
* **NotionやJiraのタスクを参照する**
* **DBからデータを取得する**
* **Figmaのデザイン情報を読む**

ただし、MCPは便利な反面、権限を広げる仕組みでもあります。

Claude Code公式ドキュメントでは、AnthropicがすべてのMCPサーバーを管理・監査しているわけではないと明記されています。

したがって、MCPは「**必要になった時に、信頼できるものだけを入れる**」が基本です。

## GitはClaude Code利用者ほど必要になる

![](https://assets.st-note.com/img/1776904197-Y0OTuXK9Ntqo6ZisP8DcWLQI.png?width=1200)

非エンジニアほど、Gitを避けがちです。

しかしClaude Codeを使うなら、Gitは必須に近いです。

理由は単純で、Claude Codeは実際にファイルを変更できるからです。

壊しても戻せる履歴があるだけで、任せられる範囲が広がります。

最低限の運用はこれで十分です。

```
git init
git add .
git commit -m "initial commit"
```

作業前にcommitする。変更後にdiffを見る。完成したらcommitする。

これだけで、Claude Codeに「今の変更を確認して」「前の状態との差分を説明して」「この変更だけcommitして」と依頼できるようになります。

Claude Codeにはcheckpoint機能もありますが、これはGitの代替ではありません。プロジェクトの履歴管理はGitで行うべきです。

## 営業追客を自動化するなら、まずCSVで十分

![](https://assets.st-note.com/img/1776904229-gh3qrWMxc9yTlQ7ufVDi4SzR.png?width=1200)

営業で失注や解約につながりやすいのは、能力不足よりも「次アクションの管理漏れ」です。

最初からCRMやMCPを複雑につなぐ必要はありません。

まずはCSVまたはGoogle Sheetsに、次の項目を持たせます。

![](https://assets.st-note.com/img/1776904405-G9ixwOlhNRXqCjHcL2dsEgAM.png?width=1200)

Claude Codeには、毎朝このデータを読み、次の条件に当てはまる顧客を抽出させます。

* **`next\_action\_at` が今日以前**
* **`status` が `open`**
* **`next\_action` が空ではない**
* **最終接触日から一定日数が経過している**

最初の自動化は「送信」ではなく「下書き」と「通知」に留めるのが安全です。

メールの完全自動送信は誤送信リスクがあります。最初はClaude Codeに追客文面を作らせ、人間が確認して送る形が現実的です。

## GASは「生成」より「検証ループ」を自動化する

![](https://assets.st-note.com/img/1776904430-OJGUt8NClb1BkHLXhpi3IVqm.png?width=1200)

GASで一番時間が溶けるのは、コードを書く時間ではありません。

生成されたGASを貼り付け、スプレッドシートで手動テストし、エラーを見て、また修正するループです。

Claude Codeに移すべきなのは、この検証ループです。

基本構成は次の通りです。

1. **clasp（Googleが提供するCLI）でApps Scriptをローカル管理する**
2. **テスト用Spreadsheetを用意する**
3. **テストデータを投入する**
4. **GASをデプロイまたは実行する**
5. **Spreadsheetの結果を読み取る**
6. **期待値と比較する**
7. **失敗したらClaude Codeが修正する**

注意点として、Google Apps Scriptのcontainer-bound scriptは親ファイルに紐づきますが、Web AppやApps Script API経由で実行する場合、`getActiveSpreadsheet()` のようなアクティブなコンテナ前提のメソッドは使えません。

API経由やWeb App経由で実行するなら、`SpreadsheetApp.openById()` のようにID指定で開く設計にしておく方が安全です。

## 外部記事のSkillsを使う時のチェックリスト

![](https://assets.st-note.com/img/1776904461-jDTLO9uGUQCqBPzca13J5hkv.png?width=1200)

Claude Code周りでは、便利なSkillsや設定例がX、Zenn、GitHubで多く共有されています。

使う前に見るべきポイントは決まっています。

特に注意すべきなのは、hooks、permissions、MCPです。

hooksは任意のコマンド実行につながります。permissionsを広げすぎると、確認なしで実行できる範囲が増えます。

MCPは外部サービスへのアクセス権限を持ちます。

外部からコピーした設定をそのまま流し込むのではなく、1行ずつ読むべきです。

## APIキーは「渡し方」より「置き場所」が重要

![](https://assets.st-note.com/img/1776904474-sQWRNyPqg62ioMeKkX8IDhmS.png?width=1200)

APIキーは、Claude Codeに直接貼り付けるものではありません。

基本は次のどれかです。

* **ローカル開発: `.env`**
* **Mac: Keychain**
* **GitHub Actions: GitHub Secrets**
* **Vercel: Environment Variables**
* **AWS/GCP/Azure: 各クラウドのSecret Manager**

`.env` を使う場合は、必ず `.gitignore` に入れます。

また、Claude Codeの認証とClaude APIの課金は別物です。

Claude CodeをPro/Max契約枠で使いたい場合、契約アカウントでログインします。

環境変数 `ANTHROPIC\_API\_KEY` が設定されていると、Claude CodeがAPIキーを優先し、サブスク枠ではなくAPI課金を使う場合があります。

契約枠で使えているかは `/status` で確認します。

## 勤怠・シフト作成は生成AIだけで解かない

現場のシフト作成は、生成AIに「いい感じに組んで」と頼むだけでは安定しません。

これは制約充足、または制約最適化の問題です。

必要なのは、まず条件を分けることです。

必ず守る条件:

* **必要人数**
* **夜勤に入れる資格**
* **1日の勤務上限**
* **連勤上限**
* **夜勤明けの扱い**
* **希望休**
* **法定ルール**

できれば満たしたい条件:

* **公平な夜勤回数**
* **希望勤務の反映**
* **ベテランと新人の組み合わせ**
* **休日数の公平性**
* **前月からの偏り補正**

この種の問題には、[Google OR-Tools](https://developers.google.com/optimization?hl=ja)のような制約最適化ソルバーが使えます。

Claude Codeの役割は、制約を整理し、CSV/Excel入出力を作り、ソルバーを組み込み、結果を人間が確認しやすい画面や表にすることです。

生成AIに最終決定を丸投げするのではなく、制約をコード化し、検証可能な形にするのが現実的です。

## まとめ

Claude Code活用で最初にやるべきことは、すごいプロンプトを覚えることではありません。

仕事を次の形に変えることです。

* **正本ファイルがある**
* **仕様がSPEC.mdにある**
* **ルールがCLAUDE.mdにある**
* **手順がSkillsにある**
* **外部接続はMCPで管理する**
* **履歴はGitに残す**
* **検証方法がある**
* **APIキーを安全に扱う**

この状態を作ると、Claude Codeに任せられる仕事の範囲が一気に広がります。

Claude Codeに依存するとは、AIに丸投げすることではありません。

AIが安全に働ける作業環境を整え、検証可能な単位で任せることです。
