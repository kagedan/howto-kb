## 目的は決まった。さぁ、AIを使いたい

前回の記事では、「セカンドブレインを作る前に、まず自分のファースト・プリンシプル（第一原理）を言語化することが重要です」と書きました。

つまり、

* 自分は何を成し遂げたいのか
* なぜそれをやるのか
* 何に価値を感じるのか
* 何を絶対に譲れないのか

といった、自分自身の軸を明確にすることが先だということです。では、**その目的が定まったら次に何をすれば**よいのでしょうか。

当然、「AIを使ってみたい」という気持ちになります。

* チーム全体の生産性を高めたい
* 日報や提案書の作成を効率化したい
* 社内の知識を蓄積したい
* 属人的なノウハウを会社の資産にしたい

こうした目的が明確になると、AIは一気に現実的なツールとして見えてきます。しかし、すぐにAIを導入するには、**情報が多すぎる**という問題があります。

[Claude Code](https://www.anthropic.com/claude-code?utm_source=chatgpt.com)、[Cursor](https://www.cursor.com?utm_source=chatgpt.com)、[GitHub Copilot](https://github.com/features/copilot?utm_source=chatgpt.com)、AIエージェント、MCP、RAG、メモリ、スキル、コンテキスト設計……。

Xやブログを見ていると、どれも魅力的に見えます。そして気づけば、

「結局、うちの会社は何から始めればいいのだろう？」

という状態になります。これは、ごく自然なことです。

目的が明確になったとしても、どのツールを、どの順番で導入するかを間違えると、かえって現場が混乱することがあります。

特に、2〜10人規模の中小企業では、

* ITスキルにばらつきがある
* 専任のエンジニアがいない
* 日々の業務に追われている
* 複雑な仕組みを運用する余裕がない

という現実があります。だからこそ重要なのは、

**「最も高度なツールを選ぶこと」**ではなく、**「全員が使い始められる入口を選ぶこと」**です。

今回の記事では、私自身が「Claude Codeから始めるべきだ」と思い込んで遠回りした末に、実はもっと先に試すべきものがあったと気づいた経験について書いてみたいと思います。

## Claude Codeより先に試すこと

私が最初に考えていたのは、AIを活用しながら、使うたびに知識やノウハウが蓄積されていく**「資産化の仕組み」**を作ることでした。

いまや、業務でAIを使うこと自体は珍しくありません。しかし、「その場で便利に使う」ことと、「使うたびに会社の資産が増えていく」ことの間には、大きな違いがあります。

私自身、[Git](https://git-scm.com?utm_source=chatgpt.com) / [GitHub](https://github.com?utm_source=chatgpt.com) や [Claude Code](https://www.anthropic.com/claude-code?utm_source=chatgpt.com) を使いながら、AIと一緒にファイルを編集し、知識を整理し、資産化の仕組みづくりを進めていました。

その過程で、生産性は確かに向上し、AIエージェントへの理解も深まりました。しかし、ある根本的な疑問にぶつかりました。

**「この仕組みを、社員全員にそのまま使ってもらえるだろうか？」**

私の会社で社員がAIにお願いしたいのは、プログラミングではありません。日報を書いたり、SNS投稿を作ったり、提案書をまとめたりといった日常業務です。

そうした業務に対して、

**「まずターミナルを開いてください」**

とは、とても言えません。

Claude Codeは非常に強力なツールです。しかし、それは主に開発者や技術者のためのものです。チーム全体にAIを広めたいのであれば、最初に必要なのは最も高機能なツールではありません。

Claude Code に関する議論の多くは、エンジニア視点で行われています。

* スキル管理
* コンテキスト設計
* CLAUDE.md の粒度
* 開発ワークフローとの統合

社長、こういうのは、会社が儲かってからやりましょう。

## claude.aiのプロジェクト機能

オススメを1つ紹介します。それは [claude.ai](https://claude.ai?utm_source=chatgpt.com) の「プロジェクト」機能です。

![](https://assets.st-note.com/img/1778978524-ZlscNtTzQUrg0OwFLqGue8M7.png?width=1200)

画面はproプランを契約している状態です

これは、プロジェクトごとに、

を設定でき、チャットを重ねるほど、そのプロジェクト専用のナレッジが蓄積されていく、というものです。

しかも、

* ブラウザで使える
* スマートフォンでも使える
* ターミナルは不要

という手軽さがあります。

たとえば、

* アプリ開発の相談をするプロジェクト
* 日報や週報を書くためのプロジェクト
* SNS投稿を量産するプロジェクト
* 海外営業の提案書を作るプロジェクト

といったように、目的ごとにプロジェクトを分けることができます。

それぞれのプロジェクトで文脈が分離されるため、Claudeは毎回同じ前提条件のもとで動いてくれます。社員に Claude Code を覚えてもらう必要はありません。チャットができれば、それだけで十分です。

たったそれだけで、単なる「質問して終わり」のAI活用から一歩進んだ、継続的に知識が蓄積される運用が可能になります。

## ナレッジ管理という本当の課題

ただし、ひとつ正直にお伝えしたいことがあります。

claude.ai のプロジェクトには、プロジェクト間でナレッジを自動共有する仕組みがありません。

つまり、

* プロジェクトAで育てた知見
* プロジェクトBで得た改善案

は、そのまま他のプロジェクトに引き継がれるわけではありません。

「これはチーム全体で使える知見だ」と気づいたときには、手動で他のプロジェクトに反映する必要があります。これは制約でもありますが、考え方を変えると、むしろ整理しやすくなります。

私は、ナレッジには2種類あると考えています。

### 1. プロジェクト固有のナレッジ

そのプロジェクトだけで必要となる情報です。

### 2. チーム全体で共有すべきナレッジ

文章の書き方、提案の型、会社の価値観など、どのプロジェクトでも使える知見です。

前者は**各プロジェクトの中**で育てます。後者は**「手順」としてテンプレート化し、新しいプロジェクトを作るたびに貼り付け**ます。完璧な自動化ではありません。

しかし、2〜10人規模の中小企業であれば、この運用で十分に機能すると感じています。
