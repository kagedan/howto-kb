## はじめに

2026-05-29、Claude Opus 4.8 のリリースと Claude Desktop のアップデート直後に、AI サブセッションが意図せず終了する事象に遭遇しました。最初は数回でしたが、この記事を書いている数時間のあいだにさらに重なり、**計 5 回** になりました。最新版をいち早く採用している以上、リリース直後の不安定さは織り込むべきトレードオフですし、原因はまだ切り分け中で、リリース影響なのか、私自身が UI 操作で意図せず終了させたのか断定できていません。

ここで本記事には 2 つの結論があります。

ひとつは、複数の AI サブセッションを並行運営して、あるバックエンドサービス群の開発を進めている最中の出来事だったにもかかわらず、**作業ロスはゼロだった** こと。これは後述する回復力 (resilience) 設計のおかげでした。

もうひとつは、それでも **回復力設計だけでは足りなかった** こと。消失が数時間で 5 回に達すると、リカバリに費やす時間が作業の前進速度を上回り始めました。そこで、実行環境を Claude Desktop からターミナル版 (CLI) に切り替える判断をしました。

一文で先にまとめると、

> **個々の AI セッションは消えうる (ephemeral) と前提し、コードと思考を二重永続化して親セッションが痕跡から再構成する「回復力設計」を組む。ただし回復力は消失を吸収する床にすぎず、リカバリコストが前進を上回ったら、消失頻度そのものを下げる手 (実行環境の見直し) を併せて打つ。**

想定読者は、Claude Code / Cursor / Cline などの AI エージェントを使った複数セッション運営に取り組んでいる開発者・エンジニアリングマネージャーです。題材の具体は伏せますが、回復力設計のパターン自体はツールを問わず転用できます。

---

## 1. 何が起きたか — 数時間で計5回、リカバリが前進を上回った

その日、複数の AI サブセッションを並行で走らせ、あるバックエンドサービス群の開発を進めていました。親セッション (オーケストレーター役) が全体を統括し、子セッションがそれぞれの担当範囲を実装していく、という構成です。

この運営の最中に、セッションが意図せず終了する事象が **数時間で計 5 回** 発生しました。親で 1 回、子で複数回です。ちょうど Opus 4.8 と Claude Desktop のアップデート直後だったため、リリース直後の不安定さを疑いましたが、UI 操作で意図せず終了させた可能性も否定できず、原因は切り分け中です。

最初の数回は、後述する仕掛けが効いて **ロスゼロで吸収できました**。「これは回復力設計が効いているな」と手応えを感じていたほどです。ところが頻度が上がるにつれ、状況が変わってきました。**1 回あたりのリカバリは小さくても、回数が増えれば総リカバリ時間は積み上がります。** 5 回目に至るころには、作業を前に進める時間より、消えたセッションを立て直す時間の方が長くなっていました。

ここが本記事の転換点です。回復力設計は「消失あたりの損失額」をゼロに近づけますが、「消失あたりのリカバリ時間」までゼロにはできません。頻度がある閾値を超えると、吸収できているのに前に進まない、という状態になります。

> 学び: セッション消失は「起こったら困る異常」ではなく「いつか起こる事象」として織り込むべき対象。ただし頻度が上がると、吸収コストそのものが前進を侵食し始める。

## 2. 二重の永続化 — コードと思考を別々の外部に逃がす

まず、ロスゼロを支えた回復力設計の中身を書きます。土台は **状態を外部に永続化しておくこと** です。ポイントは、永続化すべき状態が 2 種類あり、それぞれ別の置き場所に逃がす必要があるという点です。

**① コード = こまめな commit / push** — 各サブセッションは、節目 (プルリクエスト単位) ごとに commit して push します。これでコードの成果は GitHub という外部に保全されます。セッションが消えても、push 済みのコードは 1 行も失われません。AI 駆動開発に限らず当然の習慣ですが、セッションが消えうる前提に立つと「こまめさ」の重要度が一段上がります。

**② 思考・設計 = 作業ファイルへの永続化** — コードだけでは足りません。なぜその設計にしたのかという判断記録、タスク着手前の事前検討 (kickoff)、設計書、ドラフト段階のコード、検討中の選択肢といった **思考の成果** は、コミット可能な完成形になる前の段階ではコードに現れません。これらを作業ディレクトリのファイルとして書き出しておきます。

この二重化が肝心です。コードだけ永続化していると、消失時に「何が出来ているか」は復元できても「なぜそうしたか」「次に何をするはずだったか」が欠落します。思考だけ永続化していてもコードの実体がなければ復元できません。**両方が揃って初めて、復旧時に欠落がない状態** になります。

> 学び: 永続化すべき状態はコード (commit/push) と思考 (作業ファイル) の 2 種類。片方だけでは復旧時に必ず欠落が出る。

---

## 3. 親による痕跡からの再構成 — 「どこまで進んだか」は復元できる

二重に永続化されていても、消えたセッションが「どこまで進んでいたか」を誰かが把握し直さなければ復旧は始まりません。並行運営では、この役割を **親 (オーケストレーター) セッションが担います**。

子セッションが消えたとき、親は痕跡を突き合わせて仕掛かりを再構成します。具体的には、

* `git log` / `git status` で、どのコミットまで進み、どのファイルが未コミットで残っているかを確認する
* `gh pr` / `gh issue` で、プルリクエストやチケットの状態から進捗段階を読み取る
* 作業ディレクトリのファイル (kickoff・設計書・判断記録) から、思考がどこまで進んでいたかを把握する

これらを突き合わせれば、「コードはここまで、設計判断はここまで確定、残作業はこれ」という仕掛かり状況を、消えたセッション本人がいなくても外部から再構成できます。**痕跡が十分に残っていれば、進捗は復元可能な情報になる** ということです。

この「親が子の痕跡から復元する」構造こそが、並行運営における回復力の要です。個々の子セッションが消えても、親が生きていれば全体は止まりません。そして仮に親が消えても、親の作業ファイルと全体管理ページが残っていれば、新しい親セッションが同じ手順で全体を再構成できます。

> 学び: 進捗は「セッションの頭の中」ではなく「git・gh・作業ファイルの痕跡」に宿る。痕跡が十分なら、本人不在でも進捗は外部から復元できる。

## 4. 復旧プロンプトの構造化 — self-contained な4要素

痕跡から仕掛かりを再構成できたら、最後はそれを使って **新しいセッションを立ち上げ直す** 工程です。ここで効いたのが、復旧プロンプトを定型フォーマットで構造化しておくことでした。

親が生成する復旧プロンプトには、次の 4 要素を **そのプロンプト単体で完結する (self-contained)** 形で盛り込みます。

1. **仕掛かり状況** — どこまで進み、何が未完で残っているか
2. **再開手順** — 何から着手すれば作業を継続できるか
3. **必読資料** — 設計書・全体管理ページ・関連記事など、文脈把握に必要な参照先
4. **確定済みの設計判断** — すでに決まっていて蒸し返す必要のない判断事項

そして起動時は「役割の提示 + 全体管理ページ + 復旧プロンプトのパス」という定型で再起動します。新しいセッションは context ゼロから立ち上がりますが、この復旧プロンプトを読むだけで、消えたセッションが持っていた文脈を引き継いで作業を再開できます。

ここで重要なのは、復旧プロンプトが **self-contained であること** です。「前のセッションの会話を覚えている前提」で書くと、新しいセッションは何も覚えていないので機能しません。逆に、確定済みの判断まで含めて自己完結的に書いておけば、同じ設計判断を一からやり直す無駄も、蒸し返しによる手戻りも防げます。

ちなみに、復旧プロンプトの生成と再起動には毎回それなりの手間がかかります。1 回なら誤差ですが、これが 5 回続くと、この「手間 × 回数」が次章の問題に直結します。

> 学び: 復旧プロンプトは「仕掛かり状況 / 再開手順 / 必読 / 確定判断」を self-contained に書く。新セッションは何も覚えていない前提で、それ単体で再開できる粒度にする。

## 5. 回復力の限界 — 吸収コストが前進を上回ったら頻度を下げる

ここからが今回いちばん書き残したい学びです。

回復力設計は強力で、消失あたりの **損失額** はゼロにできました。しかし消失あたりの **リカバリ時間** はゼロにできません。痕跡の突き合わせ、復旧プロンプトの生成、新セッションの起動、文脈の読み直し — どれも小さいながら時間を食います。

ここで単純な式が効いてきます。

> **総リカバリコスト = (1 回あたりのリカバリ時間) × (消失頻度)**

回復力設計が改善するのは前者 (1 回あたり) です。後者 (頻度) には何も効きません。だから頻度が十分に高くなると、いくら 1 回あたりを小さくしても総コストが前進を侵食します。今回まさに、**総リカバリ時間が前進時間を上回る損益分岐点** を越えてしまいました。

この損益分岐を越えたら、打つ手は「吸収をさらに磨く」ではなく「**頻度そのものを下げる**」に切り替わります。そこで取った具体策が、実行環境を Claude Desktop からターミナル版 (CLI) に切り替えることでした。

念のため補足すると、これはどちらの実行形態が優れているという話ではありません。Claude には Desktop 版とターミナル版 (CLI) という複数の実行形態があり、用途によって向き不向きがあります。私のユースケース — **複数サブセッションを長時間並行で走らせ続ける運営** — においては、ターミナル版の方が安定して回せると判断した、というだけのことです。あくまで自分の使い方に合う方を選んだ、用途適合の選択です。

この切り替えには、副次的な効果もあります。**原因切り分けの実験を兼ねている** という点です。原因は依然として未確定ですが、実行環境を変えて事象が再発しなくなれば、環境固有の要因だった可能性が高まりますし、再発するなら別の要因を疑う材料になります。手を打つことが、そのまま切り分けの一手にもなっているわけです。

> 学び: 回復力設計は「消失あたりのリカバリ時間」を下げるが「頻度」には効かない。総リカバリ時間が前進時間を上回ったら、吸収を磨くのをやめ、頻度を下げる手 (実行環境の見直し) に切り替える。

## 6. まとめ — 吸収と頻度低減の二段構え

数時間で 5 回のセッション消失を経て見えたのは、回復力には二段構えが要る、ということでした。

1. **消失を吸収する床 = 回復力設計** — ①二重の永続化 (コードは commit/push、思考は作業ファイル)、②親による痕跡からの再構成、③復旧プロンプトの構造化。これで消失あたりの損失額はゼロにできる。
2. **消失頻度を下げる手 = 実行環境の見直し** — 総リカバリ時間が前進を上回ったら、吸収を磨くより頻度を下げる。今回はターミナル版 (CLI) への切り替え。

これらを貫く前提は一つです。**個々の AI セッションは消えうる (ephemeral)。永続的なのは外部状態 (git / GitHub / ファイル) だけ** ということ。セッションを「消えない前提の作業場」とみなすと脆くなりますが、「いつでも作り直せる一時的な実行環境」とみなし、価値ある状態を片端から外部に逃がしておけば、消失は復旧可能な事象に変わります。

### 原因が未確定でも、待つ手と打つ手は両立する

今回の消失の原因はまだ切り分け中です。リリース直後の不安定さなのか、私の UI 操作起因なのか、断定できていません。普通なら「原因が分からない」状態は不安で、調査が終わるまで止まりたくなります。

ですが、待つ手と打つ手は両立します。Anthropic はこれまでも迅速に改善リリースを出してきた実績があるので、改善は落ち着いて待てます。同時に、自分の側でも頻度を下げる手 (実行環境の切り替え) を能動的に打てます。回復力設計があるからこそ、原因という不確実性に振り回されず、「待つ」と「打つ」を冷静に併用できる、というのが今回の実感です。最新版をいち早く採用することと、消えうる前提の設計を持っておくことは、矛盾せず両立します。

### 永続化は必要条件、運用が十分条件

最後に補足を。「外部に永続化しておけば安心」とまでは言い切れません。姉妹記事 [AI時代の急がない勇気と旧世代発想のアンラーニング](https://zenn.dev/fixu/articles/ai-era-courage-not-to-rush) でも触れたとおり、**永続化は必要条件であって十分条件ではありません**。永続化された痕跡を親が能動的に読み返し、復旧プロンプトに組み立て直す運用がセットになって初めて回復力として機能します。

なお、本記事を書くこと自体が、姉妹記事で提案した「クリティカルパスとは別レイヤーの活動 (知見共有)」の実践でもあります。並行運営で空いた余白を — そして今回は、リカバリの合間を — こうした振り返りの言語化に振り向けるのは、回復力設計の知見を組織に残す投資でもあると考えています。
