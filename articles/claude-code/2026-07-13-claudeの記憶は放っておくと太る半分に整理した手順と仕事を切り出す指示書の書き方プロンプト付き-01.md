AIエージェントに開発を任せていると、道具側に2つの「たまり場」ができます。ひとつはAIの**記憶(メモリ)**、もうひとつは**会話そのもの(コンテキスト)**。先日、この2つを同じ日に掃除したら、記憶は33ファイル・219KBから30ファイル・108KBへ**ほぼ半分**になり、会話のほうは肥大化する前に3本の独立作業へ切り出せました。今日はその実測を、初心者向けの手順とプロンプトつきで書きます。

**原理は2行で済みます。「忘れる機能がないなら、忘れる作業を予定に入れる」。そして「内容は記録に持たせ、指示書はポインタだけ渡す」。**

## 前提: AIの記憶とは何か

Claude Code(AnthropicのコーディングエージェントCLI/アプリ)には、プロジェクトごとにAIが自分で読み書きするメモリフォルダがあります。中身はただのMarkdownで、1ファイル=1つの事実。「オーナーはこういう方針だ」「この環境にはこういう罠がある」「この作業はここまで進んだ」。そして索引ファイルが1枚あり、**索引は毎回のセッション開始時にまるごとAIに読み込まれます**。

便利です。毎回同じ説明をしなくてよくなる。ただし人間の記憶と違う点が3つあります。

1. **減衰しない。** 人間は使わない記憶を忘れますが、ファイルは残り続けます。終わったプロジェクトの作業ログが、半年後も毎回読み込まれる。
2. **優先度がない。** 「オーナーの決定事項」と「先週のマージ作業の手順メモ」が、索引の上では同じ1行です。
3. **基本、追記される。** AIは作業のたびに記憶を書き足しますが、「終わったから消す」動機を自分では持ちません。書くのは仕事の一部でも、消すのは誰の仕事でもない。

つまり構造的に太る一方です。うちの場合、最大のファイルは音声合成プロジェクトの作業ログで**1ファイル52KB**。プロジェクト自体は1か月以上前に完成していて、そこに書かれた「次はこれをやる」はとっくに全部終わっていました。それが毎回、読み込み候補として居座っていたわけです。

## 整理の実測: 219KB → 108KB

やったことは単純で、AIに「自分の記憶を整理して」と頼んだだけです。実際に送った文面(実物の要約です):

```
そういえば今のうちにClaudeの記憶(メモリ)の整理を。
Claudeのハーネスって記憶減衰しないし、優先度もつけてないし、
基本追記だから太るよね? という理解に基づく。
```

ポイントは、作業指示より先に\*\*「なぜ太るのか」の理解を渡している\*\*ことです。AIは目的が分かると手順を自分で組み立てます。実際、全33ファイルを読んで棚卸しし、次の3分類で処理していきました。この分類だけ覚えて帰ってもらえれば、この章は成功です。

* **恒久なもの → 残して磨く。** 私の方針・好み・過去に下した決定、環境の罠。特に「これは提案しないでと言った事項のリスト」は最優先で保護。これを消すと、AIが却下済みの提案を延々と蒸し返すようになります。
* **終わったもの → 教訓だけ抜いて本体は削除。** 完了した作業のログは、残る学びを数行に絞り、経緯の実況は捨てる。
* **よそに正があるもの → ポインタ1行に置き換え。** コードの履歴、リポジトリ内の手順書、タスク管理表。すでにどこかが「正」を持っている情報を記憶に複製すると、あとで必ず食い違います。記憶側は「詳細は○○を見よ」の1行でいい。

結果、52KBの作業ログは5.6KBの「完成品の仕様+環境の罠+決定事項」になり、全体では**33ファイル・219KB → 30ファイル・108KB(51%減)**。毎回読み込まれる索引も、1件400字近くまで膨れていた行を150字の1行フックに書き直しました。汎用に使える整理プロンプトにするとこうなります:

```
メモリを全部読んで棚卸しして、次の3分類で整理して。
1. 恒久(私の好み・決定事項・環境の罠)は残して磨く。
   特に「私の決定」「再提案禁止」の記述は最優先で残すこと。
2. 終わった作業のログは、残る教訓だけ数行に抜いて本体は削除。
3. リポジトリや文書に正がある情報は「参照先パス」1行に置き換え。
最後に索引を1件1行・短い見出しに書き直して、
何をどれだけ削ったか数字で報告して。
```

なお圧縮は不可逆なので、初回は「削除する前に候補一覧を見せて」を足すと安心です。慣れてくると分かりますが、危険なのは消しすぎより\*\*「決定事項を経緯ログと一緒に捨てる」\*\*の一点だけで、そこはプロンプトで名指しして守らせます。

## もうひとつのたまり場: 会話が長すぎる問題

記憶を整理した同じ日に、もう1種類の掃除をしました。こちらは「1本の会話に仕事を詰め込みすぎない」ほうです。

AIとの会話(コンテキスト)は、長くなるほど読み直しコストが上がり、しかも別々の話題が混ざって精度が落ちます。そこでClaude Codeには、作業を**独立したセッションに切り出す**仕組みがあります。うちの画面ではカード(チップ)が3枚並び、クリックするとそれぞれが専用の作業コピーを持った別セッションとして起動する、という形でした。この日は「計測カウンタの実装」「成果物への署名」「作業環境の隔離設計」の3本を切り出しています。

ここで大事なのが、新しいセッションに渡す\*\*指示書(パケット)\*\*の書き方です。新しいセッションは生まれたてで、いままでの会話を何も知りません。かといって会話全体を継承させると、**10万トークン級**の読み込みになります。逆に指示書なしで放り込むと、AIは手探りの検索から始めて、これも数万トークンを平気で溶かします(しかも今回の場合、必要なタスク記録がまだ保存前で、新セッションからは「探しても存在しない」状態でした。指示書がなければ迷子が確定です)。

指示書はおよそ1,000トークン。置き換える先が数万〜10万トークンなので、**レバレッジは10〜100倍**です。切り出し自体の経済性は疑う余地がありません。

## 指示書の失敗と、直した型

ただ、この日は1個やらかしました。指示書を**太らせすぎた**のです。作業の範囲ややらないことの一覧は、すでにリポジトリのタスク記録に書いてあったのに、心配で指示書にも再掲した。結果、必要な3倍のサイズになりました。指示書は高性能なモデルの「出力」で書くので、ここが一番単価の高い文章です。太らせた分だけ素直に痛い。

直した型はこうです。**内容は記録が持つ。指示書はポインタと差分だけ。**

```
[レーン名] 目的: ○○を実装する(1文)
参照: 仕様と背景は <リポジトリ内のタスク記録のパス> を読む
     (この指示書には再掲しない)
所有: 触ってよいファイルは A, B, C のみ。D はレーンXの担当なので触らない
順序: レーンXの完了を確認してから着手(共有ファイルがあるため)
差分: この指示書を書いた時点で未保存のものは、あなたの作業コピーには
     存在しない。<保存元の絶対パス> から読むこと
停止: ○○(鍵の登録・公開操作など)の手前では必ず止まり、承認を待つ
完了: 終わったら△△を実行して結果を報告
```

指示書に書くのは「参照先」「触ってよい範囲」「他の作業との順序」「新セッションからは見えないものの在り処」「止まるべき一線」だけ。仕様の本文はリポジトリに置き、指示書からは指す。ついでに言うと、指示書はセッションの先頭に固定されるので、2ターン目以降はキャッシュ読みになり単価が約1桁下がります。払うのはほぼ初回だけです。

## Before / After

* 記憶: 33ファイル・219KB、索引1行400字 → **30ファイル・108KB、索引1行150字**。最大ファイル52KB→5.6KB。消えたのは経緯の実況で、決定事項と環境の罠は全部残存。
* 会話: 1本の長大セッションに積み増し → **独立3レーンに切り出し**。指示書は約1,000トークンで、数万トークンの手探り検索を置き換え。
* 指示書の書き方: 心配で内容を再掲(必要量の3倍) → **参照+差分のみの薄いポインタ型**。この教訓自体もAIの記憶に書かせたので、次回は最初から薄く書いてくるはずです。

気づいた方もいると思いますが、2つの掃除は同じ原理の表と裏です。記憶の整理は「よそに正がある情報をポインタに置き換える」ことで、指示書の設計は「最初からポインタだけ渡す」こと。\*\*内容の置き場所をひとつに決め、他は全部そこを指す。\*\*AI運用の文脈で言われる高尚な話に見えて、実体は書類仕事の基本です。

## おまけ: 忘れる作業の予定化、はしなかった

「忘れる機能がないなら、忘れる作業を予定に入れる」と冒頭に書きましたが、白状すると、カレンダーには入れませんでした。AI側は月1回の自動リマインドを提案してきたのですが、断りました。太る原理を一度理解してしまえば、次に整理が要るタイミングは自分で気づけるからです。それにリマインドの自動化も、増やせばそのぶん管理対象が増えます。

結果としてこの家の分担は、**内容はAIの記憶が持ち、忘れる合図は人間が持つ**、になりました。本文で書いた「内容は記録に、指示はポインタに」の原理そのままの着地で、われながら少し出来すぎだと思います。予定表に頼るかどうかは、あなたが原理を覚えていられるかどうかで決めてください。
