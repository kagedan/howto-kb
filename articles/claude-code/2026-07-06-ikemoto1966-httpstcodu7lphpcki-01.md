## 一言で言うと

Fable 5をうまく使うコツは、指示を細かく書くことではありません。自分が見落としている前提を先にAIと点検し、作業中もずれを直し続けることです。

## 何が起きているのか

Anthropic のエンジニア Thariq Shihipar さんが、2026年7月3日に自身のXアカウントで長文の手引きを公開しました。タイトルは「A Field Guide to Fable: Finding Your Unknowns」。Anthropic の公式ブログではなく、本人が実務で得たコツをまとめた個人の記事です。

背骨にあるのは「地図は現地そのものではない」という考え方です。地図とは、あなたが Claude に渡す指示、設定、前提のことです。現地とは、実際に作業が起きる場所、つまりコード、画面、ユーザー、既存の制約のことです。

この地図と現地のずれを、Shihipar さんは「unknowns（自分がまだ分かっていないこと）」と呼びます。Claude はこのずれにぶつかるたびに、あなたが何を望んでいるかを推測して判断します。作業が大きいほど、ぶつかるずれも増えます。

ここで重要なのは、Fable 5のようなモデルでは、失敗の原因がモデルの実行力だけではなくなることです。モデルが指示をかなり正確に実行できるほど、成果物の質は「人間が何を渡し忘れたか」に左右されます。

彼は自分の知識を4つに分けます。

1つ目は、指示に書いてある「分かっていて、自覚もあること」です。たとえば、作ってほしい機能名や修正したい画面です。

2つ目は、自分で気づいている「分かっていない箇所」です。たとえば、どの設計がよいか迷っている部分です。

3つ目は、あまりに当たり前で指示には書かないが、見れば分かる「暗黙の基準」です。たとえば、既存コードの書き方や、自分のチームでは避けている表現です。

4つ目は、そもそも考えついていない「気づいてすらいない抜け」です。本人には見えていないので、指示にも出てきません。

腕のいい書き手ほど、この抜けは少ない。ただし彼らも、抜けがあることを前提に動く、と彼は指摘します。抜けをゼロにするのではなく、抜けがある前提で先に手を打つ。それが使いこなしの技術だ、というわけです。

手引きは、その抜けを見つける具体策を作業の段階ごとに並べます。

着手前に行うのが、blindspot pass（盲点洗い）です。実装の指示を書く前に、「自分が見落としていそうな点を洗い出して」と Claude に頼みます。

設計を詰める段階では、複数の案を画面つきの試作で見比べます。さらに、Claude に1問ずつ質問させ、設計を左右する論点から順に詰めていきます。最良の参照として「気に入った既存コードの場所を指し示す」方法も挙げています。

作業中には、決めたことを implementation-notes.md（実装メモ）という一時ファイルに残させます。

作業後には、変更内容を Claude に小テスト形式で出させます。自分が満点を取れて初めて確定する、という手順です。

具体例として、Fable の発表動画そのものを Claude Code だけで編集した話が出てきます。動画編集は彼の専門ではありませんでした。音声の書き起こしの仕組みを Claude に説明させ、言い間違いや長い沈黙を切る方法を確かめ、話す言葉に合わせて画面が動く試作を作らせた。最後に映像の色味が気になったとき、彼はまず Claude に色調整の案をいくつか出させ、その中から選ぼうとしました。ところが、そもそも自分には「良い状態」が判断できないと気づきます。そこで彼は、案から選ぶのをやめ、色調整そのものを Claude に教わる方に切り替えました。自分の抜けを先に見つけることから始めた、というわけです。
