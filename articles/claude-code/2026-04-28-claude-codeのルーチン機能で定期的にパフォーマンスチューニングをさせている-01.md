Claude Codeのルーチン（Routines）機能を、何に使うのが良いのかしばらく考えていました。  
クラウド上でプロンプトを定期実行できる便利な機能なのですが、定期的に動かして意味のあるタスクは何だろう、と。

たどり着いたのが、パフォーマンスチューニングです。  
「速くなったか」は数値で判断できるので、ベンチマーク基盤さえあれば、あとはAIに任せられます。  
機能としてデグレしていないかもテストが充実していれば自動で確認できますし、ブランチを切って進めるので本流には影響しません。設計の創造性があまり要らないのもAIに任せやすいところですね。

私が開発しているRepomixというCLIで試してみたところ、実行速度が 約2.4倍 になりました。

<https://github.com/yamadashy/repomix/releases/tag/v1.14.0>

ある程度再現性のある形に落ち着いてきたので、プロンプトとベンチマーク基盤の話を中心に共有します。

## ルーチンとは

Claude Codeには、クラウド上でプロンプトを定期実行できるルーチン（Routines）という機能があります。  
対象のGitHubリポジトリを指定しておけば、実行結果はブランチへ自動でpushしてくれます。ローカルで動く `/loop` と違って、ラップトップを閉じていてもクラウド側で勝手に進んでくれるのが良いところですね。

<https://code.claude.com/docs/en/routines>

## ルーチンの設定

[claude.ai/code/routines](https://claude.ai/code/routines) のWeb UIから設定できます。今は2時間ごとのcronで回しています。

![ルーチンの設定画面](https://static.zenn.studio/user-upload/deployed-images/69339a3ae0f72aa3d0431816.png?sha=8ab6785db7fb6a615e289b29aa528f7ea0d83192)

プロンプトはこちら。

```
# 自動パフォーマンスチューニング

パフォーマンス改善作業を行う。

## 設定

- 作業ブランチ: `perf/auto-perf-tuning`
- ベースブランチ: `main`
- 対象フォルダ: `src`
- PRタイトル: `perf(core): Automated performance tuning by Claude`
- 改善指標: `1~2秒かかるCLIの1回の実行に対して、数msの改善は対象外とし、実行全体の2%以上の改善`
- 調査サブエージェント数: `5`
- サブエージェントのモデル: `sonnet`

## 方針

<対象フォルダ> とその関連コード（テスト、設定、依存関係）のパフォーマンス改善またはメモリ消費削減を行う。
<改善指標> をもとに測定可能なインパクトのある変更を目指す。
機能的な変更は行わず、既存の振る舞いを維持したままパフォーマンスのみを改善すること。

## 進捗チェックリスト

```
- [ ] 1. ブランチ準備
- [ ] 2. 既存PRの確認
- [ ] 3. 調査・計画
- [ ] 4. 実装
- [ ] 5. ベンチマーク・検証
- [ ] 6. コミット
- [ ] 7. ローカルでのレビュー
- [ ] 8. PR作成・更新
```

## 作業手順

### 1. ブランチ準備

- <作業ブランチ> が存在しない場合、最新の <ベースブランチ> から <作業ブランチ> を作成する。すでに存在する場合、チェックアウトする。
- その後、<ベースブランチ> が <作業ブランチ> より先に進んでいる場合、<ベースブランチ> をマージしてpushする。コンフリクトが発生した場合、対応が別で取り込まれている可能性を考慮して解決する。
 
### 2. 既存PRの確認

- <作業ブランチ> に対するPRがすでに存在するか確認する。
- PRが存在する場合、PRの説明とレビューコメントを読み、コンテキストを把握する。

### 3. 調査・計画

まずリポジトリの構造・処理フローを把握する。その後、<調査サブエージェント数> 分の重複しない調査スコープを自分で設計し、各スコープを担当するサブエージェント（モデル： <サブエージェントのモデル>）を並列に起動する。全エージェントの報告を統合し、改善計画を立てる。

スコープはリポジトリの特性に応じて自由に決めてよい。以下は一例：
- クリティカルパス特定（ボトルネック箇所の特定）
- I/O・外部通信（不要な再読み込み、同期I/O、バッファサイズ）
- データ構造・アルゴリズム（計算量、文字列操作、コレクション選択）
- 並列化・非同期（直列処理の並列化余地、不要な逐次await）
- 依存ライブラリ・初期化コスト（軽量代替、遅延ロード）

複数の改善点が見つかった場合でも、必ず1つだけに絞る。今回のチューニングで最もインパクトの大きい変更1つのみに集中する。

<改善指標> を満たす改善が見つからなかった場合、調査結果を報告して作業を終了する。

### 4. 実装

計画を実装する。

### 5. ベンチマーク・検証

- ベンチマークを実行し、変更が <改善指標> を満たすことを測定で確認する。
- その後、lint と test を実行し、リグレッションがないことを確認する。
- <改善指標> を満たさなかった場合、変更をrevertし、調査結果を報告して作業を終了する。

### 6. コミット

- `perf` プレフィックスでコミットする。
- コミットボディに対応内容とベンチマーク結果を詳しく記載する。

### 7. ローカルでのレビュー

- ローカルで複数のサブエージェントにレビューさせる。対応が必要な指摘があった場合、修正して再度レビューする。修正時はamendを使い、コミットは1つにする。
- 修正箇所がなくなった場合、pushする。

### 8. PR作成・更新

- PRがすでに存在する場合は更新する。存在しない場合は新規作成する。
- Draft で作成する。
- タイトル: <PRタイトル>
- 説明に対応内容とベンチマーク結果を追記する。
```

プロンプトの概要

* 設定セクションを冒頭に置いて調整しやすく
* 1回の実行で1つの改善に絞る
* 調査は5並列のサブエージェントでそれぞれ別視点
* 改善指標を満たさないならrevertさせて作業終了
* push前にローカルでサブエージェントレビュー
* 1つのPRに対応を積み上げる

特に「1回の実行で1つの改善に絞る」のが重要ですね。  
複数の改善を一度にやるとどれが効いたか分からなくなるので、変更とベンチマーク結果が1対1で見えるように保ちたい、というのが理由です。

## ベンチマークの計測

ルーチンではClaude Codeが勝手にベンチマークコードを書いて計測してくれていて、改善指標を満たしたものだけコミットを積むようにしています。

CI側でも私の確認用としてベンチマークを取っていて、こちらはGitHub Actionsで自動実行されて結果がPRにコメントされます。

<https://github.com/yamadashy/repomix/blob/main/.github/workflows/perf-benchmark.yml>

ざっくり以下のことをやっています。

* PRブランチとmainブランチを別々にチェックアウトしてビルド
* Ubuntu / macOS / Windowsの3OSでpack時間を計測（runsはOSごとに調整）
* 結果をPRにコメントで貼る

複数OSで計測しているのは、特定のOSだけ速い／遅いといった環境依存の改善を避けるためです。

新しいPRが毎回立ち上がると煩わしいので、`perf/auto-perf-tuning` という1つのPRへ積み上げる構成にしており、改善の履歴もそのまま積み上がっていきます。

![PRに積み上がっていくベンチマーク履歴](https://static.zenn.studio/user-upload/deployed-images/cbb37af2f84429370f9f69dd.png?sha=72a5425c8a51238b4cd3e8bc264a4d5425514011)

mainの改善履歴も眺めたかったので、mainへのpushごとに同じベンチマークを回して、結果を [github-action-benchmark](https://github.com/benchmark-action/github-action-benchmark) でgh-pagesに保存しています。

<https://github.com/yamadashy/repomix/blob/main/.github/workflows/perf-benchmark-history.yml>

このグラフはgh-pagesで公開され、コミットごとの実行時間を時系列で追えます。

<https://yamadashy.github.io/repomix/dev/bench/>

![Repomix Pack 実行時間のコミット推移（Linux）](https://static.zenn.studio/user-upload/deployed-images/435ee121fc070f2e7e214586.png?sha=4e131478da48b3cf196fdfd6b1f8d74c596e0fa5)

階段状に下がっていく様子と、ところどころでの踊り場や急降下が読み取れますね。  
眺めているだけで楽しい。

## mainへの取り込みまでの流れ

ルーチンが定期的に `perf/auto-perf-tuning` ブランチに積み上げてくれて、そのブランチに対するDraft PRができます。

<https://github.com/yamadashy/repomix/pull/1514>

このPRに記録されているベンチマーク履歴を眺めて、改善幅が大きそうなコミットを確認します。  
良さそうならClaude Codeに「mainからこの変更を再実装してPRにして」と頼んで、改めて正式なPRをmainに向けて切ってもらう、という流れですね。

ルーチンの自動PRをそのままマージしないのは、たまにデグレが混ざることもあれば、小さな改善のわりに大量のコード変更を入れてくることもあるためです。コミット単位で1つずつ確認して、責任を持って取り込んでいきたい、というのが理由です。

## やってみてわかったこと

### 改善のアイデアが無限に出てくる

定期的に回しても改善案が尽きないのは、ちょっとした驚きでした。  
面白いのは、自分の直感に反する提案が出てくるとき。「昔はこれで速くなったけど、今の構成だと戻したほうが速い」みたいなことを淡々と指摘してくるんですよね。  
コードベースは日々変わっているのに、過去の最適化って意外と見直さないもので、数値しか見ていないAIのほうがあっさり気付いてくれます。

### 単発では着手しない改善との相性が良い

v1.14.0には20本以上のPRが乗っていて、個別では-2%や-3%の地味な改善も多い。普通なら思いついても面倒でやらない類のものです。  
それでも、計測結果までセットで出てくると「じゃあ取り込むか」と素直に思える。判断の腰が軽くなるのは、自動化のいいところだなと思います。

## 改善したいこと

今のプロンプトはかなりシンプルで、計測方法・目標・制約あたりはざっくりとしか書いていません。  
たとえば「ベンチマークでは具体的に何を測るのか」「どのくらいの劣化までなら許容するか」「絶対に変えてほしくない振る舞い」をもっと丁寧に書ければ、AIの判断はもっと正確になりそうだなと思っています。

また、ベンチマークの基盤をリポジトリに置いていないのも気になっています。事前に用意しておけば、もっと正確に測れたりトークン数も抑えられただろうなと。

このあたりは少しずつ詰めていきたいですね。

## おわりに

ベンチマーク基盤を作る手間はそこそこありますし、伸びしろが減ってくれば空振りも増えるはず。AIの判断がすべて正しいわけでもないので、レビューでひっくり返すこともあります。

それでも「寝ている間にPRが育つ」体験は新しかったです。朝起きるとPRに新しい最適化コミットがいくつか積まれていて、ベンチマーク結果も添えてある。人間がやることはレビューだけ。

気になる方は、まずベンチマークをActionで取れるところから始めて、ぜひ手元のプロジェクトでも試してみてください。  
そこさえあれば、Claude Codeのルーチンが勝手に育ててくれます。

良いゴールデンウィークを！
