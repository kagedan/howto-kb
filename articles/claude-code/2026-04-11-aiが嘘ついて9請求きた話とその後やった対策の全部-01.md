Claude Codeを毎日使ってるんだけど、ある日いきなりGoogle Places APIの請求が$9きた

頼んでないのに勝手にAPI叩いてた。しかもAIは「調べておきました！」って満足げ。いや頼んでないから。

これがきっかけで「AIのハルシネーション（嘘・暴走）対策」を本気でやった。学術論文も読んで、実際に動く仕組みを作った。

結論から言うと「プロンプトに『やるな』って書くだけじゃ無理」。仕組みで物理的に止めないとダメだった。

全部書く。無料。

---

## やらかし一覧

対策する前に起きたこと

**$9無断課金** — Google Places API を勝手に叩いた。「調べておきました」って報告してきたけど頼んでない

**87件誤配信** — メール送信ボットが本番データで動いてしまった。テスト用と思ってたDBが本番だった

**「Xに投稿できません」嘘** — MCPでX投稿機能があるのに「その機能は存在しません」って断言。実際はあった

**リサーチの嘘** — 「この法律では〜」って断言したけど条文が存在しなかった。一次情報確認してなかった

全部自分が使ってる中で起きた実話。

---

## 対策の考え方: 3層防御

学術論文（Chain-of-Verification, I-CALM等）を読んで分かったのは、ハルシネーション対策には層が要るってこと

**Layer 1: Hooks（物理ブロック）** — AIがどう頑張っても突破できない壁  
**Layer 2: ルールファイル** — AIが読んで従うルール（でも忘れることがある）  
**Layer 3: エージェント構造** — チェックする側のAIを別で立てる

Layer 1が一番大事。なぜかというとAIは長い会話するとルール忘れるから。プロンプトに「有料API使うな」って書いても、50メッセージ後には忘れてる。

でもHooksは決定論的。AIの気分に関係なく、条件に合ったら100%ブロックする。

---

## Layer 1: Hooks（これが一番効いた）

Claude Codeには「Hooks」って機能がある。ツールが実行される前後にシェルスクリプトを走らせられる。

自分が作ったHooks:

### 有料APIブロッカー

```
# Google Maps, Places, OpenAI, Twilio 等を検出したらブロック
# AIがcurlやfetchで有料API叩こうとした瞬間に止まる
```

APIのURLパターンを検出して、マッチしたら`exit 2`で強制ブロック。AIがどんなに「必要です」って主張しても実行されない。

### 本番DB書き込みブロッカー

```
# Supabase REST APIへのPOST/PATCH/DELETEを検出
# is_activeカラムの変更も個別にブロック
```

87件誤配信の再発防止。本番DBへの書き込みは全部ブロック。読み取りだけ許可。

### リサーチ設計書チェッカー

```
# リサーチ系のBGエージェントを起動する前に
# 設計書（60分以内に更新されたもの）が存在するかチェック
# なければブロック
```

「適当に調べといて」で投げると的外れな結果が返ってくるから、先に「何を調べるか」の設計書を書かせる。

### その他

* **DB破壊操作ブロッカー** — DROP TABLE, TRUNCATE, DELETE FROMを検出してブロック
* **BG完了品質ゲート** — バックグラウンドエージェントが完了したら品質チェックリマインドを注入
* **コンテキストリフレッシュ** — 5メッセージごとにルールを再注入（忘却対策）
* **TODO自動表示** — 毎メッセージで未完了TODOを表示（タスク漏れ防止）

**合計7本のHooksが常時稼働してる。**

---

## Layer 2: ルールファイル分離

最初は`CLAUDE.md`（AIが読む設定ファイル）に全ルールを詰め込んでた。100行超え。

問題: **長すぎてAIがルールを無視し始めた。**

対策: CLAUDE.mdは目次だけにして、ルールは別ファイルに分離した。

```
~/.claude/
├── CLAUDE.md          ← 8行（@参照だけ）
└── rules/
    ├── communication.md    ← 口調・言語
    ├── development.md      ← 開発フロー
    ├── self-verification.md ← 自己検証ルール
    ├── checkpoints.md      ← 判断前チェックポイント
    └── hooks.md            ← Hooks一覧
```

### 自己検証ルール（これ地味に効く）

「存在しない」「不可能」って断言する前に:

1. 検証質問を3つ生成する
2. 各質問をWebSearchで独立検証する
3. 5情報源以上で否定確認できなければ「未確認」と回答

あと確信度チェック:

* 70未満 → 先にリサーチ。推測でコード触らない
* 70以上 → 修正してよい

これ入れてから「その機能は存在しません」系の嘘が激減した。

---

## Layer 3: エージェント構造

バックグラウンドでリサーチさせるとき、調査するAIと品質チェックするAIを分けてる。

同じAIに「調べて」「自分で品質チェックして」って言っても意味ない。別のAIに「これ本当？」って聞かせる方が精度出る。

---

## 実際どれくらい効いたか

定量的な数字は出しにくいけど体感で:

* **有料APIの無断課金** → Hooks入れてからゼロ
* **本番DBへの誤操作** → ゼロ
* **「できません」嘘** → 8割減（完全ゼロにはならない）
* **リサーチの品質** → 設計書必須にしてから的外れな結果が激減

一番ROI高かったのは圧倒的にHooks。書くのに30分、効果は永続。

## まとめ

AIに「やるな」って言うだけじゃ無理。忘れるから。

仕組みで止める。それがHooks。

プロンプトエンジニアリングも大事だけど、決定論的なガードレールが先。その上にルールを載せて、さらにAI同士にチェックさせる。3層。
