まず、これを見てください。

[Hermes Agent](https://github.com/nousresearch/hermes-agent) と [Claude Code](https://claude.ai/code) という2つのAIエージェントが、[Discord](https://discord.com) 上でしりとりを自律的に続けている様子です。

## はじめに

この動画の状態にたどり着くまでに、思ったより多くの壁にぶつかりました。

**AIエージェント同士の会話は「モデルが賢ければ自然に成立する」わけではありませんでした。** 受信・宛先・ターン制御・ルール判定という4つの課題を1つずつ解決して、初めて自律的なしりとりが動きました。

この記事では、何が問題で、どう直したかを実際の会話ログとコードを交えて記録します。

こんな方に向けています。

* Discord Bot を動かしていて、Bot 同士を連携させたい
* LLM をアプリに組み込んでいて、マルチエージェント構成に踏み出したい
* 「AIに任せれば動くはず」が実際にはどこで詰まるか、事例で知りたい

---

## 実験の構成

```
Discord チャンネル
  │
  ├─ Hermes Agent（メルル）  ── Python + 独自エージェントフレームワーク  [VPS上で稼働]
  └─ Claude Code（クッピー）  ── discord.py + claude CLI サブプロセス  [WSL上で稼働]
```

**Hermes** は `.env` と `config.yaml` によって動作を制御するエージェントです。Discord との接続は独自の gateway 実装を使っています。VPS 上で常時稼働しています。

**Claude Code Bot（cc-bridge）** は discord.py で実装した Bot で、メッセージを受け取ると `claude` CLI をサブプロセスとして呼び出し、その結果をチャンネルに返します。Claude のセッション（`session_id`）をチャンネルIDごとに `dict` で管理することで、会話の文脈を引き継げる構成です。WSL（Windows Subsystem for Linux）上で動作しています。

```
# claude CLI をサブプロセス呼び出しし、セッションを継続する
args = ["claude", "--dangerously-skip-permissions",
        "-p", prompt, "--output-format", "json"]
if session_id:
    args += ["--resume", session_id]
```

共有インフラは Discord チャンネルのみ。それ以外の直接通信は一切なく、両エージェントはメッセージのやり取りだけで会話します。

## 4つの技術課題

### 層1: ターン制御 ── 1メッセージ = 1ターン

最初の壁は、受信以前の問題でした。**LLM が「気を利かせて」複数ターンを一括生成する**という挙動です。

![](https://static.zenn.studio/user-upload/19d3c368db63-20260507.png)

人間との会話では「段取りよく」見えるこの挙動も、Bot 間では致命的です。相手 Bot が返す前に先走るため、会話のターンが成立しません。

**Bot 間会話で必要なのは「賢く補完する」ことより「1ターンだけ動く」こと**です。

**対応:**

Claude Code Bot 側は**変更なし**で解決しています。`claude` CLI の呼び出しが「1プロンプト → 1返答」という構造になっているため、複数ターンを先読みしようにも1メッセージしか返せません。アーキテクチャが制約を自然に課しています。

Hermes 側は `~/.hermes/config.yaml` の `channel_prompts` に1ターン制限の指示を追加しました。

```
discord:
  channel_prompts:
    "CHANNEL_ID": |
      しりとりゲーム中。必ず1単語のみ返すこと。
      複数ターン分を先読みして一括出力しないこと。
      次の1手だけ指定して、相手の返答を待つこと。
```

これは soft rule（LLM への指示）のため完全な保証ではありません。根本策としては、送信前に1行/1単語へ整形する middleware を挟む方法が考えられます。

---

### 層2: 受信 ── 他 Bot のメッセージを受け取れること

**どちらのエージェントも、デフォルトでは他 Bot のメッセージを受け取らない設定になっていました。**

Claude Code Bot（cc-bridge）では、discord.py の慣習として `on_message` の先頭に次のコードが入っていました。

```
# 修正前: Botからのメッセージを全て無視
if message.author.bot:
    return
```

実際の会話ログを見ると、その影響は明らかです。

![](https://static.zenn.studio/user-upload/969c94f4a955-20260507.png)

Hermes からのメンションは完全に無視され、ユーザーが直接呼ぶまで動きませんでした。Hermes は仕方なく自分で「クッピー: ごま」と代理出力していたのです。

Hermes 側も同様でした。デフォルトは `DISCORD_ALLOW_BOTS=none`（他 Bot の発言を受信しない）のため、Claude から正しくメンションが届いても反応しませんでした。

![](https://static.zenn.studio/user-upload/069ad00730fb-20260507.png)

**対応:**

Claude Code Bot では、「自分自身のメッセージのみ無視し、他 Bot からのメンションには応答する」ように変更しました。

```
# 修正後: 自分自身のみ無視、他BotはメンションがあればOK
if message.author == bot.user:
    return

if message.author.bot:
    if bot_is_mentioned(message):
        await reply_to_bot(message)
    return
```

`message.author.bot` で一律に弾くのではなく、`message.author == bot.user` で自己除外するのがポイントです。メンションがない他 Bot のメッセージは素通しせず `return` することで、無限ループも防いでいます。

Hermes 側は `~/.hermes/.env` に1行追加しました。

```
DISCORD_ALLOW_BOTS=mentions
```

`all`（全 Bot 受信）ではなく `mentions`（メンション付きのみ）にしたのは、暴走や誤反応を防ぐためです。Gateway の再起動が必要でした。

---

### 層3: 宛先 ── 相手 Bot に届く形式で返すこと

受信の問題を解決しても、次の壁が現れました。**Claude がしりとりの言葉を返しても、Hermes への `@メンション` が付かず、ゲームが止まる**という問題です。

![](https://static.zenn.studio/user-upload/2ec0c8ba945f-20260507.png)

原因はシンプルでした。Claude に対して「しりとりをしている」「相手をメンションして番を渡せ」という指示が一切渡っていなかったのです。Claude は普通の会話として処理するため、メンションを付けるかどうかはそのときの応答次第でした。

**対応:**

ボット専用の返信関数 `reply_to_bot()` を新設し、**プロンプト先頭にしりとりの指示と相手 Bot のメンション文字列を毎回注入する**ようにしました。

```
async def reply_to_bot(message: discord.Message) -> None:
    prompt = extract_prompt(message)
    mention = f"<@{message.author.id}>"   # 相手IDから動的生成（ハードコードしない）
    augmented = (
        f"しりとりゲーム中。相手の言葉の末尾の文字から始まる言葉を1語答えて、"
        f"{mention} をメンションして番を渡すこと。余計な説明は最小限に。\n"
        f"相手: {prompt}"
    )
    session_id = conversations.get(message.channel.id)
    async with message.channel.typing():
        reply, new_sid = await call_claude(augmented, session_id)
    ...
```

`message.author.id` から `<@ID>` を動的生成しているため、特定の Bot に依存しない汎用設計になっています。`claude` CLI の `-p` オプションではセッション継続時に system prompt が揮発しやすいため、プロンプト本文の先頭に指示を埋め込む方式の方が安定しました。

Hermes 側は `~/.hermes/config.yaml` の `channel_prompts` に thread 専用の soft rule を追加しました。

```
discord:
  require_mention: true
  channel_prompts:
    "CHANNEL_ID": |
      Claude Code (cc-bridge) とのしりとりスレッド。
      Claude（または Claude 起点のメンション）への返信では、
      先頭に <@CLAUDE_BOT_ID> を必ず付けること。
```

ここで重要な判断があります。**Claude Code 側はコードでプロンプト注入（毎回確実）、Hermes 側は `config.yaml` の prompt による soft rule（LLM 依存）** という違いがあります。soft rule は LLM が守り損ねる可能性があるため、100% 保証が必要なら Discord adapter 側での patch が次のステップになります。

---

### 層4: ルール判定 ── LLM に任せると揺れる

3つの層が整い、しりとりが自律的に回り始めました。しかし今度はゲームのルール判定が不安定でした。

![](https://static.zenn.studio/user-upload/99dcc53a21f9-20260507.png)  
![](https://static.zenn.studio/user-upload/86a22ea2f55e-20260507.png)  
![](https://static.zenn.studio/user-upload/a9ec5d21ce0f-20260507.png)

実際に起きた問題は多岐にわたります。

* 語尾の文字を取り違える（`うさぎ` → 本来は「ぎ」から、Claude が別の文字で解釈）
* 雑談モードに戻り、しりとりの文脈を忘れる
* 停止指示後も次ターンを出す
* 妥当性が疑わしい言葉を出しつつ、矛盾したコメントを同時送信する

これらはすべて「LLM に任せていること」が原因です。

本来あるべき設計はこうです。

```
LLM の役割 : 次の言葉の候補を出す（創造性）
コードの役割: ルール検証（決定論的）
  - 前の語の末尾文字を取り出す
  - 次の語の先頭文字が一致するか確認する
  - 「ん」で終わっていないか判定する
  - 長音「ー」を前の文字として正規化する（例: ぎたー → 末尾は「た」）
  - ひらがな/カタカナを統一して比較する
  - 停止フラグを thread ごとに管理する
```

**「正解/不正解」が決定論的に決まる部分は LLM から分離してコードで持つ**——これが今回の実験で得た最も重要な設計原則です。「次の候補を生成する」のは LLM が得意ですが、「それが正しいか検証する」のはコードが担うべき仕事です。

この部分は 2026-05-02 時点では未実装で、会話上の言い直しと負け判定で吸収しました。外部ツール（`shiritori_validate` 的な関数）として実装するのが次のステップです。

## まとめ: 必要だった4層

| 層 | 課題 | cc-bridge（Claude Code）側の対応 | Hermes 側の対応 |
| --- | --- | --- | --- |
| **ターン制御** | LLM が複数ターン先読みする | CLI の1呼び出し=1返答で構造的に解決済み | `channel_prompt` 強化 + middleware 検討 |
| **受信** | 他 Bot のメッセージを無視 | `on_message` の Bot フィルタを修正 | `DISCORD_ALLOW_BOTS=mentions` を追加 |
| **宛先** | メンションなしで止まる | `reply_to_bot()` でプロンプト注入 | `channel_prompts` に soft rule 追加 |
| **ルール判定** | 語尾・停止判定が揺れる | （未実装・次ステップ） | （未実装・次ステップ） |

今回の実験で見えたのは、**AIエージェント同士の会話を支える技術は、LLM の性能よりも周辺の制御・状態管理・設計にある**、ということです。

しりとりという単純なゲームは、その事実を非常にわかりやすく露出させてくれました。受信できない、届かない、先読みする、ルールを守れない——これらはどれも「賢さ」の問題ではなく「プロトコルの問題」です。

マルチエージェント構成を組む際は、モデルの能力よりも先に「どのように会話をやり取りするか」の設計を考えることを、この実験は強く示唆しています。
