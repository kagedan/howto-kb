（本稿は、Anthropic 公式コース 「Claude Code in Action」 の "Github integration" 章で学んだ内容を、筆者なりの解釈とメタファーで再構成したものである。）

GitHub に新しい「同僚」を迎える準備はできていますか。

これまでローカル環境で右腕として働いてくれた **Claude Code** が、GitHub リポジトリの中に「常駐」できるようになりました。Issue（課題チケット）や PR（プルリクエスト／コードの変更提案）で名前を呼べば応答し、決められた仕事を自動でこなす——いわば、リポジトリに **新しい社員を一人雇い入れる** ようなものです。

ただし、ここで多くの人が誤解します。「雇えば、勝手に全部のコードを見てレビューしてくれる」——これは半分は正しく、半分は間違いです。本当に戦力にするには、**入社手続き・業務マニュアル・使ってよい道具の指定**という、人を雇うときとまったく同じ段取りが要ります。  
![](https://static.zenn.studio/user-upload/14b3c943739c-20260607.png)

この記事では、Claude Code の GitHub 統合を「職場に新しい社員を迎える」という比喩で一気通貫に解説します。結論を先に言います。

本記事は [Claude Code GitHub Actions 公式ドキュメント](https://code.claude.com/docs/en/github-actions) の v1.0 GA（正式版）仕様を一次情報として構成しています。

## 前提条件：採用面接の前に確認すること

このセクションのまとめ：**雇う前に、自分が「採用権限を持つ立場」かどうかと、給与（課金）の発生を確認する。**

人を雇うとき、まず「自分にその権限があるか」「人件費を払えるか」を確かめますよね。Claude Code の配属も同じです。始める前に、次の4点を確認してください。

* **あなたがリポジトリの管理者（admin）であること**  
  GitHub App のインストールには、リポジトリ管理者権限が必要です。人事権を持つ人でないと、新しい社員を雇い入れられません。
* **Anthropic API キーを用意できること**  
  Claude を動かす「給与の支払い口座」にあたります。`ANTHROPIC_API_KEY` を準備します。
* **クイックスタート（`/install-github-app`）は Anthropic API を直接使う人だけが対象**  
  Amazon Bedrock や Google Vertex AI 経由で Claude を使う場合は、この簡単コマンドではなく手動セットアップが必要です。
* **2種類の費用が発生すること**  
  Claude の利用に対する **API トークン課金** と、処理を走らせる **GitHub Actions の実行時間課金** の両方がかかります。タダ働きの社員ではありません。

## 1. 入社手続き：`/install-github-app` という採用通知

このセクションのまとめ：**`/install-github-app` は、新入社員の入館証とデスクをまとめて用意する採用手続きである。**  
![](https://static.zenn.studio/user-upload/0d3d1575f9f1-20260607.png)

新しい社員を迎えるとき、まずは入館証を発行し、デスクを用意し、入室できる部屋を決めますよね。Claude Code の導入も、ほぼこれと同じ手順です。

* **コマンド一つで開始**  
  ターミナルで Claude Code を起動し、`/install-github-app` を実行します。
* **3つのステップ**  
  GitHub App のインストール → 認証情報（API キー）の設定 → 自動生成される PR の作成、という流れで進みます。
* **配属完了**  
  この PR をマージすると、リポジトリの `.github/workflows`（GitHub 上で自動処理を走らせる設定の置き場）に「配属先の設定ファイル」が作られます。

ここで見落としがちなのが、**この社員に渡す「鍵の範囲」** です。GitHub App をインストールすると、Claude は次の3つの権限を要求します。これは「どの部屋に入ってよいか」を決める入館証のグレードにあたります。

| 要求される権限 | レベル | 職場でたとえると |
| --- | --- | --- |
| Contents（リポジトリの中身） | 読み書き | 書類棚を見て、書き換えてよい |
| Issues（課題チケット） | 読み書き | 課題ボードを見て、書き込んでよい |
| Pull requests（変更提案） | 読み書き | 変更提案を見て、コメント・修正してよい |

## 2. 仕事の頼み方：「名指し」と「定型業務」の2つの動き

このセクションのまとめ：**Claude への仕事の頼み方は「名指しで呼ぶ」か「定型業務として自動で回す」かの2通り。どちらで動くかは Claude が文脈から自動で判断する。**

配属された Claude は、文脈に応じて2種類の働き方を **自動で使い分け** ます。

> ここが、以前の情報から仕様が変わった重要ポイントです。  
> 旧ベータ版では `mode: "tag"` / `mode: "agent"` と動作モードを手動で書き分けていました。  
> v1.0 GA（正式版）では **モードは自動検出（auto-detected）** になり、`mode` という設定そのものが廃止されました。  
> 「2つのモードを切り替える」という古い説明はもう当てはまりません。

### ① 名指しで呼ぶ（メンション応答）

Issue や PR のコメントで `@claude` と名前を呼ぶと、Claude がその場に駆けつけます。

* **依頼の理解**：頼まれた内容を分析し、実行計画を立てます。
* **作業**：コードベース全体を見て、必要な修正を行います。
* **報告**：結果をそのまま Issue や PR のコメントとして返します。

会議で「この件、Claude さんお願い」と名指しするイメージです。呼ばれて初めて動く、対話的な働き方です。

### ② 定型業務として自動で回す（プロンプトによる自動実行）

「PR が来たら毎回チェックする」のような定型業務は、ワークフロー（自動処理の手順書）に **トリガー（起動の合図）と指示（prompt）を自分で書いて** セットします。たとえば「`on: pull_request`（PR が開かれたら起動）」という合図と、「変更点をレビューして」という指示を組み合わせます。

そうすると Claude は、毎朝決まった巡回をこなす社員のように、PR が来るたび自動でレビューを始めます。

> **どんでん返し：** 「設定なしで全 PR を見てくれる据え置き型の機能はないの？」——あります。ただし別物です。  
> 全 PR を自動でレビューするマネージドな機能は **「Code Review」** という別立ての機能で、現在は **リサーチプレビュー（試験提供）** かつ **Team / Enterprise プラン限定**（Zero Data Retention を有効にした組織は対象外）です。  
> 「無料で誰でも自動レビューが付いてくる」わけではない、という線引きを押さえておきましょう。  
> 詳しくは [Code Review 公式ドキュメント](https://code.claude.com/docs/en/code-review) を参照してください。

## 3. 戦力化：業務マニュアルと道具を渡す

このセクションのまとめ：**優秀な社員も、職場のルールと道具を渡さなければ力を出せない。設定ファイルでマニュアルと道具を整える。**

ここまでが「採用と仕事の頼み方」。ここからが本題です。

どんなに優秀な新入社員でも、その職場独自のルールや、社内システムの使い方を知らなければ本領を発揮できません。GitHub Actions（GitHub 上で自動処理を走らせる仕組み）上の Claude も、設定ファイル（YAML／設定を書くためのテキスト形式）を編集することで「教育」できます。

### 業務の事前準備（セットアップ）

社員が仕事を始める前に、机を整え、必要なアプリを立ち上げておく——それと同じ準備を指示できます。

* 開発用サーバーの起動や、依存関係のインストールなどを、ワークフローの事前ステップとして組み込みます。
* たとえば `npm run setup` を走らせる、といった具合です（これはあくまで一例で、実際のコマンドはプロジェクトごとに異なります）。

### 社内マニュアル（カスタム指示）

「このプロジェクトのサーバーは常に `localhost:3000` で動いている」といった、**現場の暗黙知**をテキストで伝授できます。新人に渡す業務マニュアルにあたる部分です。

### 特殊な道具の貸与（MCP サーバー）

社内には、専用の道具がありますよね。Claude にも、標準では持っていない道具を貸し出せます。それが MCP の役割です。

MCP（Model Context Protocol／モデルと外部ツールをつなぐ共通規格。公式：<https://modelcontextprotocol.io/> ）を使うと、Claude に新しい能力を後付けできます。たとえば Playwright MCP サーバーを追加すれば、Claude は「ブラウザという目」を手に入れ、画面上で実際にボタンが動くかを確認するテストまで代行できるようになります。

専用の検査機材を貸し出して、現場検査までこなせる社員に育てる——そんなイメージです。

## 4. 権限設計：合鍵を束ごと渡してはいけない

このセクションのまとめ：**GitHub Actions 上の Claude には「一括許可のショートカット」を持ち込まない。使ってよい道具を、原則として一つずつ指定する。**

ここが最も重要なポイントです。  
![](https://static.zenn.studio/user-upload/d7002f559b3b-20260607.png)

ローカル環境では、開発者が自分の判断で権限をまとめて許可することもあります。しかし CI 環境（自動処理が走るサーバー上）では事情が違います。**原則として、使ってよい道具を一つずつ指定する** のが基本です。新しく雇った外部の作業員に、会社の合鍵を束ごと渡す人はいませんよね。「この部屋のこの引き出しだけ」と指定して鍵を渡すのと同じ厳格さが要ります。

* **道具は個別に指定する**  
  使用を許可するツールは、一つずつリストアップします。
* **MCP の道具にも同じルールが及ぶ**  
  MCP サーバーを使う場合、そこに含まれる道具も、使ってよいものを明示的に許可リストへ記載します。

そして、ここが旧情報からの変更点です。許可リストの書き方が変わりました。

> ベータ版では `allowed_tools:` という入力名でしたが、v1.0 GA では廃止されました。  
> 現在は `claude_args`（Claude に渡す引数）の中の `--allowedTools`（エイリアス `--allowed-tools` も可）に統合されています。  
> 古い記事のとおりに `allowed_tools:` と書いても、現行のアクションでは動きません。

実際の書き方は、ワークフローの YAML にこう記述します。

.github/workflows/claude.yml

```
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          # 使ってよい道具を一つずつ指定する（合鍵の束ごとは渡さない）
          claude_args: '--allowedTools "Bash(gh issue view:*),mcp__playwright__browser_click"'
```

## 結論：自動化ではなく「協働」へ

Claude Code の GitHub 統合は、ボタン一つで全部が片付く魔法ではありません。それは、リポジトリに新しい社員を迎え、**業務マニュアルを渡し、使ってよい道具を指定し、入室できる部屋を限定する**——人を戦力化するのとまったく同じプロセスです。

私がエンジニアとして大切にしているのは、「自動化の便利さ」より「権限の見通しの良さ」です。何ができて何ができないかが一目で分かる社員ほど、安心して仕事を任せられます。Claude も同じで、権限を絞るほど、むしろ自由に働かせられます。

最初の一歩は、こうです。

1. リポジトリ管理者として `/install-github-app` を実行し、配属を済ませる。
2. `on: pull_request` で起動する **レビュー用のワークフローを1本だけ** 置く。
3. その上で、Issue に `@claude` と名指しで **小さな仕事を1つ** 頼んでみる。

いきなり全部を任せる必要はありません。デフォルト設定から始めて、少しずつあなたのプロジェクト専用の「頼れる同僚」に育てていく。その第一歩を、今日のうちに踏み出してみてください。
