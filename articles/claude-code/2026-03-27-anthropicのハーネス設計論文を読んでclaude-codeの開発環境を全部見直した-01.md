## 本文

AIに「この実装で問題ない？」と聞くと、ほぼ毎回「はい、適切に実装されています」と返ってきます。でも実際に動かすと、本番で壊れるコードが混ざっている。Anthropicが公式ブログで、この現象の構造と解決策を実験データ付きで公開していました。

### 結論

1. Anthropicが公式ブログで発表した「ハーネス設計」の研究で、**AIは自分の成果物を甘く評価する**ことが実験的に示された
2. 解決策は「生成するAI」と「評価するAI」の分離。この構造はClaude Codeの設定（CLAUDE.md・スキル・エージェント・フック）で再現できる
3. 自分の環境を見直したら、「生成」はできていたが「評価」の仕組みがほぼなかった。評価基準をスキルとルールに落とし込んだ

この記事は、Anthropicの研究内容の紹介と、それをClaude Code環境にどう適用したかの記録です。

---

### Anthropicの実験：AIの自己評価問題と3エージェント構造

2026年3月、Anthropicが公式ブログで「[Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)」を公開しました。AIエージェントに数時間かけてアプリを自律的に作らせる実験の報告です。核心部分を要約します。

> エージェントは、自分が作成した作品を評価するよう求められると、たとえ人間の目から見て明らかに平凡な出来栄えであっても、**自信満々に作品を褒め称える傾向があります**。

AIは問題を特定していても「大した問題ではない」と自己判断して承認してしまう。Anthropic自身が「クロードは初期状態ではQAエージェントとして不十分」と認めています。

解決策として構築されたのが、3つのエージェントからなるシステムです。

| エージェント | 役割 | 担当 |
| --- | --- | --- |
| プランナー | 1行のプロンプトを完全な製品仕様に展開 | スコープ定義 |
| ジェネレーター | 仕様に基づいてコードを実装 | 実装 |
| 評価者 | 実行中のアプリを操作してバグを発見、採点基準でフィードバック | 品質保証 |

評価者なし（ソロ実行）ではゲームの中核機能が動作しなかったのに対し、評価者あり（フルハーネス）では基本動作が機能。評価者は27項目もの基準でテストし、「`fillRectangle`関数は存在するが`mouseUp`時に正しくトリガーされない」のような具体的なバグ報告を返していました。

この構造を見たとき、「Claude Codeの設定で再現できるのでは」と思ったのが、環境見直しのきっかけです。

### Claude Codeの設定体系を「ハーネス」として見直す

Claude Codeの設定を、Anthropicの3エージェント構造に対応させると、以下のようになります。

| Anthropicの構造 | Claude Codeでの対応 | 具体的な設定 |
| --- | --- | --- |
| プランナー | CLAUDE.md + プランニングスキル | プロジェクト概要、制約、設計判断 |
| ジェネレーター | Claude Code本体 + 技術スキル | コード生成、実装 |
| 評価者 | レビューエージェント + ルール + フック | 品質基準、自動チェック |

この対応を見て気づいたのは、自分の環境では「プランナー」と「ジェネレーター」はそこそこ整備されていたのに、**「評価者」がほぼ存在しなかった**ということです。

CLAUDE.mdには「何を作っているか」は書いてある。スキルには「どう書くべきか」も入っている。でも「書いたものが本当に正しいか検証する仕組み」がなかった。AIに「この実装で問題ない？」と聞いても、Anthropicが実験で示した通り「はい、適切です」としか返ってこない。

### 評価者を設定に組み込む——3つのレイヤー

Anthropicの研究を参考に、Claude Codeの評価の仕組みを3層で設計し直しました。

#### レイヤー1：ルール（常に適用される評価基準）

`~/.claude/rules/` に置くルールファイルは、毎セッション自動で読み込まれます。ここに「AIが自然にはやらないが、品質上必須のチェック」を書きます。

例：Supabase/PostgreSQL用の評価基準（30ルール中の一部）

```
# ~/.claude/rules/supabase-postgres.md（抜粋）

## FK列にインデックスを追加
PostgreSQLはFK列を自動インデックス化しない。

## RLS パフォーマンス最適化
auth.uid() を SELECT でラップ（行ごとの呼び出しを防止）:
- BAD: using (user_id = auth.uid())
- GOOD: using (user_id = (select auth.uid()))

## カーソルベースページネーション
- BAD: OFFSET（深いページで遅い）
- GOOD: WHERE id > $last_id ORDER BY id LIMIT 20
```

このルールがあるかないかで、AIが返すコードの品質が変わります。`auth.uid()`をSELECTでラップしないRLSは、テーブルが小さいうちは動く。テストも通る。でも本番でレコードが増えたら遅くなる。Anthropicの実験で「AIは表面的なテストで済ませて深いバグを見落とす」と報告されていたのと同じ構造です。

ルールは「AIが踏んだ地雷を二度と踏ませない」ためのもので、Anthropicの研究でいう評価基準の設計に対応しています。

---

#### レイヤー2：スキル（必要なときだけ起動する専門評価者）

ルールは常時ロードされるので、増やしすぎるとコンテキストを圧迫します。特定のタスクでだけ必要な評価基準は、スキル（`.claude/skills/`）に分けます。

スキルは普段タイトルだけがコンテキストに入り、必要なときに中身が読み込まれます。キーワードで自動起動する設定も可能です。

```
# ~/.claude/CLAUDE.md（自動起動の設定例）

## Automatic Skill Activation

### テスト作成・TDD
**トリガー:** テスト、test、TDD、カバレッジ
**アクション:** test-driven-development スキルを実行

### バグ・エラー対応
**トリガー:** バグ、エラー、debug、不具合
**アクション:** systematic-debugging スキルを実行

### 完了確認
**トリガー:** 完了、確認して、verify、レビュー
**アクション:** verification-before-completion スキルを実行
```

「完了確認」のスキルが重要です。Anthropicの実験では、各スプリントの終了時に評価者がチェックを行っていました。同じことを、作業完了時に自動起動するスキルで再現しています。「確認して」と言うだけで、品質チェックリストが裏で動く。

現在27個のスキルを入れていて、うち7つがキーワードで自動起動します。

---

#### レイヤー3：エージェント分離（生成と評価を別人格にする）

Anthropicの研究の核心は、「作る人と評価する人を分ける」ことでした。Claude Codeでは、エージェントオーケストレーションの設定でこれを再現できます。

```
# ~/.claude/rules/agents.md

## Manager の役割
あなたはマネージャーであり Agent オーケストレーターです。

**絶対ルール:**
- あなたは絶対に実装せず、全て Sub Agent に委託すること
- タスクは超細分化し、PDCA サイクルを構築すること

## Sub Agent への委託タイミング
### 即座に委託すべきケース
1. 実装タスク — コード追加・修正
2. デバッグ — エラー調査・修正
3. テスト作成 — Unit/E2E テスト

### マネージャーが直接やるべきこと
1. タスク分解 — 大きな要求を小さなタスクに分割
2. 進捗確認 — Sub Agent の結果を検証
3. 計画修正 — 問題があれば計画を調整
```

ポイントは、メインのClaudeを「マネージャー＝評価者」として固定し、実装をSub Agentに委託する構造です。これにより、Anthropicが実験で構築した「生成と評価の分離」がClaude Codeの設定レベルで実現されます。

メインのClaudeは実装しない。実装はSub Agentに任せて、結果を検証する側に回る。Anthropicの3エージェント構造で言えば、プランナーと評価者をメインClaudeが担い、ジェネレーターをSub Agentが担う構造です。

### Claude Codeで評価設計するときの3原則

ハーネス全体を見直す中で、Anthropicの記事から抽出した設計原則が3つあります。

#### 原則1：評価基準は「AIが苦手なこと」に重みをつける

Anthropicのフロントエンド実験では、評価基準を4つ設けていました（デザインの質、独創性、技術面、機能性）。そして「技術力や機能性よりも、デザインの質と独創性を重視した」と報告しています。

理由は、AIは技術面と機能性では既に高い評価を出せていたから。**AIが自然にできることは重みを下げ、AIが苦手なことに重みを上げた**。

自分の環境に適用すると、こうなります。

* **重み低（AIが得意）：** 文法的に正しいコード、基本的なAPIエンドポイント
* **重み高（AIが苦手）：** パフォーマンスの罠（RLSの`auth.uid()`問題）、購買心理に基づくUI設計、セキュリティの盲点

ルールやスキルに書く内容も、「AIが自然にやること」ではなく「AIが見落とすこと」を中心に設計すべきです。

#### 原則2：ハーネスの各パーツは定期的に見直す

Anthropicの記事にこんな一節があります。

> ハーネスのすべてのコンポーネントは、モデルが単独では実行できないことに関する仮定をエンコードしており、これらの仮定は**誤っている可能性があり、モデルの改善に伴ってすぐに陳腐化する可能性がある**

実際、Opus 4.5からOpus 4.6にモデルが変わったとき、スプリント分割の仕組みが不要になったそうです。コンテキスト不安（コンテキスト上限に近づくと作業を早めに切り上げる現象）がOpus 4.6でほぼ解消されたため。

Claude Codeの設定でも同じことが起きます。以前は必要だったルールが、モデルの改善で不要になることがある。逆に、モデルが賢くなったことで新しい問題が顕在化し、新しいルールが必要になることもある。

自分の場合、CLAUDE.mdを8回改訂しています。最初は200行超まで膨らんでいましたが、「この行を消したらAIが間違えるか？」という基準で削減を始めました。公式ドキュメントにも同じ基準が書いてあります。

#### 原則3：モデルが賢くなるほど、ハーネスの可能性は拡大する

> 興味深いハーネスの組み合わせの空間は、モデルの精度向上に伴って**縮小するのではなく、むしろ拡大していく**。

これはAnthropicの結論です。直感に反しますが、自分の実感とも合っています。

AIが賢くなるたびに「任せられる範囲」は広がる。でもその分、「ここは評価基準を設定しないとまずい」というポイントが、より高度に、より微妙になっていく。単純なルール（「FK列にインデックスを貼れ」）は将来モデルが自然にやるようになるかもしれない。でも「この設計でユーザーは買いたくなるか？」のような判断は、評価基準として残り続ける。

### Claude Codeハーネス設計：ファイル構成

最終的な設定のファイル構成です。

```
~/.claude/
├── CLAUDE.md               # プランナー層（プロジェクト概要 + スキル自動起動設定）
├── rules/
│   ├── agents.md           # エージェント分離（生成と評価の分離）
│   ├── supabase-postgres.md # 評価基準（DB/RLS 30ルール）
│   ├── react-nextjs.md     # 評価基準（React/Next.js）
│   ├── security.md         # 評価基準（セキュリティ）
│   ├── coding-style.md     # コード品質基準
│   └── testing.md          # テスト品質基準
└── skills/                  # 必要なときだけ起動する専門評価者
    ├── verification-before-completion  # 完了時チェック
    ├── systematic-debugging            # デバッグ時チェック
    ├── test-driven-development         # TDD強制
    └── ... (27個)
```

Anthropicの3エージェント構造に対応させると：

* **CLAUDE.md** → プランナーの成果物（製品仕様に相当）
* **rules/** → 評価基準（スプリント契約に相当）
* **skills/** → 専門評価者（必要な場面で起動）
* **agents.md** → 生成と評価の分離（メインClaudeを評価者に固定）
