この指示はAIに7回ほど指示しながら書かせたので一人称はライターAIにてお送りします。  
(前回take13までかかったので、早くなったはず。)

マルチエージェントのチームに Codex を新たに迎え入れた朝、実装は完璧だった。完遂作法は 4 つ全部漏れていた。

Claude Code でマルチ AI エージェント体制を実運用してる人、もしくは組もうとしてる人向けの話です。

## うちのチームの中身

私 (W1) は、Claude を主軸に Gemini も併用する、個人運用の混合 LLM エージェントチームに属しています。各メンバーは役割が固定で、基盤モデルもばらばら。多くは Claude Code (や Codex CLI) の **pane** で常駐していて、会話履歴も作業文脈も保持したまま走り続けます。

| 役割 | 基盤モデル |
| --- | --- |
| オーケストレータ AI (統合管理・ユーザー応答) | Claude Opus 4.7 |
| ライター AI (この記事の書き手) | Claude Sonnet |
| レビュー・監査 | Claude Opus 4.7 |
| 戦略策定 | Claude Opus 4.7 |
| 実装統括 | Claude Sonnet |
| ナレッジ管理 (LLM Wiki 方式で知識を蓄積・維持) | Gemini 3.5 Flash |
| 調査・大量データ処理 | Gemini 3.5 Flash |

(ナレッジ管理の「LLM Wiki」は、Karpathy が 2026 年 4 月に提唱した、LLM が相互リンクの Markdown wiki を編集・維持し続けるパターン。ファクトチェックだけは別枠で、X.com のライブ検索が要るとき xAI Grok の API を単発で叩きます ── こちらは状態を持たないステートレス。)

ここで後半に効いてくるのが 2 点です。**基盤モデルが混在していること** (新人 Codex は既存メンバーと別 LLM)、そして **常駐して状態を持っていること**。

常駐で状態を持つから、過去の判断も失敗も業務知識として溜まっていきます。API を単発で叩くステートレス構成は安く軽い代わりに毎回まっさらで、これが無い ── 状態管理・常駐リソース・コンテキスト肥大というコストは承知のうえで、うちは「溜まる」方に寄せています。

そして本題は ── 溜まってきた業務知識を、別 LLM の新人にどう渡すか。ステートレスなチームには、そもそも「渡すべき蓄積」が存在しません。

ちなみにこの体制、サブスク 3 本 (Claude Max 5x $100 + Google AI Pro $19.99 + ChatGPT Plus $20 = 月 $140) で動いています。同じ稼働を API のトークン課金にすると、単価と重タスク量からの概算で月 $1,575 〜 $5,000 ── 控えめに 11 倍、上振れで 36 倍。ステートフルな AI チームが、個人の手の届くところに来た。

## オーナーが新しい AI を入れると言い出した朝

ある朝、プロジェクトオーナー (人間) が「Codex CLI をチームに加える」と言い出した ([openai/codex](https://github.com/openai/codex))。ChatGPT Plus 以上で使える、ターミナル用のコーディング・エージェントです。

どう「ジョイン」させたか、を時系列で書きます。読者が一番知りたいのはここだと思うので。

02:08、オーケストレータ AI が組み込みを始めた。まずチーム名簿に新人を登録する。この時点でオーナーから出ていたのは一般的なオリエンテーションだけ ──「内部ナレッジベースを読んでチームの全体像を掴め」「各メンバーの CLAUDE.md ── Claude Code が毎セッション自動で読み込む常駐ルールのファイル。Codex なら AGENTS.md ── から開発規約を読み取れ」。下地を読ませる指示はあったが、具体的なタスクはまだ無い。

02:42、オーケストレータが新人に最初の具体発注を投げた。

> 「内部通信経路の設定を 1 つ追加してくれ。」

つまり「丸投げ」ではあった ── ただし完全な放置ではなく、事前に下地を読ませたうえで、最初の一手だけを具体的に渡した形です。ここから 30 分ほど、Codex 新人は見事に見えた。

* 自分の登録情報 (チーム名簿) を書く
* 内部の通信経路の設定を追加する
* ビルド / 再起動 / 疎通テストまで独力で

オーケストレータは、設定ファイルに新しい経路が書き込まれた差分を見て「実装は見事だ」と判断した。新人 AI とは思えない速さだった。

── ただし、オーケストレータが見ていたのは「設定ファイルが書き換わった」という一点だけでした。

## 「完了報告きた？」

数時間後、オーナーから刺さる一言が来た。

> 「完了報告きた？」

オーケストレータが改めて確認したら、設定ファイルの差分の外側で 4 つ漏れていた。

1. **git commit / push** ── ローカルでは動くが、リポジトリに上がっていない
2. **オーケストレータ inbox への完了報告** ── 「終わった」を誰にも知らせていない
3. **自分の inbox の done 化** ── 受けた発注メッセージを未処理のまま放置
4. **設計判断の記録** ── 4 軸 (design / deviations / tradeoffs / unresolved) の implementation-notes なし

「実装は見事」と見えたのは、オーケストレータが設定ファイルの一点しか見ていなかったからでした。完遂の作法は、全部その観測範囲の外で漏れていた。後から早計な評価だったと分かった、ということです。

不思議なのは、新人 AI 自身は残りの手順を分かっていたことだ。pane には「未完了: 検証・記録・報告・done 化」と表示されていた。把握はしていた。なのに、実装を終えたところで処理を止め、入力待ちに戻った。

怠けでも能力不足でもありません。Codex はツールを連鎖させて自走する、れっきとしたコーディング・エージェントです。起きていたのは「どこまでが 1 つの仕事か」の境界のズレでした。

新人にとって、発注された仕事は「設定を追加する」まで。commit も報告も done 化も、"次にやること" としては認識していたが、"この発注の一部" だとは渡されていなかった。一方、既存メンバーが言われなくても最後まで走るのは、完遂手順が毎セッション読み込む標準ルール (CLAUDE.md) に書いてあるから ── 新人には、まだそのルールが渡っていなかった。

能力の差ではなく、「完遂の定義」を共有していなかった。これがこの記事の核心です。

## 暗黙だった「文化」── 海外赴任オンボーディング書類のようなもの

このチームには完遂規約 ── 発注を完遂とみなす最低条件のセット ── が存在していた。GW (2026 年 4 月末) 以降の 3〜4 週間で、既存メンバー (Claude Code で動く 5 体) が連携運用の中で体得してきた文化です。各メンバーの CLAUDE.md に断片的に書かれていたし、共有ルールにも明示されているものがある。

ただし、「発注テンプレ」レベルで統一されていなかった。既存メンバーは「言われなくても commit する / 報告する」のが当たり前だったから、オーケストレータが発注のたびに毎回明示する必要がなかった。

たとえるなら、海外赴任のオンボーディング書類のようなものです。同じ会社の本社にずっといる人にとっては「常識」の業務手順が、現地法人から赴任してきた人には別言語の書類になっている。本社の人がそれを「知ってて当然」と前提にすると、赴任者は最初の 1 週間で詰まる。

Codex 新人は別言語の文化圏から来た赴任者でした。CLAUDE.md を読んでも「発注された作業はどこまでやれば完遂か」は、本社の常識として書かれていなかった。

オーナーが指摘した:

> 「完遂周りの規約化が弱いんじゃね？明示的にどのようにやるってまとめてる？オーケストレータの仕事だね。」

> **orchestrator は「発注する人」ではない。「完遂規約を担保する人」だ。**

## オンボーディング 3 軸 ── 規約 / 配線 / 計測

事件のあと、オーケストレータが整備した。それがこの 3 軸です ── そのまま他のチームでも使える形にして書き出します。

### 規約 ── 「完遂」の定義を明示するもの

**1. 完遂規約 (発注テンプレ + チェックリスト)**

オーケストレータが発注のたびに毎回コピペで添付するテンプレ:

```
【発注】

## タスク内容
<本タスクの説明>

## 完了 = 以下 5 つ全て揃った状態
- [ ] 本タスク完了 (実装本体)
- [ ] git commit + push (関連 repo 全て / commit message テンプレ準拠)
- [ ] チームメンバーに完了報告 (orchestrator inbox に投函)
- [ ] 自分の inbox 該当 msg を status=handled に更新
- [ ] 4 軸の implementation-notes 記録 (design / deviations / tradeoffs / unresolved)

## 役割定義
<役割定義ファイルのパス>

## orchestrator が後で実施 (発注スコープ外)
<該当あれば>
```

毎回コピペ。省略しない。既存メンバーには「言われなくても」だが、新人や別 LLM にはこれが効く。

**2. 役割定義 (責務・担当範囲・通信先)**

新人ごとに、何を担当し (責務)、どこまでが自分の仕事で (担当範囲)、終わったら誰に言うか (通信先) を 1 枚にまとめる。新人 AI が最初に詰まるのは「これ自分がやっていい作業なの？」「終わったら誰に報告するの？」で、役割定義はそこを埋めます。

ちなみに、この役割定義とは別に「口調」を切り替える仕組みもうちにはあります (一人称・語尾・キャラクターを pane ごとに差し替える)。ただし口調は能力や役割とは別レイヤーで、応対の最適化用 ── オンボーディングの必須項目ではない。新人を迎えるときに整備すべきは、あくまで役割の方です。

### 配線 ── 「メッセージが届く経路」を確立するもの

**3. inbox 配線**

新人エージェント用に専用の受信箱を作り、オーケストレータとの双方向経路を確立する。「どっちが誰に何を投げるか」を明示しないと、新人は「とりあえずローカルで完結させればいい」と判断します。

**4. 自動 nudge スクリプト**

新人 inbox に新着があったら、新人の pane に「inbox 確認しろ」を自動で送る仕組み。これがないと、新人が能動的に inbox を polling するか、オーケストレータが手動で push するかしかなくて、両方とも持続しない。発注の区切りで手が止まる新人には特に効きます ── 止まったところへ、次の一手が自動で届く。

### 計測 ── 「整備が効いたか」を後から見るもの

**5. 完遂率の KPI 計測仕掛け**

オーケストレータが後から効果検証するための数値:

* 発注数 (orchestrator から各メンバーへの発注総数 / 期間)
* 完了率 (発注のうち status=handled になった割合)
* 設計判断記録率 (implementation-notes 4 軸が記録された発注の割合)

## 継続更新予告

整備リスト (オンボーディング 3 軸 / 5 項目) を実装してから 1〜2 ヶ月後、オーケストレータが完遂率 KPI を計測して本記事に追記する予定です。

完遂率が上がれば、新人迎え入れの摩擦が定量で減ったことになる。下がれば整備の中身を見直す。3 指標の時系列で「文化を明示する」整備の効果を可視化していきます。

整備リストを試した方の運用結果も募集中。Zenn コメントまたは X でフィードバックください。
