こんにちは！[satto workspace](https://workspace.satto.me) で QA エンジニアをしている中井知子です。

今回、iOS アプリの E2E テストを、**Maestro** というツールと **AI エージェント（Claude Code）** の組み合わせで自動化しました。

最初にお伝えしたいのは、今回の取り組みが「大変だった話」ではないことです。むしろ逆で、**驚くほど苦労しませんでした**。

わたし自身はコードをほとんど書いていません。Claude Code のセッションを「Mobile 担当」として立ち上げ、環境構築からテスト実装、レビュー指摘への対応までを AI エージェントが担当。わたしが担ったのは、テスト対象の決定、レビュアーとのやりとり、そして「この挙動は問題ないか」といった品質判断です。

この記事でお伝えしたいのは、この**分業体制**と、その過程で見えてきた「**AI 時代のテストツール選定基準**」です。

## Maestroとは — 実機がなくても「本物」を動かせる

Maestro は、モバイルアプリの E2E 自動化ツールです。テストを YAML という読みやすい形式で書き、シミュレータ上のアプリを実際に操作してくれます。

わたしが一番価値を感じたのは、**iOS の実機がなくても、本物のモバイルアプリを PC 上で動かしてテストできる**ことです。

| プラットフォーム | 従来 | 本物を PC 上で動かせる？ |
| --- | --- | --- |
| デスクトップアプリ | PC に本物がいる | ✅ もともと可能 |
| Web | ブラウザに本物がいる | ✅ もともと可能 |
| モバイル | 実機が必要だった | ✅ シミュレータ＋Maestro で可能に |

モバイルだけが抱えていた「実機がないと本物を動かせない」というハードルが下がり、**デスクトップや Web と同じ「本物が手元にある」土俵で E2E 自動化を進められる**ようになりました。

今回はこの構成で、本物のアプリを本物のサーバーに接続したまま、録音→アップロード→議事録生成の実経路をそのまま動かしてテストしました。

## Mobile MCPとMaestro MCP — 名前は似ているが、対象が違う

この Maestro のテストを、AI エージェントとどう作ったのか。ここで、名前のよく似た 2 つの道具が登場します。**Mobile MCP** と **Maestro MCP** です。わたしも最初は違いがまったくわかりませんでした。

MCP とは、AI エージェントに道具を持たせる仕組みのことです。この 2 つは**担当している対象がまったく違います**。

| AI への頼みごと | Mobile MCP | Maestro MCP |
| --- | --- | --- |
| 「録音画面にどんなボタンがあるか調べて」 | ✅ 実際にアプリをタップして要素一覧を返す | ❌ アプリは触れない |
| 「長押し操作は Maestro でどう書くの？」 | ❌ Maestro のことは知らない | ✅ 公式ドキュメントから正しい書き方を答える |
| 「書いたテストの構文チェックして」 | ❌ Maestro のことは知らない | ✅ 構文チェックを実行する |

* **Mobile MCP** ＝ AI の目と手（スマホの操作担当）
* **Maestro MCP** ＝ AI が持つ Maestro の取扱説明書

どちらか一方を選ぶものではなく、**2 つ揃って初めて「調べる→書く→検証する」が AI の中で完結します**。なお、書き上がったテストの実行そのものは Maestro 本体（CLI）の仕事なので、CI で回す段階では MCP は不要です。

### 実際のテスト作成：「探索」と「定着」の分業

**① Mobile MCP で探索する。** AI がアプリを実際に操作しながら、画面ごとの要素一覧を作ります。

```
[
  {"type":"StaticText","label":"00:00:05"},
  {"type":"StaticText","label":"録音中"},
  {"type":"Button","label":"一時停止"},
  {"type":"Button","label":"終了"}
]
```

**② Maestro からの「見え方」を裏取りする。** 実際、ホーム画面の録音中カードにある停止ボタンはどちらのツールからも見えず、「この操作は自動化できない。アプリ側に testID の付与依頼が必要」と**テストを書き始める前に判明**しました。探索の段階で自動化できる範囲が確定するのです。

**③ Maestro YAML に定着させる。** 探索で得た操作手順を、テストとして書き起こします。

```
- tapOn: "一時停止"
- extendedWaitUntil:
    visible:
      text: "一時停止中"
    timeout: 10000
- assertVisible: "再開"
```

この流れで第 1 弾のテストスイートを実装し、クリーンな環境で**すべてパス**しました。全テストが実アプリ×実サーバー、モックなしでの結果です。

ところでこの YAML、プログラミングの知識がなくても「一時停止をタップして、一時停止中の表示を待って、再開ボタンがあることを確認する」と読めるのではないでしょうか。ノーコードツールが GUI で実現している「誰でも読めるテスト」を、Maestro はテキストのまま実現しています。テキストだから、**組み立てるのも、UI の変更に合わせて直すのも、AI に任せられる**。

この「読める」は、のちほどツール選定の話で効いてきます。

## Appiumとの比較 — 機能表ではAppiumが勝っていた

モバイルのテスト自動化ツールといえば、老舗の **Appium** が有名です。わたしたちのチームでも当初は Appium 案が有力でした。

* デスクトップ E2E 自動化（Playwright/TypeScript）と知見を共通化できる
* 好きなプログラミング言語で書ける
* 複雑な条件分岐やロジックが書ける
* エコシステムが豊富

機能の比較表を作ると、勝つ項目は Appium のほうが多かったのです。でも、この比較表をよく見ると、あることに気づきます。**どの項目も「人間がコードを書き続ける前提」の評価軸**なのです。

## AIがテストを書く時代、選定基準が変わる

チーム内でこんな議論がありました。

> AI エージェントがいなかった頃は「複雑なロジックが実装しやすいか」がツール選定の重要ポイントだった。でも今は、複雑な実装は AI がなんとかしてくれる。だからそこはもう、選定の本質ではなくなった。

「書きやすさ」は AI が吸収してくれる。では代わりに何が重要になるのか。2 つあると考えています。

**1 つ目：生成から検証までが AI の中で完結するか**

AI が生成したテストを人間が全件レビューする運用にはしていません。実際の流れはこうです。

1. AI が書く（Mobile MCP で探索 → YAML 作成）
2. AI が検証する（Maestro MCP で構文チェック → 実行して裏取り）
3. 人間は要約された結果を確認して判断する（「すべてパス。ただし 1 本だけ間欠的に失敗する挙動があり、原因はこう」）

この**自走ループが成立するツールチェーンかどうか**が、まず効いてきます。

**2 つ目：人間が要所を素早く確認できるか**

自走ループがあっても、最後の品質判断は人間の仕事です。「このテストは本当に確認すべきことを確認しているのか」を見たいとき、Maestro の YAML なら数十秒で読めます。Appium+TypeScript のコードでは、相応のプログラミング知識と時間が必要です。全件レビューしないからこそ、**見たいときにすぐ見られる**ことの価値が上がります。

|  | AI 以前の基準 | AI 時代の基準 |
| --- | --- | --- |
| 重視すること | 人間にとっての書きやすさ・柔軟性 | 生成→検証が AI で完結するか＋人間が要所を素早く確認できるか |
| 複雑なロジック | 書けるほど良い | AI が吸収するので差がつかない |
| 記述形式 | 自由度が高いほど良い | **制約が強いほど AI の検証も人間の確認もしやすい** |

YAML の「複雑なことが書けない」という制約は、AI 以前の基準では弱点でした。AI 時代には、**AI の出力を安全に受け取るための強み**に反転するのです。

## エラー系などは「モック」を使う

ここまで「モックなしで本物を動かした」という話をしてきましたが、モックが要らなくなったわけではありません。**デスクトップ、Web、モバイル——どのプラットフォームの E2E 自動化でも、エラー系（異常系）の検証にはモックを使います**。実サーバーをわざとエラーにするわけにはいかないからです。違うのは、モックの登場回数と使いどころです。

今回の satto アプリで、どこを実経路のまま通せて、どこがモックになったかを並べるとこうなります。

|  | UI・画面遷移 | 録音・権限などの OS 機能 | 異常系（エラー） |
| --- | --- | --- | --- |
| satto モバイルアプリ | 実経路 | シミュレータで動くので**実経路** | モック |
| satto デスクトップアプリ | 実経路 | テスト環境で動かせず**モック** | モック |

大事なのは、**この差は Maestro や Playwright といったツールの差ではなく、製品側の作り込みによるもの**だということです。奥をどこまで本物で通せるかは、その製品がテスト環境で実物をどこまで動かせる作りかで決まります。**自動化しやすい作り込み方と、しにくい作り込み方がある**——今回それを実感しました。

なお、モック部分のテストが成功しても、それは「代替品が想定どおり応答した」ことの確認にすぎません。奥の実際の処理は別途確かめる必要があり、「**そのテストは何を保証したいのか**」で使い分けることになります。

裏を返すと、**モックなしで正常系を自動化できる作りは、使い分けが要らないぶん自動化しやすく、品質も担保しやすい**。今回のモバイルはその状態に恵まれていました。テストのしやすさは、製品の作り込みの段階から始まっているのだと感じます。

## それでも「手動テストはいらない」にはならない

「E2E を自動化しました」という記事を読むと、「もう手動テストって不要なのでは」と感じるかもしれません。でも実際に設計してみると、そんなことはまったくありませんでした。

今回の iOS E2E は、リグレッションシナリオ 19 本のうち、Maestro で自動化すると設計判断したのが 14 本（第 1 弾から順次実装中）。**残り 5 本（約 26%）は、設計の段階から手動と判断しています**（2026 年 7 月時点の設計）。手動 5 本の内訳には共通点があります。

* 旧バージョンをインストールした状態からの強制アップデート確認
* ネットワーク断・MDM（会社の端末管理）・ロック画面などの例外系
* アプリ未インストールの初期状態からの初回起動・認証
* 3 時間録音のような長時間シナリオ

自動化が難しいのは「操作」ではなく「**テストの前提条件を作ること**」なんです。旧バージョンが入った端末、MDM 管理下の環境、ロック画面。こうした「状況」はシミュレータでは再現できず、ツールがどれだけ進化しても自動では作れません。

自動化は手動テストをなくすものではなく、**手動テストを「人間にしかできない領域」に集中させるもの**——今回の設計を通じて、その実感が強まりました。

## まとめ

* Maestro とシミュレータの組み合わせで、モバイルも「**本物が手元にある**」土俵で E2E 自動化を進められるようになった
* AI がテストを書く時代、ツール選定の基準は「人間が書きやすいか」から「**生成→検証が AI の中で完結し、人間が要所を素早く確認できるか**」へ変わる
* どのプラットフォームでもエラー系の検証にはモックを使う。「**そのテストは何を保証したいのか**」でアプローチを使い分ける
* モックが要る／要らないはツールの差ではなく**製品の作り込みの差**。自動化しやすい作りは、そのまま品質を担保しやすい
* テストの前提条件を作る領域は自動化できない。**手動テストはなくならず、人間にしかできない領域に集中していく**

テスト自動化ツールを検討する際は、比較表で機能の数を数える前に、「そのコードは誰が書いて、誰が検証して、人間はどこを見るのか」を考えてみてください。答えが「AI が書いて、AI が検証し、人間は要所を確認する」なら、選ぶべきツールは変わってくるかもしれません。
