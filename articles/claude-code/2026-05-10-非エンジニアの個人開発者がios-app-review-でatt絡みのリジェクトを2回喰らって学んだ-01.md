# はじめに

こんにちは、takeMiyaです。

エンジニアではない立場で、Flutterを使って個人開発に挑戦しています。  
先日（2026年4月）ボードゲームアプリ「TicTacToe GO」を App Store と Google Play でリリースしました。  
アプリには Google AdMob のインタースティシャル広告を組み込んでいます。

このアプリを iOS で公開するまでに、Apple の App Review で **2回リジェクト** されました。どちらも「ATT（App Tracking Transparency）」絡みの指摘です。

正直、最初のリジェクトメールが届いたときは「ATTのコードなんて書いていないのに、なぜ？」と頭を抱えました。  
2回目は「英語の説明文は書いたはずなのに、なぜまた？」と二度詰まりました。

この記事では、

* なぜリジェクトされたのか
* どう対応して、3回目で通過できたのか
* 最初から知っていれば避けられた落とし穴

を、非エンジニア目線でできるだけわかりやすく整理します。

これから iOS アプリを App Store に申請する個人開発者の方、  
特に **AdMob を使うアプリを作っている方** にとって、申請前のチェック資料として使ってもらえる内容を目指しました。

同じく  
非エンジニアでアプリ開発に挑戦している方にも、「自分にもできそう」と感じてもらえたら嬉しいです。

---

# 結論：3行でわかる対応ポイント

時間がない方のために、結論を先にまとめます。

1. **AdMob を使うなら ATT 実装は必須**  
   自分で ATT のコードを書いていなくても、AdMob SDK が ATT フレームワークを  
   自動的に参照するため、Apple のレビューで「ATT 対応が必要なアプリ」として  
   扱われます。
2. **`NSUserTrackingUsageDescription` は対応言語すべてに用意する**  
   `Info.plist` に英語で書くだけでは不十分。日本語など他言語に対応している  
   アプリは、`InfoPlist.strings` を言語ごとに分けて、各言語版に  
   `NSUserTrackingUsageDescription` の翻訳を入れる必要があります。
3. **1回目の申請でこの2つをセットでやれば、リジェクトは避けられる**  
   私は1つずつ対応してそれぞれリジェクトされましたが、最初から両方を  
   やっていれば、リジェクト2回・追加で約1週間のロスは防げました。

詳しい経緯と実装手順は、このあと順を追って説明します。

# 1回目のリジェクト：ATTプロンプトが出ていない

## Apple からの指摘内容

App Store Connect から届いたリジェクトメールの該当箇所は、ガイドライン番号  
**`Guideline 2.1 - Information Needed`** でした。

要旨は以下の通りです。

> ATTフレームワークを使用しているにも関わらず、ATTのアクセス許可要求が  
> 見つかりませんでした。

シンプルで具体的な指摘ですが、当時の私には心当たりがまったくありませんでした。

## 心当たりがなかった理由

「TicTacToe GO」はパズル系のボードゲームアプリです。プレイヤーの行動を  
分析して何かに使うつもりはなく、**ユーザートラッキングなどするつもりは  
そもそも無かった**のです。

ATT 関連のコードも書いていません。`Info.plist` に  
`NSUserTrackingUsageDescription` を追記した記憶もありません。

「ATT フレームワークを使っている」と言われても、自分では使っている認識がない状態。これが最初に頭を抱えた理由でした。

## 落とし穴：AdMob SDK が ATT を自動的に呼ぶ

原因にたどり着いたのは、AI とのやり取りの中ででした。

リジェクトメールの内容を共有して原因を尋ねた際、**「AdMob SDK が内部でATT フレームワークを参照しているため、ATT の対応がアプリ側で必要になる」** という指摘を受けました。

この説明を聞いた瞬間、ずっと前に読んだ記憶が蘇りました。  
Apple がプライバシー強化の一環として ATT を導入したというニュース。当時は一般的なビジネスニュースとして読み流していました。実際の開発現場にどう影響があるか、までは理解していなかったのです。

整理するとこういう構造です。

* 自分のコードでは ATT を呼んでいない
* でも `google_mobile_ads`（バージョン 5.1.0）を導入している
* AdMob SDK は内部で IDFA（広告識別子）にアクセスする可能性がある
* そのため AdMob SDK は ATT フレームワークを自動的にリンクする
* Apple のレビューはこのリンクを検知して「ATT 対応必須」と判定する

つまり、**広告 SDK を入れた時点で、自分は ATT 対応の責任を負うアプリの  
開発者になっていた**ということです。

## 対応：app\_tracking\_transparency パッケージの導入

Flutter で ATT プロンプトを出すには、`app_tracking_transparency` パッケージを使うのが定番のようです。  
バージョン `^2.0.6` を導入しました。

`pubspec.yaml`：

```
dependencies:
  app_tracking_transparency: ^2.0.6
```

実装は `lib/main.dart` に書きました。  
**AdMob の初期化よりも前に ATT のリクエストを呼ぶ**のがポイントです。  
順番が逆だと、トラッキング許可が下りる前に AdMob が動き出してしまうことになります。

```
import 'package:flutter/material.dart';
import 'package:app_tracking_transparency/app_tracking_transparency.dart';
import 'package:google_mobile_ads/google_mobile_ads.dart';

Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();

  // ATT のリクエストを AdMob 初期化前に行う
  await _requestTrackingAuthorization();

  // AdMob の初期化はその後
  await MobileAds.instance.initialize();

  runApp(const MyApp());
}

Future<void> _requestTrackingAuthorization() async {
  final status = await AppTrackingTransparency.trackingAuthorizationStatus;
  if (status == TrackingStatus.notDetermined) {
    await AppTrackingTransparency.requestTrackingAuthorization();
  }
}
```

`Info.plist` には `NSUserTrackingUsageDescription` を追記しました。  
これがないと、そもそも ATT のプロンプトを表示できません。

```
<key>NSUserTrackingUsageDescription</key>
<string>Used to show you more relevant ads.</string>
```

ここまでを修正して、**2回目の申請**に出しました。  
これで通るはず、という気持ちでした。

## このセクションで得た学び

リジェクト1回目で得た学びは、技術的な対応方法そのものよりも、  
**「広告 SDK を入れる = プライバシーポリシーの責任を負う」という構造の  
理解**でした。

AdMob は「導入する」だけで動く便利なツールに見えますが、その裏では  
Apple のプライバシーポリシーと密接に絡み合っています。盲目的に導入するのではなく、**ATT などトラッキングの仕組みと、それに対する Apple の方針を周辺ごと理解しておく**必要がある、と痛感しました。

幸い、AI とのやり取り（いわゆるバイブコーディング）を通じて、調べる →  
理解する → 実装するというサイクルを高速に回せたのは、非エンジニアの  
個人開発者として大きな助けになりました。リジェクト体験そのものが、  
理解を深める良い機会になったとも言えます。

# 2回目のリジェクト：ローカライズが足りない

## Apple からの指摘内容

ATT 実装と `Info.plist` への `NSUserTrackingUsageDescription` 追記を済ませて、2回目の申請に出しました。  
今度こそ通るだろうと思っていたところ、再びリジェクトメールが届きます。

該当箇所のガイドライン番号は **`Guideline 4 - Design`**。  
1回目とは別のカテゴリでした。

要旨は以下の通りです。

> アプリの権限要求が、アプリのローカライズ言語と一致していません。

具体的には、**日本語環境のデバイスで動作確認した際に、英語の権限要求文が表示されてしまった**という指摘でした。

## 何が起きていたか

1回目のリジェクトを受けて、私は `Info.plist` に英語で `NSUserTrackingUsageDescription`  
を書きました。これで「説明文は書いた」つもりになっていました。

しかし TicTacToe GO は、App Store のローカライズ対応として **英語と日本語**の2言語に対応しています。  
アプリ全体は両言語に対応しているのに、  
**ATT の権限要求文だけ英語固定**という状態だったのです。

これでは日本語環境のユーザーが、ATT のプロンプトを見たときに英語の  
説明文を読まされることになります。  
Apple が指摘したのはこの不整合でした。

整理するとこうなります。

* アプリの対応言語:英語・日本語
* ATT の権限要求文:英語のみ
* 結果:日本語環境では英語の説明文が表示される
* → ローカライズの整合性が取れていないため、ガイドライン違反

## アプリ設計の前提と、今回の方針転換

ここで少し背景を書きます。  
TicTacToe GO は**アプリのコンセプト・基礎設計の時点で、なるべく特定の言語に依存しないつくり**を目指していました。

ボードゲームというシンプルな題材であることもあり、画面内のテキストを  
最小限にして、アイコンや色で意味を伝える設計を心がけていました。  
ローカライズの手間を減らすこと、そして将来的に多言語展開する際の  
コストを抑えること。  
これが当初の方針でした。

ところが、今回のリジェクトで **「アプリ本体のテキストは絞れても、OS が表示する権限要求のような外側の領域では、ローカライズから逃げられない」**  
ことを実感しました。

そこで方針を整理し直しました。

* 完全に特定言語非依存のアプリにすることは諦める
* ただし対応する言語は **英語と日本語の2種類** に絞る
* これを「最低限のポリシー」として運用していく

この2言語に絞った設計が、今後どういう効果を発揮するか（サポート負荷・ユーザー満足度・展開のしやすさなど）は検証中です。  
結果が見えてきたら、別の記事で改めて紹介する予定です。

## 対応：InfoPlist.strings の言語別配置

ローカライズされた権限要求文を表示するには、`InfoPlist.strings` というファイルを言語ごとに用意します。  
配置は以下の通りです。

```
ios/Runner/
├── en.lproj/
│   └── InfoPlist.strings
└── ja.lproj/
    └── InfoPlist.strings
```

中身はそれぞれ次のようにしました。

`en.lproj/InfoPlist.strings`：

```
NSUserTrackingUsageDescription = "Used to show you more relevant ads.";
```

`ja.lproj/InfoPlist.strings`：

```
NSUserTrackingUsageDescription = "より関連性の高い広告を表示するために使用されます。";
```

ファイルの作成と配置は Claude Code に任せました。  
Xcode を直接いじらずとも、ファイル構造を理解した AI に指示すれば、適切な場所に正しい形式で配置してもらえます。  
非エンジニアにとっては、こうした「Xcode の作法」を AI に肩代わりしてもらえることの恩恵は大きいと感じます。

## 動作確認：iPhone の言語設定を切り替える

実装後、ローカライズが効いているかを実機で確認しました。

私の iPhone は普段から英語設定で使っていたため、英語環境では問題なく  
英語の文言が表示されます。問題は日本語環境でした。

そこで iPhone の言語設定を一時的に日本語に切り替え、アプリを再起動して  
ATT プロンプトを再表示。日本語の説明文が表示されることを確認しました。

```
設定 → 一般 → 言語と地域 → iPhoneの使用言語 → 日本語
```

普段使う設定ではないので少し手間ですが、**多言語対応するアプリでは  
すべての対応言語で実機確認するのが鉄則**です。  
1回目のように「コードは書いたから大丈夫」と思い込まず、実際にユーザーが見る画面を自分の目で確認することの重要性を改めて感じました。

## このセクションで得た学び

2回目のリジェクトで得た最大の学びは、**アプリの設計思想と、OS が要求するローカライズの責任は別物だ**ということです。

「言語非依存のアプリにしよう」という設計判断は、アプリ内部のコンテンツには適用できます。  
しかし、OS が表示する権限要求のような領域は、開発者の設計思想ではなくプラットフォームのポリシーに従う必要があります。

この線引きを理解した上で、TicTacToe GO は**英語と日本語の2言語対応**  
というラインを最低限のポリシーとして固めました。

多言語展開のコストと、  
プラットフォーム要件への対応のバランスを、初めて自分ごととして考えたタイミングでもあります。

# 3回目で通過

ATT 実装に加えて、`InfoPlist.strings` のローカライズ対応を済ませた状態で、3回目の申請に出しました。

提出は金曜の朝8時（日本時間）。すぐに審査が始まるとは限らないため、  
おそらく週末をまたぐだろうと予想していました。

案の定、土日には音沙汰がありません。来ないだろうと思いつつ、週末は何度もメールチェックをしてしまいました。

承認の通知が届いたのは、火曜の朝9時(日本時間)。  
レビュアーからの  
特記事項はなく、  
件名は **「Welcome to the App Store」** でした。

この一文を見て、ほっと胸をなでおろしました。ようやくスタートラインに  
立った、という気持ちです。

## 個人開発で初めて経験したこと

仕事ではこれまで、アプリの App Review 通過を何度か経験しています。  
ただ、個人開発で自分のアプリを通すというのは、**仕事のときとは明らかに気持ちが違う**経験でした。

特に、申請前に頭をよぎっていた懸念がいくつかあります。

* これは新規の Apple Developer アカウントである
* 過去のリリース実績がない
* 個人名義（ブランド名義）での申請である

これらの条件が重なると、「新規アカウントは厳しめに見られる」「実績が  
ないアカウントは指摘が多くなる」といった噂を耳にしていたこともあり、  
**もっと多くの指摘が来ることを覚悟していました**。

ところが、実際に来た指摘は2回。  
しかも内容はどちらも明確で、再現性のある技術的な不備に対するものでした。  
**企業アカウントかどうかに関わらず、基準に照らしてフラットに審査されている**という印象を持ちました。

これは、これから個人開発で App Store に申請する人にとっては、ある種の安心材料になるかもしれません。  
新規アカウントだから不利、ということは、少なくとも私のケースでは感じませんでした。

# 振り返り：最初からやるべきだった3つのこと

2回のリジェクトを経て、3回目で App Store 通過にたどり着きました。  
振り返ってみると、技術的な実装そのものよりも、 **「何を前提として  
受け入れて開発に臨むか」** という設計思想の段階で見落としていたことが  
原因だったと思います。

これから iOS で個人開発を始める方に向けて、私が最初から知っていたら  
2回のリジェクトを避けられたであろう3つのことを共有します。

## 1. 広告 SDK を入れる = プライバシー対応の責任を負う

**原則**：AdMob のような広告 SDK を組み込んだ瞬間に、自分のアプリは  
「ユーザーの行動データに関わるアプリ」として Apple の審査対象になります。  
自分のコードでトラッキングを一切書いていなくても、SDK が内部で広告  
識別子（IDFA）にアクセスする可能性があるため、ATT 対応はアプリ側の  
責任として求められます。

**私の失敗**：「パズルアプリだからトラッキングはしない」と思っていたため、  
ATT を完全に自分の関心の外に置いていました。AdMob を導入した時点で、  
自分が「ATT 対応が必要な開発者」になっていたことに、リジェクトされる  
まで気付けませんでした。

**これから始める人へ**：広告 SDK を pubspec.yaml に追加するときは、  
**ATT 実装と permission string の追加もセット**として最初から組み込んで  
ください。「広告を入れる」は単なる収益化の判断ではなく、プライバシー  
対応の枠組みに入る判断です。

## 2. アプリ設計の言語非依存と、OS 要求のローカライズは別物

**原則**：アプリ内のコンテンツを言語非依存に設計することと、OS が表示する権限要求文（ATT プロンプトなど）をローカライズすることは、まったく別のレイヤーの話です。  
前者は開発者の自由ですが、後者はプラットフォームの要件です。

**私の失敗**：TicTacToe GO は「特定言語に依存しないアプリ」を設計思想としていました。  
アイコンや色で意味を伝え、テキストを最小限にする方針です。  
その思想を OS 領域にも無意識に持ち込んでしまい、ATT の説明文を英語  
1本で済ませた結果、日本語環境のユーザーには英語の説明文が表示されてしまいました。

**これから始める人へ**：アプリの対応言語を決めた瞬間に、その言語すべてにpermission string を用意してください。  
多言語対応とは「アプリ内の翻訳」だけではなく、**OS が代わりに表示してくれる文言まで含めた整合性**のことです。  
私は最終的に、英語と日本語の2言語を「最低限のポリシー」として固めました。

## 3. 申請前の動作確認は、対応言語すべてで行う

**原則**：実装が正しくても、対応言語の数だけ確認すべき画面パターンが  
あります。1言語だけの動作確認では、他言語環境での不整合は見つけられ  
ません。

**私の失敗**：私の iPhone は普段から英語設定でした。実装後、英語環境でATT プロンプトが表示されることを確認して「OK」としてしまいました。  
日本語環境でどう見えるかを実機で確認していれば、2回目のリジェクトは  
申請前に防げたはずです。

**これから始める人へ**：実機の言語設定を切り替えて、対応言語すべてで  
ATT プロンプトを目視確認してください。
