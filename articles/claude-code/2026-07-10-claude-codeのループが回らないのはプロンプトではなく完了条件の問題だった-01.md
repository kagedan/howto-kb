## はじめに

先日、Claude Codeに自律作業を任せてログを見返していたら、ぞっとする光景がありました。ターン終了をブロックするhookが、2分間に同じフィードバックを6連発していたんです。Claude Codeは毎回「修正しました」と言い、hookは毎回同じ理由で差し戻し、リポジトリの状態は1バイトも変わっていない。完全な無限ループでした。

こんな経験ありませんか？

* 自律ループを組んだのに、「できました」と言うだけで実際は何もできていない
* 逆に、いつまでも終わらずに同じ修正を繰り返している
* プロンプトを強い言葉に書き直しても、どっちの症状も直らない

僕も最初は「指示の書き方が甘いんだろう」と思って、プロンプトを何度も書き直していました。でも実測してみて分かった結論は違いました。ループが回らないのは、プロンプトの問題ではなく、完了条件を決定論的か非決定論的かで分類できていない問題だったんです。

気づくまでに事件を2つ踏んだので、その記録と、たどり着いた分類をメモしておきます。

## 完了条件をhookで強制してみた

前提として、僕はClaude Codeのサブエージェント委任に「検証可能な完了条件」を必須にしています。完了条件のない委任は、「できたように見える」以外の合否信号を持てないからです。

仕組みは単純で、PreToolUseフックでAgentツールの呼び出しを検査し、委任プロンプトに `<success_criteria>` ブロックがなければdenyします。要点だけ抜き出すとこうです。

```
# PreToolUse(Agent) で委任プロンプトを検査する
input=$(cat)
prompt=$(printf '%s' "$input" | jq -r '.tool_input.prompt // ""')

if ! printf '%s' "$prompt" | grep -q '<success_criteria>'; then
  jq -n '{hookSpecificOutput: {
    hookEventName: "PreToolUse",
    permissionDecision: "deny",
    permissionDecisionReason: "完了条件がありません。success_criteria ブロックに検証可能な合否条件とエビデンス要求を書いてください。"
  }}'
  exit 0
fi
```

denyの理由文がそのままClaude Codeへのフィードバックになるので、次の呼び出しでは完了条件付きのプロンプトが来ます。ここまでは順調でした。順調に見えていました。

## 事件1: XMLだけ書いて、中身はノリ

hookを入れてしばらくして、委任ログを読み返した僕は勘違いに気づきました。タグは確かに毎回書かれている。でも中身がこうなっていたんです。

```
<success_criteria>
- タスクが正しく完了していること
- 品質が十分であること
</success_criteria>
```

これ、何も言っていないんですよね。「正しく」も「十分」も検証手段がないので、サブエージェントは結局「できたように見える」を合否信号にするしかない。僕はタグの存在をチェックして満足していましたが、形式を強制しても意味は強制できていませんでした。

ムキになった僕は、対策として語彙lintを書きました。「exit」「テスト」「コマンド」みたいな検証語彙が中身に含まれていなければdenyする、という追加チェックです。

そして数日後、この語彙lintは自分で撤去しました。「`ls /` がexit 0であること」と書けば素通りできる床の上で、語彙の検査だけ厳しくしても防御になっていないからです。今のhookに残っているのは「空白除去後20文字以上」という形の床だけで、中身が空虚かどうかの判定は、委任の書き方を定めた知識側のskillとレビューに寄せました。hookで守れる線と守れない線がある、というのが最初の学びです。

## 事件2: 止まらないループ

2つ目の事件が、冒頭の無限ループです。

僕はターン終了時のStop hookに検証ゲートを仕込んでいます。プロジェクトの検証コマンドがFAILしたら `decision: block` でターン終了を差し戻し、通るまで「完了」と言わせない仕組みです。事件はこのゲートと、達成不能な完了条件の組み合わせで起きました。

条件が満たせないとき、Claude Codeは修正を試みて、失敗して、ブロックされて、また同じ修正を試みます。リポジトリの状態も検証の失敗出力もまったく同一のまま、同じフィードバックが2分間に6連発。進捗ゼロの空転にAPIコストだけが溶けていく実況を見て、正直ムカつきました。書いたのは僕なんですが。

対策は2段です。まずhook側に無進捗のサーキットブレーカーを入れました。git状態と失敗出力が前回FAILとバイト同一なら「無進捗」と判定し、2回目で「これ以上リトライせず、未検証・失敗として報告して終われ」とエスカレーションし、3回目以降はブロックせずに解放します。進捗のある失敗は粘らせて、何も変わらない再ブロックだけを打ち切る、という線引きです。

もう1つは条件側で、自律ループの条件文に上限句を必須にしました。

```
/goal src/auth のリファクタ完了 — npm test が exit 0、
公開 API のシグネチャは不変、or stop after 15 turns
```

この「or stop after 15 turns」がないと、条件が達成不能だったときの脱出口がありません。上限句のない条件を検出したら注意を注入するhookも足しました。

## 気づき: 完了条件は2種類ある

2つの事件を並べて眺めていて、ようやく気づきました。事件1は「検証をhookで判定できない条件を、hookで守ろうとした」失敗で、事件2は「達成できるか分からない条件を、達成型のゲートで縛った」失敗です。つまりどちらも、完了条件の性質を見ずに同じ強制のかけ方をしたのが原因でした。

見出しの「2種類」は、検証が決定論的か非決定論的か、という一番大きな分かれ目のことです。前者はコマンドのexit codeで合否を判定でき、後者はできません。ここに「ゴールを事前に明示できるか」というもう1つの軸を掛け合わせると、完了条件は次の3つの型に落ちます。

| 型 | 検証 | ゴール | 完了の定義 | 例 |
| --- | --- | --- | --- | --- |
| 到達型 | 決定論的 | 事前に明示できる | コマンドがexit 0 | テストを通す、ビルドを直す |
| 収束型 | 決定論的 | 総数が事前に不明 | 井戸の枯渇 + 上限句 | バグの洗い出し、移行漏れ検出 |
| rubric+judge型 | 非決定論的 | rubricで書ける | judgeが改善点を挙げられなくなる | ドキュメント品質、設計の良さ |

順番に見ていきます。

## 型1: 到達型

検証が決定論的で、ゴールも明示できる型です。「npm testがexit 0」「このファイルが存在する」のように、条件文がそのまま実行可能なコマンドになります。

この型の強制は簡単で、Stop hookに検証スクリプトを1本置くだけです。ちなみにこの記事を書いているリポジトリにも入っていて、中身は10行くらいしかありません。

```
#!/bin/bash
# stop-gate: exit 0 = PASS(ターン終了許可) / 非0 = FAIL(継続を強制)
set -o pipefail
cd "$(dirname "$0")/.." || exit 0
ARTICLE="articles/467b1233efe811.md"
if [ ! -f "$ARTICLE" ]; then
  echo "FAIL: $ARTICLE が存在しない" >&2; exit 1
fi
python3 .agents/skills/zenn-gemcook-article-writer/scripts/validate_article.py "$ARTICLE"
```

記事のfrontmatterや禁止表現を機械判定するvalidatorが通るまで、Claude Codeはターンを終われません。到達型に落ちるタスクは、プロンプトで「必ずテストを通してください」と念押しするより、この10行の方がはるかに強いです。

## 型2: 収束型

検証は決定論的だけど、ゴールの総数が事前に分からない型です。バグの洗い出しや移行漏れの検出がここに入ります。「バグを全部見つけたらexit 0」というコマンドは書けません。全部が何個か誰も知らないからです。

この型の完了は、到達ではなく井戸の枯渇で定義します。「2回連続で新規発見がゼロなら完了」のように、探索を続けても何も出てこなくなった状態を完了とみなす書き方です。

そして事件2の教訓どおり、収束型には上限句が必須です。井戸が枯れない（発見が出続ける、あるいは同じ発見を繰り返す）ケースがありえるので、「or stop after 20 turns」を付けて脱出口を確保します。僕は上限句のない収束型の条件は書かない、と決めてからループの空転がなくなりました。

## 型3: rubric+judge型

一番厄介なのが、検証が非決定論的な型です。「ドキュメントが分かりやすい」「設計が良い」は、exit codeでは判定できません。事件1で僕がhookに守らせようとして失敗したのは、まさにこの領域でした。

この型は3つの部品で書きます。

1. 決定論的な必要条件は列挙する
   * リンク切れゼロ、コード例が実行可能、などhookで判定できる床は普通に書く
2. 十分条件は形容詞ではなく、読者・利用者の行動で書く
   * 「分かりやすいこと」ではなく「初見の読者が30秒で読み始め位置を判断できること」
3. 判定は書いた本人ではなく、独立したjudgeエージェントにやらせる
   * judgeには成果物だけを渡し、実装の経緯や会話履歴は渡さない

3つ目が特に効きます。書いた本人に採点させると「できたように見える」しか信号が出ないので、執筆文脈を持たないレビュアーを分離するわけです。実際この記事も、下書き完成後にレビュー専用のサブエージェントへ記事ファイルのパスだけを渡して、PASSが出るまで直しています。

合否の閾値は「judgeが80点を付けたら合格」より、「judgeが改善点を挙げられなくなったら完了」の収束条件にした方がjudgeの採点ブレに頑健でした。気づけばこれ、型2の井戸の枯渇と同じ形なんですよね。非決定論的な検証は、収束型に変換して初めてループの終了条件になる、と理解しています。

## どの型にも落ちないタスクは委任しない

最後に、3つの型のどれにも落ちないタスクがあります。検証も非決定論的で、rubricすら書けない。「いい感じの設計を考えておいて」みたいな依頼です。

僕の答えは、そのまま自律委任しないです。無理に到達型のふりをした条件を書くと、事件1の空虚な条件に戻ります。代わりに次のどちらかに倒します。

* 成果物を「比較可能な複数案 + 人がレビュー可能な状態」に再定義して委任する
* 先にrubric作り自体をタスクにして、3型のどれかに落ちてから委任する

**完了条件が書けないのはプロンプト力の不足ではなく、タスクがまだ委任できる形になっていないシグナル**だと考えるようになってから、この判断で迷わなくなりました。

## わかったこと

一連の事件を経て、自律ループの設計は「条件の分類が先、強制の仕組みが後」という順番に落ち着きました。

* 到達型はhookで機械的に強制する。ここはプロンプトより仕組みが強い
* 収束型は井戸の枯渇で完了を定義し、**上限句を必ず付ける**。脱出口のないループは事故る
* rubric+judge型は必要条件だけhookに守らせ、十分条件は独立judgeの収束で判定する
* どの型にも落ちなければ、落ちる形になるまで委任しない

逆に言うと、冒頭の「プロンプトを強い言葉に書き直す」が効かなかった理由も分類で説明できます。無限ループは条件に脱出口がない構造の問題で、空虚な条件は検証の非決定論性をhookに押し付けた設計の問題。どちらも言い回しの層には原因がなかったんです。

一方で、まだ悩ましい点もあります。語彙lintを捨てて「空虚な条件の防止は知識とレビューに寄せる」と決めた判断は、今でも少し迷いがあります。hookの床を下げた分、レビューをサボった瞬間に空虚な条件が復活する構造なので、どこまでhookに持たせるべきだったかは運用しながら探っている最中です。上限句の値も「15ターンって何を根拠に？」と聞かれると、正直まだ勘です。

## おわりに

自律ループが空転していたら、プロンプトを磨く前に、その完了条件がどの型かを見てみてください。**決定論的に検証できる条件なのか、収束で定義すべき条件なのか、judgeに渡すべき条件なのか**。分類が決まれば、強制のかけ方はほぼ自動的に決まります。

それではまた！
