## はじめに

「前任者が全部頭の中に入れていた。引き継ぎ書がないから、聞くたびに手が止まる」

こんな話は、どの規模の会社でも一度は聞きます。

属人化の本質は「手順が人の頭にある」ことです。

それをAIの世界に置き換えると、「同じ指示を毎回チャットに貼り直す」という作業がそれに当たります。毎日同じ確認リストを入力し直す。毎回同じ注意事項をコピーする。これはAIを使いこなしているようで、実は「使う人のスキル」に依存した状態です。

Anthropic社のAIアシスタント「Claude Code（ターミナルやエディタから呼び出して使う、コーディング支援・業務支援AIツール）」に、この問題を一気に解消できる仕組みが整いました。

2026年5月末のアップデートだけでも、複数の新コマンドが立て続けに追加されています。スラッシュコマンドの進化と、その中核にある「スキル」機能の話です。

  

---

## 「/」から始まるコマンドとは何か

Claude Codeを操作するとき、チャット欄に

```
/usage
```

のように「スラッシュ（/）＋コマンド名」を入力すると、特定の処理がすぐに走ります。これを「スラッシュコマンド」と呼びます。

MacのキーボードショートカットやSlackの「/remind」に近いイメージです。長い文章を打たなくても、一言で意図が伝わります。

2026年5月以降のアップデートで、この組み込みコマンドが立て続けに追加・強化されました。業務運用に直接関わるものを整理します。

### 費用とプランの利用状況を確認する

```
/usage
```

セッション中にかかった費用・プランの利用上限・利用統計をその場で表示します。Claude Pro/MaxやTeam/Enterpriseプランでは、スキル・サブエージェント（仕事を分担させる複数のAIの使い方）・プラグイン・MCPサーバー（外部ツールとの接続口）ごとの内訳まで確認できます。

```
/cost
```

```
/stats
```

上記2つは同じ機能のエイリアス（別名）です。「今月どれだけ使っているか」が手元でわかるのは、コスト管理の観点から見て小さくない変化です。

### 作業メモリの残量を把握する

```
/context
```

AIが一度に読み込める「作業メモリ（コンテキスト）」の使用量を、色付きの図で可視化します。長い会話が続くとこのメモリが圧迫されて回答精度が落ちる場合があります。「なんか最近ズレた回答が増えた」と感じたとき、原因がひと目で分かります。

```
/compact
```

これまでの会話を要約・圧縮して、メモリの空きを作ります。長いセッションを続けるときの必須コマンドです。

### 裏で分身に作業させる、会話をコピーして枝分かれさせる

![](https://assets.st-note.com/img/1780467698-vEQ3celwsbdjHn1paVRIAzL5.png?width=1200)

似て非なる2つ、/fork と /branch の違い

公式X（Anthropic）の発表によると、

```
/fork
```

と

```
/branch
```

は、似て非なる2つの別コマンドです。混同しやすいので整理します。

```
/fork
```

は、今のセッションの状態（システムプロンプト・使えるツール・会話履歴・モデル・プロンプトキャッシュまで）をそっくり引き継いだ別働のエージェントを、バックグラウンド（裏）で走らせるコマンドです。裏で動かした結果は、今のセッションに返ってきます。

経営の文脈で例えると、「今の自分の状況をすべて把握した分身に、裏で別件を片付けさせて、結果だけ受け取る」イメージです。手元の作業を止めずに、もう一件を並行して進められます。

```
/branch
```

は、これまでの会話の記録（トランスクリプト）を新しいセッションにコピーして、そのコピーを自分で操作して別の流れを試すコマンドです。元の会話はそのまま残ります。

こちらは「議事録ごと別室にコピーして、そっちの議論を自分で進める」イメージです。「この指示で動かしてみたいが、今の流れは消したくない」という試行錯誤が安全にできます。

元の流れに戻りたい場合は

```
/resume
```

で戻れます。

### 定期処理をスケジュールする

```
/schedule
```

Anthropicのクラウド上で定期実行する「ルーティン（routine：決まった時間に繰り返す処理）」を作ります。会話形式で設定できるため、コードを書かなくても「毎週月曜の朝にこの確認を走らせる」という設定が可能です。エイリアスは

```
/routines
```

です。

### AIへの指示書の雛形を作る

```
/init
```

プロジェクトに

```
CLAUDE.md
```

という「AIへの指示書ファイル」の雛形を自動生成します。「このプロジェクトでは何を優先するか」「どの形式で出力するか」といったルールをファイルに書いておけば、毎回伝え直す手間がなくなります。

### 最近追加されたコマンド（2026年5月〜）

2026年5月28日リリースのv2.1.154では、さらに3つのコマンドが追加されました。

```
/goal
```

ゴール（達成条件）を設定すると、その条件を満たすまでClaudeが複数ターンにわたって作業し続けます。

```
/workflows
```

複数の作業を自動で段取りして進める「動的ワークフロー」の進捗確認・一時停止・再開・保存をまとめて管理します。

```
/simplify
```

変更したコードの整理（クリーンアップ）を、4役のAIが並列で確認・修正まで適用します。エンジニアが使う場面が多いコマンドですが、「複数の視点でレビューしてから適用する」という設計思想は、業務文書の校正にも応用できます。

また、v2.1.152では

```
/reload-skills
```

が追加されました。セッション中に新しく作ったスキルを、再起動せずその場で反映させるコマンドです。スキルを育てながら使う場面で効いてきます。

## 最大の変化——スキルでコマンドを自分で作れるようになった

組み込みコマンドの追加よりも、実は業務インパクトが大きいのがこちらです。

**カスタムコマンドが「スキル（Skills）」という仕組みに統合されました。**

スキルとは、自分専用のスラッシュコマンドを誰でも作れる機能です。

### 作り方はフォルダ名とファイル1枚だけ

たとえば

```
.claude/skills/deploy/SKILL.md
```

というファイルを作ると、それだけで

```
/deploy
```

というコマンドが使えるようになります。フォルダ名がそのままコマンド名になります。

```
SKILL.md
```

の中に書くのは、シンプルなYAMLフロントマター（ファイルの冒頭に書く設定情報のメモ欄。「description:〜」の1行）と、コマンドを実行するときにClaudeへ渡す手順・指示の本文です。

従来の

```
.claude/commands/
```

配下に置いたファイルも引き続きそのまま動くため、既存の設定を崩す必要はありません。

### 個人用とプロジェクト用の2種類

![](https://assets.st-note.com/img/1780467470-9dvKaSPsNVuopT7gXL1m0nC5.png?width=1200)

個人用とプロジェクト用、スキルの2つの置き場所

スキルには置き場所が2種類あります。

1. 個人用：ホームディレクトリ直下のskillsフォルダに置く形で、自分のすべてのプロジェクトで使えます
2. プロジェクト用：そのプロジェクト内のskillsフォルダに置く形で、そのプロジェクトだけで有効です

会社全体のルールは個人用、特定業務の手順はプロジェクト用、と使い分けが可能です。

### 「呼ばれたときだけ読み込まれる」設計が重要

スキルの本文は、そのコマンドを実行したときにだけ読み込まれます。

普段の会話・作業中はメモリに常駐しないため、いくら長い手順書をスキルに書き込んでも、普段の動作が重くなる心配はありません。

これはCLAUDE.mdに何もかも詰め込むのとは根本的に違う設計です。CLAUDE.mdは常時読み込みの「就業規則」だとすれば、スキルは「必要なときに取り出す手順マニュアル」です。

## 中小企業の業務で何が変わるか

スキルの仕組みを、経営の文脈で読み直してみます。

### 「属人化した手順」がコマンド1つになる

「月次報告のチェック項目、先月誰が作ったんだっけ」「納品前の確認リスト、またチャットに貼り直すの？」

こうした場面は、中小企業では日常的です。

スキルを使えば、こうした手順を

```
.claude/skills/monthly-check/SKILL.md
```

に一度書いておくだけで、あとは

```
/monthly-check
```

と打つだけで呼び出せます。手順が「人の記憶」ではなく「チームで共有できるファイル」に移ります。

引き継ぎのコストも下がります。新しいスタッフが入ってきたとき、「このコマンドを打てば手順が走ります」と伝えるだけで済む状態に近づきます。

### スキルを作るタイミングは公式も明言している

公式ドキュメントでは、スキルを作るとよいタイミングとして次の場面が挙げられています。

* 同じ指示やチェックリスト・手順を何度もチャットに貼り付けているとき
* CLAUDE.mdの一節が「事実の共有」ではなく「実行手順」に育ってきたとき
* 「このプロジェクトではこう動いてほしい」という動作を毎回文章で説明しているとき

「また同じこと説明している」と感じる瞬間が、スキル化のサインです。

### 費用の可視化は経営判断の材料になる

```
/usage
```

コマンドで利用状況・費用がリアルタイムで確認できるようになったことも、見逃せない変化です。

「AIを使い始めたけど、どれだけコストがかかっているか分からない」という状態は、経営者にとって不安の種になります。**コスト管理ができる状態になって初めて、AIの業務導入が「投資」として議論できます。**

なお、Claude Codeの一部機能はClaude ProやMaxなどの有料プランを前提にしています。導入を検討する際は、プランの確認と費用試算を先に行うのが現実的です。

## Claude Platformを操作する新CLIツール「ant」

スラッシュコマンドの進化と並行して、もう一つの動きがあります。

公式X（Anthropic）の発表によると、Claude Platformのすべてのエンドポイント（APIの操作口）をターミナル（コマンドライン）から直接実行できる新しいCLIツールが公開されました。コマンド名は

```
ant
```

です。

できることは、Messages API（ClaudeにメッセージをAPIで送って応答を得る操作）の呼び出し、Claude Managed Agents（Anthropicが管理する形で動かすAIエージェント）の立ち上げ、実行結果をそのままシェル（ターミナルの操作環境）へ流し込む操作、といった内容です。

経営の文脈での意味は、「これまでエンジニアがコードを書いて呼んでいたAPI操作が、ターミナルの一行コマンドで動かせる方向に進んでいる」という流れです。

さらに、このツールはClaude Code自身が

```
claude-api
```

スキルを通じてよく理解しているため、Claude Codeに「antでこれをやって」と自然言語で頼む形でも扱いやすい関係性になっています。

freeeとClaudeをAPI経由で連携させるような業務自動化を検討している場合、API操作の入り口がエンジニアでなくても扱いやすい方向へ動いているという文脈として、頭の片隅に置いておく価値はあります。

## で、結局うちは何をすればいいのか

技術の話が続いたので、一度立ち止まります。

「スキルとかコマンドとか、結局うちの会社に何の関係があると？」という問いは、まともな問いです。

### 毎週、誰かが繰り返している作業を見つける

請求書の確認、見積もりの内容チェック、新人への同じ説明。

どんな業種・どんな規模の会社にも、「これ、毎回同じ手順でやっているな」という作業が必ずあります。担当者が変わると途端に止まる、あの作業です。

AIを使う・使わないにかかわらず、まず紙に書き出してみるだけでも十分です。

1. 毎月・毎週、誰かが必ず繰り返している手順を1つ選びます
2. その手順を「誰でも見ればわかる」レベルで紙やメモに書き出します
3. 「この手順、何分かかっているか」を記録します

道具はいりません。パソコンも、AIも、この段階では不要です。

### 書き出した手順を、コマンド1つに変える

![](https://assets.st-note.com/img/1780467774-dLpsnEAGD9VNm70t3u4HeOxB.png?width=1200)

似て非なる2つ、/fork と /branch の違い

書き出した手順が3つ以上たまってきたとき、Claude Codeのスキルが使いどころになります。

スキルとは、先ほど説明した「自分専用のスラッシュコマンドを作る仕組み」です。フォルダ1つとファイル1枚を用意して、そこに手順を書くだけで、次からはコマンド1つで同じ手順が呼び出せるようになります。

たとえば「毎月の請求書チェック手順」をスキルにすれば、次からはチャット欄に

```
/invoice-check
```

と打つだけで、Claudeが手順どおりに動き始めます。手順書の在り処が「あの人の頭の中」から「チームで共有できるファイル」に移ります。

新しいスタッフが入ったときも、「このコマンドを打てば手順が走ります」と伝えるだけで引き継ぎが完結する状態に近づきます。

### 探すのは「繰り返し作業」だけではありません

スキルやClaudeの使いどころは、繰り返し作業の自動化だけに限りません。もう少し視点を広げると、使いどころがぐっと増えます。

1. 攻めの使い方（売上を作る側）——提案書のたたき台、営業メールの文面、新サービスの企画出しなど、「人手と時間が足りなくて後回しにしていた、売上につながる仕事」を前に進める使い方です
2. 苦手分野の壁打ち相手（即席の相談役）——社内に詳しい人がいない分野について、考えを整理する相手として使う使い方です
3. ミスや抜け漏れの見張り役（品質チェック）——請求金額の照合、提出期限の確認、書類の必須項目チェックなど、「同じ観点で毎回確認する」手順を担わせる使い方です

攻めの使い方では、よく使う提案書の型や営業文面のパターンをスキルに登録しておくと、毎回ゼロから書き起こす手間がなくなります。スキルが「営業の型」を持つ状態になります。

壁打ち相手としての使い方では、見慣れない制度や契約条件の論点を一次確認の前に整理する、といった場面が典型的です。「詳しい人がいなくて止まっていたこと」を前に進める助けになります。ただし、最終的な判断は専門家への確認が前提です。

品質チェックの使い方では、「同じ観点で毎回チェックする」という手順こそがスキルに向いています。人がやると見落としやすい確認を、漏れなく一定の質で繰り返せるようになります。

守りの自動化（繰り返し作業）だけでなく、売上づくり・苦手の補完・ミス防止まで視点を広げるほど、スキルの使いどころは自然と見えてきます。

### で、儲かるのか

直接お金が増える魔法ではありません。正直に言います。

効き方はこちらです。

仮に、毎月の定型チェックに1人が**2時間**かけているとします。それがスキルで数十秒の呼び出しで済むようになれば、月に約2時間・**年間24時間**がほかの仕事に回せる計算です。

さらに大きいのは「人が辞めたときのリスク」が下がる点です。手順が人の頭にある状態では、その人が辞めた瞬間に業務が止まります。手順をスキルに移しておけば、引き継ぎのコストと「辞めた後の混乱」が構造的に小さくなります。

新人教育で同じ説明を毎回1時間かけているなら、スキルにしておけばコマンドを1つ伝えるだけで済みます。

派手に売上が増える話ではありません。今いる人の時間を守り、特定の人に依存しない仕組みを地道に積み上げる打ち手です。

中小企業にとって「特定の人が辞めても回る状態」は、売上増より先に解決すべき問題のことが多いです。スキルは、その問題に静かに効く仕組みです。

## Claudeに業務情報を渡す前に確認したい3つのこと

スキルの使い方が広がるほど、Claudeに渡す情報の種類も増えていきます。請求書の数字、取引先との契約内容、スタッフの勤怠データ——こうした情報を扱う前に、線引きを先に決めておくのが現実的です。

1. プラン（契約形態）を確認します——入力した内容がAIの学習に使われない設定になっているか、業務利用に適した商用プランやAPI利用になっているかを確認します。無料・個人向けの設定のまま会社の情報を入力しないことが基本です
2. 秘密保持（取引先との約束）を確認します——取引先とのNDA（秘密保持契約：見聞きした情報を外部に出さない取り決め）がある情報を、外部サービスに渡してよいかを事前に確認します。「便利だから入れてみた」が後で問題になるケースは、この確認を省いたときです
3. 個人情報は伏せます——氏名・連絡先・マイナンバーなど個人を特定できる情報は、Claudeに渡す前に伏せ字にするか削除します。情報を渡す段階で不要な個人情報を含めない習慣が、トラブルを未然に防ぎます

便利な道具ほど、使う前に「何を渡してよいか」を決めておく必要があります。スキルに手順を登録するタイミングで、合わせて「この手順でどんな情報を扱うか」を確認する習慣にしておくと、あとから見直す手間が省けます。

## おわりに

「うちの規模でカスタムコマンドなんて作れるの？」という問い合わせは、よく耳にします。

答えはシンプルで、作れます。必要なのはフォルダ1つとファイル1枚、そして「毎回同じことを貼り直している手順はどれか」という問いだけです。

コマンドは増え続けています。ただ、増えた数を覚えることより、「どの手順をコマンド化するか」という設計判断のほうが、業務インパクトははるかに大きいです。

技術の話として読むより、「手順をどこに置くか」という組織設計の問いとして読んだほうが、使いどころが見えやすい機能です。

属人化が続く現場ほど、この仕組みは効きます。
