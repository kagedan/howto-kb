Claude Codeでサブエージェントを並列起動したところ、数日で数万ドル規模の請求が来た——こうした事例が海外コミュニティで報告されています。

Anthropicの公式情報を参照した@IT 2025年6月4日付記事（https://t.co/9gbb2qBvVW ）によれば、「サブエージェントを多用するワークフローは、シングルスレッドセッションと比べて大幅に多くのトークンを消費する可能性がある」とされています。実務でClaude Codeを使うチームが知っておくべき、コスト増加の構造とその対策を整理します。

まず、サブエージェントを使うべき場面は限られています。「コードベースを複数の異なる角度で同時検索する」「メインセッションを肥大化させたくない重いコンテキスト作業」がその典型です。各サブエージェントは独自のコンテキストウィンドウで動作し、タスク完了後は結果サマリーだけをメインに返します。途中の中間作業はサブエージェント内に閉じ込められるため、メインセッションを長持ちさせる効果があります。軽量タスクには低コストモデル、複雑なアーキテクチャ判断には高性能モデルを使い分けることが有効です（各モデルの用途適性は公式ドキュメントで確認してください）。

一方、避けるべき反パターンは4つあります。

1つ目は大量並列起動です。多数のサブエージェントを同時に立ち上げると、トークン消費が急増します。2つ目は長いセッションの引き続き使用です。セッションが長くなるほど、過去メッセージを毎回再読するコストが積み重なり、1ターンあたりのトークン消費量が大幅に増加します（@IT記事では具体的なターン数と消費量の例が紹介されています）。3つ目はMCPサーバーの繋ぎっぱなしです。複数のMCPサーバーを接続したままにすると、作業開始前からターンごとに相当量のトークンオーバーヘッドが発生し得ます（同記事で言及されている数値の詳細は原記事をご確認ください）。4つ目はextended thinkingの適用範囲の誤りです。シンプルなタスクに適用すると不要なコスト増につながるため、複雑な判断が必要な場面だけに絞ることが推奨されます。

今日から使える判断基準は3点です。まず、トークン消費状況の可視化から始めてください。Claude Codeではセッション中のトークン使用量を確認する手段がいくつか提供されており、サブエージェントやMCP接続ごとの消費割合を把握することが起点になります（具体的なコマンドや画面は利用環境のバージョンで異なるため、公式ドキュメントで確認してください）。次に、ツール検索設計の最適化です。@IT記事では、適切なツール検索設計によってトークン使用量を大幅に削減できる事例が紹介されています（具体的な削減数値は原記事をご参照ください）。最後に、実行前の計画確認を習慣化することです。Claudeが複数ファイルを読もうとしている場合、実際に必要なファイルだけに絞って実行前に修正できます。計画段階での修正コストはほぼゼロですが、半実行済みのアプローチを修正するコストはそれまでの全コンテキスト分かかります。コードレビューやドキュメント生成など、一定の遅延が許容できる処理については、Batch APIの割引制度を活用することも選択肢に入ります（割引率・適用条件は変更される場合があるため、公式ページで最新情報を確認してください）。

ツールの多機能さと請求額は比例しません。構造を理解して使う場所を絞ることが、Claude Codeを業務コストとして正当化できる唯一の道です。
