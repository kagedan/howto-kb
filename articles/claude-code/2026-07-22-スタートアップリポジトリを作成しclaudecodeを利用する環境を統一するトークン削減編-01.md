#### 注意点

**RTKが出力を圧縮・整形すること**、または **他のhook/ツールと重なること**による運用上の罠が多かったです。
そこはAIの出力もみつつ、あれ？とおもったらRTKのことも含めて伝えて、
再度確認するくらいの運用はする必要があるなという感想です。

| # | 事象 | 起こったこと | 原因 | 対策 |
| ---: | --- | --- | --- | --- |
| 1 | PR diff のフィルタが効かない | `gh pr diff` を `awk '/^diff --git/'` 前提でフィルタしたが、期待通り除外できなかった | RTK が `gh pr diff` の出力を圧縮・整形し、生の unified diff ではなくなっていた | 生のdiffが必要な処理では `rtk proxy gh pr diff ...` を使う。必要ならファイルへ保存してから処理する |
| 2 | diff の生フォーマットが取れない | PRレビュー用に diff を取得したい場面で、RTK が diff を圧縮していた | RTK hook が `git` / `gh` の出力を自動圧縮する | `rtk proxy gh pr diff <PR>` や `rtk proxy git diff ...` で bypass する |
| 3 | worktree の `git status` が main リポジトリのものに見えた | worktree作業中に `git status` の結果が main 側の状態に見え、状態把握を誤りかけた | RTK の `git status` 圧縮表示により、情報が要約されて誤解を生んだ | `git status --porcelain` など機械可読・最小形式で確認する。必要なら `rtk proxy git status` も使う |
| 4 | RTK未対応コマンドがある | `git checkout`、`ollama list`、`ollama run` などはRTKの対象外だった | `rtk discover` で未対応と判定。RTKが全コマンドをカバーするわけではない | Claude-StartUp側では追加対応不要と判断。必要ならRTK上流へIssue化する |
| 5 | 統計に乗らない/壊れる可能性 | `DIFF=$(git rtk diff HEAD)` のような形式がRTK hookで壊れ、統計に乗らない可能性があると記録されていた | shell substitution と RTK の書き換え/計測の相性が悪い可能性 | 統計に乗せたい処理とraw取得したい処理を分ける。diff取得は明示的に `rtk diff` / `rtk proxy` を使い分ける |

1件詳細を記述しておきますと、以下のようになります。

#### 1. diff フィルタと RTK 圧縮の衝突

PRレビュー等でdiffが必要なときに、`gh pr diff` の出力を `awk '/^diff --git/'` でファイル単位にフィルタする場面があった。

しかし RTK が `gh pr diff` の出力を圧縮・整形すると、rawな unified diff の前提が崩れる。その結果、`awk` フィルタが効かず、除外したいファイルが残ったり、期待した diff ブロック単位で処理できなかった。

対策は、**diff構文をパースする処理では RTK を bypass する**こと。

```bash
rtk proxy gh pr diff 191 > /tmp/pr191.diff
```

別プロジェクトのログでも、`gh pr diff 2 | awk ...` が効かず、「RTK がフォーマット変換済み」と判断して `git diff main..branch -- ':!file'` へ切り替えていた。

#### 5. 壊れる可能性

これはRTK本体側の未解決バグで、現状は回避するしかない部分です。

RTKはClaude CodeのBash hookで `git diff HEAD` のようなコマンドを `rtk git diff HEAD` に置き換えます。  
ただし、`$()` の中に `git` コマンドがある場合、置き換え位置がずれてコマンドが壊れることがあります。

```bash
DIFF=$(git diff HEAD)

# 本来期待する置き換え
DIFF=$(rtk git diff HEAD)

# 実際に起きる壊れた置き換え
DIFF=$(git rtk diff HEAD)
```

https://github.com/rtk-ai/rtk/issues/1109

この結果、$() 内のコマンド置換がおかしくなり、git rtk diff HEAD のような不正なgitサブコマンドとして実行されます。
そのため、正しくRTKが呼ばれず、RTKの削減対象にも統計にも乗らないケースがあります。
この手のケースが解消されれば、RTKの削減効率はもう少し上がる可能性があります。

## Lean-CTXの導入

https://github.com/yvgude/lean-ctx

RTKを導入したわけですが、この手のトークン削減系の話は、
その当時(たしか5月とかそのくらいの時期)少し話題が沸騰していた時期でした。
(Claude Codeが、トークン消費のバグを抱えていて、消費が増えてすぐリミット到達してしまうというpostが多かった時期。なお、2026/07/07時点でも該当IssueはOpenのまま)

https://github.com/anthropics/claude-code/issues/46917

Lean-CTXも、そんな流れの中でたまたまpostを見たので、軽い気持ちでClaudeに聞いてみて、
導入するかという流れにした記憶があります。

ただ、Lean-CTX自体は、RTKの動作部分との競合に加えて、Knowledge-RAGやKizamiがすでに分担していた、
長期記憶に関する分野の機能も持ち合わせているため、
RTKでカバーしきれていない部分を期待して、導入して利用してみようという判断をしました。

具体的には、以下のような表になります。

| 観点 | RTK | Lean-CTX |
| --- | --- | --- |
| 主な対象 | CLI出力 | ファイル読み込み、再読み込み、MCP経由の操作 |
| 強いところ | `git` / `gh` / `grep` などの出力圧縮 | ファイル再読み込みキャッシュ、読み取りモード、セッション文脈管理 |
| 当時の評価 | CLI出力フィルタとして有効 | RTKにない「ファイルキャッシュ」が魅力 |
| 懸念 | 生のdiffなどを壊す可能性 | MCP呼び出しコスト、RTK/Kizamiとの役割重複 |

リポジトリとしての目的は、同一環境の構築ですが、このリポジトリを利用している環境でのゴールは、
開発、保守、運用の実務的な領域になります。

その観点で有効そうだと判断した材料として、Lean-CTX側の説明では以下のような削減例が挙げられていました。

| 対象 | 通常 | Lean-CTX適用後 |
| --- | ---: | ---: |
| ファイル再読み込み | 約2000 tokens | 約13 tokens |
| `git status` 生出力 | 約800 tokens | 約120 tokens |

:::note warn
この数値は、自分の環境での実測値ではなく、Lean-CTX調査時に公式情報・README等から拾った代表値です。
:::

`git status` のようなShell出力は、Claude Code上での呼び方やhookの順序によってRTK側が処理することも、Lean-CTX側が処理することもあります。

この点は厳密に制御しようとすると、hook設定やallowlistの調整が必要になります。
ただ、導入目的はあくまでClaude Codeへ渡るトークン量を減らすことなので、
RTKとLean-CTXの担当が一部重なることは許容とし、
どちらかが圧縮してくれていれば目的は達成できている、という判断をしています。

#### 注意点、ハマったこと

Lean-CTX単体でも起こり得るもの、RTKとの複合に起因するものどちらも踏んだ感じでした。

| # | 事象 | 起こったこと | 原因 | 対策・判断 |
| ---: | --- | --- | --- | --- |
| 1 | `ctx_shell` 経由だとRTKが効かない | `ctx_shell` で `rtk gain` を実行しようとすると、RTKがスルーされた | Lean-CTXのallowlistで `rtk` がブロックされた | RTK確認はnative Bashで実行。`ctx_shell` はLean-CTX側の圧縮として扱う |
| 2 | `lean-ctx hook rewrite` とRTK hookが競合 | `git status` → `rtk git status` → `lean-ctx -c 'rtk git status'` のように二重に包まれる | `lean-ctx onboard` がBashコマンドを `lean-ctx -c` でラップするhookを追加する | 競合しうることを認識。必要ならBash rewrite hookを外す |
| 3 | `ctx_read` が特定ファイルでパニック | `setup.sh` を `full` モードで読むとLean-CTX内部エラーになった | `─────` のようなU+2500罫線文字を含むbashファイルでパーサが落ちていた | `lines:N-M` / `map` で読む、または区切り線をASCIIに寄せる |
| 4 | `bash -c` / `trap` がClaude Code側のshell allowlistに引っかかる | SessionStart hookで起動時に継続的なブロックエラーが発生 | `bash -c '...' &` や `trap "" INT TERM` がshell allowlistでブロックされた | hookから `bash -c` / `trap` パターンを除去し、直接呼び出しに寄せる |

最終的には、Lean-CTX側のshell allowlistはかなり緩めています。

```toml
shell_security = "on"
shell_allowlist = []
```
:::note warn
`shell_allowlist = []`の設定は、実質的にはLean-CTXのフィルタをほぼ素通りさせる設定になります。
そのため、Claude Codeの `permissions.deny` など、別のレイヤーで危険なコマンドを制限しておくことを強く推奨します。
:::

`shell_security`自体は有効のままにしつつ、`shell_allowlist = []` にすることで、
Lean-CTX側のシェルコマンドフィルタリングは実質的に無効化しました。

危険なコマンドについては、Lean-CTX導入前からある程度Claude Code側の `permissions.deny` で制御していたため、
最終的にはそちらで管理する方針にしています。

## RTKとLean-CTXを併用してみた結果

最初にRTKを導入し、後からLean-CTXを導入してみたのですが、
結局RTKを外す、あるいはLean-CTXを外して一本化する判断はしませんでした。

達成したい項目としては、トークン削減が目的だったため、
その観点から見たとき、以下のような結果になっています。

| ツール | Commands | Tokens saved | 削減率 / Compression |
| --- | ---: | ---: | ---: |
| RTK | 2,453 | 746.7K | 3.2% |
| Lean-CTX | 1,575 | 635.6K | 34.1% |

単純にトークン数だけ集計すると、以下になります。

| 指標 | 値 |
| --- | ---: |
| RTK saved | 746.7K |
| Lean-CTX saved | 635.6K |
| Total saved | 1.38M |

:::note warn
RTKとLean-CTXは集計対象や計測方法が異なるため、
削減率をそのまま横並びで比較するのはあまり適切ではありません。
また、hookやMCP経由の処理が重なる部分もあるため、上記の合算値は「厳密な総削減量」というより、自分の環境で各ツールが記録していた削減実績の合計として見ています。
:::

併用したことによる問題や、Lean-CTX側の設定の問題等もありましたが、
Claude Codeのリミットにかかるまでの時間が伸びるならヨシ、
ということで、ここは併用続行という判断をしました。
