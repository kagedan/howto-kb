Claude Codeを使うようになって、実装にかかる時間はかなり短くなりました。

ただ、開発が速くなっても、タスクを作る、着手状態にする、PRを紐付ける、レビュー中に変える、完了にする、といった管理作業は残ります。1回ごとの操作は小さくても、開発の途中で何度もLinearを開くのは地味に面倒です。更新を忘れると、ボードと実際の進捗もずれていきます。

現在はNotion、Claude Code、Linear、GitHubをつなぎ、仕様の整理から実装、レビュー、タスク完了までを次のように進めています。

```
Notionに仕様をまとめる
        ↓
Claudeと設計を詰める
        ↓
実装計画をタスクへ分解する
        ↓
LinearにIssueを登録する
        ↓
Claude CodeがIssueを取得する
        ↓
実装計画を作る
        ↓
人間が承認する（必要があれば方針修正）
        ↓
Claude Codeが実装・PR作成
        ↓
GitHubでレビュー
        ↓
PRをマージ
```

この流れで特に便利だったのが、LinearとGitHubの連携です。

GitHub上でレビューを依頼するとLinearのIssueが`In Review`になり、PRをマージすると`Done`になります。開発者は普段どおりGitHubで作業するだけで、Linear側の進捗も追従します。

この記事では、現在の開発フローと、なぜ自動化が進んでもLinearにIssueを登録しているのかをまとめます。

## 仕様はNotionに集める

インフラ構成やシステム全体の仕様はNotionにまとめています。

リポジトリ内のドキュメントだけで管理する方法もありますが、要件や設計の背景まで含めてチームで共有する場所としては、今のところNotionが扱いやすいと感じています。

Claude CodeにはNotion MCPを接続しています。設計を始めるときは、対象のページを読んでもらった上で、既存コードも確認しながら壁打ちします。

たとえば、次のような点を先に詰めます。

* 変更の目的
* 現在の構成で困っていること
* 採用する設計
* 今回は対応しない範囲
* 移行手順
* 動作確認の方法

設計が固まったら、その内容を実装計画へ落とし込みます。

## Claudeに実装計画をタスク分解してもらう

実装計画を作った後は、ClaudeにLinearのIssue単位へ分解してもらいます。

人間がすべてのIssueを一から書くのではなく、Claudeが仕様書と実装計画を読んで、実装可能な大きさへ分けて登録します。

実際のIssueは、だいたい次の構成です。

```
## 作業内容

- 変更する対象
- 実装する内容
- 守るべき設計上の制約

## 受け入れ条件

- 既存機能がこれまでどおり動くこと
- 指定したコマンドやテストが成功すること
- 変更範囲が想定内に収まっていること

## 参照

- Notionの仕様書
- リポジトリ内の設計ドキュメント
- 関連Issue
```

重要なのは、Issueを単なる作業名にしないことです。

たとえば「ディレクトリを整理する」だけでは、どこまで変更してよいのか判断できません。「今回は挙動を変えず、移動とimport修正だけにする」のように、変更してよい範囲と変更してはいけない範囲を明記します。

## Issueの目的は、Claudeへの指示だけではない

ここまでClaudeが作業するなら、Linearを挟まず、実装計画をそのままClaude Codeへ渡せばよいようにも見えます。

個人開発であれば、それでも問題ないケースはあると思います。

一方、チーム開発では、次の情報を他のメンバーから確認できる必要があります。

* 現在どのタスクを進めているか
* どこまで完了しているか
* 何がレビュー待ちなのか
* 次に着手できるタスクは何か
* 仕様と実装がどのPRで結び付いているか

Claude Codeのセッション内だけで作業を進めると、実装している本人には分かっても、他のメンバーからは進捗が見えません。

LinearのIssueは、Claudeへの入力であると同時に、チームへ進捗を公開する単位になっています。

つまり、Issueを作る目的はタスクを細かく管理することではなく、**開発中の状態をチームから見えるようにすること**です。

## Claude Codeには、まず実装計画だけを作ってもらう

Linear MCPを使い、Claude Codeから対象のIssueを取得します。

指示は毎回長く書かず、曖昧点を排除し実装計画を作成するskillか、次のような内容です。

```
LinearのJP-78を取得し、関連する仕様書と既存実装を確認してください。

まだ実装は開始せず、以下を含む実装計画を作成してください。

- 変更対象
- 実装手順
- テスト方法
- 想定される影響
- 不明点や懸念点
```

Issueを読んですぐ実装させるのではなく、実装計画を一度確認します。

ただし、毎回細かく計画を書き直す運用にはしたくありません。人間がレビューする内容は、できるだけ次の判断に絞ります。

* Issueの範囲内に収まっているか
* 既存仕様を壊す変更が含まれていないか
* テスト方法が妥当か
* 不要なリファクタが混ざっていないか

Issueに作業内容、制約、受け入れ条件まで書かれていれば、多くの場合は計画を読んで承認するだけで済みます。

実装計画のレビューをなくすというより、**修正作業ではなく承認作業に近づける**イメージです。

## 承認後はsubagent-drivenで実装する

実装計画を承認した後は、Claude Codeに実装を進めてもらいます。

現在は、調査、実装、テスト、レビューなどを必要に応じてサブエージェントへ分ける、subagent-drivenな進め方をしています。

メインのコンテキストだけですべてを処理するより、調査結果やテストログが混ざりにくく、役割ごとに確認しやすくなります。

実装後はClaude自身にも差分をレビューさせ、テスト結果と変更内容をまとめてPRを作成してもらいます。

もちろん、Claudeのセルフレビューだけでマージするわけではありません。最終的なPRレビューとマージ判断は人間が行います。

## GitHubで開発しているだけで、Linearの進捗が更新される

LinearのIssueとGitHubのPRを紐付けると、PRの状態に応じてIssueのステータスを自動更新できます。

私の環境では、次のように設定しています。

```
PRでレビューを依頼
        ↓
Linear: In Review

PRをマージ
        ↓
Linear: Done
```

PRとIssueを確実に紐付けるため、ブランチ名やPRタイトルにはLinearのIssue IDを含めます。

```
issue-78-reorganize-core-adapters
```

Claude CodeにPRを作ってもらう場合も、Issue IDをブランチ名やPRタイトルへ入れるように指示しています。

以前は、PRを作った後にLinearを開き、`In Review`へ変更し、マージ後にもう一度`Done`へ変更していました。

現在は、開発者が意識する操作は次の3つです。

タスクの状態は、その開発行為から自動的に決まります。

「PRを作ったのにIssueがIn Progressのまま」「マージ済みなのにDoneになっていない」といったずれも起こりにくくなりました。

## タスク管理はなくならないが、タスク管理の作業は減らせる

AIによる実装が進むと、Issue自体が不要になるようにも見えます。

実際には、チーム開発でタスク管理が不要になるわけではありません。誰が何を進め、どこで止まっているのかを共有する仕組みは必要です。

減らしたいのは、進捗を共有するために人間が同じ情報を何度も入力する作業です。

GitHubには、すでに実際の開発状況があります。

* PRが作られている
* レビューが依頼されている
* 変更が承認されている
* PRがマージされている

この情報をLinearへ連携すれば、人間が手動で同じ状態を入力する必要はありません。

Linearはチームが進捗を見る場所として残しつつ、更新はGitHub上の開発活動から自動で行う。この分担が、現在の運用には合っていました。

## Issueを作るときに意識していること

このフローではIssueの品質が実装計画の品質に直結します。特に意識しているのは次の4点です。

### 1. 1つのPRで確認できる大きさにする

Issueが大きすぎると、実装計画もPRも確認しにくくなります。

「バックエンドを改善する」ではなく、「既存挙動を変えずにcoreとadaptersを分離する」のように、完了条件を判断できる単位へ分けます。

### 2. やることより、やらないことを明記する

Claudeは周辺の改善案まで提案してくれるため、今回の対象外を明記しておく方が安定します。

```
## 対象外

- API仕様の変更
- UIの変更
- 既存ロジックの書き換え
```

### 3. 受け入れ条件を確認可能な形にする

「適切に動くこと」ではなく、コマンド、テスト、画面操作など、完了を判断できる条件を書きます。

```
## 受け入れ条件

- `make server`が成功する
- 既存のメール選択から実行までの操作が動く
- ロジックの差分が移動とimport修正のみに収まる
```

### 4. 参照元を残す

Issueだけですべてを説明しようとすると、仕様が重複して更新漏れが起きます。

詳細な設計はNotionやリポジトリ内のドキュメントに置き、Issueには実装に必要な要点と参照先を載せます。

## 注意点

この運用がすべてのチームにそのまま合うとは限りません。

たとえば、本番デプロイまでを完了とするチームでは、PRマージ時点で`Done`にするのは早すぎます。その場合は、マージ時に`Ready for Release`などへ移し、デプロイ完了後に`Done`へ変更する方が実態に合います。

また、GitHub連携の自動更新を正しく動かすには、PRとLinear Issueの紐付けルールを統一しておく必要があります。Issue IDの入れ忘れがあると、PRをマージしてもステータスは更新されません。

自動化すること自体を目的にせず、チームにとって「そのステータスが何を意味するか」を先に決めておくことが重要です。

## まとめ

現在の開発フローでは、Notionを仕様の置き場所、Linearをチームから見える実装単位、GitHubを実際の開発状態の置き場所として使っています。

Claude Codeは、それぞれをMCP経由でつなぎます。

```
Notion
  チームに見える仕様と設計
    ↓
Claude Code
  設計・タスク分解・実装
    ↓
Linear
  チームに見えるIssue
    ↓
GitHub
  PR・レビュー・マージ
    ↓
Linear
  進捗を自動反映
```

AIがコードを書くようになっても、チームに進捗を共有する必要はなくなりません。

ただ、そのために開発者がIssueのステータスを手動で更新し続ける必要はありません。

普段どおり実装し、レビューを依頼し、PRをマージする。その結果がLinearへ自動で反映されるようになってから、タスク管理を意識して画面を行き来することはなくなりました。

私がこの連携に感じている一番の価値は、AIが実装してくれることだけではなく、**開発の流れとチームの進捗が自然に一致すること**です。
