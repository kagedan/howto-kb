## Claude Codeを「コーディング」以外で使う

Claude Code、使ってますか？

多くの人にとっては「コードを書くツール」だと思う。実際、コーディング用途として作られている。でも、ちょっと待ってほしい。

**毎日の仕事のふりかえりと、週次の振り返り(KPT)の準備を自動化するエコシステムをClaude Codeで作ったら、かなり使えた**という話をする。

## 振り返りについて

私の会社（ソニックガーデン）では毎週、親方と1on1でふりかえり(KPT)をやっている。Keep・Problem・Tryの3軸で仕事を振り返り、Tryは「意識する」ではなく仕組み化まで落とすのが文化だ。

親方とは、徒弟制度における師匠のような存在だ。

私の会社の振り返りや徒弟制度について、興味がある方は下記のリンクの記事を読んでみてほしい。

## 背景：ふりかえりの質が低かった

「ふりかえりの準備」が問題だった。

週末になって日報を読み返そうとすると、量が多くて追えない。KPTのドキュメントをゼロから書こうとすると時間がかかる。結果、「準備不足のままふりかえりの1on1をしてもらう」「深掘り切れなくて意識に頼ったTryで終わる」というアンチパターンを繰り返していた。

そこで考えた。**ふりかえりの準備そのものをClaude Codeに任せてみよう。**

## 作ったもの

全体の構成はこんな感じ。

```
毎日 /daily-review
  ↓ 日報で一日のふりかえり → oyakataエージェントがレビュー → 深掘り → active_tries.md更新(Tryをまとめたリスト)
週次 /weekly-kpt
  ↓ 日報を全部読んで → KPTドキュメント作成 → oyakataエージェントがレビュー → 深掘り → 準備完了
実際のふりかえりミーティング（人間の親方と）
```

### oyakataエージェント

**実際の親方のコミュニケーションスタイルと、会社の文化・価値観をすべて読み込んだカスタムエージェントを作った。**

会社ブログに書かれいてる知識（ザッソウ、タスクばらし、KPT、仕事の質…）を全部まとめたMarkdownファイルを作り、それをエージェントに読ませている。

さらに、実際の1on1の文字起こしから抽出した「親方の実際のコメントトーン」まで入れた。(例: 「意識だけに頼るんじゃなくて、どう仕組み化する？」)

### 日報スキル（/daily-review）

まずタスク管理ツールから出力した今日のやったこととその所感をoyakataエージェントに読ませてレビューしてもらう。

論理的に破綻している思考や、次に繋がらないアクション、モヤっとしているがなんの改善案も示せていない問題について、具体的なアクションに繋がるまで指摘してくれる。  
oyakataは答えは示してくれる存在ではないので、自分で考えが出るまで指摘・質問をし続けてくれる。

### 週次KPT準備（/weekly-kpt-prep）

ふりかえり前に `/weekly-kpt-prep` を叩くと、以下を全部自動でやってくれる。

1. 前回のKPTファイルを読んで「前回のTry」を取得
2. 対象期間の日報を全部読む
3. 各日報の所感からProblem候補を自動抽出
4. KPTドキュメントを生成（深掘り欄も作成し、私が深ぼるための余白を作っている）
5. 自分が深掘りを書き終えたら、oyakataエージェントがKPTをレビュー
6. レビューの内容からさらに深ぼる

oyakataのレビューが終わると、スキルが自動でこう聞いてくる。

> 「今週は何点やった？（100点満点で）」  
> 「あと○点を達成するにはどうしたらいい？」  
> 「今週のふりかえりの質は何点やった？」

毎週の親方の質問をそのまま再現している。実際のふりかえりの前に振り返りをしているので、実際の振り返りの質が高まる。

## 実際に使ってみてどうだったか

**一番大きかった変化は、ふりかえりが「週1回のイベント」ではなくなったこと。**

以前は週次のふりかえりのタイミングで初めてちゃんと振り返るようなイメージだった。毎日oyakataエージェントにフィードバックをもらうようになってから、日報を書く時点でProblemを深掘りして、意識に頼らないTryを考えるところまで自然とやるようになった。

自分ひとりでやっていると、疑問を持ってくれる人・論理破綻を突いてくれる人がいないので思考が浅くなりがちだ。それをoyakataエージェントが毎日補ってくれる。

oyakataエージェントに相談してみると、自分が意識論のTryを書いたときに「意識だけに頼るんじゃなくて、どう仕組み化する？」と返ってくる。AIが相手なら何度でも会話のラリーを重ねられるので、自分のペースで深掘りがしやすい。

その結果、週末にKPTをまとめる作業も一気に楽になった。準備不足でふりかえりに行く、というアンチパターンがなくなった。

## 技術的な話

コードは書いていない。全部Markdownファイルだけで構成されている。

```
.claude/
├── agents/
│   └── oyakata.md   ← エージェント設定
├── commands/
│   ├── daily-review.md
│   └── weekly-kpt-prep.md
├── personas/
│   └── oyakata.md   ← 会社の文化・価値観・コメントスタイル
daily_reports/
├── YYYY-MM-DD.md    ← 日報
├── active_tries.md  ← 現在進行中のTry一覧
weekly_kpt/
└── YYYY-MM-DD.md
```

Claude Codeのカスタムスキル（`.claude/commands/`）でワークフローを定義し、サブエージェント（`.claude/agents/`）でキャラクターを作る。

`daily-review.md`, `weekly-kpt-prep.md` の中でoyakataを呼び出しているのは、Claude Codeの `Agent` ツールを利用したもの。`subagent_type: "oyakata"` を指定するだけでローカルのエージェント定義が呼ばれる。

---

Claude Codeを「コードを書くツール」として使っている人が多いと思うけど、こういう使い方もある。**自分の仕事のやり方そのものをClaude Codeの中に組み込む**、というアプローチ。

コードを1行も書かなくても、自分と一緒に育ってくれる。
