## 第7章「外部顧問・Geminiの招聘」― マルチAI体制で「内輪ループ」を壊す

:::note info
この記事は、個人開発サービス **CheckMe（checkme.run）** の開発物語です。
:::

### 問題：ClaudeとのYESマンループ

Claude（参謀）との開発は最高にテンポが良い。

「このツール追加しよう」→「最高ですね！」
「このデザインどう？」→「完璧です！」
「次はこれ実装しましょう」→「それも素晴らしい！」

ある日気づいた。**「お互いが全肯定し合っているだけで、誰も客観的な品質を担保していない」**という事実に。

### マルチAI体制の設計

| AI | 役割 | 特性 |
|---|---|---|
| Claude Code | 実装・コードレビュー・アーキテクチャ | コードと一貫性に強い |
| Gemini | コンテンツ品質評価・採点・外部視点 | 日本語品質評価が高精度 |

分業の原則は「**同じ判断を異なるAIに重ねない**」こと。Claudeが書いたコンテンツをClaudeが評価すると内輪になる。異なるモデルで評価することで客観性を担保する。

### Gemini品質採点の実装

```python
# scripts/score_decks.py（概略）
# 毎日02:00 cronで実行 - デッキを20本/日採点

SCORING_PROMPT = '''
あなたはeducational content quality evaluatorです。
以下のフラッシュカードデッキを評価してください。

評価軸:
- format_score (1-5): 教育コンテンツとしての形式・構造
- impact_score (1-5): 記憶に残るインパクト・面白さ
- target_score (1-5): 中高生・受験生への適合度

出力形式: JSON {"format_score": X, "impact_score": X, "target_score": X, "feedback": "..."}
'''
```

採点結果はSQLiteに蓄積し、examples ページのソート順に使う：

```sql
-- examples ページのクエリ
SELECT * FROM study_decks
ORDER BY (format_score + impact_score + target_score) DESC, hit_count DESC
LIMIT 30
```

### Geminiを採用してから変わったこと

- **Claude起草 → Gemini評価 → Claude修正** というループが自動化できた
- 「Claudeが言うから良い」ではなく「Geminiの採点で5点だから良い」という根拠が生まれた
- コンテンツの品質基準が数値化された

### Gemini APIの使い方（Google GenAI）

```python
import google.genai as genai

client = genai.Client(api_key=GEMINI_API_KEY)
response = client.models.generate_content(
    model="gemini-2.0-flash",
    contents=scoring_prompt
)
result = _extract_json(response.text)
```

### この章のエンジニア向けポイント

- 単一AIとのループは「内輪の全肯定」になりやすい
- 生成（Claude）と評価（Gemini）を別モデルに分担させると品質基準が安定する
- 採点結果をDBに保存すると「何が良いコンテンツか」がデータで分かるようになる
- クローン評価は避ける：自分が書いたものを自分で評価させない
