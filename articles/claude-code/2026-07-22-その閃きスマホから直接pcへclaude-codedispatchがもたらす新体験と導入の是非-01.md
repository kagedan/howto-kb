**【結論を先に】**  
Dispatchは「自律実行の権限管理（パーミッション）」と「PCの常時稼働環境（スリープ回避等）」を完璧にコントロールできる人にとっては、神ツールです。一方で、なんとなく設定して重いタスクを投げると、バックグラウンドでエラーを吐いたまま沈黙します。明確な「読む→要約する→下書きを作る」などの定型パイプラインを持つ研究者やエンジニアにのみ、強く推奨します。

## 1. なぜこのテーマを選んだのか？

「あ、この記事あとでPCで試そう」と思ってスマホのブラウザでタブを開きっぱなしにし、そのまま永遠に忘却の彼方へ……。皆さんもそんな経験ありませんか？ 私は日常茶飯事です。

特にAIやマルチモーダル領域は、arXivの論文やX（旧Twitter）での有志の検証リポジトリが毎日大量に流れてきます。これを**見つけたその瞬間（スマホ）に、開発環境（PC）へコンテキストとして叩き込み、PCを開いた時にはすでに検証の土台ができている状態**にできれば、生産性は劇的に上がると確信していました。そこで今回、Claude Codeの最新機能であるDispatchを徹底的に検証してみることにしたのです。

## 2. 【最重要】この技術は採用すべき？（意思決定の支援）

実装方法を見る前に、「あなたの今のチーム・個人開発にDispatch機能は必要か？」をジャッジしましょう。

### Why / When（なぜ・いつ使うべきか）

* **いつ：** 移動中、会議の合間、あるいは寝転がりながらインプットをしている時。
* **なぜ：** 「情報の発見」と「作業の着手」のタイムラグをゼロにするため。スマホで見つけたURLやスクショを投げるだけで、PC側のClaude Codeがリポジトリをクローンしたり、マークダウンで要約レポートを生成したりしてくれます。

### 向き・不向き（ぶっちゃけどうなの？）

* **クリティカルに効くケース：**
  + 論文や英語記事のURLを投げ、「この内容を自分のプロジェクトの `docs/` 配下に日本語でまとめておいて」といった、**外部依存の少ないテキスト処理・ファイル生成タスク**。
* **ぶっちゃけ微妙なケース：**
  + 「このリポジトリをクローンして環境構築し、テストまで通しておいて」といった、**ローカル環境の依存関係エラーが出やすいタスク**。途中でコケた場合、スマホからリカバリーの指示を出すのが結局面倒になります。

### 導入判断のための比較表

スマホからAIにタスクを投げるアプローチとして、他の代表的な手法と比較してみましょう。

| 比較軸 | Claude Code (Dispatch) | GitHub Copilot Workspace | Slack / Discord Bot (自作) |
| --- | --- | --- | --- |
| **主な用途** | ローカルPC環境の直接操作・ファイル生成 | クラウド上での設計・PR作成 | チーム内での情報共有・要約 |
| **スマホからの操作性** | ◎ (公式アプリからシームレス) | ◯ (ブラウザ経由) | ◎ (チャットアプリで完結) |
| **ローカル環境への干渉** | **◎ (直接ファイルを書き換える)** | △ (最終的にPullが必要) | × (クラウド完結) |
| **セットアップの手間** | ◯ (CLIのペアリングのみ) | ◎ (GitHub上で完結) | △ (サーバー・API構築が必要) |
| **最大の弱点** | PCがスリープすると死ぬ | ローカルの細かい環境変数を読めない | コードの直接編集には不向き |

**💡 結論：**  
「自分のローカルPCの中にある既存コード（未pushのコード等）」をスマホから操作・参照したいなら、**Claude Code Dispatch一択**です。クラウドで完結するタスクならCopilot Workspaceの方が手軽です。

## 3. 技術の仕組みと関連調査

では、どうやってスマホからローカルPCを操っているのでしょうか？ 魔法ではありません。

Anthropicのドキュメントやアーキテクチャ設計によれば、Dispatch機能（および `claude remote-control`）の裏側は、セキュアなリレーサーバーを介した非同期通信です。

1. **PC側の待機：** PC上で `claude` CLIが起動・ペアリングされ、Anthropicのバックエンドに対してセッションのポーリング（またはWebSocket接続）を行います。
2. **スマホからの発火：** Claude公式モバイルアプリから「Dispatch」先を指定してプロンプト（URLや画像を含む）を送信します。
3. **リモート実行：** バックエンドがPC側のClaude Codeエージェントにタスクをプッシュ。エージェントはローカルのファイルシステム（`~/.claude/` のコンテキストやカレントディレクトリ）にアクセスし、Bashコマンドの実行やコードの編集を行います。
4. **同期：** 実行結果の標準出力やファイルのDiffがリアルタイムにモバイルアプリ側に同期され、スマホ上で「今PCが何をしているか」を確認できます。

非常に強力ですが、**PC側からAnthropicへのアウトバウンド通信**を利用しているため、ファイアウォールの内側からでもNAT越えで操作できてしまう点には、セキュリティ的な注意が必要です。

## 4. 実装・検証（判断の裏付けとして）

実際に試してみましょう。今回は「電車の中で見つけた技術記事のURLを投げ、PC上にZennの草稿（Markdown）を作らせる」というシナリオです。

### ステップ1：PC側の準備

まずはPCのターミナルで対象のプロジェクトディレクトリに入り、リモートコントロールの待機状態を作ります。

```
cd ~/my-zenn-articles
# リモートコントロールを有効化して待機
claude remote-control
```

*(※Desktopアプリ版を利用している場合は、モバイルアプリとのペアリング設定から直接Dispatchを有効にできます)*

### ステップ2：スマホからの指示（Dispatch）

通勤中、Xで面白そうなVLM（Vision-Language Model）の論文解説記事を見つけたとします。Claudeモバイルアプリを開き、PC側のセッション（Desktop等）を選択して以下のプロンプトとURLを投げます。

> \*\* (スマホから):\*\*  
> `https://example.com/awesome-vlm-paper`  
> この記事の内容を読み込んで、Zennの技術記事の草稿を作成して。  
> 保存先は `./articles/vlm-review.md` にして、僕のいつもの文体（ユーモア交じり）でお願い！

### ステップ3：PC側での自律実行

PC側では、Claude Codeが勝手に動き出します。

1. `curl` 等でURLのテキストを取得
2. 内容を解釈
3. `articles/vlm-review.md` を新規作成し、内容を書き込む

### 😱 ここで丸一日溶かしました（リアルな落とし穴）

順調に見えるこの機能ですが、私は**検証初日に絶望的なミス**を犯しました。

電車の中で意気揚々と重いタスク（「このリポジトリをクローンして、`npm install` して動くか確かめて」）を投げたんです。スマホには「処理を開始します」と表示され、「ふふふ、俺のPCが今頃唸りを上げているぜ」とドヤ顔で帰宅しました。

しかし、PCを開いてみると……ターミナルには無情にも以下の文字が。

```
> Claude Code wants to run command: npm install
> Do you want to allow this? [Y/n] _
```

**……パーミッション確認で止まっとるやないかい！！！**

Claude Codeはデフォルトで、破壊的変更やコマンド実行の前に人間（User）の承認（`permission_prompt`）を求めます。PCの前に誰もいないのに、ずっと `[Y/n]` を待ち続けていたのです。さらに最悪なことに、数時間放置したせいで**Macがスリープモードに突入**し、セッション自体が切断されていました。

**【回避策】**  
リモートから完全に自律駆動させる場合は、承認プロセスをスキップするか、安全なDockerコンテナ等のサンドボックス内で動かす設計が必須です。また、Macなら `caffeinate` コマンドでスリープを防止するのも忘れずに。

```
# スリープを防止しつつ、危険なパーミッションをスキップ（※自己責任で！）
caffeinate -i claude remote-control --dangerously-skip-permissions
```

## 5. 筆者の考察

このDispatch機能、一見すると単なる「遠隔操作」ですが、本質は**人間の思考の非同期化**にあると私は考えています。

従来、エンジニアの作業は「机の前に座っている時間」に縛られていました。しかし、アイデアが閃いたり、有益な情報に出会ったりするのは、シャワーを浴びている時や満員電車の中だったりします。

Dispatchは、その時の「閃きの瞬間」と「実行の瞬間」をデカップリング（分離）してくれます。モバイルから意図だけを投げ込み、泥臭い実行はPCリソースとAIエージェントに任せる。これは、AI時代の新しい開発体験のスタンダードになる可能性を秘めています。

ただし、先ほどの失敗談のように「自律エージェントに対する権限委譲の境界線」は、依然として大きな課題です。どこまでAIを信頼して `[Y/n]` をスキップさせるか。このチキンレースを制した者だけが、真の「スマホで寝ながらコーディング」を手に入れられるのでしょう。

## 6. まとめ

いかがだったでしょうか。今回の意思決定のポイントをまとめます。

1. **ローカル環境をスマホから直接操作できる**点が、他のツールにはないDispatchの圧倒的な強み。
2. **パーミッション確認による停止**と**PCのスリープ**が最大の罠。環境構築に工夫が必要。
3. まずは「情報の要約」「草稿作成」など、**安全でファイル生成のみのタスク**から導入するのがおすすめ。

今後は、TelegramやDiscordのBot機能（Channels連携）と組み合わせ、CI/CDの失敗通知を受け取った瞬間にスマホから「Claude、ログ見て直しといて」と投げるような、さらに怠惰……もとい、効率的なワークフローを構築してみたいと思います。
