## TL;DR

* エージェントのコンテキスト管理の焦点は「プロンプトの文言」ではなく「モデルに何を見せるか」という情報設計に移行している（Sally-Ann Delucia, [02:16]）
* 単純な先頭打ち切り・LLM 要約はいずれも失敗する。対話の連続性を壊し、重要な詳細を無音で消す
* Arise が採用した現時点のベターケースは **Head+Tail 保持・外部メモリへの中間保存・重処理のサブエージェント委譲** の 3 点セット
* 「先頭 100 文字 / 末尾 100 文字」という数値は経験則に過ぎず、コンテキスト品質の定量評価指標は業界全体で未確立
* 長期記憶（セッションをまたぐ記憶）・巨大コンテキスト問題は現時点で未解決。銀の弾丸はない
* 登壇者の観察によれば、Claude Code のコードから Anthropic も同様の打ち切り＋圧縮戦略を使用していることが確認されており、業界標準解がまだ存在しないことを示唆している

## 対象読者と前提

**対象：**

* AI エージェントを本番運用しているまたは設計中のエンジニア・テックリード
* 「長い会話になると応答品質が落ちる」「コンテキスト上限でクラッシュする」という症状に直面している人
* コンテキスト管理の具体的な実装パターンと設計判断の根拠・限界条件を知りたい人

**前提：**

* LLM API（OpenAI / Anthropic など）を直接呼び出す経験がある
* エージェントループ（tool call → 結果受け取り → 再呼び出し）の基本構造を知っている
* Python 程度の読み書きができる（本記事の擬似コードは Python 風）

**扱わないこと：**

* 特定フレームワーク（LangChain, LlamaIndex など）の API 詳細
* ベクトル DB の構築・運用手順
* プロンプトエンジニアリングの基礎

## 背景

### Arise エージェント Alex の自己崩壊

この記事の出発点は、Sally-Ann Delucia（Arise Head of Product）による講演「How we solved Context Management in Agents」（2026-05-10 公開）で語られた実話だ。

Arise が開発したエージェント Alex は、観測データを分析するタスクを担っていた。そのデータとは、自分自身のトレーススパンだ。ここで問題が生じた。

> 「データを分析しているシステムが、そのデータ自体によって制約されていました。」— Sally-Ann Delucia [04:43]

崩壊の構造はこうだ：

```
分析タスク実行
  → トレーススパン増大
  → コンテキスト上限超過
  → Alex 失敗
  → 失敗ログがさらにスパン増大
  → 再び上限超過 → 再失敗
  → ループ継続…
```

誰かがバグを仕込んだわけではない。設計の構造的問題だった。「自分の監視データを処理するエージェント」というユースケース固有の問題とも言えるが、同種の自己参照ループはどんなエージェントにも潜在する。

### なぜエージェントのコンテキストは自然に腐っていくのか

エージェントが会話を続けるほど、コンテキストウィンドウにはデータが積み上がっていく：

* ユーザー発話履歴
* ツールコールとその結果（多くは冗長）
* 中間推論ステップ
* エラーログ・デバッグ情報

これらは放置すると LLM プロバイダーのトークン上限に達し、システムが破綻する。問題を「プロンプト改善」で解こうとすると本質を外す。Delucia の言葉を借りれば：

> 「最高のコンテキスト戦略とは、エージェントが必要なことを記憶し、そうでないことを忘れさせてくれるものです。」— Sally-Ann Delucia [02:16]

これは「コンテキストエンジニアリング」と呼ばれる設計領域の話だ。

---

## 全体アーキテクチャ：コンテキストとメモリの役割分離

解決の核心は概念の分離にある。

> 「コンテキストはモデルが何を見るかを決定し、メモリは何が生き残るかを決定します。」— Sally-Ann Delucia [07:48]

これを設計に落とすとこうなる：

3 つの責任がある：

1. **コンテキストウィンドウの管理**（何を今 LLM に見せるか）
2. **外部メモリへの永続化**（何を消えないように保存するか）
3. **処理の委譲**（どの処理をサブエージェントに任せるか）

---

## 実装パターン 1：Head+Tail truncation

### なぜ先頭と末尾なのか

まず、Arise が試みて失敗した 2 つのアプローチを確認する。

**失敗アプローチ A：先頭打ち切り（先頭のみ保持）**

> 「（単純な打ち切りにより）エージェントは結局すべてを忘れてしまいました。フォローアップは新しい会話のように見えました。」— Sally-Ann Delucia [05:43]

会話の先頭（システムプロンプト・最初の指示）だけ残すと、最近の文脈が完全に失われる。

**失敗アプローチ B：LLM による要約**

LLM に中間コンテキストを要約させる試みも機能しなかった（[05:16]）：

* 何が重要かを LLM が制御できない
* 特定のツールコール結果など重要な詳細が無音で消える
* 結果が一貫しない

これらの失敗を踏まえて採用されたのが Head+Tail 戦略だ。

### 擬似コード

```
def smart_truncate(messages: list[dict], max_tokens: int) -> list[dict]:
    """
    Head+Tail truncation
    - head: システムプロンプト + 会話の冒頭（最初の指示・文脈）
    - tail: 直近の会話（最新の文脈）
    - middle: 外部メモリに保存し、コンテキストから除外
    """
    if count_tokens(messages) <= max_tokens:
        return messages  # 上限未満なら何もしない

    system_messages = [m for m in messages if m["role"] == "system"]
    conversation = [m for m in messages if m["role"] != "system"]

    # 経験則: 登壇者は「先頭100 / 末尾100」を使用（後述の注意事項を参照）
    HEAD_SIZE = 100   # ← この数値は自分のユースケースで要検証
    TAIL_SIZE = 100   # ← この数値は自分のユースケースで要検証

    head = conversation[:HEAD_SIZE]
    tail = conversation[-TAIL_SIZE:]
    middle = conversation[HEAD_SIZE:-TAIL_SIZE]

    # 中間部分を外部メモリに保存（パターン2で詳述）
    if middle:
        save_to_external_memory(middle)

    return system_messages + head + tail

def deduplicate_and_compress(messages: list[dict]) -> list[dict]:
    """
    重複メッセージや長いツールコール出力を圧縮する前処理
    - 重複したシステムプロンプトを除去
    - 長いツールコール結果をサマリー参照に置き換える
    """
    seen_content = set()
    result = []
    for msg in messages:
        content_hash = hash(str(msg.get("content", "")))
        if content_hash not in seen_content:
            seen_content.add(content_hash)
            result.append(msg)
    return result
```

### 設計判断のポイント

* **システムプロンプトはリセットしない**：対話の連続性はシステムプロンプトが保持している。head に必ず含める
* **重複メッセージを前処理で除去する**：ツールコールが繰り返されると同じ出力が積み上がる。truncation 前に除去する
* **100 という数値は経験則**：後述の「落とし穴」を参照

## 実装パターン 2：外部メモリへの中間保存

### なぜ外部メモリが必要か

Head+Tail 戦略で切り捨てた中間部分は「ゴミ」ではない。後から参照が必要になる可能性がある。外部メモリ（ベクトル DB や KV ストアなど）に保存し、エージェントが必要なときに検索して取得する設計が必要だ。

### 擬似コード

```
class ExternalMemoryStore:
    """外部メモリ：コンテキストウィンドウから溢れた情報の永続化"""

    def save(self, messages: list[dict], session_id: str) -> str:
        """中間メッセージを外部ストアに保存し、参照 ID を返す"""
        memory_id = generate_id()
        payload = {
            "id": memory_id,
            "session_id": session_id,
            "messages": messages,
            "timestamp": current_timestamp(),
            "embedding": embed(messages)  # 検索用ベクトル
        }
        self.store.upsert(memory_id, payload)
        return memory_id

    def search(self, query: str, session_id: str, top_k: int = 5) -> list[dict]:
        """クエリに関連する過去のメッセージを検索して返す"""
        query_vector = embed(query)
        results = self.store.search(
            vector=query_vector,
            filter={"session_id": session_id},
            top_k=top_k
        )
        return [r["messages"] for r in results]

def build_context_with_memory(
    current_messages: list[dict],
    user_query: str,
    memory: ExternalMemoryStore,
    session_id: str
) -> list[dict]:
    """
    現在のコンテキスト + 外部メモリからの関連情報を組み合わせる
    """
    truncated = smart_truncate(current_messages, max_tokens=CONTEXT_LIMIT)

    # 現在のクエリに関連する過去のメモリを取得
    relevant_memories = memory.search(user_query, session_id)

    # メモリをコンテキストに注入（head の直後に挿入）
    system = [m for m in truncated if m["role"] == "system"]
    rest = [m for m in truncated if m["role"] != "system"]

    memory_block = format_as_context(relevant_memories)
    return system + [memory_block] + rest
```

### 設計上の注意

* **セッション ID でメモリを分離する**：複数ユーザー・複数セッションが混在すると情報漏洩リスクがある
* **注入量を制御する**：外部メモリをすべて注入するとコンテキストが再び爆発する。`top_k` で関連度上位のみを選ぶ
* **このパターンはセッション内メモリに限定**：セッションをまたぐ長期記憶は現時点で未実装（後述の「未解決課題」を参照）

## 実装パターン 3：重処理サブエージェントへの委譲

### なぜサブエージェントに分けるのか

Alex の崩壊の直接原因は「対話のメインエージェントが重いデータ処理まで担っていたこと」にあった。

> 「ゲームチェンジャーでした。私たちは多くのサブエージェントを展開しました。」— Sally-Ann Delucia [10:50]

登壇者の主観評価ではあるが、変更の前後はこうだ：

* **変更前**：単一エージェントがチャット履歴 + 重いデータ検索の両方を担当 → コンテキスト爆発
* **変更後**：メインエージェント（対話・軽量文脈管理）+ サブエージェント（重いデータ処理） → コンテキストを分離

マイクロサービスの責任分離と同じ思想だ。

### 擬似コード

```
class MainAgent:
    """
    対話と軽量コンテキスト管理に専念するメインエージェント。
    重い処理はサブエージェントに委譲し、結果のみを受け取る。
    """

    def __init__(self, sub_agents: dict[str, "SubAgent"]):
        self.sub_agents = sub_agents
        self.memory = ExternalMemoryStore()

    def run(self, user_message: str, session_id: str) -> str:
        context = self.load_context(session_id, user_message)

        # LLM に判断させる：このタスクはどのサブエージェントに委譲すべきか
        routing_decision = self.llm_call(
            context=context,
            instruction="このタスクを自分で処理すべきか、サブエージェントに委譲すべきかを判断し、委譲先を返せ"
        )

        if routing_decision.delegate_to:
            sub_agent = self.sub_agents[routing_decision.delegate_to]
            # サブエージェントには必要な情報のみを渡す（コンテキスト全体は渡さない）
            result = sub_agent.run(
                task=routing_decision.task_spec,
                relevant_data=routing_decision.required_data
            )
            # サブエージェントからは「結果のみ」を受け取り、メインコンテキストに追加
            return self.llm_call(context=context, tool_result=result.summary)
        else:
            return self.llm_call(context=context)

class HeavyDataSubAgent:
    """
    大量データ検索・集計など、コンテキストを大量消費する処理を担うサブエージェント。
    メインエージェントのコンテキストを汚染しない。
    """

    def run(self, task: TaskSpec, relevant_data: dict) -> SubAgentResult:
        # サブエージェントは自分のコンテキストウィンドウで重い処理を行う
        raw_result = self.process(task, relevant_data)
        # 結果をサマリーに圧縮してメインエージェントに返す
        return SubAgentResult(
            summary=summarize(raw_result),
            full_result_ref=save_to_storage(raw_result)  # 必要なら参照 ID で取得可能
        )
```

### インターフェース設計の原則

* **サブエージェントにはコンテキスト全体を渡さない**：必要なタスク仕様と最小限のデータのみを渡す
* **結果はサマリーで返す**：生のデータをそのままメインコンテキストに注入するとコンテキスト爆発が再発する
* **フルデータは参照 ID で保存**：詳細が必要なときだけ外部ストアから取得できる形にする

## 検証と落とし穴

### 落とし穴 1：「100 文字」という数値を信じすぎる

Head+Tail 戦略における「先頭 100 文字 / 末尾 100 文字」は、登壇者の経験則に過ぎない（[11:18]）。

* コンテキストの品質を定量的に評価する指標は業界全体で確立されていない
* コンテキスト予算配分の体系的アプローチも現時点では存在しない
* ユースケース（対話型 / 処理型 / 検索型）によって最適値は異なる

**自分のユースケースでやるべきこと：**

```
# ロングセッション評価の骨格：後半の応答品質劣化を検出する
def long_session_eval(agent, scenario: list[str], turn_threshold: int = 20) -> dict:
    """
    20ターンを超えた会話での品質劣化を検出する（数値は登壇者の経験則値 [11:18]）。
    閾値は自分のユースケースで調整すること。
    """
    results = []
    for turn, message in enumerate(scenario):
        response = agent.run(message)
        quality_score = evaluate_response(
            response=response,
            reference=EXPECTED_ANSWERS[turn],
            context_coherence=check_context_continuity(response, scenario[:turn])
        )
        results.append({"turn": turn, "score": quality_score})

    # 後半 N ターンのスコアが前半と比較して劣化していないかを確認
    early_avg = mean([r["score"] for r in results[:turn_threshold // 2]])
    late_avg = mean([r["score"] for r in results[turn_threshold // 2:]])

    return {
        "early_avg": early_avg,
        "late_avg": late_avg,
        "degradation": early_avg - late_avg  # 正値 = 後半に品質劣化あり
    }
```

### 落とし穴 2：自己参照型データ処理のループ

Alex が陥ったパターンは再現しやすい。自分のログ・トレース・メトリクスを扱うエージェントを設計するとき：

* 処理の成功・失敗が自分のデータソースを増やす
* データが増えるほどコンテキストが膨らむ
* コンテキスト上限に達して失敗し、また失敗ログが積まれる

対策：自己参照型エージェントでは処理前にデータサイズ上限チェックを入れ、上限超過時にはサブエージェントへの委譲を強制するか、データを事前にフィルタリングする。

### 落とし穴 3：LLM 要約の「無音の情報消失」

LLM に要約させると「何が消えたかが分からない」。特に：

* 特定のツールコール結果（数値・ID・状態遷移）
* エラーメッセージの詳細
* 後続のステップで必要になる中間計算結果

これらは要約で自然に消える。「重要かどうかの判断を LLM に委ねない」ことが原則で、明示的なルール（「ツールコール結果は必ずそのまま保持する」など）を実装側で定義する。

### 落とし穴 4：Claude Code の観察について

Sally-Ann Delucia の観察によれば、Claude Code のコードから Anthropic も同様の打ち切り＋圧縮戦略を使用していることが確認されたとされている（[13:31]）：

> 「Claudeのコードリークで、彼らが我々と同様の打ち切りと圧縮戦略を使っていることに驚きました。何か秘訣が得られるかと期待していたのですが。」— Sally-Ann Delucia [13:31]

**重要な留保：これは登壇者個人の観察であり、Anthropic の公式見解ではない。** ただし、同様の戦略が複数の独立した実装で採用されているという事実は、「業界全体がまだコンテキスト管理の標準解を持っていない」という現状の補強材料にはなる。

## 未解決課題（正直な限界の開示）

現時点で Arise の実装でも解決されていない問題が 3 つある（[11:18]）：

### 1. セッションをまたぐ長期記憶

現状の実装はセッション内に限定されている。ユーザーが新しいチャットを開始すると、過去の対話内容は記憶されない。「真の長期記憶」は次のフロンティアとして位置づけられているが、本番実装はまだない。

### 2. 100 文字という数値の正当化

「先頭 N 文字 / 末尾 N 文字」という数値をどう決めるかの体系的な方法論がない。コンテキスト品質の定量評価指標も確立されていない。つまり、現状は全員が経験則で戦っている。

### 3. 巨大コンテキスト問題

Head+Tail 戦略とサブエージェント分離を組み合わせても、非常に大きなプロンプトや入力が与えられた場合、LLM プロバイダーのトークン上限に達してシステムが破綻するリスクは残る。巨大コンテキストは依然として未解決問題だ。

## まとめと次に試すこと

### 再利用できる設計原則

* **コンテキスト（今見せるもの）とメモリ（永続化するもの）を役割で分ける**
* **単純打ち切りと LLM 要約はどちらも失敗する。Head+Tail + 外部メモリが現時点のベターケース**
* **単一エージェントに対話と重処理を同居させない。責任を分割する**
* **自己参照型エージェント（ログ・トレース処理）は特に慎重に設計する**
* **数値（Head/Tail サイズ、セッション長）は推奨値として受け取らず、自分のユースケースで検証する**

### 次のアクション（優先度順）

1. **ロングセッション評価の実装**：今のエージェントで 20 ターン超えの会話をシミュレートし、後半の品質劣化を測定する。数値 20 は登壇者の経験則値（[11:18]）であり、自分のユースケースに合わせて調整すること
2. **コンテキストサイズの可視化**：各ターンでのトークン数をログに記録し、上限到達タイミングを把握する
3. **Head+Tail の数値を探索**：HEAD\_SIZE / TAIL\_SIZE を 50 / 200 など変化させて応答品質への影響を評価する
4. **サブエージェントの委譲境界を定義する**：「何トークン以上の処理はサブエージェントに委譲する」という閾値と判断ロジックを設計する
5. **長期記憶の要件を整理する**：セッションをまたぐ記憶が自分のプロダクトで本当に必要かを確認し、必要なら外部 DB との統合設計を始める
