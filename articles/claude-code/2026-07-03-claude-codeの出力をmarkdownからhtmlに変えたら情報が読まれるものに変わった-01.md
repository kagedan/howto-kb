先日、AnthropicのClaude Codeチームメンバー・Thariq氏が書いた記事「**Using Claude Code: The Unreasonable Effectiveness of HTML**」がXで1,100万インプ超えのバズを記録しました。

要約すると「**Claude Codeの出力はMarkdownじゃなくHTMLにしろ**」という話です。

最初は「それだけ？」と思いました。が、実際に試してみたら、**情報が「存在するもの」から「読まれるもの」に変わった**感覚があって、これは本物だと確信しました。

この記事では、HTMLファースト出力の**メリット・デメリット両方を正直に**書きつつ、実際のプロンプト例・CLAUDE.mdへの設定方法まで紹介します。

---

## なぜMarkdownでは限界なのか

Thariq氏はこう書いています。

> "I find it difficult to read a markdown file of more than a hundred lines."  
> （100行を超えるMarkdownファイルを読むのは難しいと感じる）

心当たりありませんか？Claude Codeに設計書を書かせたら200行のMarkdownが出てきて、半分も読まずに閉じた経験。

さらにThariq氏が指摘するのが、**「人間が手で編集する」というMarkdownの最大のメリットが、もはや薄れている**という点です。

今は自分で仕様書を編集するより「ここ直して」とClaudeに頼む方が速い。つまりMarkdownの設計思想（人間が読み書きしやすい）が、AIとの協働では必ずしも最適ではなくなっています。

---

## HTMLファースト出力のメリット・デメリット

正直に両方書きます。

### ✅ メリット

* 情報密度が圧倒的に高い（タブ・カラー・図解）
* 実際に読まれる（Slackに添付したMarkdownより閲覧率が高い）
* URLで即共有できる（S3やVercelに置くだけ）
* 双方向操作ができる（スライダー・ボタン等）
* 図・フロー・グラフをSVGで埋め込める
* レスポンシブ対応でモバイルでも読める
* デザインで情報の優先度を伝えられる

### ❌ デメリット

* 生成時間がMarkdownの2〜4倍かかる
* トークン消費が多い（体感1.5〜2倍）
* git diffが読みにくい（バージョン管理が不向き）
* 手で直接編集しづらい
* grepや全文検索がしにくい
* Notionなど他ツールへの貼り付けが面倒

## HTML vs Markdown — どう使い分けるか

| 用途 | 推奨 | 理由 |
| --- | --- | --- |
| 計画書・設計書・仕様書 | HTML ✅ | タブ・図解で整理できる |
| コードレビュー・PR説明 | HTML ✅ | diff＋注釈＋重要度カラーが作れる |
| プロトタイプ・デザイン確認 | HTML ✅ | スライダーでパラメータを即調整できる |
| 共有する報告書・レポート | HTML ✅ | URLで誰でもブラウザで開ける |
| Zenn/Qiitaに投稿する記事 | Markdown ⚠️ | プラットフォームがMarkdown前提 |
| git管理する仕様書 | Markdown ⚠️ | diffが読みやすくレビューしやすい |
| チャットの短い返答 | Markdown ⚠️ | HTMLにする必要なし |
| Notionに貼る予定のメモ | Markdown ⚠️ | 貼り付け時の変換が面倒 |

## 設定方法 — CLAUDE.mdに1回書くだけ

毎回プロンプトに「HTMLで」と書かなくて済むよう、`~/.claude/CLAUDE.md`（グローバル設定）に追記します。

```
# 出力フォーマット方針

計画書・仕様書・レポート・コードレビュー・設計案など「人が読む成果物」は、
MarkdownではなくHTMLファイルで出力すること。

- タブ・カラーコーディング・図解・SVGフロー図などを活用して情報密度を高める
- <script> や CSS でインタラクティブにする（スライダー、コピーボタン、タブ切り替え等）
- ブラウザで開ける .html ファイルとして保存する
- デザインはシンプルかつ読みやすく。ダークテーマ推奨
- 短い回答・コードスニペット・会話の返答はこの限りではない
```

## ユースケース別プロンプト例

### 設計・計画フェーズ

複数の選択肢を一画面で比較したいとき。

```
オンボーディング画面の方向性を決めたい。
6つの全く異なるアプローチをHTMLファイルで生成して。
レイアウト・トーン・情報密度をそれぞれ変えて、
1ファイルのグリッドで並べて比較できるようにして。
各案のトレードオフをラベルに書いて。
```

### コードレビュー

GitHubのdiff画面より読みやすいレビューを作る。

```
このPRをレビューするHTMLファイルを作って。
ストリーミング/バックプレッシャーのロジックを重点的に。
実際のdiffをインライン注釈付きで描画して、
重要度で色分けして（🔴高リスク / 🟡要確認 / 🟢OK）。
```

### デザイン検討

「もうちょっと速く」をスライダーで伝える。

```
新しいチェックアウトボタンのプロトタイプを作りたい。
クリックでアニメーションして色が変わる。
スライダーでアニメーション速度・イージング・色を
リアルタイムで調整できるHTMLファイルを作って。
気に入ったパラメータをコピーするボタンもつけて。
```

### 技術の学習・理解

コードベースを読んで1枚のHTMLで解説させる。

```
このレートリミッターの仕組みがわからない。
関連コードを読んで、1枚のHTML解説ページを作って。
・トークンバケットのフロー図（SVG）
・重要なコードスニペット3〜4個に注釈付き
・最後に「落とし穴」セクション
1回読んで理解できるように最適化して。
```

### カスタム編集UI

「その作業のためだけ」のUIをその場で作る。

```
30件のチケットの優先順位を整理したい。
各チケットをドラッグ可能なカードにして、
Now / Next / Later / Cut の4列に分けるHTMLファイルを作って。
「Markdownとしてコピー」ボタンで最終順序をエクスポートできるようにして。
```

## まとめ

* ✅ **HTMLを使うべき場面：** 人に見せる成果物、情報量が多い設計書、共有したいレポート、デザインの比較検討
* ⚠️ **Markdownのままで良い場面：** git管理する仕様書、Zenn/Qiitaへの投稿、Notionに貼るメモ、チャットの短い返答
* 🔧 **設定は1回だけ：** `~/.claude/CLAUDE.md` に書いておけば毎回指示しなくていい

Thariq氏の言葉を借りると：

> "I feel much more in the loop than ever before when using HTML."  
> （HTMLを使うことで、かつてないほど「ループの中にいる」と感じる）

AIに任せきりにするのではなく、出力をHTMLにすることで「人間が理解・判断・フィードバックする」サイクルが回ります。これがAIとの協働の本質だと思います。

まずは次にClaude Codeに計画書を作らせるとき、「HTMLで作って」と一言加えてみてください。
