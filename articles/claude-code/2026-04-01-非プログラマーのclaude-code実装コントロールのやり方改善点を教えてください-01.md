## はじめに

私はデザイナーです。これまでにプログラミングには何度もとりかかっては挫折してきました。

昨年にAIを先生として、もはやn回目（←カウントできない）となるプログラミング学習を始め、今回はApp Storeでプロダクトをリリースすることが出来ました。メトロノームとインターバルタイマーをかけ合わせたアプリです。

**「TempoRound」**  
<https://apps.apple.com/app/id6759990980>

最初はVibe Codingでプロトタイプを作成しました。問題はその後で、「ここをもっとこうしたい」と手を入れ始めると、自分がすでにOKを出したところに勝手な変更が入ったり、直した箇所と別のところが期待通りに動かなかったり。やりたいことは明確なのに、意図した通りに進まない。それをなんとか乗り超えてアウトプット出来ました。

この先も個人開発を続けていこうと思っていますが、なにぶん一人でやっているため、現在のやり方がベストかどうか常に暗中模索です。  
ぜひ改善点や、より効率的なやり方があれば百戦錬磨のみなさんからご指南いただけると嬉しいです。

## 役割の分担

開発は以下の3つの役割で進めました。

* **自分（ディレクター/デザイナー）**：何を作るか決め、デザインをする。実機で検証する。提案に判断を下す
* **Claude.ai（プランニングパートナー）**：仕様を一緒に練る。画面プロトタイプを作る。Claude Code用のプロンプトを生成する
* **Claude Code（実装担当）**：実際のファイルを操作してコードを書く

Claude.aiは対話的に仕様を練るのに向いていて、Claude Codeは実ファイルの読み書きに向いています。それぞれに渡す情報の粒度が違うので、1つのツールで全部やると精度が落ちることに気づき「思考」と「実行」を分離した構造にしました。

## コントロールの仕組み

ここが本題です。私が取り組んだことを6つ紹介します。

### 1. プロジェクト機能で情報をプールする

Claude.aiの「プロジェクト」に関連する会話をまとめておくと、過去の会話を横断検索でき、メモリ（記憶）も蓄積されます。毎回ゼロから説明しなくてよくなるのが一番ありがたいところです。

プロジェクト機能を使わず散発的に会話していた初期は、同じ質問を繰り返したり、以前決めた仕様と矛盾する提案が返ってきたりしていました。情報をプールするようにしてから、会話の立ち上がりが格段に速くなりました。

### 2. 1つの機能＝1つの会話で区切る

地味ですが最も効果が大きかった仕組みです。

長い会話の中で複数の機能について話すと文脈が混ざって、AIの回答精度が目に見えて落ちます。1機能1会話に閉じたら、的外れな回答が減りました。

ただし会話を分けると文脈が引き継がれません。そこで「ハンドオフプロンプト」を使っています。会話の終わりに、次の会話に必要な文脈をMarkdownにまとめてもらい、次の会話の冒頭に貼ります。

```
# [機能名] — ハンドオフプロンプト

## 確定した仕様
前の会話で決まったこと（画面構成、挙動、制約）

## 未決事項
次の会話で決める必要があること

## 技術的な制約・注意点
実装時に気をつけるべきこと（過去にハマったポイント含む）

## 次の会話でやること
具体的な作業リスト
```

ポイントは「確定した仕様」と「未決事項」を明確に分けること。これがないと、決めたはずのことをまた議論し直す羽目になります。

### 3. CLAUDE.mdを生きた仕様書として育てる

開発の中盤くらいから仕様書駆動開発という概念を知り、CLAUDE.mdをメンテし始めました。

Claude Codeはプロジェクトルートの `CLAUDE.md` を作業開始前に最初に読みます。ここにルールを書いておけば、毎回プロンプトで指示しなくても従ってくれます。

使っていて感じたのは、構造がかなり大事だということです。Claude Codeは先頭に近い情報を優先参照する傾向があるので、先頭にコマンド（ビルド方法、テスト手順）とトラップ（過去のハマりポイント）を置いて、詳細仕様は後半にしました。

トラップはIf-Then形式で書くと効きます：

```
- AudioContextを生成する前 → 必ず setAudioSessionOptions() と
  setAudioSessionActivity(true) を手動で呼ぶこと。
  理由：AudioEngineの内部setActiveが他アプリの音声を中断するため
```

「いつ」「何をするか」「なぜか」の3点セットです。「なぜ」がないと、Claude Codeが「この制約はもう不要では」と判断してスキップすることがありました。

\*\*CLAUDE.mdの更新を怠った時期が、手戻りの多い時期とぴったり一致していました。\*\*仕様が変わったのにCLAUDE.mdが古いままだと、古い仕様で実装されます。書いて終わりではなく、実装が進むたびに更新するのが大切だと身をもって知りました。

### 4. 調査と実装を分ける

Claude Codeに「直して」と言うと、すぐにコードを書き始めます。でも現状を把握しないまま修正すると、別の場所に影響が出たり、そもそも根本原因が違っていたりします。

そこで、まず「調査だけ」をさせるようにしました。

```
## 調査プロンプト（read-only）

以下の状況を調査してください。
コードの変更は一切行わないでください。
計画ファイルの作成も不要です。

### 調べてほしいこと
- [現在の問題の状況]
- [関連するファイルの構造]

### 出力
- 現状の整理
- 考えられる原因
- 修正方針の選択肢（メリット・デメリット付き）
```

「コードの変更は一切行わないでください」を明示的に書くのがコツです。これがないと善意で修正を始めます。

調査結果を見てから実装プロンプトを作ることで、「何を変えるか」「何を変えないか」を自分で判断できるようになりました。この一拍を置くだけで手戻りがかなり減りました。  
そして途中からClaude codeはPlanモードをデフォルト設定にしました。

### 5. 見た目を先に確定させる

Claude Codeでプロンプト入力だけでのUI実装で最も手戻りが多かったのは「作ってみたけど思っていたのと違う」でした。

そこで、まずはClaude.aiのアーティファクト機能でJSXプロトタイプを作り、ブラウザ上で見た目を確認するところから始めました。このとき、Claude.aiのプロジェクトにFrontend Designという外部のスキルファイル（UIの設計ルールやベストプラクティスをまとめた指示書）を読み込ませていました。これがあると、自分が「こんな感じの画面にしたい」と伝えるだけでも、デザイン的に整ったプロトタイプが出てきます。とはいえ結果的にホーム画面は7回作り直して、装飾を削ぎ落とす過程で最終形が決まりました。

JSXプロトタイプで方向性が固まったら、次はそれを見ながらFigmaでデザインを起こします。Figmaで色、余白、フォントサイズなどの数値を確定させて、FigmaのMCP連携を使ってClaude.aiがその数値を直接取得し、Claude Code向けのプロンプトに埋め込みます。Claude Codeには直接Figmaを参照する手段が当時なかったため（と言ってもほんの一ヶ月前の話ですが…！）、数値はプロンプトに明示的に書き出す必要がありました。

つまり「JSXプロトタイプで方向性を確認 → Figmaでデザインを確定 → 数値をClaude Codeに渡して実装」という流れです。見た目の確認を2段階に分けたことで、Claude Codeへの指示が「この見た目をこの数値で再現して」という具体的なものになり、曖昧さから生まれるズレがなくなりました。

### 6. lessons.mdでハマりポイントを蓄積する

実装→実機検証→問題発見、というサイクルを回していると、同じ種類のミスが繰り返し起きることに気づきました。そこで `tasks/lessons.md` というファイルをプロジェクトに作って、Claude Codeが実装中に踏んだ落とし穴や、自分のフィードバックで判明した問題を記録していくルールにしました。

```
# Lessons Learned

## Expo / React Native
- Expo SDK 54ではexpo-avの音声精度が不十分 → react-native-audio-apiに移行した
- dev clientとExpo Goでは挙動が異なる。音声系は必ずdev clientで検証

## Claude Codeとの協業
- プロンプトにライセンス全文を含めるとハングする → プレースホルダー＋フォローアップ
- Figma MCPのノードIDはURLの `-` をAPIでは `:` に変換が必要
```

ポイントは、**Claude Codeの新しいセッション開始時にこのファイルをレビューさせる**ことです。CLAUDE.mdに「作業開始前に `tasks/lessons.md` を確認すること」と一行書いておくだけで、過去にハマった問題の再発が減ります。

CLAUDE.mdとの使い分けですが、CLAUDE.mdには「必ず守るべきルール」を書いて、lessons.mdには「経験から得た知見」を時系列で蓄積しています。lessons.mdの中から重要度が高いものをCLAUDE.mdのトラップに昇格させる、という流れです。

## おわりに

ここ最近はClaude Codeのベスプラ的な記事も多く、ここで書いたことがもはやコモディティ化してる気もします。

とはいえ、仕様を言語化する。事象を正確に伝える。提案に判断を下す。実機で検証する。完全に実装した機能でも、UXが複雑すぎればリリースから外す…やっぱりモノを作ってアウトプットするのは楽しいですね。

アプリのアップデートも行いたいですし、新しいデジタルプロダクトもつくりたい。なので今の方法よりももっと効率化できそうだと思ってます。
