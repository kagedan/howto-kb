# 1. はじめに

こんにちは。アルダグラムで業務委託QAをしている miyashita です。

今回は、`qa-orchestrator` という AIエージェント（Claude Code の skill / agent 群）について、**なぜ作ったか / どう設計したか / 結果どうなったか** を一通り書きます。

> **3 行サマリ**
>
> * QA が開発のボトルネック化してきて、属人化も気になっていた
> * Claude Code 上で QAプロセスを担う AIエージェントを作った
> * 結果：テスト設計工数 **約 45% 削減**、クオリティ・フォーマットも一定化（ただし課題も残っている）

**AIをQAプロセスにどう組み込もうとしたか、その思考のログ** として読んでいただけると嬉しいです。同じ悩みを持つ方の「自分の現場ではどうしようか」のスタート地点になれば幸いです。

---

# 2. なぜqa-orchestratorを作ったのか — QAボトルネックと属人化の壁

まず、何に困っていたのかから書かせてください。問題意識は大きく3つあります。

## ① QAが、開発のボトルネックになり始めていた

AIの登場で、開発スピードはこの1年で明らかに加速しました。実装を始めてからQA依頼が来るまで、以前より早く回るようになっています。

一方で、**QAだけがその速度に置いていかれている**感覚がありました。実装は終わったけど、QAの手が回っていなくてテスト実施に着手できない…という状況が徐々に生まれ始めていました。

この「リリースしたいのにQAが終わらない」がじわじわ常態化していく兆候を見ていて、これは、QAの組織的な役割としてかなり**深刻な状態**だと感じていました。

## ②ChatGPTによるテスト設計自動化の限界

では、QAチームは全くAI活用できていなかったのかというと、そうではありません。テスト設計は以前からChatGPTによる自動化に取り組んでいました。（以下参照）

<https://zenn.dev/aldagram_tech/articles/3099d55dabd6ac>

この取り組みによって確かに成果物作成の作業自体は早くなりました。

しかし、一つ大きな落とし穴がありました。「**足りない情報を人間が手動で補完していた**」という点です。

不足した情報があった時に、人間がプロンプトでAIに修正や追加インプットの指示を出したり、補完資料を手動で作成して読ませていたのです。

結果、テスト設計の作業工数は早くなったものの、別で時間を取られてしまい作業期間はトントンなんてことに。

そのため、もっと抜本的な解決策を練る必要性を感じていました。

## ③ QAスキルの属人化

もうひとつ、ずっと引っかかっていたのが**属人化**です。

### クオリティの属人化

AI活用についてチーム内で「こういう取り組みいいよ！」「こんなカスタムAI作ってみたよ！」を共有して横展開しても、同じクオリティの成果物にならない。  
結局、個人のスキル・経験・センスに依存してしまっていました。

### ナレッジの分散

ナレッジが個人の頭の中に閉じこもり、組織としての財産として定着できない。  
一部はドキュメント化されているものの、情報が分散するため参照コストが高いのです。

## ④ 成果物のフォーマットがバラバラ

現在、QAチームが作成する成果物のフォーマットは、最低限の決め事はあるものの、担当者ごとに少しずつカスタムされており、完全には固定化されていません。

その結果、**レビュアーが毎回フォーマットに適合する**ことが必要になります。

細かいですが、ここもスピードが鈍化する要因となっています。

## どう解決しようと思ったか

この4つを同時に解決しようとしたのが、`qa-orchestrator` の作成です。

「属人化しているプロセスや知見をskill化してしまえば、みんな同じ水準になるかも？」というありがちな発想が出発点でした。

1. **インプットドキュメントの自動解析**
2. **テスト成果物作成の自動化**

正直、テスト実施の自動化までやりたい気持ちはありました。でも、最初から欲張ると確実に作り切れません。なので、**まずは設計工程まで**にスコープを絞りました。

# 4. qa-orchestratorの全体像 — 何でできているか

`qa-orchestrator` は、Claude Code上で動く **skill** と **agent** の集合体です。ざっくり言うと、

* **orchestrator役のskill**: ユーザーとの対話を仕切り、どのプロセスを実行するかを管理する
* **プロセスごとのagent**: 仕様レビュー、影響分析、テスト分析、機能調査、観点設計、ケース設計、データ設計、計画策定…と、QAプロセスの工程ごとに分かれている
* **セッション情報ファイル**（`qa-session.json`）: 進捗とインプットを保持する

まさに、`qa-orchestrator` という下請け組織を作ったというイメージです。

## 全体構成のイメージ

ユーザーから見ると、入り口は1つだけです。Claude Code上で `/qa-orchestrator`と呼び出すと、その先で「何をしたいか」と問われ、「どのプロセスを実行するか」「どの資料をインプットにするか」を順に聞かれます。選んだ内容に応じて、orchestrator が裏側で対応する agent を呼び出し、成果物を Notion や手元のドキュメントに書き出していく流れです。

ユーザー自身が個別の agent を直接叩くことは基本的にありません。常に orchestrator を入り口に置き、agent はその指示で動く構造にしています。こうすることで「どの順番でどのプロセスを走らせるか」の判断がブレないようにしています。

![](https://static.zenn.studio/user-upload/f08ea24a9738-20260513.png)

## skill同士の呼び出し関係

各 agent は独立して動くわけではなく、**前段の成果物を読み込んだうえで、自分の工程の出力を作る**前提で設計しています。たとえば観点設計の agent は、その前に行われている「テスト分析」「影響分析」「機能調査」の出力をすべて参照する。これによって、いきなり観点を出すよりも、状況を踏まえた厚みのある観点が出せるようになります。

orchestrator はこの呼び出し順序を裏で管理していて、ユーザーが「次のプロセスへ進む」を承認すると、必要な前段の成果物を集めて次の agent に渡してくれます。

![](https://static.zenn.studio/user-upload/9af2eed9b541-20260513.png)

## 作業フローと skill / agent 一覧

qa-orchestrator では、以下の Phase 順に skill / agent が起動します。承認ゲートは省略していますが、各 Phase の区切りで人間の承認が入る前提です。

（実行Phaseをカスタムすることもできます）

| **Phase** | **skill / agent** | **プロセス** | **AIがやること** | **人間がやること** | **インプット** | **アウトプット** |
| --- | --- | --- | --- | --- | --- | --- |
| 0 | qa-orchestrator (親) | 初期ヒアリング | ユースケース特定・資料収集・実行計画立案 | インプット資料の指定・実行計画の承認 | ユーザーの初回指示 | qa-session.json（実行計画） |
| 1 | defect-analysis | 不具合傾向分析 | 過去の不具合チケットを分析しパターンを抽出 | 分析結果のレビュー・承認 | テスト分析ページ URL | 不具合傾向サマリー |
| 1 | impact-analysis | 影響分析 | コード変更による影響範囲を整理 | 影響範囲のレビュー・承認 | PR / チケット / 仕様書 | 影響分析レポート |
| 2 | test-analysis | テスト分析 | テストタイプ・スコープ・リスク評価を分析 | 分析結果の確認 | 仕様書 / 不具合傾向サマリー / 影響分析レポート | テスト分析結果 |
| 3 | spec-review（モード1） | 仕様レビュー | 仕様の曖昧箇所検出 + UI/UX改善提案 | 指摘への対応判断・仕様へのFB | 仕様書 / テスト分析結果 / 機能調査レポート（あれば） | 仕様レビュー結果（モード1） |
| 4 | test-planning | テスト計画 | スコープ・スケジュール・体制を策定 | 計画の承認 | テスト分析結果 / 影響分析レポート | テスト計画書 |
| 5 | feature-investigation | 機能調査 | Figma + コードからUI・バリデーション・権限制御を抽出 | 調査結果の確認 | Figma URL / プラットフォーム / 仕様書 / 機能タイプ | 機能調査レポート |
| 6 | test-viewpoint | テスト観点抽出 | 「何をテストすべきか」を網羅的に抽出 | 抽出結果のレビュー・追加・削除 | テスト分析結果 / 機能調査レポート / 影響分析レポート / 観点チェックリスト等 | テスト観点一覧 |
| 7 | spec-review（モード2） | 期待結果レビュー | 期待結果の曖昧箇所検出 + 具体化提案 | 指摘への対応判断 | テスト観点 / 機能調査レポート / 仕様書 / モード1結果（あれば） | 期待結果レビュー結果（モード2） |
| 8 | test-case | テストケース作成 | 観点から具体的なテスト手順・期待結果を展開 | テスト可能性・実現性の確認 | テスト観点 / 機能調査レポート | テストケース一覧（Web / App） |
| 9 | test-data | テストデータ設計 | テストアカウント・テストデータを設計 | データ準備の確認 | テストケース | テストデータ仕様 |
| 10 | qa-orchestrator (親) | 完了レポート | 全成果物のサマリを自動生成 | 最終確認 | 各Phaseの成果物URL | 完了レポート |
| 11 | qa-improvement | QA改善提案 | プロセス改善・skill改善を自動提案 | 提案の採否判断 | 1セッションの実行履歴 / ユーザーフィードバック | QA改善提案レポート |
| 随時 | review-export | レビュー・エクスポート | Notion成果物をMarkdownに変換 | エクスポート対象の指定 | 施策の Notion 成果物 | Markdown ファイル |

## 各 skill / agent の紹介

ここからは、それぞれが何をするものかを一つずつ紹介します。

### Phase 1 / defect-analysis — 不具合傾向分析

```
## このSKILLが行うこと

1. 不具合の記録（ID・深刻度・優先度・再現手順・期待動作）
2. 深刻度 / 優先度の分類
3. 根本原因分析（なぜ不具合が発生したか）
4. 再発防止策の定義（プロセス・コード・テスト改善）
5. 不具合サマリー（件数・ステータス・深刻度別集計）
6. Notionページ生成
```

過去の不具合チケットを機械的に分析し、機能領域別の不具合傾向・頻出パターン・根本原因のサマリーを作成します。「過去の地雷がどこに埋まっているか」を先に可視化することで、後のテスト観点設計で重点的に当てる領域を見極めやすくします。

### Phase 1 / impact-analysis — 影響分析

```
## このSKILLが行うこと

1. **変更対象の特定** — PR/チケット/仕様書から変更ファイル・コンポーネントを特定する
2. **コード依存関係の調査** — 変更コンポーネントを参照している既存画面・機能を検索する
3. **既存機能への影響サマリー** — 影響を受ける既存機能の一覧と影響内容を整理する
4. **自動テストへの影響サマリー** — 修正・追加・削除が必要な自動テストを特定する
5. **影響分析レポートの出力** — Notionページ（またはMarkdown）として出力する
```

PR・チケットと仕様書をインプットに、コード変更が既存機能・自動テストに与える影響範囲を整理します。「どの機能が巻き込まれるか」「自動テストはどこを書き直す必要があるか」を、テスト分析の前段で確定させる役割です。Phase 1 では qa-defect-analysis と並行で動きます。

### Phase 2 / test-analysis — テスト分析

```
## このSKILLが行うこと

1. **テスト対象の洗い出し** — 機能・非機能要件からテスト対象を列挙する
2. **テスト種別の分類** — 機能/性能/セキュリティ/ユーザビリティ等に分類する
3. **リスク評価** — 各テスト対象の「発生確率」×「影響度」でリスクレベルを判定する
4. **テスト方針の決定** — リスクに応じたテスト方針（重点テスト/標準/省略可）を定義する
5. **Notionページ生成** — 分析結果をNotionの構造化ページとして出力する
```

仕様書、不具合傾向サマリー、影響分析結果を読み込み、テスト対象の特定、リスク評価、テスト戦略の策定までを担当します。後続フェーズ全体の方向性を決める「司令塔」のような工程で、ここで決まったことがテスト観点・ケース・データすべてに影響します。

### Phase 3 / spec-review（モード1）— 仕様レビュー（仕様の曖昧箇所）

```
## このSKILLが行うこと

1. **曖昧箇所の検出** — 仕様書・テスト成果物から曖昧・未定義・矛盾を特定する
2. **曖昧さの分類** — カテゴリ別に整理する（用語・条件・動作・エラー・境界値・権限）
3. **影響度の判定** — テスト設計・実装への影響度を評価する
4. **UI/UX最適な仕様案の提案** — 各曖昧箇所に対してユーザー体験を考慮した具体案を提示する
5. **Notionページ生成** — レビュー結果をNotionに出力する
```

仕様書とテスト分析結果をインプットに、仕様の曖昧箇所を洗い出し、UI/UX 改善提案も添えて出力します。**「テスト設計に入る前に、仕様の穴を埋める」** ためのワンクッション。私が一番こだわった工程で、章5-1 で詳しく書きます。

### Phase 4 / test-planning — テスト計画

```
## このSKILLが行うこと

1. テストスコープの定義（対象範囲・除外範囲）
2. テストスケジュール作成（フェーズ別・工数見積もり）
3. テスト体制の定義（担当チーム・役割・責任）
4. 開始条件 / 終了条件の設定
5. テストリスクと対策の整理
6. 自動テストへの影響評価（既存自動テストの修正・新規追加の検討）
7. Notionページ生成
```

テストのスコープ、スケジュール、体制、開始/終了条件、リスク対応をまとめた計画書を出力します。テスト分析と影響分析の結果を踏まえて作成するので、プロジェクト関係者との合意形成にそのまま使えるレベルを目指しています。

（正直工数見積もりはまだまだ精度が低いので、現状ここは人間の作業となっています。）

### Phase 5 / feature-investigation — 機能調査

```
## このSKILLが行うこと

1. **Figma調査** — 画面構成・UI要素名・画面遷移をFigma MCPで確認する
2. **コードリポジトリ調査** — バリデーション・エラーメッセージ・URL・権限をサブエージェントで調査する
3. **プラットフォーム別の差分整理** — Web/iOS/AndroidのUI要素名の差異を明確にする
4. **機能調査レポートの出力** — 確認済み情報をNotionページとしてまとめる
```

このagentだけ2回登場します。

1回目では、Figma URL とリポジトリパス、仕様書をインプットに、UI 要素・状態遷移・バリデーション仕様・権限制御を網羅的に抽出して、機能調査レポートとしてまとめます。下流の qa-test-viewpoint / test-case が「推測ゼロ」で書ける状態を作る、**情報基盤として最重要**の工程です。

### Phase 6 / test-viewpoint — テスト観点設計

```
## このSKILLが行うこと

1. **テスト観点の洗い出し** — 画面/機能単位で確認すべき観点を列挙する
2. **観点の階層整理** — 大項目→中項目→小項目の3階層で構造化する
3. **リスクの判定** — 各観点のリスクレベル（高/中/低）を設定する
4. **自動化可否の判定** — 各観点の自動テスト適性を評価する
5. **Notionデータベースへの出力** — テスト観点DBにページを作成する
```

テスト分析・機能調査・影響分析の結果に、QAチーム独自のテスト観点チェックリストと外部の汎用ノウハウ（JSTQBシラバスの要約等）を掛け合わせて、網羅的なテスト観点を生成します。観点ごとにリスク評価も付与し、後段の優先度付けに使える形にします。

### Phase 7 / spec-review（モード2）— 期待結果レビュー

2回目の登場ですが、ここでは入力と検出対象が異なります。テスト観点レベルの「期待結果として明文化できない曖昧箇所」を洗い出し、具体化案を提示する役割です。テストケース化の前に最後の曖昧さを潰します。

### Phase 8 / test-case — テストケース設計

```
## このSKILLが行うこと

1. **テスト観点→テストケースへの展開** — 観点の備考欄パターンごとにケース化する
2. **実施条件の定義** — ログインユーザー・事前データ・事前操作を明確にする
3. **実施手順の具体化** — 番号付き操作手順を記述する
4. **期待結果の明確化** — 曖昧表現を排除し、具体的な状態・値・表示を記述する
5. **Notionデータベースへの出力** — テストケースDBにページを作成する
```

テスト観点を、実行可能なケースに展開します。前提条件・手順・期待結果を **1ケース 1検証点** で記述し、「正常に」「仕様通り」等の曖昧表現は禁止。Web と App はデータベースを分けて個別出力しています。

現状はNotionDBに出力していますが、後述するレビュー問題やコンテキスト消費問題に直面し、今後はCSV出力に落ち着こうとしています。

### Phase 9 / test-data — テストデータ設計

```
## このSKILLが行うこと

1. テストデータの種別分類（正常系・異常系・境界値・性能テスト用）
2. データ内容の定義（具体的な値・フォーマット）
3. データ作成方法の決定（手動/スクリプト/ファクトリ/DBシーディング）
4. データ管理方針の定義（有効期限・クリーンアップ方法）
5. 個人情報・機密データの取り扱いガイドライン
6. Notionページ生成
```

テストケースの実行に必要なテストデータ（種別・値・作成方法・テストアカウント）を設計します。設計後、テストケース側の「ログインユーザー」欄に仮置きされていたアカウントを実値で更新します。

### Phase 11 / qa-improvement — QA改善提案

```
### 提案タイミング

以下のタイミングで改善提案を行う：

1. **QA完了後**（qa-orchestratorのPhase 9完了後） — 今回の実行を振り返り改善案を提案
2. **ユーザーが明示的に要求した時** — 「改善提案して」「振り返りをして」
3. **実行中に問題を検出した時** — コンテキスト不足、情報取得失敗等
```

1 セッションの QA 活動を振り返り、(A) コンテキスト効率 / (B) 出力品質 / (C) プロセス効率 / (D) 新規 skill 提案、の4観点で改善案を提示します。承認された改善は、その場で skill ファイルに直接反映できるのが特徴です。**qa-orchestrator が自己進化していくための仕組み**として位置づけています。

もちろん、途中で改善指示をした場合も動いてくれます。

### 随時 / review-export — レビュー・エクスポート

```
## このSKILLが行うこと

1. 施策名をキーにNotion DBから成果物を取得する
2. 指定された成果物種別（または全種別）をMarkdownに変換する
3. レビュー用のローカルファイルとして出力する

**Notionへの書き込みは一切行わない（読み取り専用）。**
```

Notion 上の成果物を Markdown やCSVに変換して、ローカルファイルとして出力します。フルフローには組み込まれていない読み取り専用の skill で、レビュー用に外部共有したいときや、Notion 環境外でレビューしたい場合に呼び出します。

## バケツリレー的に、情報の純度が上がっていく

ポイントは、各 agent が**前の agent が整えた情報を受け取る**こと。バケツリレー的に、最後のテストケース設計に向けて情報の純度が上がっていく構造になっています。

最初は「仕様書」というやや雑多なインプットから始まり、仕様レビュー・影響分析・テスト分析・機能調査の段階を経て、観点設計に到達する頃には「**設計に使える形に整えられた情報**」になっている。これによって、最後のテストケース出力が「ゼロから書く」のではなく「整った情報を翻訳する」作業に近づきます。

# 5. 設計のこだわりポイント — 「こう考えた」の本丸

## 📌5-1. 仕様レビュー工程を意識的に組み込んだ

前述の通り、ChatGPTでテスト成果物を作っていた頃に毎回ぶつかっていたのが、「**仕様が曖昧なまま設計に入ってしまう**」問題でした。

仕様書を読み込ませてもAIは曖昧さに気付かず、設計と並行して「あ、ここの仕様明記されてないな、曖昧だな」と人間が気付く。そのたびに補足資料を手書きで作って、AIにインプットさせる。これがすごく面倒だったんです。

そんな経験から、以下の願望が芽生えました。

> 「**むしろ、もっと早めにAIから曖昧なところを思いつく限り指摘してほしい**」

これが、qa-orchestratorのプロセスに仕様レビューを組み込んだ直接の動機です。

仕様レビューには、主に以下の検出を指示しています。

```
## 曖昧さの検出カテゴリ

| カテゴリ | 検出パターン | 例 |
|---------|------------|-----|
| **用語の曖昧さ** | 未定義の用語、複数解釈が可能な表現 | 「管理者」が指すロールが不明、「大量」の定義がない |
| **条件の未定義** | 分岐条件・前提条件が記載されていない | 「○○の場合」の逆パターンが未定義 |
| **動作の曖昧さ** | 操作結果・状態遷移が不明確 | 「エラーを表示する」（何のエラー？どこに？） |
| **エラーハンドリング** | 異常系の動作が未定義 | 通信エラー時、タイムアウト時、二重送信時の動作 |
| **境界値の未定義** | 上限・下限・文字数制限が不明 | 「文字数制限あり」（具体的な数値がない） |
| **権限・ロール** | 権限による表示/動作の差異が未定義 | 権限Aのユーザーがアクセスした場合の動作 |
| **状態の組み合わせ** | 複数条件が重なった場合の動作が未定義 | 編集中に他ユーザーが削除した場合 |
| **UI/UX不整合** | Figmaと仕様の矛盾、プラットフォーム間差異 | 仕様書の文言とFigmaのラベルが不一致 |
| **ドキュメント間齟齬** | 仕様書・企画書・Design Doc・Figma間の矛盾 | 仕様書では「削除不可」だがDesign Docでは「削除可能」と記載 |
| **データ整合性** | データの整合性・一貫性に関する未定義 | 親データ削除時の子データの扱い |
```

設計のポイントは、テスト設計と**並行**ではなく、設計の**前**に曖昧さを潰す順序にしたことです。これだけで後工程の手戻りがかなり減りました。

実際の運用においては、ここから仕様議論のタネになることも多く、導入してとても良かったと感じる工程です。

保険としてテスト観点出力後にもこのagentは呼び出されており、保留事項がうっかりそのまま進んでしまったときにも再び指摘をしてくれるため、大変助かっています。

## 📌5-2. テスト設計前に情報を厚く集める

当たり前ですが、AIの出力品質はインプット品質に直結します。

なので、テスト観点を出す前に **「影響分析」「テスト分析」「仕様レビュー（モード1）」「機能調査」** の4つで情報を厚く集めるようにしました。

![](https://static.zenn.studio/user-upload/3b006fb7a58b-20260513.png)

これにより、「観点を作る」工程に進む頃には、新機能の仕様、周辺の既存仕様、コード上の影響範囲、UI/バリデーションの細かい仕様まで揃っている状態になります。万全のインプットを整えてから本番の出力に向かうことで、なるべく手戻りをなくそうとしています。

実際、ChatGPTで出力していたころに頻発していた「**仕様を勘違いしている**」「**テスト実施手順が曖昧、間違っている**」などの不毛かつ厄介な手戻り作業はほぼほぼなくなりました。

結果として、人間の主な作業は「**Too Muchなテストケースを削る**」「**テスト方針を決める**」といった判断にフォーカスされるようになりました。

## 📌5-3. 人間/外部のテスト設計ノウハウを埋め込む

属人化を解くアプローチとして、私が選んだのは「**ノウハウを外に出してAIに参照させる**」ことでした。

* **人間の頭の中にあるノウハウ**: プロダクトドメインを踏まえたテスト観点一覧やチェックリスト
* **外部の体系的ノウハウ**: JSTQBのシラバス、汎用的なテスト観点集

これらは元々QAチーム内にあったものをそのまま流用しています。新しく作り起こしたわけではなく、**既存のチーム資産をAIに渡せる形に整える**作業がメインでした。

![](https://static.zenn.studio/user-upload/8e8e094ad028-20260513.png)

ちょっとした工夫として、JSTQBのシラバスは原文だと長すぎるので、Claude Codeに**要約させたバージョン**を使っています。「AIで使う情報をAIで整える」というのも、地味ですが効きました。

## 📌5-4. 選択肢入力ツールでの対話型進行

これはプロセスの設計思想として一番強く意識した部分かもしれません。

このツールの目的は「**誰でも一定水準以上のクオリティを出せること**」です。だとすると、ユーザーが自由にプロンプトを書ける形式にしてはいけない。プロンプト品質が人によってバラつき、最終アウトプットの品質も必ずバラつくからです。

なので、ユーザーには「**プロンプトで何とかする余地をできるだけ与えない設計**」を意識しました。各ステップで、Claude CodeのAskUserQuestionツールを使い、基本は**選択肢から選ぶだけで完結する**形にしています。「自由に書ける」よりも「**選ぶだけ**」のほうが、結果としてレールから外れにくいからです。

![](https://static.zenn.studio/user-upload/2869c6b5f1ff-20260513.png)

結果的に、レビュー時に「あの観点がごっそり抜け落ちてる…」といったことはなくなりました。

## 📌5-5. 各Stepに人間の承認ゲートを設置

AIが作ったドキュメントの**責任は、最終的に人間が負います**。

そのため、AIに任せきりでサクサク進めていくと、出来上がった成果物への**人間の責任意識がどんどん薄くなります**。これは本当に怖いです。

なので、次のStepに進むには**人間の承認が必要**、という設計にしました。レビューと承認を強制的に挟むことで、「自分が承認した成果物」という手触りを残すように意識づける目的です。

![](https://static.zenn.studio/user-upload/afca19a9f085-20260513.png)

これがないと「AIが勝手に作ったドキュメント」になってしまい、「いや、コレAIが作ったやつなんで」なんて言われてしまう可能性もあります。**責任のグリップ**を残すための仕組みです。

## 📌5-6. セッション情報の保持

承認ゲートを置いた以上、作業が中断するのは前提です。加えてrate limitに引っかかることもあります。再開のたびに最初から説明し直していたら、せっかく自動化したのに本末転倒です。

なので、`qa-session.json` というファイルを作って、ユーザーが初回に指定した

を保持するようにしました。これにより、skillを再度呼び出した時にこのファイルを参照してサクッと再開できます。

「途中で止まること」を前提に設計するのは、AIエージェントを作るうえで意外と忘れがちなポイントだと思います。

## 📌5-7. skill改善提案の自動化

skillを使っていると、「ここの出力フォーマットを修正したいな」「ここのフローをもう少し分かりやすくしたいな」といった思いが出てきます。

これも本来は人間が思いついてから一つずつ指示して修正するのも良いですが、あまりに多いと負担になってしまいますよね。

なので、AIからも提案をもらえたら嬉しいなと考え、qa-orchestratorのフローを完走した後に、今までの会話を踏まえてskillの改善案を提案してもらう設計にしました。

これにより、**人間の改善案 + AIの改善案** を総括して、更なるブラッシュアップができるようになります。

# 6. で、結局のところ精度はどうなの？

みなさん気になっているかと思いますが、肝心の出力精度の話です。

定性的にいうと、ボリュームにもよりますが、**全成果物の作成は人力で作るより圧倒的に速く、かつ抜け漏れが少ない**印象です。

とはいえ、改善余地があるのも事実。

特に重要なテスト観点とテストケースの出力精度についての現状の評価を包み隠さずにお伝えします。

## テスト観点

!

### 👍Good

* **インプット資料のクオリティもあるが、ある程度正確に出力される**
  + 主に 仕様書 / Design Doc / 影響調査レポート/ 機能調査レポート
* **仕様書外の影響画面も代表的に拾ってくれる**

!

### 👎Bad

* **粒度が細かい / 重複ケース / Too Much な観点が多い**
  + **粒度:** リアクション付与 + チップ表示確認の様にテストケースが分割されてしまい、ケース数がかさむ
    - 「1テストケース = 1検証ポイント」という指示が悪さをしている可能性大
  + **重複:** 各アプローチから観点が作成されるが、重複している内容が多い
  + **Too Much:** 実装しないために「○○できない」と仕様書に記載されたものも、テスト観点として出力されてしまう
* **権限観点の条件指定が曖昧**
* **デグレ観点が「従来通り動作すること」「正確に動作すること」といった曖昧な期待結果になりがち**
  + 指示がうまく伝わっていない可能性が高いので、改善余地あり

## テストケース

!

### 👍Good

* **機能調査レポートの操作手順が反映されるため、テスト手順がほぼ正確に出る**
  + 機能調査レポートを正しい情報に保っておけば、テスト実施手順で指摘が必要になることはほぼない

!

### Point

* **良くも悪くもテスト観点の精度に依存する**
  + テスト観点で方針、粒度、曖昧箇所の整理をしっかりしておけば、テストケースで大がかりな修正が発生することはない。

正直、まだまだ「これで完璧」とは言えません。でも、「ゼロから人間が書く」のと比べたら、設計のスタート地点が明らかに前進している、という手応えがあります。

# 7. QAプロセスはどう変わったか — 課題は解決できたのか

qa-orchestratorの作成により、当初の課題は改善できたのか、ここで評価したいと思います。

## ① QAが、開発のボトルネックになり始めていた

### 結果: ◯

直近の開発施策においては、以下の通りテスト設計工数を削減することが出来ました。

| **工程** | **手動見積** | **実績** | **削減** |
| --- | --- | --- | --- |
| テスト分析・テスト計画 | 10 h | 5 h | −5 h（50%） |
| テスト観点 | 8 h | 4 h | −4 h（50%） |
| テストケース | 6 h | 2.5 h | −3.5 h（約58%） |
| レビュー対応 | 4 h | 4 h | ±0 h（0%） |
| **合計** | **28 h** | **15.5 h** | **−12.5 h（約45%）** |

「実績工数はもう少し短縮できたのでは？」と思うところもありますが、途中仕様追加などありましたので、それを加味するとなかなか工数削減できたのかなと思います。

この施策がデビュー戦だったのでまだまだサンプルが少ないですが、今後も工数削減が期待できる結果となりました。

しかし、根本の解決に向けてはテスト実施も自動化する必要があるのは事実です。

## ②ChatGPTによるテスト設計自動化の限界

### 結果: ◎

①では工数削減効果を示した通り、テスト設計工数削減を実現できました。

また、これによる副産物として、**仕様について考える時間が増えました。**

これは、手を動かす時間が減った分、より頭を使う時間が増え、「この仕様って、そもそもこれでいいんだっけ？」「ここ、まだ仕様が曖昧だなぁ」という**UI/UXについて考える時間を増やす**ことが出来ました。

更には、**仕様やプロジェクト進行の早期明確化を意識づける**こともできました。

「曖昧なまま進めるとAIに弾かれる」というのが目に見えているので、**最初に曖昧なところは全部潰そうという**意識が強くなったり、「社内用の権限表って、更新しなきゃいけないんだっけ？」「社外向けのドキュメントって、作成しなきゃいけないんだっけ？」といった、**今までプロジェクト終盤**で出ていた話題も、早めに出るようにもなりました。

## ③ QAスキルの属人化

### 結果: ◯

ここは、正直△と判断に迷う部分がありますが、概ね良しということで◯にしました。

改善された点としては、**誰がやっても一定水準には届くようになった**ことです。  
今まであったような、「ここのテスト観点が丸ごと抜け落ちている…」というヒヤッとすることはなくなりました。

**成果物のクオリティが間違いなく底上げされた**ことは、非常に大きな改善点だと感じています。

一方で、担当者のセルフレビュースキルによるクオリティの差はやはり生じてしまいます。

結局は複数人のレビューを通すのでクオリティを確保した状態でテスト実施に進むのですが、レビュー依頼時点でのギャップを誰がどう詰めるか、は△にするか迷った点で、残課題です。

## ④ 成果物のフォーマットがバラバラ

### 結果: ◎

これは間違いなく◎です。

今までバラバラだったフォーマットが一定出力になるため、統一されることになったからです。

# 8. 直面/表面化した課題

上で述べたように、課題は解決できた一方で、`qa-orchestrator`を導入したが故に直面/表面化した課題もありました。

## ☑️リスク策定基準の曖昧さが表面化

リスク基準は、これまでも存在していました。当時は「具体化しすぎるとルール決めのコストが肥大化してしまう」ということで、あえて抽象化する方針でした。

しかし、結局のところ各QAメンバーが「雰囲気で決めていた」ものがいくつもありました。

それがAI主導にしたタイミングで通用しなくなってしまい、より明確で納得感のある基準作りが急務となりました。

このように、「AIを入れたら一気に解決」ではなく、「**人間がやっていた曖昧な部分が、可視化されて課題化する**」 — これがAI導入のリアルだなぁと感じています。

## ☑️レビュー方法が迷子になってしまった

テスト成果物はこれまでスプレッドシートを使っていましたが、今回試しにNotionに移行してみました。

NotionDBはテストケース管理やテスト実施はしやすい一方で、レビュー時には**レビュアーがページを一つずつ開く**必要があり、レビュー用ドキュメントが別途必要になってしまいました。

「では、スプレッドシートに戻したらいいのでは？」という疑問が出ますが、これもMCP制約があり、今の環境ではCSVファイルだとレビュアーから受けたコメントが`qa-orchestrator`に読み込ませられないということが発覚。

.xlsxファイルならコメントを保持して読み込ませられることが分かったので活路は見えてきていますが、ここはまだ試行錯誤中です。

## **☑️コンテキスト消費問題**

現在Claude TeamのStandard シートがデフォルトとなっていますが、この環境下では完走するまでに5h rate limit に平均5~6回引っかかってしまいます。

Premium シートの場合は、ほとんど引っかかることなく完走できます。

そのため、Premium シートのメンバーが一部出力を代行するなど、なるべく業務が止まらないように工夫しています。（しかし、他メンバーとのやり取り含め、工数増加要因となっている）

ここは、skill改善と運用面でどのように折り合いをつけるべきか、検討中の段階です。

# 9. これからやりたいこと

## **💡リスク基準の明確化**

実は、これは既に他のQAメンバーのご尽力のおかげで明確化することが出来ました。今後の開発施策で検証を行って、更に磨きをかけていきたいところです。

特にレビュー方法です。こちらも現在QAチーム内で議論中ですが、運用しやすいレビュー方法を引き続き模索していきたいと思います。

根本的なQAのボトルネック解消のためには、テスト設計だけでなく、テスト実行・不具合起票までAIで完走できる仕組みを作らなければいけないと考えています。

こちらも今後本格的に取り組みたいところです。

## **💡生成クオリティの向上**

そもそものアウトプット出力を高めることで、人間のレビューによるクオリティ格差縮められるよう、試行錯誤を繰り返していきたいと思っています。
