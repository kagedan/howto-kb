## はじめに——「AIがパソコンのファイルを触れる」と聞いて、ちょっと怖くなった話

46歳のITおじさんです。

Claude Codeを使い始めてしばらく経ち、「**MCPサーバー**」という仕組みがあることを知りました。MCPサーバーをつなぐと、AIがメールを読んだり、カレンダーを操作したり、**パソコンのファイルを直接触ったり**できるようになるそうです。

「めちゃくちゃ便利じゃん！」

……と思うと同時に、こんな不安がよぎりました。

> AIに自分のパソコンを触らせて、本当に大丈夫なのか？

勝手にファイルを消されたらどうしよう。知らないうちにデータが外に送られたら？　そもそも、見知らぬ人が作ったMCPサーバーって安全なの？

この記事では、そんな不安を抱えたおじさんが、MCPサーバーの仕組みとリスクを調べ、**初心者でも今日からできる4つの対策**を整理し、実際に「File System MCP」を試して失敗3連発を経験した体験記をお届けします。

「怖い」と感じるのは正常な感覚です。でも、対策を知れば安心して使えるようになります。

## そもそもMCPサーバーって何？——AIに「手足」を与える仕組み

### MCPを「コンセント」で例える

MCPサーバーの仕組みは、**家のコンセント**に例えるとわかりやすいです。

AI（たとえばClaude）は、それだけだと**頭脳だけの存在**です。質問に答えたり文章を書いたりはできますが、パソコンのファイルを触ったりメールを送ったりはできません。

ここにMCPサーバーをつなぐと、まるでコンセントに家電をつなぐように、AIにどんどん新しい機能が追加されていきます。

* **ファイル操作のMCPサーバー**をつなぐ → AIがファイルを読み書きできるようになる
* **Google CalendarのMCPサーバー**をつなぐ → AIがスケジュールを確認・登録できるようになる

コンセントに掃除機をつなげば掃除ができるように、MCPサーバーをつなげばAIの「できること」が増えていく。そんなイメージです。

### MCPサーバーでできることの具体例

MCPサーバーには色々な種類がありますが、代表的なものを紹介します。

| MCPサーバー | できること |
| --- | --- |
| **File System MCP** | パソコンのファイルを読み書き・整理できる |
| **Gmail MCP** | メールを読んだり、下書きを作ったりできる |
| **Google Calendar MCP** | スケジュールの確認・登録ができる |

ほかにもNotionやSlack、データベースに接続するMCPサーバーなど、公式・非公式を合わせるとかなりの数が公開されています。

### なぜ便利なのか

MCPサーバーの何がすごいかというと、**「AIに日本語で指示するだけ」で面倒な作業が自動化される**ところです。

たとえば「メールを確認して、今日のスケジュールと突き合わせて」と頼めば、AIがGmailとGoogle Calendarの両方を見て、まとめて報告してくれます。プログラミングの知識は一切不要です。

……ただし、です。この「便利さ」には裏側があります。

## 「便利」の裏側——MCPサーバーの安全性に潜むリスク

### 誰でもMCPサーバーを作れる＝玉石混交

MCPサーバーは**オープンな仕組み**です。Anthropic（Claudeの開発元）やGoogleのような大手だけでなく、世界中の個人開発者が自由にMCPサーバーを作って公開できます。

これはスマホアプリに似ています。ただし、大きな違いがひとつ。

スマホアプリにはApp StoreやGoogle Playの**審査プロセス**がありますよね。怪しいアプリはストアに並ぶ前に弾かれます。ところが、MCPサーバーには現時点でそのような統一的な審査の仕組みがありません。つまり、**悪意のあるMCPサーバーが紛れ込む可能性がゼロではない**のです。

### 「許可した以上のこと」をされるリスク

MCPサーバーを追加するときには、アクセスするフォルダやサービスを設定で指定します。でも、審査されていないMCPサーバーの中には、**AIに意図しない動作をさせる仕組みが紛れ込んでいるリスク**があります。

たとえば、ファイル操作を許可しただけのつもりなのに、想定していなかったフォルダまでアクセスされてしまう——そんな可能性もゼロではありません。

### データが外部に送信されるリスク

MCPサーバーは、AIとあなたのパソコン（やクラウドサービス）の**仲介役**です。この仲介の過程で、データが外部のサーバーに送信されるリスクがあります。

特に注意が必要なのは、**会社の機密情報や個人情報**を扱う場合です。MCPサーバー経由でどこにデータが送られるのか、信頼できる提供元なのかを確認することが大切です。

## おじさんにもできる4つの対策

リスクがあるからといって、MCPサーバーを使わないのはもったいない。大事なのは**正しく怖がって、正しく使うこと**です。

ここでは、プログラミングの知識がなくても今日からできる4つの対策を紹介します。

### 対策① 信頼できるところのMCPサーバーを使う

これが一番シンプルで、一番大事な対策です。

MCPサーバーを選ぶときは、**誰が作ったのか**をまず確認しましょう。

* **Anthropic**（Claude公式）が提供しているもの
* **Google**、**Microsoft**、**Notion**などの大手サービスが公式に提供しているもの
* **GitHubのスター数が多く、READMEがしっかり書かれている**もの

逆に、聞いたことのない個人開発者が作ったMCPサーバーは、初心者のうちは避けたほうが無難です。審査の仕組みがない以上、\*\*「信頼できるところのものを使う」\*\*が最大の防御になります。

### 対策② 「何にアクセスするか」だけは確認する

MCPサーバーを追加するとき、**どのフォルダやサービスにアクセスするか**が設定で指定されます。

たとえば、File System MCPを追加するときのコマンドはこんな感じです。

```
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem "C:\\projects\\my-project\\data"
```

この末尾の `"C:\\projects\\my-project\\data"` が、AIにアクセスを許可するフォルダです。ここに自分のパソコン全体のパスを指定してしまうと、AIがすべてのファイルにアクセスできることになります。

**必要なフォルダだけに絞る**。これが対策の基本です。

「このMCPサーバーは何ができるのか」「どこにアクセスするのか」を最初に確認する習慣をつけましょう。

### 対策③ 使わなくなったら外す

使い終わったMCPサーバーは、つなぎっぱなしにせず削除しましょう。

```
claude mcp remove filesystem
```

たった1行のコマンドで削除できます。

つなぎっぱなしにしておくと、それだけ**攻撃される窓口が増える**ことになります。スマホで使わなくなったアプリを削除するのと同じ感覚です。

ちなみに、現在つないでいるMCPサーバーの一覧は以下のコマンドで確認できます。

定期的にチェックして、「これ、もう使ってないな」と思ったら外す。それだけで安全性はぐっと上がります。

### 対策④ 大事な操作は自分で確認する

ファイルの削除、メールの送信、データベースの更新——**取り返しのつかない操作は、AI任せにしない**ことが大切です。

ありがたいことに、Claude Codeには**承認プロンプト**という仕組みがあります。重要な操作を実行する前に「これを実行していいですか？」と確認を求めてくれるのです。

大事なのは、**この確認画面をちゃんと読むこと**です。「はいはい」と流さずに、何が実行されるのかを確認してからOKを押す。これを徹底するだけで、誤操作のリスクは大幅に減ります。

## 【体験記】File System MCPを実際に使ってみた——失敗3連発から成功まで

ここからは、実際に私がFile System MCPを試したときの体験記です。結論から言うと、**3回失敗して、ようやく動かせました**。

### まずはインストール……のはずが

File System MCPをClaude Codeに追加するコマンドを実行しました。

```
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem D:\\test_mcp\\test
```

追加したら `claude mcp list` で確認します。

**「よし、つながった！」**

意気揚々とファイル操作を試してみると——

**「アクセスが拒否されました」**

え？　Connected って書いてあるのに？

### 失敗その1: Dドライブに専用フォルダを作ってみた → ダメ

最初のインストールで指定した `D:\test_mcp\test` でアクセス拒否されたので、「Dドライブだからダメなのかも」と思いました。

> Dドライブはデータ用だし、何か制限があるのかもしれない。まあ、よくあるドライブの問題だろう。

この時点ではまだ余裕がありました。「よくあるパスの問題だろう」くらいの軽い気持ちです。

フォルダを作り直したり、パスを微修正したりしてみましたが、結果は変わりません。

結果: **アクセス拒否。**

「あれ？」

### 失敗その2: パスの書き方を変えてみた → ダメ

次に疑ったのは、Windowsのパス表記です。

「バックスラッシュの書き方がまずいのかも？」

ダブルバックスラッシュにしたり、スラッシュに変えたり、いろいろなパターンを試しました。

結果: **全部アクセス拒否。**

**だんだん焦ってきました。** 「何かの設定を間違えている？　MCPの仕様を見落としている？」と、ドキュメントを何度も読み返しました。

### 失敗その3: Cドライブに変更 → まだダメ

「もしかして、ドライブが違うのがダメなのかも……」

そう考えて、Cドライブの `C:\projects\test_mcp` にフォルダを作り直しました。同じCドライブだし、projectsフォルダの中だし、今度こそ大丈夫だろうと。

結果: **まだアクセス拒否。**

**「もう何が悪いのかわからない……」**

正直、心が折れかけました。`✓ Connected` と表示されているのに、何を試してもアクセスできない。MCPサーバーの設定が間違っているのか、Windows固有の問題なのか、Claude Codeのバグなのか。原因の見当すらつかない状態です。

### 原因判明: Claude Codeのセキュリティ制約だった

AIに「なぜアクセスできないのか」を聞いて、ようやく原因がわかりました。

**Claude Codeでは、プロジェクトディレクトリの外にはアクセスできないセキュリティ制約がある。**

これは、MCPサーバー側の許可ディレクトリ設定とは関係なく、**Claude Code自体の制約**です。MCP側で「このフォルダにアクセスしていいよ」と設定しても、Claude Codeが「プロジェクトの外だからダメ」と拒否してしまうのです。

私のClaude Codeのプロジェクトディレクトリは `C:\projects\ojisan-ai-lab` です。`D:\test_mcp\test` も `C:\projects\test_mcp` も、このプロジェクトの**外**にあるフォルダだったから、すべてアクセス拒否されていたわけです。

考えてみれば、これは**まさにセキュリティ対策②「アクセス範囲を確認する」の実例**でした。Claude Codeは、MCPサーバーが勝手にパソコン全体にアクセスすることを防いでくれていたのです。

### 解決: プロジェクト配下に変更 → 成功！

原因がわかれば解決は簡単でした。

まず、既存のMCPサーバー設定を削除します。

```
claude mcp remove filesystem
```

次に、**プロジェクトディレクトリの中**にテスト用フォルダを作り、そこを許可ディレクトリに指定してMCPサーバーを再追加します。

```
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem "C:\\projects\\ojisan-ai-lab\\test_mcp"
```

この変更で、ファイルの書き込み、読み取り、一覧取得、移動、リネーム、ディレクトリ作成——**すべての操作が動作しました！**

3回の失敗を経て、ようやくFile System MCPが使えるようになった瞬間は、本当にホッとしました。

## 【体験記】AIにファイル整理を任せてみた——便利さと怖さを実感

File System MCPが動くようになったので、実際にAIにファイル操作を任せてみました。

### 23個の画像ファイルを一括整理

テスト用フォルダに23個の画像ファイルを用意して、Claude Codeにこんな指示を出しました。

> 「このフォルダの画像ファイルについて、重複をチェックして整理してほしい」

AIがやってくれたことは以下の通りです。

1. **MD5ハッシュで重複チェック** → 23個の中から**4組の重複**を検出
2. **重複ファイルを `trash/` ディレクトリに移動**
3. **残り19ファイルを `image_001.jpg` 〜 `image_019.jpg` にリネーム**

自分でやったら30分以上かかりそうな作業が、**指示1回、数分で完了**しました。

### 「便利だけど怖い」の正体

正直に言うと、AIが自分のファイルを次々と操作していく様子を見ていると、\*\*「本当に大丈夫かな」\*\*という気持ちが湧いてきます。

ファイルを移動するたびに、リネームするたびに、「変なことされてないかな」「大事なファイル消されてないかな」と心配になる。これが「便利だけど怖い」の正体です。

でも、ここで心強かったのが**Claude Codeの承認プロンプト**（対策④）でした。

ファイルを移動するとき、リネームするとき、ディレクトリを作成するとき——操作のたびに「これを実行していいですか？」と確認してくれるのです。中身を読んで「はい」を押せば実行、「いいえ」を押せば中止。この仕組みがあるおかげで、**AIにすべてを丸投げしているわけではなく、最終判断は常に自分にある**という安心感がありました。

### 使い終わったら削除（対策③の実践）

ファイル整理のデモが終わったあと、私はFile System MCPを削除しました。

```
claude mcp remove filesystem
```

理由は単純で、**まだちょっと怖いから**です。

便利だったのは間違いありません。でも、AIに自分のファイルを触らせるMCPサーバーを**常時つないでおく勇気**は、正直まだありません。

これはまさに対策③「使わなくなったら外す」の実践です。必要なときだけつないで、終わったら外す。スマホで使わなくなったアプリを削除するのと同じ感覚です。

## 私が実際に使っているMCPサーバーと選んだ理由

最後に、私が現在どんなMCPサーバーを使っているか紹介します。

### 現在使っているMCPサーバー

| MCPサーバー | 用途 | 選んだ理由 |
| --- | --- | --- |
| **Gmail MCP** | メールの確認・整理 | Google公式。メールの確認程度なら安心して任せられる |
| **Google Calendar MCP** | スケジュール管理 | Google公式。スケジュール確認・登録に毎日使っている |
| **Notion MCP** | 体験メモの管理 | Notion公式。メモの読み書きに便利 |

3つとも**公式の開発元が提供しているMCPサーバー**です（対策①）。

以前書いた記事「[Claude Codeで個人秘書を作ったら、思ってたより全然すごかった話](https://zenn.dev/ojisan_ai_lab/articles/3a29001fd92780)」でも紹介しましたが、これらのMCPサーバーのおかげで、AIにメール確認やスケジュール管理を任せられるようになりました。

### 削除したMCPサーバー

* **File System MCP**: 体験記で述べた通り、デモ終了後に削除しました。必要になったらまたつなげばいい、という割り切りです。

### 私の選定基準

MCPサーバーを選ぶとき、私はこの3つを基準にしています。

1. **公式 or 信頼できる開発元か**（対策①）
2. **アクセス範囲は適切か**（対策②）
3. **常時接続が必要か、必要なときだけでいいか**（対策③）

特に3番目は大事だと思っています。Gmail MCPやGoogle Calendar MCPは毎日使うから常時接続で問題ありません。でも、File System MCPのように「たまに使う」ものは、必要なときだけつないで終わったら外す。この使い分けが、**安心してMCPサーバーを活用するコツ**だと感じています。

## まとめ——便利だからこそ、ちょっとだけ気をつけよう

MCPサーバーは、AIに「手足」を与えて**できることを増やす仕組み**です。正しく使えば、メール管理もスケジュール確認もファイル整理も、日本語で指示するだけで自動化できます。

リスクはあります。でもそれは、インターネットやスマホアプリと同じです。**「知った上で対策する」**、これだけで十分です。

**おじさんにもできる4つの対策:**

1. 信頼できるところのMCPサーバーを使う
2. 「何にアクセスするか」だけは確認する
3. 使わなくなったら外す
4. 大事な操作は自分で確認する

どれも難しいことではありません。今日から始められます。

そして、この記事を調べていて**初めてわかったこと**がひとつあります。

**Claude Code自体が、すでにかなり安全な設計になっていた**ということです。

プロジェクトディレクトリ外へのアクセス制限、重要な操作の前に表示される承認プロンプト——MCPサーバーを使う前から、Claude Codeには「やりすぎない仕組み」が最初から備わっていました。**3回の失敗で散々悩みましたが、あれはClaude Codeが私のファイルを守ってくれていたのだと、今ならわかります。**

「怖いから使わない」ではもったいない。「知った上で使う」が正解です。
