「LLM使ってますか?」と聞くと、だいたいこう返ってきます。  
「使ってる使ってる。調べものと、メールの文面と、たまに設計書をアップロードしてレビュー」

それ、**チャット画面**ですよね。  
LLMの賢さの、たぶん3割しか使えていません。  
残りの7割は「エージェント」——ファイルを読み書きし、コマンドを実行し、結果を見て次の行動を決めるLLM——の側にあります。

で、「エージェント」という専用ソフトがあるかというと、実は純粋にエージェントと呼べるものはリリースされていません。  
その代わり、**コーディング用のエージェントとしてすでにリリースされています**。  
ClaudeCodeのような、IDEやターミナルで動くやつです。  
「IDE=コードを書く人の道具」という思い込みで、コードを書かない人が素通りしている。

今日はそれを、いつものSI業務で使い回します。  
先に言っておくと、**この記事で私は一行もコードを書きません**。書くのは向こうです。

## フォルダごと、読ませる

サーバ更改案件のフォルダで起動して、ひとこと。

> この案件フォルダを読んで、要件定義と基本設計の不一致を重要度順に。

```
⏺ Read 要件定義書v1.3.md
⏺ Read 基本設計書_draft.md
⏺ Read パラメータ一覧.csv
```

> **LLM:** 重要度順に3件です。
>
> 1. **【矛盾】NTPの向き先が新旧混在** — 基本設計「3.2 時刻同期」は新NTP(172.16.1.10)、パラメータ一覧は旧NTP(10.1.1.5)のまま
> 2. **【要件との不一致】バックアップ保持世代** — 要件定義v1.3は「日次7世代」、基本設計「5.1」は5世代
> 3. **【記載漏れ】切り戻しの判断基準** — 切り戻し手順はあるが、「何が起きたら戻すか」の基準と判断者の記載なし

チャットとの違いは地味で、決定的です。  
**ファイルを1個ずつアップロードしていない**。  
フォルダを勝手に開けて、必要なものを自分で読んでいます。  
10ファイルでも同じ一言で済みます。

## Excelで、出させる

レビューを反映したら、次は試験項目。今度は成果物の形式まで指定します。

> この設計書から結合試験の項目表を作って、`試験項目_結合.xlsx` で保存して。1シート目に観点サマリ、2シート目に項目表。

```
⏺ Write make_test_sheet.py   # openpyxlで項目表を生成するスクリプト
⏺ Bash python make_test_sheet.py
  → 試験項目_結合.xlsx を保存しました(48項目・2シート)
```

フォルダに、**本物のExcelファイルができています**。開けば2シート、罫線も入っています。

種明かしをすると、LLM自身はExcelを出力できません。  
エージェントは\*\*「Excelを作るコード」を書いて、実行した\*\*んです。  
私はそのコードを一度も見ていません。見る必要がないので。  
WordもPowerPointも同じ要領で通ります(「この調査結果を日次報告のdocxで」で終わり)。  
正本はテキストで持って、納品の体裁は出力で作る——IaCの記事で書いた話と同じ思想です。

コーディング能力って、コードという成果物のためだけのものじゃないんですよ。  
**あらゆる成果物を作るための手足**なんです。

## SSHで、見てこさせる

机上だけではありません。手足はネットワークの向こうまで届きます。

> 検証サーバ(踏み台経由で 172.16.1.20)にSSHして、ディスク使用率と直近1時間のエラーログを見てきて。

```
⏺ Bash ssh -J bastion appuser@172.16.1.20 "df -h /var"
  → /var 92%(残 1.1GB)
⏺ Bash ssh -J bastion appuser@172.16.1.20 "journalctl -p err --since '-1h' | tail -5"
  → app[1123]: log rotation failed: /var/log/app/debug.log (4.2GB)
```

> **LLM:** /var が92%です。原因はアプリのデバッグログ(debug.log 4.2GB)のローテーション失敗。移行時のデバッグ設定が残っている可能性があります。  
> 対処案: ①応急 — 旧debug.logを圧縮して空きを確保 ②恒久 — アプリのdebugフラグをオフ。  
> ①を実行しますか?

私はSSHクライアントを開いてすらいません。「見てきて」が、そのまま通ります。

そして最後の一行に注目してください。**勝手には直さず、聞いてきます。**  
Claude Codeのようなエージェントは、変更を伴うコマンドの前に人間の承認を挟む設計になっています。  
読み取りは任せる、変更は承認する——ここの線引きは、GitでもIaCでも書いた「差分を見る目は人間に残す」と同じ話です。

## 種明かし — エージェントの中身は3つだけ

なぜIDEのLLMは何でもできて、チャットのLLMは調べもので止まるのか。  
エージェントの構成要素は、実は3つしかありません。

1. **ファイルを読み書きできる**
2. **コマンドを実行できる**(コードを書いて、動かせる)
3. **結果を見て、次の行動を自分で決める**(うまくいくまで繰り返す)

チャット画面には1と2がありません。だから賢くても「調べる・下書きする」で止まる。  
そしてこの3つを最初から全部持っていた道具が、たまたま**コーディングの道具**でした。  
フォルダ横断もExcel生成もSSHも、裏側は全部「ファイルとコマンド」です。

「エージェント専用ソフトが出たら本気出す」は、順序が逆です。もう出荷されていて、コーディング用の顔をしているだけです。

## よくある疑問

**Q. 会社の情報を入れていいの?**  
所属の利用規程が最優先。その上で、入力を学習に使わない法人契約・プランと、顧客名など固有情報のマスキングが基本線です。  
\*\*個人の無料アカウントに顧客情報を貼るのはNG。\*\*SSHに至っては、まず検証環境からにしましょう。

**Q. サーバ操作までさせて、事故りません?**  
読み取り系(見る・調べる)は任せて大丈夫です。  
変更系は承認が挟まるので、そこで差分を読む。本番への変更は、慣れるまで任せない。  
この順番を守る限り、手で深夜作業するより安全側だと私は思っています。

**Q. どのツールを選べば?**  
この記事の内容は、ClaudeCodeでもCursorでもVSCodeでもAntigravityでも、だいたい成立します。  
ツール名より「**手足(ファイル・コマンド)のあるLLMか**」で選んでください。  
チャット欄しかないものは、何を選んでも今回の話はできません。

## 締め

理屈を一段落だけ。チャットのLLMとエージェントのLLM、**中身の賢さは同じ**です。  
違いは手足——ファイルの読み書き、コマンド実行、結果を見て続けるループ——があるかどうかだけ。  
そしてSIの机上の仕事は、手足さえあれば大半が回ります。

「IDEはコードを書く人のもの」。この思い込みが、いまいちばんもったいない。
