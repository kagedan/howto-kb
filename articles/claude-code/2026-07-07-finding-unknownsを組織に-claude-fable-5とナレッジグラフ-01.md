# Finding Unknownsを組織に — Claude Fable 5とナレッジグラフ

> この記事は独立して読めます（約10分）

Anthropic の Thariq Shihipar 氏が公開した「[A field guide to Claude Fable 5: Finding your unknowns](https://claude.com/blog/a-field-guide-to-claude-fable-finding-your-unknowns)」（意訳すると **未知の洗い出し**）が、SNS や開発者コミュニティで話題になっています。

原典は主に **コーディング** 向けの記事です。技法の細部はそちらを読むのが早いので、本記事では次の 2 点に絞ります。

1. **なぜ今、話題になっているのか**
2. **会社の業務に当てはめると、原典だけでは足りない視点は何か**

---

## なぜ今、話題になっているのか

要するに、「プロンプトを長く書く」より **自分が何を知らないかを先に洗い出す** 方が、Claude Fable 5 時代の成果を左右しやすい、という話です。

Shihipar 氏の整理では、これまでのように「モデルが弱いから失敗する」というより、**書いていない前提を AI が勝手に補完してしまう** ことがボトルネックになりやすい、と述べられています。情報が足りないときも、書きすぎて柔軟性を失うときも、根っこは同じで、「自分の未知の扱い」が鍵になります。

話題になっている理由として、筆者は次の 3 点を挙げます。

1. **「地図と領土」の比喩が分かりやすい** — プロンプトは地図、実際のコードや制約が領土。ズレる理由を説明しやすい。
2. **単発のプロンプト術ではない** — 作業の前・中・後で回す手順として書かれている。
3. **実務者の体験談として出ている** — ベンダーの一般論ではなく、Claude Code チームの現場感がある。

---

## 原典に足りない視点 — 「一人で完結」と「組織で動く」の差

原典は、**一人でコードを書く人** を想定しています。同じ人が指示を出し、リポジトリを読み、不明点を減らしていく。開発現場ではそれで十分な場面が多いです。

一方、会社の業務は **一人では完結しません**。答えは誰か一人の頭の中にはなく、営業・サポート担当・法務がそれぞれ持つ情報を足し合わせ、会議や承認を経て決まっていきます。

| 原典（一人で完結） | 組織（複数人で動く） |
| --- | --- |
| 1 人が前提を更新する | 担当者が変わるたびに前提が引き継がれる |
| リポジトリが「現場の実態」 | CRM・チケット・人事データがバラバラに残る |
| 答えはコードに落ちる | 答えは会議・合意・例外承認を経る |
| 不明点は個人のスキル | 不明点は部門のサイロや権限の壁に隠れる |

商談からオンボーディングへ渡すとき、営業のメモ・サポート担当の引き継ぎチェックリスト・法務の特約は、それぞれ **地図の断片** です。誰も全体を一人で見通せるとは限りません。

### 外から見た会社と、中から見た会社は違う

AI の学習データや業界の「こうあるべき」は、基本的に **外から見た会社** の言葉でできています。そのため、きれいな回答が返ってきても、社内の人から見ると **そのままでは使えない** ことがあります。

* 「こうすべき」は、**誰が決める権限を持つか** で変わる
* 「標準フロー」は、現場では **例外対応とセット** で回っている
* 「顧客満足」も、部門ごとに **見方が違う**

社風や暗黙のルール、会議で初めて出る前提などは、プロンプトに全部書くのも難しいし、システムから機械的に読み取るのも難しい領域です。

### 会社で使うときのゴールの置き方

原典は「個人が未知を言語化するスキル」として書かれています。会社の業務に当てはめるなら、ゴールは **「AI に正解を出させる」ことではなく、「会議のたたき台を揃える」こと** に近いです。

1. 盲点になりそうな論点を候補として出す
2. 推測と事実を分ける
3. 関係者に渡して、社内で合意する

この読み替えが、本記事の出発点です。

## たとえ話：商談が決まった直後（架空の例）

次は、場面を想像しやすいように据えた **架空の例** です。実在の案件ではありません。

営業が大型案件を受注したとします。顧客には「来月からオンボーディング開始」と伝え、社内ではサポート担当に「いつものフローで」と連絡した。よくある流れです。

ここで AI に「オンボーディング計画を作って」とだけ頼むと、見栄えのよい工程表が返ってくる想定です。しかし社内では、たとえば次のような論点が、営業側の前提に入っていないことがあります。

1. サポートに **既知の不具合** があり、その顧客の利用機能と重なる
2. 法務が **データ処理の特約** をまだ確認していない
3. サポート担当者が来月異動し、**引き継ぎ先が決まっていない**

AI の案は外から見ると正しく、社内から見ると **そのままでは実行できない** ことがあります。

Finding Unknowns（未知の洗い出し）の使い方は、完成した計画を作ることではなく、**上の 1〜3 を「要確認リスト」として先に表に出す** ことです。最終的な正解は、そのあとサポート担当・法務の会議で作ります。

---

## 明日から試せる手順（営業 → オンボーディングの想定例）

前のたとえ話を踏まえた **想定例** です。プロンプトはそのままコピーして使えますが、出力内容は環境ごとに変わります。

### ステップ 1：盲点の洗い出し（Blind Spot Pass）

原典で推されている **Blind Spot Pass**（盲点の洗い出し）は、プロンプトに次の英語をそのまま入れると伝わりやすい技法です。

* `blind spot pass`
* `unknown unknowns`（気づいていない未知）

自分の立場も正直に書きます。以下は **記載例** です。

```
この商談からオンボーディングまでのプロセスについて、私は営業視点しか持っていません。
blind spot pass で、私の unknown unknowns を洗い出してください。
参照したシステム名・ドキュメント名を列挙し、推測は推測とラベル付けしてください。
```

CRM やチケット、社内 Wiki を AI から読めるなら、それが「現場の実態」の代わりになります。なければ、手元の資料を添付するだけでも一歩進みます。

### ステップ 2：一問ずつ深掘り（Structured Interview）

盲点の候補が出たら、**Structured Interview**（構造化インタビュー）を依頼します。たとえば前のたとえ話なら、「顧客の過去 SLA を未確認」「サポート担当への引き継ぎが未設計」といった候補が挙がったとします。そのうえで AI に一問ずつ質問させ、「答えがプロセスや顧客への約束を変えるもの」だけを優先させます。

```
先ほどの盲点候補について、structured interview してください。
一問ずつ、答えがプロセスや顧客への約束を変える質問だけを優先してください。
```

上記の盲点候補を踏まえたうえで、返ってきそうな質問の例です（実際の出力をそのまま引用したものではありません）。

* 「SLA 違反の履歴を確認していませんでした。オンボーディング SLA を現状のまま約束してよいですか？」
* 「サポート担当者の異動予定を把握していませんでした。引き継ぎ計画は誰が埋めますか？」

### ステップ 3：会議に渡す 1 ページにまとめる

AI の出力をそのまま「答え」にしないでください。次の 4 項目に整理してから、サポート担当・法務に渡すと会議が始めやすくなります。

| 項目 | 書くこと |
| --- | --- |
| 自分の立場 | 営業のみ、オンボーディング未経験、など |
| 要確認リスト | 推測と事実を分けて列挙 |
| 聞きたい相手 | サポート担当、法務 など（複数でもよい） |
| 決めたいこと | 開始日を約束してよいか、例外承認が要るか、など |

「なぜ通常フローから外れるのか」も短くメモしておくと、次の引き継ぎに役立ちます。

## 社内ナレッジグラフがある場合に追記するプロンプト例

顧客・案件・障害情報が **社内ナレッジグラフ（KG）** でつながっている環境なら、ステップ 1 のプロンプトに次を足します（こちらも **記載例** です）。

```
社内ナレッジグラフ上の関連データも横断してください。
根拠となる記録 ID と関係を列挙し、権限のない領域は「参照不可」と明示してください。
```

たとえば、営業が知らなかった「過去の障害対応」や「法務の特約」が、会議に持ち込める候補として出やすくなる、といった使い方が考えられます。KG は正解を決める装置ではなく、**会議のたたき台の質を上げる補助** と考えるとよいです。

探索のルールづけは [Graph Traversal Contract](https://zenn.dev/knowledge_graph/articles/graph-traversal-contract-skill)、社内データがバラバラに残る構造の話は [AIエージェントが毎回データを取りに行く設計の限界](https://zenn.dev/knowledge_graph/articles/kg-agent-memory-first-design) を参照してください。

## まとめ

Finding Unknowns が話題になっているのは、「モデルの性能」より「自分が何を知らないか」の方が成果を左右する、という **見方が広がっている** からだと筆者は考えます。

会社の業務では、答えは一人が持たず、外から見たきれいな正論が、社内から見ると的外れになることもあります。だから AI の出力は **答え** ではなく **議題** として扱うのが現実的です。

明日から試すなら、次の 3 ステップで十分です。

1. 業務判断の前に **盲点の洗い出し** を 1 回試す
2. 推測と事実を分け、**1 ページのたたき台** にする
3. そのたたき台を **会議の議題** として関係者に渡す

社内データがグラフでつながっているなら、同じ流れに「根拠付きの要確認リスト」を足すだけで、議論の出発点が一段具体になります。
