## TL;DR

* MinToはAIとの対話でWebサイトを作ることをコンセプトにしたホスティングサービス
* UIを作るとデプロイ操作がAIの会話フローから分断される
* ブラウザ操作はスクリーンショットによるトークン消費が大きく、日常利用には現実的でない
* StreamableHTTP + OAuth 2.1対応のMCPサーバーを実装し、テキストベースで完結する設計にした

## 背景

[MinTo](https://lp.mintoai.net) は静的サイトのホスティングサービスです。コンセプトは「AIとの対話でWebサイトを作る」こと。Claude Codeなどで生成したサイトをそのままデプロイできる環境を目指して作りました。

設計を始めたとき、最初に悩んだのが「UIを作るかどうか」でした。一般的なホスティングサービスはダッシュボードUIを持っています。でもMinToはあえてそこを作らず、CLIとMCPだけで完結する設計にしました。

## なぜUIを作らなかったか

### デプロイがAIの会話フローから切り離される

MinToのユースケースはこうです。ユーザーがAIに「こういうサイトを作って」と伝える。AIがHTMLやCSSを生成する。そのままデプロイして公開する。

UIがあるとこの流れに断絶が生まれます。AIがファイルを生成したあと、ユーザーはブラウザを開いてダッシュボードに移動し、ファイルをアップロードしてボタンを押す。この「UIに移動する」という操作は一見小さいですが、AIとの対話セッションの外に出ることを意味します。

CLIならAIがそのままコマンドを実行できます。

```
minto-cli deploy ./dist --site-id <site-id>
```

会話の中でデプロイが完結するので、体験が途切れません。

### ブラウザ操作はコンテキストを大量に消費する

「AIエージェントがブラウザを操作すればUIでもいいのでは」という考え方もあります。Computer UseやBrowser Useのような技術は確かに存在します。

ただ現実的な問題があります。ブラウザ操作をするAIエージェントは、画面のスクリーンショットを逐次読み込みながら動作します。1枚のスクリーンショットで数百〜数千トークンを消費するため、「ログイン→ダッシュボード移動→ファイルアップロード→デプロイボタンを押す」という一連の操作だけで容易に数千トークンが吹き飛びます。

CLIコマンド一行のトークン消費と比べると桁が違います。日常的に使うユーザーにとってこれはコストとレートリミットの両方に直結する話で、設計として採用するには無視できない差です。

加えて、ブラウザ操作の精度自体もまだ安定していません。ボタンの認識ミスやフォーム入力の失敗が起きやすく、リトライのたびにさらにトークンを消費する悪循環になります。テキストベースで完結できるなら、そちらを選ぶべきというのが結論でした。

## MCPの設計

### トランスポートと認証

MinToのMCPサーバーはStreamableHTTPで実装しています。

stdioベースのローカルMCPサーバーという選択肢もありますが、それはMinToの想定ユーザーには現実的でないと判断しました。ローカルMCPの場合、利用者はCLIツールをインストールしてサーバーを起動する必要があります。MinToのターゲットは非エンジニアも含むため、その手順を踏ませることは体験として成立しません。リモートのMCPサーバーとして提供することで、URLを設定に追加するだけで使い始められます。

認証はOAuth 2.1に対応しています。MCPクライアントがサーバーにアクセスする際、OAuthフローでアクセストークンを取得し、以降のリクエストに付与します。これによりユーザーごとの認可を安全に管理できます。

### 実装したツール

MCPサーバーには以下のツールを実装しています。

**アカウント・テナント系**

* ユーザー情報の取得
* テナント情報の取得・作成・更新

**サイト操作系**

**課金系**

AIが「サイトを作ってデプロイして」と指示されたとき、サイト作成→デプロイという2つのツールを順に呼び出すだけで完結します。課金系のツールを持つことで、AIが「現在のプランで追加デプロイできるか」を確認してから操作するようなフローも実現できます。

### 実際の動作イメージ

```
ユーザー:「このdistフォルダをmy-blogにデプロイして」
AI: [get_user_info でユーザー確認]
AI: [get_sites でサイト一覧を取得]
AI: [deploy_site でデプロイ実行]
AI:「デプロイしました。https://my-blog.mintoai.net で公開されています」
```

CLIは自分でコマンドを叩く開発者向け、MCPはAIに操作を委譲したいユーザー向けという棲み分けですが、どちらも「テキストベースで完結する」という設計の軸は変わりません。

## AIと使う前提で設計するということ

既存のホスティングサービスはUIを中心に設計されていて、CLIやAPIはそのオプションとして提供されています。MinToはこれを逆転させて、CLIとMCPを主軸にしました。

AIが日常的に使われる環境では、「UIに移動する」というコストが思いのほか体験を損なう。この判断が正しかったかどうかは使ってもらいながら検証していきますが、「AIと使う前提で作る」という設計の軸はブレさせたくないと思っています。
