## はじめに

![](https://static.zenn.studio/user-upload/52da3ab3a81e-20260721.png)  
Unity公式のCLIであるUnity CLIがリリースされました。

<https://unity.com/ja/blog/meet-the-unity-cli>

今回はUnity CLIのインストール方法、AIエージェントの連携について解説し、最終的にはゲーム制作まで行います。

## TL;DR

* Unity公式のスタンドアロンCLI「**Unity CLI**」（ベータ版）が公開されています。エディタのインストールからプロジェクト操作までできます
* 真価は `com.unity.pipeline` パッケージとの連携です。**起動中のUnityエディタをHTTP経由でフル遠隔操作**できます（recompile、テスト実行、C#のeval、GameObject操作、スクリーンショットまで約140コマンド）
* これをClaude Codeに渡したところ、**エディタに一度も触らずに**ブロック崩しの実装→シーン構築→Play Modeでの動作検証→デバッグまで完走しました

## Unity CLI とは

Unity CLI は Unity Hub とは独立して配布されるコマンドラインツールです（現在ベータ）。

下記のコマンドでインストールが可能です。

```
# インストール (macOS / Linux)
curl -fsSL https://public-cdn.cloud.unity3d.com/hub/prod/cli/install.sh | UNITY_CLI_CHANNEL=beta bash
```

```
# インストール(Windows)
$env:UNITY_CLI_CHANNEL='beta'; irm https://public-cdn.cloud.unity3d.com/hub/prod/cli/install.ps1 | iex
```

インストール後はパスが通っていないことがあるので、`source ~/.zshrc`かターミナルを再起動します。

下記のようなコマンドが使えます

```
unity install lts                 # Unity Editor(LTS版)のインストール
unity install 6000.2.10f1         # バージョンを指定したUnity Editorのインストール
unity open ./MyProject            # プロジェクトを開く
unity test ./MyProject            # テストをバッチ実行
unity editors -i --format json    # JSON出力（CI向け）
```

従来の Hub CLI（`"Unity Hub" -- --headless ...`）の後継で、JSON/TSV出力と予測可能な終了コードを備えた「CI・自動化ファースト」な作りになっています。

## Unity CLIからUnity Editorを操作する

Unity CLIからUnity Editorを操作するためには`com.unity.pipeline` パッケージが必要になります。

Unityプロジェクトを作成し、プロジェクト直下で`unity pipeline install`を実行すると`com.unity.pipeline` パッケージがインストールされ、エディタ内にローカルHTTPサーバーが立ち上がります。

![](https://static.zenn.studio/user-upload/920399f00d1a-20260721.png)

* バインドは 127.0.0.1 のみ（外部到達不可）で、Bearerトークン認証つきです
* エディタの発見は `Library/Pipeline/.unity-pipeline-port` に書かれるポート記述ファイル経由で行われます
* CLI から `unity command <name>` で実行します

```
unity command                          # 使えるコマンド一覧（約140個）
unity command editor_status            # エディタの状態
unity command recompile                # 再コンパイル（非同期、statusをポーリング）
unity command run_tests --mode editor  # テスト実行
unity command eval --code 'return 2+2;'  # Roslynで任意のC#を実行(!)
unity command screenshot --view game   # Gameビューのスクショ
```

コマンドはエディタ制御・シーン/GameObject/プレハブ操作・アセット管理・UPMパッケージ操作・ビルド・ベイク・アニメーション編集までカバーしています。破壊的操作は `--confirm true` が必須で、`--dry_run` でプレビューできるという安全設計です。

地味に重要なのが `set_autotick` です。**エディタが非フォーカスでも tick し続ける**ようになり、ターミナル作業中もコンパイルやテストが裏で進みます。

## ハマりどころ: エディタが見つからない

サーバーは生きているのに CLI が「No Unity Editor instances found」と言い出すことがありました。調べた結果は以下のとおりです。

* CLI はポート記述ファイル `Library/Pipeline/.unity-pipeline-port` **だけ**を頼りにエディタを発見します（ブロードキャスト等はありません）
* このファイルが何らかのタイミング（ドメインリロード時の削除→再作成の競合?）で消えると、サーバーがポート7800でLISTENしていても接続手段がなくなります
* **復旧はエディタメニューの Pipeline ▸ Stop Server → Start Server** です（記述ファイルが書き直されます）

![](https://static.zenn.studio/user-upload/72ebb9d1ad18-20260721.png)

切り分けに使ったコマンドはこちらです。

```
unity pipeline list                      # パッケージ状態とServer Reachable
lsof -nP -iTCP:7800-7849 -sTCP:LISTEN    # LISTEN中ポートの確認
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:7800/api/status  # 401なら生存
```

## 本題: AIエージェントにブロック崩しを作らせる

このコマンド群は要するに「**エディタ操作の完全なAPI**」なので、AIエージェントに渡すと何が起きるか試してみました。パッケージには Claude Code 用のスキル定義まで同梱されています（`.claude/skills/unity-pipeline`）。

Claude Code に「このコマンドを使ってブロック崩しを作って。完成するまでトライを続けて」とだけ指示した結果、人間がエディタに触ることなく以下が全自動で完了しました。

### 1. シーン構築とコーディング

```
unity command create_scene --path Assets/Scenes/Breakout.unity
# → C#スクリプト4本をディスクに書く（GameManager / パドル / ボール / 入力抽象化）
unity command recompile && unity command recompile_status   # エラーゼロを確認
unity command create_gameobject --name GameManager
unity command attach_script --target GameManager --type Breakout.BreakoutGame
unity command save_scene
```

設計上の工夫として、アリーナ（壁・パドル・ボール・ブロック40個）は GameManager が起動時にコードで生成する方式にして、CLIでのシーン組み立てを最小化していました。

### 2. Play Mode での自動検証

ここが一番おもしろいところです。エージェントは Play Mode に入り、**キーボードの代わりに `eval` でゲームを操作して**動作検証しました。

```
unity command editor_play
# evalでボールを発射 → 8秒後に状態を観測
unity command eval --code '...FindFirstObjectByType<BallController>().Launch();...'
# → 「ブロック2個破壊・スコア200・ボール落下でライフ減」を確認
```

勝利条件のテストでは、evalでパドルを画面幅いっぱいに広げてボールが落ちない状態を作り、全40ブロック破壊 → Won 状態への遷移を確認していました。リスタートとゲームオーバーの遷移もリフレクション経由で検証しており、テストハーネスを自分で発明している感があります。

```
unity command capture_game_view --save_path Temp/breakout.png   # 見た目の確認もスクショで
```

### 3. 完成

このような流れを経てAIが10分ほどでプレイできるブロック崩しを一発で作ってくれました。  
[半年前に検証したとき](https://zenn.dev/tkada/articles/00b32b7f41a2ed)は、動くけど動作が微妙で何度か修正を加えながら進めていく感じだったので、今回の結果はAIの進化を感じるものがあります。

## 感想と考察

* **「エディタ操作のAPI化」のインパクトが大きいです。** これまでUnityの自動化はバッチモード（エディタ起動のオーバーヘッドが毎回かかる）か、自前のエディタ拡張が必要でした。起動しっぱなしのエディタに外からコマンドを流し込めるので、開発ループが桁違いに速くなります
* **AIエージェントとの相性がとても良いです。** 構造化出力（JSON）・予測可能な終了コード・dry\_run・confirm ゲート・非同期コマンドのステータスポーリングと、エージェントが安全に使うための道具が最初から揃っています。`eval` があるので「コマンドにない操作」もエージェントが自力でなんとかできます
* **スクショで自己検証できる**（`screenshot` / `capture_game_view`）ので、エージェントが「作って終わり」ではなく見た目まで確認するループが閉じます
* 一方で**ベータ相応の不安定さ**はあります（ポート記述ファイル消失問題）。
* OnGUI のHUDはキャプチャに写らない、`run_tests` はテスト失敗時に詳細が取れないことがある、などの細かい制約もあります

## まとめ

Unity CLI + Pipeline パッケージは「ターミナルからUnityエディタを操作する」を公式が本気でやりに来た機能で、CI・自動化だけでなく **AIエージェント時代のUnity開発ワークフロー**の土台になりそうです。実験的パッケージなので今後の変更には注意しつつ、遊んでみる価値は十分あります。
