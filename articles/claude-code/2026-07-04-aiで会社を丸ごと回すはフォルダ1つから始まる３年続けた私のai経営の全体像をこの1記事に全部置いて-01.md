AIの情報は毎日流れてきます。  
  
新しいツールを試してみる。その場では確かに便利。でも気づくと、契約したツールとアカウントばかりが増えて、肝心の事業そのものは何も変わっていない——そんな感覚を持っている方は、多いのではないでしょうか。

私も最初の頃はそうでした。

いま、私は1人で会社を経営しています。集客の記事づくり、営業の提案書、制作、顧客対応の返信、請求書の準備、経理の記帳。この全部に、毎日AIが関わっています。人は雇っていません。

それでも会社は回っています。

その土台になっているのは、高度なプログラミングでも高価なツールでもありません。**事業に関わるすべてを、1つのフォルダにまとめて、AIと共有する**——たった1つ、この工夫です。

この記事では、その全体像を全部公開します。

考え方から、フォルダの実際の中身、毎晩動いている自動バックアップの仕組み（スクリプト全文つき）、経理をAIに任せるときの安全装置まで。約2万字あるので、読み物というより「手元に置いて、作りながら使う地図」として書きました。

私の実際のフォルダは顧客情報だらけでお見せできないため、構造をそのまま写し取った架空の会社を1つ用意して、その中身を開いて見せながら進めます。

## 第1部　なぜ「1つのフォルダ」に会社を丸ごと入れるのか

### AI活用が続かない理由は、ツールの性能ではない

うまくいかないときのパターンは、だいたい決まっています。

* 文章はChatGPT、画像はデザインツール、タスクは別のアプリ、データはスプレッドシート——**作業ごとにAIとツールを足していく**
* 便利になった気がするが、ツールごとに情報が分断されて、**管理する場所だけが増えていく**
* AIに何か頼むたびに、「私はこういう事業をしていて、この案件はこういう経緯で……」と**毎回ゼロから説明**する
* 説明が面倒になって、だんだん使わなくなる

![](https://assets.st-note.com/img/1783068506-jMTidv9UcWJB0KX6HhxqP5V8.png?width=1200)

これはツールの性能の問題ではありません。構造の問題です。

考えてみると、AIは「ものすごく優秀だけれど、あなたの会社のことを何ひとつ知らない新人」です。

優秀な新人でも、会社の資料があちこちに散らばっていて、どこに何があるか誰も説明できない状態では、働きようがありません。

逆に、資料が1か所に整理されていて「ここを読めば全部わかる」状態なら、初日から動けます。

**AIの性能はもう十分に高い。足りていないのは、AIが読める形に整理された「事業の全体」のほう**——これが、私がこの数年AIで会社を回してきて到達した結論です。

### 発想を逆にする：AIに合わせて、会社を「読める形」にする

そこで発想を逆にします。ツールを足していくのではなく、先に「**AIが事業全体を読める場所**」を1つ作る。

事業に関わるもの——集客用の記事、SNSの下書き、顧客とのやり取りのメモ、見積もりの控え、経理データ、自動化のスクリプト、そしてAIへの指示書——を、最初から全部1つのフォルダに入れておく。  
すると、こうなります。

さらにもう一歩進めて、私はフォルダの中を**会社の組織図と同じ形**にしています。

事業ごとのフォルダ、顧客案件のフォルダ、経理のフォルダ、そして全体を見渡す司令室のフォルダ。1人の会社でも、構造だけは「会社」として持つ。そうすると、AIはその構造ごと事業を理解します。

![](https://assets.st-note.com/img/1783068496-jh70aUHlfFB251X4ndLgkue8.png?width=1200)

「フォルダの構造＝組織の構造」。この記事で伝えたいことの半分は、実はこの1行です。残りの半分は、それを実際にどう作り、どう毎日回すか。ここから順番に見ていきます。

## 第2部　土台の作り方——「賢いフォルダ」と自動バックアップ

まず土台からです。

この部だけ少しコマンドが出てきますが、すべてコピペで動きますし、後述するAIに「作って」と頼めば代わりにやってくれるものばかりです。仕組みの意味だけ押さえてください。

### 事業フォルダを「git」で賢くする

事業用のフォルダを1つ作り、「git（ギット）」という仕組みで管理します。gitと聞くと身構えるかもしれませんが、要は「**変更履歴を全部自動で記録してくれる、賢いフォルダ**」です。

いつ何を変えたかが残るので、間違えても巻き戻せるし、まるごとクラウドに控えを置けます。

```
mkdir my-company
cd my-company
git init
```

次に、GitHubというサービスで**非公開（private）のリポジトリ**を1つ作り、手元のフォルダと繋ぎます。  
「リポジトリ」はgitで管理されたフォルダの呼び名です。これで、手元の変更がクラウド側にも控えられる状態になります。

```
git remote add origin <あなたのGitHubリポジトリのURL>
git add -A
git commit -m "最初のコミット"
git push -u origin main
```

中身の整理は、最初はこれくらいで十分です。

* data/ — 残しておきたいデータ（議事録、要件、メモ）
* output/ — AIが作った使い捨ての生成物（再生成できるもの）
* scripts/ — 自動化のスクリプト
* assets/ — 画像・動画（後でGoogle Driveに逃がします。後述）
* CLAUDE.md — AIへの説明書（これが効きます。すぐ後で説明）

大事なのは、**完璧な構造を最初から目指さないこと**。  
フォルダ分けは後からいくらでも直せます。「1つの場所に集める」ことのほうが、何倍も価値があります。

### AIへの説明書「CLAUDE.md」を置く

ここからがAIと組む本番です。私はAnthropicの「**Claude Code**」というAIを使っています。チャット画面にコピペするタイプではなく、**パソコンのフォルダの中身を直接読み書きできるタイプ**のAIです。  
フォルダに全部入れる工夫が最大限に効くのは、このタイプと組んだときです。

フォルダの一番上に CLAUDE.md というファイルを置きます。これは「**AIへの説明書**」で、Claude Codeはこのフォルダで作業するとき、毎回まずこれを読んで文脈を理解してから動きます。  
書いておくのは、たとえばこんなことです。

たったこれだけで、AIの仕事の精度がまるで変わります。「どこに何があるか」「何を大事にしているか」を分かった状態で動けるので、丸投げしても的を外さなくなる。

会社でいえば、就業規則と業務マニュアルの1枚目にあたるファイルです。

### 毎晩23時、自動でバックアップされる状態にする

土台の仕上げは、「**気づいたら毎日バックアップされている**」状態を作ることです。手で毎日保存操作をするのは、まず続きません。

なので、毎日決まった時刻に変更を自動で記録してGitHubへ送る仕組みを入れます。Macなら「launchd」という標準機能で動かせます。

これが、私が実際に毎晩動かしているスクリプトです（1リポジトリ向けに整理したもの）。そのまま使えますし、Claude Codeに「このフォルダを毎日23時に自動でcommitとpushするスクリプトを書いて」と頼めば、あなたの環境に合わせたものを作ってくれます。

```
#!/bin/bash
# git-autocommit.sh — リポジトリの変更を自動でcommit+push
REPO_DIR="$HOME/work/my-company"            # ← 自分の事業フォルダに変える
LOG_FILE="$HOME/Library/Logs/git-autocommit.log"  # ← Desktop配下は避ける（後述）
DATE=$(date +"%Y-%m-%d %H:%M:%S")

cd "$REPO_DIR" || exit 1

# 変更がなければ何もしない
if [ -z "$(git status --porcelain)" ]; then
  echo "[$DATE] 変更なし" >> "$LOG_FILE"
  exit 0
fi

# 全変更をコミット
git add -A
git commit -m "chore: 自動バックアップ $DATE" >> "$LOG_FILE" 2>&1

# リモートと同期してからpush（競合したら中断して通知）
branch=$(git branch --show-current)
git fetch origin "$branch"
if ! git rebase "origin/$branch"; then
  git rebase --abort
  osascript -e 'display notification "競合でpushを中断しました" with title "自動バックアップ"'
  exit 1
fi
git push origin "$branch" >> "$LOG_FILE" 2>&1
echo "[$DATE] バックアップ完了" >> "$LOG_FILE"
```

このスクリプトを毎日23時に動かす設定ファイルがこちらです。~/Library/LaunchAgents/com.yourname.git-autocommit.plist という名前で保存します。

```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.yourname.git-autocommit</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>/Users/あなた/work/my-company/scripts/git-autocommit.sh</string>
  </array>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key><integer>23</integer>
    <key>Minute</key><integer>0</integer>
  </dict>
</dict>
</plist>
```

保存したら、有効化はこの1行です。

```
launchctl load ~/Library/LaunchAgents/com.yourname.git-autocommit.plist
```

これで、毎晩23時にその日の変更が自動で記録され、GitHubに送られます。**「バックアップを忘れる」という事故が、構造的に起きなくなります。**

### 画像・動画は、Google Driveに逃がす

1つだけ大事な注意があります。**画像や動画を、このリポジトリに直接入れてはいけません。**

gitは文章やコードのような軽いテキストの管理は得意ですが、重い画像・動画を入れるとフォルダが一気に膨らんで、動作が重くなります。

そこで役割を分けます。

やり方は、画像類の実体を「Google Drive for desktop」の同期フォルダに置き、リポジトリの中からは**symlink（ショートカットのようなもの）で参照する**だけです。

```
# assetsの実体はGoogle Driveの同期フォルダに置き、リポジトリ内にショートカットを張る
ln -s "$HOME/Google Drive/My Drive/company-assets" assets
```

そして、git管理からは外します。

```
# .gitignore に1行追加
assets/
```

これで、テキストはGitHub、素材はGoogle Drive、**2つのクラウドに自動で控えが取られる状態**になります。

![](https://assets.st-note.com/img/1783068487-l2MVmZ0hXJHbDgSUKFTivawO.png?width=1200)

### 私がハマった詰まりどころ（先回りして潰しておく）

ここは実際に私が踏んだ落とし穴です。先に知っておくと、丸一日溶かさずに済みます。

**① 自動実行が「権限」で失敗する。** launchdからスクリプトを動かすと、DesktopやDocuments配下のフォルダはmacOSのプライバシー保護に弾かれて失敗することがあります。→「システム設定 > プライバシーとセキュリティ > フルディスクアクセス」で、スクリプトを動かすシェル（/bin/bash など）に許可を与えます。

**② ログの出力先もDesktop配下を避ける。** 同じ理由で弾かれます。→ ログは ~/Library/Logs/ のような場所に出すのが安全です（上のスクリプトもそうしています）。

**③ ログや一時ファイルはgit管理から外す。** 毎晩の自動記録がノイズで汚れます。→ .gitignore にログや使い捨ての output/ を入れておきます。

**④ 自動送信の競合。** 複数の場所から同じリポジトリを触ると、自動pushがぶつかることがあります。→ 上のスクリプトのように「競合したら中断して通知」にしておけば、勝手に壊れることはありません。

もうひとつ、これは最近の実話です。私の自動バックアップは、ある時期**11日間、誰にも気づかれずに止まっていました**。原因は通信時の認証待ちで固まったこと。以来、スクリプトに「応答がなければ待たずに中断する」設定を足し、失敗したらMacの通知が出るようにしました。自動化は「作って終わり」ではなく、**止まったことに気づける仕掛け**までがセットです。この失敗談だけで1本書けるくらいなので、詳しくは別記事にします。

ここまでで土台は完成です。履歴が残り、毎晩勝手にバックアップされ、AIが全体を読める「賢いフォルダ」ができました。ここから、これを「会社の形」にしていきます。

## 第3部　会社の形にする——架空の会社「sample-company」の中身を全部見せます

ここからが本題です。実際の私のフォルダは顧客情報の塊なのでお見せできません。かわりに、**構造をそのまま写し取った架空の1人会社**を用意しました。設定はこうです。

ごく普通の、1人でいくつかの仕事を回している状態です。この会社のフォルダ全体がこちら。

![](https://assets.st-note.com/img/1783068477-hkbDWafHyZKAsd50OgM7R9lo.png?width=1200)

```
sample-company/
├── CLAUDE.md            ← AIへの会社全体の説明書
├── businesses/          ← 自社事業（1事業=1フォルダ）
│   ├── coffee-shop/         コーヒー豆ショップ
│   └── ai-newsletter/       AIニュースレター
├── clients/             ← 顧客案件（1クライアント=1フォルダ）
│   ├── _template/           新規案件の雛形
│   ├── acme-cafe/           カフェのSNS運用代行
│   └── sakura-clinic/       クリニックの予約対応AI
├── back-office/         ← 経理部
│   ├── billing/             請求（お金を受け取る側）
│   └── accounting/          会計（帳簿をつける側）
├── command/             ← 司令室（全体を見渡す場所）
│   ├── CLAUDE.md            この部署の定義
│   ├── DASHBOARD.md         全体の状態を一望
│   └── NOW.md               今週の優先順位
├── scripts/             ← 会社の設備（自動バックアップ等）
├── assets/              ← 画像・動画（Google Drive側に実体）
└── output/              ← AIの使い捨て生成物
```

見た目はただのフォルダです。でもこれは**組織図**です。自社事業の部署、顧客案件の部署、経理部、そして司令室。1人の会社の全部が、この1枚に収まっています。順番に開けていきます。

### businesses/ — 自社事業は「1事業＝1フォルダ」

自社でやっている事業は、事業ごとにフォルダを分けます。コーヒー豆ショップのフォルダには商品リストとSNSの下書きが、ニュースレターのフォルダには配信の下書きとネタ帳が入っている。それだけです。

ポイントは、各事業フォルダにも小さな CLAUDE.md（説明書）を置くこと。「この事業は何か・何を大事にするか・文章のトーン」を書いておくと、AIに「コーヒーショップの今週のSNS下書きを作って」と頼んだとき、**その事業の文脈で**書いてくれます。ニュースレターの硬めの文体と、ショップの柔らかい文体を、AIがフォルダごとに使い分けるようになります。

「説明書」といっても、身構えるようなものではありません。コーヒー豆ショップの説明書は、たとえばこの程度です。

```
# coffee-shop（コーヒー豆のネットショップ）

## この事業について
自家焙煎のコーヒー豆を、オンラインで週替わり販売している。
お客さんは30-40代、コーヒーは好きだが詳しくはない人が中心。

## 文章のトーン
- 専門用語を使わない（「浅煎り」はOK、「シティロースト」はNG）
- 押し売りしない。淹れる時間の心地よさを伝える
- 絵文字は1投稿に1つまで

## 置くもの
- products.md — 商品リスト（価格・在庫・説明文）
- sns下書き.md — 投稿の下書き置き場
```

10行ちょっとの、ただのメモです。でもこれがあるだけで、AIの出力から「それっぽいけど、うちの店じゃない文章」が消えます。**AIへの指示を毎回頑張るのではなく、一度だけ書いた説明書に働かせる。** この感覚が掴めると、AI活用は一気に楽になります。

### clients/ — 顧客案件は「全部同じ形」にする。これが効く

受託の仕事は clients/ に、**1クライアント＝1フォルダ**で入れます。ここで一番大事なのは、**どの案件フォルダも完全に同じ構造にする**ことです。

なぜ同じ形にこだわるのか。理由は3つあります。

* **AIが迷わない。** どの案件でも「messagesを読んで状況をまとめて」が通じる。案件ごとに構造が違うと、AIへの説明が毎回必要になる
* **自動化が効く。** 「全案件の状態を一覧にして」のような横断的な頼み事が、構造が揃っていて初めて可能になる
* **自分も迷わない。** 3か月ぶりに開いた案件でも、どこに何があるか体が覚えている

そして各案件の CLAUDE.md の冒頭には、こういう「機械が読めるメモ」を書いておきます。

```
---
client: Acme Cafe
stage: active（進行中）
next_action: 7月分の投稿カレンダーを作って共有
next_action_by: 2026-07-05
---
```

現在の状態（進行中か、見積もり中か）、次にやるべきこと、その期限。たった数行ですが、これを全案件で揃えておくと、AIが**全案件の頭の数行だけを読んで、会社全体の状況を数秒で把握できる**ようになります。「期限が近い案件は？」「返事待ちで止まっている案件は？」に、AIが即答できる。後で説明する司令室は、この仕組みの上に成り立っています。

![](https://assets.st-note.com/img/1783068467-cZtzBfLjVENPIa2yDCQ951xm.png?width=1200)

やり取りの記録も、ルールを1つだけ決めています。ファイル名を 日付-手段-用件.md（例: 2026-07-01-email-見積もり送付.md）にする。これだけで、フォルダを開けば時系列に並び、AIも「経緯を追って」が正確にできます。

この「記録が案件フォルダに揃っている」状態が、実務でどれくらい効くか。たとえば、しばらく止まっていた案件から急に返信が来たとき、私はこう頼むだけです。

**私**「さくらクリニックさんから返信が来た。これまでの経緯を要約して、返信の下書きを作って」

**AI**「messagesフォルダを確認しました。経緯：5月に予約対応AIの相談があり、6月に要件を整理（診療時間外の一次対応・予約変更の受付が主目的）、こちらから概算を提示して返答待ちでした。今回の返信は『院内で検討し、進めたい』という内容です。次の段階は正式見積もりなので、お礼と、見積もり送付の日程を伝える返信案を作りました——」

過去のやり取りを読み返して思い出す時間が、ほぼゼロになります。記憶を自分の頭ではなく**フォルダに置いておき、思い出す作業はAIにやらせる**。案件が増えるほど、この差は大きくなります。

もう1つ、細かいけれど効くのが \_template/（雛形）です。新しい案件が来たら、この雛形フォルダをコピーして名前を変えるだけ。3分で「いつもの形」の案件フォルダが立ち上がります。構造を揃える仕組みは、意思の力ではなく雛形で担保する——散らからないコツです。

### back-office/ — 経理をAIに任せる。ただし「安全装置」つきで

経理は2つに分けています。**billing（請求）＝お金を受け取る表側**と、**accounting（会計）＝帳簿をつける裏側**。請求書の準備と入金の確認はbillingで、売上や経費の記帳はaccountingで。1人でやっていると混ざりがちなこの2つを、フォルダで物理的に分けます。

そして、ここが今日いちばん強調したいところです。経理はAI活用の効果が大きい領域ですが、**間違いが許されない領域**でもあります。だから私は、AIに渡す権限を3段階に分けています。

* **読む — 自由。** 帳簿の照会、集計、「今月の経費をまとめて」——確認だけの作業はAIに任せ放題にする
* **書く — 下書きまで。** 記帳や仕訳の登録はAIが**下書き（未確定）の状態まで**。確定ボタンは必ず人間が押す
* **出す — 人間だけ。** 請求書の送付、税務書類の提出、振込——**会社の外に出る操作はAIには絶対にさせない**。AIは準備まで

![](https://assets.st-note.com/img/1783068457-fVOeoJuvrtTjyRsiqHNxAng2.png?width=1200)

この線引きをフォルダの CLAUDE.md に明文化しておくと、AIはそれを読んで、自分から「下書きまで作りました。確定はご確認ください」という動き方をするようになります。**AIを信用して使い続けられるのは、信用しすぎない仕組みがあるから**です。ここを曖昧にしたまま経理をAIに触らせるのは、おすすめしません。

実際の月次の流れは、こうなっています。月初に会計フォルダでAIに「先月分の記帳をまとめて」と頼む。AIが売上の記録・経費のメモ・決済サービスの明細を突き合わせて、記帳の下書きを一覧で出してくる。私はその一覧を眺めて、違和感のある行だけ質問して直し、まとめて確定する。以前は月末のたびに半日単位で溶けていた作業が、**「一覧を確認して確定する」だけの30分**になりました。

もう1つ、経理まわりで大事な運用ルールがあります。**口座残高や顧客の実データのような機密は、バックアップ用のクラウド（GitHub）に乗せない**こと。私は経理フォルダの中に「git管理から外した専用の置き場」を作り、生データはそこかGoogle Drive側にだけ置いています。フォルダに全部集める方針と、機密の置き場所を選ぶ慎重さは、両立できます——というより、**両立させるために置き場のルールを最初に決めておく**のが正解です。

### command/ — 司令室。「もう1人の自分」を置く場所

最後が、この仕組み全体の要です。command/（司令室）には、実作業のファイルを一切置きません。あるのは3枚のメモだけです。

* CLAUDE.md — **定義**。「ここは会社全体を見渡して優先順位を決める部署。実作業はしない」とAIに教える
* DASHBOARD.md — **状態**。全案件・全事業・経理の現状を1枚で一望する
* NOW.md — **行動**。今週やることを、全社横断で優先順に並べたリスト

「状態を見る紙」と「動く順番を決める紙」を分ける。そしてAIには「ここは司令室だ」と教えておく。すると何が起きるかというと——**朝、このフォルダを開いてAIに「今日は何から?」と聞くだけで、会社全体を踏まえた答えが返ってくる**ようになります。

1人で会社をやっていて本当に怖いのは、作業量ではなく「**見落とし**」です。返事を忘れていた案件、静かに過ぎていた期限、止まっていることに気づかなかった自動化。司令室は、その見落としを拾うための「もう1人の自分」です。ちなみに私は自分の司令室のAIに名前をつけて運用しています（Argus——ギリシャ神話の百目の巨人から取った、私が勝手に命名した秘書AIです。名前をつけると、指示が「Argusに聞いて」で済むようになります）。

これで会社の形は完成です。フォルダを見れば会社の全部が分かる。そして重要なのは、**あなたに見えているものと同じものが、AIにも見えている**ということです。

## 第4部　この会社は、1日をこう回っている

形ができたところで、実際の1日の流れをお見せします。sample-companyの設定で再現しますが、やっていることは私の毎日そのままです。

### 朝——司令室を開いて、AIと今日を決める

朝、司令室フォルダでAIを起動して、一言聞きます。

**私**「今日は何から手をつけるべき？」

**AI**「NOW.mdと全案件の状態を確認しました。最優先は**さくらクリニックの見積もり送付**です。期限が今日（7/3）になっています。次にAcme Cafeの7月分投稿カレンダー、こちらは期限まで2日です。気になる点が1つ——ニュースレターの配信が前回から9日空いています。下書きだけでも今日作っておきますか？」

この間、数十秒です。AIは全案件のフォルダの頭にある状態メモ（第3部のあれです）を横断して読み、期限と状況を突き合わせて答えています。私が全フォルダを開いて回る必要はありません。**「今日何をすべきか」を思い出すという、地味に消耗する仕事が朝から消えている**——これが司令室の効きめです。

### 日中——各部署のフォルダで、AIと実作業

日中は、各フォルダに「出社」して作業します。ここでも、頼み方はすべて同じです。フォルダを開いて、AIに頼む。

* **clients/acme-cafe で**：「先月の投稿の反応を踏まえて、7月分の投稿カレンダー案を作って」→ 過去のやり取りと実績がフォルダにあるので、文脈込みの案が出てくる。私は直して確定するだけ
* **clients/sakura-clinic で**：「これまでの要件メモから見積もりの下書きを作って。医療案件なので"AIの回答は必ず人が確認する運用"を明記して」→ 下書きができる。**送るのは私**
* **businesses/ai-newsletter で**：「ネタ帳から今週の配信下書きを1本」→ 事業フォルダのトーン設定どおりの文体で下書きが出る

共通しているのは、**AIは下書きと準備まで、外に出す最後の1歩は必ず人間**という原則です。メールの送信、SNSの公開、請求書の発行——ぜんぶ同じ。この線を守っている限り、AIがどれだけ深く事業に入っても、事故は「下書きの手直し」で済みます。

作業して状況が動いたら、案件フォルダの状態メモを更新しておきます（「見積もり送付済み、返事待ち」など）。これは翌朝の司令室の精度に直結するので、私は**メッセージを送ったらメモも更新**、をワンセットの癖にしています。

なぜここまで癖として徹底しているかというと、サボって痛い目を見たからです。一時期、状態メモの更新を後回しにしていたら、司令室のAIが**1か月前の情報で「この案件の期限が過ぎています」と誤報**してくるようになりました。AIは嘘をつきませんが、**古い情報を渡せば、古い情報のまま正確に間違えます**。この仕組みの唯一の弱点は「現実とメモのズレ」で、それを防ぐのが「動いたら更新」のワンセット習慣です。逆にいえば、守るべき規律は実質これ1つしかありません。

![](https://assets.st-note.com/img/1783068447-OZanAwHxbmvy829iSgKXDfVl.png?width=1200)

### 夜23時——何もしない。自動バックアップが走る

夜は何もしません。23時になると、第2部で仕込んだ仕組みが勝手に動いて、その日の変更をぜんぶ記録してクラウドに送ります。翌朝のAIは、その履歴も含めて会社を把握した状態で立ち上がります。

### 週に1度——AIと一緒に会社を棚卸しする

毎日のリズムに加えて、週に1度だけ「棚卸し」をやります。司令室でAIに頼むのはこれだけです。

**私**「週次レビューをお願い。止まっているもの、忘れているものを全部挙げて」

**AI**「確認しました。3点あります。①Acme Cafeの状態メモが10日間更新されていません——作業は進んでいるようなので、メモが現実とズレている可能性があります。②ニュースレターの配信間隔が先月平均5日→今月9日に伸びています。③見積もりを送って2週間返答がない案件が1件。そろそろ一度だけ様子うかがいの連絡を入れる時期です」

人間の記憶と注意力は、案件が5つを超えたあたりから確実に取りこぼしを始めます。私がこの仕組みで一番買っているのは、作業の速さではなく、**この「忘れていることを思い出させてくれる」機能**です。1人でやっていると、指摘してくれる同僚はいないので。

### 会社の仕事の全体で見ると、AIはここまでやっている

1日の流れを、業務の種類で並べ直すとこうなります。私の実際の分担です。

* **集客**：記事・SNSの下書き、ネタ出し → ほぼAI。公開の判断と最終手直しが人間
* **営業**：提案書・見積もりの下書き → AIが8割。数字の確定と送付は人間
* **制作**：サイトや資料づくりの実作業 → AIが主力。設計の判断と検収が人間
* **顧客対応**：返信の下書き、経緯の要約 → AI。送信は必ず人間
* **請求**：請求書の準備、入金の確認 → AI。発行・送付は人間
* **会計**：記帳の下書き、集計、月次の締めチェック → AI。確定は人間
* **経営管理**：全体の状況把握、優先順位の提案、期限の見張り → ほぼAI（司令室）。決めるのは人間

![](https://assets.st-note.com/img/1783068437-dq7gtphAlfm26SLcTnvB5NrK.png?width=1200)

眺めてもらうと分かるとおり、パターンは1つしかありません。**手を動かすのはAI、決めて世に出すのは人間。**

これを全業務で徹底しているだけです。そしてこの分担が成立しているのは、繰り返しになりますが、AIが会社全体を読める「1つのフォルダ」があるからです。

## 第5部　どの順番で積み上げるか——全部一気にやらない

ここまで読んで「やることが多い」と感じたかもしれません。安心してください。私もこの形に一度で辿り着いたわけではなく、**順番に積み上げた結果**がこれです。おすすめの順番を書きます。

**ステップ1：土台だけ作る（初日）。** フォルダを1つ作って git init、GitHubに繋いで、自動バックアップを仕込む。第2部の内容だけ。中身の整理はまだしなくていい。

**ステップ2：CLAUDE.mdを書く（2日目）。** あなたの事業をAIに説明する1枚を書く。完璧を目指さず、10行でいい。AIの働きが目に見えて変わるのはここからです。

**ステップ3：一番「時間が溶けている」業務を1つだけAI化する（1〜2週目）。** 全部やらない。選び方の目安は、(1) 同じ情報を2回以上手で書き写している所、(2) 毎週必ず発生する所。顧客への返信づくり、記事の下書き、データの転記——どれか1つに絞って、そのフォルダを作り、AIに任せる型を作る。

**ステップ4：うまくいった頼み方を「手順書」として残す（随時）。** 一度うまくいったら、その頼み方と手順をmdファイルにしてフォルダに保存しておく。次からは「あの手順書どおりに」で同じ品質が再現されます。これを繰り返すと、**会社に「やり方」が資産として貯まっていきます**。人を雇っていないのに、フォルダの中に社員が増えていく感覚です。

**ステップ5：案件が2つ以上になったら、司令室を作る（1〜2か月目）。** DASHBOARDとNOWの2枚から。管理するものが増えて「見落としが怖い」と感じ始めたときが作りどきです。

![](https://assets.st-note.com/img/1783068427-7bfvg1wJjrp8k04i9XBKeQyF.png?width=1200)

### 「手順書」は、これくらい簡単でいい

ステップ4の手順書について、イメージが湧くように実例を1つ。顧客への返信づくりを型にした手順書は、たとえばこうです。

```
# 手順書：顧客への返信下書き

1. その案件の messages/ を新しい順に読み、経緯を把握する
2. 返信の下書きを作る。守ること：
   - 結論を最初に書く（お礼→回答→次の一歩の順）
   - こちらから提示する日程は必ず2択にする
   - 専門用語はお客さんが使った言葉だけを使う
3. 下書きの最後に「送信前チェック：金額・日付・宛名」を添える
4. 私が確認して送信する（AIは送信しない）
```

これも、ただのmdファイルです。次からは「あの手順書どおりに、この案件の返信を作って」で、同じ品質が出てきます。コツは、**AIがうまくやれた直後に「今のやり方を手順書にして保存して」と頼む**こと。手順書を書く作業すらAIにやらせるわけです。こうして貯まった手順書の束が、そのまま「あなたの会社のやり方」の資産になります。

逆に、やらないほうがいいのは「**最初から完璧な構造を設計する**」ことです。構造は事業と一緒に育つものなので、最初の設計は必ず外れます。フォルダ分けの正解を悩む時間があったら、git init して1つ目の業務をAIに任せるほうが、100倍速く前に進みます。

## よくある質問——先回りで答えておきます

この話をすると必ず聞かれることを、まとめて置いておきます。

### Q. エンジニアじゃなくても、本当にできますか？

できます。この記事に出てきたコマンドは全部コピペで動きますし、詰まったら**エラー文をそのままClaude Codeに貼り付けて「直して」と言う**のが現代の解決法です。大事なのはコマンドを理解することではなく、「フォルダに集める・説明書を書く・下書きはAI・確定は自分」という運用の考え方のほうです。考え方さえ合っていれば、技術的な部分はAIが埋めてくれます。

### Q. Windowsでも同じことができますか？

できます。gitもGitHubもClaude CodeもWindowsで動きます。違うのは自動実行の仕組みだけで、Macの「launchd」の代わりに「タスクスケジューラ」を使います。ここもClaude Codeに「このスクリプトを毎日23時にタスクスケジューラで動かす設定にして」と頼めば設定まで案内してくれます。

### Q. 顧客の情報をAIに読ませて大丈夫なんですか？

ここは丁寧に答えます。私の運用ルールは2段階です。まず、**機密度の高い生データ（契約書の実データ、口座明細、個人情報）は、そもそもAIが日常的に読むフォルダに置かない**。専用の置き場を分け、バックアップ用クラウド（GitHub）にも乗せません。次に、AIに読ませる情報も「業務に必要な範囲」に絞る——案件の経緯メモは読ませるが、たとえば相手の与信情報のようなものは要りません。あわせて、使っているAIサービスのデータ取り扱い規約（入力内容が学習に使われるかどうか等）は、自分のプランについて一度確認しておくことをおすすめします。**道具の問題というより、置き場所のルールの問題**です。ルールを最初に決めれば、両立できます。

### Q. 費用はどれくらいかかりますか？

私の構成では、AIの利用料が月数千円〜（プランによります）、GitHubは非公開リポジトリでも無料、Google Driveは既存の容量で足ります。つまり**ほぼAIの利用料だけ**です。この記事に書いた内容に、高価なツールは1つも要りません。

### Q. 途中でフォルダ構造を変えたくなったら？

変えて大丈夫です。というより、必ず変えたくなります（事業が変わるので）。そのときは、フォルダの移動や改名を**参照の修正ごとAIに頼む**のがコツです。実は昨日、私は自分の会社フォルダで経理まわりの再編成と部署フォルダの改名をやったのですが、フォルダ名に触れている数百箇所の記述の洗い出しと修正を、ほぼ全部Claude Codeにやらせました。「◯◯を△△に改名して、壊れる参照も全部探して直して。終わったら残りがないか検証して」——この頼み方で通ります。構造は育てるもの、育て直しの作業はAIの仕事。この分担なら、最初の設計に悩む必要がなくなります。

## まとめ——今日やるのは、フォルダを1つ作ることだけ

長い記事をここまで読んでいただき、ありがとうございました。全体を4行にまとめます。

* 事業をまるごと**1つのフォルダ**（リポジトリ）に集める。フォルダの構造＝組織の構造
* CLAUDE.md でAIに会社を理解させ、**毎晩の自動バックアップ**で守る
* 顧客案件は**全部同じ形**、経理は**3段階の安全装置**、全体は**司令室**で見渡す
* 分担は常に1つ：**手を動かすのはAI、決めて世に出すのは人間**

派手なAIアプリは1つも出てきませんでした。でも、この地味な土台があるかどうかで、AI活用が「積み上がる」か「散らかって消える」かが決まります——これが、1人で会社を回してきた私の、いちばん確かな実感です。

まず今日は、フォルダを1つ作って git init するところから。それだけで、AIと本気で組む準備が始まります。
