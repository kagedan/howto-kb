## この記事で伝えたいこと

オーケストレーションツール自体をCLIなどでAIが触れるようにして任せてみると、承認の手間から解放されるかもということ

## 承認・応答が手間という困りごと

AIコーディングエージェント向けのオーケストレーションツールを自作して使っています（Claude CodeとCodexで）。

仕組みとしてはシンプルで、いろいろなループを定義してフローとして登録することができるもの。

今メインで使ってるのはタスク実行フローで、自作のタスク管理ツールにタスクを登録しておくと、オーケストレーターがタスクを定期的にポーリングして拾い、AIエージェントに実行させてくれるというもの。タスクごとにTeam Leadエージェントが立ち上がって、必要に応じてExecutor（実装担当）やReviewer（レビュー担当）といったサブエージェントを起動してよしなに進めてくれます。

やってることはこれとほぼ同じです。

<https://developers.openai.com/codex/integrations/linear>

他にも探索系のフローなどをいくつか登録して、不定期で使っています。

### 困っていたこと

ツール自体はSDKとCodex App Serverを使って作成しているのと、フローにつかっているSkillチェーンの内容もあまり対話はないようにしています。

なので、Claude Codeのような応答待ちで停止するみたいなのは基本的にないんですが、dangerousで実行する様にしていないため、ツール実行時に承認作業が発生して、毎回承認する手間が発生しています（settingsの設定をサボっているのも原因）。

安全なコマンドがほとんどなので基本フル承認するだけになって、結構面倒に感じていました。

パソコンに張り付くのも面倒の一つだったのでスマホからも承認・応答をできるようにしたので、ある程度楽にはなりましたが、それでも面倒。

## 起動・承認・応答をAIに代行してもらえるCLIを作ったら解放されそう

最近ペインでブラウザを開けるのが便利ということでcmuxを使い始めて、別ペインでClaude Codeを起動して、そこにシグナルを送れる機能を知りました。この記事がとても参考になりました。

<https://zenn.dev/hummer/articles/cmux-ecosystem-claude-code>

この機能を使ってサブエージェントではなく目視で動きを確認したい時などに、別ペインでClaudeを立ち上げてもらって、よしなに操作してもらっていました。その際、承認作業はどうしますか？みたいなことをたしか聞いてくれたので、安全な内容に関しては対応してもらう様に伝えるといい感じにやってくれて、便利に使っています。

これを使っていた時に、今使っているオーケストレーションツールの承認・応答自体も同じ様にAIに対応してもらえるようにすれば楽になるかも？ということで、オーケストレーションツール自体を操作するCLIを作りました。

### どんな使い方ができるCLI？

オーケストレーションツールは、承認・応答・フローの起動・停止などの操作はすべてUIから行えるようになっていました。なので、CLIはその薄いクライアントとして実装したので、基本的にUIでできることを全部できるようにしています。加えて、ツール自体の起動などもできるようにしています。

主にできることは以下です。

* 全体の起動・停止: バックエンド+フロントエンドをまとめて起動・停止。個別にも操作可能
* ステータス確認: 今何が動いているか、承認待ちがあるかを一覧表示
* ツール承認・拒否: 名前の通り
* 質問への応答: エージェントからの質問に回答
* フローの起動・停止: 事前定義した任意のフローの起動と停止
* ブロッキング待機: WebSocket接続を維持して、次のイベントが来たら内容を返して終了。`--timeout` で待機時間も指定可能

オーケストレーションツール側では、エージェントがツールを実行しようとするとWebSocket経由で承認許可イベントをbroadcastして、応答が来るまでエージェントの実行をブロックします（UIではこれをリアルタイムで受け取って承認ボタンを出しています）。

CLIでは `wait` して待機していると、承認要求が来た瞬間にClaude Codeに通知が届くので、内容を見て判断して `approve` もしくは `reject` を返す。終わったらまた `wait` を実行する、というループになります。

### skillにして使ってもらっている

あとは、この使い方をskillにして、よしなにやってもらっています。

1. `status` で溜まっている承認・応答を確認
2. 安全な操作なら自動承認、危険なもの（破壊的コマンド等）は人間にエスカレーション
3. `wait` で次のイベントをブロッキング待機
4. イベントが来たら処理して、1に戻る

```
## 概要

<オーケストレーションツール> の運用を自律的に行うためのプロシージャ。
セッション開始時にデフォルト方針を提示して確認を取り、その後は自律ループで承認・応答・レビューを処理する。

## ワークフロー

### Phase 1. 方針確認

デフォルト設定を提示し、ユーザーに確認を取る。AskUserQuestion ツールを使って質問する。

提示する内容とデフォルト値

- サーバー起動: バックエンド＋フロントエンド両方（デフォルト）
  - バックエンドのみも選択可
- 起動する workflow: <ワークフロー名>（デフォルト）
- 承認の原則（承認リクエストは settings.json の allow/deny でフィルタ済み。来るのは ask に該当するもの、またはどのルールにもマッチしなかったもの）
  - 自動承認: タスクの目的に沿っており、かつ影響が限定的で回復可能と判断できる操作
  - ユーザーに確認: 操作の意図がタスクの目的から外れている、影響が不可逆または広範囲、自信を持って判断できない、のいずれかに該当する場合
- エージェントからの質問 → 自分で答えられるものは答える。ユーザーの意思決定が必要なものや自信を持って答えられないものはユーザーに転送
- Run レビュー → 結果を要約してユーザーに判断を仰ぐ

質問の形式例

「以下の方針で <オーケストレーションツール> を運用します。修正があれば教えてください。

1. バックエンド＋フロントエンドを起動（`<オーケストレーションツール> start`）
2. <ワークフロー名> workflow を起動
3. タスク目的に沿った限定的な操作は自動承認、それ以外はあなたに確認
4. エージェントの質問は自分で答えられるものは答え、判断に迷うものはあなたに転送
5. Run 完了時は要約してあなたにレビュー依頼

これで進めていいですか？」

ユーザーの回答が OK 系ならそのまま進む。
ユーザーが「フロントはいらない」と言った場合はバックエンドのみ起動に変更。
修正がある場合はその内容を反映して Phase 2 に進む。

### Phase 2. 初期化

1. Phase 1 で確認した起動構成に従ってサーバーを起動
   - 両方起動（デフォルト）: `<オーケストレーションツール> start`
   - バックエンドのみ: `<オーケストレーションツール> server-start`
2. 起動結果を確認し、エラーがあればユーザーに報告
3. `<オーケストレーションツール> status` でオーケストレーターの状態を確認
4. 接続エラーの場合はユーザーに報告して指示を仰ぐ
5. 起動する workflow のオーケストレーターが停止中なら: `<オーケストレーションツール> orchestrator-start <id>`
6. 既に起動中ならそのまま進む

### Phase 3. 自律ループ

以下を繰り返す

1. `<オーケストレーションツール> status` で現在の状態を確認
2. `actions` 配列の pending アイテムを Phase 1 の承認原則に従って処理
   - 自動承認と判断したもの → `<オーケストレーションツール> approve <tool_use_id>` を即実行
   - ユーザー確認が必要と判断したもの → 操作の内容と判断理由を要約してユーザーに確認を取る
   - human_input → 自分で回答できるものは `<オーケストレーションツール> respond` で回答。ユーザーの意思決定が必要なものはユーザーに転送
   - run_review → 結果を要約してユーザーにレビュー依頼。承認なら `orch run-approve`、修正なら `<オーケストレーションツール> run-revise`
3. 全 pending を処理したら `<オーケストレーションツール> wait` でブロッキング待機
4. イベントが来たら同じ原則で処理
5. 3 に戻る

### エスカレーション

以下の場合はユーザーに報告して指示を仰ぐ

- WebSocket 接続が切れた場合
- 自信を持って判断できない承認リクエストが来た場合
- Run が失敗した場合
- ユーザーから中断の指示があった場合

### 終了処理

ユーザーが停止を指示した場合、またはオーケストレーターの作業が完了した場合:

1. 起動中のオーケストレーターを停止: `<オーケストレーションツール> orchestrator-stop <id>`
2. Phase 2 で起動したサーバーを停止
   - 両方起動していた場合: `<オーケストレーションツール> stop`（フロントエンド→バックエンドの順で停止）
   - バックエンドのみの場合: `<オーケストレーションツール> server-stop`
3. 停止完了を報告して終了

### CLI コマンドリファレンス

| コマンド | 説明 |
|---------|------|
| `<オーケストレーションツール> start` | バックエンド＋フロントエンドを両方起動 |
| `<オーケストレーションツール> stop` | バックエンド＋フロントエンドを両方停止 |
| `<オーケストレーションツール> server-start` | バックエンドのみ起動 |
| `<オーケストレーションツール> server-stop` | バックエンドのみ停止 |
| `<オーケストレーションツール> frontend-start` | フロントエンドのみ起動 |
| `<オーケストレーションツール> frontend-stop` | フロントエンドのみ停止 |
| `<オーケストレーションツール> status` | 状態確認 |
| `<オーケストレーションツール> wait` | イベント待機（ブロッキング） |
| `<オーケストレーションツール> approve <id>` | ツール実行を承認 |
| `<オーケストレーションツール> reject <id>` | ツール実行を拒否 |
| `<オーケストレーションツール> respond <run_id> [text]` | human_input に回答 |
| `<オーケストレーションツール> run-approve <task_id>` | Run 完了を承認 |
| `<オーケストレーションツール> run-revise <task_id> [comment]` | Run の修正を要求 |
| `<オーケストレーションツール> orchestrator-start <id>` | オーケストレーター起動 |
| `<オーケストレーションツール> orchestrator-stop <id>` | オーケストレーター停止 |
```

承認ポリシーはセッション開始時に方針を確認する形にしています。「基本的に安全なコマンドは自動承認、判断に困ったらエスカレーション」くらいの粒度で伝えておけば、あとはやってくれます（多分）。

## まとめ

まだ使い始めたばかりなのでどこまでうまく動くか検証中ですが、いまのところうまくうごいてくれてます。

「とりあえず承認をしなくてよくなった」「画面に貼り付かなくてよくなった」がいけそうなので、とりあえず便利になっています。

なので、今はいったんうまく動くことを祈りつつ改修していくのに加えて、

* 承認・応答以外に、エラーでハマってそうなら調整する
* 微妙な動きをしていたら調整する

みたいなことができたら良さそうだと思っています。
