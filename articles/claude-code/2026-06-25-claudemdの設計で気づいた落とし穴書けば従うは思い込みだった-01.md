## はじめに

Claude Codeを使って個人開発をしていると、こんな経験をしたことはないでしょうか。

「CLAUDE.mdにちゃんと書いたはずなのに、セッションを始めるたびに同じことを言われる」

最初は自分の書き方が悪いのかと思いました。次にバグかと疑いました。しばらくの間、原因がよくわからないまま「まあいっか」とスルーし続けていました。

この記事は、その「なんか変だな」をちゃんと向き合った記録です。

---

## 自分のプロジェクトについて

FlutterでオフラインのAIメモアプリを個人開発しています。特徴はAIをすべてオンデバイスで動かす点で、flutter\_gemmaというパッケージを使ってGemma 3 1Bをローカル実行しています。

Claude Codeでの開発管理には、CLAUDE.mdとTASKS.mdの2ファイルを使っています。CLAUDE.mdにプロジェクトのルールや制約を書き、TASKS.mdで6つのフェーズに分けたタスクを管理するという構成です。

割とちゃんと設計したつもりでした。それでも問題は起きました。

---

## 何が起きていたか

問題の制約はこれです。

CLAUDE.mdにこう書いておけば、Claudeは順番にタスクをこなしてくれると思っていました。ところが実際には、セッションを開始するたびに確認のような文言が出てきます。

```
並行してタスクを進めないよう指示されていますが、確認してもよいですか？
```

オートモードで動かしているので、作業自体は止まりません。ただ、毎回こういった文言が警告のように表示されるので、だんだん気になってきました。「これ、ちゃんと機能しているのか？」「自分の書き方がまずいのか？」と不安になってくるわけです。バグなのか仕様なのか、しばらく判断できないままでした。

---

## 原因の考察

調べていくうちに、ある程度原因が見えてきました。ただし以下は推測も含みます。

### CLAUDE.mdの指示は全て従われるわけではない

Claude Codeの挙動を分析したHumanLayerのレポートによると、Claude Codeはセッション開始時にCLAUDE.mdの指示の上に次のような内部リマインダーを自動挿入しているそうです。

> "This context may or may not be relevant to your tasks."  
> （このコンテキストはタスクに関連する場合とそうでない場合があります）

つまりCLAUDE.mdに書いた内容は「絶対に従うべきコマンド」ではなく、「参考にすべきコンテキスト」として扱われています。Claudeは毎回、その指示が現在のタスクに関連するかどうかを自分で判断しているわけです。

参考：[Claude Code Best Practices - DataCamp](https://www.datacamp.com/tutorial/claude-code-best-practices)

### 公式ドキュメントはプロジェクト並行処理を推奨している

さらに調べると、Claude Codeの公式ドキュメント（Best Practices）では、プロジェクトの並行処理は明確に**推奨されている機能**であることがわかりました。

> "Run multiple Claude sessions in parallel to speed up development, run isolated experiments, or start complex workflows."

つまり私がCLAUDE.mdで「並行して進めないこと」と書いた制約は、Claude Code本来の設計思想と真っ向から衝突していた可能性があります。衝突する制約だからこそ、毎回「本当にいいですか？」と確認してきていたのではないかと考えています。

参考：[Claude Code Best Practices - 公式ドキュメント](https://code.claude.com/docs/en/best-practices)

---

## 気づいた判断軸

ここで一度立ち止まって考えました。

**そもそもこの制約、本当に必要だったのか？**

選択肢は2つあったと思っています。

### ① プロジェクトの並行OKにする

Claudeが並行処理を得意としているなら、それを制限していたのは自分の思い込みだったかもしれません。個人開発でコンフリクトのリスクが低い場面なら、並行処理を許可したほうがむしろ開発が速く進む可能性があります。

### ② 完全禁止にする

禁止するなら中途半端な書き方が一番よくないです。たとえば以下のように、強い文言で明示する必要があったと考えています。

```
## 絶対に守ること
- タスクは必ず1つずつ順番に進めること
- 複数タスクの同時実行は禁止
- 前のタスクが完了するまで次に進まないこと
```

ただし**この書き方でも確認文言が消えるかどうかは、まだ試していません**。HumanLayerの分析が正しければ、内部リマインダーがある以上、Claudeは依然として「この指示は今のタスクに本当に関係あるか？」を判断し続けるはずで、根本解決にならない可能性もあります。これは今後検証する予定です。

今回の失敗は、禁止したいのか許可したいのかが自分の中で曖昧なまま、なんとなく制約を書いてしまったことでした。

**「なんとなく書いた制約」が一番機能しない。**

これがこの経験から得た一番の気づきです。CLAUDE.mdは書けば従ってもらえるファイルではなく、Claudeが毎回「関連するか？」を判断した上で参照するファイルです。設計意図が明確でないと、判断が曖昧になります。

---

## 現状の構成と運用

この経験をふまえて、CLAUDE.mdに書くものと書かないものを整理しました。

### CLAUDE.mdに書くもの

* 技術スタック（Flutter、flutter\_gemma、Gemma 3 1Bなど）
* ディレクトリ構成と各フォルダの役割
* 絶対に守ってほしいルール（強い文言で、1行1ルール）
* 開発環境の前提条件

### CLAUDE.mdに書かないもの

* 曖昧な制約や「できればこうしてほしい」レベルの希望
* タスクの進め方の細かい手順（これはTASKS.mdの役割）

### TASKS.mdの役割

タスクの進行管理はTASKS.mdに完全に分離しました。6つのフェーズに分けて、各フェーズのタスクと進捗状態を管理しています。CLAUDE.mdはプロジェクトの「憲法」、TASKS.mdは「作業指示書」という位置づけです。

この分離によって、CLAUDE.mdがすっきりし、Claudeが迷う場面が減りました。

---

## まとめ

問題の根本はまだ解決していません。強い文言に変えれば確認文言が消えるのかも、まだ試していない段階です。ただ、この経験を通じて明確になったことがあります。

* CLAUDE.mdの指示は全て従われるわけではない。Claudeが毎回「関連するか？」を判断している
* Claude Codeの設計思想と衝突する制約ほど、確認や警告が出やすい
* 曖昧な制約を書くくらいなら、許可するか完全禁止にするかを明確に決める
* 役割が違うファイルは分ける（CLAUDE.md＝ルール、TASKS.md＝タスク管理）

同じような経験をした方がいれば、ぜひコメントで教えてください。
