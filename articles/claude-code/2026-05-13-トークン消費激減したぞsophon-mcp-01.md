## はじめに

Claude Codeを使い始めて、こんな経験はないでしょうか。

* セッション途中でコンテキストが溢れて、AIが「さっきの話」を忘れる
* `package-lock.json` を読ませた瞬間、残りコンテキストが壊滅的になる
* 「全部読んでから考えて」と指示したら、トークンが爆発してセッション終了

これらはすべて、**コンテキストウィンドウの使い方**の問題です。

今回ご紹介する **Sophon MCP**（ソフォン）は、AIに渡すトークン量を最小化することに特化したMCPサーバーです。実際にこのツールを使ったところ、要件定義・設計・実装をひとつのセッションで完走できました。その体験をお伝えします。

---

## Sophon MCPとは

> ギリシャ語 *σοφόν*「賢明なもの」より。

Claude CodeなどのAIコーディングツールでは、ファイルを読む際にデフォルトでファイル全体を取得します。小さなファイルなら問題ありませんが、`package-lock.json`（筆者計測で **88,759トークン**）のような巨大ファイルが存在するプロジェクトでは、一回の読み取りだけでコンテキストの大部分が消滅します。

Sophonは「**必要な情報だけを必要なタイミングで渡す**」という思想のもと設計されたMCPサーバーです。npmパッケージとして公開されており、設定はたった数行です。

```
{
  "mcpServers": {
    "sophon": {
      "command": "npx",
      "args": ["sophon-mcp", "--root", "/path/to/your/project"]
    }
  }
}
```

## 何ができるのか

Sophonは大きく6つのカテゴリのツールを提供します。

### 1. 段階的ファイル読み取り

ファイルを「全読み」する代わりに、まず構造だけを取得し、必要な部分だけを追加取得します。

| やりたいこと | 使うツール | 削減効果 |
| --- | --- | --- |
| Markdownの概要把握 | `read_markdown_toc` → `read_markdown_section` | **最大74.7%削減** |
| コードの関数構造確認 | `read_code_skeleton` → `read_code_body` | **最大41.7%削減** |
| JSONのキー構造確認 | `read_json_yaml_keys` → `read_json_yaml_value` | **最大92.6%削減** |
| キーワード検索 | `search_file` | **最大96.4%削減** |

たとえばコードを読む場合、まず `read_code_skeleton` でこのような一覧が返ってきます。

```
[
  {"id":"func:21-25","type":"function","name":"formatDate","signature":"export function formatDate(date: Date, locale: string): string"},
  {"id":"class:70-116","type":"class","name":"UserService","signature":"export class UserService"},
  {"id":"method:78-80","type":"method","name":"findById","signature":"findById(id: string): User | undefined"}
]
```

AIはこのシグネチャ一覧を見て「`findById` だけ詳しく読みたい」と判断し、`read_code_body` で該当部分だけを取得します。ファイル全体を読む必要がありません。

### 2. テキスト圧縮

`compress_text` は、Markdownのノイズ・コードコメント・重複行・余分な空白などを除去します。特に `trim_tool_response: true` オプションは、`null` や空配列だらけのAPIレスポンスを圧縮するのに効果的で、実測で **75.7%削減** を達成しています。

### 3. Webページ取得

`fetch_webpage` はWebページをMarkdownに変換し、圧縮して返します。ドキュメントを参照しながらコーディングする際に、生のHTMLをコンテキストに流し込まずに済みます。

### 4. プロジェクト横断の記憶管理（`memory_*`）

設計上の決定事項や注意事項をSQLiteに永続保存できます。

```
「このプロジェクトはTypeScript strict mode」
「DBはPostgreSQL 15、マイグレーションはPrisma」
「認証はJWT、有効期限は24時間」
```

こういった情報を `memory_save` で保存しておけば、セッションが切り替わっても `memory_list` で即座に復元できます。READMEを毎回全文読み直す必要がなくなります。

### 5. タスク管理（`task_*`）

作業タスクをSQLiteで永続管理します。親子階層にも対応しており、長期にわたる実装作業でも進捗を維持できます。セッションをまたいだ作業でも `task_list` で前回の続きから再開できます。

### 6. セマンティック検索（`semantic_search`）

キーワードが分からなくても、「エラーハンドリングの処理」「認証チェックに関係する部分」のような意味的なクエリでコードを検索できます。内部でClaude CLIをサブプロセス呼び出しするため、外部APIキーは不要です。

## 実際のワークフロー：ワンセッション完走体験

以下は、筆者が実際に体験した流れです。Next.jsプロジェクトにSophonを導入し、新機能の実装を行いました。

### セッション開始時

```
Claude: task_list を確認中...（前回のタスクなし）
Claude: memory_list を確認中...（前回の記憶なし）
Claude: read_directory_tree でプロジェクト構造を把握中...
Claude: read_json_yaml_keys で package.json の依存関係を確認中...
```

この段階で消費したトークンは約 **804トークン**。もし標準ツールで同じことをしていたら **1,588トークン** 消費していたはずです（**約49%削減**）。

### 実装フェーズ

```
Claude: read_code_skeleton で既存のUserServiceを確認中...
Claude: search_file で「async」を含む処理を検索中...（96.4%削減）
Claude: read_code_body で findById メソッドのみ取得中...
```

関係のないコードは一切コンテキストに入りません。AIの「注意」が実装対象だけに集中します。

### セッション終盤（実装完了後）

```
Claude: memory_save → 「新機能の設計決定：楽観的ロックを使用」を保存
Claude: task_update → 実装タスクを「done」に更新
```

次のセッションでは `memory_list` と `task_list` を呼ぶだけで、今日の作業の全コンテキストが復元されます。

## 数字で見るコスト削減効果

実際に計測された数値をまとめます。

| テストケース | 標準 | Sophon | 削減率 |
| --- | --- | --- | --- |
| `package-lock.json` キー取得 | 88,759t | 6,573t | **92.6%** |
| キーワード検索（1件ヒット） | 836t | 30t | **96.4%** |
| README 部分参照 | 360t | 115t | **68.1%** |
| プロジェクト初期調査（4ファイル） | 1,588t | 804t | **49.4%** |

Claude Sonnet 4.6 API（$3.00/百万トークン）換算での試算：

* `package-lock.json` を1回読む場合：標準 **$0.266** → Sophon **$0.020**（**$0.246節約**）
* これが10回/日なら月次で **$73.8の節約**

API利用者だけでなく、Proプランの月次上限を気にしながら使っている方にとっても、コンテキストを節約できることは直接「セッションの持ち」に影響します。

---

## 注意点：効果が薄いケース

すべてのケースで効果があるわけではありません。

* **200トークン以下の小ファイル**：構造化レスポンスのオーバーヘッドが削減分を上回ることがある（v1.1.1より自動フォールバック対応済み）
* **1ファイルだけの小さなgit diff**：statサマリのオーバーヘッドにより逆効果になる場合がある
* **ファイル全体を必ず読む必要がある処理**：段階的取得のメリットがない

## インストールと設定

```
# Claude Code の設定ファイルに追加するだけ
# ~/.claude/settings.json または プロジェクトの .claude/settings.json
```

```
{
  "mcpServers": {
    "sophon": {
      "command": "npx",
      "args": ["sophon-mcp", "--root", "/path/to/your/project"]
    }
  }
}
```

`--root` で指定したディレクトリ外へのアクセスはパストラバーサル対策として禁止されており、セキュリティ面も考慮されています。プロジェクトのデータは `{project_root}/.sophon/sophon.db`（SQLite）に保存されます。

## まとめ

Sophon MCPが解決する問題は明確です。

**「AIコーディングにおけるトークンの無駄遣い」**

コンテキストウィンドウは有限です。そこに何を入れるかは、AIの出力品質に直結します。Sophonを使うことで：

* 大規模ファイルの参照コストを劇的に削減できる
* セッションをまたいでも設計決定・タスク状態を維持できる
* Proプランのコンテキスト制限内で「計画→実装」をワンセッション完走できる

AIコーディングの効率を上げたい方に、ぜひ試していただきたいツールです。
