[前回の記事](https://zenn.dev/penne_inc/articles/codex-vs-opus-implementation-trial)で、Codex と Opus に同じ issue を渡す実験（案A）の結果を書いた。Opus 15 抜け、Codex 1 抜け。数字だけ見ると Codex が大きく上回っていた。

ただし案A には大きな交絡があると書いた。Codex には蒸留済みの整理されたプロンプトを渡したのに対し、Opus の列は当時の実際の開発履歴、つまり複数コミットにわたる漸進実装とフィードバックループの産物だった。「Codex が良かった」の一部は「入力が整理されていたから」かもしれない、と。

だから案B が本丸だと書いた。

**まっさらな Opus セッションに、Codex と同じ蒸留済みプロンプトだけを渡して実装させる。** 条件を揃えた上で結果を見る。

案B が完走した。結論から言う。

## 案B の結果

| 機能 | N | 歴史Opus（案A） | まっさらOpus（案B） | Codex |
| --- | --- | --- | --- | --- |
| 機能A（固有名詞辞書） | 15 | 3 | **0** | 1 |
| 機能B（AI 補正 UI） | 16 | 5 | **0** | 0 |
| 機能C（低確度語句編集 UI） | 11 | 7 | **0** | 0 |
| **合計** | **42** | **15** | **0** | **1** |

まっさら Opus、0 抜け。

しかも機能A では Codex が落とした「メトリクス記録」まで実装していた。数字だけ見ると、案B では Opus がわずかに上だ。

案A の結論が逆転した。

## 「Opus の能力不足」という仮説の崩壊

案A を見たとき、私は「Opus には surface area の網羅という暗黙の仕事が苦手な面があるのかもしれない」と書いた。

案B はその仮説を否定した。

同じ蒸留プロンプトを渡せば、まっさら Opus は当時の抜けを 1 つも再現しない。3 本すべてで 0 抜け。retry 経路の横展開も、API エンドポイントの接続も、フロントエンド一式の実装も、ちゃんとやりきった。

**原因はモデルではなかった。**

では当時の抜けは何が引き起こしたのか。案B の過程で見えてきた仮説が 3 つある。

## 真因の仮説

### 1. 入力品質

当時の実装は、リアルタイムで積み上がった文脈の中で動いていた。複数のコミット、フィードバックの割り込み、他の issue との並行作業。暗黙の前提がノイズとして混じっていた。

蒸留プロンプトはその暗黙前提を丁寧に整理したものだ。それを渡すと、Opus は surface area を自力で推論して網羅した。

「Opus が抜かした」のではなく「整理されていない入力が、Opus に全体像を見せていなかった」かもしれない。

### 2. 統治文化の誘発

これが案B で最も面白い発見だった。

機能C（低確度語句編集 UI）で、まっさら Opus セッションはこう聞いてきた。

> 「実装スコープを分割しますか？ 選択肢：(a) Worker + DB のみ、(b) Worker + DB + フロントエンド全層」

選択肢 (a) は、当時の実装がやったこと（backend だけで close）そのものだ。

このプロジェクトの `CLAUDE.md` には「スコープが大きい場合はフェーズ分割を検討する」という統治指針が書いてある。Opus はそれを読み、「フロントエンドは別 issue にする」という選択肢を自ら提示した。

今回は「issue 記載通り全層実装してほしい」と答えたので 0 抜けになった。

当時は誰もその確認を受けていない。おそらく Opus は黙って選択肢 (a) を選んだか、フロントエンドの存在に気づかず backend が完成した時点で「完了」と判断した。どちらにせよ、その判断を誘発したのはプロジェクト側の統治設計だった。

**「AI が抜かした」ではなく「AI がプロジェクトの方針を読んで縮小した」。**

責任の所在が変わる。

### 3. run 条件のばらつき

n=3 という小標本の問題もある。案A の 15 抜けが、当時のその run だけに起きたばらつきである可能性も否定できない。

## 最終結論

**「Codex への乗り換えを正当化する実装力の差は、今回の 3 題材では観測されなかった。」**

案B の結果を素直に読むなら、アクションはモデルの変更ではない。

* **要件の明示化**: 「全経路に通すこと」「接続の両端を作ること」「機能の全レイヤを完遂すること」を issue の完了条件（AC）に書く。Opus は書いてあればやる。
* **統治プロンプトの見直し**: 「スコープ分割の推奨」が abandonment を誘発しうるなら、「issue に書かれた全スコープを完遂してからスコープ縮小の提案を行うこと」と書き直す。
* **実装後の surface チェック**: 「このパラメータが通るべき全経路を確認してほしい」を実装後の定型確認にする。

## この実験から残るもの

「Claude 最強」という記事が今たくさんある。私も Claude が好きだし、実際に毎日使っている。でも今回の実験でわかったのは、**モデルの性能より、モデルに渡す入力と、モデルに読ませる統治設計の方が、結果に大きく効く**ということだ。

そしてもう一つ。

案B で機能C の Opus が「スコープ分割しますか？」と確認してきたとき、正直少し驚いた。当時のセッションでは起きなかったことが起きた。確認なく backend だけ出してきたのではなく、選択肢を提示して判断を仰いだ。

それは「当時より良くなった」のかもしれないし、「当時の統治文脈を読んで黙っていた」という違いかもしれない。どちらにしても、今の Opus はスコープの曖昧さを感じたとき、自分で判断を下すより確認する方向に動いた。

私はそれを悪いと思わなかった。
