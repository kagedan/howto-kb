CI に AI エージェントを入れる話は、だいたい prompt から始まる。

どう指示すれば賢く直してくれるか。どういう role を与えるか。失敗したらどう言い換えるか。

prompt は大事だ。けれど CI の中で agent を走らせるなら、最初に見るべき場所はそこではない。

どの token で動くのか。どの permission を持つのか。誰に課金されるのか。何 credit で止まるのか。あとから誰が session log を読めるのか。

ここを曖昧にしたまま prompt だけ磨くと、賢い bot を作ったつもりで、止まり方の分からない workflow を作ることになる。

## PAT が消えても、リスクは消えない

GitHub Actions 上の Copilot CLI が、長期の Personal Access Token なしで動くようになった。Actions にもともとある `GITHUB_TOKEN` で認証できる。

これはかなり良い変化だと思う。

CI 用に PAT を発行し、secret に入れ、期限や権限を管理し、退職者や bot user の棚卸しで悩む。こういう運用は、放っておくとすぐ薄暗くなる。`GITHUB_TOKEN` に寄せられるなら、入口のリスクは確実に下がる。

ただし、ここで「secret を置かないから安全」と言い切ると、読み違える。

PAT が不要になるというのは、agent のリスクが消えるという意味ではない。リスクの置き場所が、個人 token から workflow の実行面へ移るという意味である。

Copilot CLI を Actions で動かすには、workflow 側に `copilot-requests: write` のような permission を書く必要がある。さらに organization billing policy、cost center、usage dashboard、session limit といった話が絡んでくる。

つまり、AI agent が CI の正式な実行主体になり始めている。

だったら prompt は、その上に乗る設定の一部にすぎない。先に設計するのは、workflow が agent に何を許すかである。

## `GITHUB_TOKEN` は便利なだけでなく、仕様書になる

CI で何かを動かすとき、token はただの認証材料ではない。

それは「この job は何をしてよいか」を表す仕様書でもある。少なくとも、AI agent に対してはそう読んだほうがいい。

人間がローカルで CLI を叩くなら、まだ曖昧さが残る。本人の terminal、本人の判断、本人の課金、本人の責任。もちろん危険はあるが、拍は人間側にある。

CI は違う。push、schedule、pull request、manual dispatch。きっかけが来たら、workflow は走る。眠くならないし、遠慮もしない。だから `GITHUB_TOKEN` で動く agent は、便利な自動化というより、repo の中に置かれた実行者として扱ったほうがいい。

ここで雑に broad permission を渡すと、prompt の丁寧さでは回収できない。

`copilot-requests: write` をどの job に付けるのか。全部の workflow に入れるのか。特定の branch だけか。pull request from fork ではどうするのか。manual approval を挟むのか。write 権限を持つ job と test だけの job を分けるのか。

これは prompt engineering ではない。repo policy である。

自分なら、最初は agent 専用 job に閉じ込める。trigger も絞る。permission も job 単位で書く。成果物は patch か PR までにして、merge は別の review path に乗せる。

大げさに聞こえるかもしれないが、実際には逆だ。最初に小さく閉じるほうが、チームで試しやすいし、壊れたときにも読みやすい。

## 課金は「個人の使いすぎ」ではなくなる

billing も避けて通れない。

GitHub の説明では、organization に直接請求される場合、user-level budget は適用されない。ここはかなり重要である。

チャット型の AI なら、「誰が使いすぎたか」で見られる場面が多かった。個人の subscription、個人の quota、個人の使い方。組織契約でも管理はあるが、感覚としては人に紐づきやすい。

CI 上の agent は違う。

ある repository の workflow が走る。agent が model call をする。subagent や background work も session に入る。失敗して retry する。すると、コストは個人の遊びではなく、repo や organization の実行コストになる。

感覚としては、CI minutes に近い。

誰かが重い test を増やしたら、チーム全体の CI 時間が伸びる。cache の設計が悪ければ、毎回同じ dependency を落とす。timeout がなければ、落ちた job が長く居座る。

AI agent も同じ目で見たほうがいい。誰が悪いかではなく、どの workflow がどの条件で、どれくらい走るのかを見る。

cost center を分ける。usage dashboard を見る。repo ごとに予算を持たせる。実験用 workflow と本番運用 workflow を分ける。

地味だが、ここを飛ばすと後で揉める。

## `--max-ai-credits` は timeout と同じ種類の制御である

Copilot CLI / SDK には session ごとの AI credit limit を設定できる。noninteractive run なら `--max-ai-credits` のような形で上限を書ける。

これは単なる節約機能ではない。かなり実務的な安全装置である。

CI agent における timeout と同じ種類の制御だと思う。

普通の CI では、timeout を置く。test が無限に終わらない、network が詰まる、build が hang する。そういうときに job を止めるためだ。誰も timeout を「開発者を信用していないから」とは言わない。自動実行には止まる条件が必要だから置く。それだけの話である。

AI credit limit も同じである。

agent は model call をする。tool を呼ぶ。subagent を使うかもしれない。background compaction のような作業も session の中に入る。人間から見ると「一回頼んだだけ」でも、内部では複数の仕事が動く。

だから、予算は「1 response」ではなく「1 workflow run」に対して置くべきだ。

ただし、credit limit は完全なハード停止ではない。上限に達しても、実行中の response は終わってから wrap up する。つまり soft cap として読む必要がある。

ここも timeout に似ている。SIGTERM を送っても即座にきれいに終わるとは限らない。cleanup の猶予、artifact の保存、失敗時の通知が必要になる。

AI agent でも同じで、上限に達したら何を残すかを決めたほうがいい。

patch を残すのか。ログだけ残すのか。途中の plan を artifact にするのか。人間 review queue に投げるのか。

「止める」だけでは運用にならない。止まったあと、人間が読める形で残すところまでが CI 設計である。

## session log はチャット履歴ではない

agent の session streaming や usage record を、便利な履歴機能としてだけ見るのは少し危ない。

CI に agent を入れるなら、これは監査面になる。

何を prompt されたのか。どう返したのか。どの tool call をしたのか。どこで止まったのか。どの workflow run と紐づくのか。事故が起きたとき、ここを読めないと、最終的な diff だけを見て原因を推測することになる。

これはかなりつらい。

人間の作業でも、CI log、commit、PR discussion、review comment があるから後から読める。AI agent だけ「なんとなくこういう差分が出ました」では困る。困るというか、運用に乗らない。

特に CI 上の agent は、ローカルの会話よりも実行環境に近い。repository の状態、token、permission、billing、workflow trigger と結びつく。ならば session log も、会話の思い出ではなく、build log や audit log に近いものとして扱ったほうがいい。

小さいチームで、最初から SIEM 連携までやる必要はない。

ただ、最低限、workflow run と session の対応、prompt、tool call、credit 使用量、失敗理由、生成された artifact は追えるようにしたい。

あとから読めない自動化は、成功している間だけ気持ちいい。失敗した瞬間に、ただの不思議な箱になる。

## 最小構成はこれでいい

CI に AI agent を入れるとき、最初から大きな governance を作る必要はない。

むしろ最初は、かなり小さくていい。小さく始めるほうが、あとで増やす判断もしやすい。

* agent 専用 job に分ける
* trigger を絞る
* job 単位で最小 permission を書く
* `copilot-requests: write` を必要な場所にだけ渡す
* organization billing と cost center を確認する
* 1 workflow run ごとに AI credit cap を置く
* timeout と failure artifact を設定する
* session log / usage record を workflow run と紐づける
* 出力は PR または patch に止め、人間 review を通す

これだけでも、かなり違う。

これで、「AI agent を CI で走らせています」という状態から、「agent がどの権限で、どの予算内で、どのログを残して走るかを repo が知っている」状態へ移る。

ここまで来て初めて、prompt の話をしてもいい。

どんな指示を与えるか。どの粒度で issue を渡すか。どのファイルを触らせるか。どんな出力形式にするか。

それらは重要だ。ただし、agent が走る床が抜けていないことが前提である。床が抜けているなら、prompt はうまく効きすぎることすらある。

## prompt は最後に効く

AI エージェントを CI に入れるとは、賢い bot を workflow に呼ぶことではない。

勝手に走る作業を、予算・権限・停止条件・ログを持った実行単位にすることだ。

PAT が不要になったのは良いニュースだ。session limit が入ったのも良い。agent session を後から追える方向へ進んでいるのも、かなり健全だと思う。

ただ、それらは「もう安全です」という印ではない。

むしろ逆で、AI agent が CI の正式な部品になり始めたからこそ、普通の CI と同じように control plane を持たせる段階に来た、ということだ。

短い prompt より、短く止まれる workflow。

賢い指示より、読めるログ。

強い model より、狭い permission。

CI に agent を入れるなら、最初に書くべきものは prompt ではない。budget、permission、log。そこを書いてから、ようやく agent はチームの道具になる。
