## はじめに

前回の記事「[「AI駆動開発チームの作り方・育て方」を読んで](https://zenn.dev/spectee/articles/ai-driven-team-book-review)」の続きです。

1ヶ月間、手を挙げて社内のClaude Code導入推進リーダーを務めました。  
ミッションは「Claude Codeを活用したエンジニアの開発速度を向上させるプロセスの構築」です。

それまで社内ではDevinやCopilotを活用したAI支援開発を行っていましたが、コーディングサポートにとどまっていました。Claude Codeの導入により、開発プロセス全体をAI駆動に転換することを目指しています。

目標は「Claude Codeを使っていこうという社内の雰囲気を作ること」。  
実際に触ってみる機会を増やして欲しかったので、 **ハンズオンやSlackでのTips配信を軸に発進** しました。

AIツールの導入を検討している組織に向けて、推進の戦略や直面した課題をまとめます。自チームでの導入効果は開発メトリクスで計測しているので、そちらも紹介します。

## 推進の戦略

### パイロット導入で進めた理由

推進チームは正式メンバー3名 + サポート1名の構成でした。

この人数では、**社内の全チームを同時かつ個別にサポートするのは事実上不可能です。** 推進PJTのメンバーもメイン業務を抱えているため、リソースの使いどころが重要です。

また個人的には、ある程度**トップダウンでルールを統一した方が長期的には効果を挙げやすいのではと感じて**いました。ただ、**推進チームには「この方向で行きましょう」と全社的な方針を決める権限はありませんでしたし、チームごとに状況も異なります**。  
準備期間も限られる中で「さぁ一気に導入です」というわけにはいきませんでした。

「本来なら全社統一ルールを作りたいが、時間が足りない」「各チームの状況が違いすぎて、一律のルールでは回らない」「失敗するリスクを冒して全社導入するか、成功事例を作って広げるか」という葛藤がありました。

結局、まず**自チームでパイロット導入して成功事例を作り、その実績をもとに他チームへ広げる戦略を選択**しました。自チームは新規プロジェクトで関心も高く、成功しやすい条件が揃っていたのでちょうど良かったです。  
これは会社・チームによって取るべき戦略が変わってくるので、慎重に検討することをおすすめします。

実は、元々は**全社的にツールを統一してスピーディに移行しようというロードマップを組んでいた**のですが、こうした進め方が社内の空気感にマッチしていないことを肌で感じたため、大幅に軌道修正を図りました。

### 「使う雰囲気」を作る

**「あ、これ使えそう」と思ってもらう瞬間を増やす**ことを優先しました。そのため、できる限りClaude Codeや仕様駆動開発フレームワークの設定を触って試してもらえるようなハンズオン・ワークショップ形式のセッションを実施することを重視しました。

**やったこと（具体的な活動）**:

* ドキュメント作成（使い方、スクラム開発での活用パターン、OpenSpec導入ガイド）
* ハンズオンセッション開催（基本操作、スクラム開発適用、OpenSpec導入）
* Slackでの「Claude Code Tips」配信
* テンプレートリポジトリ（`.claude/` 配下の設定一式）の整備と各リポジトリへの自動配布
* フィードバック収集→コンテンツ改善のサイクル

ちなみに、テンプレートリポジトリはGitHub Actionsで各リポジトリに自動配布する仕組みにしました。手動での展開はメンテナンスが追いつかないためです。

幸いにも「開発が変わりそう、やってみたい」という熱量は割と初期から感じられたので、「実際に手を動かす機会があれば自然と活用が進むだろう」と楽観的に捉えられたのは非常にありがたかったです。

## 自チームでの導入効果（4週間の計測データ）

自チームでClaude Code + 仕様駆動開発（SDD）を導入し、4週間の開発メトリクスを計測しました。Week 0（導入前）とWeek 1〜3（導入後）の比較です。

### チームの特徴

防災情報の配信・分析基盤を開発する4名+マネージャーのチームです。メンバーは新人2名と若手1名で、新規立ち上げチームということもあり新しい技術やツールに対して好意的な環境でした。立ち上げから約2ヶ月が経過し、**開発フロー自体は確立されつつありましたが、自動テストやCI/CDの整備は十分ではありませんでした。** 率直に言ってこれは反省すべき点です。ただ、**この状態がSDD導入の効果測定としては適切なタイミングだった**とも考えています。

また、新規立ち上げのため既存のNotionなどにドキュメント資産があまりなく、GitHub Projectsでタスク管理を含めてすべて集約できたことも移行のスムーズさに寄与しています。

以下の特徴が、短期間で効果が出やすかった要因です。

* **新規プロジェクト・チーム**: 技術的負債が少なく、新しい開発フローへの抵抗感が低かった
* **小規模チーム**: 意思決定が速く、全員が同じフローで動ける
* **推進リーダーがチーム内にいた**: 自分自身がチームメンバーなので、導入のフィードバックループが即座に回った
* **テスト・CI/CDに伸びしろがあった**: SDD導入の効果が数値に現れやすかった

### 定量的な変化（Week 0 → Week 3）

| 指標 | Week 0（導入前） | Week 3（導入後） | 変化 |
| --- | --- | --- | --- |
| Issueクローズ数 | 8 | 27 | **3.4倍** |
| 平均クローズ日数 | 14.9日 | 4.9日 | **67%短縮** |
| PRマージ数 | 7 | 18 | **2.6倍** |
| 平均PRサイズ | 2,136行 | 987行 | **54%縮小** |
| mainコミット数 | 3 | 32 | **10.7倍** ※ |
| テスト関数（累積） | 11 | 111 | **10.1倍** |

※ mainコミット数の増加は、PRサイズの縮小（54%減）と合わせて「小さく頻繁にデリバリーするスタイルに変わった」ことを示しており、生産量が10.7倍になったことを示すわけではありません。チームごとに定義された指標を定量的・定期的に計測することが重要だと思います。

特にテストの変化が顕著でした。導入前は11関数しかなかったテストが、SDD導入後3週間で100関数追加されています。仕様書にテスト要件を含めるフローにしたことで、「まずテストを書く」が自然な開発プロセスの一部になりました。

PRマージ数が2.6倍、mainコミット数が10.7倍に増加しているので、テストとCI/CDの整備がなければこのペースでのマージ・デプロイは怖くてできなかったと思います。**「自動テストが通っている」という安心感があるからこそ、速いサイクルで開発が回せるようになりました。**

### 定性的な変化

| 観点 | SDD導入前 | SDD導入後 |
| --- | --- | --- |
| 仕様管理 | `docs/` に仕様書を個別配置 | OpenSpecで仕様→タスク→実装→アーカイブのサイクル |
| テスト | テストなしの実装PRが主流 | テストファーストのアプローチ定着、E2Eテスト環境整備 |
| Issue管理 | 手動管理 | Claude Code[スキル](https://code.claude.com/docs/ja/skills)による構造化 |
| コード品質 | ルール散在 | [`CLAUDE.md`](https://code.claude.com/docs/ja/memory) + `rules/` で体系的に整備 |
| CI/CD | 未整備 | 関数別テスト実行・デプロイワークフロー追加 |

### 短期間で効果が出た理由の考察

数値だけ見ると劇的な改善に見えますが、いくつかの補足が必要です。

* **ツール導入だけの効果ではない**: チーム立ち上げから2ヶ月が経ち、開発フロー自体は安定していたものの、テスト・CI/CDの整備はこれからでした。SDD導入と並行してこれらを整備したため、改善のすべてがClaude Code / SDDの効果とは言い切れません。ただし、SDDを入れたことでテスト要件が仕様書に組み込まれ、テスト整備が加速したのは事実です
* **Week 0のベースラインが低い**: テスト11関数、mainコミット3件という状態からのスタートなので、倍率が大きく出やすい
* **Week 2にマージ停滞があった**: マージ率20%と低迷し、Week 3で滞留PRを一掃（マージ率106%）。なぜ停滞したかというと、レビュー体制が追いついていなかったためです。「新しくトライしている仕様駆動のフローとして正しいか」「AIによるレビューはどこまで信頼して良いのか」という点が不明瞭で混乱した時期がありました。
  + 実装直後にAIによるファーストレビューを必須化・自動化してから人間がセルフチェック（2ndチェック）→レビュアーに提出するという新たなレビューモデルを整備しました。
  + 一見フローが肥大化して余計な時間がかかりそうですが、人間にしかできないポイントのレビューに集中できるようになってマージが進んだという感触を得ています。
* **PRサイズはまだ改善途上**: チームガイドライン推奨の500行はまだ超過している

SDD導入の効果を正確に切り分けるのは難しいですが、仕様書ベースの開発フローが定着したこと、テストを書く習慣（というか仕組み）が根付き始めたことは明確な変化です。

### 逆に難しいと思われる条件

自チームは前述の通り効果が出やすい条件が揃っていました。逆に、以下のような環境では導入のハードルが上がると感じています。

* **レガシーシステム**: コンテキストが膨大で、AIの提案精度が下がりやすい
* **仕様が曖昧**: SDDの前提となる「仕様を書く」こと自体が難しい
* **テストが不足している**: AIの提案を検証する手段がないため、効果を実感しにくい

ただし、難しい環境こそ「共通の設定やルールを育てるチャンス」でもあります。特にテスト不足はSDD導入をきっかけに大きく改善できる可能性があり、自チームでもそうでした。  
また、**仕様書が充実してくることでAIが生成するコードの精度も上がってくるので、初期に時間をかけてでもじっくり現行コードの仕様書を生成・レビューしてコンテキストに追加することは非常に重要**です。

## 失敗と学び

**おすすめ設定集の配布は成功したとは言えませんでした**。

VSCode拡張や `.claude/` 配下のテンプレート一式をGitHub Actionsで100以上のリポジトリに自動配布しましたが、PRのマージ率は数%程度にとどまっています。

**フィードバック**:

* 「情報量が多くて、どこから手をつけていいか分からない」
* 「自分のチームには関係ない設定も多い」
* 「既にチームの AI活用に積極的な人がおすすめ設定を作り込んでいる」
* 「チームごとに状況が違うので、テンプレートが役に立たない」

すでにClaude Codeを触りまくっている人にとっては当たり前の情報だし、これから慣れようという人には情報量が多すぎました。「ちょうど良い」コンテンツになるパターンは少なかったです。

限られた時間の中で、初動でルールを整備しきるのは難しかったです。感度の高い人からのフィードバックに応えようとするあまり、少しずつ慣れていく段階の人たちに十分なサポートができなかった反省もあります。

**効いたのはキュレーション情報**:

* Slackでの「Claude Code Tips」配信
* 各チームの導入事例を共有（なぜこの設定にしたのか含む）

**自前カスタマイズは「捨てる前提」で作る** 。

[Git worktree](https://git-scm.com/docs/git-worktree)を使った並行開発や、非公式[MCPサーバー](https://modelcontextprotocol.io/)の導入なども進めていました。MCPサーバーを利用した非公式スキルも複数実装しました。ところが、1ヶ月の間にClaude Code公式から[worktree対応](https://code.claude.com/docs/ja/common-workflows#git-worktrees-%E3%82%92%E4%BD%BF%E7%94%A8%E3%81%97%E3%81%A6%E4%B8%A6%E5%88%97-claude-code-%E3%82%BB%E3%83%83%E3%82%B7%E3%83%A7%E3%83%B3%E3%82%92%E5%AE%9F%E8%A1%8C%E3%81%99%E3%82%8B)、[Agent Teams](https://code.claude.com/docs/ja/agent-teams)、[Remote Control](https://code.claude.com/docs/ja/remote-control)などが次々と発表され、自前で作ったものと機能が被るケースが頻発しました。当然、公式実装の方が安定性・メンテナンス性ともに優れているので、そちらへの移行が何度も発生しました。

**どうしても急ぎで欲しい機能は自前で作るしかないが、そうでなければ公式・準公式の発表を待つ方が賢い** 。そして自前で作ったものも、 **公式が出たら気軽に捨てて乗り換える** 。このスタンスが重要です。AI駆動開発のエコシステムは進化が速すぎるので、自作ツールに固執するとかえって足かせになります。

**LLMにファイル・CLI操作を任せることへの考慮不足**  
Claude Codeを利用する上でのガードレールが不足していたという反省もありました。  
[本番DBを壊したくないのでClaude Codeのdeny/Hookを突破できるか本気で試してみた](https://qiita.com/dai_chi/items/8c351cf7812dd31b7767)などの記事にもあるように、適切な防御層を設けてもすり抜けは発生しうると前提で、それでも安全に運用できる環境を作っていくことが必要だと感じています（[実際に本番データが吹っ飛んだ話](https://alexeyondata.substack.com/p/how-i-dropped-our-production-database)も話題になりました）。

## 具体的な開発プロセス

ここまでは推進活動全体の振り返りでした。  
次に、自チームで実際にどのような開発プロセスを構築したかを紹介します。

### OpenSpecによる仕様駆動開発

仕様駆動開発のフレームワークとして[OpenSpec](https://github.com/Fission-AI/OpenSpec)を採用しています。軽量で取り扱いやすい点と、仕様がGitで管理される点が気に入っています。

OpenSpecのディレクトリ構成

```
openspec/
├── config.yaml        # スキーマ設定（ワークフロー定義）
├── schemas/           # カスタムスキーマ
├── specs/             # 現在の仕様（唯一の真実源）
└── changes/           # 変更提案ワークスペース
    └── [feature]/
        ├── proposal.md    # 変更提案（なぜやるか）
        ├── design.md      # 技術設計
        ├── specs/         # この変更に関する仕様
        └── tasks.md       # タスク一覧
```

`changes/` フォルダで変更ごとにワークスペースが分離されるので、「今何を実装中か」が分かりやすいです。完了したら `/opsx:archive` で整理できます。

自チームでは標準のspec-drivenフローにAIレビューステップを追加した `spec-driven-with-review` というカスタムスキーマを作成しています。これは次に紹介する2層レビューモデルと統合したものです。

### スクラム開発での活用

スクラムの各フェーズでClaude Codeが活躍します。自チームではIssueのクローズ速度が3.4倍になったと前述しましたが、それを支えているのがスクラムの各プロセスへの組み込みです。

**仕様を固める段階** では壁打ち相手として使えます：

仕様の壁打ち

```
claude "このユーザーストーリーの受け入れ条件を整理して。
抜け漏れがあれば指摘して"
```

**実装フェーズ** で威力を発揮：

テストファーストで進める流れ

```
claude "specs/feature-x.md を読んで、まずテストを書いて"
claude "テストが通るように実装して"
claude "リファクタリングして"
```

**レビュー** もサポートできます：

PRレビュー

```
claude "このPRの変更をレビューして。
セキュリティ上の問題があれば指摘して"
```

### 2層レビューモデル（AI 1st / Human 2nd）

レビューではさらに踏み込んで、AIとHumanで役割を分担する仕組みを導入しました。

|  | AI（1st Checker） | Human（2nd Checker） |
| --- | --- | --- |
| 担当範囲 | 正確性、セキュリティ、信頼性、パフォーマンス、可読性、テスト、ドキュメントの全7カテゴリ | 仕様適合性、アーキテクチャ、ビジネスロジック、運用準備状況 |
| 指摘分類 | MUST / RECOMMEND / NITS | 同左 |

AIが網羅的なチェックを担い、Humanは「仕様通りか」「設計として正しいか」に集中できます。**OpenSpecでは[カスタムスキーマという機能で開発フロー自体をカスタマイズできる](https://github.com/Fission-AI/OpenSpec/blob/main/docs/customization.md)ので、デフォルトフローの開発完了ステップ直後にAIレビューを組み込んで、仕様策定→実装→AIレビューを一気通貫で回しています。**

## 今後の展望

### ナレッジ共有の仕組み化

各チームでの活用が進むとフィードバックが溜まってきます。これをチーム間で連携して、ナレッジ共有がさらにはかどる環境を作りたいです。

特に **仕様駆動開発が浸透すると、仕様書の置き場所を社内で統一しないとコンテキストが散らばります。** スクラムマスターと連携しながら、最低限のルールを整備していく必要があります。

また、社内でClaude Codeを利用する際のセキュリティ設定（破壊的なコマンドのdenyリスト登録など）のガイドライン整備も重要な課題です。導入を検討している方は、[公式ドキュメントの権限設定](https://code.claude.com/docs/ja/security)を参考にチームのポリシーに合わせた設定をおすすめします。

### テストカバレッジの向上

テスト関数の数は増えましたが、カバレッジの計測・可視化はまだできていません。まずは `pytest --cov` などによるカバレッジ計測のCI組み込みでテストの量を可視化するところから始めます。

### 細く長く続ける体制へ

準備期間が短かったこともあり、前半は推進チームの稼働が大きくなりました。メイン業務に支障が出ない、無理なく続けられる体制に移行していく必要があります。

## さいごに

正直に言うと、最初からきれいに進められたわけではありません。**推進チームでの事前検証から全社アカウント配布までの期間が短く、当初想定していた「まず全体設計を固めてから展開する」というアプローチは早々に断念**しました。途中から **「まず動かしてみて、フィードバックを受けて軌道修正する」というアジャイルな進め方に舵を切った** 形です。

結果的にはこの判断が良かったと思っています。AI駆動開発や仕様駆動開発という概念自体、ここ1年ほどで急激に盛り上がってきた領域です。**ベストプラクティスがまだ確立されていない中で、完璧な計画を立ててから動くのは現実的ではありません。小さく試して、うまくいったことを横展開する。うまくいかなかったことは素直にやめる。** テンプレート配布がうまくいかなかったのも、キュレーション情報が効いたのも、やってみて初めて分かったことです。

自チームの計測データからは、SDD導入とテスト・CI/CD整備の相乗効果が確認できました。ただし、これはパイロット導入に適した条件が揃っていたからこそ出た数値でもあります。他チームへの展開にはまた別の工夫が必要です。

パイロット導入だけでは全社への浸透に限界がある感覚もありますが、トップからの後押しが加われば大きく加速するはずです。**ツールも概念も日々進化しています。「完成」を目指すのではなく、変化に適応し続ける姿勢**で、今後も推進を続けていきます。
