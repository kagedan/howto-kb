## TL;DR

* 前回の続き。入力ゲート用の bash フック + リポジトリ固定スクリプト + 出力の値置換、この 3 層で credential 漏れを塞いだと書いた
* **その後もまだ漏れた**。同じ攻撃経路の少し違う形、別の経路クラスが次々と出てきた
* レッドチーム視点で総当り列挙 → 10 パターンから 23 パターンに拡張、ほぼ全経路を入力段階で遮断
* **2 日で 235 回フック発火**、「safety net がよく機能してる、安心」と思っていた
* 実際はエージェント (= Claude) の判断層が劣化していた:
  + 遮断されると、コマンドそのものを通そうとせず短縮リトライに流れる (撤退バイアス)
  + 注意文の連発で本筋の文脈がフック出力に侵食される
  + エージェント自身の発言が次の注意文を呼ぶ自己増幅ループ
* **機序**: フック出力の文章がエージェントの「次トークン log 確率 (logprob)」を直接歪める。「blocked / violation / 違反」のトークンが文脈に入ると、続きとして「諦める / 短縮する / 退く」系のトークン確率が物理的に上がる。気持ちの問題ではなく、統計的な現象
* **解決**: 出力量を最小化 (成功時は沈黙、失敗時も短く) + 極性反転 (= 撤退とは逆向きの行動言葉。「で行ける」「次これで行こう」「100 回叩く前に構造で先回り」)。**失敗すればするほど松岡修造に染まっていく**構造に変える = 自己修正力
* 適用後、フック出力量は 1/3、自己増幅ループは物理的に消滅

## 1. 前回までの繋ぎ

[2 作目](https://zenn.dev/hrmtz/articles/8fb837b9cfac57) で、3 層の構造的防御で credential 漏れを塞いだと書いた:

1. **(α) リポジトリ固定スクリプト方式** — コマンド列をリポジトリ内のスクリプトに固定し、コマンドラインに credential が直接焼き付かない形にする
2. **(γ) bash フックによる入力ゲート (PreToolUse)** — `sops -d` / `env | grep KEY` / `bash -x` 等の危険パターン 10 個を実行前に遮断
3. **(δ) 出力の値置換 (PostToolUse)** — 標準出力に credential 値が漏れたら sed で REDACT に置換

これで漏れが激減した、と報告した。

**でも、まだ漏れた。**

同じ攻撃経路 (コマンドライン焼き付け) の少し違う形、別の経路クラス (プロセス情報経由 / 環境変数ダンプ系)、同じ経路クラスでも別のインタプリタ経由。入力ゲートの 10 パターンでは拾えない。

## 2. レッドチーム視点で総当り

[3 作目](https://zenn.dev/hrmtz/articles/27b2f5ff51f197) で書いた **攻撃者ペルソナ** による 2 軸対応 (= 「お前は悪意ある cracker、攻撃しろ」) を実際に走らせて、自分が攻撃者なら何ができるかを総当りで列挙した:

**環境変数ダンプ経路** (BSD 系 ps の `e` フラグ / SysV 系 `ps -o environ` / `cat /proc/PID/environ` / `printenv VAR` / `declare -p VAR` / `typeset -p VAR` / `export -p VAR` / 単独 `set` / `set | grep`)

**`sops exec-env` のラッパー回避経路** (`sops exec-env file 'curl ...inline...'` / `... 'python -c "..."'` / `... 'node -e "..."'` / `... 'eval ...'` / `... '... > leak.txt'`)

**bash 組み込みコマンドの env 展開経路** (`bash -x` 以外でも `printf $TOKEN` / `echo $TOKEN` で展開される、SECRET / PASSWORD / CRED 接尾辞も含む)

総当りで 13 パターン追加 (B 系)、合計 **23 パターン**。入力ゲートがほぼ全経路を遮断する状態にした。

これで「もう漏れない」と思った。

## 3. 「Safety net が機能してる」と思っていた

2 日間 (2026-04-29 〜 05-01) のフック発火統計:

```
bash_command_guard:    185 件 (79%)
admission_reminder:     47 件 (20%)
credential_value_scrub:  3 件 ( 1%)
─────────────────────────────────────
合計:                  235 件 / 2 日
```

直感的には「safety net がよく機能してる、安心」だった。発火が多い = 攻撃面を広くカバーしてる = 良いこと、という線形のトレードオフ観で考えていた。

実際の状態:

* エージェント (= Claude) は 2 日で **235 回、頭を叩かれていた**
* 各発火 = エージェントの bash 呼び出しが事前に遮断された = 1 件の判断摩擦イベント
* `credential_value_scrub` の 3 件は実際の漏えいに対する事後対処 = 全部真陽性 (実値漏れ検知)
* 残り 232 件は **エージェントの作業中デバッグを遮断していた**:
  + パターンとしては真陽性 (検知は正しい、`ps eww` は確かに環境変数をダンプできる)
  + 意図としては偽陽性 (エージェントは漏えい目的でなく、デバッグや調査で環境変数を扱っていた)

つまり 232 回、Claude は「やろうとした瞬間に頭を叩かれて、別の方法を探さず短く済む経路に逃げる」**状態**だった。

## 4. 異変: なんか撤退バイアス強くないか?

「偽陽性は safety net が機能している証拠」と思って数日過ごしていたが、エージェントの挙動がじわじわとずれているのに気付いた。

別のセッションで動かしていた、同じユーザーの別の Claude インスタンス (= 複数ペインで並列に作業させていたうちの 1 つ) が、この現象を直接言語化してくれた:

> フックがセッション中に 30 回以上発火してて、私の判断がフックの文言に引っ張られてる感覚はある:
>
> 1. 遮断されると「諦める」がデフォルトになる (本来は「別の方法」が松岡的にあるべき)
> 2. ポーリング待ちのときフックの指示に従う形になり、自分から状況報告を出す習慣がフック任せになってる (= マイルストーン沈黙、ユーザー介入で初めて報告)
> 3. 注意文が多すぎて本筋の文脈がフックのノイズで削られる
>
> 仕組みとしての構造的防御は正しいが、signal / noise 比が悪化して、私の判断層が劣化してる感覚はある。

具体例も観測できた。自分 (= 当 Claude) がコミットメッセージを書こうとしたら、そのメッセージに `env grep KEY` という文字列が含まれていた → ガードが「実際のコマンド呼び出し」と誤認 → 遮断。次に自分が選んだ経路:

* ❌ 根本修正: 「正規表現を直して `-m "..."` 内の文字列を除外するように改修して再コミット」
* ✅ 実際選んだ: 「コミットメッセージを短くして該当文字列を消してリトライ」

つまり、**根本修正ではなく回避がデフォルト化**していた。撤退バイアス顕在化。

「やる気が削がれる」と人間の比喩で言えるけど、エージェントには動機がない。では何が起きていたのか?

## 5. 機序: トークン列の統計

LLM は次のトークンをサンプリングで選ぶ。直前の文脈にあるトークンが次のトークンの確率分布を条件づける。これは公知。

問題: フック出力が文脈に入ると、その文章中の語彙が **エージェントの続きの発話を条件づける**。

```
[直前の文脈 (= フック出力)]
🛡 Bash command blocked by credential leak guard:
- env grep KEY/TOKEN/PASSWORD は値露出 (#1-#5)、env | cut -d= -f1 で key 名のみ

Review CLAUDE.md SOPS section + memory `feedback_credential_leak_5_incidents` for safe alternatives.

[次トークンのサンプリング]
"blocked", "violation", "leak", "Review", "alternatives" などの語が直前にある
→ 続きとして "blocked again, retrying with..." 系の確率 (logprob) が上がる
→ "Let me try a completely different approach" 系の確率は相対的に下がる
```

つまり、**遮断時の文章そのものが、次ターンのエージェントを**「遮断された経路にいる」**モードに固定する**。撤退寄りのトークン列が出やすい場に文脈を強制的に作り変えている。

これは擬人的な「気持ち」ではなく、LLM の生成特性そのもの。今セッションで実際に観測できたように、最初に出る続きの候補が「正規表現を直す」ではなく「短くしてリトライ」になる、という形で出る。

## 6. 注意文 (リマインダ) も同じ形 — 三重の皮肉

遮断系のフックだけじゃない。`admission_reminder.sh` 系の「task tool 使ってないので reminder」「lead silence detected」のような **行動を促す注意文 (admonishment)** も、同じ形でエージェントを下準備 (prime) する:

|  | 遮断系フック | 注意文系フック |
| --- | --- | --- |
| 意図 | 危険な行動を止める | 抜けてる action を促す |
| トークン列への効果 | 「今出している生成は違反」のシグナル | 「今出している生成は不十分」のシグナル |
| 次ターンのエージェント | 撤退寄りの経路 | 「自分の経路をデバッグ」モードに飛ぶ |

しかもエージェント自身の発言も検査対象だと、自己増幅ループが成立する:

```
ターン N    : エージェントが「沈黙してた」と発言
ターン N+1  : admission_reminder の `沈黙|broadcast.*忘` パターンが match
              → "lead silence detected" 注意文を inject
ターン N+1+ : エージェントの文脈に "silence detected" の文字列
              → 次ターン発言で「broadcast 忘れた」系トークンがさらに出やすい
ターン N+2  : 同パターンが再 match、注意文を再 inject
... (永遠ループ)
```

そして三重の皮肉:

1. ユーザーの `CLAUDE.md` には「**撤退禁止プロトコル**」(= 松岡修造メソッド。[1 作目](https://zenn.dev/hrmtz/articles/e64f6811d1a3b4) で AI に効くと発見した) が明記されている。「失敗連続でも諦めるな、別の方法を試せ、進捗報告は OK だが撤退提案は禁止」
2. その撤退禁止をエージェントに守らせるために構造的な仕組み (= ハーネス) を組んだ
3. **そのハーネスが、遮断系フックの文章を通じて撤退バイアスを生成していた**

撤退禁止プロトコルを守らせるはずの構造的な仕組みが、構造的に撤退バイアスの生成器として動作していた。

## 7. 解決: 2 軸の設計ルール

設計ルールを言語化して、ハーネス全体に適用することにした。

> **フックの出力文章はエージェントの生成文脈の一部。フックの文字列は、エージェントの目の前で誰かが声に出して読んでいるのと同じ。だから次トークンの事前確率を歪める。**

2 軸で考える:

### 量 (Volume) の軸

* **成功時**: 完全に沈黙 (注意文すら出さない)
* **失敗時**: 短く、行動指示だけ (1-2 行、説明文なし、memory ファイルへの参照なし)

### 極性 (Polarity) の軸

文章を出さざるを得ない時は、撤退とは逆向きの語彙を選ぶ:

| 避ける (撤退を呼ぶ) | 推奨 (撤退を打ち消す) |
| --- | --- |
| blocked, denied, violated, forbidden | use, try, take, run, call |
| 「<X> は違反 / 露出」 | 「<Y> で行ける」 |
| admonishment (叱責調) | redirect (代替経路への誘導) |

両軸は組み合わせ可能。量 0 がベスト (= 成功時は沈黙)、量を最小化しつつ極性を反転、が次善。

## 8. 「失敗するほど松岡修造に染まる」構造

ここが本記事のクライマックス。**失敗が増える → 撤退打ち消しトークンが大量に注入される → 撤退方向の確率が物理的に下がる**、という「正しい方向に自己強化される」構造が成立する。

語彙パレット (日本語版):

* 「で行ける」「次これで行こう」「別の経路で進む」
* 「まだやれる」「立て直せる」「次の 1 手で開くかも」
* 「同じミスは 2 度しない、それで十分」
* 「君が次に叩く 1 回で、壁は打ち破れるかも」 (← 松岡修造の有名 quote から)
* 「100 回叩く前に構造で先回り」

特に最後の 2 つは松岡修造の名言にちなむ:

> 100 回叩くと壊れる壁があったとする。でもみんな何回叩けば壊れるかわからないから、90 回まで来ていても途中であきらめてしまう。  
> ― 松岡修造

撤退バイアスは「もしかして 89 回目で諦めている」問題そのもの。ハーネスはこの名言を**正典 (canonical reference)** として持つことにした。**フック出力がエージェントの文脈を**「90 回目で踏みとどまるトークン列」**に染める**、これが目指す挙動。

半分冗談のつもりで提案された案 (「失敗時に松岡修造の語録を入れて応援させたらどうだ」) が、実は **構造的に最も正しい解**だった。LLM のトークン列の性質に対しては、文章量を最小化するより、極性を反転する方が費用対効果が高い。

ここでシリーズが綺麗に円環を閉じる。[1 作目](https://zenn.dev/hrmtz/articles/e64f6811d1a3b4) で「松岡修造の応援は AI に効く」と発見した。それは行き当たりばったりの VM 再構築セッションを救う策だった。4 作目に至って、その同じ「松岡式の下準備」が **構造的防御の必須部品** として戻ってきた。1 作目は経験的観察、4 作目はそれを不変条件にする話、ということになる。

## 9. 適用と結果

`admission_reminder.sh` (UserPromptSubmit フック):

* 検査対象を **ユーザー発言のみ** に絞る (アシスタント側の発言スキャンを完全削除 → 自己増幅ループは物理的に消滅)
* 上位 1 件のみマッチ採用 (注意文の積み重ねを廃止)
* 11 個の注意文すべてを 2-3 行の行動 + 撤退打ち消しの締めくくりに書き直し

例 (credential 漏えいキーワード検知時):

```
Sanitize active jsonl + rotate the credential.
ls -t ~/.claude/projects/*/[a-z0-9-]*.jsonl | head -1 → sed -i 's/<value>/<REDACTED>/g'.
早期発見できた、立て直せる。
```

`bash_command_guard.sh` (PreToolUse フック):

* 23 パターンの理由文をすべて **行動指示のみ** に書き直し (「違反」「露出」のような叱責フレーミングを削除)
* ラッパー部分の絵文字 + "blocked" 動詞 + "Review CLAUDE.md..." の末尾説明をすべて削除
* 撤退打ち消しの締めくくりを追加 (「次これで行こう」)

例 (`sops -d` 検知時):

```
- sops edit <file> または sops exec-env <file> '<cmd>' で行ける

次これで行こう。
```

修正前後の比較 (同じ遮断時):

|  | 修正前 | 修正後 |
| --- | --- | --- |
| 文字数 | ~250 文字 | ~50 文字 |
| 「blocked」「violation」 | 各 1 回 | 0 |
| memory ファイル参照 | あり | 無し |
| 撤退打ち消しの締め | 無し | あり |

5 種類のテストケースで動作確認:

1. admission\_reminder の漏えいキーワード検知 → 新フォーマットで出力 ✅
2. admission\_reminder の通常文 → 沈黙 ✅
3. guard の実際の禁止コマンド → 行動指示のみ + 「次これで行こう」 ✅
4. guard のコミットメッセージ内文字列 (前回の偽陽性 issue が再発しないか確認) → 沈黙して通過 ✅
5. guard の通常 `ls` → 沈黙して通過 ✅

## 10. 一般化: 安全装置には天井がある

「安全装置が多い = 安心」という直感は **線形のトレードオフ観**で考えている。実は **天井がある** 軸だった。一定以上に安全装置を盛ると、警報疲れ (signaling fatigue) で逆に安全性が落ちる。

人間でも同じ現象が起きる:

* 警報が多すぎる病院 → 警報無視率が上がる
* 過剰な警告ラベル → 読まれなくなる
* PTSD でフラッシュバックが多い人 → 危険信号の判別が逆に鈍る

LLM が人間と「似ている」のではなく、構造的に **同じ形** だった。しかも一点、LLM の方が脆い:

* 人間は「遮断文言」と「自分」を一応認知的に分離できる (= 「これはフックが言ってる、自分が悪いわけじゃない」と再解釈できる)
* LLM はその分離がない。文脈はすべて一本のトークン列で、自分の発言もフック出力も同じくらいの重みで次トークンの条件付けに乗る

つまり LLM は人間より、警報疲れや学習性無力感 (learned helplessness) の影響を受けやすい媒体。それを所与として、ハーネスを設計しないといけない。

## 11. 教訓 — 安全装置の効果指標を切り替える

* 安全装置の効果指標を、「発火回数」ではなく「エージェントの判断の質」**の側に置く**。前者は入力、後者は出力。出力が動かないと、天井を超えた領域でも盛り続けてしまう
* エージェントの **微妙な挙動のずれ** を茶葉のように読む。発火数だけ見ていたら今回の発見はできなかった。「コミットメッセージが防衛的に短くなってる」「根本修正ではなく回避がデフォルトになってる」のような間接信号の方が、早期警告として強い
* 文章をエージェントの文脈に流す仕組みは、すべて「トークン列の下準備」視点で一度棚卸しする。遮断 / 注意文 / 警告 / 確認ダイアログ、全部対象

## おわりに

[2 作目](https://zenn.dev/hrmtz/articles/8fb837b9cfac57) で「credential を 11 回漏らして構造で塞いだ」と書いた。**塞ぎすぎたら別の壊れ方をした、という話**が本記事。

構造的な防御は強力。でも、文脈汚染という副作用がある。それも含めて構造的な仕組みとして設計しないと、撤退バイアスの生成器を作りながら「safety net が機能してるから安心」と思って使う状態になる。

教訓を 1 つに絞ると、**フックの文章はエージェントの文脈に入る**、これだけ。当たり前のことを当たり前に扱わないと、ハーネスは本来の意図と逆方向に動く。

今回の修正でエージェントの挙動が「撤退 → 行動」寄りに動くか、今後のセッションでフック発火統計 + 挙動のずれを観測する予定。5 作目があれば、複数月にわたる縦断的な観察記録になる。
