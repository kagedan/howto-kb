**結論：非エンジニアがClaude Codeを使う価値はあるか**

あります。ただ「AIに全部任せる」という使い方は推奨しません。

全く知識のない領域をそのまま委ねると、人間の判断が必要な場面で判断できなくなります。  
「この方向でいいのか」「これは正しいのか」を判断するには、最低限の文脈理解が自分の中にないといけません。

自分が感じている価値はむしろ別のところにあります。  
Claude Codeは、業務の文脈・判断基準・蓄積データを継続的に育てていくことで、使い勝手が良くなっていく感覚があります。まだ自分も手探りですが、「育てる」という感覚でいると、長く使えるツールになると思っています。

* 向いてる使い方：特定業務の自動化・リサーチ・壁打ち・個人開発
* まだ検証しきれていない使い方：チームの自動化、MCP連携が必要な業務

## 1. なぜClaude Codeを使い始めたか

ChatGPTをメインで使っていましたが、Claude CodeのAgent機能を使って「仮想会社を立てて業務を効率的に回す」という記事を見て興味を持ち、「エンジニアじゃない自分でも試せそう」と思ったのがきっかけです。

## 2. Claude / Claude Codeとは

ClaudeはAnthropicが開発するAIです。  
ChatGPTと同じくチャット形式で使えるWeb版もありますが、Claude Codeはターミナル（文字を打ってPCを操作する画面）から動かすツールです。ブラウザのような画面はなく、命令文を直接入力して使います。ファイルを直接読み書きしたり、プロジェクト単位でAIとの会話の文脈を管理できるのが特徴です。

## 3. ChatGPTとの一番大きな違い：記憶と文脈の蓄積

ChatGPTを使っていて感じていた違和感は、会話がリセットされることでした。  
背景を理解していない瞬間が多々あり、毎回説明し直す感覚がありました。

Claude Codeはファイル構造で会話の文脈を保持する仕組みがあります。  
CLAUDE.md（Claudeへの指示書となるファイル。自分の役割やルールを書いておくと毎回読み込まれます）やメモリ機能によって、使うほど自分のことを理解してくれる感覚があります。  
メモリを資産としてストックしていけることが、今のところ一番の価値だと感じています。

## 4. セットアップはどれくらい大変か

非エンジニアが一番心配するのはここだと思います。  
結論、半日もかかりませんでした。

Web版のClaudeとやり取りしながら、具体的なコマンドや手順を出してもらう形で進めました。エラーが出たらそのままターミナルの画面をスクショして貼れば解決策を返してくれます。ターミナルに慣れている人であればもっと早いはずです。

<https://code.claude.com/docs/ja/vs-code>

## 5. 実際の日常業務での使い方

最初は「仮想会社」的な構造を作ろうと、ops（リサーチ・作業実行）やpm（進捗・優先順位管理）といったフォルダを立てて試みました。ただ実際に使っていくうちに、projectsフォルダの中に案件ごとのフォルダを作って管理するシンプルな形に落ち着きました。

実際の構成はこんな感じです。

クライアント案件は `overview / notes / tasks` が基本で、業務内容に応じて `data`（KPI・流入データ）や `meta-ads`（広告運用）などを足していく構造になっています。

```
ASTRSK/
├── CLAUDE.md                  # Claudeへの全体ルール・自分の役割・行動指針
├── shared/
│   └── company-context.md     # 会社・ブランドの基本情報（横断して参照）
├── projects/
│   ├── client-name/           # クライアント案件（共通構造）
│   │   ├── overview.md        # 案件概要・目標・担当範囲
│   │   ├── notes.md           # 議事録・打ち合わせメモ
│   │   ├── tasks.md           # タスク・進捗
│   │   ├── data/              # KPI・流入データなど（マーケ案件）
│   │   └── meta-ads/          # 広告運用（広告案件）
│   └── workout-app/           # 個人開発アプリ
│       ├── overview.md        # アプリ概要・画面構成・データ構造
│       ├── notes.md           # 開発メモ・技術設定の備忘
│       ├── tasks.md           # 開発タスク管理
│       ├── tech-stack.md      # 使用技術・バージョン管理
│       └── app/               # 実際のアプリコード（開発本体）
└── secretary/                 # 日次メモ・思考整理置き場
```

実際にやっていること：

* リサーチ（競合調査・市場調査など）
* 広告運用データを入れての改善提案の壁打ち
* 現状データをもとに「次に何ができるか」を一緒に考える

タスク管理はNotionで別に運用しているため、そこはClaudeに任せきれていないのが正直なところです。

リサーチ系はルール設定が効きました。たとえば「具体的な数字を出すときは必ず参照リンクを貼る」というルールを設けることで、情報を精査する手間が少なくなりました。

## 6. Vibe Codingでアプリを作ってみた（トレーニングメモアプリ）

Vibe Codingとは、コードを書かずにAIとの対話だけでアプリを作っていくスタイルのことです。非エンジニアでもここまでできるとは、という驚きがありました。

最初はワイヤーをきちんと書いてから設計しようと思っていましたが、UI/UXを考えながらだとどうしても時間がかかります。そこで方向転換。ワイヤーとして機能する最低限のデザインでまず実装してもらうことで、動くものを触りながらUI/UXを調整するという体験ができました。

「ざっくり動くもの → デザイン投入 → UX改善 → リリース版」という流れで進められるのが、非エンジニアにとってはやりやすかったです。

具体的には、筋トレログアプリを作るにあたって最初はワイヤーをきちんと設計してから実装しようとしました。ただ、種目の選び方・セット管理・プリセットの概念など、触ってみないと判断できない部分が多く、先に設計しようとすると手が止まりました。

そこで「UX優先でモックを触りながら設計を固める」方針に切り替えました。まず最低限動くものを実装してもらい、実機で触りながら「この導線は違う」「ここに画面が必要」と調整していく流れにしました。結果的に、プリセット画面だけで5画面の構成が固まっていきました。

## 7. まだ使いきれていない正直なところ

**Agent機能**  
最初の動機だった「仮想会社を回す」はまだ実現できていません。Agent機能（Claudeが自律的に複数の作業を連続で実行してくれる機能）が処理量をかなり消費することがわかったり、そこまで自動化したい業務が今のところなかったり、後述のMCP接続をしていないことが理由です。Claude Code自体に慣れてきたので、ここにも着手していきたいと思っています。

**Skills・スラッシュコマンド**  
`/commit`や`/review`のように「/」から始める命令文を打つだけで特定の作業を実行できる機能があります。存在は知っていましたが習慣化できていませんでした。

**MCP・自動化**  
MCP（SlackやNotionなどの外部ツールとClaudeを繋ぐ仕組み）の接続もほぼしていません。クライアント案件が多く、自動化できるところが少ないというのが正直なところです。

## 8. まとめ

非エンジニアでも使う価値はあります。ただし、自分がある程度理解している業務に使うのが前提だと思います。知識のない領域を丸投げするのではなく、自分の文脈や判断基準をClaude Codeに育てていく感覚で使うのが、今のところ一番しっくりくる使い方だと感じています。
