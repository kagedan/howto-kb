AI エージェント型コーディングツール（OpenCode, Claude Code 等）の普及により、**ローカル開発環境**での API キーをはじめとする機密情報の管理が新たなリスクになっている。`.env` ファイルに平文で保存する従来の方法では、エージェントによる予期せぬ露出が避けられない。

この記事では、macOS 標準の `security` コマンドを使って、機密情報を **ファイルとして一切残さず** OS の暗号化領域で管理する方法を紹介する。

## TL;DR

```
# 保存（対話入力 → シェル履歴に残らない）
security add-generic-password -a "$USER" -s "MY_PROJECT_API_KEY" -w
# → プロンプトにキーを貼り付けて Enter

# 呼び出し
export API_KEY=$(security find-generic-password -a "$USER" -s "MY_PROJECT_API_KEY" -w)
```

ファイル不要。`.env` そのものが存在しないため、AI エージェントが読み取ることも、誤って Git に追加することもない。

# 1. なぜ .env ではなくキーチェーンなのか

## `.env` 運用の最大のリスク：AI エージェントによる露出

従来からある `.gitignore` 設定漏れによる誤コミットリスクに加え、AI エージェント型ツールの台頭で新たな脅威が生まれている。AI エージェントは `.env` を**単なるテキストファイル**として扱うため、次のような露出シナリオが発生する。

* **コンテキストへの取り込み**: エージェントが `.env` を読み取り、会話コンテキストや解析対象に含める。内容がリモートの LLM に送信される。
* **ログ・デバッグ出力への混入**: エージェントがエラー調査の過程で環境変数やファイル内容を出力し、ログに平文のキーが残る。
* **ファイルの複製・移動**: エージェントが `.env` を別の場所にコピーしたり、`.env.example` を実値入りで生成したりする。
* **意図しない編集**: エージェントが `.env` の内容を書き換え、キーが破損または上書きされる。

これらの根本原因は、機密情報が**ファイルとして存在している**ことにある。

## macOS キーチェーンが解決すること

macOS の**キーチェーン**（Generic Password）に保存すれば、データは暗号化されてディスクに格納され、必要なときだけメモリ上に展開される。`.env` というファイル自体が存在しないため、エージェントがファイルとして持ち出すことは原理的に不可能になる。

# 2. security コマンドの基礎

## security とは

`security` は macOS に標準搭載されている CLI で、キーチェーンに対する操作を行う。Apple の公式ドキュメントは `man security` または [security(1) man page](https://keith.github.io/xcode-man-pages/security.1.html) を参照。

本記事で扱うのは、汎用的なパスワード（Generic Password）を操作する以下のサブコマンドである。

## アカウント名とサービス名について

`security` コマンドでは、`-a`（アカウント名）と `-s`（サービス名）の2つでエントリを識別する。これらは**キーチェーン内部での任意の検索タグ**であり、外部サービスと連携しているわけではない。

* **アカウント名 `-a`**: そのキーの「所有者」を示すラベル。基本的に `$USER`（macOS のログインユーザー名）で問題ない。
* **サービス名 `-s`**: そのキーの「用途」を示すラベル。**プロジェクトごと・プロバイダごとに一意な名前をつける**（例: `OPENAI_MYAPP`, `GITHUB_WORK`）。

両者の組み合わせでキーチェーン内のエントリを一意に特定できる。

## add-generic-password の主要引数

| 引数 | 意味 | 必須 |
| --- | --- | --- |
| `-a` | アカウント名。`$USER` でよい（任意の検索タグ） | 任意 |
| `-s` | サービス名。プロジェクト単位で命名して複数キーを区別する | **必須** |
| `-w` | 保存する値（API キー、トークン等）。**省略すると対話入力になる** | 保存時任意 |
| `-l` | ラベル。キーチェーンアクセス.app での表示名 | 任意 |

## find-generic-password の主要引数

| 引数 | 意味 |
| --- | --- |
| `-a` | 検索フィルタ（アカウント名） |
| `-s` | 検索フィルタ（サービス名） |
| `-w` | **値のみ**を標準出力。環境変数へのセット用 |
| `-g` | 全メタデータを表示（確認用） |

# 3. 実践：キーの保存・呼び出し・削除

## キーを保存する

```
# 推奨: 対話入力（シェル履歴にキーが残らない）
security add-generic-password -a "$USER" -s "OPENAI_API_KEY_PROJ_A" -w
# → プロンプトが表示されたらキーを貼り付けて Enter
```

サービス名 `-s` はプロジェクトごとに変える。例：`ANTHROPIC_API_KEY_MYAPP`、`GITHUB_TOKEN_WORK` など。

## キーを環境変数として設定する

```
# キーの値のみを取得し、環境変数にセット
export OPENAI_API_KEY=$(security find-generic-password -a "$USER" -s "OPENAI_API_KEY_PROJ_A" -w)
```

## キーを確認する

```
# メタデータを含めて確認
security find-generic-password -a "$USER" -s "OPENAI_API_KEY_PROJ_A" -g
```

キーチェーンアクセス.app（`/Applications/Utilities/Keychain Access.app`）から GUI で確認・削除することもできる。

## キーを削除する

```
security delete-generic-password -a "$USER" -s "OPENAI_API_KEY_PROJ_A"
```

# 4. おまけ：シェル起動時・プロジェクト単位の自動設定

毎回 `export` を打つ手間を省くには、次の2つの方法がある。

## `.zshrc` / `.bashrc` に書く

常に使うキーが1つだけなら、シェル設定ファイルに直接書けばよい。

```
# ~/.zshrc
export OPENAI_API_KEY=$(security find-generic-password -a "$USER" -s "OPENAI_API_KEY" -w)
```

* **用途**: 個人用の API キーが1つで、全プロジェクト共通でよい場合等。

## direnv でプロジェクトごとに切り替える

プロジェクトごとに異なるキーを使い分けるなら、[direnv](https://direnv.net/) を使う手がある。`.envrc` にキーチェーン呼び出しを書いておけば、そのディレクトリに `cd` したときだけ環境変数がセットされる。

* **用途**: 複数のプロジェクトなどで、リポジトリごとにキーを設定したい場合等。

# 5. AI エージェントとの付き合い方

本記事の構成であれば、`.env` には**機密情報が一切含まれていない**ため、エージェントが `.env` を読み取ってもキーは漏洩しない。

もう一つ重要なのは、**エージェントに `security` コマンドを実行させない**ことだ。キーチェーンへの保存・呼び出しはあくまで開発者自身が行い、エージェントには環境変数として渡すだけに留める。ツール側のガードレール（OpenCode の `permission` 設定等）で `security` コマンドの実行をブロックすれば、エージェントがキーチェーンに触るリスクも排除できる。

!

ただし、環境変数として読み込む特性上、あえてecho等で中身を勝手に出力されてしまえば、マスクされないままログなどに出力されてしまう可能性は拭いきれない。基本的には、もし何かあった場合は、APIキーをいつでも破棄できるよう使い捨てる運用が前提となりそうだ。

macOS標準機能を利用した本稿の方法はお手軽ではあるものの、ローカル環境でそもそもクレデンシャルを環境変数から読み込まない方法は別途紹介するかもしれない。
