# はじめに

Claude Opus 4.6 の **1M トークン（100万トークン）コンテキスト** が使えるようになってから、大規模リポジトリの扱いがガラッと変わりました。

これまで Claude Code で大きな Terraform リポジトリを扱うときは、

* 「関連ファイルだけ絞って読ませる」
* 「Agent tool で分割探索させる」
* 「要点だけまとめた memory を持たせる」

といった工夫が必要でした。**1M コンテキストはこの前提を一度リセットします。**

この記事では、170 ファイル以上ある実プロジェクトの Terraform を実際に 1M コンテキストで扱ってみた体験を、コスト・精度・運用面から紹介します。

# 前提：扱ったリポジトリの規模

| 項目 | 内容 |
| --- | --- |
| Terraform ファイル数 | 172 |
| 総行数 | 約 18,000 行 |
| モジュール数 | 24 |
| 環境 | dev / stg / prd |
| トークン概算 | 約 25 万トークン（コメント込み） |

つまり **Terraform 全体を読ませても 1M の 1/4 程度**。アプリケーションコードを同時に読ませても余裕があります。

# 試したこと 1：全ファイル一括読み込みで整合性チェック

## やったこと

`terraform/` 配下の `*.tf` を全部 Claude に読ませた上で、こう指示しました。

> 全モジュールの IAM ポリシーを横断的にレビューして、最小権限違反・リソース ARN のワイルドカード使用・未使用の Statement を洗い出して

## 結果

* **検出した問題：12 件**（うち Critical 3 件）
* **処理時間：約 90 秒**
* **入力トークン：約 27 万**
* **出力トークン：約 8,000**

特に価値があったのは、**モジュール A の IAM ロールがモジュール B のリソースに過剰権限を持っている**というクロスモジュールの指摘でした。これは単一ファイルを読ませていた頃には絶対に見つけられなかったやつです。

## 体感の精度

指摘の妥当性は高かったですが、**誤検知もゼロではない**です。具体的には：

* `data.aws_iam_policy_document` で定義されて他モジュールに export されているケースを「未使用」と判定したことがありました
* CloudFormation でしか参照されない IAM リソースを「使われてない」とした例も

ここは 1M だから解決するわけではなく、**Claude にどこまで渡すか**の設計が必要です。

# 試したこと 2：リファクタリング案の提示

## やったこと

> 現状の Terraform モジュール分割を見て、密結合になっているモジュールを特定して、分割リファクタリング案を出して

全 172 ファイルを読ませた状態で提案させました。

## 結果

* モジュール間の依存グラフを文章で再構築した上で、
* 「`module.api` と `module.worker` は共有 DynamoDB テーブルの定義を通して密結合になっている」
* 「`module.shared-resources` を新設して、テーブル定義を切り出すべき」

というレベルの提案が返ってきました。これは**コード全体を見渡せないとできない仕事**で、1M コンテキストの真価が出たケースです。

# 試したこと 3：アプリコード + Terraform の整合性チェック

## やったこと

TypeScript のアプリケーションコード（約 3 万行、12 万トークン）と Terraform（25 万トークン）を**両方同時に**読ませ、

> Lambda 関数の IAM ポリシーで付与されている権限と、実際にコードから呼ばれている AWS API の整合性を確認して。過剰権限を洗い出して

と指示しました。

## 結果

入力トークンは約 40 万。これが**リアルタイムにやれる**のが衝撃でした。

* `DynamoDB:Scan` 権限を持っているが実際は `Query` しか使っていない関数 → 5 件
* `S3:*` 権限を持っているが実装上 `GetObject` しか使わない関数 → 3 件
* そもそも Terraform で権限が付与されているが、該当関数がコードから消えている → 2 件

これは**アプリとインフラを同じコンテキストで見られる**ことで初めて成立する分析です。

# コスト面のリアル

## プロンプトキャッシュが命

1M コンテキストでは **プロンプトキャッシュ** が効くかどうかでコストが1桁変わります。

| モード | 入力コスト（概算） |
| --- | --- |
| キャッシュなし | 1.0x |
| キャッシュ読み込み | 約 0.1x |

同じリポジトリに対して立て続けに質問するときは、キャッシュの **5 分 TTL** が切れる前に次の質問を投げるのがコツです。Claude Code のセッションを開きっぱなしにしていれば、基本的にヒットします。

## コンテキストをフルに使うのは「ここぞ」のとき

常に 1M フルに読み込ませるのはオーバーキルかつ高コストです。私の使い分けはこんな感じです。

| 用途 | コンテキスト量 | モデル |
| --- | --- | --- |
| 単一ファイル編集 | 数千トークン | Haiku / Sonnet |
| 特定モジュールの修正 | 数万トークン | Sonnet |
| クロスモジュール分析 | 数十万〜1M | Opus 1M |
| リポジトリ全体リファクタ | 1M | Opus 1M |

# 1M コンテキストでも注意すべきこと

## 1. 入れればいいわけではない

「全部読ませれば精度が上がる」と思いがちですが、**ノイズが増えると逆に迷う**ケースがあります。特に、

* 自動生成ファイル（`.terraform.lock.hcl` など）
* ロックファイル（`package-lock.json`、`go.sum`）
* 大量のテストフィクスチャ

は**意図的に除外**した方が精度が上がりました。

## 2. 出力は相変わらず有限

入力が 1M 扱えても、**1 回のレスポンスで出せる出力トークン数は従来と同程度**です。大量のコード変更を一気に出させようとすると途中で切れます。

対策：

* 「ファイル単位で順番に出して」と指示する
* 複数ターンに分けて進める
* 大きな変更は plan → chunked execution に分ける

## 3. 長いほど「中間が見落とされる」は依然存在

これは人間もそうですが、コンテキストが長いほど**中盤に入った情報が軽視されがち**な傾向は残っています。重要な制約は冒頭または末尾に置くのが引き続き有効です。

# 運用上のベストプラクティス

私が落ち着いた運用は以下です。

1. **通常作業は Sonnet + 必要ファイルだけ**（コスパ重視）
2. **クロスモジュール分析・リファクタ検討のときだけ Opus 1M を召喚**
3. **除外パターンを `.claudeignore` 的に明示**してノイズを減らす
4. **1M セッションは長めに保持**してキャッシュを活かす
5. **重要な制約は CLAUDE.md 末尾で再掲**して中間埋没を回避

# まとめ

Claude Opus 4.6 の 1M コンテキストは、単なる「入力上限の拡大」ではなく、**コードレビューと設計相談の質を一段階変える機能**でした。

* 170 ファイルの Terraform が丸ごと読める
* アプリ × インフラの整合性を同時に見られる
* モジュール間の密結合をグローバルに指摘できる

ただし **コスト・ノイズ・出力上限** というトレードオフは残るので、普段は Sonnet、ここぞで Opus 1M という使い分けが今のところ正解だと感じています。

大規模リポジトリを触っている人ほど、1M コンテキストを体験しておく価値はあると思います。
