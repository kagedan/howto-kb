## はじめに

個人でメディア（ブログや note のような媒体）を続けていて、最初に枯れるのは「書く力」ではなく「ネタ」でした。

何について書くかが尽きれば、更新は止まる。つまり書く力よりも先に、**ネタを切らさない仕組み**が要る。なのに自分の場合、そこが手作業と気分まかせで、属人化していました。

RSS やSNSの巡回まではすでに n8n（self-hosted）＋ LLM ＋ スプレッドシートで自動化していたのですが、**動画と音声（YouTube・Podcast）が手つかず**でした。良質な情報の多くが動画で流れている時代に、ここを拾えていないのはもったいない。

そこで「動画/音声 → ネタの種」に変えるパイプラインを作り、まず **10本でパイロット** を回しました。この記事は、その**実装の記録と、机上では分からなかった発見**をまとめたものです。結論から言うと、いくつもの前提が手を動かした瞬間に崩れました。

対象読者として想定しているのは、個人開発者、n8n ユーザー、LLM で情報収集や要約を自動化したい人、そして **ローカル Whisper（faster-whisper）の実用性を知りたい人**です。

## 作ったものの全体像

```
[源] RSS / SNS / YouTube / Podcast
   ↓
[n8n] 収集・重複除去・前処理（週次バッチ）
   ↓
[文字起こし] 字幕あり → yt-dlp で字幕取得
            字幕なし → faster-whisper（ローカル・GPU）
   ↓
[抽出] Claude API が JSON で
        「ネタ候補 / 言及された研究・概念 / 視聴者の生の声」を返す
   ↓
[スプレッドシート] に蓄積 → 最終的に何を書くかは人間が選ぶ
```

最初に強調しておきたい設計思想がひとつあります。\*\*目的は「全文を一字一句正確に書き起こすこと」ではなく、「ネタの種・調べ直すべき手がかり・読者の生の言葉を拾うこと」\*\*です。この一点が、後で出てくる判断（モデル選択・字幕優先）をすべて決めました。完璧な文字起こしは要らない、という割り切りが効率を生みます。

## 大前提：文字起こしは「発見装置」であって、原稿ではない

技術の話に入る前に、いちばん大事な線引きを書いておきます。

このパイプラインは、**他人の動画や音声を文字起こしして、それをリライトして記事にするものではありません**。それは内容の再構成になりかねず、著作権の面でもグレーで、何より自分の言葉になりません。

やっているのは、こういう流れです。

1. 動画/音声から「語られている研究・概念」「視聴者コメントに出てくる生の言葉」を**拾う**
2. その手がかりをもとに、**一次情報（元の論文や公式情報）に自分で当たり直す**
3. 確認できた事実を、**自分の体験と文脈で、自分の言葉で書く**

つまり文字起こしは「世の中で今どんな話題・研究が語られているか」を知るための**発見装置**であって、原稿ではない。ここを最初に決めておくと、ツールの使い方も「全文の精度」ではなく「手がかりが拾えるか」に最適化できます。倫理と設計が、きれいに一致します。

## 発見①：「字幕なし動画」は YouTube にほぼ存在しなかった

当初の設計はシンプルでした。「字幕がある動画は字幕を使う。字幕がない動画は Whisper で起こす」。

ところがパイロットで分かったのは、**YouTube はほぼすべての動画に自動字幕を生成している**という事実です。手動字幕がなくても、自動生成字幕（auto-generated captions）がほぼ必ず付いている。つまり、**「字幕が本当に存在しない YouTube 動画」は稀**でした。

これは設計を変える発見でした。

* **YouTube は、字幕取得（yt-dlp）でほぼ足りる。** わざわざ音声をダウンロードして Whisper に通す必要は、ほとんどない。
* **Whisper の本当の出番は Podcast。** 音声しかなく字幕が存在しないコンテンツこそ、ローカル文字起こしの価値が出る。

手を動かす前は「動画＝Whisper」だと思い込んでいましたが、実際は「YouTube＝字幕、Podcast＝Whisper」が正解でした。パイロットの一番の収穫は、この前提の修正だったかもしれません。

字幕だけを取るなら、yt-dlp はこうです（音声はダウンロードしない）。

```
yt-dlp \
  --write-auto-subs \
  --sub-langs "ja,en" \
  --skip-download \
  --sub-format vtt \
  "https://www.youtube.com/watch?v=XXXXXXXXXXX"
```

「字幕があるか」の判定は、yt-dlp のメタ情報で `subtitles`（手動）と `automatic_captions`（自動）のキーを見れば分かります。今回プローブした範囲では、ほぼ全動画で `automatic_captions` に `ja` か `en` が入っていました。

ひとつ実務的な注意。**自動字幕の VTT は、同じ行が少しずつ伸びながら重複します**（rolling）。そのまま結合すると同じ文が何度も出るので、畳む処理が要ります。

```
import re

def parse_vtt(path):
    lines = []
    for s in path.read_text(encoding="utf-8", errors="ignore").splitlines():
        s = s.strip()
        if not s or s == "WEBVTT" or "-->" in s:
            continue
        if s.startswith(("Kind:", "Language:", "NOTE")) or re.fullmatch(r"\d+", s):
            continue
        s = re.sub(r"<[^>]+>", "", s)   # <c> や <00:00:00.000> タグを除去
        if s:
            lines.append(s)
    # rolling 重複を畳む：前行が現行の接頭辞なら現行で置換、完全一致はスキップ
    out = []
    for line in lines:
        if out and line == out[-1]:
            continue
        if out and line.startswith(out[-1]):
            out[-1] = line
            continue
        if out and out[-1].startswith(line):
            continue
        out.append(line)
    return "\n".join(out)
```

## faster-whisper：large-v3 と medium を RTX 4090 で比べた

Podcast 用（と、字幕がない例外用）に、ローカルの文字起こしを faster-whisper で組みました。環境は RTX 4090 + CUDA。モデルサイズを large-v3 と medium で比較しています。

最小コードはこれくらいです（実際はモデルを使い回し、RTF＝実時間比も計測しました）。

```
import time
from faster_whisper import WhisperModel

# モデルは毎回ロードせず使い回す。4090 なら float16。
_model = WhisperModel("large-v3", device="cuda", compute_type="float16")

def transcribe(audio_path, lang_hint=None):
    t0 = time.perf_counter()
    segments, info = _model.transcribe(
        str(audio_path),
        language=lang_hint,   # "ja"/"en" を渡すと安定。None なら自動判定
        vad_filter=True,      # 無音区間をスキップ（高速化＋誤認識減）
        beam_size=5,
    )
    text = "".join(seg.text for seg in segments).strip()
    elapsed = time.perf_counter() - t0
    rtf = round(elapsed / info.duration, 3) if info.duration else None  # 小さいほど速い
    return text, rtf
```

音声は yt-dlp の bestaudio をそのまま渡せました。

```
yt-dlp -f "bestaudio/best" -o "audio.%(ext)s" "https://..."
```

### 速度

RTX 4090 での体感はこうでした。

* **large-v3**：実時間の約 1/11（≒11倍速）。30分の音声が約2.5分で起こせる。
* **medium**：さらに速く、約 1/15。

週次バッチで回す前提なら、どちらも十分すぎる速度です。

### 精度

差が出たのは精度でした。

* **medium**：内容は取れるが、**日本語で同音語の誤りが目立つ**。実際に「習慣」が「集会」になるような取り違えが出ました。さらにトークン間に余計なスペースが入り、**固有名詞（研究名・人名）が崩れる**ケースがありました。
* **large-v3**：日英ともに自然で、固有名詞・専門語も安定。

今回の用途は「語られている研究名や人名を正しく拾って、後で一次情報に当たり直す」こと。固有名詞が崩れると発見装置として機能しません。**速度は medium が上でも、4090 なら large-v3 でも十分速い**ので、迷わず large-v3 を採用しました。

教訓として、「精度が最優先ではない」用途でも、**固有名詞を拾いたいなら large-v3** が無難です。クラウドの文字起こしAPIや日本語特化モデルとの比較も検討しましたが、large-v3 で要件を満たせたので、現時点では不要と判断しました。

## 抽出は Claude Haiku で十分だった（コストの話）

文字起こし（または字幕）から「ネタ候補・言及された研究・視聴者の生の声」を構造化するのは、LLM の仕事です。ここを Claude の2つのモデルで比較しました。

JSON で返させるのに、プロンプトで「JSONだけ出して」と頼むより、**構造化出力（json\_schema）でスキーマを強制**する方が堅いです。実際のコードはこんな形です（プロンプトの文面は媒体のジャンルに合わせて書くので、ここでは一般化しています）。

```
import os, json, anthropic

client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
MODELS = {"haiku": "claude-haiku-4-5", "opus": "claude-opus-4-8"}

# 受け取りたい「種」の構造
SCHEMA = {
    "type": "object",
    "properties": {
        "neta_candidates": {"type": "array", "items": {"type": "string"}},  # ネタ候補（テーマ/場面）
        "research_leads":  {"type": "array", "items": {"type": "string"}},  # 研究・概念・書籍の手がかり
        "audience_voice":  {"type": "array", "items": {"type": "string"}},  # 読者の生の言葉（日本語で短く）
        "exclusion_flags": {"type": "array", "items": {"type": "string"}},  # 媒体が扱わないテーマ 等
    },
    "required": ["neta_candidates", "research_leads", "audience_voice", "exclusion_flags"],
}

SYSTEM = (
    "あなたは、ある個人メディアのリサーチ担当です。"
    "文字起こしと視聴者コメントから、記事の『種』を発見します。\n"
    "原則：文字起こしは発見装置。転載・詳細要約はしない。取るのは——\n"
    "・ネタの種（テーマ/場面）\n"
    "・研究への手がかり（人名・概念・書籍など、後で一次情報に当たれるもの）\n"
    "・読者の生の言葉（コメントに最も濃い）\n"
    "出力は必ず日本語に統一する（英語ソースでも日本語で出す）。"
)

def extract(model_id, user_text):
    resp = client.messages.create(
        model=model_id,
        max_tokens=4000,
        system=SYSTEM,
        messages=[{"role": "user", "content": user_text}],
        output_config={"format": {"type": "json_schema", "schema": SCHEMA}},
    )
    text = next(b.text for b in resp.content if b.type == "text")
    data = json.loads(text)
    usage = (resp.usage.input_tokens, resp.usage.output_tokens)  # コスト計測用
    return data, usage
```

ひとつ実装上のコツ。再生数やコメント総数のような**数値は、モデルに推測させず、yt-dlp のメタ情報から取った実値で後から上書き**します（LLM に数字を作らせない）。

比較した結果はこうです。

* **Haiku 4.5**：必要な「種・手がかり・生の声」は十分に取れる。研究名（例：ある特性に関する研究者の名前や書籍名）も追跡できる形で拾えるし、コメント欄から「そのまま記事の入口になる生の言葉」も抽出できる。
* **Opus 4.8**：Haiku より**網羅的・逐語的**（英語原文も併記してくる等）。ただし上積みは「網羅度・逐語性」であって、発見装置としての要件は Haiku で満たせている。

そしてコスト。**10本での実測で、Haiku ≈ $0.10、Opus ≈ $0.61** でした。約6倍の差です。

結論は「**普段は Haiku、特集回など気合を入れるときだけ Opus**」の二段運用。週次バッチで回しても、抽出コストは月数十円のオーダーに収まります。「とりあえず一番賢いモデル」を選ばず、用途に対して**十分な最小のモデル**を選ぶだけで、ランニングコストが一桁変わる好例でした。

## ハマりどころ集（再現するなら、ここで詰まります）

実装で時間を溶かしたポイントを残しておきます。

**1. faster-whisper が GPU を掴まない（CUDA / cuDNN の DLL 解決）**  
Windows で faster-whisper を動かすと、`cublas64_12.dll is not found or cannot be loaded` のような cuDNN/cuBLAS 系の DLL エラーで落ちることがあります。中身の ctranslate2 が **PATH 経由**で DLL を解決するため、`os.add_dll_directory` だけでは足りず、**venv 内の nvidia パッケージの bin を PATH に前置**する必要がありました。

```
import os, sys
from pathlib import Path

def add_cuda_dlls():
    """ctranslate2(faster-whisperの中身)は PATH 経由で cublas/cudnn/cudart を解決する。
    add_dll_directory だけでは 'cublas64_12.dll is not found' になるので PATH 前置が要る。"""
    nv = Path(sys.prefix) / "Lib" / "site-packages" / "nvidia"
    if not nv.is_dir():
        return
    bins = [str(sub / "bin") for sub in nv.iterdir() if (sub / "bin").is_dir()]
    os.environ["PATH"] = os.pathsep.join(bins) + os.pathsep + os.environ.get("PATH", "")
    for b in bins:
        try:
            os.add_dll_directory(b)
        except Exception:
            pass

add_cuda_dlls()   # faster_whisper を import する前に呼ぶ
```

併せて、`pip install nvidia-cuda-runtime-cu12`（と cublas/cudnn の cu12 パッケージ）を入れておきます。

**2. YouTube 字幕取得の HTTP 429（レート制限）**  
同じIPから字幕を連続取得すると 429 で弾かれます。**1言語ずつ取得＋バックオフ**で回避しました（実際は固定ステップで十分でした）。

```
import time

for attempt in range(4):
    try:
        download_subtitles(url, lang)      # yt-dlp を1言語ぶん実行
        break
    except Exception as e:
        if "429" in str(e) and attempt < 3:
            wait = 25 * (attempt + 1)       # 25, 50, 75 秒
            print(f"[429] {lang} retry in {wait}s")
            time.sleep(wait)
            continue
        raise
```

**3. ffmpeg は要らなかった**  
最初は「音声変換に ffmpeg が要るだろう」と身構えましたが、yt-dlp の bestaudio をそのまま faster-whisper に渡せたので、今回の構成では不要でした。

**4. コメントは「上位だけ」取る／総数はメタから取る**  
視聴者の生の言葉は、コメント欄の\*\*上位（高評価順）\*\*に濃く出ます。yt-dlp の `getcomments` で `comment_sort: top` にし、`max_comments` を25件程度に絞って取りました（audience\_voice にはこれで十分）。注意は、**ここで取れる件数（≒25）と、動画の総コメント数は別物**だということ。温度指標に総数を使いたいときは、メタ情報の `comment_count`（数千件のことも）を別に拾って補完します。

## いちばんの授業料：AIエージェントを「従量課金」認証で動かして二重払いした話

最後に、いちばん高くついた失敗を正直に書きます。技術というより運用の話ですが、同じ轍を踏む人が多いはずなので。

実装には AI コーディングエージェント（Claude Code）を使いました。ところが、これが**サブスクリプション（定額プラン）ではなく、API 従量課金の認証で動いていた**ことに、しばらく気づきませんでした。

長時間のセッション（パイプラインの監視を含む数時間）を回した結果、**気づいたら従量課金で $24 ほど溶けていました**。定額プランにも入っているので、実質二重払いです。「残高不足」のエラーで初めて発覚しました。原因は、既定で最上位モデルが選ばれており、長時間の見張り作業でトークンを大量に消費していたこと。

対処と教訓:

* **エージェントを起動したら、まず認証方式（定額 or 従量）を確認する。** Claude Code なら `/status` で確認でき、`/login` で定額側に切り替えられます。
* **長時間の監視・見張り系タスクは、モデルを一段下げる。** 最上位モデルを張り付けっぱなしにすると、枠もコストも一気に減ります。

「便利なエージェントほど、裏で何の財布から払っているかを見落としやすい」というのが、$24 の学びでした。

## 自動化で、何が変わったか（費用・時間・その先）

### 費用：ランニングはほぼゼロ

* 文字起こしは**ローカル（faster-whisper）なので API 課金はゼロ**。かかるのは電気代だけ。クラウドの文字起こしAPIだと、動画を量で回すほど従量課金が積もるが、そこを RTX 4090 が吸収してくれる。
* 抽出（Claude Haiku）は、週次バッチでも**月数十円**のオーダー。
* 初期に $24 の授業料（前述の認証ミス）はあったが、これは一回限り。**定常コストはほぼ無視できる**水準に収まった。

### 時間：「ネタ探し」という工程が消えた

* 以前は、動画を自分で漁ってメモする…という能動的な作業で、気分と時間に左右されていた。
* いまは週次バッチが、勝手に「種・研究の手がかり・読者の生の声」をシートに溜めてくれる。**自分の仕事は「溜まった候補から選ぶ」だけ**になった。
* 厳密な削減時間は測っていないが、体感としては\*\*「探す」工程がまるごと無くなり\*\*、浮いた時間を「書く」「一次情報に当たり直す」という、本当に価値のある作業に回せるようになった。

### その先：情報収集力は「次の打ち手」の土台になる

情報収集が仕組み化できると、効いてくるのは目先の効率だけではありません。

* **ネタ切れしない＝続けられる。** 個人メディアがいちばん死ぬ理由は「書くことが尽きて更新が止まる」こと。種が絶えず供給されるだけで、継続性が担保される。
* **質が上がる。** 読者の生の言葉が手元にあると、独りよがりでなく「読者が実際に困っていること」に沿って書ける。研究の手がかりから一次情報に当たれば、エビデンスのある記事になる。
* **横展開できる。** このパイプラインは、検索クエリとソースを差し替えるだけで別ジャンルにも複製できる。つまり\*\*「ネタ収集の仕組み」そのものが、再利用できる資産\*\*になる。
* **上に積める。** 収集と蓄積ができたことで、次は「トレンドの温度を見て出すタイミングを最適化する」「記事を短尺動画の台本に変換する」といった打ち手を、この土台の上に載せられる。

地味な自動化に見えて、**「情報の入口」を持つことは、その後にできることの幅を変える**——これが、10本のパイロットを終えての一番の手応えでした。

## まとめと、次の一歩

10本のパイロットで分かったことを整理します。

* パイプラインの3レーン（**字幕取得 / ローカル Whisper / LLM 抽出**）はすべて機能した。
* 本番構成は **「字幕優先 × faster-whisper large-v3（フォールバック・Podcast用）× Claude Haiku 抽出」**。ランニングは月数十円規模。
* 前提は手を動かすと崩れる。「動画＝Whisper」ではなく「YouTube＝字幕・Podcast＝Whisper」だった。
* モデルは「賢い順」ではなく「用途に十分な最小」を選ぶと、コストが一桁変わる。

次は、このパイロットを n8n の本ワークフローに組み込み（既存を壊さないよう複製ブランチで）、収集タイミングの最適化や短尺コンテンツへの展開を進める予定です。スクリプトは整理して公開も検討しています。

同じように「個人メディアのネタ収集を自動化したい」人の、最初の一歩の参考になれば幸いです。
