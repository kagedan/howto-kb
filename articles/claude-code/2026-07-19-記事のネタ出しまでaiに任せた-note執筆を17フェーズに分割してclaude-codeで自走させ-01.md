この記事の企画は、AIが出しました。Claude Codeが候補を11個、採点表つきのレポートにして提案してきて、筆者はその総合1位を選んだだけです。テーマ選びから調査、執筆、採点、校閲まで、note記事の制作はほぼClaude Codeが自走しています。

こう書くと「AIに丸投げしたら記事ができた」という話に聞こえますが、実態は逆です。丸投げは何度も失敗しました。うまく回り始めたのは、品質をプロンプトではなく仕組みで作るように設計を変えてからです。この記事では、その仕組みの設計図を公開します。記事執筆に限らず、繰り返しの知的作業をAIに任せたい方なら、型ごと転用できるはずです。

## 「いい記事を書いて」が失敗する理由

最初は一発の長い指示で書かせていました。ターゲットはこう、文体はこう、構成はこう、事実確認はこう。指示を足すほど品質が上がると思っていましたが、実際は逆でした。長い指示は、守られる部分と流される部分に分かれます。どこが流されるかは毎回違う。つまり品質が運次第になります。

行き詰まって、人間の編集部を思い出しました。雑誌もWebメディアも、一人が企画から校閲まで一気にやったりしません。企画会議、調査、執筆、編集者のチェック、校閲。別々の人が、別々の基準で、順番に仕事をします。

そこで記事制作を17のフェーズに割りました。担当はAIが12フェーズ、人間が5フェーズです。主要な流れはこうなっています。

```
テーマ考案 →【人間: 選定】→ ヒアリング →【人間: 方針承認】
→ 調査（3エージェント並列）→ たたき台 → 執筆 ⇔ 採点（合格まで改善ループ）
→ 最終レビュー → 校閲 → SNS投稿文 → 振り返り →【人間: 投稿】
```

【人間】と書いた場所だけが人間の仕事です。開発でいえばプルリクの承認やCIのゲートに近い位置づけで、それ以外の区間はAIが走ります。記事の性質によっては一部フェーズを省く軽量フローも用意しています。

分割して一番効いたのは、品質よりも先に「人間の判断ポイントが数カ所に固まった」ことでした。テーマの選定、方針の承認、レビュー。判断がそこにしかないなら、人間はPCの前に居続ける必要がありません。前回の記事で書いたスマホからの遠隔運用は、この分割があって初めて成立しています。

## 仕組みの核は「スキル」という手順書

フェーズ分割を支えているのが、Claude Codeのスキルという公式機能です。`SKILL.md` というファイルに手順を書いておくと、Claudeの道具箱に加わります。関連する作業のときにClaudeが自分で取り出すか、`/スキル名` で直接呼び出せます。

公式ドキュメントには、スキル化の基準として面白いことが書いてあります。「同じ指示やチェックリストを何度もチャットに貼っているなら、それがスキルにするサイン」。筆者のワークフローは、まさに毎回同じ指示を貼る作業の塊でした。フェーズごとに手順書を書き、仕事を棚に並べる。それだけで、毎回のプロンプト設計がゼロになります。

作り方は見た目より簡単です。プロジェクトの `.claude/skills/` の下にスキル名のフォルダを作り、`SKILL.md` に説明文と手順を書くだけ。説明文はClaudeが「今この道具を使うべきか」を判断する材料になるので、どんな場面で使うかを具体的に書いておきます。

スキルの本文は使われるときだけ読み込まれる仕様なので、手順書をどれだけ厚くしても普段の動作は重くなりません。だから遠慮なく書き込めます。

運用しながら分かったコツがひとつあります。スキルには「何をするか」だけでなく、**「なぜそうするか」と「やってはいけないこと」を書く**と挙動が安定します。たとえば採点担当のスキルは、実物から抜粋するとこんな中身です。

```
# 採点スキル（実物からの抜粋）
- 評価は4項目100点: フック力25 / 実用性・具体性30 / 構成・論理展開25 / 信頼性・正確性20
- 各項目の得点に必ず具体的な根拠を書く。根拠なしの採点は禁止
- 「なんとなく良い」は0点と同じ。記事のどの箇所がどう良い/悪いかを指摘する
- 記事に情報を追加しない。評価と提案のみ行う
```

手順だけ書くと、AIは手順の隙間で楽をします。理由と禁止事項を書くと、隙間でも設計者の意図に沿って動く。人間の新人教育と似ています。

## 品質は「採点とループ」で作る

仕組みの心臓部が、執筆→採点→改善のループです。

執筆担当が初稿を書くと、採点担当が4項目100点満点で採点します。フック力25点、実用性30点、構成25点、信頼性20点。合格ラインは段階式にしていて、1回目は80点、2回目は85点、3回目以降は90点。回数を重ねるほど要求が上がるので、ループがダラダラ続きません。

もうひとつ、効果が大きかったルールがあります。**指摘は全件対応してから再採点に出す。部分対応での再提出は禁止**。上位の指摘だけ直して再提出、また指摘、また一部だけ直す――人間のレビューでもよく見る泥沼を、ルールで最初から断っています。

実際の記録を出します。前回のDiscord運用記事は、初稿が86点で1回目に合格しました。それでも指摘は5件付いたので、全件対応してから確定しています。合格したから終わり、ではなく、合格しても指摘は消化する。この積み重ねが次の初稿の質を上げます。

先に白状しておくと、この点数はAIによる身内の採点です。基準を固定して運用のブレを見る道具としては機能しますが、これ単体では品質の証明になりません。だからこそ、身内ではないレビュアーと人間の関所を別に置いています。後述します。

## 事実の守り方: AIに書かせるなら一番大事な部品

AIに文章を任せるとき、最大のリスクは事実です。それらしい数字を作る、体験していないことを体験風に書く。ここを仕組みで塞がないと、他の全部が良くても公開できません。

筆者のワークフローでは、執筆前に「検証済み事実集」というファイルを1つ作ります。調査結果から確定した事実だけを集め、出典を付け、逆に「裏取りできなかったので使わない情報」も明記します。執筆担当のルールはシンプルで、**このファイルにない数値は書けない**。事実確認が「ファイルとの照合」という機械的な作業に変わります。

情報源には格付けがあります。公式の一次情報、著名メディア、個人の発信の3段階。数値や仕様は公式での裏取りを必須にして、個人発信だけの情報は主張の根拠にしません。体験談も同じ扱いで、「一次体験として書ける事実」を先にリスト化し、その境界の外は書かない。

ちなみに調査そのものは3つのAIエージェントを並列で走らせ、前回は30を超える情報源に当たりました。実装に凝ったことはしていません。Claude Codeには下請け作業を任せられるサブエージェント機能が標準であり、担当違いの指示書を3通り書いて同時に走らせているだけです。指示するトピックは最低限にして「周辺の面白い情報も広く拾ってこい」としてあるのがポイントで、記事の骨格になった素材の多くは、指示していない周辺情報から出てきています。

## 実例: 「次に何を書くか」まで型にした

冒頭の話に戻ります。今回新しく作ったのが、テーマ考案のスキルでした。ネタ出しは一番AIに任せにくい仕事に見えますが、分解してみると型がありました。

情報源は4系統です。過去記事と振り返りの棚卸し、筆者への2〜3問のヒアリング、Webのトレンド調査、そして発信テーマの偏りの分析。ここから候補を11個作り、X拡散性・実用性・一次体験度・時事性・執筆コストの5軸で採点して、推薦トップ3を出させます。候補の顔ぶれは、過去記事の続編、まだ書いていない発信ジャンルの開拓枠、直近の話題に乗る時事枠が自然に混ざっていて、正直、人間のネタ帳より視野が広いと感じました。

最終決定は人間です。前回の記事は推薦3位を選びました。この記事はレポートの総合1位です。「AIが企画した記事をAIが書いてAIが採点する」と聞くと不安になるかもしれませんが、回る理由ははっきりしています。各フェーズの責務が独立していて、人間の判断が明確な関所として挟まっているからです。

## 正直な課題

うまくいっている話だけでは設計図として不誠実なので、課題も隠さず書きます。

**自己レビューバイアスは消えない。** 書いたAIと採点するAIが同じ文脈を共有していると、欠点を見逃します。対策として、前回から仕上げ段階で、別ベンダーのモデルで動くAI CLIに辛口レビューをさせ始めました。執筆の文脈を一切共有していない相手を、意図的にレビュアーに選んでいます。指摘は4件来て、妥当な3件だけ反映し、こちらのブランド方針と衝突する指摘は理由を付けて却下。反映後の再採点は91点でした。

文脈を知らないレビュアーの指摘は読者の実感に近い一方、鵜呑みも危険です。指摘を選別する編集長役だけは、いまも人間側の設計が要ります。

**仕組みは作って終わりではない。** 17フェーズの最後は振り返りです。そこで出た改善点をスキルに書き戻して、次の記事に効かせます。実際、「表は最初からnote対応の記法で書く」という改善は、以前の振り返りから書き戻されたもので、直近の制作では表まわりの手戻りがゼロになりました。

**万能ではない。** 型のない仕事、初めての種類の仕事はスキル化できません。順序としては、まず人間が1回やってみて、手順が見えたら2回目からスキルにする。スキルは経験の圧縮であって、経験の代わりではありませんでした。

## まとめ: 丸投げではなく「判断を固める」

この仕組みの本質は、AIへの丸投げではありません。**人間の判断を数カ所に固める設計**です。判断が固まっているから品質が安定し、固まっているからスマホからの遠隔運用も成立します。

転用するときの型は4つだけです。

1. 繰り返している作業をフェーズに割る
2. フェーズごとに手順書（スキル）を書く。理由と禁止事項も添えて
3. 採点基準と合格ラインを決め、指摘は全件対応にする
4. 事実の境界線を執筆前に引く

まずは、何度もチャットに貼っている指示を1つ、スキルにするところから始めてみてください。前回の記事（スマホとDiscordでこのワークフローを見守る話）とあわせて読むと、全体像がつながるはずです。生成AI関連の記事は[マガジン](https://note.com/horizon_it00/m/md69e7754df22)にまとめています。よければフォローをお願いします。
