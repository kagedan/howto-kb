# RAG入門：LLMに外部情報を使わせる

## 1. RAGとは何か

RAGとは Retrieval-Augmented Generation の略で、日本語では「検索拡張生成」と呼ばれます。  
LLMが回答する際に外部の文書やデータベースを検索し、ヒットした情報をもとに回答を生成する仕組みです。

LLMは基本的に、学習時のコーパスに含まれていた情報をもとに回答します。  
そのため、学習後に公開された最新情報や、学習データに含まれていない社内資料・個人のローカル資料については、その内容について正確に答えられません。

第一回でも説明した通り、次のような質問にはLLM単体では限界があります。

* 今日公開されたニュースについて教えて
* 大量のPDFから必要な情報だけ探して
* このコードのDBの実装について教えて

そこで、必要な情報を外部から検索し、その検索結果をLLMに渡して回答させる必要があります。

## 2. RAGの基本フロー

RAGの基本フロー

1. ユーザーが質問
2. 質問をもとに検索クエリを作成  
   (ユーザーの質問をそのまま検索に使うこともあるが、検索しやすい形に書き換えることもあり)
3. 文書データベースやWeb、社内資料、コードベースなどから関連する情報を検索
4. 検索された文書の中から、質問に関係する部分を取り出し、LLMの入力に追加
5. ユーザーの質問と検索結果の両方を見ながら回答を生成

講義資料や小さいコードなら、全ての文書をLLMに渡して処理することは可能です。  
しかし、扱うデータが膨大になるとコンテキストが圧迫され、良い出力が得られなくなります。  
社内文書、過去の議事録、コードベースなど長大な文章を扱う場合、すべてを毎回LLMに渡すことは非常に無駄が多いです。  
そのため、必要な情報だけを検索して渡すRAGが重要になります。

## 3. キーワード検索とベクトル検索

RAGでは、ユーザーの質問に関係する文書を検索してLLMに渡します。そのため、検索方法がLLMの出力の質に大きな影響を与えます。  
検索方法には大きく分けて、キーワード検索とベクトル検索があります。

### キーワード検索

キーワード検索は、クエリに含まれる単語と、文書に含まれる単語の一致をもとに文書を探す方法です。  
例えば、

と検索したとき、文書中に `Cursor` という単語が含まれていれば、その文書は関連している可能性が高いと判断されます。

代表的な手法としては、TF-IDF や BM25 があります。

キーワード検索の強みは、固有名詞や専門用語、関数名などを探すときに強いことです。  
例えば、次のような検索ではキーワード検索が有効です。

* `Cursor`
* `電通`
* `GPT-4o`
* `OpenAI API`

これらは意味を広く解釈するよりも、文字列として一致しているかどうかが重要です。  
一方で、キーワード検索には弱点もあります。  
それは、**表現が少しでも変わると検索しづらくなる**ことです。  
例えば、文書には「新歓BBQ」と書かれていて、検索クエリでは

```
新しく入った人と既存メンバーが食事しながら仲良くなる企画
```

と入力した場合、「新歓」や「BBQ」という単語が一致しないため、キーワード検索ではうまく見つけられないことがあります。  
つまり、キーワード検索は「同じ単語が使われている文書」を探すのは得意ですが、「同じ意味のことを違う言い方で書いている文書」を探すのは苦手です。

### ベクトル検索

ベクトル検索は、文章を数値のベクトルに変換し、意味が近い文章を探す方法です。  
このとき使われるベクトル表現を **Embedding** と呼びます。  
Embeddingでは、文章の意味が近いものほど、ベクトル空間上でも近くに配置されるように学習されています。

例えば、

```
新しく入った人と既存メンバーが食事しながら仲良くなる企画
```

というクエリと、

```
新歓BBQを開催しました。学年を超えて気軽に話せる交流の場となりました。
```

という文書は、使っている単語はかなり違います。  
しかし、どちらも「新しい人と既存メンバーが交流するイベント」という意味を持っています。  
そのため、ベクトル検索ではこのような文書を見つけやすくなります。  
ベクトル検索の強みは、表現揺れに強いことです。  
例えば、次のような検索に向いています。

* 新入生向けのイベントはある？
* 他大学と共同でイベントを開催
* 年末に交流会をやった
* スタートアップイベントで団体紹介した

このように、文書と完全に同じ単語を使っていなくても、意味が近ければ検索できる可能性があります。  
一方で、ベクトル検索も万能ではありません。  
固有名詞、日付、型番、関数名のように、文字列そのものが重要な場合は、キーワード検索の方が有効なことがあります。  
例えば、

```
GPT-4o
search_ai_ken_docs
```

のような検索では、意味が近いかどうかよりも、その文字列が文書に含まれているかどうかが重要です。

### キーワード検索とベクトル検索の違い

まとめると、キーワード検索とベクトル検索には次のような違いがあります。

| 検索方法 | メリット | デメリット |
| --- | --- | --- |
| キーワード検索 | 固有名詞、専門用語、型番、関数名などを正確に探す | 言い換えや表現揺れに弱い |
| ベクトル検索 | 意味が近い文書を探す、言い換えに強い | 固有名詞や細かい文字列の一致には弱いことがある |

重要なのは、どちらか一方が常に優れているわけではないということです。  
RAGというと「ベクトル検索を使うもの」と思われがちですが、実際にはキーワード検索も非常に重要です。  
特にコード検索では、`grep` や `ripgrep` のようなキーワード検索が強力です。  
関数名、クラス名、ファイル名、エラーメッセージを探す場合、意味検索よりも文字列一致の方が有効な場面が多くあります。  
例えば、Claude Code や Codex のようなコードエージェントも、裏側ではコードベースを検索し、必要なファイルを読み、さらに関連する関数を探すという処理を行います。

実務ではキーワード検索とベクトル検索を組み合わせたハイブリッド検索がよく使われます。  
キーワード検索で固有名詞や正確な語を拾い、ベクトル検索で意味的に近い文書を拾うことで、より安定した検索が可能になります。

## 4. ワーク

ここからは、実際に検索を試してみます。

リポジトリ: <https://github.com/ReiMinamoto/waseda_ai_club/tree/main/contents/study/02-RAG>

今回は、AI研のHPをもとに作成した小さなデータセットを使います。  
このデータセットには、AI研の活動内容、理念、過去のイベント情報などが含まれています。

検索方法として、次の3つを比較します。

キーワード検索は、単語の一致をもとに検索します。  
ベクトル検索は、文章の意味の近さをもとに検索します。  
ハイブリッド検索は、その両方を組み合わせた検索方法です。

### 実行方法

今回の実習では、次のようなコマンドで検索できます。

```
uv run python waseda_ai_club_search.py "初心者がAI開発を体験できるイベント" --method keyword
uv run python waseda_ai_club_search.py "初心者がAI開発を体験できるイベント" --method vector
uv run python waseda_ai_club_search.py "初心者がAI開発を体験できるイベント" --method hybrid
```

`--method` を変えることで、検索方法を切り替えられます。

### キーワード検索が強い例

まず、固有名詞を含む検索を試してみます。

```
uv run python waseda_ai_club_search.py "フジテレビ" --method keyword
```

このような検索では、キーワード検索が強くなりやすいです。  
これは、`フジテレビ` という単語が文書中に含まれていれば、その文書はかなり高い確率で関連しているからです。  
同じように、次のような検索もキーワード検索に向いています。

固有名詞、イベント名、ツール名、企業名などは、意味の近さよりも文字列として一致しているかどうかが重要です。

### ベクトル検索が強い例

次に、言い換えを含む検索を試してみます。

```
uv run python waseda_ai_club_search.py "新しく入った人と既存メンバーが食事しながら仲良くなる企画" --method vector
```

このクエリには、`新歓BBQ` という単語は含まれていません。  
しかし、意味としては「新歓BBQ」にかなり近い内容です。  
このような場合、キーワード検索では単語が一致しないため、うまく探せないことがあります。  
一方で、ベクトル検索では文章全体の意味をもとに検索するため、交流イベントや新歓に関係する文書を見つけやすくなります。  
以下のケースでもうまくいくはずです。

```
新入生向けのイベントはある？
他大学と共同でイベントを開催
年末に交流会をやった
スタートアップイベントで団体紹介した
```

ただし、ベクトル検索も完璧ではありません。

例えば、このクエリで「新歓BBQ」が必ず1位になるとは限りません。  
「交流」「年末交流会」など、意味的に近い別の文書が上位に来ることもあります。  
これは、ベクトル検索があくまで「意味が近そうな文書」を探しているからです。

### 検索結果を見るときのポイント

検索結果を見るときは、単に1位だけを見るのではなく、次の点を確認するとよいです。

* 目的の文書は上位に入っているか
* 関係ない文書がどれくらい混ざっているか
* なぜその文書が上位に来たのか
* キーワード検索とベクトル検索で順位がどう変わったか

RAGでは、検索結果がそのままLLMの回答品質に影響します。  
そのため、単に「検索できたか」ではなく、**必要な情報をどれだけ上位に、どれだけ少ないノイズで取得できるか**が重要になります。

## 5. RAGをうまく動かすための工夫

ここまでで、RAGでは「検索してLLMに渡す」ことが重要だと説明しました。

しかし、実際のRAGシステムでは、単に検索するだけではうまくいかないことも多いです。

RAGを実用的に動かすには、いくつかの工夫が必要になります。

### Chunking

Chunkingとは、長い文書を小さな単位に分割することです。

例えば、100ページのPDFをそのまま1つの文書として扱うと、検索結果として返ってきても、どこに必要な情報があるのか分かりにくくなります。

そこで、文書を段落単位、見出し単位、一定の文字数単位などに分割します。

```
長いPDF
↓
チャンク1
チャンク2
チャンク3
...
```

こうすることで、質問に関係する部分だけを検索しやすくなります。

ただし、チャンクは小さければよいわけではありません。

小さすぎると前後の文脈が失われます。  
大きすぎると関係ない情報まで一緒に入ってしまいます。

そのため、文書の性質に応じて、どのくらいの単位で分割するかを考える必要があります。

### Rerank

検索で取得した文書が、必ずしも最適な順番で並んでいるとは限りません。

そこで、一度検索した結果を、別のモデルやルールで並び替えることがあります。  
これを Rerank と呼びます。

例えば、最初にキーワード検索やベクトル検索で候補を20件取得し、その後に「この質問に本当に関係しているか」をより丁寧に判定して、上位5件だけをLLMに渡す、といった使い方をします。

```
検索で候補を多めに取得
↓
関連度が高い順に並び替え
↓
上位だけをLLMに渡す
```

RAGでは、最初の検索だけで完璧な結果を出すのは難しいです。  
そのため、検索とRerankを組み合わせることで、より安定した結果を目指します。

### Context圧縮

検索結果をそのまま全部LLMに渡すと、入力が長くなりすぎることがあります。

そこで、検索された文書の中から、質問に関係する部分だけを抜き出したり、要約したりします。  
これを Context圧縮 と呼びます。

例えば、ある文書全体ではなく、該当する段落だけを渡す。  
あるいは、複数の文書から重要な部分だけを取り出して、短くまとめて渡す。

このようにすることで、LLMが不要な情報に惑わされにくくなります。

RAGでは「たくさん情報を渡せばよい」わけではありません。  
LLMにとって必要な情報を、なるべく分かりやすい形で渡すことが重要です。

### 出典管理

RAGでは、どの文書を根拠に回答したのかを管理することも重要です。

例えば、次のような情報を検索結果に含めておくと、回答の根拠を示しやすくなります。

```
doc_id
title
section
url
content
score
```

出典が残っていれば、LLMの回答に対して「どの文書をもとにそう言っているのか」を確認できます。

これは、社内文書、法律文書、論文、決算資料などを扱う場合には特に重要です。

RAGの目的は、ただそれっぽい回答を作ることではありません。  
根拠となる情報に基づいて、確認可能な回答を作ることです。

### 再検索

1回の検索だけで十分な情報が得られるとは限りません。

例えば、最初に「AI研の初心者向け活動」を検索したとします。  
その結果から「AIアプリ作成体験会」という活動が見つかったとしても、さらに詳しく知りたい場合は、次にそのイベント名で追加検索する必要があります。

```
初心者向け活動を検索
↓
AIアプリ作成体験会を発見
↓
AIアプリ作成体験会について追加検索
```

このように、検索結果を見て次の検索を決めることがあります。

これは、次の章で扱う Agentic RAG につながる考え方です。

## 6. エージェントとRAG

ここまで説明したRAGは、基本的には

という1回の流れでした。

しかし、実際の調査や開発では、1回検索して終わりということはあまりありません。

人間が調べ物をするときも、最初の検索結果を見て、足りない情報があれば追加で検索します。  
関連する資料を開き、さらに別のキーワードで調べ、必要な情報を少しずつ集めていきます。

AIエージェントでも同じことができます。

```
質問
↓
検索
↓
検索結果を読む
↓
足りない情報を判断
↓
追加検索
↓
回答
```

このように、LLMが検索結果を見ながら次の行動を決めるRAGを、Agentic RAGと呼ぶことがあります。

### コーディングエージェントもRAGに近い

例えば、Claude Code や Codex のようなコーディングエージェントを考えてみます。

ユーザーが

と質問したとします。

このとき、エージェントはコードベース全体を最初からすべて読むわけではありません。

まず、ファイル名や関数名、エラーメッセージなどを手がかりに検索します。

```
grep
↓
関連ファイルを読む
↓
関数名で追加検索
↓
関連する実装を読む
↓
回答・修正
```

これは、検索対象がWebやPDFではなくコードベースになっただけで、広い意味ではRAGに近い処理です。

特にコード検索では、ベクトル検索よりも `grep` や `ripgrep` のようなキーワード検索が強い場面も多いです。

関数名、クラス名、ファイル名、エラーメッセージのように、文字列そのものが重要な情報だからです。

ここからも分かるように、RAGはベクトル検索だけを指すものではありません。

必要な情報を探して、LLMが使える形で渡す仕組み全体がRAGです。

### MCPとRAG

MCPは、LLMやAIエージェントが外部ツールを呼び出すための仕組みです。

例えば、今回の実習で使った検索スクリプトを、MCPのツールとして公開することを考えます。

```
search_ai_ken_docs(query, method, top_k)
```

このようなツールを用意すると、AIエージェントは必要に応じてAI研HPのデータを検索できるようになります。

今は人間がコマンドラインから検索しています。

```
uv run python waseda_ai_club_search.py "初心者が参加しやすい活動" --method hybrid
```

しかし、MCP化すると、AIエージェント自身が次のような検索ツールを呼び出せるようになります。

```
search_ai_ken_docs(
  query="初心者が参加しやすい活動",
  method="hybrid",
  top_k=5
)
```

その結果を読んで、必要なら追加で検索し、最終的に根拠付きで回答します。

つまり、MCPはRAGをエージェントに組み込むための手段の1つだと考えることができます。

検索関数を単なるスクリプトとして使うのではなく、LLMが呼び出せるツールとして公開することで、エージェントが自分で情報を取りに行けるようになります。

## 7. まとめ

今回の講義では、RAGの基本的な考え方と、キーワード検索・ベクトル検索の違いについて学びました。

RAGは、LLMに単に知識を追加する技術ではありません。

RAGの本質は、**必要な情報を探し、LLMが使いやすい形で渡す設計**にあります。

そのためには、検索方法の違いを理解することが重要です。

キーワード検索は、固有名詞、ツール名、関数名、型番など、文字列そのものが重要な情報を探すのに向いています。

一方で、ベクトル検索は、表現が違っていても意味が近い文書を探すのに向いています。

実際のRAGシステムでは、どちらか一方だけを使うのではなく、両方を組み合わせることが多いです。

また、RAGをうまく動かすには、検索だけでなく、Chunking、Rerank、Context圧縮、出典管理、再検索といった工夫が必要になります。

さらに、AIエージェントでは、1回検索して終わりではなく、検索結果を読んで、必要に応じて追加検索することが重要になります。

Claude Code や Codex のようなコードエージェントも、コードベースから必要な情報を探し、読んで、判断するという意味では、RAGに近い仕組みを使っています。

最終的に覚えておいてほしいのは、次の3点です。

* RAGは「知識を足す技術」ではなく「情報を探して渡す設計」
* キーワード検索とベクトル検索には、それぞれ得意・不得意がある
* AIエージェントを作る上で、RAGは情報取得の中核になる

LLMをうまく使うには、モデルそのものの性能だけでなく、**何を検索し、何を渡し、何を根拠に回答させるか**を設計することが重要です。
