## はじめに

前回の記事で、AIエージェントを使ったWeb開発において、HW設計の仕様書ドリブンなワークフローが有効に機能したことを書きました。

今回は、その開発で意図的に使い分けた2つのツールチェーン、**Claude Desktop + Claude Code** と **ChatGPT + Codex** の比較所感を書きます。

AIエージェントを使うのはこれが初めてでした。初体験だからこそ1つのツールに閉じず、同じワークフローを2系統に適用して、違いを体感したいと考えました。

* **AILBA（AI Log Brain Agent）**: Claude Desktop で仕様策定 → Claude Code で実装
* **MSH（Manufacturing Service Hub）**: ChatGPT で仕様策定 → Codex で実装

どちらも、壁打ちで仕様を固め、プロンプト（≒仕様書）をAIと一緒に作成し、それをエージェントに投入するという同じプロセスで進めました。

---

## 仕様策定フェーズ：壁打ちの感触

### ChatGPT：テンポが速い

ChatGPTとの仕様策定は、会話のテンポが速いと感じました。

こちらが要件を伝えると、すぐに「次はこれを決めましょう」「この部分はどうしますか？」と、次のステップを提示してきます。壁打ちの相手としてテンポよく仕様を詰めていける感覚がありました。

バイブコーディング的なアプローチ――アイデアをぶつけながら、勢いで形にしていく――にはChatGPTの方が向いていると感じました。

### Claude Desktop：構造的に整理してくれる

Claude Desktopとの仕様策定は、ChatGPTとは少し違う感触でした。

テンポの速さよりも、こちらが伝えた要件を構造的に整理し直してくれる印象が強かったです。機能要件を階層化したり、エッジケースを体系的に洗い出したりする場面で、HW設計の仕様レビューに近い丁寧さを感じました。

どちらが良い・悪いではなく、仕様策定の進め方のスタイルが異なるという印象です。

---

## 実装フェーズ：エージェントの挙動の違い

### Codex：テストコードを必ず書く

Codexに実装を任せて印象的だったのは、**特に指示しなくてもテストコードを生成してくれた**ことです。

正直なところ、当時はAIが生成したテストコードの品質を正確に評価できるほどの知見はありませんでした。ただ、テストが存在すること自体が安心材料になりました。少なくとも「テストが一切ない状態」よりは、コードの信頼性を担保する一つの層ができたと感じています。

MSHは製造トレーサビリティという業務システムで、機能的にもAILBAより複雑でした。テストコードが自動的に付いてくるのは、複雑なシステムの開発においてはありがたい挙動でした。

### Claude Code：プランモードとサブエージェント化

Claude Codeで驚いたのは、**プランモード**の存在です。

プロンプトを投入すると、Claude Codeがまず実装計画を立て、タスクを分割して提示してくれます。AILBAの開発では、フロントエンドとバックエンドを自動的に分割し、それぞれを順番に実装していく動きを見せました。

これは、ある種のサブエージェント化です。「まずAPIのエンドポイントを設計・実装し、次にフロントエンドのコンポーネントを構築する」といった形で、自律的にタスクを分解して進めてくれました。

HW設計でいえば、トップダウン設計でモジュール分割してから各モジュールを実装するアプローチに近い。仕様書ドリブンの思考に慣れた自分にとっては、Claude Codeの進め方の方が構造的に理解しやすいと感じました。

## 比較まとめ

2026年2月時点での所感を整理します。

**仕様策定（壁打ち）フェーズ**

ChatGPT：テンポが速い。「次は？次は？」と前に進める推進力がある。バイブコーディング的なアプローチに向いている。

Claude Desktop：構造的に整理してくれる。仕様の階層化やエッジケースの体系的な洗い出しに強い。仕様レビューに近い丁寧さがある。

**実装（エージェント）フェーズ**

Codex：指示しなくてもテストコードを生成する。実装のテンポが速い。

Claude Code：プランモードでタスクを自動分割。フロントエンド/バックエンドをサブエージェント的に分けて実装。実装の構造が見えやすい。

**共通して言えること**

どちらの系統でも、「仕様を先に固めてプロンプトとして投入する」ワークフローは有効でした。仕様の質がアウトプットの質を左右する構造は、ツールが変わっても同じです。

## 当時と今の差分

正直に書きます。

2026年2月の時点では、AIエージェントとの協働は完全に手探りでした。テストコードの品質を評価する知見も、エージェントの挙動を細かく制御するスキルもありませんでした。壁打ちで仕様を決め、プロンプトを作成してもらい、それを投入する。当時はそれが精一杯でした。

今はだいぶ違います。

Claude Codeのスキル機能やフック、CLAUDE.mdによるプロジェクトルールの設定など、エージェントの挙動を制御する仕組みを理解してきました。テストの指示も具体的に出せるようになりましたし、プロンプトの構造化もより精緻にできるようになっています。

同じシステムを今作り直したら、よりロバストなエージェントコーディングができると確信しています。

この差分こそが、「まず動かしてみる」ことの価値だったと思います。完璧な知識を身につけてから始めるのではなく、手探りでも実際に2本のシステムを作りきったことで、AIエージェントとの協働の勘所が身につきました。

## シニアエンジニアへ：正直に話します

ここから少し、同世代のエンジニアに向けて書かせてください。

昨今、AI系YouTuberの動画は、AIのトレンドをキャッチアップするのに非常に有効です。新しいモデルのリリース、ツールの使い方、プロンプトの技法。情報は大量にあります。

ただ、すべてのシニアエンジニアがそれをフォローできているかというと、正直、疑問です。

私自身の話をします。経歴はハードウェア設計が主ですが、30年の中でそれなりにソフトウェア開発もやってきました。C、C++、C#。設計の現場で必要なツール類は自分で書いてきました。

それでも、設計エンジニアとしてはもはやロートルです。最新のフレームワークを追いかけているわけでもなければ、モダンなWeb開発の作法を熟知しているわけでもありません。

でも、今回の経験で一つ確信したことがあります。

**AIを活用すれば、エンジニアビギナーのAIヘルパーにはまだ勝てます。**

若手がAIに「とりあえずこれ作って」と投げて返ってきたコードをそのまま使うのと、30年の設計経験を持つエンジニアがAIに仕様を渡して作らせるのとでは、アウトプットの質が違います。入出力の定義、例外パスの洗い出し、実用レベルの見極め。これらは何十年もの現場経験がないと出てこない判断です。

問題は、その武器を使っていないことです。

AI系のYouTubeやブログを見て「すごいな」と思うだけで止まっていませんか。「自分の領域はまだ関係ない」と思っていませんか。

関係あります。というより、ハードウェアの深い知識を持つシニアエンジニアこそ、AIエージェントと組んだときに最も力を発揮できる立場にいます。なぜなら、AIに仕様を定義する力が、そのままAIのアウトプットの品質を決めるからです。

ロートルだからこそ強い。ただし、動かなければその強さは伝わりません。
