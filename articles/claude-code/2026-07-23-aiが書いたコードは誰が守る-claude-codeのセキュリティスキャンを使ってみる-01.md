## 

AIにコードを書かせると、開発のスピードは一気に上がります。けれども、速く書けるということは、危ないコードも速く紛れ込むということでもあります。パスワードの直書き、外部からの入力をそのまま実行してしまう処理、古い書き方による脆弱性——こうした問題は、悪意がなくても自然に入り込んでしまいます。

Anthropicは、この課題に対して「コードを書くAI」と「そのコードを点検するAI」を組み合わせる方向で機能を整えてきました。その中心にあるのが、Claude Codeに追加できる **Claude Securityプラグイン** です。この記事では、その仕組みと使い方、そして周辺のセキュリティ機能との関係を整理します。

---

## Claude Securityプラグインとは何か

Claude Securityプラグインは、Claude Codeのセッションの中で **複数のAIエージェントがチームを組み、あなたのコードの脆弱性を探し出す** 機能です。ひとつのエージェントが全部を担うのではなく、役割を分けて動くのが特徴です。

流れはこうです。まずエージェントが全体の構造を把握し、次に「どこが狙われやすいか」という脅威モデルを組み立てます。その上で脆弱性を探し、見つけた候補は **別のエージェントが独立して検証してから** レポートに載せます。人間のセキュリティ研究者が、まず設計を理解してから弱点を探すのに近いやり方です。

このプラグインはあなたの手元（ローカル）のセッションで動き、スキャンにかかる分はプランの利用枠から消費されます。企業向けには、接続したリポジトリを継続監視する **マネージド版のClaude Security** も別に用意されていますが、プラグイン版はGitLabやBitbucket、外部から接続できない社内ネットワーク上のコードなど、マネージド版が届かない場所にも使えるという利点があります。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

<https://claude.com/product/claude-security>

## 使い始める前に必要なもの

導入のハードルは高くありません。必要なのは次のとおりです。

* 有料プランの **Claude Code v2.1.154以降**（スキャンが使う「動的ワークフロー」のため）
* `python3` として使える **Python 3.9.6以降**（追加インストールは不要で、標準ライブラリのみ使用）
* Linux / macOS / Windows のいずれか
* 変更点のスキャンやパッチ作成には **Git** が必要（リポジトリ全体のスキャンだけならGitなしでも可）

インストールは、Claude Codeのセッション内で次の1行を実行するだけです。

`/plugin install claude-security@claude-plugins-official`

そのあと `/reload-plugins` を実行すると、再起動せずにその場で有効になります。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

## 3つの仕事を、1つのコマンドで

プラグインを入れると `/claude-security` というコマンドが1つ増えます。ここから **「コードベース全体のスキャン」「変更点だけのスキャン」「パッチの提案」** という3つの仕事を選べます。基本の流れは、全体をスキャンして、出てきた問題をパッチに変える、というものです。

うれしいのは、変更点だけを絞ってスキャンできる点です。あるブランチにコミットがあれば、その差分だけを調べられるので、**マージする前のブランチをチェック** できます。オープン中のプルリクエストや、「commit abc1234をスキャンして」といった特定のコミット単位の指定も可能です。ただし調べられるのはコミット済みの変更だけなので、作業中の編集はいったんコミットするか、作業ツリーごと読む全体スキャンを使います。

大きなリポジトリでは、木全体を一度に見るのではなく、「API層だけ」「認証まわりだけ」といったように **一部分ずつ** スキャンするのがコツです。何を調べて何を調べなかったかは、レポートのカバレッジ欄に明記されます。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

## スキャン結果はどこに残るのか

スキャンをすると、リポジトリの中に日時つきのフォルダ（`CLAUDE-SECURITY-<日時>/`）が作られ、そこに結果が書き出されます。中身は主に次の3つです。

* **結果レポート（Markdown）**：見つかった問題に `F1` のようなIDが振られ、影響・悪用シナリオ・深刻度・確信度・推奨対応がまとまっている
* **機械可読の結果（JSONL）**：同じ内容を1行1件のJSONで
* **リビジョン記録（JSON）**：どのコミットを、どの精度で、どこまで検証してスキャンしたかを刻印。レポートが「どのコードの話なのか」を常にたどれる

このフォルダには専用の `.gitignore` が入っているので、うっかり `git add` してもレポートがコミットに紛れ込みません。逆に、監査の記録として残したいときは、その `.gitignore` を消してコミットすればよいだけです。

ひとつ知っておきたいのは、**スキャンには「ゆらぎ」がある** ことです。同じコードでも2回のスキャンで違う指摘が出ることがあります。だからこそ定期的に走らせ、リビジョン記録でどのコード・設定の結果かを結びつけておくのが安心です。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

## 見つかった問題を「パッチ」に変える

指摘を読んだら、次は修正です。`/claude-security` のメニューから「Suggest patches（パッチの提案）」を選ぶか、「F3を直して」と普通の言葉で頼めば、どの指摘を直すか選べます。

ここで丁寧なのは、**元のソースには一切触れず、複製したコピーの上でパッチを作る** 点です。しかも、パッチを書いたエージェントとは別のエージェントがレビューし、テストがあれば実際に走らせ、差分を独立した目で読み直します。「その1件の問題を確かに直している」「新しい脆弱性を持ち込んでいない」「それ以外の動きは変えていない」——この3つを保証できたときだけパッチが作られ、保証できないときは理由を添えた短いメモが返ってきます。

そして最も大事なルールがこれです。**パッチが自動で適用されることは絶対にありません。** 適用するかどうかは常にあなたの判断で、自分のシェルから次のように実行します。

`git apply CLAUDE-SECURITY-<日時>/patches/F1.patch`

しかも「1つのパッチにつき1つのプルリクエスト」で適用するのが推奨です。こうすれば、それぞれを個別にレビュー・テストできます。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

## 信頼できないコードを調べるときの注意

このプラグインは、基本的に **自分が管理しているコード** を対象に設計されています。「このコードにどんなバグがあるか」を問う道具であって、「このコードが悪さをしようとしているか」を防ぐ道具ではない、という点は押さえておきましょう。

スキャンはあなたのセッション・あなたの権限で動き、それ自体は追加の隔離をしません。つまり、リポジトリに含まれる `.claude/` の設定やフック、`CLAUDE.md` は、普段のセッションと同じように効いてしまいます。プラグインはコードやコメント、指摘の文面を「命令」ではなく「点検対象のデータ」として扱いますが、それは悪意あるリポジトリへの防御そのものではありません。

だから、第三者の依存パッケージや素性のわからないコードを調べたいときは、**セッションごとサンドボックス（隔離環境）に入れてから** 動かすのが安全です。Anthropicが実験的に公開している `sandbox-runtime` は、OSレベルでファイルアクセスや通信を制限してくれます。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

<https://github.com/anthropic-experimental/sandbox-runtime>

## 「深いスキャン」は多層防御の一枚

Claude Securityプラグインは万能の一手ではなく、いくつかの守りを重ねる「多層防御」の中の一枚です。段階ごとに、次のような道具が並びます。

* **書いている最中**：セキュリティ・ガイダンスプラグインが、Claudeが書いたコードの中のよくある脆弱性をその場で警告し、同じセッションで直す
* **必要なときに1回**：`/security-review` コマンドが、今のブランチにセキュリティの一発チェックをかける
* **必要なときに深く**：今回のClaude Securityプラグインが、リポジトリや差分を多エージェントで深くスキャンし、独立レビュー済みの指摘とパッチを出す
* **プルリクエスト時**：Code Review（TeamおよびEnterpriseプラン）が、コードベース全体を踏まえて正しさとセキュリティを多エージェントでレビュー
* **マネージドで**：Claude Security（Enterpriseプラン）が、接続したリポジトリを継続的に監視
* **CIの中**：これまで使ってきた静的解析ツールや依存関係スキャナが、言語ごとのルールやサプライチェーンのチェックを担う

大事なのは、**プラグインが既存のツールを置き換えるわけではない** ことです。人間の研究者のように「コードの意味」を推論するのが得意なプラグインと、機械的で確実な既存ツールは、互いを補い合う関係にあります。

ちなみに、いちばん軽い「書いている最中」の層を担うセキュリティ・ガイダンスプラグインは無料で全プラン向けに提供され、ファイル編集のたびに危険なパターン（`eval()`、`os.system()`、危険なデシリアライズ、XSSの入口など）を検知します。まずはこの軽い層から試し、腰を据えて調べたいときに深いスキャンを足す、という順番が現実的です。

**根拠：**

<https://code.claude.com/docs/en/claude-security>

<https://claude.com/plugins/security-guidance>

## まとめ：AIに書かせるなら、AIにも点検させる

AIがコードを書くのが当たり前になるほど、「そのコードは安全か」を確かめる工程の重みは増していきます。Claude Securityプラグインは、その点検を **複数のAIエージェントの協働** として、しかも **最後は必ず人間が判断する** かたちで組み込むための道具です。

「速く書く」と「安全に書く」は、本来トレードオフではありません。書く力と点検する力を両方AIに持たせ、最終判断を人間が握る——その設計思想を、実際に手を動かしながら体験できる機能だと言えます。
