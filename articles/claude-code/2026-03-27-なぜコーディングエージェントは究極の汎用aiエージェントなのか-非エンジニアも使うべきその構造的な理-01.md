## はじめに

2025年後半から2026年にかけて、「非エンジニアが Claude Code で業務を自動化した」「コーディングエージェントでスライド作成や経費精算までやっている」といった記事が急増している。Zenn、Qiita、note、個人ブログなど、あらゆるプラットフォームでコーディングエージェントの「コーディング以外」の活用事例が共有されている。（中には高額の有料noteを量産している層もいて正直げんなりするが、それはさておき……）

これらの記事の多くは「こんなこともできた」という体験報告だ。「なぜコーディングエージェントがそこまで汎用的なのか」を構造的に説明した記事は少ないと感じる。

本記事では、AIエージェントのアーキテクチャという視点から、コーディングエージェントが他のどのタイプのAIエージェントよりも汎用的である構造的理由を論じる。これは僕個人の主張であり、やや大げさな物言いであることは自覚している。が、コーディングエージェントを使いこなしているエンジニアはみんな同じ感覚を持ってるんじゃないかと思うし、なんなら「何をそんな今更」という声も聞こえてきそうだ。

この話は、エンジニアだけに閉じた話ではない。むしろ、エンジニアではない人にこそ知ってほしい話だ。

## 第1章：AIエージェントはどうやって「外の世界」とつながるのか

AIエージェントの汎用性を決定づけるのは、「外部とどれだけ多様な方法で連携できるか」だ。ここでは、チャット型エージェントとコーディングエージェントのアーキテクチャを比較する。

### チャット型エージェント（ChatGPT、Gemini、通常の Claude など）

チャット型エージェントが外部と連携する手段は、基本的に以下の2つに限られる。

1. **組み込みツール** … Web検索、画像生成、コード実行（サンドボックス）など、プロバイダーが事前に用意した機能
2. **MCP（Model Context Protocol）やプラグイン** … 外部開発者が作成したツールを接続する仕組み

つまり、チャット型エージェントにとっての「できること」は、**誰かが事前に用意したツールの範囲**に制約される。新しいAPIと連携したければ、誰かがそのMCPサーバーを作ってくれるのを待つ必要がある。

### コーディングエージェント（Claude Code、Cursor、Kiro CLI、Gemini CLI など）

コーディングエージェントは、上記の手段に加えて、さらに2つの強力な連携手段を持つ。

3. **CLI（コマンドラインインターフェース）経由の実行** — ターミナルから叩けるコマンドは何でも実行できる。`curl` でAPIを叩く、`ffmpeg` で動画を変換する、`git` でバージョン管理する、`aws` CLIでクラウドリソースを操作する……といった、 OS が実行可能な全てが射程範囲に入る。
4. **コードの生成と実行** — 必要なプログラムをその場で書いて、その場で実行できる。Pythonスクリプトでデータを加工する、Node.jsでAPIクライアントを書く、シェルスクリプトで複雑なワークフローを組む……といった形で、プログラミング言語で表現できることはすべて実行可能になる。

つまり、コーディングエージェントは、チャット型エージェントができることをすべてできる上に、CLI実行とコード生成・実行という追加の能力を持つ。**コーディングエージェントの能力は、チャット型エージェントの完全な上位互換**だ。

## 第2章：「コードが書ける」とはどういうことか — MCP vs CLI 論争のその先

2025年後半から2026年にかけて、エージェント開発者の間で「MCP vs CLI」が最もホットな論争になっている。この論争を追うと、コーディングエージェントの構造的優位性が見えてくる。

### MCP vs CLI — エージェントのツール連携をめぐる論争

2024年11月にAnthropicがMCP（Model Context Protocol）を発表して以来、各社がMCPサーバーを競って実装した。しかし2025年後半から風向きが変わり始める。

きっかけの一つは、Mario Zechner氏による[ベンチマーク](https://mariozechner.at/posts/2025-08-15-mcp-vs-cli/)だ。Playwright MCPの21ツール定義が13,700トークンを消費するのに対し、同等機能のCLIツールのREADMEはわずか225トークン。ブラウザ操作のテストでは、MCPのアクセシビリティツリーが52,000トークンを食う一方、CLIの選択的クエリは1,200トークン。43倍の効率差だ。

2026年2月にはEric Holmesが「[MCP is dead. Long live the CLI](https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html)」を公開し、Hacker Newsで大きな反響を呼んだ。彼の指摘で特に効いているのは、「ほとんどのMCP実装は結局CLIツールをラップしているだけ」という点だ。GitHub MCPサーバーはGitHub CLIの再実装であり、ファイルシステムMCPサーバーは標準のファイル操作コマンドのラッパーにすぎない。抽象化のレイヤーを足した結果、トークンコストが増え、認証が複雑になり、デバックが困難になっている。

日本語圏でも、hiraly氏がZennで「[MCPはなぜCLIに負けたのか](https://zenn.dev/hiraly/articles/3409b886607274)」という包括的な分析記事を公開し、X上でもMCP不要論をめぐる議論が盛り上がった。

もちろん、MCPにも利点はある。企業の社内システムをAI対応させる際のガバナンスや認証の一元管理など、MCPが適切な領域は確実に存在する。ただし、この話は本記事の主題から外れるので、別の機会に譲りたい。

### Anthropic自身がMCPの限界を認めている

MCPの設計者であるAnthropic自身が、MCPのスケーリング問題を公式に認めている。[同社のエンジニアリングブログ](https://www.anthropic.com/engineering/code-execution-with-mcp)によれば、社内ではツール定義だけで134,000トークンを消費するケースがあった。そして彼らが提案した対策は興味深いことに、MCPサーバーをコードAPIとして提示し、**エージェントがツールを直接呼ぶ代わりにコードを書いてやり取りする方式**だ。この方式により、従来150,000トークンかかっていたワークフローが約2,000トークンにまで削減されたという。

つまり、MCPの設計者自身が「コードを書いてやり取りする方が効率的」と認めたわけだ。これは、コーディングエージェントのアプローチが構造的に正しいことの、ある意味でもっとも強力な傍証だと思う。

### コーディングエージェントは「橋をその場で架ける」

この論争を踏まえると、コーディングエージェントの優位性がより鮮明になる。MCPは素晴らしい仕組みだが、**誰かが事前にツールを作っておく必要がある**という本質的な制約がある。ニッチなSaaSにMCPサーバーが存在しなければ、チャット型エージェントはそのサービスと連携できない。

コーディングエージェントなら、MCPサーバーが存在しなくても、APIドキュメントを読んでその場でコードを書いて実行できる。**MCPツールそのものをその場で作る**ことすらできる。MCP vs CLI の論争は「どちらのツール連携方式が優れているか」という議論だが、コーディングエージェントはその両方を使える上に、さらにコードも書ける。この論争自体が、コーディングエージェントの構造的優位性を浮き彫りにしている。

### 「プログラムで表現できること」の射程

プログラミング言語で表現できることの範囲は、事実上無限に近いと思う。

* 任意のHTTP APIとの通信
* ファイルシステムの操作（読み書き、変換、整理）
* データの加工・分析・可視化
* 外部コマンドの実行と結果の解釈
* 複雑な条件分岐やループを含むワークフロー
* データベースへの接続とクエリ

チャット型エージェントが「事前に用意されたツールの数」で汎用性が決まるのに対し、コーディングエージェントの汎用性は「プログラミング言語で表現できること」で決まる。

「それなら ChatGPT の Code Interpreter でもいいのでは？」と思うかもしれない。確かに Code Interpreter もコードを書いて実行できる。しかし、コーディングエージェントとは決定的な差がある。

* **再現性** … Code Interpreter のコードはサンドボックス内で消える。コーディングエージェントが書いたコードはファイルとしてローカルに残るので、同じ処理を何度でも再実行できる。
* **トレーサビリティ** … git で差分管理できるので、「いつ、何を、なぜ変えたか」が追える。問題が起きたときのトラブルシューティングも容易になる。
* **外部コマンドの実行** … Code Interpreter は Python サンドボックスに閉じている。`git`、`aws`、`curl`、`ffmpeg` といった OS 上のコマンドは叩けない。
* **ファイルシステムへの永続的アクセス** … 既存のプロジェクトや設定ファイルを直接読み書きできる。Code Interpreter の一時ファイルとは根本的に違う。
* **複数ステップの自律的な実行** … 「ファイルを読んで→コードを書いて→実行して→結果を見て修正」というループを自律的に回せる。Code Interpreter は基本的に1回の実行で完結する。

Code Interpreter は「コードを実行できるチャット型AI」であり、コーディングエージェントは「コンピュータ全体を操作できるAI」だ。この差は大きい。

## 第3章：「Bash is all you need」— 世界はもう気づき始めている

この「コーディングエージェント＝究極の汎用エージェント」という認識は、僕だけの妄想ではない。国内外で、同じ方向を指す議論が同時多発的に起きている。

### 海外の動向

**Vercel の「Bash is all you need」**

Vercelのソフトウェア責任者 Andrew Qu 氏は、AIエージェントの開発において興味深い発見をしている。当初、エージェントに専用ツールを大量に用意し、プロンプトエンジニアリングを重ね、メタデータを充実させていた。しかし、逆のアプローチ、つまりエージェントにbashの実行能力だけを与える方がうまくいったという。

> "We were building custom tools for what Unix already solves"

この知見は [The New Stack の記事](https://thenewstack.io/the-key-to-agentic-success-let-unix-bash-lead-the-way/)で紹介され、大きな反響を呼んだ。

**[Reddit「Bash is all you & agent need」](https://www.reddit.com/r/ClaudeAI/comments/1ql4t00/)**

Redditでは「エージェントにbashという1つのツールだけ与えれば、何でもできる。Bash is already a universal interface」という議論が盛り上がった。16行のPythonコードでClaude Codeのようなエージェントが作れるというデモも共有されている。

**[CLI-Anything（香港大学）](https://github.com/HKUDS/CLI-Anything)**

香港大学のプロジェクト「CLI-Anything」は、「Making ALL Software Agent-Native」を掲げ、GIMP、Blender、Audacity、LibreOffice、OBS Studio など、あらゆるGUIソフトウェアにCLIハーネスを被せてエージェントから操作可能にしている。これは「CLIで叩ければエージェントが使える」という原理の実践的な証明だ。

### 国内の動向

日本語圏では、「構造的理由」を論じた記事は少ないものの、体験的に同じ結論に到達している人は多い。MCP vs CLI の議論は日本語圏にも波及しており、MCPの優位性が失われた経緯の分析が提示されている 。

コーディングエージェントの「コーディング以外」の活用については、勉強・記事執筆・資料作成への活用 や、経費精算・スライド作成・メールリマインドまでの日常業務自動化 といった実践記事が増えている。書籍『実践Claude Code入門』の著者は「Claude Codeは、あらゆる業務を遂行する汎用エージェントとしても活用できます」と明言しており 、国内で「汎用エージェント」という言葉を使った最も明確な記事の一つだ。

2026年に入ってからは、非エンジニアによる「何でもできてしまう」という体験談 や「もうエンジニアだけのものじゃない」という主張 が note を中心に急増している。体験的に「何でもできる」と気づいた人が増えている。

## 第4章：それでも残る限界 — コーディングエージェントは万能ではない

公平を期すために言っておくと、コーディングエージェントは万能ではない。

### GUIしか存在しないシステム

CLIやAPIが提供されていないシステムに対しては、コーディングエージェントも直接操作できない。ただし、コーディングエージェント自身が Playwright スクリプトを書いてブラウザを操作するアプローチが実用化されており、この制約は急速に縮小している。

### リアルタイム性・インタラクティブ性

人間とのリアルタイムな対話が主目的の場合（カスタマーサポートのチャットボットなど）、コーディングエージェントの「コードを書いて実行する」というループは遅すぎる。この用途ではチャット型エージェントの方が適切だ。

### セキュリティリスク

CLI実行やコード実行ができるということは、裏を返せば破壊的な操作も可能だということだ。`rm -rf /` のような事故や、意図しない外部通信のリスクは、チャット型エージェントよりも高い。権限管理と承認フローの設計が不可決になる。

### 利用のハードル

現時点では、多くのコーディングエージェントはターミナル操作を前提としている。非エンジニアにとってのハードルは依然として高い。ただし、Anthropic の Cowork のようなデスクトップアプリ化の動きもあり、このハードルは急速に下がりつつある。

## 第5章：だから、エンジニアじゃなくてもコーディングエージェントを使うべき

「でも、自分はエンジニアじゃないし……」と思うかもしれない。しかし、ここが重要なポイントだ。**あなたがコードを書く必要はない。コーディングエージェントがコードを書くのだ。**

あなたがやることは、自然言語で「こういうことがしたい」と伝えることだけだ。エージェントが必要なコードを書き、必要なコマンドを実行し、結果を返してくれる。あなたにとってのインタフェースは、チャット型AIと何も変わらない。違うのは、その裏側で動いている「できることの幅」だけだ。

コーディングエージェントは、名前に「コーディング」と付いているせいで、エンジニアのためのツールだと思われがちだ。しかし、その本質は「**プログラミングという万能の道具を使いこなせるAIエージェント**」であり、その恩恵を受けるのにプログラミングの知識は必要ない。

僕は、コーディングエージェントこそが現時点で最も汎用的なAIエージェントだと考えている。そしてそれは、エンジニアだけでなく、すべてのビジネスパーソンにとっての話だ。

## おわりに

「Bash is all you need」や「MCP is dead. Long live the CLI」という声が出てきている。MCPの設計者であるAnthropic自身が、MCPの回避策として「コードを書いてやり取りする方式」を提案している。

僕が本記事で主張したいのは、これらの議論のさらに先にある話だ。bashだけでなく、コードを書いて実行できるということは、**まだ存在しないツールをその場で生み出せる**ということだ。MCP vs CLI の論争は「既存のツールをどう繋ぐか」の話だが、コーディングエージェントは「ツールが無ければ作ればいい」という次元にいる。これは、事前に用意されたツールに依存するチャット型エージェントとは、質的に異なる能力だ。

コーディングエージェントの「コーディング」は、目的ではなく手段だ。その手段が、他のどんなエージェントアーキテクチャよりも広い「できること」を実現している。だからこそ、コーディングエージェントは究極の汎用AIエージェントなのだと、僕は思う。
