## はじめに

[前回の記事](https://zenn.dev/btiw/articles/489b19dae63782)では、個人の健康データを収集・集約・分析するシステムの全体像と設計の考え方を紹介した。本記事では、そのシステムにMCP（Model Context Protocol）サーバーを組み込んだ実装について紹介する。

普段からスマホのClaudeアプリで気になったことを質問している。「今日筋トレしたほうがいいか？」といつも通り聞いたところ、必要なデータを参照した上で回答が返ってきた。MCPサーバーを組み込んだことで、AIがいつもの会話の中で自分の実データを参照するようになったのである。

「γ-GTが高い時の体重と体脂肪率を教えて」のように直接データを聞くこともできる。ダッシュボードも引き続き使うが、普段のAI利用の延長で自分のデータにアクセスできるのは、また違った便利さがある。

## MCPとは

MCP（Model Context Protocol）は、AIモデルが外部のデータやツールにアクセスするためのプロトコルである。JSON-RPC 2.0をベースとしており、サーバーが「ツール」を定義し、AIがそれを呼び出す形でやり取りする。

Claude.aiやClaude Codeには、MCPサーバーに接続する機能が組み込まれている。サーバーを用意すれば、AIが自分のデータにアクセスできるようになる。

## 構成

vitalogは健康データの集約・可視化・分析を担うRailsアプリケーションである。MCPサーバーは、このRailsアプリにエンドポイントを1本追加する形で同居させた。

```
POST /api/mcp   ← MCPリクエスト（JSON-RPC 2.0）
```

マイクロサービスとして分離する選択肢もあったが、MCPサーバーが参照するのはvitalogのDBそのものであり、データの取得ロジック（Health Digest APIや各モデルへのクエリ）もvitalog内に既にある。別サービスにすると、同じデータアクセスを二重に実装するか、内部APIを新設することになる。また、本番環境はRender上で動いており、サービスを追加すれば月額コストが増える。個人利用でトラフィックも限られるため、同居が合理的と判断した。

MCPサーバーの処理は以下の流れで動く。

```
POST /api/mcp
  → JSONパース
  → Origin検証・認証
  → JSON-RPCメソッドのディスパッチ
    → initialize / ping / tools/list / tools/call
  → レスポンス返却
```

`tools/call`が呼ばれると、指定されたツールが実行され、結果がMarkdown形式のテキストとして返る。AIはこのテキストを読んでユーザーに回答する。

### 実際の問い合わせ例

具体的にどういう体験になるか、1つ例を示す。

スマホのClaudeアプリで「今週の睡眠はどうだった？」と聞くと、MCPサーバーの`get_health_data`ツールが呼ばれ、直近1週間の睡眠データが取得される。AIはそのデータを読み、「平均睡眠時間は6時間48分で、木曜だけ5時間を切っています」のように要約して答える。

ダッシュボードを開いて睡眠ページに移動して日付を絞り込む、という操作は不要になる。聞くだけでいい。

## Claude.aiに繋ぐにはOAuth 2.1が必要だった

### Bearer tokenでいけると思っていた

vitalogにはもともとingest API用のAPIキー認証（Bearer token）がある。MCPサーバーも同じ方式で認証すればよいと考えていた。

しかし、Claude.aiのカスタムMCPコネクタは、認証方式としてOAuth 2.1を要求する。APIキーを直接設定する方法は提供されていない。

つまり、Claude.aiからMCPサーバーに接続するには、vitalog側にOAuth 2.1の認可サーバーを実装する必要があった。

### なぜOAuthが必要なのか

MCPサーバーには個人の健康データが入っている。Claude.aiが「このサーバーにアクセスしてよいか」をユーザーに確認し、ユーザーが明示的に許可する仕組みが必要である。OAuthはまさにこの認可フローを標準化したプロトコルであり、Claude.ai側がこれを前提にしているのは理にかなっている。

結果として、Bearer tokenによる認証は開発・テスト用に残しつつ、本番のClaude.ai接続にはOAuth 2.1を使う構成になった。

### 実装が必要だったもの

Claude.aiのカスタムコネクタが要求する仕様に合わせて、以下のエンドポイントを実装した。

| エンドポイント | 役割 | 準拠するRFC |
| --- | --- | --- |
| `GET /.well-known/oauth-authorization-server` | 認可サーバーのメタデータ公開 | RFC 8414 |
| `GET /.well-known/oauth-protected-resource` | 保護リソースのメタデータ公開 | RFC 9728 |
| `POST /oauth/register` | 動的クライアント登録（DCR） | RFC 7591 |
| `GET /oauth/authorize` | 認可画面の表示 | RFC 6749 |
| `POST /oauth/authorize` | ユーザーの認可処理 | RFC 6749 |
| `POST /oauth/token` | アクセストークンの発行 | RFC 6749 + RFC 7636 |

ひとつずつ見ていく。

### メタデータエンドポイント

Claude.aiがMCPサーバーに接続する最初のステップは、サーバーの仕様を知ることである。`.well-known`エンドポイントで、認可サーバーのURL、対応するコードチャレンジ方式、クライアント登録エンドポイントなどを公開する。

```
# oauth/metadata_controller.rb
def authorization_server
  render json: {
    issuer: oauth_base_url,
    authorization_endpoint: "#{oauth_base_url}/oauth/authorize",
    token_endpoint: "#{oauth_base_url}/oauth/token",
    registration_endpoint: "#{oauth_base_url}/oauth/register",
    code_challenge_methods_supported: [ "S256" ],
    token_endpoint_auth_methods_supported: [ "none" ]
  }
end
```

`token_endpoint_auth_methods_supported: ["none"]`は、クライアントシークレットを使わないPublic Clientであることを示す。Claude.aiはブラウザベースのクライアントであり、シークレットを安全に保持できないため、この設定が必要になる。

### 動的クライアント登録（DCR）

通常のOAuthでは、クライアント（ここではClaude.ai）を事前に登録してclient\_idを発行する。DCRはこの登録をAPI経由で自動化する仕組みである。Claude.aiがMCPサーバーに初めて接続する際、自動的にクライアント登録が行われる。

```
# oauth/registration_controller.rb
ALLOWED_REDIRECT_URIS = %w[
  https://claude.ai/api/mcp/auth_callback
  https://claude.com/api/mcp/auth_callback
].freeze

def create
  redirect_uris = params[:redirect_uris]

  # 同じredirect_urisセットなら既存クライアントを再利用（冪等性の確保）
  existing = OauthClient.find_by(redirect_uris: redirect_uris)
  if existing
    render json: client_response(existing), status: :ok
    return
  end

  client = OauthClient.create!(
    client_id: SecureRandom.hex(32),
    client_name: params[:client_name] || "MCP Client",
    redirect_uris: redirect_uris
  )
  render json: client_response(client), status: :created
end
```

ここでのポイントは、redirect\_urisのホワイトリスト制御である。本番環境では`claude.ai`と`claude.com`のコールバックURLのみを許可し、任意のURLへの認可コード送信を防いでいる。

### 認可フロー（PKCE）

ユーザーがClaude.aiからMCPサーバーに接続すると、以下のフローが走る。

```
Claude.ai
  → GET /oauth/authorize?client_id=...&code_challenge=...&code_challenge_method=S256
  → vitalogのログイン画面（Devise）
  → 認可画面（「健康データへのアクセスを許可しますか？」）
  → ユーザーが「許可する」をクリック
  → POST /oauth/authorize → 認可コード発行
  → リダイレクト: claude.ai/api/mcp/auth_callback?code=...
  → POST /oauth/token（code + code_verifier）
  → アクセストークン発行（30日有効）
```

PKCE（Proof Key for Code Exchange）は、認可コードの横取り攻撃を防ぐ仕組みである。クライアントはランダムな`code_verifier`を生成し、そのSHA256ハッシュ（`code_challenge`）を認可リクエストに含める。トークン交換時に元の`code_verifier`を送り、サーバー側でハッシュを検証する。

```
# oauth/token_controller.rb
def valid_pkce?(code_verifier, code_challenge)
  return false if code_verifier.blank?

  expected = Base64.urlsafe_encode64(
    Digest::SHA256.digest(code_verifier),
    padding: false
  )
  ActiveSupport::SecurityUtils.secure_compare(expected, code_challenge)
end
```

`secure_compare`を使うことで、タイミング攻撃（文字列比較の処理時間の差から情報を推測する攻撃）を防いでいる。

### MCPリクエストの認証

発行されたアクセストークンは、以降のMCPリクエストの`Authorization`ヘッダーに含まれる。

```
# concerns/mcp_authenticatable.rb
def authenticate_mcp!
  return if unauthenticated_mcp_method?  # initialize等は認証不要

  provided = extract_bearer_token

  return if valid_mcp_api_key?(provided)   # 開発用APIキー認証
  return if valid_oauth_token?(provided)   # OAuth トークン認証

  # 認証失敗時、WWW-Authenticateヘッダーでメタデータの場所を通知
  response.headers["WWW-Authenticate"] =
    %(Bearer resource_metadata="#{oauth_base_url}/.well-known/oauth-protected-resource")
  render json: { error: "Unauthorized" }, status: :unauthorized
end
```

`initialize`メソッドは認証なしで呼び出せるようにしている。MCPプロトコルでは、最初のハンドシェイク（`initialize`）でサーバーの情報を取得し、その後の認証フローに進む必要があるためである。

認証失敗時に`WWW-Authenticate`ヘッダーを返すのも仕様上の要件である。Claude.ai側はこのヘッダーからOAuthメタデータの場所を知り、認可フローを開始する。

### 健康データを扱う上での設計判断

健康データは機微な情報であり、OAuthの仕組みだけでなく、アクセスの範囲と条件を意識して設計した。

* **単一ユーザー前提**: このシステムは自分専用であり、OAuthで認可されるのも自分だけである。マルチテナントの権限管理は不要と割り切った
* **読み取り専用**: MCPサーバーはデータの参照のみで、書き込みや削除のツールは提供していない。誤操作や意図しないデータ変更のリスクを排除している
* **トークンの有効期限**: アクセストークンは30日で失効する。長すぎず、頻繁な再認可も不要なバランスとした
* **返すのは整形済みデータ**: ツールが返すのはMarkdownに整形された結果であり、DBの生レコードをそのまま返すわけではない。用途に合わせて必要な項目を抽出・整形し、AIに渡る情報を絞っている

## ツール設計

### 5つのツール

MCPサーバーに実装したツールは以下の5つである。

| ツール名 | 用途 | 入力 |
| --- | --- | --- |
| `get_health_data` | 指定期間の健康データを横断取得 | 期間（必須）、セクション指定（任意） |
| `search_blood_tests` | 血液検査の検索 | マーカー名（部分一致）、期間 |
| `search_exam_findings` | 画像診断の所見検索 | 検査種別、キーワード、期間 |
| `search_medical_visits` | 診察記録の検索 | キーワード、期間 |
| `get_patient_profile` | 病態プロファイルの取得 | なし |

### 「汎用1本」ではなく「目的別に分けた」理由

最初は「何でも聞けるツール」を1本だけ用意する案も考えた。しかし、ツールを分けたほうがAIの精度が上がる。

たとえば「γ-GTの推移を教えて」という問い合わせに対して、汎用ツール1本だと、AIはまず全データを取得してからその中を探すことになる。`search_blood_tests`というツールがあれば、AIは「これは血液検査の検索だ」と判断し、マーカー名を引数に入れて直接検索できる。ツール名と入力スキーマが、AIにとってのヒントになる。

一方で、ツールを細かく分けすぎると、AIがどれを使うべきか迷う場面が増える。5つという数は、用途の区別が明確で、かつAIが選択に迷わない程度のバランスとして落ち着いた。

### get\_health\_dataの設計

`get_health_data`は最も汎用的なツールで、期間を指定して複数種類のデータを横断的に取得できる。

```
# mcp/tools/get_health_data.rb
DEFINITION = {
  name: "get_health_data",
  description: "指定期間の健康データを取得します（体組成、睡眠、歩数、食事、筋トレ、血液検査、診察記録、画像診断など）",
  inputSchema: {
    type: "object",
    required: %w[from to],
    properties: {
      from: { type: "string", description: "開始日 (YYYY-MM-DD)" },
      to: { type: "string", description: "終了日 (YYYY-MM-DD)" },
      sections: {
        type: "array",
        items: { type: "string",
                 enum: %w[body_compositions sleep steps meals
                          workouts blood_tests medical_visits
                          fibroscan_results exam_findings] },
        description: "取得するセクション（省略時は全セクション）"
      }
    }
  }
}.freeze
```

`sections`パラメータで、必要なデータ種別だけを取得できるようにした。AIが「睡眠について聞かれているから`sleep`だけ取ればいい」と判断でき、無駄なデータ取得を避けられる。

### 出力形式をMarkdownにした理由

すべてのツールは、結果をMarkdown形式のテキストで返す。構造化データとしてはJSONのほうが再利用しやすいが、MCPのレスポンスはAIが読んでユーザーに伝えるものである。厳密な機械処理より、対話の品質を優先した。Markdownのテーブルなら、AIが構造を理解しやすく、人間が直接読んでも意味がわかる。

```
## 体組成
| 日付 | 体重 (kg) | 体脂肪率 (%) | 骨格筋量 (kg) |
|------|-----------|-------------|-------------|
| 2026-04-01 | 72.5 | 22.3 | 31.2 |

## 睡眠
| 日付 | 就寝 | 起床 | 睡眠時間 |
|------|------|------|---------|
| 2026-04-01 | 23:30 | 6:45 | 7時間15分 |
```

また、このMarkdownレンダリングは、通院レポートや週次チェックの生成でも共通のサービスクラスを使っている。既存のレポート基盤を再利用できた点も、Markdownを選んだ理由のひとつである。MCP用にはサブクラスで食事の表示レベル（1日分は全品目、複数日は上位5品目）を切り替えている。

### 検索ツールのセキュリティ

検索ツールではユーザー入力（マーカー名やキーワード）をSQLのLIKEに渡すため、SQLインジェクション対策が必要である。

```
sanitized = ActiveRecord::Base.sanitize_sql_like(params["marker_name"])
query = query.where("blood_markers.name ILIKE ?", "%#{sanitized}%")
```

`sanitize_sql_like`で`%`と`_`をエスケープし、プレースホルダーでバインドすることで安全に検索している。

## 使ってみて

### 便利な場面

日常的に使うのは、ちょっとした振り返りである。

体重が一番増えた時期の食事内容を聞いて、何が影響していたかを確認する。γ-GTが上がっている時期の生活習慣を突き合わせて振り返る。こうしたデータの横断的な問い合わせが、一言で済む。

ダッシュボードでも同じデータは見られるが、MCPの良さはAIとの対話の流れを切らずにデータを参照できる点にある。別の画面を開いて探すのではなく、会話の中で自然にデータが出てくる。

### まだ足りないところ

現時点では、データの参照のみで書き込みはできない。たとえば「今日の体調メモを記録して」といった操作には対応していない。現状は読み取り専用で十分だが、将来的にはライトバック機能も検討の余地がある。

また、長期間のデータを取得するとレスポンスが大きくなる。AIのコンテキストウィンドウには上限があるため、期間が長い場合は自動的に要約するような仕組みがあるとよいかもしれない。

## おわりに

MCP本体の実装——JSON-RPCのハンドラーとツールの定義——は薄い。実用化の山場はOAuth 2.1だった。執筆時点では、Claude.aiのカスタムコネクタに接続するためにOAuth 2.1が必要であり、複数のRFCにまたがるエンドポイントをRailsに実装するのが最も手間のかかった部分だった。

しかし、それだけの手間をかけた価値はあったと思う。スマホから一言聞くだけで自分の健康データに基づいた回答が返ってくる体験は、ダッシュボードとはまた違った便利さがある。データを集めて蓄積する仕組みは前回の記事で紹介したが、MCPによって「データへのアクセス方法」の選択肢が広がった。
