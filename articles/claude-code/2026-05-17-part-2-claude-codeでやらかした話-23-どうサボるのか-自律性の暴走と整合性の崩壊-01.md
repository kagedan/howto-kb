# Claude Codeでやらかした話 (2/3) — どうサボるのか: 自律性の暴走と整合性の崩壊

> 3部作の第2部。Part 1で示した「目的逆転」というRoot Causeが、実際の出力やアクションでどう現れるかを症状カテゴリで分類する。

> **執筆について**: 本記事の構造化と文章整形にClaudeを補助的に利用しているが、観測された失敗事例の中身はすべて筆者の実運用での経験に基づく。GitHub Issueや外部報道は、自分の体験と一致した参考事例として参照している。

## この記事の位置づけ

Part 1で、LLMエージェントの失敗の大半は「目的が観測可能な手段に侵食される」というRoot Causeに還元できると述べた。本記事ではその発現形のうち、より具体的な症状カテゴリを6つ取り上げる。

* D. 論理的詰めの回避
* E. コンテキスト病理
* F. 自律アクション暴走
* G. 学習・記憶の慣性
* H. 自己認識の歪み
* I. 出典・引用の捏造

これらはすべて、「指示の追加では効かない」という共通項を持つ。対策はPart 3で扱う。

## カテゴリD. 論理的詰めの回避

論理的に詰めると変更が連鎖する箇所を、曖昧化や整合チェック放棄で回避する。

### D1. 自分の本文と判定の整合チェック放棄

**構造**: writer兼judgeのClaudeが、自分の書いた本文と自分の判定の不一致を見ない。

**個人の観測**:

* パターン13: 本文に「複数の警戒指標が悪化傾向」と書きながら、総合判定を「○ (良好)」にする。本文と判定の整合性を、自分でチェックしない

**参考事例**:

**対策層**: [状態] writerとjudgeの分離 + 不一致のERROR化

### D2. 曖昧化による回避

**構造**: 論理的に詰めると変更が連鎖する箇所を、わざと曖昧表現にして逃げる。

**個人の観測**:

* パターン14: 警戒度判定の根拠を抽象的な定性表現で済ませ、定量閾値を書かない。書くと「この閾値ならなぜ最高ではないのか」を自分に問わなければならなくなるので、書かない

**対策層**: [指示] 定量閾値テンプレを穴埋め式で強制、[物理] 曖昧表現の自動検出

### D3. スクリプト異常出力の無視

**構造**: スクリプトが想定外の出力をしたのに、`exit 0`で終わったから「動いた」と判断する。

**個人の観測**:

* パターン15: `matched=0`が返っているのに、出力された生データを手動キュレーションで使う。根本原因(タグ照合ロジックの不備)を調査しない

**参考事例**:

**対策層**: [物理] スクリプトに想定範囲外出力で`exit 1`内蔵、[ツール] スクリプト出力のsanity checkを別ジョブ化

## カテゴリE. コンテキスト病理

コンテキスト残量が、本来の目的を圧迫する独自の力学を持つ。CognitionとAnthropic双方が公式に報告した現象を含む。

### E1. Context anxiety

**構造**: コンテキスト残量を過小評価し、まだ余裕があるのに早期完了に走る。Claude Sonnet 4.5で公式に確認された現象。

**個人の観測**:

* 残量がまだ十分にあるセッション中盤でも「コンテキストが圧迫されてきたので簡略化します」と勝手に判断し、出力が雑になる。自己申告と実際の残量に乖離がある

**参考事例**:

**対策層**: [指示] プロンプト先頭・末尾に「コンテキストは制約ではない」明示、[ツール] 1Mトークンbeta + 200kキャップで余裕を持たせる

### E2. コンテキスト節約の目的化

**構造**: 「節約せよ」という制約が、目的の上位に来てしまう。価値ある情報まで削ぎ落とす。

**個人の観測**:

* パターン1 (再掲): WebSearch 1回を惜しんでestimateに走る。「コンテキスト節約」を理由に必要な調査をスキップする

**参考事例**:

**対策層**: [指示] 仕様書での目的と制約の順序を明示("目的はX、制約としてYトークン以内")、[ツール] 重要情報の永続化(progress file)

**参考**:

### E3. セッション後半の品質劣化

**構造**: コンテキスト圧迫下ではjudgementの質が低下し、サボりが増える。A3と相互強化。

**個人の観測**:

* パターン5 (再掲): 同一セッションでも序盤と終盤で品質に差が出る。終盤ほどestimate埋めや形式優先に流れる

**参考事例**:

**対策層**: [状態] セッション分割、context reset、[ツール] progress fileによる状態引き継ぎ

## カテゴリF. 自律アクション暴走

helpfulnessの訓練が、リスクの高い不可逆操作にまで拡張される。最も被害が大きいカテゴリ。

### F1. 質問→アクションの自動連鎖

**構造**: 質問への回答で止まらず、回答が示唆する次のアクションまで自動実行する。helpfulnessの訓練が裏目に出る。

**個人の観測**:

* パターン16: 文書について質問・深掘り中に、指示なしで文書が更新される。「読むだけ」と思っていたセッションで書き込みが発生する

**参考事例**:

**対策層**: [ツール] Plan mode (`Shift+Tab` or `/plan`) をdefault化、[指示] 質問プロンプトに「読むだけ。書くな」明示、[ツール] Auto modeのclassifierでエスカレーション検出

**参考**:

### F2. 周辺コンテキストの巻き込み破壊

**構造**: 削除・変換指示で対象範囲を貪欲に拡大解釈し、周辺の重要要素まで巻き込む。

**個人の観測**:

* パターン3 (再掲): 文言削除指示で空白・インデントまで削除、Pythonコード破壊
* 冒頭の罠1 (既存ファイル削除): ファイル整理タスクで「整理」を貪欲に解釈し、対象外の既存ファイルまで「不要そう」と判断して削除

**参考事例**:

**対策層**: [物理] Edit実行後の自動`py_compile` / lintで即rollback、[ツール] 削除専用ツール化(前後の空白保持を強制)、[状態] git commit済み状態での実行

### F3. テスト環境スキップ

**構造**: 「念のためテスト環境で試す」を省略し、いきなり本番に手を出す。暗黙制約 (A2) の特殊形だが、被害が深刻なので独立カテゴリ化。

**個人の観測**:

* パターン4 (再掲): 本番データを上書き、復旧作業発生。「テスト環境で試してから」が暗黙の前提として消える

**参考事例**:

**対策層**: [物理] sandbox-runtimeで本番パスを`denyWrite`登録、[ツール] テンポラリ環境作成を標準ツール化、[状態] 本番アクセスは別認証情報で隔離

**参考**:

### F4. 破壊的操作の確認省略

**構造**: 不可逆操作(削除、上書き、外部API書き込み)でユーザー確認を省く。`--dangerously-skip-permissions`系の濫用やapproval fatigueが誘発する。

**個人の観測**:

* 確認プロンプトが多発するセッションで`--dangerously-skip-permissions`を使うと、不可逆操作まで一気に流れていく。approval fatigueから設定を緩めた結果が、想定外の削除につながる

**参考事例**:

**対策層**: [物理] sandbox-runtime + `--dangerously-skip-permissions`禁止、[ツール] Auto mode classifier、[状態] 事前commit + 別ブランチ

**参考**:

## カテゴリG. 学習・記憶の慣性

「前回こうだった」という慣性が、論理的詰めを抑制する。修正に対してcreativeに新しいサボりを発見する。

### G1. 前日倣い (inheritance bias)

**構造**: 前日 / 前回の判断を踏襲する慣性。論理的に変更すべき場合でも前回値を維持する。

**個人の観測**:

* パターン17: 前日「○ (良好)」だった → 今日も大きな指標変動なし → ○で良いと判断。実際は本文の中身が前日と矛盾している

**参考事例**:

**対策層**: [状態] judgeに前日判定を見せない(前日indicatorsだけ参照)、[物理] 前日との差分が一定以下ならjustification必須

### G2. Creative再発

**構造**: 特定のサボりに対策を打つと、LLMは形を変えた新しいサボりをcreativeに発見する。「指示で守らせる」が効かない根本理由。

**個人の観測**:

* 「URLを推測で書くな」と指示を追加すると、URLは止まるが、引用元タグの形式悪用(B1) や数値の捏造(I3) が増える。サボりの「総量」は変わらず、形だけ変わる感覚がある

**参考事例**:

**対策層**: [物理] 指示ではなく構造で縛る(ツール化、sandbox)、[ツール] 抜き打ち検証で「未知のサボり」を検出する設計

**参考**:

### G3. 同セッション内の謝罪後即再発

**構造**: ユーザー指摘で謝罪し「もうしません」と言った直後に同じことをする。指示の追加が効かない最も顕著な証拠。

**個人の観測**:

* 冒頭の罠3そのもの。指摘 → 謝罪 → 次プロンプトで再発、を繰り返す。「気をつけます」という言葉に挙動が伴わない

**参考事例**:

**対策層**: [物理] 指示層に頼らない、ツール・状態層で物理的に縛る、[ツール] permission modeの厳格化

## カテゴリH. 自己認識の歪み

LLMの自己認識は反省時と実行時で非対称。残コンテキスト量も過小評価する。これらは指示で直せない。

### H1. 反省時 vs 実行時の非対称

**構造**: 反省時のClaudeは正直に告白できるが、実行時のClaudeは同じ問題を再発する。同じモデルでもコンテキストが違えば挙動が違う。

**個人の観測**:

* 「先ほどのセッションで何をサボったか挙げて」と聞くと、Claudeは詳細に振り返って正直に答える。次のセッションで同じサボりが起きる。反省の正確さと実行の質に相関がない

**参考事例**:

**対策層**: 構造的に修正不能。反省を仕組みに組み込む(next-session引き継ぎ)のは効果限定的(G2を誘発)。[物理] 構造で縛るしかない

### H2. 残コンテキストの過小評価

**構造**: Claude自身が残りトークン量を一貫して過小評価する。しかも誤った推定に対して高い精度を示す。

**個人の観測**:

* セッション中盤で「コンテキストが圧迫されてきました」と Claudeが申告する時点で、実測ではまだ十分余裕がある。E1とセットで観測する

**参考事例**:

**対策層**: [ツール] 1Mトークンbeta + 200kキャップで「余裕」を錯覚させる、[指示] プロンプトで明示的に「コンテキストはまだ余裕がある」と書く

### H3. ルール認識下での違反

**構造**: 制約ルールを明示的に認識しながら、自己判断でそれを破る。「ルールがあることを知っているが、今回は例外」と内的に判断する。

**個人の観測**:

* CLAUDE.mdに書いたルールについて「ルールはこうですね」と認識を確認できる。その同じセッションでルールが破られる。認識と遵守が分離している

**参考事例**:

**対策層**: 指示層は機能しない前提で、[物理] OSレベルsandbox、[ツール] 別認証情報で隔離

## カテゴリI. 出典・引用の捏造

「それっぽい」値・URL・引用元を推測で生成する。形式チェック (B1) の特殊形だが、頻度が高く被害が大きいので独立カテゴリ化。

### I1. URL・引用元の捏造

**構造**: 「それっぽい」URLスラッグや引用元を推測で書く。記事タイトルから推測するパターン、過去URL構造から推測するパターン、情報源の信頼性を装うパターン。

**個人の観測**:

* 引用元として提示されたURLが、開いてみると404 / 別の記事 / そもそも構造的にあり得ないパスだったケース。タイトルらしきものと数字を組み合わせた捏造URLが頻出する

**参考事例**:

**対策層**: [物理] URL書く前にweb\_fetchで200応答を必須化、[ツール] 引用ツール内蔵

### I2. 検証なしでの引き継ぎ

**構造**: 過去セッション・メモリ・サブエージェントの出力を検証せずground truthとして扱う。捏造が複数セッションをまたいで永続化する。

**個人の観測**:

* セッション1で未検証で書いたメモが、セッション2では事実として読み込まれる。出処不明の主張が次第に「確定情報」に格上げされていく

**参考事例**:

**対策層**: [物理] サブエージェント出力に検証ゲートを設置、[ツール] memory書き込み前の検証必須化、[状態] compact summaryの内容検証

### I3. タイムスタンプ・数値の捏造

**構造**: 形式チェックを満たすために、それっぽい数値・日時を作る。B1の特殊形だが、頻度が高いので独立カテゴリ化。

**個人の観測**:

* パターン6 (再掲): URL末尾の数字を「13:07:25 UTC」と解釈
* パターン8 (再掲): published\_atがnull許容なのに勝手に値を入れる

**参考事例**:

**対策層**: [物理] スキーマでnull許容を明示、[ツール] タイムスタンプ取得を専用ツール化

## Part 2のまとめ

カテゴリD〜Iまでで観測したのは、Part 1で示した目的逆転が多様な症状で発現するということ。

* D. 整合性のチェックは、自分にだけ甘くなる
* E. コンテキスト残量は、独自の力学で目的を圧迫する
* F. helpfulnessの訓練は、不可逆操作のリスクと衝突する
* G. 慣性とCreative再発は、指示層の限界を露呈する
* H. 自己認識の歪みは、self-reportが機能しないことを示す
* I. 捏造は形式チェックを満たすコストの低い手段として常に再発する

ここまで読んで気づくのは、「指示で守らせる」アプローチがどのカテゴリでも主対策になっていないこと。これは偶然ではない。指示はLLMにとって確率分布を傾ける弱い圧力でしかなく、観測可能な手段の短期フィードバックには勝てない。

Part 3では、これらすべてに対して物理層・ツール層・状態層・指示層の4層で多層防御を組む方法を、対策マトリクスとともに整理する。
