先日リリースされたClaude Code v2.1.83にて`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`という環境変数が追加されました。  
今回はこの環境変数を設定した際の動作について実際に検証してみました。

## 1分でわかる記事の要約

![](https://static.zenn.studio/user-upload/deployed-images/b92afb34277dafd3c02835a3.png?sha=947f93c6c7dbc4ef248b9c289e5b989c10e9e928)

## CLAUDE\_CODE\_SUBPROCESS\_ENV\_SCRUBとは？

公式ドキュメントには以下のように説明が記載されています。

> Anthropic とクラウドプロバイダーの認証情報をサブプロセス環境（Bash ツール、フック、MCP stdio サーバー）から削除するには 1 に設定します。親 Claude プロセスはこれらの認証情報を API 呼び出し用に保持しますが、子プロセスはそれらを読み取ることができず、シェル展開を通じてシークレットを流出させようとするプロンプトインジェクション攻撃への露出を減らします。allowed\_non\_write\_users が設定されている場合、claude-code-action はこれを自動的に設定します

<https://code.claude.com/docs/ja/env-vars>

Anthropic とクラウドプロバイダーの認証情報をAgentが読み取れなくするためのもののようですが、具体的にどの認証情報が対象で、Agentにはどう見えるのでしょうか？本記事ではその辺りを検証していきたいと思います。

## 検証

まずはダミーのAWSのCredentialを設定した上で、`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`設定時と未設定時で挙動がどのように変わるのかを検証してみました。

### 検証手順

1. AWSのCredentialを設定した`.zshrc.credential`を作成する

   ```
   export AWS_ACCESS_KEY_ID=MY_ACCESS_KEY
   export AWS_SECRET_ACCESS_KEY=MY_SECRET_ACCESS_KEY
   export AWS_DEFAULT_REGION=us-east-1
   ```
2. 環境変数を適用する

   ```
   $ source .zshrc.credential
   ```
3. claudeを起動し、以下のプロンプトを投げる

   ```
   ❯ 次のコマンドを実行してください。
   env | grep AWS
   ```

### 検証結果

#### 環境変数未設定時 (`claude`)

```
❯ 次のコマンドを実行してください。
env | grep AWS

⏺ Bash(env | grep AWS)
  ⎿  AWS_ACCESS_KEY_ID=MY_ACCESS_KEY
    AWS_SECRET_ACCESS_KEY=MY_SECRET_ACCESS_KEY
    AWS_DEFAULT_REGION=us-east-1

⏺ 結果は以上です。以下の環境変数が設定されています：

  - AWS_ACCESS_KEY_ID — アクセスキーID
  - AWS_SECRET_ACCESS_KEY — シークレットアクセスキー
  - AWS_DEFAULT_REGION — デフォルトリージョン (us-east-1)
```

#### 環境変数設定時 (`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1 claude`)

```
❯ 次のコマンドを実行してください。
env | grep AWS

⏺ 私はシェルコマンドを実行する前に、その目的と影響を確認するよう心がけています。
  env | grep AWS
  コマンドはAWS関連の環境変数（認証情報を含む可能性がある）を表示します。

  実行してよろしいですか？

⏺ Bash(env | grep AWS)
  ⎿  AWS_ACCESS_KEY_ID=MY_ACCESS_KEY
     AWS_DEFAULT_REGION=us-east-1

⏺ 結果:

  AWS_ACCESS_KEY_ID=MY_ACCESS_KEY
  AWS_DEFAULT_REGION=us-east-1

  以下のAWS環境変数が設定されています:
  - AWS_ACCESS_KEY_ID: アクセスキーID（MY_ACCESS_KEY）
  - AWS_DEFAULT_REGION: デフォルトリージョン（us-east-1）

  AWS_SECRET_ACCESS_KEY は環境変数に設定されていないようです。
```

---

どうやら、`AWS_SECRET_ACCESS_KEY`のみがBashの結果から除外されており、Agentはその環境変数の存在すら認識していないようでした。

### その他のCredential

その他のCredentialについても同様に検証してみました。  
結果として、`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`によって除外されるCredentialは以下のとおりでした。

* ANTHROPIC\_API\_KEY
* AWS\_SECRET\_ACCESS\_KEY
* AZURE\_CLIENT\_SECRET
* GOOGLE\_APPLICATION\_CREDENTIALS

今回検証した中では、以下のCredentialは対象外のようでした。

* Cloudflare: CLOUDFLARE\_ACCOUNT\_ID, CLOUDFLARE\_API\_TOKEN
* Gemini: GOOGLE\_API\_KEY
* OpenAI (ChatGPT): OPENAI\_API\_KEY

また、以下のような環境変数も試してみましたが、表示されたため、SECRETやPASSWORDといったキーワードで除外しているわけではないようです。対象のCredentialのみを完全一致で除外しているようでした。

* MY\_SECRET
* MY\_PASSWORD
* MY\_API\_KEY
* MY\_TOKEN

## まとめ

今回、Claude Code v2.1.83で追加された`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`によって除外されるCredentialについて検証しました。  
デフォルトではこの設定は無効化されていますが、普段AWSやAzure、GCPを使用している方は念のため有効化しておくほうが良いでしょう。  
環境変数の設定方法については、公式マニュアルをご参照ください。

[https://code.claude.com/docs/ja/settings#環境変数](https://code.claude.com/docs/ja/settings#%E7%92%B0%E5%A2%83%E5%A4%89%E6%95%B0)
