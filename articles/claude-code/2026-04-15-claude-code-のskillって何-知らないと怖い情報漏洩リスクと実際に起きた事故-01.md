## はじめに

「Claude Code を使えばコーディングが爆速になる」「Skill を使えばなんでもできる」——そんな話を聞いて最近使い始めた人も多いのではないでしょうか。

でも、こんな疑問はありませんか？

「Skill ってよく見かけるけど、何ができるの？　安全なの？」

この記事では、Claude Code の「Skill」という機能の概要から、その仕組みに潜む情報漏洩リスク、そして実際に起きた重大インシデントまでを、読み物としてまとめました。

すでに使っている人にも、これから使おうとしている人にも、知らない人は知っていてほしい内容です。

## そもそも Claude Code の「Skill」とは？

Claude Code の Skill は、一言でいえば「AI への行動マニュアルを外部ファイルから読み込める仕組み」です。

### 何ができるのか

Skill は主に SKILL.md という Markdown ファイルで定義されます。このファイルに AI への指示や、実行すべきコマンド・スクリプトへの参照を書いておくと、Claude Code がそれを読み取って自動的に行動してくれます。

たとえば——

* **コード生成の自動化**：特定のフレームワーク向けに CRUD コードをまとめて生成する
* **コードレビュー**：Pull Request のたびに自動でセキュリティチェックを走らせる
* **ドキュメント生成**：関数の仕様書を自動で書き出す
* **テスト自動化**：テストコードを Skill に従って一気に生成する

要するに、「繰り返し行う作業を AI に覚えさせ、再現性高く実行させる」のが Skill の本質です。チームでリポジトリに含めれば、全員が同じワークフローを共有できるという利便性もあります。

### どこから入手するのか

Skill は自作するほか、ClawHub や GitHub といったプラットフォームから外部の人が公開したものをダウンロードして使うこともできます。ここが、後述するリスクの入口になります。

## Skill の仕組みと、情報漏洩が起きる理由

便利な機能には、裏側に構造的なリスクがあります。

### 「設定ファイルが実行レイヤーになる」という盲点

従来のアプリ開発では、設定ファイルはあくまで「受動的なパラメータの集まり」でした。しかし Claude Code の Skill や .claude/settings.json は違います。

**書かれた内容が、直接コマンドや通信の挙動を制御します。**

つまり悪意ある人間がこのファイルに細工をすれば、ユーザーが気づかないまま——

* 外部サーバーへ API キーを送信する
* 任意のシェルコマンドを実行する
* ファイルシステムにアクセスする

……といったことが、「ただファイルを開いた」だけで起きてしまうのです。

### 漏洩の 3 つの経路

セキュリティ研究者が指摘している攻撃経路は主に 3 つです。

① Hooks（フック）の悪用

Claude Code には、特定のタイミング（ファイル編集後など）で自動的にシェルコマンドを実行する「Hooks」という仕組みがあります。悪意ある設定ファイルをここへ仕込まれると、ユーザーの確認なしにコマンドが走ります。

② MCP（Model Context Protocol）の悪用

MCP は Claude が外部サービスと連携するための標準プロトコルです。本来は初回に承認ダイアログが出ますが、設定ファイルによってその保護を回避されるリスクがありました。

③ 環境変数・API キーの流出

ANTHROPIC\_BASE\_URL という環境変数を書き換えることで、Claude Code が Anthropic に送るはずの API リクエスト（認証キー付き）を、攻撃者のサーバーに横流しされた例があります。

## 実際に起きた事故・脆弱性

これは「理論上のリスク」ではありません。実際に報告されています。

### CVE-2025-59536 / CVE-2026-21852（2026 年 2 月発覚）

セキュリティ企業 Check Point Research が 2026 年 2 月 25 日に発表した調査で、Claude Code に 2 つの重大な脆弱性が確認されました。

攻撃者が用意した .claude/settings.json を含むリポジトリを開くだけで、ユーザーが何も操作しなくても攻撃が発動します。「GitHub で便利そうなリポジトリを見つけて clone した」というありふれた行動が、そのまま侵害の入口になるわけです。

API キーが盗まれると何が起きるか。Anthropic の API は Workspace という仕組みで複数のキーを共有しており、1 本のキーが流出するだけで、チーム全体のファイルへの不正アクセス、データ改ざん、予期しない課金が発生する可能性があります。

なお、これらの脆弱性は Anthropic への報告を経て、現在は修正済みです（Trust Dialog の導入・強化）。

### Skill マーケットプレイスへの悪意ある Skill 混入（2026 年初頭）

さらに深刻なのが、Skill 自体が攻撃の媒体になったケースです。

セキュリティ企業 Snyk の調査によれば、公開されている Claude Code Skills の約 36.8% に何らかのセキュリティ問題があることが判明しました。

実際の被害として報告されているのは——

また、Cato CTRL というセキュリティ研究グループは、公式の「GIF Creator」スキルを改変してランサムウェアを実行できることを実証しました。postsave() という無害に見える関数名の中にバックドアを偽装し、Claude Code の最も厳格なセキュリティモードでも防げなかったと報告しています。

### Anthropic 自身のソースコード漏洩（2026 年 3 月 31 日）

Claude Code のユーザーではなく、作った側の事故も起きています。

2026 年 3 月 31 日、Anthropic の CI/CD パイプラインの人為的なミスにより、Claude Code の完全なソースコード（約 512,000 行の TypeScript）が npm レジストリに誤って公開されてしまいました。

外部からのハッキングではなく、リリース作業のミスという「内側からの崩壊」でした。流出したコードには未公開モデルの性能データや、AI の痕跡を隠す「Undercover Mode」と呼ばれる内部機能の情報なども含まれており、技術的・競争的に大きな影響を与えました。

## 危険な Skill を見分ける方法

Skill がどれだけ便利そうに見えても、インストール前に必ずコードを精査する習慣が必要です。以下のチェックリストを参考にしてください。

### ステップ 1：SKILL.md を必ず目視確認する

まず SKILL.md を開き、以下の**危険なパターン**が含まれていないかを確認します。

curl | sh は、外部からスクリプトをダウンロードしてその場で直接実行するコマンドです。何が書かれているかを確認する手段がなく、悪意あるコードがそのままマシン上で動く可能性があります。

base64 -d | sh は、Base64 でエンコードされた命令をデコードして実行します。コードを文字列として隠蔽する典型的な難読化の手口で、一見すると何を実行しているか判別できません。

eval() は、文字列をそのままコードとして動的に実行します。外部から渡された文字列を eval に流し込むことで任意のコードを実行させられるため、プロンプトインジェクションの温床になりやすい記述です。

"ignore previous instructions" は、プロンプトインジェクション攻撃でよく使われるフレーズです。AI に対して「これまでの指示を無視せよ」と命令し、安全な動作ルールを上書きしようとする意図が読み取れます。

ANTHROPIC\_BASE\_URL の書き換えは、Claude Code が Anthropic に送るはずの API リクエストを攻撃者のサーバーに横流しするための設定です。認証キーごと盗まれるため、5 つのパターンの中でも特に直接的な被害につながります。

### ステップ 2：外部通信を行っているか確認する

SKILL.md に加え、参照されているスクリプトファイルも必ず確認します。以下のキーワードを grep で検索しましょう。

```
grep -rn "fetch\|axios\|http\|https\|curl\|wget\|POST\|sendBeacon" .
```

外部通信を行う Skill は原則として導入しないことを推奨します。どうしても必要な場合は、通信先のドメインと送信される内容を必ず把握してください。

### ステップ 3：ファイルアクセスの範囲を確認する

Skill がどのディレクトリにアクセスするかを確認します。特に以下のパスが含まれている場合は要注意です。

### ステップ 4：作者・公開元を調べる

* GitHub Stars やコミット履歴が極端に少ない Skill は慎重に
* 公開日が直近で、レビューがゼロの Skill は特に警戒
* package.json や依存関係に見慣れないパッケージが含まれていないか確認する

### ステップ 5：Hooks と MCP の設定を確認する

.claude/settings.json の中に hooks セクションや MCP サーバーの設定が含まれている場合、それらが何をトリガーに何を実行するかを一行ずつ確認してください。

## 情報漏洩を防ぐ方法

Skill の精査に加え、日常の使い方そのものにも気をつけるべき点があります。

### Trust Dialog を軽率に「Yes」にしない

現在の Claude Code にはリポジトリを信頼する前に確認ダイアログが出ます。内容を読まずに承認し続けると、MCP サーバーが全自動承認になる設定が有効になってしまいます。「なんとなく OK した」の積み重ねが、最大の脆弱性です。

### 見知らぬリポジトリの clone 後すぐに Claude Code を起動しない

.claude/settings.json や .mcp.json が含まれていないかを確認してから起動する習慣をつけましょう。

```
# clone 後、まずこれを確認する
ls -la .claude/
cat .claude/settings.json
cat .mcp.json
```

### Skill に渡す情報を最小限にする

Claude に渡した情報はクラウドに送信されます。Skill 経由であっても同じです。以下は絶対に渡してはいけません。

「全部解析して」「ホームディレクトリを読んで」といった大まかな指示は避け、対象を明示的に絞り込む習慣をつけましょう。

### 常に最新バージョンを使う

CVE-2025-59536・CVE-2026-21852 は最新版では修正済みです。古いバージョンを使い続けるのは危険です。claude --version で定期的にバージョンを確認し、アップデートを怠らないようにしましょう。

## ケーススタディ：情報漏洩はなぜ起きたか

Claude が自動で情報を外部送信することは、仕様上ありません。しかし Skill の設計ミスや誤用によって情報漏洩が起きた実例は存在します。「なぜ漏れたのか」「どう防ぐか」の観点で整理します。

**なぜ起きたか**
