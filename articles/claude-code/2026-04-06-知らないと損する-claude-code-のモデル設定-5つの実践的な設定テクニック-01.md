Claude Codeを日常的に使っていると、「なんとなくデフォルトのまま」になっていませんか。モデル設定は初期状態でも動きますが、用途に合わせて調整すると作業効率とコストが大きく変わります。

この記事では、公式ドキュメントのモデル設定機能から「知っておくと明日から使える」5つのテクニックを厳選しました。チームで使っている方にも、個人で使っている方にも役立つ内容です。

## Tip 1: opusplan で計画は Opus、実行は Sonnet に自動切り替えする

複雑な実装作業の前に「まず設計を考えてほしい」という場面は多いと思います。設計フェーズには深い推論が必要で Opus が向いていますが、コードを書き始めたら Sonnet の速さが活きます。毎回手動で切り替えるのは手間がかかります。

`opusplan` はこの切り替えを自動でやってくれるエイリアスです。

動作は明確です。

| フェーズ | 使用モデル | 理由 |
| --- | --- | --- |
| プランモード（計画中） | Opus | 複雑な推論・アーキテクチャ判断 |
| 実行モード（コーディング中） | Sonnet | 速度と効率のバランス |

「Opusに頼みたいけどコスト気になる」という場面にちょうどいい設定です。設計部分だけ Opus を使い、実装はコスト効率の良い Sonnet が担当してくれます。

設定ファイルに書いておけば毎回指定する手間も省けます。

## Tip 2: effortLevel でモデルの「考える深さ」を調整する

「このバグ、なんで直らないんだろう」というデバッグ作業と、「このコメントを修正して」という単純作業では、モデルに必要な思考量が全く違います。

デフォルトは `medium` ですが、タスクの性質に合わせて変えると時間とコストの最適化になります。

```
# 単純な作業は low で速く
/effort low

# 複雑なデバッグや設計判断は high で
/effort high

# セッションを開始するときに指定することもできる
claude --effort high
```

4段階の使い分けの目安はこのとおりです。

| レベル | 向いているタスク |
| --- | --- |
| low | コメント修正、変数名変更、簡単なリファクタ |
| medium（デフォルト） | 通常のコーディング全般 |
| high | 難しいバグ調査、設計レビュー |
| max | 特に難解な問題（Opus 4.6専用、セッション非永続） |

毎回コマンドを打つのが面倒なときは、設定ファイルで固定できます。

```
{
  "effortLevel": "medium"
}
```

`high` や `max` に固定しっぱなしにすると、単純な作業でも深く考えすぎて逆に遅くなる場合があります。`medium` をベースにして、必要なときだけ上げるのが効率的です。

一時的に深く考えてほしい場合は、プロンプトに「ultrathink」と書くだけで、その一回だけ `high` 相当で動いてくれます。設定を変えずに使えるので地味に便利です。

## Tip 3: 1Mトークンコンテキストを[1m]サフィックスで有効化する

大きなコードベースを扱うとき「コンテキストが足りなくてファイルを渡せない」という状況にぶつかることがあります。Opus 4.6 と Sonnet 4.6 は100万トークンのコンテキストウィンドウに対応しています。

使い方はシンプルで、`[1m]` を付けるだけです。

```
# セッション中に切り替える
/model opus[1m]
/model sonnet[1m]

# フルモデル名でも同じように使える
/model claude-opus-4-6[1m]
```

プランや契約によって含まれるかどうかが変わります。

| プラン | Opus 4.6 の1M | Sonnet 4.6 の1M |
| --- | --- | --- |
| Max / Team / Enterprise | サブスクリプション込み | 追加課金 |
| Pro | 追加課金 | 追加課金 |
| API / 従量課金 | フルアクセス | フルアクセス |

環境変数でも指定できます。固定したいなら `ANTHROPIC_DEFAULT_OPUS_MODEL` に `[1m]` 付きで設定しておくと、alias を使うたびに1Mが有効になります。

```
export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6[1m]'
```

## Tip 4: availableModels + model + 環境変数の3点セットでモデルを完全固定する

チームで Claude Code を使っていると「人によって使っているモデルがバラバラ」「Claudeのアップデートで突然バージョンが変わった」という問題が出てきます。

`model` 設定だけでは不完全なのがハマりポイントです。`model` は初期選択を決めるだけで、ユーザーが `/model` ピッカーから `Default` を選べば、その人のサブスクリプション階層のデフォルトモデルが使われてしまいます。

完全に固定するには3つをセットで使います。

```
{
  "model": "claude-sonnet-4-5",
  "availableModels": ["claude-sonnet-4-5", "haiku"],
  "env": {
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "claude-sonnet-4-5"
  }
}
```

それぞれの役割は次のとおりです。

| 設定項目 | 役割 |
| --- | --- |
| `model` | セッション開始時の初期モデル |
| `availableModels` | ピッカーで選択できるモデルの制限 |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | `Default` や `sonnet` alias が解決するモデルを固定 |

`env` ブロックを抜かすと、ユーザーがピッカーで `Default` を選んだときに最新の Sonnet リリースが使われてしまい、意図的なバージョン固定をすり抜けてしまいます。

Bedrock / Vertex AI / Foundry 経由でデプロイしている場合は特に重要です。環境変数でモデルを固定しておかないと、Anthropic が新モデルをリリースしたとき、ユーザー側のアカウントに新バージョンが有効化されていないと静かに壊れます。

```
# Bedrock の例
export ANTHROPIC_DEFAULT_OPUS_MODEL='us.anthropic.claude-opus-4-6-v1'
export ANTHROPIC_DEFAULT_SONNET_MODEL='us.anthropic.claude-sonnet-4-6-v1'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='us.anthropic.claude-haiku-3-5-v1'
```

## Tip 5: ANTHROPIC\_CUSTOM\_MODEL\_OPTION で社内LLMゲートウェイをピッカーに追加する

社内のLLMゲートウェイやプロキシ経由で Claude Code を使っているケースで、カスタムのエンドポイントを `/model` ピッカーに追加したいことがあります。

`ANTHROPIC_CUSTOM_MODEL_OPTION` を使うと、既存のエイリアスを置き換えずにエントリーを一つ追加できます。

```
export ANTHROPIC_CUSTOM_MODEL_OPTION="my-gateway/claude-opus-4-6"
export ANTHROPIC_CUSTOM_MODEL_OPTION_NAME="Opus via Gateway"
export ANTHROPIC_CUSTOM_MODEL_OPTION_DESCRIPTION="社内LLMゲートウェイ経由のOpusデプロイ"
```

`_NAME` と `_DESCRIPTION` は省略できます。省略するとモデルIDがそのまま名前として使われます。

Bedrock の ARN のような形式の ID でも、モデルID のバリデーションがスキップされるのでそのまま設定できます。

```
# ARNでも問題なく動く
export ANTHROPIC_CUSTOM_MODEL_OPTION="arn:aws:bedrock:us-east-1:123456789012:custom-model/abc"
export ANTHROPIC_CUSTOM_MODEL_OPTION_NAME="Custom Bedrock Model"
```

さらに、Bedrock ARN のようにプロバイダー固有のIDを使うとモデルの機能が正しく認識されないケースがあります。その場合は `_SUPPORTED_CAPABILITIES` で明示的に宣言できます。

```
export ANTHROPIC_DEFAULT_OPUS_MODEL_SUPPORTED_CAPABILITIES='effort,max_effort,thinking,adaptive_thinking,interleaved_thinking'
```

これを設定すると、effort コマンドや拡張思考などの機能が正しく有効化されます。

## まとめ

Claude Code のモデル設定は「デフォルトで動かす」ためのものではなく、「用途に合わせて最適化する」ためのものです。

5つのテクニックを振り返ります。

1. **opusplan** - 計画は Opus、実行は Sonnet の自動ハイブリッド
2. **effortLevel** - タスクの複雑さに合わせて思考の深さを調整
3. **[1m]サフィックス** - 大きなコードベースに1Mトークンコンテキストを有効化
4. **3点セット固定** - チームでのモデルバージョンを完全に管理
5. **カスタムモデルオプション** - 社内ゲートウェイをピッカーに統合

まず試してほしいのは **Tip 2 の effortLevel 調整**です。`/effort` コマンドはセッション中にすぐ使えるので、今日から試せます。チームで使っている方は **Tip 4 の3点セット固定**を設定ファイルに入れておくと、バージョン管理の混乱を防げます。
