2026年1月に TOKIUM へ入社し、QA チームでテストの自動化に取り組んでいる ikedan です。  
これまでの記事では、Claude を「もう一人の同僚」として迎えたときに、[振り返りと試運転を仕込む話](https://zenn.dev/naoto70100304/articles/human-discipline-for-ai-colleague)や、[AIの知識をどこに置くかを整理する話](https://zenn.dev/naoto70100304/articles/claude-knowledge-organization)を書いてきました。どちらも「**1人のAI同僚を、1つのセッションの中でどう律するか**」という話でした。

本記事はそこから一歩スケールします。テーマは、**複数のAI同僚を同時に走らせたときに、その全員をどう回すか**です。  
前の記事で「先回りで止める仕組み」の記事を書くと宣言しましたが、脱線してこちらを執筆しました。

きっかけは、机の上が散らかったことでした。  
テスト設計の自動化が乗ってくると、やりたいことが一度に増えます。  
「この観点の生成」「あのE2Eの実行」「評価パイプラインの修正」を、1つのウィンドウで順番にやっていては待ち時間ばかりになる。  
そこで Claude Code を2つ3つと同時に立ち上げてみたら、今度は別の問題が起きました。

**どのウィンドウが何をやっているのか、自分でも分からなくなった**のです。

少し背景を書きます。私のチームがテストしているのは経費精算・請求書の SaaS で、私の仕事は、機能ごとに **テスト観点を作り → レビュー用に自動評価し → テストケースへ展開し → また評価する**、という多段のパイプラインを回すことです。  
ほかに、E2E の実行や、画面の調査、評価ロジックそのものの改修もあります。

ここで効いてくるのは、**これらの多くが「機能Aの観点生成」と「機能BのE2E」のように互いに独立している**ことです。  
1つの機能の評価ループ（数分かかります）を待っている間に、別の機能の観点生成を別の窓で進められる ── テスト設計の自動化は、構造的に並走と相性がいいのです。

これは QA に限った話ではありません。  
複数の feature ブランチを並行で進める、レビュー待ちのタスクが溜まる、生成 AI に独立した生成を何本もさせる ── そういう場面はどれも同じ形をしています。  
逆に言えば、1つの窓で足りているうちは、Foreman のような仕組みは要りません。2つ目の窓を開いて「あれ、どっちで何をやっていたか分からなくなった」と感じた瞬間から、この記事はあなたの話になります。

## 並走を始めたら、タスク管理が破綻した

複数の Claude を同時に走らせると、こんなことが起きました。

1. **どこに何があるか分からない**：タスクは引き継ぎ用のメモや、各ウィンドウの会話の中に散らばっていて、「未処理が全部でいくつあるのか」を一覧できない
2. **二重にやる**：別のウィンドウがもう着手しているタスクを、こちらは知らずにゼロから始める（実際、同じ機能のテストケース生成を、2つの窓が別々に走らせていたことがありました）
3. **取りこぼす**：「あとでやる」と思ったタスクが、どのメモにも残らず消える

一度、笑うしかない出来事がありました。

この協調の仕組みを実装している、まさにその最中に、ダッシュボードを見たら **別のウィンドウが、同じ仕組みの別パートを並行で実装していた**のです。  
私が土台を作り、相手が終了フローを作っていた。並走がぶつかることを、その対策を作る作業そのものが証明してしまった、という皮肉な一幕でした。

別の日には、こんなこともありました。  
私が長い画面調査をしている間に、別のセッションが同じ評価設定ファイルを書き換えていて、私は「さっき見た状態」のつもりで、そこに上書きをかけようとしたのです。  
編集の直前に行番号がずれていることに気づいて、ようやく並走に気づきました。コミット履歴を見てもまだ何も起きていないように見えたので、**まだコミットされていない変更こそが、別の窓で作業が進行中である一番確かなサインだった**、というのが教訓でした。

最初に試したのは、引き継ぎドキュメントを丁寧に書くことでした。  
でも、ドキュメントは「読んだ人がそう解釈した」という派生情報で、**いま本当に未処理なのはどれか**という問いには答えてくれません。文書が増えるほど、どれが最新かが分からなくなりました。

そこで、考え方を一段変えました。**未処理タスクの「正本」を、ただ1か所に決めてしまう。**

引き継ぎメモも、進行中の会話も、すべてはそこから派生したビューにすぎない、と。この「唯一の正本（SSoT）」を、追記専用の1ファイルに置いたのが Foreman の出発点です。  
これが、冒頭で挙げた問題①「どこに何があるか分からない」への、最初の答えでした。

## Claude 公式の「Agent View」とは、別の問題を解いている

未処理タスクの正本を1か所に決める ── そう決めたとき、公式機能を知っている人ほど、当然の疑問を持つはずです。「Claude Code には公式のマルチエージェント機能があるのに、なぜ自作したのか」。

2026年に入って、Claude Code には複数エージェントを束ねる公式機能が次々と入りました。私が調べた範囲では、主に4つあります。

| 公式機能 | 何をするか | 「次に何を走らせるか」を決めるのは |
| --- | --- | --- |
| [Agent View](https://code.claude.com/docs/en/agent-view)（リサーチプレビュー） | バックグラウンドの複数セッションを1画面で監視・返信する | ユーザー（＝指揮者） |
| [Subagents / Task tool](https://code.claude.com/docs/en/agents) | 1つのセッションの中から子エージェントを spawn して副作業をさせる | 親のClaude |
| [Dynamic Workflows](https://code.claude.com/docs/en/workflows) | 1つのスクリプトが多数のサブエージェントを並列に走らせ、結果を自動検証する（同時に走らせられる数には上限があります） | スクリプト |
| [Agent Teams](https://code.claude.com/docs/en/agent-teams)（実験的機能） | チームの共有タスクリストを teammate が自分で取って（claim）分担する | リーダー役のエージェント |

これらは本当に強力で、実はこの記事の下調べ（公式機能の事実確認や、私の台帳の集計）にも Dynamic Workflows を使いました。

ただ、**これらはすべて「縦の協調」です**。1人の指揮者（ユーザー、親Claude、スクリプト、あるいはチームのリーダー役）がいて、その配下のサブエージェントやメンバーをまとめる構造。誰が次に何を走らせるかは、その指揮者が決めます。

一方で私がぶつかっていたのは、**「横の協調」でした**。

独立した人間主導のセッションが何個も立ち上がっていて、互いに対等で、指揮者はいない。  
それぞれが同じコードベースを並走で触り合う。

この「**指揮者のいない対等なセッション同士が、ロックも使わずにどうぶつからずに済ませるか**」というユースケースは、公式機能が主に想定している形とは、少し違っていました（共有タスクリストを持つ公式機能もあります。記事の後半で改めて触れます）。

**公式：縦の協調（指揮者あり）** ── 1人の指揮者が配下をまとめる

**Foreman：横の協調（指揮者なし・対等）** ── 対等なセッションが台帳で申し送りする

この「横の協調」を、ロックも中央サーバも使わずに成立させようとしたのが、ここから説明する Foreman です。

## Foreman とは何か ── ロックも中央サーバも持たない協調機構

Foreman（フォアマン＝現場監督）は、並走する複数の Claude セッションが、**相互排他ロックなしで**タスクの受け渡しと衝突回避を成立させるための仕組みです。  
中央サーバもデータベースも持たず、ファイルシステム上の「追記専用ログ」と「セッションごとの小さなファイル」だけで構成しています。

設計の柱は5つです。

| 柱 | 中身 |
| --- | --- |
| **advisory / no-lock** | フックは他のセッションを止められない（後述の構造的制約）。だから衝突は「検知して知らせる」だけにする |
| **固定パスに集約** | 台帳とセッション一覧は、作業ディレクトリに依存しない固定の場所（`~/.claude/hooks/`）に集める。場所がバラバラだと、別フォルダで起動したセッション同士がお互いを見えなくなるから |
| **event-sourcing（追記のみ）** | 正本の台帳は1行1イベントの追記専用ファイル。状態を変えるときは「上書き」ではなく「新しい行を足す」。読むときに畳んで最新状態にする |
| **fail-open（壊れても止めない）** | 監視や協調の仕組みがバグで落ちても、AI本体の作業は止めない。例外は握りつぶして、黙って通す |
| **書き込むのは4か所だけ** | 共有ファイルに書くのは、心拍記録・セッションID記録・台帳追記・衝突監査の4つだけ。集計して見せる側は読むだけで書かない |

中でも要は **advisory（助言）** という割り切りです。  
理由は、Claude Code のフック（コマンドや編集の直前に割り込める自前スクリプト）には、はっきりした構造的制約があるからです。

**フックは、隣で走っている別のセッションを停止したり・順番待ちにさせたり・強制終了したりできません。**

割り込めるのは「自分のセッションの、これから動かす操作」だけ。隣のウィンドウには手が届かない。

実は以前、素直にロックで解決しようとして一度つまずいています。  
共有ファイルにロックを取らせる方式を試したところ、ロックファイル周りのエラー（文字化けや権限エラーでのクラッシュ）が積み上がって、ひとつのログが百数十KBにまで膨れ、**ロックで守るはずの仕組みが、ロックのせいで壊れる**という本末転倒になりました。  
その反省から、ロック方式そのものを捨てています。

だから Foreman は「止める」ことを諦め、**ぶつかりそうなら知らせる／正しい順番は台帳で共有する**ことに振り切りました。  
これはデータベースでいう「楽観的並行制御（衝突は滅多に起きない前提で、ぶつかったときだけ後始末する）」と同じ発想です。

役割は2つに分かれます。

仕事を撒く側が **orchestrator**（起点セッション。タスクの整理・台帳の整合・撒く先の生成・進行の監視を担う）、

撒かれて実作業をする側が **worker**（別ウィンドウのセッション。1タスクを自分で確保＝claim して、実装やE2Eや執筆をやり、終わったら done を打つ）。

ここで Foreman が自分に課している規律が1つあります。

**orchestrator は、実作業を自分のウィンドウの中でやらず、必ず worker の別ウィンドウへ撒く。**

起点が抱え込むと、結局1つのウィンドウに仕事が詰まって並走の意味がなくなるからです。

名前で誤解されそうなので、先に断っておきます。「Foreman（現場監督）」と名付けましたし、仕事を撒く orchestrator という役も出てきます。  
「結局それが指揮者では？」と思うかもしれません。でも、これは部下を仕切る“監督者”とは少し違います。イメージしてほしいのは、現場監督が壁に掛けておく“作業台帳”のほうです。

たしかに、次の仕事を台帳へ並べ、worker の窓へ渡す役（orchestrator）はいます。  
けれど、それは固定の司令塔ではなく、そのとき仕事を撒いているセッションが一時的に担うだけの役です。  
しかも、いったん走り出した他のセッションを止めることも、命令し直すこともできません（前に書いたとおり、その権限がそもそも無いからです）。  
各セッションは、受け取った仕事を自分で確保（claim）して進め、途中経過を上へ“報告”する上司もいません。書き込む先は、みんなが同じように読み書きする1枚の台帳だけです。  
全体を仕切っているのは、人でも中央のエージェントでもなく、その台帳そのもの ── だから私は“指揮者なし”と呼んでいます。

## 仕組み ── 「歩く」5つのコマンド

Foreman の運用は、5つのコマンドを「歩く」ことに集約されます。難しい同期処理はなく、着手前と終了時に台帳を確認する、という地味な手順です。

* **`/sessions`**（読むだけ）：いま誰が・どのブランチで・生きているか・衝突の疑いはないかを一覧する
* **`/pickup`**（着手前）：台帳から次のタスクを確保（claim）する（台帳へ書き込むのは、この claim と、終了時の `/closeout` の登録の2つだけ）
* **`/dispatch`**（撒く）：orchestrator が worker を別ウィンドウとして起こす起動コマンドを生成する
* **`/closeout`**（終了時）：コミットから「次のタスクの台帳登録」までを畳む
* **`/foreman`**（運用モードを読み込む）

とくに着手前の `/pickup` には、「**黙って奪わない**」という原則が効いています。流れはこうです。

1. いま生きているセッションの一覧を読む
2. これから自分が触るファイルが、他の生きたセッションが直近5分以内に書いたファイルと被っていないか照合する。被っていたら対象や順番を変える
3. タスクを確保する。もし別の生きたセッションが既に確保済みなら、**勝手に奪わず、人に「奪う／別のタスクにする／待つ」を尋ねる**
4. 死んでいるセッションが握ったまま放置されたタスクなら、解放してから確保する

さらにもう一段、確保の前に「**そのタスクは、自分が把握しているものか**」も確かめます。  
台帳の各タスクには「誰が・いつ作ったか」の印が付きます。

**別のセッションが今日になって積んだ、自分の知らないタスク**は、中身のドキュメントに潜り込む前に、いったん差し戻す（release）のが既定です。

無人で走らせている AI 同士だと、誰も頼んでいない“幻のタスク”が台帳に積み上がることがあり、それを人が反射的に拾って走り出さないための、地味だけれど効くゲートです。

終了時の `/closeout` も同じ思想です。  
「検証なき完了なし」を前提に、完了を別の手段で実証してから、意味のある単位でコミットし、**残った作業を必ず台帳に登録し直してから**畳む。  
次に拾う worker が迷わないよう、登録時には ready\_prompt の先頭に【種別｜起動する skill｜完了の定義】というヘッダを置きます。こうして「次の一手」を決定的に手渡します。  
着手前の照合が問題②「二重にやる」を、終了時の登録が問題③「取りこぼす」を、それぞれ受け持つ形です。

技術的に一番おもしろかったのは、**台帳の状態を「畳む」ところ**です。  
台帳はただの追記専用ログなので、同じタスクの状態を変えるときも、行を上書きするのではなく、新しいイベント行を足すだけ。  
読むときに、同じIDの行を時系列で畳み込んで「最新状態」を作ります。

畳み方の肝は、**「状態を含むイベントだけが状態を動かし、状態を含まないイベントは今の状態を保つ」という加法的なルール**にしたことです。

おかげで「このタスクを別ウィンドウに撒いた」という情報を、タスクの進行状態を一切壊さずに後から足せます。ロックも、比較交換（CAS）も、上書きも要りません。

文章だけだと分かりにくいので、1つのタスクに起きたことが台帳にどう積まれるかを、簡略化して見せます。

```
// 同じ id の行が、上書きされずに1行ずつ積まれていく
{"event":"add",   "id":"L-...-a1b2c3", "status":"queued",  "title":"機能Xのテスト観点を作る"}
{"event":"claim", "id":"L-...-a1b2c3", "status":"claimed", "by":"session-7f43"}
{"event":"done",  "id":"L-...-a1b2c3", "status":"done"}
// 読むときに同じ id を時系列で畳む → この id の最新状態は status:"done"
```

実際のイベントにはもう少しフィールドがありますが、要点はこれだけです。  
状態を持つイベント（`claim` や `done`）だけが status を動かし、状態を持たないイベント（「別ウィンドウに撒いた」の記録など）は status に触れずに情報だけを足す。  
この「後勝ちで畳む」ルールひとつで、ロックも上書きも無しに状態管理が成り立ちます。

### 例：ひとつのテスト設計タスクが流れていく

言葉が続いたので、実際の仕事がどう流れるかを一度追ってみます。「ある機能のテスト観点を作る」という仕事を例にします。

1. 起点の窓（orchestrator）で、この仕事を台帳に登録（add）する。先頭には【観点生成｜使う skill｜完了の条件】というヘッダを付けておく
2. `/dispatch` で worker の別窓を起こす。**確保（claim）は、起こされた worker が自分でやる** ── 「誰がやっているか」の記録を、実際に手を動かす側に持たせるためです
3. worker の窓では、観点生成 → 自動評価 → テストケースへの展開、という多段パイプラインが回る。その間、起点や他の窓では、別の機能の E2E や評価ロジックの改修が並行で進んでいる
4. 終わったら worker が done を打ち、残りを台帳に書いてから畳む（closeout）

この4ステップのどこにも、「他の窓を止める」操作が無いのがポイントです。各窓は台帳を読み書きするだけで、全体を仕切る指揮者はいません。  
それでも、同じ仕事を二重にやることも、途中で取りこぼすことも、起きにくくなります。

## 余談 ── 生きているか確認しようとして、危うく相手を殺しかけた話

死んだセッションのタスクを別のセッションが引き継ぐには、「相手がもう落ちているか」を知りたくなります。

ここで一度、技術的にいちばん怖い地雷を踏みました。

プロセスの生死を確認する定番に `os.kill(pid, 0)` というコードがあります。  
多くの解説で「シグナルを実際には送らず、相手が存在するかどうかだけ分かる安全な確認方法」と紹介されます。

ところが **Windows の Python では中身が違い、シグナル0でも内部で対象プロセスを終了させてしまう**ことがあります。

生死を見ようとした瞬間に、監視するはずの別セッションを巻き添えで落としかねない。

「止める権限が無いはずなのに、生きているか確認しようとしただけで相手を殺してしまう」── この記事のテーマそのものの形をした事故でした。

今は `os.kill` を避け、OS の関数を直接呼んで「終了済み／まだ生きている／判定できない」の三段階で見ています。  
そのうえで判定を**非対称**にしました。**死んだ確証があるときだけ引き継ぎに動き、「生きている」「分からない」は安全側に倒して何もしない。**

確実な情報だけで動き、曖昧なときは触らない。この非対称な設計は、「ぶつかる前に避ける」という台帳運用の思想と、まったく同じ形をしていました。

## Agent View に何を足したのか ── 差分を整理する

縦と横の違いを踏まえたうえで、Foreman が Agent View に具体的に何を足したのかを整理します。

前半で挙げた4つの公式機能のうち、監視という土台の面で Foreman にいちばん近いのは Agent View です。  
どちらも「複数の背景セッションを扱う」道具だからです。ただ、両者は対立するものではありません。

**Foreman は、Agent View がくれる土台をそのまま活かして、その上に「横の協調」の層を足したもの**、という足し算の関係になっています。

下の表は、上の3行が Agent View がすでに提供している土台（Foreman もそのまま使う部分）、そこから下が **Foreman で足した差分**です。

| 機能 | Agent View | Foreman |
| --- | --- | --- |
| 複数の背景セッションを1画面で監視する | ✓ | ✓ |
| 走っているセッションに返信・介入する | ✓ | ✓ |
| タスクを投入してバックグラウンドで走らせる | ✓ | ✓ |
| **↓ ここから下が Foreman で足した差分** |  |  |
| 未処理タスクの共有正本（台帳・SSoT） | ― | ✓ |
| セッション同士のタスク受け渡し（確保／引き継ぎ／差し戻し） | ― | ✓ |
| 同じファイルの並走編集・二重起動を「検知して知らせる」 | ― | ✓ |
| セッション死活の三値判定＋放置された確保の自動回収 | ― | ✓ |
| 着手前・終了時に申し送りを歩くコマンド群（/pickup・/closeout など） | ― | ✓ |

上半分（監視・返信・タスク投入）は、Agent View がうまくやってくれる部分です。  
Foreman はそこを作り直したりはせず、むしろ worker を撒くときに **Agent View をそのまま開いて使う**こともあります。  
その土台の上に、下半分 ── 対等なセッション同士が同じコードベースでぶつからないための「申し送りの層」── を重ねたのが Foreman です。

衝突の避け方も、よく「対照的だ」と語られます。  
公式の並走は、作業場所（git の worktree）を分けて「そもそも同じファイルを触らせない」ことで衝突を予防します。いっぽう Foreman は、同じ作業ツリーを共有したまま、台帳で「ぶつかりそうだ」と知らせ合います。

ただ、この2つを「逆向きの思想」として対立させるのは、正直にいうと少し誇張でした。理由は2つあります。

ひとつ目。同じファイルへの書き込み衝突そのものは、作業場所を分ければ根絶できます。あとから検知して知らせるのは、分けなかったための後始末であって、予防より強い手ではありません。実際、私も index やコミットの競合で何度か後始末をしてきました。ですからファイルの衝突だけを見れば、分離するほうが強い ── ここは公式のやり方に分があります。

ふたつ目。では Foreman はなぜ分けないのか。じつは台帳は、作業ツリーの外（固定パス）に置いてあります。つまり「未処理タスクを一元管理して二重実行を防ぐ／セッションを跨いで申し送る」という Foreman の中心は、worktree で分けても壊れません。両者は本来 either/or ではなく、重ねられます。私が同じツリーを共有しているのは「検知のほうが偉いから」ではなく、数時間〜数日つづく対話セッション同士が「いま誰が何を触っているか」をその場で見合える使い勝手 ── trunk ベースの継続的統合に近い感覚 ── を優先したからです。これは予防と引き換えの選択で、正解がひとつに決まる話ではありません。

もうひとつ、正直に付け加えます。実は公式にも「共有タスクリスト」を持つ機能があります ── 前半で挙げた表の Agent Teams（実験的機能）です。  
チームのタスクを teammate が自分で取って進める点は、Foreman の台帳とよく似ています。  
ただ Agent Teams は、(1) リーダー役のエージェントが仕切る（＝指揮者がいる）、(2) 1つのセッションに閉じたチームで、独立したセッションを跨げない、(3) 取り合いの防止にファイルロックを使う（私がまさに捨てた方式）── という設計でした。

だから、私が欲しかった「**指揮者なし・独立したセッション同士・ロックなし**」の並走協調は、Agent Teams でも埋まりませんでした。

言い換えると、Foreman は Agent Teams と同じ「各自がリストからタスクを取る」やり方を採りながら、**そのリーダー役だけを外したもの**です。

この“リーダー不在”こそが、Foreman に司令塔がいないという意味です。裏を返せば、私に足りなかったのは公式機能そのものではなく、公式が主に想定する「縦」のモデルだけでは埋まらない「横」の並走のほう、ということになります。

更に言い換えると、Foreman は Agent View の置き換えではなく、**Agent View に「共有台帳」と「衝突回避」を足したもの**です。

## 実際に17日間まわして、件数を数えてみた

ここからが本題です。

「便利になった気がする」では記事になりません。

台帳は追記専用なので、過去のイベントが全部残っています。これを集計すれば、**自分の働き方が実際にどう変わったかを数字で見られる**はずです。やってみました。  
対象は、台帳が動き始めた **2026年6月13日から7月1日までの約2週間半**。生のイベント行は約1,200行ありました。

### 導入前 ── 数えることはできても、動かせなかった

正直に言うと、Foreman を入れる前も、タスクを「数える」こと自体はできました。  
バックログや引き継ぎメモに書き出せば、残りがいくつあるかくらいは分かります。

本当に困っていたのは、そこではありませんでした。**書き出したタスクを、記録として“動かす”手段が無かった**のです。

たとえば、誰が着手中かを記録して二重取りを防ぐ、状態を「着手 → 進行 → 完了」と遷移させる、終わったものや不要になったものを掃除する、溜まった未整理のタスクを「今すぐ着手できる／保留／もう不要」に仕分ける ── こうした操作が、バックログの Markdown ではできませんでした。  
Markdown は“読む”ためのもので、タスクを操作する受け皿ではなかったからです。だから、古いタスクが誰にも触られないまま積み上がっていきました。

Foreman では、タスクが追記専用の台帳に載る“レコード”になり、ひととおりの操作（CRUD）と整理ができるようになりました。導入の前後を、操作の観点で並べるとこうなります。

| 操作 | 導入前（バックログ Markdown） | 導入後（Foreman の台帳） |
| --- | --- | --- |
| 作成（Create） | 手で書き足す | `add`（種別・完了条件つきで登録する） |
| 読取（Read） | 目視で数える | `list`／`next`（状態・担当・鮮度つきで一覧する） |
| 更新（Update） | 上書きする（誰の版が最新か曖昧） | `claim`／`progress`／`done`（状態遷移が履歴に残る） |
| 削除（Delete） | 行を消す（履歴ごと消える） | `cancel`／`release`（取り消し・差し戻しも履歴に残す） |
| 整理（Triage） | 手作業・基準なし | 未整理のタスクを「着手可／保留／もう不要」に仕分け、まとめて掃除する |

数えるだけなら、以前もできました。変わったのは、**タスクを“動かせる記録”にできた**ことです。  
この「整理（トリアージ）」の一手が入ったことで、溜まった未処理を放置せず、着手できるものから捌けるようになりました。

台帳を入れた最初の数日も、まだ綱渡りでした。

1つのセッションで回していた6月14〜15日は、**タスクを追加する速度が、片付ける速度を上回っていました**。  
たとえば6月15日は、33件を新規に積んで、終わったのは12件。差し引き21件が、その日のうちに backlog へ溜まっていく。1つのウィンドウで順番にこなす限り、やりたいことの増加に処理が追いつかなかったのです。

### 導入後 ── 別ウィンドウへ撒けるようになって、流れが変わった

転機は6月16日でした。台帳をSSoTにして、orchestrator が worker の別ウィンドウへタスクを撒く運用が回り始めると、1日に台帳へ活動を記録するセッション数が一気に増えました。この期間ぶんを集計すると、こうなります。

| 指標 | 実測値（2026-06-13〜07-01・約2週間半） |
| --- | --- |
| 追跡したタスク（distinct） | **342 件** |
| 完了（done） | **289 件（完了率およそ 85%）** |
| キャンセル / 未完了のまま（進行中・未着手） | 25 件 / 28 件 |
| 起点とは別のウィンドウが処理したタスク | **256 件（全体の約 75%）** |
| 別セッションへ引き継がれたタスク | 21 件 |
| 1日に台帳へ活動を記録したセッション数（最大） | **35**（※同時並走ではなく、1日の延べ） |

この342件が実際に何だったかも、台帳から拾えます。  
種別を付けていたものだけを数えると、E2E の実行・修正が18件、テスト観点やテストケースの生成が14件、画面の調査が13件、評価パイプラインの改修が9件、そのほかツール整備などが続きます（大半はきちんと種別を付けていないので、あくまで内訳の目安です）。

いずれも、逐次なら1つずつ順番待ちになっていた種類の作業です。

週ごとの「完了した件数」を並べると、流れの変化がはっきり出ます。

台帳が動き出した最初の数日（週末の6/13〜14）に完了したのは10件でした。  
並走が回り始めた最初のフル稼働週は100件、次の週は134件まで上がります。

そして**追跡したタスク全体の約75%（256件）は、私が起点で作ったのとは別のウィンドウのAIが処理していました**。1つのウィンドウで順番待ちしていたら、物理的にこなせなかった量です。

注目したいのは、件数だけでなく **backlog の向き**が変わったことです。  
導入直後は「追加＞完了」で溜まる一方でしたが、並走が定着してからは、完了が追加に追いつき、最終的に追跡した342件のうちおよそ85%を片付けられました。

**やりたいことの増加に、処理がようやく追いついた**のです。これが、私が実感している一番大きな変化です。

ひとつ補足します。ここで数えている「完了」は、ただ手を動かし終えた数ではありません。  
とくにテスト観点やテストケースの生成については、そのつど自動評価（eval）にかけ、私たちの品質基準（100点満点で90点以上）を通ったものだけを「完了」として数えています。

**件数を並走で稼いでも、この品質の門は下げていません**。速さのために質を落としていないか ── そこは測って担保しているつもりです。

## 測っても分からないこと ── この運用の正直な限界

ここまで数字を並べてきましたが、守れることだけを書くと信用できない記事になるので、**この数字が証明していないこと**も、はっきり書いておきます。限界は4つあります。

**1つ目は、「35セッション」は同時並走ではないこと。** これは「その日のうちに台帳へ何かを書いた、入れ替わり立ち替わりの延べセッション数」です。実際に同じ瞬間に動いていたのは、せいぜい数本（私の体感と他の検証では3〜5本がスイートスポット）。瞬間最大同時数ではない、という但し書きが要ります。

**2つ目は、件数の増加を Foreman 単独の効果と切り分けられないこと。** この17日間は、ちょうど私が Claude Code の使い方に習熟した時期であり、ツール改善の作業（harness の整備）に集中していて稼働密度が高かった時期でもあります。完了件数が増えたのは、Foreman のおかげなのか、私が単に慣れて多く働いたからなのか、両方なのか ── n=1 の運用ログでは、原因をきれいに分離できません。「台帳を入れたら2倍になった」と言いたくなるのをこらえて、「並走で撒けるようになり、こなせる総量が物理的に増えた」という、構造として説明できる部分だけを主張に留めます。

**3つ目は、「追加した件数」は必ずしも「やった量」ではないこと。** 台帳を使い始めれば、当然これまで記録していなかったタスクも記録するようになります。だから add が増えたのは、働きが増えたのか、記録が増えただけなのか、曖昧さが残ります。ただ、完了（done）の方もおよそ85%という高い率で連動して増えているので、「記録が増えただけ」では説明しきれない、とは言えそうです。

**4つ目は、この運用が結局のところ人に依存していること。** Foreman は advisory（助言）であって強制ではありません。フックは他セッションを止められないので、安全は「着手前と終了時に台帳を歩く」という人の習慣に乗っています。実際、終了時の締めの確認（closeout）がちゃんと記録されたのは、全セッションの1割にも満たないのが現状です。歩く運用をサボれば、二重作業も取りこぼしも、また起きます。仕組みで全部を縛れてはいない、というのが正直なところです。

ただし、この「人に依存している」部分のうち、「同じファイルへの書き込み衝突」だけは、もっと構造的に潰す手が残っています。各セッションを別々の作業場所（git の worktree）に分ければ、書き込みの衝突は「検知して知らせる」に頼らず根絶できます。しかも台帳は作業ツリーの外（固定パス）に置いてあるので、「台帳はそのまま／作業場所だけ分ける」という重ね掛けができます。じつは、これは以前から見えていた宿題で、まずは台帳づくりを優先したぶん、後回しにしていました。次は、書き込み衝突が実害を出しているファイルから、この「台帳＋worktree」のハイブリッドを試すつもりです。

この4つ目に関連して、もうひとつ正直な話をします。  
「orchestrator が、撒くべき仕事を自分のウィンドウで抱え込んでしまうのを、機械的に止められないか」を検討しました。

けれど、**作る前に独立したエージェントで敵対的に検証したところ、これは止められない（No-Go）と結論しました。**

orchestrator かどうかを判定する決定的な手掛かりが無く、無理に止めれば正当な作業まで誤検知する。対象も事実上1人で、計測する母数が小さすぎる。  
だから「弱い手掛かり＋自問」で据え置きにしました。**止められないものを、止められるふりをして作らない。** これは前にも書いた「測ってから決める」姿勢の、もうひとつの形です。

最後に、コストの話も正直に書いておきます。  
複数の窓を同時に走らせれば、その数だけ利用枠（トークン）を消費します。  
**並走が得になるのは、独立して進められるタスクが何件も積み上がっているときだけ**で、単発の作業では、並走はむしろ割に合いません。

数を増やすこと自体が目的にならないよう、使いどころは絞っています。

## Foreman を使わなくても持ち帰れる3つのこと

ここまで私の自作ツールの話をしてきましたが、Foreman そのものを真似する必要はありません。  
この17日でいちばん効いたのは、実はツールではなく、その裏にある3つの判断でした。並走する AI を扱わない人にも、たぶんそのまま使えます。

1. **止められないなら、止めるふりをしない**。強制できない相手には、ロックで縛るより「知らせるだけ」の助言（advisory）に徹したほうが、仕組みは壊れにくい
2. **生死の判定は非対称にする**。「死んだ確証があるときだけ動く／生きている・分からないは何もしない」。確実な情報だけで動き、曖昧なら触らない
3. **危ないものを全部止める前に、測る**。機械的に止めるか（block）は、誤検知を測ってから決める。止めても意味がない・止めきれないものは、正直にそう認めて作らない

道具は環境に依存しますが、この3つは、たぶん依存しません。

## おわりに ── 伝える・整える・束ねる

ここまでの記事をつなげると、こうなります。

| 記事 | やったこと | 効かせ方 |
| --- | --- | --- |
| 1本目 | 振り返りと試運転を AI に翻訳 | 起きた後に学び、作った直後に試す |
| 2本目 | 知識を4つの置き場所に整理 | 何を読むかを整える |
| 本記事 | 複数のAIを台帳ひとつで束ねる | 止めずに、台帳で申し送りする |

1〜2本目は、1人のAI同僚を1つのセッションの中で律する話でした。  
本記事は、複数のセッションが同じ場所を取り合うときに、**指揮者を立てずに、台帳ひとつで申し送りして束ねる**話です。

効いたのは、「もっと強く統制する」ことではありませんでした。  
フックは他セッションを止められない ── その構造的制約を嘆くのをやめて、**止める代わりに、未処理の正本を1か所に決め、ぶつかる前に確認し合う**。  
この地味な申し送りに振り切ったとき、私のこの2週間半では、こなせるタスクの総量が目に見えて増えました。同時に、その数字が Foreman 単独の手柄だとは言い切れない、という限界もはっきり見えました。

人とAIが一緒に品質を守る時代に、さらに「複数のAIを同時に走らせる」時代が重なってきます。  
そのとき効くのは、全員を順番待ちにさせる強い力ではなく、**指揮者がいなくても互いの仕事を確認できる、ゆるい台帳**のほうかもしれません ── 少なくとも、現場監督（Foreman）を名乗らせた私の自作ツールは、そういう働き方をしていました。

次は、この台帳の上で「無人で仕事を撒き続ける全自動ループ」を、どこまで人の手から離せて、どこから先は離せないのか ── その境界線について、また別の記事で書こうと思います。
