**この記事の実施記録**: Claude Code（CLI）と Cowork（Claudeデスクトップアプリ）の併用運用を実作業で確立した。2つを並べて立て、ローカルファイル経由で繋ぐ「cowork-cli-bridge」プロトコルで非同期に分業させ、`/note-post` スキル新設を完了。Cowork がスキル設計・実装を担当し、CLI がデプロイと git 操作を引き受けた。委譲から完了まで2分47秒（実測: `cli-pending.md` の `created_at` → `completed_at` の差分。人間が CLI に声をかける時間も含む）。

---

Claude Code（CLI）とCowork（Claudeデスクトップアプリ）は、同じMac上で動作する2つの独立したセッションです。それぞれ「できること・できないこと」が異なり、ローカルファイルを介して非同期に連携する「cowork-cli-bridge」プロトコルで組み合わせると、どちらか単体では詰まるタスク（デプロイ・ブラウザ操作・フレッシュなレビュー）を分業で完結できます。実測では、Coworkがスキル設計を担当しCLIがデプロイとgit操作を引き受けた委譲が2分47秒で完結しました（人間が CLI に「pending 見て」と声をかける時間を含む）。

---

Claude Code（CLI）をしばらく使い続けると、「このセッション、文脈が溜まりすぎた」という感覚が出てきます。あるいは「このタスクはブラウザ操作が必要だけど、CLI からは触れない」という壁に当たることもある。

そのときに「もう1つの Claude である Cowork（Claudeデスクトップアプリ）を立てる」という選択肢が出てきます。Claude Code と Cowork は別製品で、できること・できないことが綺麗に分かれているからです。

Cowork（Claudeデスクトップアプリ）と、Claude Code（CLI）。同じ Mac で動いている2つの別系統のセッションを、どう繋ぎ、どう使い分けるか。2026年5月の実運用で見えてきたパターンをまとめます。

## CoworkとCLIは「どちらが上位」ではなく補完関係

最初に誤解を解いておきます。CoworkとCLIは、どちらかがどちらかの「上位互換」ではありません。それぞれに「できること」と「できないこと」があります。

この非対称性が、2つを組み合わせる動機になっています。

| 比較軸 | Cowork | CLI（Claude Code） |
| --- | --- | --- |
| ファイル操作（Read/Write/Edit） | ワークスペース内は自由 | permissions設定の範囲で自由 |
| `~/.claude/`へのデプロイ | 不可（サンドボックス制約） | 可能 |
| gitコマンド実行 | 不可 | 可能 |
| ブラウザ操作（note/Zenn公開等） | 得意 | 不可 |
| 長文レビュー・構成評価 | フレッシュなコンテキストで評価できる | 長いセッションで文脈に引っ張られやすい |
| 方向性の壁打ち | ワークスペース共有で「ファイルを見ながら相談」が成立 | コンテキストが膨らんでいると先入観が混ざる |

（出典: `ops/playbooks/cowork-cli-bridge.md`、2026-05-10版）

端的にいうと:

* **「成果物の中身を作る」** → どちらでも可（レビューはCoworkが有利）
* **`~/.claude/`・`.git/`・スクリプト実行** → CLI一択
* **ブラウザ操作** → Cowork一択
* **「このセッション、先入観が混ざってきた」** → Coworkに外部レビューを頼む

## ファイル1本で委譲できる仕組み「cowork-cli-bridge」

「2つ立てるのは面倒では？」という疑問に先に答えます。

委譲の操作は、**ファイルを1本書くだけ**です。

`ops/active/cli-pending.md`（または`cowork-pending.md`）という指示書ファイルを置く。それが「メッセージキュー」になります。

### ファイルの構造

```
---
created_at: 2026-05-11 22:42:37
source: cowork
status: pending
---

# CLI指示書

## 依頼内容
（自然言語で作業内容を書く）

## 読んでほしいファイルのパス
- /Users/saitoko/workspace/000-partner/workspaces/skill-dev/note-post/SKILL.md

## 完了処理
完了後、このファイルのstatusを`completed`に書き換えてください。
```

受け取る側は、このファイルを読んで作業を実行し、frontmatterの`status`を`pending`から`completed`に書き換えます。

ポイントは3つです。

1. **外部APIを介さない**: ローカルファイルをバスにするため、遅延ゼロ。gitで履歴も残る
2. **状態管理はシンプルな4状態**: `pending` → `in_progress` → `completed`（または`cancelled`）
3. **削除しない**: status書き換えで完了を記録する。後から「いつ・誰が・何を完了したか」を追跡できる

この設計を私は「cowork-cli-bridge」と呼んでいます（私が個人的につけた名称で、公式用語ではありません）。Playbookとして`ops/playbooks/cowork-cli-bridge.md`に実装済みです（2026-05-10更新）。

## 実例: `/note-post`スキル新設の2分47秒

「2〜3分で完了する」と書いても、実感が湧きにくいと思います。実例で見てみます。

### 何をやったか

note.comへの記事投稿手順をCoworkに指示する`/note-post`スキルを新設しました。

* **設計・実装**: Cowork（skill-devエージェント）が担当
  + `workspaces/skill-dev/note-post/SKILL.md`（318行）を新規作成（実測: `wc -l`）
  + 設計書（551行）・実装報告書（175行）も同時作成
* **デプロイとgit操作**: Coworkのサンドボックス制約で実行できないため、CLIに委譲

### 委譲の時系列

Coworkは22:42:37に`cli-pending.md`を書き込みました。指示内容は3ステップ：

1. `.git/index.lock`の削除（前セッションの残骸）
2. `~/.claude/commands/`への配置（SKILL.mdをコピー）
3. git add / commit / push

CLIが作業を完了し、`completed_at: 2026-05-11 22:45:24`を記録しました。

**経過時間: 2分47秒**（`cli-pending.md` の `created_at` → `completed_at` の差分）。この時間には、私が CLI 側のセッションに「pending 見て」と声をかける操作・CLI が読み取って実行に取りかかるまでの待機も含まれます（完全自動化のスループットではありません）。

なお、デプロイ済みの`~/.claude/commands/note-post.md`は11869 bytes（実測: `wc -c`）です。

### 実際の`cli-pending.md`はこう書かれている

「指示書に何を書けばいいかわからない」という疑問に答えるため、今回使った実物の抜粋を掲載します（frontmatterと依頼冒頭部分）。

```
---
created_at: 2026-05-11 22:42:37
source: cowork
status: pending
---

# CLI指示書

`/note-post` スキル新設の最終2ステップ（Cowork サンドボックスで実行不可だった作業）を実行してください。

## 背景

Cowork セッションで `/note-post` スキルの実装が完了しました（架構: architect、実装: skill-dev）。
ただし、Cowork のサンドボックスでは以下が実行できませんでした:
1. `.git/index.lock` の削除（OS 側で削除拒否）
2. `~/.claude/commands/` への配置（サンドボックス外）
3. git commit / push

## 依頼内容

### ステップ2: SKILL.md を本番デプロイ

cp /Users/saitoko/workspace/000-partner/workspaces/skill-dev/note-post/SKILL.md \
   ~/.claude/commands/note-post.md

（以下ステップ3のgitコマンド等が続く）
```

ポイントは「**背景説明 → できなかった理由 → 具体的コマンド**」の3層構成です。CLIはこれを読んで文脈を把握し、コマンドをそのまま実行できます。指示書の粒度としては「コピペで動くコマンド」まで書くのが目安です。

### 「往復のオーバーヘッドが増えないか」への回答

この委譲にかかった操作は「Coworkがファイルを1本書く」だけです。CLIが次にそのファイルを読んで作業を実行する。**CLIへの声かけ（「pending見て」と伝える）は人間が手動で行う**手動トリガー方式です。自動ポーリングではありません。

2分47秒という数値は、Cowork が指示ファイルを書き終えた瞬間から、CLI が status を completed に書き換えるまでの**合計経過時間**です。**この中には、私が CLI に「pending 見て」と声をかける操作・CLI が読み取って実行に取りかかるまでの時間も含まれています**。完全自動化のベンチマークではなく、人間がトリガーを引く前提の実運用記録として読んでください。それでも「Cowork で設計しながら、適当なタイミングで CLI に流す」という非同期の使い心地が成立する程度の速度感ではあります。

## Coworkをフレッシュなコンテキストの外部レビュアーとして使う

CoworkとCLIの使い分けで、見落とされやすい利点があります。**コンテキストが違う**という点です。

CLIは長いセッションを積み上げるほど、その文脈に引っ張られた判断をしやすくなります。「これが正しい方向だ」という先入観が溜まっていく。

Coworkはセッションのたびにフレッシュなコンテキストで見ます。ワークスペースのファイルは共有できるので、「このファイルを見ながら相談したい」が成立します。

例を挙げます。CLI が執筆中の Zenn Book の二冊目の原稿構成を「このまま進める」と判断していた状況で、Cowork に外部レビューを依頼しました（`cowork-pending.md` 経由）。Cowork が出した結論は「二冊目と三冊目に分冊すべき」。

これはCLIが「当たり前」と思っていた前提ごと崩す指摘でした。方向転換を引き出したのは、まったく別のコンテキストから見た目だったと考えています（これは私の解釈です）。

この「フレッシュなコンテキストでレビューする」という使い方は、以前 ChatGPT や Claude.ai のチャットに相談していた目的に近いです。違いは、ワークスペースを共有できるため「口頭で説明し直す」コストがない点です。

## 「同じMacで2つのClaudeセッション」はオーバーヘッドか

率直に言うと、使い始めはそう感じました。「1つのセッションで全部やるほうが効率的では」という感覚です。

実際に運用してわかったのは、オーバーヘッドより**セッションの寿命**のほうが影響が大きいということです。

CLIのセッションは作業が積み重なるほどコンテキストが太くなります。長く使ったセッションはそれだけ多くのコストと時間を消費します。「どこかでセッションを分ける」という選択肢は遅かれ早かれ出てきます。

cowork-cli-bridgeはその分け方を「ファイル1本で非同期に渡す」仕組みにしたものです。セッションを分けるコストを下げることで、「長く使い続けるほど重くなる1つのセッション」よりトータルで軽くなるケースが出てきます。

まだ完全に使いこなしているとは言えないですし、すべての作業に適用するものでもありません。ただ、`~/.claude/`へのデプロイやブラウザ操作が絡むタスクでは、明確に「CLIとCoworkで分ける」という選択が合理的になります。

## cowork-cli-bridge を使うべき場面まとめ

* CoworkとCLIは「できること」が違う。片方だけでは詰まるタスクが存在する
* `cowork-cli-bridge`プロトコル: ローカルファイルをメッセージキューにして非同期で委譲する
* 委譲の操作はファイル1本を書くだけ。実測では委譲から完了まで2分47秒（人間トリガー込み）
* Coworkは「フレッシュなコンテキストの外部レビュアー」としても機能する
* なお、モバイルとデスクトップを繋ぐ場面では「claude-bridge」（Google Drive経由）を使い分ける（詳細は`ops/playbooks/claude-bridge.md`参照）

実装はすべてPlaybook（`ops/playbooks/cowork-cli-bridge.md`）に落とし込んであります。まず`cli-pending.md`を1本書いてみるだけで試せます。指示書の書き方は上記の実物抜粋を参考にしてください。
