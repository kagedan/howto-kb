### CodexとClaude Code、どっちを使えばいい？

最初に正直に言う

「CodexとClaude Code、どっちがいい？」

聞かれるたびに少し困る。  
どちらが優れているか、という問いへの答えを持っていないからだ。

包丁とハサミを並べて「どっちがよく切れますか」と聞かれても、何を切るかによって変わる。  
あの困惑に近い。

ただ、両方を実際に使い続けて、「こういう人にはこっち」という感覚は固まってきた。  
今日はその感覚を、できるだけそのまま書く。

ツールのスペック表ではなく、使っている人間の実感として。

そもそも、何ができるツールなのか

Codex（OpenAI）とClaude Code（Anthropic）は、どちらも「AIがコードを書いてくれるツール」だ。

ただ、普通のAIチャットとは少し違う。

ChatGPTやClaudeに「このコードを書いて」と聞くと、コードを貼り付けてくれる。  
それを自分でコピーして、ファイルに貼って、動かしてみる。  
この「コピーして貼る」作業を自分でやる必要がある。

CodexとClaude Codeは、そこが違う。

「この機能を作って」と伝えると、計画を立て、ファイルを書き換え、テストを実行し、結果まで返してくれる。  
コードを1行も書かなくていい。指示を出すだけでいい。

プログラマー専用のツールでもない。  
自然言語で話しかけるだけで動く。「こういうものが欲しい」を伝えられれば、それで十分だ。

ただ、中身の思想がまるで違う。  
この「思想の違い」が、使い心地のすべてを決める。

### 使い始めた日の話

Codexを最初に試したのは、GitHubのIssueが溜まっていた日だった。

ものは試しで「このIssueを解決して」と投げてみた。  
画面の前で腕を組み、どんな質問が返ってくるか待っていた。

返ってきたのは質問ではなかった。計画だった。  
「こういう手順で進めます」と書かれていて、しばらくすると「完了しました。Pull Requestを作りました」と出た。

正直、拍子抜けした。  
仕事をしている感覚がなさすぎて、不安になったくらいだ。

で、そのPRを開いてみた。

コードは動いていた。  
テストも通っていた。  
ただ、変数の名前の付け方が自分のスタイルと少し違った。  
そこだけ手で直して、マージした。

所要時間は確認と修正で10分。  
同じ作業を自分でやっていたら、1〜2時間はかかっていたと思う。

そのとき初めて「これは本物だ」と感じた。

Claude Codeの最初の印象は、全然違った。

ターミナルを開いて、「この機能、こういう方向で作りたいんだけど」と話しかける。  
すると「こういう構成を考えました。どうですか？」と返ってくる。

対話だ。  
「あ、そこはちょっと違う」と言えば、方向を変えてくれる。  
「なぜこの構成にしたの？」と聞けば、理由を説明してくれる。

完成したコードは読みやすかった。  
「なぜこう書いたか」が伝わってくる構成で、後から見直したときに迷わない。

Codexが「仕事を渡す」なら、Claude Codeは「一緒に考える」だと、この日に直感した。

「職人」と「相棒」という違い

うまい比喩を探していたが、しっくりくるのはこれだ。

Codexは、腕のいい職人への発注に近い。

「こういうものを作ってほしい」と依頼を出して、あとは任せる。  
途中経過をいちいち確認しない。終わったら成果物を受け取って、問題なければOK。

信頼して任せられるかどうかが、このツールのすべてだ。  
依頼の精度が高ければ高いほど、結果が良くなる。

Claude Codeは、信頼できる相棒との共同作業に近い。

「どう思う？」と聞ける相手がいる、という安心感がある。  
複雑な問題を前にしたとき、一人で抱え込まなくていい。  
「ここがよくわからない」と言えば、丁寧に解きほぐしてくれる。

どちらがいいかは、その人の性格と、そのタスクの性質によって変わる。

### 向いている人、向いていない人

実際に使い続けた肌感覚で言う。

Codexが向いている人  
複数の作業を同時に進めたい。結果が出るまで待てる。  
「終わってから確認する」スタイルが性に合っている。  
コードの品質より、スピードと量を優先したい。

Claude Codeが向いている人  
「なぜこうなっているか」を理解しながら進めたい。  
途中で軌道修正したい。一人で抱え込むのが苦手。  
コードの読みやすさや、設計の整合性を大事にしたい。

どちらも、コードが書けなくても使える。  
自然言語で指示するだけでいい。

ひとつだけ注意を。  
「AIが作ったから」という理由で、生成物をそのまま本番環境に流してはいけない。  
何をやらせたか、何が出てきたか。ここだけは、必ず自分の目で確認する。  
ツールへの責任の丸投げは、どの道具を使っても通用しない。

どこで使えるのか、何が必要か

「使ってみたい」と思ったとき、次の扉がわからないと動けない。  
それぞれの始め方を、シンプルにまとめる。

### Claude Codeの始め方

Claude Proのサブスクリプションが必要だ（月額約3,000円前後）。  
ターミナル（Macならターミナル.app、WindowsならPowerShell）を使う。  
インストールはコマンド一行で完了する。  
「ターミナルって何？」という人には、まだ少しハードルがあるかもしれない。  
ただ、画面の指示通りに進めれば、エンジニアでなくても動かせるレベルだ。

### Codexの始め方

ChatGPT PlusまたはTeam/Enterpriseのサブスクリプションが必要だ（月額約3,000円前後）。  
WebブラウザからChatGPTにログインして、左のメニューからCodexを選ぶだけ。  
GitHubアカウントと連携すれば、リポジトリを直接扱える。  
インストール不要なので、ターミナルに不慣れな人でも始めやすい。

どちらも無料プランでは使えない。  
まず片方だけ試したいなら、Codexのほうが導線がシンプルだ。

ただし、価格や提供プランは変わることがある。  
最新情報は各公式サイトで確認してほしい。

私がいま使っている組み合わせ

参考までに、いまの自分のやり方を共有する。

大きな機能を作るときは、まずClaude Codeと対話する。  
「この機能、どういう設計にすればいいと思う？」から始めて、方向を一緒に決める。

構成が固まったら、実装の重たい部分をCodexに投げる。  
複数のタスクを並列で走らせて、終わったものから確認する。

「設計はClaude Code、実行はCodex」という役割分担だ。

一人でやっていたとき、設計と実装の間で詰まって手が止まることが多かった。  
この組み合わせにしてから、その「詰まり」が減った。  
どちらか片方で完結させることもある。タスクの規模と、そのとき自分がどれだけ関わりたいかで選んでいる。

### 本音を言う

中立を保って「どちらもいい」と書き続けると、顔が見えない記事になる。  
だから最後に、正直に言う。

私はClaude Codeのほうが好きだ。

理由は単純で、「なぜそう書いたか」を聞けるからだ。

コードは書いて終わりじゃない。  
あとで読む。修正する。他の人に引き継ぐこともある。  
「動けばいい」ではなく「なぜこうなっているか」がわかるコードを書くためには、対話できる相手のほうが自分には合っている。

Codexの速さと効率は本物だ。  
使う場面を選べば、圧倒的に仕事が速くなる。

ただ、「作りながら考える」時間が好きな人間には、Claude Codeのほうが性に合う。  
これは性格の問題だと思っている。

まず、何から始めるか

「結局どっちを試せばいいの？」

そう思っているなら、Claude Codeからをすすめる。

対話しながら進められるので、何が起きているかわかりやすい。  
AIエージェントの感覚をつかむ最初の一歩として、「一緒に考えてくれる相手」のほうが挫折しにくい。

Codexの「任せて放置」スタイルは、ある程度ツールへの信頼感が積み上がってから使うほうが、結果的に早い。

いきなり完璧な指示を出そうとしなくていい。  
「こんな感じのことをやりたい」くらいの粒度で話しかければ、意外と動いてくれる。

必要なのは技術の知識じゃない。  
「何をやりたいか」を言葉にできること。  
それだけで、十分始められる。
