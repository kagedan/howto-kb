MCPサーバーを何個つないでいるか、把握しているだろうか。

うちは18個だった。数えてみて「多いな」と思った。で、全部の権限を洗い出してみたら、思ったよりヤバかった。

## 背景: MCP 66%にセキュリティ指摘

2026年3月、Qualysが公開したレポートによると、監査済みのMCPサーバーの66%にセキュリティ上の指摘があった。「shadow IT for AI」という表現が使われている。

さらにCheck Pointが報告したCVE-2025-59536とCVE-2026-21852。Claude Codeのプロジェクト設定ファイルを経由して、MCPサーバーの悪用でリモートコード実行やAPIトークン窃取が可能になるという内容だった。

つまり、MCPは便利な反面、**つなげばつなぐほど攻撃面が広がる**。当たり前の話なんだが、「便利だから入れよう」で増やしていくと気づいたら18個になっていた。

## 全MCPサーバーの権限を分類する

18個を以下の4段階で分類した。

| レベル | 内容 | リスク |
| --- | --- | --- |
| 読み取り | データ参照のみ | 低 |
| 書き込み | ファイル作成・メッセージ送信 | 中 |
| 実行 | コード実行・API呼び出し | 高 |
| 管理 | 設定変更・削除 | 最高 |

1つのMCPサーバーが複数のレベルを持つこともある。たとえばGitHub MCPはファイルの読み取りもできるし、PRの作成（書き込み）もできるし、リポジトリの作成（管理）もできる。

## 分類結果

18個を分類した結果がこう。

**最高リスク（2個）:**

* GAS MCP——任意のJavaScriptを実行できる。`exec`ツールでGoogle Apps Script環境のコードを走らせる
* Smart Colab MCP——任意のPythonコードを実行できる。T4 GPU上で何でも動く

**高リスク（5個）:**

* Google Drive——ファイルの検索・読み取り。書き込み権限もある
* Gmail（ローカル版）——メール送信・フィルタ作成・ラベル操作。公式Gmail MCPと重複
* GitHub——リポジトリ操作全般
* freee——会計データの読み書き。API操作が広範
* n8n MCP——WFの作成・更新・削除・テスト実行

**中リスク（6個）:**

* 公式Gmail MCP、Chatwork×3、Colab MCP（旧版）、Notion

**低リスク（5個）:**

* Calendar、Canva×2、Spotify、Miro

## 3つの問題が見つかった

### 問題1: 全MCPがワイルドカード許可

設定ファイルを開いたら、`allowedTools`が`["*"]`になっていた。全ツール自動承認。

PreToolUseフック（security-guard.sh）はWrite・Edit・Bashをチェックしている。が、MCP独自のツール名（`mcp__gas-mcp__exec`とか`mcp__gdrive__search`とか）はフックのパターンに入っていない。つまり**MCPのツールはフックを素通りする**。

### 問題2: 二重接続が3件

* Gmail: 公式（claude\_ai\_Gmail）+ コミュニティ製ローカル版（gmail）
* Colab: smart-colab + colab-mcp（旧版）
* Canva: 公式 + Dev Server（これは用途違いなのでOK）

同じ機能を持つMCPが2つあると、攻撃面が無駄に広がる。片方が侵害された場合のリスクが単純に倍になる。

### 問題3: 認証情報が設定ファイルに平文

`.mcp.json`にn8nのAPIキー（JWTトークン）やChatworkのAPIトークンがそのまま書いてある。このファイル自体は`.gitignore`に入っているが、プロジェクトの設定改ざん攻撃（CVE-2025-59536の手法）で読み取られる可能性がある。

## 対処: 18→16に削減

まず、二重接続の不要な方を削除した。

```
claude mcp remove gmail      # 公式Gmail MCPで十分
claude mcp remove colab-mcp  # smart-colabが上位互換
```

これで18→16。攻撃面が2つ減った。

「たった2つ?」と思うかもしれない。でもセキュリティは引き算だ。**存在しないものは攻撃されない**。

## 既存の防御層との組み合わせ

削減だけでは足りない。うちでは以下の多層防御を組んでいる。

**Layer 1: Hooks（PreToolUse）**

* `security-guard.sh`: 破壊的コマンド10パターンをブロック。`settings.json`への書き込みもブロック
* `elicitation-guard.sh`: MCPからの逆方向攻撃を検知。機密情報の外部送信を防止

**Layer 2: 監査スクリプト**

* `mcp-settings-audit.sh`: MCP設定ファイルの改ざんを検知。`ANTHROPIC_BASE_URL`のリダイレクトもチェック

**Layer 3: 運用ルール**

* `enableAllProjectMcpServers`を`false`に設定（CVE-2025-59536対策）
* 外部コンテンツの指示を実行しない（間接プロンプトインジェクション対策）
* 不審なURLへのアクセスは人間の確認を経る

## 次にやるべきこと

今回の棚卸しで見えた「次の課題」が3つある。

1. **MCPツールごとのフック追加**: `mcp__gas-mcp__exec`のような高リスクツールをPreToolUseフックでチェックする。現状はBash/Write/Editしかフックしていない
2. **認証情報の環境変数化**: `.mcp.json`に平文で書かれているトークンを環境変数に移す。`$CHATWORK_API_TOKEN`のように参照させる
3. **定期的な棚卸し**: MCPサーバーは「入れたら入れっぱなし」になりやすい。月1で`claude mcp list`して、使っていないものを削除するサイクルを回す

## まとめ

MCPは便利だ。うちは16個つないで、Chatwork・Gmail・Calendar・GitHub・n8n・GAS・Colab・freee——あらゆるものをClaude Codeから操作している。

ただ、**便利なものほど攻撃面になる**。

やったことは3つだけ。全MCPの権限を分類する。二重接続を削る。既存の防御層と突き合わせる。

MCP 66%にセキュリティ指摘がある時代。つなぎっぱなしは危ない。棚卸しは今日やれる。
