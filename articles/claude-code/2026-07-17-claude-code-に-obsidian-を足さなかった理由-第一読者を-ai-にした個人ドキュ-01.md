Claude Code で運用している私のリポジトリには、目次がありません。ノートを眺めるための画面も、リンクを辿るためのグラフも用意していません。Obsidian を検討して、足さないことにしたからです。

先に断っておくと、Obsidian が良くないという話ではありません。私の前提と噛み合わなかった、という話です。そして噛み合わなかったのは Obsidian の欠点ではなく、私が先にひとつ捨てたものがあったからでした。

## ドキュメントは、誰のために書かれてきたか

そもそも、なぜ私たちはドキュメントを書くのでしょうか。

答えははっきりしています。**自分以外の誰かに、非同期で伝えるため**です。会議で口頭で言えば済むことをわざわざ文章にするのは、その場にいない人が、別の時間に、一人で読んで理解できるようにするためです。

だからチーム開発では Notion なり Confluence なりにコストを払います。目次を整え、見出しを立て、前提から順に書き下し、図を添える。読み手が知らないことを補い、読む順序を設計する。この手間はすべて「読み手が人間で、しかも文脈を持っていない」という前提から発生しています。

## 個人開発では、読み手が縮退する

ところが個人開発では、この前提が崩れます。読み手は自分ひとりです。そして今は、そこにもう一人加わります。AI です。

読み手が「自分と AI」の2人に縮退したとき、人間可読性に払っていたコストはどうなるか。

自分は、そもそも自分が書いたものを読み返しません。書いた本人なので文脈を持っているからです。半年後に忘れる、という反論はあります。でも実際に忘れたとき、私はドキュメントを探しませんでした。隣にいる AI に口頭で尋ねて、それで用が足りてしまった。

つまり**私のドキュメントを日常的に読んでいるのは、AI だけ**です。それなら、AI が読みやすい形にするのが素直です。

## 第一読者を AI に置くと、何が消えるか

いわゆるセカンドブレインを、人間ではなく AI が読む前提で組み直す、ということです。読み手を AI に固定して設計をやり直すと、人間向けに用意していたものが順番に不要になっていきました。消えたものと、その代わりに何が来たかを並べるとこうなります。

| 人間が第一読者のときに要るもの | AI が第一読者になった後 |
| --- | --- |
| 目次・見出しで全体像を掴む | AI が最初に読むルーティング表（`context.yaml` 1枚） |
| クイック検索の UI | frontmatter の `tags` を照合するフック |
| 通し読みできる物語の流れ | 1ファイル1事実（原子化） |
| リンクを辿って関連に行く操作 | ファイル名の規約と grep |
| WYSIWYG のエディタ | 対話（言えば AI が書く） |

左の列は全部「**人間が、自分の目と手で、探して・辿って・読む**」ための装置です。その動作を AI に委譲した瞬間、装置ごと不要になります。

代わりに効いてくるのは、人間には少し無味乾燥に見えるものばかりでした。ファイル名は語感ではなく grep で引けるかで決める（`journal/2026/07/17.md`）。1つの事実を1つのファイルに切って、frontmatter に `tags` と `description` を付ける。索引は読み物ではなく、AI が最初に読むルーティング表として置く。ルールは説明文ではなく、読み込まれてそのまま振る舞いを変える注入物として書く。

この設計を実際に動かしているテンプレートは公開しています（[claude-code-life](https://github.com/takotyan00/claude-code-life)）。`memory/` の1事実1ファイルと frontmatter、`tags` で引く呼び出しフック、`.claude/rules/` に分割された振る舞い。中身を見ると、人間向けの導線がほとんど無いことがわかると思います。設計思想の全体像は [リポジトリは人間のために書かない【全体像】](https://zenn.dev/takotyan00/articles/claude-code-life-repository) に、`CLAUDE.md` とルール分割の実装は [CLAUDE.md は書くほど効かなくなる](https://zenn.dev/takotyan00/articles/claude-code-claude-md-design) に書きました。

## Claude Code に Obsidian は必要か

ここが本題です。結論から言うと、人間が第一読者・第一ナビゲーターであるうちは必要で、その役割を AI に渡したなら要らなくなる、というのが私の答えです。

まず Obsidian が解いている問題は、実在します。ノートは増えるほど、どこに何を書いたか分からなくなる。目次を手で保守するのは続かない。そのためリンクを張って辿れるようにし、バックリンクで逆から引けるようにし、グラフで全体を俯瞰できるようにする。これは人間の記憶とナビゲーションの限界に対する、よくできた回答です。実際、私がもし毎日自分のノートを自分の目で眺める運用をしていたら、間違いなく入れていました。

問題は、Obsidian が提供している価値の多くが「人間がノートを読む・辿る・眺める」ための機能だ、ということです。グラフビュー。バックリンクパネル。WYSIWYG 編集。人間用のクイック検索。

そして私の設計は、その人間ナビゲーション層そのものを落とすことが核でした。落とした層を提供してくれるアプリを足しても、乗るところがありません。Obsidian の主機能と、私の設計の狙いが、真正面から相殺します。

同じ md の上に、どちらのナビゲーション層を乗せるかという選択だと考えると分かりやすいと思います。

分岐しているのはファイルではなく、その上に誰を立たせるかです。私は下の経路を選んだので、上の経路のために作られた道具は、置き場所ごと無くなりました。

### AI から Obsidian を叩く経路は、ちゃんとある

ここまで読んで「プラグインを入れれば AI からも Obsidian を触れるはずだ」と思った方がいると思います。そのとおりで、実際にあります。

Local REST API というコミュニティプラグインを入れると、vault のファイル読み書き・検索・見出し単位の部分編集が HTTP 経由でできるようになります。その上に乗る MCP サーバーの実装も複数あって、star が数千付いているものもある。Dataview のクエリを REST 経由で投げることもできます。だから「Obsidian は人間専用のツールだ」と切り捨てるのは、正確ではありません。AI に開くための橋は、コミュニティの手で架かっています。

それでも私が入れないのは、その橋が「向こう岸にすでに Obsidian が建っている人」のためのものだからです。

これらの経路はどれも、Obsidian アプリが起動していることが前提になります。プラグインである以上、アプリのプロセスの中で動くので当然です。ヘッドレスで動かす工夫もありますが、アプリ自体を動かし続ける前提は変わりません。一方で私のドキュメントは、Claude Code がファイルを直接開いて読んでいます。ここに橋を架けるというのは、経路が1本増えることではありません。渡った先に建物が無いので、まず建物を建てるところから始めることになります。

そして、その建物を経由して増える能力が、私の場合ほとんどありません。ファイルの読み書き、全文検索、frontmatter の読み書き、見出し単位の編集。このあたりは Claude Code が素の md に対してすでにやっています。Dataview のクエリでやるようなフィルタや集計も、md を読んで自分で数えれば済む範囲でした。

調べたかぎり、API を挟んで純粋に増えるのは3つです。Obsidian のランタイム機能を外から叩けること（他のプラグインを発火させるなど）。計算済みのバックリンクやグラフを再利用できること。そして Smart Connections のようなプラグインによる意味検索です。

前の2つは、私には効きません。プラグインを使っていないし、グラフも見ないからです。ただし3つ目は正直に書いておきます。意味検索は、私の設計が代替できていない唯一の穴です。grep は字面が一致しなければ引けないので、言葉が違うだけで同じことを指しているファイルには辿り着けません。埋め込みを使う意味検索は、そこを越えてきます。今のところ私の使い方では困っていませんが、困らないのは私の運用規模がその程度だから、というだけの話です。

コストの向きも逆でした。私が md にこだわっているのは、素の md ならどこにでも持ち出せる、つまりロックインしないからです。Obsidian を入れると、アプリ依存・vault の形式・同期の設定が増えます。Dataview のインラインフィールドのようなプラグイン固有の記法を使えば、それは Obsidian の外ではただの文字列に戻る。可搬性を上げるために md を選んだのに、md の上に別のレイヤーを積むことになります。

そのため「Obsidian はダメ」ではありません。**前提が噛み合わない**、が正確です。橋の向こうに建物がある人には、あの橋には価値があります。

## それでも人間可読を残している場所

ここまで書いた設計は、無条件では成立しません。境界が2つあります。

### 外部に向いた面

GitHub で公開しているテンプレートの README は、人間が読みます。OSS を触る人も、協業する人も、AI を挟まずに直接読む。私のリポジトリは実際には二層になっていて、内部の作業用ドキュメントは AI ネイティブ、外部インターフェースは人間可読です。前者の作法を後者に持ち込むと、ただ不親切なだけになります。

### AI が常駐している間だけ成立する

「読みたければ AI に聞く」は、聞ける相手がいる前提の話です。AI 抜きで自分が直接ファイルを開く場面はゼロにはならない。そのため冗長性をゼロまで削ってはいません。grep で引ける命名と frontmatter は、そのときの保険として残しています。これは「AI が探しやすい構造にする」と同じことをしているだけなので、追加のコストは払っていません。

誤解のないように言うと、これは可読性を捨てたわけではありません。構造化は**別種の可読性**を生みます。1事実1ファイルで frontmatter が付いていれば、人間だって grep で引けます。捨てたのは「最初から最後まで読み下してもらう前提の散文」だけです。

## 道具の前に、読み手を決めていた

振り返ると、私は Obsidian を評価して落としたのではありませんでした。その前に「第一読者は AI」と決めていて、その時点で Obsidian が効く場所が消えていた、という順番です。

道具を比べるとき、私たちは機能表を並べがちです。でも機能表は「その機能が誰のために働くか」を書いてくれません。グラフビューは、グラフを見る人がいて初めて機能になります。見る人がいなければ、それはただの描画です。

だから比較の前に決めるべきなのは、道具の良し悪しではなく、**自分のドキュメントを毎日読んでいるのは誰なのか**、でした。私の場合その答えが AI だったので、人間向けの機能から順に消えていった。それだけの話です。

自分のノートを毎日自分で読む人にとっては、Obsidian を入れない理由はひとつも無いと思います。
