追記：  
この記事を公開したタイミングで、Claude Desktop の Linux版（Ubuntu / Debian）beta も提供され始めた。

これにより、Claude Desktop / Cowork / Claude Code 周りの選択肢はさらに増えたが、この記事ではまずブラウザ上で試せる Claude Design に絞って初回検証している。

Linux版Claude DesktopやCoworkについては、ローカル環境・保存先・アクセス範囲の確認が必要になるため、別記事で扱う予定。

## はじめに

前回まで、AIエージェントを安全に試すための検証環境について整理していた。

前回の記事:  
<https://zenn.dev/imaginarygate/articles/46e2514135bc01>

もともとは、WSL / Ubuntu 上で Codex CLI や Claude Code を比較しながら、ローカルでAIエージェントを使うときの安全な作業範囲を確認していた。

一方で、Claude Desktop / Cowork / Design など、Claude側にも複数の新しい機能もあり、今回はCoworkが期間限定で2倍使用制限などが実施されていた。

ただし、Coworkはデスクトップアプリやローカルファイルの扱いが絡むため、いきなり試すには少し慎重になりたい。

そこで今回は、まずブラウザ上で使える Claude Design を使い、架空SaaSのLPを作ってみることにした。

## 今回やりたかったこと

今回の目的は、Claude DesignでどこまでLPのたたき台を作れるかを確認することだった。

やりたかったことは以下。

* 架空サービスのLPを作る
* 実在企業名や個人情報は使わない
* 外部画像は使わない
* ポートフォリオに載せられそうな題材にする
* 共有やエクスポート周りの挙動を見る
* CoworkやClaude Code連携にはまだ踏み込まない

Claude Designはデザイン生成機能ではあるが、今回は「きれいな見た目を作る」だけではなく、AIツールを使って成果物を作るときに、どこまで人間側で要件や制約を設計する必要があるかも見たかった。

## Coworkではなく、まずDesign単体を試した

最初はClaude Coworkも気になっていた。

ただ、Coworkはローカルファイルやデスクトップアプリ側の設定が関係するため、保存先やアクセス範囲をきちんと確認してから触りたい。

一方で、Claude Designはブラウザ上で完結するため、ローカルファイルには触れない。CoworkやClaude Desktopのように保存先やアクセス権限を気にする必要がなく、まずはこちらを先に試すことにした。

今回の切り分けはこうした。

```
Claude Design
→ ブラウザでLPやモックアップを作る
→ 今回の検証範囲ではローカルファイルには触れない

Claude Cowork
→ ローカルファイルやデスクトップ側の挙動確認が必要なので後回し

Claude Code / Claude Code Web
→ Designから送る導線は見えたが、今回は押さない
```

いきなり全部連携させるのではなく、まずはDesign単体で何ができるかを見る。

## 架空サービス「Cowork Sandbox」を作ることにした

今回作る題材は、架空サービスのLPにした。

サービス名は `Cowork Sandbox`。

AIエージェントを安全に試すための、隔離されたワークスペースという設定にした。

最初の要件は以下。

```
サービス名：
Cowork Sandbox

目的：
AIエージェントを安全に検証するための架空SaaS LPを作る

条件：
- 実在企業名は使わない
- 個人情報は使わない
- 外部画像は使わない
- 架空サービスとして作る
- 安全な検証環境をテーマにする
- ポートフォリオ掲載を意識する
```

最初はかなりシンプルな要件だった。

その結果、Claude Designは以下のようなLPを生成した。

* Hero
* About
* Features
* CTA
* Footer

最初にエクスポートしたPDFでは、1ページ目にヒーロー、2ページ目にABOUT / FEATURES / CTA が分かれて表示された。内容としては「Build together, in a safe space.」というヒーローコピーや、「AIチームのためのサンドボックス環境」という説明が含まれていた。

## Claude Designの画面構成

ここで、Claude Designの画面構成について簡単に触れておく。

Claude Designは左右に分かれた画面構成になっている。左側がチャット（会話で指示を出す場所）、右側がデザインのプレビューになっている。

右側のプレビュー画面の上部には、以下のようなナビゲーションバーがある。

```
Mark up     … デザイン上に直接マーカーやコメントを書き込む
Comments    … コメント一覧を見る
Edit        … テキストや要素を直接編集する
Hide tweaks … 右に出るTweaksパネルを隠す/表示する
100%        … 表示倍率（ズーム）
Present     … プレゼンモード（全画面で見せる）
Share       … 共有やエクスポート
```

### Tweaksパネル

特徴的だったのは、Claude Designが自動で生成する「Tweaks」パネルだった。

今回のLPでは、以下のようなカスタマイズ用のコントロールが自動で作られた。

```
accentColor   … アクセントカラーの切替（グリーン、ティール、ブルー、パープルなど）
heroHeading   … ヒーローセクションの見出しテキストを直接編集
darkMode      … ダークモードのON/OFF
```

プロンプトで「グリーン基調で」と指定していたが、Tweaksパネルからワンクリックで他のカラーに切り替えて比較できる。会話でいちいち「色を変えて」と指示し直さなくても、スライダーやボタンで即座にバリエーションを確認できるのは便利だった。

下部には「Reset」（変更を元に戻す）と「Save as defaults」（設定をデフォルトとして保存する）ボタンがある。

### Editモード

上部ナビゲーションの「Edit」を押すと、左側にパネルが展開され、テキストや要素の細かい編集ができる。会話で全体の方向性を指示しつつ、細部はEditモードで直接手を入れる、という使い分けが可能だった。

### 使用モデル

画面下部には「Opus 4.8 High」と表示されていた。

## 最初の生成結果を見て感じたこと

最初の生成結果は、見た目としてはかなり整っていた。

薄いグリーン基調で、余白も広く、クリーンな印象があった。

ただし、LPとして見たときに、少し抽象的にも感じた。

たとえば、最初の段階では以下の点が気になった。

* 誰に向けたサービスなのかがやや曖昧
* 「安全なAI作業空間」という雰囲気はあるが、具体的な課題が弱い
* 架空SaaSとしては成立しているが、実在感を出すにはもう少し訴求を絞りたい
* 「ワンクリックデプロイ」のように、本番反映を強く連想させる表現は今回のテーマと少しズレる

今回のサービスは「AIを安全に試すためのサンドボックス」なので、本番環境へすぐ反映するよりも、検証範囲を限定する、安全に試す、作業ログを確認する、といった方向に寄せた方が自然だと感じた。

## ターゲットを絞って再設計した

そこで、ターゲットをもう少し具体化した。

```
ターゲット：
AIエージェントを安全に試したい個人開発者・小規模チーム

ユーザーの悩み：
- 本番環境や個人データを汚したくない
- AIにどこまでアクセスを許してよいか不安
- 検証環境をすばやく用意したい
- チームで安全に試したい
- 作業ログや変更内容を後から確認できるようにしたい

提供価値：
- AI実験用の隔離されたワークスペース
- チームで共有できる安全な作業環境
- AIごとに触らせる範囲を分けられる設計
- 作業履歴や変更内容の見える化
- 検証から改善までを小さく安全に回せること
```

この条件をもとに、Claude Designへ再設計を依頼した。

その際、構成も次のように指定した。

```
Hero
Problem
Solution
Features
Workflow
Use cases
CTA
Footer
```

また、画像については、実写画像や実在サービスのロゴを使わず、UIモック、概念図、シンプルなアイコンを中心にする方針にした。

## 再生成されたLP

再生成されたLPでは、かなりLPらしい構成になった。

ヒーローには次のようなコピーが入った。

その下には、アクセス許可やログ確認を連想させる小さなUIモックが置かれていた。

セクション構成は以下のようになっていた。

```
Hero
Problem
Solution
Features
How it works
Use cases
Footer
```

Problemでは、以下の見出しが入り、AI導入前の不安を扱う構成になっていた。

Featuresには、以下のような項目があった。

```
隔離されたワークスペース
権限コントロール
操作ログの可視化
チーム共有
```

この方向性は、今回のサービス設定とかなり合っていた。

一方で、まだ改善できそうな点もあった。

## 改善したい点

全体のトーンはかなり良かった。

薄いグリーン基調で、クリーンで安全そうな印象があり、LPとしての見た目も整っていた。

ただ、制作物として載せる前提で考えると、具体化したい点が多かった。

### ターゲットがまだ少し広い

「AIを安全に試したい人」には向いているが、個人開発者向けなのか、小規模チーム向けなのか、社内検証担当者向けなのかが少しぼやけていた。

想定読者をもう少し明確にすることにした。

たとえば以下のように寄せると、よりサービスとして伝わりやすい。

```
AI導入を検討している小規模チーム・社内検証担当者向け
```

または、

```
本番環境に触れさせる前に、AIエージェントを安全に試したいPM・デザイナー・開発チーム向け
```

### Heroの説明をもう少し具体化したい

ヒーローの見出しは良かった。

ただし、サブコピーにはもう少し具体的な便益を入れたい。

たとえば、

```
本番データや日常ファイルから切り離した専用環境で、
AIエージェントやブラウザ自動化を安全に検証できるサンドボックス。
```

または、

```
AIに触らせる範囲を限定し、作業ログも確認しながら、
チームで安全に検証を進められるワークスペース。
```

このくらい具体的にした方が、「何ができるサービスなのか」が伝わりやすい。

### Problemの内容をもう少し刺さる表現にしたい

Problemセクションでは、AI利用時の不安を扱っていた。

ここは方向性としては良い。

ただし、もう少し読者の「あるある」に寄せてもよさそうだった。

たとえば、

```
本番データを誤って触られそうで怖い
AIがどこまでアクセスできるのか分からない
試したいけど、検証環境が整っていない
```

このくらい具体的にした方が、導入前の不安として伝わりやすい。

### 最後のCTAを強くしたい

LPの最後に濃いグリーンの大きな余白があった。

雰囲気は良かったが、そのままだと少し寂しく感じた。

ここには、最後のCTAセクションを入れると良さそうだった。

たとえば、

```
AIを試したい。
でも本番環境は触らせたくない。
そのための最初の一歩が、Cowork Sandboxです。

[ ベータ版に登録する ] [ デモを見る ]
```

このような締めがあると、LPとしての着地が分かりやすくなる。

## 共有・エクスポート・Claude Code連携について

Claude Designを触っている途中で、共有やエクスポートに関する導線も見えた。

### 共有メニューの構成

Shareを押すと、上部に3つのタブが表示された。

```
Share link   … リンクで共有する（共有範囲の詳細は未確認）
Export       … ファイルとしてダウンロードする
Send to...   … 他のツールに送る（CanvaやClaude Codeへのハンドオフなど）
```

### エクスポート形式

Exportタブでは、以下の4つの形式が選択できた。

```
PDF              … 現在のファイルをPDFとして書き出す
PowerPoint .pptx … PowerPointやKeynoteで編集できるスライドファイル
Project archive .zip … プロジェクト内の全ファイルをZIPでまとめてダウンロード
Standalone HTML  … オフラインでも動く、1つの自己完結HTMLファイル
```

PDFを選択すると、さらに出力方法を2つから選べた。

```
Instant（推奨）
→ 今見えているデザインをそのまま印刷ビューで出力する。速い。

Reformat for printing via Claude
→ Claudeがデザインをレターサイズ（8.5 × 11 in）に合わせて自動的にレイアウトし直す。
```

「Instant」はスクリーン表示をそのままPDFにする形。「Reformat」は紙に印刷することを想定して、Claudeがページ収まりを自動で調整してくれる。

今回の検証用LPでは「Instant」で十分だった。

### Claude Code連携

「Send to...」タブには、Claude Codeに送る導線も見えた。

ただし、今回はここでは止めた。

理由は、共有範囲やClaude Code側に何が渡るのかを、まだ十分に確認できていなかったからだ。

特に、Claude Code WebやCoworkに進むと、GitHub連携やローカルファイル、作業環境の話が絡む可能性がある。

そのため今回は、以下の範囲にした。

```
Design単体でLPを作る
Exportを試す
共有やClaude Code連携は観察だけにする
```

## PDFエクスポートについて

PDFエクスポートも試した。

最初のLPではPDFとして出力できたが、環境によってはレイアウトが崩れたり、うまく表示できないことがあった。

また、LPとしては縦長1ページに見えていても、PDF上ではページが分割されることがある。

ポートフォリオとして載せる場合は、PDFをそのまま使うよりも、以下のどれかにした方がよさそうだと感じた。

```
- 画面キャプチャとして掲載する
- 画像として書き出す
- 自分のポートフォリオページに再構成する
- PDFを使う場合は、表示崩れがないか確認する
```

生成物そのものだけでなく、出力形式ごとの見え方も確認が必要だった。

## 利用制限が来た

作業を続けている途中で、利用制限が来た。

画面には「Paused — you've hit your limit.」と表示され、実行中のタスクが一時停止された。

このとき、完了済みのタスクにはチェックマークが付き、まだ終わっていないタスクも一覧で表示されていた。画面下部には「Work so far is saved」と表示されており、途中までの作業は保存されていることが確認できた。

「Resume」ボタンも表示されており、利用枠がリセットされた後にここを押せば作業を再開できる仕組みだった。

利用枠のリセットタイミングは「View usage」リンクから確認できたが、表示されたのはチャット側の使用量のみで、Design側の利用枠がどのくらい残っているかは確認できなかった。

リセット後に再開してみたが、改善指示を数回出したところで再びリミットに達した。Design側の利用枠は、現時点ではかなりタイトな印象だった。

現時点では、以下までは確認できた。

```
- Claude Designで架空LPを生成できる
- ターゲットや悩みを渡すと、構成やコピーが改善される
- エクスポートはPDF / PPTX / ZIP / Standalone HTMLの4形式
- Share / Claude Code連携らしき導線がある
- 利用制限が来ても作業は保存され、リセット後にResumeで再開できる
- ただし、Design側の利用枠の残量表示は現時点では確認できなかった
```

成果物として扱うなら、人間側のレビューと再設計が必要に感じた。

## Figma Makeと比べた所感

以前、Figma Makeでも要件を渡してUIを生成する流れを少し試したことがある。  
Figma Makeは多機能で魅力的だが、業務利用を考えると外部ツールであるため、別のAIに相談する際にどこまで情報を渡してよいか判断が難しいと感じる場面があった。  
その点、Claude Designは会話から生成まで一つの環境で完結できるのが良かった。  
個人的には、要件からLPのたたき台を作る初期検討はClaude Designの方が手軽に感じた。ただ、利用制限がかなりタイトなのがネックだ。

## 今回分かったこと

今回、Claude Designを初めて試してみて分かったことは以下。

```
- 要件がかなり短くても、LPのたたき台は作れる
- ターゲットや悩みを明確にすると、訴求がかなり改善される
- 架空SaaSのモックアップ制作には向いていそう
- ただし、最初の生成結果は抽象的になりやすい
- ポートフォリオに載せるなら、人間側のレビューと再設計が必要
- Tweaksパネルでカラーやダークモードを即座に切り替えて比較できる
- エクスポートはPDF / PPTX / ZIP / Standalone HTMLの4形式
- PDFにはInstantとReformat（Claudeによる自動レイアウト調整）がある
- 利用制限が来ても作業は保存され、Resumeで再開できる
- Design側の利用枠の残量は現時点では確認できなかった
- Claude Designはブラウザで完結し、ローカルファイルには触れない
- 生成にはClaude Opus 4.8（High設定）が表示されていた
```

## 次回やること

今回は制限が来たので、ここまでにした。

次回やるなら、以下を確認したい。

```
- Shareの公開範囲の詳細
- 画像としてきれいに書き出せるか
- Claude Codeに送ると何が渡るのか
- Claude Code WebとローカルClaude Codeの違い
- Coworkに進む場合の保存先とアクセス範囲
- ポートフォリオ掲載用に、最終版LPへ整える
```

ただし、次に進むとしても、いきなりCoworkやClaude Code連携までは触らない。

まずはDesign単体で、共有・エクスポート・公開範囲を確認する。

そのうえで、必要があればCoworkやClaude Code連携を別記事として扱う。

## まとめ

Claude Designを使うと、架空SaaSのLPモックアップはかなり速く作れる。

特に、サービス名、ターゲット、ユーザーの悩み、提供価値、セクション構成を渡すと、LPとしての土台はかなり整う。

一方で、初回生成のままだと、見た目は良くても訴求が抽象的になりやすい。

成果物として扱うなら、人間側のレビューと再設計が必要。

```
- 誰に向けたLPなのか
- どんな課題に刺しているのか
- サービスとして自然か
- 共有範囲は安全か
- エクスポート結果は崩れていないか
- 架空サービスであることが分かるか
```

今回は、Claude Designの初回検証として、架空SaaS LPの作成、エクスポート形式の確認、共有・連携導線の確認まで行った。

まだ分からないことも多いが、少なくとも「AIデザインツールを使って、要件からLPのたたき台を作る」流れは十分に試せた。

次回は、共有範囲とエクスポート周りをもう少し確認してから、ポートフォリオ掲載用に仕上げていきたい。

## 補足

この記事は、筆者が実際にAIエージェント検証用端末を構築する中で得た作業ログや試行錯誤をもとに、ChatGPTおよびClaude（Web版）との対話を通じて整理・執筆したものです。

手順や設計方針は、実際に試した内容と、その中で得られた学びをベースにまとめています。

Claude Designの画面構成や機能については、Claude（Web版）との対話の中でスクリーンショットを共有しながら確認した内容をもとに記述しています。

環境やClaude Designのバージョンによって、画面の表示や利用制限は変わる可能性があります。

実行する場合は、自分の環境で表示内容を確認しながら進めてください。
