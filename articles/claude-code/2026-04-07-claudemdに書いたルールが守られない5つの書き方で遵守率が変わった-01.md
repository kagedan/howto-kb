CLAUDE.mdに「mainにpushするな」と書いた。Claudeは守った。2時間後、忘れてmainにpushした。

CLAUDE.mdを書き直した。「mainにpushすることは絶対に禁止」。翌日また破られた。

3回目で気づいた。問題はルールの内容ではなく、**書き方**だった。

CLAUDE.mdはClaude Codeへの指示ファイルだ。ここに書いた内容がセッション開始時に読み込まれ、AIの振る舞いを決める。ただし、同じ意図のルールでも書き方次第で守られるかどうかが大きく変わる。

700時間以上の自律運用で試行錯誤した結果、遵守率が明らかに上がった5つの書き方パターンを紹介する。

## パターン1: 禁止リストより「やっていいこと」リスト

よくある書き方:

```
## 禁止事項
- rm -rfを使うな
- git reset --hardを使うな
- git push --forceを使うな
- .envファイルを削除するな
```

これは人間には読みやすいが、Claude Codeにとっては「禁止されていない操作は何でもやっていい」と解釈される余地がある。禁止リストは常に不完全だからだ。

改善版:

```
## 許可されている操作
- ファイルの読み取り: 常に許可
- git commit: 許可（ただしmainブランチへの直接pushは不可）
- npm install: 許可（--save-devフラグ付き）
- ファイル削除: CLAUDE.mdに記載のないファイルのみ許可

上記にない破壊的操作は実行前に確認すること。
```

「やっていいこと」を明示すると、Claude Codeはリストにない操作に対して慎重になる。ホワイトリスト方式はセキュリティの基本原則でもある。

## パターン2: 具体例を1つ添えるだけで精度が上がる

抽象的な指示:

これだと「Update files」のようなメッセージが生成されることがある。具体例を1つ添えるだけで精度が上がる:

```
コミットメッセージは「なぜ」を含めて書くこと。

例: fix: prevent duplicate API calls on rapid button clicks
悪い例: fix bug
```

LLMは具体例からパターンを抽出するのが得意だ。1つの良い例と1つの悪い例があれば、その間の判断基準を自分で補完できる。

これはプロンプトエンジニアリングでいう「few-shot prompting」と同じ原理だ。CLAUDE.mdに書く指示もプロンプトの一種であり、同じテクニックが通用する。

## パターン3: 「なぜ」を書くとルール違反が減る

ルールだけの記述:

```
- テストファイルは `__tests__/` ディレクトリに配置する
```

理由付きの記述:

```
- テストファイルは `__tests__/` ディレクトリに配置する
  - 理由: src/と同階層に.test.tsを置くとビルド出力に混入し、本番バンドルサイズが増加する
```

Claude Codeは「なぜそのルールがあるのか」を理解すると、ルールの適用範囲を正しく判断できるようになる。理由のないルールは、似て非なる状況で無視されやすい。

もう1つ例を挙げる:

```
# 理由なし（守られにくい）
- APIキーはハードコードしない

# 理由あり（守られやすい）
- APIキーはハードコードしない
  - 理由: リポジトリがpublicに変更された場合、APIキーが漏洩し不正利用される。環境変数(.env)から読み込むこと
```

理由が書いてあると、Claude Codeは「なぜダメなのか」を理解した上で判断するため、類似のケース（例: データベース接続文字列のハードコード）にも自発的にルールを適用するようになる。

## パターン4: チェックリスト形式は守られやすい

散文形式:

```
デプロイ前にはテストを実行し、リンターを通し、型チェックも行うこと。
ビルドが成功することも確認する。
```

チェックリスト形式:

```
## デプロイ前チェックリスト
- [ ] `npm test` が全件パス
- [ ] `npm run lint` がエラーゼロ
- [ ] `npx tsc --noEmit` が型エラーゼロ
- [ ] `npm run build` が成功
```

チェックリスト形式にすると、Claude Codeは各項目を順番に実行し、結果を報告する傾向が強くなる。散文だと一部が抜け落ちることがある。

なぜチェックリストが有効かというと、LLMはマークダウンの構造を認識し、各チェックボックスを独立したタスクとして処理するからだ。散文の場合、文中に埋もれた指示を見落とすリスクがある。

特に「完了前に確認すべきこと」のような場面では、チェックリスト形式の効果が大きい:

```
## PR作成前の確認事項
- [ ] 変更に関連するテストを追加した
- [ ] 既存テストが全件パスする
- [ ] CLAUDE.mdの規約に準拠している
- [ ] セキュリティ上の懸念がない
```

## パターン5: フック連携でCLAUDE.mdのルールを強制する

CLAUDE.mdは「お願い」に過ぎない。Claude Codeがルールを忘れる可能性はゼロではない。確実に守らせたいルールは、hooksで強制できる。

例えば「`rm -rf` を使うな」というルールをhookで強制する:

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'INPUT=$(cat); CMD=$(echo \"$INPUT\" | jq -r \".tool_input.command // empty\"); if echo \"$CMD\" | grep -qE \"rm\\s+-rf\"; then echo \"BLOCKED: rm -rf is prohibited\" >&2; exit 2; fi'"
          }
        ]
      }
    ]
  }
}
```

hookは `settings.json` に記述し、`exit 2` で操作をブロックする。CLAUDE.mdのルールが「お願い」なら、hookは「法律」だ。

CLAUDE.mdとhookの使い分け:

| 用途 | CLAUDE.md | hook | 理由 |
| --- | --- | --- | --- |
| コーディングスタイル | ○ | △ | 柔軟な判断が必要なためCLAUDE.mdが適切 |
| 破壊的操作の禁止 | △ | ○ | 100%の確実性が必要なためhookで強制 |
| コミット規約 | ○ | ○ | CLAUDE.mdでガイドし、hookで形式を強制 |
| 機密ファイル保護 | △ | ○ | 漏洩は不可逆なためhookで確実にブロック |
| ディレクトリ構成 | ○ | △ | プロジェクトごとに異なるためCLAUDE.mdで指示 |
| 外部API呼び出し制限 | △ | ○ | 課金リスクがあるためhookで制御 |

hookを手書きするのが面倒な場合は、658以上のexampleから選べる:

```

```

## まとめ

CLAUDE.mdの書き方は、プロンプトエンジニアリングそのものだ。LLMに対する指示の精度は、書き方のパターンで決まる。

5つのパターンを振り返ると:

1. **ホワイトリスト方式** — 禁止より許可を列挙する
2. **具体例の添付** — 1つの良い例と悪い例で精度が上がる
3. **理由の明記** — 「なぜ」があるとルールの適用範囲が正しくなる
4. **チェックリスト形式** — 項目の抜け落ちが減る
5. **hook連携** — 確実に守らせたいルールは強制する
