Anthropicが「社内で数百のSkillを運用して分かったこと」を公開しました。

自社でもSkillを実運用していますが、記事の知見はほぼ実感と一致します。AIエージェントを業務に組み込む上で本質的な内容だったので、要点を解説します。👇

まず前提。Skillは「Markdownファイル」ではなく「フォルダ」です。

中にスクリプト、データ、テンプレート、参照用ドキュメント、さらにはセッション中だけ有効なhook（フック）まで入れられます。

Anthropicは社内の全Skillを9カテゴリに整理していました。代表例つきで：

①ライブラリ/API解説
②動作検証
③データ取得・分析
④業務自動化
⑤コード雛形生成
⑥コードレビュー
⑦CI/CD
⑧障害対応ランブック
⑨インフラ運用

また優れたSkillはこれらの領域を重複せず、いずれかの1カテゴリに収まるように作るべきとのことです。

記事で最もインパクトがあったのがここ。

「検証系Skillが、社内で最も効果を測定できたカテゴリ」。
エンジニアが丸1週間かけて検証Skillを磨き込む価値がある、とまで書かれています。

具体策も生々しい：
・Claudeに動作を録画させ、何をテストしたか目視で確認する
・各ステップで状態をプログラム的にassert（検証）させる

「動いた"つもり"」を潰す仕組みへの投資が、一番効くということです。

作り方のコツ①「当たり前を書くな」

Claudeは既にコードが書けるし、リポジトリも読める。デフォルトでやることを書いても、コンテキストを消費するだけで価値はゼロです。

書くべきは"Claudeの通常の判断をズラす情報"だけ。
例：評価の高いfrontend-design Skillは「Interフォントと紫グラデを避けろ」という、AIが放っておくとやりがちな癖を矯正する内容でできています。

作り方のコツ②「Gotchas（落とし穴）欄」が最も価値が高い

Claudeが実際に踏んだ罠を書き溜める場所です。記事の例が秀逸でした：

・このテーブルは追記型。欲しい行は"最新の作成日時"ではなく"最大バージョン"の行
・このIDはAPIゲートウェイ側ではrequest_id、課金側ではtrace_id。中身は同じ値
・Stagingはwebhook未処理でも200を返す。本当の状態は別テーブルを見ろ

一般論ではなく、現場固有のクセを残すほど精度が上がります。

作り方のコツ③「レールに乗せすぎない」

Skillは再利用される前提です。指示を具体的にしすぎると、少し違う状況に適応できなくなる。

必要な情報は渡す。でも最終判断の余地は残す。
このさじ加減が、地味に一番難しいところです。

作り方のコツ④ descriptionは「要約」ではなく「発動条件」

Claudeはセッション開始時に全Skillのdescriptionを読み、「この依頼に使えるSkillはあるか?」を判断します。

つまりここは"いつ呼び出すか"のトリガー定義。
例：PR監視Skillのdescriptionに「babysit（子守り）」のような起動ワードを仕込むと、狙い通りに呼び出せます。

作り方のコツ⑤「記憶を持たせる」

Skillの中に履歴ファイル（ログ/JSON/SQLite）を置くと、Claudeが過去の実行記録を読み、"前回との差分"を判断できます。

例：日報Skillが過去の投稿ログを参照し、毎回"昨日からの変化"だけを出力する。

本来セッションごとに記憶がリセットされるはずのAIに、文脈が宿ります。

作り方のコツ⑥「スクリプト（部品コード）を持たせる」

よく使う処理を関数として渡しておくと、Claudeは毎回ゼロから定型コードを書く手間から解放され、"次に何をするか"の判断に集中できます。

「火曜日に何が起きた?」と聞くだけで、その場で分析スクリプトを組み立てて答える——そんな芸当が可能になります。

またスキル内にはスキルが呼ばれた時にだけ有効にできる「hook」を作ることができます。

例えば以下です。
・/careful … rm -rf / DROP TABLE / force-push / kubectl delete をブロック。本番を触る時だけONにする
・/freeze … 特定ディレクトリ以外への編集をブロック。「デバッグ中、つい無関係な箇所を"修正"してしまう」事故を防ぐ

常時ONだと邪魔。スキル利用時だけフックを使用するのが良いです。
