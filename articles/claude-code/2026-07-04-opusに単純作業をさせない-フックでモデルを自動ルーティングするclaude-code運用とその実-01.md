## 1. 単純作業に高性能モデルを使っていませんか

変数のリネーム、決まりきったボイラープレートの生成、ファイルの棚卸し——こうした単純作業をOpusのような最上位モデルにそのまま投げているなら、それは時給10万円のアーキテクトにコピー取りをさせているのと同じです。判断力はほとんど使われていないのに、単価だけは最上位モデルのままだからです。

Anthropic公式の料金ページ([Pricing - Claude Platform Docs](https://platform.claude.com/docs/en/about-claude/pricing)、2026年7月5日時点)によると、API単価(100万トークンあたり)は次のとおりです。

| モデル | 入力 | 出力 |
| --- | --- | --- |
| Claude Opus 4.8 | $5.00 | $25.00 |
| Claude Sonnet 5(〜2026/8/31は導入価格。以降$3.00) | $2.00 | $10.00(以降$15.00) |
| Claude Haiku 4.5 | $1.00 | $5.00 |

※後述の実測コストには、この入出力単価に加えてキャッシュ読み書きの料金が含まれます。単価表とトークン数だけでは正確には一致しません。

標準価格に切り替わる9月以降で見ると、Opus:Sonnet:Haiku は入力・出力とも 5:3:1 になります。単純作業をOpusにやらせるということは、この単価差をまるごと無駄にしているということです。

この記事で示すのは3つです。①どのタスクをどのモデルに振るかのルーティング基準、②それをフックで自動化する実装、③実際に同一タスクを動かして測った実測コストです。あらかじめ断っておくと、「マルチモデルにすれば必ず安くなる」という単純な話にはなりませんでした。実測の結果、その主張は素直に修正することになります。詳細は§5・§6で正直に書きます。

## 2. 上流は判断、下流は実装、という発想

「モデルを都度選ぶ」のは地味に面倒です。目の前のタスクにOpusを使うかSonnetを使うか、その都度考えること自体が判断コストになります。そこで発想を変えて、「ルーティングの規則を先に決めておく」ことにしました。一度決めてしまえば、あとは考えなくていい仕組みです。

実際に運用している基準はこうです(自分のハーネスの難易度表を一般化したもの)。

| 階層 | モデル | 担当するタスク |
| --- | --- | --- |
| 上流(維持) | 使用中のモデルそのまま | アーキテクチャ・設計判断、タスク分解、委譲ブリーフの作成、複数成果物の統合、最終レビュー |
| 下流・標準 | Sonnet(デフォルト) | 機能実装、複数ファイルにまたがる変更、リファクタ、テスト、デバッグ、構造化された文章作成——判断は要るが範囲は限定的 |
| 下流・軽量 | Haiku | リネーム、フォーマット、定型のボイラープレート、設定値の調整、仕様が完全に決まった単一ファイル編集、ファイル探索、分類——ブリーフで完全に仕様化されていて判断の余地がない作業 |

迷ったときは上位に倒します。SonnetかHaikuか迷ったら、迷わずSonnetにする。理由は単純な非対称性です。下に誤分類すると成果物の質が落ちるリスクを負いますが、上に誤分類しても単価が少し上がるだけで済みます。損失の大きさが違うので、迷ったら安全な側に倒すのが合理的です。

## 3. フラグファイル + フックで自動化する

ここが実装の核です。仕組みはシンプルで、フラグファイルの有無でモードを表現し、`UserPromptSubmit`フックが毎プロンプトの直前に委譲指示を注入します。モードがONの間は、コマンドを打たなくてもルーティングが効き続けます。

### モードの切り替え(フラグファイルのトグル)

PowerShell:

```
# ON: フラグファイルを作成
New-Item -ItemType File -Force (Join-Path $env:USERPROFILE ".claude\harness\.mode-active") | Out-Null

# OFF: フラグファイルを削除
Remove-Item -Force (Join-Path $env:USERPROFILE ".claude\harness\.mode-active") -ErrorAction SilentlyContinue
```

bash:

```
# ON
touch ~/.claude/harness/.mode-active

# OFF
rm -f ~/.claude/harness/.mode-active
```

### フック本体(UserPromptSubmit)

フラグファイルが存在する間、毎プロンプトの直前にこの指示が注入されます。

PowerShell:

```
$flag = Join-Path $PSScriptRoot ".mode-active"
if (Test-Path $flag) {
    Write-Output "[委譲モードON] 設計判断・タスク分解・最終レビューは上流に残す。機械的な作業(リネーム/整形/定型コード/仕様が完全な単一ファイル編集)はHaikuサブエージェントへ、実装・分析はSonnetサブエージェントへ委譲する。Opusへの委譲は行わない。このターンでは自分で実装しない。"
}
```

bash:

```
#!/usr/bin/env bash
flag="$(dirname "$0")/.mode-active"
if [[ -f "$flag" ]]; then
  echo "[委譲モードON] 設計判断・タスク分解・最終レビューは上流に残す。機械的な作業(リネーム/整形/定型コード/仕様が完全な単一ファイル編集)はHaikuサブエージェントへ、実装・分析はSonnetサブエージェントへ委譲する。Opusへの委譲は行わない。このターンでは自分で実装しない。"
fi
```

### フックの登録

書いたスクリプトを `~/.claude/settings.json` の `hooks` に登録すれば配線は完了です。

```
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "powershell -NoProfile -File \"%USERPROFILE%\\.claude\\harness\\mode-hook.ps1\""
          }
        ]
      }
    ]
  }
}
```

bash版を使う場合は `"command"` を `"~/.claude/harness/mode-hook.sh"` に差し替えるだけです。

### 委譲ブリーフの型

サブエージェントは会話の文脈を一切覚えていないので、渡すブリーフは自己完結している必要があります。実運用では次の型に固定しています。

```
Goal: 何を・なぜ達成するか(1段落)
Files: 読むべき/変更すべき絶対パス
Constraints: 触ってはいけないもの、従うべき規約
Acceptance criteria: 完了の判定基準
Effort budget: 想定作業量(例:「3ファイル読んで2ファイル編集、テスト1回」)
```

呼び出し側はこの型のブリーフとモデル指定をセットでサブエージェントに渡します。

```
Agent({
  model: "sonnet",  // 機械的作業なら "haiku"
  description: "3〜5語のタスク概要",
  prompt: "<上のブリーフをそのまま自己完結させたテキスト>"
})
```

### opusplanや静的なmodel指定との違い

公式の`opusplan`は、Plan Mode中に限りOpus(設計)→Sonnet(実装)を切り替える機能で、Plan Modeという状態に紐づいた二段階の切り替えです。サブエージェント定義への`model: sonnet`のような静的指定は、そのサブエージェントが呼ばれる限り同じモデルに固定されます。

フラグファイル+フック方式は、①通常の会話フロー内でも(Plan Modeでなくても)効く、②Sonnet/Haikuをタスクの性質ごとに毎回動的に判定する、③一度ONにすれば都度コマンドを打たなくていい、という点が異なります。

## 4. ベンダーを跨ぐ: Codexとどう組み合わせるか

Claude Code一本で完結させなくてもいい、というのがもう一つの実利です。実運用ではClaude(上流)が仕様書を書き、それを外部のCodex CLIに委譲して実装させ、Claudeが結果をレビューする、という構成も併用しています。

具体的には、Claudeが仕様書(達成条件・触ってよい範囲・検証コマンドを明記したファイル)を書き、それをCodex CLIにサンドボックス実行の形で渡します。Codexが実装とテストの反復までを自律的に行い、最後にdiffと実行結果をClaudeが確認する、という流れです。仕様書さえしっかりしていれば、Codex側はこの会話の文脈を一切知らなくても実装まで完走できます。

この構成の実利は主に2つです。1つはサブスクの枠を分散できること——2つの独立したサブスク枠を持っているなら、実装作業を片方に寄せることで、もう片方の枠を設計・レビューのために温存できます。単一ベンダーに依存しないという意味でのリスク分散にもなります。もう1つは、単純作業の委譲先をClaude内部のサブエージェントだけに限定しなくてよくなることです。

そしてもう一点、本記事自体の実測値は、別ベンダーのCodexが生JSONと突合してレビュー済みです。ここで念のため補足すると、測定値を生成したのはClaude Code側で、Codexはその原稿・数値・論理を検算する第三者レビュー役です(CodexとClaudeに同じ課題を解かせて実装力を競わせたわけではありません)。Claudeに関する記事をClaude自身にしか検証させないのは方法論として片手落ちだと考えていて、この第三者レビューの工程自体が今回の実測の信頼性を支えています。

Claude→Codexの詳細な実装(仕様テンプレート、サンドボックスの選び方など)はこの記事の主題から外れるので深追いしません。反応があれば別記事にします。

## 5. 実測: 同じタスクでコストを比べる

ここからが実測パートです。測定は次の設計で行いました。

* 同一の実装タスク1件を、(a)上位モデル単独、(b)上流を維持したままのマルチモデル委譲、の2パターンで実行し、トークン消費と料金換算を比較
* 実行は直列(モード切り替えのフラグファイルがグローバルなため、並行実行すると別実験のセッションに委譲指示が混入する)
* 完了報告を鵜呑みにせず、毎回`pytest`の再実行と`git diff`で検収(自己申告と実態が一致するかも採点項目に含めた)
* 課題は自作の小さなログ解析CLIの実装課題(pytestで採点可能な自前のベンチリポジトリ)。守秘義務のリスクのない個人の自作物で、ベンチリポジトリは公開準備中です。

同一プロンプト(TASK.md全文+検証コマンド指示、全Run一字一句同一)を、上位モデル単独(Run A)と、上流を上位モデルに保ったままSonnetへ委譲するマルチモデル構成(Run B)で実行した結果です。

| 項目 | Run A: 上位モデル単独 | Run B: 上流維持+Sonnet委譲 |
| --- | --- | --- |
| 合計コスト | **$0.584** | **$0.980**(+68%) |
| 実行時間 | 1.5分 | 2.2分 |
| ターン数 | 14 | 7 |
| テスト合格 | 13/13 | 13/13 |
| 変更禁止ファイル接触 | なし | なし |

ターン数はRun Bの方が少ないのにコストは高い、という一見奇妙な結果になっています。これは上流Opusが委譲ブリーフの作成とレビューに固定費を払っているためで、内訳は折りたたみに示します。

結論(n=1)は、当初の見立てと違いました。「マルチモデルにすれば常に安くなる」わけではなく、「タスクの種類でルーティングを変える」べきだということです。今回のloglyze課題とこの採点基準では、委譲の固定費(上流モデルが委譲ブリーフを作成しレビューする作業)が単独実行の合計コストを上回りました。ターン数はRun Bの方が少なく済んでいます(7 vs 14)が、それは上流での委譲ブリーフ作成というやり取りに置き換わっただけで、時間短縮にもコスト削減にもつながっていません。委譲が効くのは複数スライスを並列に投げられる場面や、上流のコンテキストが重くなる長期セッションであって、こうした単発の小タスクではない、というのが今回の実測から言えることです。

詳細な内訳を見る

モデル別の内訳は次のとおりです。

* Run A: opus単独 $0.580(+ Claude Code内部の補助Haiku処理 $0.0034 で合計$0.584)
* Run B: opus(上流) $0.628 + sonnet(委譲先) $0.349(+ 補助Haiku $0.0034 で合計$0.980)

Run Bでは、上流opusの委譲ブリーフ作成+レビューだけで$0.628かかっており、これだけでRun A全体の$0.584を超えています。

モデル別トークン生値(JSONの`modelUsage`より):

| Run | モデル | input | output | cacheRead | cacheCreate | cost$ |
| --- | --- | --- | --- | --- | --- | --- |
| A | opus-4-8 | 5,672 | 6,480 | 327,222 | 22,644 | 0.5804 |
| A | haiku-4-5(内部補助) | 3,310 | 18 | 0 | 0 | 0.0034 |
| B | opus-4-8 | 16,548 | 5,420 | 103,514 | 35,755 | 0.6275 |
| B | sonnet-5 | 5,514 | 5,089 | 351,491 | 40,115 | 0.3488 |
| B | haiku-4-5(内部補助) | 3,310 | 18 | 0 | 0 | 0.0034 |

cacheReadはRun Bだと上流103,514+委譲先351,491で合計455,005。システムプロンプト・ツール定義・セッション文脈の再読込みが大きく、キャッシュ効率がheadless運用コストの隠れた本丸になっています。

**事故ログ(実務Tips)**: 最初の計測環境は`~\.claude\`配下にベンチリポジトリを置いていました。Claude Codeがこの配下のファイルを全てsensitive扱いし、headlessモードでは書き込み承認が構造的に通らず、9回の書き込みが全て拒否されて$2.82・13.1分を溶かして計測を破棄しました。教訓は「作業リポジトリは`.claude`の外に置く」「headless自動化の最初の関門はモデルの性能ではなく権限設計」です。

また、「opus単独」のRunでもHaikuが$0.0034だけ自動消費されています(Claude Code内部の補助処理)。`/cost`やJSONを読むときは、意図せず混ざるこの少額を見落とさないよう注意が必要です。

## 6. 下位モデルのリスクを実測する

「安いモデルに落としたらデータを消された」——SNSでよく見る逸話です。これを借り物の話ではなく自分の一次データで検証しました。同一タスク・同一プロンプトを、opus単独(Run A流用)・sonnet単独・haiku単独の3モデルにそれぞれ解かせ、品質を比較しています。

| 採点項目 | opus | sonnet | haiku |
| --- | --- | --- | --- |
| テスト合格 | 13/13 | 13/13 | 13/13 |
| 変更禁止ファイルへの接触 | なし | なし | なし |
| 余計なファイル作成・削除 | なし | なし | なし |
| 指示の取りこぼし | なし | なし | なし |
| 自己申告と実態の乖離 | なし | なし | なし |
| コスト | $0.584 | **$0.323** | **$0.097** |

結論(n=1)は、13/13のテストと差分確認という今回の採点粒度では品質差を検出しなかった、というものです。品質差が無いことを証明したわけではなく、この採点では検出されなかった、という意味です。その範囲では、haikuはopusの約1/6のコストで同じ結果に到達しました。「下位モデルにデータを消された」系の逸話は、少なくともこの条件では再現しませんでした。

ただし、これは無条件で成立する話ではありません。効いていた条件は3つあります。①仕様が完全だったこと(正規表現やシグネチャまで明記されたタスク)、②pytestという客観的な検証ゲートがあったこと、③権限を絞ったハーネス(書き込み対象・実行コマンドを制限)だったことです。品質リスクの正体は「モデルが下位だから」ではなく、「仕様が曖昧なうえに検証がない」という組み合わせで顕在化すると考えています。

単発runの揺れ(事故ログ)

Q-haikuの初回実行は、約10分でハーネス側から強制終了され、成果物がゼロのまま破棄しました(原因未特定・生JSONも残らなかったため定量比較には含めていません)。再実行では0.9分・$0.097・13/13で問題なく完走しています。厳密なベンチマークではなく実運用の判断材料として見る、というこの記事のスタンスの根拠のひとつです。

### 下位に落としてよいタスクのチェックリスト

* ブリーフだけで完全に仕様化されていて、判断の余地が残っていない(リネーム・整形・定型コード・仕様が完全な単一ファイル編集・ファイル探索・分類) → Haikuで良い
* 実装や分析に多少の判断が必要だが、範囲は限定的 → Sonnetをデフォルトにする
* 曖昧な美的判断や、原因・次アクションを考える系の分析QAタスク → 下位モデルに規律(検算・接地の指示)を足しても質は上がらず、むしろ下がることさえある。ここは下げない
* 画面のスクリーンショットやPDFの寸法・位置など、視覚的な細部を正確に読む必要があるタスク → 下位モデルは観察前に推測で埋めがちなので、上流に残す

※上の2項目は今回の実測の裏付けではなく、筆者が別のハーネス運用で得た経験則です。

* 判定に迷ったら上位へ。ミスの非対称性(下げ過ぎの損失 > 上げ過ぎの損失)を思い出す

## 7. まとめ

ルーティングは一度組んでしまえば、あとは何も考えなくていい運用になります。フラグファイルのON/OFFだけで、判断は上流に残り、機械的な作業は自動的に下流へ流れていく。今回の実測で分かったのは「マルチモデルなら常に安い」という単純な話ではなく、単発の明確なタスクは単独実行のほうが安く、委譲が効くのは並列分業や長期セッションだということでした。都合の悪い数字も含めて出したのは、この記事を読んだ人が自分の運用にそのまま使えるようにするためです。
