## はじめに

Claude Code / Desktop / Codex / Gemini の記憶 OSS「KIOKU」を作っています。これまでの流れ:

* **v0.6 (4/24)**: スキルを 4 種類のエージェントで共有できるように
* **v0.7.0 (4/28)**: 記憶 (= セッションログの自動記録) も 4 種類のエージェントで共有
* **v0.7.1 (4/30)**: `agent:` フィールド + 5 件の小さな安全策 (磨き上げ release)

ここまでで「マルチエージェント対応の second brain」の機能としては一通り揃いました。が、KIOKU を実際にしばらく自分で使っていると、**毎回もやっとする 2 つの疑問**が残っていることに気付きます:

1. **「自分の環境でちゃんと動いてるかな?」** — 設定ファイルが 4 〜 5 箇所に分かれていて、どれか 1 つでもズレるとサイレントに動かなくなる。原因を探るのも面倒
2. **「Wiki が知らないうちに腐ってないかな?」** — second brain は使うほどページが増えて、リンク切れ / 重複 / 孤立した古いページがじわじわ蓄積する。気付いた時には「ぐちゃぐちゃ」状態

5/7 〜 5/8 の 8 日間で、この 2 つの疑問にそれぞれ答える **自己診断ツールを 1 つずつ追加した** のが本記事の主題です。タイトルにある 2 つの問いに、**ターミナルで 1 行のコマンド** または Claude に **1 行のお願い** で答えが返ってくるようになりました。

## 1. 「自分の環境がちゃんと動いてる?」を 5 秒で確認する `kioku doctor`

Claude Code の設定 (`~/.claude/settings.json`) / Claude Desktop の設定 / Codex / Gemini の設定 / KIOKU 内部のメタデータ — KIOKU が正しく動くには **4 〜 5 箇所の設定** が揃っている必要があります。これまでは「動かない」と思った時、それぞれを順番に目視で確認するしかありませんでした。

`bash scripts/doctor.sh` 1 行で済むようにしました:

```
KIOKU Doctor

[ok]   OBSIDIAN_VAULT is set: /Users/ryusei/...
[ok]   Vault has wiki/ session-logs/ raw-sources/ templates/
[ok]   ~/.claude/settings.json registers KIOKU session-logger hook
[fail] Claude Desktop config does not register KIOKU MCP server
[warn] Codex config.toml does not register KIOKU MCP server
[ok]   All 4 metadata files report version 0.7.2
[mode] Current install mode: Mode A (MCP-only)
       MCP: ok / Hooks: skip / Cron: skip / Git sync: skip

Summary: 19 ok / 2 warn / 1 fail

Next actions:
  - bash scripts/install-mcp-client.sh --apply
  - Add [mcp_servers.kioku] block to ~/.codex/config.toml
```

22 個の項目を 7 カテゴリ (環境変数 / ランタイム / CLI エージェント / Hook 設定 / MCP 設定 / メタデータ整合 / 依存) に分けて出力。**`[fail]` や `[warn]` には必ず「次にこれを実行してください」が併記されます**。「動かない」と「どこを直せばいい」が同時にわかる、というのが大事です。

`--json` 出力モードもあるので、CI / 運用ダッシュボードに繋げて自動監視することもできます。

ちなみに doctor の出力末尾に出る **`[mode] Current install mode: Mode A (MCP-only)`** は v0.7.3 で追加した部分。KIOKU は v0.7.3 から **インストールの粒度を 3 段階** で選べます:

* **Mode A: MCP-only** — 一番軽量、Claude Desktop / 試用 / セキュリティ意識高めの方向け。Hook なし、自動セッションログなし、Git 同期なし
* **Mode B: Read-only** — *v0.7.x で予定*、エンタープライズ / 法務レビューで「閲覧だけ」させたい場合
* **Mode C: Full memory** — 既存の Manual Setup (全部入り)、本来の使い方

「初日に "全部入り" を強制されない」のがポイントで、まず Mode A で試して気に入れば Mode C に格上げ、という階段を user 自身が選べる。doctor が現在のモードを表示するので、「いま自分は Mode A」と認識した上で評価できます。

### 裏方の安定化作業も同時進行 (簡潔に)

doctor を user に届けるには、テスト基盤と CI が信頼できる状態である必要があります。同時に以下も入れました:

* **メタデータ drift テスト**: バージョン番号 / インストールコマンド / tool 一覧が複数のファイルに分散していて drift しがちな問題を、9 個のテストで自動検出。実は 4/28 にインストールコマンドが旧構文のまま残る事故 (`claude marketplace add` → `claude plugin marketplace add` への変更を見落とし) がありかけたので、その回帰を機械化
* **テストの fast / slow 分離**: 日常テストが 9.8 秒で完走 (`tests/run-quick-suite.sh`)、ネットワーク系の重いテストは別suite に隔離。開発中にテストの timeout を毎回待つ苦痛が消えました

ここはユーザーの直接体感には来ない裏方ですが、「doctor が信頼できる」基盤として効いています。

## 2. 「Wiki が腐ってない?」を 11 軸で測る `kioku_health`

second brain で一番怖いのは「**Wiki が静かに腐る**」こと。リンク切れ、孤立ページ、重複した題、古い note — 誰も気付かない裏でどんどん溜まっていきます。

v0.7.4 で **11 番目の MCP tool** として `kioku_health` を追加しました。Claude Desktop / Code に「自分の KIOKU は健康?」と聞けるようになります:

```
ユーザー: 「kioku_health で私の Vault の健康度を教えて」

Claude: 
  Pages total: 151
  Orphan: 8        (= リンクが張られていない孤立ページ)
  Stale: 0         (= 30 日以上更新されていないページ)
  Duplicate: 0     (= 同じ題のページが複数)
  hot.md age: 14d  (= 短期記憶ファイルの最終更新からの経過)
  Last ingest: 4s ago (74 logs)
  Unprocessed: 5   (= 自動取り込み待ちのセッションログ)
  
  Next actions:
  - 8 つの孤立ページに wikilink を追加するか、wiki/.archive/ に移動
  - 5 件のセッションログが取り込み待ち、cron 確認推奨
```

各指標に **「次にこれをすると改善する」** が必ず付いてきます。Claude が **読み取りのみ** で診断するので、Wiki を勝手に書き換える心配はありません。

### 初日に走らせて分かったこと

v0.7.4 が land した日の朝、自分の 151 ページの Vault に対して走らせてみました。**孤立ページが 8 件、取り込み待ちが 5 件** すぐに見つかりました。「リンクし忘れ」で寝かせていた古い概念ページ達です。Sprint 2 の目的だった「**見えなかった "腐りかけ" を機械的に表面化する**」が、初回実行で達成されました。

### v0.7.5 で 5 種類の指標を追加 (合計 11 種類に)

同じ日の夜の v0.7.5 で、もう 5 種類の指標を追加:

* **`broken_wikilink`** — `[[link]]` で参照しているのに、リンク先のページが存在しない件数 (= リンク切れ、ページ名変更後の貼り直し漏れ)
* **`source_sha256_duplicate`** — 同じ source ハッシュを持つページが複数 (= 同じ素材を 2 回取り込んでしまった)
* **`pages_warm_zone`** — 7 〜 30 日前に更新されたページ件数 (= 更新が新しくも古くもない「ぬるま湯」帯、stale 化前に再訪する候補)
* **`page_count_by_type`** — type (concept / project / decision / source-summary / log / index / meta / その他) 別の件数 (= 記憶の偏りが見える)
* **`summaries_growth_rate`** — 直近 7 日 / 30 日の取り込み要約ページ数の増加 (= 取り込みパイプラインが健康に回っているか時間軸で測る)

私の Vault が 155 ページに増えた状態で v0.7.5 を走らせると:

```
Broken wikilinks: 21
Source sha256 duplicates: 2
Pages warm zone (7-30d): 99
Page types: analysis=14, bug=14, concept=9, decision=23, ...
Summaries growth (7d/30d): 0 / 33
```

21 件の **リンク切れ** — ページをリネームした後に参照元を直し忘れた跡。2 件の **重複** — 自分で手動追加した文書と auto-ingest が独立に処理した結果のダブり。Sprint 2 で狙っていた「**深い淀み**」が surface (表面化) されました。

### 「機械が測れること」と「LLM にしか判断できないこと」の役割分担

v0.7.5 の地味だが構造的に大事な変更が、**`kioku_health` (機械検査) と `auto-lint` (LLM 監査) の役割分担を明示化** したこと。

これまで KIOKU の `auto-lint` は LLM (`claude -p`) に 6 観点で wiki を監査させていました: 矛盾 / 孤立 / 概念欠落 / 古い主張 / 相互リンク / frontmatter 不備。**このうち 2 つ (孤立 / frontmatter) は `kioku_health` が機械的に検出可能**。LLM のトークンを払って Node スクリプトで分かることを検出していた、というムダがあった。

v0.7.5 で `auto-lint` を 4 観点に絞り込み、**`kioku_health` の 11 種類は明示的に exclude**:

1. 概念の矛盾 (LLM が複数ページを横断して意味を比較)
2. 概念の splinter (= 同じ概念で merge 候補を見つける)
3. 専用ページ昇格候補 (sources で繰り返し言及されているのに、まだ概念ページがないもの)
4. 意味的なリンク欠落 (リンク構文の broken は kioku\_health で検出、ここは「意味的に繋がるべきなのに張られてない」)

→ **機械検査 (Node、速い、安い)** と **LLM 判断 (slow、トークン高い)** が **重ならない範囲を担当する** 設計に変わりました。LLM のトークン予算は「LLM にしかできない判断」に集中。

## 3. (副次) なぜ 8 日でこの規模が回せたか — multi-Claude セッションの運用

ここからはエンジニア向けの副次トピックです。「8 日で 4 リリース、5 PR、自己診断ツール 2 つ」という cadence は heroic な sprint の結果ではなく、**運用構造の帰結** という話。

KIOKU のリリースパイプラインは **役割の異なる 4 つの Claude セッション** で回しています:

```
[Codex CLI セッション]              ← 戦略担当、月単位
   ↓ ロードマップを書く (4/30 執筆)
[PM Claude セッション]              ← 戦術担当、Sprint 単位
   ↓ delegation handoff (1 PR につき 1 通)
[制作 Claude セッション (executor)] ← 実装担当、PR 単位
   ↓ branch + PR で land
[マーケ Claude セッション]          ← retrospect 担当 (この記事を書いている)
```

各セッションが **明確に分離された役割** を持ち、互いの領分には踏み込まない。「PM 思考」と「executor 思考」が同じセッションで競合しない、というのが効きます。コストとしてはセッション間の通信 overhead がありますが、handoff document が必ず書面で残るので、新しいサイクルのセッションも再現できる。

具体的な codification:

* **`kioku-delegation-handoff` スキル** (PR #101): 6 サイクル共通の雛形を抽出。handoff 作成のコストが **30 分 → 5 〜 10 分** に短縮、本サイクル (v0.7.4 / v0.7.5) はスキル使用 1 サイクル目 / 2 サイクル目
* **LEARN convention** (#5 / #6 / #10 / #11 / #12 / #13 / #14): 過去のサイクルで実際に踏んだ失敗パターンをルール化、各 executor セッションが session 開始時の precheck で trip wire として使う
* **branch + PR mandatory** (LEARN#12): main 直 commit を構造的に禁止。executor セッションは branch assert を毎 commit / push の前に実行、drift があればコマンドが abort

「OSS のリリース運用を multi-Claude セッションのチェーンで機械化する」と書くと大袈裟ですが、**1 人 + AI セッションチェーンで 1 リリース / 日 ペース** が可能な状態は実現できる、というのが本サイクルでの実証でした。

## 試し方

```
# Plugin marketplace (推奨)
claude plugin marketplace add megaphone-tokyo/kioku
claude plugin install kioku@megaphone-tokyo

# 「自分の KIOKU はちゃんと動いてる?」 — 5 秒で診断
bash scripts/doctor.sh
bash scripts/doctor.sh --json | jq .summary.install_mode

# 「Wiki は腐ってない?」 — 11 種類の指標で診断
bash scripts/generate-health.mjs --dry-run    # 端末出力のみ (書き込みなし)
bash scripts/generate-health.mjs              # wiki/meta/health.md を生成

# Claude Desktop / Code から
# 「kioku_health で私の Vault の健康度を教えて」
```

`bash scripts/generate-health.mjs --dry-run` は **読み取りのみ** で 11 種類の指標を端末に出力します (Wiki には何も書きません)。気軽に試してみて、気になる数字があれば対応する `next_action` に従う、という流れ。

## 次の Sprint と結び

5 月末から 6 月にかけて、**Sprint 3「価値の可視化」** が始まります。3 つのキーワード:

* **Visualizer の初期画面の改善** — 4/24 から保留されていた V-1 hotfix の延長線
* **`raw-sources → summary → wiki` のグラフ表示** — 取り込みから知識化までのパイプラインを連結グラフで見せる
* **マーケティング用スクリーンショット + デモ動線** — ランディングページに使える素材

Sprint 1+2 との連結は明快で、`kioku_health` の 11 種類の指標と `auto-lint` の 4 観点が **Visualizer の入力** になります。リンク切れや重複ページをグラフ上で highlight、ぬるま湯帯のページを stale と違う色で塗る。Sprint 3 は **記憶の健康が「測れる」から「見える」に変わる Sprint** です。
