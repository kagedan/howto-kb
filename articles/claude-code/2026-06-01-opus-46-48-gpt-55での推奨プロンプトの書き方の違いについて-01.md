## 概要

Claude Opus 4.6/4.7 / 4.8やGPT 5.5では、各社が提示している推奨プロンプトの形式が結構異なります。この記事では、公式ドキュメントと関連論文をもとに、各モデルでプロンプトの書き方を整理します。

---

## 先に結論

| モデル | 向いているプロンプト | 使い方のイメージ |
| --- | --- | --- |
| Opus 4.6 | 相談型・ペアプロ型 | 曖昧な依頼でも意図を補ってもらう |
| Opus 4.7 / 4.8 | 仕様明示型 | スコープ、制約、受け入れ条件等も書く |
| GPT 5.5 | 目標設定型 | ゴールだけ先に決める。目標達成までの経緯はAI任せ |

よく分からない、という方はとりあえずOpus 4.6を選んでおけば外れは少ないかと思います。

ただOpus4.6以前とそれ以降のモデル(4.7/4.8/GPT5.5)  
でガラッとプロンプト処理の性質が変わるので  
その点を今後意識した方がよりパフォーマンスが出せるかと思われます。

## 先に結論2

Opus4.6以前と移行のモデルでそれまでの常識がガラリと変わりました。  
不足しているコンテキスト情報を推論で補っていたのが4.6以前です。

これは長所でもあり、短所でもありました。  
指示した事以外の能力をくみ取ろうとするあまり、指示追従性能の低下にも繋がっていました。

それを改善したのが4.7や5.5です。

進化した方向性は随分異なりますが、両モデルに共通して言えるのは「それまでのプロンプトの書き方は通用しなくなる」点です

ネットやXでOpus4.7は性能が下がったとか言われがちなのはこの推奨プロンプトの形式の変化によるものが大きく、それに気が付かないで使用している人々が多い事が一因かと思われます。

今後もこの2社のモデルを使っていくなら、**これまでの「いい感じにやって」型のプロンプトだけでなく、新しい書き方に慣れていく方がよさそう**です。

## Opus 4.6: 従来の相談型プロンプトが通じやすい

Opus 4.6は、従来のClaudeらしい「一緒に考えてくれる」感覚が強いモデルです。  
たとえば、こういう依頼です。

```
この実装、ちょっと怪しいので見てください。
必要ならテスト観点も含めて、直し方を提案してください。
```

このような書き方でも、それまでのやり取りや前後の文脈を見て意図を補ってくれることが多いです。何が必要なのかをモデル内部で推論を重ね、推測する事でより良い回答を引き出そうとします。これは非常に便利な点です。

コードレビュー、設計相談、記事の構成、仕様の壁打ちのようなあらゆる作業において、この曖昧さを含んだまま相談ができます。AIに不慣れな人にとってはこちらの仕様の方がありがたいでしょう。

一方で、欠点もあります。意図を補ってくれるということは、こちらが明示していない範囲までその解釈の意図を広げる可能性があるということです。

と頼んだときに、古いfixtureだけでなく、命名や周辺設計まで触りにいくかもしれません。  
rulesやhooksで行動を縛ることも出来るでしょうが、不確定性に関わるのは間違いないでしょう。

## Opus 4.7 / 4.8: 仕様明示型に寄っている

Opus 4.7で大きく変わったのは、指示の解釈です。  
AnthropicのMigration guideでは、Opus 4.7について次のように説明されています。

> Claude Opus 4.7 interprets prompts more literally and explicitly than Claude Opus 4.6.  
> (Claude Opus 4.7 は、Claude Opus 4.6 よりもプロンプトを文字通りに、そしてより明確に解釈する)

さらに

* 書かれていない要求を推測しない
* ある項目への指示を別項目に勝手に一般化しない

といった説明もあります。

よくOpus4.6と4.7を比較して  
「回答内容が無機質/冷たくなった」「性能が低下した」という人もいますが  
この変更点が関係しているからかもしれません。  
(リリース当初のバグの問題とかも関係あるかもしれませんが)

曖昧な補完よりも、堅実性を重視する思想へシフトしている、と言えるのではないでしょうか。

ともあれ、これらの詳細はclaude code公式が出している  
マイグレーションガイドの[破壊的な変更]の項目に詳しく書かれています。

<https://platform.claude.com/docs/ja/about-claude/models/migration-guide>

### Opus 4.7(4.8)の推奨プロンプトについて

とある関数の修正を指示したいとき。  
4.7(4.8)では、こういう書き方の方が合います。

```
対象:
- 対象関数のパスや位置

要件:
- 型は〇×である
- 特定条件下の時にはこのIF条件を追加せよ
    -　ケースA　〇×▽
    -　ケースB　××▽
制約:
- 外部ライブラリは使わない
- 他ファイルは変更しない

受け入れ条件:
- テストを列挙する
- 仕様上の未決定点があれば最後に明記する
```

「いい感じに」といった曖昧さに頼るのではなく  
その時のスコープ、要件、制約、受け入れ条件等を細かく書く事がポイントです。

逆に言うと上記のように細かく書かないなら、Opus4.6の方が良いです。

あと余談ですがOpus4.6と4.7は使用しているトークナイザーが異なります。  
これが最大1.3倍ほど違います。

> Claude Opus 4.7 uses a new tokenizer

> roughly 1x to 1.35x as many tokens

と公式に名言されており、全く同じテキストでも4.7は4.6より最大約35%多くトークンを消費する場合があると説明されています。

## GPT 5.5: 目標設定型

GPT 5.5について、OpenAIのドキュメントで特に重要なのは `outcome-first prompts` です。

OpenAIの説明では、GPT 5.5は明確なゴールから作業を進め、制約を保持し、プロダクトの意図を具体的な次のステップへ変換するのが得意だとされています。

つまりゴールの設定だけとにかくする事。  
道中の細かい経路などは、AIに判断をさせた方がうまくいきますよ、という思想です。

むしろ正確な手順が必要な場合を除き、細かいstep-by-stepのプロセス指示は減らすことが推奨されています。

> GPT-5.5 works best when prompts define the outcome and leave room for the model to choose an efficient solution path.  
> (GPT-5.5は、プロンプトで結果を明確に示し、モデルが効率的な解決策を選択できる余地を残しておく場合に最も効果を発揮します)

<https://developers.openai.com/api/docs/guides/prompt-guidance>

例としてはこうです。

```
Goal:
会計ロジックを処理するための関数を完成させる。

Success criteria:
- 不正値、空文字、負数、過去日時を安全に扱う
- テストしやすい設計にする
- 境界値テストの観点を提示する

Constraints:
- 外部ライブラリは使わない
- 実装上の未決定点は推測せず、明示する

Output:
- 実装案
- テスト観点
- 注意点
```

これは、Opus 4.7 / 4.8向けの「対象、制約、受け入れ条件を厳密に書く」プロンプトと似ていますが、焦点が少し違います。

Claude 4.7 / 4.8向けは、作業範囲や禁止事項をかなり明示する。  
GPT 5.5向けは、達成すべき状態を定義して、そこへの経路はモデルに任せる、ですね。

GPT 5.5に対しては「この順番でこうやって」よりも、  
「最終的に何が満たされていれば成功か」を目標を示す方が合います。

この差はかなり大きいです。

## 具体例

ここでは、エンジニア向けに小さな例で見てみます。  
題材は、HTTPの `Retry-After` ヘッダーを扱う関数です。

`Retry-After` は、仕様上ざっくり次の2種類があります。

```
Retry-After: 120
Retry-After: Wed, 21 Oct 2015 07:28:00 GMT
```

数字なら秒数、HTTP-dateなら現在時刻との差分として扱う、という小さな実装です。  
ただし、実務では不正値、空文字、負数、過去日時などの扱いも必要になります。

元コードをこう置きます。

```
function parseRetryAfter(value: string | null): number | null {
  if (!value) return null;
  return Number(value);
}
```

この同じタスクでも、モデルごとに渡し方を変えた方がよさそうです。

### Opus 4.6向け: 相談型

```
このRetry-Afterヘッダーのパース処理が少し不安です。

function parseRetryAfter(value: string | null): number | null {
  if (!value) return null;
  return Number(value);
}

HTTPヘッダーとして安全に扱えるようにするには、
どんなケースを考えるべきか、実装方針とテスト観点を含めて提案してください。
```

4.6では、曖昧さを残して相談する形が使いやすいです。  
複数のチャットを通し、少しづつコンテキストを集める事で精度を上げさせます。

プロンプトに曖昧さが残っていても、AIに不慣れな人にとってはその方が逆に品質が向上します。

### Opus 4.7 / 4.8向け: 仕様明示型

```
以下の関数だけを修正してください。

対象:
- parseRetryAfter 関数のみ

仕様:
- value が null または空文字なら null を返す
- 数字文字列は秒数として返す
- HTTP-date形式は now からの差分秒数として返す
- 不正値、負数、過去日時は null を返す

制約:
- 戻り値は number | null のまま
- テストしやすいように now: Date を第2引数で受け取る
- 外部ライブラリは使わない
- 関数以外のコードは変更しない

受け入れ条件:
- 境界値テストを箇条書きで列挙する
- 仕様上の未決定点があれば最後に書く
```

4.7 / 4.8では、曖昧な期待を空気で読ませるより、条件を明示する方が合います。

特に4.7は、公式に「よりliteral」と説明されています。  
つまり、こちらが書かなかった暗黙の期待は、以前より拾われにくい可能性があります。

### GPT 5.5向け: 目標設定型

```
Goal:
Retry-Afterヘッダーを安全に扱える小さなユーティリティを完成させる。

Context:
現在の実装は Number(value) だけなので、HTTP-date形式や不正値に弱い。

Success criteria:
- 数字形式とHTTP-date形式の両方に対応する
- null、空文字、不正値、負数、過去日時を安全に扱う
- テストしやすい設計になっている
- 境界値テストの観点が分かる
- 仕様上の未決定点は明示されている

Constraints:
- 外部ライブラリは使わない
- 戻り値は number | null を基本にする

Output:
- 実装案
- テスト観点
- 注意点
```

(ここまで細かく書かなくて良いと思いますが)GPT 5.5では、実装手順を細かく指定するより、ゴールと成功条件を渡す方が公式の推奨に近いです。ベストな経緯はモデル自身が考えてくれます。

## まとめ

今の時点で、自分の作業内で迷うならOpus 4.6を選ぶのはかなり現実的です。

曖昧な相談に強く、ペアプロ的に扱いやすいからです。  
とりあえず壁打ちして、途中で人間が方向修正する使い方には向いています。

ただし、今後もずっと4.6が使えるとは限りません。  
新しいモデルほど、次の方向に寄っています。

* 指示をより明示的に解釈する
* 作業のスコープや制約を重視する
* ツール利用や長時間作業を前提にする
* ゴールと成功条件から自律的に進める

これは、AI側が賢くなったから人間が何も書かなくてよくなる、という方向ではありません。  
むしろ、**人間側が仕事の渡し方をうまく構造化する必要が出てきている**ように見えます。

「いい感じに直して」だけで済む時代から、

```
何を達成したいのか
何を変えてよいのか(変えてはいけないのか)
何をもって完了とするのか
どの情報を根拠にしてよいのか
```

を明示する時代へ移っているのかもしれません。  
とはいえこの記事はあくまでプロンプトの話です。

ルールベースでドキュメントを整備しスペック駆動開発へ移行できていれば、当然また話は違ってくるでしょうが…本記事ではプロンプトを扱った記事の為、この辺りの話は割愛します。
