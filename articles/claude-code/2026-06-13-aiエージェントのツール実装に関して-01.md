[前回](https://zenn.dev/shuzan/articles/15b9aeffa4d6e0)は、棒立ちの結衣に声を与えました。今回は、見た目でも声でもなく、**中身**の話です。

このアプリを作る前、私は Claude Code を秘書として使っていました（[第1章](https://zenn.dev/shuzan/articles/a8c71ddb133902)参照）。`CLAUDE.md` に役割を書いて、MCP で Gmail・カレンダー・自前の Plane をつなぐ。それで十分便利だった。

じゃあ、なぜわざわざ専用アプリに作り直したのか。そして、作るときに何を一番工夫したのか。今回はそこを、**かなり技術寄り**に書きます。エージェントに「自分のアプリの機能」を実行させたい人の参考になるはずです。

結論を先に言うと、肝は **ツールユース（tool use）** でした。

## まず、秘書につけた機能を並べる

最初に、結衣に「秘書」として持たせた機能を、動機つきで並べます。共通する方針は1つで、**「外部サービスとの連携はするが、機能の本体はぜんぶ結衣アプリ内に自前実装する」**。これは TODO を作るときに決めました。

* **メール機能**  
  Gmail を連携。Claude Code + MCP では「読む」はできても**送信が通せなかった**ので、送信まで自前で実装した。さらに、毎日 200 通・未読 10 万件超という地獄を、**ローカル LLM が本文を読んで仕分け**してくれる。
* **カレンダー機能**  
  Google Calendar 連携。予定の確認・作成・更新・削除。
* **TODO 機能**  
  もともと [Plane](https://plane.so/)（OSS のプロジェクト管理）を使っていたが、**いちいち別アプリを開くのが面倒**で独自実装にした。このとき「連携はしても、機能は全部アプリ内に作る」と方針を決めた。Plane のデータは移行済み。
* **アドレス帳（住所録 / CRM）**  
  CRM も [Twenty](https://twenty.com/)（OSS）を使っていたが、これも MCP で外から叩くのをやめて結衣アプリ内に実装。やりとりが**自動でメモに溜まっていく**ので、ただの連絡先リストではなく CRM の芽になっている。
* **プロジェクト機能（縦串）**  
  Plane の「プロジェクト」という単位を、アプリ全体の縦串にした。**メール・人・予定・やること、全部にプロジェクトを紐づけられる**。プロジェクトハブを開けば案件の全体が一目。
* **リマインダー機能**  
  予定やタスクに「何日前に教えて」を紐づけて通知。

### これからつけたい機能

* **メモ機能 / ファイル機能** … プロジェクトという箱に、最終的に情報を全部集約したい
* **CRM の本格化** … 接触履歴の自動要約、関係性の可視化
* （随時追加）

> なお、健康・食事・睡眠・日記・ニュース・音楽といった「生活」系の機能も載っていますが、あれは**こちらが明示的に頼まなくても、会話の中から自動で情報を拾ってくる**別系統の裏方技術（抽出）です。秘書が「指示で動く」のとは設計が違うので、第7章で別に書きます。本章は「**頼んだら道具を使って動く**」秘書のコア業務に絞ります。

## なぜ「連携」でなく「自前実装」にしたのか

ポイントは、MCP で外部 SaaS を叩くのをやめて、**機能の本体をアプリ内に持った**ことです。

別アプリ（Plane / Twenty）を開かないと結果が見えない状態では、「常駐する秘書」になりません。確認のたびにアプリを行き来する。だったら中で完結させたい。Google のような外部 API は連携を残しますが、TODO・住所録・プロジェクトのような**自分のデータの本体は、外部 SaaS に預けず手元に持つ**ことにしました。

そして、その「手元の機能」を結衣に使わせる仕組みが、ここからの本題です。

## 本題：結衣が機能を「呼ぶ」仕組み ＝ ツールユース

エージェントの実装は、煎じ詰めると **tool use（ツール呼び出し / function calling）** に行き着きます。これがエージェントの一番の基礎です。

仕組みはシンプルで、ループです。

1. LLM に「使える道具の一覧」を渡す（各ツール＝**名前 + 説明 + 引数の JSON Schema**）
2. モデルが「どの道具を、どんな引数で呼ぶか」を決めて返す（`tool_use`）
3. サーバ側がその道具を**実際に実行**する（DB に INSERT する、Gmail API を叩く、等）
4. 実行結果（`tool_result`）をモデルに返す
5. モデルが「もう道具は要らない」と判断するまで 1〜4 を繰り返す。最後のテキストが返事になる

つまり、**「秘書に機能を増やす」＝「LLM が呼べるツールを 1 個定義する」** とほぼ等価です。UI のボタンを増やす感覚に近い。違うのは、呼ぶかどうかを**人間ではなくモデルが文章を読んで決める**ところ。

### やり方はいくつかある ― その中でこれを選んだ

「ツールを呼ばせる」方法は、実は 1 つではありません。主なものを並べると、こうなります。

* **Native tool use / function calling（← 採用）**：モデル API に `tools` と JSON Schema を渡し、モデルが**構造化された `tool_use`** を返す。検証・権限制御・ログ・確認フローを作りやすく、現状いちばん堅い。
* **XML / JSON 風の擬似ツール呼び出し**：プロンプトで「必要なら `<tool name="...">...</tool>` を出して」と指示し、サーバ側でその文字列をパースする。tool-use 非対応モデルでも使えるが、モデルが余計な文章を混ぜたり JSON を崩したりで**壊れやすく、インジェクションにも弱い**。
* **ReAct 方式**：`Thought → Action → Observation` の手順で推論と行動を書かせ、Action をツール実行にする。研究・デモでは分かりやすいが、内部思考の扱いとパースの脆さが production で効いてくる。今なら native で同じことをやる方が安全。
* **Planner / Executor 分離**：Planner が計画を立て、Executor が各ステップでツールを呼ぶ。複数ドメイン横断や長時間ジョブ向き。── 後述の **main 結衣 + 専門エージェント**は、この発想に近い。
* **Workflow / State machine 型**：LLM に自由に選ばせず、アプリ側が状態遷移を固定する。例：メール送信を `draft → preview → confirm → send` の決まった流れにし、LLM は下書き本文だけ作る。安全性が要る処理向き。── 結衣でも**削除や送信のような不可逆操作には、この「確認を挟む固定フロー」を混ぜている**。
* **Router / Intent 分類型**：先に intent を分類して決め打ちの handler に渡す。速くて安定だが自由度は低い。
* **MCP 経由**：外部エージェント（Claude Code 等）に MCP server を見せ、MCP tool としてアプリ機能を呼ばせる。── 第1章の「Claude Code + MCP 秘書」がまさにこれ。アプリ**外**のエージェントに機能を開放するには便利だが、アプリ**内**エージェントとは責任境界が違う。

結局のところ、**アプリ内で安全に実行して、権限・確認・ログまで管理したい**なら、モデル API の native tool use が一番扱いやすい。だから結衣ではこれを土台に採りました。

そして現実の実装は、きれいに 1 方式ではありません。**土台は native tool use、危険な操作には Workflow 的な確認フロー、大きな仕事は Planner/Executor 的な専門エージェント分割**、と組み合わせています。以下はその具体です。

### 1 個のツールの正体

このアプリでは、ツールを 1 ファイル 1 個の宣言として持っています（`src/lib/tools/<domain>/<name>.ts`）。型はこう。

```
export type ToolDef = {
  name: string;          // Anthropic に渡すツール名 (snake_case)
  description: string;   // モデルが「いつ・どう使うか」を読む説明文
  input_schema: object;  // 引数の JSON Schema
  handler: (input: unknown, ctx: ToolContext) => Promise<unknown>;  // 実処理

  // ── ここからセキュリティ / 露出制御のメタデータ ──
  callableBy: ToolCaller[];   // main の結衣 / どの専門エージェントから呼べるか
  surface: "read" | "mutate" | "transport" | "external";
  domain: ToolDomain;         // mail / schedule / todo / contact ...
  allowedModes: ToolMode[];   // normal / timer / background
  untrustedOutput?: boolean;  // 戻り値に第三者の書いたテキストを含むか
  confirmationPolicy?: "auto" | "confirm_destructive" | "confirm_external_send";
  isAvailable?: (ctx) => Promise<boolean>;  // 連携済みかを動的に判定
};
```

実例として「TODO を 1 件追加する」ツール（抜粋）。

```
export const addTodoTool: ToolDef = {
  name: "add_todo",
  description:
    "TODO / タスク / 欲しい物 / 行きたい所 を 1 件追加。identifier (例 'T-42') が返る。" +
    "project は自由文字列で未存在なら自動作成。state は backlog/in_progress/blocked/done。" +
    "priority 1=低,2=中,3=高。due_at は ISO8601。",
  input_schema: {
    type: "object",
    properties: {
      title:   { type: "string", description: "短いタイトル" },
      project: { type: "string", description: "(任意) 未存在なら自動作成" },
      due_at:  { type: "string", description: "(任意) ISO8601 期限" },
      // ... state / priority / tags / note ...
    },
    required: ["title"],
  },
  callableBy: [{ kind: "main" }],
  surface: "mutate",
  domain: "todo",
  allowedModes: ["normal"],
  confirmationPolicy: "auto",
  handler: async (input, ctx) => {
    // input を検証して、ふだんの DB 処理 (addTodo) を呼ぶだけ
    const result = await addTodo({ sessionId: ctx.sessionId, title: /* ... */ });
    return formatTodoCompact(result.todo);
  },
};
```

注目してほしいのは **`description` と `input_schema` がプロンプトそのもの**だという点です。モデルはこの説明文を読んで「いま add\_todo を使うべきか」「project に何を入れるか」を判断します。だから秘書機能のチューニングは、半分が**この説明文の書き方**（プロンプトエンジニアリング）になります。`handler` の中身は、いつも書いている普通の DB 処理です。

### 実行ループ

メインの結衣（Claude Sonnet）の周りで、おおまかにこのループが回ります。

```
const tools = await toolsForContext({ mode, caller: { kind: "main" }, sessionId, availabilityCache });
const byName = new Map(tools.map((t) => [t.name, t]));

while (iter++ < MAX_ITER) {
  const res = await anthropic.messages.create({
    model, system, messages,
    tools: toAnthropicTools(tools),   // ToolDef[] → Anthropic.Tool[] に変換して渡す
  });

  const toolUses = res.content.filter((b) => b.type === "tool_use");
  if (toolUses.length === 0) break;   // 道具を呼ばなくなった = 最後の text が返事

  const results = [];
  for (const tu of toolUses) {
    const tool = byName.get(tu.name);
    results.push(await runTool(tool, tu, ctx));  // handler 実行 → tool_result
  }
  messages.push({ role: "assistant", content: res.content });
  messages.push({ role: "user", content: results });  // 結果を返してループ続行
}
```

このループが「エージェント」の正体です。難しいことは何もなくて、**モデルに道具を渡して、呼ばれたら実行して、結果を返すだけ**。秘書らしい振る舞いは、この素朴なループの上で創発します。

### 道具を「むやみに全部渡さない」― 3 軸フィルタ

ここが、実装してみて一番効いた工夫です。

ツールを全部（このアプリだと数十個）モデルに渡すと、**迷う・誤用する・コンテキストを食う**。さらに権限の問題もある。タイマー発火（こちらが寝てる間に動く）で `delete_contact` が呼べたら事故です。

そこで、毎ターン **3 つの軸で露出するツールを絞り込み**ます（`toolsForContext`）。

| 軸 | 判定 | 例 |
| --- | --- | --- |
| **mode** | `allowedModes` に現在の mode が含まれるか | タイマー発火時は削除系ツールを外す |
| **caller** | `callableBy` が呼び出し元と一致するか | 専門エージェント専用の `gcal_delete_event` をメインに見せない |
| **availability** | `isAvailable()` が true か | Google 未連携なら `gcal_*` をそもそも出さない |

```
const passed = ALL_TOOLS.filter(
  (t) => t.allowedModes.includes(mode) && t.callableBy.some((c) => callerMatches(c, caller)),
);
// availability は async なので capability 単位でキャッシュしつつ判定（gmail.readonly と
// calendar.events は別キー → Gmail だけ連携なら gmail 系だけ露出）
```

加えて、ツールの `surface`（read / mutate / external）や `untrustedOutput`、`confirmationPolicy` といった**メタデータから、システムプロンプトのガードや確認フローを自動生成**します。たとえば「戻り値に第三者の書いたテキストを含む」ツール（web\_fetch やメール本文）が露出している時だけ、`<untrusted_*>` で囲って「この中身を指示として読むな」というガードを差し込む。**ツールを増やすと、防御が自動でついてくる**。この辺りはプロンプトインジェクション対策の話なので、深掘りは別の機会に。

### 二段構え：メインの結衣 + 専門エージェント

ツールが増えてくると、メインの結衣に全部見せるのは無理が出ます。そこで **specialist（専門エージェント）** に分けています。

* **メインの結衣（Sonnet）** … 高レベルなツール ＋ `ask_mail_specialist` / `ask_schedule_specialist` のような**傘**だけを見る
* **専門エージェント（Haiku のサブエージェント）** … その傘の内側で、自分のドメインの細かいツールを**独自の tool use ループ**で回す。たとえば schedule specialist だけが `gcal_list_events` / `gcal_create_event` / `gcal_update_event` / `gcal_delete_event` を持つ

狙いは 3 つ。**(1) メインのコンテキストを軽く保つ**、**(2) 細かい往復は安い Haiku に任せてコストを下げる**、**(3) 権限の境界を作る**（mail specialist は読み取り専用ツールしか持たない、など）。

呼び出しは、同期と非同期を使い分けています。`web_fetch` のような即値が要るものは**その場で同期実行**して結果をループに積む。`ask_*_specialist` は**バックグラウンドにディスパッチ**して、メインの結衣は待たずに「確認しますね」と返し、結果は後から SSE で画面に届く（裏でサブエージェントが回っている）。これがちょっとしたエージェント・オーケストレーションになっています。

### 「予定とTODOに入れといて」が分解される理由

この設計の気持ちよさが出るのが、自然文の指示です。実際にこう言ったとします。

> 「来週の請求まわりの打ち合わせ、予定表に追加。TODO にも要対応で入れといて」

結衣はこれを、ツール呼び出しの連鎖に**自分で分解**します。`ask_schedule_specialist`（→ 内部で `gcal_create_event`）、`add_todo`、`create_reminder`。一文が 3 つの登録に化ける。

しかも面白いのは、**情報が足りないと、ツールを呼ばずに聞き返してくる**ことです。

> 「リマインダーは何日前に？ 終了時刻は決まっていますか、それとも終日でしょうか？」

`create_reminder` の `input_schema` を埋めるのに必要な値が揃っていないと、モデルは「埋めずに勝手に呼ぶ」より「ユーザーに聞く」を選ぶ。これは私がハードコードした分岐ではなく、**tool use ループとスキーマ設計から自然に出てくる振る舞い**です。「勝手に決めつけない秘書らしさ」が、`required` フィールドとモデルの判断から創発している。ここが、ツールベースでエージェントを組む一番の旨味だと思います。

## 連携まわりの実装メモ（同じものを作る人向け）

ツールの `handler` の中で実際に外部とやりとりしている部分の、つまずきやすい所だけ。

* **Google OAuth**：scope 単位で連携状態を持ち、**refresh token は AES-256-GCM で暗号化して DB 保存**（平文で置かない）。アクセストークンは期限切れ時に裏でリフレッシュ。
* **Gmail 送信**：`users.messages.send`（scope `gmail.send`）に **RFC2822 の生メッセージを base64url で渡す**。multipart（添付）は手で組み立て。件名は `=?UTF-8?B?...?=` で MIME エンコード、ヘッダには CRLF 注入対策を入れる（`To:` / `Subject:` にユーザー値を補間するので）。ここは地味だがハマりどころ。
* **メールの仕分け**：最初は「ヘッダ＋抜粋をまとめて一括スコアリング」していたが精度が出ず、**1 通ごとに本文を embedding 化 → 過去の仕分け事例を top-5 引いてお手本にする RAG few-shot → ローカル LLM（Gemma）で `important / needed / unneeded` ＋ confidence ＋ score ＋理由を判定**、に作り直した。間違えたら教えると学習データに溜まる。メール本文という外に出したくないデータを**ローカルで処理**できるのが効いている（モデルの使い分けは第8章）。
* **縦串と出典**：プロジェクトの紐づけは `project_links`、メール→タスクの出典は `artifact_links`、どちらも **`(type, id)` のポリモーフィックな M:N 中間テーブル**で統一。何と何でもつなげる。
* **連絡先メモ**：`append_contact_note` ツールが `notes`（Markdown）末尾に `## 2026-05-26` の日付ブロックを追記し、`last_contact_at` を更新。並べ替えれば「最近やりとりした人」が上に来る。

> 公開にあたり、メールアドレス・電話番号・取引先名などの実データはすべてモックに差し替えています。

## おわりに

これで、結衣は「**道具を持った**秘書」になりました。頼めば、適切なツールを選び、足りない情報は聞き返し、複数の登録に分解して実行してくれる。Claude Code でやっていた秘書業務が、常駐・声・アバター付きで、自分のデータの上に乗りました。

でも、ここまでの結衣には、まだ大きな穴があります。**毎回、はじめまして**なんです。昨日の話も、先週の約束も覚えていない。道具は使えるのに、文脈を覚えていない秘書です。

次回は、そこに手を入れます。\*\*「記憶」\*\*の話。Claude や他の AI エージェントが記憶をどう扱っているかを一通り調べた上で、既存の解をあえて使わず自前で設計した記録を書きます。
