45個。

東大院生がClaude Codeを使って日常タスクをcronジョブで45個自動化した記事を公開していて、1000いいね以上を集めています。メール処理・日程調整から論文の新着監視、MLコード開発、システム自己監視まで、設計原則まで全部公開しているものです。

こんにちは、さかしーです。

この投稿への返信コメントに、妙に納得する一言がありました。自動化って『作ること』より『何を任せて何を手元に残すか』の判断軸が先にある気がして、そこが曖昧なままcron増やしても管理コストばかり増える、というものです。

自分でもClaude Codeを使った自動化パイプラインをいくつか組んでいるので、今回はこの話をしてみます。

## 「何を自動化するか」より「何を残すか」

僕がClaude Codeで最初に組んだのは、note記事の下書き生成パイプラインです。Xのバズった投稿をネタに拾って、リファレンス分析まで全部自動でやる仕組みです。

最初は公開まで全自動にしようとしていたんですが、途中でやめています。理由は、公開する前に自分が一度読んだ確認を挟みたかったからです。自動化が暴走したときの取り返しつかなさを考えると、判断が必要な部分だけは手元に残しておく方がいいという結論になりました。

45個という数字は確かにすごいですが、それより何を残したかの方が気になりますね。

## cronで放っておいても回る状態を作る

cronジョブは、定時に処理を自動で走らせる仕組みです。エンジニア以外にはなじみが薄い言葉ですが、Claude Codeに頼むとエンジニアじゃなくても簡単に組めます。

僕が試したのは、毎朝Xのトレンドを収集して候補ツイートを選ぶ処理です。以前は手作業でやっていたのが、今は起きたらDiscordに候補が届いている状態になっています。

起動するだけでゼロから作れるという感覚のまま放置すると、エラーが出たときに何もできません。どういう処理が動いているか自分が理解していることが大切だと思っています。

もう少しシステムに任せても大丈夫という確信が得られたら、完全に自動で回してもいいかなと考えています。

## Claude Codeに任せて良かった4カテゴリ

自分なりに整理すると、こういう分類です。

* 繰り返しの情報収集（APIで取得してファイルやSlackに吐き出すもの）
* テキストの変換・整形（マークダウンへの変換、フォーマット揃え、要約）
* スキャンした資料の分類作業（レシート、イベントの資料など）
* 通知トリガー（条件を満たしたらDiscordやSlackに飛ばすもの）

逆に任せなかったものは、記事の最終確認と公開ボタン、費用が発生するAPI呼び出し量の上限見直し、読者のコメントへの返信あたりです。

自動化は確かに便利なんですが、ガードレールなしで自動化すると月末の請求書でびっくりする羽目になります。実際に僕も一度やりました。

## まず1個だけ動かしてみる

45個を見て自分には無理と感じるようであっても、最初から45個を目指す必要はありません。毎日15分かけている、という作業ひとつだけを探して、まずそこからやってみるとよいでしょう。

Claude Codeで始めるなら、定型文の生成や毎日の同じ作業の情報収集を最初に試してみると、自動化の感触がつかめると思います。動いたとき感動しますよ。
