Claude Projectsで、技術担当・事業担当・学習コーチ担当など、役割の異なる複数のAIエージェントを並行運用しています。現在は8体います。

しばらく運用して分かったのは、「AIに何度も同じ説明をする」という問題は、モデルの性能だけでは解決しないということでした。

セッションが変わるたびに、前回どこまで進めたか、何を決めたか、何を保留にしたかが消えます。エージェントが増えるほど、この問題も大きくなりました。

その結果作ったのが、Agent Memory Server（AMS）というセルフホスト型のメモリサーバーです。今回、OSSとして公開しました。

<https://github.com/kscscafe/agent-memory-server>

この記事では、機能の一覧ではなく、「何がうまくいかず、なぜこの設計になったのか」を中心に書きます。

## AIエージェントを増やしたら何が壊れたか

最初は一体のエージェントに会話履歴を読ませれば足りていました。

ところが、役割の異なるエージェントを並行して使うようになると、いくつかの問題が起きました。

### 同じ説明を繰り返す

技術担当に話した内容を、事業担当は知りません。

それ自体は当然ですが、セッションをまたぐと同じ担当であっても前回の文脈が切れます。

* 前回どこまで作業したか
* 以前どのような設計判断をしたか
* そのエージェントは何を担当しているか
* 何が完了し、何が未完了か
* 触ってはいけない設定は何か

これらを毎回人間が説明していては、AIを増やすほど引き継ぎの負担も増えてしまいます。

### 別の担当領域へ書き込んでしまう

複数のエージェントが同じメモリ基盤を使うと、あるエージェントが別のエージェントの記憶を上書きする可能性があります。

例えば、技術担当が、事業担当の判断として保存されている記憶を書き換えてしまうようなケースです。

単に記憶を共有するだけでは足りず、「誰がその記憶を所有しているか」も必要になりました。

### エージェントの自発性に依存する

最初は、エージェントへの指示に、

* セッション開始時に記憶を検索する
* 確定した情報は記憶へ保存する

と書けばよいと考えていました。

しかし、実際には安定しませんでした。

会話が長くなったり、別の指示が増えたりすると、エージェントがメモリを見ないことがあります。保存すべき確定事項が出ても、保存ツールを呼ばない場合があります。

プロンプトへルールを書くだけでは、毎回確実に実行されるとは限りませんでした。

## 「ユーザーを覚えるメモリ」ではなかった

AI向けのメモリシステムを調べると、多くは`user_id`や`session_id`を中心に設計されています。

例えば、次のような情報を保存します。

* このユーザーは朝の会議を好む
* このユーザーは特定の食べ物が好き
* 前回の問い合わせ内容は何だったか
* このチャットセッションで何が話されたか

これは、複数の利用者へAIサービスを提供する場合に必要なメモリです。

一方、私が必要としていたのは、利用者を分離するためのメモリではありませんでした。利用者は私一人です。その代わり、複数のAIに異なる担当があります。

```
一人の人間
├── 技術支援を担当するAI
├── 事業や経営を担当するAI
├── 学習管理を担当するAI
├── 特定プロジェクトを担当するAI
└── 全体の優先順位を扱うAI
```

必要だったのは、それぞれのAIが自分の担当、判断、作業履歴を継続して持てる仕組みでした。

言い換えると、

> ユーザーを覚えるAIメモリではなく、仕事を引き継げるAIチームの記憶基盤

です。

## 一時的なサブエージェントとも違う

「マルチエージェント」という言葉は、複数の意味で使われています。

Codexなどでは、大きなタスクを複数のサブタスクへ分解し、複数のエージェントが並行して処理できます。この方式では、管理役がその場で作業者を編成し、タスクが終わればチームも解散します。

AMSが対象にしているのは、それとは異なるマルチエージェントです。

```
タスク分解型
  タスク発生 → 一時的に作業者を編成 → 完了 → 解散

長期ペルソナ型
  担当を持つ → 複数セッションで継続 → 判断と経験を蓄積
```

AMSの中心は後者です。

技術担当は過去の技術判断を覚えている。事業担当は以前の市場検討や方針を覚えている。学習担当は前回までの進捗を覚えている。

AIを単発の回答装置ではなく、担当を持つ継続的なチームとして使うための設計です。

## AMSの基本構造

AMSはFastAPIで動作し、記憶はSQLiteへ保存します。

検索にはFTS5と`sqlite-vec`を使い、全文検索とベクトル検索の結果をReciprocal Rank Fusionで統合します。

記憶は大きく3種類に分けています。

### Semantic memory

プロジェクトの状態、確定した事実、設計判断、担当など、後から参照したい知識です。

```
{
  "key": "project_api_policy",
  "value": "外部APIの失敗時は再試行を3回まで行う",
  "category": "design_decision",
  "agent": "engineer"
}
```

### Procedural memory

エージェントが作業するときに守る手順やルールです。

「コードを変更する前に既存構造を確認する」「作業終了時に確定事項を保存する」といった、行動に関する記憶を扱います。

### Episodic memory

セッションや出来事の記録です。

何を行い、どこまで進み、どのような問題が起きたかを残します。次のセッションでは、これを使って前回の続きから作業を再開します。

## 記憶には所有者が必要だった

複数のエージェントが同じデータベースを使うと、別の担当の記憶を誤って上書きする可能性があります。

そこでAMSでは、semantic memoryに`owner`を持たせました。

すでに所有者が設定されているキーを別のエージェントが更新しようとすると、HTTP 403で拒否します。

```
engineer が保存した記憶
        ↓
owner = engineer
        ↓
business が同じキーを更新
        ↓
403 Forbidden
```

これは高度なRBACではありません。個人が運用する小規模なAIチーム向けの、単純な越境防止です。

現在、REST／MCPから直接保存した記憶と、手動承認された記憶ではowner gateが働きます。

一方、スケジューラーの自動承認経路で保存された記憶は、現在`owner=NULL`になります。この経路ではowner gateが働きません。

これは既知の制限としてREADMEとIssueに明記しており、次のバージョンで統一する予定です。

<https://github.com/kscscafe/agent-memory-server/issues/1>

## AIの自発性だけに依存しない

当初は、「エージェントがセッション開始時にメモリサーバーを見に行く」という設計でした。

しかし、エージェントが見に行くのを忘れたり、その時点では不要だと判断してスキップしたりすることがありました。

そこで、記憶を保存・検索するAPIだけでなく、各エージェントの文脈を生成し、セッション開始時に渡す仕組みを作りました。

私の運用環境では、スケジュールジョブが各エージェント用の`agent_context.md`を生成し、Claude Projectsへ配信します。

目標は「AIが気を利かせて覚えてくれること」ではありません。

> 覚えるタイミングと想起するタイミングを、できるだけシステム側で決定的にする

ことです。

## 古い記憶をどう扱うか

記憶は増え続けます。

古い判断や、すでに完了したタスクをそのまま残すと、検索結果のノイズになります。さらに、古い前提が現在も有効な情報として参照される可能性があります。

初期のAMSには、semantic memoryを物理削除するMCPツールがありませんでした。「陳腐化した」という印を本文へ書き込むだけで、削除できない状態でした。

OSSとして公開する準備の中で、この問題に気づき、`delete_memory`を追加しました。

あわせて、陳腐化を示すマーカーが付いた記憶を抽出し、人間が確認してから削除する運用にしています。

長期記憶では、保存方法だけでなく、訂正・削除・棚卸しも必要でした。

## 保存前に候補を確認できる

AIによる記憶では、保存漏れだけでなく、誤った記憶の混入も問題になります。

一度誤った前提が保存されると、その後のセッションでも繰り返し参照される可能性があります。長期記憶は便利ですが、誤りまで長期化させます。

AMSには、直接保存する`save_memory`とは別に、候補として保存する`save_candidate`があります。候補はpending queueへ入り、承認または却下できます。

現在はカテゴリに基づく自動承認経路もあり、Codexから提案された候補は手動レビューの対象になります。

すべてを自動保存するのでも、すべてを人間が手入力するのでもなく、情報源や用途に応じて保存経路を分けています。

## なぜSQLiteなのか

AMSは、大人数が利用するSaaS向けの基盤ではありません。

一人の人間が、自分のMacや小規模なサーバーで、自分のAIチームを動かすことを想定しています。

そのため、外部のRedisやベクトルデータベースを必須にしませんでした。

* 一つのSQLiteファイルとして所有できる
* バックアップしやすい
* 中身を直接確認できる
* 外部サービスへ記憶を預けずに済む
* 個人運用に必要な規模では十分に機能する

スケールよりも、所有できることと運用の単純さを優先しています。

## MCPとRESTの両方を用意した

AMSはREST APIとMCPの両方を公開します。

REST APIは、スケジューラーや外部連携など、決定的に動かしたい処理から利用します。

MCPはClaude DesktopやClaude Codeなど、MCPに対応したクライアントから記憶を検索・保存するために使います。

主なMCPツールは次のとおりです。

* `search_memory`
* `get_memory`
* `list_memories`
* `get_context`
* `save_memory`
* `save_candidate`
* `list_pending_candidates`
* `delete_memory`
* `inbox_add`

記憶を特定のチャットサービス内部へ閉じ込めず、外部の共通レイヤーとして扱うための構成です。

## Redis Agent Memory Serverとは同名だが、対象が違う

公開準備を進める中で、Redisにも同名のAgent Memory Serverがあることを知りました。

Redis版は、ユーザーやセッションを軸に、AIアプリケーションへメモリ機能を提供するプロダクション向けの基盤です。

このAMSは、エージェントのペルソナを軸に、一人の人間が複数の専門AIを継続運用するための基盤です。

```
Redis Agent Memory Server
  多数の利用者 × AIアプリケーション

このAMS
  一人の所有者 × 複数の長期AIペルソナ
```

同じ「メモリサーバー」でも、分離する単位と解決する問題が異なります。

Redis版を小さく作り直したものでも、代替しようとするものでもありません。

## 導入方法

リポジトリをcloneし、Pythonの仮想環境を作成します。

```
git clone https://github.com/kscscafe/agent-memory-server.git
cd agent-memory-server

python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
```

`.env`へ最低限必要な値を設定し、REST APIを起動します。

```
uvicorn main:app --host 127.0.0.1 --port 8000
```

MCPサーバーは別プロセスで起動します。

詳細な環境変数、認証、Claudeへの登録方法はREADMEを参照してください。

<https://github.com/kscscafe/agent-memory-server>

## 現在地

AMSは、私自身のAIエージェント運用から生まれたソフトウェアです。

2026年7月23日に、最初の公開版`v0.1.0`をリリースしました。

<https://github.com/kscscafe/agent-memory-server/releases/tag/v0.1.0>

現時点で、次の機能があります。

* semantic／procedural／episodic memory
* 全文検索とベクトル検索を組み合わせた検索
* エージェント単位のowner gate
* 記憶候補と承認フロー
* セッション開始時のコンテキスト取得
* 記憶の削除と棚卸し
* MCP／REST API
* LINE／Slack／Notion／Google Driveとの任意連携
* スケジューラーと自動実行処理

一方で、誰でもすぐ導入できる完成品になったとは考えていません。

現在は、私の環境で実際に動いているものを、他の人が読んで試せる形へ整理した段階です。

今後は、セットアップ手順の改善、自動承認経路におけるowner gateの統一、実際の利用者から得た問題の修正を進めます。

## おわりに

AIエージェントを増やせば、そのままAIチームになるわけではありません。

チームとして継続させるには、少なくとも次のものが必要でした。

* 担当
* 記憶
* 所有権
* 引き継ぎ
* 人間による確認と訂正

AMSは、これらを一つの小さなセルフホスト型サーバーへまとめたものです。

「一人で複数のAIを継続運用したい」「毎回同じ説明をするところから抜けたい」という人に、設計やコードが何かの参考になればと思います。

GitHubで公開しています。

<https://github.com/kscscafe/agent-memory-server>
