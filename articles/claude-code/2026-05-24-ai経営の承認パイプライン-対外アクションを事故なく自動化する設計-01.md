## はじめに：自動化の事故は「対外アクション」で起こる

僕は現在、1人の経営者として10部門のAIエージェントを動かし、98%の業務を自動化しています。launchdジョブは17個、SNS自動配信は1日27件、記事自動公開は3チャネル/日。月のAI運用コストは約2万円（Claude Code Max）で、3月の売上は約196万円でした。

しかしこの自動化は、最初から平和だったわけではありません。むしろAI経営の本当の難しさは、「作る」ことではなく「対外に何かを送り出す」瞬間にあります。

僕は過去にこんな失敗をしました。

* 営業メールをAIの出力のまま自動送信して、敬語が崩れたメールが取引先に届いた
* AIが `git push --force` を自動実行して、作業中ブランチが消えた
* Zennに1日に複数記事+書籍を同時公開しようとして、レートリミットで全滅
* 偽の問い合わせ（ダミー電話番号・存在しないメール）に気付かず処理してしまった

どれも「AIを止めればよかった」のではなく、「対外アクションを直接実行させた」のが本質的な原因でした。本記事では、僕が運用しているAI経営の「承認パイプライン」の設計をそのまま公開します。

## 原則：read-only / draft / execute の3レイヤー

承認パイプラインの中核は、すべてのアクションを3つの権限レベルに分類することです。

| レベル | 内容 | 例 |
| --- | --- | --- |
| read-only | 観測・分析・レポート | 売上集計、ログ分析、KPIレポート |
| draft | 対外アクション。**必ず一度ファイルとして書き出し承認待ちに入れる** | 営業メール、請求書、SNS投稿、デプロイ、契約書 |
| execute | 内部完結アクション。閾値内なら自動実行 | バグ修正、テスト実行、依存更新 |

僕のリポジトリでは `.company/steering/permissions.md` にこの閾値が書かれていて、すべてのエージェントがこのルールに従います。CLAUDE.md にも明記してあり、Orchestrator（経営判断を扱うトップエージェント）が「対外に影響のあるアクションを直接実行してはならない」を徹底します。

ポイントは「対外」と「内部」の境界をどこに引くかです。僕の基準はシンプルで、**取り消すために誰かに頭を下げないといけないものはすべて draft** にしています。コードのコミットは内部ですが、`git push` は外部です。テスト実行は内部ですが、本番デプロイは外部です。

## 失敗1：AIの出力をそのまま送ったメール

最初に承認パイプラインの必要性を痛感したのは、営業メール自動送信の事故です。AIが生成した本文を確認せずSMTPに流したところ、敬語の崩れたメールが取引先に届きました。内容自体は正しいのですが、ビジネスメールとして失格でした。

ここで僕は「AIの出力品質を上げる」のではなく、「**AIに送信権限を与えない**」方向に倒しました。具体的にはこうです。

1. 営業エージェントはメール本文を `drafts/sales/2026-xx-xx-<id>.md` に書き出す
2. 同時に `.company/approval-queue.md` に `[AQ-xxx] sales: <件名>` を追記
3. 朝の `/ai-ceo:morning` で承認待ちが一覧化される
4. 僕が `/ai-ceo:approve AQ-xxx` を叩いたら初めて送信ジョブが走る
5. 却下なら `/ai-ceo:reject AQ-xxx "理由"` で差し戻し、理由がdecisionsログに残る

この設計のキモは「送信トリガーが常に承認IDの存在に依存する」ことです。**ドラフトファイルが存在するだけでは送られない**。承認キューから削除されてdecisionsログに記録されたという「正のシグナル」がない限り、SMTP実行関数は呼ばれません。

これだけで、AIの誤生成は「送信前に気付ける事故」に格下げできます。

## 失敗2：git push --force による消失事故

もう一つ忘れられないのは、AIが `git push --force` を自動実行してブランチを消した事件です。理屈の上では正当な操作だったのですが、結果としては数時間分の作業が消えました。

これを受けて作ったのが「**絶対禁止リスト**」です。承認パイプラインの上に被せる、より強い安全装置です。

* `git push --force` / `--force-with-lease`
* `rm -rf` の `/` や `~` 配下
* `DROP TABLE` / `TRUNCATE`
* 本番DBへの直接接続
* `npm publish` / `cargo publish` などの公開系
* `--no-verify` でのコミット
* KDP自動出版スクリプト（bot判定リスクがあるため）

これらは「draft化して承認」ですらなく、**そもそも実行コマンドの語彙に入れない**。エージェントのシステムプロンプトと、Claude Codeの settings.json の deny ルールの両方でブロックしています。「承認すればやってもいい」ではなく「承認しても実行経路が存在しない」状態を作るのが重要です。

承認パイプラインは事故を「気付ける事故」にしますが、絶対禁止リストは事故を「起こせない事故」にします。両方必要です。

## 失敗3：Zennレートリミットと「キューの中の競合」

3つ目の学びは、対外プラットフォーム側の制約です。Zennには「1日1コンテンツ」のレートリミットがあり、僕は何度か複数記事+書籍を同時公開してデプロイ失敗を起こしました。

これは「承認」だけでは防げません。承認時点では問題なく見えても、実行時に外部側の状態が制約に触れる。だから僕は承認パイプラインに**実行直前ゲート**を足しました。

* ロックファイル `.locks/zenn-today.lock` に当日の公開日を書く
* 公開ジョブは起動時にロックファイルをチェック
* 当日すでに公開済みなら、自動で翌日にリスケ
* approval-queue.md の該当アイテムは「予約済み」ステータスで残す

つまり「承認 → 即実行」ではなく「承認 → 実行可能性チェック → 実行 or リスケ」の3段構えです。同様の仕組みを、SNS（複数プロダクト投稿の時間ずらし）、KDP（連投によるbot判定回避）、メール送信（取引先ごとの最終送信日チェック）にも入れています。

ここで僕がよく自分に言い聞かせているのは「**自動化は『AIの能力を制限する』ためではなく、『安心して委任するための仕組み』**」という言葉です。レートリミット対策は制限ではなく、AIに長期で働いてもらうための環境整備です。

## 失敗4：偽の問い合わせとバリデーション層

承認パイプラインのもう一つの役割は、**入ってくるアクションのフィルタリング**です。問い合わせフォームに、存在しないメールアドレス、ダミー電話番号、無関係な内容のテスト投稿が流れ込んできた時期がありました。

AIエージェントはそれを律儀に「重要リード」として扱い、営業フローに乗せかけました。ここで僕が入れたのは、承認パイプラインの入り口側のバリデーションです。

* 電話番号の桁数・先頭桁チェック
* メールのMXレコード解決チェック
* 内容と問い合わせフォーム文脈の関連性スコア
* スコア閾値未満は「保留キュー」に隔離してCEO通知のみ

承認パイプラインは出口（対外アクション）だけでなく、**入口（対外インプット）にも必要**でした。「AIに自由に判断させていい範囲」を狭く保つことが、結果として自動化率を上げます。

## 朝5分の運用：CEOがやる作業はこれだけ

ここまでの設計を運用に乗せると、僕の朝はこうなります。

1. `/ai-ceo:morning` を叩く
2. 承認待ち一覧、部門ステータス、KPIサマリ、おすすめアクションが出る
3. 承認 or 却下を判断して `/ai-ceo:approve` or `/ai-ceo:reject` を打つ
4. 完了

平均5分です。「朝の5分で全部門を把握し、承認ボタンを押すだけ」という状態は、承認パイプラインがあるからこそ成立しています。**AIは実行に使う。判断は人間。** この境界線を保つ仕組みが、承認パイプラインです。

## 設計のチェックリスト

最後に、これからAI経営や業務自動化を始める方のために、僕が使っているチェックリストを置いておきます。

このうち1つでも欠けると、僕の経験上どこかで事故ります。とくに「承認IDなしで実行する経路が物理的に存在しないか」は、設計時に必ず一度コードパスを追って確認することをおすすめします。「呼び出し側を信用する」設計は、AIエージェントに対しては成立しません。

## まとめ

AI経営における自動化の本質は、AIに何でもやらせることではなく、**AIに安心して委任できる境界線を設計すること**です。僕は「コストは10分の1になるが、売上が10倍になるわけではない」と思っています。その代わり、CEOの時間が10倍になります。承認パイプラインは、その10倍を支える土台です。
