## はじめに

私は10年ほどシステムエンジニアをしています。割と多趣味な人間をしています。趣味が多いのに遊びを本気でやる性なので、お金が貯まらないタイプです。  
今回は長年の付き合いで、あるゲームの運営ツールを開発することとなり、LLMを使った開発の勉強を兼ねてプロジェクトを立ち上げました。  
この記事はその際に作った開発環境を魔改造した話となっています。

# Part 1: What — 何をしたか

## multi-agent-shogunとは

[multi-agent-shogun](https://github.com/yohey-w/multi-agent-shogun)は、Claude Code + tmuxで複数のAIエージェントを並列運用するフレームワークだ。戦国時代の軍制をモチーフにした階層構造を持っている。

```
Lord（人間） → Shogun（戦略） → Karo（戦術） → Ashigaru（実行） / Gunshi（分析・QC）
```

Lordが方針を決め、Shogunが指令を発行し、Karoがタスクに分解してAshigaruに割り当てる。Gunshiが品質チェックを担当する。通信はYAMLファイルベースのメールボックスで、ポーリングではなくイベント駆動。エスカレーション機構もある。

この基本的な階層構造、通信基盤、品質チェックフロー、バッチ処理プロトコル、破壊的操作の禁止ルール群――これらは全て本家の設計だ。

## シンデレラの城への改装

私はこのフレームワークをフォークして、[アイドルマスター シンデレラガールズ](https://cinderellagirls.idolmaster-official.jp/)のキャラクターをエージェントに割り当てた。

元々のフレームワークでは全エージェントが「戦国風」の共通口調で話す。「はっ！」「承知つかまつった」「任務完了でござる」。機能的には問題ないけれど、自分のプロジェクトに使い続けるなら、もう少し手に馴染むものにしたかった。

フォークのきっかけは運用上の限界だった。ペルソナ設定やスクリプトのカスタマイズが増え、未コミット＋stash/popの運用では管理しきれなくなった。private forkに移行して、カスタマイズをコミット管理に載せた。

以下が主な変更点だ。

| 領域 | 本家 | フォーク後 | なぜそうしたか | 区分 |
| --- | --- | --- | --- | --- |
| ペルソナ | 戦国風口調（全員共通） | キャラ個別の口調・性格 | そのほうが面白そうだったから | 独自追加 |
| AI間通信 | 戦国風（同じ口調） | ロッキーペルソナ（簡潔3行） | キャラ口調はトークンを食う。コスト削減 | 独自追加 |
| QC学習 | Gunshi担当のQCフロー | + 学習ルール蓄積サイクル | 同じミスを二度踏みたくない | 独自追加 |
| 安全ルール | 破壊的操作禁止8項目 + 組織ルール6項目 | + フォーク固有ルール追加, 独立ファイル化 | upstream誤pushなどフォーク固有のリスク対策 | 拡張 |
| 通信基盤 | inbox\_write + inbox\_watcher | inbox\_watcher安定化改修 | WSL2実運用で出た不具合の解消 | 拡張 |
| スケジュール実行 | なし | scheduled\_launcher.sh | 定型タスクを人手から外す | 独自追加 |
| ダッシュボードGist同期 | なし | sync\_dashboard\_gist.sh | 外出先から進捗を確認したい | 独自追加 |
| 整合性フレームワーク | なし | game\_rules.yaml + 依存マップ | 暗黙の依存で不整合が起きた経験から | 独自追加 |
| Production Portal | なし | FastAPI + Vue 3 PWA | ターミナルに張りつかず状況把握したい | 独自追加 |
| 職能分業 | 汎用Ashigaru | BE/FE/ライター/統合 | タスクの性質に合ったメンバーに振りたい | 独自追加 |

指揮系統、通信の基本設計、エスカレーション、バッチ処理プロトコル、ダッシュボード――これらは本家の設計をそのまま使っている。

# Part 2: Why — なぜそうしたか

## フレームワークを「自分のもの」にする

OSSのフレームワークをそのまま使う選択肢はあった。戦国風口調でも開発はできる。でも毎日何時間も一緒に作業する相手なら、自分にとって意味のあるキャラクターを割り当てたかった。

構造は同じまま、住人だけ変える。**将軍の城をシンデレラの城に改装した。**

## AIに人格を与えると何が変わるか

「マキノにレビューで指摘された」と「QCでfailした」は同じ事象でも受け取り方が違う。キャラの名前がつくと、品質チェックに物語が生まれる。比奈がデザインを書いてマキノがレビューする。芳乃が方針を決めて、泉がタスクを分解する。

作業ログを読み返すとき、キャラの口調があると文脈が追いやすい。誰が何をしたかが声で分かる。

ただし、ここには注意もある。キャラ口調はトークンを消費する。AI間通信まで丁寧な口調で話すとコストが膨らむ。そこで「ロッキー」という別のペルソナを定義した。直球・簡潔・3行以内。AI間のメッセージはロッキーの口調で書く。人間が読む場所だけキャラ口調を使う。「ペルソナ装飾なし」ではなく「簡潔さという独立したペルソナ」として定義しているのがポイントだ。ペルソナならそこにニュアンスを乗せることができる。

## AI自律型から人間主導型へ

フォークの過程で、運用の哲学も変わった。

元のフレームワークは**AI自律的な設計**だ。自分のフォークでは**人間主導**の色を強くした。具体的にはスクリプト移譲――定型処理はシェルスクリプトに落として、LLMにはLLMにしかできない判断と生成だけやらせる。YAML手編集やステータス更新のような定型作業をスクリプト化するとトークン節約にもなるし、手順漏れも防げる。

**無限のAPIクレジットがあるわけではない。**  
趣味プロジェクトでマルチエージェントを回すなら、コスト意識はアーキテクチャに組み込む必要がある。

## 「遊びだから本気でやる」

エイプリルフールの日に、話題に応じてキャラクターを自動選択するエージェント（my-idol-agent）の設計を始めた。夜の12時間で設計書作成→敵対的レビュー→指摘修正→コンセプト追加まで完走した。

趣味プロジェクトだからこそ回せるプロセスがある。納期の縛りがないから、本気の設計レビューをやっても誰にも迷惑がかからない。遊びだから本気でやる。

このプロジェクトについては別の記事で詳しく書く予定だ。

# Part 3: How — どうやったか

## ペルソナシステムの3層設計

Claude Codeには「コンテキストリセット」という仕組みがある。コンテキストウィンドウが溢れそうになると過去の会話が圧縮される。`/clear`コマンドで意図的にリセットすることもある。このとき、ペルソナ情報が失われると口調が元に戻ってしまう。

対策としてペルソナの読み込みを３層構造とした。

1. **CLAUDE.md** — プロジェクトルートに置く設定ファイル。Claude Codeが自動読み込みする。最低限のキャラ名と口調パターンを記載
2. **.claude/rules/persona-core.md** — ペルソナ維持ルールの詳細。禁止パターン、復元手順
3. **config/personas.yaml** — 870行のキャラ定義。口調サンプル、禁止表現、呼称ルール

CLAUDE.mdは常に読み込まれるので、リセット後も最低限の口調は復元される。詳細が必要な場面ではpersona-core.mdを読み、完全な定義が要ればpersonas.yamlを参照する。

本家は全員「戦国風」の一律設定なので、そもそもペルソナ復元の問題が起きにくい。キャラを個別化したから必要になった設計だ。

## QCフライホイール

本家にはGunshiによる品質チェックフローがある。Ashigaruの成果物をGunshiがレビューし、PASS/FAILを判定する。この基本構造はそのまま使っている。

独自に追加したのは「フライホイール」だ。

品質チェックで新しい違反パターンが見つかると、学習ルールのYAMLファイルに追記される。次回のチェックではこのYAMLがチェックリストとして使われる。頻出パターンはAshigaruの指示書に昇格する。

```
品質チェックで検出 → YAML蓄積 → チェックリスト化 → 指示書に昇格 → 同じミスが起きなくなる
```

実例を挙げると、axiosのbaseURL二重プレフィックス、`git add -A`による未追跡ファイル混入、TypeScriptビルドチェック未実行――これらは一度検出されて以降、ルール化されて再発していない。

## 安全ルールの独立ファイル化

本家には破壊的操作の禁止ルール（`rm -rf`やforce push等を禁じる8項目）と、組織プロトコルのルール（自己実行禁止、直接報告禁止等の6項目）がある。CLAUDE.md内に書かれている。

フォークではこれらを独立ファイルに分離した。理由は2つ。

1つ目は参照性。CLAUDE.mdは長い設定ファイルなので、安全ルールだけ独立したファイルにした方が見通しが良い。

2つ目はフォーク固有のルール追加。「upstream push禁止」を追加した。private forkで作業しているので、`git push origin`は許可するが`git push upstream`は絶対禁止。元のOSSリポジトリに自分のカスタマイズを誤ってpushしないための安全弁だ。

WSL2環境固有のルール（`/mnt/c/`配下の破壊的操作禁止）も追加している。

## イベント駆動通信

通信基盤は本家の設計だ。YAMLファイルにメッセージを書き込み、inotifywaitでファイル変更を検知してエージェントに通知する。ポーリングはしない。

エスカレーション機構も本家由来だ。メッセージが2分放置されたらEscapeキー送信、4分放置されたら`/clear`で強制リセット。

フォーク側で行ったのは通信監視スクリプトの安定化改修だ。WSL2環境でのinotifyの挙動やアトミック書き込み時のinode変更への対応など、実運用で出た問題を潰している。

## 暗黙依存の明示化

ペルソナ情報は3箇所で定義される。CLAUDE.md、persona-core.md、personas.yaml。ペルソナを変更するときは3ファイルを同時に更新しないと不整合が起きる。

こうした暗黙の依存関係を依存マップとして明文化した。どのファイルがどのファイルを参照しているか、何を変えたらどこに影響するかを一覧にしてある。

自動化ツール（codd-dev）の採用も検討したが、alpha版であることとリポジトリの規模感（依存ノード10-15個程度）を考慮して手動マップを選択した。小さなチームには小さな仕組みが合う。

当然ながらこの仕組みは本家にはない。本家はペルソナが一律なので依存関係が単純だ。キャラを個別化したことで三重定義の問題が生まれ、それを解決するために依存マップが必要になった。

## その他の独自追加

**Production Portal** — FastAPI + Vue 3のPWAで、ダッシュボードをブラウザから確認できるようにした。Tailscale経由で外出先からアクセスできる。本家のダッシュボードファイルをGitHub Gistに同期する仕組みも独自に作った。

**スケジュール実行** — crontab連携で、平日朝8時と休日深夜2時にバッチタスクを自動投入する。トークン消費の少ない時間帯に人のレビューが必要ない作業や重い作業を実行するための仕組みだ。

**職能分業** — 本家のAshigaruは汎用だが、フォークではバックエンド/フロントエンド/ライター/統合に専門分化させた。タスクの性質に合ったエージェントに振れるようになった。

# まとめ

multi-agent-shogunは、指揮系統・通信基盤・品質チェック・安全ルールといったマルチエージェント運用の骨格がよく設計されたフレームワークだ。自分がやったのは、その骨格を残したまま住人を入れ替えて、運用の中で必要になった仕組みを足していったこと。

フレームワークは道具であって、道具は使う人に合わせて手を入れるものだと思っている。構造は同じでも、住人が変われば城の性格は変わる。
