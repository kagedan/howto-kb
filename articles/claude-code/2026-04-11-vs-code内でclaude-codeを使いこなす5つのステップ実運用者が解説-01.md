Claude Code の VS Code 拡張機能、「とりあえず入れてみたけど使いこなせてない」状態になりがちです。チャットパネルが開いて質問に答えてくれるのはわかった、でもそれ以上の使い方がよくわからない——そういうフェーズで止まっている方に向けて、実際に差が出る5つのポイントを整理しました。

対象読者: Claude Code VS Code 拡張機能をインストール済みで、基本的な使い方は把握している方。VS Code 1.98.0 以上が必要です。

## ステップ1: パーミッションモードを使い分ける

ほとんどの方がデフォルトのまま使っていますが、**パーミッションモードの選択が作業効率に直結します**。

プロンプト入力欄の下部にあるモード表示をクリックすると、3つのモードを切り替えられます。

| モード | 動作 | 向いている場面 |
| --- | --- | --- |
| `default` | アクションごとに確認を求める | コードレビューや慎重に進めたい変更 |
| `plan` | 変更前に計画を提示して承認待ち | 影響範囲が広いリファクタリング |
| `acceptEdits` | 編集を自動承認して進める | 信頼できる繰り返し作業 |

`plan` モードが特に便利で、Claude が何をしようとしているかをマークダウンで展開表示してくれます。そこにインラインでコメントを書き込めるので、「この部分だけ変えて」と修正指示を入れてから実行させる、という流れが作れます。

設定でデフォルトを変えておくことも可能です。`Cmd+,` でVS Code設定を開き、`claudeCode.initialPermissionMode` を変更してください。

## ステップ2: @-メンション と Option+K を使いこなす

「ファイル全体を読み込ませる必要があるのかな」と思いながら漠然と質問してしまうのは、初見だと仕方ないですが、もったいない使い方です。

**選択テキストは自動的に Claude に渡されます。** コードを選択した状態でプロンプトを入力すると、プロンプトボックスの下部に「何行選択中」と表示され、その内容が自動的にコンテキストに含まれます。

さらに精度を上げたいときは `Option+K`（Mac）/ `Alt+K`（Windows/Linux）を使います。エディタでコードを選択した状態でこのショートカットを押すと、`@app.ts#5-10` のような形式でファイルパスと行番号が自動挿入されます。

ファイル名やフォルダ名を直接 `@` で参照するときは、ファジーマッチが効くので完全一致でなくても大丈夫です。

```
> @auth を使っている箇所を全部教えて
> @src/components/ の構成を説明して
```

フォルダを指定するときは末尾にスラッシュをつけることがポイントです。

プロンプトボックスの目のアイコンをクリックすると、選択テキストを Claude から隠せます。「選択は維持しつつ Claude には見せたくない」という場面で使えます。

## ステップ3: チェックポイントで安全に試行錯誤する

ここは設計がうまいと思う部分です。

Claude が編集を行うたびに、メッセージにチェックポイントが自動記録されます。メッセージにホバーすると巻き戻しボタンが出てきて、3つの選択肢が表示されます。

* **Fork conversation from here**: コード変更はそのまま、会話だけここから分岐
* **Rewind code to here**: ファイルをこの時点に戻す（会話履歴は残る）
* **Fork conversation and rewind code**: 会話もコードも両方この時点から分岐

「3ファイル変更してもらったけど、最後のファイルだけ元に戻したい」というときに Rewind を使うと一発です。Git の `git stash` や手動での差し戻しが不要になります。

ただし、チェックポイントはあくまで VS Code 内のファイル変更のトラッキングです。データベースへの書き込みや外部 API へのリクエストは巻き戻せません。その点は注意です。

## ステップ4: 複数タブで並行作業する

Claude Code をタブとして複数開けることは、意外と知られていないポイントです。

`Cmd+Shift+Esc`（Mac）/ `Ctrl+Shift+Esc`（Windows/Linux）で新しいタブを開くか、コマンドパレットから "Open in New Tab" を選ぶだけです。各タブが独立した会話とコンテキストを持ちます。

実際の使い方としては、

* メインタブ: 機能実装の会話
* サブタブ: テストコードの生成
* 別ウィンドウ: バグ調査の会話

という分け方が現実的です。タブが裏に隠れているときに Claude が応答を終えると、スパークアイコンにオレンジの点が表示されます。パーミッション待ちのときは青い点です。

ドラッグでパネルを右サイドバーに配置しつつ、追加の会話はタブで開く——この組み合わせがデュアルモニターなしでの最大活用に近い構成です。

## ステップ5: [@browser](/browser "browser") でブラウザ操作を自動化する

この機能は目立たないですが、フロントエンド開発をしている方には刺さります。

Chrome に [Claude in Chrome 拡張機能](https://chromewebstore.google.com/detail/claude/fcoeoabgfenejglbffodgkkbkcdhcgfn)（バージョン 1.0.36 以上）を入れると、VS Code から Chrome を直接操作できるようになります。

```
@browser go to localhost:3000 and check the console for errors
@browser fill in the login form with test@example.com and submit
```

Claude はブラウザのログイン状態を共有しているので、認証が必要なページもそのまま操作できます。ローカル開発サーバーのコンソールエラーを確認してもらって、その場でコードを修正する——という一連の流れを VS Code から出ずに完結できます。

正直、これは使いどころが限られますが、手動での確認作業が多い開発フェーズだと効果を感じやすいです。

## 設定として押さえておくべきもの

5つのステップとは別に、最初に確認しておくと地味に助かる設定があります。

`~/.claude/settings.json` に以下を追加すると、VS Code 上で設定のオートコンプリートが効くようになります。

```
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json"
}
```

設定の項目が多くてどれを変えればいいかわからないときに、インライン補完と説明が出るのでかなり助かります。

---

5つのステップを全部一度に試す必要はないです。まず `plan` モードと `Option+K` だけでも意識的に使ってみると、「ただ質問するだけ」の使い方から一段上がった実感が得られると思います。
