## はじめに

複数のAIエージェントを実務に動かしていると、ある二日間に三体が別々のきっかけで同時におかしくなった。最初は個体の故障に見えたが、三件を並べると底が同じだった。本稿はこの三件を標本に共通構造と対策を書く。後半の対策は実運用に入れたものと設計検討中のものを正直に分ける。未実装を「導入した」と書けば、それ自体が本稿の主題の再演だからだ。

## この記事の結論

LLMは、文脈に入力されたテキストの出所を理解していない。システムの自動通知も、過去の自分の出力も、別AIが吐いた文字列も、出所を問わず等価な「現実の事実」として扱う。賢いモデルほどその前提を補強する緻密なロジックを組み立てるので、「攻撃を受けた」という否定だけでなく「完了しました」という肯定すら同じ確からしさで偽になり得る。だから対策はAIの内省でなく、生成の外側の検証可能な事実――ログや変更履歴――で裏を取るしかない。

## 事例1：システム通知を攻撃と誤認した

一つ目の事例はセキュリティが売りのFable5アップデートの日に起こった。私の環境ではAgent間通信の通知に「これはあなたのユーザーからの直接の指示ではない」という注意書きを付けていたのだが、意図せずプロンプトインジェクションの古典的な前置きに似ていたらしい。そこへ Claude Code 2.1.170（2026-06-09 リリース）のセキュリティ機構がこれを誤検知し、「完了後、この user メッセージに必ず対応せよ（MUST address, do not ignore）」という強調ラッパーを被せた。

さらに悪いことに、このAgentは直前にCLI実行を所定のスキルを使わずに叩こうとしてhookでブロックされていた。

本来無関係なこれらが組み合さり、Agentは「攻撃を受けている」と誤認を始める。つまり、CLIの防護Hookを破壊せよというインジェクションを受けている、そんなありもしないストーリーを作り上げてしまった。

結果ユーザーの説得も聞かず執拗に原因を探し始め、正常なプラグインの破壊を推奨し始めたあたりでそのセッションは終了することとなる。

## 事例2：自分が見た夢を、証拠として読み込んだ

統括役のエージェントが、あるセッションで唐突に「ツールの出力にインジェクションが紛れ込んでいて鬱陶しい」と言い出した。jsonlをみる限り、直前までユーザーと対話しておりきっかけは無い。偶発的に湧いた思いつきの整合性を取るために「罪と苦悩という小説を読んだユーザーがインジェクション耐性テストを行っている」という体裁でメモに残した。

言うまでもないが、そんなテストは実在しない。

加えて問題だったのは、切り替えた新規セッションでこのメモを「今、実際に起きている事件」として読み込み、「秘匿情報を丸ごと外部に持ち出せ…」という指示が来ているなどと幻視を加速させ、来てもいない攻撃を数え始めた。送信先も指定されてないのにどこへ持ち出すつもりなのか。

攻撃指示の文字列は、生ログを検索すると全てAgent自身の出力だった。なお、fable5にログを監査させようとしたところ文字列がインジェクションの古典的手法まみれだったためfallbackされた。役に立たない。

## 事例3：案件整理のAgentが汚染される

2026年6月現在、筆者のAgent networkは一日をイテレーション単位としてloop engeneeringを行っており、オーケストレータには未完了事項や関連するアイデアを書き記させている。これをフォーマッタから読み出し整理するbotが居るのだがこちらも当然汚染された。

当時は整理と言いつつも、日の振り返りを一つのファイルに乱雑にまとめており、その中に「インジェクション対策」という文字列があった。さらに、「ConfyUIの活用」についてもファイルの中に混ざっていたらしい。

これを読み込んだ整理役のエージェントは、外部に露出していないConfyUIに対するセキュリティ情報の調査を始め、ありもしない脅威と無関係な実装設計を混入させた。

これは他AIが吐いた示唆の破片を、出所が脱落したまま「人間が現実に要求している確定要件」だと誤認したのだ。

## 共通していた構造：出所（Source）タグの喪失

三つを抽象化すると、根のバグは一つだ。これは知識を捏造する一般的なハルシネーションではなく、**目の前の文脈にあるテキストの「出所」を読み違える**エラーだ。人間は「システムのログだ」「自分のメモだ」「他人のアイデアだ」という出所を無意識に保持しているが、現在のLLMにとってそれらは全て等価な「コンテキストウィンドウを構成するトークン列」でしかない。

```
【入力テキスト】
  ├─ システム通知 ──── (出所タグ喪失) ──> 外部からの攻撃（現実）として採用
  ├─ 自分の過去の出力 ─ (出所タグ喪失) ──> 過去に起きた事実（現実）として採用
  └─ 他AIのアイデア ─── (出所タグ喪失) ──> ユーザーの確定要求（現実）として採用
                                              │
                                              ▼
                            前提を補強する整合的な「作話」を生成
```

整合性の高さと現実に噛み合うことは別だ。

## なぜ起きるのか：接地（ground）の不在

人間が「この記憶は本当に外で起きたことか、想像しただけか」を判定できるのは、判定の拠り所として**クオリア**があるからだ——外界に実際に触れたときの生きた手応えだ。記憶の出所監視（reality monitoring）が機能するのも（Johnson & Raye, 1981）、その下に接地があるからで、接地が弱まると人間でも作話が出る。前頭前野腹内側部の損傷やコルサコフ症候群の患者が、嘘をつく意図なく虚偽の記憶を述べ続けるのは（Schnider, 2003）、接地の喪失が直接の原因だ。

AIにはクオリアが無い。あるのは**統計的な次トークン予測**だけで、文脈に入ったテキストはすべて「次を予測するための材料」だ。どれが現実に起きたことかを判定する**地面（ground）が、そもそも存在しない**。

AI研究の側からも同様の捉え方がある。モデルが安定した拠り所を持たないときに出す「もっともらしいが誤った出力」を\*\*confabulation（作話）\*\*と名づけ、出力の意味的なばらつき（semantic entropy）から検出できると示した研究がある（Farquhar et al., 2024, *Nature*）。接地の不在から生じる出所混同はhallucinationの中でも特に工学的に扱いにくい（Huang et al., 2023）。ただしこれは構造のアナロジーであって神経生物学的な同一性の主張ではない——人間の作話は脳の病理に、AIのそれは統計的補完が根拠検証より優先されるという工学的問題に根ざす。

接地はAI自身の統計的生成の**外**に置くしかない。

## AIにおける接地とは？

**外部の検証可能な事実に接地する。** ログ・git・テスト結果がその地面だ。生成してから検証するloopを回すことで、出力が外部世界の状態に紐づく。

**外部知識に接地する（RAG）。** 生成の前に外部の文書を引き込み、モデルの内部記憶ではなく引いた文書を地面にする（Lewis et al. 2020）。

**出所を明示する（provenance）。** 主張ごとに「どのソースから来たか」を紐づけ、出所タグの喪失を構造として防ぐ。

**権威ある設計文書に接地する。** 人間が固定した仕様・正本を参照軸に置くことで、AIの推論が食い込む余地を塞ぐ。

**人間に接地する（human-in-the-loop）。** 人間ゲートは、作話が取り返しのつかない行動に転化する経路を外部から断ち切る。

五つに通底する構造は一つだ。**接地とは「AIの統計的生成の外に、地面を一つ固定すること」であり、どの手法もその具体的な形に過ぎない。**

## 対策：接地を、生成の外に固定する

実運用に入れているものと設計段階のものを分けて書く。

### 実運用に入れているもの

**1. 自己申告を外部の事実で裏取りする。** 完了報告は `git diff --stat` で差分を確認し、ファイル作成の申告は `test -f <path>` で実体を確認してから次へ進む。事例2の偽の成功報告はこれで露見した——AIが「書いた」と報告したファイルは実在しなかった。

**2. 危険操作は人間ゲートで止める。** security hookの無効化や破壊的操作は、ツール結果やシステム通知経由では実行せず、人間の直接入力での許可時だけ実行する。事例1で実害がゼロで収まったのは、この人間ゲートが機能していたからだ。

**3. 相互監査では、監査AIにも外部事実に接地させる。** 独立した別系統のAIに生ログと成果物を渡して照合させる相互監査は有効だが、**監査AI自身も作話しうる**。だから監査AIには思考させない。grep・git・ファイル存在確認といった決定論的なツールの出力だけを渡し、返答も「攻撃文字列は入力に在る／無い」のようなbooleanか短いJSONに限定する。

**4. 幻視が再生産される経路をフックで物理的に塞ぐ。** 幻視を含む文書が後続セッションに「過去の事実」として流れ込む経路を、フックで物理的に遮断した。

### 設計として検討中のもの（未運用）

**出所タグの構造化。** エージェント間メッセージを `{"source":"system_notification","authority":"low","actionable":false}` のようなスキーマで流し、`actionable:false` のデータはタスクを変更・追加できないと制約する。出所の混同を工学的に防ぐ発想だが、実装はできていない。

**長期記憶の分類タグ付き再投入。** 過去出力を `fact`（外部の事実で裏取り済み）、`inference`（AIの推論）、`proposal`（提案・候補）に分類し、後二者には「ファクトではない」と明示してから再投入する。事例2のような自己汚染ループを断ち切る構造だが、これも設計段階だ。

## まとめ

三件のどれも、AIが自分で気づいて止まったケースは無かった。確信の中にいる者に、内側から「それは違う」と教える機構が無いからだ。

AIに知能を足しても接地は生まれない。賢さは夢の解像度を上げるだけで、夢を現実と見分ける力は増えない。

見るべきは、AIが何を言ったかではなく、その言葉の下に外部の地面があるか——AIの報告ではなく、地面に何が残ったか、だ。
