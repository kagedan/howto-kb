AI へのレビュー依頼が、これまでで一番うまくいった日があった。指摘は鋭く、見当違いがなく、そのまま修正に回せた。数日後、別の変更で同じレビューを頼もうとして、あの指示をどう書いたか思い出せなかった。チャット履歴のどこかにはある。だが、二度と同じレビューは返ってこなかった。この記事は、うまくいったプロンプトを書き捨てにしていた失敗と、プロンプトを成果物としてリポジトリで管理するようにした記録だ。

ここでいうプロンプトは、単発の依頼文だけではない。AI に渡す定型の指示、観点、手順、制約を含む「指示資産」のことだ。

特定のツールを紹介する記事ではない。以下では実例として Claude Code の Skills を使うが、本質は、AI 向けの定型指示をコードと同じ変更管理に乗せることにある。想定読者は、これから生成 AI を開発に本格的に組み込もうとしているエンジニア。個人開発の実体験を主に書くが、末尾でチームに持ち込む場合の注意にも触れる。

## 背景

Windows GUI アプリを AI 主導で開発している個人プロジェクトがある。設計書の作成、実装、レビューの一次確認は AI に任せ、自分は判断と検収に回る。そういう体制を実験しているリポジトリだ。

前回、[設計の決定を会話ではなくリポジトリに置く話](https://zenn.dev/urario/articles/repo-as-memory)を書いた。決めたことは、短いレコードにして残るようになった。ところが、AI への頼み方そのものは、相変わらず毎回その場で書いていた。レビューの依頼文。設計ドキュメントを書かせるときの型の指定。提出前の自己チェックの指示。どれも会話のたびに手で打つ。使っているうちに「この言い回しは効く」という発見が積み上がっていくのだが、その発見が残る場所は、チャット履歴しかなかった。

## 課題：効いたプロンプトを、二度と再現できなかった

冒頭のレビュー依頼の話を少し詳しく書く。観点を並べ、出力の形を指定し、指摘には必ず根拠を付けろという一文を足した版が、一番よく効いた。指摘の質が明らかに変わったので、覚えている。その場で使って、会話を閉じた。

数日後、別の変更のレビューを頼むとき、記憶を頼りに同じ指示を打ち直した。似たものは書けた。だが返ってきた指摘は前回より粗く、根拠の付き方も甘い。おかしいと思って履歴を遡り、原文を探した。似たレビュー依頼が何十件も並んでいて、どれが「あの日の」だったのか判別できない。原文はたぶん見つかった。たぶん、というのは、それを使っても手応えが完全には戻らず、確信が持てなかったからだ。

もう一つ、根の深い方の話がある。以前、AI の設計レビューが 4 往復した話を書いた。あのとき往復のたびに違う観点の指摘が小出しに来たのだが、後から掘り返すと、原因の一端はこちら側にあった。レビュー依頼を毎回手で打っていて、観点の顔ぶれが毎回少しずつ違っていた。境界の観点を入れ忘れた回は、境界の指摘が来ない。当たり前のことだ。レビューの質が日によって違うのを AI の気分のせいにしていたが、実際には、こちらの指示が日によって違った。

これが自分ひとりの開発で起きている。もし隣に同僚がいたら、あの効いた言い回しは、そもそも渡ることすらない。

## なぜ素朴なやり方では解けないのか

「今度からメモしておこう」では解けなかった。実際、一時期は個人のメモアプリに貼っていた。保存先が変わっただけで、埋もれ方は同じだった。検索は効かず、どれが最新版か分からなくなり、プロジェクトの進行につれて中身が古びても誰も直さない。メモは書いた瞬間から死んでいく。

根はもっと単純なところにある。プロンプトを成果物として扱っていなかった。コードなら、リポジトリに入れ、差分をレビューし、挙動が変わったら履歴を疑う。プロンプトは同じ開発工程の入力でありながら、書き捨ての消耗品扱いだった。成果物でないものは、レビューの対象にも、改善の対象にも、再利用の対象にもならない。

そして AI の出力は、指示の細部に敏感だ。指示が変われば出力が変わる。標準の指示がなければ、レビュー品質が変わったときに、対象の違いなのか、モデルの揺らぎなのか、こちらの指示の違いなのかを切り分けられない。人間のレビュアーなら観点は頭の中に残るが、AI には毎回こちらが観点を渡し直す。渡すものが毎回違えば、返ってくるものの比較も難しくなる。

前回の決定レコードは、この問題の半分しか解いていなかった。「何を決めたか」は残るようになった。だが「どう AI を動かすか」は、まだ全部こちらの手打ちに掛かっていた。記憶には二種類あって、片方だけ資産にしても、もう片方が揮発し続ける。

## 打ち手：プロンプトを成果物としてリポジトリに入れる

やったことは前回と同じ発想だ。繰り返し使う AI への定型の指示をファイルにして、コードと同じリポジトリに置いた。レビュー観点のカタログ。設計ドキュメントを書かせるときの型。提出前の自己チェックリスト。どれも普通の Markdown ファイルで、コードと同じようにバージョン管理される。

効かせどころは二つある。

一つ目は、置き場所を「AI が場面に応じて呼び出せる場所」にすることだ。ファイルにしても、毎回手で渡す運用だと、渡し忘れた日から元の手打ちに戻る。いま使っている Claude Code には[スキル](https://code.claude.com/docs/en/skills)という仕組みがあり、リポジトリの `.claude/skills/design-review/SKILL.md` に置いた指示を、AI が必要に応じて参照できる。必須のレビューなら、説明文からの自動選択だけに任せず、明示的にスキルを呼び出す運用にする。この形式は [Agent Skills](https://agentskills.io) という複数の AI ツールで採用されるオープン標準に沿っている。ツール固有の配置や拡張機能には差があるものの、指示本文を別のツールへ持ち出しやすい。レビュー観点のカタログをスキルにしたものが、こういう形だ。

```
.claude/
└── skills/
    └── design-review/
        └── SKILL.md
```

```
---
name: design-review
description: 設計ドキュメントのレビュー。観点カタログを全部一度に当てる。
  「設計をレビューして」で使う。
---

対象の設計ドキュメントに、次の観点をすべて同時に当てる。
指摘には必ず根拠(どの記述とどの記述が食い違うか)を付ける。

## 観点カタログ

1. 整合性: 同じ状態を更新する箇所が複数ないか。用語が途中で変わっていないか
2. 境界: 入力が空・上限ちょうど・タイムアウトのときの挙動が書いてあるか
3. 異常系: 途中で失敗したら巻き戻すのか、途中状態を残すのか
4. 依存方向: UI 層がドメインの内部構造を直接触っていないか

## 出力の形

観点ごとに「指摘 / 根拠 / 提案」の 3 点セット。指摘がない観点は「指摘なし」と書き、確認した範囲も添える。
```

これを入れてから、レビューの頼み方は「設計をレビューして」の一言に近づいた。観点はスキルが運ぶので、少なくとも自分の書き忘れによって観点の顔ぶれが変わることはなくなった。あの「効いた言い回し」は、思い出すものではなく、リポジトリに置いてあるものになった。

二つ目は、バージョン管理に乗せることだ。これはただの保管ではない。観点を一つ足すのも差分になり、履歴に残る。レビューの質が急に変わったと感じたら、コードだけでなくプロンプトの変更履歴も疑える。つまり、プロンプトのバグを修正できるようになる。書き捨てていた頃は、入力条件の何が変わったのかを比べる基準すらなかった。

役割ごと固定したいものは、エージェント定義にしている。前回、ナレッジの手入れ役を定義として見せたのと同じ形式で、レビュー担当を定義し、観点スキルとセットで運用する。定型の指示はスキルに、役割と制約はエージェント定義に、という分担だ。

一つだけ、明文のルールにしたことがある。**リポジトリ内の版を正本とし、個人環境のコピーを直接の配布元にはしない。** 手元で改善した内容は PR として正本へ戻し、合意された版を改めて配る。この順番を決めておかないと、いつの間にか全員が微妙に違う版を使い出して、振り出しに戻る。

前回の言葉で言えば、決定レコードが「何を決めたか」の記憶で、プロンプトの資産化は「どう動かすか」の記憶だ。二つ揃って、会話の揮発に頼る部分を減らせる。

## 残った課題

正直に書くと、増やしすぎた。仕組みが気持ちよくて、思いつくたびにスキルを足していた時期があり、一度も呼ばれないまま古びたものがいくつかできた。数が増えるほど、一つあたりの出番は減り、更新も追いつかなくなる。いまは「二回以上、手で打った指示だけを資産にする」に線を引き直して、呼ばれないものは棚卸しで消した。一回きりの指示は、その場で書けばいい。全部を標準化する必要はない。

もう一つ、プロンプトも古くなる。観点カタログにない種類のバグがレビューを素通りしたら、それはカタログの改善要求だ。バグを見つけたらテストを足すのと同じ回路で、漏れを見つけたら観点を足す。この回路が回っていないカタログは、少しずつ現実とずれていく。次の課題は、プロンプトの変更を履歴に残すだけでなく、過去の事例で回帰評価できるようにすることだ。

## 組織で使うなら

個人開発なら、効いたプロンプトが埋もれても、損をするのは自分だけだ。組織では、うまい人のプロンプトがその人のローカルに閉じる。すると、チームの AI 活用の品質は、各自の言い回しの巧拙に比例してばらつく。ばらつきは前提であって、責める話ではない。仕組みで吸収する。標準の観点カタログと依頼の型をリポジトリに置けば、少なくとも誰が頼んでも同じ観点をレビューの出発点にできる。うまい人の言い回しは、その人の技からチームの装備に変わる。

プロンプトの改善が PR になることも、組織ではより効く。「観点を一つ足したい」が差分としてレビューされ、合意されてから標準になる。口伝やチャットの「こう頼むといいらしい」で広まるのと違って、いつ誰が何を変えたかが残る。

ただし、増やしすぎの失敗は、組織のほうが速く進む。全員が善意で足していくと、カタログはすぐに読まれない厚さになる。足す仕組みだけでなく、削る側の役割と基準を先に決めておいたほうがいい。

なお、リポジトリに入れる指示もコードと同じく情報分類の対象になる。公開リポジトリでは、社内用語、内部 URL、認証情報、顧客固有情報を含めない。

AI が増えるほど壊れやすいのは、同じ作業を頼んでいるつもりで、実は全員が違う指示を出していることだ。そして人間に最後まで残るのは、どの言い回しを標準に昇格させ、どれを削るかという取捨だと思う。プロンプトが資産になると、その取捨がチームの設計判断になる。

## 明日試せる最小の一歩

直近一週間で二回以上手で打った AI への指示を、一つ選ぶ。それをファイルにしてリポジトリに置き、次からは「このファイルを読んでやって」で頼む。スキルの仕組みがあるツールならそれに乗せればいいし、なくてもファイルと一文で始められる。自分の場合、最初に入れたのは観点 4 つのレビューカタログ 1 枚だけだった。それだけで、日によって観点を入れ忘れる問題はなくなり、レビュー品質が変わったときにプロンプトの履歴も調べられるようになった。
