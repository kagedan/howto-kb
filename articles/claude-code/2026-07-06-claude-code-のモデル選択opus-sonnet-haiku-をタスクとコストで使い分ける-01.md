Claude Code はデフォルトのまま使えますが、**タスクの難しさとコストに合わせてモデルを切り替える**と、体感速度と請求額が大きく変わります。この記事は「どこで・どう切り替えるか（操作）」と「何を基準に選ぶか（判断）」を、Claude Code の実際の設定に落として整理します。

要点を先に：切り替え手段は4つあり優先順位が決まっていること、`opusplan` という“計画はOpus・実行はSonnet”のハイブリッドがあること、`/fast`（fast mode）は**モデルを変える機能ではない**こと。ここを押さえると迷いが消えます。

!

**対象読者と前提**

* Claude Code（CLI）を日常的に使うエンジニア
* コンテキスト長・トークン単価の概念が分かる人
* モデルの単価・コンテキスト長・提供状況は更新されます。数値は **2026年時点の目安**として扱い、**最新は Anthropic 公式の pricing / models ページで確認**してください。切り替え仕様は Claude Code 公式ドキュメント（code.claude.com/docs）で裏取りしています。

## 結論：まず「切り替え手段の優先順位」を頭に入れる

Claude Code のモデルは、次の4か所で指定できます。**上にあるものほど強い（優先される）** のがポイントです。

モデル指定の優先順位（上が強い）

```
1. 起動時フラグ         claude --model opus
2. 環境変数             ANTHROPIC_MODEL=claude-sonnet-5
3. セッション内コマンド  /model  （その場で切り替え）
4. 設定ファイル         .claude/settings.json の "model"
```

「普段はSonnet、この案件だけOpus」を環境変数や `--model` で上書きし、恒久的な既定は `settings.json` に置く——という重ね方ができます。順に見ます。

## 1. `/model`：セッション内でその場で切り替える

対話中に `/model` を打つとモデル選択メニューが出ます。選べる値は用途別のエイリアスと、フルのモデルIDです。

/model で選べる主な値

```
default   … Claude Code の既定
opus      … 最上位の推論力（例：claude-opus-4-8）
sonnet    … コーディング/エージェント作業の主力（例：claude-sonnet-5）
haiku     … 速さと低コストが持ち味（例：claude-haiku-4-5）
opusplan  … 計画中は opus、実行フェーズで sonnet に自動切替（後述）
best      … 最上位クラスの性能を選ぶ
（この他 claude-opus-4-8 のようなフルIDも直接指定可）
```

選択時の挙動は「**Enter＝以後の既定として保存**」「**s＝今のセッションだけ切り替え**」です。試しに上げたいときは `s`、恒久的に変えたいときは Enter、と使い分けます。

## 2. 恒久設定：`settings.json` の `model`

いつも同じモデルで始めたいなら、設定ファイルに書きます。プロジェクト単位なら `.claude/settings.json` です。

.claude/settings.json

```
{
  "model": "sonnet"
}
```

値はエイリアス（`opus` / `sonnet` / `haiku` / `default` / `opusplan` など）でもフルIDでも構いません。**長いコンテキストを使いたい**ときは `opus[1m]` / `sonnet[1m]` のように 1M コンテキスト指定のサフィックスも使えます。

.claude/settings.json（1Mコンテキストを既定にする例）

```
{
  "model": "sonnet[1m]"
}
```

## 3. 一時上書き：`ANTHROPIC_MODEL` と `--model`

「この起動だけ別モデル」は環境変数か起動フラグでやります。設定ファイルより強いので、既定を汚さずに上書きできます。

```
# この1回だけ Opus で起動
$ claude --model opus

# シェルセッション中は Sonnet 5 に固定
$ export ANTHROPIC_MODEL=claude-sonnet-5
$ claude
```

優先順位は再掲すると `--model`（最優先）→ `ANTHROPIC_MODEL` → `settings.json`。CI やスクリプトからは環境変数、手元の一発は `--model` が扱いやすいです。

## 4. 使い分けの基準：タスク × コスト

ここが本題です。3系統の性格を、**得意な仕事**と**相対コスト**で並べます。単価は 2026年時点の目安で、比率の感覚（入力で概ね Opus : Sonnet : Haiku ＝ 5 : 3 : 1）を掴むのが目的です。

| 系統 | エイリアス | 向いているタスク | コンテキスト | 入力/出力 単価（1Mトークン, 目安） |
| --- | --- | --- | --- | --- |
| Opus | `opus` | 難しい設計判断・長時間の自律作業・込み入ったデバッグ | 1M | $5 / $25 |
| Sonnet | `sonnet` | 日常のコーディング・エージェント作業の主力 | 1M | $3 / $15 |
| Haiku | `haiku` | 単純/定型・分類・サブエージェントの下請け | 200K | $1 / $5 |

判断の目安はシンプルです。

* **Opus**：一発で失敗したくない設計変更、原因が読めないバグ、長い自律実行。単価は高いが手戻りを減らせる場面で効く
* **Sonnet**：普段の実装・リファクタ・テスト作成。品質と速度のバランスが良く、既定の主力に向く
* **Haiku**：定型の分類・要約・整形、そして**サブエージェントの下請け**。速くて安い

## 5. `opusplan`：計画は Opus、実行は Sonnet

「設計は賢く、実装は安く速く」を1つのエイリアスにしたのが `opusplan` です。**プランモード中は Opus** で考え、**実行フェーズに入ると自動で Sonnet に切り替わります**。

.claude/settings.json

```
{
  "model": "opusplan"
}
```

汎用の“よしなに切り替え”ではなく、**計画↔実行の境目でモデルが切り替わる**という明確な線引きです。「難所の設計はOpusに任せたいが、手を動かす部分までOpus単価は払いたくない」ときの既定として噛み合います。

## 6. `/fast`（fast mode）はモデル変更ではない

よくある誤解の整理です。`/fast` で入る fast mode は **Opus をより速く動かす設定**であって、別モデルへの切り替えではありません。

* **Opus 専用**（Sonnet / Haiku 等では使えない）
* 同じ Opus を高速に動かす代わりに、**トークン単価は割高**
* 「速さが欲しい」＝「安いモデルにする」ではない点に注意

安くしたいなら Sonnet / Haiku へ切り替え、Opus のまま速くしたいなら fast mode、と目的で使い分けます。

## 7. サブエージェントのコスト設計：`model: inherit`

`.claude/agents/*.md` で定義するサブエージェントは、frontmatter で自分のモデルを持てます。既定は `inherit`（親＝メイン会話と同じモデル）です。

.claude/agents/log-scanner.md

```
---
name: log-scanner
description: ログを走査して該当行を抜き出す下請け
model: haiku
---
あなたはログ調査の下請けです。指定パターンに一致する行だけを抜き出して返します。
```

メインを Opus/Sonnet に置きつつ、**単純な走査・抽出の下請けは Haiku に落とす**と、賢さが要る所だけ高いモデルを使えます。ここは `model: inherit` のまま放置しがちですが、下請けを明示的に安いモデルへ振ると効きます。

## 8. コスト感：単価から試算し、キャッシュで下がる

表の単価から実際の一往復を試算してみます。あくまで**公表単価からの算術（想定例）** で、ベンチマークではありません。

試算（想定例）— 入力200K + 出力20K トークンの1往復

```
Opus   : 0.2M×$5 + 0.02M×$25 = $1.00 + $0.50 = $1.50
Sonnet : 0.2M×$3 + 0.02M×$15 = $0.60 + $0.30 = $0.90
Haiku  : 0.2M×$1 + 0.02M×$5  = $0.20 + $0.10 = $0.30
```

同じ作業でも Opus と Haiku で単価ベースはおよそ5倍差。ただし実効コストはもっと下がります。**Claude Code はプロンプトキャッシュを活用する**ため、繰り返し送られる前提（システムプロンプトやファイル文脈）は**キャッシュ読み出しで概ね1/10**の単価になります。つまり表の単価は“上限の目安”で、長い会話ほど実効単価はそこから下がっていきます。

## まとめ

* 切り替えは4手段、優先順位は **`--model` > `ANTHROPIC_MODEL` > `/model` > `settings.json`**
* 基準は **タスクの難しさに単価を合わせる**：難所=Opus / 日常=Sonnet / 定型・下請け=Haiku
* **`opusplan`** は計画=Opus・実行=Sonnet のハイブリッド。設計だけ賢くしたい既定に向く
* **`/fast` はモデル変更ではない**（Opus を速くする・単価は割高）。安くしたいなら安いモデルへ
* サブエージェントは **`model: inherit` を見直し**、下請けを Haiku に落とすと効く
* 単価・コンテキスト長・提供モデルは更新される。**数値は目安、最新は公式で確認**
