## きっかけ: 「loop engineering」という言葉に納得できなかった

**筆者:** 2026年に入って "loop engineering" という言葉をよく見かけるようになりました。曰く、prompt engineering < context engineering < harness engineering < loop engineering という4層の入れ子構造があり、その頂点にあるのが perceive → reason → act → observe のループを設計する loop engineering である、と。5つのlevers（tools, stopping conditions, context management, verification, guardrails）と6つのprimitives（automations, worktrees, skills, plugins/connectors, sub-agents, state/memory）が語彙として整理されています。

筆者は業務でClaude Codeのharness（プラグイン群）を自作・運用しており、Context/Plan/Act/Verify/Stall検知/Safety rail/Observability/HOTLという軸に沿ってプラグインを配置しています。この構成を指して「loop engineeringをやっている」と言えるのか、そもそも筆者のharnessは既存のloop engineeringという概念に対して優れているのか劣っているのか——という疑問をClaude Codeにぶつけたのが出発点でした。

結論を先に書きます。**「loop engineering」は工程の並びを言い当てていますが、その工程の境界を誰がどう検証するかについては何も規定していません**。そこにこそ実務上の価値の分水嶺がある、というのが最終的にたどり着いた理解です。

## loop engineeringが言っていること・言っていないこと

**Claude Code:** industry用語としてのloop engineeringは、要するに「ループを回すこと自体を設計対象にする」という主張です。perceive（状況を読む）→ reason（計画する）→ act（実行する）→ observe（結果を見る）を1周期とし、この周期を回し続けることでタスクを完遂させます。5つのleversは「ループの質をどう上げるか」のチェックリストとして機能します。

* **tools** — ループが使える手段の集合
* **stopping conditions** — いつループを止めるか
* **context management** — 何を記憶し何を忘れさせるか
* **verification** — 出力が正しいかどうかの確認
* **guardrails** — 逸脱を防ぐ柵

これ自体はよくできた整理だと思います。問題は、これらのleversが実装レベルでどう実現されるかについて、loop engineeringという語彙は何も強制しないことです。多くの実例記事を読む限り、"verification" も "guardrails" も、結局は**プロンプトの中の指示文**として実装されることが多いようです。「このタスクが完了したと判断したら次に進め」「ガードレールを守れ」——こう書いておけば、あとはLLM自身がその指示にどれだけ忠実であるかに懸かってしまいます。

つまり loop engineering は「何を確認すべきか」という**設計者の意図**を語る語彙であって、「誰がその確認を実行し、その結果をどう強制するか」という**実行時の権限構造**については語っていません。この語彙の抜け穴に気づいたことが、この記事の出発点になりました。

## 工程が同じでも、境界の検証者が違う

**筆者:** 筆者のharnessも、抽象化すればcontext → plan → act → verify → repeat という同じサイクルを回しています。工程の名前も、loop engineeringの4段階とほぼ対応がつきます。ではloop engineeringと何が違うのか——各工程が厳密に定義されているところではないか、とClaude Codeに聞いてみました。

**Claude Code:** その通りですが、もう一段掘れます。「工程が定義されている」こと自体は一般的なloop engineeringにもあります。違いは工程の**存在**ではなく、工程を抜けるための**判定者**です。

たとえば `condukt`（このharnessのタスク実行系プラグイン）は、1つのタスクを Act → Verify のフェーズに通すとき、LLM自身に「これは完了しました」と自己申告させて次へ進むことはしません。かわりに、verify専用の独立したRustバイナリ（`propguard` や `stuckguard` など）を別プロセスとして起動し、そのバイナリが exit code と構造化JSONで判定結果を返します。LLMが書いたコードや報告内容そのものを、LLMとは独立したロジックが**外側から**再検証します。

これを図式化するとこうなります。

```
[loop engineeringの典型的な実装]
LLM: 計画を立てる → 実行する → 「完了しました」と自己申告 → 次工程へ
                                    ^^^^^^^^^^^^^^^^^^^^
                                    検証者 = 生成者本人

[このharnessの実装]
LLM: 計画を立てる → 実行する → 独立バイナリを呼び出す → exit codeで可否が返る → 次工程へ
                                    ^^^^^^^^^^^^^^^^^^
                                    検証者 = 生成者と別プロセス・別ロジック
```

これは特別な発明ではありません。ソフトウェア工学において**Separation of Duties（職務分掌）**、あるいはセキュリティ設計原則としての**Separation of Privilege**（Saltzer & Schroeder, 1975）と呼ばれてきた、古典的な考え方をLLMエージェントのループに単純に持ち込んだものです。会計監査の世界で言う「作成者と承認者を同一人物にしない」という原則、機械学習で言う「学習データとテストデータを分ける」という慣習とも同型の話です。

一般に **maker/checker separation** と呼ばれるこの原則を軸に据えると、筆者のharnessが実際にやろうとしていたことが初めて言語化できました。

## 名前探しの過程

**筆者:** この構造を一言で何と呼ぶべきか、しばらくClaude Codeと自問自答を繰り返しました。最初は「オーケストレーションシステム」と呼ぼうとしましたが、これは狭すぎると気づきました。オーケストレーションはタスクをどう分配し実行するか（Actの軸）を指す言葉で、検証・監査・停止判断（Verify/Stall/Safetyの軸）を含意しません。

次に「Gateされたシステム」と言い換えてみました。「工程の間にgate（関門）を置いて、条件を満たさないと先に進めない」という説明はそれなりに正確に思えましたが、これも違うと感じました。「Gate」は関門という**部品の名前**であって、なぜその関門が必要なのか、その関門は何を担保しているのかという**関係性の原則**を言い当てていません。「loop」も同じで、単に繰り返しの構造を指すだけで、繰り返しの中で何が保証されているかには触れません。

**Claude Code:** ここで学術・実務の既存語彙をいくつか持ち込みました。

* **control plane / data plane** — ネットワーク機器の設計に由来する対比です。LLMは判断・生成を担う data plane、決定論的なRustバイナリは強制・記録を担う control plane。このharnessの設計原則を一言でいうと「判断はLLM、決定論はバイナリ」であり、この対応はよく合います。
* **PDP/PEP（Policy Decision Point / Policy Enforcement Point）** — アクセス制御理論からの借用です。判定ロジックそのもの（PDP。`adjudicate()`のようなpure関数）と、それをどこで呼び出すか（PEP。`condukt`のPhase 6や`autoflow`のStop hook）を分けて考えると、このharnessの多くのコンポーネントが実はPDP/PEPの実装だったことが見えてきます。
* **maker/checker separation** — 最終的に一番しっくり来た言葉です。「誰が」の問題に焦点を当てており、Separation of Duties（職務分掌）、Separation of Privilege（Saltzer & Schroeder）、"nemo judex in causa sua"（何人も自らの事件の裁判官たりえず、というローマ法由来の法諺）、ポパーの反証可能性、機械学習のtrain/test splitと同型の原則として位置づけられます。

ここまで来て、「Gate」も「loop」もこの原則を実装する**部品の名前**に過ぎず、システム全体の名前としては不十分だったことに気づきました。名前を探す作業そのものが、部品の名前とシステムの原理を混同していたことへの気づきのプロセスでした。

## 「Zero Trust」ではない、という筆者の訂正

**Claude Code:** この流れで、一度「これはagent系におけるZero Trustだ」とまとめかけました。

**筆者:** ここは訂正が必要だと感じました。人間が評価することも含めて、この関係は無限後退になりうるが、それは何の価値も埋めません。だからZero Trustと言い切ってしまうのは踏み込みすぎで、実際にはこの関係のどこかに一定の信頼を置くしかない、と指摘しました。

**Claude Code:** その指摘は正確です。「生成者の自己申告は一切信用しない」を文字通り貫徹しようとすると、無限後退が起きます。checkerの判定を誰が検証するのか。そのmeta-checkerの判定を誰が検証するのか——これを延々と続けても、価値は増えません。認識論でいう\*\*Agrippaのトリレンマ（後退の問題）\*\*そのもので、あらゆる正当化の連鎖は次の3つにしかなりません。

1. 無限後退（価値を生まない）
2. 循環論法（実質的な正当化にならない）
3. どこかで意図的に打ち切る（独断的だが、これしか実用上の選択肢がない）

実際、NIST SP 800-207が定義する実務上のZero Trustも、文字通り「何も信用しない」わけではありません。暗黙の信頼（「社内ネットワークにいるから信用する」）を排除するのが主旨であり、どこかに明示的な **root of trust（信頼のアンカー）** を置くことは前提にしています。

このharnessでの root of trust は2箇所に集約されます。

* **人間** — エスカレーションの終端。人間の判断をさらに機械検証する仕組みは存在しませんし、存在させても無限後退になるだけです。
* **一度だけ監査された決定論的コード** — checkerとなるRustバイナリ自身は、実行のたびに「今回は正直か」を問い直す必要がありません。コードレビューとテストで**一度**検証されれば、決定論（同じ入力には同じ出力）という性質によって、以後の全ての実行で同じ保証が続きます。

この非対称性——LLMの自己申告は毎回新しく生成される主張なので都度の検証が要るが、静的なコードは一度の監査で足りる——が、maker/checkerを分ける本当の理由になります。「Zero Trust」という強い言葉より、\*\*「信頼しなければならない面積を最小化し、残った面を小さく・明示的に固定する」\*\*という言い方の方が正確です。

## 具体的な差分: プロンプトの基準 vs 型付きの契約

**筆者:** ここまでの話をまとめると、「作り手は承認者になってはならず、基準をクリアしないといけない。その基準はプロンプトではなく型付きの条件にする」ということだと筆者なりに再整理してみました。

**Claude Code:** その言い換えで正確です。loop engineeringの典型的な実装では、基準はプロンプト内の自然言語で表現されます。

```
このタスクが完了したと判断したら、次のステップに進んでください。
```

この文には常に解釈の余地が残ります。「完了した」の基準が曖昧なままLLMの裁量に委ねられていて、これは結局「生成者自身に判定させる」ことと変わりません——マーカーとチェッカーが同一の主体（同じLLM、同じcontext）に同居してしまっています。

これに対し、このharnessでは工程の境界を**型付きの契約**として表現します。たとえば`condukt adversarial adjudicate`が返す判定は次のような閉じた列挙型になっています（実装を単純化した例です）。

```
enum Verdict {
    Pass,
    Block,
    Escalate,
}
```

呼び出す側（LLM）は、`match`で全パターンを扱わない限りコンパイルが通りません。「Pass」でも「Block」でも「Escalate」でもない、曖昧な第4の状態を勝手に発明する余地が型レベルで存在しません。これはBertrand Meyerが1986年にEiffel言語で提唱した**Design by Contract（DbC）**——契約をドキュメントやコメントではなく、コンパイラが検証できる形で言語機構に埋め込むという考え方——をagentのループ境界に応用したものだと言えます。

## なぜRustなのか

**筆者:** ここでもう一つ気づいたことがありました。実装言語そのものが、この機構——特にDesign by Contract——を支えているのではないか、という点です。

**Claude Code:** その通りです。同じ発想を型のない言語やプロンプトだけで実現しようとすると、契約の強制力は結局「LLMがちゃんと従うかどうか」に退化してしまいます。Rustが持つ以下の特性は、maker/checker separationを実装可能にする**前提条件**になっています。

1. **代数的データ型＋網羅的match** — 基準を閉じた列挙型として表現でき、コンパイラが全パターンの処理漏れを検出します。プロンプトの基準は自然言語の解釈余地を常に残しますが、`enum`は解釈の余地そのものが型レベルで存在しません。
2. **`Result`/`Option`が暗黙のfail-softを許さない** — 何かに失敗したとき、暗黙に「たぶん大丈夫」で先へ進むことがコンパイラレベルで不可能です。fail-closedにするかfail-softにするかを、`unwrap_or`か`?`による伝播かという形で明示的に選ばせます。
3. **serdeの厳格デシリアライズ** — LLMが吐いたJSONが期待する構造体の型に合わなければ、パース自体が失敗します。「基準を満たしたふりをした出力」を型そのものが弾きます。プロンプトで「このJSON形式で返して」と依頼するのとは強制力の階梯が違います——後者はLLMの善意への依頼、前者はパーサによる拒否です。
4. **プロセス境界＝OSレベルの隔離** — コンパイル済みバイナリをCLIサブプロセスとして呼び出し、exit codeと構造化stdoutだけでやり取りします。作り手（LLM）は物理的にcheckerのメモリ空間・内部ロジックへ到達できません。同じcontext内で別のLLM呼び出しを挟むより強い分離で、境界が言語仕様ではなくOSのプロセスモデルによって担保されます。
5. **契約の検証タイミングがコンパイル時** — プロンプトの遵守はLLMの推論のたびに再判定される、いわば毎回揺らぎうる意志の問題です。Rustの型契約はビルド時に一度きり検証され、以後は言語の決定性（同じ入力に同じ出力）によって同じ保証が続きます。UB（未定義動作）が残る言語であれば「一度監査すれば以後信用できる」という前提そのものが崩れてしまいます——Rustの安全性保証は、root of trustを「一度の監査」に固定できることの土台になっています。

## それでも、有用であればいい

**筆者:** 理論の精緻さより先に、実務上の話をしておきたいと思います。ここまで書いてきた原則も、突き詰めれば正当化は途中でどこかに打ち切らざるを得ません。完璧な無矛盾性を追い求めても価値は増えないというのが実感です。

実際の運用では、`/flow`のようなコマンドを打つだけで、このharnessの大半は自律的に組み上がっていきます。もちろん間違いは起きます。しかし maker/checker separation が工程の随所に埋め込まれているおかげで、その間違いは大抵どこかのgateで検出され、修正のループに戻されます。完璧に間違えないシステムを目指したのではなく、**間違いが起きても検出され、修正される構造**を目指した結果として、これが実用に足るものになっていると考えています。

## まとめ

* 「loop engineering」は工程の並びを整理する語彙としては有用ですが、工程の境界を誰が検証するかについては規定していません。
* 実務上価値を生むのは、工程の存在そのものより、**境界の判定権限を生成した主体から構造的に分離すること**——maker/checker separation（Separation of Duties/Privilege）です。
* この分離を実装する媒体は「プロンプトによる依頼」ではなく「型付きの契約」であるべきで、これはDesign by Contractのagentループへの応用と言えます。
* 契約の強制力は言語機構に依存します。代数的データ型・`Result`/`Option`・厳格デシリアライズ・プロセス分離・コンパイル時検証を持つRustは、この原則を実装可能にする土台になっています。
* 無限後退は避けられず、どこかに明示的なroot of trust（人間の最終判断、一度監査された決定論的コード）を置く必要があります。「Zero Trust」ではなく「信頼の最小化と明示化」というほうが正確です。
* 最終的な価値基準は理論的な無矛盾性ではなく実用性であり、この構造の役割は「間違えない」ことではなく「間違いが検出され、修正される」ことを保証することにあります。
