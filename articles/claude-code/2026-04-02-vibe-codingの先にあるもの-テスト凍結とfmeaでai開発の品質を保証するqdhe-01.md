## AI開発の現在地 — 速さの代償

2025年、AI駆動開発は爆発的に普及した。GitHub Copilotの補完に始まり、CursorやWindsurf、Claude Codeといったエージェント型ツールが次々と登場し、「プロンプトを書けばコードが出てくる」時代が現実になった。

だが、速さには代償がある。

Veracodeが100以上のLLMを対象に実施した調査（2025 GenAI Code Security Report）では、**AI生成コードの45%がOWASP Top 10に該当するセキュリティ脆弱性を含んでいた**。しかもこの数値は、モデルが大型化・新型化しても改善していない。LLMは構文的に正しいコードを書く能力は向上しているが、セキュアなコードを書く能力は横ばいのままだ。

一方、METR（Model Evaluation & Threat Research）が2025年2〜6月に実施したRCTでは、経験豊富なOSS開発者16人が246タスクをAI使用/不使用で実施した結果、**AI使用時に完了時間が19%増加**した。さらに衝撃的なのは、開発者自身は「AIで20%速くなった」と感じていたことだ。知覚と現実の間に約40ポイントの乖離がある。

つまり我々は今、**速く書けるが壊れやすく、速くなった気がするが実は遅い**という二重の罠の中にいる。

## AI開発手法の変遷 — 3つの時代

この状況を理解するために、AI開発手法の変遷を整理しておきたい。

**2023〜24年：プロンプトエンジニアリング期。** AIは補助ツールだった。開発者がコードを書き、AIがそれを補完する。主役はあくまで人間で、ChatGPTやCopilotに「この関数を書いて」と頼む使い方が中心。プロンプトの質が出力の質を決めた。

**2025年：コンテキストエンジニアリング期。** AIは共同作業者になった。CursorやWindsurfが台頭し、`CLAUDE.md`や`AGENTS.md`といった設定ファイルでAIの振る舞いを制御するようになる。「何を頼むか」から「どんな文脈を渡すか」に焦点が移った。AIはコードベース全体を理解し、ファイルをまたいだ変更を提案できるようになった。

**2026年：ハーネスエンジニアリング期。** AIはチームメンバーになった。OpenAI CodexやClaude Codeの自律エージェントが、issue単位で独立して実装を進める。人間はコードを書かず、**AIが正しく動作するための環境（ハーネス）を設計する**。METR研究の参加者が「AIなしで作業するのは、Uberに慣れた後に歩いて街を横断するようなものだ」と語ったように、もはやAIなしの開発には戻れない段階に入りつつある。

しかし、ここで本質的な問いが浮かぶ。**AIが自律的にコードを書く時代に、品質は誰が保証するのか？**

vibe codingの「AIに任せて雰囲気でコードを書く」スタイルは生産性を感じさせるが、先述の通りセキュリティ脆弱性を45%の確率で埋め込む。人間がレビューする前提なら許容範囲かもしれないが、エージェントが自律的に動く世界では、そのレビュー自体をどう担保するかが問題になる。

## QDHEの着想 — 品質を「仕組み」で保証する

この問題に対する自分の答えが、**QDHE（Quality-Driven Harness Engineering）** だ。

着想の原点は3つある。

まず**V字モデル**。組込み開発でお馴染みの、要件→設計→実装→テスト→検証という対称構造。AIが書くコードにも、要件からテストまでのトレーサビリティは必要だ。

次に**TDD（テスト駆動開発）**。テストを先に書き、実装はテストを通すために書く。AI開発では特に重要で、「AIが書いたコードが正しいか」を判定する基準が先に必要になる。

そして**FTA/FMEA**。信頼性工学の故障解析手法だ。FTA（Fault Tree Analysis）はトップダウンで「何が起きたら致命的か」を分析し、FMEA（Failure Mode and Effects Analysis）はボトムアップで「各コンポーネントがどう壊れるか」を列挙する。

この3つを**ハーネスエンジニアリング**の思想で統合した。ハーネスとは本来、テスト対象を制御・観測するための治具のこと。QDHEでは、AIエージェントが正しく動作するための制約・ルール・検証機構の総体をハーネスと呼ぶ。

## QDHEの仕組み — 10ステップ × 7エージェント

QDHEは10ステップのパイプラインと、7つの専門AIエージェントで構成される。

**人間が関与するのはStep 1の要件伝達と、各ステップ間の承認ゲートだけだ。** コードは1行も書かない。

**7つのエージェント**は、Claude Code Agent Teamsとして協調動作する。

* **Orchestrator** — 全体の進行管理。ステップ間のゲートキーピング
* **Planner** — 要件定義、設計、テストリスト作成
* **Reviewer** — テスト観点のレビュー、抜け漏れ指摘
* **Writer** — ドキュメント生成
* **Coder** — 実装コードの生成
* **Validator** — テストコードの凍結・検証、ミューテーションテスト
* **Analyst** — FTA/FMEAによるリスク分析

### テスト凍結 — AIの「うっかり改ざん」を防ぐ

QDHEの中核機能の一つが**テストコードのSHA256ハッシュ凍結**だ。

AIエージェントにテストを書かせ、実装も任せると、テストが通らないときにAIがテストの方を書き換えるという問題が頻繁に起こる。人間なら「テストを直すな、実装を直せ」と判断できるが、エージェントにとってはどちらも「テストをGREENにする手段」でしかない。

QDHEでは、Step 6でテストコードが確定した時点でSHA256ハッシュを記録し、以降のステップでテストファイルの改変を機械的に検出・拒否する。これは精神論やプロンプトの指示ではなく、物理的な制約だ。

### ハーネス3層構造

QDHEのハーネスは3つの層で構成される。

重要なのは、3層目の存在だ。テストが正しいことを誰が保証するのか？ QDHEではReviewerエージェントがテスト観点をレビューし、ミューテーションテストでテストの殺傷力を検証し、ハッシュ凍結で改ざんを防止する。**テストをテストする仕組み**が組み込まれている。

## 実戦：AI人狼ゲームを1時間で作る

理論だけでは説得力がないので、実際にQDHEでゲームを作った。

**仕様：人間1人 + AI NPC 4体の5人人狼**。技術スタックはTypeScript、React + Node.js + WebSocket + Claude API。フロントエンドでプレイヤーが操作し、バックエンドでAI NPCがClaude APIを使って推理・議論・投票する。

結果から言うと、**1時間でStep 7（実装+全テストGREEN）まで完走した**。

自分がやったことは、最初の要件伝達と、途中数回の承認ゲートでの「OK」だけだ。コードは1行も書いていない。

### タイムライン（概算）

| 経過時間 | ステップ | 内容 |
| --- | --- | --- |
| 0〜5分 | Step 1-2 | 要件定義・設計。人間が仕様を伝達 |
| 5〜15分 | Step 3 | FTA/FMEA。Analystがリスク分析を実行 |
| 15〜25分 | Step 4-5 | テストリスト104件作成 → Reviewerが13件追加して117件 |
| 25〜30分 | Step 6 | テストコード102件を凍結（SHA256ハッシュ記録） |
| 30〜60分 | Step 7 | 実装+テスト。Coderが実装し、全102件GREEN |

## FMEAが見つけたもの — 設計前のバグ退治

FMEA（故障モード影響解析）は、コードを1行も書く前に「何がどう壊れるか」を網羅的に洗い出す手法だ。Analystエージェントが人狼ゲームの各コンポーネントに対してFMEAを実行した結果、**RPN（Risk Priority Number）上位7件**が特定された。

最も高いRPNは**135**、故障モードは「**タイムアウト後の二重処理**」（ID: N-02）だった。WebSocketでの通信タイムアウト後にリトライが走り、サーバー側で同じアクションが二重に処理される問題だ。人狼ゲームで言えば、投票が2回カウントされて結果が狂う。

これは実装後のデバッグで見つけていたら、修正コストが何倍にも膨らんでいただろう。FMEAが設計段階で検出し、テストケースとして先に組み込まれたことで、Coderは最初から二重処理防止のロジックを実装せざるを得なかった。

RPN≥100の上位7件を表にまとめるとこうなる。

| ID | コンポーネント | 故障モード | RPN |
| --- | --- | --- | --- |
| N-02 | NightProcessor | タイムアウト後の二重処理 | 135 |
| G-06 | GameEngine | 非同期フェーズ遷移の競合（排他制御なし） | 126 |
| A-03 | AIController | Claude APIが脱落済みtargetIdを返した際のフォールバック未実施 | 120 |
| A-06 | AIController | AiActionContextにnightActionsが混入し情報漏洩 | 120 |
| G-01 | GameEngine | checkWinCondition()が一部の投票パスで呼ばれない | 108 |
| G-03 | GameEngine | day=3投票後のdraw宣言でday++タイミング競合 | 108 |
| G-07 | GameEngine | getState()が内部状態の参照を返し外部から書き換え可能 | 105 |

注目すべきは、7件中3件がAI固有の故障モードだということだ。

**G-06（RPN=126）：非同期フェーズ遷移の競合。** GameEngineがフェーズ遷移処理中に別のアクションが到着すると、同一ゲームにphaseが2つ存在する状態が起きる。AI NPCは人間と違って複数が同時に応答を返すため、人間だけのゲームより発生頻度が高い。排他制御の実装が必須になる。

**A-03（RPN=120）：Claude APIが脱落済みプレイヤーのIDを返す。** LLMは「生存者リスト」をプロンプトで渡されても、過去のコンテキストに引きずられて脱落済みプレイヤーを指名することがある。AIControllerのバリデーションとフォールバックが必要になる。

**A-06（RPN=120）：nightActionsの情報漏洩。** AIのプロンプト構築時に、`PublicGameState`ではなく`GameState`を誤って渡すと、他プレイヤーの夜行動がAI NPCに見えてしまう。人間プレイヤーには送信されないが、AI NPC同士で情報が筒抜けになる。

これらはいずれも「AIがプレイヤーとして参加する」という構造から生まれる故障モードであり、従来の人間だけのゲーム開発では出てこない。FMEAがなければ、実装後にプレイテストで偶然発見するのを待つしかなかった。

FTAでは4つのトップ事象を定義した。「ゲーム進行不能」「勝利判定の誤り」「NPCの不正行動」「情報漏洩」。それぞれのフォールトツリーを展開し、末端の基本事象がFMEAの故障モードと対応していることを確認した。たとえばFTA-1「ゲーム進行不能」のフォールトツリーは、NightProcessorの永久ブロック → タイムアウトタイマー未設定（BE-2）→ FMEA N-01（RPN=81）という形でFMEAと接続される。N-01自体はRPN≥100の必須ラインには入らないが、上位のN-02（RPN=135、タイムアウト後の二重処理）と連鎖する故障モードであり、FTAの構造がこの依存関係を可視化している。

## AIがAIをレビューする — Reviewerの仕事

QDHEのStep 5では、Reviewerエージェントがテストリストをレビューする。Plannerが作ったテストリスト104件に対して、Reviewerは以下の指摘を行った。

**HIGH指摘5件**。特に重要だったのは**TC-011のテストデータ矛盾**の検出だ。「同数票のテスト」と記載されているのに、テストデータが単独最多票になっていた。他にもゲームログ記録テストの完全欠落、決選投票後の`checkWinCondition`呼び出し未検証など、Plannerが見落としていた観点が次々と指摘された。

**追加テスト13件の提案**。たとえば「ゲーム途中でプレイヤーが切断した場合のリカバリ」「E2E統合シナリオ」「脱落状態への遷移テスト」など。

結果として、テストリストは**104件→117件**に拡充され、そこからテストコードとして実装された**102件が凍結**された。117件から102件に減っているのは、フロントエンドUI系15件がバックエンドテストの範囲外だったためだ。

ここで重要なのは、**AIが書いたテストをAIがレビューしている**という構造だ。一見すると「AIがAIを評価して意味があるのか」と思うかもしれない。しかし、PlannerとReviewerは異なる観点を持つ別のエージェントであり、Reviewerはテストの網羅性・境界値・異常系に特化したプロンプトで動作する。人間のコードレビューでも、書いた本人とは別の目で見ることに価値があるのと同じだ。

## ミューテーションテストの衝撃 — 102件GREENの幻想

Step 7で全102件のテストがGREENになった時点で、自分は正直「これで品質十分やろ」と思っていた。だがQDHEのパイプラインはここで終わらない。Step 8のミューテーションテストが、その幻想を打ち砕いた。

ミューテーションテストとは、実装コードに意図的な小さな変異（ミュータント）を注入し、テストがそれを検出できるかを検証する手法だ。たとえば `if (x > 0)` を `if (x >= 0)` に変えたり、`return true` を `return false` に変えたりする。テストが壊れれば（ミュータントが「殺される」）、そのテストは有効。テストが通ったまま（ミュータントが「生存」する）なら、そのテストはその箇所を本当には検証できていない。

Strykerを使った初回実行の結果は**スコア43.89%**。閾値70%に遠く及ばない。

```
Survived（生存）: 460件
NoCoverage（未カバー）: 247件
合計未検出ミュータント: 707件
```

つまり、**102件のテストが全部GREENでも、コードの半分以上はテストが実質的に検証できていなかった**。

これがvibe codingの典型的な落とし穴だ。テストが通る = 品質が高い、ではない。テストが「何を見ていないか」を測定しなければ、カバレッジ100%でも偽の安心感に過ぎない。

### テスト凍結の解除と再凍結

ミューテーションスコアが閾値を下回ったため、QDHEパイプラインは`/qdhe-unfreeze`コマンドでテスト凍結を一時解除する。Writer 3体が並行でテスト追加に入った。

追加されたのはこんなテストだ。

* `prompt-builder.ts`専用テスト — AIプロンプト構築の境界値。空の生存者リスト、全員脱落済み、夜行動情報の非混入を検証
* `ws-server.ts`専用テスト — WebSocketメッセージルーティング、不正JSONのハンドリング、セッション管理
* 既存テストファイルへの大量追記 — 副作用検証、NoCoverageパスのカバー、境界値の網羅

テスト数は**102件 → 352件**（+250件）に激増。再実行の結果は\*\*70.95%\*\*で閾値70%をクリア。テストを再凍結してStep 9に進んだ。

### Step 9〜10: リファクタと最終検証

Step 9ではCoderエージェントがリファクタリングを実施した。型アサーション（`as`）の削除、重複コードのヘルパー関数抽出、`flatMap`による型安全な変換など。352件のテストが全て通ることを確認しながらの安全なリファクタだ。テストが凍結されているから、Coderは「テストを直して辻褄を合わせる」ことができない。実装だけを改善するしかない。

Step 10の最終検証では、Validatorが全指標を一括チェックした。

| 指標 | 結果 |
| --- | --- |
| テスト | 352件 全GREEN |
| 行カバレッジ（backend） | 95.36% |
| 分岐カバレッジ（backend） | 94.29% |
| ミューテーションスコア | 70.95% |
| テスト凍結ハッシュ | 一致 |
| TypeScriptビルド | 全パッケージ成功 |
| **総合判定** | **PASS** |

## まとめ: 人間の仕事が変わる

振り返ると、自分がこのプロジェクトでやったことは驚くほど少ない。

* 最初に「5人人狼、AI NPC 4体、TypeScript」と要件を伝えた
* Step間の承認ゲートで数回「OK」と言った
* Step 1で未決定事項12件に回答した

コードは1行も書いていない。設計もテストもFMEAも、全てAIエージェントが実行した。それでも352件のテストが全てGREEN、ミューテーションスコア70.95%、カバレッジ95%超えという品質が出た。

これは「AIがすごい」という話ではない。**AIを正しく動かすための仕組み（ハーネス）を設計すれば、品質は仕組みから生まれる**という話だ。

vibe codingの問題は、AIに品質判断を委ねていることにある。AIは「テストをGREENにする」ことは得意だが、「テストが十分か」を判断するのは苦手だ。だからテスト凍結でAIの改ざんを防ぎ、FMEAで故障モードを事前に洗い出し、ミューテーションテストでテストの殺傷力を検証する。人間の仕事は「コードを書くこと」から「品質を保証する仕組みを設計すること」に変わった。

QDHEはまだ始まったばかりだ。2層学習（実行ログからの自動改善）やフロントエンドパイプラインの統合はこれからの課題になる。OSS公開を予定しているので、興味のある方はぜひフィードバックをいただきたい。
