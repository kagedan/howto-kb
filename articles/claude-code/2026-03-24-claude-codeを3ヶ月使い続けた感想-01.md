ClaudeCodeの進化がすごい。

---

## 何がそんなに違うのか

CursorもCopilotも使っていたがどちらも便利だった。でも使い方は結局「補完ツール」だった。コードを書いている途中で候補を出してもらうだけだった。

Claude Codeは少し違う使い方をする。

ターミナルから `claude` と打って、タスクを投げる。「このAPIエンドポイント、レスポンスの型定義がないから追加して、テストも書いて」みたいな形で。するとClaudeがコードを読んで、修正して、テストを書いて、「こういう理由でこうしました」と説明してくる。

最初は慣れなかったが、1週間後には普通になっていた。

---

## CLIであることが、思ったより重要だった

Cursorは優秀だが、IDEの中でしか動かない。

Claude CodeはCLIなので、どこからでも呼べる。シェルスクリプトから呼べる。CI上で動かせる。「このブランチのdiffを見てコードレビューして」をローカルでもCIでも同じコマンドで実行できる。

これが地味に便利で、使い方の幅が広い。

IDEに縛られないということは、自分の作業フローに組み込みやすいということでもある。Vimを使っていようが、VSCodeだろうが、エディタ関係なく使える。

---

## CLAUDE.mdというファイル

プロジェクトのルートに `CLAUDE.md` というファイルを置くと、Claudeがそれを読んで作業してくれる。

```
# CLAUDE.md

## プロジェクト概要
決済系のAPIサーバー。本番障害が起きると洒落にならないので慎重に。

## コーディング規約
- エラーハンドリングは必ずtry-catchで囲む
- DBトランザクションを使う場合は必ずロールバック処理を書く
- テストなしのPRは出さない

## 触ってはいけない場所
- /src/legacy/ 以下は絶対に変更しない（理由：古い決済ロジックが動いていてテストがない）
```

このファイルを書いておくと、毎回「ここは触るな」「テスト書いて」と指示しなくていい。

「触ってはいけない場所」を書いておいたら、本当に触らなくなった。レガシーコードの地雷を踏まれなくなったのが個人的に一番ありがたかった。

---

## 業界調査でNo.1になったらしい

今年の3月、Pragmatic Engineerという海外のエンジニア向けニュースレターが約1,000人のエンジニアに調査した結果が出た。

「最も利用しているAIコーディングツール」の結果がこうだった。

| ツール | 支持率 |
| --- | --- |
| Claude Code | 46% |
| Cursor | 19% |
| GitHub Copilot | 9% |

リリースから8ヶ月でCursorとCopilotを抜いてNo.1になった。

特にスタートアップや小規模企業での採用が早くて、75%がメインツールとして使っているという数字も出ていた。

「そんな数字になってたのか」とこれを読んで少し驚いた。自分の周りでも使っている人が増えているとは思っていたが、ここまで一気に広まっていたとは思っていなかった。

---

## 正直、困ったこと

良いことばかり書くのも気持ち悪いので、困ったことも書く。

**自信満々に間違える。**

Claudeはコードを生成するとき、「こういう理由でこうしました」と説明してくる。この説明が割と自信に満ちている。で、たまに間違っている。説明が自信満々なので、読んでいるとなんとなく合っている気がしてくる。これが一番危ない。

レビューをちゃんとしていないと、もっともらしい間違ったコードがマージされる。

あとは**社内の独自ライブラリに弱い。**

有名なOSSやよく使われるフレームワークはかなり正確に扱ってくれる。でも社内独自のラッパーや、公開されていないAPIクライアントは当然知らない。知らないくせに補完してくることがあって、存在しないメソッドを平然と呼ぶコードが出てくることがある。

CLAUDE.mdに「このプロジェクトの主要なクラスはこれで、こういうインターフェースになっている」を書き足したらだいぶマシになったが、完全ではない。

---

## 結局どういう使い方が合っているか

3ヶ月使って落ち着いた使い方はこれだ。

**Claude Codeに向いているタスク：**

* テストコードの生成（正解がある程度決まっているので間違いに気づきやすい）
* リファクタリング（「この関数、責務が広すぎるので分割して」）
* ドキュメント・コメントの追加
* 型定義の補完
* 似たようなコードを大量に書く作業（CRUD系、シリアライザーなど）

**自分でやった方がいいタスク：**

* アーキテクチャの設計判断（「どうするか」を決めるのは自分）
* セキュリティに関わる部分のレビュー（最終確認は必ず目を通す）
* 社内ライブラリを絡めた複雑な実装

「コードを書く」部分はClaudeに任せて、「何を作るか・なぜそうするか」を自分が担う、という分担が一番うまく機能している気がする。

---

AIツールが出るたびに「エンジニアの仕事がなくなる」という話が出る。

3ヶ月使ってみた感想は、「自分が考える量は減っていない、むしろ増えた」だ。コードを書く時間が減った分、設計を考える時間・レビューする時間・何を作るかを決める時間が増えた。

それが良いことかどうかは、人によって違うかもしれない。コードを書くこと自体が好きな人にとっては、少しさびしい変化でもある。

でも生産性は、明らかに上がった。

---

*2026年3月時点の個人的な使用感です。アップデートが頻繁なので、数ヶ月後には状況が変わっている可能性があります。*
