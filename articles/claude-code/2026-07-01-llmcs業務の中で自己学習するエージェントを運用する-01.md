## はじめに

2026年前半時点の記録として、CS業務にLLMエージェントを組み込んで運用している構成を、残しておこうと思います。

## なぜエージェントに任せたかったか

CSの問い合わせ対応は、その多くが「調査」と「判断」の反復です。お問い合わせに至った理由をデータベースで確認し、過去の似たケースを思い出し、現在の仕様に照らして、顧客に返す文面へ落とす。一件ずつは小さくても、積み重なると相当な時間が「調べ直し」に溶けていきます。

さらに厄介なのは、その判断のノウハウが人の頭の中にしか残らないことです。担当が変われば、また同じ調査をゼロからやり直すことになる。

やりたかったのは、調査と一次判断をエージェントに任せ、人間は最終確認と難しいところだけを見る、という分担でした。そしてもう一つ、**対応するたびにナレッジが溜まっていってほしい**。この2つ目が、後述する自己学習ループの動機です。

---

## 設計の全体像

Claude Code を使用しています。

流れはこうです。

1. 顧客の問い合わせは Zendesk に入り、Slack に集約される
2. 人間が「調べさせたい」ものにリアクションを付けて選別する
3. 選ばれたものが調査キューに流れ、`/loop` で回る `poll-inquiries`スキル が古い順に1件だけ拾う
4. エージェントが Skills と Sub-agents を使い分け、BigQuery などのデータ源を調べて一次判断をつくる
5. 調査結果を元のスレッドに「対応提案」として返す
6. 人間が承認、または指摘する
7. 指摘された場合、`self-correct` が原因を調べ直し、ナレッジを直すPRまで作る

一本の直線ではなく、最後に **7 → ナレッジ → 次回の判断** というループになっているのが要点です。図でオレンジの太線にした部分がそれで、この記事のキモはそこにあります。

---

## ナレッジは「仕様」ではなく「判断ルール」で持つ

最初に迷ったのが、サービス仕様をすべてドキュメントに書き写してエージェントに読ませるべきか、という点でした。これは早々にやめました。

理由は簡単で、**仕様は変わるから**です。それをナレッジに写した瞬間、そのドキュメントは古くなった時点で誤った情報源になる。CSにとって、これは最も避けたい事態です。

そこで、ナレッジに残すのは仕様そのものではなく「**CS業務として何をするか・どう判断するか**」だけにしました。仕様の現在値が要るときは、その都度、実装リポジトリやデータを直接見に行かせる。ナレッジ側には、たとえば「アカウントが自動停止されたとき、どの順で何を確認し、どう切り分けるか」という手順と判断基準を置きます。

このナレッジ群を `docs/decisions`（判断ルール）と呼び、回答の正本にしています。ルールという形で持っているからこそ、あとで直せる。この一見地味な設計判断が、次の自己学習ループの土台になります。

---

## 人を挟む場所を絞る（HITL）

Human-in-the-Loop は聞こえはいいのですが、全工程に人を挟めば、それは自動化ではなく監視付きの手作業です。そうならないよう、人が介在する場所を意識的に2箇所に絞りました。

* **入口：調査対象の選別**　生の問い合わせを片端から流すのではなく、人が「調べる価値がある」と判断したものだけをリアクションで手渡す。ここでノイズを止めると、後段がかなり静かになります。
* **出口：回答の承認 / 指摘**　エージェントの調査結果は「提案」として返させ、そのまま顧客には飛ばさない。人が見て、良ければ承認、おかしければ指摘する。

逆に、その間の調査と一次判断は完全に任せます。ここを中途半端に覗きに行くと、待ち時間が増えるだけです。「入口で絞り、出口で見る、間は触らない」。この線引きが、運用してみていちばん効いた設計でした。

---

## 自己学習ループ

指摘を受けたら `self-correct` という専用スキルが起動し、次の順で動くようにしました。

1. なぜ間違えたのか、実データに当たって原因を特定する
2. それが「今回のデータ特有の話」か「判断ルールの誤り・不足」かを切り分ける
3. ルール側の問題なら、`docs/decisions` を直すPRを作る
4. 元のスレッドに訂正内容を返す

つまり、**人の一度の指摘が、次回以降の判断基準そのものを書き換える**。うまくいけば、同種の間違いは繰り返されなくなります。

エージェントの中身も、この流れに合わせて役割を分けています。Skills が「調査・判定・事例化」といった業務のまとまりごとの入口、Sub-agents がさらに専門化した実働部隊（データベースを掘る、仕様をコードで確認する、返信文を組み立てる、推敲する）です。

ここまで分けるのは、[以前の記事](https://zenn.dev/fivot/articles/fec94dcb2fb6a7)にも書いたとおり、LLMは一度に多くを背負わせると精度が落ちるからです。分割して受け渡すほうが、それぞれの品質が上がり、どこで間違えたかも特定しやすい。自己学習ループにとっては、この「どこで間違えたかを特定できる」ことが、そのまま「どのルールを直すか」の精度に効いてきます。

## おわりに

冒頭のとおり、これは新しい手法ではありません。人を挟み、フィードバックでナレッジを育てる、というだけの話です。

自分にとっての収穫は、それを概念から「毎日の業務ループの中で回り続ける形」まで落とし込めたことでした。指摘が判断ルールに還り、次の判断が少し賢くなる。このループはCSだけでなく、他の開発業務にも活かせそうです。
