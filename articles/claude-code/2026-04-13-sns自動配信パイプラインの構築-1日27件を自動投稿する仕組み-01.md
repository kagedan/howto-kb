## はじめに — なぜ1日27件なのか

僕は一人会社を経営している。プロダクトは9つ。SNSチャネルはX・LinkedIn・Facebookの3つ。単純計算で **9 × 3 = 27件/日** の投稿が必要になる。

これを手動でやるのは不可能だ。1件5分としても2時間以上かかる。しかも毎日。

結論から言うと、今は **朝の5分で全部門を把握し、承認ボタンを押すだけ** の状態になっている。この記事では、そこに至るまでの設計と失敗を共有する。

## 全体アーキテクチャ

パイプラインは3層構造になっている。

```
[コンテンツ生成層]
  Claude Code → プロダクト別投稿文を生成

[スケジューリング層]
  launchd (macOS) → 時間帯・曜日を分散配置

[配信層]
  Autonomos (Sales Cruise) → 各チャネルへ自動投稿
```

### なぜこの構成か

最初はcronを使っていた。しかし **macOSではcronがフルディスクアクセスの問題で全滅する** という罠にハマった。17個のジョブが一晩で全部止まっていたときの絶望感は忘れられない。

launchdに全面移行して、この問題は解決した。

## 設計の要点

### 1. 投稿の分散配置

9プロダクトの投稿を同じ時間に流すとスパム判定される。これは実際にやらかした。

対策として、**プロダクトごとに投稿時間と曜日をずらす** ルールを設けた。

```
プロダクトA: 月水金 8:00 / 12:00 / 18:00
プロダクトB: 火木土 9:00 / 13:00 / 19:00
プロダクトC: 月火水 10:00 / 14:00 / 20:00
...
```

各チャネルでも時間をずらす。Xは即時性が高いので朝、LinkedInはビジネスタイムの昼、Facebookは夕方にピークが来る。

### 2. コンテンツ生成の自動化

投稿文はClaude Codeで自動生成している。ポイントは **プロダクトごとのコンテキストファイル** を用意すること。

```
.company/products/{product-name}/
├── STATE.md          # 現在の状態
├── sns-templates.md  # 投稿テンプレート
└── target-audience.md # ターゲット情報
```

これをAIに渡すことで、プロダクトの特性に合った投稿文が生成される。テンプレートは以下のようなバリエーションを持たせている。

* 機能紹介型
* 課題提起型
* 実績・数字型
* ユーザーの声型

同じ内容の繰り返しを避けるため、過去の投稿履歴も参照させている。

### 3. 承認パイプライン

ここが一番重要だ。

初期は **AIの出力をそのまま自動送信** していた。結果、敬語がおかしい投稿や、文脈がずれた内容が流れた。取引先向けの投稿で不適切な表現が出たときは冷や汗をかいた。

今は **draft → 承認 → 実行** の3ステップを必ず通す。

```
1. AIが投稿文を生成 → draft状態で保存
2. 承認キューに追加
3. CEOが承認 → 自動投稿
```

「自動化は『AIの能力を制限する』ためではなく『安心して委任するための仕組み』」というのが僕の考え方だ。

### 4. bot検知対策

自動投稿で避けて通れないのがbot検知だ。当社では **18層のbot検知対策** を自社開発した。

詳細は書けないが、以下のレイヤーで対策している。

* ブラウザフィンガープリントの自然化
* 操作タイミングのランダム化
* セッション管理の最適化
* Playwrightベースの実装

## 失敗から学んだ5つの教訓

### 教訓1: cronを信じるな（macOSでは）

前述の通り。launchd一択。

### 教訓2: 同時投稿はスパム

複数プロダクトの投稿は必ず時間をずらす。最低30分は空ける。

### 教訓3: AIの出力を無検証で送るな

承認パイプラインは必須。特に対外コミュニケーション。

### 教訓4: レートリミットを甘く見るな

Zennで1日に複数記事と書籍を同時公開してデプロイが全滅した経験がある。プラットフォームごとのレートリミットを把握し、ロックファイルで制御する仕組みを入れた。

### 教訓5: 10事業は多すぎる

SaaSを4つ作ったが全部ユーザーゼロだった。AIは何でも作れるが、何でも売れるわけじゃない。3事業に集中してから、ようやく仕組みが回り始めた。

## 実装のポイント

### launchdのplist例

```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.company.sns-post-productA</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>/path/to/sns-post.sh</string>
    <string>productA</string>
    <string>twitter</string>
  </array>
  <key>StartCalendarInterval</key>
  <array>
    <dict>
      <key>Weekday</key>
      <integer>1</integer>
      <key>Hour</key>
      <integer>8</integer>
      <key>Minute</key>
      <integer>0</integer>
    </dict>
  </array>
</dict>
</plist>
```

### 投稿スクリプトの骨格

```
#!/bin/bash
PRODUCT=$1
CHANNEL=$2
LOCK_FILE="/tmp/sns-${PRODUCT}-${CHANNEL}.lock"

# 重複実行防止
if [ -f "$LOCK_FILE" ]; then
  echo "Already running. Skipping."
  exit 0
fi
touch "$LOCK_FILE"

# コンテンツ生成（Claude Code呼び出し）
CONTENT=$(claude --print "Generate SNS post for $PRODUCT on $CHANNEL" \
  --context ".company/products/$PRODUCT/sns-templates.md")

# 承認キューに追加
echo "$CONTENT" >> .company/approval-queue.md

# ロックファイル削除
rm "$LOCK_FILE"
```

実際のスクリプトはもっと複雑だが、骨格はこの通り。

## まとめ

1日27件のSNS自動投稿は、一見すると大げさに見える。しかし仕組みさえ作れば **月¥22,000で毎日自動で回る**。

ポイントは3つ。

1. **分散配置** — スパム判定を回避する
2. **承認パイプライン** — 安心して委任できる仕組み
3. **プラットフォーム別の制約把握** — レートリミット、bot検知への対策

「コストは10分の1になるが、売上が10倍になるわけではない。」これは事実だ。しかし、**仕組みの初期投資は大きくても、一度作れば自動で回る。** そこに価値がある。
