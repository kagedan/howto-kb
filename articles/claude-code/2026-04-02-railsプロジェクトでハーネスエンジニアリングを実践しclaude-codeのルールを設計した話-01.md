## はじめに

こんにちは、クラシルでレシチャレの開発をしているkiyokuroです。

今回は、RailsプロジェクトでClaude Codeのルールファイル群を設計・構築した取り組みについて紹介します。

Claude Codeを日常的に使って開発していてセッションごとに設計提案や実装スタイルのブレが気になっていました。例えばあるセッションではサービスクラスにキーワード引数を使う実装を提案し、別のセッションでは位置引数で書いてくる。同じプロジェクトなのに、Claudeの提案に一貫性がありませんでした。

この問題を解決するために「ハーネスエンジニアリング」の概念を調査し、実際にルールファイル群を構築しました。この記事では、その過程で得られた知見と試行錯誤について紹介します。

## ハーネスエンジニアリングとは

ハーネスエンジニアリングは、AIエージェントが正しく動ける「環境・制約・仕組み」を設計するエンジニアリングです。語源は馬具（ハーネス）。強力だけど暴走しうるAIを、手綱で正しい方向に導くイメージです。似た概念のプロンプトエンジニアリングやコンテキストエンジニアリングとの違いを整理すると、以下のようになります。

| 用語 | 対象 | やること |
| --- | --- | --- |
| プロンプトエンジニアリング | 入力（プロンプト） | AIへの指示を最適化 |
| コンテキストエンジニアリング | 文脈（情報の与え方） | AIに渡す情報の選別・構造化 |
| ハーネスエンジニアリング | 実行環境全体 | 制約・フィードバックループ・リンター・ドキュメント・ライフサイクル管理をまるごと設計 |

## Step 1: まずルール追加を支援するskillsを作った

いざ`.claude/rules/`にルールを書こうとすると、最初に困ったのは「どのようなファイルを作って、何を書けばいいのかわからない」ということでした。

そこでまず取り組んだのが、ルールの記入を支援するskillsの作成です。Anthropicの公式ドキュメント（Best Practices for Claude Code）にある判断基準をskillsに組み込みました。

> Keep it concise. For each line, ask: "Would removing this cause Claude to make mistakes?" If not, cut it.  
> （簡潔に保つこと。各行について「この行を削除したらClaudeが間違えるか？」と問いかける。Noなら削除する。）

この基準をskillsに埋め込んでおくことで、「このルールを書くべきか？」の判断をClaude自身に委ねられるようになりました。ルールを直接書き始めるのではなく、ルールを書くための仕組みから作るというアプローチです。

実際に作成したskillsの骨格は以下のようなものです。

.claude/skills/add-rule/SKILL.md

```
# ルール追加スキル

## Phase 1: 追加判断
### 追加すべき（Claudeが間違える情報）
- プロジェクト固有の規約でコードから推測できないもの
- 複数のパターンが混在していて「どちらに合わせるべきか」の判断基準
- インフラ・運用の知識（Terraform連携、ECSサービス分割等）
- 外部システムとの連携ルール（別リポジトリ、マイクロサービス）

### 追加不要（コードを読めばわかる情報）
- Claudeが既存コードを2-3ファイル読めば発見できるパターン
- RubyやRailsの標準的なベストプラクティス

### 判断の問いかけ
「この行を削除したらClaudeが間違えるか？」→ Noなら追加しない

## Phase 2: 記載場所の選定
## Phase 3: 既存ルールとの整合性チェック
- 全ルールファイルを確認し、新規ルールと意味的に矛盾する記述がないかチェック
- 矛盾が見つかった場合はユーザーに報告し、どちらを優先するか確認する

## Phase 4: ルールの記述
## Phase 5: 確認と適用
```

Phase 1の「追加すべき/追加不要」の判断基準がポイントです。また、Phase 3では新しいルールを追加する際に既存の全ルールファイルと矛盾がないか確認するフローを入れています。この矛盾チェックが重要だった話はStep 4で詳しく触れます。

## Step 2: エージェントチームでコードベースを分析した

skillsができたので、次はコードベースの調査です。

対象のRailsプロジェクトはサービス・モデル・コントローラーだけで数百ファイル規模あります。人の目で全容を把握するのは現実的ではありません。Claude Codeのエージェントチームを使い、各レイヤーを並列に調査しました。

* コントローラー層: ルーティング規約、パラメータの受け渡しパターン
* サービス層: クラス設計、引数パターン、エラーハンドリング
* ジョブ/バッチ層: 非同期処理の実装パターン
* テスト層: テストの書き方の慣習、ファクトリの使い方

AIにコードベースの特徴を分析させた後、Step 1で作ったskillsを使って「この特徴はルールとして記入すべきか？」の判断もAIに行わせました。こうして最初の叩き台ができました。

### AIのブレは、自分たちのコードのブレだった

分析結果で印象的だったのが、サービスクラスの引数パターンです。

キーワード引数と位置引数がほぼ半々に割れていました。

ここで気づいたのは、Claudeの提案がセッションごとにブレていたのは、そもそも人間の書き方がブレていたからだということです。もちろんこれは長期間サービスを開発していく中で徐々に開発チーム内で方針が変化していった結果だと思います。Claudeは既存コードを参考にして提案するため、既存コード自体が一貫していなければ、AIの出力も当然ブレます。

ルール構築は「AIを矯正する」作業だと思っていましたが、実際には明文化されてないようなチームのコーディング規約を明文化する作業でもありました。

## Step 3: 1つのリポジトリだけでは足りなかった

叩き台ができた時点で、あることに気づきました。AIがコードベースを分析して作成したルールは、1つのRailsリポジトリの中で完結する情報だけでした。

しかし、実際の業務はリポジトリをまたいで行われます。

例えば、Railsサービス内で特定の処理を行う場合、社内の別マイクロサービスにAPIリクエストを送る必要があるケースがあります。仕様を正確に把握するにはそのマイクロサービスのリポジトリも参照しなければなりません。

また、新しくDynamoDBテーブルを追加したいなどの新規でAWSのリソースを使いたい場合は、まずTerraformのリポジトリでリソースを作成し、その後Railsのリポジトリで設定ファイルへの追記やモデルの実装を行うという複数ステップの手順が必要です。

このようなリポジトリをまたいだ関係性や実装手順は、AIが1つのRailsリポジトリを見ただけでは知ることができません。これはまさに「削除したらClaudeが間違える」類の情報であり、人間が明示的にルールとして書かなければならない部分でした。

実際にルールとして書いた内容の一部を紹介します。

.claude/rules/{project}/index.md

```
## マイクロサービス連携
- 特定のドメイン操作は全て専用の名前空間のクラス経由で行う。
  ロジックをRailsアプリ内に直接実装しない

## データストア追加手順
新しいデータストアを追加する場合は以下の全てが必要:
  1. IaCリポジトリでリソースを作成
  2. Railsリポジトリの設定ファイルにテーブル設定を追加
  3. app/models/ にモデルを追加
  4. 環境変数をECSのタスク定義に追加

## 非同期ジョブ
- コンテナはSpotインスタンスで稼働しており、意図しないタスク終了があり得る。
  非同期処理は冪等かつ短時間で完了するよう設計する
```

マイクロサービスの連携ルール、インフラリソースの作成手順、Spotインスタンスの制約。いずれもコードを読むだけでは知ることができない、運用経験から得られた知識です。

こうした情報を手動で追記していくことで、現在運用中のルールが完成しました。最終的なファイル構成は以下の通りです。

```
.claude/
├── rules/
│   └── {project}/
│       ├── index.md          # 全体ルール・チェックリスト
│       ├── services.md       # Service層ルール
│       ├── controllers.md    # Controller層ルール
│       ├── models.md         # Model層ルール
│       ├── spec.md           # テスト層ルール
│       └── migration.md      # マイグレーションルール
└── skills/
    └── add-rule/
        └── SKILL.md          # ルール追加スキル（Step 1で作ったもの）
```

各ルールファイルには`globs`フロントマターでスコープを設定し、該当するファイルを編集するときだけ読み込まれるようにしています。

## Step 4: 不要なルールを削る

運用を始めてすぐに、ルールが過剰だった部分も見えてきました。

AIにコードを分析させて作成した叩き台には、例えばRSpecの`let`の書き方について以下のようなルールが含まれていました。

* DB保存が必要で他のlet/beforeから参照されるデータは `let!` を使う
* DB保存不要な値や計算結果は `let` でもよい
* 順序依存がある場合は必ず `let!` を使う

一見すると丁寧なルールに見えます。しかし、RSpecでは遅延評価で済むものには`let`を使い、即時評価が必要なものには`let!`を使うのが一般的な慣習です。Claudeはこの慣習を既に知っています。

「この行を削除したらClaudeが間違えるか？」── No。Claudeは元々正しく書ける。

このルールは運用開始後すぐに削除しました。RailsやRSpecの一般的な記法はClaudeが既に理解しているので、社内で特別な利用方法が必要なものだけを残すようにしています。

### ルール間の矛盾にも注意が必要だった

不要なルールの削除と合わせて、ルール間の矛盾のチェックも重要でした。ルールを書き足していくと、既存ルールと矛盾する記述が紛れ込むことがあります。矛盾するルールがあると、Claudeは任意にどちらかを選択してしまいます。どちらが選ばれるかは予測できないため、結果がセッションごとにブレる原因になります。

この問題を防ぐために、Step 1で作ったskillsのPhase 3に「既存ルールとの整合性チェック」を組み込んでいます。新しいルールを追加する際には必ず既存の全ルールファイルとの矛盾がないか確認するフローです。

## まとめ

やってみて感じたのは、ルール設計は「AIに何を教えるか」ではなく「AIが知り得ないことは何か」を考える作業だということです。コードの書き方やフレームワークの慣習はAIが既に理解しています。人間が書くべきなのは、リポジトリの境界を超えた関係性、運用経験からしか得られない制約、そしてチーム内で暗黙的に共有されていた規約です。

そして、AIの出力がブレる原因を調べたら人間のコードがブレていたという発見は、ルール構築の副産物としてチームの規約を明文化するきっかけにもなりました。

現時点ではrulesの整備が中心で、ハーネスエンジニアリングとしてはまだ入口に立ったところです。hooksを使った自動チェックの仕組みや、MCPによる外部ツール連携など、Claudeがより正確に動くための設定はまだまだ余地があります。引き続きアップデートしていきます。
