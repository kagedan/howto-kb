# ターニングポイント

私が書いてきた記事の中で屈指の有用記事だと思っている、「2025年5月22日、私が生成AIとともに生き残るために考えていること」の公開から1年が経過しました。

<https://zenn.dev/tkithrta/articles/9a6f1f2d249790>

2025年5月22日はちょうどAnthropicからClaude Sonnet 4とClaude Opus 4の発表があった日で、この頃からClineで使用するAPIの使用量をプランニングしたり、VS CodeのLanguage Model API経由でGitHub CopilotからClaude 3.5 Sonnetを使う必要がなくなり、Claude CodeをインストールしてClaude Maxの定額プランに契約さえしてしまえば全てが解決する時代に突入したことをよく覚えています。

あれから1年経ち、LLMはAGI（汎用人工知能）と呼べるほどの進化を遂げ、AIエージェントはシンギュラリティを実現するかのような振る舞いをするようになりました。

私たちがギュられないようにするためにはどうすればいいのでしょうか？

> 我々が幸せになるために必要な資源は、実はかなり少ない。今の1％以下じゃないかと思うくらいだよ。多くの不必要な活動が、忙しさや環境破壊の元凶になっている  
> [Fireside chat with Google co-founders, Larry Page and Sergey Brin with Vinod Khosla - YouTube](https://www.youtube.com/watch?v=Wdnp_7atZ0M) - Larry Page (Google)
>
> 日本語訳は[グーグル創業者が語る働き方の未来 「もう必死に働かなくて良いんじゃない？」 | 企業ニュース | キャリコネ 口コミ・評判の情報サイト](https://careerconnection.jp/company_news/buttyake/3276)より引用

これは何度でも声に出して読みたい日本語なのですが、AIは仕事を減らすためにあってほしいので、AIを使うために仕事が増えるのは避けたいところです。

そんな思いをこめて書いた、1年前の内容を引用しながらコメントしつつ振り返っていきたいと思います。

# 基礎教養を振り返る

> まず、江戸時代から続く基礎教養「読み」「書き」「そろばん（計算）」、コンピュータ利用における基礎教養「ITリテラシー」「メディアリテラシー」「セキュリティ意識」みたいないわゆる三種の神器を生成AIの活用において定義したいと思います。
>
> * Vocabulary (ボキャブラリー / 専門用語)
> * Context file (コンテキストファイル / 文脈ファイル)
> * Schema (JSON Schema / スキーマ)

これは今も変わっていませんね。  
詳しくは去年の生成AIアドベントカレンダーで書いているのでこちらをご参照ください。

<https://zenn.dev/tkithrta/articles/e4379c440c3841>

当時はまだプロンプトエンジニアリングが主流で、2025年6月頃からコンテキストエンジニアリングがバズワードになったのですが、1年前にこの3つを定義した時はまさかコンテキストファイルがここまで重要な位置づけになるとは思っていませんでした。

まあ最近はコンテキストファイルを使っているのかもよく分からない、ハーネスエンジニアリングとかいうものが流行っているみたいですが……。

> そもそも生成AIはAIや検索とどう違うのかも定義したいと思います。
>
> * TDM (テキストデータマイニング。情報抽出の自動化プロセス全般)
> * AI (AIはテキストデータマイニングの一種。機械学習システムの訓練・利用)
> * GAI / GenAI (生成AIはAIの一種。コンテンツ生成可能な機械学習の訓練・利用)
> * Search (検索はテキストデータマイニングの一種。検索インデックス生成・検索アプリケーション利用)

生成AIが驚くほど普及したのは検索の発展形として使う人がいたからだと思うんですよね。

1年前からGoogle検索した時にAIが回答する仕組みはありましたが、性能の低いモデルを使用していたのでGoogleの評価を下げる原因になっていました。  
最近はGemini 3.5 Flashを使用しているのでかなり性能が上がったように思えます。

<https://blog.google/intl/ja-jp/products/explore-get-answers/search-io-2026/>

また当時この中に推論（Infer / Inference）を加えるべきか悩んだのですが、確か推論がテキストデータマイニング（あるいはRAG）の一種なのか曖昧だったので加えることをやめた記憶があります。

> 2025年5月22日現在、最も使われているLLMは以下の三種です。
>
> * ChatGPT (OpenAI)
> * Gemini (Google)
> * Claude (Anthropic)

これは2026年5月22日現在も変わっていませんね。

> これらのLLMと開発企業は以下クラウドサービスと密接な関係性があります。
>
> * Azure / Microsoft Azure (OpenAIのGPT)
> * GCP (GoogleのGemini)
> * AWS (AnthropicのClaude)

今はMicrosoftとOpenAIの間で独占契約がなくなったので、少し変わったかもしれません。

# 生成AIを振り返る

> 生成AIの生成において最も使われるフォーマットは以下のとおりです。

この3つはなかなかバランスいい選択だったなと、今となっては思います。

> JSONだけでは表現できないケースもあるので、以下フォーマットを使います。

確かにJSONは便利ですが、YAMLやTOMLを使うことでJSONで解決できない問題を解決できるので、これらはメリットを活かして採用するのが一番ですね。

> Markdownだけでは表現できないケースもあるので、以下フォーマットも使います。

この1年はいかに人間とAIが理解しやすいFrontmatterのメタデータを作っていくかといった年だったように思えます。

Obsidianを用いたMarkdown管理、Agent Skillsを発動できるAIエージェント、Claude Codeのコンテキストエンジニアリングなどなど……。

Mermaidもまあ便利ですが、生成AIが生成したMermaidをうまくパースできなかったり、Mermaidそのものが重かったりするのがちょっと気になるので、いい感じに発展してくれたらいいんですけどね。

> 生成AIにおいて以下の要素はとても重要です。
>
> * Heading (見出し)
> * Seriation (系列化 / 分類)
>
> 例えば見出しであればMarkdownで以下の三段階に分けることができます。
>
> 他にも分類であればMarkdownで以下の三段階に分けることができます。

これめっちゃ重要ですね。  
私も1年間コーディングエージェントを効率よく使うにあたりやったことは、ドキュメントや仕様駆動開発（SPEC.md）で使用しているMarkdownの見出しと分類を構築するにあたり、上記ルールを守るように徹底しただけです。

1つ付け加えるとするなら、最近はTODO.mdでタスク管理やリスク管理を行うようになったので、以下のようなチェックボックス記法を使うようになったぐらいです。

```
- [x] これは完了したタスクです。
- [ ] これは未完了のタスクです。
```

Obsidianでも検索できるので便利です。

> また、以下内容を理解しておくと管理が楽になります。

PDFをコンテキストファイルとして追加する手法はGoogleのNotebookLMで主流になりましたね。  
Base64もindex.htmlだけでアプリを作る時とかに便利なのですが、言語化が難しい……。

> Gitは使いづらいので、以下VCSも視野に入れておきます。

後述するコーディングエージェントですらGitを正しく使うことができないのは結構問題で、Jujutsuがたまに流行ったり、GitのワークフローをハックするEntireというサービスが生まれたりしました。

<https://entire.io/>

# AIコーディングエージェントを振り返る

> エージェントとは何でしょうか？  
> John F. Sowa著"Knowledge Representation"において、エージェントとは、意図的に何かをすることができる生体と定義しています。
>
> <https://www.jfsowa.com/krbook/>

これは2025年6月に追記した内容ですが、John F. Sowa著"Knowledge Representation"も私にとって結構大きなターニングポイントだったので、気になる方は読んでみるといいかもしれません。

> 生成AIでコーディングしたり、コーディングエージェントを使いこなすには以下roadmap.shの内容を理解しておく必要があります。  
> <https://roadmap.sh/frontend?r=frontend-beginner>  
> <https://roadmap.sh/backend?r=backend-beginner>  
> <https://roadmap.sh/devops?r=devops-beginner>  
> <https://roadmap.sh/git-github?r=git-github-beginner>

コーディングエージェントを使いこなせるエンジニアと非エンジニアの違いはこういうボキャブラリーを知っているかどうかだと思っているので、roadmap.shのこの4つだけでも覚えておくといいです。

> したがって、2025年5月22日現在は以下のプログラミング言語を学んでおくとアジリティの高い開発が見込めます。
>
> * HTML (XML)
> * CSS
> * JavaScript (TypeScript)
> * Python
> * Go

2026年5月22日現在もそうですね。

ただJavaScript, Python, Goどれを使うべきかという問題はとても難しいです。

私は最近ほぼPythonしか使わなくなったので、経緯とあわせてコメントしていきたいと思います。

> Node.jsとnpmでJavaScriptやTypeScriptの開発環境を構築するのは非常に大変でしたが、最近は以下ランタイム、ツールのおかげで楽になりました。

まずJavaScript（TypeScript）を使わなくなった最も大きな理由としては、ランタイムとツールチェーンが多すぎるところにあります。

<https://zenn.dev/tkithrta/scraps/4050f601e2d061>

おそらく今後もっと増えると思っていて、そうなってくると冒頭で述べたようなAIを使うために仕事が増える本末転倒な状況になってしまうので、距離を置いている次第です（もちろんHTML + CSS + JavaScriptの組み合わせを使うことはありますが）。

Goはクロスコンパイル（例えば開発環境のLinuxからWindowsで使えるツールをgo build）した時にWindows Defenderに誤検知されることが多発するようになったので使わなくなりました（もちろんDebianやUbuntuで動かすWebアプリケーションで使うことはあります）。

このように消去法で選択していった結果、Pythonしか使わなくなったわけです。

> pipとrequirements.txtでPythonの開発環境を構築するのは非常に大変でしたが、最近は以下ツールのおかげで楽になりました。

最近はtyという型チェッカーも使っています。  
uv, Ruff, tyを作っているAstralはOpenAIに買収される程度にはすごいので、皆さんもこれらのツールチェーンだけでPythonをやっていきましょう。

<https://docs.astral.sh/uv/>  
<https://docs.astral.sh/ruff/>  
<https://docs.astral.sh/ty/>

> もちろん以下のプログラミング言語、Webアプリケーションフレームワークも学んでおいて損はないでしょう。
>
> * PHP (Laravelなど)
> * C# (F#, .NET)
> * Ruby (Ruby on Railsなど)
> * Java (Kotlin, Scala, Spring Bootなど)
> * Rust
> * SQL (SQLite, PostgreSQL, MySQL)

ここらへんのプログラミング言語はコーディングエージェントでフルスタックWebアプリケーションを作ったり、運用・保守の仕事でなくならないと思うので、学んでおいて損はないと思います。

> シェルスクリプトも学んでおくといいでしょう。以下のようなものがあります。
>
> * Bash
> * Zsh
> * Fish
> * PowerShell
> * Nushell

シェルはコーディングエージェントのログインシェルとして残ると思われるので、学んでおいて損はないと思います。

ただシェルスクリプトはコーディングエージェントと相性が悪すぎることがこの1年で分かったので、1年前に書いた内容の中で唯一書いたことを後悔した内容です。  
FishとかNushellとか使っている人全然見なくなりましたね。

> また、コーディングエージェントはGitとGitHubに強く依存していることが多く、コードレビューも必要になるため、以下の内容について理解しておく必要があります。
>
> * GitHub Copilot
> * GitHub Codespaces (VS Code)
> * GItHub CLI

特にGitHub Codespacesのようなサンドボックス環境はこの1年で非常に需要が高いものになりました。

ただ最近のGitHubはものすごく不安定で、今後も無料のサービスを維持できるのか不安になることがたまにあります。  
GitHub Copilotのプランも2026年6月に変わる予定ですからね。

> 主に使用するGItHub CLIコマンドは以下のとおりです。
>
> * gh auth
> * gh browse
> * gh codespace (gh cs)
> * gh gist
> * gh issue
> * gh org
> * gh pr
> * gh project
> * gh release
> * gh repo

コーディングエージェントが自動的にIssueやPRを処理したり、コミットメッセージやリリースノートを書くことも珍しくなくなりました。

> コーディングエージェントへ以下Gitコマンドの内容を理解させることはとても重要です。
>
> * git grep
> * git diff
> * git show
> * git log -p
> * git stash show -p
> * git reflog -p
> * git add -p
> * git worktree
> * git sparse-checkout

コーディングエージェントにコンテキストを渡すうえで重要なGitコマンドですね。  
git worktreeでコーディングエージェントを並列稼働させたり、git sparse-checkoutで必要なディレクトリだけをチェックアウトしたりできることに気づいたので、この2つは2025年6月に追記した内容です。

> あとは以下IPA試験について学んでおくとAIや生成AIを使わなくても役に立つのでお得です。
>
> * 情報セキュリティマネジメント試験
> * 基本情報技術者試験
> * 応用情報技術者試験

応用情報技術者試験なくなるらしいな？

<https://www.ipa.go.jp/shiken/syllabus/henkou/2025/20260331.html>

なので今後は情報セキュリティマネジメント試験、データマネジメント試験（仮称）、基本情報技術者試験あたりを学んでおくといいかもしれません。

特に最近はセキュリティや倫理に関する問題が生成AIによって増えまくっているので、ITパスポートすっ飛ばして情報セキュリティマネジメント試験について学ぶことをオススメします！

# 画像生成AIを振り返る

> 生成AIの画像生成において最も使われるフォーマットは以下のとおりです。
>
> PNGやJPEGだけでは表現できないケースもあるので、以下フォーマットを使います。
>
> PNGやJPEGより優れた以下フォーマットもあります。

ここらへんは鉄板ですね。  
1年で生成AIが生成するSVGのクオリティが結構上がってきたので、Marmaidかわりに使えて便利だったりします。

> 最近はChatGPT, Gemini, Grok（xAI）で簡単に画像生成できますが、pixivで見るようなイラストを生成したいのであれば、以下SDXLモデルがベースになっているモデルをHugging FaceやCivitaiで探してみましょう。
>
> * Animagine (Cagliostro Research Lab)
> * Illustrious (OnomaAI)
> * Pony (Stable Diffusion API)

この辺は1年前とかなり状況が変わっていて、ChatGPT（ChatGPT Images）やGemini（Nano Banana）が生成する画像がGrok（xAI）やSDXLモデルで生成する画像とは比べ物にならないほどクオリティが上がっていたり、プロンプトを理解できるようになってきています。

なので私も最近はSDXLモデルをHugging FaceやCivitaiで探してDiffusersでぶん回すみたいなことをしなくなってきていて、ChatGPT ImagesやNano Bananaばかり使うようになりました。

また、最近はSDXLモデルの性能が頭打ちになってきたので、Animaモデルというアニメ寄りのイラストやスタイルに強いモデルを新たに作る動きが出てきたりしています。

<https://ascii.jp/elem/000/004/375/4375749/>

> Stable DiffusionやNovelAIのような画像生成AIを使っていると、ものすごい数の画像が生成されるため、以下内容を理解しておくと管理が楽になります。
>
> 例えば7-Zipでは主に以下アーカイブと圧縮形式がサポートされています。
>
> * 7z
> * BZIP2
> * GZIP
> * TAR
> * WIM
> * XZ
> * ZIP
> * ZSTD
>
> 他にも7-Zipでは主に以下ハッシュ関数とチェックサムがサポートされています。
>
> * CRC32
> * CRC64
> * SHA1
> * SHA256
> * BLAKE2sp
> * XXH64

アーカイブと圧縮、ハッシュとチェックサムは画像生成AIだけでなく、AIエージェントやコーディングエージェントを効率よく使うために応用されるようになりました。

例えば私は2026年5月22日現在ChatGPT PlusのProjectsだけでプログラミング言語を作っているのですが、これはChatGPT Plusで使えるAdvanced Data Analysis（Code Interpreter）のPythonがZIPやTAR、GZIPを使ってユーザーに生成物をまとめたファイルで提供できるようになったり、大量のファイルの変更状況をCRC32やSHA256でチェックできるようになったことで実現できたことです。

<https://zenn.dev/tkithrta/articles/2b0134084a45d6>

アーカイブと圧縮、ハッシュとチェックサムをAIエージェントやコーディングエージェントに任せる分野はまだまだ発展の余地があると思っているので、今後も注目していきたいと思っています。
