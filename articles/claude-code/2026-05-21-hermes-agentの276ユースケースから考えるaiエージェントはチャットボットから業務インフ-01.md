## 開発領域：AIが「コードを書く」から「開発フローを持つ」へ

元投稿で最初に強調されていたのは、開発・エージェント運用の領域です。

具体例として、次のようなものが挙げられていました。

* バックエンド監視や強化学習環境のために、12個のHermesエージェントを並列実行する
* plan → code → QA → deploy の自律パイプラインを作る
* 自己進化するコーディングエージェントを作る
* リポジトリを監視するwatchdog agentを動かす
* 複数のコーディングエージェントを束ねるswarmを作る
* 夜間にコード品質をチェックするbotを動かす
* リモートのコーディングfleetを運用する
* コードベースの永続的な記憶を持たせる
* エージェントのobservabilityやアーキテクチャ健全性チェックを行う

ここで面白いのは、単に「AIがコードを書く」ではないことです。

コード生成だけなら、すでに多くのツールがあります。Hermes Agent的な文脈で重要なのは、コードを書く前後の流れも含めて扱っている点です。

```
依頼を受ける
↓
リポジトリを読む
↓
計画する
↓
実装する
↓
検証する
↓
PRやレポートとして残す
↓
次回のために手順や記憶を保存する
```

この流れまで含めると、AIは単なる補完機能ではなく、開発プロセスの一部になります。

ただし、ここで注意したいのは「完全自動で本番デプロイまで任せる」ことを最初の目標にしないことです。最初は、PR作成、検証ログの整理、レビュー前セルフチェック、Issue分解あたりから始める方が安全です。

## 個人利用・生活支援：生活導線に入るAI

生活に近い用途としては、次のような使い方も紹介されていました。

* 家族全体で1つのHermesインスタンスを共有する
* 毎朝のInbox要約を作る
* AIが寝る前の物語を生成する
* 食事記録や献立メモの補助に使う
* 生活習慣の記録補助を行う
* 折りたたみスマホ上に生活支援用途のAIを置く
* ホームオートメーションを制御する
* 長期的な文脈を扱う生活支援AIを作る
* 毎日の能動的なチェックインを行う

長期的な文脈を扱うAIでは、利便性だけでなく、保存する情報の範囲、削除可能性、第三者情報の扱いを設計する必要があります。

ただし、個人利用・生活支援の領域は機微な情報に触れやすいです。メール、家族、健康、予定、住居、デバイス制御などは、便利さとリスクが近い場所にあります。医療・健康上の判断はAIに委ねず、必要に応じて専門家に確認する前提です。

実運用では、次の線引きが必要です。

* どの情報を長期記憶に残すか
* どの情報はセッション内だけで扱うか
* 外部送信してよい内容は何か
* 家族や第三者の情報をどう扱うか
* デバイス制御はどこまで自動化するか

常駐するAIほど、忘れないことだけでなく、覚えすぎないことも大事になります。

```
↓

```

これが完全自動で動いている例として紹介されています。

コンテンツ制作は、Hermes Agentのような常駐エージェントと相性がよい領域です。理由は、成果物がファイルとして残り、レビューしやすく、公開前に止めやすいからです。

たとえば、記事制作なら次のようにできます。

```
1. 情報収集だけをAIに任せる
2. 構成案を作らせる
3. 下書きを作らせる
4. 公開リスクをチェックさせる
5. 人間が最終確認する
6. 承認後だけ公開する
```

ここで大事なのは、公開まで自動化しないことです。AIは速いですが、速く間違えることもあります。下書きで止める、レビュー観点を分ける、公開フラグを人間が変更する、という設計にしておくと、速度と安全性を両立しやすくなります。

## Hermes Agentを業務に入れるときの読み替え

元投稿は、かなり刺激的な事例が並んでいます。

ただ、実務で読むときは「すごい事例を真似する」よりも、次の4段階に分けると安全です。

```
1. 補助
   人間が都度頼み、AIが調査・要約・下書きを返す

2. 半自動
   手順化された作業をAIが実行し、成果物をファイルやPRに残す

3. 常駐
   cronやWebhookでAIが自分から動き、定期レポートや監視を行う

4. 組織化
   複数エージェント、人間レビュー、権限分離、監査ログを含む業務フローになる
```

最初から4を狙うと、だいたい複雑になりすぎます。まずは1か2から始めるのがよいです。

たとえば、最初の一歩としては次のようなものが向いています。

* リポジトリを読ませてREADME改善案を出す
* 作業ログから記事の下書きを作る
* PR本文を作る
* 差分をレビュー観点ごとにチェックする
* 毎朝のニュースや論文を要約する
* よく使う手順をSkill化する

これらは、失敗してもリスクが比較的小さく、成果物を人間が確認しやすいです。

## Hermes Agentの主要機能を実務目線で見る

Hermes Agentを使うとき、最初に押さえるとよいのは次の機能です。

| 機能 | 実務での意味 |
| --- | --- |
| CLI | まずはローカルで単発実行し、成果物を確認する入口 |
| Messaging Gateway | Discord、Telegram、Slackなどから依頼・報告できる入口 |
| Toolset | ファイル、ターミナル、ブラウザ、画像、音声などを扱うための道具 |
| Memory | ユーザー設定や恒久的な前提を覚える長期メモ |
| Skill | 成功した手順や運用ルールを再利用する作業手順書 |
| cron | 定期実行や継続監視を行う仕組み |
| MCP | 外部ツールや社内システムを扱うための接続規格 |
| Provider切り替え | タスクやコストに応じて利用モデルを切り替える仕組み |

ここでいうMCPは外部ツールをAIから扱うための接続規格、cronは決まった時間に処理を起動する定期実行の仕組みです。

この中で、最初に効くのはCLI、ファイル操作、Skillです。

いきなり全機能をつなぐより、まずはローカルで小さな成果物を作らせ、その手順をSkill化し、次にMessaging Gatewayやcronへ広げる方が運用しやすいです。

## すぐ試せる小さいユースケース

### 1. 作業ログから記事下書きを作る

```
logs/2026-05-20.md を読んで、公開してよいAI活用の学びを1つ選び、
Zenn向けに、課題、やったこと、詰まった点、再現手順、注意点をまとめる。
内部情報は抽象化し、published: false の下書きとして保存する。
```

成功条件は、次のように明確にしておきます。

* `articles/` 配下にMarkdownがある
* front matterに `published: false` がある
* 内部URLや機密情報が含まれていない
* 人間が読んでから公開判断できる

### 2. PRレビュー前のセルフチェックを任せる

```
この差分を、技術正確性、セキュリティ、初心者再現性、文章の自然さ、事業導線の観点でレビューして。
致命的な指摘と、公開前に直すべき指摘を分けて。
```

「レビューして」だけではなく、観点を分けるのがポイントです。

### 3. Discordから小さな依頼を投げる

最初は、対象チャンネルを絞り、明示メンション時だけ反応させるのがおすすめです。

```
このIssueを読んで、実装前に確認すべき前提と受け入れ条件をまとめて。
```

このような依頼なら、AIが勝手に外部送信したり公開したりするリスクが低く、日常業務に入れやすいです。

### 4. 繰り返し作業をSkillにする

同じ説明を2回以上したら、Skill候補です。

* Zenn記事作成フロー
* PR作成フロー
* レビュー観点
* 特定リポジトリの検証コマンド
* Discord連携のトラブルシュート

Skillは、AIに読ませる社内Wikiのようなものです。人間が読んでも分かるように書きつつ、AIが次回そのまま実行できる粒度にするのがコツです。

## 失敗しやすいポイント

### 1. いきなり全部自動化する

元投稿の事例は派手です。だからこそ、最初から自律企業や複数エージェント組織を作りたくなります。

しかし、実務ではまず小さく始める方が強いです。

```
下書きを作る
レビューする
人間が承認する
成功した手順をSkillにする
```

このループを先に作ると、後から自動化しやすくなります。

### 2. 権限を広げすぎる

AIにファイル、ブラウザ、GitHub、メール、チャット、外部APIを渡すと、一気に便利になります。

同時に、一気に危なくなります。

最初は、読み取り専用、対象範囲限定、公開前停止、人間承認を基本にした方がよいです。

### 3. Memoryに何でも入れる

Memoryは便利ですが、作業ログ置き場ではありません。

入れるべきなのは、次回以降も効く情報です。

* ユーザーの好み
* よく使う環境
* プロジェクトの恒久的な前提
* 繰り返し使う運用ルール

一時的な進捗や今日だけのTODOは、Issue、PR、作業ログに残す方が扱いやすいです。

### 4. AIレビューを承認と勘違いする

AIレビューは便利です。でも、最終承認ではありません。

公開記事、法務、セキュリティ、営業資料、採用判断、顧客対応などは、人間の確認を残すべきです。

## まとめ

元投稿の276ユースケースは、かなり未来感があります。

ただ、そこで語られている方向性はシンプルです。

AIは、質問に答えるだけのチャットボットから、記憶を持ち、道具を使い、スケジュールで動き、外部サービスとつながり、成果物を残す存在になりつつあります。

元投稿の最後では、次のような流れで締められていました。

```
2024: AI answered questions.
2025: AI helped humans work.
2026: People started building AI systems that work instead of them.
```

日本語にすると、こんな感じです。

```
2024年: AIは質問に答えた
2025年: AIは人間の仕事を手伝った
2026年: 人々は、人間の代わりに働くAIシステムを作り始めた
```

もちろん、すべてをAIに任せればよいわけではありません。

実務では、むしろ次の設計が重要です。

```
AIに速度を任せる。
仕組みに安全を任せる。
人間に判断を残す。
```

Hermes Agentは、この設計を作るための土台になり得ます。

まずは小さく、下書き、レビュー、調査、PR補助、定期レポートのような低リスクなユースケースから始める。うまくいった手順をSkillにし、必要な前提だけMemoryに残し、徐々にGatewayやcronへ広げる。

元投稿の派手な事例をそのまま追いかけるより、この順番で進める方が、長く使えるAIエージェント運用に近づくはずです。
