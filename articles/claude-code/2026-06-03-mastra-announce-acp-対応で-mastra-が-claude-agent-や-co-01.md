## Mastra が ACP 対応を発表

2026年6月2日、Mastra は **Agent Client Protocol（ACP）対応** を発表しました。

今回の発表をひとことで言うと、**Mastra のエージェントが、既存のコーディングエージェント実行基盤を丸ごと借りられるようになった**という話です。

公式アナウンスでは、次のような ACP 互換ハーネスが例として挙げられています。

* Claude Agent
* Codex CLI
* Cursor
* Gemini CLI
* そのほかの ACP 互換ハーネス

必要バージョンは次の通りです。

* `@mastra/core@1.34.0` 以降
* `@mastra/acp` の追加インストール

単に「外部プロセスを呼べる」ようになったのではありません。Mastra が supervisor として計画・委譲を担当し、外部のコーディングハーネスに対して **コードベース読解、ファイル編集、シェル実行、テストのやり直し、Git 操作** といった“コーディングエージェントらしい仕事”をまとめて任せられるようになった、というのが本質です。

---

## 背景: なぜ今 ACP なのか

2026年前後のエージェント界隈では、「LLM を呼ぶだけのアプリ」から「実際に作業を進めるエージェント」へのシフトが急速に進んでいます。特にコーディング領域では、その変化がとても分かりやすいです。

以前なら、アプリ側が次のものを個別に実装する必要がありました。

* リポジトリの探索
* ファイルの読み書き
* ターミナル実行
* テストの再実行
* 失敗時の自己修正ループ
* Git 差分の確認やコミット補助

ところが Claude Code や Codex CLI のような「コーディングエージェント」は、これらを単なるツール群ではなく、**ひとまとまりの作業環境**として提供します。だからこそ使い勝手がよい一方、各エディタ・各フレームワーク・各実行環境ごとに個別統合が必要になる、という問題もありました。

ACP はまさにそこを標準化しようとする動きです。ACP 公式ドキュメントでは、エディタとコーディングエージェントが密結合になりがちで、

* 新しい組み合わせごとに統合作業が必要になる
* 対応エディタや対応エージェントが限定される
* 開発者が特定ツールにロックインされやすい

といった問題がある、と説明されています。これはかなり LSP に似た構図です。LSP が「エディタと言語機能の結合」をほどいたように、ACP は **エディタ / クライアントとコーディングエージェントの結合をほどく** ための規格だと理解すると分かりやすいです。

Mastra にとって ACP が重要なのは、Mastra 自身がコーディングハーネスをゼロから再発明しなくても、**既存の強力なコーディングエージェントをサブエージェントやツールとして組み込める** からです。

## ACP とは何か

### ACP は「クライアントとコーディングエージェント」をつなぐ標準

ACP は Agent Client Protocol の略で、**コードエディタ / IDE / アプリケーションなどのクライアントと、AI コーディングエージェントのあいだの通信を標準化するプロトコル**です。

ここで重要なのは、ACP が想定している相手が「一般的な外部 API」ではなく、**コード編集や実行を伴うエージェント**だという点です。つまり単純な REST API というより、「ユーザーの代わりにコードベースへ入っていって仕事を進める相手」を扱うための規格です。

ACP 公式ドキュメントによると、現在の中心ユースケースはローカル実行です。

* ローカルでは、エージェントがサブプロセスとして起動され、`stdio` 上の JSON-RPC で通信する
* リモートエージェント対応も視野に入っているが、まだ進行中の領域がある

つまり現時点の ACP は、A2A のような「ネットワーク越しの別エージェント間通信」というより、**クライアントがローカルまたは近接したコーディングハーネスを安全に使う**ための色が強い規格です。

### “harness” とは何か

今回のアナウンスで繰り返し出てくる `harness` という言葉は、ざっくり言えば **ACP を話せるコーディングエージェント実行基盤** のことです。

単なるモデル API ではなく、たとえば次の機能をまとめて持っています。

* ファイルの読み書き
* シェルコマンドの実行
* テスト失敗時の再試行
* Git 操作
* セッション維持
* 実行中の進捗通知
* 権限要求

Mastra が今回手に入れたのは「新しいモデル」ではなく、**こうしたコーディングハーネスに仕事を委譲する接続口**です。ここがかなり重要です。

## Mastra で何ができるようになったのか

Mastra の ACP 対応は、大きく見ると 2 つの使い方があります。

### 1. `AcpAgent` としてサブエージェント化する

最も分かりやすいのはこれです。ACP 互換ハーネスを `AcpAgent` で包み、親の Mastra エージェントの `agents` に登録します。

```
import { Agent } from "@mastra/core/agent";
import { AcpAgent } from "@mastra/acp";
import { Workspace, LocalFilesystem } from "@mastra/core/workspace";

const TARGET_REPO = "/path/to/your/local/repo";

const claudeAgent = new AcpAgent({
  id: "claude-agent",
  name: "Claude Agent",
  description: "Claude Agent, enabled over the Agent Client Protocol",
  command: "npx",
  args: ["-y", "@agentclientprotocol/claude-agent-acp"],
  cwd: TARGET_REPO,
  workspace: new Workspace({
    id: "claude-agent-workspace",
    filesystem: new LocalFilesystem({ basePath: TARGET_REPO }),
  }),
});

export const supervisorAgent = new Agent({
  id: "supervisor-agent",
  name: "Supervisor Agent",
  instructions: `
    Delegate code tasks to the claude-agent subagent,
    including any relevant skills in the prompt.
  `,
  model: "openai/gpt-5.4",
  agents: {
    claudeAgent,
  },
});
```

この構成では、Mastra 側のエージェントが supervisor になり、どの作業を外部ハーネスに任せるかを判断します。ハーネス側は“手を動かす人”、Mastra 側は“段取りを決める人”です。役割分担がはっきりしています。

Mastra の ACP ドキュメントでは、`AcpAgent` だけでなく `createACPTool` も提供されています。こちらはサブエージェントとして全面委譲するのではなく、**必要なときだけコーディングエージェントをツールとして呼ぶ** ための API です。

この違いは地味ですが大事です。

* `AcpAgent`: 「専門家の部下」として委譲する
* `createACPTool`: 「高機能な道具」として必要時に呼び出す

Mastra は supervisor / subagent パターンが得意なので `AcpAgent` との相性が特に良さそうですが、既存のツール中心エージェントに後付けで組み込むなら `createACPTool` が便利です。

## 今回の実装で本当に効いているポイント

### 1. Workspace を通したファイル操作

Mastra の ACP ドキュメントで特に重要なのは、**ACP 側のファイル操作が Mastra の `Workspace` 抽象化を通る** と明記されている点です。

これはかなり Mastra らしい設計です。単に外部 CLI に好き放題ファイルを書かせるのではなく、Mastra 側で用意したワークスペースを通して読み書きを制御するので、

* どのディレクトリまで触れてよいか
* 読み取り専用かどうか
* どのスキルを参照させるか

を supervisor 側で整理できます。

公式アナウンスでも、supervisor 側のワークスペースにスキルを持たせ、委譲時にそれを一緒に渡す構成が紹介されています。つまり ACP 対応は「外部エージェントに丸投げ」ではなく、**Mastra のスキル / ワークスペース / ガバナンスと組み合わせて使う設計** になっています。

### 2. 権限要求を扱える

ACP では、エージェント側が作業の途中で permission request を返すことがあります。Mastra 側では `onPermissionRequest` を使って、この要求にどう応答するかを制御できます。

これは実運用ではかなり重要です。たとえば次のような場面です。

* このディレクトリに書き込んでよいか
* このコマンドを実行してよいか
* この変更を続行してよいか

コーディングエージェントは便利ですが、勝手に動きすぎると怖い。Mastra が permission callback を持っていることで、**ローカルポリシーや独自承認フローを差し込める** ようになっています。暴走防止の安全弁ですね。

### 3. セッションを持続できる

`@mastra/acp` は、初回利用時に ACP エージェントプロセスを起動し、デフォルトでは `persistSession: true` でセッションを維持します。

これはコーディングエージェントと相性がよい仕様です。コードベースの文脈や直前の作業内容を引き継げるので、関連する複数タスクを連続して投げやすい。逆に、毎回完全に隔離したいなら `persistSession: false` にしてワンショット実行にもできます。

このあたりは「普通の API 呼び出し」ではなく、**ある程度 stateful な作業相手** としてハーネスを扱っていることがよく分かります。

### 4. モデル選択まで踏み込める

ACP ドキュメントでは、エージェントが対応モデル一覧を公開している場合、Mastra 側から `getAvailableModels()` で取得したり、`model` オプションで選択したりできます。

つまり Mastra は単に「CLI を叩く」だけでなく、**ACP セッションが公開する機能面まである程度抽象化して取り込んでいる** ということです。

## MCP や A2A と何が違うのか

ここは少し整理しておくと、Mastra の最近の動きが見えやすくなります。

* **MCP**: エージェントにツールやリソースを渡す標準
* **A2A**: エージェント同士がネットワーク越しに仕事を依頼し合う標準
* **ACP**: クライアントとコーディングエージェントの通信を標準化する規格

感覚的にはこうです。

* MCP は「エージェントが外界の機能を使う」ための口
* A2A は「エージェントが別のエージェントに頼む」ための口
* ACP は「アプリやエディタがコーディングエージェントを呼び出す」ための口

Mastra はここ数か月で、MCP、A2A、Browser、Workspace、Remote Filesystem、LSP for Workspaces などを次々に揃えてきました。今回の ACP 対応は、その流れの中でもかなり象徴的です。**Mastra 自身が全部の能力を内製するのではなく、外部の強い実行基盤を“接続可能な部品”として扱う** 方向がよりはっきりしました。

特に ACP は、A2A よりも「コーディング体験」に寄っています。A2A がネットワーク越しの独立エージェント連携なら、ACP は **ローカルな作業環境・CLI・エディタ拡張を含んだ“手触りのあるコーディングエージェント”を組み込むための規格** です。

## どういう場面で効くか

今回の発表が特に刺さるのは、次のようなケースです。

### 1. Mastra を司令塔にして、実作業は専用コーディングエージェントに任せたい

たとえば Mastra 側では、

* ユーザー要求の整理
* タスク分解
* どの作業を誰に任せるかの判断
* 途中経過の要約
* スキルやポリシーの付与

を担当し、実際のコード変更・テスト・差分確認は ACP ハーネスに任せる、という分業ができます。

これはかなり自然です。Mastra の supervisor 能力と、Claude Agent / Codex CLI 系の“手を動かす能力”は補完関係にあります。

### 2. 既存のコーディングハーネスを再利用したい

企業やチームによっては、すでに特定のコーディングエージェントを使い込んでいる場合があります。そのとき、Mastra 導入のたびに同等機能を再実装するのは非効率です。

ACP 対応があると、Mastra を導入しても「今まで使っていたコーディングハーネス資産」を活かしやすくなります。

### 3. 安全に“コード編集付きエージェント”を組み込みたい

コード編集を伴うエージェントは便利な一方で、次の不安がつきまといます。

* どこまでファイルを触るのか
* どのコマンドを実行するのか
* 変更結果をどう監督するのか

Mastra の Workspace と permission handling を併用することで、完全に自由な野良 CLI 実行よりも、**管理された枠の中でコーディングエージェントを使いやすくなる** のが今回のうれしい点です。

## 既存手法・関連規格・競合との位置づけ

この発表の面白いところは、Mastra が「自前のコーディングエージェントを売り込む」より、**相互運用の側に立った**ことです。

ACP 公式の説明は、かなり明確に「エディタとエージェントの密結合を解きたい」と言っています。これは裏返すと、今のコーディングエージェント市場が、まだかなり分断されていることの証拠でもあります。

* あるツールは特定エディタでは強いが他では使いにくい
* あるハーネスは強力だが、別フレームワークに組み込むのが難しい
* 便利な CLI があっても、自分のアプリケーションから扱いづらい

Mastra の ACP 対応は、この分断を吸収するための一歩です。もちろん ACP 自体もまだ新しく、リモート対応は進行中で、対応ハーネスの裾野もこれから広がる段階です。ですが、**標準化の方向へ早めに乗っておくこと** には十分価値があります。

LSP が最初から完璧だったわけではないのと同じで、こういう規格は「採用する実装が増えるほど意味が出る」性格のものです。Mastra が初期段階から ACP を受け入れたのは、戦略的にかなり筋がよい動きだと思います。

## まとめ

Mastra の ACP 対応は、単に「Claude Agent や Codex CLI を呼べるようになった」という以上の意味があります。重要なのは、**Mastra がコーディングエージェント実行基盤を外部の標準規格経由で取り込めるようになった** ことです。

特に大きいポイントは次の 4 つです。

1. **既存のコーディングハーネスを再利用できる**
2. **`AcpAgent` と `createACPTool` の両方で組み込める**
3. **Workspace と permission handling により、Mastra のガバナンスと両立できる**
4. **セッション持続やモデル選択など、コーディングエージェントらしい振る舞いをそのまま扱える**

Mastra はここ数か月、Browser、A2A、Workspace、Remote Filesystems、LSP for Workspaces など、「エージェントが実際に仕事をするための周辺機能」をどんどん厚くしてきました。今回の ACP 対応は、その流れの中でも特に **“コーディングエージェントを相互運用可能な部品として使う”** という方向性を明確に示す発表です。

言い換えると、Mastra は「自分だけで完結するフレームワーク」ではなく、**優秀な外部エージェントや実行環境を接続しながら全体をオーケストレーションする司令塔** に近づいています。今回の ACP 対応は、そのためのかなり強い一手です。
