## 結論

ベンチマークの話題が先行しがちですが、業務システムに組み込む側として読むべきは性能表よりも「運用の境界線」です。

| 境界線 | 公式情報の内容 | 実務での対応 |
| --- | --- | --- |
| 拒否・フォールバック | Fable 5には安全分類器があり、APIでは拒否がHTTP 200の `stop_reason: "refusal"` として返る。`stop_details.category` には `cyber` / `bio` / `reasoning_extraction` などが入る | 「Fable 5が常に答える」前提を捨て、拒否・fallback・応答モデルをログに残す |
| 料金・アクセス | API: $10/$50 per MTok。サブスクは6/22まで追加費用なし、6/23以降は使用クレジットが必要 | 評価は6/22までに済ませ、本採用はAPI料金でコスト試算する |
| データ保持 | Fable 5は30日保持が必須。Claude APIのZDR対象外で、Bedrockでも provider data share が必要 | ゼロ保持を顧客と約束している案件では使わない。契約とデータ分類に照らす |
| モデル仕様 | `claude-fable-5`、1M token context、最大128k output、adaptive thinkingは常時ON。手動のextended thinking budgetやassistant prefillは使わない | 長文・長時間タスクに絞り、短い大量処理はOpus/Sonnet/Haikuへ逃がす |

## Fable 5とMythos 5の位置づけ

Fable 5はOpusクラスの上に新設された「Mythosクラス」のモデルで、モデル文字列は `claude-fable-5` です。同じ基盤モデルから、安全対策（セーフガード）を一般提供向けに有効化したものがFable 5、サイバー防御機関など限定パートナー向けに一部解除したものがMythos 5として提供されます。

公式ドキュメント上の統合前提は次の通りです。

| 項目 | Fable 5 |
| --- | --- |
| API model ID | `claude-fable-5` |
| 主な用途 | 最も難しい推論、長時間のagentic work |
| context window | 1M tokens |
| max output | 128k tokens |
| thinking | adaptive thinking が常時ON |
| extended thinking | 手動budget指定は非対応。`output_config.effort` で深さを調整する |
| data retention | 30日保持、zero data retention対象外 |
| 価格 | input $10 / MTok、output $50 / MTok |

性能面で公式が強調しているのは「タスクが長く複雑になるほど、既存モデルとの差が開く」という点です。公式発表では、Stripeの大規模Rubyコードベース移行事例などが紹介されています。短い単発タスクではOpusやSonnetとの差が小さい可能性があり、後述のコストを踏まえると「長時間・自律型のタスクにだけFableを割り当てる」ルーティングが現実的です。

## 境界線1：拒否とフォールバック前提でパイプラインを設計する

Fable 5には新しい分類器（classifiers）が付属し、特定領域のリクエストを検知すると応答挙動が変わります。Anthropic公式発表では、サイバーセキュリティ、生物学・化学、蒸留に関係する入力で、一般ユーザー向けの保護挙動や別モデルへの切り替えが起き得ると説明されています。

APIドキュメントでは、Fable 5が拒否した場合はHTTPエラーではなく、Messages APIの成功レスポンスとして `stop_reason: "refusal"` が返ると説明されています。`stop_details.category` には `cyber` / `bio` / `reasoning_extraction` などの分類が入ることがあり、fallbackの実装方法はClaude API、Claude Platform on AWS、Amazon Bedrock、Vertex AI、Microsoft Foundryで差があります。

つまり実装側では、次の2つを分けて扱います。

| 場面 | 実装で見るもの |
| --- | --- |
| Claudeアプリや一部サーフェスでの自動切り替え | ユーザーに通知されたか、応答モデルが何だったか |
| Claude API / Claude Platform on AWS | `stop_reason`、`stop_details.category`、`fallbacks` betaの利用有無、課金対象 |
| Amazon Bedrock / Vertex AI / Microsoft Foundry | platform adapter側のclient-side fallback、SDK middleware、provider側の対応範囲 |

公式データではフォールバックが発生しないセッションが95%超とされていますが、実務上の論点は発生率ではなく\*\*「Fable 5が必ず最後まで答える」という前提が崩れた\*\*ことです。

* セキュリティ診断ツールの解説、依存パッケージの脆弱性調査、医薬・化学系ドメインの業務など、正当な用途でも分類器に当たる可能性があります（公式も「無害なリクエストを誤検知することがある」と明言しています）
* 品質評価やリグレッションテストで「Fable 5の出力」を比較する場合、フォールバック発生の有無を記録しないと比較が成立しません
* ユーザー向けプロダクトに組み込む場合、「応答がOpus 4.8に切り替わることがある」挙動を仕様として織り込む必要があります

最低限、次の項目をログに残します。

```
{
  "requested_model": "claude-fable-5",
  "served_model": "claude-fable-5_or_fallback_model",
  "stop_reason": "refusal | end_turn | max_tokens | tool_use",
  "stop_details_category": "cyber | bio | reasoning_extraction | null",
  "fallback_strategy": "api_fallbacks_beta | sdk_middleware | client_side | none",
  "fallback_model": "claude-opus-4-8",
  "fallback_used": true,
  "domain": "coding | security | biology | finance | general",
  "billable_path": "fable | fallback | refused_without_output"
}
```

このログがないと、品質比較も、コスト比較も、顧客への説明も曖昧になります。Fable 5を入れる最初の実装は、プロンプト改善より先に、拒否とfallbackの観測から始めるべきです。

## 境界線2：6月22日と6月23日で世界が変わる

アクセス経路は段階的です。

* **API・従量課金Enterprise**: 6月9日から全面利用可能。$10 per 1M input tokens / $50 per 1M output tokens
* **サブスク（Pro / Max / Team / シート型Enterprise）**: 6月9日〜22日は追加費用なしで利用可能。**6月23日に一旦プランから外れ、以降は使用クレジットが必要**（容量が確保でき次第、プランへの復帰を目指すと公式は述べています）

つまりサブスクユーザーにとって、いまは「期間限定の評価ウィンドウ」です。実務での動き方は次のようになります。

1. 6/22までに自分の代表的なワークロードでFable 5とOpus 4.8を比較し、差が出るタスクと出ないタスクを仕分けする
2. 差が出たタスクについて、API料金（$10/$50）でのコストを試算する
3. 6/23以降も使い続ける価値があるかを、クレジット購入前に判断する

「とりあえず全部Fableに切り替える」は、6/23にコスト構造が急変するため危険です。

簡易試算はこの式で十分です。

```
1回あたり概算コスト =
  input_tokens / 1,000,000 * 10
+ output_tokens / 1,000,000 * 50
```

例えば、1回の調査・実装タスクで input 200k tokens、output 20k tokens を使うなら、概算は `$2 + $1 = $3` です。これを100回/月に広げると約 `$300/月` になります。同じ量をOpus 4.8で流すと、単価はおおむね半分です。

Fable 5は「高いから避ける」モデルではありません。ただし、短いFAQ、分類、定型要約まで全部流すモデルでもありません。

## 境界線3：30日データ保持が必須になる

見落としやすいのがデータ取り扱いの変更です。

原文: "We will require 30-day retention for all traffic on Mythos-class models, on both first- and third-party surfaces."  
日本語訳: 「Mythosクラスのモデルにおける全トラフィックについて、ファーストパーティ・サードパーティ双方のサーフェスで30日間の保持を必須とする」  
引用元: [Claude Fable 5 and Claude Mythos 5 - Anthropic](https://www.anthropic.com/news/claude-fable-5-mythos-5)

保持されたデータは安全対策目的に限定され、原則30日後に削除されると説明されています。ただし、これは**ゼロデータ保持（ZDR）を前提にした運用と両立しない**ことを意味します。

* 顧客との契約やプライバシーポリシーで「プロンプトは保持されない」と説明している場合、Fable 5の利用前に文言の確認が必要です
* Bedrockでは `anthropic.claude-fable-5` / `global.anthropic.claude-fable-5` がモデルIDとして示され、Fable 5利用には provider data share が必要です
* 社内の機密区分で「外部保持不可」とされているデータは、Fable 5ではなくOpus以下のモデルに流すルーティングで対応します

## 実務でのモデルルーティング

Fable 5を入れるなら、モデル選択を人間の気分ではなくルールにします。

| 入力タスク | 第一候補 | 理由 |
| --- | --- | --- |
| 大規模コード移行、長時間のagentic coding | Fable 5 | 長時間・複雑タスクで差が出やすい |
| 複数文書をまたぐ調査、金融・法務の深い比較 | Fable 5を評価候補 | 1M contextと長時間推論の価値がある |
| 一般的な設計相談、コードレビュー、通常の実装 | Opus 4.8 / Sonnet 4.6 | Fable 5の単価を使う必然性が弱い |
| 高頻度の分類、短文生成、FAQ | Sonnet / Haiku | コストとレイテンシ優先 |
| セキュリティ、biology/chemistry、distillationに近い入力 | Fable 5へ直送しない | 拒否・fallback・誤検知を前提に設計が必要 |
| ZDRや外部保持不可のデータ | Fable 5を使わない | 30日保持必須と矛盾する |

最初の評価では、全タスクをFable 5へ流さず、次のようなルールで始めるのが扱いやすいです。

```
model_routing:
  default: claude-sonnet-4-6
  complex_reasoning: claude-opus-4-8
  long_horizon_agentic_work:
    candidate: claude-fable-5
    requires:
      - no_zero_retention_requirement
      - no_sensitive_domain_without_review
      - cost_budget_approved
      - refusal_and_fallback_logged
```

## 失敗パターン

### 1. ベンチマークだけ見て一括切り替えする

短いタスクでは既存モデルとの差が小さい可能性があり、出力単価は高水準です。長時間・自律型タスクに絞って割り当てないと、品質向上を体感できないままコストだけ増えます。

### 2. 拒否やフォールバックを「例外」として扱う

拒否やOpus 4.8への切り替えは仕様であって障害ではありません。HTTP 200で `stop_reason: "refusal"` が返る経路もあるため、通常の例外ハンドラだけでは拾えません。発生を記録し、頻発するドメインがあれば送るモデル自体を見直します。

### 3. データ保持の変更を法務に共有しない

技術側だけで導入を判断すると、顧客説明と実態がずれます。30日保持はモデル選択の問題ではなく契約の問題なので、導入前に共有が必要です。

### 4. 「サブスクで今使える」を本番可用性と勘違いする

6/22までは評価ウィンドウとして有用ですが、6/23以降は使用クレジットが必要になります。業務システムに組み込むなら、Claudeアプリ上の一時的な利用可否ではなく、API、課金、fallback、データ保持を前提に判断します。

## まとめ

Fable 5の本質は「Opusより賢いモデルが出た」ではなく、「拒否・fallback・段階的提供・30日保持という新しい運用前提を持つティアが始まった」ことです。

最初にやるべきことは、全ワークロードをFable 5へ切り替えることではありません。代表タスクを選び、Opus 4.8と比較し、拒否とfallbackをログに残し、30日保持と契約が矛盾しないかを確認することです。

この4点がそろってから、長時間・高価値のタスクだけFable 5へルーティングします。そこまでできて初めて、「最新モデルを試した」ではなく「業務システムに入れられる形で評価した」と言えます。

モデルの使い分けを含む業務システムへのAI導入設計は[ITPRODX.com](https://itprodx.com/)でも整理しています。

APIコード、platform別fallback、BedrockのモデルIDと保持モードまで含む実装チェックは、Qiita完全版の[Claude Fable 5をAPIに入れる前に見るべき7つの本番チェックリスト](https://qiita.com/YushiYamamoto/items/3150aae08e4065f8a4d1)に分けました。Zenn側では「何を決めてから導入するか」、Qiita側では「どう実装して確認するか」を役割分担します。
