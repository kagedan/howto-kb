## 複数のAIを走らせた瞬間、あなたは分散チームを運用している

Claude Code、OpenAI Codex、GitHub CopilotのようなAI coding agentを1人で使っていても、2つ以上のtaskを並列に走らせた時点で、状況は「1人の開発」ではなくなります。

それぞれのagentがbranchを切り、pull requestを作る。しばらくすると、同じrepositoryに複数のopen PRが同時に並びます。これは実質、複数の作業者が同じrepositoryへ同時にPRを出している状態です。つまり、あなたは小さな分散チームを運用しています。

ここで詰まる原因は、多くの場合「各AIの賢さ」ではありません。

各agentは、渡されたtaskを妥当にこなします。1本のPRだけを見れば、CIも通り、diffも筋が通っている。それでも手戻りが起きるのは、**それぞれのagentが「他のPRが今まさに何を変えているか」を知らないまま作業しているから**です。

足りないのは賢さではなく、**共有コンテキスト** — 今この瞬間、他のin-flightな変更が同じrepositoryのどこへ向かっているか、という共有状態です。

## なぜ共有コンテキストが欠けるのか

共有コンテキストは、放っておくと自然には生まれません。理由はいくつかあります。

* **各agent / 各chatは、自分のtaskしか見ない。** Agent Aは自分のbranchを、Agent Bは自分のbranchを見ている。互いのPRは互いの視野の外です。
* **個別に質問して得た答えは、そのsessionに閉じる。** あるchatで「このファイルは他で触っていない？」と聞いて得た回答は、その場の一時的な返答であって、GitHub上の他のPRに反映される共有状態にはなりません。
* **状況はpush / merge / closeで動く。** 昨日Clearだったものが、別のPRが先にmainへ入った今日はClearとは限りません。共有コンテキストは1枚のスナップショットではなく、動き続ける状態です。
* **一度だけ全PRを解析しても足りない。** PRは増え、mainは進み、drafがreadyになる。解析は状況が変わるたびに更新されないと、すぐ古くなります。

このため、「AIをもっと賢くする」方向だけでは、この問題は埋まりません。埋めるべきは、個々のagentの外側にある**共有面**のほうです。

## 共有面はGitHub、そこに状況を書く

AI agentが同じrepositoryでPRを作る運用では、すでに全員が共有している面が1つあります。GitHubのpull requestそのものです。

Veripsa Coreは、この面に状況を書くGitHub Appです。特定のAIベンダーに依存しない共有面（= GitHubのcheckとPR comment）に、**PRの「中」ではなく、PRとPRの「間」の状況**を出します。

* ほかのopen PRと同じ作業面へ向かっていないか
* どのPRに着地順の注意が必要か
* どこが待つ・rebaseする・順番を変える・通常のreviewを続ける、の分かれ目か
* 根拠不足をClearと誤読していないか

大事なのは、これを**agentのsessionへ勝手に注入するわけではない**という点です。Veripsaは共有面に状況を置くだけで、人もagentも、それをPR上で読みます。agentであれば `gh pr checks <PR番号>` や `gh pr view <PR番号> --comments` で読ませられます。

そして、よく言われる「traffic control（待つ / rebase / 着地順）」は、この共有コンテキストから**導かれるaction**です。衝突はその共有コンテキストが欠けたときに出る**症状の1つ**にすぎません。順番、上流依存、下流への波及、待ち行列 — どれも「今ほかで何が動いているか」を共有できて初めて出せる助言です。

Veripsa Core自体の概要は [Veripsa Coreとは？GitHubで増えるAIのPRをmerge前に見るGitHub App](https://zenn.dev/veripsa/articles/what-is-veripsa-core) に分けています。この記事では、共有コンテキストという上流の話に絞ります。

## 図鑑: 共有コンテキストからVeripsaがPRに書くこと

「PR同士の間を見る」と言っても抽象的なので、実際にVeripsa CoreがPR上に書く内容を、共有コンテキストの種類ごとに並べます。ここに挙げるのは、いずれも現行機能としてPRに出るものです。

### A. 「待つ / 順番」— 誰が先に、どこで重なるか

* **直接衝突 → 順番に着地する**: 2本のin-flight PRが同じファイルに触れると、merge前に直接衝突を指摘し、「順番にlandする」ことを提案します。
* **待つ側・保持する側の両方に通知**: 待つPRには「どのPRの後ろか」を、保持するPRには「後ろに何本待っているか、早めにlandしてlaneを空けて」と、両側に伝えます。
* **着地順の提案**: 絡み合ったPR群には、最も土台になるものから着地する順を提案します。後続は一度だけrebaseすれば済みます。

### B. 「Heads up」— textでは見えない関係

* **意味的な結合の警告**: gitのtextual mergeでは見えない構造的な結合（別ファイルからの参照、共有テーブル、共有configキーなど）で2本のPRがつながっているとき、「これはgit conflictではないが、mergeの前に揃えたほうがよい」というheads-upを出します。
* **上流依存が今まさに変わっている**: 自分のPRが土台にしている部分を、別のin-flight変更が今まさに書き換えているとき、警告します。これはあとで作り直しになる「手戻りの罠」です。
* **波及範囲(blast radius)**: 変更の構造的な下流を示し、その広がりが大きいときは「wide」のように性質で伝えます（生の件数は出しません）。
* **繰り返しぶつかる土台ファイルの分割提案**: 独立した変更が何度も同じ土台ファイルでぶつかっているとき、そのファイルの分割を候補として提案します（そのPRが実際に触るファイルに限ります）。
* **mainが自分の下で動いた**: branchを切ったあと、自分が編集中のファイルがmain側で更新されていたら、mergeでconflictに当たる前にrebaseを促します。並列PR検知だけでは死角になりやすい部分です。

### C. 「Unknown」— 分からないをClearに混ぜない

* **Unknownの理由を分ける**: 「単にこのPRで新規追加されたファイル（merge後に計算可能、これは警告ではない）」と、「本当に解析できなかったgap（unknownとして扱う）」を区別します。どちらも都合よくClearには倒しません。
* **抑制した結合を沈黙させない**: 広く共有されたファイルを経由する結合を、群全体への警告にはしない一方で、Clearとも言わず、Unknownとして「ここは沈黙ではなく、確認できていない」と見えるようにします。

### D. hard fail と acknowledge

* **未解決のconflict marker**: PRが追加した未解決のconflict marker（git が残す未解決の衝突マーカー対）を、行の中身を読まずに検知し、check をhard-failさせます。Veripsaが自分から出す唯一のhard failです。
* **ACK（acknowledge to proceed）とその再確認**: 本物のcross-PR couplingは、`veripsa-ack`ラベルで明示的に確認するまで`action_required`のまま保たれます。これはapprovalでもpardonでもなく、「この結合を見た上で進む」という記録です。結合が変われば再度pauseし、再確認を求めます。

こうして並べると分かるのは、どれも「このPRのコードが良いか悪いか」ではなく、\*\*「今ほかで動いている変更との関係」\*\*を書いている、という点です。これが共有コンテキストの正体です。

## 実物: 2本のPRが同時にopenのとき

抽象論だけでは分かりにくいので、公開demo repositoryを見ます。

どちらのPRも `orders/pricing.py` と `tests/test_pricing.py` を編集します。2本とも同時にopenだったので、私がPR上のVeripsa commentを確認したところ、こう出ていました（確認時点の実際の表示）。

保持している側、[PR #14](https://github.com/GetVeripsa/ai-pr-collision-lab/pull/14)：

> ✓ Clear — nothing is blocking you  
> 1 in-flight PR is waiting behind this one on: `orders/pricing.py`, `tests/test_pricing.py`

後ろで待つ側、[PR #13](https://github.com/GetVeripsa/ai-pr-collision-lab/pull/13)：

> ⏸ Wait in line — a direct collision is ahead  
> You collide in `calculate_total` (in `orders/pricing.py`) — that's the same code, not just the same file.

PR #13にはさらに、`orders/pricing.py` の22行目付近でrebase時にtextual conflictが出やすい、というheads-upと、着地順（#14 → #13）が付きます。ここで読めるのは次のことです。

* どちらのPRも、単体では妥当な小さな変更です。悪いPRという話ではありません。
* 2本を**同時に見て初めて**、「同じ `calculate_total` に触っている」「#14が先」という共有コンテキストが出ます。
* PR #14の `Clear` は「実装が正しい」証明ではなく、「見えているtraffic上で先に入れるべき相手がいない」という限定された意味です。

同じ2エージェント衝突を、片方をmergeする前後のbefore/afterで追った例は [2つのAIエージェントが同じファイルを編集—PR衝突をmerge前に見るGitHub App](https://zenn.dev/veripsa/articles/two-ai-agents-same-file-pr-collision-before-merge) に、merge前に何を読むかの手順は [Claude Code/CodexのPRが衝突する前に見るべきこと](https://zenn.dev/veripsa/articles/claude-code-codex-pr-conflict-before-merge) に分けています。

## content-free — private repoに入れるための信頼境界

共有面に状況を書くと聞くと、「ソースを預けるのか」と気になります。

Veripsa Coreは、source file bodyやdiff bodyを保存・表示しません。signalを計算するためにrepository contentを処理時に一時的に読むことはありますが、永続保存したり、commentへ貼ったり、code review出力として再利用したりせず、処理後に破棄します。

これはVeripsaの中心価値そのものではなく、**private repositoryに入れてもらうための信頼境界**です。何を保存し、何を保存しないかの正確な線引きは [ソース本文を預けずに、PR 同士の衝突はどこまで見えるのか](https://zenn.dev/veripsa/articles/veripsa-content-free-analysis-scope) にまとめています。

## 今できること / まだできないこと

共有コンテキストと言っても、現行の対象範囲ははっきりしています。

* 対象は、**1つのinstalled repository内のopen PR**です。同じownerの別repositoryをまたぐcross-repository / cross-ownerの協調は、まだ提供範囲ではありません。
* Veripsaはadvisory by defaultです。agentのsessionへ状況を自動注入したり、自動でmergeやconflict解消をしたりはしません。共有面に状況を置き、人とagentがPR上で読む、という形です。
* 何をblockするかは、repository側がVeripsa checkをbranch protection / rulesetsのrequired checkにするかで決まります。

導入は、いきなりmerge gateにしなくて構いません。

1. selected repositoryを1つ選ぶ
2. branch protectionは変えず、advisoryでcheck / commentを観察する
3. `Clear` / `Heads up` / `Wait in line` / `Unknown` の出方を見る
4. 運用に合うと判断したら、required checkへの昇格を検討する

## まとめ

一人でも複数のAI agentを走らせれば、あなたはすでに分散チームを運用しています。そこで詰まるのは、各AIの賢さより先に、**「今ほかのPRが何を変えているか」という共有コンテキストの欠如**です。

* 各agentは自分のtaskしか見ない
* 個別chatの回答は共有状態にならない
* 状況はpush / merge / closeで動き続ける
* 一度だけの解析では古くなる

Veripsa Coreは、この共有コンテキストを、全員がすでに共有しているGitHubのPR上に書きます。衝突は、その共有コンテキストが欠けたときに出る症状の1つです。
