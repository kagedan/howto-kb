## TL;DR

* MCP サーバーを作ったら、公開の場は 1 つではなく**階層**になっています: 公式 MCP Registry(上流)→ Glama / PulseMCP / mcp.so / awesome-mcp-servers などのディレクトリ(下流)
* 公式レジストリへの登録は **HTTP ドメイン認証(Ed25519)+ server.json + mcp-publisher CLI** で、思ったより簡単でした
* 一番のハマりどころは意外な所にありました: **「API キーなしでも起動して tools/list に応答する」ことがディレクトリ掲載の事実上の要件**だった話を書きます
* 対象は [Argosvix(アーゴスヴィクス)](https://argosvix.com/ja) の MCP サーバー([@argosvix/mcp-server](https://www.npmjs.com/package/@argosvix/mcp-server)、87 ツール)ですが、手順は自作 MCP サーバー全般に使えるはずです

## 前提

* MCP サーバーを npm に公開済み(`npx -y @argosvix/mcp-server` で起動できる状態)
* 独自ドメインを持っている(公式レジストリの名前空間認証に使います)

## 1. 公式 MCP Registry への登録

公式レジストリ(registry.modelcontextprotocol.io)は、MCP サーバーの「正式な台帳」です。ここに登録しておくと、後述のディレクトリ群が自動クロールで拾ってくれる起点になります。

### 名前空間とドメイン認証

サーバー名は逆 DNS 形式(例: `com.argosvix/server`)で、**そのドメインの所有を証明**する必要があります。方式はいくつかありますが、私は HTTP ドメイン認証を使いました。

1. Ed25519 鍵ペアを生成する
2. **公開鍵**をドメインの `/.well-known/mcp-registry-auth` に配置する(静的ファイル 1 個)
3. `mcp-publisher login http --domain yourdomain.com --private-key ...` でログイン

ここで大事な運用上の注意がひとつ: **well-known ファイルは登録後も消してはいけません**。再ログインのたびに検証されるので、恒久的に配信し続ける前提で置き場所を決めてください(私は landing の静的ファイルとしてリポジトリ管理に含めました)。

### server.json と version 同期の罠

レジストリにはサーバーのメタデータを `server.json` で登録します。ここで踏んだ罠が、**version の三重管理**です。

* `package.json` の version(npm の正)
* `server.json` の version(レジストリの正)
* コード内の定数(initialize 応答で名乗る version)

この 3 つは一致していないと、クライアントが handshake で見る version とレジストリの表示がズレます。私は一度これで不一致を出したので、以後「**version を上げるときは 3 箇所同時**」を機械的なルールにしました。リリース手順書に 1 行書いておくだけで防げます。

### publish

```
cd your-mcp-server && npm publish
# server.json の version を合わせてから
mcp-publisher publish
```

npm に新しい version を出す → server.json を合わせる → `mcp-publisher publish`、の順です。

## 2. ディレクトリの世界 — 公式レジストリだけでは見つけてもらえない

登録して分かったのですが、ユーザーが実際に MCP サーバーを**探す**場所は公式レジストリではなく、その下流のディレクトリ群です。

awesome-mcp-servers に PR を出したところ、bot からこう返ってきました:「**まず Glama に掲載されてチェックを通し、スコアバッジを PR に付けてください**」。つまりディレクトリ間にも依存関係があり、Glama のチェック通過が他の場所の入場券になっています。

## 3. 最大のハマりどころ: 「キーなし起動」が要件だった

Glama の自動チェックは「サーバーが起動して introspection(initialize / tools/list)に応答すること」を確認します。ここで問題が発覚しました。

私のサーバーは API キー(環境変数)が未設定だと、**起動直後にエラーを出して exit する**設計だったのです。ユーザー保護としては真っ当な設計のつもりでしたが、チェッカーはキーなしで起動します。つまり構造的に掲載不能でした。

### 対応: introspection-only モード

「キーがなくても起動してツール一覧には応答し、**実行しようとしたときだけ**設定手順を案内する」形に変えました。

```
// 起動時: exit せず警告して継続
if (!API_KEY) {
  console.error("[mcp] API key is not set — starting in introspection-only mode.");
}

// tools/call の冒頭: キーがなければ isError の tool 結果で案内
if (!API_KEY) {
  return {
    content: [{ type: "text", text: "API key is not set. Get one at ... and add it to the env of this MCP server config." }],
    isError: true,
  };
}
```

ポイントは 2 つあります。

1. **プロトコルエラーではなく `isError: true` の tool 結果で返す** — こうすると接続先の LLM がエラーメッセージを読んで、ユーザーに設定手順をそのまま伝えてくれます。エラーが UX になる設計です
2. **購読(subscribe)系はキーなしでは受け付けない** — 受けてしまうと空キーでのポーリングが永続します。ここは明示的に InvalidParams で断る

やってみて気づいたのですが、これはディレクトリ対策を超えて**普通に良い改善**でした。ユーザーはキーを発行する前にツール一覧を見て「入れる価値があるか」を判断できます。「キーなしでも起動して一覧に応える」は、MCP サーバーの事実上のマナーと考えていいと思います。

## 4. リポジトリ公開の問題

もうひとつ、ディレクトリの多くは **GitHub リポジトリを掲載単位**にしています。開発リポジトリが非公開(モノレポ)の場合、npm パッケージだけでは掲載できない場所がある。

私は「npm に MIT で公開済みのパッケージのソースだけを公開ミラーリポジトリに切り出す」形で解決しました([argosvix/mcp-server](https://github.com/argosvix/mcp-server))。既に世界中に配布しているコードなので新たに公開する情報は増えず、ディレクトリの掲載単位・Issue の受付窓口・README の置き場が手に入ります。

ミラーを作る場合の注意点をひとつだけ: **公開する前に、同梱ファイルを全部見てください**。私は CHANGELOG に内部ドキュメントへの参照が残っているのを公開後のチェックで見つけて、慌てて消しました。npm の `files` フィールドに入っていないファイル(= npm では公開されていなかったファイル)をミラーに含めると、意図せぬ初公開になります。

## さいごに

MCP サーバーの配布は「npm に publish して終わり」ではなく、公式レジストリ → ディレクトリ群という階層を順に埋めていく作業でした。手順自体はどれも難しくありませんが、「キーなし起動」のような**掲載側の暗黙の要件**は、やってみるまで分かりませんでした。この記事がこれから載せる人のショートカットになれば嬉しいです。
