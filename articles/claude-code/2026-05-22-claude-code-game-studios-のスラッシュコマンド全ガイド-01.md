[前回の記事](https://zenn.dev/larry/articles/claude-code-game-studios-intro)では、Claude-Code-Game-Studiosの概要と、ネズミゲームを題材にした基本的な使い方を紹介しました。

読んだ方から「他のコマンドも気になった」という声をいただいたので、今回はスラッシュコマンドを全部並べます。リポジトリには72個あると説明されていますが、ここでは確認できた67個をカテゴリ別に紹介します。一覧をざっと眺めるだけでも、このテンプレートがどこまでカバーしているかが伝わるはずです。

## コマンドの全体像

67個のコマンドは12のカテゴリに整理されています。

| カテゴリ | コマンド数 | 担当フェーズ |
| --- | --- | --- |
| オンボーディング | 4 | プロジェクト開始 |
| ゲームデザイン | 6 | 企画・設計 |
| アート・アセット | 3 | 素材管理 |
| UX・インターフェース | 2 | UI設計 |
| アーキテクチャ | 4 | 技術設計 |
| ストーリー・スプリント | 8 | 実装管理 |
| レビュー・分析 | 9 | 品質確認 |
| QA・テスト | 10 | テスト全般 |
| プロダクション | 4 | 運営管理 |
| リリース管理 | 5 | リリース準備 |
| チーム調整 | 9 | 部門横断 |
| その他支援 | 3 | 汎用支援 |

カテゴリを見るだけで、個人開発では省略しがちなフェーズがきちんと埋まっているのがわかりますね。

## オンボーディング（4個）

プロジェクトを始めるとき、最初に使うコマンドです。

| コマンド | 説明 |
| --- | --- |
| `/start` | プロジェクトの現在地を診断して、適切な次のステップに誘導。no idea / vague concept / clear design / existing work の4段階から選ぶ。**最初に必ず打つ** |
| `/help` | ワークフローカタログを開く。どのコマンドを使えばいいか迷ったときの辞書代わり |
| `/project-stage-detect` | 既存の作業ファイルを読んで現在のフェーズを自動判定。途中参加やしばらく間が空いたときに便利 |
| `/setup-engine` | Godot 4 / Unity / Unreal Engine 5 から使うエンジンを設定。エンジンによってエージェントの振る舞いが変わる |

## ゲームデザイン（6個）

コンセプトからコアループを固めるフェーズです。

| コマンド | 説明 |
| --- | --- |
| `/brainstorm` | ゲームのアイデアを深掘りする。クリエイティブディレクターが主導してジャンル・視点・コアループの方向性を問い返してくる |
| `/design-system` | 特定のシステムを1つ掘り下げる。移動、インベントリ、視界など、コアに関わる仕組みを詳細ドキュメントに落とす |
| `/map-systems` | 複数システムが互いにどう影響し合うかを可視化する。移動→体力→AI追跡頻度、という連鎖を図に起こしてくれる |
| `/quick-design` | 設計の意思決定を素早く回す。「このUIフロー、2案のどちらがいいか」みたいな小さい判断をさくさく進めたいときに |
| `/review-all-gdds` | `design/`以下の全ゲームデザインドキュメントを横断してレビュー。矛盾や抜け漏れを拾ってくれる |
| `/propagate-design-change` | 設計変更が他のシステムに波及する影響を整理する。コアループを変えたとき、何を連動して直す必要があるかをまとめる |

## アート・アセット（3個）

ビジュアル素材の設計と管理を担います。

| コマンド | 説明 |
| --- | --- |
| `/art-bible` | プロジェクト全体のビジュアルスタイルガイドを作る。色調、ライティング、輪郭スタイルなど統一感の基準になるドキュメント |
| `/asset-spec` | 個別アセットの仕様を定義する。「ネズミのスプライトは64×64px、影なし、8方向」みたいな具体的な要件をまとめる |
| `/asset-audit` | 既存アセットのファイル名や配置が規約に沿っているかを検査する。命名ルールの混在を防ぐ |

## UX・インターフェース（2個）

UIの設計と評価をします。

| コマンド | 説明 |
| --- | --- |
| `/ux-design` | UIレイアウトとユーザーフローを設計する。HUDの配置、メニュー構造、チュートリアルの流れなどをドキュメント化 |
| `/ux-review` | 実装済みのUIをユーザビリティの観点で評価する。色盲対応やフォントサイズなどアクセシビリティの確認も含む |

## アーキテクチャ（4個）

技術設計を固めるコマンドです。

| コマンド | 説明 |
| --- | --- |
| `/create-architecture` | システムの全体構造と依存関係を設計する。テクニカルディレクターが担当し、エンジン選定の根拠や主要クラスの役割を定義 |
| `/architecture-decision` | 技術的な決定をADR（Architecture Decision Record）として記録する。「なぜ物理エンジンをXにしたか」を後から追えるようにする |
| `/architecture-review` | 実際のコードが設計通りの構造になっているかを監査する。依存関係の逆転や、本来別モジュールにすべき処理の混入を検出 |
| `/create-control-manifest` | 入力システムとコントロールスキームを定義する。キーボード・ゲームパッド・タッチの対応状況とアクションのマッピングをまとめる |

## ストーリー・スプリント（8個）

実装の計画管理を担うコマンドです。前回記事でも触れましたが、このカテゴリが日常的に最も使うことになります。

| コマンド | 説明 |
| --- | --- |
| `/create-epics` | 大きな機能をエピック単位に分解する。「ネズミの移動システム」「住人AI」「インベントリ」みたいな粒度 |
| `/create-stories` | エピックを受け入れ基準付きのユーザーストーリーに落とす。「8方向移動を実装する」「壁沿いスニーク移動が動く」などの単位 |
| `/sprint-plan` | スプリントのスコープを決めてタスクを分解する。何日で何ストーリーを消化するか、優先順位はどうするかをここで決める |
| `/sprint-status` | 現在のスプリントの進捗とブロッカーをレポートする |
| `/dev-story` | ストーリーを1つ選んで実装する。必要な専門家エージェントが自動で呼ばれる |
| `/story-readiness` | ストーリーが実装可能な状態になっているかを事前確認する。依存関係や受け入れ基準の曖昧さを着手前にチェック |
| `/estimate` | ストーリーのサイズを見積もる。スプリント計画の前に各ストーリーの難易度感を揃えるために使う |
| `/story-done` | ストーリー完了を記録する。受け入れ基準との照合が走る |

## レビュー・分析（9個）

コードと設計の品質を多角的に見るコマンドです。

| コマンド | 説明 |
| --- | --- |
| `/design-review` | 実装の方向性がゲームビジョンとズレていないかを確認する。「移動が速くなりすぎてステルス感が死んでいる」みたいな定性的なズレを指摘 |
| `/code-review` | コードの品質と規約への準拠をチェックする。ゲームプレイプログラマーとリードプログラマーが担当 |
| `/balance-check` | ゲーム経済と難易度曲線を分析する。視野角、食料補充頻度、アップグレードコストのバランスを検証 |
| `/content-audit` | ナラティブの一貫性を確認する。ゲーム内テキスト・ダイアログのトーンや設定との矛盾を拾う |
| `/scope-check` | フィーチャークリープのリスクを評価する。スコープが広がりすぎていないか、リリース目標に対して現実的かを確認 |
| `/perf-profile` | パフォーマンスのボトルネックを特定する。フレームレートを落としている処理を洗い出す |
| `/tech-debt` | 技術的負債をリストアップして優先順位を付ける。「今すぐ直すべきか、後回しにしていいか」を分類 |
| `/gate-check` | フェーズ移行の準備ができているかを確認する。コンセプトからプロダクションに進んでいいか、のゴーサインチェック |
| `/consistency-check` | ドキュメント・コード・アセットにわたって一貫性があるかを横断確認する。設計書と実装の乖離を見つける |

## QA・テスト（10個）

テスト計画から実行、分析まで揃っています。

| コマンド | 説明 |
| --- | --- |
| `/qa-plan` | テスト戦略とカバレッジ設計を作る。何をどこまでテストするかの方針をドキュメントに落とす |
| `/smoke-check` | 起動から基本操作までのクイック確認。「動くか」の最低限を素早く通す |
| `/soak-test` | 長時間プレイでの安定性テスト。メモリリーク、フレームレートの経時劣化、セーブデータの破損などを見る |
| `/regression-suite` | 過去に修正したバグが再発していないかを確認する |
| `/test-setup` | テスト環境とフィクスチャを設定する。テストで使うモックデータや初期状態を整える |
| `/test-helpers` | 再利用できるテストユーティリティを作る。複数のテストケースで共通して使う処理を関数化 |
| `/test-evidence-review` | テスト結果のドキュメントが適切に記録されているかを検証する |
| `/test-flakiness` | 不安定なテスト（たまに落ちる、たまに通る）の原因を調査する |
| `/skill-test` | エージェント自身の動作を検証する。設定したエージェントが期待通りの役割を果たしているかを確認 |
| `/skill-improve` | エージェントの動作を改善する。指示の精度を上げたり、役割の境界を調整する |

## プロダクション（4個）

プロジェクト全体の進行管理です。

| コマンド | 説明 |
| --- | --- |
| `/milestone-review` | メジャーマイルストーンの進捗を評価する。「α版の完成度はどこまで来ているか」を確認 |
| `/retrospective` | スプリントやフェーズ終了後の振り返りをする。うまくいったこと、課題、次への改善点を整理 |
| `/bug-report` | バグを記録して優先順位を付ける。再現手順、重大度、影響範囲を整理 |
| `/bug-triage` | 溜まったバグを分類して担当を割り当てる。修正順序の決定に使う |

## リリース管理（5個）

出荷前の最終確認フェーズです。

| コマンド | 説明 |
| --- | --- |
| `/release-checklist` | リリース要件の充足状況を確認する。法的表示、プラットフォーム要件、バージョン番号など出荷前に漏らしてはいけない項目を網羅 |
| `/launch-checklist` | デプロイ直前の最終確認。ストアページ、プレスキット、サポートページの準備状況も見る |
| `/changelog` | バージョン履歴のドキュメントを生成する |
| `/patch-notes` | プレイヤー向けのアップデート概要を作る。開発者視点ではなく、ユーザーが読んで分かる言葉で書く |
| `/hotfix` | ライブ版への緊急修正を展開する。最小限の変更で問題を封じ込めるための手順を踏む |

## チーム調整（9個）

複数のエージェントを横断してひとつのイニシアティブを動かすコマンドです。個別のコマンドを自分で組み合わせる代わりに、このコマンドを打つとチーム全体が連携して動きます。

| コマンド | 担当チーム |
| --- | --- |
| `/team-combat` | コンバットシステムチーム |
| `/team-narrative` | ナラティブチーム |
| `/team-ui` | UIチーム |
| `/team-release` | リリースチーム |
| `/team-polish` | ブラッシュアップチーム |
| `/team-audio` | オーディオチーム |
| `/team-level` | レベルデザインチーム |
| `/team-live-ops` | ライブオプスチーム |
| `/team-qa` | QAチーム |

「コンバットの手触りを全部見直したい」という状況で、プログラマー、デザイナー、QAをそれぞれ個別に呼ぶより`/team-combat`を一発打つ方が速いです。チームコマンドは自分のプロジェクトに合わせてカスタムも作れます。

## その他支援（3個）

上記に収まらない支援コマンドです。

| コマンド | 説明 |
| --- | --- |
| `/prototype` | アイデアの一次検証用に動くプロトタイプを素早く作る。仕様が固まる前に「動かして確認する」フェーズで使う |
| `/onboard` | 新しいチームメンバーやエージェントにプロジェクトを説明する。コンテキストを整理して引き継ぐためのドキュメントを作る |
| `/localize` | 多言語展開の準備をする。テキストの抽出、翻訳キーの整理、ロケール対応の設計をまとめる |

## よく使うコマンドのシーケンス

使い方がぼんやりしている場合は、フェーズに沿って流れを組んでみると整理しやすいです。

**企画フェーズ**

```
/start → /brainstorm → /design-system → /map-systems
```

**設計フェーズ**

```
/create-architecture → /create-control-manifest → /architecture-decision
```

**実装ループ**

```
/sprint-plan → /create-epics → /create-stories
  ↓（ストーリーごとに）
/story-readiness → /dev-story → /code-review → /story-done
  ↓（スプリント終わりに）
/sprint-status → /retrospective
```

**リリース前**

```
/qa-plan → /smoke-check → /soak-test → /regression-suite
→ /gate-check → /release-checklist → /launch-checklist
```

## コマンドの数より大事なこと

67個並べましたが、全部使うわけではありません。ソロ開発のネズミゲームくらいの規模なら、日常的に触るのは`/dev-story` `/code-review` `/balance-check` `/story-done`あたりに集中します。

カテゴリを眺めて「あ、QAの`/test-flakiness`は使えそう」とか「チームコマンドを自作してみよう」といった拡張のヒントを拾うのが、一覧の活用法としては一番しっくりきます。

リポジトリ自体がMITライセンスなので、プロジェクトに合わせてコマンドを追加・変更するのも自由です。既存のスキルファイルを見ながら同じ構造で書けば、自分のプロジェクト専用コマンドを足せます。

## おわりに

67個のコマンドをカテゴリ別に並べると、企画からリリースまでのゲーム開発の工程がそのまま見えてきますね。ソロだと後回しにしがちなQAやドキュメント管理まで、テンプレートとして用意されているのがこのリポジトリの面白いところです。
