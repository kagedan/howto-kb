## はじめに

AIコーディングアシスタントを使う開発チームが増えている。中でも **Claude Code**（Anthropicが提供するAIコーディングツール）は、ただコードを書くだけでなく「計画→実装→検証」を自分で繰り返すエージェントとして動く。ちょっとした修正からファイルをまたぐリファクタリングまで、驚くほど自律的にこなしてくれる。

私たちのチームでは、約3ヶ月にわたるプロジェクトで **全員がClaude Codeを使う体制** を試してみた。コードレビューにもAIを取り入れ、ドキュメントの自動生成も組み合わせた。

この記事では、その中で実際にぶつかった **課題と、それにどう対処したか** を率直にまとめている。「AI導入で開発が楽になった部分」と「想定外に大変だった部分」の両方を伝えることで、これからチームでの導入を考えている方の参考になればうれしい。

---

## 3行まとめ

* **AIコーディングツールはチーム開発で強力だが、「指示した以上の範囲を勝手に変える」「AIの回答が毎回変わりテストが難しい」「レビューが追いつかなくなる」といった独自の課題がある**
* **AIへの指示には「承認ゲート（実行前に人間が確認するステップ）」を設けるのが大事。何を変えるかを事前に確認してから実行するルールが品質を守ってくれる**
* **AIレビューは見落としと的外れな指摘が混在するので、チェックリスト化と人間レビューとの役割分担がカギになる**

---

## 概要

私たちのチームでは、複数の開発者がそれぞれClaude Codeを使いながら、ひとつのプロジェクトを一緒に開発するという、わりと新しいスタイルのチーム開発をやってみた。コーディングルールの制定、Claude Codeのルール定義（CLAUDE.md, rules, skills）、ドキュメントの自動作成、PRレビューまでをあわせて運用する体制だ。

Claude Codeの自律性は想像以上で、複数ファイルにまたがるコードの整理や、テストコードの生成、ドキュメントの更新まで、ひとつの指示でまとめてやってくれることもある。

でも、その「自律性の高さ」が、チーム開発ではかえって新しい問題を生むことになった。

---

## 課題1：AIが「指示した範囲」を超えて変更を広げる

**ひとことで言うと：「ここだけ直して」と頼んだはずが、関係ないファイルまで勝手に書き換えられた**

### 何が起きたか

あるファイルの修正を依頼したところ、Claude Codeは「関連しそうだ」と判断した別のファイルにも変更を加えていた。その結果、意図していない挙動の変化が複数箇所で発生し、差し戻し対応が必要になった。

AIは「全体の整合性のために必要」と判断すると、頼んでいないファイルも積極的に直そうとする。これはバグではなくClaude Codeの設計上の特性だ。ただ、チーム開発の場面では **想定外の変更が紛れ込むリスク** になってしまう。

### 対策

一番効果的だったのは、**実装前に変更対象を明示して承認を得る**フローを作ること。

Claude Codeに「これから何を変えるか」をまず宣言させて、それを人間が確認してからOKを出す。チームではオートモードを使わず、1つずつ内容を確認してから承認するように徹底した。速度は多少落ちるが、想定外の変更を防ぐ確実性のほうを優先した判断だ。

```
【変更予定】
- 対象ファイル: src/auth/login.py
- 変更内容: パスワードリセット機能の追加
- 変更しないファイル: src/auth/session.py, src/auth/oauth.py

上記のみ変更します。よろしいですか？
```

ポイントは、「確認しながら進める」の意味が「実装してから確認する」ではなく、「**実装の前に変更内容を見せてもらい、OKしてから実行する**」ということ。

加えて、Claude Codeの設定ファイル（.claude/settings.jsonのpermissions.allowやpermissions.deny）で特定のツール操作を制限する方法も有効だった。

```
{
  "permissions": {
    "defaultMode": "default",
    "allow": [
      "Read",
      "Edit",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npm run *)"
    ],
    "deny": [
      "Bash(rm *)",
      "Bash(git push *)"
    ]
  }
}
```

!

**CLAUDE.mdとは？**  
CLAUDE.mdは、Claude Codeがプロジェクト内で参照する指示書ファイル。コーディング規約や禁止パターンなどを記述しておくと、Claude Codeがタスク実行時に自動で読み込んでくれる。ただし、あくまでプロンプトとして読み込まれる「お願い」であり、AIが必ず従う保証はない。本当に守らせたい制約は`.claude/settings.json`の設定で担保するのが確実だ。

⚠️ 注意：permissions.deny による制限は、複数のバージョンで機能しないケースが報告されている。permissions.deny に明示的に設定したファイルへの読み書きがブロックされず、エラーも表示されないまま実行される Instructa Coursesというバグが確認されており、現時点での現実的な代替手段として PreToolUse フックの実装が推奨されている。 Shipyard設定ファイルによる制限は補助的なものとして扱い、過信しないことが望ましい。  
関連イシュー：#6699（2025年8月）／#13785（2025年12月）／#27040（2026年2月）

## 課題2：AIの回答が毎回変わるので、テストの設計が難しい

**ひとことで言うと：「同じ入力でも毎回違う出力が返る」という性質が、従来のテスト手法と相性が悪い**

### 何が起きたか

AIを組み込んだ処理フローを開発していて、「テストをどこまで書くか」の判断がとにかく難しかった。

AIの出力は確定的ではない。同じ入力を渡しても、返ってくる結果が毎回微妙に違う。つまり「AIへの指示文が正しいかどうか」をテストで保証するのは原理的に難しい。かといって、実際のAIを呼ぶと実行が遅いし、ダミーデータで代用すると現実との乖離が気になる。

### なぜ起きるか

従来のテストは「同じ入力なら必ず同じ出力」が大前提で、だからこそ期待値を書いて検証できる。でもAIは「確率的な関数」なので、この前提が成り立たない。

### 対策

チームで **テスト方針を明文化して、コスパの高い観点に絞る** ことにした。

| 種別 | 優先度 | 理由 |
| --- | --- | --- |
| 統合テスト（処理が通しで動くか） | 必須 | 機能全体の最低保証 |
| 外部サービス接続テスト | 必須 | 設定ミスの早期検知 |
| 複雑な分岐ロジックのテスト | 任意 | 複雑な場合のみ |
| AI出力の単体テスト | 省略 | 毎回結果が変わるので信頼性が低い。レビューで代替 |

統合テストではAI呼び出しを固定値のダミーに差し替えて、「処理フローが正しく通るか」だけを見る。「AIが何を返すか」ではなく「AIがこう返したとき、全体がちゃんと動くか」を確認するイメージだ。AIへの指示文の品質はコードレビューでカバーすることにした。

ポイントは「AIが何を返すかのテスト」ではなく「AIがこのレスポンスを返したとき、パイプライン全体が壊れないか」を見ることだ。

AI出力そのものの単体テストは省略するという割り切りも重要になる。

どうしてもLLMからのレスポンス内容を検証したい場合は、出力フォーマットをあらかじめ定義しておき、その構造が守られているかをチェックするにとどめるのがおすすめだ。

## 課題3：PRレビューがボトルネックになる

**ひとことで言うと：AIのおかげで実装は速くなったけど、人間のレビューが全然追いつかなくなった**

### 何が起きたか

全員がClaude Codeを使い始めると、**実装スピードは上がるのに、レビューのスピードがついてこない** という問題が起きた。私たちは1週間スプリントで回していて、PlanやReview・振り返りを除くと開発に使えるのは実質4営業日ほど。そこに検証・PRレビュー・PR承認のサイクルで丸1日以上かかることがあり、実装が速くなるほど「レビュー待ち渋滞」が発生する構造になった。

しかもAIは一度にたくさんのファイルを変更できるので、1回のPRの差分が大きくなりがち。レビュアーの負担はさらに増える。

### なぜ起きるか

これまでの開発では「実装の速さ」と「レビューの速さ」はだいたい釣り合っていた。でもAIで実装だけが加速すると、人間のレビューがボトルネックになるという構造的なギャップが生まれる。

### 対策

この課題には2つの方向からアプローチした。

**① 実装前に変更計画を承認して、レビュー総量そのものを減らす**

まず根本から見直したのが、**実装を始める前に変更計画を人間が承認する**フローの徹底だ。作ってからまとめてレビューするのではなく、「何を・どの範囲で変えるか」を事前に合意しておく。そうすることで方向のズレや手戻りが減り、レビューで確認すべき内容がぐっと絞られる。

これは課題1で紹介した「変更予定の宣言」と同じ考え方で、レビュー負荷の軽減にもそのまま効いてくる。

**② `/code-review` プラグインで一次レビューを自動化**

次に取り入れたのが、Claude Codeに組み込まれている `/code-review` プラグインだ。Claude Codeのリポジトリでオープンソース公開されており（[anthropics/claude-code](https://github.com/anthropics/claude-code/tree/main/plugins/code-review)）、Claude Codeが使えるプランなら追加費用なしで利用できる。

PRブランチ上で `/code-review` を実行すると、5つのエージェントがそれぞれ違う切り口（コーディング規約の準拠チェック、バグ検出、過去PRコメントとの整合性確認など）で並列に解析してくれる。各指摘には信頼度スコア（0〜100）が付き、デフォルトでは80以上のものだけが表示されるのでノイズが少ない。

人間のレビュアーはAIが「問題なし」と判断した部分をスキップして、AIが引っかけた箇所や設計判断が必要な部分に集中できるようになった。

なお、より深い解析が必要なチーム向けに、Anthropicは有償の「Claude Code Review」マネージドサービスも提供している。プラグインとの使い分けは、チーム規模やPRの量に応じて検討するとよい。

ただし、AIレビューは万能ではない。見落としや的外れな指摘もそれなりに出る。この点は課題4で詳しく触れる。

## 課題4：AIコードレビューは見落としと的外れな指摘が混在する

**ひとことで言うと：AIレビューは「気づいてほしいところ」は見落とし、「別にいいところ」を指摘してくる**

### 何が起きたか

AIによるコードレビューを使ってみて、2種類の問題がはっきり出てきた。

**見落とし**：本来は指摘してほしい問題をスルーしてしまう。特にツールやフレームワーク固有の深い知識が必要な問題は苦手で、ある設定ミスがレビューをすり抜けてコードに入ってしまったこともあった。

**的外れな指摘**：コードとしては問題ないのに、「改善の余地がある」とAIが判断して指摘してくるケース。プロジェクト固有の設計判断（例：あえて特定の共通モジュールを使わないルール）を理解できず、一般論ベースで指摘してくることがある。

### なぜ起きるか

AIレビューは「コードの文面」は読めるけれど、「なぜこの設計にしたか」という背景や文脈は持っていない。プロジェクト独自の制約や、意図的なトレードオフは、指示書にあらかじめ書いておかないと汲み取ってもらえない。

### 対策

AIレビューの担当範囲を **「一般的なコーディングルール違反の検出」に限定**して、ツール固有の問題やシステム設計の判断は **人間が担う** という役割分担を明確にした。

さらに、過去に混入したバグのパターンをチェックリストにまとめて、レビュー時の確認ポイントとして共有する取り組みも効果的だった。「このPRではこの確認をした」と記録を残すことで、見落としが起きたときの振り返りにもつながる。

## AIと人間の"ちょうどいい境界線"とはなにか

### もう少し解像度を上げると

境界線は「スキル」ではなく「タイミング」の問題です。

AIは**速く・広く・疲れずに**処理できる。人間は**文脈を読み・責任を持ち・例外を判断できる**。この特性の違いから、自然と役割が決まります。

| AIに任せていい | 人間が介在すべき |
| --- | --- |
| 選択肢の列挙 | 選択肢の採否 |
| ルール違反の検出 | ルール自体の妥当性判断 |
| 変更内容の実行 | 変更範囲の承認 |
| パターンに沿った実装 | 設計のトレードオフ判断 |

### 境界線が崩れるとき

この境界線が崩れる原因はほぼ一つで、**「AIが速いから、確認を省いても大丈夫だろう」という慣れ**です。

記事が繰り返し「承認ゲート」を強調しているのも、速さへの慣れが境界線を少しずつ侵食するからで、仕組みとして強制しないと人間側が手を抜き始める、という実感がこもっています。

結局のところ、**境界線は引いたら終わりではなく、意識的に維持し続けるもの**というのがこの記事全体の核心だと思います。

## さいごに

チームでClaude Codeを使ってみてわかったのは、AIの力を最大限に引き出すには **AIの自律性を信頼しつつ、人間がちゃんとガードレール（安全策）を設計する** ことが欠かせないということだ。

大事だった教訓を3つに絞ると、

1. **「承認ゲート」を作る**：AIに実装前に変更内容を宣言させて、人間がOKしてから実行する流れを仕組みにする
2. **テストはコスパで絞る**：AIの「毎回結果が変わる」性質を直接テストしようとせず、処理フロー・外部接続・分岐ロジックにフォーカスする
3. **AIレビューの役割を限定する**：コーディングルール違反の検出に絞って、ツール固有の問題や設計判断は人間が見る

Claude Codeはまだどんどん進化しているツールだ（この記事は2026年3月末時点の情報に基づいている）。今後のアップデートで、ここに挙げた課題の一部は自然に解消されていくかもしれない。

ただ、「AIで開発が速くなる」という期待に対して、「レビューや検証は同じペースでは速くならない」という現実のギャップは、AIの性能向上だけでは埋まらない。チームのプロセス設計・役割分担・品質基準の見直しを、AIの活用とセットで進めていくことが、現場に定着させるカギだと感じている。
