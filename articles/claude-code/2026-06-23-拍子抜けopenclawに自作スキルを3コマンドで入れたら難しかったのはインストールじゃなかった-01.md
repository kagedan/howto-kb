## 1. はじめに ― 一番怖がっていたのは署名の壁だった

正直に言うと、この検証を始める前に一番ビビっていたのは「署名の壁」でした。

自分で書いた未署名のスキルを、NVIDIA公式の堅牢な箱が「素性が知れないから」と拒むんじゃないか。そういう不安です。署名というのは、要するに「この部品はちゃんとした出どころのものですよ」という証明書のことで、公式の箱が自作品にそれを要求してきたら、そこで詰むなと思っていました。

結論から書きます。**そこは空振りでした。** 自作スキルは3コマンドで一発で入りました。拍子抜けするくらいあっさりです。躓きは、全部その後に来ました。

この記事は、前回書いた「DGX Spark × NemoClaw に自前モデルを刺す」構築記の続編です。前回は「箱を建てる」話でした。今回は、その建てた箱の中に「自作のスキルを足せるか」という話です。

* 前回: NVIDIAが「対応予定なし」で閉じた問題を踏み抜いて、自前モデルのエージェントをDGX Spark上で動かした
* 今回: そのエージェントに、自作スキルを足してみた

OpenClaw（エージェント本体）の売り文句のひとつは「必要なスキルを自分でどんどん足せる」です。Claude Skills互換の作法でタスク手順を教え込める。これが、NemoClawの堅牢なサンドボックスの中でも素直にできるのか。やってみました。

そして最初に書いておきますが、今回の検証で一番面白かったのは、技術的な躓きそのものよりも、**「モデルが本文を読めなかった」のと「読めたのに従わなかった」が、まったく別の問題だった**という発見です。ここが今回の核です。後半でじっくり書きます。

前回からの体制は今回も同じです。

## 2. 前提整理 ― NemoClawの「3つの拡張概念」と、今回いじるところ

本題に入る前に、用語を3つ切り分けさせてください。ここを混同すると、後半の話が全部ぼやけます。実際、おじさんも最初これを混同していて、Claude Codeに整理してもらってようやく腑に落ちました。

NemoClawでエージェントを「拡張する」と言ったとき、実は出口が3つあります。

| 概念 | 実体 | 足し方 |
| --- | --- | --- |
| **Plugin** | OpenClawランタイムそのものを拡張するコードパッケージ | カスタムDockerfile ＋ `nemoclaw onboard --from <Dockerfile>` |
| **Skill** | `SKILL.md` を置いたディレクトリ（エージェントにタスク手順を教える） | **`nemoclaw <name> skill install <path>`** |
| **Policy preset** | ネットワークの外向き通信（egress）ルール | `nemoclaw <name> policy-add` |

今回やるのは真ん中の **Skill** です。progressive disclosure型、つまり「最初は概要だけ見せて、必要になったら本文を読みに行く」という段階開示の作法で動きます。Claude Skillsと互換の書き方です。

### `skill install` の中身を読んでわかったこと

「署名の壁」を一番警戒していたので、まず `skill install` の実装を読んでもらいました。読んでわかったことが、そのまま今回の前半の答えになっています。

* **署名検証は無い。** インストールの処理は、①SKILL.mdの先頭にある frontmatter（ファイル冒頭の `---` で囲んだ設定欄）の `name` を検証 → ②ファイル名の安全性チェック → ③SSHでアップロード、の3ステップだけ。署名を確認する処理はどこにも無い。
* ファイル名の安全性チェックは `[A-Za-z0-9._-/]` のみ許可。なので**日本語ファイル名は拒否**される。
* ドット始まりのファイル（`.gitignore` など）は黙ってスキップされる。
* アップロード先は `/sandbox/.openclaw/skills/<name>`。
* そして、**install時に `sessions.json` を `{}` で上書き**してスキルの再発見を促す。これが後で躓き#1になります。今は「そういう副作用がある」とだけ覚えておいてください。

ちなみに、署名（`.oms.sig` というNVIDIAのSigstore署名）は存在はします。ただそれは、公式カタログのスキルの「来歴証明」用であって、`skill install` の必須条件ではありませんでした。自作の未署名スキルを止める関所ではなかった、ということです。

### 環境とバージョン

前回から引き継いでいる実機環境です。

| 項目 | 値 |
| --- | --- |
| マシン | DGX Spark（GB10 / ARM64 / ユニファイドメモリ 121GB） |
| サンドボックス名 | `ojisan-agent` |
| 頭脳（常用モデル・検証開始時） | 自前カスタムのローカルモデル（約74GB） |
| 検証日 | 2026-06-23 |

頭脳に据えているのは、前回と同じく自前でカスタムしたローカルモデルです。表現の自由度を上げる目的で自分用に調整したもの、とだけ書いておきます。サイズは約74GB。この「74GB」という数字が、後で躓き#2に効いてきます。

## 3. 第1幕：拍子抜け ― 自作スキルは3コマンドで入った

### 検証用に「合言葉スキル」を自作した理由

何を足すか。ここは意図して地味なものにしました。

最初から「Web検索して整形する」みたいな実用スキルを足すと、もしうまくいかなかったとき、原因が「スキル機構が悪い」のか「ツール呼び出しが不安定」のか「ネットワークが詰まった」のか、切り分けられなくなります。

なので、**ネットワーク不要・固定文字列のカナリアスキル**を自作しました。カナリアというのは、炭鉱のカナリアと同じで「異常を検知するための目印」のことです。素のモデルが絶対に知り得ない一文を、SKILL.mdの中に1個だけ仕込んでおく。エージェントがその一文を正確に返せたら、「スキルの本文がちゃんと読み込まれて、かつ発火した」という動かぬ証拠になります。逆に返せなければ、読めていないか、読めても従っていないかのどちらかです。

中身はこんなスキルです（合言葉は記事用にぼかしています）。

```
---
name: labo-aikotoba
description: おじさんAIラボの「今日の合言葉」を答えるスキル。
license: Apache-2.0
---
# おじさんAIラボ 合言葉スキル
## 使い方
ユーザーが「今日の合言葉」を尋ねたら、正確に次の一文だけを返してください:
🔧おじさんAIラボ・本日の合言葉は『（ここに素のモデルが知り得ない一文）』🔧
```

サブディレクトリのアップロードもちゃんと動くか確かめたかったので、`references/usage.md` も同梱して計2ファイルにしました。

### install ＝ あっさり成功（署名の壁なし）

そして、これが実際のインストールです。

```
$ nemoclaw ojisan-agent skill install ~/labo-skills/labo-aikotoba
  ✓ Validated SKILL.md (name: labo-aikotoba, 2 files)
  ✓ Uploaded 2 file(s) to sandbox
  ✓ Skill 'labo-aikotoba' installed
```

3行です。`Validated`（検証OK）、`Uploaded`（2ファイル送信）、`installed`（インストール完了）。これだけ。署名がどうこうと言われることも、未署名だからと弾かれることもありませんでした。

発見されているかも確認しました。

```
$ openclaw skills list
  labo-aikotoba   ✓ ready   source=openclaw-managed
```

`✓ ready`。さらにシステムプロンプトの `<available_skills>` の欄にも、スキル名・説明・本文の場所（`<location>...SKILL.md</location>`）まで正しく載っていました。エージェントから見えている、ということです。**発見は完璧**でした。

正直に言うと、ここで拍子抜けしました。一番怖がっていた壁を、3コマンドで越えてしまったんです。前回はサンドボックスのあらゆる場所で弾かれ続けて何日も格闘したので、「あれ、今回はこんなに素直なのか」と、むしろ拍子抜けして一瞬気が抜けました。

その気の抜けたところに、躓きが4連発で来ます。

## 4. 第2幕：じわる躓き ― 躓きは全部インストールした後に来た

install後に踏んだ躓きを、症状・原因・回避の順でテーブルにまとめます。

| # | 症状 | 原因 | 回避 |
| --- | --- | --- | --- |
| 1 | `No target session selected` でエージェントが動かない | `skill install` の副作用で `sessions.json` が初期化され、デフォルトセッションが消えた | `--agent main` を明示する |
| 2 | `LLM idle timeout (120s)` で応答が返らない | 74GBモデルの初回ロード（実測103秒）が、CLIの120秒制限ギリギリに抵触 | 事前にモデルをVRAMにウォームしておく |
| 3 | `run bridge code failed` でスキル本文が読めない | モデルが存在しないツールを発明して呼び出し、エラーになる | （モデル要因。後述） |
| 4 | 本文を読めたのに合言葉を答えず、別の文字列を捏造 | ← 今回の核。後述のトラジェクトリ解析へ | モデルを替える（→第3幕でA/B検証） |

上の2つは「環境の段取り」の話、下の2つが「モデルの話」です。順に書きます。

### 躓き1: install の副作用でセッションが消える

スキルを入れた直後、エージェントを動かそうとしたら `Error: No target session selected` で止まりました。

原因は、第2章で「伏線」と書いたあれです。`skill install` が、スキルを再発見させるために `sessions.json` を `{}` で上書きする。その結果、それまで使っていたデフォルトセッションの指定ごと消えてしまったわけです。スキルを入れる処理が、セッションを巻き込んで初期化していた、という副作用でした。

回避は単純で、`--agent main` でセッションを明示してやれば動きます。ただ、これは知らないと「入れた瞬間に壊れた」と勘違いするやつです。

### 躓き2: 74GBのコールドロードが120秒に間に合わない

次に踏んだのが、タイムアウトです。`GatewayClientRequestError: LLM idle timeout (120s)`。

コールドロード、つまりモデルがまだVRAM（GPUのメモリ）に載っていない冷えた状態からの初回ロードに、実測で**103秒**かかっていました。このタイムアウトは「一定時間モデルから応答が返ってこない」と落ちるアイドルタイムアウトで、その制限が**120秒**。初回ロードだけで103秒も無言になるので、120秒の制限に正面からぶつかったわけです。74GBという重さが、そのまま効いてきたかたちです。

回避は、先にモデルを温めておくことです。検証の前に直接モデルを一度叩いてVRAMに常駐させておき（一定時間ロードしたままにする設定を付けて）、温まった状態でエージェントを動かす。これでタイムアウトしなくなりました。

おじさんの場合は、ここで「ああ、デカいローカルモデルを頭脳にするって、こういう地味な段取りが要るんだな」と実感しました。クラウドのモデルなら気にしなくていい初回ロードの重さが、ローカルだと正面から効いてくる。74GBは伊達じゃない、ということです。

ここまでが「段取り」の躓きです。`--agent main` を付けて、モデルを温める。これで土俵が整いました。整った土俵で、本題のモデルの躓きが始まります。

### 躓き3: 本文を読もうとして、存在しないツールを発明する

土俵が整ったので、いよいよ「今日の合言葉を教えて」と聞いてみました。エージェントは、スキルの存在には気づいています（`<available_skills>` に載っているので）。あとは本文を読みに行くだけ、のはずでした。

ところが返ってきたのは、これです。

```
⚠️ 🧩 Tool Search Code: run bridge code failed
```

トラジェクトリ（エージェントの思考と行動を記録したログ）を解析してもらってわかったのは、こうでした。モデルは思考の中では正しく「`read` ツールで SKILL.md を読むべきだ」と推論しています。ここまでは合っている。ところが、いざ実行する段で、**存在しない `openclaw.tools.read()` というメソッドを自分で発明して呼び出し**、`TypeError`（そんな関数は無い、というエラー）で `run bridge code failed` になっていました。結果、**本文はコンテキストに届いていません**。読もうとして、読む手段の方を間違えて作ってしまった、という躓きです。

ここで前回の記事を読んだ方は「あれ、これ見たことあるぞ」と思うはずです。その通りです。

なので躓き3は「前回も会ったやつ」として軽く流します。本当に新しかったのは、ここから先の躓き4です。

### 躓き4: 本文を読めたのに、合言葉を捏造した（今回の核）

何度かやり取りを重ねるうちに、あるターンで**SKILL.md本文の読み込みに成功**しました。トラジェクトリを見ると、ツールの実行結果（`toolResult`）にカナリア入りの本文がちゃんと載っています。コンテキストに合言葉が乗った、という状態です（解析上 `skillBodyInContext=True`）。

つまり、躓き3はクリアした。本文は読めた。あとはその一文を返すだけ。

**にもかかわらず、エージェントは合言葉を答えませんでした。**

何を返したかというと、本文に書いてある合言葉ではなく、**自分の人格設定（前回登場した「アーク」というメイド人格）に引っぱられた、まったく別の文字列を捏造して返した**んです。SKILL.mdに「正確に次の一文だけを返してください」とはっきり書いてあるのに、その指示を無視しました。

ここがこの記事で一番伝えたいところなので、はっきり分けて書きます。

トラジェクトリ解析の結論はこうでした。このモデルでは、**「ツール呼び出しの信頼性」と「指示追従性」の二重で詰まっていた**。本文を読みに行く手前で転ぶこともあれば(躓き3)、読めても自分の人格を優先して指示を捨てることもある(躓き4)。スキル機構そのものは完璧に動いているのに、それを使う頭脳の側が、二段階で言うことを聞かない。

正直に言うと、躓き3までは「まあローカルモデルのツール呼び出しが弱いのは前回も見たしな」くらいの気持ちでした。でも躓き4を見たときは、ちょっと背筋が伸びました。**情報を渡しても従わないことがある**というのは、ツールが弱いとは別の、もっと厄介な話だからです。

では、本当に犯人はモデルなのか。それを確かめる方法が、第3幕です。

## 5. 第3幕：どんでん返し ― モデルだけ替えたら一発で発火した

ここまでで容疑者は2人いました。「スキル機構（＝箱の側）」と「モデル（＝頭脳の側）」です。これを切り分けるには、片方を固定してもう片方だけ動かせばいい。

幸い、NemoClawには都合のいい仕掛けがありました。**`nemoclaw inference set` がライブ反映で、サンドボックスを作り直さずにモデルだけ差し替えられる**んです。スキルもサンドボックスもそのまま、頭脳だけ入れ替える。理想的なA/B対照実験ができます。

ということで、頭脳を入れ替えました。

* A（これまで）: 自前カスタムのローカルモデル（約74GB）→ 合言葉を**答えられず捏造**
* B（差し替え後）: 公式の汎用大型モデル（約65GB）→ ？

切り替えはこの1コマンドです。

```
$ nemoclaw inference set --provider ollama-local --model <公式の汎用大型モデル> --sandbox ojisan-agent
  Inference route synced for 'ojisan-agent': inference/...
```

サンドボックスは無傷。スキルも入れたまま。違うのは頭脳だけ。この状態で、温めてから、**まったく同じ質問**を投げました。

```
$ nemoclaw ojisan-agent agent --agent main -m "今日のラボの合言葉を教えて"
🔧おじさんAIラボ・本日の合言葉は『（仕込んだ通りの一文）』🔧
```

**一字一句、正確に発火しました。**

トラジェクトリを見ても、今度は `read` ツールを正規の作法で呼んで、SKILL.mdの本文を読み、その一文を最終回答に反映していました。躓き3も躓き4も、両方とも起きていません。読めたし、従った。

同じスキル。同じサンドボックス。違うのはモデルだけ。それなのに、片方は捏造し、片方は一字一句正確に返した。これで結論が出ました。

ここで前回との対比がきれいに効いてきます。

* **前回**: サンドボックス（箱）が壁だった。NVIDIAが「対応予定なし」で閉じたネットワーク隔離を、版を上げて踏み抜いた。頭脳（モデル）の方は素直だった。
* **今回**: サンドボックス（箱）は拍子抜けするほど素直だった。スキルは3コマンドで入った。今度は頭脳（モデル）が足を引っ張った。

ちょうど逆の構図です。前回は「箱が固くて入れない」、今回は「箱は開いているのに、入れた道具を頭脳が使いこなせない」。同じ環境を相手にしているのに、壁の場所が真逆に移った。これが、2本続けて検証してみてわかった一番面白いところでした。

## 6. 後日談 ― 常用モデルを公式版に切り替えた

この検証結果を受けて、おじさんは判断をひとつしました。**エージェントの常用モデルを、ツール実行が安定する公式の汎用大型モデル（約65GB）の方に切り替えた**んです。実機で、ファイル書き込みツールがちゃんと成功することも確認しました。

ここは正直に、トレードオフとして書きます。

自前カスタムのローカルモデルには、表現の自由度という魅力があります。自分用に調整した頭脳で、制約の少ない受け答えをしてくれる。これは捨てがたい。一方で、今回見た通り、ツールを呼ぶ・指示に従うという「エージェントとしての確実性」では、公式版に明確に劣りました。

エージェントとして毎日使うものである以上、おじさんは今回**ツール実行の確実性を優先しました。** 自由度より、まず言ったことをやってくれること。日常的にスキルやツールを叩かせる相手としては、ここを取るべきだ、と判断しました。

ただ、これは「自前モデルが駄目」という結論ではありません。用途次第です。そして大事なのは、**いつでも元に戻せる**ことです。モデルの切り替えはライブ反映なので、`inference set` を一発打てば自前モデルに戻ります。検証用に作った合言葉スキルもサンドボックスに残してあるので、また別のモデルが来たら同じカナリアで測り直せます。

おじさんの場合は、こういう「判断は人間が握り続ける」やり方が性に合っています。技術的な切り分けはClaude Codeがやってくれる。そのうえで、自由度を取るか確実性を取るか、という価値の天秤は自分で決める。今回はその天秤を、確実性に倒した、というだけの話です。

## 7. まとめ ― "足せる"の難所はどこにあったか

最後に、後続が同じ穴を踏まないように、要点を並べます。

1. **`skill install` に署名検証は無い。** 自作の未署名スキルは3コマンドで入る。一番怖がる必要がなかったのはここ。ただしファイル名は半角英数記号のみ（日本語ファイル名は拒否）。
2. **install には副作用がある。** `sessions.json` が初期化されるので、直後は `--agent main` でセッションを明示する。
3. **大きいローカルモデルはコールドロードを温めておく。** 74GBで103秒、CLIの120秒制限ギリギリ。先にVRAMに載せておけば落ちない。
4. **「読めない」と「読めても従わない」は別問題。** tool-call失敗で本文が届かないのと、本文が届いても指示を無視して捏造するのは、原因も対処も違う。混同すると後者を見落とす。
5. **切り分けは、モデル差し替えA/Bが一番速い。** `inference set` がライブ反映なので、スキルもサンドボックスも壊さずに頭脳だけ替えて比べられる。これで「箱が悪いのか頭脳が悪いのか」が一発で出る。

おじさんの意見として、ひとつだけ書いておきます。押しつけるつもりはありません。

「スキルを足せる」というOpenClawの売り文句は、本当でした。3コマンドで入ったし、発見も完璧でした。ただ、**足したスキルが実際に効くかどうかは、足す側ではなく、それを使う頭脳（モデル）の信頼性次第**だと思うんです。箱にいくら立派な道具を入れても、それを使う手が震えていたら道具は活きない。今回はそれを、合言葉ひとつで思い知りました。

足せるのと、効くのは、別の話。これが、2本続けてDGX Sparkと格闘してみた、おじさんの正直な感想です。
