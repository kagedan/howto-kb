## この更新、正直かなり実務寄り

今回のポスト、最初は「細かい改善だな」で流れがちなんですが、個人的には**かなり本質的な更新**だと思いました。

理由は単純で、AIコーディングツールって結局、**長く読めるか**と**安全に触らせられるか**で使い勝手が決まるからです。 Codeは、ターミナルで動くAI開発支援ツールです。コードを読んで、修正して、コマンドも叩いてくれる。CursorやGitHub Copilotと競合する文脈で語られやすいですが、強みは**長時間の作業を崩れにくく回せること**にあると思っています。Opus 4.6は大きなコードベースでの信頼性やコードレビュー性能を強め、Sonnet 4.6も長文コンテキストやコーディング能力をかなり伸ばしています。 ---

## 何が変わったのか

ポストの軸は、ざっくり言うと二つです。

ひとつは、**出力トークン上限の引き上げ**。今回の更新抜粋では、Opus 4.6の既定の最大出力が64kまで上がり、Opus 4.6とSonnet 4.6で上限が128kまで広がったと読めます。つまり、AIが一回で返せるコード量や説明量が増える、ということです。長いリファクタ、複数ファイルの修正案、テスト付きの大きめの提案をまとめて返しやすくなる。ここはかなり効きます。 もうひとつは、**allowRead の追加**です。もともとClaude Codeの設定では、`denyRead` で読ませたくないパスを止める仕組みや、権限ルールで `Read(...)` を制御する仕組みがありました。今回の抜粋では、その「読ませない」設定の中でも、**必要な場所だけ再許可する**方向に一段細かくなった、と受け取れます。 > 長いコードを一気に返せる  
でも、読ませる範囲は細かく絞れる

この組み合わせが、地味に見えてかなり強いんですよね。

## なぜ重要なのか

AI駆動開発でいちばんつらい瞬間って、モデルが賢くない時より、**あと少しで使えるのに詰めが甘い時**だったりします。

長い変更を途中で切る、共有ロジックを重複して書く、触らせたくない設定まで見に行く。こういう細かなストレスが積み重なると、結局人間が細切れに指示して面倒を見ることになる。  
Sonnet 4.6は、 Codeの初期テストでSonnet 4.5より好まれた割合が約70%とされていて、「修正前に文脈をよく読む」「重複せず共有ロジックに寄せる」といった評価が出ています。ここに今回の出力上限拡大が重なると、**読む力**と**返す力**の両方が底上げされる。だから体感が変わりやすいんだと思います。  
ちなみに公開されているモデル仕様では、 .6系は1Mコンテキスト対応で、API上のmax outputはOpus 4.6が128k、Sonnet 4.6が64k表記です。なので今回のポストは、API仕様そのものというより、**Claude Code側の運用上限や既定値の調整**として読むのが自然だと思います。ここは少し注意して見たほうがいいです。platform.claude.com

## AI駆動開発の視点で見ると

個人的には、今回の更新は「AIがさらに賢くなった」というより、**AIをチームに入れやすくした**更新だと思っています。

個人開発なら長文出力はそのまま速度になりますし、スタートアップや受託開発では権限制御の細かさがそのまま導入ハードルを下げます。

CursorやGitHub Copilotも便利ですが、これから差がつくのは、単発補完の精度だけじゃなくて、**大きい文脈を持ったまま安全に作業を継続できるか**じゃないかなと思います。AIに全部任せる世界というより、**AIに長い作業を渡して、人間はレビューと意思決定に寄る**形です。 > セキュリティは締める  
でも必要な場所だけ開ける

この感覚は、今後の技術スタック選定でもかなり大事です。モデル性能だけでなく、**権限設計・監査性・運用のしやすさ**まで含めて選ぶ流れが強くなるはずです。

正直、派手さだけならもっと目立つ更新はいくらでもあります。

でも、毎日Claude Codeを触る人ほど、今回みたいな改善の価値は大きく感じるはずです。こういうアップデートが積み重なると、AIは「便利な補助」から「普通にチームの一員」へ近づいていく。僕はそこがいちばん面白いです。
