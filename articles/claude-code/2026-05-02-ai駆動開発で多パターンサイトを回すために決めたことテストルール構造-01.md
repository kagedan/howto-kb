## 軽い気持ちで始めたら、パターンが爆発した

前提として、StatPlayの開発はClaude Codeを使って、主にSonnet 4.6とOpus 4.7で進めています。Opusはクレジット消費が大きいので、普段はSonnetで作業して、設計判断や大規模な変更のときだけOpusに切り替える運用です。この使い分け自体もAI側に提案してもらっています。

StatPlayはもともとダークモードだけで作っていました。サイバーパンク調の暗い背景にネオンカラーのグラフが映える、そういうデザインです。

ところが、作っていくうちにグラフの画像をいろんなところで使いたくなってきた。資料に貼る、SNSに載せる、スライドに入れる。暗い背景のスクリーンショットって、意外と使いにくいんですよね。「ライトUIもあったほうがいいかな」と、わりと軽い気持ちでライトモードを追加しました。

英語対応も似た流れです。「英語でも出せばいいのでは」くらいのノリで始めました。統計の概念は言語に依存しないし、グラフが主役だから翻訳量もそこまでじゃないだろうと。

さらに、StatPlayにはハブページ（index.html）と個別トピックページがあります。同じグラフが両方に登場する。つまり、1つのインタラクティブグラフに対して「ハブ＋個別ページ」×「日本語＋英語」×「ダーク＋ライト」×「デスクトップ＋スマホ」のパターンが存在するわけです。

どれも「あったほうがいいな」で気軽に足していった結果でした。1つ1つは自然な判断だったんですが、組み合わせが掛け算で増えていくことに、最初は気づいていなかった。

Claude（AI）にコンテンツを追加してもらうと、指示した通りに全パターン書いてくれます。律儀に、忠実に。で、最初のうちは「ちゃんと全部やってくれるじゃん」と思っていたんですが、コンテンツが増えてくると、ほころびが見えてきた。

英語版を開いたらボタンのラベルだけ日本語のまま。ライトモードに切り替えたらネオンカラーが白背景に溶けて読めない。ハブと個別ページでグラフの挙動が微妙に違う。AIは何も言わずに納品してくるので、全パターン目視で確認してようやく気づく。

1回や2回ならいいんですが、これがじわじわ蓄積するんです。今のところは制御できていますが、そこに至るまでにいくつかの段階がありました。

## まずテストで検知できるようにした

最初にやったのはテストです。

```javascript
// 実コード (test_site.mjs): セクション内の日英ペアが揃っているか
const titlesJa = (sec.match(/<div class="panel-title" data-lang="ja"/g)||[]).length;
const titlesEn = (sec.match(/<div class="panel-title" data-lang="en"/g)||[]).length;
ok('T85 bilingual parity', titlesJa === 3 && titlesEn === 3);
```

```javascript
// ダーク/ライト両テーマでコントラスト比4.5:1以上を保証
const crDark  = contrast([0,0xf3,0xff], [0x0a,0x0f,0x2c]);
const crLight = contrast([0,0x70,0x8c], [0xf0,0xf3,0xfa]);
ok('T38 contrast >=4.5 on both themes', crDark >= 4.5 && crLight >= 4.5);
```

日英のペアが揃っていなければテストが落ちる。コントラストが足りなければ落ちる。今は212件が毎回走っていて、修正漏れを機械的に潰してくれます。

ただ、テストって**起きた問題を検知する**仕組みなんですよね。**問題を起こさない**仕組みではない。コンテンツを追加するたびにAIが同じ種類のミスを繰り返して、テストが落ちて、また直して……というループ自体がコストになっていました。

## 仕様を割り切って、レギュレーションに書いた

次にやったのは、仕様の割り切りを決めることでした。

「将来にわたって、すべてのコンテンツは日本語と英語の両方で存在させる。ダークとライトの両方で表示できる状態にする。そうじゃないパターンは作らない」——これを方針として確定させました。

たとえば、StatPlayには「日本人の年収データで重回帰を体験する」というコラムがあります。

- 日本語版: https://statplay.sasailab.com/columns/income_prediction.html
- 英語版: https://statplay.sasailab.com/en/columns/income_prediction.html

本当は各国ごとにローカライズしたかったんです。アメリカなら州別の所得データのほうが英語話者にとって実感のある例になる。でも、そこまでやると手が回らない。「重回帰で予測ができる」という体験自体は、データが日本のものでも成立します。だから英語版では「In Japan, the income distribution looks like this」という形でそのまま載せています。

英語圏の読者には少し物足りないかもしれません。でも「じゃあ英語版を作らない」にしてしまうと、ビルドスクリプトに条件分岐が要る。テストに例外処理が増える。「このページだけ英語版を生成しないで」みたいな特殊ルールが1つできると、AIはそれを他にも適用しようとし始めるんです。

だから「すべてのコンテンツは日英両方で存在する。機械的にローカライズする。例外なし」と決めて、CLAUDE.md（AIへのレギュレーションファイル）に書き込みました。テーマやレスポンシブも同じです。ダーク/ライトでHTML構造は完全に同じ、違いはCSS変数の値だけ。スマホ幅でも情報量は減らさない。

レギュレーションに方針を書いてからは、確かにAIは迷わなくなりました。新しいコラムを追加するとき「英語版も生成しますか？」と聞かれなくなった。黙って日英ペアで出してくる。

ただ、正直に言うと、ルールを書いただけでそれを信用して開発を進めることはできませんでした。少し矛盾して聞こえるかもしれませんが、ルールを書けばAIは「迷いにくく」なる。でも「絶対に守る」保証はない。長いセッションの途中で抜け落ちることもある。だからレギュレーション＋テスト（検知）はセットで必要でした。

---

## 「そうしか作れない構造」に変えた

ルール＋検知のさらに先として、構造的にルール違反が起きようがない状態にしました。コンテンツを共通パーツ（パーシャル）として1箇所に書き、ビルドで展開する構成です。

```html
<!-- content/partials/ci_main.html — 信頼区間のCanvas部分 -->
<div class="panel" data-autorun="ciRun">
  <div class="controls">
    <div class="ctrl">
      <label><span data-lang="ja">信頼度</span><span data-lang="en">Confidence</span>
        = <span class="val" id="ciLvlVal">95%</span></label>
      <input type="range" id="ciLvl" min="80" max="99" step="1" value="95">
    </div>
    <button class="btn" id="ciRun">
      <span data-lang="ja">▶ 300回 生成</span>
      <span data-lang="en">▶ Generate 300</span>
    </button>
  </div>
  <canvas id="ciCanvas" height="240" aria-busy="true"></canvas>
</div>
```

`data-lang="ja"` と `data-lang="en"` が1ファイルに同居。index.html側にはマーカーを置くだけ：

```html
<section id="ci" class="reveal">
  <!-- @panel:ci_main --><!-- /@panel -->
</section>
```

ビルドスクリプトがマーカーを見つけてパーシャルを注入し、言語別にフィルタします：

```python
# scripts/build_topics.py — パーシャル注入の実コード
def inject_partials(html: str, lang: str | None = None) -> str:
    for partial_file in sorted(PARTIALS_DIR.glob('*.html')):
        panel_id = partial_file.stem
        pattern = re.compile(
            rf'<!-- @panel:{re.escape(panel_id)} -->.*?<!-- /@panel -->', re.DOTALL)
        content = partial_file.read_text(encoding='utf-8').strip()
        if lang:  # 個別ページ: 片方の言語だけ残す
            soup = BeautifulSoup(content, 'html.parser')
            strip_lang(soup, lang)
            content = str(soup)
        html = pattern.sub(lambda _: replacement, html)
    return html
```

ハブには `lang=None`（日英両方残す）、個別ページには `lang='ja'` or `'en'`（単言語）で注入。同じソースから用途に応じて出し分ける。「ハブと個別ページで内容が食い違う」「片方の言語だけ存在する」が構造的に発生しない。

開発中盤のv2.0頃からこの構成に移行して、やってよかったなと思っています。2000行超のHTMLを44個のパーシャルに分割するリファクタリング自体もAIに一気にやってもらいました。人間のエンジニアとデザイナーでやったら数週間かかる規模の作業です。途中で何回か壊れましたが、「動かない、こういう症状」と伝えれば自分で直してくれる。次からはテストで追加して比かかるようにする。テスターも半導体なので気楽なものです。リアルの開発現場でやるとかなり危ない炎上の現場です。

一つ言っておくと、「テストを書いたほうがいい」「構造自体を変えたほうがいい」という提案をAIが自発的にしてきたことは一度もありません。すべてこちらから持ちかけて初めて「いい考えですね」と返ってくるレベルです。コードをどう書くかについては非常に強いけれど、サイトの保守や運用の判断は出てこない。全部お任せにしてしまうとブラックボックス化して危ない。

## たまたま、こういう形になった

そう考えると、まだAI駆動開発の成立条件は限定的に思えます。少なくとも2026年5月の今では。StatPlayがこの進め方で回っているのは、サーバーサイドスクリプトを使っていないから。ユーザー管理も課金もDBもない。これがマイグレーションや認証や決済が絡んでくると、話はまったく変わってくると思います。何倍も工数がかかるし、維持し続けるのが大変になる気がしています。

最初から「静的サイトだけでAI駆動の利点を最大化しよう」なんて考えていたわけじゃないんです。作っていくうちに「このスコープならAIとうまく回るな」と気づいて、そこに寄せていった。StatPlayの目的は「統計の概念を、触ってイメージできるようにする」こと。それにはサーバーサイドが不要で、静的サイトで十分だった。たまたまAIが得意な領域にハマっていた、というのが正直なところです。

計画というよりは、試行錯誤の末にこういう形に落ち着いた。

……まあ、AGIが実現したら、こういう「スコープの割り切り」も「ルールブックの設計」もAIが勝手に考えてくれるのかもしれませんね。そうなったら私は何をすればいいんでしょう。とりあえず今は、ルールブックを書く仕事がまだ人間に残されているようなので、もうしばらくこのやり方で続けていこうと思います。
