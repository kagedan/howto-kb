AIエージェントのAIモデルはできるだけ差し替え可能にしておいた方がよい、という話について書きます。

## はじめに

自分は Slack から呼び出せるAIエージェントを運用しています。最初は Claude Code を Slack から使えたら便利そう、くらいの軽い気持ちでした。

ただ、使っているうちに役割が少しずつ変わってきました。Webアプリ開発、Flutter、スマートグラス（Even G2）用アプリ開発、研究パイプラインの実行など、個人開発の入口が Slack に寄っていきました。気づくと、これは単なるチャットUIではなく、自分の作業環境の一部になっていました。

そこで実感したことが、特定のAIベンダーや CLI にどこまで依存してよいのか、という点です。この記事では、自作 Slack AI bot でLLMアダプタという抽象化レイヤーを作り、Claude Code と Codex CLI を差し替え可能にした話をします。

## Slackが作業環境になった

現在運用しているAIエージェントは、自宅の Mac mini 上で動いています。

Slack で bot にメンションすると、Mac mini 上の `worker.py` がリクエストを受け取り、Claude Code を起動して、Slack のスレッドに返答します。

↓ 実際の様子はこんな感じです（2つ目はAIが自律的に動く例）  
![](https://static.zenn.studio/user-upload/b832609e8afb-20260526.png)

<https://x.com/aokiti_tech/status/2041152629175386598>

これだけ見ると、直接 CLI で Claude Code を叩いたり、`/loop` や Agent Teams のような仕組みを使えばいいじゃんとなるかもしれませんが、Slack を入口にすると少し性質が変わります。チャンネルが作業コンテキストになり、スレッドが作業ログになり、通知がそのまま完了報告になります。

また、この bot は現時点で60以上の Skills があります。Webアプリ開発、スマホアプリ開発、スマートグラス（Even G2）アプリ開発、研究など用途ごとに Slack チャンネルに分け、必要な Skills だけを注入する仕組みを自作することで、コンテキスト節約にも繋がっています 。

ここまで使うようになると、AIエージェントは「便利なツール」から「日常的に依存する作業基盤」に近づいていきます。すると、便利さとは別の問題が見えてきます。

## 便利になるほど移行しづらくなる

Claude Code はとても便利です。コーディングに強く、ツール実行もでき、セッション継続もでき、Skills のような仕組みもあります。普通に使う分にはかなり快適です。

一方で、便利な機能をそのままアプリケーションの中心に置くと、少しずつ移行しづらくなります。

たとえば、以下のような変化は普通に起こります。

* 価格体系や利用条件が変わる
* コーディングに強いモデルの勢力図が変わる
* CLI や SDK の入出力仕様が変わる
* あるモデルでは使える機能が、別のモデルでは使えない

ちょうど5月中旬には、Claude Code を非対話的（ヘッドレス）に実行する `claude -p` がサブスク利用から外れることが発表され、実質的に値上げになったとのことで大きな話題になりました。

<https://x.com/ClaudeDevs/status/2054610152817619388>

これを機に、自分もSlack botの基盤エージェントを Codex に変更しようと取り掛かりました。

ただ、LLMは進化が速いです。ある時期は Codex が自分の用途に合っていても、別の時期には Gemini CLI の方が使いやすいかもしれません。今後は Qwen 系や DeepSeek 系のようなオープンウェイトモデルを、ローカルまたは自前 GPU で使う選択肢も増えていくと思います。

このとき、AIエージェントの内部が Claude Code の出力形式や挙動に強く依存していると、「モデルを変えたい」だけなのに、Slack 表示、ツール呼び出し、セッション管理、エラー処理まで全て修正し、細かい挙動を使いながらデバッグし続けることになります。

## 課題は「モデル」よりも「周辺仕様」にある

一般的なプロダクトでAIモデルを差し替えると聞くと、単に API キーやモデル名を変えるだけのように見えます。

しかし、AIエージェントの場合はそこまで単純ではありません。モデル本体だけでなく、その周辺にある仕様もまとめて変わります。

たとえば、Claude Code と Codex CLI では、ストリーミング出力の形式が違います。ツール呼び出しの表現も違います。thinking ブロックの有無も違います。セッション継続の方法も違います。

bot 側から見ると、欲しいものはシンプルです。

* テキストが少しずつ返ってくる
* ツールを実行したことが分かる
* 実行が成功したのか失敗したのかが分かる
* 次回の会話に使うセッション ID が分かる

![](https://static.zenn.studio/user-upload/9192d7430d3e-20260526.png)

一方で、これらを取り出す方法はモデルや CLI によって違います。

つまり、差し替えたいのは「モデル名」だけではなく、「モデルを使ったときに起きるイベントの解釈」そのものです。

ここを bot 本体に混ぜると、LLMを取り換えるたびに大きな修正が伴い、複雑になります。そこで今回、変更容易性を高めるために、モデル独自の仕様を外側に追い出すアーキテクチャの構築に取り掛かりました。

## AIモデルの仕様をクリーンアーキテクチャの「詳細」として扱う

ここで使った考え方は、クリーンアーキテクチャの依存ルールです。ソースコードの依存は内側に向かい、ユースケースは外側の円にある具体的な技術やデータフォーマットを知らないようにします。

![](https://static.zenn.studio/user-upload/d46f4b689f01-20260526.png)

Webアプリケーションで考えると分かりやすいと思います。

たとえば、ユースケースが MySQL の SQL 文や DynamoDB の SDK 呼び出しを直接知っているとします。この場合、DB を差し替えたいだけなのに、ユースケースの中身まで変更する必要が出てきます。

そこで、ユースケース側には Repository や Gateway のインターフェースを置き、MySQL や DynamoDB の具体的な実装は外側に置きます。ユースケースは抽象だけを知り、どの DB から読むかは知りません。

AIエージェントでも同じことが起きていると考えました。

Claude Code や Codex CLI は便利ですが、bot のユースケースから見ると、それらの呼び出し方、ストリーミング形式、ツール呼び出し表現、セッション継続方法、エラー表現は「詳細」です。言い換えると、Frameworks & Drivers 側に置きたい関心事です。

この設計では、AIモデルの CLI を直接扱う処理をユースケースから分離し、LLMアダプタとして扱います。LLMアダプタの役割は、Claude Code や Codex CLI などの外側のデータフォーマットを、bot 内部の共通イベントへ変換することです。この記事では、この共通インターフェースを `LLMAdapter`、bot 内部で扱う共通イベントを `LLMEvent` と呼ぶことにします。用語としては、Interface Adapter / Gateway に近い役割です。

* bot 本体は Claude Code や Codex CLI の出力形式を知らない
* 各 LLM アダプタが、モデルや CLI 固有の出力を共通イベントに変換する
* bot 本体は、共通イベントだけを見て Slack に表示する

![](https://static.zenn.studio/user-upload/19dd45d9e6e5-20260526.png)  
*図： Use Case から見た、DBとAIエージェントの依存関係を簡略化して比較*

Claude Code 用のLLMアダプタは、`claude -p --output-format stream-json` の出力を読み、bot が扱いやすいイベントに変換します。Codex CLI 用の LLM アダプタは、`codex exec --json` の出力を読み、同じイベント形式に変換します。

これにより、外側のデータフォーマットが変わっても、その影響をLLMアダプタ側に閉じ込めやすくなります。AIモデルや CLI を差し替えるときも、ユースケースや Slack 表示ロジックへの影響を小さくできます。

## すべてを同じように扱うわけではない

ただし、抽象化するときに「すべてのモデルを完全に同じものとして扱う」必要はありません。

Claude Code では thinking ブロックや SSH 経由のリモート実行に対応できます。一方で、Codex CLI やオープンウェイトな LLM では同じ方法では扱えない機能もあります。画像入力に対応しているか、セッション継続に対応しているかも、LLMアダプタごとに違います。

そこで、各モデルが対応している機能を `manifest.toml` に宣言します。

```
[capability]
supports_thinking = true
supports_resume = true
supports_images = true
supports_remote_exec = true
supports_streaming = true
```

bot 本体はこの capability を見て、機能を有効化するかどうかを決めます。

たとえば、リモート実行が必要な Skill が選ばれたとき、現在のLLMアダプタが `supports_remote_exec = false` なら、その場で対応していないことを返します。`supports_resume = true` なら前回の session\_id を使って会話を継続します。

LLMアダプタごとの差分を manifest に出しておくと、「このモデルでは何ができて、何ができないのか」が見えます。すべてのモデルで同じ体験を完全再現するのではなく、各モデルが対応している能力を明示し、その範囲で差し替えられるようにする、という考え方です。

## LLMアダプタの契約をテストする

抽象化しても、実際に差し替えたときに動かなければ意味がありません。

ここも、DB や外部 API の差し替えと同じように考えられます。

たとえば `UserRepository` というインターフェースを作ったとしても、MySQL 実装と DynamoDB 実装で返す値の形やエラーの扱いが違っていたら、結局ユースケース側が壊れます。抽象を作るだけでは不十分で、「その実装が抽象の契約を満たしているか」を確認する必要があります。

AIモデルでも同じです。

ただし、ここでテストしたいのは LLM の回答内容そのものではありません。LLM は非決定的なので、自然言語の出力を完全に固定するのは難しいです。テストしたいのは、各LLMアダプタが `LLMAdapter` として期待される入出力の契約を満たしているかです。

この bot では、各LLMアダプタに対して契約テストを用意しています。ここでいう契約テストは、各実装が `LLMAdapter` の契約に沿って共通の `LLMEvent` を返せるかを確認するテストです。たとえば以下を検証します。

* `TextDelta` と `ResultEvent` が返る
* 致命的エラーは例外として漏れず `ErrorEvent` として返る
* ストリーミング対応のLLMアダプタでは、完了前に進捗イベントが返る
* resume 対応のLLMアダプタでは、前回セッションを継続できる
* ツール呼び出しが `ToolUseStart` / `ToolUseStop` として対応して返る

これらは実際に LLM を動かしてテストします。

![](https://static.zenn.studio/user-upload/2da241951c9e-20260526.png)

また、ここでも `manifest.toml` が効いてきます。`manifest.toml` は、そのアダプタが契約のどの部分に対応しているかを宣言するものです。  
たとえば `supports_streaming = false` のLLMアダプタの場合、ストリーミングのテストは skip します。`supports_images = true` なら画像入力のテストを実行します。

つまり、「このモデルはこの機能に対応している」と宣言したなら、その機能が実際に契約どおり動くことをテストで確認できます。

これにより、新しいLLMアダプタを追加するときの負担がかなり減ります。Qwen 系や DeepSeek 系などのLLMアダプタを追加する場合も、`manifest.toml` を書き、`LLMAdapter` を実装し、契約テストを通す、という流れにできます。

新しいAIモデルを追加するときは「このテストを通せるようにLLMアダプタを実装して」とAIに依頼できます。

## 抽象化して見えてきた境界

今回の実装で整理できたのは、bot 本体から見た実行イベントの差し替えです。

一方で、抽象化しても残る差分があります。渡した Skills をどれくらい適切に読めるか、長いタスクをどこまで自律的に進められるか、ツール実行後に自分で状況を立て直せるか、といった振る舞いはモデルごとに違います。

こうした振る舞いの差は、イベント形式を揃えるだけでは消えません。契約テストで最低限の I/O は確認できますが、タスク遂行能力や指示追従性までは完全には固定できません。

モデルを差し替えられるようにするというと、何でも簡単に置き換えられるようにする話に見えるかもしれません。しかし実際には、「設計で吸収できる部分」と「モデル選定や運用で受け止める部分」を分けて理解できるようにする話でもあります。

この境界が見えるだけでも、新しいモデルを試すときに、どこを実装で揃え、どこを評価や運用で見るべきか判断しやすくなると思います。

## おわりに

AIエージェントが大きくなるほど、特定のAIモデルやベンダーへの依存は重くなっていきます。

Claude Code は便利ですし、Codex CLI も便利です。今後はローカル LLM もさらに現実的になっていくと思います。だからこそ、どれか一つにアプリケーション全体を寄せるのではなく、モデルを交換可能な外部依存として扱えると安心です。

自分の bot では、`LLMAdapter`、`LLMEvent`、`Capability manifest`、契約テストという形で、そのための土台を作りました。

今回作った仕組みで、すべてのモデルを同じように扱えるわけではありません。それでも、モデル固有の差分を外側に寄せ、対応している機能を明示し、契約テストで確認できるようになったことで、かなり扱いやすくなりました。

AIエージェント時代のアーキテクチャでは、AIモデルをコアではなく外側の詳細として扱う。この考え方は、これからも変わらず重要なのではないかと思います。

モデル差し替え可能な設計にしておくと、新しいモデルが出るたびに「また全部作り直しか……」ではなく、「ではLLMアダプタを1個足すか」と言えます。精神衛生にもよいです。

## 補足： 実装側ではどうやっているか

実装上の対応を簡単に載せておきます。

LLMアダプタは、AIモデルや CLI を呼び出すための共通インターフェースです。記事上では `LLMAdapter` と呼びます。bot 本体は `LLMAdapter.invoke()` を呼び出し、共通の `LLMEvent` を受け取ります。

依存方向で見ると、以下のようになります。bot 本体は具体的な Claude Code や Codex CLI には依存せず、`LLMAdapter` という抽象だけを呼び出します。具体的なモデルや CLI への接続は、その外側にある Adapter 実装が担当します。

![](https://static.zenn.studio/user-upload/5773c798f00c-20260526.png)

各LLMアダプタは、モデル固有の出力を以下のようなイベント（`LLMEvent`）に変換します。

| イベント | 内容 |
| --- | --- |
| `TextDelta` | テキストの差分 |
| `ToolUseStart` | ツール呼び出しの開始 |
| `ToolUseInputDelta` | ツール引数の差分 |
| `ToolUseStop` | ツール呼び出しの完了 |
| `ThinkingStart` / `ThinkingDelta` / `ThinkingStop` | thinkingブロック |
| `ResultEvent` | 実行完了 |
| `ErrorEvent` | 致命的エラー |

Claude Code 固有の `claude -p --output-format stream-json` のパースは Claude Code 用のLLMアダプタに閉じ込めます。Codex CLI 固有の `codex exec --json` のパースは Codex CLI 用のLLMアダプタに閉じ込めます。

実際には、前後に hooks を差し込みたい、ユーザー確認が必要な操作をどう扱うか、... といったユースケースによって、ほかにもいろいろなイベントが考えられます。いずれの場合も、bot 本体から見た時の CLI の出力や状態は `TextDelta` や `ToolUseStart` などの共通イベントとして追加し、扱います。
