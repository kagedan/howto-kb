**カメラで手のひらを撮影するとAIが手相を読み解いてくれるアプリ**をFlutterで作ったので、その実装ログを書きます。

これまでタロット占いアプリやライフプランのシミュレーションゲームなど、「AI × 占い・シミュレーション」系のアプリをいくつか作ってきており、今回もその流れに則っています。

## 作ろうと思った理由

きっかけは「カメラ機能をちゃんと使ったアプリをポートフォリオに加えたい」でした。これまでの自作アプリはUI操作とAPI連携が中心で、端末のハードウェアを活かした実装の経験がなかったためです。

題材を手相にしたのは、以前作ったタロットアプリとの連続性です。タロットアプリでは「ケルト十字の10ポジションそれぞれに役割と文章量をプロンプトで明示すると、AIの出力が安定する」という知見を得ていました。手相も「生命線・知能線・感情線・運命線」という古典的な枠組みがあり、**タロットのポジション定義パターンがそのまま横展開できる**と考えました。

## 作ったもの

* 手のひらをカメラで撮影すると、4つの線(生命線・知能線・感情線・運命線)+総評をAIが読み解いてくれる
* 左手は「生まれ持った資質」、右手は「現在と未来」として、選んだ手で語りの視点が変わる
* 解釈はカード形式で項目別に表示

## 技術スタックと選定理由

| 技術 | 用途 | 選定理由 |
| --- | --- | --- |
| Flutter | アプリ本体 | これまでの自作アプリと共通 |
| camera | 撮影・プレビュー | ガイドオーバーレイを重ねるため、image\_pickerではなくプレビューを自前で持てるcameraを採用 |
| CustomPaint | 撮影ガイドの手型オーバーレイ | 後述 |
| Claude API(Vision) | 手相の解釈生成 | 画像を直接読ませて構造化テキストを返させる |
| flutter\_secure\_storage | APIキー保管 | 過去アプリと共通の方式 |

開発の進め方はいつも通り、**Claude.aiで仕様書を作り、それをClaude Codeに渡して実装する**方式です。今回は着手前に「撮影ガイドUI」「解釈生成(プロンプト設計)」「免責・プライバシー設計」の3本の仕様書を用意しました。仕様書に「要検討事項」という欄を作っておき、実装しながら判断した点はそこに追記していく運用にしています。

## 設計の柱: AIに「何を読むか」を決めさせない

このアプリの設計で一番こだわったのは責務分離です。過去に家計簿シミュレーションアプリで「決定論的な計算ロジックとAI解釈を分離する」設計をして手応えがあったので、今回も同じ思想で組みました。

* **アプリ側(PalmReadingDefinition)**: 読む項目・順序・観察観点・文字数の定義を持つ。AIに依存しない純粋なデータ定義
* **AI側(PalmVisionService)**: 渡された枠組みに沿って語るだけ。項目の追加・変更には関与しない

こうしておくと、「太陽線も読みたい」となったときに定義リストへ1項目足すだけで、プロンプトも結果画面も追従します。

プロンプトの構造はこんなイメージです(抜粋):

```
以下の項目を、この順序で読み解いてください。

1. 生命線(life_line): 親指の付け根を囲む線。長さ・カーブの大きさ・
   線の濃さを観察し、活力やバイタリティの傾向として200字前後で語る。
...

## 出力JSON形式
{
  "hand": "右手",
  "readings": {
    "life_line": "...",
    ...
  }
}
```

「各項目の役割と文章量を明示する」のはタロットアプリで効果があった手法で、今回も出力の長さのばらつきがかなり抑えられました。

## つまずいたポイントと解決法

### ガイドの手型シルエットが左右逆に表示される

**症状**: 実機で撮影画面を確認したところ、手の選択(右手/左手)に対して、点線の手型シルエットが左右逆に表示されていました。右手を選んだのに、枠に合わせようとすると親指の位置が合わない状態です。

**原因**: 左右の反転処理の「解釈の間違い」でした。仕様書には「左手選択時はシルエットを水平反転する」と書いていたのですが、そもそも**どちら向きのシルエットを基準(反転なし)とするか**の認識が、仕様と実装でずれていました。

考えてみると、これは間違えやすいポイントです。右手の手のひらをカメラに向けると、画面上では親指が**左側**に写ります。鏡で自分の手を見るときの感覚と逆になるため、「右手用のシルエット」を頭の中だけで考えると向きを取り違えやすいのです。仕様書に文章で「水平反転」と書いただけでは、この曖昧さは解消できていませんでした。

**解決**: 症状をClaude Codeに伝えたところ、反転処理の修正自体はすぐに完了しました。修正コストは小さかったものの、「左右」「向き」のような空間的な仕様は、文章だけでなく図や参考画像で示すべきだったというのが反省点です。

## 検証: 「寿命・健康に言及しない」は守られたか

生命線は「寿命」と結び付けられがちな線なので、仕様の受け入れ基準に「生命線の解釈に寿命・健康への言及が含まれないこと」を入れ、複数回生成してのチェックを実施しました。結果は**すべてクリア**で、システムプロンプトの禁止事項(「病気・寿命・健康状態・医療に関する言及を一切しない」「活力・バイタリティの傾向としてのみ語る」)が有効に機能していることを確認できました。

プロンプトで「〜しない」を守らせる指示は破られることも多い印象でしたが、「代わりにどう語るか」(活力の傾向として語る)をセットで指定したことが効いたのではないかと考えています。

## 免責とプライバシーの設計

占い系アプリなので、「これは娯楽である」ことの伝え方も設計対象にしました。

免責を**UI表示とプロンプト制御の二層**で担保しています。

* **UI側**: 初回起動ダイアログ → 撮影前注記 → 結果画面フッター → 詳細ページ、と段階的に配置。1枚の長い規約を読ませるのではなく、行動の直前に短い注意を置く方式です
* **プロンプト側**: システムプロンプトの禁止事項で「病気・寿命・健康状態への言及」を明示的に封じています。生命線は「寿命」と結び付けられがちな線なので、活力・バイタリティの傾向としてのみ語らせる設計にしました

また手のひら画像はセンシティブになり得るデータなので、ライフサイクルを「撮影 → 縮小 → API送信 → 解釈完了時に端末からも削除」と固定し、履歴には解釈テキストのみ保存する方針にしました。異常終了で画像が残るケースの保険として、アプリ起動時に一時ディレクトリを掃除する処理も入れています。

## 学び

* タロットで得た「ポジション定義」のプロンプト設計パターンは、題材がカードから手のひら画像に変わってもそのまま通用しました。**出力を安定させたいなら、AIに構造を考えさせず、構造はこちらで渡す**のが有効だと改めて感じます
* 「〜に言及しない」という禁止指示は、**「代わりにどう語るか」をセットで指定する**と守られやすくなる、というのが今回の検証からの手応えです
* カメラ機能は「撮影する」こと自体より、\*\*良い画像を撮らせるためのUI(ガイド・確認・撮り直し導線)\*\*の設計が本体でした
* 「左右」「向き」のような空間的な仕様は文章だけでは曖昧さが残ります。**図や参考画像を仕様書に含めるべき**というのが今回一番の反省点です
* 仕様書に「要検討事項」欄を設けて暫定判断を明記しておくと、Claude Codeへの指示がぶれず、実装しながらの判断も記録に残せて記事化が楽でした
