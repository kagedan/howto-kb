## 実践ガイド — 3エージェント連携のTipsと開発で得た知見

---

### はじめに

連載最終回は、3つのAIエージェントを連携させる際の実践的なTipsと、開発・運用を通じて得た知見を共有します。エージェント設計のパターン、ハマりポイントとその対策、大規模コードベースへの適用方法、そしてカスタマイズの方法を解説します。

### 3エージェント連携の実践手順

まず、3つのエージェントを実際にどう連携させるか、具体的な手順を示します。

#### Step 1: Refactoring Agent で現状分析

```
cd refactoring-agent-for-claude-code

# 対象コードベースの分析
/refactor-system-cmd ./path/to/legacy-system

# 出力確認
open reports/00_summary/full-report.html

# ナレッジグラフで関係性を探索
/query-graph "主要なドメインエンティティの依存関係"
```

ここで得られた MMI 評価とドメイン対応表が、後続の判断材料になります。

#### Step 2: Architecture Redesign Agent で再設計

```
cd architecture-redesign-agent

# Stage 1 の成果物をコピー
cp -r ../refactoring-agent/reports/ ./reports/

# 評価フェーズから開始（分析は Stage 1 で完了済み）
/workflow ./path/to/legacy-system
# → 「3. 設計から実行」を選択
```

Stage 1 の `reports/` をコピーしておけば、Architecture Redesign Agent は分析フェーズをスキップし、評価（Phase 2）から実行できます。これにより、**分析の重複を避けて時間を節約**できます。

#### Step 3: Coding Agent for ScalarDB で実装計画

```
cd coding-agent-for-scalardb

# Stage 2 の成果物を参照用にコピー
cp -r ../architecture-redesign-agent/reports/ ./reference/
cp -r ../architecture-redesign-agent/generated/ ./reference/

# ワークフロー開始
/plan
/plan-step 01
# → Architecture Redesign Agent の出力を参照しながら要件を具体化
```

### エージェント設計から学んだ5つの教訓

3つのエージェントの開発を通じて、AIエージェント設計に関する重要な教訓が得られました。

#### 教訓1: スキルは「単一責任」に絞る

最初は全フェーズの指示を1つの巨大な SKILL.md に詰め込もうとしました。結果は散々で、エージェントは後半のステップを忘れたり、前半の指示と混同したりしました。

```
❌ 1つのSKILL.md に全フェーズの指示を記述
  → 後半が忘れられる、指示が混同される

✅ フェーズごとに独立したSKILL.md
  → 各スキルが明確な責務を持つ
```

ソフトウェア設計の「単一責任原則」は、エージェントのプロンプト設計にもそのまま適用できます。Refactoring Agent の18スキル、Architecture Redesign Agent の37スキルという分割は、この原則に従った結果です。

#### 教訓2: 「即時出力ルール」でコンテキスト喪失を防ぐ

LLM のコンテキストウィンドウには上限があります。長時間のパイプライン実行中にコンテキストが上限に達すると、それまでの分析結果が失われます。

**対策: 各ステップ完了時に即座にファイルを出力する。**

```
❌ 全ステップ完了後にまとめて出力
  → 途中でコンテキスト上限 → 全部消失

✅ 各ステップ完了ごとに即座にファイル出力
  → 途中で中断しても途中成果物が残る
  → --resume-from で再開可能
```

Architecture Redesign Agent では、この原則をルールファイル（`output-conventions.md`）に明記し、すべてのスキルが強制的に従うようにしています。

#### 教訓3: サブエージェントでコンテキストを保護する

大規模コードベースのすべてのファイルをメインエージェントに読み込ませると、コンテキストが溢れます。

```
❌ メインエージェントが全ファイルを直接読む
  → コンテキスト溢れ

✅ サブエージェントに調査を委託し、要約だけ受け取る
  メイン → Task(Explore): "パッケージ構造を調査"
         ← 箇条書き5行の要約
```

Architecture Redesign Agent では8つのサブエージェントパターンを体系化しました。中でも **並列探索パターン** は特に有効で、3つのサブエージェントを同時起動して異なる観点を並行調査できます。

#### 教訓4: 前提条件チェック（Step 0）は必須

スキルを分割すると、「どのスキルがどのスキルの出力に依存しているか」の管理が課題になります。

各 SKILL.md の冒頭に **Step 0: 前提条件チェック** を設けることで解決しました。

```
### Step 0: 前提条件チェック

以下のファイルの存在を確認してください：

**必須:**
- reports/01_analysis/ubiquitous-language.md
- reports/01_analysis/domain-code-mapping.md
- reports/02_evaluation/mmi-overview.md

ファイルが存在しない場合、エラーメッセージを表示して処理を中断してください。
```

前提フェーズが未実行のまま後続スキルを実行しようとした場合、明確なエラーが返ります。

#### 教訓5: レビューシステムは品質の要

Coding Agent for ScalarDB の5観点並列レビューは、設計品質の大幅な向上に貢献しています。

人間のレビュアーは得意な観点に偏りがちですが、5つの異なる観点を強制的に並列実行することで、**漏れのない品質チェック**が実現します。

```
1つの観点でのレビュー → 見落としが発生しやすい
5観点並列レビュー → 整合性 × ScalarDB適合性 × 運用性 × セキュリティ × 実装可能性
```

### Mermaid図と日本語の相性問題

3つのエージェントはすべてMermaid図を生成します。日本語テキストとMermaidの相性には多くの落とし穴がありました。

#### よくあるエラーと対策

**1. スペースを含む日本語テキスト**

```
%% NG: パースエラーになる
A[監査セット 作成] --> B[ユーザー認証]

%% OK: ダブルクオートで囲む
A["監査セット作成"] --> B["ユーザー認証"]
```

**2. サブグラフ名**

```
%% NG
subgraph Phase 1 分析

%% OK
subgraph "Phase 1 分析"
```

**3. エッジラベルのハイフン**

```
%% NG
A -->|Customer-Supplier| B

%% OK
A -->|"Customer-Supplier"| B
```

**4. 数字始まりのノードID**

```
%% NG
1[First] --> 2[Second]

%% OK
node1["First"] --> node2["Second"]
```

対策として、これらのルールを `.claude/rules/mermaid-best-practices.md` に集約し、すべてのスキルが参照するようにしました。さらに `/fix-mermaid` スキルで生成後の自動検証・修正を行います。

### 大規模コードベースへの適用Tips

サンプルコード（356行）での動作確認後、実際のプロジェクト（数万行〜）に適用する際のTipsです。

#### Tip 1: 段階的に実行する

全フェーズ一括実行ではなく、段階ごとに結果を確認しながら進めます。

```
# Step 1: まず調査と分析だけ
/system-investigation ./src
/analyze-system ./src
# → 結果を確認。ドメイン用語は正しく抽出されているか？

# Step 2: 評価
/evaluate-mmi ./src
/ddd-evaluation ./src
# → MMIスコアは妥当か？DDD適合度の判定は正しいか？

# Step 3: 設計（評価結果に納得してから）
/ddd-redesign ./src
/design-microservices ./src
```

#### Tip 2: スコープを限定する

大規模コードベースを一度に全部分析するのではなく、ドメインやディレクトリを絞って段階的に進めます。

```
# 特定ドメインだけを分析
/full-pipeline ./src --domain=Order,Customer

# 特定のサブディレクトリだけを調査
/system-investigation ./src/main/java/com/example/order
```

#### Tip 3: ナレッジグラフを活用する

一度 `/build-graph` でグラフを構築すれば、コードを読み返すことなく関係性を探索できます。

```
/query-graph "Orderに依存するすべてのエンティティ"
/query-graph "凝集度が低いモジュール内のクラス一覧"
/query-graph "境界コンテキスト間で共有されているエンティティ"
```

#### Tip 4: 不要なフェーズはスキップする

```
# ScalarDB を使わない場合
/full-pipeline ./src --skip-analytics

# セキュリティ分析が不要な初期検討の場合
/full-pipeline ./src --skip-security

# データモデル分析が不要な場合
/full-pipeline ./src --skip-data-model
```

#### Tip 5: 人間の判断を挟む

エージェントの出力を鵜呑みにせず、各ステージの出力を人間がレビューしてから次に進むことが重要です。

```
Stage 1 完了 → MMI評価は妥当か？ドメイン用語は正しいか？
  ↓ 人間のレビュー
Stage 2 完了 → 境界コンテキストの分割は適切か？
  ↓ 人間のレビュー
Stage 3 完了 → 実装計画は実行可能か？
  ↓ 人間のレビュー + GO/NO-GO判定
実装着手
```

特に境界コンテキストの分割判断は、ビジネス上の判断が必要な部分であり、AIの提案をそのまま採用するのではなく、ドメインエキスパートとの議論を経て決定すべきです。

### カスタムスキルの追加方法

3つのエージェントはすべて Claude Code のスキルシステム上に構築されているため、**Markdownファイルを1つ作成するだけ**で新しいスキルを追加できます。

#### 最小限のカスタムスキル

```
---
name: my-custom-analysis
description: カスタム分析スキル
user_invocable: true
---

# My Custom Analysis Agent

## 概要
独自の観点でコードベースを分析します。

## 使用方法
/my-custom-analysis [対象パス]

## 実行プロンプト

### Step 0: 前提条件チェック
引数で指定されたパスが存在することを確認してください。

### Step 1: コード調査
Task(Explore) サブエージェントで対象ディレクトリのコードを調査し、
特定のパターンを検出してください。

### Step 2: 分析結果出力
reports/custom/my-custom-analysis.md に出力してください。

## 出力ファイル
- reports/custom/my-custom-analysis.md
```

これを `.claude/skills/my-custom-analysis/SKILL.md` に保存すれば、`/my-custom-analysis ./src` で呼び出せます。

#### カスタムスキルのベストプラクティス

1. **Step 0 で前提条件を検証する** — 必須入力ファイルの存在確認
2. **サブエージェントでコンテキストを保護する** — Task(Explore) で調査を委託
3. **即時出力ルールに従う** — 各ステップ完了時にファイルを書き出す
4. **フロントマターを付与する** — title, phase, skill のメタデータ
5. **Mermaidルールに従う** — 日本語テキストはダブルクオートで囲む

#### ルールファイルの追加

スキル横断の共通知識を追加したい場合は、`.claude/rules/` にMarkdownファイルを追加します。

```
<!-- .claude/rules/my-convention.md -->

## カスタムコンベンション

- 出力ファイルのエンコーディングは UTF-8
- 日付形式は ISO 8601
- 金額の単位は明示する（USD, JPY 等）
```

CLAUDE.md で参照するだけで、すべてのスキルから自動的に利用可能になります。

### 3つのエージェントの使い分けガイド

最後に、プロジェクトの状況に応じたエージェントの使い分けをまとめます。

#### パターン1: フルコース（全3エージェント）

**シナリオ:** レガシーモノリスをScalarDBベースのマイクロサービスに全面刷新

```
Refactoring Agent → Architecture Redesign Agent → Coding Agent for ScalarDB
```

最も包括的なアプローチ。分析の重複を最小限に抑えつつ、各エージェントの強みを活かします。

#### パターン2: クイックスタート（Architecture Redesign Agent のみ）

**シナリオ:** とりあえず現状把握からコード骨格生成まで一気に

```
Architecture Redesign Agent（/full-pipeline）
```

Architecture Redesign Agent は Refactoring Agent の全機能を含んでいるため、単体でも分析からコード生成まで完結します。

#### パターン3: 現状把握のみ（Refactoring Agent のみ）

**シナリオ:** リファクタリングの方向性を探る初期検討段階

```
Refactoring Agent（/refactor-system-cmd）
```

コード生成は不要で、現状の可視化と改善方針の策定だけが必要な場合。

#### パターン4: 実装計画のみ（Coding Agent for ScalarDB のみ）

**シナリオ:** アーキテクチャは人間が設計済み。ScalarDBの詳細設計・実装計画が必要

```
Coding Agent for ScalarDB（/plan）
```

16の調査ドキュメントがScalarDBの知識ベースとして機能するため、ScalarDB初心者でも高品質な実装計画を策定できます。

### 今後の展望

#### 多言語対応

現在は主にJavaプロジェクトを対象としていますが、分析ロジックの多くは言語非依存です。Serena MCP が対応する言語（Python、TypeScript等）への拡張は比較的容易です。

#### CI/CD統合

現在はスポット実行ですが、CI/CDパイプラインに組み込んで「コミットごとにMMIスコアを計測する」「PRマージ時にDDD適合度をチェックする」といった継続的な品質モニタリングも実現可能です。

#### エージェント間の自動連携

現在はエージェント間の成果物の受け渡しを手動で行っていますが、将来的にはエージェント間の自動連携（一方の出力を自動的に次のエージェントに渡す）も検討しています。
