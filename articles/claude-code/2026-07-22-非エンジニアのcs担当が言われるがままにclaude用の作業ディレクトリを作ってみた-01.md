Claudeの作業用ディレクトリを作成した時の話を書きます。

## なんでそんなに細かく分けるの？てか作業用ディレクトリってなに？

ある日、先輩社員とオンラインで画面共有をしながら、自社プロダクトの色々な操作をClaudeのスキルで行う方法について教わっていた時、先輩から「Claude用の作業ディレクトリを設定した方がいいよ」とアドバイスをもらい、その時に先輩のディレクトリ構造を見せてもらいました。

担当しているプロダクトやお客様ごと、その中でも議事録やコンテキストなど役割ごとにかなり細かくディレクトリが分かれていて、案件ごとに背景を記載したドキュメントや議事録、資料の置き場などが設定されていました。

細かくてすごいなーとは思ったのですが！！！

非エンジニアの私としては、ディレクトリはPC内に貯めているファイルを貯めておく整理場所というイメージしかなく、そこにClaudeの作業場所を作ってあげることの意味や有効性が全くわかりませんでした。

私は最近ターミナルの操作を学んでいるところでして、ターミナルやClaudeがcdコマンドでなどで作業ディレクトリを移動する習性があるということ自体は知っていたのと、一応CSとしてお客様とミーティングを行った際の議事録が貯まるディレクトリを作って、次回定例の壁打ちなどは実践していたのですが、それ以上にいくつもいくつもこんなに細かくディレクトリを作らなくてはClaudeを活用しきれないのかと疑問に思ったので、  
ディレクトリ作成依頼をClaude Codeに投げつつなぜこのようなことをしなければならないのかについてそのまま聞いてみました。

## AIは実は前回のことをそこまで覚えていないらしい

いや同じセッションで聞けばそこで与えた文脈とかやりとりはばっちり全部覚えてるんじゃないの？そこまでやる必要ある？と思って聞いてみた回答をそのまんま貼ります。  
結論ダメみたいです。

1. コンテキストウィンドウ（一度に覚えられる量）に上限がある  
   セッションを続けても、会話が長くなると古いやり取りから押し出されて実質的に忘れます。無限メモリではなく、机の広さが決まっているイメージ。長い案件は1セッションに収まりません。
2. 長いほど劣化する  
   情報が溜まるほど、関係ない過去のやり取りに引きずられて精度が落ちたり(いわゆる文脈の希釈)、応答が遅く・高コストになります。「全部盛り」は品質的にも不利です。
3. セッションはいつか切れる／落ちる  
   アプリを閉じる、別PCで作業する、時間が空く、クラッシュする。そのたびに記憶ゼロに戻ります。人間なら「昨日の続き」ができますが、セッション頼みだとそれが飛ぶ。

## まずは小さく作ってみた

いやそうだとしてもそこまで細かく作成しないといけないの？？？と思ったので、まずはコンパクトに必要そうなものだけで始めてみることにしました。

```
CLAUDE.md            # 私の自己紹介と、AIへの頼み方の前提
.claude/rules/
  communication.md   # メール・チャットのトーン
  workflow.md        # 作業の進め方
areas/
  _template/         # 新しい案件はこれをコピーして使う雛形
    meetings/        # 議事録
    references/      # 先方資料
    drafts/          # 下書き
```

ちなみに、細かくディレクトリを分けたところで、Claudeはどこが何のためのディレクトリやファイルかわかってくれるのか？と思いましたが、説明ファイルを作成すればそれをちゃんと読んでくれるとのことでした。

`CLAUDE.md` は、AIが作業の前に必ず読むファイルです。名前や所属、「結論から短めに書いて」みたいなお願いを書いておけばよいそうです。

READMEも作ってくれて、そこには「ここは議事録を入れる場所」みたいに書いておけば、AIが名前から想像するのではなく用途を読んで分かってくれるようです。

## まとめ

ディレクトリは、ファイルの保管庫というイメージしかなかったのですが、Claudeくんの作業場所として整備してあげることでよいアウトプットにつながりそうです。

使ってみてどうだったか、使いやすいかなどまた書きたいと思います。
