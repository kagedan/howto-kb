## 「これ、Taskでいいんだっけ」に毎回2秒固まる問題

Claude Code Agentを本格的に使い始めると、コードを書く前にここで止まる。`Task` toolで投げるか、`TeamCreate`でチームを組むか、それともメインスレッドのまま書き切るか。

最初は「並列で速くなるから Task でいいか」くらいの軽さで決めていたのだが、これがじわじわ効いてくる。1秒で終わる grep に subagent を起動して10秒待たされたり、互いに干渉し合う調査を素朴な Task 並列で組んで身動きが取れなくなったり、という地味な失敗が積み上がっていく。

判断軸を整理して、3問のフローに圧縮したら迷いが消えた。その話をする。

## まず前提: Task の本来の目的は「並列化」ではない

`Task` tool の第一目的は **context isolation** だ。並列実行はその副産物に過ぎない。

メインスレッドで Read / Grep / WebFetch を連発すると、その全ログがメインのコンテキストウィンドウを食う。一方 `Task` で subagent を呼ぶと、subagent 側のコンテキストで探索が走り、メインに戻るのは**最終出力だけ**。途中の grep ヒット数千行はメインの目に触れない。

これが Task の本質だ。「速くなる」より「コンテキストが長持ちする」が主効果。この前提がずれていると、後の判断がすべてずれる。

## 3つのモードの違い

| モード | 通信 | 向いている問題 |
| --- | --- | --- |
| メインスレッド | なし | 短い検索・単一ファイル編集・既知の修正 |
| Task subagent | 親→子→親（一方通行） | 探索 heavy・分割可能な並列タスク |
| Agent Team | メンバー間で双方向 DM | 相互依存のある探索（バグ調査・監査） |

Task と Agent Team の本質的な違いは**通信方向**だ。Task は依頼して結果を受け取るまで一直線。Agent Team は走っている最中にメンバー同士でメッセージを送り合えるので、A の発見を受けて B が調査方向を変える、という動的な連携ができる。

## 判断フロー: 3問で決まる

### Q1: 分割できるか

3ファイル以下・3手順以下の作業なら、subagent の起動コストのほうが高くつく。メインで素直に書く。

単一ファイルの設定更新や、差分を読んでコミットメッセージを整えるような作業がこれに当たる。ここに subagent を挟むのは、コンビニのおにぎりを買うのにタクシーを呼ぶ感じだ。

### Q2: 探索ログをメインに残したいか

ユーザーと対話しながら探索したい、途中の grep 結果を見て次の判断をしたい——こういう場合は subagent に隔離すると経過が見えなくなる。意思決定が挟まる調査はメインで走らせる。

逆に「結果だけ欲しい」「途中のヒット数十件はメインに残したくない」なら、subagent に隔離する。これが Task の本来の用途だ。

### Q3: 途中で他の調査ラインの結果を参照するか

**独立して走り切れる** → Task を並列で複数呼ぶ。たとえば3つの異なるディレクトリで型エラーを集計するような作業。

**A の発見が B の方向を変える** → Agent Team。ログで遅延を発見したら、コード担当が同時刻のデプロイ差分を見に行く、というような動的な連携が必要な場合。

「並列調査だから Agent Team」ではなく、「**途中で互いに影響し合うかどうか**」で決める。

## なぜこう書くか: モード選択のトレードオフ

| モード | 起動オーバーヘッド | メインのコンテキスト消費 |
| --- | --- | --- |
| メイン | なし | tool call ごとに直接消費 |
| Task subagent | 数秒〜十数秒 | 最終出力分のみ |
| Agent Team | チーム作成 + spawn でさらに乗る | メンバー数 × 起動コスト分はメインに乗らない |

Task も Agent Team も「速くするため」に使うと判断がずれる。正しい動機は：

* **Task**: メインのコンテキストを汚さずに、別プロセスに探索・生成を任せたい
* **Agent Team**: 走行中に動的な方向転換（相互通信）が要る
* **メイン**: 小さく・速く・結果をユーザーに見せながら進めたい

この動機がフィットするモードを選べば、3問のフローとだいたい一致する。

## 実際にやらかした3つのパターン

**パターン1: 短いタスクに subagent を使う**

「念のため Task でやっておくか」の積み重ねで、起動オーバーヘッドが積算されてレイテンシが地味に伸びる。速くしたつもりが遅くなる、最も悲しいパターン。Q1 で「分割不要」が出たらメインで書く、を徹底する。

**パターン2: 干渉する並列を Task で組む**

「3つの仮説を Task 並列で投げれば速い」と思って組んだら、A 線が答えを出した後も B / C が止まらない。動的な方向転換が必要だったので、本来は Agent Team が正解だった。Q3 の問いはこの失敗の予防策だ。

**パターン3: Agent Team を「とりあえず並列」で使う**

逆方向の失敗。並列実行が目的なら Task で十分なのに、`TeamCreate` を使って team\_name 管理 / SendMessage の hub-and-spoke 設計まで書いて、複雑度だけが増す。動的協調が要らないなら Team は overkill だ。

## 自分はこう使っている

判断を3問のフローに落とし込んでから、「2秒固まる」が消えた。

* コミットメッセージ生成・画像リサイズ・単一ファイル更新 → **メイン一択**
* ブログ公開の直列4ゲート・独立した並列検索 → **Task**
* バグ調査の動的な3線並列・インシデント対応 → **Agent Team**

新しいタスクを書く前に「分割できるか？」「ログを残したいか？」「途中で参照するか？」の3問を通すだけで、モード選択の議論が消えた。選択が消えると、実装に集中できる。
