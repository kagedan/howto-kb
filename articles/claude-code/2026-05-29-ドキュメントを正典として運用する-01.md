# ドキュメントを「正典」として運用する

ここで紹介するのは、Claude Code（Anthropic の CLI 型コーディングエージェント）を実務開発に使う中で固まってきた、ドキュメントの運用方法です。エージェントに正しく動いてもらうための「足場」をどう設計するか、という話になります。また2026年時点での話であり、これからもかわっていくことでしょう。

AI コーディングエージェントにとって、ドキュメントは補助資料ではなく**判断の根拠にする一次情報**です。曖昧・古い・所在不明な記述は、そのまま誤った実装になって返ってきます。

所在は明確に分けます。指示書（常に効くルールファイル）には**規範とルーティングだけ**を書き、**設計の正式仕様は `docs/` にカテゴリ別**で置きます。

* `docs/architecture/` — 認証フロー・構造・ログシステムなどの**正式仕様**
* `docs/servers/` `operations/` `development/` `troubleshooting/` — 構成・手順・開発・障害記録

以下、`docs/` を正典として機能させる設計を、**なぜそうするか**とあわせてまとめます。

## ルールとドキュメントを混ぜない

規範（「必ず△△する」）は指示書へ、正式仕様（「こう動く」）は `docs/` へ。それぞれ一か所だけに書きます。

\*\*理由は、\*\*混ざると、矛盾した記述に挟まれてエージェントが誤動作します。二か所に書けば必ず片方が腐り、古いほうを根拠に動きます。

## ルールを領域別に分散し、自動で読み込ませる

ルールを一枚の指示書に全部書くのではなく、領域ごとに分けて配置します。横断的なルールだけを最上位に置き、**領域固有のルールは各サブディレクトリの指示書に分散**し、その領域のファイルを編集するときだけ自動で読み込まれるようにします。最上位には「どの領域を触るとどの指示書が読み込まれるか」の**ルーティング索引**だけを置きます。

* 配備スクリプト → `scripts/` の指示書（デプロイ・ビルド・archive 規則）
* フロント → `web/` の指示書（URL とコードの対応、API 名前空間）
* バックエンド → `admin/` の指示書 / ログ署名 → `logging/` の指示書

\*\*理由は、\*\*全ルールを常時読ませると指示書が膨張し、肝心の規範が薄まって見落とされます。領域別に分け、編集対象に応じて必要な規範だけを文脈に載せれば、常時ルールは薄く保ったまま、その場で効くルールだけを確実に効かせられます。新しい領域のルールは該当ディレクトリに足し、索引に一行加え、最上位からは消す——**一か所だけに書く**原則をここでも守ります。

## ルールは「短く明瞭」を最優先にする

ルール本体は短く言い切り、背景や例外は本体に書きません。

\*\*理由は、\*\*長いルールは見落とされます。長文に埋め込まれた一文は「読んだが効かない」状態になる。短いほど確実に守られます。

## 用語の正典を一つ持ち、言葉を厳密にする

`user`・`key`・`session` などの多義語は用語集（glossary）を一つだけ正典とし、表記レイヤーを分けます。

* **DB 層** snake\_case / **概念層** PascalCase / **散文層** 日本語

**理由は、**用語がぶれると出力もぶれます。言葉を一つに固定して、はじめて正確な出力を期待できます。なお**既存記述の機械的一括置換は禁止**します。用語集の効果は「これから書くもの」の統一にあり、過去の一括書き換えは相互参照を壊すリスクが高く効果も小さいためです。

## 巨大なドキュメントは「全文を読ませない」前提で構造化する

一度読んだファイルは以降のやり取りに残り続け、累積コストになります。そこで部分読み前提に構造化します。

* 冒頭に\*\*目次（TOC）\*\*を置き、必要な節だけ部分読みする
* 行数が増えたらインデックスと実体を**分割**する
* 指示書に\*\*「全文 Read 禁止リスト」\*\*を明記する

**理由は、**狙いは二つ。**指示書の膨張を防ぐ**こと——本体にはルーティングだけを残し、読ませる範囲を明瞭にして読み込みを必要時まで遅延させます。もう一つは総コスト削減。横断調査や広い探索は**探索サブエージェントに委譲**し、要約だけ受け取って本体に長大なファイルを載せません。

## コードと外部リソースを mapping する

URL・エンドポイント・画面といった**外部から見えるリソース**と、**コードの起点・ログの場所**の対応表を領域ごとに用意します。

**理由は、**これがないと、エージェントは**全体を grep / find で舐め回すところから調査を始めます**。遅く、文脈を無駄に消費する。対応づけておけば「この URL → このファイル → このログ」と最短で到達でき、特定できないときだけ grep に切り替えられます。

## キーワードから正典へ引ける索引を持つ

**キーワード（環境変数・ファイル・API・概念）→ 該当ドキュメントの該当行**への細粒度索引を別に持ち、まずそこを引きます。最上位には**索引・用語集・目次の三つ**を置き、これだけ読めば全体を俯瞰できる状態にします。

\*\*理由は、\*\*本文を整えても入口が分からなければ読まれません。索引一つで「どこを読むか」が確定すれば、本体を膨らませず到達性だけ上げられます。新しく参加するエージェントも人も、まず索引を読めば「どこに何があるか」を把握でき、いきなり本文を全部読む必要がなくなります。

## ドキュメントを定期的に蒸留する

ドキュメントは書いて終わりではなく、古くなった記述を見分けて整理し続けます。

* 廃止された記述は消すか、**「SUPERSEDED（どれに置き換わったか）」を明記**して残す
* **自動チェック**（broken link・廃止用語の混入・肥大化や TOC ずれ）を回し、コミット前に逸脱を警告する
* 個人ノートに溜めた知見が一般化・安定化したら、`docs/` に**昇格**させる

**理由は、**ドキュメントは放置すると必ず現実と乖離し、エージェントは古い記述を正典として読んでしまいます。蒸留を「特別な作業日」にすると後回しになって溜まる。だから**通常作業の副産物として常時行い、自動チェックで取りこぼしを拾う**ことで、正典が正典であり続ける状態を保ちます。
