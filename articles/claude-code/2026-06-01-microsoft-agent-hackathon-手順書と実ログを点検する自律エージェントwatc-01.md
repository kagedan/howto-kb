## 暗黙知の形式知化を支援する自律駆動型のAIエージェントを作りたい

こんにちは、先端技術開発グループ（WAND）の加藤です。

私は先端技術開発グループに所属しながら、シニアエンジニアの立場で様々なプロジェクトに参画することが多く、  
特に案件相談やAI開発プロジェクト初期の要件定義に携わることがよくあります。

最近では、AIエージェント構築のプロジェクトが増え、金融・エネルギーなど長くお付き合いがある領域に加えて、  
製造業の皆様とお話する機会も非常に多くなってきました。

そんな中で、特に製造業のお客様からお問い合わせいただくのが、いわゆる「暗黙知の形式知化」です。  
日本の多くの製造業では、長く業務に携わられているベテラン社員の方も多く、こうした方の高齢化にともなって、知識の伝達が危ぶまれており、  
AIエージェントを用いて、暗黙知の形式知化に取り組みたいというお客様が多くいらっしゃいます。

この暗黙知の形式知化という課題には様々なアプローチがありますが、今回は自律駆動型のAIエージェントを用いて、  
それを実現するアプローチをMicrosoft Agent Framework 1.0を用いて検証してみました。

## 製造業における作業手順書と実ログの差

製造業を一例に挙げてみます。

多くの製造業の皆様において、製品製造やサービスを定められた条件の元で管理された状態を維持することは非常に重要です。  
ISO 9001などの国際規格を取得している工場では、作業手順書 (Standard Operating Procedure, 以下 SOP) が業務の規範として書面化されています。  
一方、こうした書類は、改訂が一定の頻度で発生します。新しい検査項目が増えた、リコール後の確認手順が追加された、有害物質規制が更新された、といった変更があると、  
すぐに反映することが求められます。

一方で、こうした変化をすぐに業務で実施したり、安定的に運用していくことは非常に手間がかかるものとなっています。

これは製造業に限った話ではなく、医療、金融、物流など、**SOP と実行ログの両方が存在する業務すべて**で起きています。BPR (Business Process Reengineering) は一度実施すれば終わる仕事ではなく、規範と実態の乖離を **誰かが継続的に監視し続ける** 仕組みが求められています。

## これからのエージェントシステムをどう設計するか

具体的な実装の話に入る前に、設計思想を書いておきます。

### パーソナルエージェントが「中心」になる時代

ここ数か月で、**[Microsoft 365 Copilot Cowork](https://www.microsoft.com/en-us/microsoft-365/blog/2026/05/05/copilot-cowork-from-conversation-to-action-across-skills-integrations-and-devices/)** やClaude Coworkなど **個人につくパーソナルエージェント** が一気に出揃いました。一人ひとりに提供されたパーソナルAIエージェントが業務UIとして常駐し、そこに必要なツールを繋いでいくものと考える方も多いと感じます。

この前提に立つと、これまでよくあった「統括エージェントと作業エージェントがA2Aで会話する」 ようなマルチエージェント設計は、コンテキスト汚染のリスクを低減したり、並列実行が可能といったメリットはあるものの、デメリットもあると考えています。  
一つの示唆として、[*Single-Agent LLMs Outperform Multi-Agent Systems on Multi-Hop Reasoning Under Equal Thinking Token Budgets*](https://arxiv.org/abs/2604.02460) では、**思考トークンの予算を揃えた条件下で 単一エージェント が マルチエージェント を マルチホップ推論で上回る** ことを実験的に示しています。実運用側でも、Anthropic が自社の multi-agent research システムで **chat 比で約 15× のトークン消費** を[報告](https://www.anthropic.com/engineering/built-multi-agent-research-system)したうえで、「価値が十分高いタスクでないと経済的に成立しない」 と明言しています。

**自律的に動くべき塊は、独立した 1 つのエージェントとして閉じておく方が結局合理的** であることがわかってきています。

### 一方、パーソナルAIエージェントだけでは継続監視は困難

一方で、Coworkのようなパーソナルエージェントは「ユーザーが何か聞きに来る」ことが前提の指示待ち構造です。SOP と実態の乖離を 24 時間ずっと見続ける、といった業務には向きません。

そこで自分が取ったのは、次のような 2 層の構造です。

* パーソナルエージェント：ユーザーが Teams や Outlook で「先週の異変は？」と聞けるプル型のエージェント
* 独立自律エージェント：裏で動いて気づいたら通知を投げるプッシュ型のエージェント

ポイントは、パーソナルエージェントと独立した自律エージェントを分けたうえで、両者を同じツールレジストリの傘下に置きつつ、**エージェントごとに持たせるツールの権限を変える**ことだと考えています。たとえばパーソナルエージェントには read 中心、自律エージェントには detect・record と限定的な write、という具合に分けておけば、「どのエージェントが何をできるか」を一元的に管理でき、同じ操作を意図せず両方ができてしまう状態も防げます。

今回の PoC では利用しているのが Watchdog だけなので権限分離までは実装していませんが、本番では [Entra Agent ID](https://learn.microsoft.com/en-us/entra/agent-id/what-is-microsoft-entra-agent-id) や [Agent Governance Toolkit](https://opensource.microsoft.com/blog/2026/04/02/introducing-the-agent-governance-toolkit-open-source-runtime-security-for-ai-agents/) を使い、エージェントごとに許可するツールの権限をレジストリ側で管理する想定です。パーソナルと自律を分離し、ツールは権限で出し分ける。スケールしても管理しやすい構成を、ここで目指しています。

次章から、具体的な中身を見ていきます。

## 手順書と実ログを巡回点検する自律型エージェント「Watchdog Agent」

今回作った Watchdog Agent を一言でいうと、完全自律的に気づきを返してくれる業務プロセス監視 AI です。

従来型のLLM チャット系の PoC の多くは、ユーザーが質問しに来ないと何も起きない受動的なシステムです。一方、手順書と実態の乖離を継続的に見つける、というような用途では、ユーザーから調べてと言われる前に動いてくれることが重要です。

そこで Watchdog Agent は、裏側で自律的に動き、気づいたことを返してくる構造にしています。

1. SOP や実行ログに変化があると、変化点から調査タスクを組み立てる
2. 必要なツールを 24 種類のなかから自分で選び、連鎖的に実行
3. 検出した異変を、業務担当者の方が読める言葉でレポート化
4. 過去の判断 (棚上げした所見 / 採用した方針) を踏まえ、次の調査を自分で予約

整理すると、**事象駆動・自律的なツール選択・メタ認知・自己拡張** の 4 点で、自律的な振る舞いを実行するエージェントになっています。

## Watchdog が見ている 3 つの軸

Watchdog が見ているのは、次の 3 つの軸です。

*手順書と現場の実態、両方の変化を突き合わせて初めて『守るべき本当の手順』が見えてくる*

| 軸 | 何を見るか | 製造業での例 |
| --- | --- | --- |
| **規範の変化** | SOP の時系列差分 | 検査項目の追加、規制更新時の手順変更 |
| **観察の変化** | 実行ログの時系列差分 | 工程実績の頻度急変、新規バリアント出現 |
| **規範 ↔ 観察 の乖離** | 規範と実態の乖離 | 新検査項目が現場で実施されていない |

この 3 軸が時間方向で重なり合う核となるプロセスは、 **関係者が合意していて、実際に守られていて、規制にも反していないやり方** であり、これは **ゴールデンプロセス** と呼ばれるものです。これはまさに、ベテランの暗黙知を含む「本来あるべき業務の形」を、規範と実態の両面から形式知として固定していく作業に当たると思っています。Watchdog はその手前の段階で、危険信号を継続的に拾い続けるレイヤを担っています。

## 1 体のエージェントで 4 つの役割を回す

「Agentic AI らしい振る舞い」をどう実装するか、というのは今回のハッカソンでのテーマの一つでした。Watchdog では、1 つのエージェントが内部で 4 つのフェーズを順に回す構造にしています。

| フェーズ | 役割 | 主な MCP tool |
| --- | --- | --- |
| **Plan** | 今回の巡回の調査計画を立てる | `list_workflows`, `get_watchdog_stats`, `get_backlog_status` |
| **Investigate** | 自律的に検出器を選んで深堀り | `detect_norm_drift`, `detect_obs_drift`, `detect_norm_obs_gap`, `scan_ambiguity` |
| **Judge** | 報告に値する所見を選別 | `list_findings`, `cluster_findings`, `propose_root_causes`, `reassess_severity` |
| **Report** | レポート保存 + 次タスク投入 | `record_watchdog_report`, `enqueue_watchdog_task` |

複数のエージェントを立てて「統括エージェントと作業エージェントが会話する」構成は、先述の理由から今回は採りませんでした。代わりに、1 体のエージェントに **業務担当者の方が自然にやっていること** を 3 つ持たせる形にしています。

### ① 過去の判断を覚えている (メタ認知)

人間の業務担当者の方であれば、「先週この異変は誤検出と判断したので、今週は別の角度で見よう」と自然に考えられると思います。Watchdog も同じように振る舞います。

具体的には、エージェントが新しい調査を始めるとき、次の情報が自動でシステムプロンプトに挿入されます。

* 直近 3 件のレポート (どんな結論を出したか)
* 棚上げした調査結果の傾向 (drift\_type 別 + 理由)
* 現在のバックログ状況 (どんな調査が予約されているか)

これにより、同じ誤検出を繰り返しにくくなります。一度「これは無視してよい」と判断したパターンは、次回以降は最初から優先度を下げて扱われます。業務担当者の方の記憶を、実装として再現したものです。

### ② 自分で次の調査を予約する (自己拡張)

エージェントが 1 つの所見を見たときに、「これだけでは根本原因が分からない、別のワークフローも見ておきたい」と気づくことがあります。そうした場合、エージェント自身が `enqueue_watchdog_task` を呼んで **次の調査をバックログに追加** します。

1 回の実行で結論まで辿り着けなくても、エージェントが自分でフォローアップ調査を予約してくれるため、巡回をまたいだ仮説検証が進みます。人間が「次に何を調べるべきか」を逐一指示しなくて済むのは、地味ですが効いてくる利点だと感じています。

### ③ 業務判断の中身を LLM に担わせる

LLM の使い方は、ともすると「個々の所見を日本語に翻訳するだけ」になりがちです。Watchdog では一歩踏み込んで、3 つの **業務判断系のツール** を独立した MCP tool として実装しました。

| Tool | 何をする | 業務担当者で言うと |
| --- | --- | --- |
| `cluster_findings` | 蓄積した異変を意味的にクラスタリング | 「最近溜まってるこの問題群、テーマ分けすると...」 |
| `propose_root_causes` | 各クラスタの根本原因仮説を生成 | 「これって全部、新人教育の手順が古いのが原因では？」 |
| `reassess_severity` | 全所見の重大度を見直す | 「この一連の重要度、いまの状況だと再評価が必要かも」 |

これによりエージェントは、個別の所見の翻訳係にとどまらず、**横断的な因果分析と優先順位の見直し** まで担えるようになります。LLM の "少し考えさせる" 力を、業務に効くポイントに絞って使う狙いです。

このように 3 つの役割を 1 体のエージェントに持たせることで、マルチエージェント化のコスト（エージェント間のやり取りで LLM 呼び出しが大きく増える）を払わずに、エージェンティックな振る舞いを実装できたと思っています。

## MCP を中心にした疎結合な構成

先に述べた「中心 = パーソナルエージェント、周辺 = 独立自律エージェント、両者をツールレジストリの傘下に置く」という構造を、実装としてどう実現したかをここで具体的に書いていきます。

自律エージェントである Watchdog は stdio 経由で MCP server のツールを呼び、外部のホスト（Claude Desktop / Copilot Studio / Microsoft 365 Agents SDK / Copilot Cowork）は Streamable HTTP 経由で同じツール群を共有します。Watchdog も外部ホストも、必ずこの MCP server を経由してしかツールに触れません。

今回は利用しているのが Watchdog だけなので全 24 ツールを呼べる状態ですが、本番設計の狙いはレジストリ側でエージェントごとに権限を絞ることにあります。たとえば、自律 Watchdog は detect・record 系まで含めた広めの権限、Copilot Cowork は read 中心、Copilot Studio で組む業務ワークフローエージェントは write を含むが承認フロー経由のみ、という分け方を、同じ MCP サーバー上でエージェントとツールの対応として表現するイメージです。

この構造のうえで、以下のシナリオが直線でつながります:

| 連携パターン | 方向 | 例 |
| --- | --- | --- |
| **プッシュ（検知 → アクション）** | Watchdog → Cowork | 重大度の高い所見を Watchdog が検出 → Cowork が Teams で関係者に確認依頼を投げる |
| **プル（利用者 → 検知）** | Cowork → MCP → Watchdog | ユーザーが Teams で「先週の異変は？」 → Cowork が `list_findings` を呼んで返す |
| **A2A 協調** | 3 体リレー | Watchdog (検知) ⇔ Copilot Studio エージェント (ワークフロー) ⇔ Cowork (SOP 改訂 PR) |

本実装の MCP server は Streamable HTTP transport にも対応しているので、Copilot Studio の「MCP server をエージェントに追加」ウィザードから URL を入れるだけで、Teams や Outlook で動く Copilot から Watchdog の 24 ツールを呼べる状態に持ち込めます。

Microsoft は 2026 年に Copilot Studio が MCP server を取り込めるようにしており、エージェント間連携 (A2A) も今後広がっていくと見ています。本実装はその流れにそのまま乗れる構造にしているため、「自社業務に特化した Watchdog を作る」→「Teams から呼べるようにする」→「他のエージェントと連携させる」という発展を、追加実装をほとんど書かずに段階的に進められると考えています。

## Azure上での構築と工夫点

普段、私はAWSをメインのクラウドとして使用していますが、今回はAzureでの実装を行いました。デモのためだけの構成ではなく、審査期間中は常時稼働させ、業務にもそのまま持ち込める構成を目指しました。

### 採用したサービスと、その役割

| カテゴリ | サービス | 役割 |
| --- | --- | --- |
| AI 基盤 | Azure OpenAI gpt-5.5 (Microsoft Foundry, 2026-05 GA) | Watchdog の LLM。ナラティブ生成、SOP 解析 |
| Agent SDK | Microsoft Agent Framework 1.0 (2026-04 GA, Semantic Kernel 2.0 の後継) | 4 フェーズの orchestration、function-calling の auto-invoke、`FunctionMiddleware` による tool 観測 |
| ツール抽象 | MCP（オープンプロトコル / Azure 非依存） | 24 ツールを集約し、Microsoft 側のホスト群と共有 |
| 実行基盤 | Azure Container Apps | FastAPI / Watchdog のホスト。scale-to-0 対応 |
| データ | Cosmos DB (NoSQL, Serverless) | グラフのノード / エッジを永続化 |
| フロント | Static Web Apps + Linked Backend | UI と API を別ドメインに分離 |
| ネットワーク | VNet + Private Endpoint | Cosmos のパブリックアクセスを無効化 |
| 認証 | Managed Identity / Easy Auth | キーレスな Cosmos 接続と、公開エンドポイントの保護 |
| IaC | Terraform | 構成全体をコード化し、差分管理 |

Agent Framework 1.0 については、`FunctionMiddleware` を使うとツール呼び出しを 1 行も書かずに観測できる点が便利で、これを使って 24 ツールすべてのリアルタイム可視化を実装しました。

### デプロイ構成

公開している入口は Static Web Apps の 1 つだけで、API も Cosmos も VNet の内側に閉じています。外部ホストは MCP 経由で同じツール群につながり、Container App から Cosmos へは Managed Identity + Private Endpoint、推論は Azure OpenAI を使います。

![Watchdog Agent の Azure 構成図](https://static.zenn.studio/user-upload/deployed-images/868091f5c76670c2423b6d7e.png?sha=190556b5ba1eb99155e8271443e1e0d021245d7d)

*外からの入口は1つだけ、データは内側に閉じる。セキュリティを優先した構成*

### コストと運用

scale-to-0 にしているのでアクセスの無い間のコンピュート料金はかからず、稼働中は月 25 ドル前後 + Azure OpenAI の従量です。審査期間外はコンピュートと公開経路だけを落とす stop / resume スクリプトで月 12 ドル前後に圧縮でき、構成全体は Terraform で管理しています。

## ツールの中身：古典的なグラフ解析を LLM が呼べる形にする

ここまで「Watchdog がツールを自律的に選ぶ」と書いてきましたが、同じくらい大切なのは、そのツールの中身が何か、という点だと思います。今回はプロセスマイニングとグラフ理論の分野で 20 年以上積み上げられてきた解析手法を色々と実装しています。

業務プロセスの研究分野では、SOP や実行ログを有向グラフとして扱い、構造的な欠陥や逸脱を測る指標が体系的に整理されています。BPIC (Business Process Intelligence Challenge) で繰り返し検証された手法や、Wil van der Aalst らによるプロセスマイニングの教科書的なアプローチ、そしてグラフ理論の古典 (PageRank / Betweenness / Louvain) などです。これらを Python + NetworkX で実装し、その結果を LLM が読める形に整えた、というのが今回の中身になります。

### レイヤ構造と利用パッケージ

全体を層で見ると、次のようになっています。どの層でどの OSS パッケージを使っているかも合わせて示しました。

*LLM は『何を調べるか』を決めるだけ。実際の計算は昔ながらの確実な手法に任せている*

上から、起動トリガー（`watchdog` のファイル監視 + バックログ）、エージェント・ループ（`agent-framework` 1.0 + `openai` / `httpx`）、ツール層（`mcp` / FastMCP を `fastapi` + `uvicorn` で公開）、アナリティクス（`networkx` / `leidenalg` + `igraph` / `scipy` / `pm4py`）、永続層（`azure-cosmos` + `azure-identity`）という構成です。

LLM は最上層で「今このワークフローの構造品質を見たい」と判断するだけで、その下のグラフ計算はすべて古典的なアルゴリズムが受け持ちます。LLM の役割は計算そのものではなく、何を計算するかを選ぶことです。この分業が、エージェンティックでありながら不確実性は最小限、という性格につながっているように思います。

### ツール一覧：何をするか

24 ツールの主なものを、何をするか（背後にある古典手法）とあわせて挙げます。

| ツール | 何をするか（手法） |
| --- | --- |
| `get_structural_quality` | SOP の構造欠陥（孤立・行き止まり・循環）をグラフ探索で検出 |
| `list_activities` | PageRank・媒介中心性で「重要な手順／ハブ」を抽出 |
| `compare_workflows` | 複数プロセスの Jaccard 類似度・共通遷移から再利用候補を発見 |
| `list_top_variants` | 実行ログから DFG を作り、頻出パターンを抽出（プロセスマイニング） |
| `detect_obs_drift` | 週ごとの DFG のエッジ頻度差分で、観察（実態）の変化を検出 |
| `take_norm_snapshot` + `detect_norm_drift` | SOP を KPI ベクトル化し、時系列差分で規範の変化を検出 |
| `detect_norm_obs_gap` | SOP と実ログを突き合わせ（コンフォーマンス検査）、未浸透・未文書化を検出 |
| `get_coverage` | Step × 規制／役割／文書 の網羅マトリクスで「穴」を発見 |
| `scan_ambiguity` | 「適切に」「速やかに」等の曖昧表現を NLP + LLM で検出 |
| `cluster_findings`／`propose_root_causes`／`reassess_severity` | 所見のクラスタ化・根本原因仮説・重大度の再評価（LLM が担う業務判断） |

検出系のツールはいずれも **決定論的** で、同じデータなら同じ結果が出ます。LLM が担うのは最後の 3 つ、つまり所見をまとめて共通テーマや根本原因を語り、全体を見て重大度を見直すという業務判断の部分です。古典手法で安定して検出し、LLM で業務文脈を解釈する。この役割分担が、検出精度と説明のしやすさの両立につながっていると思います。

なお今回のデモでは、BPIC 2012 (公開ベンチマーク、ローン申請プロセスの合成版) と、公開情報サンプル SOP から AI が生成した KG (KYC 口座開設 / 住宅ローン審査) を使用しています。実在組織の内部 SOP は含みません。

## デモの解説

## 残課題と次の展開

現状でも本番で運用できそうな状態になっていますが、課題はいくつか残っています。

* **規範 ↔ 観察 の乖離検出の精度**：今は SOP の手順名と実ログの作業名を、手作業で作った対応表（CSV）で突き合わせている。今後はエンベディングによる意味的な対応付けで精度を上げたい
* **マルチテナント対応 / Agent 365 への登録 / A2A の実機接続**：本番運用に向けた次の宿題
* **状態の Cosmos 永続化**：今回はデモ用のグラフデータと初期状態を Container イメージにそのまま同梱（焼き込み）して代替している。本番では Cosmos に永続化したい
* **自律エージェントの暴走防止**：Watchdog は変化検知に加えて `enqueue_watchdog_task` で自分でタスクを積めるため、構造上はタスクが無限に増殖しうる。指示なしで動くエージェントだからこそ、実行回数やタスク深さの上限、人間の確認を挟むチェックポイントといったガードレールと監視の仕組みが要る

これからは実際のデータでどこまでうまく稼働するかを検証してみたいと思います。

## まとめ

「問われずとも気づきを返す AI」が経営判断の材料になる未来は、もう手の届くところまで来ていると感じています。**規範と実態の乖離を見続ける** という、地味で人手も足りない仕事をエージェントが引き受けることで、BPR を "一度実施すれば終わる仕事" ではなく "エージェントが継続的に支える活動" に変えていけるのではないでしょうか。

そして冒頭に挙げた **暗黙知の形式知化** という観点でも、「規範 (手順書) と実態 (実ログ) の両面から、本来あるべき業務の形を継続的に固定していく」という今回のアプローチは、一つの有効な手段になり得ると考えています。

Watchdog の適用範囲は製造業に限りません。SOP と実ログの両方を持つ業務であれば、どこでも当てはまります。MCP 経由で 24 ツールを外部に公開しているので、自社業務の KG に差し替えれば、そのまま自社向けの Watchdog として動かせる構成にしています。

Microsoft Agent Hackathon 2026 のテーマ「業務改革につながる Agentic AI」に対して、一つの実装解を提示できたのではないかと考えています。
