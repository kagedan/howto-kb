## この記事でやること

**Claude Code が計画を立てて指示を出し、Codex が実装する。** タスクの受け渡しも、実装中の質問も、エージェント同士が直接やりとりします。人間がやるのは、最初にやりたいことを伝えることと、ときどき様子を見て、承認や仕様の判断に答えることくらいです。

![herdr の1画面に leader / coder / reviewer の3ペインが並び、サイドバーには project-a と project-b の2チームが表示されている](https://static.zenn.studio/user-upload/deployed-images/05f2c70a0306127a390d91db.png?sha=621c6a62698e17f07db322bb7a47ad5cfe4d59f2)  
*この記事の到達点。leader(Claude Code)の指示で coder(Codex)が実装し、reviewer(Codex)が承認を返す。サイドバーには2プロジェクトのエージェントが並ぶ*

使う道具は2つだけです。どちらもインストールは1行で、難しい設定は出てきません。シェルスクリプトも書きません。

* [agmsg](https://github.com/fujibee/agmsg) — エージェント同士がメッセージを送り合うための道具(**会話**の担当)
* [herdr](https://herdr.dev/) — 複数のエージェントを1画面に並べて見守るための道具(**見守り**の担当)

話は4段階で進みます。

1. **agmsg でエージェントが協業できる** — リーダーと実装担当の2体で、指示と質問が行き交うのを確かめる
2. **herdr で協業状態が確認しやすい** — 誰が働いていて、誰が承認待ちで止まっているか、一目でわかるようにする
3. **リアルタイムに受け取れない・止まってしまっても、確認が取れる** — Codex ならではの弱点と、動かなくなったときの対処を押さえる
4. **チームを増やして、複数プロジェクトを同時に走らせる** — リーダー1人+役割分担のチームをもう1つ組み、2案件を並行で進める

## なぜ「Claude が指示、Codex が実装」なのか

最初に役割分担の理由を説明しておきます。

**リーダーは Claude Code。** 人間の曖昧な依頼をくみ取って計画に翻訳したり、上がってきた成果を検収したりする「窓口」の仕事は、対話が得意な Claude 系が向いています。リーダーが意図を取り違えるとチーム全体が間違った方向に走るので、ここには一番信頼できるモデルを置きます。

**実装は Codex。** 手を動かす仕事は数をこなす必要があるので、実装力があってサブスクの消費枠も別になる Codex に任せます。実装とレビュー(検収)を別のモデルに分けると、「自分で書いたコードを自分で甘めにチェックする」問題も避けられます。

この記事でやるのは、この「窓口は Claude、数をこなす実装は Codex」という役割分担を、**既製の道具2つだけで体験できる最小構成**です。

なお、この組み合わせは必須ではありません。使えるモデルや契約プランに応じて、リーダーと実装を入れ替えても仕組み自体はそのまま動きます。

## 1. agmsg でエージェントが協業できる

### agmsg とは

エージェント同士がメッセージを送り合うための道具です。仕組みはシンプルで、パソコンの中にメッセージの保管庫(SQLite というデータベースファイル1個)を置いて、みんなでそこを郵便受けとして使うだけ。サーバーもネットワーク設定も要りません。必要なのは `bash` と `sqlite3` だけです。

そして大事なのが、agmsg は**スキル**(エージェントに新しい能力を追加する仕組み)として入ることです。インストールしたあとの操作は、ぜんぶエージェントへの日本語の指示で済みます。Claude Code だけでなく、Codex、Gemini CLI、GitHub Copilot CLI など主要な CLI エージェントに対応しています。

### セットアップ

インストールはこの1行です。

次に、デモ用のプロジェクト `project-a` を `~/projects` の下に作ります。

```
mkdir -p ~/projects/project-a && cd ~/projects/project-a
```

このプロジェクトに、チームの運用規約を `CLAUDE.md` として1枚書きます。

CLAUDE.md

```
# チーム運用規約(team-a)

このプロジェクトは leader と coder の2役割で協業する。
自分の役割は agmsg に登録した名前で決まる。未参加なら人間の指示に
従って参加し、以後は自分の役割の節に従うこと。

## 共通ルール
- 役割間の連絡はすべて agmsg で行う
- セッション開始時は自分宛の未読と history を確認し、
  途中からでも文脈を引き継ぐ

## leader の役割
- 要件をタスクに分解し、coder に agmsg で依頼する
- 成果物を検収(レビュー)し、修正が必要なら具体的に指摘して差し戻す
- 自分では実装しない

## coder の役割
- leader からの依頼に従って実装する
- 不明点は勝手に判断せず、agmsg で leader に質問する
- 完了したら変更内容の要約を leader に報告し、次の指示を待つ
```

ポイントは coder 側の「**不明点は leader に質問する**」の一文です。エージェントはわからないことがあると推測で突っ走りがちなので、「聞いていい相手がいる」と教えておくと手戻りがぐっと減ります。

規約に「自分の役割は agmsg に登録した名前で決まる」と書いてあるのもミソで、役割ごとにファイルを分けなくても、**1枚の規約で全員が自分の節だけを読む**ようになります。あとでメンバーを増やすときも、この規約に節を足すだけです。

規約を書いたら、最後に仕込みをもう1つ。Claude Code は `CLAUDE.md`、Codex は `AGENTS.md` という名前のファイルを、プロジェクトの指示ファイルとして自動的に読み込みます。いま書いた規約を両方に読ませたいので、シンボリックリンクを張っておきます。

```
ln -s CLAUDE.md AGENTS.md   # どちらのエージェントにも同じ規約を読ませる
```

### チームに参加させる

Claude Code と Codex を1体ずつ起動して、それぞれに話しかけてチームに参加させます。この時点では、普通にターミナル2枚でOKです。

**Claude Code(leader)側。** ターミナルを1枚開き、`project-a` で `claude` を起動して、こう話しかけます。

```
/agmsg で team-a に leader として参加して
```

途中で「配信モード」(メッセージの受け取り方)を聞かれたら、そのまま既定値でOKです。Claude Code は `monitor`(保管庫を常時見張っていて、約5秒で届くリアルタイム受信)になります。

**Codex(coder)側。** もう1枚ターミナルを開き、同じ `project-a` で `codex` を起動して、こう話しかけます。

```
$agmsg で team-a に coder として参加して
```

こちらも配信モードは既定値でOKです。Codex は `turn`(自分の作業の区切りごとに郵便受けを確認)になります。

![左: Claude Code が team-a に leader として参加(monitor モード)、右: Codex が coder として参加(turn モード)](https://static.zenn.studio/user-upload/deployed-images/4031910440398c5cf2bf46a9.png?sha=c2ee6a3d60cc2b37c81a8db4fcc32a50b62563a8)  
*左が leader(Claude Code)、右が coder(Codex)。それぞれ既定の配信モードで team-a に参加したところ*

**この受け取り方の違いが、第3章の伏線です。**

### まずは疎通確認

いきなり実装を頼む前に、メッセージが往復するかだけ確かめておきます。

登場するのは、**あなた(人間)・leader(Claude Code)・coder(Codex)** の3者です。あなたが話しかけるのは leader だけで、coder への連絡は leader が agmsg で行います。この関係は、この後の実装デモでもずっと同じです。

**leader(Claude Code のターミナル)** に、こう話しかけてください。

```
coder に「聞こえますか？」と送って、返事が来たら教えて
```

leader は agmsg で coder にメッセージを送ります。ただし、このまま待っていても coder は返事をしません。**turn モードの Codex は、自分が動いたときにしか受信箱を確認しない**ため、暇にしている coder はメッセージが来たこと自体に気づけないのです(この問題は第3章で解決します)。

今回は人間が起こしてあげます。**coder(Codex のターミナル)** に、こう話しかけてください。

coder がメッセージに気づいて返事を送ると、常時見張り(monitor)の leader は数秒で受け取り、あなたに「返事が来ました」と報告します。確認ポイントは次の3つです。

* leader に「team を確認して」と頼むと、leader と coder が同じ team-a のメンバーとして表示される
* coder の返事を leader が報告してくれる
* leader に「history を見せて」と頼むと、いま往復した2通が残っている

### 動かしてみる

疎通が取れたら、リーダーに実装を頼んでみます。結果をそろえやすいよう、技術スタックまで指定しておきます。

```
Node.js と Express で、メモリ保存の TODO API を作りたい。
POST /todos(追加)、GET /todos(一覧)、PATCH /todos/:id/complete(完了)の3つ。
npm test で動作確認できるテストも付けて。
計画を立てて、実装を coder に依頼して、完成まで面倒を見て。
```

すると、こんな流れで進みます。

1. leader が計画を立て、agmsg で coder に実装タスクを送る
2. coder が実装に着手。途中で不明点があれば leader に質問する
3. leader が約5秒で受信して回答(自分で決めていい範囲なら決め、仕様に関わるなら人間に確認)
4. coder が実装を終えて完了報告
5. leader が検収して、人間に「終わりました」と報告

計画・指示・質問・回答・報告の中身を、人間は一度も取り次いでいません。ただし進行のトリガーだけは別です。疎通確認と同じ理由で、**coder は新着に自分から気づけません**。leader が指示を送ったのに coder が動かない、leader が回答したのに coder が再開しない、というときは、そのつど coder に「agmsg の受信箱を確認して」と声をかけて進めてください。この声かけの面倒くささは、第3章でなくします。

![左: leader が計画を立てて agmsg で coder に仕様を送信、右: coder が受信箱を確認して実装に着手](https://static.zenn.studio/user-upload/deployed-images/cee14e23973969508ab977ef.png?sha=511fe83b56a6f874dc9a79c3cc8897081851f2cd)  
*左の leader が計画を立てて仕様を送り、右の coder が受信して実装方針を立てたところ。leader の依頼にはエンドポイント仕様や検証手順まで書かれている*

これで柱の1本目、「agmsg でエージェントが協業できる」は確認できました。ただ、ターミナル2枚を行ったり来たりして眺めるのは、もう窮屈になってきたはずです。

## 2. herdr で協業状態が確認しやすい

### herdr とは

複数のエージェントを1画面に並べて見守るための道具です。「コーディングエージェント用の tmux」と紹介されることが多いですが、tmux を知らなくても大丈夫。WezTerm や Ghostty のような新しいターミナルアプリに乗り換えるわけではなく、いま使っているターミナルの中で `herdr` を起動すると、その中にエージェントを並べる画面が広がる。いわば **「ターミナルの中のターミナル」** だと思えばOKです。

嬉しいのは次の4点です。

* **マウスで操作できる**。ペインの分割やリネームは右クリック、ワークスペースやペインの切り替えはクリックででき、tmux のような複雑なキー操作を覚える必要がありません
* **状態が色でわかる**。各エージェントの今の状態がサイドバーに色で一覧されます。状態は **working**(作業中)/ **blocked**(承認・質問・権限確認など外部からの入力や判断待ち)/ **done**(作業を終えたが人間が未確認)/ **idle**(手空き)の4つで、本記事ではこの名前で呼びます
* **閉じても続く**。裏でサーバーが動き続けるので、ターミナルの画面を閉じてもエージェントたちは働き続けます(ただし PC 本体がスリープすれば止まります)
* **外からも見える**。SSH で入れるマシンで herdr を動かしておけば、スマホからも様子を確認できます

### セットアップ

インストールはこちらも1行です。

```
curl -fsSL https://herdr.dev/install.sh | sh
```

インストールできたら、いつものターミナルで起動します。この後の作業はすべて herdr の画面の中で行います。

### エージェントを herdr の中で立ち上げ直す

herdr を起動したら、人間がやるのは leader を立ち上げるところまでです。

1. 起動するとワークスペース(作業画面)が1つ開いている状態で、最初はホームディレクトリにいるので、`cd ~/projects/project-a` で移動する
2. `claude` を起動して、「team-a に leader として参加して。未読と history も確認して」と再参加させる(チームのやりとりは agmsg の保管庫に残っているので、依頼や質問の経緯はそのまま引き継がれます)

残りのペイン(区画)の用意は、leader にやらせてしまいます。herdr はコマンドでペインの分割・リネーム・入力ができるので、leader にこう頼むだけです。

```
herdr pane split で右にペインを作って codex を起動して。
ペイン名は、自分を leader、隣を coder にして。
codex が起動したら「$agmsg で team-a に coder として参加して」と入力して、
チームに参加させて。各ステップがうまくいったか確認しながら進めて。
```

leader が `herdr pane split --direction right` でペインを作り、`herdr pane run` で codex を起動して名前を付け、参加の指示まで隣のペインに打ち込んでくれます。人間は眺めているだけで、チームが herdr の中で立ち上げ直されます。

なお「参加して」といっても、team-a には第1章で登録済みなので、メンバーが重複して増えるわけではありません。agmsg が同じプロジェクトの登録を見つけて、同じ leader / coder として続きから参加する形になります。

ペイン名は各ペインの上部に表示されるうえ、第3章で leader が相手のペインを探す手がかりにもなるので、**agmsg のエージェント名と同じにしておくこと**が大事です。

(もちろん、右クリックメニューから手動でペインを分割・リネームすることもできます)

![herdr の画面。左にサイドバー、右に leader と coder の2ペインが並んでいる](https://static.zenn.studio/user-upload/deployed-images/6ad5f2ebbfc55cb4b16a7047.png?sha=d9ffc3b92652bdd20e52bc9a094f996c77736855)  
*leader が自分でペインを分割して codex を起動し、チームを herdr の中で立ち上げ直したところ。左のサイドバーにエージェントが一覧される*

### 何が変わるか

今度は、第1章で作った TODO API への機能追加を leader に頼んでみます。

```
TODO API に DELETE /todos/:id(削除)を追加したい。
実装を coder に依頼して、完成まで面倒を見て。
coder に agmsg でメッセージを送ったら、coder のペインに
「agmsgの受信箱を確認して」と入力して知らせて。
```

最後の一文がポイントです。coder は turn モードで新着に自分から気づけないので(第1章の「気づかない問題」)、送りっぱなしにせず、ペインへの入力で知らせるところまで leader に頼んでいます。

見え方がこう変わります。

* leader に指示を出すと、サイドバーで leader が working に
* leader が指示を送って coder を起こすと、coder も working に。**2つ並んで働いているのが一目でわかる**
* coder がファイル編集の承認待ちで止まると **blocked** に変わる → 気づいた人間が承認するだけ
* 会話の中身が気になったら、leader に「history 見せて」と言えばチームのやりとりが時系列で出てくる

![DELETE /todos/:id の依頼を受けた leader が agmsg でタスクを送り、coder のペインに「agmsgの受信箱を確認して」と入力して起こしている herdr の画面](https://static.zenn.studio/user-upload/deployed-images/8c613989297ea4ddca7beef7.png?sha=683404258f8ff4daf7f5eaf97d8e6fcbc9d08f75)  
*leader がタスクを送り、coder のペインに通知文を入力して起こしたところ。coder が受信して実装を始める様子まで1画面で見渡せる*

そして夜に依頼してターミナルを閉じ、翌朝つなぎ直したら実装と検収が終わっている。herdr が常駐してくれるおかげで、これができます。ただし動き続けるのは、PC 本体が起きている間だけです。ノート PC の蓋を閉じるとスリープで止まります。本格的に放置したい場合は、スリープしない常時稼働のマシン(リモートサーバーなど)で herdr を動かします。手元からは SSH で接続して `herdr` を再実行すれば、動いているセッションにそのままつながります(スマホの SSH クライアントからでも同じです)。

!

**コラム: 放置運転と承認のトレードオフ**

夜中に coder が blocked(承認待ち)で止まると、朝まで止まったままです。放置運転を本気でやるなら、承認を減らす起動オプションを検討します。

* Claude Code: `claude --permission-mode acceptEdits`(ファイル編集だけ自動承認)
* Codex: `codex --sandbox workspace-write --ask-for-approval never`(書き込みをワークスペース内に制限したうえで、承認の問い合わせをなくす。枠の外の操作は承認を求めず、失敗としてエージェントに返ります)

さらに全部スキップする手段(Claude Code の `--dangerously-skip-permissions`、Codex の `--dangerously-bypass-approvals-and-sandbox`)もありますが、エージェントが何でも実行できるようになります。使うならデモ用プロジェクトの中だけにして、普段の作業環境では慎重に。

柱の2本目、「herdr で協業状態が確認しやすい」もできました。ここまでは順調ですが、実はここまでのデモには、たまたまうまくいっていた部分があります。

## 3. リアルタイムに受け取れない・止まってしまっても、確認が取れる

### Codex は「声をかけられるまで気づかない」ことがある

第1章の伏線を回収します。2体の受信方法はこう違いました。

|  | 受信方法 | 新着に気づくタイミング |
| --- | --- | --- |
| Claude Code(leader) | monitor: 常に郵便受けを見張る | 約5秒 |
| Codex(coder) | turn: 自分の作業の区切りで郵便受けを確認 | **次に動いたとき** |

`turn` の Codex は、**何もしていない(idle の)あいだは新しいメッセージに気づけません**。leader が指示を送っても、coder が暇なままじっとしている、ということが実際に起きます。

(agmsg には Codex 向けのリアルタイム受信もベータ版として用意されていますが、公式 README でも実験的な仕組みに依存すると注記されているため、本記事では既定の `turn` で進めます)

だからこそ第1章では、人間がそのつど coder に「受信箱を確認して」と声をかけて進めていました。エージェントに任せているはずなのに、伝書鳩の役目だけ人間に残っている状態です。

### 解決: 起こす係も leader にやらせる

第2章の機能追加では、依頼文に「coder に知らせて」と書き足すことでこの問題を乗り切りました。ただ、毎回依頼文に書くのは面倒ですし、書き忘れればそこで止まります。そこで、このルールを規約に書いておいて、**起こす係は leader に固定します**。

`CLAUDE.md` の leader の節に、2行足します。

CLAUDE.md

```
- agmsg でメッセージを送ったら、herdr で「agmsgの受信箱を確認して」と入力して起こす
- 相手が working(作業中)や blocked(入力待ち)のときは、何も打ち込まずに待つ
```

規約を書き換えたら、leader に「CLAUDE.md を読み直して」と一声かけて(再起動でもOK)、機能追加をもう1つ頼んでみます。

```
TODO API に GET /todos/count(件数)を追加したい。
coder に依頼して、完成まで面倒を見て。
```

今度は、coder が idle のままでも人間の出番はありません。leader が指示を送り、ペインの状態を確かめ、自分で「受信箱を確認して」を打ち込んで起こすところが観察できるはずです。質問への回答を返すときも同じルールが働くので、質問→回答→再開の往復も途切れません。伝書鳩の役目は、こうして leader に引き継がれました。

![GET /todos/count の依頼で leader が規約に従って coder を起こし、完了報告を待っている herdr の画面](https://static.zenn.studio/user-upload/deployed-images/c10cbbc2c85a10adde19bc6c.png?sha=b8b08364a644d5bf5fafd22298c150fe842f5b23)  
*leader が自分で coder を起こして「working になったので規約に従って完了報告を待つ」と判断しているところ。人間は何もしていない*

### 止まってしまったときの見分け方と対処

もう1つの「止まった」も、3パターンに分けて考えれば怖くありません。

**① coder が暇そうにしている(idle なのに仕事が残っている)。** 起こし損ねです。メッセージは agmsg の保管庫に残っているので消えてはいません。leader に「coder に声をかけて」と頼めば回収できます。

**② blocked(入力待ち)で止まっている。** これは故障ではなく、承認・質問・権限確認など、外部からの入力や判断を待っている正常な状態です。サイドバーで気づいたらペインをのぞき、人間の承認待ちなら承認する、leader への質問が滞っているなら leader 側の様子を見る、と待ち先に応じて対応します。なお blocked のペインに「受信箱を確認して」を打ち込むと承認画面に文字が流れ込んでしまうので、声をかけるのは idle のときだけにします。

**③ ペインごと落ちている。** herdr の画面でわかります。復旧も leader に頼めます。

```
coder のペインで codex を起動し直して。起動したら、team-a への再参加と、
未読・history・git の差分の確認、途中だった作業の再開を指示して。
```

チーム間のやりとりは agmsg の保管庫に全部残っているので、依頼・質問・回答の経緯はそこから復元できます。落ちたエージェントの内部の思考までは戻らないため、実際の作業状態は git の差分やテスト結果で確かめさせるのがポイントです。

これで柱の3本目もそろいました。**指示する・見守る・起こす/立て直す**、ぜんぶ日本語の指示と画面の確認だけで回ります。

## 4. チームを増やして、複数プロジェクトを同時に走らせる

2体で回ることがわかったら、あとは同じやり方の延長です。仕上げに2つ広げてみます。

* チームごと増やして**別プロジェクトを並行で走らせる**
* メンバーを増やして**リーダー1人+役割分担のチーム**にする

### project-b: leader / coder / reviewer の3体チーム

別のプロジェクト `project-b` で、レビュー担当を足した3体チームを組みます。

| ペイン名 | エージェント | 役割 |
| --- | --- | --- |
| leader | Claude Code | 計画・指示・質問対応・最終検収 |
| coder | Codex | 実装(このチームで唯一ファイルを編集する役割) |
| reviewer | Codex | コードレビュー(実装した本人以外の目) |

まず、herdr で新しいワークスペースを開きます。サイドバーの spaces 欄の下にある「new」をクリックするだけです(**project-a のチームは、そのまま動かしておけます**)。新しいワークスペースが開いたら、プロジェクトを作って移動します。

```
mkdir -p ~/projects/project-b && cd ~/projects/project-b
```

![サイドバーの spaces 欄に project-a と project-b が並び、新しいワークスペースでプロジェクトを作成したところ](https://static.zenn.studio/user-upload/deployed-images/4b2b75c3940001ffbce52f1e.png?sha=9331d43d8a8f5cbf62845d9ccde4a841c0545be4)  
*「new」で開いた2つ目のワークスペース。project-a のチームは裏でそのまま動いている*

あとは project-a と同じです。規約を `CLAUDE.md` に書き、シンボリックリンクを張る。共通ルールはそのままで、reviewer の節が増え、3体で回すためのルールが少し足されています。

CLAUDE.md

```
# チーム運用規約(team-b)

このプロジェクトは leader と coder と reviewer の3役割で協業する。
自分の役割は agmsg に登録した名前で決まる。未参加なら人間の指示に
従って参加し、以後は自分の役割の節に従うこと。

## 共通ルール
- 役割間の連絡はすべて agmsg で行う
- セッション開始時は自分宛の未読と history を確認し、
  途中からでも文脈を引き継ぐ

## leader の役割
- 要件をタスクに分解し、coder に agmsg で依頼する
- agmsg でメッセージを送ったら、herdr で「agmsgの受信箱を確認して」と入力して起こす
- coder や reviewer から一報を受けたら、その宛先も同じように起こす
- 相手が working(作業中)や blocked(入力待ち)のときは、何も打ち込まずに待つ
- reviewer の承認報告を受けて最終検収する。仕様レベルの判断は leader が行う
- 自分では実装もコードレビューもしない

## coder の役割(このチームで唯一ファイルを編集する役割)
- leader からの依頼に従って実装する
- 実装を終えたら reviewer に直接レビューを依頼し、leader にも一報を入れる
- reviewer の指摘に対応して再依頼する。不明点や仕様の疑問は leader に質問する

## reviewer の役割
- coder からの依頼でコードをレビューする。ファイルは直接編集しない。
  指摘はすべて agmsg で coder に返し、返したことを leader にも一報する
- 指摘が解消したら leader に承認を報告する。仕様の疑問は leader に確認する
```

coder と reviewer のルールに「**leader にも一報**」が入っているのがポイントです。coder も reviewer も turn モードの Codex なので、「気づかない問題」は3体になっても同じ。起こす係(herdr のペイン操作)は leader なので、当事者同士のやりとりにも一報を挟んで、leader が「次に誰を起こすべきか」を知れるようにしています。leader の節の「一報を受けたら、その宛先も同じように起こす」が、それを受け止めるルールです。

規約を書いたら、project-a と同じようにリンクを張ります。

```
ln -s CLAUDE.md AGENTS.md
```

続いて `claude` を起動し、チームへの参加からペインの用意まで、まとめて頼みます。

```
/agmsg で team-b に leader として参加して。
参加できたら herdr pane split でペインを2つ作って、それぞれ codex を起動して。
ペイン名は、自分を leader、2つを coder と reviewer にして。
codex が起動したら、coder のペインには
「$agmsg で team-b に coder として参加して」、
reviewer のペインには
「$agmsg で team-b に reviewer として参加して」と入力して、
チームに参加させて。各ステップがうまくいったか確認しながら進めて。
```

leader が自分の参加を済ませ、ペインを作り、codex を起動し、参加の指示まで打ち込んでくれます。

![leader が team-b への参加とペイン作成から coder / reviewer の参加確認まで「全ステップ完了」と報告している herdr の画面](https://static.zenn.studio/user-upload/deployed-images/e8d94fc95f84d7878e28945f.png?sha=3b5a03eb8ac74cf9c1b7ed37e42a73d2679bef8a)  
*3体チームのセットアップを終えた leader の報告。coder と reviewer のペインでも team-b への参加が確認できる*

依頼はこんな感じです(`project-b` は空のプロジェクトなので、題材も新規に作らせます)。

```
Node.js と Express で、期限(due date)付きの TODO API を新しく作りたい。
要件はタスクの追加・一覧・完了と、期限の登録・期限順の一覧。
計画を立てて coder に依頼し、レビューと検収まで進めて。
```

流れはこうなります。

1. 人間 → leader に依頼
2. leader → coder に実装指示(idle なら起こす)
3. coder → 実装を終えたら reviewer にレビュー依頼、leader に一報 → 一報を受けた leader が reviewer を起こす
4. reviewer → 指摘を coder に返し、leader に一報 → leader が coder を起こす
5. OK が出たら leader が検収して人間に報告

「実装した本人がレビューもする」構図が消えるので、2体のときより品質の底が上がります。

### 2チームを同時に走らせる

ここまでで、herdr には `project-a` と `project-b` の2つのワークスペースができています。実はこの2チーム、**同時に走らせられます**。

* agmsg のチームは名前(`team-a` / `team-b`)でわかれているので、メッセージが混線することはありません
* herdr のサイドバーには全ワークスペースのエージェントが並ぶので、**2案件・5体の状態が一目でわかります**
* 人間の仕事は、各チームの leader に依頼を1つずつ投げて、あとは blocked を見つけたら承認するだけ

夜に2案件仕込んでおき、朝つなぎ直したら両方の検収報告が届いている(第2章で触れたとおり、マシンが起きていることが条件です)。複数プロジェクトの同時進行が、規約のコピーと参加の指示だけで手に入ります。

ここからさらに、調査係や進行管理係を足して役割を細かく分けていくこともできます。やることは同じで、規約に節を足して、参加の指示を1人分増やすだけです。

## ハマりどころ

* **参加直後の1通目に反応しないことがある。** monitor モードは、そのセッションで最初のターンを踏むまで1通目の受信に反応しません(公式 README に明記されている仕様です)。チームを組んだ直後は、monitor モードの leader に「よろしく」でも何でも一声かけておくと確実です
* **依頼には「完成まで面倒を見て」を入れる。** 「指示して」だけだと、指示を送った時点で自分の仕事が終わったと思ってしまうことがあります。往復の終了条件まで言うのがコツです
* **ペイン名は agmsg のエージェント名と必ず揃える。** leader が起こす相手のペインを探す手がかりはこの名前だけです(人間が画面上で見分ける目印にもなります)
* **leader が herdr の使い方を知らないことがある。** herdr は新しい道具なので、AI が学習していないことがあります。それでも通常は `herdr --help` などを自分で実行して、調べながら進めてくれます。手が止まっているようなら、「herdr のヘルプを確認して」と促すか、使うコマンド(`pane split` / `pane rename` / `pane run`)を依頼文に書き添えれば進みます
* **どちらの道具も開発が活発。** この記事は冒頭のバージョンで検証しています。挙動が違ったら公式ドキュメントへ

## まとめ

* **agmsg でエージェントの協業は成立する。** Claude Code(leader)が計画して指示し、Codex(coder)が実装して、わからないことは質問で返す。人間の仕事は要所の承認と判断だけになる
* **herdr で協業状態が確認しやすい。** 状態はサイドバー、会話は history、画面はペイン。ターミナルを閉じても続き、SSH 環境があればスマホからも見回れる
* **リアルタイムに受け取れなくても、止まっても、確認が取れる。** 起こす係は herdr のペイン操作を規約で教えた leader に任せ、blocked は人間が対応し、落ちたら再起動して history と git 差分から立て直す
* **チームの拡張は「規約の節+参加の指示」が増えるだけ。** リーダー1人+役割分担のチームも、複数プロジェクトの同時進行も、同じやり方の延長で組める
