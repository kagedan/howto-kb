私は常に利用制限に怯えながら利用しています。

そこで、トークン量やAPI料金に置き換えると5時間制限はどれくらいのキャパなのか。自分の環境で計測してみました。

この記事では、5時間枠の推定を主な結果として扱います。7日枠は表示上の増分が小さいため、かなり粗い参考値です。

| 参考: 今回条件での7日枠100%換算 | 約8,859万トークン相当 | 約7,087万〜1億1,812万 | 約$850（約$680〜$1,134） |

## なぜ測ったのか

Claude Codeには利用制限があります。[公式ヘルプ](https://support.claude.com/ja/articles/11647753-%E4%BD%BF%E7%94%A8%E9%87%8F%E3%81%A8%E9%95%B7%E3%81%95%E3%81%AE%E5%88%B6%E9%99%90%E3%81%AF%E3%81%A9%E3%81%AE%E3%82%88%E3%81%86%E3%81%AB%E6%A9%9F%E8%83%BD%E3%81%97%E3%81%BE%E3%81%99%E3%81%8B)でも、利用量は会話の長さ、添付ファイル、現在の会話量、使うモデルや機能によって変わり、セッションベースの利用制限は5時間ごとにリセットされると説明されています。

ただ、普段使っていると、表示される%だけでは判断しにくい場面があります。

「5時間制限って、実際どのくらい使えるの？」

「自分はなぜ2時間くらいで100%まで使い切ってしまうの？」

「週次制限も4日目くらいで尽きるけど、使い方が悪い？」

「追加利用すると、どれくらい費用が増えるの？」

こういう疑問が残ったままだと、ずっと残量を気にしながら作業することになります。必要なときは追加利用すればよいのですが、どれくらい重い使い方なのかが見えないまま追加し続けるのは落ち着きません。

そこで、Claude Codeの利用率がどれくらいのトークン量で増えるのかを測ってみました。

## 検証結果

今回の検証では、固定の大きめのプロンプトをClaude Codeに投げ、5時間利用率が50ポイント以上増えたところで停止しました。その開始前と終了後の利用率を比較しています。

5時間利用率は、7%から62%まで増えました。差分は+55ポイントです。

この区間で、Claude CodeのJSON出力から集計したトークン内訳は以下です。

| 種別 | トークン数 |
| --- | --- |
| 通常入力 | 50 |
| キャッシュ作成 | 3,344,213 |
| キャッシュ読込 | 179,920 |
| 出力 | 19,500 |
| 観測トークン合計（単純合算） | 3,543,683 |

この3,543,683トークンで5時間利用率が55ポイント増えました。同じトークン構成が続くと仮定して、単純に100%へ線形換算するとこうなります。

```
3,543,683 tokens * 100 / 55 = 6,443,060 tokens
```

つまり、今回の条件では、5時間枠100%は表示値ベースで約644万トークン相当に見えました。

同じ区間で、7日利用率は64%から68%まで増えました。差分は+4ポイントです。こちらも同じように線形換算すると、7日枠（週次制限）100%は表示値ベースで約8,859万トークン相当に見えます。

ただし、7日枠は差分が4ポイントしかありません。整数%表示の丸めだけでも大きく振れるため、5時間枠よりかなり粗い参考値です。丸めを考えると、目安は次の範囲まで広がります。

| 制限枠 | 表示上の差分 | 丸めを考えた差分の目安 | トークン換算の目安 |
| --- | --- | --- | --- |
| 5時間枠 | +55ポイント | +54〜56ポイント | 約633万〜656万 |
| 7日枠 | +4ポイント | +3〜5ポイント | 約7,087万〜1億1,812万 |

## API料金に換算するとどれくらいか

トークン量だけだと直感的にわかりにくいので、API料金にも換算しました。

2026年5月6日時点の[Claude API Docsの料金ページ](https://platform.claude.com/docs/ja/about-claude/pricing)を参照し、Opus 4.7の入力、出力、キャッシュ作成、キャッシュ読込の単価を使って再計算しています。

今回使った単価は以下です（MTok = 100万トークン）。

| 種別 | 単価 |
| --- | --- |
| 通常入力 | $5 / MTok |
| 5分キャッシュwrite | $6.25 / MTok |
| 1時間キャッシュwrite | $10 / MTok |
| キャッシュread | $0.50 / MTok |
| 出力 | $25 / MTok |

今回のキャッシュ作成は、Claude Codeの出力上で `ephemeral_1h_input_tokens` として観測されました。そのため、5分キャッシュwriteではなく、1時間キャッシュwriteの単価で計算しています。

観測区間の内訳はこうなります。

| 種別 | トークン数 | 単価 | 金額 |
| --- | --- | --- | --- |
| 通常入力 | 50 | $5 / MTok | $0.00025 |
| 1時間キャッシュwrite | 3,344,213 | $10 / MTok | $33.44213 |
| キャッシュread | 179,920 | $0.50 / MTok | $0.08996 |
| 出力 | 19,500 | $25 / MTok | $0.48750 |
| 合計 | - | - | $34.01984 |

今回のAPI料金換算は、ほとんどが1時間キャッシュwriteです。観測区間の約$34.02のうち約$33.44、つまり約98%を1時間キャッシュwriteが占めています。

つまり、約$62相当という値は、今回のように1時間キャッシュwriteが多く発生したトークン構成を、Claude APIの価格に当てはめた場合の金額です。

同じClaude Codeの利用率増分でも、出力トークンが多い作業、キャッシュreadがよく効く作業、Sonnetを使う作業では、API料金換算の結果は大きく変わります。

| 制限枠 | 表示値ベースのAPI料金換算 | 丸め誤差を考えた目安 |
| --- | --- | --- |
| 観測区間 | 約$34 | - |
| 今回条件での5時間枠100%換算 | 約$62 | 約$61〜$63 |
| 今回条件での7日枠100%換算 | 約$850 | 約$680〜$1,134 |

Claude Codeが返すJSONにも `total_cost_usd` はありますが、今回はローカル推定値として扱いました。この記事では、公式価格表ベースで再計算した値を主にしています。

ここでのAPI料金換算は、サブスクリプション枠の中で使った分をAPI価格で見直した参考値です。実際にこの金額が請求されたわけではありません。

上限到達後に追加利用へ切り替えた場合はAPI価格が関係しますが、この記事の約$62は、今回の観測区間を5時間枠100%へ外挿した値です。追加利用で同じ金額が請求されるという意味ではありません。

補足として、以前、制限が100%になった状態で追加利用に切り替えて使い続けたとき、1時間も経たずに$50を超えた記憶があります。そのときの作業内容やキャッシュの効き方は今回と同じではありませんが、今回の換算値が極端に外れている印象はありませんでした。

## この数字の前提

今回の「約644万トークン相当」は、Claude Codeの5時間制限が常に644万トークンである、という意味ではありません。

今回わかったのは、Team PremiumシートでOpus 4.7 xhighを使い、1時間キャッシュ作成が大きく発生した固定プロンプトを処理した場合、利用率55ポイント分がこのくらいのトークン構成に見えた、ということです。

プラン、モデル、推論設定、キャッシュの効き方が変われば、トークン内訳やAPI料金換算は変わります。Claude Codeの利用率表示は整数%なので、丸め誤差もあります。特に7日枠は+4ポイントしか動いていないため、5時間枠よりかなり粗い参考値です。

また、本記事では通常入力・キャッシュ作成・キャッシュ読込・出力を単純合算した「観測トークン合計」から利用率を外挿しています。Claude Codeのサブスクリプション利用率が、この単純合算トークン数で内部的に計算されている、という意味ではありません。

xhighでは、見えている出力以外にthinking由来のトークンが発生する可能性もあります。この記事ではClaude CodeのJSON出力に含まれる `usage` を正として集計しており、見た目の出力量ではなく、`usage` 上の `output_tokens` を使っています。

## どうやって測ったか

Claude Codeは、Claude Teamプランのサブスクリプション枠で実行しました。`ANTHROPIC_API_KEY` を指定してAPIキー課金で実行したものではありません。

測ったのは、固定ベンチの前後で、Claude Codeのサブスクリプション枠にある5時間・7日利用率がどれくらい動いたかです。組織全体の利用量、日次集計、APIの実請求額を測ったものではありません。

利用率は、ClaudeのSettings > Usageに表示されるPlan usage limitsの進捗バーと同じ種類の値を、計測前後のスナップショットとして記録しました。

計測中は直列で10回実行し、2026-05-05 14:23:06〜14:31:39 JSTの同じ5時間ウィンドウ内で完了しました。同じアカウントでClaude Web、Desktop、他端末のClaude Codeを使う操作はしていません。

固定プロンプトの本文はこの記事では省略しますが、各runで同じ大きめのプロンプトを投げています。Claude CodeのJSON出力では、通常入力は各run 5トークンで、大きい部分は通常入力ではなく1時間キャッシュ作成として計上されていました。

この記事で扱う7日利用率は、保存したレスポンス上の `seven_day` 枠です。別途 `seven_day_sonnet` も取得できていましたが、今回の計測対象には含めていません。`seven_day` とUI上の週次バーの厳密な対応は未確認です。そのため、7日枠の換算値は5時間枠よりもさらに粗い参考値として扱っています。

大まかな流れは次の通りです。

1. 開始前に5時間利用率と7日利用率を保存する
2. 固定プロンプトをClaude Codeに複数回投げる
3. 5回ごとに利用率スナップショットを保存する
4. 5時間利用率が+50ポイント以上動いたら止める
5. Claude CodeのJSON出力からトークン内訳を集計する
6. 利用率差分から100%相当のトークン量へ線形換算する

Claude Codeのトークン内訳は、各実行のJSON出力に含まれる `usage` フィールドを集計しています。トークン消費やコストの扱い方は、OSSの[ccusage](https://ccusage.com/)を参考にしました。ccusageはClaude CodeのローカルJSONLファイルからトークン使用量を集計できるツールです。ただし、今回の制限枠換算はccusageの出力そのものではなく、保存したスナップショットとJSON出力をもとに自分で集計・外挿した値です。

今回の実行では10回成功し、5時間利用率が+55ポイント動いたところで停止しました。

固定プロンプトを繰り返しているため、通常のAPIプロンプトキャッシュであれば2回目以降に大きくcache readへ寄ることを期待するかもしれません。しかし今回のClaude Code実行では、各runで約334K tokensが1時間cache writeとして観測されました。Claude Codeが内部的に構成するコンテキストやセッション状態により、固定プロンプトの大部分がそのままcache hitする形にはなっていないようです。


## 自分の使い方に当てはめる

今回の主な収穫は、私の環境・今回のトークン構成に限れば、「利用率55ポイント分がAPI価格換算で約$34、5時間枠100%へ外挿すると約$62相当」という目安が得られたことです。

たとえば、今回と同じトークン構成に近い重めの作業で5時間枠を2時間で使い切るなら、単純計算では1時間あたり約322万トークン相当のペースで消費していることになります。

週次枠についても、粗い推定ではありますが、表示値ベースで約8,859万トークン相当という参考値があると、4日で尽きるペースがどれくらい重いのかを考えやすくなります。

もちろん、実務のClaude Code利用は固定プロンプトの繰り返しとは違います。実際には、リポジトリの大きさ、読み込むファイル数、auto-compact、やり直し、レビューの往復などで消費量はかなり変わります。

それでも、何も数字がないよりはずっと判断しやすくなりました。

私の場合は、少なくとも「足りない気がする」ではなく、「この使い方だと5時間枠に対して明らかに速いペースで消費している」と言えるようになりました。追加利用するか、作業を分散するか、軽いモデルや別ツールを併用するか——数字があることで選択肢を比較しやすくなります。

## おわりに

Claude Codeの利用制限は、メッセージ数や時間だけで見ると実態がつかみにくいです。今回、トークン量とAPI料金という軸で可視化したことで、「なんとなく足りない」を少し具体的にできました。

[公式ヘルプ上の米国価格・税抜](https://support.claude.com/ja/articles/9266767-team-%E3%83%97%E3%83%A9%E3%83%B3%E3%81%A8%E3%81%AF%E4%BD%95%E3%81%A7%E3%81%99%E3%81%8B)では、TeamプランのPremiumシートは月払いで月額$125です。今回のトークン構成をAPI価格に当てはめると、5時間枠だけで約$62相当になります。価格は地域や時期で変わり得ますし、サブスクリプション枠をAPI価格で評価した参考値にすぎません。サブスクリプション枠は体感では足りなくなることがありますが、API価格に置き換えて見ると、含まれている利用枠自体はかなり大きいと感じました。

一方で、体感としては以前より制限が厳しく感じる場面もあります。高性能モデルを日常的に使うほど、チームとしてのコスト管理は避けにくくなっています。限られた制限枠の中でどれだけ効率的にClaude Codeを使えるかが、ますます大事になりそうです。

利用制限に悩みながらClaude Codeを使っている人にとって、1つの目安になれば嬉しいです。
