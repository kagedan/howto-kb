## 概要

この動画は、Mastra の番組「Agents Hour」で、Ito の CTO である Evan Marshall が、AI 時代のコードレビューと QA の変化を語る回です。主張はかなり明快で、**コード生成そのものより、生成されたコードをどう検証するかが新しいボトルネックになっている**、というものです。モデルが良くなったことで修正案はすぐ出るようになった一方、本当に安全にマージできるかを確かめる工程は、あまり自動化されていません。

そこで Evan が紹介するのが Ito です（註：動画内では「**イ**ートゥ」と発音していました）。Ito は PR に対して実際にコードを動かし、統合テストや E2E テストに近い観点で確認を行い、スクリーンショット、動画、ログ、再現手順まで返してくれるレビュー基盤です。Reptile や CodeRabbit のような「レビューの unit test 層」に近いツールに対して、Ito は **integration / end-to-end 層の観測性と検証性** を担うものとして位置付けられています。

動画の後半では、なぜハーネス engineering が今重要なのか、なぜタスク分解は「手順」より「自然なゴール」で考えるべきなのか、そしてなぜエンタープライズでは AI コード生成の導入効果が 先行期待（hype） ほど大きく見えないのかが語られます。単なるプロダクト紹介ではなく、**AI コード生成の時代に QA とレビューがどう再定義されるべきか**を考える材料になっています。

## 要点

1. **AI コード生成のボトルネックは実装ではなく QA へ移っている**。修正案はすぐ出ても、それが正しいか確かめる時間は依然として重い。
2. **Ito は「実際にコードを動かすコードレビュー」を目指している**。PR に対してスクリーンショット、動画、ログ、再現手順を返し、開発者がローカル確認しなくても変更を把握しやすくする。
3. **既存のレビュー AI が unit test 層だとすれば、Ito は integration / E2E 層を狙っている**。見た目や状態遷移、実環境に近いフローの確認に重心がある。
4. **モデル改善が鈍化するほど、価値の差はハーネスに移る**。Evan は、固定されたモデル能力から追加の成果を引き出すには harness engineering が重要だと述べる。
5. **良いハーネスは「やり方」を細かく縛るのではなく、決定論的な境界とゴールを与える**。これにより、モデルが改良されてもハーネスが劣化しにくくなる。
6. **タスク分解は Step A / B / C より「自然なゴール」で分けるべきだ**。code generation と code review が別 bot になるのも、その方がエージェントにとって自然だからだという。
7. **Ito のデモは、PR レベルでの観測性の次の層を見せていた**。人間がクリックしなくても、PostHog や Excalidraw のような UI を伴うプロダクトの検証結果が動画付きで残る。
8. **PR ごとに差分ベースのテストケースを生成し、manual testing をスケールさせる発想が中核にある**。CI に常設するのではなく、その差分に必要な検証だけを走らせる。
9. **エンタープライズで AI 導入効果が hype ほど出ない理由は、validation layer が未整備だからだ**。高ステークス領域ほど「速く書ける」より「安全に出せる」ことが重要になる。
10. **Ito は Playwright や既存テスト群、dev containers、ESLint などとも接続し、既存の開発環境を踏まえて検証する**。単独のデモツールではなく、既存の開発フローに入り込む前提で設計されている。

## 詳細

### 1. コード生成よりも、コード検証が遅い

最近のシステム開発現場では、モデルの進歩によって「修正案を出す」こと自体はかなり速くなりました。Claude Code や各種コーディングエージェントを使えば、1 分でパッチが出る場面も珍しくありません。

しかし、その次が問題です。本当に動くのか。UI は壊れていないか。副作用はないか。状態遷移で変なことは起きないか。これらを確かめるには、結局プレビュー環境を触る、ローカルに pull する、ログを見る、スクリーンショットを比較する、といった人間の manual work が残りがちです。

Evan は、この構図を「ボトルネックが code development から QA に移った」と述べます。AI がコード生成を加速した結果、遅い部分が前に出てきた、という整理です。

### 2. Ito が狙うのは unit test 層ではなく integration / E2E 層

この文脈で Ito がどこを狙っているかも、動画ではかなり明示的に説明されます。Evan は、既存のコードレビュー AI を「unit test layer」に近いものとして位置付けます。差分を読んでコメントする、明らかな危険点や改善点を指摘する、という役割です。

それに対して Ito が担いたいのは、**コードが実際にどう振る舞うかを統合レベルで確かめる層**です。PR の中に、動画、スクリーンショット、run log、再現手順、stub / mock の情報まで入れて、変更の結果を見せる。これによって、レビューの重心を「テキスト差分の読解」から「変更後の挙動の観測」へ寄せようとしています。

この発想は、AI コード生成が普及するほど効いてきます。人間が 1000 行の diff をレビューするとき、問題は「コードがそれらしく見えるか」ではなく、「本当にプロダクトとして期待通り動くか」だからです。

### 3. 今の差を作るのはモデルよりハーネスである

動画の中盤では、Ito の説明から一歩進んで、いわゆる harness engineering の話に移ります。ここでの Evan の主張は、最近のモデル進歩はゼロではないが、以前より漸進的になっており、だからこそ追加の価値はハーネス側で作るべきだ、というものです。

面白いのは、彼らが内部で自分たちの仕組みを「carriage」と呼ぶことがある点です。単なる 1 本の harness ではなく、複数の agent を束ねた馬車のように考えているわけです。これは比喩に見えて、設計思想もよく表しています。単体の強いモデルに全部やらせるのではなく、境界と役割を持った複数の要素をまとめて運用する。

また、harness engineering は外部ベンチマークを眺めるだけでは足りず、自前の eval set を育て続ける必要があるとも話しています。これは、エージェントシステムの価値がプロダクト固有のワークフローに強く依存するからです。

### 4. エージェントには「手順」より「ゴール」を与える

Evan の話で特に重要だったのは、エージェントに何を与えるべきかという点です。人間はつい「Step A をやって、次に B をやって、最後に C をやって」と指示したくなります。しかし彼は、それよりも **決定論的に検証できる境界とゴール** を与えることが重要だと述べます。

理由は、モデルに細かくやり方を指定し過ぎると、モデルが改良されたときにその制約が足かせになるからです。モデルはもっとよい方法を思い付けるかもしれない。だからハーネス側は、やり方を固定するより「このチェックに到達すれば成功」と定義するほうがよい。

さらに、複数の目標を一度に与えるほどエージェントは弱くなる、とも説明します。「これもやれ、あれもやれ」では優先順位が不明になるからです。したがってタスク分解は、能力別や手順別ではなく「自然なゴール」で分けるべきだ、というのが彼の考えです。code gen bot と code review bot が別物なのは、その目標が本質的に異なるからだ、と説明されます（実際、あるモデルが作ったコードを、同じモデルがレビューしてクリティカルな指摘をしまくる、というのはよくありますよね）。

### 5. デモが見せたのは、PR レベルの観測性の次の層

デモでは、Ito が PostHog のような実アプリケーションを完全に自律的に動かし、その結果を PR コメントとして返す様子が示されます。そこには動画、ログ、再現手順、stub / mock 情報まで含まれます。

このとき重要なのは、単に「テストが通った / 落ちた」ではなく、**何をどう試し、何が見えたかまで PR レベルで残る**ことです。統合テストや E2E レベルの確認は、人間が一番時間を取られやすいのに、最もブラックボックス化しやすい部分でもあります。Ito はそこに観測性を持ち込もうとしているわけです。

Evan は、セキュリティ上の問題や rapid remount が作る stale side effect （UI の再マウントが短時間に起きたときに古い副作用が残ってしまうバグ事例）のように、人間がパッと見では見抜きにくい問題も挙げていました。差分レビューだけでは拾いづらい種類の不具合に、プロダクトに近い層から当てにいくのが Ito の強みだと分かります。

### 6. CI に積み上げるのではなく、PR ごとの manual testing を拡張する

Ito のテストケースは、既存の E2E スイートをそのまま呼ぶのではなく、PR から生成されるのが基本です。ここがかなり重要です。

多くのチームでは、E2E テストは CI に常設され、壊れやすく、保守コストも高いものとして扱われています。Ito の発想はそれと少し違い、**人間が普段やっている差分確認を PR ごとにスケールさせる**ことにあります。だから 1 PR ごとに 30〜50 ケースを追加しても、それがそのまま CI の常設コストにはならない。

この設計は、AI コード生成時代にかなり相性が良さそうです。diff ごとに確認したい点は変わるので、固定の E2E スイートだけでは追い切れません。差分に応じて test plan を作り、その PR で必要な検証をだけ深く行うほうが合理的です。

### 7. 難所は test case alignment にある

もっとも、テストケースを自動生成するなら「そのテストケース自体が妥当か」という問題が出てきます。Evan も、ここを大きな課題として認めています。テストケースを最後まで実行できたかどうかは比較的測りやすい一方、ユーザーが本当に見たい検証と一致しているかは曖昧です。

そのため Ito では、PR 上で「このテストはいらない」「こちらを見てほしい」といったフィードバックを返せるようにして、alignment を改善していく想定になっています。これは、生成 AI の評価でよく出る「成功率」だけでは測れない領域であり、実務ではむしろこちらのほうが難しいことをよく示しています。

### 8. エンタープライズで 先行期待（hype） ほど速くならない理由

動画の終盤で印象的なのは、なぜ大企業が X 上の hype ほど AI の効果を享受できていないのか、という説明です。Evan は、その理由を validation layer の欠如に求めます。

小さな実験なら「速く書ける」だけで十分です。しかしヘルスケア、フィンテック、巨大 SaaS のように、何かが壊れたときの影響が大きい環境では、現場は簡単に不安を捨てられません。C-suite が導入を急いでも、実際に事故を防ぐ仕組みがなければ、エンジニアは全面的には乗れない。

ここで Ito が狙っているのは、「生成速度そのもの」ではなく「安全に速く出せる状態」を作ることです。この視点は、AI コード生成の議論を hype から実運用へ引き戻すものとして重要だと思います。

### 9. Playwright、dev containers、開発者ツールとの接続

最後の Q&A では、既存の Playwright ベースの大きな E2E コレクションを持つチームでも使えるのか、という質問が出ます。Evan の回答から分かるのは、Ito が既存資産を無視してゼロから置き換えるツールではないということです。

オンボーディング時にリポジトリの既存構成を調べ、必要なら dev containers を使って本物のサービスを起動する。Docker Compose や Dockerfile を含む実環境に近いセットアップを前提にしているので、単なる headless browser デモにとどまりません。

さらに、ESLint 設定変更のようなケースでは、実際にコマンドを走らせて型エラーや設定崩れまで確認できると語られます。つまり Ito は、ブラウザ自動操作だけの製品ではなく、コードレビューを「実行と観測の場」に変える基盤として作られているわけです。
