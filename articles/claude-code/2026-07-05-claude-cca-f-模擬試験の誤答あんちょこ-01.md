# Claude CCA-F 模擬試験の誤答あんちょこ

Claude Certified Architect Foundations、略して CCA-F の模擬試験で間違えた問題を、復習用に整理した自分用あんちょこです。

目的は、正答を丸暗記することではありません。

**同じ型の誤答を二度しないこと**です。

---

## まず覚えるべき判断軸

### 1. プロンプトで「誘導」するのか、APIで「強制」するのか

ここを間違えると連続失点します。

* プロンプト
* XMLタグ
* Few-shot
* temperature

これらは、モデルの挙動を**誘導**するものです。

一方で、次は実行や形式を**強制**する設計です。

* `tool_choice`
* JSON Schema
* `required`
* API設定
* ツール定義

判断ルール：

```
確率を上げる → プロンプト / XML / Few-shot
実行を保証する → tool_choice / schema / API設定
```

---

### 2. 巨大コンテキストは「全部入れる」ではなく「必要部分だけ渡す」

長文PDFや契約書を扱うとき、やってはいけないのは全文を盲目的に詰め込むことです。

* 60ページPDF：本文を先、質問を最後
* 500ページ契約書：RAGで必要チャンクだけ取得
* 長い会話：重要情報を最初と最後に置き、古い履歴は要約

判断ルール：

```
長文 = 配置を設計する
巨大文書 = RAGで検索して必要部分だけ渡す
会話が長い = 要約して重要情報を末尾に戻す
```

---

MCPの分類は単純です。

```
MCP Resource = 読むデータ
MCP Tool     = 実行する処理
MCP Prompt   = 再利用する指示テンプレート
```

例：

* 社員ハンドブックPDF：Resource
* 社員住所の更新API：Tool
* 人事問い合わせ用の定型プロンプト：Prompt

モデル重みをMCP Resourceでホストする、という発想は誤りです。

---

### 4. Agentic Architectureは「誰が何をするか」で分ける

エージェント設計では、名前の雰囲気ではなく責務で判断します。

```
階層型 = オーケストレーターが順序制御・集約
P2P = エージェント同士が交渉・交換
Hub & Spoke = ハブが専門エージェントへ委任
Reflection = 生成結果を別エージェントが批評・検証
Semantic Routing = 意図分類して適切な処理先へ振る
```

---

### 5. temperatureは万能ではない

過去の誤答では、何度もtemperatureに引っ張られています。

temperatureは主に出力のばらつきに関係します。  
次の用途には不十分です。

* ブランドトーンを正確に合わせる
* 逐語抽出を保証する
* JSONスキーマ準拠を保証する
* ツール実行を保証する

判断ルール：

```
文体制御 → 実例 + 明示的トーン指定
逐語抽出 → 要約・言い換え・変更を禁止
構造化出力の保証 → tool_choice
ツール実行の保証 → tool_choice
```

## Q6：JSON/XMLなどの出力形式をプロンプトで強制する場合

### 正答

* JSON/XMLのみを出力させ、会話フィラーを禁止する
* 正確なターゲット形式の例をFew-shotで示す

### 誤答原因

構造化出力を狙うときに、Markdownラッパーや中間形式に引っ張られる。

ただし、生JSONが必要な場合、Markdownで包むのは逆効果です。

### 覚える

```
プロンプトで構造化出力を狙うなら、
フィラー禁止 + 正確な例示
```

## Q10：API 429 レート制限への設計対応

### 正答

指数バックオフ、ジッター、サーキットブレーカーを組み合わせる。

### 誤答

`bypass_ratelimit=true` のような存在しない回避策を選ぶ。

### 誤答原因

API制限を設計で扱うべき場面で、抜け道を探している。

### 覚える


## Q15：Few-shot例の最適な構成

### 正答

* XMLタグでFew-shotセクションを明確に区切る
* 例の入力/出力パターンを本番タスクと完全一致させる

### 誤答原因

「悪い例を最後に置く」に引っ張られる。

Few-shotで重要なのは、悪い例の強調ではなく、正しいI/Oパターンの提示です。

### 覚える

```
Few-shot = XMLで区切る + 本番I/Oと一致
```

## Q18：階層型エージェントを採用すべき場面

### 正答

オーケストレーターが実行順序を指示し、分離されたサブエージェントの結果を集約する厳格なワークフロー。

### 誤答

エージェント同士が交渉・リソース交換するタスク。

### 誤答原因

階層型とピアツーピア型を混同している。

### 覚える

```
階層型 = 上位司令塔
P2P = エージェント同士の交渉
```

---

## Q19：Reflectionパターン

### 正答

評価用サブエージェントが、メイン生成結果をユーザー送信前に批評・検証する。

### 誤答

ユーザーの性格を反映する。

### 誤答原因

Reflectionを会話トーンのミラーリングと誤解している。

### 覚える

```
Reflection = 生成 → 批評 → 修正または採用
```

---

## Q28：逐語的なテキスト抽出で言い換えを防ぐ方法

### 正答

「要約・言い換え・単語変更を禁止」と明示的な否定制約を書く。

### 誤答

temperatureを1.0にする。

### 誤答原因

厳密性をパラメータで解決しようとしている。

temperatureを上げると、むしろ出力はばらつきます。

### 覚える

```
逐語抽出 = 変更するな、と明示する
temperatureではない
```

---

## Q32：JSONスキーマ準拠を最も堅牢に保証する方法

### 正答

定義済みフォーマットツールを `tool_choice` で強制する。

### 誤答

XML生成後に正規表現でJSONへ変換する。

### 誤答原因

構造化出力を後処理で無理に直そうとしている。

### 覚える

```
JSONスキーマを保証したい = tool_choiceでツール強制
```

補足：

```
プロンプト指示 = 確率を上げる
Few-shot = 確率をさらに上げる
tool_choice = スキーマ準拠を強制する
```

## Q40：Hub & Spoke型マルチエージェントの利点

### 正答

中央オーケストレーターが専門タスクをサブエージェントに委任し、各エージェントのコンテキストを小さく保てる。

### 誤答

状態や会話履歴管理が不要になる。

### 誤答原因

Hub & Spokeは状態管理を不要にしません。  
むしろ、ハブが状態や履歴を管理します。

### 覚える

```
Hub & Spoke = 責務分離 + コンテキスト削減 + 集中維持
```

---

## Q44：ツール誤用を減らす設計

### 正答

* 曖昧なケースのFew-shotルーティング例を示す
* ツール名を意味的に明確に差別化する

### 誤答原因

出力形式や巨大な汎用ツールで解決しようとしている。

### 覚える

```
ツール選択精度 = 明確なツール名 + エッジケース例
```

NG例：

巨大な汎用ツールは、かえってモデルを混乱させます。

## Q46：JSON Schemaでoptional項目を表す方法

### 正答

optional項目を `required` 配列から外す。

### 誤答

descriptionに「Optional」と書きつつ、`required` に残す。

### 誤答原因

自然言語説明とJSON Schema仕様を混同している。

### 覚える

例：

```
{
  "type": "object",
  "properties": {
    "query": { "type": "string" },
    "date_range": { "type": "string" }
  },
  "required": ["query"]
}
```

## Q51：長文会話で中盤の情報を忘れる問題

### 正答

* 重要情報をプロンプトの最初と最後に置く
* 会話履歴を頻繁に要約し、古いターンを要約に置き換える

### 誤答原因

lost in the middleを、単なるコンテキスト長の問題と見ている。

実際には、情報の配置位置も重要です。

### 覚える

```
Lost in the middle対策 = 端に置く + 要約する
```

## Q55：Claude Codeのカスタムスラッシュコマンド

`CLAUDE.md` では複雑なロジックを書かず、既存の認証済み標準CLIスクリプトや `make` ターゲットを参照する。

### 誤答

C++動的ライブラリが必要。

### 誤答原因

Claude Codeのワークフロー設定を、拡張機能開発のように誤認している。

### 覚える

```

ロジック本体を書く場所ではない
```

## Q60：Semantic Routing

### 正答

ユーザープロンプトを受け取り、意図を分類し、生成モデルを呼ぶ前に適切なテンプレートや専門エージェントへ振り分ける。

### 誤答

RAGのベクトルDBを置き換える。

## Q8：MCPコンポーネントの正しいマッピング

### 正答

* 従業員ハンドブックPDFをMCP Resourceとして公開する
* 住所更新のようなDB書き込み処理をMCP Toolとして定義する

### 誤答原因

Resourceを「何かを置く場所」と雑に理解している。

MCP Resourceはモデルや実行環境をホストする場所ではありません。

### 覚える

```
MCP Resource = 読み取り専用データ
MCP Tool = 実行・副作用あり
MCP Prompt = 再利用可能なプロンプトテンプレート
```

## Q22：60ページPDFを効率的に処理するプロンプト構造

### 正答

大量のPDF本文を先に置き、`<document>` などのXMLタグで囲み、具体的な質問や指示を最後に置く。

### 誤答

質問を最初に置き、その後にPDF本文を置く。

### 誤答原因

人間向け文書構成と、LLM向けコンテキスト配置を混同している。

### 覚える

推奨構造：

```
<document>
大量のPDF本文
</document>

<task>
上記文書を根拠に、次の質問に答えてください。
質問：...
出力形式：...
</task>
```

## Q52：XMLタグ認識のアーキテクチャ上の意味

### 正答

* 命令とユーザーデータを分離し、プロンプトインジェクションや混乱を減らす
* ClaudeはXML的な構造を認識しやすく、文書構造・論理階層の解析に強い

### 誤答原因

XMLタグの効用を、構造理解ではなくシステム機能拡張と誤認している。

### 覚える

```
XMLタグ = 境界分離 + 構造理解
XMLタグ ≠ 認証 / HTML変換 / トークン制限回避
```

# 模擬試験 #6

## Q57：5つのサブエージェントから必ず1つを選ばせるルーティング

### 正答

API構成で `tool_choice: {"type": "any"}` を使う。

```
{
  "tool_choice": {
    "type": "any"
  }
}
```

### 誤答

`<do_not_talk>` XML区切り文字を使う。

### 誤答原因

XMLタグによる誘導と、APIパラメータによる強制を混同している。

### 覚える

```
ツールを必ず1つ選ばせる = tool_choice: {"type": "any"}
```

整理：

```
どれか1つのツールを必ず使わせる → tool_choice: {"type": "any"}
特定の1ツールを必ず使わせる → tool_choice: {"type": "tool", "name": "..."}
ツール使用を任せる → tool_choice: {"type": "auto"}
ツールを使わせない → tool_choice: {"type": "none"}
```

# 再発しやすい誤答パターン

## 1. パラメータ万能思考

次をtemperatureで解決しようとしない。

* トーン一致
* 逐語抽出
* スキーマ準拠
* ツール実行保証

必要なのは、明示指示、例示、スキーマ、tool\_choiceです。

---

## 2. プロンプト誘導とAPI強制の混同

XMLタグやFew-shotは有効です。  
ただし、保証ではありません。

```
保証という言葉が出たら、まずtool_choice / schema / API設定を疑う
```

---

## 3. 巨大情報をモデル能力で解こうとする

巨大文書や長文会話は、モデルの気合いで解く問題ではありません。

必要なのは設計です。

* チャンク化
* 検索
* 要約
* 重要情報の配置
* RAG
* コンテキスト管理

---

## 4. Agentic Architectureを雰囲気で読む

「エージェントが複数いる」だけでは不十分です。

見るべきは責務です。

```
分類するのか
実行するのか
集約するのか
評価するのか
交渉するのか
```

この責務分離で読めば、階層型、P2P、Hub & Spoke、Reflection、Semantic Routingを区別できます。

---

# 最終結論

この範囲で最も重要なのは、次の1行です。

```
誘導はプロンプト、保証はtool_choice/API設計。
```

ここを固定すれば、Structured Output、Tool Design、MCP、Agentic Architectureの失点を大きく減らせます。
