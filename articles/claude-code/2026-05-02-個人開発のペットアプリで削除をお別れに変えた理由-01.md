Claude CodeやCode Xなどコードエージェントの台頭によって、プロダクトは一晩で作れる時代になった。

「ペットの健康記録アプリを作って」と AI に頼めば、ものの５分でそれらしいものが返ってくる。  
もちろん完璧ではないが、少し手直しさえすればリリースすることができる。  
数日も使わずに、App Store には何かが並ぶ。  
それ自体は素晴らしい。

でも、AI には取れないものが一つある。**責任**だ。

私はいま、ペットの毎日を記録するアプリ「Petla」を個人で作っている。設計と実装は AI と並走している。コードは Claude に書いてもらう、レビューも Claude にやってもらう。それでも、何を AI に頼まないか。そこに、自分の責任の輪郭がある。

## ある日、自分のアプリに違和感を持った

開発中、自分が作った設定画面を眺めていて手が止まった。

ペットのプロフィール画面に  
「ペットを削除する」ボタンが、当たり前のように並んでいた。

もちろんアプリの機能としてはなんら問題のない。  
**しかし、何かが明らかに間違っているのだ。**

ペットアプリで「ペットを削除する」時はどんな時だろうか？

誤って追加してしまったケースを除いて  
おそらくペットが亡くなってしまったケースが大半だろう。

自分の愛するペットが亡くなり、別れを受け入れる時に  
「ペットを削除する」というテキストラベルで済ませてしまうことは  
本当にペットアプリとして適切なのだろうか？  
ペットを亡くした飼い主がアプリを開いて「ペットを削除する」というダイアログを表示させることは私のプロダクトではとてもじゃないができない。

## 「お別れ」というモデル設計

![](https://static.zenn.studio/user-upload/01d7e0121472-20260502.png)

機能の名前を「削除」から「お別れ」に変えた。

加えて「お別れ」した後もペットのデータは残り  
アルバム形式でお別れした自分の愛するペットとの思い出を振り返れるようにした。

データモデルから設計する必要があった。

Pet.swift

```
@Model
final class Pet {
    var name: String
    var birthDate: Date

    /// nil の間は「一緒に過ごしている」状態。
    /// 値が入った時点でお別れ済み。物理削除はしない。
    var passedAwayAt: Date?
}

extension Pet {
    var hasPassedAway: Bool { passedAwayAt != nil }

    /// 一緒に過ごした日数。お別れ後も計算可能。
    var daysSpentTogether: Int {
        let endDate = (passedAwayAt ?? Date()).startOfDay
        let start = birthDate.startOfDay
        return Calendar.current.dateComponents(
            [.day], from: start, to: endDate
        ).day ?? 0
    }
}
```

`passedAwayAt` というたった一つの nullable な日付フィールドが、アプリ全体の人格を切り替えるスイッチになる。

データは消えない。記録も写真も、すべての日々が残る。ただ「もう一緒には過ごしていない」という事実が、静かに加わる。

## 文言ひとつにも責任がある

![](https://static.zenn.studio/user-upload/1d10fd3b64c2-20260502.png)

UI の文言を全部見直した。

| Before | After |
| --- | --- |
| ペットを削除する | {ペット名}とのお別れを記録する |
| 本当に削除しますか？ | {ペット名}と過ごした日々を、想い出として残します |
| 削除しました | これからは、想い出の中で一緒です |

「削除」という二文字は、AI に頼めば一秒で出てくる。

しかし、そこに愛と責任はない。

AI が生成する文言は中央値を返す。どこかで見たような単語が並ぶ。プロダクトの輪郭は、その中央値から **どう離れるか** で決まる。

これに答えられるのは、人間だけだ。

## 周辺機能も連動して変える

![](https://static.zenn.studio/user-upload/f81321c9c132-20260502.png)

「削除を『お別れ』に書き換えた」だけでは、責任を取ったことにならない。お別れ後の体験は、こう変わる。

* 通知は止まる（「{ペット名}の体重を記録しましょう」は届かない）
* 誕生日のお祝いも止まる
* ホーム画面のトーンが変わる（明るい祝祭から、静かで柔らかい色彩へ）
* でも、過去の記録・写真・思い出は、絶対に消さない
* 「メモリアル」セクションが用意され、想い出に再訪したい時にだけ訪れられる

基本姿勢は **「触りたくなければ触らなくていい」**。

ユーザーの感情を先回りしすぎない。「悲しいですよね」と話しかけない。「{ペット名}の想い出はここにあります」と置いておくだけ。

それが、自分が考える礼儀だ。

## AI には責任が持てない

![](https://static.zenn.studio/user-upload/a5c2fd196f59-20260502.png)

AI に「これでいいですか」と聞いて「いいです」と返ってきても、それは責任の移転にはならない。

判断したのは自分だ。出したのも自分だ。  
ユーザーの前に立つのも、自分だ。

AI が当たり前になった時代、プロダクトの輪郭は「何を作るか」より「何を作らないか」で決まる。そしてその判断に、最後まで名前を添えられるか。

責任を持てないプロダクトは、作らない。  
責任を持てる範囲のプロダクトを、丁寧に作る。  
それだけだ。
