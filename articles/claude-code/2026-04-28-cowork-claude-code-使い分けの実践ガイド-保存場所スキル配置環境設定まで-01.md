## はじめに

Claudeを「ChatGPTの代わりに使ってみる」段階から、「業務に組み込んで毎日使い倒す」段階に来ると、必ず直面する壁がある。

**「これってCoworkでやるべき？それともClaude Code？」**

そして次に来るのが、

という配置・設定の悩みだ。

私自身、Claude公式アプリのCoworkモードと、CLIツールのClaude Codeを両方常用している。Chrome拡張「N Graph」（Notion × Google Analytics連携ツール）の開発・運用を1人で回す上で、両者の役割分担を明確にしないと作業効率が一気に落ちることを身をもって学んだ。

この記事では、両ツールを実際に使い込んだ立場から、**保存場所・スキル配置・環境設定**の3点に絞って、現場の使い分けを共有する。同じ悩みを抱える人の試行錯誤を3時間ほど短縮できれば本望だ。

---

## そもそもCoworkとClaude Codeは何者か

最初に、両者の立ち位置を1行で整理する。

中身のClaudeモデル（Sonnet/Opus/Haiku）は同じだ。違うのは**インターフェース**と、**Claudeが触れる範囲**である。

Coworkは「ユーザーが選択したフォルダ」と一時的なサンドボックスを行き来する。Claude Codeは「起動したディレクトリ配下すべて」を作業領域とする。この違いが、保存場所の使い分けを決定づける。

一言で結論を出すなら、

> **Git管理するもの＝Claude Code、Git管理しないもの＝Cowork**

これだけ覚えておけば、初見でも7割は迷わない。残り3割は「両方使う場面」で、ここから先で詳しく扱う。

## 保存場所とコミット先の使い分け

### Coworkの保存先

Coworkはユーザーがフォルダを「マウント」して使う。macOSなら標準は ~/Documents/Claude/Projects/<プロジェクト名> 配下だ。Cowork内で作ったWord・Excel・PowerPoint・Markdown・PDFは、すべてここに書き出される。

このフォルダの特徴は、**Gitリポジトリではないこと**だ。バージョン管理されないので、上書き・削除に弱い。逆に言えば、軽い気持ちでドラフトを量産できる。

向いているのは以下のような成果物だ。

* 顧客向け資料、提案書、プレゼン
* ブログ記事・note・SNS投稿の下書き
* リサーチレポート、市場調査
* メールテンプレート、議事録、ブリーフィング
* 一過性の集計、Excelで作る簡易ダッシュボード

要するに、**Git管理する価値がない（or できない）作業ファイル**である。

### Claude Codeの保存先

一方、Claude Codeはターミナルで起動した位置がそのまま作業ディレクトリになる。普段はVSCodeで開いているプロジェクトリポジトリと同じ場所だ。

Claude Codeで触れるべきは、**Gitで管理されるすべて**。

* ソースコード（フロント・バック・スクリプト全部）
* 設定ファイル（package.json、tsconfig、.env.example など）
* インフラ定義（Dockerfile、Terraform、CI/CDワークフロー）
* 開発者向けドキュメント（README、ADR、CHANGELOG）
* テストコード、フィクスチャ

コミット先も基本ルールは普段のGit運用と同じ。featureブランチを切って、Claude Codeに作業させ、レビューしてmainにマージする。Claude Codeはブランチ操作・PR作成までこなすので、**Claudeに「ここまでやったらPR出して」と頼める**のは小さなことに見えて大きい。

### 「両方使う」プロジェクトの整理術

実プロジェクトでは、両方使うケースが大半だ。私のN Graph運用を例に取る。

* **コード本体**（Chrome拡張のTypeScript、ビルド設定、APIクライアント）→ Claude Code（VSCodeで開いているリポジトリ）
* **プロモ素材**（note記事、X投稿原稿、ユーザーへのメール文、スクリーンショット指示書）→ Cowork（~/Documents/Claude/Projects/N Graph運営）
* **両方触るもの**（リリースノート、ユーザー向けドキュメント）→ どちらでも書けるが、最終的にはGit管理側へ寄せる

迷ったときの判断基準は1つ、**「3ヶ月後にこのファイルを参照する可能性があるか」**。あるならClaude Code（Git）側に置く。なければCowork側でいい。

## アプリ・スキルはどこに置くべきか

ここが一番混乱しやすい領域だ。Claude周辺の用語を先に整理する。

* **MCP（Model Context Protocol）**：外部サービス連携の規格。Notion、GitHub、Stripe、Google系などをClaudeから呼び出せるようにする「コネクタ」の総称。
* **スキル（Skill）**：特定タスクのためのプロンプト＋スクリプトを束ねたフォルダ。SKILL.mdが中核。「PDF作成」「Excel編集」など、**Claudeに作業手順を教える定型書**。
* **エージェント（Agent / subagent）**：特定領域の専門家として振る舞うAIの設定。役割・口調・参照スキルを定義する。
* **プラグイン**：複数のスキル・MCP・エージェントを束ねた配布単位。.pluginファイルとしてまとめて配布できる。
* **マーケットプレイス**：プラグイン群を配布するGitHubリポジトリ。/installコマンドで他人のマシンに配れる。

### 配置場所の選択肢

スキルやプラグインを置く場所は、用途に応じて3パターンある。

**1. ローカル個人利用**：~/.claude/skills/<スキル名>/SKILL.md に直置きする。最速で動く。1人で使うなら十分。

**2. GitHub Private/Public でマーケットプレイス化**：プラグインの形に整え、リポジトリで配布する。/plugin install で他人もインストールできる。**チーム共有・複数端末利用にはこれ一択**。

**3. プロジェクト同梱**：プロジェクトリポジトリ直下の .claude/ に置く。プロジェクトを開いた人全員が使える。**プロジェクト固有の業務フローを共有したい時**に便利。

### Cowork向け？Claude Code向け？

**結論：両対応で書くのが基本。**

スキルの中身は単なるMarkdown＋シェルスクリプト＋Pythonスクリプトの組み合わせなので、CoworkでもClaude Codeでも同じものが動く。実際、私が運用する自作プラグイン群は、両方の環境で同じスキルを呼び出している。

例外は2つだけ。

迷ったら両対応で書き、SKILL.mdの冒頭で「主にClaude Codeで使う」「主にCoworkで使う」と注記しておけば運用上問題ない。

### チーム配布・複数端末運用のおすすめ

スキルが10個を超えてきたら、GitHub privateリポジトリでまとめて管理することを強く勧める。

私の場合、PenClaw Marketplace（GitHub private）に10個のエージェントとスキルを集約し、Claude Code/Cowork両方からインストールできるようにしている。新しいマシンを買い替えた時も、/plugin install penclaw 1コマンドで環境を復元できる。**dotfiles感覚でClaudeの作業環境ごと持ち運ぶ**イメージだ。

## Claude Code環境設定の勘所

Claude Codeの環境設定は、ファイルが分散していて全体像が掴みにくい。最小限の3ファイルから始める。

### 1. CLAUDE.md（プロジェクトルート）

プロジェクトのルートに置くMarkdownファイル。**Claudeが毎回起動時に読む永続メモリ**だ。ここに書くべきは以下の通り。

* プロジェクトの目的・哲学（なぜ作っているか）
* 技術スタック・主要な依存関係
* ディレクトリ構成の説明
* コーディング規約・命名規則
* ビルド・テスト・デプロイのコマンド
* 触ってはいけない領域、注意すべき領域

ここをサボると、Claudeは毎回ゼロから探索する羽目になる。**最初の30分でCLAUDE.mdを書くと、その後の作業が体感3倍速になる**。これは誇張ではない。

### 2. ~/.claude/settings.json（個人グローバル設定）

ユーザー単位の設定。MCP接続、デフォルトのモデル、テーマ、パーミッション制御などをここに書く。複数プロジェクト共通で使いたい設定はここに置く。

注意したいのは**MCP接続の取捨選択**だ。MCPを盛り過ぎるとコンテキストを圧迫し、Claudeの応答品質が下がる。**最初はGitHubとファイルシステムだけ**から始め、必要になったら追加する方針が安全だ。私の構成は、開発時はGitHub＋Notion＋Stripe、運営時はGoogle系＋Slack＋Chatworkという形で、**プロジェクトごとにMCP構成を切り替えている**。

### 3. .claude/ ディレクトリ（プロジェクト固有設定）

プロジェクトリポジトリ内に置く。プロジェクト固有のスラッシュコマンド、フック、サブエージェントをここに定義する。チームで共有したい設定はここに入れ、Gitにコミットする。

特に有用なのが\*\*Hooks（フック）\*\*だ。コミット前にlint・テストを走らせる、PRを出す前にCHANGELOGの更新を確認する、といった「忘れがちな手順」を自動化できる。Claude任せにせず、**人間の手作業をフックで肩代わりさせる**発想だ。

### おすすめ初期設定3点

立ち上げ時、最低これだけはやっておきたい。

1. **プロジェクトルートにCLAUDE.mdを必ず置く** ― 短くてもいい、書かないよりマシ。
2. **~/.claude/skills/ をGit管理する** ― dotfilesリポジトリに含めると、新マシンでも30秒で復元できる。
3. **MCPは最低限から** ― 「とりあえず全部入れる」は罠。コンテキストが食われ、応答が遅くなる。

## 実例：N Graphの開発・運用で両方をどう使っているか

私が運営するN Graphは、Notionにアクセスしながら Google Analytics（GA4）の数字を直接Notion画面に表示できるChrome拡張機能だ。Pro版は月額3ドル。1人で開発・サポート・マーケティングまで回している。

* **Claude Code（VSCodeで開いた N Graph リポジトリ）**：拡張機能の本体コード、ビルド・リリース作業、Chrome Web Storeへの提出ファイル準備、バグ修正、機能追加。すべてGitで管理され、Claude Codeがブランチを切ってPRを出す。
* **Cowork（~/Documents/Claude/Projects/NGraph運営）**：プロモーション素材、note記事の下書き、X投稿の原稿、Pro版ユーザーへの返信文面、日次のGA4・Stripe数字レポート。Git管理しない、流れていくコンテンツ。
* **両方を繋ぐPenClaw Marketplace**：自作スキル群を入れたGitHub privateリポジトリ。たとえば「N Graphの日次運営レポートを作る」スキルは、Cowork側で呼ばれて毎朝のブリーフィングに使っているが、中身は同じスキルがClaude Codeでも動く。

この役割分担の結果、CLI操作が苦手な日常タスク（メール返信、note執筆、画像差し替え指示）はCoworkで気軽に進み、コードに触る作業はClaude Codeでガッツリ片付く。\*\*「迷う時間がほぼゼロ」\*\*になったのが最大の収穫だ。

## まとめ

最後に、判断に迷った時に立ち戻れる原則を3行で。

* **Git管理するもの＝Claude Code、それ以外＝Cowork**
* **スキルは両対応で書き、配布はGitHubプライベートリポジトリで**
* **Claude Codeは CLAUDE.md から始め、MCPは最低限から**

Claude周辺は変化が速く、半年後にはまた別のベストプラクティスが出ているかもしれない。それでも\*\*「保存場所を分ける」「設定を記述する」「配布の仕組みを持つ」\*\*という骨格は変わらないはずだ。
