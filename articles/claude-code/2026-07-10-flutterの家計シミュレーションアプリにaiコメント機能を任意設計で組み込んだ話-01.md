## 背景・作ろうと思った理由

最近Flutterアプリを作り続けています。今回作ったのは、収入・支出・貯蓄額を入力すると将来のキャッシュフローをグラフで見せてくれる、いわゆる「ライフプラン簡易診断」アプリです。

私はFP2級を持っているのですが、資格の知識をそのままコードに落とし込もうとすると、ふと立ち止まる瞬間がありました。「この試算結果に一言アドバイスを添えたら親切だな」と思った直後に、「それって個別具体的な助言になってしまわないか？」という懸念が頭をよぎったのです。

FP2級の試験でも、税理士法との関係で「一般的な説明」と「個別具体的な助言」の境界線はたびたび問われます。仮の事例に基づく計算や一般的な制度説明は問題ない一方、個別の顧客の状況に踏み込んだ助言は資格外の行為になり得る、という整理です。

今回は完全に自分専用ツールで対顧客サービスではないので法的リスクは低いのですが、「後から人に見せる」「将来BtoC化する」可能性も考えると、最初から誠実な設計にしておきたいと思い、AIコメント機能そのものを「任意・総評のみ・免責必須」という設計に落とし込むことにしました。この記事はその設計判断の記録です。

## 技術選定とその理由

* **Flutter + Riverpod + sqflite**: 他の自作アプリ（タロットアプリ）と同じ構成を踏襲。データは端末内完結が前提。
* **Claude API**: AIコメント生成に使用。既存アプリでも使っていた構成をそのまま流用。
* **flutter\_secure\_storage**: APIキーの保管用。sqfliteの平文テーブルに入れないと決めていたので、ここは最初から分離設計にしました。

計算ロジック（`CashflowSimulator`）とAI連携（`AiAdvisorService`）を明確にレイヤー分離したのが、今回一番意識した点です。

```
domain/usecases/cashflow_simulator.dart   ← 数値計算（決定論的、AI不使用）
                ↓ 結果を渡す
data/services/ai_advisor_service.dart      ← Claude APIへの問い合わせ（任意機能）
```

AIには「計算」をさせず、「計算結果の解釈」だけをさせる。この分離を最初に決めたことで、後述する免責設計もシンプルになりました。

## 実装の流れ

Claude Codeに渡す指示書（`life_plan_app_spec.md`）を事前にClaude.aiとの設計相談で作成し、それをプロジェクトルートに置いて章単位で実装を依頼する、というのがいつものワークフローです。

指示書には最初から「AIコメントの粒度は総評1本のみ」「APIはトグルでON/OFFできる」という制約を明記しておきました。実装時に判断がぶれないよう、プロンプトの厳守事項まで指示書に書き込んでおいたのが効きました。

```
String _buildPrompt(SimulationResult result) {
  return '''
以下はユーザーの家計キャッシュフロー試算の要約です。
この結果全体を俯瞰し、一般的な傾向についての総評を3〜4文で作成してください。

【厳守事項】
- 特定の金融商品・保険商品・投資先を推奨しない
- 断定を避け、「〜という視点もあります」等の表現を使う
- 税務・法律の個別具体的な助言はしない
- 不安を煽らない

【試算結果の要約】
${result.toPromptSummary()}
''';
}
```

ポイントは`toPromptSummary()`で渡す情報を要約統計（最終年の資産額、貯蓄がマイナスに転じる年など）に絞り込み、個別の収入・支出項目の明細を渡さないようにしたことです。渡す情報の粒度を制限すること自体が、AIの出力を総評レベルに留めるガードレールになります。

## つまずいたポイントと解決法

AI連携の実装自体は素直に進んだのですが、その手前の`CashflowSimulator`（キャッシュフロー計算ロジック）で想定より時間を使いました。

躓いた原因は、収入・支出項目に「開始年・終了年・成長率」という3つの可変要素を持たせたことで、年次ループの中で「その年に有効な項目かどうか」の判定と「何年目の成長率を掛けるか」の基準点がずれるバグを作り込んでしまったことです。具体的には、`start_year`からの経過年数ではなく、シミュレーション開始年からの経過年数で成長率を計算してしまい、収入項目を途中で追加すると数値が不自然に跳ねる、という状態になりました。

これは単体テストを先に書いていたおかげで早期に発覚しました。「収入・支出のみの単純な右肩推移」「途中で収入項目が終了するケース」など、指示書に書いておいたテストケースを実装前に用意していたのが功を奏した形です。結果的に、成長率の基準点を項目ごとの`start_year`に統一することで解決しました。

この経験があったからこそ、「計算ロジックは決定論的に保ち、AIは計算結果の解釈だけを担当する」という役割分担の重要性を再確認しました。もし計算自体をAIにやらせていたら、こういうバグの切り分けはもっと難しくなっていたはずです。

## 免責設計について

AI連携部分で一番時間をかけたのは、コードよりもむしろ「免責をどこに、どう出すか」の設計でした。1箇所にまとめて終わりにするのではなく、接触点ごとに段階的に表示する構成にしています。

1. **初回起動時**: 免責事項全文への明示同意がないと先に進めないモーダル
2. **結果画面**: 常時表示される短縮版の免責バナー
3. **AIコメント直下**: 「AIによる一般的なコメントです。専門家の助言に代わるものではありません」という注記を隣接表示
4. **設定画面**: 免責事項全文をいつでも閲覧できるページ

特に3番目は、AIコメントを結果画面の他の情報（数値・グラフ）と視覚的に区別することも兼ねています。AIラベル（アイコン + 「AIコメント」というテキスト）を明示することで、人間の専門家の見解と混同されないようにしました。

```
class AiCommentSection extends StatelessWidget {
  Widget build(BuildContext context) {
    return Card(
      child: Column(
        crossAxisAlignment: CrossAxisAlignment.start,
        children: [
          Row(children: [
            Icon(Icons.auto_awesome, size: 16),
            Text('AIコメント', style: labelStyle),
          ]),
          Text(comment),
          const SizedBox(height: 8),
          Text(
            DisclaimerText.aiCommentDisclaimer,
            style: captionStyle,
          ),
        ],
      ),
    );
  }
}
```

また、AI機能自体をデフォルトOFFにし、設定画面でトグルをONにした時だけAPIキー入力欄が現れる設計にしました。OFF時は`AiAdvisorService`への通信そのものを発生させない、という制約も指示書に明記しています。「機能を使わない自由」を最初から用意しておくことで、免責の負担も利用者側の選択に委ねられる形になったと思っています。

## 学び・次にやること

計算ロジックとAI解釈レイヤーを分離する設計は、バグの切り分けのしやすさと免責設計のシンプルさの両方に効いてくることを実感しました。「AIに何をさせて、何をさせないか」を最初に決めておくと、後工程の設計判断（今回で言えば免責の出し方）が芋づる式に楽になります。

次は、このアプリのライフイベントテンプレート（結婚・住宅購入・教育資金など）をもう少し充実させたいと思っています。
