長くClaude Codeを使っていると、こんな感覚に襲われる。

朝、立ち上げたばかりの頃は反応が早い。指示も的確に通る。でも3時間目、4時間目になると——同じ指示を繰り返させられる。前に決めたことを忘れる。返答が遅い。

これはClaudeが疲れているわけではなく、**コンテキストウィンドウが埋まり始めている**サインだ。

最終章のこの章では、コンテキストの仕組みとコスト最適化を扱う。地味だが、ここを押さえると Claude Code との付き合い方が一段安定する。

## コンテキストウィンドウとは「机の広さ」

Claudeには「一度に覚えていられる情報量」の上限がある。これがコンテキストウィンドウだ。本のページ数や、作業机の広さに例えるとイメージしやすい。

Sonnetクラスのモデルだとおよそ200,000トークン。これがどれくらいかというと、原稿用紙にして数百枚分。一見ものすごく広いが、Claude Codeで実作業をしていると意外と早く埋まる。何が机を埋めるかというと——

* **会話ログ**：これまでのやり取り全部
* **読み込んだファイル**：「このファイル見て」と言うたびに机に乗る
* **コマンド実行結果**：`grep` の出力もBashの戻り値も全部記憶に入る
* **画像**：スクリーンショット1枚で数千トークン
* **MCP経由のデータ**：Webの検索結果、Slackの履歴なども入る

これらが積み重なっていくと、机の広さを使い切る。

## 埋まるとどうなるか

埋まるにつれて以下のような症状が出る。

1. **前の指示を忘れる**——古い会話から押し出されて消える
2. **同じ確認を繰り返す**——一度伝えたことをもう一度聞かれる
3. **応答が遅くなる**——処理する情報量が大きい
4. **精度が落ちる**——重要な情報を探しながら答えるので注意が散漫になる
5. **料金が膨らむ**——APIは「入力トークン数」で課金されるので、机が厚いほど1ターンあたり高くなる

「Claudeがバカになった」ように見える瞬間は、たいていこのどれかが起きている。

## `/compact` で机を整理する

Claude Codeには **`/compact`** というコマンドがある。これまでの会話を要約・圧縮して、机の上を片付けてくれる。

タイミングは **コンテキストが7〜8割埋まったあたり**。「まだ余裕ある」と思ううちに打つのがコツで、埋まり切ってからでは出力品質に響いた要約しかできない。

実行すると、Claudeは

* 重要な意思決定や方針
* 進行中のタスクと現状
* 未解決の問題

を残したサマリーを作り、それを新しい起点として会話を続ける。「これまでの細かいやり取り」は消えるが、「次にやるべきこと」と「なぜそれをやるか」は引き継がれる。

## コンテキストを節約する4つの習慣

`/compact` に頼り切らず、日頃から机を散らかさない工夫も大事だ。

**1. 話題を混ぜない**

朝は営業メール、昼はSEO、夜はPythonスクリプト——これを1つのセッションでやると、Claudeは全部を「同じプロジェクト」として机に積み上げる。プロジェクトが変わったら新しいセッションを開く、これだけで体感がだいぶ違う。

**2. 大きいファイルを丸ごと読ませない**

「この3000行のコード全部読んで」は危険。机の半分が一瞬で埋まる。`grep` で該当部分を絞ってから渡すか、「まずこの関数だけ見て」と限定する。

**3. スクショを大量に貼らない**

画像1枚で数千トークン。「設計の参考に」と10枚貼ると、数万トークンが一気に消費される。**いま判断に必要な1〜2枚だけ**にとどめる。

**4. 終わった話題は新セッションに切る**

長い会話を引きずるより、節目で新しいセッションを開いたほうがクリアに動く。「前回はこれをここまでやった」と短くまとめて渡せば、文脈は十分引き継げる。

## モデルを使い分ける——コストの大半はここで決まる

Claude Codeでは複数のモデルが選べる。同じ作業でもどのモデルで動かすかで、コストと速度が大きく変わる。

**Haiku（速い・安い）**

* 簡単な変換、フォーマット整え、定型処理
* 大量に繰り返し走らせるタスク
* 下書きや初稿の生成

**Sonnet（バランス型・標準モデル）**

* 普段の実装、リファクタ、複数ファイル横断の作業
* ドキュメント執筆、構成設計
* 大半のユースケースはここでOK

**Opus（高精度・高コスト）**

* アーキテクチャ設計の意思決定
* 複雑なバグの原因究明
* 重要な判断を含むレビュー

コスト感のおおまかなイメージは、**Haiku 1 : Sonnet 3〜5 : Opus 15〜25**。SonnetをデフォルトにしておいてHaikuに落としたりOpusに上げたり、というのが普通の使い方になる。

## 月額コストの現実的な目安

「Claude Codeって月いくらかかるの？」とよく聞かれる。

サブスクリプション（ProやMaxプラン）を使うのか、APIキーで従量課金にするのかで話は変わるが、ビジネス用途で1日1〜2時間使う前提の概算はこのくらい。

| 使い方 | 月額目安（API課金の場合） |
| --- | --- |
| ライト（Haiku中心、相談少なめ） | $20〜30 |
| 標準（Sonnet中心、1日1時間） | $50〜100 |
| ヘビー（Sonnet+Opus、1日2〜3時間） | $150〜300 |
| 並列Agent多用 | $300〜500以上 |

サブスクリプション（定額プラン）を使えば、月額固定でこの何倍も使える形にできる。「毎月いくらまで」と上限を決めたい人にはサブスク、使った分だけ払いたい人にはAPI、と性質が分かれる。

月額が想定より大幅にオーバーするときは、たいてい**コンテキストの無駄使い**か**並列Agentの使いすぎ**が原因。前者は `/compact` 習慣で、後者は「本当に並列が必要か？」の問い直しで多くが解決する。

## コスト最適化、3つの実践原則

1. **ルーチン作業はHaiku** ——スクレイピング、簡単な整形、定型タスク
2. **判断と設計はSonnet以上** ——アーキテクチャ、レビュー、複雑な分岐
3. **コンテキストが詰まる前に `/compact`** ——7割を超えたら反射的に打つ

この3つを意識するだけで、月額コストは無理なく30〜40%下がる。しかも精度も上がる。詰まった机では良い仕事ができないからだ。

## まとめ——「机を広く保つ」が一番効く

| 困りごと | 対処 |
| --- | --- |
| Claudeが「同じこと聞き返してくる」 | `/compact` で圧縮 |
| 応答が遅くなった | 新しいセッションを開く |
| 月額コストが高い | モデルの使い分けと並列の見直し |
| 大きいファイルを読ませると重い | 該当部分だけ切り出して渡す |

コンテキストウィンドウは作業机の広さ。広く使えているうちに整える習慣が、長期的なClaude Codeとの付き合いを快適にする。

「埋まってから何とかする」ではなく、「埋まる前に整える」。それだけで、より速く、より安く、より精度の高い作業ができる。
