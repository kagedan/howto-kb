## 0. 「使い倒さなきゃ」の焦りと、答えの見つからなさ

Fable 5が今の形で使える期間には、限りがあります（正確には、Proプランなどで使い放題の状態から、従量課金のみの提供に移行します）。「消える前に、なんとか使い倒さなきゃ」——そう思いつつ、**しかしどう使えばその価値を最大化できるのか、わからない**。そんな方が多いのではないでしょうか。

白状すると、私もそうでした。Fableが最初にリリースされたとき、触った瞬間にその凄みは体感しました。体感したのに、「で、この力をどこに向ければいいのか」が見えない。凄いものを前にして、日々のタスクを少し上手にこなさせるだけ。宝の持ち腐れという言葉が頭をよぎる日が続きました。

そこに、あの衝撃です。米国政府による規制で、Fableが一時的に使えなくなった。

ただ、これが結果的に効きました。「使い倒さなきゃ」の焦りから強制的に切り離されて、**考える時間**が生まれたんです。手が止まったので、頭を動かすしかない。私はふと、これまでの「最強モデル」たちがどうなったかを振り返ってみました。

すると、答えはおのずから見えてきました。

## 1. フロンティアモデルは、必ず「過去」になる

Claude Fable 5を使い始めて、最初に驚いたのは回答の賢さではありませんでした。**構成力**です。

提案書の壁打ちをしたときのこと。私が渡したのは要件の断片だけなのに、返ってきたのは「論点の切り方」「反論の先回り」「捨てる情報の判断」まで練り上がった構成でした。前職はコンサルでしたが、正直に言うと「これ、レビュー要らないな」と思う瞬間が何度もありました。思考の粗さを指摘するつもりで読み始めて、指摘が見つからずに読み終わる。あの感覚です。

でも、ここで一度立ち止まりたいんです。

GPT-4が出たとき、私たちは「もうこれ以上は要らない」と思いました。Claude 3 Opusのときも、その後のモデルでも同じことを言いました。そして全部、過去になりました。**「最強」は例外なく一時的**です。Fable 5もいずれ、より新しいモデルに置き換わるか、価格や提供形態が変わります。

では、消えていった歴代の「最強」たちから、手元に残ったものは何だったか。

答えは、**そのモデルで作った仕組みとデータ**でした。モデルは減価する。でも、モデルに作らせた「土台」は残る。

だとしたら、Fable 5という最強の思考力が使い放題である"今"の正しい使い方は、ひとつしかありません。

## 2. 知能の裁定取引——最強モデルは使い倒すな、「缶詰」にしろ

私はこれを**知能のアービトラージ（裁定取引）**　だと考えています。

Fable 5の知能を「日々の回答」に使うと、それは消費で終わります。使った瞬間に価値が発生して、その瞬間に消える。一方で、Fable 5の知能を「**構造**」に変換すると、それは資産になります。

構造とは何か。設計書、テンプレート、エージェントの規律、スキーマ、採点基準。つまり「**次からは考えなくていい形**」に固定されたものです。

ここで重要な検証結果をひとつ。Fable 5が復活してからの数日、意図的に同じタスクをFable 5とOpus 4.8、Sonnet 5で比較してきました。結論はこうです。

> **設計と指示が練り上がっていれば、Opus 4.8やSonnet 5クラスでも十分に戦える。**

素の状態で「提案書の構成を考えて」と投げれば、差は歴然です。でも、Fableに作らせた設計書・レイアウト定義・チェックリストを渡した上で「この型に従って中身を埋めよ」と指示すると、下位モデルの出力は見違えます。つまりモデル間の性能差は、かなりの部分を「**設計の質**」で吸収できる。

ならば、やるべきことは明確です。

**Fableで設計する → 型に固定する → 廉価モデルで運用する。**

私はこれを「知能の缶詰」と呼んでいます。旬の知能を、保存の効く構造に詰めておく。開けるのは、Fableがいなくなった後の未来の自分です。

私は普段から「できない → できる → 型化 → 渡す」というループで仕事を組み立ててきましたが、AIの世界ではこの「渡す」先が後輩や仕組みだけでなく、**未来の安いモデル**になった。そう捉えると、今やるべきことが一気にクリアになりました。

## 3. 実例：私が缶詰にした3つ

抽象論だけでは深みが出ないので、私が実際にFable 5で設計し、Claude Codeに作らせている3つを紹介します。

3つに共通する設計思想は一貫しています。

> **賢さを「LLMの実行時判断」に置かない。テンプレート・中間表現・機械的ガードに"前払い"で埋め込む。**

実行するモデルが賢い必要をなくす。これが缶詰の本体です。

### 3-1. KCHIVE：全AIとの対話を、自分だけのナレッジグラフに

**何を作ったか**：Claude、ChatGPT、Gemini、ローカルのClaude CodeやCodex——あらゆるAIとの対話履歴から「再利用可能な知識」だけを蒸留し、Obsidianのナレッジグラフとして一元化するツールです。過去ログの一括取り込み（バックフィル）と、以後は `kb add` 一発の追記運用ができます。

完成したので、動作動画とリポジトリを置いておきます。

<https://www.youtube.com/watch?v=0ezuMer6YPw>  
<https://github.com/ykishimotoy/kchive>

**どこに知能を前払いしたか**：  
知識の抽出プロンプトとカードのスキーマ（frontmatter設計・分野タクソノミー・重複判定基準）をFableで練り上げて固定しました。その結果、**日々の抽出処理はHaiku級の廉価モデルで回ります**。「会話ログの要約」ではなく「知識の蒸留」をさせる、この差分は全部プロンプトとスキーマ側に埋まっているからです。

そしてもうひとつ。モデルは入れ替わっても、**蒸留された「自分のデータ」だけは複利で増え続けます**。これが3つの中で最も長寿命の資産です。

### 3-2. HANCHO：開発チームを、サブエージェント編成で回す

**何を作ったか**：Web開発を「親玉エージェント＋8体のサブエージェント」で回すClaude Code基盤です。要件定義だけは人間と綿密にやりとりし、設計以降は自律で走る。レビューも自律、テストも自律です。

**どこに知能を前払いしたか**：  
一番の見どころは「**AIが自分に甘いテストを書く問題**」の解決です。普通に作ると、実装したモデル自身がテストも書くので、実装の癖に合わせた"通るためのテスト"が生まれます。

HANCHOではこれを、モデルの良心ではなく**構造**で潰しました。

* テスト作成エージェントは、**実装コード（src/）を読むこと自体が禁止**。要件定義書だけからテストを書く
* 実装エージェントは、**tests/ への書き込みがhooksで物理的にブロック**される
* テストが落ちたとき「テストが悪い」と裁定できるのは、書いた本人ではなく別のエージェント

「賢いモデルなら公正にやってくれるだろう」に頼らない。**規律をhooksという機械に埋め込んだので、実行モデルの格が落ちても品質担保の構造は劣化しません。**

これも構造が出来上がったのでGitHubレポジトリを共有します。  
<https://github.com/ykishimotoy/hancho>

### 3-3. DECK：アイデアを入れるだけで、コンサル品質のパワポが出る

**何を作ったか**：テンプレートに則ったpptxを生成するClaude Codeスキルです。生成 → 画像化 → ルーブリック採点 → **90点を超えるまで自動修正**、というQAループを内蔵しています。

**どこに知能を前払いしたか**：  
核心は「**LLMにコードもXMLも書かせない**」ことです。LLMが書くのは圧縮されたYAML（スライドの中身の指定）だけで、座標・配色・整列・フォントはすべて事前定義のレイアウトカタログとPythonレンダラが担います。

トークン消費の比較がこちら（10枚デッキの生成時の概算）：

| 方式 | 出力トークン | 廉価モデル耐性 |
| --- | --- | --- |
| python-pptxコード直書き | 15,000〜25,000 | 座標ミス・API誤用が頻発 |
| OOXML直編集 | 30,000以上 | ほぼ無理 |
| **YAML→レンダラ方式** | **2,500〜4,000** | レイアウトは型が保証 |

約1/6です。しかも廉価モデルの弱点である空間推論や数値精度は、**そもそも出番がない**。文字数上限もレイアウト側に定義してあるので、オーバーフローが原理的に起きにくい。「Sonnet 5クラスで動く」のは根性ではなく設計の帰結です。

ただし1点だけ補足を。これはまだ調整中で、本当に上手くいくかは未知数です。でもその検証をもとに「この方法は実装してみたけどダメ、別の方法論はどうか」と高速で試行錯誤できるのも、AI時代ならではだと思っています。

### 3つの共通構造

並べてみると、全部同じ形をしています。

```
┌─────────────────────────────────────┐
│  型（Fableが設計）                    │  ← 一度だけ、最強モデルで作る
│  スキーマ / レイアウト / エージェント規律 │
├─────────────────────────────────────┤
│  中身（廉価モデルが実行）              │  ← 日常運用。安く、何度でも
│  知識抽出 / 実装 / スライドspec        │
├─────────────────────────────────────┤
│  検証（機械が担保）                    │  ← LLMを信用しない層
│  重複判定 / hooks / 採点ループ         │
└─────────────────────────────────────┘
```

知能は上の層に前払い済み。真ん中は安いモデルで回り、下の層が品質を機械的に守る。**Fableがいなくなっても、この構造は1ミリも劣化しません。**

## 4. あなたの缶詰は何か——この2つを貼れば、Fableが答えを出す

さて、ここまでの3つはあくまで「私の立場」での最適解です。ITコンサルで、コミュニティを運営していて、本を書いている人間の缶詰。あなたの缶詰は、あなたの仕事と生活の中にしかありません。

……と書くと「じゃあ自分で考えなきゃ」となりますが、**その必要すらない**んです。Fable級の思考力なら、正しい問いと"あなたの生情報"さえ渡せば、作るべき土台の草案は一発で出ます。

必要なのは2ステップ、所要2分です。

### Step 1：普段使いのAIから「自分」を抽出する

履歴が一番溜まっているAI（ChatGPTでもGeminiでもClaudeでも）に、これを貼ってください。

```
あなたが私との過去のやりとり・メモリから把握している私の情報を、
第三者のAIに渡すためのプロファイルとしてYAML形式で出力してください。
推測には (推測) と付記。各項目3〜5個、なければ「不明」でよい。

profile:
  職業・役割:
  繰り返している知的作業:   # 週次・月次で繰り返し依頼してくること
  よく作る成果物の型:       # 資料、コード、文章、企画書など
  判断の癖・こだわり:       # 私が毎回直させるポイント、譲らない基準
  私だけが蓄積しているデータ: # 他人が持っていない情報・経験・記録
  苦手で外注したい作業:
  使用中のAIツール・環境:
```

### Step 2：出てきたYAMLを、Fableにこれと一緒に貼る

```
以下は私のプロファイルです。前提：フロンティアモデル（あなた）は
いずれ使えなくなり、日常運用は廉価モデルで回すことになります。

この前提で、私が「今、最強モデルの思考力があるうちに作っておくべき
土台・雛形」を3つ提案してください。各案は次の形式で：
1. 名前と一言コンセプト
2. 缶詰にする知能：何の判断・構造を今のうちに型に固定するか
3. 廉価モデルで回る理由：実行時のLLMの役割をどこまで小さくできるか
4. 最初の一歩：Claude Codeに渡す最初の指示文（そのままコピペ可能な形で）

条件：ツール紹介ではなく、私のプロファイル固有の設計であること。
汎用的な提案（「メモを取ろう」等）は禁止。

--- 以下、プロファイル ---
（Step 1の出力を貼る）
```

これで、あなた固有の「作るべき土台」の草案と、Claude Codeに渡す最初の指示文まで手に入ります。

種明かしをすると、このプロンプト自体が「知能の前払い」の実演です。問いの設計をあなたの試行錯誤に任せず、記事側で固定してある。だから2分で試せる。そして私のKCHIVE・HANCHO・DECKも、実質このプロファイル→設計の流れから生まれています。

出てきた草案の見極め方は1行で足ります。**「そのモデルが消えても、価値が残るか？」**——残るものだけ作ってください。

## 5. 余談：Fableの「燃費」の肌感

土台構築以外の話も少しだけ。執筆や調べ物の方面でも、Fable 5はめちゃくちゃ有用でした。書籍の構成検討では、章立ての論理矛盾を自分より先に見つけてきます。

ただし警告をひとつ。**調べ物系はトークンが一瞬で溶けます。** [AI業界を俯瞰する本](https://zenn.dev/ykishimotoy/books/de8bc95c9b3d57)の下調べと執筆をFableに任せたら、Max 5xプランの上限に達し、2回目の上限解放でやっと吐き出し切りました。深く広く調べさせると、消費は本当に容赦がない。

でも逆に言えば、「Fableは何にどれくらいかかるのか」という肌感を**乱暴に試せるのも今だけ**です。上限に殴られるのも経験値。私はこの1ヶ月で「重い調査は骨子だけFable、収集は廉価モデル」という配分感覚を体で覚えました。これも一種の缶詰です。

そして、「自分の人生を棚卸しして、可能な限り生情報としてプロンプトに書き込んで作成した自己啓発書」も作成してみたのですが、これはProプラン（maxではなく）の上限ギリギリで回りきり、この時の生成は従量課金でやったとしても28ドルで済むことがわかりました。文章プロンプトで完結する原稿執筆は相性が非常に良い、ハイクオリティの本の原稿をこの価格で出せる、これもやってみないとわからない経験知ですね。

## 6. まとめ：モデルは減価する、土台は複利で増える

* Fable 5の凄さは思考力と構成力。ただし「最強」は例外なく一時的
* 設計さえ練り上がっていれば、Opus 4.8やSonnet 5クラスで十分戦える
* だから今やるべきは、**Fableの知能を構造に変換して固定する**こと——知能の缶詰
* 型はFableが作り、中身は廉価モデルが埋め、検証は機械が守る。この3層にすれば、モデルが入れ替わっても資産は劣化しない
* あなた固有の缶詰は、上の2つのプロンプトを貼れば2分で草案が出る

**Fable 5が消える前にすべきことは、Fable 5がいなくても回る仕組みを作ることです。**
