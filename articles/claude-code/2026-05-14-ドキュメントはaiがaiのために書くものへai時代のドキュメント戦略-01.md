## 🚀 はじめに

「動くソフトウェア > 包括的なドキュメント」というアジャイルマニフェストの一節は、長らく「ドキュメントは後回しでいい」という免罪符として使われてきました。また、生成 AI の普及により「AI がコードを直接理解できるから、ドキュメントは書かなくていい」と誤解してしまっている方もいらっしゃるかもしれません。

しかし、AI 時代においてこの考え方は **致命的** です。

AI コーディングエージェントの出力品質は、入力されるコンテキストの品質に直接依存します。ドキュメントがなければ、AI は何も知りません。何も知らない人間が書くコードも困りものですが、何も知らない AI はもっとたちが悪い。**人間の数十倍の速度で、見当違いな実装を大量に作り出してしまう** からです。

**ドキュメントはもはや、AI エージェントが、未来の別の AI エージェントのために残す記録です**。今の AI が下した設計判断、採用しなかった選択肢、プロジェクトのルールといった情報を記録として残さなければ、次にやってくる AI エージェントは同じ過ちを繰り返します。

この記事では、DORA の実証データや Thoughtworks の議論を踏まえ、AI 時代のドキュメント戦略として **3 つの原則** を提案します。

## 📊 背景：なぜ今ドキュメント戦略を見直すべきか

### DORA が実証した「ドキュメントは AI の戦力倍増器」

Google の DORA（DevOps Research and Assessment）チームは、2021 年にドキュメント品質を独立したケイパビリティとして初めて測定しました。以降の研究で、ドキュメントの重要性は年々明確になっています。

| 年 | 主要な発見 |
| --- | --- |
| **2021** | ドキュメント品質を**初めて独立ケイパビリティとして測定**。高品質な内部ドキュメントがソフトウェアデリバリと運用パフォーマンスを有意に向上させることを発見 |
| **2022** | ドキュメント品質が**戦力倍増器**として機能することを実証。高品質ドキュメント + トランクベース開発 → **組織パフォーマンス 1,525% 向上**。CI では高品質ドキュメントで **750%**、低品質で 34%（**22 倍の差**） |
| **2024** | AI 導入の効果を初の定量モデルで測定。AI 採用率向上に伴い個人の生産性は向上する一方、**デリバリ安定性は 7.2% 悪化**。AI は万能ではなく、基盤となるプラクティスの質が結果を左右することが示唆された |
| **2025** | 「**AI は増幅器**」と結論。組織の既存の強みも弱みも増幅する。7 つのケイパビリティ（AI-accessible internal data 等）が AI の恩恵を決定。ドキュメント品質は独立項目から「AI-accessible internal data」ケイパビリティに吸収された |

DORA 2025 でドキュメントが「AI アクセス可能な内部データ」に吸収されたことは、ドキュメントの役割が「人間向けの品質属性」から「**AI が消費するコンテキストデータ**」へと変化したことを意味しています。

> *"You can't connect an AI to information that doesn't exist, and you shouldn't connect it to information that is wrong."*  
> （存在しない情報に AI を接続することはできないし、間違った情報に接続すべきでもない。）  
> — DORA, [AI-accessible internal data](https://dora.dev/capabilities/ai-accessible-internal-data/)

### 認知的負債：AI が蓄積する新しい負債

2026 年 2 月、Margaret-Anne Storey は Thoughtworks のリトリートで **認知的負債（cognitive debt）** を提唱しました。

> *"Technical debt lives in the code; cognitive debt lives in developers' minds."*  
> （技術的負債はコードに宿る。認知的負債は開発者の頭の中に宿る。）  
> — Margaret-Anne Storey, 2026

技術的負債がコードに蓄積するのに対し、認知的負債は **開発者の頭の中にだけ存在する暗黙知** に蓄積します。AI コーディングエージェントは以下の理由でこの負債を加速させます。

1. **セッション間でコンテキストが失われる** — AI は新しいセッションを開始するたびに前回の知識を忘却する
2. **AI が生成したコードの「なぜ」が残らない** — AI はコードを書けるが、設計判断の背景を自発的には記録しない
3. **速度が仇になる** — 人間の数十倍の速度でコードを書くため、ドキュメント化されていない判断が高速に蓄積する

> *"This velocity multiplier becomes a debt accelerator."*  
> （この速度乗算器は、負債の加速器になる。）  
> — Rachel Laycock (Thoughtworks CTO), 2026

### アジャイルマニフェストの再解釈

> *"Working software over comprehensive documentation"*  
> （動くソフトウェアを、包括的なドキュメントよりも重視する）  
> — Agile Manifesto, 2001

マニフェストが否定しているのは **"comprehensive"（包括的な）** ドキュメントであり、ドキュメントそのものではありません。

| 2001 年の暗黙の前提 | AI 時代の現実 |
| --- | --- |
| ドキュメント執筆は人間の時間を大量に消費する | AI は人間の数十倍の速度で書ける |
| ドキュメントはすぐに陳腐化する | AI が CI ゲートで一貫性を維持できる |
| 対面コミュニケーションが最も効率的 | AI は対面会話にアクセスできない |

AI がドキュメントを高速に書ける今、**ドキュメントを書かない理由がなくなった** のです。

2026 年 2 月の Deer Valley リトリートでは Laura Tacho がこう述べました。

> *"The Venn Diagram of Developer Experience and Agent Experience is a circle."*  
> （開発者体験とエージェント体験のベン図は、一つの円である。）  
> — Laura Tacho, 2026

人間にとって良い開発環境は、AI にとっても良い開発環境です。以降で述べる 3 つの原則は、人間のためでもあり、AI のためでもあります。

## 📝 原則 1：すべてのドキュメントを Markdown で書く

| ❌ 避けるべき形式 | ✅ 推奨する形式 |
| --- | --- |
| 📊 PowerPoint | 📝 Markdown |
| 📸 ホワイトボード画像 | 🧜 Mermaid / PlantUML |
| 🖼️ draw.io バイナリ | 📁 テキストベース DSL |
| 📄 Word / PDF | 📋 ADR（Markdown） |

AI 時代のドキュメントは、**人間にとっての可読性** と **AI にとっての機械可読性** を両立する必要があります。この両立を実現する最適な形式が **Markdown** です。

### なぜ Markdown か

1. **人間も AI も読み書きできる** — プレーンテキストでありながら構造化された情報を表現できる
2. **Git diff がクリーン** — PR で変更点が人間にも AI にも一目瞭然
3. **ツール非依存** — 特定のソフトウェアに依存せず、どのエディタでも編集可能
4. **GitHub でネイティブレンダリング** — リポジトリ内でそのまま読める

### 図表も Markdown 内にテキストで書く

ドキュメントに図表は不可欠ですが、AI が正確に読み取れない形式は避けるべきです。

* **PowerPoint で描いた図** — pptx は ZIP 圧縮された XML の集合体であり、AI にとってノイズが多い
* **ホワイトボードに書いてカメラで撮った画像** — OCR の精度に依存し、構造情報が失われる
* **draw.io のバイナリ形式** — Git diff も効かず、AI も読めない

代わりに、**Mermaid** や **PlantUML** のようなテキストベースの DSL で表現します。中でも Mermaid は GitHub の Markdown 内でネイティブにレンダリングされ、AI が生成・修正・解釈しやすく、第一選択肢として最適です。

テキスト DSL の決定的な利点は、**AI が図を生成・更新できる** ことです。AI が設計判断を下したときに、アーキテクチャ図やシーケンス図もセットで生成・更新させれば、ドキュメントとコードの乖離を防げます。

### ADR は Markdown で書く最重要ドキュメント

ADR（Architecture Decision Records）は AI 時代において最も重要なドキュメント形式です。

1. **「なぜ」を記録する唯一の仕組み** — コードは「何を」表現するが「なぜ」は表現しない
2. **AI のリグレッション防止** — AI が過去に否定された選択肢を再提案するのを防ぐ
3. **認知的負債への直接的な対策** — セッション間で失われるコンテキストを永続化する

```
## Context（背景）
<!-- 意思決定が必要になった状況・制約を価値中立に記述 -->

## Decision（決定）
<!-- "We will ..." で始める。能動態で完全な文 -->

## Alternatives Considered（検討した選択肢）
<!-- 検討したが採用しなかった選択肢と、不採用の理由 -->

## Consequences（結果）
<!-- 正・負・中立すべての影響 -->
```

特に **`Alternatives Considered`** セクションが重要です。AI がセッション間でコンテキストを失っても、このセクションを読めば「なぜ他の選択肢を採用しなかったか」を理解できます。これがないと、AI は毎回同じ議論を蒸し返します。

なお、Devin の DeepWiki のようにリポジトリからドキュメントを自動生成するツールも登場していますが、これらはソースコードの **現在の状態** を読み取って記述するだけであり、「なぜこの設計になったのか」「どの選択肢を検討して棄却したのか」までは読み取れません。ADR のような「なぜ」の記録は、人間と AI が対話の中で意識的に残す必要があります。

## 📁 原則 2：すべてのドキュメントをソースコードと同じリポジトリで管理する

「Confluence や SharePoint にドキュメントを置いても、AI がアクセスできれば問題ないのでは？」と思うかもしれません。しかし、問題は **アクセス可能性** ではなく、**バージョン管理の一体性** にあります。

### 分離管理の構造的問題

ソースコードとドキュメントのバージョン管理が分離していると、以下の問題が構造的に発生します。

* **バージョンの紐づけが困難** — ソースコードのあるバージョンに対応するドキュメントのバージョンが管理しきれない。AI が古いドキュメントを参照して見当違いな実装を生成するリスクが常につきまとう
* **アクセス可能性の喪失** — ドキュメントの移動、削除、リンク切れなどにより、AI からも人間からもアクセスできなくなる
* **同時更新の保証がない** — ソースコードを更新してもドキュメントが更新されない。DORA 2025 が警告する「間違った情報に AI を接続する」状態に陥る

同じコミットでソースコードとドキュメントが一緒に更新され、`git checkout` で任意の時点のコードと仕様を同時に復元できる。この一体性が、AI が信頼できるコンテキストを得るための前提条件です。

### CI にドキュメントゲートを設ける

「ドキュメントは後で書く」を構造的に防止するには、CI パイプラインにドキュメントゲートを組み込みます。ドキュメントゲートには 2 つの層があります。

1. **構造・文体のチェック** — textlint, markdownlint による Markdown の品質検証
2. **コードとの整合性チェック** — AI がコード変更を理解し、ドキュメントとの乖離を検知

#### 従来のドキュメント Linter の限界

従来はドキュメントの Lint といえば、textlint や markdownlint といったツールを使うことが主流でした。これらはドキュメントの文体や Markdown の構造をチェックするには有効です。しかし、これらが検証できるのはあくまで **ドキュメント単体の品質** であり、AI 時代に必要な以下のチェックはできません。

* コード変更についてなぜこの変更をするに至ったのか、理由や経緯がドキュメントに記録されているか
* コード変更に対応するドキュメントが正しく更新されているか
* コードの振る舞いとドキュメントの記述に乖離が発生していないか
* 新機能の追加に対して必要なドキュメントが作成されているか

わざわざ CI ゲートでドキュメントが書かれているかチェックしなくても、リポジトリを読み取り自動的にドキュメントを生成させればいいのでは、と思うかもしれません。しかし、先述のとおり自動生成ツールは「何があるか」を記録できても、「なぜそうなったか」は記録できません。CI ゲートの目的は、この「なぜ」が確実に記録されるようにすることです。

従来のルールベースの GitHub Actions では、のコードの意味的な変更を理解し、それに対応するドキュメントの整合性を判断することはほぼ不可能です。「`src/` が変更されたら `docs/` にも変更があるか」程度のファイルパスベースのチェックは書けますが、内容の整合性までは検証できません。

#### AI を活用したより高度なドキュメントゲート

GitHub が発表した [**Agentic Workflows**](https://github.blog/jp/2026-02-16-automate-repository-tasks-with-github-agentic-workflows/) は、この課題に対する現実的な解決策を示しています。PR をトリガーとして AI エージェントを起動し、コード差分とドキュメントを横断的に読み取って乖離を自動検知する仕組みが構築できるようになれば、従来の CI では不可能だった「コードとドキュメントの意味的な整合性チェック」が実現できるでしょう。

2026-05-15 時点ではまだ Agentic Workflows は一般公開されていないので自動化はできませんが、手動でこのようなチェックを促すことはできるでしょう。スキルを作り、PR のテンプレートに「`/docs-lint` スキルを使用して AI にドキュメントチェックを行わせること」といったチェック項目を書いておけば、ドキュメントゲートの効果は期待できます。

## 🤖 原則 3：カスタムインストラクションを整備し、AI が自動的にドキュメントを残す仕組みを作る

原則 1（Markdown）と原則 2（リポジトリ管理）は、ドキュメントの **形式** と **置き場所** を定めました。しかし、最も重要な問題が残っています。**誰がドキュメントを書くのか？** です。

人間がすべてを手書きするのは非現実的です。AI の開発速度に人間のドキュメント執筆は追いつきません。答えは、**AI に書かせる** ことです。カスタムインストラクションを整備し、AI が対話の中で決定された設計判断やルールを自動的にドキュメントとして残す仕組みを作ります。

### コンテキストエンジニアリング：何を永続化し、どう読み込ませるか

AI が受け取るコンテキストには永続性の異なる複数の層があります。[LangChain が提唱した「コンテキストエンジニアリング」](https://blog.langchain.com/the-rise-of-context-engineering)の考え方をもとに、永続性の観点で整理すると以下のようになります。

| 層 | 特性 | 例 |
| --- | --- | --- |
| **プロジェクト知識** | 永続・構造化（Git 管理） | インストラクションファイル、ADR、Design Doc |
| **AI メモリ** | セッション横断・非永続 | Copilot Agentic Memory（28 日）、Claude Auto Memory |
| **タスク記録** | 永続だが未整理 | Issue、PR（議論・試行錯誤を含む） |
| **対話履歴** | セッション限り | 自動（有限） |
| **ツール出力** | 一時的 | AI が自動取得 |

DORA 2025 もケイパビリティ #3「AI-accessible internal data」の実践手法としてコンテキストエンジニアリングに言及しており、AI に内部ドキュメントやコードベースを接続する仕組みの重要性を強調しています。

#### 整理してプロジェクト知識に昇格させる

最上段の **プロジェクト知識** だけが、構造化され、いつでも AI が信頼して参照できる情報です。それ以外の層は、揮発性が高い（対話履歴・ツール出力）か、永続であっても試行錯誤や議論が混在して未整理（Issue・PR）か、期限付き（AI メモリ）です。

これらの層に散在する情報の中から、重要な設計判断やルールを **整理して** プロジェクト知識に昇格させる必要があります。

AI メモリ層（Copilot Agentic Memory や Claude Auto Memory）はこの昇格の中間地点です。AI が作業中に発見した知識を自動的に蓄積しますが、永続ではありません（例：Copilot は 28 日で期限切れ）。ここに溜まった知識のうち重要なものを、人間がレビューして ADR やインストラクションファイルとしてコミットする。これが原則 3 のサイクルの核心です。

#### 読み込みは必要なときに必要なものだけ

プロジェクト知識に整理・昇格させたドキュメントであっても、すべてを AI のコンテキストに詰め込めばよいわけではありません。DORA 2025 はこう警告しています。

> *"While feeding large, raw documents into an AI's context window is tempting, it is often counterproductive."*  
> （大量の生ドキュメントを AI のコンテキストウィンドウに流し込みたくなるが、これは往々にして逆効果である。）  
> — DORA 2025

コンテキストウィンドウは有限です。パスベースルールと遅延読み込みで「必要なときに必要なものだけ」読み込ませる設計が重要です。情報の整理・永続化（何を残すか）とロード戦略（どう読み込ませるか）は別の問題であり、両方を意識する必要があります。

### インストラクションファイルの整備

主要な AI コーディングエージェントはいずれも、リポジトリ内のインストラクションファイルを読み取る仕組みを持っています。

|  | GitHub Copilot | Claude Code | Cursor |
| --- | --- | --- | --- |
| **メインファイル** | `copilot-instructions.md` | `CLAUDE.md` | `.cursor/rules/` |
| **パスベースルール** | `.instructions.md` + `applyTo` | `paths:` YAML | `globs:` YAML |
| **AI 自動学習** | Agentic Memory（28 日） | Auto Memory | なし |

記載すべき内容は各ツール共通です。

```
## Project Overview                    ← 言語・フレームワーク・目的
## Critical Build & Validation Steps   ← コミット前に実行するコマンド
## Project Structure                   ← ディレクトリツリー
## Testing Guidelines                  ← テストの書き方
## Code Style & Linting                ← 命名規則
## Common Errors & Solutions           ← エラー→修正のFAQ
## Important Reminders                 ← NEVER / ALWAYS ディレクティブ
```

### AI にドキュメントを自動生成させる仕組み

カスタムインストラクションの真価は、AI に「コードだけでなくドキュメントも書け」と指示できることにあります。

具体的には、インストラクションファイルに以下のようなルールを記載します。

* **設計判断を下したら ADR を書く** — 新しいライブラリの採用、アーキテクチャの変更など、重要な判断には `docs/architecture-decisions/` に ADR を作成する
* **対話で決まったルールはインストラクションに追記する** — プロジェクト固有の制約やパターンが対話で明らかになったら、インストラクションファイルに記録する
* **PR にはドキュメント更新を含める** — コード変更に影響を受けるドキュメントを同時に更新する

主要ツールはすでに **AI が自律的にドキュメントを蓄積する仕組み** も備えています。GitHub Copilot の Agentic Memory は作業中に発見した知識を自動保存し、Claude Code の Auto Memory は `MEMORY.md` に記録を残します。ただし、これらの自動記録は揮発性が高いため、重要な判断は人間がレビューしたうえで ADR やインストラクションファイルに昇格させるべきです。

このサイクルにおいて **人間の役割はレビューと承認** です。AI が書いたドキュメントを PR でレビューし、正確性を確認してマージする。AI の開発速度を活かしつつ、人間の判断力で品質を担保する分業です。

## 🎯 まとめ

AI は増幅器です。良い開発プラクティスを増幅もすれば、悪い開発プラクティスも増幅します。

ドキュメントがなければ、AI は増幅するものがありません。ドキュメントが間違っていれば、AI は間違いを増幅します。

**ドキュメントは、AI という増幅器の入力信号そのものです。** 3 つの原則でこの信号の品質を高めましょう。

1. **Markdown で書く** — 人間と AI の両方が読み書きできる形式を選ぶ
2. **リポジトリで管理する** — コードとドキュメントのバージョンを一体化する
3. **AI に自動記録させる** — カスタムインストラクションで AI がドキュメントを残す仕組みを作る
