2026年5月28日にリリースされた Claude Opus 4.8 を、化学の実験レポートの型（目的・理論・結果・考察・結論）で、数値を中心に読み解きます。「すごい」で消費せず、どの数字がどれだけ動き、何が本当の変化なのかを切り分ける——その試みの第0版です。

![](https://assets.st-note.com/img/1780271886-MJrkyEpb03dzLXS8FOnaDH9t.png?width=1200)

## 1. 目的

データを「目的・方法・結果・考察」という型で、印象ではなく、何を測り、何が分かって、何が分からなかったのかを分けて記録する。

第0版の目的は二つあります。ひとつは、Claude Opus 4.8 と同時公開の Dynamic Workflows の変更点を、できる限り具体的な数値で整理すること。もうひとつは、このシリーズで一貫して使う観測の枠組み（後述の3軸）を整えることです。

## 2. 理論・背景

### 2.1 リリースの経緯と系譜

![](https://assets.st-note.com/img/1780272054-viTOzDhnQj1xSgosYV6tHZ0l.png?width=1200)

図1：リリース系譜タイムライン

Claude Opus は Anthropic の最上位モデル系列です。4.8 のリリースで目を引くのは、その更新ペースでした。<br>**前バージョン Opus 4.7 からわずか41日**。これは公式系譜のなかでも最速のペースです [5]。

背景として、4.7 はリリース当時あまり芳しい評価を受けておらず、コメントの冗長さやツール呼び出しの不安定さが指摘されていました [8]。その意味で 4.8 は、純粋な能力向上であると同時に、前バージョンの「手直し」という側面も持っています。短い間隔は、その性格を裏づけているように見えます。

### 2.2 前提（測定条件）

数値を読む前に、測定の前提を押さえます。

* **対象**：Claude Opus 4.8（API モデルID: claude-opus-4-8）[5]
* **比較対象：前バージョン Opus 4.7、他社の GPT-5.5・Gemini 3.1 Pro [3]
  提供環境：Claude API / Amazon Bedrock / Google Cloud Vertex AI、コンテキスト長は最大100万トークン、最大出力12.8万トークン [5]**
* **測定条件**：システムカードの標準設定は「adaptive thinking を max effort、特記なき場合は5試行の平均」[3]。エフォートは既定が high で、その上に xhigh・max があります [4]
* **価格**：入力 $5・出力 $25（100万トークンあたり）。4.7 と同一です [5][7]

「同じ価格のまま中身が変わった」という点は、後のコスト評価で効いてくる前提です。

### 2.3 観測の枠組み：能力・信頼性・コストの3軸

新モデルを「すごい」で終わらせないために、このシリーズでは次の3軸で見ます。本稿の結果・考察もこの順で並べます。

* **能力**：何が、どれだけできるようになったか（ベンチマーク）
* **信頼性**：どれだけ安心して任せられるか（自己申告・暴走耐性・一貫性）
* **コスト**：それを、いくらで・どれだけの速度で得られるか

## 3. 結果

### 3.1 能力評価：主要ベンチマーク比較（SWE-bench / USAMO） 公開された主要ベンチマークを領域別にまとめます。

※比較対象の GPT-5.5 および Gemini 3.1 Pro の数値は、各社が 2026年4月〜5月に公開したテクニカルレポートの最新値に基づきます [3][7]。

![](https://assets.st-note.com/img/1780273052-uA1WP8JMonQdxreRq5h9Epib.png?width=1200)

![](https://assets.st-note.com/img/1780273010-pU9r6WtGN1MZQ3zKY7VDgBsy.png?width=1200)

数字を領域ごとに読むと、傾向がはっきりします。

**コーディング**では、すでに飽和に近い SWE-bench Verified は +1.0（87.6→88.6）にとどまる一方、より難しく汚染の少ない SWE-bench Pro が +4.9（64.3→69.2）伸び、GPT-5.5（58.6）に10.6ポイント差をつけました [2][3]。「簡単な問題はもう差がつかない、難しい問題で差が開く」という構図です。ただし**端末エージェント（Terminal-Bench 2.1）は GPT-5.5（78.2）が 4.8（74.6）を上回ります** [3][4]。

**エージェント／コンピュータ操作**は、OSWorld-Verified が 83.4、Online-Mind2Web が 84 と、競合を上回りました [3][6]。

**推論・数学**では、USAMO 2026 が 69.3→96.7 と、1サイクルで +27.4 ポイント跳ねました [3][6]。これは Opus 系で過去最大級の数学の伸びです。一方 GPQA Diamond は 94.2→93.6 とわずかに後退しており、これは飽和した指標で起きる典型的な揺らぎです [3][10]。

### 3.2 信頼性：自己申告の正確性

本リリースで最も特徴的なのが、この軸です。Opus 4.8 は、**コードの欠陥を指摘せず見逃す確率が 4.7 比で約4分の1**に下がりました [7][8]。さらに、欠陥のある結果を無批判に報告するスコアが **0% になった初の Claude モデル**で、過信（overconfidence）は 4.7 比で**10倍以上**減ったと報告されています [3]。提供元も、自分の作業についての不確実性をより申告しやすく、根拠のない主張をしにくくなったと説明しています [1]。

### 3.3 コスト・速度・労力

価格は据え置きですが、コストの選択肢が増えました。

* 標準価格は入力 $5・出力 $25（100万トークンあたり）で、4.7 と同一です [5][7]。
* 新しい fast mode は約2.5倍の速度で、料金は2倍（$10／$50）。ただし 4.7 の fast mode より3倍安い水準です [4][5]。
* コスト面の注意点として、**GPT-5.5 は出力トークンあたり約4割安い**という指摘があります [7]。価格で見るなら無条件にお得とは言えません。

### 3.4 Dynamic Workflows

![](https://assets.st-note.com/img/1780272305-f2yG6DYLIm3juCbqxvsBMnpd.png)

図3：Dynamic Workflows フロー図

Opus 4.8 と同時に、Claude Code 向けの新機能 **Dynamic Workflows** がリサーチプレビューとして公開されました [1][3]。中核の動きは、①Claude がオーケストレーション用スクリプトを動的に生成 → ②1セッションで数十〜数百の並列サブエージェントを起動 → ③敵対的エージェントが結論に反証 → ④収束まで反復 → ⑤報告、というものです [3]。

効果は数値にも出ています。たとえば BrowseComp は単一エージェントで +5.0 ですが、**マルチエージェントのオーケストレーターを使うと 88.5% まで上がります** [4]。「1体より、検証し合う複数体」が効くことを示す一例です。

## 4. 考察

### 4.1 数字の読み方：どこが本当に伸びたのか

10個の数字をそのまま眺めると圧倒されますが、読みどころは3つに絞れると考えられます。第一に、伸びは「飽和した簡単な指標」ではなく「難しい指標」に集中していること（SWE-bench Pro +4.9 に対し Verified は +1.0）。第二に、USAMO の +27.4 は単なる改善ではなく、数学的推論の質的な跳躍を示唆していること。第三に、GPQA のわずかな後退が示すとおり、**すべての数字が一様に上がったわけではない**こと。これらを踏まえると、今回の最適化は「論文用の指標」ではなく「実務で効く難しいタスク」に向いていると考えられます [10]。

### 4.2 「正直さ」を数値で評価することの意味

定量結果（3.1）だけ見れば順当な性能向上ですが、実務でのインパクトが大きいのは、むしろ信頼性（3.2）だと考えられます。理由はシンプルです。実務で人を困らせる不具合の多くは、「モデルが間違うこと」そのものより、「間違ったまま、それを報告しないこと」から生まれているように思います。「欠陥を見逃す確率が4分の1」「無批判報告0%」という数字は、ベンチマークの脚注ではなく、運用上の信頼性に直結する変化として読めるのではないでしょうか。

### 4.3 Dynamic Workflows との一貫性

![](https://assets.st-note.com/img/1780272330-xeE928BqKlyQX3h0rMIfRtdN.png?width=1200)

この見方に立つと、Dynamic Workflows（3.4）の設計も同じ方向を向いて見えてきます。敵対的エージェントにあえて反証させ、収束まで反復する仕組みは、ひとつのエージェントが陥りがちな過信を、アーキテクチャのレベルで抑える試みだと考えられます。モデル単体の「正直さ」（3.2＝第一層）と、複数エージェントの「相互反証」（3.4＝第二層）は、どちらも『AIの過信をどう抑えるか』という同じ問いへの、層を変えた回答なのかもしれません。BrowseComp が単体より多体で 88.5% まで上がる事実は、この第二層が数値としても効くことを示しています。

### 4.4 実装者の視点：信頼性とコストの読み

信頼性は、実装側の言葉では「どれだけの時間、人間が見ていなくても任せられるか」と言い換えられるように思います。無人で長く走らせる用途ほど、瞬間的な賢さより、誤りを申告し・暴走しない性質が効いてきます。

コストも、価格表だけでは判断を誤りそうです。標準価格は据え置きで、性能だけ上がったので「性能あたりの価格」は確かに改善しています。一方で fast mode は速度2倍化に料金2倍、GPT-5.5 は出力が約4割安い。**「速さが要るのか」「正確さが要るのか」「単価が要るのか」で最適解が変わる**、と考えられます。

### 4.5 限界と留意点（実機検証への課題）

数値を扱う記事として、限界を明記します。 本稿（第0版）は、公開されたシステムカードおよび一次資料の精読に基づく「理論値」の整理です。したがって、第三者の視点による再現的検証は、次稿（Vol.1）以降の主題となります。

* **端末コーディングでは GPT-5.5 が優位**で（Terminal-Bench 2.1: 78.2 vs 74.6）、用途で最適モデルは変わります [3][4]。
* Terminal-Bench はハーネス（実行環境）で結果が変わります。公開ハーネスでは GPT-5.5 優位、各自のハーネスでは別の数字になりうるため、**同条件比較**で読む必要があります [3]。
* OSWorld の伸びには評価ハーネスの更新が混ざっており、純粋なモデル改善とは切り分けて読むべきです [6]。
* **自己申告精度の数値は、提供元や早期テスターの報告**に基づきます。第三者の再現的検証は、本シリーズの今後の主題です。
* 本稿の数値はシステムカード由来（max effort・5試行平均が中心）であり、シングルショットの体感とは異なる可能性があります。

## 5. 結論

第0版の結論を、三点に整理します。

1. Claude Opus 4.8 の伸びは「飽和した簡単な指標」ではなく「難しい指標」に集中しており（SWE-bench Pro +4.9、USAMO +27.4）、実務寄りの最適化だと考えられます。
2. ただし本質的な変化は性能よりも**信頼性**にあります。「欠陥を見逃す確率1/4」「無批判報告0%」は、無人運用ほど効いてくる変化です。Dynamic Workflows も、その思想を複数エージェントへ広げたものとして読めます。
3. これらが**価格据え置き**で提供されている一方、端末コーディングは GPT-5.5 が優位、出力単価も GPT-5.5 が約4割安く、用途で最適解は変わります。

そして本シリーズの物差しを、**能力・信頼性・コスト**の3軸として整えました。今後のリリースは、この物差しで測っていきます。

**——次回 Vol.1 の予告です。** 「信頼性」を公開数値の引用だけで終わらせては、結局スペック表の言い換えになってしまいます。そこで次回は、最も検証の難しいこの軸に絞り、自分の手で確かめます。Claude Opus 4.8・4.7・Sonnet 4.6 に同じ問いを一度だけ投げ、返ってきた主張を一つずつ実在の一次資料に当てて答え合わせします。問いはシンプルです——**正直なモデルは、その代わりに使いにくくなってしまうのでしょうか**。
