ここから下という部分をコピペしてCodexに読み込ませたらOKな仕様にしています。  
ただし、後で出てくるUltra部分の「最適化」に関しては**「最適化（超高速、軽量、安定（かつできる限りキャッシュに持たせて）、高度セキュリティ化）」**と当社の場合は書き換えています。

ちなみにClaude codeは8月1日で一旦解約予定です。fable5のUltracodeで考えてOpus4.8で実装をしてきましたがOpusだと世界で勝てないなって思い、Codex5.6 solのMaxをメインに日々使うことにここ数日で悟り、カルトレインの中で決断しました。

GPT-5.6世代の登場により、AI駆動開発（AI-Driven Development）は新たなフェーズに入りました。

しかし、「どのモードをどう使えばいいのか」「上位モデルで完璧な設計を作ったのに、実装フェーズでなぜかコードが劣化してしまう」と悩む開発者も多いのではないでしょうか。

本記事では、AIのパフォーマンスを限界まで引き出しつつコストを最小限に抑える「Ultra / Max / Medium」の黄金使い分け術と、実装時のデグレードを完全に防ぐプロンプトテクニックを解説します。

## 1. 100点満点の「モデルモード使い分け」

結論から言うと、各モードの役割分担は以下のように定義するのが最も効果的です。

**モード得意な役割採用する理由  
🔴 Ultra**アプリ全体のスクリーニング・構造把握複数エージェントの並列処理で依存関係や競合を横断レビューできるため  
**🟡 Max**バグ修正・高度なロジック設計・開発計画深層思考（ディープ・リーズニング）による脳内シミュレーションが得意なため  
**🟢 Medium**素直なコード実装（UI・定型ロジック）余計な深読みをせず、最速・最安のトークン消費で指示通り動くため

### 🔴 Ultra：アプリ全体のスクリーニング

パッケージ依存関係の調整、肥大化したアーキテクチャの最適化など、「全体を俯瞰して隠れたリスクを炙り出す」作業にはUltraが最適です。バックグラウンドで複数の評価エージェントが並列走るため、人間が見落としがちな依存関係のバグの火種を網羅的に洗い出せます。

### 🟡 Max：高度なロジック・バグ原因の特定・設計

1つのバグの根本原因究明や、複雑なアルゴリズム、マイルストーンの設計には「深層思考（ディープ・リーズニング）」が不可欠です。Maxは「このコードを修正すると他の画面にどう影響するか？」を限界まで思考シミュレーションするため、シニアエンジニア並みの鋭い回答を出力します。

### 🟢 Medium：素直なコードの量産・実装

明確な設計図がある状態でのクラス作成やUI構築では、AIに余計な「深読み」をさせない方がスピードも早くコストも抑えられます。GPT-5.6はMediumでも基礎能力が非常に高いため、指示通りストレートに綺麗なコードを書いてくれます。

## 2. 開発スピードが爆速化する「黄金サイクル」

この3つを組み合わせることで、無駄なトークン消費を抑えながら最高効率で開発を進めることができます。

* **【ステップ1：Ultra】**
* **【ステップ2：Max】**
* **【ステップ3：Medium】**

## 3. 【注意】実装フェーズで「コードが劣化する」理由と対策

上位モデル（Ultra/Max）で完璧な設計書を作ったにもかかわらず、Mediumでコード化する際に「古くて不適切な書き方」や「バグを生む安易な実装」が混入してしまう現象があります。

### なぜコンテキストの先祖返り（デグレード）が起きるのか？

原因はMediumの性能不足ではなく、「文脈（コンテキスト）の引き継ぎ不足」です。

AIはチャットごとに毎回コンテキストを整理して思考します。Maxで作った高度な設計書を「じゃあこれを目安に書いて」と大雑把に渡すと、Mediumは設計の背景や意図を理解できず、自身が持つ一般的な「手癖パターン」でコードを書いてしまいます。

### 解消法：「設計書の凍結（プロンプト・エンジニアリング）」

この落とし穴を防ぐには、Maxで作った設計書を「絶対的なルール」として固定し、Mediumの思考を制限するガイドレールを敷く必要があります。

💡 1. Maxへのプロンプト（設計フェーズ）

設計を出力させる際、最後に「下位モデル実行用の仕様書」を作らせます。

> **Max宛の指示文例:**

「〜〜の設計をMaxの高度な推論で構築してください。出力の最後には、下位モデル（Medium）が余計な解釈を挟まず機械的に1発で完璧な実装コードを出力できるよう、厳密なクラス定義、インターフェース、状態管理のルールをまとめた『実装用仕様書』を出力してください。」

💡 2. Mediumへのプロンプト（実装フェーズ）

Mediumには「考えるな、仕様書をただトレースしろ」という役割（ペルソナ）を与えます。

> **Medium宛の指示文例:**

「以下の【Maxが作成した厳密な仕様書】に従い、指定されたコードのみを実装してください。**注意：あなた独自の推論や、一般的な実装パターンのアレンジは一切不要です。** 仕様書に記載されているアーキテクチャ、命名規則、状態管理のフローを100%厳密にトレースしてコード化してください。【Maxが作成した厳密な仕様書】（ここにコピペ）」

こうすることで、Mediumは余計な下位思考を挟む余地がなくなり、**Max譲りの高品質なコードを爆速・最安コストで出力**できるようになります。

## 4. 迷った時の「最終防衛ライン」

仕様書を固定してもなお不安が残るような「一ミリの妥協も許されない超重要ロジック」については、無理にMediumを使わず、実装まで一貫して「Max」で押し切るのが安全です。

* **Mediumで任せるべきもの**

  + 画面UI（レイアウト、コンポーネント）
  + シンプルなAPI通信ロジック
  + 基本的なバリデーション
* **Maxで一貫して行うべきもの**

## まとめ：AIを「設計者」と「作業者」に役割分担させる

「設計はUltra/Max、実装はMedium」という戦略は、「上位モデルに完璧な仕様書を作らせ、下位モデルには思考ではなく『作業』をさせる」という主従関係を徹底することで真価を発揮します。

このメリハリと連携テクニックを意識して、快適で高効率なAI駆動開発を進めていきましょう！
