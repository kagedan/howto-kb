## セッションが変わるたびに、同じ説明を繰り返している

Claude Codeを使っていると、同じ説明を何度も繰り返していることに気づきます。

議事録の整え方、レポートの構成、ファイルの命名規則。前のセッションでうまくいったやり方を、新しいセッションでまた最初から伝え直す。この手間を減らしてくれるのが、Skillsという仕組みです。

この記事では、Skillsとは何かを簡単に押さえたうえで、自分の仕事に合ったスキルをどのようにつくるかを、2つのアプローチに分けて紹介します。

私自身、これまでに十数以上のスキルを業務に組み込んできました。その経験から、つくり方をこの2つに分けて考えると、迷いにくくなると感じています。

## Skillsの正体は、Claudeに渡す手順書

Skillsは、Claude Codeに特定の仕事の進め方を参照させる仕組みです。

実体はシンプルで、SKILL.mdという名前のMarkdownファイルです。冒頭にスキルの名前と説明、つまり「どのような依頼を受けたときに使うか」を書き、本文に具体的な手順や判断基準を記載します。

このファイルを、ホームフォルダ直下の所定の場所に置くものです。

```
~/.claude/skills/スキル名/
```

するとClaude Codeは、関連する依頼を受けたときにそのファイルを参照し、書かれた内容に沿って作業を進めます。

![](https://assets.st-note.com/img/1784494382-LoNuPVXlCj6g3rOf7hGD9Bxe.png?width=1200)

Skillsがあると何が変わるか（Before／After）

重要なのは、説明文であるdescriptionが、そのスキルを使う条件になることです。

たとえば、「議事録を整えてと言われたら、このスキルを使う」と書いておけば、スキルの正式名称を覚えていなくても、普段の言葉で頼むだけで利用できます。

毎回詳しく説明する必要がなくなるだけでなく、作業の進め方やアウトプットの品質も安定しやすくなります。セッションが変わっても同じ仕事の進め方を再現できるように、業務の手順書を置いておく仕組みと考えると分かりやすいと思います。

![](https://assets.st-note.com/img/1784494410-VW4eCv0TXU9ZlnuQ3r7m2kN8.png?width=1200)

SKILL.mdの中身のイメージ

もうひとつ大事なのは、SKILL.mdを自分ですべて書く必要はないということです。

基本的には、Claude Code自身に書かせます。これから紹介する2つのアプローチも、突き詰めれば、Claudeにどのような過程を経てスキルを書かせるかの違いです。

## つくり方は2つに分けると考えやすい

スキルのつくり方は、大きく2つに分けられます。

ひとつは、つくりたいものの構想から要件を固めていく**設計型**。もうひとつは、目の前の仕事をClaude Codeと一緒にやり切ったあとで、その過程をスキルにする**実践型**です。

順に見ていきます。

![](https://assets.st-note.com/img/1784494432-agr5mQ9SJApyDkG24H6w3lXo.png?width=1200)

2つのアプローチの全体像

## 設計型：ヒアリングさせて、要件の解像度を上げる

つくりたいもののイメージはあるものの、細部までは言葉にできていない。そうしたときに使いやすいのが、Claudeに要件を聞き出してもらう方法です。

たとえば、会議の文字起こしから議事録をつくるスキルが欲しいとします。

この時点で伝えられることは、「文字起こしを渡したら、読みやすい議事録にしてほしい」という程度かもしれません。しかし、この状態ですぐにスキルを書かせると、一般的な議事録の形式ができあがり、自分の実務には少し合わないものになることがあります。

そこで、次のように頼みます。

> 会議の文字起こしから議事録を作るスキルを作りたい。詳細な要件を定義するために、重要と思われる点を私に一問一答でヒアリングして。

すると、Claudeから一つずつ質問が返ってきます。

議事録は誰が読むのか。決定事項と宿題をどのように分けるか。発言者名を残すか。分量はどの程度にするか。雑談や脱線した部分をどう扱うか。

こうした質問に答えていくうちに、自分でも曖昧だった要件が少しずつ明確になります。

たとえば、次のような条件が固まっていきます。

* 読み手は、欠席者を含む部内メンバー
* 決定事項、担当者、期限は表にまとめる
* 発言者名は、意思決定に関係する箇所だけ残す

ここまで具体化できたら、あとは「この内容でスキルにして」と伝えるだけです。

![](https://assets.st-note.com/img/1784494456-7KvoqT1pe5ZsIc4S8P6idRrB.png?width=1200)

設計型の進め方（一問一答ヒアリングの会話イメージ）

この方法の良さは、質問に答える過程で、自分が重視している条件に気づけることです。

頭の中にある完成イメージを、最初から自分だけで整理して書き出すのは簡単ではありません。一問一答であれば、聞かれたことに答えていくだけで、必要な条件が具体化されていきます。

なお、要件を整理する相手はClaude Codeである必要はありません。ChatGPTとの対話で要件を詰め、まとまった内容をClaude Codeに渡してスキル化する、という分担でも問題なく進められます。

## 実践型：一度やり切った仕事を、そのままスキルにする

もうひとつは、最初からスキルをつくろうとしない進め方です。

まずは、目の前にある具体的な仕事をClaude Codeと一緒に完成させることに集中します。

たとえば、毎月の請求書処理を考えてみます。今月分の請求書PDFが20枚あり、そこから支払一覧のExcelをつくりたいとします。

Claude Codeに処理を任せてみると、最初はうまくいかない箇所も出てきます。

スキャン画像になっているPDFは読み取り方を変える必要がある。源泉徴収のある請求書は列を分けたい。出力するファイルの名前も統一したい。

こうした調整を重ね、今月分の支払一覧が完成したとします。

その時点で、次のように伝えます。

> いまやった一連の作業を、来月も同じようにできるようにスキルにして。

Claudeは、このセッションで行った試行錯誤を把握しています。そのため、最終的にうまくいった手順だけでなく、途中でつまずいた箇所や、その回避方法も含めてSKILL.mdに整理できます。

翌月からは、「今月の請求書処理をして」と頼むだけで、前月に固めた進め方を再現しやすくなります。

![](https://assets.st-note.com/img/1784494478-sDYrGno6fi3wRmjIK7vCBbAc.png?width=1200)

実践型の進め方（試行錯誤→完成→スキル化）

実践型の強みは、要件定義を強く意識しなくても、結果として自分の仕事に合った条件が整理されていることです。

実物を完成させるまでの過程で、必要な手順や例外処理を実際に確認しているためです。頭の中だけで要件を考えるのではなく、作業を通じて要件を確かめているともいえます。

横展開もしやすい方法です。請求書処理のスキルができれば、その進め方を参考にして、経費精算や入金確認などの似た仕事にも広げられます。

## どちらから入るか

2つのアプローチの使い分けは、次のように整理できます。

目の前に具体的な仕事がすでにある場合は、実践型が向いています。まず一度完成させ、その過程をスキルにする方が、速く、確実に進められます。毎月や毎週繰り返している作業の多くは、この方法でスキル化できます。

一方、まだ自分でも経験したことのない業務や、アウトプットの形式が特に重要な仕事には、設計型が向いています。決まったフォーマットで納品する資料などは、作業を始める前に要件を固めておいた方が、手戻りを減らせます。

完成イメージが自分の中で曖昧な場合も、質問に答えながら条件を整理できる設計型が役立ちます。

![](https://assets.st-note.com/img/1784494501-DfxY2yFzogOtdrh174pAIkcu.png?width=1200)

使い分けの条件比較

実際には、どちらか一方だけを使うとは限りません。

実践型でつくったスキルを使い続けるうちに、追加したい条件が見えてくることがあります。その段階で設計型のヒアリングを行い、スキルを補強するという流れもよくあります。

## つくって終わりではなく、使いながら育てる

スキルは、一度つくったら完成というものではありません。

実際に使ってみると、想定と違う部分や、追加したい条件が見つかります。そのたびに「いまの修正内容をスキルにも反映して」と頼めば、スキルを少しずつ自分の仕事に合わせていけます。

私が使っているスキルも、最初につくった状態のまま使い続けているものはほとんどありません。使うたびに気づいた点を反映してきたことで、依頼するときの言葉は少しずつ短くなり、求めるアウトプットも安定するようになりました。

Skillsは、高度な自動化だけを目的とした仕組みというより、自分の仕事の進め方を言葉にして残しておく場所だと感じています。

設計型は、仕事を始める前に考えながら言語化する方法です。実践型は、実際に仕事を進めたあとで、その過程を振り返りながら言語化する方法です。入口は異なりますが、自分の仕事の進め方を再利用できる形で残すという点では共通しています。

まずは、毎週または毎月繰り返している仕事をひとつ選び、一度Claude Codeと一緒に完成させたあとで、「この一連の作業をスキルにして」と頼んでみる。そこから始めると、Skillsを使う感覚をつかみやすいと思います。
