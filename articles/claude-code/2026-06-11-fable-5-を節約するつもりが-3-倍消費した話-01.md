こんにちは！ついに昨日、[Claude Fable 5 (Mythos クラス初の一般公開モデル)](https://www.anthropic.com/news/claude-fable-5-mythos-5) が出ましたね！

私も昨日からその性能の高さにワクワクで色々試しています。  
一方で、トークン消費が Opus 4.8 の 2 倍と激しかったり、6/23 からは Claude Pro/Max のサブスクリプション枠内では使えなくなり usage credits での従量課金が必要になることがアナウンスされていたりと、この知性をどんなことに、どうやって使おうか、と考えている人も多いのではないでしょうか？

巷では、

* 「リファクタリングをこの際まとめてやってもらおう！リファクタリングのロードマップをFableに立てさせると良さそう！」
* 「Fableはレビュー役で、コードを書くのはより小さいモデルにやらせるのが良いのでは？」
* 「Fableは指示役で、（以下略）」

みたいな意見をいくつか見かけました。これがうまくいくなら、Fable を節約しつつ上手に使えそうです。

そこで、Fable 5 + Cursor Composer 2.5 Fast を使って検証してみました。

## 先に結論

今回のお題(ゼロからブラウザゲームを 1 本作る)では、**Fable に直接書かせる方が良い**結果でした。その理由も考察してみたので興味のある方は続きをどうぞ！

## 検証内容

「ブラウザで動くスーパーマリオブラザーズ風ゲームを作る」というお題を、まったく同じプロンプトで 2 つの体制に投げました。

* **体制 A**: Fable 5 が直接実装する
* **体制 B**: Fable 5 は指示役(タスク分解・指示書作成・レビュー・統合)に徹し、実装は headless の Cursor CLI で動かす Composer 2.5 Fast(Cursor の高速・低価格モデル)に並列でやらせる

```
ブラウザで動くsuper mario brothersを作成してください。可能な限り原作を再現してください。操作方法はspaceでジャンプ、左右の矢印キーで移動です。
```

### 共通の前提

* 空ディレクトリに `git init` しただけの状態からスタート
* Claude Code (v2.1.170)
* Fable 5 (medium effort)
* 同時刻に起動して並走
* 体制 A はプロンプトをそのまま、体制 B は同じ文面を `/cursor-impl`(後述の自作 skill)付きで投入

### 自作 skill `cursor-impl`

体制 B を場当たり的にやると検証にならないので、手順を skill として固定しました。

📦 skill は[こちらで公開しています](https://github.com/Nasubikun/skills/tree/main/cursor-impl)。`npx skills add Nasubikun/skills --skill cursor-impl` でインストールできます。

ワークフローは 6 ステップです。

1. **Preflight**: working tree の確認
2. **タスク分解**: 担当ファイルが重ならないように分割。共有部分は Fable が自分で書く
3. **指示書(brief)作成**: タスクごとに brief.md を書く
4. **並列ディスパッチ**: headless の `cursor (agent)` CLI(`composer-2.5-fast`)をバックグラウンドで並列実行
5. **レビュー**: Fable が変更ファイルを読み、問題があれば feedback.md を書いて差し戻し(最大 3 ラウンド)
6. **統合**: 全体の検証を回してレポート

委譲していいのは「出力量は多いが判断は少ない」仕事だけ、設計判断が要る変更は Fable が自分で書く、というルールも入れています。

## どう動いたか

体制 A は、単一の `index.html`(1,142 行)をほぼ一発で書き上げ、構文チェックをしてブラウザで開いて終了でした。

体制 B は次のように進みました。

1. Fable がまず共通コード(`index.html`、`package.json`)を自分で書く
2. 「ゲーム本体」「オーディオ」の 2 タスクに分解し、それぞれに 100 行超の brief.md を書いて Composer を 2 つ並列で起動
3. Composer は速く、ゲーム本体(約 2,900 行)を**約 200 秒**で出力
4. オーディオは 1 発 approve。一方ゲーム本体は、Fable のレビューで **9 件の指摘**を差し戻し(うち**起動不能のバグが 5 件**、加えて単色矩形だったスプライトの全面描き直しなど)
5. 修正ラウンド 1 回で収束。軽微な修正は Fable が自分で対応し、最後に headless ブラウザでゲームを操作して検証、PASS を報告

## 結果

まず、この検証の主題である「Fable を節約できたのか？」から。

|  | 体制 A(直接実装) | 体制 B(指示役 + Composer) |
| --- | --- | --- |
| 所要時間 | 6 分 18 秒 | 約 25 分 |
| Fable のトークン消費 | 約 5 万 | **約 14 万** |
| Composer のトークン消費 | — | 入力 11 万 / 出力 3 万 |

**節約するはずだった Fable が、直接実装の約 3 倍のトークンを消費しています。** タスク分解・brief 作成・全ファイル読みのレビューはどれもコンテキストを食う仕事で、「Fable は美味しいところだけ」とはいきませんでした。

そして肝心の出来栄えも、Fable に直接書かせた体制 A の方が圧倒的にいい出来でした。

## 考察

### なぜ体制 B は負けたのか

以下は筆者の考察です。

**① brief の時点で情報が落ちる。**  
体制 A の結果を見る限り、Fable は「マリオらしさ」をかなり高い解像度で持っています。しかし brief に書けるのは、そのうち言語化できる部分だけ。ジャンプの重力の効かせ方や敵を踏んだときの跳ね返りのような感覚は、100 行の brief を書いてもこぼれ落ちます。直接実装なら、このロスがそもそもありません。

**② レビューは「壊れていない」までしか保証できない。**  
言語化できた 9 件の指摘は修正 1 ラウンドでちゃんと直りました。しかし初回出力の品質が低いと、レビューは壊れている箇所を潰すので手一杯で、「良い」まで持っていく余裕がありません。

**③ 手触りをフィードバックで直すのは割に合わない。**  
仮に「ジャンプが軽すぎる」と指摘できても、修正の往復ごとに①のロスが乗ります。この体制は、正しさには収束しても質には収束しません。

まとめると、指示役から実装役に渡るのは言語化できた情報だけで、手触りのような言語化できない部分は伝わらないまま失われます。フロントエンドや UX のような言語化が難しいお題は、特に向いていないといえそうです。

## 検証のツッコミどころ

### レビュー後の修正は Fable がやればいいのでは？

実は軽微な修正は Fable が直しています(今回も「スプライト下端の空行でキャラが浮く」は Fable が自分で修正)。

では大きな修正も Fable がやればいいかというと、それだと Composer の書いた 2,900 行を読み込んで直すことになり、自分で書くのと手間が変わらなくなります。Fable が手を動かすほど、「Fable に書かせない」ことで成り立っていた節約は消えていく。なら最初から Fable に書かせた方がいい気がします。

今回は特に「検証」としてはここの境界を曖昧にすると差が見えづらくなると思い、このように設定しました。

### 実装役をもっと強いモデルにすればいいのでは？

これはありそうです。今回の負け筋には Composer の素の実装力も含まれるので、実装役を GPT 5.5 あたりに替えたら結果は変わるかもしれません。（とはいえ、Composer もそんなに低性能なモデルではないと思うのですが。。）これは今後試したいと思っています。

ただ、GPT 5.5 が使えるなら指示役と実装役を分けず最初から GPT 5.5 に任せればいいタスクも多そうで、結局「強いモデルに直接書かせればよくない？」な気もします。

### お題の設定が特殊では？

「そりゃ既存のコードベースもなくて、1 枚で書き切れるサイズなら Claude が直接書く方がいいよね」と思った方、私もそう思います。実際このお題は、

* 既存コードがないので、守るべき規約をすべて brief に書き起こすしかない
* 合否が「遊んで面白いか」で決まるのに、テストも lint もないので、レビューで拾えるのは「壊れているか」まで
* 1,000〜2,000 行は Fable が一発で書き切れる量

と、直接実装に有利な条件が揃っていました。

## では、delegation が勝てる実務タスクはあるのか？

理屈の上では候補を挙げられます。が、今回の検証を経て、現段階ではどれも確実にそうとも言い切れないな、という感覚です。

1. **参照実装がある反復作業**  
   「この API と同じパターンで残り 10 本」のような仕事なら、brief は「参照実装はここ。同じように」で済む。とはいえ、これは Fable にとっても簡単な部類の仕事で、わざわざ委譲するまでもない。brief を書く方がトークンを食いかねないし、Composer も完全無料ではない
2. **受け入れ基準が機械検証できる仕事**  
   typecheck / lint / テストが揃っていれば、Composer の判断力不足をツールが補える。それでもテストが守るのは品質の下限で、命名や構造といった「次に読む人のための質」には結局差が出る
3. **出力量がボトルネックの仕事**  
   大量の codemod や i18n の文字列外出しなど、判断は軽いが総量が膨大な仕事。  
   ここはちょっと可能性がありそうです。一方で、Delegation を行うべきタスクかどうかの判断を誤ると品質を下げる結果になりますし、そもそも大量変更はそんなに頻繁には起きないはずなのでわざわざこんな仕組みを組むほどではなさそうな気がします。

この形がハマるタスクは一部にはあると思います。ただ「このタスクはハマるか?」を毎回見極めるコスト自体がばかにならず、見誤ったときの損(今回のような品質低下と 3 倍のトークン)も大きい。なので現時点の私の立場は、「一旦全部 Fable に書かせる方がトータルのコスパが良さそう」かなと思います。もちろん、従量課金になれば話は別ですが。。

## おわりに

「Fable に指示させて下位モデルが実装すればいい」は、今回の検証ではうまくいきませんでした。ハマるタスクが一部にあるとしても、それを見極めるコストを考えると、一旦全部 Fable に書かせる方が良さそうかなーという立場です。

とはいえ、これは Fable が出た翌日に「とりあえず試した」n=1 の検証で、実務的な検証はまだまだ途中です。私自身、実コードベースでこれから試していくつもりです。

実際に Fable が従量課金になる頃には全然違うことを思っている予感も沸々としています。それだけ進歩が早いということですね。
