## 1. はじめに

この記事は、自分が個人開発している Roblox ゲーム **Wobble Survival: Don't Fall!**（近日リリース予定）を作る過程で得た知見をまとめたものです。落ちないようにみんなで生き残り、落ちそうな人を助けると高得点、という協力型のサバイバルゲームです。

普段は **Claude.ai で設計し、Claude Code で実装する** というワークフローで開発しています。この記事では、その進め方の全体像と、実際にこのワークフローで「期間別ランキング」を作ったときにハマった `os.date` の週番号問題、そしてその解決策を紹介します。

こんな人に読んでほしい記事です。

* **Claude Code が気になっているゲーム開発者** — 実際のゲーム開発にどう組み込むかの具体例として
* **AIを使って、もっと凝ったゲームを作りたい人** — 設計から実装までAIとどう役割分担するかの一例として

ゲーム自体はまだリリース前ですが、もうすぐ公開します。よければ楽しみにしていてください。それでは本題へ。

## 2. 開発ワークフロー全体像

このゲームは、設計と実装を別のツールに分けて進めています。**Claude.ai が設計判断とMarkdownの指示書づくりを担当し、Claude Code（Cursor）が実装を担当する**、という分業です。

流れはこうです。Claude.ai と設計を詰めて仕様をMarkdownの指示書に落とし、それを Claude Code に渡して実装してもらう。出来上がったものを Roblox Studio で動かして確認し、気づいた点を Claude.ai に戻してまた設計を直す。このループを回し続けています。

ポイントは、**Claude.ai にはコードを書かせず「設計と指示書」に専念させ、Claude Code には「実装」に専念させる**こと。役割を分けることで、設計の議論が実装の細部に埋もれず、逆に実装の試行錯誤が設計判断を汚さずに済みます。

## 3. Claude Codeに課しているルール：「Step 0調査 → 報告 → 承認 → 実装」

AIのコーディングエージェントを使っていて一番ヒヤッとするのは、**こちらの意図を取り違えたまま、いきなり実装を始めてしまう**ときです。前提を1つ勘違いしていると、もっともらしいコードが大量に生成されて、あとから巻き戻すのが大変になります。

そこで自分は、Claude Code に対して必ず次の順番を踏ませるルールを敷いています。

1. **Step 0：調査** — まず関連コードや既存の実装を調べる。この時点では一切コードを書かない。
2. **報告** — 「現状こうなっている」「この修正にはここが影響する」「方針はA/Bが考えられる」と報告させる。
3. **承認** — 自分が報告を読み、方針を承認（または修正）する。
4. **実装** — 承認された方針に沿って初めてコードを書く。

たったこれだけですが、効果は大きいです。実装前に「何をどう変えるつもりか」が言語化されるので、**こちらの認識とズレていればコードが1行も書かれる前に止められる**。手戻りの大半はここで防げます。

実際、後述する `os.date` の問題も、Step 0 の調査報告の中で「この週番号指定子、Luauで本当に動くのか要検証です」という一言が出てきたことで、実装前に地雷の存在に気づけました。指示書の側にも「`%G`/`%V` のLuauサポートは要検証。非対応なら手計算でフォールバック」と先回りで書いておけたのは、この調査フェーズがあったからです。

設計（Claude.ai）と実装（Claude Code）を分けたうえで、実装側に「いきなり手を動かさせない」ルールを足す。これが、個人開発で破綻せずにスピードを出すための芯になっています。

## 4. 題材：期間別ランキングを作りたい

このゲームには、プレイヤーのスコアを競うランキング掲示板があります。最初は「全期間（歴代）」のランキングだけを置いていたのですが、これには弱点があります。**上位が固定化して、新しく始めた人にチャンスが見えない**のです。歴代トップは長くプレイした人で埋まり、後発の人は「どうせ追いつけない」と感じてしまう。

そこで、ランキングに**期間タブ**を付けることにしました。

* **全期間** — 歴代の累計・ベスト
* **今日** — その日のぶんだけで集計（毎日リセット）
* **今週** — その週のぶんだけで集計（毎週リセット）

「今日」「今週」が毎日・毎週リセットされることで、**誰にでも上位を狙えるチャンスが定期的に生まれる**。これは競争のフックとしてかなり強く、デイリーで戻ってくる動機にもなります。

実装方針としては、データを期間ごとに別々のDataStoreへ分けて持つことにしました。「全期間用」「今日用」「今週用」のストアを分け、ストア名に日付や週を埋め込んでおけば、日付や週が変わった瞬間に自動的に別のストアへ書き込まれ、古いストアは放置するだけで自然にローテーションされます（手動の掃除が不要）。

ここで1つ、地味だけれど無視できない問題が出てきます。**「今週」をどう識別するか**です。「今日」はUTCの年月日をそのまま使えば済みますが、「今週」はそう単純ではありません。次の節で、ここでハマった話をします。

## 5. ハマりどころ：`os.date("!%G-W%V")` がLuauで弾かれる

期間別ランキングでは、データを「今日のぶん」「今週のぶん」に分けて集計する必要があります。素直に考えると、集計の単位ごとに**キー文字列**を作って、それをDataStoreの名前に含めて分離するのが楽です（古いストアは放置すれば自然にローテーションされる）。

日付キーは簡単で、UTCの年月日を使えば済みます。

```
local dayKey = os.date("!%Y-%m-%d")  -- 例: "2026-06-03"
```

問題は週キーです。最初は「ISO週番号でいけるだろう」と考えて、こう書こうとしました。

```
-- やりたかったこと: ISO週番号で週キーを作る
local weekKey = os.date("!%G-W%V")  -- "2026-W23" のような文字列が欲しい
```

ところがこれは動きません。**Luauの `os.date` は ISO年 `%G` と ISO週番号 `%V` の変換指定子に対応していない**のです。`%Y %m %d %H %M %S %*t` あたりは使えるのに、`%G`/`%V` は弾かれてエラーになります。

標準のLua/C由来の感覚で `%V` を使ってしまいがちですが、Robloxの実行環境ではここで足を取られます。

## 6. 解決：月曜(UTC)の日付を週キーにする

### 6.1 考え方

そもそも欲しいのは「同じ週なら同じ文字列、週が変わったら別の文字列」になるキーであって、**ISO週番号という形式そのものが欲しいわけではありません**。

だったら、**その週の月曜日(UTC)の日付**をそのまま週キーにすればいい。同じ週の中ならどの曜日に実行しても同じ月曜日を指すので、キーは安定します。週が変われば月曜の日付も変わるので、自然に別ストアへ分離されます。

### 6.2 実装

```
-- 現在時刻（UTC）
local now = os.time()
local t   = os.date("!*t", now)  -- t.wday: 1=日曜 .. 7=土曜

-- ISO方式の曜日番号に変換（月曜=1 .. 日曜=7）
local isoDow = (t.wday == 1) and 7 or (t.wday - 1)

-- その週の月曜に当たる時刻を逆算（時刻は now と同じまま）
local mondayTime = now - (isoDow - 1) * 86400

-- キー文字列を生成
local weekKey = os.date("!%Y-%m-%d", mondayTime)  -- 例: "2026-06-01"
local dayKey  = os.date("!%Y-%m-%d", now)          -- 例: "2026-06-03"
```

あとは、このキーをDataStoreの名前に埋め込んで分離します。

```
Leaderboard_week_2026-06-01
Leaderboard_day_2026-06-03
```

`os.date("!*t", now)` の `!` はUTCで時刻テーブルを取る指定で、返ってくる `wday` は「日曜=1 .. 土曜=7」。これを「月曜=1 .. 日曜=7」のISO方式に直してから、`(isoDow - 1)` 日ぶんを秒数（`86400`）で引きます。ここで引くのは「丸ごとの日数」なので、`mondayTime` は厳密には0:00ではなく **その週の月曜の「`now` と同じ時刻」** を指します。ただし最終的に使うのは `%Y-%m-%d` で取り出す日付部分だけなので、週キーとしては月曜の日付が正しく得られれば十分です（時刻のズレは結果に影響しません）。もし0:00きっかりのタイムスタンプ自体が必要なら、`now` から `t.hour`・`t.min`・`t.sec` を引いて当日0:00に正規化してから日数を引けば得られます。

### 6.3 なぜこの方式が破綻しないか

* **週番号の計算が要らない** — ISO週番号は年初の扱いなどでロジックが地味に面倒ですが、月曜の日付方式ならその計算自体が不要です。
* **年跨ぎの週でも壊れない** — 12月末〜1月始めにまたがる週も、「その週の月曜の日付」という1つの基準で一意に定まります。
* **日数の引き算が正確** — UTCには夏時間(DST)が無いので、1日 = 86400秒が常に成り立ちます。ローカルタイムで同じことをやるとDSTの切り替え日に1時間ズレる危険がありますが、UTC基準なら起きません。

ISO週番号という「形式」にこだわらず、「同じ週で安定する一意なキー」という**要件**に立ち返ったことで、シンプルかつ頑健な実装に落とせました。

## 7. このワークフローの効きどころと限界

半年ほどこのやり方で開発してきて、効くところと、まだ人間がやるしかないところがはっきりしてきました。

**効きどころ**

* **実装はもうほとんど任せられる。** コードを書く部分の大半はAIで十分です。バグが出ても、的確な指摘ポイントを渡したりプロンプトを工夫したりすれば、うまく回せる。「自分で1行ずつ書く」フェーズは、もうかなり減りました。
* **Step 0で方向をすり合わせると、完成イメージがズレない。** ここが想像以上に大きい。すり合わせずに進めると、最終的にどんなものになるかのイメージが湧かないまま手が動いてしまったり、気づいたら違う方向に進んでいたりします。先に方向を合わせてから始めると、自分の意図がちゃんと通る。しかも、すり合わせの議論の中で\*\*「それならこうした方が良いのでは」という、最初は考えていなかった案や発見が出てくる\*\*のが面白いところです。AIを"実装係"だけでなく"壁打ち相手"として使えている感覚があります。
* **手戻りが激減する。** Step 0の調査・報告・承認を挟むことで、認識のズレがコードになる前に潰せます。os.date問題のように「実装前に地雷に気づく」ことが何度もありました。

**まだ人間がやるしかないところ**

* **実際に遊んで確かめること。** AIはゲームをプレイできません。落ちるタイミングの手触り、行ける場所・行けない場所、操作した時の感覚——こういうものは、自分で実際に動かして確かめるしかない。仕様書の上では正しくても、遊ぶと違う、というのは普通に起きます。
* **視点が固定化したときに、別の角度から突っ込むこと。** AIと長く詰めていると、お互い同じ方向・同じ前提に固まってしまって、抜け落ちに気づけなくなることがあります。「そもそもこれ、別の見方をすると違うのでは?」と引いて指摘するのは、人間の役割だと感じています。設計判断の責任を最後に持つのは、結局こちら側です。

## 8. まとめ／次回予告

この記事では、個人開発のRobloxゲームで使っている「Claude.aiで設計 → Claude Codeで実装」というワークフローと、その中で踏んだ `os.date` の週番号問題を紹介しました。

要点をまとめると：

* 設計（Claude.ai）と実装（Claude Code）を、ツールごとに役割分担する
* 実装側には「Step 0調査 → 報告 → 承認 → 実装」を踏ませて手戻りを防ぐ
* `os.date` の `%G`/`%V` はLuau非対応 → ISO週番号という「形式」を捨て、「その週の月曜(UTC)の日付」という要件ベースのキーに切り替えて解決した

最後の `os.date` の話は、「形式にこだわらず要件に立ち返る」という、AIと組むときにも効く考え方の具体例でもありました。

次回は、同じゲームで遭遇した **「DataStoreの書き込み予算が枯渇した話 ― 書き込みキューで捌く」** を書く予定です。少人数プレイでも意外と早くDataStoreの予算は尽きる、という運用フェーズのリアルな話です。
