Claude Codeは更新が速く、設定の正解がすぐ古くなります。この記事では2026年5月時点で現役の設定だけを選び、最短セットアップ、本当に使う設定ファイル、廃止された落とし穴、そして設定を活かす開発フローまでを一気に整理します。

## なぜ「結局なにを設定すればいい？」になるのか

Claude Codeは毎週のように新機能が増えます。便利な反面、少し前の解説記事に書かれた設定が、もう名前を変えていたり削除されていたりします。ネットの情報を真似たのに動かない、という場面は珍しくありません。

つまずきの正体は、機能が多すぎることではなく、「いま現役のものはどれか」が分かりにくいことです。だから最初の一歩は、全部を覚えることではなく、現役の最小セットに絞ることです。

ここでは2026年5月下旬の公式ドキュメントで確認した内容をもとに、迷いどころだけをまとめます。

## 結論：最小セットアップ＋3つのファイル＋自己改善する流れ

先に全体像です。やることは三段階だけです。まず数分で動かす。次に設定ファイルを三つだけ押さえる。最後に、設定が活きる開発フローを回す。

設定ファイルは、権限やモデルを決めるsettings.json、AIに渡す前提メモのCLAUDE.md、繰り返し作業をまとめるスキルの三つが軸です。これだけで土台は十分に整います。

## まずは5分で動かす：最短セットアップ

専門知識が浅くても、ここまでは数分で終わります。

![Claude Codeの最短セットアップ4ステップ（インストール・ログイン・/init・/model）を示す手順図](https://assets.st-note.com/img/1779755375-lpx7dsS0y9hkU2vHciJFQutE.png)

1. **インストール**。今はパッケージをダウンロードするネイティブインストーラが推奨です。MacやLinuxならターミナルでinstall.shをbashに流す一行、WindowsはPowerShellでinstall.ps1を実行します。HomebrewやWinGetでも入ります。ネイティブ版は自動更新されるので、古い設定に悩まされにくいのも利点です。
2. **ログイン**。claudeと打って起動すると、ブラウザで認証を求められます。一度通せば次回から不要で、入り直したいときは入力欄で「/login」。
3. **CLAUDE.mdを作る**。プロジェクトのフォルダで起動し、「/init」と打つだけ。コードベースを読んで、AIに渡す前提メモのたたき台を自動で作ってくれます。
4. **モデルを選ぶ**。「/model」で一覧から選びます。詳しくは後ろで触れます。

ターミナル以外にも、Web版やデスクトップアプリ、VS Code・JetBrainsの拡張から同じように使えます。

## 2026年5月に“現役”の設定ファイル

土台になるのは次の三つです。深い網羅より、まずここを押さえます。

### settings.json（権限・モデル・環境変数）

動作の中心になる設定ファイルです。同じ項目を複数の場所に書けて、狭い範囲ほど優先されます。優先順位は高い方から、管理者設定、コマンドライン引数、このリポジトリ専用のローカル設定、チーム共有のプロジェクト設定、自分の全体設定の順です。

![settings.jsonの優先順位ピラミッド（管理者＞コマンドライン＞ローカル＞プロジェクト＞ユーザー、権限はマージでdeny最優先）](https://assets.st-note.com/img/1779755378-YDgP2IL0EZWVipcMdoO5Frmy.png)

最初に触ると効くキーは多くありません。permissionsで許可・確認・拒否のルールを決め、modelで既定モデル、envで全セッション共通の環境変数、hooksで節目に動く処理を指定します。チームで共有したいものはプロジェクト設定へ、自分の手元だけのものはローカル設定へ。なお権限ルールだけは上書きではなく全範囲が合算され、拒否は常に最優先になります。

### CLAUDE.md（AIに渡す前提メモ）

プロジェクトの約束ごとや前提を書いておくと、毎回説明しなくても踏まえてくれます。広い設定から狭い設定へと連結して読まれ、上書きではなく積み重なります。先頭に@を付けた記述で別ファイルを取り込むこともでき、長くなったらトピック別に分けた専用フォルダへ逃がせます。

2026年に入って、AI自身が学びをメモに溜める自動メモリも標準で動くようになりました。1ファイルは200行ほどに抑えるのが目安です。

### スキル（繰り返し作業をまとめる）

よくやる手順は、スキルとしてまとめておくと「/名前」で呼び出せます。置き場所はプロジェクトなら.claude/skillsの下、自分専用なら同じ構成をホーム側に作ります。最小構成はSKILL.mdが一枚あれば十分です。

ここで一つ重要な変更があります。以前のカスタムコマンド（.claude/commandsの下）は今も動きますが、新しく作るならスキルにまとめる形が推奨に変わりました。設計のコツは前編の記事で詳しく扱っています。

## つまずきポイント：廃止・名称変更された設定

ここが今いちばん混乱しやすい部分です。古い記事のとおりに設定して動かない原因の多くは、ここにあります。

![2026年5月に変わった設定・名称の対応表（古い→今）。output-styleコマンド削除や docs URL移転、attribution、MCPのHTTP化など](https://assets.st-note.com/img/1779755381-jEM2p4g0wfPUruHDFx8tLKOZ.png)

確認できた主な変更はこうです。

* **公式ドキュメントの場所が移りました**。古いアドレスは新しいcode.claude.comへ転送されます。リンクを保存しているなら更新を。
* **出力スタイルの誤解に注意**。「/output-style」というコマンドは削除されましたが、出力スタイルの機能そのものは健在です。「/config」の画面か、設定キーのoutputStyleで今も使えます。「廃止された」と早合点しないことです。
* **コミット署名の設定名が変わりました**。includeCoAuthoredByは古い名前で、今はattributionです。
* **MCPの接続方式**。SSEという古い接続は非推奨で、HTTPが推奨です。
* **MCPのスコープ名**。以前のprojectは今のlocal（既定）、以前のglobalは今のuserに変わりました。
* 環境変数で小型モデルを指定していた古いキーも、新しい名前に置き換わっています。

一方で、CLAUDE.local.mdは「廃止された」という噂がありますが、公式では現役です。誤って消さないようにしてください。

## 設定が活きる開発フロー：4つの原則

設定を整えたら、最後はそれを活かす進め方です。実務で固まった四つの原則を挙げます。

![設定を活かす開発フロー4原則（分けすぎない・レビューゲート・検証の自動化・自己改善ループ）の2x2図](https://assets.st-note.com/img/1779755382-6S0kX5BsxzL1mfnP7ohT8wFE.png)

**分けすぎない**。サブエージェントは便利ですが、親の文脈を引き継がないため、状況把握に余計な時間とトークンがかかり、指示の取りこぼしも起きます。基本はメインエージェントに一貫して作業させ、明確に文脈を混ぜたくないときだけ分けます。

**レビューゲートを置く**。仕上げる前に、レビュー役のエージェントを別に立てて点検させます。Must・Should・Couldで優先度を分けるMoSCoW法が便利で、最大三回まわし、Mustがゼロになったら抜けます。本人に自己採点させると甘くなるので、レビューは分離するのがコツです。観点も「読みやすさ」「正確さ」のように分けて指定すると、指摘が鋭くなります。これは人のレビューと同じ理屈です。

**検証方法を先に用意する**。品質の低い成果物を人が確認するのは時間の無駄です。AI自身が確かめられる手段を先に持たせます。表示の確認なら自動操作ツール、コードならlinterの実行手順、「30件送る」なら送信済みリストを作らせて件数を数えるスクリプト。基準を満たすまで終われない仕組みにすると、人は最後の確認に専念できます。

**自己改善ループを回す**。作業のあとに自分の動きを振り返らせ、次に活きる形でスキルの設定を更新させます。ただし一度きりの出来事をそのままルールにすると、かえって質が落ちます。「これは一般化する価値があるか」を一段挟むのが肝心です。この考え方は前編で詳しく書きました。

## まとめ：現役の最小セットから始める

更新の速さに振り回されないコツは、現役の最小セットに絞ることです。

* 数分で動かす：インストール、ログイン、/init、/model
* 設定はsettings.json・CLAUDE.md・スキルの三つを軸に
* 廃止・名称変更（出力スタイルのコマンド削除、attribution、MCPのHTTP化など）を踏まないようにする
* 設定を活かすフロー：分けすぎない、レビューゲート、検証の自動化、自己改善ループ

まずは/initでCLAUDE.mdを作るところから始めてみてください。あなたが「これは入れてよかった」と思う設定があれば、ぜひコメントで教えてください。スキルそのものの設計は前編の[Claude Codeスキル設計｜2026年最新のベストプラクティス](https://note.com/modern_tech/n/naefca5c8c0aa)で詳しく扱っています。
