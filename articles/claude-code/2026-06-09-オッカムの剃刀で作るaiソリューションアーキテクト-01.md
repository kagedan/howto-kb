先端技術開発グループ（WAND）の小島です。前回の記事で、AWSが2026年5月に発表した「[Agent Toolkit for AWS](https://aws.amazon.com/jp/about-aws/whats-new/2026/05/agent-toolkit/)」を試してみました。今回はこれを使って、**Claude CodeでAWSのアーキテクチャ図（draw.io形式）を作成する**方法を紹介します。

<https://zenn.dev/exwzd/articles/20260601_agent_toolkit_aws_1>

ただ単に作らせるだけだと、要素を盛りすぎた「何をしたいのかわからない図」になりがちです。そこで本記事では、**Claude Codeのスキル**（**Skills**）に「**オッカムの剃刀**」（引き算的思考）と「**不明点は聞き返す**」という2つの原則を組み込み、人間が描くような要点を絞ったアーキテクチャ図を作る`solution-architect`スキルを設計していきます。Agent Toolkit for AWSのレビューと組み合わせる使い方も紹介します。

## Claude Codeにアーキテクチャ図は作成できるのか

結論からいうと**可能**です。検証で使ったClaude Opus 4.7（1Mトークン）に、

> 3層アプリケーションのAWSアーキテクチャ図をdraw.io で作って。Well-Architected に沿うようにして

というと、以下のようなアーキテクチャ図が生成されます。

![](https://static.zenn.studio/user-upload/deployed-images/aded2ebf6f860da9145d8f17.png?sha=104a4c563ccf76b525d4e70304e092b2585c7f9b)  
*実際に生成されるのはdraw.ioのYAMLファイルで、それをVSCodeの拡張機能でPNGレンダリングしたものを掲載*

以前はDraw.io MCPなどを併用する例もありましたが、LLMの性能向上により、これらの**MCPがなくても十分生成できるように**なっています。

Claudeの素の回答はLLMの事前知識に基づくもので、AWSの公式情報に基づくベストプラクティスを反映したものではありません。そこで以下のようなフォローアップの質問を入れます。

> これをAgent Toolkit for AWS Plugin を使ってレビューして修正して

すると、以下のようにパワーアップしたアーキテクチャ図になります。

![](https://static.zenn.studio/user-upload/deployed-images/a16967a8ba3c9b942416bf5a.png?sha=ff2448ca56568a1c74888a7f2b6cfbeae5acf947)  
*プラグインのレビュー後のアーキテクチャ図。ADOT（AWS Distro for OpenTelemetry）のサイドカーをあえて書くのは初めて見ました*

## AIによるアーキテクチャ図生成の課題

ここで大きな課題が見えてきました。「**何をしたいアプリなのかが、生成された図からわからない**」という点です。

例えば、3Dモデル×工場オンボーディングの例では、人間（AWSの方）が作ったアーキテクチャ図は以下のようになります。

![](https://static.zenn.studio/user-upload/deployed-images/88480bfccd9f60e341ffd733.png?sha=8416dd9a7b45de128824a99f1dbcd7c36b896bdd)  
*[以前の記事（AWS Summit 2025）](https://zenn.dev/exwzd/articles/20250715_aws_summit_builders_fair)からの再掲、もとは[B-042A 産業用 AI の仲間 関連資料](https://pages.awscloud.com/summit-japan-2025-doc-download.html?id=B-042A)　より（現在リンク切れ）*

人間が作ったアーキテクチャ図は、何がやりたいかが明確で、程よく削ぎ落とされた情報になっています。これを見れば展示されているアプリケーションについて会話できる状態です。

つまり、**LLMは要素の足し算は得意だが、やりたいことを引き算的に抽出し、ポイントのみを描画させる作業は、まだ人間のほうが得意**な領域であるということがわかります。

こういった発想をAIに取り入れられないのか、というのが次なる工夫です。

## オッカムの剃刀

「引き算的思考」については、ITや科学界隈でも古くからよく引用される指針があり、**オッカムの剃刀**（Occam's razor）と呼ばれます。これは14世紀の哲学者・神学者であるオッカム村のウィリアムにちなみます。

![](https://static.zenn.studio/user-upload/deployed-images/2f20a39b23816a10bb45c4a4.jpg?sha=e3f4a2cf49648360ef2b40ca909970b2ef504dd8)  
*画像は[Wikipedia](https://ja.wikipedia.org/wiki/%E3%82%AA%E3%83%83%E3%82%AB%E3%83%A0%E3%81%AE%E5%89%83%E5%88%80)から*

なぜLLMの作成したアーキテクチャ図が複雑になってしまったのかというと、

* **要件に必要のない仮定をおきすぎた**
* **そもそも要件をきちんと聞いていなかった**

というのが大きな理由かと思われます。特に前者に効くのが「オッカムの剃刀」です。

今回の例は「3層アプリケーションを作ってほしい」「3Dモデルで工場のオンボーディングアプリケーションを作ってほしい」と、ふわっと指示した形でした。要件を明確化しないまま進めたため、どこまでが本当に必要な仮定なのかを判断できず、結果として不必要な仮定をおいてしまった——というのが筆者の仮説です。

後者については「不明点は聞き返せ」と明示的に指示するようにします。これをClaude Codeで毎回やらせるには、**スキルとして定義しておくと便利**なので、次はオッカムの剃刀と聞き返しを組み込んだスキルを設計します。

## オッカムの剃刀をスキルに組み込む

Claude Codeなどのコーディングエージェントには、**スキル**（Skills）という仕組みがあり、特定の操作や処理を自然言語ベースで再利用可能な形で導入できます。

スキル自体もClaude Codeから作ることができます。上記の2点を加味して、`solution-architect`というスキルを作ってみましょう。

solution-architectのスキルの実装

```
---
name: solution-architect
description: ユーザー要件に基づいたアーキテクチャ図(draw.io形式)を作成する。要件が不明な点は必ず聞き返し、AWS Well-Architected Framework に沿った設計を行う。アーキテクチャ設計、構成図作成、サービス選定の相談に使用する。
---

# Solution Architect

## 原則：オッカムの剃刀

> 「必要が無いなら多くのものを定立してはならない。少数の論理でよい場合は多数の論理を定立してはならない。」— オッカム

**要件を満たす最も単純な構成が最良の構成。** 想像上の将来要件や技術的好みで複雑化しない。

## プロセス

### 1. 要件を聞く（不明点は必ず質問）

設計判断に影響する不確実性は `AskUserQuestion` で聞き返す。憶測で進めない。確認対象の例：

- ユースケース・想定ユーザー
- 規模(req/s、データ量)、可用性(SLA)、レイテンシ要件
- 予算感、既存環境、コンプライアンス要件

すべて聞く必要はない。**設計が分岐する論点のみ**聞く。

### 2. Well-Architected Framework に沿って設計

6つの柱を念頭に、要件に応じて濃淡をつける（過剰設計しない）：
運用上の優秀性 / セキュリティ / 信頼性 / パフォーマンス効率 / コスト最適化 / 持続可能性

### 3. draw.io 形式で図を出力

`.drawio` (mxGraph XML) ファイルとして書き出す。AWS公式シェイプ (`shape=mxgraph.aws4.*`) を使う。

最小テンプレート：

\```xml
<mxfile host="app.diagrams.net">
  <diagram name="Architecture" id="arch1">
    <mxGraphModel dx="800" dy="600" grid="1" gridSize="10" guides="1" tooltips="1" connect="1" arrows="1" fold="1" page="1" pageScale="1" pageWidth="1169" pageHeight="826" math="0" shadow="0">
      <root>
        <mxCell id="0"/>
        <mxCell id="1" parent="0"/>
        <!-- ノード例: Lambda -->
        <mxCell id="lambda1" value="Lambda" style="sketch=0;points=[[0,0,0],[0.25,0,0],[0.5,0,0],[0.75,0,0],[1,0,0],[0,1,0],[0.25,1,0],[0.5,1,0],[0.75,1,0],[1,1,0],[0,0.25,0],[0,0.5,0],[0,0.75,0],[1,0.25,0],[1,0.5,0],[1,0.75,0]];outlineConnect=0;fontColor=#232F3E;gradientColor=none;fillColor=#ED7100;strokeColor=#ffffff;dashed=0;verticalLabelPosition=bottom;verticalAlign=top;align=center;html=1;fontSize=12;fontStyle=0;aspect=fixed;shape=mxgraph.aws4.resourceIcon;resIcon=mxgraph.aws4.lambda;" vertex="1" parent="1">
          <mxGeometry x="200" y="200" width="78" height="78" as="geometry"/>
        </mxCell>
        <!-- エッジ例 -->
        <!-- <mxCell id="e1" style="endArrow=classic;html=1;" edge="1" source="lambda1" target="ddb1" parent="1"><mxGeometry relative="1" as="geometry"/></mxCell> -->
      </root>
    </mxGraphModel>
  </diagram>
</mxfile>
\```

よく使うAWSシェイプ：`resIcon=mxgraph.aws4.<service>` の形で、例: `lambda` / `s3` / `dynamodb` / `api_gateway` / `cloudfront` / `ec2` / `rds` / `sqs` / `sns` / `eventbridge` / `cognito` / `cloudwatch`。

### 4. 設計の根拠を示す

各コンポーネントについて1行で：「**なぜ採用したか／代替を退けた理由／どの柱に効くか**」を明記する。

## 出力フォーマット

\```
## アーキテクチャ概要
（1〜3文）

## 構成図
（.drawio ファイルとして出力／パスを提示）

## コンポーネントと選定理由
- <Service>: <役割> — <理由 / Well-Architected との対応>

## トレードオフ・前提
- （明示しておくべき前提や切り捨てた要件）
\```

## アンチパターン

- 要件確認なしの「とりあえずベストプラクティス」
- 「将来○○かも」を理由にした複雑化（YAGNI）
- 6つの柱を均等に作り込む過剰設計
- 流行りのサービスを並べる、要件起点でない設計
```

このスキルのポイントは「オッカムの剃刀」の**原文和訳をスキルに引用している点**です。

> 「必要が無いなら多くのものを定立してはならない。少数の論理でよい場合は多数の論理を定立してはならない。」— オッカム

また、設計判断で迷う部分は`AskUserQuestion`**で聞き返せ**と指示しています。`AskUserQuestion`はClaude Codeに組み込まれたツールで、コンソール上の対話型ダイアログでユーザーに質問するものです。

## スキルを試してみる

### 3層アプリケーションの例

> solution-architectのSkillsを使い、testディレクトリに3層アプリケーションのアーキテクチャ図を作成して

としてみます（Skillsの使用を明示しなくてもおそらく起動するはずです）。

![](https://static.zenn.studio/user-upload/deployed-images/245859fe7cbfa80d0bc620b1.png?sha=863d46317125cdd43ea698a1d466d3ec6212e6eb)

![](https://static.zenn.studio/user-upload/deployed-images/a83d200a2f36b72a7f845adb.png?sha=e927eb6aa2e61ff486c8ac5f9343e126e4bcd08b)

`AskUserQuestion`のおかげで、対話ベースで要件のヒアリングが進んでいきます。アプリ種別、規模・SLA、DB種別という一般的な内容を聞いてきますね。

生成されたアーキテクチャ図が以下の通りです。

![](https://static.zenn.studio/user-upload/deployed-images/f40376f8fb503c17d6480407.png?sha=3a83e273db27ca65804296d51f0f22e92e30f044)

一気にシンプルになり、人間にとって馴染みやすいものになりました。

ここで興味深いのが、「Well-Architected Frameworkに沿うようにしろ」とスキルで指示しているのに、**可用性を捨ててシングルAZ**にしている点です。要は「社内向けの小規模なツールなら、可用性は意識しなくてよく、むしろコストとシンプルさを取ってEC2で良い」という**人間っぽいトレードオフ思考を実現できている**というわけです。

また、これをAgent Toolkit for AWSでレビューさせると以下のアーキテクチャ図になりました。

![](https://static.zenn.studio/user-upload/deployed-images/8cd291b84e1ad13d17ba9d9c.png?sha=02247e7ec00747d91675b24c45f5f19cc6794005)

開発時に、Session Managerで接続できないためVPCエンドポイントを追加すべきというフォローはナイスです。しかし、このレビューは相変わらず足し算の方向に向かうようです。

### 3Dモデル×工場オンボーディングの例

同様に「3Dモデル×工場オンボーディング」の例でも、スキルを使ってアーキテクチャ図を作成してみます。

![](https://static.zenn.studio/user-upload/deployed-images/aa8e6d231edca516f3c36a6d.png?sha=637965b37beac46c18877f895594b4eb21df6324)

![](https://static.zenn.studio/user-upload/deployed-images/1533b442bf23ce2785f5db07.png?sha=0a666b74164c3791e11f73c1fee1a6d746f5eeee)

こちらも`AskUserQuestion`でヒアリングしてきましたが、規模といった一般的な内容だけでなく、**3D描画方式、LMS機能、外部連携といったアプリケーション特有の要件までヒアリング**してきました。この質問の柔軟性は素晴らしいですね。

![](https://static.zenn.studio/user-upload/deployed-images/7a4bc9f2e0087fe55c2a1b99.png?sha=bacaac712677f6ad4643a5c7ea2b3a7f0dc3fb00)

作成されたアーキテクチャ図は、AppStream 2.0を基軸としたものになりました。以前のアイコンが大量に並んだ図と比べると、何をしたいアプリなのかがだいぶわかりやすくなっています。

![](https://static.zenn.studio/user-upload/deployed-images/bae71114acd90863ddd840ec.png?sha=44736cb935d6ee830cde34f9558d7ec67e73b3a2)

Agent Toolkit for AWSのレビューを次に入れると、エラーハンドリングや可観測性・セキュリティ・コストが大きく増えています。

こちらも相変わらず足し算のレビューになっており、「GameLift Streamsを使え」といった構成を根本から見直すレビューにはなっていません（`AskUserQuestion`で要件を別の方向に分岐させていれば、初手で選ばれていた可能性はあります）。

### プラグインのレビューで過剰な足し算をさせないようにするには

あくまでアイディアレベルですが、ここまで見たように、初手のアーキテクチャはシンプルでも、プラグインによるレビューで過剰に盛られてしまう傾向があります。

そこで、`solution-architect`スキル自体に処理を2段階に分けさせ、後段フェーズでAWSプラグインのレビューをかけたうえで「オッカムの剃刀の原則に反していないか」をリフレクションさせながらリファインすれば、シンプルさを維持できるのではないかと考えています。

## まとめ

この記事では、Claude CodeのスキルとAgent Toolkit for AWSを活用して、外部MCPなしでdraw.io形式のアーキテクチャ図を作成する方法を紹介しました。そして、古くからの指針である「オッカムの剃刀」をスキルに組み込むこと、そして不明点をユーザーに聞き返すことが、より要点を絞った人間らしいアーキテクチャ図を作るうえで有効だと確認できました。
