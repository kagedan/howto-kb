## はじめに

僕は Claude Code のアカウントを2つ持ってます。仕事用と個人用です。

仕事用は会社の契約で使ってますが、個人用は自分で別途契約したもの。今回は [Anthropic の Built with Claude 4.7 Hackathon](https://cerebralvalley.ai/e/built-with-4-7-hackathon) に参加するため、個人アカウントを使いたかったんです。ハッカソンで作るものは完全に個人の成果物なので、仕事用アカウントのプロジェクトメモリやログに混ざるのは避けたかったです。

最初に考えたのは「ログアウト→ログイン」の繰り返しでした。でもすぐに諦めました。仕事とハッカソンを並行して進めると、1日に何十回もアカウントを切り替えることになりますよね。さすがに現実的じゃないです。

### そこで思いついた解決策：作業ディレクトリごと分けてみよう

ログイン切り替えが無理なら、**アカウントごとに Claude Code の作業ディレクトリ自体を分けてしまえばいい**——ここに辿り着きました。

Claude Code の全設定・全データは `~/.claude/` に集約されてるので、このディレクトリを2つ作って使い分ければ、アカウントごとの状態が物理的に独立します。ログイン情報を含むセッション (`.claude.json`) もディレクトリの中にあるため、ディレクトリを切り替える = アカウントを切り替える、ということになります。

### なぜこの解決策が最善なのか

Claude Code はデフォルトで `~/.claude/` に全データを入れてます:

- アカウント情報・セッション（`.claude.json`）
- ルールファイル・カスタム指示（`rules/`、`CLAUDE.md`）
- プロジェクトごとのメモリ（`projects/`）
- フック・スクリプト（`hooks/`）
- スキル・プラグイン（`skills/`、`plugins/`）
- 会話履歴・コスト記録

これをアカウントごとに丸ごと別ディレクトリに分けると、ログイン切り替えなしに**アカウントごとの完全な独立環境**が作れます:

- 仕事用には仕事のコンテキストとルールだけ
- 個人用にはハッカソンのコンテキストと自分好みの設定だけ
- スキルやプラグインも用途に応じて別々に有効化できる
- コスト・使用量もアカウントごとに独立して記録される

`CLAUDE_CONFIG_DIR` という環境変数でディレクトリを切り替えられるので、シェルエイリアスと組み合わせれば**1コマンドで環境ごと切り替わります**。ログイン切り替えより圧倒的に速いです。

ただ——「完全に分離できてる」と思ったら、実はそうじゃなかった、という落とし穴がありました。それが本記事の本題です。

### 結果として起きたこと

設定ディレクトリを分離した後も、分離が**不完全**で、毎ターン大量のコンテキストが無駄にシステムプロンプトへ注入されてました。ここではその原因と、実際に取った対処を書いておきます。

---

## セットアップの全体像

```
~/.claude/              # 旧デフォルト（移行前はここにすべてがあった）
~/.claude-personal/     # 個人用アカウント
~/.claude-work/         # 仕事用アカウント
~/.claude.bak/          # 移行後、~/.claude をリネームしたバックアップ
~/.claude.json          # HOME直下 — CLAUDE_CONFIG_DIR に関係なく常に読まれる
```

```bash
# ~/.zshrc に追加したエイリアス
alias cc-work='CLAUDE_CONFIG_DIR=~/.claude-work claude'
alias cc-personal='CLAUDE_CONFIG_DIR=~/.claude-personal claude'
```

---

## 核心的な問題：`CLAUDE_CONFIG_DIR` は完全に分離しない

これが一番の誤算でした。

`CLAUDE_CONFIG_DIR=~/.claude-work` を指定してても、Claude Code は **`~/.claude/` をフォールバックとして引き続きスキャンします**。つまり両方のディレクトリからファイルを読み込んでるんです。

### 実際にシステムプロンプトに現れた証拠

同じルールファイルが、2つの異なるラベルで2回注入されていました:

```
Contents of ~/.claude-work/rules/...    ← "user's private global" として注入
Contents of ~/.claude/rules/...         ← "project instructions" として注入（ラベルも間違い）
```

内容は完全に同一です。同じルールファイルを毎ターン2回払ってたことになります。

---

## 毎ターン注入されていたコンテキストの構造

実測したところ、ユーザープロンプトやツール出力の前に、以下のような層のコンテキストが毎ターン注入されていました:

- **ルールファイルの二重注入**: `~/.claude/` と `~/.claude-<account>/` で同じ内容が重複
- **使わない言語向けのルール**: 初期状態でインストールされていた他言語向けの翻訳ルール群（自分が使わない言語なら完全に無駄）
- **大量のスキル説明文**: 有効化されてるプラグインが提供する全スキルの説明が一覧として展開される
- **プロジェクト `CLAUDE.md`**: プロジェクト直下の指示書。情報を詰め込みすぎると肥大化する
- **auto-memory システム前文**: メモリシステムの使い方説明
- **各種フック注入テキスト**: 有効化した拡張機能が毎ターン追加するテキスト

これらが積み重なって、**毎ターン100KB以上**がユーザーの入力より前に消費されていました。

---

## プラグイン肥大化の仕組み

プラグインを有効化すると、そのプラグインが提供する**全スキルの説明文**が「利用可能なスキル」ブロックに自動的に追加されます。1プラグインで数十〜百超のスキルを含むものもあるため、複数有効化すると簡単に100超のスキル説明が毎ターン展開されます。

使わないスキルでも、有効化されてる限りトークンを消費し続けます。

---

## 取った対処

### 1. デフォルトディレクトリをリネーム（最大の改善）

```bash
mv ~/.claude ~/.claude.bak
```

これでフォールバック読み込みがなくなって、ルールの二重注入が解消されます。

`.bak` は念のためバックアップとして残し、しばらく安定稼働してから削除する想定です。もしリネーム後もフォールバックが戻ってくるなら、`CLAUDE_CONFIG_DIR` が永続的に設定されてない可能性があります:

```bash
grep CLAUDE_CONFIG_DIR ~/.zshrc ~/.bashrc ~/.zshenv
```

---

### 2. 使わない言語向けのルールを削除

インストール時に付属していた、自分が使わない言語向けの翻訳ルール群を両アカウントから削除しました。

```bash
rm -rf ~/.claude-work/rules/<使わない言語>/
rm -rf ~/.claude-personal/rules/<使わない言語>/
```

削除前に、他のファイルから `@import` や設定参照で依存されてないことを確認するのが大事です。

---

### 3. フックのパスを修正（両アカウント）

`~/.claude/` をリネームした後、Bash ツールを使うたびにこんなエラーが出るようになりました:

```
PreToolUse:Bash hook error — No such file or directory
```

原因は `settings.json` のフックパスが**旧ディレクトリを絶対パスで指したまま**だったからです。

```json
// 修正前（両アカウント共通の誤り）
"command": "~/.claude/hooks/some-hook.sh"
```

```json
// 修正後（各アカウントスコープに変更）
// ~/.claude-work/settings.json
"command": "~/.claude-work/hooks/some-hook.sh"

// ~/.claude-personal/settings.json
"command": "~/.claude-personal/hooks/some-hook.sh"
```

statusline のコマンドパスも同様に修正が必要でした。

**ポイント: フック・スクリプト内のパスは絶対パスで書くのが鉄則です。ディレクトリを分けるなら、パスも分けたディレクトリを指すようにしないと静かに壊れます。**

---

### 4. プラグインレジストリのパスを修正（両アカウント）

リネーム後、一部のスキルが消えて、片方のアカウントのプラグインレジストリがリセットされました。

原因は両アカウントの `plugins/installed_plugins.json` と `plugins/known_marketplaces.json` が**旧デフォルトディレクトリを参照したまま**だったからです:

```json
// 修正前
"installPath": "~/.claude/plugins/cache/..."
"installLocation": "~/.claude/plugins/..."
```

全エントリを各アカウントのスコープに修正しました。片方はレジストリが破損していたため、既知のプラグイン構成から手動で再構築しました。

---

## 残課題（手動で対処予定）

- **プラグインの棚卸し**: 実際に使ってないプラグインを `/config` で無効化。スキル説明文がトークン消費の主要因になるため、使う物だけを残す
- **プロジェクト `CLAUDE.md` の縮小**: 詳細は別ファイル（`plan/` 等）に切り出し、CLAUDE.md 自体は最小限のポインタだけにする
- **ルートに置いた大きなドキュメントの見直し**: アカウントディレクトリ直下に大きな `.md` を置くと自動で読み込まれる可能性があるので、参照されてないなら移動または削除

---

## 検証チェックリスト

新しいセッションを起動して、以下を確認してみてください:

- `~/.claude/rules/...` のコンテンツがシステムプロンプトに現れない（リネームで消えるはず）
- 削除した言語向けルールディレクトリが現れない
- 最初の Bash ツール呼び出しで `PreToolUse:Bash hook error` が出ない（パス修正済み）
- 有効化したはずのスキルが available skills 一覧に表示される（レジストリ修正済み）

---

## 診断コマンド集

```bash
# ディレクトリのサイズ確認
du -sh ~/.claude-*/* | sort -hr

# ディレクトリ間の差分確認（分離前の状態調査に便利）
diff -rq ~/.claude ~/.claude-work

# 現在有効な config dir
env | grep CLAUDE_CONFIG_DIR

# rules ディレクトリの合計バイト数（肥大化チェック）
cd ~/.claude-work/rules && find . -name "*.md" -exec cat {} + | wc -c

# ~/.claude.json のキーごとサイズ（上位15件）
python3 -c "
import json, os
d=json.load(open(os.path.expanduser('~/.claude.json')))
sizes={k:len(json.dumps(v)) for k,v in d.items()}
for k,s in sorted(sizes.items(),key=lambda x:-x[1])[:15]: print(f'{s:>8} {k}')
"
```

## まとめ

`CLAUDE_CONFIG_DIR` による分離は**不完全**です。環境変数を設定してても、Claude Code はデフォルトの `~/.claude/` をフォールバックとしてスキャンし続けます。2アカウント構成でこのデフォルトディレクトリを放置すると、ルールファイルが毎ターン二重注入されてトークンコストが倍になります。

完全に分離するには、**デフォルトディレクトリを物理的に削除またはリネームする**必要があります。`CLAUDE_CONFIG_DIR` だけじゃ足りないです。

移行で実際に壊れた順番:

1. フックパス（即座にエラーが出るので気づきやすい）
2. プラグインレジストリ（スキルが静かに消えるので気づきにくい）
3. ルール二重注入（システムプロンプトを見ないと気づけない）

3番目が一番コスト高く、一番気づきにくい問題でした。
