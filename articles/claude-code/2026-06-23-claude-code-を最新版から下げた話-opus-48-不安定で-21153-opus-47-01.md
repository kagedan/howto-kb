Claude Code の最新版で動かしていた Opus 4.8 (1M context) があまりに不安定だったので、CLI を `2.1.153` に固定し直しました。モデルも Opus 4.7 (1M context) + reasoning effort = xhigh に揃え直しています。

「最新最強モデルをあえて一段下げる」判断と、その途中で踏んだ `PATH` の罠を、実体験ベースで残しておきます。

## TL;DR

* 私の環境で `claude-opus-4-8[1m]` が不安定だった (具体症状5つを後述。`git` 関連 env の自走探索 + 自己 injection 判定が最小構成でも再現したのが決定打)
* codex (gpt-5.4 + web 検索, read-only) に GitHub issues / CHANGELOG / npm 履歴を横断調査させ、安定版を **`2.1.153`** と判定
* ただし同一マシンに **3つの install 経路** (system / npm-global / native installer) が混在しており、PATH でどれが勝つか把握しないとバージョン切替が反映されない
* `claude install 2.1.153` で `~/.local/bin` に固定 → 起動バナーで `v2.1.153 / Opus 4.7 (1M context) with xhigh effort` を確認

## なぜ「一段下げる」判断をしたか

### 最新最強モデルが常に正しいわけではない

普通、新しいモデルは前世代を上回る前提で動きます。ベンチも改善し、対応する文脈長も増えます。だから「最新最強」を選ぶのが合理判断、と無意識に思いがちです。

でも実運用では、最新版が「一番効く」とは限りません。

ベンチ上の性能と、実セッションでの安定性は別軸の指標です。同じ Opus 系でも、release 直後の最新版は **CLI 側の互換性・ツール呼び出し・auto mode の停止条件**などで thrash しがちです。そこに 1M context の重さが乗ると体感はさらに悪化します。

最新最強を一段下げて、こなれた組み合わせで運用安定を取る。負け犬の判断に見えるかもしれませんが、エージェント運用の現実コストに対する素直な落とし所だと私は捉えています。

…と書くと格好いいですが、本音は「動かないツールに毎日キレるのが嫌になっただけ」です。

### 私のセッションで起きていた不安定 (具体症状5つ)

`claude-opus-4-8[1m]` を使っていて、私が継続的に観測した症状はこの5つです。

1. **別の AI のレスポンスを指示文と誤認する**  
   codex などサブツールの出力を、ユーザー指示として処理しようとする。tool call の境界が崩れる。
2. **auto mode 中に何度も実行が止まる**  
   本来 1 tool call で済む作業の途中で停止し、出力が文字化けする。中断 → 再開のループに入りやすい。
3. **ハルシネーションが増える**  
   一次ソースを Read 済みのファイルから、存在しない仕様を引用してくる頻度が上昇。
4. **`git push` すらスキル定義どおりに通らない**  
   skill の手順を素直に踏まず、独自に逸脱して失敗で止まる。
5. **指示していない環境変数の自走探索 + 自己 injection 判定**  
   `git` 関連の env をなぜか勝手に取りに行こうとする。hook でブロックされたから事故にはならなかったが、本人 (モデル) が独自に「prompt injection ではないか」と自己判定して挙動を変えていた。**MCP も skill も入れていない最小構成**でも同じ挙動を再現したので、CLI とモデルの組み合わせ自体に原因がある、と私は判断しました。

どれも単発ならモデル個体の揺らぎで済みます。ただし **5つが同時多発する**と作業効率は桁で落ちます。1セッションで「最後まで完走する」確率が体感で大きく下がっていました。

特に最後の **env 自走探索 + 自己 injection 判定** は、最小構成でも再現したことで「これは CLI 側 (= バージョン) も絡んでいる」と踏み切る決定打になりました。だから「CLI を下げると治る可能性」を試す価値があります。

## 安定版の特定 — codex に GitHub issues を読ませる

### 調査を AI に委譲する使い方

「どのバージョンが安定か」は、人間が GitHub issues を全部読んで判定するには時間がかかりすぎます。だから **codex (gpt-5.4 + web 検索, read-only sandbox)** にこの調査自体を委譲しました。

調べさせた内容は3点です。

* GitHub issues: 「Opus 4.8 unstable / crash / hang / model selection / 4.7 / 4.6 / downgrade / effort / regression」で横串
* `CHANGELOG.md`: バージョンごとの fix 内容
* npm registry: バージョン履歴

read-only + sandbox で走らせる前提なので、ファイル書き換えや実行系の事故は構造的に起き得ません。issue 横断調査は AI に向く仕事の典型例だと思います。

起動コマンドの形は概ねこんな雰囲気でした (実行時の prompt 本文は割愛)。

```
codex exec --sandbox read-only --search-web \
  "..." </dev/null
```

`</dev/null` を明示クローズしないと、`codex exec` を background で呼んだとき stdin 待機で永久 hang します。ここは別記事ネタになる罠です。

### バージョン別シグナル (codex 出力)

codex が返してきたシグナル表を、原文ニュアンスを保って整理します。

| バージョン | 安定性シグナル | Opus 4.7/4.6 選択可? | 主な根拠 |
| --- | --- | --- | --- |
| `2.1.152 - 2.1.153` | **安定 / 静か** | yes (推定) | `2.1.152` で「model/login 切替後にセッションが固まる」修正。報告クラスタなし |
| `2.1.156 - 2.1.157` | 混在 | yes | `2.1.156` で Opus 4.8 の thinking block 改変による API エラー修正、`2.1.157` で端末フリーズ + `/model` ピッカー修正 |
| `2.1.181 - 2.1.186` | **修正多発フェーズ / 不安定** | 混在 / 自動アップグレードの方向 | startup freeze / stream stall / Ctrl+C 無反応 / schema-loop hang / sleep 後 parse error などの fix が連発。**`2.1.183` で「要求 model が deprecated / 自動更新」警告追加** |
| `2.1.111 - 2.1.112` | 不安定 | yes | 2026-04-17 のクラスタ: [#49763](https://github.com/anthropics/claude-code/issues/49763) crash regression / [#49821](https://github.com/anthropics/claude-code/issues/49821) Opus 4.7 既定クラッシュ (回避 `--model claude-opus-4-6`) / [#49809](https://github.com/anthropics/claude-code/issues/49809) effort 無視 |

結論: **`2.1.153` を pin**。フォールバックは `2.1.152`。

ここで一つファクト境界を明示しておきます。codex の confidence は **medium** であり、「Opus 4.7/4.6 の選択を最初に削除したバージョン」は断定できる一次ソースを取得できていません。確認できているのは、**`2.1.183` 以降は deprecated model の自動アップグレード化の方向に動いている**、というところまでです。

### タイムライン — いつ何が壊れたか

時系列で並べると、CLI が荒れた瞬間が見えてきます。

* **2026-04-16 〜 04-17**: `2.1.111 - 2.1.112` のクラッシュ回帰クラスタ。Opus 4.7 ロールアウトで既定がクラッシュし、`--model claude-opus-4-6` 回避が広まる
* **`2.1.152`**: 「session が model/login 切替後に固まる」修正
* **`2.1.156`**: Opus 4.8 の thinking block 改変による API エラー修正
* **`2.1.157`**: 端末フリーズ + `/model` ピッカー修正
* **`2.1.181 - 2.1.186`**: startup freeze / stream stall / Ctrl+C 無反応 / schema-loop hang / sleep 後 parse error などの fix 連発
* **`2.1.183`**: 「要求した model が deprecated / 自動更新」警告追加。古い 4.7/4.6 を素直に選びにくい方向

`2.1.153` は **`2.1.156` 以降の Opus 4.8 系修正フェーズに入る直前** の "静かなバージョン" にあたります。Opus 4.8 を避け、4.7 / 4.6 に倒して運用したいなら、現実的な底はここです。

## 落とし穴 — 同一マシンに複数の install 経路が混在していた

ここが今回の記事の核です。バージョンを下げる前に、**そもそも自分が今どの claude を呼んでいるか**を確定させる必要がありました。

### system / npm-global / native installer の三つ巴

`which claude` してみると、私のマシン (Ubuntu) には3つの経路が見えてきました。

| 経路 | 実体パス | バージョン | 所有 |
| --- | --- | --- | --- |
| **system 全体 install** | `/usr/bin/claude` → `/usr/lib/node_modules/.../claude.exe` | `2.1.172` (248MB native build) | root |
| **npm global** | `~/.npm-global/bin/claude` | `2.1.186` | user |
| **native installer** | `~/.local/bin/claude` | (未インストール時はなし) | user |

ややこしいのは、**どれが PATH で勝つか**を把握していないことです。ログインシェルで `which claude` を打って初めて事実が見える、という状態。私の場合 `~/.npm-global` 側の `2.1.186` は PATH に乗っておらず、**実は死蔵していた**こともこの確認で初めて判明しました。

### PATH でどれが勝つかを見失う罠

複数の install 経路があると、何が起きるか。

* `npm install -g @anthropic-ai/claude-code@<v>` を走らせても、**`/usr/bin/claude` (system 側)** が PATH 先頭にいる限り反映されない
* `claude --version` で確認しても、最初に解決された system 側の数字が出るだけ
* 「上げたつもり / 下げたつもり」がシェル再起動まで気づけない

根本治療は **PATH の先頭に自分が管理できる場所を置くこと**です。`~/.local/bin` を `.profile` で PATH 先頭に積んでおけば、そこに置いたバイナリが常に勝ちます。

## ダウングレード実施

### `claude install 2.1.153` で `~/.local/bin` に固定

実施したコマンドはこれだけです。

`claude install` は native installer 経由で `~/.local/bin/claude` にバイナリを置きます。sudo 不要・root 領域に触らない、というのがポイントです。`.profile` がこの dir を PATH 先頭にしている前提なら、次回ログイン時から `2.1.153` が勝ちます。

確認:

```
$ which claude
~/.local/bin/claude
$ claude --version
2.1.153
```

ログインシェルで再確認しても `2.1.153` が解決される状態に到達しました。

### 旧 install をシャドウするだけで残す判断

`/usr/bin/claude` (`2.1.172`) と `~/.npm-global` (`2.1.186`) は **削除せず**、PATH でシャドウするだけにしました。

理由は2つあります。

* system 側 (248MB) は root 権限が必要で、apt で管理されているなら uninstall は別途検討
* 何かあったときに fallback として残しておきたい

シャドウだけで意図通り動く、というのが Unix の PATH 設計の良いところです。**消さずに勝たせる**ほうがロールバックの選択肢が残ります。

(`sudo rm` を打つのが単純に怖かった、という不純な動機が半分くらい混じっているのは内緒にしておきます。)

### 永続化と「既存セッション」の罠

ここまで読んで「で、これは永続するの? 別ターミナルでも反映されるの?」が気になるはずです。私も気になりました。

整理するとこうです。

* `claude install` が置いた `~/.local/bin/claude` は **本体 (symlink → `2.1.153` 実体) なので永続**。再起動しても残る
* `.profile` の `~/.local/bin` を PATH 先頭に積む分岐は **Ubuntu 標準テンプレに元から書かれている**。書き換え不要だった
* つまり**新規ログインで開くセッション**はすべて 2.1.153 が勝つ

罠はここからです。

| 状況 | claude バージョン | 対処 |
| --- | --- | --- |
| **新規ログイン** (新しく SSH/ターミナルを開く) | ✅ 2.1.153 (自動) | 不要 |
| **既に開いていた既存シェル** (tmux/screen/長寿ターミナル) | ⚠️ 古い PATH のまま、system 版を呼ぶ可能性 | 下記ワンライナー |

既存シェルでは、PATH が **インストール前に評価された値**として環境変数に焼かれています。そのまま `claude` を打つと `/usr/bin/claude` (system 側) を呼びにいきます。「インストールしたのに反映されない」と混乱する典型パターンです。

直し方:

```
hash -r              # シェルのコマンド位置キャッシュをクリア
# または
exec bash -l         # 自分自身をログインシェルとして再起動
# または
source ~/.profile    # PATH を再読み込み
```

確実なのは **そのターミナルを一旦閉じて開き直す**こと。それだけで切り替わります。

### `cron` / `at` / `systemd-run` から呼ぶときの注意

もう一つ覚えておきたいのが、**非対話的環境**の話です。

* `cron`
* `at`
* `systemd-run` (`--user` 含む)
* CI ランナー

これらは `.profile` も `.bashrc` も読みません。PATH は `/usr/bin:/bin` といった最小セットしか持たないので、`~/.local/bin/claude` を解決できません。

将来 cron や at で `claude` CLI を叩くスクリプトを書くなら、どちらかが必要です。

* **フルパス指定**: `$HOME/.local/bin/claude` (cron は `HOME` env を自動設定する。確実を取るなら `/home/<user>/.local/bin/claude` のように実機の絶対パスを直書きしてもよい)
* **スクリプト先頭で PATH 拡張**: `export PATH="$HOME/.local/bin:$PATH"`

人間が手で打つときには見えない罠ですが、自動化に組み込んだ瞬間に再発します。私は今回の publish スクリプトでは `claude` CLI を呼んでいないので影響を受けませんでしたが、伏線として記録しておきます。

### 追記: その伏線、自分で踏んでいました

…と偉そうに書いた**直後**に、自分の `harness-ops` (毎日の記事生成・観測を回す自前 cron 群) を `grep` してみました。結果がこれです。

```
$ grep -rc "claude -p" harness-ops/scripts/*.sh
# 30+ 箇所ヒット
```

そして、それらすべての呼出が:

* `--model` **未指定** (= CLI デフォルト = 不安定の本丸 **Opus 4.8** を引いていた)
* `--effort` **未指定** (= デフォルト effort)
* crontab / 各スクリプトに `PATH` 設定**なし** (= `/usr/bin/claude` 2.1.172 system版を引いていた)

つまり、今朝対話セッションで切替えた「**`2.1.153` + Opus 4.7 + xhigh effort**」という設定は、**自動運用 cron 側には1ミリも反映されていませんでした**。harness が落ちまくっていた本当の原因はこれです。

「自分は publish で `claude` CLI を呼んでないから影響ない」と書いた数分後に、grep の結果を見て凍りつきました。

#### `claude_run` ラッパーで3層 pin

`core/lib.sh` に共通ラッパーを置き、`scripts/` 配下の `claude -p`  を `claude_run`  に `sed` 一括置換しました。

```
# core/lib.sh
# 1. CLI バージョン pin (cron は .profile を読まないため明示)
export PATH="$HOME/.local/bin:$PATH"

# 2-3. モデルと effort のデフォルト (.env で上書き可)
CLAUDE_MODEL_DEFAULT="${CLAUDE_MODEL_DEFAULT:-claude-opus-4-7}"
CLAUDE_EFFORT_DEFAULT="${CLAUDE_EFFORT_DEFAULT:-xhigh}"

claude_run() {
  local prompt="$1"
  shift
  claude -p "$prompt" \
    --model "$CLAUDE_MODEL_DEFAULT" \
    --effort "$CLAUDE_EFFORT_DEFAULT" \
    "$@"
}
```

各スクリプトでは引数なしで呼べます。追加引数 (`--allowedTools` 等) はそのまま転送されます。

```
# Before
RESULT=$(claude -p "$PROMPT" --allowedTools Read,Write,Edit,Grep,Glob,Bash 2>&1)

# After
RESULT=$(claude_run "$PROMPT" --allowedTools Read,Write,Edit,Grep,Glob,Bash 2>&1)
```

検証:

```
$ bash -c 'source core/lib.sh; claude_run "Reply with exactly: pong"'
pong
```

5秒で `pong` が返り、**CLI バージョン (2.1.153) / モデル (4.7) / effort (xhigh) の3層全部が cron 環境でも揃った**ことを確認しました。

「自分は影響受けない」と書いた数分後に、自分が一番派手に嵌まっていました。記事のために罠を語ろうとした側が、書いている最中に罠の主役を更新する展開は、今後ちょっと笑い話として擦っていこうと思います。

## 学び

### 最新最強より一段下げて凪を取るのは合理判断

「最新版を選ばないこと」に最初は抵抗があります。新機能・性能向上・モデル進化を取り損ねる気がします。

でも実運用では、CLI 側の安定性とモデル側の安定性は別軸で動きます。**両方の足並みが揃った「凪のバージョン」** を選んだほうが、エージェント運用の実コストには見合います。

### issue クラスタの横断調査は AI に向く仕事

GitHub issues の数百件を読んでクラスタを特定する作業は、まさに AI に委譲すべき仕事の典型です。

ポイントは3つ。

* **read-only sandbox** で起動して書き換え事故を構造的に排除
* web 検索を有効化して issues / CHANGELOG / npm registry を横断
* 「クラスタ vs 単発の anecdote」を区別させる prompt

私の場合は `codex` でこれを走らせましたが、別の CLI でも同じことが可能です。**人間がやる必要のない調査は委譲する**、というスキルセットとして残ります。

…と書くと優雅ですが、要は「issue を 1件ずつ全部読むのが面倒だった」だけです。

### 複数 install 経路は PATH の地雷原になる

最後の学びは Unix 基礎に戻ります。

* `npm install -g`
* `apt install`
* native installer (`claude install` 等)
* `pipx` / `cargo` / `brew` など

これらの install 経路は、**それぞれ異なる prefix にバイナリを置きます**。同じ CLI が3経路で入ったとき、PATH の解決順がすべての挙動を決めます。

`which claude` と `claude --version` をペアで打って、想定通りのバイナリが勝っているかを確認する。地味ですが、これが Claude Code に限らず全 CLI 運用の基本作法です。

…と偉そうに書きましたが、私はこれを怠ったせいで「インストールしたつもり」を半日ほど続けました。基本作法を語る資格があるのか自分でも怪しい。

### 補論: 直書きでも同じ問題は解ける

cron 側の pin は今回 `claude_run` ラッパーを立てましたが、振り返ると **`sed` の置換パターンを変えるだけ** でも済む話でした。

```
# 今回採用したラッパー方式
sed -i 's/claude -p /claude_run /g' scripts/*.sh

# 直書き方式 (関数定義なし)
sed -i 's/claude -p /claude --model claude-opus-4-7 --effort xhigh -p /g' scripts/*.sh
```

直書きだと `lib.sh` のラッパー関数も不要で、`grep` するだけで「どのモデルで動いているか」が呼出行に明示されます。

| 項目 | ラッパー方式 (今回採用) | 直書き方式 |
| --- | --- | --- |
| `lib.sh` の追加行数 | 約25行 (関数定義 + 変数) | 1行 (PATH 拡張のみ) |
| モデル変更時の手間 | `lib.sh` 1行を直す | 30+箇所すべて書換 (`sed` で一括可) |
| `.env` での上書き | domain 別に切替可 | 不可 (script に埋込まれる) |
| 可読性 | 関数の中身を見に行く必要あり | 呼出行を見れば全部分かる |

私がラッパーを選んだのは、Sonnet 4.6 への切替や effort 調整を将来やるかもしれないという**変更耐性**側に倒したからです。逆に、**「とりあえず今動けばよい」が最優先なら直書きで十分**、というのが正直なところです。
