AIにWordPressのフォームを修正してもらおうとしたとき、最初に気づいたことがあります。

「Contact Form 7のフォームを修正して」と頼むと、AIは詰まります。

設定がデータベースに入っているので、AIには見えない。スクリーンショットを渡して、管理画面の操作手順を説明して、AIが返してきた手順を見ながら自分で操作する。これはAIとの「協働」ではなく、AIを「検索エンジン代わり」に使っているだけです。

フォームがコードになっていないと、AIエージェントにとって「見えない壁」になります。

---

## AIエージェントのコストは2種類ある

AIを使った開発のコストを考えるとき、2つの軸があります。

**① トークンコスト**  
入出力のトークン数。長い説明を与えるほど、複雑な出力が返るほど、コストが上がります。

**② スキルコスト**  
AIに状況を説明する手間。「このGUIツールの、このボタンを押して、この設定を変えて…」という説明は、AIには伝わりにくく、人間側の説明コストが高くなります。

GUIフォームビルダーは両方のコストが高いです。DBに入った設定はAIから見えないので、スキルコストで「スクリーンショット＋操作手順の説明」が必要になる。出力も「手順書」になるので、トークンコストも増えます。

フォームがコードであれば、両方が下がります。

---

## フォームがコードだとコストが下がる理由

HXFEではフォームをPHPの配列で定義します。

```
$schemas['contact'] = [
    'id'      => 'contact',
    'to'      => 'admin@example.com',
    'subject' => 'お問い合わせ: {name}',
    'fields'  => [
        [ 'key' => 'name',    'type' => 'text',     'label' => 'お名前',    'required' => true ],
        [ 'key' => 'email',   'type' => 'email',    'label' => 'メール',    'required' => true ],
        [ 'key' => 'message', 'type' => 'textarea', 'label' => 'メッセージ', 'required' => true ],
        [ 'key' => 'hp',      'type' => 'honeypot' ],
    ],
];
```

このスキーマをAIに渡して「電話番号フィールドを追加して」と頼むと：

```
[ 'key' => 'tel', 'type' => 'tel', 'label' => '電話番号' ],
```

1行のコードが返ってきます。

**入力（スキルコスト）：** スキーマを渡すだけ。操作手順の説明不要。  
**出力（トークンコスト）：** 差分が明確な1行のコード。手順書ではない。  
**レビュー：** `git diff` で確認できる。何が変わったか一目瞭然。

このサイクルが、AIとの協働コストを下げ続けます。

## HXFEのエコシステムがさらにコストを下げる

「HXFEを初めて使うAI」でも正確なコードを出せる仕組みがあります。

```
hxfe-code-first-forms/
├── llms.txt          ← AIエージェント向けのAPI仕様サマリー
├── ai-reference.md   ← スキーマキー一覧・使用例・注意点・lint警告一覧
└── CLAUDE.md         ← Claude Code向けの開発コンテキスト
```

これらのファイルをAIコーディングツールが読み込むことで、HXFEのスキーマ仕様を事前に把握できます。

**llms.txt** はAPIサマリーです。フィールドタイプ・スキーマキー・バージョン履歴がまとめられています。

**ai-reference.md** はより詳細な仕様書です。使用例・lint警告の一覧・注意点が書かれています。AIがコードを生成するとき、よくある間違い（`subject`の書き忘れ・honeypotフィールドの省略など）を事前に避けられます。

**CLAUDE.md** はClaude Code向けの開発コンテキストです。「このプロジェクトは何か・どういう方針で開発するか」が書かれています。

これらがあることで、AIへの説明コストがさらに下がります。「HXFEのスキーマに電話番号を追加して」と言うだけで、正確なコードが返ってきます。

---

## エラーログもAIが読める形で設計した

v1.4.0でエラーログ機能を追加しましたが、ここもローコスト運用を意識しました。

SMTPエラー・Webhookエラー・reCAPTCHAエラーをデータベースではなく `wp-content/hxfe-logs/` のテキストファイルに記録します。

```
[2026-06-13 10:00:00] SMTP_ERROR | form:contact | wp_mail() failed — To: admin@example.com | Subject: お問い合わせ
[2026-06-13 10:01:00] WEBHOOK_ERROR | form:contact | HTTP 500 | URL: https://hooks.zapier.com/...
```

このログをAIに渡して「このエラーの原因と対処法を教えて」と頼めば、すぐに回答が返ってきます。

データベースに入っていたら、SQLで取り出してAIに渡す手間が発生します。テキストファイルなら、そのままコピーして貼るだけです。

「AIが読める形でエラーを記録する」という設計が、運用コストを下げます。

---

## ローコストで運用できるエコシステム

HXFEを使った開発・運用のサイクルを整理すると：

**開発時：**

1. スキーマをAIに渡す（数十行のPHPコード）
2. 「このフィールドを追加して」「この条件分岐を変えて」と指示
3. 返ってきたコードをレビューして適用
4. `git diff` で変更を確認

**運用時：**

1. エラーログをAIに渡す
2. 「このエラーの原因を教えて」と聞く
3. 対処法をコードで返してもらう

**バージョンアップ時：**

1. changelogをAIに渡す
2. 「このバージョンの変更点に合わせてスキーマを更新して」と指示
3. 差分を確認して適用

全てのフローで「コードを渡す → コードが返ってくる」という一貫した流れになっています。GUIツールのように「画面を説明する → 手順が返ってくる → 自分で操作する」という迂回が発生しません。

---

## 「AIと一緒に開発する」前提でツールを選ぶ

2026年現在、AIエージェントを使った開発が当たり前になりつつあります。

この前提でWordPressのエコシステムを選ぶと、自然な答えが出てきます。

* **フォームはコードで定義する** → AIが直接触れる
* **設定はDBではなくファイルで持つ** → AIが直接読める
* **エラーはテキストで記録する** → AIに渡しやすい

HXFEはこの方向で設計したWordPressフォームプラグインです。AIエージェントとの協働を前提に置いたとき、「PHPの配列でフォームを定義する」という選択がいかに合理的かが、使っていくうちにわかってきます。

---

## AIサービスへの依存リスクを下げる設計

2026年、AIを取り巻く環境は急激に変化しています。

* Claude・ChatGPT・GeminiなどのAIサービスが**突然プランを変更・値上げ**する
* Claude Fable 5など最新モデルが**国や地域によって利用制限**を受ける
* 「AIをフル活用して開発・公開したけど、AIサービスの変更で**更新ができなくなった**」という事例も出始めている

AIに依存した開発フローは、AIサービスの変化に振り回されるリスクがあります。

HXFEはこのリスクに対して、構造的な答えを持っています。

**フォームの定義はPHPコードです。AIサービスが変わっても、フォームは動き続けます。**

* Claude → ChatGPT → Gemini → どのAIに乗り換えても、PHPの配列はそのまま渡せる
* AIサービスが使えなくなっても、既存のフォームは一切影響を受けない
* llms.txt・ai-reference.mdはAIサービスを問わず読める標準的なテキストファイル

「AIと協働できる」ことと「AIに依存しない」ことは矛盾しません。HXFEはその両立を目指した設計です。
