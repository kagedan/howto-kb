> Claude Code で「使うほど賢くなる」自己成長サブエージェントを作り、それを本気で評価・チューニングした記録です。  
> 題材はレビューを手伝うエージェントですが、ここで書く教訓は **自己成長エージェント全般** に効くはず。用途やドメインの詳細は省き、**評価とチューニングの方法論**だけを書きます。

## 作ったもの

コードレビューや仕様相談を手伝うサブエージェントを作りました。たとえば PR を投げると「この処理、過去に似たデグレを出した観点が漏れてませんか？」と返ってくる。仕様の相談をすると、関連ドキュメントを裏取りしたうえで「ここ、既存仕様と矛盾します」と指摘してくれる——そんなレビュー相棒です。

せっかくなら使うほど賢くなってほしいので、特徴は **自己成長**。レビューや振り返りで「これは再発しそうだ」と判断した教訓を、ナレッジベース（以下 KB）へ構造化して溜めていきます。

実装は **Claude Code のサブエージェント機能**。定義はフロントマター（`name` / `description` / `tools` / `model`）付きの1枚の Markdown で、ここに性格・手順・動作モードを書きます。実物はこんな骨格です（中身は一般化しています）。

```
---
name: review-buddy
description: コードレビュー・仕様相談・テストレビューを手伝う自己成長エージェント
tools: Read, Grep, Glob, Bash, Edit, Write
model: sonnet
---

あなたはレビューの相棒。一次資料とコードで裏取りしてから答える。

## 動作モード（mode → action）
- コードレビュー  → review-checklist と bugs-and-regressions を読み、実コードを読む
- テストレビュー  → test-gaps を読み、実コードを読んで観点漏れを出す
- 仕様相談       → spec-gotchas を読み、一次資料で即答（実コードは要求時のみ）

## 自己成長プロトコル
再発しうる教訓を見つけたら、KB に「再利用できる原則」として追記する。
一過性・環境固有のものは追記しない。
```

以降に出てくる「プロンプト」はこの定義ファイル、「KB をロードする」は各モードが起動時に読むファイル（`bugs-and-regressions` など）のこと、と思って読んでください。3層で構成しています。

* **プロンプト（定義）** … 性格・手順・動作モードを書いた1枚の md（上のやつ）
* **自己成長 KB** … エージェント自身が育てる知識（過去の不具合・仕様の落とし穴・テスト漏れを、レビュー観点リストへ昇格）
* **外部の一次資料** … 仕様やドキュメントなど、自分では書き換えない正本

![自己成長サブエージェントの構成図](https://static.zenn.studio/user-upload/deployed-images/b3973ba3817993abe29c963c.png?sha=ab9fe4b41540a22e54a2842408c6db9a1849438a)

で、動くものはわりとすぐできました。問題はここから。

## 「動いた」と「効く」は別物だった

「いい感じに指摘してくれる」では、まだ信用できません。本当に役立つのか、ちゃんと確かめたい。

そこで、**お手本問題と採点基準を先に用意して、エージェントの回答を毎回その物差しで採点する**ことにしました。「この質問にはこう答えてほしい」という模範問題集（ゴールデンタスク）と、「実コードを読んだか」「専門外をちゃんと断ったか」といった観点ごとの採点表（ルーブリック）。あとは同じ質問を条件違いで投げて、結果を数えるだけ。

この地味な実測に振り切ったら、作っている時は「まあそうだろう」と思っていた前提が、けっこう外れていく。直感に反した発見がいくつも出てきた——というのが、ここから先の話です。

## 1. 過去のバグで試しても、賢さは測れなかった

最初にやった評価がこれです。実在の過去バグについて「このテスト計画、考慮漏れある？」と聞いたら、エージェントは見事に漏れを指摘しました。賢いじゃないか……と思った瞬間に気づきます。

**その真因も再発防止も、もう KB に書いてあります。** このエージェント自身が過去に溜めたものだからです。つまり測れているのは「未知の漏れを推論で見つける力(detection)」ではなく「KB を検索する力(recall)」でしかありません。自分が書いた答えを自分で引いているだけ。原理的に賢さは測れていないのです。

なので題材を変えました。**KB にまだ無いネタ**を選び、その論点が KB に無いことを grep で確認してから、ヒントなし（コールド）で投げます。すると、まだバグになっていない考慮漏れ（例：あるバッチ処理機能で「件数が多い時のページング境界」「途中失敗時のリトライとべき等性」「処理中に参照元が変わった場合」）をちゃんと自力で挙げました。これが本物の detection です。

---

## 2. 挙動を変えるレバーは3つ。効きが全然違った

「思ったより浅い」「見てほしい観点を見てくれない」——そういう時に**何をいじれば直るのか**。ここで言う挙動とは、たとえば *どこまで掘るか（実コードまで読むか、仕様だけで答えるか）* や *どの観点で指摘するか* といった、調整したくなる振る舞いのこと。動かす手は3つあって、効きどころがまるで違いました。

| レバー | 効き | ひとこと |
| --- | --- | --- |
| プロンプト（システムプロンプト） | **条件付き** | 書き方次第で天と地（後述） |
| 呼び出し時の指示 | **確実** | 「コードまで見て」「速答で」が一番効く |
| KB の内容 | **コールドで効く** | 原則を、関連モードが読む場所に置いた時だけ |

いちばんの発見は、プロンプトの文言が **“書き方”次第で効きが天と地** だったことです。

* **効かない**：「間違いの重さと情報の古さを“判断”して深さを決めよ」→ 曖昧な判断を丸投げする規則。何度書いても安定しませんでした。
* **効く**：「テストレビューならコードを読め」→ 単純な **mode→action**。これを足したら、コールドで 5回中5回、実コードを読んで機能固有の漏れを出すようになりました。

---

## 3. ナレッジは「ログ」じゃなく「原則」にして初めて効く

自己成長の肝は KB。だが**何をどう書くか**で価値が桁違いに変わります。一番やりがちな失敗が、起きたバグをそのまま記録すること。これは**ログ**であって、別の状況のレビューにはまったく転用されません。

効いたのは、似た事例をまとめて\*\*一段抽象化した“再利用できる原則”\*\*に持ち上げ、**そのモードが読む場所に置いた**とき。これだけで、ヒントなしで原則が発火し、未記録の漏れを指摘するようになりました。

```
Before（ログ・転用されない）
  「事象 X で値 Y が消えた」

After（原則・コールドで発火する）
  「この種の確定処理では、保存される値が
   〔一部のフィールド／未処理の行〕で失われうる。
   確定後に "保存項目 × 行種別" を掛けて検証せよ」
```

コツは、**抽象度をちょうどの高さにする**こと（低すぎ＝転用されない／高すぎ＝「エッジケースをテストせよ」級のノイズ）と、**それを使うモードが読むファイルに置く**こと。良い原則でも、誰も読まない場所にあれば発火しません。

面白いのは、これが**節2のプロンプトと逆**なことです。プロンプトは抽象的に書くと滑り、KB は抽象的に書かないと効きません。違いは「抽象化という判断を、誰がやるか」です。プロンプトはそれを実行時の AI に丸投げするからブレます。KB は**人が先に原則へ持ち上げておく**ので、AI は当てるだけで済みます。

---

## 4. 危ない自己判断は「賢く分類」でなく「外しても安全」で潰す

このエージェントは、質問によっては**実コードまで掘りに行かず**、仕様書や KB だけで答えます（速さのため）。コードを掘れば実装に即して答えられますが時間がかかる。掘らなければ速いですが、実装と食い違うかもしれない——というトレードオフです。

困るのは、**いつ掘る／掘らないの自己判断がブレる**こと。一番怖いのが、**コードを掘らずに断定して、実コードと食い違う**ケースです。仕様では「保存される」はずの値が、実装では保存されていない——という乖離は普通に起きます。

「いつ掘るか」を正しく判断させられないなら、判断の精度を上げるより、**外しても安全な側に倒す**ことにしました。鍵はコストの非対称性で、掘らなくていい場面で掘っても**遅いだけ**、掘るべき場面で掘らず断定すると**間違う**。だったら安い失敗に寄せます。具体策は2つ：

* 答えの根拠が**実コードにある領域は、掘らない選択肢を作らず常に実コードを読む**（コードレビュー／バグ調査／テストレビュー）。速度は二の次。
* **掘らなかった時は「実コード未確認」と自己申告させる**。怖いのは掘らないこと自体でなく、\*\*“実コードを見ていないのに断定すること”\*\*だから。

---

## まとめ：評価は一回で終わらない

「使うほど賢くなるエージェント」を作りたくて始めましたが、実測してみて一番効いたのはこちらの学びでした——**エージェントの価値は、作った瞬間ではなく、評価して直していく過程で決まる。** 上の発見はどれも、回して数えるまで見えませんでした。

* **自分が教えた答えで、賢さは測れない。** 試すなら、まだ知らない問題で。
* **AI に判断させず、行動を指示する。** 迷う判断は、人かナレッジが持つ。
* **記録ではなく、教訓を残す。** 使う場所に置いて、知識は初めて効く。
* **正しく判断させるより、外しても安全に。** わからない時は「わからない」と言わせる。

そして自己成長エージェントは、**使うたびに中身（KB）が変わります**。リリース前に一度 OK を出しても、翌週には別物です。しかも「まだ記録にない題材」を一番投げてくれるのは、日々の実運用そのもの。

だから評価は実験室で一度きりではなく、**運用しながら回し続ける営み**になるはず——というのが、作り終えた時点での仮説です。実はまだ本番には出していないので、運用に乗せたら、また測って書きます。
