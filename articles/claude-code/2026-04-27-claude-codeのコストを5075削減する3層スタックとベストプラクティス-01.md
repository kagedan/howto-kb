**要約：** Claude Codeのトークンコストを継続的に50〜75%削減するには、単一のテクニックだけでは不十分です。この記事では、プランファースト、モデルティアリング、サブエージェント委任、トークンバジェッティング、3層スタックを組み合わせた4つの実践的コンボを紹介します。AIコストをシステマティックにコントロールするための手法です。

## トークン最適化に「銀の弾丸」が存在しない理由

第1回は習慣について、第2回はキャッシュについて扱いました。1記事しか読めないなら第2回を選んでください — キャッシュが最大のレバレッジです。ただし、**Claude Codeのコストを50〜75%継続的に削減したい**なら、どの手法も単独では足りません。

**トークンの無駄遣いは複数の場所で発生します** — 各テクニックはそれぞれ1箇所を塞ぐにすぎません。すべてを塞ぐには、重ねて使う必要があります。

| 無駄の発生源 | 対応するテクニック |
| --- | --- |
| 過剰なインプット（余分なファイル読み込み） | ナレッジグラフ、`.claudeignore`、Read offset、ネイティブツール |
| 過剰なヒストリー（長い会話） | 戦略的な `/compact`、ローリングサマリー |
| コンテキストの拡散（探索フェーズ） | サブエージェント委任 |
| シンプルなタスクに高性能モデルを使いすぎ | モデルティアリング、opusplan |
| 冗長な出力 | `max_tokens`、stop\_sequences、フォーマット制約 |
| セッションごとにゼロから再構築 | バッチ処理、フォーキング、キャッシュ規律（第2回） |

この記事では4つの実践的コンボを紹介します — 複数のテクニックが協調してAIコストを最大限に削減する組み合わせです。

## コンボ1：Plan-First × モデルティアリング × サブエージェント — 複雑なタスクのコストを50〜75%カット

複雑なタスクに対する「ダイヤモンドコンボ」です。正しく適用すれば、「いきなりコードを書く」ワークフローと比べてコストを50〜75%削減できます。

### Plan-first：合意なしにClaudeを実行させない

**Plan-first**は、コードを1行も書く前に詳細な計画を作成させるテクニックです。認識のズレを早期に発見し、トークンを無駄にする修正ループを避けることが目的です。

よくある失敗パターン：*「Feature Xを実装して」* といきなり指示する。Claudeが意図を推測 → コードを書く → 認識のズレに気づく → 修正 → 書き直す。このループ1回ごとにトークンのかたまりが消費されます。

より良いやり方：

```
1. Feature XのTODOプランを作成させる（まだコードは書かない）
2. プランをレビューし、必要なら修正する
3. ステップごとに実行させる
```

なぜ効果的か：

* コードが間違う前に、**トークンを使う前に**認識のズレを発見できる
* 各実行ステップが小さくなるため、コンテキストの蓄積が遅くなる
* 最初から正しい結果が出やすい → 修正ループが減る

Plan-firstは`EnterPlanMode`や類似のカスタムコマンドと相性が良いです。

### モデルティアリング：タスクに適したClaudeモデルを選ぶ

**モデルティアリング**は、タスクの複雑さに応じて異なる価格帯のClaudeモデルに振り分ける戦略です — すべてにOpusを使う代わりに。

すべてにOpusを使うのは、ファイルにラベルを貼るためにチーフアーキテクトを雇うようなものです。3つの階層に分けましょう：

| モデル | 典型的なタスク比率 | 向いているタスク |
| --- | --- | --- |
| **Haiku**（$0.25 / $1.25 per Mトークン） | 25〜35% | JSONフォーマット、docstring、変数リネーム、シンプルなロギング |
| **Sonnet**（$3 / $15 per Mトークン） | 55〜65% | 機能実装、ユニットテスト、API連携 |
| **Opus**（$15 / $75 per Mトークン） | 5〜10% | アーキテクチャ設計、パフォーマンス最適化、セキュリティレビュー |

モデル選択の基準は**複雑さ**であって、**重要度**ではありません。コミットメッセージが重要だからといってOpusが必要なわけではありません — Haikuで十分です。

### OpusPlanパターン：Opusで設計し、Sonnetで実行する

**OpusPlan**は2つのモデルを組み合わせるパターンです：Plan ModeでOpusを使ってアーキテクチャを検討し、その後Sonnetに切り替えて実行します — 設計品質を落とさずにコストを削減できます。

有効化方法：`/model opusplan`

認証リファクタリングタスクでの実際の数値：

* Opus全体：**$1.23**
* OpusPlan（Opusで計画 + Sonnetで実行）：**$0.57**
* **54%の削減**

最適なモデルティアリングで1,000タスクにスケールした場合：**$897** 対 Opus全体の **$3,645** — **75%の削減**。

### サブエージェント委任：メインコンテキストへの汚染を防ぐ

**サブエージェント委任**は、調査タスクや大量のファイル読み込みを別のコンテキストで動くエージェントに委任するテクニックです。メインセッションに余分なデータが混入するのを防ぎます。

*「src/全体を読んで認証関連のコードを探して」* と指示すると、**すべてのファイル内容**がメインコンテキストに流れ込みます。メインセッションが汚染され、キャッシュが壊れ、コストが急騰します。

代わりにこう言いましょう：

```
「サブエージェントを使ってsrc/を調査し、
 認証関連のファイルをfile:line形式で要約して返して」
```

サブエージェントの特徴：

* **独立したコンテキスト**で動く — メインを汚染しない
* 処理完了後に50,000トークンの生データではなく**200トークンのサマリー**を返す
* サブエージェントには**Haiku**を使う：`CLAUDE_CODE_SUBAGENT_MODEL=haiku` → コンテキストが軽く、モデルコストも安い

役割分担を明確に：

* **メインセッション**：判断、設計、統合
* **サブエージェント**：調査、大量読み込み、定期タスク、マイグレーション実行

並列リサーチへの応用：4つのサブエージェントを同時実行（`codebase-locator`、`codebase-analyzer`、`thoughts-locator`、`thoughts-analyzer`）→ `thoughts/shared/research/<topic>.md` に集約。その後のプランニングフェーズではこのファイルだけ読めばよく、コードベースを再読する必要がなくなります。

### まとめ：実際の複雑なタスクでの活用

タスク：「新しいOAuthプロバイダーをサポートするよう認証ミドルウェアをリファクタリングする」

完全なワークフロー：

1. **サブエージェントによるリサーチ**：3つのエージェントが並列で現在の認証実装・OAuthプロバイダーAPI・テスト構造を調査 → 500トークンのリサーチサマリーファイルを生成
2. **OpusによるPlan Mode**：リサーチファイルを読み込み、8ステップのプランとトレードオフ分析を提示
3. プランをレビューし、承認する
4. **Sonnetに切り替え**：ステップごとに実行し、各ステップ後すぐにテスト
5. サブエージェントが独立したコンテキストでフルテストスイートを実行 → pass/failを報告

ナイーブなワークフロー（「OAuthをサポートするよう認証をリファクタリングして」）との比較：**コスト60〜70%削減**、修正ループが少なく、プランが明確。

## コンボ2：ネイティブツール × Read Offset × 構造化出力 — ワークフローを変えないトークンハイジーン

「トークンハイジーン」の3点セット — ワークフローを変えず、Claudeへの指示の仕方を変えるだけです。

### Claude Codeのネイティブツールの方がBashより安い

組み込みのGrep/Glob/Readは**構造化された出力**を返し、コンテキストに入る前にClaude Codeが処理・圧縮します。Bash経由で`rg`/`find`を実行すると**生のテキスト**がそのまま流れ込む — 同じ情報量でもトークンをより多く消費します。

| タスク | 悪い例（トークン多め） | 良い例（トークン少なめ） |
| --- | --- | --- |
| パターン検索 | `Bash: rg "TODO" src/` | **Grepツール** |
| ファイル検索 | `Bash: find . -name "*.tsx"` | **Globツール** |
| ファイル読み込み | `Bash: cat file.ts` | **Readツール** |

CLAUDE.mdに記載しましょう：*「Prefer built-in Grep/Glob/Read over bash equivalents.」*

複数のシェルコマンドを組み合わせる場合はBashが必要ですが、検索と読み込みはネイティブツールを優先します。

### Read offset + limit：大きなファイルの読み込みトークンを削減する

ROIが高く、手間はほぼゼロです：

* `Read Form.tsx` → 2,000行がコンテキストに入る
* `Read Form.tsx offset=120 limit=50` → 50行だけコンテキストに入る

**二重のメリット：** コンテキストが小さくなる → ターンごとのトークンが減る → autocompactの発動が遅くなる → 4層圧縮パイプラインの呼び出しが減る。

標準ワークフロー：Grepで行番号を特定 → Read offsetで精確に読み込む。

### 構造化インプット：CSV形式はプローズより50%トークンを節約できる

繰り返しパターンのデータは**テーブル/CSV**で渡す — プローズではなく。

**プローズ（約60トークン）：**

> 「ユーザーの張三、25歳、北京出身、商品Aを購入、100元、2024-01-15。ユーザーの李四、30歳...」

**CSV（約30トークン、同じ情報量）：**

```
name,age,city,product,price,date
Zhang San,25,Beijing,A,100,2024-01-15
Li Si,30,Shanghai,B,200,2024-01-16
```

約50%の削減。ボーナス：モデルはCSVの方がパースしやすく、精度も向上します。

### 出力制約：冗長さを2つの層で防ぐ

LLMはデフォルトで長く話したがります。制御しないと、出力トークンは必要量の3〜5倍になることがあります。

**層1 — ソフト制約（プロンプト）：**

* 「5つの箇条書きで回答し、各項目は15語以内」
* 「verdict、reason、scoreのキーを持つJSONで」
* 「前置きなし、結果だけ」

**層2 — ハード制約（API）：**

* `max_tokens` をユースケースに応じて設定
* `stop_sequences`：`\n\n`、`</answer>`
* 低temperatureにする → 出力が短く、一貫したものになる

ユースケース別プロファイル：

```
{"code_gen":      {"max_tokens": 1000, "temperature": 0.2},
 "data_analysis": {"max_tokens": 400,  "temperature": 0.3},
 "simple_qa":     {"max_tokens": 200,  "temperature": 0.5}}
```

### プロンプトの冗長さを排除する：67%削減、品質は変わらない

Claude Code利用時のROIが高い最適化の一つ、手間はゼロ：**プロンプトから余分な言葉を削る**。

**削減前：**

> 「こんにちは、このコードを分析するのを手伝っていただけると嬉しいです。これはデータを処理するPython関数です。見ていただいて、問題点と改善方法を教えていただけますか？日本語でお答えください。」

**削減後：**

> 「以下のPython関数の問題点と改善点を日本語で分析してください：」

品質は変わらない。LLMが処理するのは情報であって、丁寧な言い回しを採点しているわけではありません。

4つの実践ルール：

1. 挨拶や回りくどい説明を省く
2. 命令形の動詞を直接使う
3. 情報を繰り返さない
4. ヘッジ表現を省く（「もしかしたら」「よかったら」「できれば」）

特に**システムプロンプト**では重要です — **すべての**リクエストでカウントされるからです。

## コンボ3：トークンバジェッティング × コンテキストプレッシャープロトコル — 大規模プロジェクトの財務規律

大規模プロジェクト向けのパターンです。Claude Codeのコンテキストに「財務規律」が必要な場面で使います。

### トークンバジェッティングとは何か、なぜ必要か

**トークンバジェッティング**は、セッション開始前にコンテキストの各カテゴリ（システムプロンプト、ルール、ツール定義、取得コンテキスト、レスポンス）に明示的な上限を設けるテクニックです。コンテキストが際限なく膨らむのを防ぎます。

明示的な上限がなければ、誰も気づかないうちにコンテキストが占有されていきます：必要以上に長いルール、十分に整理されていない取得コンテキスト、制限のないレスポンス。

事前に予算を設定しましょう：

```
system prompt       4K
project rules       800
tool definitions    12K
retrieved context   15K   ← 最も膨らみやすい
response budget     8K
safety margin       2K
───────────────────────
total               42.3K
```

いずれかのカテゴリが上限を超えたら → 自動で整理するか、エラーを報告します。

**500セッションのケーススタディ：** トークンバジェッティングを適用したところ、静的コンテキストが **51,300 → 19,600トークン** に削減 — **セッションあたりのコストが62%減**。1日100セッションで：月約$1,530の節約。

取得コンテキストの影響が最も大きいことが多いです — 関連度の高いファイルを予算内で優先し、重要度の低いセクションを完全に省くのではなく30%に要約するだけで、45Kから15Kに削減できます。

### コンテキストプレッシャープロトコル：暗黙のコーナーカットではなくエラー報告させる

**コンテキストプレッシャープロトコル**は、CLAUDE.mdに記述するルールセットです。コンテキスト使用率が70%を超えた際にClaude Codeが取るべき行動を指定します — モデルが通知なしに検証ステップをスキップする代わりに。

コンテキストが満杯に近づくと、Claude Codeは**検証ステップを徐々に省略する**傾向があります — ファイルを読まずに修正する、Plan modeをスキップする、テストを飛ばす。最も問題なのは：**これが無言で行われること**です。

CLAUDE.mdに明記しましょう：

```
## コンテキストプレッシャープロトコル
コンテキスト使用率が70%を超えた場合：
- このセッションで読んでいないファイルを修正してはならない
- テスト/検証ステップをスキップしてはならない
- 調査タスクにはサブエージェントを優先して使う
- 品質を下げないとタスクを完了できない場合：
  「コンテキストが少なくなっています。compact/forkを推奨します」と通知して停止すること
  暗黙のコーナーカットをしない
```

原則：**通知付きの低品質アウトプットは、無言の低品質アウトプットよりも優れています**。この期待値を明文化しておきましょう。

### MCPと拡張機能：「ゼロコスト幻想」への注意

すべての拡張機能がClaude Codeでコンテキスト上無料というわけではありません：

| 種類 | ロードタイミング | コンテキストコスト |
| --- | --- | --- |
| **CLAUDE.md** | セッション開始時、毎リクエスト | 高、固定 |
| **MCPサーバー** | セッション開始時、毎リクエスト | 高、固定 |
| **スキル** | 最初に説明文、呼び出し時に本体 | 徐々に増加 |
| **サブエージェント** | 独立したコンテキスト | **メインに対して0** |
| **フック** | エージェントループ外 | **0** |

実践的な判断基準：

* Reasoningが不要な自動化 → **Hook**（コンテキストゼロ）
* 大きなワークフロー → **スキル**（遅延ロード）。あまり使わないスキルは`disable-model-invocation: true`を設定 — 手動で呼び出すまでコンテキストから隠れる
* 必要な外部ツール → **MCP**（ただし厳選 — 各サーバーは固定コスト）
* CLAUDE.mdは500行以下に保つ；長いドキュメントはスキルに変換する

## コンボ4：3層トークンスタック — 包括的なコスト削減のマスターパターン

**3層トークンスタック**は、Claude Codeにおける3つの異なるトークン無駄遣いの発生源をROI優先順位に従って解決する3層アーキテクチャです — 互いに代替するのではなく、重ねて使います。

### ROI優先順位順の3層

1. **ナレッジグラフ**（または同等のもの） — Claudeが読む前にインプットトークンを削減
2. **サブエージェント委任** — 作業中のコンテキスト拡散を削減
3. **戦略的な `/compact`** — 蓄積後のヒストリートークンを削減

### この順序が最適な理由

* **層1 — ナレッジグラフ**：Claudeが読む*前に*入力を削減 → 最高ROI
* **層2 — サブエージェント**：作業*中に*コンテキストの拡散を削減 → 中程度のROI
* **層3 — `/compact`**：蓄積*後に*ヒストリーを削減 → 最低ROIだが依然として必要

### 3層スタックを使った典型的なワークフロー

```
[タスク：フォーム処理に関連する5つのReactコンポーネントをリファクタリング]

Step 1 — グラフクエリ（層1）：
  「src/内のフォーム処理に関連するコンポーネントを探す」
  → 15ファイルが関連（src/の120ファイル全部を読む代わりに）

Step 2 — サブエージェント分析（層2）：
  「サブエージェントを使って15ファイルを分析し、
   コンポーネントのprops、stateの形状、イベントハンドラーを返して」
  → 200トークンのサマリー（50,000トークンの生データの代わりに）

Step 3 — メインセッション + /compact（層3）：
  サマリーを元にリファクタリングプランを立て → 実行
  → 各フィーチャーが完了したら：保持指示付きで /compact

Step 4 — MCPによる永続化（Claude Code v2.1.91+）：
  大きなグラフクエリ結果をディスクにキャッシュ → 再クエリ時にトークンを消費しない
```

### Codebase-Memory：超大規模コードベース向けのツール

**Codebase-Memory**はC++で書かれたMCPサーバーで、tree-sitterとLSPを使って大規模コードベースをインデックス化し、ファイル全体をコンテキストに読み込まずに構造的なクエリ結果を返します。

実際のパフォーマンス：2800万行のLinuxカーネルを3分でインデックス化し、クエリをミリ秒以下で返します。5回の構造的クエリで約**3,400トークン**、ファイル探索の約**412,000トークン**と比較して — **120倍の削減**。

知っておくべきトレードオフ：**回答品質はファイル探索の92%に対して83%です**。このギャップは、深いロジック理解が必要なタスク（複雑なデバッグ、セキュリティレビュー）では重要になります。

合理的な組み合わせ方：「*Xに関連するものは何か*」という質問にはCodebase-Memory、「*Xのロジックはどうなっているか*」にはターゲットを絞ったファイル読み込み。

## Claude Codeのトークン節約についてよくある質問

**Q：どのテクニックから始めればいいですか？**  
A：プロンプトキャッシュ（[第2回](https://zenn.dev/ruralwritter/articles/4656b52b64a0ed)）から始めてください — 最大のレバレッジで最高のROI、ワークフローを変える必要もありません。その後、plan-firstとモデルティアリング（この記事のコンボ1）を追加します。トークンバジェッティングと3層スタックはプロジェクトが大きくなってから適用します。

**Q：重要なタスクにHaikuを使うのは安全ですか？**  
A：基準は**複雑さ**であって、重要度ではありません。HaikuはJSONフォーマット、docstring、変数リネーム、コミットメッセージを問題なく処理できます — たとえそれらがプロジェクトにとって重要なタスクであっても。深い推論が必要なタスク（アーキテクチャ設計、セキュリティレビュー）だけOpusが必要です。

**Q：サブエージェント委任はどんなタスクに向いていますか？**  
A：大量のファイルの調査、テストスイートの実行、大量のログ読み込み、マイグレーション。メイン会話で直接結果が必要なタスクや、常に相互参照が必要なタスクには向いていません。

**Q：コンテキストプレッシャープロトコルは必須ですか？**  
A：必須ではありませんが、長いセッションを実行するプロジェクトや大規模なコードベースには必要です。このプロトコルがないと、コンテキストが70%を超えたときにClaude Codeが暗黙のうちに検証ステップをスキップします — アウトプットの品質が下がっても、その原因に気づきにくくなります。

## 結び：Claude Codeのコスト削減はシステム全体の取り組みである

**Claude Codeのコストを50〜75%継続的に削減できる単一のテクニックは存在しません。**

真のコスト削減は重ね合わせから生まれます：正しい方向に進むためのplan-first、適正な価格を払うためのモデルティアリング、コンテキストをクリーンに保つためのサブエージェント委任、何にコストを払っているかを把握するためのトークンバジェッティング、1トークンも無駄にしないためのプロンプトハイジーン、毎セッションゼロから再構築しないためのキャッシュ規律、そしてひとつひとつを意識しなくても済むようにするための基盤となる習慣。

1つのテクニックを選ぶのではありません。**すべてを日々のワークフローに織り込んで、意識しなくてもできるようにすること**が目標です。
