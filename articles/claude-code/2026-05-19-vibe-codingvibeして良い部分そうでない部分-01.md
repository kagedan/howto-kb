Vibe という言葉には、妙な魅力があります。

効率がよさそうで、楽そうで、しかも未来感があります。  
努力しなくてもいい。学ばなくてもいい。近道を通って、オールドタイプの「プログラマー」を一気に追い越せる。

そんな響きがあります。

では、実際のところはどうなのでしょうか。

部分的にはそのとおりですが、そうでない部分もあります。

数行の指示だけで、AI が中規模以上のプロジェクトを完璧に作り上げる。人間は一切監督しなくていい。  
いつかはそうなるかもしれません。しかし、少なくとも今はまだそこまで来ていません。おそらく明日もまだ無理でしょう。

私たちは今日の技術で、今日のプロダクトを作らなければなりません。  
だからこそ、どこは Vibe Coding に任せてよくて、どこは自分で握るべきなのかを見極める必要があります。

AI の進化は、これまで何度も人間の予想を裏切ってきました。  
プログラマーが真っ先に置き換えられると思われていた時期もありましたが、実際には先に棋士、画家、作家の領域が侵食されています。

そのため、私は「AI が自分の生きている間にプログラマーを完全に置き換える」と決めつけてキャリアを考えるつもりはありません。

そもそも、仮に本当にその日が来たとしても、特別な準備が必要だとは思っていません。  
プログラマーを完全に代替できる AI が手元にあるなら、こちらは欲しいものを伝えるだけで済むはずです。あとは AI が作ってくれるでしょう。

そこまで来たのなら、準備も何もありません。

もっとも、そんな強力な AI が一般ユーザーに気軽に開放されるとも思っていません。今でさえ、私たちは Claude Mythos に触れることすらできないのですから。

少し脱線しました。

私は仕事でも個人開発でも、AI エージェントをかなり使っています。  
最近では Neovim を開く機会もかなり減りました。

長い時間をかけて作り込んだ設定も、体に染み込ませたショートカットキーも、以前ほど出番がありません。

ただ、仕事ではほとんど Vibe Coding をしていません。  
AI エージェント、主に Claude Code と対話しながら、こちらが主導権を持ってコードを書かせています。

AI が書いたコードに思わぬヒントをもらうこともありますし、逆にこちらが誤りを直すこともあります。  
この形は、かなり前向きな変化だと感じています。私は以前よりも良いコードを書ける、いえ、産出できるようになりました。

仕事の速度も品質も上がりました。  
ただし、青天井で限界突破できたわけではありません。結局のところ、ボトルネックは私でした。人間は遅いのです。

それでも、効率のためにコントロールを手放すつもりはありません。  
なぜなら、怖いからです。

AI の不確実性が、重大な障害として本番環境に入り込むかもしれない。  
それが怖いのです。

私の関わっているプロジェクトには、信頼できるテストがあります。しかし、非常に重要で、使用期間中に障害が起きてしまうと大きな損失につながるシステムでもあります。

私は安心して眠りたいのです。  
だからこそ、本番に出すコードは、自分が自信を持っているものでなければなりません。

一方で、個人開発ではかなり自由に試せます。  
実際、私はいくつものプロジェクトを Vibe Coding で作りました。

結果は思った以上に良く、少し大きめのプロジェクトでもそれなりに動きました。

ただ、「それなりに動く」で本当に十分なのでしょうか。

数百行程度の小さなツールであれば、Vibe Coding はかなり合理的です。  
むしろ、実装の細部をいちいち読むほうが時間の無駄かもしれません。

では、数千行を超えるような中規模以上のプロジェクトではどうでしょうか。  
小さなツールでは気にならなかった問題が、急に無視できない大きさになります。

プログラミングの世界では、「銀の弾丸はない」という言葉がよく知られています。  
この言葉に異論のある人は少ないでしょう。

言語、フレームワーク、設計方針、さらには if 文の書き方ひとつを取っても、正解はひとつではありません。  
ここが Vibe Coding の難しいところです。

AI は完璧なコードを書けません。  
そもそも、完璧なコードなど存在しないからです。

プロジェクトが大きくなり、複雑になるほど、AI が一度に見渡せる範囲は相対的に狭くなります。  
小さなプロジェクトのように、毎回すべてのファイルを読ませるわけにはいきません。

AI は確かに優秀です。  
しかし、部分最適が全体最適とは限らないのです。  
局所的には正かった判断が、プロジェクト全体にとって正しいとは限りません。

そこに問題があるのです。

設計レベルの小さな歪みが、AI の手によって驚くほどの速度で増幅されます。  
構造を少し変えれば簡単に解決できる問題でも、AI は目の前の目的を効率良く達成するために、パッチを重ね続けることがあります。

その場では動きます。  
テストも通るかもしれません。  
しかし、その修正は次の開発を少しずつ難しくしていきます。

やがてコードの複雑さが、AI 自身にも扱いきれないところまで膨らんだとき、詰みます。

AI は効率が良すぎるのです。  
完全に壊れる直前まで、問題がないかのように物事が進むでしょう。

もちろん、人間も完璧なコードなど書けません。  
私たちはいつも、過去の自分が下した判断に苦しめられています。

それでも、人間には経験と勘があります。  
その経験と勘こそが、AI の弱い部分を補える力です。

プロジェクトの構造が健全かどうかを見る。  
設計の方向が破綻していないかを判断する。  
それは、まだ人間が担うべき仕事です。

もちろん、人間も間違えます。  
ただ、そのときは AI が助けてくれます。

AI エージェントがもたらした大きな恩恵のひとつは、リファクタリングのコストを劇的に下げたことです。

開発の途中で設計ミスに気づいたとき、修正には大きな作業量が伴います。  
その作業量のせいで、私たちは何度も妥協してきました。

ディレクトリ構成を最初にうまく切れなかった。  
namespace の分け方を間違えた。  
でも、今さら動かすと大量の import を直さなければならない。

そうして、コードは間違った場所に置かれたままになります。

しかし、今なら違います。  
一言指示すれば、AI エージェントは黙々とリファクタリングしてくれます。

だからこそ、現時点で一番現実的なのは、人間と AI を組み合わせた「選択的 Vibe Coding」だと思っています。

人間が全体を見て、どこを AI に任せるかを決め、そして、どこを自分で判断すべきかを決めるのです。

ある部分では、関数の中身まで読む必要はないかもしれません。  
ただし、関数のシグネチャは把握しておくべきでしょう。

別の部分では、実装の細部を追わなくてもよいかもしれません。  
ただし、クラス設計全体が妥当かどうかは見ておくべきでしょう。

どこで自分の判断が必要になるのか。  
それ自体を判断する必要があります。

そうでなければ、それは開発ではなく、AI で遊んでいるだけです。  
AI が全部うまくやってくれることを祈る、ただのガチャになってしまいます。

AI が全自動で解決できない場面でも、安心して進めたいなら。  
数か月後も、プロジェクトを無理なく前へ進められる状態にしておきたいなら。

選択肢はありません。

今の答えは、選択的な Vibe Coding です。

もちろん、Vibe Coding だけで大規模プロジェクトを安定して扱えると分かったなら、私は迷わず考えを変えます。  
状況を見て立場を変えられることも、エンジニアに必要な能力です。
