## はじめに

2026年05月06日（米国時間）、Anthropic が [`Higher Limits`](https://www.anthropic.com/news/higher-limits-spacex) を発表しました。  
Claude Code の 5 時間 rate limit が 2 倍化、ピーク時間帯制限が撤廃、API Opus の rate limit も大幅に増えるという内容です。価格は据え置き、効果は即日。

公式発表によると、背景には SpaceX の Colossus 1 データセンター（220,000+ NVIDIA GPUs）に 1 ヶ月で 300+ MW の容量を追加したというインフラ拡張があります。絶対量が一段増えたタイミングでの開放、という構図です。

毎日 Claude Code を触っているエンジニアとして、この変更で「実際に何が変わって、何が変わらないのか」を整理してみます。今後の運用を考える材料になればと思います。

## 何が変わったのか

公式発表で確認できる変更は主に 4 つです。

1. **Claude Code の 5 時間 rate limit が 2 倍化**（Pro / Max / Team / Enterprise）
2. **ピーク時間帯制限が撤廃**（Pro / Max）
3. **Claude API の Opus rate limit が大幅に増加**
4. **価格は変更なし**

サブスクユーザーにとっては「同じ料金で 2 倍使える」という変化です。  
Max 5x の月 $100、Max 20x の月 $200、Pro の月 $20 は据え置きで、5 時間あたりに送れるメッセージ・トークン量が単純に倍になります。

ピーク時間帯制限の撤廃は、特に夜業時間帯に影響します。「速度が一時的に落ちる」「限界に当たりやすい」という挙動が、Pro / Max では消える方向です。

API ユーザーにとっては、Opus の rate limit 増が大きな意味を持ちます。  
これまで「個人で Opus 中心の運用は枠的に厳しい」と感じていたケースが、緩和される方向です。

## エンジニアにとって何が変わるか

自分の使い方を振り返ると、直近 30 日で 274 時間ほど Claude Code を触っていました（自分でログ集計した数字、別記事で詳細）。1 日平均 9 時間程度です。

この使い方だと、以前は 5 時間枠の終盤でレスポンスが詰まったり、夜業帯で「待ちが入る」瞬間が日常的にありました。2 倍化とピーク時間帯撤廃は、この「待ち」を体感ゼロに近づける方向です。すでに同じことを感じている方も多いと思います。

ここから先は仮説ですが、エンジニアの使い方は次の方向に振れると予想しています。

**1. Sonnet 中心 → Opus 中心へのシフト**

これまで「コスト効率」のために Sonnet を多用していたケースが、Opus 中心に振り直される可能性があります。サブスクユーザーは料金変動なし、API ユーザーも rate limit 緩和で「Opus の絶対量制限」が緩むため、能力を取りに行く動機が強まります。

**2. session を短く切る習慣の見直し**

「context が長くなりすぎる前に切る」という運用を持っている人は多いと思いますが、5 時間枠が広がったことで「長尺 session で運用する」選択が現実的になります。context 維持で前後関係の説明が要らなくなるメリットが、rate limit リスクを上回りやすくなる、という構図です。

**3. subagent / parallel tool 利用の積極活用**

並列 tool 呼び出しや subagent への委譲は rate limit 消費の観点で控えていた側面がありましたが、枠に余裕が出たことで気軽に使える選択肢になります。1 セッション内の同時並列度が一段上がりそうです。実際に自分も、5/06 以降の 1 セッションで subagent 並列を 4-5 回起動する運用を試しています。これまで「枠が惜しくて控えていた」瞬間が確かにあったな、と気づきました。

## 何が変わらないのか

同じく整理しておきたいのは、変わらない部分です。

**週次 rate limit は据え置き**

Claude Code には 2025年08月 から導入されている週次 rate limit（Pro / Max 向け、リセット 7 日ごと、全体使用 + Opus 別の 2 軸）があります。  
今回 2 倍化されたのは 5 時間 rate limit のみで、**週次の総量上限は変わりません**。「5 時間あたりが 2 倍」=「週次総量が 2 倍」ではない、という点が誤解されやすい構図です。

「これまで週次レートに当たっていなかった人」にとっては実質的な総量上限はほぼ動きませんが、5 時間枠の「待ち時間」体感が消えるので生産性は上がる、という読み方になります。逆に「もともと週次レートに張り付いていた人」（subscriber 全体の数 % 程度と推計されています）には今回の発表で増える総量はありません。

**月額予算の上限は変わりません**

Pro $20、Max $100 / $200 のサブスク料金は据え置きです。  
「2 倍使える」=「使用量が増える方向に振れる」ということでもあります。元から枠を使い切っていた人ほど「コスパ」は実質的に上がりますが、API 従量課金で運用している人にとっては「rate limit が緩和された」だけで、月額請求額は使った分だけ増えます。

**「自分が何にトークンを使っているか」を把握する必要性**

むしろ重要性が増します。  
rate limit 倍増で使用量が増えやすくなる = 「気づいたら過剰に Opus を呼んでいた」「気づいたら subagent を使いすぎていた」が起きやすくなります。

把握していない状態での使い方の自動拡大は、API ユーザーには直接コストとして跳ね返り、サブスクユーザーには「次のプラン判断」を曖昧にします。

**コスト構造の理解**

input / output / cache\_read / cache\_create のトークン単価差は変わらず、キャッシュ戦略を含めた最適化余地はそのままです。1 セッションを長くすればキャッシュヒット率は上がる、subagent はキャッシュ層をまたぐので並列度を上げると create 側が増える、といった構造的な話は引き続き重要です。

## 自分の使い方を可視化する手段

自分のトークン使用量を集計するツールはいくつかあります。

最初に試すなら [ccusage](https://github.com/ryoppippi/ccusage) です。Claude Code の JSONL ログ（`~/.claude/projects/<project>/<session>.jsonl`）を集計する OSS の定番で、コマンド一発で日次・モデル別の使用量と API 換算コストが出ます。

私が個人的に開発している [koji-lens](https://github.com/etoryoki/koji-lens) は、ccusage が日次集計、koji-lens は session 内訳という棲み分けで作っています（β）。  
同じ JSONL を読み、ツール別 / subagent 別 / セッション別の内訳と Web ダッシュボードを追加するアプローチです。

## おわりに

まとめると、`Higher Limits` で「使える量」は単純に 2 倍になりますが、「自分が何にトークンを使っているか把握する必要性」はむしろ増えます。枠の余裕で運用が自動拡大しやすいぶん、見えていない状態のリスクが上がる、という構図です。

2 倍化で待ち時間が消えた感触をすでに持っている方、Opus 比率を上げてみた方、長尺 session に切り替えた方、いろんな運用変化があると思います。
