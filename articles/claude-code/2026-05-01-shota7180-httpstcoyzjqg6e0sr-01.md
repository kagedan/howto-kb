デザイン業務は、私たちが想像する以上に時間がかかります。

依頼を受けて、意図を読み取り、ラフを作り、修正して、確認して、また修正する。
この往復がある限り、数時間単位の工数になるのは当たり前でした。内容によっては、半日以上かかることもあります。

この課題に対して、SHIFT AIのデザイン部ではAIエージェント活用による業務の再設計に取り組んでいます。

この取り組みを進めているのが、当社 デザイン部の責任者である川合さん(@kawai_design)です。現在はAI導入支援や研修にも携わりながら、実務レベルでのAI活用を進めています。
現時点で、制作時間は平均で約1/2まで短縮されています。さらに、フローを最大限活用できた場合の試算では、6時間かかっていた作業を30分程度(最大1/12)まで圧縮できる見込みです。

ただし、やっていることは「効率化」ではありません。デザイン業務を分解し、別のものとして作り替えています。

ここで起きているのは、ツールの活用ではなく、業務そのものの再設計です。

# **AI駆動型のデザインフロー**

![](https://pbs.twimg.com/media/HHMsjTQbkAApDHk.jpg)

ここで紹介するフローは、デザイン部が 理想像として目指している全体像 です。100%この通りに動いているわけではなく、現時点では一部が未整備、または運用上の判断で停止している箇所もあります。

目指しているフローはこうです。

・Slackで依頼を受け取る
・Claude Codeが内容を解析して要件定義を作る
・新規か既存かを分岐する
・生成用のプロンプトを組み立てる
・ChatGPT Images 2.0で画像を生成する
・結果をチェック・修正する(ターゲットがその画像を見て行動するか、AIの中に仮想的な受け手を置いてテストする構想も含む)
・確認に回す

一つひとつはシンプルです。重要なのは、ここが「分断された工程」ではないという点です。最初から、一つの流れとして設計されています。

このフローは単に工程を並べているわけではありません。重要なのは「どこで精度が決まり、どこで修正コストが発生するか」を前提に設計されている点です。

たとえば、要件定義の前に人が判断する工程を挟むと、判断のばらつきがそのまま後工程に影響します。そのため、最初の段階でClaude Codeに要件を整理させ、基準を揃えた状態で次の工程に渡す設計にしています。

# **精度は「要件定義」で決まる**

この仕組みで最も効いているのは、要件定義です。

デザインは感覚の仕事に見えますが、実際には

・誰に
・何を
・どう伝えるか

でほぼ決まります。

ここを曖昧にしたまま作るから、修正が増えます。

デザイン部では、過去のクリエイティブや、ターゲット情報をもとに要件を組み立てています。さらに、どのクリエイティブがどの成果につながったか — インプレッションやウェビナー参加者数などの成果データの紐付けも進めている段階です(この部分は現時点では完全には整備できておらず、これから本格的に蓄積していく前提です)。

「感覚やセンスで作る」をやめ、最初に言語化する。これだけで、後工程は大きく変わります。実際には、この要件定義は人が書いているわけではありません。

Slackに流れた依頼をもとに、Claude Codeが内容を読み取り、過去のクリエイティブやターゲット情報を参照しながら要件定義書を自動で作成する設計になっています。なお、Slack上の依頼をClaudeが自動で巡回・取得する仕組みも構築済みですが、現在はトークン消費を抑えるため一時的に停止しており、運用上は依頼を都度Claudeに渡す形を取っています。

参照しているのは、単なるテキストではありません。

・過去に制作したデザイン
・どのターゲットに向けたものか
・(整備中)どのクリエイティブがどの成果を出したか

こうしたデータを紐づけて読み込ませています。

さらに、ブランドのルールやフォント、配色などもすべてDESIGN.mdというテキストファイルとに定義されています。AIはそのファイルを読み込んだうえで、要件を組み立てます。ここでやっているのは、「感覚で作る」ことの排除です。最初に全ての条件を言語化することで、後工程の修正を減らしています。
