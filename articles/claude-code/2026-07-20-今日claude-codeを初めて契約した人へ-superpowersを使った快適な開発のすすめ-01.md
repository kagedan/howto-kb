# 今日Claude Codeを初めて契約した人へ 〜superpowersを使った快適な開発のすすめ〜

Claude Codeを契約した日にやることは、たいてい同じです。「これを作って」と打ち込みます。数秒後、コードが返ってきます。速いです。感動します。

そして数十分後、もう一度同じ画面を見ることになります。動くには動きました。でも思っていたものと違います。作り直しになります。

腕の差ではない、と分かったのは`superpowers`という拡張を1ヶ月使ってからでした。プラグインを何も入れていないClaude Codeに「作りたいもの」だけを渡すと、こうなる可能性が高いです。

## 何も入れていないClaude Codeに何を渡すと、何が起きるか

「認証機能を作って」と頼みます。Claude Codeはその場でファイルを開き、コードを書き始めます。指示が具体的であればあるほど、この着手は速くなります。

問題は、指示の外側にある判断です。パスワードの再発行はどうするか。セッションの有効期限は。既存のユーザーテーブルとどう繋ぐか。こうした判断は、指示に書かれていない限りClaudeが仮に決めます。仮に決めた結果は、書いた本人が見るまで妥当かどうか分かりません。

だから「動くには動いた。でも違う」が起きます。指示が悪かったわけではありません。**判断を先に固める工程が、そもそも存在しなかった**だけです。

## superpowersを入れると、何が変わるか

superpowersは Claude Code向けの拡張([obra/superpowers](https://github.com/obra/superpowers))で、ブレインストーミング・計画・TDD・レビューといった「進め方」をスキルとして持たせます。依頼を投げると、いきなり実装に入らず、まずこの順で進みます。

```
「〇〇を作って」
  → brainstorming              要件と判断を先に洗い出す
  → writing-plans               実装計画を文章化し、小さなタスクに分割する
  → test-driven-development     タスクごとに RED → GREEN → REFACTOR を繰り返す
  → requesting-code-review      完了前にレビューを依頼する
```

抽象的な流れだけだとイメージしにくいので、手元で開発しているX風SNSクローンに「返信機能」を追加したときの話をします。

依頼は「投稿に返信できるようにしてほしい」の一文でした。この一文の裏には、実は5つの判断が隠れています。詳細ページは作るのか。返信はタイムラインに流すのか。返信への返信はどこまで表示するのか。親投稿を消したら返信はどうなるのか。返信を書く導線はどこに置くのか。

brainstormingの段階でこの5つを一つずつ潰しました。詳細ページを新設し、タイムラインには最上位の投稿だけを流す。スレッドは1段だけ表示し、それ以上は遷移して見る。親を削除したら子も連鎖して消す。返信はモーダルではなく詳細ページ内のフォームで書く。ここまでを設計メモとして先に文章にしてから、実装計画に進みました。

実装計画は、この設計をそのままコードに変換するのではなく、7つの小さなタスクに分解されていました。DBスキーマの変更、返信件数を数えるロジック、API、投稿フォームの改修、一覧表示、詳細ページ、最終検証。1つのタスクごとにコミットが積まれます。

なかでも「返信件数を数えるロジック」のタスクは、そのままTDDの手順書になっていました。先に落ちるテストを書く。実際に落ちることを確認する。実装を書く。テストが通ることを確認する。この4ステップが、テストコード付きでタスクの中に書かれていました。5つの判断を大きな1本の実装として書くのではなく、小さなタスクに割って、それぞれを赤から緑にしていく。これが、素のClaude Codeとの一番の違いでした。

## 唯一気になった点:離席中は止まっていることがある

良いことばかりではありません。superpowersはClaudeの判断で危険な操作(破壊的なgitコマンド、本番反映など)を勝手に実行しません。安全のためですが、これには裏があります。

寝る前にまとまったタスクを投げて、朝起きると、まだ確認待ちで止まっていました。何かを壊したわけではありません。ただ、Claudeが「ここは人間に判断してもらったほうがいい」と止まった場所で、確認する人間がいなかっただけです。

長時間目を離すなら、確認が発生しそうな場面をあらかじめ想定して指示に含めておくか、承認の範囲を事前に整理しておく必要があります。これは欠点というより、安全側に倒した設計が要求する「準備」だと捉えています。

## 使い始めるときの設定:普段はSonnet、判断が必要な場面は自動でOpus

普段のモデルはSonnetにし、実装を始める前や「これで完了かもしれない」というタイミングでは、より強いモデル(Opus)に経緯ごと相談する仕組みを裏で動かします。使い始めてすぐこの組み合わせに落ち着きました。

* 普段のモデルは`/model`コマンドでSonnetに設定する
* `/advisor`コマンドで、判断が必要な場面の相談役をOpusに設定する
* どちらも設定するのは最初の1回だけです。あとは実装前・完了報告の前・行き詰まったときに、Sonnetが自分からOpusを呼びます。毎回`/advisor`と打つ必要はありません
* 常時Opusにするよりコストを抑えたまま、判断が必要な局面の質は落としません。

全部Sonnetで動かし、判断が必要な瞬間だけ裏でOpusが呼ばれる。今のところ、これで困ったことはありません。

## 今日契約した人へ

Claude Codeは、渡した指示の分だけ動きます。渡していない判断は、Claudeが仮に埋めます。これは今日も、この先も変わりません。

superpowersが変えたのは、その「渡していない判断」を実装前に引き出し、小さなタスクに割って一つずつ確かめていく工程を、毎回自分で用意しなくてよくした点です。1個1個のスキルを知っていても、必要なタイミングで自分から呼び出す運用は長続きしません。勝手に適切な場所に挟まってくる、というところに価値があります。

最初の「これを作って」を打ち込む前に、superpowersを入れておくことをすすめます。
