Claude Code は、ユーザーが 1 文字打つ前に 3 万トークン超を API に送っています。
この事前送信ぶんがプロンプトキャッシュ込みで実際にいくらになるのかを、自作の捕捉プロキシと API の usage 報告値で実測しました。
「キャッシュがあるから問題ない」という定番の反論が本当かどうか、API 従量課金で払う立場から数字で確かめます。

## 何を測るか

エージェント CLI がユーザーの入力前に送るトークン量(システムプロンプト+ツール定義)の比較が、海外コミュニティで大きな議論になりました。
毎ターン定価で再送されるなら高くつきますが、プロンプトキャッシュの読み出しは基本料率の 0.1 倍。
なら実害はどこにあるのか。

測るのは 3 点です。
初回リクエストで何トークンが送られ、何がキャッシュに書かれるか。
ターンやセッションを重ねたとき、ターンあたりの支払いがいくらに落ち着くか。
そして、キャッシュが無効になる条件と、そのとき何を払い直すか。

この数字は、定額プランの利用者には請求として現れません。
請求に直結するのは、API 従量課金で Claude Code を使う場合と、Agent SDK などで同等のハーネスを動かす場合です。
複数 CLI の横並び比較はしません。
量を並べる話は冒頭の議論で出尽くしていて、知りたいのは 1 つの CLI をキャッシュ込みで使い続けたときの支払いだからです。

## 計測設計

Claude Code は接続先を環境変数 `ANTHROPIC_BASE_URL` で差し替えられます。
そこでローカルにリバースプロキシを置き、リクエスト本文と SSE レスポンスをファイルに保存しながら本物の API へ転送しました。
TLS への介入が不要なので、証明書のインストールなしで動きます。

![capture-setup.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3456995/90cd823d-6e35-434d-a5ea-d2d46d37545a.png)

検証環境は次のとおりです。

- WSL2(Ubuntu 24.04 / カーネル 6.6.87.2-microsoft-standard-WSL2)
- Claude Code 2.1.216 / モデル claude-fable-5
- Go 1.23.4(プロキシの実装に使用)
- 計測日:2026-07-22

プロキシは 1 ファイルで完結します。

```go
package main

import (
	"bytes"    // リクエスト本文を転送用に包み直すため
	"fmt"      // 連番ファイル名の生成に使う
	"io"       // 本文の読み取りに使う
	"log"      // 捕捉サマリの出力に使う
	"net/http" // プロキシ本体
	"os"       // 捕捉ファイルの保存に使う
	"time"     // TTFB と所要時間の計測に使う
)

// 転送先(本物の API)
const upstream = "https://api.anthropic.com"

func main() {
	// 捕捉ファイルの保存先(引数で変更可)
	dir := "capture"
	if len(os.Args) > 1 {
		dir = os.Args[1]
	}
	// 保存先ディレクトリを作る
	os.MkdirAll(dir, 0o755)
	// リクエスト連番
	seq := 0
	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		// リクエスト連番からファイル名の接頭辞を作る
		seq++
		id := fmt.Sprintf("%03d", seq)
		// リクエスト本文を全部読んでファイルに残す
		body, _ := io.ReadAll(r.Body)
		os.WriteFile(fmt.Sprintf("%s/%s-req.json", dir, id), body, 0o644)
		// 上流へのリクエストを組み立てる(クエリ文字列も引き継ぐ)
		url := upstream + r.URL.Path
		if r.URL.RawQuery != "" {
			url += "?" + r.URL.RawQuery
		}
		req, err := http.NewRequest(r.Method, url, bytes.NewReader(body))
		if err != nil {
			http.Error(w, err.Error(), 500)
			return
		}
		// 認証ヘッダ等をそのまま引き継ぐ
		req.Header = r.Header.Clone()
		// 圧縮を無効化して生の SSE をそのまま記録できるようにする
		req.Header.Del("Accept-Encoding")
		// ここから上流との往復を計時する
		t0 := time.Now()
		resp, err := http.DefaultClient.Do(req)
		if err != nil {
			http.Error(w, err.Error(), 502)
			return
		}
		defer resp.Body.Close()
		// レスポンスヘッダとステータスをクライアントへ返す
		for k, vs := range resp.Header {
			for _, v := range vs {
				w.Header().Add(k, v)
			}
		}
		w.WriteHeader(resp.StatusCode)
		// レスポンス本文の保存先
		f, _ := os.Create(fmt.Sprintf("%s/%s-resp.txt", dir, id))
		defer f.Close()
		// ストリームを崩さないよう、読めた分だけ即座に返しつつ複製する
		buf := make([]byte, 32*1024)
		var ttfb time.Duration
		first := true
		for {
			n, rerr := resp.Body.Read(buf)
			if n > 0 {
				// 最初の1バイトが届いた時刻を TTFB として記録する
				if first {
					ttfb = time.Since(t0)
					first = false
				}
				w.Write(buf[:n])
				// バッファリングさせず即送出する(SSE を壊さないため)
				if fl, ok := w.(http.Flusher); ok {
					fl.Flush()
				}
				f.Write(buf[:n])
			}
			if rerr != nil {
				break
			}
		}
		// 1リクエスト分のサマリを出力する
		log.Printf("[%s] %s %s status=%d req=%dB ttfb=%dms total=%dms",
			id, r.Method, r.URL.Path, resp.StatusCode, len(body),
			ttfb.Milliseconds(), time.Since(t0).Milliseconds())
	})
	log.Println("listening on 127.0.0.1:8899 ->", upstream)
	// ループバックのみで待ち受ける(外部公開しない)
	log.Fatal(http.ListenAndServe("127.0.0.1:8899", nil))
}
```

実行は 3 コマンドです。

```bash
go mod init proxy && go build -o proxy .   # ビルド
./proxy capture &                          # プロキシ起動
ANTHROPIC_BASE_URL=http://127.0.0.1:8899 claude -p "1+1の答えだけを返して"
```

支払いの根拠になるのは、レスポンスに含まれる usage ブロックです。
捕捉した SSE から grep で取り出せます。

```bash
grep -h -o '"\(input_tokens\|cache_creation_input_tokens\|cache_read_input_tokens\)":[0-9]*' capture/003-resp.txt | sort -u
```

```text
"cache_creation_input_tokens":38254
"cache_read_input_tokens":0
"input_tokens":2
```

計測条件は「プロジェクトファイルも CLAUDE.md も無い空ディレクトリ、MCP サーバ無し」に統一しました。
トークン数は API が返す報告値で、独立した 3 セッションで同じ 38,254 が返ることを確認しています。

## 初回リクエストの中身:38,254 トークン、全部キャッシュ行き

初回の本命リクエストは本文 105,101 バイトでした。
内訳はツール定義が 29 個で 86,670 文字と全体の 8 割超を占め、システムプロンプトが 11,078 文字、環境情報などの初回メッセージ部が 7,703 文字(いずれもリクエスト JSON 上の文字数)。
usage 報告では `cache_creation_input_tokens` が 38,254、`input_tokens` はわずか 2 でした。

つまり Claude Code は、ユーザーのプロンプト本体まで含めた初回リクエストのほぼ全体を、TTL 1 時間のキャッシュとして書き込みます。
`cache_control` の切れ目は 3 箇所で、システムプロンプトの冒頭付近と末尾、そして初回メッセージ列の最後に置かれていました。
5 分 TTL(書き込み 1.25 倍)を想定して計算すると誤ります。
現行の Claude Code が要求するのは 1 時間 TTL。
書き込み単価は基本料率の 2 倍になります。

本命とは別に、712 トークンの副リクエストも毎セッション飛んでいました。
セッションタイトルの生成用で、こちらはキャッシュされません。

面白かったのは内訳の分解です。
専用トークナイザが無くても、キャッシュ境界で割れた write/read の値が、そのまま各部分のトークン数として読めます。
作業ディレクトリだけ変えて新規セッションを開始すると、usage は書き込み 6,177・読み出し 32,079 に分かれました。
ツール定義+システムプロンプトの 32,079 トークンはディレクトリが違っても共有ヒットし、環境情報を含むメッセージ部の約 6,177 トークンだけが書き直された、ということです。

![request-layout.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3456995/8813ae86-74e6-4b58-b0c5-b9cbc8ce7558.png)

| 実験 | write | read | input |
|---|---|---|---|
| 新規セッション ターン1 | 38,254 | 0 | 2 |
| 同一セッション ターン2(`--continue`) | 16 | 38,254 | 2 |
| 同一プロンプトの別セッション | 0 | 38,254 | 2 |
| 別ディレクトリの新規セッション | 6,177 | 32,079 | 2 |
| `--append-system-prompt` で41文字追記 | 38,306 | 0 | 2 |
| 60分放置後の再開(`--continue`) | 6,229 | 32,079 | 2 |

3 行目は測定ミスではありません。
同じプロンプトで別セッションを開始したら、書き込みゼロの全ヒットになりました。
キャッシュはセッションではなくプレフィックスに付くので、内容が一致すればセッションを跨いで効きます。
複数ターンの「書き込みサンプル」を取るつもりで同一プロンプトのセッションを 3 本実行したら、2 本目以降が全部読み出しになった…。
プレフィックスキャッシュの仕様どおりですが、計測設計としては見落としていました。

## 単価に換算する

claude-fable-5 の公式レート(2026-07-22 時点、[公式料金ページ](https://platform.claude.com/docs/en/about-claude/pricing)より)は、入力 $10 / 1h キャッシュ書き込み $20 / キャッシュ読み出し $1 / 出力 $50(いずれも 1M トークンあたり)です。
実測値を当てはめます。

![cache-economics.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3456995/e8eab232-2356-4ee0-be7b-6b8f2aca4ef7.png)

- ターン1:38,254 × $20/1M = **$0.765**
- ターン2以降:38,254 × $1/1M = **$0.038/ターン**
- キャッシュが無かった場合:38,254 × $10/1M = $0.383/ターン

10 ターンのセッションなら、オーバーヘッド分の支払いは $0.77 + 9 × $0.038 = 約 $1.11。
キャッシュ無しの想定 $3.83 に対して 7 割減です。
「キャッシュがあるから無問題」論は、継続するセッションに限れば数字で裏付けられます。

ただし損益分岐には注意点が 1 つ。
1h TTL の書き込みは 2 倍なので、上乗せ分(+1.0 倍)を読み出し(1 回あたり −0.9 倍)で回収するには 2 回、つまり 3 ターン目までかかります。
読み出しが 1 回も発生しないまま TTL が切れる使い方は、毎回 2 倍単価の書き込みだけを払う最悪のケース。
どの使い方がそこに落ちるかは、最後の節で整理します。

## キャッシュが剥がれる瞬間

本題はここからです。
継続セッションの $0.038/ターンという数字は、プレフィックスが 1 バイトも変わらないことが前提です。

試しに `--append-system-prompt` で短い一文(システムプロンプト+41 文字)を足して同じセッションを継続すると、usage は書き込み 38,306・読み出し 0。
システムプロンプトへの 41 文字の追記で、38,000 トークンぶんの再書き込み($0.77)が発生しました。
プレフィックスは tools → system → messages の順に並ぶため、前方に近い変更ほど破壊範囲が広くなります。
セッション途中で MCP サーバを足す・許可ツールを変えるといった操作も、同じ理屈で全書き込みに戻ります。

1 時間の放置も試しました。
セッションを放置してから `--continue` で再開すると、usage は書き込み 6,229・読み出し 32,079。
意外なことに、全量の再書き込みにはなりませんでした。
分解するとこうです。
会話履歴を含む部分は最終アクセスから 60.6 分が経過しており、TTL 切れで書き直し(6,229 トークン、$0.125)。
一方ツール定義+システムプロンプトの共有部分は、直前に実行していた別ディレクトリの実験が 59.5 分前に読み出していたため、TTL 内に収まってヒットしました。
この共有部分の最初の書き込みは 62.8 分前です。
書き込み起点の TTL なら消えているはずなので、読み出しのたびに TTL が更新されていることの実証になりました。
つまり同じマシンで Claude Code を使い続けている限り、共有部分(約 32k)が冷えることは稀で、放置で失うのは会話固有部(約 6k)だけです。
ただしマシン全体が 1 時間止まれば共有部分も消えるので、設計上は全量 $0.77 を見込むのが安全です。

もうひとつ、予想が外れた点があります。
キャッシュヒットなら TTFB(最初の 1 バイトが返るまでの時間)も目に見えて縮むと考えていました。
実測は、全量書き込みの 2 回が 2,226ms と 1,254ms、読み出し 5 回が 1,092〜1,656ms、部分書き込みを含む 3 回が 1,489〜6,480ms。
傾向としては読み出しが速いものの、この規模ではばらつきに埋もれます。
38k 程度のプレフィックスでは、キャッシュの価値はレイテンシではなくほぼコスト側にあると見ています。

## それで、38,000 トークンは問題なのか

実測を踏まえた私の答えはこうです。
「キャッシュがあるから無問題」は、使い方が 3 つの条件に収まっている間だけ正しい。
セッションを 3 ターン以上続ける。
システムプロンプト・ツール・MCP といった構成には途中で触らない。
そして 1 時間以内に戻ってくる。

条件から外れたとき何を払うのか、使い方別に並べます。

| 使い方 | オーバーヘッド分の支払い |
|---|---|
| 同じセッションでターンを重ねる | $0.038/ターン |
| 新しいセッションを立てる(直近 1 時間以内に利用あり) | $0.04〜$0.16(共有部分がヒットするため) |
| 1 時間超の間隔でワンショットを定期実行 | 毎回 $0.77(キャッシュ無し $0.38 の 2 倍) |
| セッション途中で構成を変える | その場で +$0.77 |

得をする条件は 1 つで、キャッシュが温かいうちに次のリクエストを発生させることです。
一番損なのは、1 時間を超える間隔で `claude -p` を 1 発ずつ実行する定期ジョブ。
読み出しが 1 度も発生しないまま TTL が切れるので、2 倍単価の書き込みだけを払い続けます。
この形を従量課金で運用しているなら、実行間隔を 1 時間以内に寄せるか、`--continue` で 1 本のセッションに繋ぐだけで支払いが変わります。
構成変更を伴う作業は、セッションの頭にまとめるのが安全です。

なお、コンテキスト圧迫という観点では 38k は claude-fable-5 の 1M ウィンドウの 4% 弱で、以前ほど深刻ではないかなと。

## 所感

オーバーヘッド論争は「何トークン送っているか」という量の話に終始しがちですが、測ってみると財布に効くのは量そのものより「いつ全書き込みが走り直すか」でした。
フラグ 1 つの追記で $0.77 が消えると知ってからは、セッション途中で設定をいじる手が少し止まるようになりました。
副産物として、キャッシュ境界の write/read 差分だけでリクエスト内訳をトークン単位で分解できたのは収穫です。
トークナイザが手元に無くても、usage の突き合わせだけでここまで割り出せます。
