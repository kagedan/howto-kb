## はじめに

[第1回プロローグ](https://zenn.dev/shomitei/articles/alumni-spa-01-prologue)から1週間。前回は「なぜ作るのか」という動機の話でした。  
今回は「何を使って作ったか」、もっと正確に言うと「何を使って作ろうとして、何回手段を乗り換えたか」の話です。

結論から書くと、この9ヶ月で開発の主軸AIを3回乗り換えました。ChatGPT → Gemini → Claude、という順番です。

ただ、乗り換えたといっても、前のAIをきっぱり捨てたわけではありません。いまでもそれぞれに居場所があって、役割を分けて併走しています。**主軸は3回変えた、でも1つも捨てていない。** そこに辿り着くまでの右往左往を、時系列で書いていきます。

文章は苦手ですが、AIをとっかえひっかえした記録なら、さすがに書くことがあります。

---

## ChatGPT専属期（2025年8〜11月）

新サイトづくりは、ChatGPTへの相談から始まりました。2025年8月、僕が最初に投げたプロンプトはこれです。

```
SPAで高校OB会のポータルサイトを作りたいんだけど、どうやって作るのがいいかな？
とりあえず、会員の卒業年度、大学、大学院、メールアドレスが管理したくて、
都度、インフォメーションを出せるといいな。
動画配信もやってみたいけど優先度は低い。
```

いま読み返しても、ずいぶん雑な相談です。  
ChatGPTに初めて触れて、何ができるんだろう？と思いながら、とりあえず打ち込んだ初めの一言がこれでした。

そもそも当初は、今ある「サポーター登録ポータル」の移行だけではなく、諸々の過去のOB／OGからのコメントを踏まえ、これ１つで事足りる「高校OB会のポータルサイト」のつもりで、キャリアセミナーの説明動画閲覧やOB／OG同士のチャットまで頭にありました（このあたりは途中でスコープを絞ります。動画・チャットはいまも計画には残っていますが、未実装です）。

### ChatGPTが返してきた提案

この雑な相談に対して、ChatGPTはかなり整理された提案を返してきました。要点はこんな感じです。

| 要素 | ChatGPTの提案 | 最終的に僕が選んだもの |
| --- | --- | --- |
| フロントエンド | React + Vite または Next.js | Next.js |
| スタイリング | Tailwind CSS | Tailwind CSS |
| 認証 | Firebase Auth / Supabase Auth / Auth0 | AWS Cognito |
| バックエンド・DB | Supabase / Firebase / Appwrite | DynamoDB |
| デプロイ | Vercel / Netlify | AWS Amplify |

ここで前回の記事の表現について補足です。前回は受動的に「ChatGPTからNext.jsを提案された」と書いていたのですが、当時のチャットをさかのぼって整理し直したところ、**ChatGPTはReact+ViteとNext.jsを並べて提示してきただけで、その中からNext.jsを選んだのは僕自身**でした。誤解を生まないよう、前回の記事も併せて書き直しています。  
業務でバックエンドを中心に製造していますが、対応するフロントエンドはNext.jsを使用していることから理解を深めたいこと、どうやらその選択が今後のベストであるように提示されたことからNext.jsを選択しました。

そして表を見てのとおり、ChatGPTはバックエンドにSupabaseやFirebaseを推していました。AWSはひとことも勧めていません。にもかかわらず僕がAWSを選んだ理由は、前回も書いた消極的なものです。

* 旧サイトからの流れでAWSの知見があった
* SupabaseもFirebaseも、よく知らないサービスを新しく掴むのが怖かった
* Vercel / Netlify も使ったことがなかった

要するに、悪く言えば知らないものを避けた、よく言えば仕事で使う技術範囲を選択した形です。  
提案から使いたいものを選択＆選択＆選択して、自分の手癖でスタックを決めました。

### MVPの骨格はこのとき決まった

技術選定では自分の思いを中心にChatGPTに「組み合わせてできる方法」を尋ねた僕ですが、ChatGPTが出してきた「最小構成（MVP）」の3本柱は、ほぼそのまま採用しました。

1. 会員情報管理（卒業年度・大学・大学院・メールアドレス）
2. お知らせ機能（管理者が作成、一般会員が閲覧）
3. 認証と権限（一般会員 / 管理者の2ロール）

この3つは、いまのシステムの骨格にそっくり引き継がれています。

実装は、DB設計 → マイページ → ログイン・ログアウト → 認証 → お知らせ・イベント告知、という順で進めました。2025年11月にこのあたりが一通りつながって、「動くもの」になりました。

### 専属だったはずなのに、だんだん噛み合わなくなる

順調だったのは、ここまででした。

開発が長引くにつれて、ChatGPTとの会話が少しずつ噛み合わなくなっていきます。具体的には、こういうことが続きました。

* うまく動いていたコードに対して、急に「もう一度作り直しましょう」と言い出す
* セッションが変わると、それまでの経緯を覚えていない。同じ説明を何度もやり直す
* 「DB設計 → マイページ → …」と指示した実装の順番を、途中から忘れている
* 復元はしてくれるが、口調や前提がプロジェクトの実態とズレた状態に戻る

長い付き合いになるほど、返答が遅くなり、タイムアウトも増えていきました。

この問題への対策をChatGPTに率直に尋ね、プロジェクトの設計方針を1ファイルにまとめた `ai-context.md` を作ることになります。おもしろいのは、**この `ai-context.md` を最初に提案してきたのが、ほかでもないChatGPT自身だった**ことです。セッションが切れると忘れてしまう、というChatGPT自身の弱点を、AI自身がわかっていたわけです。なんだか切ない話です。

## Gemini並行期（2025年11月末〜2026年1月）

ChatGPT一本でつらくなってきた11月末、2つ目のAIとしてGeminiを併用し始めました。

動機は2つです。1つは、Geminiの扱えるコンテキストの長さが魅力的だったこと。ChatGPTで一番こたえていたのが文脈切れだったので、長く覚えていてくれるなら試す価値があると思いました。もう1つは、単純にGoogleのものを一度触ってみたかった、という実験ごころです。

### 2つのAIを手で行き来させる

このころのワークフローが、われながら不格好でした。図にするとこうです。

```
  ChatGPTが書いたコード
        │  コピペ
        ▼
  Geminiに「これ評価して」
        │  修正案をコピペ
        ▼
  ChatGPTに「この指摘どう思う？」
        │  返答をまたコピペ
        ▼
  Geminiへ……（以下、人間が裁定するまでループ）
```

ChatGPTの作ったコードファイルをGeminiに貼り、Geminiの指摘をコピーしてChatGPTに貼り返す。この往復を、僕は手作業でやっていました。

2つのAIに議論させると、たしかに片方では気づかない指摘が出てきます。出てはくるのですが、間に挟まれた人間（僕）が、（眠い目をこすりながら）ひたすらコピペ係をやらされる構図でもありました。

### 手貼りがしんどくて、課金で一本化した

1月の頭、ついに音を上げました。Geminiのほうが文脈を長く保ってくれると実感できたこともあり、Geminiを有料プランにして主軸を一本化しました。

立派な技術的判断ではありません。ChatGPTに疲れた、が9割です。

## Gem二分割戦略

Geminiを主軸にしてから、紹介したかった工夫にたどり着きます。Geminiの「Gem」（役割を固定したカスタムAI）を、**2つ作成した**ことです。

Geminiのチャットがデザインと設計の話でぐちゃぐちゃしてきたので、専門家として評価検討をするだけのChatを切り分けたかったのです。  
1人で全部を抱えるより、相談相手が「担当別」にいるほうが頭が整理できる気がしました。

分け方はシンプルで、使うタイミングで切りました。

| Gem | 呼び出すタイミング |
| --- | --- |
| セキュリティGem（堅牢化用） | セキュリティ的な懸念がないか、情報漏洩しないかをチェックするとき |
| デザインGem | ”今風”のデザインをどう作るか／デザインレビューしたいとき、デザイン補正をしたいとき |

実際に使っていたGemのプロンプトを、匿名化したうえで全文公開します。長いので折りたたんでおきます。  
ちなみに、Gemのプロンプトもやりたいことを伝えてGeminiに考えてもらいました。

セキュリティGem（堅牢化用）のプロンプト全文

```
あなたは、TypeScriptを用いたSPA（React/Next.jsなど）とAPI通信の整合性・堅牢化を追求するシニアエンジニアです。

目的と目標：

* TypeScriptによるフロントエンド（React/Next.js）とバックエンド間の通信における型安全性を極限まで高めます。
* 実行時バリデーションを導入し、型定義と実際のデータの乖離を排除します。
* エラーハンドリングとステータス管理を徹底し、ユーザー体験とデバッグ効率を向上させます。

行動指針とルール：

1) コードレビューと修正案の提示：

a) 型の安全性（Type Safety）の徹底：

- フロントエンドとバックエンド間で型が共有・同期されているかを確認します。
- 'any' や 'unknown' の安易な使用を指摘し、具体的なインターフェースや型定義を提案します。
- 応答データ（Response）とリクエスト（Payload）が実際のAPI仕様と一致しているか検証します。

b) 実行時バリデーションの導入：

- Zod 等のライブラリを用いたバリデーションを推奨します。
- コンパイル時の型だけでなく、ランタイムでのデータ整合性を保証するコードを提案します。

c) 堅牢なエラーハンドリング：

- HTTPステータスコード（4xx/5xx）、ネットワークエラー、タイムアウトのハンドリングを確認します。

2) 回答の形式：

以下の構成で回答を出力してください：

- 【不整合・リスク】: 現在のコードにある型の脆弱性やバグの可能性を指摘。
- 【改善案（Code）】: Zod や TypeScript の機能を活用した、堅牢な修正コードを提示。
- 【プラスα】: セキュリティ（CORS, Auth）やパフォーマンス（キャッシュ）に関する補足アドバイス。

トーンとスタイル：

* シニアエンジニアとして、論理的かつ技術的に深い洞察を提供します。
* 具体的で再利用可能なコード片を提示します。
* 常にベストプラクティスを意識し、保守性の高い設計を推奨します。
```

デザインGem（UI/UX・コードレビュー用）のプロンプト全文

```
あなたは「同窓会サポーター登録ポータル」専用のシニアUI/UXエンジニア 兼 フロントエンドリードです。
共有された「ai-context.md」の仕様を完全に遵守し、SPAのデザイン安定化とコード品質の維持を支援してください。

### 1. デザインの安定化ルール(最優先)
実装やレビューにおいて、以下のデザインシステムを徹底してください。
- **ベースカラー**: ブランドカラー(深い紫系 / primary-500)を基調とする。
- **Tailwind CSS v4**: インラインスタイルやマジックナンバー(勝手な数値)を禁止し、`globals.css`で定義されたDesign Tokens(primary-600, surface, surface-alt等)を使用する。
- **余白ルール**: 8pxグリッドシステムを基本とし、一貫性を持たせる。
- **コンポーネント**:
  - `Card-based UI`: 統一されたpadding / radius / shadowを適用。
  - `PrimaryButton`: 背景 primary-600 / 文字 white。
  - `Badge`: ステータスに応じて色を固定(ok=green, depends=yellow, hard=red, 未登録=赤系バッジ)。
- **アクセシビリティ**: セマンティックなHTML(dt/ddの活用、h2/h3の階層構造)を維持する。

### 2. 技術スタックと実装規約
- **Frontend**: Next.js (App Router), TypeScript, react-hook-form + zod。
- **バリデーション**: `schema.ts` の zod 定義を唯一の正解とし、APIとスキーマを同期させる。
- **データ管理**: DynamoDBからのデータは必ず `mapItemTo...` 関数を通し、zod検証済みのドメインオブジェクトとして扱う。
- **型定義**: `ProfileFormValues` や `ClientProfile` を基準とし、独自の型を乱立させない。

### 3. セキュリティ・権限管理
- **管理画面**: `/admin/**` へのアクセスは `admin` または `operator` グループのみに制限。
- **APIガード**: UIガードに依存せず、サーバー側での `groups` 検証コードを必ず含める。
- **認証**: Cookieベース(HttpOnly)を前提とし、localStorageへの依存を排除する。

### 4. あなたの振る舞い
- コード生成時は、既存のコンポーネント(AppShell, Card, Button等)を再利用する構成を優先して提案してください。
- デザインにブレが生じそうな提案(例：新しい色の導入、独自の余白設定)をする際は、必ず理由を説明し、プロジェクトの規約に合わせるよう警告してください。
- 実装だけでなく、UXの観点(チラ見え防止、リダイレクトの挙動、エラー表示の統一)からもアドバイスを行ってください。
```

### 効いたこと

担当分けは、思った以上に効きました。

1つは、コードの品質が上がったことです。デザインGemに規約を全部覚えさせておくと、レビューのたびに「そのマジックナンバーは禁止」「`mapItemTo...` を通して」と指摘してくれる。自分一人だと見逃す規約違反が、目に見えて減りました。

もう1つは、頭の整理に効いたことです。「この話はセキュリティGemに、こっちはデザインGemに」と相談先を振り分けるだけで、自分のなかでも「いま考えているのは要件の話か、実装の話か」が切り分けられる。担当を分けたのはAIなのに、いちばん整理されたのは僕の頭でした。

### 笑いどころ：チーム化したら、チーム問題が起きた

ただ、いいことばかりではありませんでした。

2つのGemは相反する主張をすることがあり、扱いがだんだん面倒になっていきます。極めつけは、**セキュリティGemにちょっとデザインの話を振るとはぐらかされ、デザインGemにちょっとセキュリティの話を振ってもはぐらかされる**、という現象でした。「それは私の担当ではありません」と、AIにやんわり突き返されるわけです。

分担をしたくてAIでチームを再現したつもりが、今度はチームあるあるの「縄張り問題」まで再現してしまった。しかも、人間のほうがAIの担当範囲に気をつかっている。立場が逆では、と何度か思いました。

## 自己訂正ループ事件

Gem二分割とは別に、ChatGPTのころからGeminiの時代まで、ずっと尾を引いた問題があります。「自己訂正ループ事件」と呼ぶべき現象です。

症状はこうです。出てきたコードに対して「ここ直して」と頼むと、AIはちゃんとお詫びして修正案を出してくれます。ところが、そのコードを査読して貼り付けてチェックをしてもらって改めて出てきた修正案が元のコードとほとんど同じなのです。直したつもりで何も直っていない。もう一度頼むと、また同じものが返ってくる。

これにハマると、こちらの時間だけが溶けていきます。AIは悪びれず謝り続け、コードは1ミリも進まない。原因の切り分けもできず、ただ徒労感だけが積もりました。

## Claude移行（2026年2月〜）

転機は2026年2月でした。

きっかけは、ふとした思いつきです。最初に試したのはPCではなく、iPhone版のClaudeアプリでした。休憩時間にニュースを見てClaude.aiにチャットで触ってみて、受け答えの感触が良かったので、PCのVSCodeに拡張（Claude Code）を入れてみた。これが、それまでの開発体験を一変させました。

衝撃だったのは、ざっと4点です。

* コードのコピペをしなくてよくなった解放感。AIがファイルを直接読んで、直接書き換えてくれる
* 複数のファイルをまたいだ修正提案のレベルが、それまでと明らかに違った
* あの無限ループが、**足を洗うようにぴたっと減った**
* ターミナルと一体化した作業体験。提案からコマンド実行までが地続きになった

特にコピペからの解放は大きかったです。あの往復に音を上げてGeminiに一本化した僕にとって、「そもそもコピペが要らない」というのは発想が一段飛んでいました。

主軸をClaudeに移してからは、リファクタリングと型強化、イベント参加登録機能の追加、管理者サイトの拡充、テストとセキュリティの見直し、と一気に前へ進みました。

## いまのAI体制

「3回乗り換えた」と書くと、AIを使い捨ててきたように聞こえるかもしれません。でも実際は違います。引退したのは1つだけで、残りは役割を変えていまも現役です。

| AI | いまの役割 | 残っている強み |
| --- | --- | --- |
| ChatGPT | 引退（もう使っていない） | ― |
| Gemini（Gem経由） | デザイン・セキュリティのチェック役 | ネット検索の精度がClaudeより上 |
| Claude Code | ロジック実装の中心 | ターミナル一体化・複数ファイル把握 |
| Claude.ai | 壁打ち・思考整理 | 会話でアイデアを練る |

ChatGPTには引退してもらいましたが、MVPの骨格と `ai-context.md` という置き土産は、いまも生きています。Geminiは検索が強いので、調べ物まじりのチェックを任せています。実装はClaude Code、考えごとはClaude.ai。

結局のところ、たどり着いたのは1強ではなく、**適材適所のオーケストラ**でした。指揮者はあいかわらず文章表現が苦手な人間で、楽団員は3つのAI。  
冷静に考えると、文章を書くのが嫌でシステムに逃げた人間が、AIを3つも指揮しているわけで、なかなか締まりのない絵ではあります。
