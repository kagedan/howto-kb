`CLAUDE.md` を置いたのに指示が効かない。よくある原因は「中身」ではなく「どこに・どの順で読み込まれるか」を押さえていないことです。CLAUDE.md は**毎セッション冒頭にコンテキストへ読み込まれる指示ファイル**で、複数の場所に置けて、置き場所ごとにスコープと読み込み順が決まっています。

この記事では、CLAUDE.md を**読み込みの仕組みから逆算して設計**します。4つのスコープの読み込み順、実務で書く三層のテンプレ、`@import` の解決規則、そして壊れがちな書き方を before→after の diff で整理します。「とりあえず書く」から「効くように配置する」へ寄せるのがゴールです。

!

**対象読者と前提**

* Claude Code をプロジェクトのルートで起動できるエンジニア
* すでに `CLAUDE.md` を1つは書いたが、指示の効き方に納得していない人
* `/init`・`/memory` の出力例はすべて「想定例」です。バージョンや環境で文言は変わります

## 結論：CLAUDE.md は「強制」ではなく「毎回読まれる文脈」

先に肝を1つ。CLAUDE.md は設定ファイルのように**強制されるものではなく**、毎セッション冒頭にコンテキストとして差し込まれる「文脈」です。だから内容が曖昧だったり矛盾していると、従われ方も揺れます。

公式ドキュメントもこの性質を明言しています。**「コミット前に必ず lint を回す」のように特定タイミングで確実に実行させたい指示は、CLAUDE.md ではなく hooks に書く**のが正解です（hooks はライフサイクルイベントでシェルコマンドとして実行され、Claude の判断に関係なく走ります）。CLAUDE.md は「振る舞いを寄せる」ためのもの、と割り切ると設計がぶれません。

## 1. 4つのスコープと読み込み順を実パスで掴む

CLAUDE.md は次の4スコープに置けます。読み込み順は**広いスコープ → 狭いスコープ**。後に読まれた指示ほど「直近の文脈」として効きます。

| スコープ | 置き場所 | 共有範囲 |
| --- | --- | --- |
| managed（組織ポリシー） | Linux/WSL: `/etc/claude-code/CLAUDE.md` ／ macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md` ／ Windows: `C:\Program Files\ClaudeCode\CLAUDE.md` | 組織の全ユーザー |
| user（個人・全プロジェクト） | `~/.claude/CLAUDE.md` | 自分だけ（全プロジェクト） |
| project（チーム共有） | `./CLAUDE.md` または `./.claude/CLAUDE.md` | リポジトリのチーム |
| local（個人・このプロジェクト） | `./CLAUDE.local.md`（`.gitignore` 推奨） | 自分だけ（このプロジェクト） |

さらに、ディレクトリ階層も効きます。作業ディレクトリから**ルート方向へ親をたどって** `CLAUDE.md` / `CLAUDE.local.md` を探し、見つかったものを**全部連結**します。順序はファイルシステムのルート側 → 作業ディレクトリ側。つまり「起動した場所に近い指示ほど後に読まれる＝強く効く」と理解できます。

読み込み順（foo/bar/ で起動した場合・上から順に連結）

```
~/.claude/CLAUDE.md          # user（全プロジェクト）
foo/CLAUDE.md                # project（親）
foo/bar/CLAUDE.md            # project（作業ディレクトリ。最後＝直近の文脈）
foo/bar/CLAUDE.local.md      # local（同じ階層では CLAUDE.md の後に追記）
```

## 2. 実務で書くのは「三層」——テンプレで分担を固定する

managed は組織配布なので、個人が日々書くのは実質 **user / project / local の三層**です。役割を混ぜると破綻するので、層ごとに「書くもの」を固定します。

### user 層（`~/.claude/CLAUDE.md`）：自分の癖だけ

全プロジェクトに効くので、**プロジェクト固有の事実は書かない**のが鉄則です。

~/.claude/CLAUDE.md

```
# 個人の共通設定（全プロジェクト）

- 日本語で応答する
- コミットメッセージは Conventional Commits（feat:/fix:/docs: …）に従う
- 破壊的変更の前に必ず差分の要約を出す
```

### project 層（`./CLAUDE.md`）：チームで共有する事実

ここが本丸です。**新しいメンバーに毎回説明することを書く**と決めると、内容が安定します。具体的に書けるほど従われやすくなります。

./CLAUDE.md

```
# プロジェクト共有メモリ

## ビルド・テスト
- セットアップ: `make setup`
- テスト: `npm test`（コミット前に必ず実行）
- 型チェック: `npm run typecheck`

## 配置規約
- API ハンドラは `src/api/handlers/` に置く
- インデントは 2 スペース

## やらないこと
- `main` へ直接 push しない（PR 経由のみ）
```

### local 層（`./CLAUDE.local.md`）：自分のサンドボックス

検証用 URL やお試しデータなど、**コミットしたくない個人設定**。`.gitignore` への追加を忘れずに。

./CLAUDE.local.md

```
# 個人ローカル設定（コミットしない）

- 動作確認は http://localhost:5173 を使う
- ダミーデータは ./fixtures/local/*.json
```

## 3. `@import` で分割する——解決規則を外さない

CLAUDE.md が長くなったら `@path/to/import` で外部ファイルを取り込めます。取り込まれたファイルは起動時に展開され、参照元と一緒にコンテキストへ載ります。

./CLAUDE.md

```
プロジェクト概要は @README を、npm コマンドは @package.json を参照。

# 追加指示
- git ワークフロー @docs/git-instructions.md
```

外しやすい規則が3つあります。

1. **相対パスの基準は「import を書いたファイル」**。作業ディレクトリ基準ではありません。
2. **再帰インポートは最大4ホップ**まで。深いネストは展開されません。
3. **コードスパン／フェンス内はインポートされない**。パスを「文字として」書きたいときはバックティックで囲みます（`` `@README` `` はリテラル、`@README` はインポート）。

## 4. でかくなったら `.claude/rules/` でパススコープ化

CLAUDE.md は**1ファイル200行以内が目安**。超えるとコンテキストを食い、従われ方も落ちます。トピックごとに割りたいときは `.claude/rules/` にファイルを分けます。

ディレクトリ構成

```
your-project/
├── .claude/
│   ├── CLAUDE.md          # 常時読む共通指示
│   └── rules/
│       ├── testing.md     # テスト規約
│       └── api-design.md  # API 規約（パススコープ）
```

`paths` フロントマターを付けると、**マッチするファイルを触ったときだけ**読み込まれます。常時コンテキストに載せたくない規約はこちらへ。

.claude/rules/api-design.md

```
---
paths:
  - "src/api/**/*.ts"
---

# API 開発ルール
- すべてのエンドポイントで入力バリデーションを行う
- エラー応答は共通フォーマットを使う
```

`paths` を書かない rules は無条件で起動時に読まれ、優先度は `.claude/CLAUDE.md` と同じです。モノレポで他チームの CLAUDE.md まで拾ってしまう場合は `claudeMdExcludes` で除外できます。

## 5. 壊れる書き方（アンチパターン）を diff で直す

### アンチパターン1：曖昧で検証できない指示

「ちゃんと」「適切に」は効きません。**検証できる具体**に直します。

./CLAUDE.md

```
- - コードはきれいにフォーマットする
- - テストはしっかり書く
+ - インデントは 2 スペース
+ - コミット前に `npm test` を実行する
```

### アンチパターン2：層をまたいだ矛盾

user 層と project 層で食い違う指示があると、Claude はどちらかを**任意に選びます**。定期的に `/memory` で読み込まれているファイル一覧を確認し、矛盾を消します。

~/.claude/CLAUDE.md（user 層から「プロジェクト固有」を抜く）

```
- - パッケージマネージャは pnpm を使う   # ← プロジェクト依存。project 層へ移す
+ （user 層には個人の普遍的な好みだけを残す）
```

### アンチパターン3：人間向けメモを本文に直書き

block レベルの HTML コメント（`<!-- ... -->`）は、**コンテキストへ注入される前に除去**されます。メンテナ向けメモはコメントに入れれば、トークンを消費しません（コードブロック内のコメントは保持されます）。

./CLAUDE.md

```
- メンテナへ: この規約は 2026Q3 に見直し予定（このままだとトークンを食う）
+ <!-- メンテナへ: この規約は 2026Q3 に見直し予定（注入前に除去される） -->
```

## 6. 出力で確認する：`/init` と `/memory`

ゼロから書くなら `/init` がたたき台を作ります。コードベースを分析して、ビルド・テストコマンドや規約を拾った CLAUDE.md を生成します（既存があれば上書きせず改善提案）。

Claude Code セッション内（想定例）

```
> /init

Analyzing codebase...
Created CLAUDE.md with:
  - Build & test commands (npm install / npm test)
  - Project structure (src/, tests/)
  - Detected conventions (2-space indent, ESLint)
Review and refine as needed.
```

いま何が読み込まれているかは `/memory` で確認できます。指示が効かないときの一次切り分けは「そもそもロードされているか」です。

Claude Code セッション内（想定例）

```
> /memory

Loaded memory files:
  ~/.claude/CLAUDE.md            (user)
  ./CLAUDE.md                    (project)
  ./.claude/rules/testing.md     (rule)
Select a file to open in your editor.
```

`/memory` の一覧に出てこないファイルは、Claude からは見えていません。置き場所（前掲の4スコープ）を疑います。

## まとめ

* CLAUDE.md は**強制ではなく毎回読まれる文脈**。確実に走らせたい処理は **hooks** へ
* スコープは **managed → user → project → local** の順に読まれ、**後に読まれた指示が強い**
* 日々書くのは **user/local＝個人、project＝チーム共有**の三層。役割を混ぜない
* 分割は `@import`（最大4ホップ・相対は import 元基準）、容量削減は **パススコープ rules**
* 曖昧・矛盾・人間向けメモ直書きを避け、`/init`・`/memory` で「効く配置」を検証する
