## はじめに

AIエージェントのオーケストレーションツール（OSS）「**TAKT**」のしくみと利用方法についてを解説します。本連載では、TAKTを実際の開発に使って得た知見を、**解説（第1部）→実体験（第2部）→同一プロンプトでのClaudeとの比較（第3部）** の3部構成で記事を書こうと考えています。  
第1部の本記事では、まずTAKTが何をするツールなのか、どう動くのか、どう使うのかを整理します。

先に、連載全体の結論を書いておきます。

**オーケストレーションは「人間の都度確認」を「ワークフロー定義」に置き換えます。タスクを丸投げして完了まで放置できる代わりに、指示書とワークフローの設計が人間の新しい仕事になります。**

※ 読者として、Claude Code等のAIコーディングツールの利用経験がある方を想定しています（LLMやAIコーディングの基礎説明は省きます）。

TAKT（**TAKT Agent Koordination Topology**）は、nrslib氏が公開しているOSSです（MITライセンス / TypeScript製）。

<https://github.com/nrslib/takt>

GitHub上の説明文は次のとおりです。

> Define how AI agents coordinate, where humans intervene, and what gets recorded — in YAML

「AIエージェントがどう協調し、人間がどこで介入し、何が記録されるかをYAMLで定義する」。この一文がTAKTのほぼすべてを表しています。

## なぜTAKTを使ったのか

きっかけは単純で、**オーケストレーション環境を一から自作するのは難しい**と感じたことです。

Claudeを単体で使っていると、計画させて確認し、実装させて確認し、レビューさせて確認し……と、工程の節目ごとに人間の確認と指示出しが発生します。この進行役の仕事ごと仕組みに任せたいのですが、自作しようとすると、ステップ遷移・コンテキストの受け渡し・レビューの強制・ループの暴走検知・実行記録までを全部自前で作ることになります。

TAKTにはこれらが最初から揃っていました。READMEに書かれた設計思想は「AIエージェントは単に信頼するのではなく、外側から制御する対象として扱う」というもので、プロンプトでお願いするのではなく、宣言的なワークフロー定義でプロセスを強制します。作者自身も「AIの見張り番をやめよう」という記事でこのツールを公開しており、まさに私が解決したかった課題でした。

<https://zenn.dev/nrs/articles/c6842288a526d7>

リポジトリやコミュニケーション等が活発に動いており、今回の採用を決めました。

## オーケストレーションツールとは何か

Claudeを普通に使う場合との違いです。

|  | Claude単体 | オーケストレーションツール |
| --- | --- | --- |
| 進行 | 人間が都度指示・確認する | ワークフロー定義が進行を制御する |
| レビュー | 人間が依頼して実施させる | 定義されたループとして強制される |
| 完了まで | 人間が張り付く | 放置できる |
| 記録 | 会話ログ | 各ステップの指示・応答・レポートが構造的に残る |

つまり「誰が・何をして・結果がどうなったら次にどこへ進むか」をあらかじめ定義しておき、進行をツールに任せるのがオーケストレーションツールです。

なお、この領域のOSSはTAKTだけではありません。たとえばマルチエージェントワークフローの決定論的オーケストレーションを掲げる[Microsoft Conductor](https://opensource.microsoft.com/blog/2026/05/14/conductor-deterministic-orchestration-for-multi-agent-ai-workflows/)、並列コーディングエージェント向けの[agent-orchestrator](https://github.com/AgentWrapper/agent-orchestrator)（旧ComposioHQ、現AgentWrapper帰属）、ロールベースのマルチエージェントを協調させる汎用プラットフォームの[CrewAI](https://crewai.com/open-source)などがあります。

## TAKTのしくみ

TAKTという名前は音楽用語に由来します（taktはドイツ語で「拍子」、指揮で全体のテンポを合わせる語です）。その世界観に沿って、もともとは **ワークフロー全体をpiece（ピース）**、**その中の個々のステップをmovement（ムーブメント）** と呼んでいたようです。指揮者が楽曲（piece）の各楽章（movement）を順に導く、というイメージですね。

ただ、この音楽由来の呼び名は少し直感に反して飲み込みづらかったのか、現在のTAKTでは名称が改められ、**ワークフロー全体を「workflow」、個々のステップを「step」** と呼びます。本記事でも以降は、現行のworkflow / stepという呼び方で進めます。指揮者が楽曲を導くように、TAKTは定義に従って各stepを順に進めていく——という世界観自体は変わっていません。

### ワークフローはYAMLで宣言する

TAKTのワークフローはYAMLで定義します。READMEに掲載されている例がそのまま分かりやすいので引用します。

```
name: plan-implement-review
initial_step: plan
max_steps: 10

steps:
  - name: plan
    persona: planner
    edit: false
    rules:
      - condition: Planning complete
        next: implement

  - name: implement
    persona: coder
    edit: true
    required_permission_mode: edit
    rules:
      - condition: Implementation complete
        next: review

  - name: review
    persona: reviewer
    edit: false
    rules:
      - condition: Approved
        next: COMPLETE
      - condition: Needs fix
        next: implement    # ← fix loop
```

各ステップにpersona（planner / coder / reviewerなどの役割）を割り当て、`edit`でファイル編集の可否を絞り、`rules`に「条件 → 次のステップ」を宣言します。`review`で「Needs fixならimplementに差し戻す」と書いてあるとおり、**レビュー⇄修正のループが定義として強制される**のがポイントです。

実行エージェントはClaude Code / Claude Agent SDK / OpenAI Codex / opencode / GitHub Copilot CLI / Cursor Agent / Kiro CLIなど8種に対応しています（README記載）。また、ビルトインで推奨ワークフロー6種（default / default-mini / default-high / frontend / backend / dual）とペルソナ28種が同梱されており、最初は手ぶらで始められます。

### なぜ自律的に実行できるのか

[公式のworkflowsドキュメント](https://github.com/nrslib/takt/blob/main/docs/workflows.md)によると、自律実行を支えるしくみは大きく3つです。

1. **遷移ルールの自動判定（3方式）**: 「次にどのステップへ進むか」を、人間ではなくエージェントの出力から自動で判定します。判定方式は3つあり（下表）、終端は`COMPLETE`（成功）/ `ABORT`（失敗）として明示されます。
2. **ステップ間のコンテキスト受け渡し**: 前ステップの出力が`{previous_response}`として次ステップの指示に自動注入されます。レポートを`output_contracts`でファイルとして書き出し、後続ステップから参照させることもできます。サブワークフローの呼び出し（`workflow_call`）もあります。
3. **ループ監視**: `loop_monitors`がreview→fix→reviewのような反復を検出し、しきい値を超えるとAIジャッジが「進捗があるか」を評価して継続/中断を判断します。

遷移判定の3方式（しくみ①）は、次のように使い分けます。

| 方式 | 判定のしかた |
| --- | --- |
| タグベース判定 | エージェント出力の`[STEP:N]`タグで次ステップを指定する |
| AIジャッジ | `ai("条件文")`で、出力が条件を満たすかをAIが判定する |
| 集約（`all()` / `any()`） | 並列サブステップの結果を`all()`（全成功）/ `any()`（いずれか成功）で束ねて分岐する |

先ほどのYAML例の遷移を図にすると、次のようなループになります。

reviewの結果がApprovedなら終端`COMPLETE`へ、Needs fixならimplementへ差し戻し——という遷移を、人間ではなく定義が決めます。「条件分岐」「受け渡し」「無限ループ対策」が揃っているので、人間が次の指示を出さなくても完了（または中断）まで進む、という理屈です。

もうひとつ面白いのが[Faceted Prompting](https://github.com/nrslib/takt/blob/main/docs/faceted-prompting.md)という設計です。プロンプトを5つの独立した関心事（facet）に分解し、ステップごとに宣言的に合成します。

| Facet | 担うもの |
| --- | --- |
| Persona | エージェントの役割・専門性 |
| Policy | 禁止事項・品質基準・優先順位 |
| Instruction | ゴール・ステップ固有の手順 |
| Knowledge | ドメイン文脈・参照資料・API仕様 |
| Output Contract | 出力構造・レポートテンプレート |

ドキュメントいわく、ソフトウェア工学の「関心の分離」をプロンプト設計に適用したものです。同じPersonaを別のPolicy / Instructionと組み合わせて再利用でき、各ステップには現フェーズに必要な知識・制約だけが渡されるため、コンテキストの肥大化を防げます。たとえば先のYAML例に出てきた`planner` / `coder` / `reviewer`といったペルソナを、ステップやプロジェクトが変わってもそのまま流用し、その都度のPolicy・Instructionだけを差し替える——というイメージです。

## 使い方

インストールはnpm一発です。

`~/.takt/config.yaml`に実行エージェントとモデルを書きます。

```
provider: claude
model: sonnet
language: en
```

普段の運用はほぼ2コマンドです。

```
takt      # 対話的にタスクを定義（ワークフロー選択 → タスク記述 → キューへ）
takt run  # キューしたタスクを自律実行
```

ほかにも、用途に応じた補助コマンドが用意されています（いずれもREADME記載）。

| コマンド | 何をするか |
| --- | --- |
| `takt add #6` | GitHub Issueをそのままタスクにする |
| `takt list` | タスクやブランチを管理する |
| `takt eject` | ビルトインワークフローをコピーしてカスタマイズする |

## まとめ

第1部では、TAKTのしくみと使い方を解説しました。

* TAKTは「AIエージェントを外側から制御する」オーケストレーションOSSです。ワークフロー（workflow）全体をYAMLで宣言し、遷移ルール（タグ／AIジャッジ／集約）・ループ監視・実行記録が最初から揃っています
* 各ステップ（step）にpersona・edit・rulesを割り当てることで、レビュー⇄修正のループを定義として強制できます。Faceted Promptingでプロンプトを関心事ごとに分解・再利用できます
* インストールは`npm install -g takt`、運用は`takt`（タスク定義）と`takt run`（自律実行）のほぼ2コマンドです

冒頭の結論をもう一度書きます。**オーケストレーションは「人間の都度確認」を「ワークフロー定義」に置き換えます。** 丸投げして放置できる代わりに、指示書とワークフローの設計が人間の新しい仕事になります。
