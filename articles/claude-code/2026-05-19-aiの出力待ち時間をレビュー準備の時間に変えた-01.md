## はじめに

Claude Code や Cursor にコードを書かせると、数十秒から数分の待ち時間が生まれます。  
最初、自分はその時間をただの待ち時間だと思って、生成中の画面をぼんやり見たり、Slack を見たり、X を開いたりしていました。

ただ、ターミナルに戻ると自分が何を確認するつもりだったのか忘れたり、記憶が薄れていることに気づきました。

前回、[AI に全部書かせたら自分には何も残らなかった話](https://zenn.dev/kaji_kaji/articles/b78aa0d793399a) を書きました。  
その続きとして、AI で書いたコードをレビューで説明できなかった話も書きました。

そのときは、AI に投げる「前」の準備、つまり設計メモや PR 説明、テスト観点を自分で言語化することに焦点を当てていました。

今回はその次の話をできればと思います。  
AI に投げた「あと」、出力を待っている時間をどう使うか。  
ここを変えたら、AI の出力との付き合い方がだいぶ変わる気がします。

## 待ち時間に文脈を手放すと、AI の出力に流される

AI依頼後に皆さんはどのように時間を使っているでしょうか。  
私は、ついSlack、X、ブログ、別Issue、別PRのレビューをしてしまい  
戻ってきたときには、何を依頼したのか、何を確認するつもりだったのか忘れていることがあります。

その状態で AI の出力を見ると、 コードはそれっぽく整っていて、ローカルでも動くので 「動きそう」「読めそう」で受け入れてしまいます。  
AI の出力は形になっている分、判断を持っていない自分には余計に正しく見えてしまいます。

これは単なる気分の問題だけではないようです。  
タスクを切り替えたあとも元のタスクへの注意が一部残る、Attention Residue という考え方があります。  
自分の体感としても、AI の待ち時間に Slack を見たあとは、戻ってきたコードを読むときに文脈へ戻るのが明らかに遅くなっています。

待ち時間に別作業をすると、出力後に AI の結果だけをみることになりますが、  
何を依頼したかを遡る→結果を理解する→説明を読む→質問をする  
なんとなくわかった気になる。てか疲れる。。みたいな感じになっているのが私の現状でした。

## AI の待ち時間は、レビュー準備の時間だった

ある時、待ち時間を「暇な時間」ではなく「レビュー準備の時間」として考えたらどうだろうと思いました。

AI がコードを書いている間に、自分は何をもって OK とするかを決めます。  
目的・責務・失敗予想・テスト観点・レビュー質問。  
このあたりを、待ち時間のうちに数行のメモに落としておきます。  
もしくは先に基準を決めておいて、出力中は頭の中でこのようなコードが返ってくるだろうと予測を立てます。

そうすることで、今までは AI の出力を「採用するかどうか」で見ていたのが、「自分の基準と合っているか」で見られるようになりました。  
個人的にはこれは大きい違いでした。  
レビューする側に立てると、AI のコードに対して「ここは違う」と言えるようになります。

## 待ち時間の長さ別にやっていること

毎回フルでやる必要はあるとは思いませんが、待ち時間の長さによって、やることを変えています。

| 待ち時間 | やること | 目的 |
| --- | --- | --- |
| 〜30 秒 | 依頼内容を読み返す | 何を頼んだか忘れない |
| 1〜3 分 | AI がやりそうな失敗を 3 つ書く | 出力後のレビュー観点を作る |
| 3〜5 分 | テスト観点・レビュー質問を書く | 「動いたから OK」を防ぐ |
| 5 分以上 | 戻るためのメモを書いてから休憩する | 文脈を捨てずに休む |

### 〜30 秒：依頼内容を読み返す

短い待ち時間ではメモを書く余裕がありません。  
代わりに、自分が AI に何を頼んだのかを読み返します。

* 何を頼んだのか
* 何を期待しているのか
* どのファイル・どの関数に関係するのか

これを読み返すだけでも、戻ってきた出力の置き場所がイメージできる感覚が掴めました。  
プロンプトをスクロールして見直す、それくらいで十分かなと思います。

### 1〜3 分：AI がやりそうな失敗を 3 つ書く

少し時間があるときは、AI がやらかしそうなことを先に予想して書きます。  
3 つで十分です。多すぎると書いている間に出力が返ってきます。

たとえば、こういう失敗を予想することが多いです。

* 既存の責務分割を無視する
* エラーケースを浅く扱う
* テストを書いたように見えて重要なケースが抜ける
* UI 側にドメインロジックを寄せすぎる
* 既存の命名規則とズレる
* 過剰に汎用化する

予想を書いておくと、出力を見たときに違和感を拾いやすくなります。  
予想が外れるのはむしろ良いことで、AI が思ったより踏み込んだ提案をしてきたサインになります。

### 3〜5 分：テスト観点・レビュー質問を書く

もう少し長いときは、テスト観点とレビュー質問を書きます。  
これは前回の記事で書いた「投げる前の準備」と少し被りますが、待ち時間で書くものはもっと粗くてかまいません。

テスト観点なら、こんな粒度で並べます。

* 正常系
* 異常系
* 権限なし
* データなし
* 境界値
* 既存挙動の回帰

レビュー質問は、自分がレビュアーだったら何を聞くかを書きます。

* なぜこの層に置いたのか
* この責務はここでよいのか
* このテストで十分か

書いておくと、出力後のセルフレビューがそのまま PR 説明文の素材にもなります。

### 5 分以上：戻るためのメモを書いてから休憩する

5 分以上の待ち時間なら、無理に画面で待機しなくても良いかなと思います。  
ただし、離れる前に「戻ったら何を見るか」を 軽くメモすればいいと思います。

これを書かずに席を立つと、私は、戻ってきたときにわからなくなっていることが多いのでメモを書くようにしています。。  
メモを書いておけば、コーヒーを淹れている間に頭が休まっても、戻ってすぐタスクに頭を切り替えることが可能です。

水を飲む、目を休める、立って歩く程度なら良いと思いますし、実際に私もやっています。  
ただ、Slack や X を開くと別の文脈で上書きされるので、長い待ち時間こそ避けるようにしています。

## 実際に書いているメモ

待ち時間中に書いているメモはこのくらいの粒度で書いています。  
エディタの隅にメモを開いて、3 分くらいで書き殴っています。

```
## 今回の目的
入力エラー時に、ユーザーがどの項目を直せばよいか分かるようにする。

## 責務の仮説
- UI 表示: component
- 状態管理: custom hook
- バリデーション: pure function

## AI がやりそうな失敗
- hook に寄せすぎる
- エラーケースを正常系だけで済ませる
- 既存の命名とズレる

## 出力後に見る順番
1. 変更ファイル
2. 責務の置き場所
3. 型
4. 異常系
5. テスト
```

このメモはドキュメントとして残すものでもありません。  
書いてみると分かりますが、責務の仮説や見る順番は、AI に投げる前にもうぼんやり頭にあります。  
それを言葉にして残すかどうかで、出力後の自分の気持ちが変わってきます。

## 逆に、待ち時間にやらないようにしていること

待ち時間に避けるようにしているのは、こういうことです。

* Slack を見る
* X を見る
* YouTube を見る
* 別 Issue に飛ぶ
* 生成中のコードをぼんやり眺める
* 出力されたらすぐローカルで動かす
* 動いたらすぐ採用する
* AI に追加修正を投げ続けて、自分で読まない

もちろん、毎回完璧にできているわけではありません。 急いでいると Slack を見てしまうこともあります。  
ただ、AI の待ち時間ごとに別の文脈へ移動すると、戻ってきたときに自分の判断基準がブレブレになります。  
少なくともレビューが必要な変更では、待ち時間に別のデジタル情報を入れすぎないようにしています。

「Slack を見るな」と言いたいわけではなく AI の出力に対して自分の判断を持ちたいなら、待ち時間に文脈を手放さない方が割に合う、というだけの話です。

## やってみて変わったこと

待ち時間の使い方を変えてから、いくつか変わったことがあります。

* AI の出力に流されにくくなった
* 「これは違う」と言いやすくなった
* 出力後の確認が速くなった
* PR 説明文を書きやすくなった
* レビューで聞かれそうな点を先に潰せるようになった
* 待ち時間がただの休憩時間ではなくなった
* AI を使っているのに、自分も考えている感覚が戻った

劇的に生産性が上がった感覚はありませんが、出力後に迷う時間や、レビュー指摘を受けてから手戻りする時間は明らかに減りました。  
合計時間で言うとおそらくトントンくらいですが、AI疲れが大きく改善したように感じます。

## 想定される疑問

ここまで書いていて、自分でもツッコミを入れたくなったポイントが 3 つあります。

### Q. Claude Code のメモリや `CLAUDE.md` に書いておけば、毎回失敗予想を書かなくてもいいのでは？

メモリと待ち時間メモは、役割が違うと感じています。

* メモリ／`CLAUDE.md` は、出力**前**に AI に守らせるためのルール。プロジェクト全体に再利用可能なものを置く場所
* 待ち時間メモは、出力**後**に自分が判断するための観点。このタスク固有の目的・責務・観点を書く場所

たとえば「既存の責務分割を尊重する」は `CLAUDE.md` に書けます。  
ただ「**今回**の入力エラー UI の責務を component / hook / pure function のどこに置くべきか」は、タスクごとに変わるのでメモリには書ききれません。

それに、`CLAUDE.md` に書いてあっても AI は普通にやらかすことがあります。  
ガードを増やしても、最終的にそのコードが今回の要件と合っているかを判断するのは人間なので、待ち時間メモはそのための準備として書いている、というのが自分の整理です。

### Q. レビューも AI にやらせて、人間は休めばいいのでは？

AI にレビューさせること自体は、自分もよくやっています。  
PR を投げる前に Claude や CodeRabbit にざっと見てもらうのは便利です。

ただ、「AI が書いて、AI がレビューして、人間は Approve ボタンを押すだけ」になると、ちょっと危ないと感じています。  
PR を出した瞬間、その実装は自分の判断として記録されます。レビュアーや障害対応で「なぜこう実装したのか」を聞かれたとき、「AI レビューが OK を出していたので」では通らないからです。

これは前回の記事で書いた「レビューで説明できなかった」問題と、本質的には同じです。  
AI レビューは補助線としては使えますが、判断の最終責任まで AI に渡してしまうと、結局「説明できないコード」を量産することになります。  
だから、AI レビューを併用する場合でも、自分の中に判断基準を持っておく価値はあると考えています。

### Q. そもそもタスクを小さく分けて、待ち時間自体を発生させない方がいいのでは？

完全にその通りだと思います。  
タスクを小さく区切れば、AI に頼む単位も小さくなり、1 回あたりの待ち時間も短くなります。  
レビューもしやすくなりますし、AI の出力に置いていかれる感覚も減ります。  
むしろ、タスクを小さく切れる場面では、そちらを優先すべきだと考えています。

ただ、現実には小さく切れない場面もあります。

* 既存コードの調査込みのタスク
* リファクタや責務再配置のように、ある程度まとまった単位で考えないと判断できないもの
* 仕様の探索が必要で、そもそも切る粒度が決まっていないもの

こうした場面では、どうしても 1 回の依頼が長めになります。  
今回書いた待ち時間運用は、そういう「小さく切れない、もしくは切るのが難しかった」場面での保険として使っているイメージです。  
まずタスクを小さく切ることを試みて、それでも待ち時間が出るときに、今回の内容を検討するという関係になっていると思います。

## まとめ

うまくいっている話ばかり書きましたが、やはり疲れているとどうしても疎かになってしまいます。  
夜中に作業をしていると無意識にPRを作ってしまうこともあり、翌日にチェックから始まるので追いつくのに時間がかかったりしています。  
やはり疲れている時は大人しく寝ることに集中したほうがいいと思いました。

今後はAIの待ち時間中にタスクから離れすぎずに作業に集中していきたいと思います。
