海外の開発者 X で、Codex と Claude Code、そしてサブエージェントの使い方をめぐる実験投稿が伸びています。

話題になっていたのは Theo 氏の投稿です。表示回数は確認時点で約32.1万。内容は、GPT-5.6 Sol を使ったコーディング環境の実験で、メインのモデル設定を高い推論レベルにしたとき、そこから呼び出されるサブエージェントまで同じ高コスト設定で動いてしまう、という問題提起でした。

ここで大事なのは「どのツールが勝ったか」という単純な比較ではありません。重要なのは、AI エージェントが複数体で動く時代には、どの作業にどれだけ深く考えさせるかを設計する必要が出てきた、という点です。

何が起きたのか

OpenAI も Anthropic も、エージェントが別のエージェントを呼び出して並列に作業する流れを強めています。OpenAI の Multi-agent ドキュメントでは、モデルが複数のサブエージェントを並列に動かし、その結果を統合できると説明されています。Codex 側にも、調査・実装・レビューのような並列化しやすい作業でサブエージェントを使う考え方が整理されています。

一方で、Claude Code 側のドキュメントには、モデルや effort を設定ファイル、スキル、サブエージェント単位で調整する発想があります。つまり、これからのエージェント運用では「メインモデルを強くする」だけでは足りず、役割に応じて推論の深さを変えることが重要になります。

Theo 氏の投稿が刺さった理由はここです。高い推論レベルは強力ですが、サブエージェントが複数走ると、同じ設定が何倍にも増幅されます。1人の優秀なエンジニアに深く考えさせるのと、10人の調査担当すべてに最高レベルで長考させるのでは、コスト構造がまったく違います。

日本の読者が見るべきポイント

日本で Codex や Claude Code を使っている人は、まず「成功例の真似」から入りがちです。強いモデルを選ぶ。高い effort にする。長い指示を書く。これ自体は悪くありません。

ただ、サブエージェントが入ると話が変わります。

たとえば、1つの機能追加を AI に任せるとします。メインエージェントが設計を考え、別のエージェントが既存コードを調べ、別のエージェントがテスト方針や UI 影響を確認する。こういう構成はとても便利です。

しかし、すべてのサブエージェントに最高レベルの推論をさせる必要はありません。

既存ファイルの場所を探すだけなら低めでよい。テストログを要約するだけなら中程度でよい。仕様の矛盾を検出する、セキュリティ上の判断をする、設計の分岐を比較する、といった場面だけ高い推論を使う。こうした配線ができるかどうかで、同じ AI ツールでも費用対効果が大きく変わります。

実務での分解

サブエージェントを使うときは、作業を次の3種類に分けると扱いやすくなります。

1つ目は「探索」です。ファイルを探す、関連箇所を読む、既存の命名規則を集める、ドキュメントの該当部分を拾う。ここは広く浅くで十分なことが多く、低から中程度の effort が向いています。

2つ目は「判断」です。どの設計を採用するか、既存仕様と矛盾しないか、変更範囲をどこまでに抑えるか。ここはメインエージェントか、信頼できるレビュー用サブエージェントに高めの effort を割り当てる価値があります。

3つ目は「実行」です。コードを書く、テストを追加する、文章を整える、チェックリストを埋める。ここは作業内容によって変わります。単純な機械作業なら低めでよく、複数ファイルにまたがる実装なら中から高が必要です。

この分解をしておくと、「とりあえず全部最高設定」という使い方から抜けられます。
