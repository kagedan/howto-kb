## はじめに ― 今日が引っ越しの日

こんにちは、素人ITドクターの井手です。第4回です。

第3回で、引っ越しの**理屈**はすべて整理しました。OneDriveの罠、自宅サーバーへの集約、3層バックアップ戦略 ―― 頭では分かった、けど、実際に手を動かすのはやっぱり怖い。そして、今日いざ実演しようと思ったら ―― **新たな不安**が湧いてきました。

## 😨 引っ越し中に、本番のサービスが止まったらどうしよう?

clinic-match は、**実際に使っているサービスとしましょう**。  
もし引っ越し作業中にバグが入って、サービスが止まったら…  
**院長として、それだけは絶対に避けたい**。

これは医師読者の先生方も、もし自分のクリニックでITサービスを動かしていたら、**絶対に同じ不安**を持つはずです。

結論から言うと、これも AI に聞けば**明確に解決**します。第4回の最初のプロンプトは、これを確認することから始めます。

## プロンプト⓪:本番に影響しないか、AIに確認する

引っ越し作業を始める前に、まず**不安を消す**ことから。Antigravity の Claude Code に、こう聞きました ↓

### 📋 PROMPT 00 ― 本番に影響しないことの確認

```
これから プロジェクトを、
他のPCの別のフォルダ(C:\dev\)に git clone で引っ越します。

医師として一番不安なのは
「本番で動いているサービスが止まること」です。

以下を明確に答えてください:

1. git clone する作業は、本番(Vercel・Firebase)に影響を与えますか?
2. npm install や npm run dev は、本番に影響しますか?
3. 引っ越し作業中、本番で動いているアプリは止まりませんか?
4. もし新環境で操作を間違えても、本番が壊れない仕組みになっていますか?
5. 本番に影響を与える唯一の操作は何ですか?

医師にも分かるように、明確に教えてください。
```

> **🩺 AI の回答(医師にも分かる「本番影響度」明確回答)**  
>   
> **Q1. git clone は本番に影響を与える?**  
>   
> **✅ 絶対に影響しません**  
>   
> AIの比喩:「**医学雑誌の最新号を1冊買って自宅に持ち帰る**ようなもの。雑誌の出版社(GitHub)に『1冊ください』とお願いするだけ。雑誌の内容(本番アプリ)は一切変化しない」  
>   
> **Q2. npm install や npm run dev は本番に影響する?**  
>   
> **⚠️ npm install は安全。npm run dev だけ要注意**  
>   
> **npm install**:✅ 完全安全(自宅キッチンに調理器具を揃えるだけ)  
> **npm run dev**:⚠️ **1つだけ盲点あり** ←【超重要、後述】  
>   
> **Q3. 引っ越し中、本番アプリは止まらない?**  
>   
> **✅ 絶対に止まりません**  
>   
> AIの比喩:「**あなたが自宅で電子カルテの設定をしている間も、クリニックの受付・診察室は通常通り稼働しているのと同じ**」  
> Vercel・Firebase・Firestore は**クラウド上で24時間稼働継続**。  
>   
> **Q4. 操作を間違えても本番が壊れない仕組みは?**  
>   
> **✅ 5つの安全装置(フェイルセーフ)あり**  
>   
> ① GitHubは読み取り専用 / ② npm install は自分のPCのみ / ③ npm run dev は localhost のみ / ④ 本番デプロイは明示コマンド必要 / ⑤ Firestoreセキュリティルールで防御  
> → 医療の**「ダブルチェック・トリプルチェック」と同じ思想**  
>   
> **Q5. 本番に影響を与える「唯一の操作」は?**  
>   
> **⚠️ 明示的な4種類のコマンドだけ**  
>   
> 🔴 git push origin main(Vercelが本番を再デプロイ)  
> 🔴 firebase deploy(Cloud Functionsを上書き)  
> 🟠 firebase deploy --only firestore:rules  
> 🟡 firebase deploy --only firestore:indexes  
>   
> → 引っ越し作業中、これらは**一切打ちません**。

## 🚦 AI が示してくれた「3ゾーン分類」

医師の**トリアージ**と同じ感覚で、操作を3段階に分類してくれました

> **🟢 フリーゾーン ―― いくらやっても安全**  
>   
> ✅ git clone https://github.com/<your-account>/clinic-match-app.git  
> ✅ cd clinic-match-app  
> ✅ npm install (frontend / functions の両方)  
> ✅ コードを開いて読む・編集する  
> ✅ git status / git log でリポジトリ状態を確認  
>   
> → 自宅でカルテを「閲覧」しているのと同じレベルの安全性

> **🟡 イエローゾーン ―― 注意して行う**  
>   
> ⚠️ npm run dev で開発サーバー起動 → **画面を眺めるだけならOK**  
> ⚠️ ただし「**オファー送信**」「**アカウント作成**」等のボタンを押すと**本番DBに本物のデータが入る**  
>   
> **→ → ボタンを押すこと自体はOK。ただしテスト用と分かる名前で入力 → 後で削除のルール運用が必要。詳しくは次のセクションで。**

> **🔴 レッドゾーン ―― 引っ越し作業では絶対しない**  
>   
> 🔴 git push origin main  
> 🔴 firebase deploy  
> 🔴 Firebase コンソールでの直接操作  
>   
> → これらが本番を変える操作。**引っ越し作業では一切使いません**。

## ⚠️ 知っておくべき盲点:npm run dev のデータは本番DBに入る

AI の答えで、僕も**初めて気付いた重大な事実**がありました。

### 🩺 自宅で電子カルテのテストをしているつもりが、本物の患者カルテに書き込んでしまう状態

これが npm run dev の**唯一の盲点**です。  
画面は新PCの中(localhost:3000)で動いていますが、**データの読み書き先は本番のFirebase**。  
つまり、開発画面でボタンを押すと、**本番DBに本物のデータが入る**ということ。

![](https://assets.st-note.com/img/1778074181-X7HUgKJs9SNFdVhpv6eyP23k.png?width=1200)

**🩺 素人ITドクターの判断:むしろテストデータを入れて確認したい**

AIの警告を読んで、最初は「ヤバい、ボタン押せない」と思いました。  
でも冷静に考えると ―― **動作確認のためにはボタンを押したい**。

そもそも引っ越し後に「本当に動くか?」を確かめないと、引っ越しの意味がない。眺めるだけでは、ログイン処理・データ保存・メール送信が壊れてないか分からない。

そしてこれは、AI が「**絶対NG**」と言ってるわけではありません。AI が言ってるのは「**知らずに押すと本番DBに入りますよ**」という**事実の警告**。**知った上で意図的に入れる**のは、判断の問題です。
