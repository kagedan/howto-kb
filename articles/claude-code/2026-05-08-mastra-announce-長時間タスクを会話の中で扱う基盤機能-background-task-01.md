## Mastra が「Background Tasks」を発表

2026年5月7日、Mastra は **Background Tasks for Mastra Agents** を発表しました。

アナウンスの要点を一文で言うと、**時間のかかるツール実行をエージェントの応答フローから切り離し、進捗を返しながら裏で走らせられるようになった**、ということです。

公式の表現を借りると、エージェントは長時間実行されるツール呼び出しに対して、処理の前・途中・後でユーザーへフィードバックを返せるようになります。しかもそのあいだ、エージェントのループ全体が固まるわけではありません。さらに、背景実行が有効化されたツールを複数呼び出した場合は、**並列実行**もできます。

これは地味な改善に見えて、実際にはかなり大きいです。AI エージェントをアプリに組み込むと、時間のかかる処理はすぐに出てきます。深い調査、重いワークフロー、外部 API 待ち、人間の承認待ち、長いコーディングタスク。これまでは「応答を返すまで待たせる」か、「自前でジョブキューや SSE を組む」かのどちらかになりがちでした。Background Tasks はそのギャップを、Mastra のエージェント実行モデルの中で埋めようとする機能です。

---

## 背景：なぜ「バックグラウンド実行」が必要になるのか

LLM ベースのエージェントは、単純なチャットボットより長い処理を抱えがちです。とくに Mastra のように tools、subagents、workflows を組み合わせるフレームワークでは、1 回のユーザー入力の裏で次のようなことが起こります。

* 検索ツールで複数の情報源をたどる
* サブエージェントにリサーチや執筆を委譲する
* ワークフローを起動して数分かかる処理を待つ
* 人間の承認が返ってくるまで止まる
* 複数の外部 API をたたいて進捗を集める

ここで厄介なのは、**ユーザー体験と実行モデルの都合がぶつかる**ことです。

普通の同期処理なら、エージェントはツールの完了を待ってから次に進みます。これは実装としては分かりやすいのですが、ツールが 30 秒、1 分、あるいは数分かかると UI では「何も起きていない」ように見えます。ユーザーは固まったと思うし、サーバー再起動や接続切断にも弱くなります。

一方で、バックグラウンドジョブの仕組みを自前で作ると、今度はかなり面倒です。

* タスク ID の払い出し
* 実行状態の永続化
* 再起動後の復元
* SSE や WebSocket での進捗配信
* 完了後にエージェントへ結果を戻す制御

Background Tasks は、この「長時間タスクをまともに扱うための足回り」を Mastra が標準で持つ、という意味があります。

## 何ができるのか

Background Tasks のコアはシンプルです。**ツール呼び出しをその場で完了させず、別管理のタスクとして実行し、その状態変化をストリームへ流せる**ようにします。

公式ドキュメントを読むと、ポイントは次の 5 つです。

### 1. 長時間ツールを非同期で実行できる

ツールに `backgroundTasks: { enabled: true }` を付けるか、エージェント側で特定ツールを background 対象にすると、そのツールはバックグラウンド実行されます。

```
import { createTool } from "@mastra/core/tools";
import { z } from "zod";

export const researchTool = createTool({
  id: "research",
  description: "Run a deep research job on a topic",
  inputSchema: z.object({ topic: z.string() }),
  backgroundTasks: {
    enabled: true,
    timeoutMs: 600_000,
  },
  execute: async ({ context }) => {
    // 長い処理
  },
});
```

ここで重要なのは、「すべての tool call が勝手に非同期になる」わけではない点です。Mastra はデフォルトでは従来どおり前景実行で、**明示的に opt-in したものだけ**を背景実行します。短時間で終わるツールは今までどおり同期のほうが単純です。

### 2. ユーザーへ進捗を返せる

単に裏で走るだけではなく、ストリーミングイベントとして次のような状態を拾えます。

* タスクが開始された
* 実行中になった
* 出力の一部が届いた
* 完了した
* 失敗した
* キャンセルされた
* 一時停止された
* 再開された

つまり、UI 側では「調査を始めました」「まだ実行中です」「完了しました」という状態を、エージェントの対話の流れの中で自然に見せられます。ここが単なるジョブキューとの差です。

### 3. 完了後にエージェントが続きの思考を再開できる

Mastra は `agent.streamUntilIdle()` を用意しており、バックグラウンドタスクが完了した結果をメモリへ書き戻したうえで、**エージェントループを再入場させる**ことができます。

```
const stream = await agent.streamUntilIdle("Research SpaceX for me", {
  memory: { thread: "t1", resource: "u1" },
});

for await (const chunk of stream.fullStream) {
  // 初回応答、進捗、完了後の継続ターンがここに流れる
}
```

この仕組みのおかげで、エージェントは最初に「調査を始めます」と返し、その後バックグラウンドで実処理を走らせ、終わったら「結果を踏まえると結論はこうです」と続けられます。ユーザーから見ると 1 回の会話の延長に見えますが、内部的には複数ターンに分かれて処理されています。

### 4. 複数の背景タスクを並列で流せる

公式アナウンスでも明示されていますが、background 対象ツールを複数呼ぶと、それぞれが並列で走ります。

これは、たとえば次のようなケースで効きます。

* 3 つの検索ソースを同時に調べる
* 複数のサブエージェントへ並行して仕事を振る
* 別々の API からレポートを収集する

エージェントは本質的に「考える処理」と「待つ処理」が混ざるので、待ち時間を並列化できるだけで体感速度がかなり変わります。

### 5. 一時停止と再開ができる

これもかなり重要です。Background Task は、長い処理を走らせるだけでなく、**途中で suspend して外部シグナルを待つ**ことができます。ドキュメントでは、人間の承認待ちや webhook 待ちの例が示されています。

これは要するに、「AI エージェントの処理を、単なる非同期関数ではなく、状態を持つ長寿命タスクとして扱う」方向に Mastra が舵を切っていることを意味します。

## 使い始めるには何が必要か

最小構成で必要なのは 3 つです。

### 1. Mastra インスタンス側で有効化する

```
import { Mastra } from "@mastra/core";

export const mastra = new Mastra({
  storage: {
    // ...
  },
  backgroundTasks: { enabled: true },
});
```

公式ドキュメントでは、ここに `globalConcurrency`、`perAgentConcurrency`、`backpressure`、`defaultTimeoutMs` なども設定できます。つまり単なる ON/OFF ではなく、**同時実行数や詰まり方まで管理できるタスクマネージャ**として設計されています。

### 2. Storage を設定する

これは初心者が見落としやすいのですが、**storage は必須**です。理由は単純で、バックグラウンドタスクは状態を永続化しないと成立しないからです。

* タスクが何を実行中か
* 今どの状態にいるか
* 結果やエラーは何か
* 再起動後に再開できるか

これをメモリ上だけで持つと、プロセス再起動で全部消えます。なので Mastra は storage バックエンドを必須にしています。これは「ちょっと便利な async オプション」ではなく、**永続化された実行モデル**だと理解すると分かりやすいです。

### 3. 対象ツールやエージェントを opt-in する

ツール単位で `backgroundTasks.enabled` を付ける方法のほか、エージェント側でどのツールを background 対象にするかをまとめて制御できます。

```
import { Agent } from "@mastra/core/agent";

export const researcher = new Agent({
  id: "researcher",
  instructions: "You research topics and answer questions.",
  model: "openai/gpt-5.4",
  tools: { researchTool, summarizeTool },
  backgroundTasks: {
    tools: {
      researchTool: {
        enabled: true,
        timeoutMs: 600_000,
      },
      summarizeTool: false,
    },
  },
});
```

この設計は実務的です。ツール定義そのものは再利用しつつ、どのエージェントで background 実行を許すかを分けられるからです。

## どう理解するとよいか

この機能は「Mastra にジョブキューが入った」と理解しても完全には間違いではありません。ただし、もっと正確に言うなら、**エージェントループと統合されたジョブ実行基盤**です。

普通のジョブキューとの違いは、完了した結果が単に DB に保存されるだけでなく、**エージェントの記憶とストリームに戻ってくる**ことです。これによって、ユーザーの会話体験が次のように変わります。

1. ユーザーが長い依頼をする
2. エージェントがすぐに着手メッセージを返す
3. 裏でツールやサブエージェントが走る
4. 進捗が UI に流れる
5. 完了すると、その結果を踏まえてエージェントが続きの返答を行う

この流れは、ディープリサーチやコーディングエージェントの UX と非常に相性がいいです。最近の AI プロダクトでは「すぐ最終回答を返す」のではなく、「いま何をしているかを見せながら進める」方向へ寄っています。Background Tasks は、その UX を Mastra で実装しやすくするための基盤機能だと言えます。

## ロードマップとのつながり

今回の発表は、[2026年5月発表のロードマップ](https://zenn.dev/shiromizuj/articles/f8ed4ed72c414b) で触れられていた「長時間稼働するエージェント」の流れときれいにつながっています。

ロードマップウェビナーでは、5 月の注力テーマとして次が挙がっていました。

* Background Agents
* Durable Agents
* Dynamic Sub-Agents

このうち今回の Background Tasks は、少なくとも **Background Agents 的な体験を支える低レベルの実行基盤**と見ると理解しやすいです。長時間ツールを裏で走らせ、進捗を返し、終わったら会話へ戻す。これはまさに「長く動くエージェント」の最初の足場です。

また、ロードマップでは「ウィンドウを閉じても続きが見られる durable な体験」や、「スケジュールと文脈を持って自律的に動く Heartbeats」の話も出ていました。今回の発表だけでそれらすべてが完成したわけではありませんが、**タスク状態を永続化し、途中経過や完了イベントを扱う仕組み**は、そうした将来機能とかなり相性が良いはずです。

言い換えると、Mastra はこの数か月、Observational Memory、Workspaces、Remote Sandboxes、Channels、ClickHouse などを順に出してきましたが、ここへきて「長時間・並列・継続実行」という軸を明確に強め始めた、という見方ができます。

## 実務でどこに効くか

個人的には、Background Tasks は次のようなユースケースで特に効くと思います。

### Deep Research 系

検索、抽出、要約、比較、執筆までを 1 回でやるリサーチエージェントは、どうしても数十秒から数分かかります。こうした処理を前景同期で待たせるのは UX が悪いので、背景実行との相性が非常にいいです。

### コーディングエージェント系

リポジトリ探索、LSP 検索、テスト実行、複数ファイル編集、結果の検証。こうした流れは、まさに「進捗を返しながら長く動く」対象です。Mastra が最近強化している Workspace や Browser、LSP との相性も良いでしょう。

### Human-in-the-loop 系

承認待ち、レビュー待ち、追加入力待ち。Background Tasks の suspend / resume はこの種の処理に直結します。チャットで依頼し、必要なところだけ人間が介入し、残りはエージェントが進める設計が取りやすくなります。

### UI 付き業務アプリ系

SSE で進捗を見せる管理画面、タスク一覧、失敗時の再試行、状態監視。Mastra の `backgroundTaskManager.stream()`、`getTask()`、`listTasks()` は、この手の UI を組み立てる前提機能としてかなり扱いやすそうです。

## 注意点もある

便利な機能ですが、何でも background にすればよいわけではありません。

### すぐ終わるツールには不要

数百ミリ秒から数秒で終わる処理なら、通常の `stream()` や `generate()` のほうが単純です。Background Tasks はタスク管理と永続化を伴うぶん、構成も理解コストも少し上がります。

### Storage 設計が重要になる

この機能をちゃんと使うなら、アプリ全体の storage 戦略を考える必要があります。ローカル開発なら手軽でも、本番では durability と可観測性を含めた設計が必要です。ちょうど数日前に発表された ClickHouse 対応とも、こういう長時間運用の流れでつながっています。

### ストリーミング UI の設計も問われる

Background Tasks が入ると、UI は「最終テキスト 1 発」では足りません。開始、進捗、完了、失敗、再開などのイベントをどう見せるかを考える必要があります。フレームワークの機能追加は、同時にプロダクト設計の自由度と責任も増やします。

## まとめ

Background Tasks は、Mastra における長時間エージェント処理の基盤整備です。単なる async 実行ではなく、**永続化されたタスク状態、ストリーミングでの進捗通知、完了後のエージェント再開**までを含めて一つの実行モデルとして提供している点が肝です。

ロードマップ動画で見えていた「長時間稼働するエージェント」への流れが、今回かなり具体的な API として姿を見せたと言ってよいでしょう。Background Agents、Durable Agents、Heartbeats のような話題がこれからどう積み上がるかを見るうえでも、かなり重要な第一歩です。

これからもMastraの機能追加から目が離せません。
