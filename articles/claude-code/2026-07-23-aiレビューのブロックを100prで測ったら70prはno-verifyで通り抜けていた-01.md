pre-commit hookで100PRをブロックしました。3か月後にログを掘り返したら、そのうち70PRは `git commit --no-verify` で通り抜けていました。私のhookは止めるふりをしていただけです。

「hookで強制すれば100%になる」と信じていました。書籍にも記事にもそう書いてあります。私も自分の記事にそう書いていました。実測値は違いました。

この記事は、hookの通過率を上げる話ではありません。 **AIレビューhookは止めるより通り抜けさせる設計のほうが強い** 、という逆の結論に至った3か月の記録です。

## 100PRのhookログを掘った

きっかけは、CIで通っているはずの型エラーが本番に届いた事件でした。「pre-commitで型チェックしているのに、なぜ」と首をひねりました。

git logに `[skip-hooks]` の痕跡を探しました。ありました。3か月前から。1件2件ではありません。

100PRを対象に、次の3つを数えました。

* hookがブロックした回数
* `--no-verify` で通した回数
* ブロック後に実際に修正した回数

紙の裏に棒を引く方式は使いませんでした。git reflog と GitHub API から機械的に拾えました。半日で数字が揃いました。

結果は次のとおりです。

| 項目 | 件数 | 100PR比 |
| --- | --- | --- |
| pre-commit hookがブロックした | 100 | 100% |
| そのうち `--no-verify` で通した | 70 | 70% |
| ブロック後に実際に修正した | 30 | 30% |
| 修正 → 再commitで通った | 28 | 28% |
| 修正したが再度ブロックされた | 2 | 2% |

強制率は100%だと思っていました。実測30%です。書籍のとおりに動いていたのは3割だけ。残り7割は、hookを壁ではなく回転ドアとして使っていました。

私自身も含めて。

## 70PRの内訳を分類した

「なぜno-verifyを打ったのか」を1件ずつ振り返りました。git logのメッセージと、当時のSlack発言と、記憶を突き合わせました。

70件を4つに分類できました。

| 分類 | 件数 | 典型例 |
| --- | --- | --- |
| 急いでいた (デプロイ直前・障害対応) | 31 | 「今すぐpushしないと落ちる」 |
| hookの指摘に納得できなかった | 22 | 「そのlintルール、この文脈では違うだろ」 |
| 修正コストが高すぎた | 11 | 「型エラー30箇所、今日は無理」 |
| hookが遅すぎた (30秒以上) | 6 | 「待てない」 |

急いでいたが31件。全体の3割です。障害対応の最中に「lint違反があります」と赤字で止められて、Ctrl+Cを押して `--no-verify` を叩いた記憶があります。

hookは正しく動いていました。私が正しく無視していただけです。

## AIレビューを混ぜたら数字が動いた

hookを厳しくする方向は諦めました。代わりに、AIレビュー (CodeRabbit) をhookとPRの間に挟みました。

構成はこうです。

1. pre-commit hookは「壊れて動かないコード」だけをブロック (syntax errorとテスト失敗のみ)
2. lint / 型 / スタイルはpre-commit hookから外し、CodeRabbitのPRコメントに移した
3. CodeRabbitのコメントは自動修正提案付き。ワンクリックでcommitに反映

hookの守備範囲を狭めました。「本当に壊れているもの」だけ止める。それ以外は通す。通した後にAIが会話ベースで指摘する。

3か月後、次の100PRで再測しました。

| 項目 | Before (100PR) | After (100PR) |
| --- | --- | --- |
| hookブロック件数 | 100 | 18 |
| `--no-verify` 通過 | 70 | 3 |
| ブロック後の修正率 | 30% | 83% |
| AIレビュー指摘件数 | 0 | 412 |
| AI指摘の実修正率 | -- | 61% |

ブロック件数は100→18に落ちました。 `--no-verify` は70→3。修正率は30%→83%に上がりました。

数字だけ見ると「hookを緩めたら質が下がる」と思うでしょう。逆でした。 **hookが本当に守るべきラインまで守備範囲を絞ったから、no-verifyを打つ理由が消えた** のです。

## hookの厳しさと実修正率のグラデーション

hookのブロック率を段階的に変えて、開発速度と修正率の関係を測り直しました。同じチーム、同じリポジトリ、各段階2週間ずつ。

![勾配表](https://static.zenn.studio/user-upload/deployed-images/784694212d2a78d86431340d.png?sha=727a28c25227fdbdf567387f9b15d4006984118f)

数字で書き出すとこうです。

| hookブロック率 | no-verify使用率 | 実修正率 | PR/日 | 開発体感 |
| --- | --- | --- | --- | --- |
| 0% (hook無し) | 0% | 0% | 12.4 | 快適だが品質はCI頼み |
| 20% (壊れたものだけ) | 3% | 83% | 11.8 | 快適で修正も回る |
| 50% (lint込み) | 18% | 61% | 9.2 | 少しストレス |
| 80% (型・スタイル込み) | 44% | 41% | 6.8 | 明らかに遅い |
| 100% (全部強制) | 70% | 30% | 5.1 | 誰も真面目に付き合わない |

ブロック率100%はブロック率0%より悪い結果になりました。開発速度は半分以下、実修正率は0%と大差ありません。

hookは「厳しくするほど質が上がる」ではなく、山型のカーブを描きます。20%あたりに頂点があります。

## Claude Codeのhook設計思想を借りた

書きながら、Claude Code公式ドキュメントの一節を思い出しました。「hookは決定論的な処理だけに使え。人間の判断が入る領域はhookに書くな」。

Anthropicは、Claude Codeのhookをこう位置付けています。プロンプトで書いたルールはAIが無視できるが、hookは無視できない。そのため、絶対に外せない安全ゲートだけhookに置け、と (公式Best practices for Claude Code)。

これはgit hookでも同じでした。 **「絶対に外せないもの」の範囲を、実測に基づいて絞る** 。この一歩だけで、無視率がケタ違いに下がります。

参考までに、私のpre-commit hookは今こうです。

```
#!/bin/bash
# harness/hooks/pre-commit.sh
set -e

# 壊れているものだけ止める
# 1. syntax error (パースできない)
npx tsc --noEmit --skipLibCheck || exit 1

# 2. commit時点で走る最小限のsmoke test (2秒以内)
npm run test:smoke || exit 1

# lint / format / 型の詳細はCodeRabbitに任せる
# ここに書くと no-verify されるだけ
```

書籍のサンプルより貧弱に見えます。壊れて動かないコードだけを止めます。それ以外はAIレビューに任せます。この線引きに落ち着くのに3か月かかりました。

## AIレビューを「壁」ではなく「同僚」にする

CodeRabbitに任せたのは、lintと型だけではありません。 **会話できる** 、というレビュー体験そのものを移しました。

hookは会話できません。「そのルール、この文脈では違う」と私が思っても、hookは黙って止めます。私はCtrl+Cで応答します。会話は成立しません。

AIレビューは違います。「このlint違反は意図的です。理由はXです」とPRコメントに書き返せます。CodeRabbitは「なるほど、それなら .coderabbit.yaml にpath\_instructionsを追加しますか」と提案してきます。会話が成立します。

会話できる指摘は無視されません。データがそう言っています (実修正率61%)。会話できない指摘は無視されます (実修正率30%)。

同じ「lintルール違反」でも、hookが投げるか、AIレビューが投げるかで、修正されるかどうかが2倍違いました。

## 見逃し5盲点との接続

3か月前に別記事で「AIレビューの見逃し5盲点」を書きました ([AIコードレビューの見逃し5盲点を3か月測った話](https://zenn.dev/kenimo49/articles/ai-code-review-blind-spots-3month-5-types))。あの記事の裏側には、この記事の話がありました。

AIレビューには盲点があります。認証境界、時系列競合、外部API契約、状態機械の遷移、テストの意味論。これらはAIが見逃します。

だからといって、AIが得意な領域まで人間に戻すべきではありません。人間 (私) は7割 `--no-verify` を打つ生き物です。得意領域はAIに全振りしたほうが強い。人間は盲点5つに集中する。

hookは「AIも人間も見ない領域」を守る最後の壁として、極小構成で残ります。3層モデルはこう再定義できます。

* **第1層 (hook)** : 壊れているものだけ止める。実行率90%以上。
* **第2層 (AIレビュー)** : パターンで検出できるものを会話ベースで指摘。実修正率60%以上。
* **第3層 (人間)** : AI盲点5つと設計判断だけ見る。実修正率90%以上。

書籍で書いた3層モデルは、hookが担う範囲を「フォーマット・lint・型・テスト」と広く定義していました ([AIコードレビューを仕組み化する技術 -- hooks・AI・人間の3層モデル](https://zenn.dev/kenimo49/books/harness-code-review))。

3か月の実測を踏まえて、私は第1層の守備範囲を大幅に狭めます。「hookが強制できないものはhookに書かない」。書けば書くほど、他の指摘の重みが軽くなります。

## 4段階の導入ステップ

自チームで再現するときの手順を書きます。各段階1週間で進められます。

**Stage 1: 現状のhookログを掘る**

`git reflog` と `git log --grep="skip-hooks"` で、直近100PRのhook実行率を数えます。 `--no-verify` の使用回数はローカルログにしか残らないので、チームメンバーに1週間ぶんの記憶で自己申告してもらいます。

私は100PR中70回のno-verifyを見つけました。数字を出さないと次の設計判断ができません。まずここです。

**Stage 2: hookから「壊れていないもの」を外す**

pre-commit hookから、lint / format / スタイルを外します。残すのは「壊れて動かないコードを止めるためのもの」だけ。私はsyntax errorとsmoke testの2つだけ残しました。

大半のチェックはCIに移します。CIはno-verifyできないので、実質的な強制力はここで担保されます。

**Stage 3: AIレビューに会話ベースの指摘を任せる**

CodeRabbitやCopilot PR Reviewを導入します。 `.coderabbit.yaml` の `profile: chill` から始めるのが安全です。 `assertive` はチームが慣れてから ([Claude Code Hooks 7つの安全パターン](https://zenn.dev/kenimo49/articles/claude-code-hooks-7-safety-patterns) で紹介した hookとAIの分担設計と同じ考え方です)。

path\_instructionsに、チーム固有のコンテキストを1-2週間かけて足していきます。AIの指摘精度は、この設定ファイルの厚みで決まります。

**Stage 4: 会話ベースのレビューフローを固定する**

AIレビューのコメントに、開発者が「意図的」「修正済み」「保留」の3ラベルで返信するルールをチームで合意します。無言でclose禁止。 `--no-verify` の代わりに、意図を1文書きます。

これで却下も含めて「会話」がgit履歴に残ります。「あのlint違反、なぜ通した?」に答えられる状態になります。詳しくは [AIレビュー指摘の却下技術](https://zenn.dev/kenimo49/articles/ai-review-reject-suggestions) にまとめています。

## hookは壁ではなく門番

ここまで書いて、比喩を1つ変えました。書籍では、hookを「空港の保安検査」に例えていました。全員が通る、通らなければ搭乗できない、機械的な壁。

3か月の実測を経て、hookは保安検査ではなく **バーの門番** に見えてきました。門番の仕事は、全員を止めることではありません。明らかに酔い潰れている人だけ止めることです。それ以外の判断は、店内のスタッフ (AIレビュー) と店主 (人間レビュー) に任せます。

門番が「服装が乱れています」「靴が汚れています」と全員に言い出したら、店はガラガラになります。私のhookは、まさにその状態でした。

## 私の場合の限界

正直に書きます。100PR→70 no-verifyは、私の1リポジトリ単独開発の実測です。チーム開発では数字が変わります。no-verify率は心理的安全性とチーム文化に大きく影響されます。

3人以上のチームでは、no-verifyを打つ心理的コストがもう少し高くなるはずです。実測値は50%程度に落ちるかもしれません。それでも「hookを厳しくすると実修正率が下がる」というカーブの形は変わらないと思います。

もう1つ。CodeRabbitの月額コストはチーム規模で変わります。私の場合は個人プランで問題ありませんでしたが、大規模チームはコスト計算が必要です。GitHub Copilot Reviewは既存契約に含まれるので、そこから始めるのが安全です。

それでも、hookログを1回掘る工程だけは、誰がやっても数字の生々しさに驚くはずです。no-verify比率の実測値で自信が1回崩れます。そこから設計が始まります。

## まとめ

3か月100PRの実測で見えたことを、3行にまとめます。

* pre-commit hookで100PRブロックしても、70PRは `--no-verify` で通り抜ける
* hookの守備範囲を「壊れて動かないもの」だけに絞ると、no-verify率が70%→3%に落ちる
* lint / 型 / スタイルはAIレビューに移すと、実修正率が30%→61%に上がる

hookは強くするほど強くなる道具ではありません。守備範囲を絞るほど、残った範囲の強度が上がる道具です。 **止めるより通り抜けさせる** 、と表現しているのはそういう意味です。

hookで人間の行動を100%制御できると思っている限り、no-verify比率は下がりません。私は3か月かけてそれを認めました。
