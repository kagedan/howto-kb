「AIを使えば量産できる」と聞いて始めたのに、最初の1週間は1日2〜3本が精一杯でした。それが今は2日で20本を作れるようになったので、何が変わったかを正直に書きます。

## そもそも何を量産しているか

**コピペ de UI** というサイトをやってます。業務Webアプリでよく使うUIコンポーネントを、動くデモ・サンプルソース・AIプロンプトの3点セットで提供するやつです。

5月16日にリリースして、6月11日時点で事例77本・記事30本を公開しています。

最初から「量産できると思っていたか」というと、正直そうでした。AIに指示すれば大量に作れるんじゃないかと。実際は全然そうじゃなかった。

## 最初の2週間：手作業で詰まっていた

最初の2週間は毎日事例を作っていましたが、1日2〜3本が限界でした。

なぜかというと、あらゆる判断を毎回その場でやっていたからです。

* 「このコンポーネントの仕様、どうする？」
* 「デモのHTMLの構造、今回はこれでいいか？」
* 「サンプルソースをどこまで削るか？」

ClaudeCodeに作ってもらいつつも、方向性を決める部分は全部自分で考えていた。毎回ゼロから考えていた、というのが正確です。

そしてもうひとつ問題がありました。5月26日頃に気づいたんですが、デモとサンプルソースが乖離していたんですよ。

> デモはちゃんと動いているのに、サンプルソースを見ると別のコードが書いてある。

自分で作っておいて、自分で気づいてないやつです。恥ずかしい。

## check-sample.jsを入れた：5月26日

デモとソースの不一致が怖くなって、チェックスクリプトを作ることにしました。

「チェックツールをClaudeCodeに作らせる」という発想の転換です。自分でコードを書くのではなく、仕様を伝えてツール自体を作ってもらう。

> 「デモのHTMLとサンプルソースを自動で突き合わせて、乖離があったら警告を出すスクリプトが欲しい」

それだけ伝えたら `check-sample.js` ができあがりました。コードを自分では1行も書いていないんですが、品質チェックがちゃんとできるようになった。

この体験が後のワークフローの考え方に影響しています。「詰まってから道具を作る」という順序です。最初から完璧な仕組みを作ろうとするんじゃなくて、手作業で詰まったところを自動化していく。

## スラッシュコマンド化：5月31日

check-sample.jsを入れてから5日後、今度は作業プロセス全体をスラッシュコマンド化しました。

「事例を作るときの手順」をそのままコマンドのMDファイルに書いただけです。

```
/create-example
→ 仕様確認 → HTML作成 → デモ確認 → サンプルソース整理 → check-sample.js実行
```

これ以降、「何をどの順番でやるか」を毎回考えなくてよくなった。ClaudeCodeにコマンドを呼ぶだけで、手順通りに動く。

> 判断を減らすと、速度が上がる。

当たり前のことなんですが、これを実感したのはコマンド化してからです。

## 2日で20本：6月9〜10日

この状態で「簡単解説」という新シリーズを作り始めたら、2日間で約20本を一気に公開できました。

なにか特別なことをしたわけじゃないです。ルールMDがあって、コマンドがあって、チェックスクリプトがある。その状態で「ではシリーズを書こう」とやったら、2日間で20本になっていた。

初期の1日2〜3本と比べると、10倍以上の速度です。

> 仕組みを作るのに2週間かかって、仕組みで作るのが残りの期間。

「最初から速く作れると思っていた」という期待との乖離を最初に書きましたが、正直今は「仕組みを作る期間が2週間で済んだのは速かった」と思ってます。

## ツールは手作業で詰まってから作るほうがいい

振り返ると、仕組みを作る順序が大事だったと思います。

最初から完璧なワークフローを設計しようとしていたら、たぶん設計で詰まっていたはずです。手作業でやってみて「これが面倒くさい」「ここで詰まる」を体で覚えてから道具を作ったから、ちゃんと使える道具になった。

1日2〜3本の時期は無駄じゃなかった、と今は思ってます。ただ当時はそんなこと思ってなかったですが。
