## はじめに

[株式会社Sally](https://sally-inc.jp/) 所属エンジニアの @wellPicker です。  
弊社では、スマホやパソコンでマダミスを遊べるアプリである[ウズ](https://www.uzu-app.com/ja)や、マダミス情報・予約管理サイト[マダミス.jp](https://mdms.jp/)、マダミス開発ツール[ウズスタジオ](https://www.uzu-app.com/ja/studio)を開発しています。  
マダミスについては[こちら](https://www.uzu-app.com/ja/what-is-mdms)をご覧ください。

Claude Code の応答が「なんとなく初動が遅い」「無関係な文脈まで踏まえた脱線が増えた」と感じるようになり、`CLAUDE.md` と `MEMORY.md` の肥大化を疑って中身を確認したところ、想像を超えて肥大化していました。しかも一度整理を依頼して「1/3 に圧縮された」と報告を受けていたにも関わらず、です。

本記事では、この「一度圧縮したはずなのに再度肥大化してしまった」原因と、そこから得られた md ファイルの運用方針の結論について紹介します。

## きっかけ: CLAUDE.md と MEMORY.md の肥大化

Claude Code はセッションを起動するたびに、`~/.claude/CLAUDE.md` (グローバル指示) と、プロジェクト直下の `CLAUDE.md`、そして auto-memory の `MEMORY.md` を全文コンテキストに載せます。

特に `MEMORY.md` は Claude 自身が「これは覚えておくべきだ」と判断したメモを自動で追記していく仕組みなので、意識せず使っていると気付いたら数百行になっています。私の場合、mobile / backend / マダミス.jp と扱うレイヤーが多いこともあり、`MEMORY.md` は最初の見直し時点で 215 行まで達していました。

## 一度目の整理: 「1/3に圧縮しました」報告を鵜呑みにする

`MEMORY.md` が長くなってきたことに気付き、Claude 自身に整理を依頼しました。依頼内容はシンプルに「冗長な項目や、書かなくても自動で分かる項目を削除して 200 行以内に収めて」というものです。

依頼の結果、最終的に「215 行 → 71 行 (約 1/3) に圧縮した」という報告を受けて、その時はそれで満足していました。数字だけ見て「良い仕事をしてくれた」と思い、圧縮後の具体的な中身を目視で確認していませんでした。

## 二度目の気付き: 実は「圧縮」ではなく「見えない場所に移した」だけだった

それから 1 ヶ月半ほど経って、また応答が重く感じ始めたので、今度は自分の目で `MEMORY.md` を VS Code で開いてみました。開いてみて、そもそも Claude がどんな方針で圧縮したのかをこの時初めて知り、状態が想像以上に悪化していることが分かりました。

行数自体が再び 170 行まで増殖していることも困りましたが、問題はその中身でした。以下は実際のファイルの一部を抜粋したものです。

```
- [根治優先・暫定対応は指示がない限り禁止](feedback_root_cause_over_workaround.md) — symptom を塞ぐ workaround に逃げず、まず根本原因を解明して原因そのものを直す
- [置換前に挙動差を全列挙](feedback_enumerate_behavior_diffs_before_replacing.md) — 似たコンポーネントへの共通化は着手前に挙動差を表で提示しユーザー判断を取る
- [2回失敗したら設計から見直す](feedback_step_back_after_two_failed_attempts.md) — 同じファイル群への小手先修正で2回失敗したら、責務の置き場・依存方向から見直す
- [タイミングバグは診断モード commit で fact 取り](feedback_diagnostic_overlay_for_timing_bugs.md) — イベント駆動の修正で 2 回失敗したら、画面表示型の診断モード commit で実機観測してから本修正
- [スコープ逸脱で立ち止まる](feedback_stop_when_scope_creeps.md) — 依頼スコープ外のファイルに手を伸ばす前にユーザーに確認する
```

最初の修正依頼時に「行数を減らせ」と依頼した結果、Claude は各項目について `MEMORY.md` 内では見出しと参照ファイルへのリンクの 1 行だけを記し、参照ファイル内でそれぞれ数行から数十行の内容を書く、という形式を勝手に選んでいました。その結果、実際の内容は数百行分にも膨れ上がっていたようです。

つまり、一見 `MEMORY.md` の分量は大きくないのに、実際には大量の記述がコンテキストを圧迫して Claude のパフォーマンスを下げている事態が起きていたのに、開くまでそれに気付けなかった、ということです。

## 全削除

状況を確認して、個人用の `CLAUDE.md`, `MEMORY.md`, および `/memory` 配下の個別の md ファイル (143 ファイルもありました) を一度思い切って全て削除しました。結果として、かなり動作は安定するようになったと思います。

## 再発防止: 許可のない md 書き込みを禁止する

一度全削除しても、Claude Code の auto-memory は放っておけばまた勝手にメモを追記し始めます。そこで、グローバル `~/.claude/CLAUDE.md` を最小構成で書き直すついでに、以下のルールを追記しました。

```
## メモリ・ルール系ファイル

`CLAUDE.md` / `MEMORY.md` / `~/.claude/projects/*/memory/*.md` を
作成・編集・削除する前に、適用後の全文を提示してユーザーの明示承認を得る。
承認なしの書き込みは禁止。
```

これで Claude は memory 系のファイルに書き込む前に、必ず「これを書きます、いいですか?」と確認するようになりました。「勝手に増殖する」問題を根本から止める、シンプルで効きの良い一撃です。

## まとめ

Claude Code の「なんとなく遅い」「なんとなく脱線が多い」の一因は、`CLAUDE.md` と `MEMORY.md` の肥大化にあります。

私自身、Claude に整理を任せた結果として採用されていた「1 項目 1 行 + 参照リンク」形式を「1/3 に圧縮された」の報告だけで受け入れて、一度は解決したつもりになっていました。しかし実態は「本体は短くなったが、リンク先を含めた総量では何倍にも膨れ上がっている」状態で、しかもそれを本体を眺めているだけでは察知できない、という最悪の構造ができ上がっていました。整理を任せた後に、必ず結果を自分の目で確認しておくべきだったと思います。

加えて、`MEMORY.md` は Claude 自身が自動で追記していく仕組みなので、放っておくと肥大化は不可避です。書き込み時に必ず承認を挟ませるルールを 1 行入れておくだけで、再発は大きく抑えられます。

トークン数は応答速度にも API コストにも直接効くので、Claude Code を日常的に何セッションも走らせている方は、`CLAUDE.md` / `MEMORY.md` の中身と、参照 md まで含めた総量を今一度見直してみてはいかがでしょうか。
