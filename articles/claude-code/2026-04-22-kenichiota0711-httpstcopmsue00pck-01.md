プロジェクトごとに記憶は蓄積できた。ただ、もっと大きな「事業・組織・人」における意思決定軸があるはずだと思いました。

# 記憶とスキーマは、別の層にある

認知科学では、人間の認知を「事実の記憶」と「世界の構造」に分けて考える。後者を"スキーマ"と呼びます。**理解のための"枠組み"です。**

記憶が「何があったか（What happened）」だとすれば、スキーマは「世界はどう動くか（How things work）」。例えば「3/10に評価面談をした」は記憶。「評価面談とはこの構造で、この関係者がいて、この判断基準で動く」はスキーマです。

AIにはこのスキーマが存在しません。だから記憶を蓄積しても、文脈を読めない。事実を積み上げても、判断の軸が立ち上がらない。

# 認知科学のスキーマ理論

スキーマは発達心理学者ピアジェの理論。人間は世界を理解するための「型」を持っていて、新しい情報には既存の型を当てはめて（**同化**）処理する。型で説明できない情報には、型そのものを更新する（**調節**）。

人間は出来事をそのまま記憶しているのではなく、スキーマを通して再構成している。**同じ話を聞いても、人によって記憶が違うのはこのためです。**

神経科学には予測符号化理論があります（Friston, 2010）。脳は常に世界を予測するマシンとして働き、予測と現実のズレから学習する。知能とは事実の蓄積ではなく、世界を予測する枠組みそのもの。

スキーマは、知能の土台です。

# 記憶は時間軸で増え、スキーマは構造として深まる

この2つは"育ち方"が違う。記憶は、時間とともに線形に増えていく。1on1ログ・議事録・決定事項。日々蓄積されるフロー情報。

スキーマは、構造として深まっていく。頻繁には更新されない。しかし更新されるときは"世界の理解"そのものが変わる瞬間。

両者を同じ場所で管理しようとすると混乱が起きます。**AIが参照しても、出来事と原理の区別がつかない。よって、層を分けて設計する必要がありました。**

# 前回の設計と、足りなかった層

前回記事のディレクトリ構造は以下。

```markdown
~/projects/
├── management/
│   ├── context/     ← このプロジェクトの文脈（人事制度、事業、1on1の型）
│   └── members/
├── note-writing/
│   └── context/     ← このプロジェクトの文脈（文体、プロフィール）
└── ai-promotion/
    └── context/     ← このプロジェクトの文脈（AI戦略、ツール評価）
```

各プロジェクトに context/ があります。ただし、これはそのプロジェクトを動かすための文脈です。人事制度のコンテキスト、特定施策のコンテキスト、AI推進のコンテキスト。それぞれが閉じた領域の前提。

足りなかったのは、すべてのプロジェクトの上に存在する層。**事業そのもの、組織そのもの、デザイナーとしての自分そのもの。**どのプロジェクトに取り組むときも背景として存在している、判断の大前提です。

# schemas/ 世界そのものを置く場所

グローバル設定配下に schemas/ を追加。

```markdown
~/.claude/
├── CLAUDE.md
├── memory/              # 記憶の層（What happened）
│   ├── context-log.md
│   ├── frameworks.md
│   └── preferences.md
└── schemas/             # ← 新設：世界の層（How things work）
    ├── SCHEMAS.md       # インデックス
    ├── domain/          # 事業・業界
    │   ├── business-model.md
    │   └── industry.md
    ├── organization/    # 組織
    │   ├── company.md
    │   ├── values.md
    │   └── design-org.md
    └── stakeholders/    # 関係者
        └── map.md
```
