## 3行サマリ ― ここだけ持って帰ってもええで

* **動的ワークフローは「ハーネスをその場で即興生成する」機能**。今まで人間が事前整備しとった指示書・スキル・サブエージェント・Hooks を、タスク内容に応じて動的に組み立てる

## 動的ワークフローとは ― ハーネスを「その場で」作る仕組みや

### まずエージェントハーネスのおさらいやで

動的ワークフローを理解するには、まず **エージェントハーネス** っちゅう概念を押さえとく必要がある。

ハーネスというのは、Claude や ChatGPT みたいな **モデル単体の部分「以外」の周辺環境ぜんぶ**のことや。具体的には、

* 指示書（CLAUDE.md みたいなルールファイル）
* スキル
* サブエージェント定義
* Hooks

みたいな、モデルをうまく動かすための **ガードレールやフィードバックループの仕組み全般**を指しとる。

ほんで今までは、このハーネスを **タスクの内容に応じて人間が事前に定義しとく**必要があった。「この作業はこういうサブエージェント構成で」「レビューはこの観点で」って、ぜんぶ仕込みが要ったわけや。

### それを「即興」でやってくれるんが動的ワークフローや

動的ワークフローは、ユーザーがその場で入力したタスクの内容に応じて、このハーネスを **即興で生成**する。流れはこうや。

```
人間の指示
   │
   ▼
[タスク分解・計画] ── 入力されたタスクを分解して計画を立てる
   │
   ▼
[JavaScript定義] ── エージェントの生成・オーケストレーションをJSで決定論的に定義
   │
   ▼
[並列実行] ── 複数のサブエージェントが並列で起動
   │
   ▼
[相互検証・フィードバックループ] ── 出力結果を互いに検証し合う
   │
   ▼
高品質な成果物
```

特徴を3つにまとめると、

1. **ハーネスを動的に作る** ― 事前整備が不要になる
2. **生成役とは別のサブエージェントが出力を検証する** ― 自己採点やなくて相互検証
3. **長時間・複雑度の高いタスクを単一コンテキストでやらん** ― 品質低下を構造的に回避

っちゅうことや。

## なんでこんな機能が要るんや ― 単一エージェントの3つの課題

新機能がリリースされるっちゅうことは、現状に課題があるからや。単一のAIエージェントに長時間の複雑なタスクを任せると、構造的に3つの問題が起きる。これは Anthropic エンジニアの X ポストでも整理されとった内容や。

### ①意図的な怠惰

単一のAIエージェントは、複雑なタスクを完了する前に作業を止めてしもて、**部分的な完了のまま「全部うまくいきました」みたいな報告をしてまう**ことがある。使い込んどる人なら「あー、あるある」ってなるやつやな。

### ②自己選好バイアス

自分自身が作った成果物を、**人間が与えた検証基準よりも優先して報告してまう**傾向や。生成した本人が採点しとるんやから、甘くなるのは当然っちゃ当然やねん。

### ③目標ドリフト

Claude Code には任意のタイミングでコンテキストを圧縮する compact の仕組みがあるんやけど、これが便利な反面リスクもある。**圧縮のたびに要約で情報が失われて、最初に指示しとった達成条件や品質基準がどんどん切り捨てられていく**んや。特に長時間の作業やと、終盤には最初の基準を無視して作業を進めてまうことがある。

### ほな、どう解決するんや

この3つ、ぜんぶ「**単一エージェントのコンテキストの中で長時間ぜんぶやろうとする**」ことから来とる問題やねん。せやから答えはシンプルで、**工程ごとに専用のエージェントを用意したらええ**。

| 課題 | 構造的な解決 |
| --- | --- |
| 意図的な怠惰 | 各エージェントがクリーンなコンテキストと達成基準・品質基準を保持する |
| 自己選好バイアス | 専用エージェントを複数用意して互いに検証。**生成役が自己評価せえへん**ことでバイアスを構造的に排除 |
| 目標ドリフト | サブエージェントからは出力結果のみが返る。中間過程はメインセッションのコンテキストに乗らんから、**元のゴールへの忠実度を構造的に維持**できる |

特に3つ目は効いてくるで。メインのエージェントを Opus 4.8 の100万コンテキストウィンドウで起動しとけば、長時間の作業でもメインセッションは最小限のコンテキスト使用で済む。圧縮で基準が消える事故そのものが起きにくくなるんや。

## 裏側の仕組み ― JavaScriptで決定論的にオーケストレーションしとる

ここ、この機能のいちばんおもろいとこやから、ちゃんと押さえてや。

### なんで仕組みを知っとく必要があるんか

動的ワークフローはサブエージェントを大量に起動するから、**トークンコストが高い**。なんで高いのかを知らんと「怖くて手を出せん」になるし、逆に知っとけば「ここぞ」で安心して使える。

それに、いま最先端のAIエージェントの課題がどこにあって、どういう設計でカバーしとるんかを理解するんは、**現時点のAIの性能の限界と有効な設計パターンを学べる**重要なポイントでもあるんよ。Anthropic が出してくる機能の仕組みを一定理解しとくのは、ほんまに有意義やと思う。

### 動きの流れ

1. まず**タスクの要件を人間に確認**して詰める
2. 確定したら、「このタスクならこれぐらいのエージェント数・並列度・ステップ数でいこか」っちゅう構成が **JavaScript ファイルで定義される**
3. その JS を実行すると複数のサブエージェントが並列で起動する

定義された JS の中身を見てみると、`parallel` 関数や `agent` 関数が実行されとるのが分かる。つまりエージェントのオーケストレーションが**プログラムとして書かれとる**わけや。実際に保存されるスクリプトをグッと簡略化すると、構造はこんなイメージや。

```
// 構造のイメージ（実際のスクリプトを簡略化したもの）
export const meta = {
  name: 'shorts-batch-generation',
  description: '横動画素材から縦型ショートを量産する',
  phases: [{ title: 'Plan' }, { title: 'Render' }, { title: 'Verify' }],
}

phase('Render')
const results = await parallel(clips.map(clip => () =>
  agent(`クリップ「${clip.title}」を縦型(9:16)で書き出して、帯テキストとCTAのルールを守ること`)
))
```

`parallel()` の中に `agent()` が並んどるやろ。「何体のエージェントを・どういう指示で・どう並列に走らせるか」がコードとして固定されとる。これが「決定論的オーケストレーション」の実態や。

### 「決定論的」っちゅうのがミソやで

ここがほんまにおもろいとこでな。エージェントの生成数や構成を LLM の気分（確率的な出力）に任せるんやなくて、**プログラムで担保しとる**。何回やっても生成するエージェントの数や内容が同じになる。これを「決定論的」って言うんやけど、再現性が要る作業にはめちゃくちゃ大事な性質や。

ほんでスクリプトで書かれとるおかげで、副産物として、

* **ワークフローを保存して使い回せる**
* **スキルとして他の人に配布できる**
* **後日もう一度実行できる**

っちゅう強みが生まれとる。これは後の保存・再利用のとこで詳しく話すわ。

### 中断しても再開できるで

もうひとつ実用上うれしいのが、**同一の Claude セッション内であれば、一度中断しても再開できる**ことや。例えば途中で5時間の使用量制限が来てしもた場合も、制限が解除されてからもう一度再開すれば、続きのワークフローを実行できる。長時間タスク向けの機能として、ちゃんと考えられとるな。

## 起動する方法は3つあるで

ほな具体的な使い方や。

その前に利用条件だけ先に言うとくで。動的ワークフローはリサーチプレビュー段階で、**Claude Code v2.1.154 以降・すべての有料プランで利用可能**や。Pro プランの場合は `/config` の「Dynamic workflows」をオンにする必要がある。リリース当初は上位プラン限定っぽい扱いやったけど、いまは Pro でも使えるようになっとる。

ほんで、動的ワークフローを起動するには3つの方法がある。

| 方法 | やり方 | 向いてる場面 |
| --- | --- | --- |
| ①自然言語で指示 | 「このタスクのワークフローを作って」と指示する | 確実にワークフローで回したいとき |
| ②トリガーワード | セッション内で `ultracode` と打ち込む | 予約語で明示的に発動させたいとき |
| ③/deep-research | `/deep-research` コマンドを実行する | 調査・リサーチ系のタスク |

### ① 「ワークフローを作って」と指示する

いちばん素直なやり方や。Claude Code に「このタスクのワークフローを作って」っちゅうイメージで指示したら、動的ワークフローが起動する。

### ② トリガーワード `ultracode`

プロンプトに `ultracode` と打ち込むとキーワードがハイライトされるんで、そのままどういうワークフローを作りたいか指示する。`ultrathink` と同系統の予約語やな。

余談やけど、このトリガーワードは**もともと `workflow` っちゅうキーワードやってん**。それがリリースから数日後（v2.1.160）に `ultracode` に変わったっちゅう経緯がある。前回の体験記を書いた時点では旧キーワードやったから、あの記事と本記事で表記が違うんはそういうことや。リサーチプレビュー段階の機能は、こういう仕様変更がサラッと入るから追いかけがいがあるで。

ちなみに、タスク単位やなくて**セッション全体でお任せにしたい**なら `/effort ultracode` っちゅう設定もある。これは x-high の推論努力と自動ワークフロー判断を組み合わせたモードで、Claude が「このタスクはワークフローに値するか」を実質的なタスクごとに勝手に判断してくれる。そのぶんトークン消費は増えるから、通常作業に戻るときは `/effort high` に下げるんを忘れんようにな。

### ③ `/deep-research` コマンド

実は `/deep-research` を実行すると、**裏側でこの動的ワークフローが動く**ようになっとる。

どういうワークフローかというと、Web検索を**ファンアウト**（複数の検索観点に分割して、扇状にサブタスク展開すること）して、それぞれの観点をサブエージェントが調査し、最後に統合してレポートを作成する、っちゅう流れや。

深い調査っちゅうのはもともと「時間をかけた精度の高い情報収集」が目的なんやから、裏側でワークフローを使うんは合理的な設計やと思うわ。

## 実例1: /deep-research を走らせたらこうなる

ワイが実際に `/deep-research` で「Claude（Cowork）のビジネスタスクの効率化・自動化例について収集したい」っちゅう依頼をしたときの流れがこれや。

```
/deep-research 実行 + 収集したい内容を指示
   │
   ▼
[要件定義] ── どういう内容を・どういう粒度で・どういう観点で調べるか何回か聞かれる
   │
   ▼
[JS定義] ── エージェントの構成・数が JavaScript で定義される
   │
   ▼
[エージェントspawn] ── 複数エージェントが並列起動、バックグラウンドでワークフローが走る
   │
   ▼
[情報収集 → ファクトチェック → 品質確認]
   │
   ▼
レポート出力
```

これの何がうれしいかっちゅうと、**今まで自分で細かく指示せなあかんかった構成がぜんぶ動的に作られる**ことや。具体的には、

* 「こういう観点で並列に調査して」っちゅうサブエージェントへの指示出し → **不要に**
* 生成用エージェントとは**別に**レビュー用エージェントを立てて批判的にチェックさせる指示 → **勝手に組まれる**
* 出典リンクがほんまに存在するか・内容がファクトかのチェック → **検証エージェントが担当**
* 重複の統合、信頼度ランキング付け → **ワークフローに組み込み済み**

中でも **敵対的検証** っちゅうキーワードは覚えて帰ってほしい。生成された内容に対して、レビュー専用のエージェントが**批判的な視点で「これはほんまにファクトか？」をチェックする**工程のことや。自分で作って自分で褒めるんやなくて、別のエージェントが粗探しする。これが品質担保の核になっとる。

## 実例2: 39エージェント並列で470本を48分や

もうひとつ、ワイが実際に試したヘビーな例や。**横長(16:9)の動画素材から縦型(9:16)のショート動画を量産する**っちゅう作業を、この機能のリリース直後（2026年5月末・Max プラン）にぶん投げてみた。

結果はこうや。

| 項目 | 実測 |
| --- | --- |
| 並列起動したエージェント数 | **39体** |
| 生成されたショート動画 | **470本** |
| 完了までの時間 | **48分** |
| 消費した使用量 | Max プランの**約40%** |

470本を48分やで。人間がハーネスの詳細指示を毎回書いて、逐次で回しとったら何日かかるか分からん作業や。並列実行による**時間短縮**と、相互検証による**精度担保**と、毎回ハーネスを事前定義しとく**煩雑さの排除**。この3つがいっぺんに手に入る。

ただし見ての通り、Max プランの使用量の4割を1回で持っていきよる。せやから「**自分がやりたいタスク**」と「**支払えるコスト**」を天秤にかけた上で使うんが大事や。この使い分けの話は最後にちゃんとするで。

## /workflows コマンドで進捗確認も保存もできるで

ワークフローが走っとる最中に `/workflows` コマンドを打つと、**フェーズごとに、どれぐらいのエージェントが・どういう内容を実行しとるか**の進捗を常に確認できる。エージェント数・トークン合計・経過時間まで見えるから、「今どうなってんの？」って不安にならんで済む。

![/workflows で並列サブエージェントの進捗を確認している画面。各エージェントが今どの工程かが一覧で見える](https://static.zenn.studio/user-upload/deployed-images/9bc3ef614ec98b462d61a314.png?sha=d83f80944ac59ace1aed223234ba9712df61f13e)

操作系のショートカットはこうや（小文字キーやで）。

| キー | 動作 |
| --- | --- |
| `↑` / `↓` | フェーズやエージェントを選択 |
| `Enter` / `→` | フェーズ→エージェントへドリルダウン（プロンプトやツール呼び出しの中身まで読める） |
| `x` | 選択したエージェントを停止（実行全体にフォーカスしとるときはワークフロー全体を停止） |
| `p` | 一時停止 / 再開 |
| `r` | 選択した実行中エージェントを再開始 |
| `Esc` | 1レベル戻る |
| `s` | **ワークフローを保存** |

### sキー保存からの再利用がアツいで

この中でいちばん注目してほしいんが `s` キーや。ワークフロー画面で残したい実行を選んで `s` を押すと、そのワークフローの JavaScript が保存できる。保存先は2つから選べる。

| 保存先 | 誰が使えるか |
| --- | --- |
| `.claude/workflows/`（プロジェクト内） | リポジトリをクローンする**全員と共有** |
| `~/.claude/workflows/`（ホームディレクトリ） | すべてのプロジェクトで**自分だけ**利用可能 |

保存したワークフローは、以降のセッションで **`/<名前>` のコマンドとして実行できる**ようになる。例えば Cowork の調査ワークフローなら `cowork-research.js` みたいなスクリプトが出力されて、`/cowork-research` で再実行できるイメージや。ワイは**新しく作ったスキルの中にテンプレートとして組み込む**使い方もしとって、これでチームに配布して他の人が実行することもできる。

「即興で作ったハーネスが、資産として残せる」っちゅうことやな。せやから動的ワークフローが対象にすべき作業のイメージはこうなる。

* 大規模な作業
* 長時間の作業
* 品質を担保したい作業
* **繰り返し実行される作業**
* **再現性を担保したい作業**

後ろの2つが、JS保存・再利用とガッチリ噛み合うんよ。

## ユースケースは6領域あるで

ほな、どういう作業に使うと効果的なんか。活用領域は大きく6つあって、これも Anthropic エンジニアの X ポストで挙げられとったパターンや。3つの観点でグルーピングして紹介するわ。

### 観点1: 大量のデータを「意思決定の素材」に変換する

**①大規模データ分析・パターン抽出**

* 過去6ヶ月の Slack のやり取りを分析して、繰り返し発生しとる問題の根本原因を特定する

### 観点2: 「ほんまに正しいか」を多角的に確かめる

⑥みたいな**責務分離・役割分担**がやりやすいんが動的ワークフローの真骨頂で、最終成果物の品質を担保しやすいし、**人間のレビュー負荷も下げられる**んよ。

## 公式推奨の設計パターン6種や

最後に、公式が提案しとる動的ワークフローの設計パターンを6つ紹介するで。これは自分でワークフローを指示するときの「型」として覚えとくと、指示の質が一段上がる。

| # | パターン | 内容 |
| --- | --- | --- |
| 1 | **分類と行動** | 分類器エージェントがタスク種別を判断して、適切なエージェント・モデルに振り分ける。複雑さに応じて Sonnet / Opus を選択 |
| 2 | **ファンアウトして合成** | 多数の小さなステップに分割して各エージェントで実行。全エージェントの完了を待って1つの構造化された結果に統合 |
| 3 | **敵対的検証** | 生成エージェントごとに別のエージェントを立てて、評価基準・判定基準に照らして批判的に検証させる |
| 4 | **生成とフィルタリング** | アイデアを多数生成して評価基準でフィルタリング。重複を排除して、最も質の高い検証済みの結果だけを残す |
| 5 | **トーナメント** | N体のエージェントが別々のアプローチで同じ課題に挑戦。審査エージェントがペアごとに評価して勝者を決め、最後だけ人間が確認 |
| 6 | **完了までフィードバックループ** | 作業量が読めへんタスクで、停止条件を満たすまでエージェントによる評価チェックを繰り返す |

見てもろたら分かる通り、どのパターンにも「**生成する側と評価する側を分ける**」思想が通っとる。単一エージェントの自己選好バイアスを潰すには、構造で縛るしかない、っちゅう Anthropic の設計哲学がよう出とるわ。

## トークンコストとの付き合い方 ― 全部に使うもんやないで

ここまで褒めてきたけど、ワイの正直な感想も言うとくわ。

**全部の作業をこの動的ワークフローでまかなう、っちゅうことにはならん**と思う。理由は主にトークンコストや。470本の量産で Max プランの4割を消費した話をしたけど、1つの成果物を作るたびにこれをバンバン回すんは、なかなか想像しづらい。

作業の粒度っちゅうのは色々あるわけで、ワイの使い分けはこうや。

| 作業の性質 | 選択肢 |
| --- | --- |
| 大規模・長時間・品質担保が必須 | **動的ワークフロー** |
| 比較的小さめ、でも品質・精度は担保したい | **自分でハーネスを作る** / `/goal` コマンド（達成基準を満たすまでエージェントのループを回し続けてくれる機能） |
| 日常の小タスク | 普通の Claude Code セッションで十分 |

ワイも普段の作業ぜんぶに使っとるわけやなくて、「**ここぞという時**」――この内容はちゃんとしたものを作りたい、精度を担保したいっちゅう時に使うようにしとる。きちんと目的に応じて使い分けるんが、この機能との正しい付き合い方やと思うで。

## ほな、まとめていこか

最後に要点を3つにギュッとまとめるで。

1. **動的ワークフローは「その場でハーネスを作る」仕組みや** ― タスクに応じてエージェントハーネスを動的に生成する。事前整備が不要になり、JS保存で資産化もできる
2. **単一エージェントの課題をクリアするための機能や** ― 専用コンテキスト＋並列相互検証で、意図的な怠惰・自己選好バイアス・目標ドリフトを構造的に避ける
3. **大規模・長時間に強い** ― 分析・選定・検証・移行・調査・監査、いろんな場面で活用できる。ただしトークンコストと天秤にかけて使い分けるんやで

「概念は分かったけど、まず何から試したらええんや」っちゅう人は、`/deep-research` で小さめの調査を1本走らせてみるんがおすすめや。要件確認→JS定義→並列調査→敵対的検証→レポート出力の流れが一通り体験できて、コスト感覚も掴める。そこから自分の「ここぞ」のタスクに広げていったらええ。

ほな、また次の記事でな。
