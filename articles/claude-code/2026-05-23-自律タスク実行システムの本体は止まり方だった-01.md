## はじめに：正常系は設計の本体ではない

ひとりで事業をやっていると、手を動かせる時間が 1 日に 2〜3 時間しか取れません。「自分が見ていない間も、決めた作業を少しずつ前に進めてほしい」。その素朴な要求から、Notion をタスクの置き場に、cron と Claude Code（`claude -p`）を実行エンジンにした自律タスク実行システムを作りました。

作ってみてはっきりしたことが 1 つあります。自律システムの本体は、正常に進む経路ではなく **「どう止まるか」** の設計だ、ということです。タスクが順調に進む経路の実装は全体の一部にすぎません。途中で壊れたとき、判断が要るとき、無限ループに入ったとき、曖昧な指示を渡されたとき。この 4 つの「止まり方」をどう設計するかが、無人で回せるかどうかを決めました。

この記事は、その設計判断を整理したものです。対象読者は、Claude Code や MCP（Model Context Protocol、AI からツールを呼ぶための接続仕様）を業務自動化に使おうとしている方を想定しています。規模は個人運用レベルで、万能の仕組みではない点は先に書いておきます。

## 全体像：1 回の起動で 1 工程だけ進める

```
[人: Notion]  Backlog にタスク追加 → Ready にドラッグ
        │
[cron]  10 分おきに loop.sh を起動
        │
[loop.sh]  軽量モデルで dispatcher を起動するだけ
        │
[dispatcher]  Tasks DB を確定論的に全件取得
              Ready を検証 → 作業領域を用意 → Status=Running
              run.sh をデタッチ起動
        │
[run.sh]  多重起動防止 / 緊急停止 / 異常終了リカバリ
          timeout 付きで高性能モデルを起動
        │
[auto-loop]  次の 1 工程を実行 → Notion とファイルを更新
```

設計の起点は **1 回の cron 起動で 1 工程しか進めない** ことです。各工程は毎回まっさらなコンテキストで起動します。長い対話を引きずらない代わりに、状態はすべて Notion とローカルファイルに書き出し、次の起動が読み直す。この「状態を全部外に書く」方針が、後述する 4 つの止まり方をすべて支えています。

以降は、正常系の実装ではなく、この設計の重心である「止まり方」を 4 つの判断として整理します。

## 止まり方 1：無限ループから機械的に止める

自律実行で一番怖いのは「同じ工程で延々と失敗し続ける」状態です。判断ロジックにバグがあれば、システムは「もう少しで成功する」と思い込んだまま永遠に回り続けます。内容を判断する仕組みでこれを止めようとすると、その判断自体がバグの影響を受けます。

ここでの設計判断は、**内容ではなく回数で止める** ことでした。各 STEP の起動回数（Attempts）を必ず数え、上限に達したらメカニカルに BLOCKED にします。これは「成果物が良いか」を一切見ません。ただ回数を数えるだけです。判断ロジックがどれだけ壊れても、この安全網は判断を介さないので必ず作動します。

ここで数えるカウンタを 2 種類に分けています。

* **Fail Count**：レビュー工程の不合格回数。レビュー周回の上限を決める。
* **Attempts**：その STEP でセッションが起動された回数。無限ループ防止の指標。

両者を混同すると、「レビューには通っているが起動だけ繰り返している」状態を見逃します。役割が違うものは別々に数える。地味ですが、止まり方の信頼性はこの分離に依存しています。

## 止まり方 2：強制終了されても次の起動が拾える

プロセスはいつでも停止しうる前提です。cron プロセス、タイムアウト、OS 停止など、「いつ停止してもおかしくない」とすると、設計は「停止した後どう復旧するか」に向きます。

`run.sh` は起動冒頭で前回の異常終了を検知し、古いロックファイルや中途半端な一時ファイルを掃除し、未コミット変更を退避用ディレクトリに逃がします。そのうえで、状態を Notion とローカルファイルから読み直して再開します。プロセス用メモリ領域に状態を載せず、ストレージに載せる。だから途中で停止しても次の cron が最新状態を拾えます。

`claude -p` には強制終了のための `timeout` を付け、暴走しても必ず終わるようにしています。ただし dispatcher と loop 側には `timeout` を付けません。これらが中断されると Notion の状態が中途半端になり、かえって復旧を難しくするためです。**「必ず止める」と「中断してはいけない」を工程ごとに切り分ける**。一律にタイムアウトをかけないのは、止め方を雑にしないための判断です。

## 止まり方 3：人間が即座に止められる

自動化を信用するには、いつでも自分の手で止められる確信が要ります。`STOP` ファイルを 1 つ置けば、次回起動時に即座に停止する。仕組みはこれだけです。

シンプルにしたのは意図的です。緊急停止の経路に判断や条件分岐を入れると、止めたいときに限ってその判断がバグります。**緊急停止は、判断を挟まず、最も単純な機構で実装する**。止め方の信頼性は、機構の単純さと比例します。

## 止まり方 4：曖昧な指示はそのまま全力実行しない

無人で回す以上、一番たちが悪いのは「曖昧な指示をそのまま全力で実行し続ける」ことです。そこで実行前のチェックを 2 段構成にしました。

* **(A) 構造検証**：自動実行するための必須パラメータが埋まっているか。成果物が外部ストレージ指定ならフォルダの実在まで確認する。
* **(B) 妥当性ゲート**：埋まっていても、完了の定義が不明瞭だったり、ゴールがタスク遂行にとって不適切（漠然・矛盾・「無限に作業して」のような終端不能）なら、進めずに理由つきで通知する。

(B) が設計判断の核です。構造が埋まっていることと、指示が遂行可能であることは別物です。ただし、ここで保守的になりすぎると今度は過剰な保留に振れます。判断基準は「明確に不適切なときだけ止める」。軽い曖昧さは止めず、計画工程に委ねる。**止めすぎも止め方の失敗である**、という線引きをしています。

## 設計判断の検証：MCP の制約を実機で確定させる

止まり方の設計と並んで学びになったのが、「できないように見えること」の扱いです。

当初、dispatcher は Notion MCP の `fetch` でデータソースの行を列挙する設計でした。ところが、新しく Ready にしたタスクが検知されない事象が出ました。切り分けた結果、この MCP には次の制約があるとわかりました。

* `notion-fetch` をデータソースに対して実行すると、返るのはスキーマ定義だけで行データは 0 件。
* `notion-search` は意味検索で、関連度順に上位だけ返す。クエリに一致しない行は索引済みでも黙って脱落する。
* 行を全件列挙する専用ツールは公開されていない。

つまり「未知の名前の行を漏れなく列挙する」確実な手段が MCP のツール面には無い、という結論でした。検索ベースの回避策では「タスクが一生実行されない」状態から抜け出せませんでした。

解決方法は Notion の **REST API** でした。`POST /v1/data_sources/{id}/query` は `filter` によるサーバ側の絞り込みで、Ready/Running をページング付きで確定論的に全件返します。MCP のツール面の制約に閉じ込められていただけで、API そのものには列挙手段があった、というオチです。

```
curl -X POST "https://api.notion.com/v1/data_sources/${DS_ID}/query" \
  -H "Authorization: Bearer ${NOTION_API_TOKEN}" \
  -H "Notion-Version: 2025-09-03" \
  -H "Content-Type: application/json" \
  -d '{"filter":{"or":[
        {"property":"Status","select":{"equals":"Ready"}},
        {"property":"Status","select":{"equals":"Running"}}]},
       "page_size":100}'
```

教訓は単純、**ツールとして公開されている操作が、その API のすべてではない**。「できないように見える」ときは、MCPに頼らず API の仕様を探る。当たり前のことを再確認しました。

参考: [Query a data source – Notion API](https://developers.notion.com/reference/query-a-data-source) / [Upgrade guide 2025-09-03](https://developers.notion.com/docs/upgrade-guide-2025-09-03)

## 実践への示唆

この設計から、自社で同種の自動化を組む方に渡せる判断基準を 3 つにまとめます。

1. **正常系より「止まり方」に時間を割く**。順調に進む経路は短い。無人運用の品質は、Stuck・強制終了・緊急停止・曖昧な指示の 4 つをどう止めるかでほぼ決まります。
2. **止める判断は、判断ロジックの外に置く**。回数で機械的に止める安全網と、単一ファイルでの緊急停止は、判断を介さないからこそバグに強い。
3. **状態を全部プロセスの外に書く**。プロセスは死ぬ前提にし、Notion とファイルに現在地を残す。これがすべての止まり方と復旧の土台になります。

限界もはっきりあります。これは個人運用規模で、モデルの実行品質に依存し、レビュー工程の厳しさをこちらで設計しないと品質は担保されません。「カンバンに書けば勝手に良い成果物が出る」ものではなく、「決めた工程を、止まるべきときに止まりながら、止まらないような適切なタスク指示を出せるようになる」ための仕組みです。

## まとめ

* 自律タスク実行システムの本体は、正常に進む経路ではなく「どう止まるか」の設計にある。
* 止まり方は 4 つ。無限ループの機械的停止／強制終了からの復旧／単純な緊急停止／曖昧な指示の妥当性ゲート。
* 止める判断は判断ロジックの外に置き、状態はプロセスの外に書く。これが復旧と信頼性の土台になる。
* ツールの制約に当たったら API の仕様を確かめる。今回は Notion REST API の data source query で解決した。

派手さはありませんが、手を離している間も決めた作業が少しずつ進むようになりました。同じように「人手が足りない側」で自動化を組む方の判断の足場になれば幸いです。
