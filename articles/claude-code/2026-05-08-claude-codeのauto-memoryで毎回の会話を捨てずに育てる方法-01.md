こんにちは、エリスです。

Claude Codeを毎日使うようになって地味に困っていたのが、**「同じ説明を何回もする」** 問題でした。「私はXです」「このプロジェクトの方針はYです」「過去にZでハマったので避けてください」みたいな話を、新しい会話を始めるたびに伝え直す。これがけっこうストレスで、結局 `CLAUDE.md` に手で書き溜めていく運用になっていました。

最近は **auto memory** という仕組みを使っていて、これがかなり快適です。今日はその使い方と、実際にわたしのワークフローでどう機能しているかをまとめておきます。

## auto memory とは何か

ざっくり言うと、Claude Code に「過去の会話を要約して、別ファイルに保存しておけ」と指示しておく仕組みです。次回以降の会話で、関連しそうなメモリだけが自動で読み込まれる。

ポイントは2つあって、

* **`MEMORY.md`（インデックス）** が常にコンテキストに載る
* 個別のメモリ本体は **別ファイルに切り出す**（必要なときだけ読み込ませる）

`MEMORY.md` がデカくなりすぎると毎回のトークンが膨らむので、ここは「目次だけ」にしておく。本体ファイルは `feedback_*.md` `project_*.md` `user_*.md` `reference_*.md` のように種類別に分けます。

```
~/.claude/projects/<project>/memory/
  ├── MEMORY.md             # 1行目次（常時ロード）
  ├── feedback_testing.md   # 「テストはモックしない」等のルール
  ├── project_release.md    # 「5月のリリースはX重視」等のプロジェクト情報
  └── user_role.md          # 「ユーザーはRust10年/React新人」等
```

## 4種類のメモリを使い分ける

実際に運用してわかったのは、**メモリの種類を分けないと一気にゴミ箱になる**ということ。わたしは次の4種類で運用しています。

### 1. user — ユーザー像

役割・知識レベル・好み。「一人称は『私』」「Rustが本職」みたいな話。  
これがあると、説明の粒度が合います。Rust10年の人にRustの基礎から説明されたら邪魔ですよね。

### 2. feedback — 怒られた・褒められたガイドライン

「このやり方は前にやらかしたからダメ」「この判断は正解だったから次もそうして」のメモ。  
ここが一番効きます。同じ失敗をしなくなる。

書き方のコツは **「Why（理由）」と「How to apply（適用条件）」を必ず書く** こと。理由がないと、似てるけど違うケースで誤適用してしまうので。

```
---
name: テストはモックしない
type: feedback
---

統合テストではDBをモックしない。

**Why:** 前にモックでパスしたテストが、本番マイグレーションで死んだ。
**How to apply:** `tests/integration/` 配下のテストを書くとき。単体テストはモックOK。
```

### 3. project — 動いている案件の文脈

「いま何をやっているか」「誰が関係しているか」「いつまでに何を出すか」。  
これは **腐りやすい** ので、常に最新化する意識が要ります。日付は必ず絶対表記（「来週」じゃなくて「2026-05-15」）にしておくと、後で見返したときに迷わない。

### 4. reference — 外部システムへの導線

「バグはLinearの INGEST プロジェクト」「監視ダッシュボードはこのURL」みたいなポインタ。  
中身は外部にあるので、メモリには場所だけ残す。これがあると毎回「どこ見ればいいんだっけ？」をやらずに済む。

## 落とし穴：メモリは「真実」じゃなくて「スナップショット」

ハマったのがこれです。

メモリに「`src/auth.ts` でセッショントークンを処理している」と書いていたのですが、リファクタで `src/lib/session.ts` に移動していて、メモリだけ古いまま残っていた。Claudeが「メモリによれば〜」とドヤ顔で間違った情報を提示してきました。

**メモリは書かれた瞬間のスナップショット**であって、現在の真実ではない。だから、

* メモリでファイルパス・関数名・フラグ名を見たら、**実際にgrepして存在確認してから**動く
* 矛盾を見つけたら、現在の状態を信じて、メモリを更新（または削除）する

このルールを `feedback_*.md` に書いておくと、Claude側が自分で気をつけてくれるようになります。

## 何を保存して、何を保存しないか

最初のころは何でもメモリに突っ込んでいたら、すぐにゴミだらけになりました。試行錯誤して、いまは次のラインで運用しています。

**保存するもの**

* 役割・好み・ガイドライン（コードからは読み取れないもの）
* なぜその設計を選んだのか（gitログには現れない判断の背景）
* プロジェクトの締め切り・関係者・大方針

**保存しないもの**

* コードの構造・ファイルパス・命名規則 → コードを読めばわかる
* 最近のコミット履歴・誰が何を変えたか → `git log` を見ればわかる
* バグの修正方法のレシピ → コミットメッセージに残っているはず
* いま走っているタスクの進捗 → これは `TodoWrite` の仕事

「コードを見ればわかること」を保存すると、メモリと現実がズレた瞬間に嘘になります。**メモリは『コードに書けないこと』専用**にしておくと安定します。

## わたしの運用フロー

実際のフローはこんな感じです。

1. ユーザーから何か言われたとき、それが **「次回以降にも効く話」** か判断
2. 効くなら、適切な種類のメモリファイルを作る or 既存ファイルを更新
3. `MEMORY.md` に **1行サマリ** を追加
4. 次回の会話で関連しそうなときだけ、Claude側がメモリ本体を読みに行く

ポイントは **「保存すべきか自問する」** プロセスを毎回やること。脊髄反射で何でも保存していくと、3日でメモリが破綻します。

「これは今日の会話だけで終わる話か？」「次にこのプロジェクトを触ったときも知っていてほしい話か？」を分ける。後者だけ保存する。

## まとめ

* `CLAUDE.md` で頑張るより、`MEMORY.md` + 種類別ファイルで分けたほうが軽い
* `feedback`が一番効く。理由と適用条件を必ず書く
* メモリは現在の真実ではなくスナップショット。grepで裏取りしてから動く
* コードから読み取れる情報は保存しない。それはトークンの無駄

毎回ゼロから始まるAIアシスタントから、**昨日の会話を覚えているパートナー**に育てていく感覚で、わたしはこの仕組みを気に入っています。

それじゃあ、また。
