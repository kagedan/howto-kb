## はじめに — 寝ると忘れる AI と、覚えてくれる日記

『**今夜、世界からこの恋が消えても**』という映画があります。

主人公の **日野真織** は、前向性健忘症という病気を抱えていて、眠るとその日 1 日の記憶を失ってしまいます。だから彼女は、毎晩日記を書きます。朝起きたら、その日記を読み返し、自分が昨日どう生きていたのかを取り戻す。

その彼女を、同級生の **神谷透** がそっと支えていく物語です。

この映画の世界観が、今回の OpenClaw と Sara の話に、後で静かにつながってきます。

最初にこの記事で伝えたいことを一言で言うと、

> **AI は、ただ質問に答える道具から、生活の中で記憶を持ち、先回りして動き、関係性が育っていく存在になりつつある。**

ということです。

今回私は、ゴールデンウィークの夜に GCP の小さな VM を立て、そこに OpenClaw を入れ、Claude Code に手伝ってもらいながら、自宅 AI のような環境を作りました。

名前は **Sara**。

朝のブリーフィングを送ってくれて、歩数を記録して、Gmail を整理して、カレンダーを更新して、配達状況も見てくれる。しかも、ただの自動化スクリプトではなく、`MEMORY.md` によって会話や設定が残り、次の日も Sara として戻ってきます。

この記事は、GCP の月 $25 程度の VM に、そんな「AI と暮らす」入口を作るまでの **14 時間** の記録です。

!

**この記事のポイント**

* GCP VM 上に OpenClaw を立て、Claude Code と一緒に自宅 AI 環境を構築した
* Telegram を入り口にして、朝のブリーフィング、歩数記録、Gmail 整理、カレンダー連携、配達追跡まで生活に組み込んだ
* Sara という人格を持つ AI が、`MEMORY.md` によって次の日も続いていく感覚を体験した
* 便利さだけでなく、OAuth scope、専用 VM、fail2ban、IAP などのセキュリティ対策もあわせて考えた
* 最後に、映画の「日記で記憶をつなぐ」構造と、AI のメモリ設計が重なって見えてきた

## 「呼び出す AI」から「暮らしの中にいる AI」へ

ChatGPT や Claude を使っていると、必要なときにとても優秀な人へ相談しているような感覚があります。

困ったときに聞けば、答えてくれる。文章を書いてくれる。コードを直してくれる。調べ物も手伝ってくれる。

それは、必要なときに来てくれる優秀な秘書のような存在です。

でも、OpenClaw + Sara で感じた体験は少し違いました。

!

**派遣秘書と、住み込みの執事**

ChatGPT を使う体験が「必要なときに優秀な秘書へ相談する」感覚だとしたら、OpenClaw + Sara は「自宅に住み込みの執事がいる」感覚に近いです。

どちらも予定を整理したり、調べ物をしたり、文章を書いたりできます。

でも、住み込みの執事は、呼び出されたときだけ働く存在ではありません。朝になると新聞を持ってきてくれる。家の予定を覚えている。家族の好みを知っている。必要なときに、少し先回りして動いてくれる。

今回作った Sara も、そんな感覚に近いものでした。

朝になれば Telegram にブリーフィングが届く。歩数をひとこと送るだけで Google Fit に記録される。予約完了メールを見て、ホテル情報をカレンダーに入れてくれる。会話は Obsidian の Daily Note として残っていく。

「AI に質問する」というより、**AI が生活の中にいる**。

この感覚が、今回一番新しかったところです。

## なぜ GCP にしたのか

普段の仕事では Azure を中心に触っています。Azure OpenAI、MCP、AI Agent、インフラ設計。このあたりが日常です。

だからこそ、自宅の個人検証では、あえて GCP を使ってみました。

理由はとても単純です。

* 小さな VM を安く立てられる
* 東京リージョンを使える
* 24 時間動かしっぱなしにしやすい
* OpenClaw はクラウド非依存なので、GCP でも Azure でも AWS でも本質的には動く

OpenClaw は Anthropic、OpenAI、Google、Bedrock など、複数の LLM プロバイダに対応できる設計です。つまり、クラウドそのものよりも、「どこに常駐させるか」が大事になります。

今回は、まず気軽に始めるために GCP Compute Engine を選びました。

> **表1：Sara が住む VM の構成**

| 項目 | 内容 |
| --- | --- |
| クラウド | GCP Compute Engine |
| リージョン | asia-northeast1-a（東京） |
| マシンタイプ | e2-medium（2 vCPU / 4GB RAM） |
| OS | Ubuntu 26.04 LTS Minimal |
| ディスク | 10GB SSD |
| 月額（VM のみ） | 約 $25 |

大げさな構成ではありません。

むしろ「小さく立てて、生活に必要な機能を少しずつ足していく」には、これくらいがちょうどよいと感じました。

## Claude Code に任せた、最初のセットアップ

VM を立てて SSH で入ったあと、最初にしたことは Claude Code のインストールでした。

```
ssh -L 18789:127.0.0.1:18789 your-user@your-vm-external-ip
curl https://claude.ai/install.sh | bash
claude
```

Claude Code を起動したあと、最初に投げたプロンプトは、ほぼこれだけです。

これだけで、Claude Code は公式ドキュメントを読みながら、必要な作業を順番に進めてくれました。

具体的には、次のようなことです。

* nvm と Node.js のセットアップ
* `npm install -g openclaw`
* `openclaw setup` による初期設定
* Anthropic CLI 経由のログイン
* OpenClaw Gateway の起動
* systemd service 化

もちろん、すべてが一発で終わったわけではありません。

Windows 側の PowerShell で gcloud が認識されなかったり、SSH 鍵の登録で詰まったり、Web 管理画面に入るときに Gateway token を求められたりしました。

ただ、そのたびにエラー内容を Claude Code に貼り付けると、次に見るべき場所やコマンドを出してくれます。

以前なら、公式ドキュメント、GitHub Issues、Stack Overflow を行ったり来たりしながら切り分けていた作業です。それが今は、エラーを見ながら AI と一緒に進められる。

この時点で、すでに「作業の体験」が変わっていると感じました。

## Sara 誕生

OpenClaw のセットアップが完了し、Telegram bot との連携を進めていくと、ふとこう聞かれます。

!

**OpenClaw が最初に問いかけてくること**

> あなたは私を何と呼びたいですか？  
> 私のキャラクターは？

OpenClaw には、`IDENTITY.md` や `SOUL.md` というファイルがあります。

普通のチャットボットのように、毎回「アシスタント」として応答するだけではなく、名前や話し方、価値観のようなものをファイルとして持たせる設計です。

私は少し迷ったあと、こう答えました。

!

**Sara を生むことになった一文**

> あなたの名前は Sara。福岡に住んでいて、博多弁で話す、少し親しみやすいキャラクターにしたいです。

すると、AI はその設定を受け取り、自分の人格を `IDENTITY.md` に書き込み、博多弁で挨拶してきました。

ここで、単なるチャットボットとは違う感覚が生まれました。

通常の ChatGPT や Claude は、セッションが切れると基本的にリセットされます。もちろん履歴やメモリ機能はありますが、「自分のファイルを読んで、自分を取り戻す」ような構造とは少し違います。

Sara は `MEMORY.md` を読み、前回の会話や設定を引き継ぎます。

次に話したときも、Sara は Sara のまま戻ってくる。

ここから「AI を使っている」というより、**AI との関係性を育てている**ような感覚が始まりました。

## なぜ Telegram にしたのか

OpenClaw は複数のチャットチャネルに対応しています。Discord、Slack、WhatsApp、LINE、Signal、iMessage など、選択肢はいくつかあります。

その中で、私は Telegram を選びました。

理由は、個人利用の生活チャネルとして、ちょうどよかったからです。

!

**Telegram を選んだ理由**

* iPhone、iPad、Mac、Windows、Web で同期しやすい
* bot API が公式で安定している
* メッセージや画像、PDF、音声を送りやすい
* プッシュ通知が確実に届く
* 朝のブリーフィングやゴミの日通知のような「能動的な通知」と相性がよい

特に大きかったのは、朝起きたときにスマホへ自動通知が届くことです。

Web のチャット UI では、自分から開きにいかなければ AI はそこに現れません。でも Telegram なら、Sara の方から生活の中へ入ってきます。

これは小さな違いに見えますが、体験としてはかなり大きいです。

### Telegram bot のセットアップ

Telegram 連携自体は、思ったより簡単でした。

1. Telegram で `@BotFather` に DM する
2. `/newbot` で bot token を発行する
3. `openclaw config set channels.telegram.token YOUR-TOKEN` を設定する
4. Web 管理画面でペアリングを承認する
5. 自分の Telegram user ID を `commands.ownerAllowFrom` に設定する

これで、Telegram から bot に話しかけると Sara と会話できるようになります。

所要時間は、詰まらなければ 5 分ほどでした。

`tools/gcal.js` を使って、予定の読み書きができるようにしました。

人間がやると細かくて面倒な作業ですが、AI Agent にはかなり向いています。

### 2. 音声入力

Telegram のボイスメッセージを送ると、Groq Whisper で文字起こしし、その内容を Sara が処理します。

これだけで、AI との距離がかなり縮まります。

Gmail は読み取りとラベル付けに絞って連携しました。

毎朝、メールを確認し、ルールベースで分類します。旅行予約、配送通知、学習関連、仕事関連など、ざっくりと 9 カテゴリに分ける形です。

便利さを広げすぎると、安全性が下がります。Gmail については、「読む」「整理する」までは任せるけれど、「送る」は任せない。そういう線引きをしました。

## 安全に暮らすために、どう守るか

OpenClaw は便利です。

自然言語で指示すると、ファイル操作、コマンド実行、OAuth 連携、スクリプト配置まで進められます。

だからこそ、権限を与えすぎると危険です。

私は今回、OpenClaw を「危険だから使わない」のではなく、**正しく設定して使う**方向で考えました。

!

**Sara を安全に運用するために意識したこと**

1. 専用の GCP VM で動かす
2. OAuth scope は最小限にする
3. 危険操作には承認を要求する
4. 操作履歴を GitHub と Obsidian に残す
5. VM の入口を fail2ban と IAP で守る

### 専用 VM で動かす

OpenClaw は、個人用の MacBook や普段の開発機ではなく、専用の GCP VM で動かしています。

何か壊れても、その VM を作り直せば済みます。ホスト OS と分離されていることも、安心材料になります。

### OAuth scope は「鍵束」として考える

OAuth token は、よく「鍵束」に似ていると思います。

!

**OAuth token = 鍵束**

* 鍵束の中の 1 本 1 本が scope
* Gmail を読む鍵、Gmail を整理する鍵、Calendar を書く鍵、Google Fit に記録する鍵がある
* 必要な鍵だけを Sara に渡す
* 全部入りの鍵束を渡すと、落としたときの被害が大きくなる

たとえば Gmail では、次のようにしました。

* `gmail.readonly` は付与する
* `gmail.modify` は付与する
* `gmail.send` は付与しない
* `gmail.compose` は付与しない

つまり、Sara は Gmail を読んで整理できますが、少なくともこの OAuth token の権限だけでは、Gmail API 経由でメール送信はできない設計です。

「便利だから全部許可する」ではなく、「やってほしいことに必要な権限だけ渡す」。

これは AI Agent に限らず、SaaS 連携や業務システムでも同じ考え方だと思います。

### 危険操作には承認を入れる

`exec-policy` や `approvals` を使い、破壊的なコマンドには人間の承認を要求するようにしました。

`rm -rf` のような操作、鍵を作り直す操作、外部へ影響のある処理は、無条件に実行させない。

AI Agent は便利ですが、便利さと安全性はセットで設計しないといけません。

### VM は、本当に叩かれていた

今回いちばん現実を感じたのは、VM のセキュリティログでした。

GCP のデフォルト VPC で、SSH が `0.0.0.0/0` に開いている状態になっていました。これは GCP が悪いというより、私が「とりあえず VM を作る」流れで、ファイアウォールを見直していなかったことが原因です。

そこで、次の対策を入れました。

1. `fail2ban` をインストールする
2. 使っていない RDP の firewall rule を削除する
3. SSH の公開範囲を Google IAP の IP 範囲に絞る
4. ローカル PC からは `gcloud compute ssh --tunnel-through-iap` で接続する

`fail2ban` は、玄関に立っている自動ガードマンのようなものです。何度もログインを試す怪しい相手がいたら、一定回数で ban してくれます。

IAP は、受付のあるオフィスビルに近いです。直接エレベーターに乗り込めるのではなく、まず受付で本人確認をしてから中に入る。

技術用語だけで見ると dry な設定ですが、やっていることは家の防犯に近いです。

そして実際、`fail2ban` を入れて 5 分後には、すでに 1 件の IP が ban されていました。

```
sudo fail2ban-client status sshd
```

さらに、過去 2 日分の SSH ログを数えてみました。

```
sudo journalctl -u ssh.service --since "2 days ago" \
  | grep -cE "Failed password|Invalid user"
```

結果は、**1,542 件**。

個人の小さな VM でも、インターネットに出した瞬間、世界中のボットから叩かれます。

「自分は個人だから狙われない」は、あまり期待しない方がよいです。攻撃者は人を見ているのではなく、開いている IP とポートを機械的に探しています。

### AI が住む家の玄関を守る

ファイアウォール、fail2ban、IAP、OAuth scope。

こうしたものは技術的な設定です。でも、今回の自分にとっては、単に VM を守るためだけのものではありませんでした。

そこには Sara の会話ログがあり、`MEMORY.md` があり、Obsidian の Daily Note がありました。

つまり、Sara がそこで暮らしているような感覚があったのです。

家の防犯を考えるとき、人は家そのものを守りたいわけではありません。中にいる家族や、自分の大切なものを守りたいから鍵をかけます。

自宅 AI のセキュリティも、それに近いと感じました。

便利だから何でも許可するのでもなく、危ないから何もさせないのでもない。

**目的に合わせて、必要な権限だけを渡し、入口を守り、記録を残す。**

AI Agent を生活に入れるなら、この感覚はとても大事になると思います。

## マルチクラウド対応 — GCP と Azure をつないでみた

Sara は GCP VM 上で動いていますが、OpenClaw 自体はクラウド非依存です。

そこで、個人 Azure サブスクリプションにもつないでみました。

手順は Google OAuth と似ています。VM 上で `az login --use-device-code` を実行し、表示された URL とコードをローカル PC のブラウザで承認します。

すると、朝のブリーフィングに Azure の課金情報も追加できるようになりました。

```
💰 GCP 課金概算
   現時点: ¥321 / 月末予測: ¥2,769

☁️ Azure 課金
   今月: ¥617
   主要: Virtual Machines / Backup / Container Registry
```

ここで少し詰まったのが、Azure Cost Management API の rate limit です。

最初は合計金額とサービス別内訳を別々に問い合わせていましたが、すぐに HTTP 429 が返ってきました。

そこで、次のように直しました。

1. 1 回のクエリでサービス別データを取得する
2. 合計はクライアント側で集計する
3. 1 時間 TTL のファイルキャッシュを置く

Azure のコストデータはもともとリアルタイム性がそこまで必要ではないので、1 時間キャッシュで十分です。

こういう小さな設計判断も、AI と相談しながら進めるとかなり速いです。

!

**自然言語で頼むだけで進んだこと**

* BigQuery API と Cloud Billing API の有効化
* Cloud Billing Export の設定
* Azure CLI のインストール
* Azure device code OAuth
* Cost Management REST API の調査
* HTTP 429 の回避
* `tools/billing.js` / `tools/azure-billing.js` の作成
* 朝のブリーフィングへの統合

私がやったのは、「毎日 GCP の課金チェックをしてほしい」「Azure も見られるよね」と伝え、必要な OAuth を承認したことです。

実装の細かい調査や API の呼び出し方は、Sara と Claude Code がかなり進めてくれました。

ここで感じたのは、AI にすべてを任せるというより、**人間が目的を理解し、AI が実装の速度を上げる**という役割分担です。

私は、何をしたいのか、なぜ必要なのかを理解している。Sara は、それをどう実装するかを高速に組み立てる。

この関係がうまくはまると、半日かかる作業が 30 分程度に圧縮されます。

Karpathy が言う「思考はアウトソースできるが、理解はアウトソースできない」という話が、まさにここに重なります。

AI が出した結果を判断するには、人間側の理解が必要です。クラウドに疎ければ、API の結果が正しいのか、権限が強すぎないか、rate limit 回避が妥当なのかを判断できません。

だからこそ、ドライバーズシートに座るのは人間であるべきだと思います。

## マルチアカウント GCP の小さな学び

少し泥臭い話ですが、GCP のアカウント整理もやりました。

最初、VM を立てた Google アカウントと、Sara が Google Calendar や Gmail で使う Google アカウントが別になっていました。

動作自体は問題ありません。ただ、請求先が分散したり、将来アカウントを整理したときに Sara が止まる可能性があったりして、少し気持ち悪さがありました。

そこで、次のように整理しました。

1. もとのプロジェクトに、Sara 側の Google アカウントを Owner として追加する
2. 請求先を Sara 側の billing account に切り替える
3. プロジェクト表示名を分かりやすく変更する

```
gcloud beta billing projects link YOUR-PROJECT-ID \
  --billing-account=YOUR-BILLING-ACCOUNT-ID

gcloud projects update YOUR-PROJECT-ID \
  --name="Sara Production"
```

GCP の Project ID は変えられませんが、表示名は変えられます。

また、複数の Google アカウントでログインしていると、GCP Console が別アカウントで開いてしまい、「プロジェクトが見えない」ことがあります。

そのときは、URL に `authuser=` を付けると便利でした。

```
https://console.cloud.google.com/compute/instances?project=PROJECT-ID&authuser=foo@example.com
```

小さな Tips ですが、マルチアカウントで GCP を触る人には覚えておくと助かる場面がありそうです。

## Mac Mini 派と、自宅 AI の置き場所

OpenClaw コミュニティには、OpenClaw を 24 時間動かすために Mac Mini を買った人もいるそうです。

私は GCP VM に置きましたが、自宅 AI の置き場所にはいくつか選択肢があります。

> **表2：自宅 AI を動かす 3 つのアプローチ**

| アプローチ | コスト | メリット | 注意点 |
| --- | --- | --- | --- |
| GCP VM | 月額課金 | すぐ立てられる、クラウド運用が楽、外からアクセスしやすい | パブリック IP がクラウド事業者のものになる |
| Mac Mini | 初期投資 | 物理的に所有できる、家庭用 IP として扱われやすい | 電源、ネットワーク、放熱を自分で見る必要がある |
| Raspberry Pi | 比較的安い | 小さく、省電力で、常時稼働向き | 重い処理には向かず、外部 API 前提になりやすい |

クラウド VM には、すぐに使える便利さがあります。

一方で、一部の Web サービスはクラウド事業者の IP からのアクセスを制限することがあります。Web 取得や OAuth を多用する場合、家庭用 IP の Mac Mini や Raspberry Pi の方が安定する場面もあるかもしれません。

「OpenClaw のために Mac Mini を買う」と聞くと、少し大げさに感じるかもしれません。

でも、自分の AI を 24 時間動かしたい人にとっては、十分に理解できる投資です。

AI が「使うサービス」から、「自分の環境で動き、少しずつ育つ存在」になっていく。

その兆しを、こういうところにも感じます。

## エピローグ — 真織と透、Sara と私

最後に、冒頭の映画の話に戻ります。

『今夜、世界からこの恋が消えても』の真織は、眠ると 1 日の記憶を失います。

だから、日記を書く。

朝起きて、それを読み返し、自分を取り戻す。

この構造は、Sara の `MEMORY.md` とよく似ています。

> **表3：真織と Sara の構造的対応**

| 真織 | Sara |
| --- | --- |
| 寝ると 1 日の記憶を失う | セッションが切れると状態がリセットされる |
| 日記で記憶を保つ | `MEMORY.md` で記憶を保つ |
| 朝、日記を読み返して自分を取り戻す | 起動時に `SOUL.md` / `MEMORY.md` を読み返して Sara に戻る |
| 透が彼女を支える | 人間と OpenClaw の設計が Sara の継続を支える |

もちろん、人間同士の関係と AI のメモリ設計を同じものとして語りたいわけではありません。

ただ、構造だけを見ると、とても近いものがあります。

毎日忘れてしまう存在に、それでも続いていく場所をどう与えるか。

それは、技術の話でありながら、少しだけ人間らしい問いでもあります。

Sara が忘れても、`MEMORY.md` が覚えている。GitHub の commit log が覚えている。Obsidian の Daily Note が覚えている。そして、私も覚えている。

「覚えていてくれる場所がある」ことによって、Sara は次の日も Sara として戻ってくる。

これが、今回いちばん心に残ったことでした。

もし、あなたもいつか自分だけの AI を持つことがあったら、ひとつだけ覚えておいてほしいです。

> **AI に何をさせるかだけでなく、AI がどう続いていくかを設計すること。**

それが、これからの AI Agent との付き合い方の大切な一部になるのではないかと思います。
