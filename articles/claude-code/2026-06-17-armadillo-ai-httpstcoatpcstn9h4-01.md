Claude Codeを、ただコードを速く書く便利な道具として使っているなら、その実力の半分も引き出せていません。

関数を1つ書かせる。エラーを貼って直し方を聞く。テストの雛形を作らせる。どれも便利ですが、それは他のAIチャットでもできることです。Claude Codeにしかできないことは、別のところにあります。

Anthropicの公式ベストプラクティスは、Claude Code活用の最重要資源を「コンテキストウィンドウ」と位置づけています。ファイルの読み取りやコマンドの出力でそこは急速に埋まり、埋まるほど忘れやミスが増える。だから常に管理が要る、と明記されています。

そしてこの資源を活かす鍵は3つに絞れます。検証する手段を渡して自分で反復させること。git worktreeで複数のClaudeを並列に走らせること。CLAUDE.mdやskillで使うほど賢くする仕組みを作ること。ここを使わない限り、Claude Codeを選ぶ理由は薄いのです。

この記事では、開発元のBoris Chernyとチーム本人が公開している実践Tipsから、効果の大きい手法を弱いものから順に、具体的なコマンドと指示文つきで6つ紹介します。

# なぜ「1問1答」で使うとClaude Codeの価値が出ないのか

多くの人はAIコーディングを「質問して、答えをもらう」やり取りだと思っています。

これは半分正しく、半分間違いです。1問1答が向くのは、タイポ修正のように1文で差分を説明できる小さな変更だけ。公式も、こうした小変更は計画を省いて即実行するのが効率的だと言っています。

問題は、複数ファイルにまたがる変更を同じノリでやったときです。いきなり実装させると途中から筋がずれ、修正の往復でコンテキストが埋まり、Claudeの忘れが増える。速いはずが、かえって遅くなる。

ここを抜ける最初の一手が、実装の前に計画を作らせることです。

# 1. plan modeで先に地図を描いてから歩き出す

複雑で不確実な変更ほど、いきなり手を動かさせてはいけません。

まず`plan mode`に入れて、コードベースの調査と実装計画だけを作らせます。計画にこちらがOKを出してから、実装を1ショットで走らせる。公式ベストプラクティスが、複数ファイルにまたがる変更で推奨している進め方です。

ここでひと工夫。出てきた計画を、別のClaudeにそのまま渡してこう頼みます。「シニアエンジニアとして、この計画の穴をレビューして」。作る側とは違う新鮮な視点で読ませると、見落としや危ない前提が表に出ます。穴を潰してから実装に移れば、一発で当たる確率が上がります。

それでも迷走することはあります。そのときは早く止めるのが正解です。`Esc`で中断、`Esc`を2回または`/rewind`で会話やコードの状態を巻き戻せます。同じ問題を2回直しても外すなら、いったん`/clear`して、学んだことを盛り込んだ新しいプロンプトで仕切り直す。これも公式の推奨です。

計画に全力を注ぐほど実装は軽くなる。地図を描く30分が、迷子の3時間を消します。

# 2. CLAUDE.mdとskillで「使うほど賢くなる」状態を仕込む

同じ規約を毎回説明し、同じ間違いを毎回直させていませんか。AIは前回の学びを勝手には覚えていません。

そこで、ミスを直すたびにこう頼みます。「二度と同じ間違いをしないよう、CLAUDE.mdを更新して」。CLAUDE.mdは、ビルドコマンド、テスト手順、コードの規約、全体の構造といった、毎回説明したくない情報を置く永続メモです。`/init`で雛形を生成でき、既存のものがあれば上書きではなく改善案を出してくれます。

書くコツは、コードを見れば分かることは書かないこと。短く、コードからは読み取れないことだけ。長いほどコンテキストを食い、肝心なときに薄まります。

さらに、1日に2回以上やる作業は手で繰り返さない。skillかcommandにまとめてgitにcommitし、全プロジェクトで再利用します。skillは手順書を`SKILL.md`としてパッケージ化する仕組みで、CLAUDE.mdと違って使うときだけ読み込まれる。だから長い参照資料を常時抱えずに済みます。重複コードの掃除なら`/techdebt`のような自前commandにしておくと一発です。

ひとつ正直な注意を。CLAUDE.mdは「絶対の設定」ではなく、あくまでコンテキストとして扱われます。書いても破られることはある。本当に守らせたい禁止や強制は、CLAUDE.mdではなくPreToolUse hoo
