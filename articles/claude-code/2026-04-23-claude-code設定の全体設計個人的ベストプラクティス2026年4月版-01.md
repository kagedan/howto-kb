Claude Codeを本格的に使い始めて半年ほど経ち、dotfilesを育てるように、Skills、hooks、CLAUDE.mdにはかなりの運用知識が蓄積されている感覚があります。この記事では、どのように現状設計しているかを書いてみます。あくまで2026年4月時点の個人的な設計指針です。

## プロジェクト構成

自分は ~/Develop/<会社名>/ のような事業プロジェクトを起点にClaude Codeを使っています。ここに全部を集約していました。<会社名>の部分は、会社名なり事業やプロジェクトなどが該当するでしょう。

```
~/Develop/<会社名>/
├── .claude/
│   ├── skills/          ← 数十個のスキル
│   ├── hooks/           ← ガード系スクリプト
│   └── settings.json    ← 権限設定・hooks定義
├── CLAUDE.md            ← 運用指示
└── ...
```

例えばここで、別リポジトリの `~/Develop/openclaw/` でClaude Codeを起動するとします。そうなると、Claude Codeのスキルディスカバリは、起動ディレクトリの `.claude/skills/` と `~/.claude/skills/` しか見ないので、スキルが1つも出てこない。親ディレクトリの `.claude/skills/` は継承されないためです。

## Claude Codeの継承モデルは非対称

ここが今回の記事の核心にはなりますが、Claude Codeの設定ファイルには、継承されるものとされないものがあります。

| 設定 | 親ディレクトリから継承？ |
| --- | --- |
| CLAUDE.md | される（ディレクトリツリーを遡って全連結） |
| .claude/skills/ | されない（起動ディレクトリ + ~/.claude/skills/ のみ） |
| .claude/settings.json | されない（起動ディレクトリのみ） |
| .claude/rules/ | されない（起動ディレクトリのみ） |

言うならば、CLAUDE.mdだけが「上に遡る」動きをします。skills、hooks、rulesは起動ディレクトリに閉じている。これを知らずに「親ディレクトリにスキルを置けば子プロジェクトでも使えるだろう」と思っていると、何も起きません。

この問題、dotfilesの管理と同じ構造をしています。

* `.zshrc` はホームディレクトリに1つ。どこでも効く
* `.eslintrc` はプロジェクトごと。必要なリポにだけ置く
* `.gitconfig` はグローバル設定とリポ設定を使い分ける

Claude Codeのスキルも同じです。「どこでも使いたいもの」と「このプロジェクトでだけ使うもの」を分ける必要があります。そのため、数十あるスキルを眺めて、分類軸を考えました。最終的にシンプルな問いに落ち着いています。

> このプロジェクトを使わなくなっても使うか？

* Yes → パーソナルスキル（自分に紐づく）
* No → PJT・事業固有スキル（その事業に紐づく）

もう1つ、「このリポジトリ以外で使うか？」がNoなら、リポ固有スキル。

これで3層になります。

| 分類 | 判定 | 配置先 |
| --- | --- | --- |
| パーソナル | 事業を問わず使う | `~/.claude/skills/` |
| プロジェクト・事業固有 | 特定事業のAPI/データに依存 | プロジェクトの `.claude/skills/` |
| リポ固有 | 特定コードベースでだけ意味がある | そのリポの `.claude/skills/` |

具体例:

```
パーソナル（~/.claude/skills/）:
  shared-logo-generator        ← SVGロゴ生成。どこでも使う
  shared-design-reviewer       ← デザイン批評。プロジェクト不問
  shared-blog-composer         ← ブログ執筆ワークフロー

事業固有（<会社名>/.claude/skills/）:
  shared-fetch-shopify-kpi     ← Shopify APIトークンが必要
  shared-weekly-kpi-report     ← GA4 + Sheets連携
  shared-morning-routine       ← タスク管理ツール + カレンダー

リポ固有（openclaw/.claude/skills/）:
  shared-openclaw-loop         ← openclawプロジェクト固有の自律ループ
```

## 正本をgit管理する

パーソナルスキルの置き場所は `~/.claude/skills/` です。ここに置けば、どのディレクトリからClaude Codeを起動してもスキルが使えます。

ただし `~/.claude/` 自体はgit管理されていません。スキルを直接ここに置くと、バックアップも履歴もない。dotfilesを直接ホームディレクトリで編集するのと同じ危うさがあります。

なので、正本は別のgitリポに置いて、symlinkで配るようにしています。

```
# 正本: git管理されたリポジトリ
~/Develop/ai-tooling/
├── skills/
│   ├── shared-logo-generator/
│   ├── shared-design-reviewer/
│   └── ... (43スキル)
└── hooks/
    ├── block-x-webfetch.sh
    ├── claude-md-size-check.sh
    └── commit-review-opt-in.sh

# symlinkを貼る
ln -s ~/Develop/ai-tooling/skills/shared-logo-generator ~/.claude/skills/shared-logo-generator
```

dotfilesリポから `~/.zshrc` にsymlinkを貼るのと全く同じパターンです。

## hooksも同じ原則で分離する

hooksもスキルと同じ問題があります。`~/.claude/settings.json` に定義したhooksは全プロジェクトで発火します。プロジェクトの `settings.json` に定義したhooksはそのプロジェクトでだけ発火する。

全プロジェクトで効いてほしいhooks（`~/.claude/settings.json`）:

例：

* X (Twitter)のURLをWebFetchしようとしたらブロック → 専用スキルを使わせる
* CLAUDE.mdのサイズチェック → 肥大化防止
* VOICEVOX音声通知 → タスク完了時に読み上げ

特定プロジェクトでだけ効いてほしいhooks（プロジェクトの `settings.json`）:

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-repository.sh",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

ここで1つ判断に迷ったのが `commit review`（コミット前のセキュリティチェック）です。全プロジェクトで効いてほしい気もするけど、実験リポでいちいちレビューが走ると邪魔になる。結局、opt-in方式にしました。hooks自体は `~/.claude/settings.json` に定義するけど、リポのルートにマーカーファイルがある場合だけ発火するようにしてみています。

```
#!/usr/bin/env bash
# commit-review-opt-in.sh
set -euo pipefail

REPO_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || echo "")
if [ -z "$REPO_ROOT" ] || [ ! -f "$REPO_ROOT/.claude-enable-commit-review" ]; then
  exit 0  # マーカーがなければスキップ
fi

# レビュー処理に進む...
```

本番リポには `.claude-enable-commit-review` を置いておく。実験リポには置かない。hooksのスコープを手動で制御できます。

## CLAUDE.mdは自然に階層化できる

CLAUDE.mdは親ディレクトリを遡って連結されます。これを活かして、指示も3層にしています。

```
~/.claude/CLAUDE.md                    → 個人の好み（出力スタイル、VOICEVOX設定）
~/Develop/CLAUDE.md                    → ワークスペース共通（安全ルール）
~/Develop/<会社名>/CLAUDE.md     → 事業固有（タスク管理サービス連携、保護リポ）
```

`~/Develop/CLAUDE.md` はboundary guardとして薄く書いています。実際の中身はこれだけです:

```
# Develop Workspace

~/Develop 配下は複数の独立リポジトリの集合。

- まず対象リポジトリを確認すること
- ルートを横断して勝手に変更しない
- secrets を workspace 直下に置かない
- 個別事業のルールは各プロジェクトの CLAUDE.md を優先する
```

分厚く書くと、どのプロジェクトから起動しても読まれてしまうので、必要最小限にしています。

## CLAUDE.mdに何を書くか

ここが意外と悩むポイントだと思います。最近は、7tsunoさんの「CLAUDE.mdに本当は何を書くべきなのか」という記事を読んで、自分の運用も見直しました。

<https://zenn.dev/cureapp/articles/65b9a99d22ce2b>

大事な前提として、CLAUDE.mdはSystem Promptではなく、User Messageとして注入されます。つまり、セッションが長くなるとコンテキストウィンドウの「古い方」に押しやられて、効き目が弱まっていく。これを踏まえると、CLAUDE.mdの役割がはっきりするのかなと思いました。

> CLAUDE.mdは「セッション開始時の作業を助ける情報」を置く場所

たとえばこういうものです:

* プロジェクトの構造と所在（「正本は repos/hoge-pdm/ にある」等）
* ツール指定（「Google系は gog コマンドを使う」等）
* タスク実行フロー（「まずタスク管理ツールでタスク確認 → 実行 → 完了」）
* 禁止事項（「保護リポジトリに書き込まない」）

一方で、「テストを書け」「命名規則を守れ」「型定義を省略するな」のような繰り返し守らせたいルールは、CLAUDE.mdに書いても後半で忘れられるリスクがあります。

こういうルールは `.claude/rules/` に置く方が安定します。rulesはツール使用時に条件付きで注入されるので、セッションの長さに影響されにくい。しかも `paths` 指定でファイルパターンごとに出し分けできます:

```
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
---

- TypeScriptでは `any` を避け、適切な型を定義すること
- 関数の引数と戻り値には型注釈を付けること
- interfaceよりtypeを優先すること（unionが使えるため）
```

`.ts` / `.tsx` を触るときだけこのルールが注入されます。Pythonファイルを編集しているときには出てきません。Progressive Disclosureの考え方で、必要なときに必要な指示だけが効きます。

自分の `~/Develop/CLAUDE.md` が4行しかないのはこの理由です。CLAUDE.mdを薄く保って、ルールはrulesに、スキルはskillsに分散させる。CLAUDE.md一枚に全部書いて肥大化させるより、Claude Codeが用意してくれた分散先を使う方が、長いセッションでも指示が効き続けます。

## syncスクリプトの役割を変える

以前はすべてのスキルをsyncスクリプトで各リポに配布していました。移行後は、パーソナルスキルを除外して事業固有スキルだけを配布するようにしています。パーソナルスキルは `~/.claude/skills/` のsymlinkで常に使えるので、syncの責務は事業固有の配布だけになりました。

結局、「どのディレクトリから起動するか」が設計上大事だとわかりました。

| 作業 | 起動ディレクトリ | 使えるもの |
| --- | --- | --- |
| メイン事業の日常業務 | `~/Develop/<会社名>` | 事業固有CLAUDE.md + Memory + hooks + 全スキル |
| openclaw開発 | `~/Develop/openclaw` | パーソナルスキル（`~/.claude/skills/`経由） |
| 複数リポを横断する作業 | `~/Develop` | boundary guard CLAUDE.md + パーソナルスキル |

「全部を1つの場所から操作する」のではなく、「正しい場所から起動すれば、必要なものが揃う」状態を作ります。
