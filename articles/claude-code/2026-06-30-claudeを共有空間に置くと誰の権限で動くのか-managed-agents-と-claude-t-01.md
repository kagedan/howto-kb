## はじめに

こんにちは。FLINTERSの河内です。

社内で開いているAI活用の勉強会で、Anthropic から相次いで出てきた2つの機能、**Claude Managed Agents** と **Claude Tag** をハンズオンで触ってみました。今回、2026-06-29 のこの会では発表担当の斉藤さんに画面を共有してもらいながら、その場で設定をいじって挙動を確かめていく形で進めました。

斉藤さんがこの2つを選んでくれたのは、どちらも「Claude の環境が手元にない人にエージェントをどう届けるか」という共通のテーマを持っているからでした。私がとくに引き込まれたのは、そこから一歩進んだところです。AIエージェントを作ってグループ全体に配ろうとすると、機能の派手さよりも先に、地味だけれど避けて通れない問いにぶつかります。**共有された場所でAIが動くとき、それは誰の権限で動いているのか**。今回のハンズオンは、まさにこの認可とアイデンティティの設計を、手を動かしながら確かめる時間になりました。

本記事は勉強会の内容を AI を活用しながらまとめています。

## 今日のお題 — 「Claude が手元にない人」にどう届けるか

斉藤さんの導入はこうでした。

> 両方とも、いわゆる Claude の環境が手元にない人たちがエージェントを呼び出して使える機能になっているので、今後グループの生成AIを配るときに参考になるかなと思います。

Messages API がモデルに直接プロンプトを投げるパターンなのに対して、Managed Agents は「エージェントハーネス」として提供される、という位置づけです。スキルやフックも含めた一式をセットアップして動かせる実行基盤、というイメージですね。

一方の Claude Tag は、Slack 上で @Claude をタグ付けして会話に参加させる機能です。出口は違いますが、根っこにある「個人の Claude アカウントを前提にしない配り方」という発想は共通していました。

## Managed Agents を触る — 自然言語で要約エージェントを作る

まず Managed Agents です。公式ドキュメントの定義はシンプルで、「マネージドインフラ上で動く、事前構築済みで設定可能なエージェントハーネス」。自前でエージェントループ・ツール実行・ランタイムを組む代わりに、Claude がファイル読み取り・コマンド実行・ウェブ閲覧・コード実行を安全に行える完全マネージドな環境が手に入ります。長時間・非同期のタスク向けです。

<https://platform.claude.com/docs/en/managed-agents/overview>

ドキュメントは4つのコアコンセプトで整理されています。

| 概念 | 内容 |
| --- | --- |
| Agent | モデル、システムプロンプト、ツール、MCP サーバー、スキルの組み合わせ |
| Environment | セッションが動く場所の設定（Anthropic 管理のクラウドサンドボックス、または自前インフラ上のセルフホスト） |
| Session | 環境内で動くエージェントのインスタンス。特定のタスクを実行し出力を生成する |
| Events | アプリとエージェント間でやり取りされるメッセージ（ユーザーターン、ツール結果、ステータス更新） |

動作の流れも、エージェントを作る → 環境を作る → セッションを開始する → イベントを送ってレスポンスを SSE でストリーミングする → 途中で操舵・中断する、という5ステップで素直です。Session はサーバー側に永続化される append-only のイベントログで、コンテナが落ちても残る、という設計になっています。

今回は Claude Console から触りました。これは API キーを発行する開発者向けコンソールに近いもので、Claude のサブスクリプション側でかける組織統制設定（managed settings など）はここには効きません。トークン消費がそのまま課金になるので、上限を決めて少額デポジットした状態で試しました。

もともと用意していた題材は、Claude アカウントを持たない人に向けて OSS スキルパック GBrain（Garry Tan / Y Combinator CEO が公開）を配布する、というものでした。ただ、すぐに後述する認証の議論へ深入りしたので、その場ではよりシンプルな Slack 要約エージェントを自然言語で作って試しています。

<https://github.com/garrytan/gbrain>

クイックスタートには「何を作りたいですか？」という入力欄があり、自然言語で指示を出すとエージェント設定が生成されます。試しに「Slack の公開チャンネルから昨日のダイジェストをまとめるものを作って」と頼んでみたところ、ほぼ一発でエージェントの YAML が書き上がりました。モデルに Sonnet、ツールに Slack の MCP サーバーが定義され、エージェントツールセットが紐づく、という具合です。

そこから Environment を作り（パッケージやネットワークアクセスを持つクラウドコンテナのテンプレート）、Session を開始します。Slack の MCP サーバーを使うには認証情報を入れる Vault が必要だ、という案内が出てきました。

## ぶつかった「認証の壁」

ここで議論が一気に深まりました。私が最初に気になったのは、認証情報の扱いです。

> ここで斉藤さんが自分の Slack アカウントで認証を通すと、このワークスペースにアクセスできる。ほかのメンバーや私も、その認証情報を使って Slack にアクセスできてしまうのか?

公式の Vault の仕様を読むと、トークンの「中身」は確かに守られていました。Vault に登録した認証情報は書き込み専用フィールドとして扱われ、API レスポンスにも返らない。Claude の生成コードが走るサンドボックスからもトークンには到達できない設計です。実際に接続してみても、認証情報の存在は分かるけれど中身は見えませんでした。メッセージを送ると前日に Slack でやり取りしていた内容まで拾ってきて、ちゃんと要約が返ってきます。動くことは動きました。

<https://platform.claude.com/docs/en/managed-agents/vaults>

問題はその先、**配布する段になってからの体験**でした。調べていくと、話はそう単純ではありません。

まず、エンドユーザーごとの Vault は自動では作られません。公式ドキュメントの手順は、開発者が API で `display_name: "Alice"` のようにユーザーぶんの Vault を作り、そこに各ユーザーの OAuth トークンを登録し、セッション作成時に `vault_ids` で参照する、というものです。つまり本当にユーザー単位で認証を分けたければ、自分で OAuth フローとバックエンドを組んでトークンを同期する必要がある。コンソールをポチポチ触るだけで実現できるものではありませんでした。

しかも公式は、Vault と認証情報は **ワークスペーススコープ**で「同じワークスペースの API キーを持つ人なら、誰でもセッション作成時に参照できる」と明記しています。トークンの中身は読めなくても、それを使うこと自体は他の人にも開かれている。私が会の中で「個人専用の Vault を作っても、他の人が使えない状態にできるわけではないのでは」と引っかかったのは、まさにこの仕様のことでした。

これがそのまま配布の難しさに直結します。各セッションの内容は管理者のモニタリング画面から見えてしまいますし、Vault がワークスペーススコープである以上、エンドユーザーに API を直接叩かせる構造は危なくて作れません。結局、間にフロントエンドを一枚噛ませて認証とアクセスを制御する必要が出てきます。

実際にエージェントを配るときは、むしろユーザー別の認証こそ欲しい場面が多い。それなのに今の Managed Agents は、ユーザーごとの認証を気軽に組み込んで配布する構造にはなっていない、という所感に落ち着きました。共通のワークスペース認証で済むものなら使いやすいけれど、という温度感です。

そして「ユーザーごとの認証をどう扱うか」という宿題が、そのまま次のお題につながりました。

## Claude Tag と「エージェントアイデンティティ」

Claude Tag は、Claude と Slack を連携させる新しい方法です。任意のチャンネルで @Claude をタグ付けしてタスクを割り当てると、組織のツールと周囲の共有コンテキストを使って実際の作業を行います。チャンネルでのタグ付けのほか、Claude との DM、AI アシスタントパネルからの呼び出しという3つの入り口があります。

ここで Anthropic が打ち出しているのが **エージェントアイデンティティ** という設計思想でした。発表の中で斉藤さんが説明してくれた要点はこうです。

> 今 Claude Code とかでやっているエージェントは、個人の認証情報を扱って認証していますよね。自分の Gmail や Slack のアカウント、自分のクレデンシャルで作業している。それが今後弊害になりそうだから、AI に対してユーザー権限じゃなくて AI 権限を持たせましょう、という話です。

つまり、個人のユーザー認証情報を借りるのではなく、ワークスペースが管理するサービスアカウントとして Claude が動く。Slack では Claude アプリとして投稿し、GitHub は Claude GitHub App 経由で操作し、データウェアハウスは管理者がプロビジョンしたサービスアカウントで照会する、という形です。問いを「このユーザーは何ができるか」から「このエージェントはこの区画で何ができるか」へ転換する、という整理ですね。

設定画面には、Google カレンダーを連携させる際の注意書きとして「個人の Google カレンダーアカウントは使わず、Claude 専用のアカウントを作ることを推奨します」と出てきました。個人アカウントで認証してしまうと、@Claude でカレンダーを読み出した瞬間に個人の予定が全部読めてしまう。だから専用アカウントを割り当てる、という発想です。

ここで私は引っかかりました。

> エージェント用のアクセス権限が区画されているのはいいことだと思うんです。でも理想を言えば、私がカレンダーにアクセスするときは、私がアクセスできる範囲をさらに絞り込んだサブセットをエージェントに使ってほしい。今の話だと、カレンダー用の共有アカウントを作ってみんなで使い回しましょう、という話に聞こえる。

斉藤さんも「いわゆるサービスアカウントですよね」と。本当にこれがベストプラクティスなのかは、正直まだ腹落ちしていません。ただ、この設計には背景があります。

この設計の背景は、Anthropic がこの機能に合わせて公開したブログ「Agent identity: a new access model for autonomous, team-wide AI」に詳しく書かれています。

<https://claude.com/blog/agent-identity-access-model>

従来の「1ユーザー = 1アシスタント」モデルは、チームの共有空間で AI が複数人に操作される状況では機能しなくなる、という指摘です。中心にある問いはこうでした。

> But when more than one person is steering, whose permissions apply? There's no single choice of person that'd be right all of the time.
>
> （複数人が操縦しているとき、誰の権限が適用されるのか? 常に正しいと言える一人を選ぶことはできない。）

だからこそ、問いを「このユーザーは何ができるか」から「このエージェントはこの区画で何ができるか」へ移す。記事の言葉では *"Agent identity replaces the question 'what can this user do?' with 'what can this agent do in this compartment?'"* と表現されています。共有チャンネルが誰かの個人ドキュメントへの裏口になってしまうのを防ぐために、エージェント自身のアイデンティティを立てる。言われてみれば、共有空間で個人権限を借りる方が筋が悪い、というのは確かにその通りでした。

## アクセスバンドルと3層モデル

では権限はどう設計するのか。Claude Tag は、到達できる範囲を3つのレベルで設定します。各レベルは上位の権限とメモリを継承します。

* **組織全体**: Claude Tag がインストールされている全ての場所に適用される認証情報とリポジトリ
* **ワークスペース**: Slack ワークスペース内の全パブリックチャンネルに適用（組織全体を継承）
* **プライベートチャンネル**: ワークスペースが付与済みのものに加えて、追加の認証情報やリポジトリ。機密接続を小さなグループに閉じ込める

この単位が **アクセスバンドル** と呼ばれるものです。ハンズオンで分かったのは、アクセスバンドルが単なる認証情報の束ではない、ということでした。公式の定義では「認証情報・リポジトリ権限・プラグイン・指示」のセットで、実際の画面でも、アクセスを許可する外部ドメインや、使えるプラグイン、標準の指示・コンテキストまでが一式で束ねられていました。これをチャンネルに紐づけて運用します。

リポジトリ単位でも設定できるので、各プロジェクトチャンネルに特定のリポジトリを紐づけておけば、そのスコープの中から「設定項目はこうでした」と返してもらう、といったレビューに近い使い方もできそうでした。MCP サーバーの指定や curl コマンドの許可もバンドルの中に入れられます。

ただ、運用のリアルとしては悩ましい点もありました。管理者役のメンバーからは、こんな本音が出ました。

> 各チームにカスタムしてほしいんですけど、全部「こう変更してください」と僕のところに来たら対応しきれない。設定内容は柔軟なんですが、管理者がポチポチやるしかなくて、辛いです。

「このチャンネルにこのリポジトリを紐づけてください」という依頼が一人に集中すると持たない。柔軟だけれど、誰がどのバンドルを設定・許可できるかという委譲の仕組みがまだない、というのが現時点の手触りでした。

なお、Claude Tag はネットワークのアウトバウンドをかなり絞っている印象でした。許可リストにドメインを足さないと外に出られない。Slack には企業情報が集まるからこそ、情報流出をさせないために egress を絞っている、という整理なのだと思います。実際、未承認ホストへの通信はブロックされ、全アクションはサービスアカウント下でログに残る、という設計です。

ちなみに Claude Tag と Managed Agents は別物で、課金のクレジットも別枠です。Claude Tag はチャンネルでタグ付けすると組織に請求され、DM は各自の Claude アカウントに請求されます。

## で、どこで効くのか

ハンズオンを通じて、向き・不向きの線引きが見えてきました。

**相性が良さそうなのは、共有リソースを扱う定型業務**です。あるチャンネルで特定の業務を頼むとき、毎回その都度エンドユーザーに特定ファイルの権限を付与しなくていいケース。社内ヘルプデスクや各種の申請プロセスのように、利用者ごとのコンテキストの差分があまりないものは、サービスアカウント型ときれいに噛み合います。Slack のワークフローが大量にあって「どれを選べばいいか分からない」といった問い合わせの一次受けボットとしても良さそうです。実際に今は人が手で返している内容を考えると、使用量課金でもコスパが見合う可能性は十分あります。

**逆に向かないのが、パーソナルアシスタント的な使い方**です。スレッドで問い合わせが来たときに、そのスレッドに参加していない自分だけの情報まで参照して返信文を作ってほしい——それが本来の理想ですが、共有チャンネルでそれをやると、他の人が見てはいけない個人の情報がチャンネルに漏れてしまう。発表でも整理されたとおり、個人の情報を含めて返すなら、そもそも共有チャンネルでやる意義がない。そういうパーソナルな用途は、DM や個人の Claude アカウント側でやってください、というのが Claude Tag の設計思想なのだと理解しました。

もうひとつ物足りなかったのが、ヒューマンインザループの仕組みです。調べてみても、実行前に人間が承認を挟むステップは Claude Tag には組み込まれていませんでした。承認を入れたければカスタム指示のプロンプトで促すか、その層を自分で作り込むしかありません。Slack のワークフローの途中ステップだけを切り出して自然言語で対応させる、といった柔軟さも今のところなさそうです。

## まとめ — やりたい世界観は見えた

触り終えての率直な感想は、「やりたい世界観は理解できた。でも痒いところにはまだ手が届かない」でした。

Managed Agents は認証情報の保護こそ堅牢ですが、エンドユーザーごとの認証を組み込んで気軽に配る、という用途にはまだ向いていません。Claude Tag のエージェントアイデンティティは、「共有空間で誰の権限を使うか」という答えのない問いに対する、Anthropic なりの一つの解です。個人権限のサブセットを絞り込んでほしいという私の理想とはまだ距離がありますが、共有チャンネルで個人権限を借りる方がむしろ筋が悪い、という指摘には納得感がありました。名前が「Claude Tag」なのも、チャンネルに対して「ここはこの権限」というタグ付けを寄せていく場所、と考えると腑に落ちます。

グループ全体に生成AIを配る立場から見ると、結局のところ鍵を握るのは、機能そのものよりも「どこを権限の寄せポイントにするか」でした。Slack × Google × Notion のような構成だと権限統制がばらけて大変になりがちなところに、Claude Tag は寄せポイントを作りにいっている。チームプランの中に内包してくれれば一番ありがたいのですが、まずは社内の申請やバックオフィスの一次受けボットあたりから、現実的な使いどころを探っていきたいと思います。
