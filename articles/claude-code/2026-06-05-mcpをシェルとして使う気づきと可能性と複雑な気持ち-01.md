## 発端

MacのClaude Desktop（Cowork）からWindowsのWSL2をMCP経由で操作できるようにした。  
詳細な手順は[前記事](https://zenn.dev/keiichi_okamoto/articles/xx_260531_mac-win-mcp) にまとめてある。

この設定をしてから気づいたことが、手順そのものより面白かった。

---

## Claude CodeはシェルとしてMCPに繋がっている

「WSLでDockerを再起動して」と指示したとき、内部ではこうなっている：

各MCP呼び出しで起動されるClaude Codeは、コマンドを実行して結果を返すだけの存在だ。お互いを知らないし、文脈も持っていない。指示を分解し、結果をまとめ、次の操作を判断し、最終的に報告しているのはCowork側のClaudeだけだ。

つまりこの構成では、**Claude CodeはAIとして判断していない。シェルとして動いている。**

この構造を理解すると「mcpsh」という概念が見えてくる。軽量なMCPサーバーがシェルとして動き、上位のAIが指揮者として判断する——今回のClaude Code（mcp serve）はその先駆け的な形だ。

---

## 同じMac内でもDockerコンテナにアクセスできる

WSLへのMCP接続を設定した後に気づいたことがある。同じMac内でもMCPを使うと、Dockerコンテナの中にさえアクセスできる。

「DBの中身を確認して問題がないか見て」という指示で：

```
docker compose exec db psql -U appuser -d appdb -c "SELECT * FROM users;"
docker compose exec db psql -U appuser -d appdb -c "SELECT * FROM devices;"
docker compose exec db psql -U appuser -d appdb -c "SELECT * FROM subscriptions;"
```

これらを実行して結果を見て「subscriptionsに1件NULLが残っているのはSNSのEndpoint作成が失敗した痕跡」のような評価まで返してくれる。

---

## 気をつけるべきこと

便利な反面、注意が必要な点もある。

**CLAUDE.mdのルールが効かない場合がある**  
パターンA（HOMEで起動）の場合、読み込まれるのは`HOME/CLAUDE.md`だけ。プロジェクト固有の`CLAUDE.md`は読まれない。普通の会話の流れでMCPへのリクエストが走ってしまう可能性もある。

対策として`$HOME/CLAUDE.md`に書いておくと確認を求めるようになる：

```
## MCP操作の確認
mcpでの作業と思われる場合には、どのmcpに対してリクエストするか必ず確認すること。
```

**セキュリティ上の境界が曖昧になる**  
パターンA（HOME起動）はHOME以下全体にアクセス権がある。本番の認証情報（`.env`など）が含まれるディレクトリにもアクセスできてしまう。プロジェクトごとに個別MCPを登録するパターンBの方がアクセス範囲を限定できる。

**Anthropicのサーバーを経由する**  
「何を実行するか」の判断はAnthropicのサーバーを通る。実際のコマンド実行はMac→WSL間のローカルで完結するが、`.env`の中身などをClaudeに読ませるときは注意が必要。

## 使い分けの現実

この設定をしてから、Windows Appを使ってWSL2に入る機会が大幅に減った。

| 方法 | 向いている場面 |
| --- | --- |
| MCP経由（Cowork） | 日本語で指示してClaudeに任せたい、Mac側の作業と組み合わせたい |
| SSHで直接ログイン | 複雑なコマンドを自分で打ちたい、インタラクティブに操作したい |
| Windows App | Windowsのデスクトップ環境そのものが必要な場面 |

副産物としてMacのターミナルからSSHでWSL2に直接入れるようにもなった。Windows Appのリモートデスクトップと比べてキーボードの挙動がMacのままで使え、日本語入力の切り替えも楽で、画面遅延もない。

## 複雑な気持ち

便利なのは間違いない。「WSLでDockerを再起動して」の一言で動く。PostgreSQLの中身を確認して評価もしてくれる。Dockerコマンドのオプションを覚えなくてもいい。

でも、それが問題でもある。

Dockerコマンドを自分で叩いて、エラーに当たって、ログを読んで、直す——という体験が薄れていく。「できる」と「分かっている」は別の話で、AIに任せることで前者だけが残って後者が育たない可能性がある。

「Dockerで他に何ができるか」を知るには、ある程度自分で触った体験が必要だ。引き出しがなければAIへの指示の質が下がるし、AIの提案が正しいかどうかの判断もできない。

今のスタンスとしては「概念を理解した上で任せる」を意識したい。コマンドの細かいオプションを暗記する必要はないが、「何をやっているか」は把握しておく。AIが「これが正解です」と言っても、それが自分のプロジェクトに合っているかどうかは経験からしか判断できない。

## ビルドエラーの確認・修正まで完結させる

WSL側のMCPが繋がっていれば、Dockerのビルドエラーを見ながら修正して再ビルドまで一気にできる。

```
1. WSL側で docker compose up --build を実行（MCP経由）
2. ビルドエラーのログをCoworkで確認
3. Mac側のファイルを修正（Read/Editツール）
4. git push → WSL側でpull
5. 再ビルド
```

ただしMacとWSLでファイルが別管理なので、Macで修正→push→WSLでpullという手間が入る。

### Mac側のClaude CodeもMCPで繋げば手間がなくなる

Mac側のClaude Codeも同様にMCPで繋げば、Mac内で完結できる。

`claude_desktop_config.json`に以下を追加するだけ：

```
"mac_claude_code": {
  "command": "/Users/okamoto/.local/bin/claude",
  "args": ["mcp", "serve"]
}
```

こうするとMacのDockerビルドエラーもCoworkから直接確認でき、ファイル修正→再ビルドまでgit push/pullなしで完結する。

「ビルドエラーを見ながら修正して最終成果物を出す」という作業をCoworkから一気通貫で任せられる。

## まとめ

MCPはシェルとして機能する。Coworkが指揮者として判断し、Claude Code（mcp serve）が実行エンジンとして動く。この構造を理解すると「何を任せられるか」「何を自分で判断すべきか」が見えてくる。

設定自体はシンプルで、SSH鍵の準備と`claude_desktop_config.json`への数行の追記だけだ。一番ハマるのはパスフレーズなし専用鍵の作成で、それさえ分かれば動く。

便利なツールを持ったからこそ、どう使うかの判断は人間の仕事として残り続ける。
