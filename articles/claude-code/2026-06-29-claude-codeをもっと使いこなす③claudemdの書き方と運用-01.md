# Claude Codeをもっと使いこなす③｜CLAUDE.mdの書き方と運用

こんにちは、開発日和です。

Claude Codeを使ううえで、地味に大事なのが `CLAUDE.md` の存在です。これがあるかないかで、Claudeの「使えるやつ感」が大きく変わります。

今回は `CLAUDE.md` の役割・書き方・運用のコツを実体験ベースでまとめました。

---

## `CLAUDE.md` とは何か

`CLAUDE.md` は、**Claude Codeが毎回自動で読み込む設定ファイル**です。ここに書いた内容は、セッションをまたいでも常にClaudeのコンテキストに含まれます。

「毎回同じことを説明している」と感じたら、それは `CLAUDE.md` に書くべき内容です。

### 読み込まれる場所と優先順位

`CLAUDE.md` は複数の場所に置けます：

```
~/.claude/CLAUDE.md           ← ユーザーグローバル（全プロジェクトに適用）
/path/to/project/CLAUDE.md   ← プロジェクトルート（そのプロジェクトに適用）
/path/to/subdir/CLAUDE.md    ← サブディレクトリ（そのディレクトリ内に適用）
```

複数存在する場合は**すべて読み込まれ、より具体的なもの（サブディレクトリ）が優先**されます。

---

## グローバルCLAUDE.md ― 自分の「定数」を書く場所

`~/.claude/CLAUDE.md` には、プロジェクトをまたいで共通の情報を書きます。

```
# 基本方針
- 会話はすべて日本語で行う
- 複雑な説明はステップごとに日本語で区切って記述する
- サンプルコードには必ず日本語のコメントを添える
- 英語ドキュメントの説明が必要な場合は日本語訳も添える

# 私のプロフィール

## 役割
- システムエンジニア（SE案件メイン）

## 技術スタック

### バックエンド
- PHP（メイン言語）
- Laravel（主要フレームワーク）
- Livewire（フロントエンドとの連携）
- MySQL（主要DB）

### フロントエンド
- JavaScript

### インフラ・クラウド
- AWS（主要クラウド）
- Docker（コンテナ環境）

---

# Claude への共通指示

## 出力スタイル
- **必ず日本語で回答する**

## 回答の品質基準
- 推測・不確かな情報は「※推測:」と明記してから述べる
```

ポイントは「自分がエンジニアとしてどういう人間か」「どんな回答スタイルが好みか」を書くこと。毎回「日本語で答えて」「PHPが主な言語です」と説明しなくて済むようになります。

---

## プロジェクトCLAUDE.md ― Claudeへの「引き継ぎ書」

プロジェクトルートの `CLAUDE.md` には、そのプロジェクト固有の情報を書きます。新メンバーへの引き継ぎ書をイメージするとわかりやすいです。

### 書くべき内容

**1. よく使うコマンド**

```
## よく使うコマンド

# 開発サーバー起動
php artisan serve

# テスト実行
php artisan test
php artisan test --filter=UserTest

# マイグレーション
php artisan migrate
php artisan migrate:fresh --seed
```

**2. アーキテクチャの概要**

```
## アーキテクチャ

- PHP 8.2 / Laravel 11
- MySQL 8.0
- フロントエンドはBladeテンプレート + Alpine.js
- 認証はLaravel Breeze
- キャッシュはRedis

## ディレクトリ構成の補足

- `app/Services/` ← ビジネスロジックはここに集約
- `app/Repositories/` ← DB操作はリポジトリパターンで分離
- `app/Http/Controllers/` ← コントローラーは薄く保つ方針
```

**3. コーディング規約・注意事項**

```
## コーディング規約

- 変数名・関数名は英語、コメントは日本語
- マジックナンバーは定数化する（`const` で定義する）

## 注意事項

- マイグレーションは `artisan migrate:fresh` を本番で実行しないこと
```

---

## 書かなくていいこと

`CLAUDE.md` に書くと逆効果なこともあります：

* **コードの内容そのもの**（Claudeはコードを読めるので書かなくていい）
* **当たり前のこと**（「バグを作らないように」「セキュリティに気をつけて」など）
* **長すぎる説明**（読み込みに使われるコンテキストが増えてコストが上がる）

「コードを見れば分かること」は書かない。「コードを見ても分からない背景・方針・例外」だけを書くのがコツです。

---

## /init を使って自動生成する

ゼロから書くのが面倒なときは、まず `/init` で自動生成してもらうのが早いです。

ディレクトリ構成・設定ファイル・READMEなどを読んで草案を作ってくれます。出力された内容を見ながら「ここはいらない」「ここは追記したい」と編集するのが効率的です。

---

## 運用のコツ

`CLAUDE.md` は一度作って終わりではなく、育てていくものです。

**更新のタイミング**

* 新しい技術・ライブラリを導入したとき
* チームの規約が変わったとき
* 「何度も同じことをClaudeに説明している」と気づいたとき

**定期的な見直し**

* 古くなった情報（使わなくなったコマンド、変わったディレクトリ構成）は削除する
* 肥大化してきたら整理する。長いほどいいわけではない

---

## まとめ

| 場所 | 何を書くか |
| --- | --- |
| `~/.claude/CLAUDE.md` | 自分のプロフィール・回答スタイル・技術スタック |
| `プロジェクトルート/CLAUDE.md` | よく使うコマンド・アーキテクチャ・規約・注意事項 |

`CLAUDE.md` を整備すると「毎回同じ説明をする手間」が省け、Claudeへの指示が短くなっていきます。地味ですが効果が大きい設定です。ぜひ試してみてください。
