IT未経験でアプリ開発を進めているKKです。少しだけ自己紹介させてください。

大学：文系  
職歴：営業歴10年（個人、法人など）開発経験なし  
資格：ITパスポート

金融や人材業界で営業をしていた経験から、IT未経験でもITに興味を持っていました。そこに、生成AIが一気に身近なものになりました。

プログラミングとか難しそう、、、と長く思っていましたが、生成AIのおかげでその壁を越えられる。期待感を持ってAIを使っていました。最近ではClaude CodeやCodexを使いながら、アプリ開発に取り組んでいます。

わからないことだらけで日々色々な実験をしながら進めています。こんな経歴だからこそ、一般的なIT企業のセオリーや組織体制、管理方法に縛られることなく、AIネイティブな体制を作ることができたと思います。

人間にとってわかりやすい管理方法は、本当にAIにとってもわかりやすいのか。AIを使って開発を進めるなら、AIの認知構造に合わせて運用を作った方がいいのではないか。

この記事では、複数のアプリ・事業アイデアをAIと進める中で考えるようになった「AIネイティブな事業開発」について書きます。よくある「AIで簡単に稼ぐ方法」ではありません。AIに任せるために、人間側がどう文脈を整え、判断の土台を作るべきか、という実践記録です。

その中で試したこと、うまくいったこと、まだ難しいことをシェアします。個人開発をやっている・これから個人開発をやりたい・会社の体制に違和感を持っている。こんな方にぜひ読んでほしい内容になっています。

## 人間向けの資料に感じた違和感

企画系の部署と一緒に仕事をしていたとき、事業責任者向けにPowerPointを作ることがありました。チャットツールを使ったサービスを営業現場に取り入れるためのプロジェクトです。

複数のサービスを比較して、どのサービスを採用するかを説明する資料を作成しました。その際に、使わない予定のサービスも比較のために問い合わせをし、商談も行いました。工数が余っていないのに社内で開発する選択肢も検討しました。

本命のサービスとそれらを比較する表を作りプレゼンしました。そして本命のサービスが採用されました。

もちろん比較自体は必要です。ただ、実際には「これがよさそう」というサービス候補が先にあり、それを通すために他サービスとの比較表を作っているように感じる場面もありました。いや、実際そうなんでしょう。

意思決定のための資料というより、すでにある結論に「イエス」と言ってもらうための資料。30分の会議のために、事前検討・商談で2週間。資料作成・チェックで1週間。3週間の時間をかけてサービスを決めました。

そこから自社向けにプロダクトをカスタマイズしてもらい、実際に使い始めたのは3か月後。合計で4か月かけてやっとサービスを使えるようになったんです。

当時はそれに違和感を持っていませんでした。そういうもんかと。でも今だから言えます。AIを前提にすると、もっと減らせる作業が多かった。

一般的な会社や組織では、PowerPoint、Excelを作成し、会議や商談に臨みます。でもそれらは、人間の認知限界を補うために作られています。

人間は一度に大量の情報を処理できません。記憶力にも限界があります。複数案件を同時に進めるのも得意ではありません。だから、資料を作り、会議をし、議事録を残し、意思決定をします。

しかし、AIを中心に仕事を進めようとすると、この前提が変わります。AIにとってはコンサルが作るようなビジュアル化されたPowerPointはわかりやすい資料ではありません。Excelの表やグラフが理解しやすいわけではありません。

これまで当たり前に行ってきた仕事。そこには「人間にとって理解しやすい」という前提があったんです。AI時代の仕事なのに、私たちはまだ人間向けの形式にAIを合わせようとしていました。

その前提が変わります。「AIにとって理解しやすい」という前提を持ってすべてを設計することが必要になりました。この考え方こそが、AIネイティブだと思います。

PowerPointをAIに読ませる。ExcelをAIに解釈させる。会議で決まったことを後からAIに要約させる。もちろん便利です。ただ、それは人間ネイティブな仕事をAIで補助している状態です。

私が試したかったのは、その逆でした。  
最初からAIが読みやすく、判断しやすい形で事業開発を設計したらどうなるのか。

## AIネイティブな事業開発とは何か

私が考えるAIネイティブな事業開発とは、人間の資料作成文化をAIに持ち込むことではありません。AIが理解しやすい単位で、事業の文脈を整理することです。

| 人間ネイティブ | AIネイティブ |
| --- | --- |
| PowerPoint | Markdown |
| Excel管理 | 構造化テキスト |
| 会議 | プロンプト |
| 議事録 | メモリファイル |
| 組織図・肩書き | コンテキスト・役割定義 |

この対比は、単なるツールの違いではありません。仕事の前提が違います。

人間に見せる資料ではなく、AIが次の行動を判断するための文脈を作る。会議で共有するのではなく、プロンプトとして明示する。肩書きではなく、そのAIが何を知っていて、何を判断してよいかで役割を決める。

AIにとって重要なのは、「何を知っているか」です。だから、私は事業ごとに情報をMarkdownで整理し、プロジェクト番号をつけ、ステージを定義し、現在地と次のアクションを明示するようにしました。

たとえば、事業の状態を「なんとなく進んでいる」「まだ微妙」「そろそろ出せそう」といった曖昧な言葉で扱うと、AIは判断しにくくなります。そこで、S0、S1、S2、S3、S4、S5、S6、S7、S8、S9のようにステージを共通言語にしました。

今どの段階にいるのか。次に進むには何が足りないのか。人間が判断すべきことは何か。AIが進めてよいことは何か。毎回ゼロから説明しなくても、同じ言葉で扱えるようにする。

これが、私にとってのAIネイティブな運用の第一歩でした。

## PM AIの登場

最初は作りたいアプリやサービスごとに事業部チャットという形でチャットを作成し相談していました。次に取り組んだことは、PMを行うAIの作成です。

事業（アプリやサービス）を複数作っていく中で、私が全事業の最新状況をキャッチアップし、指示を出すのが難しくなってきました。事業が5を超えた際に感じました。そこで事業横断型で最新状況を把握し、指示を出せるAIが必要だと考えました。そこで誕生したのがPM AIです。

PM AIに最初に任せたのは、各事業部の最新状況をまとめてもらうことでした。複数の事業やアプリを同時に進めていると、まず「今どれがどうなっているのか」を思い出すだけで時間がかかります。

そこで進捗報告フォーマットを作成し、それを各事業AIから提出してもらうようにしました。提出された内容をPM AIに集約するようにしました。毎朝フォーマットを送ることで進捗状況が最新化されていきます。

報告フォーマットには、たとえば次のような項目を入れています。

* 現在の完成度
* 前回から進んだこと
* 現在のボトルネック
* PMに判断してほしいこと
* 次にやるべきこと  
  これは、人間向けのきれいな資料ではありません。AIが比較しやすい入力形式を作るためのものです。

PM AIは、それぞれの事業の状態を見たうえで、今日どの事業部のどのタスクから取り組むべきかを表で出してくれます。たとえば、こんなイメージです。

| 優先順位 | 事業部 | 今日やること |
| --- | --- | --- |
| #1 | PRJ-A | ストア掲載文の初稿を作り、公開可否をPMが判断する |
| #2 | PRJ-B | 画面遷移の認識ズレがないか確認し、修正指示を出す |
| #3 | PRJ-C | 進捗報告が未提出のため、事業部AIに最新状況を再確認する |

もちろん、AIが出した順番をそのまま実行するわけではありません。私から色々要望を出します。  
「今日は2番目の事業を進めたい」  
「このタスクは登録作業が必要で、今日は少し重いから別のタスクから進めたい」  
「この事業部から報告が来ていないが大丈夫か」  
そうした判断は人間が行います。

AIネイティブな運用とは、人間の判断をなくすことではありません。人間が判断する前の整理コストを下げることです。

## 作業者から意思決定者へ

Claude CodeやCodexを使う前のAI活用は、まだ私が作業者でした。

AIチャットに「こういうコードを出してほしい」と頼む。出てきたコードを自分で貼り付ける。動かしてみる。エラーが出る。エラー内容をまたチャットに戻す。AIは便利でしたが、人間側が手を動かさないと進まない状態でした。

確かにプログラミングの知識がなくても教えてくれる。でも、結局人間の手間は残るな、と感じていました。

Claude CodeやCodexを使うようになって驚いたのは、AIがかなりの部分を勝手に進めてくれることです。指示を出せば、勝手にコードを書き換え、エラーを修正し、状況を確認してくれる。私は出てきたアウトプットの確認と意思決定だけすればよくなりました。

結果としてこれまでと仕事の重心が変わったと思います。以前は作業者でした。今は、意思決定者になりました。

PM AIや事業部AIからは、「これはどうしますか」「この方針で進めてよいですか」「このコストを払いますか」といった問いが返ってきます。そこで必要になるのは、作業ではなく判断です。

これは、まだAIだけでは正確に判断しきれない部分です。例えばAIは、ある月額サービスを「大した金額ではない」と言ってきました。しかし私の感覚では「高すぎる」と感じていました。

そういったずれが出てきます。すべてをAIに任せていたら、不要な有料プランを選んでしまう可能性もまだあるんです。

コスト感覚、時間感覚、リスク許容度、今日どの事業を進めたいかという気分。こうしたものは、人間が判断し、AIに伝える必要があります。

AIネイティブな事業開発とは、人間が不要になることではありません。  
人間がすべてを抱え込まず、AIが動きやすい文脈を整え、人間は判断すべきところに集中することだと思っています。

## まだ難しいこと

上ではかなり偉そうなことを言っていますが、うまくいったことばかりではありません。

まず、情報はすぐ古くなります。AIは便利ですが、最新の事業状況を勝手に覚え続けてくれるわけではありません。メモリファイルや進捗レポートを更新しないと、古い前提で判断してしまいます。

次に、AIは放っておくと一般論に戻ります。  
「新規事業では顧客理解が重要です」  
「MVPを作って検証しましょう」  
正しいのですが、杓子定規的な考え方では意味がありません。だからこそ、抽象論ではなく、現在ステージ、制約、ボトルネック、完了条件を渡し、情報を更新し続ける必要があります。

さらに、複数AIを使うと認識ズレも起きます。PM AIと事業部AIに伝えている内容がズレると、PM AIが見ている仕様と、事業部AIが理解している仕様が違う状態になります。実際に、同じ事業部内で似たサイトを複数作っていたとき、サイトAとサイトBの認識が混ざり、トップページと導線が食い違うことがありました。

これは第3回で詳しく書く予定ですが、この失敗から学んだのは、AIを増やすことによるリスクも想定し、対応する必要があるということでした。

AIを増やすなら、文脈の同期が必要です。PM AIは意思決定と指示、事業部AIは実行。その役割を明確にしないと、AI同士の認識がズレます。

## 次回以降で書くこと

今回の記事では、AIネイティブな事業開発という考え方を書きました。次回は、実際に私が使っている自分専用ポータル「MyMonitor」について書く予定です。

もともとはNotion、Google Keep、OneNoteなどで状況管理をしようとしていました。しかし、最新状況を更新し続けるのが面倒だったり、一画面で見づらかったり、画面遷移が多かったりして、なかなか続きませんでした。人間がコピペして貼らないといけないことも面倒でした。

そこで、AIに相談しながら、自分専用のポータルサイトを作りました。今日の予定、積み残し、各プロジェクトのTODO、AIニュースなどを一画面で見られるようにしています。

既存ツールに自分を合わせるのではなく、自分とAIが使いやすい業務画面を作る。これも、AIネイティブな事業開発の一部だと感じています。

また次回も読んでもらえるとうれしいです。ここまで読んでいただきありがとうございました。
