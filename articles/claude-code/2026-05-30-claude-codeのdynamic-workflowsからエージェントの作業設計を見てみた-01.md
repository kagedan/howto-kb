Claude CodeのDynamic Workflowsで、実行されたエージェントのプロンプトを見てみました。

最初は「ワークフロー上でどんなエージェントが動いているのか」を知りたくて見ていました。

しかし実際に面白かったのは、そこに書かれていたのが単なる手順ではなく、エージェントにどう仕事を進めさせるかという作業設計そのものだった点です。

見る観点をどう分けるか。  
調査範囲をどう絞るか。  
何を指摘として報告させるか。  
見つからなかったときにどう返させるか。  
Outcomeを後続作業に渡せる形にするにはどうするか。

この記事では、各エージェントに表示されていたプロンプト全文の転載ではなく、Prompt / Activity / Outcomeの構造から、エージェントのワークフロー設計として参考になりそうな点を整理します。

Dynamic Workflows自体については、Anthropicの公式記事「[Introducing dynamic workflows in Claude Code](https://claude.com/blog/introducing-dynamic-workflows-in-claude-code)」で紹介されています。

公式説明を雑に要約すると、Claudeがタスクに応じてワークフローを組み立て、複数のサブエージェントに作業を分解して進める仕組みです。

## 何を見たのか

確認した環境は、2026-05-30時点のClaude Code v2.1.158です。

対象にしたワークフローは`product-defect-audit`です。

入力した依頼は、自分の作っているプロダクトについて、セキュリティ面とUI/UX面の欠陥を探してほしい、というものでした。

この記事は、プロンプト全文や具体的な指摘内容を紹介するものではありません。

見たいのは全文そのものではなく、そこから読み取れた構造です。

* Prompt / Activity / Outcome がどう分かれているか
* フェーズやエージェントの分解がどう見えるか
* エージェントごとの担当観点や調査範囲がどう指定されているか
* 報告条件がどう絞られているか
* Outcomeがどう後続作業に渡せる形になっているか
* CLAUDE.mdやカスタムコマンドに何を持ち帰れるか

結論から言うと、Dynamic Workflowsで表示されるエージェントごとのプロンプトは、単なる指示文というより「作業の型」に近く見えました。

## Dynamic Workflowsの画面で見えたもの

Claude Codeで`/workflows`コマンドを実行すると、そのセッション内で作成されたワークフローの一覧を見られます。

たとえば、次のように表示されます。

```
Dynamic workflows
  2 completed

    ✔ audit-findings-to-requirements  26 agents · 1.2m tok · 9m 2s
  ❯ ✔ product-defect-audit  61 agents · 2.4m tok · 10m 6s

  ↑/↓ to select · Enter to view · s to save · Esc to close
```

ここからワークフローを選ぶと、その作業で実行された内容を確認できます。

詳細画面では、フェーズごとにどのエージェントが動いたか、どのモデルが使われたか、トークン数やツール実行数、実行時間も見られました。

```
product-defect-audit
Audit a product for security, UI/UX, and generation-quality defects...
61/61 agents · 10m05s · done

╭ Phases ──────────┬ Review · 9 agents ─────────────────────────────────────────╮
│ ❯ ✔ Review   9/9 │  ✔ find:sec-authz       Sonnet 4.6             118.4k tok │
│   ✔ Verify 52/52 │  ✔ find:sec-secrets     Sonnet 4.6                58k tok │
│                  │  ✔ find:ux-a11y         Sonnet 4.6             127.5k tok │
│                  │  ✔ find:gen-prompt      Opus 4.8 (1M context)   84.9k tok │
│                  │  ✔ find:gen-pipeline    Opus 4.8 (1M context)   84.6k tok │
╰──────────────────┴───────────────────────────────────────────────────────────╯
```

つまりDynamic Workflowsでは、単に「ワークフローが実行された」だけでなく、表示されている範囲では、その内側でどんな作業単位が作られ、どのモデルに割り当てられたのかまで確認できます。

個人的に一番参考になったのは、個別のエージェントを選んだときに表示される内容です。

おおまかに、次の三段構成になっていました。

```
Prompt · 20 lines
  You are auditing this codebase...
  FOCUS: Frontend & route-level security.
  Report each finding with file+line evidence.

Activity · last 3 of 50 tool calls
  Grep(...)
  Read(...)
  StructuredOutput

Outcome
  Four confirmed defects, each with concrete file+line evidence:
  ...
```

`Prompt`は、そのエージェントに渡された担当観点と調査範囲です。

ここは全体を確認できました。

`Activity`は、実際にどのようなツール呼び出しを行ったかの実行履歴です。

ただし、表示上は`last 3 of 50 tool calls`のように末尾の一部だけが見えており、すべてのツール呼び出しを一覧できるわけではなさそうでした。

そのため、Activityから完全な実行ログを復元できるわけではありません。あくまで「どの種類の作業をしていたか」を見るための手がかりとして捉えるのがよさそうでした。

`Outcome`は、そのエージェントが最終的に返した結果です。

今回見た例では、ここに検出した欠陥がまとめられていました。

単に「問題がありそう」と書くのではなく、次のような情報を含む監査レポートに近い形でした。

* 何が問題か
* どのファイル・行にあるか
* 重要度や分類は何か
* なぜそれが問題なのか
* どう直せばよいか

つまり今回見たOutcomeは、エージェントの作業ログというより、後続の修正作業に渡せる成果物に近い形になっていました。

この三段構成になっていたので、「何を指示され、どんな種類の作業をして、何を返したか」を後から追いやすいと感じました。

## 何が面白かったか

印象的だったのは、Dynamic Workflowsが「最終回答」だけではなく、作業の分解過程を見える形で残している点です。

通常、AIエージェントに仕事を頼むと、最後の回答だけを見て良し悪しを判断しがちになります。

しかしDynamic Workflowsでは、少なくとも表示されている範囲で、次の3つを分けて確認できます。

* どんなプロンプトでエージェントが起動されたか
* そのエージェントがどんな種類の作業をしたか
* 最終的に何を成果物として返したか

これによって、結果だけでなく、作業設計そのものをレビューしやすくなります。

たとえばOutcomeが良くなかったときに、単に「AIの精度が低い」と見るのではなく、次のように切り分けられます。

* Promptの焦点が曖昧だったのか
* Activityを見る限り、十分な調査が行われていなさそうなのか
* Outcomeのまとめ方が悪かったのか
* そもそもエージェントの分け方が粗かったのか

もちろん、Activityで見えるツール呼び出しは一部なので、実行内容を完全に検証できるわけではありません。

それでも、最終回答だけを見るよりは、どこで失敗した可能性があるのかを考えやすいです。

## エージェントごとのプロンプトから見えた作業設計

今回見たプロンプトは、抽象的な作業指示ではなく、かなり具体的なエージェント向けの指示になっていました。

ここから、自分でエージェント向けの指示を書くときにも参考になりそうな点がいくつかありました。

### エージェントごとに見る観点が具体的

各エージェントには、かなり絞られた担当観点が与えられていました。

たとえば、セキュリティ全般をなんとなく見るのではなく、認可、シークレット、入力、フロントエンドのように観点が分かれています。

これは、人間のレビューでもよくやる分け方に近いです。

1人に全部見せるより、観点ごとに担当を分けた方が、何を見るべきかが明確になります。

エージェントにとっても、「セキュリティを見て」という大きな依頼より、「認可境界に絞って見て」の方が動きやすそうに見えました。

### 調査範囲が絞られている

プロンプトには、どの領域を見るべきか、どの観点で見るべきかが書かれていました。

ここが曖昧だと、エージェントは広く浅く見てしまいます。

逆に、調査範囲が絞られていると、Outcomeも具体的になりやすいです。

今回の例でも、エージェントごとに見るべき観点が分かれていたことで、Outcomeが「なんとなく全体を眺めた感想」ではなく、特定の問題に寄った報告になっていました。

### 報告条件が厳しい

印象的だったのは、報告条件がかなり厳しく書かれていたことです。

「それっぽい懸念」ではなく、実際のファイルと行番号に基づいた、説明できる欠陥だけを返すような指示になっていました。

これは重要だと思います。

エージェントのレビューは、油断すると一般論や可能性の話が混ざります。

「この実装は危険かもしれない」  
「この設計は問題になる可能性がある」  
「一般的にはこうするべき」

こういう指摘も、場合によっては役に立ちます。

しかし、欠陥監査として後続の修正作業に渡したい場合は、推測よりも証拠が重要になります。

今回見たプロンプトでは、そこをプロンプト側で抑えようとしているように見えました。

### 空振りを許容している

見つからなかった場合は、無理に何かを出すのではなく、空の結果を返してよいという扱いになっていました。

これも地味に大事だと思いました。

これまでエージェントにレビューを頼んできた体験でも、「必ず何か見つける」設計にすると、エージェントは弱い指摘を作りがちになると感じています。

本当は重大ではないものを、それっぽく欠陥として報告してしまいます。

逆に、空振りを許容すると、「証拠をもって言えるものだけ出す」方向に寄せやすいです。

今回のワークフローでは、そうやってOutcomeの信頼性を上げようとしているように見えました。

### Outcomeが修正可能な形になっている

Outcomeは、単なる要約ではありませんでした。

見つかった欠陥について、何が問題か、どこにあるか、なぜ問題か、どう直すかまで含めていました。

これはかなり参考になりました。

エージェントにレビューを頼むとき、つい「問題を探して」とだけ言ってしまいます。

でも実際に次の作業につなげるには、「問題の一覧」だけでは足りません。

少なくとも、次の情報があると修正に入りやすいです。

* 対象ファイル
* 対象行
* 問題の説明
* 影響範囲
* 重要度
* 分類
* 修正方針

今回見たOutcomeは、後続の修正作業にそのまま渡しやすい形になっていました。

ここは、自分でCLAUDE.mdやカスタムコマンドを書くときにも取り入れたいと思いました。

## プロンプトを読む価値

Dynamic Workflowsで表示されるエージェントごとのプロンプトを読む価値は、「Claude Codeが裏側で何をしているかを完全に知れる」ことではありません。

Activityも一部しか見えないので、内部の実行過程をすべて復元できるわけではありません。

それでも、表示されているPrompt / Activity / Outcomeを見ることで、Claude Codeが複雑な作業をどう分解しようとしているのかを観察できます。

今回見たプロンプトでは、少なくとも次の点が明示されていました。

* エージェントの担当観点
* 調査対象の範囲
* 見つけるべき欠陥の種類
* 読むべきファイルや観点
* 報告するときの条件
* 空振りだった場合の扱い
* 成果物の形式

この構造はDynamic Workflows専用の話ではありません。

CLAUDE.mdやカスタムコマンドを書くときにも、そのまま参考にできます。

エージェント向けの指示を書くときは、「何をしてほしいか」だけでは足りません。

むしろ重要なのは、「どう進めてほしいか」だと思います。

## CLAUDE.mdやカスタムコマンドに持ち帰るなら

今回見たプロンプトから、自分の指示にも取り入れられそうだと思ったのは次の構造です。

* 担当観点: 何を見るエージェントなのか
* 範囲: どのファイル・領域を見るか
* 観点: 何を欠陥とみなすか
* 証拠: ファイルと行番号など、報告に必要な根拠
* 除外: 推測や一般論を出さない
* 空振り: 見つからなければ空で返す
* 形式: 後続作業に渡せる形で返す

たとえば、レビュー用のカスタムコマンドを書くなら、単にこう書くよりも、

次のように書いた方がよさそうだと思いました。

```
認可境界に絞ってレビューしてください。

実際のファイルと行番号で説明できる問題だけを指摘として報告してください。
推測や一般論だけの指摘は含めないでください。
該当する指摘がない場合は、空の結果を返してください。

各指摘には、問題の内容、対象ファイル、対象行、影響、重要度、修正方針を含めてください。
```

これは今回見たプロンプトそのものではありません。

ただ、Dynamic Workflowsで表示されていたエージェントごとのプロンプトを見て、自分の指示にも持ち帰れそうだと思った型です。

「何を見るか」だけでなく、「何を指摘として認めるか」「何を報告しないか」「どんな形で返すか」まで書く、ということです。

これだけでも、エージェントの出力はかなり変わりそうに思います。

## まとめ

今回は、自分の作っているプロダクトに対して「セキュリティ面とUI/UX面の欠陥を探して」と頼んだときに、Claude CodeのDynamic Workflowsでどのように作業が分解されていたのかを見ました。

一番の収穫は、最終的な指摘だけでなく、その指摘に至るフローの一部を見られることでした。

どんなエージェントが立ち、どんなプロンプトが渡され、どのような観点で指摘を探し、Outcomeとしてどうまとめるのかを確認できました。

もちろん、Activityで見えるツール呼び出しは一部なので、すべての実行過程を確認できるわけではありません。

それでも、Prompt / Activity / Outcomeを分けて見られることで、エージェントの作業設計をかなり観察しやすくなります。

Dynamic Workflowsは、便利機能として使うだけでなく、Claude Codeが複雑な作業をどう分解しているかを観察する教材としても面白いです。

今後、自分でCLAUDE.mdやカスタムコマンドを書くときは、今回見たように、担当観点、範囲、報告条件、空振り時の扱い、Outcomeの形式まで含めて設計したいです。
