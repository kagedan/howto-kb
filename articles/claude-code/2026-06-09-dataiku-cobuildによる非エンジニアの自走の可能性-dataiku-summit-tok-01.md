# Dataiku Cobuildによる非エンジニアの自走の可能性

## はじめに

2026年6月9日、Dataiku Summit Tokyo に参加しました。

光栄にも弊社の上村くんが登壇することになり、その応援とともにDataikuの最新機能や事例を知るのが目的でした。

<https://meet.dataiku.com/dataiku-summit-tokyo-20260609/>

基調講演の中で発表されたのが、6月リリース予定の Dataiku Cobuild です。自然言語の業務要望から Dataiku の Visual Flow（pipeline、model、agent、application）を直接生成するという、これまでの Dataiku には無かった新しい AI アシスト機能でした。

実は先月、社内のアナリストから一つの要望が挙がっていました。「AIエージェントをスムーズに開発し、業務に展開できるようにしたい」という、シンプルだが、答え方が無数にある要望です。その答えを、私はずっと探していました。

Cobuild のセッションを見ながら、こう思いました。

> 「これは、私が探していた答えなのではないか」

当時、この要望に答えるために、私はClaude単体でのAIエージェント実装 → Dataikuへのビルトイン → Streamlit in Dataikuへのリファクタリングによるコーディング量削減 → Claude Code in Dataiku（以下 **CCiD**）によるDataiku内バイブコーディング → CCiDによるSnowflake Cortex Agentラッパー開発、という5段階のツール・実装方式を乗り換えて、開発手法を整理していきました。

CCiDに到達した時点で、私自身の開発生産性は劇的に上がりました。しかし、当初のアナリスト要望を満たすにはまだ難易度がある、というのが実感でした。

なぜ届かないのか。どこまで行ってもエンジニア的UI（CLIによる、狭いユーザー体験）であり、これを民主化、市民開発とするのは違うなと考えました。

そして Cobuild が、そのもやもやに対する一つの仮説を与えてくれたように思います。

この記事では、その思考の過程を書きます。技術機能の解説というよりは、 **「アナリストに AI エージェントの開発を自走できるようにするための最適なプロセスとツールは？」** という問いを、5段階の探索史を辿りながら Cobuild への期待として再構成したものです。

Cobuild はまだ実機で触れていないので、本記事の Cobuild に対する評価はすべて仮説となります。可能性の提示として読んでいただければと思います。

---

## 第1部：民主化ではなく自走化

業界には「データの民主化」という言葉があります。

この10年、データプラットフォームの主要ベンダー、Snowflake も Dataiku も Tableau も、Power系も、皆が「データの民主化」を掲げてきました。Dataiku は "Everyday AI" を看板に、「誰もが AI を使える世界」を訴え続けています。

私自身、この方向性に異論はありません。実際、Vポイントマーケティングの中でも Dataiku を使うアナリスト層は着実に広がっており、Visual Recipe や Visual Agent、Webapp といった機能を使って、自分たちで分析や簡単なアプリケーション構築まで踏み込めるようになっています。これは10年前には考えられなかった世界です。

ただ、今回のアナリスト要望に向き合う中で、私は **「民主化」という言葉では足りない** ことに気づきました。

民主化は、 **ツールを誰でも触れる状態にすること** です。Visual Recipe で誰でもデータを加工でき、Visual Agent で誰でもエージェントを定義できる、という状態を作る。これは確かに重要な達成です。

しかし、ツールが触れる状態と、アナリストが本当に自分の業務課題を **自分で解決し続けられる状態** は、別の話だと考えるようになりました。後者を表す言葉として、私は **「自走化」** を提案したいと思います。

自走化は、民主化を含んで、なお超えるものです。

| 概念 | 焦点 | 達成条件 |
| --- | --- | --- |
| 民主化 | ツールへのアクセス | 誰でもツールを使える環境がある |
| 自走化 | 課題解決の継続性 | アナリスト自身が課題を解決し続けられる |

民主化されただけのアナリストは、ツールの学習コスト、機能の天井、新たな要件への対応力、これらを自分で乗り越えられないと、結局エンジニアの手を借りることになります。「ツールはあるけれど、結局誰かに頼まないと進まない」という状態は、民主化されていても自走化されていない状態だと言えます。

私が今回のアナリスト要望で向き合ったのは、まさにこの「ツールはあるが自走できない」というギャップでした。Dataiku も Visual Agent も、ツールとしては既に存在しています。それでもアナリストが自分で AI エージェントを構築・運用し続けられるかというと、そこにはまだ大きな距離がありました。

なぜか。次の部では、まず Dataiku 自体に存在していた「学習コストの壁」について書きます。

---

## 第2部：Dataikuは、まだ難しかった

Dataiku は確かに「ノーコード／ローコード」のプラットフォームです。Visual Recipe を選んでカラムを指定すれば、SQL を書かなくてもデータ加工ができる。Visual Agent でツールとプロンプトを定義すれば、コードを書かなくてもエージェントが動く。これらは大きな抽象化です。

ただ、 **「コードを書かなくてよい」と「簡単に使える」は、同じではありません** 。

実際の業務でアナリストが Dataiku に触れたとき、彼らが直面したのは以下のような壁でした。

* Flow のどこに何を置くべきか、という設計判断
* 同じ加工が複数の Visual Recipe で可能なときの選択（Group, Join, Prepare, ...）
* Recipe が失敗したときのエラー画面の読み解き
* Visual Recipe で実現できない処理に行き当たったときの代替手段の選択
* Project Variables, Scenarios, Schedules の意味と使い分け
* 自分が触っている Dataset が「上流の何から派生しているか」の追跡

これらはどれも、 **コードを書かないことで隠蔽されているわけではなく、Dataiku 固有の概念として新しく学ぶ必要のあるもの** です。「コードを書かなくてよい」は、 **代わりに Dataiku の概念体系を学ぶ必要がある** ことを意味します。

そして、業務の複雑度が一定を超えると、Visual Recipe の天井に行き当たります。Window 関数、再帰的な処理、複雑な条件分岐。これらが必要になった瞬間、Python Recipe を書くか、Code Studio で何かを実装するか、という選択肢に切り替わる。ここでアナリストの自走は、ほぼ確実に止まります。

つまり Dataiku は、 **「民主化されているが、自走の途中で必ず壁にぶつかる」** プラットフォームでした。

この前提の上に、AI エージェントを業務で使えるようにする、という要望が乗ってきたわけです。エージェントは Visual Agent で定義できますが、業務に組み込むためには、データ取得、前処理、結果のハンドリング、UI、運用、これらすべての設計が必要になります。 **Visual Agent 単体ではエージェントは「使える」状態にならない** 、というのが私の出発点でした。

ここから5段階の探索が始まります。

---

## 第3部：5段階の探索

ここからは、私自身が辿った道のりを時系列で書きます。

補足として、この5段階を通じて **私自身のコーディング作業のほとんどは Claude との対話によるバイブコーディングで進めました** 。コードを自分で書くのではなく、Claude に書かせて自分は判断する、というスタイルです。これは、Cobuild に対する評価の根底にも繋がる姿勢です。

注意していただきたいのは、 **これは私が開発者として最適化していった過程であり、アナリストに渡せる完成形を作っていった過程ではない** ということです。この区別が、後の議論で重要になります。

### 段階1：Claude 単体で Webhook エージェント

最初のステップは、Claude を使って Webhook 経由で動く小さなエージェントを書くことでした。  
業務上の要望に対して、「そもそも AI エージェントで何ができるのか」「どこまでの応答品質が出せるのか」を、自分の手で検証する段階です。

このフェーズは Claude との対話的開発で、生産性は高い。プロトタイプはすぐ動きます。

ただし、致命的な制約がありました。 **Dataiku の本番データにも、社内の既存パイプラインにも、直接はアクセスできない** ということです。Webhook で立てたエージェントは、組織のデータ統制の外側にいる、孤立した実験環境でした。

このまま業務に組み込むことはできない。次のステップは、Dataiku の中に持っていくことでした。

### 段階2：Dataiku にビルド

エージェントのロジックを Dataiku Project の中に取り込み、Webapp や Code Recipe として動くようにしました。これでようやく、本番データに正しい権限でアクセスし、出力を Dataset として残せるようになります。

ここで得たものは非常に大きく、本番資産との接続、ガバナンスの中での実行、Flow としての可視化。  
失ったもの、というよりむしろ新しく直面した課題は、 **UI の窮屈さ** でした。  
Dataiku 標準の Webapp は、業務担当者に渡す体験としては作り込みが必要で、しかも作り込もうとすると HTML/CSS/JavaScript を書く比重が一気に増えます。

「自由度を取りに行くとコード量が爆発する」という、フロントエンド実装によくある罠に、Dataiku の中でも同じように直面した、という感覚でした。

### 段階3：Streamlit へリファクタリング

そこで Streamlit に移しました。Streamlit はデータアプリに特化したフレームワークで、 **HTML/CSS/JavaScript 直接実装と比べると 1/3 程度、Dataiku 標準 Webapp の作り込みと比べてもおおむね半分以下にコード量が圧縮されました**。データを見せる、入力を取る、結果を返す、という基本動作だけなら、本当に数十行で書けます。

この時点で、コード量の観点では明らかに楽になりました。

ただし、別のコストが発生していたことに、後から気づきます。 **Dataiku と Streamlit の境界をまたぐ「中継作業」が増えていた** のです。

Streamlit のコードから Dataiku の Dataset を参照するためには、パスを書く、エクスポートする、認証情報を渡す、結果を別フォルダに書き出す、Dataiku 側で取り込む、といった作業が発生します。コード自体は短いのですが、 **コードを取り囲む手作業の総量** は、必ずしも減っていませんでした。

自由度の高い Web アプリ実装と比べれば確かに楽になっている。しかし、「Dataiku を使う意味」の観点では、なにか中途半端な状態にいました。

### 段階4：CCiD への移行

転換点は CCiD でした。

CCiDの本質は、 **Claude Code が `dataiku` Python SDK を経由して、Project 内のオブジェクト（Dataset, Folder, Recipe, Project Variables など）を直接認識した上でコードを書ける** 、という点にあります。

このCCiDについて、keywalker社の記事がとても分かりやすく、私も参考にしながら構築しました。  
<https://www.keywalker.co.jp/blog/dataiku-claudecode.html>

Dataiku オブジェクトを認識させる経路は、私の理解では大きく3つあります。

* **経路A**：Project 構造を CLAUDE.md に手動で記述する
* **経路B**：Dataiku Python SDK 経由で Claude Code が動的に取得する
* **経路C**：MCP サーバー経由（執筆時点で、公式の Dataiku 向け Claude Code MCP は確認できていません）

私は **経路B** を採りました。Code Studio 内で起動した Claude Code は、現在の Project の context を SDK が自動的に保持するため、`dataiku.Dataset("...")` を書けば即座に該当 Dataset を参照できます。CLAUDE.md の手動メンテも、外部認証情報の管理も最小限で済みます。

この瞬間、 **段階3まで存在していた「中継作業」がほぼ消えました** 。

Streamlit に Dataset を渡すために CSV をエクスポートする必要も、結果を別フォルダに書き出して Dataiku に取り込み直す必要もない。 **コードは Dataiku Project の内側で、Project の資産を直接読み書きしながら動く** 状態になりました。

ただし、CCiD には CCiD 固有の運用上の留意点があります。経路B を本格的に使う中で、私が直面した主なものは以下の通りです。

1. **探索コードの繰り返しによるトークン消費**：Claude Code は毎セッション、Project メタデータを取得するコードを再実行しがちで、Project 内のオブジェクト数が増えるとコンテキストを圧迫します。
2. **認証情報の取り扱い**：Code Studio 外で CCiD を使うときの API key の扱い、CLAUDE.md や Git 履歴に混入させない運用ルールが必要になります。
3. **破壊的 SDK 操作のリスク**：`dataset.clear()` や `project.delete()` のようなメソッドが SDK には存在しており、これを Claude Code が「クリーンアップして再生成しよう」と判断して呼ぶ事故は理屈上起こりえます。CLAUDE.md に明示的な禁止リストを置く必要があります。
4. **SDK バージョン差異**：Claude Code は学習時点の SDK 仕様で書くため、本番 Dataiku のバージョンと食い違うコードを生成しうる。CLAUDE.md にバージョンを明記し、不明なメソッドは公式ドキュメントを確認させる指示が要ります。

これらは罠ではあるものの、CLAUDE.md の設計とガードレール（`.claude/settings.json` 等）でかなりの部分は塞げます。 **塞ぐ前提で運用すれば、CCiD は確かに強力な開発基盤になります** 。

### 段階5：CCiD で Snowflake Cortex Agent ラッパーとパイプラインを開発

段階4 で CCiD が定常運用できる土台になったので、より複雑な実装に踏み込みました。 **Snowflake Cortex Agent をラップした業務 AI と、それを業務データに接続するパイプラインの開発** です。

Cortex Agent 自体は Snowflake 側にあり、API 仕様も比較的新しく、サンプルコードも限られています。これを業務で使える形にラップするには、認証、リクエスト構造、レスポンス処理、エラーハンドリング、リトライ、ログ、と書くべきものが多い。一人の開発者がスクラッチで書こうとすると、それなりの工数になります。

CCiD はここで本領を発揮しました。Cortex Agent の ドキュメントを CLAUDE.md に参照させ、Dataiku Project 内の Dataset を入出力として直接読み書きするコードを、対話的に組み上げていく。仕様の確認、エラー再現、修正のループが極めて速い。

この段階で、私の開発生産性は明確に最大化されました。

そして同時に、私は気づきました。 **これはアナリストには渡せない** と。

---

## 第4部：CCiD の限界 — 開発者の最適化はアナリストに届かない

段階5 まで来た私は、開発者として CCiD という強力な道具を手にしていました。Dataiku オブジェクトを認識する Claude Code、Cortex Agent ラッパー、本番接続、テスト可能な開発環境。 **私の生産性は最大化されました** 。

しかし、そこにアナリスト要望の答えは無かった、というのが正直なところです。

なぜか。CCiD を使うために必要な前提を並べてみるとよくわかります。

* ターミナル／CLI 操作
* Python の読み書き
* Dataiku Python SDK の知識
* Code Studio の起動と環境管理
* CLAUDE.md の設計
* `.claude/settings.json` 等のガードレール設定
* Git 連携と認証情報の扱い
* 生成された Python コードのレビュー

これらは個々に見れば、エンジニアにとってはどれも当然のスキルです。しかし **アナリストにとっては、いずれも乗り越えるべき新しい学習対象** になります。Dataiku 自体の学習コスト（第2部）の上に、CCiD のための学習コストが積層する形になります。

整理するとこうです。

| 層 | 学習対象 | アナリストへの負荷 |
| --- | --- | --- |
| Dataiku 層 | Flow、Recipe、Project構造、Visual Agent | 既に学習コストあり |
| CCiD 層 | CLI、Python、SDK、CLAUDE.md、ガードレール | さらに上乗せ |

さらに、CCiD は Anthropic API に直接接続する形態のため、Dataiku の LLM Mesh などの組織横断的な AI ガバナンス機構を経由しない、という側面もあります。CLAUDE.md とガードレールで個別の Project レベルの統制はできますが、組織全体の AI 利用を統合的に監視・統制する経路としては、別途設計が必要な領域でした。

これに加えて、運用していて気になった技術的な限界がもう一つあります。**CCiD で利用する Claude Code は強力ですが、Dataiku 固有の仕様や機能、特に新しい機能やバージョン差異については、推測でコードを書くことがありました**。SDK のメソッド名を間違える、引数の仕様を取り違える、最新版で変わった挙動を反映できない、といった事象です。CLAUDE.md でバージョンや参照ドキュメントを明示することである程度は緩和できますが、 **Dataiku 特化の知識が深いほど望ましい** 、というのは CCiD 運用上の実感でした。この点は後の第6部の議論につながります。

CCiD は **開発者の自走化** には完璧に効きました。私自身、CCiD なしで Cortex Agent ラッパーを同じ速度で書ける気はしません。  
しかし **アナリストの自走化** には届かない。むしろ、 **エンジニアと業務側の距離を「自走化の経路としては」広げてしまう** 側面すらあると感じました。

これが、 CCiD まで到達してなお、当初の要望に答えられた感覚が持てなかった理由でした。

私が探していたのは、開発者の生産性ではなく、 **アナリストが自分で AI を使い続けられる経路** だったのです。

「みんなも勉強すればできるよ！」と内部レビューで説明はしましたが、それは選ばれた数名に閉じた話になるだろうと想像していました。

そして、それに対する解決策が見えないまま、私は Dataiku Summit Tokyo の会場に座っていました。

---

## 第5部：Cobuildを見た瞬間

Dataiku Summit Tokyo 2026 のセッションで Cobuild が発表されたとき、画面に映っていたのはこういうデモでした。

業務担当者が自然言語で「こういうエージェントを作って、こういうデータを入力にして、こういう承認フローで運用したい」と書く。すると Dataiku のキャンバスに、Flow と Agent と Application が、構造化された Visual Flow として生成される。生成された Flow は、Visual Recipe や Visual Agent と同じ画面上で、人間がレビュー・修正・承認できる。

公式の説明は、AI アシスト型の開発が監査可能で透明なワークフローを生む、という主旨でした。Dataiku の CEO は、コーディングアシスタント型が「コードが推論パスの中に埋もれて、業務ユーザーが読めず、監査人がたどれなくなる」と批判し、Cobuild はそうではない、と語っていました。

これは Dataiku 公式が掲げる「AI 支援開発のブレイクスルー」というメッセージです。

ただ、 **私が会場で感じたブレイクスルーは、それとは少し違うものでした** 。

私が見ていたのは、第2部で書いた Dataiku の学習コストの壁と、第4部で書いた CCiD の追加学習コストの壁、 **その二重の壁を Cobuild が一気に下げる可能性** でした。アナリストが Dataiku の概念体系も、Python も、CLI も、CLAUDE.md も意識せずに、業務要望を Dataiku のネイティブ資産に変換できる経路が、初めて見えた瞬間だったのです。

Cobuild が解決するのは「AI 支援開発の透明性」だけではない、と私は思いました。  
解決しているのは、 **アナリストの自走化を阻んできた、構造的な学習コストの壁** です。

## 第6部：根拠ある期待 — CCiD の経験から推論できる実現可能性

ここで、Cobuild への期待を「Dataiku 公式が発表したから」という他者依存の根拠ではなく、 **自分の経験から推論できる根拠** で支えてみたいと思います。

### 推論1：Visual Flow 生成は技術的に実現可能と考えられる

私は CCiD を使って、自然言語の依頼から Dataiku Project 内で動く Python コードを生成し、Streamlit や Cortex Agent ラッパーとしてリリース可能な状態にまで持っていく経験をしてきました。

このプロセスは、

1. 自然言語の理解
2. Dataiku オブジェクトの認識（経路B）
3. 目的のターゲット形式への変換（コード）

という3ステップで構成されています。

Cobuild が行うことは、上記3ステップの **ステップ3 を「コード」ではなく「Visual Flow」に置き換えたもの** だと整理できます。1 と 2 は CCiD で既に実証されている動作であり、Dataikuの Project Export を見ると、Flow は YAML や JSON 形式の構造化データとして表現されています。ターゲット形式の変換は、Dataiku 自身が最も詳しい領域です。

つまり **Cobuild の基本機能の実現可能性は、CCiD の経験から推論できる** と考えています。ターゲット形式の差は、出力テンプレートの差にすぎないからです。

ただし、 **「実現可能」と「実用品質」は別** です。生成品質、複雑度の天井、エラー時のフォールバック、保守性、これらは出力形式が変わる以上、CCiD の経験から直接外挿はできません。ここは実機検証を待つ必要があります。

### 推論2：Dataiku 特化型 AI による「安心感」の構造

ここで、第4部の末尾に書いた論点に戻ります。CCiD の Claude Code は、 **Dataiku 固有の仕様や機能について推測でコードを書くことがありました** 。Claude Code は汎用的に強力ですが、Dataiku 特化の知識という意味では、学習時点の知識と、CLAUDE.md で渡せる範囲の参照情報に依存しています。

これは Dataiku のドキュメント整備やバージョン進化のスピードと、Claude Code 側の知識更新のサイクルが必ずしも揃わないことから来る、構造的な限界でした。CLAUDE.md でカバーするにも限界があり、結果として **Dataiku の流儀から外れたコード設計が混入することがあった** 、というのが運用上の実感です。

Cobuild は、 **Dataiku 自身が開発する AI である以上、Dataiku の仕様・機能に習熟したコーディングと設計支援を提供することが期待できます** 。これは、汎用 LLM をベースとした CCiD には構造的に難しい領域です。

ここに、私が CCiD で長く向き合ってきたガバナンス課題への、構造的な答えが見えました。

CCiD のガバナンス課題は、突き詰めると3つです。

1. 生成物がコードのため、業務側がレビューしきれない
2. Dataiku 固有の仕様への推測が混入し、Dataiku の流儀から外れた設計になることがある
3. 生成経路が LLM Mesh 等を経由しないため、既存のガバナンス機構の外側にいる

Cobuild の設計は、これらに対して以下のように答える可能性があります。

| CCiD の課題 | Cobuild の構造的回答 |
| --- | --- |
| 生成物のレビュー困難 | Visual Flow として可視化される（誰でも同じ画面で見える） |
| Dataiku 仕様への推測 | Dataiku 自身が開発する AI として、仕様・機能への習熟が組み込まれる |
| ガバナンス機構との分離 | Dataiku ネイティブのため Govern / Agent Management と連携することが期待される |

特に重要なのは、 **Dataiku 特化型 AI による生成の精度** だと考えています。

これまで、汎用 LLM ベースの AI コーディング支援は、対象プラットフォームの仕様を「推測する」前提で動いていました。プラットフォーム特化の AI は、推測を排除し、 **「そのプラットフォームの正しい使い方に沿った生成物」が初期状態から得られる** ことを意味します。

なお、Snowflake Intelligence の "Sandbox & Skills" のような、 **プラットフォーム特化の再利用部品の概念は既に他プラットフォームに存在** しており、Cobuild も同様の方向性で進んでいると見ることができます。

「アナリストに自由を渡しつつ、組織は安心していられる」という状態は、これまで矛盾する要請でした。 **Cobuild が Dataiku 特化型 AI として、Visual Flow を生成し、人間が GUI でレビュー可能な形で提供する。この構造は、その矛盾を緩和する可能性を持つ**ように思えます。

## 第7部：仮説として残しておく論点

ここまで Cobuild への期待を書いてきましたが、実機検証ができていない以上、 **すべては仮説** です。むしろ、仮説段階だからこそ明示しておきたい不確実性が、いくつかあります。

第一に、 **生成品質と複雑度の天井** 。  
会話的な反復生成で、業務要件をどこまで正確に Flow に落とせるか。複雑な条件分岐、エッジケース、業務固有のルールが多い場合、Visual Flow としてどこまで表現可能か。複雑度が一定を超えた領域で Visual Flow が肥大化したとき、 **「ビジュアルを読み解く負担」が「コードを読む負担」を超える** 可能性があります。

第二に、 **Dataiku 特化知識と組織業務知識の接続** 。  
Cobuild が Dataiku の仕様に習熟していることと、ある組織の業務固有のルール・データモデル・命名規則を理解していることは、別の問題です。組織側で何を Cobuild に教え込む必要があるのか、それをどう継続的にメンテナンスするのか。Dataiku Govern との連携設計を含めて、実機を触ってから判断する領域です。

第三に、 **CCiD との境界線** 。  
Cobuild が解決するのは、おそらく自走化の前半、つまり「アナリストが標準的な業務 AI を構築する」段階です。一方、複雑な統合、既存コード資産のリファクタリング、性能チューニング、独自のラッパー実装、こうした領域は Cobuild の射程外として CCiD が残る、と私は予想しています。 **Cobuild は CCiD の上位互換ではなく、補完関係に立つ** 、というのが現時点の私の見立てです。

第四に、 **Cobuild の生成物の保守性** 。  
生成された Flow をその後どう保守するか。要件変更時に再生成するのか、Flow を直接編集するのか。再生成型と編集型のどちらが運用に乗るかは、組織のリテラシーとガバナンス次第で、現時点では断言できません。

これらの仮説は、 **6月の正式リリース後の実機検証で更新する余地のあるもの** として、本記事の論旨を裏付ける根拠としては「あくまで仮説」と位置付けておきたいと思います。

## 第8部：二層構造としての提言

仮説段階での提言ですが、Cobuild が想定通りの設計で出てくるとすれば、 **Dataiku 上の AI 開発は「二層構造」になる** と考えています。

| 層 | 主な担い手 | 主なツール | 主な責任範囲 |
| --- | --- | --- | --- |
| 自走化層 | アナリスト | Cobuild + Visual Recipe / Agent | 標準的な業務 AI の構築と運用 |
| 拡張層 | エンジニア | CCiD + Python / SDK | 複雑な統合、既存資産の保守、性能最適化 |

二層構造を組織として機能させるためには、3つの設計と、1つの前提条件があります。

### 層の境界を明示する

アナリストがどこまで Cobuild で自走できるのか、どこからエンジニアに委譲するのか、という境界の暫定的な指針を、組織として持っておく必要があります。「Visual Flow が一定の規模を超えたら」「Cobuild が Dataiku の標準的な機能セットを超える統合を要求されたら」のような、運用可能な判断基準が要ります。

### 委譲経路を整備する

アナリストが詰まったとき、エンジニア側にスムーズに引き継ぐ経路を作ります。Cobuild で生成された Flow を、エンジニアが CCiD で読み解き、必要な部分だけコードで補完する、というハイブリッドのワークフローが運用に乗れば、自走化層と拡張層の往復が機能します。

### 両層の監査経路を統合する

Cobuild の生成物（Visual Flow）と CCiD の生成物（コード）は、ガバナンスの観点では異なる経路を辿ります。Dataiku Govern / Agent Management が前者を、コードレビューと Git 履歴が後者を担うとすれば、両者を統合的に俯瞰できる仕組みを組織として持つ必要があります。

### ライセンスとアクセスの前提

公式情報によれば、Cobuild は当初 Designer-level users 向けに提供される予定です。「アナリストの自走化」を語るうえで、組織内のどの層が Designer ライセンスを持っているか、追加ライセンスのコスト、Snowflake 連携版を使う場合の Snowflake Cortex クレジット消費、といった現実的な制約が前提条件として効いてきます。技術的な可能性と、実際に組織で運用に乗るかは、ライセンス設計と切り離せません。

民主化の時代は「ツールを誰でも使える環境を作る」が組織の仕事でした。 **自走化の時代は「層構造を設計し、層の往復を機能させる」が組織の仕事** になります。

なお、本記事は Cobuild 本体を中心に論じていますが、2026年5月には Snowflake 連携版である **Cobuild on Snowflake** も発表されています。

Snowflake をデータ基盤として併用する組織にとっては、Cobuild 本体と Cobuild on Snowflake のどちらをどの用途で使うか、という選択も別途検討すべき論点になります。

## おわりに

冒頭の問いは「アナリストに AI エージェントを使えるようにする、とはどういうことか」でした。

5段階の探索で見えてきたのは、 **民主化（ツールを渡す）だけでは答えにならない** ということでした。アナリストが自分で業務課題を解決し続けられる **自走化** の状態をどう作るかが、本当の問いだったのです。

Cobuild が、その認識転換の起点になるのか。Visual Flow 生成と Dataiku 特化型 AI という二つの構造が、「民主化と統制」というこの10年解けなかった矛盾に答えを出せるのか。 **6月のリリースと、実機での検証** を経て、また書きたいと思います。

「データの民主化」から「アナリストの自走化」へ。  
6月を、楽しみに待ちたいと思います。
