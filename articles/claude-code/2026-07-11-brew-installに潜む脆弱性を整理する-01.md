## はじめに

きっかけは、AIコーディングエージェント（Claude Codeなど）が、タスク達成のために**結構自由に `brew install` を実行しようとする**場面に何度か遭遇したことでした。

「これ、会社支給のPCで本当に大丈夫なんだっけ？」

管理者権限がないとできない操作には別途承認プロセスが挟まる前提はあるものの、`brew install` の多くはユーザー権限だけで完結してしまいます。つまり、その承認プロセスの外側で色々なものが入り得る。

この記事では一旦AIエージェントの話は脇に置き、**`brew install` というコマンドそのものが技術的にどんな脆弱性を内包しているか**を自分の頭の整理も兼ねて棚卸しします。最後に、AIエージェントが絡むとその脆弱性がどう増幅されるかにも触れます。

---

## Formula と Cask の違い（前提知識）

Homebrewには大きく2種類のパッケージ形式があります。

* **Formula**: CLIツールやライブラリ（`git`, `wget`, `node`など）。ソースやビルド済みバイナリ(bottle)を取得し、`/opt/homebrew/bin`などにコマンドとして配置される。
* **Cask**: GUIアプリケーション（`Google Chrome`, `Slack`, `Docker Desktop`など）。`.dmg`や`.pkg`をダウンロードし、`/Applications`にインストールする、いわば「Homebrew経由のアプリインストーラ」。

どちらもRubyのDSLで書かれた「レシピ」であり、`homebrew/core` / `homebrew/cask` というGitHubリポジトリでメンテナンスされています。

---

## 1. サプライチェーン層のリスク

### 配布元のハイジャック

Formulaは多くの場合、アップストリーム（GitHub Releases、公式サイトなど）からソースやバイナリを取得します。Homebrew自体が健全でも、**配布元サーバーが侵害されればそのまま影響を受ける**という構造です。

### checksum検証の限界

Homebrewはダウンロード後にSHA256を検証しますが、そのchecksum自体がformulaファイル内に記述されています。formulaファイルが侵害されれば、checksumごと差し替えられてしまい、検証の意味が薄れます。

### サードパーティtap

`brew tap` で公式以外のリポジトリを追加すると、Homebrew本体が持つレビュー・検証体制の管理下から外れます。信頼度は一段階落ちると考えるべきです。

---

## 2. 実行時のコード実行リスク

* Formulaは Ruby DSL で書かれており、`install` ブロック内で**任意のシェルコマンドを実行できる**。
* Caskには `preflight` / `postflight` / `uninstall` などのフックがあり、ここにも任意スクリプトを仕込める。
* Caskで配布される `.pkg` は、その内部でmacOSのインストーラスクリプトを持つことがあり、Homebrew側はその中身までは逐一検証しない（署名検証止まり）。

つまり `brew install` は単なる「ファイルをコピーする」操作ではなく、**任意コードの実行を伴い得る操作**だという理解が重要です。

---

## 3. 権限・ファイルシステムのリスク

* Apple Siliconでは `/opt/homebrew`、Intelでは `/usr/local` が、デフォルトで**現在のユーザーの書き込み権限**で構成されます。sudo不要という利便性の裏で、そのディレクトリに書き込めるプロセス（ローカルの他プロセスやマルウェア）がいれば、既存のbrewバイナリを差し替えられるリスクがあります。
* PATHの優先順位次第で、システム標準コマンドをbrew版が先にシャドーイングします。意図せず改変されたバイナリが先に呼ばれる可能性。

---

## 4. ガバナンス・可視性のリスク（会社PC文脈）

* MDM/EDRはApp Store経由やpkgインストーラの導入は検知しても、**brew経由での導入を見落とす**ことがある。
* 自分の会社PCでは管理者権限は都度・時限付き（15分など）でしか付与されないが、その承認は「管理者権限を使うこと」自体への承認であり、**昇格後に何をインストールするかまでは見ていない**。昇格した数分の間に、任意のtap追加やパッケージインストールができてしまう点が本質的な論点。

---

## 5. AIエージェントによる増幅要因

ここまでは `brew install` 自体が元々持つ技術的な脆弱性でした。AIコーディングエージェントが関わると、これらのリスクは以下のように増幅されます。

* 人間なら都度吟味する「このパッケージ本当に要る？」という判断を、AIはタスク達成のために**高速・機械的に**実行し得る。
* tap追加やパッケージ選定の妥当性（公式tapか、最小権限か）を吟味せずに提案・実行してしまう可能性。
* 「なぜそれを入れたか」という説明可能性・ログが弱く、事後の監査が難しい。

---

## 対策の方向性

### 個人でできるチェック

* 見慣れないFormula/Caskを入れる前に、`brew info <name>` や `brew cat <name>` でレシピの中身（何をダウンロードし、どんなスクリプトを実行するか）を軽く確認する
* `brew tap` で公式以外のリポジトリを追加する前に、そのtapの信頼度（メンテナ・更新頻度など）を見る。使い終わったら `brew untap` で外す
* 迷ったら公式tap (`homebrew/core`, `homebrew/cask`) の範囲で完結させる
* `brew update` でformula定義を最新に保ち、古いchecksum情報のまま使い続けない
* AIエージェントに `brew install` を実行させる場合、提案されたパッケージ名・tapを一度自分の目で確認してから承認する

### 組織・AIエージェント向けの施策

* AIエージェントのpermission設定で、`brew install` / `brew tap` 系のコマンドは**都度確認を挟む**モードにする
* 会社PCでは、許可されたFormula/Caskのホワイトリスト運用を検討する
* 公式tap (`homebrew/core`, `homebrew/cask`) 以外の追加は原則禁止にする
* 定期的に `brew list` / `brew tap` で棚卸しし、脆弱性検証

---

## まとめ

`brew install` は便利な反面、

1. サプライチェーン（formula/配布元の信頼性）
2. 実行時のコード実行
3. 権限・ファイルシステム
4. ガバナンス・可視性

という4層でリスクを内包しています。個人レベルでは「入れる前にレシピを見る」「知らないtapを増やさない」だけでもリスクはかなり下がりますが、AIエージェントの自律性が高まるほど、この4層のリスクが**人間のレビューを介さずに顕在化するスピード**が上がります。だからこそ組織レベルでは、前節で挙げたような仕組み化が必要になる、というのが今回整理して見えてきた本質だと思います。
