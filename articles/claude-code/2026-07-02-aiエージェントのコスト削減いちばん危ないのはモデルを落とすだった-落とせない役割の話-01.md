## TL;DR

* X で「Fable 5は高いから、チームを組んでモデルを使い分ければ節約できる」という話が回ってきた。自分がずっとやってきたことだったので、実際に運用してみて分かった“その続き”を書く。
* 「重い役はいいモデル、軽い作業は安いモデル」は正しい。ただ、安いモデルに落とせない役がある。自分の場合は検証役（Critic）で、ここを安くしたら普通にバグを見逃した。
* そこで発想を変えた。モデルを落とすんじゃなくて、渡す入力の方を減らす。検証ゲートを3段に割って、安いモデルにテスト出力を潰させ、高いモデルにはコードだけ読ませる。
* うまくいかなかった案も書く。「git diffだけ渡す」は新規開発だと意味がなかったし、「書き手の一段上のモデルでレビュー」も筋が悪かった。
* 最後に、この節約の設計そのものを Fable 5自身にレビューさせた話をする。自分の初案は、そのFableにあっさり却下された。

土台になっているマルチエージェント基盤そのものは前回書いたので、そっちは前提として置いておく。

リポジトリ: <https://github.com/KoyanagiAyuha/team-framework>

## 発端はXのポスト

きっかけはこれ。

> Use Fable 5 as orchestrator and Opus + Codex to execute (to save fable usage):  
> Fable 5 (max reasoning) = orchestrator  
> Opus = deep reasoning subagent  
> Sonnet = mechanical work subagent  
> Codex = peer Sr. engineer, different perspective  
> — [@diegocabezas01](https://x.com/diegocabezas01/status/2072436501263339841)

このあとに具体的な手順（Fable をメインに設定して、opusとsonnetのサブエージェントを作り、Codex を別視点として足す…）が続くんだけど、言いたいことは「役割でモデルを使い分けて Fable の消費を抑える」の一点。  
Claude Codeにも`/model opusplan`（計画はOpus、実行はSonnet）が最初から入っているくらいで、「頭を使うところは上位モデル、手を動かすだけのところは安いモデル」はもう当たり前になっている。  
まとめ記事だと「これで4〜9割安くなる」なんて数字も見かける。

で、自分も前からマルチエージェント基盤で同じことをやっていた。  
「あ、これ自分のやつだ」と思った。  
ただ、実際に毎日回していると、この定石だけでは足りない場面にぶつかる。  
今日はそこの話をする。

## そもそも Fable が高い

料金はこう（2026年7月時点、入力/出力の $ / 100万トークン）。

| エイリアス | 実体 | 入力 / 出力 |
| --- | --- | --- |
| `sonnet` | Claude Sonnet 5 | $3 / $15 |
| `opus` | Claude Opus 4.8 | $5 / $25 |
| `fable` | Claude Fable 5 | $10 / $50 |

Fableは思考が常時オンで、レビューやデバッグ、計画あたりは頼りになる。  
ただ出力単価がOpusの2倍ある。  
何も考えず全部の役をこれで回すと、当然すごい勢いで溶ける。

ちなみに自分は$100のサブスクなんだけど、レビューを全部Fableで走らせていた頃は30分で上限に達した。  
従量課金なら金額、サブスクなら回数制限。  
どっちにしても「いいモデルを雑に使い倒す」は続かない。

なので今の構成では、Fableを使うのは重い判断のところ（Plannerの分解と Criticゲート）だけに絞っている。  
全役ではなく、いちばん頭を使う場所にだけ。  
この前提で以下を読んでほしい。

## 定石どおり、役割でモデルを分ける

うちは役割を認知機能で定義しているので、素直にモデルを当てるとこうなる。

| 役割 | やること | モデル | なぜ |
| --- | --- | --- | --- |
| Orchestrator | 全体の統括（メインセッション） | 人が `/model` で選ぶ | 橋渡しに徹する |
| Planner | 重い分解・計画 | fable | 立てるのは重い分解のときだけ。手前で手戻りを潰せる |
| Worker | スコープ内の実装 | sonnet | ほとんどのタスクは Sonnet で足りる。複雑なやつだけ opus |
| Critic | 成果物の検証 | fable | いちばん判断が重い関門 |

WorkerをSonnet既定にするだけで、実装の量に比例していたコストがぐっと落ちる。  
ここまでは記事の受け売りどおりで、問題はこの先だった。

## Critic を安いモデルに落としたら、見逃した

最初はCriticもOpusに落とそうとした。  
いちばん呼ばれる役だし、ここを安くできれば効きそうだと思ったので。

OpusのCriticが素通りさせたバグを、まったく同じコードをFableにレビューさせたら見つけた。

n=1の話なので鵜呑みにはできない。  
でも方向としては腑に落ちる。  
深いモデルのレビューが強いのは、コードを読みながら「なんか引っかかる」と気づく瞬間で、事前にチェック項目を並べる段階じゃない。  
既存コードとの噛み合わせとか、暗黙に守られてる前提が崩れてるとか、そういう「言われないと気づかない」やつ。

つまりCriticを安くすると、いちばん拾ってほしいバグから先に取りこぼす。  
ここは落とせない。

そうすると「重いのは分かってるけどFableのままにしたい役」が残る。  
で、Fableのまま回したら、今度はトークンが爆発した。

## 何が爆発していたか、そして外した2案

検証ゲートは当初こう動いていた。  
変更ファイルを全部Readして、テストやコンパイラなどの出力を丸ごと飲み込んで判定する。  
これがFableのコンテキストを直撃していた。

削り方を2つ試して、どっちもダメだった。

### 「git diffだけ渡す」は新規開発だと効かない

最初の本命はこれ。  
全文じゃなく`git diff`だけ渡せば軽くなるはず。  
既存の大きいファイルを数行いじるだけなら、差分は全文よりずっと小さい。

でも自分の仕事はほぼ新規開発で、新しく作ったファイルは`git diff`でも全行が`+`で出る。  
つまり差分イコール全文、何も減らない。  
新規コードはそもそも全部読まないとレビューできないので、読む量に下限がある。  
diff作戦は既存改修には効くけど、自分のケースには効かなかった。

### 「書き手の一段上でレビュー」も筋が悪い

次に考えたのが、Sonnetが書いたらOpus、Opusが書いたらFable、と一段上のモデルにレビューさせる案。  
きれいに見えた。

でもさっきのバグの件を思い出すと崩れる。  
あれは「できあがったコードをOpusとFableに読ませ比べた」話で、誰が書いたかは関係ない、レビュアー単体の実力差だった。  
だから「書き手の難しさでレビュアーを決める」のは軸がずれている。  
しかも書き手はだいたいSonnetなので、一段上＝Opusレビューが多数派になる。  
Opusは見逃す実績があるのに、それを主戦力に据えることになってしまう。

なので書き手基準はやめた。  
将来レビューモデルを振り分けるとしても、軸は書き手じゃなく「ロジックの濃さ × 壊れたときの被害」だろう。  
パーサや認証や整合性みたいな急所は上位モデル、設定やボイラープレートは安いモデル。  
ここは今回は入れず、頭の隅に置いてある。

## 結局、モデルじゃなく入力の方を分けた

行き止まりで、そもそも何が重いのかを考え直した。  
爆発してたのは本当にコードか？どうも違う。  
ビルドがコケるとコンパイラの連鎖エラーやスタックトレースが延々出て、コードの何倍もの分量になる。  
で、この出力を読むのにFableの頭はいらない。

正直に書いておくと、この「出力がコードの何倍」は設計相談で聞いた見積もりで、自分の実測じゃない。  
本当は爆発した1セッションの内訳（Readした量vsテスト出力の量）を数えて主因を確かめるべきで、そこはまだ宿題。  
だからこの3段化も「効くはず」の設計で、どれだけ減ったかの数字はこれから測る段階だ。

判断のいらない重い入力（生ログ）は安いモデルに畳ませて、判断のいるコードだけをFableに読ませる。  
そのためにゲートを3段に割った。

```
Stage1  Worker（sonnet / 複雑なやつだけ opus）
          実装する

Stage2  収集役（sonnet・判断はしない）      ← 節約の肝
          テスト・tsc・ビルドを走らせて、
          生ログを「exit code ＋ 失敗箇所の抜粋」に畳む

Stage3  Critic（fable）
          コードは自分で読む（ここは減らせない）
          テストは再実行せず Stage2 の結果を使う
          合否を出す
```

コードを読む工程だけはFableに残す。  
ここを剥がすと最初の「見逃す」問題が戻ってくるので、削れない。  
削るのは生ログの方。  
ログを畳むのは判断ゼロの単純作業だから、Sonnetで十分。  
ひとつだけ気をつけているのは、exit codeは生の値のまま通すこと。  
数値は要約でごまかせないので、Sonnetが「だいたい成功」みたいに丸める事故を防げる。

あと粒度の背圧も入れた。  
1タスクの新規・変更行が目安600行を超えたら、Fableを呼ぶ前に「これは大きすぎるから分割して」と手前に突き返す。  
読む量はどう頑張っても粒度でしか下げられないので、でかい塊は計画側に戻す。

## 一般化するとこうなる

今回で言葉にできたのは、だいたい次のあたり。

モデルを落とす前に「落として何を失うか」を確かめる。  
Workerは落として平気だったけど、Criticは落とすと見逃した。  
役によって答えが違う、というのが実感。

落とせない役には、別の削り方がいる。  
テスト実行やログ整形みたいな判断のいらない部分を安いモデルに切り出して、高いモデルには判断に必要なものだけ渡す。  
生のビルドログを最上位モデルに直接流さない、はけっこう効くはず。  
ログはコードより長いことが多いので。

それでも残る「コードそのもの」の量は、タスクを小さく割るしかない。

このへんはあまり書かれてないけど、モデルの割り当てより「何を渡すか」を詰める方が、実際には効いてる手応えがある。

## 締め：この節約設計を、Fable 自身にレビューさせた

いちばん書きたかったのはここ。  
この一連の設計を、当のFableにレビューさせた。

まず設計相談。  
最初に持っていったのは「Fableが検証プランを作って、安いモデルが実際に検証して、Fableが最後に判定する」という別の3段案だった。  
これが却下された。  
返ってきた理由を、要旨で引く。

> Opus が集めた“証拠”には、Opus が見逃したバグの痕跡がそもそも写っていない。だから最後に Fable がその証拠を見ても、欠陥の写っていない資料を見て「合格」を出すだけ。守りたかった Fable の強みを、構造の方で殺している。

このときFableが代わりに勧めてきたのが「diffだけ渡す2段」、つまりこの記事の行き止まり①だった。  
ところが後日「自分は新規開発が中心なんだ」と伝えて相談し直したら、Fableは自分のその案を引っ込めた。  
新規ファイルはdiffがほぼ全文だから節約にならない、と。  
前提が変われば最上位モデルでも前言を撤回する。  
Fableに訊けば正解が出る、わけではない。  
2回目のやりとりでようやく「コードを読む工程はFableに残す、削るのは生ログ」に落ち着いた。

実装が終わってからは、同じFableに検証ゲート役として敵対的にレビューさせた。  
判定はREQUEST\_CHANGES。  
ドキュメントで約束した挙動が一部そもそも実装できていない、みたいな見落としを4件突かれて、全部直した。  
どれも例外的な状況でしか発火しないやつで、普通に動かしているだけでは絶対に見つからなかった。

最後に本物のパイプラインを1回だけ走らせて、Worker（隔離）→収集役→Criticの受け渡しが通ることを確認した。

結局やったのは、Fableを要所にだけ置くこと。  
その一行にたどり着くのに、ずいぶん遠回りした。

## まとめ

* 「モデルを階層化して安く」は正しい。Worker を安いモデルにするだけでも効く。
* ただし落とすと壊れる役がある。自分の場合は Critic。そこはモデルじゃなく入力を階層化して、安いモデルに生ログを畳ませ、高いモデルにはコードと最小の証拠だけ渡す。
* 「どのモデルを当てるか」の一段下に「何をどれだけ渡すか」がある。
* この3段化はまだ設計を通しただけで、どれだけ削れたかは測っていない。数字が出たら追記する。
* そして、その節約設計を最上位モデル自身にレビューさせると、例外系の穴まで出てきた。高いモデルは、使い所を絞ればちゃんと元が取れる。
