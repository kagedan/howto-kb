## はじめに

Claude Code（Anthropic公式CLI）を使い込んでいくと、ルールやスキルの設定ファイルがどんどん肥大化していきます。

私の環境では rules/ が747行、skills/ が2,563行まで膨らんでいました。これらは会話のたびにコンテキストウィンドウに読み込まれるため、肝心の作業に使えるコンテキストが圧迫されます。

この記事では、ハーネス設定を「インデックス化」することで、常時ロードされる rules/ を **747行→173行（77%削減）**、スキル呼び出し時のロード量を **平均160行→44行（73%削減）** に縮小した方法を紹介します。

## 前提：Claude Codeのハーネス構成

Claude Codeには、エージェントの振る舞いを制御する仕組み（ハーネス）がいくつかあります。

| 配置先 | 役割 | ロードタイミング |
| --- | --- | --- |
| `rules/` | 原則・禁止事項 | **常時ロード** |
| `skills/` | 作業手順・チェックリスト | **スキル呼び出し時** |
| `agents/` | ロール定義 | エージェント起動時 |
| `hooks/` | 機械的強制 | 自動実行 |
| `CLAUDE.md` | エントリーポイント | **常時ロード** |
| `memory/` | 経験・教訓 | 常時ロード（indexのみ） |

ポイントは **ロードタイミング** です。`rules/` と `CLAUDE.md` は毎回の会話で必ずコンテキストに読み込まれます。ここが大きいと、それだけで作業用のコンテキストが減ります。

## 問題：全部読ませていた

整理前の私の環境はこうでした。

### rules/ に手順・フォーマットが混入（747行）

```
rules/test_process.md（247行）
  ├── 絶対禁止事項（原則）← ここだけがrulesの役割
  ├── C1〜C4チェック手順 ← これはskills向き
  ├── DA1〜DA4監査手順   ← これもskills向き
  └── エビデンス確認フォーマット ← これもskills向き
```

原則・禁止事項だけなら42行で済むのに、手順やフォーマットが混入して247行になっていました。これが **毎回の会話で常時ロード** されていたのです。

### skills/ にコード例・チェックリストが全部入り（2,563行）

```
skills/security/SKILL.md（186行）
  ├── 原則・禁止事項（手順）← ここだけが呼び出し時に必要
  ├── .envの設定例          ← 必要な時だけ見ればいい
  ├── NGコード/OKコード例   ← 必要な時だけ見ればいい
  └── レビューチェックリスト ← 必要な時だけ見ればいい
```

スキルは呼び出し時にSKILL.mdの全文がコンテキストにロードされます。手順だけなら52行で済むのに、コード例やチェックリストまで毎回ロードしていました。

## 解決策：インデックス化

やったことは単純です。**「全部読ませる」から「目次だけ読ませて、詳細は必要な時だけ辿る」に変えました。**

### 1. rules/ → 原則・禁止事項だけに（747行→173行）

手順・フォーマット・チェック手順をすべて skills/ に移行し、rules/ には「〜してはならない」「〜を守る」だけを残しました。

**Before（247行の一部）:**

```
## 成果物整合チェック（C1〜C4）

### C1: テスト一覧 ↔ ケース詳細
- テスト一覧の全IDがケース詳細に存在するか確認
- ケース詳細のIDが全てテスト一覧に記載されているか確認
...（以下200行の手順が続く）
```

**After（6行）:**

```
## テスト成果物整合・監査（参照先）

- 成果物整合チェック（C1〜C4）の具体手順 → test-process-quality スキル を参照
- 監査エージェントのロール定義 → agents/role-auditor.md を参照
- 禁止事項 → test_process.md を参照
```

### 2. skills/ → SKILL.md + reference.md に分割

実は、Claude Codeの skills/ にはSKILL.md（スキル呼び出し時にロード）の他に **reference.md**（リンクで辿った時だけロード）を置ける仕組みが公式に用意されています。

ただし、**エージェント（Claude）自身はこの仕組みに気づいていませんでした。** 私が「referenceとかさ、その全貌と現状の差を知りたいな」と聞いて調査させた結果、初めてreference.mdの存在が判明しました。自分たちが提供している機能を把握できていなかったわけです。

この仕組みを使って分割しました。

**SKILL.md（手順・原則のみ、呼び出し時にロード）:**

```
# セキュリティ原則

> **このファイルは手順・原則のみ。詳細はreference.mdに書く。**

## 絶対禁止事項
- ソースコード・設定ファイルへの認証情報の直書き
- ログに認証情報・個人情報を出力すること
...

## 詳細（コード例・チェックリスト）
→ reference.md を参照
```

**reference.md（詳細・コード例、必要時のみロード）:**

```
## コード例（NG/OK）

# NG: 絶対にやってはいけない
SECRET_KEY = 'django-insecure-xxxxxxxxxxxx'

# OK: 必ずこの形式にする
SECRET_KEY = os.environ['DJANGO_SECRET_KEY']

## コミット前セキュリティレビュー チェックリスト
- [ ] SECRET_KEY・パスワード・APIキーが含まれていないか
...
```

### 3. 配置基準を5W1Hで確定

どこに何を書くか迷わないよう、配置基準を確定しました。

| 書きたいこと | 配置先 | 対応する問い |
| --- | --- | --- |
| 原則・禁止事項 | rules/ | Why・What |
| 作業手順 | skills/SKILL.md | How |
| コード例・チェックリスト | skills/reference.md | How（詳細） |
| ロール定義 | agents/ | Who |
| 機械的強制 | hooks/ | When |

## 結果

### コンテキスト消費の変化

| 項目 | Before | After | 削減率 |
| --- | --- | --- | --- |
| rules/（常時ロード） | 747行 | 173行 | **77%** |
| skills/（呼び出し時） | 平均160行/スキル | 平均44行/スキル | **73%** |
| reference.md | なし | 平均114行/スキル | 必要時のみ |

### 数字の裏側

* **rules/ 747行→173行**: 574行分のコンテキストが **毎回の会話で** 節約される
* **skills/ 平均160行→44行**: スキル1回の呼び出しで116行分の節約
* **reference.md 平均114行**: 詳細が必要な場面でのみロードされるため、普段はゼロ

合計で、ハーネス設定に使われるコンテキスト量が大幅に削減されました。内容の欠落はありません（全行を機械的に突き合わせて確認済み）。

## 肥大化を防ぐ仕掛け（skills/のみ必要）

インデックス化しても、将来の更新で詳細が追記されれば元に戻ってしまいます。

ただし、**肥大化防止が必要なのは skills/SKILL.md だけ**です。rules/ は常時ロードされるので、肥大化すれば自分（ユーザー）が直接困る。そのため自然と気をつけられます。また rules/ に書く内容は「原則・禁止事項」だけなので、そもそも量が増えにくいです。

一方 skills/SKILL.md は呼び出し時にだけロードされるため、肥大化の痛みが見えにくい。エージェントは「詳細も書いておいた方が便利だろう」と善意で追記し、気づいたら160行に戻っている――というのが実際に起きたことです。

Claude Code にはファイルへの追記を仕組みとして制御する機能がないため、全21スキルのSKILL.md冒頭に役割を明記しました。

```
> **このファイルは手順・原則のみ。詳細・コード例・チェックリストは reference.md に書く。**
```

これにより、次のセッションのエージェントがSKILL.mdを更新しようとした際に「ここには詳細を書かない」と認識できます。

## 補足：内容が消えていないことの確認方法

大量のファイルを分割・移動すると「知らないうちにルールが消えていた」というリスクがあります。

分割前の各行をソートし、分割後（SKILL.md + reference.md）の合算と `comm -23` で比較することで、全スキルについて欠落がないことを機械的に確認しました。

```
# 分割前の内容を抽出
git show <分割前コミット>:skills/security/SKILL.md | sort -u > /tmp/before.txt

# 分割後のSKILL.md + reference.md を合算
cat skills/security/SKILL.md skills/security/reference.md | sort -u > /tmp/after.txt

# 分割前にあって分割後にない行を検出
comm -23 /tmp/before.txt /tmp/after.txt
# → 出力なし = 欠落なし
```

## まとめ

* Claude Codeのハーネス設定は、使い込むほど肥大化する
* 肥大化した設定は **毎回の会話でコンテキストを圧迫** する
* **「全部読ませる」→「目次だけ読ませて、詳細はリンクで辿る」** に変えるだけで大幅に削減できる
* rules/ は原則・禁止事項だけ、skills/SKILL.md は手順だけ、詳細は reference.md に分離
* 分割後は必ず「内容が消えていないか」を機械的に確認する
* SKILL.md冒頭に役割を明記して、将来の肥大化を防ぐ

ハーネス設定を「書く」ことに集中しがちですが、**「どう読まれるか」を意識して配置する**のが、コンテキストを有効に使うコツだと思います。
