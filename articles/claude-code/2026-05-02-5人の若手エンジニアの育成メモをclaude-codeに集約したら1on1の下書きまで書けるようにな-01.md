## 散らばっていた育成情報

5人の若手エンジニアを見ていると、情報がすぐに散らばります。日々の観察メモは手元、週次のKPT（ふりかえり）はGoogle Docs、日報は社内SNS、育成会議のメモはGoogleスプレッドシート ── 「この人に今どんなアドバイスをすべきか」を考えようとすると、記憶を辿りながら4つのツールを開いて回ることになります。

少し背景を補足します。私の会社（ソニックガーデン）には「徒弟制度」があります。宮大工の師弟関係をモデルにした育成制度で、若手エンジニアを「弟子」、その指導役を「親方」と呼びます。親方は弟子のコードを見て、仕事のやり方を観察し、フィードバックを繰り返します。

親方の仕事は「聴く」より「観る」です。弟子の成長段階を把握し、ちょうど頑張ればできるレベルの仕事を渡し、些細なやり方の違いも指摘する。一貫した判断軸を持ち、弟子が頭の中に「リトル親方」── 親方の判断軸の縮小版 ── をインストールできるようにします。

5人の弟子を「観る」ためのデータが散らばっていては、判断軸の一貫性も保てません。そこでClaude Codeを使い始めました。コードを書くためではなく、**散らばった育成情報を集約してアドバイスを考えるため**です。

## 集約と取込の仕組み

Markdownファイルで弟子ごとの情報を管理し、Claude Codeに「業務マニュアル」を渡して、記録・取込・要約・執筆を任せています。

### 情報の構造

```
team-notes/
├── members/           # 日常の観察メモ（親方が気づいたことを随時記録）
├── 1on1/              # ふりかえり（1on1）の記録
├── daily/{名前}/      # 日報記録（四半期ごとにファイル分割）
├── knowledge/         # 会社の文化・制度の前提知識
└── CLAUDE.md          # Claude Codeへの指示書
```

弟子ひとりにつき、観察メモ・ふりかえり記録・日報の3種類のファイルがあります。これが弟子を「観る」ためのデータベースになります。

### 日常の記録

何か気づいたら、Claude Codeに話しかけるだけです。

```
「Aさんの今日の様子を記録して：PRレビューが丁寧になってきた」
「Bさんについて気になったことを記録して：タスクバラシ（タスクの細分化）が雑になっている」
```

Claude Codeが日付と種別タグをつけて、該当メンバーのファイルに追記します。手帳にメモする感覚と同じですが、あとから検索・集約できるところが異なります。

### 外部からの取込

冒頭で挙げた外部ツールから、それぞれスクリプト経由でリポジトリに取り込みます。

```
「ふりかえりを取り込んで」  → Google DocsからKPTを取得・整理
「日報を取り込んで」        → 社内SNSのAPIから日報を取得・整理
「育成会議メモを取り込んで」 → スプレッドシートから育成会議メモを取得
```

取込後の整理もClaude Codeが行います。KPTのポイントを要約したり、日報を四半期ファイルに振り分けたり。「取り込んで」の一言で、散らばっていた情報がひとつのリポジトリに集まります。

## 下書きまで書かせる

情報が集まったら、本題の「弟子へのアドバイスを考える」に入ります。振り返りや育成会議メモの下書きまでClaude Codeに書かせて、自分は事実確認と微調整だけ担当します。

### 育成の振り返り

Claude Codeが以下のファイルを読み込んで、振り返りの下書きを作ります。

* 日常の観察メモ
* ふりかえり（1on1）の記録
* 日報
* 振り返りの書き方ガイド

出力は決まった構成に沿います。

1. **最近の目標** — 本人が掲げている目標
2. **課題感** — なぜうまくいっていないか、深掘り
3. **良かったこと** — 具体的なエピソード
4. **いまいち** — うまくいかなかったこと + 親方側の反省
5. **所感** — パフォーマンス段階、自立度、量/質フェーズの評価
6. **次のアクション** — 「じゃあどうする？」

ここで重要なのが、`knowledge/` ディレクトリの存在です。

### 前提知識を共有する

`knowledge/` には、会社の文化や制度を言語化したドキュメントを置いています。

* **徒弟制度**：観察を重んじる、判断軸の一貫性、厳密に指摘する、といった親方の心得
* **成長の原則**：まず量をこなす（量から質へ）、仕事を小さく切って渡す、フロー状態をつくる
* **ふりかえりの考え方**：正解を教えない、内発的動機を育てる
* **パフォーマンスの段階**：自立度に応じた「ベテラン／メンバー／アシスタント」の段階定義

Claude Codeはこれらを参照しながら振り返りを書くので、「この会社ではこう考える」という文脈を踏まえた出力になります。たとえば「この弟子は今レベル2（設計レビュー＋レビュー指摘1往復が必要）で、量のフェーズ」といった評価が、社内の共通言語で出てきます。

### 育成会議メモの下書き

親方同士が集まって弟子の様子を共有する育成会議があり、そこに提出するメモもClaude Codeに下書きを頼んでいます。

CLAUDE.mdには、自分の文体ルールまで定義してあります。

CLAUDE.md

```
- 句読点は使わない。文の区切りは改行にする
- 口語的で短い文。メモ書きのような自然体
- 箇条書きは使わない。文章で書く
- 「まぁ」「なぁ」「〜のでよい」など口語表現を使う
- 300文字程度に収める
```

出力例はこのような形です。

```
Cさん
ちょっとマンネリしているのか仕事が完璧にこなせているわけではないのに
緊張感や工夫が足らないような感じがある
何か新しい大きめの仕事や刺激があったほうがいいのか
```

自分が書いたような文章が出てくるので、あとは事実確認と微調整をするだけです。ゼロから書こうとすると手が止まりがちですが、下書きがあれば「ここは違う、こっちのほうが近い」と修正ベースで考えられます。

## 任せているのは「思考の下ごしらえ」

補足しておくと、アドバイスの判断そのものをClaude Codeに任せているわけではありません。任せているのは**思考の下ごしらえ**です。

* 散らばった情報を1箇所に集める（取込）
* 時系列で変化を整理する（要約）
* 社内の評価軸に沿って構造化する（振り返り）
* 自分の言葉で下書きを作る（執筆）

これらが揃っていると、「この弟子に今何を伝えるべきか」を考えることに集中できます。情報を探す・整理する・文章を起こすといった手間がなくなった分、**観察と対話そのもの**に時間を使えるようになりました。

親方の仕事は「観る」ことです。観るためには、観たものを記録し、蓄積し、振り返る仕組みが必要です。Claude Codeはその仕組みの部分を担ってくれています。

## 気をつけていること

### AIの出力を鵜呑みにしない

振り返りや育成会議メモの下書きは、あくまで下書きです。Claude Codeは記録の行間を埋めようとして憶測で補ってくることがあります（言ってないニュアンスを足したり、推測の評価語を盛ったり）。「本当にそう言ってたっけ？」を自分の記憶や元の記録で確認してから使います。事実誤認を含んだまま会議に持っていくと、それこそ弟子に失礼になります。

### 「観る」は代替できない

Claude Codeに任せているのは観察そのものではなく、観察した記録の整理です。日々のコードレビュー、隣で仕事している様子、雑談の中で出てきた違和感、こうしたものは引き続き親方が自分の目で「観る」必要があります。Claude Codeはその観察結果を蓄積・整形するパートだけを担います。

## 仕組みのポイント

同じようなことをやってみたい方に向けて、仕組み面のポイントをまとめておきます。

### CLAUDE.mdに「業務マニュアル」を書く

CLAUDE.mdには以下を定義しています。

* メンバー一覧とファイルパスの対応
* 記録フォーマット（日付・種別タグ）
* 各取込フローの手順
* 執筆時の文体ルール
* 会社の前提知識への参照

Claude Codeは毎回CLAUDE.mdを読むので、プロジェクト固有の文脈を常に把握しています。毎回説明し直す必要がありません。

### Markdownでシンプルに

特別なDBやSaaSは使っていません。Markdownファイル＋Gitというシンプルな構成です。Claude Codeが直接読み書きできますし、エディタでも普通に開けます。気になったときにさっと見返せるのも利点です。

### knowledgeディレクトリで暗黙知を言語化する

「この会社ではこう考える」をドキュメントにしておくと、Claude Codeの出力が文脈を踏まえたものになります。これは新しいメンバーに会社の文化を伝えるのと同じで、言語化しておくこと自体にも価値があります。

### 外部サービスとはスクリプトでつなぐ

Google Docs・スプレッドシートは CDP（Chrome DevTools Protocol）経由、社内SNSはAPI経由で取得しています。

CDP方式は、`--remote-debugging-port=9222` で起動したブラウザに対して `fetch()` でCSVエクスポートURLを叩くだけのシンプルな仕組みです。ポイントは、**ブラウザでログイン済みであれば、その権限をそのままスクリプトから利用できる**こと。APIキーやサービスアカウントの発行・管理は不要で、Googleの認証フローも考えなくていい。社内ツールでよくある「公式APIはあるけど認証が面倒」「APIが用意されていない」ケースで特に効きます。

社内SNSはAPIトークンを `.env` で管理して、Pythonスクリプトから叩いています。`scripts/` ディレクトリに置いたスクリプトを Claude Code が必要に応じて実行する形です。

## おわりに

Claude Codeを「コードを書くツール」としてだけ見ると、こういう使い方は思いつきにくいかもしれません。ただ、実態としては「CLAUDE.mdを読んでターミナル上で何でもやってくれるアシスタント」なので、マネジメント業務にも自然に使えます。

5人の弟子の情報を常に頭に入れておくのは難しいですが、Claude Codeは毎回すべて読んでくれます。そのうえで会社の文化や評価軸も踏まえて整理してくれます。

親方として「この弟子に今なにを伝えるか」を考える時間が、一番価値のある時間です。散らばっていた情報を集める手間がなくなった分、5人の弟子と向き合う時間そのものに使えるようになりました。
