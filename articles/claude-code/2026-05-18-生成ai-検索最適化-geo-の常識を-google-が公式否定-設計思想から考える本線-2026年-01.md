「**生成AI 検索 (Google の AI Overview / AI Mode など) のために llms.txt を作るべき**」「コンテンツは Chunking して短くしないと AI に拾われない」 — こうした **GEO (Generative Engine Optimization = 生成AI 検索向け最適化)** 業界の「正解」 とされる手法を、Google が2026年に公式に否定しました。

本記事では Google が否定した5 myth を「なぜ Google は否定したのか」 という設計思想レベルから整理し、agentic experiences を含む2026年以降の本線を深掘りします。

## 設計思想の核 ： AI Overview は RAG + Query fan-out で動く

Google の生成 AI 機能 (AI Overview / AI Mode) は別 ranking system ではなく、**既存 Search ranking の上に乗っている**。これが Google ガイドの最大の整理です。

具体的には次の2仕組みで動きます。

* **RAG (Retrieval-augmented generation)**: Search index から関連 page を取得 → grounding (= 引用元として保持) → 生成 AI が回答を作成 + clickable citation を提示
* **Query fan-out**: 1ユーザークエリから関連クエリを並列展開 (例: 「how to fix a lawn full of weeds」 → 「best herbicides for lawns」「remove weeds without chemicals」「how to prevent weeds in lawn」)

つまり「Google 検索で上位に来るページ」 と「AI Overview に引用されるページ」 は基本的に同じ条件で評価される。**GEO ≠ 別最適化**・**GEO ⊂ SEO** という構造です。

この設計思想が分かれば、業界 myth の多くが「なぜ Google が否定したか」 が見えてきます。

## 1. Google が否定した GEO myth 5件 ： 設計思想ベースの解釈

### myth 1： llms.txt や AI 向け専用ファイル

Google 公式 verdict (英語原文):

> "LLMS.txt files and other special markup: You don't need to create new machine readable files, AI text files, markup, or Markdown to appear in generative AI search."

**設計思想ベースの理由**: AI Overview は RAG で「Search index 上の通常 page」 を引用する。llms.txt 専用 file は Search index にも入らないし、AI 機能の retrieval pipeline からも参照されない。**専用 channel を別に作る発想自体が Google の architecture と合っていない**。

他 AI (Anthropic / OpenAI / Perplexity) は retrieval architecture が異なるので、それぞれの crawler 仕様に従う別議論。

### myth 2： Chunking (細切れ化)

> "Chunking content: There's no requirement to break your content into tiny pieces for AI to better understand it. Google systems are able to understand the nuance of multiple topics on a page."

**設計思想ベースの理由**: RAG の retrieval 単位は「page 全体」 ではなく「page 内の relevant section」。Google のシステムは multi-topic page から関連部分を切り出して LLM に渡す snippet 化を内部で実行。つまり **「人間が事前に chunking する」 = システムが内部でやっている処理の二度手間**。

しかも1トピック1ページに分けると、Google 側で「同一トピックの page が大量に増えた」 と認識して duplicate content / scaled content abuse 評価に振れるリスクすらある。

### myth 3： AI 専用の書き換え

> "Rewriting content just for AI systems: You don't need to write in a specific way just for generative AI search. AI systems can understand synonyms and general meanings."

**設計思想ベースの理由**: LLM は意図/synonym/general meaning を理解する。「AI 向け文体」 と「人間向け文体」 を分ける前提が成り立たない。むしろ「人間にとって読みやすい page」 = page experience score が上がる = 全体的に上位表示しやすくなる。

long-tail KW スタッフィングも同じ理由で不要。意図マッチングは model 側で処理される。

### myth 4： Inauthentic mentions

> "seeking inauthentic mentions across the web isn't as helpful as it might seem."

**設計思想ベースの理由**: Google の core ranking は **content quality** を見る・spam system は **mention の authenticity** を見る。組織的に仕込んだ mention は spam system の filter で除外される。さらに recent 動向では「mention の文脈 (なぜそれが言及されたか)」 まで model 側で評価される傾向。

自然な engagement (Reddit 議論・technical blog 引用) は別軸として依然価値あり。

### myth 5： 構造化データへの過信

> "Overfocusing on structured data: Structured data isn't required for generative AI search, and there's no special schema.org markup you need to add."

**設計思想ベースの理由**: schema.org は **rich result (検索結果の拡張表示)** を狙う場合のシグナル。生成 AI 検索の retrieval pipeline は schema.org を必須としていない。**「rich result 用の通常 schema.org」 はそのまま OK・GEO 用に special markup を盛る必要なし** という整理。

## 2. GEO の本線 ： 設計思想ベースで何をするか

Google が「やるべき」 と明示した3軸を設計思想で読み解くと:

### non-commodity content

「7 Tips for First-Time Homebuyers」 (commodity・誰でも書ける) ではなく「Why We Waived the Inspection & Saved Money」 (non-commodity・1次経験) を書く。

設計思想: RAG の grounding step は **「他に同じ情報源があるか」** で重み付けされる。commodity 情報は他に1000個ある = 引用優位性低い。non-commodity は他にない情報 = 引用優位性高い。

### technical clarity

indexable + crawlable + snippet eligible + good page experience (Core Web Vitals + モバイル対応 + main content の明示性)。

設計思想: AI Overview に出るには **まず Search index に入る + snippet eligible** が前提。技術 SEO の標準を満たさない page は AI 機能の retrieval pipeline 自体に乗らない。

### scaled content abuse 回避

「ユーザーが検索する全 variation 用に別 page を作る」 NG。

設計思想: variation 量産は **structural completeness (Pricing / FAQ / Brand を1度埋める = 1 page) と別軸**。RS の「空白を作らない」 原則 (Pricing / FAQ / Brand / Persona の埋め込み) は structural completeness で OK・variation 用 page 量産は NG。

## 3. 新規軸 ： agentic experiences の設計思想

Google ガイド末尾の **agentic experiences** は、Search 外の AI agent 経済への伏線です。

```
従来:    人間 → Google Search → 人間が click → 人間が page を読む
新規軸:  人間 → AI agent → agent が複数 site を navigate → agent が予約/購入/情報統合 → 人間に summary 返す
```

この変化で必要になるのが:

* **DOM 構造 + accessibility tree の機械可読性**: 人間向け visual だけでなく semantic HTML / ARIA labels で agent が理解できる構造
* **Universal Commerce Protocol (UCP) 対応**: 商品データ・在庫・価格・配送条件を protocol 経由で agent に返す API 設計
* **MCP server 公開**: 自社データを生成 AI が query できる接続点

RS は自社 MCP server を公開しており、agent 経済への準備を進めています。Google 公式が言及した web.dev/articles/ai-agent-site-ux は browser agent 向けの best practice resource として参照価値あり。

**2026後半以降の GEO 戦場の分岐**:

* 人間向け SEO + AI Overview 引用 (本記事の myth 1-5 + 本線3軸の話)
* AI agent 向け navigation + commerce protocol 対応 (agentic experiences 軸)

両方を視野に入れた website 設計が、3ヶ月後・半年後の差を作ります。

## まとめ

Google 公式の設計思想ベースで GEO を整理すると、業界 hack の多くは「architecture と合っていない」 から否定された側に振れる。RAG + Query fan-out が AI Overview の根幹である以上、SEO 本線をきちんとやることが GEO 対策そのもので、その上で agentic experiences の新規軸が次の6-12ヶ月の戦場になります。
