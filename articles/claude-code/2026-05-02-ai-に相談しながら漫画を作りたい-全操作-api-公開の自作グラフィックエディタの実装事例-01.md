## はじめに — AI に相談しながら作るには

グラフィック制作ツールで AI を使うというと、Photoshop の **マジックエディット** や iPhone 写真の **マジック消しゴム** のような、**ツール側のボタンを押すと AI が一発で何かをやってくれる** 機能が思い浮かぶ。それも便利だが、実際の制作で本当に価値があるのは、それより手前の **「これを実現するにはどんな処理をしたらいい?」を AI に聞きながら、一緒に進めていく** 対話的な使い方のほうだと思う。

文章を書く、コードを書く、データを整理する——テキスト系の作業はもうこの形で進められる。AI チャットに「ここの構成どう思う?」「この段落、別の言い方ない?」と聞きながら、自分の手を動かす。

漫画作りも同じことができたらいい、と私は思っていた。「このコマ、もう少し引いた構図にしたい。どう変えれば伝わる?」「3 ページ目から 5 ページ目の流れ、テンポが悪い気がする。どこを変えたらいい?」——そんな相談を AI とできれば、制作はずっと楽しい。

ところがグラフィックエディタは Photoshop を除いて GUI 専用で、AI が **「いまどうなっているか」を読めない・触れない**。状態が見えなければ提案もできない。AI が制作のパートナーになるためには、**ツールの状態と操作のすべてが AI から API で見える/触れる** 必要がある。

そこを埋めるために自作したのが **manginus**（"manga" + "engine"）。漫画特化のグラフィックエディタで、**全操作を API として公開し、MCP 経由で AI から触れる** 形にしてある。

技術スタックは FastAPI / React / WebSocket / ComfyUI / PIL / Playwright / Blender Python / MCP——名前が並ぶと多そうに見えるが、どれも **既存の枯れた技術** で、新しさはない。あるのは **何を AI に公開するか・どう公開するか** の組み合わせだけ。本記事はその実装事例。各技術の役割は本文で順に説明していく。機能側の話（コマ＝マスクという発想・LINE スタンプ・動画化）は別途 [yatmita.com の labo 連載](https://yatmita.com/labo/posts/category/ai-manga) にある。

> **本記事の前提知識**: HTTP / REST API / Python / JavaScript の基本がわかれば読める想定。AI 連携系（MCP・ComfyUI・画像生成 API など）は登場時に簡単に説明する。

最初は既存の OSS マンガエディタを **MCP（Model Context Protocol：AI エージェントに外部ツールを使わせるためのプロトコル。Anthropic が提唱、JSON-RPC ベースの薄い仕様）** で操作できるようにしようとした。12 個の MCP ツールでラップしたが、もとのエディタの内部に外から触れる API がない構造だったので挫折した。**API ファーストで作り直す** ほうが早いと判断した、というのが自作の経緯になる。

なお、本記事は **manginus（漫画を組み上げるエディタ）** の話に絞る。シナリオやコマ割りの設計（ネーム）は別系統の **「nemu-kaigi」** という自作エンジンでやっていて、複数の AI モデルにネームを並行生成させて投票で選ぶ仕組み。これは **別の機会に記事にする予定** （概要は [yatmita の labo 連載](https://yatmita.com/labo/posts/category/ai-manga) にある）。manginus は **nemu-kaigi が出力したネームを食べて、絵にする** ツール、という関係になっている。

## 設計の核 — 全操作 API / サーバー駆動 / UI state も API

manginus の設計判断は 3 つに集約される。

### 1. データの真実はサーバーが持つ

ブラウザの中（JavaScript の変数）にデータを置く設計だと、外から触るには「ブラウザを開いて DOM を操作する」しか方法がない。manginus は逆で、サーバー側にデータの本体を置く。

* **FastAPI**（Python の Web フレームワーク。型ヒントから自動でバリデーションとドキュメントが生成される）でサーバーを立てる
* **Pydantic**（Python のデータモデリングライブラリ。`BaseModel` を継承するとそのクラスが JSON のスキーマになる）でデータの形を定義する
* ブラウザは WebSocket（後述）で「サーバーで起きたこと」を購読して、画面を更新するだけ

こうすると **AI から見ると「サーバーに REST で問い合わせれば状態がわかる」** 形になる。AI は GUI を操作しなくていい。

```
from enum import Enum
from pydantic import BaseModel, Field

class NodeType(str, Enum):
    PAGE = "page"
    PANEL = "panel"
    IMAGE = "image"
    TEXT = "text"
    SPEECH_BUBBLE = "speech_bubble"
    EFFECT = "effect"
    GROUP = "group"
    COLUMN = "column"

class Transform(BaseModel):
    """座標は親に対する相対値 (0〜1)、scale/rotation/opacity は絶対値"""
    x: float = 0
    y: float = 0
    width: float = 0
    height: float = 0
    rotation: float = 0
    scale_x: float = 1.0
    scale_y: float = 1.0
    opacity: float = 1.0

class Node(BaseModel):
    id: str
    type: NodeType
    transform: Transform = Field(default_factory=Transform)
    children: list["Node"] = []
    properties: dict = {}  # type 別の追加情報（image src, bubble shape, text body 等）
```

ノードは派生クラスで分けず、**単一の `Node` クラス + `NodeType` enum + `properties: dict`** という緩い構造にしている。型ごとに厳格に分けるよりも、**dict で柔軟に持つほうがツリー操作（追加・削除・再帰探索）が単純** に書けるからだ。

座標を「親からの相対値（0〜1）」で持つのは普通の手法だが、AI 連携の文脈で効いた。AI が「このコマを少し左に詰めて」と指示するときに `transform.x -= 0.05` のような **相対操作で意味が通る**。絶対座標だと「ページの寸法を考慮して計算する」必要があって、AI が間違えやすい。

### 2. すべての操作は REST API

ブラウザを開かなくてもマンガが完成する設計（ヘッドレス）。コマ追加、画像生成、エクスポートまで全部 REST API として用意する。

```
POST   /api/pages/{id}/panels      コマを追加
PATCH  /api/nodes/{id}             ノードの一部を更新
POST   /api/images/generate        画像を生成
POST   /api/export/pdf             PDF にエクスポート
```

API の粒度は「**AI エージェント（Claude Code 等）がコマンド 1 行で操作できる単位**」に揃える。「ページを追加してコマを 5 個並べる」のような複合操作は、サーバー側に専用エンドポイントを作らずに、AI が個別 API を順番に叩いて再現する粒度を残す。**AI が自分で工夫して使い回す余地** を残すのがコツ。

### 3. UI state までサーバー側で API 化する

普通なら **React**（ブラウザ UI のフレームワーク）の `useState` でローカルに置く UI 状態（選択中のノード・アクティブページ・各パネルの開閉・モーダル表示など）も、サーバー側に **dict で持って REST API で操作する**。

```
# サーバー側の UI 状態（dict で柔軟に持つ）
ui_state = {
    "scene": "welcome",
    "panes": {
        "left":  {"visible": True, "width": 240, "activeComponent": "pages"},
        "right": {"visible": True},
    },
    "currentPageIndex": 0,
    "selectedNodeId": None,
    "modal": None,
    "checkedPanelIds": [],
    "components": {
        "nemuPanel":      {"selectedPage": 0, "expandedPanel": None},
        "characterPanel": {"selectedId": None, "previewImage": None},
        # ...
    },
}

def update_ui_state(patch: dict) -> dict:
    """部分更新（ネストされた dict は再帰的にマージ）"""
    _deep_merge(ui_state, patch)
    return ui_state
```

dict で持つことで **「UI 部分の更新は patch を deep merge」** という単純なルールに揃う。新しい UI コンポーネントが増えても、`components.<name>` を 1 つ生やすだけでスキーマ拡張なしに追加できる。Pydantic で厳格に型付けする道もあるが、UI は変更頻度が高いので柔軟性を優先した。

ブラウザでユーザーがクリックする操作は、**「UI 操作の API を叩く」** に翻訳される。これによって AI から「12 ページ目を開いて、3 番目のコマを選択して」と REST で命令できる。リロードしても画面状態が戻るし、AI が REST で操作したページ遷移が人間の画面でも自動で起こる。**AI と人間が同じ画面を共有する** のに必須の設計判断だった。

サーバー側で起きた変更を即座にブラウザへ流すのに **WebSocket**（HTTP と違って接続を張りっぱなしにできて、サーバー側からブラウザに能動的にメッセージを送れる仕組み）を使う。manginus では **サーバー → ブラウザの一方向プッシュ** だけに使い、ブラウザからサーバーへの操作は REST API に統一している。

これで AI が REST で触っても、人間がブラウザで触っても、別タブで触っても、すべて同じ経路でブラウザに反映される。

## 画像生成 API の設計

画像生成モデルの進化は早く、いまや **1 ページまるごと AI に出させてキャラ一貫性も保たれる** 制作方法も普通にできる時代だ。コストも下がっている。

それでも manginus は「**コマ単位で素材（絵）を生成し、吹き出しとセリフは別レイヤーで組む**」スタイルを採用した。理由は 3 つ。

* **吹き出しを自動計算で組み込みたい**: 1 ページまるごと AI に出させると、絵と吹き出しが一緒に焼き付いた画像として出てくる。ところが私には **写植の技術（手作業で文字をきれいに組む技術）がない** ので、文字数とコマサイズから **吹き出しの大きさを計算で自動調整する** ほうが楽だ。そのためには、AI には **絵だけ描かせて、吹き出しとセリフは manginus 側で別レイヤーとして自動配置する** 必要がある
* **部分修正のしやすさ**: ページ単位で出すと「このコマだけ差し替えたい」「ここの構図だけ変えたい」が通らず、毎回 1 ページ全部を再生成することになる
* **演出責任の置きどころ**: コマ割り・読者の視線誘導・セリフの間など、演出は人間または **専用のネーム生成エンジン（nemu-kaigi）** が判断する粒度に置きたかった。nemu-kaigi 側では「**コンセプト → ストーリー脚本 → コマ単位のプロンプト**」を出力するフローを試している。これを manginus に流し込めば「**絵を描く AI / 演出を考える AI / 組み立てるエディタ**」と役割分担できる（nemu-kaigi の詳細は別記事で書く予定）

API も「AI が完成形を作る」のではなく「**AI が試行錯誤しながら部品を作り、人間が組み立てを判断する**」を前提に設計する。

### 生成と採用を分ける

漫画の品質要件では「3 枚出して 1 枚選ぶ」が基本。だから API も「生成」と「採用」を分ける。

```
POST /api/images/generate    →  candidates[] に追加（複数枚）
POST /api/nodes/{id}/adopt   →  どの candidate を本採用にするか指定
```

ノードは `candidates: list[Image]` と `adopted_index: int` を持つ。AI がリテイクしても過去の候補は残るので、人間が選び直せる。

### プロンプトは「書く」から「設計する」へ

プロンプトをフラットに書いても精度が出ない。Jinja2 で「**脚本コンテキスト**」と「**場面の指示**」を分離して渡す。脚本のト書きをそのまま渡すと AI が「空気感」を拾ってくれる。**指示だけ渡すと無機質な絵、脚本を渡すと感情が乗る**。

API の `POST /api/images/generate` も `script_context` と `scene_directive` を別フィールドで受け取るスキーマにしてある。**プロンプト設計の知見を API のスキーマに織り込む** と、AI エージェントもこの分離を勝手に学習する。

### JSON で部分修正

「構図はいいけど髪が茶色い」のとき、全部描き直さずに済む。Gemini に画像構造を JSON で抽出させて、差分だけ入れて再生成する。`POST /api/images/edit` に `node_id` と `instruction` を渡す形で API 化した。**ただし参照画像を受け付けないので顔は直せない**——この制約は MCP ツールの `description` に書いておくと、AI が自分で適切な API を選んでくれる。

## 合成は古典の API で — レイヤー演出

**一般的な** AI 画像生成は「1 枚の絵を出力する」ことしかできない。レイヤー分けや透明度の概念がない（LayerDiffusion などレイヤー出力の研究は進んでいるが、業務利用できる形ではまだ限定的）。だから「背景を薄く敷いて手前にキャラ」のような演出は **AI に 1 枚で描かせると失敗する**。

代わりに **AI に素材だけ生成させて、合成は古典の API でやる**。

1. **背景レイヤー**：風景画を生成して、**PIL**（Python Imaging Library。Python の標準的な画像処理ライブラリ。ピクセル操作・透明度・フォーマット変換などができる）で透明度 30% にして敷く
2. **切り抜きキャラ**：立ち絵を生成して、**ComfyUI**（ノードベースの画像生成 GUI ツール。`POST /prompt` でワークフロー JSON を投げて実行できる API も持っている）経由で **rembg**（画像の背景を AI で除去するライブラリ）にかける
3. **重ね合わせ**：manginus のノードツリーで親子関係を組んで前景に配置

これは Photoshop でやるのと同じ流れを API 化しただけだ。`POST /api/transform` に `effect: rembg` を指定すれば背景透過になる。透明度の設定はノードに `opacity: 0.3` を渡す。**従来 GUI でしかできなかった画像処理を API として公開する** という地味な仕事が、AI 連携で大きく効く。

PIL は 20 年以上前からある古典的なライブラリだが、AI 生成と組み合わせると **「AI に描けないものを補う層」** として再評価される。ComfyUI と PIL は両方とも画像処理ができるので、境界は **「ワークフローとして他のプロジェクトにも再利用したいなら ComfyUI / このサービスでしか使わない処理なら PIL」** で割っている。

### おまけ：AI に「見せる」ための座標グリッド

PIL の用途で地味に効いているのが、**AI が画像を確認するときの補助オーバーレイ** だ。`POST /api/transform` に `effect: "grid"` を渡すと、PIL で **座標グリッドを焼き込んだ確認用画像** が返る。

縦横に分割線を引いて、各位置に `0.25` / `0.50` / `0.75` のラベル、四隅に `0,0` / `1,1` の文字を焼き込んだ画像になる。manginus はノード座標を「親からの相対値（0〜1）」で持っているので、AI がこの画像を見ながら「**ここの位置（0.30, 0.45）にあるオブジェクトを 0.50, 0.45 に動かしたい**」のような相対指示を組み立てやすい。

「AI に画像を見せる」のは生成だけでなく **状態確認** の用途もあって、そっちのために PIL でひと手間加える、というのが対話的な制作には効く。AI 連携対応のツールでは、**AI が「見る」ためのお膳立て** が地味に効果ある領域だ。

## 壊れて直す — AI 前提のツールの防御設計

AI ツールは **壊れる**。ネームを再インポートしたらノードが消えた、サーバー再起動でセッションが消えた、ドキュメント通りの API が動かなかった——日常茶飯事だ。

普通の Web アプリ以上に **防御的な実装** が要る。AI が何をするか予想できないので、人間が「保存してね」とリマインドできない。manginus で追加した「当たり前」の機能は 3 つ。

* **サーバー終了時の自動セーブ**：シャットダウン処理で「未保存の変更があるか」判定して、あれば自動保存
* **タイムラプスの永続化**：撮影状態を JSON に書き出して、再起動後に続行
* **UI 状態の復元**：開いていたページ・選択ノードを保存、リロードしても戻る

UI 状態をサーバーに置いている設計のおかげで、復元は自動セーブの一環として処理できる。**設計判断（UI state も API）が防御設計（永続化）にも効いた**、という連鎖。

タイムラプスは少し複雑で、**2 層に分けて** 実装している。

* **キャプチャ層**（manginus 本体内）: Playwright 常駐スレッドで 3 秒ごとに定期スクショ + state 変更時の追加スクショ。チャットログ・API 呼び出し履歴も同時に記録する
* **編集層**（別リポジトリ、**Remotion**（React で動画コンポジションを書ける仕組み）で実装）: 上のフレーム + ログを取り込んで、**チャットログを字幕として焼き込んだデモ動画** にレンダリング

「素材を撮る層」と「動画として組み立てる層」を別 repo に分けてある。マンガ制作の素材生成 → 編集と同じ構造で、後段はフロントエンド技術（React + Remotion）の世界に切り替わる。**AI と人間の協業プロセスを可視化する** ためのツール群がここに集まっている。

初回のタイムラプスは、生成スクリプトのタイムアウトや重複リクエストでクオータを使い切るなどのドタバタを全部拾ってしまい、結果として **NG 集** になった。せっかくなのでそれを Remotion で字幕付き動画化して公開してある。

ついでに開発フローの話。テスト 82 件のうちブラウザ起動待ちで 30 秒タイムアウトしているテストがあって、全体 2 分かかっていた。**`sleep(1)` を `sleep(0.5)` にして停止判定を入れる、たった 2 行の修正で 112 秒が 37 秒になった**。本来は固定 sleep ではなく **ポーリング（条件を満たすまで待つ）** に書き直すのが筋で、これは「**AI にテストを書かせると `sleep()` を多用しがち**」という典型的なパターンでもある。次の改善として控えているが、まず固定 sleep の値を見直すだけでも効果は大きい。AI に開発を任せるとループあたりの待ち時間が積み重なるので、テスト時間は何倍にも効く。

「壊れたとき次に同じ壊れ方をしないこと」が AI ツール開発のコツで、`CLAUDE.md` がその経験の蓄積場所になる。**ドキュメントに失敗を書く、テストに失敗パターンを入れる、コードに防御処理を足す**——壊れた経験が次の安定を作る。

## エクスポート API のバリエーション

「全操作を API 公開する」のエクスポート編。PNG エクスポートを **Playwright**（Microsoft 製のブラウザ自動化フレームワーク。テストや E2E でよく使われるが、Python から呼んでヘッドレスブラウザを操作できる）**でブラウザのプレビュー画面をスクショする** 方式にしている。

エクスポートのたびに毎回ブラウザを立ち上げ直すと、Chromium プロセスの起動コストが重い（1 回数秒）。なので **バックグラウンドのワーカースレッドにブラウザを 1 つだけ起動しておいて、エクスポート要求が来るたびに同じブラウザで連続スクショする** 形にしている。これで 17 ページの単行本も、3 秒ごとのタイムラプスも、ブラウザ起動オーバーヘッドが 1 回だけで済む。

実装上の地味な制約として、**Playwright の sync API はスレッド親和性がある**（同じスレッドからしか操作できない）ので、専用の 1 スレッドにブラウザを閉じ込めて、外部からは **キューでタスクを投げる** 形にしてある。

```
import queue, threading
from playwright.sync_api import sync_playwright

_queue: queue.Queue = queue.Queue()

def _browser_thread():
    """専用スレッドのメインループ。キューからタスクを受けて実行"""
    pw = sync_playwright().start()
    browser = pw.chromium.launch()
    while True:
        item = _queue.get()
        if item is None: break
        func, result_queue = item
        try:
            result_queue.put(("ok", func(browser)))
        except Exception as e:
            result_queue.put(("error", e))

def run_in_browser(func, timeout=120):
    """別スレッドからキュー経由でブラウザを操作"""
    result_queue: queue.Queue = queue.Queue()
    _queue.put((func, result_queue))
    status, value = result_queue.get(timeout=timeout)
    if status == "error": raise value
    return value

# 使い方
def shoot(browser):
    page = browser.new_page(viewport={"width": 800, "height": 1131}, device_scale_factor=3)
    page.goto(url)
    page.locator("#page-canvas").screenshot(path=out_path)
    page.close()

run_in_browser(shoot)
```

`device_scale_factor=3` を指定すると 3 倍解像度の高品質 PNG が出る。CSS のレイアウトはそのままで縦書き・特殊フォントもブラウザに任せられる。**プレビューとエクスポートが完全一致** するので、AI が触った状態と出力画像が構造的にズレない。

PNG さえ出れば、派生フォーマットは組み立てるだけだ。

* **PDF**: `img2pdf`（PNG を結合して PDF にする Python ライブラリ）で結合。`POST /api/export/pdf` に `margin_mm` を渡す。これは **セブンイレブンのマルチコピー機で B5 印刷したらコマ枠の端が 4mm 切れた** ので追加したパラメータで、エミュレータでは出ない実機特有のバグだった
* **EPUB3**（電子書籍の標準フォーマット）: PNG + メタデータ + spine（ページ順序）。`page-progression-direction: rtl` で右綴じ（日本語漫画）、`ltr` で左綴じ（英語版）
* **コマ単位**: Panel ノードごとにスクショ → 透過 PNG（Playwright の `omit_background=True` でブラウザの背景を透明にできる）
* **コラムノード**: 漫画ページの中に Markdown / HTML を流し込める特殊ノード。あとがき・キャラ紹介・目次を漫画の合間に配置できる
* **単行本化**: `POST /api/projects/{id}/import-pages` で別プロジェクトのページを引っ張ってきて、`--insert-at` で挿入位置を指定

汎用的な API を出しておくと、**後から思いつかなかった用途で AI が使い回す**。これが API 公開の効きどころで、たとえばコマ単位エクスポートは元々インスタ用だったが、後で LINE スタンプ用途に使い回された（次節で書く）。

## データ再利用 — 予期しない用途で API が活きる

API として「データを綺麗に持っておく」と、後から **予期しない使い道** が出てくる。manginus で起きた 2 つの例。

### 1. コマエクスポート → LINE スタンプ

コマ単位エクスポートは最初インスタ投稿用に作った。が、漫画のコマは「**はみ出す**」（吹き出しがコマ枠を越える、エフェクトが隣のコマにかかる）ので、コマ単位で切り出すと不自然にカットされる。

「お蔵入りか」と思っていたら **LINE スタンプ用に完璧** だった。スタンプは 1 枚 1 枚が独立した画像で、コマ枠のはみ出し問題がそもそもない。透過 PNG → ComfyUI rembg → PIL でテキスト合成 → リサイズ。1 日で 24 種出せた。

実際に販売したスタンプ:

<https://line.me/S/sticker/33578841>

### 2. コマ座標 → Blender で動画化

コマ単位のデータが今度は **動画化パイプライン** に活きた。「カメラがコマからコマへ移動しながら撮影している」表現を **Blender Python**（3D ツール Blender に内蔵されている Python API。`bpy` モジュールで 3D シーンを Python から操作できる）で量産可能にしたとき、**manginus が持っているコマの座標 JSON をそのまま Blender に渡す** だけで済んだ。

```
@app.get("/api/projects/{id}/panels.json")
def export_panels_json(id: str):
    return [{"page": idx, "panels": [...]} for idx, panels in iter_pages(project)]
```

地味なハマりどころが 1 つあった。Blender 4.3 のデフォルトの色空間設定（**View Transform = AgX**。HDR 用のトーンマップで、写実的な絵の彩度を抑える効果がある）が、もとから完成している sRGB の漫画画像だとコントラストを薄くしてしまう。`scene.view_settings.view_transform = "Standard"` に戻すと sRGB のコントラストが復活する。Blender ドキュメントを読めば書いてあるレベルの基本設定だが、知らないと「色が薄い」と悩む。

実際の出力例:

<https://www.youtube.com/watch?v=Np2dA4ZjKPk>

manginus のコマ座標が、印刷・電子書籍・スタンプ・動画と連鎖的に活きていく。**「データを 1 箇所に集めて API で出しておけば、後から複数の用途で再利用できる**」という古典的な設計判断が、AI 連携時代も普通に効く。

## MCP — 既存 REST への薄いラッパーで十分

最後に、MCP（AI エージェントに外部ツールを使わせるためのプロトコル）対応の話。manginus は Python の MCP SDK の `FastMCP` を使い、FastAPI アプリ内に mount する形で実装している。

```
from mcp.server.fastmcp import FastMCP

mcp = FastMCP(
    "manginus",
    stateless_http=True,    # 各リクエストを独立処理（HTTP の互換性が高い）
    json_response=True,
    instructions=(
        "manginus は MCP ネイティブな漫画エディタサーバーです。"
        "初回接続時は必ず get_claude_md ツールを呼び出して操作ガイドを取得してください。"
    ),
)

# ツールモジュールをドメイン別に分割して登録
from . import (
    tools_project, tools_story, tools_pages, tools_nodes,
    tools_nemu, tools_characters, tools_generate, tools_export,
)
```

3 つの工夫がある。

1. **`instructions` で初回ガイドを誘導**: AI が接続したら最初に `get_claude_md` ツールを呼ぶように指示する。これで「ワークフロー、ネーム YAML フォーマット、API の使い方」を `CLAUDE.md` 1 ファイルから一括で渡せる。AI が手探りで API を叩く前に、運用ルールを把握させる仕組み
2. **ツールはドメイン別モジュールに分割**: `tools_pages` `tools_nodes` `tools_generate` のように REST のリソース構成と対応させる。新しい操作を増やしたいときに、対応するモジュールに 1 関数足すだけで済む
3. **MCP ツールにロジックは書かない**: ツールの中身は内部で REST API を `httpx.post()` で叩くだけ。バリデーションも実装も REST 側に集約

```
@mcp.tool()
async def add_panel(page_id: str, x: float, y: float, w: float, h: float) -> dict:
    """ページにコマを追加する"""
    res = await httpx.post(f"{BASE}/api/pages/{page_id}/panels",
                           json={"transform": {"x": x, "y": y, "width": w, "height": h}})
    return res.json()
```

このスタイルだと:

* **REST と MCP で挙動がズレない**: 両方が同じエンドポイントを叩く
* **新しい操作の追加が早い**: REST に 1 つエンドポイントを足すと、MCP には数行追加するだけで AI からも使える
* **デバッグが楽**: MCP で挙動が怪しいときも、`curl` で同じ操作を試して切り分けられる
* **AI 専用の隠れ API がない**: 人間が触れない API が出来上がるのを防ぐ

「LLM 用の特殊な API」を設計し始めると API が二重化して保守コストが上がる。**REST API を AI が使いやすい粒度に揃えておけば、MCP は機械的なラッパーで済む**。これは MCP に限らず、Function Calling や OpenAI Tools API にも同じ考え方で応用できる。

## 振り返り

manginus に使った技術はどれも **新しくないし派手でもない**。FastAPI / Pydantic / WebSocket / React / ComfyUI / PIL / Playwright / Blender + bpy / MCP——枯れた技術ばかり。

新しさはない。あるのは「**全操作を AI 側に API として公開する**」という方針と、**役割分担の組み合わせ方** だけ。

具体的にやったことを並べると、どれも普通の設計判断:

* データの真実は FastAPI に置く
* UI 状態も FastAPI に置いて REST API で操作できるようにする
* ブラウザは React state にミラーするだけ
* WebSocket は一方向プッシュで使う
* 画像生成は外部、画像処理は ComfyUI と PIL に分担、エクスポートは Playwright
* MCP は REST API への薄いラッパー、AI 専用の隠れ API を作らない
* Blender 連携も「コマ座標の JSON を渡すだけ」の単純な接続

「AI チャットで頼めば全部やってくれる」が当たり前になる時代に、ツール側がまだ追いついていない領域はたくさんある。グラフィックエディタはその一つに過ぎない。3D モデリングも、音楽制作も、ゲームエンジンも、CAD も——同じ問題を抱えている。

「**全操作を AI 側に API として公開する**」というシンプルな方針さえ守れば、AI 連携対応のツールは枯れた技術の組み合わせで作れる。これが manginus を作って 2 ヶ月で得た一番の知見だった。
