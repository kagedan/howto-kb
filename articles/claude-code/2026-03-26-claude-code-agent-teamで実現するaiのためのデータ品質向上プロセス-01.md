こんにちは、クラシルでデータエンジニアをしているharry([@gappy50](https://x.com/gappy50))です。

我々データチームでは、昨年[Tier定義の記事](https://zenn.dev/dely_jp/articles/dely-ai-ready-dataops-tier)で紹介した通り、データモデルにTier（用途と重要度に応じた品質格付け。Tier3以上をAI利用可能とする）を設定し、品質を段階的に上げていく仕組みを運用しています。

<https://zenn.dev/dely_jp/articles/dely-ai-ready-dataops-tier>

Tierが上がるほどデータの信頼性が高まり、AIを含めた高度な活用が可能になる。この仕組み自体は順調に回り始めていますが、**Tierを上げるプロセス自体がボトルネック**になるという新たな課題に直面しました。

その中で、私たちのデータチームではClaude Code Agent Teamを使って、データの品質向上と分析のアジリティを両立させる取り組みを進めています。

目指すのは「AI Readyなデータ活用を組織全員で自走し続けられる状態」です。

![Step3: AI Readyなデータ活用を組織で自走し続ける](https://static.zenn.studio/user-upload/deployed-images/e12f792d789a0f1d638f44a1.png?sha=8ff2214149f3fd29fd3d1afc32e19ad58cb792db)

本記事では、このボトルネックをClaude Code Agent Teamでどう解消しているかを、実際の体験をお見せしながら紹介します。

先に結論をお伝えすると、**Tierを上げるプロセスをAgent Teamで自律化することで、少人数のデータチームでも品質と分析のアジリティを両立できるようになりました。** 鍵になるのは設計と実装の分離、そしてチーム自体を育て続ける仕組みです。

## AI活用を前提としたデータ品質の課題

AI時代のデータ基盤では、データの品質と分析のアジリティの両立がこれまで以上に重要になっていると考えています。AIに品質の低いデータを渡せば、当然ながら品質の低い結果しか返ってこない。一方で、品質を追求するあまりデータの整備に時間がかかりすぎれば、ビジネスの意思決定スピードに追いつけません。

特にAIにとって重要なのがSSoT（Single Source of Truth: 唯一の信頼できる情報源）の確立です。同じ「月間売上」を計算するクエリが複数存在し、それぞれ微妙に異なる数値を返す状態を想像してみてください。人間であれば「このクエリが公式」と判断できますが、AIにはその文脈がありません。SSoTが確立されていなければ、AIは「どのデータを信じるべきか」を判断する根拠を持てないのです。

我々のTier定義では、Tierを上げるタイミングでディメンショナルモデル（factテーブルとdimensionテーブルに分離する設計パターン）を作成し、重要なデータの品質を高めながらよりアジリティの高い分析が行える土台を作っています。adhocモデル（ビジネスニーズに応じてその場で作られた、設計規約に基づかないdbtモデル。Tier4に相当）のままでは、作成者の暗黙知に依存した構造になりがちです。カラム名が曖昧だったり、JOINの前提条件がコメントにしか書かれていなかったりすると、データ管理のガバナンスが効きにくく、AIがそのモデルを安全に利用することも困難になります。

一方で、このTierを上げるプロセスには3つの課題があります。

* **属人的な対応がボトルネック**: ディメンショナルモデルの設計・実装はデータエンジニアリングのスキルが求められる作業です。2〜3人のデータエンジニアで7ドメインのデータを支えている我々のようなチームでは、要望に応えるキャパシティが圧倒的に不足します
* **検証コストの高さ**: 移行前後で数値が変わっていないことの確認、既存のデータモデルに潜む不具合の検知、ディメンション別の整合性チェック。これらをすべて手動で行うと1時間以上かかり、Tierを上げるたびにこの検証を繰り返す必要があります
* **設計原則のブレとレビューコスト**: データチームの中でも、ディメンショナルモデルの設計原則が人によってぶれることがあります。どのカラムをdimension/factのどちらに配置するか、既存のadhocモデルからどのロジックを引き継ぐか。こうした判断がブレると手戻りが発生し、レビューコストも膨らみます

品質向上にかかる工数をAIで削減できなければ、組織の力学は品質よりも質の低いデータを量産するほうにすぐに傾くことになり、データが資産にならない。この課題を解決するために、Claude Code Agent Teamによる品質向上プロセスの自律化に取り組みました。

## Agent Teamの全体像

まず、Agent Teamの構成要素であるskills（スキル）とAgents（エージェント）を紹介します。skillsとはClaude Codeの仕組みで、エージェントに与える「特定タスクの実行手順と判断基準をまとめた定義ファイル」です。`.claude/skills/` ディレクトリに配置し、スラッシュコマンドやエージェントのスキル定義から呼び出して使います。

我々のTier定義に合わせて、データ関連のリポジトリには現在以下のようなスキルを用意しています。

| スキル名 | 品質向上の段階 | 内容 |
| --- | --- | --- |
| promote-to-tier3 | adhocモデル → SSoTかつ品質の高いデータモデルへの昇格 | generic test（dbtが標準で提供するデータ品質チェック）追加、SSoTなデータ統合のための検証とデータモデルの作成、メタデータ整備、adhoc依存の解消 |
| migrate-to-star-schema | 品質の高いモデルをディメンショナルモデルへ構造改善 | 4Phaseでfact/dimensionテーブルに分離し、分析のアジリティを向上 |
| migration-test | 移行時の品質検証 | MCP経由でSnowflakeに接続し、移行前後の数値整合性を検証 |

これらのスキルを実行するエージェントは、「書き込み権限の最小化」と「判断と実行の分離」を設計方針として6つに分けています。

| エージェント | 役割 | モデル | 権限 |
| --- | --- | --- | --- |
| data-architect | 設計・計画のみ。コードを書かない | Opus | read-only |
| dbt-data-modeler | SQL/YAML実装 | Sonnet | write権限あり |
| migration-tester | 数値検証 | Sonnet | MCP経由Snowflake接続、read-only |
| migration-reviewer | コードレビュー（data-architect兼任） | Sonnet | read-only |
| model-build-optimizer | パフォーマンス分析 | Sonnet | read-only |
| data-engineer | インフラ・パイプライン | Opus | 全権限 |

ここで注目してほしいのは、6つのエージェントのうちファイルへの書き込み権限を持つのはdbt-data-modelerとdata-engineerの2つだけということです。分析やレビューを行うエージェントが「ついでに直しておきました」とファイルを書き換えてしまうと、変更の追跡が困難になります。Tier定義で品質を管理している以上、「誰が・いつ・何を変えたか」の追跡性は生命線です。

また、判断が必要な役割（設計の意思決定やインフラの全体最適化）にはOpus、明確な指示に基づく実行中心のタスクにはSonnetを割り当てています。OpusとSonnetにはかなりのコスト差があるため、「どこに知性のコストを集中させるか」を意識的に設計することが大切だと考えています。

ここからは、このAgent Teamが実際にどう動くのかを見ていきます。Tier昇格プロセスの中で最も工数がかかるディメンショナルモデル移行を題材に、4つのPhaseで進める流れを体験ベースで紹介します。

## ディメンショナルモデル移行をAgent Teamでやってみた

各Phase間にユーザー承認ポイントがあるのがポイントです。完全自律ではなく、人間が要所で判断を挟みます。

### Phase 1: data-architectによるギャップ分析

まず、data-architectが既存のadhocモデルを読み込み、ディメンショナルモデルとの差分を分析します。どのカラムをdimension/factテーブルに配置するか、既存のadhocモデルからどのロジックを引き継ぐかを整理し、移行計画を策定します。

data-architectはadhocモデルのSQLロジックとカラムの利用パターンを読み込み、スキル定義内に記述された設計原則（「集計対象のメトリクスはfactに配置」「分析軸として使われるカラムはdimensionに配置」等）に基づいて移行計画を策定します。設計原則がスキル定義ファイルに明文化されているため、人によって判断がブレるという課題を構造的に防止できます。

data-architectの定義には「**DO NOT CODE JUST PLAN**」と明記しており、コードは一切書きません。設計と実装を同じエージェントが担うと、目の前のコードの都合に引っ張られて設計がぶれる。先述した「設計原則のブレ」という課題を、エージェントレベルで構造的に防いでいます。

この移行計画を確認し、意図通りであることを承認してから次のPhaseに進みます。Phase 1の設計承認は、その後の実装すべてに影響するため、ここでの確認は省略しません。

### Phase 2-3: dbt-data-modelerによる実装とadhoc書き換え

設計が承認されたら、dbt-data-modelerがSQL/YAMLを実装します（Phase 2）。実装が完了したら、data-architect（migration-reviewerを兼任）が設計意図通りかをレビューし、CIが通ったらmigration-testerが数値検証を行います。

Phase 3では既存のadhocモデルを新しいディメンショナルモデルに書き換え、同様に検証します。

### 数値検証の実際

Phase 2・3で行われる数値検証がこのプロセスの肝です。migration-testerはMCP（Model Context Protocol: AIエージェントのセッション内からリアルタイムに外部ツールを呼び出す標準プロトコル）を使ってSnowflakeに接続し、以下の5項目を自動で検証します。

![数値検証レポートの実例](https://static.zenn.studio/user-upload/deployed-images/63d2b2bf3818b5a4027fb29c.png?sha=b5ff561030d582dfd5fb08b4c2fda54a286a2350)

| 検証項目 | 内容 | 例 |
| --- | --- | --- |
| **全体集計比較** | レコード件数、ユニークユーザー数、D7/D14/D30集計等の一致確認 | 本番16,708,302件 → 移行後16,708,302件、差分0 |
| **ディメンション別比較** | 各ディメンション値ごとのレコード数やメトリクスの数値を本番と突合 | prod\_rows 12,921,802 = new\_rows 12,921,802 |
| **EXCEPT差分チェック** | 本番→新・新→本番の両方向でEXCEPTを実行し差分行を検出 | 差分行: 0件 |
| **NULL分布比較** | カラム別のnon-null件数を本番と移行後で比較（EXCEPT差分チェックと合わせてNULLの変化を検出） | 全カラムで差分0 |
| **総合判定** | 全検証項目のPASS/FAILからマージ可否を自動判定 | 全項目PASS → マージ判定PASS |

従来、この検証を手動で行う場合、検証クエリの作成に1-2時間、クエリ実行と結果待ちに30分-1時間、結果の目視比較と判定に30分-1時間と、トータルで数時間以上かかっていました。migration-testerではスキル定義に基づく検証クエリの自動生成からMCP経由での実行、自動判定まで含めて10分で完了します。しかも、人間が見落としがちなNULL分布の変化やディメンション別の偏りまで網羅的にチェックしてくれます。

### Phase 4: `/retro` で振り返り

最後に、チーム全体でレトロスペクティブを実施します。`/retro` コマンドを実行すると、各エージェントに生産性・品質・役割越境の3軸で振り返りを求め、改善点を `.claude/` 配下のファイル（エージェント定義・スキル定義）に反映したPRとして提出します。

「次回気をつけよう」で終わらせず、気づきをコードベースに刻むことで、次のセッションでは改善が反映された状態でスタートできます。エージェントが完璧である必要はない。フィードバックを受け取ってコードに反映する仕組みさえあれば、チームは育っていきます。

実際に、dbtリポジトリでは `/retro` から生まれた改善PRが複数マージされています。たとえば、数値検証に必要なCIの自律監視ルールが新たにCLAUDE.mdに追加されたり、dbtのタグ配置ルールや不足しているコンテキストの補完のためにskillsの中にインタビューのプロセスが追加されました。

## adhocモデルからSSoTモデルへの昇格プロセス

ディメンショナルモデル移行と並行して、adhocモデルをSSoTかつ品質の高いデータモデルへ昇格させるプロセスも自動化しています。promote-to-tier3スキルは、以下の3つのステップでTier昇格を進めます。

### ステップ1: 要件チェックと不足分の補完

Tier3への昇格には3つの要件があります。

* generic testが追加されていること
* ビジネスメタデータ（description等）がすべて埋まっていること
* 他のadhocモデルへの参照がないこと（Tier3はadhoc間の依存を許容しない）

Agent Teamはまずこの3要件を自動でチェックし、不足しているgeneric testやメタデータを補完します。たとえば、not\_null制約やunique制約が未設定のカラムを検出し、適切なgeneric testを追加する。descriptionが空のカラムに対しては、SQLのロジックやカラム名から推測してビジネスメタデータのドラフトを生成し、ユーザーに確認を求めます。

### ステップ2: SSoT検証

次に、SSoT検証を行います。同一のビジネスイベント（たとえば「注文」や「ユーザー登録」）を表現するモデルが重複して存在しないかをチェックします。adhocモデルは同じデータを別の人が別の目的で作りがちなため、Tier3に昇格する前にデータの唯一性を担保する必要があります。

この検証をPASSしたモデルには `ssot_verified` タグが付与され、`adhoc_verified` ディレクトリに移動することでTier3昇格かつAI利用可能なモデルであることを明示します。

### ステップ3: 不足コンテキストのインタビュー

スキルの実行中に情報が不足していると判断した場合、Agent Teamはユーザーにインタビューを行います。たとえば「このカラムのビジネス上の意味を教えてください」「この集計ロジックは日次/週次どちらで利用されていますか」といった質問を通じて、メタデータの精度を高めます。このインタビュープロセス自体も `/retro` を通じて改善されてきた仕組みです。

手動で最大2日かかっていたTier昇格の一連の作業（要件チェック・generic test追加・メタデータ補完・SSoT検証）が、Agent Team導入後は1~2時間以内で完了するようになりました

Tier3以上のモデルには正確なメタデータとgeneric testが揃っています。この「正解データ」があることで、AIエージェントは新しいモデルの品質を既存の高品質モデルと比較しながら判断できるようになります。品質の高いデータが増えるほど、AIが参照できる「お手本」が増え、品質向上の精度が上がっていくと期待しています。Tier3以上のデータをAIがどう利用するかについては、別の記事で詳しく紹介する予定です。

## まとめ

本記事では、Tierを上げるプロセスをClaude Code Agent Teamで自律化する取り組みを紹介しました。

我々が実現したいのは、AIを前提としたデータの品質と分析のアジリティの両立です。Tier定義で品質の段階を設け、Tierを上げるタイミングでディメンショナルモデルを作成し、重要なデータの品質を高めながらアジリティの高い分析基盤を作っていく。Agent Teamは、このプロセスを少人数チームでも高速に回すための仕組みです。

試してみたい方は、まず日々の繰り返し作業の手順をMarkdownで書き出すことから始めるのがお勧めです。手順が言語化できれば、それがそのままスキル定義の原型になります。

本記事はSnowflake+dbtを前提に書いていますが、設計原則自体はBigQuery+DataformやDatabricks等の他のスタックでも同様に適用できると考えています。大事なのはツールの選定よりも、役割分離とフィードバックの仕組みです。
