## はじめに

こんにちは。おっぴー@AI推進です。世の中はコーディングエージェントで開発のスピードが格段に早くなったが、同時にこのままでいいのだろうか？と感じるエンジニアの方もいらっしゃるのではないでしょうか。私もその一人です。便利になる反面トレードオフとして付きまとう制約が存在しており、そのトレードオフにどう向き合うかで今後エンジニアとしてのスタンスが大きく変わると考えています。  
本記事では著名人も警鐘を鳴らしているCognitive Debt(認知的負債)について整理し、今後どう向き合っていくべきかについて考察する。

## Cognitive Debtとは何か

「技術的負債(Technical Debt)」という言葉は、多くの開発者にとって馴染み深い概念だ。急いで書いたコード、後回しにしたリファクタリング、雑に済ませたテスト——これらは将来の開発速度を蝕む「借金」として認識され、返済(リファクタリングやテスト整備)の必要性も広く共有されている。

一方で、「Cognitive Debt(認知的負債)」は、技術的負債とは似て非なる問題を指す。この用語は、MIT Media Labの研究チームによるEEG研究で2025年に注目を集めたもので、最近になってSimon Willisonらの紹介を通じて開発者コミュニティに広まり、AI Engineer World's FairでのGeoffrey Litt氏の講演「[Understanding is the new bottleneck](https://www.geoffreylitt.com/2026/07/02/understanding-is-the-new-bottleneck.html)」では、この概念に「Understand to Participate(参加するために理解する)」という新しい切り口を持ち込んだ。技術的負債が「コードそのものの質」の問題であるのに対し、Cognitive Debtは「そのコードに対する人間の理解」の問題だ。

Cognitive Debtの起源詳細

「Cognitive Debt(認知的負債)」という用語自体は、MIT Media Labの研究チームによるEEG研究「[Your Brain on ChatGPT](https://arxiv.org/abs/2506.08872)」(Kosmyna et al., 2025)で注目を集め、ソフトウェア開発の文脈ではMargaret-Anne Storeyが[2026年2月のブログ記事](https://simonwillison.net/2026/Feb/15/cognitive-debt/)で概念化した(Storey自身はPeter Naurの古典「Programming as Theory Building」(1985)——プログラムの本質はソースコードではなく開発者の頭の中の理論である——を基盤に挙げている)。Simon Willisonらの紹介を通じて開発者コミュニティに広まったのが最近の経緯。

コードベース自体は綺麗で、テストも通り、設計も破綻していないかもしれない。しかし、それを書いた(あるいは書かせた)人間の頭の中に、そのコードが「なぜそうなっているのか」「どう動くのか」というモデルが存在しない——この状態がCognitive Debtである。コードは負債を抱えていないのに、人間の理解だけが実態から乖離していく。

## なぜ今問題化するか

Cognitive Debtという概念自体は新しいものではない。大規模なコードベースを複数人で触っていれば、以前から「自分が書いていないコードを完全には理解できない」という状況は存在した。

しかし、Claude Codeをはじめとするコーディングエージェントの登場によって、この問題の性質が変わった。筆者が考える理由は主に3つある。

**1. 変更の規模と速度が人間のキャッチアップ速度を超える**

エージェントは一度の指示で数百行、時には数千行規模の変更を、数分で生成できる。人間がその差分をレビューし、意味を咀嚼する速度は変わらない。生成速度と理解速度のギャップは、エージェントの能力が上がるほど広がり続ける。

**2. 「動く」ことと「理解している」ことが分離する**

エージェントが生成したコードは、テストが通り、要求も満たしていることが多い。この「動作確認」だけで開発者は満足しがちになる。しかし「動く」は「理解している」の証明にはならない。CIがAll Greenであることと、次にそのコードを触るときに正しい変更を提案できることは、まったく別の能力だ。

**3. 概念形成のプロセスがスキップされる**

人間がゼロからコードを書くとき、設計の選択肢を検討し、トレードオフを比較し、既存コードとの整合性を考える——このプロセス自体が、頭の中に概念モデルを構築する行為になっている。エージェントに丸投げすると、この「副産物としての理解形成」が起きないまま、成果物だけが手に入る。

## Understand to Participate

Geoffrey Litt氏が提示した原則が「Understand to Participate(参加するために理解する)」だ。この言葉が示唆するのは、コードを理解する目的が「レビューのため」ではなく「参加し続けるため」だという転換である。

レビューは受動的な行為だ。提示された変更を読み、問題がなければ承認する。これに対して「参加」は能動的な行為であり、次の一手を自分から提案・修正できることを含む。

Litt氏はここで、「understand to verify(検証するために理解する)」と「understand to participate(参加するために理解する)」を対比させる。前者はエージェントの出力が正しいかを検証するための理解であり、エージェント自身の自己検証能力が上がるほど重要性は下がりつつあるとLitt氏は指摘する。一方、後者はプロジェクトを前進させる当事者であり続けるための理解であり、エージェントの能力向上とは独立に、人間側に求められ続ける。

Litt氏の主張を整理すると次のようになる。

* プロジェクトを前進させるには、頭の中に「豊富な概念セット」(コードベースの設計判断・命名規則・依存関係などについて自分なりに持っている理解の引き出し)が必要
* 概念セットが欠けていると、次に何を提案・修正すべきかの判断ができなくなる
* 理解の深さについてLitt氏自身は明示的な定義を与えていないが、講演では変更内容に関するクイズに答えられるかを実質的な基準として用いている。筆者はこれを「次に何をすべきか提案・修正できる能力」と解釈している

つまり、Cognitive Debtを避けるための基準は「読んで分かるか」ではなく「次の変更を自分で駆動できるか」に置かれるべきだということだ。この基準は、テストが通っているか・PRが承認されたか、といった従来のチェックポイントとは独立している。

## 負債が蓄積した先にあなたに起きること

Cognitive Debtを放置した場合に何が起きるかは、技術的負債よりも見えにくい形で進行する。

まず、**提案ができなくなる**。エージェントが変更案を出したときに、それが妥当かどうかを判断する基準が自分の中になくなる。結果として「エージェントが出したものをそのまま受け入れる」以外の選択肢が消える。

次に、**修正ができなくなる**。エージェントの出力にバグや設計上の問題があっても、それを的確に指摘し別の実装を指示することができない。曖昧な言葉で「なんか違う」と伝えるしかなく、修正の往復回数だけが増える。

懸念の実証研究による裏付け (Anthropicの研究)

この懸念は実証研究でも裏付けられつつある。[Anthropicの研究](https://www.anthropic.com/research/AI-assistance-coding-skills)(Shen & Tamkin, 2026, [arXiv:2601.20245](https://arxiv.org/abs/2601.20245))では、52名の開発者を対象とした比較実験で、AI支援を受けた群の理解度クイズ正答率は50%と、手動で書いた群の67%を17ポイント下回った(差が最も大きかったのはデバッグ課題)。一方、AIに概念的な質問を投げかけながら使った参加者は65%以上のスコアを維持し、コード生成を丸投げした参加者は40%を下回った——負債の蓄積は「使うか使わないか」ではなく「どう使うか」に左右されることも同時に示されている。また前述のMIT研究では、LLMを使ってエッセイを書いた群は直後に自分の書いた文章を引用できない割合が83%に達した(ただしこの研究には[方法論への批判](https://arxiv.org/abs/2601.00856)があり、対象もコーディングではなくエッセイ執筆である点には留意が必要)。

最終的に起きるのは、**プロジェクトへの実質的な発言権の喪失**である。技術的な意思決定の場において、その人はもはや「エージェントの出力を承認するボタン」以上の役割を果たせなくなる。これは肩書き上のエンジニアではあっても、実質的にはプロジェクトの操縦席から降ろされた状態に近い。

チーム開発においては、これが個人の問題にとどまらない点も重要だ。特に承認プロセスが特定のメンバーに依存しているチームでは、そのメンバーがCognitive Debtを抱えた状態でエージェントの出力を無批判に承認し続けると、承認がチェック機能として働かなくなり、その経路を通るコードの品質保証が形骸化する。

なお、これに対しては「認知的負債はレバレッジであり、全コードの理解はもともと不要」という反論もある。実際、我々はコンパイラの出力を読まないし、依存ライブラリの内部を全て理解しているわけでもない。抽象化に理解を委ねること自体はソフトウェアの歴史そのものだ。問題は理解を手放すこと自体ではなく、「どこまでを自分の理解の境界とするか」を無自覚に決めてしまうことにある。エージェントの出力はコンパイラの出力と違い、自分のプロジェクト固有の設計判断を含む。そこを理解の境界の外に置くと、前述の発言権の喪失に直結する。

## 検討されている可視化・軽減策の方向性

Cognitive Debtが技術的負債と決定的に違うのは、**可視化する確立した標準的な仕組みがまだ存在しない**ことだ。技術的負債であれば静的解析ツールやコードカバレッジ、循環的複雑度といった指標がある。Cognitive Debtについても、技術的負債・認知的負債・意図の負債の3層でソフトウェアの健全性を捉え直す「Triple Debt Model」([arXiv:2603.22106](https://arxiv.org/abs/2603.22106))など学術的な研究は進みつつあるが、「この開発者はこのコードをどれだけ理解しているか」を測る、実務で広く使われる指標はまだない。

ここには具体的な研究・実装の余地がある。

**セッションログからの概念抽出**

Claude Codeのようなエージェントとの対話セッションログには、エージェントが導入した新しい設計判断、命名規則、依存関係の変更などが時系列で記録されている。これを解析し、「今回のセッションで導入された新規概念・パターン」を自動抽出して要約できれば、開発者が振り返って学習すべきポイントを可視化できる。

extract\_new\_concepts.py

```
# セッションログから新規導入された概念候補を抽出するイメージ
# (実際のログは会話・diff主体の非構造データであり、new_symbols や
#  agent_rationale を得るには、ここより前段でLLMによる抽出処理が必要になる)
import json

def extract_new_concepts(session_log_path: str) -> list[dict]:
    with open(session_log_path) as f:
        events = [json.loads(line) for line in f]

    candidates = []
    for event in events:
        if event.get("type") != "file_edit":
            continue
        # 新規追加された関数・クラス・設定キーなどを差分から抽出
        for symbol in event.get("new_symbols", []):
            candidates.append({
                "name": symbol["name"],
                "kind": symbol["kind"],  # function, class, config_key, etc.
                "file": event["file_path"],
                "rationale": event.get("agent_rationale"),
            })
    return candidates
```

このような抽出結果をもとに「今日のセッションで理解しておくべき概念リスト」を自動生成し、開発者に提示するワークフローが考えられる。

**理解度確認クイズによる定量測定**

Litt氏は講演の中で、変更内容を説明させる[explain-diffスキル](https://gist.github.com/geoffreylitt/a29df1b5f9865506e8952488eac3d524)(実装が公開されている)と理解度チェックのクイズを組み合わせる手法や、対話的なmicro-worlds(小さなデバッガーやシミュレーターのようなもの)を使って理解度を検証する手法を紹介している。これに倣い、大規模diffの適用後に、その変更内容に基づく簡単な質問(「なぜこの関数は非同期化されたか」「このリトライ処理の上限値の根拠は」)を自動生成し、開発者に答えさせる仕組みも有効だろう。回答の正確さや回答までの時間を追跡すれば、Cognitive Debtの蓄積度合いを定量的に近似できる可能性がある。

これらはLitt氏の提案や実験段階のアイデアを参考にしたものだが、共通しているのは「エージェントの出力を人間が消費するだけの一方通行のパイプライン」に、理解を強制的に差し込むフィードバックループを追加するという発想だ。

## 既存ツールの現状

では、今すぐ使えるツールはどこまで揃っているのか。Cognitive Debtへの対策は「予防(理解しながら進める)」「検証(理解度を測る)」「回復(失った理解を再構築する)」の3つに整理でき、それぞれで成熟度が大きく異なる。

**予防: すでに実用段階**

**回復: OSS・商用とも選択肢が豊富**

* [deepwiki-open](https://github.com/AsyncFuncAI/deepwiki-open)(約17k stars)はリポジトリからWikiを自動生成しRAGチャットで質問できる。ローカルLLMでも動く
* [Understand-Anything](https://github.com/Egonex-AI/Understand-Anything)はコードベースを知識グラフ化し、依存順に学べるガイドツアーや変更の影響分析を提供する。受動的にドキュメントを読むのではなく、能動的にグラフを探索させる設計が特徴だ
* 商用では[Unblocked](https://getunblocked.com/)がコードに加えてPR・Slack・Jiraを横断する知識グラフを構築し、「なぜこの設計になったか」に出典付きで回答する。コードの外にある意思決定の文脈(意図の負債)までカバーする点で他と一線を画す

**検証: 明確な空白地帯**

一方で、「AI生成コードに対する開発者の理解度を測定・記録する」ことに特化した確立ツールは、調査した限り存在しない。汎用のクイズ出題MCPサーバー([quiz-mcp](https://github.com/karerckor/quiz-mcp)など)は部品として存在するが、diffからクイズを自動生成し、正答率を追跡してCognitive Debtの蓄積度合いとして可視化する統合的な仕組みは、前節で述べたとおりまだアイデア段階にとどまる。

つまり現状は、**理解の予防と回復は道具が揃いつつあるが、「自分がどれだけ負債を抱えているか」を知る手段だけが欠けている**——負債の可視化こそが管理の第一歩であることを考えると、この空白は次の1年で最も動きがある領域になるだろう。

## 明日からあなたのチームで実践できること

研究レベルのツールが整うのを待たずとも、個人・チームレベルで今日から取れる対処はいくつかある。

* **概念セットを意識的に構築する**: エージェントに変更を依頼する前に、自分なりの設計仮説を一度言語化してみる。エージェントの出力と自分の仮説を比較することで、理解のギャップが可視化される。
* **自分にクイズを課す**: 大きな差分を受け取ったら「なぜこの実装を選んだか」「この設定値の根拠は何か」を自問し、即答できない箇所をその場でリスト化する。  
  Litt氏が紹介する理解度クイズの簡易版として、答えられなかった箇所は後日改めて確認する習慣をつける。「動くかどうか」ではなく「次にこの部分を自分が変更するとしたらどう書くか」を自問しながら差分を読む姿勢が、受動的な承認から能動的な参加への切り替えを起こす。
* **理解できない差分は縮小して依頼し直す**: 一度に大きすぎる変更をエージェントに依頼すると、理解の追いつかない部分がそのままCognitive Debtになる。理解できる粒度まで指示を分割することは、速度を落とす代償を払ってでも価値がある。
* **チームで「理解の共有」を明示的なプロセスにする**: PRレビューの場で「このコードが何をするか」だけでなく「なぜこの設計になったか」を説明し合う時間を設ける。エージェントが生成した理由づけをそのまま鵜呑みにせず、チームの言葉で再構築する。

Cognitive Debtは、蓄積が指標に表れる技術的負債とは違い、外からは見えないまま返済コストが膨らみ続け、気づいたときには参加のハードルが大きく上がっている、という性質を持つ。コーディングエージェントの生成速度が上がり続ける今こそ、「理解すること」自体を意識的な実践として設計し直す必要があると筆者は考える。  
それでは良きAIライフを！
