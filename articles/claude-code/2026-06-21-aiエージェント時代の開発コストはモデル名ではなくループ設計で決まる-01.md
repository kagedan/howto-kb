AIコーディングのコストを考えるとき、まずモデル名や料金表を見たくなる。高いモデルを使ったから高い。軽いモデルに替えれば安くなる。そう考えるのは自然だ。

でも agentic coding で本当に怖いのは、1回の返答の値段ではない。AIに何周させるかである。

補完やチャットなら、こちらが聞いて、AIが答えて、そこで一区切りになる。もちろん長いやりとりはあるが、人間が会話の拍を握っている。ところがエージェントになると、AIは読む、探す、直す、テストする、失敗する、もう一度読む、別の場所を直す、PRを作る、という作業ループに入る。

このループは便利だ。便利だからこそ、放っておくと伸びる。

GitHub Copilot の usage-based な課金や AI credit の話が出てくると、つい「月額固定の時代が終わるのか」という価格ニュースとして読んでしまう。だが開発者にとってもっと重要なのは、請求の単位がチャットの回数ではなく、エージェントが実際に回した作業量へ近づいていることだと思う。

つまり、AI開発のコスト管理は経理の仕事ではない。かなり開発設計の話である。

## 月額補完の感覚では読めない

これまでのAIコーディング支援は、なんとなく「賢い補完」や「横にいる相談相手」として理解できた。IDEで候補を出す。チャットで関数を説明する。エラー文を貼る。こちらの操作を少し速くする道具だった。

この感覚のままエージェントを見ると、見積もりを間違える。

GitHub の cloud agent のような仕組みでは、issue から作業が始まり、リポジトリを調べ、実装計画を作り、ブランチを切り、必要ならPRまで進む。これはもう「1回質問した」ではない。小さな作業者をキューに投入している感覚に近い。

人間なら、途中で「あ、この依頼は大きすぎる」と気づいて止まることがある。あるいは、仕様が曖昧なので先に確認する。エージェントも確認できる場合はあるが、よくできたモデルほど、曖昧な依頼にも筋の通った解釈を作って進めてしまう。

ここでコストが出る。モデルが高いから、だけではない。探索範囲が広い。成功条件が薄い。テストで落ちたときの戻り方が決まっていない。そういう状態でAIに任せると、ループが長くなる。

そして長いループは、トークンだけを消費するわけではない。人間のレビュー時間も食う。

## 高いのはモデルではなく、曖昧な依頼である

もちろんモデル価格は無視できない。強いモデルは高くなりがちだし、複雑な作業では実際に強いモデルを使いたい場面もある。

ただ、現場で効く差はそこだけではない。

「この画面をいい感じに直して」

この依頼は短い。短いが、安い依頼ではない。AIは関連ファイルを探す。既存コンポーネントを見る。CSSを読む。テストを探す。デザイン上の意図を推測する。場合によっては複数の画面を直す。動作確認で失敗すればまた戻る。

同じ作業でも、こう書くとかなり変わる。

「このフォームのエラーメッセージだけを、既存の `FormField` の表示ルールに合わせる。対象は `app/settings` 配下。文言変更とスタイル調整まで。バリデーション仕様は変えない。`pnpm test settings-form` が通れば完了」

長いプロンプトが偉い、という話ではない。境界があるかどうかだ。

触ってよい場所。変えてはいけない仕様。通すべき確認。完了条件。これらがあると、AIは迷う時間を減らせる。人間もレビューしやすい。失敗したときも「どこからやり直すか」が見える。

逆にここがないと、AIは親切に広く考える。親切な広さは、エージェントの世界では課金と差分量になる。

## issue は小さな仕様書ではなく、実行境界である

AIに渡す issue を、ただの小さな仕様書だと思わないほうがいい。

仕様書は「何を作りたいか」を説明する。エージェント向けの issue は、それに加えて「どこまで動いてよいか」を決める必要がある。

たとえば、次のような項目があるだけで、作業の性質はかなり変わる。

* 触ってよいディレクトリ
* 触ってはいけないファイルや設定
* 期待する差分の大きさ
* 先に調査だけしてほしいのか、実装まで進んでよいのか
* テストやビルドの確認方法
* 失敗したときに止まる条件
* PR本文に残してほしい判断

これはプロンプト術というより、作業の契約である。

人間の同僚に頼むときも、経験のある人ほど暗黙にこういう境界を読む。だがAIにはチームの空気がない。過去の経緯も、リポジトリの地雷も、レビュー担当者の好みも、明示されていなければ分からない。

だから agent-ready な issue は、説明が長い issue ではない。狭く、検証でき、途中で止まれる issue である。

## 節約は「使わない」ではなく「短く失敗させる」

AIコストの話になると、すぐ節約術に寄ってしまう。軽いモデルを使う。出力を短くする。毎回フルコンテキストを渡さない。どれも間違ってはいない。

ただ、エージェントで一番効く節約は「短く失敗できる単位にする」ことだと思う。

いきなり「この機能を作ってPRまで出して」と渡すのではなく、まず調査だけを切る。次に設計案だけを出させる。問題なければ実装へ進める。最後に検証とPR説明を整える。こう分けると、AIの作業は少し遅く見えるかもしれない。

だが実際には、手戻りが見えやすい。

調査の段階で前提が違えば、そこで止められる。設計案が大きすぎれば、実装前に狭められる。テストが通らなければ、どの判断が怪しいか戻れる。最初から最後まで一気に走らせるより、レビューする人間の頭も壊れにくい。

これはAIを疑うための手続きではない。AIを実務に入れるための足場だ。

エージェントは、うまくいくと本当に速い。だからこそ、失敗するときも速い。速く広がった失敗は、人間が読むには重い。

## PRが出てからコスト管理しても遅い

チームでAIエージェントを使うなら、PRが出てから悩むのでは遅い。

大きすぎるPRが出てきたあとに「なぜこんな差分になったのか」と聞いても、もうコストは使われている。レビュー担当者の時間も使われる。CIも回っている。場合によっては、別の人の作業と衝突している。

見るべきなのは、投入前である。

この作業はAIに渡す単位として小さいか。成功判定は機械的に確認できるか。差分が大きくなりそうなら、どこで止めるか。人間は何を読めば責任を持てるか。こういう問いを最初に置く。

ここを決めておくと、モデル選びも少し冷静になる。探索が広いタスクには強いモデルを使う。単純な置換や限定的な修正なら軽いモデルで足りる。判断が必要なところだけ人間が挟まる。道具の使い分けが、料金表ではなく作業の形から決まる。

逆に、タスクの形が曖昧なままだと、どのモデルを選んでも同じ問題に戻る。賢いモデルは広く走り、軽いモデルは迷う。どちらもレビューしづらい。

## agent-ready な開発者は、ループを設計している

AIエージェントを使いこなす、という言い方には少し危うさがある。まるで、うまいプロンプトを知っている人だけが得をするゲームのように聞こえる。

実際にはもっと地味だ。

agent-ready な開発者は、AIに何を頼むかだけでなく、何周させるかを考えている。探索をどこまで許すか。実装へ進む条件は何か。テストに失敗したら何回まで直させるか。PRに出てきた差分を人間が読める大きさに保てるか。

この設計があると、AIは強い味方になる。ないと、便利なはずの自動化が、見えにくい請求と読みにくい差分になって返ってくる。

AI開発のコスト管理は、料金表を眺めることではない。

開発者がタスクをどう切り、どこで止め、何をレビューするかを設計することだ。モデル名は変わる。価格も変わる。だが、ループを短く、検証可能に、レビューできる形へ戻す力は、しばらく変わらない。

結局そこが、AIエージェント時代の開発スキルになる。
