## 「修正しました！」の何割かは、修正されていない

Claude Code を毎日使っていると、必ずこの場面に出会います。

> 「テストも通り、修正が完了しました！」

……と元気に報告されたのに、実際にビルドすると通らない。テストはそもそも実行されていない。LLM は「完了したと**言う**こと」と「完了して**いる**こと」を、必ずしも区別してくれません。もっともらしい完了報告は、完了の証拠にならないのです。

私は Windows 上で Claude Code を業務基盤として常用しており（開発だけでなく週次レビューやタスク管理まで任せています）、この「自己申告完了」問題にここ数か月、仕組みで対抗してきました。この記事では、その対策を**4つの層**に分けて紹介します。

1. **規約層**: CLAUDE.md に「完了の定義」を書く
2. **強制層**: hooks で機械的にチェックする
3. **退避層**: 検証できないものは「未検証」としてバックログに落とす
4. **評価層**: 作った本人にレビューさせない

どれも特別なツールは不要で、層1だけなら3行の追記で今日から始められます。

## 層1: CLAUDE.md に「完了の定義」を書く

まず規約です。私のグローバル CLAUDE.md（`~/.claude/CLAUDE.md`）には、冒頭にこう書いてあります。

```
## 完了の定義（なぜ: LLMの自己申告は完了の証拠にならない）

- コード変更後は build / test / 実行のいずれかで exit code を確認してから完了報告する
- 検証できなかった場合は「未検証」と明記して報告する
```

ポイントは2つあります。

**「なぜ」を併記する。** LLM はルールの意図を理解していると、例外的な状況でも意図に沿った判断をしやすくなります。「守れ」とだけ書かれたルールは、文脈が長くなると形骸化しがちです。

**完了の証拠を「exit code」に固定する。** 「動作確認する」のような曖昧な表現だと、「コードを読んで問題ないことを確認しました」という*目視確認*で完了報告されてしまいます。exit code という機械的な証拠に限定することで、抜け道を塞ぎます。

ただし、正直に言うと**規約だけでは守られないことがあります**。コンテキストが長くなったとき、会話が要約されたときに、ルールの優先度が下がるのを何度も観察しました。そこで層2です。

## 層2: hooks で機械的に強制する

Claude Code には [hooks](https://code.claude.com/docs/ja/hooks) という仕組みがあり、ツール実行の前後やセッションの停止時に任意のスクリプトを差し込めます。重要なのは、hooks が **LLM への「お願い」ではなくハーネス側のチェック**だという点です。プロンプトの指示と違い、コンテキストが長くなっても劣化しません。

使うイベントは2つです。

| イベント | 役割 |
| --- | --- |
| `PostToolUse` | Edit / Write が走ったら「未検証フラグ」を立てる。Bash でテスト系コマンドが走ったらフラグを消す |
| `Stop` | 応答を終えようとした時点でフラグが残っていたら、停止をブロックして検証を要求する |

`~/.claude/settings.json` の設定はこうなります。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": "pwsh -NoProfile -File ~/.claude/hooks/mark-dirty.ps1" }]
      },
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "pwsh -NoProfile -File ~/.claude/hooks/mark-verified.ps1" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "pwsh -NoProfile -File ~/.claude/hooks/verification-gate.ps1" }]
      }
    ]
  }
}
```

なお、Windows では hook コマンド内の `~` 展開が効かない環境があるため、実運用では絶対パス指定が安全です。

フラグを操作する2つの補助スクリプトはこれだけです。

```
# mark-dirty.ps1: コード系ファイルの編集で「未検証フラグ」を立てる
$p = [Console]::In.ReadToEnd() | ConvertFrom-Json
if ($p.tool_input.file_path -notmatch '\.(md|txt|json)$') {
    New-Item -ItemType File -Force (Join-Path $env:TEMP "claude-dirty-$($p.session_id)") | Out-Null
}
```

```
# mark-verified.ps1: テスト/ビルド系コマンドの実行でフラグを消す
$p = [Console]::In.ReadToEnd() | ConvertFrom-Json
if ($p.tool_input.command -match 'test|build|pytest|vitest') {
    Remove-Item -Force -ErrorAction SilentlyContinue (Join-Path $env:TEMP "claude-dirty-$($p.session_id)")
}
```

mark-dirty 側で `.md` などドキュメント系の拡張子を除外しているのは伏線で、初版でここを絞らずに運用したら、ドキュメント編集だけのセッションでゲートが発火する誤検知に悩まされました（後述）。

ゲート本体の簡略版です（実運用版から要点だけ抜き出した**概念デモ**です）。

```
# verification-gate.ps1（簡略版）
$payload = [Console]::In.ReadToEnd() | ConvertFrom-Json

# 重要: Stop hook のブロックで再実行された場合は素通しし、無限ループを防ぐ
if ($payload.stop_hook_active) { exit 0 }

$flag = Join-Path $env:TEMP "claude-dirty-$($payload.session_id)"
if (Test-Path $flag) {
    # 実運用版はここで transcript の最終応答に「未検証」の明記があれば通す
    # （未検証エスケープ）。この簡略版では省略している
    @{
        decision = "block"
        reason   = "コード変更後に build/test/実行の exit code 確認がありません。検証を実行するか、『未検証』と明記して pending-verification.md に追記してください。"
    } | ConvertTo-Json -Compress
    exit 0
}
exit 0
```

`decision: "block"` を返すと Claude は停止できず、`reason` の内容が指示として渡ります。つまり**停止する前に必ず一度、「検証するか、未検証と明記するか」を機械が要求してくれる**わけです。正確に言うと、これはハードな強制ではありません。次項の無限ループ対策により督促は停止試行1回につき一度で、2周目は素通しします。それでも「検証漏れが誰にも指摘されずに完了する」ことだけは起きなくなります。

実装上の注意を3つ。

* **無限ループ対策は必須。** Stop hook がブロックすると Claude は作業を続け、再び停止しようとして hook が再発火します。ペイロードの `stop_hook_active` を見て2周目は素通ししないと、永遠に終われないセッションが生まれます（一度やらかしました）。
* **fail-open で設計する。** ゲートスクリプト自体が例外を吐いたときは `exit 0`（許可）に倒します。品質ゲートのバグで全セッションが停止するのは本末転倒なので、「壊れたら通す・ただしログは残す」が正解でした。
* **フラグ消しの判定は緩めから始める。** 簡略版ではテスト系コマンド(`test|build|pytest` など)の実行をもって検証済みとしています。厳密に exit code まで見たければ、Stop hook 側で `transcript_path` のセッションログ(JSONL)を読み、最後の Edit/Write 以降に成功した Bash 実行があるかを判定する方式が堅牢です。私は運用しながら段階的に厳しくしました。

## 層3: 「未検証」をバックログとして管理する

現実には、その場で検証できないことがあります。実 API キーが必要、外部サービスの応答待ち、人間の実機確認が必要──などです。

このとき最悪なのは、「検証できなかったこと」が完了報告の中に溶けて消えることです。私は**未検証専用のバックログファイル**を1枚置いています。

```
# pending-verification.md（未検証バックログ）

| # | 項目 | 未検証の理由 | 起票日 | 状態 |
|---|---|---|---|---|
| 25 | 学習アプリの実API経路 | APIキー記入待ち | 07-12 | 未検証 |
| 26 | 新hookの実発火 | 次セッションで確認 | 07-13 | 未検証 |
```

運用ルールは1つだけ。**「未検証」と報告に書いた項目は、必ずこのファイルに追記する。解消したら日付つきで解決済みへ移す。** これで「未検証」が免罪符ではなく追跡可能なタスクになります。層2のゲートも、実運用版では検証の代わりに最終応答への「未検証」明記を認める判定を入れてあり（前掲の簡略版では省略しています）、「検証するか、未検証と記録するか」の二択に落ちる設計です。

## 層4: 作った本人にレビューさせない

exit code で担保できるのはコードだけです。ドキュメント、設計、文面のような成果物はどうするか。

やってはいけないのは、**同じ会話の中で「今作ったものをレビューして」と頼む**ことです。直前に自分が作った成果物への自己レビューは、体感でほぼ確実に甘くなります。生成時の文脈（こう作るのが正しいという前提）を引き継いだままレビューするので、前提ごと間違っているケースを検出できません。

私は CLAUDE.md にこう書いています。

```
- 生成物の品質判定は、作った本人と別コンテキスト（reviewer）で行う
  （なぜ: 同一コンテキストの自己レビューは迎合で甘くなる）
```

具体的には、読み取り専用ツールだけを持たせた reviewer サブエージェントを別コンテキストで起動し、「弱点の発見・反証・受け入れ基準チェック」だけをさせます。修正権限を持たせないのは、レビュアーが自分で直し始めると批判が手心に変わるからです。外部に出す文面は、この reviewer の合格記録をファイル冒頭にコメントで残す運用にしています（`<!-- reviewer: 2026-07-07 条件付き合格 -->` のように）。

## 効果はどうだったか

正直に書くと、厳密な A/B 測定はしていません。その上で観察できた変化は:

* **「未検証」という語が報告に日常的に現れるようになった。** 導入前は「たぶん動く」ものが完了として流れていました。今は未検証バックログに月十数件が起票され、そのほとんどが後日検証で回収されています。
* **Stop hook のブロックは月に数回発火します。** 発火が意味するのは「検証されないまま完了しかけた変更を捕捉した」ことまでで、そのすべてが実際に壊れていたわけではありません（下記の誤検知も含みます）。それでも、検証漏れが誰にも止められず素通りする経路が塞がれていることの確認にはなっており、頻度が低いこと自体が層1の規約が普段は効いている証拠だと解釈しています。
* **副作用もありました。** 初版ではドキュメント編集だけのセッションでもゲートが発火する誤検知があり、対象ツール・対象拡張子を絞る調整が必要でした（前掲 mark-dirty の拡張子フィルタはその産物です）。ゲートは「厳しく作って緩める」より「緩く作って締める」方が運用が楽です。

## まとめ: 今日からの最初の一歩

4層をまとめると:

| 層 | 手段 | コスト |
| --- | --- | --- |
| 規約 | CLAUDE.md に完了の定義(3行) | 5分 |
| 強制 | hooks で検証ゲート | 半日 |
| 退避 | 未検証バックログ1ファイル | 10分 |
| 評価 | 別コンテキスト reviewer | 30分 |

全部やる必要はありません。**まず層1の3行**をあなたの CLAUDE.md に足してください。それだけで完了報告の景色が変わります。「未検証」という言葉が Claude の口から出るようになったら、この記事を思い出して層2以降を足していく──そのくらいの温度感がちょうどいいと思います。

LLM との協働で大事なのは、賢さを信頼することではなく、**検証を仕組みにすること**でした。同じ問題で消耗している方の参考になれば嬉しいです。
