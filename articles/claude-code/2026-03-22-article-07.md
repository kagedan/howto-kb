🚀 AmazonやDisneyで7年SWEを経験し
現在エンタープライズ向け
AIエージェント開発のCTOをやっている人が書いた

「Claude Code完全攻略ガイド」が
海外でバズってたので日本語で解説

対象：Claude Codeを使い始めた〜中級者のエンジニア

本人いわく「何百万ユーザーが触るシステムを落とせない環境で開発してきた」とのこと
その経験からClaude Codeをプロダクション品質で使い倒すノウハウをまとめている

✅ Think First（まず考えろ）

いきなりプロンプト打つな
これが最大の間違い

Shift+Tab×2でPlan Modeに入る
ここで設計を詰めてから実装に入ると、出力の質が劇的に変わる

悪い例：「auth system作って」
良い例：「既存のUserモデルでemail/password認証、Redisにセッション24時間保存、/api/protected以下にミドルウェア追加」

この5分の投資が、後の数時間のデバッグを消す

✅ CLAUDEmdの極意

起動時に必ず読まれるファイル
ここに書いた内容がすべての会話に影響する

ただし注意点がある

- Claude Codeのシステムプロンプトで既に50個程度の指示枠を使っている（本人の体感）
- モデルが同時に守れる指示は150〜200個が目安
- つまり詰め込みすぎると、ランダムに無視され始める

書き方のコツ
- 「何を」だけでなく「なぜ」を書く
- 「TypeScript strictモード使え」より「implicit anyで本番バグ出したからstrict必須」の方が効く
- 新人向けドキュメントではなく「明日記憶喪失になる自分へのメモ」というイメージ

会話中に#キーを押すとhttps://t.co/NOe1NFDGAyへ自動追加できる
同じことを2回修正したら、それはファイルに書くべきサイン

✅ コンテキストウィンドウの罠

200,000トークンあるから大丈夫でしょ？

実体験ベースだと、20-40%あたりで出力品質が落ち始める感覚があるらしい
100%まで使えるわけじゃない

/compactしても、既に劣化した後だと回復しづらい

対策
- 1会話1機能を徹底（auth作ってからDB設計、は別会話に分ける）
- SCRATCHPADmdやplanmdに外部記憶を書き出す
- 次回はそのファイルを読ませて再開

コピペリセット術（これ使える）
1. 重要部分をコピー
2. /compact で要約取得
3. /clear で完全リセット
4. 重要部分をペースト

これで新鮮なコンテキストに必要な情報だけ入る
劣化した状態で戦うより遥かにマシ

✅ プロンプトが全て

「Claudeの性能が足りないから上手くいかない」
そう思ったことあるかもしれない

でも振り返ると、ほとんどの場合プロンプト側に改善余地がある

曖昧な入力 → 曖昧な出力
これはほぼ物理法則

具体的に書く
制約を明示する
Claude 4.5は特に過剰設計しがちだから「シンプルに。1ファイルで。余計な抽象化するな」と釘を刺す

そして「なぜ」を伝える
「毎リクエストで呼ばれるから速度重視」
「これはプロトタイプで捨てる予定」
こういう背景があると、Claudeのトレードオフ判断が変わる

✅ モデルの使い分け

Opus = 計画・アーキテクチャ決定
Sonnet = 実装

Shift+Tabで切り替え可能
CLAUDEmdがあるから、モデル間の引き継ぎはスムーズ

OpusでPlan Mode使って設計 → Sonnetで実装
このフローが費用対効果良い

（API経由でOpus 4.5を実装にガンガン使うとかなりコスト嵩むので注意）

✅ MCP・Hooks・カスタムコマンド

使わないのはもったいない

MCP（Model Context Protocol）
- Slack、GitHub、DB等への接続
- いちいちコピペしてる作業があるなら、MCPで自動化できる可能性高い

Hooks
- ファイル変更後にPrettierや型チェックを自動実行
- 技術的負債を溜めないガードレール

カスタムコマンド
- .claude/commands/にmd追加
- /commandname で呼び出し
- 繰り返すタスクはコマンド化

Pro Max使ってるなら、全機能試す価値あり

✅ ハマったときの対処法

Claudeがループし始めたら

まず/clear
蓄積したコンテキストが混乱の原因かもしれない

タスクを分割する
複雑なタスクで詰まるなら、Plan Modeの設計が甘い可能性
