## はじめに

Claudeをチャットで使う人は増えました。多くは「1回頼んで、1回返ってくる」使い方です。一方で、Anthropicのエンジニアは、もう違う使い方をしています。Claude Codeをつくっている Boris Cherny 氏は、公開イベント（Fortune Brainstorm Tech）で、こう語っています。

> I don't prompt Claude anymore, my job is to write loops.

訳すと「もうClaudeにプロンプトは打たない。私の仕事はループを書くことだ」。彼はこの考え方を「ループエンジニアリング」と呼んでいます。実際の現場でも、指示を出すのは人ではなく別のClaudeだと話しています（<https://fortune.com/2026/06/08/anthropics-boris-cherny-creator-of-claude-code-says-there-are-days-he-manages-tens-of-thousands-of-ai-agents-at-once/>）。

この記事は、「ループとは何か」をふんわり紹介して終わりにしません。**ループならではの難しさ**まで踏み込みます。Claudeをある程度使っていて、自動化やエージェントにもう一段踏み込みたい人向けです。

## Skillとループは何が違う？

前に「Skill」を紹介しました。ここでよく聞かれるのが「ループってSkillと何が違うの？」です。違いは1点に尽きます。

|  | Skill | ループ |
| --- | --- | --- |
| 与えるもの | やり方（1回をちゃんと） | それを**自分で回す制御** |
| 次の一手を決めるのは | あなた（呼ぶ） | **Claude自身** |
| 品質の出どころ | 1発の出来 | **回す中で良くなる** |
| あなたの役割 | 依頼者 | **監督・設計者** |

Skillは「1回をちゃんとやる」道具。ループは「それを自分で何度も回す」仕組みです。足されるのは**自律**（次を自分で決める）と**反復**（何度も回す）の2つ。そして、**この2つが価値と難しさの両方を生みます**。

![「指示する人」から「設計する人」への変化を左右で比べた図](https://static.zenn.studio/user-upload/38d2c1625b70-20260630.png)  
*図1: 毎回1対1で指示する人から、ループを設計して回す人へ*

## ループの心臓は「評価役」

ループは、ひとことで言えば「**作る → 評価する → ダメなら直す**」の輪です。公式も、エージェントを「AIがツールを繰り返し使うもの」と定義しています（<https://www.anthropic.com/research/building-effective-agents>）。

ここで一番大事なのは「作る役」ではありません。できあがりを見て「これでOK？」を判定し、ダメなら作り直させる「**評価役**」です。人が毎回チェックしないぶん、ループ自身がダメ出しできないと、まちがいをそのまま量産します。評価役がいるから、ループは回るほど良くなります。

![作る役と評価役が回るループの図。ダメなら作り直し、OKなら完成](https://static.zenn.studio/user-upload/3558985411cc-20260630.png)

*図2: 作る役と「ダメと言える評価役」。これがループの心臓部*

これはAnthropic自身がやっていることでもあります。Borisは、コードレビューを「**ペルソナの違うClaudeのチームが協働してやり、ほぼ全部のバグを捕まえる**」と話しています。1人のClaudeに採点させるのではなく、別の視点を持つ複数のClaudeに評価させる。評価役を何重にも置いたループ、というわけです。

## ループならではの難所

評価役を中心に置くと、ループ特有の課題が見えてきます。どれも「**人が毎回見ていない**」から生まれます。Skillには無い、ループだけの難所です。

### 1. どこで止めるか（停止条件）

ループは1回で終わりません。「完成」を定義しないと、止まらない、あるいは無駄に回り続けます。実務では、止め方を複数用意します。

* **達成条件** … 評価役の点数がしきい値を超えた／テストが通った、で止める
* **上限** … 回数・トークン・時間に天井を設ける（保険）
* **収束** … 2回連続で改善がなければ止める
* **エスカレーション** … 行き詰まったら人に投げる

止め時の設計を怠ると、次の「暴走」に直結します。

### 2. 暴走とコスト

ループは、反復と並列でトークンを一気に使います。これはコストに直結します。APIで使うなら利用料がかさみ、Claudeの有料プラン（Pro/Maxなど）で使うなら使用量の上限にすぐ当たります（いわゆる「すぐ無くなる」やつです）。「とにかく回す」と、お金や使用枠がどんどん溶けていきます。

Borisも、Claudeチームのレビューを「**ほぼ全バグを捕まえるが、とても高価**」と認めています。トークンを大量に投じる「テスト時計算（test-time compute）」だからです。だから歯止め（回数上限・予算上限）と、安価版との使い分けがセットになります。

コストの見方も変わります。Borisの助言は「昔のAIサブスクと比べるな。**エンジニアが同じ作業をしたらいくらか、と比べろ**」。回すコストそのものより、それで何時間ぶんの仕事が消えるか、で測るという考え方です。

### 3. 誤りが積み上がる（ドリフト）

ループは前の出力に次が乗ります。だから**最初の小さなズレが、後工程で大きくなります**。一発のミスより、連鎖のほうが怖い。

対策は2つ。各段で評価役に検証させること。そして「毎回きっちり同じ」が要る処理は、**AIの判断に任せずスクリプトで固定する**こと。ここはSkillの出番です（決まった手順はSkillやスクリプトに、判断はループに）。

### 4. 数が増えると見えなくなる（観測）

Borisは「1日に数百のエージェントを動かす日もある」と言います。1個なら見ていられますが、数十・数百を同時に回すと、**どれがコケたか分からなくなります**。

設計の定石は、サブエージェントを「独立したコンテキストで動かし、**結果（要約）だけ親に返す**」こと（<https://code.claude.com/docs/en/sub-agents>）。途中の大量のログは親に流さず、要点だけ上げる。人の役割は「全部を見る」から「**異常を捕まえる**」へ変わります。

### 5. 自律ほど、安全に気を配る

自律で回すほど、人の確認が間に挟まりません。だから慎重さが要ります。

Anthropic自身、確認を全部すっ飛ばす危険なモードを「ほとんどの状況で危険」とし、より安全なモードでも「**人のレビューの代わりにはならない**」と明言しています（<https://www.anthropic.com/engineering/claude-code-auto-mode>）。実際、安全寄りのモードでも危険な操作を見逃す確率はゼロではない、と公表しています。

だから現場の定石は、サンドボックス（隔離環境）で動かし、最新モデル（不正な指示に強い）を使うこと。そのうえで\*\*「何を任せ、何を人に残すか」の線引きを決めます\*\*。全自動＝安全ではありません。

## まとめ

整理します。

* **Skillは「1回をちゃんと」、ループは「それを自律で回す」。** 足されるのは自律と反復。
* **ループの心臓は評価役。** 作る→評価→直すの輪で品質が出る。作る役と評価役は分ける。
* **自律の代償が、ループ固有の難所**：止め時／暴走とコスト／誤りの蓄積／観測／安全。どれも「人が毎回見ていない」ことから生まれる。

ループを書くとは、作る役を用意することではありません。**評価役と歯止めを設計すること**です。Anthropicが数百のClaudeを回せるのは、そこを作り込んでいるからです。まずは小さな1つのループで、「ダメと言える役」を1人立てるところから始めるのが、現実的な第一歩になりそうです。
