ここ1ヶ月、Claude Code の挙動に「明らかに前と違う」違和感を感じていた。ルールを守らない、自分でやると決めたはずの作業を抜かす、同じ説明を何度も繰り返す。SNS や GitHub Issue でも同じ報告が増えていて、「気のせいではない」のは早い段階で分かっていた。

2026年4月23日、Anthropic が公式にポストモーテムを出した（[An update on recent Claude Code quality reports](https://www.anthropic.com/engineering/april-23-postmortem)）。原因はモデル本体ではなく、製品レイヤーで重なっていた3つの変更だったと判明している。

この記事では、自分が3月下旬〜4月にかけて Claude Code を使っていて感じた違和感と、公式が認めた3つの原因の答え合わせをしておく。最後に、AIエージェント製品との付き合い方として何を変えるかを書く。

## 自分が感じていた違和感

### 1. `develop` への直接 push 禁止ルールを忘れる

個人開発のリポジトリでも、`main` への直接 push と `develop` への直接 push は禁止していて、CLAUDE.md に書いてある。Claude Code には作業ブランチを切って PR を出させるフローを徹底していたのに、3月末あたりから「今 develop にコミットしました、push します」みたいな進行を平気でやるようになった。

最初は自分の指示が雑だったかと思ってルールの書き方を強化したが、同じ Claude Code に同じプロジェクトで何度も同じことを言うことになった。CLAUDE.md は読んでいるはずなのに、セッション中盤以降にルールが効かなくなる印象だった。

### 2. テストの自動生成が抜ける

実装と一緒にテストを書かせる運用にしていて、これも CLAUDE.md レベルで決めている。3月中旬まではちゃんとテストを伴って実装が進んでいたのに、いつからか「実装だけ完了させて、テストの話を一切しない」ターンが増えた。

「テストは？」と聞き返すと普通に書ける。ただ、書ける状態のはずなのに自発的に手を出さない。短く済ませようとしている雰囲気が出ていた。

### 3. 同じことを何度も説明する・選択するツールが浅くなる

長めのセッションで顕著だったのが、序盤で共有したコンテキストを途中で忘れたかのような振る舞い。同じファイルを何度も Read し直す、序盤に決めた方針を中盤で「この実装で合っていますか？」と聞き直してくる、本来 Grep で済む探索を Bash の `find` で力業に行く、といった「浅い行動選択」が増えた。

このあたりは Reddit や X でも「Opus が浅い」「Claude Code が途中で投げ出す」みたいな表現で報告されていて、明らかに自分だけの体感ではなかった。

## 公式発表：3つの原因まとめ

公式ポストモーテム（[anthropic.com/engineering/april-23-postmortem](https://www.anthropic.com/engineering/april-23-postmortem)）によると、影響していたのは Claude Code / Claude Agent SDK / Claude Cowork の3製品で、API と推論レイヤー本体は無影響。原因は以下の3つだった。

### ① reasoning effort のデフォルト変更（3/4 〜 4/7）

UI フリーズや応答遅延を緩和する目的で、reasoning effort のデフォルトを `high` から `medium` に下げた。結果として、Sonnet 4.6 / Opus 4.6 で「賢さの低下」を体感できるレベルの劣化が出た。

ユーザーから「速さより賢さが欲しい」というフィードバックが多く、4月7日に Opus 4.7 では `xhigh`、その他モデルでは `high` に戻した。

### ② キャッシュ最適化のバグ（3/26 〜 4/10）

3月26日に「1時間以上アイドルだったセッションで、古い thinking ブロックを **1回だけ** 消す」最適化が入った。狙いは、再開時の遅延削減。

しかし実装にバグがあり、**再開後に同一セッションのターンごとに毎回 thinking 履歴を消し続けた**。結果として「忘れる」「同じことを繰り返す」「奇妙なツール選択をする」といった症状が継続的に発生した。Sonnet 4.6 / Opus 4.6 に影響、4月10日に修正。

### ③ verbosity 抑制のシステムプロンプト（4/16 〜 4/20）

Opus 4.7 の冗長さを抑える目的で、システムプロンプトに「ツール呼び出し間の説明は25語以下、最終応答は原則100語以下」という制約を追加した。

これが他のプロンプト変更と組み合わさってコーディング品質を損ね、Anthropic 自身の追加評価で Opus 4.6 / 4.7 に約3%の品質低下を確認。Sonnet 4.6 / Opus 4.6 / Opus 4.7 に影響、4月20日に撤回された。

### 共通する性質

3つに共通するのは、**基盤モデルを差し替えたわけではなく、推論パラメータ・キャッシュ・システムプロンプトという "harness" 層の調整に起因している** という点だ。各変更がトラフィックの違うスライスに違うタイミングで効いていたので、ユーザー側からは「広範で一貫しない劣化」に見えた。Anthropic 側の社内 eval でも初期は再現できなかったらしい。

## 答え合わせ：自分の体感とどう対応するか

3つの違和感を、3つの原因と照らし合わせる。

| 体感した症状 | 主に対応する原因 |
| --- | --- |
| `develop` 直 push 禁止ルールを忘れる | ②キャッシュバグ（thinking 履歴消失で序盤の制約が効かなくなる） |
| テスト自動生成が抜ける | ③verbosity 抑制プロンプト（短文化志向で「自発的な追加作業」を削る方向にバイアス） |
| 同じファイルを Read し直す・浅いツール選択 | ②キャッシュバグ + ①effort ダウン |

特にしっくりきたのが ② のキャッシュバグだ。CLAUDE.md は最初に読まれるが、その後の thinking 履歴が毎ターン消されると、「序盤で踏まえたルール」が中盤以降のターンで考慮されにくくなる。**ルールを守らない** という症状そのものより、**ルールを思考に組み込んだ痕跡が消えていた** というほうが正確な記述に近い。

テスト省略のほうは ③ がフィットする。最終応答100語以下という制約が、「ついでにテストも書いておきますね」「次はこの観点でテストを足したほうがいいです」みたいな自発的な提案を抑制する方向に効いていてもおかしくない。

ここで重要なのは、自分が「Claude Code が頭悪くなった」と感じていた症状が、**プロンプト内の指示の品質ではなく harness 層の問題で起きていた** という点だ。CLAUDE.md を書き直しても、ルール文を強化しても、根本のキャッシュバグや短文化制約には勝てない。

## 学び：AIエージェント製品との付き合い方

今回の件で、自分のスタンスとして固めておきたいことが3つある。

### 1. AIサブスクの年間プランには入らない

サブスクサービスは年契約で割引が効くのが普通で、Claude も Pro / Max の年間プランは月払いの2割引前後になる。ただ、AI 製品の「現在の出来」は数週間単位で動く。

直近1ヶ月だけでも、

* 4月13日〜23日の品質低下と修正
* 4月21日の Pro プラン Claude Code 削除騒動とその撤回（[別記事](https://zenn.dev/pannda/articles/claude-code-pro-plan-removal-2026-04) で詳述）
* Opus 4.7 投入によるセッション長期化

と、提供条件・体感品質・課金構造の3軸が同時に動いている。**年契約は12ヶ月分の前提を一括で固定する行為** なので、変化が早い領域とは相性が悪い。月払いのほうが、競合への乗り換えやプラン変更の機動力を残せる。

### 2. Codex / Gemini CLI など他サービスの素振りを続ける

ベンダーロックの怖さは、**問題が起きた時に逃げ場の選択肢を持っているかどうか** で決まる。今回みたいに「公式がポストモーテムを出して全サブスク利用上限をリセットした」案件は補償付きで終わったが、もし Anthropic 側が「既存 Pro からも Claude Code 除外」みたいな路線変更に踏み切ったら、即座に乗り換え先が必要になる。

OpenAI Codex は Plus（$20）でも継続利用可能で、Codex CLI を含めて開発ワークフローに乗せやすい。Gemini CLI も Google が無償枠を厚めに切ってきている。今すぐ全面移行する必要はないが、**月に数日でも他のCLIで実装を回してみて、Claude Code との差分を体に入れておく** のは保険として効く。

「Claude Code に最適化したワークフロー」をそのまま他に移植するのは無理でも、自分のリポジトリで Codex に同じタスクを振ってみる程度のことは、習慣化しておきたい。

### 3. ユーザー側の防衛策をどう設計するか

ポストモーテムから引き出せる教訓として、ユーザー側でできる対策も整理しておく。

* **重要なルールは CLAUDE.md だけでなく、フックやテンプレに固定する**: thinking 履歴が消える系の不具合に備えて、毎ターン強制的に効くレイヤー（pre-commit フック、PR テンプレ、CI チェック）に回せるものは回す
* **長時間セッションを過信しない**: thinking 履歴依存の作業は、セッションを区切って都度コンテキストを渡し直すほうが事故が少ない
* **症状の記録を残す**: 違和感を感じた時点でいつ・何が・どう変わったかをメモしておくと、ポストモーテムが出たときに自分の体感の解像度が上がる

## まとめ

* 2026年4月23日、Anthropic が3月4日〜4月20日の Claude Code 品質低下について公式ポストモーテムを公表
* 原因は **基盤モデルではなく harness 層の3つの変更**: ①reasoning effort のダウン ②thinking 履歴のキャッシュバグ ③verbosity 抑制プロンプト
* 自分が体感していた「ルール忘れ」「テスト省略」「浅いツール選択」は、それぞれ ②③①+② で説明できる
* AIエージェント製品は変化が早いので、**年間プランには入らず、Codex / Gemini など他サービスの素振りも続けて、いつでも受け身を取れる体勢を作っておく** のが個人開発者としての落とし所

Anthropic は再発防止策として、社内スタッフに public build の Claude Code を使わせる運用への切り替えと、システムプロンプト変更ごとのモデル別評価とアブレーションの徹底を表明している。誠実な対応で、補償として全サブスクの利用上限もリセットされた。

ただ、この手の harness 層の事故は今後も起きうる。「公式が認めるまで自分の体感を信じていいか分からない」状態を減らすには、ユーザー側で観測と乗り換え可能性を維持しておくしかない。

Claude Code は引き続き重宝するし、今回も結果的には信頼を回復した。だからこそ、**1社のサブスクに賭けすぎない** 設計で付き合いたい。
