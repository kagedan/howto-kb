## `subagent_type: "fork"` は本当に存在するのか

Claude Codeでサブエージェントを起動するとき使う `subagent_type` に、`fork` という値があるらしいという話を見かけた。実在するのかを確認してみると、通常のエージェント起動が持つ公開エージェント一覧には出てこない。

一方でワークフロー機能内部のサブエージェント起動関数のオプション（agentType）の説明にはこう書かれている。

> opts.model overrides the model for this agent call ... **Ignored for subagent\_type: 'fork' — forks always inherit the parent model.**

つまり `fork` は独立した公開エージェント種別ではなく、複数エージェントを連携させるワークフローの内部で使う特殊な `agentType` の一つとして実在する。名前が示す通り「親エージェントのcontextをそのまま引き継いで分岐する」挙動を持ち、その帰結として **モデルの上書きが無視される**（親と同じモデルを強制的に継承する）という一点だけが仕様として明記されている。

これは裏を返すと、それ以外の通常のサブエージェント起動は**親の会話を一切見ない**独立起動だということでもある。この非対称性が今回の話の出発点になった。

## `fork`の優位性: contextの継続はタダではない

`fork`でない通常のサブエージェント起動は、親の会話履歴を一切引き継がない。必要な情報はすべて呼び出し側が**手動でテキストとして切り出して**プロンプトに埋め込む必要がある。

これは裏を返せば、`fork`の優位性は「その手間がゼロになること」に尽きる。

* 直前の議論の文脈、既に握った前提、途中まで積んだ推論を説明し直す必要がない
* 呼び出し側が「何を渡すべきか」を都度判断するコストが消える
* 浅く・短く継続するだけのタスクなら、フルの会話履歴を持ったまま続きをやらせるのが最速

つまり `fork` は「**同じ推論スレッドをそのまま伸ばしたい**」場面に強い。

## `fork`を使うことの問題: context rot がそのまま伝播する

ここに落とし穴がある。長く続いた会話には、脱線・矛盾する仮説・却下した案・ノイズになった過去のツール出力が積もっていく。いわゆる **context rot**（コンテキストの劣化）だ。

`fork`は「contextを一切フィルタせずに継承する」経路なので、rotもそのまま子エージェントに引き継がれる。これは通常の1セッションが長くなるほど劣化していく現象と、本質的に同じ速度・同じ仕組みで子エージェントにコピーされる。fork of fork of fork のように連鎖させると、rotは複利的に蓄積する。

これに対する解毒剤が **蒸留 (distill)** という発想だ。生の履歴を「決定事項・現在の状態・制約・未解決の問い」だけの圧縮版に落とし込み、そこからforkする（あるいはそこから新規に起動する）ことで、rotをリセットする。

まとめるとこの時点で3つの経路が見えてくる。

| 経路 | contextの扱い | rotへの態度 |
| --- | --- | --- |
| `fork` | 親のcontextを丸ごと継承 | フィルタなし、rotも継承 |
| distillしてから継続 | 圧縮済みのチェックポイントから再開 | rotに対する防波堤 |
| fresh spawn（自己完結プロンプト） | 継承ゼロ、呼び出し側が手動で選別 | 強制的な手動distill |

## 実例: 多段オーケストレーションはなぜforkを避けるのか

この整理は、実際に大きなタスクを「分解 → 並列実装 → 検証」のように多段のサブエージェントへ委譲する仕組みを触っていて実感したものだ。こうした仕組みでは、各段階（分解担当・実装担当・検証担当）は毎回まっさらな新規起動で、呼び出し側が必要な情報（触るべきファイル、完了条件、参照すべき規約など）を**手動でテキスト化して**渡すように作られている。段階が変わるたびに強制的な蒸留のゲートを通す形になっており、`fork`は使われない。

理由は単純で、呼び出し側（=会話全体を持つメインループ）のコンテキストが長くなると応答が遅くなりコストも上がる。各段階を独立起動することで、「重い作業の出力（ビルドログ・差分全文など）がメインループのコンテキストに積み上がらない」という効果を狙っている。

さらに興味深いのは、こうした仕組みが**contextを介さない第3の経路**も併用している点だ。実装担当が書いた成果物を検証担当がどう知るかというと、両者の会話は繋がっていない代わりに、ファイルシステムや外部の状態ストアを共有メモリとして使う。検証担当は実装担当の思考過程を継承する必要がなく、実際の成果物と完了条件さえ読めれば判定できる。これは「rotを薄めて渡す」のではなく「そもそもcontextに依存しない」解決策だ。

同じ発想は、長時間の同一セッションを継続する場面にも当てはまる——会話が育ちすぎたら検出して圧縮し、必要な部分だけ復元するという「蒸留」の仕組みは、エージェント切り替えの瞬間（forkの分岐点になり得る場所）だけでなく、単一セッションの継続にも効いてくる。

## forkを使ってはいけない場面: 監査・検証

ここまでは「contextをどれだけ引き継ぐか」という**量**の話だった。しかしもう一つ、量では解決できない軸がある——「**誰の**推論経路を引き継ぐか」という質の問題で、これは監査・検証・レビューの場面で顕在化する。

前述のような多段オーケストレーションでは、この問題を正面から扱う設計がよく見られる。たとえば、検証担当に使うモデルを実装担当と**あえて変える**、というルールだ。狙いは共有ブラインドスポットの排除——同じモデルなら同じ癖・同じ見落としを持ち込みやすいので、意図的に別モデルへ振り分ける。

モデルの一致すらわざわざ避けるということは、それより遥かに強い共有——contextそのものの継承——であるforkを、実装担当から検証担当へ使うことは設計上ありえない。理由は次の2点に集約される。

1. **共有ブラインドスポット**: 実装担当が「これで正しい」と自分を納得させた推論の癖・見落としを、そのまま検証担当も引き継いでしまう。distillしてcontextを圧縮しても、圧縮された中身が実装担当自身の正当化ロジックなら意味がない。
2. **自己確認バイアス**: 実装担当の会話には「なぜこの実装で良いか」という説得的な文脈がすでに積まれている。そこからforkすると、検証担当は中立な第三者ではなく「既に説得された延長」になってしまう。

同じ理由から、N人の独立したレビュワーによる反証パネルや、複数の独立サンプルによる多数決といった手法も、独立性そのものが価値の源泉になっている。forkでcontextを共有した瞬間、N個のサンプルが同じ誤りに相関して倒れてしまい、独立サンプルによる多数決という手法そのものが無意味になる。

## forkの妥当性そのものも、いずれ測るべき対象になる

ここまでの話は「forkを使うべきか」を、context rotや監査独立性という**質的な理由**だけで判断してきた。しかしforkを既定の選択にすることが本当にコスト最適か——fork自体のオーバーヘッド、prompt cache共有の効果、実行時間——は、定性的な確信だけで決め切ってよい話ではない。

こうした「モデル選択」や「実行戦略の選択」を、実績データから決定論的に決める仕組みを持つ基盤であれば、fork/inlineの選択も同じ枠組みで将来学習させたい、という発想は自然だ。ただし比較データがまだ無い段階では、いきなり自動判定ロジックを実装するのは筋が悪い。まず「実行ごとにコスト・所要時間・どちらの戦略を使ったか」を記録できる最小限の計測基盤だけを足し、十分な実績が貯まってから既定を見直す——というのが妥当な順序になる。

ただしこの「measureしてから判断する」という発想自体、計測ループの設計を誤ると同じ種類の脆さを持ち込みかねない。

1. **既定バイアスが比較サンプルを枯渇させる。** 既定を常にforkにして例外を稀にしか許さない運用だと、inline実行側のサンプルがいつまでも十分に貯まらない可能性がある。
2. **記録ステップが手動でゲートが無い。** 実行後に結果を記録するかどうかを呼び出し側の「思い出し」任せにすると、これを強制する決定論的な仕組みが無い限り記録漏れが起き続けても誰も気づけない。
3. **自由記述のラベルは表記ゆれに弱い。** `fork`/`Fork`のような表記のブレが、後の閾値判定に静かに混入しうる。

つまり「forkを使うかどうかを最終的にはデータで決めたい」という発想自体は正しいのに、そのデータを集める仕組みそのものが、これまで議論してきたのと同じ種類の脆さ（判断をLLMの記憶に依存させる・既定バイアスが分布を歪める）を持ちうる、という入れ子構造になっている。

## まとめ: 使い分けの指針

* **forkが向く場面**: 同一の推論スレッドをそのまま伸ばしたいだけの、浅く短い継続作業
* **distillを挟むべき場面**: contextが育ってきた／矛盾や行き止まりが混ざってきた継続作業。生の履歴から直接forkせず、圧縮済みのチェックポイントを経由する
* **fresh spawn（fork厳禁）が必須な場面**: 独立したサブタスクの並列実行、そして何より**検証・監査・反証**の役割全般
* **既定の選択自体も、いずれデータで検証し直す対象**: 「forkを既定にする」という判断すら、本当にコスト最適かはまだ定性的な仮説にすぎない。計測基盤を先に足し、実績が貯まってから既定を見直す——ただしその計測ループ自体（既定バイアスによるサンプル偏り・手動記録の未強制・自由入力の表記ゆれ）にも新しいrotの入り口があることを忘れてはいけない

監査の価値は「対象と違う視点から見ること」そのものにある。forkは「違う視点」を構造的に潰してしまうため、distillでは代替できない別の制約として扱う必要がある。そして、たとえ質的な理由でforkを選ぶのが正しいとしても、その選択が本当にコスト最適かは別問題で、いずれ実績データで裏を取る必要がある——ただしデータを集める仕組み自体も、判断をLLMの記憶に委ねたり既定バイアスで分布を歪めたりすれば、また同じ種類の脆さを持ち込んでしまう。逆に言えば、`fork`というオプション一つを深掘りするだけで、「contextをどれだけ渡すか」「独立性をどう担保するか」「その判断自体をどう検証し続けるか」という、エージェント設計の3つの軸がくっきり見えてくる。
