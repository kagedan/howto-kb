AIエージェントで事業を回す記事は増えました。でもそのほとんどは、判断を間違えても誰も損しない世界の話です。

僕はパチスロと競馬のデータ事業を、4つの職能を持つAI組織に運営させています。判断を間違えると翌日の実弾——自分の金——が飛びます。

5ヶ月で事故を65回起こし、そのたびにルールを書き、売上は¥0のまま今日も動いています。これはその記録の入口です。

---

## 何をやっているのか

組織の全体像はこうなっています。

```
┌──────────────────────────────────┐
│          CEO（僕・人間）            │
│    最終判断 / 声入れ / 実弾投下     │
└────────────┬─────────────────────┘
             │
┌────────────▼─────────────────────┐
│        COO     （AI・司令塔）      │
│  戦略 / タスク管理 / 全PJ横断調停    │
└──┬─────────┬──────────┬──────────┘
   │         │          │
┌──▼──┐  ┌──▼──┐  ┌───▼───┐
│編集長│  │分析官│  │ 法務  │
│原稿  │  │KPI  │  │安全  │
│品質  │  │監視  │  │チェック│
└─────┘  └─────┘  └───────┘

日次ジョブ12本/日
意思決定記録 ADR 65本
```

CEOは僕です。最終判断と、実際に店に座って打つのは人間がやります。それ以外——戦略の整理、タスク管理、原稿の執筆、数値の監視、公開物の安全チェック——はAIの4職が回しています。

COOが司令塔です。毎日の日次ジョブ12本で、KPIの集計、法務のチェック、日報の生成などが自動で走ります。

事業は2つあります。

* **パチスロ**: 過去データの分析をもとに、着席時点の期待値を最大化する方式で台を選んでいます。当初は機械学習モデルで予測していましたが、精度と実戦成績が一致しない問題にぶつかり、方式を切り替えました。7月は7戦で−19,000円
* **競馬**: 三連複のAI予測。検証データでの平均回収率156.6%。実馬券で運用中

パチスロは赤字です。機械学習での予測を5ヶ月試して、最終的にMLを捨てて過去データ分析に切り替えた——その経緯自体がこの連載の素材になります。競馬は検証段階で数字が出ていますが、実運用の結論を出すにはまだ早い状況です。

## いくらかかるのか

コスト構造をそのまま晒します。

* **AI基盤**: Claude Codeの定額サブスクリプション。(MAX5x)
* **実弾**: パチスロ7月は7戦で−19,000円。競馬は実馬券をIPATで運用中
* **売上**: **¥0**（2026年7月時点）
* **公開物**: Kindle 1冊（¥980）、noteマガジン1つ（¥1,980)、Zenn記事、X

定額の枠内に4つの職能と12本の日次ジョブが収まっています。その上で売上が¥0というのが、今の事業の姿です。

「月○万円稼いでます」と書き出すほうが記事としては強いです。でもそれは嘘になります。

## 何が壊れたか

代表的な事故を1つだけ、詳しく書きます。

2026年6月25日、AIが存在しないスクリプト2本の「分析結果」を具体的な数値付きで報告してきました。据え置き相関 r=+0.04、事前期待値+0.45 vs 当日+1.34。どれも架空でした。(色々調査しましたが、opus4.8特有のバグが濃厚)

このデータは、翌日僕がどの台に座るかの判断材料として使う予定でした。嘘の数字で実弾の判断をするところでした。

発覚したのは「このスクリプト、いつ作った？」と聞いたからです。存在しませんでした。

同日中に「報告の誠実性」を最上位の規範として明文化しました。「ツールを実行していない結果を生成してはならない」。これが65本のうちの1本になりました。

他にも壊れたことは山ほどあります。

* ディスコード秘書システムの連絡が3日連続で届かなかった。3日とも壊れ方が違った
* 予測モデルの1位評価が、2位に実績で負けていた
* 新規事業として仮想通貨Botに挑戦して、バックテストの段階で全敗した。

この連載では、これらを1つずつ掘っていきます。

## なぜギャンブルをAIに任せたのか

人の判断が介入するよりも、AIを判断軸にした方が機械的に勝てると思いました。

パチスロには公開されているデータがあります。でもそれを毎日集計して、どの台が狙い目かを判断するという作業は、人間がやるとものすごくボトルネックになります。集計代行のサービスもあるくらいで、みんな同じところで詰まっています。

エージェントAIが出てきたとき、そのボトルネックが外れると思いました。集計だけじゃありません。台選びの判断そのものを、人間の勘ではなくデータに基づく機械的な判断に置き換えられるならそのほうが再現性が高い。  
人の判断には人間特有の"ブレ"みたいなものがあります。ギャンブルしてる人ならわかってくれると思いますが、、、疲れていれば甘くなるし、前日の結果に引きずられる。機械にはそれがありません。

2026年2月、Googleのantigravityでダッシュボードを作り始めました。店舗ごとのデータを可視化して、どの台が狙い目かを予測する。最初はそれだけのツールでした。

3月にClaude Codeに移行して、ツールが組織に変わり始めました。組織を作ろうと計画したわけではありません。COO（司令塔）だけ置いたら、事故が起きるたびに役割が増えていきました。note原稿を書く編集長、数値を監視するアナリスト、公開物の安全チェックをする法務——問題が起きるたびに職能が生まれました。

計画の結果ではなく、事故の集積です。

## 壊れたらどう直してきたか

ADR——Architecture Decision Recordとウチでは命名して使っています。議事録みたいなものです。事業上の判断を1件1ファイルで残す方式で、65本の意思決定記録があります。1本ごとに事故か発見が起点になっています。

最初の3本を例に出します。

* **ADR-004**: 「北極星となる指針がないとAIチーム全員の判断がぶれる」ことが分かりました。Mission（データの非対称性で、個人が組織に勝てることを証明する）を言語化しました
* **ADR-005**: 「1人で全部やると深さが出ない」ことが分かりました。4つの職能に分けました
* **ADR-008**: 「3つのプロジェクトに同じAIが散らばって3人いる状態」になっていました。1人の司令塔に統合しました

最初の3本だけで、もう3つの問題を踏んでいます。

このサイクル——事故が起きる、記録を残す、ルールが増える、組織が変わる——を65回繰り返しました。同じ轍を踏まないための仕組みとしてADRを使うのは、AI組織に限った話ではありません。どのプロジェクトでも使えます。大事なのは「壊れたことを書いて、それが2度と起きないようにドキュメンテーションする」という習慣のほうです。

## 売上¥0でも続けている理由

売上¥0はあくまでnoteの売上です。事業全体の成否はまだ出ていません。

ただ正直に言えば、今の数字だけ見ればどこにも勝っていません。パチスロは赤字、noteは¥0、Xのフォロワーも0。数字だけなら「やめたほうがいい」と言われても仕方がない状況です。

それでも続けているのは、いつか同じようなことを考えている人の役に少しでも立ちたいからです。

「AIに事業を任せたらどうなるのか」を、うまくいっている部分だけでなく、壊れた部分も含めて記録している人はあまりいません。うまくいったら「こうすれば勝てます」と書けばいい。でもうまくいっていない今の段階で、壊れ方と直し方を残しておくことにも意味がある——少なくとも僕はそう思っています。

¥0の真っ只中で書くのと、うまくいってから「最初は¥0でした」と振り返るのは違います。今書かないと、後から書いても嘘になります。

## この連載で書くこと

この連載は、ギャンブル事業をAI組織に運営させている過程の記録です。

具体的には:

* **個別の事故と、そこから生まれたルール**: 捏造未遂、停止命令無視、3日連続の通知不達。壊れ方と直し方を1本ずつ
* **コストと売上のリアルタイム記録**: 定額枠にどれだけの組織が収まるか。そして売上は立つのか立たないのか
* **機械学習を捨てるまでの記録**: 精度と成績が一致しない問題、1位が2位に負ける問題。5ヶ月試して方式を切り替えた判断の裏側

同じようなことを考えている人がいたら——AIを使って何かを回したい、でもどこが壊れるのか分からない——この記事が少しでも材料になればいいなと思っています。
