私たちは Codens という AI 開発スイートを作っている Corevice のチームです。Codens 自身の開発にも Codens を使っており、**マージされた PR の 65% が AI 生成**です(GitHub API による実測値。<https://www.codens.ai/stats/> で週次更新のダッシュボードを公開しています)。PR 作成からマージまでの中央値は 2 分、93% の PR が 1 時間以内にマージされています。

「AI に任せたらそんなに速く回るわけがない」と思われるかもしれません。実際、モデルの賢さだけではこの数字にはなりません。この記事では、この運用を支えている 3 つの構造——**決定論的ワークフロー(Purple)× レビューゲート(Orange)× 自動修復(Red)**——の設計を解説します。宣伝というより、「AI エージェントに SDLC を任せるとき何を構造で縛るべきか」の設計メモとして読んでもらえれば幸いです。

## 設計原則:「次に何をするか」を AI に委ねない

自律エージェントの典型的な失敗は、タスク遂行の順序判断までモデルに委ねてしまうことです。実行のたびに経路が変わり、再現も監査もできなくなります。

Codens の実装ハブである Purple Codens は、この判断を**ステップベースの状態機械**に固定しています。ブランチ作成 → 作業環境準備 → AI による実装 → push と PR 作成 → E2E テスト → デプロイ確認・通知、という進行はワークフロー側が決定論的に制御し、AI はステップ内のコード生成だけに集中します。

重要なのはゲートステップの扱いです。実装検証(`verify_implementation`)のようなゲートは、**失敗時に最大 2 回まで自動リトライ(retry cap)し、それでも解消しなければ人間にエスカレーション**します。つまり「無限に頑張る」ことが構造的にできません。そして PR のマージは**人間レビュー待ち(Waiting Human)がデフォルト**で、auto-merge はリポジトリ単位のオプトインです。全ステップの実行はライブセッションビューで逐次確認でき、完了後は監査ログとして残ります。

## レビューゲート:マージ前に立つもう一人のレビュアー(Orange)

実装が自動化されると、今度はレビューがボトルネックになります。そこでマージ前の品質ゲートを担うのが Orange Codens です。

* PR が作成・更新されると、差分をコンテキスト込みで解析し、バグ・競合状態・null 参照・リソースリーク・設計リスクを**重大度付きの finding** としてコメントする
* SCA/SBOM による依存ライブラリの脆弱性監査とシークレット検出を、コード品質レビューと同じゲートに統合する
* 重大度の高いセキュリティ指摘は、**Opus モデルによる二段階の Verify pass** を通してから通知する(低確度の指摘はサマリに畳み、inline コメントは重大なものに絞るノイズ設計)

検出した finding はコメントで終わりません。Purple / Red に **handoff** され、修正 PR が自動生成され、Orange が再レビューして解消を確認したら APPROVE します。この handoff → 修正 PR → APPROVE → 自動マージのループは**平均約 8 分で 1 周**します。

自動化同士が呼び合うループには発振のリスクがあるので、**Bot 由来の PR はループガードで除外**し、handoff はマージ前には suggestion ゲート、自動起票は PR マージ時、と役割を分けています。

## 自動修復:エラー検知そのものをトリガーにする(Red)

3 つ目の構造は、本番エラーの処理です。Red Codens は人間の指示ではなく、**Sentry や CI からの Webhook がそのまま起動トリガー**になります。

1. エラーを Webhook で受信(Sentry / GitHub Actions / CI / 自社監視ツール)
2. Claude API(Sonnet)が 6 ステップの体系的デバッグ手法でスタックトレースを解析。依存関係グラフから**関連ファイルを最大 100 件まで動的に取得**して根本原因を特定
3. 影響範囲を最小化した差分で修正コードを生成
4. プロジェクト指定のテストコマンドで検証。失敗したらテスト出力をフィードバックとして**最大 3 回まで自動再生成(3-Retry)**。リトライ時は不足ファイルも追加取得
5. テストが通った修正だけが GitHub PR になり、Slack に通知

ここでも「自動マージはしない」が徹底されています。**修正 PR は必ず人間レビューを経てマージするのがデフォルト**です。社内実績では 1 件あたり約 3 分で処理し、成功率は約 92%。裏を返せば 8% は失敗するので、だからこそ「テストを通った修正だけが PR になる」「マージは人間」という 2 つのフィルタが要ります。

さらに QA 側では、Blue Codens(Playwright E2E の自動生成・実行)がテスト失敗を検知するとバグを Red に自動起票し、Red の修正 PR を Blue が再テストする、という検知 → 修正 → 再検証のループが無人で回ります。

## 横断の安全装置:暴走を「構造的に」不可能にする

3 製品に共通する横断設計も、公開している範囲で紹介します。

* **リトライ上限はモデル別**: fix/verify ループにはモデルファミリーごとのリトライ上限がデフォルトで設定されています(例: Claude 系 3 回、Qwen 系 6 回)。無限ループは構造的に発生しません
* **コストの可観測性**: 全タスクのクレジット消費は Billing Control Plane に記録され、ダッシュボードでリアルタイムに確認できます。プロジェクト単位の月次上限も設定可能です
* **ルールの二重化**: リポジトリの CLAUDE.md / .claude/rules/ は常にプロンプトへ注入されます。加えて、命名規約・import 順・禁止 API のような機械判定可能なルールは、AI の判断とは独立した**決定論的な enforcement レイヤー**がマージ前にブロックします
* **クレデンシャル分離**: API キーや DB 接続情報は AWS Secrets Manager に置かれ、各タスクはタスクスコープの IAM Role 経由でのみアクセスします。AI に平文の資格情報は渡りません。本番 DB への破壊的操作(DROP / TRUNCATE / 大量 DELETE)はワークフローレベルでガードされ、明示的な承認が必要です
* **追跡可能性**: 全操作は組織単位のテナント分離のもとで監査ログに記録されます。「AI が裏で何をしたか分からない」状態を作らないことが、人間がマージ判断を持ち続けるための前提になっています

## まとめ

Codens が Codens を作る、というドッグフーディングを続けて得た結論はシンプルです。**AI エージェントの自律性は、モデルの賢さではなく構造で制御する。**

* 進行の判断はワークフロー(状態機械)が持ち、AI はステップ内に閉じ込める
* ゲート(verify chain・レビュー・テスト)を通過したものだけが先に進み、リトライには必ず上限を置く
* デフォルトは常に「人間のマージ待ち」。自動化はオプトインで広げる

この構造の上でなら、マージ PR の 65% を AI が書いても開発は壊れませんでした。同じ問題を考えている方の参考になれば嬉しいです。
