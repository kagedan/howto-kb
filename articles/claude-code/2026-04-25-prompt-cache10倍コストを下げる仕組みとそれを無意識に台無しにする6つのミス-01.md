**シリーズ: Claude Codeのトークン節約術**

## 思わず姿勢を正したくなる数字

第1回では、作業の習慣を変えることでトークン消費を削る話をしました。今回はまったく別の角度からのアプローチです。削るのではなく、**トークンそのものを10倍安くする**という話。理屈に合わないように聞こえますが、以下の数字を見れば納得していただけるはずです。

Claude Sonnet 4.5の実際の料金はこうなっています。

* **通常のinput：** $3.00 / 1M token
* **Cache write (TTL 5分)：** $3.75 / 1M（通常の約1.25倍）— 初回キャッシュへの書き込み
* **Cache read：** $0.30 / 1M — **ちょうど10倍安い** — 2回目以降のキャッシュからの読み込み

現実的なセッションで考えてみます。固定コンテキスト100Kトークン（長いコードファイルや設計ドキュメントなど）で20ターンのやり取りをする場合。キャッシュを使わなければ、毎ターンそのコンテキスト分を支払うことになります。

* キャッシュなし：20 × $0.30 = **$6.00** のinputコスト
* キャッシュあり：$0.375（初回write）+ 19 × $0.03（以降のread）= **$0.945**
* 削減率：**84%**

しかも損益分岐点はわずか**2ターン目**。3ターン目以降は、ほぼ無料の高速道路を走っているようなものです。

一番嬉しいのは、**設定作業が一切不要**な点。Claude Codeが自動的にキャッシュのマーカーを配置してくれます。ただし仕組みを理解すれば、「たまたまキャッシュが効いた」状態から、**意図的にコストを下げる仕組み**へと変えられます。

## 仕組み：キャッシュはprefix単位で動く

キャッシュをイメージするなら、ノートのようなものです。Claudeのサーバーはinputのすべて — tool definitions、system prompt、会話全体 — を一本の長い文字列としてノートに書き込んでいます。次のリクエストが来たとき、先頭から前回のセッションと同じかを照合します。同じなら差分だけ課金。違えばゼロから計算し直しです。

覚えておくべきルールはひとつだけ。**inputの先頭が前回と同じ文字列なら cache hit（再利用）。1文字でも違えば、そこから先は cache miss（全額課金）**。

リクエストを送るたびに、Claude Codeは以下のパーツを固定順で連結した一本の長い文字列を組み立てます。

```
[Tool definitions] → [System prompt (CLAUDE.mdを含む)] → [Messages (会話)]
```

サーバーはこの文字列を前回のものと先頭から比較し、最初に差分が出た地点までをキャッシュヒットとして扱います。そこから先は再計算です。この**階層構造**を理解することが鍵になります。

| 変更される階層 | 影響範囲 |
| --- | --- |
| **Tool definitions**（MCPの追加、skillのON/OFF） | 🔥 **3階層すべて**を破壊 |
| **System prompt**（CLAUDE.mdの修正、モデル変更） | System + Messagesの2階層を破壊 |
| **Messages**（新しいメッセージ追加） | 変更地点以降のみ破壊 |

この階層構造を把握しておくことが重要です。一見無害に見える操作が、実は上位階層のキャッシュを丸ごと吹き飛ばしているケースが非常に多いからです。詳細はアンチパターンの章で解説します。

### TTL：キャッシュは永遠には残らない

キャッシュには弱点があります。永久保存ではないのです。アクティビティが途切れると、サーバーはリソース節約のためキャッシュを破棄します。このTTL（Time To Live）はプランによって異なります。

| プラン | TTL |
| --- | --- |
| Pro / API | **5分** |
| Max | **1時間** |

メッセージを送信してキャッシュの読み込みが成功するたびに、TTLカウンタは**0にリセット**されます。しかしコーヒーを淹れに離席したり、電話に出たり、ちょっとしたミーティングを挟んだりして、Proプランで6分経過してしまえば、キャッシュは消えています。次のターンはゼロから再計算 — キャッシュ読み込みと比べて約6倍高くつきます。

Proプランには小技があります。5分経過する前に軽いメッセージ（「OK、続けて」など）を送ってTTLをリセットするという方法。美しいやり方ではありませんが、どうしてもキャッシュを保持したい場面では機能します。一方Maxプランは話が別で、30分席を外してから戻ってきてもキャッシュは健在です。じっくり考える癖があったり、中断が多い働き方をしているなら、MaxがProより本当に割に合う理由のひとつです。

## キャッシュの「育て方」：4つのテクニック

ここからが実践編です。キャッシュを最大限に活用する方法を紹介します。限られたトークン予算で働いている人たちが使っている、4つのテクニックです。

### テクニック1 — すべてを最初のメッセージにフロントロードする

思い出してください。キャッシュは先頭（prefix）から比較されます。制約を後出しで追加していくと、先頭部分はキャッシュ済みでも、追加された新しい部分は毎回再計算になります。結果として、キャッシュの効果が本来の威力を発揮しません。

まずは**悪い例**から。

```
Turn 1: "Form.tsxをリファクタして"
Turn 2: "あ、新しい依存は追加しないで"
Turn 3: "あと、testsもグリーンで"
Turn 4: [スクリーンショット添付]
```

毎ターンprefixが少しずつ伸びていくので、キャッシュはブロックごとに部分的にしか効かず、全体最適にはなりません。

**正しい例**はこうです。

```
Turn 1:
Goal: Form.tsxをリファクタする。
Constraints:
- 新しい依存は追加しない
- 既存のtestsをグリーンのまま保つ
Done when: すべてのtestsがpassする。
[スクリーンショットはメッセージ末尾に配置]
```

コンテキスト全体が最初からキャッシュに乗ります。以降のターンは末尾に1〜2行追加するだけで済みます。

**あまり知られていないコツ：** 画像は**それより後ろにある全ての内容のキャッシュを破壊します**。なのでスクリーンショットはメッセージの最終行、テキスト全部の後に配置するのが鉄則。この配置ならテキスト部分（価値の99%を占める）のキャッシュはそのまま残ります。

### テクニック2 — Batching：複数の質問を1セッションに集約する

100Kトークンの大きなファイル群に対して、5つの質問をしたい状況を想像してください。

**悪い方法** — 5つのセッションを別々に開く：

* それぞれでcache writeが発生 → 毎回1.25倍のコスト
* 合計：5 × 1.25倍 = **6.25倍**のファイル読み込みコスト

**良い方法** — 1セッションにまとめる：

* コスト：1.25倍（初回write）+ 4 × 0.1倍（以降のread 4回）= **1.65倍**
* 約**3.8倍安い**

原則はとてもシンプルです。**コンテキストを読み込んでキャッシュに乗せたなら、離席する前にできるだけ使い切る**。

実践への応用：

* **コードレビュー：** レビュー用の質問をすべて1セッションにまとめる。ファイルごとに新セッションを開かない
* **本番ログの分析：** ログを1回読み込んだら、perf、errors、patternsなど複数の観点で同じセッション内で質問する
* **注意点：** 「念のため」でファイルを読み込まないこと。そのセッションで使わなければ、cache writeのコストだけ払って何もリターンが得られません

### テクニック3 — autocompactを待たずにRolling summarizationする

長いセッションはよくこういう状態に陥ります。50ターン目あたりで、**コンテキストの80%以上がもうゴミ** — デバッグは終わった、議論も決着済み、機能も完成済み。Claude Codeは167Kトークンに達すると自動圧縮（autocompact）してくれますが、そのタイミングで主導権を失います。何が残り何が捨てられるかは、システム任せになるのです。

**より良いパターン：** `conversation_memory.md` というファイルを用意し、10ターンごとくらいに能動的に更新する。

```
## Completed ✅
- authミドルウェアのリファクタ完了
- /api/ordersのunit testsがpass

## Current Task 🔄
- workerキューのrace conditionをデバッグ中
- 試したこと：timeoutを延長（不十分）、retryロジック（duplicate発生）

## Key Decisions
- pollingではなくRedis pub/subを採用
- authはmiddlewareに集約、分散させない

## Constraints
- Node 16との互換性必須
- public APIは変更しない
```

更新タイミングは**120Kトークン付近**がベスト。167Kの自動圧縮閾値に達する前です。実際にこの運用をしているユーザーの数字では、突然の自動圧縮が86%減少。40ターン超のセッションコストは $13.24 から $4.67 に — **65%削減**という結果が出ています。

なお、`/compact` は進行中のセッションをその場で圧縮するコマンドです。ただし別のアプローチとして**Context snapshot**というやり方もあります。新しいセッション用に `task_context.md` で要点をまとめたファイルを作り、新セッションでは `@task_context.md` と参照するだけ。147Kトークンを読み込み直していたところが、わずか847トークンで済む。99.4%削減です。

### テクニック4 — 並列調査のためのfork session

よくあるもうひとつのシチュエーション。同じコードベース上で3つのバグを同時に発見し、並列で調査したい（DBタイムアウト、auth 401、メモリリーク）。反射的に3つの独立セッションを開きたくなりますが、それは大きな間違い。それぞれでファイルコンテキストのcache writeが発生してしまいます。

正解はこうです。

```
# Step 1: baseセッション — 共有コンテキストを一度だけ読み込む
claude
> Read src/db/queries.ts, src/api/routes.ts, src/middleware/auth.ts
/rename bug-investigation-base

# Step 2: 3方向に並列fork（各ターミナルで実行）
# Terminal 1:
claude --resume bug-investigation-base --fork-session
> "productionログ内のDBタイムアウトを調査して。"

# Terminal 2:
claude --resume bug-investigation-base --fork-session
> "token expiryが401の原因になっているか分析して。"

# Terminal 3:
claude --resume bug-investigation-base --fork-session
> "workerプロセスのメモリリーク源を特定して。"
```

3つのforkはすべて**同じキャッシュを共有**します。prefixが同一だからです。コスト内訳：

* Base：1.25倍（cache write 1回）
* 3 fork：3 × 0.1倍（各cache read）
* 合計：1.55倍（3 × 1.25倍 = 3.75倍の代わりに）
* 3つの並列調査で**約2.4倍節約**

forkは元のセッションを無傷で残します。いつでも同じチェックポイントに戻って再度forkできる。複数ターミナルでの同一セッションresumeより、`--fork-session` を優先すべきです。resumeはメッセージ順序が乱れがちですが、forkはクリーンに別ファイルを切ります。

### ボーナステクニック：multi-turn API呼び出しのキャッシュ

Claude Code CLIではなく、Anthropic APIで自前のアプリを作っているなら、ひとつコツがあります。**最後のメッセージを除く全メッセージ**に `cache_control: {"type": "ephemeral"}` を付ける。最後のメッセージを除外する理由は、それがまだキャッシュに載っていないから — 再利用できるのは過去のターンの内容だけです。

具体例を出します。10ターンの会話、1ターンあたり約500トークン。通常なら27,500トークンのinputを支払うところ、キャッシュありなら実質約5,800トークンで済む。会話が長くなるほど、削減幅はさらに大きくなります。

## キャッシュを密かに殺す6つのアンチパターン

ここからが最重要パートです。一見無害に見えて、実は**蓄積したキャッシュを丸ごと破壊する**操作を紹介します。ほとんどの人が請求書を見るまで気づかない類のミスです。

### アンチパターン1 — 作業中にTTLを切らす

Proプラン（TTL 5分）でキャッシュが温まっている状態で、作業の手を止める。コーヒーを取りに行く、電話に出る、10分のミーティング — その間にキャッシュは消滅。次のターンで支払うのは、いま打ったメッセージ分ではなく、**開始時点からのコンテキスト全額**です。

**対策：** Proプランでは対話のリズムを一定に保ち、ターン間隔を5分以上空けない。越えそうなときや長い休憩が必要なときは、そのセッションだけMax（TTL 1時間）に切り替えるか、再計算を覚悟するか選ぶ。

### アンチパターン2 — 大きなコンテキストを早すぎるタイミングで読み込む

「念のため」とturn 1で大きなファイルを読み込んだものの、結局そのファイルについて質問しなかった。それでも先頭でcache writeが発生し、**以降の全ターン**でcache read feeも払い続けています。使わないのに、です。

**対策：** 実際に必要になってからファイルを読み込む（lazy loading）。そのセッションで絶対使うと確信できるファイルだけ読む。「全部読んでおけば安心」という習慣は今日で卒業。

### アンチパターン3 — セッション途中で `/model` でモデルを切り替える

各モデルは**別々のキャッシュ**を持ちます。Sonnetで作業していて80Kトークン分キャッシュが温まっている状況を想像してください。ふと気になって `/model opus` で別モデルの反応を試し、Sonnetに戻る。結果：同じ内容で**cache writeを2回支払った**ことになります。まさに無駄遣い。

**対策：** コンテキストを読み込む**前に**モデルを決めておく。両モデルを比較したいなら、`--fork-session` で2セッションに分岐させ、途中で切り替えるのは避ける。

### アンチパターン4 — `/rewind` を使う

`/rewind` は最後のメッセージを削除します → input文字列が変化 → **その地点から先のキャッシュが全滅**。50Kのコンテキストで30ターン進んだところでrewindした? それは30ターン分の蓄積キャッシュを、別方向を試すためだけに捨てたことになります。

**対策：** 「同じコンテキストで別方向を試したい」なら、戻りたい地点から `--fork-session` で分岐する。元セッションはそのまま、キャッシュも健在。forkで新しい枝だけが生えるイメージです。

### アンチパターン5 — 作業フローの途中でCLAUDE.mdを編集したりMCPをインストールする

階層構造を思い出してください。

* CLAUDE.md修正 → **2階層破壊**（System + Messages）
* MCPサーバーのインストール/アンインストール → **3階層すべて破壊**

40ターンのタスク中盤、キャッシュは温まっている。ルールをひとつ足したくなってCLAUDE.mdに追記 → 再起動 → **キャッシュは完全にコールド状態**。また一から積み直しです。そのルール、本当にそのコストに見合いますか? たいていの場合はNoなのに、もう支払った後。

**対策：** CLAUDE.mdやMCPの変更はまとめて、**タスク完了後**または**翌日の朝、新セッションの最初**に適用する。作業中に思いついたら「ルールXとYを追加したい」とメモだけ残して、その場で編集しない。

### アンチパターン6 — MCPサーバーをバラバラの日にインストールする

月曜にMCPサーバーAをインストール、火曜にB、水曜にC → 翌日のセッションを迎えるたびに**3回コールド状態からスタート**。同時にまとめてインストールしていれば1回で済んだはずです。

**対策：** 試したいMCP/pluginのリストを溜めておく。週に1回30分だけ時間を取って一気にインストールし、その週はそれで固定。日々バラバラに入れない。

### ボーナス：キャッシュ視点でのCLAUDE.md圧縮ルール

グローバルCLAUDE.mdの修正は**月1回が上限**と考えるべきです。修正するたびに、以降の全セッションでキャッシュがコールド状態に戻る。これは非常に大きなコストです。代わりに次の戦略を取ります。

* **グローバルCLAUDE.md：** 20行以内に収め、スタイルとcommitフォーマットだけ。変更は最小限に。
* **プロジェクト別CLAUDE.md**（`<project>/.claude/CLAUDE.md`）：ビルドルールやプロジェクト固有のworkflow。このファイルのキャッシュはグローバルと独立なので、互いに影響しません。
* **作業中のタスク固有のコンテキスト：** ひとつのタスクのためにCLAUDE.mdを編集する必要はない。セッションの最初のメッセージに直接入れれば十分。

## 警告：キャッシュが無言で壊れることがある

知っておくべきリアルなリスクがあります。2025年3月、Anthropicが**2つのキャッシュバグが重なった**ことを公式に認めました。cache writeが効かず、既存のcacheもreadできない状態。結果、すべてのリクエストが全額課金に。あるエンジニアが45Kコンテキストで50リクエストのセッションを実行したところ、$0.80で済むはずが $6.75（8.4倍）を請求されたそうです。1日3〜4セッションこのペースでやると、$200の割り当ては**47時間**で底をつきます。

一番怖いのは、システムが**何のエラーも出さない**こと。機能的にはツールは普通に動いているように見える — キャッシュが無効になっていることに気づけません。

ここから得られる教訓：

* 感覚や期待に頼るのは禁物です。実際のcache hit rateを観測できる仕組みが必要になります。
* 第1回で紹介した朝晩の `/cost` が第一の防衛線。
* 説明のつかない異常な請求額を見たら即座に止める。トークンを注ぎ込み続けない。

## まとめ

Prompt cacheはClaude Code最大のレバーです。過去に使った内容に対して**10倍安い**。Claude Codeが自動でキャッシュマーカーを配置してくれるので、やるべきことはシンプル：**それを無意識に破壊しないこと**。うまくやれば、日常的なセッションで60〜84%の請求削減が見えてきます。

**キャッシュを育てる4テクニック：**

1. 目標・制約・ファイルをすべて最初のメッセージに集約（画像は末尾に!）
2. 関連する質問を1セッションにまとめる（約3.8倍安い）
3. システム任せの自動圧縮を待たず、能動的にrolling summarization（長セッションで約65%削減）
4. 並列調査にはfork session（3方向で約2.4倍安い）

**避けるべきキャッシュ破壊の6習慣：**  
TTL切れ・ファイル先読み・途中でのモデル切替・`/rewind` の使用・作業中のCLAUDE.md編集・MCPのバラバラ導入

第1回と第2回で基盤は揃いました。次の第3回ではすべてを統合します — plan-first、モデルの階層化、subagent、token budget — これらを、この学習過程を乗り越えてきた人のための実践的なworkflowへ仕立てます。
