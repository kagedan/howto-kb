# 7/7いっぱいで、Fable 5が"使い放題の最強モデル"じゃなくなる。このプロンプートだけ打っておけ

## ⚡ 気づいたら手遅れだった、にしたくない

7/7を境に、Fable 5は「気軽に投げていいモデル」じゃなくなる。

Anthropicの公式発表はこうだ。**7/1の復旧から7/7まで、週次利用枠の最大50%まで含まれる形で使える**。**7/8からは従量課金のusage credits経由**になる（時刻は非公開だが、米国基準の日付なので日本時間だと7/8の午前中には切り替わっている可能性が高い）。「トークン単価を気にせず投げていい期間」は、今日（7/6）と明日（7/7）の2日間で終わる。

これ、静かに一番怖いパターンだと思ってる。「まだ大丈夫」と思ってるうちに枠を溶かして、気づいたら明日の夜、一番聞きたかった相談ができなくなってる。だから今日のうちに、Fable 5にしかできない仕事を全部終わらせて、日常運用は他モデル＋Skillsで再現できる形に落とし込む。そのために実際に自分が投げたプロンプトを、そのまま公開する。

🛠️ 開発用途（Web開発、GitHub運用、Cursor/Claude Code/MCP、技術ブログ、業務効率化、資格学習）に寄せて書いてるけど、変数部分（用途・目標・プロジェクト名）を書き換えれば誰でもそのまま流用できる。

## 🛠️ 1. 今のうちに"武装"しておくべきSkills

Skillsは「フォルダ＋SKILL.md」という構造。モデルが変わっても（Sonnet 5でもOpus 4.8でも）そのまま持ち運べる資産になる。Fable 5が持ってる「長期記憶・大規模文脈理解」という強み、それをSkillsという形で外部化しておくのが今回の本丸。

以下、自分の用途に直結するものを優先度順に紹介します（実際に返ってきた回答の要約です）。

### 🥇 優先度: 最高

**① WebRTCデバッグ・実装Skill**
- **作る価値**: WebRTC特有の落とし穴（ICE候補の扱い、SDPネゴシエーション、TURN/STUNの設定ミス、Safari/Chromeの実装差異など）は、汎用モデルだと毎回同じ説明をゼロからさせられがちです。これを一度体系化すれば、Fable 5でなくても再現性の高い支援が受けられます。
- **活用シーン**: 接続不安定の原因切り分け、SDP/ICEログの読み方、getUserMedia周りのブラウザ差異対応、新規実装時のアーキテクチャ相談。
- **具体的なSkill設計**:
  - `SKILL.md`にWebRTCの接続確立フロー（Signaling→ICE→DTLS→SRTP）をチェックリスト化
  - よくあるエラーパターン集（`ICE connection failed`、`m-line順序不一致`など）と切り分け手順を`reference/common-errors.md`に分離
  - 過去に自分が踏んだ罠を`gotchas.md`として追記していく運用にする（Fable 5に「今回のプロジェクトで学んだ罠を要約して」と頼んで種を作るのが効率的）

**② 技術ブログ執筆Skill**
- **作る価値**: 構成・トーン・読者層（日本のITエンジニア）を固定した執筆補助を毎回ゼロから説明せずに済む。
- **活用シーン**: WebRTC技術解説、個人開発の開発記、AI活用ノウハウ記事。
- **具体的なSkill設計**:
  - 導入・課題提示・実装・学びの4部構成テンプレ
  - コードブロックの言語表記ルール、見出し階層のスタイルガイド
  - 「結論から書く」「専門用語は初出時に一言補足」などの文体ルール

### 🥈 優先度: 高

**③ README/ドキュメント生成Skill**
- **作る価値**: 複数サービスにまたがるGitHub運用では、README/ドキュメントの陳腐化が起きやすい。生成ルールを固定化しておけば更新コストが下がる。
- **活用シーン**: 新機能追加時のREADME更新、DB/APIスキーマ変更時のドキュメント反映。
- **具体的なSkill設計**:
  - 自分のプロジェクト特有の構成（DBテーブル定義、CI/CDフロー）をテンプレの雛形として組み込み
  - Mermaid記法でのアーキテクチャ図生成ルールを含める

**④ コードレビューSkill（GitHub Actions/Supabase/Next.js特化）**
- **作る価値**: 自分のプロジェクト固有の設計原則（外部通信の分離方針、DBアクセス権限の最小化方針など）を、レビュー基準として固定できる。
- **活用シーン**: PRレビュー、Claude Code / エージェントが生成したコードの検証。
- **具体的なSkill設計**:
  - チェック項目：外部API通信を許可された経路だけで行っているか／DB接続の権限が過大でないか／型の扱いが安全か
  - 過去に事故った変更パターン（設定テーブルの想定外の複数行化など）を再発防止チェックとして明記

**⑤ 設計レビュー・技術判断Skill**
- **作る価値**: Fable 5の強みである「長期文脈での一貫した設計判断」を、判断基準の言語化によって部分的に代替。
- **活用シーン**: 新機能の技術選定、アーキテクチャの意思決定（構成A vs 構成Bのトレードオフ整理など）。
- **具体的なSkill設計**:
  - 過去の判断ログ（なぜその構成を選んだか、なぜ特定のアプローチを避けたか）を`decisions.md`として蓄積
  - 新しい判断のたびにこのファイルを参照させ、一貫性を担保

### 🥉 優先度: 中

**⑥ 基本情報技術者試験 学習支援Skill**
- **作る価値**: 出題範囲・頻出分野・自分の弱点分野を固定しておけば、モデルが変わっても同じ精度で問題演習・解説が受けられる。
- **活用シーン**: 通勤時間などでのスキマ学習、過去問の解説依頼。
- **具体的なSkill設計**:
  - 出題分野（テクノロジ系/マネジメント系/ストラテジ系）ごとの理解度メモ
  - 間違えた問題のログを`weak-points.md`に蓄積し、復習優先度を自動判定

**⑦ 業務日報・作業ログ生成Skill**
- **作る価値**: 個人開発の進捗を振り返りやすくし、技術ブログのネタ元にもなる。
- **活用シーン**: Claude Code / エージェントの実行結果をベースにした日次サマリー生成。
- **具体的なSkill設計**:
  - GitHubのコミットログ・PRから自動で「今日やったこと」を抽出するフォーマット
  - ブログネタ候補にフラグを立てる項目を追加

## ⚡ 2. Fable 5にしかできない仕事、今夜中に終わらせる

Skillsで代替しきれない、**モデル自体の地力に依存する作業**は今のうちに片付けておく。

1. **メインプロジェクトの全体アーキテクチャ分析＋改善提案**：リポジトリ全体をFable 5に読み込ませ、既存の実装方針の改修案や、次に作りたい機能の設計骨子を作ってもらう。長期文脈と自己検証能力が効くタスクです。
2. **上記①〜⑦のSkillのたたき台一括生成**：Fable 5は「これまでの会話・プロジェクトの蓄積から要点を抽出する」作業に強みがあるため、各Skillの初稿をまとめて作らせるのに向いています。
3. **中長期の運用プレイブックの精度検証**：自分が立てた目標達成までの週次マイルストーンや判断基準を、長期プランニング能力の高いFable 5に一度レビューさせておく。
4. **将来実装予定の機能の設計書作成**：「あとで実装しよう」と後回しにしている機能について、今のうちに詳細設計書を作っておけば、実装フェーズでは安価なモデルに任せられる。
5. **decisions.md・gotchas.mdの初期構築**：これまでの会話履歴を振り返らせて、「なぜその設計にしたか」を言語化・ファイル化する棚卸し作業。これは今しかできない「記憶の外部化」そのものです。

## 🌏 3. 海外・国内で今起きている議論

Web検索で拾った最新の議論、まとめるとコミュニティの結論はほぼ一致してる。

- **モデルの役割分担が定着しつつある**：Sonnet 5が実装・調査の一次対応、Opus 4.8がレビュー・検証、Fable 5は「長期プランニングや高難度の移行作業」だけに絞って使う、という3層構成の運用が主流になりつつあります。日常タスクにFable 5を使うのは「トークンの無駄遣い」という指摘が目立ちます。
- **Skillsは「小さく始めて育てる」のが鉄則**：Anthropicの公式ガイダンス、および実務者の記事の双方で、最初から完璧なSkillを作ろうとせず、数行のメモと1つの失敗パターンから始めて、使うたびに追記していくアプローチが推奨されています。
- **SKILL.mdのdescriptionは「トリガー文」として書く**：「このSkillは〇〇をする」ではなく「〇〇するときにこのSkillを使う」という書き方にすることで、自動発火の精度が上がるとされています。
- **CLAUDE.md／SKILL.mdの二層構造**：プロジェクト全体のルールはCLAUDE.mdに、特定タスクの手順はSkillsに分離する構成が、コンテキスト消費を抑えつつ精度を保つベストプラクティスとして紹介されています。
- **usage creditsの枯渇に備える**：7/7以降の高コスト化に備え、キャッシュ利用（プロンプトキャッシュで最大90%割引）やBatch API活用でコストを抑える工夫が複数の記事で共有されていました。
