Anthropicが2026年5月11日、Claude Codeに新機能「Agent View」を追加しました。  
リリースからまだ6日しか経っていない、Research Preview段階の新機能です。「複数のClaudeを並列で動かして、社員を増やしたかのように仕事ができる」という触れ込みです。  
  
期待して試しました。  
結論を先に書きます。  
  
毎日の業務には要りませんでした。   
でも月初・月末の重い作業では、武器になる機能でした。  
「いつ使うか」を整理すると、価値がはっきり見えました。

---

  

## Agent Viewとは

ターミナルから複数のClaudeセッションを同時に立ち上げて、1画面で一覧管理できる新機能です。  
  
これまで：1つのClaudeに1つの作業を頼んで、終わるまで待つ  
これから：複数のClaudeに別々の作業を頼んで、並列で動かす  
  
コマンドはシンプルでターミナルにてバックグラウンドにタスクを送る：

```
claude --bg "○○のタスクをやって"
```

一覧を見る：

```
claude agents
```

各セッションの状態（作業中・指示待ち・最終更新時刻）が1画面で確認できます。

---

## 実際に試した手順

### STEP 1：Claude Codeをアップデート

v2.1.139以降が必要です。

```
npm install -g @anthropic-ai/claude-code
```

### STEP 2：PowerShellで起動

VSCode内のClaudeでは/bgが動きません。  
「/bg isn't available in this environment.」とエラーが出ます。  
Agent ViewはCLI（コマンドライン）専用機能。PowerShellを開いて、

```
claude
```

から起動します。

### STEP 3：タスクを依頼してバックグラウンドへ

> 「今月の広告レポートのたたき台を作って」

Claudeが質問してきます。  
答えてやり取りが始まったところで/bgを打つと、バックグラウンドに送られます。セッションIDが発行されて、こう表示されます。

```
backgrounded: 0f3d238b
claude agents          → セッション一覧を見る
claude attach 0f3d238b → このセッションに戻る
claude logs 0f3d238b   → 最近の出力を見る
claude stop 0f3d238b   → このセッションを止める
```

ここまでは便利でした。

## 試した直後に感じた疑問

「これ、普通にClaudeに頼んだ方が早くない？」  
  
バックグラウンドに送る一手間、確認のために戻る一手間がある。  
 1つのタスクだけなら、直接対面でラリーした方が早い。  
ここで自分の環境と照らし合わせて整理しました。

---

## 僕の環境

3つの設定が揃っています。

1. --dangerously-skip-permissionsを許可 確認ダイアログなしで自律的に作業を進められる状態
2. voiceEnabled: trueで音声入力ON タイピングの代わりに話すだけで指示できる
3. CLAUDE.mdに業務ルールを書いている 毎回の前提共有が不要

つまり、1つのセッションで「権限のあるClaudeが自律的にどんどん仕事を進める」環境が整っています。

---

## 2つの機能は別レイヤーだった

「権限を与える」と「並列セッション」は別の話でした。

権限を与える（bypass permissions） 1つのClaudeに「自由」を与える。  
確認なしに作業させられる。1セッションでの自律性の話。  
Agent View（並列セッション） Claudeを複数並べる。  
それぞれが独立したコンテキスト・履歴を持つ。Claudeを増やす話。

---

## 毎日の業務には不要だった理由

僕の毎日の業務は、こういうものです。  
  
クライアントへの広告レポート作成 営業メールへの返信 提案書のたたき台修正 社内資料の整理  
どれも「Claudeが出した結果を見て、トーン修正・数字確認・部分書き換えのラリーをする」業務です。  
  
ラリーが発生する業務には、1セッションでの直接対話が一番速い。  
バックグラウンドに送ってから戻る手間が、ラリーのテンポを壊します。

---

## ただし、活かせる場面が3つある

ここからが本題です。  
  
「日常では不要」で終わると判断が雑すぎました。  
深掘りすると、確実に活きる場面が3つありました。

---

### 活かせる場面①：複数クライアントの独立コンテキスト管理

広告代理店として複数のクライアントを抱えています。  
  
これまで：1つのセッションで「A社の話」「B社の話」「C社の話」を混ぜていた  
理想： セッションA：A社専用（A社のCLAUDE.md・履歴  
 セッションB：B社専用（B社のCLAUDE.md・履歴）  
セッションC：C社専用  
  
これだと、Claudeが「これはどっちの会社の話だっけ？」と混乱しません。  
クライアントごとに専用Claudeを持つ感覚です。

---

### 活かせる場面②：月初・月末の重い作業

月初・月末は「リサーチ系・集計系・分析系」が集中します。  
  
これまで：1つずつ順番に投げて、終わるのを待っていた  
理想：5本まとめて並列で仕掛ける  
  
□例  
セッション1：先月の全クライアントのCPA集計  
セッション2：競合10社の新規広告コピー収集  
セッション3：自社の月次経営レポート作成  
セッション4：来月の提案書5本のたたき台  
セッション5：今週のSNS投稿7日分の量産  
  
「人間が見ていなくても完成するタスク」は、並列化が効きます。  
コーヒー飲んでる間に5本全部終わっている、という状態が作れる。

---

### 活かせる場面③：時間外活用

寝る前に5本仕掛ける → 朝起きたら全部完成している  
会議に出る前に2本仕掛ける → 会議後に結果がまとまっている

これまでは「自分が起きている時間にしかClaudeが動かない」発想でした。 Agent Viewなら「自分が寝ている時間にもClaudeが働いている」状態が作れます。  
時給ゼロのClaudeを24時間稼働させる感覚です。

---

## 整理：いつ使うか

毎日（修正ラリーが多い業務）→ 通常モード 月初・月末（重いリサーチ・集計が集中）→ Agent View クライアント案件を独立管理したい時 → Agent View 寝ている間・移動中・会議中 → Agent View

つまり「常用ツール」ではなく「月数回・特定シーンの武器」でした。

---

## どんな人に向くか

**毎日活かせる人**  
 複数案件を完全に独立管理したいフリーランス・コンサル・代理店経営者 リサーチ・分析・大量データ処理が日常業務に組み込まれている人  
  
**月数回活かせる人**  
僕のような業務スタイル（毎日は修正ラリー業務、月初月末に重い作業が集中）  
  
**ほぼ使わない人**  
1日に1つのタスクをじっくりやる人 すぐ返ってくる短いやり取りばかりの人

---

## 結論

Agent Viewを「不要だ」と切り捨てるのは間違いでした。  
「常用ツールじゃない」が正しい評価です。

毎日は要らない。  
でも月初・月末には別人のような生産性を出せる。 24時間稼働するClaudeを持てる。  
これがAgent Viewの本当の価値だと、試して気づきました。

---

## まとめ

Agent ViewはCLI専用機能。PowerShellなどから起動が必要 権限を与えることと並列セッションは別レイヤーの話 毎日の修正ラリー業務には1セッションの方が速い 月初・月末の重い作業、複数案件の独立管理、時間外活用には強い 常用ツールではなく「月数回の武器」として持っておく
