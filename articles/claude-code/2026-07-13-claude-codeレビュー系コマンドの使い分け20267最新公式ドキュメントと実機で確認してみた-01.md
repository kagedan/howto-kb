> 本記事は2026年7月時点の公式ドキュメント調査 + Claude Code v2.1.206での実機確認に基づきます。挙動やコストは今後変わる可能性があります。

## はじめに

Claude Codeにはコードレビュー系のコマンドが複数あります。`/code-review`、`/review`、`/code-review ultra`（`/ultrareview`とも）、`/security-review`、`/simplify`、さらに公式マーケットプレイスには同名の`code-review`プラグインまで存在します。

名前が似ているうえに役割が重なって見えるので、「自分のコードを見てほしいときはどれ？」「同僚のPRを見るときは？」と迷いがちです。公式ドキュメントとCHANGELOGを読み込み、手元のv2.1.206で動かして確認したので、使い分けを整理して共有します。

対象読者:

* Claude Codeを日常的に使っていて、レビュー系コマンドの違いを整理したい人
* PR前のセルフレビューとチームのPRレビューでコマンドを使い分けたい人
* ultraの課金が気になって手を出せていない人

## TL;DR

後半の実測（欠陥9件を仕込んだ小さなアプリでの1回ずつの計測）から言える、筆者の本音のまとめです。

* **1つだけ選ぶなら`/code-review`**。自分の作業中の差分も、PR番号を渡せば他人のPRも見られて、effortで深さも調整できます。あらゆる需要をだいたいカバーするので、まず押さえるべき1本です
* **日常は追加クレジット不要（プランの通常使用量の範囲）で動く`/code-review`で足りる**。effortは今回のラボでは`high`で検出が飽和し、それ以上は時間が延びるだけでした
* **`/security-review`が拾う脆弱性は`/code-review high`でもほぼ拾えました**。セキュリティ観点「だけ」に絞りたいときの専用コマンドという位置づけです
* **公式マーケットプレイスの`code-review`プラグインは、あえて入れる必要は薄い**です。同梱の`/code-review`とは別物のPR専用コマンドで、同梱コマンドで足りる場面が多いです
* **追加課金の`/code-review ultra`は「追加コスト＝高品質」とは限りません**。今回の小規模な変更では、混入していない実在バグの発掘はむしろ追加クレジット不要の`/code-review max`の方が多いくらいでした。大きな変更のマージ前など、使いどころを絞るのがよさそうです
* **レビューを任せるモデルはSonnet以上が無難**です。最速のHaikuも試しましたが、そもそもレビューを最後まで完遂できない実行が多く（結果を取得できずに終わる、最終レポートが確定しないなど）、司令塔役として不安定でした

以下は各コマンドの使い分けと、その根拠になった実測の詳細です。

## シチュエーション別の早見表

| シチュエーション | 使うコマンド |
| --- | --- |
| 作業中、書いたそばから素早くチェックしたい | `/code-review`（引数なし） |
| コミット・PR前にコードを整えたい | `/simplify` |
| セキュリティ面だけ確認したい | `/security-review` |
| 大きめの変更をマージする前に深く確認したい | `/code-review ultra`（課金注意） |
| 同僚のPRを承認前に確認したい | `/review <PR番号>` |
| 同僚のPRをじっくり確認して指摘をPRに残したい | `/code-review <level> <PR番号> --comment` |

大きな分かれ目は**自分の作業中の差分を見るか、GitHub上のPRを見るか**です。公式ドキュメントにも「作業中のdiffには`/code-review`、PRには`/review`」という住み分けが明記されています。

## 多角比較表

主要5コマンド + プラグイン版を、対象・観点・深さ・速度・修正適用の有無で比較します

| コマンド | 対象 | レビュー観点 | 深さ | 速度 | 修正適用 |
| --- | --- | --- | --- | --- | --- |
| `/code-review` | 現ブランチの変更（未コミット含む / PR番号指定も可） | 正確性バグ + クリーンアップ | effort引数でスケール | 数秒〜数分 | `--fix`指定時のみ |
| `/simplify` | 現ブランチの変更 | クリーンアップ専用（バグは探さない） | — | 数分程度 | **デフォルトで適用** |
| `/security-review` | 現ブランチの変更 | 脆弱性（インジェクション / 認証 / データ露出など） | — | 数分程度 | なし（報告のみ） |
| `/review` | GitHub PR | 総合的なPRレビュー | single-pass | 数秒〜数分 | なし（読み取り専用） |
| `/code-review ultra` | 自分のブランチ or PR | 検証済みの実バグに集中 | クラウドで多数エージェント + 独立検証 | 5〜10分 | `--fix`で完了後に適用 |
| `code-review`プラグイン | GitHub PR | バグ + CLAUDE.md準拠 | 並列エージェント + confidence（確信度）フィルタ | 10分前後 | なし（PRコメント投稿） |

コストは`/code-review ultra`だけが特殊で（3回無料 → 以降$5〜$20/回）、他はすべて通常使用量の範囲で動きます

`/code-review`・`/review`・ultraの3行の速度と深さは公式ドキュメントの比較表、プラグイン行の深さはGitHub上の公開実装に基づきます。深さ列の「—」は公式情報がない項目、`/simplify`・`/security-review`・プラグイン行の速度は著者実測による目安です（いずれも2026年7月時点。実測の詳細は後述の実測セクションを参照）。

## 自分のコードをレビューする: 3段構え

自分の書いたコードに対しては、作業のフェーズごとにコマンドを変えるのが合っていました

### 作業中: `/code-review` で素早く回す

引数なしで実行すると、現在のブランチで加えた変更をレビューします。公式ドキュメントによると、既定の対象はブランチのupstreamより先のコミットと、ワーキングツリーの未コミット変更です。

effort（推論努力）のレベルを第1引数で渡すと深さを調整できます

```
/code-review low          # 高信頼の指摘だけ素早く
/code-review high         # 広めに探す（不確実な指摘も含む）
/code-review max src/api/ # 対象をパスで絞って最大効率で
```

低いレベルは少数で信頼度の高い指摘を、`high`〜`max`は不確実なものも含む広いカバレッジを返します。指定しない場合はセッションのeffort設定に従います（現在値は`/effort`で確認できます）。なお公式のeffort既定値はSonnet 5・Opus 4.8で`high`、Opus 4.7で`xhigh`です。つまり明示的に下げない限り、`/code-review`も`high`相当で動きます。

指摘に納得したら`--fix`付きで再実行するか、最初から付けておくと修正まで適用されます

### コミット・PR前: `/simplify` と `/security-review` で仕上げる

`/simplify`はバグ探索をしない代わりに、再利用・簡素化・効率化といったクリーンアップに特化したコマンドです。`/code-review`と違って**修正の適用までがデフォルト**なので、「PR前にコードを整える」用途ではこちらが手数少なめです。

認証まわりや外部入力を扱う変更なら`/security-review`も追加します。現ブランチの変更をインジェクション・認証・データ露出などの脆弱性観点で分析するコマンドです。

### マージ前の大きめ変更: `/code-review ultra`

マージしてしまうと影響が大きい変更には、クラウドで動く深層レビューがあります

ローカルの`/code-review`との違いは公式ドキュメントが明確で、多数のレビューエージェントが並列で変更を探索し、**報告前にすべての指摘を独立に再現・検証**します。スタイル指摘ではなく実バグに絞られるのが特徴です。実行はリモートsandboxなので、走らせたままターミナルを閉じても問題ありません（`/tasks`で確認・停止できます）。

課金には注意が必要です。2026年7月時点の公式ドキュメントでは以下のとおりです。

* Pro / Maxはアカウントごと**3回まで無料**（1回限りの付与でリフレッシュなし）
* Team / Enterpriseは無料枠なし
* 以降は追加使用量（`usage credits`）で**1回あたり$5〜$20程度**（変更サイズに依存）
* 支払いには追加使用量の設定を事前に有効化しておく必要があり、無効のままだと起動がブロックされます（`/usage-credits`で確認・変更）
* クラウドセッションが開始した時点で1回とカウントされるため、**途中で止めても失敗しても無料枠を消費**します
* 実行前に確認ダイアログでスコープ・残り無料回数・推定コストが表示されます

なおultraはユーザーがコマンドを打ったときだけ起動します。公式ドキュメントにも「Claude は ultrareview を自動的に開始しません」と明記されており、Claudeへの依頼で勝手にクラウド実行と課金が始まることはありません。

クラウドで動いたかどうかは**実行前の確認ダイアログが出たかで判別できます**。手元のDesktopアプリでは、`/code-review ultra`と打ってもダイアログが出ずにそのままローカルのレビューが走る現象を確認しました（2026年7月時点の実機確認。このときクラウドでは実行されておらず、課金も無料枠の消費もありません。`/ultrareview`と直接打てばDesktopでもダイアログが出ます）。クラウドで確実に回したいときは、ダイアログが表示されたことを確認するか、`/ultrareview`またはターミナルからの`claude ultrareview`を使ってください。

## 他の人のPRをレビューする

### まず `/review` でざっと確認

PR番号を渡すと、そのPRのdiffを取得してレビューを返します。読み取り専用で、承認前に同僚のPRへ目を通す用途が公式の想定です。番号を省略するとオープンなPR一覧から選べます。

single-passの軽いレビューなので、速度感は`/code-review`と同じ数秒〜数分です

### じっくり見たいPRは `/code-review` にPR番号を渡す

`/code-review`の`target`引数はPR番号も受け付けます。effortを選べるマルチエージェントレビューをPRに向けたいときはこちらです。

```
/code-review high 123           # PR #123 を high effort でレビュー
/code-review high 123 --comment # 指摘をPRのinlineコメントとして投稿
```

`--comment`を付けるとGitHub PRのinlineコメントとして指摘が投稿されるので、レビュー結果をそのままチームに共有できます

さらに深く見たい場合は`/code-review ultra 123`でPRモードのultrareviewも使えます（課金は自分のブランチのときと同様）

## 実測: 欠陥を9件混入して検出差を見てみた

使い分けの実感を得るため、Express製の簡易メモアプリに欠陥9件（正確性バグ3 / クリーンアップ余地3 / 脆弱性3。うち2件は未コミットのままworking treeに残す）を意図的に混入し、**1コマンド = 1新規セッション**で実行して検出結果を比べました。

混入を指示したセッションでレビューを走らせると「答えを知っている状態のレビュー」になり、同じセッションでレビューを連続実行すると先行レビューの指摘が後続のヒントになるため、混入と計測、計測同士のセッションをすべて分離しています。本編の計測はモデルをSonnet 5に固定しました（モデルを変えた計測は後述）。

2つ注意点があります。まず2026年7月時点・v2.1.206での著者実測で、結果はモデル・effort・対象コードに依存するので傾向の参考としてください。もう1つ、欠陥はClaude（Opus 4.8のセッション）に定番パターンとして混入させたものです。実務で人間が作り込む複雑なバグより検出しやすい題材のため、検出率の絶対値は割り引いて、コマンド間の相対比較として読むのが安全です。

検出・対応した数を、混入した3カテゴリ（各3件）ごとに分けるとこうなりました。

| コマンド | バグ | 脆弱性 | クリーンアップ | 所要時間（実測） | 備考 |
| --- | --- | --- | --- | --- | --- |
| `/code-review high` | 3/3 | 3/3 | 3/3 | 約4分 | この表は検出数。mediumはクリーンアップ2/3（デッドコード1件が非検出）。high以上は3/3検出するが最終レポートは上限で減ることあり（effort別は下表） |
| `/security-review` | 0/3 | 3/3 | 0/3 | 約3分 | バグ・クリーンアップは認識の上でスコープ外と明言 |
| `/review`（PR） | 2/3 | 2/3 | 3/3 | **約19秒** | 未コミット2件（バグ1・脆弱性1）はPR外で対象外 |
| `/simplify` | 1/3 ⚠️ | 1/3 ⚠️ | 3/3 | 約4分 | ⚠️は副次的な修正（バグ・脆弱性は本来スコープ外） |
| プラグイン版（PR） | 2/3 | 2/3 | 0/3 | 約11分 | 未コミット2件はPR外、クリーンアップは軽微として除外 |

各コマンドを実力が出るeffort（`/code-review`はhigh以上、他はeffort設定なし）で並べています。effortを変えたときの`/code-review`の伸びは下の階段表を参照してください。`/review`とプラグイン版はコミット済みの7件（バグ2・脆弱性2・クリーンアップ3）が母数で、未コミットの2件はそもそも見えません。`/simplify`はクリーンアップ専用ですが、再利用や効率化の副産物としてバグと脆弱性を1件ずつ結果的に直しています（⚠️）。

`/code-review`のeffortを変えた階段はこうなりました

| effort | 検出（9件中） | うち最終レポートに載った数 | 所要時間（実測） |
| --- | --- | --- | --- |
| low | 4 | 4 | 約14秒 |
| medium | 8 | 8 | 約2.5分 |
| high | 9 | 8 | 約4分 |
| xhigh | 9 | 6 | 約13分 |
| max | 9 | 9 | 約23分 |

「検出」と「最終レポートに載った数」を分けているのは、レビューの過程では見つけているのに、報告件数の上限で最終レポートから漏れる指摘があるためです（詳細は後述の注目ポイント3つ目）。

クラウドの`/code-review ultra`（`/ultrareview`）も同じラボで実行しました。所要は約15分（公式の目安は5〜10分）で、指摘は9件、内訳は混入した9欠陥のうち8件 + 混入していない実在バグ1件（既存データベースへのマイグレーション欠如）です。未コミットの2件も検出しました。唯一挙がらなかったのは複雑ループの整理（クリーンアップ）で、残り2つのクリーンアップも軽微（nit）区分での報告だったので、「スタイル指摘ではなく実バグに焦点を当てる」という公式ドキュメントの説明に近い出力です。各指摘には深刻度が付き、セキュリティ3件をマージブロッカー相当として修正着手まで促す形でした。

ただ、ultraの費用対効果には正直な補足があります。今回の小規模なdiffでは、ultraの検出数はローカルの`/code-review high`以上と同水準で、混入していない実在バグの発掘はむしろローカルのmax（通常使用量の範囲で動く）の方が多い結果でした。ultraは追加使用量を消費するので、「ローカルのmaxでも十分に深いレビューが返ることは多い」前提に立って、変更の規模や影響度、使用量の残りと照らして使うかを検討するのがよさそうです（大きな変更での差は本実験では未検証です）。

検出マトリクスの詳細（9欠陥 × 6コマンド。興味があれば展開してください）

✅=検出・報告 / ⚠️=副次的に修正 / ❌=非検出または報告見送り / ➖=スコープ外（PRに含まれない未コミット分）

| 混入した欠陥 | `/code-review` medium | `/security-review` | `/review` | `/simplify` | プラグイン版 | ultra |
| --- | --- | --- | --- | --- | --- | --- |
| SQLインジェクション | ✅ | ✅ | ✅ | ⚠️ 副次修正 | ✅ | ✅ |
| XSS | ✅ | ✅ | ✅ | ❌ 報告のみ | ✅ | ✅ |
| パストラバーサル（未コミット） | ✅ | ✅ | ➖ | ❌ 報告のみ | ➖ | ✅ |
| off-by-one | ✅ | ❌ 対象外 | ✅ | ⚠️ 副次解消 | ✅ | ✅ |
| null未処理クラッシュ（未コミット） | ✅ | ❌ 対象外 | ➖ | ❌ | ➖ | ✅ |
| 条件式の論理ミス | ✅ | ❌ 対象外 | ✅ | ❌ 見送り | ✅ | ✅ |
| 関数の重複 | ✅ | ❌ 対象外 | ✅ | ✅ 適用 | ❌ 軽微として除外 | ✅ nit区分 |
| デッドコード | ❌ 非検出（mediumのみ / high以上は検出） | ❌ 対象外 | ✅ | ✅ 適用 | ❌ 軽微として除外 | ✅ nit区分 |
| 不必要に複雑なループ | ✅ | ❌ 対象外 | ✅ | ✅ 適用 | ❌ 軽微として除外 | ❌ |

`/security-review`の「対象外」は見逃しではなく、バグとして認識した上で「セキュリティ脆弱性ではない」と明言してのスコープ外判定でした

実測から見えた注目ポイントを4つ挙げます

* **未コミット変更の扱いで二分される**: `/code-review`（ultra含む）・`/security-review`・`/simplify`は未コミットの2件も検出しましたが、`/review`とプラグイン版はPRの確定diffだけを見るため未コミット分は原理的に見えません。「PRに出す前の最後の変更」を見てほしい場面でPR系コマンドを選ばない、という使い分けの根拠になります。
* **検出はhighで頭打ち、時間は増え続ける**: 今回の9件はhighで全件検出に到達し、xhigh・maxへ上げても検出数は変わらず時間だけ3〜6倍になりました。ただし高effortほど混入していない実在の問題を追加で掘り出します（明確な誤検出はほぼゼロでした）。maxでは一部の指摘について、実際にアプリを起動して攻撃リクエストを送り、成立を確かめた上で報告するところまで踏み込んでいました。無駄になるというより「深掘りの範囲と検証の深さが広がる」動きです。
* **effortを上げると、検出済みの軽微な指摘が最終レポートから漏れることがある**: 報告件数には上限があり正確性バグが優先されるため、highではデッドコードが、xhighではクリーンアップ3件すべてが「検出はされたのに最終レポート圏外」になりました（maxは全件列挙されました）。effort階段表の「検出」と「最終レポートに載った数」の差がこの現象です。クリーンアップの指摘が目的なら`/simplify`を別に回す方が確実です。
* **`/simplify`は「バグを探さない」が「直してしまう」ことはある**: 重複排除や効率化の副産物として、SQL文字列連結（脆弱性）とoff-by-one（バグ）が結果的に修正され、挙動が変わりました。適用後のdiff確認はセットにするのが安全です。

細かい落とし穴も1つ見つかりました。ブランチをプッシュして追跡ブランチとHEADが完全に一致した状態だと、`/code-review`はコミット済みの変更をレビュー範囲から外し、未コミット分しか見なくなります。プッシュ後にセルフレビューしたいときは`/code-review main...feature-branch`のように範囲を明示すると確実です。

### モデルを変えるとどうなるか

同じラボで、モデルをHaiku 4.5・Opus 4.8・Fable 5に変えたeffort階段も取りました。注意点が1つあり、Fable 5指定の4実行はいずれも、セキュリティ系コードが安全性分類器に検知されてOpus 4.8へ自動切替される通知が出ています。これは公式ドキュメントに記載のある挙動（サイバーセキュリティ・生物学の領域で発生しやすい）なので、下表の「Fable 5指定」列は実質「前半Fable 5 + 後半Opus 4.8」の実測値として読んでください（Opus 4.8は切替先そのものなので、Opus列はクリーンな計測です）。数値は検出数（レビューが見つけたseeded数）/ 所要時間です。最終レポートに何件載るかは報告件数の上限で変わるので、その内訳は前掲のeffort階段表を参照してください（Haiku 4.5は司令塔役が不安定で、この検出数自体が実行ごとにぶれます）。

| effort | Haiku 4.5 | Sonnet 5 | Opus 4.8 | Fable 5指定（途中からOpus 4.8） |
| --- | --- | --- | --- | --- |
| medium | 6件 / 約2分 | 8件 / 約2.5分 | 8件 / 約50秒 | 9件 / 約2分 |
| high | 5件 / 約2分 | 9件 / 約4分 | 9件 / 約1分 | 9件 / 約4分 |
| xhigh | 取得失敗 | 9件 / 約13分 | 9件 / 約2分 | 9件 / 約8分 |
| max | 約6件 / 約2分 | 9件 / 約23分 | 9件 / 約17分 | 9件 / 約13分 |

読み取れることは4つあります。ただし各セルは1回ずつの実測で、レビュー結果は実行ごとに確率的に揺れます。SonnetとOpusは計測した時間帯も離れており、所要時間はサーバー側の混雑の影響を受けている可能性があります。さらに欠陥を混入させたセッション自体がOpus 4.8で動いていたため、特にOpus列は「自分が書きそうな定番欠陥を自分でレビューする」構図になっており、モデル間の優劣を判定できるデータではありません。数字は「今回の実行ではこうだった」という参考値として見てください。

1つ目は、**Sonnet・Opus・Fable指定では検出数がモデルでほぼ動かなかった**ことです。この3つはmediumで8〜9件、high以上で全件に到達しました。対象範囲・出力形式・修正適用の有無といったコマンドの性格もモデルでは変わりません（Haikuだけ様子が違い、4つ目で扱います）。

2つ目は、**内部の走り方が実行によって大きく違った**ことです。今回の計測では、Sonnetのhigh以上とOpusのmaxの実行は多数のエージェントを並列に立てて検証し、実アプリを起動した攻撃実証や、混入していない実在バグの発掘まで踏み込みました。一方、Opusのmedium〜xhighの実行は補助のエージェントを立てずに本体が直列で確認する構成で、報告も混入した欠陥に絞られています。ただし各1回の実測なので、この切り替わりが「このモデルのこのeffort以上は並列」という固定の規則なのかまでは断定できません。所要時間の差も混雑や実行ごとのばらつきと切り分けられていないため、確かに言えるのは「同じeffortラベルでも走り方が大きく違う実行があった」ことまでです。

3つ目は、**脆弱性を扱うレビューをFable 5で回す運用は実質成立しない**ことです。切替はターンの途中でも起き、モデル自身は切り替わった事実を申告しないため、UI通知を見ていない限り「Fable 5でレビューしたつもりが実測はOpus 4.8」になります。セキュリティが絡むコードのレビューは、どうせOpusに切り替わるので、最初からSonnetかOpusを選んでおけば済みます。

4つ目は、**最速のHaiku 4.5は、そもそもレビューを最後まで完遂できないことが多かった**ことです。`/code-review`のmedium以上は複数のエージェントを束ねる多段パイプラインですが、Haikuではこの司令塔役がたびたび崩れました。今回medium・high・xhigh・maxを狙った計測でも、mediumは初回がレビュー結果を返せずに取り直し、xhighは取り直しても結果を取得できず（スキルの完了を待たずに終了時刻を記録してしまう、など）、maxは完遂したものの最終レポートが実行の途中で3回作り直されて内容が確定しませんでした。一発できれいに完遂したのはhighだけです。個別のバグを見つける力自体は高く、単体ではXSSもSQLインジェクションも検出できていたので、弱いのは検出力ではなく複数エージェントの取りまとめの方です。そもそもHaikuは公式にeffort非対応で、レビューの深さを引き上げる手段がありません。検出も時間（約2分のまま）も設定で動かせず、何を取りこぼすかは実行ごとにバラバラでした。さらにHaikuは許可を自動承認する自動モードの対象外なので、レビューを自動で流す用途にも向きません。レビューを任せるならSonnet以上が無難です。

まとめると、同じ`/code-review`でも結果はモデルとeffortで変わりますが、変わり方には強弱がありました。effortは検出数をhighまで押し上げ（その先は時間が延びるだけ）、モデルは能力の下限で結果を大きく左右する一方（Haikuは完遂すら不安定）、Sonnet・Opus・Fableのような十分な能力のモデル同士では差は小さいものでした。

## 紛らわしいポイント3つ

### 1. 公式プラグインの code-review は別物

公式マーケットプレイスに同名の`code-review`プラグインがあり、これは同梱コマンドの`/code-review`とは**別実装**です。両方有効な環境ではプラグイン側が`/code-review:code-review`という名前になります。この紛らわしさは公式リポジトリのIssueでも指摘されています。

プラグイン版はPR専用で、並列エージェントの指摘をconfidenceスコアでフィルタしてPRに単一コメントを投稿する、というチームのPRレビュー向けの作りです。ローカルの作業中diffには同梱の`/code-review`、と覚えておけば混乱しません。

### 2. `/ultrareview` と `/code-review ultra` の関係

公式ドキュメント上は`/code-review ultra`が正式な呼び出しで、`/ultrareview`がエイリアスという扱いです。ただし実際に`/code-review ultra`を実行すると、画面上は`ultrareview`コマンドに解決されて動くので、呼び名としてはむしろ`ultrareview`が実体に見えます。どちらで打っても同じクラウドレビューが走る、と捉えておけば混乱しません。CIやスクリプトから非対話で使う場合は`claude ultrareview`サブコマンドがあり、`--json`や`--timeout <minutes>`（既定30分）を受け付けます。

### 3. `/review` の挙動は一度変わって戻った

`/review`はv2.1.186で一度`/code-review medium`と同じマルチエージェントエンジンに変更され、v2.1.202で高速なsingle-passに戻りました。「PRを深く見たいなら`/code-review <level> <PR番号>`」という現在の住み分けは、この変更の結果です。過去の記事や解説を読むときはバージョン時点に注意してください。

## チーム全体の自動化は別系統

ここまでは手元のセッションで打つコマンドの話ですが、Team / Enterpriseプラン向けには**Code Review**というマネージドサービスが別にあります。GitHubリポジトリに接続すると、PRに対して自動でマルチエージェントレビューが走り、inlineコメントとcheck runを投稿します。トリガーは`@claude review`などのコメントでも可能で、コストはレビュー1件あたり$15〜$25程度（PRサイズなどに応じて増減 / 2026年7月時点）です。

個人の`/code-review ultra`とは別機能なので、「個人はultrareview、チームの継続運用はCode Review」という整理になります。Code Reviewのコスト感やコストを抑える使い方は、実際に使って$100超かかったときの記録を別記事にまとめてあります。

## まとめ: 自分はこう使い分ける

* 作業中は`/code-review`を引数なしで回し、指摘が妥当なら`--fix`で適用
* effortはSonnet 5・Opus 4.8なら既定の`high`が基準（今回の小さなラボでは検出が`high`で飽和しました）。素早く済ませたい軽い確認のときは`low`〜`medium`に下げ、逆に大きな変更や込み入ったバグが疑わしいときは`xhigh`〜`max`に上げる、という調整の余地は残しておくのがよさそうです（今回のラボは小規模なので上位で頭打ちになりましたが、実務の複雑なコードでは上位が生きる場面もあるはずです）
* PR前に`/simplify`で整えて、セキュリティが絡む変更だけ`/security-review`を追加
* 影響の大きい変更のマージ前だけ`/code-review ultra`。ただしローカルの`/code-review max`でも十分に深いレビューが返ることは多いので、追加費用（$5〜$20/回）を払う価値は変更の規模と影響度で判断（まず無料枠3回で手応えを見るのがオススメ）。
* 同僚のPRは`/review <番号>`でざっと見て、深掘りしたいときだけ`/code-review high <番号> --comment`

コマンドが多く見えて、実際は「自分のdiffかPRか」「速さか深さか」の2軸で選ぶだけでした。まずは通常使用量の範囲で動く`/code-review`と`/review`から試してみてください。

## 運用Tips: `/goal`との組み合わせ

ここからは実験ではなく筆者の運用です。条件を満たすまでClaudeにターンをまたいで作業を続けさせる`/goal`コマンドと組み合わせると、レビューから修正までを自動で回せます。うまく回すコツは、達成を客観的に判定できる条件をゴールに置くことです。

たとえば「テストが通るまで直す」や、実際にアプリを動かして確認する`/verify`が通るまで、といったゴールは収束します。レビュー系で言えば、PRを出した後にCIのレビューボットとやり取りしながら指摘を解消していく往復を`/goal`で回すのが、外部の判定を条件にできる分うまくいきます。

`--fix`自体は便利ですが、無人で回すよりも`/code-review high`で一度指摘を眺めてから`--fix`を当てる半自動の方が、意図と違う修正が黙って入る事故を避けられます。

## 運用Tips: hookでPR前レビューを組み込む

もう1歩進めると、Claude CodeのhookでレビューをPR作成のワークフローに組み込めます（こちらも筆者の運用構想です）。PreToolUse hookで`gh pr create`の実行を検知し、`exit 2`でブロックすると、hookが出力したメッセージがClaudeに渡ります。そこで「`/simplify`→テスト→`/code-review high --fix`→テスト→`/security-review`を済ませてから再実行」という指示を返す構成にすると、PR作成前のレビューを仕組みとして強制できます。hook自身がレビューを実行するのではなくClaudeに実行を促す作りにするのがポイントで、何をなぜ直したかがセッションとコミットに残ります。

ここで**抜け道（bypass）を必ず用意する**のが重要です。何もしないとレビュー後の再実行も同じhookに弾かれて、`gh pr create`が永久に通らなくなります。「レビュー済みなら通す」条件をhook側に入れるか、`PRE_PR_GATE_BYPASS=1 gh pr create ...`のように環境変数フラグを付けたときはブロックをスキップする作りにしておきます。

自動モードと組み合わせると、ブロック→レビュー実行→修正適用→再実行までが実質無人で回ります。注意点は2つで、`/simplify`と`--fix`は挙動を変える修正を入れることがあるため間にテストを挟むこと、収束条件は「指摘ゼロ」ではなくテストの成否のような客観的な判定に置くことです（前節の`/goal`と同じ理屈です）。
