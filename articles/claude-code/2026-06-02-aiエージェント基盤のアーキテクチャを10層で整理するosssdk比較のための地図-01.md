## はじめに

「このSDK、かなり良さそうですね」

チームの検証共有で、あるRuntimeの話が出ました。使い勝手はよく、すぐに社内展開できそうに見えました。

ただ、話を聞いているうちに違和感が出てきました。これはRuntimeの話なのか。モデル選定の話なのか。Sandboxの話なのか。あるいは、長時間タスクをどう再実行するかという話なのか。

同じAIエージェント基盤の話をしているはずなのに、見ているレイヤーが違っていました。

チームでは、メンバーそれぞれが異なるOSS/SDKを分担して検証していました。自分はあえて、実行エンジン以外 — コードの隔離実行、エージェントに仕事を渡す仕組み、失敗時に再開する仕組み — を見ていました。

学びを持ち寄って設計方針を議論する段階になったとき、壁にぶつかりました。

あるメンバーはRuntimeの使いやすさを話していました。別のメンバーはSandboxの安全性を気にしていました。自分は、そもそもタスクの再実行やObservabilityが必要ではないかと考えていました。

どれも正しい。ただ、同じ表の上で話していませんでした。

そこで、自分が検証を通じて整理していた全体像をチームに共有しました。すると、「ここは見えてきている」「ここはまだ誰も手を付けていない」が可視化できました。

便利なRuntimeを選ぶことと、基盤を設計することは同じではない。

この記事では、AIエージェント基盤のアーキテクチャを10層+2横断で整理し、OSS/SDKを比較するための地図を紹介します。後半ではDocker Agentを実際にマッピングします。

先に全体像だけを並べると、10層+2横断は次のような構成です。

| 層 | 名前 | ざっくり言うと |
| --- | --- | --- |
| L10 | Task Source | エージェントを動かす仕事の起点 |
| L9 | Durable Execution / Orchestration | 長時間タスクを落とさず進める仕組み |
| L8 | Multi-Agent Coordination | 複数エージェントの役割分担・並列実行 |
| L7 | Agent Runtime / Loop SDK | 単体エージェントの実行ループ |
| L6 | Capability Layer | ツール・スキルなど外部能力との接続 |
| L5 | Memory & State | 知識やタスク状態の保持 |
| L4 | Execution Sandbox | コード実行環境の隔離 |
| L3 | Governance / Policy | 実行前のポリシー判定 |
| L2 | LLM Gateway / Router | モデル接続・ルーティング・制御 |
| L1 | Model | 推論・計画・対話を担うLLM |
| X | Observability | 全層にまたがる観測 |
| Y | Supply Chain & Repo Contract | 依存管理とリポジトリ側の作業契約 |

この地図があると、「このSDKはRuntimeには強いが、長時間実行や観測は別に考える必要がありそうだ」といった形で、議論の場所を揃えやすくなります。

この記事は、AIエージェントを「LLMにツールを渡してタスクを進める仕組み」として触ったことがある人を主な読者にしています。

一方で、Runtime、Sandbox、MCP、Durable Executionといった用語をすべて知っている前提にはしません。必要な概念は、レイヤーの説明の中で簡単に補足します。

## AIエージェント基盤としてのAgent Harness

ここでいうAgent Harnessは、LLMそのものではなく、LLMをエージェントとして動かすための周辺の仕組み全体を指します。

たとえば、モデルを呼び出す、ツールを実行する、コード実行環境を隔離する、途中で失敗したタスクを再開する、実行ログを観測する、といった部分です。Chat UIやLLM API単体ではなく、「エージェントを実際に動かし続けるための土台」と考えると近いです。

ただし、Agent Harnessという言葉には厳密な標準定義がなく、文脈によってRuntime、Tool Loop、Orchestration、Sandboxなど、指す範囲が異なります。だからこそ、チームで議論するためにレイヤーに分けて定義することに価値があります。

### Runtime中心では足りなかった

最初は、自分たちもRuntimeの比較をしているつもりでした。

しかし検証結果を並べると、Runtimeの比較だけでは説明できない差分が出てきました。あるOSSはSandboxまで含んでいる。別のSDKはTool Loopには強いが、長時間タスクの再実行は別問題になる。Observabilityは、そもそも誰も見ていない。

そこで、Runtimeを中心に見るのをやめて、責務ごとにレイヤーを分けることにしました。

この分類は、AIエージェント関連のOSS・SDK・標準プロトコルを横断して見たうえで、チームでOSS/SDKを比較判断できる粒度に再整理したものです。Anthropicが提唱したハーネス設計原則も参考にしています。

「10層が唯一の正解」ではありませんが、この切り口で見ると比較や議論がしやすくなります。

## AIエージェント基盤のアーキテクチャを10層+2横断で見る

新しいOSS/SDKを見たとき、最初に聞くべき問いは「これは良いSDKか」ではありません。

* このSDKは、どの層を引き受けてくれるのか
* 逆に、どの層は自分たちで持たなければならないのか

この2つです。

以下の10層+2横断が、その問いに答えるための地図になります。役割が近い層を5つのグループと2つの横断層にまとめました。

各層では、イメージしやすいように代表的なプロダクト名も挙げます。ただし、特定プロダクトを推すためではありません。あくまで地図の読み方を掴むための例として扱います。

たとえば「GitHub Issueを起点に、エージェントが調査し、コードを修正し、Pull Requestを作る」タスクを考えます。

IssueがL10 Task Source、エージェントの実行ループがL7 Runtime、コード実行の隔離がL4 Sandbox、GitHubやMCPサーバーとの接続がL6 Capabilityです。タスクが途中で止まったときに再開する仕組みはL9 Durable Execution、何が起きたかを追うログやトレースはX Observabilityにあたります。

たとえば「Claude Agent SDKを採用するか」という問いも、この地図ではL7のRuntime選択とL1のモデルロックインに分けて議論できます。

まずはざっと全体感を掴んでもらえれば十分です。各層が具体的にどう効いてくるかは、後のDocker Agentのマッピング例で見ていきます。

### モデルと接続（L1・L2）

エージェントが「考える」ために必要な、LLMへのアクセスに関する層です。

**L1 Model**は、推論・計画・対話を担うLLMそのものです。

Claude、GPT、Qwenのようなモデルがここに入ります。

**L2 LLM Gateway / Router**は、複数モデルを同じインターフェースで扱うための層です。

たとえばLiteLLM、OpenRouter、PortkeyのようなサービスやOSSがこの役割を担います。モデルの切り替え、フォールバック、キャッシュ、予算管理をここで扱います。

最初は特定のモデルを直接呼ぶだけでも動きます。ただ、複数モデルを使い分けたい、障害時に別モデルへ逃がしたい、利用料金を制御したい、といった要求が出るとL2が必要になります。

L2を持たないと、特定モデルへの依存がそのまま基盤全体のロックインになります。

### 安全と統制（L3・L4）

エージェントに「何を許可し、何を禁止するか」を制御する層です。

**L3 Governance / Policy**は、ツール呼び出しやリソースアクセスの前に「実行してよいか」を判定する層です。

たとえばMicrosoft Agent Governance Toolkitのように、ポリシーに基づいてコマンド実行や外部アクセスを制御する仕組みがここに入ります。

**L4 Execution Sandbox**は、仮にコードが実行されても影響範囲を閉じ込める層です。

Docker Sandbox、E2B、Daytona、Modal、Rivet agentOSのようなコンテナ、microVM、Wasm/V8系の隔離環境が候補になります。

L3は「実行してよいか」を事前に止める層です。L4は「実行されても壊れない場所に閉じ込める」層です。

Runtimeが便利になるほど、この層が曖昧だと実行リスクもそのまま広がります。

### 能力と記憶（L5・L6）

エージェントが「何を知っていて、何ができるか」を決める層です。

**L5 Memory & State**は、長時間タスクやセッションを跨いだ知識・状態を保持する層です。

ローカルファイル、S3のようなファイルストレージ、Cloudflare Agent Memoryのような構造化メモリがここに入ります。過去の調査結果、ユーザー設定、途中まで進んだタスク状態などを保持します。

**L6 Capability Layer**は、エージェントに外部世界への手足を与える層です。

GitHubを読む、Slackに投稿する、社内DBを検索する、といった操作はここに入ります。標準化の観点ではMCPサーバーやAgent Skillsが代表例です。

ここが弱いと、エージェントは毎回ゼロから考え、同じ失敗を繰り返します。

逆にL6だけが強くても、L5がなければ前回の文脈を引き継げません。

### 実行ループ（L7）

エージェントの「動き方」を定義する中核の層です。

**L7 Agent Runtime / Loop SDK**は、多くの人が「Agent SDK」と聞いて最初に想像する部分です。

LLMに次の行動を考えさせ、必要ならツールを呼び、その結果をまたLLMに戻して次の判断につなげる。この繰り返しを実装する層です。

Claude Agent SDK、OpenAI Agents SDK、OpenHands、Docker AgentのRuntime部分などがこの層に入ります。

多くのOSS/SDKが最初にカバーするのがこの層です。チーム内の議論でもここに集中しがちです。

ただ、この地図の出発点は、L7だけでは基盤として足りないという問題意識です。

### スケールと協調（L8・L9・L10）

エージェントを「組織的に運用する」ために必要な層です。

**L10 Task Source**は、エージェントを動かす「仕事」の起点です。

GitHub Issues、Linear、Jira、Slack、Cronなどがここに入ります。

**L9 Durable Execution**は、その仕事を途中で落とさず進めるための層です。

長時間タスクのdispatch、停滞検知、失敗時の再試行、人間承認待ちなどを扱います。

Temporal、Cloudflare Dynamic Workflows、OpenAI Symphonyのようなワークフロー/オーケストレーション基盤が候補になります。

**L8 Multi-Agent Coordination**は、複数エージェントの役割分担や品質ゲートを扱う層です。

たとえば調査担当、実装担当、レビュー担当に分ける。あるいは複数のレビュアーを独立に走らせ、再現可能な指摘だけを採用する。

LangGraph、CrewAI、Claude Code Ultrareviewのようなパターンがここに入ります。

検証フェーズでは、手元のCLIから1回実行できれば十分に見えます。

しかし実運用ではIssueやSlackから継続的に仕事が入り、途中で落ちたタスクを再開し、必要に応じて複数エージェントで分担する必要があります。本番運用に近づくほど、この層の不在が効いてきます。

### 横断層（X・Y）

特定の層に属さず、すべての層にまたがる関心事です。

**X Observability**は、何が起きたかを後から追えるようにする層です。

どのモデル呼び出しにコストがかかったのか、どのツール呼び出しで失敗したのか、エージェントがどこで止まったのかを見えるようにします。

OpenTelemetry GenAI Semantic Conventions、Langfuse、Traceloop、Datadog、Honeycombなどが関係します。

**Y Supply Chain & Repo Contract**は、エージェントが動く前提をリポジトリ側で管理する層です。

AGENTS.mdのような指示書、Agent Skills、MCPサーバー、依存管理の仕組みがここに入ります。エージェントが毎回同じ前提で動けるようにし、ツールやスキルのサプライチェーンリスクも管理します。

ここまでが地図の全体像です。次に、この地図を持つことで何が変わるのかを見ていきます。

## OSS/SDK比較でこの地図が役立つ場面

レイヤーで整理する意味は、大きく3つあります。

### 1. OSS/SDKのカバー範囲が比較できる

新しいOSS/SDKが登場したとき、単純な優劣ではなく「10層のうちどこをカバーしているか」で見ることができます。

あるプロダクトがL7（Runtime）とL6（Capability）に強くても、L4（Sandbox）やL9（Durable Execution）は別途必要かもしれません。この見方があると、採用時に補うべき層まで含めて判断できます。

### 2. 足りない層が見える

チームの検証状況を地図に重ねると、「ここは学びがある」「ここは誰も見ていない」が一目でわかります。

自分たちのチームでは、L7（Runtime）についてはメンバーの検証で知見が溜まっていた一方、L9（Durable Execution）やX（Observability）はほぼ未着手でした。地図がなければ、この空白に気づくのはもっと遅れていたと思います。

### 3. 新しいOSS/サービスが出ても位置付けやすい

AIエージェント領域は新しいプロダクトやサービスが次々と登場します。地図があると、「これはL9の話だな」「L2の新しい選択肢か」と素早く位置付けられ、既存の構成のどこに影響するかが見えます。

### Runtime選定で起きるモデルロックインの判断

ここまでの3点を、実際のチーム議論で起きた例で見てみます。

メンバーは普段 Claude Code CLI を使っており、Claude Agent SDK であれば使い方をすでにイメージできる状態でした。学習コストの低さからL7（Runtime）の有力候補でしたが、Anthropic以外のモデルを使いたいという要求には対応できません。L1（Model）にロックインが発生します。

この整理があると、たとえば「短期的にはSDKのロックインを許容して社内展開を始め、並行してマルチベンダー対応のRuntimeへの移行設計も進める」といった選択肢が自然に議論のテーブルに上がります。

多くのレイヤーをまとめてカバーするプロダクトを使うと、立ち上げは速くなります。その代わり、どこかの層でロックインを受け入れることになります。

それを「良い/悪い」ではなく、「どの層のロックインを、どの時間軸で許容するか」という形で議論できる。これが地図の使いどころです。

## Docker Agentを10層アーキテクチャにマッピングする

地図の使い方を具体的に示すために、チームメンバーが実際に検証したDocker Agentを10層にマッピングします。

Docker Agentは、Docker社が提供するYAML/CLI駆動のAgent Harnessで、Docker DesktopやOCI registryとネイティブに統合されています。

このマッピングでは、「その層をDocker Agentに任せられるか」という観点で見ます。

充足度は、単体で任せやすいものを✅、基本機能はあるが外部補完が必要なものを⚠️、該当機能が確認できないものを❌としています。複数の観点を含む層では、`✅ / ❌` のように分けて表記します。重要なのは点数を付けることではなく、どの層を任せ、どの層を自分たちで補うかを見えるようにすることです。

| 層 | 充足度 | 備考 |
| --- | --- | --- |
| L1 Model | ✅ | 外部provider（OpenAI / Anthropic / Gemini / Bedrock）とlocal model provider（Docker Model Runner / Ollama）を設定可能 |
| L2 LLM Gateway | ⚠️ | 複数providerとmodel routing / fallbackはある。専用Gateway相当のキャッシュ・組織横断予算管理・障害時failoverは外部 |
| L3 Governance | ⚠️ | permissions / hooksで実行制御は可能。OWASP Agentic Top 10やAGT相当の包括的Governanceは外部設計が必要 |
| L4 Sandbox | ✅ | Docker Sandboxでネイティブ完結 |
| L5 Memory | ⚠️ | SQLite backed memoryとsession横断Tasksあり。自動的な知識抽出・整理を行う高度な長期メモリは外部 |
| L6 Capability | ✅ | built-in toolsが豊富。MCP / Skillsに加え、hooksでshell command / scriptも差し込める |
| L7 Runtime | ✅ | YAML/CLI駆動。OCI registryでエージェントをパッケージ配布 |
| L8 Multi-Agent | ✅ | sub\_agents / transfer\_task / handoffs / background\_agentsにより、セッション内の委譲・切替・並列実行をサポート |
| L9 Durable Exec | ⚠️ | Tasksでsession横断タスクDB、priority、dependenciesはある。停滞検知・再試行・workflow replayは外部 |
| L10 Task Source | ⚠️ | Tasksで内部タスクはsession横断管理できる。GitHub IssuesやSlackなど外部Task Sourceとのnative integrationは外部 |
| X Observability | ⚠️ | 匿名telemetryでcommand / tool / token / session eventsはある。OpenTelemetry GenAI準拠の運用監視・trace連携は外部 |
| Y Supply Chain & Repo Contract | ✅ / ❌ | Supply ChainはOCI registry配布で強い。Repo ContractはAGENTS.md相当の標準対応が確認できず、別途設計が必要 |

### マッピングから見えること

**L1 / L4 / L6 / L7 / L8が強い** — 外部・ローカルのモデルprovider、Docker Sandbox、豊富なbuilt-in tools、YAML/CLI駆動のRuntime、セッション内のmulti-agent実行までをDockerエコシステム内で扱えます。ローカル完結するエージェント基盤としてはかなり有力な選択肢です。

**外部補完が必要な層もはっきりしている** — L2は専用Gateway相当のキャッシュや組織横断の予算管理、L3はOWASP/AGT相当の包括的Governance、L9は停滞検知・再試行・workflow replay、XはOpenTelemetry GenAI準拠の運用監視が外部になります。

**Supply Chainは強く、Repo Contractは別設計** — OCI registryでagentを配布できるのはDockerらしい強みです。一方で、AGENTS.mdのようなrepoごとの作業契約を標準的に読む仕組みは別途設計が必要です。

大事なのは、Docker Agentを単純に評価することではありません。「どの層を任せ、どの層を別の手段で補うか」で議論できること — それがレイヤーで整理する意味です。

## おわりに

この記事では、AIエージェント基盤を10層+2横断のレイヤーモデルで整理する考え方を紹介しました。

要点をまとめます。

* AIエージェント基盤はRuntimeだけでは語れない。Sandbox、Gateway、Observability、Durable Executionなど、多くのコンポーネントで成り立っている
* 全体像を持つことで、チームの議論の前提が揃う
* OSS/SDKは「どのレイヤーをカバーするか」で比較すると見通しが良くなる
* どの層を外部に任せ、どの層を自前で持つかは、組織ごとの制約と時間軸に応じて判断する
* 新しいOSS/サービスが出てきても、レイヤーで理解しておくと「どこの話か」がすぐにわかる

自分たちのチームでは、この地図を共有してからメンバー間の議論が明らかに噛み合うようになりました。「これはL7の話をしている」「L9がまだ空白だから次はここを検証しよう」と、同じフレームで会話できるようになったのが一番の成果です。

この地図はまだ完成形ではありません。実際の基盤設計を進めるなかで、層の粒度や定義は更新し続けています。
