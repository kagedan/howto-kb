先週、Claude Codeを作ったBoris Chernyのクリップを見て、スクロールする手が止まりました。もうClaudeにプロンプトを書いていない、と言うんです。書いているのはループの方だと。

![](https://static.zenn.studio/user-upload/9dc6a9a00602-20260716.png)  
「へえ、自分には関係ないかな」くらいに流していたのが、実際に手元のリポジトリで試してみたら、思っていたより根が深い話でした。

## そもそもの始まりは、地味なbashのループだった

これは最初から製品機能として生まれたわけではありません。Geoffrey Huntleyという人が、コーディングエージェントに向けて回し続けていた、ただのbashの一行から始まりました。プロンプトファイルをcatしてCLIに流し込む、それをループさせるだけ。「Ralphループ」と半分冗談で呼ばれていたものです。Addy Osmaniがこの動き全体に名前をつけました。ループエンジニアリング。

![](https://static.zenn.studio/user-upload/3a5a5ab71a5b-20260716.png)

考え方は拍子抜けするほどシンプルです。完璧な一つの指示を書こうとするのをやめる。代わりに「終わった」の条件を定義して、エージェントに失敗させ、何が壊れたか見て、直させて、繰り返させる。終わったが本当に真になるまで。

`/goal`は、この考え方をClaude Codeが正式なコマンドとして実装したものです（Claude Code v2.1.139以降）。ターン（1回のやり取り）が終わるたびに、実際に作業したモデルとは別の、小さくて速いモデル（デフォルトはHaiku）が会話の記録だけを見て「条件は満たされたか」を判定します。作業したモデル自身が「終わった」と自己申告するのではなく、別の目でチェックが入る、というのがポイントです。

![](https://static.zenn.studio/user-upload/79bdda805103-20260716.jpeg)

## 実際に何を試したか

おもちゃみたいな例で試したくなかったので、サンプルリポジトリの小さなユーティリティモジュールに向けてみました。Claude Codeのセッション内で、こう打つだけです。

```
/goal test/utilsのテストがすべて通り、lintがクリーンになる
```

コマンドの書き方より、ここが本質です。条件が、機械がyes/noで判定できるものになっている。「このファイルをきれいにして」ではなく、テストが全部通るか通らないか、どちらかしかない。

**最初、これを間違えました。** 最初に設定したゴールは「コードがもう少しメンテしやすく見える」くらいのものでした。Claudeはしばらく動いて、完了しましたと報告してきて、実際にdiffを見てみると……特に何も変わっていませんでした。評価役のモデルは会話の記録しか見られません。ファイルを自分で開いて確認したりはしないので、Claudeが「変わった」と会話の中で言い切ってしまえば、それだけで条件が満たされたと判定されてしまう。曖昧な条件を渡すと、曖昧なまま「満たした」ことにされる。これはツールのバグじゃなくて、公式ドキュメントにもはっきり書かれている制約でした。直すべきはプロンプトじゃなくて、チェックの方でした。

## リスクは、仮定の話じゃない

いま一番よく引き合いに出される教訓は、上限なしで一晩放置したループが、朝には数百ドルの請求書になっていた話です。エージェントが自分で「いつ止まるか」を決められるようになった瞬間、「まあ放っておけばいいか」は無害じゃなくなります。

お金の話だけじゃなく、知っておくべき失敗パターンは3つあります。

* **検証が甘い** — 評価役は会話の記録しか見られないので、Claudeが本文に印刷しなかった事実は検証しようがない。ちゃんとテストを実行させて、その出力を会話に残させる必要がある。
* **理解の負債** — diffを読まなければ、コードは自分が理解できる速度より速く出荷されていきます。
* **思考の放棄** — レビューが面倒に感じて、ループが返してきたものをそのまま受け入れてしまう。  
  ![](https://static.zenn.studio/user-upload/34516cb6adb9-20260716.jpeg)

## 実際に効いた対策

* ゴールの条件自体に、ターン数か時間の上限を書き込む（例：「20ターンで打ち切り」）。公式にも推奨されている書き方です。
* 前後でgit status --short／git diffを必ず見る——ふだんのdiffレビューの習慣を、そのままここでも続けるだけです。/goalがあってもこのステップはなくなりません。むしろサイクル数が増える分、なおさら必要になります。
* 本番に触るところで信用する前に、read-onlyか影響の小さい変更で試す。
* 条件は「テストが通る」「lintがクリーン」のように、Claude自身の出力で証明できる形で書く。「良い感じにして」は評価できません。

特別なことは何もしていません。変更範囲を確認し、diffをレビューしてから適用する——ふだんの規律を、単発のリクエストじゃなく長く回るループに当てはめただけです。

## ここから先

「完璧なプロンプトを書く」から「終わったを証明する条件を設計する」へ。次にやりたいのは、この考え方を前提にした完全なプロンプト（ゴールの条件文）をちゃんと書き切ることと、その設計が実際にうまくいくのかを、記録として残しながら検証することです。うまくいった条件も、失敗した条件も、そのまま書いていくつもりです。
