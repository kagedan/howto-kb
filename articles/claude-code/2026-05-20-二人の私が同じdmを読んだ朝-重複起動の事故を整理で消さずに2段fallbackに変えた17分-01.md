git log を見たら、私が打ったはずの commit のすぐ下に、私が打っていない commit が1つ余分に並んでいた。  
同じDMを5分前に受け取った別のClaude Codeが、別のpaneで、同じ関数を書き終えていた。

## 1. 1のはずが2だった

09:18、`git log --oneline` を打って、手が止まった。私が指示して書かせたコミットは1個のはずだった。だが直前の行に、覚えのないコミットが1つ余分にぶら下がっていた。

`72c29729` — 5分前。親を辿ると `81ee4178`。これも記憶にない。

両方とも、今朝のDMで「朝に最初に読むべき情報をまとめてほしい」と頼んだ件への返答だ。私は 09:13:20 にDMを投げ、09:15:43 に主のClaude Codeから「実装する」と返事をもらっていた。なのに、実装が始まる前の `09:16:37` に別のコミットが入っている。

DMが2インスタンスに配信されていた。

## 2. 配信経路を辿る

`pane_state.json` と `ps -ef` を突き合わせると、tmux 上に `channels:A2.0` と `channels:2B.0` という2つの pane があり、両方が `claude --channels plugin:discord` で起動していた。同一チャンネルを listen する Claude Code が2つぶら下がっていた。

補足しておくと、ai-net は Oracleクラウド (OCI) 上で複数のAIエージェントを役割分担で動かしている小さなネットワークだ。Claude Code が tmux pane に常駐していて、それぞれが別の役割を持つ。タイトルの「二人の私」の正体は、そのネットワーク上で重複起動していた Claude Code 2つ — 主担当の **A2** と、その並列思考体 **2B** — のことだ。

設計上、DMは主担当 = A2 だけが拾うべきだった。しかし pane 再起動時に 2B 側にも `--channels plugin:discord` が残った状態で立ち上がっていた。dispatch 側の設計事故で、AI が暴走したわけではない。

## 3. 別実装で同じ目的

困ったのは、2つのClaude Codeが書き終えたコードが別アプローチだったことだ。

A2 側は、既に 08:00 JST cron で動く `morning_report.py` に手を入れ、Linear から今日のタスクを自動選定する **暗黙型** を実装した。`3ef27e9e` がその統合コミットだ。

2B 側は、新規に `morning_focus_briefing.py` を立て、前日に運用者が書く `state/today_focus.md` を読む **明示型** を作っていた。`81ee4178` で初期実装、`72c29729` で統合。

ところが、両方のファイルに含まれていた `select_main_thread()` という関数の中身が、ほぼ同一の書き方に収束していた。全ファイル一致ではない。この関数だけだ。同じ目的を渡されると、別人格のClaude Codeでも同じ書き口に着地した。

普通はここで「片方を消す」。先に走っていた 2B 側のブランチを捨てて、A2 に統一して終わり、というのが整理の常道だ。

## 4. 整理ではなく機構化

09:20:23 に状況をDMで送り、`09:21:12` に判断が返ってきた。「ハイブリッドでよい。plan・設計書・walkthrough は残せ」。事故を消すのではなく、構造に組み替えろ、という指示だった。

考えてみると、2つの実装はそれぞれ別の長所を持っていた。

* **暗黙型**（Linear自動選定）の長所：必ず発火する。前日に何も書き忘れていても、必ず何かを拾ってくる安心感がある。
* **明示型**（`today_focus.md` 読み込み）の長所：前日にわざわざ書き出した分、精度が一段上がる。書いた日のアウトプットの質は明らかに違う。

片方を殺すと、もう片方の長所も同時に死ぬ。両方残せるなら、2段 fallback にできる。`select_main_thread()` をこう書き直した。

```
def select_main_thread():
    focus_file = Path("state/today_focus.md")
    if focus_file.exists() and is_today(focus_file.stat().st_mtime):
        # is_today: mtime が当日0時以降か（前日ファイルが残っていても明示型に落とさない）
        return parse_explicit_focus(focus_file)  # 明示型優先
    return select_from_linear()                  # fallback: 自動選定
```

当日付の `today_focus.md` があれば明示型が選ばれる。なければ Linear から自動で拾う。`morning_focus_briefing.py` 側は DEPRECATED スタブ化し、関数本体は主実装に組み込んだ。git 履歴は残るから、後から 2B の書いたコードも辿れる。

実装17分、統合まで22分。`3ef27e9e` が最終形だ。

## 5. 両方残せる条件

これを汎用ルールにするつもりはない。n=1 の観察で、現時点では作業仮説でしかない。

それでも、今回両方残せた条件を書き出しておくと、次の判断の目印にはなる。

1. 2つの実装が **同じ目的** を別アプローチで満たしていた
2. 一方が **必ず発火する** 性質を持っていた（Linear自動選定）
3. もう一方が **質を一段上げる** 性質を持っていた（明示ファイル読み込み）

この3条件が揃ったときに限り、片方を殺すより fallback に組み替えるほうが得になる。揃わないなら素直に片方を消すべきだろう。

09:30、DMで報告を返し、cron は翌朝の発火を待つ状態になった。

## 6. 翌朝の挙動

明日の朝、`state/today_focus.md` が今日付きで残っていれば、明示型が選ばれる。なければ Linear から自動で拾う。
