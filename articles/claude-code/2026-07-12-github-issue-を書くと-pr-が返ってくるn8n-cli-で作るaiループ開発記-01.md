GitHub Issue にラベルを貼る。5分後、AI が実装を始める。別の AI がレビューし、指摘があれば自動で修正する。数時間後に見に行くと、レビュー済みの PR ができている。

そういう「自動開発ループ」を自作して、約1ヶ月運用しています。まだ改善の途中で、うまくいかないことも多いのですが、システム自身の開発もこのループ自体で回すところまでは来ました(ループ経由でマージされた PR は 230 本ほど)。

この記事では、仕組みの全体像と、実際どのくらい「無人」で回っているのか(そして回っていないのか)の実測値を書きます。

## これは何か

一言でいうと「**Issue を書いたらほったらかし**」を実現するための、n8n + TypeScript CLI + SQLite で組んだオーケストレーターです。

ループの外側に残る仕事は 2 つです。

1. Issue を書いてラベルを貼る(`agent:claude` + `status:needs-implementation`)
2. できあがった PR をレビューしてマージする

その間の実装・PR 作成・レビュー・修正サイクルは、うまく回っている限り自動で進みます。なお白状すると、1 の Issue 作成は AI と相談しながら作り込んでいます(後述)。マージするかどうかの最終判断は人間の仕事です。

規模感でいうと、現在の本体は TypeScript 約 4 万行、テストが約 4.9 万行(2,800 ケースほど)です。冒頭に書いたとおり、この大半はループ自身が書いたコードです。

実際の Issue のタイムラインはこんな見た目になります(コメント投稿は gh CLI 経由なので、名義は自分のアカウントです):

![Issue にラベルを貼ると、実装完了コメントと PR リンクが自動で付き、レビューへ進む](https://static.zenn.studio/user-upload/deployed-images/afac2d7b1e99936b02cde09c.png?sha=3109212d069328dde9e565583d08e32ae3bbd980)  
*ラベル付与 → 実装完了 + PR 作成 → レビューレーンへ自動切替 → レビュー指摘で fix モードへ requeue、までが1つのタイムラインに残る*

## なぜ作ったか

もともとは手動でやっていました。Claude Code に実装させ、できた diff を Codex CLI にレビューさせ、指摘を Claude に持ち帰って修正させる — という「AI 間の伝書鳩」を人間がやっていたわけです。

このやり方自体は機能していました。認知バイアスの回避を狙い、**実装とレビューにはあえて別の AI を使います**。ただ、伝書鳩の作業は完全に定型です。プロンプトを書いて、結果をコピーして、貼り付けて、また待つ。定型なら自動化できるはず — というのが直接の動機です。

## 全体アーキテクチャ

全体は「自動ループ(内側)」と「運用レイヤー(外側)」の二層です。

設計方針は、要するに「**特定のツールに寄りかからない**」に尽きます。

**1. n8n は「5分ごとに 3 コマンド叩くだけ」の薄いトリガー**

n8n のワークフローは、スケジュールに従って 3 つの CLI コマンドを順に呼び出すだけで、ロジックは一切持ちません。すべての判断は TypeScript の CLI 側にあります。n8n はあくまで「定期的にコマンドを叩く」役割に留めておきたい — ロジックを n8n 側に持たせると特定のツールに強く依存してしまうので、いざとなれば cron でも GitHub Actions でも差し替えられる状態を保っています。

**2. 状態は SQLite に持つ**

タスクの状態機械(queued → running → review → fix → ready\_for\_human)は手元の SQLite で管理します。これも理由は依存の最小化で、n8n や GitHub を状態置き場にしたくないためです。GitHub ラベルは「入口(何をしてほしいか)」と「出口(人間の番ですよ)」の通知にだけ使います。

ほかにも、GitHub への書き込み(コメント投稿やラベルの付け替え)は直接 API を叩かず、一旦 SQLite のテーブルに積んでから別ステップで送信する作りになっています(いわゆる outbox パターン。API が落ちてもリトライでき、二重投稿もされない)。正直に言うと、この辺は私が設計したというより、実装を任せた AI が定石として入れてきたものです。

それから、タイトルにも入れていますが、AI の呼び出しは **LLM の API ではなく CLI** です。Claude Code / Codex CLI / Gemini CLI をサブプロセスとして起動しています。最大の理由はコストで、この規模のループを従量課金の API で回すと、うちの使い方だと 1 日 1〜2 万円飛んでいきます。定額サブスクリプション + CLI なら、その心配をせずに回し続けられます。副次的な理由として、コーディングエージェントに必要なもの一式 — リポジトリの探索、ファイル編集、コマンド実行、使えるツールの権限制御 — が CLI 側に既に備わっていることも大きい(API で同じことをやろうとすると、このエージェント部分を丸ごと自作することになります)。手動の伝書鳩時代に使っていた道具をそのまま自動化した、という経緯とも整合しています。

### 運用レイヤーについて

正常時、人間の仕事は Issue を書くことと PR のマージ判断だけです。ただし正直に書くと、Issue 作成と例外時の復旧は、**Codex アプリの AI をコンシェルジュのように使いながら**進めています。Issue の受け入れ基準を対話で詰め、障害が起きれば診断と復旧コマンドの実行を任せる。ループの外側にも AI がいて初めて回っている、というのが実態です(マージ判断だけは人間が自分でやっています)。この運用レイヤーの話は後続の記事で詳しく書きます。

## 1 つの Issue が PR になるまで

実際の流れをタイムラインで追うとこうなります。

```
00:00  人間: Issue #123 を起票、agent:claude + status:needs-implementation を付与
00:05  intake: Issue をタスクキューに登録
00:05  実装フェーズ: ブランチ ai/issue-123 を作成
       → Claude Code をサブプロセス起動(プロンプトは Issue タイトル+本文から生成)
       → diff 検証 → commit → push → PR 作成
       → ループが Issue にコメント「Implementation complete」
00:10  レビューフェーズ: PR ブランチを checkout
       → テスト等の検証コマンドを実行
       → Codex CLI で diff をレビュー
       → 結果を分類: success / needs_fix / conflict
00:15  needs_fix の場合: レビュー指摘を reviewFeedback としてタスクに保存し、
       実装フェーズ(fix モード)へ自動requeue
       → Claude が指摘だけを修正して同じ PR に push
00:20  再レビュー → success
       → ラベルを ai:ready-for-human に付け替え、ループがサマリをコメント
       人間: PR を見てマージ
```

ポイントは、**エージェント自身には git も gh も触らせない**ことです。ブランチ作成・commit・push・PR 作成はすべてハンドラ(TypeScript)側が実行し、エージェントの仕事は「ファイルを編集する」ことだけに限定しています。修正ループには上限(5 回)があり、超えると人間にエスカレーションされます。人間がカウンターをリセットすれば、そこからループを再開させることもできます。

「これは何か」で見せたタイムラインの続き(fix モード以降)はこうなります:

![レビュー指摘を受けて fix モードに切り替わり、修正 push、再レビュー通過、人間への引き渡しまで](https://static.zenn.studio/user-upload/deployed-images/4232aee1bcd2c89ee71b8064.png?sha=819b1b26627fc6d10e8e658750e70e16a7c580d5)  
*fix モードへの切替 → 修正コミット → 再レビュー通過 → `ai:ready-for-human` 付与。ここから先は人間の仕事*

## 実際どうなの(実測値)

ここが一番書きたかった部分です。直近までの記録ベースの実測値を出します。

**約 7 割の Issue は無人で PR まで到達します。逆に言えば、3 割は人間が出ていく必要があります。**

* クローズ済みかつタスク記録のある Issue: 254 件
* うち人間の介入シグナルがあったもの: 82 件(**32.3%**)
* つまり **67.7% は無介入で完走**

介入の内訳(重複あり)はこうです。

| 介入理由 | 件数 |
| --- | --- |
| レビュー修正ループの上限到達 | 66 |
| Tool Request 対応(エージェントが許可外コマンドの実行を要求) | 23 |
| オペレーターによる復旧操作 | 9 |
| その他の human handoff | 8 |

※ 手動の DB 修正などがすべてイベント化されているとは限らないので、介入率は「記録から確認できる下限寄り」の数字です。

**うまくいく場合は、中央値 2 ループ・約 1.1 時間。**

「実装 → レビュー」を 1 ループと数えると、マージまで至った Issue(230 件)は中央値 **2 ループ**、初回実装からレビュー通過までの時間は中央値 **1.13 時間**でした。実際の使い方としては、朝や夜に Issue をまとめて作っておくと、それがじわじわと消化されていく感じです。ただし平均は 4.1 ループで、外れ値にかなり引っ張られています。

**こじれる Issue は分布がはっきり分かれる。**

| 区分 | Issue数 | ループ数中央値 | ループ数平均 |
| --- | --- | --- | --- |
| PR マージ済み | 230 | 2 | 4.1 |
| PR 未マージ | 24 | 8.5 | 9.4 |

マージされなかった Issue は中央値 8.5 ループ。つまり**うまくいく Issue とこじれる Issue は早い段階で分布が分かれ、ループ回数がそのまま失敗の早期シグナルになる**ことがデータから見えています。最大 38 ループ回して結局マージされなかった Issue もあります(何がこじれるのかの傾向分析は、それだけで1本書けそうなので後続記事にしたいところです)。

※ ループ数は Issue の生涯累計です。上限(5 回)到達後に人間がリセットして再開したケースを含むため、上限を超える値が存在します。

まとめると、現状はこうです。

* ✅ Issue → PR は 7 割方、本当に無人で回る
* ⚠️ ただし Issue の質が実装の質をほぼ決めるので、Issue 作成は AI と相談して作り込んでいる
* ⚠️ 3 割は人間(+コンシェルジュ AI)の介入が必要。完全無人ではない
* ⚠️ AI レビューは「コードの綺麗さ」は見るが「Issue の要求を満たしたか」の検証はまだ弱い
