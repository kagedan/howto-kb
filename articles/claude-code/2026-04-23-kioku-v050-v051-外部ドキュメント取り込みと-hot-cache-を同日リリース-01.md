## はじめに

Claude Code / Desktop の記憶 OSS「KIOKU」を作っています。[前回の v0.4.0 記事](https://zenn.dev/megaphone_tokyo/articles/e31afe2926de5c) は「新機能ゼロで既存コードを監査して直したリリース」の話でした。

その続きとして、2026-04-23 に **v0.5.0 と v0.5.1 を同日に** 出しました。

* **v0.5.0**: PDF / Markdown / EPUB / DOCX の統一 ingest router
* **v0.5.1**: session 跨ぎの短期記憶を繋ぐ hot cache + PostCompact hook

「**外部の知識を取り込む (ingest) → 取り込んだ知識を session 間で持続させる (persist)**」という連続した一対のリリースです。この記事ではそれぞれの設計判断と、間に挟まる「**作らなかった機能**」の話を書きます。

<https://github.com/megaphone-tokyo/kioku>

## v0.5.0: 外部ドキュメント取り込みの統一

### 変更点

v0.5.0 までは PDF 用・Markdown 用・URL 用で**別々の MCP tool** が並んでいました。呼び出し側 (Claude Code / Desktop) が拡張子を見て正しい tool を選ばなきゃいけないし、将来 RTF / ODT を足すたびに tool が増えていく。

v0.5.0 では `kioku_ingest_document(path)` という 1 本の入り口を切って、router が拡張子で dispatch する形に変えました。

```
kioku_ingest_document("論文.pdf")   → PDF handler
kioku_ingest_document("note.md")   → PDF と同じ extractor (plain text chunk)
kioku_ingest_document("book.epub") → EPUB handler (新規、yauzl ベース)
kioku_ingest_document("spec.docx") → DOCX handler (新規、mammoth + yauzl)
```

旧 `kioku_ingest_pdf` は deprecation alias として v0.5〜v0.7 window で残存、v0.8 で削除予定。新規統合では `kioku_ingest_document` を使う導線にしました。

### EPUB 取り込み: 8 層防御の ZIP 展開

EPUB は実体が ZIP なので、攻撃者が vault 配下に書き込みたがる典型的な流入経路です。Readability に食わせる前に、**zip-slip / symlink / 爆弾 EPUB / XXE / XHTML 内 script 経由 XSS** をまず止める必要がある。

実装は [yauzl](https://www.npmjs.com/package/yauzl) ベースで 8 層の防御を直列に並べました。

| 層 | 防ぐもの |
| --- | --- |
| 1 | zip-slip (`../` を含む entry path) |
| 2 | symlink entry (展開時に vault 外を指すリンク) |
| 3 | 累積 size cap (爆弾 EPUB、展開後 500 MB まで) |
| 4 | entry count cap (ZIP 爆弾の別変種、10,000 エントリまで) |
| 5 | NFKC filename 正規化 (CJK 記号の同一視、Unicode normalization attack) |
| 6 | nested ZIP skip (入れ子 ZIP は展開しない) |
| 7 | XHTML に対する XXE pre-scan (`<!DOCTYPE` 拒否) |
| 8 | XHTML 内 `<script>` / `on*=` sanitize |

全層を通過した XHTML だけを Readability + Turndown で Markdown 化し、`spine` (EPUB の目次順) の順で `.cache/extracted/epub-<subdir>--<stem>-ch<NNN>.md` に保存します。2 章以上ある場合は `-index.md` も一緒に出力して、auto-ingest cron が後から LLM 要約を作るときに目次として使えるようにしました。

### DOCX 取り込み: mammoth + yauzl の二段構え

DOCX は ZIP + XML なので、素直に [mammoth](https://www.npmjs.com/package/mammoth) で Markdown 変換できます。ただ mammoth は内部で `jszip` を使っていて、ここが XXE / zip-slip に対して無防備です。

なので KIOKU の DOCX handler は、**mammoth に渡す前に yauzl で ZIP を先に開く**構造にしました。

1. `yauzl` で ZIP を開き、8 層防御 (EPUB と同じ) を通す
2. `word/document.xml` と `docProps/core.xml` に `assertNoDoctype()` で XXE pre-scan
3. 全部通過したら、ZIP を閉じて mammoth に `Buffer` を渡して Markdown 変換
4. Metadata は `--- DOCX METADATA ---` fence で囲って **untrusted** 注釈付きで保存

画像 (VULN-D004/D007) と OLE 埋め込み (VULN-D006) は MVP では skip しました。画像対応は EPUB と同じ pattern で足せますが、OLE は threat surface が広すぎるので別 PR で詰める方針です。

### Unicode ファイル名対応

EPUB / DOCX の extractor は cron (`auto-ingest.sh`) から argv でファイル名を受け取るので、`raw-sources/<subdir>/<stem>.<ext>` を分解する正規表現が要ります。ASCII 前提の `\w` では日本語・中国語のファイル名が拾えないので、Unicode プロパティ escapes を使いました。

```
const m = argv[2].match(/raw-sources\/([\p{L}\p{N}_-]+)\/([\p{L}\p{N}_-]+)\.(epub|docx)$/u);
//                                     ^^^^^^^^^^^^^^^^^                   ^^^^^^^^^^^^^^^^^   ^
//                                     Unicode-aware                       Unicode-aware     unicode flag
```

`\p{L}` は任意のスクリプトの文字 (Latin / Cyrillic / CJK / Arabic / Devanagari など)、`\p{N}` は数字。`/u` flag が必須です。これで `論文.epub` や `日本語メモ.docx` も cron 経路で正しく拾えます。

## v0.5.1: hot cache で session 間の記憶を繋ぐ

release 同日、続けて v0.5.1 を出しました。「外部から取り込んだ知識を、**次のセッションが始まったときに呼び戻す**」側の話です。

### 何が解決したかったか

Claude Code は長い session を走らせると途中で context を **compaction** (要約圧縮) します。compaction 後は直前までの作業記憶がかなり減っていて、LLM が「さっきの話忘れた」状態になりがちです。

KIOKU は以前から `SessionStart` Hook で `wiki/index.md` を system prompt に注入していましたが、これは「蓄積された全知識の目次」であって、「**いま進行中の短期記憶**」ではない。

### 設計: hot cache pattern

解は `wiki/hot.md` という 1 ファイルです。

```
---
type: hot-cache
updated: 2026-04-23
---

## Recent Context

- いま何を作っているか
- 直前の session で決めた設計判断
- 次の session で覚えておきたいコンテキスト
```

500 word soft limit、hard cap 4000 文字。LLM が session 終わりに書き換える前提の、短いメモです。これを SessionStart と **PostCompact (新しい hook event)** の両方で注入します。

| Event | 注入内容 | 理由 |
| --- | --- | --- |
| SessionStart | hot.md + index.md の両方 | session 冒頭は全 context が必要 |
| PostCompact (新) | hot.md **のみ** | index.md は context に残存、token 節約 |

### 設計判断: Stop hook は default opt-out

「session 終わりに hot.md を書き換える」prompt を Stop hook で出すかどうかは、中で議論がありました。結論は **default opt-out**、`KIOKU_HOT_AUTO_PROMPT=1` を明示的に set したときだけ出る形にしました。

理由はセキュリティ境界の違いです。`session-logs/` は各マシンローカル (`.gitignore` で除外) で Git に入りません。でも **`hot.md` は Git sync 対象** なので、書き換え内容がそのまま GitHub private repo に commit されます。マスクされるはずの API key が紛れ込んで push される経路ができうる。

「自動化で気持ちよく UX」を優先したい気持ちはあるけど、**隔離境界が違うものを同じ自動化の扱いにするのは良くない**、という判断でした。opt-in すれば使える、しなければ出ない、で分ける。

### Security boundary の実装

* hot.md 注入前に `applyMasks()` 適用 (API key / token パターン伏字化、既存 `mcp/lib/masking.mjs` を共有)
* `scan-secrets.sh` の scan 対象に `wiki/hot.md` を追加
* `realpath` で symlink escape (vault 外パス参照) を拒否
* 4000 文字超過は truncate + WARN log

### Claude Code v2 の per-event schema 対応に 4 回 iteration

hot cache の実装自体は数時間で終わりましたが、Claude Code v2 の hook output schema が **event ごとに違う** ことが分かって、release 当日 (2026-04-23) の release セッション内で 4 回の schema 対応 iteration を重ねることになりました (全て v0.5.1 タグ前に commit)。

v1 では全 event で共通の flat schema が使えました。

```
{ "additionalContext": "hot.md の内容..." }
```

v2 ではこれが event によって形が違います。

| Event | 使うべき schema |
| --- | --- |
| PreToolUse / UserPromptSubmit / PostToolUse | `hookSpecificOutput` wrapper |
| SessionStart | `hookSpecificOutput` (lenient) |
| **PostCompact** | **top-level `systemMessage`** (hookSpecificOutput 非対応) |
| **Stop** | **top-level `systemMessage`** (hookSpecificOutput 非対応) |

v2 で旧 v1 flat を渡すと **validation error を出さず silent に無視** されます。dev 中に気づくのに時間がかかる failure mode です。release 当日 (2026-04-23) の iteration は:

* **round 1**: SessionStart injector を `hookSpecificOutput` でラップ
* **round 2**: hot.md を index.md より前配置 + `MAX_INDEX_CHARS=10KB` cap
* **round 3**: PostCompact が silent 無効だったことに気づき、top-level `systemMessage` に差し替え
* **round 4**: Stop の opt-in prompt も同じ問題だったことに気づき、top-level に統一 (PM review で catch、release 前)

round 3 の時点で、本来は全出力箇所に grep をかけるべきでした。「schema が event ごとに違う」と分かった時点で、同じ schema を使っている全箇所を一括で audit すれば、round 4 は要らなかった。session-logger.mjs:369 の Stop opt-in prompt が v1 flat のまま残っていたのは、「SessionStart と PostCompact を直した」という部分最適で満足していたせい。最後の PM review で catch されて release 前に塞げたものの、review が無ければそのまま ship されていた経路でした。

### 組織知化: LEARN#9「外部 schema drift は grep 監査とセット」

この経験を開発ルールに組み込みました。

> 外部 API / CLI schema (Claude Code v2 hook schema 等) に依存する箇所は、**1 箇所修正した時点で「同じ schema を使っている全箇所」を grep で列挙し、同一 PR 内で audit する**。さもないと per-site 修正で片側だけ v2 対応されて **silent 無効化の regression** を生む。

具体的な再発防止:

```
# 外部 schema を更新する時は必ず grep で全出力箇所を列挙
grep -rn "stdout.write.*JSON.stringify" hooks/ scripts/
```

さらに、**「古い shape の不在を assertion する」negative test** を入れました。

```
assert.ok(!parsed.additionalContext, 'v1 flat schema を使わない');
```

negative assertion は schema migration で特に価値があります。通常のテストは「期待する shape が存在する」を見るだけなので、半分だけ移行した regression が通る。「古い shape が存在しない」を pin すれば、中途半端な rollback や部分 migrate を自動で catch できる。

## 作らない判断: defuddle を skip した話

v0.5.1 の直後、本当は v0.5.2 で **defuddle** (web 広告 / boilerplate 除去で token 40-60% 削減) を作る予定でした。roadmap 上は確定事項として積んでいて、既存 LLM Wiki 系 OSS でも打ち出されている機能です。

で、着手前に **実測データを取る probe** を 30 分ほどやりました。

### 測ったこと

| 指標 | 実測 | 判定 |
| --- | --- | --- |
| `raw-sources/articles/` 10 件の body 長 | 1,640 〜 170,204 bytes、中央値 ~10KB | **全件 substantial、抽出失敗ゼロ** |
| `session-logs/` 38 件 grep (`short-content` / `readability.*fail` / `extracted.*empty`) | 0 件 | **実害の証拠なし** |
| `.cache/extracted/url-*` の失敗 cache | 0 件 | 判断材料外 |

既に使っている `@mozilla/readability` が広告 / boilerplate を十分除去していて、defuddle の前段を足す理由が測定データに存在しない、という結論。

### skip 判定

2 つの選択肢がありました:

* **(A)** roadmap 通り defuddle を足す (6-10h)。「データで不要でも、将来のために入れておく」路線
* **(B)** skip する。6-10h を v0.6.0 Phase C (LP β + plugin marketplace 登録) に振り分ける

(B) を選びました。判断を `handoff/open-issues.md §19` に **再燃条件付き**で記録しています。

> 以下のいずれかが観測された時点で本 §19 を close、Phase B' を再評価:
>
> * `raw-sources/articles/` に Readability が空に近い抽出 (< 500 bytes) を生成するケースが複数発生
> * session-logs で iframe / embed / JavaScript-heavy SPA の本文抽出失敗が user 体感で記録される
> * 競合が defuddle 採用で明確な UX 優位を示した時

「今は要らないけど、こういうシグナルが出たら再検討する」という形にしておけば、「作らない判断」が永久棚上げにならずに済みます。

### 考えたこと: 測ってから作る、という落ち着き

個人 OSS だと「roadmap に書いた」「競合がやってる」「技術的に興味ある」で作ってしまいがちです。でも、**データで「不要」と示された機能を作ると、自分のコードベースに保守負担が乗る**。EPUB 8 層防御を見てもらうと分かるけど、KIOKU はもう単純な toy project ではなくて、新機能を 1 つ足すたびに threat surface と test suite が膨らむ。

30 分の probe で 6-10h + その後の保守負担を回避できるなら、**実測ベースで「作らない」と判断する** のは十分 ROI が高い。副次的に、**再燃条件を書く**という行為が「なぜその機能が要るのか」を言語化させてくれるので、「何となく roadmap に載せていた」項目の棚卸しにもなります。

## それ以外に入ったもの

上の 3 つの話に絞りましたが、v0.5.0 と v0.5.1 には他にも小さい修正が入っています。

* **v0.5.0**: `fast-xml-parser` の XMLBuilder 系 CVE-2026-41650 については、**KIOKU は XMLParser (読み込み側) しか使っていないので exploit 不可**。dependabot alert 解消のため v0.5.1 で `^5.7.0` upgrade 済。`html-sanitize` の `DANGEROUS_TAGS` に `meta` / `base` / `link` を追加 (EPUB consumer 経路向け defense-in-depth)
* **v0.5.1**: 上述の security boundary (applyMasks / scan-secrets 拡張 / realpath / truncate) に加え、hot.md は `wiki/` 配下の Git sync 対象に入るので、`templates/vault/CLAUDE.md` に opt-in 運用ルールを追記
* テスト: v0.5.0 は Bash 158 assertions + Node 全 suite green、v0.5.1 は Node 47 件 + Bash 488 assertions 全 green。`npm audit` は runtime deps で 0 vulnerabilities

詳細は [v0.5.0 Release Notes](https://github.com/megaphone-tokyo/kioku/releases/tag/v0.5.0) と [v0.5.1 Release Notes](https://github.com/megaphone-tokyo/kioku/releases/tag/v0.5.1) にまとまっています。

## v0.6 に向けて

v0.5 系で ingest → persist の基盤はだいぶ揃ったので、次は **外部 user を迎え入れる** フェーズに入ります。v0.6.0 Phase C で予定しているのは:

* **Agent Skills cross-platform 対応**: Cursor / Windsurf / Gemini CLI / Codex に symlink で広げる
* **Claude Code plugin marketplace 登録**: `claude plugin install kioku@megaphone-tokyo` で入る
* **Delta tracking**: `.raw/.manifest.json` で source-level hash 管理 (同 PDF 2 回 ingest で skip)
* **Obsidian Bases dashboard**: `wiki/meta/dashboard.base` で ingest 履歴 / orphan 一覧を動的表示
* **LP β (note.com ベース) + Discord soft launch**: 1 人 dogfooding から外部 feedback を受け取る体制へ

「1 人で作って 1 人で使う」から「他人に使ってもらって、feedback を受ける」に移る最初の release になる予定です。
