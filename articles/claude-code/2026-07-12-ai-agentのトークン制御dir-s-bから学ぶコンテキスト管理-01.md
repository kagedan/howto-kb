僕が初めてAI Agentのトークン制御についてちゃんと考えたのは、複雑なマルチAgentシステムからではなく、ごく普通のWindowsコマンドからでした。

Windows Command Prompt、つまり `cmd.exe` では、次のようにファイルを再帰的に一覧表示できます。

見出しやタイトルを簡潔にするため、以降（タイトルを含む）ではこのコマンドを `dir /s /b` と略記します。ただし本文中のすべての例・出力・議論は、`/a:-d`（ディレクトリを除外し、ファイルだけを対象にする）を含むこの完全なコマンドを前提にしています。

このコマンドがやっていることはシンプルです。カレントディレクトリ以下のファイルパスを再帰的に一覧表示し、ファイルだけを対象にしてディレクトリは対象にしません。

ここで前提を先に整理しておきます。`dir /s /b /a:-d` は `cmd.exe` における `dir` の引数のスタイルです。PowerShellでも `dir` と入力できますが、これは通常 `Get-ChildItem` のエイリアスであり、引数の書き方や意味は完全には一致しません。PowerShellで似た意図を表現するなら、僕なら次のように書きます。

```
Get-ChildItem -Recurse -File
```

この記事で論じたいのは `dir` コマンドそのものではなく、「ファイルを再帰的に一覧表示する」というこの小さな例を通じて、AI Agentが動いているときに実際どこにトークンを使っているのか、そしてコンテキストをどう管理すべきなのかを理解することです。

## 結論を先に：OSコマンドの実行はLLMトークンで課金されない

以前の僕は、「Agentがコマンドを呼び出した」ことと「モデルがトークンを消費した」ことを混同しがちでした。

その後、この2つを切り分けてみると、境界は実はとても明確でした。

* Windowsが `dir /s /b /a:-d` を実行すること自体は、LLMトークンで課金されない。
* Shellが生成する `stdout` は、単なるプロセスの出力にすぎない。
* Runtimeやツール実行層が `stdout` を捕捉することは、それがモデルのコンテキストに入ったことと同じではない。
* シリアライズされてモデルのコンテキストに送られた入力テキストだけが、LLMの入力トークンコストを生む。
* モデルが生成するツール呼び出しや通常の回答、最終回答などのテキストが、LLMの出力トークンコストを生む。
* 後続のターンでツール呼び出しの記録やツール結果が保持されていれば、それらも再び入力コンテキストの一部になりうる。

つまり、「コマンドの出力が存在する」こと自体がLLMトークンを消費するのではなく、「その出力がモデルの可視コンテキストに置かれる」ことがLLMトークンコストになるのです。

もちろん、コマンドの実行にはCPU、ディスクIO、プロセス時間、プラットフォームの実行クォータ、ツール呼び出しの制限など、他のコストがかかる場合もあります。ただしこれらはLLMトークンコストではありません。Agentエンジニアリングの観点では、まず2種類のコストを分けて考えます。1つはruntime/toolの実行コスト、もう1つはモデルの入出力トークンコストです。

## `dir /s /b /a:-d` は実際に何をしているのか

以降の議論がぶれないように、まずこの `cmd.exe` コマンドを分解しておきます。

* `dir`：ファイルとディレクトリを一覧表示する。
* `/s`：サブディレクトリを再帰的にスキャンする。
* `/b`：bare formatを使い、見出しやサイズ、日付などの余計な情報を出さず、パスだけを出力する。
* `/a:-d`：ディレクトリを除外し、ファイルだけを一覧表示する。

なので、その出力は通常こんな感じになります。

```
C:\project\README.md
C:\project\src\app.ts
C:\project\src\components\Button.tsx
C:\project\package.json
```

プロジェクトが小さければ問題ありません。でも再帰スキャンが大きなディレクトリに当たると、出力の規模は簡単に制御不能になります。`node_modules`、ビルド成果物、ログディレクトリ、キャッシュディレクトリを含むプロジェクトなら、数千行、時には数万行のパスが出力されることも珍しくありません。

本当に危険なのは、Shellが2万行を出力できるかどうかではなく、Agentのツール層がその2万行をそのままモデルのコンテキストに詰め込んでしまうかどうかです。

Shell、Runtime、ツールのラッピングの違いによって、コマンドがどう実行され、出力がどう捕捉され、バッファ制限があるかどうかは変わります。でも実装の詳細がどうであれ、僕は今、いつも同じ問いを立てます。「元の出力は最終的にLLMの可視コンテキストに入ったのか、入らなかったのか？」

## LLM、Runtime、Executor、Shellをどう区別するか

すべてを「Agentが実行している」とひとくくりにしないために、1回のツール呼び出しを大まかに4つの層に分けます。

* LLM：ツールを呼び出すかどうかを決め、ツール呼び出しの引数を生成し、ツールの返り値を読み、回答を生成する。
* Runtime（実行時）：対話管理、ツール登録、コンテキストの組み立て、結果の差し戻しなどのプロセスを管理する。
* Executor（実行層）：実際にコマンドを起動したり関数を呼び出したりし、終了コード、`stdout`、`stderr` を捕捉する。
* Shell/OS：`dir` のようなシステムコマンドを実行する。

この視点で見ると、LLMは自分でWindows上でコマンドを打っているわけではありません。通常は構造化されたツール呼び出しリクエストを出力しているだけです。RuntimeまたはExecutorがそれを受けてコマンドを実行し、何らかの形の結果をモデルに返します。

だから僕がトークンを分析するときは、1つの動作に注目します。「何がLLMのコンテキストに戻されたか」です。

## トークンは実際どこで消費されているか

1回のAgentの作業では、トークンコストは最終回答だけでなく、いろいろな場所から発生します。

僕はこれを、絶えず変化していくコンテキストのパッケージとして理解しています。

この図で、初心者に特に注意してほしい点が2つあります。

1つ目。多くのAgent Runtimeでは、ツール定義とschemaがモデルの可視コンテキストに置かれると、それらもトークンを消費します。あるAgentが多くのツールを公開していて、それぞれが長い説明・引数構造・例・制約を持っている場合、まだ何のコマンドも実行していなくても、モデルの入力はすでに膨らんでいる可能性があります。もちろん、製品によっては必要に応じたロード、圧縮、ルーティング、実装の隠蔽を行うので、あるRuntimeの振る舞いをすべての製品に共通する固定ルールだと考えるべきではありません。

2つ目。ツールの実行結果は、実行後の主要なリスクです。`dir /s /b /a:-d` の2万行のパスがそのまま丸ごと差し戻されると、巨大なコンテキストの負担になります。

つまりトークン制御には2つの端点があります。

* 実行前：無関係なツール・無関係な説明・無関係な履歴をできるだけロードしない。
* 実行後：生の結果をできるだけ差し戻さず、要約・参照・ページング可能なデータを多く返す。

## 最悪の設計：stdoutの全量をそのまま差し戻す

Agentにプロジェクトをスキャンさせるとします。

ツール実行層は2万行のパスを手に入れました。もっとも直接的だけれど、もっとも最悪な設計はこうです。

> 完全なstdoutをツールの返り値として、そのままLLMに渡す。

この設計は一見「情報が完全」に見えますが、エンジニアリング的には非常に危険です。

いくつかの弊害が生まれます。

* トークンコストが急速に上昇する。
* コンテキストウィンドウがパスのリストに占拠される。
* モデルの回答が遅くなる。
* 本当に重要なユーザーの意図・システム上の制約・コードの断片が薄まってしまう。
* Runtimeがツール結果を後続のコンテキストに保持していれば、次のターン以降もこの低価値なノイズを背負い続けることになる。
* モデルが大量のパスの中で迷子になり、かえって判断が難しくなる。

このとき僕は初めて気づきました。Agentのツール結果は、完全であればあるほど良いわけではありません。LLMに返す内容は、次の意思決定に役立つべきものであって、下層の世界をそのままコンテキストに運び込むものではないのです。

## 4つの層：stdout、ツールの返り値、LLMのコンテキスト、外部ストレージ

その後、僕は自分なりの判断基準を1つ決めました。「コマンドが何を出力したか」だけを問うのではなく、「出力がどの層で止まっているか」を問う、というものです。

ここに1つ重要なポイントがあります。この連鎖は、必ずしも全量が貫通しなければならないわけではない、ということです。

Shellは2万行を出力してよく、Executorはそれを丸ごと捕捉してよく、ツール層も完全な結果をファイル、データベース、オブジェクトストレージ、一時的な結果キャッシュに保存してよいのです。そのうえで、要約・件数・サンプル・`result_id` だけをLLMに返せばよいのです。

つまり、完全なデータは存在してもよいけれど、デフォルトでコンテキストの中に存在すべきではない、ということです。

これはAgentエンジニアリングにおいて、とても重要な判断だと思います。

> 外部ストレージは事実を保存できる。コンテキストは、いま必要な意思決定に必要な情報だけを載せるべきである。

## ツール層が先に力仕事をすべき

「どうやってLLMに2万行を全部見せるか」という問いを、「ツール層が先にどんな処理をすべきか」という問いに変えたとたん、設計はずっとクリアになりました。

ファイルスキャンのツールで言えば、ツール層は先にこういったことをできます。

* コマンドを実行するか、ファイルシステムAPIを使ってファイルをスキャンする。
* 結果をストリームで読み込み、一度にすべての出力をメモリに詰め込まない。
* 総数を集計する。
* 拡張子・ディレクトリ・サイズ・更新日時で集約する。
* `node_modules`、`.git`、`dist` などを無視するフィルタ条件を適用する。
* 完全な結果を外部の場所に保存する。
* 短い要約と少量のサンプルを生成する。
* 必要に応じてページングトークンを提供し、後で必要な分だけ展開できるようにする。

こうすれば、LLMは2万行のパスを見なくても、「このプロジェクトはだいたいどんな構造で、ファイルはどこに集中していて、次にどの部分を調べるべきか」を把握できます。

この図の要点は、完全な結果が消えてしまうわけではなく、外部化されているということです。LLMが受け取るのは、それが推論を続けるために必要な最小限の情報です。

## ツールに何を返してほしいか：最小限のプロトコル

もし僕がファイルスキャンツールの返り値のプロトコルを設計するなら、完全なstdoutをそのまま返すことはしません。むしろこんなJSONを返してほしいと思います。

```
{
  "summary": "プロジェクトのファイルをスキャンしました。主なファイルはsrc、tests、docsに集中しており、依存ディレクトリとビルドディレクトリは除外済みです。",
  "count": 1842,
  "sample": [
    "README.md",
    "src/app.ts",
    "src/components/Button.tsx",
    "tests/app.test.ts",
    "docs/architecture.md"
  ],
  "truncated": true,
  "result_id": "scan_20260711_001",
  "next_page_token": "page_2",
  "filters_applied": {
    "exclude_dirs": ["node_modules", ".git", "dist"],
    "files_only": true,
    "recursive": true
  }
}
```

このプロトコルでは、各フィールドに明確な役割があります。

* `summary`：LLMが結果を素早く理解できるようにする。
* `count`：LLMに規模を伝える。
* `sample`：LLMに少量の実際のサンプルを渡し、要約が抽象的になりすぎるのを防ぐ。
* `truncated`：返された内容が完全な結果ではないことを明示する。
* `result_id`：外部に保存された完全な結果を指す。
* `next_page_token`：後でページ単位に展開できるようにする。
* `filters_applied`：今回のスキャンの範囲をLLMに伝え、誤判断を防ぐ。

これは「モデルに2万行のパスを渡す」よりもAgentに適しています。なぜならLLMの仕事はファイルリストのデータベースになることではなく、十分な情報をもとに次の判断を下すことだからです。

後で本当に展開が必要になったら、Agentはもう一度ツールを呼び出せます。

```
get_scan_page(result_id="scan_20260711_001", next_page_token="page_2")
```

あるいは局所的に問い合わせることもできます。

```
query_scan_result(result_id="scan_20260711_001", path_prefix="src/components")
```

こうすることで、コンテキストの増え方は「無差別な流し込み」から「必要に応じた局所展開」に変わります。

## 実行前の制御：無関係なツールとコンテキストをできるだけ露出させない

ここまでは、ツールを実行した後に結果をどう制御するかという話でした。実は、ツールを実行する前の時点で、すでにトークンコストが発生している可能性があります。

僕が以前見落としていたのは、ツールがまだ呼び出されていなくても、ツールの説明とschemaがすでにモデルのコンテキストに入っている可能性がある、という点です。

もしあるAgentが一度にすべてのツールをモデルに露出させていたら——ファイルツール、ブラウザツール、データベースツール、デプロイツール、画像ツール、Notionツール、GitHubツールなどを全部詰め込んでいたら——モデルは毎ターン、必ずしも関係のない大量のツール説明を背負うことになります。

より合理的な実行前の制御には、次のようなものがあります。

* Tool / Skill Lazy Loading：タスクに必要なときだけ、関連するツールの説明をロードする。
* Task Router：先にタスクの種類を判断してから、公開する能力のグループを決める。
* Tool Grouping：ツールをファイル、コード、ブラウザ、データなどの能力ごとにグループ化する。
* Schema Compression：ツールのschemaは明確だけれど簡潔に書き、冗長な例でコンテキストを埋め尽くさない。

僕はこれを、実行前の制御は「モデルがどれだけの能力の説明を見られるか」を制御することだ、と理解しています。

## 実行後の制御：生の結果をできるだけ差し戻さない

実行後の制御が対象にしているのは、「ツールがどれだけの結果を生み出したか、そのうちどれだけがコンテキストに入るか」です。

ここでの手段には、次のようなものがあります。

* `summary`：先に短い要約を返す。
* `count`：規模の情報を返す。
* `sample`：少量の代表的なサンプルを返す。
* pagination：ページ単位で展開する。
* external storage：完全な結果を外部に置く。
* `result_id`：全文の代わりに参照を使う。
* 局所展開：特定のディレクトリ、特定の種類のファイル、特定のログ区間だけを問い合わせる。

この図は、僕にとって実用的な覚え方を1つ作ってくれました。

* 前段の入力面：モデルが最初から無関係なツールや履歴を背負いすぎないようにする。
* 後段の結果面：モデルが実行後に生の出力を飲み込みすぎないようにする。

トークン制御は何か1つの魔法のスイッチではなく、Agentのライフサイクル全体を貫く設計習慣です。

## Subagentは自動的にトークンを節約してくれるわけではない

トークン制御の話をしていると、subagentのことをすぐに思い浮かべます。僕も最初は、「タスクをsubagentに分割しさえすれば、必ずトークンを節約できる」という誤解を持っていました。

でもその後、この言い方は不完全だと気づきました。

もしいわゆるsubagentが、同じ長いコンテキストの中での「ロールプレイ」にすぎない場合——例えば同じ会話の中で「今からスキャン担当を演じて」「次は要約担当を演じて」と言っているだけの場合——それは本当の意味でコンテキストを隔離できていません。すべての履歴、ツール結果、説明は依然として同じコンテキストに積み上がっており、トークンコストが必ずしも下がるわけではありません。

本当にトークン節約に役立つsubagentは、少なくとも3つの条件を満たす必要があります。

* 独立したコンテキスト：各subagentがメインの対話の全履歴を引き継がない。
* 入力の絞り込み：各subagentが、タスクの完了に必要なファイル・要約・パラメータだけを読み込む。
* 成果物の外部化：subagentが完全な中間結果をファイルや外部ストレージに書き出し、要約・パス・`result_id` だけをメインagentに返す。

つまりsubagentの価値は、「subagentという名前がついていること」ではなく、コンテキストの境界を短く、明確にすることにあるのです。

## 低コンテキストなsubagentパイプラインの例

ファイルスキャンと分析のタスクをsubagentパイプラインとして設計するなら、僕はメインagentにスケジューリングだけをさせ、生の結果をすべて自分で飲み込ませることはしません。

ここでの鍵は、メインagentが完全なファイルリストを背負う必要がないことです。スキャンsubagentも、対話全体の履歴を知る必要がありません。分析subagentも、自分が分析すべき局所部分だけを読めばよいのです。

こうすることの利点は、「マルチAgentは高度だ」といった抽象的な話ではなく、とても具体的です。

* 各コンテキストが短くなる。
* 各段階の入力が明確になる。
* 中間成果物が追跡可能・再利用可能になる。
* メインagentの注意力が意思決定と集約に集中する。
* 生データが後続の対話を無意味に汚染しない。

僕が今理解しているsubagentは、単なるタスク分業の方法というより、コンテキストの隔離と中間成果物の管理方法に近いものです。

## `dir /s /b` に戻って：このツールをどう設計するか

もし今、Agentのために「ファイルをスキャンする」ツールを設計するなら、僕はこう設計するのは避けます。

```
run_shell("dir /s /b /a:-d")
```

そしてstdoutを全量返す、というやり方です。

僕はむしろ、意味がもっと明確なツールとして設計したいと思います。

```
scan_files(
  root=".",
  recursive=true,
  files_only=true,
  exclude_dirs=["node_modules", ".git", "dist"],
  max_sample=20
)
```

内部では `cmd.exe` の `dir` を使ってもいいし、PowerShellの `Get-ChildItem` を使ってもいいし、言語標準のファイルシステムAPIを使ってもかまいません。LLMにとってそれは重要ではありません。重要なのは、ツールの返り値のプロトコルが制御可能であることです。

デフォルトで返してほしいのは：

* スキャン範囲。
* 適用したフィルタ条件。
* ファイルの総数。
* ディレクトリごとの分布。
* 拡張子ごとの分布。
* 少量のサンプル。
* 切り詰められているかどうか。
* 完全な結果の `result_id`。
* 次のページのトークン、または局所的な問い合わせ方法。

もしユーザーが後で「`src` 以下のReactコンポーネントを分析して」と聞いてきたら、Agentは `result_id` をもとに `src` の局所的な結果を問い合わせればよく、最初のターンからリポジトリ全体のパスリストをコンテキストに入れる必要はありません。

## この例から学んだ判断のことば

`cmd.exe` の `dir /s /b /a:-d` というこの例を通じて、僕はAgentのトークン制御についていくつかの判断を持つようになりました。

1つ目。コマンドの実行とモデルのコンテキストは別の話である。

Shellの出力はそのままLLMのトークンコストにはなりません。OSコマンドの実行自体には実行コストがありますが、LLMトークンで課金されるわけではありません。出力がシリアライズされてモデルのコンテキストに送られたときに初めて入力トークンコストになり、モデルがツール呼び出しや回答を生成するときに出力トークンコストが発生します。

2つ目。ツールの返り値は1つのエンジニアリング設計であり、コマンド出力の自然な転送ではない。

ツール層は自ら集計・フィルタ・集約・ページング・切り詰め・保存を行うべきであり、生のstdoutをそのままモデルに渡すべきではありません。

3つ目。コンテキストは希少なリソースである。

コンテキストは、いま必要な意思決定に必要な情報だけを保存すべきであり、すべての中間事実を保存すべきではありません。完全な事実は外部ストレージに置き、`result_id`、ファイルパス、ページングトークンで参照すればよいのです。

4つ目。subagentがトークンを節約できるかどうかは、本当にコンテキストを隔離しているかどうかにかかっている。

すべてのagentが同じ長いコンテキストを共有しているなら、それは呼び方を変えただけにすぎません。各agentが独立して絞り込まれた入力を受け取り、成果物を外部化して初めて、メインコンテキストの負担を減らせる可能性があります。

## 結論：優れたAgentは、何をコンテキストに入れないかを決められる

今の僕がAI Agentのツール呼び出しを見るとき、「ツールを呼び出せるかどうか」だけを問うことはありません。こう問い続けます。

* ツールを呼び出す前に、どれだけ無関係なschemaをロードしているか？
* ツールを呼び出した後、どれだけ生の結果を差し戻しているか？
* 要約・件数・サンプル・ページングはあるか？
* 完全な結果を外部化しているか？
* `result_id` を使って後続の局所展開をサポートしているか？
* そのsubagentは本当に隔離されているのか、それとも同じ長いコンテキストの中でのロールプレイにすぎないのか？

これが、`dir /s /b /a:-d` というこの小さな例から僕が学んだ、いちばん大きな変化です。

AI Agentのエンジニアリング能力は、より多くのツールを呼び出せることでも、より多くの情報をモデルに詰め込むことでもありません。多くの場合、より重要な能力とは、どの情報をコンテキストに入れるべきでないかを判断することです。

`stdout`、ツールの返り値、LLMのコンテキスト、外部ストレージをきちんと切り分けられるようになったとき、トークン制御はもう抽象的なコストの問題ではなく、設計し、検証し、繰り返し改善していける1つのエンジニアリングプロトコルになります。
