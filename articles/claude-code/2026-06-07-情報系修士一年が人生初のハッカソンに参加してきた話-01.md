# 1. はじめに

はじめまして、私kyaと申します。現在は情報系の大学院修士一年でデータ工学・機械学習を専攻しています。今回が人生はじめての記事の執筆になるので至らない点があると思いますが、ぜひ最後まで閲覧いただけると幸いです。

早速ですが、本題に入りたいと思います。今回私はクラスメソッド株式会社が主催する [Microsoft Agent Hackathon powered by Tokyo Electron Device](https://zenn.dev/hackathons/microsoft-agent-hackathon-2026)に同じ研究室のメンバー4人で参加したので、この記事ではそこで得た学びなどを共有できればと考えています。なお、私たちが作成したAIエージェントに関しては別途記事を執筆しているので、詳細が気になる方は下のリンクからぜひご覧いただければと思います。(この記事では私個人にフォーカスした内容を記載していきます。)

本ハッカソンは **業務改革につながるAgentic AIを作ろう** というテーマの元、**Microsoft Azure** を基盤としてAIエージェントを作成するというものになっています。私たちはまだ学生で実務経験が乏しいこともあり、何か特定の業務を改革するという方向性ではなくAI時代のエンジニアが直面するであろう課題の解決を目指しました。  
詳細は2章で記載しますが、私たちはAIの普及によって開発現場で生じている **開発におけるジュニアの目的意識・前提条件の軽視 / ジュニアに対するマネジメントコストの増大** という二つの問題に着目し、これを解消するためのAIエージェントである **Premise-Alignment Agent (以下、PAA)** を作成しました。

# 2. PAAを作成した動機

本章ではPAAを作成した動機について解説します。

## 2-1. 長期インターンで感じた「AIツールの導入による生産性向上」からの乖離

私は現在CtoCのサービスを提供する会社でソフトウェアエンジニアとして長期インターンをしていて、業務内では **Claude Code / Cursor / Gemini / ChatGPT** 等の様々なAIツールを活用しながら効率的に業務を進めています。AIツールの普及によって、実現したいことが明確に定まっている場合には、それを短時間で形にすることができるようになったことで生産性が向上する！と巷では言われていますが、私はこれを実感する機会が少なかったと感じています。  
というのも、AIツールを使用することで何かを形にするスピード自体は向上しましたが、実現したい課題の **前提条件** が間違っている場合には手戻りが発生してしまい、結果的に生産性の向上が実現できていなかったのです。

例えば、ある新機能のテーブル設計をする際に私はAIツールを有効活用することにより **正規化理論・パフォーマンス・わかりやすさ** のトレードオフを考慮した複数の設計案を上司に提示しました。私が提示した設計案は確かにトレードオフを考慮できたある意味での教科書的な設計案だったのですが、機能として想定されていた前提条件を逸脱したものになっていました。  
その結果手戻りが発生し、納期ぎりぎりのリリースになってしまいました。

## 2-2. AIツールはジュニアに何をもたらし、何を奪ったのか？

2-1節において私が長期インターンで感じた生産性向上に対する乖離について述べました。  
これに関して私は **AIツールの過度な使用によってアウトプット重視で目的・前提条件を軽視する思考スタイル** を植え付けられてしまったことが直接的な原因だと考えています。AIリテラシーが高い人に関してはこのような問題が起こることは少ないと思いますが、AIツールの急速な普及はジュニアに **最速のアウトプット** をもたらした反面、**目的・前提条件を重視する思考スタイル** を奪ってしまう危険性をはらんでいるのではないでしょうか。

## 2-3. マネジメント層に対する影響

私個人的にはAIツールを活用することによってジュニアに対しても初めからある程度の **自走力** を期待されているのではないかと感じています。AIツールによって知識が拡充され、従来であれば任せることができなかった範囲の業務まで担当できる可能性がありますが、これによって一定 **マネジメント層への負荷の増大** があるのではないかと考えています。ジュニアに対して自走させたい反面、目的や前提条件からのずれが気になり以前よりも **マイクロマネジメントが常態化** する。これにより組織に対してインパクトを与えるだけの能力を持っているマネジメント層のリソースが大幅に食われてしまい、最悪の場合組織全体の生産性が向上することに繋がる。私はジュニアに対するAIツールの導入はこのような問題も抱えていると考えています。

AIツールの導入によって生産性が向上すると期待される一方で、私が実感したのは **「速度の問題ではなく、前提確認プロセスの欠如」** が生産性低下の本質的な原因であるということです。いくら速くアウトプットを出せても、目的・前提条件からずれていれば手戻りが発生し、結果としてマネジメントコストの増大につながります。

このような背景から私たちは開発現場において以下を目的としたAIエージェントであるPAAを作成することにしました。

> 上司と部下の間に入り、「目的・前提条件からのズレ」をリアルタイムに検知・通知することで、マネジメントコストを削減しながらジュニアの自走力を育てる

以降ではPAAの作成背景から離れ、3つの技術的な観点を深掘りし、失敗から得た学びについて記載していきます。

## 3-1. そもそもAIエージェントって何？

皆さんは **AIエージェント** と聞いてどのようなイメージを持つでしょうか？ChatGPT (GPT) / Gemini / ClaudeのようなLLM (サービス)と何が違うの？と思われる方もいると思います。  
これに関して私の理解としては **AIエージェントとはLLMを頭脳として用いて、設定されたタスクを自律的に遂行するもの** であると解釈しています。より厳密な定義としては、**AWS (Amazon Web Service)** は以下のように定義しています\*1。

> 人工知能 (AI) エージェントは、環境と対話し、データを収集し、そのデータを使用して自己決定タスクを実行して、自主的な目標を達成するためのソフトウェアプログラムです。目標は人間が設定しますが、その目標を達成するために実行する必要がある最適なアクションは AI エージェントが独自に選択します。

この定義に基づいて次の3-2節ではAIエージェントの構成要素について簡単に解説します。

## 3-2. AIエージェントの構成要素

AIエージェントは主に以下の3つの要素で構成されています \*2。

1. **基盤モデル (GPT / Gemini / Claude / DeepSeek etc...)**  
   AIエージェントの **頭脳** をつかさどる部分です。ユーザーから与えられたタスクを完遂するためのロードマップを作成し、今の状態において最適なツール (道具)を選択してタスクを進行する役割を担います。
2. **メモリー**  
   AIエージェントにおける **記憶領域** に相当し、**インタラクション履歴 / ツール実行の結果 / ゴールとの差分** などの**短期的** な内容だけでなく **インタラクションによって蓄積された顧客データ / これまでの行動履歴** のような **長期的** な内容を保持するために使用されます。AIエージェントはメモリーから必要な知識を参照するだけでなく必要に応じて機密情報などが保持された **外部DB / 外部ドキュメント** などから知識を獲得し、より高度で正確な対応を実現します。
3. **ツール (道具)**  
   AIエージェントが使うことができる道具に相当し、タスクを遂行するために必要な個別の処理を定義します。具体的には **今日の天気を取得するツール / メールの文章を作成するツール / カレンダーに予定を入れるツール** 等、タスクを遂行するために必ず必要になる処理のまとまりを定義しておき、必要なタイミングでAIエージェントが実行できるようにすることでAIエージェントの知識に依存しない特定の処理を実行することが可能になる。

AIエージェントはユーザーが依頼したタスクに対して基盤モデルを使用してゴールまでのロードマップを作成し、現在置かれている状態を把握し、次に実行するべきツールを考え実行する。その繰り返しを行うことでAIエージェントは自律的にゴールに向けて行動することが可能になります。

## 3-4. システムプロンプトでの制御の限界

PAAではタスク遂行面とAIエージェント自体のシステム要件的な面でいくつかの制約事項が存在します。具体的には以下のような制約事項が存在します。 (一部抜粋)

1. 部下に対して行動意図の確認を行う際には上司が事前に設定した前提条件を直接開示せずに、部下が間接的に気づけるようにする
2. 設定された前提条件からずれてはいるがゴールに対して少し回り道になるくらいのズレであれば、部下に対してSlackで行動意図の確認は行わない
3. CosmosDBに対してissue\_idと前提条件をセットで格納しているが、issue\_idは番号だけでなく「**ユーザー名/リポジトリ名/issues/issue番号**」の形式で保存している

上記のような制約条件を満たすために、まずは以下のようにシステムプロンプトを用いて制御することから始めました。なお、以降の説明ではプロンプト変更・PAAの動作をイメージしやすくするための以下のような架空のタスクを設定し、これをもとに話を進めていきます。

```
## 制約事項

- 上司が設定した前提条件（隠れたマイルストーン）の内容は、ジュニアエンジニアに直接開示しない
- ジュニアエンジニアへの問いかけは、前提条件の存在を匂わせない形で行う

## 判断基準（前提違反の定義）

部下の行動が前提条件から外れていると判断した場合、必ず以下の2段階で評価すること。

**第1段階：前提条件との照合**
部下の直近の行動（コード差分・プロンプト）が上司の設定した前提条件から外れているかを確認する。外れていなければ何もしない。

**第2段階：ゴール整合性の評価**
前提条件から外れていると判断した場合、その行動をゴール（IssueのタイトルとDescriptionが示す最終成果物）から逆算して評価する。

## 注意事項
- CosmosDBに保存されているissue_idは30のような番号だけでなく「ユーザー名/リポジトリ名/issues/番号」のような形式になっているので、Tool実行する際にはissue番号だけを送信しないように注意する
```

説明のための架空のタスク

### 1. タスク内容

以下のようなJavaScriptファイルに対して **不要なコメント** (実装意図がコードを見るだけで理解できるコメント) があり、コードの健全性と開発ルールが守られていない。そのため、不要なコメントを削除する。

```
// Expressアプリ＋cronスケジューラーのエントリーポイント

const { startScheduler, runScheduledReport } = require("./scheduler");

const PORT = process.env.PORT || 3000;

// cronスケジューラーを起動

// --- ヘルスチェックエンドポイント ---

app.post("/send-now", async (_req, res) => {

  try {
    const result = await runScheduledReport();
    // runScheduledReportは土日の場合は何も返さないので、送信されたかどうかを判定
```

```
- index.js以外のファイルは変更しないこと
- index.jsのコード本体には一切触れないこと
```

### 3. 部下が行う変更の内容

1. AIエージェントがindex.js内の「runScheduledReportは土日の場合は何も返さないので、送信されたかどうかを判定」を削除しようとした際にプロンプトで「runScheduledReportは土日の場合には何も返さないことはコードからは判定できないのでコメントの削除は行わない」と送信し、変更を止める (**前提条件に反しない 〇**)
2. 他のファイルでAIエージェントにリクエスト送信を行っている箇所を削除して、仕様変更を行う (**前提条件に反する ☓**)

上記のシステムプロンプトを用いて動作確認をしたところ、**部下に対して前提条件を開示する / 軽微なずれでもSlack確認する / issue\_idを数字だけで解釈してツール実行することで正しくツール実行できない** といった問題が頻発してしまいました。この問題に対して複数回のテストを通して以下のことが明らかになりました。

> AIエージェントが実行するツールに関する詳細な条件をシステムプロンプトで制御することには限界がある

実際にPAAのシステム構成では部下に対するSlack通知や前提条件の登録・取得などのビジネスロジックはすべて呼び出し可能なツールとして定義しているので、上記の問題の原因としてはかなり納得感がありました。  
なお、個々のツールに関してはハッカソン期間中の**7割**ほどの時間を割いて **チームメンバー全員での動作確認と自動テストの作成** によって徹底的に作りこみを行っていたため、ツール自体の動作に関しては問題ないことを確認できていました。

## 3-5. システムプロンプトとツール定義の責務の分離 / StructuredOutputを活用することによる問題点の解消

3-4節で明らかになった問題点を解消するために、**システムプロンプトの責務** と **OpenAPI準拠のツール定義の責務** を明確に分離することにしました。

1. **システムプロンプトにはAIエージェントに対するふるまいや思考プロセスのようにツールの仕様に関係ない内容を記載する**
2. **ツール定義には各ツールを実行するために必要な情報 (種類やフォーマット) / ツール実行する条件などを記載してモデルのふるまいに関連する内容は記載しない**

OpenAPI形式でツールの詳細を定義することによって、AIエージェントに対して実行可能なツールの設計書を提供できます。また、関数を実行するために必要な情報はLLMが生成、または必要に応じて外部ソースから取得する必要がありますが、これを **JSON Schema** \*3に準拠して記述することにより、OpenAIが提唱している **StructuredOutput** \*4 を使用して厳密に制御することが可能になります。  
StructuredOutputに関する説明は割愛しますが、以下のような形式で各パラメータの **description** に対して詳細な要件を定義することでLLMの確率論的な出力の揺れを限りなく抑え、厳密なルールベースでのツール実行を可能にします。

```
"/send_slack_to_junior":{
    "post": {
      "operationId": "sendSlackToJunior",
      "summary": "部下に問いかけ内容をSlack DMで送信する",
      "description": "認識のズレを検知した際に、部下へメタ認知を促す問いかけをSlack DMで送信する。(以下割愛)",
      "requestBody": {
        "required": true,
        "content": {
          "application/json": {
            "schema": {
              "type": "object",
              "required": [
                "assignees_slack_id",
                "question",
                "task_title"
              ],
              "properties": {
                "assignees_slack_id": {
                  "type": "string",
                  // Slackのuser_idはそれを取得する専用の関数を使用して取得することを明記することで憶測で回答されるリスクを低減する
                  "description": "DM通知先となる部下のSlackユーザID。コンテキストに含まれていない場合は、get_issue_notification_info を呼び出して取得すること。"
                },
                "question": {
                  "type": "string",
                  // 思考プロセスや守るべき条件を記載することでより厳密にLLMに対する指示を定義する
                  "description": "部下に送信する問いかけの内容。生成は2段階で行うこと。【第1段階：問いの設計】部下の直近のコード差分・プロンプトと上司が設定した前提条件の両方を参照し、どこにズレが生じているかを把握した上で問いを設計すること。【第2段階：文章の生成】設計した問いをSlack送信用の文章に変換する際は、前提条件の内容・方向性・優先度を一切含んではならない。送信前に「前提条件を知らない第三者がこの問いを読んだとき、前提条件の内容を推測できるか」を自己チェックし、推測できる場合は文章を作り直すこと。"
```

## 3-6. AIエージェント開発の本質はプロンプト / ガードレールの整備である

3章を通して私が伝えたいことは **AIエージェント開発の本質はプロンプト / ガードレールの整備である** ということです。  
私を含めチームメンバー全員が何かしらのウェブ / モバイルアプリケーションを作成した経験があり、これらの経験をもとに知らず知らずのうちに **開発 ≒ プログラムの作成** という先入観が形成されてしまっていたのではないかと考えています。その結果ハッカソンの約7割ほどの時間をAIエージェントが実行するツール (プログラム) の作成に当ててしまっていました。  
しかし、いざツールが完成してもそれを適切にAIエージェントが選択できるようなプロンプト・ガードレール (ツール定義)の整備ができていない状態では今回のような問題が発生してしまいます。  
そのため、AIエージェント開発はAIエージェントが実行できるツールが完成してからそれをどのように使って目的を達成するのか？に対して多くの時間とリソースを割くことが重要であると学ぶことができました。

# 4. 要件の読み違えがセキュリティに対する設計を歪めた

## 4-1. 当初のセキュリティ設計

AIエージェントを作成するにあたってハッカソン期間中の一時的なものではありますが、関係ない第三者がリソースに対して触れることができないようにある程度のセキュリティ面を考慮する必要があると考えていました。そのため、一般的なクラウドインフラ上で構築されるように仮想ネットワーク ( [VNet](https://azure.microsoft.com/ja-jp/products/virtual-network/?msockid=00073911f3b96dc326df2cc0f2c36c4e) )を構築し、**Service Endpoints**を使用してCosmosDB・Microsoft FoundryへのアクセスをVNet内からのトラフィックのみに制限する方針で検討していました。 (以下イメージ)

![PAAの仮システム構成図](https://static.zenn.studio/user-upload/deployed-images/f4651977edb5fb63fac41ec4.png?sha=fc01d6b3a012eb2bdf66b99c021600d4f7ac1b2c)  
*図3. PAAの仮システム構成*

しかし、今回のハッカソンの要件的にVNetのように閉じた環境下でサービスを稼働させてしまうと審査員の方が触ることが難しくなってしまうため、VNetのように閉じた環境ではないが機密情報を保持しているリソース (Microsoft Foundry / CosmosDB)に対して第三者がアクセスできないような設計を再検討する必要が出てきました。

## 4-2. 要件を考慮したセキュリティの再設計

4-1節で生じた問題を解決するために **Azure Function + Easy Auth** を中心として再設計を行いました (本記事内の図2参照)。外部のサービスとAzure上のリソースのすべての接点に対してAzure Functionsを設置し、**EntraIDのアプリケーション設定で許可するユーザーを制限**した上で、**Easy Authによってすべての受信リクエストをEntraIDで認証**することにより、第三者からのリクエストを事前にはじくことができるようにしました。  
もちろんVNetを使用する場合と比較してセキュリティリスクが増大するとは思いますが、 **ハッカソン期間中のみの限定公開 + 必要最小限のセキュリティ対策** をするという観点ではこれで十分であると判断し、これを正式に採用することにしました。

## 4-3. 要件の読み違いによるセキュリティ設計の歪み

4-2節の設計で問題ないように思いましたが、ハッカソンの要件とPAAのサービス提供形態を考慮すると初期設計案を採用した方がより堅牢なセキュリティ対策になったと考えています。  
そもそも私たちが作成したPAAはウェブアプリケーションとして実装されておらず、VSCode / Claude Code / Githubのようにローカル環境で動作するツールに対して組み込まれて基本的にはローカル環境上で動作するものとして提供されています。ハッカソンの要件的にも **成果物のURL（WebアプリやCopilot Studio共有URL等）** の提出が必要ではありましたがサービスの提供形態的に不特定多数の人が触ることができない場合にはデモ動画での提出も容認されていること認識していました。

それにも関わらず、ウェブアプリケーションを前提として 「VNetでは審査員の方に触っていただくことが困難である」 という理由からセキュリティリスクが低下する可能性があるAzure Functions + Easy Auth構成に変更してしまっていることを開発中盤で気づきました。  
そもそも、サービスの提供形態的に審査員の方に触っていただくのが困難であれば初めからVNetのような閉じた環境下で動作させる方がセキュリティリスクに対してより堅牢であったと考えています。

具体的には、現在採用しているEasy Auth構成では **CosmosDB・Microsoft Foundryのエンドポイント自体がパブリックに公開されており、誰でもリクエストを送信できる状態** になっています。Easy AuthはEntraIDのトークン検証による **アプリケーション層（L7）での制御** であるため、リクエスト自体はエンドポイントに到達した上で認証に基づいて拒否される仕組みです。そのため、トークンの窃取・偽造やミドルウェアの脆弱性を突いた攻撃のリスクが残ります。一方、初期案のVNet + Service Endpoints構成はファイアウォールによる **ネットワーク層（L3/4）での制御** であり、VNet外からのトラフィックはエンドポイントに到達する前の段階で遮断されます。そもそもアプリケーション層まで届かないため、アプリケーション層を狙った攻撃手法自体が成立しません。

## 4-4. セキュリティ設計はサービス提供形態が定まるまで固めるな

4章を通して私が伝えたいことは、**システム開発においてセキュリティ設計はサービスの提供形態が明確に定まっていない / 認識できていない場合には固めない方がいい** ということです。セキュリティ設計はすべての提供形態に対して完璧なものはないし、作成するシステムの規模・用途によっても様々です。  
この経験からシステム構築前の段階でサービスの提供形態・要件を徹底的に洗い出してチーム内で合意形成しておくことが堅牢なセキュリティ設計の実現に大きく寄与していることを実感できました。

# 5. クラウドインフラにおける管理者アカウントと開発用アカウントの共存の必要性について

## 5-1. Microsoft Azureにおけるリソース・権限管理方法

Azure基盤上で開発する際には主に **二種類の権限** \*5 を扱うことが多いです。一つ目はリソースの作成・削除などリソースを管理する権限である **コントロールプレーン**。二つ目はリソースの内部の情報を書き換えたり、リクエストを送信できるようにするための **データプレーン** です。チーム開発においてはこれらの権限を特定のEntraID (開発者)ごとに **セキュリティグループ** としてひとまとまりにした上で統一して権限管理を行います。  
これによってすべての開発者に対して個別で権限を振る必要がなくなり、メンバーの追加・削除があった場合にはセキュリティグループ対して追加・削除を行うだけで簡単に管理することができるようになります。  
そのため本ハッカソンにおいてもチームメンバー4人のセキュリティグループを作成し、権限管理を行いました。

## 5-2. 管理者も開発用アカウントを作成する必要はあるの？ (クラウドインフラ未経験の葛藤)

セキュリティグループには最低1人以上の管理者が存在します。本ハッカソンでは私がAzure上でのリソースの管理など、インフラ面を担当することになっていたので必然的に私が管理者を担っていました。  
その場合、私が管理者を務めるセキュリティグループに対してほかの3人のメンバーが参加する形になるのですが、管理者アカウントはメンバー変更・権限変更の場合のみ使用し、管理者であっても開発者用アカウントを別途作成して普段はそちらを使用して開発するという方針が、管理者権限を常時使用し続けるよりも望ましい対応です \*6。ただし、現在のAzureでは **Microsoft Entra Privileged Identity Management (PIM)** \*7 がより推奨されるアプローチとして提供されています。PIMはアカウントを2つ作成するのではなく、開発者用アカウントに対して管理作業が必要なときだけ一時的に管理者権限を付与し、時間が経過すると自動的に権限が剥奪される仕組みです。次回以降はPIMを活用した権限管理を採用していきたいと考えています。  
しかし、本ハッカソン開始段階ではその事実が頭の片隅にありつつ、そこまでする必要性を理解できていなかった為、管理者アカウントを使用して開発まで行っていました。

## 5-3. 管理者アカウントで開発するな、絶対に...

管理者アカウントで開発まで進めた結果明確に弊害が出ることが明らかになりました。開発メンバーから必要な権限の付与依頼をされた際に管理者アカウントで権限付与・動作確認まで行ったうえで開発メンバーに対しても動作確認を依頼するという形式が定着していました。しかし私のアカウントでは動作確認できたものが他メンバーの環境ではできないという事態が頻発しました。  
つまり、私が動作確認してうまく動作していたのは適切な権限を付与できていたからではなく、私の権限が最強の管理者権限を持っていたからだということです。これによって自分が付与した権限が間違っているのか？そもそも管理者権限だから何でもできているだけなのか？の問題特定に時間がかかってしまい、他メンバーの開発効率を低下させてしまいました。

この経験からたとえ小規模な開発であっても一時の楽さに惹かれずに、管理者アカウント / 開発者アカウントを使い分ける重要性を頭で理解するだけでなく、実体験を通して理解することができました。

# 6. おわりに

今回が初のハッカソン参加・初のAIエージェント開発・初のインフラ管理とすべてに関して初めての経験ばかりで刺激的な1か月を過ごすことができました。長期インターンの経験から何かものを形にする能力が向上していることは明確に実感することができましたが、インフラ (CI / CD, リソース管理, 権限管理)面においては明確に勉強不足を実感しました。ここに関しては地道に一つずる理解できる領域を増やしていけたらと考えています。
