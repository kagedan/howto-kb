## はじめに

Claude Codeに`/goal`というSkillがあります。事前設定したゴールを満たすまでエージェントが作業を続けるSkillです。今回、`/goal`を使った開発ワークフローの調査、実験をしました。その中で調査、アプリケーションの仕様作成、実装、テストなど`/goal`を使ってワークフローが成り立つ箇所、そうでない箇所が明らかになりました。

## 作ったもの

私が主催しているコミュニティイベント「裏テック無尽」での発表内容に関連したデモアプリです。

<https://ft8.numa08.dev/>

<https://github.com/numa08/try-ft8>

アマチュア無線の世界で利用されるプロトコル**FT8**の仕様解説をするので、デモとしてスマホ上でFT8の音声信号のエンコード・デコードを試すことができます。

## goal による自走実装のためにやったこと

Anthropicのドキュメントで`/goal`は以下のように説明されています。

> /goal コマンドはコンプリーション条件を設定し、Claude はあなたが各ステップをプロンプトすることなく、その条件に向かって動作し続けます。各ターンの後、小さく高速なモデルが条件が成立しているかどうかをチェックします。成立していない場合、Claude はあなたに制御を返す代わりに別のターンを開始します。条件が満たされると、ゴールは自動的にクリアされます。

プログラミング言語で言うところの`for`や`while`のような制御構文的なふるまいをエージェントにさせられますね。ドキュメント内の他の文章から**コンプリーション条件**が大切そうだと読み解きました。実際、ドキュメント内で以下の例が評価器として挙げられています。

> 1 つの測定可能な終了状態：テスト結果、ビルド終了コード、ファイル数、空のキュー  
> 述べられたチェック：Claude がそれをどのように証明すべきか、例えば「npm test が 0 で終了する」または「git status がクリーンである」  
> 重要な制約：その過程で変更されてはならないもの、例えば「他のテスト ファイルは変更されない」

このうち「重要な制約」はゴールをずらさないための制約ですね。また終了判定は軽量モデルで実行されますからシンプルなものが求められます。例にある`npm test が 0 で終了する`、`git status がクリーン`はまさしくシンプルですしさらに決定論的です。以上を踏まえてまずデモ実装作業ワークフローを整理、`/goal`を使って自走させるかどうか検討します。

### FT8仕様調査

デモで利用したいFT8の仕様について調査し理解します。実際のシステム開発では「外部API調査」とか「連結システム調査」と呼ばれる箇所でしょう。作るシステムの仕様は変えられるが、こちらは変えられないし合わせる必要があります。とは言え調査と言っても仕様ドキュメントをフォルダに入れてClaude Codeに読み込ませただけですが。

### アプリケーションの仕様決め

別で調査していたタスクがあったので今回はClaude DesktopでやりましたがClaude Codeでも問題ないでしょう。

最終成果物はSPEC.mdです。そしてこのファイルが実装時に`/goal`で終了判定に使う受入テストを作成するデータソースとなります。受入基準はEARS記法で、機械的に検証可能な粒度で書かせました。

EARS(Easy Approach to Requirements Syntax)は、要件を数パターンの定型文に落とし込む書き方です。たとえば「いつ〜したとき、システムは〜しなければならない」「もし〜したら、システムは〜しなければならない」といった型に当てはめていきます。曖昧さが減り、1つの受入基準が1つの検証可能なふるまいに対応するので、後段のテストと1対1で結びつけやすくなります。実際、SPEC.mdの受入基準はこんな調子です。

> AC-6 いつ ユーザーが運用中に名前を変更したとき、システムは 3 文字ルールで検証し、以降の送信で新しい名前を用いなければならない。

あわせて非目標(Non-Goals)も明記させ、エージェントが「推論・作りすぎ」をしないようにしました。「本物のコールサイン対応はしない」「名前衝突の自動回避はしない(その場で手動修正)」といった具合に、やらないことを先に潰しておくイメージです。

### 受入テストの作成と凍結

SPEC.mdをもとに受入テストコードを実装させます。しかし今回はFT8という外部APIとも言うべき機能を使うことから「すべての機能をコードで完璧にテスト」できません。たとえばスピーカーとマイクを実際の音で結合させる交信、マイク許可ダイアログの目視、収録した音声を保存・送信していないことの担保などは、実機や人の目でしか確認できません。そこで少なくとも「自動テストできる物・できない物」に分類させ、可能な物からテストの実装をするアプローチを取りました。この「自動化できない部分」が、あとで効いてきます。

テストが完成したら実行し失敗することを確認します。ここで作成した受入テストの通過が`/goal`の完了条件です。

## /goalで自走させる

用意が整ったら`/goal`を使います。今回は以下のプロンプトを使いました。

```
/goal SPEC.md の全受入基準(AC-1〜AC-32)に対応するテストが pnpm test で exit 0、
lint・型チェックがクリーン、Playwright で AC-24 の起動〜メイン画面描画が通ること。
SPEC.md と test/ 以外の既存テストは変更しない。Non-Goals に記載の項目には着手しない。
```

終了条件「pnpm test で exit 0, lint・型チェックがクリーン、Playwright でE2Eテスト」ができること、制約「SPEC.mdとtest/ 以外のテストの変更。Non-Goals に記載された項目の着手」の禁止。これらを明記して自走させます。

## 完成品を動かして分かったこと

今回、トークンリミットやターン数に関する制約は付与しなかったため完成するまで動き続けました。しかしスマホ2台で交信を実験してみると、テキストの音声信号化はできましたが、音声信号のテキスト化に失敗しました。どうやら片方のスマホの時間設定はNTPが返す値に非常に近い正確な時間でしたが一方で片方は正確な時間から1.7秒進んでいました。FT8には「15秒おきに送信と受信を繰り返す」仕様があります。したがって交信する端末同士が同じ時間を刻んでいないと正しく送受信できません。ちなみに実際の無線運用ではGPSを用いた時刻補正などをします。

問題の解決方法としてHTTPのDateヘッダーに含まれる秒にあわせて送受信タイミングをオフセットする方法を採用しました。調査の上でSPEC.mdを更新し(このとき時刻同期まわりの受入基準が増えたので、`/goal`の完了条件で数える受入基準の範囲も広げています)、テストを用意し、そして再び `/goal`による実装の自走を実施しました。ここで大事なのは、**問題を人間が切り分けて仕様に書き戻し、そのうえでもう一度自走させた**という流れです。自走ループそのものは時刻同期の不具合を直してくれません。

## /goalによる自走で得た学び

今回FT8と言う「外部システム」あるいは「物理現象」を取り扱ったことで一連のワークフローを実現できました。

ここで冒頭に書いた「`/goal`でワークフローが成り立つ箇所、そうでない箇所」を整理しておきます。

| 工程 | /goalで自走できたか | 理由 |
| --- | --- | --- |
| 実装(受入テストを緑にする) | ◎ 自走できた | `pnpm test`の終了コードで決定論的に判定できる |
| 受入テストの実装 | ○ おおむね自走 | ただし「自動化できる範囲」の見極めは人が判断 |
| 仕様(SPEC.md)の決定 | △ 人が主導 | 何を作る/作らないかは人が決める。EARS化の作業自体は補助できる |
| 実機で初めて分かる問題の切り分けと設計変更 | ✗ 人が必要 | 時刻同期の不具合のように、動かして初めて分かる |

自走ループの内側、つまり「決定論的に緑/赤を判定できる作業」は`/goal`が非常に強い一方で、その外側、つまり「何を正解とするかを決める作業」は人が担う、というのが今回の実感です。先ほどの時刻同期の問題は、まさにこの"外側"に落ちた出来事でした。この内と外の切り分けを、ワークフロー全体の図にするとこうなります。

「実装の自走」とか「自律実装」とかのワードがもてはやされますが、実際に効いてくるのは「決定論的なゴール条件」と「それを作るための明確な仕様」だと強く感じました。昔から大切だったやつですね。特に今回は仕様をEARS記法で書きました。だったら「EARS記法になっているか、いないか？」をゴール条件にして、調査や仕様作成のフェーズも自走化する、というのも面白そうです。
