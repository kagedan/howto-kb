> 「AIはコードを書く。人間はコードを読む。」

私はソフトウェアエンジニアとして15年のキャリアがある。コードを書くことが仕事だと思っていた。  
今は違う。コードを書くのはAIで、私は仕様を書いている。  
今、私が書くコードは体感で全体の1%に満たない。業務もプライベートも関係なく、コードを書くのはAIだ。

残業時間は大幅に削減され、生産性は飛躍的に向上した。  
これは私に起きた変化の記録だ。

---

## TL;DR

* **仕様書を書く** — ClaudeにはSpec.mdを渡す。あいまいな指示はあいまいなコードを生む
* **AIが書いたコードを読む** — 全承認は品質の低下。コードを読む力が今一番大事なスキルである
* **わからない領域はClaudeに聞きながら広げる** — T型人材への横展開コストをAIが下げてくれる

## きっかけ：2025年9月、VSCodeに拡張機能が来た

転機は2025年9月だった。

AnthropicがClaude Code 2.0を発表し、**VSCode拡張機能**が本格化した。6月にマーケットプレイスへの公開は始まっていたが、2.0でサイドバーパネルやインライン差分表示が加わり、IDEに深く統合された。私はすぐに試した。

ただし当時は**業務での利用ができなかった**。個人のOSS活動の中でひっそりと使い始めた。

感触はあった。「これは仕事を変える」という確信もあった。ただひたすら、使い続けた。

2025年9月29日、**Claude Sonnet 4.5**がリリースされた。

このタイミングで、実用レベルで使える手応えを初めて感じた。コードの品質、文脈理解の精度、指示への追従性。どれも一段上がった感覚があった。「これは本物だ」と思った瞬間だった。

そして2026年2月17日、**Claude Sonnet 4.6**がリリースされる。コーディング・エージェント計画・長文脈推論すべてが底上げされたフルアップグレードだ。Claude Codeユーザーの約70%がSonnet 4.5よりSonnet 4.6を好んだという。その感覚は正直だと思う。私も同じだった。

2026年初頭、業務での利用が許可された。

ここからすべてが変わった。個人のOSS活動でこっそり使っていたものが、業務の中心に来た。  
残業時間は大幅に削減され、以前では考えられなかったペースで実装が進むようになった。

数字の変化よりも、**時間の使い方**が変わったことのほうが大きかった。

## 仕様駆動開発：人間が担う「上流」

私の今の仕事はこうだ。

1. 課題をヒアリングする
2. 機能要件に落とし込む、タスク一覧表を作成する
3. **Spec.md（マークダウンの仕様書）を書く**
4. その仕様書をClaude Codeに渡す
5. Claude Codeがコードを書く
6. テスト設計をする
7. テストをClaude Codeに書かせる
8. 内容をチェックする
9. PRを作成する。pushする
10. Claude Code Reviewでレビューする。クリティカルが出なくなるまで回す。
11. マージする

コーディングという作業は、ほぼClaudeがやる。私は**仕様を定義する人間**になった。

ただし、「人間がやる」と「人間が決める」は違う。アーキテクチャ設計やテスト設計も、Claudeに選択肢とトレードオフを出させ、人間がそれを評価・決定するという形で進めることが多い。  
ゼロから考えるより、Claudeの出力を批判的にジャッジするほうが、スループットも質も上がる。  
最終的な判断と責任を人間が持つ、というのが正確な分業だ。

上流工程の質が、そのまま成果物の質になる。曖昧な仕様を渡せば曖昧なコードが返ってくる。  
クライアントの言葉の裏にある本当の課題を見抜き、それを正確な言語で仕様書に落とす。  
そこで活きるのは、**課題の構造化能力とドメイン知識**だ。この2つがSpec.mdの質を決め、成果物の質を決める。大切なのは目利き力だ。

## 失敗と仕組み化：AIは「終わりました」と言う

正直に言う。困ったことがある。

**AIはエラーを握り潰す。**

こんなことがあった。Claude Codeに実装を依頼し、「完了しました」とコードが返ってきた。  
成果物のコードを確認すると、`try-except`の中が`print`一行で終わっていた。例外をキャッチして、コンソールに出力して、そのまま正常終了。フロントには何も伝わらない。  
エラーは「なかったこと」になっていた。

AIは嘘をつかない。ただ、聞かれていないことは言わない。エラーハンドリングが甘くても、「実装した」という事実に嘘はないから「完了しました」と報告してくる。これが一番厄介だった。

対策として取り入れたのが、**CLAUDE.mdにハーネスの仕組みを定義すること**だ。

```
tasks/lessons.md に失敗を記録させる
```

Claudeが同じミスをしたとき、`lessons.md`に学習内容を書かせる。次のセッションでその記録を参照させることで、**同じミスを繰り返さない仕組み**を作った。

AIの失敗を、チームの学習資産に変える。これは人間のチームマネジメントと同じ発想だ。

CLAUDE.mdの設計については、Claude Code生みの親Boris Chernyの知見が参考になる。  
<https://zenn.dev/acntechjp/articles/5409d8e2ad0767>

ChernyはほぼすべてのセッションをまずPlan modeで開始し、プランが納得できるまでClaudeと往復した後、auto-accept edits modeに切り替えるという手法を実践していた。  
Plan modeでの合意が成功率を2〜3倍に引き上げると本人が述べている点は、Spec.mdを渡してから実装させるという私のワークフローと本質的に同じ発想だ。  
なお、モデルの信頼性向上とともにCherny自身のスタンスは変化しており、2026年時点では「まずauto modeを使え」が最大のTipsと語っている。

また、コードレビューには**Claude Code Review**（[anthropics/code-review](https://github.com/anthropics/claude-code/tree/main/plugins/code-review)）を活用している。  
PRのレビューをAIが担うことで、実装速度が上がってもレビューがボトルネックにならない仕組みを作った。テスト実装もAIに任せ、テスト設計だけ人間が行う。ボトルネックが移動するたびに、AIで吸収していった。

<https://zenn.dev/shintaroamaike/articles/86187d64045449>

### AI駆動開発におけるテスト設計の例

![](https://static.zenn.studio/user-upload/540fa95f0dff-20260626.png)

特にミューテーションテストは効いている実感があった。

AIは既存コードに対してテストを書くとき、自身が書いた実装をなぞるだけのテストを生成しがちだ。テストが実装と同じ誤りを抱えていても、両方がAIの産物であれば検査が通ってしまう。  
テスト設計を人間が先に定義しておくのは、この「AIが自分自身を採点する」状況を防ぐためだ。  
ミューテーションテストはその品質を事後的に検証する手段として機能する。  
無意味なテストを減らすことで認知負荷を下げる効果もある。

<https://mutmut.readthedocs.io/en/latest/index.html>

### mutmut の使い方（公式ドキュメント準拠）

**インストール**

**最小構成で実行**

```
# testsまたはtestフォルダを自動検出して実行
mutmut run

# ソースとテストを明示指定する場合
mutmut run --paths-to-mutate "src/" --tests-dir "tests/"

# 特定モジュールのみ
mutmut run "my_module*"
mutmut run "my_module.my_function*"
```

**結果確認**

```
# 生き残ったミュータントの一覧
mutmut results

# 特定ミュータントの差分を確認
mutmut show 3

# インタラクティブTUIで閲覧（推奨）
mutmut browse
# TUI内: f=関数単位で再テスト、m=モジュール単位で再テスト

# HTMLレポートを生成
mutmut html
```

**CI（GitHub Actions）への組み込み**  
CI/CDに組み込むことを思いつくが、テストの仕様上かなり重いため  
導入は慎重になるべきである。

```
- name: Run mutation testing
  run: |
    pip install mutmut
    mutmut run --paths-to-mutate src/
    mutmut results
```

自分だけ生産性が上がっても意味がない。

社内で積極的に発信した。`CLAUDE.md`の定義、`DESIGN.md`の書き方、実際のワークフロー。  
「こう使えばうまくいく」を言語化して共有した。

結果として、**チームメンバーが自走できるようになった事例も生まれた。** AI駆動開発は個人の武器にとどまらず、チームの開発体制そのものを変えつつある。

## 誤解：「AIがあれば誰でもアプリが作れる」の危うさ

ここで正直に言っておきたいことがある。

確かに今は、ソフトウェアエンジニアリングの経験がない人でもAIを使って「動くアプリ」を作れる時代だ。でも、見た目が動いていることと、品質が高いことは別の話だ。

セキュリティの問題、エラーハンドリングの漏れ、スケーラビリティへの配慮。これらは動作確認では見えない。**経験者がAIを使うからこそ、生産性が本当の意味で上がる。**

AIは道具であって、判断する主体ではない。

---

## 結論：AIが書いたコードが、読めますか？

最後に一番大事なことを言う。

AI駆動開発で一番多い失敗は、**全承認**だ。

Claudeが生成したコードを読まずに承認し続ける。何を作っているかを理解しないまま進める。  
これでは品質は担保できない。最終的な成果物の責任は、AIではなく人間にある。

私が培ったスキルのうち、今一番効いていると感じるのは**コードを読む力**だ。

生成されたコードが何をしているか。どこに問題があるか。仕様通りに動いているか。この判断は、経験なしには難しい。

プロンプトエンジニアリング、コンテキストエンジニアリング——AIの使い方を表す言葉はいろいろ生まれてきた。だが今求められているのはもっとシンプルなスキルだと思う。

AIをディレクションし、その成果物を判断する力。  
AIは優秀な部下だ。指示が明確なら期待以上の仕事をする。でも最終的に「これでいい」と判断するのは人間でなければならない。AI駆動開発における人間の役割は、オーケストレーターであり、審判者だ。

そしてこの「審判者」には、幅広い知識が必要になる。

フレームワークは日々変わり、インフラも変わる。一つの領域の専門性だけでは、AIの成果物を正しく判断できない場面が増えてくる。AIが答えを出してくれるからこそ、その答えが正しいかを判断できる人間でなければならない。日々キャッチアップを怠らず、技術を磨き続ける姿勢が重要だ。

**「学ばなくなった瞬間、あなたはAIに代替される。」**

これからはT型人材として、プロフェッショナルな専門領域を持ちながら、インフラもセキュリティもデータベースも「やってみる」幅広さが求められると強く思う。

AIはその横展開のコストを劇的に下げてくれた。新しい領域を学ぶとき、AIは最高の入門書であり、壁打ち相手でもある。

**「わからないことは、Claude Codeに聞け。これでいい。」**  
AIはあなたの味方だ。ペアプログラミングの相手として、いつでもそこにいる。

インフラ設計に自信がなくても、調べながら進めれば十分だ。  
最初はみな、何も知らなかった。

AIに頼りすぎるとコードを読む力が育たないのでは、という不安はもっともだ。  
私の考えはこうだ。AIが生成したコードを一行ずつ読んで「なぜこう書いたか」を問い続けること自体が、学習になる。

かつては参考書を読み、Googleで検索するのが主流だった。今はAIに聞いて壁打ちし、その結果で実装するほうがよほど近道だ。動いているコードを読みながらAIに質問し続けることで、実践的な力がつく。

AIは答えを出す機械ではなく、問いに付き合ってくれる相手だ。  
ただし、土台のない状態でAIに頼るのは危険だ。基礎なき者にAIは「それらしい嘘」を教える。  
書籍を読み、基礎を固めたうえでAIを使う。その順番は変わらない。  
考えることを手放したとき、あなたはただのAIの承認者になる。

---

AIは確実に、ソフトウェアエンジニアリングを変えている。コードを書く時間は劇的に減った。でも、**考える時間は増えた。**

日本のIT人材不足を解決する鍵は、一人ひとりがAIをレバレッジとして使い、個人の生産性を高めることだと思っている。

エンジニアとしての価値は、コードを書けることではなくなってきた。  
**クライアントの課題を定義し、AIの成果物を判断する。それがこれからのエンジニアの仕事だ。**  
AIは手段に過ぎない。何を作るべきかを決めるのは、いつも人間だ。
