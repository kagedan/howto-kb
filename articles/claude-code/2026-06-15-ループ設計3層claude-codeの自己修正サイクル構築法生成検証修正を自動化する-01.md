> Claude Codeに「テストが通るまで直して」と頼んだのに、1回直して終わり。同じエラーが残ったまま「完了しました」と返ってきた経験はありませんか？

**この記事でわかること:** Claude Codeに自己修正ループ（生成→検証→修正の反復サイクル）を設計する3つの層——CLAUDE.mdでのルール定義、Hooksでの自動検証、Skillsでのワークフロー化——の具体的な構築手順がわかります。

このnoteでは、非エンジニアでも再現できるAI活用法を実測データつきで解説しています。Claude Codeを使っている方で、「AIが1回で完璧なコードを出してくれない」と感じている方向けの記事です。エンジニアでなくても「AIに繰り返し修正させる仕組み」を知っておくと、バイブコーディングの成功率が上がります。

## Claude Codeの自己修正ループとは

Claude Codeは基本的に「指示→実行→完了」の1方向で動きます。しかし、現実のコーディングは「書く→テスト→失敗→直す→テスト→成功」の繰り返しです。

この「生成→検証→修正」のサイクルをClaude Codeに組み込む仕組みが**自己修正ループ**です。

私は記事制作パイプラインとSaaS開発を並行していますが、自己修正ループを導入する前と後で、Claude Codeの出力品質が大きく変わりました。

この違いは、Claude Codeに「何を検証するか」「失敗時にどうするか」を明示的に設計しているかどうかで決まります。

Claude Codeの公式ドキュメント（[Best Practices](https://docs.anthropic.com/en/docs/claude-code/best-practices)）でも、最も重要なプラクティスの1つとして\*\*「Give Claude a way to verify its work」\*\*（Claudeに作業を検証する手段を与えよ）が挙げられています。

## Claude Codeの自己修正ループ構築手順

自己修正ループは3つの層で構成します。下の層ほど自動化の度合いが高くなります。

* **第1層:** [**CLAUDE.md**](http://CLAUDE.md)**（ルール層）** — 「検証してから完了しろ」をルールとして書く
* **第2層: Hooks（自動検証層）** — ファイル編集のたびにテストやリントを自動実行する
* **第3層: Skills（ワークフロー層）** — 生成→検証→修正の手順をスキルとして定義する

1つだけでも効果がありますが、3層を組み合わせると「Claude Codeが自力で品質を担保する」状態に近づきます。

### Step 1: CLAUDE.mdに検証ルールを書く（第1層）

最もシンプルで、最も効果が大きい施策です。CLAUDE.mdに「検証してから完了しろ」というルールを書くだけで、Claude Codeの自己修正率が上がります。

**なぜ効くのか:** Claude Codeはセッション開始時にCLAUDE.mdを読み込み、全ての行動の判断基準にします。「テストを実行してから完了する」と書いてあれば、Claude Codeは自分の出力をテストで検証し、失敗すれば修正を試みます。

以下は、私が実際に使っている検証ルールの構造です。

```
# 検証ルール

## コード変更時の必須手順
- コード変更後は必ず `npm test` を実行する
- テストが失敗した場合、エラーを読んで原因を特定し、修正してから再実行する
- 全テストが通過するまでコミットしない
- lint エラーが残っている場合もコミットしない

## バグ修正時のルール
- まず失敗するテストを書く
- テストが失敗することを確認してから修正を実装する
- 修正後にテストが通ることを確認する
```

**ポイントは3つです。**

**1. 検証コマンドを具体的に書く。** 「テストを実行する」ではなく「`npm test` を実行する」と書きます。Claude Codeは具体的なコマンドがあると、実行→結果確認→判断のサイクルに入りやすくなります。

**2. 失敗時の行動を明示する。** 「テストが失敗した場合、エラーを読んで原因を特定し、修正してから再実行する」のように、失敗→修正→再検証の流れを書きます。これがないと、Claude Codeは「テスト失敗しました」と報告して止まります。

**3. 完了条件を明確にする。** 「全テストが通過するまでコミットしない」は、Claude Codeに「テストが通っていない＝まだ完了していない」と判断させます。

私は最初、「テストを実行すること」としか書いていませんでした。するとClaude Codeはテストを1回実行して、失敗しても「テストを実行しました。いくつか失敗があります」と報告するだけで、修正に入りませんでした。失敗時の行動を書き足してから、自動的に修正→再テストのループに入るようになりました。

### Step 2: Hooksで検証を自動化する（第2層）

CLAUDE.mdのルールは「推奨」です。Claude Codeが忙しい（コンテキストが長い、複雑なタスクに集中している）と、ルールを無視することがあります。

Hooksは「強制」です。settings.jsonに設定したHookは、条件に合致するたびに必ず実行されます。

**PostToolUse Hookで、ファイル編集のたびにテストを自動実行する設定を入れます。**

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm test 2>&1 | tail -20"
          }
        ]
      }
    ]
  }
}
```

この設定をプロジェクトの `.claude/settings.json` に配置します。

**動作の流れ:**

1. Claude Codeがファイルを編集する（Edit or Write ツール）
2. PostToolUse Hookが発動し、`npm test` が自動実行される
3. テスト結果の末尾20行がClaude Codeに返される
4. Claude Codeはテスト結果を読み、失敗があれば修正に入る

**CLAUDE.mdとの違いは「結果が勝手に返ってくる」点です。** CLAUDE.mdのルールでは、Claude Codeが「テストを実行しよう」と判断する必要があります。Hookなら、編集するたびにテスト結果がフィードバックされるので、Claude Codeは判断不要で修正サイクルに入れます。

### Hookの設計で私が失敗したこと

最初、`npm test` の出力を全部返していました（`tail` なし）。テストファイルが多いプロジェクトでは出力が数百行になり、Claude Codeのコンテキストウィンドウを圧迫しました。`tail -20` で末尾だけ返すようにして解決しました。

もう1つの失敗は、`matcher` を空（全ツール対象）にしていたことです。Read ツールでファイルを読んだだけでもテストが走り、無駄な実行が増えました。`"Edit|Write"` に絞ることで、実際にコードを変更したときだけテストが走るようになりました。

### Python プロジェクトの場合

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python -m pytest --tb=short 2>&1 | tail -20"
          }
        ]
      }
    ]
  }
}
```

`--tb=short` でトレースバックを短縮し、コンテキスト消費を抑えます。

### リント（コード整形）も自動化する

テストに加えて、リントも自動実行できます。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --quiet $(jq -r '.tool_input.file_path // empty') 2>/dev/null | head -10 || true"
          }
        ]
      }
    ]
  }
}
```

`jq` で編集対象のファイルパスを取得し、そのファイルだけをlintします。`|| true` でlint対象外のファイル（画像等）ではエラーを無視します。

### Step 3: Skillsで修正ワークフローを定義する（第3層）

第1層（[CLAUDE.md](http://CLAUDE.md)）と第2層（Hooks）で「検証→修正」の基盤ができました。第3層では、特定のタスクに対する**修正手順そのもの**をSkillとして定義します。

Skillsは `.claude/skills/` ディレクトリに配置するMarkdownファイルです。Claude Codeのスラッシュコマンドとして呼び出せます。

### コード修正ループのSkill

```
---
name: fix-loop
description: テスト失敗を修正するループワークフロー
---

## 修正ワークフロー

以下の手順でテスト失敗を修正してください。

### Phase 1: 問題の特定
1. $ARGUMENTS のエラーメッセージを読む
2. スタックトレースからファイルとエラー行を特定する
3. 該当ファイルを読み、周辺のコードを理解する

### Phase 2: 修正の実施
4. エラーの根本原因を特定する（表面的なエラーではなく、なぜそのエラーが起きたか）
5. 最小限の修正を実施する（防御的コーディングではなく、原因の修正）

### Phase 3: 検証
6. テストを実行する: `npm test`
7. 全テストが通過した場合 → 修正内容を報告して終了
8. 新しいエラーが出た場合 → Phase 1 に戻り、新しいエラーで繰り返す

### 制約
- 1回のPhase 2で複数の問題を同時に直さない（1原因1修正）
- テストを書き換えてパスさせることは禁止（テストは仕様）
- 5回ループしても解決しない場合、原因分析の結果を報告して終了する
```

このSkillを `.claude/skills/fix-loop/SKILL.md` に保存し、以下のように呼び出します。

```
/fix-loop npm test が TypeError: Cannot read properties of undefined で失敗している
```

**Skillにワークフローを書く利点は3つあります。**

**1. 再利用性。** 同じ修正パターンを何度でも使えます。毎回プロンプトで修正手順を説明する必要がありません。

**2. 制約の明示。** 「テストを書き換えてパスさせることは禁止」のような制約をワークフローに含められます。CLAUDE.mdに書くと全セッションに適用されますが、Skillなら修正タスクのときだけ適用されます。

**3. ループの上限設定。** 「5回ループしても解決しない場合は報告して終了」のように、無限ループを防ぐ条件を入れられます。

### 記事制作ループのSkill

私の記事制作パイプラインでは、生成→校正→修正のサイクルをSkillの連携で実現しています。

```
1. /draft でテーマから記事を生成
2. /polish で校正・SEO評価（.polish.yaml に結果を出力）
3. polish結果を読み、指摘事項を反映
4. 必要に応じて再度 /polish で確認
```

このフローの中で、Step 2→3が自己修正ループです。polish結果（corrections配列）が「修正すべき点」を構造化データで返すため、Claude Codeは何を直すべきか明確に判断できます。

**構造化された検証結果がループの品質を決めます。** テストの pass/fail、lintのエラーメッセージ、polishの corrections配列——いずれも「何が問題か」「どこが問題か」を明示的に返すから、Claude Codeが次のアクションを判断できるのです。

### Step 4: 3層を組み合わせた実践例

ここまでの3層を組み合わせると、以下のような自己修正システムが構築できます。

**例: Next.jsプロジェクトでの機能追加**

```
# CLAUDE.md（第1層）

## 検証ルール
- コード変更後は `npm test` と `npm run build` を実行する
- TypeScript型エラーが残っている場合はコミットしない
- テスト失敗時はエラーを読んで修正し、再実行する
```

```
// .claude/settings.json（第2層）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx tsc --noEmit 2>&1 | tail -10"
          }
        ]
      }
    ]
  }
}
```

```
<!-- .claude/skills/fix-loop/SKILL.md（第3層） -->
（Step 3 で紹介したワークフロー）
```

**動作の流れ:**

1. Claude Codeに「ユーザー認証のAPIエンドポイントを追加して」と依頼
2. Claude Codeがコードを書く（Write ツール）
3. **第2層が発動**: PostToolUse Hookで `tsc --noEmit` が自動実行、型エラーがフィードバック
4. Claude Codeが型エラーを見て自動修正
5. 再びWrite → Hook発動 → 型チェック通過
6. **第1層が効く**: CLAUDE.mdのルールに従い、`npm test` を実行
7. テスト失敗 → エラーを読み、修正、再テスト → 成功
8. `npm run build` を実行 → 成功
9. コミット

型エラーはHookが即座にキャッチし、テストとビルドはCLAUDE.mdのルールに従って実行される。この2層の組み合わせで、ほとんどのエラーがコミット前に解消されます。

### Step 5: /goal でループの終了条件を設定する

Claude Codeの `/goal` コマンドを使うと、特定の条件が満たされるまでClaude Codeが自動的に作業を続けます。

```
/goal npm test が全て通過し、npm run build がエラーなしで完了する
```

`/goal` を設定すると、Claude Codeは各ターン終了時に小型モデル（Haiku）が条件を評価し、未達であれば次のターンに進みます。

**効果的な /goal の書き方:**

```
/goal pytest tests/ が exit code 0 で終了する
```

```
/goal eslint src/ がエラーゼロで通過する
```

```
/goal npm run build が成功し、dist/ にファイルが生成される
```

**避けるべき書き方:**

```
/goal バグを修正する
```

これでは「何が成功か」が曖昧で、Claude Codeが正確に判断できません。**検証可能な条件**（コマンドの exit code、ファイルの存在、テストの pass/fail）で書くことが重要です。

## Claude Codeの自己修正ループ設計の注意点

**1. CLAUDE.mdのルールは「推奨」であり「強制」ではない。**

Claude Codeはコンテキストが長くなったり、複雑なタスクに集中していると、CLAUDE.mdのルールを無視することがあります。絶対に守らせたいルール（危険コマンドのブロック等）はHooksかsettings.jsonの `deny` リストで強制してください。

**2. Hookの実行時間は短く保つ。**

PostToolUse Hookは編集のたびに実行されます。テスト実行に5秒以上かかると、Claude Codeの作業テンポが崩れます。私は最初 `npm run lint` をHookに入れていましたが、プロジェクト全体のlintに8秒かかり、編集するたびに待たされました。対象ファイルだけをlintするように絞って解決しました。

**3. 検証結果は構造化するほどループの精度が上がる。**

「テスト失敗」だけでは、Claude Codeは何を直すべきかわかりません。「src/auth.ts:42 TypeError: Cannot read properties of undefined (reading 'email')」のように、ファイル名・行番号・エラー種別が含まれた結果を返すようにしてください。`--tb=short`（pytest）や `--verbose`（Jest）のフラグが有効です。

**4. 全てを自動化しようとしない。**

自己修正ループが有効なのは「正解が検証可能なタスク」です。テストの pass/fail、ビルドの成否、lintのエラーゼロ——これらは検証可能です。一方、「このAPIの設計は適切か」「このUIは使いやすいか」は自動検証できません。設計判断は人間が行い、実装の正確性をループで担保する、という使い分けが重要です。
