## ターミナル=開発者の道具、という思い込み

Claude Code は「コーディングエージェント」として紹介されることがほとんどです。でも数か月使い込んだ今、私の中での位置づけは変わりました。**これは個人の業務基盤（バックオフィス）である**、と。

私は一人でスモールビジネスを複数運用しています。コードも書きますが、業務時間の多くは開発ではありません。顧客候補への連絡、意思決定、週次の振り返り、タスクの棚卸し──いわゆる「経営の雑務」です。この記事では、その非開発業務を Claude Code に任せている実例を4つ紹介します。

ちなみに Anthropic のコミュニティアンバサダー募集ページにも「開発者の肩書きは不要です。」と明記されています（出典: <https://claude.com/ja/community/ambassadors> 2026-07-14閲覧）。ツールの側も、開発者以外に開かれ始めています。

コードは1行も出てきません。出てくるのは **Markdown のファイルと、ルールの設計**だけです。

## 事例1: 週次レビューを代行させる

金曜の週次レビュー、続いていますか？ 私は続きませんでした。集計が面倒だからです。

今は `/weekly-review` と打つだけです。スキル（Claude Code に覚えさせた手順書）が以下を実行します。

1. 今週の活動ログ・タスク台帳・記録ファイルを集計する
2. 人間に**2問だけ**質問する（例えば、今週の判断で迷ったこと・来週の最優先）
3. 回答を記録ファイルに追記し、来週分の棚卸しを更新する

設計の核心は「**集計はAI、判断だけ人間**」の分離です。週次レビューが続かない原因は判断が嫌だからではなく、判断の前段の集計が重いからでした。前段を全部 AI に渡したら、私が金曜にやることは2問に答えるだけになり、そして続くようになりました。

### 摩擦の非対称設計

このレビューには習慣ガードの昇格・降格ルールも組み込んでいます。

* **昇格（制限を強める）は無摩擦**: 警告後も続行した日数が週2回を超えたら、警告モードからブロックモードへ自動で上がる
* **降格（制限を緩める）は有摩擦**: 緩めるときは理由の記入が必須

人間は自分に甘い方向へは滑らかに動くので、そちらにだけ摩擦を置く。ルールを自分で書けるのが Claude Code の面白さで、これは「自分の意志力を外部化する」体験です。

## 事例2: 意思決定の滞留を毎朝つきつけられる

一人で仕事をしていると、誰も意思決定を催促してくれません。「決めるだけで前に進む案件」が、気づくと2週間止まっている──これが一人事業の隠れたボトルネックでした。

対策として、意思決定は必ず Markdown の台帳に「起票」するルールにしました。

```
## 意思決定キュー（決めるだけで1ループ閉じる）

| 決定事項 | 起票日 | 選択肢 | 推奨 | 状態 |
|---|---|---|---|---|
| 施策Aを適用するか | 07-12 | 適用 / 修正して適用 / 見送り | 適用 | 未決定 |
```

そして Claude Code が台帳から HTML の「decision board」を自動生成し、セッション開始時に毎回こう注入されます。

```
[decision] 未決定 3件（最古 1日滞留）→ decision-board.html
```

ポイントは、**AI に決めさせるのではなく、決まっていないことを可視化させる**ことです。起票日から滞留日数を自動計算し、放置が数字で見える。それだけで「未決定のまま忘れる」がほぼ消えました。選択肢と推奨案を AI に整理させるところまでは任せ、決定ボタンは常に人間が押します。

## 事例3: タスクの置き場を「1問」で決める

タスク管理ツールの選定で消耗した経験は誰にでもあると思います。私の結論は、ツールより先に**境界ルール**でした。

> **『残っている作業は、人間の送信・記入・公開・決定か？』**
>
> * Yes → 人間の台帳（Markdown）へ
> * No（実装・検証・調査）→ 機械のタスクキューへ

この1問で、すべてのタスクの置き場が一意に決まります。機械のキューに入ったものは Claude Code が消化し、完了時には検証の証跡を書かせます。人間の台帳に残るのは、本当に人間にしかできないこと──送る・記入する・公開する・決める──だけです。

導入して分かったのは、人間の台帳に残るタスクの少なさです。「自分がやらなければいけないこと」の大半は、実はそうではなかった。逆に言えば、本当に人間がやるべき数件が、雑多なタスクの山に埋もれていたわけです。

## 事例4: 「作りすぎ」を仕組みで止める

最後は失敗談です。AI で生産性が上がると、**内部整備ばかりが進む**という病気にかかります。

ツールを磨く、テンプレを整える、管理の仕組みを作る──AI と一緒だとこれが楽しく、しかも成果物がどんどん完成する。一方で顧客に連絡する・成果物を公開するといった「市場に触れる仕事」は AI が代行できないので、相対的にどんどん重く感じられる。私の観測（送信だけが残った数件のタスクについての個人の記録で、厳密な統計ではありません）では、機械側のタスク（資料・ツール・計画）が90点台まで完成する一方、\*\*人間の送信だけが残ったタスクの放置率はほぼ100%\*\*でした。作る力が上がった分だけ、送らない在庫が積み上がったのです。

対策はここでも仕組み化でした。

* 週次で見る指標を「**市場接触数/週**」の1つに固定する（内部の完成度スコアは進捗指標にしない）
* セッション開始時に接触数を自動注入する: `[market-first] 直近7日の市場接触: 6件 | 最終接触: 4日前`
* 週の最初の作業枠30分は、送信キューの最上位から消化する。ツール磨きはその後

AI 活用の記事は「どれだけ作れるようになったか」を語りがちですが、個人の実務では\*\*「作ったものを外に出す人間側の行動」が新しいボトルネック\*\*になります。ここに気づけたこと自体が、Claude Code を業務基盤にして得た一番の学びかもしれません。

## 横断する設計原則

4つの事例に共通する原則を抽出すると、こうなります。

1. **人間にしかできないことだけを人間に残す** — 送信・記入・公開・決定。それ以外は委譲の候補
2. **リマインドは通知ではなくセッション注入で** — 通知は無視できるが、作業開始時に目の前に置かれた1行は無視しにくい
3. **摩擦は非対称に置く** — 望ましい方向は無摩擦に、望ましくない方向にだけ摩擦を
4. **AI は判断材料の整理まで、決定は人間** — 責任の所在を曖昧にしない

## 非エンジニアの始め方

「でもターミナルでしょ？」という方へ。なお、Claude Code 本体のインストールとログインは済んでいる前提で書きます（セットアップ自体は[公式ドキュメント](https://code.claude.com/docs/ja/setup)の手順どおりで、ここが最初の山です）。その先で私の構成が使っている道具は、実は次の2つだけです。

* **CLAUDE.md**: Claude Code が毎回読む指示書。ここに自分の業務ルールを日本語で書く
* **Markdown の台帳**: 意思決定キュー、週次記録、タスク表。ぜんぶただのテキストファイル

最初の一歩としておすすめなのは、事例2の縮小版です。

1. `decisions.md` というファイルを作り、決めていないことを表で3件書く
2. CLAUDE.md に「セッション開始時に decisions.md を読み、未決定の件数と最古の起票日を教えて」と書く
3. 毎朝 Claude Code を開く

これだけで「決まっていないことが毎朝見える」が手に入ります。プログラミングの知識は不要です。うまく回り始めたら、週次レビューの集計、タスクの棚卸しと、1つずつ委譲範囲を広げていけばいい。

私自身、この使い方の知見を共有したくてこの記事を書きました。コードを書かない業務にこそ、伸びしろが大きく残っています。「エンジニアじゃないから関係ない」と思っていた方の、最初の一歩になれば嬉しいです。
