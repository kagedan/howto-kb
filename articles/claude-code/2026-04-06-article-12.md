after turboquant and qwen3.5-35b-a3b, i got curious: how realistic is it to use kv cache as a document store today? to have vectorless, RAG-less search. so i prefilled 258K out of 262K context window on L4 (a budget GPU popular in prod). ~99% of the slot is pre-computed and stored, users load it on the fly in ~1s. system prompt + query append to the end, generation takes ~3K tokens, enough for search. at 99% fill rate, decoding runs ~20 tps on L4.

i prepared some ego datasets (jina papers, which i know best), plus popular novels in chinese and english. the results are actually pretty good. some hallucination, but most answers are solid and well-grounded. what's more interesting is the cost: ~$0.26/h on L4 spot. single LLM. no vector database, no embedding model, no workflow/pipeline engineering.

using kv cache as document store is nothing new, like the old CAG paper. but with quantized kv cache and modern attention (hybrid SSM-attention, GQA, MQA, MLA), the economics are changing fast. if we solve cold-prefill speed and decoding speed, and budget GPU costs keep dropping, the future of search could be vectorless. radical, but possible.

KV cache can be a way of embedding knowledge. on L4 for 258K tokens, it takes 572s to embed, much slower than any embedding model we have. but once embedded, every query runs against full attention over the entire document. no chunking loss, no retrieval miss. recall is theoretically perfect - you're attending to everything. try the demo here: https://t.co/TydWqZyU6P

the obvious ceiling is the context length: 262K tokens is roughly one book. to scale to 10 books or an enterprise KB, you'd need multi-slot routing - pick the right embedded slot per query.
