## この記事でわかること

* AIエージェントの「記憶」が3種類に分類される理由と対応するストレージ
* ChromaDB（ベクトルDB）とSQLiteを使い分ける設計判断の背景
* 「全部RAGに入れる」アンチパターンと私が経験した失敗
* `unified_search` — 情報取得順序をプロンプトではなくコードで強制する仕組み
* コレクション設計の具体的なルール

## はじめに

AIと対話していて、こんな経験をしたことはないでしょうか。

> 昨日、良い設計方針が出た。よし、続きをやろう。  
> 翌朝、新しいセッションを開いて続きを依頼すると…ゼロから始まる。

これはAIの「記憶力の問題」ではありません。**「どこに情報を置くか」という設計の問題**です。

私はこの問題を解決するために、RAG（ChromaDB）とSQLiteを組み合わせた知識基盤システム「KA（Knowledge Assistant）」を構築してきました。構築と運用を通じてたどり着いた結論は、**AIに「何を覚えさせるか」より「どこに置くか」の設計精度が、システムの品質を9割決める**という事実でした。

第3回でエージェントチームの設計を紹介しました。今回はそのチームが「何を、どこから参照するか」——知識基盤そのものの設計論に踏み込みます。

## 前提知識

以下のキーワードを事前に押さえておくと読みやすくなります。

* **RAG（Retrieval Augmented Generation）**: LLMに外部データベースを組み合わせ、検索した情報をもとに回答させる手法。本の索引のようなもので、LLM自体に記憶させるのではなく「外部の本棚から引っ張ってくる」
* **ChromaDB**: ベクトル（意味）検索に特化したデータベース。「概念が似ている情報」を取り出せる
* **MCP（Model Context Protocol）**: Claude Codeからツールを呼び出す仕組み。AIが外部システムを操作できるようにするプロトコル

## 第1節：AIに「記憶」させるとはどういうことか

人間の記憶には種類があります。

* **長期記憶**: 本で学んだ知識、過去の経験の蓄積
* **短期記憶（作業記憶）**: 今考えていること、目の前のタスクの文脈
* **手続き記憶**: 自転車の乗り方のような「体で覚えた」行動原理

AIエージェントの知識基盤も、これとほぼ同じ構造で分類できます。

| 人間の記憶 | KAの対応 | 格納する情報の例 |
| --- | --- | --- |
| 長期記憶 | ChromaDB（RAG） | 技術ドキュメント・Web検索結果・設計書 |
| 短期記憶 | CLAUDE.md | 全セッション必須の最重要ルール・制約 |
| 手続き記憶 | エージェントメモリ `.md` | 行動原理・禁止事項・チームの教訓 |

この分類を無視して「全部RAGに入れれば良い」と考えると、後で必ず困ります。私はその失敗を経験しました。

## 第2節：失敗談 — 「全部ChromaDBに入れていた時代」

KA構築初期（内部でPhase 1と呼んでいます）、私は楽観的でした。

> 「RAGがあればなんでも引き出せる。まず全部入れよう」

行動原理・禁止ルール・技術情報・ログ・会話履歴——すべてをChromaDBの同一コレクションに入れ続けました。

2週間後、何が起きたか。

1. **検索精度が低下した**: 「エージェントが守るべきルール」と「X APIの仕様書」が同じベクトル空間に混在し、意味的に近くても用途が全然違う情報が上位に来る
2. **「どこに何があるか」が分からなくなった**: コレクションが1つなので、何を削除・更新すべきかの判断基準が作れない
3. **行動ルールが揮発した**: 毎回検索で引き出さないと適用されない構造になってしまい、事実上ほぼ守られない

その後、エージェントチームの討議で「Phase 2 再設計」に舵を切りました。

**根本問題は「用途が違う情報を同じ場所に置いていた」ことでした。**

## 第3節：設計の転換点 — 「何を、どこに、なぜ」

Phase 2再設計で確立したルールがこちらです。

| 情報の種類 | 置き場所 | 理由 |
| --- | --- | --- |
| ドキュメント本文・技術情報 | ChromaDB `content_{id}` | 意味検索が必要 |
| 汎用技術知識・チーム共有知見 | ChromaDB `common_knowledge` | 横断的に参照したい |
| ファイルカタログ（パス・名前） | ChromaDB `file_index_{id}` | ファイル名・パスで検索 |
| 会話履歴・記事メタ・ログ | SQLite | 日時・件数・一覧の構造化クエリ |
| 行動原理・禁止事項・教訓 | エージェントメモリ `.md` | コンテキスト注入で「常時適用」させる |
| 最重要制約 | CLAUDE.md | 全セッション必須のため |

設計判断の核心は、たった1つの問いに集約されます。

> **「この情報は、意味（意図）で検索するか？それとも構造（ID・日時・件数）で検索するか？」**

* 「X APIの認証方式を教えて」→ 意味検索 → ChromaDB
* 「先週の会話ログを5件並べて」→ 構造クエリ → SQLite
* 「今すぐ行動原理を適用して」→ 常時注入 → `.md` / `CLAUDE.md`

この問いに答えられない情報は、まだ「どこに置くか」を決める準備ができていない状態です。

## 第4節：ChromaDB × SQLite ハイブリッドの実装

「なぜ1つのDBに統一しなかったのか」はよく聞かれます。

答えは「**役割が根本的に違うから**」です。

ChromaDBはベクトル（意味の近さ）で検索します。「PythonのFAQが欲しい」という曖昧な質問には強い半面、「2026年3月の記事を日付順に並べて」は苦手です。逆にSQLiteは集計・JOIN・日付範囲は得意ですが、「なんとなく似た情報」という曖昧な検索はできません。

**両者は競合ではなく、補完関係です。**

なぜChromaDBを選んだのか。私は当初、以下の代替技術も検討しました。

| 技術 | 不採用理由 |
| --- | --- |
| Qdrant（Rust製ベクトルDB） | 現段階ではChromaDBで十分。スケール時の移行先候補 |
| FAISS（Meta製） | サーバー機能がなく自前ラップが必要 |
| Elasticsearch | リソース消費大（8GB RAM環境では同居困難） |
| SQLite + sqlite-vss | エコシステムがChromaDBより未成熟 |
| MongoDB Atlas Search | クラウド前提、ローカル完結ができない |

VPSのリソース制約（RAM 8GB）とローカル完結要件が選択を絞り込みました。「最強のDB」を選ぶのではなく、**自分の環境と用途に合った組み合わせを選ぶ**のが設計の本質です。

## 第5節：unified\_search — 「お願い」から「強制」へ

第2回で「情報取得の優先順位（RAG→メモリ→Web→学習データ）」を紹介しました。ここでは「なぜその順序をコードで強制するに至ったか」を説明します。

最初はプロンプトに書いていました。

```
# 情報取得ルール
回答前に必ずRAGを検索すること。
学習データだけで回答してはいけない。
```

結果：AIは「確率的に」従います。調子が良いときは守る。文脈が積み重なったときは忘れる。

プロンプトエンジニアリングの限界です。**AIへの「お願い」は、人間でいえば「念のため確認して」と同じ重みしかありません。** 守ったり守らなかったりするのは当然です。

解決策は、MCPレイヤーで処理順序をコードに焼き込むことでした。

このツール `unified_search` は、クエリの意図を自動判定して最適な実行パスを選びます。

* 「会話履歴ID `abc-123` の内容を教えて」→ SQLite直接検索（UUID）
* 「先週のRAG設計に関する会話を出して」→ SQLite先行（日時・一覧系）
* 「X APIの仕様について教えて」→ ChromaDB先行（概念系）

AIにどのDBをいつ使うかを「判断させない」。コードが自動的に決定する。

１ヶ月運用して分かったことがあります。**コードで強制しない限り、AIは学習データに戻ろうとします。** それはモデルの設計上の特性であり、「お願いで変えようとする」のが間違いでした。

これが、プロンプトエンジニアリングからコンテキストエンジニアリング（「何を、いつ、どのDBから渡すか」の設計）への移行です。

## よくある落とし穴

### 「とりあえず全部入れる」症候群

RAGを導入した直後は「とにかく情報を増やせば精度が上がる」と思いがちです。しかし品質スコアの低い情報、陳腐化したドキュメント、役割が違うデータが混在すると検索精度は下がります。「入れる量」より「整理する仕組み」に投資する方が費用対効果は高いです。

### 行動ルールをRAGに入れてしまう

「AIにルールを守らせたい」とRAGに入れても、毎回検索で引き出されるとは限りません。常時適用したいルールは `.md` ファイルとしてエージェントメモリに置き、コンテキスト注入で「毎回必ず読み込む」構造にするべきです。

### 検索順序をプロンプトで制御しようとする

前述の通り、プロンプトによる「お願い」は確率的にしか機能しません。情報取得のパイプラインはコードとして実装し、AIに判断させない設計を目指してください。

## まとめ

今回の核心を一言で表すなら：

> **「覚える量」ではなく「設計の精度」が知識基盤の品質を決める**

KAが到達した設計原則は3つです。

1. **用途で分類する**: 「意味で探す」情報はChromaDB、「構造で探す」情報はSQLite、「常時適用」したい情報は `.md` に置く
2. **行動ルールはRAGに入れない**: 常時適用が必要なルールはメモリファイル経由でコンテキスト注入する
3. **検索順序はコードで強制する**: プロンプトへの「お願い」では確率的にしか機能しない

ChromaDB × SQLite のハイブリッドは「2つのDBを管理する手間」を生みます。しかし、それ以上に「正確な情報が正確なタイミングで引き出される」精度向上をもたらします。

次回（第5回）は「設計が決まったら、次は自動で情報を仕入れる仕組み」——知識基盤の自動補充とメンテナンス戦略を扱います。
