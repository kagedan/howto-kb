AIコーディングツールを選ぶとき、ついモデル名を見てしまう。どのモデルが賢いか。どれが速いか。どれが一番長いコンテキストを持てるか。

でも実務で差が出るのは、そこだけではない。むしろチームに入れた瞬間に効いてくるのは、AIがどこで動き、何を読めて、どのコマンドを実行できて、失敗したときに何が残るかである。

同じ「AIコーディングエージェント」でも、ローカルCLIで動くもの、クラウド環境でブランチを切るもの、GitHubのissueからPRまで進むもの、IDEの中で作業者の手元に寄り添うものでは、まったく別の道具になる。生成されるコードだけを見ると似ている。運用すると違いが出る。

## ローカルCLIは自分の作業台を広げる

Codex CLI や Claude Code のように、ターミナルからリポジトリを触るタイプは気持ちがいい。既存の作業ディレクトリにいて、テストを走らせ、差分を読み、必要ならファイルを直す。人間の開発フローにかなり近い。

この型の強さは、試行錯誤の速さにある。小さな修正、調査、リファクタ、テスト追加のような仕事は、チャット画面にコードを貼るより自然に進む。AIがプロジェクトの中へ入ってくるので、コンテキストスイッチが少ない。

ただし、これは「自分の作業台を拡張する」道具でもある。作業台に何を置いているかは、こちらの責任になる。

読ませるディレクトリはどこまでか。`.env` やローカル設定に触れさせるのか。`npm install` やマイグレーションを実行してよいのか。削除や大きな移動はどこで止めるのか。こういう判断を曖昧にしたまま「賢いから任せる」と、AIの性能とは別のところで危なくなる。

ローカルCLI型は、個人開発ではかなり強い。けれどチームでは、各メンバーの端末ごとに境界がばらつきやすい。だからこそ、permission prompt や sandbox は邪魔な確認ではなく、作業の輪郭を決めるUIとして見たほうがいい。

## クラウド型はレビュー導線にAIを入れる

一方で、Codex cloud や GitHub Copilot coding agent のようなクラウド/PR型は、発想が少し違う。AIを自分の端末に呼ぶのではなく、issue、branch、pull request、CI、review という既存のチーム導線に入れる。

これは単に「クラウドでも動く」という話ではない。作業の単位が変わる。

ローカルCLIでは、AIとのやりとりは手元の試行錯誤として残りやすい。クラウド/PR型では、issueの依頼文、作られたブランチ、PR description、CI結果、レビューコメントが作業のインターフェースになる。AIがどれだけ賢いか以前に、依頼がレビュー可能な形で切られているかが効いてくる。

ここで雑なissueを投げると、雑なPRが返ってくる。これは人間でも同じだが、AIの場合は「なんとなく意図を察して先回りする」能力があるぶん、ズレた先回りも起きる。

だからクラウド型を入れるチームは、AI用に新しい魔法のプロンプトを書くより先に、issue template を見直したほうがいい。何を直すのか。触ってよい範囲はどこか。受け入れ条件は何か。テストは何を通すのか。PRに残してほしい判断は何か。

クラウド型の本質は、AIがコードを書けることではない。AIの作業を、人間がレビューできる単位に変換することにある。

## permission は安全装置というより、依頼の境界である

AIエージェントが怖いのは、悪意を持つからではない。むしろ逆で、よかれと思って進みすぎるから怖い。

「このテストを直して」と頼んだら、本体コードを変えるかもしれない。「この警告を消して」と頼んだら、設定を緩めるかもしれない。「型エラーを直して」と頼んだら、公開APIの形まで変えるかもしれない。人間なら途中で「それは今回やることですか」と聞く場面でも、AIは筋の通った解釈を作って走れる。

ここで permission prompt や workflow approval が効く。これはAIを信用していないから挟むものではない。依頼の境界を、実行の途中で具体化するために挟むものだ。

実務では、最初の依頼文だけで完璧に範囲を切るのは難しい。コードを読んで初めて、修正が大きくなることもある。だから「このコマンドを実行してよいか」「このファイル群を変更してよいか」「この外部アクセスを許すか」という確認は、作業のブレーキではなく、設計上のチェックポイントになる。

賢いエージェントほど、この境界は重要になる。能力が低い道具は、そもそも遠くまで行けない。能力が高い道具は、頼んでいない場所まで行けてしまう。

## repo instruction はプロンプト術ではなく操作契約になる

AGENTS.md や CLAUDE.md のようなリポジトリ内の指示ファイルも、単なる便利プロンプトとして扱うともったいない。

チームでAIを使うなら、個人のチャット履歴よりリポジトリに残るルールのほうが強い。ビルドコマンド、テストコマンド、触ってはいけないディレクトリ、レビュー前に見るべきファイル、既存の設計判断。こういうものは、人間向けREADMEにも書けるが、AIに渡す操作契約として明示すると意味が変わる。

たとえば「このプロジェクトではDBスキーマ変更を勝手に作らない」「UIコンポーネントは既存のデザインシステムを使う」「テストが落ちたらスナップショットを安易に更新しない」といったルールは、モデル性能では吸収しきれない。プロジェクトごとの癖であり、チームの価値判断だからだ。

AIに毎回口頭で説明するのは弱い。説明した人しか再現できないし、忘れる。リポジトリに置いた操作契約なら、ツールが変わっても残る。Codex、Claude Code、Copilot、Cursor のどれを使うにしても、チームがAIに何を許すかという設計は移植できる。

## ツール比較は「失敗したときの戻し方」で見る

AIツールの比較表を作るなら、モデル名や対応IDEだけでなく、失敗したときの戻し方を入れたい。

どのログが残るのか。どの差分を人間が読むのか。テスト結果はどこに出るのか。権限確認はどの粒度で止まるのか。外部ネットワークやシークレットに触れる境界はどこか。作業が大きくなりすぎたとき、誰がどこで止めるのか。

ここを見ないと、ツール選定が「一番賢そうなものを選ぶ」話になる。短期的にはそれでも動く。だが、チームに入れると運用のほうが先に問題になる。

AIが作ったPRを誰がレビューするのか。レビュー担当者は、AIが何を根拠に変えたかを読めるのか。CIが落ちたとき、AIに再実行させるのか、人間が切り戻すのか。依存更新や設定変更のような危険な差分を、通常の機能追加と同じ扱いにしてよいのか。

こういう問いに答えられるチームは、モデルが入れ替わっても強い。逆にここが曖昧なまま最新モデルへ飛びつくと、毎回同じ混乱を新しい名前で繰り返す。

## agent-ready とは、AIに全部任せることではない

agent-ready なチームというと、AIにどんどん作業を投げられるチームを想像しがちだ。実際には少し違う。

agent-ready とは、AIが動く場所を決めていることだ。読ませるものを決めていることだ。実行してよいことと、止めるべきことを分けていることだ。最後に人間のレビューへ戻す道筋を持っていることだ。

モデル性能はこれからも変わる。今日の最強モデルは、来月には別の名前になっているかもしれない。そこを追うのは必要だが、そこだけ見てもチームは強くならない。

残るのは運用設計である。AIをどこで動かし、何を許し、どう失敗を見える形に戻すか。AIコーディングツールの本当の性能は、モデル単体ではなく、その境界まで含めた開発環境として決まる。

## Source notes

* OpenAI Codex cloud docs: Codexをクラウド側の作業者として走らせ、ブランチやPRに接続する文脈の確認に使用。
* OpenAI Codex CLI repository: ローカルCLI型の実行環境、sandbox、承認フローを考える材料として使用。
* Claude Code overview / security docs: ターミナル、IDE、background agent、permissions、hooks、MCP、project instructions の位置づけを確認。
* GitHub Copilot coding agent docs: issueからbranch / PR / review flowへつながるクラウドエージェント型の整理に使用。
* "Configuring Agentic AI Coding Tools": repo-local instructions を個人プロンプトではなく操作契約として扱う背景に使用。
* "Overeager Coding Agents": permission boundary を、頼んでいない作業を止める authorization の問題として扱う補助線に使用。
