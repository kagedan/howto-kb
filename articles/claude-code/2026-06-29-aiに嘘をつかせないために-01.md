### はじめに

「AIって嘘をつくから信用できないよね」

という言葉は、IT業界に関わる人なら必ず聞いたことはあるでしょうし、その他業種の方でも度々目にする言葉だと思います。  
ただ、果たしてAIは、意図的に嘘をつきたくてついているのでしょうか（もしかしたら、そういうパターンも我々の知らぬところであるかもしれないですが...）。  
人間の場合は悪意や焦り、保身、気まぐれ、さまざまな状況によって嘘をつく理由が生まれます。が、本当に嘘をつきたくてAIは嘘をついているのでしょうか。

### 人間もAIも嘘をつく

ざっくり言うと、生成AIは「事実を知ってる機械」というより、「次にそれっぽい言葉を予測する機械」です。  
そのため、知らないことでも“もっともらしく埋める”性質があります。これを一般に「ハルシネーション」と呼びます。

=============

AIが嘘をつくのってどうして？　と質問したらこんな風に返されました。  
上記の質問相手はChat GPTですが、GeminiとClaude Codeにも上の文章については同意されたので真っ赤な嘘ではないのでしょう。こんな話は今更の議論で、言及するまでもないことだとは思います。

インターネットにある膨大な情報・入力された言葉に対する予想を返しているだけ。だからこそ平然と嘘をつきますし、無知な相手（この場合は質問した人間側）を納得させるような巧みな言葉で会話ができるのです。この回答って、皆が同じようなことを言ってるからこそ「そういう回答を返せば良い」と思われているのかもしれません。

これって、人間と一緒ですよね。  
人間は実体験のように、現状のAIには身に付けられない学習方法を持ちますが、私達だって今まで得てきた情報と経験則で会話を重ね、場合によっては相手に悪気なく嘘をつきます。

### じゃあどうすればいいのか

とはいえ、仕事でAIに嘘をつかれても困りますし、なるべく情報は正確に返して欲しいわけです。難しい準備もハードルが高い。

その対策の1つとして、運用に乗っているソースコードに追従した「AIが読み取るための仕様書（マークダウン形式）」を準備することで、ハルシネーションが起こる確率を下げています。  
現状は、下記のようなフォーマットで、1つの機能に対してAI用仕様書を作成することで嘘をつけないようにします。

| ファイル名 | 役割 |
| --- | --- |
| README.md | 仕様書全体の目次・概要。改修履歴の管理 |
| requirements.md | 要件定義。ユーザーストーリーと受け入れ条件 |
| design.md | 設計ドキュメント。アーキテクチャ概要、インタフェース、処理フロー |
| overview.md | 機能説明。モジュールの目的と主要機能の概要 |
| architecture.md | アーキテクチャ詳細。レイヤー構成、依存関係、ディレクトリ構造 |
| business-logic.md | ビジネスロジック詳細。判定条件、計算ロジック、エラーハンドリング |
| data-models.md | データモデル詳細。エンティティ定義、DB テーブル構造 |

全ての情報を1つのマークダウンにまとめてしまうと分量も多くなり、内容を確認するのが一苦労。人間がそう思う感じるのですから、きっとAIだって同じ気持ちでしょう。そのためのマークダウンの分散です。

### 上記のmdをどう書き分けるか

ファイルを分ける理由は前述の通りですが、何をどこに書くべきなのでしょうか。実際に運用しているテンプレの骨子を紹介します。  
分け方の基準は至ってシンプルで、**AIがその粒度の質問を受けたとき、最短で正確な根拠にたどり着けるか** です。

* **README.md** … 目次と改修履歴。AIが最初に読む入口。「どのマークダウンに何が書いてあるか」のリンク集、最終更新日・対応コミットを置きます。ここが古いと、他が正しくても全体の信頼が崩れます。
* **requirements.md** … 要件定義。ユーザーストーリーと受け入れ条件。「なぜこの機能が存在するのか」を書きます。仕様の意図を取り違えさせないための土台です。
* **design.md** … 設計。アーキテクチャ概要・主要インタフェース・処理フロー。「どう作られているか」の全体像を渡します。
* **overview.md** … 機能の目的と主要機能の要約。design より粒度を粗くし、まず全体像を掴ませる用です。
* **architecture.md** … レイヤー構成・依存関係・ディレクトリ構造。「どのファイルが何を担当するか」の地図。調査時にAIが当たりをつけるために使います。
* **business-logic.md** … 判定条件・計算ロジック・エラーハンドリング方針。**最もハルシネーションが起きやすい業務ルールをここに集約します。**「上限はいくつか」「エラー時にどう振る舞うか」の唯一の正解を置く場所です。
* **data-models.md** … エンティティ定義・テーブル構造。項目名と型の正解。AIが存在しないカラムを創作する誤動作を防ぎます。

コツは「事実が変わりやすい順に独立させる」こと。とくに `business-logic.md` と `data-models.md` は改修を行うと影響を受けやすいので、ここを最新に保つだけでハルシネーションの大半は抑えられます。

書き出しのテンプレ例（business-logic.md）はこんなイメージです。

```
## 貸出上限
- 一般会員: 3冊まで
- プレミアム会員: 10冊まで
- 延滞中の本が1冊でもある場合は新規貸出不可（理由: 督促運用の都合）

## エラーハンドリング
- 対象書籍が存在しない: 404 を返す（ログは Info）
- 在庫情報の取得に失敗: 500 を返す（ログは Error）
```

「事実」と「根拠」をセットで、箇条書きで端的に。AIは長文の散文より、こうした構造化された記述の方が正確に情報を拾ってくれます。

### 仕様書がある／ないで、どう変わるのか

AI用仕様書の有無で回答がどう変わるかを、完全に架空の題材で比較してみます。  
題材は「図書館の蔵書貸出API（ book-lending ）」です（実在するjinjerのプロダクトとは一切関係ありません。あくまで仕組みを説明するための作り話としてご覧ください）。

```
# book-lending モジュール — 会話ログ（サンプル / 架空の題材）
# ※ 実在システムとは無関係の、図書館の蔵書貸出を題材にした架空の例です。

────────────────────────────
【パターンA：仕様書なしで聞いた場合】
────────────────────────────

開発者
book-lending っていうAPI、引き継いだんだけど、会員1人あたりの貸出上限って何冊だっけ？

AI
一般的な図書館システムだと、会員1人あたり5冊までが上限になっていることが多いです。
延滞があると貸出停止になるのが通例ですね。

開発者
（…これは実装内容の話？　一般論を答えられている気がする）

→ 仕様書がないと、AIは「それっぽい一般論」で空白を埋めてしまう。
　 これがハルシネーションになる。もっともらしい内容なため、かえって危ない。

────────────────────────────
【パターンB：仕様書を読ませた場合】
────────────────────────────

開発者
さっきの質問、仕様書のbusiness-logic.md と data-models.md を読みつつ、コードも確認した上でもう一度答えて。貸出上限は何冊？

AI
business-logic.md に記載があります。このAPIの貸出上限は会員ランクで分かれていて、
- 一般会員: 3冊まで
- プレミアム会員: 10冊まで
です。加えて「延滞中の本が1冊でもある場合は新規貸出不可」というルールがあります
（business-logic.md「貸出上限」節）。
先ほどの“一般論の5冊”ではなく、こちらが正です。

開発者
なるほど、ランクで内容が変わるのね。データ構造はどうなってる？

AI
data-models.md を見ると、書籍の貸出状況は status（available / lent / reserved）で管理され、
会員ランクは member.rank に持っています。
推測ではなく、data-models.md の定義に沿った値です。記載のない項目については「仕様書に記述がない」と明示します。

────────────────────────────

開発者
ありがとう。最初の5冊のまま実装してたら危なかったです。

AI
仕様書がない状態だと、私は知らない部分を一般論で埋めてしまいます。
仕様書があると、推測ではなく*根拠を示して*答えられますし、「分からない」と言うこともできます。
```

同じ質問でも、AはAIの想像、Bは仕様書という典拠に基づく回答です。  
この差こそが、AI用仕様書を用意する一番の目的です。

### 何が良いのか

上のA／Bの比較のように、仕様書がないとAIはそれっぽい一般論で空白を埋めてしまう可能性があります。これがハルシネーションです。  
お見せしたのは簡略化した架空のサンプルなので内容は分かりやすいですが、実際の業務で使われているソースコードは非常に複雑で、コードの行数は数万行を超える機能もあります。にも関わらず、急務となる障害調査や問い合わせには1秒でも早い対応が求められる厳しい状況は突然発生します。  
特に私が現在担当するバッチ処理は1つ1つの機能が重く、人間の目で処理を把握すると経験（勘に近い能力）が求められ、下手したら実装をした本人しか触れない・調査ができない、いわゆる属人化の原因となるわけです。俗に言う「あの人がいないと困る」という、システム開発あるあるですね。

とはいえjinjerのような大規模なシステムで、ドメイン知識が求められる状況で完全に属人化が防げるというわけではありません。が、AI用の仕様書を共有することにより誰でも調査を行える状況を作り出し、システムそのものを把握するという行為にも繋げられる状況を作り出しつつあります。

### 継続して行うべきこと

明確なデメリットとして、提示した仕様書のメンテナンスは継続して行わないといけません。コードの改修の度、AI自身に仕様書を追記・更新する手間が増えるのは事実です。但し、AIに嘘をつかせないという大きなメリットが付属します。

現状チーム内では、この仕様書を指定のGitリポジトリで共有する施策をとっています。  
新規実装・改修等があれば都度仕様書をメンテナンスすることは忘れないようにすることで、その効果は波及しつつあります。このマークダウンとSteeringと並行して利用すると、より効果的です。

### とはいえ

AIに調査を丸投げ！　という未来はまだまだ先のことかもしれません。  
人間が学習を止める時、AIの嘘を判断できなくなってしまうことは危惧するべきです。そういう意味では、我々エンジニアと呼ばれる存在が不要になることはないのでしょう。

それでも、仕様書という典拠を持たせるだけで、AIは知らないことを一般論で埋めるのではなく、根拠を示して答え、時には質問に答えられない（分からない）という正直なフィードバックを言えるようになります。完全に嘘を封じることはできなくとも、口当たりの良い嘘を大幅に減らすことはできる。  
「AIに嘘をつかせない」とは、AIに正解を暗記させることではなく、嘘をつかなくて済む環境を人間側が用意してあげることなのだと思います。

AIを使っていることがスキルと見做される日は近く、IT業界で働く以上避けることはできないでしょう。  
なので、我々はこの「言葉巧みな、口が上手い知ったかぶりのアシスタント」と上手く付き合っていけば少なくともマイナスにはならないのでしょう。
