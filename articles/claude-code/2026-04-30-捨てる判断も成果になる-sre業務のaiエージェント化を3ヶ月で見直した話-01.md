## はじめに

こんにちは、カンリーでSREをしている吉村です。

今回は、**SRE業務をAIで再設計したくて Notion AI Agent で「SRE調査兵団」を作ったが、3ヶ月で停止し、Claude Code の skill に寄せる判断をした話** を書きます。

ざっくりまとめると、

* SREの問い合わせ業務をAIで再設計したいと考えた
* Notion AI Agent を選び、「SRE調査兵団」というエージェントを作った
* 運用してみたら、課題解決方法の提案までで止まる構造的な課題が見えた
* 修正対応・実行までAIに任せたい流れの中で、Claude Code の skill に寄せる判断をした

という3ヶ月でした。図にするとこんな流れです。

「作ったけど捨てた」話ではあるのですが、**捨てる判断ができたこと自体が、今回いちばんの収穫** だったと思っています。  
同じようにAIエージェントの内製を検討している方の参考になれば幸いです。

## 🤔 背景：SREの問い合わせ業務を、AIで再設計したかった

SREチームでは、業務プロセスそのものをAIに載せていくことを考えていました。

ここで言う「業務プロセスをAIに」というのは、コーディング補助やミーティング要約のような局所的な活用ではなく、**問い合わせ対応・調査・記録のような日々の業務フローそのものをAIに載せる**、という意味です。

最初に対象として選んだのは、頻出するオペレーションである「SREへの問い合わせ・依頼の受付」でした。Slackのチームメンションを起点に発生していた問い合わせを、AIで再設計できるかを検証していくことにしました。

問い合わせ運用は、こういう状態になりがちです。

* 過去に同じ問い合わせがあった気がするが、どこにあるか思い出せない
* ナレッジに書いてあるはずだが、検索しても出てこない
* 結局、その場で人が答え、また同じ会話が別のスレッドで繰り返される

つまり問題は、ナレッジが無いことではなく、ナレッジ・履歴・経緯を横断して引き当てる作業を毎回人がやっていることでした。

ここに「問い合わせの受付 → 横断検索 → 回答提示 → 記録」という流れをAIに載せれば、SREは「AIの一次回答を見て判断する」という付加価値の高い役割に集中できるはず。それが出発点でした。

## 📐 4つの代替先候補を同じ軸で比較する

実装の候補として、次の4つを挙げました。

評価は、定量と定性の2軸で行いました。

* **定量（QCDスコアリング）**：1回あたりの推論単価、月額固定費、初期構築工数、応答速度、連携データソース数、共通質問10問での回答正解率
* **定性（Pros/Cons）**：SRE実務への適合性、ナレッジ自動同期、運用メンテナンス性、コスト変動リスク、精度限界、問い合わせ履歴の保持性

特に共通質問10問は、Datadogのログコスト切り分け、ECSのALB配下サービスのヘルスチェック、RDSのバックアップとリストアなど、**SREに実際よく飛んでくるタイプの問い合わせ** を揃え、全候補にフラットに投げて比較しています。

全項目を載せると煩雑になるので、ここでは特に判断に効いた観点だけサマリ表にまとめます。

| 候補 | コスト | 連携の容易さ | カスタマイズ性 | 初期構築工数 | 主な懸念 |
| --- | --- | --- | --- | --- | --- |
| Dify | 中 | 高 | 高 | 中〜大 | RAGチューニング前提で初期学習コストが大きい |
| Amazon Q Business | 高 | 高 | 中 | 小 | ユーザー数分のライセンス＋ベクトルDB課金 |
| Notion AI Agent | 低〜中 | 高（Notion中心） | 低 | 小 | ロジックがブラックボックス／ベータ版 |
| Claude Code | 低〜中 | 中（要実装） | 高 | 大 | Lambda+Slack App実装、セッション非保持、応答遅延 |

各候補の所感を簡単にまとめると、

* **Dify** は柔軟性が高い一方、まともに使うにはチャンク分割・古いドキュメントのクリーニング・ハイブリッド検索の比率調整といったRAGチューニングが前提で、片手間で立ち上げるには重い
* **Amazon Q Business** はコネクタが揃って連携は容易だが、エンドユーザー全員分のライセンス＋ベクトルDBのユニット課金が発生し、問い合わせbot用途にはコストが厳しい
* **Claude Code** はMCP経由で各種連携が可能で社内利用も進んでいたが、検証時点では **セッション非保持・応答に5分以上・Notionピック精度の劣後** といった課題があり、Lambda＋Slack App実装の初期工数も重かった
* **Notion AI Agent** は高度なチューニングをせずともNotion・Slackをベースに一定精度で回答でき、エンジニア部内に活用事例もあって **学習・展開コストが他候補より明確に低かった**

各候補で実際に議論したポイント（クリックで展開）

### Dify

* Notion / Slack / Drive / GitHub / Backlog などとの接続が可能で、Slack botとしての運用もできる
* AIで回答するだけでなく、Backlogへの自動起票のような **アクションのカスタマイズも可能**
* Bedrock経由でモデル指定もできる
* 検証時点で試してみたところ、Notion連携の設定自体は簡単だが、まともに使えるレベルにするには相応のチューニングが必要だった
  + 「Notionに書いてあるのに『分かりません』と返ってくる」「古い手順書を優先して回答してしまう」といった事象が起きる
    - これを回避するには **チャンク分割の最適化、古いドキュメントのクリーニング、ハイブリッド検索の比率調整** といったRAGのチューニング知見が必要
* 社内に利用事例がないため、初期学習コストが見込まれる
* 長期運用を前提に腰を据えてカスタマイズするなら有力。ただし「片手間で立ち上げる」用途には向かない

### Amazon Q Business

* Slack / Notion / GitHubのコネクタは標準で揃っており、連携は容易
* 一方で、BacklogやGoogle Workspaceにはコネクタがなく、その場合は別途カスタマイズが必要
* 検証時点では、料金が問い合わせbot用途には厳しい構成だった
  + **botを呼び出すエンドユーザー全員分のライセンスが必要**
  + 加えてベクトルDBのユニット課金が発生。データソースが増えれば、さらに増える
  + ※ 料金体系は変動する可能性があるため、最新情報は[公式ページ](https://aws.amazon.com/jp/q/business/pricing/)を参照

### Notion AI Agent

* Notionのドキュメントだけでなく、**GitHub・メール・カレンダー・Figma・Slack** とも接続可能。
* 高度なチューニングをせずとも、Notion・Slackをベースに一定精度で回答してくれる
* 検索ロジックはブラックボックスでカスタマイズできないが、データソースを絞り込むことでノイズを下げる、というアプローチは取れる
* 全社でNotion活用が進んでおり、エンジニア部内でNotion AIによる問い合わせ対応の事例もすでにあったため、**学習・展開コストが他候補より明確に低い**
* ベータ版なので、料金体系が今後どう変わるかは不明

### Claude Code

* MCP経由でNotion / GitHub / Backlog などと連携可能
* DifyのようにGUI/ノーコードで組むのではなく、**LambdaとSlack Appを組み合わせて実装する必要があり、初期工数は重い**
* 社内でClaude Codeの利用が標準化されつつあり、ナレッジを活かせる期待があった
* セキュリティ面ではBedrock経由ならデータがモデル学習に使われず、IAMの権限範囲で運用できる
* 一方、検証時点では **セッションを保持できず対話用途に向かない**、応答に5分以上かかる、Notionドキュメントのピック精度がNotion AI Agentより劣る、といった課題が見えていた

### 結論：Notion AI Agent の利用を決断

総合的に比較した結果、**Notion AI Agent の利用を決断しました**。  
決め手は次の3点です。

1. SREのナレッジがすでにNotionにあり、問い合わせ履歴の保持もNotion DBで完結する
2. 実装・運用保守コストが低く、少ない工数で「やりたいこと」が実現できる
3. ベータ版のためコスト可視化と上限設定は必要だが、初期検証としては十分

## 🛠️ SRE調査兵団の設計

採用したNotion AI Agentに「**SRE調査兵団**」という社内向けの愛称を付けました。  
Slackのリアクションをトリガーにした非同期UXとし、発言にリアクションを付けるだけでエージェントが起動するようにしています。

### 全体アーキテクチャ

ざっくりとした全体像はこうです。

### 3種類のリアクションと役割

用意したリアクションは3種類です。

| リアクション | 用途 | 主な処理 |
| --- | --- | --- |
| ナレッジDB検索 | 軽く類似事例を探したい | ナレッジDBを検索し、結果を返信 |
| 問い合わせ受付・調査 | 受付・調査・回答・記録まで進めたい | 履歴DBに起票/更新 → 横断検索 → 結論・根拠・方針を返信 |
| 完了 | 調査の決着 | ステータスを完了に更新し、要約を残す |

実運用では、それぞれに専用のSlackカスタム絵文字を割り当てています。

### リアクションの運用フロー

基本の運用フローは「**まずナレッジDB検索 → 情報が不足しているなら問い合わせ受付・調査 → 解決したら完了**」というシンプルな構造です。

完了時の情報は履歴DBに蓄積され、次の問い合わせでの回答精度を向上させる設計です。  
問い合わせから記録までを、リアクションを押すだけで一気通貫に進められる形を狙いました。

## 🔍 回答精度を上げるための「検査のループ」

動き始めてからは、回答精度の向上が焦点になりました。  
やっていたのは、人がレビュー役に回るシンプルなフィードバックループです。

AIが回答したあと、人が正否を判断し、

* 正しければ「このまま進めてください」と返す
* 違えば「この部分はこう解釈してください」と返す

という形で、毎回スレッドにフィードバックを残します。  
これが履歴DBに蓄積されることで、次の問い合わせの参照元になる。つまり「精度を上げる行為」と「ナレッジを蓄積する行為」を、同じ動作で行えるようにする設計です。

並行して、データソース側のチューニング（参照Notionページの追加、Webアクセスの有効化、特定URLの許可追加）も実施しました。検索ロジックそのものをカスタマイズできない以上、**「何を見せるか」で精度をコントロールする** のが、Notion AI Agent でできる現実的な精度向上策でした。

## 📉 実運用で見えてきた「Notion AI Agentでは届かない領域」

チーム宛メンションが発生した際にSRE調査兵団を使ってみる、という形で実運用を始めました。

過去のSlackやり取りを横断的に参照しないと答えられないような問い合わせに、AIだけで回答できるケースも出てきました。狙っていた「専用エージェントだから出せる価値」は、一定機能している実感がありました。

ただ、運用を回しているうちに、Notion AI Agentでできることの限界も同時に見えてきました。

* できるのは、対応策の **「提案」やコード例の「提示」** まで
* 実際に修正コードを書き、コミットし、PRを出すといった作業は、結局人が手を動かす必要がある
* つまり「問い合わせに **答える**」まではカバーできたが、「問い合わせを **直す**」までは届かない

「SRE業務をAIで再設計する」という当初の目的に照らすと、提案までで止まるエージェントは、本当にゴールに対して足りているのか？という疑問が湧いてきました。  
問い合わせ対応の本当のゴールは「相手の困りごとが解消されること」であって、「対応策が提示されること」ではないからです。

運用していて、こんな点も気になり始めました。

* 一般論やベストプラクティス系の質問は、専用エージェントを介さずAIに直接聞いた方が早い
* リアクションを押してから返信が返るまでの待機時間が、チャットでAIに直接聞く速度感に比べて遅い

「専用エージェントだから出せる価値」は確かにある。でも、その上限が見え始めていた。それがこのフェーズでした。

## 🛑 停止の判断 ―「答える」から「直す」までAIに任せる時代へ

実運用で見えてきた上限に加えて、もう一つ大きかったのが Claude Code が「実行までできるAI基盤」として社内で育ってきたことです。振り返りで、こんな議論が出ました。

> 調査兵団は停止でいいかもしれない。  
> Claude本体が Slack / Notion / GitHub / AWS / Google Cloud / Datadog にアクセスできるようになり、Claude Code を中心に業務を回せる。修正対応まで含めるなら、Notion AI Agent ではどうしても足りない。

検討開始時点と停止判断時点では、前提が大きく変わっていました。  
AIから接続できるデータソースは Notion 中心から各種SaaS・クラウドへ広がり、AIに任せられる範囲も「提案・コード例の提示」から「修正コードの作成・実行」まで踏み込めるようになっています。  
Claude Code 自体も社内で日常的に使われる共通基盤になりつつあり、SRE業務の実装先としても、Notion AI Agent のプロンプトより Claude Code の plugin / skill のほうが筋が良くなってきていました。

検討開始時点で「最もコスパの良い選択肢」と判断していた前提が、3ヶ月で大きく変わった形です。実運用で感じていた「提案で止まる違和感」も、要は"実行まで"やれる時代に、提案で止まる仕組みを使っていたから、ということでした。

とはいえ、Notion AI Agent が不要になったわけではありません。開発者以外が Slack から気軽に問い合わせる用途では、提案・案内までで十分なことも多く、手軽さという強みは今も変わりません。今回の停止は、用途の切り分け直しに近いです。

## 🔁 次のフェーズ：Claude Code の skill として再実装

SRE調査兵団が担っていた機能は、Claude Code の skill として再実装する方向に動いています。狙いは次の3点です。

1. 個別のAIエージェントを作り込むのではなく、Claude Code を共通基盤として plugin / skill を積み上げていく
2. 「問い合わせの受付・調査」だけでなく、修正対応・実行までを同じ基盤の上で完結させる
3. SREだけでなく、エンジニア組織全体で再利用できる形に寄せていく

Claude Code であれば、skill から実際のコード変更・PR作成・運用作業までAIに任せられます。  
「業務をAIで再設計する」という当初の目的に、Notion AI Agent より明確に近づける。これが、次のフェーズに進む理由です。

## ✍️ まとめ：作って捨てた3ヶ月で得たもの

短期間で立ち上げて短期間で停止、と並べると「失敗」に見えるかもしれません。ただ、振り返ってみると得たものは多かったと思います。

* **現状プロセスの明示化と To-Be 像の定義**  
  SRE調査兵団を作る過程で、どこをAIに任せ、どこを人が担うかを整理できた
* **AIエージェント選定の評価軸**  
  複数のサービスを同じ軸で比較するフレームは、別領域での選定にも使い回せる
* **「いつ作り、いつ捨てるか」の意思決定**  
  短いサイクルで方針を切り替えられたこと自体が、チームの動き方として残った

**小さく作って小さく捨てられる状態を保つこと** が、変化の早いAI領域では結局いちばん続けやすいやり方だと思います。

SRE調査兵団は停止しますが、「SRE業務をAIで再設計したい」という方向性は変わりません。Claude Code を共通基盤に skill / plugin を積み上げる次のフェーズを、引き続き進めていきます。
