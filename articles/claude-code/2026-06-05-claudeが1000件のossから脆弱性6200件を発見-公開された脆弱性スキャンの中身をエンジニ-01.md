1,000以上のオープンソースプロジェクト。  
検出された脆弱性は23,019件。  
そのうち high か critical と推定されるものが6,202件。

Anthropic が2026年4月に始めた [Project Glasswing の初期報告](https://www.anthropic.com/research/glasswing-initial-update)で、この数字が出ました。  
同時に、そのスキャンの土台になったフレームワーク [defending-code reference harness](https://github.com/anthropics/defending-code-reference-harness) が GitHub で公開されています。

「Claude が脆弱性を見つける」という話は、これまでどこか他人事でした。  
でも実物のツールが手元に降りてくると、急に自分の現実になります。

この記事では、まず数字の正確な意味を整理して、公開されたツールの構造と前提条件を読み解きます。  
そのうえで、個人やチームが本当に使えるのかを、誇張なしで評価します。

  

## まず数字を正確に読む

ニュースの見出しは「1万件超の脆弱性」と書きます。  
ですが数字は何種類もあって、混同すると意味を取り違えます。  
最初にここを整理します。

* スキャンしたOSSプロジェクト: 1,000以上
* 検出した脆弱性の総数: 23,019件（medium・low を含む）
* うち high か critical と推定: 6,202件
* 第三者のセキュリティ企業が検証した high/critical 候補: 1,752件
* 検証の結果、genuine（本物）と確認: 1,587件
* 実際に high か critical と確認: 1,094件

一番大事なのは、検証に回した1,752件のうち**1,587件（90.6%）が本物の脆弱性だった**という点です。

AI が出した脆弱性レポートは「どうせ誤検知だらけだろう」と思われがちです。  
ところが独立した第三者が確かめて、9割が実在する脆弱性でした。

一方で、実際に high か critical だったのは62.4%です。  
重大度の見積もりはやや盛り気味、でも脆弱性が存在すること自体はほぼ正しい。  
これがこの数字の正直な読み方です。

スキャンには [Claude Mythos Preview](https://www.anthropic.com/glasswing) という高性能モデルが使われています。  
参加したのは AWS・Apple・Cisco・Google・Microsoft・NVIDIA・Palo Alto Networks など約50の組織です。  
個人の実験ではなく、業界の主要プレイヤーが束になった取り組みだと押さえておけば十分です。

## ボトルネックは「発見」から「修正」に移った

今回の報告でAnthropicが一番言いたいのは、数字そのものではありません。

[公式の表現](https://www.anthropic.com/research/glasswing-initial-update)はこうです。

> これまでソフトウェアセキュリティの進歩は、いかに速く新しい脆弱性を見つけられるかで律速されていた。今は、見つかった脆弱性をいかに速く検証・開示・修正できるかで律速される。

つまり、ボトルネックが**発見から修正へ移った**という主張です。

これを自分の仕事に引き寄せると、かなり生々しい話になります。

これまでセキュリティ対応は「見つける人が足りない」前提で組まれていました。  
脆弱性は人間が地道に探すもので、数は限られていました。

そこにAIが入ると、脆弱性は一晩で数千件出てきます。  
すると詰まるのは、見つける工程ではなく、その後の検証・トリアージ・パッチ適用です。

CVE対応の工数配分も、レビューの優先順位の付け方も、ここから変わっていきます。  
「探す」より「捌く」が仕事になる、と読むのが自然です。

## 公開された harness の構造

公開された defending-code reference harness は、脆弱性を見つけて直すまでを7段階のパイプラインにしたものです。

Build → Recon → Find → Verify → Dedupe → Report → Patch、という流れで進みます。

ビルドして、対象を偵察し、脆弱性を探し、再現を検証し、重複を排除し、レポートにまとめ、最後にパッチ案を出す。  
人間のセキュリティエンジニアがやる作業を、そのまま工程に分解した構成です。

操作は Claude Code のスキルコマンドで行います。

* **/quickstart**: 初期設定とガイド実行
* **/threat-model**: 対象の脅威モデルを構築
* **/vuln-scan**: 静的な脆弱性スキャン
* **/triage**: 検出結果の確認・重複排除・優先順位付け
* **/patch**: 修正案の生成と検証
* **/customize**: パイプラインを自分のターゲットに適応させる

デフォルトで対象にしているのは、C/C++ のメモリ系脆弱性です。  
バッファオーバーフローや Use-After-Free を、**ASAN**（AddressSanitizer）で検出する作りになっています。

実行時のセキュリティもしっかり設計されています。  
エージェントは **gVisor** のサンドボックス内で動き、外部通信は Claude API だけに制限されます。  
脆弱性を探すコードを動かす以上、隔離は必須という思想です。

## 大量の脆弱性レポートは、誰が直すのか

このニュースには、見つける側とは別の緊張があります。  
脆弱性を大量に受け取る、OSSメンテナー側の負担です。

AIが一晩で数千件の脆弱性を出せるようになると、レポートを受け取る側が追いつきません。  
検証して、本物かを見極めて、パッチを当てて、リリースする。  
この工程は、結局のところ人間とコミュニティの時間で動いています。

Glasswing が示した90.6%という精度は心強い数字です。  
それでも、残りは誤検知で、メンテナーはその切り分けにも時間を使います。  
善意で投げられた大量のレポートが、小規模なOSSプロジェクトを疲弊させる、という構図は現実に起こり得ます。

「AIで脆弱性が見つかる時代になった」の裏には、「見つかった後を誰が引き受けるのか」という問いが必ずついてきます。  
ここを無視して発見数だけを礼賛するのは、エンジニアとしては片手落ちだと思います。

## 本番がほしいなら Claude Security が別にある

harness を読み解いて物足りなさを感じた人向けに、補足しておきます。

今回公開されたのは、あくまで参照実装です。  
複数プロジェクトを継続的に回したり、検出結果のライフサイクルを管理したりする本番運用は、想定されていません。

そうした用途には、Anthropic が **Claude Security** というマネージド製品を別に用意しています。  
harness とこの製品は別物なので、混同しないようにしておくと話が整理できます。

手元で仕組みを学ぶなら harness、本番で回すなら製品版、という住み分けです。

## まとめ

今回のポイントを振り返ります。

* Glasswing は1,000以上のOSSをスキャンし、6,200件超の高深刻度脆弱性を見つけた
* 第三者検証で9割が本物と確認され、AIの脆弱性発見は実用段階に入った
* ボトルネックは発見から、検証・修正へ移った
* 公開された harness は7段階パイプラインの参照実装
* デフォルトは C/C++ 限定で、本番運用には別途 Claude Security がある

エンジニアとして今取るべき行動は、シンプルです。

C/C++ の資産を持つチームなら、[harness](https://github.com/anthropics/defending-code-reference-harness)を読んで自分たちのパイプラインの設計図にする。  
そうでないなら、まずは「脆弱性は捌く時代になった」という前提だけ頭に入れておく。

AIが脆弱性を見つける話は、もう未来の話ではありません。  
見つかった後の検証と修正を、自分の現場でどう回すか。  
そこを考え始めるのに、ちょうどいいタイミングだと思います。
