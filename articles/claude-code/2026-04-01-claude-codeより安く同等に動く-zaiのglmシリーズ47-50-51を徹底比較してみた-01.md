最近、コーディングエージェント界隈でひそかに注目を集めているのが、[中国Z.ai](http://xn--Z-kq6ay5z.ai)（旧ZhipuAI）が開発するGLMシリーズです。

「Claude Opus 4.6の94.6%の性能を、もっと安いコストで出せる」という話が開発者コミュニティに広まり、OpenClawやClaude Codeとの組み合わせで使う人が増えています。

この記事では、**GLM-4.7 → GLM-5.0（GLM-5）→ GLM-5.1** の3世代の違いを整理し、実際にどう使うかまでを解説します。AIツールをよく使う方、コーディングエージェントに興味がある方を想定して書いています。

## GLMシリーズって何？ざっくり概要

[GLMはZ.ai](http://xn--GLMZ-dd4c.ai)（zai-org）が開発するオープンソース寄りの大規模言語モデル（LLM）です。以前は学術・研究寄りのイメージが強かったですが、GLM-4世代あたりから一気にコーディング・エージェントに本気で取り組み始め、2025〜2026年にかけて急激に性能が向上しました。

特に注目すべきは **Agentic（長期自律エージェント）性能** で、「Vending Bench 2」というエージェントベンチマークではオープンソースモデル1位を獲得。Claude Opus 4.6に肉薄するところまで来ています。

しかも重み（weights）が公開されているため、APIだけでなくローカル実行も可能。Claude系モデルと異なり、商用利用の柔軟性が高い点も魅力です。

## 何がどう良くなったか？

### GLM-4.7 → GLM-5.0 ：最大のジャンプ

この世代間の進化が一番大きいです。性能面で定量的に比較すると：

* **SWE-bench Verified**（実際のGitHub issueを自動修正するベンチ）：73.8% → 77.8%
* **Vending Bench 2**（長期エージェントタスク）：オープンソース1位
* **BrowseComp・Terminal-Bench** でも明確な改善

ただし数値より重要なのが**タスクの質的変化**です。GLM-4.7が「コードを書ける」モデルだとすると、GLM-5.0は「複雑なシステムを長期間かけて自律的に構築できる」モデルに近づいています。

バックエンドAPIとフロントエンドUIを同時に設計・実装して、エラーを自己修正しながら完成まで持っていく——そういった「Agentic Engineering」が格段に上手くなりました。

また、非同期RL（強化学習）基盤 **「slime」** を導入したことで、ポストトレーニング（学習後の微調整フェーズ）が効率化され、トレーニングコスト対効果が高まっています。

### GLM-5.0 → GLM-5.1 ：Coding Agentへの最適化

GLM-5.1はGLM-5の性能をベースに、**Claude CodeやOpenClawなどのCoding Agentとの統合に特化して最適化**されたバージョンです。

主な変化点：

* コンテキスト長が204,800 tokens、出力最大131,072 tokensと明記・安定化
* Claude Code / OpenClaw / Cursor / Cline / Kilo Code など主要エージェントツールとの互換性を確認・最適化済み
* Reasoningモード（推論ステップを明示的に踏むモード）に対応
* GLM-4.7をデフォルトモデルから置き換えられるよう設計

実際のClaude Codeを使ったコーディング評価では **45.3点（Claude Opus 4.6は47.9点）**、つまりOpusの94.6%相当というスコアが出ており、コーディング用途であればほぼ遜色ない性能を発揮します。

デメリットとして、**画像入力が非対応**になっています。マルチモーダルな用途（スクリーンショットを読ませてUIを直す、など）にはGLM-5.0を使う必要があります。また一部ユーザーから「長コンテキスト時に出力が不安定になる」「5.1特有の奇妙な出力が出ることがある」という報告もあります。

## 実際にどう使うか：Claude Code & OpenClawへの設定方法

ここが多くの人が知りたいポイントだと思います。GLM-5.1はClaude CodeやOpenClawのバックエンドモデルとして設定できます。

### 前提：[Z.ai](http://Z.ai) APIキーの取得

1. [Z.ai](https://z.ai) にアカウントを作成
2. APIキーを発行（Coding Planへの加入が必要）
3. 環境変数 `ZAI\_API\_KEY` にキーを設定

### Claude CodeでGLM-5.1を使う

Claude Codeは環境変数またはsettings.jsonでモデルのマッピングを変更できます。

**方法1：settings.jsonを編集する（推奨・永続的）**

`~/.claude/settings.json` を開き、以下のように追記します：

```
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "YOUR_ZAI_API_KEY",
    "ANTHROPIC_BASE_URL": "https://api.z.ai/api/anthropic",
    "API_TIMEOUT_MS": "3000000",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "glm-4.5-air",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "glm-5.1",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "glm-5.1"
  }
}
```

**各項目の説明：**

* `ANTHROPIC\_AUTH\_TOKEN`：Z.aiのAPIキーを入力
* `ANTHROPIC\_BASE\_URL`：エンドポイントをZ.aiに切り替え
* `API\_TIMEOUT\_MS`：**重要**。タイムアウトを約50分に設定。GLM-5.1は複雑なタスクで時間がかかるため、デフォルトのタイムアウトでは処理が途中で止まることがあります
* `ANTHROPIC\_DEFAULT\_HAIKU\_MODEL`：Haiku相当として `glm-4.5-air` のままにすることをお勧め。Haiku層もGLM-5.1に変更することは一般的ではありません
* `ANTHROPIC\_DEFAULT\_SONNET\_MODEL` / `ANTHROPIC\_DEFAULT\_OPUS\_MODEL`：両方ともGLM-5.1に上書きするのが標準

**設定後の確認方法：**  
新しいターミナルで `claude` を起動して以下を実行します：

```
/status
```

または

```
/stats
```

出力に「glm-5.1」と表示されたら成功です。既存のターミナルセッションは反映されないため、**必ず新しいターミナルウィンドウ**で起動してください。

**方法2：環境変数で設定する（一時的な切り替えに便利）**

```
export ANTHROPIC_AUTH_TOKEN=your_zai_api_key
export ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic
export ANTHROPIC_DEFAULT_SONNET_MODEL=glm-5.1
export ANTHROPIC_DEFAULT_OPUS_MODEL=glm-5.1
claude
```

こちらはターミナルセッションごとの設定なので、試し使いに向いています。

### OpenClawでGLM-5.1を使う

OpenClaw（Claude Codeのオープンソース互換ゲートウェイ）を使っている場合は、以下の手順です。

**設定ファイルを編集する**

`~/.openclaw/openclaw.json` を開き、以下を追加・変更します：

```
{
  "models": {
    "providers": {
      "zai": {
        "models": [
          {
            "id": "glm-5.1",
            "name": "GLM-5.1",
            "contextWindow": 204800,
            "maxTokens": 131072
          }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "zai/glm-5.1"
      }
    }
  }
}
```

**ゲートウェイを再起動**

```
openclaw gateway restart
```

動作確認は `openclaw tui` でチャット画面を開き、GLM-5.1が表示されていればOKです。

以下は実際のユーザー報告に基づいた注意点です：

2. **長コンテキスト時に出力が不安定になることがある**
3. **画像入力非対応**
4. **プラン確認が必須**

## どれを選べばいい？シチュエーション別まとめ

判断の軸は大きく「コスト」「タスクの複雑さ」「ローカル実行が必要か」の3点です。

**GLM-4.7 を選ぶケース**

* 日常的なコーディング補助（関数生成、デバッグ、コードレビューなど）
* APIコストを最小化したい
* 多言語コーディング・翻訳など幅広いタスクをバランス良くこなしたい
* ローカル推論でVRAMを節約したい（アクティブパラメータが少ない）

**GLM-5.0 を選ぶケース**

**GLM-5.1 を選ぶケース**

* Claude CodeやOpenClawをメインのコーディングエージェントとして使っている
* Coding Planでコスト固定したい
* Claude Opus 4.6と近い性能をより安く試したい
* Reasoningモード（ステップバイステップの推論）を使いたい

## まとめ

GLMシリーズは「チャットAI」から「実務で使えるコーディングエージェント」へと着実に進化しています。特にGLM-5.1は、Claude Codeのバックエンドとして設定するだけでOpus相当の性能をずっと安く使える可能性があり、コーディングエージェントをよく使う人には試す価値があります。

一方で、「長コンテキストで不安定になることがある」「5.1は画像非対応」「設定に若干の手間がかかる」といった点も正直なデメリットとしてあります。いきなりすべて切り替えるより、まずClaude Codeの設定にGLM-5.1を追加してみて、体感で比較するのが一番おすすめです。

今後もGLMシリーズは更新が続くと思われるので、引き続き追っていきます。
