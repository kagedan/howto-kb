## 設計の話はわかった、で、実際どうなの？

[前回の記事で](https://zenn.dev/just2enough/articles/a0f050bfd25c4d)、Claude Code のセッション間引き継ぎを `docs/handover.md` でやる話を書きました。要点だけ整理すると、handover は2層構造で、**先頭は「次セッションで即再開するためのブロック」、その下が「あとで参照する決定ログ」**。決定ログには「決定」だけでなく\*\*「不採用の案」と「なぜそう決めたか（理由）」を並べて残す\*\*——というのが骨子でした。

ただ、前回は設計思想に寄せた分、**決定ログの具体例がゼロ**でした。綺麗な設計論はいくらでも後付けできます。なので今回は、実際に運用しているデスクトップアプリ[Petatto.md](https://just2enough.github.io/petatto-md-site/)（Obsidian の md を Windows のデスクトップに付箋として貼る個人プロジェクト）の handover から、決定ログの実物をそのまま引っ張ってきます。

## 決定ログ、実物はこう見える

まずは1件、実際のエントリを貼ります。マウント遅延する vault（GoogleDrive 上の Obsidian フォルダなど）で起動すると付箋が立ち上がらない、というバグを直したときの決定です。

細部は読み飛ばして、**型**を見てください。番号・日付・種別、そして「決定」「理由」「詳細（コミット）」。理由欄が泥臭くて、「ログが無い別 PC のバグを、自分の PC で vault フォルダを起動前にリネームして無理やり再現した」みたいな、コミットメッセージには書ききらない経緯がここに記録されています。

```
### 140 2026-06-12: マウント遅延 vault の再走査で付箋を開けるよう修正（issue #20）

- 種別: 設計判断（バグ修正。spec §6.3 未反映＝要追記）
- 決定: GoogleDrive 等のマウント遅延で起動時に vault 不在だと付箋が立ち上がらず、watcher も起動失敗のまま残る不具合を修正。新 IPC `rescan_vault` で、起動時と同じ突合＋付箋ウィンドウ生成を行う再走査経路を新設
- 理由: watcher 起動失敗は実機ログで確認。別PC発生でログの無い issue を、本PCで vault フォルダを起動前に退避リネームし「起動時に不在→後から出現」を決定的に再現して特定した
- 詳細: コミット `b8a8841`
```

このエントリの「理由」が後日どう使われたか、を少しだけ。決定欄に何気なく書いた\*\*「watcher も起動失敗のまま残る」\*\*の一行が、翌セッションの起点になりました。コードレビューで「稼働中に vault が一瞬 unmount→remount されると watcher は死んでいるのに『稼働中』と誤判定し、再走査でも監視を回復できない」という、より深い穴が見つかった（No.141）。さらにその次、再走査スレッドが裏で静かに失敗する経路を表に出す対処（No.142）へとつながった。理由欄は「なぜそうしたか」の言い訳であると同時に、**次に掘る場所のしおり**になっていた、という話です。

## ハイライト①：覆った決定こそ資産になる

ここが本記事の山場です。「ロックされたファイルへの書き込みをどう扱うか」という、地味な一点について、方針が2回ひっくり返りました。3つのエントリを時系列で並べます。

付箋化のとき、Obsidian で開きっぱなしのノートだと、ファイルの原子的置換が一過性のロックに弾かれて失敗する。最初はこれを「リトライで救う」と決めました。

```
### 120 2026-06-05: Obsidian の一過性ロックで付箋化が落ちる問題を段階バックオフ・リトライで救済

- 種別: 設計判断（データ保全 / UX）
- 決定: vault 内 md への `write_atomic` を `commands.rs::write_atomic_vault` でラップし、ロック系失敗を段階バックオフ（50/120/250ms・最大3回）でリトライ。枯渇時のみ新 i18n 文言 `error.file_locked_by_other_app` を返す。in-place フォールバック案は不採用
- 理由: Obsidian で開いたままのノートを新規付箋化すると `ReplaceFileW` の元ファイル削除段が Obsidian の一過性ロックにぶつかり失敗していた。Obsidian は短時間でロックを離すため段階バックオフで実用上ほぼ吸収（実機の反復操作で再現せず確認）
- 詳細: 罠は `trap.md`
```

ところが、改めてコードレビューをしてみたら、このリトライが二重に筋が悪いと分かった。で、**真逆に倒しました**。リトライを消して、即座に失敗を返す（fail-fast）。

```
### 125 2026-06-05: ロック救済のリトライを廃止し fail-fast＋明確メッセージへ転換（No.120/121 を改める。結合レビュー #1-3）

- 種別: 設計判断（データ保全 / UX。No.120 / No.121 の方針転換）
- 決定: `write_atomic_vault` の段階バックオフ・リトライ（No.120）を撤去。ロック時は即 `error.file_locked_by_other_app` を返す fail-fast に。フロントは握りつぶしていたロック文言をトーストで表示
- 理由: 結合レビューで、リトライは ①持続ロック（Obsidian 開きっぱなし＝通常）では全段失敗で待つだけ無駄 ②一過性ロック成功時は backoff 中に挟まった他アプリ保存を古いスナップショットで上書き（クロバー＝データ消失）の二重苦と判明。失敗時メッセージで「閉じて再試行」を促せば一過性ロックは手動の一拍リトライで吸収でき、リトライの実益は薄い
- 詳細: コミット `5a17d38`。副次で `thread::sleep` 撤去によりワーカースレッド占有も解消
```

ここで普通なら話は終わりです。「リトライを入れて、消した」。第三者が見たら「最初からそうしろよ」案件ですね💦けれど数日運用して、**No.125 の前提そのものが崩れました**。Obsidian を閉じていても、インデクサや Defender がファイルを一瞬掴んで同じ失敗が出る。この「Obsidianを閉じている側」のケースでは、No.125 が挙げた却下理由（①②）がどちらも成り立たない。で、3度目の方針転換。

```
### 132 2026-06-09: ロック誤検知を「クロバー安全リトライ」で救済（No.125 を改める。前提が変わったため）

- 種別: 設計判断（データ保全 / UX。No.125 のリトライ撤去を改める）
- 決定: `write_atomic_vault` はロックを `VaultWriteError::Locked` として型で返すだけにし、新ヘルパ `write_with_lock_retry` がロック時のみ backoff（旧 No.120 と同値）でリトライする。各リトライは sleep 後・書き込み直前に再 read → ハッシュ照合し、不変なら書き直し／変化なら書かずに中断（Conflict ＝クロバー回避）
- 理由: No.125（fail-fast）の前提が変わった。No.125 の却下根拠（①持続ロックでは待つだけ無駄 ②リトライ成功＝Obsidian 保存をクロバー）は全て「Obsidian が開いている」前提だった。だが運用中、Obsidian を閉じていてもスキャナが一瞬ファイルを掴み失敗する誤検知が頻発（実機ログで確認）。この閉側ケースでは①は短リトライで成功し②はクロバー相手が居ないため、fail-fast は純粋な誤検知。採用した「毎回再 read 再検証リトライ」は No.125 の検討時に選択肢として既出だが「複雑化」で見送られていた。着手前に決定ログを読み直し既出と確認した
- 詳細: `commands.rs` / `atomic_write.rs`。Obsidian 開/閉の両方で実機検証
```

注目してほしいのは No.132 の理由欄です。**「No.125 の却下根拠①②は全て『Obsidian が開いている』前提だった」**と、ひっくり返す相手の理由を名指しで引いている。これができたのは、**No.125 を消さずに残してあったから**です。もし「fail-fast にしたから古いリトライの記録は要らない」と消していたら、No.132 のとき、また一から「リトライ vs fail-fast、どっちが正しい？」をゼロから議論し直していました。実際にはそうならず、**「前提のどこが変わったか」だけを一点、差分で書けば済んだ**。

しかも理由欄の最後の一行「採用案は No.125 の検討時に既出だが複雑化で見送られていた。着手前に決定ログを読み直し既出と確認した」。これは前回書いた「**同じ機構を3回直すなら、着手前に履歴を1回探す**」をそのまま実演しています。覆った決定が消されずに積まれているから、3度目の自分が「これ昔却下したやつでは？」に自分で気づけた。**覆った決定こそ資産になる**、の実物です。

## ハイライト②：仕組みが拾った「繰り返すミス」

先に身も蓋もないことを言っておくと、ここで出すのは**私（と Claude）が何度も同じミスを繰り返した記録**です。ただ、retrospective には「再発カウンタ」という機構が組み込んであって、**そもそも『人間も Claude もミスを繰り返す』ことを前提に設計されています**。

具体例。私の開発環境は Windows で、Bash と PowerShell を行き来します。この2つは作業ディレクトリ（cwd）を共有しないので、Bash 側で相対パスのまま `cargo test` を投げると、想定と違うフォルダで走って失敗する。これを**何度もやりました**。retrospective の「悪かった」欄に、`(05-27) cwd 不一致を3回目踏んだ` と再発回数つきで記録され、翌日また `(05-28) また躓いた` が積まれ、最終的に「観測5回超」まで数えられました。

フェーズが一区切りした時点で、retrospective を蒸留して恒久ルールに昇格させる運用（前回書いた還流ハーネス）が回ります。このとき cwd 問題は、再発回数の多さから真っ先に拾われ、`~/.claude/rules/` に「cargo/git/npm を Bash で投げるときは無条件で絶対パス」というルールとして昇格しました（決定 No.138）。

……で、ここからが正直枠です。**翌セッションで、このルールを外しました**（決定 No.139）。

理由は、昇格基準に razor をかけ直したからです。「**昇格していいのは、黙って壊れる・遅れて現れる失敗だけ。即座にエラーで気づけて、やり直しも安いミスはルールにしない**」。cwd 不一致は、間違えた瞬間に "No such file or directory" で即コケる。観測した5回とも即時エラーで、毎回その場で絶対パスに直して1往復で復旧していた。つまり\*\*「黙って壊れる」側ではなかった\*\*。CLAUDE.md や rules/等のルール枠は貴重なコンテキストであり「注意資源」で、増やすほど一つひとつが守られなくなります。だから「やり直しが1往復で済むミス」のためにルールは足さない、と判断して、昇格を**巻き戻した**わけです。

その結果、cwd の再発は止まっていません。でもそれでいい、というのがこの節の結論です。**仕組みは「ミスを数えて昇格させる」だけでなく、「昇格させすぎを巻き戻す」方向にも作動した**。拾うことと同じくらい、捨てることをちゃんとやれたかどうかを見ています。

## 正直枠：運用のひっかかり・うまくいっていない点

ここまで「仕組みが役立った」良い話に寄ったので、運用してみて「これは仕組みでは消せないな」と諦めているひっかかりを2つ。ここから先は**引用がありません**。どちらも handover をいくら読み返しても出てこない、私の記憶だけにある類の話だからです。

**1つ目は、書くコストは毎回かかるのに、報われるのは時々だということ。** 決定ログは、4〜5行を書く手間が毎セッション確実に発生します。一方それが効くのは「後で読み返したとき」だけで、その場面は数セッションに1回あるかどうか。ハイライト①・②で見せた「効いた瞬間」の裏には、**一度も読み返されないまま終わった大量のエントリ**があります。正直、律儀に理由まで書いている最中は「これ、今わざわざ書く意味ある？」という気持ちと毎回小競り合いしています。ROI が遅延して目に見えないからです。実際、疲れたセッション終了時に handover 更新がなかなか終わらず、待ち時間が発生する、は起きます。handoverを更新してくれるのはClaudeCodeですが、ここは消えません。コストの実感は、払った本人にしか見えないからです。

**2つ目は、Claude は放っておくと盛ること。** これは前記事の「Claude って長文書くの大好き」の実務版です。「1決定4〜5行」というルールを置いても、handover 更新を Claude に任せると、副次効果も経緯も補足も全部書こうとして、1エントリがすぐ10行15行に育ちます。決定ログ運用の体感、その半分くらいは**この饒舌を刈り込む作業**かもしれません。皮肉なもので、肥大化を防ぐために作った仕組みの運用そのものが、肥大化との手作業の格闘になっている。ルールは「上限はここ」と示すだけで、上限まで削るのは結局こちらの手。ここを自動化しきれていないのが、地味にだるい所です。

この2つは、skill にブレーキを足せば消える類ではありません。書くコストの遅延 ROI も、盛り癖の刈り込みも、最後は運用者の手作業に残ります。仕組みで全部は殺しきれないなぁ、というのが運用してみての実感です。

## まとめ：実例から言える一般則を一つだけ

実例を並べて、はっきりしたことが1つあります。**決定ログは、書いた瞬間より、読み返した瞬間に価値が出る。**

No.132 が「前提が変わった」を一点の差分で書けたのは、No.125 を**読み返せた**から。3度目の自分が「これ昔却下したやつでは」に気づけたのも、過去の自分の理由を**読みに行った**から。書いた瞬間にはどれもただの記録で、効いたのは全部「後で読まれたとき」でした。

だとすると、投資すべきは**書く労力ではなく、読まれる導線と、肥大化対策**です。前記事で書いた「2層構造」「1決定4〜5行」「3件を超えたらアーカイブ」「再発カウンタと razor」——あれは全部、決定ログを**読み返せる状態に保つ**ための仕掛けでした。山積みの300行は誰も読み返さない。読み返されないログは、書いた瞬間の自己満足で終わります。とにかくClaudeに読ませる情報を肥大化させないことが大事。

綺麗な決定ログを書くことより、**未来の自分（と Claude）が、最小の文脈でそれを読み返せること**。実例から言える一般則は、結局それだけかなと思っています。
