# はじめに

最近は、AIに指示を出すだけでコードを書いてくれる時代になっています。

たとえば、

* 画面を作る
* エラーを直す
* APIを作る
* テストを書く
* 既存コードを読んで修正する

といった作業を、AIコーディングエージェントに任せられるようになっています。

「AIを使うのはズルい」とか「AIを使わずに全部自分で書くべき」という考え方もありますが、実際の開発現場では、AIを使って速く作り、AIが出したコードを理解し、レビューし、自分の責任で出せることが重要になっています。  
これからのエンジニアは AIを使えることが前提 になっていきます。

AIを使わないことが偉いのではありません。

AIを正しく使い、その結果に責任を持てること が大事です。

この記事では、初心者がAIコーディングエージェントを使うときに知っておくべき考え方と、具体的な使い方を整理します。

⸻

# AIコーディングエージェントとは

AIコーディングエージェントとは、開発者の指示をもとに、コードを書いたり、修正したり、ファイルを読んだり、コマンドを実行したりしてくれるAIツールです。

普通のチャットAIと違うのは、単にコードを提案するだけではなく、実際のプロジェクトの中で作業してくれることです。

たとえば、

ログイン画面を作ってください。  
このエラーの原因を調べて修正してください。  
プロフィール編集機能を追加してください。

のように指示すると、AIがプロジェクト内のファイルを読み、必要なファイルを編集しながら作業してくれます。

小さなアプリであれば、設計、実装、修正、テストのかなりの部分をAIに手伝ってもらえます。

ただし、ここで大事なのは、

AIが作業できることと、AIが責任を持てることは別

ということです。

⸻

# なぜコードを勉強するのか

AIがコードを書けるなら、なぜ私たちはコードを勉強するのでしょうか。

理由はシンプルです。

AIはコードに責任を持てないからです。

AIが書いたコードでエラーが起きても、AIは謝ってくれません。

そのコードを使った人に迷惑がかかったとき、説明するのはAIではありません。

そのコードを確認し、採用し、プルリクエストを出した人です。

つまり、AIに指示したあなたです。

だから、AIが書いたコードであっても、

* 何を変更したのか
* なぜその実装になったのか
* 仕様を満たしているのか
* 危ない処理が入っていないか
* エラーが起きたときに説明できるか

を理解する必要があります。

企業は「AIに指示を出すだけの人」にお金を払っているわけではありません。

AIが出したコードを理解し、レビューし、自分の責任で出せる人 にお金を払っています。

だから、私たちはコードを勉強します。

AIを使わないためではありません。

AIを使いこなすために、コードを勉強します。

## AIに守ってほしいルールを書く

AIコーディングエージェントを使うときは、最初にプロジェクトのルールをAIに伝える必要があります。

そのために使うのが、AGENTS.md や CLAUDE.md、agents.md のようなファイルです。

ツールによって、読むファイル名は少し違います。

ツール 代表的なルールファイル  
Codex AGENTS.md  
Claude Code CLAUDE.md  
Antigravity agents.md  
Cursor Rules / project rules 系の設定

この講座では Antigravity を使うため、まずはプロジェクトの中に agents.md を作る方針で進めます。

役割としてはシンプルです。

AIに、このプロジェクトで守ってほしいことを伝えるファイル

です。

⸻

agents.md に書くべきこと

初心者向けには、まず安全面のルールを書くのが大事です。

たとえば、次のように書きます。

### AIへの指示

* 勝手に git commit しないでください。
* 勝手に git push しないでください。
* 勝手にデプロイしないでください。
* .env ファイルは編集しないでください。
* 秘密情報やAPIキーをコードに直接書かないでください。
* 大きな変更をする前に、まず変更方針を説明してください。
* 実装後は、何を変更したのかを説明してください。
* 判断に迷う場合は、勝手に進めず質問してください。

ここで大事なのは、agents.md に何でも書きすぎないことです。

毎回必ず読んでほしい最低限のルールだけを書くのがよいです。

アプリの詳しい設計、DB構造、API仕様などを全部 agents.md に書くと、長くなりすぎて扱いづらくなります。

そのため、詳しい情報は docs/ に分けます。

⸻

### 詳しい情報は docs/ に分ける

アプリの詳しい設計や仕様は、docs/ に分けておくと扱いやすいです。

たとえば、次のようにします。

docs/  
architecture.md # 全体設計  
database.md # DB構造  
api.md # API仕様  
ui.md # 画面仕様  
coding-rules.md # 実装ルール

そして、agents.md には次のように書きます。

### 参照するドキュメント

実装前に、必要に応じて以下のドキュメントを確認してください。

* 全体設計: docs/architecture.md
* DB構造: docs/database.md
* API仕様: docs/api.md
* 画面仕様: docs/ui.md
* 実装ルール: docs/coding-rules.md

こうすると、AIに必要な情報を適宜読ませながら作業できます。

## Planning mode を使う

初心者がAIコーディングエージェントを使うときは、いきなり実装させるのではなく、まず計画を出させるのがおすすめです。

Antigravityには Planning mode があります。

Planning modeでは、AIがすぐにコードを書き始めるのではなく、まず実装方針や作業リストを出してくれます。

初心者にとっては、これがかなり大事です。

なぜなら、AIが何をしようとしているのかを確認してから実装に進めるからです。

⸻

Planning mode の使い方

基本の流れは、次の通りです。

1. やりたいことを日本語で書く

ログイン後に、ユーザーのプロフィールを表示するページを作りたいです。  
名前、メールアドレス、自己紹介文を表示してください。

2. まず実装方針を出してもらう

実装する前に、どのファイルを変更するか、どのような方針で進めるか説明してください。

3. Planning mode で計画を確認する

AIが Implementation Plan や Task List のような形で、作業方針を出してくれます。

ここで、いきなり承認せずに内容を読みます。

確認するポイントは次の通りです。

* 変更するファイルは正しそうか
* 関係ないファイルを触ろうとしていないか
* 仕様と違うことをしようとしていないか
* 危ない処理を入れようとしていないか
* 自分が理解できない変更がないか

わからない部分があれば、そこでAIに質問します。

この変更方針のうち、〇〇の部分がよくわかりません。  
初心者にもわかるように説明してください。

4. 問題なければ実装させる

方針に納得できたら、実装に進めます。

その方針で実装してください。  
ただし、勝手にコミットしないでください。

5. 実装後に変更内容を説明させる

何を変更したのか、初心者にもわかるように説明してください。

6. 自分で動作確認する

AIが「できました」と言っても、それを信じるだけでは不十分です。

自分で画面を開き、動作を確認します。

初心者におすすめのプロンプト

初心者は、次のようなプロンプトを使うと安全に進めやすいです。

実装前

```
実装する前に、変更方針を説明してください。
どのファイルを変更する予定かも教えてください。
まだコードは変更しないでください。
```

実装するとき

```
{#変更がある場合はここに記載（ない場合はその旨を伝える）}
```

実装後

変更内容を初心者にもわかるように説明してください。  
また、動作確認の方法も教えてください。

わからないコードがあるとき

```
このコードが何をしているのか、1行ずつ説明してください。
```

レビューしたいとき

```
この変更に危ない点や、もっと良くできる点がないかレビューしてください。
```

MCPは発展編

MCPとは、AIコーディングエージェントに外部ツールを接続するための仕組みです。

たとえば、

* GitHubを読ませる
* Figmaのデザインを読ませる
* データベースに接続する
* ドキュメント管理ツールに接続する

といったことができます。

MCPを使うと、AIがプロジェクト外部の情報も参照しながら作業できるようになります。

たとえば、Figma MCPを使えば、Figma上のデザインをもとに画面実装を進めることができます。

GitHub MCPを使えば、IssueやPull Requestの情報をもとに作業できます。

ただし、初心者が最初からMCPまで使う必要はありません。

まずは、

* AIコーディングエージェントを使う
* agents.md を書く
* docs/ を整える
* Planning mode で実装方針を確認する
* AIの変更を読む
* 自分で動作確認する

ところまでできれば十分です。

MCPは、余裕がある人向けの発展内容として扱います。
