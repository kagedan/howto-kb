AIに情報を渡せば渡すほど賢くなる。多くの人がそう信じています。

でも実際は、逆に働くことがあります。Chromaが2025年7月に公開した技術レポートは、18種類のLLMを検証し、入力が長くなるほど、単なる検索や複製のような簡単な作業でさえ性能が一律ではなく落ちていくと報告しました。しかも「情報がそこにある」だけでは足りず、どう見せるかで結果が変わるといいます。Anthropicも、AI設計の焦点を、プロンプトの文面から、AIに渡すトークン全体を選び抜く作業へ移すべきだと説明しています。

つまり勝負は「どれだけ持たせるか」ではなく「何を、どう見せるか」。この記事では、Claude Codeの記憶のしくみを分解しながら、AIを賢くするための記憶の設計を5つの視点で整理します。

# 「記憶は多いほど賢くなる」は逆効果になりやすい

情報は多いほど正確になる。直感的にはそう思えます。実際のデータは、逆を示しています。

Chromaのレポートが調べたのは、入力の長さと性能の関係です。18種類のLLMに対して、文章の中から特定の一節を探す、あるいはそのまま写すといった、人間なら間違えようのない作業をやらせました。結果、入力が長くなるほど正答率が下がり、その下がり方もモデルごとにバラバラだったのです。

なぜこうなるのか。AIは渡された文章を、端から順に均等に読むわけではありません。量が増えると、本当に見てほしい一行が、大量の背景情報に埋もれます。人間が分厚いマニュアルの中から一文を探すとき、ページが増えるほど見落とすのと同じです。情報がそこにあること自体は、読んでもらえる保証になりません。

ここで発想を切り替えます。記憶とは、貯めるほど価値が上がる貯金ではない。多いほど探しにくくなる、散らかった机に近いものです。

- まず「全部渡す」をやめて、今の作業に関係するものだけ手元に置く
- 使わない資料は捨てるのではなく、必要なときに開ける場所へしまう
- これは30秒の意識の切り替えで始められます
正直に言うと、情報を削るのは最初こわいものです。でも、探し物が減るほうがAIは速く正確になります。では、そもそもAIの「記憶」とは何を指すのか。ここを誤解したまま整理しても空回りします。

# AIの「記憶」の正体は、起動時に配られる一枚の資料

AIには人間のような脳の記憶があって、話したことを覚えている。多くの人がそんなイメージを持っています。少なくともClaude Codeの場合、実態は違います。

Claude Codeが持つ永続的な知識は、大きく2系統です。ひとつは人が手で書く`CLAUDE.md`。もうひとつはClaudeが会話中の修正や好みから自動で書き足していくAuto memoryです。この2つは、どの会話でも開始時にまとめて読み込まれます。

ここが肝心なところで、読み込まれた中身は絶対の命令ではありません。あくまで「参考にしてね」という位置づけの資料として扱われます。だから同じことを書いても、状況次第で守られたり守られなかったりします。

記憶とは脳ではなく、毎回配られる一枚の資料。この記事ではこれを「配布資料」と呼びます。会議の前に配られる資料と同じで、厚ければ読まれるわけではなく、要点が絞られているほど中身が伝わります。

- 覚えさせようとするのではなく、毎回渡す資料を編集する感覚で考える
- 「これは毎回配る価値があるか」を1項目ごとに問う
- 迷ったら、その1行を消しても仕事が回るか試してみる
才能や賢さの問題ではありません。配る資料の設計の問題です。となると、次に効いてくるのは、その配布資料そのものである`CLAUDE.md`の中身です。

# CLAUDE.mdが太るほど、指示は守られなくなる

ルールはとにかく`CLAUDE.md`に書いておけば安心。そう考えて、あれもこれも書き足していく人は多いです。これが精度を下げます。

Claude Code側は、`CLAUDE.md`を1ファイル200行未満に抑えることを目安として示しています。理由は明快で、長いファイルはそれだけでコンテキストを食いつぶし、書いてある指示の遵守率をむしろ下げる可能性があるからです。冒頭で見たとおり、量が増えれば大事な一行は埋もれます。ルールブックが分厚くなるほど、肝心のルールが読み飛ばされるわけです。

では何を書くべきか。Claude Codeの案内では、毎回必ず必要になるもの、と線が引かれています。

- 毎回使うビルドコマンドや実行コマンド
- コードの書き方の規約、命名のルール
- プロジェクトの全体構造
- 常に守ってほしい方
