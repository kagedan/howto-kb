個人でAI献立アプリを準備しています。冷蔵庫から使いたい食材を取り出して撮ると、AIが献立を提案してくれるアプリです。運営は私一人と、AIがひとり。実装はすべてClaude Code（AI）にやってもらい、私は動作とレシートを確認する係です。

月500円のサブスクで成立させたいので、AIの原価は1円単位で気にしています。その節約の本命がprompt caching——同じプロンプトの前半部分をキャッシュして、2回目以降の入力料金を約10分の1にする仕組みでした。

実装して、リクエストを送って、レスポンスの `usage` を見たら、こうなっていました。

```
"cache_creation_input_tokens": 0,
"cache_read_input_tokens": 0
```

エラーはありません。献立もちゃんと返ってきます。ただ、キャッシュだけが黙って仕事をしていない。

この記事は、その原因（ドキュメントに書いてあるのに気づけない仕様）と、「プロンプトを削る」のではなく**わざと13,000トークンまで太らせて**解決した話です。最後に実測のコストも載せます。

## 何が起きていたか

書いたコード自体は、ドキュメント通りの素直なものでした。システムプロンプトに `cache_control` を付けるだけです。

```
// Cloudflare Worker から Claude API (Haiku 4.5) を呼ぶ
system: [
  { type: "text", text: MENU_SYSTEM_PROMPT, cache_control: { type: "ephemeral" } },
],
```

これで「初回はキャッシュ書き込み（`cache_creation_input_tokens` が付く）、2回目以降は読み込み（`cache_read_input_tokens` が付く）」になるはず。ところが何度送っても両方ゼロ。マーカーの付け方、ブロックの分け方、TTL……ひと通り疑いましたが、どれも合っている。

## 原因：モデルごとに「キャッシュされる最低トークン数」がある

答えはAnthropicの公式ドキュメントに書いてありました。プロンプトキャッシュには**最小キャッシュ可能長**があり、プレフィックスがそれに満たない場合は「エラーにならず、ただキャッシュされない」のです。

| モデル | 最小キャッシュ長 |
| --- | --- |
| Opus 4.5〜4.8、**Haiku 4.5** | **4096トークン** |
| Sonnet 4.6 | 2048トークン |
| Sonnet 4.5 | 1024トークン |

うちの献立生成はHaiku 4.5で、システムプロンプトは4096トークンにまったく届かない規模でした。

この仕様のいやらしいところは、**何のシグナルも出ない**ことです。400も警告も返らず、`cache_control` を付けたリクエストは正常に成功し、ただ `write=0 / read=0` になる。「書き方が悪いのでは」と実装側を疑い続ける構図になります。しかも最小長はモデルによって4倍違うので、「Sonnetでは効いていたのにHaikuに切り替えたら効かなくなった」も普通に起きます。

## 解決：削るのではなく、太らせる

普通のチューニングは「プロンプトを短くして安くする」方向です。でもこの壁の前では逆で、**4096トークンを越えない限り、1トークンもキャッシュされない**。

かといって、意味のない文字列で水増しするのは最悪です。キャッシュ読み込みにも通常の10%の料金はかかるので、ゴミを13,000トークン読ませ続けるのはただの浪費になります。

そこで、アプリがもともと持っていた資産を使いました。このアプリには**内蔵レシピが113品**あります。その一覧（ジャンル・主菜・副菜・調理時間・カロリー・主材料）をテキスト化して、システムプロンプトに埋め込みました。

```
【和食】定番の生姜焼き｜主菜:豚の生姜焼き｜副菜:キャベツの千切り、味噌汁｜20分｜約450kcal/人｜子どもOK｜主材料:豚ロース、…
【洋食】…（×113品）
```

これでシステムプロンプトは約13,000トークン。壁を越えました。

この方法の良いところは、パディングが**実利を持つ**ことです。AIが内蔵レシピのラインナップを知った上で献立を組めるようになるので、アプリ内のレシピと矛盾しない提案が出やすくなる。「キャッシュのために太らせた」のと「提案品質のために文脈を渡した」が同じ一手になっています。

レシピ一覧は手書きせず、レシピデータから自動生成するスクリプトにしました。

```
// public/recipes.js から recipe-ref.js を再生成する
// レシピを追加・変更したら実行: node scripts/gen-recipe-ref.js
const lines = RECIPE_DB.map((r) =>
  `【${r.genre}】${r.title}｜主菜:${r.main}｜副菜:${r.sides.join("、")}｜${r.mainTime}分…`
);
```

一方で、ユーザーごとに変わる部分（選んだ食材、アレルギー設定、品数）はキャッシュを壊すので、システムプロンプトには入れず、userメッセージ側に寄せました。こちらは圧縮して**1リクエストあたり約80トークン**です。「変わらない13,000トークンをキャッシュに、変わる80トークンだけを毎回払う」という形になります。

## 実測

修正後の `usage` はこうなりました。

* 初回リクエスト: `cache_creation_input_tokens: 13060`（書き込み）
* 2回目以降: `cache_read_input_tokens: 13060`（読み込み）

ゼロが並んでいた欄に、13060がきれいに移動しました。

料金への効き方はこうです（Haiku 4.5は入力$1/100万トークン。キャッシュ読み込みは0.1倍、書き込みは1.25倍。1ドル150円換算）。

|  | 13,060トークンの入力コスト |
| --- | --- |
| キャッシュなし（毎回素読み） | 約2.0円/回 |
| キャッシュヒット時 | **約0.2円/回** |
| 初回の書き込み | 約2.4円（初回のみ） |

出力トークン等を含めた実測では、献立生成1回あたり**キャッシュヒット時1〜2円、ミス時3〜5円**。応答は6〜18秒でした。月500円のプランでユーザーが毎日3回使っても、原価は月100〜300円に収まる計算で、採算ラインに乗ります。

「短いプロンプトのままなら毎回0.1円以下では？」というツッコミは正しくて、純粋な金額だけならそれが最安です。ただその場合は113品のレシピ文脈を捨てることになる。**この文脈を渡したい前提に立った瞬間、「4096の壁を越えてキャッシュに乗せる」が唯一の正解**になります。中途半端に3,000トークンだけ渡すのが一番損な設計です。

## ハマらないためのチェックリスト

1. **`usage` を必ず見る。** キャッシュが効いているかはエラーではなく `cache_creation_input_tokens` / `cache_read_input_tokens` でしか分からない。両方ゼロが続いたら何かがおかしい。
2. **モデルの最小キャッシュ長を確認する。** Haiku 4.5とOpus系は4096トークン。Sonnetより壁が高い。モデルを乗り換えたときに黙って効かなくなるのはここ。
3. **太らせるなら「安定していて役に立つ」内容で。** キャッシュはプレフィックスのバイト単位の一致で判定されるので、日時やユーザーIDを埋め込むと毎回無効化される。変わる情報はuserメッセージ側へ。
4. **埋め込んだ参照データの更新はキャッシュの書き直し。** うちの場合、レシピを1品追加したら再生成スクリプトを回し、その直後の1回だけ書き込みコスト（1.25倍）が発生する。頻繁に変わるデータはこの手法に向かない。
5. **TTLは5分（デフォルト）。** 書き込みが1.25倍なので、5分以内に2回目のリクエストが来れば元が取れる。キャッシュは組織単位で共有されるので、別のユーザーのリクエストでもプレフィックスが同じならヒットする。ユーザーが増えるほどヒット率が上がる、数少ない「スケールすると安くなる」構造。
