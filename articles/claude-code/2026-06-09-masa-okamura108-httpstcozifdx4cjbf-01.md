# ループ エンジニアリングとは何か

プロンプト エンジニアリングが1回で良いプロンプトを作ってLLMに送り、1回の応答をもらうのに対して、ループ エンジニアリング**「AI エージェントを反復（ループ）して動かし、エージェントの判断でスキルやツールを使用し、自走を繰り返してゴール達成まで動かす仕組みを構築すること」**です。

プロンプト エンジニアリングが「AIに何を言うか」に重心を置く考え方だとすれば、ループ エンジニアリングは「AIが何度も考え、試し、検証し、必要なら人間に戻す仕組みをどう設計するか」に重心を置く考え方だと捉えています。

良いループを考えるときに、以下の6要素で見ると整理しやすいと感じています。

1: Trigger: 何をきっかけに動くか

2: Context: どの情報を渡すか

3: Action: 何を実行するか

4: Verification: 成功したかをどう確認するか

5: Memory: 結果や学びをどこに残すか

6: Escalation: AIだけで判断できない時にどう人間へ戻すか

「いい感じに処理して」と曖昧に頼むのではなく、成功条件と停止条件まで設計することが重要です。

・・・ここで疑問がでるかもしれません。

**「ループ エンジニアリングというけど、ただの定期実行では？」**

これは半分Yes。

結局のところ、5分ごと、1時間ごと、1日1回のようにAI（CodexやClaude Code）を定期実行させる発想は、ループエンジニアリングの中に必ず含まれます。

つまりCronで定期的にClaude Codeなどを呼び出して実行する、と言うことです

定期実行と聞くと「毎回同じ処理を行う」という響きがあるので混同しますが、ループの場合は「発火は定期実行だが、処理内容はAIが判断して毎回違った処理を行う」となります。

また、別の論点があるとすれば、上でもあげた「トリガー」で、例えば以下のような動作をトリガーに動かす場合は定期実行の枠からは外れます。

・Slackでメッセージを受信したら動く。

・Gmailが来たら動く。

・Stripeで決済が行われたら動く。

・GitHubでPRが更新されたら動く。

こうしたイベントをトリガーにAIを稼働させるのも、ループ エンジニアリングのスコープではないかと思います。

大事なのは、定期実行かイベント駆動かではなくAIが何かをきっかけに動き、タスクを実行するということです。

# 弊社で回しているループ

弊社では「AIカンパニー」を構築して、複数のAI社員に業務を担当させています。その中で、すでに複数のループを回しています。

## 1. Slackを監視してNotionにタスクを入れるループ

まずはSlackの監視です。

Slack上でクライアントやチーム内でコミュニケーションしているとタスクが発生します。これを5分単位で確認するループを構築しています。

タスクが発生したら自動的にプロジェクトごとのNotion DBに登録します。

これは、AI駆動PMの一種です。
