AIエージェントが自身のハーネスを観測し、改変提案を返す仕組み — メタハーネス — は、コーディングエージェントとパーソナルAIの領域で実装が進み、設計パターンとして輪郭が見え始めている。Stanford / KRAFTON / MIT による Meta-Harness 論文(Lee et al., 2026)、UBC / Meta / Edinburgh らによる HyperAgents(Zhang, J. et al., 2026)、Fudan / Peking University らによる Agentic Harness Engineering(AHE; Lin, J. et al., 2026)といったプレプリント系統が並行して進行し、auto-harness や SICA、HALO のようなオープンソース実装も公開されている。これらは、評価関数が明確で、改変対象が単一プロセス内に閉じている領域で機能する。本稿が扱うのは、その理論を企業AIエージェントの運用環境に持ち込んだときに何が変わるか、という設計論である。

本稿の主張は以下のとおりである。メタハーネスの骨格 — エージェントが自身のハーネスを観測し改善案を提案するループ — は、運用主体が個人から企業に変わっても維持できる。しかし、設計上の前提が複数の軸で反転する。何が反転し、それを構造としてどう収容するか、それが本稿の主題である。

筆者は前稿で個人AIエージェントのメタハーネス実装を扱ったが、本稿ではメタハーネスの企業導入そのものの設計論について論じる。

## 本稿のスコープ

メタハーネスという用語は、研究文脈と運用文脈の間で意味が揺れる。本稿では明確に定義しておく。

**ハーネス** とは、AIモデルが入力を受けて出力を返すまでに通過するすべての周辺構造 — プロンプト、ツール定義、参照データ、ワークフロー、メタファイル、ガードレール — を指す。同じモデルでも、ハーネスが異なれば別のエージェントになる。

**メタハーネス** とは、AIエージェント自身が、自分を取り巻くハーネスを観測・評価し、改変提案を返す仕組みを指す。AIが直接ハーネスを書き換えるとは限らない。本稿では、観測・提案は AI が担い、改変の実行は人間の承認を経て初めて反映される設計を、企業運用における基本形とする。

本稿が扱う対象は、AIエージェント周辺のハーネス層(プロンプト、ルールセット、分類体系、スキル定義、評価ロジック)の自己改善である。

補足しておくと、AIがコード変更案を生成し、人間の承認を経て本番に反映する運用 — Claude Code や Codex CLI に代表される PR ワークフロー — は既に本番運用に入っており、本稿はそれ自体を否定するものではない。本稿が実用域にないとして対象外に置くのは、Meta-Harness 論文の outer loop に代表される **AIが自身のコードやハーネスを完全に自律的に改変し続けるシナリオ** を、企業の業務システム本体に対して回す形態である。これは現時点のハルシネーション制御では実用域にない。制約というより、現時点で守るべき設計上の対象範囲設定である。根拠は本稿後半で改めて整理する。

## エンジニアの責任範囲、メタハーネスの責任範囲

本論に入る前に、企業メタハーネスを語る上で最も重要な区別を先に示しておく。**システムの要件を定めることはエンジニアの責任範囲であり、定められた要件の範囲内で改善を行うことがメタハーネスの責任範囲である。**

これは別々の仕事を二者が並行して行っているという話ではない。同じ「企業AIエージェントを健全に運用する」という一続きの設計問題があり、その中でどこからどこまでが人間が引き受けるべき責任で、どこからが自動化に委ねられる責任かを区別している。境界線が明確になれば、両者の関係性も自然に決まる。

この区別は前稿で論じたパーソナルAIエージェントの実装論において提示した原則 — "ハーネスは境界を定義する。エージェントはその境界内で自律する" — を企業文脈に持ち込んだ帰結である。

エンジニア(社内エンジニア、外部の受託エンジニア、Forward Deployed Engineer 系の職能など、組織によって呼称はさまざまだが、本稿では便宜上 "エンジニア" と総称する)の責任範囲には次が含まれる。

* AIエージェントの担当タスク、利用ツール、参照データ、出力先 — つまりエージェントの職務範囲そのものの定義
* 多部署からの要望や業務上のフィードバックを集約し、システム要件に落とし込む作業
* どこをメタハーネスの改善対象とするか、どこを触らせないかの境界設計
* 改善提案の評価基準、デプロイ条件、承認権限の階層設計

メタハーネスの責任範囲は、エンジニアが事前に宣言した境界の内側に閉じた、極めて限定的な領域である。

* 構造化された信号(ツール実行ログ、定量メトリクス、A/B テスト結果など)を観測する
* 観測結果から、定義済みの改善対象(プロンプト、スキル定義、分類体系など)に対する改善案を生成する
* 改善案を承認パイプラインに送り出す

この責任範囲の境界線を最初に固定することで、後段の議論で繰り返し現れる疑問 — "誰が要件の発散を抑えるのか"、"どのような機能を実装するかは誰が決めるのか" — に統一した答えが置ける。これらはエンジニアの責任範囲に属する判断であり、メタハーネスに委ねるべき判断ではない。メタハーネスは、エンジニアが定めた境界の内側で機能する仕組みとして設計するのが望ましく、境界そのものを動かす権限は持たせないほうがよい。

この前提の下で、企業メタハーネスの設計要件を具体化していく。

## 企業メタハーネスの設計要件

企業AIエージェントのメタハーネスは、3つの要件を満たさなければならない。

### 1. Bounded — 改善スコープを事前に閉じる

エージェントには明確な職責が割り振られ、改善の対象は **職責内での精度向上**(正答率、失敗率、応答時間、誤判断率)に限定される。スコープの拡大 — scope creep — を防ぐこと自体が要件になる。

この要件は設計の方向性そのものを規定する。Bounded な改善ループは、信号源・改変対象・評価基準を事前にすべて定義しなければ動かせない。「まず運用を開始し、状況を見ながら調整する」という運用方針は、企業文脈では成立しない。事前にすべての境界を宣言する設計コストは、運用開始の閾値として最初に支払うべきコストになる。

そしてこの設計コストを支払うのは、前節で述べたとおりエンジニアである。

### 2. 信号源は観測可能で構造化されたデータに限定される

メタハーネスの自動ループに直接乗せられる信号は、観測可能で、事前に構造化された信号に限定される。具体的には次のとおりである。

* ツール実行ログ(成否・エラーコード・実行時間・呼び出しパターン)
* ベンチマークスコア(正答率、F1、レイテンシなど、定義済みの定量メトリクス)
* A/B テストや shadow deployment の結果(バージョン間の定量比較)
* 構造化されたユーザー操作(good/bad ボタン、A/B 選好クリック、出力の採用/却下/編集の操作ログ)
* 明文化されたルール違反シグナル(社内規約や SLA の閾値超過、コンプライアンス・チェッカからの自動検知)

これらは AI が直接読み取って改善案を生成できる。一方、自由記述のフィードバックや、組織内の複数部門からの要望をどう集約するかといった問題は、メタハーネスの自動ループの外で扱う。なぜそうなるかは、後段の「メタハーネスが機能する条件」で論じる。

### 3. 改変可能領域は極端に狭い

AIエージェント周辺のデータは、mutability の観点から2つに分かれる。

**Type A(append-only で改竄不可なデータ):** 履歴、ログ、判断記録、契約、取引データ、議事録、ナレッジベース、ツール実行 trace、承認/却下の判断ログ。事後に正しさを検証できる状態で残されなければならない監査・コンプライアンス対象データの大半がここに入る。

**Type B(AI による改変提案の起点になり得るデータ):** エージェントの行動ルール、スキル定義、プロンプトテンプレート、ワークフロー定義、エラー処理手順、分類体系。エージェントの振る舞いを規定するメタファイル群。

企業運用では、多くのデータが Type A 側へ偏在する。企業データの多くが監査要件と運用要件のため改竄不可で扱われる必要があるためである。その結果、メタハーネスの Type B は **極めて限定的** になる — 上記の限定的なリストに収まる。改善対象の領域が狭いということは、その狭い領域の中で何をどう設計するかが、極めて精密に問われるということでもある。

この Type A / Type B 区分は本稿全体を通じて参照される設計概念であり、後段の議論(スコープ設計、承認パイプライン、改変提案の範囲)はすべてこの2分類の上に立つ。

ここまでの設計要件を実装に落とすと、企業AIエージェントのメタハーネスは **2層のスコープ設計** によって構成される。

**Layer 1: エージェント本体のスコープ。** このエージェントは何のタスクを担当するか。どのツールを使えるか(読み取り専用か書き込み可能か)。どのデータを参照するか。出力はどこへ渡すか。エージェントの「職務範囲」の定義。

**Layer 2: 改善ループのスコープ。** メタハーネスは何を信号源にするか。何を改変提案できるか(プロンプト、ルールセット、分類体系、スキル定義など)。どこには絶対に触れさせないか(システムアーキテクチャ、業務コード本体、人事情報など)。ベンチマークはどう定義するか。

この2層は **一致していなければならない**。これが Scoped Meta-Harness 設計の最重要原理である。

Layer 1 で「API エラー分析エージェント」と定義したのに、Layer 2 の改善ループが企業システム全体を観測できる状態になっていれば、改善提案は必然的に越境する。「このエラーの本質は認証フローの設計にある。Auth0 から Cognito への移行を推奨」といった提案が、エラーログ分類エージェントの改善ループから上がってくる。技術的に筋が通っていても、運用観点で即座にその提案が受け入れられる可能性はほぼゼロで、トークンを浪費するだけで終わる。

### スコープの宣言は設計時に固定する

Layer 1 のスコープは、運用開始前に明示的に宣言され、構造化された設定ファイル(TOML や YAML などのスキーマ付き定義ファイル)として固定される。担当タスク、許可ツール、参照データ、出力先、各 I/O の制約 — これらを設計時に書き切る。後からエージェントに新しいツールや新しい職責を「便利だから」と追加していく運用は、構造的にスコープを溶解させていく。

これは美学の問題ではなく、性能上の問題でもある。Berkeley Function Calling Leaderboard(Patil et al., 2025, ICML)を中核とする一連の研究は、ツール数が増えるほど LLM の正しいツール選択精度が急落することを繰り返し示してきた。Gan & Sun(2025)の RAG-MCP は、MCP プール規模の拡大に伴うツール選択精度の単調な低下を独立に実証しており、選択肢の増加が単にトークン消費の問題ではなく選択精度そのものの問題であることを明示している。具体的には、同論文の MCP stress test において、ベースライン手法(Blank Conditioning)のツール選択正答率は、ツール候補数の増加に伴い 13.62% まで低下し、retrieval ベースのツール選択導入によって 43.13% へと3倍以上に改善することが示されている。さらに同論文の position 100 以降の評価では、選択失敗が成功を上回る領域へ突入することが報告されており、ツール数の拡大が単なるトークン消費の問題を超えて選択精度そのものを破綻させる現象が定量的に確認されている。Zhang, K. et al.(2026)は、意味的に類似した distractor ツールが存在する状況下では、ツール追加そのものが精度を下げる "tool-use tax" として作用することを定式化し、ツール拡張が想定外の文脈で他ツール選択を阻害する機序を示した。Anthropic の内部測定でも、58 個のツール定義は約 55,000 トークンを消費し、選択肢が増えるに従って正解率が低下する。LiveMCPBench(arXiv:2508.01780)による独立した検証では、12 個のフロンティア LLM を実環境の MCP タスクで評価した結果、Claude-Sonnet-4 が 78.95% のタスク成功率に到達した一方で、大半のモデルは 30–50% の範囲に留まり、失敗の約半数がツール検索段階の誤りに起因することが報告されている。

これらの事実は、Layer 1 設計に対して厳格な要請を突きつける。「念のため使えるツールを増やしておこう」「いずれ必要になりそうだから職責を広げておこう」という設計判断は、現在使われていない選択肢が、いずれ使うことになる選択肢の精度を確実に下げる。スコープの広さは、それ自体がコストである。

そして Layer 2(改善ループ)は、この Layer 1 の宣言に厳密に従う。改善エージェントは、自身が改善対象とするエージェントの宣言ファイルを最初に読み、そこに記された担当タスク・許可ツール・参照データの範囲内でのみ改善提案を行う。スコープ外のエラー、スコープ外の改善対象、スコープ外のツールの追加 — これらは構造的に提案不可能になるよう、改善エージェントのプロンプトとアクセス制御の両方で閉じる。

### ケーススタディ: API エラー分類エージェント

具体例で見る。本番 API のエラーログを継続的に分類し、根本原因仮説を提示するエージェントを設計する。

**Layer 1 の設計(設計時に固定):**

* タスク: 本番 API のエラーログから既知パターンへの分類、および根本原因仮説の提示
* 使用ツール: ログ検索、スタックトレース解析、変更履歴参照(すべて read-only)
* 参照データ: 直近30日のエラーログ、既知障害DB、デプロイ履歴
* 出力先: オンコールエンジニアの Slack チャンネル

**Layer 2 の設計:**

* 信号源: 分類結果に対するオンコールエンジニアの採否、根本原因仮説の検証採用率(いずれも構造化された定量信号)
* 改変提案可能領域: エラーパターン分類体系、根本原因ルールセット、入力プロンプト
* 改変不可領域: API コード本体、インフラ構成、Slack 通知ルール本体、新規ツールの追加、新規データソースの追加
* ベンチマーク: 分類精度、仮説採用率、誤分類率の推移

この設計で本質的にやっていることは、「エージェントが見るデータ範囲」と「メタハーネスが改善対象とする範囲」を意図的に揃え、それ以外の改変経路を構造的に閉じる ことである。

スコープを切らずに同じエージェントを動かせば、AI は「このエラーパターンが頻発するのは認証アーキテクチャの設計ミスです。Auth0 からの脱却を推奨します」「エラーハンドラの設計を全面リファクタリングすべきです」といった、運用観点では即座に却下される提案を出力する。これは AI の判断が誤っているわけではない。**そうした提案が生じる構造を、ハーネス設計の段階で組み込んでしまっている** ことが、構造的な原因である。

この設計問題は、診断型エージェントに限定されるものではない。営業通話の文字起こしから決裁者・予算・導入時期を抽出して CRM に書き込む抽出型エージェント、契約書を要約して条項リスクを評価する評価型エージェント、社内ナレッジから FAQ 候補を生成する生成型エージェント、いずれにおいても Layer 2 のスコープを切らなければ、AI は「そもそもこの CRM のフィールド構成が不適切であり、再設計すべき」等の越境提案を返す。タスクの形式が異なっても、構造的失敗は同じ場所で発生する。Scoped Meta-Harness は、エージェントの型に依存しない汎用設計原理として機能する。

### 改善と仕様変更の区別

ここで、Scoped Meta-Harness を運用する際に決定的に重要な区別を導入する。**改善(improvement)と仕様変更(spec change)は、別物として扱われなければならない。**

**改善:** 設計時に宣言された Layer 1 スコープの内側で、精度・効率・正答率・誤判定率を向上させる作業。プロンプトテンプレートの修正、スキル説明文の精緻化、エラーハンドリングロジックの調整、分類カテゴリの細分化 — これらは改善である。

**仕様変更:** Layer 1 スコープそのものを変更する作業。新しいツールの追加、新しいデータソースへの接続、エージェントの担当タスクの拡張、出力先の追加 — これらは仕様変更である。

メタハーネスが扱うのは **改善のみ** である。仕様変更は、メタハーネスの自動ループの中で発生してはならない。

なぜか。仕様変更を許せば、改善ループは無限にスコープを拡張できるようになる。ある時点で「このエラー解消には新しい API を呼べる必要がある」という提案が通れば、その API が新しい信号源になり、新しいエラーパターンが観測対象になり、それを解決するためにまた新しい capability が必要になる — こうした連鎖が止まらなくなる。改善ループは表面上は継続的に「改善」を続けているが、エージェントの実体は元の設計から完全に乖離していく。前節で触れたツール数増加に伴う精度低下は、この拡張のたびに累積的に重なる。仕様変更を含む自己改善は、長期的には実装を発散させ、エージェント自体の信頼性を破壊する可能性がある。

ただし、この区別は実務で常にクリアに引けるわけではない。境界が曖昧になる典型例を3つ挙げておく。

**分類カテゴリの細分化。** エラー分類体系で「認証エラー」を「OAuth トークン失効」「セッション期限切れ」「権限不足」に分割する提案を考える。既存の信号源(エラーログ)と既存の参照データ(エラーパターンDB)の内側で新しい区別が成立するなら、これは改善である。しかし、新しい区別を成立させるために認証フローのトレースデータが新規に必要になる場合、それは新しい信号源の追加 — 仕様変更に該当する。表面上は「分類を細かくしただけ」に見えても、実装の前提条件が変わっていれば仕様変更である。

**Few-shot 例の差し替え・追加。** プロンプトに含める例示の質を上げるのは改善であるが、例示が新しいデータカテゴリ(これまでエージェントが扱っていなかった種別)を含み、結果としてエージェントが「これも自分の担当範囲だ」と解釈するように変わるなら、それは職務範囲の暗黙的な拡張で仕様変更に該当する。Few-shot は単なる例示に見えて、エージェントの判断範囲の宣言として機能する側面があるため、ここには細心の注意を要する。

**ツール記述の編集。** AHE 論文が改変対象として明示している領域であり、メタハーネス研究で最も活発に議論されている改変経路の一つでもある。同じツールの呼び出し方を明確化するのは改善であるが、記述変更によって「呼び出してよい状況」の解釈が広がる場合 — 例えば「読み取り専用クエリに使う」を「データ確認全般に使う」と書き換えるような場合 — 実質的に行動ルールの変更であり、仕様変更の境界に乗る。

これらの境界例から見えてくるのは、判定基準が **「文面上の変更が小さいか」ではなく「変更後にエージェントが触れる信号源・参照データ・出力範囲が拡張されるか」** という一点に集約されるということである。改善エージェントの自問は、より正確には次のように言い換えられる: 「この変更は、Layer 1 で宣言された I/O 境界を一切動かさずに実装可能か?」 これを満たさない提案は、改善ループの外に上げる。

この自問を改善エージェントのプロンプトレベルで明文化し、I/O 境界の差分を構造的に評価できるようにする。提案がこの境界条件を満たさない場合、改善エージェントは **提案を行わず、代わりに「これはスコープ拡張要件であり、人間による仕様判断が必要」とフラグを立てるに留める**。

仕様変更の判断は、人間 — エンジニアと業務責任者 — が、改善ループの外で行う。冒頭の役割分担に従えば、これは当然の帰結である。新しい capability が本当に必要か、その追加が他のエージェントや業務にどう影響するか、ツール数増加によって既存スキル選択精度がどう変化するか、コンプライアンス影響はどうか — これらは改善ループのスコープを超えた経営・運用判断であり、AI が自律的に決めるべきものではない。

## メタハーネスが機能する条件

ここまでで、メタハーネスは「観測可能で構造化された信号があり、明確な評価基準がある」ときに機能する仕組みであることを述べてきた。逆に言えば、その条件が揃わないところでは、メタハーネスを無理に動かそうとしても発散する。

この成立条件は、コーディングエージェントの自己改善研究によって実証的にも裏付けられている。Robeyns et al.(2025)の SICA は SWE-Bench Verified という ground truth が確立された領域で、エージェントの自己改善を 17% から 53% に引き上げた。Agrawal et al.(2025)の GEPA は、構造化されたテキスト feedback(実行ログ、トレース、エラーメッセージ等)が、スカラー報酬ベースの強化学習を上回る効率で prompt を最適化できることを示している。いずれも、評価関数が verifiable で信号が構造化されているという前提下での結果であり、本稿が企業文脈における必要条件として整理する3条件と一致する。

条件をもう少し具体化すると、自動ループが成立するのは次の3つが揃ったときである。

* **信号が構造化されている:** 成功/失敗がバイナリで取れる、定量メトリクスが定義済み、選好データが離散的に取れる、など
* **信号源の役割が単一の文脈に閉じている:** 信号の発生源がエージェントの職務範囲と一致しており、他の文脈の信号が混入しない
* **評価基準が事前に明文化されている:** 何をもって「改善」と判定するかが、リリース前に決まっている

この3条件が揃わない領域 — 多部署からの自由記述要望、組織横断の優先順位調整、業務要件そのものの定義 — は、そもそもメタハーネスのスコープではない。それらはエンジニアの責任範囲として、メタハーネスの外側で扱うべきだ。

このことは、企業導入の現場で率直に認識する必要がある。「企業メタハーネスを導入しました」と謳いつつ、実体としては自由記述フィードバックをエンジニアが解釈・構造化して改善対象を特定する運用は、本稿の定義では狭義のメタハーネスではない。広義のメタハーネスとして含めることはできるかもしれないが、Lee et al. の論文で示された評価関数駆動の outer loop が自律で回るのとは、自律性のレベルで明らかに差がある。

つまり企業AIエージェントのメタハーネスは、**評価基準と観測信号が事前に明確に定義された領域でこそ自動化が成立する仕組み** である。次節では、その成立条件を満たす具体的な実装パターンを示す。

## メタハーネス実装パターン

ここからは、企業環境で実際にメタハーネスを動かしている、あるいは動かし得る具体的な実装パターンを4つ紹介する。いずれも前節の3条件 — 構造化信号、文脈の単一性、明文化された評価基準 — を満たすケースである。

### パターン 1: ツール実行エラーの自己改善

最も基本的な実装パターンである。エージェントが利用するツール(API、スクリプト、CLI)の実行ログを信号源とし、エラー率の高いツールやスキルの定義・実装を改善する。

**信号源:**

* ツール実行ログ(成否、エラーコード、引数パターン)
* 同一エラーの発生頻度とコンテキスト

**改善エージェントの動作:**

* 日次・週次で実行ログを集計し、エラー率の閾値を超えたツールを抽出する
* エラーパターンを分類し、根本原因仮説を立てる
* 仮説に応じて改善案を生成する:
  + 引数の使い方が誤っている → スキル定義ファイル(SKILL.md)の記述を改訂
  + 想定外のエッジケース → スクリプト本体やエラーハンドリングロジックを改訂
  + API 仕様変更への追従漏れ → 当該 API 呼び出し部の改訂

**デプロイフロー:**

* 改善案を開発環境にデプロイ
* 過去ログのリプレイテストで回帰がないことを確認
* shadow deployment または A/B テストで本番影響を測定
* 改善が認められた場合、エンジニアが承認 → クライアント承認 → 本番デプロイ

このパターンは、Layer 1 で許可されたツールセットそのものを変えるわけではない(それは仕様変更領域)。あくまで既存ツールの呼び出し方や周辺ロジックを改善するに留まる。

### パターン 2: チャットボットの構造化されたユーザーフィードバック

社内向け FAQ ボットやサポートチャットボットなど、エンドユーザーから直接フィードバックを受けられる領域である。前節の3条件が最も整って揃う代表例である。

**信号源:**

* 各応答に対する good / bad ボタン操作
* 同一質問に対する新旧バージョン A・B の選好クリック
* エスカレーション率(ボットの応答後、人間オペレータに引き継がれた割合)
* 平均応答時間、コンプライアンス違反検知率

**改善エージェントの動作:**

* 低評価の応答クラスタを抽出し、共通パターンを特定
* プロンプト、応答テンプレート、参照ナレッジの取得ロジックの改善案を生成
* 候補バージョンを開発環境に生成

**デプロイフロー:**

* 新バージョン候補を shadow deployment で旧バージョンと並走
* 一定トラフィックを経過した時点でメトリクスを集計
* 事前定義された複数のメトリクス閾値(例: 正答率が旧バージョン比 +3%以上、エスカレーション率が悪化していない、応答時間が許容範囲内、コンプライアンス違反率が増えていない)をすべて満たすか判定
* 基準を満たす候補のみが次バージョンとして昇格

このパターンの肝は、**信号が構造化されている**(クリック操作はバイナリまたは離散選択)、**信号源の役割が単一**(ユーザーとしての操作のみで、業務要件への意見は混入しない)、**評価基準が事前に定義済み**(リリース基準はメトリクス閾値) という3条件がすべて成立している点である。

### パターン 3: RAG エージェントのスキル更新

時間とともに参照データが蓄積・更新されていく RAG(Retrieval Augmented Generation)エージェントは、メタハーネスが活きる領域の一つである。データの内容や構造が変化するに伴い、それを取得・処理するスキルも更新が必要になる。

**信号源:**

* RAG 検索結果と最終応答に対する正答率(評価セットによる)
* 検索ヒット率、検索結果の利用率(ヒットしたが使われなかった割合)
* ユーザーからの「該当する情報が見つからなかった」型の構造化フィードバック
* 新規追加されたデータカテゴリの統計

**改善エージェントの動作:**

* 検索精度のメトリクスが低下した領域を特定
* 原因を分類:
  + 検索クエリ生成プロンプトが新しいデータカテゴリに追従していない → プロンプト改訂
  + スキル定義に書かれた検索戦略が陳腐化している → SKILL.md 改訂
  + チャンク化やリランキングのロジックが不適切 → 処理スクリプト改訂
* 改善案を生成

**デプロイフロー:** パターン 2 と同様に、評価セットでの自動評価と shadow deployment を組み合わせた段階的リリース。

このパターンは、データの増減・変化に伴って継続的な微調整が必要であり、人手で追従し続けるのが現実的でない領域で特に価値を持つ。

### パターン 4: 分類・抽出タスクの精度改善

請求書からのフィールド抽出、サポートチケットの分類、契約条項のカテゴライズなど、**ground truth が(完全ではないにせよ)取れる** タスクである。

**信号源:**

* 人間レビュアーによる修正ログ(エージェントの抽出/分類結果と最終確定値の差分)
* カテゴリ別の正解率、誤分類のパターン

**改善エージェントの動作:**

* 修正ログを集計し、誤りパターンが集中している分類カテゴリ・抽出フィールドを特定
* 分類体系の細分化・再定義、プロンプトの改訂、Few-shot 例の差し替えなどを提案

**デプロイフロー:** 過去データでのバックテスト → エンジニア承認 → 本番反映、または shadow mode での新旧比較を経たうえで本番反映。

---

これら4パターンに共通するのは、いずれも **観測可能で構造化された信号** と **明確な評価基準** が存在し、Layer 1 スコープの内側に改善対象が閉じている点である。逆に、これらの条件が揃わない領域では、メタハーネスの自動ループは成立しない。「メタハーネスをどうやって企業システムに取り込むか」という問いに対する実務的な答えは、**この条件が揃う領域を選び抜くこと、そして条件が揃うようにエンジニア側が信号源と評価基準を事前に整備すること** にある。

### 適用が成立する条件

ただし、上記4パターンの条件をクリアできる領域を抱える組織であっても、Scoped Meta-Harness の構築は無条件に投資対効果として成立するわけではない。信号源インフラ、評価セット、shadow deployment 基盤、承認パイプライン、改変提案のキャプチャレイヤー — これらを揃える初期工数は、エージェント1体に対して概ね数人月規模で発生する。立ち上げ後も、評価セットの維持、信号源の健全性監視、改変提案の承認運用 — これらが継続的なコストとして残る。これに見合う規模感が前提条件になる。

経済的に成立しやすい典型は次のいずれかである。

* 単一エージェントの呼び出し量が大きく、わずかな精度改善でも累積効果が大きい(社内 FAQ ボット、本番 API のエラー処理、大量の請求書処理パイプラインなど)
* 同種のエージェントを複数本運用しており、改善基盤を横展開できる
* 改善の遅延がコンプライアンスや SLA 違反に直結し、人手の追従が現実的でない

これに該当しない組織 — エージェントが少数で呼び出し頻度も限定的、改善対象が低頻度の業務エッジケース、といった条件下では、無理にメタハーネスを組まず、定期的な人手チューニングで運用する方が ROI が成立する場合が多い。「AI エージェントを導入したからメタハーネスも組む」という連鎖は、規模が伴わない場合に過剰投資になりやすい。

適用順序の観点もある。最初に組むべきは、本稿で示した4パターンのうち **信号源の構造化コストが最も低い領域** である。チャットボットの good/bad ボタンは UI に1ボタン追加するだけで済むが、契約条項評価の修正ログを構造化シグナルとして取得するには、人間レビュアのワークフロー自体を組み替える必要がある。前者から始めて運用ノウハウと承認運用の慣行を蓄積し、後者へ拡張するのが現実的な導入パスになる。最初の1本でガバナンスの型を作り、2本目以降でその型を再利用する — ここに横展開の経済性が生じる。

## 沈黙が意味するもの

エージェントの暴走は、フィードバック不足によって発生するわけではない。それは別の機構 — 承認パイプライン、改変対象の限定、tool access の構造的制限 — で抑える。これらが機能していれば、誰もフィードバックを返さなくても、AI が独自にハーネスを改変して崩壊することはない。

しかし、企業運用における沈黙は別の意味で深刻である。改善可能性を秘めたデータが、改善に変換されないまま蓄積し続ける。エージェントの出力に違和感を持つ場面があっても、構造化された信号として記録されない限り、そのシグナルはどこにも残らない。エージェントは依然として動き続けるが、運用側が想定する水準と、実際の出力品質の間にギャップが開いていく。

沈黙を構造化するのも、メタハーネスではなくエンジニアの責任範囲である。具体的には次のような対応が必要となる。出力に対する採用/却下/編集の操作を構造化シグナルとして自動記録するインフラを敷く。定期レビューサイクルでサンプリング評価を行い、評価結果を定量化する。明示的なシグナルが極端に少ないエージェントを「改善機会の喪失リスク」として可視化し、能動的にレビューを入れる。これらはすべて、メタハーネスを動かす前段の信号源整備の問題である。

ここでも役割分担は変わらない。信号を作るのは人間、信号から改善案を出すのが AI である。

## 承認パイプライン: 基本はサジェスト、条件付きで自動デプロイ

Scoped Meta-Harness の最後の構成要素は、改変提案がどう本番に反映されるかのフローである。

企業運用における基本形は次のとおりである。※単純化された例

1. 改善エージェントが構造化信号源を読む
2. Layer 1 スコープを読む
3. スコープ内に収まる改善案をサジェストする
4. エンジニアが評価する(スコープ内か、仕様変更を含まないか、技術的に妥当か)
5. 必要に応じてクライアントの承認プロセスに上げる
6. 承認された変更だけがハーネスに反映される

この基本形が要求される理由は、技術的なものと制度的なものの2つである。技術的には、メタハーネスのサジェストを生成する AI 自体がハルシネーションを起こす可能性がある以上、無監督の自己改変は構造的に許容できない。制度的には、本番エージェントの行動ルールやスキル定義の変更は、運用責任者の判断として実行される必要があり、AI の自動判断として実行されてはならない。

実装としては、メタハーネスからの改変提案は **必ずキャプチャレイヤーに書き出され、本体のハーネスファイルとは別ファイルとして保管される**。本体ファイルへのマージは、別のプロセス(reconciler)が承認済みのものだけを対象に実行する。キャプチャとマージを別ファイル・別タイミングに分離することが、AI 提案と人間承認の境界をディレクトリ構造として保持する設計になる。

承認権限の階層は、企業の組織構造に従って設計される。スキル定義の些末な記述修正は運用担当者の承認で十分であるが、エージェントの行動ルールの構造的変更には業務マネージャの承認が、コンプライアンス関連の判断ロジックの変更には法務側の承認が、それぞれ必要になる。**承認権限の階層を組織図に重ねる設計判断** が、ここで要求される。

### 例外: 定量基準下での自動適用

ただし、すべての改変が人間承認を経るべきとは限らない。明確な基準がルールとして事前に定義できる場合は、自動適用も成立し得る。

例えば、前節で示したチャットボットのパターンを想起していただきたい。新バージョン候補を shadow deployment で並走させ、事前定義された複数のメトリクス閾値(正答率が旧バージョン比で一定割合以上、エスカレーション率が悪化していない、応答時間が許容範囲内、コンプライアンス違反率が増えていない、など)を **すべて満たした場合に自動でリリースされる** という仕組みは、技術的にも制度的にも成立する。

成立条件は厳密である。

* 評価メトリクスがすべて事前に定量定義されている
* 各メトリクスの閾値が事前に承認されている
* 閾値判定の対象トラフィック量が事前に決まっている(統計的に有意な評価ができる量)
* ロールバック条件と手順が事前に整備されている

これらの条件を実装に落とす段階で、運用側はいくつかの非自明な技術判断を迫られる。統計的に有意な評価に必要なトラフィック量は、改善対象の効果サイズと許容する誤検出率から逆算する必要があり、社内 FAQ ボット程度の利用量では有意差検出に数日から数週間を要する場合がある。これは「shadow deployment を回せば数時間で結果が出る」という直感とは大きく乖離する。トラフィック分配も、単純なランダムスプリットでは不十分なことが多い — ユーザー属性や時間帯の偏りを補正したサンプリングが要求され、ピーク時間帯と閑散時間帯の挙動差で誤った判定が出るのを避ける設計が必要になる。ロールバック手順は、新バージョンが古いバージョンのデータ依存(キャッシュ、累積した分類結果、ユーザー側に蓄積された応答履歴など)を抱え込まない設計が前提であり、これを満たさないと「ロールバック可能なはずだったが、データレベルで前進していて戻せない」事態が発生する。これらは自動適用を成立させる前段で整備されるべきインフラ要件であり、メタハーネスを動かす前のエンジニア責任範囲に含まれる。

もう一段深い問題として、評価メトリクスそのものが改善ループによってハッキングされるリスクがある。Goodhart の法則として知られるとおり、ある指標が最適化の目標になった瞬間、その指標は本来意図していた品質の代理指標として機能しなくなる。チャットボットの正答率が上がっているにもかかわらず、実体としては「ユーザーが気にしない誤りを量産しているだけ」という方向に最適化が進む事例は、強化学習文脈で多数報告されてきた。改善ループでも同じ機序が働く。完全な防御策はなく、複数メトリクスのコンジャンクションを要求すること、人間によるサンプリング監査を定期的に挟むこと、メトリクスの定義自体を定期的に見直すこと — これらの組み合わせで崩壊までの時間を稼ぐのが現実的な対処になる。自動適用の判定が単一メトリクスの閾値で組まれていれば、その単一指標を狙い撃ちした改善提案が通り続け、ある時点で表面化する形で崩れる。

そしてこれらの条件が揃わない場合、自動適用は許容すべきではない。

さらに重要なこととして、自動適用が許されるのは、**業務上クリティカルでない領域に限定すべき** である。社内 FAQ ボットの応答改善は、誤適用しても影響は限定的でロールバックも容易であるが、契約書のリスク評価や与信判断のロジックは、誤った変更が業務にクリティカルなインパクトを与え得る。後者では、定量基準を満たしていても人間によるチェックを残すべきである。

自動適用と人間承認の線引きそのものを誰が決めるかと言えば、これもまたエンジニア — そして業務責任者・コンプライアンス部門 — の責任範囲である。「どのエージェントのどの種類の変更が、どの基準下で自動適用可能か」を事前に定義し、明文化することが、自動デプロイ運用の前提条件になる。

## 業務システム本体を扱わない理由

冒頭で対象範囲を「AIエージェント周辺のハーネス層」に限定した。なぜ業務システム本体は対象外なのか、改めて整理する。

第一に、改変対象が広がるほど、改変提案の評価コストが指数関数的に上昇する。エラーログから推測される「分類体系の誤り」の検証は数分でできるが、「業務システムのデータベース設計の誤り」の検証は数日を要する。トークンと運用コストの観点から、現実的に運用可能なスコープは狭く保たれる必要がある。

第二に、業務システム本体は AI 以外のステークホルダーが多すぎる。データベース設計、API インターフェース、認証フロー — こうした要素は AI エージェントの利用とは独立に存在し、独立に変更管理される。AI からの改変提案が業務システム本体に及ぶと、その提案を受ける側の組織体制が AI を前提に設計されていないため、運用が破綻する。

第三に、ハルシネーションのインパクトが許容外となる。スキル定義ファイルの誤った書き換えは、影響範囲がそのエージェントの範囲に限定される。一方、業務システムへの変更がハルシネーション由来で誤っていた場合、影響範囲は事業全体に及び得る。エージェント固有のハルシネーションは Lin, X. et al.(2025)によって体系的に分類・整理されており、OWASP も Top 10 for LLM Applications 2025 において LLM06: Excessive Agency(過剰な自律性付与)を独立したリスクカテゴリとして明文化している — いずれも、エージェントへの自律性付与とハルシネーション制御を独立に扱うことができない、という業界共通の認識を反映している。コーディングエージェントが PR を生成し人間がレビューする運用は、この非対称性に対して人間ゲートで対処しているが、それを **自律ループ** として企業の業務システム本体に回すことは、現時点では運用可能なリスク水準を超える。リスクの非対称性こそが、メタハーネスの対象範囲設定の根拠になる。

## 境界設計の担い手

ここまで論じてきた設計判断 — Type A/B 境界の引き直し、Layer 1/2 のスコープ整合、改善と仕様変更の区別、メタハーネスが機能する3条件の見極め、実装パターンの選定、自動適用と人間承認の線引き、改変対象の範囲設定 — は、いずれも純粋な技術判断ではない。業務文脈、組織構造、規制要件、運用熟度のすべてを参照した上で初めて成立する判断である。

そして本稿で繰り返し述べてきたとおり、これらの判断はメタハーネスではなくエンジニアの責任範囲に属する。メタハーネスは境界の内側で機能する仕組みとして設計するのが望ましく、境界そのものを設計するのは人間の役割として残す — これが本稿全体を貫く設計姿勢である。

この設計判断を担える人間は、現時点で組織内におそらくほとんど存在しない。クライアント企業の側は AI エージェントの内部設計を知らず、AI 製品側は顧客のドメイン文脈を知らない。両方を理解し、システム境界の設計判断ができる人材が要求される。

業界では、こうした職能の一つとして Forward Deployed Engineer(FDE)が注目されつつある。2026年に入って AI 主要ラボがいっせいに FDE 採用を加速させているのは、ここで論じてきたような設計判断を担える人材の不足が、エンタープライズ AI 導入の実質的なボトルネックになっているからだと観察できる。

ただし、FDE という職能名そのものは現時点でまだ流動的で、組織や文脈によって職務の重心は分かれている。本稿の論旨は、メタハーネスの設計判断を引き受ける役割が必要だということであって、それを FDE が独占的に担うと断定するものではない。社内エンジニア、業務責任者、コンサルタント、あるいはそれらが組み合わさったチーム — どの形態であれ、AIエージェントの運用と業務システムの両方を理解した者がこの境界設計を引き受ける必要がある、というのが本稿の主張である。

そのうえで予測を述べれば、本稿で論じたメタハーネスの構築は、今後数年において FDE の中核業務の一つになっていく可能性が高い。理由は単純で、運用業務と AI システムの双方を深く理解する職能が、ちょうどこの境界設計の要求と重なるからである。職能名がどう定着していくかは別として、設計問題自体は確実に残り続ける。

## 参考: 主要な文献的裏付け

### メタハーネスとハーネス自己改善の理論系統

* **Lee, Y., Nair, R., Zhang, Q., Lee, K., Khattab, O., Finn, C. (2026).** *Meta-Harness: End-to-End Optimization of Model Harnesses.* arXiv:2603.28052. — 評価関数駆動の outer loop による harness 自己改善の定式化。本稿はこれを企業文脈に持ち込んだときの制約と例外を論じている。
* **Lin, J. et al. (2026).** *Agentic Harness Engineering: Observability-Driven Automatic Evolution of Coding-Agent Harnesses.* arXiv:2604.25850. — Component / Experience / Decision の3層 observability により改変を falsifiable contract 化し、harness 自己改善ループが trial-and-error に崩壊するのを防ぐ設計。本稿の Layer 1/2 スコープと改変提案の評価可能性の議論に直接対応する。10 iterations で Terminal-Bench 2 の pass@1 を 69.7% → 77.0% に改善し、frozen harness が SWE-bench-verified に転送可能であることも示している。
* **Zhang, J. et al. (2026).** *Hyperagents.* arXiv:2603.19461. — Task agent と meta agent を単一の編集可能プログラムに統合し、メタレベル機構そのものを編集可能とすることで、coding 領域を超えた領域(論文査読、ロボティクス報酬設計、Olympiad 数学採点)への一般化を実証。本稿が「企業運用ではメタレベルを編集可能にすべきでない」と主張する対極の研究系統として、設計上の引き合いになる。
* **Robeyns, M. et al. (2025).** *A Self-Improving Coding Agent (SICA).* ICLR 2025 Workshop SSI-FM. arXiv:2504.15228. — SWE-Bench Verified という ground truth が確立された領域で、エージェントの自己改善を 17% から 53% へ引き上げ、メタエージェントと target エージェントの統合による自己改善ループの実現可能性を実証。本稿の「verifiable な評価関数下でのみメタハーネスが成立する」という命題の独立した実証データ。

### Bounded スコープと工具数依存性

* **Patil, S. G. et al. (2025).** *The Berkeley Function Calling Leaderboard (BFCL): From Tool Use to Agentic Evaluation of Large Language Models.* ICML 2025 (PMLR 267:48371-48392). — ツール数増加に伴う関数呼び出し精度の劣化を体系的に評価。Layer 1 スコープを設計時に固定すべき根拠の中心的論文。
* **Gan, T., Sun, Q. (2025).** *RAG-MCP: Mitigating Prompt Bloat in LLM Tool Selection via Retrieval-Augmented Generation.* arXiv:2505.03275. — MCP プール規模の拡大に伴うツール選択精度の単調な低下を独立に実証。Retrieval ベースのツール選択により、ツール選択精度が 13.62% から 43.13% へ改善することを示しており、「ツール数の問題は単なるトークン消費ではなく選択精度そのものの問題」であることを明示する。さらに position 100 以降では選択失敗が成功を上回る領域に入ることが報告されている。
* **Zhang, K. et al. (2026).** *Are Tools All We Need? Unveiling the Tool-Use Tax in LLM Agents.* arXiv:2605.00136. — 意味的に類似した distractor ツール存在下では、ツール追加そのものが精度を下げる "tool-use tax" として作用することを Factorized Intervention Framework で定式化。Layer 1 で「念のため」追加したツールが、想定外の文脈で他ツール選択を阻害する機序を理論化する。
* **LiveMCPBench (arXiv:2508.01780).** *Can Agents Navigate an Ocean of MCP Tools?* — 12 のフロンティア LLM を実環境の MCP タスクで評価。Claude-Sonnet-4 が 78.95% のタスク成功率に到達した一方で、大半のモデルは 30–50% の範囲に留まり、失敗の約半数がツール検索段階の誤りに起因することを報告。スケール下でのツール選択がボトルネックであることを独立に裏付ける。
* **Anthropic 内部測定** — 58 ツール定義が約 55,000 トークンを消費し、ツール数増加とともに選択精度が低下することを報告(anthropic.com/engineering/advanced-tool-use)。Tool Search Tool 導入による on-demand ツールロードで、Opus 4 の MCP 評価精度が 49% → 74%、Opus 4.5 が 79.5% → 88.1% に改善することも示されている。

### 構造化信号と prompt 最適化

* **Agrawal, L. A. et al. (2025).** *GEPA: Reflective Prompt Evolution Can Outperform Reinforcement Learning.* arXiv:2507.19457. — 構造化されたテキスト feedback(実行ログ、トレース、エラーメッセージ、制約違反等)を用いた reflective prompt 最適化が、スカラー報酬ベースの強化学習を rollout 効率で上回り得ることを実証。本稿の「信号源は観測可能で構造化されたデータに限定される」という要件と、それが自動最適化を成立させる根拠として直接的に対応する。

### エージェント固有のハルシネーションと自律性リスク

* **Lin, X. et al. (2025).** *LLM-based Agents Suffer from Hallucinations: A Survey of Taxonomy, Methods, and Directions.* arXiv:2509.18970. — エージェント固有のハルシネーションを体系的に分類した最初の包括的サーベイ。エージェントのワークフローを段階別に分解し、各段階で発生し得るハルシネーションのタイプを taxonomy 化している。本稿が業務システム本体を対象外に置く理由のうち、「ハルシネーション影響の非対称性」を裏付ける文献的根拠。
* **OWASP (2025).** *OWASP Top 10 for Large Language Model Applications 2025.* — LLM06 として "Excessive Agency"(過剰な自律性付与)を独立したリスクカテゴリとして明文化。LLM が API 呼び出し、データ操作、自律的判断を行うようになるにつれ、無監督のオートメーションが指数関数的にリスクを増幅させると指摘している。本稿の「自律ループを企業の業務システム本体に回すことは、現時点では運用可能なリスク水準を超える」という主張に対する業界規範文書としての裏付け。
