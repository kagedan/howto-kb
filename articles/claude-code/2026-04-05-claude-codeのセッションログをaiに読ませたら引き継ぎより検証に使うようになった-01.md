## はじめに

AIに自分の過去のセッションログを読ませています。セッションの引き継ぎ目的で始めましたが、運用するうちに用途が変わっていきました。AIが書いたIssueの背景を後から確認させる、作業パターンの漏れを検証させる。むしろこちらの方に価値を感じています。

この記事はその実践報告です。

## やっていること

セッションの会話ログはJSONLで自動保存されます。それを読み出すCLIツールを作り、CLAUDE.mdに書いておきます。

```
## セッションログ検索（推奨）

session search "keyword"
session list --since 2025-12-20
session view <session_id>
```

MCPは使っていません。AIがBash経由でコマンドを叩くだけです。session-logsスキル（jq+ripgrepでセッションログを検索するClaude Code用スキル）など同じ発想のものは既にあるので、専用CLIを作らなくても始められます。

ポイントは、ログを読むのが人間ではなくAIだということです。cclog、ccsearch、Claude Session Monitorなど、ログを人間が読むツールは既に揃っています。AIに読ませると、人間では現実的でない使い方ができるようになります。

## 引き継ぎだけなら既存手法でもできる

一番よく使っているのは引き継ぎで、毎セッション冒頭で「前回のログ読んで」と言っています。AIが直近のログを読んで前回の作業を把握し、続きから始める。

ただ、引き継ぎだけならhandover方式やAuto Memoryでもできます。セッション終了時に何もしなくていい手軽さと、任意の過去セッションに遡れる柔軟さは生ログならではですが、それだけなら記事にするほどの話ではありません。

## セッションログは防御壁

ベストプラクティスでは、判断の経緯もIssueやcommitメッセージに残します。AIにもそう指示する。でも実際には漏れます。Issueの記述から背景が抜け落ちていたり、commitメッセージが変更内容の羅列だけだったり。AIが書くアウトプットの品質は安定しません。

後から読んで「何これ、なんでこうなったの」となったとき、セッションログがあれば当時のやりとりに遡れます。Issueやcommitに書き漏れた経緯、検討して捨てた選択肢、その時点での前提条件。セッションログは、アウトプットの品質が漏れたときの防御壁として機能します。

この防御壁を、別のセッションのAIに読ませて活用する。これが運用の中で見つかった使い方でした。

## 具体的にどう使っているか

**Issueの背景確認。** AIに書かせたIssueの内容が、後から見ると文脈が抜け落ちていることがあります。別のセッションでAIに「このIssueが起票された当時のセッションログを読んで、背景を把握してほしい」と指示する。Issueには書かれなかった検討経緯や、採用しなかった選択肢まで含めて把握した上で、そのIssueの調査や実装に入れます。

**commitの意図の復元。** AIが作ったcommitのメッセージだけでは、なぜその変更が必要だったかがわからないことがあります。当時のセッションログをAIに読ませれば、「この変更はXXXの問題を解決するためにYYYのアプローチを取った、ZZZは検討したが却下した」という経緯が出てきます。

**作業パターンの漏れ検出。** ある工程が毎回漏れている気がしました。確証はなく、直感だけ。AIに「全リポジトリのセッションログから調べてほしい」と指示したら、横断検索して実施状況を調べてくれました。gitログやIssueには「工程をスキップした」という記録は残りませんが、セッションログにはAIがその工程を実行しなかった（あるいは言及すらしなかった）事実が残っています。

## なぜ生ログなのか

要約を作る手もありますが、それでは上記の使い方が成り立ちません。

要約は「次のセッションで必要そうな情報」を事前に選んで残すものです。しかし、後からIssueの背景を確認したくなるか、特定の工程の漏れを全件検証したくなるかは、その時点では予測できない。検討して捨てた選択肢は、要約には含まれません。

「何が必要になるかわからないから、全部残しておいてAIに判断させる」。これが生ログの利点です。

1Mコンテキストが使える今でも、ログを丸ごとコンテキストに入れるのはコンテキスト汚染の原因になります。CLIで必要な部分だけ読ませる方が精度は高い。

## まとめ

引き継ぎ目的で始めた仕組みですが、「AIが作ったアウトプットの背景を、別のAIに読ませて確認する」用途が予想外に広がりました。Issueの経緯確認、commitの意図復元、作業パターンの検証。スキルで手軽に始められるものもあるので、専用ツールを作らなくても試せます。

AIのアウトプットの品質は安定しません。セッションログは、その漏れに対する最後の防御壁になります。
