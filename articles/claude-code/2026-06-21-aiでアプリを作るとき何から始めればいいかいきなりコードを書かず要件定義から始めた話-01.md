## はじめに

将来が不安で仕方ない、どこにでもいる有象無象のサラリーマンです。

「AIに触っとかないとマジでやばい」という焦りから、家の夕飯当番をLINE Botで自動化した。この記事はその連載の②。テーマは、**作り始める前に何をしたか**。

具体的には「要件定義」「AI（Claude）の使い方」「環境構築のざっくり」を書く。手順やコードの細部、ハマった失敗は後続記事に送るので、ここは"作る前の地ならし"に徹する。

（全体像はこちら → [全体記事](https://zenn.dev/uzomuzo_salary/articles/74f72ee05c5b36)）

結論から言うと、一番やってよかったのは **いきなりコードを書かせず、要件定義から始めたこと**だった。

## やりがちな失敗：いきなり「作って」と言ってしまう

AIにアプリを作らせるとき、最初にやりたくなるのは「LINE Botを作って」と丸投げすること。だがこれをやると、AIは足りない情報を勝手に推測で埋める。出てきたものが自分の想定とズレていて、結局あとから直す羽目になる。

なので今回は、Claudeに対して最初にこう言った。

```
いきなり作り始めるのではなく、要件定義から始めたい。
まずは何から始めればよいか。
```

これで「作る」前に「決める」フェーズに入れた。地味だがこれが効いた。

## ステップ1：要件定義（ここが本体）

Claudeに質問を投げてもらいながら、ひとつずつ要件を固めていった。AIに「全部聞いて」と任せるのではなく、**1問1答で詰めていく**と、抜けが見つけやすい。

最終的に固まった要件はこう。実際に決めた内容をそのまま載せる。

### 誰が・何のために使うか

* 家族2人で使う（自分＋もう1人）
* 毎週の夕飯当番のメニュー決め〜レシピ確認を、LINEグループ上で回す

### 機能要件

* **日曜の昼**：その週のメニュー候補をLINEグループに自動送信
* **メニュー選択**：グループで番号またはメニュー名を送ると選択。**最後に選ばれたものが確定**（2人が別々を選んでも最後が勝つ）
* **選択状態のリアルタイム共有**：選ぶたびに「○○が選択されました」を全員に通知
* **月曜の昼**：その時点で確定しているメニューのレシピ（材料・手順）を自動送信
* **月曜の朝**：未選択ならリマインドを1回送信
* **レシピ管理**：1メニュー1ファイルの `.txt`（Markdown）。ファイルを足すだけで選択肢が増える

### 決めごと（仕様の細部）

* LINEはグループトークを使う
* 通知は全員に届く
* リマインドは月曜10:00に1回だけ
* 選択状態は毎週リセット

ポイントは、**「最後に選ばれたものが確定」みたいな細かいルールまで先に言語化した**こと。ここを曖昧にすると、実装後に「2人が違うの選んだらどうなる？」で必ず揉める。先に決めておくと、AIへの指示も一発で通る。

## ステップ2：技術スタックを決める

要件が固まってから、構成をClaudeに相談した。このとき大事なのは、**自分のスキルレベルを正直に伝える**こと。

自分はこう伝えた。

* Pythonは少し触れる
* サーバーのデプロイ経験はゼロ

これを伝えると、提案の粒度がそのレベルに合わせて調整される。背伸びして「経験あります」と言うと、説明が一気に省略されて詰む。正直が一番ラク。

結果、AIが選んだ構成はこう（詳細や、なぜここに落ち着いたかは後続記事で）。

* LINE Messaging API（窓口）
* Python + Flask（サーバー）
* Render（ホスティング、無料枠）
* 選択状態の保存先（当初はDB。後にスプレッドシートへ移行）
* 定期実行（当初はGitHub Actions。後にGASへ移行）

> 注：「後に移行」が2回出てくる通り、最初の構成は完成形ではない。そのへんの壊れた経緯が、このシリーズの失敗編の中身になる。

## ステップ3：環境構築（やったことの列挙）

ここは詰まりポイントが多いので、本記事では**やったことの一覧**にとどめる。各手順の細かい罠は別記事で供養する。

* **LINE Developers**：チャンネル作成、アクセストークン発行、Botをグループに追加、応答メッセージOFF
* **Render / GitHub**：アカウント作成、リポジトリ作成
* **ローカル環境**：Python導入、仮想環境（venv）、必要ライブラリのインストール
* **ngrok**：ローカルのサーバーを一時的に外部公開して、LINEのWebhook疎通を確認

このあたりは「公式手順どおりにやればいい」と思いきや、PowerShellで仮想環境が有効化できない、ngrokがアカウント認証を要求してくる、など細かい段差が地味に多かった。ここで心が折れかける人は多いと思う（自分は折れかけた）。

ただ、進め方そのものはシンプルだった。**基本はAIの言うとおりに手を動かすだけ**。そして詰まったら、自分で抱え込まずに**エラー文や不明点のスクリーンショットを全部AIに投げる**。これで大半は解決した。

具体的には、

* ターミナルに出たエラーは、そのままコピペするかスクショで貼る
* 画面のどのボタンを押せばいいか分からない設定画面も、スクショを撮って「これはどこを押せばいい？」と聞く
* 「この警告は許可していい？」みたいな判断も、画面ごと投げて確認する

自分で原因を調べてからAIに聞く、みたいな丁寧なことはしていない。**わからん状態のまま、画面をそのまま見せる**のが一番速かった。エラーの解読こそAIが得意なところなので、ここは遠慮なく頼っていい。

## ステップ4：実装はAIに投げる

環境が整ったら、実装はClaude Codeに任せた。丸投げではなく、**要件定義で固めた内容を、具体的な指示に落として**渡す。実際に投げた依頼文がこれ。

```
以下の構成でLINE Botのバックエンドを作成してください。

## 構成
- Python + Flask
- LINE Messaging API（line-bot-sdk）
- 定期実行（スケジューラ）
- 選択状態の保持（DB）

## 機能要件
- LINEグループへのWebhook受信
- レシピフォルダ内の .txt ファイル名一覧をメニューとして送信
- ユーザーがメニュー名または番号を送信したら選択状態を更新し、
  グループ全員に「○○ が選択されました」と通知
- 選択状態は保持し、毎週日曜にリセット
- 日曜12:00にメニュー一覧を自動送信
- 月曜10:00に未選択ならリマインド
- 月曜12:00に選択済みメニューのレシピを送信

## 環境変数（.envから読み込む）
- LINE_CHANNEL_SECRET
- LINE_CHANNEL_ACCESS_TOKEN
- LINE_GROUP_ID
```

要件定義をちゃんとやっておくと、この指示文が**そのまま要件のコピペで済む**。「要件定義は実装の下書き」だと実感した。

さらに言うと、**このClaude Codeへの指示文すら、自分でゼロから書いていない**。要件定義のやり取りをしていたClaude（チャットの方）に「Claude Codeに投げるプロンプトを作って」と頼んで、出てきたものをそのまま貼っただけ。

つまり流れはこう。

```
Claude（チャット）と要件定義する
   ↓
そのClaudeに「Claude Code用の指示文を作って」と頼む
   ↓
出てきた指示文を Claude Code に貼る
   ↓
コードが生成される
```

「AIに渡す指示をAIに書かせる」。横着に聞こえるが、要件をいちばん理解しているのは直前まで会話していたAI自身なので、これが理にかなっていた。自分は内容を読んで確認するだけでよかった。

## 学び・次に活かすこと

* **「作って」の前に「決めて」**。要件定義をAIと一緒にやるだけで、手戻りが激減する
* **細かいルールほど先に言語化する**（最後の選択が確定、未選択時はどうする、など）。曖昧なまま進むと必ず後で揉める
* **自分のスキルを正直に伝える**と、AIの説明の粒度が自分に合う。見栄を張ると詰む
* 要件定義をしっかりやると、それがそのまま**実装指示のテンプレ**になる
* **詰まったら抱え込まず、エラー文も設定画面もスクショで丸ごとAIに投げる**。自分で調べてから聞く必要はない
* **AIに渡す指示は、AIに書かせていい**。要件を理解しているAI自身に指示文を作らせると速い

次は、なぜそもそもこんなことを始めたのか──将来不安とAIの話を、少しだけ正直に書く（→ [動機の記事](https://zenn.dev/uzomuzo_salary/articles/8b387618f189d4)）。そのあと、作る過程でぶつかった失敗を1つずつ供養していく。
