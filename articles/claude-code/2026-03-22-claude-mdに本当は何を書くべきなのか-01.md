## 要点

CLAUDE.mdは「システムプロンプト」ではなく「セッション開始時のユーザーメッセージとして注入される」もの。会話が進むにつれ影響力は薄れるため、セッション全体を通じて守らせたいルールの置き場には向かない。

## CLAUDE.mdに書くべき内容

- **プロジェクト概要**: 何のためのプロジェクトで、何をしているか
- **モジュール構成**: 名前から推測しにくいディレクトリ構造の説明
- **セッション開始時の手順**: worktree作成など、毎回実行される初期化処理

## CLAUDE.mdに書くべきでない内容

- セッション全体を通じて守らせたいルール（テスト方針、コミットメッセージ、命名規則など）
- 特定の用途でしか使わない情報（開発時のみ、レビュー時のみの指示）
- 量の多い情報（毎セッション無条件にコンテキストを消費するため）

## ルールの正しい置き場

セッション全体で守らせたいルールは `.claude/rules/` ディレクトリにconditional rulesとして配置する。該当ファイルを初めて扱うタイミングで注入されるため、CLAUDE.mdより新しいメッセージとして機能する。

```yaml
paths:
  - "src/api/**/*.ts"
```

のようにYAML frontmatterの`paths`フィールドで、特定ファイル操作時のみ読み込ませることが可能。

## 結論

CLAUDE.mdに何を書くかは「毎回の会話で最初に伝えるべきこと」への答え。本当に必要な初期情報だけを厳選して記載するのが、トークン効率とルール遵守の両面で最適。
