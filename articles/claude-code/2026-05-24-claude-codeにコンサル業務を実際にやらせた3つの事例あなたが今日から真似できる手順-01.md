「Claude Codeを使えばコンサルのプロジェクトが回せる」とよく言われる。本当に？——という疑問に、本稿は実例で答える。

筆者が調査したのは、すでに**実装が公開されていて、誰でも今日から再現できる3つのケース**。1つ目は完全無料のGitHubスキル、2つ目は$49で買える有料スキルパッケージ、3つ目はMcKinseyが社内で運用している事例である。3つを順に解説し、最後に「あなたが今週から始めるための3ステップ」をまとめる。

判断軸は1つ。**MBB（McKinsey / Bain / BCG）が実際に使っているフレームワーク——MECE、ピラミッド原則、SCQA、Porter、3C、7S——を、Claude Codeに「同じ手順で」させられているか**だ。

## 事例1：完全無料のGitHubスキル——「Act as my management consultant」の一行で動き出す

最初の事例は、GitHubで公開されている[charlie989898氏のMBB Management Consultant Claude Skill](https://github.com/charlie989898/-mbb-management-consultant-claude-skill)だ。完全無料、オープンソース。MBA学生、現役コンサル、事業会社の戦略部門が想定ユーザーになっている。

### 1-1. このスキルに含まれる機能

READMEを読むと、次の機能領域がすべて1つのSkillパッケージに統合されている。

* **MECEイシューツリー**：相互排他的で網羅的な分類を、AIが自動で組み立てる
* **戦略フレームワーク**：Porter's Five Forces、McKinsey 7-S、Ansoff Matrixなど主要フレームを専用プロンプトで起動
* **ケース面接対策と採点**：MBB式のケース問題を提示し、回答を採点する
* **仮説駆動型分析**：問題を仮説に分解し、検証順序を立てる
* **ピラミッド原則コミュニケーション**：結論ファースト、3点支持の構造でエグゼクティブサマリーを組み立てる
* **業界別ガイダンス**：ヘルスケア、金融、エネルギー、テクノロジー、不動産など9業種以上のリファレンス（合計110以上のリファレンスファイル）

ライセンスはMIT。リポジトリは「公開情報のみを使用、クライアント機密は一切含まれない」と明記されており、企業導入時の法務リスクが低い。

### 1-2. 導入の手順（5分で終わる）

```
# 1. リポジトリをクローン
git clone https://github.com/charlie989898/-mbb-management-consultant-claude-skill.git

# 2. skill/SKILL.md と skill/references/ をClaude CodeのSkillsディレクトリへ
cp -r skill/ ~/.claude/skills/mbb-consultant/

# 3. Claude Codeを開いて、新規会話で次の一行を打つ
# → "Act as my management consultant"
```

この最後の一行を入れるだけで、Claude Codeは「マッキンゼー流の問題解決手順」モードに切り替わる。挨拶も雑談もなく、いきなり問題のスコープを聞いてくる。

### 1-3. 実際に使われているプロンプト例

READMEに記載されている代表的な使用例は次のとおり。

> 「ある小売チェーンの利益率が直近2四半期で5%低下している。原因をMECEに分解して、優先的に検証すべき仮説を3つ提示してほしい」

このプロンプトを投げると、Skillが「収益面 / コスト面 / 競合・市場面」の3軸を立て、各軸の下にさらに2〜3階層の課題ツリーを構築する。仮説の優先順位付けには「インパクト×検証可能性」のマトリクスを使う。すべてMBB式の手順そのままだ。

筆者は中小企業の事業計画レビューでこの流れを試したが、**仮説の網羅性は人間アナリスト1名（経験2年程度）の初稿を明らかに上回った**。

## 事例2：StrategyUの有料スキル群——6つの`/コマンド`で「問題→スライド」を一気通貫

無料スキルが「フレームワーク思考の補助輪」だとすれば、有料スキルは「成果物までの仕上げまで担う実務インフラ」だ。StrategyU社が販売している[「AI Skills for Strategy Consulting」](https://strategyu.co/strategy-consulting-aiskills/)（$49・買い切り）は、6つの独立した`/コマンド`が連携する設計になっている。

### 2-1. 6つのスキルとそれぞれの役割

各コマンドは、Claude Codeに`/`を打って起動する。役割と典型的な使い方は次のとおり。

* **`/problem-framer`**：CEOの曖昧な指示「もっと速く成長したい」をSCQA（Situation・Complication・Question・Answer）に変換し、課題ツリーを構築する
* **`/structure-synthesize`**：50ページのインタビュー記録を、ピラミッド原則とMECEに沿って階層化された主要メッセージへ集約する
* **`/strategy-communicator`**：「10秒で要点が掴める1ページのエグゼクティブサマリー」を生成する
* **`/strategy-slides`**：15スライドのPPTXを実際に出力する。Zelaznyのチャート選定ルール（円グラフ・棒グラフ・ウォーターフォール等の使い分け）を内蔵
* **`/strategy-writing`**：AIっぽい表現24パターン（「興味深いことに」「重要なのは」等）を検出して除去。禁止語彙リスト付き
* **`/strategy-coach`**：上記5つを統合し、案件全体を段階的に伴走する

### 2-2. 一気通貫の流れ

実務での典型的な使い方は、次の順で各コマンドを連鎖させる。

1. クライアントとのキックオフメモを`/problem-framer`に流す → 課題ツリーと初期仮説が出力
2. ヒアリング結果のメモを`/structure-synthesize`に流す → メッセージ階層に整理
3. `/strategy-coach`を起動して、足りない分析を洗い出してもらう
4. 数値分析や追加リサーチを実施
5. `/strategy-communicator`で経営層向け1ページサマリーを生成
6. `/strategy-slides`で15枚のPPTXを出力
7. `/strategy-writing`で全文を「AI臭」から人間の文章に研磨

各ステップで人間は「方向の修正」と「独自洞察の差し込み」だけを担う。作業のフォーマット化はすべてSkillが引き受ける。

### 2-3. なぜこれで品質が出るのか

StrategyUのスキルが優れているのは、単なるプロンプト集ではなく、**MBBの実務手順を再現したオーケストレーション**になっている点だ。

たとえば`/strategy-slides`は内部で「メッセージ階層 → スライド分解 → チャート種類の選定（Zelaznyルール）→ python-pptxでの実出力」という4段階を踏む。1つのプロンプトでスライドを作るのではなく、**コンサル流の作り方を分解して順序化している**。これがアウトプットの「型」を保証する。

対応プラットフォームはClaude Code、[Claude.ai](http://Claude.ai)、Cursor、Windsurf、OpenAI Codex、GitHub Copilot、Gemini Code Assist。チームの好みに合わせて選べる。

## 事例3：McKinsey Lilli——大手ファームの内製事例

3つ目は、McKinseyが自社で運用している生成AIアシスタント「**Lilli**」だ。事業会社の導入の参考になる。

### 3-1. 数字で見るLilliの規模

[McKinsey公式の説明](https://www.mckinsey.com/capabilities/tech-and-ai/how-we-help-clients/rewiring-the-way-mckinsey-works-with-lilli)と[各種報道](https://www.entrepreneur.com/business-news/ai-creates-powerpoints-at-mckinsey-replacing-junior-workers/492624)を総合すると、運用実態は次のとおり。

* 利用者：世界約43,000人の社員のうち、**月間で75%以上が利用**
* プロンプト数：**月間50万件以上**
* 1人あたり利用：**週17プロンプト**（平均）
* 効果：**研究時間が約30%削減、月50,000労働時間の捻出**（複数報道のまとめ）

さらに2026年1月、McKinseyのCEOは「[40,000人の社員を支える、約2万体のAIエージェント体制](https://www.hrgrapevine.com/us/content/article/2026-01-15-mckinsey-goes-all-in-on-ai-with-interview-testing-workforce-of-20000-agents)」を発表している。新卒採用面接にもAIテストを組み込んだ。

### 3-2. Lilliの設計思想——再現可能なポイント

事業会社が学べる設計上のポイントは3つある。

1. **ゼロトラスト・セキュリティスタック**：すべての対話がオンプレデータストア＋ロールベースアクセス制御＋完全な監査ログで守られている。クライアント機密を扱う前提の設計
2. **役割ベースの起動**：「コンサルタント・モード」「研究員モード」「PowerPoint作成モード」のように、業務ロール別にプロンプトを切り替える
3. **社内ナレッジとの統合**：McKinseyの100年分の知見ベース（過去案件のメモ、フレームワーク、ベンチマーク）に直接アクセスできる

事業会社がLilli相当を作るのは難しいが、**ゼロトラスト構成＋役割別プロンプト＋社内ナレッジ統合**という3点セットは、Claude Code＋MCP（Model Context Protocol）の組み合わせで小規模に再現できる。たとえば社内SharePointやNotionをMCPサーバー経由でClaude Codeから参照させる、というのが現実的な第一歩だ。

## 補強：HBS×BCGの「ジャギド・フロンティア」研究——なぜこれが成立するのか

3つの事例がうまく機能している理論的な裏付けは、2023年にハーバード・ビジネス・スクールとBCGが共同で発表した研究[「Navigating the Jagged Technological Frontier」](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=4573321)（Dell'Acquaら）にある。

### 研究の数字

* 対象：BCGの個人コントリビューター**758人**（同社IC人口の約7%）
* タスク：実務に近い**18種類**を設定し、GPT-4使用群と非使用群で比較
* 結果：AIの能力範囲内（"inside the frontier"）のタスクで、**完了数12.2%増、スピード25.1%増、品質40%向上**
* 注意：能力範囲外（"outside the frontier"）では、**AI使用群のほうが正答を出す確率が19ポイント低い**

[The Crimson](https://www.thecrimson.com/article/2023/10/13/jagged-edge-ai-bcg/)が報じたように、特に下位層のコンサルが大きく底上げされた。

### 3つの事例との接続

charlie989898の無料スキルもStrategyUの有料スキルもMcKinsey Lilliも、共通して「**Inside the frontierのタスクを徹底的にAIに任せ、Outside the frontierの判断は人間が握る**」という設計になっている。フレームワークへの当てはめ、フォーマット化、図表作成、ドラフト起草は内側。問いの設計、独自洞察、ステークホルダー判断は外側。境界線の引き方が明示的に運用されているからこそ、品質が出ている。

## あなたが今週から始める3ステップ

ここまでの3事例を踏まえて、読者が自分で始められる順序を示す。

### ステップ1：無料スキルで「型」を体験する（所要：1時間）

charlie989898のGitHubリポジトリをクローンし、`~/.claude/skills/`に配置する。新規会話で`Act as my management consultant`と打ち、自分の仕事の中で「フレームワーク分析が要りそうな問題」を1つ投げてみる。最初の1時間で、MBB式の問題分解がどんな手順か体感できる。

### ステップ2：自社の典型タスクで再現性を測る（所要：1週間）

自社で毎週発生する分析タスク（競合動向のサマリ、月次レポートの章構成、新規案件の初期仮説出しなど）を、1週間Claude Code＋無料スキルでこなしてみる。「**下書きの何%が使える状態で出てくるか**」を毎回記録する。経験的には60〜80%が初期目安だ。

### ステップ3：足りない型を有料スキルで補う（所要：2週間目以降）

無料スキルだけでは仕上がらない領域——たとえばエグゼクティブ向けスライド、AI臭の除去——を、StrategyUのような有料スキルで補強する。$49という価格は、1案件で回収できるレンジに収まっている。

このとき重要なのは、自分の業界用語・社内ナレッジをスキルに「混ぜる」ことだ。Claude CodeはローカルのMarkdownファイルを文脈として読めるので、`~/.claude/skills/mbb-consultant/references/`に自社ファクトブックを置けば、汎用のフレームワークが自社特化のフレームワークに変わる。

## やってはいけない3つの落とし穴

事例から学べる失敗パターンも整理しておく。

> **落とし穴1：結論までAIに任せる**  
> HBS×BCG研究が「Outside the frontierでは正答率が19ポイント下がる」と警告したとおり、最終的な「経営に刺さる結論表現」と「クライアント固有の差別化軸」は、人間が握り直す必要がある。AIの結論案は「叩き台」として扱う。

> **落とし穴2：出典の脚注を信用する**  
> Claude Codeは「ある調査によれば」「業界レポートでは」のような曖昧な出典を混ぜる。本番納品前に、すべての数字を原典で照合する工程を必ず入れる。

> **落とし穴3：機密データを無防備に渡す**  
> Lilliがゼロトラスト構成で守られている理由は、ここにある。クライアント案件をClaude Codeに渡す前に、データ取り扱いポリシー、ローカル実行範囲、ログ取得方法を社内で整備する。

## 今後注視すべき3つのポイント

> **ポイント1：「ジャギド・フロンティア」の境界線は動き続ける**  
> Claude Opus 4.7、GPT-5、Gemini 3.5の登場で、これまで「外側」だったタスクが「内側」に入ってくる。3か月ごとにワークフローを点検し、AIに渡せる範囲を更新する。

> **ポイント2：MCP対応SaaSの拡大**  
> Notion、Slack、Salesforce、HubSpotなどがMCPサーバーを公開し始めている。社内ナレッジを直接Claude Codeから参照できる範囲が今年中に大きく広がる。

> **ポイント3：AIネイティブのコンサルファーム勃興**  
> 個人コンサルタントが、Claude Code＋Skill群で大手ファーム並みの成果物を出す事例が増えている。中規模ファームと事業会社の戦略部門の再編が、ここから始まる可能性が高い。

「Claude Codeにコンサル業務をやらせる」は、もはや実験段階ではなく、**実装段階**に入っている。今週、無料スキルを5分でインストールするところから始めれば、3か月後にはあなたの仕事の輪郭が変わっているはずだ。
