ハーネスエンジニアリングは、AIエージェントの出力品質をプロンプト単体ではなく周辺環境（ルール・コマンド・ワークフロー）の設計で改善していく手法です。  
プロジェクトにより実践の形は様々ですが、自分の場合はコマンド3つだけで構成しています。その運用サイクルと得られた知見を共有します。

## ハーネスエンジニアリングの3つのコアコマンド

ハーネスエンジニアリングで日常的に使用しているコマンドは **`log → distill → promote`** の3つだけです。  
あとは状態把握やメンテナンスの補助コマンドが少しあるだけです。3つのコマンドの役割、フィードバックループ一回分の実際の流れ、運用を通じて実感した変化を共有します。

---

## `/harness:*` コマンドで行うフィードバックループ

実際に使用しているコマンド名を共有します。 名前と役割から、ワークフローの骨格をつかんでいただけるかと思います。

| コマンド | 役割 |
| --- | --- |
| `/harness:log` | Claudeの改善点を、そのチャットでの会話に基づいてジャーナル（改善記録）として書き出す |
| `/harness:distill` | ジャーナルを読み、共通するパターンをルールやエージェント定義の候補に蒸留する |
| `/harness:promote` | ローカルで効果的だったルールやエージェント定義を、人が選定して共有設定とする |

> **ディレクトリ構成の前提**
>
> ```
> .claude/
> ├── agents/         ← チーム共有（git管理）
> │   └── local/      ← 個人用（gitignore）
> ├── commands/       ← チーム共有（git管理）
> │   └── local/      ← 個人用（gitignore）
> ├── rules/          ← チーム共有（git管理）
> │   └── local/      ← 個人用（gitignore）
> └── harness/
>     └── log/        ← ジャーナル（gitignore）
> ```
>
> ルールとコマンドには **「チーム共有」と「個人用（`local/`）」の2段** があります。`/harness:log` はジャーナルを `harness/log/` に書き出し、`/harness:distill` がそれを 各 `local/` ディレクトリに出力します。`/harness:promote` は、個人用として効果を感じたものを1段上のチーム共有（`.claude/rules/`）に引き上げる操作です。

### フィードバックループ一回分のサイクル

例えばこういうことが起きたとします。

* **Claude** が、特定のlintルールに沿わないコードを出力する。 **開発者** がその場で指摘し、 **Claude** に修正してもらう。
* 修正が終わったあとに **開発者** が `/harness:log` を実行することにより、「何が起きたか」「どういう条件で起きたか」などがジャーナルに出力される。
* 任意のタイミングで **開発者** が `/harness:distill` を実行すると、 **Claude** がジャーナル群の中から「lint違反を作り込みやすい頻発パターン」を候補ルールとして抽出される。
* **開発者** がその候補を `/harness:promote` でチームの共有ルールに昇格させると、次の週からは **Claudeの前提知識に含まれることになる** 。

### ジャーナル出力からチーム共有までのワークフロー

各ステップが何を生み出すか、フォーマットの例を示します。

**`/harness:log` のジャーナル出力例:**

```
- date: 2026-04-01
- event: Claudeがクォートスタイルの規約に沿わないコードを書いた
- context: モデル層の編集時
- frequency: 今週3回目
```

**`/harness:distill` の出力例（候補ルール）:**

```
- source: log 3件
- pattern: 文字列リテラルのクォートスタイルを守らない
- proposed rule: 文字列は式展開がない限りシングルクォートを使う
```

**`/harness:promote` 後のルール:**

```
## Quote Style
- 文字列は式展開がない限りシングルクォートを使う
- 違反パターン: "string" → 修正: 'string'
```

この3段階を見ると、ジャーナルがルールとして組み込まれる過程がイメージしやすいかと思います。

## フィードバックループで変わったこと

このフィードバックループにより、「テストで気づく」→「lintで気づく」→ **「そもそもClaudeがそう書かなくなる」** というところまで前倒しになります。Böckelerの言う "keep quality left"（品質チェックをできるだけ上流に寄せる）が自然に達成される感覚です。

以前は同じ指摘を何度もしていたものが、ルールとして定着するとほとんど再発しなくなりました。指摘の繰り返しがなくなると、会話の中でより本質的な設計の議論に時間を使えるようになったと感じています。

## 補助コマンド

本筋は `log → distill → promote` の3つですが、運用の中であると便利な補助コマンドもいくつか用意しています。

| コマンド | 役割 | いつ使うか |
| --- | --- | --- |
| `/harness:status` | 現在のハーネスの状態を俯瞰する | ハーネスがどれくらい育ったか見たいとき |
| `/harness:score` | ハーネスのカバレッジ・健全度を数値化する | チームに進捗を共有するとき |
| `/harness:rebuild` | 蒸留された信号から、ルール／コマンド／スキルを再構築する | ルール群が肥大化してきたとき |
| `/harness:clean-local` | 意図からずれたローカル調整や古くなった設定を整理する | ローカルの `local/` 配下が散らかってきたとき |

補助コマンドは **必須ではありません** 。まずは `log → distill → promote` の3つだけでも大丈夫です。ハーネスが大きくなってきたら、必要に応じて足していけばいいと思います。

## まとめ

* ハーネスエンジニアリングは `log → distill → promote` の3コマンドで行われています
* 「その場の指摘」をジャーナルに記録し、蒸留してルール化することで、同じ指摘の繰り返しがなくなっていきます
* 結果として、Claudeとの会話が指摘から設計議論へとシフトしていく感覚があります

あくまで運用例なので万能ではないと思いますが、ハーネスエンジニアリングをこれから始める方の参考になればと思います。
