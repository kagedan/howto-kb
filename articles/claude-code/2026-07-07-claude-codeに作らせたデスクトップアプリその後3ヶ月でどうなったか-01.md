## はじめに

3ヶ月ほど前、[Claude Code にデスクトップアプリを作らせたら7日で初リリースできた](https://zenn.dev/kashioka/articles/ec6d8434468bc5)という記事を書きました。Markdown 読み専用のデスクトップアプリ **Rendu**（Tauri + React）を、コードの 99% を AI に書かせて作った話です。

あの時点で書いたのは「作り始めてから最初のリリースまで」でした。じゃあ**その後、放置されずに動き続けたのか**、**AI に書かせたコードは運用に耐えたのか**。これは、そこそこリアルな「AI 製 OSS のその後」の記録です。

結論から言うと、一番面白かったのは **「AI が書いたコードを AI に監査させたら、本物の脆弱性（1クリックで任意コード実行）が出てきた」** ことでした。この話をメインに据えつつ、3ヶ月ぶんの数字と失敗を並べていきます。

## まず数字（前回との比較）

|  | 前回記事の時点 | 現在（v0.10.0） |
| --- | --- | --- |
| コミット数 | 112 | **199** |
| マージ済み PR | ??? | **86** |
| stable リリース | 3 | **15**（beta 込みで 27） |
| テストケース | 174 | **280**（TS 236 + Rust 44） |
| 外部コントリビューター | 0 | **2** |
| コード（テスト除く） | 約 3.2K 行 | 約 5.3K 行 |
| テストコード | — | 約 2.5K 行 |

相変わらず、機能コードとほぼ同量のテストが AI から生成されています。そして**コードを書いているのは今でも 99% AI**です。私がやっているのは、後述するとおり「何を作るか」と「本当に正しいか」の判断だけ。

3ヶ月で足された主なもの：

* **KaTeX 数式レンダリング**（`$$...$$`）
* **内部 Markdown リンクのアプリ内遷移**（`[link](./other.md)` をクリックすると履歴付きで移動）
* **起動高速化**（数式を含まない文書では KaTeX を一切ロードしない遅延読み込み）
* **多言語対応**（簡体字中国語 + System テーマ）← これは**外部コントリビューター**が入れてくれた
* **セキュリティ体制**（Dependabot / cargo audit / npm audit / 週次スキャン / SECURITY.md）

## 「その後」に起きた、地味だけど大きな変化

### 1. PRが来た（しかも中国から）

前回記事の最後に「AI が書いたコードって言うと微妙な空気になる」という社会的気まずさの話を書きました。その後、ある日 GitHub の通知が来ていて、開いたら**見知らぬ中国の方からの Pull Request** でした。内容は**簡体字中国語ロケール**の追加。

正直、外部からPRいただけるのを想定していなかったのでちょっと慌てました。AI 製と明言している個人 OSS に、海の向こうから人が貢献してくれる。こんなことある？と。……で、ここからが**あたふた**タイムです。まず私は中国語が読めない。追加された翻訳が妥当なのか、その場で判断できない。しかもその PR には「System テーマ（OS のダーク/ライト追従）」まで一緒に入っていて、こっちはこっちで既存のテーマ処理に食い込む変更でした。

結果、ロケール追加だけだったらわかりやすかったのにと思いつつ、**カラーピッカーが消える / Dark と Light の配色が混ざる / リセット導線がない**といった細かい問題が出て、フォローアップの PR もいれて。英語でメッセージを送りつつ、いつものようにAI に「この Tauri のイベント API の使い方、公式の推奨パターンと合ってる？」と裏取りをさせつつ、なんとかマージまで漕ぎ着けました。

ここで痛感したのは、**外部 PR のレビューこそ人間（私）の仕事**だということです。十分にアプリの方向がAIに伝わっていればいいですが、なかなかAIに未来像や方向性は伝えきれません、他人のコードはなおさらで、この修正が必要なのか否か、Renduにあっているのか否か判断することが必要です。「AI にレビューさせればいい」で済まない領域が、人が来た瞬間に初めて立ち上がった。嬉しさとあたふたが同時に来る、いい経験でした。

### 2. セキュリティ運用が「1人 + AI」で回るようになった

個人開発でいちばん後回しになりがちなのがセキュリティ運用です。でも今は、

* Dependabot が毎週 npm / cargo / GitHub Actions の更新 PR を出す
* CI で `npm audit` と `cargo audit` が回る
* 週次スキャンで脆弱性が出たら Issue が自動で立つ

という体制を、AI に組ませて維持できています。実際、この 3 ヶ月で **quick-xml の RUSTSEC 脆弱性**（依存の依存で、上流が semver 非互換のため即上げられない）や **DOMPurify のランタイム XSS** が飛んできて、その都度「ignore して追跡 Issue を立てる」「override で潰す」といった判断をしました。判断はするけど、調査と実装は AI がやる。**個人開発の守備範囲が、明らかに広がった**という実感があります。

## 本編：AI 製コードを AI に監査させたら、本物の脆弱性が出た

ここが今回いちばん書きたかった話です。

アプリがある程度成熟したのでfable5で熱狂しているタイミングだったので、「一度、全ソースをレビューしよう」と思い立ちました。やり方は、**4 つのレビューエージェントを並列で走らせる**（Rust バックエンド / フロントエンド中核 / フロントエンド周辺 / 設定・CI）というもの。それぞれに領域を割り当てて、独立に指摘を出させました。

すると、3 系統から**独立に同じ指摘**が上がってきました。要約するとこうです：

> Markdown 内のリンク `[付録](../../evil.command)` をクリックすると、`..` でワークスペース外に出たローカルパスが OS の「デフォルトアプリで開く」に渡され、macOS では `.command` ファイルが**実行される**。

つまり、**信頼できない .md ファイルを開いてリンクを 1 回クリックするだけでコード実行が成立する**。しかも HTML サニタイズを回避する必要すらない（`<a href>` の相対リンクは普通に通るので）。これは AI が書いたコードで、私が「内部リンク遷移を実装して」と気軽に指示した結果でした。

正直、ヒヤッとしました。と同時に、面白いとも思いました。**AI が書いたコードの穴を、AI のレビューが見つけた**わけです。

### 「AI が見つけた」で終わらせない

ここで大事なのは、レビューエージェントの指摘を**鵜呑みにしない**ことでした。LLM のレビューは「もっともらしいが間違っている」指摘も普通に混ぜてきます。なので、指摘が来たら私が実際のコードと突き合わせ、さらに **Playwright で実ブラウザを立ち上げて「本当に 1 クリックで実行されるか」を再現**して確定させました。再現できたものだけを「本物」として扱う。

修正は多層防御にしました：

1. フロントで未知スキーム（`file:` / `vscode:` など）を弾く
2. 開いていいのは PDF・画像・文書などの受動的な拡張子だけ。それ以外は確認ダイアログ
3. **Rust 側で実行可能拡張子（`.command` / `.exe` / `.app` / `.pkg` …）をハード拒否**（万一フロントを突破されても最後の砦になる）

そして修正 diff を、今度は**敵対的にレビューさせる**。Codex（別の AI レビュアー）に「この拒否リスト、抜けはないか」と投げると、「`.pkg` と `.msix` が抜けている」→ 追加 → 「まだ ClickOnce の `.application` が抜けている」と、ラウンドを重ねて詰めてくれました。ここでも、**どこで打ち止めにするかを決めるのは人間**です（拒否リストは無限に伸ばせるが、本命の防御は allowlist 側なので、と判断して 3 ラウンドで止めた）。

同じレビューで、ほかにも「更新通知の URL 無検証（`file://` も通る）」「webview に `$HOME` 全域への書き込み権限を渡していた」といった穴が出てきて、まとめて塞ぎました。

## CI が通っても実機で壊れる、という当たり前の話

セキュリティ修正のあと、**実機検証**をしました。ここで、テストも CI も全部グリーンなのに**実機でしか出ない**バグが 2 つ見つかりました。

**その1：PDF 出力のダイアログが固まる。**  
PDF の保存先ダイアログを Rust 側から出すようにしたら、ダイアログは表示されるのに OK ボタンがグレーアウトして操作できない。原因は **Tauri v2 の同期コマンドはメインスレッドで動く**という仕様を知らなかったこと。その中でネイティブの保存ダイアログ（これもメインスレッドを要求する）を呼ぶと、自分で自分をブロックしてデッドロックする。`async` 化して別スレッドに逃がしたら直りました。

**その2：Mermaid 図のノードが大量に消える。**  
複数行ラベルを含むフローチャートを開くと、矢印とテキストだけ残ってボックスが消える。これも実機で報告を受けて発覚。原因を **Chromium と jsdom の両方で再現**して特定したところ、Mermaid が SVG ラベルに吐く HTML の閉じない `<br>` を、こちらが**厳格な XML パーサ**で再パースしていて、最初のエラーで図の大半が切り捨てられていた。パーサを `text/html` に変えるだけで直りました（26 ノード中 8 しか描画されていなかったのが、26 全部出るように）。

どちらも、**テストと CI をすり抜けて実機だけで出た**バグです。前回記事で「AI は手であって頭ではない」と書きましたが、その「頭」の仕事には、**最後に自分の手で実物を触る**ことが確実に含まれる。ここは自動化しきれない。

## 直せないものもある（正直な話）

全部きれいに直ったわけではありません。

* **リリースの途中で Apple Developer の契約が失効していて、公証が 403 で落ちた。** これはコードの問題ではなく、developer.apple.com で契約に同意し直すしかない。AI にはどうにもできない領域。
* **旧バージョンの「更新通知の URL」は直せない。** OS 別の直リンクを出す修正を入れたけど、それは新しいバージョンにしか載らない。**すでに出荷済みのバイナリの挙動は、コードでは変えられない**。前進方向でしか直らないので、旧バージョンから来た人向けにはリリースノートの冒頭を工夫する、くらいしか打つ手がなかった。

こういう「技術では解決しない／後追いでしか直らない」問題があるのは、3 ヶ月運用してみて実感したことです。

## メタな話：「判断の質」から「検証の質」へ

前回記事の結論は、\*\*「AI は手であって頭。何を作るかを決める判断の質が差になる」\*\*でした。3 ヶ月運用して、この結論を少しアップデートしたくなりました。

コードを書くコストがほぼゼロになると、ボトルネックは「書く」ことから **「本当に正しいか確かめる」** ことに移ります。今回の 3 ヶ月で私が実際にやっていたのは、ほとんどが検証でした：

* レビューエージェントの指摘を**鵜呑みにせず実コードと実挙動で確かめる**
* 修正を**敵対的にレビューさせる**（「反証してみろ」と投げる）
* 複数エージェントに**多角的に**見させる
* そして最後に**自分の手で実機を触る**

「何を作るか」の判断はもちろん今でも人間の仕事です。でもそれ以上に、**「AI が出してきたものを、どう疑い、どう確かめるか」** が個人開発の質を決めるようになった。速度が上がったぶん、検証と責任はむしろ重くなっている、というのが 3 ヶ月後の実感です。

AI が書いたコードだから危ない、のでもない。AI がレビューしたから安全、でもない。**多層で疑って、実物で確かめる**。それが今のところの答えです。

## おわりに

Rendu は今 v0.10.0 で、相変わらず MIT ライセンスで、Homebrew でも入ります。

```
brew install --cask kashioka/tap/rendu
```

3 ヶ月経っても、このアプリは 99% が AI 製のままです。でも「AI に丸投げして放置」とは正反対の 3 ヶ月でした。書くのは一瞬、確かめるのは地道。そのバランスが、たぶんこれからの個人開発の実像なんだと思います。
