## 要点

1. **Prompt Caching はコスト削減だけでなくレイテンシ改善にも効く**。長いプロンプトほど前処理の再利用メリットが大きい。
2. **入力トークンにも計算コストがある**。モデルは入力を attention 用の表現に変換する必要があり、その分が課金対象になる。
3. **OpenAI では長い prompt に対して caching が自動で有効化される**。ただし「自動で有効」と「高いヒット率が出る」は同義ではない。
4. **キャッシュは exact prefix rule で決まる**。最初に変化したトークン以降はキャッシュが無効になる。
5. **動的な値を prompt の先頭に置くのは悪手**。時刻、request ID、更新される working memory などはキャッシュ破壊要因になる。
6. **静的な情報を上、動的な情報を下に置く**のが基本設計になる。
7. **長大な知識ベースは、変動しやすいブロックの後ろに置かない**。そうしないと一番重い部分を毎回再計算する。
8. **Prompt を「複数ブロックの積み重ね」として設計する考え方が有効**。超安定、ユーザー固有で安定、セッション固有で不安定、直近会話、といった層に分ける。
9. **ProjectDiscovery の事例が示す通り、設計ミスだけでキャッシュヒット率は大きく落ちる**。修正後は大幅な改善が見込める。
10. **Prompt Caching は OpenAI 固有ではない**。Anthropic なども提供しているが、細部はプロバイダごとに違う。

## 前提

動画解説の前に、前提となる知識を解説します。

### そもそも Prompt Caching とは何か

Prompt Caching とは、簡単に言えば、**前にも処理したことがあるプロンプトの一部について、その計算結果を再利用する仕組み**です。LLM はテキストを受け取るたびに、ただ文字列を読むのではなく、各トークンを内部表現へ変換しながら順番に処理しています。長い system prompt や大量の会話履歴を毎回まるごと読み直させると、そのぶん計算時間もコストも増えます。

そこで、プロバイダ側が「この先頭部分はさっきと同じだ」と判定できる場合、すでに計算済みの部分を使い回して、毎回ゼロから前処理しなくて済むようにします。これが Prompt Caching です。利用者から見ると、同じ長い前置きや共通コンテキストを含むリクエストを繰り返すと、**入力コストが安くなったり、応答が速くなったりする**形で効果が見えます。

重要なのは、これは「回答そのものを丸ごとキャッシュして返す」仕組みではないということです。Web の HTTP キャッシュのように、同じリクエストなら同じレスポンスを返すという話ではありません。そうではなく、**モデルが入力を読む途中の計算を再利用する**仕組みです。そのため、後ろに新しい質問を足したり、一部だけ文脈が変わったりしても、先頭の共通部分が保たれていれば効果が出る余地があります。

エージェント文脈でこれが重要なのは、エージェントが毎回かなり長い共通プロンプトを抱えがちだからです。たとえば system instructions、few-shot examples、ユーザー設定、ナレッジベース、過去の会話履歴などです。これらの並べ方が悪いと、せっかく共通部分が多くてもキャッシュを活かしにくくなります。逆に、安定した部分を前に、変わりやすい部分を後ろに寄せると、同じエージェントでもコストとレイテンシをかなり改善できることがあります。

### 具体的にどのように Prompt Caching が実現されるか

ここで混乱しやすいのは、「Prompt Caching を誰が実装しているのか」という点です。結論から言うと、Prompt Caching は通常、**モデルプロバイダの推論基盤側で実現される機能**です。つまり、GPT や Claude といったモデルを提供する OpenAI や Anthropic の API 基盤が、同じ prefix を持つ prompt を見つけて、内部で計算済みの前処理結果を再利用します。モデルそのものが自律的に「今回は前と同じだからキャッシュしよう」と判断しているわけでもなければ、Claude Code や Codex のようなハーネスがローカルに同じ種類の KV cache を持っている、という話でもありません。

一方で、**そのキャッシュが当たりやすいように prompt を組み立てる責任**は、アプリケーションやハーネス側にあります。Claude Code、Codex、GitHub Copilot のような製品は、system prompt、ツール定義、会話履歴、ワーキングメモリなどをどう並べるかを決めています。この並べ方が悪いと、プロバイダ側に caching 機能があってもほとんどヒットしません。逆に、安定した prefix を長く保つように作れば、同じ API を使っていてもコストとレイテンシはかなり改善できます。つまり、**キャッシュを保持するのはプロバイダ、キャッシュしやすい形にするのはハーネス**、という分担で考えると分かりやすいです。

OpenAI では、この仕組みはかなり自動化されています。[OpenAI の prompt caching ガイド](https://developers.openai.com/api/docs/guides/prompt-caching) によると、recent models では **1024 トークン以上の prompt で自動的に prompt caching が有効**になります。基本的には `cache=true` のような明示パラメータは不要です。利用者側が見るべきなのは、レスポンスの `usage.prompt_tokens_details.cached_tokens` です。これを見ると、どれだけの入力トークンが cache hit だったかが分かります。また、OpenAI には `prompt_cache_key` という任意パラメータがあり、同じ長い prefix を持つリクエスト群を似たルーティングへ寄せて cache hit を改善しやすくできます。さらに、モデルによっては `prompt_cache_retention` に `in_memory` や `24h` を指定して保持方針を変えられます。つまり OpenAI では、**キャッシュ自体は自動で効き、必要に応じて routing と retention を調整できる**、という形です。

Anthropic では、[Anthropic の prompt caching ドキュメント](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-caching) にある通り、OpenAI よりも制御が明示的です。Anthropic には大きく分けて 2 つの使い方があります。1 つはトップレベルに `cache_control` を置く **automatic caching** で、会話の最後の cacheable block までを自動で breakpoint にしてくれます。もう 1 つは、`system` や `messages.content` の個別 block に `cache_control` を付ける **explicit cache breakpoints** です。こちらは「この system prompt までは固定だからここで cache write したい」といった細かい制御ができます。デフォルト TTL は 5 分で、`cache_control: { "type": "ephemeral", "ttl": "1h" }` のように 1 時間へ伸ばすこともできます。効果を観測するときは `usage.cache_creation_input_tokens`、`usage.cache_read_input_tokens`、`usage.input_tokens` を確認します。

この違いを実務的に言い換えると、**OpenAI は自動寄り、Anthropic は明示制御寄り**です。ただし、どちらでも本質は同じです。長い共通 prefix を保ち、変動しやすい部分を後ろに寄せない限り、キャッシュ効率は上がりません。API にパラメータがあるかどうかは違っても、効率化の前提になる prompt 設計は共通です。

では Claude Code、Codex、GitHub Copilot のような「エージェント製品」はどう考えればよいでしょうか。これらは通常、モデル API の上に載った **ハーネス** です。会話履歴の持ち方、ツール呼び出し、ファイル内容の挿入、作業メモリの差し込み、要約の生成などを管理しています。この層でも「前回の結果を再利用する」処理はいろいろありますが、それは prompt caching とは限りません。たとえば会話要約を保存して次回 prompt に再注入するのはメモリ戦略であって、プロバイダの KV cache 再利用とは別です。同じように、同じ質問に同じ回答を返すレスポンスキャッシュも別物です。

読み分けるポイントは、**API の usage に cached token 系の値が出るか**、**`cache_control` や `prompt_cache_key` のような API パラメータがあるか**です。これらがあるなら、基本的には provider-side prompt caching です。逆に、アプリケーション内部で要約やベクトル検索を使って文脈を節約しているだけなら、それは別の最適化です。動画で語られている Prompt Caching は前者、つまり **モデル API の prefill を再利用する仕組み**として理解するのが正確です。

## 動画詳細

前提知識の解説が長くなりましたが、動画の内容の紹介に入ります。

### まず誤解を正す: 「自動で有効」と「最適化不要」は違う

冒頭では、まず「Prompt Caching でトークンコストを 90% 下げられる」という話を置いたうえで、「でも多くの人はやり方を間違えている」という問題提起に進みます。ここでの論点は、OpenAI では prompt caching が自動で動くため、利用者が構造を意識しなくても恩恵を受けられるように見えることです。

しかし実際には、キャッシュはどんな prompt にも均等に効くわけではありません。**同じ prefix がどれだけ維持されるか**で効き方が決まります。つまり、自動機能ではあるけれど、プロンプトの設計しだいで効率は大きく変わる、というのがここでの主張です。

この説明が重要なのは、エージェント開発でありがちな誤解に直接つながるからです。新しい機能が「自動」と書かれていると、人はつい「なら設計は気にしなくていい」と思いがちです。でも実際には、モデルの温度設定でも、ツール説明でも、メモリの入れ方でもそうですが、**自動化された仕組みほど、入力構造の設計差が結果に出る**ことがよくあります。Prompt Caching もその典型だ、というのがこの動画の出発点です。

### なぜ入力トークンにも課金があるのか

次に Alex は、初心者が見落としやすい前提を置きます。なぜ出力だけでなく入力にも課金されるのか、という話です。これは LLM に少し慣れた人でも、意外と感覚的にしか理解していない部分かもしれません。

モデルは、入力されたテキストをそのまま読んでいるわけではありません。トークン列を内部表現に変換し、attention で参照できる形に整えてから推論します。ここで出てくるのが、いわゆる KV cache で使われるようなキー・バリュー表現です。つまり、入力は「ただ置いてある文字列」ではなく、モデルが使える中間表現に変換される必要がある。そしてその処理自体に計算資源が必要です。

モデルは、入力されたテキストをそのまま読んでいるわけではありません。まずテキストはトークンという単位に分割され、それぞれが数値ベクトルへ変換されます。そのうえで Transformer の各層で、「どの過去トークンをどれだけ参照すべきか」を計算するための query・key・value の表現が作られます。このようにトークン列を内部表現に変換し、attention で参照できる形に整えてから推論するのです。Prompt Caching で再利用されるのも、単なる文字列そのものではなく、こうした前処理済みの内部表現です。つまり、入力は「ただ置いてある文字列」ではなく、モデル内部で再利用可能な形に変換される必要があり、その処理自体に計算資源が必要です。

この仕組みが理解できると、Prompt Caching が単なる値引き施策ではないことが分かります。値引きされているのは「前に見たことがある文字列」だからではなく、**すでに計算済みの前処理結果を再利用できるから**です。

### Prompt Caching は何を再利用しているのか

動画では、OpenAI が直近 10 分程度のあいだに同じ prompt prefix を見ていれば、その前処理結果を再利用できると説明されます。ここで大事なのは、「同じ質問ならキャッシュされる」のではなく、**同じ先頭列が続いていること**が条件だという点です。

この違いはかなり重要です。たとえば、エージェントの最終的な意図が同じでも、system prompt の冒頭にタイムスタンプや可変のワーキングメモリを置いてしまえば、毎回 prefix が変わってしまいます。人間からすると「ほぼ同じ prompt」に見えても、キャッシュの観点では別物です。

補足すると、ここでいう prefix は「文章の意味的なまとまり」ではなく、もっと機械的な概念です。先頭から順に並んだトークン列の一致です。だから、意味が同じでも文言が少し違うだけでキャッシュ対象から外れます。Prompt Caching が semantic caching ではなく、あくまで prefix reuse の仕組みだと理解しておくと、このあと出てくる設計指針が腑に落ちます。

### Mastra エージェントの 10,000 トークン例が示すこと

Alex は、長い knowledge base を system instruction に持った Mastra エージェントを例に出します。コールドスタート時には、その約 10,000 トークン全体を通常価格で処理します。これは当然です。まだその prefix はキャッシュされていないからです。

ところが、その後に follow-up message を送ると話が変わります。前段の 10,000 トークン近いコンテキストは前回と同じなので、そこはキャッシュされた形で再利用され、新しく追加したぶんだけを高いコストで処理すればよくなります。ここでようやく「長い prompt なのに、二回目以降がだいぶ安く・速くなる」という現象が具体的に理解できます。

この例からは、Prompt Caching がチャット履歴のあるアプリケーションで特に効きやすい理由も見えてきます。エージェントやチャットボットでは、毎ターン、過去の履歴や system prompt を抱えたまま新しいメッセージを足していくことが多い。その構造自体がキャッシュに向いています。ただし、それは**不必要に prefix を壊していなければ**の話です。

### 核心: exact prefix rule

動画の本当の山場はここです。Alex は「Prompt Caching only works when the beginning of the prompt matches exactly」と言い切ります。つまり、キャッシュは最初に変わったトークンまでしか効かない。

このルールを理解すると、プロンプト最適化の方向性はかなり明確になります。先頭に近い場所ほど重要で、そこに置いた変動値の影響は大きい。逆に、後ろのほうで変わるなら、それ以前の重いコンテキストは守れる可能性が高い。

これは、従来の prompt engineering の感覚と少しズレることがあります。人はしばしば「大事な情報は先頭に置くべき」と考えますし、それ自体は多くの場合正しいです。ただし Prompt Caching を強く効かせたいなら、そこでいう「大事さ」は二種類に分ける必要があります。

* モデルに必ず先に読ませたい重要度
* なるべく固定してキャッシュを守りたい安定度

そして Prompt Caching の文脈では、後者が非常に重要になります。大事だけれど頻繁に変わる情報は、先頭に置くとコスト面では不利です。ここがこの動画を通して分かる、実務的な面白さではないでしょうか。

### ProjectDiscovery の例が示す「設計の悪さはすぐ数字に出る」

Alex は、ProjectDiscovery のチームがキャッシュヒット率 7% しか出ていなかったという話を紹介します。原因は、working memory がほぼ毎ステップ変わり、それが prompt の前のほうにあったためにキャッシュの大半を毎回無効化していたことでした。

これはかなり示唆的です。Prompt Caching の最適化というと、つい「数% 良くなる細かいチューニング」のように聞こえますが、実際には設計が悪いだけで 7% まで落ち込むことがある。つまり、これは微調整ではなく、**構造設計の問題**です。

エージェントの world state や scratchpad、working memory は、まさに変化しやすい情報の代表です。多くの開発者は「現在の状態だから上の方に置いたほうがよさそうだ」と考えがちですが、キャッシュの観点では逆効果になりやすい。この事例だけでも、構造設計がキャッシュ効率に直結することがよく分かります。

ここから先のアドバイスは、対象読者がやや混ざります。まず、Claude Code や Codex や GitHub Copilot のような既製のエージェント製品を**利用するだけの人**にも通じる原則があります。たとえば「変動値を先頭に置かない」「安定した指示を先に置く」といった話は、custom instructions や長い固定プロンプトを自分で調整できる範囲ならそのまま役立ちます。

一方で、working memory、retrieval 結果、会話履歴、ツール定義をどの順序で prompt に積むかといった話になると、これは主に **Mastra のようなフレームワークで AI エージェントを実装する人**向けです。さらに、prompt をどの block に区切り、どの層で何を固定・可変にするかという整理は、エージェントアプリ開発者だけでなく、**エージェントフレームワークを設計する人**にも直接関係します。

なので、このあたりの助言は 1 つの読者層だけに向けられているというより、同じ原理が異なる抽象度で語られていると見るのが正確です。ざっくり言えば、「先頭に何を置くか」は上級利用者にも関係し、「prompt を block 列としてどう構成するか」はエージェント開発者やフレームワーク開発者により強く関係する、という切り分けです。

### 先頭に置いてはいけないもの: 時刻、request ID、変動値

Alex の第一の実践アドバイスは明快です。**時刻や request ID のような動的値を system prompt の先頭に置くな**。これは一見地味ですが、かなり強い原則です。

とくに時刻を prompt に埋め込む実装はよく見かけます。「現在日時を知らないとタスクを正しく処理できないかもしれない」という理由で、毎回 system prompt 冒頭に now を差し込むパターンです。しかし、分単位で時刻が変わるだけで prefix も変わるため、キャッシュ効率は大きく落ちます。

ここで実務的に考えるべきなのは、「その動的値は本当に毎回 prompt の先頭に必要なのか」という問いです。多くの場合、必要なら後ろへ寄せる、あるいは必要なときだけツールで取得する、といった設計に置き換えられます。Prompt Caching は単に安くするための話に見えて、実際には**どの情報を prompt に直書きし、どの情報をツール呼び出しに逃がすべきか**という設計判断にもつながっています。

### 一番安定したものを先に置く

動画の中盤以降は、プロンプトをどう並べるべきかという整理に入ります。結論は非常に分かりやすく、**最も安定したものを先に、最も変わりやすいものを後に**です。

この原則は、広い意味では「エージェント利用者」にも当てはまります。たとえば、長い custom instructions や固定の役割設定を自分で書いているなら、時刻やその場限りの事情を冒頭に入れないほうがよい、という助言として読めます。ただし、動画で想定している主な適用先は、より強く prompt 構造を制御できる **AI エージェント開発者** です。

もし大きな knowledge base の前に working memory を置くと、working memory が変わるたびに knowledge base 側まで巻き添えで無効化されます。逆に knowledge base を先に置いておけば、後ろの部分が変わっても重い土台は守りやすい。

ここで重要なのは、Prompt Caching の最適化が「何を削るか」ではなく「どう並べるか」の問題として説明されていることです。多くの最適化話は、「もっと短くしろ」「不要な文脈を減らせ」で終わりがちです。しかしこの動画では、長いコンテキストが必要であること自体は否定していません。むしろ、**長いコンテキストを持つなら、その長い部分をいかに再利用可能な位置に置くか**が重要だと言っています。

これは、RAG やメモリ、会話履歴を複数抱える実際のエージェントにとってかなり現実的な助言です。長さをゼロにはできないなら、構造を良くするしかありません。

### Prompt は「ブロック列」として設計する

最後の整理として Alex は、プロンプト全体を block の並びとして考えるとよいと言います。これは初心者にも上級者にも有効なフレームです。

ただし、ここでの「block」は、単なる文章の段落というより、system instructions、ユーザー固有情報、memory、retrieval 結果、直近会話履歴といった **アプリケーション側で別々に組み立てている prompt の構成要素** を指しています。その意味で、この節は明らかに **エージェント利用者一般** よりも、**AI エージェント開発者** や **エージェントフレームワーク開発者** に強く向いた話です。既製品の利用者にとっては、その内部がこう設計されているかもしれない、と理解するための見方として役立つ節だと言えます。

だいたい次のような層に分けるイメージです。

1. **超安定ブロック**: system instructions、固定ルール、few-shot examples
2. **比較的安定したユーザー固有ブロック**: プロフィール、設定、恒常的な前提
3. **セッション固有ブロック**: memory、retrieval 結果、中間状態
4. **直近会話ブロック**: sliding window の最新メッセージ群

この整理によって、Prompt Caching を「テクニック」ではなく「設計原則」として扱いやすくなります。何をどこへ置くかを場当たり的に決めるのではなく、ブロックごとの安定性を基準に配置できるようになります。

特に直近 n メッセージを最後尾に置くべき、という指摘は重要です。会話履歴はチャットアプリでは不可欠ですが、sliding window である以上、一定件数を超えると毎ターン変化します。つまり最も揮発性が高い。だから一番下で扱うのが理にかなっています。

### この話は OpenAI だけのものではない

最後に Alex は、Prompt Caching は OpenAI 特有ではなく、Anthropic もサポートしていると触れます。ただし、各プロバイダで仕組み、価格、条件、細かな挙動は異なります。

というのも、この動画の価値は「OpenAI の仕様暗記」ではなく、**prefix reuse の考え方を身につけること**にあるからです。一度この原理を理解すれば、あとは各社のドキュメントを読んで、自分のプロダクトに合わせて適用できます。逆に言えば、仕様だけ覚えて原理を理解していないと、プロバイダが変わった瞬間に最適化の勘所を失います。

Mastra のようなフレームワークを使う場合も同じです。フレームワークが何かを自動化してくれるとしても、背後で何が効いているかを知っていると、system prompt、memory、retrieval、chat history の配置に自信を持って意思決定できるようになります。短い動画なのに価値が高いのは、まさにそのためです。
