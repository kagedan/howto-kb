# OpenClawで「AI経営OS」を構築してみた：9体のエージェントが会社を動かす仕組みと設定方法【2026年版】

## はじめに：なぜ「AI経営OS」が必要になったのか

2026年現在、AIツールは爆発的に増えた。ChatGPT、Claude、Gemini——何でも答えてくれるLLMは揃っている。でも「便利なAIがある」と「AIで会社が動いている」はまったく別の話だ。

Claude Codeが登場してから、私はずっとこう考えていた。「AIに単発の質問をするのではなく、AIに経営チームを組ませたらどうなるか」と。

その実験の結果が**OpenClaw**だ。CEO・CFO・CMO・COO・CTOの5つの役職エージェントと、秘書・ナレッジ管理・プロダクト管理・クライアント管理の4つのオペレーションエージェント。合計9体のAIエージェントが毎日稼働する経営OSを構築した。

この記事では、その**設定方法・実装の詳細・経営的な効果**をエンジニア視点でまとめる。フリーランス独立を考えているSESエンジニアにとっても、AI経営の具体的なイメージをつかむ参考になるはずだ。

## OpenClawとは何か

OpenClawは、**Claude Code（AnthropicのエージェントAI CLI）をベースに構築した経営自動化フレームワーク**だ。

Claude Codeは、AIがコードを書くだけでなく、ファイルを読み書きし、Bashコマンドを実行し、外部APIを叩ける。これをスキル（`/skill-name`）という形で構造化すると、「この指示が来たらこの動作をするエージェント」を定義できる。

OpenClawはこの仕組みを使って、会社の機能別エージェントを揃えた「経営チーム」を作った。

```
openclaw-keiei-os/
  ├── ceo          # 経営判断・優先順位付け
  ├── cfo          # 財務分析・仕訳
  ├── cmo          # マーケティング戦略
  ├── coo          # オペレーション管理
  ├── cto          # 技術判断・アーキテクチャ
  ├── cc-secretary # inbox/today/archive管理
  ├── cc-knowledge # 教訓・パターン・インシデント記録
  ├── cc-products  # プロダクト管理
  └── clients      # クライアント管理
```

これが「9体の経営エージェント」の全体像だ。

## セットアップ：Claude Codeのスキルシステムを使う

### 前提条件

# Claude Codeのインストール

# バージョン確認
claude --version

# プロジェクトディレクトリに移動して初期化
cd ~/ses-content-automation
claude
```

### CLAUDE.mdでエージェントの振る舞いを定義する

OpenClawの核心は、`CLAUDE.md`に**エージェントの役割と制約を書く**ことだ。Claude Codeは起動時にこのファイルを読み込み、以降の動作の「性格」として内部化する。

例えば、CFOエージェントのCLAUDE.md設定：

```markdown
# CFOエージェント設定

## 役割
財務データの分析、仕訳の確認、キャッシュフロー予測

## 制約
- 実際の送金・支払い操作は実行しない
- 数値を提案する前に実データを確認する
- 推測値と実績値を明示的に区別する
- ピーク値を再現可能な値として扱わない

## アクセス可能なデータ
- /data/finance/ 配下のCSVファイル
- neonctl connection-string で接続するDBの財務テーブル
```

### スキルファイルの構造

各エージェントは `.claude/skills/` ディレクトリに定義ファイルを置く：

```bash
.claude/
  skills/
    ceo.md
    cfo.md
    cmo.md
    coo.md
    cto.md
    cc-secretary.md
    cc-knowledge.md
    cc-products.md
    clients.md
  settings.json  # フック・パーミッション設定
CLAUDE.md      # グローバル行動規範
```

スキルファイルの中身（例：cc-secretaryスキル）：

```markdown
---
name: cc-secretary
description: inbox/today/archiveのYAML操作。タスク管理と日次スケジュール
trigger: "秘書", "タスク追加", "今日のスケジュール", "inbox"
---

## 動作仕様

1. inbox.yamlを読み込む
2. ユーザーの指示に応じてtask追加・移動・完了マーク
3. today.yamlに本日のタスクを整理して返す
4. archive.yamlに完了タスクを移動

## YAMLスキーマ

```yaml
# inbox.yaml
tasks:
  - id: "uuid"
    title: "タスク名"
    priority: high|medium|low
    due: "2026-06-15"
    tags: []
```
```

### settings.jsonでフックを設定する

Claude Codeの`settings.json`では、特定のイベントをトリガーにシェルコマンドを自動実行できる。これが「エージェントの自律動作」の鍵になる。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "skill:cmo",
        "command": "node scripts/fetch_gsc_data.js > /tmp/gsc_latest.json"
      },
      {
        "matcher": "skill:cfo",
        "command": "python scripts/refresh_finance_cache.py"
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "echo '[PRE] $(date): コマンド実行前チェック' >> /tmp/agent_log.txt"
      }
    ]
  },
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(neonctl *)",
      "Bash(python scripts/*)",
      "Read",
      "Write",
      "Edit"
    ]
  }
}
```

## 9体のエージェントの実際の役割と使い方

### 1. CEOエージェント：経営判断の壁打ち相手

```bash
/openclaw-keiei-os:ceo
```

朝一番に起動して「今週の優先事項を整理してほしい」と投げると、inbox・財務状況・プロジェクト進捗をまとめて経営判断の材料を出してくれる。

実際のやり取り例：

```
ユーザー: 今週の重点を3つ決めてほしい

CEO: 現状のinboxとfinanceデータを確認しました。
推奨優先順位：
1. A社との契約更新交渉（期限：6/15、対応なしの場合リスク高）
2. 新規プロダクトのベータリリース準備（cc-productsに7件の未解決issue）
3. 6月末の資金繰り確認（CFOへの連携を推奨）

根拠：inbox.yamlに8件の未処理タスクがあり、
うち3件がA社関連。finance/2026-05.csvでは
6月末に大口支出が集中しています。
```

ポイントは、CEOエージェントが**他のエージェントが管理するデータを横断的に参照する**ことだ。孤立したAIではなく、「経営チームのまとめ役」として機能する。

### 2. CFOエージェント：財務データ分析の自動化

フリーランスとして独立を考えるなら、**財務管理とデータ分析**は避けられない課題だ。CFOエージェントはこの部分を大幅に自動化する。

```bash
/openclaw-keiei-os:cfo
```

DBアクセスの設定が肝だ。macminiで運用する場合、Neonへの接続を安全に行う：

```bash
# .env（gitignoreに追加必須）
NEON_PROJECT_ID=your-project-id

# 接続文字列の取得（毎回動的に取得する）
CONNECTION=$(neonctl connection-string --project-id $NEON_PROJECT_ID)
psql "$CONNECTION" -c "SELECT * FROM monthly_summary;"
```

CFOエージェントが参照するデータ分析スクリプト：

```python
import pandas as pd
from pathlib import Path

def analyze_monthly_cashflow():
    """月次キャッシュフロー分析"""
    df = pd.read_csv(Path('data/finance/transactions.csv'),
                     parse_dates=['date'])
    
    monthly = df.groupby(df['date'].dt.to_period('M')).agg({
        'income': 'sum',
        'expense': 'sum'
    }).reset_index()
    monthly['net'] = monthly['income'] - monthly['expense']
    monthly['cumulative'] = monthly['net'].cumsum()
    
    # 推測値ではなく実績値のみを返す
    return monthly[monthly['date'] <= pd.Period.now('M')]

if __name__ == '__main__':
    result = analyze_monthly_cashflow()
    result.to_json('/tmp/cashflow_latest.json', orient='records',
                   force_ascii=False)
    print(f'分析完了: {len(result)}ヶ月分のデータ')
```

### 3. CMOエージェント：コンテンツとマーケティング戦略

```bash
/openclaw-keiei-os:cmo
```

Google Search ConsoleやSNSのデータを読み込んで、コンテンツ戦略を提案してくれる。GSCスキルとの連携が特に強力だ：

```javascript
// scripts/fetch_gsc_data.js
const { google } = require('googleapis');

async function fetchSearchConsoleData() {
  const auth = new google.auth.GoogleAuth({
    keyFile: process.env.GSC_KEY_FILE,
    scopes: ['https://www.googleapis.com/auth/webmasters.readonly']
  });
  
  const searchconsole = google.searchconsole({ version: 'v1', auth });
  const response = await searchconsole.searchanalytics.query({
    siteUrl: process.env.GSC_SITE_URL,
    requestBody: {
      startDate: '2026-05-01',
      endDate: '2026-06-09',
      dimensions: ['query', 'page'],
      rowLimit: 50
    }
  });
  
  return response.data.rows;
}
```

### 4. cc-knowledgeエージェント：組織の学習記録

これが長期的に最も価値があるエージェントだ。失敗パターン・成功パターン・インシデントを蓄積する「組織の記憶」として機能する。

```bash
/openclaw-keiei-os:cc-knowledge
```

記録フォーマット：

```yaml
# knowledge/incidents/2026-04-sns-shadowban.md
---
type: incident
date: 2026-04-30
severity: high
tags: [sns, meta, 自動化]
---

## 何が起きたか
同一アカウントに15分で15件の自動投稿 → Metaのシャドウバン発動

## 根本原因
cronジョブの間隔設定を誤り、投稿間隔が1分になっていた

## 再発防止
hooks.jsonに投稿前の間隔チェックを追加。
最低15分間隔を強制するガードを設置。

## 適用ルール
同一アカウントへの連投は最低15分間隔を厳守
```

このナレッジが蓄積されると、次回から同じ状況でClaude Codeが自動的に「過去のインシデントに類似しています」と警告を出すようになる。

## フリーランス独立準備への応用

SESエンジニアとして客先常駐で働いていると、「独立したい」という気持ちは自然に湧いてくる。ただ**フリーランス 独立 準備**で最も詰まるのは、技術よりも「経営管理」の部分だ。

OpenClawのエージェント構成は、まさにこの課題を解決するために使える。

### 独立前に整備すべき3つの管理システム

**1. 財務管理（CFOエージェント）**

フリーランスの**単価相場**を自分のデータで把握するには、稼働記録の定期的な**データ分析**が必要だ。CFOエージェントに稼働履歴を読み込ませると、月次の収支・稼働率・実質時間単価を自動計算してくれる。

```python
def calculate_freelance_metrics(records_df):
    """フリーランス稼働指標の計算"""
    metrics = {
        # 実績値のみ使用（推測値は含めない）
        'avg_daily_rate': records_df['日当'].mean(),
        'utilization_rate': records_df['稼働日数'].sum() / \
                           (records_df['月'].nunique() * 20),
        'effective_hourly': records_df['収入'].sum() / \
                           records_df['実働時間'].sum()
    }
    return metrics
```

**2. 契約書管理（cc-secretaryエージェント）**

フリーランスに欠かせない**フリーランス 契約書 テンプレート**の管理もOpenClawで自動化できる。

```
.claude/templates/
  contracts/
    業務委託基本契約書_テンプレート.md
    個人情報取扱い覚書_テンプレート.md
    秘密保持契約書(NDA)_テンプレート.md
    準委任契約書_テンプレート.md
```

テンプレートをMarkdownで管理し、cc-secretaryエージェントが案件ごとに自動生成：

```
使い方：
/openclaw-keiei-os:cc-secretary
> 新規案件「B社Webシステム開発」の業務委託契約書ドラフトを生成してほしい
> 単価：月80万円、期間：2026年7月〜12月、準委任形式
```

エージェントがテンプレートを選択し、案件情報を埋め込んだドラフトを生成する。**最終確認と締結は必ず人間が行う**のが鉄則だ。契約書のAI生成はあくまで「下書き作成の加速」であり、法的効力の担保は別の話だ。

## 実装上のハマりポイントと解決策

### macOS固有の問題：`timeout`コマンド不在

```bash
# macOSには timeout コマンドがない
# ❌ 動かない
timeout 60 python analysis.py

# ✅ gtimeoutを使う
brew install coreutils
gtimeout 60 python analysis.py
```

### PM2でエージェントを常駐管理

launchdは無音停止することがあるため、PM2を使う：

```bash
# 日次ブリーフィングのcron設定
pm2 start "claude --print /prompts/daily_briefing.md" \
  --name "daily-ceo-briefing" \
  --cron "0 8 * * 1-5" \
  --no-autorestart

pm2 save
pm2 startup  # macOS再起動後も自動起動

# ログ確認
pm2 logs daily-ceo-briefing --lines 50
```

### DBアクセスの安全管理

```bash
# ❌ ローカルDBに化ける事故パターン（実際に発生した）
psql ""

# ✅ Neon接続は明示的に
CONNECTION=$(neonctl connection-string --project-id $NEON_PROJECT_ID)
psql "$CONNECTION" -c "SELECT * FROM monthly_revenue;"

# エラーを握り潰さない（デバッグのため）
# ❌ 2>/dev/null はNG
# ✅ エラーはログファイルに流す
psql "$CONNECTION" -f query.sql 2>> /tmp/db_errors.log
```

### Claude認証のkeepalive監視

Claude CLIの認証が切れると、全cronジョブが401エラーで全滅する。PM2でwatchdogを設定する：

```javascript
// scripts/check_claude_auth.js
const { execSync } = require('child_process');

try {
  execSync('claude --print "ping" 2>&1', { timeout: 10000 });
  console.log('Claude auth: OK');
} catch (error) {
  // アラート送信（Slack webhook等）
  console.error('Claude auth FAILED - 要再認証');
  process.exit(1);
}
```

```bash
pm2 start scripts/check_claude_auth.js \
  --name "claude-auth-watchdog" \
  --cron "*/30 * * * *"
```

## 経営的な成果：何が変わったか

率直に言う。**「AI経営OS」は魔法ではない**。エージェントは間違える。データが古いと判断も狂う。指示が曖昧だと的外れな出力を返す。

ただ、確実に変わったことがある：

**意思決定の速度**
朝のCEOブリーフィングで、その日の重点事項が5分以内に決まる。以前は「何から手をつけるか」の整理だけで30分かかっていた。

**財務の可視化**
CFOエージェントが毎週末に収支サマリーを生成する。数字を「見ていなかった」状態から「常に把握している」状態に変わった。データ分析の習慣が強制的に作られる。

**ナレッジの蓄積**
cc-knowledgeエージェントへの記録が習慣化したことで、同じ失敗を繰り返さなくなった。特にデプロイ前の差分確認、SNS投稿の間隔ルールなど、過去の事故から学んだルールが自動的に参照される。

**コンテキストの保持**
複数案件を並行しているとき、前回の会話をAIが覚えていないのは大きなフラストレーションだった。cc-knowledgeとclientsエージェントがデータを永続化することで、「前回どうだったっけ」を都度説明しなくて済む。

## まとめ：OpenClawで作った経営OSの全体像

| エージェント | 役割 | 主な入力 | 主な出力 |
|------------|------|---------|----------|
| CEO | 経営判断・優先順位 | inbox, 財務, PJ状況 | 週次/日次ブリーフィング |
| CFO | 財務分析・データ分析 | 収支CSV, DBデータ | 月次レポート, 資金繰り表 |
| CMO | マーケティング戦略 | GSCデータ, SNS分析 | コンテンツ計画, KPI報告 |
| COO | オペレーション管理 | タスク, スケジュール | プロセス改善提案 |
| CTO | 技術判断 | コードレビュー依頼 | アーキテクチャ提案 |
| 秘書 | タスク/スケジュール管理 | inbox.yaml | today.yaml |
| ナレッジ | 教訓・パターン記録 | インシデント報告 | 再発防止策 |
| プロダクト | プロダクト管理 | 機能要件, バグ | ロードマップ更新 |
| クライアント | 顧客・契約管理 | 契約情報, MTG記録 | 案件サマリー |

フリーランスとして独立する際、この9体のエージェント構成はそのまま「一人フリーランスの経営OS」として機能する。**単価相場の把握**も**フリーランス 契約書 テンプレートの管理**も**フリーランス 独立 準備**のチェックリストも、全てこのフレームワークの中に組み込める。

コードとYAMLとMarkdownで定義されたこのシステムは、Gitで管理でき、Claudeのアップデートに合わせて進化させられる。エンジニアが経営を「コードとして書く」——それがOpenClawの本質だ。
