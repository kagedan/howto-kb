# 個人開発を何度も未ローンチで終わらせてきた僕が、GWだけで3つ公開できた理由

## はじめに

GWだけで、3つのプロダクトをローンチしました。

作ったのは以下の3つです。

正直、技術的にすごいことはあまりしていません。

Vercelにデプロイして、Claude Codeにかなり実装を手伝ってもらいました。  
高度なアーキテクチャや、独自の技術的ブレイクスルーがあるわけではありません。

ただ、今回の個人開発で自分にとって大きかったのは、**とにかくローンチまで持っていけたこと**でした。

僕はこれまで、個人開発で何度も失敗してきました。

ドメインだけ取ったものも、少なくとも3つくらいあります。  
最初はテンション高く作り始めるのですが、途中で仕様を増やしたくなったり、UIに納得できなくなったり、コードの汚さが気になったりして、結局ローンチまで至らない。

飽き性で、かつ完璧主義。

この組み合わせは、個人開発においてかなり厄介です。  
僕の場合、基本的に1ヶ月以上かかるプロダクトは、ほぼローンチまでたどり着きません。

作りかけのリポジトリだけが増えていく。  
ドメイン更新メールだけが、過去の自分のやる気を知らせてくる。つらい。

そんな中で、今回かなり意識したのが、

> Done is better than perfect.

という考え方でした。

今回は、生成AIとVercelで「自分が飽きる前に出せるサイズ」まで開発期間を圧縮することを意識しました。

## 作ったもの

### IfYouMust

[IfYouMust](https://chromewebstore.google.com/detail/ifyoumust/mgaciibaamhmchajndeegkkhdcegifbf) は、誘惑サイトを見る前に小さなタスクを強制するChrome拡張です。

例えば、YouTubeやXなど、自分がつい見てしまうサイトを開こうとすると、その前に腕立て・深呼吸・読書などの小さなタスクを求められる、というものです。

普通のサイトブロッカーだと、結局オフにしたり、別の抜け道を探したりしてしまいます。

そこで、誘惑を完全に遮断するのではなく、「見るなら先に小さな良い行動をする」という形に変えたら続くのではないかと思いました。

誘惑に向かう瞬間を、「小さな良い行動」のトリガーに変えることを狙っています。

![](https://static.zenn.studio/user-upload/d9f3e36dba4e-20260506.png)

### SHUNKAI

[SHUNKAI](https://shunkai.app/about) は、数学の答えではなく「解法方針」をAIが評価する学習アプリです。

作ろうと思ったきっかけは、既存の数学学習が「実際に解くこと」を前提にしていると感じたからです。

もちろん手を動かして解くことは大事ですが、電車の中でノートを開いて最後まで解き切るのは少し重い。

一方で、数学は「どう解くか」が見えた時点で、半分くらい勝負がついている感覚があります。

そして、解法方針を考えるだけならスマホでもできます。

ただ、これまではその方針が正しいかどうかを判定するのが難しかった。  
生成AIによって、そこが初めて現実的になったのではないかと思いました。

SHUNKAIは、「解く」のではなく「方針を立てる」ことだけに特化した数学学習アプリです。

単に正解・不正解を見るのではなく、

* その方針で解けるか
* より効率的な解き方があるか
* どの発想を使うべきだったか

をフィードバックすることを目指しています。

今回作った3つの中では、SHUNKAIだけが少しアプリケーション寄りで、OpenAI APIや最低限の永続化を使っています。

![](https://static.zenn.studio/user-upload/ef13140b15af-20260506.png)

### 終電詰んだ.com

[終電詰んだ.com](https://shudentsunda.com/) は、終電を逃した人向けに、駅別の避難先をまとめたサイトです。

終電を逃したときに知りたいのは、きれいな観光情報ではなく、

* どこで寝られるか
* どこで朝まで過ごせるか
* 近くにネカフェはあるか
* コンビニはあるか
* タクシーで帰るならいくらくらいか

といった、かなり切実な情報です。

終電を逃した瞬間にほしいのは、読み物ではなく、今すぐ使える選択肢です。

「駅名 終電逃した」「駅名 寝過ごした」のような検索に対して、すぐ行動できるページがあれば需要があるのではないかと思いました。

そこで、駅ごとに「最短で寝る」「最安でしのぐ」といった観点で、現実的な選択肢を整理するサイトを作りました。

これはプロダクトというより、ニッチな検索需要を狙ったSEOメディアの実験に近いです。

![](https://static.zenn.studio/user-upload/a6d18d409725-20260506.png)

## これまで何度もローンチできなかった

今回3つローンチできたことは、自分にとっては結構大きな変化でした。

なぜなら、これまでの個人開発では、ローンチまでたどり着かないことが何度もあったからです。

アイデアを思いつく。  
ドメインを取る。  
リポジトリを作る。  
トップページを少し作る。  
READMEを書く。

このあたりまでは、だいたい楽しいです。

ただ、少し進むと、認証、DB、課金、管理画面、OGP、利用規約などが気になり始めます。

最初は小さかったはずのプロダクトが、頭の中で勝手に高尚になっていく。

そして、立派になればなるほど、公開できなくなります。

公開できないまま時間が経つ。  
時間が経つと、熱量が落ちる。  
熱量が落ちると、細かい修正が面倒になる。  
そのうち、別のアイデアの方が面白く見えてくる。

結果、また未ローンチのリポジトリが1つ増える。

この流れを何度もやってきました。

ドメインだけ取って終わったものもあります。  
たぶん3つくらいはあります。

自分の中ではかなり見慣れた失敗パターンです。

## 1ヶ月かかる個人開発は、自分には無理だと認めた

今回一番大きかったのは、「1ヶ月かかる個人開発は、自分には無理だ」と認めたことです。

これは技術力の問題というより、性格の問題です。

僕は飽き性です。  
そして、完璧主義です。

この2つが組み合わさると、個人開発ではかなり高い確率でローンチまでたどり着きません。

完璧主義なので、作っている途中で細かいところが気になります。  
飽き性なので、時間がかかると熱量が落ちます。

この2つが同時に発動すると、かなり厳しいです。

なので今回は、「良いプロダクトを作る」より前に、「自分が飽きる前に出せるサイズにする」ことを優先しました。

機能を削る。  
設計を簡単にする。  
認証を入れなくていいなら入れない。  
DBが不要なら使わない。  
本格的な管理画面は作らない。  
有償機能も最初は作らない。

とにかく、GW中にローンチできるサイズまで落とす。

この制約を置いたことで、ようやく最後まで持っていけました。

僕の場合は、1ヶ月かけて80点を目指すより、数日で60点を出す方が向いていました。

Done is better than perfect.

この言葉は昔から知っていましたが、今回ようやく少し実践できた気がします。

## 今回は「60点で複数出す」と決めた

今回、あえて1つのプロダクトに絞らず、3つ作りました。

理由は、最初から1つのアイデアに大きく賭けるよりも、小さく作って公開し、反応を見る方が自分には合っていると思ったからです。

個人開発で何かがヒットする確率は、正直かなり低いと思っています。

仮に、1つのプロダクトがヒットする確率を `p` とします。  
そして、独立したプロダクトを `n` 個出すとします。

そのとき、少なくとも1つがヒットする確率は、

です。

例えば、1つのプロダクトがヒットする確率がたった5%だったとしても、10個出せば少なくとも1つがヒットする確率は約40%になります。

1 - (1 - 0.05)^{10} = 約40\%

もちろん現実はこんなに単純ではありませんが、少なくとも「0本より3本出す方が可能性はある」とは言えます。

だから今回は、最初から「完璧な1本」ではなく、「そこそこの質で複数本出す」ことを意識しました。

雑に作るという意味ではありません。  
最低限、人に触ってもらえる状態にはする。

ただし、最初から100点は目指さない。  
まず60点で公開して、反応があれば80点、90点に近づけていく。

今回3つ作ったのは、そのための実験でもあります。

## かかった費用

今回の3つをローンチするために追加でかかった費用は、ほぼドメイン代だけです。

ドメイン取得が1つあたり年15ドルくらい。  
それ以外は、基本的に自分の時間だけでした。

しかも、その時間もたいしてかかっていません。

以前なら、1つ作るだけで数週間から1ヶ月以上かかっていたと思います。  
でも今回は、生成AIとVercelのおかげで、かなり短い時間で公開まで持っていけました。

1本あたり年15ドルくらいで打席に立てる。  
この軽さは、今回かなり大きかったです。

## 僕に必要だったのは、技術力よりも損切りルールだった

今回、3つとも最初から「ヒットしなければ深追いしない」と決めています。

改善予定もありますが、全部「ヒットしたらやる」です。

逆に言うと、ヒットするまではやりません。

ドメインも更新しない設定にしています。

これは、ある意味で損切りルールです。

僕の場合、反応がないプロダクトに愛着が湧いて捨てられないというより、むしろ逆です。

気づいたら忘れています。

最初は勢いよく作る。  
ドメインも取る。  
でもローンチまで至らない。  
そのうち別のことに興味が移る。  
存在を忘れる。

そして1年後、突然ドメイン更新の課金だけがやってくる。

これが嫌でした。

使われていないプロダクトや、ローンチすらしていないアイデアに、忘れた頃にお金だけ取られる。  
地味ですが、かなり虚無です。

だから今回は、最初から決めました。

* ヒットしたら有償機能を追加する
* ヒットしなければ何もしない
* ドメインも更新しない
* 反応があるものだけ育てる

冷たく見えるかもしれませんが、僕にとってはこれが必要でした。

これはプロダクトへの愛情がないというより、忘れた頃にドメイン代だけ請求される未来を防ぐためのルールです。

打席数を増やすには、作る力だけではなく、捨てる前提を先に置くことも必要だと思いました。

## 技術スタック

一応、使った技術も簡単にまとめておきます。

ただ、この記事の主題は技術的な工夫ではなく、「どうすれば自分がローンチまで持っていけるか」です。  
なので、ここはあくまで補足です。

### 共通

3つでフレームワークは揃えていません。  
それぞれの性質に合わせて、なるべく軽く作れる構成にしています。

* ホスティング: 全プロダクトともVercel
* 言語: TypeScript（IfYouMustのみ素のJavaScript）
* スタイル: Tailwind CSS（SHUNKAI / 終電詰んだ.com）
* 計測: Vercel Analytics

フレームワークは以下です。

* SHUNKAI: Next.js（App Router）
* 終電詰んだ.com: Astro（静的生成）
* IfYouMust: フレームワーク不使用（Chrome拡張のため素のHTML / CSS / JavaScript）

### IfYouMust

IfYouMustは、Manifest V3のChrome拡張として作っています。

フレームワークは使わず、素のHTML / CSS / JavaScriptです。

ざっくり構成は以下です。

* Manifest V3
* backgroundはservice worker
* ブロック画面はcontent scriptからShadow DOMでオーバーレイ注入
* ルールはchrome.storage.localに保存
* 権限はstorage / scriptingと、optional host permissionsの`<all_urls>`
* セッション解除はタブ単位のメモリ保持

誘惑サイトが描画されてから隠すと意味が薄いので、content scriptは`document_start`タイミングで注入しています。  
ページ側のスクロール、クリック、キー入力も止めるようにしています。

ただし、最初から複雑な同期やアカウント管理は入れていません。  
Chrome拡張として最低限成立するところまでに絞りました。

### SHUNKAI

SHUNKAIは、今回の3つの中では一番アプリケーションらしい構成になっています。

主な構成は以下です。

* Next.js（App Router）
* Vercel
* OpenAI API
* Upstash Redis
* KaTeX
* Vercel Analytics

OpenAI APIを叩く必要があるため、APIキーをフロントエンドに置くことはできません。  
そのため、Next.jsのroute handlerからOpenAI APIを呼び出しています。

また、最低限の制限として、IPアドレスによる利用制限も入れています。

ユーザーのメールアドレスや利用に必要なデータは、Upstash Redisに保存しています。

本格的なDBを使う選択肢もありましたが、今回は速度優先で、必要最低限の永続化に絞りました。

数式表示にはKaTeXを使っています。

終電詰んだ.comは、SEOメディアとして作っています。

構成はかなりシンプルです。

* Astro
* output: static
* Tailwind CSS v4
* @astrojs/sitemap
* Vercel
* Vercel Analytics

駅ごとのデータは、`src/data/stations/*.ts` にTypeScriptオブジェクトとして持っています。

それを `src/pages/[slug].astro` で動的ルーティングし、ビルド時に静的ページとして書き出しています。

地図はGoogle Maps APIを使っていません。  
Google Mapsの検索URLへのディープリンクだけで実現しています。

アフィリエイトや外部APIもまだ入れていません。  
まずは検索流入があるかを見ることに絞っています。

## 今後やること

基本方針は、**ヒットしたら有償機能を追加する**です。

IfYouMustなら、より細かいルール設定や端末間同期。  
SHUNKAIなら、問題数の拡充や学習履歴、苦手分野の分析。  
終電詰んだ.comなら、ホテル予約や広告などの収益化導線。

ただし、どれも今すぐは作りません。

まずは、使われるかを見る。  
反応があれば育てる。  
なければ深追いしない。

それくらいの温度感でいます。

## おわりに

GWだけで3つのプロダクトをローンチしてみて、個人開発の感覚が少し変わりました。

自分にとって一番大きかったのは、「すごいものが作れるようになった」というより、「自分が飽きる前に公開できるサイズまで、開発期間を圧縮できるようになった」ことでした。

僕自身、これまでは完璧に作ろうとして、何度もローンチまでたどり着けませんでした。

でも今回、技術的にすごいことをしなくても、生成AIとVercelを使えば、短期間で複数のプロダクトを公開できることが分かりました。

しかも、追加でかかった費用はほぼドメイン代だけです。  
1つあたり年15ドルくらいで、打席に立てる。

もちろん、3つ出したからといって、どれかが必ず当たるわけではありません。

ただ、未公開のまま0本で終わるより、60点でも3本出した方が、明らかに可能性は広がります。

Done is better than perfect.

AI時代でも、何を作るかを決めるのも、最後に公開ボタンを押すのも、まだ人間です。

そして、自分のような飽き性で完璧主義な人間にとっては、完璧を目指すよりも、まず公開することの方がずっと大事なのだと思いました。

それぞれのプロダクトの実装や設計については、別の記事でもう少し詳しく書いていこうと思います。
