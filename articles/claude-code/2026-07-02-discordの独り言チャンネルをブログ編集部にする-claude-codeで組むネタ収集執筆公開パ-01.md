ブログが続かない原因は文章力ではなく、「ネタ出し」と「執筆」が毎回ゼロから始まることだと思っています。私は非エンジニアの一人社長ですが、この2つをClaude Codeに渡すパイプラインを組んで、オウンドメディアを運用しています。この記事では、その仕組みをエンジニアの方向けに構成要素ごと公開します。

## 全体像

```
Discord独り言チャンネル（ネタ帳）
  │  scripts/discord.mjs（botトークンで差分取得。常駐なし）
  ▼
knowledge/*.md（1ファイル=1ノウハウのナレッジ置き場）
  │  blog-write スキル（ドラフト作成）
  ▼
記事ドラフト ──▶ サブエージェント2体（公開可否レビュー / ファクトチェック）
  │  人間レビュー
  ▼
blog-publish スキル（ビルド→コミット→push→IndexNow通知）
```

人間が手を動かすのは「独り言を書く」「ネタを選ぶ」「最終レビュー」の3点だけです。

## 1. ネタ帳はDiscordの自分専用チャンネル

仕事中の気づき・うまくいった手順・失敗と解決策を、思いついたそばからDiscordの独り言チャンネルに投げ込みます。音声入力の走り書きも多く、整理は一切しません。**後工程のAIが整理する前提なので、人間は書き留めることだけに集中する**のが設計思想です。

Claude CodeとDiscordの接続は、常駐botを開発するのではなく、botトークンを使った小さなスクリプトで「必要なときにメッセージを読みに行く」だけにしています。実例では7日分で37件の投稿が取れました。

## 2. ナレッジは「1ファイル=1ノウハウ」+ステータス管理

取得した独り言は、そのまま記事にせず、一度 `knowledge/` ディレクトリのMarkdownに落とします。ポイントはfrontmatterでのステータス管理です。

```
---
status: ネタ          # ネタ → 記事化済み に更新
article: ''           # 記事化されたら /blog/<slug>/ を記入
---

# ノウハウのタイトル

## 誰のどんな悩みに効くか
## 中身（一次情報）
## 関連
```

ルールは1つだけ厳格にしています。**一次情報（自分で実測した数字・実際にやった手順・失敗と解決）だけを書く**。どこかで読んだ一般論を混ぜると、生成される記事が「どこにでもある記事」になるからです。

## 3. 変換はスキル（手順書）に分業させる

工程ごとにClaude Codeのスキルを分けています。

| スキル | 役割 |
| --- | --- |
| knowledge-harvest | Discordの差分取得→一次情報の仕分け→ナレッジ化 |
| blog-write | ナレッジ→記事ドラフト（トーン・frontmatter・内部リンクの定型化） |
| blog-publish | ビルド確認→コミット→push→IndexNow通知→ナレッジのステータス更新 |

1本の巨大プロンプトにせず工程で切るのは、失敗したときに「どの工程が悪いか」を特定して手順書だけ直せるようにするためです。プロンプトのデバッグではなく、手順書のメンテナンスとして運用できます。

## 4. 人間の関所は「選定」と「公開判断」の2つだけ

全自動は最初から目指していません。

* **ネタの選定**: どの気づきを記事にするかは目視で選ぶ。事業の方向性の判断はAIに委ねない
* **公開前レビュー**: ドラフトは公開可否レビューとファクトチェックの専用サブエージェント（別記事で解説予定）を通した上で、最後は必ず人間が読む

AIに渡すのは「作業」、人間に残すのは「判断」。この線引きだけ守れば、ネタ帳をSlackやNotionに替えても、出口をブログからメルマガに替えても、同じ型が機能します。

## まとめ

* ネタ帳×AIエージェントの接続という型を作ると、コンテンツ制作の重い工程（変換作業)が仕組みに移る
* ナレッジは1ファイル=1ノウハウ+ステータス管理。一次情報縛りが品質の生命線
* 工程はスキルに分業させ、人間は選定と公開判断だけ残す
