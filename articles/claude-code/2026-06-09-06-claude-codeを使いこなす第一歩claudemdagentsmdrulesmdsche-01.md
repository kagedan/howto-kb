## はじめに

> 「CLAUDE.md・AGENTS.md・RULES.md・SCHEMA.md を読んでください。読み終わったら Step 4（部位マスタAPI）を実装してください。」

これが私の毎回の書き出しになりました。

最初はそうではありませんでした。Step 1のときは「このAPIを実装して」と一言送っていました。でもStep 2になると「あ、このプロジェクトはFastAPIで、テーブルはこういう構成で、命名規則はこうで…」と毎回説明し直す自分がいました。

**Claude Codeはチャットをまたぐと、前の会話内容を忘れます。**

新しいチャットを開くたびに「プロジェクトの文脈」がリセットされるので、毎回ゼロから説明するのは非効率ですし、説明漏れがあればコードの品質にも影響します。

「毎回読ませるファイル」として指示書を設計すれば、この問題が解決できる——そこから4ファイルの設計が始まりました。

この記事では、LoadLogの開発で実際に使った指示書の構成と、なぜそのファイル分割にしたのかを解説します。

## Claude Codeとは

VSCode等の上で動くAIコーディングツールです。プロジェクトのファイルを読み込んで「〇〇を実装して」という指示に従ってコードを生成・編集してくれます。

**通常のChatGPTやClaudeとの大きな違いは「プロジェクトのファイルを直接読み書きできる」点です。**

## 4ファイルの構成と役割

| ファイル | 読むタイミング | 内容 |
|---|---|---|
| CLAUDE.md | 常に最初 | プロジェクトの玄関口。技術スタック・ディレクトリ構成・起動手順・他ファイルへの参照 |
| AGENTS.md | 実装指示をするとき | Step 1〜8の実装順序・完了条件・指示テンプレート |
| RULES.md | 迷ったとき | 命名規則・コーディングルール・エラー形式・セキュリティルール |
| SCHEMA.md | 実装中に参照 | 全テーブル定義・全APIエンドポイント一覧・エラーコード一覧 |

### なぜ1ファイルにまとめないのか

**更新頻度が異なるためです。**

- RULES.md（命名規則等）は一度決めたらほとんど変わらない
- SCHEMA.md（テーブル定義・API）はAPIを追加するたびに更新する

1ファイルにまとめると「どこを更新すればいいか」が分かりにくくなります。更新漏れがあれば、AIが古い仕様に基づいてコードを書いてしまいます。


## CLAUDE.md：玄関口として機能させる

CLAUDE.mdはClaude Codeが最初に読むファイルです。ここに全ての情報を詰め込まず、「詳細は各ファイルを参照」という形にしました。

```markdown
# LoadLog — CLAUDE.md

## プロジェクト概要
（2〜3行）

## 技術スタック（早見表）
| レイヤー | 技術 | バージョン |
...

## ディレクトリ構成
（ツリー形式）

## ローカル起動手順
（コマンド）

## ドキュメント参照ガイド
| 知りたいこと | 参照先 |
| 実装の順序 | AGENTS.md |
| 命名規則   | RULES.md  |
| テーブル定義| SCHEMA.md |
```

開発中に「ディレクトリ構成はCLAUDE.mdで作ってくれる認識ですが合ってますか？」と確認したことがありました。認識がズレていたら実装の方向性ごと変わってしまいます。**指示書の内容を「自分はこう理解している」と言語化して確認するクセは、テスト仕様の読み合わせに近い感覚です。**


## AGENTS.md：実装の順序と完了条件を定義する

AGENTS.mdはClaude Codeへの実装指示書です。「何をどの順番で実装するか」「完了とはどういう状態か」を定義しました。

```markdown
## 実装ステップ（Phase 1）

### Step 3 — 認証API

**作業内容**
- POST /api/users/register
- POST /api/users/login

**完了条件**
- pytest tests/test_auth.py -v で全テスト合格
- 重複メールで409、誤パスワードで401が返る
```

### 完了条件を「テストが通ること」にした理由

「実装して」だけでは何をもって完了とするかが曖昧です。テスター経験から「完了条件は客観的に確認できる形で定義する」という習慣があり、これがAIへの指示にも自然に活きました。


## RULES.md：コーディングルールを明文化する

RULES.mdはコーディング規約です。AIへの指示に一貫性を持たせるために作りました。

```markdown
## 命名規則

| 対象 | ルール | 例 |
|---|---|---|
| 変数・関数 | snake_case | get_current_user |
| クラス | PascalCase | TrainingSession |
| APIエンドポイント | kebab-case | /api/body-parts |

## レイヤー責務ルール

- routers/ : リクエスト受取・レスポンス返却のみ
- services/: ビジネスロジック・DB操作
- routers/にビジネスロジックを書かない
```

### コメントルールを追加した経緯

Claude Codeが生成したコードには、最初ほとんどコメントがありませんでした。コードは動きます。テストも通ります。でも「なぜこう書いているのか」が全く分かりません。

1週間後に見返したとき、自分が書いたわけでもないコードを読み解くのは想像以上に大変でした。**「動く」と「読める」は全くの別物です。**

テスターとして「仕様書がないとテストできない」という感覚は身に染みていました。コードにとってのコメントは仕様書と同じです。「何をしているか」ではなく「なぜそうするのか」が書かれていることで、コードが初めて「読めるもの」になります。

そこでRULES.mdに以下のコメントルールを追加しました。

```markdown
## コメントルール

使用タグ：【目的】【前提】【注意】【TODO】の4種類のみ

方針：
- 「なぜ」を優先して書く
- 自明な処理にはコメントしない
- 1関数あたり3〜5個を目安にする
```

### コメントルールの実例

実際のLoadLogのコードでどう使うか見てみましょう。「部位を削除するAPI」を例にします。

```python
def delete_body_part(
    part_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # 【目的】指定した部位を削除する。紐づくメニューがある場合は削除を拒否する
    # 【前提】get_current_userで認証済みであること
    # 【注意】他のユーザーの部位は取得できないようにuser_idでフィルターしている
    # 【注意】404を返す理由は、他ユーザーの部位の存在を知らせないためでもある

    part = db.scalar(
        select(BodyPart).where(
            BodyPart.part_id == part_id,
            BodyPart.user_id == current_user.user_id
        )
    )
    if not part:
        raise HTTPException(status_code=404, ...)

    menu_count = db.scalar(
        select(func.count()).where(Menu.part_id == part_id)
    )
    # 【注意】紐づくメニューがある場合は400を返す。削除するとメニューが孤立するため
    if menu_count > 0:
        raise HTTPException(status_code=400, ...)

    db.delete(part)
    db.commit()
    return {"message": "削除しました"}
```

ポイントは **「何をしているか」ではなく「なぜそうするのか」** を書くことです。

`404を返す理由は他ユーザーの部位の存在を知らせないため` というコメントは、コードを読むだけでは絶対に分かりません。セキュリティ上の意図が込められていますが、コメントがなければ「なぜ404？」と疑問が残ったままになります。

こういった「なぜ」を残しておくことで、後から読んだときの理解が格段に変わります。


## SCHEMA.md：実装中に参照するリファレンス

SCHEMA.mdにはDB定義とAPIエンドポイントをまとめました。

```markdown
## training_sets テーブル

| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| set_id | INT | PK | セットID |
| weight_kg | FLOAT | NULL | 重量。自重はNULL |
| pre_score | INT | CHECK(1〜10) | 主観的運動強度 |

## APIエンドポイント一覧

| メソッド | エンドポイント | 認証 | 概要 |
|---|---|---|---|
| POST | /api/sessions/{id}/sets/bulk | 要 | セット一括登録 |
```


## 実際の指示テンプレート

これらのファイルを使った実際の指示文はこうなりました。

```
CLAUDE.md・AGENTS.md・RULES.md・SCHEMA.md を読んでください。

読み終わったら Step 4（部位マスタAPI）を実装してください。

実装時はRULES.mdのコメントルールに従い【目的】【前提】【注意】【TODO】タグを
使って各関数に3〜5個のコメントを必ず追加してください。

実装完了後はAGENTS.mdの【完了メッセージのフォーマット】セクションを参照し、
7項目すべてを埋めて報告してください。項目を省略しないでください。
```

**「読んでください」→「実装してください」→「コメントルールに従って」→「完了メッセージで報告して」の4点セットが基本形です。**


## 運用してみて気づいたこと

### 毎回4ファイルを読ませる理由

Claude Codeはチャットをまたぐと情報をリセットします。Stepが変わるタイミングで新しいチャットを開くとき、毎回この4ファイルを読ませることで「プロジェクトの文脈」を再注入できます。

最初はこれが手間に感じていましたが、逆に「新しいチャットを開くタイミング＝Stepの区切り」という習慣になり、実装の単位が自然と整理されていきました。


## 失敗から学んだこと

### コメントルールが守られなかった

指示文に「コメントを追加してください」がなければ、AIはコメントなしのコードを書きます。**AIは指示されたことしかやりません。** RULES.mdに書いてあっても、指示文で触れなければ参照されないことがあります。

重要なルールは指示書に書くだけでなく、**毎回の指示文にも明示的に書く必要があります。**

### 指示書に書いたことと実態を一致させる

AGENTS.mdのStep 8（フロントエンド実装）にこう書いていました。

```
画面デザインは添付のモックアップ（ダークテーマ）に従う
```

しかし実際にはモックアップファイルをプロジェクト内に配置していませんでした。

結果として、Claude Codeは「添付のモックアップ」を参照できず、自分の解釈でダークテーマのUIを作りました。**実装された画面とモックアップにズレが生じました。**

**指示書に書いたことは、実態と一致して初めて機能します。**

正しい対応はこうです。

```
LoadLog/
└── docs/
    └── mockups/
        ├── SCR-01_login.png
        ├── SCR-03_transfer.png
        └── ...（全画面分）
```

```markdown
# AGENTS.md Step 8 の修正版
画面デザインは docs/mockups/ 配下の各モックに従う
- SCR-01: docs/mockups/SCR-01_login.png
- SCR-03: docs/mockups/SCR-03_transfer.png
```

これはテスト設計でいう「テスト仕様書に書いた手順が実際に実行できる状態か確認する」と同じ考え方です。**仕様書と実態の乖離は、テスターとして最も気をつけるべきポイントでした。** それが自分の指示書で起きていたことに後から気づきました。


## まとめ

- CLAUDE.md（玄関）・AGENTS.md（指示書）・RULES.md（ルール）・SCHEMA.md（リファレンス）の4ファイルに分ける
- 分ける理由は「更新頻度が異なるから」
- 指示文には「読んでください」→「実装してください」→「ルールに従って」→「完了報告して」の4点を毎回含める
- 重要なルールは指示文にも明示的に書く
- 指示書に書いたことは実態と一致させる。ファイルの参照先は実際に存在するパスで書く
