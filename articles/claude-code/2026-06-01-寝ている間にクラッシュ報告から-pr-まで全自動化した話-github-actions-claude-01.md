## はじめに

GitHub Actions の cron 実行や Claude Code Routines のクラウド実行を組み合わせると、**寝ている間にアプリのクラッシュ対応が進む仕組みが作れるんじゃないか** と思って試してみました。

リリース済みアプリのクラッシュ対応、地味に時間を食いますよね。

1. Play Console や Firebase Crashlytics でクラッシュを見つける
2. チケットを起票する
3. 担当者をアサインする
4. コードを読んで原因を特定する
5. 修正してブランチを切る
6. PR を作る
7. レビューを依頼する
8. 開発者テストの指摘を反映する

このうち **1〜6 と 8 は Claude に任せられる**んじゃないか、と思って組んでみたら想像以上にきれいに回るようになりました。一次レビューも Claude にやらせて、人間は最終確認と開発者テストだけする運用です。

本記事では、[**SUBLINE**](https://www.subline.jp/)（スマホで内線・外線通話ができる IP 電話アプリ）で運用中のこの仕組みを、構成要素ごとに解説します。チケットトラッカーには Backlog を使っていますが、GitHub Issues などに置き換えても同じ思想で組めます。

## 30 秒サマリ

* **Before**: クラッシュ 1 件あたり「発見 → 起票 → 調査 → 実装 → PR 作成 → テスト項目記入」で 1〜2 時間／件
* **After**: 朝起きるとブランチに修正コミットと PR が並んでいて、人間がやるのは **「PR の差分とテスト項目を見る 5〜10 分」** だけ
* **作り方**: GitHub Actions と Claude Code Routines の 2 系統で、状態管理＋ stage 分割＋途中再開＋類似 PR 統合を組む

## 全体像

図の点線（自己ループ）は **「ネットワーク失敗・モデル turn 切れ・runner timeout が起きても、次回の cron / トリガーで途中から再開する」** 設計を表しています（[4 章](#4.-%E6%AE%B5%E9%9A%8E%E7%9A%84%E5%86%8D%E9%96%8B%E3%81%AE%E4%BB%95%E7%B5%84%E3%81%BF)で詳述）。

ポイントは 3 つ:

1. **人間が触る場所を「最終確認」と「開発者テスト」だけに絞り込む**
2. **途中で失敗しても続きから再開できる**ようにチケット状態と属性で進行を管理
3. **開発者テスト後の指摘も Claude に追加対応させる**ことで、人間のキャッチボール回数を最小化

## 1. クラッシュログ取得からチケット起票

最初のステップは「クラッシュを見つけてチケットに登録する」自動化。GitHub Actions で実装しています。

### 1-1. Android 限定: Play Console から取得（簡単）

Android はクラッシュログが **Google Play Console に自動で集約される**ので、Play Developer Reporting API を叩くだけです。

ただし **Play Console が拾えるのは「端末の設定で『使用状況と診断情報の共有』を ON にしているユーザー」のクラッシュに限られる** 点には注意してください。OFF にしているユーザーの分は別の経路（後述の Crashlytics）で拾う必要があります。

**事前準備**:

1. **Google Cloud Platform（以下 GCP）でサービスアカウントを作成**し、Play Developer Reporting API を **API ライブラリから有効化**
2. **Play Console > ユーザーと権限** で新しいユーザーを招待し、サービスアカウントのメールアドレスを追加。アカウントの権限で「**アプリ情報の閲覧**」「**一括レポートのダウンロード（読み取り専用）**」を付与（**Play Console API のアクセス制御はここで決まります。GCP IAM のロールではなく Play Console 側の権限が本体**）
3. サービスアカウントの **鍵 (JSON) をダウンロード**し、**base64 化**して `GCP_SA_KEY_BASE64` という名前で **GitHub Secrets** に保存

#### なぜ base64 化するか

JSON をそのまま Secrets に貼ると、内部の改行や `"` のエスケープで事故りやすい（特に `run:` で `echo` する時）。base64 で 1 行文字列に変換しておくと、安全に Secrets に格納できて GitHub Actions 側で復元するだけで済みます。

```
# macOS（クリップボードにコピー）
cat service-account.json | base64 | tr -d '\n' | pbcopy

# Linux（標準出力に 1 行で出力）
cat service-account.json | base64 -w0
```

GitHub Actions 側で復元:

```
- name: Restore service account
  run: |
    echo "${{ secrets.GCP_SA_KEY_BASE64 }}" | base64 --decode > service-account.json
```

これで GitHub Actions からクラッシュログ取得が可能になります。

```
# .github/workflows/crash-to-backlog.yml（抜粋）
on:
  schedule:
    - cron: '0 14,21 * * *'   # 毎日 JST 23:00 と 6:00
  workflow_dispatch:
    inputs:
      min_users:
        description: '最低影響ユーザー数'
        default: '5'
```

`min_users` は **影響ユーザー数のしきい値**。1 にすると再現性のない単発クラッシュまで拾って起票数が爆発し、無駄対応でかえって品質が落ちかねないため、**最低でも 5 以上を目安に、アプリの DAU やクラッシュ頻度の実態に合わせて調整**してください。

Play Console の Reporting API でクラッシュ一覧を取得します。Python の標準ライブラリで完結させて、外部依存は最小化:

```
# Play Developer Reporting API でクラッシュ取得
result = api_get(
    f"{REPORTING_BASE}/apps/{enc_pkg}/errorIssues:search",
    params={
        "filter":   "errorIssueType = CRASH",
        "orderBy":  "distinctUsers desc",
        "pageSize": 100,
    }
)
```

ANR も同じ API で `errorIssueType = APPLICATION_NOT_RESPONDING` を指定すれば取得できます。

### 1-2. iOS / Android 両対応: Firebase Crashlytics + Cloud Logging（少し面倒）

Crashlytics は **iOS でも Android でも使え、端末側の診断情報共有設定に依存せずクラッシュを集約できる** のが強みです。Play Console で漏れる Android ユーザーや、Play Console を使えない iOS のクラッシュをここで拾う形にもできます。ここでは iOS 側の導入を中心に解説します。

iOS のクラッシュログは Xcode の Organizer で見るのが一般的で、Apple 公式の **App Store Connect API** にもクラッシュ系エンドポイント（TestFlight Feedback API、Analytics Reports API）が存在します。ただし**公開済みアプリのクラッシュ集約と AI への投入**を考えると、Crashlytics 経由の方が運用しやすかったので、[SUBLINE](https://www.subline.jp/) では **Firebase Crashlytics** を導入して **Cloud Logging** にエクスポートし、GCP API で取得する形にしました。

**事前準備（iOS 側）**:

1. **Firebase コンソール**で対象アプリの **Crashlytics を有効化**
2. アプリに **Firebase Crashlytics SDK を追加**
3. Firebase コンソールの **設定 > 統合** から **Cloud Logging を有効化**
4. GCP サービスアカウントに **ログ閲覧者**（`roles/logging.viewer`、必要に応じて **Error Reporting 閲覧者**）を付与
5. Android と同様にサービスアカウント鍵を base64 化して GitHub Secrets に保存

`firebasecrashlytics.googleapis.com` 自体には `v1alpha` の `topIssues` などのエンドポイントが存在しますが、**安定版の REST API は未提供**です。Firebase 公式ドキュメントでは [BigQuery エクスポート](https://firebase.google.com/docs/crashlytics/bigquery-export) と [Cloud Logging エクスポート](https://firebase.google.com/docs/crashlytics/cloud-logging-export) の 2 つが案内されていて、[SUBLINE](https://www.subline.jp/) では後者で運用しています。

### 1-3. 重複防止と運用上の工夫

* **dedup\_key（重複起票防止）**: クラッシュには Play Console / Crashlytics 側で固有 ID が振られています。チケット起票時にその ID を description の中に `<!-- [CRASH:issue_id] -->` というコメント形式で埋め込んでおくと、次回の自動実行で「**この ID を含むチケットが既にあるか**」を Backlog 検索で確認でき、同じクラッシュに対して二重起票するのを防げます
* **MIN\_USERS フィルタ**: 影響ユーザー数が少なすぎるものは起票しない（ノイズ削減）
* **製品版バージョン限定**: 最新の公開バージョンコードを取得して、それと一致するクラッシュだけ拾う

起票時に **担当者・カテゴリー・タイトル形式** をルール化しているのが、後続の自動化を成り立たせる要点です。

## 2. チケットの状態管理ルール

ここが自動化の中核で、**「状態と属性だけで次の挙動が決まる」設計**にしてあります。

### 2-1. ステータスの遷移

| Status ID | 名前 | 何を意味するか |
| --- | --- | --- |
| 1 | 未対応 | 起票直後（または開発担当者が再対応を依頼した直後） |
| 2 | 処理中 | Claude が feature ブランチに修正を push 済み |
| 3 | 処理済み | Claude が改修・自己確認まで終え PR 待ち |
| 4 | 完了 | PR マージ済み or 別チケットに統合済み |

> もう一度やり直してほしい場合は、チケットのステータスを「未対応」に戻して担当者を Claude Code に戻せば、再ピックアップされて頭から処理が走ります。

### 2-2. 属性

* **担当者**:
  + **Claude Code**（Backlog 上に AI 用として作成したユーザー）→ 自動処理対象
  + **開発担当者**（人間）→ Claude は触らない（仕様確認待ち等）
* **カテゴリー**: `アプリ(Android)` / `アプリ(iOS)` → どのリポジトリの自動化が動くかを切り替え
* **マイルストーン**: `3.9.7` のような形式 → これが `release-SUBLINE-V3-9-7` のようなブランチ名に変換される。**マイルストーン未設定のチケットは PR 作成までは進まず、改修ブランチだけ作って PR 待ちで停止する**

### 2-3. 状態 × ブランチ × PR の対応表

実際の進行状況は **「チケットステータス」「チケットコメント」「ブランチの状態」「PR の状態」「レビューコメント」** の 5 つで決まります:

| ステータス | チケットコメント | ブランチ | PR | レビュー | 次にやること |
| --- | --- | --- | --- | --- | --- |
| 未対応 | なし | なし | なし | なし | Claude が方針コメント投稿 |
| 未対応 | 方針コメントあり | なし | なし | なし | Claude が `feature-SUBLINE-XXXX` ブランチで修正開始 |
| 処理中 | 方針コメントあり | 修正済み（push 完了） | なし | なし | Claude が改修不足や追加対応の要否を調査 |
| 処理済み | （省略） | 修正済み | なし | なし | Claude が PR 作成（マイルストーン記載がある場合のみ） |
| 処理済み | （省略） | 修正済み | 作成済み | なし | Claude が PR レビュー観点コメントを投稿 |
| 処理済み | （省略） | 修正済み | 作成済み | レビュー済み | 開発担当者の開発者テスト待ち |

**マイルストーンが未設定**だと、改修ブランチだけ作って PR 作成は停止します。**「次回リリースに乗せると決めたタイミングでチケットのマイルストーンにバージョンを追記」** すれば、自動で PR 作成まで進みます。基本は放置しておけば改修ブランチが溜まっていく形です。

### 2-4. この設計の旨味

**担当者を切り替えるだけで「Claude が手を出すかどうか」をコントロール**できます。たとえば、

* Claude に「期待動作わからない」と確認コメントを出された
* そのコメントに開発担当者が答える
* 担当者を Claude Code に戻す
* → 次回の自動実行で続きから処理される

開発担当者と Claude の引き継ぎがチケットの担当者欄だけで完結する、というのが運用してみると地味に効きます。

## 3. Claude による自動改修（2 パターン）

「いざ修正」のフェーズは、**GitHub Actions パターン**と **Routines パターン**の 2 つで運用しています。両者は同じ動作をするバックアップ関係で、コスト構造と細かい挙動だけが違います。比較は[後述](#%E6%AF%94%E8%BC%83)。

### パターン A: GitHub Actions（`claude-autofix-backlog.yml`）

`anthropics/claude-code-action@v1` を使って、ステージごとに Claude を呼び出します。Action の使い方は [Claude Code GitHub Actions 公式ドキュメント](https://code.claude.com/docs/ja/github-actions)を参照してください。

#### staged 構造の旨味

各ステージで **「Claude にやらせること」と「shell でやらせること」を明確に分ける**ことで、Claude のトークン消費を抑えられます。

* 決定的な処理（Backlog API、`gh` コマンド、git 操作、JSON パース）→ shell
* 判断が必要な処理（コード読解、改修方針、類似判定、レビュー文生成）→ Claude

```
- name: "[Plan] Claude: 改修方針を策定"
  uses: anthropics/claude-code-action@v1
  with:
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
    # 実運用では --max-turns / --model をステージごとに調整します（数値は後述の表参照）
    claude_args: "--max-turns 12 --model claude-opus-4-7"
    settings: |
      {"permissions": {"deny": ["WebFetch", "WebSearch", "AskUserQuestion", "Edit"]}}
    prompt: |
      ## チケット: ${{ matrix.issue_key }}
      タイトル: ${{ env.ISSUE_SUMMARY }}
      ...
      Step 2: plan.txt か clarification.txt のどちらかを Write する。
```

認証は `anthropic_api_key` を Secrets に入れて従量課金で動かす形にしています。

Claude には `plan.txt` か `clarification.txt`（質問が必要なケース）を書かせるだけ。投稿は shell で行います:

```
# Plan ステージ後の shell で
content = open("plan.txt").read().strip()
post_comment(content)  # Backlog API 呼び出し
```

### パターン B: Routines（claude.ai）

claude.ai 側の **Claude Code Routines**（以下「Routines」）という機能で、スケジュール実行や GitHub イベント発火で動く Claude セッションを作れます。Routines 本体は **claude.ai の Web UI** で作成・編集するのが標準ですが、**Claude Code セッション内では `RemoteTrigger` ツール**を経由して Routines を一覧取得・作成・更新・実行することもできます。実際この記事の元になっている [SUBLINE](https://www.subline.jp/) の Routines も `RemoteTrigger` で作らせました（GitHub イベントトリガーの最終設定だけは Web UI が必須）。

```
# Claude Code から呼べる RemoteTrigger ツールのアクション
RemoteTrigger list                    # 一覧
RemoteTrigger get    <trigger_id>     # 詳細取得
RemoteTrigger create <body>           # 作成
RemoteTrigger update <trigger_id>     # 更新
RemoteTrigger run    <trigger_id>     # 即時実行
```

外部システムから Routines を発火させたい場合は、公開された `/fire` エンドポイントが使えます（[公式ドキュメント](https://platform.claude.com/docs/en/api/claude-code/routines-fire)）:

```
curl -X POST https://api.anthropic.com/v1/claude_code/routines/{trigger_id}/fire \
  -H "Authorization: Bearer ${ROUTINE_FIRE_TOKEN}" \
  -H "anthropic-version: 2023-06-01" \
  -H "anthropic-beta: experimental-cc-routine-2026-04-01" \
  -H "Content-Type: application/json" \
  -d '{"text": "任意の追加コンテキスト"}'
```

`ROUTINE_FIRE_TOKEN` は Anthropic API key そのものではなく、**Routines ごとに発行する fire 用の bearer token** です。Routines 詳細画面で発行・管理します。混同しないよう環境変数名を分けておくと安全。

こちらは **一つの長大なプロンプト**で plan / fix / review / 統合分析 / PR 作成 / レビューコメントまで全部やらせます。Claude がすべてのツール（Bash, Read, Write, Edit, Glob, Grep）を駆使して工程を進めます。

#### 比較

|  | GitHub Actions | Routines |
| --- | --- | --- |
| トリガー | cron / push / GitHub イベント / dispatch | スケジュール / GitHub イベント / API |
| 課金 | **Anthropic API key（従量）** | **claude.ai サブスク内** |
| トークン効率 | ◎（shell と分担） | △（全部 Claude） |
| 月額コスト感 | 件数 × API 料金が積み上がる | サブスクのプラン上限内で動く |
| **モデル切替** | **◎ ステージごとに切替可** | △ Routines ごとに固定（1 セッション = 1 モデル） |
| 構造 | staged YAML | 単一プロンプト |
| デバッグ | ジョブログ | Routines の実行ログ |
| メンテ | YAML 更新で即反映 | Web UI で更新（`RemoteTrigger` 経由も可） |

#### ステージごとにモデルを使い分けられるのは GitHub Actions だけ

GitHub Actions パターンの隠れた強みが **ステージごとに `--model` を変えられる**点です。`claude-code-action@v1` の `claude_args` で指定するだけ。

```
- name: "[Plan] Claude: 改修方針を策定"
  uses: anthropics/claude-code-action@v1
  with:
    claude_args: "--max-turns 8 --model claude-opus-4-7"   # 判断重要 → Opus

- name: "[Consolidate] Claude: 類似 PR の判定"
  uses: anthropics/claude-code-action@v1
  with:
    claude_args: "--max-turns 6 --model claude-haiku-4-5"  # 軽量タスク → Haiku
```

[SUBLINE](https://www.subline.jp/) の実運用では以下のように分けています:

| ステージ | モデル | 理由 |
| --- | --- | --- |
| 改修方針策定 | **Opus** | 原因分析と方針判断が後続の精度を左右 |
| コード修正 | **Opus** | 実装品質が PR 品質に直結 |
| 差分確認・追加修正 | **Opus** | 追加実装の可能性がある |
| 類似 PR 判定 | Haiku | パターンマッチ、軽量 |
| PR 本文生成 | Sonnet | 要約とテストケース提案 |
| PR レビューコメント生成 | Sonnet | コードレビュー判断 |
| 統合追記レビュー生成 | Haiku | 短い追記 |

単価の相対比は **Opus が Sonnet の約 1.7 倍、Haiku の 5 倍**（2026 年 5 月時点の output 単価ベース）。絶対値は変動するので [公式ページ](https://platform.claude.com/docs/en/about-claude/pricing) を都度参照してください。**重要な判断だけ Opus・テンプレ埋めは Haiku** に分けるだけでコスト効率がはっきり変わります。

一方の **Routines は 1 セッション = 1 モデル**で動く仕様（Routines ごとにモデルを指定するが、1 つの Routine 内ではタスクごとに切り替えられない）。Opus にしておけば品質は高いけど Haiku で十分な軽量タスクにも Opus が使われる無駄が出ます（サブスク内なので追加課金にはなりませんが、プランの利用枠（5h あたりのメッセージ上限）には早く到達します）。

このあたりは併用すれば「いいとこ取り」というよりは**トレードオフが明確に出る部分**なので、件数・コスト・運用の好みに合わせて好きな方を主軸にするのが現実的です。

#### コスト面の現実的な話

* **GitHub Actions パターン**: shell との分担でトークン消費は抑えられているが、Anthropic API key の従量課金は **件数に応じて積み上がる**
* **Routines パターン**: トークン効率は悪いが、**Claude.ai のサブスク内で動くので追加課金は発生しない**（プランの利用枠（5h あたりのメッセージ上限）には早く到達する点に注意）

件数とサブスクのプラン状況によって、どちらが得かは結構変わります。

#### 代替 AI ツールも視野に

API 課金で運用する場合、Anthropic 以外の選択肢も:

* **OpenAI Codex CLI / OpenAI API**（GPT 系）→ Claude API より単価が安いケースもある
* **Gemini CLI**（Google）→ 無料枠が広い

GitHub Actions の構造（shell + AI 判断ステップ）は AI ベンダーに依存しない形になっています（OpenAI / Gemini への置き換えは [SUBLINE](https://www.subline.jp/) では未検証）。Claude action の部分だけ差し替えて他ツールに置き換えやすい設計です。

#### 実運用での選択

**[SUBLINE](https://www.subline.jp/) ではメインを Routines** で運用しています。サブスク内で完結するのが楽で、claude.ai/code/routines の実行ログでも経過を追えるためです。GitHub Actions パターン（`claude-autofix-backlog.yml`）も同じ動作で組んであり、**バックアップとしていつでも動かせる状態**で温存しています。

二段構えにしておけば、運用状況や件数の変化に応じて主軸を切り替えられます。

#### 共通設計: 初動の安全装置としての件数キャップ

両パターンとも **1 回の実行で対象チケットを N 件に絞る** キャップを入れています（[SUBLINE](https://www.subline.jp/) 現状: GHA = 10 件、Routines = 5 件）。各チケットは独立に処理されるので 1 件あたりのクオリティは件数で変わりません。設計上はタイムアウトしても次回再開できるので、理屈の上では制限不要です。

ただ実運用の初動では、

* サブスク 5h メッセージ枠の瞬発消費を抑える
* 万一プロンプトのバグが潜んでいた時の被害範囲を限定する
* Backlog 通知や PR レビュー対象が一気に増えるのを防ぐ

の 3 点から保守的に絞っています。残りのチケットは次回 cron / トリガーで拾うので最終的にはすべて処理されます。**運用が安定したら段階的にキャップを上げていく方針**で、Phase 1（GHA 10 / Routines 5）から Phase 2（GHA 15 / Routines 7）→ Phase 3（上限なし）と広げていく予定です。Routines 側を少なめにしているのは、1 セッションが長期化するとコンテキストが詰まりやすい挙動への配慮。

## 4. 段階的再開の仕組み

自動化で一番怖いのは「途中で失敗したときに次にどこから再開すべきか分からなくなる」こと。**この設計が本フロー最大の工夫**です。

2-3 節の対応表（ステータス × コメント × ブランチ × PR × レビュー）がそのまま **再開判定表** を兼ねています。加えて以下 2 つの軽い check を組み合わせるだけで、毎回ゼロから「今どこから始めるべきか」が決まります:

* **OPEN PR の有無**: PR が既にあれば PR 作成スキップ、レビューコメントだけ追記
* **feature ブランチのコミット数**: ブランチだけあってコミット 0 件なら fix から再実装

代表的な再開ケース:

| 落ちたタイミング | 次回どこから再開するか |
| --- | --- |
| `git push` 直後にネットワーク失敗 | review から |
| モデルの turn 切れで plan だけ済んだ | fix から |
| 開発担当者が手動で空 feature ブランチを作ってた | fix から（コミット 0 件として再実装） |
| PR は出来てたがレビュー追記が落ちた | section 6 のみ実行（救済モード） |

**「何度実行しても同じ状態に落ち着く」のが目標**です。これがないと、自動化が失敗する度に人間が引き継ぐ羽目になり、結局工数が減りません。

### 担当者切り替えで「再開可能ポイント」を作る

Claude が「期待動作わからないので確認してください」とコメントしたら、**担当者を Claude → 開発担当者に変更**します。これで、

* 次の自動実行ではフィルタで除外されて Claude は触らない
* 開発担当者が答えて担当者を Claude に戻すと、再ピックアップされて plan からやり直し

確認コメントには本文先頭に `[Claude確認]` という marker を入れていて、**「開発担当者が回答済み」と判別できるようにしています**:

```
[Claude確認]
SUBLINE-XXXX の対応で確認です。

Q1. 〇〇の動作はどちらが期待値ですか？
  A) 〇〇〇（現状維持）
  B) 〇〇〇（変更後）

確認後、担当者を Claude Code に戻していただければ対応します。
```

## 5. PR 統合と吸収チケットの自動クローズ

同一リリースに向けて複数のチケットが類似修正だった場合、**個別 PR を量産せずに統合**する機構があります。

### 類似判定

統合判定ステージで、同じリリースブランチ向けの他 OPEN PR と diff を比較します。Claude に以下の観点で判定させます:

* **同じファイルへの変更**
* **同じ例外・エラーパターンへの対処**（同じ例外クラスや同じエラーメッセージへの対応など）
* **同じガードパターンの追加**（null チェック、状態確認、try-catch 等）
* **同じメソッド・関数への修正**

判定基準はプロンプトに自然言語で書くだけで、ロジックを実装する必要はありません。[SUBLINE](https://www.subline.jp/) の場合は Android 固有の例外パターンや Fragment のガードを観点として列挙しています。

### 吸収処理

「類似」と判定されたら:

1. 処理対象の feature ブランチを **leading PR のブランチにマージ**
2. leading PR 本文に **関連チケットリンクと統合理由を追記**
3. 処理対象のチケットに `[Claude統合通知]` コメントを投稿
4. 処理対象のチケットを **`statusId=4` (完了)** に変更

これで、複数のクラッシュログ起票がほぼ同じ原因だった場合に、PR が 1 本にまとまります。

#### push の安全弁

leading ブランチへの push は、**素の `--force` ではなく `--force-with-lease` を使うこと**。さらに、

* 統合前に **leading ブランチに人間の追加コミットが入っていないか** を確認（最後の commit author が GitHub Actions / Claude Code Bot かを check）
* マージ衝突が出たら統合せず、Backlog に `[Claude確認]` コメントを残してエスカレーション

の 2 つを入れておくと、開発担当者が手動で追ったコミットを AI が上書きする事故を防げます。`force-push` は強力なだけに、AI 自動化フローでは安全弁を 2 段つける価値があります。

### コメント例

```
[Claude統合通知]
本チケットの修正は SUBLINE-2434 の PR #113 でまとめて対応するため、
本チケットは完了としてクローズします。

統合理由: 同じファイルへのガード追加で対処内容が重複するため。
```

## 6. 開発者テスト後の指摘も Claude に任せる

PR ができても、実機で動作確認すると不備が見つかることがあります。「ProgressBar が消えなくなった」「特定条件でエラーが出る」など。

こういう **追加指摘も Claude に対応させる仕組み**を 2 通り用意しています。

### 6-1. パターン A: `@claude` メンションで対応（GitHub Actions）

PR や Issue のコメントに `@claude` を含めると `claude-on-mention.yml` が発火し、Claude が対応してくれます。

```
# .github/workflows/claude-on-mention.yml（抜粋）
on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]

jobs:
  claude:
    if: contains(github.event.comment.body, '@claude')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
```

PR コメントに「@claude ProgressBar が消えない件、対応してください」と書くだけで、Claude が PR の文脈を読んでブランチに追加コミットを乗せてくれます。**ピンポイントな指示**に向いています。

### 6-2. パターン B: `claude-fix` ラベルで対応（Routines GitHub イベントトリガー）

Claude Code Routines には **GitHub イベントトリガー**機能があり、`pull_request.labeled` イベントで起動できます。

設定:

* 対象 PR に `claude-fix` ラベルを付ける
* フィルタ: `Labels` `is one of` `claude-fix`

ラベルを付けると Claude が起動し、PR の全コメント（レビュー本文・インラインコメント・通常コメント）を統合的に読んで判定:

| 分類 | 動作 |
| --- | --- |
| (A) コード修正が必要な明確な指摘 | ブランチに修正をコミット → 完了コメント → ラベル外し |
| (B) 質問・確認のみ | 回答コメントだけ投稿 → ラベル外し |
| (C) 判断が分かれる大規模変更 | 確認コメントを残してエスカレーション（ラベルは付けたまま人を待つ） |

**指摘をまとめて投げる**運用に向いています。複数の指摘を PR コメントに書き終えてからラベルを 1 個付けるだけで、Claude が全部読んで対応します。

### 6-3. 使い分け

|  | `@claude` メンション | `claude-fix` ラベル |
| --- | --- | --- |
| 起動条件 | コメント本文に `@claude` | ラベル付与 |
| 適した場面 | ピンポイントの単発指示 | 複数指摘をまとめて投げたい |
| 誤発火リスク | 中（言及で誤発火しうる） | 低（明示的にラベルを付ける必要） |
| 実装ベース | GitHub Actions | Claude Code Routines |

両方併用していますが、メイン運用は **`claude-fix` ラベル方式**に寄せています。複数指摘を整理してからラベル 1 個で発火、という方が落ち着いて運用できるためです。

## 7. 実運用での効果

### 工数削減（体感ベース）

クラッシュ 1 件の従来工数（体感ベース）:

| 作業 | 時間 |
| --- | --- |
| クラッシュ発見 → Backlog 起票 | 5〜10 分 |
| 原因調査 → 修正実装 | 30〜60 分 |
| PR 作成 → テスト項目記入 | 30〜60 分 |
| **合計** | **約 1〜2 時間** |

これが今は **「開発担当者が PR の差分とテスト項目を確認する 5〜10 分」** になっています。1 件あたり **約 1〜2 時間の削減** の体感です。月に 10 件起票されるチームなら月 10〜20 時間ほどの効果になる計算です。

定量的に追うなら **完走率 / 開発者テストでの差し戻し率 / AI 修正後に人手で修正を入れた率** あたりを記録すると、自動化の品質を継続的に評価しやすくなります（[SUBLINE](https://www.subline.jp/) では運用が固まったらここを計測していきたい）。

### 体感としての変化

「定時の時間以外に作業が進んでいる」感覚は、運用してみると想像以上に効きます。朝起きるとブランチに数件分の修正コミットが乗っていて、レビュー対象の PR が並んでいる、という状態は当初思い描いていた **「寝ている間に作業が進む」理想** に近いです。

人間側の時間の使い方も変わります:

* これまで: クラッシュ発見 → 調査 → 実装 → PR → テストで 1 日が埋まる
* 今: PR レビューと開発者テストに集中、空いた時間で **「何を作るか」「何が求められているか」** を考える方に回せる

### 副次的な効果: AI と並走する感覚

工数削減や品質維持の話とは別に、副次的な効果として「夜中に AI が並列で改修を進めてくれている」体験そのものが、助手というか部下というか、**一緒に作業してくれるチームができたような感覚** に近いです。進捗管理や対応方針の判断を独断で進めていて、並走する作業者がいない運用では、自分以外に同じ手を動かしてくれる存在がいるだけで地味に効きます。

### 開発担当者が今やっていること

* 担当者切り替えのトリガー操作（5 秒）
* マイルストーンへのリリース予定バージョン記入（10 秒）
* Claude が確認質問を出してきたときの回答（1〜3 分）
* PR のコード差分の最終チェック（5〜10 分）
* マージ操作（10 秒）

### 実運用で見えてきた弱点

* **同じ箇所の細かい改修方針のブレ**: Claude のセッションごとに微妙に方針が違うので、レビューで開発担当者が微調整することはある
* **コミットメッセージのスタイル**: 統一感が出にくい（プロンプトで指示しているがブレる）
* **テスト項目の具体性**: 「正常に動作すること」みたいな曖昧表現を防ぐために、プロンプトに **品質基準（NG ワードリスト含む）** を入れている。それでもたまにブレるので、テンプレ調整が地味に大事
* **Plan ステージの turn 不足で失敗するクラッシュがある**: 例外パターンが複雑で読むファイルが多いケースは `--max-turns` を増やして対応中。失敗してもステータスは未対応のまま残るので、再実行で再着手できる設計が効きます

## 8. 他のケースにも応用できる設計パターン

ここまでは [SUBLINE](https://www.subline.jp/) の話ですが、**仕組み自体は AI に定型作業を任せる多くのケースに応用できる** はずです。本記事の本文で出てきた工夫のうち、特に他のドメインで効きそうなものを 4 つに絞って整理します。

### 8-1. 状態 × 属性の組み合わせで「途中から再開」を担保する（本文最大の工夫の再掲）

[2-3 節](#2-3.-%E7%8A%B6%E6%85%8B-%C3%97-%E3%83%96%E3%83%A9%E3%83%B3%E3%83%81-%C3%97-pr-%E3%81%AE%E5%AF%BE%E5%BF%9C%E8%A1%A8) と [4 章](#4.-%E6%AE%B5%E9%9A%8E%E7%9A%84%E5%86%8D%E9%96%8B%E3%81%AE%E4%BB%95%E7%B5%84%E3%81%BF) で書いた通り、タスクトラッカー上の **ステータス・コメント・成果物の有無** を組み合わせて、AI が「今どこから再開すべきか」を毎回ゼロから判定できるようにします。

応用先:

* **Web アプリの Issue 自動対応**: GitHub Issues のラベルやコメント、PR の状態を判定
* **データパイプライン**: 中間ファイルの有無で次のステップを決める
* **記事作成 AI**: 下書き・校正・公開の段階を Notion などのプロパティで管理

**「自動化が失敗したら最初からやり直し」だと、結局人間が引き継ぐ羽目になる**。再開性をどう設計するかで自動化が実用に耐えるかが決まります。

### 8-2. 類似タスクは 1 つのアウトプットに統合する

[5 章](#5.-pr-%E7%B5%B1%E5%90%88%E3%81%A8%E5%90%B8%E5%8F%8E%E3%83%81%E3%82%B1%E3%83%83%E3%83%88%E3%81%AE%E8%87%AA%E5%8B%95%E3%82%AF%E3%83%AD%E3%83%BC%E3%82%BA) で書いた PR 統合の仕組みを、他の領域でも使えます。

応用先:

* **Web アプリのバグ修正 PR**: 同じファイルへの修正は 1 PR にまとめる
* **記事の更新**: 似たトピックの執筆依頼は 1 つの記事にマージ
* **データ修正バッチ**: 関連するレコード修正をまとめてリリース

判定基準は AI に自然言語で渡せばいいので、ロジックは数行で書けます。**人間のレビュー負荷を抑える鉄板パターン**。

### 8-3. テンプレ + 曖昧 NG リストで品質を揃える

AI に成果物を作らせると **フォーマットや品質がブレる** のが最大の弱点。本文の PR テンプレでも触れた通り、テンプレ + 禁止表現リストを渡すだけで出力品質が安定します。

品質基準の例（PR テスト項目用にプロンプトへ埋め込み）:

```
- 「確認する」「操作を行う」など曖昧な表現は禁止
- ユーザーが行う具体的な操作手順・条件を書く
- 「正常に動作すること」「表示されること」など曖昧な期待値は禁止
- 画面上の具体的な変化・表示内容を書く
```

応用先:

* **コードレビューコメント**: 観点リストと NG 表現を渡してフォーマットを統一
* **議事録生成**: 「決定事項」「TODO」「保留事項」のセクション必須化
* **顧客対応メール**: 構成と禁止表現をプロンプトに含める

AI の出力品質は **プロンプトに書いた品質基準で 7〜8 割決まる** 体感です。

### 8-4. Marker 文字列 + ファイル早見表でトークン浪費を防ぐ

2 つセットで効く小ネタです。

**Marker 文字列**: AI が過去に書いたコメントを次回参照する時、毎回内容を読んで分類するのはトークンの無駄。先頭に固定 marker を入れて文字列マッチで瞬時に判別する。

| Marker | 意味 |
| --- | --- |
| `[Claude改修方針]` | 方針コメント。次回 fix ステージから再開できる |
| `[Claude確認]` | 仕様確認の質問。人間の回答待ちなので一時停止 |
| `[Claude統合通知]` | 別 PR に統合された旨。チケットは完了扱い |

**ファイル早見表**: AI に「目的のファイルを探させる」のは時間とトークンの無駄。プロジェクトルートに `CLAUDE.md` を置き、機能 → ファイル対応をプロンプトにも埋め込みます。

```
# CLAUDE.md（プロジェクトルートに置く。Claude Code が自動で読む）

## 機能 → ファイル対応
| 機能 | パス |
|---|---|
| 機能A | app/.../FeatureAScreen.kt |
| 機能B | app/.../FeatureBScreen.kt |
```

応用先:

* **Web プロジェクト**: 「機能名 → コンポーネント / ルーター / コントローラ」のマッピング
* **ドキュメント運用**: 「トピック → docs 配下のディレクトリ」の対応表

#### 補足: プロンプトに「迷ったら進む」しきい値を書く

AI は完璧主義で確認質問を投げがちなので、しきい値を明示します:

```
- 明確な不具合 → 即着手
- 実装中に判断できることは自分で判断してよい
- 80% わかれば進める。完璧な仕様は不要
- 迷ったら進む。確認は「仕様で複数解釈が成り立つ場合」のみ
```

これがないと、些細な疑問で自動化が止まります。

> ※ 旧バージョンの「担当者欄を AI と人間のバトン渡しに使う」「Marker 文字列で自分の過去出力を識別」は [2-4 節](#2-4.-%E3%81%93%E3%81%AE%E8%A8%AD%E8%A8%88%E3%81%AE%E6%97%A8%E5%91%B3) と本節 8-4 で既に触れているので、ここでは応用例の列挙のみに留めています。

## 9. 今後広げたいスコープ

現状は「原因と修正方針が比較的明確なクラッシュ対応」に絞っているからこそ自動化が回っています。次のステップとして広げたい領域は 2 つ:

### 9-1. 新機能の事前調査・実現可能性まとめ

ざっくり「こんな機能を入れたい」というチケット起票だけで、AI が:

* **既存画面の拡張で対応できるか**（流用元の特定）
* **必要な SDK 拡張があるか**（外部依存の見立て）
* **似た要件を過去にどう実装したか**（コードベース横断調査）

を調べて Backlog コメントとして返してくれる、というフローを足したい。クラッシュ修正の plan ステージとほぼ同じ構造で組めるはずです。

### 9-2. アイデア出しの土台作り

AI に常時 watch させて以下を提供してもらう体制があると、定時中の人間の集中先が「企画判断」に寄せられます:

* アプリに組み込んでいる SDK で **新たに使える機能** の紹介
* プロダクトと相性が良さそうな **技術トレンド情報** の整理
* ユーザー要望・レビューの **傾向分析**

### ゴール: 「人間は何を作るかを考える」「AI は調査と実装を回す」

理想は、定時中に人間は **「何が求められているか・何を作るか」** を考えることに集中して、調査と実装の手を動かす部分は AI がチケット駆動で進めてくれる分担です。クラッシュ自動化はそのフローの **最初のピース** という位置付けで、ここから少しずつ範囲を広げていきたいと思っています。

## まとめ

クラッシュ報告 → 改修 → PR → 開発者テスト指摘対応まで、**人間の介入を「開発者テストの実機確認」と「最終確認の 5〜10 分」だけ**に絞り込めるフローを Claude × GitHub Actions × Routines で組みました。

### 効いたポイント（持ち帰り用 7 項目）

* **🥇 状態と属性の組み合わせで再開ステージを決める** — 本フロー最大の工夫。途中失敗が前提でも回る
* **担当者欄を「AI と人間のバトン渡し」インターフェースに使う** — 既存のタスクトラッカー機能だけで対話が成立
* **類似 PR の自動統合** — レビュー対象の PR を 1 本にまとめて人間の負荷を抑える（force-with-lease + 人間コミット検出で安全弁を 2 段）
* **PR テンプレ + 曖昧 NG リスト** で AI 成果物の品質を揃える
* **`CLAUDE.md` / ファイル早見表 / marker 文字列** でトークン浪費と状態判定コストを下げる
* **開発者テスト指摘も `claude-fix` ラベル + Routines の GitHub イベントトリガーで自動対応**
* **GitHub Actions と Routines の 2 系統**を組んで、コスト・要件に応じて主軸を切替（現状は Routines がメイン、GHA はバックアップ）
* **初動は件数キャップで安全運転**（Routines 5 / GHA 10）— 運用が安定したら段階的に上げていく

### コストとツール選択の落とし所

* **サブスクをすでに使っている・件数が多い** → Routines に寄せると追加コストはサブスク内に収まる（**[SUBLINE](https://www.subline.jp/) ではこちらで運用中**）
* **ログを段階別に追いたい・CI と連携したい・モデルをタスク別に最適化したい** → GitHub Actions（API key 従量。バックアップとして待機）
* **更にコストを下げたい** → OpenAI / Gemini / 他 CLI への置き換えを検討（GitHub Actions の AI ステップだけ差し替える形にすれば、shell ＋ AI 判断ステップの構造はベンダー非依存）

### この自動化フローが向かないケース

* **大規模リファクタ**: 範囲が広く判断ポイントが多いものは、本フローのように自動でクローズさせるのは難しい（個別に AI に手伝わせる形が現実的）
* **要件が曖昧な機能追加**: 確認コメントの往復が増えて手戻り工数が増えがち（要件定義を先に固める必要あり）
* **テスト要件が複雑なもの**: 複数モジュール連携やタイミング依存があると、テスト項目のテンプレ化が効きにくい

クラッシュ対応のような\*\*「原因と修正方針が比較的明確で、コード変更がローカル」なタスク\*\*との相性が抜群です。同じ性質を持つタスクなら、本記事のフローはそのまま転用できると思います。

寝ている間に並走してくれる仕組みは、想像以上に開発体験を変えます。参考になれば嬉しいです。
