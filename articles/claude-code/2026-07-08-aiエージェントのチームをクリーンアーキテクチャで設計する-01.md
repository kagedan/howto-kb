こんにちは！株式会社カンリーでエンジニアをやっている市川です。

## はじめに

Claude Code のようなコーディングエージェントで「複数のエージェントにチームを組ませてレビューさせる」というのは、いまや珍しい話ではありません。役割ごとにサブエージェントを分け、それぞれの観点で調査・報告させ、リードが集約する——という構成は、多くのチームが試し始めていると思います。

この記事で伝えたいのは、その **一歩先** です。

私たちは、AIエージェントのチームを組むときに、**アプリケーション本体で使っているクラス設計の規約（クリーンアーキテクチャ）と、まったく同じレイヤー構成で組みました。** 「エージェントを分ける」だけでなく、「エージェントたちが依存するプロンプト・知識・手順」そのものを、レイヤードアーキテクチャの流儀で構造化した、という話です。

狙いは大きく2つです。

* 🪶 **コンテキストの節約** — 無駄なプロンプト（Markdown）の読み込みを減らし、各エージェントに「必要なものだけ」を渡す
* 📦 **ドメイン知識の外出し** — プロジェクト固有の制約や知識を独立させ、使い回しやすい構成にする

---

## 背景：なぜ「チーム」が必要になったのか

きっかけは、大規模なコードレビューを自動化したいという要求でした。

1つのレビュー対象に対して、確認したい観点がとにかく多い。破壊的変更、依存パッケージの影響、コード全体への波及範囲、テストの妥当性、セキュリティ——といった具合に、観点が増えるほど1つのプロンプトに詰め込むのは無理が出てきます。しかも変更規模が大きいケースでは、1回のレビューで数百〜千を超えるファイルが動くこともあり、単一のエージェントでは物理的に追いきれません。

そこで、**観点ごとに専門のエージェントを分業させるチーム構成**を作りました。ここまでは「よくあるAIチーム」です。

本題は、そのチームの **内部設計** をどうやって破綻させずに保つか、でした。

---

## 前提：AIチームも「普通のソフトウェア」と同じ問題に陥る

エージェントを分けて増やしていくと、通常のソフトウェア開発とまったく同じ問題にぶつかります。

* **密結合**：あるエージェントのプロンプトを直したら、別のエージェントの挙動まで壊れる
* **バラバラ実装**：似たような調査手順を、エージェントごとに少しずつ違う書き方で持ってしまう
* **肥大化**：共通ルールを全エージェントにコピペし、どれが正なのか分からなくなる

これは要するに、コードで散々戦ってきた問題です。だったら、**コードで使っている解法をそのまま持ち込めばいい** ——というのが今回のアイデアの出発点でした。

私たちのプロジェクトでは、新規コードを追加するときのルールとして、レイヤードアーキテクチャを採用しています。

```
Controller → UseCase → Service → Repository
```

* **Controller**：リクエストの受付とレスポンス返却のみ。ロジックを持たない
* **UseCase**：ユースケース単位のオーケストレーション。Service を組み合わせる
* **Service**：単一責務のビジネスロジック
* **Repository**：データアクセス（DB・外部API）

依存の向きは常に上から下への一方向。この規約を、**そのままAIチームの設計にも適用しました。**

---

## `.claude/` をレイヤーで構成する

エージェントの定義ファイル群（`.claude/` 配下）を、アプリと同じ考え方で5つのレイヤーに割り当てます。

| レイヤー | 配置場所 | 役割 | アプリ側の対応 |
| --- | --- | --- | --- |
| **Controller** | `.claude/skills/` | スラッシュコマンドを受けてチームを起動するだけ | `Controllers/` |
| **UseCase** | `.claude/agents/` | チーム（Lead + Specialist + Critic）。Service を組み合わせて仕事を遂行 | UseCase クラス |
| **Service** | `.claude/services/` | 単一責務の手順（PRの読み取り、影響範囲分析、制約チェック…） | Service クラス |
| **Repository** | `.claude/repositories/` | データアクセスの方法（ソースコード検索、外部データ取得…） | Repository クラス |
| **Domain** | `.claude/domain/` | 知識・制約（フレームワークの制約、検索パターンなど） | Model / 値オブジェクト |

ポイントは、アプリ本体を触っているエンジニアなら **「どこに何があるか」が説明なしで分かる** ことです。エージェント設計のための新しいメンタルモデルを覚える必要がありません。

---

## 抽象基底クラス：Lead / Specialist / Critic

エージェントには、役割ごとに共通する振る舞いがあります。そこで、Template Method パターンの要領で **抽象基底クラス** を用意しました（`.claude/agents/_abstract/`）。

| 基底 | 役割 | 共通化されている処理 |
| --- | --- | --- |
| **Lead** | オーケストレータ（コードは編集しない） | 準備 → 委譲 → 集約 → 判断 という一連のフェーズ |
| **Specialist** | 担当領域の調査・分析・報告 | 推測の禁止 / 報告フォーマット / 権限の制約 |
| **Critic** | Lead の報告を厳格にレビュー | 曖昧さ・楽観視・根拠不足の追及方法 |

個別のエージェントは、この基底を `extends:` で継承し、**自分に固有の部分だけを書きます。** 「報告は必ずこのフォーマットで」「推測で埋めるな」といった共通ルールを、各エージェントにコピペする必要がなくなります。

---

## エージェントの宣言は「クラス図」そのもの

具体的に、あるリード役のエージェントの定義（frontmatter）を見てみます。

```
---
name: review-tech-lead
description: 全レビュー結果を集約し技術的リスクを正直に報告するテクニカルリード
tools: Read, Grep, Glob, Write, Task
model: opus
extends: agents/_abstract/lead.md      # ← Lead 基底クラスを継承
uses:                                  # ← 依存する Service を宣言
  - .claude/services/read-pr.md
  - .claude/services/check-constraints.md
  - .claude/services/search-related-docs.md
delegates:                             # ← 委譲先の Specialist
  - review-breaking-changes
  - review-test-reviewer
  - review-dependency-reviewer
  - review-code-impact
  - review-security-reviewer
reports_to:                            # ← レビューを受ける相手（Critic）
  - review-pm
---
```

`extends` / `uses` / `delegates` / `reports_to` の4つで、継承・依存・委譲・報告の関係を宣言する。これはもう、**クラス図をそのまま書き下している** のと同じです。関係が明示されているので、あとから読んでも「このエージェントが何に依存し、誰に何を任せているか」が一目で追えます。

---

## `uses:` は「依存宣言」

`uses:` は、そのエージェント／手順が依存する下位レイヤーを宣言するものです。実際に連鎖を辿ってみます。

```
# ① agents/review-tech-lead.md
uses:
  - .claude/services/check-constraints.md   # ← Agent が Service に依存
```

```
# ② services/check-constraints.md
uses:
  - .claude/repositories/read-framework-constraints.md  # ← Service が Repository に依存
  - .claude/repositories/read-language-constraints.md
  - .claude/repositories/search-source-code.md
```

```
# ③ repositories/read-framework-constraints.md
uses:
  - .claude/domain/framework-constraints/   # ← Repository が Domain に依存
```

そして、リード役のエージェントがこの依存グラフを **事前に辿り、1つのファイルに連結（バンドル）** してから各 Specialist に渡します。Specialist は1回の読み込みで、「自分の仕事に必要な手順・知識・制約」がすべて手元に揃う——という仕組みです。

---

## 狙い①：コンテキストの節約

ここが、このアーキテクチャの本命の狙いの1つです。「必要なものだけを渡す」を、3つの軸で徹底しています。

| 軸 | 分けなかったら | 分けてやっていること |
| --- | --- | --- |
| **① `uses:` の連鎖** | 全エージェントが全 Markdown を抱え込む | 依存先の Markdown だけが届く（無関係な制約は届かない） |
| **② must-read の分離** | 全役割のルールを全員が読む | 自分の役割分の must-read だけが届く |
| **③ データの役割別配分** | 巨大な diff や検索結果が全員に届く | 担当領域の差分だけが、担当エージェントにだけ届く |

結果として、どのエージェントも **「自分の判断に必要なものだけ」** を受け取ります。コンテキスト上限を気にせず、それぞれが自分の領域を深く掘れる、というのが大きな利点です。単一の巨大プロンプトに全部を詰め込む方式とは、判断の質がはっきり変わります。

---

## 狙い②：ドメイン知識の外出し（と、レイヤーごとの責務分担）

プロジェクトによって、言語・フレームワーク・ツールは違います。この「違い（分岐）」を、**レイヤーごとに役割を分けて吸収** します。同じ「分岐」でも、層によって持つべき責務が違うのがポイントです。

* **Service**：どの Repository を組み合わせるかを定義する。「制約チェックは、言語制約・フレーム制約・ソース検索の3つを順に使う」といった *手順* を宣言するだけ
* **Repository**：アクセス手段の分岐を定義する。「状況に応じて Grep / Glob を使い分ける」といった、言語やツールに依存しない *汎用的な手段* を持つだけ
* **Domain**：プロジェクト固有の知識を保持する。ディレクトリ構成・検索パターン・フレームワーク固有の事情は、**すべてここに閉じ込める**

```
# Service：複数 Repository を組み合わせる手順を宣言
# services/check-constraints.md
uses:
  - .claude/repositories/read-language-constraints.md
  - .claude/repositories/read-framework-constraints.md
  - .claude/repositories/search-source-code.md    # ↓ この連鎖を追う

# Repository：ツール非依存の汎用手段（Grep / Glob）
# repositories/search-source-code.md
uses:
  - .claude/domain/architecture.md                # プロジェクト固有は Domain へ

# Domain：このプロジェクトのディレクトリ構成・検索パターン
# domain/architecture.md  ← 固有の事情はここに閉じる
```

上位レイヤー（Repository / Service）を「無垢」に保ち、プロジェクト固有の値を Domain に寄せる。これがドメイン知識の外出しの実体です。

---

## 効果：多くの場合、Domain だけ触ればいい

プロジェクト固有の値が Domain に集まっているので、上位レイヤーは無修正のまま使い回せます。

| 変えたいこと | 修正するファイル（Domain のみ） |
| --- | --- |
| ログ基盤を別のサービスに変える | `domain/log-infrastructure.md` |
| 通知先チャンネルや検索キーワードが変わった | `domain/channels.md` |
| 新しいフレームワーク／言語バージョンに対応する | `domain/framework-constraints/` に追記 |

**Agent / Service / Repository は一切触らなくていい。** 依存の向きが一方向（上→下）なので、下を変えても上に伝播しないからです。差し替え・拡張のしやすさは、レイヤードアーキテクチャそのものの恩恵と言えます。

副次的に、こんな効果も得られました。

* 🔄 **差し替えが容易**：Repository を1ファイル直すだけで、全エージェントに反映される
* 🧩 **拡張が容易**：新しい Specialist は `extends:` と `uses:` を書くだけで追加できる

---

## 正直なところ：まだ道半ば

きれいに書きましたが、Domain への完全分離は **まだ道半ば** です。Repository や Service にプロジェクト固有の値が残っている箇所もあり、少しずつ Domain 側へ寄せている最中です。「一方向の依存」も、油断すると崩れます。この構造は、意識して維持し続けないと、あっという間に元の密結合に戻ります。

それでも、**「AIチームも普通のソフトウェアと同じ設計原則で組める」** という手応えははっきりありました。エージェントが増えても破綻しにくく、既存のエンジニアがメンタルモデルを流用できる。この2点だけでも、採用する価値は十分あると感じています。

---

## まとめ

* 狙いは2つ：**コンテキスト節約** と **ドメイン知識の外出し**
* そのために、アプリ本体と **同じレイヤー構成** を `.claude/` 配下に適用した
* `extends` / `uses` / `delegates` / `reports_to` で、依存関係を **クラス図のように宣言** する
* 結果、各エージェントに「必要なものだけ」を渡せるようになった
* 副産物として、**再利用・差し替え・拡張** もしやすくなった

AIエージェントのチーム設計に悩んでいる方は、「新しい設計手法」を探す前に、**すでにコードで使っている設計原則をそのまま持ち込めないか** を考えてみると、案外すっきり整理できるかもしれません。
