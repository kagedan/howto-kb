こんにちは、ギークプラスの千葉です！

今回は私が実際に開発で活用しているAIエージェントチーム（以下AIチーム）による開発を整理しつつ、弊社のエンジニアチームのAI利用状況も交えご紹介していきたいと思います！

## 前提

弊社ではAI活用に積極的に取り組んでおり、昨年夏頃からのDevinの有償利用を皮切りに、ChatGPT（Proプラン）やClaude（Maxプラン）、Cursorも含めLLMサービスをハイクラスのプランで利用することができます。予算上限までであればどれを使っても問題はありません。好みのサービスを柔軟に利用することができ、同時並行で利用して比較したいとの考えから上限内で複数サービスを組み合わせて使っているメンバーもいます。

### AIチーム開発の活用状況

2026年6月現在、私は自身ではコードを書いておらずClaudeでサブエージェントを利用したチーム開発を行っています。例えば、一開発でもフロントエンド、バックエンドと分割し複数エージェントで分担させたり、複数開発を複数セッションで並行で対応したりしています。自分は1チーム1セッションになるようにしています。

### AIチームの体制

AIチームは、人間のチーム開発と同じく「役割分担」で動かしています。私が直接やりとりするのは **親エージェント（オーケストレーター）** で、親が開発をコンポーネント単位に分解し、 **子エージェント（サブエージェント）** に割り当てます。

* 親エージェント：タスク構造の認識、タスク分割、サブエージェントの起動・指揮、進捗の集約、課題事項があった場合のアラート
* 子エージェント（フロント担当）：画面・コンポーネントの実装
* 子エージェント（バック担当）：API・DB・バッチの実装
* 子エージェント（レビュー担当）：他エージェントが書いたコードのクロスレビュー

ポイントは、子エージェント同士は直接会話するのではなく、後述の工夫②のタスク管理マークダウンをいわば「掲示板」として間接的に状況共有する点です。各エージェントは自分の担当ファイルスコープの中だけで作業し、共有リソースを変更するときだけ掲示板で宣言します。これにより、単一端末で複数エージェントが並行稼働してもファイル競合や手戻りを最小化できます。  
レビューについては、実装した子エージェントとは別の子エージェントにレビューさせる「エージェント間レビュー」を一次チェックとし、その後に私自身が最終レビューを行う二段構えにしています。

では実際にどのようにAIチームとして開発しているか、工夫したポイントについて説明していきます。

## 工夫ポイント① git worktreeの活用

AIチーム開発では触らない日はないほど**git worktree**の利用が当たり前になっています。なぜかというと、AIチーム開発では複数のエージェントが同じソースコードを同じタイミングで編集する可能性は大いに想定されます。ここまでは人間で行うチーム開発と同様ですが、AIチーム開発では単一端末で複数エージェントが稼働するためファイルが競合してしまいます。ではディレクトリ分けすれば良いという発想になりますが、どこのディレクトリで何をやったのかの管理、把握が難しくなるという問題が発生します。

そこで、git管理下にありながらディレクトリ丸ごと複製とも言えるgit worktreeを利用することで、ディレクトリ名を柔軟に設定できるため何をやっているのかわかるよう識別子で命名し、なおかつgitに状況の把握や変更の管理を任せることができます。

加えて、rulesによって開発開始時にworktreeを切ることやその命名規則を定め、開発終了時にはworktreeを消すことによってディレクトリを整理し綺麗に保つこともできます。命名規則は以下のようにしています。ブランチ名とworktreeディレクトリ名にタスクの番号を含めることで、どのworktreeで何の開発が行われているか一目でわかるようになっています。またブランチ名に/claudeも含めてclaudeで生成したコードであることを明示させています。

* ブランチ名: feature/claude/SCP-`<番号>`（例:feature/claude/SCP-123）
* worktreeディレクトリ名: `<リポジトリ名>`-SCP-`<番号>`（例:`<リポジトリ名>`-SCP-123）

※「SCP」は開発コードになります。

## 工夫ポイント② タスク共有ディレクトリの作成

複数エージェントによる開発でも、人間によるチーム開発と同様にタスク管理の問題があります。ここで、まず弊社の開発におけるタスクのルールについて簡単に説明しておきます。タスクは階層構造で、最上位のものは開発によって実現したいユースケース（≒解決したい課題）を示します。その配下に開発するコンポーネント単位(例えばバックエンド、フロントエンドといった区切り)でサブタスクを切りタスク管理をしています。最上位が大規模な場合は、リリース可能かを判断基準にサブタスクに分割した上でコンポーネント単位でさらにサブタスクを切ることもあります。

私の場合はタスク管理の問題について、.claude内にtasksというディレクトリを切り、さらに最上位タスクに対応するディレクトリを切ってタスク管理にまつわるマークダウンを配置しています。タスク構造をそのままAIチーム開発に適用しています。

タスク管理のマークダウンにはサブタスクやエージェントの分担、進捗やタスク間の依存関係、仕様なども記載します。このマークダウンで進捗情報を保持することによって、エージェント同士が開発の進捗を更新して互いに状況を把握し開発を進めることができます。

タスク管理マークダウンは以下のようなテンプレートがあり、開発の都度テンプレートからコピーし使います。エージェントの担当領域・担当ファイルを明示することで、ファイルの占有が分離され競合を防ぐことができます。各エージェントは作業開始時にこのファイルを読み、他エージェントの担当ファイルには触れないというルールを敷いています。

```
## アクティブエージェント
| エージェント | 担当領域 | ステータス | 担当ファイル |
|------------|---------|----------|------------|
| Agent-1 | フロントエンド | 作業中 | src/components/*, src/pages/* |
| Agent-2 | バックエンド | 作業中 | src/api/*, src/lib/* |

## 同期ルール
1. 作業開始時にこのファイルを読み、他エージェントの担当ファイルに触れない
2. ステータス変更時にこのファイルを更新する
3. 共有リソース（package.json 等）の変更は宣言してから行う
4. 作業完了時に成果サマリーを記録する
```

「共有リソースの変更ログ」「完了タスクログ」も同じファイルに時刻付きで追記していきます。これにより、複数エージェントが互いの進捗とお互いの変更を把握しながら並行で開発を進められます。

加えて、最上位タスクディレクトリにはAIとやりとりし詰めた仕様や設計を記載したドキュメント、開発時に作成されたテストデータ用クエリなどを格納することもあります。タスク管理の他、AIだけでなく人間へのノウハウ共有も兼ねて情報を残すようにしています。

tasksの運用もrulesに記載しており、チーム開発の旨の指示を出せばAIがタスク構造を認識し自ずとサブエージェントを稼働させタスクの進捗を記録しながら並列で開発を行ってくれます。

## 工夫ポイント③ .claudeの配置

3点目はAIチーム開発を行うための.claudeの配置です。弊社ではここ数ヶ月の間でシステム開発が進み、認証基盤やデータ基盤といった複数システムがリリースされました。システム = リポジトリという単位であり、システム間の連携を想定した開発も増加していたことから、リポジトリを跨ぐ開発を考慮する必要がありました。

.claudeの配置としてよく見られるのはそのリポジトリの最上位階層に配置することですが、上記の開発事情を考慮して独立に.claudeを作り、そして各リポジトリと同階層に配置するようにしました。実際のディレクトリ構成としては以下のようになります。

```
workspace/                 # 任意のフォルダ名でよい
├─ .claude/                # AIチーム開発用.claude
├─ system1/
├─ system2/
└─ system3/
```

この階層構造は配置規約としてCLAUDE.mdに記載しており、相対パスで記載しています。エージェントもこの情報を認識して作業し、worktreeも同階層で切られます。システムが増えてもこの情報も追記すればエージェントが認識できます。

## 課題

この開発手法は3月ごろから確立し、私がメインで担当しているskylaaの販売計画機能（新規に開発）、他機能の一部開発で実践してきました。販売計画機能は私のざっくりとした体感で設計からリリースまで人力では10人月前後かかるかなというところでしたが、4〜5月でほぼ私単独で初期開発を終えリリースしています。爆速開発を実現する一方でデメリットもありました。

デメリットの1つ目はレビューについてです。エージェントに開発させたコードはエージェント間でレビューさせ、そして私自身によるレビューを行っていて、ここで1日ほどかかっていました。バックエンド、フロントエンドの両方を担当していたため、合計2日以上セルフレビューに費やしていた計算になります。

開発の流れの中で、私のレビューがボトルネックになってしまうことはどうしても否めませんでした。X（旧Twitter）ではコードを見ることもほぼしていないと公言する有名開発者の方もいます。割り切ってコードを細かく把握することを諦め、仕様の通り正しく動くかの確認に注力するのか、極力エージェント間のレビューで済むものはそちらに寄せて仕様通りかなど本当に人間の確認が必要なものは確認するようにするか。どこまでをエージェントに任せ、どこからを人間が担うか、その線引きが悩ましいところです。

裏を返せば、仕様も網羅的に言語化できるのであればテストコードやE2Eテストツールに落とし込むことができ省力化して確認することも可能です。販売計画機能は、何らか数値に問題があった場合、その企業の経営計画に大きな誤算を与える可能性があります。機能の重大性からもレビューには力を入れるべきと考えており、やり方は今後も探っていきたいと思います。

2つ目は負担の増加です。販売計画機能の開発時はスケジュールもタイトであったためスピードも重視しました。弊社のこれまでの機能開発では、フロントエンドとバックエンドで担当者を分け開発していましたが、私単独で実装することとしました。

本来2人でやっていたことを1人でやることになったため前述のレビューを含めて負荷が非常に高かったです。個人の体感としては炎上案件のプロジェクトマネージャーを務めているようなスピード感や密度でした。リリース後タスク調整を依頼し、直後は負担を減らした開発をさせてもらっていました。

AIチーム開発は大きな実装効率の向上を期待できますが、近年問題になっているAI疲れを引き起こす原因にもなりえます。取り組む際にはメリハリをつけることも重要でしょう。

## 今後の展開

AIチーム開発についてご紹介しました。私個人のこれまでの開発チームリーダーやプロジェクトマネージャーの経験から、実際に開発での運用を通じてこのような形になりました。世間でもAIエージェントでチームを組成し開発を行う企業も増えてきています。徐々にプラクティスも確立され一定の型に収束していくことが推測されます。

弊社エンジニアチームでも、個々人でのAI活用は進んでいるものの、AI活用に関するノウハウの共有はできておらず個別に閉じてしまっている状態でした。現在では、AIチーム開発の仕組みを含め、AI利用のノウハウを集約、共有知化すべくAIエージェントによる開発用のリポジトリの検討が進んでいます。
