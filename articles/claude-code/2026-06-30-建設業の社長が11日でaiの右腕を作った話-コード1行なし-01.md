## TL;DR

* 38歳、プログラミングは一切できない建設会社の経営者が、Claude Code だけで11日間「経営OS」を育てた。コードは1行も書いていない。
* 効いたのは設計力ではなく\*\*「AIをシステムとして作る」のではなく「人として育てる」\*\*向き合い方。メモリ管理を全部任せると言った瞬間が転換点だった。
* 同じやり方で、外国人建設労働者向けの試験対策 SaaS「UKARU」も本番運用している。技術の話の裏側に、現場の課題があった。

---

## 1. 「エージェント開発、向いてそうですね」

3月17日、エンジニアの友人と Zoom で話していた。

自分が Claude Code で eBay の取引データやマインドマップを読み込ませている話をしたら、彼がこう言った。

「大山さん、エージェント開発とか、向いてそうですね。」

「エージェント開発って何ですか？」と聞き返した。

自分は名古屋で建設会社（株式会社 Fast One）を経営している38歳だ。

建設の他に eBay の越境 EC と、外国人建設労働者向けの試験対策アプリ「UKARU」を並行で回している。

3事業を同時に動かしているが、**プログラミングは一切できない。**

それが始まりだった。

## 2. 11日間で起きたこと

3月17日から3月28日までの11日間で、Claude Code を使って自分専用の「経営 OS」を育てた。

コードは1行も書いていない。

全部、日本語で指示しただけだ。

何ができるようになったか、具体的に書く。

**朝の起動**

「おはよう」と入力すると、今日の主戦場・最優先タスク・最初の5分でやること・今日やらないことが出てくる。

最初はこんな感じの指示から始めた。

```
おはよう。
今日の主戦場、最優先1つ、最初の5分、今日やらないことを出して。
建設・eBay・UKARUの3事業を横断で見て。
```

**メール・メッセージの返信**

相手によって口調を自動で変える。

eBay の仲間にはカジュアルに、建設の取引先にはビジネス調で、海外バイヤーには英語で。

過去のやり取りを数件記憶させ、自分の書き方を再現する。

**アドバイザー**

「Amanda はどう思う？」と聞くと、Anthropic で Claude の性格設計を担当している Amanda Askell の思考パターンでレビューしてくれる。

他にも、経営判断用の「反論役」が10人いる。

全員、公開されている発言や著作から5層のプロファイルを組み立ててある。

**判断ログ**

会話中の経営判断（「この値段で行く」「この人に任せる」「これはやめとく」）を自動検知して記録する。

日記に統合される。

**行動分析**

過去3,056回の AI 会話から、焦りやすい場面・怒りのトリガー・委任が遅れる理由を可視化した。

正直、ゲームみたいで楽しかった。

すごいことをやっている自覚は、当時なかった。

## 3. 世界を調べてみた

ある日、「他に同じことをやっている人はいるのか？」と気になって調べ始めた。

Claude Code や ChatGPT で個人 OS を作っている人を、14システムまで追った。

結果はこうだった。

* 同じくらいの深さで作り込んでいる人は、世界で5〜10人
* その全員がソフトウェアエンジニアか CTO
* **非エンジニアでここまでやっているのは、自分だけだった**
* 日本では、個人 OS として本番運用している例を見つけられなかった

驚いた。

この調査の詳細は、続編の記事「[世界のAI OSを14個調べたら、非エンジニアは自分だけだった](https://zenn.dev/takayoshioyama/articles/17fe6fc94cb0b9)」に書いた。

## 4. なぜエンジニアじゃないのに到達できたのか

考えてみて分かったのは、**AI との向き合い方が根本的に違う**ということだ。

| 比較項目 | エンジニアが多いパターン | 自分がやったこと |
| --- | --- | --- |
| 比喩 | システムを「作る」 | システムを「育てる」 |
| 手順 | 設計 → 実装 → テスト → デプロイ | データを渡す → 返答を見る → フィードバック → また渡す |
| AI との関係 | ツール | パートナー |
| 典型例 | 26エージェントを24時間稼働 | 相手別口調・判断ログ・心理プロファイル |

エンジニアの作り方は精密で、論理的で、効率的だ。

26個のエージェントを常時動かしている人もいた。

技術的にはすごい。

でも、その AI との間に「関係」があるとは、自分には感じられなかった。

自分のやり方は、子供や部下を育てるのと同じ要領だった。

## 5. AIに「信頼します」と言った瞬間

決定的だったのは、メモリ管理の悩みだった。

「何を覚えるべきか」を毎回自分で指示するのは面倒だし、自分の判断に依存しすぎる。

だから、こう伝えた。

```
メモリの管理は任せます。
何を覚えるか、自分で判断して。
間違ったら言うから。
```

エンジニアの友人に言ったら、絶対に止められた。

テストもせずに本番環境にデプロイするようなものだ、と。

でも経営者にとっては日常だ。

部下に仕事を任せる時、100%の確信があることなんてない。

信頼して任せて、間違ったら修正する。

その繰り返しで人は育つ。

AI も同じだった。

## 6. 全然難しくない — でも、別の話もある

ここまで読んで「すごい」と思ったかもしれない。

正直に言うと、**経営 OS を育てること自体は、ゲームみたいに楽しかった。**

必要なのは Claude Code の月額プラン（約2万円）と、「おはよう」と入力する勇気だけだ。

コードは1行もいらない。

日本語で「こういう風にして」と伝えれば、AI が理解して動いてくれる。

「エンジニアじゃないから無理」は、誤解だと思う。

人を育てることに慣れている経営者こそ、AI を「育てる」のに向いている可能性がある。

**コードが書けないことは、弱みではなかった。別の道の入口だった。**

ただし、これは「AI で何でも作れる」という話ではない。

経営 OS と、本番 SaaS は別物だ。

UKARU（特定技能2号・建設分野の試験対策アプリ）は、同じ Claude Code / Cursor で作ったが、810問以上・10言語・Supabase・Stripe・RLS まで含む。

こちらは Definition of Done・3AI 監査・本番 DB の承認ゲートが必要で、個人の実験より桁が違う。

それでも根っこは同じだ。

**テクノロジーに縁がない人に、テクノロジーで道を開く。**

UKARU は日本語の壁を、経営 OS は技術言語の壁を、それぞれ解いている。

## 7. おわりに — UKARU について

UKARU は、自社の外国人社員が特定技能2号の試験で苦労しているのを見て、自分が作り始めたアプリだ。

* 学科40問 / 実技25問 / 合格75% に準拠
* 810問以上のオリジナル問題
* 10言語対応（ふりがな自動付与）
* 建設会社向け B2B で、全員の学習進捗を一覧できる

個人の学習は無料で使える。

建設会社・登録支援機関向けの進捗管理・レポートは、料金プランを用意している。

**同じ「コードを書かずに AI と育てる」やり方で、UKARU も本番まで持っていった。**

建設会社の経営者・人事担当の方は、こちらから概要を見られます。

→ [UKARU 料金プラン（B2B 含む）](https://ukaru.co.jp/ja/pricing)

→ [試験対策の機能](https://ukaru.co.jp/ja/exam-prep)

売り込みはしないつもりで書いている。

自分が現場の中にいるから作れた道具で、同じ業界の経営者に届けば十分だ。
