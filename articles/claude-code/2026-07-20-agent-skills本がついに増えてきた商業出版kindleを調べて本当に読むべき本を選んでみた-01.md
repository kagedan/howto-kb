## はじめに：プロンプトを書く時代から、「仕事のやり方」をAIに渡す時代へ

生成AIを使い始めた頃、私たちはプロンプトを工夫していました。「あなたは優秀なコンサルタントです」「次の観点でレビューしてください」「このフォーマットで出力してください」。良い回答を得るために、毎回長い指示を書く。それが「AIを使いこなす」ということでした。

しかし、Claude CodeのようなAIエージェントを使っていると、だんだん別の問題が見えてきます。「毎回、同じレビュー基準を説明している」「毎回、同じ分析手順を書いている」「毎回、社内ルールや成果物のフォーマットを教えている」。AIが賢くなっても、人間が毎回仕事の進め方を説明していたら、結局こちらがAIの管理者になってしまいます。

そこで注目され始めたのが、**Agent Skills**です。

Anthropicの公式説明では、Skillsは「指示、スクリプト、リソースなどをまとめたフォルダ」で、Claudeが必要なときに動的に読み込み、特定の仕事を繰り返し実行できるようにする仕組みです。またClaude CodeのSkillsは、Claudeだけに閉じた独自機能ではなく、「Agent Skills」というオープンスタンダードに沿った仕組みとして設計されています。

簡単に言えば、**「AIに何を聞くか」から、「この仕事はこうやる、とAIに教えておく」へ。**

生成AI活用の中心が、少しずつ変わり始めています。

ところが、Agent Skillsについて学ぼうとすると、少し前までは公式ドキュメントやWeb記事を読むのが中心でした。それが2026年に入り、Kindleの個人出版が一気に増え、2026年6月には技術評論社から本格的な商業出版も登場しました。

そこで今回は、現在出ているAgent Skills関連本を商業出版・Kindleまで含めて調べ、「結局どれを読めばいいのか」を整理してみます。

※本記事にはアフィリエイトリンクを含む場合があります。

## 1.【まず1冊ならこれ】『Claude Codeで学ぶ Agent Skills入門』

📘 **『Claude Codeで学ぶ Agent Skills入門』佐藤亮 著**

2026年6月8日に技術評論社から発売された、Agent Skillsを体系的に学ぶための専門書です。現時点で「Agent Skillsをちゃんと理解したい」と考えたとき、最初の基準になる一冊だと思います。

Agent Skillsとは何かという基本から、既存Skillsの使い方、自分でSkillを設計・作成する考え方、再利用や共有まで、一連の流れを整理して学べるのが強みです。Kindleの個人出版にもAgent Skills本はかなり増えてきましたが、「まず正確な全体像をつかみたい」という人には、出版社の編集を経たこの本から入るのが無難です。

### 💡 この本が向いている人

Claude Codeは使っているが、Skillsはまだよく分からない。SKILL.mdを見たことはあるけれど、何を書けばいいか分からない。MCP、サブエージェント、CLAUDE.mdとの役割分担が曖昧。そんな人の「最初の一冊」に向いています。

特に重要なのは、Skillsを単なる「長いプロンプトの保存場所」と考えないことです。たとえばコードレビューなら、「レビューして」という一文だけではなく、チェックする順番、セキュリティ観点、コーディング規約、出力形式、参考資料までを一つの再利用可能な能力としてまとめられます。

ここを理解すると、Claude Codeの使い方が一段変わります。

## 2.【Skillsを深く作り込みたい】『Agent Skills 実践ガイド』

📕 **『Agent Skills 実践ガイド: AIエージェントに専門知識を渡す技術』岡崎太 著**

こちらはKindle・ペーパーバックで展開されている個人出版系のAgent Skills専門書です。商業出版の『Claude Codeで学ぶ Agent Skills入門』が「全体像を正しく理解するための入口」だとすれば、こちらはもっと深くSkill設計へ入りたい人向け、という位置づけがよさそうです。

Skillsを作るとき、本当に難しいのはMarkdownファイルを書くことではありません。「何をSkillにするべきか」「どこまで指示として固定するか」「どの情報を必要なときだけ読み込ませるか」「チームでどう品質を維持するか」という設計です。

Agent Skillsは簡単に作れるからこそ、何でも巨大なSKILL.mdに詰め込めばよいわけではありません。

### 💡 この本が向いている人

すでにSkillsを少し作ったことがあり、「もっと体系的に設計したい」「チームで再利用したい」「自分の業務知識をAIに持たせたい」という人に向いています。

ただし、ここは重要です。個人出版の技術書は、非常に速いスピードで最新トピックを扱えるメリットがある一方、商業出版社の技術書とは編集・技術レビューの体制が異なります。そのため私は、**公式ドキュメント＋商業出版を基準にして、Kindle本を実践アイデア集として使う**読み方がよいと思います。

## 3.【一番面白かった切り口】「思考」「分析」「提案」そのものをSkillにするシリーズ

[📗 **『Agent Skills実践ガイド Vol.1 思考AI編』**](https://amzn.to/4wihTgC)  
[📗 **『Agent Skills実践ガイド Vol.2 分析AI編』**](https://amzn.to/4wPm9Uz)  
[📗 **『Agent Skills実践ガイド Vol.3 提案AI編』**](https://amzn.to/4fx6e6u)

今回、Agent Skills本を調べていて最も面白いと思ったのが、この方向です。Vol.1では「考える力」、Vol.2ではSWOT、3C、BCGマトリクス、ファネル分析などの「分析する力」、Vol.3では「提案を通す力」をSkillsとして扱っています。

ここで、Agent Skillsの見え方が大きく変わります。

Agent Skillsというと、「コードレビューSkill」「テスト生成Skill」「Git操作Skill」のようなエンジニア向け機能を想像しがちです。しかし、本質的にはコードを書くためだけの仕組みではありません。

たとえば、「市場分析するときは、最初に市場規模を確認し、次に競合を整理し、最後に自社の優位性を見る」「不具合解析では、まず現象を再現し、4M変化点を確認し、仮説を立てて切り分ける」「設計レビューでは、安全性、コスト、製造性、保守性の順に確認する」。

こうした**人間が仕事で使っている“思考の順番”そのものをSkillにできる**わけです。

私は、ここにAgent Skillsの本当の可能性があると思います。

## プロンプトエンジニアリングの次は「暗黙知のSkill化」ではないか

これまで企業の生成AI活用では、「どうやって良いプロンプトを書くか」が重要なテーマでした。しかし、優秀な人の仕事を観察すると、価値があるのはプロンプトの文章そのものではありません。

何を最初に確認するのか。どこで違和感を感じるのか。どの順番で原因を切り分けるのか。何をもって「この成果物はOK」と判断するのか。こうした仕事の型に、本当の価値があります。

例えば製造業を考えてみます。ベテラン設計者に図面を見せると、若手が気づかない問題を数分で見つけることがあります。「この公差だと量産でばらつきそうだ」「ここは組付け時に工具が入らない」「過去に似た構造で不具合が出た」。本人に「どうやって見つけたんですか？」と聞いても、「経験かな」と言われる。

これが暗黙知です。

従来の生成AI活用では、設計書をRAGに入れて「検索できるようにする」という発想が中心でした。しかしAgent Skillsでは、もう一段違うことを考えられます。

**その人が“どう考えているか”を、AIが再利用できる仕事の型として残す。**

たとえば、設計レビューSkill、FMEA Skill、設備トラブル解析Skill、品質不具合の原因分析Skill、プロジェクトリスクレビューSkill、データ分析Skill、技術調査Skill。こうしたSkillsを作れる可能性があります。

もちろん、熟練者の能力をSKILL.md一つで完全再現できるほど単純ではありません。しかし、「毎回口頭で教えていた仕事の進め方」を、AIが参照できる組織資産へ変える。これは生成AI活用としてかなり大きな変化だと思います。

## 4.【Claude Code全体の中で理解したい】『Claude Code実践レシピ100本』

📙 **『Claude Code実践レシピ100本: MCP / Skills / Hooks / サブエージェント実践ガイド』**

Agent Skillsだけを単独で学ぶのではなく、Claude Code全体の中で理解したい人には、こうしたレシピ型のKindle本も候補になります。

なぜなら、Agent Skillsだけですべてを解決するわけではないからです。ざっくり整理すると、SkillsはAIに「仕事のやり方・専門知識」を渡すもの、MCPはAIを外部データやツールにつなぐもの、Hooksは特定のタイミングで処理を強制・自動実行するもの、Subagentsは特定の役割を持つエージェントへ仕事を分担するもの、と考えると分かりやすいです。

Agent Skillsを使い始めると、「これはSkillに書くべきなのか？」「サブエージェントに分けるべきなのか？」「MCPでつなぐべきなのか？」という設計問題が必ず出てきます。

Claude Codeを本格的に使う人は、Skills単体ではなく、この全体像まで理解した方がよいと思います。

## 5.【まず雰囲気を知りたい】『マンガでわかる！Claude Code「Agent Skills」超入門』

📓 **『マンガでわかる！Claude Code「Agent Skills」超入門』**

いきなり本格的な技術書を読むほどではない。でも「Agent Skillsって結局何が便利なの？」を短時間でつかみたい。そんな人には、低価格のKindle入門書という選択肢もあります。

こうした本は、技術仕様を正確に深掘りする本というより、「Skillsを使うと何が変わるのか」「自分の仕事なら何をSkillにできそうか」というアイデアを得る入口として使うのがよさそうです。

数百円のKindle本で雰囲気をつかみ、興味を持ったら商業出版や公式ドキュメントへ進む。この読み方なら十分価値があります。

## 結局、どの本を読めばいいのか

Agent Skillsを初めて学ぶなら、まずは\*\*『Claude Codeで学ぶ Agent Skills入門』\*\*。体系的に理解する基準にしやすいです。

自分でSkillsを本格的に設計したいなら、**『Agent Skills 実践ガイド』**。公式ドキュメントや商業出版と照らし合わせながら、より深く実践したい人向けです。

エンジニア以外の仕事にも応用したいなら、**『Agent Skills実践ガイド 思考AI編・分析AI編・提案AI編』**。Skillsを「コーディング機能」ではなく「人間の仕事の型」として考えるヒントになります。

Claude Codeを本格運用したいなら、**『Claude Code実践レシピ100本』**。Skills、MCP、Hooks、サブエージェントを組み合わせて考えたい人向けです。

とりあえずAgent Skillsが何か知りたいなら、**『マンガでわかる！Claude Code「Agent Skills」超入門』**。低コストで概念をつかむ入口として使いやすいです。

## Agent Skillsは「Claude Codeの便利機能」だけでは終わらない

今回、本を調べる前は、私もAgent Skillsを「Claude Codeを便利にする新しい拡張機能」という感覚で見ていました。しかし、関連本を調べていくと、少し違う見方ができるようになりました。

重要なのは「Claude CodeのSkillsを覚える」こと自体ではなく、\*\*「自分たちの仕事を、AIが再利用できる能力としてどう定義するか」\*\*なのかもしれません。

会社には、マニュアル、標準作業書、設計標準、チェックリスト、過去トラブル集、熟練者のノウハウがあります。これまでは、それらを人間が読んで仕事をしていました。

これからは、それらをAIエージェントが必要なときに読み込み、決められた手順で仕事をする世界が来るかもしれません。

そう考えると、Agent Skillsは単なるプロンプト管理ではありません。

**企業の「仕事のOS」をAIが読める形へ変換する技術**

と見ることもできます。

## 製造業こそAgent Skillsと相性がいいかもしれない

個人的に特に面白いと思うのが、製造業です。

製造業には、設計レビュー、DRBFM、FMEA、なぜなぜ分析、設備故障の原因切り分け、品質不具合対応、工程変更管理、安全レビュー、生産準備など、膨大な「仕事の型」があります。

これらは単なる文書ではありません。「どういう順番で考えるか」「何を見落としてはいけないか」「どの条件なら上司や専門部署へエスカレーションするか」という判断の積み重ねです。

例えば設備異常が起きたとします。AIに「原因を分析して」とだけ依頼するより、「まず現象を整理する。正常時との差分を確認する。4Mの変化点を抽出する。ログとアラーム履歴を確認する。仮説を複数立て、確認方法と優先順位を提示する」という仕事の進め方をSkillとして持たせる方が、はるかに再現性があります。

ここに社内文書を検索するRAG、設備データを取得するMCP、専門領域を担当するサブエージェントを組み合わせる。

すると、**Skills＝仕事のやり方、RAG＝会社の知識、MCP＝データとツール、Agent＝実行する人**という構造が見えてきます。

これは、単なるチャットボット導入とはかなり違います。

## まとめ：次に価値を持つのは「良いプロンプト」ではなく「良い仕事の型」かもしれない

生成AIの初期は、良いプロンプトを書ける人が注目されました。しかし、毎回優れたプロンプトを書くこと自体が目的ではありません。本当に欲しいのは、誰がAIを使っても一定品質の仕事ができる状態です。

そこで重要になるのが、何を確認するのか、どの順番で考えるのか、どの情報を参照するのか、どの基準で判断するのか、どの形式で成果物を出すのか、という「仕事の型」です。

Agent Skillsは、その仕事の型をAIが再利用できる形にする、一つの方法です。

2026年に入り、ようやく日本語のAgent Skills本も増えてきました。特に2026年6月に技術評論社から『Claude Codeで学ぶ Agent Skills入門』が登場したことで、公式ドキュメントだけを追わなくても体系的に学べる環境が整い始めています。

一方で、Kindleを見るとすでに、その先の世界も始まっています。思考法をSkillにする。分析フレームワークをSkillにする。提案の方法をSkillにする。

つまり、**人間が長年かけて身につけてきた「仕事のやり方」を、AIに移植しようとする試みです。**

プロンプトエンジニアリングの次に来るのは、もっと長いプロンプトを書く技術ではないのかもしれません。自分や組織が持っている暗黙知を発見し、言語化し、再利用可能なSkillへ変える力。これから生成AIを本気で業務に入れていく企業にとって、こちらの方がはるかに重要な能力になる可能性があります。

Agent Skillsを学ぶということは、Claude Codeの新機能を一つ覚えることではありません。

**「自分の仕事は、実際にはどんな知識と判断の積み重ねでできているのか」を見直すことなのかもしれません。**
