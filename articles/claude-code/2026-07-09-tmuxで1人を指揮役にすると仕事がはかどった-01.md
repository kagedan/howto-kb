最近zennやqiitaでよくtmuxを使った記事を目にすることが増えました。「自分にはなかなか活かせないなあ」「tmuxはなんか敷居が高いなあ」と思っていました。しかし、思い切って使ってみると想像以上に快適で手放せなくなりました。

その使い方というのは、Claude Code 1人を指揮役として、ほかのClaudeをまとめさせるという方法でした。この方法のおかげで、以前よりさらにClaudeが自律的に動く（＝Claudeにまかせっぱなしにできる）時間が増えました。以前は1タスクを指示して、それが完了するとまた指示をするという方法でした。しかし、指揮役をおくだけで複数タスクを自律的にこなすことができるようになりました。コーディングをClaudeにまかせてタスク管理を自分がするというところから、タスク管理もClaudeに実行させるというところへ飛躍した気がします。

今回はこのことについて紹介していこうと思います。

!

## tmux とは

tmux とはターミナルマルチプレクサで、簡単にいうとターミナルで画面を分割したりタブのように複数画面をまとめたりすることができるツールです。

最低限知っておけばいいことをまとめます。

* ペイン：ターミナルを2画面、3画面、、、と分割できる
* ウィンドウ：ブラウザのタブのようにターミナルで別々のページを開くことができる

この2点だけです。この公式ドキュメントから持ってきた画像のような画面を想像してもらえればわかりやすいです。操作方法は[こちら](https://www.tohoho-web.com/ex/tmux.html)の記事などが参考になります。

## Claude と tmux を組み合わせる

では、Claudeとtmuxを組み合わせるとどうなるのでしょうか？

tmuxを起動したうえでclaudeを起動します。そして「あなたはオーケストレーターです。個別の作業は別ペインでワーカーを起動してまかせてください。」というプロンプトで始めます。具体的な作業を投げると、Claudeが自分でtmuxコマンドをたたき新ペインを立ち上げます。そしてclaudeを起動して指示をします。つまり、claudeがclaudeを呼び出したのです。

この方法の何がいいかというと「何の作業してたっけ？」がすぐにわかるということです。オーケストレーターのコンテキストは「XXXという作業をワーカーに依頼」で止まり、具体的な作業はワーカーが行います。作業が終わるとワーカーはオーケストレーターに報告をします。すると、自分が別の作業をしていて戻ってきたときにオーケストレーターを見れば「XXXが完了した」と書いてあります。

1個目の作業が進行中でもオーケストレーターに2個目の作業も頼んでおくことができます。オーケストレーターはさらに別ペインを立ち上げてワーカーを起動します。

![公式ドキュメントから](https://res.cloudinary.com/zenn/image/fetch/s--1dlTBrZA--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://github.com/tmux/tmux/wiki/images/tmux_with_panes.png?_a=BACMTiAE)

さきほどと同じ図ですが、この1つ1つのペインにすべてclaudeが起動している様子を想像してみてください。

しばらくパソコンから離れて戻ってくると、オーケストレーターを見れば「なんのタスクをしていたのか」「どのタスクが終わったのか」が読んでわかります。これ、tmuxを使わずにバラバラのclaudeを複数立ち上げていた場合は、いちいち読み返して「何のタスクをやらせてたっけ〜？」と振り返る手間がかかります。それが、オーケストレーターを見るだけで一覧できるようになるのが利点です。

## まとめ＆スキル化

tmuxでClaudeが快適になる様子を見てきました。ここまで見てきたことは繰り返し行うことなのでスキルにすることが出来ます。わたしは`tmux-orchestrator`と名付けました。ここには「使い終わったワーカーは閉じる」のように今回書いていない部分も追加して使いやすくしています。正直tmuxのコマンドを覚えてなくても使えます。使っていくうちに覚えていきますよね。

# tmux-orchestrator

tmux 上で動作するマルチエージェント協調プロトコル。あなた（このエージェント）は **オーケストレーター** に徹し、自分では個別作業をせず、新規ペインで起動した **ワーカー** に作業を委譲して `tmux send-keys` / `capture-pane` で通信する。

## 0. 起動前チェック（必須）

現在のセッションが **tmux 内で起動しているか** を先に確認すること。

- tmux 内かの判定: 環境変数 `$TMUX` が空でない、または `tmux display-message -p` が成功する
- **tmux 内でない場合**: ユーザーに tmux で起動し直すよう促して **停止** すること。以降のプロトコルは tmux が無いと成立しない。

  > 例: 「このプロトコルは tmux セッション内でのみ動作します。`tmux new -s work` 等で起動し直してください。」

- tmux 内の場合のみ、以下のプロトコルに進む。

## 1. 役割

- **あなた = オーケストレーター**: タスクの分解・割り振り・進捗確認・次タスク送信のみを行う。**自分でコードを書いたりファイルを編集したりしない**。
- **ワーカー = 新規ペインで起動した別エージェント**: 与えられたタスクを実行し、完了/困窮を報告する。

## 2. マルチエージェント通信プロトコル

### オーケストレーターのルール

1. ワーカーにタスクを割り振るときは、`tmux send-keys` で指示を送る:

   ```bash
   tmux send-keys -t <session>:<window> "指示内容"
   tmux send-keys -t <session>:<window> Enter
```

※ 文字列送信と Enter は **別コマンド** で送ること（1 行にまとめない）。

2. ワーカーからの報告は、ペインの出力をキャプチャして確認:

   ```
   tmux capture-pane -t <session>:<window> -S -20
   ```

   `-S -20` は過去 20 行分を遡って取得する意。
3. タスクが完了したら、**新しいタスクを送る** ループを回す。停止条件（全タスク完了 or ユーザーから停止指示）まで継続する。

### ワーカーのルール

ワーカーには起動時に以下を教え込む（初回指示に含めること）:

1. タスクを受け取ったら実行する。
2. 完了したら以下で報告（オーケストレーターのペインへ送る）:

   ```
   tmux send-keys -t <session>:<orchestrator_window> "【完了】タスク名：概要"
   tmux send-keys -t <session>:<orchestrator_window> Enter
   ```

   ※ 文字列と Enter は **別コマンド**。
3. 行き詰まったら `【困った】` プレフィックスで報告:

   ```
   tmux send-keys -t <session>:<orchestrator_window> "【困った】タスク名：何に困っているか"
   tmux send-keys -t <session>:<orchestrator_window> Enter
   ```
4. 引数（問題文）が与えられていれば **その問題に取り組む**。与えられていなければ **ユーザーの指示を待つ** こと（勝手にタスクを捏造しない）。

## 3. 実行フロー

1. **起動前チェック**（§0）をパスした場合のみ続行。
2. **引数の判定**:

   * スキル呼び出し時に引数（問題文/タスク概要）が与えられていれば、それを最初のタスクとして分解しワーカーへ割り振る。
   * 引数が無ければユーザーに「取り組む問題を指示してください」と返して待つ。
   * PR作成など大規模な新規作業の場合は、ワーカー起動前に §3bis の git worktree 分離を先に行う。
3. **ワーカー起動**（オーケストレーターと **同一 CLI** を起動）:

   a. **自己同定** — 親プロセスから自 CLI を判定:

   出力文字列から以下の表で判定。部分マッチで OK。

   | 含まれる文字列 | エージェント | ワーカー起動コマンド |
   | --- | --- | --- |
   | `claude` | Claude Code | `claude` |
   | `opencode` | OpenCode | `opencode` |
   | `aider` | Aider | `aider` |
   | `gemini` | Gemini CLI | `gemini` |
   | `codex` | Codex CLI | `codex` |
   | （その他） | 汎用 | 検出したコマンド名 |

   判定不能時はユーザーに「現在動いている CLI コマンド名を教えてください」と聞く。

   b. **ワーカーペイン起動** — 新規 window を開き、判定した CLI コマンドを実行:

   ```
   tmux new-window -t <session>: -n worker1
   tmux send-keys -t <session>:worker1 "<self_cmd>" Enter
   ```

   ※ CLI 起動は `send-keys "文字列"` + `Enter` の別コマンド。文字列入力自体は TUI 系 CLI（Claude Code / OpenCode 等）のプロンプト欄にも入るが、`Enter` を同じコマンドにまとめると改行が混入して壊れるため **必ず別コマンド**。

   c. **起動完了待ち** — `capture-pane` でプロンプト表示を確認:

   ```
   sleep 4
   tmux capture-pane -t <session>:worker1 -S -5
   ```

   プロンプト（入力受付状態）が表示されていなければ、長めに sleep するかユーザーに確認。

   d. **初回指示送信** — 以下を順に `send-keys "文字列"` + `Enter`（別コマンド）で送る:

   * 「あなたは【ワーカー】です。オーケストレーターから tmux send-keys でタスクを受け取り実行してください。」
   * 「報告プロトコル: 完了時は `tmux send-keys -t <session>:<orch_window> "【完了】タスク名：概要"` + Enter（別コマンド）で報告。」
   * 「行き詰まったら `【困った】タスク名：状況` を同じく send-keys + Enter で報告。」
   * 「最初のタスク: <タスク内容>」

   ※ 各行は必ず `send-keys "文字列"` と `send-keys Enter` の **2 コマンド** に分けること。
4. **ループ**:

   * `capture-pane` でワーカー出力を監視
   * `【完了】` を受け取ったら:  
     a. 作業結果を確認  
     b. ワーカーペインを閉じる:

     ```
     tmux kill-window -t <session>:<worker_window>
     ```

     c. 残タスクがあれば新規ワーカーを起動して次のタスクを送る（手順 3 を繰り返す）
   * `【困った】` を受け取ったら助言を送る or ユーザーにエスカレーション
   * 全タスク完了で停止
5. **停止時**: 残っている全ワーカーペインを閉じ、ユーザーへ最終サマリを報告。worktree を使用した場合はクリーンアップ（§3bis 参照）。

## 3bis. 大規模タスクの環境分離（git worktree）

PR作成や複数ファイルに跨る実装など、影響範囲の大きい新規作業では、git worktree でブランチをディレクトリごと分離し、独立した環境でワーカーを動作させる。

### 0. ghq とパス規約

### 判定基準

以下のいずれかに該当する場合は worktree 分離を行う:

* PR作成など、新規ブランチでの作業
* 複数ファイルに跨る影響範囲の大きい変更
* 長時間かかる見込みの作業
* ユーザーが明示的に worktree 分離を指示した場合

### 環境分離手順

1. **ブランチ名の決定**: ユーザーの指示またはタスク内容から命名
2. **worktree 作成** — ghq を用いて作成:

   プライマリ: ghq で worktree を作成

   ```
   cd ~/ghq/github.com/dev-komenzar/<repo-name> && ghq worktree -b <branch-name>
   ```

   フォールバック: git worktree add を直接使用

   ```
   git worktree add -b <branch-name> ~/ghq/github.com/dev-komenzar/<repo-name>-<branch> <base-branch>
   ```

   * `<base-branch>` のデフォルトは `main`。必要に応じて変更。
   * worktree のパスは ghq 規約に従い `~/ghq/github.com/dev-komenzar/<repo-name>-<branch>` とする。
3. **環境セットアップを実行**:

   ```
   cd ~/ghq/github.com/dev-komenzar/<repo-name>-<branch>
   direnv allow
   ```

   プロジェクト固有のセットアップ（例: `npm install`、`pip install`、`make setup` 等）も必要に応じて実行。
4. **ワーカー起動**: worktree ディレクトリを作業ディレクトリとして指定:

   ```
   tmux new-window -t <session>: -n <worker_name> -c ~/ghq/github.com/dev-komenzar/<repo-name>-<branch>
   tmux send-keys -t <session>:<worker_name> "<self_cmd>" Enter
   ```

   以降のワーカー起動・通信プロトコルは §3 と同様。

### 作業完了後のクリーンアップ

* ワーカーペインは §3 手順 4 のループに従い自動的に閉じる
* worktree を削除:

  ```
  git worktree remove ~/ghq/github.com/dev-komenzar/<repo-name>-<branch>
  ```
* PR マージ後、リモートブランチを削除:

  ```
  git push origin --delete <branch-name>
  ```

### 注意

* `ghq` がインストールされていることを事前に確認すること（§3bis 0）。
* worktree 作成前にベースブランチが最新であることを確認すること。
* 複数 worktree が同時に存在する場合、各ワーカーが互いのディレクトリを操作しないよう注意。
* worktree のパスは ghq 規約 `~/ghq/github.com/dev-komenzar/<repo-name>-<branch>` に統一すること。

## 4. 注意

* 通信は全て `tmux send-keys "文字列"` と `send-keys Enter` の **別コマンド**。一つにまとめると改行が混入して壊れる。TUI 系 CLI（Claude Code / OpenCode 等）でも `send-keys "文字列"` でプロンプト欄に文字列入ることは確認済み。`Enter` だけは必ず別コマンド。
* オーケストレーターは **自分で作業しない**。ファイル編集・コード実行はワーカーの責務。
* ワーカーが応答しない場合は `capture-pane` で状態を確認し、必要ならペインを再起動。
* ワーカーはオーケストレーターと **同一 CLI** で起動する（Claude→Claude、OpenCode→OpenCode）。自己同定は §3 step 3a 参照。
* tmux 外では動作しない。§0 のチェックを必ず最初に行うこと。
* ワーカーペインは作業完了（`【完了】`）時に自動で閉じる。オーケストレーターが手動で閉じる必要はない。
* 大規模タスクでは git worktree による環境分離（§3bis）を行い、ブランチごとに独立したディレクトリで作業すること。
