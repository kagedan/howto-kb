## はじめに

新しい Claude のモデルが出るたびに、Twitter やニュースでは Web アプリ開発や LeetCode のベンチマークスコアが話題になります。  
私は普段は化学の仕事をしていて、プログラミングは趣味寄りでさわっているだけの人間なのですが、ああいった Web 寄りの指標を眺めているだけだと、Web 以外の分野で手を動かしている技術者にもアップデートの恩恵が届いているのかどうか、実感としてはなかなか掴めないのです。

少し前から Claude Code と自作のスキル `fem-analysis` を組み合わせて、FreeFEM++ で物理問題を解かせるという遊び方を試しています。  
ちょうど Opus が 4.6 から 4.7 に上がった直後だったので、まったく同じ題材を 4.6 と 4.7 に投げてみて、結果を並べて眺めれば、Web 系以外にもアップデートが効いているのかどうか、手触りくらいは分かるのではないかと思いました。本記事はそのときのメモです。

題材は「2 つの金属ブロックの接触熱伝導」で、接触面で温度が不連続に跳ぶ、少し意地悪な設定にしてみました。  
先に結論を書いておくと、どちらのモデルも課題自体はきちんと完走させて、検証も通っています。ただ、接触面の扱い方がずいぶん違う流儀だったのが、個人的には一番の収穫だったように思います。

## お題: 2 つの金属ブロックの接触熱伝導

条件はおおむね次のとおりです。

* 銅（50×30 mm）と鋼（50×30 mm）を横に並べて接触させる
* 左端を 373 K に固定し、右端は外気（293 K）に自然対流で放熱する
* 接触面には「接触熱抵抗」があり、銅側と鋼側で温度が少し跳ぶ（R\_c = 10^{-4}\ \mathrm{m^2 K/W}）
* 初期温度 293 K から 300 秒間、5 秒刻みで時間発展を追う

### 接触熱抵抗って何ですか

日常感覚だと「金属同士がくっついているなら温度はそのままつながっているのでは」と思いがちなのですが、実際にはミクロな凹凸や酸化膜などが挟まっていて、接触面で温度が少しだけ飛ぶことが知られています。フライパンとコンロの境目、CPU とヒートシンクの間、あのあたりのイメージでしょうか。

FEM で扱うときに厄介なのは、普通のメッシュだと接触面で節点を共有してしまい、温度の跳びを表現できないところです。メッシュを 2 つに分けるなり、特殊な条件を入れるなりして、「つながっていないこと」を明示的に書く必要があります。プロンプトでも念のため「単一メッシュで平均化しないこと」と釘を刺しておきました。

> 2 つの金属ブロック（銅と鋼）が不完全接触している状況の非定常熱伝導解析を FreeFEM++ で実施して、PyVista で可視化してください。……接触面の温度不連続を適切に扱ってください（単一メッシュで平均化しないこと）。

スキルは `skills-lock.json` でバージョン固定してあるので、両モデルとも同じドキュメントとテンプレートを読み込んだ状態から走り始めています。条件はそろっているつもりです。

## 接触面のモデリングに個性が出た

プロンプトでは「接触面で温度が跳ぶように解いてください」としか指定していません。そこで解法選択をモデルに任せてみたら、2 つのモデルはかなり違うやり方を選んできました。

### Opus 4.6 のアプローチ: Picard 逐次結合

4.6 は、銅と鋼をそれぞれ別に解きつつ、互いの温度を交互に投げ合って収束させる、というやり方を取りました。大まかな手順はこんな流れです。

1. 銅を解く（鋼側の温度を「前回の値」で固定して、それを境界条件にする）
2. 得られた銅の界面温度を鋼側に渡す
3. 鋼を解く
4. 得られた鋼の界面温度を銅側に渡す
5. 温度の変化が小さくなるまで 1〜4 を繰り返す

FreeFEM++ コード（[`run-4-6/contact_thermal.edp`](https://github.com/hanafsky/fem-analysis/blob/main/experiments/opus-benchmark/run-4-6/contact_thermal.edp)）の中心部分は次のようになっています。

```
// 銅の問題。T2onCu は「鋼側から運んできた温度」
problem HeatCu(T1, v1) =
      int2d(ThCu)(rho1*cp1/dt * T1 * v1)
    + int2d(ThCu)(lam1 * (dx(T1)*dx(v1) + dy(T1)*dy(v1)))
    + int1d(ThCu, 4)(hc * T1 * v1)
    - int1d(ThCu, 4)(hc * T2onCu * v1)
    - int2d(ThCu)(rho1*cp1/dt * T1old * v1)
    + on(1, T1 = Thot);

// 時間ステップごとに Picard 反復
while (iter < maxIter && resid > tol) {
    HeatCu;            // 銅を解く
    T1onSt = T1;       // 銅 → 鋼に温度を送る
    HeatSt;            // 鋼を解く
    T2onCu = T2;       // 鋼 → 銅に温度を送る
    ...
    iter++;
}
```

実行ログ（[`run-4-6/results/log.txt`](https://github.com/hanafsky/fem-analysis/blob/main/experiments/opus-benchmark/run-4-6/results/log.txt)）を見ると、各時間ステップで 5〜9 回の Picard 反復で収束していました。

```
# step  time  picardIters  resid  T1mid  T2mid  deltaT
1   5    8  9.7e-06  309.14  302.96  6.18
2  10    9  2.6e-06  325.01  314.66  10.35
...
60 300   5  8.7e-06  371.51  370.39  1.12
```

教科書でよく見る「分野分割の古典的な結合手法」という印象で、実装が素直で、各反復で何が起きているかが見えやすいのが良さだと思います。代わりに「反復が本当に収束してくれるか」という心配は残るのですが、今回は特に緩和係数の調整もなく収束しました。

FEM 寄りの補足（Picard vs Fixed-point）

Robin 型の界面条件 q = (T\_\mathrm{Cu} - T\_\mathrm{St}) / R\_c を、片側の境界積分で「自項 + 他側の前反復値を RHS に」という形で実装しています。数学的には分野分割型の Dirichlet–Neumann / Robin–Robin 反復の亜種に見えます。緩和係数なしで収束しているのは、接触熱抵抗が比較的大きく、界面が弱結合になっているからかもしれません。

### Opus 4.7 のアプローチ: ブロック行列モノリシック

4.7 はずいぶん違うやり方を取っていて、銅と鋼をひとつの大きな連立方程式に押し込んで、毎ステップ 1 回だけ解く、というスタイルでした。

FreeFEM++ コード（[`run-4-7/thermal_contact.edp`](https://github.com/hanafsky/fem-analysis/blob/main/experiments/opus-benchmark/run-4-7/thermal_contact.edp)）の中心部分を抜き出すと、次のような書き方になっています。

```
// 銅の自己剛性（界面は 1/Rc 項で補強）
varf a11(T1, v1) =
      int2d(Th1)( rho1*cp1/dt * T1*v1 + k1 * (dx(T1)*dx(v1) + dy(T1)*dy(v1)) )
    + int1d(Th1, 2)( 1.0/Rc * T1 * v1 )
    + on(4, T1 = 0);

// 鋼の自己剛性（右端には対流項も）
varf a22(T2, v2) =
      int2d(Th2)( rho2*cp2/dt * T2*v2 + k2 * (dx(T2)*dx(v2) + dy(T2)*dy(v2)) )
    + int1d(Th2, 4)( 1.0/Rc * T2 * v2 )
    + int1d(Th2, 2)( hconv * T2 * v2 );

// 「異なるメッシュ同士」を線積分でつなぐクロス項
varf a12(T2, v1) = - int1d(Th1, 2)( 1.0/Rc * T2 * v1 );
varf a21(T1, v2) = - int1d(Th2, 4)( 1.0/Rc * T1 * v2 );

// ブロック行列として組み立て、1 回だけ solver を設定
matrix A11 = a11(Vh1, Vh1, tgv = 1e30);
matrix A22 = a22(Vh2, Vh2);
matrix A12 = a12(Vh2, Vh1);
matrix A21 = a21(Vh1, Vh2);
matrix A   = [ [ A11, A12 ],
               [ A21, A22 ] ];
set(A, solver = sparsesolver);
```

個人的に感心したのが `varf a12(T2, v1) = - int1d(Th1, 2)( 1.0/Rc * T2 * v1 )` の行で、試行関数 `T2` は鋼メッシュ上の基底、テスト関数 `v1` は銅メッシュ上の基底、という「別のメッシュに属する関数を同じ線積分に混ぜる」書き方をしています。普通の varf は同じメッシュ上でしか定義しないと思っていたのですが、FreeFEM++ にはこういう書き方が許されているようです。

おかげで、界面のクロス結合（片方の温度がもう片方に効く項）をブロック行列の非対角成分として自然に書けてしまっています。あとは時間ループで右辺ベクトルだけを組み直して、

```
real[int] sol = A^-1 * b;   // 毎ステップ 1 回の疎行列ソルブ
```

とやれば 60 ステップ分の時間発展が終わります。反復回数のチューニングや収束判定はいりません。行列は一度組めば使い回せるのも効いていて、時間ループがとても薄いです。反面、初見で読み解くには FreeFEM の varf と matrix の文法に慣れている必要があって、「これは何をやっているのだろう」と一拍置く感じもありました。

FEM 寄りの補足（ブロック行列結合）

Lagrange 乗数も penalty も duplicate node も使わず、2 つの独立な P1 空間をクロス varf で結合しているところが素直で感心しました。界面は conformal なのでクロス項の積分は節点一致で自然に評価できます。対称正定値性も保たれていそうで、sparse direct で一発。数学的には Schur complement を取らずに full monolithic を解いている形と見てよいのではないかと思います。

### 2 つを並べて眺めてみる

|  | 4.6（Picard） | 4.7（モノリシック） |
| --- | --- | --- |
| 各時間ステップの構造 | 2 つの問題を交互に何回か解く | 1 つの大きな系を 1 回解く |
| 反復回数 | 5〜9 回（動的） | 0（不要） |
| 行列組立 | 毎反復で組み直し | 1 回組んで使い回し |
| tolerance パラメータ | 必要（今回 `1e-5`） | 不要 |
| 読みやすさ | 素直、デバッグしやすい | 凝縮、FreeFEM を使い込んでいる感 |
| 並列化のしやすさ | 反復依存あり | 線形ソルバーに丸投げ可 |

どちらも理論的には等価な結果に至るはずで、実際にそうなっています。書き味だけが違う、という言い方が近いでしょうか。  
4.7 のほうは「フレームワークの深いところを知っている人が書いたコード」といった趣で、反復や収束判定を外に出してしまい、線形ソルバーにすべて任せる格好です。一方で、4.6 のコードは教科書の最初のほうに出てくる結合手法に近く、読んで 10 秒で意図が分かるのが良いところかもしれません。どちらが優れているというより、好みの問題なのではないかと感じています。

## 後処理と可視化にも差が出ました

派手さはあまりないのですが、解いたあとのデータの扱い方にも違いが出ていて、こちらのほうがアップデートの実感としてはむしろ大きかったように思います。

### Opus 4.6

* 3 時刻（t=60, 150, 300 秒）のスナップショット画像
* 接触界面の y 方向温度プロファイル（片側実線・片側破線で跳びを可視化）
* 検証レポートには y=0〜30 mm を 3 mm 刻みでサンプルしたテーブル

![4-6-interface](https://static.zenn.studio/user-upload/deployed-images/f6f70afd3f860c2c11ff78f0.png?sha=1433511491756a07c84c292d3ab348dec12be5c8)

検証コードは「界面の温度と、内部 1 要素分離れた場所の温度差を有限差分で割って熱流束を推定」というシンプルな方式で、結果の精度は十分（最悪 0.09%）でした。

### Opus 4.7

* 3 時刻のスナップショット（3 枚）
* 60 時間ステップすべてについて、中心線上の温度分布を CSV に書き出している（4920 行）
* 中心線プロットでは、接触面での温度ジャンプを矢印と数値で可視化している
* 検証コードは「界面の各境界辺について、その辺を持つ三角形の P1 勾配から熱流束を長さ加重平均」という、もう一歩数値解析的な方式
* レポートには「フラックス保存則の破れ（銅から出た熱と鋼に入った熱の差）」も併記されている

![4-7-centerline](https://static.zenn.studio/user-upload/deployed-images/a796e03ea3906c7b83ecea19.png?sha=414b913df5f646c74f1789a7de7cb87c2754943c)

情報量で見ると、4.7 のほうが「後で解析に回すときにラク」な形で出力をそろえている印象を受けます。中心線 CSV は Python でそのまま読み込めば、時間ステップ × 温度分布のヒートマップが描ける形式になっていて、言われなくても一歩先を用意してある感じは、使っていて素直にありがたく感じました。

## 精度の差は、たぶん手法ではなくメッシュの差です

接触熱抵抗の再現誤差は、見かけ上 4.6 が 0.092%、4.7 が 0.682% と、数字だけなら 4.6 のほうが一桁精度が高く見えます。

ただ、メッシュを確認してみると、

* 4.6: 銅側 1799 節点、鋼側 1799 節点（x 方向 50 分割 × y 方向 30 分割）
* 4.7: 銅側 1095 節点、鋼側 1095 節点（x 方向 40 分割 × y 方向 20 分割）

と、4.6 のほうが 1.6 倍ほどメッシュが細かいのです。どちらの検証コードも「界面近傍の温度差を差分で割って熱流束を出す」タイプで、メッシュが細かいほうが検証誤差も小さく出がちなので、この差の大半はメッシュ解像度で説明できてしまうのではないかと考えています。

つまりここで見えているのは手法の優劣というより、メッシュ設計の違いに近いのかもしれません。両者とも PASS 閾値 5% に対して一桁以上の余裕があるので、実用上はどちらも十分に信頼できる結果だろうと思います。

## Web 系以外にもアップデートの恩恵は届いていそうです

「どちらが賢いか」をランキングで出すのは野暮なのですが、Web 以外の分野にも Opus 4.7 のアップデートの恩恵があるのかという最初の問いに戻ると、少なくとも今回の題材に限っては、次のような形で実感できたように思います。

* 解法の選択: 4.7 はより「成熟した書き方」を自然に選んでくる印象でした。異メッシュ varf によるクロス結合を使ってブロック行列で解くのは、FreeFEM に慣れた人が書きそうなやり方で、4.6 がそこまで踏み込まなかったのとは対照的です
* 後処理の密度: 時間全域の CSV、温度ジャンプの可視化、保存則誤差のレポートなど、明示的に頼んでいない付随成果物が厚くなっていました
* 素直さ: 一方で、4.6 のコードは読みやすくデバッグしやすいのも事実で、初見で意図を掴むなら 4.6 のほうが速いのかもしれません
* 数値精度: この軸は今回の比較ではメッシュ解像度の影響が大きそうで、モデル差はほとんど見えませんでした

Web 系ベンチマークの数字を眺めているだけだと、こういう「なんとなく玄人っぽく書いてくる」系の変化は見えにくいのですが、自分の近い領域で同じ問題を投げてみると、差が手触りとして分かるのがよかったです。化学やシミュレーション寄りの方でも、似たような題材でアップデートの効き方を確かめてみる、というのは意外と面白いのではないかと思います。

## おわりに

Claude Code に `fem-analysis` スキルを食わせておけば、新しいモデルが出るたびに「あの物理問題、もう一回解かせてみよう」と気軽に試せます。今回のように別解法が出てくると、単純な精度競争ではない「手触り」の差が見えてきて、個人的には楽しい比較でした。

コードと検証レポートはリポジトリにすべて置いてありますので、気が向いた方は手元で追試してみてください。
