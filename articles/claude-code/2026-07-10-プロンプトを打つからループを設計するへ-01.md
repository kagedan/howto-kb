## はじめに

2026年6月30日にAnthropicより公式ガイド[Getting started with loops(ループ入門)](https://claude.com/blog/getting-started-with-loops)の公開がありました。

最近は「Loop Engineering（ループエンジニアリング）」という呼び名が広がりつつあります。  
「エージェントに委ねるためのゴール・制約・完了条件・検証方法を明示する役割へ変わっていく」と捉えております。

ループエンジニアリングについて記事にしました。

## ループエンジニアリングとは何か

> ループとは、AIエージェントが 作業 → 検証 → 修正 → 再実行 を、停止条件に達するまで繰り返す仕組み。

プロンプトが不要になるわけではないです。プロンプトの**役割が変わります**。

1回だけの作業指示を書くのではなく、ループの材料となる「目標」「検証方法」「停止条件」を書くことになります。

> > ループ設計　= [これまでの質の良いプロンプト] ＋ [目標、検証、状態、制約、停止条件]

「AIに何を入力するか」ではなく「**AIに入力が届き続ける仕組みをどう設計するか**」が問われる、という視点の転換です。

プロンプトエンジニアリング→コンテキストエンジニアリングと来た流れの、次の段階と捉えるとわかりやすいと思います。

## 「それは、ただの定期実行では？」

ループの多くはcronのような定期実行のような振る舞いをします。  
ただし、単なる定期実行とループ設計には決定的な違いがあります。

> **状態と停止条件の設計が伴ってはじめて「ループ」になる**。

| 観点 | ただの定期実行 | ループ設計 |
| --- | --- | --- |
| 目標 | 毎回同じ処理を実行 | 検証可能なゴールに向かって前進 |
| **状態** | 実行ごとにリセット | **前回の結果・失敗理由を次の実行へ引き継ぐ** |
| **停止** | 止めるまで無限に回る | **停止条件（達成・上限・予算）で自ら止まる** |
| 人間 | 関与しない | 取り消せない判断だけ人間に戻す |

前回はどこまで進んだか、なぜ止まったかを引き継がずに回る仕組みではない。  
同じ作業を繰り返すだけの手続き(スクリプト)と異なる点です。

## Claude Code公式からの4分類

ループを「何をエージェントに引き渡すか」で4種類に整理しています。

| ループ型 | 起動 | 停止条件 | 向いているタスク |
| --- | --- | --- | --- |
| ターン型 | ユーザーのプロンプト | Claudeが完了と判断 | 単発のタスク全般 |
| ゴール型（`/goal`） | 手動プロンプト | 目標達成 or 試行回数の上限 | 検証可能な終了基準がある複雑なタスク |
| 時間型（`/loop` / `/schedule`） | 指定した時間間隔 | キャンセル or 完了イベント（PRマージ等） | 定期作業、外部システムの変化待ち |
| プロアクティブ型 | スケジュールやイベント（人間の介入なし） | タスクごとに完結、ルーチン自体は継続 | バグ対応・依存更新などの定義済みタスク群 |

![Claude Code公式による4つのループ分類 — ターン型・ゴール型・時間型・プロアクティブ型の起動・停止条件・向いているタスクをまとめた図解](https://static.zenn.studio/user-upload/deployed-images/ab21d658800b9bb192ce8d06.png?sha=e68c441390b72356dd9db2fd2ffd2a33f37a479e)  
*4つのループ型の全体像 — 下段は各型のポイント*

### ターン型 — 普段のプロンプトも実は小さなループ

普段の「プロンプトを送って結果を待つ」やり取りも、内部では 編集→テスト実行→修正 のループが回っています。  
検証ステップをスキルに書いて自動化しておくと、人間が差し戻すターン数そのものを減らせる、というのが公式の回答です。

### ゴール型 — 「何を証拠に完了とするか」を渡す

`/goal`は、検証可能な停止条件に向かって、作業→確認→続行/停止の判断を繰り返す使い方。

```
/goal Lighthouseのパフォーマンススコア90以上を達成する。試行は5回まで。
```

**定量的な判定基準**の効果です。  
「良い感じにして」ではループは止まりどころを見失います。テストの合格本数、スコアのしきい値のように、機械的に判定できる基準を渡すことがポイントとなります。。

### 時間型 — 外部の変化を待つ仕事を手放す

`/loop`はローカルで、`/schedule`はクラウドで定期実行します。  
レビュー待ち・CI待ちのように「人間が定期的に見に行っていた仕事」がループに置き換わる領域です。

```
/loop 5m PRの状態を確認し、レビューコメントに対応し、CIが落ちていたら修正する
```

### プロアクティブ型 — 人間がループの中にいない

スケジュール＋ゴール型＋スキル＋自動モードを組み合わせ、人間の介入なしで動く形です。  
「プロンプトを打つ」という行為自体がなくなり、人間はループの設計やメンテナンスを行います。

## ループを構成する6つの部品

では、外側の仕組みは何で組み立てるのか。

| 部品 | 役割 | Claude Codeでの対応 |
| --- | --- | --- |
| Automations | いつ起動するかを決める | `/schedule`（Routine）、`/loop` |
| Worktrees | 並列作業の衝突を防ぐ分離 | git worktree |
| Skills | 仕事ごとの手順書 | SKILL.md、スラッシュコマンド |
| Plugins / Connectors | 外部ツールへの読み書き | MCPサーバー |
| Sub-agents | 独立した目でのレビュー | サブエージェント、`/code-review` |
| Memory | 前回の実行結果の引き継ぎ | メモリファイル、issue、Markdown |

![ループを構成する6つの部品 — Automations・Worktrees・Skills・Plugins/Connectors・Sub-agents・MemoryとClaude Codeでの対応をまとめた図解](https://static.zenn.studio/user-upload/deployed-images/6e7b0984c71a36f9aae51377.png?sha=a35dc3c6617d3db53478ad1e6edb3744fb1f288d)  
*6部品とClaude Codeの機能の対応*

Memoryは軽視されがちです。「前回何を試したか、どこで失敗したか」が会話の中にしかないと、次の実行で消えます。  
**会話の外（ファイル・issue・スプレッドシート）に状態を残す**ことが、ループを「ただの定期実行」から分ける実装上の分岐点になります。

## ループ設計の勘所

公式ガイドから、実際に設計するときに効く点を4つに絞って挙げます。

### 1. スキルを活用する — 検証手順ごとひとかたまりに

同じ手順や手続きはスキル（SKILL.md）としてひとかたまりに定義し、自走をアシストします。

ポイントは、**検証を人間レビュアーと同じ手順で書く**こと。開発サーバーを起動する、UIを操作して確認する、ブラウザコンソールを見る、Lighthouseを回す——人間がレビューでやっていた確認手順を書き下ろせば、ループの各周回で同じ検証が自動で走ります。

### 2. 多角的にレビューする — 自己レビューには限界がある

1つのエージェントが自分の作業を自分で評価すると、レビューは甘くなります。別のエージェントに新鮮な目でレビューさせる（`/code-review`など）ことで見落としを減らせます。

### 3. 停止条件は複数の層で用意する

安定・安全にループを回すために、停止条件は1つに頼らず拡充します。目標達成だけでなく、試行回数の上限、予算、完了イベント（PRマージ）を組み合わせます。「達成できなかったときにどう止まるか」まで書いてはじめて、安心して回しっぱなしにできます。

### 4. 空論で戦わず、小さく検証を重ねる

いきなり大きなループを組まず、まず小さく試します。トークンも設計で節約できます。

* 小さなタスクにループや複数エージェントは不要
* 監視対象の変化頻度に実行間隔を合わせる
* 推論で解くより、スクリプトを書かせて実行する
* 動的なワークフローは、まず小さなスライスで試す

使用状況は`/usage`や`/workflows`で追跡できます。

> **「自分がどうやって確認しているか」を言語化できない作業は、ループに乗せられない**。

検証手順を書き下ろす力が、プロンプトを書く力に代わる中心スキルになりそうです。

## ループを取り入れる場所の検討

**ボトルネックになっているタスクから始める**。

そして次の3つを問います。

1. 検証チェックを文章で書けるか
2. 終了条件は明確か
3. 作業はスケジュール駆動にできるか

答えに応じてループ型を選び、まず小さく回します。実行結果を見て「どこで止まったか」「やり過ぎていないか」を観察し、目標・検証・停止条件を調整していく——ループの設計自体もまた、ループで改善していくわけです。

私の場合、最初の候補は「PRのレビュー対応とCI修正の待ち時間」と「記事公開後のリンク切れ・表記チェック」あたりになりそうです。どちらも検証が書き下ろせて、終了条件が明確で、時間駆動にできます。

プロンプトエンジニアリングが「1回の指示の質」を磨く技術だったとすれば、ループエンジニアリングは「指示が届き続ける仕組み」を設計する技術です。  
どちらもAIを使い業務を行う上では大切なスキルと思います。
