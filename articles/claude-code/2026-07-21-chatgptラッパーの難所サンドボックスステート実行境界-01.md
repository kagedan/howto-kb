以前、「ChatGPTラッパー」という言葉は、しばしば肯定的ではない意味で使われていました。

`LLM API`を呼び出し、少しRAGを加え、チャット画面を載せる。薄く、代替しやすく、モデルに新機能が追加されれば消えていく。そんなプロダクトを指す言葉でした。

しかし今、ChatGPTのようなプロダクトを作る難しさは、モデルの呼び出しそのものにはありません。

コードやツールを安全に実行するsandbox、作業状態を保持するストレージ、ネットワーク制御、ユーザー確認、artifactの生成と表示。問題の中心は、**ランタイム**へ移っています。

モデル自体は、APIや既存サービスとして利用できます。しかし、その周囲の仕組みは自分で設計しなければなりません。

わずか1年で、問いは「ただのラッパーか」から「sandboxをどこで動かすか」へ変わりました。

もちろん、APIの上にシンプルなUIを載せる構成自体を否定するものではありません。この記事で扱うのは、コード実行や状態保持を伴うエージェント型のプロダクトです。

この記事では、その変化を整理します。

## これまでラッパーとしてイメージされたもの

以前の構成は単純でした。

```
prompt → LLM API → (RAG) → response
        + file parsing
```

主な工夫は、プロンプト、検索インデックス、少量のグルーコードでした。小規模な試作なら、短期間で作れました。

難しい部分はモデルが担い、アプリ側の実装は比較的小さかったからです。

以前、[トークンへの支出が資産になるのは、その周辺の仕事を設計し直したときだけだ](https://zenn.dev/gyu07/articles/70e239a868f251)と書きました。以前、その「仕事」の大半はプロンプトでした。

今は大きく変化しつつあります。

## プロダクトとエンジンを分けて考える

ここでは、2つの層を分けて考えます。

1つ目は、ChatGPTのData Analysis、Canvas、Agentモードや、Claudeのartifactsのような**コンシューマー向けプロダクト**です。

これらは、プラットフォームが管理するsandboxを備えています。ユーザーは独自の実行環境を持ち込めませんが、完成したプロダクトの形からアーキテクチャを学べます。

2つ目は、CodexのCLI、app-server、SDKや、Claude Codeのheadlessモード、Agent SDKのような**開発者向けエンジン**です。

こちらは、agent loopを自分のコードから駆動するための仕組みです。

コンシューマー向けプロダクトからは、アーキテクチャの形を学べます。開発者向けエンジンからは、実装戦略を学べます。

## ランタイム設計の中心にsandboxがある

ChatGPTは、1つのモデルというより、複数の実行環境を組み合わせたプロダクトです。

| 機能 | 実体 |
| --- | --- |
| 通常のチャット | 推論と検索 |
| Data Analysis | ファイルを扱うPython / Jupyter系sandbox |
| Canvas | コード編集とプレビュー環境 |
| Agentモード | ブラウザ、コード実行、コネクタを備えた仮想環境 |

どれも、モデルだけでは実現できません。

必要なのは、次のような仕組みです。

```
managed sandbox
+ network boundary
+ user confirmation
+ tool isolation
```

何を実行できるのか。どこへアクセスできるのか。いつユーザーに確認するのか。認証情報をどこまで渡すのか。

これらは、プロンプトだけでは解決できないランタイム設計の問題です。

かつて、モデル呼び出しはプロダクトの中心でした。今では、重いランタイムを構成する部品の1つです。

## 難しいのは、ステートをどこに置くか

コードを実行し、ファイルを編集し、ターンをまたいで作業を続けるには、状態を保持する必要があります。

エージェント型のプロダクトは、少なくとも4種類のステートを扱います。

```
1. conversation state         — 会話、計画、ツールの実行履歴
2. workspace filesystem state — ファイルやリポジトリ
3. artifact / render state    — canvasやプレビュー
4. connector / auth state     — 認証情報、権限、承認
```

設計上の論点は、何をsandbox内に置き、何をDB、オブジェクトストレージ、Gitなどへ保存するかです。

これは、以前から[考えてきた](https://zenn.dev/gyu07/articles/3b4f4e6018ddd1) `state gravity`、つまり「ステートの重力」の問題です。

頻繁に更新する作業状態は、処理の近くに置きたい。一方、永続的な記録は中央に集めたい。両者をどう分けるかが難所です。

Codexが `sqlite_home` を通じて、ジョブの状態やエクスポート結果を[SQLiteで扱っている](https://developers.openai.com/codex/config-reference)のも、同じ考え方です。

作業台はローカルに、記録は別の場所に置く。

ランタイムを選ぶことは、ステートの置き場所を選ぶことでもあります。

* **Vercel**は長時間実行を扱いやすくしましたが、ステートフルなリポジトリsandboxではありません。
* **Cloudflare Durable Objects**は会話状態の保持に向いていますが、任意コードの実行環境には向きません。
* **AWS Fargate**はタスク単位の隔離を提供しますが、周辺の仕組みは自分で作る必要があります。
* **gVisorやFirecracker**を使えば細かく制御できますが、運用の負担も大きくなります。

どれを選んでも、コールドスタート、warm pool、ネットワーク制御、永続化の設計は避けられません。

Codex cloudでは、セットアップとエージェント実行を[別のフェーズ](https://developers.openai.com/codex/agent-approvals-security)に分けています。

セットアップ中は依存関係を取得できますが、エージェント実行はデフォルトでオフラインです。シークレットも実行前に取り除かれます。

自前でランタイムを作るなら、こうした境界も自分で設計しなければなりません。

## agent loopは作らず、駆動する

CodexやClaude Codeのようなコーディングエージェントでは、多くの場合、一から再実装するより既存のエンジンを利用するほうが現実的です。

[`codex exec`](https://developers.openai.com/codex/noninteractive)を使えば、Codexを非対話で実行できます。`--json`を指定すると、イベントをJSONLとして受け取れます。

`codex app-server`を使えば、stdio、WebSocket、Unixソケット経由でCodexを操作できます。[Python SDK](https://developers.openai.com/codex/sdk)も、内部ではapp-serverをJSON-RPCで制御します。

Claude Codeにも、`claude -p`とAgent SDKがあります。

agent loop、ツール制御、パッチ適用、sandboxの強制は、すでに高度なエンジンとして提供されています。

これらを作り直すより、既存のエンジンを利用し、その周囲に境界やドメイン固有の処理を実装するほうが合理的です。

ただし、app-serverのような統合面は、安定したABIとは限りません。変化を前提に扱う必要があります。

## 自分で持つべきもの

モデルもagent loopも借りられるなら、何を自分で持つべきでしょうか。

私が重視しているのは、エージェントに渡す事実を、**どのように得たか**で分類することです。

根拠から導かれた `verified` な事実なのか。それとも、パターンから推測した `estimated` な情報なのか。出所はどこか。

[根拠とラベルのある事実](https://zenn.dev/gyu07/articles/70e239a868f251)を渡せば、エージェントの推測を減らせます。

ただし、汎用的なグラウンディングの仕組みは、いずれプラットフォームに吸収されるかもしれません。

それでも、次のようなドメイン固有の事実は残ります。

* 自社システムの制約
* 非公開のデータ
* 業務固有のルール
* 間違えると危険な条件

汎用モデルは、事前情報なしにこれらを正確に推論できません。

こうした事実を生み出すパイプラインを持てば、少なくとも今、エージェントの回答をより正確にできます。

それが永続的な堀になるかは分かりません。

作る理由は、将来も守られるからではなく、今日の答えを正しくできるからです。

## よくある議論

* **多くのアプリにsandboxは必要ない**
  + APIの上にチャット画面を載せるだけで価値を出せるプロダクトは、今も多くあります。この記事が扱うのは、コードを実行し、artifactを表示し、作業状態を保つ一部のエージェント型プロダクトです。
* **単なるbuild-vs-buyではないか**
  + 以前はモデルを買い、その上にアプリを作りました。今はモデルだけでなく、agentエンジンやsandboxも利用できます。自分で作るのは、その間にあるオーケストレーション、境界、ドメイン固有のデータ処理です。
* **いずれプラットフォームに吸収されるのではないか**
  + 多くの部分は吸収されるでしょう。マネージドなagent sandboxも、いずれマネージドPostgresのように当たり前になります。それでも、非公開データや業務固有の制約は、汎用プラットフォームには吸収されにくいと考えています。
* **借り物の上に作っているだけではないか**
  + app-serverは変化し、sandboxの制約もプラットフォームに左右されます。これは安定した要塞ではありません。スタック全体を自作するより、変化の近くにいるほうが有利だという賭けです。

## まとめ

OpenAIやAnthropicは、チャット、sandbox、artifact、レイテンシを急速に改善しています。

彼らと同じ領域で、彼らより速く開発することはできません。

取れる戦略は2つです。

1. CodexやClaude Code、既存のsandboxを使い、プラットフォームの進歩に乗る
2. 自分のドメインに固有の事実を持つ

「ただのラッパー」という言葉は、以前ほど単純ではなくなりました。

現在のラッパーには、sandbox、ネットワーク境界、ステート管理、エンジンのオーケストレーションが含まれます。

かつて、モデル呼び出しがプロダクトでした。今では、プロダクトを構成する部品の1つです。

簡単な部分は、もはや自分で作らない部分になりました。
