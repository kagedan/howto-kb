「RTK」（Rust Token Killer）は、Claude Code のトークン消費量を抑えるプロキシツールです。

ツールの基本的な導入方法と仕組みについてはすでに[公式ドキュメント](https://github.com/rtk-ai/rtk)やいくつかのブログ記事が存在するため、本記事では一歩進んで、どの操作でどれだけ削れたかという数値、そして圧縮が出力をどう歪めたかというリスクについて報告できればと思います。

データとしては、私がRTKを使用した約40日間・8,587コマンドの実行履歴データベース（`history.db`）を使用します。

## 留意点

* 環境: 検証環境は単一の devcontainer 環境・単一プロジェクト（Rails モノレポ）
* 実績値: 削減量だけ見ると派手ですが、この数字は数件の巨大ログ閲覧に依存する外れ値であり、環境や設定次第で変動します。ただし、他の開発メンバーの環境でもRTKが謳う通り高い水準で削減できていることは確認済みです

## TL;DR

* **40日間で86.8Mトークン（90.4%）を削減**
  + 総入力量 96.0M トークンを 9.3M にまで圧縮。コストとコンテキストの節約には確実な実インパクトがありました
* **削減の79.2%は、わずか7回の同一ログの再読が占めていた**
  + RTKは1,000万トークンに達する開発ログを数千トークンへと超圧縮（68.9M→144K）し、本来読めない出力を閲覧可能にしました。
  + ただし集計の結果、削減の約8割がわずか7回の操作に集中。原因は、Claude Codeが成長し続けるログの末尾を `tail` 等で絞り込まず、毎回丸ごと `cat` していたため（うち3回は68秒以内に連発）でした
  + アプリ実装・テスト週（巨大ログが出る）は劇的に効く反面、仕様・レビュー週はあまり効いでませんでした
* **信頼性のリスクあり**
  + `git commit` や `diff` などの状態確認コマンドにおいて、RTKが意図しない要約やハッシュの提示、情報の欠損を引き起こしました。
  + それらのうち一部リスクはまだ残存しているので慎重にClaude Code の動作を確認するのが無難だと思います
* **バグ検知・対応が早い**  
  rtk は66.5k starの、直近3日に2日ペースでPRマージされている超活発リポジトリ。後述の通りこの期間に検知した不具合はほぼ全て、新バージョンでマージ済み or Issue/PR登録済みでした

## そもそも RTK とは

RTK を入れると、Claude Code が実行するコマンド（`cat` / `grep` / `git diff` / `rspec` など）の出力は、Claude Code にとって意味のある部分だけに圧縮されてから使用されます。

仕組みとしては、Claude Code の hook によって、通常のコマンドは自動的に rtk <cmd> へ書き換えられて実行されます（例: `git status` → `rtk git status`、`cat foo.log` → `rtk read`）。ユーザーから見ればコマンドは普段どおりですが、LLM のコンテキストに流れ込むテキストだけが圧縮されます。

#### 考察: 巨大ブロブについて

* 「削減量の合計」セクションでも1位と3位になっていたが、RTKが効くかどうかはアプリを動かしてログ/transcriptを生む作業かで決まる様子
* 実装＋動作確認＋テスト中心（例：20週）なら効くものの、既存コードを読み差分で確認しながら足す実装（例：22週）や、仕様書・レビュー・ドキュメント中心（例：23/25週）では効きが弱い傾向にある

具体的に、全削減の 8 割を生んだRails開発ログ閲覧7 件をピックアップしてみると:

| 日付・時刻 | 入力 | → 出力 | コマンド |
| --- | --- | --- | --- |
| 05-22 10:04 | 9.17M | 8.8K | `cat /workspace/app/log/development.log` |
| 05-22 10:44:22 | 9.60M | 21.3K | `cat /workspace/app/log/development.log` |
| 05-22 10:44:50 | 9.61M | 71.8K | `cat /workspace/app/log/development.log` |
| 05-22 10:45:30 | 9.61M | 2.6K | `cat /workspace/app/log/development.log` |
| 05-27 06:03 | 10.23M | 34.1K | `cat app/log/development.log` |
| 06-16 07:35 | 10.31M | 3.3K | `cat /workspace/app/log/development.log` |
| 06-16 07:50 | 10.32M | 2.3K | `cat log/development.log` |

* Claude Code エージェントの実行 7 件すべてが、同じ追記型ログファイルへの素朴な `cat`。`tail -n` でも `grep` でも行範囲指定でもなく、毎回 900〜1,000 万トークンの全文を頭から読んでいる。
* 05-22 の朝、同一ログを連続で読んでいる。
* 前後の履歴を見ると、Claude Code のエージェントは 50MB / 100MB の分析ページを `curl` で負荷テストし、`bundle exec rails runner` を立て続けに実行し`development.log` に追記されている
* RTK はClaude Code が cat するか tail/grep するかというコマンドの選択そのものは変えられないため、仕方なくこのように非効率な出力を事後的にフィルタしている模様。transcript（JSONL）に対しても同様の構造になっていると見られる

#### 考察: 検討の余地

もし調査・検討するとしたら、以下にその余地がありそうです:

## 旧版 v0.40.0 で観測された不具合と、最新版での再発状況

たまに意図しない欠落（削ってはいけない出力まで圧縮で失うこと）などが発生しました。以下に現場観測した一次情報を記録します。特に、コマンドの失敗をrtkの出力が成功に見せかける問題が複数種類存在する状態でした。完全に安全とは言えないため、慎重にClaude Code の動作を確認するのが無難かも知れません。とはいえ、rtk は66.5k starの、直近3日に2日ペースでPRマージされている超活発リポジトリ。この期間に検知した不具合はほぼ全て、一部は新バージョン（v0.42.x）でマージ・解消済み、残りは Issue/PR に登録済みでした。

観測した問題の詳細

| 事象（v0.40.0 で観測された症状） | 原因 | 最新 v0.42.x の状況 | 残存リスク |
| --- | --- | --- | --- |
| **`git commit`** ―― 日本語メッセージで RTK が panic。「成功」風の文と**実在しない SHA ハッシュ**を含む出力を返し、実際は未コミットなのに Claude が「コミット成功」と誤認した | コミット結果を要約する処理がマルチバイト文字で panic し、フォールバックが実行結果を確認しないまま「それらしい成功出力（架空ハッシュ込み）」を組み立てて返したため | 🟢 **緩和確認。** PR #1266 で対応。v0.42.4 では日本語コミットが panic なく記録されることを実データで確認 | 出力を数トークンに畳み**実ハッシュを返さない**構造は不変（Issue #2305 ほか open）。「成功したか・どの SHA か」は別途確認が要る |
| **`git diff`** ―― 圧縮後の差分が実態とズレ、`0 files changed` 等の食い違う要約が出力された。コンフリクトの有無を取り違える恐れがあった（今回は `Read` で直接確認して回避） | RTK が unified diff を `--stat` 風サマリに畳む際、要約結果が実際の差分と食い違うことがある（畳む割合が大きいほど乖離リスクが上がる） | 🟡 **未確認。** Issue #2446 ほか残存。現データに食い違いの直接証拠はないが、1.66M を 10% へ畳む土壌は健在 | 圧縮が厚いぶん、食い違えば失うものが最大。コンフリクト判断は生差分で確認する |
| **`git status`** ―― ① `--short` で 793 ファイル分の一覧が内部メッセージ 1 行に置換・情報欠損。② `git status … | head` のようなチェーンコマンドで流すと、RTK が先頭 15 件に要約する途中で `head` がパイプを先に閉じて panic し、壊れた出力を Claude が「大量のファイルが削除された」と誤読した | ①大量の status 行を圧縮した際に要約が情報を欠落。②チェーンコマンドの downstream が先に閉じてストリームが破断し RTK が異常終了 | ① 🟡 **未確認。** Issue #2320 残存。② 🟢 **修正済（0.42.3）。** チェーンコマンドによる panic は再発しない | ①圧縮がかかる status では要約・欠損が起こりうる。②チェーンコマンドを使わない限り問題なし。※巨大な `--short`（793 行超）は現在 0% 素通し（欠損なし・量は大。→ 上表） |

※凡例  
🟢 **緩和/修正**：旧版の具体的故障が、対応バージョンで再発しない（commit は架空ハッシュが現データで再現せず ―― コミット成功そのものの検証ではない。status のチェーンコマンド panic はトラッカー上 0.42.3 で修正）。  
🟡 **未確認**：本調査では再発有無を判定できず、Issue は open。**いずれも「畳む／捨てる」という圧縮の構造自体は残る**ため、緩和＝完全解決ではありません。

**共通構造**：rtk はコマンドを実行した後、「成功らしい出力があるか」を正規表現などでパターンマッチして要約します。このとき**プロセスの終了コード（exit code）を pass-through せず、出力パターンだけで成否を判定**するケースがあるため、失敗した実行が `ok`／exit 0 として Claude に届いてしまいます。

git 系（status／worktree／stash）と go build については修正 PR が出ており、マージ後は改善されます。一方 `git diff` の微小差分誤報（#781）は修正が 2 度失敗して停滞中のため、**v0.42.4 現在も再現する**唯一のバグです。重要な差分の確認には `rtk proxy git diff` または直接 `Read` で生ファイルを見るのが確実です。
