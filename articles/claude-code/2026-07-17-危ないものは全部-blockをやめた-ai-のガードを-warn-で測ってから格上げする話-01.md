**この記事は何の話か（3 行まとめ）**

* AI（Claude）の暴走を止めるガード（Hook）は、増やすほど安全になりそうですが、実際は誤検知で正当な操作まで止まり、ガードそのものが邪魔者になって、いずれ無効化されかねません
* そこで新しいガードは原則 warn（止めずに知らせるだけ）から始め、しばらく動かして「誤検知（FP）」と「見逃し（FN）」を測ってから、block（実行を止める）へ格上げすべきかを判断することにしました
* 実際に測ってみると、格上げに値したのは3件、6件は warn のまま据え置き、2件は「確実には任せきれない」と正直に格下げ・見送りにし、残り1件は昇格→検証で差し戻し→再昇格という往復を経て block に着地しました

**前提となる用語（この記事だけで読めるように）**

* Hook：Claude Code が決まったタイミング（コマンド実行の直前など）で自動的に呼び出す、自前の小さなスクリプト
* PreToolUse：Claude がコマンド実行やファイル編集を実際に動かす「直前」に、Hook が割り込めるタイミング
* block / warn：block は操作そのものを実行させないこと、warn は実行を通したうえで警告だけ出すこと
* FP（偽陽性）／FN（偽陰性）：FP は「危なくないのに止めてしまう」誤検知、FN は「危ないのに見逃す」取りこぼし
* fail-open：ガード自身がバグで落ちたとき、黙って操作を通す（＝止めない側に倒す）設計

正直に言うと、前作の結びで予告したもう一つの宿題──「無人で仕事を撒き続ける全自動ループを、どこまで人の手から離せるか」──は、本記事の執筆時点でもまだ回収できていません。ループの仕組み自体は実装・テスト済みで、実際にウィンドウを自動起動するところまでは動きます。ですが、安全に試すための前提条件（並走セッションが少ない「静かな窓」）を、日々複数セッションを同時に走らせる自分の運用の中では逆に作りづらく、本番投入の判断はまだ見送り続けています。この境界線の宿題は、また別の機会に回収するつもりです。

もう一つ、前作の時点にはまだ無かった話として、dispatch する worker にどのモデルを割り当てるかも、前作公開の2日後（2026年7月4日）から自動振り分けになりました。タスクの種類ごとに、設計・アーキテクチャ判断のような重い判断は上位のモデルへ、dispatch そのものや定型ヘルスチェックのような機械的な作業は下位の軽量モデルへ、それ以外は中間のモデルへと、種類別に固定で割り当てる仕組みです（基本は軽く、必要な時だけ重くする、という考え方です）。この振り分け自体の効果検証はまだ途中で、本記事のテーマ（warn/block の格上げ判断）とは別軸なので、ここでは深入りしません。

先回りで止める Hook を作り始めた頃、私は単純にこう考えていました。危ない操作のパターンを見つけたら、片っ端から block にすればいい。ガードは多いほど安全だろう、と。

実際にやってみて、これは間違いでした。

## ガードは増やすほど、無視される

block を増やすほど安全になる、と思うかもしれません。実際は逆でした。

block を一つ増やすたびに、摩擦が増えます。そして誤検知（FP）で正当な操作まで止まると、ガードはただの邪魔者になります。邪魔者は、いつか無効化されかねません。`--no-verify` で素通りさせたり、環境変数で丸ごと止めたり、最悪なのは「このガードはうるさいから」と消されることです。**止めすぎるガードは、止める力そのものを失います。**

前作までで作った破壊操作のガードでも、一番苦労したのは「危険な文字列」と「危険なコマンド」を区別することでした。`grep "rm -rf" file` のような検索や、コミットメッセージにたまたま入った `rm -rf` という文字列まで止めると、ガードは信用されなくなります。突き詰めれば、これは誤検知との戦いでした。

この「増やすほど無視される」のは、QA に限った悩みではありません。Google のサイト信頼性エンジニアリング（SRE：大規模なシステムを安定運用するための技術領域）の[モニタリングの章](https://sre.google/sre-book/monitoring-distributed-systems/)でも、「すべての呼び出しアラートは対応可能（actionable）であるべき」とされ、誤報が積もって対応疲れ（アラート疲れ）に陥った現場は、いずれアラートそのものを無効化してしまうと書かれています。仕組みで暴走を止める、という話のひとつ先輩にあたる分野です。

だから私は、考え方を変えました。**「止められるか」ではなく「止めて意味があるか」「止めても誤検知で自壊しないか」を、データで確かめてから決める。**

## だから、warn から始めて測る

新しいガードは、原則 warn から始めることにしました。もう一人の同僚である Claude を、いきなり全部の操作で縛るのではなく、まず横で様子を見る、ということです。

しばらく warn（止めずに知らせるだけ）で動かし、どれくらい誤って発火するか、逆にどれくらい本物を見逃すかを、数セッションぶん観察します。そのうえで、本当に block へ格上げすべきかを判断します。

「測る」というのは比喩ではありません。warn が発火するたびに、私のガードは小さなログに記録を残します。たとえば終了時チェックの発火は専用のログに、壊れたツール呼び出しの検知は別のログに、母数（その間に動いたツール呼び出しの総数）とセットで貯まっていきます。発火の絶対数だけ見ても意味がないので、母数と並べて率で見られるようにしてあります。

じつは、回数を数えて「任意のメモから必須のルールへ昇格させる」という発想そのものは、社内の先行記事 ── まっさんの[「学習する記憶の3層構造」](https://zenn.dev/tokium_dev/articles/claude-code-evolutionary-memory) ── から学んだものです。本記事の warn から block への格上げは、その昇格を、思い込みではなく誤検知率という物差しで測りながら判断してみる試みでもあります。

この「効くはずで本番投入せず、warn で測ってから block にする」という段取りは、1本目の記事で書いた「作ったその場で試運転する」という発想の、運用版でもあります。

## 測ってみたら、四つの結末に分かれた

実際に warn で測って格上げを判断したところ、ガードは大きく四つの結末に分かれました。

順番に書きます。

### 結末① 6件は warn のまま据え置いた

測った結果、格上げに値しなかったものが6件ありました。

* **終了時の締め忘れ・独立検証の抜けを促す警告**：そもそも観測が不完全です。「別の手段で完了を検証した痕跡があるか」を見たいのに、会話の履歴は圧縮や入れ替えで一部しか追えず、誤検知のクラスを閉じきれません。だから block にはせず、警告が一定回数たまったら「このガードを block に上げるべきか再評価せよ」という小さな目印（tripwire）だけを自動で足し、強制するかどうかはログがたまってから決める、という段階導入にしています。
* **別セッションとの同時編集の警告**：前作で書いたとおり、そもそも他のセッションを止める権限がありません（これは仕組み上の固い制約です）。しかも PreToolUse で「許可」を返してしまうと、許可フローを短絡して対象の編集を無条件に通してしまう。だから「知らせるだけ」に徹します。
* **判断の丸投げを止めるガード**（この記事を書いている今も、私のセッションには毎回これが出ています）：止め方には deny を使っていますが、同じ質問をもう一度出せば通過し、1セッションあたりの停止回数にも上限を設け、ガード自身が落ちたら黙って通す（fail-open）ようにしています。block でも warn でもない「摩擦つきの warn」とでも言うべき形で、完全な壁にはしていません。
* **壊れたツール呼び出しのリセット推奨**：AI が出すツール呼び出しの形式そのものが壊れている場合、それはそもそも PreToolUse に届く前に弾かれます。機械的に割り込める経路が終了時の警告しか残っていないうえ、「いったんリセットすれば直る」という対策自体が未証明なので、強制せず推奨に留めています。
* **同一ブランチで並走するコミットの、幅広い検知**：複数セッションが同じブランチで並走しているという注意信号がたまったら、機械的な deny へ格上げすべきかを継続的に測っているガードです。信号が立った時間帯と、実際に問題が起きた時間帯を突き合わせて相関を取り直しましたが、一致は見られませんでした。この記事を書いている今日の再評価でも、同じ結論（見送り）を確認し直しています。「昔測って決めた」ではなく、都度測り直して同じ結論に落ち着いている点が、他の warn 維持組と少し違います。
* **委任判断（大きな調査は外部ツールへ渡すべきという宣言）を促す、セッション終了時の後押し**：セッションの終わり際にしか出せない性質上、「もう一度その場でやり直させる」ことができません。だから最初から block ではなく、1セッションにつき一度だけそっと知らせる形に留めています。

共通しているのは、**観測が不完全だったり、止める権限が無かったり、対策の効き目が未証明だったり、そもそもそのタイミングでは「やり直し」を強制できなかったりするものは、block に上げない**という線引きです。

### 結末② 3件、block へ格上げした

逆に、格上げに値する証拠がそろったものは、3件 block へ上げました。

一つは、並走の二重取りを止める、狭く絞ったガードです。複数セッションが同じタスクを取り合うとき、「台帳にすでに記録された先行の確保（claim）を、後から知らずに追い確保する」という具体的な経路だけを止めます。独立した別エージェントによる敵対的なレビューで、この経路を遮断対象として名指しで特定し、そこで見つかった**5件の穴**（止めすぎ・判定漏れ）を是正しました。しかも、同じIDの取り直し・死んだセッションからの引き継ぎ・出し直し・回数上限といった、逃げ道を四つ残した、範囲を狭く絞った block です。

もう一つは、外部に不可逆な副作用を出す語の検知です。タスク名に「課金」「決済」「drop table」「truncate」「webhook」などが含まれ、かつ自動起動の経路に乗っている場合だけ、人の承認を挟みます。これらを選んだ理由は、実際のログで誤検知がほぼゼロだったうえ、万一見逃したときの帰結が最大（お金を動かす・テーブルを消す・外部へ送る）だからです。

もう一つは、①で「警戒はするが止めない」に据え置いた、同一ブランチ並走コミットの検知と、根っこは同じガードです。ただしこちらは条件を極端に絞っていて、「ファイルを指定しない、丸ごとの commit」かつ「その場に居合わせた別セッションが触ったファイルと、これからコミットするファイルが実際に重なっている」という2条件が同時に成立したときだけ block します。ファイルを指定した commit や、他セッションと無関係な commit は、常に通ります。同じ「並走コミットへの警戒」でも、広く緩い判定は warn のまま据え置き、狭く物的証拠が取れる判定だけを block へ上げた、というのが実態です。

どちらも、**「危ないから」ではなく「測ったら誤検知が低く、かつ見逃しの代償が大きいから」** 上げました。数字で before → after を並べると、こうなります。

| ガード | Before（warn 段階） | After（block 昇格後） | 根拠になった数字 |
| --- | --- | --- | --- |
| 並走の二重取り防止 | ログに記録するだけ（inject-only）・止める力なし | 条件成立時のみ実行を止める（narrow deny）・逃げ道4つは意図的に温存 | 敵対レビューで欠陥5件検出 → 是正後、pure/matcher/state 25件 + e2e 5件 + 回帰1件 = **31件 全PASS** |
| 外部副作用語の検知 | 「送信」「通知」等の日本語動詞も候補に入れた広いリスト案 | 誤検知ほぼゼロの7語（課金/決済/drop table 等）に厳選 | 実ログの誤検知実測＝厳選した7語は概ね0〜1件・除外した「送信」「送付」「通知」は各2件・2件・**6件** |
| 同一ブランチ並走コミットの狭域検知 | 「ファイル未指定 commit」全般を advisory で知らせるだけ・止める力なし | 「ファイル未指定」×「他セッションが触ったファイルと重複」の2条件同時成立時のみ block | 実際に block が発動したのは2件、いずれもファイル重複が実物として存在。同時期に検知した「同一ブランチ並走」の注意信号は**12件**あったが、この2件の発生時刻とは一つも重ならなかった＝広い信号ではなく、狭く絞った物的証拠だけを昇格の根拠にした |

### 結末③ 「確実には任せきれない」と、格下げ・据え置きにした

そして、いったん block や全自動を狙ったものの、測った結果「これは確実には任せきれない」と判断して、格下げ・据え置きにしたものもあります。

外部副作用を検知する単語のリストは、敵対的な検証にかけたところ、**両方向に誤る**ことが分かりました。「PR を作成」のように助詞が入った言い回しを見逃す一方（FN）、「メール……の観点」という社内のテスト項目名を、外部送信と勘違いして過剰に止める（FP）。どちらも単語の表面的な形に依存していて、人の判断を置き換えられません。

そこで、この検知は「確実な block」ではなく **best-effort（できる範囲で）** と明文化し、確実に止めたいときは、人がタスクに明示的なフラグを立てる運用へ委ねました。「机上では止められるはず」を、測って「止めきれない」と認め、正直に降ろした例です。

向きの違う「見送り」もありました。タスクの割り当てから起動までを完全に無人で回す自動化を、いったんは目指しました。けれど、実際に子プロセスを起動して本物のデータを書き換える一連の流れは、テストでは偽物の部品でしか再現できず、実環境では検証できていません。未検証のまま全自動へ上げるのは、私がいちばん気をつけている「検証なき完了」そのものです。だから、機械が候補を選び、人が起動する一段手前の形に据え置きました。これは「格下げ」ではなく、確証が取れるまでの「昇格の見送り」です。

### 結末④ 昇格→差し戻し→再昇格を経て block に着地した

三分類のどれにもきれいに収まらないものが、1件だけありました。製品ブランチへの誤ったチェックアウトを止めるガードです。

最初は warn（実際に汚染インシデントが起きた直後の応急処置）で始め、しばらくして block へ昇格させました。ところが、その昇格を独立した別エージェントに敵対的にレビューさせたところ、誤検知と、判定をすり抜ける回避策の両方が見つかりました。そこでいったん block を取り下げ、warn だけの状態に差し戻しています。判定の式を、誤検知を起こしにくい形に作り直したうえで、もう一度 block へ昇格させました。それで終わりではなく、その後もさらに敵対的な検証にかけたところ、コマンドの書き方を変えた別の抜け道が見つかり、都度直しながら、現在も block として動いています。

一直線に「warn→測る→block」と進んだ他の2件と違い、これは**昇格したあとも敵対的な検証をやめなかったからこそ、穴を塞ぎながら block を保てた**例です（②のもう1件・並走コミットの狭域検知は、そもそも一度も差し戻していません）。測って終わりではなく、昇格した後も測り続ける必要がある、という意味で、③の「測っても分からないことの限界」に近い教訓を含んでいます。

| 結末 | 何件くらいか | 代表例 | なぜそうしたか |
| --- | --- | --- | --- |
| ① warn のまま | 6件 | 締め忘れ警告／同時編集警告／丸投げガード／壊れたツール呼び出しのリセット推奨／同一ブランチ並走コミットの広域検知／委任宣言を促す終了時の後押し | 観測が不完全・止める権限が無い・効き目が未証明・そのタイミングではやり直しを強制できない |
| ② block へ格上げ | 3件 | 並走の二重取り／不可逆な外部副作用語／同一ブランチ並走コミットの狭域検知 | 誤検知が低く、見逃しの代償が大きい |
| ③ 格下げ・据え置き | 2件 | 外部副作用の単語検知（格下げ）／無人運転の自動化（昇格の見送り） | 単語の形しだいで両方向に誤る・実環境で未検証で確実には任せきれない |
| ④ 昇格→差し戻し→再昇格 | 1件 | 製品ブランチへの誤チェックアウト防止 | 一度の昇格では見抜けない穴があり、敵対検証との往復で仕上げた |

## この記事の山場 ── 「送信」を止めようとしたら、ボタン名にぶつかった

外部に副作用を出すタスクを止めたい、と考えたとき、私が最初に書こうとしたリストには、日本語の動詞も入っていました。「送信」「送付」「通知」「投稿」。外部に何かを出す操作の代表格です。

ところが、実際のログで誤検知を数えてみると、様子が違いました。

私たちが作っているのは経費精算・請求書のプロダクトです。画面には「送信」ボタンがあり、「通知」設定があり、テスト項目には「メール送付の観点」「通知の表示」といった名前が並びます。「送信」「通知」という単語は、止めたい外部操作と、止めたくない QA の作業対象とで、見分けがつかなかったのです。ログに残った発火を定性的にざっと数えると、誤検知は「通知」が6件、「送信」「送付」が各2件と、無視できない数でした。

一方で、英語の記号的な語や、不可逆な金融・破壊の語は、誤検知がほぼゼロでした。「drop table」や「truncate」をうっかりタスク名に書くことはまずないし、「課金」「決済」も QA の画面ラベルとはぶつかりません。

そこで、**同じ「外部に出す語」でも、誤検知率で選り分けて、高 FP の日本語動詞は意図的にリストから外しました。** もし測らずに「外部に出すなら全部止める」と素朴に組んでいたら、QA の正常な作業のたびにガードが発火して、真っ先に無効化されていたはずです。

この一件は、私にとって象徴的でした。**危なさは単語の意味では決まらない。自分の環境で実際にどれだけ誤って当たるかで決まる。** それは、机の上では分からないことでした。

## 測っても分からないこと ── この運用の限界

ここまで「測ってから決める」と書いてきましたが、守れることだけを書くと信用できない記事になるので、測っても分からないことも書いておきます。限界は、はっきり三つあります。

一つは、**「率」と言っても、たいてい分母がありません。** たとえば私のいちばん再発の多い検出器は、過去に92件の警告を出しましたが、独立した3つの視点で見直したところ、その約88%は構造的な誤検知でした。けれどこの「約88%」は、件数を一つずつ定性的に分類した近似であって、精密な統計値ではありません。別の監査では「改善率を出そうにも分母（本来の母数）が無く、効果は厳密には測れない」とまで指摘されました。測ったと言っても、多くは近似と定性に留まるのが正直なところです。

ただ、分母が取れる場面では、warn のまま中身を堅くするだけでも効きます。テスト項目の「1ケースに期待値が複数」を検知するチェックは、2,093件の実データに当てると最初は違反率が 9.56% で、その大半が誤検知や境界でした。そこで block にはせず、warn の判定ロジックから構造的な誤検知を順に潰したところ、違反率は 5.64% まで下がりました。**block に上げる前に、warn 側の精度を上げる**という選択肢が、いちばん地味で、いちばん効くこともあります。

二つ目は、自己参照の汚染です。自分のガードの効き目を、自分のセッションで測ると、観測そのものが挙動を変えてしまいます。実際、壊れたツール呼び出しを調査している最中に、その壊れた呼び出しを何度も自分で再発させてしまいました。測っている対象に、測っている自分が影響してしまうのです。

三つ目は、線引きが固定ではないことです。ガードは今も改修中で、warn と block の境目は測り直すたびに動きます。今回 block に上げなかったものが、ログがたまれば上がるかもしれないし、その逆もあります。これは「一度決めて終わり」ではなく、ずっと測り続ける運用です。

## おわりに ── 伝える・整える・止める・避ける・測る

5本の記事をつなげると、こうなります。

| 記事 | やったこと | 守り方 |
| --- | --- | --- |
| 1本目 | 振り返りと試運転を AI に翻訳 | 起きた後に学び、作った直後に試す |
| 2本目 | 知識を4つの置き場所に整理 | 何を読むかを整える |
| 3本目 | 先回りで止めるガード層 | 取り返しのつかない操作を完了前に止める |
| 4本目 | 止められない並走を、歩いて避ける | 止めずに、ぶつかる前に確認する |
| 本記事 | warn で測ってから block を厳選 | 増やす前に、効くかを測る |

「危ないものは全部 block にすればいい」と思っていたのが、いざ測ってみると、格上げに値したのは3件、6件は warn のまま、2件は正直に格下げになり、残り1件は昇格と差し戻しを繰り返してから block に着地しました。守りを足すこと自体が目的になりかけていた自分には、これは少し意外な結果でした。強い力を増やすより、**増やした力が誤って自分を縛らないかを測るほうが、結局は守りを長持ちさせる**。それが、warn から始めて測るという地味な段取りの、いちばんの効きどころでした。

人と AI が一緒に品質を守る時代に、ガードは「どれだけ強く止められるか」で評価されがちです。でも実際に効いてくるのは、**止めたあとに自分の作業を壊さないか、誤って邪魔者にならないか**を、データで確かめ続ける慎重さのほうかもしれません。

次は、ここまでのガードでは届かない領域 ── 壊れたツール呼び出しや、判断の丸投げのように、PreToolUse の Hook では機械的に止めきれず、AI 自身の生成側の規律でしか狭められない穴について、また別の記事で書こうと思います。
