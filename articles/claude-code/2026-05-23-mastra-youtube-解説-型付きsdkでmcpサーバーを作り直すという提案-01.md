## 概要

本動画は、Mastra が主催した **「TypeScript AI Demo Day」**（2026年4月開催）というイベントでの登壇模様です。この発表は、MCP サーバー設計がここ数か月でどう変わってきたのかを、Kevin Whinnery が「3世代」という整理で振り返るトークです。出発点は、MCP が登場した当初に多くの実装が採用した「大量のツールをそのままモデルに見せる」方式でした。しかし実運用では、ツール数が増えるほどモデルが正しいツールを選びにくくなり、ツール定義そのものがコンテキストを圧迫するという問題が顕在化します。

そこから話は、Anthropic が進めたコード生成ベースの動的 tool calling、Cloudflare の OpenAPI 検索＋HTTP 実行という二段構えの方式へと進みます。そして Kevin が最終的に推すのが、Stainless の「SDK code mode」です。これは、MCP サーバーが返すのを大量の個別ツールではなく、API 専用の型付き SDK を前提にした検索ツールとコード実行ツールの組み合わせにし、モデルに「型のあるコード」を書かせる設計です。

発表の後半では OpenAI Console を使ったデモが行われ、search docs と execute の 2 ツールだけで API のドキュメント探索から実行までを回し、最終的に GitHub 上の config repo URL を取り出す流れが見せられます。最後の質疑では、こうした方式が精度を高める一方で、コード実行型のエージェントがサンドボックスの安全性に強く依存することも正面から認められます。つまりこの発表は、「MCP の精度問題をどう解くか」と「その代償として何を新たに管理しなければならないか」を、かなり実務的に示したものです。

## 要点

1. **第1世代の MCP は「大量ツール公開型」だった**。しかしツールが多すぎると、モデルは適切な選択が苦手になり、ツール定義だけでコンテキストが膨らむ。
2. **Anthropic 型の第2世代は、モデルにコードを書かせて動的にツールを発見・実行させる**。大量ツールを静的に渡すよりは良いが、精度のばらつきは残る。
3. **Cloudflare は OpenAPI spec の検索と HTTP 実行に役割を分けた2ツール方式を示した**。少ないツールで API 空間を探索でき、トークン効率も高い。
4. **Stainless はその流れをさらに進め、API 専用の型付き SDK をモデルに使わせる方式を提案している**。
5. **型付き SDK の利点は、リクエスト、レスポンス、エラーが静的解析可能になること**。モデルにとって操作対象がずっと明確になる。
6. **MCP サーバーは search docs と execute の2ツールに絞れる**。search docs が API ごとの実践的な文脈を返し、execute がサンドボックス内で SDK コードを実行する。
7. **ドキュメント検索は README や examples、手書きドキュメント、OpenAPI spec まで含めて組み立てると強い**。単純な grep よりサーバーサイドでの文脈構成が重要になる。
8. **デモでは docs 検索とコード実行のループで、ライブ API をたどりながら目的の GitHub URL を取得した**。
9. **この方式は中間レスポンスを大量に会話コンテキストへ積まなくてよい**。複雑な API 操作でもコンテキスト膨張を抑えやすい。
10. **ただし安全性は未解決**。コード実行型エージェントのセキュリティは、サンドボックス権限と認証情報管理の設計に大きく依存する。

## 詳細

### 1. Kevin と Stainless の自己紹介

Stainless は自社 API を作っている会社というより、顧客が持つ OpenAPI Spec と設定ファイルをもとに、SDK、ドキュメント、MCP サーバー、CLI、Agent Skills などを生成するための基盤を提供する会社です。言い換えれば、API そのものを受託開発するというより、**API 提供者が自分たちの API を開発者やエージェントにどう見せるか**をまとめて設計・生成するためのプラットフォームを仕事にしている会社です。

Kevin は、いずれほとんどの開発者が API プロバイダになり、そこで必ず「エージェントに API をどう公開するか」という問題に直面すると言います。発表の主題は、**API 提供者の視点から見たエージェント向けインターフェース設計**です。

### 2. 第1世代: 何百ものツールをそのままモデルに見せる設計

最初の世代として Kevin が挙げるのは、MCP サーバーにあるツールを大量にそのままモデルへ公開する方式です。実際、現在の [GitHub 公式 MCP Server の README](https://github.com/github/github-mcp-server) を見ても、多数のツールやツールセットが並んでおり、登場当初はこの形が自然に見えました。MCP がもたらした新しさは、モデルがネットワーク越しにリモートツールを呼べるようになったことにあり、その可能性自体はたしかに大きかったからです。

ただし実運用ではすぐに限界が見えました。問題は 2 つあります。1 つは、**ツール数が多すぎるとモデルが正しいものを選べない**ことです。もう 1 つは、**ツール定義そのものがコンテキストを占有し過ぎる**ことです。API が大きくなればなるほど、この設計はスケールしません。Kevin はここをかなり率直に、「急速に持続不可能になった」と表現します。

次に出てくるのが Anthropic のアプローチです。ここではモデルにコードを書かせ、まず自分が利用できるツールを探索させたうえで、そのツールをプログラム的に呼び出させます。たとえば Anthropic が [Claude 3.5 Sonnet の SWE-bench 用エージェント](https://www.anthropic.com/research/swe-bench-sonnet) を解説した記事では、Bash Tool と Edit Tool という少数の汎用ツールだけを与え、モデル自身にリポジトリ探索、再現スクリプト作成、コマンド実行、修正、再検証の流れを判断させています。静的に何百ものツール定義を先に押し込むのではなく、必要に応じて発見させるわけです。

この方式にははっきりした利点があります。コンテキストの膨張を抑えられますし、モデルはコード生成自体はかなり得意なので、ツール呼び出しのロジックを書かせることもできる。つまり、単純なツール選択問題を、よりモデルが得意な「コードを書く問題」に変換しているわけです。

ただし Kevin は、この方式でも精度のばらつきは大きかったと述べます。ツール定義の形式がまだ新しく、API の使い方を正しく発見するプロセスも安定しきらなかったからです。大量ツールをそのまま見せるよりは改善するが、まだ決定版ではない。ここが次の世代への橋渡しになります。

### 4. 第2世代その2: Cloudflare の「検索ツール＋実行ツール」

Anthropic と並行して、Cloudflare は少し違う方向から同じ問題を解こうとしました。Kevin の説明では、ここで重要なのはツール数を 2 つにまで絞ることです。1 つは OpenAPI spec を探索するツール、もう 1 つは見つけたエンドポイントに対して HTTP リクエストを実行するツールです。Cloudflare 側でこの発想に近い公開例としては、[Cloudflare Docs 向け AI assistant の紹介記事](https://blog.cloudflare.com/introducing-ai-assistant-for-cloudflare-docs/) があります。これは Kevin が説明した「OpenAPI 検索ツール＋HTTP 実行ツール」の二段構えそのものを詳述した記事ではありませんが、Cloudflare が API やドキュメント探索を AI アシスタントへどう組み込もうとしていたかを追う参考になります。

この設計のポイントは、モデルに最初から API 全体の操作手段を細かく渡すのではなく、**API の地図を探索する能力**と、**見つけた場所へ実際にアクセスする能力**を分けることにあります。結果として、トークン効率や精度はかなり良くなる。ただし Kevin は、これを成功させるにはハーネス側の文脈設計や制御もかなり必要だと補足します。つまり、よい方向ではあるが、まだ運用者の工夫に負う部分も大きいという位置づけです。

### 5. 第3世代: Stainless の SDK code mode

ここで発表の中心的な主張が出ます。Stainless は Cloudflare 的な「探索＋実行」という枠組みを引き継ぎつつ、実際にモデルが操作する対象を HTTP レベルではなく **型付き SDK** に変えるべきだと提案します。

Kevin が強調するのは、型付き SDK が持つ情報量です。TypeScript SDK なら、リクエストの入力型、レスポンス型、エラー型などがそろっており、コード実行前に静的解析できます。これによってモデルは「ただのエンドポイント一覧」よりはるかに豊かなインターフェースを得ます。API ごとに固有の型が用意されているため、どの引数を渡せばよいか、返ってくるデータをどう扱うか、どういう失敗があり得るかが、コード構造のレベルでモデルに伝わります。

たとえば同じ「顧客を 1 件作る」操作でも、モデルが探索の先で見つける操作面はかなり違います。

```
// HTTP レベルの操作面のイメージ
POST /v1/customers
{
  "email": "ada@example.com",
  "name": "Ada Lovelace"
}
```

HTTP だと、これが「顧客作成」だと分かっても、モデルはまだ URL、メソッド、body の shape、返り値の shape を正しく扱い続けなければなりません。型付き SDK では、探索の先にある操作面そのものが、より意味的です。

```
// 型付き SDK のイメージ
const customer = await client.customers.create({
	email: "ada@example.com",
	name: "Ada Lovelace",
});
```

この場合、モデルに見えているのは単なる HTTP 断片ではなく、「customers を create するメソッドがある」「そこに email と name を渡せばよい」「返り値は customer 型で返ってくる」といった、目的に沿って整理された操作面です。つまり、**使うべき操作を見つける段階でも、見つけた後に正しく呼ぶ段階でも、型付き SDK のほうがモデルにとって扱いやすい**わけです。

なお、ここでいう型付き SDK は zod そのものを指しているわけではありません。zod のようなスキーマライブラリが型生成や実行時検証に使われることはありますが、Kevin が言っている本質はもっと広く、API ごとのメソッド名、引数型、レスポンス型、エラー型が整理された「型付きクライアント」をモデルに渡す、という発想です。

[Stainless の blog 記事「SDK code mode shows SotA accuracy and performance for agents using APIs」](https://www.stainless.com/blog/sdk-code-mode) を読むと、この主張の背景もさらに明確になります。そこでは、SDK code mode が効く理由として、単に型があることだけでなく、**idiomatic な SDK、API 固有ドキュメント、型チェック、API 固有のエラーメッセージ、実行時バリデーション**が一体でモデルへフィードバックを返せることが挙げられています。つまり SDK code mode の強みは「SDK を使う」単独の話ではなく、**検索されたドキュメントと、型付きコード実行と、良質な失敗フィードバックがつながっていること**にあります。

同記事では Increase Banking API を使った eval も公開されており、Stainless、Anthropic Code Mode、Cloudflare、Dynamic の 4 構成を比較しています。その結果、Stainless の SDK code mode は completeness 98%、efficiency 95%、factuality 53%、平均所要時間 48.5 秒で、4 構成の中で最良の総合結果を出したと説明されています。動画ではこの数表自体までは詳しく踏み込みませんが、Kevin の「現時点ではこれが最善に近い」という断言は、この blog 記事側の eval を背景に読むとかなり理解しやすくなります。

現時点で API を MCP 経由で正確に扱わせる方法としては、この SDK code mode が最善に近いのではないか。つまり「MCP をどう設計するか」の議論を、「何個のツールを出すか」から「モデルにどんな型付き操作面を与えるか」へ移すべきだという提案です。

### 6. デモ: OpenAI Console 上で Stainless API を操作する

理論の説明のあと、発表はすぐデモに入ります。舞台は OpenAI Console です。Kevin はここをかなり高く評価していて、モデル API リクエストの文脈の中でリモート MCP サーバーを視覚的に試せる点を強調します。

デモ対象は Stainless 自身の API です。この API では build 作成や spec のアップロード、SDK や MCP サーバー、ドキュメントの生成ができ、当然 SDK も存在します。ここで重要なのは、**MCP サーバー自体も TypeScript SDK の一部として生成できる**という点です。Stainless がホストするなら Cloudflare Workers 上で動かせるし、NPM に公開して自前運用することもできます。発表の主張が、単なる研究アイデアではなく、すでに生成物として成立していることを示す場面です。

ここで 1 点整理しておくと、**動画内のデモそのもの**は OpenAI Console 上で Stainless API を相手に行うライブデモですが、先ほどの blog 記事では別途、Increase Banking API を使った評価用の具体例が示されています。そちらでは `docs_search` を 2 回使って必要な操作を調べ、その後 `execute` に SDK コードを渡して複数回の試行を行い、最終的に「expense reimbursements を誰にいくら送ったか」を正しく答える流れが掲載されています。つまり動画のデモと blog の事例は同一ではありませんが、**search docs で API の使い方を絞り込み、execute で型付き SDK コードを走らせ、失敗したらエラーを見て修正する**という基本パターンは共通しています。

blog 記事で示される流れをかなり単純化すると、モデルはまずこうしたコードを書いて SDK を実行します。

```
async function run(client) {
	const results = [];
	for await (const transfer of client.achTransfers.list()) {
		const desc = `${transfer.statement_descriptor ?? ""} ${transfer.individual_name ?? ""}`.toLowerCase();
		if (desc.includes("reimburs") || desc.includes("expense")) {
			results.push({
				recipient: transfer.individual_name,
				amount: transfer.amount,
			});
		}
	}
	return results;
}
```

この種の例が分かりやすいのは、モデルが単に 1 回 API を叩くだけでなく、**SDK を使ってページネーション付きの一覧をたどり、条件で絞り込み、必要なフィールドだけを抜き出す**ところまで 1 つのコード実行の中で済ませているからです。Kevin が動画で「複雑な操作でも中間レスポンスをコンテキストへ積み上げずに済む」と言っていたのは、まさにこの点です。

### 7. search docs ツールは「API の使い方そのもの」を返す

ここから Kevin は MCP サーバーが公開する 2 ツールを具体的に説明します。1 つ目の search docs は、自由文クエリと生成したい言語を受け取り、モデルが API 統合コードを書くために必要な情報群を返します。

この説明で重要なのは、返すのが単なる API リファレンス断片ではないことです。Stainless は生成ドキュメント、手書きドキュメント、SDK の example フォルダ、README、OpenAPI Spec を見られるため、検索結果に API ごとの実践的な文脈をまとめて返せます。Kevin はこの文脈組み立て能力をかなり重視していて、単純な grep に頼るより、**サーバー側で検索してコンテキストを組み立てる設計が再評価されるはずだ**と述べます。

ここは発表全体の中でも、かなり示唆の多い部分です。MCP を単なる「リモートツール呼び出しの標準」としてではなく、**ドキュメント検索と文脈供給の面も含めた API 利用体験の設計面**として捉え直しているからです。

### 8. execute ツールは、生成した SDK コードをサンドボックスで走らせる

2 つ目の execute ツールは、モデルが作った統合コードを安全なサンドボックスで実行するものです。Kevin は Stainless にも実験的な参照実装があると述べつつ、この部分は他社の sandbox execution サービスに置き換えてもよいと説明します。

この execute ツールが重要なのは、モデルが API を「関数呼び出し」ではなく「コードを書くこと」で扱えるようになる点です。結果のフィルタリングや前処理、後処理も JavaScript などの一般言語で柔軟に書ける。中間 API レスポンスを一つひとつ会話コンテキストに積み増ししなくてよいので、複雑な処理でもコンテキストウィンドウの膨張を抑えられます。

ここで Kevin は「モデルがやるべきこと」を明確にしています。モデルにすべてのツール選択をさせるのではなく、**検索で必要な文脈を集め、型付き SDK コードを書き、サンドボックスで実行し、失敗したら修正して再試行する**というループへ役割を整理しているわけです。

### 9. デモ本番: Pet Store の config repo URL を探させる

具体例として Kevin がモデルに頼むのは、「Pet Store project の config repo の GitHub URL を教えて」という依頼です。ここでは developer message に「MCP サーバーを使って API コードを生成せよ」という一般指示があり、user message に目的タスクが置かれています。

モデルはまず利用可能なツール一覧を確認し、少し推論したうえで docs を検索します。必要な情報が一度で揃うとは限らないため、検索は複数回走ることがあります。その後、execute ツールで実際の SDK コードを実行し、ライブ API から結果を取ります。もしエラーが返れば、そこで止まらず、コードを修正してもう一度試せます。

Kevin はこのループを特別ドラマチックに誇張せず、あくまで自然な実行の流れとして見せています。実際、今回のデモでもコード実行は 2 回必要でしたし、docs 検索も何度か行われました。それでも最終的には、目的だった configure repo の URL にたどり着けています。ここで見せたかったのは、**少数ツール＋型付き SDK＋コード実行の組み合わせなら、モデルが API をかなり実務的に使える**という点です。

### 10. doc search の返り値と execute 時の前提

デモを見ながら Kevin は、各ツールが実際には何を返しているのかも説明します。doc search はクエリ結果の配列を返し、その中には API の使い方に関する高レベルの Markdown 概要もあれば、追加文脈を与える補助文書も含まれます。つまり、単一の検索ヒットではなく、モデルがコードを書くのに必要な周辺知識の束を返しているわけです。

一方で execute に渡すコードは、認証済み SDK クライアントがあらかじめ存在する前提で書かれます。モデルはそのクライアントを使って API を叩き、必要ならレスポンスをメモリ上でフィルタするコードまで書けます。この「クライアントは用意されている」という前提も重要です。モデルに毎回認証まですべて任せるのではなく、**実行環境が最低限の足場を提供し、その上でモデルに処理を書かせる**構造だからです。

### 11. まとめ: MCP を「型付きコード実行」の入口として再定義する

発表の締めで Kevin は、もしこれまで MCP 経由で API を実用的に使わせようとしてうまくいかなかったなら、その API 専用の型付き SDK をエージェントに使わせることで改善できるはずだと述べます。ここで名前としては Stainless を挙げつつも、主張は必ずしも自社製品に閉じていません。たとえ Stainless を使わなくても、この設計原理には十分な将来性がある、と言っています。

このまとめ方から見えてくるのは、Kevin が本当に訴えたいのが「Stainless を使ってほしい」よりも、**MCP の操作面を再設計すべきだ**という点だということです。大量のツールを公開するのではなく、API 固有の型付き操作面と、そこへ到達するための検索文脈を組み合わせる。発表全体はこの主張を導入からデモまで一貫して支えています。

### 12. 質疑: 精度は上がるが、セキュリティはまだ未解決

最後の Q&A では、聞き手から「execute environment になるなら、セキュリティ前提はどうなるのか。新しい攻撃面を広げてしまわないか」という非常にまっとうな質問が出ます。ここで Kevin はごまかさず、**それはまだ未解決問題だ**と認めます。

コードを API に対して実行する以上、安全性はサンドボックスの安全性に依存します。ネットワーク権限、ファイルシステム権限、その環境で何が許されているかがそのまま攻撃面になります。現状、この方式が多く試されているのは内部ツールの文脈であり、比較的強い権限を持った環境も多い。つまり、この方法はまず精度を大きく改善するが、その代わりに実行環境の権限制御という別の難題を正面から引き受けることになります。

Kevin が有望な方向として挙げるのは、スコープ付き OAuth トークンや credentials vault のような仕組みです。たとえば Anthropic の managed agents platform が紹介され、必要な権限だけを持つ認証情報をサンドボックスへ注入できれば、実行コードの能力もその範囲に閉じ込められます。また、型付き SDK コードを使うことで、実行前の静的解析や実行後の監査がしやすくなる可能性もあると述べます。

この質疑は短いですが重要です。発表全体が「精度をどう上げるか」に向いていたのに対し、最後に「その精度向上はどんな安全性モデルの上に成り立つのか」という現実的な問いを残すからです。したがってこのトークは、単に MCP を賢く使う方法の紹介ではなく、**今後のエージェント実行基盤がどこで責任を持つべきか**まで含めた問題提起として読むのが正確だと思います。

## 後日談：AnthropicによるStainless買収（2026年5月18日）

### 動画公開と買収発表は同じ日

本記事で取り上げた動画が公開された2026年5月18日、Anthropicは奇しくも**同日にStainlessの買収**を発表しました。買収額などの契約条件は非公開ですが、複数の関係者による報道では3億ドル（約450億円）超とされています。Stainlessは2022年に元StripeエンジニアのAlex Rattrayが設立し、Sequoia CapitalやAndreessen Horowitzといった著名なVCから資金調達を受けてきた注目のスタートアップでした。

### 「中立的なインフラ」が一夜にして終わった

Kevinがこの登壇で繰り返し強調していたのは、StainlessがOpenAPI仕様からSDK・ドキュメント・MCPサーバーを自動生成する**中立的な基盤**として機能しているという立ち位置です。実際、Stainlessの顧客にはAnthropicだけでなく、OpenAI・Google DeepMind・Perplexity・Groq・Cloudflareといった名だたる競合他社も名を連ねていました。

ところが買収後、Anthropicはこれらすべての顧客向けに**ホスト型サービスおよびSDKジェネレーターを2026年9月1日をもって終了する**と発表しました。既存顧客はこれまでに生成したSDKの所有権と改修の権利は維持できるものの、今後は自社でSDKの保守を行わなければなりません。OpenAIやGoogleなど業界を代表する企業たちが、開発環境の再構築を迫られる事態となりました。

### MCPの「規格」と「実装インフラ」を同時に手中に収めた

この買収が業界的に注目を集めた理由は、単なるM&Aではなく、**MCPエコシステムの主導権争い**という文脈で語られている点にあります。

AnthropicはMCPを2024年にオープンソースとして提唱し、業界標準として普及させてきました。2026年3月時点での月間SDKダウンロード数は9,700万回を超え、公開サーバー数は1万台以上にのぼります。そこへStainlessの買収が加わることで、Anthropicは**規格（MCP仕様）と、その実装を支える最大の自動化インフラの双方**を手中に収めたことになります。

これはGoogleがKubernetesをオープンソースとして業界標準に育てたのち、GKE（Google Kubernetes Engine）というマネージドサービスで自社クラウドへの誘導を図った戦略と同種のものとして分析されています。「規格をオープンにして業界を巻き込み、その実装基盤を後から独占する」という手法の再演です。

### この登壇が「味わい深い」理由

Kevinがこの発表の冒頭で述べていた「いずれほとんどの開発者がAPI提供者になり、エージェントにAPIをどう公開するかという問題に直面する」という言葉は、買収後の文脈で読み返すとひと味違う重みを持ちます。Stainlessはまさにその問いに答えるための中立的インフラとして成長し、その価値の大きさがAnthropicの買収動機そのものになったわけです。

また、発表の締めでKevinが「たとえStainlessを使わなくても、この設計原理には十分な将来性がある」と述べていた部分も、今となっては皮肉な響きを持ちます。設計思想の普遍性は変わりませんが、**最も洗練された自動生成インフラはAnthropicの傘下へ移った**ことで、競合他社にとっての開発者体験における競争条件は大きく変化しました。

なお、同様の動きは業界全体で加速しており、OpenAIも2026年3月に人気Pythonツール開発企業Astralを買収しています。ただしそちらはサービス継続を維持しており、今回AnthropicがStainlessのホスト型サービスを終了するという決断は、より攻撃的なインフラ独占の姿勢として受け止められています。AI業界の競争の主戦場が、モデル単体の性能から**周辺ツールチェーンやシステム統合の支配**へとシフトしていることを象徴する出来事といえます。
