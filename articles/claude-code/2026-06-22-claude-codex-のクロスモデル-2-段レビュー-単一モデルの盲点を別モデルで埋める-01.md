## はじめに

[第1回](https://zenn.dev/kashi923/articles/c63230ad1f8700)では、開発を Phase/Stage/Gate に構造化し、状態をファイルに逃がして `/clear` に強くする設計を紹介しました。

レビューをエージェントに任せ始めた頃、「自分の書いたコードを自分でレビューさせても、肝心の見落としはすり抜ける」という壁に何度かぶつかりました。その実感が、これから紹介する仕組みの出発点です。

今回は第1回で示した階層のうち **Step（= 1 commit）の内側**を掘り下げます。ここに私の運用でいちばん尖っている仕組みが入っています。

> **1 つの Step を 4 つの Stage に割り、各 Stage で Claude と Codex の 2 人のレビュアーに独立して見せる。**

Claude にコードを書かせて、同じ Claude にレビューさせても、自分の盲点は自分では見えません。「Step を完了させたい」というバイアスもかかる。そこで、**書いた本人とは独立したレビュアー**を 2 枚重ね、**うち 1 枚は別モデル**にする。これが本記事のテーマです。

## なぜ単一モデルのセルフレビューでは足りないのか

エージェントに「レビューして」と頼むと、それなりの指摘は返ってきます。でも次の限界があります。

1. **同一モデル・同一文脈では盲点が残りやすい。** 生成したモデルとレビューするモデルが同じだと、訓練データに由来する癖や見落としも似通う。同じ穴を同じように見逃しやすい。
2. **完了バイアス。** メインの会話を進めているエージェントには「この Step を終わらせたい」という圧がかかる。自分のコードに甘くなる。
3. **観点の混線。** 設計の妥当性とコードの綺麗さとテストの十分さを一度に見ると、どれも中途半端になる。

これを 3 つの工夫で潰します。**(a) 観点を Stage で分離する**、**(b) レビュアーをメインの会話から独立させる**、**(c) 別モデルを重ねる**。順に説明します。

## (a) 4 Stage で観点を分離する

Step を 4 段階に割り、各段階で別の品質観点だけを集中レビューします。

```
[ Stage 1. 設計 ]      docs/phaseN/design.md を更新 → Gate 1（設計レビュー）
[ Stage 2. テスト設計 ] docs/phaseN/test-plan.md を更新 → Gate 2（テスト設計レビュー）
[ Stage 3. 実装 ]      コード + テストコード、/check 通過 → Gate 3（コードレビュー）
[ Stage 4. テスト実施 ] cargo test + カバレッジ計測、test-evidence 記録 → Gate 4（結果レビュー）
       ↓
    Step 全体で 1 commit
```

設計の善し悪しと実装の綺麗さは別の頭で見るべきもの。混ぜると「設計はいいけどテストが薄い」みたいな複合状態を一度に裁こうとして精度が落ちます。Stage で割れば、各 Gate のレビュアーは 1 つの観点に集中できます。

## (b)(c) Gate の中身: Claude → 修正 → Codex → 統合判定

各 Gate は **2 人のレビュアーを順次**通します。並行ではありません。ここが設計の核心です。

```
Gate 開始
  │
[ sub-1 ] Claude subagent review（project-aware）
  - メイン agent が Agent tool で general-purpose subagent を fresh context で起動
  - 指摘を docs/phaseN/review-records-claude.md に記録
  │
[ sub-2 ] メイン agent が修正適用
  - 指摘を「修正適用 / 妥当性記録 / refactoring 化 / 要確認」に分類して反映
  │
[ sub-3 ] Codex CLI review（fresh-eyes、修正版に対して）
  - codex_review.sh を実行、指摘を review-records-codex.md に記録
  - ※ Codex には Claude の指摘も対応分類も見せない（独立性確保）
  │
[ sub-4 ] メイン Claude が Codex 指摘を統合判定
  - 各 finding を再分類して反映
  │
Gate 通過 → Stage 確定
```

役割分担はこうです。

* **Claude = primary reviewer（project-aware）。** プロジェクト規約・過去の設計判断 D-N（設計判断を番号付きで積んだログ。第1回参照）・未対応リファクタリングを踏まえて指摘する。先に Claude で直しておくことで、Codex に渡る前に「プロジェクト固有の問題」を消しておける。
* **Codex = secondary reviewer（fresh-eyes）。** Claude が修正した後の状態に対して、何も知らない第三者の目で「Claude が見落とした項目」「Claude の修正が新たに生んだ問題」を拾う。
* **メイン Claude = integrator。** Codex はプロジェクトの文脈を知らないので、その指摘をそのまま採用してはいけない。プロジェクト文脈を踏まえて最終判定する。

**順次にする理由**は効率です。Claude → 修正 → Codex の順なら、Claude が拾った問題は Codex に回らないので、Codex の指摘は純粋に「Claude が見落とした項目」に絞られる。並行だと同じ問題を 2 回処理することになり、統合も面倒になります。

## レビュアーのスタンスを明確に分ける

2 人のレビュアーには、それぞれ別の立場を与えます。

**Claude subagent（project-aware reviewer）**

* プロジェクト規約（`.claude/rules/`）、設計判断ログ（`docs/decisions.md`）、リファクタリングログを踏まえる
* 既存判断 D-N との整合、過去 Phase との一貫性、未対応リファクタリングとの衝突を見る
* メインの会話とは別の fresh context（会話履歴を引き継がない状態）で起動するので、完了バイアスから独立している

**Codex CLI（third-party / fresh-eyes reviewer）**

* プロジェクト固有のしがらみには触れさせない。コーディング規約や設計判断ログ（`decisions.md`）のような固有の文脈は渡さない
* 一般的なソフトウェア工学・設計理論・業界の best practice の観点だけで評価させる
* 起動時に「あなたは外部の第三者レビュアーである」と明示する

この「片方は文脈を知り尽くし、片方は何も知らない」という非対称が効きます。文脈を知るレビュアーは規約違反を捕まえ、何も知らないレビュアーは「新参者には伝わらない暗黙の前提」を捕まえる。両方ないと盲点が残ります。

## 観点を ID 体系で一元管理する

「ちゃんとレビューして」だけだと毎回ブレるので、観点を ID で固定し、両レビュアーに同じ定義を参照させています。命名規約はこうです。

* **B（Base）**: 両レビュアーが必ず見る基本観点。`B1`, `B2`, …
* **C（Claude 特有）**: project-aware レビュアーのレンズ。`C1`, `C2`, …
* **T（Third-party / Codex 特有）**: fresh-eyes レビュアーのレンズ。`T1`, `T2`, …

たとえば Stage 3（コード）の観点を抜粋すると:

| ID | 観点 | 担当 |
| --- | --- | --- |
| B1 | 読みやすさ・idiom・early return・凝集度 | 両者 |
| B5 | エラーハンドリング（`Result` 伝播、不要な `unwrap()`） | 両者 |
| C1 | 複雑度規約遵守（cognitive 20 / lines 80 / args 5） | Claude |
| C4 | レイヤ依存方向（`ui → analysis → domain` の一方向） | Claude |
| T2 | SOLID / DRY / KISS / YAGNI | Codex |
| T6 | 過度な防御的プログラミング | Codex |

レビュー出力では `Stage 3 / C1` のように観点 ID を必ず併記させます。これで「何を根拠にした指摘か」が追え、後から観点の抜けも検出できます。出力フォーマットも両者で統一しています。

```
- ファイル:行番号
  問題: <何が問題か。観点ID（B1〜 / C1〜 / T1〜）併記>
  推奨修正: <どう直すべきか>
```

指摘がなければ `指摘なし` の 1 行。末尾に「次 Stage に進んで OK か」の全体評価サマリを付けさせます。

## 記録は独立ファイルに分ける

2 人の出力は **干渉しない独立ファイル**に保存します。

| ファイル | 用途 |
| --- | --- |
| `docs/phaseN/review-records-claude.md` | Claude subagent のレビュー記録専用 |
| `docs/phaseN/review-records-codex.md` | Codex CLI のレビュー記録専用 |

分ける理由は独立性の担保です。Claude と Codex は互いの出力を見ない。両方を読んで統合判断するのはメイン Claude だけ。もし Codex に Claude の指摘を見せてしまうと、Codex がそれに引きずられて「色眼鏡レビュー」になり、独立性が壊れます。

各エントリには findings に加えて **対応分類**を残します。

* **修正適用**: 実際に直した。適用箇所も書く
* **妥当性記録**: 採用しなかった。理由を書く
* **refactoring 化**: 今は直さず `docs/refactoring.md` に `#NN` として起票
* **要確認**: 保留。事由を書く

「指摘を受けた」だけでなく「どう処理したか」まで残すのがポイントです。これが後から「なぜこの指摘を採らなかったか」を説明できる証跡になります。

## Codex CLI のモデル割り当て

Codex は Stage ごとに最適なモデルを割り当てます（`CODEX_MODEL` 環境変数で上書き可）。以下は Codex CLI 0.135.0・2026-05 時点の構成です。モデル名は今後変わり得ます。

| Stage | mode | レビュー対象 | デフォルトモデル |
| --- | --- | --- | --- |
| 1 設計 | `design` | `design.md` 全文 | `gpt-5.5`（最上位寄り） |
| 2 テスト設計 | `test-plan` | `test-plan.md` 全文 | `gpt-5.3-codex` |
| 3 コード | `code` | `git diff --staged --function-context` | 自動振り分け※ |
| 4 テスト結果 | `test-result` | テストファイル全文 + `test-evidence` 該当エントリ | `gpt-5.3-codex` |

※ Stage 3 は変更対象が計算・ドメインロジック（`src/analysis/` または `src/domain/`）を含むなら `gpt-5.5`、それ以外は `gpt-5.3-codex` に自動で振り分けます。コアロジックほど強いモデルを当てる発想です。

起動は 1 コマンドです。

```
bash .claude/scripts/codex_review.sh code   # Stage 3 のコードレビュー
```

認証は ChatGPT login を使い、API キーは持たせていません。

## サニタイズ: 外部に出してはいけないものを構造で止める

Codex は外部サービスです。**個人資産データやシークレットを送ってはいけない**。これを「気をつける」ではなく構造で止めます。

起動スクリプト `codex_review.sh` が、送信前にパスフィルタで以下を一律除外します。

```
絶対送信禁止（path / pattern ベース）:
- data/production/*           … 個人資産データ（最重要）
- .env, .env.*                … 環境変数（.env.example は除く）
- *.pem, *.key, *.p12, *.pfx  … 秘密鍵
- secrets/, credentials/      … 資格情報
- db/tracker.duckdb, logs/*.log … ランタイム生成物
```

逆に、ソースコードや設計ドキュメントは「個人資産値を含まない設計」なのでそのまま送れます。`src/**`、`tests/**`、`docs/**`、`data/sample/**`（コミット済みの公開サンプル）、テストフィクスチャ。

ただし、送れることと、独立性のためにレビュー入力へ渡すことは別の判断です。`decisions.md` も送信できますが、独立性を保つためレビュー入力に含めません。

スクリプトはさらに監査もします。

1. 送信前に「禁止パスに該当するか」をチェックし、該当なら warning + 中断
2. 送信内容のサイズ・行数をログに記録
3. 5 桁以上の連続数値の出現を warning ログに記録（後追い監査のため）

ここは第3回の「rule + hook の多層防御」と地続きで、「ルールで諭す」だけでなく「スクリプトで止める」を重ねる思想です。

## 現実の運用での注意点

きれいごとだけでなく、実運用でぶつかった制約も正直に書いておきます。

* **Codex のモデルと認証の組み合わせには相性がある。** ChatGPT login で使えるモデルと使えないモデルがあり、特定 Stage の Codex レビューを一時スキップしたことがある。スキップした場合は記録ファイルに「Stage N スキップ（理由）」と 1 行残し、後から追えるようにしています。
* **純粋な docs 変更や設定ファイルだけの変更ではレビューを省く。** すべての Step で全 Gate を回すとコストが見合わない。リネームやフォーマットのみの commit は該当 Stage だけ実施する、というスキップ条件を明文化しています。
* **指摘数は意外と多い。** 実際の Step では 1 つあたり 25〜28 件の指摘が出ることもあった。大半は軽微だが、別モデルだからこそ拾えた「自分のモデルが見落としていた項目」が毎回数件は混じる。ここにクロスモデルの価値があります。

## まとめ

第2回の要点です。

> **コードを書いた本人にレビューさせない。観点を Stage で割り、project-aware な Claude と fresh-eyes な Codex を順次重ね、記録を独立させて統合判定する。別モデルだからこそ、単一モデルの盲点が埋まる。**

ポイントを並べ直すと:

* 4 Stage で観点を分離 → 各レビューが 1 観点に集中
* Claude（文脈を知る）→ 修正 → Codex（何も知らない）→ 統合の順次フロー
* 観点 ID 体系で指摘の根拠を固定
* 独立記録ファイルでレビュアーの独立性を守る
* サニタイズをスクリプトで構造化し、外部送信事故を防ぐ

次の第3回では、この「ルールで諭す + スクリプトで止める」をプロジェクト全体に広げた **rule + hook の多層防御**を扱います。プロンプトインジェクション対策（外部コンテンツは指示でなくデータ）や、エージェントの操作を記録する監査ログまで含めて、AI エージェントを安全に運用する設計をまとめます。
