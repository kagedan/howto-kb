## はじめに

[前回の記事](/nenene01/articles/hermes-agent-slack-codex-setup)で、自己改善型AIエージェント「Hermes Agent」をMacに常駐させ、Slackから使えるようにしました。

その後、Slack上のHermesにこんな相談を投げました。

> チーム開発で、共通リポジトリの設計・実装を行うとします。
>
> 各担当者にClaude Codeを払い出して利用する場合、同じ工程（API設計やUT開発）で同じモデルを使っても、チーム共通の構造化されたコンテキストを使うわけではないので、成果物がブレますよね。
>
> この課題を解決するために、Hermesを複数用意したり一本化したりすれば、生成物のブレを最小にできるのではないか？

返ってきた回答が実務的で面白かったので、対話を通じて辿り着いた\*\*「サンドウィッチ構成」\*\*というアーキテクチャを整理して記事にします。

なお、ここで挙げた「各担当者にAIを払い出すと、チーム全体の成果物がブレる」という課題感は、[AIエージェント前提の開発チームを5ヶ月運用して考えたこと](/nenene01/articles/team-scaling-ai-agents)という別記事で書いた実体験がベースになっています。あわせて読むと背景が伝わりやすいかもしれません。

## 課題：個別払い出しでは成果物がブレる

![各担当者にAIを個別払い出しすると成果物がブレるイメージ](https://static.zenn.studio/user-upload/deployed-images/2ad8d33ab355b78757e368bf.png?sha=00eeac7bf09475a172a8065b33c11dd604464058)

同じモデル・同じ工程・同じリポジトリでも、以下が違うと出力はブレます。

* 各担当者が与えるプロンプトと見せているコンテキスト
* ローカルの `CLAUDE.md` / skills / memory の状態
* 仕様の曖昧な部分をAIが勝手に補完する内容
* レビュー観点と「完了」の定義
* 人間側の暗黙知

たとえばAPI設計を各自がClaude Codeに依頼すると、エラー形式・paginationの書き方・認可や冪等性の考慮・OpenAPIの粒度・命名規則がバラつきます。UT開発なら、正常系中心の人、境界値を厚くする人、mockを多用する人、fixture設計の流儀が違う人……という差が出ます。

重要なのは、これが**モデルの性能不足ではなく、入力コンテキストと評価基準が揃っていないことによるブレ**だという点です。

ブレの原因は4つに分類できます。

1. **モデルの非決定性** — temperatureを下げても完全には揃わない
2. **コンテキスト差** — 誰が・どの時点で・どのファイルを読ませたか
3. **仕様の曖昧さ** — 曖昧に伝えた部分はAIが一般論で補完する
4. **評価基準の不在** — 「良いAPI設計とは何か」が構造化されていない

つまりゴールは「生成物を完全に固定すること」ではなく、\*\*「入力・手順・評価・改善ループを中央管理して、ブレを小さくし続けること」\*\*になります。

!

ここまでの論点は、あくまで **「中央管理すればブレは小さくできるはず」という仮説** の段階です。

本記事ではこの仮説を出発点にサンドウィッチ構成を設計し、中央管理そのものがボトルネックにならないか、ローカルの自由度をどこまで残すか、標準が形骸化しないか——といった点は、実際に運用してはじめて立ち上がる課題です。

これらは後半の「[運用上の注意点](#%E9%81%8B%E7%94%A8%E4%B8%8A%E3%81%AE%E6%B3%A8%E6%84%8F%E7%82%B9)」であらためて扱います。

## 構成パターンの比較

### A. 各エンジニアがClaude Codeを個別利用

```
Engineer A → Claude Code
Engineer B → Claude Code
Engineer C → Claude Code
```

導入は簡単で個人の作業速度は上がりますが、コンテキストが分散し、暗黙知が個人に溜まり、成果物品質が担当者依存になります。

**個人ブースターとしては良いが、チーム開発の標準化基盤としては弱い**構成です。

### B. 中央AI基盤に一本化

共通skills・共通コンテキスト・統一されたレビュー観点を持てる一方、権限設計・セッション混線・基盤メンテの負担が発生します。

また全員の作業を中央に通すと、待ち時間・並列作業・IDE連携・個人の作業スタイルの問題が出ます。

### C. サンドウィッチ構成（採用案）

対話の中で辿り着いたのがこの形です。※未検証状態です。今後、この案のワークフローを提案し検証を行いたいと考えております。

**個人AIは維持したまま、その前後に標準化レイヤーを挟みます。**

* **前段のHermes**が、個人AIに渡す入力コンテキスト（必読資料・設計原則・出力形式・禁止事項・完了条件）を標準化する
* **中段の個人AI**は各自の使い慣れたツールのままで、実装・生成に集中する
* **後段のHermes**が、成果物をチーム標準に照らしてレビューする

前段で揃え、後段で検査する。**個人の自由度とチーム標準化を両立**できるのがポイントです。Hermesは個人AIを置き換えるのではなく、「各個人AIに渡す前の共通コンテキスト生成器」かつ「品質基準管理者」——いわば**AI開発ガバナンス層**として配置されます。

## 運用フローの具体例

実際の1タスクはこう流れます。

```
Step 1: Engineer が issue / task を読む
Step 2: Hermes の工程別 profile に聞く
        「この issue を API 設計する前に、Claude Code に渡すべき標準コンテキストを作って」
Step 3: Hermes が標準化済みプロンプトを生成（参照資料・手順・制約・出力形式・完了条件）
Step 4: Engineer がそれを Claude Code に渡す
Step 5: Claude Code が成果物を生成
Step 6: 成果物を Hermes reviewer profile に戻す
Step 7: Hermes が標準との差分を指摘
Step 8: 必要に応じて skill / guideline を更新
```

### 前段が生成する標準プロンプトの例（API設計）

```
あなたはこのリポジトリの API 設計担当です。
以下の文書を必ず読んでから設計してください。

必読:
- docs/architecture.md
- docs/api-guidelines.md
- docs/error-format.md
- existing OpenAPI spec

制約:
- 既存 API の backward compatibility を壊さない
- pagination は cursor 方式
- error response は project standard（RFC7807 準拠）を使う
- 認可チェックは service 層で行う

出力:
1. endpoint summary
2. OpenAPI YAML diff
3. error cases
4. authorization matrix
5. required test cases

完了条件:
- OpenAPI validation が通る
- 既存 API との命名一貫性がある
- 互換性リスクが列挙されている
```

全員が同じprofileからこれを取得して個人AIに渡すので、**最初の入力コンテキストが揃い、出力のブレが構造的に減ります**。

### 改善ループ（ここが本命）

チーム標準化で一番大事なのは初期テンプレートではなく改善ループです。

```
成果物がブレる
  ↓
なぜブレたか分析する（prompt不足？ skill不足？ guideline不足？）
  ↓
skill / guideline / checklist を更新する
  ↓
次回から標準プロンプトに反映される
```

skillsは「AI用の標準作業手順書＋レビュー観点＋チームの暗黙知」であり、一度作って終わりではなく**チームの開発標準を育てるドキュメント**です。

このメンテナンスを中央化できることが、個別払い出しとの決定的な違いになります。

## Hermesでなくても代替可能では？

対話の中で私も突っ込みました。「サンドウィッチ構成は良いが、Hermesでなくても代替可能では？」と。

返ってきた整理が明快でした。

> 「標準プロンプトを作って渡すだけ」ならHermesでなくても代替可能。
>
> ただし、チーム開発の"標準化レイヤー"を**継続運用する基盤**として見ると差が出る。

比較軸はモデル性能ではなくレイヤーの違いです。

```
Claude Code / Codex = 個人が作業するための実装エージェント
Hermes              = チームがAI利用を運用するためのオーケストレーション基盤
```

![Hermesが優位になりやすいポイントの整理](https://static.zenn.studio/user-upload/deployed-images/015192f8051ad02fbe037241.png?sha=2a4b2362fb704aca488cd2bf3a6e3458091bcdc1)

Hermesが優位になりやすいポイント：

1. **モデル非依存** — 標準化ロジックをHermesに置けば、「API設計はClaude、テスト生成はCodex、大量レビューは安いモデル、機密はローカルLLM」と用途別に切り替えても標準化レイヤーを作り直さなくてよい
2. **profile分離** — 工程・役割ごとに model / skills / memory / 権限を分けた運用単位を作れる（api-standardizer、ut-standardizer、reviewer、security-reviewer…）
3. **チーム知の蓄積** — skills（手順・評価基準）、memory（安定した前提）、session\_search（過去の意思決定の全文検索）、curator（skillsの自動メンテ）が改善ループに直結する
4. **チーム接点の中央化** — Slack常駐＋cronで「毎朝未レビューPRの標準違反リスクを一覧化」「週次でレビュー指摘を集約しskill改善候補を出す」といった自動化を組める
5. **前段・後段の共通化** — 個人ごとに前段プロンプトを作るとそこがまたバラつく。中央のprofileなら全員が同じ入口を通る
6. **multi-agentレビュー** — API互換性・セキュリティ・テスト不足・パフォーマンスの多観点レビューをworkflowとして固定化できる
7. **コスト・認証・権限の中央管理** — 誰がどのモデルをどのコストで使い、何を投げているかが可観測になる

### 逆にHermesが過剰なケース

正直な整理として、以下なら不要です。

* 2〜3人の小規模チームで、ドキュメントとレビュー文化が既に強い → `AGENTS.md` + CI + PRテンプレートで十分
* 全員Claude Codeに完全統一する方針 → `CLAUDE.md` + slash commands + CI hooksで相当できる
* 標準化対象が「プロンプトテンプレートの配布」だけ → NotionやGitHub docsでよい

Hermesを入れる価値が出るのは、テンプレ配布ではなく**文脈生成・レビュー・履歴検索・改善・profile分離・自動化・多モデル化・チーム常駐**までやる場合です。

## 運用上の注意点

サンドウィッチ構成にも副作用はあります。

* **標準プロンプトが長くなりすぎる** — プロンプトが膨大になりすぎると重要情報が埋もれるかもしれない。
* **標準が腐る** — 正しく回るまで定期的なメンテナンスが必要だが、手を回せるかが肝になる。
* **個人AIが標準を無視する** — 後段レビューを行い、改善を回すが実施しない（無視する）エンジニアへの強制手段がない。
* **儀式化すると使われない** — 「毎回10分かかる儀式」になったら失敗。軽量なテンプレート生成に留めさせないといけない。
* **最終的な品質担保はCIと人間レビュー** — Hermesは標準化を助けるが、最終保証は人(エンジニア)によるレビューを介し保証させるべきである。

## まとめ

* チーム開発でAIの成果物がブレる原因は、モデル性能ではなく**入力コンテキストと評価基準の不揃い**
* 解決は完全中央集権でも個別最適でもなく、**前段（標準化）→中段（個人AI）→後段（レビュー）のサンドウィッチ構成**
* 個人AIは「実装・探索の加速」、Hermesは「標準化・文脈統制・評価基準・改善ループ」という役割分担
* Hermesの優位性は生成性能ではなく、**チーム全体でAIを使うときの文脈・手順・評価・改善・自動化を束ねられること**にある

これまで暗黙知によってまちまちだった成果物を、個人の自由度を殺さずに統一へ寄せていく。AIエージェントが1人1台の時代になったからこそ、「AIをどう使うか」自体をチームプロセスとして設計する必要が出てきたのだと思います。

## 次回に向けた宿題

本記事はHermesとの対話から設計した**仮説の段階**であり、実運用での検証はこれからです。記事中で挙げた課題・宿題を整理すると、次の3つに集約されます。

**① 実証タスク（次回記事のスコープ）**

* 工程別 profile（`api-standardizer` / `ut-standardizer` / `reviewer` / `security-reviewer` など）と、それに紐づく skills を整備する
* 小さなチームで実際にサンドウィッチ構成を回し、結果をレポートする

**② 検証すべき仮説**

* 中央管理そのものがボトルネックにならないか
* ローカル（個人AI）の自由度をどこまで残すべきか
* 標準が形骸化しないか

**③ 運用上の副作用への対策**

* 標準プロンプトの肥大化、標準の腐敗（メンテ owner）、標準の無視への担保、儀式化の回避、CI・人間レビューとの接続

次回は実際に工程別 profile と skills を整備し、小さなチームでこの構成を回した結果をレポートする予定です。
