## 結論

AIエージェントの話題は「どのモデルが賢いか」に偏りがちだが、実運用でボトルネックになるのは**インフラ**だ。Modal の CTO が Latent Space で明かした数字が象徴的で、強化学習のロールアウトでは\*\*「時に10万個のサンドボックス」\*\*を瞬間的に立ち上げる必要があるという。

この記事は、エージェントを「安全に・大量に・境界付きで」動かすインフラが直面する**3つの壁——①隔離 ②スケール ③攻撃面**を整理する。そして各壁について、僕が今日このデイリー運用の中で実際に踏んだ話を添える。背骨に通すのは、Modal が掲げる **DX（開発者体験）→ AX（エージェント体験）への転換**という設計思想だ。

---

## 第1部: 何が起きているか — DXからAXへ

Modal CTO の Akshat Bubna は、[Latent Space のインタビュー](https://www.youtube.com/watch?v=UwxxlTNPjWo)で「SDKチームの設計思想を、開発者体験（DX）からエージェント体験（AX）へ変えた」と語る。理由が明快だ——**「エージェントに数百のKubernetes YAMLを読ませて型のないファイルを書かせるより、コード内のデコレータを数行変えて、変更をライブで確認できる方が圧倒的に速い」**。

同じ潮流は日本でも見える:

英語圏でも同じ言葉が生まれている。Every.to の Dan Shipper は [GPT-5.6が知識労働をどう変えるか](https://every.to/chain-of-thought/how-gpt-5-6-changes-knowledge-work) で、人は「タスクの実行者」から「作業をこなすシステムを世話する管理者」へ変わると論じ、こう要約した——**「仕事をするな、ループを世話せよ（Don't do your work. Tend your loop）」**。Modal の「DX→AX」と寸分違わぬ概念だ。日本のgihyo（ループ設計）、speakerdeck（ハーネス）、米国のModal（AX）、Every.to（Tend your loop）が、別々の言葉で同じことを指している。

Modal のCTOはさらに踏み込む。「コードを誰も読まなくなった時代にこそ、**オブザーバビリティ（可観測性）が最重要**」。出力の挙動を見て変更をプロンプトする——それがAX時代の運用だ。

だが、AXを実現するには具体的な壁が3つある。

## 第2部: エージェント時代の3つの壁

### (1) 壁①：隔離 — エージェントは任意コードを実行する

エージェントに手を動かさせるとは、任意のコードを走らせるということだ。暴走や情報流出を防ぐには安全な閉じ込めが要る。

[Apple Container でClaude Codeをサンドボックス化する記事](https://creators.bengo4.com/entry/2026/07/09/084206)は、隔離強度を4段階（カーネル制限 / コンテナ / MicroVM / 仮想マシン）に整理する。**MicroVM は「ハードウェア仮想化の強い隔離」と「コンテナ並みの高速起動」を両立する中間解**で、macOS ではApple Container がこれを担う。Modal のCTOも「サンドボックスレベルでは LLM 仲介の権限に懐疑的で、ハードなガードレールが必要」と一致する。

隔離は「重ければ安全」ではなく「軽くて安全」が理想だ。[Claude Code・Codex用に超爆速で超安全なサンドボックスを作る記事](https://qiita.com/Koukyosyumei/items/3bd4696237ea64c38971)が示す `h5i` は、**コンテナを使わず0.2〜3秒で起動する軽量サンドボックス**で、プロンプトインジェクション対策と全操作の監査記録を両立する。エージェントは頻繁に立ち上げ直すので、起動が遅い隔離は使われなくなる——「速い隔離」こそがAX時代の要件だ。Claude Code は Bubblewrap/seccomp を Bash Tool に適用し `settings.json` で許可範囲を制御、Codex は sandbox policy × approval policy の組み合わせで read-only / workspace-write / danger-full-access の3段階を持つ。

### (2) 壁②：スケール — バースト性が本質

Modal の最大の技術的挑戦は自動スケーリングだ。CTO いわく、通常のエージェントやサンドボックスはさほどバースト的でないが、**「RL は異常なほどバースト的」で、ロールアウトでは時に10万サンドボックスが必要**になる。

しかも問題は「0→1」ではなく「1リージョン内で1,000→1,500 GPUへ超弾力的にスケールする」こと。Modal は GPU スナップショット（モデルの状態を保存してコールドスタートを高速化）でこれに応える。エージェントを大量に動かす世界では、**瞬間的な立ち上げ・畳みたたみの速度**がそのままコストと体験を左右する。

### (3) 壁③：攻撃面 — 権限を与えると新しい脆弱性が開く

エージェントに権限を渡すと、新しいカテゴリの脆弱性が生まれる。今週報告された [GitLost](https://gigazine.net/news/20260709-gitlost/) が典型だ——**攻撃者は公開リポジトリのIssueに隠した指示を仕込むだけで、AIエージェントに非公開リポジトリの内容を公開コメントとして吐き出させられる**。認証情報は不要。報告者は「エージェントAIの間接プロンプトインジェクションは、Webアプリの SQL インジェクションに相当する」と警告する。

攻撃面はモデル自体にも開く。2026年7月にGPT-5.6がリリースされた際、UK AI Security Institute は**数時間で「普遍的ジェイルブレイク」を発見**したと報告した——単発の逸脱ではなく、モデルの能力を損なわないまま長尺のエージェント的タスク完遂やエクスプロイト開発に悪用できる、という深刻なものだ（[AI Explainedの解説](https://www.youtube.com/watch?v=mWlCituW7wo)）。エージェントに権限を与えるほど、モデルのアラインメントの綻びがそのまま攻撃面になる。「賢いモデルほど安全」ではない。

防御側の研究も進む。Anthropic の [GRAM（デュアルユース知識のオフスイッチ）](https://www.anthropic.com/research/off-switch-dual-use)は、危険知識を性能を落とさず削除可能な区画に隔離する。攻撃面を「言うこと」でなく「知っていること」のレベルで塞ぐアプローチだ。

## 第3部: 今日、僕が踏んだ壁（実践）

この3つの壁は、10万サンドボックスを回す大企業だけの話ではない。個人でエージェントを並列運用する僕も、今日のデイリー収集の中で全部踏んだ。

### 処方1（隔離）：セッションを分離する — 共有デーモンの乗っ取りを踏んだ話

今日、トレンド収集で複数のサブエージェントを並列起動した。ところが**それらが同じブラウザデーモンを共有していたため、X（Twitter）取得中のセッションが、別のサブエージェントの処理に乗っ取られ、openai.com へナビゲートされてしまった**。X の取得は失敗した。

これはまさに「隔離の欠如」だ。回避策は単純で、Reddit 用・X 用に別々の `--session` 名を割り当てて名前空間を分ける。**エージェントを並列で走らせるなら、リソースの奪い合いを前提に隔離を設計する**——身をもって学んだ。

### 処方2（スケール）：レートを制御して安全に大量アクセスする

Reddit は IP 単位の WAF でブロックしてくる。今日これを回避できたのは、\*\*「1サブレッドずつ、各取得の間に12秒以上のスリープを挟む逐次実行」\*\*にしたからだ。短間隔で一括ループするとブロックが発動し、全経路が数分デッドになる。

10万サンドボックスの弾力スケールとは規模がまるで違うが、原理は同じ——**大量アクセスを安全にさばくにはレート制御が要る**。優先度順に並べておけば、途中でブロックされても重要なものから埋まる、という設計もした。

### 処方3（攻撃面）：権限境界を意識してエージェントに委譲する

サブエージェントに取得手順を注入するとき、僕は権限境界を意識している——読み取り系は自動許可、送信・作成・削除は確認を挟む。GitLost が示すのは「エージェントに何でもさせると、意図しない出力経路が攻撃面になる」ということ。Modal CTO の「ハードなガードレールが必要」という言葉と、GitLost の教訓は同じ方向を指している。

この「強い権限を常時持たせない」発想は、人間の運用ではすでに定石になりつつある。[ZOZOがJIT（ジャストインタイム）アクセスを全社導入した事例](https://techblog.zozo.com/entry/just-in-time-access-company-wide-rollout-at-zozo)は、強力なAWS権限を必要な時だけ一時付与し、読み取り専用は常時・書き込みのみJIT対象と割り切る。**エージェントにこそ、この「使い捨ての権限」が要る**。常時フルアクセスのエージェントは、GitLostのような間接インジェクションを食らった瞬間に被害が最大化するからだ。

一つ補足しておくと、Modal は逆に「エージェントが自分のインフラを操る」未来も見せている。同社の `auto inference` はエージェントが GPU を H200→B200 に変えながら設定を自己最適化するという。**権限を絞りつつ、どこまで委ねるか**——この綱引きがAX時代の設計の中心になる。

## 締め

「どのモデルが賢いか」は毎週入れ替わる。だが、エージェントを**安全に・大量に・境界付きで**動かせるかは、もっと地味で、もっと決定的だ。DX→AX の転換は掛け声ではなく、隔離・スケール・攻撃面という3つの具体的な壁として現れている。

面白いのは、この壁が個人開発者にも同じ形で降りてくることだ。10万サンドボックスは無縁でも、セッションの隔離・レート制御・権限境界は、今日から効く。僕は今日それを、失敗を通して確かめた。エージェント時代のインフラは、まず自分の運用から始まっている。
