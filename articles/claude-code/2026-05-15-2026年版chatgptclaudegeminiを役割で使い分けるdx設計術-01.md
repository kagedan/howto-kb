## はじめに：「最強の1社」を選ぶのをやめた話

地方の中小製造業（アルミダイカスト）で、一人でDX推進をしているやまとです。

生成AIを使い始めたとき、ベンチマーク記事を読み漁って「今月最強はこれ」と1社に絞ろうとしていました。Sonnet vs GPT vs Geminiのスコア比較、コードベンチでの優劣、推論コスト…。

でも、実際に業務で使ってみると、不思議とそういう順位通りには進まないんですよね。

* 「最強」とされたモデルに頼んでも、業務文脈ではうまく出てこない
* 別の社のAIにそのまま投げたら、あっさり通る
* ベンチマーク上の順位と、業務での「使えた感」がズレる

そこからたどり着いたのが、「最強を選ぶ」のをやめて **「役割に配置する」** という発想です。

本記事では、製造業DX一人担当の現場感覚をベースに、生成AI3社（ChatGPT / Claude / Gemini）を **業務フェーズに割り当てるための設計術** をまとめます。具体的にはこんな構成です。

* 「最強選び」と「役割配置」の **As-Is / To-Be 分析**
* 業務フェーズ × 3社の **Fit & Gap 分析**
* 製造業DX現場での **実際の配置例**（Mermaid 図つき）
* 明日からセットアップする **TODOチェックリスト**

## 1. 結論：生成AIは「比較」ではなく「配置」する

先に結論だけ書きます。

> **生成AIは「ChatGPT vs Claude vs Gemini」で勝者を選ぶものではなく、業務フェーズに『配置』するもの。**

理由はシンプルで、3社とも「日常実務全般」を担えるほど汎用化していて、かつ「明確な得意領域」を持っているからです。1社で全部やろうとすると、不得意領域に無理させる時間が増えて、結局トータルで遅くなる。

設計の観点で言い換えると、

* **ChatGPT** ＝ 個人の日常実務を横断する「総合アシスタント」
* **Claude** ＝ 深く考えて長く動く「思考のエース」
* **Gemini** ＝ Google Workspace を貫く「業務基盤」

この3つを **マイクロサービス的に分担させる** イメージです。1つのモノリスではなく、責務ごとに分けたサービスにルーティングするほうが、運用も品質もスケールしやすいのと同じ理屈ですね。

「3社全部を契約・運用する」ことに抵抗があるかもしれません。私もそうでした。でも、強み領域がはっきりしているので、使い分け自体は思ったよりシンプルです。

次のセクションから、その根拠と実例を書いていきます。

## 2. As-Is / To-Be 分析：「最強選び」から「役割配置」へ

まず、「最強1社で全部やる」というよくあるアンチパターンと、「役割配置」の To-Be を並べて比較します。

| 観点 | As-Is（最強1社で全部やる） | 課題 | To-Be（役割で配置） |
| --- | --- | --- | --- |
| プロセス | 1つのAIに全タスクを投げる | 不得意領域でも無理に使う | 業務フェーズごとに最適AIを割当 |
| 時間 | プロンプト調整に時間消費 | 同じ品質に到達するのに時間がかかる | 各AIの強み領域で一発で出やすい |
| コスト | 1社のプラン上限に到達 | レート制限・トークン上限の影響大 | 3社で負荷分散、月額の最適化が可能 |
| 品質 | 不得意領域で品質低下 | 出力のばらつきが大きい | 強み領域に振るので品質が安定 |
| 体制 | ソロで全部抱える | スケールしない | 仮想チーム化（要件・設計・実装担当） |
| 学習曲線 | 1社のクセに過剰最適化 | プロバイダ依存 | 各社の思想を比べて理解が深まる |

ポイントは、**To-Be の方が「使うAIの数が多くてコスト高」とは限らない**ことです。

たとえば私の場合、Claude Code の月額上限を3日で焼き切る勢いで使っていた時期があります。これを以下の流れに分けてみました。

* 要件整理 → ChatGPT
* 設計実装 → Claude
* Sheets 整形 → Gemini

結果、Claude の使用量は減って、全体の月額はむしろ抑えられました。

「最強選び」は、結果として **1社のレート制限・サブスク上限を律速にしてしまう** 設計でもあるんですよね。

## 3. 3社の強みを「役割」で整理する

ベンチマーク順位ではなく、**「業務文脈での担当領域」** で3社を整理し直します。

### Mermaid①：3社の役割マッピング

### ChatGPT：日常実務の総合アシスタント

ChatGPT の強みは、**個人の業務全体を横断してサポートできる粒度の高さ** です。

2026年5月時点で、ChatGPT は過去チャット・保存メモリ・接続したGmailやファイルから関連文脈を引き出して、継続性のある回答を返せるようになっています。OpenAI公式の「Memory FAQ」によると、ChatGPT Plus / Pro では、過去チャット・保存メモリから関連情報を引き出して応答をパーソナライズする機能が有効になっています([OpenAI Help Center: Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq))。

開発寄りだと **Codex memories** という機能もあり、過去スレッドから技術スタックやプロジェクトの慣習を抽出して、次回以降のセッションに渡せるようになっています([OpenAI Developers: Codex memories](https://developers.openai.com/codex/memories))。Codex のホームディレクトリ配下にメモリファイルが保存され、スレッド単位で参照・生成のオン/オフを制御できる構造です。

**ChatGPTに配置したい仕事**

* 業務調査・情報収集（出典つき要約まで）
* ブログ / 資料 / 提案書の構成案づくり
* コードのたたき台作成
* エラー調査の初動相談
* 議事録・報告書のドラフト
* 過去のやり取りを踏まえた継続提案
* Gmail やファイルの文脈を活用した業務処理
* 1人ブレストの壁打ち相手

私のイメージだと、ChatGPT は **「いちばん最初に相談する人」**。最初の一手を早く出したいとき、迷ったときに脳内をダンプする相手として、毎日使っています。

### Claude：深く考えるエース

Claude の強みは、**長く・深く考える力と、作業から学習する方向への進化** です。

Anthropic は2026年5月6日、Claude Managed Agents 向けに **dreaming** という機能をリリースしました。エージェントが過去の作業・メモリを見返し、失敗パターンや回避策、ツール固有のクセを「playbook」として整理して、次のセッションに渡す仕組みです([Anthropic: New in Claude Managed Agents](https://claude.com/blog/new-in-claude-managed-agents))。

注目されたのが、法律AIスタートアップ Harvey での導入結果で、**特定のワークフロー** において dreaming を有効化したらタスク完了率が約6倍になった、という事例です([Anthropic 公式](https://claude.com/blog/new-in-claude-managed-agents) / [VentureBeat 解説](https://venturebeat.com/technology/anthropic-introduces-dreaming-a-system-that-lets-ai-agents-learn-from-their-own-mistakes))。

数字の派手さよりも、私が本質的だと感じているのは **「終わったあとに振り返って学ぶ」方向への進化** という点です。これまでの生成AIは「質問 → 回答 → 終わり」で、次に同じ質問が来てもまたゼロから考え直していました。dreaming は人間の新人が「失敗を覚えて、同じところでつまずかなくなる」のに近い動きをします。

現場に置き換えると、こういう「現場ごとの細かいクセ」を Claude 側に蓄積していけるイメージです。

* このプロジェクトでは、このライブラリのバージョンに注意する
* この社内環境では、このコマンドが通らないので別のやり方を使う
* この上司には、この形式の報告書のほうが通りやすい

**Claudeに配置したい仕事**

* 長く動く開発タスク（要件整理から実装まで一気通貫）
* 複雑な文書作成（仕様書 / 設計書 / 企画書）
* コード設計・リファクタリングの相談
* 業務プロセスの改善・再設計
* 失敗・成功のナレッジを蓄積して使い回す運用

Claude は **「深く考えてほしいときに頼むエース」**。コードを書くだけでなく、「なぜこの設計なのか」「どこにリスクが残るか」を一緒に詰めてくれる相手として配置しています。

### Gemini：Google Workspace の業務基盤

Gemini の強みは、**Google エコシステムとの一体感** です。

Google Cloud は2026年4月22日、**Gemini Enterprise Agent Platform** を一般提供開始しました([Google Cloud Blog: Introducing Gemini Enterprise Agent Platform](https://cloud.google.com/blog/products/ai-machine-learning/introducing-gemini-enterprise-agent-platform))。その中核機能のひとつが **Memory Bank** で、複数セッションを横断して会話文脈を保持・検索できる仕組みです([Google Cloud Docs: Agent Platform Memory Bank](https://docs.cloud.google.com/gemini-enterprise-agent-platform/scale/memory-bank))。

特に効いてくるのは、Google Workspace がメイン基盤の会社で、Drive / Sheets / Gmail / Docs / Slides に蓄積されたデータをそのまま「コンテキスト」として扱える点です。

**Geminiに配置したい仕事**

* Drive 配下の資料を横断した整理・要約
* Gmail の過去文脈を踏まえた業務処理
* Sheets のデータ分析・自動化
* Docs / Slides と連携した資料生成
* 社内データを使った業務エージェント構築
* Google Workspace 中心の会社の業務効率化

私の感覚だと、ChatGPT と Claude が「人の代わりに考える」方向だとすれば、Gemini は **「業務システムの中で動く基盤型の生成AI」**。スプレッドシートで生産管理データを扱っているような現場では、相性の良さが際立ちます。

## 4. 業務フェーズ別シーケンス：どこで誰に投げるか

ここからは、実際の業務フローに沿って「どのタイミングで誰に何を投げるか」を可視化します。

### Mermaid②：要件 → 設計 → 実装 → 運用 のシーケンス

このフローのポイントは、**「思考の深さ」が必要なフェーズと「データの広さ」が必要なフェーズを、別のAIに分けている** ことです。

* ChatGPT は「何を作るべきか」の段階で広く浅く動く
* Claude は「どう作るか」「どこにリスクがあるか」を深く詰める
* Gemini は「実データ」と「業務システム」の橋渡しを担う

そして最終判断と本番反映は人間が握る。ここは外せない一線だと思っています。

## 5. Fit & Gap 分析：業務フェーズ × 3社の適合度

各業務フェーズに対する3社の適合度を、Fit / Partial / Gap の3段階で評価します。

| 業務フェーズ | ChatGPT | Claude | Gemini | 推奨 |
| --- | --- | --- | --- | --- |
| 要件整理・初期ヒアリング整理 | **Fit** | Partial | Partial | ChatGPT |
| 構成案・たたき台づくり | **Fit** | Partial | Gap | ChatGPT |
| 情報収集・出典付き要約 | **Fit** | Partial | Partial | ChatGPT |
| 複雑設計・例外ケース検討 | Partial | **Fit** | Gap | Claude |
| 長文ドキュメント執筆 | Partial | **Fit** | Partial | Claude |
| コード設計・リファクタリング | Partial | **Fit** | Gap | Claude |
| コードレビュー | Partial | **Fit** | Gap | Claude |
| 失敗ナレッジの蓄積（dreaming） | Gap | **Fit** | Partial | Claude |
| Google Workspace 連携処理 | Gap | Gap | **Fit** | Gemini |
| Sheets 集計・分析 | Partial | Gap | **Fit** | Gemini |
| Drive 横断ドキュメント整理 | Gap | Gap | **Fit** | Gemini |
| Gmail 文脈を踏まえた返信ドラフト | Partial | Gap | **Fit** | Gemini |
| 最終意思決定・リリース判断 | Gap | Gap | Gap | **人間** |

### 評価の読み方

* **Fit**：その業務に対して、現状の機能でほぼ過不足なく対応できる
* **Partial**：可能だが、別のAIや人間の補助が前提になる
* **Gap**：現状の機能では不向き、もしくは非効率

### 総合判断

> **配置の最適解は「業務フェーズ単位でAIをスイッチすること」**。

理由：

* 必須要件（思考の深さ / データの広さ / 業務基盤連携）は1社ですべて満たすのが難しい
* 1社最強型はサブスク上限・レート制限がボトルネックになりやすい
* 3社配置は「仮想チーム」として運用でき、属人化を抑える効果もある

注意点：

* 「最終意思決定・リリース判断」は **すべての社で Gap** と評価しています。ここを「Claude にFit」と書きたくなる気持ちはわかりますが、リリース責任を取れるのは人間だけ、という前提は崩さない
* 業務によっては Gemini と Workspace の組み合わせが圧倒的に効くので、業種・基盤次第で重みが変わる
* ベンダー側の機能アップデートで Fit/Gap は半年単位で動くので、四半期に1回は見直す前提で

## 6. 実践：製造業DX一人担当の現場での配置例

「で、結局どう使い分けてるの？」という話を、現場ベースで書きます。

私の職場は、地方中小の製造業（アルミダイカスト）です。IT環境をゼロから整備して、生産管理システムの自社開発（バックエンド：Python + FastAPI、フロント：Next.js + TypeScript）、IoTセンサー導入、ダッシュボード整備、NASサーバー構築、外注ベンダー管理までを一人で担当しています。

人を増やせるわけではないけれど、やる仕事は増えていく。そんな「一人DX担当」が、生成AIをどう配置しているか。

### Mermaid③：仮想チーム編成図

### フェーズごとの実例

**【要件整理・情報収集】→ ChatGPT**

新しい社内システムを作るとき、まず ChatGPT に「この業務課題をどう整理すると上司に通りやすいか」を壁打ちします。

たとえば「鋳造ラインの不良率トラッキングを自動化したい」というふんわりした依頼を、誰が・いつ・どんな入力をして・どんな出力が必要か、というレベルまで分解させる。出てきた構成案を自分でレビューして、実装に落とせる粒度に仕上げる。

最初の一手の速さは、ChatGPT に投げるのが一番噛み合います。

**【設計・実装】→ Claude**

「この業務ロジック、こんな例外ケースがあるんだけど、どう設計する？」みたいな複雑な問いは Claude に。

要件整理の結果を Claude に渡して、設計判断と実装まで一気通貫で進める。出力されるコードはそのまま使うのではなく、責任を自分で持つ前提でレビューします。

報告書系の「文書設計」も Claude に深く考えさせると、構成の質が一段上がります。「上司の意思決定に必要な情報を、A4一枚に収める」みたいな制約付きタスクで強い。

**【データ活用・Google連携】→ Gemini**

生産管理データの一部は Google スプレッドシートで管理しているので、Gemini との相性がいい領域です。

複数シートを横断した集計、ピボットの組み替え、定期報告のテンプレート整形は、Gemini に任せると手間が大きく減ります。Memory Bank が乗ったあとは、特定のシート構造や報告フォーマットを覚えてくれるので、「毎週同じ作業」を渡すのがより楽になりました。

**【最終判断・リリース責任】→ 自分**

どのAIが書いたコードでも、本番に出す責任は自分が持つ。これだけは絶対に外せない一線です。

社内の業務システムが落ちたとき、AIに頭を下げさせるわけにはいかない。最後のレビューと意思決定は、人間がやる仕事として残しています。

### 配置による変化

この配置にしてから、いちばん変わったのは **「作業に追われる時間が減って、考える時間が増えた」** ことです。

* 要件定義担当 → ChatGPT
* 設計・実装担当 → Claude
* データ活用担当 → Gemini
* 意思決定・責任 → 自分

一人でも、「ちょっとした開発チームを持っている」感覚で動けるようになりました。

ベンチマーク順位を追っていた頃と比べて、3社契約の合計コストは増えています。でも、それを上回るスループットの伸びがあり、何より「不得意領域に無理させない」分、出力品質のブレが減ったのが大きいです。

## 8. AI指揮者になるための4つの力

3社を使い分けるようになると、人間側の役割も少しずつ変わってきます。

> **これまで：** 自分で調べ、自分でコードを書き、自分で報告書を書く  
> **これから：** 生成AIに分担させて、自分は判断・意思決定・責任の部分を握る

このとき必要になる力を、エンジニア向けに4つに整理してみました。

### ① ゴール設定力（＝要件定義能力）

生成AIは「作業」はできても、「何を目指すべきか」は決められません。

「この機能を作って」と投げるのではなく、「この業務課題を解決するために何を作るべきか」を自分で言語化する力が、いままで以上に効いてきます。これは要件定義そのもの。

### ② タスク分解・配置力（＝アーキテクチャ設計能力）

ChatGPT に任せる仕事、Claude に任せる仕事、Gemini に任せる仕事、自分で握る仕事を分ける。

これはマイクロサービスの責務分担、あるいはチーム編成と同じ思考です。コードを書く時間より、ここに使う時間のほうが、長期的にはROIが高い。

### ③ 出力検証力（＝コードレビュー＋ファクトチェック能力）

生成AIは優秀ですが、平気で存在しない仕様・API・関数を出してきます。私も「Claude が出したコードがそのまま通る」前提でレビューを軽くしたら、本番直前で破綻したことがあります（あれは堪えました）。

「これは本当に正しいか」を見極める力 ── つまり静的解析・テスト・ドメイン知識ベースのレビュー ── は、AI時代にむしろ価値が上がる領域です。

### ④ 判断・責任能力（＝リリース判断能力）

コードを書いても、本番リリースの責任は生成AIには取れません。報告書を社内に出す責任、顧客に提案する責任 ── これらは最終的に人間が持つしかない。

「AIに仕事を奪われるか」よりも、「**AIチームを指揮する側に回れるか**」のほうが、これからの分岐点になると感じています。

## 9. まとめ

改めて3社の役割を整理します。

* **ChatGPT** — 日常実務の総合アシスタント（調査・構成・たたき台・壁打ち）
* **Claude** — 深く考えるエース（複雑設計・長文・開発タスク・ナレッジ蓄積）
* **Gemini** — Google 業務基盤担当（Drive / Sheets / Gmail / Docs / Slides）

3社の進化の向かう先には、共通点があります。

> **生成AIは「質問に答える道具」から「仕事を進めるエージェント」へ進化している。**
