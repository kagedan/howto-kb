## はじめに

一時的に使えなくなっていたClaude Fable 5が、また使えるようになっていました。

Claude Codeを起動すると、「Fable 5 is back.」と表示されています。

![Claude CodeでFable 5 is backと表示された画面](https://static.zenn.studio/user-upload/deployed-images/04eca5f7cc666c09a3d3f5d2.png?sha=b0ddbf9b7bd63343b1d0ae13a7b947aa658f9edf)

以前の記事でも書いたとおり、Fable 5はリリース直後に使えるようになったものの、その後アクセス停止になっていました。

表示を見る限り、2026年7月7日までは課金範囲内(=追加料金なしで、プランの利用枠の中)で利用でき、それ以降は追加料金がかかるようです。つまり、また「今のうちに重めの作業を進めておく期間」が来たということです。

この記事は、Fable 5が使えるようになってからやったことの記録です。

## 何をやったか

今回Fable 5で進めたのは、派手な新規開発ではなく、今回も環境整備です。

具体的には、以下をやりました。

* Skillsの作成
* Skillsのアップデート
* Sonnet 5をサブエージェントとして使ったサイトデザインのアップデート
* Sonnet 5をサブエージェントとして使ったプレゼン資料テンプレートのアップデート

Fable 5はトークン消費が大きめなので、普段から気になっていたものの、後回しにしていた作業をまとめて片付けることにしました。

特にサイトや資料テンプレートのような作業では、Fable 5だけで全部を進めるのではなく、Sonnet 5をサブエージェント(親エージェントから作業を任される下請け役)として使う形にしました。全体方針やレビューはFable 5に任せつつ、実装や具体的な修正はSonnet 5にも分担させる、という使い方です。

このあたりについては、Xの以下のポストを参考にしました。

設定では、サブエージェントのモデルをFable以外に固定しました。

```
CLAUDE_CODE_SUBAGENT_MODEL=claude-sonnet-5 claude \
  --append-system-prompt "基本的にタスクや作業の実行は、適切な粒度でsubagentsに実行手順が明確な指示を与えて委譲すること。あなたは全体進行の俯瞰と立案を行う。自己判断による例外は認める"
```

ポイントは2つです。

* `CLAUDE_CODE_SUBAGENT_MODEL`でサブエージェントを`claude-sonnet-5`に固定する
* `--append-system-prompt`で、Fable 5には全体進行と立案を主に担当させる

これで、Fable 5は最初から「自分で全部やる」のではなく、適切な粒度でSonnet 5へ作業を投げる前提になります。

## Skillsの作成

まずやったのは、新しいSkillsの作成です。

既存の作業を何度か繰り返していると、「これは毎回プロンプトで説明するより、Skillにした方がよい」と思うものが出てきます。

特に、以下のような作業はSkill化しやすいです。

* 毎回同じ調査手順がある
* 出力フォーマットを固定したい
* 判断基準を明文化したい
* 失敗しやすい注意点がある

Fable 5に作らせると、単に手順を並べるだけではなく、前提条件、確認方法、失敗時の扱いまで含めて組み立ててくれるのが助かります。

人間が雑に「この作業をいい感じにSkillにして」と投げても、使う側のエージェントが迷いにくい形まで落とし込んでくれる印象です。

## Skillsのアップデート

次に、既存のSkillsのアップデートも進めました。

Fable 5が前回使えたときにもSkillsの改善を進めていましたが、使っているうちに、また改善したいところが出てきます。

たとえば、次のようなものです。

* 指示が曖昧で、毎回少し違う判断をする
* 本当は重要ではない指摘が強く出る
* 実際の運用とスキル内の手順がずれている
* 出力フォーマットが用途に対して少し読みにくい

こういう修正は、地味ですが効果が大きいです。

Skillは一度直すと、その後の作業すべてに効いてきます。記事作成、PRレビュー、ログ分析、資料作成など、毎回使うものほど改善効果が積み上がります。

Fable 5に任せて良かったのは、単に文章をきれいにするだけでなく、「このルールは実態と合っているか」「この指摘は自動修正してよいのか」といった運用上の副作用まで見に行ってくれるところでした。

前回も感じたことですが、こちらが指示していない改善まで勝手に足してくる感覚は今回もありました。たとえば「この項目を直して」とだけ頼んだのに、関連する別のルールの整合性や、想定していなかったエッジケースまで一緒に手を入れてくる、といった具合です。良かれと思って踏み込んでくるので、便利な反面、意図した範囲を超えていないかは毎回こちらで確認する必要があります。

## サイトデザインのアップデート

サイトデザインのアップデートにも使いました。

デザイン作業は、Fable 5の良さが分かりやすく出る領域だと思います。

ただし今回は、Fable 5にすべて直接書かせたわけではありません。サイトの構成や改善方針はFable 5に見てもらい、具体的な実装や修正ではSonnet 5をサブエージェントとして使う形にしました。

「もう少し今っぽく」「読みやすく」「情報量は落とさずに整理して」といった曖昧な指示でも、余白、文字サイズ、色、コンポーネントのまとまり方をかなり自然に調整してくれます。

実際の進め方としては、Fable 5に「このサイトを見て、古く見えるところと直す順番を出して」と投げ、出てきた修正方針をもとにSonnet 5へ実装を委譲させました。Fable 5にはレビュー役に回ってもらい、修正後の画面を見て「まだ詰まって見える」「ここの余白は広げた方がよい」といった確認をさせています。

特に良かったのは、単に見た目を派手にするのではなく、既存の構成を大きく壊さずに整える方向で提案してくれたことです。

デザインをゼロから作るより、既存サイトの違和感を直す作業の方が難しいことがあります。Fable 5はそのあたりの「今あるものを読んで、崩さずに良くする」力が強いと感じました。

一方で、細かい修正を全部Fable 5にやらせると少しもったいないです。方針決めやレビューにFable 5を使い、手を動かす部分をSonnet 5に任せると、品質とコストのバランスが良くなります。

## プレゼン資料テンプレートのアップデート

プレゼン資料のテンプレートも見直しました。

資料テンプレートは、一度作ると何度も使います。だからこそ、最初の設計が雑だと、毎回少しずつ使いにくさが出てきます。

ここでも、Fable 5をメインにしつつ、Sonnet 5をサブエージェントとして使いました。Fable 5にはテンプレート全体の方向性や品質確認を任せ、Sonnet 5にはスライドごとの調整や具体的な反映を担当させる形です。

今回見直したのは、主に次のような部分です。

* タイトルスライドの見せ方
* セクション区切りのデザイン
* 箇条書きの密度
* 図表を置いたときの余白
* 発表資料としての読みやすさ

Fable 5は、資料の「見た目」だけでなく、「話す順番」や「どこで視線を止めるか」まで含めて提案してくれるのが良かったです。たとえば、タイトルスライドを派手にするよりも、セクション区切りで一度視線を止める構成にした方が発表では使いやすい、というような見方をしてくれました。

テンプレートは使い回すものなので、ここを直しておくと今後の資料作成がかなり楽になります。

サイトデザインと同じで、Fable 5を常に作業者として使うより、レビュー役や設計役に置いた方が向いている場面があります。Sonnet 5をサブエージェントにすることで、重い判断はFable 5、具体作業はSonnet 5という役割分担にできました。

## なぜ今やるのか

理由はシンプルで、2026年7月7日まで課金範囲内で使えるからです。

Fable 5は便利ですが、トークン消費は軽くありません。普段のちょっとした修正に毎回使うというより、重めの設計、棚卸し、テンプレート整備のような作業にまとめて使うのが合っている気がします。

特に今回やったような環境整備は、成果がその場限りではありません。

* Skillを直すと、今後のAI作業の精度が上がる
* Skillを作ると、繰り返し作業が減る
* サイトデザインを直すと、公開物の見え方が良くなる
* 資料テンプレートを直すと、次回以降の資料作成が楽になる
* Sonnet 5をサブエージェントとして使うと、Fable 5の利用量を抑えながら品質を上げやすい

追加料金がかかるようになる前に、こういう積み上がる作業を優先しておくのが良さそうです。
