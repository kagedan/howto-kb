## この連載について

最初に正直なところを書くと、個人開発でいちばん時間を溶かしたのはコードそのものより、エージェントと「さっき決めたこと」が食い違う場面でした。

Claude Code や OpenAI Codex のようなコーディングエージェントを使い始めると、最初の感動の次に必ずこの壁が来ます。

* 会話が伸びるとコンテキストが自動圧縮され、さっき決めたことが微妙にズレて戻ってくる
* 「いい感じに直して」を繰り返すうちに、設計判断の根拠がどこにも残らない
* エージェントは自信満々だが、本当に品質が担保されているのか確信が持てない

私はこの半年、\*\*Rust + Dioxus で個人資産可視化アプリ（Wealth Compass）\*\*を作りながら、これらに対処するための運用設計を組み上げてきました。アプリ本体より運用設計の方が学びが多かったので、3 回の連載にまとめます。

* **第1回（本記事）**: 開発フロー全体を Phase/Stage/Gate で構造化し、状態をファイルに逃がすことで `/clear` に強くする設計
* **第2回**: Claude と OpenAI Codex を独立した 2 人のレビュアーとして使うクロスモデル 2 段レビュー
* **第3回**: rule と hook を二重化してエージェントを安全に運用する多層防御

対象読者は、すでに Claude Code や OpenAI Codex を日常的に使っている方です。「エージェントに何を任せ、何を構造で縛るか」に関心がある人を想定しています。

## 出発点: コンテキストは「揮発する作業メモリ」だと割り切る

エージェント開発でいちばん最初に内面化すべきだと感じたのは、これです。

> **会話コンテキストは信頼できる永続ストレージではない。揮発する作業メモリだ。**

長い会話は自動圧縮（auto-compaction）される。圧縮は要約なので、必ず情報が落ちる。「さっきの設計判断でこう決めたよね」がコンテキストの彼方に消え、エージェントが過去の自分と矛盾した実装を平気で出してくる。これはモデルの賢さの問題ではなく、**状態をコンテキストに置いている設計の問題**です。

だから設計の出発点はこうなります。

> **判断・進捗・証跡の「正本」は、すべて git 管理のファイルに置く。コンテキストはそれを読んで再構築する一時バッファにすぎない。**

この割り切りができると、`/clear`（コンテキスト全消去）が怖くなくなります。むしろ積極的に使える。ここが本記事の肝です。

## 階層: Phase / Task / Step / Stage / Gate

まず開発全体を 5 階層に分解しました。粒度を決めておくと「今どの単位の話をしているか」がブレません。

```
Phase  … プロジェクトを貫く大目標（1〜6）。例: Phase 5 = 品質改善と運用基盤強化
 └ Task   … Phase 内の中目標。例: 5-A アプリ側リファクタリング
    └ Step   … 1 commit に対応する作業単位。例: 5-A-1 重複解消
       └ Stage  … Step 内の 4 段階（設計 → テスト設計 → 実装 → テスト実施）
          └ Gate   … 各 Stage 完了時のレビュー関門
```

ポイントは **Step = 1 commit** という対応です。Step を「単体で動作・テストが完結する粒度」に切ることで、1 つの論理変更が 1 コミットに収まり、レビューも履歴も読みやすくなる。

Stage と Gate は第2回（クロスモデルレビュー）で深掘りするので、本記事では「Step の中はさらに 4 段階に割れていて、各段階にレビュー関門がある」とだけ押さえてください。

## 状態をどこに置くか: ドキュメント駆動の正本管理

階層を決めたら、次は「各階層の状態をどのファイルが持つか」を固定します。これが `/clear` 耐性の本体です。

| ファイル | 役割 | 正本とするもの |
| --- | --- | --- |
| `docs/STATUS.md` | 現在地を指す薄い索引 | 「今どの Phase か」だけ。中身は持たない |
| `docs/phaseN/tasks.md` | 各 Step の進捗状態を管理する表 | 各 Step の状態（pending / completed） |
| `docs/decisions.md` | 設計判断ログ（D-1, D-2, …） | 「なぜその設計にしたか」 |
| `docs/refactoring.md` | リファクタリング候補ログ | 「後で直したい」の全件 |
| `docs/phaseN/design.md` | Phase の設計 | 設計判断の詳細 |
| `docs/phaseN/test-plan.md` | テスト設計 | どのケースを検証するか |
| `docs/phaseN/test-evidence.md` | テスト実施記録 | 実行結果・カバレッジ・バグ対応の証跡 |

設計の意図は 3 つあります。

**1. 索引と中身を分離する。** `STATUS.md` には現在地への参照だけを置き、状態の中身を持たせない。こうすると、進捗が変わるたびに索引を書き換える必要がなく、ドリフト（索引と実態の乖離）が起きにくい。

**2. 判断の「なぜ」を必ず残す。** `decisions.md` に設計判断を D-N として番号付きで積む。後から「なぜここで Decimal を使ったんだっけ」を追える。エージェントにもこのファイルを読ませることで、過去の自分と矛盾しない実装を書かせる。

**3. 証跡を残してサボりを防ぐ。** `test-evidence.md` にテスト 1 件ごとの実行結果を記録する。「バグなし」の Step も省略せず書く。これは未来の自分とエージェントの両方に対する「ちゃんとやった」の担保です。

重要なのは、これらが**すべて git 管理**である点です。コンテキストが揮発しても、git には残る。`/clear` で会話を捨てても、ファイルを読み直せば判断の履歴は完全に復元できます。

## 自動化: skill で「思い出す」をなくす

正本ファイルを決めても、人間（とエージェント）はルールを忘れます。「Step 完了時に test-evidence を更新する」を毎回主体的に思い出すのは非現実的。そこで Claude Code の \*\*skill（スラッシュコマンド）\*\*で手順を固定します。

私が使っている主な skill はこれです。

| skill | タイミング | やること |
| --- | --- | --- |
| `/phase-start <N>` | Phase 着手時 | design/tasks の雛形生成、関連 rules の参照表示 |
| `/step-start <id>` | Step 着手時（`/clear` 直後） | 前提コンテキストを決定論的に再構築 |
| `/step-complete` | Step 完了時 | コード品質 + ドキュメント整合性チェック |
| `/check` | コミット前 | fmt → lint → test を順に実行 |
| `/commit` | コミット時 | Conventional Commits 形式で作成 |
| `/phase-end <N>` | Phase 完了時 | retrospective 雛形、refactoring 突合、総合レビュー |

skill の役割は「**決定論的にやるべきことを、エージェントの記憶に頼らず機械的に実行する**」ことです。たとえば `/step-start` は、tasks.md・直前 Step のレビュー記録とテスト実施記録・関連する設計判断 D-N・関連リファクタリング項目を、毎回同じ順序で Read してから着手宣言まで進めます。

ここがミソで、`/step-start` が「決定論的に正本ファイルを読み直す」装置になっているから、`/clear` で会話を捨てても、新しいセッションが同じ状態から再開できる。skill が状態再構築のプロトコルになっているわけです。

## /clear を運用に組み込む

以上を踏まえた実際のセッション運用フローがこれです。

```
1. Step を最後まで完走 → /commit → git push
2. /clear でコンテキストを全消去
3. 新規セッションで /step-start <次の Step ID>
4. skill が tasks.md / 直前 Step のレビュー記録とテスト実施記録 / 関連 D-N / 関連 refactoring を
   決定論的に Read し、着手宣言まで自動で進める
5. 通常どおり作業を進める
```

**Step の区切りで意図的に `/clear` する**のがポイントです。Step を重ねるとコンテキストは必ず膨張し、放っておけば自動圧縮が走る。自動圧縮は制御できないタイミングで起き、解釈ズレとトークン浪費を生む。だったら、Step の切れ目という**情報の連続性が切れる安全なタイミングで、自分から捨てる**方がいい。

逆に、**Stage 間では `/clear` しません**。同じ Step 内の 4 Stage（設計 → テスト設計 → 実装 → テスト実施）は情報の連続性が高く、設計判断がそのままテスト計画と実装に流れ込む。ここで clear すると再構築コストの方が高くつきます。

`/clear` できる根拠は、繰り返しになりますが「状態の正本がファイルにある」ことです。タスク表・テスト実施記録・レビュー記録・設計判断ログ・リファクタリング候補ログの各ファイルに判断と進捗が残っているから、会話を捨てても直前までの文脈は失われない。**捨てられる設計だから、安心して捨てられる。**

## この設計で何が変わったか

体感で効いたのは次の点です。

* **矛盾実装が激減した。** decisions.md を読ませることで、エージェントが過去の判断と整合した実装を出すようになった。
* **自動圧縮に振り回されなくなった。** 圧縮を「起きてしまう事故」ではなく、`/clear` で「先回りして制御するもの」に変えられた。
* **トークン効率が上がった。** Step ごとに会話をリセットするので、1 セッションが常に小さく保たれ、長大なコンテキストを毎ターン読み直すコストが消えた。
* **再現性が出た。** 「あの判断なんだっけ」が全部ファイルにあるので、数週間ぶりに戻っても `/step-start` 一発で再開できる。

## まとめ

第1回の要点はこれだけです。

> **コンテキストを永続ストレージとして信頼しない。判断・進捗・証跡の正本を git 管理ファイルに逃がし、skill でその読み直しを決定論化する。すると `/clear` が怖くなくなり、自動圧縮を自分で制御できる。**

Phase/Stage/Gate の階層は、この「状態をファイルに逃がす」ための器です。器があるから、各ファイルが何の正本かを迷わず決められる。

次の第2回では、この Step の中にある **4 Stage × 2 段レビュー**を掘り下げます。Claude（プロジェクトの文脈を知るレビュアー）と OpenAI Codex（何も知らない第三者レビュアー）を独立させて組み合わせ、単一モデルの盲点を別モデルで埋める仕組みです。
