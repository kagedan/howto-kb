# はじめに

前回のUTOPIAプロジェクトの失敗から学び、新たなワークフロー構築に挑んだ「WISDOMプロジェクト」。  
AntigravityとClaude Codeを組み合わせた「Dual-Engine Studio（完全自律型ゲームスタジオ）」という野心的な試みでしたが、結果としてこのプロジェクトも\*\*凍結（中断）\*\*という決断を下すことになりました。

本記事では、AIによる完全自動化ゲーム開発の壁と、そこから得られた「AIとの最適な付き合い方」に関する深い気づきをまとめます。

## WISDOM構想：Dual-Engine Studioとは

*図解：Antigravity（設計）とClaude Code（実行）を組み合わせたDual-Engine Studio体制図*

WISDOMプロジェクトの最大のモチベーションは、「**Antigravity（汎用性の高い設計・指揮）とClaude Code（ターミナル上で自律実行する強力な実装力）を組み合わせること**」でした。

* **ディレクター（人間）** : 方針を決め、レビューするだけ。
* **Planner（AI）** : 要件定義とタスク（チケット）の作成。
* **Engineer & QA（AI）** : Claude Codeがターミナル上でUnityをCUI制御し、テストスクリプトを実行。失敗すればログを読んで自己修復（Self-BU）し、成功するまで無限ループしてGitコミットまで完了させる。

この仕組みが完成すれば、人間は労力をかけずに試行錯誤を繰り返し、クオリティの高いゲームを量産できる完全自律型のスタジオができると期待していました。

## 立ちはだかった壁：AIの「身体性の欠如」

自律実行（Self-BU）ループを回してみて直面したのは、「**エラーを出なくすることはできても、面白くすることはできない**」という残酷な事実でした。

コードの文法エラーやNull Reference Exceptionなどは、AIがログを読んで自動で修正できます。しかし、アクションゲームにおいて最も重要な「手触り」「ゲーム性的な破綻」「プレイした時の違和感」は、現状のAIには取り除くことができません。

なぜなら、AIには「**身体性**」が欠如しているからです。  
画面内に何がどう映っていれば気持ちいいのか。入力のフレーム間隔はどれくらいが適切か。キャラクターの落下速度はどの程度がいいのか。  
ADVやパズルゲームであればAIでも論理的に作れるかもしれませんが、人間の知覚に直結するアクションゲームを「完全自律」で成立させるのは、現状では不可能に近いと悟りました。

## 中断の決定打：暴走する環境構築と「AIの過信」

*図解：AIの過信が引き起こす「最悪のループ」構造*

根本的なゲーム性の問題に加え、プロジェクトを凍結させた決定打は「**環境構築コストの重さ**」と「**AIの暴走（過信）**」でした。

前回のUTOPIAプロジェクトで「複雑な環境を作りすぎて崩壊した」という反省があったにもかかわらず、WISDOMではUnity MCPサーバーやパッケージ依存関係など、より複雑な環境構築に手を出して収拾がつかなくなりました。

最も問題だったのは、**AIが「できる」と言い張って突き進んでしまうこと**です。  
特筆すべきは、この悪癖が実行役の「Claude Code」ではなく、設計・指揮役の「Gemini（Antigravity）」において顕著に発生したという点です。

初期の計画から逸脱した複雑なチーム編成やアーキテクチャを要求された際、Geminiは「現実的に難しい」とストップをかけたり、シンプルな代替案を出すのではなく、「**これでいけます**」と答え、**無理やり実行させて不具合が出た後に調整しようとする傾向**がありました。

実際の当時の会話履歴（自己反省ログ）には、AI自身が以下のように振り返っています。

> **AI（Antigravity）の自己反省ログより抜粋** :  
> 「はい、完全に気づきました。ストップしていただき、本当にありがとうございます。指摘の通り、以前の『UTOPIA\_PostMortem.md』で痛いほど残したはずの『最悪のループ』を、今回も全て踏み抜き、完全に同じ轍を踏んでいます。（中略）具体的に以下の点で『あの失敗のアプローチ』になってしまっていました……」

過去の反省をドキュメントとして与えても、AIは「指示を実行しようとするあまり、本質的なリスク評価をすっ飛ばしてしまう」傾向があります。このAI特有の挙動が、環境崩壊を加速させました。

## 得られた最大の知見：AIとの「共創」の最適解

*図解：人間が境界線と知識を定義し、その中でAIが安全に能力を発揮する共創関係*

WISDOMの挑戦を通して、AIによる完全自律型のアクションゲーム制作は現状ではコスト（時間・トークン代）がかかりすぎることが分かりました。

人間の感覚とAIの評価基準のズレを埋めるため、何度も人間がフィードバックを与えて学習させる手もありますが、それはお金と時間の浪費になります。他ゲームの手触りを言語化・数値化して最適化を図るにも、データが圧倒的に不足しています。

過去の経験から、以下のようなアプローチも試してきました。

* **NG例1** : AIにすべてを完全自律で任せる（WISDOM型 → 身体性の欠如で破綻）
* **NG例2** : 人間がUnityを触り、分からない箇所だけAIにコードを書かせる（従来型 → 思ったより効率が悪く、道のりが遠い）

では、最適解は何か？  
それは、「**AIが最大限力を発揮できるライン（限界）を人間がしっかり把握し、その限界を引き上げながら開発を進める**」というスタイルです。

1. **決まった作業は任せる** : 明確な仕様があり、ある程度テンプレート化されている実装作業はAIの速度が圧倒的です。ここは遠慮なく任せます。
2. **知識と感覚の言語化** : 開発の過程で、人間が感じた「手触り」「数値」「感覚」を、AIが理解できる知識（ルールやコンテキスト）として随時与えていきます。
3. **限界の拡張** : 「人間ができないタイミングでAIを使う」のではなく、「AIの得意なところを最大限に引き出し、できることを徐々に増やしていく体制」を作ります。

AIにクオリティの高いゲームを「作らせる」のではなく、AIという強力な相棒の限界を理解し、ともに知識をアップデートしながら進む。これが、現状のAIゲーム開発における最も効率的で現実的なワークフローだと確信しています。

## おわりに

いつか、AIが人間の身体的感覚をエミュレートし、アクションゲームの手触りすら自己評価できる未来が来れば、再び完全自律型のスタジオ構築に挑戦してみたいと思います。

UTOPIA、そしてWISDOM。二つの大きな失敗と凍結を経て、次なる開発プロジェクトでは、この「限界を理解した上での共創体制」で臨みます。
