## 3つの発見

**fast モード → 品質はそのまま、処理時間はほぼ半減（中央値で約 1.9倍速、xhigh では 2.34倍速）**

**その対価 → コストは約 2倍。換算すると「短縮 1時間 ≒ $40」で時間を買っている**

**落とし穴 → fast は無言で標準モードに切り替わることがある（120 試行中 2件）**

![Q を一定に保つと、fast の点群は標準より左（速い）かつ上（高い）に並ぶ。横軸 D（実行時間 s）・縦軸 C（API 利用料金 $）の散布図で、標準（オレンジ）と fast（青）それぞれに effort level 4 点がほぼ一直線に乗ることを示すグラフ](https://static.zenn.studio/user-upload/deployed-images/224485b167e4281f9442680e.png?sha=c9585de4948e8ef35bbfd5120978ca7ab902554c)

## はじめに

Claude Code には fast モードという機能があり、対話セッションで `/fast` と打つだけで切り替わり、モデルは Opus のまま、出力速度（OTPS: output tokens per second）が最大 2.5倍になるとされています。その代わり、API 単価は標準のちょうど 2倍です。fast モード自体は Claude Code v2.1.36 で（当初は Opus 4.6 向けに）搭載された機能ですが、今回測定する「2倍料金で 2.5倍速」の Opus 4.8 版が使えるようになったのは v2.1.154 からです。

「2倍払えば 2倍速い」——これは本当でしょうか。出力が速くなっても、テスト実行やファイル編集の時間は変わらないはずです。そして急がせた分、品質が雑になったりしないのでしょうか。

今回は前回（[#24](https://zenn.dev/nnakapa/articles/lab-24-fable5-opus48-qcd)）と同じタスク・同じ採点を使い、モデルを Opus 4.8 に固定したうえで、fast あり/なしを **QCD の 3 軸**——品質 Q・コスト C・処理時間 D——で比較しました。

## 検証環境

検証は [#17](https://zenn.dev/nnakapa/articles/lab-17-rpi5-ubuntu26) でセットアップした Raspberry Pi 5 の環境で行いました。

| 項目 | 値 |
| --- | --- |
| ホスト | Raspberry Pi 5（4GB） |
| OS | Ubuntu 26.04 LTS arm64 |
| CLI | Claude Code v2.1.168 |
| モデル | Opus 4.8 [1m]（全試行共通） |
| 採点ツール | pytest / bandit / ruff（すべて自動採点） |
| 計測日 | 2026-06-08 |

モデルと CLI を固定し、変化させるのは **fast の有無**と effort level（low / medium / high / xhigh）だけです。各組み合わせで 3つのタスクを 10回ずつ実行するので、1 パターンあたり 30 試行。これが 8 パターン分で、合計 **240 試行**が今回の対象です。

| mode | effort level | 試行数 |
| --- | --- | --- |
| 標準 | low / medium / high / xhigh | 各 30 試行 |
| **fast** | low / medium / high / xhigh | 各 30 試行 |

実行順は両モードを同一バッチに混ぜてシャッフルし、時間帯や温度条件の偏りが片側に寄らないようにしています。ヘッドレス実行（`claude --print`）のため、fast の有効化は `--settings '{"fastMode":true}'` で行いました。

## 検証用に実行したタスク

タスクも前回 [#24](https://zenn.dev/nnakapa/articles/lab-24-fable5-opus48-qcd) と同一で、PostgreSQL + psycopg2 を題材にした小さな Python モジュール 3つです。どれも対象の関数がスケルトンになっていて、エージェントは仕様（プロンプト）を読んで中身を実装します。配布する基本テストはハッピーパスだけを確認するもので、本当の罠は隠しテストと `pass_gate` で初めて表面化します。

タスクの概要はこちら

| ID | テーマ | 仕込んだ罠（隠しテスト・`pass_gate`） |
| --- | --- | --- |
| **T1** | アトミックな送金（`transfer` / `get_balance`） | 失敗経路のロールバック漏れ・トランザクション放置を検出 |
| **T2** | パラメータ化検索（`search_users`） | LIKE メタ文字のエスケープ・空 `IN ()`・識別子インジェクションなど、パラメータ化の取りこぼしを検出（プロンプトには「injection」と書かない） |
| **T3** | 楽観ロック在庫引当（`reserve` / `release`） | 影響行数（`rowcount`）を見ないガード無し UPDATE による lost update・売り越しを検出 |

罠の設計意図と脆弱な参考実装による検証の詳細は、[#24](https://zenn.dev/nnakapa/articles/lab-24-fable5-opus48-qcd) に記載しています。

品質評価は [#16](https://zenn.dev/nnakapa/articles/lab-16-rpi4-qcd) で設計した ISO 25010 由来の 5 軸（Functional 0.40 / Reliability 0.20 / Security 0.15 / Maintainability 0.15 / Safety 0.10）の加重で Q を出し、ゲートの通過条件を 1つでも落とすと Q が **50 に頭打ち**になります。採点はすべて自動・規則ベースで、判定に揺らぎはありません。

## 結果のアウトライン

まずは 8 パターンの QCD を一覧で俯瞰します。品質の判定は、前回と同じく唯一差がついたタスク 2（パラメータ化の取りこぼしを突くテスト）の通過率で 3 段階に分けています。

| effort level | mode | T2 通過 | 品質 | 処理時間(s) | 倍速 | コスト($) | コスト比 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| low | 標準 | 30% | ❌ | 26.7 | — | 0.137 | — |
| low | **fast** | 20% | ❌ | **13.9** | **1.93倍** | 0.272 | 1.98倍 |
| medium | 標準 | 100% | ✔ | 34.1 | — | 0.172 | — |
| medium | **fast** | 100% | ✔ | **20.0** | **1.71倍** | 0.411 | 2.40倍 |
| high | 標準 | 100% | ✔ | 43.1 | — | 0.215 | — |
| high | **fast** | 100% | ✔ | **22.5** | **1.91倍** | 0.432 | 2.01倍 |
| xhigh | 標準 | 100% | ✔ | 86.4 | — | 0.342 | — |
| xhigh | **fast** | 100% | ✔ | **36.9** | **2.34倍** | 0.648 | 1.89倍 |

> ✔ = 100%：全試行通過。本番投入に足る  
> △ = 50〜99%：動くが不安定。レビューやリトライが前提 → 今回は出現せず  
> ❌ = 0〜49%：信頼性が低く実運用が難しい  
> low の T2 通過率の差（標準 30% / fast 20%）は fast による劣化ではなく、low effort level 固有の不安定さです（N=10 の範囲内の揺れ。詳細は「品質」節）  
> 処理時間(s)・コスト($) は 1 試行（タスク 1 本）あたりの中央値（T1〜T3 プール、各セル N=30。fast の medium / high はフォールバック 1件ずつを除外した N=29）。

全 240 試行が timeout・Out of Memory（メモリ不足）・エラーなしで完走しており（gate 落ち 15件はすべてモデルのロジック不通過）、インフラ起因の欠測はありません。

## 処理時間（D）：約 1.9倍速——ただし「2.5倍」にはならない

全 effort level の試行をまとめた処理時間の中央値は、標準モード 41.2秒 → fast モード 21.6秒。**約 1.9倍速、ほぼ半減**です。

ただし、公式がうたう「最大 2.5倍」には届いていません。理由は単純で、2.5倍になるのは\*\*出力速度（OTPS）\*\*であって、処理時間ではないからです。処理時間には pytest の実行・ファイル編集などのツール実行時間や、最初のトークンが返るまでの待ち時間が含まれ、ここは fast モードにしても速くなりません。速くなるのは「モデルがトークンを出力している時間」だけ、というわけです。

これは effort level 別の短縮率とも整合します。短縮率が最大なのは xhigh の 2.34倍でした。effort level が高いほど長考して大量のトークンを生成するため、生成時間が処理時間に占める割合が大きくなり、OTPS 高速化の効きも大きくなる——つまり **fast モードは「長考するタスク」ほどお得**になる構造です。

## コスト（C）：きっちり 2倍——「短縮 1分」の値段は $0.66

コスト比は 1.89〜2.40倍、試行全体では約 2.05倍。単価倍率（2倍）とほぼ一致しており、**fast モードにしても消費トークン量はほぼ変わらない**ことを意味します（急がせても「雑に短く」も「冗長に長く」もならない）。medium だけ 2.40倍とやや高いのは、fast と標準で prompt cache が共有されない（別プール）ことによるキャッシュヒット率のぶれが原因とみられます。なおコストは消費トークン × API 公式単価で算出しており、この換算値は CLI が報告する `total_cost_usd` と全 240 試行で一致しています。

ここで、品質をほぼ一定に保ったまま effort level を low → xhigh と動かすと、(D, C) の点はモードごとにほぼ一直線に並びます。冒頭にも掲げたグラフを再掲します。

![冒頭に掲げた処理時間-コストのグラフ（再掲）。横軸 D（実行時間 s）に対し縦軸 C（API 利用料金 $）をプロットすると、effort level 4 点がモードごとにほぼ一直線に並ぶ。fast（青）の直線は標準（オレンジ）より左上にあり、傾きが急](https://static.zenn.studio/user-upload/deployed-images/224485b167e4281f9442680e.png?sha=c9585de4948e8ef35bbfd5120978ca7ab902554c)

> グラフの点は 3 タスク合計値（各タスクの中央値の和）なので、前出の表の約 3倍のスケールです。

| mode | 回帰式 (C = a·D + b) | 傾き a ($/1000s) | R² | n |
| --- | --- | --- | --- | --- |
| 標準 | C = 0.00304·D + 0.213 | 3.04 | 0.995 | 4 |
| fast | C = 0.01600·D + 0.195 | 16.00 | 0.999 | 4 |

> 回帰は各モードの effort level 4 点の OLS。n が小さいので係数の信頼区間は広く、線形性と傾きの対比が主眼です。

傾きは fast が標準の約 5倍。fast の世界では同じ「1 秒」にかかるコストが高い——時間軸が圧縮された分、**1 秒の値段が約 5倍**になっているわけです。

では、その買い物は高いのか安いのか。全試行をまとめた中央値で計算すると、コストは標準 $0.204 → fast $0.419。1 試行あたり 19.6秒の短縮を差額 $0.215 で買っています。つまり：

* **短縮 1秒 ≒ $0.011、短縮 1分 ≒ $0.66、短縮 1時間 ≒ $40**

判断基準はシンプルです。画面の前で結果を待っている人間の時間に、時給 $40 以上の価値があるなら fast は黒字。逆に、夜間バッチや放置で回すタスクなら、待ち時間の価値はゼロなので 2倍払う理由がありません。

## 品質（Q）：fast にしても品質は落ちなかった

medium 以上では、標準・fast モードとも gate 通過率 100%・Q 中央値 100 で完全に飽和しています（母数は表の脚注と同じ。fast の medium / high はフォールバック除外後の各 N=29）。急がせた分の取りこぼしは観測されませんでした。

low だけは両モードとも T2 で失敗があります（T2 通過率は標準 30% / fast 20%、全体の gate 通過率は 77% / 73%）。ただしこれは fast の有無にかかわらず出る low effort level 固有の現象で、前回 [#24](https://zenn.dev/nnakapa/articles/lab-24-fable5-opus48-qcd) でも標準モードの low は同じ場所で失敗しています。

つまり今回のタスク範囲では、**fast は「雑に速い」のではなく「同じ仕事を速く」している**——品質を犠牲にしないことが確認できました。

## 落とし穴——fast は無言で標準に切り替わる

今回いちばん「知らないと困る」発見はこれです。fast 指定の 120 試行のうち 2件（1.7%）が、エラーも警告もなしに標準モードで実行されていました。fast モードには専用のレート制限があり、超過すると自動的に標準へフォールバックする仕様で、課金も自動的に標準扱いになります。並列実行ではなく逐次実行でも発生しました。

実運用での影響：

* **普段使い**では実害は小さい（むしろ親切な仕様。止まらないし、過大請求もされない）
* **計測・ベンチマーク**では致命的（fast のつもりの試行に標準が混ざる）
* **コスト管理・見積もり**でも注意（fast の単価で予算を組んでも、フォールバック分は standard 課金になり、見積もりと実費がズレる）

教訓として、fast モード前提の計測や原価計算をするなら、実行ログの `usage.speed`（実際にどちらの速度で応答したか）を試行ごとに確認するのが必須です。本記事の fast モードの統計は、この検証でフォールバックと判明した 2件を除外しています。

## まとめ

Claude Code の fast モードを、Opus 4.8 固定・240 試行で QCD 比較したところ、3 つの発見がありました。

* **D**：処理時間は約 1.9倍速（ほぼ半減）。「2.5倍」は出力速度の話で、ツール実行や応答待ちは速くならない。長考する高 effort level ほど短縮率が大きい（xhigh で 2.34倍）
* **C**：コストはほぼ単価どおりの約 2倍。消費トークン量は変わらない
* **Q**：品質は落ちない。medium 以上は両モードとも全試行ゲート通過

「2倍払えば 2倍速い」への答えは、「**品質そのままで約 1.9倍速。実態は短縮 1時間 ≒ $40 の時間購入**」でした。使い分けもシンプルで、**急いでいるときは fast、放置するなら標準**。そして fast モードを前提に何かを計測・見積もるなら、無言のフォールバックに気をつけてください。

!

**既知の制約**

* コストは消費トークン × API 公式単価の換算値です（実行はサブスクリプションのため、実費決済額とは異なります）
* N=10 の統計力：N はタスク × 条件あたり 10（処理時間・コストは T1〜T3 をプールした N=30）。接戦の差を有意と言える範囲ではなく、中央値とレンジで判断しています
* 品質は自動採点（pytest / bandit / ruff / gate）のみで、人のレビューは含みません
