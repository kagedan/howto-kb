Claude Code を日常的に使っていて、抱えているプロジェクトが2つ、3つと増えてきたころ、こんな症状に心当たりはないでしょうか。

* さっきまで話していたプロジェクト A の文脈が、プロジェクト B の話をした直後にきれいさっぱり消えている
* 「あの修正、終わった?」と聞くと、セッションはもうその話を覚えていない
* 複数のタスクを1つの会話で切り替えるたびに、状況説明からやり直す羽目になる
* 気づけば自分が「Claude の記憶係」になっていて、進捗管理を人間が肩代わりしている

これは Claude(あるいは他の LLM エージェント)の能力が足りないから起きるのではありません。**1つの会話セッションに、性質の違う複数の仕事を同時に詰め込もうとする使い方そのものが、複数プロジェクト並行という用途に合っていない**だけです。この記事では、なぜそうなるのかを構造で説明し、越えるための発想の転換を示します。

## なぜ壊れるのか: 3つの壁

原因は独立した3つの制約に分解できます。

### 壁1: コンテキストは「読むだけ」でも減っていく有限資源

コンテキストウィンドウの消費は、こちらが打つ指示の長さだけで決まりません。ファイルを読む、コマンドの出力を読む、会話を思い出す、そのすべてが消費に加算されます。

プロジェクト A のコードを読み込んだ直後にプロジェクト B の話に切り替え、また戻ってくる。これを繰り返すと、「今は関係ない情報」がコンテキストに居座り続け、本当に必要な情報を押し出していきます。厄介なのは、これが会話が長くなるほど悪化する**片道の劣化**で、セッションを再起動しない限り回復しない点です。

### 壁2: 応答は単線的で、並行はできているように見えるだけ

Claude Code は一度に1つの応答しか生成できません。長時間かかる処理を1つ走らせている間、そのセッション自体は(バックグラウンド実行の仕組みを別途使わない限り)ほかの用事に使えません。

結果、「次は何をやらせるか」のスケジューリングを人間が担うことになります。これは本来、エージェントに任せたかった仕事そのものです。

### 壁3: タスクの重さと危険度が揃わないまま同じ土俵に乗る

軽い調べ物も、重要な設計判断も、危険度の高いコード実行も、同じセッション・同じ権限設定で扱っていると、こんなことが起きます。

* 定型作業に高精度(で高コスト)なモデルを使ってしまう
* 逆に重要な判断を雑に済ませて後で手戻りする
* 「これは人間の承認が要る操作か」の線引きがその場で揺れ、外部公開や課金を伴う操作まで勢いで実行しかけている

3つの壁はバラバラの問題ではありません。根は1つ、\*\*「1つの脳に全部を任せている」\*\*ことに尽きます。

## 発想の転換: エージェントを「1人」でなく「チーム」として設計する

解決の方向性は、人間の組織運営から借りてくると理解しやすくなります。1人のマネージャーが全プロジェクトの実務を自分の手でこなすのではなく、案件ごとに担当者を立て、マネージャーは意思決定と進捗管理に専念する。これをそのままセッション設計に当てはめます。

* **監督役**: 全体を見渡し、タスクを選び、担当セッションに指示を出し、成果物を検収し、人間に報告する。**自分では実装しません**
* **実働役**: プロジェクトごとに立てる専任セッション。コードを触り、実験を回し、原稿を書きます

この分業のポイントは、単に役割を分けるだけでなく「監督役が実作業をしない」ことでコンテキスト消費が非対称になる点です。監督役が読むのは各プロジェクトの状態を要約したファイルと報告だけでよく、コードの中身やログ全文を読み込む必要がありません。結果、監督役は長期間ダウンせずに全体を見続けられます。

さらにこの構造は、危険度に応じた隔離とも噛み合います。試行錯誤の激しいプロジェクトにはサンドボックス環境を与えてホストに触れさせない。タスクの重さに応じてモデルのグレードを変える。そして人間の承認が必要な操作(外部公開・課金・外部送信)は、どのセッションであっても必ず人間に戻す、というルールを最初から埋め込んでおきます。

過大な期待は避けたいところです。この分業を組んでも、Claude 自体の能力が上がるわけではないし、実装や設計判断の質が魔法のように良くなるわけでもありません。変わるのは「人間が進捗管理者から解放される」ことと、「並行するプロジェクト数を増やしても1プロジェクトあたりの対応品質が劣化しにくくなる」ことの2点です。

## ここから先は「組み方」の問題になる

診断(3つの壁)と発想の転換(チームとして扱う)までが、この記事で共有したかったことです。

ここから先、実際に手を動かすと出てくる疑問

* 監督役と実働役の権限をどう線引きするか
* 状態をどんなファイルにどう書くか
* tmux でどう起動して、危険な操作をどう隔離するか
* 外出先からどう介入するか

これらは、実際に手を動かして再現できる粒度の話になります。この粒度まで踏み込むと、記事1本では収まりきりません。

この続きは、実際に数ヶ月この体制を運用している設定ファイル・実際に起きた障害の記録込みで「Claude Code で作る自律マルチエージェント開発体制」にまとめています。この記事で言葉にした「3つの壁」と「監督役/実働役」の分業を、実際に動く形にするところまで踏み込みたい人向けの内容です。
