## はじめに：AIを使うほど思考力が落ちる人、伸びる人

ChatGPTやClaude Codeが当たり前になった2026年、エンジニアや事業主の中で明確に二極化が起きています。

* **タイプA**：AIに丸投げして「とりあえず動くもの」を量産し、自分は何も理解していない
* **タイプB**：AIを使うほど、設計判断・問題分解・抽象化のスキルが伸びている

この差は「プロンプトの上手さ」ではなく、**AIとの対話設計**にあります。本記事では、Claude Codeを「自分を鍛えるスパーリングパートナー」として運用する具体的な手法を、コード例とともに解説します。

## 「丸投げ運用」と「壁打ち運用」の違い

まず、典型的な使い方の差を整理します。

| 観点 | 丸投げ運用 | 壁打ち運用 |
| --- | --- | --- |
| 出発点 | 「〇〇を作って」 | 「〇〇を作りたい。設計の選択肢を挙げて」 |
| AIの役割 | 実装者 | 反論者・質問者・レビュアー |
| ユーザーの動き | 結果を受け取る | 仮説を出し、反論で削る |
| 残るもの | 動くコード（理解なし） | 判断ログ + 自分の理解 |
| 半年後の自分 | AI依存が深まる | AIなしでも同じ思考ができる |

ポイントは、**判断のオーナーシップを手放さない**こと。AIは「最終決定者」ではなく「思考の鏡」として使います。

## 実践テクニック1：Grill Me（質問攻め）モード

新機能の設計段階で、自分の仕様を言語化する前にAIに「徹底的に質問させる」手法です。

```
あなたはシニアプロダクトマネージャーです。
私が今から話す新機能のアイデアに対して、以下の観点で
最低15個の質問を投げてください。回答は私から引き出すのが目的です。

- ユーザーストーリーの曖昧さ
- エッジケース
- 非機能要件（性能・セキュリティ・運用）
- 既存機能との競合
- 計測指標（成功判定）

私の最初のアイデア: 「Xの投稿を自動生成するボットを作りたい」
```

このプロンプトを投げると、Claude Codeは矢継ぎ早に質問してきます。回答していくうちに、自分が考えていなかった「投稿の品質基準」「NGワード処理」「承認フロー」といった論点が浮かび上がります。

## 実践テクニック2：Devil's Advocate（反論者）モード

設計案がまとまったら、わざと反対意見を出させます。

```
以下の設計案に対して、最も鋭い反論を3つ出してください。
「動くけど後で痛い目を見る」観点を優先してください。
忖度は不要です。

## 設計案
- Firestore に未処理投稿を保存
- GAS の cron で 10 分おきにポーリング
- Discord に承認依頼を送信
- 承認後、X API で予約投稿
```

すると、たとえば次のような反論が返ってきます。

* 「GAS の 6 分実行制限に引っかかるのでは？ 並列処理が必要なら Cloud Functions のほうが堅い」
* 「Discord 承認のタイムアウト処理が定義されていない。承認待ちのまま腐る投稿が出る」
* 「Firestore のステータス遷移が文字列ベース。Enum 化しないとタイポで詰む」

**この反論に対して自分で再反論を組み立てる**ことが、思考力のトレーニングになります。AIの意見を鵜呑みにせず、自分の根拠を言語化する練習です。

## 実践テクニック3：TDDで自分にプレッシャーをかける

Claude Code に実装を任せるとき、テストを先に書かせます。

```
TDD で進めてください。
1. まず失敗するテストを書く（実装はまだ書かない）
2. 私がテストをレビューし、OK を出してから最小実装
3. リファクタリング
各ステップで私の承認を待ってください。
```

このフローだと、**テストの段階で仕様の穴に気づく**ため、実装後の手戻りが激減します。さらに、テストコードを読むことで自分の理解度を測れます。

## 実践テクニック4：判断ログを残す

AIとの対話で得た判断は、必ず**自分の言葉で**ドキュメント化します。

```
## 判断: Firestore vs PostgreSQL

### 選択: Firestore
### 理由（自分の言葉）
- 個人利用で月100リクエスト未満 → Spark プラン無料枠で十分
- スキーマ変更が頻発する初期フェーズ → ドキュメント型が有利
- GAS との親和性が高い（公式ライブラリあり）

### 却下した選択肢
- PostgreSQL: マイグレーション運用コストが高い
- Cloud SQL: 月額固定費が発生

### 再評価の条件
- 月間リクエストが1万を超えたら見直し
- リレーション設計が3階層以上になったら見直し
```

このログは、半年後の自分が「なぜこの選択をしたか」を思い出すための装置です。AIに同じ質問を再度投げるより、自分の判断履歴を読み返すほうが学びは深い。

## FAQ

### Q1. 壁打ち運用は時間がかかりませんか？

短期的には Yes、長期的には No です。最初の数回は丸投げの2〜3倍時間がかかりますが、**同じ失敗を繰り返さなくなる**ため、3ヶ月後には逆転します。

### Q2. Claude Code と ChatGPT で運用は変わりますか？

Claude Code はターミナル統合・ファイル操作・スキル機能（カスタムコマンド）が強く、エンジニア向けの壁打ちに向いています。ChatGPT は対話UIが洗練されており、企画段階の発散には強い。**フェーズで使い分ける**のが現実解です。

### Q3. AIの回答を鵜呑みにしてしまう癖はどう直しますか？

「**最初の回答は必ず疑う**」をルール化します。具体的には、AIが何か答えたら「他の選択肢は？」「この主張の根拠は？」「逆のケースだとどうなる？」を機械的に3回投げる。これだけで依存度は劇的に下がります。

## 将来の展望：AIエージェント時代の「人間の役割」

2026年現在、AIエージェント（Claude Code, Devin, Cursor Composer 等）の自律性は急速に高まっています。半年前は「コードを書いてくれる」だったのが、今は「PR を作り、レビューに応答し、デプロイまで進める」レベルに到達しました。

この流れの中で、人間が担うべき役割は\*\*実装者から「方向性のオーナー」\*\*へとシフトしています。具体的には：

1. **問いを立てる力**：何を解くべきかの設定
2. **判断する力**：トレードオフを言語化して選ぶ
3. **検証する力**：AIの出力を疑い、テストする

これらはすべて「AIに質問させ、AIに反論させ、自分で判断する」という壁打ち運用で鍛えられます。**作業を奪われるのではなく、判断の解像度で差がつく時代**に入ったということです。

## まとめ

Claude Code を「外注先」として使うのは、もったいない。

* ✅ 質問させる（Grill Me）
* ✅ 反論させる（Devil's Advocate）
* ✅ テストを先に書かせる（TDD）
* ✅ 判断ログを自分の言葉で残す

この4つを習慣化するだけで、AIを使うほど人間の思考力が伸びる回路ができます。AI時代に置いていかれるか、加速するかは、**ツールの選択ではなく対話の設計**で決まります。
