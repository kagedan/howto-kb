## 1. はじめに ― 前回スキルが使えたので、調子に乗って天気を聞いた

前回、自作スキルがエージェントで「効く」ところまでたどり着きました。だから今回は、軽い気持ちでした。

Telegramでうちのエージェント（人格は前回登場した「アーク」というメイド役です）に、「今日の天気を教えて」と聞いただけです。天気くらい、もう一発で返ってくるだろう。そう思っていました。

結論から書きます。**返ってきませんでした。** ツール呼び出しに失敗し続けて、いつまで経っても天気が出てこない。気軽に聞いただけなのに、なぜこんなに転ぶのか。そこから半日、犯人探しが始まりました。

この記事は、DGX Spark上でNemoClaw（NVIDIAの堅牢なAIエージェント実行環境）とOpenClaw（その上で動くエージェント本体）を回しているシリーズの第3弾です。

* 前々回: 箱を建てた（DGX Sparkに自前モデルのエージェントを載せた）
* 前回: その箱に自作スキルを足した（足せたけれど、効くかは頭脳＝モデル次第だった）
* 今回: 標準の天気スキルを使おうとしたら、最初からすれ違っていた

そして先に、今回の核を書いておきます。詰まりの本体は、箱が意地悪をしたわけではありませんでした。**善意で用意された標準の天気スキルと、堅牢な安全設計（外向き通信の許可リスト）が、最初から別々の宛先を見ていた**。要するに噛み合っていなかった、というだけの話です。ただ、その「だけ」を突き止めるまでが長かった。

前回からの体制は今回も同じです。

!

**実作業はぜんぶClaude Codeです。** おじさんはSSH越しにDGX Sparkを操作するClaude Codeに方針と判断を出しただけで、コマンドを叩いたのもgateway.log（エージェントの実行ログ）を読んだのもトラジェクトリ（後述・エージェントの思考と行動の記録）を解析したのも、ほぼClaude Codeです。だからこの記事も「楽勝でした」という話ではなく、「指示しながら見ていたら、思っていたのと違うところで詰まった」という記録です。

なお頭脳（モデル）は、前回の後日談で書いた通り、ツール実行が安定する**公式の汎用大型モデル**に切り替え済みの状態から始めています。

## 2. きっかけ ― Telegramで「今日の天気」を聞いたら、延々と失敗した

前回までで、頭脳は公式の汎用大型モデルに替えてあります。スキルも「効く」状態になったはず。だから天気なんて、一番ありふれた質問だし、すぐ返ると思っていました。

ところが、Telegramに返ってきません。アークは何か考えている素振りはあるのに、最終的な答えが出てこない。何度聞いても、ツール呼び出しのところで転んでいるようでした。

正直に言うと、最初は「またモデルのクセかな」と思いました。前回、頭脳がツールを呼ぶところで散々苦労したので、その延長だと決めつけかけたんです。でも、それにしては挙動がおかしい。健全にツールを呼ぼうとしているのに、その先で失敗している。

「気軽に聞いたのに、なぜこんなに失敗するんだ」。ここで決めつけるのをやめて、ちゃんと調べることにしました。エージェントが何を考えて、どこで転んでいるのか。gateway.logとトラジェクトリを開いて、犯人探しを始めます。

掘ってみたら、失敗の原因は1つではありませんでした。ただ、主役は最初からはっきりしていました。**スキルが見ている宛先と、箱が許す宛先が、まったく別物だった**ことです。まずそこから書きます。

## 3. 犯人の本体 ― スキルは「wttr.in」を指し、箱は「open-meteo」しか通さない

ここが今回の山場です。順番に、スキルの中身と、箱のルールを突き合わせていきます。

### 標準weatherスキルは「wttr.in を curl で叩け」と書いてある

まず、エージェントが使おうとしている標準（バンドル済み）の `weather` スキルの中身を読んでもらいました。場所はここです。

```
/usr/local/lib/node_modules/openclaw/skills/weather/SKILL.md
```

`openclaw skills list` でも `✓ ready`（使える状態）と出ています。中身の要点はこうでした。

```
---
name: weather
description: "Current weather and forecasts with wttr.in via curl ..."
metadata: { openclaw: { requires: { bins: ["curl"] } } }
---
## Commands
curl "wttr.in/London?format=3"
curl "wttr.in/London?format=j1"   # JSON
...
```

読めば一目瞭然で、このスキルは **「wttr.in というサイトを curl で叩け」** と教えています。

ここで2つ用語を補足します。**curl** はコマンドラインからURLを直接叩いてデータを取ってくる、昔ながらの定番ツールです。そして **wttr.in** は、URLを叩くだけで天気がテキストで返ってくる便利なサービスです。つまりこのスキルは、「wttr.in を curl で叩けば天気が取れるよ」という、ごくまっとうな手順書になっています。

### NemoClawの箱は「open-meteo と weather.gov」しか通さない

次に、箱の側のルールです。NemoClawはサンドボックス（隔離された実行環境）ごとに **egress（イーグレス＝外向きの通信）** を許可リストで縛っています。許可した宛先以外には、原則として外に出ていけません。

その `weather` 用のポリシー定義を読んでもらいました。

```
# ~/NemoClaw/nemoclaw-blueprint/policies/presets/weather.yaml
network_policies:
  weather:
    endpoints:
      - host: api.open-meteo.com         # /v1/forecast, /v1/elevation
      - host: geocoding-api.open-meteo.com
      - host: api.weather.gov            # /points, /gridpoints, /alerts ...
```

許可されている宛先は、この**3つ**だけでした。

* `api.open-meteo.com` … open-meteo の天気予報API
* `geocoding-api.open-meteo.com` … open-meteo の地名→緯度経度の変換API
* `api.weather.gov` … 米国気象局のAPI

`nemoclaw ojisan-agent status` で許可ホストを確認しても、リストに出てくるのはこの3つです。そして、この中に **`wttr.in` はありません**。

### つまり、最初からすれ違っていた

ここまで並べれば、犯人の正体は明らかです。

* スキルは **「wttr.in を叩け」** と言っている
* 箱は **「open-meteo と weather.gov しか通さない」** と決めている

宛先がそもそも一致していません。スキルが指す宛先（wttr.in）は、箱が許す3つの宛先のどこにも入っていない。だからエージェントが素直にスキルの言う通りwttr.inを取りに行くと、箱の許可リストに引っかかって外に出られず、失敗する。これが「気軽に聞いたのに失敗し続けた」真因でした。

標準スキルを有効化しただけでは天気が取れない。理由は、スキルの宛先と箱の許可宛先が噛み合っていないから。ここが今回の背骨です。

## 4. モデルとweb\_fetchの壁 ― さらに掘ると別の壁もあった

宛先のすれ違いが主犯です。ただ、gateway.logをさらに読んでいくと、その手前と奥に、別の壁も見えてきました。ここは正直に書きますが、主役にはしません。「宛先が噛み合っていない」という本筋に、モデル側の事情が重なって、症状をややこしくしていた、という整理です。

論点は3つあります。原因と対策をセットで書きます。

### 壁その1: 頭脳がツールの呼び方そのもので転ぶ

まず、頭脳（モデル）を公式の汎用大型モデルに切り替える前の話です。それまで常用していた**自前カスタムのローカルモデル**でこの天気質問を踏むと、そもそもツールの呼び方で転んでいました。

トラジェクトリを見ると、JavaScriptを書いてツールを呼ぶモード（コードモード）で、`require is not defined`（そんな関数は無い、というエラー）を10回出しながら、**156回**もループして、天気APIに到達する手前で力尽きていました。

これは前回のworklogで書いた自作スキルの発火失敗と同じ根本原因です。ローカルモデルがエージェントループでツール呼び出しを安定させられない、という既知の弱点。だから対策はシンプルで、**頭脳を公式の汎用大型モデルに替える**。これは前回の後日談で実施済みなので、今回は軽く流します。

### 壁その2: web\_fetchはcurlの代わりにならない（疑い）

公式の汎用大型モデルに替えたら、ツールの呼び方は健全になりました。ところが、別の転び方をします。gateway.logにこう出ていました。

```
[tools] web_fetch failed: fetch failed   (url: https://wttr.in/Tokyo?format=...)
[tools] web_fetch failed: fetch failed   (url: https://tenki.jp/forecast/3/16/)
```

ここで **web\_fetch** という用語を補足します。これは、curlのようにシェルからコマンドを叩くのではなく、**モデルが内蔵で持っている「URLを取りに行く」ツール**です。エージェントは、スキルに「curlで叩け」と書いてあるのに、つい手近なこの web\_fetch を選んでしまう。そしてそれが失敗する。さらに、許可リストにも無い `tenki.jp` を勝手に取りに行こうとして、当然これも失敗していました。

ここで気になったのが、curl と web\_fetch の差です。後の章で実証しますが、**egressにwttr.inを足すと、curlではwttr.inが取れます。でも web\_fetch は同じwttr.inに失敗し続ける**。

!

**curl は通るのに web\_fetch は通らない。これは観測事実で、理由は推測で止めます。**

観測できたのは「curl（exec経由）は通る／web\_fetch は失敗する」という事実だけです。理由まで断定はしません。**web\_fetch は、ネットワークポリシーのバイナリ許可とは別の許可レイヤー・経路を持っている"疑い"がある**、という仮説までです。いずれにせよ実務的な結論は同じで、**スキルが言う通り「curl（exec経由）」を使わせる**のが正解でした。web\_fetch に逸れると失敗する、と。

なので対策は、**取得方法をexec+curlに固定する**こと。exec（エージェントがシェルコマンドを実行するツール）でcurlを叩かせて、web\_fetchを使わせない。これを後でスキル本文に明記します。

### 壁その3: 思考が出力枠を食い尽くして「空応答」になる

3つめは、答えが空っぽで返ってくる症状です。gateway.logにこう出ていました。

```
[agent/embedded] reasoning-only retries exhausted ... surfacing incomplete-turn error
```

原因を掘ってもらったら、`openclaw.json` のモデル定義で `"maxTokens": 4096` になっていました。**maxTokens** は、モデルが1回のターンで出せる出力トークン（文章の単位）の上限です。そして公式の汎用大型モデルは、答える前に **reasoning（リーズニング＝内部の思考プロセス）** を長く回します。

天気質問のように複数のツールを使う質問だと、この思考が4096の枠を食い尽くしてしまい、肝心の本文（最終回答）に到達する前に枠が尽きる。結果、空応答になって `reasoning-only retries exhausted`（思考だけで再試行を使い果たした）と出ていた、というわけです。

対策は、**maxTokens を 4096 から 16384 に拡大**することです。コンテキストウィンドウ（モデルが一度に扱える総量）は131072あって余裕があるので、本文まで届く枠を確保しました。

ここまでで、症状の全体像が見えました。主犯は宛先のすれ違い。そこに「ツールの選び方（web\_fetch）」と「思考が枠を食う（maxTokens）」が重なって、ややこしく見えていた。では、どう直したか。次の章です。

## 5. 直し方 ― 宛先・取得方法・競合の3点で噛み合わせる

直し方を考えるとき、設計の選択肢は2つありました。

* **(A) 箱をスキルに合わせる**: egressに `wttr.in` を許可して、標準スキル（wttr.inを叩く）をそのまま使う
* **(B) スキルを箱に合わせる**: 既定で許可済みの `open-meteo` を叩く自作スキルを入れる

おじさんが選んだのは **(B) open-meteo に寄せる** 方でした。理由は堅牢さです。open-meteo は箱が**最初から許可している**宛先なので、サンドボックスを作り直し（rebuild）ても消えません。一方 (A) でwttr.inを足す方式は、カスタムの追加許可なので、作り直しのたびに気を配る必要がある。日常的に使うものだから、消えにくい方を取りました。

その上で、噛み合わせを直す手は、全部で次の**3点**です。以降、この記事ではこの順番を崩しません。

**① 宛先 = open-meteo に合わせる**

自作スキル `labo-weather` を作って導入しました（標準の `weather` と別名にして競合を避けています）。データ源は箱が許可済みの open-meteo に固定。よく使う都市の緯度経度表（geocoding不要で済む）と、open-meteo の forecast URL、それに WMO weather\_code（天気を表す数値コード）を日本語に変換する表まで同梱しました。

**② 取得方法 = exec + curl に固定する（web\_fetch回避）**

スキル本文に「**web\_fetchを使うな。execでcurlを実行せよ**」と明記しました。さらに、execの正しい呼び出し形式まで書き込んでいます。ここはトラジェクトリ解析で得た実機知見です。

トラジェクトリを見ると、モデルが exec を `["-c", ...]` の形で呼ぶと `Provide a command to start`（起動コマンドを渡せ）というエラーで失敗し、`["bash", "-c", ...]` の形だと成功していました。

```
# 失敗する呼び方
exec(["-c", "curl -s '...'"])         → Error: Provide a command to start
# 成功する呼び方
exec(["bash", "-c", "curl -s '...'"]) → OK
```

なので、スキルに `bash -c "curl -s '...'"` の形を例示して、モデルが正しい形で叩けるように固定しました。

**③ 競合するバンドルスキルを `enabled: false` で無効化する**

標準の `weather` スキル（wttr.inを叩く方）が残っていると、自作スキルと競合してモデルが迷います。なので無効化します。ここで1つ実機の発見がありました。

```
$ openclaw skills disable weather
# → そんなサブコマンドは無い
```

`openclaw skills` に `disable` サブコマンドは**ありません**（あるのは check / info / install / list / search / update だけ）。じゃあどう切るかというと、`openclaw.json` の設定で直接切ります。

```
{
  "skills": {
    "entries": {
      "weather": { "enabled": false }
    }
  }
}
```

`skills.entries.weather.enabled` を `false` にする。これは 1password や apple-notes など、他の多数のバンドルスキルが無効化されているのと**同じ方式**です。可逆ですし、念のためバックアップ（`openclaw.json.bak-weatheroff`）も取りました。設定後、`skills list` で `disabled` になっていることを確認しています。

整理すると、噛み合わせを直す3点はこうです。

1. **宛先** = open-meteo に合わせる
2. **取得方法** = exec + curl に固定する（web\_fetch回避）
3. **競合スキル** = `skills.entries.weather.enabled = false` で無効化する

### 補足: wttr.inでも実際にcurlは通った

3点の本筋とは別に、3章の「遮断ではなくすれ違い」を裏付けた確認も書いておきます。

設計(B)を選ぶ前に、(A)の方も試していました。カスタムのpresetで `wttr.in` をegress許可（`policy-add` で適用・Policy version 4）し、サンドボックスの中から直接curlを叩いてみたんです。

```
$ curl -sS 'https://wttr.in/Tokyo?format=3'   →  Tokyo: 🌤️  +22°C   (exit 0)
$ curl -sS 'http://wttr.in/Tokyo?format=3'    →  Tokyo: 🌤️  +22°C   (exit 0)
```

許可さえ足せば、wttr.inにもcurlは**ちゃんと通りました**（`+22°C`、正常終了）。だから「箱がwttr.inを遮断している」のではなく、「最初から許可リストに入れていなかった＝すれ違っていた」が正確です。なお、この `+22°C` は wttr.in から取った値で、後の章で出てくる open-meteo の値とは**データ源も値も別物**なので、混同しないでください。今回は最終的に(B)を採ったので、この wttr.in 許可は使っていません（気になれば撤去できます）。

## 6. 実機で天気が取れた ― 曇り・23.4℃、実APIと一致

3点を直した状態で、もう一度、同じ質問を投げました。「今日の東京の天気を教えて」。

返ってきたのがこれです。

```
今日の東京は 曇り です。
気温 23.4℃（体感 24.6℃）/ 湿度 63% / 風速 6km/h
```

**取れました。** 曇り、23.4℃、体感24.6℃、湿度63%、風速6km/h。

この値が本物かどうかも確認しています。事前に、open-meteo へ直接curlを叩いて取った値と突き合わせました。

つまり、エージェントは仕込んだ通り open-meteo を exec+curl で叩いて、実際のデータを取得して返した、ということです。WMO weather\_code を「曇り」という日本語に変換する処理もちゃんと働いていました。そして、人格（アーク）も維持されたまま。スキルの宛先と取得方法を箱に噛み合わせたら、天気が普通に返るようになりました。

正直に言うと、ここでも一点だけ注意があります。3点を固定して大幅に安定しましたが、それでもモデルが時々 exec の引数形式や web\_fetch の選択でブレる可能性はゼロではありません。スキルで手順を固定するのは「確率を大きく上げる」ことであって、「100%にする」ことではない。ここは正直に書いておきます。

## 7. まとめ ― 「足せる/使える」と「実際に効く」の間にあるもの

最後に、後続が同じ穴を踏まないように、要点を並べます。

3点の直し方は、全部この順番で覚えておくと迷いません。

1. **宛先** = open-meteo に合わせる（スキルの宛先を、箱が許可済みの宛先に寄せる）
2. **取得方法** = exec + curl に固定する（web\_fetchに逸れると失敗するので、curlを使わせる）
3. **競合スキル** = `skills.entries.weather.enabled = false` で無効化する（`skills disable` サブコマンドは無いので設定ファイルで切る）

補足として、もう1つ。

* **maxTokens の枯渇に注意**。思考の長いモデルは、出力枠を思考が食って空応答になることがあります。複数ツールを使う質問で顕在化しやすいです。今回は 4096 → 16384 に拡大して回避しました。

そして、シリーズの地続きを書いておきます。

* 前々回 = 箱を建てた
* 前回 = スキルは足せる。でも効くかは頭脳（モデル）次第だった
* 今回 = 噛み合わせを直して、「効かせる」ところまで到達した

おじさんの意見として、ひとつだけ書いておきます。押しつけるつもりはありません。

NemoClawの安全設計、つまり「許可した宛先以外には外に出さない」というegressホワイトリストは、正しいと思うんです。堅牢で、信頼できる。ただ、その正しさゆえに、善意で用意された標準スキル（wttr.inを見る）と、最初からすれ違うことがある。箱は悪くないし、スキルも悪くない。ただ別々の宛先を見ていただけ。

「足せる」と「使える」と「実際に効く」は、それぞれ別の話なんですよね。今回はその一番最後、「実際に効く」までの間に、宛先のすれ違いという橋を架ける作業がありました。**その橋を架けるのを判断するのが、人間の仕事だった**。これが、3本続けてDGX Sparkと格闘してみた、おじさんの正直な感想です。
