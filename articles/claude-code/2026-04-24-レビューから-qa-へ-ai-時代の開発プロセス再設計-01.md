## はじめに

AI が書いた PR のレビューに、以前より時間を取られていませんか?

AI Agent によるコード生成が業務レベルで実用に届き始め、Claude Code や Codex が「コードを書く主体」として PR を出す時代になりました。

こうなったときに、開発プロセスで最も ROI が下がるのはどこでしょうか。私の答えは明確です。**人間がコードレビューに使う時間** です。

AI が書き、AI がレビューできる領域がどんどん広がっています。その中で人間は何を検証すべきか。本記事の主張はシンプルです。

ただし、ここで言う QA は「テスト実行係」ではありません。AI には代替できない **振る舞いの正しさ** を検証する役割です。本記事ではこの QA の再定義と、時間配分の組み替えを提案します。

対象読者は次の通りです。

* AI Agent をすでに業務で使っている、あるいは導入を検討している開発組織の意思決定者
* チームのレビューコストに悩んでいるテックリード
* 自分たちの役割の変化を感じている QA 担当者

## AI コードレビューが担保できる品質は、どこまで来たか

まずは事実確認から入ります。2026 年時点で、AI Agent によるコードレビューが人間並みに信頼できる領域は、どこまで広がったのか。

私の観測では、次の領域は「AI に寄せた方が速くて確実」な水準に到達しています。

* **バグ検出**: null 参照、配列境界、非同期処理の race condition、例外ハンドリング漏れなど
* **セキュリティ**: SQL インジェクション、XSS、認可漏れ、秘密情報のハードコードなど OWASP Top 10 系
* **コーディング規約**: 命名、インデント、未使用コード、推奨 API の使用など
* **設計アンチパターン**: 過剰な責務、循環依存、レイヤー違反、抽象化の過不足

共通するのは、**コードそのものから機械的に判定できる** という点です。パターンを定義できる、あるいは文脈から判定できる領域は、AI の得意領域になりました。

### Claude Code のレビュー例

たとえば Claude Code に次のような Go の関数を投げると、こんな指摘が返ってきます（出力は一例）。

user.go

```
func (u *UserService) GetUser(id string) (*User, error) {
    query := "SELECT * FROM users WHERE id = '" + id + "'"
    row := u.db.QueryRow(query)
    var user User
    row.Scan(&user.ID, &user.Name, &user.Email)
    return &user, nil
}
```

```
[high] SQL インジェクション: 文字列連結でクエリを組み立てています。
       id にユーザー入力が流れ込むとクエリが改変されます。
       プレースホルダ（?）と引数渡しに書き換えてください。

[medium] エラーを握りつぶしています: row.Scan の戻り値 err を無視しています。
         レコードが存在しない場合や型ミスマッチが呼び出し元に伝わりません。

[low] 設計: UserService が *sql.DB を直接保持しています。
      Repository パターンで分離すると、テスト容易性が上がります。
```

指摘の精度と粒度は、数年前の静的解析ツールでは再現できなかったレベルです。コンテキストを読み、優先度を付け、自然言語で意図を説明してくれます。

### 人間レビューを上乗せする価値はあるか

「AI の指摘が完璧ではない、だから人間のダブルチェックが必要だ」という反論が成立します。しかし、弱いです。

なぜなら、**人間レビューも完璧ではない** からです。むしろ人間のレビューは時間制約・疲労・関心のムラで安定しません。AI レビューの信頼水準が人間レビューの安定水準を上回るチームも、すでに珍しくありません。

## 人間がコードレビューに使う時間の ROI は下がり続けている

AI に寄せられるなら、人間のレビュー時間は減らすべきです。ところが実態は逆で、AI が書くようになってから人間のレビュー負荷はむしろ増えています。

### データ: PR pickup time が AI 時代に伸びている

LinearB の 2026 Software Engineering Benchmarks Report には興味深い数値があります（8.1M PR、4,800 チーム分析）。

| PR の種類 | pickup time（Unassisted 比） |
| --- | --- |
| 人間が書いた PR | 1.0x |
| AI 補助で書いた PR | **2.47x** |
| AI Agent が書いた PR | **5.3x** |

> Engineers dread reviewing AI-generated code and often procrastinate when it comes to reviewing such PRs.  
> （エンジニアは AI 生成コードのレビューを嫌がり、そうした PR のレビューを後回しにしがちだ）  
> — [Code Review is the New Bottleneck For Engineering Teams](https://newsletter.eng-leadership.com/p/code-review-is-the-new-bottleneck)

レビュアーが AI 生成コードのレビューを嫌って後回しにする、という行動パターンが観測されています。

### 何が起きているのか

AI はコードを大量に生成できますが、レビューする人間の数は変わりません。結果、次の循環が起きます。

1. AI がコードを生成する量が増える
2. PR 数が増える
3. 人間レビュアーが追いつかない
4. pickup time が伸びる
5. 開発速度が落ちる

AI で生産性を上げようとしたのに、レビュー工程がボトルネックになって全体のスループットが下がります。この構図です。

### 何を人間がレビューしているか — 棚卸し

一度立ち止まって、PR レビューで人間が実際に指摘している内容を棚卸ししてみてください。

おそらく、こんな内訳になるはずです。

| 指摘カテゴリ | AI で検出可能か | 人間でないと無理か |
| --- | --- | --- |
| 命名の一貫性 | ◎ | × |
| SQL インジェクション等のセキュリティ | ◎ | × |
| Null 参照、境界チェック | ◎ | × |
| レイヤー違反、循環依存 | ○ | × |
| ドメインモデリングの妥当性 | △ | ○ |
| 業務要件との整合 | × | ◎ |
| UX・使いやすさへの影響 | × | ◎ |

**◎ が立つ領域は AI に寄せるべき**、**× がある領域は人間でないと無理**。この線が明確に引ける段階に来ています。

ところが多くのチームは、AI でも検出できる領域を人間が二重にレビューしています。これは時間の浪費です。

## QA の再定義 — コードの正しさから振る舞いの正しさへ

ここから本記事の核心です。

人間がレビューを減らして、浮いた時間はどこに投資すべきか。答えは **QA の再強化** です。ただし、ここで言う QA は従来のイメージとは少し違います。

### 「コードの正しさ」と「振る舞いの正しさ」を分ける

開発成果物の品質は、二層に分けて考えられます。

* **コードの正しさ**: コードから機械的に判定可能。AI が主担当。
* **振る舞いの正しさ**: 要件・業務・ユーザーとの整合性。人間（QA）が主担当。

この区分が今までぼやけていたのは、コードの正しさが人間のレビュー負荷を食い尽くしていたからです。AI がコードの正しさを引き受ければ、人間は振る舞いの正しさに集中できます。

### システム種別で「振る舞いの正しさ」の意味は変わる

「振る舞いの正しさ」を抽象的に語ると空中戦になります。実際にはシステム種別で検証すべき軸が変わります。

| システム種別 | 振る舞いの正しさの中身 | 検証の主体 |
| --- | --- | --- |
| 社内 / 業務システム | 業務整合性・ドメイン整合性・要件との一致 | 業務知識を持つ QA / ドメインエキスパート |
| B2C サービス | UX・使いやすさ・ユーザー体験・導線 | ユーザー視点を持つ QA / UX 担当 |

業務システムでは「請求書が正しく発行されるか」「承認フローが要件どおりに流れるか」が争点になります。コードが型安全でも、業務プロセスと噛み合わなければ価値がありません。

B2C サービスでは「ユーザーが目的を達成できるか」「迷わず進めるか」が争点になります。エラー処理が完璧でも、UI 配置やコピーが悪ければユーザーは離脱します。

どちらも、**コードを読むだけでは判定できません**。実装を動かし、業務やユーザーの立場で触って初めて分かる領域です。ここは AI に寄せにくく、人間の判断が最後まで残ります。

### なぜ AI に代替されにくいのか

振る舞いの正しさを AI が判定するには、次のいずれかが必要です。

* 業務要件 / ユーザー期待値が厳密に形式化されている
* 実行環境（業務フロー / ユーザー環境）を AI が再現できる

どちらも現状では難しいです。業務要件は暗黙知を多く含み、ユーザー体験は定量化しにくい感覚が混ざります。「だいたい合っている」と「本当に要件を満たしている」の差を詰めるのは、人間の仕事として残り続けます。

## 実務への落とし込み

ここまでの主張を、明日からのチーム運営に落とし込みます。重いリストラではなく、**時間配分の組み替え** です。

### プロセス: 早期マージ → QA ハンドオフ

従来の「長いレビュー → 慎重なマージ → テスト」から、**AI レビュー → 早期マージ → QA フェーズ** へ組み替えます。

ポイントは 3 つです。

1. **AI レビューを CI に組み込む**: PR が開いた瞬間に AI レビューが走る。指摘は開発者が自己修正する。
2. **人間レビューは「大枠」だけ**: 設計意図、業務整合の大方針、破壊的変更の妥当性。コード詳細は AI に任せる。
3. **QA を独立したフェーズとして設計する**: マージ後・本番前に「振る舞いの正しさ」を検証するフェーズを挟む。

これは LinearB のレポートが推奨している方向性とも一致します。

> Implement AI code review tools in CI pipelines as a first-pass filter, enabling faster human review.  
> （AI コードレビューツールを CI パイプラインに一次フィルタとして組み込めば、人間のレビューを速くできる）

### 小規模組織向け: 共有リソース型 QA

「そもそも QA チームがない」という組織も多いはずです。そこで提案したいのが **共有リソース型 QA** です。

* 専任 QA を 1 プロジェクトに 1 人置くのは、小規模チームには重い
* 代わりに **複数プロジェクトを横断する QA エンジニア** を育てる
* QA は「テスト実行係」ではなく「振る舞いの検証設計者」として位置付ける

具体的な配置例を 2 つ挙げます。

**例 1: 5 プロジェクト × 1 QA エンジニア** — 業務システムを 5 つ抱える中規模の受託開発会社。1 人の QA エンジニアが各プロジェクトの要件を把握し、マージ後の受け入れ検証を巡回するモデルです。開発者の負荷を減らしつつ、業務整合性の検証は必ず通せます。

**例 2: プロダクト横断の QA ギルド** — 自社プロダクトを複数持つ SaaS 企業。プロダクトごとに開発チームを置きつつ、QA は「ギルド」として組織横断で持ちます。週次ローテーションで各チームに入り、振る舞いの正しさをチェックします。

どちらも「QA を専任化するコストを、横断資源化で吸収する」モデルです。

### 明日から踏み出せる一歩

最後に、明日から試せる具体的アクションを挙げます。

* **PR 合格基準の見直し**:「AI レビュー通過 + 人間の大枠 OK + 基本動作確認」の 3 点セットに組み替える
* **レビュー時間の棚卸し**: 直近 1 ヶ月の PR コメントを「AI で代替可能 / 人間にしかできない」で分類する。代替可能な時間を QA に振り替える
* **QA フェーズの定義**: マージ後・本番前に「振る舞いの正しさ」を検証するフェーズを入れる。小規模なら開発者本人が QA 視点でチェックするだけでも効く

大掛かりな組織改編は不要です。レビューと QA の時間配分を 1 ヶ月単位で組み替えるところから始めれば十分です。

## まとめ

AI 時代の開発プロセスを整理します。

* **コードの正しさ** は AI に寄せる。人間の二重チェックは ROI が下がっている
* **振る舞いの正しさ** に人間をシフトする。業務整合性（業務システム）や UX（B2C サービス）など、システム種別で中身は変わる
* **QA の再定義**:「テスト実行係」ではなく「振る舞いの検証設計者」。専任が難しければ横断リソース型で
* **時間配分の組み替え**: レビューに使っていた時間を QA に付け替える。明日から試せる

最後に、読者のチームに問いを投げて本記事を閉じます。
