あなたのエージェントは今、どのモデルを呼んでいるだろう。そして「どのモデル」は、コードのどこかにベタ書きされていないだろうか。

先週、AIエージェント企業のLindyが、製品からClaudeを丸ごと外してDeepSeekに載せ替えた。理由は性能ではなく金だ。25人の会社で、AIの利用料が人件費を上回っていたという。CEOのFlo Crivelloは「事業の生存問題だった」と言っている。

同じ週、OpenRouterがエージェント向けのMCPサーバを出した。Claude Codeに「入力100万トークンあたり2ドル以内で、いちばん強いコーディングモデルは?」と聞くと、その場の最新の価格と性能から候補を返してくれる。値段はモデル登場時ではなく、質問した瞬間の値が使われる。

この2つは別々のニュースに見えて、同じ地殻変動の表と裏だと思う。「どのモデルを使うか」という決定が、ビルド時から実行時へ移りつつある、という話だ。

![モデル選択がビルド時固定から実行時ルーティングへ移る構造転換の図](https://static.zenn.studio/user-upload/2f81746e4f88-20260627.png)

## モデルが「固定の依存」から「値付けされる商品」に変わった

少し前まで、モデル選択はアーキテクチャ上の固定点だった。「うちはClaudeで作る」と決めれば、それは前提として動かない柱になる。プロンプトもツールも、その柱の上に積んでいく。

ところが今は様子が違う。まず価格が桁で割れる。同じ週に出たGPT-5.6を見ても、ベンダー自身が1つの世代をLuna($1/$6)・Terra($2.50/$15)・Sol($5/$30)と3段の価格で刻んできた。入力と出力で10倍以上の幅がある。次に品質が追いついてきた。Snowflakeの分析では、GLM-5.2のような中国製モデルがタスクによってはClaudeに肉薄し、価格性能比では上回る場面も出ている。

つまりモデルは、設計図にベタ書きする「依存先」ではなく、リクエストごとに値付けされて選び直される「商品」に変わった。Lindyがやったのは、この商品の乗り換えだ。乗り換えを手作業の大移植としてやると数週間かかる。だがこれを設計に最初から織り込めば、移植は設定の差し替えで済む。それが実行時ルーティングという考え方だ。

![GPT-5.6のLuna・Terra・Solの3段価格を比較した棒グラフ](https://static.zenn.studio/user-upload/f4bf3b4b0bc1-20260627.png)

## ベタ書きをやめて、ルータを1枚かませる

やること自体は単純で、エージェントとモデル呼び出しの間に薄いルーティング層を1枚入れる。呼ぶ側は「コーディング用の安いやつ」くらいの抽象的な要求だけを出し、それを実際にどのモデルへ落とすかはルータが決める。

```
# 抽象的な「意図」だけを投げる。実モデル名は呼ぶ側に書かない。
answer = route(
    task="code",          # 用途
    max_input_usd=2.0,    # 入力100万トークンあたりの上限
    quality="high",       # 求める品質ティア
).complete(prompt)
```

ルータの中身は2つだ。1つはポリシーで、コスト上限・レイテンシ・品質ティアの条件から候補を絞る。もう1つはフォールバックチェーンで、1次モデルが落ちた、遅すぎる、予算を超えた、というときに次の候補へ自動で滑り落ちる。呼ぶ側のコードはモデル名を一切知らないので、来月もっと安くて強いモデルが増えたら、ルータの候補表を1行足すだけで全体が乗り換わる。

![ルータが安いモデルで試し検証ゲートを通し不合格なら上位モデルへ昇格するフロー図](https://static.zenn.studio/user-upload/3e9ba281a752-20260627.png)

## 本当の難所は「検証」のほうに移る

ここで多くの実装が見落とすのが、モデルを差し替えられる自由を手に入れた瞬間、出力品質の保証が自分の責任に回ってくることだ。Claude固定なら「Claudeがそう言ったから」で通っていた出力が、ルータが今日はDeepSeekを選んだ、では同じ品質とは限らない。自由とセットで、品質のばらつきも引き受けることになる。

だから実行時ルーティングは、必ずタスク単位の検証ゲートとセットで設計する。出力にスキーマや制約を課して機械的に検証し(JSONが妥当か・テストが通るか・型が合うか)、落ちたら品質ティアを1段上げて呼び直す。安いモデルでまず試し、ダメなものだけ高いモデルへ昇格させる。さらに一部のトラフィックを上位モデルへカナリアで流し、安いモデルの劣化を継続監視する。

この「安いので試す、検証する、ダメなら昇格する」というループが効くのは、現実の多くのタスクが弱いモデルでも通ってしまうからだ。GPT-5.6のメモでも、同じ性能を出力トークン3分の1で出すという効率改善が報告されている。全部を最上位モデルで回す必要はない。安く通る分を安く通し、通らない分だけ高い枠を使う。この振り分けが、Lindyのような会社のコスト曲線を文字どおり地面まで落とす。

## 設計の重心はどこへ動いたか

モデルをベタ書きしていた頃、エンジニアリングの価値はプロンプトとオーケストレーションにあった。それは今も要る。ただ重心はもう一段ずれた。どのモデルをいくらで呼び、出力をどう検証し、ダメなときどう昇格させ退避させるか。この「ルーティングと検証の層」が、これからのエージェントで差がつく場所になる。

モデル本体はベンダーが毎月のように強く、安くしてくれる。こちらが作るべきは、その変化を毎リクエスト取り込んで、品質を落とさずに一番安い経路を選び続ける仕組みのほうだ。来月もっと安くて強いモデルが出たとき、コードを書き換えずに乗り換えられるか。それが、ベタ書きをやめた設計に残るただ一つの問いになる。
