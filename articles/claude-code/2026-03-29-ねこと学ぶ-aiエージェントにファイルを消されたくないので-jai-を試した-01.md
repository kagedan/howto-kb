## Claude Code に `.bashrc` を消された日

先月、Claude Code にドットファイルの整理を頼んだ。「`.bashrc` と `.bash_profile` の重複してる設定をまとめて」という、わりと日常的な指示。

結果、`.bashrc` が消えた。

正確には「まとめた」結果として `.bash_profile` に統合されたんだけど、元の `.bashrc` を消す前にバックアップを取るという発想がエージェント側になかった。しかもシンボリックリンクで他のファイルから参照されてた設定まで巻き添えで壊れて、ターミナルを開くたびにエラーが出る状態に。復旧に40分かかった。

40分で済んだのは不幸中の幸いで、世の中にはもっとひどい事例がある。Anthropic の GitHub Issues を漁ると「ホームディレクトリ全消去」「家族写真の消失」「100GB超のデータロスト」みたいな報告がゴロゴロ出てくる。AIエージェントにマシンアクセスを渡すのが当たり前になった今、これは構造的な問題だ。

で、対策を探してたら `jai` というツールを見つけた。Stanford Secure Computer Systems (SCS) group が作ってる、AIエージェント向けの軽量サンドボックス。

## jai はコンテナじゃない

最初に言っておくと、jai は Docker でも VM でもない。Dockerfile を書く必要もないし、イメージをビルドする必要もない。

これだけ。`jai` のあとにコマンドを書くと、そのプロセスが隔離された環境で動く。

仕組みはこう:

* **カレントディレクトリ**: 読み書き自由（ここは作業場所だから当然）
* **ホームディレクトリ**: copy-on-write オーバーレイで保護。書き込みは別の `changes` ディレクトリに溜まり、元ファイルには触れない
* **`/tmp`、`/var/tmp`**: プライベートな領域が割り当てられる
* **それ以外のファイルシステム**: リードオンリー

つまり「仕事に必要なCWDだけ書き込ませて、それ以外は壊せないようにする」という割り切った設計。Docker みたいに「完全に隔離された世界を作る」のではなく、「事故の被害範囲を限定する」という思想。

公式サイトの設計哲学が印象的だった。"If containment isn't easier than YOLO mode, nobody will bother."（隔離が YOLO モードより面倒なら誰も使わない）。わかる。Docker でClaude Code を動かす設定を書こうとして3回挫折した身としては、この姿勢はありがたい。実際、Docker でやろうとすると「ホスト側のファイルをどこまでマウントするか」「Node.js のバージョン合わせ」「権限周りの設定」あたりで毎回ハマる。jai はそういうのを全部すっ飛ばして、Linux のマウント名前空間で直接隔離する。

## インストールしてClaude Codeで使ってみた

jai は Linux 専用で、カーネル 6.13 以上が必要。ここが最大のハードル。

```
git clone https://github.com/stanford-scs/jai.git
cd jai
./autogen.sh
./configure
make
sudo make install
sudo systemd-sysusers
```

ビルドには `autoconf` と `automake` が必要。自分は Ubuntu 24.04（WSL2ではなくネイティブの検証用マシン）で試した。カーネルバージョンは `uname -r` で確認できる。6.13 未満だとそもそも動かない。

インストールしたら、あとは本当にコマンドの前に `jai` をつけるだけ。

```
# Claude Code をサンドボックス内で起動
jai claude

# 普通のシェルをサンドボックス内で起動（動作確認用）
jai bash
```

Claude Code が起動して、普通に使える。CWD 配下のファイルは読み書きできるし、`git` も `npm` も動く。違いは、ホームディレクトリへの変更が元のファイルに影響しないこと。

jai の中にいるかどうかは環境変数でわかる。`JAI_MODE` が設定されてたらサンドボックス内。

地味に便利なのが、PID 名前空間も分離されてること。jai 内のプロセスから外のプロセスを `kill` したり `ptrace` したりできない。ファイルだけじゃなくプロセスの隔離もやってくれてるのは、ちゃんとしてるなと思った。

## 3つのモード、どれを使うか

jai には casual / strict / bare の3つのモードがある。`-m` フラグで切り替える。

```
jai -m casual claude    # デフォルト
jai -m strict claude    # 厳格モード
jai -m bare claude      # ベアモード
```

にゃんだかんだ表で見たほうが早い。

|  | casual (デフォルト) | strict | bare |
| --- | --- | --- | --- |
| ホームディレクトリ | CoWオーバーレイで保護（読める） | 空 | 空 |
| 実行ユーザー | 自分 | `jai` 専用ユーザー | 自分 |
| `.env`/SSH鍵 | 読めるが書き換え不可 | 見えない | 見えない |
| NFS対応 | OK | NG | OK |
| 用途 | 日常開発 | 機密性が必要な場面 | NFS環境での代替 |

**casual** が日常向け。ホーム内のファイルは読めるけど、書き込みはオーバーレイに吸収されるので元ファイルは安全。**strict** はガチの隔離で、専用の非特権ユーザーとして実行される。**bare** は strict のNFS非対応を回避するための選択肢で、空のホームだけど自分のUID で動く。

自分は普段 casual で使ってる。理由は単純で、Claude Code が `.claude/` 配下の設定ファイルを読めないと CLAUDE.md のルールを認識してくれなくなるから。strict だと空のホームになるのでそれが飛ぶ。

## `rm -rf ~` をやってみた

jai の売りは「ホームディレクトリを壊せない」なので、本当に壊せないのか試した。

```
# まず jai なしの環境で確認（やらないでください）
# rm -rf ~  ← 絶対にやるな

# jai 内で実行
jai bash
rm -rf ~
ls ~
```

結果: jai の中では `~` の中身が消えたように見える。`ls ~` すると空っぽ。でも `exit` して jai の外に戻ると、ホームディレクトリは完全に無傷。copy-on-write のオーバーレイ上で「削除した」という変更が記録されただけで、元のファイルには一切触れてない。

にゃるほど、これは安心感がある。

もうちょっと意地悪なテストとして、jai 内から `/etc/hosts` を書き換えようとしてみた。

```
jai bash
echo "127.0.0.1 evil.example.com" >> /etc/hosts
# bash: /etc/hosts: Read-only file system
```

リードオンリーなので弾かれる。ファイルシステムレベルで保護されてるから、`sudo` を使おうが突破できない（jai 内では権限が制限されてるので `sudo` 自体が機能しない）。

ちなみに、jai を抜けたあとにオーバーレイに溜まった変更を確認することもできる。`$HOME/.jai/default.changes` に差分が残ってるので、「エージェントがホームに対して何をしようとしたか」を事後的に監査できる。冒頭の `.bashrc` 事件も、jai 越しだったら「あ、消そうとしてるな」と気づけたはず。

## 正直な不満と限界

使ってみて「これは万能じゃないな」と思った点もある。

**Linux 限定**。macOS では動かない。自分のメイン開発機は Mac なので、これが一番痛い。ネイティブ Linux の検証用マシンでしか使えないのは、正直なところ常用ハードルが高い。カーネル 6.13 以上という要件も、2026年3月時点だとまだ新しめのディストリビューションじゃないと満たせない。

**ネットワークは隔離されない**。jai はファイルシステムの保護がメインで、ネットワークアクセスは制限しない。つまり、エージェントが `curl` で外部にデータを送信するような攻撃は防げない。これは設計上の割り切りだと思うけど、知っておく必要がある。

**CWD は完全に無防備**。カレントディレクトリは読み書き自由なので、プロジェクトのソースコードを消される可能性はある。jai が防いでくれるのは「プロジェクト外への被害拡大」であって、プロジェクト内のファイルは自己責任。まあ、そこは git があるからいいんだけど。

**設定ファイルの書き方がまだドキュメント薄い**。`.jai/claude.conf` みたいなファイルでコマンドごとの設定ができるんだけど、サンプルが少なくて手探り感がある。`dir` ディレクティブで追加のディレクトリにアクセス権を付与できるのは便利なんだけど、最初それに気づかなくて「なんで venv のパッケージが読めないんだ」と10分悩んだ。

```
# ~/.jai/claude.conf の例
conf .defaults
mode casual
dir /home/me/shared-venvs
```

`conf .defaults` を先頭に書くのがポイント。公式ドキュメントにも「全設定ファイルはこの行から始めるべき」と書いてある。これを書き忘れてデフォルトのマウント設定が丸ごと消えて焦った。

こういうのは公式に Quick Start 的なレシピ集があると助かるんだけど、まだ若いプロジェクトなので仕方ない部分かもしれない。

不満はあるけど、総合的には「コマンドの前に `jai` って打つだけ」で事故のダメージが限定されるのは、コストに対してリターンがでかい。リモート開発サーバーやCI環境なら Linux 限定も関係ないし、自分は検証用の Ubuntu マシンで Claude Code を動かすときは常に `jai claude` 経由にするようにした。Mac では相変わらず裸で動かしてるけど、怖いファイル操作をさせるときは SSH で Linux に逃がすようにしてる。

macOS 対応が来たら本気で常用すると思う。誰かポートしてくれにゃいかな。
