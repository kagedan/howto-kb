## はじめに

「Claude Code、すごく便利なんだけど…費用が気になる」

そんな声をよく聞きます。実際、Claude Codeを本格的に使い始めると、気づけば月に$100〜$200のAPIコストが発生することも珍しくありません。

しかし、ちょっとした設定と使い方の工夫で、**同じ生産性を維持しながらコストを大幅に削減できます**。本記事では、`/compact`・`.claudeignore`・モデルルーティングなど、実践的なコスト最適化テクニックを体系的に解説します。

---

## 1. まずClaude Codeの料金体系を正確に把握する

コスト最適化の第一歩は、料金構造を正確に理解することです。

### サブスクリプションプラン

| プラン | 月額 | 向いているケース |
| --- | --- | --- |
| Pro | $20 | 週数回のライトユーザー |
| Max 5x | $100 | 平日ほぼ毎日使う個人開発者 |
| Max 20x | $200 | 大規模開発・チーム利用 |

### API従量課金（主要モデル）

| モデル | 入力 | 出力 | 特徴 |
| --- | --- | --- | --- |
| Haiku | $1/100万トークン | $5/100万トークン | 高速・低コスト |
| Sonnet 4.5 | $3/100万トークン | $15/100万トークン | バランス型 |
| Opus 4.5 | $5/100万トークン | $25/100万トークン | 最高精度 |

公式データによると、**平均的な開発者は1日$6、90%のユーザーは$12以下**に収まっています。しかし、大規模リファクタリングや長時間のセッションを続けると、あっという間に上限を超えることがあります。

## 2. モデルルーティング戦略（最大のコスト削減ポイント）

コスト最適化で**最も効果が高い**のが、タスクに応じたモデルの使い分けです。

OpusとHaikuでは入力コストが**5倍**も違います。何でもOpusを使っていると、Haikuで十分なタスクにも高い料金を払い続けることになります。

### モデル選択の判断基準

| タスク | 推奨モデル | 理由 |
| --- | --- | --- |
| アーキテクチャ設計・複雑な推論 | Opus | 深い思考が必要 |
| 一般的なコーディング・コードレビュー | Sonnet | バランス◎ |
| ファイル探索・簡単な編集・grep | Haiku | 速度とコスト優先 |
| ドキュメント生成・コメント追加 | Sonnet/Haiku | 創造性不要 |

### `/model` コマンドで切り替え

```
/model haiku     # Haikuへ切り替え（探索タスク向け）
/model sonnet    # Sonnetへ切り替え（通常作業）
/model opus      # Opusへ切り替え（設計・深い推論）
```

### Thinking Budgetのコントロール

Opusの「拡張思考モード」は精度が上がる反面、トークン消費が膨大になります。環境変数で制限をかけましょう：

```
# .envや~/.bashrcに追加
export MAX_THINKING_TOKENS=0           # 通常作業時は無効化
export MAX_THINKING_TOKENS=10000       # 設計検討時は少量に制限
```

## 3. Context Rot を防ぐ `/compact` の使い方

### Context Rot とは？

Claude Codeを長時間使い続けると、会話履歴が積み上がり、**コンテキストウィンドウの40%を超えたあたりから精度が低下**する現象が起きます。これを「Context Rot（コンテキスト腐敗）」と呼びます。

コンテキストが肥大化すると：

* 重要な指示が「記憶の彼方」に埋もれてしまう
* AIの注意力（Attention）が分散し、応答品質が落ちる
* そして当然、**トークン消費量も激増**する

### `/compact` で会話を圧縮する

このコマンドで会話履歴を要約・圧縮します。実測で**60〜87%のトークン削減**が報告されています。

**使うタイミングの目安：**

* 1つの作業フェーズが終わったとき（例：機能実装 → テスト記述へ移行する前）
* コンテキストが「重くなってきた」と感じたとき
* 新しいタスクを始める前

### 自動コンパクトの設定

毎回手動で実行するのが面倒なら、環境変数で自動圧縮を設定できます：

```
# コンテキストが50%に達したら自動圧縮（デフォルトは95%）
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50
```

### ⚠️ 圧縮前に重要情報をCLAUDE.mdへ

`/compact` は会話を要約するため、**細かい指示や決定事項が消えることがあります**。重要なルールや設計決定は、必ずCLAUDE.mdに永続化してから圧縮しましょう。

```
<!-- CLAUDE.mdに追記する例 -->
## 決定事項（2026-03-26）
- DB接続は必ずコネクションプールを使う
- APIレスポンスはスネークケースで統一
```

### `/clear` との使い分け

| コマンド | 動作 | 使うタイミング |
| --- | --- | --- |
| `/compact` | 会話を要約して継続 | 同じタスクの続きをやるとき |
| `/clear` | 完全リセット | 別のプロジェクト・全く新しいタスクのとき |

## 4. `.claudeignore` でファイル読み込みを最適化

### なぜ `.claudeignore` が重要か

Claude Codeはプロジェクトのファイルを積極的に読み込みます。`node_modules` の何千もの依存ファイルや、ビルド成果物（`dist/`）を毎回読み込んでいたら、それだけで膨大な入力トークンを消費します。

**`.claudeignore`**（`.gitignore` と同じ書式）を作成することで、不要なファイルの読み込みを防げます。

### サンプル `.claudeignore`

```
# 依存関係
node_modules/
vendor/
.venv/
__pycache__/

# ビルド成果物
dist/
build/
out/
.next/
*.pyc
*.class

# ログ・一時ファイル
*.log
*.tmp
.DS_Store

# 大きなデータファイル
*.csv
*.json.bak
data/raw/

# ロック・キャッシュ
package-lock.json
yarn.lock
*.lock
.cache/
```

`.gitignore` の内容を `.claudeignore` にコピーするだけで、すぐに効果が出ます。

## 5. プロンプトキャッシュを最大活用する

### プロンプトキャッシュの仕組み

Claude APIには**プロンプトキャッシュ**機能があります。一度処理したコンテキスト（コードベースの読み込み内容など）は、次回のリクエストで**90%割引**で再利用できます。

つまり、「同じファイルを何度も参照するセッション」では、初回以降のコストが激減します。

### キャッシュを壊さないための注意点

**⚠️ 以下の操作でキャッシュが無効化されます：**

1. **`thinking budget` の変更**（`MAX_THINKING_TOKENS`の変更）
2. **CLAUDE.mdの頻繁な書き換え**
3. **システムプロンプトの大幅変更**

CLAUDE.mdは**一度確定したら安定させる**のが、プロンプトキャッシュを最大活用するコツです。

## 6. MCPサーバー・ツール登録数の最適化

### ツール定義がコンテキストを圧迫する

MCPサーバーを登録すると、そのツール定義が**システムプロンプトに自動注入**されます。10個のMCPサーバーを有効にすると、セッション開始時のトークンが相当量増加します。

**推奨上限：アクティブツール10個 / ツール定義80個以下**

### 不要なMCPを無効化する

`.claude/settings.json` でプロジェクト単位の設定が可能です：

```
{
  "mcpServers": {
    "github": { "enabled": true },
    "database": { "enabled": true },
    "slack": { "enabled": false },   // 今は使わない
    "jira": { "enabled": false }     // 今は使わない
  }
}
```

作業に不要なMCPは積極的に無効化しましょう。

## 7. コスト削減の環境変数チートシート

```
# ~/.bashrc または .env に追加

# コンパクト設定
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50   # 50%で自動圧縮（推奨）

# Thinking Budgetの制御
export MAX_THINKING_TOKENS=0                # 通常時は無効化

# 出力トークン上限（デフォルト32,000 → 半減）
export CLAUDE_CODE_MAX_OUTPUT_TOKENS=16000

# 非必須モデル呼び出しを無効化
export DISABLE_NON_ESSENTIAL_MODEL_CALLS=1
```

## 8. サブスクリプション vs API 選択フロー

自分に最適なプランを選ぶ判断基準です：

```
1日の使用時間はどのくらい？
├── 2時間以上（ほぼ毎日）→ Max 5x（$100/月）を検討
│   └── チーム10名以上 → Team/Enterprise
├── 1〜2時間（週数回）→ Pro（$20/月）で様子見
└── CI/CDや自動化バッチが主体 → API従量課金（$6/日平均）
```

**月$30未満なら Pro、月$50〜$100かかるなら Max 5x が割安**になることが多いです。`/cost` コマンドで現在のセッションコストを確認しながら判断しましょう。

```
/cost    # 現在のセッションのコスト確認
/usage   # トークン使用量の詳細
```

## まとめ：今すぐできる5つのアクション

| # | アクション | 効果 |
| --- | --- | --- |
| 1 | `.claudeignore` を作成（.gitignoreをコピー） | 入力トークン大幅削減 |
| 2 | `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50` を設定 | Context Rot防止 |
| 3 | 探索・grep系タスクは `/model haiku` を使う | モデルコスト5分の1に |
| 4 | CLAUDE.mdを500行以下に整理・安定化 | キャッシュ最大化 |
| 5 | 使っていないMCPをdisabledに | システムプロンプト軽量化 |

Claude Codeのコストは「使い方の工夫」で大きく変わります。特に **モデルルーティング** と **`.claudeignore` の設定** は即効性が高いので、まずこの2つから試してみてください。
