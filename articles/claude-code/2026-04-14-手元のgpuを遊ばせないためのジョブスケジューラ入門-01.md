## 1. はじめに

**ジョブスケジューラ**は、限られた計算資源でタスク順序を整理するためのツールです。実行待ち、資源の競合、状態管理、履歴の追跡などを一貫して扱えるジョブスケジューラは、共有の計算サーバを使う**HPC環境**（High Performance Computing, いわゆるスパコン）では昔から使われてきました。計算処理が長時間になり、試したいパターンが複雑になるケースでは、多数の計算ジョブを登録・管理することで効率的に実験を回すことができます。

一方、最近では**Claude Code**や**Codex**など、Coding Agentの進化と開発速度の加速により、個人の開発環境でもこのようなジョブ管理をすると便利なケースが増えていると感じています。1台の計算機でも、順番待ち、GPUなどのリソース衝突回避、経過を追える実行単位は、自動化するには有効といえます。

## 2. Slurmとは

![](https://static.zenn.studio/user-upload/e1dd9c165ff8-20260413.png)

ジョブ管理の世界で、もっともよく知られた存在の1つが**Slurm Workload Manager**です。Slurmは**S**imple **L**inux **U**tility for **R**esource **M**anagementの略で、ローレンス・リバモア国立研究所を中心に開発されました。

Linux向けに設計された強力なジョブスケジューラで、もともとは大規模な並列計算を効率的に管理するために作られ、多くのスーパーコンピュータや研究機関で採用されています。スパコンのランキングである[TOP500](https://top500.org/)でも、半分以上がSlurmを採用しているとされます。

<https://slurm.schedmd.com/overview.html>

Slurmは**オープンソースソフトウェア**として公開されていて、無料で利用でき、主要なLinuxディストリビューション向けにパッケージが提供されています。リソースの割り当て、割り当て済みノード上でのジョブの開始と監視、待機ジョブのキュー管理、という3つを中核機能として設計されています。

![](https://static.zenn.studio/user-upload/5864fff5ae7e-20260413.png)  
*Slurmの構成（[slurm.schedmd.com](https://slurm.schedmd.com/)の図を翻訳）*

利用者（クライアント）からみると、簡易なコマンドだけでジョブを投入・管理することができます。例えば、`sinfo`で計算ノードやパーティションの状態を確認し、`sbatch`でバッチジョブを投入し、`srun`でジョブを起動し、`squeue`で待機中や実行中のジョブを確認する…といった具合です。この昔から変わらない操作体系が広く共有されている（=枯れている）ことも、Slurmが事実上の標準として使われている大きな理由です。

Slurmが便利なのは、資源指定、キューイング、状態管理、ジョブ制御を1つの操作体系で扱えるところです。GPUを1枚使いたい、メモリは16GB欲しい、30分以内で終わるはず、といった条件を利用者はジョブに付けて投入し、あとはスケジューラに実行状態管理を任せられます。

## 3. ローカル開発のためのSlurm風サブセット`slotd`

一方で、ローカルマシン、例えば手元のデスクトップPCや[vast.ai](https://vast.ai/)などで借りたマルチGPUマシンでの用途だけを考えると、Slurm環境を構築するには少し過剰になります。Linux向けのシンプルなジョブスケジューラとしては他に[GPU Task Spooler](https://github.com/justanhduc/task-spooler)などが有名ですが、使用感はHPC向けのジョブスケジューラとは異なります。

普段使い慣れているSlurmと似た操作感でローカルで気軽に使えるジョブスケジューラが欲しいな…と感じていたので、今回Slurm風の簡易ジョブスケジューラを開発することにしました。

![](https://static.zenn.studio/user-upload/56dd6008453f-20260413.png)  
<https://github.com/ymgaq/slotd>

こちらの`slotd`は、Slurmの使い勝手のうち、ローカルマシンで使いそうな機能を切り出すことを狙って作った、Rust製の軽量ジョブスケジューラです。対象は単一ノード、単一ユーザーのローカルワークステーションで、クラスタ管理機能は持ちません。そのかわり、手元のPCやSSH先のノードで実験を回すための基本機能を軽量に提供します。

`slotd` が提供するユーザー向けコマンドはSlurmと共通しています。

* `sbatch`
* `srun`
* `salloc`
* `squeue`
* `sacct`
* `scontrol`
* `scancel`
* `sinfo`

これらはコマンド名だけでなく、使い方やオプションもSlurmと揃えています。たとえば`sbatch`でバッチジョブを投入し、`squeue`で確認し、`sacct`で完了状態を見る、という基本的な流れは共通です。ローカル環境向けの実装ですが、操作の感覚はできるだけSlurmと一致するようにしています。

内部構成はかなりシンプルで、`slotd`はRust製の単一バイナリでとして提供されます。ローカルdaemonがUnix domain socket経由でリクエストを受け付け、状態はSQLiteに永続化されます。実行ホストも1台だけです。割り切った設計として、controllerとworkerが分かれているわけでもなく、リモートノード起動の仕組みもありません。

それでも、開発に必要な主要なSlurmの機能は実装してあり、CPU、メモリ、GPUの予約ベース実行、バッチジョブ、対話実行、allocation、job array、dependency、`--begin`、`--exclusive`、`--requeue`といった機能があり、GPUの自動検出やローカルfeature constraintにも対応しています。

HPC環境は使わないけど、ジョブスケジューラを勉強したい…という人には入門も兼ねたちょうどいいツールとして利用できると期待しています。詳細を知りたい方向けに[ドキュメントページ](https://ymgaq.github.io/slotd/ja/)も整備しています。

ここからは具体的な使い方をみていきましょう。

## 4. `slotd`のインストール方法

`slotd`の導入要件は比較的少なく、Linux（あるいはWSL）上で[Rust開発環境](https://rust-lang.org/tools/install/)が使えれば動かせます。daemonを管理したい場合はsystemdコマンドの権限も必要です。また、GPUの自動検出を行いたい場合は`nvidia-smi`が実行できる必要があります。

Rust開発環境がない場合

[公式](https://rust-lang.org/tools/install/)からインストールできます。

```
curl https://sh.rustup.rs -sSf | sh
```

`slotd`のインストールには、リポジトリをcloneしてから付属スクリプトを実行します。

```
git clone https://github.com/ymgaq/slotd.git
cd slotd
./scripts/install.sh
```

installスクリプトの実行内容

このスクリプトを使うと、`slotd`本体をcargoでreleaseビルドし、`~/.local/bin`以下に本体とSlurm互換コマンドのaliasを配置します。また、runtime rootを`~/.local/share/slotd`に作成し、設定ファイルを`~/.config/slotd/slotd.env`に書き出してくれます。`systemd --user`が使える環境であれば、ユーザーサービスのインストールと起動までまとめて行われます。

インストールが済んだら、まずdaemonが動いていることを前提に基本コマンドを確認します。  
これらのコマンドの仕様は基本的にはSlurmと同様です。

```
sinfo
# PARTITION  | HOSTNAMES       | STATE | FEATURES                 | GRES_USED
# cpu        | localhost       | idle  | cpu                      | N/A
# gpu*       | localhost       | idle  | cpu,rtx5090              | gpu:RTX5090:1(IDX:N/A)

squeue
#        JOBID | PARTITION | NAME       | USER         | ST |        TIME | NODELIST(REASON)

sacct
#        JobID | Partition | JobName    | User         | State      | ExitCode
```

初回起動直後であれば、`sinfo`には設定済みパーティションが表示され、`squeue`と`sacct`は空になっているはずです。

## 5. 基本的なコマンド・使い方

Slurm系の基本的な考え方として、ジョブにはインタラクティブジョブ、バッチジョブ、パラレルジョブ、アレイジョブなど、いくつか種類があります。再現性のある実行をあとで流したいなら`sbatch`、対話的に短く試したいなら`srun`や`salloc`、似た実験を多数回したいならアレイジョブ、という切り分けです。

また、日常的によく使うコマンドの役割もある程度決まっています。`sinfo`は使える資源を見るコマンド、`sbatch`はジョブを投入するコマンド、`squeue`は今動いているジョブを見るコマンド、`sacct`は終わったジョブの履歴を見るコマンドです。加えて、不要になったジョブを止めるには`scancel`を使います。日常的にはこのあたりを行き来するだけで、かなりのことができます。

### 5.1 `sinfo`で実行可能な資源を見る

最初に確認したいのは、そのマシンでどのパーティションが有効になっているかです。ここで使うのが`sinfo`です。

```
sinfo
# PARTITION  | HOSTNAMES       | STATE | FEATURES                 | GRES_USED
# cpu        | localhost       | idle  | cpu                      | N/A
# gpu*       | localhost       | idle  | cpu,rtx5090              | gpu:RTX5090:1(IDX:N/A)
```

`slotd`では、CPU向けやGPU向けのパーティションが設定されていれば、それぞれが表示されます。ここで何を見るかというと、そのマシンでどの種類のジョブを投げられるかです。軽い前処理なら`cpu`、学習や推論なら`gpu`というように、ジョブ投入前に実行先のイメージを持つために使います。

また、`sinfo -l`でより詳細なリソース状況を確認できます。

```
sinfo -l
# PARTITION  | HOSTNAMES       | STATE | FEATURES                 |   CPUS | CPU_ALLOC |     MEMORY |  MEM_ALLOC |   GPUS | GPU_ALLOC | RUNNING | PENDING | GRES_USED
# cpu        | localhost       | idle  | cpu                      |     32 |         0 |     54218M |         0M |      0 |         0 |       0 |       0 | N/A
# gpu*       | localhost       | idle  | cpu,rtx5090              |     32 |         0 |     54218M |         0M |      1 |         0 |       0 |       0 | gpu:RTX5090:1(IDX:N/A)
```

ジョブの実行が始まらないときも、まずここから確認するとよいです。要求したCPU数やメモリ量、GPU数、実行時間が大きすぎないか、そもそも該当パーティションが使える状態か、をチェックするとよいでしょう。

### 5.2 `sbatch`でバッチジョブを投入する

実際の計算をバッチジョブとして登録する基本コマンドが`sbatch`です。対話的にその場で走らせるのではなく、順番待ちや資源管理を含めてスケジューラに任せたい処理は、これで投入します。

もっともクイックに実行する形は、`--wrap`で1行のコマンドを渡す方法です。

```
sbatch --wrap 'echo hello from slotd'
```

投入に成功すると、次のような出力になります。

ここで返ってくる数字は`job id`です。以後はこのidを使って状態確認や詳細表示を行います。まずはこの最小例で、ジョブ投入から完了確認までの流れを一通り試すのがおすすめです。

実際の運用では、`--wrap`だけでなくジョブスクリプトを使うことも多くなります。長い学習コマンドや複数行の前処理、環境変数設定を含めたい場合は、そのほうが見通しがよいからです。

たとえば次のようなシェルスクリプトを用意しておくとします。

```
#!/usr/bin/env bash
echo "start training"
python train.py --config configs/base.yaml
echo "finished"
```

これを`train.sh`として保存しておけば、次のようにスクリプトを投入できます。

```
sbatch --job-name train-base --partition gpu --cpus-per-task 8 \
    --mem 16G --gpus 1 --time 04:00:00 train.sh
```

`sbatch`でよく使うオプションは下記のようになります。

| オプション | 意味 | 典型的な用途 |
| --- | --- | --- |
| `-J`, `--job-name` | ジョブ名を付ける | 学習名や実験名を識別しやすくする |
| `-p`, `--partition` | 実行先パーティションを選ぶ | `cpu`と`gpu`など、リソースを使い分ける |
| `-c`, `--cpus-per-task` | 1タスクあたりのCPU数を指定する | DataLoaderや前処理のCPU数を調整する |
| `--mem` | 必要メモリを指定する | OOM回避のために必要量を予約する |
| `-G`, `--gpus` | 必要GPU数を指定する | 学習や推論でGPUを確保する |
| `-t`, `--time` | 制限時間を指定する | runaway jobを防ぐ |
| `-o`, `--output` | 標準出力の保存先を指定する | 実験ログをjob単位で保存する |
| `--wrap` | 1行コマンドをそのまま実行する | 軽い確認や短い評価ジョブを流す |

典型的なGPU学習の文脈では、`--partition`、`--cpus-per-task`、`--mem`、`--gpus`、`--time`、`--output`あたりをよく使います。どの資源をどれだけ使うのかを明示しておくと、他のジョブと競合しにくくなり、あとから実験条件も追いやすくなります。

#### 5.2.1 ジョブ投入の具体例

`sbatch`で投入するジョブは、スクリプトの先頭に`#SBATCH`でオプションを渡すこともできます。学習ジョブの設定をファイル側にまとめておけるので、再実行や共有がしやすいからです。

たとえば次のようなスクリプトを`train_vit.sh`として用意します。

```
#!/usr/bin/env bash
#SBATCH --job-name train-vit
#SBATCH --partition gpu
#SBATCH -cpus-per-task 8
#SBATCH --mem 16G
#SBATCH --gpu 1
#SBATCH --time 08:00:00
#SBATCH --output logs/%j.out

echo "hostname: $(hostname)"
nvidia-smi
python train.py --config configs/vit_base.yaml
```

これを次のように投入します。

この例では、`gpu`パーティションに対して、8CPU、16GBメモリ、GPU1枚を要求し、最大8時間まで実行できるようにしています。`-o logs/%j.out`としておくと、`%j`がjob idに展開されるので、`logs/17.out`のようにジョブごとにログが分かれます。学習前に`nvidia-smi`を打っているのは、実行時のGPU状態をログに残しておくためです。

このときの意図は、学習スクリプトをただ走らせるのではなく、資源条件付きのジョブとして登録することです。あとから見返したときにも、このジョブがGPU1枚、16GBメモリ、8CPUで動く想定だったことが明確に残ります。実験ごとに`train_vit.sh`、`train_llm.sh`、`eval_vit.sh`のようにスクリプトを分けておくと、ローカルでもかなり整理された運用ができます。

### 5.3 `squeue`で今動いているジョブを見る

ジョブが待機中または実行中なら、`squeue`で確認できます。

典型的な表示は次のようになります。

```
JOBID | PARTITION | NAME        | USER | ST | TIME | NODELIST(REASON)
17    | gpu       | train-vit   | ...  | R  | 8:41 | localhost
18    | gpu       | eval-vit    | ...  | PD | 0:00 | Resources
19    | cpu       | preprocess  | ...  | R  | 0:53 | localhost
20    | gpu       | export-onnx | ...  | PD | 0:00 | Dependency
```

ここで重要なのは`ST`と`NODELIST(REASON)`です。`ST`は状態を表していて、`R`なら実行中、`PD`なら待機中です。待機中なら最後の列に理由が出ます。たとえば資源不足なら`Resources`、開始時刻待ちならそれに応じた理由が表示されます。つまり`squeue`は、今なにが走っていて、なにが詰まっているかを見るためのコマンドです。

上の例だと、`train-vit`と`preprocess`はすでに動いていて、`eval-vit`はGPUが空くのを待っており、`export-onnx`はdependency付きなので前段のジョブ完了待ちだとわかります。こうした読み方ができるようになると、ローカル環境でもジョブの混み具合をかなり直感的に把握できます。

もしジョブがずっと始まらない場合は、まず`ST`が`PD`になっていないか、そのとき`NODELIST(REASON)`にどんな理由が出ているかを見ます。`Resources`なら資源待ち、`Dependency`なら依存ジョブ待ち、というようにかなり直接的に状況が出るので、最初の切り分けとしてとても有効です。

夜間に複数の学習ジョブを積んでおいたときや、Agentがいくつも評価ジョブを投げたときには、この一覧だけでかなり状況がわかります。ローカル1台運用でも、実験の本数が増えると一覧性の価値が大きくなります。

### 5.4 `scancel`でジョブを止める

もし実験を途中で止めたいときは`scancel`を使います。

これはjob id `17`のジョブを停止する、という意味です。パラメータを間違えた、想定以上にメモリを使っている、別の実験を優先したい、といった場面で使います。ジョブスケジューラを使うと、走らせるだけでなく安全に止められるのも大きな利点です。

### 5.5 `sacct`で終了済みジョブの履歴を見る

完了後の情報は`sacct`で見られます。

```
JobID | Partition | JobName      | User | State     | ExitCode
14    | cpu       | preprocess   | ...  | COMPLETED | 0:0
15    | gpu       | train-vit    | ...  | FAILED    | 1:0
16    | gpu       | train-vit    | ...  | FAILED    | 1:0
17    | gpu       | train-vit    | ...  | CANCELLED | 0:15
```

`squeue`との違いは、`squeue`が今動いているものを見るコマンドなのに対して、`sacct`は終わったジョブも含めて履歴を見るコマンドだという点です。ローカル運用では、むしろ失敗したジョブをあとから追う場面が多いので、`sacct`はかなり重要です。

ここでは`State`と`ExitCode`を見れば十分役に立ちます。正常終了なら`COMPLETED`、異常終了なら`FAILED`、手で止めたなら`CANCELLED`というように、結果を一覧で確認できます。たとえば上の例だと、`train-vit`は二度失敗してから再投入され、その次のrunでは手動で停止されたことがわかります。

### 5.6 `scontrol show job`で1つのジョブを詳しく調べる

詳細を確認したいときは`scontrol show job <id>`が使えます。

ここではジョブの状態、理由、要求リソース、作業ディレクトリ、出力先、タイムスタンプなどが確認できます。ローカル運用では、ジョブの実行後に何が起きたかを追いやすいことがとても重要です。

```
JobId=1 JobName=wrap UserId=<USER_NAME>(<USER_NAME>) Partition=cpu State=COMPLETED Reason=Completed
   NumTasks=1 CPUs/Task=1 ReqMem=512MB ReqGRES=(null) TimeLimit=UNLIMITED BeginTime=Unknown Dependency=(null) Exclusive=No
   SubmitTime=2026-04-13T09:18:24 StartTime=2026-04-13T09:18:24 EndTime=2026-04-13T09:18:24 ExitCode=0:0 ArrayTask=(null) BatchFlag=1
   WorkDir=/home/<USER_NAME>/workspace/slotd Command=echo hello from slotd StdOut=/home/<USER_NAME>/workspace/slotd/slurm-1.out StdErr=/home/<USER_NAME>/workspace/slotd/slurm-1.out NodeList=localhost
   ReqTRES=cpu=1,mem=512M AllocTRES=cpu=1,mem=512M,node=1 MaxRSS=(null)
```

この出力で見るべきポイントもある程度決まっています。`State`と`Reason`で最終状態を確認し、`Command`で実際に何を実行したかを見て、`StdOut`と`StdErr`でログの保存先を確認します。`ReqTRES`と`AllocTRES`を見れば、要求した資源と実際に割り当てられた資源も把握できます。失敗したジョブをデバッグするときは、最終的にこの出力とログファイルを見ることになります。

### 5.7 `srun`と`salloc`で対話的に使う

`sbatch`がジョブを登録してあとで流すためのコマンドだとすると、`srun`と`salloc`はその場で対話的に作業したいときに使うコマンドです。ただ、2つは似ているようで役割が少し違います。

まず`srun`は、1つのコマンドを今すぐ実行したいときに使います。たとえば学習前にGPUが見えているかだけ確認したい、前処理スクリプトを1回だけ試したい、学習済みcheckpointで短い評価を手元ですぐ回したい、といった場面です。感覚としては、普通のシェルでコマンドを打つのに近いのですが、実行自体はスケジューラ管理下に置かれます。

```
srun --label --unbuffered -- echo hello
```

この例では単に`echo`していますが、実際には次のような使い方のほうがイメージしやすいと思います。

```
srun -p gpu -c 4 --mem 8G -G 1 -- python scripts/check_gpu.py
```

これはGPU1枚を含む資源条件で`python scripts/check_gpu.py`をその場で実行する、という意味です。学習ジョブを長時間流す前に、環境が壊れていないかを短く確認したいときに向いています。`--label`は出力にtask labelを付けるオプションで、`--unbuffered`は標準出力を即時に見たいときに役立ちます。

一方で`salloc`は、1つのコマンドではなく、作業用の枠そのものを確保したいときに使います。たとえばGPU1枚を30分だけ押さえたうえで、その中で`python`や`ipython`を起動して試行錯誤したい、学習が落ちる条件を対話的にデバッグしたい、モデルのロードだけ手元で何度か試したい、といった場面です。

```
salloc -p gpu -c 4 --mem 8G -G 1 -t 00:30:00
```

```
Granted job allocation 21
```

このあと、そのallocationの中でシェルを立ち上げて作業します。もっともわかりやすいのは、確保したallocationの中で`srun`を使ってシェルを起動する方法です。

これで、そのallocationに割り当てられた資源の中で対話的なshellに入れます。つまり流れとしては、先に`salloc`でGPUやメモリを確保し、そのあと`srun --pty bash`で中に入って作業する、という形です。

その中では、たとえば次のような作業をします。

```
python
>>> import torch
>>> torch.cuda.is_available()
True
>>> model = load_model(...)
```

つまり`srun`はコマンド単位、`salloc`は作業時間ごとの資源確保だと考えるとわかりやすいです。`sbatch`でジョブが積まれているときに割り込んで実行するなどのユースケースが考えられます。`sbatch`が再現性の高いバッチ実行向けだとすると、`srun`と`salloc`は実験前の確認やデバッグ向けのコマンドです。

### 5.8 `--array`で並列ジョブを流す

さらに、実験をまとめて流すときに便利なのがjob arrayやdependencyです。job arrayを使うと、似た条件の実験をまとめて投入できます。たとえばハイパーパラメータ違いの評価を10本まとめて流したい場合、1本ずつ`sbatch`するのではなく、配列ジョブとしてまとめて投入できます。

たとえば次のように書くと、`0`から`9`までの10個のtaskを持つ配列ジョブを作れます。`%2`は同時に2個までしか走らせない、という意味です。

```
sbatch \
  --job-name eval-array \
  --array 0-9%2 \
  -o logs/%A_%a.out \
  --wrap 'python eval.py --seed $SLURM_ARRAY_TASK_ID'
```

投入に成功すると、親ジョブに対して次のような出力になります。

このとき実際には、`12_0`から`12_9`のようなtaskが作られます。`squeue`や`sacct`では、それぞれのtaskが個別に見える形になります。

```
JOBID  PARTITION  NAME        USER  ST  TIME  NODELIST(REASON)
12_0   gpu        eval-array  ...   R   0:12  localhost
12_1   gpu        eval-array  ...   R   0:08  localhost
12_2   gpu        eval-array  ...   PD  0:00  Resources
```

この仕組みの意図は、似た実験をひとまとまりとして管理することです。seed違い、データ分割違い、ハイパーパラメータ違いの評価を流すときにかなり便利です。

### 5.9 `--dependency`で実験フローを作る

`--dependency`を使えば、学習が終わってから評価を流す、評価が終わってから集計を流す、といった後続処理の連結もできます。たとえば先に学習ジョブを投入して、その完了後にだけ評価ジョブを走らせたいなら、次のように書けます。

```
sbatch \
  -J train-model \
  -p gpu \
  -c 8 \
  --mem 16G \
  -G 1 \
  -t 04:00:00 \
  -o logs/%j.out \
  --wrap 'python train.py --config configs/base.yaml'
```

```
sbatch \
  -J eval-model \
  -p gpu \
  -c 4 \
  --mem 8G \
  -G 1 \
  -t 01:00:00 \
  -o logs/%j.out \
  --dependency afterok:20 \
  --wrap 'python eval.py --ckpt checkpoints/base.pt'
```

この場合、job id `21`の評価ジョブは、job id `20`の学習ジョブが正常終了したあとにだけ実行されます。`squeue`では、学習中のあいだ評価ジョブは待機状態で見えます。

```
JOBID | PARTITION | NAME       | USER | ST | TIME | NODELIST(REASON)
20    | gpu       | train-model| ...  | R  | 8:41 | localhost
21    | gpu       | eval-model | ...  | PD | 0:00 | Dependency
```

単発のコマンド実行を超えて、実験の流れそのものをジョブとして記述できるようになるのが、ジョブスケジューラのいちばん大きな価値だと思っています。学習、評価、集計をこの形でつなげられるようになると、人間が端末の前に張り付いていなくても、計算機の遊びがなく実験が回るようになります。

その他の機能については、slotdの詳細はドキュメントページを参照してください。  
<https://ymgaq.github.io/slotd/ja/>

## 6. ジョブスケジューラ × Agent開発

これまでジョブスケジューラはHPC環境で主に使われてきましたが、Coding Agentによる開発や実験の自動化とも非常に相性がよいと感じています。最近のClaude CodeやCodexは、コードを編集し、テストを実行し、ログを読み、必要なら次の修正を行う、というループを実行できます。

![](https://static.zenn.studio/user-upload/7ea8eedd9012-20260414.png)

ただし、フルオートで学習や評価のような重い処理までAgentの対話セッションに直接ぶら下げると、実行時間、セッション同士の競合、ログ追跡、セッション寿命の問題が出てきます。そこで、Agent自体は何を試すかを決める役に徹し、重い処理はスケジューラにジョブとして投げるという使い分けをするという発想が出てきます。例えば次のような流れです。

1. Agentがコードを修正する
2. Agentが必要な学習ジョブや評価ジョブを`slotd`に投入する
3. `--dependency`を使い、処理が終わるタイミングでAgentを呼び出すジョブを登録する
4. 新しいセッションでAgentがログや成果物を読み、結果を要約する
5. 次の修正や次の実験条件を決める
6. 1に戻る

このやり方の利点は、Agentのセッションが長時間の学習に付き合う必要がなくなり、GPU利用はジョブとして管理され、結果の所在も追いやすくなります。特に、リソース管理、再試行制御、分散実行、ログ管理といった領域では、Codex やClaude Code単体よりも安定性と再現性が向上する可能性があります。

ここで面白いのは、Agent自身を後続ジョブとして呼べることです。  
たとえば、まず学習ジョブを投入します。

```
sbatch \
  -J train-exp-a \
  -p gpu \
  -c 8 \
  --mem 16G \
  -G 1 \
  -t 04:00:00 \
  -o logs/%j.out \
  --wrap 'python train.py --config configs/exp_a.yaml'
```

次に、そのジョブが正常終了したあとにだけCodexを呼び出す後続ジョブを投入します。

```
sbatch \
  -J codex-review-exp-a \
  -p cpu \
  -c 4 \
  --mem 8G \
  -t 00:20:00 \
  -o logs/%j.out \
  --dependency afterok:42 \
  --wrap 'codex exec --full-auto -C ~/workspace/project "Read AGEGNTS.md, logs/42.out and metrics/exp_a.json, summarize the run, and suggest one concrete patch for the next experiment."'
```

この場合、job id `43`はjob id `42`が正常終了したあとにだけ動きます。`squeue`では次のように見えます。

```
JOBID | PARTITION | NAME               | USER | ST | TIME | NODELIST(REASON)
42    | gpu       | train-exp-a        | ...  | R  | 6:12 | localhost
43    | cpu       | codex-review-exp-a | ...  | PD | 0:00 | Dependency
```

この形にすると、実験の実行と、その結果を読んで次の方針を考えるAgentの仕事を、dependency付きのジョブ列として自然につなげられます。学習後に評価Agentを呼ぶ、評価後に再度Agentを呼ぶ…といったサイクルに簡単に拡張できます。

!

ちなみに`slotd`自体もこのスタイルで開発しています。

初期段階では、人間が設計の大枠を示し、スケジューラとしての初期実装と簡易テストまでを対話的に誘導しながら書かせました。

その後は、codex自身がジョブスケジューラを経由して機能追加、テスト拡充、リファクタリング、ドキュメント作成といった作業のかなりの部分を、Agentが自律的に回して完成度を高めていきました。

最終的にRustのコードベースで10,000行、116件のテスト、各種ドキュメントが1日で整備できました。
