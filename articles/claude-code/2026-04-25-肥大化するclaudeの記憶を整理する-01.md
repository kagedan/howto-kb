## はじめに

Claude Codeを業務で使い込んでいると、`~/.claude/`配下のセッションログが想像以上の速さで蓄積していきます。手元のマシンを確認したところ、`~/.claude/`全体で939MB、`~/.claude/projects/`配下のJSONLファイルだけで434件382.2MBに達していました。

公式ドキュメントによれば、Claude Codeは起動時に`cleanupPeriodDays`の設定値（既定値30日）を基準に古いセッショントランスクリプトを削除します。`v2.1.117`以降は`projects/`に加えて`tasks/`、`shell-snapshots/`、`backups/`も同じ掃除の対象になりました（[Data usage](https://code.claude.com/docs/en/data-usage)、[Issue 51779](https://github.com/anthropics/claude-code/issues/51779)）。

ところがこの自動削除は日付による全件一括の仕組みで、特定のプロジェクトだけ残したい、サイズの大きい会話だけ間引きたいといった粒度の操作はできません。`cleanupPeriodDays`を`0`に設定して無効化することも、現行バージョンではバリデーションエラーで弾かれます（[Issue 23710](https://github.com/anthropics/claude-code/issues/23710)）。30日以内であってもヘビーに使えば数百MBは普通に積み上がりますし、`history.jsonl`や`plans/`は別系統のため、セッションを消しても入力履歴とプランは残り続けます。

そこで、ターミナルだけで完結する整理ツールとして[claude-history-manager](https://github.com/okamyuji/claude-history-manager)を作りました。Pythonとuvで動くCLIで、対話メニューから一覧、検索、削除、ストレージ集計ができます。本記事ではこのツールを実際に手元のClaude Code環境で動かして検証した結果と、既に公開されている類似ツールとの違いをまとめます。

## 動作環境と前提

* macOS Tahoe 26.4.1
* Python 3.11以上
* uvでパッケージ管理
* Claude Code（最新版）
* Rich（依存はこの1つだけ）

`uv sync`で依存解決と仮想環境の生成が完了するため、Python本体さえ入っていれば数秒で動かせます。

## なぜ自前のツールを作ったのか

作成前に、既存のClaude Code履歴系ツールを一通り確認しました。代表的なものを挙げます。

機能的にはどれも完成度が高いのですが、自分の使い方で課題が3つありました。

1点目は形態の重さです。Web UIやElectronは閲覧体験が良い反面、整理作業のたびにブラウザやアプリを立ち上げるのが煩わしく、シェル中心の作業フローと馴染みません。

2点目は対象範囲です。多くのツールは`projects/*.jsonl`の閲覧か削除に特化しており、`history.jsonl`、`plans/`、`debug/`、`shell-snapshots/`、`file-history/`を横断的に集計してくれるものは見つかりませんでした。

3点目はclaude-mem連携です。以前の記事で書いたとおり、claude-memのChromaDBが知らぬ間に肥大化することがあります（[claude-memプラグインのChromaDBが420GBに肥大化していた原因と対処方法](https://zenn.dev/okamyuji/articles/claude-mem-chromadb-disk-bloat-fix)）。同じツールから`VACUUM`とFTSインデックス再構築まで踏めると、整理動線が一本にまとまります。

これら3点を1つのターミナルCLIに収めたのがclaude-history-managerです。

## アーキテクチャ

5つのモジュールで構成しています。総コード量は約1100行です。

各モジュールの責務は次のとおりです。

* `config.py`は`CLAUDE_DIR`、`PROJECTS_DIR`、`HISTORY_FILE`、`PLANS_DIR`、`CLAUDE_MEM_DIR`、`JST`の定数だけを保持します
* `parser.py`はJSONL1行ずつをパースしてタイムスタンプ、ブランチ名、最初のユーザーメッセージ、アシスタント応答冒頭などのメタ情報を抽出します
* `storage.py`はファイル列挙、検索、削除、ストレージ集計、claude-mem最適化を担当します
* `ui.py`はRichの表とパネルで結果を整形します
* `cli.py`は対話メニューとサブコマンドを束ねます

`parser.py`には早期終了の最適化を入れています。最初のユーザーメッセージとアシスタント応答が取れてメッセージが6件以上見つかった時点で、残りの行は中身を解析せず行数だけ数えて読み込みを打ち切る作りです。手元の434件382MBを処理しても`uv run claude-history sessions`で約0.48秒、ファイル数が増えてもストレスなく扱えます。

## 実機での動作確認

### ストレージ全体の把握

最初に実行したのは`storage`サブコマンドです。

```
uv run claude-history storage
```

セッションログ以外にもファイル履歴が1337件8.8MB積み上がっていることが一目でわかります。`du -sh ~/.claude/`では見えない内訳まで分解できるのが利点です。

### セッション一覧と検索

`uv run claude-history`を引数なしで実行すると対話メニューが立ち上がります。`s`を入力するとセッション一覧に入り、20件ずつページ送りで表示されます。

```
セッション一覧 (80件)  ページ 1/4

    1  2026-04-25 20:02  devs/zenn (main)  160msg 418KB
       このディレクトリにあるarticlesやbooksで github.comの…
    2  2026-04-25 19:51  devs/java/cognito/auth/se... (main)  118msg 1416KB
       ...
```

`番号`で詳細表示、`d 番号`で削除、`f`で検索、`x`で一括削除に進めます。検索はユーザーメッセージ、アシスタント応答、プロジェクト名、ブランチ名、作業ディレクトリを横断的にあたります。

```
uv run claude-history search ratatui
```

実際に手元の434件から「ratatui」を検索すると、3件のヒットが0.48秒で返りました。grepと違い、JSONLの構造を理解した上で最初のユーザー発話とアシスタント応答だけを対象にしているため、ノイズの少ない結果が得られます。

### 一括削除

メインメニューから`x`を選ぶと一括削除のサブメニューに入ります。3つの軸が用意されています。

* 日付でカットする方式は`YYYY-MM-DD`を入力すると、その日より前のセッションを抽出します
* プロジェクト指定削除は対象一覧から番号で選ぶ方式で、特定リポジトリの作業履歴だけを丸ごと整理できます
* 小サイズセッション削除はサイズ閾値（既定5KB）以下の中断セッションだけを掃除します

実行直前に対象一覧と合計サイズが表示され、`本当に削除しますか？`で確認を求められます。`Confirm.ask`の戻り値が`False`なら削除関数自体が呼ばれない実装にしているため、誤操作で全消ししてしまう事故は起きにくくなっています。

入力履歴`history.jsonl`の削除では、削除前に必ず`history.jsonl.bak`へバックアップを取ってから書き換えます。`shutil.copy2`で属性ごとコピーしているため、万一の場合は`mv`で戻せます。

### claude-mem最適化

`~/.claude-mem/`が存在する環境では、メインメニューに7番目の項目「claude-mem最適化」が現れます。これはclaude-memのSQLiteデータベースとChromaDBに対して以下の操作をまとめて走らせるサブメニューです。

* メインDBの`VACUUM`と`ANALYZE`でフラグメント化を解消する
* ChromaDBのSQLiteを`VACUUM`してインデックスファイルを縮める
* FTS5インデックス（`observations_fts`、`session_summaries_fts`、`user_prompts_fts`）を再構築する
* `~/.claude-mem/logs/`から7日以上前のログを削除する

操作前後のサイズが`32.4MB → 28.1MB（-4.3MB）`の形式で表示され、効果を確認しながら進められます。claude-memの大規模肥大に直面したときは、まず`a`（すべて実行）で総点検し、それでも大きいときに個別の操作へ進むのがおすすめの順序です。

### コマンドライン引数による直接実行

対話モードを経由せずに結果だけ取りたい場合はサブコマンドを使えます。

```
uv run claude-history sessions
uv run claude-history history
uv run claude-history storage
uv run claude-history search MangaViewer
```

シェルのワンライナーやエイリアスから呼び出すことを想定したインターフェースで、CIや週次の整理スクリプトに組み込みやすくしています。

## 既存ツールとの差別化点

実装と検証を踏まえ、改めて既存ツールとの違いを整理します。

| 観点 | claude-history-manager | claude-code-viewer | claude-code-history-viewer | claude-code-cleaner | ccusage |
| --- | --- | --- | --- | --- | --- |
| 形態 | ターミナルCLI | Web UI | Electronデスクトップ | Rust TUI | Node CLI |
| 言語/依存 | Python + rich | Node + Web | Electron | Rust | Node |
| 対象範囲 | projects + history.jsonl + plans + debug + shell-snapshots + file-history | projects中心 | projects中心 | 整理対象を広く列挙 | 使用量集計 |
| 一括削除条件 | 日付 / プロジェクト / サイズ | あり（UI） | あり（UI） | あり | 対象外 |
| 横断検索 | first\_user\_msg + assistant + project + branch + cwd | あり | あり | 対象外 | 対象外 |
| claude-mem最適化 | 対応 | 非対応 | 非対応 | 非対応 | 非対応 |
| 起動時間 | 約0.48秒（434セッション） | ブラウザ立ち上げ | アプリ起動 | 比較的軽量 | 即時 |

ブラウザやElectronで会話の中身を読み込みたい用途であれば`claude-code-viewer`系列が便利ですし、トークン課金を追いたい用途であれば`ccusage`が手堅い選択です。一方で、整理と最適化を主目的にして、シェル作業の延長線で完結させたい場合にclaude-history-managerが効きます。

公式の自動削除は前述のとおり日付ベースの全件処理なので、「30日以内だが特定プロジェクトのセッションが2GBあるので片付けたい」といった粒度には届きません。Anthropicも[Issue 47038](https://github.com/anthropics/claude-code/issues/47038)で自動保持ポリシーの拡充をfeature requestとして受け付けている段階で、現時点ではユーザー側の補助ツールが必要な領域です。

## 最新のClaude Codeでも引き続き有効な理由

公式ドキュメントの[Explore the .claude directory](https://code.claude.com/docs/en/claude-directory)と[Data usage](https://code.claude.com/docs/en/data-usage)を読み直すと、`~/.claude/projects/`にJSONLでセッショントランスクリプトを保存する仕組み自体は当面変わらないことがわかります。`cleanupPeriodDays`は値を増やせば長期保持できる一方で、ストレージ消費は線形に増えます。

claude-history-managerが解決するのは、その「長期保持か全削除か」の二択を「粒度の細かい間引き」に変える部分です。具体的には次の場面で力を発揮します。

* `cleanupPeriodDays`を長め（90日や365日など）に設定しているが、特定プロジェクトの会話だけ残したい場合に効きます
* 検証用の極小セッション（5KB未満）が散らばっているのを、本筋のログを残したまま掃除したい場合に効きます
* claude-memと併用していて、ChromaDBやFTSインデックスが膨らんだときに同じツールから整理したい場合に効きます
* 過去会話を検索してから削除したいといった「読む」と「消す」を行き来したい場合に効きます

公式の自動削除に頼り切らず、定期的にこのツールで内訳を確認するワークフローに変えてから、`~/.claude/`が暴走するケースは観測していません。

## テストと品質ゲート

最後に、ツール自体の品質維持について触れます。

```
220 passed in 1.00s
TOTAL  658  10  98%
```

220件のテストが1秒で完走し、行カバレッジは98%です。テストは4つのファイルに分かれており、JSONL解析（45件）、ストレージ操作（37件）、UI出力（39件）、CLI対話メニュー（68件）を担当します。`tests/conftest.py`で一時ディレクトリ上に`.claude/`構造を組み立てるため、本物の履歴に触れずにライフサイクル全体を検証できます。

リンターは`ruff`、型チェックは`mypy`の`strict`モードを通しています。整理ツールは「うっかり削除」が致命傷になるため、削除前後の確認ダイアログ、バックアップ、空の対象集合の早期リターンといった分岐は念入りにテストを書いています。

## おわりに

claude-history-managerはターミナルだけで`~/.claude/`配下を整理することを目的にしたCLIです。実機の434セッション382MBに対して0.48秒で一覧が返り、対話メニューから日付やプロジェクト単位で粒度の細かい削除ができ、claude-memの`VACUUM`まで一本の動線でこなせます。

公式の自動削除は便利ですが、日付による全件一括の仕組みなので、長期保持と粒度の細かい整理の両立には向きません。claude-history-managerは公式機能を補完する位置付けで、ヘビーユーザーほど効果が大きい設計です。

ソースコードと使い方の詳細は[GitHubのリポジトリ](https://github.com/okamyuji/claude-history-manager)にあります。ターミナル中心の作業フローでClaude Codeを使い込んでいる方の役に立てば嬉しいです。
