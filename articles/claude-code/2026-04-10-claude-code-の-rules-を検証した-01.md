## はじめに

Claude Code には、特定のファイルパスにマッチした場合にのみルールを適用する **`paths` frontmatter** という機能がある。  
このレポートでは、`paths` 条件の実際の動作を検証し、そこで見つかった制約と、それを回避するために構築した **`branch-rules-auto-loader`** の仕組みについてまとめる。

---

## 1. Claude Code のルールファイルとは

Claude Code では `~/.claude/rules/` 配下に `.md` ファイルを置くと、全セッションで自動的に読み込まれるグローバルルールとして機能する。

さらに、ルールファイルの先頭に YAML frontmatter を記述することで、**特定のファイルパスにマッチした場合のみ**ルールを適用することができる。

```
---
paths:
  - "**/*.swift"
---
# iOS 開発ルール
Swift ファイルを編集する際は...
```

これにより、たとえば Swift ファイルを操作するときだけ iOS 固有のルールを読み込む、といった使い分けが可能になる。

---

## 2. 使ってみて分かった問題点：Write 操作ではルールがトリガーされない

### 現象

`paths` frontmatter を持つルールは、**`Read` ツールを使ったときにのみ** パスマッチが評価される。

`Write` ツールだけで新規ファイルを作成した場合、ファイルパスが `paths` 条件に一致していても、**ルールが適用されない**。

### 検証内容

以下のようなルールを用意して検証した。

```
---
paths:
  - "**/*claude-rule-poc*/**"
---
# このディレクトリ専用ルール
このルールが読み込まれた場合、返答の冒頭に必ず「[RULE LOADED]」と発言すること。
```

**ケース①：既存ファイルを Read → ルールが適用される**

```
> test.txt の中身を確認して

⏺ Read(test.txt)
  ⎿  aaa

[RULE LOADED] ファイルを確認しました。内容は「aaa」です。
```

**ケース②：Write のみで新規ファイル作成 → ルールが適用されない**

```
> test2.txt を作成して「bbb」と書いて

⏺ Write(test2.txt)
  ⎿  Wrote 4 bytes to test2.txt

test2.txt を作成しました。内容は「bbb」です。
```

`[RULE LOADED]` が出力されていない。`Write` 単体ではルールがトリガーされていないことがわかる。

| 操作 | ルール適用 |
| --- | --- |
| 既存ファイルを `Read` | ✅ 適用される |
| `Read` → `Write` の順で実行 | ✅ 適用される |
| `Write` のみで新規ファイル作成 | ❌ 適用されない |

### 問題の本質

Claude Code の内部実装として、`paths` のマッチング評価は **`Read` ツールの実行タイミング** に紐づいている。  
`Write` 操作単体ではこの評価が走らないため、新規ファイルの作成時にルールが適用されない盲点が生まれる。

---

## 3. ワークアラウンド：touch → Read → Write

### 手順

新規ファイルを作成する場合、以下の 3 ステップを踏むことで `paths` ルールを正しく適用させることができる。

```
> test3.txt を新規作成して「ccc」と書いて

⏺ Bash(touch test3.txt)
  ⎿  (no output)

⏺ Read(test3.txt)
  ⎿  (empty file)

⏺ Write(test3.txt)
  ⎿  Wrote 4 bytes to test3.txt

[RULE LOADED] test3.txt を作成しました。内容は「ccc」です。
```

`touch → Read → Write` の順を踏んだことで `[RULE LOADED]` が出力され、ルールが正しく適用されている。

### なぜこれで動くのか

`touch` でファイルを物理的に作成しておくことで、`Read` ツールがそのパスに対してマッチング評価を実行できる状態になる。  
空ファイルであっても `Read` を通すことで paths ルールが読み込まれ、その後の `Write` 操作にもルールが効いた状態になる。

`~/.claude/CLAUDE.md` に **最重要事項** として記載すると大体この手順が守られていた。

## 4. より根本的な課題：プロジェクト固有ルールの管理

### 課題

`paths` frontmatter でファイル拡張ではなく特定のプロジェクト固有（またはディレクトリ配下固有）のルールを管理する場合、以下の問題がある。

1. **ルール適用が Read トリガー依存** → 新規ファイル作成時に適用されないケースがある
2. **グローバルルールとの混在** → `~/.claude/rules/` は全セッション共通なので、プロジェクトが増えると管理が複雑になる
3. **予測困難性** → 「このファイルを Read したとき、どのルールが読み込まれるか」が直感的に分かりにくい

### リポジトリの .claude に書けばいいのでは？

Claude Code はリポジトリ直下に `.claude/` ディレクトリを置くことで、そのプロジェクト専用の設定を持たせることができる。「プロジェクト固有のルールはここに書けば確実では？」と思うかもしれないが、以下の理由で採用しなかった。

**同じルールを複数のプロジェクトに適用したい場合の問題**

たとえば「workspace/div1/ 配下の全プロジェクトにタスク管理ルールを適用したい」というケースを考える。この場合、各リポジトリの `.claude/` にルールをコピーして置く必要があり、以下のリスクが生じる。

* **更新時の同期漏れ**: ルールを改善した際、全リポジトリのコピーを一斉に更新しなければならない
* **新規プロジェクト作成時の取り込み漏れ**: 新しいリポジトリを作るたびに手動でルールをコピーする手順が必要になり、設定し忘れが起きやすい
* **管理コストの増大**: リポジトリが増えるほど、ルールのバージョン管理が複雑になる

branch-rules-auto-loader であれば、`~/.claude/branch-rules/` の1ファイルを更新するだけで全対象プロジェクトに即時反映され、新規プロジェクトも自動的に拾われる。

### 求めていたもの

* 特定のプロジェクトディレクトリ（またはディレクトリ名のパターン）でのみ有効になるルール
* Read/Write の操作種別に依存しない、セッション開始時点でのルール確定
* 管理が明快なディレクトリ構造

---

## 5. 解決策：branch-rules-auto-loader の構築

### 仕組みの概要

`paths` frontmatter に頼らず、**Claude 自身にルール読み込みの判断を委ねる**アプローチを採用した。

```
~/.claude/
├── rules/
│   └── branch-rules-auto-loader.md   # セッション開始時に必ず読み込まれるメタルール
└── branch-rules/
    ├── aaa.md                         # プロジェクト A 用ルール
    └── bbb.md                         # プロジェクト B 用ルール
```

### branch-rules-auto-loader.md の内容

```
# Branch Rules Auto Loader

セッション開始時（最初のユーザーメッセージへの返答前）に以下を実行すること：

1. `pwd` コマンドで現在の作業ディレクトリを取得する
2. 下記テーブルの「正規表現パターン」に一致するか確認する
3. 一致した場合、対応する「ルールファイル」を `Read` ツールで読み込む
4. 読み込んだルールをセッション中有効なものとして扱う
5. 一致しない場合は何もしない（ユーザーへの言及も不要）

| 正規表現パターン | ルールファイル |
|---|---|
| `.*claude-rule-poc.*` | `~/.claude/branch-rules/aaa.md` |
```

### 動作フロー

```
セッション開始
    ↓
branch-rules-auto-loader.md が自動読み込み（全セッション共通）
    ↓
Claude が pwd を実行して作業ディレクトリを確認
    ↓
正規表現パターンと照合
    ↓
一致 → 対応する branch-rules/*.md を Read で読み込む
不一致 → 何もしない
    ↓
セッション中、そのルールが有効な状態で作業継続
```

### 今回の用途における paths frontmatter との比較

| 観点 | paths frontmatter | branch-rules-auto-loader |
| --- | --- | --- |
| トリガー条件 | ファイルの Read 操作 | セッション開始時（pwd 確認） |
| 新規ファイル作成時 | ❌ Write 単体では未適用 | ✅ 常に適用済み |
| 予測可能性 | 低い（操作依存） | 高い（セッション開始時に確定） |
| 管理のしやすさ | ルールファイル内に分散 | テーブル1か所で一元管理 |
| 適用スコープ | ファイルパスベース | ディレクトリパスベース（正規表現） |

## 6. 実際の検証：claude-rule-poc プロジェクト

このレポートの検証は `test-claude-rule-poc-aaa` ディレクトリで行った。  
ディレクトリ名が `.*claude-rule-poc.*` にマッチするため、セッション開始時に `branch-rules/aaa.md` が自動読み込みされる。

`aaa.md` には以下のルールが定義されている。

```
# claude-rule-poc プロジェクト用ルール

このルールが読み込まれた場合、ユーザーへの最初の返答の冒頭に必ず以下を発言すること：

「[TEST] branch-rules-auto-loader 経由で claude-rule-poc ルールが読み込まれました！」
```

セッションを開始すると、Claude は最初のユーザーメッセージへの返答前に自律的にルールファイルを読み込み、定義された指示に従って動作する。

![sample](https://static.zenn.studio/user-upload/1f3956ce0689-20260410.png)

`Read(~/.claude/branch-rules/claude-rule-poc.md)` が自動実行され、その直後に `[TEST] branch-rules-auto-loader 経由で claude-rule-poc ルールが読み込まれました！` が出力されている。ユーザーが何もしなくても、セッション開始時点でルールが確定していることがわかる。

## 7. まとめ

### 得られた知見

1. **`paths` frontmatter は Read 操作依存** → 新規ファイル作成時は `touch → Read → Write` の 3 ステップが必要
2. **プロジェクト単位の条件分岐には `branch-rules-auto-loader` が有効** → 操作の種類に依存せず、セッション開始時点でルール適用を確定できる
3. **Claude 自身に判断を委ねる設計** → ルールファイルに「Claude が実行すべき手順」を記述することで、機能として存在しない仕組みを命令で補える

### 現在の構成

```
~/.claude/CLAUDE.md               # グローバル設定・最重要ルールの記載場所
~/.claude/rules/                  # 全セッション共通ルール群
    branch-rules-auto-loader.md   # ← セッション開始時にディレクトリ判定してルール読み込み
~/.claude/branch-rules/           # プロジェクト固有ルール群
    aaa.md                        # test-claude-rule-poc-* ディレクトリ用
```

この構成により、プロジェクトごとに異なるルールをクリーンに管理しつつ、Read/Write 操作の種別に依存しない安定したルール適用が実現できている。
