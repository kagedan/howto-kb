## はじめに

6月17日、サンフランシスコで開かれた Databricks の [Data + AI Summit 2026](https://www.databricks.com/dataaisummit) に参加してきました。その朝の基調講演で、共同創業者でCTOの Matei Zaharia さんが、週末に公開したばかりというOSS「[Omnigent](https://github.com/omnigent-ai/omnigent)」を紹介していました。個々のエージェントの上に乗る「**メタハーネス（meta-harness）**」と呼ばれるものです。この記事では、メタハーネスとは何かを深掘りしつつ、Omnigent を中心に、関連するフレームワークとその系譜をまとめます。

## そもそも「ハーネス」とは何か

言語モデルは、突きつめればトークンを吐き出すだけの存在です。そのモデルの周りを囲んで、ファイルや画面、ツールの呼び出し、権限のチェック、サンドボックスといった「外の世界との接点」を与えているソフトウェア、これがハーネスである、と Matei さんはシンプルに説明していました。

たとえば Claude Code なら、`Claude` というモデルと、それをファイルや端末やMCPにつなぐ `Claude Code` というハーネスに分けて考えられます。Codex でも Cursor でも構図は同じで、生成AIを何らかのプロダクト上で使用する場合、モデルの上に必ず何らかのハーネスが乗っています。

階層で並べると、`モデル → ハーネス → メタハーネス` という入れ子になります。いちばん下に素のモデルがあり、その上に個々のハーネスがあり、さらにその上に複数のハーネスを束ねるメタハーネスが乗る、というイメージです。

## ハーネスが乱立するまでの流れ

生成AIモデルが出た当初、最初はチャットでモデルに質問を投げかけるだけでした。そこにツール呼び出し（tool calling）が加わり、モデルが外部の関数を叩けるようになります。やがてコーディングエージェントが実用になり、開発者の多くが日常的に使うようになりました。ハーネスは比較的作りやすく、しかもモデルと一緒に設計するとぐっと性能が出るので、各社・各個人が自分に最適化されたハーネスを作り始めました。結果として、似たようなハーネスが大量に並ぶ状態になりました。

また、API の共通化により、多くのプロバイダーが OpenAI 互換の API に寄り、ツール接続は MCP に収斂してきたことで、モデルやツールの差し替えと評価がやりやすくなりました。裏を返すと、ハーネスを乗り換える障壁が下がったぶん、選択肢が増えて余計に乱立した面もあります。

このような背景から、増えたハーネスをより上位でオーケストレーションするという目的で、メタハーネスの需要が生まれてきました。

## 実際に使用されているハーネス

Matei さんの定義に基づくと、Claude Code や Codex、Cursor、Aider あたりの Coding Agent もハーネスの一種になります。

自作エージェントを組むためのフレームワークやSDKについても、設計の考え方に違いはあれど「自分でハーネスを書くための部品」と考えれば、広い意味でのハーネスに含まれると思います。以下はその代表例です（OSS可否は2026年6月時点）。

ほかにも、自前の Agent や Claude Code の Skill もハーネスと捉えられると思います。

## ハーネスの上の層で解決できること

Matei さんは、カンファレンスでハーネスをまたいで困ることを3つに整理していました。この3つが、そのままメタハーネスの存在理由になります。

1つ目は**合成**（composition）です。実際の開発では、複数のコーディングエージェントを同時に走らせる場面が増えています。あるエージェントが詰まったら別のエージェントに聞く、安いモデルと高いモデルを役割分担させる、といった組み合わせ方が広がってきました。ところが、同じハーネスの中で組み合わせるのは簡単でも、Claude Code と Codex のように別々のハーネスをまたいで組み合わせるのは、とたんに難しくなります。

2つ目は**協働**（collaboration）です。自分が片方のエージェントを使い、同僚が別のエージェントを使っているとき、二人で同じ作業をどう進めればいいのか。Matei さんは「画面いっぱいにエージェントのウィンドウが開いていて、ひたすらテキストをコピペしている」という例を挙げていました。途中で誰かに作業を引き継ぐにも、コンテキストごと渡す手段がありません。

3つ目は**統制**（control）です。エージェントはプロンプトインジェクションで乗っ取られることがありますし、乗っ取られなくてもDBを消すような事故を起こします。かといって「この100行のスクリプトを実行していい？」と毎回ユーザーに聞くのも限界があります。コストが膨らむ問題も同じ層の話です。

この3つは、どのハーネスにも共通してのしかかってくる課題です。だからこそ、個々のハーネスの上に、横断的に解く層を置こうという発想からメタハーネスが求められるようになりました。

## メタハーネスとは何か

### 「上の層」が生まれるまでの流れ

メタハーネスの発想はいくつかの系譜が合流して、今の形に近づいてきました。

ひとつは、複数のLLMやツールへのアクセスを一箇所に集めて捌く、ゲートウェイやコントロールプレーンの系譜です。OpenAI 互換のプロキシとして100以上のモデルをまとめて扱う [LiteLLM](https://github.com/BerriAI/litellm)、ルーティングやガバナンスを担う [Portkey](https://github.com/Portkey-AI/gateway)、[Cloudflare AI Gateway](https://www.cloudflare.com/products/ai-gateway/) あたりが分かりやすい例です。同じ Data + AI Summit で発表された Databricks の Unity AI Gateway も、社内のモデルやエージェント、MCP、コーディングツールを一箇所に集めて、コスト上限やポリシーを面で握ろうとしていました。どのモデルを使うか、どのツールを許すか、コストをどう抑えるかを、入口でまとめて管理しようという発想は共通しています。

もうひとつは、ワークフローのオーケストレーションの系譜です。手順を定義として持ち、失敗しても再実行できるよう状態を保つ、というアプローチで、Netflix 発の [Conductor](https://github.com/conductor-oss/conductor) や、永続実行（durable execution）で知られる [Temporal](https://temporal.io/) がここに当たります。エージェント前提のものでは、グラフでフローを組む [LangGraph](https://github.com/langchain-ai/langgraph) や、AutoGen と Semantic Kernel を統合して2026年4月に 1.0 が GA した [Microsoft Agent Framework](https://github.com/microsoft/agent-framework) が代表格で、後者は逐次・並列・ハンドオフ・グループチャットといった協調パターンをひととおり備えています。

さらに、複数のコーディングエージェントを並べて走らせるUIの系譜もあります。git のワークツリーで作業を分離し、Kanban ボードや差分レビューで並列実行を見える化する道具として [Vibe Kanban](https://github.com/BloopAI/vibe-kanban) や [Claude Squad](https://github.com/smtg-ai/claude-squad) のほか、コンテナで隔離する Imbue の [Sculptor](https://github.com/imbue-ai/sculptor)、Mac アプリの [Conductor](https://www.conductor.build/) などが出てきています（このアプリ版 Conductor は、先ほどの Netflix 由来のワークフローエンジンとは名前が同じだけの別物です）。

どれも便利ですが、見ている場所は少しずつ違います。ゲートウェイは入口の統制、オーケストレーションは手順の堅牢化、並列UIは実行の見える化で、いずれもハーネスの中身や並べ方の話です。ハーネス同士の互換性そのものには踏み込んでいません。Omnigent は、異なるハーネスをまたいで束ねることを軸としています。

### Omnigent の中身

Omnigent は、Databricks が公開した OSS のメタハーネスです。Apache 2.0 ライセンスで、まだ alpha 段階ですが、Databricks への依存はなく、手元でもサーバーでも動かせるようになっています（参考: [Databricks のブログ](https://www.databricks.com/blog/introducing-omnigent-meta-harness-combine-control-and-share-your-agents)、[GitHub リポジトリ](https://github.com/omnigent-ai/omnigent)）。

会場で示されていた全体像のスライドが分かりやすかったので、構成をそのまま追ってみます。  
![](https://static.zenn.studio/user-upload/2fea8b40c570-20260618.jpg)

中心にあるのは **Runner** と **Server** の2つで、両者を **Common API** という共通インターフェースがつないでいます。Runner は個々のエージェントをラップして、サンドボックスや信頼性まわりを受け持ちます。Server は履歴・ポリシー・成果物・カタログ・MCP・Skills を集約して、共有や中央のポリシーを担います。そしてこの共通インターフェースの上に、Terminal UI・Web UI・ネイティブアプリ・モバイル・REST API という複数のアクセス手段が並びます。いちばん下には、Coding Agents（Codex や Claude、Cursor など）と Custom Agents（自作エージェント）がぶら下がる、という構図でした。

共通インターフェースの考え方はとても素直で、「メッセージとファイルを入れると、イベントとツール呼び出しが流れ出てくる」という形に各ハーネスを共通化しています。

使い方も見ておきます。まずインストールして、既存のエージェントを `omnigent`（`omni` でも可）で包んで起動します。

```
# インストール（uv tool もしくは Homebrew）
uv tool install omnigent
# brew install omnigent-ai/tap/omnigent

# 既存のハーネスをそのまま包んで起動する
omnigent claude               # Claude Code を起動
omnigent codex                # Codex を起動

# 自作エージェントの定義を実行する
omnigent run path/to/my_agent.yaml

# 共有や中央ポリシーを使いたいときはサーバーを立てる
omnigent server start
```

`omnigent claude` のように包むだけで、いつもの端末UIに加えて、同じセッションが Web UI（既定で `http://localhost:6767`）からも触れるようになります。

おもしろいのは、エージェントの定義を YAML で書ける点です。ここが合成の入口になっていて、ハーネス（`executor.harness`）を指定しつつ、ツールとして関数だけでなく別のエージェントもぶら下げられます。

```
name: my_agent
prompt: You are a helpful data analyst.

executor:
  harness: claude-sdk

tools:
  word_count:
    type: function
    callable: mypackage.mymodule.word_count

  researcher:
    type: agent
    prompt: Search for relevant information and summarize it.
    tools:
      word_count: inherit
```

`harness` に指定できる実行系（executor）は、`claude-sdk`、`codex`、`codex-native`、`claude-native`、`cursor`、`openai-agents`、`pi`、`antigravity` などです。同じ定義のまま、裏で動かすハーネスだけ差し替えられるところに、メタハーネスらしさが出ています。

統制まわりも、ポリシーとして宣言的に書けます。コスト上限や、OS操作のときに確認を挟むといったルールを、サーバー・エージェント・セッションのいずれの粒度でも設定できます。

```
policies:
  approve_shell:
    type: function
    handler: omnigent.policies.builtins.safety.ask_on_os_tools

  budget:
    type: function
    handler: omnigent.policies.builtins.cost.cost_budget
    factory_params:
      max_cost_usd: 5.00
      ask_thresholds_usd: [3.00]
```

「5ドルを超えたら止める、3ドルで一度確認する」といった上限を、コードに手を入れずに後付けできます。コストに対するポリシーを持っている点は、他のエージェントフレームワークにあまりない特徴だと感じました。

協働のほうも仕組みが用意されています。セッションを URL で共有して同僚を招くと、相手は同じセッションを見ながらコメントしたり、エージェントに直接指示を出したりできます。会場のデモでも、READMEのレビューを別のエンジニアに振って、そのまま続きを任せていました。

ここまでをさきほどの3つの課題に当てはめると、合成は YAML でのハーネス指定、協働はセッション共有、統制はポリシーという形で、ひととおり埋まっていることになります。

### メタハーネスに関連するフレームワーク

メタハーネス的な層を担うフレームワークは、ほかにもあります。オーケストレーションの代表格が [CrewAI](https://github.com/crewAIInc/crewAI) です。エージェントに役割（role）と目的（goal）を与えてチームを組み、タスクを割り当てて回す、という考え方になります。最小構成はこのくらいで動きます（参考: [docs.crewai.com](https://docs.crewai.com/)）。

```
from crewai import Agent, Task, Crew

researcher = Agent(
    role="リサーチャー",
    goal="{topic} の最新動向を調べる",
    backstory="その分野に明るいアナリストとして振る舞う。",
)

task = Task(
    description="{topic} について調べ、要点をまとめる",
    expected_output="箇条書きで5点",
    agent=researcher,
)

crew = Crew(agents=[researcher], tasks=[task])
print(crew.kickoff(inputs={"topic": "AIエージェント"}))
```

変遷で触れた Conductor や Microsoft Agent Framework も、このオーケストレーション層の仲間です。どれも「複数のエージェントを協調させる」点ではメタハーネスと重なります。ただ、見ているのは手順の組み立て方で、Claude Code と Codex のように異なるハーネスをまたいで束ねるところには踏み込みません。同じ「上の層」でも、Omnigent とは力点が分かれています。

### ハーネスでもありメタハーネスでもあるもの

自分自身がコーディングエージェント=ハーネスでありながら、同時に複数のエージェントを束ねる=メタハーネスタイプのフレームワークやプロダクトもあります。

ひとつ目は [Claude Squad](https://github.com/smtg-ai/claude-squad) です。端末アプリで、Claude Code や Codex、Gemini、Aider といったエージェントを、それぞれ独立したワークスペースで同時に走らせます。中身では tmux で別々の端末セッションを作り、git のワークツリーで各セッションを別ブランチに分けるので、お互いの作業がぶつかりません。起動はこれだけです。

```
brew install claude-squad
# 起動コマンドは claude-squad（短縮エイリアス cs を張る使い方が一般的です）
claude-squad       # 既定は claude。`claude-squad -p "codex"` のように指定もできる
```

ふたつ目は [Vibe Kanban](https://github.com/BloopAI/vibe-kanban) です。Kanban ボードでタスクを並べ、各タスクを別々のワークツリーでエージェントに実行させて、差分をレビューする、という道具になります。対応するエージェントも幅広く、Claude Code や Codex、Gemini CLI、Amp、Cursor などを束ねられます。  
ただし Vibe Kanban は、公式サイトで開発の縮小（sunsetting）がアナウンスされています。

このほか、[OpenHands](https://github.com/All-Hands-AI/OpenHands) は Agent Canvas で複数エージェントを扱えますし、Goose や Claude Code 自身もサブエージェントを持っていて、自分の中で小さなハーネスを束ねる構造になっています。  
従来のハーネス自身が、合成、協働、統制といったメタハーネスの要素を持つように進化しています。

## おわりに

ハーネスが一気に増えた結果、その上で合成・協働・統制をまとめて引き受ける「メタハーネス」という層が立ち上がってきました。Omnigent はそこをハーネスの相互運用という角度から狙う代表例で、周辺にはワークフロー寄りのオーケストレーション系や、ハーネスとメタハーネスを兼ねるツールが並び、力点はそれぞれに分かれています。

自分自身も、Claude Code の Skill として Codex と Claude を協働させる仕組みを組んで運用していたことがあります。別々のエージェントをまたいで束ねたい場面は実際にあって、メタハーネスはその要求に正面から応える層だと感じました。まだ早い段階の道具が多いものの、方向性としてはかなり筋が良いと思っています。
