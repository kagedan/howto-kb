こんにちは、株式会社エムニの宮木です。Fable 5 が復活して従量課金の提供形態に移行する前に、Fable 5 の利用量をなるべく抑えつつ、Fable 5 と同等の品質でコーディングできるハーネスを設計できないか、実際に作って検証したので今回はその紹介です。

## TL;DR

* Fable 5 (Opus の 2 倍単価) をフェーズの起動と進行判定だけを行う薄い指揮層に限定し、トークンの重い作業 (計画・実装・検証) を Opus / Sonnet / Codex に委譲する Claude Code plugin を作りました → [pev-harness](https://github.com/myksyut/pev-harness) (OSS)
* 「マイクラ作って」というタスクで実測したところ、Fable 単独と同等品質のまま、Fable への課金は **−77.6%**、総額は **−40%** になりました
* ただしこの構造は壊れやすく、orchestrator に関する 4 つの規約のうち 1 つ破るだけで素の Claude Code より高くつきます (本文中に試算を載せています)

## 作ったもの: pev-harness

方針は「Fable には判断だけをさせて、トークンを消費する仕事は安いモデルに落とす」です。コストはトークン量 × 単価の積なので、単価が 2 倍でも量を全体の 15% 以下に抑えられれば影響は小さく、それよりも重い作業を安いモデルへ逃がす削減のほうが効きます。この役割分担を毎回のプロンプトで頑張るのではなく、plugin としてパイプラインに固定したものが pev-harness です。

<https://github.com/myksyut/pev-harness>

`/pev "タスク"` と投げると、(Triage →) Plan → Execute → Verify のパイプラインが走ります。

```
  あなた: /pev "タスク"
       ↓
  指揮層 (Fable 5) … 各フェーズの起動と進行判定だけ。実装ファイルは読まない
       ↓
  [0] TRIAGE  (Sonnet)  計画が要るタスクか 1 ターンで判定
       ↓ 要る場合のみ
  [1] PLAN    (Opus)    仕様の穴を質問 or 根拠付きデフォルトで確定 → plan.md
       ↓ ここで人間レビュー (Gate A)
  [2] EXECUTE (Sonnet or Codex CLI)  plan 通りに実装
       ↓
  [3] VERIFY  (Sonnet)  実装者とは別タスクで起動、テストを回して PASS/FAIL 判定
       ↓
  PASS → 完了    FAIL → 自動で再計画 → 再実装 → 再検証 (最大 3 回)
```

モデルの割り当てと単価は次の通りです。

| Layer | Model | 単価 in/out ($/MTok) | 責務 |
| --- | --- | --- | --- |
| Orchestrator | Fable 5 | 10 / 50 | phase の起動、Gate 判定、retry 打ち切り判断のみ |
| Triage | Sonnet (low effort) | 3 / 15 | Plan 必要性の 1 ターン判定 |
| Planner | Opus (xhigh) | 5 / 25 | 仕様の質問返し + plan.md 作成 |
| Executor | Sonnet / Codex CLI | 3 / 15 (Codex は $0) | 実装 (Codex はサブスク内で API 従量課金なし) |
| Verifier | Sonnet (xhigh) | 3 / 15 | 独立検証、verify.json に PASS/FAIL |

Fable に残しているのは、retry を続けるか打ち切るか、検証が FAIL したときにどう再計画させるか、人間にエスカレーションすべきか、といった判断の難しい仕事だけです。成果物の品質を決めるのは各 phase 側のモデルで、そちらは従来構成のままなので、品質は原理的には落ちないはず、というのが設計時の想定でした。

## 設計思想: orchestrator を薄く保つ 4 つの規約

この構造は放っておくと崩れます。orchestrator に任せておくと、ついでに自分でファイルを読み、ついでに自分で直してしまうからです。そこで次の 4 つを規約として明文化しました。

1. orchestrator は実装ファイルを Read しない。読んでよいのは triage.json / plan.md / verify.json などの制御情報だけ
2. orchestrator はコードを書かない・テストを走らせない
3. phase agent の model を Fable に引き上げない (Fable は orchestrator 専用)
4. orchestrator のトークン消費は task 全体の 15% 以下に収める

ここまで縛る理由は、規約が破れたケースを試算してみると分かります。仮に orchestrator が実装ファイルを読み始めて input が 300k tokens に膨らんだ場合、

```
Fable orch 分 = 0.30M × $10 + 0.01M × $50 = $3.50
→ 合計 $4.4 > ハーネスなし baseline $3.83
```

となり、規約を 1 つ破るだけで素の Claude Code より高くつきます。15% という数字はこの反転点から逆算して決めました。「気をつける」ではなく規約として書いてプロンプトと CI で強制する、というのがハーネスを作った理由でもあります。

## 実験: 同じ「マイクラ作って」を 4 構成に投げる

効果測定のお題は「マイクラ作って。ブラウザで動くやつ」にしました。ゼロからの実装で品質差が見た目に分かりやすいことと、放っておくとモデルがいくらでも作り込めるタスクなので、コスト管理の検証に向いていると考えたためです。同一プロンプトを 4 構成に投げました (headless 実行、コストは CLI の `total_cost_usd` の実測値)。

| 構成 | 金額 | 成果物 | 検証 |
| --- | --- | --- | --- |
| 素の Claude Code (Opus 単独) | $1.69 | ❌ 描画が崩壊したゲーム | なし |
| 素の Claude Code (Fable 単独) | $8.85 | ◎ 最高品質だが作り込みすぎ | なし (自己申告のみ) |
| **pev-harness** | **$5.33** | **◎ 同等品質** | ✅ 独立 verifier が PASS 判定 |
| pev-harness (実装バグが出た回) | $8.98 | ◎ | ✅ FAIL 検出 → 自動修正 → 再検証 PASS |

![素の Claude Code (Opus 単独)](https://static.zenn.studio/user-upload/541f7a43e066-20260708.png)  
*Opus 単独 ($1.69): 地形メッシュが崩壊していて、面欠けと浮遊断片が出ている*

![素の Claude Code (Fable 単独)](https://static.zenn.studio/user-upload/d2fa1757983f-20260708.png)  
*Fable 単独 ($8.85): テクスチャ・木・昼夜サイクルまで入った最高品質。ただし頼んでいない機能まで作り込んでいる*

![pev-harness](https://static.zenn.studio/user-upload/4e0b82ac3915-20260708.png)  
*pev-harness ($5.33): テクスチャ・10 種ホットバー・FPS 表示付きで Fable 単独に近い品質。これのみ独立 verifier の PASS 判定付き*

各構成で起きたことを順に書きます。

**Opus 単独 ($1.69)**: 一番安くて一番速い (6.7 分) のですが、成果物は地形メッシュが崩壊していて (面欠け・浮遊断片)、ゲームとして成立していませんでした。なお Opus は自分で smoke test を書いて「PASS しました」と報告してきます。テストが WebGL をスタブしていたため、描画バグは検出できていませんでした。自己申告の PASS は当てにならない、というのが最初の教訓です。

**Fable 単独 ($8.85)**: 品質は 4 構成で最高でした。56 FPS で、手続き生成テクスチャ、木、バイオーム、昼夜サイクル、AO、効果音まで付いています。ただ、頼んでいない機能もかなり含まれていて、20.5 分かけて自律的に作り込んでいった結果がこの金額です。Fable 単独が高くつく主因はモデルの能力ではなくスコープ判断で、どこで止まるかを決める仕組みがないと作り込みが止まらないようです。

**pev-harness ($5.33)**: Planner (Opus) が曖昧な点を根拠付きのデフォルトで確定し (ブロックは 8 種、効果音は Non-goal、のように範囲を明示)、実装は Sonnet が行い、検証は実装者とは別に起動された verifier が自前で 53 個の smoke test を書いて exit code で判定しました。品質はテクスチャ、10 種ホットバー、FPS 表示、フォグまで入って Fable 単独に近く、金額は Fable 単独比で −40%。4 構成の中で唯一、独立した検証が付いています。

**retry が発生した回 ($8.98)**: 別の回では、plan の受け入れ基準「ホットバーがクリックで切り替え可能」が executor の z-index バグを捕捉し、verifier が FAIL 判定、orchestrator が再実装を指示して 2 周目で PASS、という流れが全自動で走りました。この retry を続けるか打ち切るかの判断が、Fable に高い単価を払っている部分です。

## Fable の利用量はどれだけ減ったか

「Fable 単独と同等の性能を保ったまま、Fable の利用量をどこまで削れるか」という観点で、Fable 単独 run ($8.85 = 全額 Fable 課金) を基準に再集計します。

| 構成 | Fable への課金 | 削減率 | 品質 |
| --- | --- | --- | --- |
| Fable 単独 | $8.85 | — | ◎ (基準) |
| **pev-harness (現行 default)** | **$1.98** | **−77.6%** | ◎ 同等 |
| 〃 (品質基準を緩めた設定) | $1.48 | −83.3% | ○− 未達 (テクスチャなしの単色ブロック) |

仕組みとしては、harness 経由の場合に Fable が読み書きするのはタスク文、triage 結果、plan の見出し、verify の判定結果といった制御情報だけで、トークンを消費する実装・検証は Sonnet (と Codex) 側に流れるため、Fable 自身の課金対象が小さく保たれる、というだけの話です。

注意点として、「Fable 課金 −77.6%」と「総額 −40% ($8.85 → $5.33)」は別の数字です。削られた Fable 分が消えたわけではなく、Sonnet への支払いに付け替わっています。モデル別の予算管理をしたい場合は前者、支払い総額を抑えたい場合は後者を見てください。

また表の 3 行目の通り、削減率と品質はトレードオフです。成果物をどこまで作り込むかは、planner に渡している品質基準の指示文 (「テクスチャを描くこと」「ブロック選択 UI を表示すること」のように、要求品質を plan の受け入れ基準に含めさせる文言) で決まります。この指示を緩めた設定では Fable 課金は −83.3% まで下がりましたが、成果物はブロックがテクスチャのない単色 1 色で塗られただけの見た目になり、Fable 単独と同等品質とは言えませんでした。$1.69 (壊れている) から $8.85 (作り込みすぎ) までの間のどこを取るかを、この受け入れ基準で明示的に選べるようになったことが、ハーネス化のいちばんの効果だと考えています。

## 実測して分かったこと

事前の想定と違ったところを 3 つ挙げます。

1 つ目は、単価差は出力量で相殺されることです。orchestrator を Fable から Opus に差し替えた比較では、総額はほぼ同額でした ($5.33 vs $5.39)。Opus は単価が半分ですが、進行状況の説明が長く、出力トークンを約 2.5 倍使ったためです。単価の安いモデルに変えたのに安くならない場合は、トークン量側を疑う必要があります。

2 つ目は、Plan にコスト抑制効果があったことです。Plan phase は品質のための投資と考えていましたが、Fable 単独の $8.85 の主因がスコープの自律的な拡大だったことを踏まえると、planner が MVP の範囲を確定させること自体が、作り込みの暴走を止めるコスト装置として機能していました。

3 つ目は、品質指示の文言の強度で結果が振れることです。「テクスチャ相当の視覚ディテール」という指示は「単色 + 陰影で十分」と解釈されることがありました (実際にされました)。「ブロックを単色のままにしない」「ブロック選択 UI の表示は必須」まで文言を強めて、ようやく run 間の品質が安定しました。曖昧な形容詞は、コスト圧力の下では弱い方に解釈されるようです。

## 注意点

* 実測はいずれも n=1〜数 run で、run 間の分散 (retry の有無で ±$3 程度) は残っています
* ゼロからの単発タスクでは Opus 単独のほうが安いです ($1.69。ただし今回は壊れていましたが)。このハーネスの価値は「何より安い」ことではなく、Fable の判断品質を Fable 単独の 1/2〜1/4 の金額で、検証付きのパイプラインに組み込める点にあります
* 削減幅が出やすいのは、今回のような新規作成よりも、既存のコードベースに機能を足していく普段の開発です。素の Claude Code では既存コードの読み込みで入力トークンが膨らみ、その全部が高いモデルの単価で課金されますが、harness ではその読み込みを安いモデル側が引き受けるためです。さらに実装を Codex CLI に委譲する設定 (ChatGPT サブスクリプション内で動くため API の従量課金が発生しない) にすると、いちばんトークンを使う実装フェーズのコストがほぼゼロになります

## まとめ

Fable 5 を orchestrator に限定して使うと、判断品質を保ったまま Fable への課金を 1/4 以下にできました。ただしこの構造は自然には維持されないため、「実装ファイルを読まない」「コードを書かない」「トークン比率 15% 以下」を規約として強制する仕組みが必要です。また、検証は実装した agent とは別の agent に行わせるのを勧めます。今回の実験でも、実装した本人が「テストを書いて PASS した」と報告してきたのに、実際には描画が壊れているケースがありました (Opus 単独の回)。実装者の自己採点ではなく、別の agent が独立にテストを書いて判定する構造にしておくほうが安全です。

plugin は OSS で公開しています。Claude Code に plugin として追加するだけで動くので、興味があれば試してみてください。あわよくば使ってみての感想などフィードバックいただけると嬉しいです。コントリビュートがあるとさらに喜びます。
