初めまして。FLINTERS で情シス業務を担当している ishiguro です。

／  
AI 利用の潮流が速すぎる...！  
＼

そんなことを日々感じている今日この頃、先日、社内の Claude メトリクス（利用状況）を可視化する基盤を作成することになりました。  
今回、Terraform × Lambda × GAS という構成で作ったのですが、いや、正直ほぼ Claude Code に任せて "作ってもらった" のですが、本格的に Claude Code に任せて何かを作るのが初めてだった自分からすると速さと正確さにひたすら驚かされました。

「1ヶ月弱はかかるだろう」と覚悟していた作業が、Claude Code をフル活用したことで実質10日程度で終わりました。しかも、私は Terraform を触るのが今回が初めてだったのに...。

この記事は、誰かの参考になれば、というより、この驚きをそのまま聞いてもらいたいという、そんなお話。

## 作ったもの

まずは、成果物をざっくりとご紹介。

社内の Claude ユーザーが、

* 自分が今月どれくらいClaudeを使っているか
* エンジニアマネージャー陣が、組織全体の利用傾向を把握する

ということができるように、Claude のユーザー別の月次利用量・コストを取得して、日次でスプレッドシートに反映する仕組みです。全体像はこんなイメージ。

まずは「利用量・コストの可視化」というスコープに絞って最短でリリースすることを優先しました。

できる人がこの構成を見ると、「パパッと組めそう」とか「もっとブラッシュアップできそう」とか感じるかもしれませんが、自分なりに経験や時間が少ない中でよくまとめたほうだと思います。

## 前提となった開発条件

背景として、以下の3つの条件が同時に重なっていました。

1. **リリースまでの期間が限られていた**  
   a. 既に全社で Claude 利用が開始されていた状況だったので、メトリクスの可視化と分析は急務でした。
2. **LambdaやGASは経験があったが、Terraformは初めてだった**  
   a. 後述する理由から、触ったことなかった Terraform にチャレンジすることになりました。まとまった事前調査や事前学習の時間を確保できないスケジュール感だったので、とにかく不安でした。
3. **コードレビューやシステム挙動確認を担当する上長も、同時期に多くの業務を抱えていた**  
   a. この開発には「いかにして上長の負担を軽減するか」もテーマでした。  
   b. このテーマ、自分では気付けておらず、弊社の「他チームのエンジニアマネージャーと街コンのようにトークする」というイベントで、とあるマネージャーからアドバイスいただいた自分に対する課題です。このイベントの企画・開催と当該マネージャーにも大感謝！

正直、これまでの簡易的な集計ツールの開発実績から逆算すると、**1ヶ月程度はかかるだろうと覚悟していました**。この状況を変えたのが、**Claude Code** でした。

## Terraformなんて触ったことなかったのに

Lambda や GAS は経験がありましたが、Terraform は今回が完全に初めてでした（正確には、入社後の研修で軽く触りましたが、流石に忘れてしまっていました...）。AWS コンソール上で手動でポチポチ作った方が最初の一歩は早いかもしれない、とも思いましたが、世間や社内での利用潮流も汲んで「 Claude に手伝ってもらえば何とかなるか」という気持ちで Terraform を使うことに決めました。

結果、この判断は大正解でした。要件と「実現したいこと」を伝えるだけで、Claude Code がスルスルとコードを組み立ててくれました。書き方も`apply`した結果がどうなるのかも分からない状態からのスタートだったのに、想像していたような「未経験の壁」にはほとんどぶつかりませんでした。

しかも、ただコードを書いてもらって終わりではありません。出てきたコードを部分ごとに抜き出して「この`resource`ブロックは何をしているのか」と聞けば、その場でちゃんと解説してくれます。**手を動かすのはClaude、理解するのは自分**、という形で作業をしながら、自然と Terraform の書き方を覚えていくことができました。教材を一冊読むよりよっぽど早く身に着いたかも...？

極めつけに、開発の途中で検証環境から本番環境に切り替える際、構成が全てコードとして残っていたおかげで、コンソールから一つひとつ作り直すことなく、コードを流し込むだけで同じ環境をそのまま再現できました。ここでも「コードにしておいてよかった」としみじみ感じました。

## 上長のレビュー負担をどう減らすか

上長のレビュー負担を減らすために、やはりここでも Claude に上長の代わりとして働いてもらいました。やったことは大きく3つです。

### 1. PRを出す前に、Claude自身にレビュワーになってもらう

PR を出す前に、Claude に「レビュワー視点」で見てもらいました。具体的には、

* 要件との相違点はないか
* システム上の懸念点（取り分けセキュリティ上の懸念、エラーハンドリングが甘い等）はないか
* 不要な記述やリファクタリングできる箇所はないか

を指摘してもらい、その場で直す。例えば、権限設定を見せて「このロールにこの権限は本当に必要か」を聞くと、必要最小限のリソースだけに絞ったポリシーへの修正案を返してくれる、といった具合です。これを繰り返してから初めて上長に PR を出す、というプロセスにしました。結果として、上長が実際に見るコードが **「より読みやすく」、「指摘事項がより少なく」** なっており、レビューそのものが相対的に短時間・少負担になりました。

### 2. 「正確に動くエビデンス」としてテストをより網羅的に書く

テストコードがより確固たる動作の証拠となるよう、ここでも Claude を使い、書いたテストのカバレッジに抜漏れがないかを確認してもらい、不足している観点があれば追加で書いてもらいました。

### 3. READMEを「見ればわかる」レベルまで具体化する

README には、システムの目的・構成・使い方などを詳細に書いた上で、Claude に依頼してシステム構成図を作成し、README のプレビュー上で全体像がひと目で伝わるようにしました。文章だけでなく、自分の力量では書くに至れなかった「図」を作成して可視化することで、レビュー前の「何を作ったのか」の説明コストを大きく減らせたと思います。

この3つに共通しているのは **上長の時間を使わせる前に、自分の下で検証・可視化を終わらせておく** ということです。Claude Code は「コードを書く道具」としてだけでなく「レビュー前のセルフチェック相手」として使ったときに、想像以上の威力を発揮すると感じました。

## 結果：想定1ヶ月 → 実質10日

* これまでの開発実績から見積もっていた期間：**約1ヶ月**
* 実際にかかった期間：**本実装1週間＋レビュー・微修正3日 ＝ 合計10日程度**

期間だけを見ると単純に「速くなった」という話に見えますが、体感として大きかったのは**レビューのやり取り自体の往復回数が減った**ことです。レビューの前後で発生しがちな「待ち時間」そのものが、事前のセルフレビューによって圧縮された形になります。正直、リリース直前の時点で自分でも「あ、もうこれで完成か」と拍子抜けと言いますか、不思議な感覚でした。

## 学んだこと・注意点

とはいえ、何もかも Claude 任せにしていたわけではありません。驚きばかりの体験談ですが、自分の中で意識していたことも少しだけ共有しておきます。

* **丸投げはしない**：生成されたコードは必ず読んで、分からない部分はその場で Claude に聞いて理解してから次に進むようにしていました。理解を飛ばして進めると、後で自分がレビューできなくなったり、修正指示ができなくなったり、様々リスク・弊害が出てきます。
* **セルフレビューはあくまで一次チェック**： Claude によるレビューで指摘がゼロになっても、最終的な承認や責任は人間にある、という前提と意識は崩していません。
* **ドキュメントへの投資は先払いした方が安い**：README・構成図に要する時間（ Claude を働かせているのでそこまで時間は掛かっていませんが。）は一見遠回りに見えますが、結果的にレビュー時間の短縮という形で回収できると思いました。

## おわりに

Terraform未経験、時間も限られている、上長のレビューにかけられる時間も限られている...。正直、条件だけ自分の力量で見たら簡単ではないと思っていました。それでも Claude Code をフル活用したことで、想定の3分の1以下の期間で、しかもほとんど正確に想定挙動する状態で形にできてしまいました。

Claude Code の速さと正確さにひたすら驚いた、という感想をそのまま書き残しました。AI駆動開発ええやん、という気持ち。開発頻度は他チームそこまで多くないですが、他業務も含めてもっと Claude くんを業務に介入させて、バリバリ働いてもらおうと思いました。
