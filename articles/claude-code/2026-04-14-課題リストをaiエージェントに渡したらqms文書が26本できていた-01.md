前回は、4人の部長エージェントが提出した業務フローを横断レビューし、出てきた10個の課題に対応優先順位をつけて終わりました。  
前回：[4人の部長エージェントが提示した業務フローを統合したら、やっぱりMECEじゃなかった](https://zenn.dev/tetsuro_works/articles/ai-agent-frontier-log-05))

今回は、前回まとめた課題リストvol.1に対して、私から対応方針だけ指示を出して、実行はClaude Codeに任せて解決していく話です。  
本検証における「解決」とは、QMS文書を出力し定型化・標準化・明文化することと同義と捉えいただければ幸いです。

---

## スタート地点：文書数　1本

前回の終了時点で、課題「🔴 重大な抜け漏れ」に対する対応方針を伝えたことで、'品質委員会規定.md'と、'残タスク.md'という文書が生成されていました。  
残タスクはあくまでオペレーション上の文書なので、'品質委員会規定.md'のみ正式なQMS文書として採用します。

それではQMS文書体系の構築をスタートしましょう。

私のプロンプト

```
▶ 課題リストの中で緊急度の高いものから優先的に対応し、その結果をQMS文書として明文化してください。
▶ 先にQMSに必要な上位文書から作って、下位の文書に落とすようにしてください。
▶ 今後作成するすべての文書は、文書管理番号付きのファイル名にしてください。
```

指示を出すと、Claude Codeから方針に対する質問がきますので、「こんな風に進めてほしい」という希望だけ伝えればどんどん文書を作成してくれます。  
もちろん、よくわからないところは「任せる」「よしなに」でも大丈夫です。

他のことをやりつつ、合間にClaude Codeからの承認依頼をぽちぽちEnterしていると、

Claude Codeの返答：生成されたQMS文書　26本

**気づけば文書が1本→26本**に膨れ上がっていました。

Claude Code利用制限や、検証時間が取れない日もあったので、すべて出力しきるまで3日ほどかかりましたが、Claude Codeの作業時間で言えば1時間かからないくらいじゃないかと思います。  
検証を始めてからほんの数日でQMSのアウトラインが見えてきたことに、ただただ感動です。

## アウトラインが見えてきたところで、Claude Codeにレビューを依頼

ひとまず課題への対応が終わったので、全体レビューを行ってみましょう。

今回は2つの視点でレビューを行います。

1. 先に4人の部長エージェントによる主観的なレビューを依頼
2. 部長エージェントのレビュー結果を踏まえて、Claude Codeに全体レビューを依頼

私のプロンプト

```
▶ 26本の文書を各部長エージェントに各々の部署の視点からレビューをしてもらい、1つの報告書にまとめてください。
▶ それがまとまったら、あなた（Claude Code）が全体レビューを行ってください。
```

この指示によって4人の部長エージェントが並行起動し、次々に文書を読み込んでレビューを始めます。

こちらも処理量が多くProプランの5時間ごとの利用制限に引っかかったりもしましたが、翌日処理を再開して無事レビューが完了しました。

まずは部長からのレビュー結果まとめです。

## Claude Codeを使えば、QMS文書の[抜けもれ、矛盾点、是正ポイント]を洗い出すことが可能

次に、4部長の指摘事項をClaude Codeに全体レビューしてもらい、新たな課題リストとして整理してもらい、高優先のタスクを洗い出します。

| 優先度 | 事項 | 主な指摘者 |
| --- | --- | --- |
| 🔴 即対応 | RD-DOC-001確定（量産バリデーション実施） | 全員 |
| 🔴 即対応 | QA-MAN-001第3版改訂（文書一覧更新） | 田中・佐藤 |
| 🔴 即対応 | アレルゲン検査結果→アレルゲン一覧制定フロー確定 | 中村 |
| 🟠 今月中 | QA-PRO-001・QA-PRO-004の保管期間修正（5年へ統一） | 田中・山田 |
| 🟠 今月中 | 金属探知機テストピースサイズをMFG-PRO-002に記入 | 山田 |
| 🟠 今月中 | 製造指示書（MFG-FRM-001-D）フォームの制定 | 山田 |
| 🟠 今月中 | 年間校正計画書（MFG-FRM-002-A）の作成・外部校正機関へ依頼 | 山田 |

この新たな課題リストvol.2から優先度が高い順に対処していくことになります。  
ここまでの検証によって**QMS文書管理のPDCAのサイクルを高速で回していくことができる**ことがひとまず証明できたかと思います。

---

こういった品質管理の文書作成は、理想の姿を描いたあとに作りこんでいくべきものですが、**時間に追われとりあえず文書を揃えることに注力してしまい**、本質からずれてくることは珍しくありません。

Claude Codeを使えば、とりあえず揃えることにほとんど労力がかからないので、**理想の姿を描くことに時間を使っていける**という点でとても有用だと思います。

PDCAを回すうえでも、抜けもれ、矛盾点、不適合のポイントを網羅的にチェックし、全体の整合性をとりながらアクションリストに落とし込むことも可能です。

---

とはいえ、実際の運用ではアウトプットされた文書の内容を人間が理解し承認する時間が律速になるのが理想なのは、他のAI活用領域と同様でしょう。

それすらもAIエージェントに業務手順書を作成させたり、各現場の細則を学習したAIエージェントを作成したりすることでフォローできると思いますので、AIエージェントファーストの業務設計が求められる理由がよく分かりました。

次回からは、もう少し細かな課題解決に取り組んでみたり、試作品のクッキーを製造させてみたり、いろいろなイベントを発生させてClaude Codeを使った解決にトライしていこうと思います。
