## TL;DR

個人開発で複数のLLM API・CLIツールを使い倒していたら、性質の異なる4つの障害に同じ時期に立て続けに当たりました。

* **無料枠が予告なく削減される**（Gemini CLI → Antigravity CLIで1日1,500req→20reqへ）
* **依存先のモデルが無料枠対象からひっそり外れる**（Vauban運用中、`gemini-2.0-flash`が`limit: 0`に）
* **高性能モデルのレート制限に頻繁に当たる**（Claude Opusを気軽に使いすぎて詰まる）
* **API課金が個人開発の財布を溶かす**（Claude Visionでの毎日大量呼び出し）

原因はバラバラですが、教訓は1つでした。**LLMのモデル・価格・無料枠は、自分がコードを1行も変えていなくても、ベンダー都合で変わる。** 個人開発でLLM APIに依存するなら、それを前提にした設計をしておかないと、ある日突然パイプラインが止まります。

この記事では、実際に踏んだ4つのインシデントと、そこから固まった3つの設計パターン（プロバイダ抽象化・タスク別モデルルーティング・自動フォールバック）を、実コードとともにまとめます。

## 起きたこと: 4つのインシデント

### インシデント1: 無料枠が突然消えた（CLIツール）

Claude Codeをサブスク（月額固定）で使う一方、GeminiとCodexのCLIを無料枠で組み合わせられないかと調べたときの話です。

Gemini CLIは後継のAntigravity CLIに移行するタイミングで、無料枠が段階的に削られていきました。

| 時期 | 無料枠 |
| --- | --- |
| 2025年11月（登場時） | 250リクエスト/日 |
| 2025年12月 | 20リクエスト/日（92%削減） |
| 2026年3月〜 | クレジット制＋週次ロックアウト導入 |

**週次ロックアウト**は、週の上限に達すると最大168時間（7日間）使えなくなる仕様です。Opusを使うような複雑なセッションなら1〜2回で上限に達したという報告もあり、実際にロックアウトを経験しました。

回避策として「Google AI StudioのAPIキーをCLIに設定する」を試みましたが、Antigravity CLIはAPIキー非対応でした（デスクトップアプリ版のみ対応）。Codex CLIも無料枠は「評価・軽い試用向け」止まりで、本格運用にはChatGPT Plus（$20/月）以上が必要——Claude Codeとの二重課金になり、ローコストという前提が崩れました。

結果、Claude・Gemini・Codexを併走させて相互レビューさせる構想は、無料枠が実用に耐えないという理由だけで潰れました。

### インシデント2: 依存先のモデルがひっそり無料枠から外れた（CI）

複数リポジトリにセキュリティチェックの仕組み（[Vauban](https://github.com/mediiiiium/vauban)）を展開して運用していたところ、ある日 `git push` が通らなくなりました。

```
Gemini Code Review...Failed

google.api_core.exceptions.ResourceExhausted: 429 You exceeded your current quota
* Quota exceeded for metric: generativelanguage.googleapis.com/generate_content_free_tier_requests
* Quota exceeded for metric: generativelanguage.googleapis.com/generate_content_free_tier_input_token_count
```

使いすぎたのかと思って調べると、原因は逆でした。push前の差分レビュー層が呼んでいた `gemini-2.0-flash` が、いつの間にかGoogle AI Studioの無料枠対象から外れ、`limit: 0`——事実上使用不可——になっていたのです。無料枠の対象モデルはGoogle側が管理しており、古いモデルほど新モデルへの移行を促すために枠を絞られる傾向があります。「古い＝枯れてて安定」ではなく「古い＝いつ切られてもおかしくない」が実態でした。

直し方自体は簡単で、展開スクリプト（`setup.sh`）を再実行すれば `gemini-2.5-flash`（無料枠で動作確認済み）に更新されます。

```
bash ~/vauban/setup.sh ~/delivery-app none
```

問題は、この仕組みを7つのリポジトリに展開していたことです。本体が更新されても、各リポジトリのファイルは自動では追従しません。Googleが無料枠モデルを変えるたびに、更新が漏れたリポジトリだけが同じエラーで詰まります。手動巡回は現実的ではないので、Claude Codeのクラウド定期実行を使い、月次ルーティンを組みました。

1. Vaubanの最新版を参照
2. 対象リポジトリすべてに `setup.sh` を再実行
3. 変更があればコミット・プッシュ
4. 変更がなければ何もしない

ファイルをコピーしてコミットするだけの作業なので、モデルは一番軽い Haiku で十分です。

### インシデント3: 高性能モデルのレート制限に頻繁に当たる

Claude Codeで開発しているとき、最初は重要な判断もそうでない実装も、区別せず最高級モデル（Opus）に投げていました。Opusは賢いぶんレート制限がきつく、何度か「使いすぎ」で止まりました。

そこで役割を分けました。

* **Sonnetに任せる**: コードを書く・調べる・検索する・修正する——実装作業全般
* **Opusに聞く**: 方向性の判断、設計評価、重要な意思決定の前

CLAUDE.md（AIへの設定ファイル）にルールとして明文化しています。

```
Opus は以下のときだけ使う：
- コードレビュー・セキュリティレビュー
- 複雑なロジック・アーキテクチャの評価
- 重要な意思決定が必要な場面

通常の実装・調査・編集は Sonnet をデフォルトで使う。
Opus は rate limit の消耗が早いため、惜しみなく使わない。
```

判断の分岐点だけOpusに当てるようにしてから、レート制限にかかる頻度は大きく下がりました。「新人が細かい作業、中堅が実務全般、外部コンサルタントがここぞの判断」くらいの粒度感です。

### インシデント4: API課金が個人開発の財布を溶かした

ビール情報サービス（Now on Tap）で、毎朝タップリスト画像をClaude Visionに渡してJSON抽出する処理を全店舗ぶん回していました。精度は良い一方、呼び出し回数（判定+抽出で1画像あたり2回×全店×毎日、過去画像の再解析も含む）に比例して課金が膨らみ続けました。

安いモデルに乗り換えたいが、精度をこれまで潰してきた誤読・誤分類バグごと巻き戻したくない——というのが課題でした。そこでモデルを乗り換える前に、まず設計を変えました。

## 抽出できる設計パターン

4つのインシデントから、実際にコードへ落とし込んだパターンは3つです。

### パターン1: 契約を固定してプロバイダを差し替え可能にする

精度を決めているのはモデル本体ではなく、プロンプトと後処理だと割り切り、レイヤーを分離しました。

* `prompts.js` … 判定プロンプト・抽出プロンプト・JSONパース・後処理を単一ソースで持つ
* `providers/gemini.js` / `providers/claude.js` … 「base64画像とプロンプトを渡したら生テキストを返す」だけの薄い実装
* `vision.js` … 公開API（`analyzeTapList` / `extractBeers`）。**署名は不変**

```
画像 → [gemini or claude] → 生テキスト → 同じparse → 同じ後処理 → 同じ品質ガード
                ↑ここだけ差し替え可能
```

呼び出し側（毎朝のパイプライン、全店巡回、各スクレイパー）は1行も変更していません。モデルは「文字を生成する箱」に格下げされ、精度の本体（評価ロジック）は触らずに済みます。

メインをGemini 3.1 Flash Lite（無料枠: 1日500回・1分15回）に切り替え、安全マージンを見て4.2秒間隔のゲートを噛ませて逐次処理。`temperature: 0`、`thinkingBudget: 0` でブレを抑えています。さらに精度の歯止めとして、**抽出が3件未満ならタップリスト判定そのものを取り消す**品質ガードを入れ、安いモデルが空振りしても壊れたデータがDBに入らないようにしています。

### パターン2: タスクの重さでモデルを振り分ける

「実装は安いモデル、判断は高いモデル」という役割分担をCLAUDE.mdのようなルールファイルに明文化しておくと、レート制限の消耗そのものを構造的に抑えられます。CIのような軽作業（ファイルコピー＋コミットの月次ルーティン）にも同じ発想で最軽量モデルを割り当てています。

### パターン3: 自動フォールバック＋通知

無料枠は有限で、ベンダーが気まぐれで枠を絞ることもあります。そこで保険を1層挟みました（`fallback.js`）。

* Geminiが日次枠切れや失敗を返したら自動でClaudeに切替、そのプロセスは以降Claudeで走り切る
* 切り替わった瞬間にSlackへ1回だけ通知（「今回はClaude課金が発生します」）
* `VISION_PROVIDER=claude` の環境変数1つで、いつでも手動でClaude常用に戻せる

無料枠が改悪されても、コードを変えずに避難できる状態にしてあります。

### パターン4（設計というより運用）: ドリフトは定期ジョブで検知・修復する

Vaubanのように「1つの仕組みを複数箇所に展開する」構成では、本体の更新に各展開先が追従しない、というズレが必ず発生します。これは「壊れるまで気づかない」タイプの問題で、人力の巡回では確実に漏れます。月次のクラウドルーティンのように、差分があれば直して、なければ何もしない、という冪等なジョブを自動で回しておくのが唯一の現実的な対策でした。

## まとめ

| 障害モード | 対策パターン |
| --- | --- |
| 無料枠が突然削減・ロックアウトされる | 無料枠に依存し切らない設計、プロバイダ切替を前提にする |
| 依存先のモデルが無料枠から静かに外れる | 契約（インターフェース）を固定してプロバイダを薄く差し替え可能にする |
| 高性能モデルのレート制限に頻繁に当たる | タスクの重さでモデルを振り分け、ルールとして明文化する |
| 複数箇所に展開した設定が本体更新に追従しない | 定期ジョブで差分検知・自動修復し、人の記憶に頼らない |
| API課金が想定以上に膨らむ | 品質ガード付きで安いモデルに寄せつつ、自動フォールバックで保険をかける |

「いいモデルを選ぶ」より「どのモデルでも同じ品質が出る器を作る」方が、個人開発では長く効きます。モデル・価格・無料枠はこちらの制御が及ばない外部要因である以上、壊れた後に直す設計ではなく、壊れる前提で組んでおく設計のほうがコストが低いというのが、4つのインシデントを踏んだ結論です。
