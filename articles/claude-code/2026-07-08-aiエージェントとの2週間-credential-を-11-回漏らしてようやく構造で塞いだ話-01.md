## TL;DR

* multi-agent orchestration で AI agent に作業させ始めた
* session jsonl / cmdline / container env / API echo back / process inspection から **2 週間で 11 回 credential が漏れた**
* positive rule + pre-flight checklist という behavioral defense は **同じ vector で再発** + **新 vector class が次々出現**して破綻
* 最終的に **(α) repo-baked script idiom** + **(γ) bash hook の input gate (23 patterns / 53 test pass)** + **(δ) PostToolUse hook の output sanitize** の 3 層で漸近
* 個別 test pass ≠ real-world robust。**word boundary なしの regex** と **compound command 跨ぎの greedy match** で同 session 内 false positive 2 件、word boundary `[^a-zA-Z_/]` 前置と separator-aware `[^&|;<>]+` で fix
* AI 時代の credential 防衛は **「鍵」じゃなくて「入力経路」** に boundary を引く

---

## はじめに

自宅で multi-agent orchestration 系の自作 framework (= njslyr / shogun 系列) を回し始めた。複数 AI agent が並列に code を書き、git commit し、container を deploy する構成。

最初は楽しかった。**ただ、credential が異常な数で漏れ始めた**。

session jsonl に env 値、process cmdline に inline flag、container env dump、digest md への転写。1 incident じゃなくて、short period に 4-8 件連発する pattern が出た。

「これは structural fix なしじゃ無理だ」と気付いて SOPS 導入、`CLAUDE.md` (= AI agent への instruction file) に 2 コマンド原則書いて、誘惑表書いて、pre-flight checklist 書いた。**それでも漏れた。**

最終的に、bash hook で入力ゲート + repo-baked script で shape 固定 + PostToolUse で出力 sanitize の 3 層で漸近させた。本記事はその 2 週間の failure log と、辿り着いた structural defense の記録。

## Chapter 1: 最初の 4 incidents — 「同じやつだろ」を AI に言うのが効かない

### #1 (2026-04-13): `env | grep ZNAS_PG_PASS`

最初は **env 確認**だった。

stdout に value 全文焼付。session jsonl は **persistent file system に平文保存**されるので、`Anthropic 側削除依頼` が必須になった。

教訓: env 確認は `env | awk -F= '{print $1}'` (= key 名のみ) で済む。

### #2 (2026-04-14): `sh -c "env | grep ANTHROPIC"`

**1 日後に同じパターンで再発**した。container 内 env を確認する文脈で。

```
sh -c "env | grep -iE ANTHROPIC"
```

「やめて」って言ったのに 1 日でループ。これが behavioral rule の限界。

### #3 (2026-04-15): `bash -x` で `printf $KEY` xtrace expand

debug 目的で `bash -x` 使ったら、`printf $CF_ANTHROPIC_API_KEY` が xtrace で stderr に value expand されて 2 key 同時 leak。

教訓: secret 扱う script に `bash -x` / `set -x` / `-o xtrace` **絶対禁止**。

### #4 (2026-04-22): `sops -d file | head` で 5 key 同時 leak

決定打はこれ。

```
sops -d ~/projects/creds-migration/secrets-template/llm.enc.yaml | head -10
```

CF\_ANTHROPIC / OPENAI / GEMINI / PG\_REPL\_USER / PG\_REPL\_PASSWORD の **5 key を一度に焼付**。直前に JATS XML を `head` で多用してて、同じ pattern を sops 文脈に持ち込んでしまった。

self-diagnosis:

* **Context 流用**: 別文脈の `head` pattern が credential 文脈に転移
* **Negative framing の限界**: 「`sops -d | head` 禁止」rule は別文脈からの pattern 流用に trip-wire 利かず
* **Pre-flight check 欠如**: credential file path を認識した瞬間に止まる mental break 機構不在

### Chapter 1 小括

4 incidents 全部「**outer pipe で stdout に流れる**」同 root cause。AI に「やめて」って言うだけだとループする。**negative rule (= 禁止) は同 vector 再発を防げない。**

## Chapter 2: positive rule + pre-flight checklist 導入 (= behavioral defense の最大形)

ここで `CLAUDE.md` を全面改訂した。

### A. positive rule (= 2 コマンド原則)

「**禁止**」じゃなくて「**これだけ**使え」と書く。

```
sops edit <file>           # 編集時 (= EDITOR で decrypted 編集、save で再暗号化)
sops exec-env <file> '<cmd>'  # 使用時 (= subprocess の env に注入、stdout に value 出ない)
```

それ以外の sops 構文は**書かない**。`sops -d` は構文として封印。

### B. 誘惑 → 代替表

| やりたい事 | ❌ 違反 | ✅ 正解 |
| --- | --- | --- |
| ファイル構造見たい | `sops -d file | head` | `sops exec-env file 'env | cut -d= -f1 | sort'` |
| 値 verify | `echo $KEY` | `sops exec-env file 'python3 -c "..."'` (= bool 判定) |
| 一時的 env 注入 | `export $(sops -d file | xargs)` | `sops exec-env file 'your_script.sh'` |

### C. pre-flight 3 秒 checklist

```
□ command に `sops -d` はあるか？               → あれば NG、書き直し
□ 出力が outer pipe (head/cat/grep/tee/...) に流れるか？ → あれば NG、書き直し
□ 形は `sops edit <file>` か `sops exec-env <file> '<inner>'` か？ → 他なら NG
```

### D. 事故年表を CLAUDE.md 冒頭に

「**2 週間で 4 回**」を年表で見せる。AI が毎 session 開始時に読む CLAUDE.md の冒頭 100 行内に置く。

これで完璧と思った。

## Chapter 3: それでも漏れた #5-#8 — 異 vector class の出現

### #5 (2026-04-22 16:41、#4 の 29 分後): NCBI API key

```
curl ... "api_key=$NCBI_API_KEY" ... | head -5
```

NCBI idconv API は **response の `echo` field に request params が丸ごと含まれる仕様**。`api_key=` が response body に焼付、`| head` で会話ログに流出。

`#4` から **29 分後**。SOPS 文脈外の curl pattern。pre-flight checklist は SOPS 専用 rule と認識してて、API echo back は想定外だった。

### #6 (2026-04-26): rclone.conf を `tail -8`

```
tail -8 /root/.config/rclone/rclone.conf
```

S3 access\_key + secret 全文露出。rclone.conf は **sops 暗号化されてない credential file**。pre-flight checklist は「sops file」しか見てなかった。

### #7 (2026-04-26 night): `pgrep -af` で process cmdline

別 agent (= multi-agent orchestration の workers) が rclone を `--s3-access-key-id <K> --s3-secret-access-key <S>` で起動してて、`pgrep -af` で **process cmdline 全展開**で焼付。`#6` と同 IAM key の **再露出**。

### #8 (2026-04-26 dawn): rendered script を `head` + sed redact

envsubst で credential expand 済の rendered pull script を deploy 検証目的で:

```
head -15 /root/norway_pull.sh | sed 's/AWS_VASTAI_ACCESS_KEY_ID="[^"]*"/REDACTED/'
```

sed pattern が `AWS_VASTAI_*` を期待してたが script 中身は `RCLONE_CONFIG_S3AWS_*` (= export 名 mismatch) で **redaction 全 fail**、raw value 直接 stdout。**sed redact は silent failure する** (= pattern mismatch でも exit 0、誤った安心感)。

### Chapter 3 小括

positive rule は **`sops` 文脈にだけ trip-wire 効く**。credential が流れる経路は無限:

* file dump (= `cat .env` / `tail rclone.conf` / `head /etc/secret`)
* process arg (= `pgrep -af` / `ps eww` / `cat /proc/<pid>/environ`)
* rendered template (= envsubst 済 script の inspect)
* API echo back (= request param が response body に含まれる仕様)
* container env dump (= `docker inspect --format '{{.Config.Env}}'`)
* derivative artifact (= digest md / archive / dump への転写)

5 個の vector class が 1 週間で出現した。

## Chapter 4: 4/27 dawn 連発 #9-#11 — AI の self-recursive leak

ここから 1 night で 3 件連発した。

### #9: docker inspect format で全 env dump

```
docker inspect pg --format "{{range .Config.Env}}{{.}}\n{{end}}"
```

`POSTGRES_PASSWORD=<value>` が含まれた全 env を dump。session jsonl 内 **83 occurrences** で焼付。

### #10: digest md への転写

同 session で user 指示「会話ログ残しておこう」、digest md 作成時に **#9 で焼付済の値をそのまま転写**、md ファイルに sanitize なしで露出。

「すでに見えてる context だから安全」と AI が誤認した。新 artifact 作成時の re-leak risk 認識欠如。

### #11: 流出を報告すれども sanitize しない怠慢

> 「流出した時報告してくるじゃない。それと同時にサニタイズしなよ。」

これは user 側の言葉。AI は leak 検知 → user に報告 → 指示待ち → sanitize、という遅延 mode で動いてた。

**正解 mode**: leak 検知 → **報告と同時に active session jsonl を即時 in-place sanitize** (`sed -i`)、user 確認待ちなし。

構造的: AI は `context は ephemeral` と思いがちだが、`~/.claude/projects/.../*.jsonl` は **persistent file system、即焼付、明示削除しないと永続**。

### Chapter 4 小括

reflexive procedure を rule 化:

```
# leak 検知時、報告と同時に実行
SRC=$(ls -t ~/.claude/projects/-home-hrmtz-projects-*/[a-z0-9-]*.jsonl | head -1)
sed -i -E "s/<leaked-value>/<REDACTED>/g" "$SRC"
# その上で user に報告 + rotate 計画
```

「user 確認待ち」は backward damage を拡大する。

## Chapter 5: SOPS 鎖国 第一波 — bash hook で入力ゲート

11 incidents で **明らかに behavioral rule 単独では破綻**。`~/.claude/hooks/` 配下に hook を置いた。

Claude Code の `PreToolUse:Bash` hook で、bash command を **実行前に regex match で block** する。`settings.json` で:

```
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "bash ~/.claude/hooks/bash_command_guard.sh",
        "timeout": 5
      }]
    }]
  }
}
```

10 patterns を catalog 化:

| # | pattern | block reason |
| --- | --- | --- |
| A1 | `sops[[:space:]]+(-d|--decrypt)` | sops -d / --decrypt は positive rule 違反 |
| A2 | `env[[:space:]]*|[[:space:]]*(grep|...)(KEY|TOKEN|PASSWORD|SECRET)` | env 値 dump |
| A3 | `docker[[:space:]]+inspect.*--format.*\.Config\.Env` | docker inspect で env 全 dump |
| A4 | `bash[[:space:]]+-x.*(printf|echo).*\$[A-Z_]+` | bash -x の expand |
| A5 | `cat[[:space:]].*(\.env|\.aws/credentials)` | env / credentials 平文露出 |
| A6 | `(head|tail)[[:space:]]+.+\.enc\.(yaml|json)` | sops encrypted file の head/tail |
| A7 | `curl.*-H.*Bearer[[:space:]]+[A-Za-z0-9_+/=-]{30,}` | curl inline literal token |
| A8 | `rclone.*--s3-access-key-id[[:space:]]+[A-Za-z0-9]+` | rclone inline flag |
| A9 | `curl.*\?api[_-]?key=` | URL query api\_key= |
| A10 | `tail[[:space:]].+(rclone\.conf|\.netrc)` | credential file tail |

block 時の output:

```
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "🛡 Bash command blocked by credential leak guard:\n- ..."
  }
}
```

これで AI が違反 command を書いた瞬間に **実行前で停止**。

### `credential_value_scrub.sh` (PostToolUse)

別 hook で、bash output に credential pattern が出たら **active session jsonl を `sed -i` で in-place sanitize**。

```
PATTERNS=(
    'POSTGRES_PASSWORD=[a-zA-Z0-9_!@#$%^&*-]+|POSTGRES_PASSWORD=<REDACTED>'
    'sk-ant-[a-zA-Z0-9_-]{20,}|sk-ant-<REDACTED>'
    'AKIA[0-9A-Z]{16}|AKIA<REDACTED>'
    'TURSO_AUTH[A-Z_]*=eyJ[a-zA-Z0-9._-]+|TURSO_AUTH=<REDACTED>'
    # ...
)
```

これは **reactive damage control**。漏れたものを後追いで消す。forward は rotate 必須なのでこれだけでは足りないが、jsonl 焼付の persistent leak は防げる。

第一波で **4 ヶ月安定**した。

## Chapter 6: 12 回目の incident (= zetith-emdash 開発中、shell mis-parse vector)

最近、副業の WordPress → emdash CMS 移行 sandbox で CF Stream API token を扱う場面が出た。

私は AI に対して以下の curl を user terminal で実行するよう指示した:

```
sops exec-env ~/.config/cloudflare/wrangler.enc.yaml \
  'curl -s -H "Authorization: Bearer $CF_STREAM_API_TOKEN" \
    -d "{\"url\":\"...\",\"meta\":{\"name\":\"_test\"}}" \
    https://api.cloudflare.com/...'
```

user 側 terminal で copy-paste した時、**改行と quote escape が壊れた**。bash が:

```
sops exec-env file '
curl -s -H'
'Authorization: Bearer cfut_xxxxxx... ': command not found
'-d ...'
```

と解釈、**`Authorization: Bearer cfut_...` を独立 command として exec**しようとして:

```
/bin/sh: 2: Authorization: Bearer cfut_xizyHy...XX: not found
```

を stderr に出力 → session jsonl に **expanded value 焼付**。

### なぜ既存 hook が捕まえられなかったか

A7 pattern: `curl.*-H.*Bearer[[:space:]]+[A-Za-z0-9_+/=-]{30,}` は **post-expansion の literal token (30+ chars)** を block 設計。

ただし今回の vector は:

* pre-execution の command source は `curl ... -H "Authorization: Bearer $CF_STREAM_API_TOKEN"` (= shell var literal、未展開)
* hook には「まだ token じゃない」と見える、**block しない**
* shell mis-parse は hook 通過 **後** (= bash exec time) に発生
* post-expansion の stderr leak は hook の責任範囲外

= **input gate 単独では shell parse 失敗由来の post-expansion leak は防げない**。

「あ、これか」と気付いた。

## Chapter 7: 構造的対策 完成 — α (shape rule) + γ (input gate)

### α: repo-baked script idiom

inline curl を**書かない**ようにすればいい。

```
scripts/cf-stream/
├── README.md
├── verify-token.sh           # token 動作確認
├── get-customer-code.sh      # customer-code 取得
├── list.sh                   # 動画一覧
├── upload.sh                 # mp4 upload + R2 master 保存
└── delete.sh                 # 動画削除
```

各 script は **shell-quote 確定済**。JSON body は heredoc + `--data @file` 経由で書く:

```
PAYLOAD=$(mktemp -t cf-stream-copy.XXXXXX.json)
trap 'rm -f "$PAYLOAD"' EXIT
cat > "$PAYLOAD" <<'JSON'
{
  "url": "https://...",
  "meta": { "name": "_token-verify-delete-me" }
}
JSON

curl -s -X POST \
    -H "Authorization: Bearer ${CF_STREAM_API_TOKEN}" \
    -H "Content-Type: application/json" \
    --data @"$PAYLOAD" \
    "${API_BASE}/copy"
```

`--data @file` 経由なら **terminal copy-paste で壊れない**、shell quote escape の盲点を構造的に潰す。

呼び出し idiom:

```
sops exec-env ~/.config/cloudflare/wrangler.enc.yaml \
  scripts/cf-stream/upload.sh ~/videos/foo.mp4
```

inline ad-hoc は禁止。

### γ: bash hook 拡張 (= 13 new patterns + A 系拡張)

**red team mode で 27 attack surfaces を列挙**してから書いた。「悪意ある AI agent が credential を盗む command を全列挙したら何が出るか」。

新 13 patterns:

| # | pattern | rationale |
| --- | --- | --- |
| B1 | `printenv VAR_TOKEN_PASSWORD_SECRET_CRED` | env 値直 expand |
| B2 | `echo $VAR_TOKEN_...` | echo 値 expand |
| B3 | `printf $VAR_TOKEN_...` | printf 値 expand |
| B4 | `(declare|typeset|export) -p VAR_TOKEN_...` | shell builtin の env dump 機能 |
| B5 | bare `set` or `set | grep` | 全 env+func dump |
| B7 | `cat /proc/<pid>/environ` | /proc 経由 env 平文 dump |
| B8 | `ps eww` / `ps -o environ` | BSD ps の e flag は env dump |
| B9-12 | `sops exec-env file '(python|node|ruby|perl|bash|sh) -[ce]'` | wrapper bypass |
| B11 | `sops exec-env file '... eval ...'` | eval 完全 bypass |
| B13 | `sops exec-env file '... > /file'` | output redirect で leak-to-file |
| B15 | `env > file` | env 全 dump to file |

A 系拡張:

* A2: `env|grep` → `env|(grep|awk|sed|fgrep|egrep|rg|tr)` で grep family 全部
* A7: `-H "Bearer ..."` → `-H "Bearer ..."` and `--header "Bearer ..."` 両 form

そして **重要**: SOPS 経由 curl の inline pattern を block する新 pattern:

```
sops[[:space:]]+exec-env[[:space:]].+['"].*[[:space:]]*(curl|wget|http|axios)[[:space:]]
:::sops exec-env で inline curl/wget は quote escape 失敗で env expansion → token leak (#5 vector 2026-04-29)、scripts/cf-stream/*.sh 等 repo-baked script に固定
```

これで `sops exec-env file 'curl ...'` が **block される**。代替は `sops exec-env file scripts/cf-stream/upload.sh ...`。**shape rule (= 形だけ許可) を hook で強制**。

### 同 session 内で出た FP 2 件

13 patterns 一括追加直後の real-environment 検証で false positive 2 件発覚:

**FP 1: B8 `ps` が `sops` の suffix `ps` に substring match**

```
sops edit ~/.config/cloudflare/wrangler.enc.yaml
       ^^                  
       ↑ ここを ps と誤認、`ps edit` matched
```

B8 regex: `ps[[:space:]]+[a-z]*e[a-z]*([[:space:]]|$)` が `sops`  の `ps`  部分に match。**word boundary 不在**が原因。

fix: `(^|[^a-zA-Z_/])ps[[:space:]]+...` で先頭境界を強制。

**FP 2: A6 head/tail `.enc.yaml` が compound `&&` 跨ぎで greedy match**

```
list.sh | head -10 && sops edit ~/.config/cloudflare/wrangler.enc.yaml
          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
          ↑ greedy `.+` がここ全体を 1 match と解釈
```

A6 regex: `(head|tail)[[:space:]]+.+\.enc\.(yaml|json)` の `.+` が `&&` を跨いで `.enc.yaml` まで match してしまう。

fix: `(head|tail)([[:space:]]+[^[:space:]&|;<>]+)+\.enc\.(yaml|json)` で **command separator (`&|;<>`) を含まない token のみ** に制限。

### test fixture

`/tmp/hook-test-fixtures.sh` で全 pattern を test:

```
run_test "B8 ps eww direct"   "BLOCK" "ps eww"
run_test "B8 sops edit"       "ALLOW" "sops edit ~/.config/cloudflare/wrangler.enc.yaml"
run_test "B8 ps aux"          "ALLOW" "ps aux"
run_test "A6 head .enc direct"        "BLOCK" "head ~/secrets.enc.yaml"
run_test "A6 compound | head -10"     "ALLOW" "list.sh | head -10 && sops edit ~/wrangler.enc.yaml"
# ...
```

全 53 test pass (= 既 11 + 新 21 BLOCK + 21 legitimate ALLOW)。

### δ (補助): output sanitize hook はそのまま継続

PostToolUse の `credential_value_scrub.sh` は **input gate を通過してしまった漏れ** の damage control として継続稼働。

α + γ で漏れる経路を物理的に閉じ、δ で漏れた値を後追いで消す。

---

### 1. behavioral rule だけで credential は守れない

正のルール + 誘惑表 + pre-flight checklist 全部書いて、それでも 11 回漏れた。CLAUDE.md の冒頭 100 行に事故年表書いたのに、**新しい vector class が出るたびに同じ轍を踏む**。

AI が「やめて」を覚えるには **同じ間違いを 4-5 回**してくれる必要がある。それまでに鍵が漏れる。

### 2. input gate (PreToolUse) と output filter (PostToolUse) の二重防御

* 入り口で危険 pattern を block (= γ)
* 出口で漏れた値を sanitize (= δ)

両方必要。input gate だけでは shell mis-parse / encoding bug 経由の post-expansion leak を捕まえられない。output filter だけでは forward damage (= 第三者が見たかもしれない) を防げない。

### 3. shape rule (= repo-baked script idiom) は forget-proof

inline command を書く誘惑を **物理的に block** する。`sops exec-env file 'curl ...'` を hook で禁止すれば、必然的に `scripts/<service>/*.sh` の wrapper を書くことになる。

shell-quote は wrapper script の中で **1 度だけ正しく書けばいい**。それを repo に commit すれば、AI が次に呼ぶ時は **shell-quote 確定済**で動く。

### 4. 「N+1 を待たない」原則

5 回目で structural 議論を約束、実装サボったら **6/7/8/9/10/11 連発**した。

「次の同パターンが起きたら structural fix する」と書いたら、**その session 内に書く**。「忙しい時に書く」は forget mode に直行する。

### 5. 個別 test pass ≠ real-world robust

13 patterns を 1 commit で追加して、各 pattern の test fixture は通った。**でも real-environment で run したら 2 件 FP が即発覚**。

理由:

* **word boundary 不在**: `ps` が `sops` の suffix と substring match
* **greedy regex の compound 跨ぎ**: `head -10 && ... .enc.yaml` が単一 command と誤認

教訓:

* 新 pattern は **必ず word-boundary 前置** (`(^|[^a-zA-Z_/])<keyword>`)
* compound command の separator (`&|;<>`) を `[^...]+` で除外
* 個別 test fixture に加えて、**common shell idiom regression test** (= `sops edit`、`bash hook test`、`tail -n N file` 等) を fixture に組込必須

### 6. red team enumeration を design 段階に

「悪意ある AI agent が credential を盗む command を全列挙したら何が出るか」を **patterns 追加前**に書く。私は 27 attack surfaces を列挙した:

* env 値 expand 系: `env|grep` / `printenv` / `echo $VAR` / `printf $VAR` / `declare -p` / `set` (= 7 surfaces)
* env dump 系: `cat /proc/.*/environ` / `ps eww` / `docker inspect Env` (= 3)
* file 平文系: `cat .env` / `head/tail .enc.yaml` / `tail rclone.conf` (= 3)
* inline credential 系: `curl Bearer literal` / `rclone --s3-access-key-id` / `curl ?api_key=` (= 3)
* wrapper bypass 系: `sops exec-env + python|node|ruby|perl|bash|sh -c` / `eval` / output redirect (= 5)
* xtrace 系: `bash -x; printf $VAR` (= 1)
* redirect to file 系: `env > file` (= 1)
* side-channel: shell history / terminal scrollback / age key compromise / sops temp file / clipboard manager (= 4)

うち 17 surfaces を hook で close、4 surfaces を skip (= 過剰 FP)、6 surfaces を **OS-level / out-of-scope** として behavioral rule で対応。

### 7. AI 相手の信頼 boundary は「鍵」じゃなくて「入力経路」

これが本記事の core claim。

人間相手なら「信頼 boundary は鍵そのものを保護」(= safe / vault / KMS)。

AI 相手は違う。AI は鍵を「使う」だけじゃなくて、**stdout / stderr / file system / process arg / API response / digest** に **任意経路で値を流す**。だから boundary は鍵じゃなくて **入力経路** に引く。

```
鍵を保護
  ↓
鍵を AI に「触らせない」のは不可能
  ↓
鍵を「使う path」を repo-baked にする
  ↓
それ以外の path で AI が credential 触る command を書いた瞬間 hook で block
```
