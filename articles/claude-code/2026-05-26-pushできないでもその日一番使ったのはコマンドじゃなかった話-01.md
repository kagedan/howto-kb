## まず、何が起きたか

共通基盤のソースコードをリポジトリに登録する、ってタスクがあった。

手順書もある。ファイルも揃ってる。やることは「手順通りにコマンド打つだけ」のはずだった。

`git add` → OK。`git commit` → OK。`git push` → **ダメだった。**

返ってきたエラーがこれ。

```
A default branch does not yet exist for this project.
```

短い。シンプル。でも、そこから先が全然シンプルじゃなかった。

---

## 最初は「すぐ直るやろ」と思ってた

正直、最初はなめてた。

「デフォルトブランチの設定どこかでやればいいんでしょ」くらいの気持ちで、GitLabの画面をぽちぽちしてた。

でも設定画面が触れない。権限がない。

「あれ、そもそもこのリポジトリって誰が管理してるんだっけ？」ってなって、そこからが本番だった。

---

## コマンドで現状を整理した

感覚で動くのをやめて、まず「今何が起きてるか」を確認することにした。

やったのは3つで、リモートの詳細を `git remote show origin` で確認して、リモートに何があるかを `git ls-remote origin` で確認して、ブランチの状態を `git branch -a` で確認した。

一番重要だったのは `git ls-remote origin` の結果が**空だった**こと。リポジトリ自体は存在してるけど、中身がまっさら。初期化すらされてない状態だった。

そこでようやく仮説が立った。「デフォルトブランチが設定されてないか、ブランチ保護のルールが初回pushを弾いてるかのどっちかだ」と。

---

## ここでAIに聞いた

Claude Code と ChatGPT を使った。でも使い方が大事で、「答えを教えて」じゃなくて「可能性を列挙して」って聞いた。

具体的には「`pre-receive hook declined` って出たとき、GitLabでありうる原因のパターンを全部教えて」って投げた。

返ってきたのは、デフォルトブランチ未設定・ブランチ保護ルールで直接pushが禁止・ロールが Maintainer 未満で保護ブランチに触れない・CI/CDのhookがルールを実行してる、みたいな5パターンくらいの候補だった。

これが**めちゃくちゃ助かった**。

自分の経験が浅いと「知らない原因」がたくさんある。AIはその「自分が思いつかなかった可能性」を一瞬で並べてくれる。答えを出してくれるんじゃなくて、**調査の地図を渡してくれる**感じ。

で、その地図を持って一個ずつ確認していった。

---

## 問題はわかった。でも解決できるのは自分じゃなかった

調査を進めるうちに、はっきりしてきたことがあった。

**このリポジトリの設定を変えられるのは、自分じゃない。**

GitLabのリポジトリ設定って、Maintainer以上の権限がないと触れない。自分のロールは Developer だった。ブランチ保護の設定画面は見えるけど変えられない。

つまり状況はこうだった。原因はわかった。解決策もわかった。でも実行できる人が別にいた。

ここで「えっ、じゃあ自分の仕事終わり？」ってなりそうだけど、そうじゃなかった。

**「正しい人に、正しい情報を渡す」のも仕事だった。**

やったのはこれだけど、けっこう大事で、エラーの全文と実行したコマンドと結果をまとめて、「この設定をこう変えれば直るはず」って具体的に書いて、担当チームにエスカレーションした。

「よくわからないんですがpushできなくて困ってます」じゃなくて、「原因はおそらくこれで、変更が必要なのはこの設定です」って渡せたのは、調査したからこそだった。

---

## 待ってる時間も、仕事だった

エスカレーションしたあと、返答を待つ時間が発生した。

以前の自分なら「詰まってる……」って落ち込んでたと思う。でも実際は、この時間にできることがいっぱいあった。

調査メモをちゃんとしたドキュメントに整形したり、手順書に「事前に確認が必要な権限」の項目を追記する提案をまとめたり、別の積み残しを進めたり。

「待つ」って受け身に聞こえるけど、実際は**他の人の作業を前提に、自分の仕事を並列で回す**ってスキルだと気づいた。依存関係がある作業の前でフリーズしないこと、それも現場では普通に求められることだった。

---

## 一番意外だったこと

この日、技術的な調査より時間かかったのが「**誰が何をできるか**を整理すること」だった。

このリポジトリを管理してるのは誰か。連絡するチャンネルはどこか。変更してもらうには何の情報が必要か。どのくらいで対応してもらえるか。

これ、コマンドでは調べられないし、GitLabの設定画面にも書いてないし、AIでも答えられない。

**人と組織を理解することが、技術的な問題を解決するための一部になってた。**

エンジニアの仕事ってコードを書くことだと思ってたけど、現場って全然それだけじゃないんだなと、この日わりとはっきり感じた。

---

## 学んだことをざっくりまとめると

技術的なところで言うと、`pre-receive hook declined` はGitLabのサーバー側のルール起因のエラーで、空リポジトリへの初回pushはデフォルトブランチ設定やブランチ保護の影響をもろに受ける。あと `git ls-remote origin` が空なら「リポジトリはあるけど未初期化」って状態で、ロールと権限は最初に確認しておくと調査時間がめちゃくちゃ短くなる。

現場的なところで言うと、「問題を解決する ≠ 自分で実装する」で、解決できる人を特定して正確な情報を渡すのも立派な仕事。他チーム依存は例外じゃなくて普通で、調査結果をドキュメントに残すのは次に同じ問題にハマる人の時間を節約できる、地味に価値のある仕事だった。

AIは「答えを出す道具」じゃなくて「自分が知らない可能性を高速で列挙してくれるやつ」として使うと強い。

---

## 若いころの自分に言えるとしたら

> **コードが書けなくても、調査して、整理して、渡せれば、それは仕事になるよ。**

実装力はもちろん大事。でも現場では、それ以外のことに使う時間が思ってたより全然多い。

エラーに向き合う力、ログを読む力、現状を言語化する力、誰に何を頼むかを判断する力、待ちながら別のことを進める力。

これ全部、コードを書く力と同じくらい鍛える価値がある。

`git push` できない日があっていい。そこで何をするかが、じわじわ効いてくる。
