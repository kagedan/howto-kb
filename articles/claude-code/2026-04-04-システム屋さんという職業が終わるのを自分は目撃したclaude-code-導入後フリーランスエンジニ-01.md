## はじめに

こんにちは、moumu です。

この記事、読まなくても今日は困らない。でも 3 年後、「あのとき読んでおけばよかった」と思う可能性がある。具体的に言うと、月単価が下がり始めたとき、案件の声がかからなくなり始めたとき、初めてこの記事の意味がわかるかもしれない。

自分はフリーランスエンジニア、30 歳。C# .NET をメインに、業務システムの開発を生業にしてきた。「コードが書ける」ことが自分の飯の種だと、ずっと信じて疑わなかった。

それが 3 ヶ月前、完全に揺らいだ。

---

## 「自分いらなくなるかも」と感じた瞬間

3 ヶ月前、仕事で Claude Code が導入された。

最初は「まあ AI ツールね、Copilot みたいなもんでしょ」と高をくくっていた。ところが実際に触ってみて、背筋が冷えた。

Claude Code は、コードの補完をするツールじゃない。指示を出すと、自分でファイルを開いて、コードを書いて、エラーを確認して、修正して、動くところまで持っていく。**エージェントとして動く。**

試しに「シンプルな在庫管理システムを作ってくれ」と投げた。30 分後、CRUD 全部入りの Web アプリが動いていた。自分なら半日かかる作業だった。

その瞬間、頭の中に出てきたのは「すごい」じゃなくて、「自分、いらなくなるかも」だった。

これは感情論じゃない。自分が 5 年目まで磨いてきた「コードが書ける」というアイデンティティが、30 分で揺らいだ。現実として、**「要件を聞いてシステムを作る」という仕事の工程のほとんどが、今この瞬間に代替可能になっている**のを目の前で見た。

では、この恐怖とどう向き合ったか。

---

## AI エージェントの「得意と苦手」を 3 ヶ月かけて整理した

怖かったから、逃げずに向き合うことにした。

Claude Code、LangChain、MCP プロトコル、エージェント設計のパターン。仕事の合間に詰め込んだ。技術書より一次情報を当たった。実際に手を動かして、何十個もプロトタイプを作った。

その過程で見えてきたことがある。

**AI が本当に得意なことと、まだ苦手なことが、想像とまったく違った。**

得意なこと：

* 仕様が明確な実装タスク（CRUD、API 設計、テストコード）
* リファクタリング（既存コードの構造改善）
* ドキュメント生成
* エラーの原因特定と修正

苦手なこと：

* 曖昧な要件を人間と合意形成しながら固めていく作業
* 「なぜこの機能が必要なのか」をステークホルダーから引き出すヒアリング
* ビジネスの文脈を読んで優先順位を判断すること
* 人が感情的に動くポイントを掴んで、提案を通すこと

気づいたのは、**コードを書く前段階にある仕事のほとんどが、AI にはまだできない**ということだ。そしてそれは「まだ」ではなく、本質的に人間が担う領域かもしれないと感じ始めた。

では、「コードを書く」という行為に価値を置いていた自分は、何を変えるべきなのか。

---

## 「AI に仕事を奪われる」という話をひっくり返す

世間では「AI にエンジニアの仕事が奪われる」という話が溢れている。その危機感は正しい。でも、**奪われる仕事の中身をちゃんと見ていない人が多い**と思う。

奪われるのは「コードを書くという行為」だ。

「エンジニアという職業」じゃない。

もっと正確に言うと、奪われるのは「決まった仕様を、コードという形に変換する作業」だ。それは確かに今後 AI が担っていく。もう担いつつある。

でも、じゃあ誰が仕様を決めるのか。誰が「この機能、本当にユーザーが求めてるの？」と問い直すのか。誰がクライアントの本音を引き出して、プロジェクトをゴールまで導くのか。

それは人間だ。今も、これからも。

「エンジニアが消える」という話は半分正しくて半分間違っている。

> 消えるのは **"実装マシン" としてのエンジニア**だ。残るのは **"設計者・翻訳者・交渉人" としてのエンジニア**だ。

じゃあ、その「残るエンジニア」に共通するスキルは何か。

---

## では何が残るのか――「人を動かすスキル」の正体

3 ヶ月勉強して、自分が最終的に行き着いたのは技術の話じゃなかった。

**「人を動かすスキルだけが、不変だ」**

これは抽象論に聞こえるかもしれないが、自分が言っているのは具体的なことだ。

たとえば、クライアントが「売上管理システムを作ってほしい」と言ってきたとする。AI はこの一文からシステムを作れる。でも自分がまず確認するのは、「なぜ今それが必要なのか」「現状のどこに痛みがあるのか」「誰が一番困っているのか」だ。

その問いかけで、「実は売上管理じゃなくて、営業の行動が可視化されていないことが本当の問題だ」と気づくことがある。本質が変わると、作るものが変わる。作るものが変わると、クライアントが本当に助かる。

これは AI には今のところできない。なぜなら、クライアント自身が「本当の問題」を言語化できていないからだ。引き出すのは、人間の問いかけと、場の空気を読む力だ。

行動経済学の観点で言えば、人間の意思決定は論理より感情と文脈で動く。「何を作るか」より「なぜ作るか」を納得させること。「機能の説明」より「導入後の未来のイメージ」を描かせること。これができるエンジニアとできないエンジニアの差は、AI が普及するほど大きくなる。

---

## じゃあ自分はどう動いたか

AI 勉強と並行して、自分が変えたのは「仕事の受け方」だった。

以前は「この機能を実装してほしい」という依頼をそのまま受けていた。今は必ず一度立ち止まる。「その機能は、誰のどんな問題を解決しますか？」と聞く。

たった一言を変えただけで、クライアントとの会話の深さが変わった。具体的な話をすると、ある製造業のクライアントから「受発注の管理ツールが欲しい」という依頼を受けたとき、いつもの問いかけをした。話を掘っていくと、本当の問題は「担当者が退職したときに情報が引き継げないこと」だとわかった。作るものが変わり、3 ヶ月後に「他のベンダーに頼んでいたら全然違うものができていた」と言われた。単価も前案件比で 1.3 倍になった。

提案の精度が上がり、「この人に頼んでよかった」と言われる頻度が上がった。ここ半年で新規案件の問い合わせは倍近くになっている。一方で、以前の自分と同じように「言われた通りに作ります」を続けているフリーランスは、声がかかりにくくなったという話を、フリーランスコミュニティで何度も聞くようになった。半年前は「気のせいかも」と思っていたが、今は疑いようがない。

AI ツールの使い方も変わった。自分が考えるのは「何を作るか・なぜ作るか」で、「どう作るか」は Claude Code に任せるようになった。そうすると、一人でできる仕事の量が 3 倍以上になった。コードを書く時間が減った分、クライアントと話す時間を増やした。

これが今のところ自分の答えだ。

---

## まとめ：3 年後に後悔しないために、今日やること

「AI に仕事を奪われる」は本当だ。でも奪われた後に何が残るかを考えていないと、ただ怖いまま時間が過ぎる。月単価が下がり始めてから気づいても、取り返すのに倍の時間がかかる。案件の声がかからなくなってから動いても、ポジションはすでに埋まっている。

コードを書くという行為に自分の価値を置いていた人は、今すぐその前提を疑ったほうがいい。価値を置くべきは、\*\*「なぜを引き出す力」「人を動かす設計力」「感情に働きかける提案力」\*\*だ。

具体的に今日やれることを一つだけ挙げるなら、次の仕事の依頼を受けたときに「その機能、なぜ必要なんですか？」と一回だけ聞いてみることだ。

その問いかけの習慣が、3 年後の自分を全然違う場所に連れていく。

---

自分はまだ答えを探している途中だ。でも「人を動かすスキルが不変だ」という確信だけは、3 ヶ月で固まった。

> コードは道具だ。道具を使う人間の価値は、道具が賢くなるほど、問われる。
>
> あなたは今、どちらのエンジニアになろうとしているか。
