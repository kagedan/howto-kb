AIにどこまで任せるか。**Claude Code** チーム公式の「4つのループ」分類を、無人ループ運用者の目線で読み解きます。

---

## はじめに：「任せ方」の地図がなかった

前回まで[コンテキストエンジニアリング](https://note.com/ryotashacho/n/nf39da861805a)（読ませる情報の設計）、[ハーネスエンジニアリング](https://note.com/ryotashacho/n/n8ca5f92b74cf)（エージェントが動く足場の設計）を扱ってきました。ループはその最上段にあたります。

ただ、いざやろうとすると最初に詰まるのはここです。「で、どこから手放せばいいの？」

全部任せるのは怖い。かといって毎ターン自分で確認していたら、ボトルネックは永遠に自分のまま。この「どこまで委譲するか」の物差しは、これまで各自の感覚頼みでした。

そこに2026年6月30日、Claude Code チームが公式ブログに「Getting started with loops」という記事を出しました。ループの使い分けを4タイプに整理した、いわば公式の地図です。今回はこの記事を、運用者の目線で読み解きます。

## 結論：ループは「何を手放すか」で4つに分かれる

原文はまず、ループをこう定義します。

> ループとは、停止条件（stopping condition）が満たされるまで、作業のサイクルを繰り返すエージェントである。

<https://claude.com/blog/getting-started-with-loops>

そのうえで4タイプに分類します。読んでいて膝を打ったのは、これが機能の分類ではなく「人間が何を手放すか」の階段になっている点です。

![](https://assets.st-note.com/img/1784232083-hMVD1O9cWmRziHF8Nbf5IUQC.png?width=1200)

下に行くほど人間の関与が減り、自律性が上がります。順に見ていきます。

## それぞれ詳しく

### 1. ターンベース：まず「検証」を手放す

もっとも基本の形です。人がプロンプトを送り、Claude が読んで、編集して、テストして、報告する。ここでのボトルネックは、毎ターンの結果確認を人間がやっていることです。

原文の処方箋は、検証手順を SKILL.md に書き下ろすこと。例として挙がる verify-frontend-change というスキルは「開発サーバーを起動し、ブラウザで実際に操作し、コンソールのエラーがゼロであることを確かめるまで完了と報告しない」という内容です。

面白いのは「失敗したらステップ1からやり直す」という一文まで書いてあること。検証スキルはチェックリストではなく、ループの再入口として書く。ここが肝だと読みました。

### 2. ゴールベース：「いつ終わりか」を手放す

次に手放すのは停止判断です。/goal に検証可能な成功基準を渡すと、達成するまで「試行 → 評価 → 再試行」を繰り返します。

```
/goal ホームページの Lighthouse スコアを 90 以上にする。5回試行したら停止。
```

裏側では、エバリュエーターモデルが「本当に終わっていいか」を検査し、未達なら作業へ差し戻します。コツは定量的な基準とターン上限をセットで書くこと。「いい感じにして」は動きません。

### 3. タイムベース：「いつ始めるか」を手放す

3つめは開始判断の委譲です。指定した間隔で自動的に起動し、キャンセルするか作業が終わるまで繰り返します。

```
/loop 5m PRをチェックし、レビューコメントに対応し、失敗しているCIを修正する
```

ローカルで回すなら /loop、マシンを閉じていても動かしたいならクラウド実行の /schedule。注意点として原文が挙げるのは実行間隔です。監視対象が実際に変化する頻度に合わせないと、トークンをただ燃やすだけのポーリングになります。

### 4. プロアクティブ：プロンプトそのものを手放す

最終形は、人間がリアルタイムにいない状態で、イベントやスケジュールを起点に働くループです。新しい機能ではなく、ここまでの3タイプの組み合わせで作ります。

原文の例は毎時のバグ報告トリアージでした。スケジュールで起き、ゴール達成まで止まらず、修正時は3つの解決策を並列に探索して、別のエージェントに敵対的レビューをさせる構成です。人の目が消える分を、別の機械の目で補う。私はここが本記事の白眉だと感じました。

## 回し続けるための品質とコストの話

長く回すほど効いてくるのが、品質とコストの管理です。品質面で原文が挙げる戦略は4つあります。

* コードベース自体を綺麗に保つ（Claude は既存パターンに従うため、汚れは再生産される）
* 品質基準そのものを SKILL.md に書き下ろす
* フレームワークのドキュメントへ到達しやすくしておく
* レビューは実装した本人ではなく、別エージェントに任せる

コスト面の要点は3つ。「モデル選択が最大のレバー」「大規模実行の前に小規模で試す」「決定的な作業は LLM に考えさせずスクリプトにする」。使用状況は /usage や /workflows で内訳を確認できます。

正直、この章がいちばん実運用者向けです。私の経験でも、ループの費用対効果は「推論を使うべき場所の設計」でほぼ決まります。

## さいごに：どの一段から始めるか

原文の締めは「自分がボトルネックになっている仕事から始めよ」というものです。診断の問いは4つ。

1. 自分がボトルネックになっているタスクはどれか
2. その検証手順を文章に書き下ろせるか → 書けるなら検証スキル
3. ゴールを検証可能な形で定義できるか → 定義できるなら /goal
4. その仕事は定期的に発生するか → するなら /loop・/schedule

読み終えての私の結論は、4タイプに共通するコアスキルは「停止条件を言語化する力」だ、ということです。プロンプトの上手さではなく、受け入れ条件を書く力。要件定義という古典的な技術が、そのままエージェント運用の中心に戻ってきています。

いきなり最終形を組む必要はありません。まず検証を手放す。その検証が信頼できるから、停止判断を手放せる。この階段は、エージェントへの信頼を一段ずつ積む階段でもあります。
