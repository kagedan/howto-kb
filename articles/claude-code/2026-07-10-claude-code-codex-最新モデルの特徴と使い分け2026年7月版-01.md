## はじめに

AnthropicとOpenAIの双方でモデルラインナップが一新されました。

Claude側はOpusの上位に**Fable 5**が加わり、日常の主力も**Sonnet 5**へ世代交代。

OpenAI側は**GPT-5.6**が「Sol / Terra / Luna」という新しい3ティア構成で登場しました。

ラインナップが変わりましたので、改めて見直しました。

## Claudeの新ラインナップ

Claude Codeで選択できる主要モデルは4つのティアに整理できます。

| モデル | 位置づけ |
| --- | --- |
| Claude Fable 5 | 最上位（Opusのさらに上のティア） |
| Claude Opus 4.8 | 高難度タスクの主戦力 |
| Claude Sonnet 5 | バランス型・日常の主力 |
| Claude Haiku 4.5 | 最速・最安の軽量ワーカー |

従来の「Opus > Sonnet > Haiku」という3段構成の**上にもう1段（Fable 5）が追加されました**。

![Claudeの新ラインナップ2026 — Fable 5は魔法使い、Opus 4.8は学者、Sonnet 5は専門家、Haiku 4.5はサラリーマンとして擬人化した4モデルの整列図](https://static.zenn.studio/user-upload/deployed-images/8b130eff97219b7a67afc9d3.png?sha=2db6dfa6863362bf1353545557331a6e60e68e71)

### 各モデルのユースケース

**Claude Fable 5 — 「ここぞ」で投入する最上位**

* 数日単位の自律タスク（計画→実行→自己検証を回し続ける長期実行）
* 最難関の設計・調査など、他のモデルでは歯が立たない問題

詳細: [公式発表](https://www.anthropic.com/news/claude-fable-5-mythos-5)

**Claude Opus 4.8 — 高難度タスクの主戦力**

* 高難度の設計・アーキテクチャ判断
* 複雑なデバッグ、コードレビュー（実バグの発見に強い）
* 夜間・長時間の自律実行。fast mode対応

詳細: [Models overview | Claude Platform Docs](https://platform.claude.com/docs/en/about-claude/models/overview)

**Claude Sonnet 5 — 日常実装のデフォルト**

* 日常の実装・修正・テスト全般。Claude Codeのデフォルトモデルで、大多数のコーディング作業はこれで足りる
* コーディング・エージェント用途で**従来のOpus級に迫る品質**

詳細: [公式発表](https://www.anthropic.com/news/claude-sonnet-5)

**Claude Haiku 4.5 — 軽量・高速の使い捨てワーカー**

* 分類、ログの要約、定型的なコード変換などの軽量タスク
* 大量に並列実行するサブエージェントのワーカー
* レイテンシが重要な対話的ユースケース

詳細: [Models overview | Claude Platform Docs](https://platform.claude.com/docs/en/about-claude/models/overview)

## 実践的な使い分け

### 大原則 — 高価なモデルは「判断の瞬間」だけ（Claude公式）

Claude公式（[@ClaudeDevs](https://x.com/ClaudeDevs/status/2074606058128224365)）が示した大原則を押さえておきます。

公式ベンチマークでの性能維持率・コスト削減率の詳細と、この体制（Advisor=相談役 / Orchestrator=指揮役の2パターン）を実際のPR作業で回してみた実測レポートは、別記事にまとめています。  
[Fable 5は"常時フル稼働"させない — Claude公式のadvisor/orchestratorパターンを開発体制に取り入れる](https://zenn.dev/nenene01/articles/fable5-advisor-orchestrator-orca)

以下の使い分けは、すべてこの大原則の各論です。

### 基本方針

| 作業の種類 | Claude（Claude Code） | OpenAI（Codex） | 理由 |
| --- | --- | --- | --- |
| 日常の実装・修正・テスト | **Sonnet 5** | **Terra** | 速い・安い・十分に賢い。デフォルトのまま使う |
| 高難度の設計・アーキテクチャ判断 | **Opus 4.8 / Fable 5** | **Sol** | 深い推論と長期の一貫性が必要 |
| 複雑なデバッグ・コードレビュー | **Opus 4.8** | **Sol** | 上位ティアの読解・推論の深さが効く |
| 数日がかりの自律タスク・最難関問題 | **Fable 5** | **Sol** | 各社最上位の長時間実行・自己検証能力に任せる |
| 軽量・定型・大量並列タスク | **Haiku 4.5** | **Luna** | コストとスピードで圧倒的 |

### 手元の3つのつまみ — /model・effort・/fast

日々の運用で触るのは、実質この3つです。

| つまみ | 何を変えるか | 使いどころ |
| --- | --- | --- |
| `/model` | モデルそのもの | Sonnet 5（Codex側はTerra）を既定に、難所だけ上位モデルへ。Claude Code・Codex CLIとも同名コマンド |
| `effort` | 同じモデルの「考える深さ」（low〜maxの5段階） | モデルを替えるほどではない調整に。上げるほど賢くなるがトークン消費も増える。逆にFable 5をlow〜mediumで回してレート消費を抑える運用も有効 |
| `/fast` | 同じOpusの「出力速度」（約2.5倍） | 品質そのままに待ち時間を短縮。ただし価格は上がる |

原則は「**Sonnet 5（Codex側はTerra）を既定とし、必要なときだけ上位モデル・高effortに切り替える**」です。OpusやFable、Solといった上位モデルは消費クォータが大きいため、常時オンにするのではなく、問題の難度に応じて投入します。

![手元の3つのつまみ — /model（モデルそのもの）・effort（考える深さ）・/fast（出力速度）を整理したインフォグラフィック](https://static.zenn.studio/user-upload/deployed-images/350f9077151cdbb422b38919.png?sha=7516c207045c8f5d410039aa0f59888afeeb729c)

### サブエージェントへのmodel指定 — オーケストレーション時の「編成」を組む

Claude Codeのみでオーケストレーションを組むならサブエージェント（Agent tool）があります。  
**エージェントごとにモデルを指定**でき、やり方が2つ存在します。

Codexと混ぜてマルチエージェントでの対応ならOrcaのようなオーケストレーション機能を使います。 この使い分けの詳細も[前掲の記事](https://zenn.dev/nenene01/articles/fable5-advisor-orchestrator-orca)を参考にしてください。

1. **自然言語で指示（アドホック）** — 「Sonnetのワーカー3体で並列実装、探索はHaikuで」のように依頼すれば、メインセッションがサブエージェント起動時にモデルを割り当てます。定義ファイルは不要で、agent teamsをその場の命令で立ち上げてくれます。
2. **ファイルで設定** — `.claude/agents/*.md`で役割ごとのデフォルトを定義しておく。毎回明示しなくても、そのエージェントを使うたびにモデル・役割プロンプト・ツール制限が自動適用されます。

優先順位は「**呼び出し時の明示指定 > 定義ファイルの`model` > 親セッションのモデルを継承**」とのこと。定義ファイルはマストではなく、明示しなかった際のフォールバック（既定値）として機能します。

定義ファイル方式の例:

.claude/agents/explorer.md

```
---
name: explorer
description: コードベースの広域探索を担当する読み取り専用エージェント
model: haiku
---

リポジトリ全体を横断的に検索し、関連ファイルの場所と概要だけを報告してください。
```

.claude/agents/architect.md

```
---
name: architect
description: 設計判断とレビューを担当する上級エージェント
model: opus
---

実装の詳細ではなく、アーキテクチャ上のトレードオフと影響範囲を分析してください。
```

## 「モデル選択とはチーム編成である」 — agentic engineeringの視点

「モデル選択とはチーム編成である」という見方に行き着きます。アンドレイ・カーパシーがvibe codingの次の段階として提唱する**agentic engineering**。  
[Sequoia Ascent 2026の講演サマリー](https://karpathy.bearblog.dev/sequoia-ascent-2026/)から3行、引用します。

> The unit of programming changed from typing lines of code to delegating larger "macro actions".  
> The programmer is increasingly not just a code writer, but an orchestrator of agents.  
> I have never felt more behind as a programmer.

（要旨）プログラミングの単位は「行を書く」から「マクロアクションを委譲する」へ変わり、プログラマーはエージェント群のオーケストレーターになりつつある。カーパシー氏は「これほど取り残されていると感じたことはない」と漏らすほどの変化です。  
また、Anthropicは、**2026年5月時点で自社の製品コードベースにマージされた新規コードの約80%がClaudeによって書かれた**と[公表しています](https://venturebeat.com/technology/anthropic-says-80-of-its-new-production-code-is-now-authored-by-claude-how-your-enterprise-can-keep-up)。

ほかにも「ハーネスエンジニアリング」「ループエンジニアリング」という言葉が広がりつつあります。

* **ハーネスエンジニアリング** — プロンプト（AIにどう言うか）→コンテキスト（AIに何を見せるか）の先にある第3段階で、**AIが失敗しにくい開発環境そのもの**（テスト・lint・型チェック、自動修正ループ、ガードレール、マルチエージェント連携）を設計する技術。3つは置き換えではなく入れ子の関係です（参考: [3段階の進化の解説](https://zenn.dev/emuni/articles/7c1493a52a3569)）
* **ループエンジニアリング** — Addy Osmani氏が提唱。人間がプロンプトを打ち続ける代わりに、**ゴールを定義すれば完了条件を満たすまでAIが試行を繰り返す「ループ」そのもの**を設計する考え方（参考: [ループエンジニアリング初級編｜AGIラボ](https://chatgpt-lab.com/n/n9f7c7f44f9e9)）

切り口は違いますが、肝は共通だと考えています。ハーネス（足場・検証環境）を作るにも、ループ（ゴール→試行→検証の反復）を回すにも、前提になるのは**エージェントを適切に配置し、使いこなすこと**です。

これまでの仕事の重心は、コードを書くことそのものでありました。  
ですが、機能の最適化や品質の作り込みへと移ってきました。  
その次の段階では、最適化・品質担保を**遂行するエージェントの配置とモデルの選定**を設計する作業へ、さらに、エージェントに委ねるための**ゴール・制約・完了条件・検証方法を明示する**役割へと変わっていくのだろうと理解しています。

つまり、モデル選択の意味が変わりつつあります。  
委譲の単位が「コードの補完」だった時代は、モデルの賢さで選べばよかったのです。  
「サブシステムのリファクタ」や「数日がかりの調査と実装」などと大型化した今では、**どのマクロアクションをどの知能に任せるかは、人間のチームで言う人員配置そのもの**になります。  
リーダーの経験上、「この機能は難易度が高いのでAさん、タスク消化が早いBさんはこの作業」と考える場面がありました。  
今後はエージェントの選択の際に同様の判断をすることとなります。

* ジュニアに任せてよい定型作業を、シニアの時間で消化するのは無駄
* 一方、アーキテクチャを決める場面にジュニアだけを置くと手戻りが膨らむ
* リードは、メンバーへの指示の明確さ——**ゴール、制約、完了条件、検証方法**——に集中する

この先はプロンプト作成が上手・下手だけではなく、**モデルという名の人材ポートフォリオを組む力**が必要となります。  
その編成を組む第一歩が、本記事で整理してきた**各社モデルの特性と役割を把握すること**だと考えています。
