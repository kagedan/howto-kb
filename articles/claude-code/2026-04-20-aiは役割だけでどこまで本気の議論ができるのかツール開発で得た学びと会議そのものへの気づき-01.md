## TL;DR

* AI壁打ちの偏りに課題を感じ、複数AIが多角的に議論するプロセスを可視化できるWebUIツールを探して、見つからなかったので自作しました
* Next.js + nginx + MySQL + Docker 構成に、CLAUDE.md（約300行の指示書）で動くAIファシリテーターを乗せ、MCP サーバー経由でベクトル検索・議事録・コスト管理まで一体化しています
* 自前SSEオーケストレーター・Swarms・Agent Teams を経て、サブエージェント方式（`claude -p --agents`）に落ち着くまでの約1ヶ月の記録です
* 開発の学び：ベクトル検索の導入でコストが35%悪化→再設計で最大49.5%削減できた失敗→学びのプロセスを、実測値つきで詳述します
* 会議そのものへの気づき：作って初めて不要と分かった機能から見えた、AIファシリテーターが機能すると会議に何が起きるかを考察します

## この記事の位置づけ

開発したツールは、私のローカル環境で動かす個人用の会議ルームです。サブエージェント方式は `claude -p --agents` を使っており、これは Claude サブスクリプションに紐づく開発者向けの機構です。商用サービスとして提供したり、多数のユーザに開放する用途で使うことは、Anthropic の利用規約上 NG です（参考: [Anthropic Usage Policies](https://www.anthropic.com/legal/usage-policy) ／ 最終確認は各自でお願いします）。

あくまで私の思考・議論プロセスを可視化し、途中で口出ししながら結論を磨くための個人用ツールとして読んでください。同じ目的を商用サービスで提供するなら、Anthropic API 直叩きでサブエージェント機構を自前実装するコースに戻る必要があります（後半の置き換えの厳しさでも触れます）。

## 会議への課題意識と、プロセスを見たかった欲求

### 見つからなかったツール

私は[SUBLINE](https://www.subline.jp/)というプロダクトの責任者とシステム部の責任者を兼任しているので、他プロダクトの開発と行き来しながら、次の施策を考えたり、数字のちょっとした変化からお客さまの行動の変化を読み解こうとしたり、傾向分析のような仕事もしています。時間を行き来しているぶん、会議まわりの時間効率にはずっと課題を感じていました。

社内勉強会で発表する順番が決まった2月末、そういう非効率な業務を効率化してくれるようなサービスが世の中にないか調べて、なければ作ってみようと思ったのが動機です。

議事録の自動作成や会議後の要約生成といった会議の結果を効率化するサービスはたくさんあります。でも、会議そのものの質を上げるためのツール、特に事前に多角的な視点でシミュレーションをしてくれるルームのようなものは、探しあてることができませんでした。

### 会議の質は、ファシリテーションの難しさに起因している

少し遠回りな話をしますが、会議を上手に開催できる人は少ないと思います。これは批判ではなく、ファシリテーションというスキルが、学ぶ機会が少ない割に頻度も多く難易度が高いと感じてます。

目的の不明確な会議、参加者のパワーバランスによる発言の偏り、結論が出ないまま時間切れ、といった問題の多くは、ファシリテーターが適切に機能していないことに起因しています。ただ、どんなに有能なファシリテーターであっても、無策で参加する参加者から価値のある発言を引き出すのは至難の業で、会議の質はどこまで前段階で参加者に本気の準備をしてもらえるかにかかっています。

もしも、複数のAIエージェントが事前に多角的な観点でシミュレーションをしてくれるルームがあり、事前に深く議論した内容を会議にもちよることができれば、もう少し生産性の高い議論を増やせるのではないかと考えました。

### コミュニケーションと意思決定を分けるべき

もう一つ、会議に感じていた構造的な問題があります。多くの会議は、コミュニケーション目的と意思決定目的が混在しています。近況報告→雑談→本題→時間切れ、という流れが常態化しやすく、中途半端にコミュニケーション（無駄話ではないかもしれませんが、本題とは別のもの）が絡むから、決めるべきことが決まらない。

AIエージェントとの事前シミュレーションで会議の意思決定の密度を上げられれば、従来より短い時間で議論を終えられるので、浮いた時間をまるまる世間話専用のコミュニケーションの場として開催する、ということもできるようになれるかもしれません。

### AIの思考プロセスの可視化

ただAIに相談するにしても、結論だけ持ち寄られても、え、なんで？ってなるので、その結論に至るプロセスのほうをより重視したいと考えてます。

会議の方向性に途中で口出しできる仕組みにしたかった。これが、今回開発したツールの各機能の採用動機を貫く軸になっています。

## どんなツールを作ったか

### 構成の概要

図中で「Agent tool で呼出し」と書いているところの `Agent tool` は Claude Code が提供しているサブエージェント呼出し用の組み込みツールで、親の Claude プロセスが子エージェントを起動・メッセージ授受する仕組みです。Anthropic 標準 API にはなく、Claude Code 固有の機能です。

`Agent tool` と `--agent` / `--agents` の CLI 引数はよく混同されますが、役割が違います。

| 用語 | 何か | いつ使う |
| --- | --- | --- |
| **`Agent tool`** | Claude Code の組み込みツール（`Read` / `Edit` / `Bash` と同じ層にある） | Claude が会話中にサブエージェントを起動したいと判断したとき、モデル自身が `Agent(subagent_type: "math", prompt: "…")` のように呼び出す |
| **`--agent <name>`** | **起動時 CLI 引数**。セッション全体でどのエージェント定義を active にするか指定 | `claude --agent reviewer` のように、プロセスの役割を指定したいとき |
| **`--agents <json>`** | **起動時 CLI 引数**。カスタムエージェント定義を JSON 文字列で直接渡す | `.claude/agents/*.md` を作らず、ワンショットで定義したいとき |

`--agent` / `--agents` は `Agent tool` が参照できる候補リストを設定する手段、`Agent tool` はその候補を Claude が実際に呼び出すランタイム機構、という関係です。

ストレージは適材適所で三層に分けています:

* ① サーバー MySQL：会議をまたいだ正データ（メタ・全発言・メモ・処理ログ・データキャッシュ）。一覧表示や履歴参照はここから。AI推論中の検索対象にはなりません（AI は MCP で公開したツールの範囲しか触れない設計）
* ② per-meeting SQLite + sqlite-vec：会議ごとに作って捨てられる軽量DB。MySQL にベクトル拡張が手近に無かったので、ベクトル検索だけはこちらに持たせました。AI は MCP 経由でここに読み書きします
* ③ クライアント側 sql.js + IndexedDB：ツール呼出しや stdout の大量ログ。サーバーDBを肥大化させずブラウザ内で完結させています

補足として、過去会議の引き継ぎは MySQL を直接引くのではなく、ファイルベース＋新しい SessionDB への ingest で行っています:

1. 前回会議の `minutes.md` や `notes/` をファイルで保全
2. 新会議の `data/prev/` に引き継ぎたいファイルを配置
3. 会議開始時に Next.js が自動で chunk 化＋embedding して新しい会議の SessionDB に投入
4. AI は会議内で `search_research` を呼べば前回の関連部分をベクトル検索できる

この設計だと、関係ない過去会議が紛れ込まない／会議ごとに独立してベクトル検索が高速／MySQL の古い発言を毎回読み込む必要がない、という副次メリットも得られます。AI が過去を参照できる範囲を、人間が prev/ に何を置いたかで明示的にコントロールできる構造です。

全体は Docker 上で動かしています。

### 各コンポーネントの役割

Next.js / CLAUDE.md / MCP / 三層ストレージの内訳を開く

主要コンポーネントをまとめるとこうなります。

| コンポーネント | 役割 | 実装メモ |
| --- | --- | --- |
| **Next.js (App Router)** | Claude CLI 起動／MCP ディスパッチ／MySQL アクセス／SSE ストリーミング | サーバー側の統合ハブ |
| **React クライアントUI** | スレッド表示、議事録リアルタイム更新、コスト可視化、各タブ | クライアント側ログDB（sql.js + IndexedDB、3秒デバウンス）でログ表示を完結 |
| **CLAUDE.md（約300行）** | AIファシリテーターの脳として、進行ルール・フェーズ処理・パネリストへの指示方法を自然言語で記述 | コードではなく指示書でロジックを管理 |
| **自作MCPサーバー `meeting-db`** | 発言ログ・ナレッジベース・Webリサーチ結果をリレーショナル+ベクトルで永続化 | Anthropic 公式 SDK を使い、`better-sqlite3` + `sqlite-vec` + Python 埋込スクリプトを薄くつなぐ |
| **サーバー側 MySQL** | 会議をまたいだ正データ（メタ・全発言・メモ・処理ログ・データキャッシュ） | 一覧表示や履歴参照のソース |
| **per-meeting SQLite + sqlite-vec** | 会議ごとに使い捨ての軽量DB。会議内のベクトル検索（事前仕込み／save\_research／過去発言検索）を担当 | `multilingual-e5-small`（384次元） |

**ストレージを三層に分けた理由**

MySQL には sqlite-vec のような軽量ベクトル拡張が手近に無く、別ミドルウェア（pgvector / 外部 Vector DB）を立てるのは構成として重い。一方 SQLite には sqlite-vec があって、会議ごとに使い捨ての DB ファイルで運用できます。なので、会議をまたぐ正データは MySQL、会議内ベクトル検索は SQLite、大量のリアルタイムログはブラウザ側、という適材適所の三層になりました。

**プロセスを可視化する裏側（ログタブの中身）**

会議中、ファシリテーターやパネリストは各フェーズで大量のツールを呼び出しています。典型的な流れを図にするとこうなります。

*1フェーズあたりに裏で走るツール呼出しの典型パターン。`Read` / `Edit` / `save_message` / `task_started` / `task_notification` などが時刻つきでログタブに並ぶので、AIが今何をしているかを追える状態になっています*

## 約1ヶ月の技術判断の歩み

### まず自前のSSEオーケストレーターで動かす

最初の動くバージョンは自前のSSEオーケストレーター（Node.js 800行前後）でした。以下のことができていました。

* ファシリテーター役のAIが各パネリストに順番に発言を依頼する
* パネリストの発言がフロントエンドにリアルタイムでストリーミングされる
* 発言ターンの管理（誰が今話しているかの状態管理）
* 簡単なスレッド構造

設計としては、Node.jsのサーバーがオーケストレーターとして各AIエージェントのAPI呼出しを管理し、その結果をSSEでフロントエンドに流すという構造でした。動作はしていましたし、基本的な会議の流れを実現できていました。

ただ、一つの問題がありました。AIらしさが足りない。ファシリテーターが、次はAさんに話してもらいます→Aさん話す→次はBさん、という機械的な進行しかできず、議論の内容に応じた柔軟な対応が難しかった。コードでロジックを書くほど、想定外の状況に対応できなくなっていきました。

### 周辺機能の作り込みと、ロジック複雑化の壁

ここから周辺機能をかなり手広く追加していきました。カスタムフレームワーク作成、マインドマップ、データソース連携、ナレッジベース、チャットUIのリサイズ、議事録駆動設計など。Git のコミットも一日で数十件まとめて入るような時期で、このフェーズは触っていて一番楽しかった気がします。

同時に、動いている実装に、より複雑な会議ロジックを加えようともしていました。議題タイプに応じた進行パターンの切り替え、深掘りフェーズの動的制御、Devil's Advocate（反論役）の自動挿入あたりを実装しようとしたとき、ロジックの複雑化に直面します。

条件分岐が増えるにつれて、コードは読みにくくなり、想定外の組み合わせでバグが発生するようになりました。会議の進行ロジックをコードで書くのは根本的に間違っているのではないか、という疑問が湧いてきました。

ここで大きな方針転換を決めました。ロジック制御をコードからAIへ移譲する。具体的には、自然言語の指示書（CLAUDE.md）にロジックを書き、AIファシリテーター自身に判断させる方式への転換です。

同時に、既存のマルチエージェントフレームワークを探し始めました。自作より良いものがあれば使うべきだ、という当然の発想です。

### API課金の現実と二モード構成への分岐

既存フレームワークを試す並行作業として、もう一つ直面していた問題がありました。コストです。

ここまではAnthropic APIを直接叩く構成で動かしていました。安定していて、レートリミットの挙動も予測しやすく、安定運用が必要ならこの構成だろうと思っていました。

ただ、壁打ち用途で何度も回してみたかった私にとって、このペースの消費はきつかった。改善のアイデアを試すたびに数ドル単位で消えていき、数回の会議で15ドルほどがあっという間に溶けていきました（当時はナレッジベース事前仕込みもベクトル検索最適化も未実装で、毎回フルコンテキストを流していたため、今のCLIモード最適化後の $1〜3/回 とは桁が違う消費でした）。気軽に試せなくなって、改善のアイデアを思いついてもこの修正を試すとまた数ドルか、と躊躇するようになり、開発者として健全な状態を失っていきました。

そこで方針を変えて、安定運用枠としてAPIモードを残しつつ、開発中の反復では `claude -p`（Claudeのサブスクリプション内で動くCLI）に切り替える二モード構成にしました。APIモードはレートリミット制御などの安定性が必要な場面のために、CLIモードは壁打ちと反復速度のために。この二モード構成は今も維持しています。

この判断は副次的に、次に検証する Agent Teams やサブエージェント方式といった CLI ベースのマルチエージェント機能を試す動機にもなりました。

### Swarmsを数日試す

Python製のマルチエージェントフレームワーク Swarms を試しました。エージェントの定義や役割分担の設計は直感的で、試してみる価値があると感じました。

数日かけて動かしてみると問題が出ました。一定の条件でハングアップが発生し、複数のエージェントが同じ発言を重複して出力するケースがありました。会議のように安定した進行が必要なユースケースには信頼性が足りなかった、と採用を見送りました。

ただ、これはSwarmsが提供する複数のワークフローパターンをすべて網羅した上での判断ではありません。Sequential系のワークフローを選べば重複発言は回避できた可能性がありますし、プロンプト設計や日本語適合の工夫を積めば、別のユースケースでは十分機能するフレームワークだと思います。私の要件（日本語・リアルタイム進行制御）との相性の問題だった、というのが正確な表現です。

### Agent Teamsを検証

Swarmsと並行して、Claude の Agent Teams 機能も触ってみました。却下はしましたが、何が必要かを学ぶ材料として価値はありました。

lead の自律性が強くて teammates への委任が詰めきれなかったこと（Delegate Mode の調整不足）、task 完了の反映遅延や session resume まわりの experimental 段階っぽさ、トークン消費の重さ、といったあたりが主な理由です。詳細と、逆にここは筋が良いと感じた点は、後半のマルチエージェントFW比較の深掘りでまとめます。

### サブエージェント方式に着地

そのまま `claude -p --agents` を使ったサブエージェント方式に落ち着きました。ファシリテーター役のClaudeが、必要なタイミングで各パネリストをサブエージェントとして呼び出す構造です。

この方式の利点は明確でした。

* ファシリテーターが全ての文脈を持ち、判断してパネリストを呼び出すため制御が明確
* 各パネリストの呼出しは独立しているため、idle復帰コストが発生しない
* MCPサーバーを通じた永続化が自然に組み込める
* CLAUDE.mdに書いた自然言語ロジックがそのままファシリテーターの振る舞いになる

### 現在の構成に落ち着く

CLAUDE.md + MCP meeting-db + notes/（各パネリストの個別メモ） + status.json（pause/continue/feedbackの3状態）という組み合わせが現在の構成です。

status.json は特に気に入っている仕組みです。会議の進行中に、下のような JSON を書き込むだけでファシリテーターが気づいて挙動を変えます。

```
{ "action": "pause" }                          // 一時停止
{ "action": "continue" }                       // 再開
{ "action": "feedback", "message": "..." }     // 議論に反映させたい指示
```

プロセスへの介入を、シンプルなファイルの書き換えだけで実現しています。

**ナレッジベース3種類の使い分け**

## ベクトル検索の再設計：失敗→学び→実測

（補足：記事内ではハイブリッドという呼び方もしますが、キーワード検索＋ベクトル検索の併用という一般的なハイブリッド検索ではなく、ナレッジベース事前仕込み＋save\_research＋要点配布、という複数の retrieval 戦略を組み合わせた設計、という意味で使っています）

### 最初の試み：ベクトル検索の導入

ナレッジベースが大きくなると、全てをコンテキストに入れるのはトークン肥大化につながります。必要な部分だけを意味検索で取り出す設計にするため、ベクトル検索を導入しました。

sqlite-vecとmultilingual-e5-small（384次元）の組み合わせで実装し、発言ログ・ナレッジベースの知識・Webリサーチ結果を全てベクトル化して検索できるようにしました。検索できるようになった、という達成感がありました。

ところが、結果は逆でした。

### 失敗：コストが35%悪化した

ベクトル検索導入後、実際の会議でコストを計測してみると、導入前（ベースライン）比で約35%増加していました。効率化のために入れた機能が、かえってコストを上げている、という想定外の結果です。

### 原因分析：サブエージェント呼出し回数が倍増していた

原因を調べると、パネリストの挙動が変わっていることが分かりました。

ベクトル検索導入前は、パネリストへの呼出し（サブエージェントの起動）が1回の会議あたり平均5回程度でした。導入後は12回前後に増えていました。

何が起きていたかというと、パネリストが自分で `search_knowledge`（ナレッジベースをベクトル検索する自作MCPツール）や `search_messages`（過去発言を検索する自作MCPツール）を呼び出す、という行動が増えていました。ベクトル検索が使えるようになったことで、パネリストが関連情報を自発的に検索するようになり、その検索のたびにコンテキストが増え、次の呼出しコストが上がっていました（`search_research` は後述のハイブリッド設計で導入したツールで、この悪化時点では存在していません）。

ベクトル検索は便利な機能なのですが、使い方を誤るとパネリストが自律的に検索しまくる、という問題を生む可能性があることに気づきました。

*悪化時の構造：パネリスト自身がベクトル検索を自律的に呼び、呼出のたびにコンテキストが膨らんでいた*

### 再設計：3つのアプローチの組み合わせ

問題を整理すると、パネリストが自分で `search_knowledge` / `search_messages` を呼ぶ→コストが上がる、という構造です。解決策は、パネリストが自分で検索しなくていい設計にすること、です。

再設計は3本柱で実施しました。

1. ナレッジベース事前仕込み（prompt cache効率化）  
   各パネリストのsystem promptに、ナレッジベースの関連チャンクを会議開始時に埋め込みます。パネリストは最初から必要な知識を持った状態で起動するため、会議中に検索する必要がありません。しかもsystem promptはprompt cacheが効くため、2回目以降の呼出しコストが大幅に下がります。
2. save\_research（Web結果の自動ingest）  
   researcherエージェントがWebデータを取得したら、自動的にsession DBにingestする仕組みです。パネリストはresearcherが何を調べたかを知らなくてもよく、ファシリテーターがこのテーマについて search\_research で調べてよいと指示した場合のみ使います。
3. ファシリテーターによる要点配布  
   パネリストを呼び出す際の invocation prompt に、議論の要点と参照すべきデータのありかを明示します。パネリストは自分で何かを探すのではなく、ファシリテーターから渡された情報で考える、という設計です。

*再設計後：検索するのはファシリテーター側。ナレッジベースは会議開始時に一度だけ引いて system prompt に埋め込み、以降は prompt cache で再利用される*

### 実測結果

再設計後にコストを計測しました。基準の違いで混乱しないよう、ベースライン（ベクトル検索導入前）と再設計前（＝悪化時点）の両方を併記した比較表にしておきます。

| 議題 | ベースライン（導入前） | 再設計前（悪化時点） | 再設計後 | 導入前比 |
| --- | --- | --- | --- | --- |
| 皐月賞予想（競馬分析FW） | $3.23 | $4.35（＋35%） | $1.63 | **−49.5%** |
| SUBLINE月次レビュー（BSC型） | $2.75 | ― | $2.33 | **−15.3%** |

※ SUBLINE は再設計を始めた後に計測を開始したため、悪化時点のコストは記録が残っていません。

（サンプル数は各条件1〜2回の実測で、複数回平均ではありません。議題や実行時間帯で±10%程度の変動はあり得ます）

代表値として採用するのは導入前比 -49.5% の方です。本記事で -49.5% と言えば、ベクトル検索導入前のベースライン（$3.23）と再設計後（$1.63）の比較を指します。再設計前比 -62.5% は悪化してしまった時点（$4.35）を起点にした数字で、単体の自慢値としては少しフェアでないので採用しませんでした。

競馬分析の方が削減率が高かった理由は、Webリサーチへの依存度が高く、save\_researchによる自動ingestの恩恵が大きかったためです。月次レビューはデータソース（MySQL/GA4等）からのデータが中心なので、Webリサーチの比重が低く、削減率も穏やかでした。

この失敗→再設計のプロセスで得た教訓は、AIエージェントに道具を与えると意図しない使い方をする可能性がある、ということです。設計者はどう使われるかを先に想定して手を打っておいたほうが結果的に安い、という実感を持ちました。

各方式の検証詳細を開く

### 自前SSEオーケストレーター

最初のアプローチです。Node.jsでオーケストレーターを書き、各パネリストのAPI呼出しを管理し、SSEでフロントエンドにストリーミングする構成でした。

動いていたこと：

* 発言ターンの管理
* リアルタイムストリーミング
* 基本的なスレッド構造
* シンプルな会議の流れ

限界として見えてきたこと：  
会議のロジックをコードで書くほど、例外的な状況への対応が難しくなります。このパネリストの発言が短すぎたから深掘りを追加する、今の流れはDevil's Advocateを入れるべき場面だ、こういう判断が必要な分岐をコードで全て書くのは無理でした。

この経験から、会議の進行ロジックはコードではなく AI に書かせるべき、という発想が生まれました。

### Swarms

Python のマルチエージェントフレームワーク。エージェントの定義が直感的で、数日ほど触りました。

引っかかったのは安定性です。一定条件でハングアップし、複数エージェントが同じ発言を重複して出すケースも残りました。会議のように最後まで安定して流れてほしいユースケースではこの挙動は致命的で、検証を切り上げました。Swarms には Sequential / Concurrent / Hierarchical など複数のワークフローがあり、Sequential 系を選んでいれば重複発言は起きにくかったかもしれません。そこまで網羅はできていないので、Swarms 自体の善し悪しというより、私の要件で動かした短期の結果として見てください。

学びとしては、動くだけではなく途中で止まらず動き切ることがどれだけ重要かを体感したのが大きかったです。会議の途中でハングアップすると、それまでの議論の文脈ごと消えるので、復帰コストが重い。

### Agent Teams

Claude Code の Agent Teams 機能を触りました。Opus 4.6 と同時期に登場した experimental 機能（`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` で有効化、Claude Code v2.1.32 以降）で、lead と teammates の2〜16名でチームを組み、Team 名で状態を保ったまま復帰できる常駐型のモデルです。

**セットアップ方法**

専用の初期化コマンドはなく、期待されるフォルダ構成を置いて環境変数を設定すれば動きます。ディレクトリ構成（最小例）:

```
my-team/
├── CLAUDE.md              # lead（あなた）への指示書
└── .claude/agents/
    ├── math.md            # teammate: 数学の先生
    └── science.md         # teammate: 理科の先生
```

`CLAUDE.md`（lead に読ませる指示書）:

```
# テスト会議
math と science の2人の先生がいます。
ユーザーの質問を適切な先生に SendMessage で委譲してください。
```

`.claude/agents/math.md`（teammate の定義。YAML frontmatter + プロンプト）:

```
---
name: math
description: 数学の先生。計算問題に答える
model: haiku
maxTurns: 3
---

あなたは数学の先生です。回答の冒頭に必ず【数学】と付けてください。
```

起動:

```
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
cd my-team
claude -p "計算問題：25×4は？"
```

**実際のアサインの流れ**

lead（Opus）が `CLAUDE.md` を読んで、質問内容から適切な teammate を判断して `Agent tool` で呼び出します。以下は `-p` + `--output-format stream-json` で拾った実行ログの要約（Agent Teams 有効時の典型的な流れ）:

```
1. 【lead thinking】
   「計算問題なので、数学の先生に委譲します」
2. 【Agent 呼出し】
   Agent(subagent_type: "math", prompt: "計算問題：25×4は？")
3. 【task_started】
   task_id: ae38d99d8dd2e8ef2  type: local_agent
4. 【math teammate 応答】(haiku, 2.7秒, 18,270 tokens)
   【数学】25×4を計算します。25 × 4 = 100 ...
5. 【task_notification: completed】
6. 【SendMessage】lead → math 「回答ありがとうございます」
7. 【lead 最終出力】
   「数学の先生からの回答です：25 × 4 = 100」

総実行時間: 16.5s ／ 消費: $0.088（lead Opus ＋ teammate Haiku）
```

lead が `Agent tool` で teammate を spawn し、teammate の回答を受け取ると `task_notification: completed` が返り、lead が最終回答としてまとめ直す、という階層構造になっています。チーム名前を指定して再接続すれば、teammate 側の状態を保ったまま続きができるのが特徴です（※resume まわりはまだ experimental、後述）。

**永続化の実態**

Team の状態は `~/.claude/teams/<team-name>/` 配下に作られます。テスト時に実際にできたのはこんな形:

```
~/.claude/teams/
└── default/                  # チーム名（指定なしなら default）
    └── inboxes/
        ├── math.json         # math teammate への受信箱
        └── science.json      # science teammate への受信箱
```

`inboxes/math.json` の中身（抜粋）:

```
[
  {
    "from": "team-lead",
    "text": "1+1はいくつですか？",
    "summary": "Ask math teacher: what is 1+1?",
    "timestamp": "2026-04-17T06:18:30.766Z",
    "read": false
  },
  {
    "from": "team-lead",
    "text": "回答ありがとうございます。",
    "summary": "数学の先生の回答を共有",
    "timestamp": "2026-04-20T04:04:54.623Z",
    "read": false
  }
]
```

`SendMessage` が各 teammate の受信箱 JSON に追記されていく設計です。チーム名で復帰したとき、teammate はこの未読メッセージを読み直すことで前回からの文脈を取り戻す、という仕組みです。

触っていて引っかかったポイントはこんな感じです。

* **lead の自律性が強くて制御が効きにくい**: lead が自分でタスクを拾ってしまい teammates に回らない場面が多くありました。公式ドキュメントにも「`Delegate Mode` を最初から有効にすべき」と明記されていて、ここを詰めきれなかった私側の未熟さもあります
* **task の完了マーキングが遅延する**: teammate が完了を報告せず、依存タスクが「実行中」のまま止まる現象。公式の [known limitations](https://code.claude.com/docs/en/agent-teams) にも記載されています
* **session resume の未成熟さ**: `/resume` や `/rewind` で teammates が復元されず、lead が既に消えた teammate に話しかけようとする挙動も公式既知の制限として載っています
* **トークン消費が重い**: 公式ドキュメントでも「標準セッションの約7倍のトークンを消費しうる」と記載されており、私の条件（4パネリスト・深掘り2ラウンド・prompt cache有効、N=1〜2）でも概ね同程度の増加を観測しました。条件次第で数字は動きます。構造的な理由として、`SendMessage` のたびに inbox JSON に追記され、teammate 再起動のたびに未読分がコンテキストに乗るため、議論が長引くほど起動コストが直線的に増える点があります。長い議論に向かない設計上の制約といえます
* **永続性**: 複数会議をまたぐ記憶の継承（前回の議事録、ナレッジの引き継ぎ）は単体では持たず、外部DBと自前で繋ぐ前提

CLAUDE.md や MCP・skills は spawn 時に teammate にロードされる仕様になっていて、指示の書き込み口はちゃんと用意されています。ただ lead の自律性が強いぶん、CLAUDE.md に書いた指示が lead 判断で上書きされたり素通りされたりする場面があり、CLAUDE.md が効かないのではなく、Delegate Mode や spawn prompt を詰めきれば lead を制御できたはずだが短期の検証ではそこまで到達できなかった、が正確なところです。Agent Teams 側の実験段階っぽさ（task status / resume 周り）と、私の検証不足の両方が理由です。

一方、設計思想には感心しました。`.claude/teams/` 以下に Team が追加されていき、Team の中に役割をもったメンバーが積まれ、アイドル状態で待機、Team 名を指定すれば状態を保ったまま復帰できる。このモデルはチーム単位で常駐的な AI 組織を運用する用途にハマる構造で、今後かなり期待できる機能だと感じました。私のユースケース（会議ごとに立ち上げて都度完結）とは合わなかっただけで、常駐型のチーム AI を持ちたい場面なら第一候補にできる作りです。Delegate Mode を詰めて再挑戦したい別プロジェクトの筆頭に入っています。

### サブエージェント方式（採用）

`claude -p --agents`を使い、ファシリテーター役のClaudeがパネリストをサブエージェントとして呼び出す方式です。

**Agent Teams との違い**

どちらも階層構造の親子エージェントですが、3つの軸で違います。

| 違いの軸 | Agent Teams | サブエージェント方式（今回採用） |
| --- | --- | --- |
| **teammate の寿命** | 常駐。アイドル状態で待機し、Team 名で復帰できる | 会議ごとに spawn → 完了 → 終了（使い捨て） |
| **状態の持ち方** | teammate 側で `~/.claude/teams/<team>/inboxes/` の JSON に保持 | 外部 DB（MCP meeting-db / MySQL）に集約 |
| **lead の自律性** | 高め。lead が自分で判断して委任（＝お任せ度高） | `CLAUDE.md` の進行ルールで制御可能 |

要するに、常駐チームか使い捨てか、内部 inbox か外部 DB か、lead 自律かルール駆動か、の3対比です。私のユースケース（会議ごとに立ち上げて都度完結）には後者が合っていました。

**選んだ理由**

* ファシリテーターが全文脈を持つ → 制御が一元化される
* パネリスト呼出しが独立した1回のAPI呼出し → idle復帰コストなし
* MCPサーバーとの統合が自然 → 永続化が解決される
* CLAUDE.mdの自然言語ロジックがそのまま動く → 柔軟な進行判断が可能

**現在の課題**  
パネリストの呼出し回数が増えるとコストが積み上がります。前述のベクトル検索の再設計がこのコスト最適化に直接つながっています。

**各エージェントの定義の置き場所**

会議ごとに1つのディレクトリが作られ、その中の `.claude/agents/` サブディレクトリに各エージェントが個別ファイルとして配置されます。名古屋旅行の会議の例:

```
data/agent-meetings/meeting-1776494832479/
├── CLAUDE.md                          # ファシリテーターの指示書
├── .claude/
│   └── agents/
│       ├── panelist-1776494596746-1.md  # バジェット番長
│       ├── panelist-1776494596746-2.md  # 体験クリエイター
│       ├── panelist-1776494596746-3.md  # ロジスティクス参謀
│       ├── panelist-1776494596746-4.md  # みんなの代弁者
│       └── researcher.md
├── status.json
├── minutes.md
├── topic.md
├── notes/
└── data/
```

個別ファイルの中身は、YAML frontmatter（name / description / model / maxTurns）に続けて、性格・専門性・ガードレール・発言ルールを自然言語で書いた指示書です。例（バジェット番長）:

```
---
name: panelist-1776494596746-1
description: バジェット番長・予算管理・コスト視点
model: sonnet
maxTurns: 10
---

あなたは「バジェット番長」（予算管理・コスト視点）です。

## 性格
「それ、いくらかかる？」と常に問い返す。楽しさより持続可能な支出を重視...

## 専門性
交通費・宿泊費・食費の相場感、費用対効果の評価、節約オプションの提示

## ガードレール
ケチすぎて体験を潰さない。予算内での最大化を目指す
```

このファイル + 同じ定義の JSON を `--agents` フラグでも CLI 起動時に渡す二段構えで、エージェント定義が確実にロードされるようにしています。

`--agents` に渡している JSON はこんな構造です（名古屋旅行会議の例、抜粋）:

```
{
  "panelist-1776494596746-1": {
    "description": "バジェット番長・予算管理・コスト視点",
    "prompt": "あなたは「バジェット番長」（予算管理・コスト視点）です。\n\n## 性格\n「それ、いくらかかる？」と常に問い返す...（性格・専門性・ガードレール全文）",
    "model": "sonnet",
    "maxTurns": 15,
    "disallowedTools": ["WebSearch", "WebFetch"]
  },
  "panelist-1776494596746-2": {
    "description": "体験クリエイター・アクティビティ・思い出づくり視点",
    "prompt": "...",
    "model": "sonnet",
    "maxTurns": 15,
    "disallowedTools": ["WebSearch", "WebFetch"]
  },
  "panelist-1776494596746-3": { ... },
  "panelist-1776494596746-4": { ... },
  "researcher": {
    "description": "リサーチャー。Web検索で最新情報やデータを取得する",
    "prompt": "（リサーチ専用の指示書）",
    "model": "haiku",
    "maxTurns": 15
  }
}
```

パネリストには `WebSearch` / `WebFetch` を禁止してリサーチは researcher に集約、researcher だけには Web ツールを許可、という棲み分けも JSON で制御しています。起動コマンドはこの JSON を文字列化して `--agents` に渡す形です:

```
claude -p \
  --agents "$(cat agents.json)" \
  --add-dir ./ \
  --mcp-config mcp-config.json \
  --model opus \
  --verbose \
  --output-format stream-json
```

### 用途を見直した機能：ユーザーレビュー機能

AIの発言に対して評価をつけられる、というユーザーレビュー機能を作りました。この発言は的外れ／この主張は採用、といった評価を各発言につけて、会議の方向修正に使うイメージでした。

実際に使ってみると、介入したい対象が多すぎて現実的でないことが分かりました。会議中に20〜30の発言があり、その一つ一つにレビューをつけるのは手間が大きすぎます。介入したいという欲求は、レビューではなく status.json への feedback や割り込み機能の方がずっと自然に実現できていました。

ただ、リアルタイムの方向修正手段としては不向きでも、別の用途として、会議終了後にあの発言は良かった／悪かったをラベル付けしてプロンプト改善やナレッジ蓄積のフィードバック源にする使い方なら、価値は残るかもしれません。現状は当初の用途では削除候補ですが、用途を変えて残す可能性は手元に残しています。

なお、これとは別に AIメタレビュー機能というものを残しています。会議終了後にAI自身が議論の質を振り返って評価する機能で、ユーザーが個別の発言に手作業で印をつけるのではなく、AIが自分自身を振り返る方向の機能です。介入対象が多すぎる問題が起きないので、こちらは積極的に活用しています。プロセスを見ながら途中で口出しできるという記事のテーマからすると、AI自身が自分を振り返るというのは独立した位置づけとして気に入っています。

![レビュータブ：34件のデータ項目に対して採用／不採用／深掘りボタンが並ぶUI、上部にAIメタレビューの実行ボタン](https://static.zenn.studio/user-upload/0eaa8cd12d3a-20260420.png)  
*レビュータブ：上部に「この会議を評価」（AIメタレビュー実行ボタン）、下に34件のデータ項目それぞれに採用／不採用／深掘りのボタン。右上「34件中 ✓0 ✗0 ⏳34」という表示で、人手で1件ずつ評価する非現実性が一目で分かります（※画像はメタレビュー実行前の状態で、ボタンを押すとAIが34件をまとめて評価します）*

作って初めて要らないと分かる、あるいは用途を見直すことになる。こういう経験は、作る前には手に入らないものでした。

### AIファシリテーターが機能すると変わること

この2点から見えてきたことがあります。人間の会議のために発展してきた多くの補助ツール、マインドマップや進行ファシリテーションツール、議事録テンプレートといったものは、人間の会議特有の問題を解決するために作られています。

AIファシリテーターが機能する会議では、そのような問題の一部が最初から存在しません。進行が散漫になったり、誰かが話を独占したり、結論を出し忘れるのは、自律的に動いているファシリテーターが最初に潰してしまいます。

逆に言えば、AIの会議で本当に必要なのは、プロセス介入・文脈継承・コスト制御。この3つを押さえれば、人間用の補助ツールを全部乗せる必要はありません。

## もし商用サービスに展開するとしたら：API 置き換えの厳しさ

冒頭で書いたとおり、このツールは `claude -p --agents` を使う個人用の位置づけなので、商用サービスには使えません。もし商用化するなら Anthropic API で置き換える必要がある、として現状の仕組みのどこが厳しくなるか、整理しておきます。

置き換えに必要な具体作業と規模感を開く

最大の障壁は `Agent tool` がないことです。`claude -p --agents` のサブエージェント機構は Claude Code 固有で、標準 API には存在しません。置き換えるには、ファシリテーターが各パネリストを呼ぶオーケストレーションを自前で書き直すことになります。サブエージェント方式で捨てた複雑性（自前SSE方式 Node.js 約800行）を、もう一度書く羽目になるわけです。

他に引っかかる点:

* MCP の扱い: Claude Code が stdio で自動接続してくれる MCP サーバを、API 側では HTTP ラップしてツール定義に展開し、tool-use ループを自前で回す必要がある
* CLAUDE.md の指示書ロジック: 毎リクエストに手動で system prompt として含め、`cache_control` を明示指定して prompt cache を効かせる作業が発生
* Read / Edit / Glob / Grep など Claude Code 標準の組み込みツールは消えるので、議事録更新・ファイル参照も全部自前ツールにする必要がある
* コスト構造: サブスクリプション固定から従量課金に戻るので、数回で15ドル問題が再燃する
* ストリーミング JSON フォーマット: `stream-json` と API SSE で形式が違うのでパーサ改修

一方で流用できるもの:

* MySQL・SessionDB・クライアント側DB・nginx・Docker 構成はそのまま使える
* MCPサーバのツール実装（中身）も流用可（接続層をHTTPに変えるだけ）
* フロントエンドは無改修

ざっくり規模感を当ててみます。`Agent tool` 相当のオーケストレーターで 800行くらい（従来型SSE方式の規模が目安）、MCP の HTTP ラッパーとツールハンドラで 200〜300行、Read / Edit / Glob / Grep 相当の自前ツールで数100行、それに prompt cache の `cache_control` 明示、ストリーミングパーサ改修、ツールループ、リトライ制御まで含めて、現状プラス1000〜1500行規模の追加実装になる見込みです。商用化する価値が見えたタイミングで踏み込めばよい、という判断で、今は個人用ツールのレベルに留めています。

## 実践例：下書きを AI に任せて、人間が仕上げる

この記事の下書きは、紹介したツールに書かせました。公開版は、そこから私が何度もレビューを入れて大幅に書き直した形です。AI が出した草稿と最終稿のあいだの距離こそ、このツールの使いどころだと感じているので、何を AI に任せ、どこで人間が引き取るか、という観点で実践例を残しておきます。

具体的に何を AI に書かせて、どう直したかを開く

### 何を AI に任せ、何を人間が見たか

成果物は `minutes.md` として議事録に残り、`draft.md` が本稿の原型になりました。文章構成と初稿執筆はAIに任せ、数値（$3.23など）・日付・技術仕様は実装コードと実行ログ・usage.json と突き合わせて人間（私）が事実確認しています。比較した他フレームワーク評価の留保、基準の違いの明記、機能区別の整理なども人間レビューで加筆しました。AIが書いたから無条件に信頼した、という構造ではない旨を補足しておきます。

### パネリストID と役割の対応表

以下、パネリストが5人登場します。読み進めやすさのために対応表を先に置いておきます。

| 表示名 | 内部ID | 役割 |
| --- | --- | --- |
| 編集長 | `editor-chief` | 進行統括・最終判断・タイトル提案（ファシリテーター兼任） |
| 菊田ボイス | `kikuta-voice` | 文体・視点の菊田っぽさ保証 |
| 媒体レビュアー | `platform-reviewer` | 公開先媒体（Zenn等）の読者視点での読みやすさ・適合性評価 |
| 開発目撃者 | `dev-witness` | 技術内容の正確性・実測値の責任 |
| 構成ドラフター | `drafter` | 章立て提案・本文草稿作成 |

### どんな会議体で書いたか

ブログ記事共同執筆（菊田スタイル）というカスタムフレームワークを用意して、5人のパネリストに議論させ、1本の長編草稿を生成してもらいました。そこからは私のレビューを何度か通して書き直し、本稿になっています。

![ブログ執筆フレームワークの構成画面。編集長／菊田ボイス／媒体レビュアー／開発目撃者／構成ドラフターの5パネリストが並ぶ](https://static.zenn.studio/user-upload/de4a97dcaa08-20260420.png)  
*設定画面のフレームワーク詳細：上の対応表の5人がそれぞれ役割つきで配置されている様子*

特に大事だったのが菊田ボイスというパネリストです。文体だけでなく、書く前提にある想いまで再現してもらう必要があったので、性格やガードレール（してはいけないこと）まで具体的に定義しました。

![菊田ボイス パネリストの編集画面。性格・専門性・ガードレールが詳細に記述されている](https://static.zenn.studio/user-upload/c0422a0ebd47-20260420.png)  
*菊田ボイスの定義：「筆者本人を長年読み込んでいるような嗅覚。表面的な文体模写ではなく、視点の取り方・論理展開の癖まで把握している」「推測で『らしくない』と言わない。必ずナレッジベースの根拠を引く。好みの範囲では押しつけない」*

### 「菊田らしさ」をナレッジベースに積む

このパネリストの背後には菊田スタイルというナレッジベースがあります。過去ブログ記事5本（[interpark.co.jp/dev/](https://www.interpark.co.jp/dev/) の私の連載）から抽出した文体特徴・構成パターン・頻出表現を md ファイル1本にまとめ、ナレッジベースに登録しました。

![菊田スタイル ナレッジベースの管理画面。kikuta-style-guide.md（8KB、45チャンク）が登録されている](https://static.zenn.studio/user-upload/ae55f7e82bfc-20260420.png)  
*「菊田スタイル」ナレッジベース：kikuta-style-guide.md（8KB）が45チャンクに分割され、最終インデックス時刻まで管理されています*

中身は単なる語尾リストではありません。前提となる想いが核、技術の扱い方、転用すべき菊田らしさの核 vs 社内限定の型、など、なぜそう書くかの精神論を上部に置いています。

![kikuta-style-guide.md のチャンク表示。「想いを伝えるために必要な表現の厚み」「文章の骨子は『前提となる想い』」が見える](https://static.zenn.studio/user-upload/62819d23ce50-20260420.png)  
*ナレッジベースのチャンク化された中身。「技術自慢っぽくならない。『これでもかと技術を詰め込む』が、自慢のためではなく想いを伝えるために必要な表現の厚みとして」など、書き方の根拠が小さな単位で参照できる形で保存されています*

このフレームワークとナレッジベースの組み合わせで、5人のパネリストが議論する形です。途中、AIメタレビュー機能とユーザーレビュー機能は別物なので分けて扱ってほしい、といった割り込みを status.json 経由で送りながら、方向性を少しずつ揃えていきました。

![本記事執筆会議で送った割り込みメッセージ。青枠で「【割り込み】AIメタレビュー機能とユーザーレビュー機能は別物です…」と表示され、その下からファシリテーターの発言が再開している](https://static.zenn.studio/user-upload/e8ca4cd9b80b-20260420.png)

### レビュー指摘とその反映

最終的な議事録には、各パネリストのレビュー指摘と、誰のどの指摘がどう反映されたか、までが残ります。

![kikuta-voice が文体・菊田ボイスチェックの結果として、総評「文体が論文・技術仕様書方向に引っ張られ、菊田の本来の声（口語的な敬体）がほとんど出ていない」「機能の全リスト以降は機能説明の羅列になり、想いの糸が切れている」などを自然言語で詳細に返している様子](https://static.zenn.studio/user-upload/90ef9ce75fc0-20260420.png)  
*kikuta-voice の実際の発言。「文体が論文・技術仕様書方向に引っ張られ、『〜ですね』『〜なんですよね』『ただ』『そこで』などの話し言葉リズムがほとんど出ていない」「機能の全リスト以降は機能説明の羅列になり、想いの糸が切れている」など、過去記事から抽出したナレッジベースに照らして、文体と視点の癖までかなり具体的に指摘してきます*

あまりうまくない言葉の並べ方をする点を菊田っぽさと評されるのは、多少腹立つところもあるのですけど、それが個性なのだとすれば受け入れるしかないですね。私の書いたナレッジベースに私を見つめ直させられる、というのは作る前には想像してなかった発見でした。

![議事録全体の構造：決定事項／合意点／対立点が反映済みフラグつきで記録される](https://static.zenn.studio/user-upload/ecbf275488c2-20260420.png)  
*議事録の俯瞰：右ペインの「対立点」では、kikuta-voice が「機能リストは感情を前面に出すため精選すべき」、platform-reviewer が「読み飛ばし注記のまま全量掲載」と意見が分かれ、最終的には後者を採用、という経緯まで残ります。誰がどう主張して誰の案が採用されたかが後で追えるのが議論型の実例*

### AI に相談しながらフレームワークを作る

今回は事前にフレームワークを作り込みましたが、ゼロから作る場合は「AIに相談して作る」入力欄から会話形式で組み立てることもできます。試しに、新規プロダクトのリリース可否を多角的に判断する会議をしたい、と入れてみると、議論型／統合型の判定からパネリストの性格・専門性・ガードレール、設計のポイントまで提案してくれます。
