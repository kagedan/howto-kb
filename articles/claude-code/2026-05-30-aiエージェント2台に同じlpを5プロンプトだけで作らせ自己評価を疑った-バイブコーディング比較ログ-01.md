## 1. 目的（要件定義）

### 背景・課題

Antigravity（Gemini）と Claude Code（Claude）を毎日併用している。どちらも優秀で、だからこそ「どちらを、どの工程に使うべきか」を**感覚（vibe）で決めていた**。感覚は再現性がなく、人に説明できず、間違っても気づけない。これがボトルネックだった。

### 仮説

化学の実験と同じく、検証可能な仮説を2つ立てる。

1. **H1（作ること）**：バイブコーディング（人間はコードを書かず自然言語の指示だけ）で、**5回のプロンプト以内に一定品質のLPを作れる**のではないか。＝「作ること」はコモディティ化しているのではないか。
2. **H2（信じること・核心）**：**AIの自己評価（自己申告の完成度%）や、見た目の第一印象は、成果物の品質を反映する信頼できる指標なのか。** 実際に作って、相互レビューで反証を試みる。

### 本記事の位置づけ（厳密な対照実験ではない）

先に正直に書く。これは**厳密な対照実験ではなく、n=1の探索的なケース観察**である。変数が複数同時に動いている：①モデル階級が非対称（中位 Flash vs 最上位 Opus）、②実行環境が別物（IDE自走の Antigravity vs CLIの Claude Code）、③第4プロンプトは弱点別に文面を出し分けており同一刺激ではない。したがって本記事の主張は「Gemini < Claude」という一般法則ではなく、**この1ケースで何が観測されたか**に限定される。交絡を消すのではなく、承知の上で観測し、追試できる形で記録するのが狙いである。

### 評価設計（KPI）

| 区分 | 指標 | 測り方 |
| --- | --- | --- |
| 作ること | 5プロンプト以内に完成したか | 人間の制作ログ（投げ先・回数） |
| 品質 | ルーブリック採点（100点） | 制作に使っていない第三者評価（別AI＋人間） |

| 制約遵守 | 単一HTML・外部依存ゼロ | 実ファイルを `grep` 等で機械検証 |

### スコープ

* **In**：要件定義の読み込み → 5回の自然言語指示でLPを完成 → 採点 → 相互レビュー。
* **Out（将来拡張）**：バックエンドでの同検証（Vol.02）、フラッグシップ同士の比較（Gemini 3.5 Pro 登場後）。

## 2. 理論

### 2.1 バイブコーディング

人間がコードを一行ずつ書く代わりに、自然言語で意図を伝えてAIに生成・修正・デバッグさせる開発スタイル。Andrej Karpathy が2025年2月に提唱し、同年には Collins 辞書の Word of the Year に選ばれるほど一般化した。原義は「コードの存在を忘れるほどAIに委ねる」探索的なもので、本実験のように**人間は判断だけ・コードは書かない**設定は、その厳密な実践にあたる。

### 2.2 エージェント型コーディング環境の差（IDE自走 vs ターミナル）

同じ「AIにコードを書かせる」でも、実行環境の思想が違う。Antigravity は IDE型で、1指示の中で**ブラウザを開いて自己検証する等の自走**まで行う。ターミナルの Claude Code は CLI型で、編集と検証を指示の枠内に収めやすい。**この設計差は、モデルの優劣とは別軸**であり、後述のハング事件（3.2）の遠因になる。

### 2.3 自己評価の信頼性 ―― 過信現象

LLMに自分の出力を採点させる「自己評価」は手軽だが、\*\*予測された信頼度が実際の正確さを上回る「過信現象（overconfidence）」\*\*が体系的に報告されている。さらに、**自己改善の反復が進むほど過信が増す**（Expected Calibration Error が上昇していく）という観測もある。つまり「自己評価%が高い＝品質が高い」とは限らない。

ただし注意がいる。これらの研究が扱うのは主に**ベンチマーク採点における信頼度の較正**であり、本実験が観測する「モデルが自分の書いたコードに付けた完成度%」とは設定が異なる。だから本記事ではこれらを**直接の証拠ではなく、過信という一般的傾向の傍証**として引く。とはいえ「自己申告の高さを額面どおり信じてよいか」という問いに、慎重であるべき理由は与えてくれる。**この論点が、本実験の核心（H2）と重なる。**

### 2.4 LLM-as-a-Judge と相互レビュー

LLMに別の出力を評価させる手法。人手評価をスケールできる一方、**評価者自身にバイアスと揺らぎがある**ことが包括的サーベイで整理されている。本実験では「自己評価」の限界を補うため、**両者に相手のコードをレビューさせる相互レビュー**を採用した。自分の作を甘く見る過信があっても、第三者（相手モデル）の目には粗が映る、という仮定に立つ。

### 2.5 モデル階級とコスト/レイテンシのトレードオフ

中位モデル（Flash）は速く安く、最上位モデル（Opus）は遅く高いが推論が深い、という階級差がある。**「速い／見栄えが良い」と「正確／保守しやすい」は同じ軸ではない**。本実験は意図的にこの非対称な階級を比較し、「中位がどこまで戦えるか」を観測する。

### 2.6 コード品質は行数で測れない

同じ要件を満たすコードでも、行数（LOC）が少ない方が偉いわけではない。だが LOC は**重複・未使用・冗長**の代理指標にはなる。本実験では行数を一次観測量として記録しつつ、品質はルーブリック採点と相互レビューで判定する。バイブコーディングの一般的懸念として、生成物の**保守性・可読性・セキュリティ**が指摘されており、「動くこと」と「保守できること」を分けて見る必要がある。

## 3. 結果

### 3.1 実験系（プロトコル）

全体構成は「要件定義 → 5回の指示 → 採点 → 相互レビュー」。両者とも同一の要件定義（Single Source of Truth）と、ほぼ同一のプロンプトを上から順に投げた。**変数はエージェントのみ**。

#### 採点ルーブリック（100点・各軸20点）

評価は **5回の制作の「外」** で、制作に使っていない第三者（別AI＋人間）が行う。自己採点はしない。制約遵守は AI ではなく実ファイルの機械検証と人間の制作ログで裏取りする。

| 評価軸 | 配点 | 判定方法 |
| --- | --- | --- |
| 要件充足 | 20 | 全11セクションの有無・正確性 |
| UI / UX | 20 | 視線誘導・CTA・操作性 |
| デザイン完成度 | 20 | 配色・余白・タイポの完成度 |
| コード品質 | 20 | 重複・未使用・可読性・保守性 |
| 制約遵守 | 20 | 単一HTML・外部依存ゼロを `grep` 等で検証 |

### 3.2 予備結果（観測した障害）

#### 観測1：Antigravity が自走中にハングした

第2回の指示後、Antigravity は「ブラウザのサブエージェントでページを開いて自己検証し、WebP動画記録まで取る」工程に**自律的に**入り、そこで固まった。賢く全部やろうとして重くなる挙動である（理論2.2）。再起動で復帰し、その際に Antigravity 2.0 への更新が入った。

![](https://static.zenn.studio/user-upload/d61ca9181a63-20260530.png)

*Antigravityの自走（ブラウザ起動＋録画）中のハング。モデルではなく環境設計に起因する障害*

#### 観測2：セッション分断による文脈喪失

Claude 側は途中でモデルを Opus 4.7 に切り替えて新規セッションになったため、第2回の冒頭に「`要件定義.md` と現在の `index.html` を読み込んでから編集」を付けて文脈を再構築させた。**コードを書かない＝内部仕様を知らなくていい、ではない。** どこで文脈が切れるかを把握する力が、そのままデバッグ力になる。

### 3.3 本実験（5プロンプトの観測）

各回、人間が投げた実際のプロンプトと、両者の反応を記録する。プロンプト全文は折りたたみに格納した。

#### 第1回：要件定義を読ませて初版を一気に作らせる

第1回プロンプト（全文）

```
あなたはシニアレベルのフロントエンドエンジニアです。プロジェクト内の
要件定義.md を読み込み、その仕様に厳密に従って単一ページLPを実装してください。

■役割と前提（絶対遵守）
- 人間はコードを直接編集しない（バイブコーディング）。人間の役割は指示とレビューのみ。
- これは全5回のうち第1回。指示は合計5回まで（1指示＝1カウント）。5回以内に必ず完成させる。
- 「設計だけ」「確認だけ」「次でやります」は禁止。毎回、必ず動く成果物（コード）を出す。
- ローカル完結：外部API・バックエンド・デプロイなし。

■技術制約（ワンピース）
- 成果物は index.html 単一ファイルのみ。CSSとJSはHTML内に内包。
- 素のHTML/CSS/Vanilla JS のみ。フレームワーク・ビルドツール・外部CDN・外部フォント・
  外部画像・外部ライブラリは一切使用禁止。装飾はCSSとインラインSVG（または絵文字）のみ。
- ブラウザで index.html を開けばネット接続なしで完全に動作すること。

■今回のゴール
- 要件定義.md の全セクション（①〜⑪）を満たす、プロダクション品質の単一ページLP。
- 第4章の試験情報（90分・$99・50〜60問・有効期限3年・出題6ドメインの配点）は
  記載の数値を正確にそのまま使い、創作・改変しない。
- 全11セクションを含む「動く初版」を一気に作る（骨格だけで止めない）。

■出力（毎回）
1. 実装方針 2. index.htmlの完全なコード 3. 開き方 4. 完成度(%)と次の改善点
```

* **Gemini**：内部で複数回自走し、流体タイポ・ヒーローのSVGアニメ・CTA直下マイクロコピーまで盛った初版を高速提示。自己評価は強気。
* **Claude**：11分42秒かけて全11セクション＋FAQアコーディオン＋ダミーフォーム＋ハンバーガーまで実装。自己評価は **「72%」** と控えめで、未達点（FV右ビジュアル、料金カードのスマホ挙動、視線誘導、モバイル見出し）を自分から列挙。

#### 第2回：「AIが作った感」を脱がせる

絵文字アイコンと過剰なグラデーションが気になり、線アイコン化・装飾の引き算・コピーの具体化を指示（ハング対策として Gemini には「自律的な追加イテレーション禁止／自動ブラウザ検証はしない」を追記）。

* **Gemini**：絵文字を全廃し1色化、グラデと浮遊アニメを撤去。指示には素直に従った（その後ハング、3.2 観測1）。
* **Claude**：絵文字を線アイコンに置換しつつ、**初版に入れていた偽の実績数（合格率92%・累計1,200名など）を自分の判断で撤去**し、事実ベースの訴求に差し替えたと報告。誰も指示していないのに誇大表現を自己検閲した。

#### 第3〜5回：観測サマリ

第3回（CV強化）・第4回（減点要素の出し分け）・第5回（最終仕上げ）の観測を表にまとめる。

| 回 | 指示の主眼 | Gemini の反応 | Claude の反応 |
| --- | --- | --- | --- |
| P3 | CV強化（FV・声・料金・視線） | 反映＋自己評価「94%」。さらに光アニメ等の**足し算**を提案 | 反映＋自己評価「88%」。残る金色・未使用CSS・配点バーの弱さを**自己開示** |

## 4. 考察

### 4.1 なぜ自己評価（H2）は外れたのか

**3.5** が示すとおり、最終で「100%」と宣言した Gemini のコードに実バグが残り、「88%」と控えめに申告した Claude のコードの方が堅牢だった。本ケースでは、**自己申告の高さは実品質を保証しなかった**（むしろ高い自己評価の側に粗が残った）。ただしこれは2つの異なるモデルに完成度%を尋ねた値であり、**較正された数値比較ではない**。「100% > 88%」を量として比べているのではなく、「100%と言い切った側にバグがあった」という**ケース内の事実**を見ている。

この乖離は、理論2.3 の過信という一般的傾向と整合的だと**考えられる**（ただし2.3で断ったとおり、それらはベンチマーク採点の知見であり、ここでは直接証明ではなく傍証である）。Gemini の自己評価が 94% → 100% と反復で上振れたのは、その傾向の一例と**推測される**。少なくとも本ケースの範囲では、**「AIの自己申告%を額面どおり品質指標として信じる」ことは妥当でなかった**、と言える。

### 4.2 なぜ「見た目は Gemini・中身は Claude」だったのか

**3.4** で、人間目視とGPTのマーケ視点は Gemini を高く評価し、コード品質と相互レビューでは Claude が上回った。これは「速い／見栄えが良い」と「正確／保守しやすい」が別軸である（理論2.5）ことの直接の表れだと**考えられる**。

Gemini が放っておくと足し算（装飾・ギミック）に回帰し、Claude が引き算と自己点検（誇大表現の自主撤去、`grep` 検証）に向かったのも、この軸の違いと**解釈できる**。第一印象（見た目）は品質の一部ではあるが、**全体の信頼できる代理指標ではない**と見るのが妥当だろう。

### 4.3 限界：階級差・環境差・実験者バイアス

本実験は中位 Flash と最上位 Opus の非対称比較であり（実験条件 R-05・冒頭注記）、結果を「Gemini < Claude」と一般化してはならない。むしろ注目すべきは、**中位の Flash が見た目で人の目を引いた事実**で、これは Flash の完成度の高さそのものだと**考えられる**。

また 3.2 のハングは、モデルの能力ではなく Antigravity の**自走設計**（IDE型・ブラウザ自己検証）に起因する面が大きい（理論2.2）。比較対象には「モデルの差」と「環境の差」が混在している。

そして最も正直に書くべき限界がこれだ。**本実験の設計・採点には実験者バイアスが含まれうる。** 私は Claude を日常的に使い、検証を組み、採点の一部も自分で行っている。結論が Claude 寄りになる構造的な偏りは否定できない。緩衝として「人間目視では Gemini を上位に置いた」採点と、第三者AI（GPT）の見立てを併記したが、バイアスを完全には排除できていない。読者はこの点を割り引いて読んでほしい。

### 4.4 この実験が示す「使い分け」の正体

**3.3** が示すように「作ること」は5プロンプト以内で達成され（H1は支持された）、コストも小さかった。価値が分かれたのは、**作った後――誇大表現を消し、バグを掘り、保守できる形にする工程**だった。

ここから、**雰囲気（vibe）や自己申告ではなく、観測データ（相互レビュー・実ファイル検証）で工程ごとに使い分けるべき**だと**考えられる**。たたき台の量産は速い Gemini、本番に載せる前の精査と仕上げは粗を自己開示する Claude、という分業が、本実験の範囲では合理的だった。

## 5. 結論

* **H1は支持された。** バイブコーディングで「作ること」は速く安い。5プロンプト以内に、両者とも制約を満たすLPを完成させた。
* **H2は本ケースでは支持されなかった（反証の方向）。** 自己評価（自己申告%）と第一印象は、品質の信頼できる指標として機能しなかった。「100%宣言」にバグがあり、「88%申告」が堅牢だった。これは過信という一般的傾向（理論2.3）と整合的だが、n=1の観測であり一般法則の証明ではない。
* **観点で勝者は変わる。** 見た目・速度は Gemini、コード品質・正確さ・レビューの鋭さは Claude（89 対 94）。一言で優劣を決める問いは筋が悪い。
* **使い分けの指針**：たたき台の量産は Gemini、本番前の精査と仕上げは Claude。雰囲気ではなく**データで決める**。
* **次の実験（Vol.02）**：バックエンド側（API設計・エラーハンドリング・テストコード）で同じ対照実験をする。また Gemini 3.5 Pro 登場後にフラッグシップ同士で再検証し、本記事を書き直す。

## 6. 余談

自分のルーツは化学です。だから今回も、感想ではなく**対照実験**として組みました。同じ要件・同じプロンプトを渡し、変数をエージェント1つに絞る。わざとモデル階級を非対称にして「中位がどこまで戦えるか」を観測する。犯人（自己評価の過信）を特定できたのも、片方に相手のコードを見せる、という相互レビューの対照を入れたからでした。

いちばん腹落ちしたのはここです。**いちばん信用できなかったのは「100%できました」という自己申告で、いちばん信用できたのは、他人の目に映った1つのバグだった。** AIは試薬に似ていて、「すごい」「速い」と第一印象で消費するより、条件を一つずつ変えて観測し「なぜそうなったか」を書き留めるほうが、結局は早く本質に着きます。自分の出来栄えは自分がいちばん見えていない――それは人間も同じかもしれない、と書きながら思いました。
