## プロローグ

近年、「AIコーディングエージェント」が非常に強力になってきています。

私自身、日々の開発において、様々な機能をAIエージェントと一緒に構築してきました。その中で、**「人間とChatGPTで『こんな風に複数のAI同士を連携させたら、もっとスムーズに自律的なコード生成ができるのでは？もしかして動くんじゃないか？』」** と思いつき、様々な協調フロー（オーケストレーション）の実験を試してみました。

そして、そのオーケストレーションを実際に試行錯誤していくうちに、AI同士の役割分担や安全ガードレールのルールを、再利用可能な「プロンプト（スキル指示書）兼 設計書」として綺麗にドキュメントに残しておきたくなったのです。

「もしかしたらできるかも」というアイデアから出発し、いかにして実用的な自律開発のルールを整理し、安全で高品質なコード生成へとつなげる仕組みを整えていったのか。

そこで構築したのが、主エージェントの **Codex** を「指揮者（Orchestrator）」に据え、**Claude Opus/Sonnet/Fable** という優秀な「楽団（各タスク担当）」を指揮して開発サイクルを回す、自律開発ワークフローです。

「もしかしてできるのでは？」というアイデアから始まったプロンプトを、どのようにしてコード生成を自律的に行うオーケストレーションシステムへとつなげていったのか解説します。

## 1. 「Orchestrator（指揮者）」としてのCodexの役割

### そもそも「オーケストレーション」とは何か？

一般に、AIに開発を任せる際に陥りやすいのが、1つのモデルに「設計して、コードを書いて、テストして、バグがあったら修正して」とすべてのタスクを丸投げしてしまうことです。これを行うと、AIが自分の書いたコードの誤りに気付けず同じエラーで無限ループに陥るなど、視野狭窄に陥りやすくなります。

本ワークフローにおける **「オーケストレーション」とは、全体の開発ライフサイクル（要求定義 ➔ 設計 ➔ 実装 ➔ テスト ➔ レビュー ➔ 修正）を細分化し、各ステップで最も得意なAIモデルを呼び出してタスクを振り分け、それらの成果物を破綻なく統合・制御していく仕組み** のことです。

そして、この「各AIモデルへ適切にパスを出し、その結果を開発環境（ファイルやターミナル）へ適用・調整する進行役（指揮者）」の役割を担うのが、VS Codeと直接繋がっているCodex（Orchestrator）自身です。

### なぜCodexがOrchestratorなのか？

この連携アイデアを検討していた際、ChatGPTから **「VS Code上のCodexであれば、エディタやターミナルと直接繋がっているから、一連のClaudeの実行制御ができるよ」** という回答がありました。

Claudeのような外部モデルは、直接開発環境のコンパイルエラーを検知したり、自律的にコマンドを叩いてビルドを回したりすることはできません。  
そこで、開発環境と直結している**CodexがOrchestratorとなり、Claudeの実行管理をコントロールする**。そして、設計やレビューの知恵だけを外部のClaudeから引き出して、現場（ワークスペース）へ適用・調整する。このChatGPTのアドバイスこそが、本オーケストレーションの設計の鍵となりました。

## 2. 役割の分離とマルチエージェント楽団

Orchestrator（Codex）が指揮する「楽団（Claudeモデル）」は、それぞれ得意分野に応じて明確に役割が分離されています。

### 各モデルの憲法

| エージェント | 役割と責務 | ファイル編集 | 特徴 |
| --- | --- | --- | --- |
| **Codex** | **Orchestrator** ・人間からの指示受け口 ・唯一のファイル編集、コンパイル、テスト実行 ・Claudeの設計・レビュー結果の統合と実行判断 | **許可** | 実行環境（VS Code）を掌握する現場監督。 |
| **Claude Opus** | **設計士** ・要求の整理、不明点の抽出 ・モジュール構成、一方向の依存関係設計 ・テスト方針の策定 | **禁止** | 高い推論能力を活かして設計に特化。直接コードを触らせないことで先走り実装を防ぐ。 |
| **Claude Sonnet** | **検査官** ・`git diff` に対するコードレビュー ・バグ、回帰リスク、テスト不足の指摘 ・セキュリティ、性能、循環依存のチェック | **禁止** | 応答速度と精度のバランスが良く、高速なレビューに最適。 |
| **Claude Fable** | **監査官** ・認証情報、暗号化などの重大セキュリティ監査 ・他モデル間で判断が割れた場合の最終調停 | **禁止** | 高コストなため、ユーザーの**事前承認**時のみスポット起動。 |

## 3. より良いコード生成と対話を促す「プロンプトの設計」

ChatGPTと「こんな感じで試してみよう」と作ったプロンプトには、AIが自律的に動きつつも、人間（開発者）と密に協調してより高品質なコードを出力するための「3大プロンプト設計」が組み込まれています。

### 4.1 憶測の禁止と `needs_clarification` の強制

AIが最もやりがちなのが、「勝手に仕様を憶測してコードを書く」ことです。これを防ぎ、確実で高品質なコード生成を行うため、プロンプトには以下のような厳しい制約を課しています。

> 提供された情報だけでは判断できない点を、憶測で補完しないでください。  
> 不明点が結論、設計、安全性、互換性、ライセンス、テスト結果に影響する場合は、statusをneeds\_clarificationとし、ユーザーへ確認すべき質問を返してください。質問には、その情報が必要な理由と、可能な選択肢の影響を付けてください。

これにより、仕様の曖昧さから生じるバグの混入を未然に防ぎ、開発の初期段階で適切なすり合わせができる安全設計が実現しました。

### 4.2 構造化出力（JSON）の厳格なフォーマット定義

Orchestrator（Codex）が機械的に結果をパースして次の処理に進むため、Claudeには以下の3パターンのJSONフォーマットの出力のみを強要し、雑多な「はい、承知しました」といった会話テキストを徹底排除しています。

* **成功時（`success`）**: 設計結果やレビュー内容を所定のキーに格納。
* **追加確認時（`needs_clarification`）**: 人間に確認すべき質問リスト（`questions`）を格納。
* **失敗時（`error`）**: エラーの詳細（ただしAPIキーなどの漏洩を防ぐために認証情報は除外）を格納。

### 4.3 バイリンガル指示（言語の使い分け）

「人が読む設計書やコード内のコメントは日本語にしたい」が、「プログラミング言語の識別子、コマンド、JSONのキーは英語のままでなければ壊れる」という問題があります。  
プロンプト側で「回答のドキュメントやコメントは日本語で記載し、コードやJSONキーは原文のまま残すこと」を徹底的に指示し、機械処理の整合性と人間側の可読性を両立させています。

## 4. 補助スクリプト `orchestrate.py` による安全ガード

Orchestrator（Codex）がClaude Code CLIを直接呼び出す際、意図しない外部パッケージのインストールや不必要な長時間処理を制御し、効率的なコード生成を行うために `orchestrate.py` が活躍します。

## 5. 循環依存の禁止と一方向のクリーンな依存関係

AIエージェントに実装を任せると、モジュールやクラスがお互いを逆参照し合うような「循環依存」を頻繁に作り出します。これが原因でコンパイルエラーの無限ループ（クラスAが見つからず直そうとしたらクラスBが壊れる）に陥るつまずきを経験しました。

これを解決するため、Orchestratorは設計フェーズにおいて、以下の「一方向依存」のクリーンアーキテクチャ規約を Claude Opus に厳しく守らせます。

## 6. 8段階の堅牢な検証（スモークテスト）パイプライン

自律開発における手戻りを最小限にするため、オーケストレーターは一気に重い処理を走らせず、最も軽量な検証から段階的に進める「スモークテスト・パイプライン」を順番に実行します。

### 段階的検証による効果

いきなり重い設計や実装を開始して「10分待った結果、APIキーエラーで終了した」といった悲劇を防ぎます。  
特に「軽量疎通確認」や「レビュースモークテスト」といったダミーの入力データによる防衛線を張ることで、実行環境の不備を初期段階で安全かつ低コストに検知できるようになりました。

## 空応答

空、空白だけ、必須項目のないJSONを成功扱いしないでください。

Claudeには必ず次のいずれかを返させてください。

{
  "status": "success | needs_clarification | error",
  "summary": "空でない要約",
  "details": "空でない説明",
  "questions": []
}

情報不足時はneeds_clarificationとして具体的な質問を返させ、ユーザー回答前に処理を進めないでください。

空応答の場合は同じモデルで1回だけ再試行し、再度空なら停止してください。別モデルへ自動変更しないでください。

次の順番で実施してください。失敗した場合は次へ進まないでください。

9. Fableはdry-runのみ
10. Gitと安全性を最終確認

## まとめ

「ChatGPTと一緒に試作したプロンプト」と「Codexを指揮者にしたオーケストレーション環境」を整えたことで、AIペアプロにおけるコードのデグレードを防ぎ、より高品質な開発を自律的に行うためのベースラインができました。

1. **人間＋ChatGPT**でプロンプトのアイデアを試作する。
2. **Codex（Orchestrator）** が唯一の編集者として、プロンプトに沿って現場で手を動かす。
3. **Claude Opus/Sonnet（楽団）** から設計やレビューの知恵を安全に引き出す。
4. 安全なゲートウェイ **`orchestrate.py`** を中継させて制御する。

とはいえ、このワークフローはまだ始まったばかりのプロトタイプです。

今後、実際の開発現場でこの仕組みを日々使い込んでいきながら、「もっとこう指示したほうがレビューの精度が上がるな」「この役割分担はこうしたほうがスムーズだな」といった気付きを反映させ、ルールやプロンプトをさらに**煮詰めていきたい**と考えています。
