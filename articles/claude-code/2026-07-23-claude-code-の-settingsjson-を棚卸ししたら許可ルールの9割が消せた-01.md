数ヶ月使い込んだ Claude Code のプロジェクト設定を開いたら、settings.local.json に許可ルールが45個溜まっていた。特定の論文PDFをダウンロードする curl コマンドの完全一致許可。もう存在しないプロセスIDへの kill。引用符が欠けて壊れたパターン。身に覚えのないルールばかりが並んでいる。

棚卸ししてみると、45個のうち32個は消しても何も起きないゴミだった。ついでに設定ファイルの階層や自動許可の仕組み、deny の穴まで一通り整理したら権限まわりの解像度がだいぶ上がったので、わかったことを残しておく。

前提をひとつだけ。settings.json は Claude Code の挙動を決める設定ファイルで、権限まわりは allow、deny、ask の3つのリストに `Bash(git add:*)` のようなパターンを並べて書く。allow は確認なしで実行、deny は実行しない、どちらでもなければユーザーに確認、という判定になる。基本的な書き方は次の記事がよくまとまっているので、本稿は実際に運用して初めて見える側を書く。

<https://qiita.com/tomada/items/45ae90809c714bb5c78e>

先に結論を一枚で。

| わかったこと | 対処 |
| --- | --- |
| settings.local.json は許可プロンプトの押下履歴。放置すると腐る | 定期的に棚卸しして消す |
| ls や git log などの読み取り系は最初から自動許可されている | allow に書かない |
| cd に毎回許可を求められる、は錯覚 | 原因は && の後ろにいる別コマンド |
| `Bash(python3 *)` は任意コード実行の全面許可と等価 | 使うなら引き換えを自覚する |
| deny の .env だけでは秘匿ファイルを守れていない | .ssh や .aws、\*.key まで網を広げる |

## settings は4枚のレイヤーでできている

Claude Code の権限は1つのファイルではなく、4枚のレイヤーの重ね合わせで決まる。

| ファイル | スコープ | git・共有 |
| --- | --- | --- |
| `<project>/.claude/settings.local.json` | そのプロジェクト × そのマシン | 共有しない。自動生成される |
| `<project>/.claude/settings.json` | そのプロジェクト | git 管理してチームで共有 |
| `~/.claude/settings.json` | 全プロジェクト | 個人のグローバル方針 |
| `~/.claude/settings.local.json` | 全プロジェクト × そのマシン | 共有しない |

評価はプロジェクトローカルが最優先。ただし deny はどのレイヤーにあっても勝つ。グローバルの settings.json に `Read(**/.env)` の deny を1個書いておけば、どのプロジェクトの allow でも上書きできない。安全側の設定はグローバルの deny に置く。これが基本形になる。

## settings.local.json の正体は押下履歴

最初の疑問は、誰がこの45ルールを書いたのかだった。答えは自分である。

Claude Code が許可ルール外のコマンドを実行しようとすると確認プロンプトが出る。選択肢には今回だけ許可するものと、今後は確認しないものがあり、後者を選ぶと Claude Code がその場で settings.local.json に allow ルールを追記して保存する。つまりあのファイルは設定というより、過去の「今後は聞かないで」の押下ログだ。

タチが悪いのは、安全に一般化できないコマンドだ。特定URLの curl を `Bash(curl *)` に広げるわけにはいかないから、この完全一致のコマンドを今後許可する、という形で保存される。二度と実行されないコマンドの許可が1行残る。作業中にテンポよく Enter や 2 キーで流していると、今回だけのつもりが今後確認しない側を選んでいることがある。それが数ヶ月分積もると45ルールになる。

## 消せるルールの5分類

実際に消したものを分類するとこうなった。同じ棚卸しをする人はこの5パターンを探すといい。

### 一回限りの許可

特定URLの curl、特定PIDの kill、/tmp に置いた使い捨てスクリプトの実行。その場では意味があったが二度とマッチしない。数はこれが最多だった。

### 死にルール

旧マシンの絶対パスを含むルール。ユーザー名が違うので新環境では絶対にマッチしないのに、マシン移行でそのまま運ばれてくる。

### 失効した参照

MCP サーバのツール名は `mcp__<サーバID>__<ツール名>` の形式で保存されるが、コネクタを繋ぎ直すとサーバIDが変わることがある。旧IDのルールは残っていても何にもマッチしない。

### 壊れたルール

`Bash(python3 -c ':*)` のような、引用符が中途半端に切れたパターン。複雑なワンライナーに今後確認しないと答えると、こういう壊れ方をすることがある。

### 冗長ルール

`Bash(python3 *)` があるのに `Bash(python3)` が併存しているケース。広いルールに包含される狭いルールは消していい。

この5分類で45個が13個まで減った。消して困ったものは今のところない。allow の削除は最悪もう一度プロンプトが出るだけで、それ以上のダメージがない。迷ったら消す側に倒せる。

## 書いても無意味なルールがある

トランスクリプトを集計すると、Claude が実行したコマンドの上位は cd、echo、head、tail、grep、ls だった。ではこれを allow に足せばプロンプトが減るかというと、減らない。最初から自動許可されているからだ。

Claude Code は読み取り専用と判定できるコマンドを組み込みで自動許可している。cat や grep のような定番に加えて、git status や git log といった git の読み取り系サブコマンド、gh pr view、docker ps あたりまでカバーされる。これを知らないと `Bash(ls:*)` のような意味のない allow を書いてしまう。ネットの設定例にはこの手の無駄ルールがけっこう混ざっている。

## cd に毎回聞かれる、は錯覚

自動許可を知ると矛盾に気づく。cd は自動許可のはずなのに、体感では毎回許可を求められている。deny に入れてしまったかと設定を見直したが、どこにもない。

正体は複合コマンドだった。Claude は `cd /path/to/dir && python3 build.py` のように、cd と本命を && で繋いで実行することが多い。許可判定はコマンド全体に対して行われるので、後半の python3 build.py が未許可ならプロンプトが出る。そして確認ダイアログに表示されるのは cd で始まる行全体。人間の目には cd を聞かれているように見える。

だから対策は cd 側ではなく後半側に打つ。プロンプトが出たら && より後ろを見て、そこにいるコマンドを許可するかを判断する。

## インタプリタのワイルドカードは全面許可と等価

一番大事な話がこれ。`Bash(python3 *)` を allow に入れるとプロンプトは劇的に減る。Claude はスクリプト処理を大量に python でやるからだ。ただしこれは任意コード実行の全面許可と等価である。`-c` は、ファイルの代わりに引数へ書いた文字列をそのままプログラムとして実行するオプションだ。Python で書けることはファイル削除でもネットワーク通信でも全部そこに書けるから、`python3 -c` が通る時点で何でもできる。ファイルに書いて `python3 script.py` で走らせる場合も、中身は許可ルールから見えないので構図は同じだ。

同じ理由で危ないものは多い。node や ruby などのインタプリタ。`bash -c` や eval。npx や uvx のようなパッケージランナー。`npm run *` や `make *` のようなタスクランナーのワイルドカード。`npm run test` と個別に指定するなら安全だ。

自分は理解した上で `Bash(python3 *)` を使っている。毎回確認する煩わしさと天秤にかけた選択で、これ自体を否定する気はない。まずいのは等価性を知らずに入れることと、それを deny で守っているつもりになることだ。`Bash(rm -rf:*)` を deny に入れていても、`python3 -c "import shutil; shutil.rmtree(...)"` は止まらない。allow を広げた瞬間、deny の網は素通りされうる。

## deny は .env だけでは足りない

deny の定番は .env の読み取り禁止だが、見直すと穴があった。before と after で載せておく。

```
// before：定番セット
"deny": [
  "Read(**/.env)", "Read(**/.env.*)",
  "Read(**/*secret*)", "Read(**/*.pem)",
  "Read(**/id_rsa)", "Read(**/id_ed25519)",
  "Bash(sudo:*)", "Bash(rm -rf:*)"
]
```

一見手厚いが、SSH 鍵は id\_rsa と id\_ed25519 というデフォルトのファイル名しかカバーしていない。別名で作った鍵は素通りする。もうひとつ見落としがちなのが Claude Code 自身のログイントークンで、`~/.claude/.credentials.json` に平文で置かれている。

```
// after：追加した4つ
"Read(**/.ssh/**)",             // 鍵の名前によらず .ssh ごと
"Read(**/.aws/**)",             // AWS 認証情報
"Read(**/*.key)",               // 一般的な秘密鍵
"Read(**/.credentials.json)"    // Claude Code 自身のトークン
```

広げるときの注意は誤爆だ。`Read(**/*token*)` までやると tokenizer.py のような普通のファイルが読めなくなる。`*.key` もテスト用証明書を扱う開発では引っかかる。deny は引っかかれば気づけるので、まず広めに張って誤爆したら狭める運用でいい。

## allow と deny の間に ask がある

紹介記事は allow と deny の2択で説明することが多いが、実際には ask という毎回確認のリストがある。

```
"ask": [
  "Bash(rm:*)",
  "Bash(curl:*)",
  "Bash(wget:*)",
  "Bash(git reset:*)",
  "Bash(git rebase:*)"
]
```

危ないから禁止したい、でもたまに必要。そういうコマンドの置き場がここになる。rm を deny に入れると本当に消したいとき不便で、allow に入れるのは怖い。ask なら実行のたびに目視確認が挟まる。deny ほど硬直せず、allow ほど無防備でもない。この層を使い分けると設計がだいぶ楽になる。

## 同期するなら settings.local.json だけは外す

設定を複数の Mac で揃えたくなり、実体を Dropbox に置いて `~/.claude/settings.json` からシンボリックリンクを張る構成にした。dotfiles の定石通りで、読み書きともリンクを透過して実体に届く。ハマりどころは2つあった。

ひとつ目。settings.local.json を同期対象にしてはいけない。このファイルは権限プロンプトのたびに自動書き込みされるので、2台で同時にセッションを走らせるとクラウドストレージ上で競合コピーが生まれ、許可ルールが消えたり JSON が壊れたりする。実際、同期対象に入っていた時期のファイルには2台分のユーザー名の絶対パスを含むルールが混在していた。マシン固有という名前の通りに扱い、共通化したいルールだけを settings.json 側に昇格させる。

ふたつ目。シンボリックリンクは、別ファイルに書いてからリネームで差し替える保存方式に出会うと無言で剥がれる。実体への書き込みではなく、リンクというファイル自体が新しい実ファイルで上書きされるからだ。書き込みは成功するので気づかない。その瞬間から同期が静かに止まる。設定を変える操作をしたら ls -la でリンクが生きているか見る癖をつけておくといい。

## 数ヶ月に一度の掃除がちょうどいい

権限設定は一度作って終わりではなく、押下履歴が溜まり続ける生き物だった。allow の削除はプロンプトが1回増えるだけだから、迷ったら消す。読み取り系は書いても無駄。インタプリタの許可は自覚的に。deny の網は .env の先まで。この4つを数ヶ月に一度見直すくらいでちょうどいい。
