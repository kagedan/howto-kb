Claude Codeを触り始めた頃、私はこう思っていました。

> 「え、もうこれだけで開発できるじゃん」

実際、単発の実装はかなり強いです。  
しかし、数日〜数週間単位で開発を続けると、だんだん違和感が出てきます。

* 昨日決めた設計を忘れる
* 過去の指示と矛盾した実装をする
* 意図したMCPを使わない  
  ...etc

最初は「プロンプトが悪いのかな」と思っていました。  
でも使い込むほど、問題はもっと根本的なところにあると気づきます。  
Claude Codeに必要なのは、  
**「良い指示」より「良い記憶管理」** でした。

---

## AI開発で最初にぶつかる壁

Claude Codeは長い会話を続けられます。  
ですが、永遠に全てを覚えているわけではありません。  
コンテキストが増えると、古い会話は自動的に圧縮（compact）されます。  
つまり、以前の細かいニュアンスや設計意図は徐々に失われていきます。

さらに厄介なのが、Context Rotです。  
会話が長くなるほど、

* 古い指示
* 新しい指示
* 一時的な対応
* 仮実装
* 修正履歴

が混ざり合い、文章そのものが複雑化します。  
するとAIは「何が現在の正なのか」を判断しづらくなります。  
これは人間でも同じで、Slackの3000行あるスレッドを読まされて、正確に意思決定しろと言われたら辛いですよね。  
Claude Codeにも同じことが起きています。

実際、LLMの研究でも「コンテキスト量が増えるほど性能が劣化する」ことが報告されています。特に長いコンテキストの中間部分は参照されにくくなる傾向があり、会話が長くなるほどAIの精度は下がります。

## ドキュメントはAIにとっての「記憶装置」

そこで重要になるのが、ドキュメント運用です。  
ここでいうドキュメントは人間向けのものではなく、Claude Codeのための記憶装置です。

私は大きく2種類に分けています。

### 永続ドキュメント

これはプロダクト全体のルールを管理するファイルです。

例えば：

* ディレクトリ構成
* 命名規則
* 各種設計書
* UIルール
* テスト方針

など。

重要なのは、「1ファイルずつ作る」ことです。  
最初から巨大な設計書を作ろうとすると、AIも人間も破綻します。

なので、

1. まず1つ作る
2. Claudeにレビューさせる
3. 人間が承認する
4. 次へ進む

という流れにすると安定します。AI開発では、「少しずつ合意形成する」ことがかなり重要です。

### ステアリングファイル

もう1つ重要なのが、作業単位のメモです。  
これは「ステアリングファイル」と呼ばれることがあります。

ここには、

* 今回のタスク
* やることリスト
* 設計の詳細
* 決定事項
* 注意点
* 現在の実装状態（どこまでできているか）

を書きます。

これがあると、Claude Codeが途中でコンテキストを忘れても復元しやすいです。  
特にcompact（コンテキスト圧縮）前に更新しておくと効果が大きいです。

### ドキュメント構成イメージ

## 「全部メインスレッドでやる」は危険

Claude Codeを使い始めた頃、私は全部を1セッションでやっていました。  
設計、調査、レビュー、ログ解析、テスト。  
でもこれはかなりコンテキスト効率が悪い。

特に、

* テストログ
* ビルド結果
* 依存関係調査
* 大量ファイル検索

みたいな処理は、コンテキストを大量に消費します。  
そこで便利なのがサブエージェントです。

例えば：

* セキュリティレビュー
* 影響範囲調査
* テスト失敗分析
* ドキュメントの整合性チェック

などを別コンテキストで実行する。  
サブエージェントの利点は、終了後にコンテキストが破棄されることです。  
つまり、重い作業だけ切り離すことができます。  
ただし、結果も消えるので、重要な内容は外部ファイルへ保存するのが良い方法です。

なお、サブエージェント自体はClaude Codeに作ってもらうことができます。「〇〇をサブエージェントで調査して」と指示するだけで、適切なコンテキストを切り出して実行してくれます。

また、コンテキストが重くなってきたタイミングでは以下の2つのコマンドが有効です。

* `/compact` : 会話履歴を要約・圧縮してコンテキスト使用量を削減します。ただし圧縮の過程で、決定事項や細かいニュアンスが失われることがあります。**実行前に重要な内容をステアリングファイルや永続ドキュメントに書き出してから使うのがポイントです**
* `/clear` : 会話履歴を完全にリセットします。不要なツール実行結果や一時的なログを一掃したいときに使います

`/compact` は「外部化 → 圧縮」の順序を守ることで効果を発揮します。

## Claude Codeが「思った通りに動かない」理由

Claude Codeを使っていると、

> 「いや、そこは察してくれよ」

と思う瞬間があります。  
でも実際には、AIは暗黙知がかなり苦手です。

例えば人間なら、

* 「このファイルは触らない（歴史的経緯がある）」
* 「この層にビジネスロジックを書かない」
* 「既存コードのスタイルに合わせて書く」

みたいな空気を察せます。AIはそこを明文化しないとズレます。

なので、

* **スラッシュコマンド**: よく使う指示をコマンド化する（`/review` など）
* **Skill**: 特定の状況でどのツールをどう使うかをAIに事前定義する
* **フック**: ファイル保存やコマンド実行などのイベントに応じてルールを自動適用する

などを使い、「どう行動すべきか」を暗黙知も含め固定化すると安定します。  
それぞれの詳細な設定方法は、Claude Codeの公式ドキュメントをご参照ください。  
毎回プロンプトに含めることなく、ルールとして暗黙知も含めてしっかり明文化しておきます。

なお、「意図したMCPツールを使ってくれない」という問題には、Bashツールを活用してCLIをClaude Codeに直接操作させる方法が有効です。MCPへの依存を減らしながら、同じ操作を確実に実行させることができます。

### 暗黙知を減らすイメージ

## まとめ

Claude Codeを使っていて感じるのは、AI開発は「プロンプト力」というより、  
**コンテキスト管理の技術** に近いということです。

* どの情報を外部化（永続化）するか
* いかに暗黙知を明文化するか
* どこでコンテキストを切るか・節約するか

この設計次第で、AIエージェントの体験はかなり変わります。  
そしてこれは、Claude Codeに限った話ではありません。  
どのAIエージェントも、コンテキストの制約や記憶の曖昧さを抱えています。

だからこそ、

を比較するだけではなく、  
**その性能を最大限引き出すための環境設計をどう行うか**も、重要だと感じています。
