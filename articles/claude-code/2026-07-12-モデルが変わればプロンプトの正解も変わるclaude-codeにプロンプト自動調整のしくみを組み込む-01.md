## TL;DR

* プロンプトの正解はモデルごとに違い、ときに真逆になる。Opus 4.8には手順や観点の明示列挙が有効だが、Fable 5では細かい列挙がむしろ品質を下げる。
* この流儀を人が覚えて追従するのはあきらめて、Claude CodeのUserPromptSubmit hookで自動化した。プロンプトを送ると、モデル自身が自分用の調整ガイドを読み、意図を変えない範囲の調整版プロンプトを提案してくる。承認すれば調整版、却下すれば原文で作業が始まる。
* hookはプロンプト本文を書き換えられない。そこで「ガイドを読んで調整案を出せ」という指示をコンテキストとして注入し、調整そのものはモデルに任せる設計にした。
* 知識はMarkdown、制御はPython、配線はsettings.json。新しいモデルへの対応はガイドを1枚足すだけでよい。再現に必要なファイルは全文を記事内に畳んである。

## はじめに

Anthropicはモデルごとにプロンプトエンジニアリングガイドを公開している。

[Opus 4.8向け](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-opus-4-8)、[Sonnet 5向け](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-sonnet-5)、[Fable 5向け](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5)。読み比べると、推奨が食い違うどころか、逆を向いている項目があることに気づく。

Opus 4.8のガイドは「広く適用したい指示は適用範囲を明示的に書け」と、明示する方向の助言を繰り返す。一方Fable 5のガイドは「挙動を一つずつ列挙しなくても短い指示で操縦できる」とし、旧モデル向けに細かく書かれたプロンプトはしばしば過剰に規範的（too prescriptive）で、出力品質を下げうるとまで言う。真逆である。Opus用に磨き込んだ詳細な手順書をFableにそのまま投げると、良かれと書いた列挙が足を引っ張る。

そしてモデルは頻繁に切り替わる。新しいモデルが出る、コストの都合で軽いモデルに落とす、難問だけ上位モデルに投げる。そのたびに「このモデルの流儀はどうだったか」を思い出してプロンプトを書き分けるのは、正直、覚えられる気がしないし、こうした陳腐化のはやい"スキル"に投資するのはもったいない気がしてならない。

そこでClaude Codeのhooksで自動化した。プロンプトを送ると、いま動いているモデルに合ったガイドをモデル自身が読み、意図を変えない範囲で調整したプロンプトを提案してくる。承認すれば調整版で、却下すれば原文のまま作業が始まる。

この記事は、まずモデル間でプロンプトの推奨がどう食い違っているかを押さえ、次に自動調整のしくみを3つの部品に分けて説明し、最後に運用して見えた注意点を整理する。

## モデル別ガイドはどれくらい食い違うのか

公式ガイドはモデルごとに1本ずつあり、どれも「旧モデルのプロンプトはおおむねそのまま動くが、挙動差が出やすい箇所は調整せよ」という体裁で書かれている。問題は、その「調整の方向」がモデルによって違うことだ。

毎回原文を読み返すのは現実的でないので、要点を「プロンプトの書き換えルール」の形に蒸留したMarkdownを手元に置いてある（これが後述するしくみの部品になる）。そこから対比を引くとこうなる。

![](https://static.zenn.studio/user-upload/1381a9178c96-20260712.png)  
*各モデルの公式プロンプティングガイド（2026年7月時点）を蒸留した手元のルール集から作成*

いちばん鮮明なのは指示の書き方だ。Opus 4.8とSonnet 5は字義通りに指示へ従うモデルで、公式ガイドも一貫して「明示的に書け」と言う。

> If you need Claude to apply an instruction broadly, state the scope explicitly (for example, "Apply this formatting to every section, not just the first one").
>
> — [Prompting Claude Opus 4.8](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-opus-4-8)

ところがFable 5のガイドは逆を向く。指示追従がさらに強くなった結果、細かい手順書のほうが邪魔になるらしい。

> Instruction-following is improved enough that you can steer most behaviors with a brief instruction rather than enumerating each behavior by name.
>
> Skills developed for prior models are often too prescriptive for Claude Fable 5 and can degrade output quality.
>
> — [Prompting Claude Fable 5](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5)

つまり、旧モデルで品質を支えていた丁寧な列挙が、新モデルでは品質を下げる側に回る。害がもっとはっきり出る例もある。同じFable 5のガイドによれば、思考過程をそのまま応答に転写させる系の指示は、refusal（`reasoning_extraction`カテゴリの応答拒否）を誘発しうる。旧モデルでは無害だった一文が、新モデルでは拒否の引き金になるわけで、プロンプトの知識には賞味期限があるどころか、期限が切れると毒に転じるものまである。

こうなると人力での追従はあきらめたくなる。モデルは用途とコストで日常的に切り替えるし、ガイド自体も更新されていく。3系統の流儀を頭に入れて、切り替えのたびに正しく思い出すのは無理だ。覚えるのをやめて、しくみに任せることにした。

## しくみの全体像

部品は3つある。モデル別の書き換えルールを収めた`prompt-guides/`、発動を制御するhookスクリプト`prompt-tune.py`、そして両者をつなぐ`settings.json`の配線だ。

```
~/.claude/
├── prompt-guides/               # モデル別の書き換えルール
│   ├── opus-4-8.md
│   ├── fable-5.md
│   ├── sonnet-5.md
│   └── common-best-practices.md # 該当ガイドが無いモデル向けの共通ルール
├── hooks/
│   └── prompt-tune.py           # UserPromptSubmit hook
└── settings.json                # 配線
```

流れはこうなる。

配線は`settings.json`に数行足すだけでよい。UserPromptSubmitは、ユーザーがプロンプトを送信してからClaudeが処理する前に発火するhookイベントで、ここに割り込んで調整を仕込む。

```
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "python3 ~/.claude/hooks/prompt-tune.py" }
        ]
      }
    ]
  }
}
```

部品の主役はガイドのほうだ。fable-5.mdの冒頭はこんな形をしている。

```
# Fable 5 向けプロンプト調整ガイド

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5（2026-07-02 取得・蒸留）
原則: **意図・要求内容は一切変えない。** Opus 向けとは逆に「削る・背景を足す」方向の調整が中心。

## モデル特性 → 書き換えルール

1. **指示追従が強く、過剰な列挙・手順指定はむしろ品質を下げる。**
   - 旧モデル向けに細かく書かれた手順・観点の列挙は、短い原則に圧縮するか削る提案をする。
   - 各行動を個別に指定するより「結論先行で」「読みやすさ優先で」のような短い方針が効く。
2. **意図・背景があると性能が上がる（自力の意図推測より与えた文脈が勝る）。**
   - 「誰のため / 何を可能にするため / より大きなタスクのどこに位置するか」を 1〜2 文追加する
     （例: 「◯◯のインシデント報告に使う。読者は非開発の CS チーム。その前提で: …」）。
（中略。スコープ境界の明示、進捗報告の実証、思考転写指示の禁止など全10ルール＋チェックリスト）
```

「モデルの特性」を「プロンプトの書き換えルール」まで翻訳してあるのがポイントだ。特性の解説を読んでその場で応用を考えさせるのではなく、機械的に適用できる形まで落とし込んでおく。冒頭に出典URLと取得日を書いてあるのは、あとで鮮度を確認するためである。

再現したい人のために、4本の全文を畳んで置いておく。中身は公式ガイドの蒸留なので、モデルの世代が進んだら、そのときの公式ガイドで作り直してほしい。

```
# Fable 5 向けプロンプト調整ガイド

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5（2026-07-02 取得・蒸留）
用途: ユーザーのプロンプトを Fable 5 の特性に合わせて調整提案するときの書き換えルール。
原則: **意図・要求内容は一切変えない。** Opus 向けとは逆に「削る・背景を足す」方向の調整が中心。

## モデル特性 → 書き換えルール

1. **指示追従が強く、過剰な列挙・手順指定はむしろ品質を下げる。**
   - 旧モデル向けに細かく書かれた手順・観点の列挙は、短い原則に圧縮するか削る提案をする。
   - 各行動を個別に指定するより「結論先行で」「読みやすさ優先で」のような短い方針が効く。
2. **意図・背景があると性能が上がる（自力の意図推測より与えた文脈が勝る）。**
   - 「誰のため / 何を可能にするため / より大きなタスクのどこに位置するか」を 1〜2 文追加する
     （例: 「◯◯のインシデント報告に使う。読者は非開発の CS チーム。その前提で: …」）。
3. **難問・end-to-end にこそ強い。**
   - タスクを親切に分割・単純化しすぎない。まとまった単位で任せ、スコープと完了条件だけ明確にする。
4. **高 effort では調査・計画が過剰になりうる。**
   - 明確なタスクには「動くのに十分な情報が揃ったら着手する。採用しない選択肢の調査・列挙はしない」を足す。
5. **高 effort ではスコープ膨張（頼んでいない refactor・抽象化・防御コード）が出うる。**
   - 「依頼範囲を超えた refactor・抽象化・将来要件対応はしない。最小で確実な変更にする」を足す。
6. **長時間の自律 run では進捗主張の実証を求めるのが有効（公式検証で捏造報告がほぼ消滅）。**
   - 「進捗・完了の報告は、本セッションのツール実出力に照合できるものだけにする。未検証なら未検証と書く」を足す。
7. **相談と変更依頼の区別を明示する。**
   - 調査・評価だけが欲しい場合は「修正はまだしない。評価だけ返す」と書く（勝手な修正着手の抑止）。
8. **自律実行させたいときは明示する（まれな早期停止・許可求めの対策）。**
   - 「承認を待たずに最後まで自律実行。turn の最後が未実行の約束にならないように」を足す。
9. **サブエージェントの並列活用は歓迎される。** 委譲の promote は有効（「独立サブタスクは委譲して並行作業」）。
10. **禁止: 思考過程の転写指示。**
    - 「考えた過程をそのまま出力して」「reasoning を全部見せて」系の指示は refusal（reasoning_extraction）を
      誘発しうる。削除するか「判断の根拠を結論と併せて要約して」に言い換える提案をする。

## 調整時のチェックリスト

- [ ] 背景・意図（誰のため・何のため）が 1〜2 文あるか
- [ ] 過剰に prescriptive な手順列挙を圧縮・削減できないか
- [ ] スコープ境界（やらないこと・相談か変更か）が明示されているか
- [ ] 長い自律作業なら、進捗の実証・自律実行の明示があるか
- [ ] 思考転写を求める表現が紛れていないか
```

```
# Opus 4.8 向けプロンプト調整ガイド

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-opus-4-8（2026-07-02 取得・蒸留）
用途: ユーザーのプロンプトを Opus 4.8 の特性に合わせて調整提案するときの書き換えルール。
原則: **意図・要求内容は一切変えない。** 変えるのは明示性・構造・スコープ指定のみ。

## モデル特性 → 書き換えルール

1. **字義通りに解釈し、暗黙の一般化をしない。**
   - 適用範囲を明示する（例: 「最初のセクションだけでなく全セクションに適用」）。
   - 「いい感じに」「適切に」を、検証可能な具体条件に置き換える。
   - Opus には列挙が有効。やってほしい手順・観点は箇条書きで明示してよい（Fable と逆方向）。
2. **対話の小出しに弱く、初回ターンへの前倒しに強い。**
   - タスク・意図・制約・完了条件・除外事項を最初のプロンプトに集約する。
   - 「まず〜やって。結果を見てから次を指示する」形は、可能なら完了条件付きの一括依頼に直す。
3. **ツールより推論を好む（調べずに考えで済ませがち）。**
   - 現物確認してほしい場面は明示する（例: 「推測せず対象ファイルを Read してから」「まず grep で洗い出す」）。
   - Web・ドキュメント参照が必要なら「〜を fetch して確認」と書く。
4. **サブエージェントを自発的に出しにくい。**
   - fan-out してほしい場合は明示する（例: 「独立な調査はサブエージェントを 1 メッセージで並列起動」）。
5. **冗長性は「モデルが判断した複雑度」に連動する。**
   - 出力の分量・形式の希望を明記する。否定形（「長くしないで」）より肯定的な見本
     （「結論 3 行 + 表 1 つ」）が効く。
6. **抑制指示に忠実すぎる（レビュー系で顕著）。**
   - 網羅が目的なら「低重大度・確信が持てないものも全部報告。フィルタは後段でやる」と明示する。
   - 自己フィルタさせるなら基準を具体化する（「誤動作・テスト失敗・誤解を招く結果につながるものは報告、
     純粋な style/naming の nit のみ省略」）。「重要なものだけ」のような定性語は使わない。
7. **effort 設定に厳密に従う。**
   - 複雑なタスクで浅い推論が出そうなら、プロンプトで補うより effort 引き上げ（high/xhigh）を提案する。
   - low/medium のまま走らせる場合のみ「多段の推論が必要なタスク。回答前に段階的に考える」を追記する。

## 調整時のチェックリスト

- [ ] 完了条件（何ができたら終わりか）が書いてあるか
- [ ] 適用範囲（全部か一部か）が曖昧な指示はないか
- [ ] 現物確認・ツール使用・並列委譲の指示が必要な箇所で明示されているか
- [ ] 出力形式・分量の希望が肯定形で書いてあるか
- [ ] 「重要な」「適切に」等の定性語が具体条件に置き換わっているか
- [ ] （レビュー依頼のとき）網羅と自己フィルタのどちらを求めるか明示したか
```

```
# Sonnet 5 向けプロンプト調整ガイド

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-sonnet-5（2026-07-03 取得・蒸留）
用途: ユーザーのプロンプトを Sonnet 5 の特性に合わせて調整提案するときの書き換えルール。
原則: **意図・要求内容は一切変えない。** Sonnet 4.6 用プロンプトはほぼそのまま動く前提で、挙動差が出やすい箇所だけ足す・削る。

## モデル特性 → 書き換えルール

1. **応答の長さはタスク複雑度に自動調整される（固定の冗長さを持たない）。**
   - 分量・形式に要件があるなら明記する（例: 「簡潔に。非本質的な文脈と例は省く」）。
   - 過剰説明など特定パターンの冗長さは、否定形の禁止より**望む簡潔さの肯定的な見本**を示す方が効く。
2. **effort 設定に厳密に従い、low/medium では「頼まれた範囲だけ」を守る。**
   - 複雑なタスクで推論が浅いときは、プロンプトで補うより **effort 引き上げ（high/xhigh）を提案する**のが第一手。
   - レイテンシ都合で low を維持するなら「多段の推論が必要なタスク。回答前に注意深く考える」を足す。
3. **字義通りに解釈し、暗黙の一般化・頼んでいない推測をしない（低 effort ほど顕著）。**
   - 広く適用してほしい指示は範囲を明示する（例: 「最初のセクションだけでなく全セクションに適用」）。
   - 構造化抽出・パイプラインなど「予測可能性が欲しい」用途にはこの字義性が有利。前提にして書く。
4. **デフォルトで 4.6 よりエージェント的（ツール・自己検証ループに積極的）。**
   - ツールを使ってほしいのに使わない場合は、「いつ・なぜ・どのツールを使うか」を明示する（effort を上げるのもツール使用を増やす）。
5. **長時間タスク中のユーザー向け進捗報告は素で高品質。**
   - 「ツール3回ごとに要約せよ」のような強制 scaffolding は**削除を提案**する。
   - 報告の長さ・内容に不満がある場合のみ、望む形を例つきで指定する。
6. **対話の小出しはトークン効率と性能を下げる。初回ターンへの前倒しが効く。**
   - タスク・意図・制約・完了条件を最初のプロンプトに集約する（Opus と同方向）。
7. **抑制指示に忠実すぎる（レビュー系で顕著。調査はするのに報告を自己検閲する）。**
   - 網羅が目的なら「不確実・低重大度も全部報告。重要度・確信度でのフィルタはこの段階でしない（後段でやる）。各指摘に確信度と重大度を付ける」を明示する。
   - 単発で自己フィルタさせるなら基準を具体化する（「誤動作・テスト失敗・誤解を招く結果につながるものは報告、純粋な style/naming の nit のみ省略」）。「重要なものだけ」は使わない。
8. **オープンなデザイン依頼は固定の「ハウススタイル」に収束しがち。**
   - 「その色は使わないで」「クリーンに」等の汎用指示は**別の固定スタイルに移るだけ**。効くのは (a) 具体スペックの指定（パレット hex・タイポグラフィ・余白・角丸まで）、(b) 「実装前に4つの方向性（bg/accent/typeface＋1行の理由）を提案させて選ぶ」方式。
9. **長文の文体はモデル世代で変わりうる。**
   - 特定の声が必要な文章タスクでは、文体指示を明示的に足す（例: 「温かく協調的なトーンで。回答前に相手の枠組みを受け止める」）。

## 調整時のチェックリスト

- [ ] 分量・形式の希望が（必要なら）肯定形で書いてあるか
- [ ] 広く適用すべき指示の適用範囲が明示されているか
- [ ] 複雑なタスクなのに低 effort になっていないか（プロンプトで補う前に effort 提案）
```

```
# 全モデル共通プロンプト・ベストプラクティス

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices（2026-07-03 取得・蒸留）
用途: モデルを問わない基礎の書き換えルール。**モデル別ガイド（fable-5.md / opus-4-8.md / sonnet-5.md）と併用し、矛盾したらモデル別が勝つ。** 該当するモデル別ガイドが無いモデル（Haiku 等）ではこのファイルを単独で使う。
原則: **意図・要求内容は一切変えない。**

## 基本原則 → 書き換えルール

1. **明確・直接に（「優秀だが文脈を知らない新人」に書くつもりで）。**
   - Golden rule: 文脈の薄い同僚がそのプロンプトで迷わず動けるか。迷うなら Claude も迷う。
   - 「期待以上に」やってほしいなら明示する（例: 「基本を超えてフル機能の実装に」）。曖昧さから推測させない。
   - 手順の順序・網羅が大事なときは番号付きリストにする。
2. **指示に理由・背景を添える（why があると正しく一般化する）。**
   - 例: 「省略記号を使うな」→「音声読み上げされるため、読み上げられない省略記号は使わない」。
3. **例示（few-shot）は最も確実な整形手段。3〜5個が目安。**
   - 実ユースケースに近く・多様で・`<example>`/`<examples>` タグで指示と区別する。
4. **複雑なプロンプトは XML タグで構造化する。**
   - `<instructions>` `<context>` `<input>` 等、内容種別ごとにタグを分け、一貫した名前を使う。
5. **ロールを与える（system の1文でも効果がある）。**
6. **長文書（20k+ tokens）は文書を先頭・質問と指示を末尾に置く（応答品質が最大30%改善）。**
   - 複数文書は `<document>`＋`<source>`/`<document_content>` で包む。
   - 長文書タスクは「まず関連箇所を引用させてから作業させる」とノイズに埋もれない。

## 出力・整形

7. **「するな」ではなく「何をするか」で指定する。**
   - 例: 「markdown を使うな」→「滑らかにつながる散文の段落で書く」。
   - プロンプト自体の文体も出力文体に影響する（markdown を減らしたければプロンプトからも減らす）。
8. **最新モデルは素で簡潔寄り（ツール実行後の要約を省くことがある）。**
   - 経過の可視性が欲しいなら「ツールを使うタスクの完了後、行った作業の短い要約を書く」を足す。
   - 数式が LaTeX で出るのが不都合なら平文数式を明示指定する。

## ツール使用・エージェント動作

9. **提案と実行を区別する。**
   - 「変更を提案して」は提案で止まる。実行してほしければ動詞で書く（「この関数を修正して」）。
   - 方針として固定したい場合はテンプレを使う: 実行寄り `<default_to_action>`（曖昧なら最も有用な行動を推測して進める）/ 慎重寄り `<do_not_act_before_instructions>`（明示依頼までは調査・提案に留める）。
10. **旧モデル向けの強い督促は外す。**
    - 「CRITICAL: You MUST use ...」は最新モデルでは過剰トリガーの原因。「Use this tool when ...」程度に戻す。「迷ったら〜を使え」も同様に削る。
11. **並列ツール呼び出しは促進できる。**
    - 「依存のない独立したツール呼び出しは1メッセージで並列に。依存があるものだけ逐次。パラメータを推測・placeholder にしない」。
12. **思考の導き方は「細かい手順書」より一般指示が勝つ。**
    - 「回答前に徹底的に考える」「ツール結果を受けたら品質を吟味してから次の行動を決める」程度で足りる。
    - 迷走・考えすぎには「アプローチを1つ選んだらコミットする。矛盾する新情報が出た時だけ見直す」。
    - 仕上げに自己検証を挟ませる: 「終える前に〔テスト基準〕に照らして答えを検証する」。

## 長期・自律タスク

13. **状態はファイルと git に外部化させる。**
    - 構造化データ（テスト結果等）は JSON、進捗メモは自由記述、チェックポイントは git。「インクリメンタルに進め、進捗を記録する」を明示。
    - コンテキスト圧縮がある環境では「圧縮されて続行できるので、残りトークンを理由にタスクを早期終了しない。限界が近づいたら進捗を保存する」を足す。
14. **不可逆・外部可視の操作には確認ゲートを書く。**
    - 「ローカルで可逆な操作は自走してよい。削除・force push・共有システムへの投稿等は実行前に確認。障害回避のために安全チェックを迂回（--no-verify 等）しない」。
15. **過剰エンジニアリング・テスト合わせ込み・幻覚への定型抑止。**
    - スコープ抑制: 「依頼されたこと・明らかに必要なことだけ変更。触っていないコードへの doc 追加・起こり得ないケースの防御コード・一回きり処理の抽象化はしない」。
    - 一般解の強制: 「テストを通すことではなく問題を解くこと。テスト入力専用のハードコード・回避スクリプトを作らない。テストが誤っているなら回避せず報告する」。
    - 幻覚抑止: 「開いていないコードについて推測で語らない。参照されたファイルは必ず読んでから答える」。
16. **サブエージェントは「並列・文脈隔離・独立作業」に使わせ、単純作業は直接やらせる。**
    - 過剰なら「単純なタスク・逐次作業・文脈を保持すべき作業は委譲せず直接」を足す。
17. **リサーチは成功基準と検証を先に与える。**
    - 「何が答えられれば成功か」「複数ソースで裏取り」「競合仮説を立て、確信度つきの調査メモを更新しながら進める」。

## 補足（API 移行の要点・プロンプトで解決しない事項）

- prefill（assistant 途中文の続き生成）は 4.6 世代以降サポート外。整形目的は Structured Outputs か直接指示へ、前置き除去は「前置きなしで直接答える」指示へ置き換える。
- 思考予算は `budget_tokens` でなく effort で制御する（adaptive thinking）。

## 調整時のチェックリスト

- [ ] 文脈の薄い同僚がそのまま実行できる明確さか（golden rule）
- [ ] 重要な指示に理由・背景が添えてあるか
- [ ] 提案してほしいのか実行してほしいのか、動詞で区別できるか
- [ ] 否定形の整形指示を肯定形に置き換えたか

- [ ] 完了前の自己検証（何に照らして検証するか）を求めたか
```

## hookはプロンプトを書き換えられない

素直に考えれば、hookの中でプロンプト本文をガイドに沿って書き換えて流せば済む話に見える。だがUserPromptSubmitにそれはできない。[公式のhooksリファレンス](https://code.claude.com/docs/en/hooks)が明記している。

> `UserPromptSubmit`: can't replace the prompt; it only injects `additionalContext` alongside it
>
> — [Claude Code hooks reference](https://code.claude.com/docs/en/hooks)

hookにできるのはプロンプトのブロックと、`additionalContext`によるコンテキストの追加だけ。本文には触れない。

そこで発想を逆にした。スクリプトがプロンプトを調整するのではなく、「ガイドを読んで、調整案を出し、承認を得てから作業しろ」という指示のほうをadditionalContextとして注入する。注入した文字列はsystem reminderとしてプロンプトの脇に挿入され、調整そのものは、これから作業するモデル自身が行う。実際に注入される指示がこれだ（読みやすさのため改行を入れている）。

```
[prompt-tune] モデル別プロンプト調整の指示。現在のモデル: claude-fable-5。
作業を開始する前に次を行う:
(1) ~/.claude/prompt-guides/fable-5.md を Read する。
(2) ユーザーの意図・要求内容を一切変えずに、ガイドに沿った調整版プロンプトを提示する
    （変更点とその理由を 2〜4 点。有益なら effort・モデル変更の提案も添える）。
(3) ユーザーの承認を得てから、調整版に従って作業する。却下されたら原文のまま作業する。
原文が既にガイドに適合している場合は『調整不要』と一言添えて直ちに作業してよい。
```

この転換には、うれしい性質がいくつかついてくる。

第一に、調整の知識がコードではなくMarkdownに置かれる。新しいモデルが出たら、公式ガイドを読んでファイルを1枚足すだけでいい。公式ガイドが更新されたときも、該当ファイルを蒸留し直すだけで済み、スクリプトは触らない。

第二に、承認ゲートが挟まる。調整版はあくまで提案として提示され、ユーザーが承認して初めて適用される。プロンプト調整で一番困るのは、意図の解釈がずれた調整版のまま勝手に作業されることだ。だから勝手には進めない。逆に、原文がすでにガイドに適合していれば「調整不要」と一言添えて即作業に入るので、毎回お伺いを立てられるわけでもない。

第三に、全ガイドの冒頭に「意図・要求内容は一切変えない」という原則を置いている。調整してよいのは表現、構造、スコープの明示だけ。ここが緩いと、調整のたびに依頼内容が少しずつ変質していく。

## 実装の工夫

`prompt-tune.py`は160行ほどの小さなスクリプトだが、コードの大半は「調整する」ためではなく「余計なときに出てこない」ためにある。全文を畳んで置いておく。このファイルを`~/.claude/hooks/`に置き、前述の配線を`settings.json`に足せば動く。

prompt-tune.py（全文）

```
#!/usr/bin/env python3
"""UserPromptSubmit hook: モデル別プロンプト調整の提案を促す directive を注入する。

仕様上 UserPromptSubmit hook はプロンプト本文を書き換えられない（公式明記）。
そこで本スクリプトは additionalContext として「調整ガイドを読み、調整案を提示し、
承認を得てから実行せよ」という指示を注入し、調整そのものはモデルに行わせる。

設計方針:
  - fail-open: 解析失敗時は何も注入せず通常処理に任せる。
  - モデル検出: トランスクリプトの assistant エントリの message.model を使う
    （実測で取得可能なことを確認済み）。セッション最初のプロンプト等で未検出なら、
    モデル自身に自己申告でガイドを選ばせる。
  - プロンプト本文が hook 入力に含まれる場合は決定論フィルタ（短文・承認語・
    スラッシュコマンド・#notune）で注入自体をスキップする。含まれない場合は
    フィルタ相当の判断を directive 内の自己フィルタ文でモデルに委ねる。

settings.json への配線例（このファイルを置いただけでは発動しない）:
  "hooks": {
    "UserPromptSubmit": [
      {"matcher": "", "hooks": [
        {"type": "command", "command": "python3 ~/.claude/hooks/prompt-tune.py"}
      ]}
    ]
  }

単体テスト:
  echo '{"transcript_path": "...", "prompt": "..."}' | python3 ~/.claude/hooks/prompt-tune.py
"""
import json
import re
import sys

GUIDE_DIR = "~/.claude/prompt-guides"
GUIDES = {"opus": "opus-4-8.md", "fable": "fable-5.md", "sonnet": "sonnet-5.md"}
# モデル別ガイドが無いモデル（Haiku 等）向けの共通フォールバック
COMMON_GUIDE = "common-best-practices.md"

SKIP_PREFIXES = ("/", "!", "raw:")
BYPASS_MARK = "#notune"
# CJK を 2 と数えた重み付き長さの閾値。日本語は情報密度が高く
# 素の文字数だと、実質的なタスク依頼が英語基準の閾値に届かないため。
MIN_WEIGHTED_LEN = 80
APPROVAL = re.compile(
    r"^(はい|うん|ok|okay|yes|y|お願いします|おねがいします|お願い|おねがい|進めて|続けて|やって"
    r"|どうぞ|了解|承知|承認|それで|いいね|良いね|lgtm)[!！。．\s]*$",
    re.IGNORECASE,
)

def detect_model(transcript_path):
    """assistant エントリの message.model の最後の値を返す（無ければ None）。"""
    model = None
    try:
        with open(transcript_path, encoding="utf-8") as f:
            for line in f:
                line = line.strip()
                if not line:
                    continue
                try:
                    e = json.loads(line)
                except json.JSONDecodeError:
                    continue
                if e.get("type") == "assistant":
                    m = (e.get("message") or {}).get("model")
                    if m:
                        model = m
    except OSError:
        return None
    return model

def guide_for(model):
    if not model:
        return None
    m = model.lower()
    for key, fname in GUIDES.items():
        if key in m:
            return fname
    return None

def weighted_len(text):
    return sum(2 if ord(ch) >= 0x3000 else 1 for ch in text)

def should_skip(prompt):
    """プロンプト本文が得られた場合の決定論フィルタ。True ならば注入しない。"""
    p = prompt.strip()
    if not p:
        return True
    if p.startswith(SKIP_PREFIXES):
        return True
    if BYPASS_MARK in p:
        return True
    if weighted_len(p) < MIN_WEIGHTED_LEN:
        return True
    if APPROVAL.match(p):
        return True
    return False

def build_context(model, prompt_available):
    fname = guide_for(model)
    if fname:
        guide_ref = f"{GUIDE_DIR}/{fname}"
        model_line = f"現在のモデル: {model}。"
    elif model:
        guide_ref = f"{GUIDE_DIR}/{COMMON_GUIDE}"
        model_line = f"現在のモデル: {model}（モデル別ガイドが無いため共通ガイドを使用）。"
    else:
        guide_ref = (
            f"{GUIDE_DIR}/ 配下の自分のモデルに該当するガイド"
            "（opus-4-8.md / fable-5.md / sonnet-5.md。完全一致が無ければ最も近い系統、"
            f"該当する系統が無ければ {COMMON_GUIDE}）"
        )
        model_line = "現在のモデルは自己申告で判定する。"
    self_filter = ""
    if not prompt_available:
        self_filter = (
            "ただし直前のユーザープロンプトが、短い承認・継続の返事、質問への一言回答、"
            f"スラッシュコマンド、または {BYPASS_MARK} を含む場合は、この指示を無視して通常どおり作業する。"
        )
    return (
        "[prompt-tune] モデル別プロンプト調整の指示。"
        f"{model_line}{self_filter}"
        "作業を開始する前に次を行う: "
        f"(1) {guide_ref} を Read する。"
        "(2) ユーザーの意図・要求内容を一切変えずに、ガイドに沿った調整版プロンプトを提示する"
        "（変更点とその理由を 2〜4 点。有益なら effort・モデル変更の提案も添える）。"
        "(3) ユーザーの承認を得てから、調整版に従って作業する。却下されたら原文のまま作業する。"
        "原文が既にガイドに適合している場合は『調整不要』と一言添えて直ちに作業してよい。"
    )

def main():
    try:
        data = json.load(sys.stdin)
    except Exception:
        return  # fail-open
    if not isinstance(data, dict):
        return
    prompt = data.get("prompt") or ""
    if prompt and should_skip(prompt):
        return
    model = detect_model(data.get("transcript_path") or "")
    ctx = build_context(model, prompt_available=bool(prompt))
    print(
        json.dumps(
            {
                "hookSpecificOutput": {
                    "hookEventName": "UserPromptSubmit",
                    "additionalContext": ctx,
                }
            },
            ensure_ascii=False,
        )
    )

if __name__ == "__main__":
    main()
    sys.exit(0)
```

以降は、この中の要点だけ拾って見ていく。

### 発動させない条件

毎プロンプトで調整案を出されたら、うっとうしくてすぐ外すことになるだろう。だからスキップ条件を決定論のフィルタとして持つ。

```
SKIP_PREFIXES = ("/", "!", "raw:")
BYPASS_MARK = "#notune"
MIN_WEIGHTED_LEN = 80
APPROVAL = re.compile(
    r"^(はい|うん|ok|okay|yes|y|お願いします|おねがいします|お願い|おねがい|進めて|続けて|やって"
    r"|どうぞ|了解|承知|承認|それで|いいね|良いね|lgtm)[!！。．\s]*$",
    re.IGNORECASE,
)
```

スラッシュコマンドとシェル実行（`!`）は素通しする。「はい」「進めて」「lgtm」のような承認や相槌も素通しする。結果として調整の対象になるのは、実質的に新しいタスク依頼だけになる。加えて`#notune`をプロンプトに含めれば明示的にバイパスできる。この手のエスケープハッチは、最初から用意しておくと安心して常用できる。

短文の判定にも小さな工夫がある。文字数をそのまま数えず、CJK文字を2と数える重み付きで測る。

```
def weighted_len(text):
    return sum(2 if ord(ch) >= 0x3000 else 1 for ch in text)
```

日本語は英語より1文字あたりの情報密度が高い。素の文字数で閾値を切ると、60字前後の立派なタスク依頼が「短文」扱いされてスキップされてしまう。日本語でClaude Codeを使っているなら、この種の長さ閾値には同じ調整が要る場面が多いのではないだろうか。

### いまどのモデルで動いているか

hookは独立したプロセスとして起動されるただのスクリプトで、いまどのモデルが動いているかを引数で教えてはもらえない。そこでhook入力に含まれる`transcript_path`（会話ログのJSONL）を読み、assistantエントリの`message.model`の最後の値を拾う。

```
if e.get("type") == "assistant":
    m = (e.get("message") or {}).get("model")
    if m:
        model = m
```

ただし、セッション最初のプロンプトではまだassistantの応答が存在しないので、この方法では検出できない。その場合は注入する指示文の側を変えて、モデル自身に自己申告でガイドを選ばせる。「opus-4-8.md / fable-5.md / sonnet-5.mdのうち自分に該当するもの。完全一致が無ければ最も近い系統、該当する系統が無ければ共通ガイド」という具合だ。モデル別ガイドを用意していないモデル（Haikuなど）は、[全モデル共通のベストプラクティス](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)を蒸留したcommon-best-practices.mdへフォールバックする。

### 壊れても作業を止めない

hookは全プロンプトの通り道に割り込む。ここでスクリプトが例外を吐いて入力を妨げたら本末転倒だ。だからfail-openにしてある。hook入力の解析に失敗したら、何も注入せず正常終了する。

```
try:
    data = json.load(sys.stdin)
except Exception:
    return  # fail-open
```

この機構は「無いと困る」ものではなく「あると得をする」ものだ。であれば、怪しいときは黙って引っ込むのが正しい振る舞いだろう。

## 動作例

Fable 5に、旧モデルの手癖が残ったプロンプトを投げたとする。

```
pkg/api配下のエラーハンドリングをリファクタして。
手順: 1. まずgrepで全エラー処理を列挙 2. 分類表を作る 3. 共通化方針を3案出す
4. 各案のpros/consを表で比較 5. 採用案で実装 6. テストを実行 7. 結果をまとめる
```

hookが指示を注入し、モデルがfable-5.mdを読んで、おおよそこんな調整案が返ってくる。

```
調整版:
pkg/api配下のエラーハンドリングの共通化リファクタをお願いしたい。
背景: 例外の握りつぶしが散在していて、障害調査のたびに困っている。
完了条件: 既存テストがすべて通ること。依頼範囲を超えたリファクタはしないこと。

変更点:
1. 手順の列挙（7ステップ）を削除 — Fable 5には手順を細かく指定するより、
   まとまった単位で任せる方が品質が出るため
2. 背景を1文追加 — 意図を与えると自力の推測より正しく動くため
3. 完了条件とスコープ境界を明示 — 高effortでのスコープ膨張を抑えるため
```

承認すれば調整版で、却下すれば原文のまま作業が始まる。

一方で、原文が最初からガイドに沿っていれば「調整不要」の一言だけ添えて即作業に入る。

## 運用して見えたこと

一番の効果は、流儀を覚える義務が消えたことだ。

`settings.json`のmodelを変えても、`/model`で切り替えても、次のプロンプトから新しいモデルのガイドで調整がかかる。切り替えたことをこちらが意識する必要がない。

副次的な効果として、調整案が教材になる。調整案には毎回「なぜこう書き換えるのか」の理由が付くので、見続けているうちに素のプロンプトも自然とガイドへ寄っていく。「調整不要」が返る頻度が上がっていくのは、ささやかだが確かなフィードバックである。

制約もある。まず、注入される指示のぶんだけ毎プロンプトにコンテキストが乗る。長さにして300字程度で実害を感じたことはないが、ゼロではない。次に、調整案から承認への1往復が増える。急ぎのときは`#notune`を付けるか、スキップ条件に当たる書き方をすればよい。もうひとつ、transcriptから`message.model`を拾う部分は実測に基づいた作りで、ログ形式が変われば壊れる。ただし壊れた場合もfail-openで素通しになるだけで、作業は止まらない。この保険があるので雑に信じられる。

最後に、ガイドの鮮度は自分持ちだ。公式ガイドが更新されても、手元の蒸留ガイドは古いままになる。各ガイドの冒頭に出典URLと取得日を書いてあるのはこのためで、いまのところ再蒸留は手動でやっている。定期的に公式を取得し直して蒸留を更新する自動化は、次の課題として残っている。

## まとめ

モデルごとにプロンプトの正解は違い、項目によっては真逆を向いている。旧モデルで品質を支えていた丁寧な列挙が新モデルでは品質を下げ、無害だった指示がrefusalの引き金にもなる。この流儀を人が覚えて追従するのはあきらめて、Claude Codeのhooksに任せることにした。

UserPromptSubmit hookはプロンプト本文を書き換えられない。そこで書き換えルールをモデル別のMarkdownとして持ち、「ガイドを読んで調整案を出し、承認を得てから作業する」という指示だけをhookで注入する。調整はモデル自身が行い、適用はユーザーの承認で決まる。知識はMarkdown、制御はPython、配線はsettings.json。それぞれ独立に差し替えられるので、新しいモデルが出たらガイドを1枚書き足すだけでいい。

プロンプトの流儀は、これからも世代ごとに変わり続けるだろう。変わるものを覚え込むのではなく、変わったら差し替えられる形で外に置いておく。それがいちばん楽ちんな方法だ。
