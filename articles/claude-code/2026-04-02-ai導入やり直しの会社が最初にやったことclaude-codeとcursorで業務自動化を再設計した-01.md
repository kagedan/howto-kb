## 「AI導入に投資したのに、何も変わらなかった」——その理由、わかりますか？

2026年春、AI導入の話題はもはや珍しくありません。OpenAIが1,220億ドルという史上最大規模の資金調達を完了し、Anthropicのモデルはコーディングから文書作成まで実務の中核に食い込み、AIエージェント時代が本格的に幕を開けました。

一方で、私たちGiving Firstが現場で支援していて感じるのは、「AI導入に100〜300万円を投じたが、何一つ定着しなかった」という声が依然として多いという現実です。

今回ご紹介するのは、そういった"失敗から仕切り直した"実際の支援事例をベースにした話です。**ツールを変えたのではなく、設計思想を変えた**。そこだけが、失敗と成功を分けた境目でした。

---

## なぜ最初の導入は失敗したのか——「ツール先行」の構造的な問題

最初の導入時、その会社がやったことは以下の3つでした。

1. ChatGPT Plusを全社員に契約（月額コスト：約30万円/年）
2. 外部ベンダーにRAGボットの開発を依頼（費用：約180万円）
3. 社内研修を2回実施（費用：約20万円）

**合計230万円超を投じて、結果はほぼゼロ**。

RAGボットは精度が低く、誰も使わなくなった。ChatGPTはいつの間にかプライベート利用専用になった。研修で学んだプロンプト技術は、日常業務に繋がらなかった。

失敗の根本原因を整理すると、明確でした。

ここを直さない限り、何度ツールを変えても同じ結末になります。

## 仕切り直しのステップ2——Claude Code + GASで「転記業務」を撲滅する

最初のターゲットに選んだのは、毎朝30分かかっていた売上集計レポートの転記作業です。

技術スタックはシンプルにしました。

* **Claude 3.5 Sonnet（API経由）**：データの整形・要約・異常値コメントの生成
* **Google Apps Script（GAS）**：スプレッドシートの読み書き・トリガー実行
* **Claude Code**：実装コードの生成・デバッグ・リファクタリング

Claude Codeの使い方は、プロジェクトルートに`CLAUDE.md`を置くところから始めます。

```
# プロジェクト概要
Google Apps Scriptを使って、スプレッドシートの売上データを読み取り、
Claude APIで整形・要約し、Gmailで送信する自動化スクリプトを構築する。

# 使用技術
- Google Apps Script (V8ランタイム)
- Anthropic Claude API (claude-3-5-sonnet-20241022)
- SpreadsheetApp, GmailApp

# ディレクトリ構成
/src
main.gs # メイン処理
claude.gs # Claude API呼び出し
formatter.gs # データ整形
```

このCLAUDE.mdを読み込んだ状態でClaude Codeに投げると、コンテキストが維持されたまま実装が進みます。途中でAPIのエラーハンドリングや、GASの1回あたりの実行時間上限（6分）を考慮したバッチ処理の分割まで、指示せずとも提案してきます。

実際のClaude APIへのリクエスト部分はこう実装しました（GAS）。

```
function callClaudeAPI(prompt) {
const url = 'https://api.anthropic.com/v1/messages';
const payload = {
model: 'claude-3-5-sonnet-20241022',
max_tokens: 1024,
messages: [{ role: 'user', content: prompt }]
};
const options = {
method: 'post',
contentType: 'application/json',
headers: {
'x-api-key': PropertiesService.getScriptProperties().getProperty('CLAUDE_API_KEY'),
'anthropic-version': '2023-06-01'
},
payload: JSON.stringify(payload)
};
const response = UrlFetchApp.fetch(url, options);
return JSON.parse(response.getContentText()).content[0].text;
}
```

**APIキーはスクリプトプロパティに格納するのが鉄則**です。ソースコードに直書きしてGitHubに上げてしまう事故は後を絶ちません。ちょうど先日、AnthropicがClaude Codeのソースコードを誤って公開してしまったというニュースが出ていましたが（機密情報の流出はなかったとのこと）、API鍵の管理はどのプロジェクトでも最初に設計すべき箇所です。

この自動化で、毎朝30分かかっていた作業が**トリガー設定により午前8時に全自動で完了**するようになりました。

## 仕切り直しのステップ3——Cursorで「社内向けツール」を内製化する

次のフェーズでは、問い合わせ返信の自動下書き生成ツールをCursorで内製しました。

Cursorのエージェントモードを使って、**ゼロからWebアプリを1週間で構築**しています。技術スタックはNext.js + TypeScript + Claude API。

Cursorで特に有効だったのは、`.cursor/rules`ディレクトリに業務ルールを記述した`coding-standards.mdc`を置いておく手法です。

```
---
description: このプロジェクト固有のコーディングルール
---
- Claudeへのリクエストはすべて/lib/claude.tsに集約する
- エラーはすべてconsole.errorではなくlogger.tsを通す
- 環境変数はzodでバリデーションしてから使う（.env.example必ず更新）
- コンポーネントはServer Components優先、クライアント処理はuseClient明示
```

このルールファイルがあると、Cursorが複数ファイルにまたがる実装をするときにもブレが出にくくなります。「なんかコードが一貫していない」と感じたときの多くは、このルール設計が甘いケースです。

完成したツールは、Gmailの受信トレイから問い合わせを自動検出し、Claude 3.5 Sonnetが返信下書きを生成して担当者がワンクリックで確認・送信できる構成です。担当者の確認時間は1件あたり平均4分から40秒に短縮されました。

## 仕切り直しのステップ4——「担当者を1人決める」という組織設計

ツールと実装が揃っても、もう一つ変えたことがあります。

**AI活用の担当者を明確に1人決めた**ことです。

役職は関係ありません。この会社では、総務兼経理の方がAI推進担当を兼務しました。その方が毎週1時間、「自動化できる業務の棚卸し」をする時間を設け、月1回のレポートを経営陣に上げる仕組みにしました。

導入したツールの話よりも、この「人と仕組み」の設計のほうが定着に効いています。深津貴之さんがOpenClawを使った全自動開発について発信するなど、AIによる自律的な実行が注目されていますが、**中小企業においては「人間が週1時間だけAIを管理する」設計のほうが現実的で持続可能**です。

## まとめ——失敗した会社の「やり直し」に共通するパターン

今回の事例から得られた教訓を整理します。

| 失敗時 | やり直し後 |  
|--------|------------|  
| ツールから発想 | 課題から発想 |  
| 全社員に展開 | 1人の担当者が推進 |  
| 外部に丸投げ | 小さく内製 |  
| 成果指標なし | 時間削減を週次で計測 |  
| 研修で終わり | 自動化を業務フローに組み込む |

**重要なのは、AIを「使う」のではなく「業務フローに組み込む」という発想の転換**です。

Claude CodeやCursorは、今や実務で使い込むほど威力を発揮するツールになっています。ただし、それを正しく設計し、継続させる「人と組織の仕組み」がなければ、どれだけ優秀なモデルを使っても定着しません。

AI導入の成否は、ツールの選定より**設計思想と組織運用**で決まります。
