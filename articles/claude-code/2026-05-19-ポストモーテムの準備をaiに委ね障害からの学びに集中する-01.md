こんにちは。株式会社EVERSTEELでソフトウェアエンジニアをしているebikenです。

EVERSTEELでは障害の振り返りとしてポストモーテムを作成し、チームで再発防止策を議論、実施しています。そのポストモーテムにおける情報整理の作業を Agent Skills を使って半自動化した取り組みについて紹介します。

## ポストモーテムとは

ポストモーテムとは、Google SRE Book の定義を借りれば「インシデントの記録、影響、緩和・解決のために取られた対応、根本原因、再発防止策をまとめた文書」です。

> A postmortem is a written record of an incident, its impact, the actions taken to mitigate or resolve it, the root cause(s), and the follow-up actions to prevent the incident from recurring.  
> — [Google SRE Book, Chapter 15](https://sre.google/sre-book/postmortem-culture/)

また PagerDuty は、ポストモーテムを「過去のインシデントから学ぶためのプロセス」と定義しています。

> A process intended to help you learn from past incidents. It typically involves an analysis or discussion soon after an event has taken place.  
> — [PagerDuty](https://www.pagerduty.com/resources/learn/incident-postmortem/)

文書としての側面とプロセスとしての側面の両方がありますが、いずれにおいてもその目的は個人を罰することではなく、組織として障害から学ぶことにあります。

前提となるのが **Blameless**（個人を責めない）の文化です。「誰がミスしたか」ではなく「なぜそれが起きたか」に焦点を当てることで、関係者が率直に情報を共有でき、再発防止に向けた建設的な議論が成り立ちます。障害の原因を個人の行動に帰属させるのではなく、システムやプロセスの問題として捉える姿勢が、継続的な改善の土台になります。

ちなみに、postmortem はラテン語の "post mortem"（死の後に）を語源とし、元々は死因を特定するための医学検査（検死解剖）を指す言葉のようです。「事後に原因を調べる」という意味がソフトウェアエンジニアリングの文脈にも借用されています。

## EVERSTEEL での障害対応とポストモーテム

僕らのチームでは、障害が発生した際にはおおよそ以下の流れで対応しています。

1. CloudWatch Alarms からのSlack通知や Sentry エラーのSlack通知、お客様や社内メンバーからの報告などによって問題を検知
2. Slack の障害対応チャンネルにて対応スレッドを作成し状況を共有
3. 開発チーム側のSlackチャンネルでも調査スレッドを作成し、調査内容を記録
4. ロールバックや修正デプロイによって復旧を進める

インシデントが収束した後はポストモーテムを議論するミーティングの準備を担当者が実施します。僕らのチームでは Notion でポストモーテム(文書、ミーティングどちらも)を管理しており、用意したテンプレートに記入します。テンプレートは以下のような形で構成されています。

```
1. インシデントの内容
  - 不具合の内容
  - タイムライン
  - サービスへのインパクト
  - 発生のきっかけと直接原因
  - どのように検出したか
  - どのように緩和、復旧したか
  - 関連リンク
2. 対応振り返り
  - 問題にすぐ気づくことができたか
  - 報告・連携は適切だったか
  - 原因をすぐに特定できたか
  - 原因特定後すぐに修正できたか
  - 修正後の報告内容は適切だったか
3. 再発防止
  - 根本原因は何か
  - 今後発生しないようにどのような対策を行うか
```

ポストモーテムは公開されているテンプレートも多く、[PagerDuty](https://www.pagerduty.com/resources/learn/incident-postmortem/) や[Atlassian](https://www.atlassian.com/incident-management/postmortem) などが参考になりました。

準備されたポストモーテムを活用し、運用に携わっているエンジニアを集めて週次でミーティングを実施しています。そのミーティングでは、前回までのミーティングで決まった再発防止策の進捗確認と、新規で作成されたポストモーテムについて、内容を確認して根本原因の究明と再発防止策を議論します。先ほど紹介したテンプレートを活用して事前に必要十分な情報を埋めておくことで、ミーティングの時間では議論に集中しやすくしています。

## ポストモーテムの情報整理の半自動化を進めた背景

適切なポストモーテムの作成には、正確な情報を複数のツールから収集する必要があります。例えば Slack のアラート通知や Biz チームとのやりとり、開発チーム内部の調査ログといった各種やり取りの内容とタイムスタンプが必要ですし、CloudWatch Logs, Alarm に格納されているエラーログやエラー率の推移といった、発生していた事象とその影響範囲も必要です。他にも Sentry で管理されているエラーイベントやスタックトレース、GitHub でのデプロイ履歴や変更差分といった情報も必要に応じて収集する必要があります。

これらの情報収集やタイムラインの構築、直接原因や影響範囲の精査と Notion への入力作業がポストモーテム作成のたびに発生します。ミーティングでの適切な議論や、しばらく時間が経ってあとから正確に状況を把握するためには必要ではあるのですが、億劫になりがちな作業でした。

この課題に対して、最終的に採用したのは Agent Skills を使ってドラフト作成を半自動化するアプローチでした。プロンプトの変更がデプロイなしで即時反映される軽さと、ユーザーと対話的に進めながら判断の曖昧な箇所をその場で確認できる柔軟さが、僕らの状況に合っていたためです。他に比較検討した2案も合わせて、それぞれ内容と懸念点を記載します。

### 案 A: CLI スクリプト + LLM API

リポジトリに Node.js スクリプトを配置し、各データソースの API を叩いてデータを収集、LLM API でドラフトを生成する案。バージョン管理が可能で再現性が高い一方、LLM API のトークン従量課金が AI コーディングエージェントの利用コストと重複すること、修正のたびにスクリプトの更新とテストが必要になることが懸念でした。LLM API を使って対話的なエージェントを構築することも可能ですが、会話の状態管理やツール呼び出しの制御など処理が複雑になりがちです。

### 案 B: Lambda + Slack Bot

Lambda 関数として実装し、Slack のスラッシュコマンドから起動する案。チーム全員が Slack から利用でき、認証も中央管理できるメリットがあります。Slack Bot を介した対話的なエージェントも構築可能ですが、Bot を常駐させる必要があること、Slack のメッセージングと LLM の会話管理を組み合わせた処理が複雑になりがちなことが懸念でした。加えて、Lambda・IAM・Secrets Manager などのインフラ整備のオーバーヘッドや、プロンプトやワークフローを調整するたびにデプロイが必要になることも初期フェーズでは障壁になると判断しました。

### 案 C: Agent Skills（採用）

Agent Skills とは、Claude Code や OpenAI Codex CLI といったAI コーディングエージェントに対して、特定のワークフローや手順を SKILL.md などのフォーマットで定義し、再利用可能な形で実行させる仕組みです。スキル定義にはワークフローの手順、使用するツール、守るべきルールなどを記述でき、エージェントにその定義に従って対話的にタスクを進めさせることができます。

参考:

この方法では、デプロイが不要でプロンプトの変更が即座に反映されます。案 A・B と異なり、セクションごとにユーザーの確認を挟みながら進められるため、タイムラインの区切りや影響範囲の判断など曖昧さが残る箇所をその場で修正できます。LLM の利用コストもサブスクリプションに加入している場合は追加の API 課金は発生しません。エンジニアのローカル環境に依存するため再現性はスクリプトほど高くなく、利用者も開発環境のセットアップが必要なエンジニアに限られますが、現状ポストモーテムの準備はエンジニアが担当するため問題にはなりませんでした。

## 作ったもの

今回作成した Agent Skill は、以下のように動作します。

以下を最初の入力としてユーザーに求めます。

* 障害報告や調査が行われた Slack スレッドの URL
* 書き込み先の Notion ページの URL
* 影響を受けた対象サービス名
* AWS プロファイル名

障害の内容によって必要な情報は多少変化しますが、最低限上記のコンテキストを先に揃えておくことで、その後の調査が進めやすくなります。

```
❯ /es-postmortem

⏺ Postmortem ドラフトの作成を始めます。まず前提条件とコンテキストを確認させてください。

  Step 0: 前提条件

  CloudWatch Logs 検索および Slack/Notion/Sentry のクレデンシャル取得には AWS SSO ログインが必要です。
  未実行の場合は事前に以下を実行してください:

  aws sso login --profile <本番環境の AWS プロファイル名>

  Step 1: コンテキスト収集

  以下の情報を教えてください:

  1. Biz メンバーとのやりとり Slack スレッドURL（必須・複数可）— #xxxxx 等のスレッド
  2. 開発チーム内部の調査 Slack スレッドURL（基本的に必須・複数可）— #xxxxxx 等
  3. 書き込み先 Notion ポストモーテムページのURL（必須・1つ）
  4. AWS プロファイル名 — CloudWatch Logs / Secrets Manager アクセス用の --profile 値
  5. 対象サービス
  6. その他補足情報（あれば）

  ※ 発生時刻・収束時刻・インシデント概要は収集データから算出するためお聞きしません。

✻ Baked for 8s
```

ユーザーの入力が完了すると、Slack からスレッドの内容の取得を始め、並行して GitHub のデプロイ履歴・Release・PR の調査、CloudWatch Logs へのクエリ実行、Sentry のエラー情報取得を進めます。そうやって収集したデータを突き合わせてタイムラインを構築し、影響範囲・関連 PR・使用したログクエリといった、テンプレートで求められている内容を適切なフォーマットで Notion のページに書き込みます。書き込みが完了したらユーザーに確認を求め、都度入力内容に関する微調整なども対話的に実行します。

以下が Skill のイメージです。(直接社内の構成情報やインフラ設計などを記入している部分があるため公開用に調整しています。)

Skill（公開用に簡略化）

```
# Postmortem ドラフト作成

あなたはポストモーテムのドラフト作成を支援するアシスタントです。
以下のワークフローに従い、対話的にドラフトを作成してください。

## 記載ルール（厳守）

- **データ・ファクトに基づく記載のみ**。推測に基づく内容は記載しない。データから確認できない事項は「不明」と記載する
- **改善策は具体的で実行可能なアクションにする**
  - OK: 「5xx エラー率が閾値を超えたら Slack に通知するアラートを追加する」
  - NG: 「今後はもっと注意する」
- 各項目には必ず情報源を明記する
  - URL を取得できるものはリンクを付与する
  - CloudWatch ログの場合は利用したクエリをコードブロックで明記する
- **タイムライン**: 全てのタイムスタンプを JST で、日付を含む形式（MM-DD HH:mm）で記載する。各データソースのタイムスタンプを突き合わせ、以下のカテゴリに分類して時系列順に整理する:
  - エンバグ（原因となった変更のデプロイ等）
  - 障害発生 / 障害検知 / 止血対応 / 原因調査 / 復旧
  - 形式: `MM-DD HH:mm` [カテゴリ] イベント内容 [情報源](リンク)
- **再発防止セクション**: AI の推測として候補を提示し、チームでゼロベースの議論を促す
- **対応振り返りセクション**: 一切記入しない。空欄のままにする
- **個人・チームを批判しない**。システムやプロセスの改善点にフォーカスする

## ワークフロー

### Step 1: コンテキスト収集

ユーザーに以下の情報を確認する：

1. Slack スレッド URL（障害報告・調査）
2. 書き込み先の Notion ページ URL
3. 対象サービス名
4. AWS プロファイル名
5. その他補足情報

**注意:** 発生時刻・収束時刻はユーザーに聞かない。Step 2 で収集したデータから算出する。

### Step 2: データ収集

以下のデータを収集する。依存関係のない調査は可能な限り並行に実行する。

#### 2-1. Slack スレッド取得
スレッド内の全メッセージを取得し、タイムスタンプを JST に変換してタイムラインに使用する。
アラート発火・障害報告・復旧報告などの文脈から発生時刻と収束時刻を推定する。

#### 2-2. GitHub デプロイ履歴の確認
deploy/rollback のタイミングとバージョンを特定する。
workflow run の履歴から「いつ・何を・どの環境に」デプロイしたかを追跡する。

#### 2-3. CloudWatch Logs 検索
Slack の障害報告時刻とデプロイ日時をもとに時間範囲を設定し、以下のクエリを実行する。

- エラー・警告ログの取得
- エンドポイント別エラー率の時系列（影響開始・収束時刻の特定）
- 影響を受けたテナントの一覧
- 特定リクエスト ID のトレース
- ステータスコード分布の時系列（復旧確認）

インシデント起因のエラーとノイズを区別するため、インシデント前のベースラインと比較する。

#### 2-4. GitHub 原因の深掘り
バージョン間の diff から原因 PR とコード変更を特定する。

#### 2-5. Sentry エラー情報
インシデント期間中のイシュー一覧を取得し、スタックトレースから原因コード箇所を特定する。
新規イシュー（期間中に初発）と既存イシュー（急増）を区別する。

### Step 3: ドラフト生成 + Notion 書き込み

1. Notion API でページのブロック構造を読み取る
2. 各セクションのドラフト内容をユーザーにテキストで提示して確認を取る
3. 承認を得られたセクションから順に Notion に書き込む
4. Markdown → Notion ブロック変換は専用コマンドで行う
```

#### URL や CloudWatch Logs のクエリなど、情報源や情報の取得方法を記録させる

可能な限り情報源へのリンクを残すようにプロンプトを設計しました。Slack のスレッドや GitHub Actions の実行履歴など URL を取得できるものにはリンクを付与し、CloudWatch Logs では検索に使用したクエリをコードブロックで Notion に記録します。再発防止を進める中で改めてログを確認したくなることは珍しくありませんし、AI が生成した内容の確からしさを検証する際にも、リンクやクエリが記載してあると便利です。

#### 根本原因の究明と再発防止策の議論はチームメンバーで行う

AI は収集したデータの範囲でしか判断できません。情報の見落としや、文章化されていないコンテキストが存在する可能性がある以上、再発防止策をすべて AI に委ねるのは避けたいです。例えば、障害対応時にオフィスで口頭で共有した、お客様の環境では当時ネットワークの工事をしていて調子が悪かった、お客様やBizメンバーと電話でやりとりした、といったコンテキストを全てAIに渡すことは現時点では難しいと考えています。そのため、根本原因の究明と再発防止策の議論についてはAIの出力をそのまま利用するのではなく、必ずチームメンバーで行うこととしました。

とはいえ、ある程度論点の洗い出しや提案をAIで事前に記入しておくことは議論を円滑に進めることに繋がると思い、あくまで参考としてではありますが記入させるようにしました。漫然と対策の洗い出しを指示するだけでは表層的な案が並びがちでしたが、根本原因の究明におけるなぜなぜ分析のように適切な手法の活用を指示すると、それなりに筋の通った出力をしてくれることが多いです。

## 設計の工夫

### MCP を使わずシェルスクリプトでデータソースにアクセスする

AI コーディングエージェントには MCP（Model Context Protocol）サーバーを接続して外部サービスにアクセスする仕組みがありますが、今回は採用しませんでした。MCP でも同様のアクセス制御や機能提供は可能ですが、今回必要なのは比較的限定的な操作です。そのため、Slack・Notion・Sentry それぞれで薄いラッパーシェルスクリプトを用意し、スクリプト内で curl で各種 API を叩く設計を選択しました。Slack API は `conversations.replies` の1エンドポイントしか使いませんし、Bot Token（`xoxb-*`）によってボットが招待されたチャンネルのみに読み取りを制限するなど、必要最小限の機能を実装する方針では、薄いラッパースクリプトの方が初期実装コストと保守コストを小さくできると判断しました。

### Secrets Manager による秘匿情報管理

秘匿情報の管理には、チームでの共有のやりやすさや漏洩リスク、ローテーションの容易さを考慮して AWS Secrets Manager を採用しました。前述したシェルスクリプトを実行すると、内部で Secrets Manager から各サービスのAPIキーを取得してそのままAPIを叩きます。そうすることで、AI コーディングエージェントが直接 Secrets Manager から秘匿情報を取得してログに記録されてしまうことを避けています。加えて、Claude Code や Codex CLI 側の実行できるコマンドの制限設定なども行なっています。

### Markdown 形式での Notion への書き込み

Notion への書き込みは、前述したデータの読み取りと同様の理由で、薄いラッパーシェルスクリプトでページのブロック構造読み取り/書き込みを行う形にしています。書き込みの際、当初は Notion Block API の JSON をエージェントに直接組み立てさせていましたが正しいフォーマットで作れないことがあったため、AI コーディングエージェントには Markdown で出力させ、Markdown -> Notion ブロックの変換は既存のスクリプトを利用する形に責務を分離しました。こうすることで、安定した、任意のフォーマットでの Notion への書き込みを実現しています。

## 導入してみて

現在ポストモーテム作成に関わるチームメンバー全員がこの Agent Skill を毎回利用しています。これまで30分や1時間以上時間を確保して調査/整理していたのが、スキルに指示を出し、出力された内容をレビューし、必要に応じて調整する、という作業に変化し、ミーティングの準備にかかる時間はかなり短くなりました。正確な時間は測定していないのと、事例によってかかる時間はまちまちですが、体感で元々調査にかけていた時間の8割以上を削減できたのではないかと感じています。

また、以前はインシデントが複数重なってしまったり、業務がかなり忙しい時期などにポストモーテムの作成が雑になってしまうこともありましたが、今は必要十分な内容が安定して記載されるようになり、担当者による質のばらつきも小さくなりました。その結果として、週次のミーティングではより本質的な根本原因の究明と再発防止策の議論に集中できるようになり、そこに割ける時間も増えたと感じています。

なお、ポストモーテム作成の自動化は私たちに限った取り組みではなく、既に汎用的な Agent Skill テンプレートが公開されていたり、インシデント管理プラットフォーム自体にドラフト自動生成機能が組み込まれるケースも増えています。

まずはこうした汎用テンプレートを試したり、incident.io のようなプラットフォームを利用している場合はその機能を利用する方が良いと思います。私たちのように社内固有のデータソースへのアクセスやテンプレートに合わせた細かいカスタマイズが必要になった場合に、独自の Agent Skill 構築を検討するのが良いと思います。

## おわりに

ポストモーテムは障害から学び、同じ問題を繰り返さないための重要なプロセスです。しかしその準備には、複数のデータソースからの情報収集やタイムラインの構築といった定型的な作業が少なくありません。今回はその部分を Agent Skill に委ねることで、本来集中すべき議論と学びにより多くの時間を使えるようになりました。

Agent Skills というアプローチはまだ発展途上ですが、デプロイ不要でプロンプトの変更が即座に反映されるという特性は、社内ワークフローの自動化と相性が良いと感じています。弊社では今回のポストモーテム作成以外にも、PR作成、テストケース最適化、各種サービスのベストプラクティスを踏まえたコードレビューなど、様々なワークフローの自動化に取り組んでいます。

本ブログが、ポストモーテムの作成に負担を感じていたり、障害の振り返りをこれからやろうとしている方の参考になれば幸いです。
