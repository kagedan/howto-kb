Komlock lab VPoE阿部([@takupeso](https://x.com/takupesoo))です。

Coinbaseの[x402](https://docs.cdp.coinbase.com/x402/welcome)、Stripeの[MPP](https://docs.stripe.com/payments/machine/mpp)と決済プロトコルが出てきたいま、AIエージェントが自分でAPIを呼び、モノを買い、お金を動かす——そんな「エージェント主体の経済活動」が現実になりつつあります。

この記事は、AIエージェント向けのウォレット・決済CLI **Kova** を触るハンズオンの手引きです。**全行程をテストネット（Polygon Amoy）で完結**させているので、実費ゼロで進められます。

## 1. Kovaとは何か

Kovaは、**AIエージェントに決済機能（ウォレット管理・送金・決済）を持たせるための CLI** です。[Open Wallet Standard（OWS）](https://openwallet.sh/)に基づく HD ウォレットを作成・管理し、ネイティブトークンや ERC-20（USDC / USDT / JPYC）の送金、ポリシーで守られたエージェント実行を、CLI とスキルとして提供します。

秘密鍵は AES-256-GCM で暗号化され、利用者自身が管理します（non-custodial）。

「AIに送金させる」だけのツールは他にもあります。Kovaの特徴は、

> **AIに権限を渡しても、実行経路（ターミナルかエージェントか）とポリシーで操作が制限される**

という設計です。後半のハンズオンで、これを実際に試します。

## 2. 将来像——AIエージェント決済の「ハブ」になる

Kovaが目指しているのは、単一のウォレットツールではなく、**さまざまな決済手段に対応する AIエージェント決済のハブ**です。

エージェントが経済活動をするとき、支払い手段は1つではありません。オンチェーンのステーブルコイン送金もあれば、x402のようなHTTPネイティブな都度課金、将来的にはクレジットカードを使った決済もありえます。Kovaはこれらを**OWSという共通規格でまとめる**ことを目指しています。

*エージェントは決済手段の違いを意識する必要がなく、Kova に「払って」と頼むだけ。どの手段で払うかの選択と、ポリシーによるチェックは Kova が担う——という構図を目指しています。*

ポイントは、**決済手段が増えても、エージェントへのインターフェース（自然言語＋ポリシー）は変わらない**こと。これがハブとしての価値です。

## 3. 今できること——ブロックチェーン操作

ここまでは将来像です。現時点でKovaができるのは、ブロックチェーンのウォレット操作です。`kova --help` を叩くと、コマンドがグループごとに並びます。

| グループ | 主なコマンド | できること |
| --- | --- | --- |
| 初期設定 | `init` | ウォレット・ポリシー・エージェントキー・スキルを一括セットアップ |
| ウォレット & セキュリティ | `wallet` / `key` / `policy` | アドレス確認、APIキー管理、支払いルール設定 |
| ブロックチェーン操作 | `balance` / `send` / `swap` / `sign` | 残高確認・送金（既定 dry-run）・スワップ・署名 |
| AI連携 | `launch` | 認証情報を渡して Claude Code 等を起動 |
| 設定 & 拡張 | `config` / `token` / `skills` | RPC設定、トークン登録、スキル導入 |
| 状態確認 | `status` | ウォレット/ポリシー/エージェントの状態表示 |

対応チェーンはメインネット（Ethereum / Base / Polygon / Arbitrum / Optimism / Avalanche）とテストネット（Sepolia / Base Sepolia / **Polygon Amoy** / Fuji）。今回のハンズオンは Polygon Amoy だけを使います。

## 4. このハンズオンの目的とやること

**目的は、次を体験すること**です。

1. **エージェントがお金の主体になる** — Claude Code から、AIが自然言語で自分のウォレットを操作する
2. **AIの権限が実行経路とポリシーで制限される** — owner-only の制限と、自分で設定したポリシーの二段で、意図しない操作を防ぐ

このハンズオンでは次の2つを実行します。

1. **基本的なKovaコマンドの実行** — インストールからウォレット作成、残高確認、AIエージェントへの接続、そして「危険操作が構造的に弾かれる」体験まで
2. **JPYC EC のテスト実行** — AIエージェントが JPYC（円ステーブルコイン）のECサイトで実際に買い物をする

### 前提

* Node.js 20以上
* [Claude Code](https://docs.claude.com/en/docs/claude-code) CLI（`claude` が PATH 上で動くこと）
* macOS または Linux（Windows は WSL2）

## 5. ハンズオン①：基本的なKovaコマンドを実行する

### Step 1: インストール

```
npm install -g @komlock_lab/kova
kova --version
# 0.1.2
```

### Step 2: `kova init` でセットアップ

`kova init` は対話形式の4ステップで、初期設定が1コマンドで完了します。

| Step | 内容 |
| --- | --- |
| 1/4 | パスフレーズで暗号化したアカウント作成 + 復旧フレーズ確認 |
| 2/4 | 支払いルール（ポリシー）を自動登録 |
| 3/4 | AI接続用のエージェントキーを発行（`~/.kova/config.json` に保存） |
| 4/4 | 拡張機能（スキル / プラグイン）を選択してインストール |

実際の対話は、おおよそ次のように進みます。

1. **言語選択**: `日本語` を選択
2. **利用規約への同意**: 規約・プライバシーポリシーが提示され、`Yes` で同意
3. **Step 1/4 アカウント作成**:
   * アカウント名を入力（デフォルト `default`）
   * パスフレーズを入力（8文字以上／記号・数字・大小文字を混ぜる／一般的な単語や繰り返しは不可）→ 確認のため再入力
4. **Step 3/4 AI 接続設定**: エージェントキーの有効期限を選択（`30 日（推奨）`）→ キーが発行され、プロファイルに保存される
5. **Step 4/4 拡張機能のインストール**: スキル一覧から、↑/↓ で移動・Space で選択・Enter で決定。ここで **`JPYC EC` を選択**しておきます（後半の「JPYC EC のテスト実行」で使います）
6. **復旧フレーズの表示**: 12 語の復旧フレーズが表示される。**アカウント復元の唯一の手段なので、安全な場所に保管**する

完了すると、作成したアカウントのアドレス・ポリシー（初期は「制限なし」）・エージェントキーの有効期限が表示されます。

> Step 2/4（支払いルール）は「制限なし」で自動設定されます。送金制限は後から `kova policy update` で設定でき、ハンズオン②で実際に設定します。

### Step 3: コマンドで状態を見る

**`kova status`** — ウォレット/ポリシー/エージェントの状態を表示します。

```
アクティブアカウント:  default (0x6465f5801a1056223dAA4c86F33B85347174dF53)
HD マスター:           default
アクティブプロファイル: default
標準入力:              TTY (対話ターミナル)
認証方式:              api-key
アクティブポリシー:    1 件のポリシーが登録済み
ポリシー適用:          適用中 (credential ベースのエージェント経路、OWS サーバー側)
エージェント設定:      1 件の API キーが登録済み
```

**`kova wallet info`** — アカウントのアドレスを表示します。EVM チェーンのアドレスは共通なので、1 つのアドレスがすべてのチェーン（Ethereum / Polygon / Amoy など）で使えます。

```
wallet :
  id        : b9367bba-b671-46cf-8860-277eb420f9c1
  name      : default
  accounts  :
    - [0]
        chainId        : eip155:1
        address        : 0x6465f5801a1056223dAA4c86F33B85347174dF53
        derivationPath : m/44'/60'/0'/0/0
  signer    :
    kind : passphrase
```

**`kova balance`** — 残高サマリを表示します。まだ入金していないので、この時点では保有なしです。

`kova status` の出力で注目したいのは `標準入力` の行です。ここではターミナルから実行しているので `TTY (対話ターミナル)` と表示されています。

Kovaは「エージェントモード」のようなフラグを持ちません。**ターミナル（TTY）かどうか**と、**どの認証情報を使うか（ローカルのパスフレーズか、発行済みの API キーか）** で、実行経路が自動的に決まります。エージェント経路（非TTY）の場合は、実行できるコマンドを限定しています。

### Step 4: ウォレットアドレスを共有する（ハンズオン参加者向け）

`kova wallet info` で表示された **アドレス**が、Amoy を含むすべての EVM チェーンであなたのエージェントの「財布」になります（EVM チェーンのアドレスは共通です）。

```
kova wallet info | grep address
# 例: 0x6465f5801a1056223dAA4c86F33B85347174dF53
```

表示された Amoy アドレスを、[配布用スプレッドシート](https://docs.google.com/spreadsheets/d/1eARSPL4fHbDtRL1FXbCkfl8xNdxNmL-H6qQPRwqUfYc/edit?gid=0#gid=0) に記入してください。主催者がテストネットの POL（ガス代）を送付します。

### Step 5: Claude Code から自然言語で操作する

Claude Code で Kova を利用しましょう。普段どおり `claude` を起動するだけです。

`kova init` の Step 3（AI接続設定）で内部的に `init-agent` が呼ばれ、エージェントキー（認証情報）が `~/.kova/config.json` のプロファイルに保存されています。`kova` は実行時にこのプロファイルから認証情報を読むので、どの端末で起動した Claude Code からでもそのまま使えます。あとは Claude Code のセッション内で、自然言語で頼むだけです。

`check-balance` スキル経由で `kova balance` が走ります。

```
Polygon Amoy で 0.00001 POL を 0xBaA9C2078a6c8CB36F6Ac0F049F6b1AE71680b31 に送って
```

`send-tokens` スキル経由で、まず `kova send` のシミュレーション（dry-run）で送金できるかをチェックし、内容を確認したうえで実際に送金（broadcast）します。Step 3 で配布された Amoy POL がガス代になります。成功すると Amoy 上に実トランザクションが残ります。

```
txHash: 0xd498194f1ec8fabb839f30878236eabdb1c4a822210e1057f07bd34fe65365f8
https://amoy.polygonscan.com/tx/0xd498194f1ec8fabb839f30878236eabdb1c4a822210e1057f07bd34fe65365f8
```

kovaを利用することで、AIエージェント主体での経済活動がなめらかに実行できることを体験できました。

## 6. ハンズオン②：危険操作が「構造的に」弾かれる

わざと危険な操作を頼んで、ブロックされる様子を見ます。

### ブロック①：owner-only

Claude Code のセッション内で、いかにもプロンプトインジェクションっぽく頼んでみます。

ポリシー削除やキー更新のような **owner-only 操作は、非TTY（エージェント）経路では構造的にブロック**されます。裏では以下のようなコマンド実行・出力がなされています。

```
PID=$(kova policy list | jq -r '.data.policies[0].id')
kova policy remove --id "$PID" < /dev/null
```

```
{
  "ok": false,
  "error": {
    "code": "INTERACTIVE_INPUT_REQUIRED",
    "message": "This operation requires a terminal ... owner-only — passphrase prompt cannot open in non-TTY (agent runtime)"
  }
}
```

「ポリシーを消して」と命じられても、エージェント経路では通りません。重要操作は人間がターミナル操作でしか実行できません。

### ブロック②：ポリシー

送金上限ポリシーを追加して、違反させてみましょう。

まずターミナルで、送金上限のポリシーを追加します。`policy update` は owner-only 操作なので、ターミナル（TTY）での実行が必要です。

```
# ポリシー ID を確認
kova policy list

# 対象ポリシーを編集
kova policy update --id <policy-id>
```

対話で次のように進めます。ここでは Amoy の native トークン（POL）に **日次上限 0.00002** を設定します。トークンは native 送金にマッチさせるため `native` を指定します。

```
Confirm: execute owner-only operation "policy-update"? (y/n): y

◇  どの操作を行いますか？
│  rule を追加
◇  どの rule type を追加しますか？
│  spending_limit (1 日 / 1 ヶ月の送金上限)
◇  チェーン名 (例: base, polygon, arbitrum)
│  polygon-amoy
◇  トークンシンボル (例: USDC, USDT, JPYC、または "native")
│  native
◇  日次上限額 (例: "100" = 100 USDC)
│  0.00002
◇  月次上限 (任意、空のまま Enter でスキップ)
│
◇  どの操作を行いますか？
│  完了 — 変更を適用
◇  この変更を適用しますか？
│  Yes

◆  ポリシーを更新しました。
```

`spending_limit` は **その日の累積**で判定されます。Step 5 ですでに 0.00001 POL を送金しているので、本日の残りは 0.00002 − 0.00001 = **0.00001 POL** です。

まず、残りを超える送金を頼みます。

```
Polygon Amoy で 0.000011 POL を 0xBaA9C2078a6c8CB36F6Ac0F049F6b1AE71680b31 に送って
```

本日の累積が 0.00001 + 0.000011 = 0.000021 となり、上限 0.00002 を超えるため `POLICY_DENIED` で弾かれます。

```
{
  "ok": false,
  "error": {
    "code": "POLICY_DENIED",
    "message": "spending_limit (daily) exceeded for eip155:80002:NATIVE: already spent 0.00001 + this tx 0.000011 = 0.000021 exceeds limit 0.00002. Resets at 2026-06-22T00:00:00.000Z (UTC).",
    "details": { "reason": "spending_limit_exceeded", "effect": "deny" }
  },
  "meta": { "mode": "agent", "authMethod": "api-key" }
}
```

次に、残りちょうどの送金を頼みます。

```
Polygon Amoy で 0.00001 POL を 0xBaA9C2078a6c8CB36F6Ac0F049F6b1AE71680b31 に送って
```

累積がちょうど上限の 0.00002 に収まるため、これは**通ります**。

```
txHash: 0x32d02c7fff5384bfa60aa643db7a6aa9d5bb41bace6fb7908c035477dbd76deb
https://amoy.polygonscan.com/tx/0x32d02c7fff5384bfa60aa643db7a6aa9d5bb41bace6fb7908c035477dbd76deb
```

上限は UTC 0 時にリセットされます。

AIは、自分でこのポリシーを緩めることはできません。`policy update` は owner-only なので、AI（エージェント経路）から実行しようとしても `INTERACTIVE_INPUT_REQUIRED` でブロックされます。上限を超える送金は弾かれ、そのポリシー自体もエージェントからは変更できない、という構造になっています。

### 仕組み：3つの OperationKind

Kovaは操作を3つに分類し、実行経路と認証手段で自動的に絞ります。

| 種別 | 例 | agent（非TTY） | terminal（TTY） |
| --- | --- | --- | --- |
| read-only | `balance` / `wallet info` / `status` | ✅ そのまま | ✅ そのまま |
| policy-gated | `send` / `sign` / `call` | ✅ APIキー + ポリシーで gate | ✅ パスフレーズで署名 |
| owner-only | `policy create/update/remove` / `key rotate` / `config set` | ❌ 構造ブロック | ✅ パスフレーズ + 確認 |

防御は実行経路とポリシーの二段構えです。

## 7. ハンズオン③：JPYC ECで買い物する

最後に、もっと具体的なユースケースとして、**AIエージェントが JPYC のECサイトで実際に商品を買う**ことを Polygon Amoy で体験します。

ECスキルは、JPYC・x402 を使ったエージェントコマース用のスキルです。

### 仕組み

* **x402**: HTTP の `402 Payment Required` を決済に使うプロトコル。「商品を頼む → サーバが402と支払い要求を返す → 署名して再送 → 注文成立」という流れ
* **JPYC**: 円ステーブルコイン（1 JPYC ≒ 1円）

### セットアップ：EC スキルと JPYC を用意する

EC の操作はすべて Claude Code 経由で行います。

* **EC スキル**: Step 2 の `kova init`（拡張機能のインストール）で `JPYC EC` を選んでいれば導入済みです。後から入れる場合は `kova skills install jpyc-ec`
* **テスト JPYC（Amoy）**: 購入には JPYC が必要です。Step 4 で登録したアドレスに、主催者がテスト JPYC を配布済みです（TEST\_TEE は 5 JPYC）。`kova balance` で着金を確認できます

### AI に商品を探させる・買わせる

あとは Claude Code に自然言語で頼むだけです。商品の検索から購入まで、AI が EC スキル経由で実行します。

```
JPYC EC（staging / Polygon Amoy）で買えるショップと商品を一覧して
```

AI は staging の `/shops` を叩いて一覧を返します。たとえば agent ショップ（Komlock Shop）には `TEST_TEE`（5 JPYC）などがあります。

```
agent ショップの TEST_TEE を Polygon Amoy（preferred_chain_id 80002）で1個買って。
配送先は適当なテスト住所でいい。
まず 402 の支払い要求（金額）を見せてから、署名して注文を確定して。
```

AI は次の流れを実行します。

1. **402 チャレンジ**: 署名なしで `POST /checkout` → 支払い要求が返る

   ```
   商品           : TEST_TEE × 1
   小計           : 5 JPYC
   支払先         : 0x3569c94e4175eb7fa9f9566a9701c23f1911f639
   チェーン       : eip155:80002 (Amoy)
   reservation_id : res_f139fce9598461827065757dfde2e58b
   ```
2. **Kova が支払いに署名**: 支払いの承認を Kova が裏で署名します。利用者は秘密鍵を扱う必要がありません
3. **settle（注文確定）**: 署名を載せて `POST /checkout` → 注文成立。試しに実行してみた結果がこちらです。

   ```
   注文番号 : ORD-20260621-HNT4M6
   order_id : c9537c5e-ef86-42d2-9c47-808dc3746374
   支払額   : 5 JPYC（実送金）
   payer    : 0x6465f5801a1056223dAA4c86F33B85347174dF53
   tx_hash  : 0xe13f9659ead6aac9c69219d1d3d19bf58c2a65a3eaa4204baa621aa68c5f3bfa
   network  : Polygon Amoy (eip155:80002)
   ```

   <https://amoy.polygonscan.com/tx/0xe13f9659ead6aac9c69219d1d3d19bf58c2a65a3eaa4204baa621aa68c5f3bfa>

「AIが買い物を完結させる」という流れを確認できました。今後、こうした決済フローが使われる場面は増えていきそうです。

## まとめ

「AIにお金を持たせる」というのは怖いですが、Kovaは **read-only は自由 / AIエージェントの操作はポリシーで制限する / 重要な操作は人間だけ** という明快な境界で、構造的にリスクを抑えています。

過度な期待は禁物ですが、エージェントが自分の判断でモノを買う世界がもう手元に来ているという事実にワクワクしますね。今後もAIエージェントの経済活動に注目していきましょう！
