## 概要

MastraのAIエンジニアAlexが、エージェントに記憶を持たせるための4つのアプローチ——「会話履歴」「ワーキングメモリ」「セマンティックリコール」「オブザーベーショナルメモリ」——を体系的に解説した動画です。各レベルがどのように機能するか、何を解決しどんな限界を持つかを丁寧に説明しながら、業界最難関のメモリベンチマーク「LongMemEval」での実測スコアも交えて比較しています。最終的に紹介される「オブザーベーショナルメモリ」は人間の記憶・忘却の仕組みに着想を得た最新システムで、同ベンチマークで最高スコアを記録。Mastraのオープンソースコーディングエージェントにも実装されており、長時間タスクをこなすAIエージェント開発の現場での実用性も実証されています。

## 要点

1. LLMはデフォルトでは「金魚」並みの記憶力しか持たない——会話履歴のみでは、スレッドをまたいだ記憶は一切引き継がれない
2. 会話履歴はシンプルで効果的だが、コンテキストウィンドウの上限・コンテキストドリフトという根本的な制約を抱える
3. ワーキングメモリは事前定義フィールドへの値充填でメモリを補完し、会話が長くなってもキーとなる情報を保持し続ける
4. セマンティックリコールはベクターDBへの意味検索でスレッドをまたいだ想起を実現するが、プロンプトキャッシュが効かないというコスト面の課題もある
5. オブザーベーショナルメモリは「オブザーバー」と「リフレクター」の2つのアンビエントエージェントで会話を動的に圧縮する
6. LongMemEvalベンチマーク：ワーキングメモリ55% → セマンティックリコール80% → オブザーベーショナルメモリ84%（Gemini 2.5 Flashでは95%）
7. 観察（Observations）は安定しているためキャッシュが有効になり、本番環境でのトークンコスト削減に直結する
8. Claude Codeのコンパクションと異なり、オブザーベーショナルメモリはバックグラウンド非同期で動くためユーザー体験を中断しない
9. ノイズの多いツール呼び出し結果の圧縮（10K→160トークン、63倍削減）が顕著な効果を発揮する
10. 実際のユースケース（ワークショップ説明文エージェント）では、数週間の使用でユーザーの文体・好みを自動学習していく

## 詳細

### LLMは「金魚」——記憶問題の出発点

動画の冒頭でAlexは、LLMの記憶問題をユーモアを交えて提示します。「自転車が好き」と書いてから事実を聞くと、まるで映画『メメント』の記憶を持たない主人公と話しているようだ、と。

このたとえは単なるジョークではなく、エージェント開発の核心を突いています。デフォルト状態のLLMはステートレスです。リクエストのたびにゼロから始まる。会話履歴を「渡す」ことで擬似的な記憶を与えることはできますが、それがどこまで通用するか、という話がこの動画全体のテーマです。

### レベル1：会話履歴（Conversation History）

最も基本的なアプローチ。LLMへのリクエストに過去のメッセージリストを含めることで、「前の文脈を知っている」状態を作り出します。

仕組みとしてはシンプルで直感的です。「自転車が好き」と書いた後に「それについて教えて」と送れば、モデルはリストを見て「自転車の話だ」と理解できます。

**ただし、限界は明確です。**

* **コンテキストドリフト**：インプットに詰め込む量が増えるほど、モデルの注意が分散し、応答の質が落ちる
* **ウィンドウの上限**：特にツール呼び出しが多い場合、想定より早くコンテキストウィンドウ上限に達する
* **スレッドをまたげない**：新しいスレッドを開いた瞬間、すべての記憶はリセットされる

Alexは「最初のメッセージにユーザーの目標があって、その後30メッセージが続いたら？ウィンドウが下にずれてしまう」と語ります。直近の10〜30メッセージだけを送るという実用的な対応策はあっても、それ自体が情報の喪失を意味します。

### レベル2：ワーキングメモリ（Working Memory）

会話履歴の「補完」として機能するのがワーキングメモリです。事前に定義されたフィールド（値はなし）をエージェントに持たせ、会話の中でそのフィールドを埋めるべき情報を探し出すよう促します。

例えば「1000万ドルのSaaSを作りたい、ミスは嫌いだ」とエージェントに伝えると、次のフィールドが自動的に更新されます。

```
goal: $10M SaaSの構築
preference: ミスを避けること
```

仕組みを少し覗くと、メモリシステムはシステムインストラクション（ユーザーには見えないがモデルには見える）を拡張します。テンプレートと現在の値が渡され、`update working memory` というツールでいつでも更新できることがモデルに伝えられます。

**向いているケース：**

* コーディングエージェントや長時間タスクを扱うエージェント
* 名前や連絡先のような変わらない事実の記録
* 「簡潔な返答が好き」といった安定したユーザー設定の保持

**限界：**

* フィールドの事前定義が必要（あまりエージェントらしくない）
* 大量のデータや長期的な記憶には不向き
* LongMemEvalスコア：**55%**（ただし元々長期記憶向けには設計されていない）

### レベル3：セマンティックリコール（Semantic Recall）

「意味」で過去を検索し、コンテキストウィンドウに動的に注入するアプローチです。従来のRAG（検索拡張生成）の考え方をエージェントメモリに応用したものと言えます。

**動作の流れ：**

1. 送信されたメッセージをエンベッディングに変換してベクターDBに保存
2. 次のメッセージ送信時、同様にエンベッディング化してDBを検索
3. 意味的に関連するメッセージを取得し、コンテキストウィンドウに追加
4. モデルへの入力に「これらは別のスレッドから想起されたメッセージです」という形で注入

このおかげで、「犬が好きで、Nas Barkleyという名前だ」とAに伝えれば、別スレッドBで「私の犬の名前は？」と聞いても「Nas Barkley」と返ってきます。

また、「犬について教えて」という直接的な検索だけでなく、「どんな動物が好き？」という意味的な問いでも正しく関連情報を取得できるのが特徴です。

**調整が必要なパラメーター：**

* **Top K**：上位何件の結果を返すか（例：3件）
* **メッセージレンジ（Message Range）**：周辺メッセージをどれだけ含めるか

**メリット：**

* スレッドをまたいだリコールが可能
* 長い履歴にもスケールし、関連情報だけを選択的に取り込める
* フィールドの事前定義が不要で、より自動的

**デメリット：**

* エンベッダーとベクターストアの導入が必要（コスト・レイテンシの増加）
* **クエリごとにシステムインストラクションが変化するため、プロンプトキャッシュが効かない**——本番環境では最大のコスト節約源を失うことを意味する
* リコールの精度が不確実な場合があり、Top Kやメッセージレンジの丁寧な調整が必要

LongMemEvalスコア：**80%**

### レベル4：オブザーベーショナルメモリ（Observational Memory）

動画が最も時間をかけて解説する、最高レベルのメモリシステムです。「人間はどのように覚え、どのように忘れるか」に着想を得ています。

**人間の記憶との類比**

人間の脳は過去に見聞きしたすべての情報を等価に処理しているわけではありません。目標達成に必要な重要部分に集中し、時間が経つにつれて不要になった情報は自然に忘れていきます。オブザーベーショナルメモリはこのプロセスを2つのアンビエントエージェントで実装します。

**2つのアンビエントエージェント**

1. **オブザーバー（Observer）**：会話履歴が一定のトークン数（しきい値）を超えると起動し、メッセージ履歴を「観察（Observations）」の密なリストに圧縮する。各観察には優先度インジケーターと日時アノテーションが付く。
2. **リフレクター（Reflector）**：観察リスト自体も大きくなると起動し、複数の観察ブロックをさらに凝縮した形にまとめる。緑の絵文字（重要度低）の観察は削除され、ユーザー設定は1行にまとめられる。

**具体的なデモの結果**

動画中、Mastraのllms.txtファイル（非常に大きいファイル）を取得するツール呼び出しを行うと、レスポンスに10.1Kトークンが含まれ、2Kトークンのしきい値を大幅に超えました。オブザーバーが起動し、この10K超のトークンを160トークンに圧縮——**63倍の削減**です。保持されたのは「ユーザーの名前」と「目的のドキュメントへのリンク」のみ。エージェントが実際に必要としていたのはその2点だけでした。

旅行計画エージェントのデモでは、3.1Kトークンの会話履歴がオブザーバーにより480トークンまで圧縮されました。

**セマンティックリコールとの決定的な違い**

| 観点 | セマンティックリコール | オブザーベーショナルメモリ |
| --- | --- | --- |
| システムインストラクションの変化 | クエリのたびに変化 | 観察は安定している |
| プロンプトキャッシュ | 効かない | **有効**（コスト削減の恩恵を受けられる） |
| メモリの保存形式 | ベクターDB内の元メッセージ | 圧縮された観察リスト |
| ノイズの多いツール呼び出し | 全文がそのまま蓄積される | 圧縮時に重要部分だけ抽出 |

**Claude Codeとの比較**

Claude Codeにも類似した「コンパクション」機能がありますが、大きな違いがあります。Claude Codeのコンパクションは同期的・ブロッキングで発生し、重要な情報が失われる可能性があり、ユーザーは1分以上待たされることがあります。

オブザーベーショナルメモリは**バックグラウンドで非同期・ノンブロッキング**で動作します。実際のデモでは同期的に動いていましたが、これは説明のためです。実用環境ではユーザーの体験がほとんど中断されません。

**LongMemEvalスコアとベンチマーク比較**

| メモリシステム | スコア |
| --- | --- |
| ワーキングメモリ | 55% |
| セマンティックリコール | 80% |
| オブザーベーショナルメモリ | 84%（同モデル比較） |
| オブザーベーショナルメモリ + Gemini 2.5 Flash | **95%**（検証可能な最高記録） |

### 実際のユースケース：成長するエージェント

Alexは自身が実際に使っているワークショップヘルパーエージェントを紹介します。Mastraでは毎週木曜日にワークショップを開催しており、このエージェントはLumaページの説明文を書いてくれます。オブザーベーショナルメモリを有効にしたまま数週間使い続けた結果、エージェントは次のようなことを学習していました。

* 説明文はすべて二人称で書く
* フック（最初の一文）をとても重視している
* 簡潔さを好む
* 「Mastra Code」のリンクは `code.mastra.ai`

「エージェントが私と一緒に成長し続けている」とAlexが語るように、他のメモリシステムでは同じレベルのキャリブレーションとシンプルさを両立させることはできません。

### Mastra Code：コーディングエージェントへの実装

このメモリシステムの起源は、MastraのエンジニアTyler Barnesが実際のコーディングエージェント開発の現場で「メモリこそが長い時間軸で動く高性能なディープエージェントの最大の鍵」と気づいたことにあります。現在、Mastraのオープンソースコーディングエージェントハーネス「Mastra Code」はオブザーベーショナルメモリを採用しており、長時間にわたるコーディングタスクを扱う際の記憶管理に活用されています。
