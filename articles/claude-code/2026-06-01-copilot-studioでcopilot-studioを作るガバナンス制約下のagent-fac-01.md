## 「エージェントを使いこなす人」と「作れない人」の間にある壁

<https://www.youtube.com/watch?v=9HoZ9IcVv-4>

AIエージェント元年と言われて久しい。Copilot Studioも機能が急速に充実してきた。

でも現場の担当者はこう言う。

> 「作りたいんだけど、どこから始めればいいか分からない」  
> 「Claude Codeで作ったサンプルを見せてもらったけど、うちの社内では使えない」  
> 「Copilot Studioは触ったことがある。でも白紙から作るのは無理だ」

この「壁」の正体は、スキルだけじゃない。**ガバナンスとセキュリティ**だ。

```
❌ 外部LLM API            → セキュリティ審査NG
❌ Python / Node.js      → ローカル環境への導入不可
❌ Claude Code           → 社内未認可ツール
❌ GitHub Copilot        → ライセンス未取得

✅ Copilot Studio        → 使えている
✅ Power Automate        → 使えている
✅ Teams                 → 使えている
```

「使える範囲で、どこまでできるか」——この問いへの答えとして作ったのが **Agent Factory（エージェントファクトリー）** だ。

---

## Agent Factory とは何か

一言で言えば、**「作りたいエージェントを自然言語で伝えると、Copilot Studioにインポートできるファイル一式が自動生成される」** システムだ。

ユーザーが行う操作はこれだけ：

```
Step 1. Teams上でエージェントに話しかける
        「議事録を自動で要約して、承認フローに流すエージェントを作りたい」

Step 2. エージェントと対話する（名前・機能・トピックをヒアリングされる）

Step 3. Google Drive にフォルダが生成される（〜25ファイル）

Step 4. ファイルを全選択 → ダウンロード → Copilot Studio にインポート
```

## なぜ「あえてCopilot Studioだけ」なのか

### ポジショニング

世の中のエージェント自動生成アプローチは大きく3つに分類できる。

| アプローチ | 品質 | ガバナンス適合 | 学習コスト | 誰が使えるか |
| --- | --- | --- | --- | --- |
| **最先端型**（Claude Code / GPT-4 API等） | ⭐⭐⭐ | ❌ 多くの企業でNG | 高 | 開発者のみ |
| **標準機能型**（Microsoft自動作成系） | ⭐ | ✅ | 低 | 誰でも |
| **Agent Factory** 👈 | ⭐⭐ | ✅ | 低 | 誰でも |

最高品質を諦める代わりに得たもの：**ガバナンスの壁を越えずに、現場担当者が自分でエージェントを作れる状態**。

### 「最初の1本」の価値

完璧なものを最初から作る必要はない。**動くものが手元にある状態**から始められることが、定着の第一歩になる。

## 設計の核心：なぜこの構造になったか

### ① エージェントを2つに分けた理由

最初は**1エージェントで全部やろうとした**。結論から言うと、動作が安定しなかった。

「対話」と「生成」は、求められる性質が根本的に違う。

同じコンテキストに混在させると：

* 対話の「柔軟さ」が生成の精度を下げる
* 生成の「厳格さ」が対話の自然さを壊す

**コンテキスト・セッションを分離することで、それぞれに最適化できた。**

### ② 処理を4段階に刻んだ理由

**一括処理にしなかった理由：**

各ステップのインプット・アウトプット形式が異なる。

| ステップ | インプット | アウトプット | なぜ分けるか |
| --- | --- | --- | --- |
| 対話→要件 | 自然言語（曖昧） | 構造化テキスト | 曖昧さを排除するため |
| 要件→設計 | 構造化テキスト | JSON | 機械処理に適した形式へ |
| 設計→ファイル | JSON | ファイル群 | RPAとして自動化するため |

JSON解析を挟むことで「AIの回答」ではなく「構造化データ」をフローに渡せるようになり、**ファイル生成が安定した自動処理になった。**

## プロンプト設計の詳細

### 本体エージェントのプロンプト構成

| # | セクション | 内容 |
| --- | --- | --- |
| 1 | **役割定義** | Agent Factoryとしての振る舞いを定義 |
| 2 | **動作フロー** ⭐ | どのタイミングで何をするかを明示。「ヒアリング完了後にツールAを呼ぶ」など |
| 3 | **ヒアリング項目** | 基本項目＋ツール関連項目。1〜2項目ずつ進める指示あり |
| 4 | **ツールの使い方** ⭐ | 仕様生成ツールとファイル生成ツールの使い分けを明示 |
| 5 | **レビュー観点** | JSON受領後の項目チェックリスト。不足→エラー防止の二重チェック |
| 6 | **完了後の案内** | README・ガイドの存在アナウンス／Google Driveの操作手順 |
| 7 | **重要原則** | ヒアリングの進め方・禁止事項等 |

### 仕様生成エージェントのプロンプト構成

こちらは**シンプルを徹底**した。

| # | セクション | 内容 |
| --- | --- | --- |
| 1 | **役割（最小限）** | 最低限の役割定義のみ。余計な指示を入れない |
| 2 | **出力JSONの構造定義** ⭐ | トピック（2〜5個）とツール（最大3個）それぞれのタグ・値を詳細に定義 |
| 3 | **生成ルール（厳格に）** | 1文字ズレるとインポートエラーになるため、ルールとして明示 |
| 4 | **自己検証ルール** | 生成後にJSON整合性を自己チェックする手順を組み込む |

### エージェント間の橋渡し：Few-shot in ディスクリプション

本体エージェント → 仕様生成エージェントへの情報の渡し方は、**ツールのディスクリプション（説明欄）にFew-shotプロンプトを入れる**という方法をとっている。

---

## ツール部分はモック——「あえて全自動にしない」理由

Copilot Studioのエージェントだけで完結できる範囲と、できない範囲がある。

**完全自動化しないメリット：**

| 観点 | 内容 |
| --- | --- |
| **ガバナンス** | 外部APIや未承認ツールに触れずに済む |
| **学習効果** | ツール差し替えを自分でやることで理解が深まる |
| **柔軟性** | モックを見れば「ここを差し替えればいい」が分かる |
| **到達可能性** | 完全自動を目指さないことで、確実に1本目が渡せる |

## 試行錯誤の軌跡——今の構造になるまで

最初のアーキテクチャは今とは全然違った。

| バージョン | 結果 | 変更内容・判明したこと |
| --- | --- | --- |
| v1 — 1エージェント全部入り | ❌ 失敗 | 対話と生成が同じコンテキストに混在 → AIが「できないことを無理やりやろうとする」現象が頻発。生成ファイルがインポートエラーになり続けた |
| v2 — エージェント分離 | 🔄 改善 | 対話と生成を分離 → 対話は安定。ただし生成物のインポートエラーが引き続き頻発 |
| v3 — 4段階パイプライン化 | 🔄 改善 | JSON解析を挟んでファイル生成をRPA化 → 安定。論理→物理の橋渡しをFew-shotで定義し再現性が向上 |
| v4 — プロンプト精緻化 | ✅ 完成 | 二重チェック構造を確立。並列処理で生成速度も向上。現在の安定版 |

具体的に何がうまくいかなかったか（詳細）

**v1の問題：** 1エージェントで全部やろうとした結果、AIが「できないことを無理やりやろうとする」現象が発生。生成されたファイルがインポートエラーになることが多発。

**v2の問題：** エージェントを分けたことで対話は安定したが、生成物の品質がまだ不安定。特に「論理設計（人が読みやすい形）」の結果をそのまま仕様生成エージェントに渡すと、出力がブレた。

**v3で解決：** 橋渡し部分にFew-shotを入れ、「どんな形式で渡すと安定するか」を定義したことで再現性が大幅に向上。JSON解析を挟むことでフロー側の処理が安定。

**品質の源泉：** 試行錯誤の回数。プロンプト・フロー構造・アーキテクチャ全てを変えながら繰り返した結果が今の状態。

## この設計で届けたいもの

> **「誰も置いていかない」**

最先端のAIツールを使いこなせる人だけが恩恵を受ける世界は、正直面白くない。

セキュリティ審査が通っていない。予算がない。スキルがない。そういう「壁」の内側にいる人たちも、AIエージェントを自分で作れる状態を作りたい。

弊社が積み上げてきた「お客様が自分で使いこなせる状態を作る」という考え方が、この設計の根幹にある。

Claude Codeで作ればもっと高品質なものが作れる。それは知っている。でも「そうじゃない」という選択に意味がある。
