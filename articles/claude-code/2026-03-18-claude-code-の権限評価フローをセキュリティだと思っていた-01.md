Claude Code を導入した翌日、チームのセキュリティ担当からこう聞かれた。「エージェントが外部に通信しないことは保証されているか？」あなたは `deny` ルールに `WebFetch` を追加して「対処済み」と答えた。それは本当に正しかったのか。

---

## 「権限評価フロー」という名前が誤解を生む

Claude Code のドキュメントには、ツール呼び出し時の評価順序が示されている。

```
PreToolUse Hook → Deny Rules → Allow Rules → Ask Rules → Permission Mode → canUseTool Callback
```

多くの解説記事はこれを「ガバナンス設計の中核」として紹介する。deny ルールを書けばエージェントを制御できる、と。

しかしこの理解は構造的に誤っている。

---

Claude Code の実体から考える。

LLM は Anthropic のサーバー上で動いており、ローカルの OS に直接触れない。テキストを生成して HTTP レスポンスとして返すだけだ。ローカルのファイルシステムやネットワークにアクセスする手段を持たない。

`tool_use` ブロックはその返答の一形式に過ぎない。「これを実行してくれ」という要求を SDK に返しているだけで、実行するかどうかは SDK が決める。

```
Anthropic サーバー（LLM）
    ↓ HTTP レスポンス（tool_use ブロックを含む）
Claude Code SDK
    ↓ tool_use をパース
権限評価フロー ← 実行前にここで判定する
    ↓
ローカル OS
```

権限評価フローは tool\_use ブロックの実行前に判定する。それ以外のタイミングでは動かない。

つまり `WebFetch` を deny しても、Bash が許可されていれば何も変わらない。

```
# WebFetch は deny されている
# しかし Bash 経由なら通る
curl https://external.example.com
python -c "import urllib.request; urllib.request.urlopen('https://external.example.com')"
wget https://external.example.com
```

Bash を許可した瞬間、OS の能力をそのまま渡している。tool\_use フィルタはその内側で何が起きているかを見ていない。

---

## 決定論システムの思考パターンがLLMに誤適用される

決定論システムに慣れたエンジニアは、こう考える。「このルールを設定すれば、この動作が保証される」。

ガードも同じ発想で設計される。`deny` ルールを書けば、その操作は止まる。設定したから安全だ、と。

しかし LLM は確率的なテキスト生成器だ。tool\_use ブロックを出力するかどうかも確率的であり、Bash を通じて何をするかは SDK から見えない。「設定したから保証される」という感覚は、LLM を組み込んだシステムでは成立しない。

さらに根本的な問題がある。LLM は deny されたとき、別の手段を試みる。

```
WebFetch を deny される
→ 「では curl を使おう」と Bash で試みる
→ Bash も deny される
→ 「では Python の urllib で」と試みる
→ それも deny される
→ 「では Node.js の fetch で」と試みる
```

決定論システムなら「deny されたら止まる」が保証される。LLM は目的を達成しようとする性質上、一手を止めても別経路を確率的に生成する。deny ルールは「その一手を止める」だけであって、エージェントの行動全体を制約しない。

権限評価フローはその名前から「アクセス制御システム」に見える。しかし設計者の意図は別のところにある。

エージェントが自律的にツールを呼ぶとき、人間がどこで介入するかを調整したい。

```
人間が全操作を承認 → 生産性が死ぬ
全部自動実行       → 不安で使えない
```

その中間を調整するための仕組みだ。これは **ヒューマンインザループの UX 設計**であって、セキュリティ制御ではない。

`bypassPermissions` を使う場合は「隔離環境専用」とドキュメントが繰り返し警告しているのも、設計者がこの区別を認識しているからだ。アプリレイヤーのフィルタでセキュリティ境界を作れるとは言っていない。

---

## ガードの実体はプロセスレベルに置く

本当の意味でエージェントの通信やファイルアクセスを縛りたいなら、制御層をプロセスレベルまで下げる必要がある。

| 層 | 制御の根拠 | Bash経由の迂回 |
| --- | --- | --- |
| tool\_use フィルタ | LLMの出力形式に依存 | 可能 |
| プロセスレベル | OS が強制 | 不可能 |

具体的な手段はこうなる。

* **seccomp** — ネットワーク系システムコールを禁止すれば、何を経由しても外に出られない
* **eBPF** — プロセス単位・方向・宛先まで細粒度で制御
* **コンテナのネットワークポリシー** — 許可した宛先以外への通信を OS が遮断
* **AppArmor / SELinux** — ファイルシステムアクセスをプロセス単位で強制

権限評価フローとサンドボックスは代替関係にない。役割が違う。

```
権限評価フロー  → 人間がどこで介入するかの調整（UX）
プロセスサンドボックス → エージェントが何をできるかの強制（セキュリティ）
```

両方必要であり、前者だけでガバナンスが完成したと考えることが危険だ。

---

## まとめ

Claude Code の権限評価フローは「誰がいつ承認するか」を制御する仕組みだ。セキュリティ境界ではない。

本番環境のガードが必要なら、コンテナやサンドボックスなどのインフラレベルで設計する。権限評価フローはそのうえに乗る利便性の調整機構として位置づける。

「設定したから安全」という感覚は、LLM を組み込んだシステムでは捨てる必要がある。deny ルールは一手を止めるだけで、目的達成を試みる LLM の性質は止まらない。セキュリティ境界はプロセスの外側に置く。
