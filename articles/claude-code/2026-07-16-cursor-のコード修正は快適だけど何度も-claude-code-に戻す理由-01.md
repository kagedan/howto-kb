Claude Code に月17ドル。最近は Cursor も月20ドル。2つのAIコーディングツールの課金が重なって「本当に両方必要か」と疑い始めた。試しに1週間、コード修正を全部 Cursor に任せてみた。快適だった。でも、週の途中で Claude Code に戻したくなる瞬間が来た。

## 1週間コード修正を Cursor に任せた

思い立ったのは6月初旬。自動投稿パイプラインのバグ修正や新機能追加が続いていて、毎日のコード修正くらい Cursor で十分では、と考え始めた。

エディタ画面の快適さと Tab 補完の速さは Claude Code (ターミナル経由) より直感的だし、何より課金の圧縮が目標だった。

試してみると、日常のコード修正ならほぼ全部 Cursor でこなせた。

* Python スクリプトのバグ修正
* 条件分岐の追加
* 設定ファイルの編集

細かい調整の連続は、Composer (複数ファイルへの指示) で一気に片付く。「これなら Cursor に一本化できるか」と思い始めた。

快適なのは確かだった。でも、1週間やってみるとある場面で「あ、ここは Claude Code じゃないと詰まるな」と気づいた。

## それでも戻した1つの作業

それは「**複数の独立した修正を並列で回す**」場面だった。

自動投稿パイプラインは毎朝1本の記事を投稿する一連の工程を無人で回す。記事を生成するエージェント、サムネイル生成のエージェント、完了メール送信……複数の機能が同時に動く設計になっていて、うち1つが失敗しても他の処理は止まらないようにしている。

この「複数のエージェントを並行管理・各自に指示を出す」という場面で、Cursor の Composer だと指示が単線になってしまう。1個のエージェント (コードブロック) にあれこれ指示してから、「では次はこっちのエージェントをお願い」と別ファイルに指示を出す。

直線的なタスク処理は得意だが、「同時に複数の並列タスクを管理してほしい」という要求には向いていないと感じた。

一方、Claude Code のターミナルは「同時に複数のサブプロセスを背景で回して、親は進行状況を見守る」という構造がしっくり来る。複数の修正指示を一度に Claude に投げて、ファイルはどれから何をするか判断させる。失敗を処理してリトライもできる。

この「重い並列作業の全体管理」は Claude Code の方が頭がいい。

1週間で感じたのはそれだけだった。日常のコード修正は Cursor で十分。でも「複数タスクを同時に回して監督する」という場面だけは Claude Code に戻したくなる。

## 今の安い/重いの使い分け

だから今は2つを分けて使っている。

**日常のコード修正は Cursor に任せる**。ファイルを開いて、「ここの条件分岐を変えてほしい」と指示をテキストで投げ、Composer で一気に直してもらう。これで十分だし、$20で快適。

**並列で重い作業は Claude Code に戻す**。複数のエージェント、複数のリトライロジック、同時実行の管理……こういう「全体を見守りながら制御する」仕事は Claude Code でないと落ち着かない。多少高くても、ここは無理やり安くしない。

単なる「安い方に置き換える」ではなく、自分の使い方に合わせて「何をどちらで回すか」を決めた。それが今のところの線引きだ。

AIツールを実際に触って試した試行錯誤を不定期で残してます。同じところで詰まってる人向けに書いてるので、また読んでくれたら嬉しい。
