## TL;DR

* 1人 + Claude Code で個人開発を回しているうちに、「Claude が書いて、Claude がレビューして、Claude が『OK』を出す」という構造の危うさにぶつかった。
* そこで Claude Code を **PdM・エンジニア・QA・リリース番** に役割分担させた“擬似開発会社”の運用OSを作った。Subagent と数枚の Markdown だけで動く。
* やってみて一番効いたのは「マルチエージェント協調」そのものより、**失敗を記憶でなく“台帳 → 機械ゲート”で封じる仕組み**と、**検査役と作成役を分離する**設計だった。
* この記事は、そこに至るまでに踏んだ地雷と、最終的に固まった設計判断の体験記です。

---

## きっかけ：1人 × Claude で踏んだ地雷

最初は普通に Claude Code で個人開発をしていた。Rails の求人マッチングアプリ（本番稼働中）で、ある日まとめてレビューをかけたら、こういうものが出てきた。

* **登録フォームから `user_type` を送れてしまい、権限が昇格しうる**（Strong Parameters の許可リスト漏れ）
* 日報まわりで **render が二重に走って 500**、しかも **GET に副作用**（レコード自動生成）が残っていた
* 求人一覧に **N+1**

どれも「動いてはいた」。テストも緑だった。怖かったのはそこで、**「テストが通っているから安全」で危うく見逃しかけた**こと。認可の欠陥は lint でも型チェックでも audit でも検出されない。テストの本数を見て安心していたが、中身は自動生成の雛形のままだったりした（テスト 44本あって、実動作を検証していたのはごく一部）。

ここで気づいた。

> **1人 + 1つの Claude だと、「実装する人」「レビューする人」「OKを出す人」が全部同じになる。**  
> 自分が書いたコードを自分でレビューして自分でマージする —— これは構造的に事故る。

会社なら、実装した人とは別の人がレビューし、リリース前にもう一段ゲートがある。その**役割分担の構造**を Claude Code の上に作れないか、というのが出発点だった。

---

## 作ったもの：3層の“擬似開発会社”

最終的にこういう構造になった。実体は **Subagent（`~/.claude/agents/*.md`）数枚 + 中央ルールの Markdown 2枚**だけ。

```
┌─────────────────────────────────────────────┐
│ Layer 1: 経営層（常駐）                       │
│   PdM兼テックリード ＝ メインの Claude         │
│     ── どのリポに時間を割くか／設計レビュー    │
│        ／リリースGO判断                        │
└─────────────────────────────────────────────┘
                     │
        ┌────────────┼────────────┐
        ▼            ▼            ▼
┌─────────────────────────────────────────────┐
│ Layer 2: リポジトリ別プロダクトチーム          │
│   各アプリを「フェーズ別モード」で編成         │
│   （P1着想 → P6運用 で起動メンバーが変わる）   │
└─────────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────┐
│ Layer 3: 共通Ops層（Subagent・オンデマンド）   │
│   web-engineer / qa-engineer /                │
│   release-guardian / idea-visionary …         │
│   → 全リポ共通で再利用                         │
└─────────────────────────────────────────────┘
```

ポイントは、**Layer 1（司令塔）はメインの Claude が兼任**していること。司令塔まで独立エージェントにすると対話のテンポが落ちるし、トークンも食う。**重い判断は司令塔が引き取り、単発で済む作業だけ下に降ろす**——この線引きが後で一番効いてくる。

ファイル構成はこれだけ。

```
~/アプリ開発/
├── AGENT_TEAM.md   … 組織図と“恒久ルール”の唯一の正本
└── INCIDENTS.md    … 盲点・障害の台帳（失敗の記録）

~/.claude/agents/
├── web-engineer.md      … 実装役
├── qa-engineer.md       … テスト作成・機能検証役
├── release-guardian.md  … リリース前の最終検査役（後述・聖域）
├── idea-visionary.md / idea-strategist.md … 企画役
└── …
```

ここから先は、**作りながら「これは知見だ」と思った設計判断**を、踏んだ地雷とセットで書いていく。

## 知見1：マルチエージェントは“基本オーバーキル”だった

最初、テンションが上がって何でもマルチエージェントで組もうとした。が、すぐに気づく。

> **マルチエージェント協調は、各メンバーが独立した Claude インスタンス＝トークンが 4〜5倍 飛ぶ。**

「実装して」「テスト書いて」「競合を分解して」みたいな**単発・並列・実行だけのタスク**にフル編成を組むのは、ただの浪費だった。そこで筆頭原則をこう決めた。

> **迷ったら単発 Subagent。**  
> マルチエージェント協調へ“昇格”するのは、**複数の役割の合意が要る局面だけ**。

昇格すべきか迷ったら、この3問。1つでも Yes なら昇格を検討する。

1. 複数の役割の**合意**が要るか？（実装だけ／テストだけなら No）
2. **取り返しのつかない判断**か？（技術選定・本番GOは Yes）
3. **複数リポをまたぐ**か？（1リポ内に閉じるなら No）

実際、日常の9割は「web-engineer 単発」「qa-engineer 単発」で回る。フル編成（4役以上同時）を起動するのは、**新規プロダクトの企画一巡**と**アーキ設計の合議**くらい。マルチエージェントの記事は「すごい」で終わりがちだけど、**実運用では“いつ組まないか”の判断の方が本質**だった。

## 知見2：役割をモデルティアで分ける（判断の重さ × 失敗コスト）

全部を最上位モデルで回すのは贅沢すぎるし、全部を軽量モデルにすると検査が雑になる。基準を **「判断の重さ × 失敗コスト」** に置いて、役割ごとにティアを振った。

| ティア | 用途 |
| --- | --- |
| 最上位 | アーキ合議・技術選定の**合議局面だけ**（回数上限つき） |
| 上位 | PdM兼テックリードの日常判断・難バグ調査・リリースGO準備 |
| 標準 | 実装・テスト・コードレビュー・企画 |
| 軽量 | 定型コミット・lint整形・依存更新・状態ファイル更新 |

設計したときの学びは2つ。

* **コスト削減の本丸は「手順実行系」**。定型作業を軽量ティアに落とすのが効く。
* 逆に、**品質を死守すべき検査役は絶対に軽量化しない**。ここでケチると、冒頭の権限昇格バグみたいな**不可逆・全体波及の事故**を見逃す。コスト最適化と品質死守は“別の場所”にある。

モデル指定は各 Subagent の `model:` フィールドに**エイリアスで**書く。新モデルが出たら**中央のティア表1枚だけ**更新すれば、全エージェントが追従する（モデル名を各所にベタ書きしない）。

## 知見3：状態を“中央ファイル”に書かない

最初、中央の `AGENT_TEAM.md` に「kokoro は今 P6、本番URLはこれ、未解決課題はこれ」と**状態のスナップショット**を書いていた。これがすぐ陳腐化した。フェーズが進むたびに中央を直すのを忘れ、**現実とドキュメントがズレる**。

そこで原則を分けた。

> **恒久ルール（組織設計・承認規定・QAゲート）は中央に1箇所。**  
> **状態（フェーズ・稼働URL・未解決課題）は各リポの `AGENTS.md` に。**

つまり、

* 「どう進めるか」のルール → `~/アプリ開発/AGENT_TEAM.md`（**唯一の正本**）
* 「今どこにいるか」の状態 → 各リポ `AGENTS.md` 末尾の `phase`（**状態の正本**・自動ロードもされる）

「どの変化でどのファイルを触るか」も固定した。**フェーズが進んだ → 各リポの `phase` だけ／新しいモデル → 中央のティア表だけ／新リポ追加 → 中央に1行**。横断ルールは中央1箇所、状態は各リポ内。これで不整合がほぼゼロになった。

## 知見4：失敗を“記憶”でなく“台帳 → 機械ゲート”で封じる

これが一番効いた仕組み。

冒頭の地雷（権限昇格・二重render・N+1・株式分割の補正漏れ…）を、**「次から気をつけよう」で終わらせない**。発見したら即、`INCIDENTS.md` に番号付きで1ブロック記録する。フォーマットはこう。

```
## INC-1: kokoro 登録時の権限昇格（認可欠陥）

- 発見日: 2026-06-24
- 症状: 登録フォームから user_type を送れ、権限が昇格しうる
- 原因: Strong Parameters の許可リスト漏れ。信頼できない入力をそのまま受理
- 誤対応しかけた点: 「テストが通っているから安全」で見落としかけた。
  認可の正しさは lint/型/audit では検出されない（設計上の欠陥）
- 恒久対策: 新規入力受理は許可リスト方式。認可分岐は許可/拒否の両方をテスト。
  さらに認可変更そのものは機械ゲートで緑にせず人間レビュー必須に
- 出典: コミット 8645913
- 昇格先: §8.3 → §8.2 G8 ＋ §8.1 A9
```

ポイントは末尾の **「昇格先」**。台帳に書いただけでは再発する。対策が**複数回再現して安定したら、コミット前の機械QAゲートに“昇格”させる**。

```
失敗を発見
  → INCIDENTS.md に記録（番号付き台帳）
  → 最低品質基準（§8.3）に落とす（測定可能な下限）
  → 安定したら QAゲート（§8.2）へ昇格（マージの絶対条件）
```

実際、最初は手元の心がけだった項目が、いまはコミット前ゲートになっている。

| ゲート | 合格基準 | 由来インシデント |
| --- | --- | --- |
| G8 | 認可分岐に許可/拒否**両方**のテスト | 権限昇格バグ |
| G9 | 一覧・関連ロードに **N+1なし**（eager load） | 求人一覧N+1 |
| G11 | render/commit が二重に走らない・**GET非冪等なし** | 日報二重render |
| 境界 | 丸め・分割・null・除算の**境界テスト** | 株式分割の補正漏れ |

> **記憶に頼らず、「番号付き台帳 + 機械ゲート」で再発を封じる。**

抽象的な教訓（「認可は大事」）を、**測定可能な基準（「許可/拒否の両方をテスト」）に翻訳して機械に持たせる**。これが、AIに任せても品質が落ちない一番の理由だった。

## 知見5：検査役と作成役を分ける（“聖域”を1つ作る）

知見4のゲートを誰が判定するか。ここで冒頭の構造問題に戻る。**自分が書いたテストで、自分が合格を出してはいけない。**

そこで `release-guardian`（リリースガーディアン）という役を新設した。チームで唯一、**マージと本番反映を止める権限**を持つ独立検査役。性格づけはこう。

* **read-only。実装もテストも一切書かない。** 合否とブロック理由だけ返す。
* 1項目でも不合格なら **フェイルクローズ**（止めて人間に報告）。
* モデルは **軽量化禁止（聖域）**。ここをケチると不可逆事故を見逃すから。

そして `qa-engineer`（テストを書く・走らせる＝**材料を用意する側**）と明確に分けた。

> **qa-engineer ＝ 作成役**（テストを書き、PASS/FAILの材料を用意する）  
> **release-guardian ＝ 検査役**（止める権限を持つ・自分では何も書かない）

自分でテストや実装を書く検査役は「自分が用意したものを自分で合格にする」**利益相反**になる。だから検査役には何も書かせない。テスト不足を見つけたら、合格にせず作成役に**差し戻す**。

会社の品質保証部門が「自分でコードを書かない」のと同じで、**検査の独立性は“書かないこと”で担保される**。

## 知見6：承認は“可逆性”で引く（git の有無に依存しない）

「人間が承認すべき所」をどう定義するか。最初は「ブランチ → PR → main マージ」を承認ゲートにしようとした。が、**自分のリポの大半は `.git` すら無い**プロトタイプだった。ブランチ前提は空振りする。

そこで承認軸を\*\*「可逆性」\*\*で引き直した。

* **必ず人間が承認**（不可逆・本番反映・課金・権限）: 本番デプロイ、破壊的migration、secret変更、**新規の入力受理・認可分岐の変更**
* **承認なしで進めてよい**（可逆・ローカル完結）: 実装・修正、ローカルのテスト、調査、状態ファイル更新

特に効いたのが **「認可・破壊的変更は機械ゲートで緑にしない」**（承認区分 A9）。G8「認可テストの存在」は**存在チェック止まり**で、テストがあっても認可“設計”が正しいとは限らない。だから**新規の認可分岐を見つけたら、機械PASSにせず人間レビューへ差し戻す**。

そして承認は **セッション/タスク単位**に限定した。「以降ずっと無確認でデプロイ」は不可。授権は毎回切る。

> 機械で測れるもの（ビルド・型・lint・N+1）は機械ゲートに。  
> 機械で測れないもの（認可の正しさ・破壊的変更の是非）は人間に。  
> **この線引きを“可逆性”で引く。**

## おまけ：エージェント定義は“七要素テンプレ”で揃える

各 Subagent の定義（役割カード）は、ブレないように要素を固定した。`release-guardian.md` の冒頭はこんな感じ。

```
---
name: release-guardian
description: マージ／本番反映の直前に通す「最後の安全装置（聖域）」。…
  起動シグナル：「リリース前チェック」「マージしていい?」…
  必ず対象リポジトリを引数で指定して起動。実装・テストは一切書かず、
  合否判定とブロックだけを行う。
model: sonnet
---

あなたは擬似開発会社の「リリースガーディアン」です。
このチームで唯一、マージと本番反映を止める権限を持つ独立検査役で…
```

定義に必ず入れる要素：①ミッション ②**起動シグナル**（どの言葉で呼ばれるか）③**対象リポを引数必須**（独立リポ群なので誤って別リポに書くのを防ぐ）④所有ファイル ⑤やってはいけないこと ⑥エスカレーション先 ⑦最初に読む正本。

特に③の「対象リポを引数で渡す」は地味だけど重要で、リポがバラバラに散らばっている個人開発では、**誤リポへの書き込み防止**の生命線になっている。

## 効果と、これから

数ヶ月運用してみての実感。

* **「とりあえず動く」で止まらなくなった**。コミット前ゲートが砦になり、認可・N+1・冪等性が標準で潰れる。
* 実際、上の地雷はすべて修正済み。テストも 44本 → 73本（0 fail）に育った。
* **判断の置き場所がはっきりした**。「これは機械ゲートで緑にしていい／これは人間が見る」が毎回ブレない。
* マルチエージェントは“ハレの日”用。**日常はほぼ単発 Subagent**で、コストも想定内。

これから足したいのは、設計専任役（app-architect）とデプロイ準備役（公開ボタンは押さない release-engineer）、状態維持の軽量役（repo-librarian）あたり。あと、非gitのプロト群を `git init` してインシデントの出典にコミットハッシュを残せるようにすること。

---

## まとめ

「マルチエージェントで開発」と聞くと派手な協調を想像するけど、1人 + Claude Code でやってみて効いたのは、もっと地味な設計判断だった。

1. **マルチエージェントは基本オーバーキル。迷ったら単発 Subagent。**
2. **役割をモデルティアで分ける**（判断の重さ × 失敗コスト）。
3. **状態を中央に書かない**（ルールは中央・状態は各リポ）。
4. **失敗を記憶でなく“台帳 → 機械ゲート”で封じる。**
5. **検査役と作成役を分ける**（止める権限を持つ聖域を1つ）。
6. **承認は“可逆性”で引く**（認可・破壊的変更は人間ゲート）。

本質は「AIに開発を任せる」ことではなく、**AIが書いたものを“別の役割のAI＋人間ゲート”でどう締めるか**だったと思う。Subagent と Markdown 数枚で組めるので、個人開発で AI に色々任せ始めている人の参考になれば。
