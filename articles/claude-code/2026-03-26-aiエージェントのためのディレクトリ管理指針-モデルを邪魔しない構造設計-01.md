2026年現在、AIコーディングエージェント（Claude Code、Cursor、Aiderなど）は急速に進化しています。しかし、エージェントが長時間稼働すると、**人間が書いたコード**だけでなく**モデル自身が生成したコード**がコンテキストを汚染し、推論の質を低下させる問題が顕在化しています。

本記事は、これまでの実践から導き出した\*\*「モデル中心主義」のディレクトリ管理指針\*\*を精密にまとめます。  
核心は「**薄く・隔離・積極的忘却**」です。エージェントの本質はモデルである以上、ディレクトリ構造はモデルの自由を最大化し、過去の生成物を邪魔にしないよう設計しなければなりません。

### 1. 基本原則（これだけは厳守）

1. **モデルに優しい薄さ**  
   コンテキストを圧迫しない。AGENTS.mdやrulesファイルは極力短く（推奨：30〜80行程度）。詳細は必要なときにのみ段階的に開示。
2. **生成物の隔離**  
   モデルの実験・一時生成コードは本体から完全に分離。成功した最小限の変更だけを本体にマージ。
3. **積極的忘却とリセット**  
   過去の生成コードを過度に尊重せず、現在の要件からシンプルに再生成するよう明示的に指示。不要ファイルは定期的に削除提案。
4. **ツール非依存の基盤**  
   AGENTS.mdを正規の共有ファイルとし、ツール固有のファイル（CLAUDE.md、.cursor/rulesなど）はこれを参照またはミラーする形に。

これにより、「自分のコード（人間＋モデル両方）がモデルの邪魔になる」現象を大幅に軽減できます。

### 2. 推奨ディレクトリ構造（最小主義版）

最もモデルに優しい、現実的に安定する構造です：

```
project-root/
├── AGENTS.md          # 最重要：プロジェクトの憲法。薄く保つ
├── .agents/           # ツール共通の隠しフォルダ（.claude/ や .cursor/ と併用可）
│   ├── rules.md       # 全体ルール（短く、必須項目のみ）
│   └── skills/        # 再利用スキル（1スキル1フォルダ、肥大化したら分割・削除）
├── src/               # 本体コードのみ。ここを最終的にクリーンに保つ
├── sandbox/           # モデルの実験・一時生成物の専用領域（定期クリア推奨）
├── temp/              # さらに捨てやすい一時ゴミ箱
├── docs/              # 人間向けドキュメント中心（モデルは最小限参照）
└── .gitignore         # sandbox/, temp/, *.generated などを徹底除外
```

#### 各要素の役割と精密な運用ルール

* **AGENTS.md（ルート直下）**  
  ツール非依存の正規ファイル。すべてのエージェントが最初に参照。  
  内容は最小限に抑え、以下のセクションのみ推奨：

  ```
  # Project Overview
  [プロジェクトの簡潔な目的と目標]

  ## Core Rules
  - 過去の生成コードに縛られるな。現在の要件からゼロベースでシンプルに考え直せ。
  - 新しい作業はまず sandbox/ で実験。成功したら最小限の差分だけ src/ に適用。
  - sandbox/ 内のファイルは基本的に一時的。不要になったら即削除を提案。
  - rules.md と skills/ は必要なときだけ参照。全内容を記憶しようとするな。
  - ディレクトリ構造を複雑化するな。新規フォルダ作成時は明確な理由を記述。

  ## Build & Test
  [最小限のコマンドのみ記載]

  ## Code Style
  [必須の命名規則・アーキテクチャ制約のみ。詳細はコード内のコメントやlintに任せる]
  ```

  長くなりそうならサブファイルに分割し、AGENTS.mdから「必要な部分だけ参照せよ」と指示。
* **.agents/ フォルダ**  
  隠しフォルダとしてツール共通の設定をまとめる。

  + rules.md：常に適用される制約（スタイル、テスト方針など）。短く保つ。
  + skills/：再利用可能なワークフロー。1スキルあたりSKILL.mdを中心に5000トークン以内。不要スキルは積極的に削除。
* **sandbox/ と temp/**  
  これが最も重要な隔離領域。  
  モデルに「ここで好きに実験せよ。良い結果だけ最小限マージ」と指示。  
  長時間稼働時はエージェント自身に「sandbox/ の不要ファイルをリストアップ・削除提案せよ」と定期指示。
* **src/**  
  最終的な本番コード。人間のレビューを経てクリーンに保つ。モデル生成物は直接ここに置かず、マージ時に最小化。

### 3. 運用指針（AGENTS.md に必ず記載すべきメタルール）

* **コンテキスト管理**：全ファイルを一気に読み込ませず、関連ディレクトリやファイルのみ段階的に開示。
* **自己生成コードへの態度**：過去の出力は「参考」止まり。修復しようとせず、再生成を優先。
* **クリーンアップ**：セッション終了時や一定ターンごとに「不要ファイル・ディレクトリを提案せよ」。
* **ツール対応**：AGENTS.mdを正規とし、CLAUDE.mdや.cursor/rulesは「AGENTS.mdを参照」と簡潔に記述して重複を避ける。
* **禁止事項**：不要なファイル複製、ディレクトリの深掘り過多、古い生成物の無批判な継承。

### 4. 注意点と実践Tips

* **コンテキスト汚染対策**：研究でも、過度に詳細なAGENTS.mdは逆に成功率を下げ、コストを増やすケースが指摘されています。薄さを最優先に。
* **複数ツール環境**：AGENTS.mdを単一ソースオブトゥルースに。ツール固有ファイルはこれをインポートまたは同期。
* **長時間エージェント**：sandbox/ を徹底活用し、本体（src/）への影響を最小化。定期的に「白紙から再考」指示を入れる。
* **モニタリング**：生成コードがスパゲッティ化していないか、人間が定期レビュー。モデルに「この構造はモデルを邪魔していないか自己評価せよ」と聞くのも有効。

この指針の根底にあるのは、「**エージェントはモデルである**」という事実です。  
ディレクトリ構造はモデルの推論を支援する「薄いハーネス」でなければならず、逆に制約になってはなりません。

人間の整理欲を抑え、モデルに「決めさせる」部分を最大化する——それが2026年現在のモデル中心エージェント設計の鍵です。

あなたのプロジェクトでこの構造を試してみて、具体的な課題（例：特定のスキルが肥大化する、sandboxのクリアタイミングなど）があれば、さらに精密に調整できます。  
実践事例やフィードバックを基に、指針を進化させていきましょう。
