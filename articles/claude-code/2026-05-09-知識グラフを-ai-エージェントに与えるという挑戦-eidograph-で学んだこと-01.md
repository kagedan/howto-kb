AI エージェントを毎日使っていて「**もっと自分のことを分かったうえで動いてほしい**」と思ったことはありませんか？ そこで、その答えを「**知識グラフ**」に求めた個人プロジェクト **EidoGraph** の開発記録です。論文に出会って設計を引き直した話、ベクトル検索ではなくグラフを選んだ理由、そして AI エージェントのコンテキスト基盤として知識グラフが何を変えるのかを、開発を通じて得た学びとともに書き残しました。

## 0. はじめに — このプロジェクトについて

EidoGraph は、X のポストやブログ記事といった生データから、書き手の **価値観・思考パターン・判断基準** を抽出し、知識グラフとして構造化する個人プロジェクトです。目的は単純で、

> 「未知の状況でも本人ならどう判断しそうか」を AI が推論できるためのコンテキストを作る

というもの。LLM のファインチューニングではなく **プロンプト・コンテキストベースの人格模倣** をターゲットにしています。

ちなみに名前の由来は「Eidos」で、以下の語源があるらしいので採用しました。

> 主にギリシャ語由来の「形相・イデア」を指す哲学用語

自分は Unity エンジニアであり、バックエンド関連の知識はそこまで多くなく、知識グラフも未経験でした。グラフデータベースである Neo4j も未経験です。Claude Code を相棒にしつつ、GitHub のマイルストーン機能を活用し設計と実装を進め、現在は次のような 11 パッケージのモノレポになっています。

```
contracts ─ llm ─ graph-db
                 ├─ ingestion ─ extraction ─ graph-builder ─ graph-query ─ feedback
                 │                                ↓
                 └─ pipeline ─ api-server / mcp-server / demo-chat
```

今振り返ると、知識グラフに出会って構想を立てたものの、さらに他の論文や概念と出会って、「なぜ AI エージェントに知識グラフが活きるか」ということそのものを学べたのが大きかったなと思っています。

## 1. そもそも知識グラフとは何か

知識グラフ (Knowledge Graph) を一言で説明すると以下です。

> **「ノード（事物）」と「エッジ（関係）」で世界を表現するデータ構造**。エッジには型があり、「`A` は `B` の `DERIVES_FROM` である」といった意味を保持できる。

例えば以下のように表現されます。信念がなにによって形成されたか、などを保持できるわけです。

```
[Belief: 本番環境の安全性最優先]
      │
      │ DERIVES_FROM
      ↓
[DecisionRule: JWT にはリフレッシュトークンの有効期限テストを書く]
      │
      │ EVIDENCES
      ↓
[Episode: 2025年3月 Project X で認証脆弱性が発覚]
```

リレーショナル DB との違いは、**関係そのものをファーストクラスのデータとして扱える** ことです。RDB でも JOIN すれば類似のことはできますが、グラフ DB は関係をたどる操作（traversal）に最適化されています。Neo4j では Cypher クエリで `(r:DecisionRule)-[:DERIVES_FROM]->(b:Belief)` のように、ASCII アートのように、図に書くような形でクエリが書けるのが特徴です。

書籍で学んだこととして、グラフ DB は、この「関係性をたどる」ことに重点が置かれているため、検索の際はポインタのような形でリンクを形成します。そのため、RDB ではコピー・結合といった手間が発生してしまうようなケースでも、グラフ DB だと高速に検索、利用ができる、ということのようです。

ベクトル DB との違いはもっと本質的で、

|  | ベクトル DB | 知識グラフ |
| --- | --- | --- |
| 検索の単位 | 「意味的に近いテキスト片」 | 「明示的に関係づけられたノード」 |
| 推論 | 暗黙（埋め込み空間の近さ） | 明示（エッジ型の意味で辿る） |
| 説明可能性 | 弱い（なぜ近いかは説明しにくい） | 強い（どのエッジを辿ったかが残る） |
| 弱点 | 意味の曖昧さに弱い | 構築コストが高い |

ベクトル検索が「似ているもの」を返すのに対し、知識グラフは「**どう関係しているか**」を返す。これが後述する AI エージェント文脈で効いてきます。

## 2. 学びの変遷

このプロジェクトは大きく分けて 3 度、根本から設計が変わりました。自分が知識グラフやその周辺知識をつけていくにつれて解像度が上がっていったため、それをフィードバックして変化した、という感じです。

### 2.1 第一段階：「ノードとエッジ」の整理

前述のように、元々知識グラフの知見がなかったため、どういう構成にすべきかから Claude Code に相談しながら開発を行いました。ただ、ざっくりとしたイメージはあって、なにかしらの事実を抽出して、それらをエッジでつなげばいいだろう、というイメージは持っていました。

たとえば「テストは先に書きたい」「TDD でいきたい」「テストファーストが好み」という 3 つの異なる発言があったとき、これらは 3 つのノードではなく、1 つの `Preference` ノード「テストファースト開発を好む」に統合されます。元発言は `Evidence` ノードとして別レイヤーに保持し、`SUPPORTED_BY` エッジで結ぶ、という具合です。

ただ、それをどうしていいか分からなかったのでディスカッションを続けました。

### 2.2 第二段階：ルールベース vs LLM の責務分離

最初の実装では「LLM に全部やらせれば賢くやってくれるだろう」と考えていました。チャンク分割もノイズ除去も意味抽出も LLM。ただ、それだとコストが跳ね上がる可能性があり、読み込ませる文章量が多いと知識グラフを構築するだけでお金がかかってしまいます。そこで、LLM を使わない方法で情報を抽出できる方法を探しました。（というか Claude Code に聞いた）

そこで返ってきた答えが、ルールベースで抽出し、エッジなどの「関係性をつなぐ部分のテキストを生成してもらう」というハイブリッド構造でした。

| ステージ | 担当 | 理由 |
| --- | --- | --- |
| ノイズ除去・形式分類・トピックタグ付与・頻度集計 | ルールベース | 機械的に処理可能。LLM に任せる価値が薄い |
| 正準文生成・関係抽出・意味統合・confidence 推定 | LLM | 曖昧さの解決にこそ LLM の価値がある |

LLM は「生成 AI」と言われる通り、テキストを生成することが得意。言い換えると、すでにあるデータから抽出できるものを整理するのに使うのではなく、そこから導き出される内容を「生成」することに利用した、という形です。一方で、すでに与えられたデータからルールベースで抽出する方法は、LLM 登場以前から存在している技術であり、それを組み合わせたわけです。（まぁ当然、ルールベース部分も自分の専門分野からは離れているので、壁打ちしながら Claude Code に実装してもらったものではありますが）

Claude Code の言葉を借りれば、

> **LLM は「曖昧なものを意味づける」のは得意だが、「明らかにノイズなものを捨てる」のはコスパが悪い**。前者は確率分布の中央値が正解になり、後者は分布の端を切るだけだから、ルールで足りる。

ということ。

### 2.3 第三段階：HumanLM 論文との出会い、A 層の追加

最大の転換点は、ある論文との出会いでした。AI エージェント、アシスタントのメモリシステムについて興味があって、色々な方法で情報を集めていたあるときに出会った論文。

> **HumanLM: Simulating Users with State Alignment Beats Response Imitation** (Wu et al., 2026)  
> <https://arxiv.org/abs/2603.03303>

この論文が主張するのは、

> 表面的な言い回しを真似る (Response Imitation) では、人格は再現できない。心理学的に根拠のある内的状態を整合させる (State Alignment) 必要がある。

実験では、教師あり微調整 (SFT) による表面模倣の改善幅は **6.5%（最悪値）**、一方で内的状態整合は **16.3%** の改善を達成したという話でした。「言葉のクセ」と「考え方のクセ」は別物で、後者を扱わないと模倣にならないという主張です。

要するに、話し方を表面上で模倣しても、その人の裏の真意までは測れず浅い発言になってしまうわけです。それを「考え方」から模倣することで、表面的ではなく内面的にも、環境に即した模倣が可能になった、と理解しています。

論文は人格を 6 つの心理学的次元で記述しています：

| 次元 | 定義 |
| --- | --- |
| Belief（信念） | 何を真実と考えているか |
| Goal（目標） | 何を達成したいか |
| Value（価値観） | 何を根本的に大切にするか |
| Stance（立場） | 特定コンテキストでの立ち位置 |
| Emotion（感情） | 情報処理に影響する感情的傾向 |
| Communication（コミュニケーション） | 情報をどう構造化して伝えるか |

これを読んで、実装中の知識グラフに取り込むべきと感じ、再設計を依頼してこの内容を盛り込みました。

当時のグラフは `DecisionRule`（「JWT にはリフレッシュトークンテストを書く」）と `Preference`（「TypeScript が好み」）と `Experience`（具体エピソード）が中心の **B 層 + C 層** でした。これは「**既知の状況での判断ルール集**」としては機能するものの、

* 既存ルールに該当しない **未知の状況** でどう判断するか？
* 複数のルールが **矛盾する状況** でどちらを優先するか？

に答えられません。これを解決するための方法がまさに論文の内容ということですね。

そこで、HumanLM の論文を参考に再設計を行い、**A 層（信念・価値観の抽象層）** を追加しました。

```
A 層（抽象）  : Belief / Goal / Value / Stance / Emotion / Communication
B 層（ルール）: DecisionRule / Preference / ReasoningPattern
C 層（具体） : Experience / Evidence / TopicInterest
```

層間関係も新設しました：

* **DERIVES\_FROM** (B → A)：「このルールは、この信念から導出されている」
* **REINFORCES** (C → A)：「このエピソードが、この信念を強化した」
* **CONFLICTS\_WITH** (A ↔ A)：「特定条件下でこの 2 つの価値観は矛盾する」

それぞれを Neo4j 上でいくつか検索してみると以下のように表示されます。

![KG Sample](https://static.zenn.studio/user-upload/deployed-images/b77063af27051e19100d3819.png?sha=df9e20c74042559a512bfa7c7c4bde0792bcd2e6)

右下のほうにズームした結果が以下です。

![KG Sample Detail](https://static.zenn.studio/user-upload/deployed-images/a975131b81013df059caf664.png?sha=1d45f445005999953809d596ab274c40b5ec24e0)

これで「なぜそのルールを持っているのか」をたどれるようになりました。`未知の状況に直面 → 該当する具体ルールが無い → A 層から推論する` という経路が成立したわけです。

ここでの学びは、

> **知識グラフのスキーマは、「何を覚えるか」よりも「何を推論できるか」で決まる。**

ということでした。スキーマは仮想的な推論エンジンの入力ポートなのです。

### 2.4 第四段階：データの偏りと収集の難しさ

スキーマを整えても、データが偏っていれば意味がありません。実際、ブログ 178 記事から構築したグラフを集計してみると、こんな分布になっていました。

| カテゴリ | ノード数 | 望ましい数 |
| --- | --- | --- |
| TopicInterest（興味領域） | 58 | 十分 |
| Communication | 39 | 十分 |
| Belief | 23 | おおむね良好 |
| **Value** | **1** | **15+ 欲しい** |
| **Stance** | **2** | **10+ 欲しい** |
| **Emotion** | **0** | **5+ 欲しい** |

技術ブログが情報源なので技術的な興味関心は厚いですが、「**何を根本的に大切にしているか (Value)**」「**何にワクワクするか (Emotion)**」といった、人間としての軸に関わる情報は壊滅的に少ない。

ここで気づいたのは、**生データの種類が、抽出できるノードの種類を規定する** ということでした。`DecisionRule` を抽出するには技術メモ的なテキストが向きますが、`Value` を抽出するには「二者択一を迫られて選んだ場面」「強い感情が出た場面」が必要になります。これらは技術ブログにはほぼ含まれません。

ただもともと、手元のデータでいきなり構築できるとは考えておらず、作成した知識グラフを元に AI アシスタントを動かして、知識グラフを育てていく、という考えでいました。そこで、別の AI アシスタントに「日常会話の中から A 層を中心に拾って Feedback API に投げる」というスキルを書きました。データソースとしての **「会話」を組み込む** という発想です。

## 3. なぜ AI エージェントに知識グラフなのか

ここまでが学びの話。ここからは、なぜ AI エージェント全般にとって知識グラフが意味を持つのか、現時点で見えていることを書いていきます。

### 3.1 ファインチューニング、RAG、知識グラフの違い

AI エージェントに「個人の文脈」を与える手段は複数あります。

| 手法 | 何を覚えるか | 強み | 弱み |
| --- | --- | --- | --- |
| ファインチューニング | モデルの重みに分散して焼き込む | 推論時のコンテキスト消費ゼロ | 更新が高コスト・説明不可・忘却制御が困難 |
| RAG（ベクトル検索） | 生テキストを類似度で引く | 構築が安い・即時更新可能 | 「考え方」ではなく「似た文章」しか返らない |
| 知識グラフ | 構造化された関係 | 推論経路が明示・更新粒度が小さい | 構築コストが高い（ここを LLM で下げるのが本プロジェクト） |

ファインチューニングは強力ですが、たとえば「価値観が時間とともに変わった」を表現するのが極めて難しいです。RAG は逆に柔軟ですが、「品質と速度なら品質を優先する」という抽象的な判断軸を、`類似度` という尺度で安定して取り出すのは難しいです。

知識グラフの強みは、**「未知の状況に対して、どの抽象から推論するか」を明示的に設計できる** ことです。エージェントが新しい質問を受けたとき、

1. 関連する具体ルール (B 層) を引く
2. 該当ルールが無ければ、上位の信念・価値観 (A 層) を引いて推論する
3. 矛盾する価値観があれば、その矛盾自体をエージェントに渡す

という処理を、ベクトル類似度の中央値ではなく **明示的なエッジの traversal** として実装できます。

### 3.2 エージェントのハーネス側から見ると何が嬉しいか

Claude Code のような AI エージェントを日常的に使っているとよく分かりますが、**コンテキストウィンドウは有限の資源** です。CLAUDE.md に何でも書けるわけではないし、書いたところで、状況に関係ない情報まで毎ターン読み込まれて、推論がぼやけます。

知識グラフをコンテキスト基盤に置くと、ハーネス側からは次のように見えます。

* **動的にコンテキストを組み立てられる**：今の話題が「セキュリティ」なら、関連する Belief / DecisionRule / Experience だけを抽出して投げる。
* **トークン予算を意識できる**：「上位 N 件、トークン上限 X」のような retrieval policy を組める。
* **更新が局所的になる**：価値観が変わったら、該当ノードの strength を下げて、新ノードを追加するだけ。プロンプト全体を書き直す必要がない。
* **説明責任が持てる**：エージェントが「あなたなら品質を優先するから、こちらを推奨します」と言ったとき、根拠ノードと根拠エピソードまで提示できる。

このうち最後の **説明責任** は、人格模倣以外の文脈、たとえば社内ナレッジを扱う AI エージェントでも極めて重要になると思っています。「なぜこの判断をしたか」をエッジでたどって示せるかどうかは、今後のエージェント信頼性の分水嶺になるはずです。

### 3.3 「秘書エージェント」というユースケース

EidoGraph の元となった想定は、**マスター（人間）の代わりにワーカーエージェントへ指示を出し、報告を一次レビューする秘書エージェント** です。秘書がやるべきことは大きく 3 つ：

1. マスターの指示スタイルを模倣して、ワーカーへ指示を生成する
2. ワーカーからの報告をマスターの判断基準でレビューする
3. **未知の状況でも、マスターの信念から推論して自律判断する**

3 つ目が知識グラフの本領です。マスターが過去に書いていない判断は、過去の発言からの完全一致では引き出せません。ですが、

* A 層に「コード品質はスケジュールより優先」(Value, strength: 0.9) がある
* B 層に「未検証技術は本番投入前に負荷テストする」(DecisionRule) がある
* C 層に「Project Y で早期マイクロサービス化が失敗した」(Experience) がある

という構造があれば、新規の質問「マイクロサービス化を始めるべきか」に対して、**直接マッチするルールがなくても A 層と C 層から推論を生成できる** のです。

この「**直接ルールがない → 抽象から推論する**」という挙動こそ、ファインチューニングや RAG では再現しにくい、知識グラフの固有の価値だと考えています。

## 4. 設計上の重要な分離

実装してみて、後付けで「これは大事だった」と気づいた分離をまとめています。汎用的に効く設計指針なので、別プロジェクトでも使えるはずです。

### 4.1 confidence と weight を分ける

```
type GraphNode = {
  confidence: number  // 抽出・統合の確からしさ（システム側の事情）
  weight: number      // 人格模倣上の重要度（ユーザー側の事情）
  // ...
}
```

最初は両者を「スコア」一つで管理しようとして失敗しました。

* `confidence` が低い ＝ 抽出が怪しいので、別の方法で再確認したい
* `weight` が低い ＝ 抽出は確実だが、本人にとって重要ではないので普段は出さない

混ぜるとどちらも判断できなくなります。**「システムの自信」と「ユーザーにとっての重要度」は別軸** なのです。これは KG だけでなく、検索ランキングや推薦システム一般に効く分離だと思います。

抽出は「この文からシグナルを取る」、統合は「複数シグナルから正準文を作る」。

* 抽出は決定的（同じ入力に同じ出力）にできるのでテストが楽
* 統合は LLM 依存で確率的なので、評価戦略が変わる
* 抽出だけ差し替えたい（別のソース種別を追加したい）ケースがある

責務の分離は「テスト戦略の分離」と一致します。これは経験則として強く言えます。

### 4.3 LLM プロバイダ抽象化

`LlmClient` インターフェースを切って OpenAI / Claude CLI / OpenAI 互換（LM Studio, Ollama, vLLM）を差し替えられるようにもしました。最初は OpenAI のみでしたが、コスト削減の観点から後者ふたつを追加しました。ただ、Claude Code はキャッシュがうまく効かないのか、あるいは今回実装した内容がかみ合わないのか、トークン消費量がかなり多かったのであくまで使える、くらいの感じになっています。

## 5. 精度評価

ローカル LLM を導入した時点で評価フレームワーク (Intrinsic / Extrinsic / LLM-as-Judge) を作ったのですが、意外と Qwen3 のローカル LLM のほうが、OpenAI の gpt-5.4-mini よりスコアが高かったのは意外でした。

評価の自動化は、

* どのモデルが構築した KG が、persona context として品質が高いか
* 同じモデルでも、プロンプト変更で品質がどう変わるか
* ある変更が回帰を起こしていないか

を客観的に答えられる土台になります。**LLM を活かす個人プロジェクトで、評価フレームワークの優先度は思っているより高い** と思いました。やはり、LLM は確率で出力するのと、そもそも人間が読める量をはるかに凌駕する文章量を出力するので、こうした機械的なチェック機構は今後とても重要になるなと感じています。

## 6. これから AI エージェントに何を期待するか

最後に、これからの AI エージェントの使い方について現時点で考えていることを書いておきます。

### 6.1 「持ち歩ける人格コンテキスト」

EidoGraph で構築した KG は、いま MCP Server 経由で外部の AI アシスタントから引けるようになっています。これが意味するのは、

> 「自分の価値観・思考パターン」を **別の AI エージェント間で持ち歩ける**

ということです。Claude Code でコードを書くとき、Cursor でドキュメントを書くとき、別の AI チャットで雑談するとき、すべてが同じ知識グラフを参照する世界が来ると考えています。言ってみれば、自分を深く理解してくれている AI アシスタントです。アイアンマンのジャービスみたいな。あるいは転スラのラファエルみたいな。今の AI 文脈の最大の弱点は **「会話を変えると毎回ゼロから」** ですが、KG を中央にすると、文脈の継続が DB レベルで担保されます。

### 6.2 エージェントが KG を能動的に育てる

現在は人間がフィードバックを与えて KG を更新していますが、エージェントが自律的に「この発言は新しい Value 候補だ」と検出して Feedback API に投げ込む構図はもう試作しています。**エージェントが KG の利用者であると同時に、KG の育成者になる** という方向は、自己改善ループとして自然だと思います。

ただし、ここには「ハルシネーションされた価値観が KG に書き込まれる」というリスクがあります。EidoGraph では `pending_review` ステータスを噛ませて、人間の最終承認を残しています。**エージェントによる KG 自動拡張は、確証バイアスのループになりやすい** と考えており、当面は半自動を維持する予定です。

### 6.3 KG の「忘れ方」を設計する

人間の価値観は時間で変わります。 EidoGraph では `lastSeenAt` / `weight` を持っており、時間減衰や `deprecated` フラグで「古い価値観を忘れる」を表現できる設計にはなっています。

ただし「**何を忘れるか**」のポリシーはまだ未着手です。RAG では「インデックスから消す」で済みますが、KG では「あるノードを忘れると、それに依存していた DERIVES\_FROM 関係はどうなるか」という波及があります。これは今後の研究テーマだと思っています。

## 7. まとめ — 何を学んだか

1. **知識グラフのスキーマは「何を覚えるか」ではなく「何を推論できるか」で決まる**。スキーマは仮想推論エンジンの入力ポート。
2. **生文をそのままノードにせず、正準文に統合する**。出典は別レイヤーに保持。これがないと類似ノードが爆発する。
3. **LLM を全部に使わず、ルールベースと責務分離する**。LLM は曖昧さの解決にこそ価値があり、ノイズ除去には向かない。
4. **抽象層 (A 層) を持たない KG は、未知の状況で推論できない**。HumanLM 論文の指摘どおり、「言い回し」と「考え方」は別物。
5. **データソースの種類が、抽出できるノード種類を規定する**。技術ブログから Value は出ない。会話・ジャーナル・対話を組み合わせる必要がある。
6. **confidence と weight を分ける**。「システムの自信」と「ユーザーにとっての重要度」は別軸。
7. **評価フレームワークは想像より本質的**。LLM が絡むプロジェクトでは早めに用意したほうがいい。
8. **AI エージェントの未来において、KG は「持ち歩ける人格コンテキスト」になる**。複数エージェント間で文脈を継続させる中央 DB として機能しうる。

知識グラフは、構築コストが高く、即効性のあるリターンも見えにくいため、ベクトル DB に比べて少し知名度が低いかもしれません。ですが、**AI エージェントに「考え方」を渡したい場面では、ほかに代わるものが見当たらない** というのが率直な感想です。

ファインチューニング不要、RAG だけでは足りない、その間にある狭くて深い領域が知識グラフの居場所だと思います。本稿が、同じ問いを抱える誰かの設計の出発点になれば幸いです。
