Claude Code（Claude CLI）とMCPサーバーを使ったPoCを、チームで共有しながら進めていました。  
その中で、開発者ごとにClaudeの挙動が微妙に違うことがありました。

最初はプロンプトやMCPサーバー側の差分を疑っていたのですが、調べてみると原因は **Claude Code の自動メモリ** でした。

## 起きていたこと

同じプロジェクトを触っているはずなのに開発者によってClaudeの判断が少し違うことがありました。

たとえば、

* ある人の環境では特定の実装方針を覚えている
* 別の人の環境ではその前提がない
* 同じ依頼をしても、参照する情報や判断が少し変わる

という状態です。

チームで同じPoCを進めている場合、これは少し困ります。  
個人ごとの学習内容が混ざると、「プロジェクトとして共有している前提」と「その人の環境だけにある前提」が分かりづらくなるためです。

## 原因は自動メモリだった

Claude Code には自動メモリ機能があります。  
公式ドキュメントでは、Claudeが作業中にビルドコマンド、デバッグの知見、アーキテクチャメモ、コードスタイルの好みなどを保存できる機能として説明されています。

この自動メモリは便利ですが、チーム開発では注意が必要です。

公式ドキュメントによると、自動メモリは各プロジェクトごとに `~/.claude/projects/<project>/memory/` に保存されます。  
つまり、リポジトリに入っている共有ファイルではなく、基本的には各開発者のローカル環境に保存されます。

そのため、開発者AのClaudeと開発者BのClaudeで、裏側に持っているメモリが違う状態になりえます。

今回のように、チームで同じ挙動を期待したいPoCでは、この差分がノイズになりました。

## 対応：プロジェクト設定で自動メモリをオフにする

対応自体はシンプルでした。

プロジェクト直下に `.claude/settings.json` を作り、以下を追加します。

```
{
  "autoMemoryEnabled": false
}
```

これで、そのプロジェクトでは自動メモリが無効になります。

`.claude/settings.json` はプロジェクト設定なので、Git管理してチームで共有できます。  
一方、`.claude/settings.local.json` は個人用のローカル設定で、Git管理しない用途です。

今回のように「チーム全員で同じ挙動に寄せたい」場合は、`.claude/settings.json` に入れるのが自然だと思います。

## 自動メモリを切ったほうがよさそうなケース

自動メモリ自体は便利な機能です。  
個人開発では、過去の作業内容を覚えてくれることで助かる場面もあると思います。

ただ、次のような場合は切る選択肢もありだと感じました。

* チームで同じPoCを進めている
* 開発者ごとの挙動差を減らしたい
* 暗黙知ではなく、共有ファイルに前提を寄せたい
* Claudeの判断理由を追いやすくしたい

今回の目的は、Claudeの個人最適化ではなく、チームで再現しやすい検証環境を作ることでした。  
その意味では、自動メモリをオフにしたほうが扱いやすかったです。

## おわりに

Claude Codeの自動メモリは便利ですが、チームで同じプロジェクトを共有して使う場合は、開発者ごとのローカルな記憶が挙動差の原因になることがあります。

今回のように再現性を重視するPoCでは、`.claude/settings.json` に以下を入れて、自動メモリを切るのがシンプルでした。

```
{
  "autoMemoryEnabled": false
}
```
