## この記事で作るもの

Step 08・09で実装した「書き込みの前に人間が確認するフロー」を紹介します。

エージェントがファイルを書き換えようとすると、いきなり変更が適用されるのではなく、こんな確認画面が表示されます。

```
==================================================
[変更確認] target/test_patch.py
--- a/target/test_patch.py
+++ b/target/test_patch.py
@@ -1,3 +1,4 @@
 def greet(name):
-    print("hello " + name)
-    return None
+    message = f"Hello, {name}!"
+    print(message)
+    return message
==================================================
実行しますか？ [y/N]:
```

`y` を入力すれば適用、それ以外はキャンセル。シンプルですが、エージェントが自律的に動きすぎないための重要な安全装置です。

このフローを支えるのは3本のツールです。

| ツール | 役割 |
| --- | --- |
| `show_diff` | 変更前後の差分をunified diff形式で表示（読み取り専用） |
| `patch_file` | ファイルの一部を置換する（書き込み） |
| `write_file` | ファイルを全体上書きする（書き込み） |

---

## なぜ人間確認が必要か

前回のA4記事で作った探索ツール（`list_files` / `read_file` / `search_text`）は「読み取り専用」でした。エージェントがどれだけ動き回っても、ファイルは変わりません。

しかし `patch_file` や `write_file` は違います。**一度実行するとファイルが書き換わります。** しかもエージェントは確認なしに次々とツールを呼び出すので、気づいたときには意図しない変更が積み重なっている可能性があります。

「自律的に動けること」と「人間がコントロールできること」は相反します。この2つのバランスをどこで取るかが、エージェント設計の核心です。

今回は\*\*「取り消せない操作には必ず確認を挟む」\*\*という方針を採りました。読み取りは自動、書き込みは人間が判断する。Claude Code自身も同じ設計思想で動いています。

## show\_diff の実装

src/tools/diff\_tools.py

```
import difflib
from pathlib import Path

def show_diff(original: str, modified: str, filename: str = "file") -> str:
    if original == modified:
        return "差分はありません（変更なし）"

    original_lines = original.splitlines(keepends=True)
    modified_lines = modified.splitlines(keepends=True)

    diff = difflib.unified_diff(
        original_lines,
        modified_lines,
        fromfile=f"a/{filename}",
        tofile=f"b/{filename}",
        lineterm="",
    )

    diff_text = "\n".join(diff)

    if not diff_text:
        return "差分はありません（変更なし）"

    return diff_text
```

Pythonの標準ライブラリ `difflib.unified_diff` を使って、GitのDiff表示と同じ形式の差分テキストを生成します。

重要なのが `show_diff` を\*\*「表示専用ツール」\*\*として設計した点です。引数として `original` と `modified` を文字列で受け取るだけで、ファイルには一切触れません。「差分を見せる」と「ファイルを書く」を完全に分離しています。

分離する理由は確認フローを組みやすくするためです。`repl.py` で `_confirm_tool_call` を実装するとき、`show_diff` を純粋な文字列変換として呼び出せると、確認画面の組み立てが単純になります。

## patch\_file の実装

src/tools/diff\_tools.py

```
def patch_file(path: str, original_snippet: str, new_snippet: str) -> str:
    try:
        file_path = Path(path).resolve()
        cwd = Path.cwd().resolve()
        try:
            file_path.relative_to(cwd)
        except ValueError:
            return f"エラー：作業ディレクトリ外への書き込みは禁止されています: {path}"

        if not file_path.exists():
            return f"エラー：ファイルが見つかりません: {path}"

        content = file_path.read_text(encoding="utf-8")

        if original_snippet not in content:
            return (
                f"エラー：指定されたテキストがファイル内に見つかりません。\n"
                f"ファイル: {path}\n"
                f"検索テキスト:\n{original_snippet}"
            )

        new_content = content.replace(original_snippet, new_snippet, 1)
        file_path.write_text(new_content, encoding="utf-8")

        return f"✅ ファイルを更新しました: {path}"

    except PermissionError:
        return f"エラー：書き込み権限がありません: {path}"
    except Exception as e:
        return f"エラー：パッチ適用に失敗しました: {path} ({e})"
```

`patch_file` は「ファイルの一部だけを置換する」ツールです。LLMが `original_snippet`（置換前テキスト）と `new_snippet`（置換後テキスト）を指定します。

`.replace(original_snippet, new_snippet, 1)` の末尾の `1` がポイントです。これで**最初の1件だけを置換**します。同じテキストがファイル内に複数あっても、意図しない箇所まで書き換わらないようにする安全策です。

`original_snippet` がファイル内に存在しない場合は、書き込まずにエラーメッセージを返します。LLMが誤ったスニペットを指定しても、ファイルは変更されません。

## write\_file の実装

src/tools/file\_tools.py

```
def write_file(path: str, content: str) -> str:
    try:
        file_path = Path(path).resolve()
        cwd = Path.cwd().resolve()
        try:
            file_path.relative_to(cwd)
        except ValueError:
            return f"エラー：作業ディレクトリ外への書き込みは禁止されています: {path}"

        file_path.parent.mkdir(parents=True, exist_ok=True)
        file_path.write_text(content, encoding="utf-8")
        return f"✅ ファイルを保存しました: {path}"
    except PermissionError:
        return f"エラー：書き込み権限がありません: {path}"
    except Exception as e:
        return f"エラー：ファイル書き込みに失敗しました: {path} ({e})"
```

`write_file` はファイルを**全体上書き**します。新規ファイル作成にも使えます（`parent.mkdir(parents=True, exist_ok=True)` でディレクトリも自動生成）。

`patch_file`（部分置換）と `write_file`（全体上書き）の使い分けは次の通りです。

| 操作 | 使うツール |
| --- | --- |
| 関数1つだけ書き換えたい | `patch_file` |
| ファイル全体を新しい内容で置き換えたい | `write_file` |
| 新規ファイルを作りたい | `write_file` |

LLMは状況に応じてどちらを使うか判断します。

## 人間確認フローの実装

書き込み系ツールの呼び出し前に確認を挟む処理は `repl.py` の `_confirm_tool_call` に実装しました。

src/repl.py

```
from tools.diff_tools import show_diff

def _confirm_tool_call(tool_name: str, arguments: dict) -> bool:
    print("\n" + "=" * 50)

    if tool_name == "patch_file":
        path = arguments.get("path", "")
        original = arguments.get("original_snippet", "")
        new = arguments.get("new_snippet", "")
        diff = show_diff(original, new, path)
        print(f"[変更確認] {path}")
        print(diff)

    elif tool_name == "write_file":
        path = arguments.get("path", "")
        content = arguments.get("content", "")
        print(f"[上書き確認] {path}")
        print(f"内容（先頭200文字）:\n{content[:200]}")

    elif tool_name == "run_command":
        command = arguments.get("command", "")
        print(f"[コマンド実行確認]\n$ {command}")

    print("=" * 50)
    answer = input("実行しますか？ [y/N]: ").strip().lower()
    return answer == "y"
```

確認が必要なツールは `dispatcher.py` の `REQUIRES_CONFIRMATION` セットで管理しています。

src/dispatcher.py

```
REQUIRES_CONFIRMATION = {"write_file", "patch_file", "run_command"}

def requires_confirmation(tool_name: str) -> bool:
    return tool_name in REQUIRES_CONFIRMATION
```

REPLのエージェントループでは、`requires_confirmation` がTrueのときだけ確認を取ります。

src/repl.py

```
for tc in tool_calls:
    tool_name = tc["name"]

    if requires_confirmation(tool_name):
        confirmed = _confirm_tool_call(tool_name, args)
        if not confirmed:
            result = "ユーザーがキャンセルしました。"
            print("[キャンセル] 操作を中止しました。")
        else:
            result = dispatch(tool_name, tc["arguments"])
    else:
        print(f"[ツール実行] {tool_name}...")
        result = dispatch(tool_name, tc["arguments"])
```

`[y/N]` の入力で `y` 以外はすべてキャンセルとしています。Enterだけを押しても `n` と同等です。「安全側に倒す」設計です。キャンセル時は `"ユーザーがキャンセルしました。"` という文字列をツール結果としてLLMに返すので、LLMは「操作が中止された」と認識して次のアクションに移れます。

## 動作確認の様子

`step09_patch.py` で単体テストを行いました。

**変更を適用する（y）**

```
[差分確認]
--- a/target/test_patch.py
+++ b/target/test_patch.py
@@ -1,3 +1,4 @@
 def greet(name):
-    print("hello " + name)
-    return None
+    message = f"Hello, {name}!"
+    print(message)
+    return message

この変更を適用しますか？ [y/N]: y

[結果] ✅ ファイルを更新しました: target/test_patch.py

[変更後のファイル]
def greet(name: str) -> str:
    message = f"Hello, {name}!"
    print(message)
    return message
```

**変更をキャンセルする（n または Enter）**

```
この変更を適用しますか？ [y/N]: n
[キャンセル] 変更を中止しました。
```

どちらのケースも、確認前にファイルの中身は変わりません。`y` を入力して初めて `patch_file` が呼ばれます。

## 躓いた点・気づき

**`keepends=True` と `lineterm=""` の組み合わせで行間に空行が入る**

`splitlines(keepends=True)` で各行末に改行文字を残したまま `unified_diff` に渡し、さらに `lineterm=""` を指定して `"\n".join(diff)` でつなぐと、行間に空行が入ったように見える出力になります。

具体的には `splitlines(keepends=True)` が返す `"print("hello " + name)\n"` という文字列が diff に含まれ、そこに `"\n".join` の改行がさらに加わるためです。見た目は少し気になりますが、diff の内容自体は正しいので、今回は許容しました。

**`show_diff` を `repl.py` に直接importして確認フロー内で呼び出す**

`show_diff` はLLMが呼び出すツールとしても登録していますが、`repl.py` の `_confirm_tool_call` 内でも直接importして使っています。ツールとしての `show_diff` と、確認UIのヘルパーとしての `show_diff` が同じ関数を共有する形です。

コードの重複がなく、diffの表示ロジックが一箇所に集まる点でこの設計は合理的でした。

## まとめ

* `show_diff`（表示専用）→ 確認 → `patch_file`（書き込み）の3ステップが安全な書き込みフローの骨格です
* 「自律性」と「安全性」のバランスを取る方法は「取り消せない操作だけを確認対象にする」こと
* `REQUIRES_CONFIRMATION` セットでどのツールが確認対象かを宣言的に管理することで、ツールが増えても確認ロジック自体は変わりません
