Claude Code、高いですよね。  
最近エンジニア以外の人たちにもClaude Codeの存在が知れ渡るようになりましたが、満足に使おうとすると月額$200のプラン、最低でも月額 $100プランは必要になってきます。月額$20のProプランでも使えなくはないですが、最高性能モデルのOpusを選択しているとすぐにレート制限にかかってしまいます。

一方で、非エンジニア界隈ではほとんど話題に上がらないのですが、実はOpenAIもClaude Codeと同じAgentic Coding用のツールとしてCodexを商用展開しています。このCodexですが、gpt-5.3-codexが登場したあたりからコーディングエージェントとしての性能はOpusモデルと遜色ない実用レベルになっていると感じています。もちろんモデルによって特徴は異なりますが、CodexにもClaude Codeにない良さもあるので正直好みの問題だったり、使い方次第でどうにかなる程度の違いになっています。

そんなCodexの一番嬉しい点は、なんと言っても破格の価格設定だと思います。Claude Codeでは最高性能モデルであるOpusを満足に使おうと思うと最低でも月額$100を要求される一方で、Codexでは同等性能のモデルを月額$20で使えてしまいます。なんとお値段1/5。破格ですね。CodexのレートリミットがClaude Code Proプランと同等かというとそんなことはなく、$100プランと同じくらいのクレジットが与えられている印象です。

Codexのレートリミットは5時間枠と1週間枠の2種類存在し、どちらも使い切るとその枠のクレジットが復活するまでは使えなくなる仕様です。クレジット消費2倍でアウトプット速度2倍にするモードを使わなければ、あまり使い切る心配はない印象です。

Claude Codeを使ってみたいけど$100は払えないという人には 、Codexを使ってみることをお勧めします。

この記事では、そんなCodexの使い方について画像を交えながら紹介します。

# 1. Codexの拡張機能を追加する

Codexには、アプリ版とCLI版の二種類存在します。私はCLI版しか使ったことがないので、ここではCLI版を前提として説明していきます。  
Codex CLIはターミナル上で使うこともできるのですが、VSCodeやCursorで開発しながらCodexを使いたい場合、拡張機能を追加することをお勧めします。拡張機能では使い勝手のいいUIを用意してくれており、セッションの移動や各種設定の変更が直感的にできるのでお勧めです。  
![](https://static.zenn.studio/user-upload/1e88c1cbef80-20260330.png)

# 2. Codex用のドキュメントを作成する

まず最初に、Codexがタスクをこなすにあたって事前に知っておいてもらうべき情報をまとめたドキュメントであるAGENTS.mdを作成します。AGENTS.mdはいわばエージェントのためのREADMEであり、エージェントがプロジェクトで作業する際に必要な背景情報やタスクの進め方などを確実に伝えるためのドキュメントです。  
AGENTS.mdは、コーディングエージェント全般で広く使える標準化されたフォーマットなので、Codex以外にもCursorやVS Codeなどでも同じファイルを使い回し可能です。

これで、毎回前提条件などをプロンプトで入力しなくてもCodexがAGENTS.mdを参照して自動でコンテキストに追加してくれるようになります。

## 例

```
# AGENTS.md サンプルファイル

## 開発環境のヒント
- `ls` コマンドで探すのではなく、`pnpm dlx turbo run where <プロジェクト名>` を使ってパッケージへ直接移動しましょう。
- Vite、ESLint、TypeScriptがパッケージを認識できるように、`pnpm install --filter <プロジェクト名>` を実行してワークスペースに追加してください。
- `pnpm create vite@latest <プロジェクト名> -- --template react-ts` を使うと、TypeScriptチェックが設定済みの新しいReact + Viteパッケージをすぐに作成できます。
- 正しい名前を確認するために、各パッケージ内の `package.json` にある `name` フィールドを確認してください（トップレベルのものは除外してください）。

## テスト手順
- CIプランは `.github/workflows` フォルダの中にあります。
- `pnpm turbo run test --filter <プロジェクト名>` を実行して、そのパッケージに対して定義されているすべてのチェックを実行してください。
- パッケージのルートディレクトリからは、単に `pnpm test` と打つだけで実行可能です。マージする前に、コミットがすべてのテストをパスすることを確認してください。
- 特定のテストに絞る場合は、Vitestのパターンを使用して `pnpm vitest run -t "<テスト名>"` と実行してください。
- 全てのテストが通る（緑色になる）まで、テストや型のエラーを修正しましょう。
- ファイルの移動やインポートの変更を行った後は、`pnpm lint --filter <プロジェクト名>` を実行し、ESLintやTypeScriptのルールに問題がないか確認してください。
- 誰かに指示されていなくても、変更したコードに対しては必ずテストを追加または更新してください。

## PR（プルリクエスト）の手順
- タイトルの形式: [<プロジェクト名>] <タイトル>
- コミット前には必ず `pnpm lint` と `pnpm test` を実行してください。
```

# 3. Planモードで実装計画を立てる

Planモードをオンにして実装を依頼することで、実装計画を立ててくれます。これで、自分の意図と方針がずれていないかを確認できます。  
方針を明確にするために質問が必要な場合は、Codex側から質問を投げてくれます。

![](https://static.zenn.studio/user-upload/3973d0669382-20260330.png)

質問に答えていくと、最終的に実装計画が出力されます。計画が意図とずれている場合には、再度手直しの指示を出して修正できます。修正がない場合、計画に従ってエージェントが実装を始めます。

## Plan出力例

```
# 四則演算関数の追加

## Summary
`src.app` に加減乗除の関数をそろえて追加し、既存の `add()` に合わせて型ヒントと docstring を付ける。除算は整数除算として実装し、`divide(5, 2) == 2` の挙動にする。公開再エクスポートとテストも同時に更新する。

## Key Changes
- [`/Users/xxx/development/sample_codex/src/app.py`](/Users/xxx/development/sample_codex/src/app.py)
  - 既存の `add(a: int, b: int) -> int` を維持。
  - `subtract(a: int, b: int) -> int` を追加。
  - `multiply(a: int, b: int) -> int` を追加。
  - `divide(a: int, b: int) -> int` を追加し、`//` を使った整数除算にする。
  - 4関数すべてに簡潔な docstring を付け、型ヒントを統一する。
- [`/Users/xxx/development/sample_codex/app.py`](/Users/xxx/development/sample_codex/app.py)
  - `src.app` から新関数も import し、`__all__` に追加してトップレベル互換を保つ。
- [`/Users/xxx/development/sample_codex/tests/test_app.py`](/Users/xxx/development/sample_codex/tests/test_app.py)
  - `subtract`, `multiply`, `divide` の正常系テストを追加。
  - `divide` は整数除算の期待値で検証する。
  - ゼロ除算は Python 標準挙動に合わせて `ZeroDivisionError` を期待するテストを追加する。

## Public Interface
- 追加する公開関数:
  - `subtract(a: int, b: int) -> int`
  - `multiply(a: int, b: int) -> int`
  - `divide(a: int, b: int) -> int`
- `app.py` の再エクスポート対象にも同じ関数群を含める。

## Test Plan
- `add(1, 2) == 3`
- `subtract(5, 3) == 2`
- `multiply(4, 3) == 12`
- `divide(5, 2) == 2`
- `divide(6, 3) == 2`
- `divide(1, 0)` で `ZeroDivisionError`
- 仕上げに `uv run ruff check .` と `uv run pytest` を実行して通過確認する。

## Assumptions
- 入力は既存 `add()` に合わせて `int` に限定する。
- 除算はユーザー指定に従い整数除算を採用する。
- `main.py` の表示文言は今回の対象外で、四則演算関数の追加とテスト整備に留める。
```

# 4. CodexにPluginsを追加する

## Agent Skills

Pluginsについて説明する前に、Agent Skillsに触れる必要があります。Agent Skillsは、エージェントにドメイン固有の専門知識を与えるための再利用可能なファイルベースリソースです。再利用できるので、誰かが一度作成したSkillsをそのまま自分のプロジェクトに適用することもできます。

Skillsを使用することで、短い指示文でもSkillsからコンテキストを補うことで毎回意図した通りの方法でタスクを実行してくれるようになります。チーム開発においても、Skillsを使用することでメンバーによって品質がばらつくことを防ぐことができます。

Codexでも、このSkillsを使用することができます。  
参考URL: <https://developers.openai.com/codex/skills>

### Skillsの例

```
---
name: gh-address-comments
description: gh CLIを使用して、現在のブランチのGitHubプルリクエスト（PR）に対するレビューコメントやIssueコメントへの対応を支援します。最初にghの認証を確認し、ログインしていない場合はユーザーに認証を促します。
metadata:
  short-description: GitHub PRレビューのコメントへの対応
---

# PRコメントハンドラー

現在のブランチに対応するオープンなPRを見つけ、gh CLIを使ってそのコメントに対応するためのガイドです。すべての `gh` コマンドは、ネットワークアクセス権限を昇格させた状態で実行してください。

前提条件：`gh` が認証されていることを確認してください（例えば、一度 `gh auth login` を実行します）。その後、権限を昇格させた状態（workflowおよびrepoスコープを含める）で `gh auth status` を実行し、`gh` コマンドが成功するようにします。サンドボックス環境で `gh auth status` がブロックされる場合は、`sandbox_permissions=require_escalated` を指定して再実行してください。

## 1) 対応が必要なコメントの確認
- `scripts/fetch_comments.py` を実行して、PR上のすべてのコメントとレビューのスレッドを表示します。

## 2) ユーザーへの確認
- すべてのレビューのスレッドとコメントに番号を振り、修正に必要な作業の簡単な概要を提示してください。
- どの番号のコメントに対応すべきか、ユーザーに尋ねてください。

## 3) ユーザーがコメントを選択した場合
- 選択されたコメントに対する修正を適用します。

注記：
- 実行中に認証エラーやレート制限の問題が発生した場合は、ユーザーに `gh auth login` で再認証するよう促してから、やり直してください。
```

## Agent Plugins

Skillsと、Skills内で定義されるタスクを実行するために必要なツール類をパッケージにしたものがAgent Pluginsという概念です。例えば、SkillsでSlackに返信する際の文章スタイルを定義していたとして、これまではSlackアプリとの接続設定は自分でやっていましたが、PluginsではSlackアプリとの連携まで自動でやってくれるようなイメージです。（まさにこの記事を執筆している最中に！)CodexでもこのPluginsが使えるようになったというリリースが出ました。今後はCodexでもPluginsを使うのが主流になっていくと思います。  
参考URL: <https://developers.openai.com/codex/plugins>

画像の通り、設定画面からPluginsやSkillsを追加できるようになっています。PluginsやSkillsは自分で作成することもできます。現時点では、公式で用意されたものか自分で作成したもののどちらかを追加できるみたいです。
