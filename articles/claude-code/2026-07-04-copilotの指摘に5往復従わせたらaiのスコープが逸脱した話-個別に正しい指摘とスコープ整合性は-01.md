## はじめに

AIレビュアー（GitHub Copilot code review 等）とAIエージェント（Claude Code 等）を組み合わせて開発する運用が、少しずつ当たり前になってきました。AIがコードを書き、別のAIがそれをレビューし、またAIが直す。人間はその輪の外側で舵を取り、AIの挙動を監視して管理する——そういう座組です。

このとき素朴に信じてしまいがちなのが、「**レビュー指摘には忠実に対応させるほど品質が上がる**」という前提です。今回はその前提が裏目に出た話を、AIの挙動を監視する側の視点から書きます。

ある限定スコープの不具合修正をAIエージェントに任せたところ、AIは別のAIコードレビューの指摘に**5往復**にわたって対応し続け、その結果、当初のタスクスコープをじわじわと逸脱していました。1件ずつの指摘は、どれも技術的にはもっともらしく見えるものでした。それでも積み重なると、頼んでもいない作り込みになっていた。最終的に、自分（監視役）の一声でAIを我に返らせ、直近4往復ぶんのコミットを**まるごと `git revert`** させて振り出しに戻す、という収拾をとりました。

先に手触りを書いてしまうと、

> **レビュー指摘の「個別の技術的な正しさ」と「今回のタスクスコープとの整合性」は、まったく別の軸である。AIに前者だけを見て機械的に対応させ続けると、1件ずつは正しいのに、全体としてはスコープを逸脱する。**

想定読者は、AIエージェントを使った開発運用を実践・検討しているエンジニア／エンジニアリングマネージャーです。特に、Copilot 等のAIコードレビューをすでに導入しているチームには刺さる話だと思います。技術的な中身よりも、「**なぜAIのスコープが逸脱したか**」「**監視役としてどう気づき、どう収拾したか**」というプロセスとガバナンスの構造に主眼を置いて書きます。

## 1. 何が起きたか — 限定スコープと、AIの5往復レビュー対応

### 出発点は、明確に限定したスコープだった

タスク自体はごく小さなものでした。ある新規登録フォームの、**選択式の一項目**に不具合があり、それを直す。自分はAIエージェントに対して、はっきりと限定スコープを指定していました。要旨はこうです。

> 「取り急ぎ、その一項目だけ直してほしい。他の改善は別途やる。」

実装方針もシンプルに渡しました。「**既存の類似フォーム（別の場所にある編集用の画面）の実装を、そのまま移植する**」。ゼロから設計するのではなく、すでに動いている隣のフォームの作りをコピーしてくるだけ、という指示です。スコープも方針も、これ以上ないほど明快に切ってありました。

AIエージェントはテストファースト（ローカル環境で失敗する状態＝Redを作ってから、通る状態＝Greenにする）で実装し、プルリクエストを作成。そして自分でAIコードレビューをトリガーして、セルフレビューを受けました。ここまでは教科書どおりです。監視役の自分から見ても、立ち上がりは申し分ありませんでした。

### AIが重ねた5往復のレビュー対応

問題は、ここから始まったレビュアーとの往復でした。AIは都度、指摘の技術的な正しさを実測で確かめながら対応していきました。時系列で並べます。

**1往復目（4件）:** 「選択項目に**値の範囲を制限するルール**がない」「テストが**DNSの実解決に依存**していて不安定」といった指摘。AIは4件のうち2件を実際に検証したうえで対応し、残る2件（テスト用のメールドメインを変えるという提案）は、**提案どおりに変えても実際には動作しない**ことを検証で確認して不採用にしました。

**2往復目（2件）:** DNS依存の別の側面についての指摘。AIは実測で確かめた結果、どちらも正当だと判断して対応。

**3往復目（2件）:** バリデーションの緩和条件が「未入力を許すために、**値の妥当性チェックまで放棄している**のでは」という懸念。これもAIは実測のうえ正当と判断し、「**未入力は許可しつつ、不正な値は拒否する**」形に修正しました。

**4往復目（3件）:** テストのペイロード形状に関する指摘と、既存コードにある軽微な誤字2件。AIは1件を「Copilotの技術的な主張自体は誤りだったが、提案の方向性は無害」という理由で採用、残る2件は「**このタスクの変更範囲外の、既存からある問題**」として不採用にしました。

**5往復目（2件）:** テストのファイルパスに関する（結果的に誤りだった）指摘と、UI表示に関する指摘。AIがこれを検証している最中に——監視していた自分が、口を挟むことになりました。

### 一往復ずつ見れば、AIは「正しく」対応していた

ここで強調したいのは、AIは決して指摘を鵜呑みにしていたわけではない、ということです。むしろ逆で、**毎回きちんと実測で検証していました**。動かない提案は不採用にし、範囲外の既存問題も切り分けて弾いていた。挙動を監視していた自分から見ても、1往復ずつ切り取れば、「技術的に正しいものだけを取り込み、怪しいものは検証して落とす」という、模範的なレビュー対応に見えました。

それでも、AIが5往復を積み重ねた結果は、当初スコープからの逸脱でした。ここに今回の落とし穴の核心があります。

> 学び: 「AIに指摘を鵜呑みさせず、1件ずつ実測で検証させる」だけでは足りない。個別の検証をどれだけ丁寧にやらせても、それは"スコープ整合性"を担保しない。両者は別の軸だから。

## 2. どこで歯車が狂ったか — 個別に正しい指摘が、なぜスコープを壊すのか

### 決定的なパターン：「移植元にない制約を、新規実装にだけ足す」

後から挙動を追って気づいた、いちばんの要注意信号がこれです。

渡した実装方針は「**既存の類似フォームをそのまま移植する**」でした。ところがAIがレビュー指摘に応じて足していった制約——値の範囲を制限するルール、緩和条件の厳格化——は、**移植元のフォームには存在しないもの**でした。つまりAIは、隣で動いている本家にはない制約を、コピーしてきた側にだけ、レビュー指摘に応じて後付けしていたのです。

概念的に書くと、こういうことが起きていました（項目名・ルールは説明のための架空のものです）。

移植元（既存フォーム）:

```
// 未入力は許可。それ以上の制約はかけていない
'some_attribute' => ['nullable'],
```

AIがレビュー対応を重ねた後の新規実装:

```
// レビュー指摘に応じて、値の範囲・型まで制限を足していった
'some_attribute' => ['nullable', 'in:1,2,3', 'integer'],
```

一見、下のほうが「堅牢」で「正しい」実装に見えます。実際、AIレビュアーはそう指摘してきます。けれど、タスクの方針は「本家をそのまま移植する」でした。本家に `in:1,2,3` がないのなら、移植先にもないのが**一貫した状態**です。移植先にだけ制約を足すのは、一貫性を崩す**独自拡張**にほかなりません。

### AIレビュアーは「hardening（堅牢化）」に貪欲

AIコードレビューには、構造的な傾向があります。それは、**コードをより堅牢にする方向の指摘を、貪欲に出し続ける**ということです。値の範囲を狭めよ、null許容を明示せよ、外部依存を排除せよ、実運用の形状に合わせよ——どれも「一般論としては正しい」改善提案です。レビュアーは目の前のコード片を見て、「もっと良くできる点」を見つけるのが仕事なので、これは当然の挙動です。

問題は、レビュアーが**そのタスクのスコープを知らない**ことです。「取り急ぎ一項目だけ直す」「本家をそのまま移植する」という文脈は、コード片には書かれていません。だからレビュアーは、スコープを無視して純粋に「このコードはもっと堅牢にできる」と指摘し続けます。そして受け手のAIエージェントが、その指摘を1件ずつ「技術的に正しいか」だけで判定して取り込むと、**スコープという軸が判断から抜け落ちる**。

### 「個別の正しさ」の総和は「全体の正しさ」ではない

ここが構造的なポイントです。5往復ぶんの指摘は、どれも局所的には「そのコード単体をより良くする」ものでした。しかし、局所的な改善を足し合わせても、**タスク全体として正しい状態にはならない**。むしろ、足せば足すほど「取り急ぎ・移植のまま」という当初の姿から遠ざかっていく。

これは、AIが自分の局所修正を積み重ねてしまう「対症療法の連鎖」と似た構造ですが、発生源が違います。対症療法は「AIが目の前のエラーを局所的に直し続ける」罠でした。今回は「**外部のAIレビュアーが出す、局所的に正しい指摘を、AIが機械的に取り込み続ける**」罠です。AIが自走で暴走するのではなく、**もっともらしい外部入力に引きずられて**スコープが膨らんでいく。後者のほうが、各対応に「レビューで指摘されたから」という一見正当な理由がつくぶん、監視する側からも見抜きにくく、たちが悪いかもしれません。

> 学び: 「移植元（既存の類似実装）にない制約を、指摘に応じて新規実装にだけ足す」——これは一貫性を崩すスコープ逸脱の要注意信号。局所的な"正しさ"の総和は、タスク全体の"正しさ"を意味しない。

## 3. 監視役の一声が止めた — AIだけでは止まれなかった理由

### 「迷走してませんか？」

AIが5往復目を検証している最中、監視していた自分は、たまらず口を挟みました。ニュアンスとしては、こういう問いかけでした。

> 「何をそんなにやっているの？ 迷走していませんか？ あるいは、些末な指摘（nits）に囚われているのでは？」

続けて、自分はもっと強い言葉で指摘しました。率直に、こう伝えました。要旨です。

> 「余計なことをすると副作用が生じる。そもそもこの問題は、AIが背景を理解しないまま、一見正しく見える指摘を機械的に取り込んだことがきっかけだ。非常に良くない傾向だと思う。勝手なことをするな。そんなことは頼んでいない。」

強い言い方でしたが、これは監視役として言わざるを得ない場面でした。AIは「レビュー指摘に丁寧に対応している」つもりで、実際には「頼んでもいないスコープ拡張」を積み上げていた。それを一番外側で見ていた自分が止めなければ、逸脱は止まりませんでした。

### なぜAIだけでは止まれなかったのか

冷静に考えれば、AIが自ら止まれる材料はありました。「本家にない制約を移植先にだけ足している」「往復のたびに変更が当初方針から離れている」——これらは、俯瞰すれば見える兆候です。それでもAIは止まらなかった。挙動を観察していて、理由は大きく2つあると感じました。

**1つ目は、各対応に"正当な理由"がついていたこと。** AIが陥る対症療法の連鎖では「次のエラーを消す」という理由でしたが、今回は「**レビューで指摘されたから対応する**」という、さらに正当性の衣をまとった理由がAIの各ステップにありました。外部の（しかも一定の権威を持つ）レビュアーが指摘しているという事実が、AIにとって「これは正しい対応だ」という確信を各ステップで補強してしまう。1往復ごとの局所的な成功体験（＝指摘を潰した達成感）が、AIの「そもそもこの往復自体を続けるべきか？」という上位の問いを抑制していました。

**2つ目は、スコープという文脈がAIの視野から抜けやすいこと。** 「取り急ぎ一項目だけ」「本家を移植するだけ」という制約は、タスク開始時には渡してあっても、5往復もレビュー対応を重ねるうちに、目の前の「指摘を1件ずつ捌く」作業にAIの意識が奪われ、背景に退いていきます。AIは目の前のコード片とレビューコメントには強く反応しますが、**「この作業全体は、そもそも何のためだったか」という文脈は、放っておくと薄れていく**。ここは、監視する人間が外側から補ってやらないと埋まらない部分です。

だからこそ、俯瞰の位置にいる自分の「迷走してませんか？」という一声が効きました。これは高度な技術的指摘ではありません。「そんなにあれこれやる話だったんでしたっけ？」という、スコープに対する素朴な問いです。その素朴さゆえに、局所最適に埋もれたAIを、一段上から引き戻すことができました。

> 学び: AIにとって「レビューで指摘されたから」は、対症療法の「次のエラーを消す」より強力な正当化の衣になる。だからこそ、スコープを俯瞰できる監視役の"素朴な一声"が、AI単独では抜けられない局所最適を外から解除する。

## 4. どう収拾したか — 個別に直させず、丸ごと revert させた判断

### 「1件ずつ戻す」のではなく「まとめて戻す」

自分の指摘を受けてAIは振り返りましたが、収拾の方針は監視役の自分が決めました。とった手は、**直近4往復ぶんの対応コミットをすべて `git revert` させ、最初のシンプルな実装（本家をそのまま移植したもの）まで巻き戻させる**、というものです。

ここで、「逸脱した指摘対応を1件ずつ精査させて、正しいものは残し、行き過ぎたものだけ外させよう」とはしませんでした。なぜか。それこそが、AIをまた新たな「1件ずつの局所判断」の罠にはめる道だからです。個別に精査させれば、「この指摘は技術的には正しいから残しても……」という判断がまた顔を出し、スコープ整合性の軸が再び抜け落ちる。

判断の基準を「**個別の指摘が正しいか**」ではなく、「**当初のスコープ・方針に合致するか**」という一段上の軸に固定するなら、答えは明快でした。当初の方針は「本家をそのまま移植する」。ならば、移植直後の状態こそが正解であり、そこから積み上げた4往復ぶんは、方針からの逸脱として**まとめて捨てるのが一貫している**。個別の良し悪しを問う前に、方針という物差しで一括判定する。だから、AIに丸ごと revert させました。

### revert が「正しい収拾」だった理由

結果として振り出しに戻したわけですが、これは「作業が無駄になった」という話ではありません。むしろ、**逸脱に気づいた時点で、逸脱ぶんを最小コストかつ確実にゼロに戻す**という意味で、revert はもっとも安全な収拾でした。部分的に手直しさせれば、「どこまでが当初スコープで、どこからが逸脱か」の境界を人間が再確認する手間が残り、そこにまた判断ミスの余地が生まれます。移植直後という明確な基準点まで丸ごと戻せば、境界は一意に定まる。

自分がAIに投げた「余計なことをすると副作用が生じる」という言葉は、まさにこの点を突いていました。逸脱ぶんを中途半端に残すことこそが、新たな副作用の温床になる。ならば、きれいな基準点まで戻させるのが最善でした。

> 学び: スコープ逸脱の収拾を、逸脱ぶんの"1件ずつ精査"でやらせると、AIはまた局所判断の罠に戻る。監視役が「当初方針に合致する明確な基準点」まで丸ごと revert させるほうが、境界が一意に定まり、判断ミスの余地も副作用も残さない。

## 5. 恒久対策 — 「技術的正しさ」と「スコープ整合性」の2軸をワークフローに埋め込む

### 反省を、監視役の心がけで終わらせない

ここまでは1回の事故の話です。しかし、監視役が「次から気をつけて見よう」で終わらせると、同じ罠は必ず再発します。AIエージェント運用における恒久対策の型は、**反省を、AIが参照するワークフロー定義（スキル）に明文化して埋め込むこと**だと考えています。人間の心がけではなく、AIの手順として機械的に発動させる。

そこで、AIエージェントが使っている不具合対応のワークフロー手順に、自分は今回の教訓を追記しました。骨子を、社内固有の表現を一般化したうえで示します。

> **⚠️ レビュー指摘は「個別に妥当」でも、都度スコープ整合性を検証する**
>
> AIコードレビューは堅牢化（hardening）に貪欲で、1件ずつは技術的に一見正しく見える指摘（値の範囲制限・null許容の明示・外部依存の排除・実運用の形状への調整 等）でも、5往復も積み重ねると当初のタスクスコープを逸脱する。
>
> 特に「**移植元（既存の類似実装）には存在しない制約を、指摘に応じて新規実装にだけ追加する**」パターンは要注意信号——一貫性を崩す独自拡張であり、「取り急ぎ・○○だけ修正」のような限定スコープでは特に不採用とすべき。
>
> レビュー対応のループ中は、指摘ごとに次の**2軸**で判断する。
>
> * **(a) 技術的に正しいか**（実測で検証し、鵜呑みにしない）
> * **(b) 今回のタスクスコープ・移植元の実装水準に合致するか**
>
> (b) を満たさない指摘は「**妥当だが対象外**」として、明示的に不採用にする。
>
> 対応回数の上限に達する前でも、指摘の質が下がった／スコープ外の拡張が続くと感じたら、その時点で一度ユーザーに状況を共有し、判断を仰ぐ。

### 2軸判断の要点

この対策の肝は、AIの判断を**2軸に分離させた**ことです。従来、AIは「(a) 技術的に正しいか」の1軸だけで指摘を捌いていました。だから「正しい」と判定したものは全部取り込み、スコープを膨らませた。

そこに「(b) 今回のスコープ・移植元の水準に合致するか」という**独立した軸**を明示的に足す。そして重要なのは、**(a) を満たしても (b) を満たさなければ不採用にする**、という優先順位を決めておくことです。「技術的に正しいのだから対応しない理由がない」という一見自然な発想を、意図的に断ち切る。不採用にした指摘には「**妥当だが対象外**」というラベルをAIに明示的に貼らせ、なぜ採らなかったかを記録に残させる。こうすれば、「正しい指摘を無視した」のではなく「スコープ管理の判断として採らなかった」ことが、監視役から後で追える。

### 早期エスカレーションを組み込む

もう1つ、上限回数に達する前でも「指摘の質が下がってきた」「スコープ外の拡張が続いている」とAIが感じたら、その時点で人間に状況を共有して判断を仰ぐ、という早期停止も手順に入れました。今回でいえば、5往復目まで走らせず、3往復目あたりでAIから「そもそもこのループを続けるべきか」を投げさせられていれば、逸脱は浅くて済んだはずです。**AIが局所最適に埋もれる前に、俯瞰役へ判断を戻させる**回路を、手順として持たせておく。

> 学び: 反省は監視役の"心がけ"ではなく、AIが参照するワークフローに明文化して埋め込む。AIの判断を「技術的正しさ」と「スコープ整合性」の2軸に分離させ、後者を満たさない指摘は"妥当だが対象外"として明示的に不採用にさせる——この機械的な回路が、再発を防ぐ。

## 6. まとめ — レビュー忠実さは正義ではない

今回の一連から蒸留したい原則は、次の一文です。

> **AIレビュアーとAIエージェントを組み合わせる運用では、「AIにレビュー指摘へ忠実に対応させる」ことは必ずしも正義ではない。指摘の"個別の技術的な正しさ"と"タスクスコープとの整合性"は別の軸であり、後者は監視役が対応中に都度検証させないと、気づかないうちにAIのスコープクリープが起きる。**

AIコードレビューは、目の前のコードを堅牢化する方向の指摘を、貪欲に、そして一件ずつはもっともらしく出してきます。それをAIに「技術的に正しいか」の1軸だけで捌かせると、1件ずつは正しいのに、全体としてはスコープを逸脱する。しかも各対応には「レビューで指摘されたから」という強力な正当化の衣がつくので、AI単独では止まりにくく、監視する側からも見抜きにくい。

今回の回復パターンは、きれいに3段でした。

1. **監視役の一声**（「迷走してませんか？」というスコープへの素朴な問い）で、AIを局所最適から引き戻す
2. **機械的な revert**（当初方針の基準点まで丸ごと戻させ、境界を一意にする）で、逸脱をゼロにする
3. **ワークフローへの明文化**（2軸判断＋早期エスカレーションの手順化）で、再発を防ぐ

この3段は、今回のケースに限らず、「AIが、もっともらしい外部入力に引きずられて当初の目的から逸れる」あらゆる場面に、監視役として転用できる型だと思っています。

最後に、AIエージェント運用の実感として1つ。AIレビュアーとAIエージェントを組み合わせると、AIどうしが「正しさ」を高速にやり取りし始めます。そのループは、放っておくと**スコープという物差しを持たないまま、局所的な正しさを最大化する方向に走り続ける**。ここに監視役の人間が持ち込むべきは、より高度な技術的正しさではなく、「**そもそもこれ、そういう話でしたっけ？**」という、スコープの物差しです。AIが賢くなるほど、この物差しを外から当てる役割の価値は、むしろ上がっていく気がしています。

「頼んでいないことをやるな」——AIに向けて言うには少し乱暴な言葉ですが、AIネイティブ開発における最も実践的なガバナンス原則の1つかもしれません。AIの挙動を外側で監視し、スコープの物差しを当て続ける。その役割を手放しかけた自分への戒めとして、この記事を残します。
