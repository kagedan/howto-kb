## Claude Code DevContainer 3ファイル解説

`.devcontainer/` の中にある 3 つのファイルが、VS Code で「コンテナを開く」と  
自動で作られる **使い捨ての Linux 環境** を組み立てている。ということは聞いてわかったのですが、実際にこれらのファイルがどのようにして作用し、ルールを決めているのかを詳しく調べた結果をまとめました。

コンテナ関連の知識が抜けていた自分にとってはとても勉強になります。

---

## 0. 全体のイメージ

まず前提の用語から。

* **コンテナ**：Docker が作る「薄い仮想マシンみたいなもの」。  
  中身はほぼ Linux PC だが、PC 1 台まるごとではなく、  
  **1 つのプロジェクト専用の使い捨て作業部屋**という感覚。
* **Dev Container**：VS Code がこのコンテナの中でエディタのサーバー部分を動かし、  
  ホスト（Windows 側）の VS Code からリモート操作する仕組み。  
  「Claude Code がホスト PC を壊す可能性」を隔離する目的で使う。

3 ファイルの関係は次のようになっています。

```
devcontainer.json   ← VS Code への指示書（どんな部屋を作るか）
      │
      │ "build.dockerfile" で参照
      ▼
Dockerfile          ← 部屋（コンテナ）の組み立てレシピ
      │
      │ COPY でコンテナ内にコピーされ、
      │ devcontainer.json の postStartCommand から呼ばれる
      ▼
init-firewall.sh    ← 部屋の出入り口に立つ「関所」の設定スクリプト
```

ざっくり:

| ファイル | 何を決めるか | いつ動くか |
| --- | --- | --- |
| `devcontainer.json` | VS Code の動き方、拡張機能、ボリューム、ユーザーなど | コンテナを開くたび |
| `Dockerfile` | コンテナの中身（OS、入れるツール、ユーザー設定） | 最初の「ビルド」1 回だけ |
| `init-firewall.sh` | 通信を許可する相手と拒否する相手（ネットワークの関所） | コンテナ起動のたび |

## 1. `devcontainer.json` — VS Code への指示書

VS Code の Dev Containers 拡張が最初に読むファイル。  
「どういうコンテナを作ってほしいか」「起動後に何をしてほしいか」を JSON で指定します。

### 各項目の意味

```
"name": "Claude Code Sandbox"
```

* VS Code 左下などに表示される、この環境の**表示名**。  
  機能には影響しない。「Claude Code 用の砂場」くらいの意味。

```
"build": {
  "dockerfile": "Dockerfile",
  "args": {
    "TZ": "${localEnv:TZ:America/Los_Angeles}",
    "CLAUDE_CODE_VERSION": "latest",
    "GIT_DELTA_VERSION": "0.18.2",
    "ZSH_IN_DOCKER_VERSION": "1.2.0"
  }
}
```

* このコンテナは **同じフォルダにある `Dockerfile` を使ってビルドする**、という宣言。
* `args` は Dockerfile に渡す材料（引数）。
  + `TZ`：タイムゾーン。`${localEnv:TZ:America/Los_Angeles}` は  
    「ホストの環境変数 `TZ` があればそれを使い、なければロサンゼルス時間」という意味。
  + `CLAUDE_CODE_VERSION`：コンテナに入れる Claude Code のバージョン。`latest`=最新。
  + `GIT_DELTA_VERSION`：git の差分をきれいに色分け表示するツール `delta` のバージョン。
  + `ZSH_IN_DOCKER_VERSION`：Docker 内で zsh（高機能シェル）を整えるスクリプトのバージョン。

```
"runArgs": ["--cap-add=NET_ADMIN", "--cap-add=NET_RAW"]
```

* `docker run` に渡す追加オプション。
* `NET_ADMIN` / `NET_RAW` は **ネットワーク系の管理権限**。  
  これがないと `iptables`（通信ルール変更コマンド）を実行できないので、  
  後述の `init-firewall.sh` が動かない。**関所を作るための権限**と思えばよい。

```
"customizations": { "vscode": { "extensions": [...], "settings": {...} } }
```

* コンテナに接続した VS Code に適用する設定。
* `extensions`：自動でインストールする拡張機能。
  + `anthropic.claude-code`：Claude Code 本体。
  + `dbaeumer.vscode-eslint`：JavaScript/TypeScript の文法チェッカー。
  + `esbenp.prettier-vscode`：コードの整形ツール。
  + `eamodio.gitlens`：git 履歴を見やすくする拡張。
* `settings`：
  + `editor.formatOnSave: true`：**保存時に自動整形**。
  + `editor.defaultFormatter`：整形には Prettier を使う。
  + `editor.codeActionsOnSave.source.fixAll.eslint: "explicit"`：  
    保存時に ESLint の自動修正を走らせる（明示的に有効化）。
  + `terminal.integrated.defaultProfile.linux: "zsh"`：  
    VS Code のターミナルを開くとデフォルトで zsh になる。
  + `terminal.integrated.profiles.linux`：bash と zsh の 2 つを選べるよう登録。

* コンテナ内で **どのユーザーとして作業するか**。
* ベースイメージ `node:20` に最初から存在する `node` ユーザーを使う。
* root（管理者）ではなく一般ユーザーで動かすことで、  
  万一コマンドが暴走してもシステム全体は壊れにくい。

```
"mounts": [
  "source=claude-code-bashhistory-${devcontainerId},target=/commandhistory,type=volume",
  "source=claude-code-config-${devcontainerId},target=/home/node/.claude,type=volume"
]
```

* コンテナに外部の保存領域（**Docker ボリューム**）をつなぐ設定。
* コンテナは使い捨てなので、中のファイルは基本消える。  
  でも以下の 2 つは消えると困るので、専用の保存庫にしまう。
  + `/commandhistory`：**ターミナルでの入力履歴**（↑キーで出てくるやつ）。
  + `/home/node/.claude`：**Claude Code のログイン情報や設定**。  
    これが消えると毎回ログインし直しになる。
* `${devcontainerId}` はこのプロジェクト固有の ID に置換され、  
  他プロジェクトと履歴・設定が混ざらない。

```
"containerEnv": {
  "NODE_OPTIONS": "--max-old-space-size=4096",
  "CLAUDE_CONFIG_DIR": "/home/node/.claude",
  "POWERLEVEL9K_DISABLE_GITSTATUS": "true"
}
```

* コンテナ内で常にセットされる **環境変数**。
  + `NODE_OPTIONS=--max-old-space-size=4096`：Node.js が使えるメモリ上限を 4GB に。  
    Claude Code や各種ビルドがメモリ不足で落ちにくくなる。
  + `CLAUDE_CONFIG_DIR`：Claude Code に「設定ファイルはここを見てね」と伝える。  
    上の mounts で永続化している場所を指している。
  + `POWERLEVEL9K_DISABLE_GITSTATUS=true`：  
    zsh の見た目テーマ（powerlevel10k）が毎回 git 状態を調べて重くなるのを  
    オフにする。**見た目を軽量化する設定**。

```
"workspaceMount": "source=${localWorkspaceFolder},target=/workspace,type=bind,consistency=delegated",
"workspaceFolder": "/workspace"
```

* **Windows のプロジェクトフォルダをコンテナ内の `/workspace` として直接見せる**設定。
* `type=bind` なので、Windows 側でファイルを保存するとコンテナでもすぐ反映されるし、逆も同じ。
* `consistency=delegated` は Mac 向けの速度チューニング指定（Windows では実質無視される）。
* `workspaceFolder` は VS Code が「ここを開く」と見なすパス。

```
"postStartCommand": "sudo /usr/local/bin/init-firewall.sh"
```

* コンテナが **起動したあと** に自動で実行するコマンド。
* ここで **後述のファイアウォール（関所）を張る**。
* `sudo` で root 権限で実行（ネットワーク設定変更には管理者権限が必要）。

```
"waitFor": "postStartCommand"
```

* VS Code に「**`postStartCommand` が終わるまで準備完了とみなさないで**」と指示。
* つまりファイアウォール設定が完了してから開発を始められる。  
  → 今回のエラーではこのコマンドが失敗していたため、接続が中断された。

## 2. `Dockerfile` — コンテナの組み立てレシピ

Dockerfile は **料理のレシピ**のようなもの。  
上から順に命令が実行され、1 つの「イメージ」（コンテナのひな形）ができ上がります。

### 各ブロックの意味

* 土台になるイメージ。**Node.js 20 入りの Debian Linux**。
* ここに自分たちの追加設定を積み上げる。

* `ARG`：ビルド時だけ使う引数（`devcontainer.json` の `build.args` から渡される）。
* `ENV`：コンテナ実行中ずっと残る環境変数。
* 合わせて「外から受け取ったタイムゾーンを、コンテナ内にも設定」。

```
ARG CLAUDE_CODE_VERSION=latest
```

* 後で Claude Code をインストールする時に使うバージョン指定。デフォルト `latest`。

```
RUN apt-get update && apt-get install -y --no-install-recommends \
  less git procps sudo fzf zsh man-db unzip gnupg2 gh \
  iptables ipset iproute2 dnsutils aggregate \
  jq nano vim \
  && apt-get clean && rm -rf /var/lib/apt/lists/*
```

* Debian のパッケージマネージャ `apt-get` で**必要なコマンド類を一気にインストール**。
* 主なもの:
  + `git` / `gh`：git と GitHub CLI。
  + `sudo`：一時的に root 権限を借りるコマンド。
  + `zsh` / `fzf`：高機能シェルとあいまい検索ツール。
  + `iptables` / `ipset` / `iproute2`：**ファイアウォール系**（`init-firewall.sh` で使う）。
  + `dnsutils`：`dig` など DNS 解決ツール。
  + `aggregate`：IP レンジをまとめるツール。
  + `jq`：JSON をシェルで扱うためのツール。
  + `less`, `nano`, `vim`, `man-db`, `procps`, `unzip`, `gnupg2`：一般的な作業用。
* 最後の `apt-get clean` ... はキャッシュ削除で**イメージを軽くする定番処理**。

```
RUN mkdir -p /usr/local/share/npm-global && \
  chown -R node:node /usr/local/share
```

* npm のグローバルインストール先フォルダを作り、`node` ユーザーが書けるように所有権を変更。
* これをやらないと一般ユーザーで `npm install -g` が失敗する。

```
ARG USERNAME=node
RUN SNIPPET="export PROMPT_COMMAND='history -a' && export HISTFILE=/commandhistory/.bash_history" \
  && mkdir /commandhistory \
  && touch /commandhistory/.bash_history \
  && chown -R $USERNAME /commandhistory
```

* **bash のコマンド履歴を `/commandhistory/.bash_history` に保存**する仕込み。
* `devcontainer.json` でこのフォルダをボリューム化しているので、  
  コンテナを作り直しても過去の入力履歴が消えない。

* 「この環境は Dev Container 内である」というマーク。
* 一部のツール（Claude Code など）がこれを見て挙動を切り替えることがある。

```
RUN mkdir -p /workspace /home/node/.claude && \
  chown -R node:node /workspace /home/node/.claude

WORKDIR /workspace
```

* `/workspace`（プロジェクト置き場）と `/home/node/.claude`（Claude 設定置き場）を作成。
* 所有者を `node` にしてから、デフォルト作業ディレクトリを `/workspace` に設定。

```
ARG GIT_DELTA_VERSION=0.18.2
RUN ARCH=$(dpkg --print-architecture) && \
  wget "https://github.com/dandavison/delta/releases/download/${GIT_DELTA_VERSION}/git-delta_${GIT_DELTA_VERSION}_${ARCH}.deb" && \
  sudo dpkg -i "git-delta_${GIT_DELTA_VERSION}_${ARCH}.deb" && \
  rm "git-delta_${GIT_DELTA_VERSION}_${ARCH}.deb"
```

* `git-delta` を GitHub リリースから CPU 種別に合った `.deb` パッケージで取得してインストール。
* git diff が色付きで見やすくなる。

* 以降のコマンドは **一般ユーザー `node`** として実行する切り替え。
* root でむやみに操作しない、というセキュリティ方針。

```
ENV NPM_CONFIG_PREFIX=/usr/local/share/npm-global
ENV PATH=$PATH:/usr/local/share/npm-global/bin
```

* npm で `-g`（グローバル）にインストールするときの置き場所と、  
  そこに入った実行ファイルを PATH に追加。

```
ENV SHELL=/bin/zsh
ENV EDITOR=nano
ENV VISUAL=nano
```

* デフォルトシェルを zsh に、既定のテキストエディタを nano に。
* `EDITOR` / `VISUAL` は git commit のメッセージ編集などで使われる。

```
ARG ZSH_IN_DOCKER_VERSION=1.2.0
RUN sh -c "$(wget -O- https://github.com/deluan/zsh-in-docker/releases/download/v${ZSH_IN_DOCKER_VERSION}/zsh-in-docker.sh)" -- \
  -p git -p fzf \
  -a "source /usr/share/doc/fzf/examples/key-bindings.zsh" \
  -a "source /usr/share/doc/fzf/examples/completion.zsh" \
  -a "export PROMPT_COMMAND='history -a' && export HISTFILE=/commandhistory/.bash_history" \
  -x
```

* 外部スクリプト `zsh-in-docker.sh` を使って **oh-my-zsh + powerlevel10k テーマ + プラグイン** を一括導入。
* `-p git -p fzf`：git と fzf プラグインを有効化。
* `-a "..."`：`.zshrc` に追記する行。fzf のキー割り当てと履歴保存設定。

```
RUN npm install -g @anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}
```

* **Claude Code 本体をグローバルインストール**。コンテナ内で `claude` コマンドが使えるようになる。

```
COPY init-firewall.sh /usr/local/bin/
USER root
RUN chmod +x /usr/local/bin/init-firewall.sh && \
  echo "node ALL=(root) NOPASSWD: /usr/local/bin/init-firewall.sh" > /etc/sudoers.d/node-firewall && \
  chmod 0440 /etc/sudoers.d/node-firewall
USER node
```

* ホスト側の `init-firewall.sh` をコンテナの `/usr/local/bin/` にコピー。
* 実行権限を付ける (`chmod +x`)。
* **sudoers に特別ルール**を書き込む:
  > `node` ユーザーは **パスワード無しで `init-firewall.sh` だけ** root として実行できる
* これで `devcontainer.json` の `postStartCommand` の `sudo /usr/local/bin/init-firewall.sh`  
  がパスワード入力なしに動く。最後にまた `node` ユーザーへ戻す。

**ここで `COPY` している点が重要**：  
コンテナ内の `init-firewall.sh` は**イメージに焼き込まれたコピー**。  
ホスト側の `.devcontainer/init-firewall.sh` をあとから直しても、  
**Rebuild Container しないと反映されない**のはこのため。

## 3. `init-firewall.sh` — ネットワークの関所

このシェルスクリプトは、コンテナから外に出られる通信先を **許可リスト方式** で絞り込みます。  
「Claude Code がうっかり怪しいサイトに情報を送らないように」という**安全装置**の役目。

スクリプトの流れは大きく 6 段階。

### 3.1 安全に動くためのおまじない

```
set -euo pipefail
IFS=$'\n\t'
```

* `set -e`：途中で 1 つでもコマンドが失敗したら即終了。
* `set -u`：未定義の変数を使ったら即終了。
* `set -o pipefail`：パイプ `|` の途中の失敗も見逃さない。
* `IFS=$'\n\t'`：単語区切りをスペース無しに。空白を含むパスで事故らないため。
* 合わせて **「事故ったらすぐ止まる厳格モード」**。

### 3.2 既存ルールの初期化

```
DOCKER_DNS_RULES=$(iptables-save -t nat | grep "127\.0\.0\.11" || true)

iptables -F
iptables -X
iptables -t nat -F
iptables -t nat -X
iptables -t mangle -F
iptables -t mangle -X
ipset destroy allowed-domains 2>/dev/null || true
```

* まず **Docker 内部の DNS ルール**（`127.0.0.11` 宛）だけを控えておく。
* そのあと既存のファイアウォールルールと IP セットを全部消して、**まっさらな状態**にする。
* 後で DNS ルールだけは戻す（社内 DNS が死ぬと名前解決が壊れるため）。

### 3.3 DNS ルール復元と基本ルール

```
if [ -n "$DOCKER_DNS_RULES" ]; then
    echo "Restoring Docker DNS rules..."
    iptables -t nat -N DOCKER_OUTPUT 2>/dev/null || true
    iptables -t nat -N DOCKER_POSTROUTING 2>/dev/null || true
    echo "$DOCKER_DNS_RULES" | xargs -L 1 iptables -t nat
fi

iptables -A OUTPUT -p udp --dport 53 -j ACCEPT   # DNS 問い合わせ許可
iptables -A INPUT  -p udp --sport 53 -j ACCEPT   # DNS 応答許可
iptables -A OUTPUT -p tcp --dport 22 -j ACCEPT   # SSH 送信許可
iptables -A INPUT  -p tcp --sport 22 -m state --state ESTABLISHED -j ACCEPT
iptables -A INPUT  -i lo -j ACCEPT               # 自分自身(loopback)の入力
iptables -A OUTPUT -o lo -j ACCEPT               # 自分自身の出力
```

* さっき控えた Docker の DNS ルールを復元。
* そのあと **必ず通したい通信**に穴を開ける:
  + **DNS**（ドメイン名 → IP の変換）53 番ポート。
  + **SSH**（git で使う）22 番ポート。
  + **localhost 通信**（コンテナ内の通信）。

### 3.4 許可 IP セットを作る（関所の通行手形リスト）

```
ipset create allowed-domains hash:net
```

* `allowed-domains` という名前の **IP アドレス集合**（IP セット）を作成。
* 以後、ここに追加した IP だけが外部通信を許可される。

#### 3.4.1 GitHub の IP レンジを許可

```
gh_ranges=$(curl -s https://api.github.com/meta)
... (検証) ...
while read -r cidr; do
    ...
    ipset add -exist allowed-domains "$cidr"
done < <(echo "$gh_ranges" | jq -r '(.web + .api + .git)[]' | aggregate -q)
```

* GitHub が公式に公開している `api.github.com/meta` から  
  GitHub のサーバー群の **IP アドレス帯**を取得。
* `web` / `api` / `git` 用の IP をまとめて IP セットに追加。
* `aggregate -q` で無駄に重なった範囲を 1 つにまとめる。
* `-exist` は **「既に登録済みでもエラーにしない」** フラグ。  
  これが無いと重複時にスクリプト全体が落ちる（← 今回修正した箇所）。

#### 3.4.2 よく使うドメインを名前解決して追加

```
for domain in \
    "registry.npmjs.org" \
    "api.anthropic.com" \
    "sentry.io" \
    "statsig.anthropic.com" \
    "statsig.com" \
    "marketplace.visualstudio.com" \
    "vscode.blob.core.windows.net" \
    "update.code.visualstudio.com"; do
    ips=$(dig +noall +answer A "$domain" | awk '$4 == "A" {print $5}')
    ...
    while read -r ip; do
        ...
        ipset add -exist allowed-domains "$ip"
    done < <(echo "$ips")
done
```

* リスト中のドメインを 1 つずつ `dig` で解決して IP を取り出す。
* それぞれの IP を `allowed-domains` に追加。
* ドメインの意味:
  + `registry.npmjs.org`：npm パッケージの配布元。
  + `api.anthropic.com`：**Claude 本体の API**。
  + `sentry.io` / `statsig.*`：エラー収集や機能フラグなど、Claude Code の裏側サービス。
  + `marketplace.visualstudio.com` / `vscode.blob.core.windows.net` / `update.code.visualstudio.com`：  
    VS Code の **拡張機能ダウンロードと更新**。

#### 3.4.3 ホスト PC との通信を許可

```
HOST_IP=$(ip route | grep default | cut -d" " -f3)
HOST_NETWORK=$(echo "$HOST_IP" | sed "s/\.[0-9]*$/.0\/24/")
iptables -A INPUT  -s "$HOST_NETWORK" -j ACCEPT
iptables -A OUTPUT -d "$HOST_NETWORK" -j ACCEPT
```

* コンテナのデフォルトゲートウェイ（＝ Docker のホスト側）の IP を割り出し、  
  その同じサブネット（例：`172.17.0.0/24`）との通信を許可。
* これが無いと、VS Code サーバーとホスト PC の通信が切れて操作できなくなる。

### 3.5 デフォルトを「全部拒否」にしてから、許可リストだけ通す

```
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT DROP

iptables -A INPUT  -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

iptables -A OUTPUT -m set --match-set allowed-domains dst -j ACCEPT
iptables -A OUTPUT -j REJECT --reject-with icmp-admin-prohibited
```

* まず全方向の通信を**基本 DROP（黙って破棄）**。
* 既に確立済み／応答パケットは通す（普通の通信を壊さないため）。
* 外向き通信は **`allowed-domains` に入っている宛先のみ許可**。
* それ以外は即 REJECT し、アプリ側に「拒否されました」とすぐフィードバック。
* これで **許可リスト方式のホワイトリスト**が完成する。

### 3.6 動作確認（関所が本当に機能してるか）

```
if curl --connect-timeout 5 https://example.com >/dev/null 2>&1; then
    echo "ERROR: ..."; exit 1
else
    echo "Firewall verification passed ..."
fi

if ! curl --connect-timeout 5 https://api.github.com/zen >/dev/null 2>&1; then
    echo "ERROR: ..."; exit 1
fi
```

* `example.com` は許可リストに無い → **到達できてしまったら失敗**（設定ミス）。
* `api.github.com` は許可リストに入っている → **到達できなければ失敗**。
* 両方の期待通りならファイアウォール設定は成功。

## まとめ

* **`devcontainer.json`** は「どんな部屋を VS Code に用意してほしいか」の指示書。  
  ビルド引数・拡張機能・ボリューム・起動後コマンドなどをここで宣言する。
* **`Dockerfile`** はその部屋をゼロから組み立てるレシピ。  
  OS、開発ツール、Claude Code、シェル設定、そしてファイアウォールスクリプトを焼き込む。
* **`init-firewall.sh`** はコンテナ起動のたびに走る安全装置。  
  許可リストに載ったサーバーとしか通信できないよう `iptables` と `ipset` で関所を張る。

3 つが揃うことで、「Claude Code が壊してよい隔離された Linux 環境」＋  
「でも必要な相手とはちゃんと通信できる」状態が作られます。
