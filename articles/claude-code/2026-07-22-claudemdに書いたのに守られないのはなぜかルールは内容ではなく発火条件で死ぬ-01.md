## まず、あなたの書き方だけの問題ではない

CLAUDE.mdにルールを書いた。守られない。表現を変えた。まだ守られない。もっと強い言葉で書いた（「絶対に」「必ず」「IMPORTANT」）。それでも守られない。

まず、問題の構造から見ていきます。そもそも「書いておけば守られる」という前提は成り立ちません。

* HumanLayerの分析（[Writing a good CLAUDE.md](https://www.humanlayer.dev/blog/writing-a-good-claude-md)）によると、指示が増えたとき、LLMは**末尾の指示だけを無視するのではなく、すべての指示に従いにくくなります**。フロンティアモデルでも、指示が150〜200個ほどになると遵守率が直線的に低下します。しかも、Claude Code自身のシステムプロンプトには、すでに約50個の指示が含まれています。つまり、あなたのCLAUDE.mdに書いたルールは、最初から数多くの指示と注意を奪い合うことになります。
* さらに、Claude Codeのハーネス（実行基盤）はCLAUDE.mdの内容をコンテキストに加える際、**「このコンテキストは現在のタスクに関係しない可能性がある。強く関係しない限り反応するな」という趣旨のsystem reminderを自動で添えます**。これは実際のセッションログでも確認できます。つまりあなたのルールは、モデルに「無視してよい候補」として渡されているのです。

CLAUDE.mdは設定ファイルではありません。**ターンごとに、ほかのあらゆるコンテキストと優先順位を争う、ただのテキスト**です。この前提に立つと、考えるべきことが変わります。「どう書けば守られるか」ではなく、\*\*「競争の中で、必要な瞬間にだけ勝ち残るルールは、どのような形をしているか」\*\*です。

## 「短く書け」は延命であって根治ではない

検索するとよく出てくる対策があります。短くする、200行以内に収める、ファイルを分割する、見出しだけを残す。どれも効果はあります。ただし、これらは**一つのファイルに平たく並んだ指示の量を減らしているだけ**です。

この構成の問題は、すべてのルールが毎回まとめて読み込まれ、そのたびに注意を奪い合うことです。ファイルを短くすれば競合するルールが減り、遵守率は上がります。しかし、ルールが増えれば再び同じ問題にぶつかります。根本的な解決策は後で説明するとして、まずは一つのファイルにまとめたままでも効果のある書き方を紹介します。

筆者はこの半年、AIエージェントに外部記憶システムを持たせ、継続的に運用してきました。ルールは記憶ノードとして保存され、**どのルールがいつ読まれたかをアクセスログで追跡できる**環境です。そのため、「どのルールが使われ、どのルールが使われなかったか」を継続して観察できます。そこから分かったことが3つあります。

## 観測1：「気づいたら〜する」系のルールは、発火率が最も低い

これが一番の収穫でした。次のようなルール、CLAUDE.mdに書いていませんか。

```
- 冗長なコードを書き始めたと気づいたら、シンプルに書き直すこと
- 話が脱線してきたら、元のタスクに戻ること
- 推測で実装していると感じたら、手を止めて質問すること
```

このような**自己観察を求めるルール**は、ほかのルールに比べて発火率が際立って低いグループでした。正確に言えば、定期的に記憶を棚卸しするときには読まれます（そして、たいていはそこで削除対象になります）。しかし、**本来発火すべき瞬間、つまり失敗しかけているその場で自発的に読まれることは、ほとんどありません**。

理由は単純です。**失敗の渦中にいるエージェントには、自分の状態を振り返る余裕がない**からです。ルールが発火すべき瞬間、モデルの注意は目の前の作業に向ききっています。つまり、「気づいたら」と書いても、その「気づく」きっかけ自体がありません。

実際に機能したルールの発火条件は、例外なく次の2種類でした。

1. **入力から判別できるシグナル**：「ユーザーが〜と言ったら」「エラーメッセージに〜が含まれていたら」
2. **特定の操作の直前**：「〜を実行する直前に」「〜をコミットする直前に」

どちらも、失敗が起きる**前**に、モデルの外側から確認できる出来事をきっかけに発火します。たとえば、次のように書き換えられます。

| ✗ 発火しにくい | ○ 発火しやすい |
| --- | --- |
| ハードコードに気をつけること | 新しい定数を書く直前に、既存の設定ファイルを確認すること |
| 重複コードを書いていたら共通化すること | 3つ目の類似関数を書く直前に、共通化を提案すること |
| 推測で実装しないこと | 仕様に書かれていない挙動に触れる実装の前に、必ず質問すること |

左列にあるのは心構えで、右列にあるのは具体的なきっかけです。**ルールが実行されるかどうかを左右するのは、内容の正しさではなく、発火条件が観測できる出来事に結びついているかどうか**でした。

## 観測2：定員を設けないと、全部が一緒に死ぬ

HumanLayerの「一様劣化」は、運用でもそのまま観測されました。ルールを増やすと、特定のルールが死ぬのではなく、**全体の遵守率が静かに下がります**。劣化が一様であることの怖さは、壊れた瞬間に気づけないことです。

筆者の記憶システムでは、優先度ごとに件数の上限を設けています。最優先のグループはライブラリ全体で5件まで、その次は15件までです。**新たに1件の優先度を上げるには、既存のどれかを下げなければなりません**。この制限があることで、「これは本当に常時読み込ませる価値があるのか」を毎回吟味するようになります。

CLAUDE.mdに置き換えて考えると、**常時読み込ませるルールの上限を先に決める**ということです（たとえば10件）。上限を超えたら、追加するのではなく既存のルールと入れ替えます。指示が増えるほどすべての遵守率が下がる以上、「全部重要」は「全部無視」と同じ意味です。

## 観測3：矛盾したルールを併存させると、挙動がセッションごとに揺れる

古いルールを消さずに新しいルールを書き足していくと、互いに矛盾するルールが自然に生まれます。人間は「新しい方を優先してくれるはず」と期待しますが、モデルにはルールの新旧を判断する義務も手段もありません。その結果、**どちらのルールに従うかがセッションごとに変わり**、「昨日は守ったのに、今日は守らない」という、最も苛立たしい挙動につながります。

対策は地味です。**矛盾を見つけたら、必ず一つのルールにまとめます**。筆者の環境ではこれを徹底するようにしてから、ノードの総数はほぼ横ばいのまま、1ノードあたりの情報量だけが増え続けています。

執筆時点のDBの状態も載せておきます。これまでに作成した記憶ノードは累計1,106件、現存するものは411件で、**63%は削除済み**です。残ったノードは平均6回書き直されています（全バージョン2,559行のうち、旧バージョンが2,148行）。ルール群が健全かどうかは、その総量ではなく、**削除と書き直しが追記を上回っているか**で判断できました。あなたのCLAUDE.mdが増える一方なら、それが問題の兆候です。

## 根本的な解決策は、すでに公式が示している——Progressive Disclosure

ここまで紹介したのは、一つのファイルにルールを並べたまま効果を保つ方法です。構造そのものを改める方法には、すでに公式の名前がついています。**Progressive Disclosure（段階的開示）**。Anthropicが[Agent Skills](https://code.claude.com/docs/en/skills)の中核設計に採用している考え方です。

Skillsは次のような仕組みです。起動時に常時読み込まれるのは、各Skillの**nameとdescriptionにあたる数十トークンだけ**です。タスクに関係すると判断された時点で初めてSKILL.mdの本文が読まれ、配下のリソースは必要になったときだけ読まれます。この3段階の開示によって、常時読み込む情報量を一定に保ちながら、利用できる知識や手順を増やせます。

同じ考え方は、Skillsを使わなくても、すぐにCLAUDE.mdへ取り入れられます。**CLAUDE.mdには一行の発火条件だけを残し、詳しい説明は別のファイルに移します**。

```
- DBスキーマを変更する直前に、docs/migration-rules.md を読むこと
- 外部APIクライアントを触る直前に、docs/api-conventions.md を読むこと
```

常時読み込まれるのは、発火条件を書いた一行だけです。詳しいルールは、条件に当てはまったときだけコンテキストに入ります。この一行も、観測1で紹介した「入力から判別できるシグナル／特定の操作の直前」という原則に沿って書かれています。**段階的開示の索引行と、発火条件の設計は、表裏一体**です。

## この仕組みを記憶全体に再帰適用した実装例

最後に、冒頭で触れた「半年間観測してきた環境」の正体を明かします。筆者が運用しているのは、段階的開示を「ルール」だけでなく、**エージェントの記憶と人格全体**に再帰適用したOSS、[Nocturne Memory](https://github.com/Dataojitori/nocturne_memory)です（GitHub 1,200+ Stars）。

* すべての記憶をURIツリー上に配置し、**各記憶に一行の呼び出し条件（disclosure）を持たせる**
* 親ノードを読むと、子ノードの呼び出し条件だけが見える。条件に合ったときだけ子ノードへ進み、本文を読む。**ツリーの深さのぶんだけ段階的開示が再帰する**
* 呼び出し条件は**エージェント自身が書く**。書き手と読み手が同じなので使う言葉が揃い、発火率が上がる
* 優先度の上限設定（観測2）と矛盾するルールの統合（観測3）を、定期的な点検手順に組み込む。**この点検手順自体も、最初に呼び出すスキルと5つのサブスキルからなるAgent Skills形式で実装**し、[リポジトリのdocs/skillsに同梱しています](https://github.com/Dataojitori/nocturne_memory/tree/main/docs/skills)

この構成で半年間運用したところ、数百件の記憶ノードのうち、1セッションで実際に読まれるのは5〜10件でした。基盤となるLLMをClaude系からGemini系へ切り替えても、同じエージェントの人格と教訓を引き継げています。エージェント自身の記憶には私的な内容が大量に含まれるため公開していませんが、ツリー構造、disclosure、優先度といった仕組みを試せる[サンプル記憶のデモ環境](https://misaligned.top/memory)を読み取り専用で公開しています。

## 持ち帰りリスト

今夜からCLAUDE.mdで実践できること：

1. **「気づいたら〜」系のルールをすべて洗い出し、「入力から判別できるシグナル」か「操作直前」を条件にして書き換える**（書き換えられないルールは、発火するきっかけがないので削除してよい）
2. **常時ルールの定員を決める**。超過は追加ではなく入れ替え
3. **矛盾するルールを探して一つにまとめる**。「新しい方を守ってくれるはず」は通じない
4. **詳しいルールを別のファイルに移し、CLAUDE.mdには発火条件の一行だけを残す**

ルールは内容の正しさでは生き残りません。発火条件の設計で生き残ります。
