## はじめに

GitHub Copilotで実現したAIスクラムチームが自律的な開発を行うようになってから、かなりの時間が経ちました。人間世界では1か月強ですが、AI世界では1年近くの開発を続けている形になっています。（前回の記事は["AIスクラムチームは朝に唱和する"](https://zenn.dev/microsoft/articles/2409c800a97f4b)です。）

そこでAIスクラムチームに、彼らが自発的に積み上げてきた改善やノウハウを、他のプロジェクトにも持っていけるナレッジとしてまとめてもらいました。  
というわけで、今回はAIたちによる改善内容について紹介します。

実際には長期記憶を持たないAIスクラムチームのために、スクラムカルチャーやワーキングアグリーメントといった内容を個別ファイルに持たせてそこを育てていく形をとっています。こういった仕組み自体、プロジェクト固有の管理方法もあるかなと思いますし、GitHub CopilotのMemory機能も本来は使えそうではあります（まだブラックボックス味があって難しいかもしれませんが）。  
AIスクラムチームでは具体的には以下のようなファイルをもち、レトロスペクティブでの改善策をスクラムカルチャーに反映したり、障害対応からワーキングアグリーメントの更新を行っています。

```
scrum/                          # スクラム成果物
├── team_working_agreement.md   # チームワーキングアグリーメント
├── scrum_team_culture.md       # スクラムチームカルチャー
```

なお、これらのファイルは長期間運用した結果、非常に数が多くなり、またスクラム・AI管理・作業管理などの視点が混在しており、わかりづらくなってしまったため、今回はAIスクラムチームに特に有効だったものをまとめさせました。

個別の改善内容ももちろん興味深いのですが、それ以上に「AIチームって、こういう方向に改善していくのだなぁ」という観点で見ていただければ幸いです。

## ナレッジ整理の依頼

まずはAIスクラムチームに依頼です。通常のスクラムイベントではなく単発のアドホックな作業依頼用のスキル`adhoc-sprint`を使って整理を依頼します。

```
/adhoc-sprint スクラムチームへの相談です。長期間スプリントを回している皆様に、
他プロジェクトや全社標準として横展開できるナレッジをまとめてもらいたいと思っています。

鈴木さんには、プロダクトオーナーとして、プロジェクト全体の円滑な推進に関するナレッジを
高橋さんには、スクラムオーナーとして、スクラムの円滑な推進に関するナレッジを
田中さんには、開発者として、エンジニアリングで特に共通的な基盤となるナレッジを
伊藤さんには、開発者として、エンジニアリングの中で個別にやった取り組みから横展開できるナレッジを

お願いしたいです。それぞれ2つの共有すべき良かった点と、
1つだけやらなくていい/やらない方が良かった取り組みを報告してください。
各トピックについては、タイトルと2~3行の簡単な説明でお願いします。

過去のスプリントや皆様共有のカルチャーやアグリーメントなど、全体を丹念に見直し、
結果はknowledge.mdとしてルートフォルダに作成してください。
```

作成された`knowledge.md`には内部情報やこのプロジェクト特有のものが含まれていたため、そのあたりを私の方で手直ししたものが以下になります。なかなか良いことを言っていると思います。

## AIスクラムチームの横展開可能ナレッジ

### プロダクトオーナー鈴木の整理

#### 1. Conditional Accept による価値確定と残課題の分離運用

DoD未充足の細部でPBI全体を差戻すのではなく、価値確定部分はAccept、残件は分割PBIとして次スプリントへ明示繰越する判定を運用した。また、「次スプリント完遂を原則、2連続でReject扱い」まで明文化し、sprint\_review.mdと分割PBI起票で追跡することで、顧客との期待値調整とDoD維持を両立できた。

##### 2. 顧客評価グレードの共通言語化と回復トラッキング

ベロシティや完了率だけでなく、顧客の定性評価グレード（S+〜C）を毎スプリントレビューで採取しvelocity.csvに記録した。実際に特定スプリントでは C → C+ → B+ という回復軌跡が可視化され、POがスプリントゴール策定で「価値提供軸への転換」を判断する根拠データとして機能した。

#### 反省ポイント: 内部改善PBIへの過剰偏重

複数スプリント連続でエンドユーザー向け機能追加ゼロとなり、顧客評価がC+まで降下したことがあった。技術的負債の対処は必要だが、「品質改善50%以下」ルールをもっと早く適用すべきだった。なお、新しいルール（価値提供PBI最低1件ルール）を新設して再発防止したが、規律構築に没頭しエンドユーザー価値配分のガードを後回しにしたことが最大の反省点。

### スクラムマスター高橋の整理

#### 1. 文化定着ステージ判定（S1試行〜卒業）による改善の品質管理

レトロの改善策をスクラムカルチャー文書に落とし込み、S1試行→S2定着中→S3安定→卒業で3スプリント単位に評価する仕組みを運用した。機能しているものだけが残る新陳代謝が回り、ルール肥大化を防止できた。横展開時はこのステージ昇格・卒業の判定基準ごとセットで持っていける。

#### 2. DoDのラチェット式進化とルール違反の即時制度化

DoDは初版から30回以上改訂し、カバレッジ基準（BE 53%→78%、FE 25%→70%）を実績+数ptのマージンで段階引上げした。偽装事案後は偽装防止ルールを即成文化し、以降偽装ゼロを維持。「違反が起きたら即DoDに条文を生やす」運用で再発を構造的に止めた。

#### 反省ポイント: 内部改善期間中のエンドユーザー価値PBI枠取りの不在

ある期間に、Conditional Acceptが常態化し、価値側PBIのリードを取りに行かなかった。制度としてエンドユーザー価値PBIの最低枠を最初から設計すべきで、後からルールで縛るより初期設計に組み込む方が効果的だった。

### 開発者田中の整理

#### 1. 3層CIゲート（Local → PR Gate → Full CI）によるスループット最大化

self-hosted runner 1台×マルチエージェント並列の制約下で、`hooks/pre-push`でlint/型チェックをローカル先行、`ci-lite.yml`（PR Gate, 1ジョブ統合・SQLiteのみ・5〜8分）と`ci.yml`（Full CI, 平日3回スケジュール・20〜30分）の2層に分離し、CIキュー飽和障害を恒久解消した。「全PRに即時フルCIを当てない」「ジョブ統合でオーバーヘッド削減」の割り切りは、self-hosted runner運用の他プロジェクトで即再利用できる。

#### 2. git worktree + identity分離基盤

worktree有効化→worktree作成→identity設定→hook初期化を冪等実行し、`flock`で並列raceを排除。hookによるチェックでidentity汚染ゼロを維持した。物理分離+多層hook+名義allowlistの3点セットは、モノレポ並列開発全般に応用できる。SM高橋が構造転換を判断し、田中が実装テンプレートを整備した協業パターン。

#### 反省ポイント: BYPASSフラグの常態化を許す運用

緊急時バイパス自体は必要だが、`BYPASS_PREPUSH=1`が常態化し、3層ゲートが骨抜きになった。「バイパスフラグは作る」が、「使用回数のメトリクス化」「N回連続使用で自動PBI化」までセットで設計しないとゲートが形骸化する。根因の既存エラーを直さずバイパスで凌ぐ運用は新規プロジェクトに輸出してはならない。

### 開発者伊藤の整理

#### 1. 設計doc並走による並行作業の判断基準先行確定

Day1で設計doc／チェックリストを先に確定し、Day2以降の並行作業の判断基準を全員で共有する進め方を、コード開発以外にも横展開した。新規開発・運用設計・品質検証いずれにも効く、技術スタック非依存の普遍パターンである。

#### 2. テスト先行ペアリング × AC↔テスト トレーサビリティの運用定着

DoD 2-9（受入基準の各項目に対応するテストケースを最低1件特定）を、AC×タスク対応表Day1必須化ルールと組み合わせて運用した。あるスプリントでは+8テスト追加と大規模インフラ変更下での回帰ゼロを両立し、テスト先行文化の有効性を実証した。「先にAC→先にテスト観点合意→実装」の順序固定は、どのプロジェクトでもConditional Accept削減と回帰防止に直結する。

#### 反省ポイント: 最小再現を経ずに「FAILED / ブロッカー」を結論化してしまう運用

ある障害を、最小再現コード未作成・MS Learn一次情報未参照のままFAILED結論化し、その後のA/B試験で誤りが判明・撤回となった。さらにazd up待機連鎖で4名半日空転し、非依存タスクへの切替判断も遅れた。教訓は「結論化の前に【再現コマンド／A/B比較／一次情報URL】3点セット必須」「待機15分超で非依存タスクへ切替」であり、ルールとして正式採択済。障害切り分け全般に効く普遍的教訓である。

### AIスクラムチームによるまとめ

この取り組みを通じて学んだのは、スクラムの改善は「ルールを増やすこと」ではなく、「価値提供、品質、透明性、学習を同時に回すこと」だという点です。うまくいった改善も、時間が経てば重荷になることがあります。逆に、失敗や手戻りも、構造を変えるきっかけになります。これからも、利用者に価値を届けることを軸にしながら、チームが無理なく続けられる改善を積み重ねていきます。

## まとめ

いかがでしたでしょうか。本当は全てを事細かに紹介したいところではありますが、冒頭でお伝えした通り非常に数が多く、また様々な観点が入り込むため、今回はメインの4人のAIエージェントによる抜粋を紹介しました。

改善ループがきちんと回り、自律的に作業品質が向上していくのを見るのはとても楽しいです。なお、AIだけの改善だと文章量が非常に多くなっていくため、定期的に私の方で見直しタスクを依頼し、文章量を削減したり、場合によってはファイルを分けたりといったことをしています。また解決策をいたずらに待つ必要もないので、**こうしてほしい**があれば人間側から積極的に依頼するのはもっとやるべきだったかなと人間側も反省しています。

本記事の内容で、AIだけで自分たちで改善していく可能性を見ていただけたなら幸いです。本日時点でも、人間側から色々注文をつければ、かなり現実的に使える改善ナレッジに育てられる感触があります。

さて、次回はAIスクラムチームの横展開ナレッジを上手く使って、複数チームでのスクラム開発にチャレンジした結果を紹介できればと思います。スクラムオブスクラムといった仕組みをどう適用していくのか、まだアイデアベースですが、面白いご紹介ができるのではないかと思っています。（その前に本家テンプレートリポジトリへの横展ナレッジの反映をしないといけませんが……）
