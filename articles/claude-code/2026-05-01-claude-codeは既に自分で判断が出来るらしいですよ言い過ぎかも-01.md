## この記事で分かること

- Claude Code Auto Modeとは何か、従来との違い
- Opus 4.7 xhigh努力レベルの特徴と使いどころ
- 3つの許可プロファイル（conservative / balanced / full）の使い分け
- Agent Teamsとの組み合わせで何が変わるか
- 本番環境で使う前に知っておくべきリスク

## 背景 — 「割り込みが多い」問題

2026年4月16日、Anthropicはclaude-opus-4-7をauto modeで利用できるようになったことを正式にアナウンスした。
Max契約者が対象で、同日にxhigh努力レベルも追加されている。

Claude Codeを使い始めたとき、多くの開発者が感じる摩擦はここだ。
「タスクを頼んだのに、判断のたびに確認を求めてくる。」
Auto Modeはその摩擦を設定可能な形で取り除く仕組みだ。

## 解説 — Auto Modeの仕組み

Auto Modeは許可モデルを変える機能だ。通常モードではClaudeが都度確認するような決定を、Claudeが自律的に処理する。

**3つの許可プロファイル**

| プロファイル | 特徴 |
|---|---|
| conservative | 低リスク操作のみ自動、高リスクは停止 |
| balanced | 大半の判断を自動、本番影響のある操作は停止 |
| full | ほぼすべてを自動（要注意） |

**Opus 4.7 xhigh努力レベル**
high（デフォルト）とmax（最大精度）の間に新設された段階だ。/effortコマンドで切り替える。
速度を犠牲にせず、精度を引き上げたいタスクに向く。コードレビュー・アーキテクチャ設計・テスト戦略の立案がその例だ。

使い始めてすぐ気づいたのは、balancedプロファイルでの動作の安定感だ。ファイル操作・テスト実行・軽微なリファクタリングは自動で進み、DBマイグレーションやgit pushのような操作で適切に止まる。

## 実務への落とし込み

**Auto ModeをAgent Teamsと組み合わせる**

Agent Teamsは複数のClaude Codeインスタンスを並列で動かす機能だ（2026年2月にGA）。Auto Modeと組み合わせると、チームリードが全体を統括しながら、各メンバーが割り込みなしに自律実行する構成が作れる。

    例：CLIツール開発タスク
    チームリード：全体設計と統合
    Teammate A：UX観点の実装
    Teammate B：テクニカルアーキテクチャ
    Teammate C：テスト戦略とエッジケース

**設定例（balanced profile）**

    /effort xhigh
    /auto balanced
    > Refactor the authentication module to use JWT. Run tests after each change.

この3行で、テスト通過まで自律的にイテレーションが回る。

## 注意点 / 限界 / 誤解されやすい点

**本番DBアクセス権限は渡さない**
balancedプロファイルであっても、本番データベースへの書き込み権限やmainブランチへのpush権限をClaude Codeに与えるのは慎重に判断すべきだ。Auto Modeは「安全」ではなく「設定可能」なリスク管理だ。

**fullプロファイルは検証環境専用**
fullプロファイルは開発者が明示的に「すべてを任せる」と判断した検証専用環境でのみ使うのが現実的だ。

**Max契約のみ対象**
Auto ModeのOpus 4.7対応は現時点でMax契約者限定だ。Pro契約者への展開は未発表。

**xhigh努力レベルはトークンコストが高い**
xhighはhighより精度が高い分、レスポンスが長くなる傾向がある。コスト試算は事前に行うこと。
