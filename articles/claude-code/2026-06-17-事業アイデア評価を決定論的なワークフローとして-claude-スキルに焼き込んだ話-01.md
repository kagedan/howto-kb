LLM に

> いい事業アイデアを考えて

と聞くと、それっぽい答えは返ってくる。

ただし毎回違う。

創造性としては正しい。

しかし評価には向かない。

比較したいなら、毎回同じ手順を通る必要がある。

そこで評価ロジック自体を、プロンプトではなく **スキル（再利用可能ワークフロー）** として固定化した。

これが `business-idea-explorer` である。

---

## 設計思想

### 創造性はモデルに、規律はスキルに

役割を明確に分離した。

### Claude に任せること

### スキルで固定すること

つまり、

> 発想の自由度は維持しながら、評価の自由度を消す

ことが目的である。

---

## ワークフロー

```
0. 前提インテイク
1. 対象の絞り込み
2. ペイン収集
3. クラスタリングと順位付け
4. 商材アイデア生成
5. 事業モデルと価格設計
6. 最初の100人獲得施策
7. 検証設計（GO/NO-GO）
```

評価軸は固定。

```
支払いの濃さ
×
既存解決策の不在
×
到達可能性
```

件数順に並べない。

このルールは重要なので明示的に禁止事項として実装している。

---

## description を起動条件として使う

Claude スキルでは `description` が単なる説明文ではなく起動条件になる。

そのため description は UX ではなくルーティングとして設計した。

```
name: business-idea-explorer
description: >
  事業アイデアを考えたい
  ペインを探したい
  このアイデア評価して
  ニッチを探したい
  ...
```

日本語と英語の両方を埋め込み、

曖昧な問い合わせからでも発火するようにしている。

---

## コンテキスト節約

大きな情報源は遅延ロードする。

```
skills/business-idea-explorer/
├── SKILL.md
└── references/
    └── playbook.md
```

SKILL.md には評価ルールのみ。

検索クエリ集や価格アンカーは playbook 側へ分離。

必要なフェーズでのみロードする。

---

## 実行時の工夫

* フェーズ対話型
* TaskCreate による進捗管理
* 並列 Web 検索
* Markdown レポート固定出力
* 市場別ローカライズ

重要なのは「最後まで勝手に走らない」こと。

各フェーズでユーザー確認を挟み、方向修正できるようにしている。

## まとめ

LLM の課題は能力不足ではなく、一貫性不足であることが多い。

今回やったのは、

に切り分けたこと。

起動条件、順序、禁止事項、出力スキーマをファイル化すると、

事業アイデア評価は「毎回違う相談」ではなく「再現可能な道具」になる。
