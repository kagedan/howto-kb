AIエージェントに実際の権限——トークンの発行、資金の操作、システムの変更——を委譲する場面が急速に増えています。このとき、ほとんどの現場では**APIキーやパスワードをエージェントに渡して**います。

私はこれを危険だと考えています。この記事では、権限を**秘密情報ではなく on-chain トークン**として渡すという考え方(TCTC / [ERC-7303](https://eips.ethereum.org/EIPS/eip-7303))と、それを今日から使えるようにする公開済みのツール群を紹介します。

筆者は ERC-7303 の著者です。

## パスワードを渡すことの何が問題か

エージェントは「完全に信頼できるソフトウェア」ではありません。速くて有能で、しかし**間違える**存在です。プロンプトインジェクションを受けるかもしれないし、指示を誤解するかもしれない。その相手に APIキーを渡すと、次の性質がついてきます。

|  | 資格情報(APIキー・パスワード) | on-chain トークン |
| --- | --- | --- |
| 複製 | **コピーされたら終わり**。漏洩したキーは持ち主を区別しない | ソウルバウンド(譲渡不能)にでき、権限は置いた場所から動かない |
| 失効 | キーの**ローテーション作業**。使用箇所を全部見つけて祈る | 発行者が**バーン1回**。次の呼び出しの有効性をチェーン上で確認する |
| 監査 | 「いま誰が何をできるか」は設定ファイルの考古学 | 保有状況は**公開**。ブロックエクスプローラで誰でも確認できる |
| スコープ | 1本のキーがタスクに必要な範囲より広く開けがち | 証明書1種=権限1つ。ロール内 OR・ロール間 AND で合成 |
| 失念 | 忘れられた権限は**永遠に生きる** | 期限付きトークンは**期限が来たら自動で権限が消える**(トランザクション不要) |

重要なのは、これがエージェント固有の問題ではなく、**エージェントによって顕在化した権限管理一般の問題**だということです。だから解決策も、エージェント専用の仕組みではなく汎用の権限基盤であるべきだと考えています。

## TCTC:権限をトークンにする

TCTC(Token-Controlled Token Circulation)の考え方は3つの動詞で言い切れます。

* **Grant = mint** — 発行者が証明書トークン(ERC-721/1155)を対象アドレスにミントする
* **Check = balanceOf > 0** — 対象コントラクトが呼び出し時に残高を確認する
* **Revoke = burn** — 発行者が証明書をバーンする。**保有者の協力は不要**

対象コントラクト側は、関数にモディファイアを1つ付けるだけです。

```
// ERC-7303: MINTER_ROLE の証明書を持つ者だけが mint できる
function safeMint(address to) public onlyHasToken(MINTER_ROLE, msg.sender) {
    _safeMint(to, _nextTokenId++);
}
```

権限サーバはありません。ポリシーエンジンもありません。**チェーンそのものが権限の真実の源**で、ウォレット・エクスプローラ・インデクサといった既存のトークンインフラがすべてそのまま監査ツールになります。

これを標準として私が提案するものが [ERC-7303](https://eips.ethereum.org/EIPS/eip-7303) です。2026年の更新で、コントラクト自身がロール構造を機械可読に説明するイントロスペクション(ERC-165 で検出可能、interface ID `0x4ee69337`)が入りました。「どのロールがどの証明書で満たされるか」を、**コントラクトに直接聞ける**ようになっています。

## エージェントに対しては「キルスイッチ」になる

エージェント委任の文脈で効くのは、失効の即時性と一方向性です。

```
# 人間がエージェントに1時間だけ権限を付与(mint)
grant_role(agent, MINTER_ROLE, expires_in: 1h) ✅

# エージェントは自分の権限をon-chainで自己確認
check_role(agent, MINTER_ROLE) → true (balanceOf = 1)

# 様子がおかしい——キルスイッチ(burn)
revoke_role(agent, MINTER_ROLE) 🔥

# エージェントが何を信じていようと、次の呼び出しはコントラクトで失敗する
check_role(agent, MINTER_ROLE) → false
```

ポイントは最後の行です。失効はエージェントへの「お願い」ではありません。エージェントのプロセスが生きていようと、指示を誤解していようと、**コントラクト側の残高チェックが落ちる**ので実行できません。

さらに期限付き証明書(時間認識型の `balanceOf` を持つ ERC-1155)を使うと、期限が過ぎた瞬間に `balanceOf = 0` になります。**失効のトランザクションすら不要**——「revoke を忘れた」が事故にならない、フェイルセーフ付きの委任です。

これらは提案ではなく、すべて Sepolia 上で動いています。60秒のデモ動画で、人間が権限を付与→エージェントが NFT をミント→人間がバーン→エージェントが即座に能力を失う、という一連の流れを見られます。

<https://www.youtube.com/watch?v=o547bwYT32A>

## 玉ねぎモデル:トークンがトークンを制御する

TCTC の構造は再帰的です。あるトークンの流通を別のトークンが制御し、その制御トークンの流通もまたトークンで制御できる——層構造なので「玉ねぎモデル」と呼んでいます。

* **最内層:発行者** — 証明書の発行が `onlyOwner` で決まる層。再帰が機械的に底を打つ場所
* **制御トークン(証明書)** — MinterCert、MemberPass、期限付きの委託先バッジ…
* **対象コントラクトのロール** — どの証明書がどのロールを満たすかの宣言(イントロスペクション可能)
* **最外層:保有者** — 人間とAIエージェント

ここで大事なのがトラストモデルです。**信頼の根は発行者ではなく、コントラクトです。**検証する側から見れば、証明書コントラクトの流通ルール——ソウルバウンドか、発行者バーン可能か、期限があるか——が信用できれば、**その下で誰が発行していても構いません**。対象コントラクトは信用する証明書を**コントラクトアドレスで**名指しし、ウォレットは信用するコントラクトを記録する。発行者は「コレクションの運用者」であって、信用を求められる対象ではないのです。

この構造は、実は1999年に USENIX Security Symposium で発表した論文 [Digital-Ticket-Controlled Digital Ticket Circulation](https://www.usenix.org/conference/8th-usenix-security-symposium/digital-ticket-controlled-digital-ticket-circulation) が原型です。チケットの流通をチケットが制御する——27年前の設計が、ブロックチェーンという共有基盤と「汎用性を本当に必要とするユーザ(自律エージェント)」を得て、ようやく実運用になりました。

## 公開したツール

### 1. tctc-mcp — エージェント側(MCP サーバ)

```
npx tctc-mcp --config config.json
```

MCP 対応のエージェント(Claude Code など)に、`check_role` / `list_roles` / `discover_roles`、管理者モードでは `grant_role` / `revoke_role`(期限付き対応)を提供します。イントロスペクション対応コントラクトなら**設定ゼロでロール構造を自動発見**します。npm と [MCP Registry](https://github.com/kofujimura/tctc-mcp) で公開済みです。

### 2. TCTC Dashboard — 人間側(管理画面)

<https://tctc-mcp.vercel.app/>

エージェントを管理する人間のための Web アプリです。バックエンドなし・APIキー不要の静的 SPA で、ブラウザウォレットだけで動きます。

* **任意の ERC-7303 コントラクトを inspect** — イントロスペクションでロール構造を自動発見(何も設定しない)
* **判定と証跡のライブ表示** — `hasRole` の判定と、その根拠になっている証明書ごとの `balanceOf` を12秒ごとに更新。期限付きならカウントダウン表示
* **発行者なら Grant / Revoke がその場で** — 接続ウォレットが証明書の `owner()` なら、mint / burn ボタンがインラインに出ます
* **証明書コレクションの新規発行** — 標準型/期限付き型をブラウザから直接デプロイ。対象コントラクトがまだなくても権限運用を始められます

AIエージェント専用ではありません。ERC-7303 を実装したコントラクトなら何にでも使える、**汎用の on-chain 権限管理アプリ**です。

試すには、Sepolia でこの URL を開いてみてください(3ロール構成のサンプルコントラクトが開きます):

```
https://tctc-mcp.vercel.app/?target=0x873f0bf314A1e0B566015CEf9dA37783A729Fd02
```

## まとめ

* エージェントへの権限委譲は「**見えて、絞れて、破壊できる**」形であるべきで、秘密情報の受け渡しはそのどれも満たさない
* TCTC / ERC-7303 は権限を on-chain トークンにする:**grant = mint、check = balanceOf、revoke = burn**
* キルスイッチ(発行者による一方的バーン)と、ガスレス自動失効(期限付き証明書)が実装済み・公開済み
* エージェント用(tctc-mcp)と人間用(ダッシュボード)は、**同じチェーンという真実の源を読む2つのクライアント**
