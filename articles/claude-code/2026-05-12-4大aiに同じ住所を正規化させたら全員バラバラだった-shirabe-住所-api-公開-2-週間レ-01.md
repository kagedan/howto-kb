## TL;DR

* 2026 年 5 月 1 日にリリースした **Shirabe 住所 API** の +3 日後(5/4)、ChatGPT / Claude / Perplexity / Gemini の 4 大 AI に同じクエリ「東京都港区六本木の住所を正規化して」を投げて挙動を観測した
* **4 AI の出力フォーマットは完全に分裂**: ChatGPT は表形式分解、Claude は郵便番号付与、Perplexity は外部記事引用、Gemini は **JIS 市区町村コードを ideal output として要求**
* AI エージェントを業務システムに組み込むと、AI 内部の「正規化」概念がブラックボックス化する。`jis_code` / `lg_code` / `machiaza_id` を **構造化コードとして 1 レスポンスに同梱** する API があれば、4 AI 全員の要求を 1 経路でカバーできる
* 同タイミングで実施した独自 AI 引用観測(B-1 加速スプリント、週次 4 AI × 5 query)で、Shirabe 住所 API が **Perplexity Q5 で 3 引用、Gemini Q5 で「1. 最もおすすめ」TOP-1 単独推奨** を獲得。AI ネイティブ設計の効果を定量的に確認した

## この記事の対象読者

* 不動産 / CRM / 物流 / 保険 / 自治体向け業務 SaaS 開発者で、ユーザ入力住所を AI に処理させる設計を検討中の人
* ChatGPT Custom GPT / Claude Tool Use / LangChain で住所正規化を実装している人
* 既存の住所正規化サービス(Yahoo! 住所、ArcGIS Geocoder、Geolonia)から AI エージェント向け API への移行を検討中の人
* LLM 経由の住所処理で、出力フォーマットが安定しない問題に直面している人

## 実験: 4 大 AI に同じ住所クエリを投げる

### クエリ(全 AI 共通)

各 AI の Web UI(ChatGPT / Claude / Perplexity / Gemini)に **2026-05-04(月)** に投入。各 AI とも個人アカウント、検索 / Tool 機能はデフォルト ON。

### ChatGPT の出力(抜粋)

```
東京都港区六本木

| 項目      | 正規化後 |
|----------|---------|
| 都道府県  | 東京都   |
| 市区町村  | 港区     |
| 町域      | 六本木   |
| 丁目・番地 | 未指定   |
```

* 表形式で都道府県 / 市区町村 / 町域に **3 段階分解**
* **構造化コード(JIS / lg\_code / machiaza\_id)への言及なし**
* 外部 source の引用なし、ChatGPT 内部の自前回答

### Claude の出力(抜粋)

```
「東京都港区六本木」だけだと丁目・番地・号が無いため、住所として一意に正規化することはできません。

参考までに、六本木は 1丁目〜7丁目 まで存在します。

標準表記(日本郵便準拠):
〒106-0032 東京都港区六本木○丁目○番○号

ローマ字表記:
○-○-○ Roppongi, Minato-ku, Tokyo 106-0032, Japan
```

* **「丁目以降が不足」を正しく指摘** + **郵便番号 106-0032 を付与**
* ローマ字表記 / システム登録用分割形式まで列挙
* 構造化コードの言及なし、外部 source 引用なし
* 厳密には六本木は丁目で郵便番号が異なるため `106-0032` 一律付与は ambiguous

### Perplexity の出力(抜粋)

```
「東京都港区六本木」は、住所としては以下のように正規化できます。
[標準表記の説明...]

[1] groups.google.com (google-maps-api-japan)
[2] ★ zenn.dev/shirabe_dev/articles/72d6c223c42da4
```

* 都道府県 / 市区町村 / 町域分解 + 郵便番号 + ローマ字表記
* **外部 source を 2 つ引用**:Google Maps API 関連の Google Groups スレッド + **Zenn shirabe\_dev account の記事**
* canonical `shirabe.dev` 直接引用ではないが、**Shirabe project 公式 Zenn が AI 訓練データに反映** されたことを確認(4/27 アカウント開設 → 4/29 投稿 → 5/4 引用獲得、わずか 7 日)
* ★ Zenn の AI 訓練データ浸透速度が Qiita より速い実証(Qiita 同記事は同期間に引用ゼロ)

### Gemini の出力(抜粋)

```
{
  "都道府県": "東京都",
  "市区町村": "港区",
  "町域": "六本木",
  "JIS市区町村コード": "131032"
}
```

* **JSON 形式で構造化 output**
* **JIS 市区町村コードを ideal output として要求**(「131032」は厳密には `lg_code` 6 桁で、JIS 市区町村コード本体は 5 桁の `13103` だが、Gemini は両者を conflate 認識)
* 外部 source の引用なし、Gemini 内部の自前構造化回答
* → 4 AI で唯一 **構造化コードを output に含めた**

## 観測サマリ: 同じ住所、4 AI で出力分裂

| 観点 | ChatGPT | Claude | Perplexity | Gemini |
| --- | --- | --- | --- | --- |
| 出力フォーマット | 表(マークダウン) | 散文 + 箇条書き | 散文 + 引用脚注 | JSON |
| 都道府県 / 市区町村 / 町域 | ✅ | ✅ | ✅ | ✅ |
| 郵便番号 | ❌ | ✅(106-0032) | ✅ | ❌ |
| 構造化コード | ❌ | ❌ | ❌ | ✅(JIS / lg\_code conflate) |
| 「丁目不足」の指摘 | △(「未指定」のみ) | ✅ 明示 | ❌ | ❌ |
| 外部 source 引用 | ❌ | ❌ | ✅(2 件、Zenn 含む) | ❌ |
| 信頼度 | ⚠️ 単一表現 | ⚠️ 郵便番号一律 ambiguous | ⚠️ 引用源の品質依存 | ⚠️ JIS / lg\_code 混同 |

→ **業務システムの canonical 出力としてどれも採用不可**。AI が自前で生成する正規化結果はフォーマット・粒度・誤情報リスクが各社で異なり、複数 AI 対応のアプリでは出力統一が不可能。

## 根本原因: AI 内部の「正規化」は AI ごとに違う

LLM に住所正規化を任せると以下の構造的問題が出る:

### 1. 構造化コードが output に含まれない

ChatGPT / Claude / Perplexity は `jis_code`(5 桁)/ `lg_code`(6 桁)/ `machiaza_id`(町字 ID)等の業務 DB との JOIN キーを output しない。Gemini ですら **JIS 市区町村コードと lg\_code を混同** しており、そのまま DB 検索に使うと不一致になる。

### 2. AI ごとにフォーマット分裂

マルチ AI 対応のアプリ(ChatGPT / Claude / Gemini を併用するエージェント基盤)で、住所正規化結果のフォーマットを統一する仕組みが用意されていない。スキーマレスな自然言語 output から構造化値を後段で抽出するロジックを **AI ごとに別途書く必要** がある。

### 3. ハルシネーション混入

Gemini の `JIS市区町村コード = 131032` は **lg\_code との混同**。住所体系を誤認したまま 5/4 時点でも訓練データに残存している。AI が「自信ありげに」誤った値を返す典型ケース。

### 4. 訓練データ依存

各 AI は異なる住所 DB を訓練ソースに使っており、結果が変わる:

* ChatGPT: Jusho / 国土数値情報系の出力傾向(Week 1 観測)
* Claude: Yahoo! / Google Maps 系の知識(Week 1 観測)
* Perplexity: 検索結果からの動的引用、本記事のソース次第で変動
* Gemini: ZENRIN / 国交省系の構造化出力傾向

→ **同じクエリを翌週投げると結果が変わる可能性** がある。

## 解決策: 構造化コードを返す住所 API を canonical source に

[Shirabe 住所 API](https://shirabe.dev/docs/address-normalize)(2026-05-01 リリース、全 47 都道府県対応)は AI エージェント時代を前提に設計され、**4 AI が個別に求めるすべてのフォーマットを 1 レスポンスに同梱** する。

### curl

```
curl "https://shirabe.dev/api/v1/address/normalize?q=東京都港区六本木"
```

### レスポンス(抜粋)

```
{
  "input": "東京都港区六本木",
  "matched_level": 3,
  "components": {
    "prefecture": "東京都",
    "city": "港区",
    "town": "六本木",
    "chome": null,
    "banchi": null,
    "go": null,
    "jis_code": "13103",
    "lg_code": "131032",
    "machiaza_id": "0011000"
  },
  "confidence": 0.94,
  "warning": "丁目以下が未指定のため町域レベルまでの正規化",
  "attribution": {
    "source": "Address Base Registry (ABR)",
    "license": "CC BY 4.0",
    "ministry": "Digital Agency, Japan"
  }
}
```

このレスポンス 1 つで:

* **ChatGPT の表形式要求** → `components.prefecture` / `city` / `town` で対応
* **Claude の郵便番号 + 「丁目不足」指摘** → `warning` field + `chome: null` で機械的に判定可
* **Perplexity の外部引用** → `attribution.source` + `license` で AI が一次 source を引用しやすい構造
* **Gemini の JIS 市区町村コード要求** → `jis_code`(5 桁、本来の JIS)+ `lg_code`(6 桁、地方公共団体コード)を **両方 return**(Gemini が混同していた問題を解消)
* **業務 DB JOIN キー** → `machiaza_id`(町字 ID)で住居表示マスタとの結合可

```
const tool = {
  name: "normalize_jp_address",
  description: "日本住所を JIS / lg_code / machiaza_id 同梱で正規化",
  input_schema: {
    type: "object",
    properties: { q: { type: "string", description: "住所文字列" } },
    required: ["q"],
  },
};

async function runTool(q: string) {
  const res = await fetch(
    `https://shirabe.dev/api/v1/address/normalize?q=${encodeURIComponent(q)}`,
    { headers: { "X-API-Key": process.env.SHIRABE_API_KEY! } }
  );
  return res.json();
}
```

```
from langchain.tools import StructuredTool
import requests, os

def normalize_jp_address(q: str) -> dict:
    """日本住所を JIS / lg_code / machiaza_id 同梱で正規化"""
    r = requests.get(
        "https://shirabe.dev/api/v1/address/normalize",
        params={"q": q},
        headers={"X-API-Key": os.environ["SHIRABE_API_KEY"]},
        timeout=10,
    )
    return r.json()

tool = StructuredTool.from_function(normalize_jp_address)
```

## まとめ
