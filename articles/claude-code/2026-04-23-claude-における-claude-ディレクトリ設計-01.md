## 概要

Claude Code における .claude ディレクトリと、その周辺の構成要素の役割および設計方針について整理してみた。

Claude Code では、プロジェクト記憶、条件付きルール、再利用可能な手順、subagent、hooks、settings などを組み合わせて挙動を構成する。.claude は、それらをプロジェクト単位で管理するための領域として扱うと理解しやすい。

この観点では、Claude 単体ではなく、モデルとその外側の実行環境を合わせて設計対象とする。

LangChain においても Agent は Model と Harness の組み合わせとして整理されている[(the-anatomy-of-an-agent-harness)](https://blog.langchain.com/the-anatomy-of-an-agent-harness/)

ここでは、.claude をその Harness の一部として位置付けて整理する。

## ディレクトリ構成

典型的なディレクトリ構成は以下の通りである。

```
repo/
├─ CLAUDE.md
├─ .claude/
│ ├─ agents/
│ ├─ hooks/
│ ├─ rules/
│ ├─ skills/
│ ├─ CLAUDE.local.md
│ ├─ settings.json
│ └─ settings.local.json
```

CLAUDE.md はプロジェクト記憶として、リポジトリ直下または .claude/ 配下のいずれかに配置できる。

CLAUDE.local.md および settings.local.json は、個人用のローカル上書きとして扱うと整理しやすい。

これらは通常 gitignore 対象とし、開発者ごとの環境差分や一時的な調整を記述する用途で利用する。共有前提の設定とは分けて管理することを推奨する。

本構成は単なる設定の分割というより、Claude の振る舞いをレイヤーごとに分けて管理するための構造として設計する。

## .claude の役割

Claude は推論や生成を行うモデルであるが、実務においてはそれだけでは不十分であり、周辺の構造と組み合わせて利用する必要がある。

具体的には、以下のような要素が求められる。

* どのルールに従うか
* どのツールを使用するか
* どの手順で作業を進めるか
* どの範囲まで操作を許可するか

これらはモデル単体でもある程度は扱えるが、安定した挙動を実現するためには、外部構造として明示的に設計・管理することが重要となる。

.claude はこれらの要素をプロジェクト単位で定義し、Claude の挙動に反映させるための領域として機能する。

この観点では、.claude は単なる設定ファイルの置き場ではなく、Claude を実務で運用可能な形にするための外部構造、すなわち Harness の一部として捉えることができる。

## 各構成要素

### CLAUDE.md

CLAUDE.md はセッション開始時に読み込まれるプロジェクト記憶として扱う [(memory)](https://code.claude.com/docs/ja/memory)

Claude の判断のベースとなる前提がここに含まれるため、頻繁に参照される情報に限定して記述することが重要である。

主に以下を記述する。

* アーキテクチャ方針
* 基本的な設計原則
* 共通ルール
* 利用頻度の高いコマンド

全てのルールや仕様をここに集約するのではなく、常時参照する必要のある前提に限定することを推奨する。

内容が大きくなりすぎる場合は、rules への分割や、@path/to/file による import を利用して構造化する。

また、詳細な仕様やドメイン知識については docs/ などに分離し、CLAUDE.md から参照先を示す形にすることで、コンテキストの肥大化を防ぐことができる。

### アンチパターン

CLAUDE.md はセッション開始時に参照される前提であるため、ここにあらゆる情報を集約しようとすると、かえって扱いにくくなる。

まず避けたいのは、仕様書、手順書、背景説明、設計思想、運用メモなどを 1 ファイルにまとめて書いてしまう構成である。

この場合、常時参照させたい前提と、その場でしか必要ない情報が混在し、Claude にとっても人間にとっても、どこが重要なのかが分かりにくくなる。

また、docs/ に置くべき詳細仕様まで CLAUDE.md に転記してしまう構成も避けたい。

CLAUDE.md は仕様書のコピーではなく、判断のベースとなる前提をまとめる場所である。詳細な仕様をそのまま書き込むと、情報量が増えるだけでなく、仕様の更新が二重管理になりやすい。

さらに、rules や skills に分けるべき内容まで CLAUDE.md に入れてしまうと、レイヤーの役割分担が崩れる。

例えば、言語別の細かな規約や、レビュー手順、リリース手順などは、CLAUDE.md に書くよりも、それぞれ rules や skills に分離した方が構造として分かりやすい。

加えて、「とりあえず困りそうなことを全部書く」という運用も避けたい。

CLAUDE.md は常時参照される前提であるため、内容が増えるほど、重要な前提が埋もれやすくなる。結果として、最も守ってほしいことがかえって伝わりにくくなる。

### ベストプラクティス

CLAUDE.md を安定して運用するためには、ここを「常時参照する最小限の前提を書く場所」として扱うことが重要である。

まず、ここに書く内容は、頻繁に必要になる判断基準に限定する。

例えば、アーキテクチャ方針、設計上の原則、共通ルール、利用頻度の高いコマンドのように、セッションのたびに参照される価値が高い内容を優先する。

一方で、詳細仕様、背景説明、長い手順、特定領域だけに適用されるルールは、別レイヤーに分離する。

仕様やドメイン知識は docs/ に置き、条件付きの規約は rules に分け、再利用可能な手順は skills に分離することで、CLAUDE.md の役割を明確に保ちやすくなる。

また、CLAUDE.md 自体が大きくなってきた場合は、無理に 1 ファイルに維持するのではなく、rules への分割や @path/to/file による import を利用して構造化する方が扱いやすい。

このときも、import 先に詳細を寄せつつ、CLAUDE.md 自体には「何を常時意識すべきか」が分かる状態を残しておくことが重要である。

さらに、CLAUDE.md には「常時読ませたい理由が説明できる内容だけを書く」という基準を持つと整理しやすい。

逆に言えば、「この情報は特定のタスクでしか使わない」「特定のファイルを触るときにだけ必要」「順序だった手順として呼び出したい」といった内容は、CLAUDE.md 以外に置く方が自然である。

要するに、CLAUDE.md は何でも書ける場所ではなく、Claude の判断のベースとなる共通前提を最小限に整理して置くためのレイヤーとして設計するのがよい。

## rules

rules/ は規約を分割して管理するための領域とする。

ただし、rules/ に配置したファイルが自動的に条件付き適用になるわけではない点には注意が必要である。

paths frontmatter を持たないルールは常時ロードされる前提で扱い、CLAUDE.md を補完する共通規約として利用する。

一方で、特定のファイル群に対してのみ適用したいルールは、YAML frontmatter で paths を明示することで条件付き適用とする。

例えば、以下のように記述する。

```
paths:
  - "src/api/**/*.ts"

# API Development Rules
- API の入力検証を必須とする
- 標準エラーレスポンス形式を維持する
```

このように、rules は「置き場所」だけで条件付きになるのではなく、paths の有無によって常時適用か条件付き適用かが決まる。

主に以下のような内容を対象とする。

* 言語別のコーディング規約
* フレームワーク別の実装ルール
* ディレクトリ単位の責務や制約
* テストコードの書き方
* セキュリティ上の注意事項
* API やスキーマ変更時の確認事項

例えば frontend.md、backend.md、terraform.md、testing.md、security.md のように、トピック単位で分割して管理することを推奨する。

また、モノレポや複数コンポーネントを含むリポジトリでは、特定パスにのみ適用したいルールは paths を明示したうえで rules/ に分離することで、無関係な文脈の流入を抑制できる。

本レイヤーでは「その場限りの手順」ではなく、「繰り返し適用される規約」を定義することを基本とする。

なお、Claude Code の memory ドキュメントでも、常時参照する前提は CLAUDE.md に置き、条件付きで適用したい内容は分離して管理する考え方が示されている [(memory)](https://code.claude.com/docs/ja/memory)

## アンチパターン

rules/ を設計する際には、単にファイルを分割すればよいわけではない。特に以下のような構成は、意図した挙動にならず、かえって Claude の判断ノイズを増やしやすい。

まず避けたいのは、すべてのルールを paths なしで配置する構成である。

```
.claude/rules/
├─ go.md
├─ api.md
├─ db.md
├─ terraform.md
└─ security.md
```

この場合、各ルールは条件付きではなく常時ロード対象となるため、CLAUDE.md を分割したつもりでも、実質的には常時参照する前提を増やしているのと近い状態になる。

また、1つのルールファイルに背景説明、設計思想、詳細仕様、実装手順まで含めて長文化する構成も避けたい。

rules は継続的に守らせたい規約を定義する場所であり、仕様書や手順書そのものを置く場所ではない。説明が長くなりすぎる場合は docs/ や skills/ に分離した方が整理しやすい。

さらに、トピック単位で分割しているように見えても、paths を付けずに frontend.md、backend.md、terraform.md のようなファイルを並べるだけでは、条件付き適用にはならない点にも注意が必要である。

加えて、paths を使っていても、適用対象の粒度が曖昧なまま広いパターンを書いてしまうと、意図しない場面でルールが候補に入りやすくなる。

例えば src/**/\*.ts のように広く取りすぎるよりも、src/api/**/*.ts や src/frontend/\*\*/*.tsx のように責務単位で絞る方が扱いやすい。

## ベストプラクティス

rules/ を安定して運用するためには、常時参照するルールと条件付きで適用したいルールを明示的に分けて設計することが重要である。

まず、paths を持たないルールは少数に限定し、CLAUDE.md を補完する共通規約として扱う。

ここには、言語全体で常に守りたい最小限の原則や、リポジトリ全体に共通する制約のみを置く。

一方で、特定のファイル群にのみ適用したいルールは、必ず paths frontmatter を付けて条件付きルールとして定義する。

例えば、以下のように責務単位で分けると整理しやすい。

```
.claude/rules/
├─ common.md
├─ go.md
├─ api.md
├─ frontend.md
└─ terraform.md
```

このうち common.md のような共通ルールは paths なしで最小限に保ち、api.md や frontend.md、terraform.md などは paths を明示して適用対象を限定する。

また、1ファイル1トピックを基本とし、1つのルールファイルに複数の関心事を混在させないことも重要である。

例えば API の互換性制約とセキュリティルールを別ファイルに分けておくと、後から見直す際にも保守しやすい。

さらに、rules には「その場限りの手順」ではなく「繰り返し適用される規約」を書く、という原則を維持する。

レビュー手順やリリース手順のように順序が重要なものは skills に切り出し、仕様や背景説明は docs/ に置き、rules には最終的に守らせたい制約だけを残す方が構造として安定する。

要するに、rules は仕様書のコピーでも手順書の置き場でもなく、Claude に継続的に意識させたい規約を、常時適用と条件付き適用に分けて定義するためのレイヤーとして設計するのがよい。

| 状態 | 影響 |
| --- | --- |
| pathsあり | 必要なときだけ候補 |
| pathsなし | 常に候補 |
| rules多い | ノイズ増加 |
| rules長い | token圧迫しやすい |

## 補足

paths frontmatter による条件付き適用は有用である一方で、実装上の制約や挙動の差異により、意図通りに動作しないケースが報告されている。

例えば、起動時にすべてのルールがロードされるように見える挙動や、特定の操作時に path 条件が適用されないケースなどが議論されている。

2026年4月時点では、paths frontmatter に関する挙動についていくつかの Issue が報告されており、全件ロードされる挙動や、Write ツール実行時に path-scoped rules が適用されないといったケースが指摘されている

<https://github.com/anthropics/claude-code/issues/16299>

* .claude/rules/ の path指定ルールが無視され、常に全ルールがグローバルに読み込まれる
* 本来は対象パス時のみ適用されるべきなのに、セッション開始時から全部コンテキストに入る
* 結果として 不要なルール混入・コンテキスト肥大化・挙動のブレが発生する

<https://github.com/anthropics/claude-code/issues/23478>

* .claude/rules/ の path指定ルールが Write（新規作成時）では読み込まれず、Read時のみ適用される
* その結果、新規ファイル作成時のルール（例：ヘッダ追加）が無視される
* 回避には Write後にRead→Editするなどのワークアラウンドが必要

このため、セキュリティ制約や必ず守らせたい処理については、rules のみに依存せず、hooks や settings.json 側で制御する構成を併用することが望ましい。

## skills

skills/ は再利用可能な作業手順を定義するための領域とする。

ここでは知識そのものではなく、作業の進め方を定義する。

主に以下のような内容を対象とする。

* Pull Request レビュー
* リリース前チェック
* 障害対応フロー
* 定期運用作業
* 調査および切り分け手順

同一の作業を繰り返す場合は、プロンプトで都度指示するのではなく、skill として定義することで、出力の一貫性を維持しやすくなる。

また、作業手順を明示的に分離することで、どのような順序で確認し、どのような観点で判断するかを構造化しやすくなる。

例えば review-pr、release-check、incident-triage のように、反復的に実施するタスク単位で分割して管理することを推奨する。

実装上は、skills は .claude/skills/<skill-name>/SKILL.md をエントリポイントとするディレクトリ単位で管理され、必要に応じて examples、template、scripts などの補助ファイルを同梱できる。

そのため、skills は単なる 1 ファイルのメモではなく、再利用可能なワークフローを最小単位のパッケージとして構成するレイヤーとして捉えると分かりやすい。

本レイヤーでは「常時参照する前提」ではなく、「必要な場面で呼び出す手順」を定義することを基本とする。

加えて、skills は SKILL.md の frontmatter に含まれる description が自動選択の手がかりとして使われるため、どのような場面で利用されるべき skill なのかが分かるように、用途の記述を明確にして設計することが重要である。

なお、Claude Code の skills ドキュメントでも、skills は再利用可能なワークフローを定義する仕組みとして説明されており、Claude は関連する場合に skill を使用するか、/skill-name で直接呼び出すことができる[(skills)](https://code.claude.com/docs/ja/skills)

## アンチパターン

skills/ を設計する際には、単に作業内容を列挙すればよいわけではない。特に以下のような構成は、意図した場面で利用されにくく、かえって運用しづらくなる。

まず避けたいのは、skill を長いメモや背景説明の置き場として使う構成である。

skills は再利用可能な手順を定義する場所であり、仕様書や設計思想をそのまま保存する場所ではない。背景説明やドメイン知識が中心になってしまうと、どのような場面で skill を使うべきかが曖昧になる。

また、description を曖昧に書く構成も避けたい。

例えば「コードを改善する」「いろいろ調査する」のような description では、どの場面で呼び出されるべきかが不明確であり、自動選択の精度が下がりやすい。

さらに、1つの skill に複数の関心事を詰め込みすぎる構成も避けたい。

レビュー、修正、リリース、調査のような異なるタスクを 1 つの skill にまとめると、手順が長くなり、結果として再利用性も下がる。

加えて、常時参照させたい前提や規約まで skills に入れてしまう構成も避けたい。

共通前提は CLAUDE.md、継続的に守らせたい規約は rules に置くべきであり、skills は必要時に呼び出す手順に集中させた方が整理しやすい。

要するに、skills は「何でも入れられる補助ファイル」ではなく、呼び出し条件と手順が明確な再利用可能ワークフローとして設計する必要がある。

## ベストプラクティス

skills/ を安定して運用するためには、1つの skill に対して 1つの目的と 1つの作業フローを対応させることが重要である。

まず、skill 名と description を見ただけで、どのような場面で利用するかが分かる状態にする。

例えば review-pr、release-check、incident-triage のように、目的が明確で、呼び出し条件が想像しやすい名前にすると扱いやすい。

また、SKILL.md では背景説明よりも、Claude に実行させたい手順、確認観点、期待する出力形式を優先して記述する。

これにより、同じ作業を繰り返したときの出力のぶれを抑えやすくなる。

さらに、description は自動選択の手がかりになるため、「何をするときに使う skill なのか」を具体的に書くことが重要である。

例えば「Pull Request をレビューするとき」「障害の初動切り分けを行うとき」のように、利用場面を具体的に書く方が運用しやすい。

加えて、skill は必要に応じて直接呼び出せるように、明示呼び出しでも違和感のない粒度に保つことが望ましい。

そのため、1つの skill に複数の大きな目的を詰め込むのではなく、タスク単位で分割して管理することを基本とする。

要するに、skills は詳細仕様の保管場所ではなく、「どの場面で」「どの順序で」「どの観点で」作業を進めるかを再利用可能な形で定義するためのレイヤーとして設計するのがよい。

## agents

agents/ は subagent を定義するための領域とする。

rules や skills が主会話の Claude に対して前提や手順を追加するためのレイヤーであるのに対し、agents は役割ごとに独立した実行主体を定義するためのレイヤーとして捉えると分かりやすい。

subagent は独自の context window、system prompt、tool access、permissions を持つため、単なる追加情報ではなく、主会話から委譲される補助的な実行単位として機能する。

このため、agents では「どのように作業するか」そのものよりも、「どの役割に何を担当させるか」を定義することを基本とする。

例えば researcher、reviewer、fixer のように、反復的に現れる役割単位で分割して管理することを推奨する。

また、subagent は description に一致するタスクに遭遇したときに利用されるため、どのような場面で委譲すべき役割なのかが分かるように、用途の記述を明確にして設計することが重要である。

なお、Claude Code の subagents ドキュメントでも、subagent は独立した context window と専用設定を持つ専門化された補助的実行単位として説明されている[(sub-agents)](https://code.claude.com/docs/en/sub-agents)

## アンチパターン

agents/ を設計する際には、単に役割名を並べればよいわけではない。特に以下のような構成は、意図した場面で委譲されにくく、かえって運用しづらくなる。

まず避けたいのは、subagent を「追加のメモ」や「長い補助プロンプト」の置き場として使う構成である。

agents は主会話の Claude に追加情報を渡す場所ではなく、独立した実行主体を定義する場所である。そのため、背景説明や一般論だけが長く書かれていても、どの役割として使うべきかが分かりにくい。

また、description を曖昧に書く構成も避けたい。

例えば「いろいろ調査する」「必要なら直す」のような description では、どのようなタスクを委譲すべきかが不明確であり、自動選択の精度が下がりやすい。

さらに、1つの agent に複数の大きな責務を持たせすぎる構成も避けたい。

調査、レビュー、修正、計画立案を 1 つの agent にまとめると、役割が広すぎて境界が曖昧になり、結果として主会話側で分業する意味が薄れる。

加えて、すべての agent に広いツール権限を与える構成も避けたい。

subagent は役割ごとに責務を分離できることが利点であるため、必要最小限の tool access と permissions に絞った方が、安全性と見通しを保ちやすい。

要するに、agents は何でもできる万能 worker の集合ではなく、明確な委譲先として設計する必要がある。

## ベストプラクティス

agents/ を安定して運用するためには、1つの agent に対して 1つの役割と 1つの責務のまとまりを対応させることが重要である。

まず、agent 名と description を見ただけで、どのようなタスクを委譲すべきかが分かる状態にする。

例えば researcher、reviewer、fixer のように、役割が直感的に分かる名前にすると扱いやすい。

また、description には「何をする agent か」だけでなく、「どのような場面で使うべきか」を具体的に書くことが重要である。

例えば「コードベースの探索や関連箇所の調査を行うとき」「差分のレビュー観点を整理するとき」のように、利用場面まで明示した方が運用しやすい。

さらに、agent ごとの tool access や permissions は必要最小限に限定する。

読むだけの役割には読み取り専用ツールだけを与え、修正を担当する役割にのみ書き込み権限を与えるようにすると、責務分離と安全性を両立しやすい。

加えて、agents は主会話の Claude を補助する「別の実行主体」であることを意識し、前提や規約を書く場所として使わないことも重要である。

共通前提は CLAUDE.md、継続的に守らせたい規約は rules、再利用可能な手順は skills に置き、agents には委譲先としての役割定義だけを残す方が構造として安定する。

要するに、agents は追加のプロンプト置き場ではなく、「どの役割に、どの責務を、どの権限で委譲するか」を定義するレイヤーとして設計するのがよい。

## hooks

hooks は Claude の動作に対する自動処理および強制処理を定義する[(hooks-guide)](https://code.claude.com/docs/en/hooks-guide)

特定のイベントをトリガーとして、必ず処理を実行することができる。

主に SessionStart、SessionEnd、UserPromptSubmit、PreToolUse、PostToolUse、Stop、StopFailure などのイベントに対して処理を紐付ける形で利用する。

例えば、Bash コマンドの実行前後に処理を差し込みたい場合は、PreToolUse / PostToolUse に matcher を設定して制御する。

例としては以下がある。

* 危険コマンドの拒否
* フォーマット処理の自動実行
* 実行前チェック
* 実行ログやメトリクスの記録

hooks は Claude の推論とは独立して動作するレイヤーであり、文章による指示ではなく、仕組みとして強制的に制御できる点が特徴である。

このため、守らせたい制約や必ず実行したい処理については、可能な限り hooks 側に寄せることで、挙動の安定性を高めることができる。

## アンチパターン

hooks を設計する際には、単に自動化すればよいわけではない。特に以下のような構成は、意図しない副作用や運用の複雑化を招きやすい。

まず避けたいのは、Claude への文章指示で実現できる内容まで hooks に寄せてしまう構成である。

hooks は強制力の強いレイヤーであるため、すべてをここで制御しようとすると、挙動の柔軟性が失われやすい。

また、副作用の大きい処理を無制限に実行する構成も避けたい。

例えば、すべてのコマンド実行時に重い処理を走らせると、パフォーマンス低下や意図しない遅延につながる。

さらに、複数の hooks が相互に依存したり、順序に依存する構成も避けたい。

hooks はイベント単位で独立して動作する前提で設計した方が、トラブル時の切り分けが容易になる。

加えて、「何が実行されるか分からない状態」を作ることも避けるべきである。

hooks は自動的に実行されるため、どのイベントで何が動くのかを明確にしておかないと、デバッグや運用が困難になる。

要するに、hooks は強力な仕組みであるがゆえに、責務を絞り、影響範囲を意識して設計する必要がある。

## ベストプラクティス

hooks を安定して運用するためには、「強制すべきものだけをここに置く」という原則を持つことが重要である。

まず、セキュリティや安全性に関わる制約は優先的に hooks で制御する。

例えば、危険コマンドのブロックや、特定ディレクトリへの書き込み制限などは、文章指示ではなく hooks によって確実に制御する方が安全である。

また、必ず実行したい前処理・後処理も hooks に適している。

例えば、コード整形、lint 実行、メトリクス収集、ログ記録などは、イベントに紐付けて自動化することで、運用の一貫性を保ちやすくなる。

さらに、各 hook はできるだけ単機能に分割し、1つのイベントに対して1つの責務を持たせると扱いやすい。

これにより、問題が発生した際にも影響範囲を限定しやすくなる。

加えて、どのイベントでどの hook が実行されるかを明示的に整理しておくことで、挙動の可視性を高めることができる。

要するに、hooks は「確実に実行したい処理」「絶対に守らせたい制約」を担うレイヤーとして設計し、他のレイヤー（CLAUDE.md、rules、skills）とは明確に役割を分離するのがよい。

## settings.json

settings.json は実行環境および実行制御に関する設定を管理する[(settings)]<https://code.claude.com/docs/en/settings>)

Claude の推論内容そのものではなく、外側から動作を制御するためのレイヤーとして扱う。

主に以下を扱う。

* permissions（実行可能な操作の制御）
* hooks の接続設定
* 環境変数
* モデル設定

settings.json は Claude の出力を誘導するものではなく、実行可能な範囲や環境そのものを制御する役割を持つ。

このため、制約や禁止事項のうち「確実に守らせたいもの」については、文章による指示ではなく、このレイヤーで制御することを基本とする。

また、hooks と組み合わせることで、「いつ実行するか」と「何を許可するか」を分離して設計することができる。

## アンチパターン

settings.json を設計する際には、単に設定を増やせばよいわけではない。特に以下のような構成は、意図しない制約や運用の複雑化を招きやすい。

まず避けたいのは、Claude への指示で制御すべき内容まで settings.json に寄せてしまう構成である。

settings.json は実行環境を制御するためのものであり、「どう考えるか」や「どう判断するか」を定義する場所ではない。

また、permissions を過度に広く設定する構成も避けたい。

すべての操作を許可してしまうと、安全性が低下するだけでなく、agent や hooks による責務分離の意味も薄れる。

さらに、settings.json に複雑な制御ロジックを持たせようとする構成も避けたい。

実行タイミングや条件分岐は hooks 側で扱うべきであり、settings.json はあくまで静的な制御に留めた方が分かりやすい。

加えて、プロジェクト設定と個人設定を混在させることも避けたい。

settings.local.json に分離せず、個人用の設定を settings.json に含めてしまうと、チーム内での再現性が崩れる原因になる。

要するに、settings.json は万能な制御レイヤーではなく、「環境と許可範囲を定義する場所」として責務を限定する必要がある。

## ベストプラクティス

settings.json を安定して運用するためには、「実行可能な範囲を明示する」という観点で設計することが重要である。

まず、permissions は最小権限の原則に従って設定する。

必要な操作だけを許可し、それ以外は許可しないことで、安全性と意図しない動作の抑制を両立できる。

また、hooks と組み合わせて役割分担を明確にする。

「いつ実行するか」は hooks で定義し、「何を許可するか」は settings.json で定義することで、制御の見通しが良くなる。

さらに、プロジェクト共通の設定と個人ごとの設定は分離する。

共有すべき設定は settings.json に置き、開発者ごとの環境依存設定は settings.local.json に分離することで、チーム全体の再現性を保ちやすくなる。

加えて、settings.json は「Claude の振る舞いを変える」のではなく、「Claude が動ける範囲を定義する」レイヤーであることを意識する。

この区別を明確にすることで、CLAUDE.md や rules、hooks との役割分担が整理され、全体構造が安定する。

要するに、settings.json は実行環境の境界を定義するレイヤーとして設計し、他のレイヤーと責務を混在させないことが重要である。

## 設計方針

.claude の設計においては、以下の分離を意識する。

* 前提は CLAUDE.md
* 条件付き規約は rules
* 手順は skills
* 分業は agents
* 強制は hooks
* 制御は settings

この分離により、Claude の挙動をレイヤーごとに整理し、構造的に管理できるようになる。

ただし、これらは完全に独立したものではなく、役割に応じて重なりながら機能するため、「どのレイヤーに置くと最も自然か」という観点で配置を判断することが重要である。

## 仕様はどこに置くか

アプリケーションの仕様については、すべてを CLAUDE.md に書くのではなく、役割に応じて置き場所を分ける方が扱いやすい。

まず、仕様そのものは通常のドキュメントとして管理する。

例えば、ドメイン仕様、API 仕様、要件、画面仕様などは、docs/ や ADR、既存の設計書に配置する。

一方で、Claude に常時意識させたい前提は CLAUDE.md に置く。

ここに書くのは仕様の全文ではなく、「何を参照すべきか」「どの前提を外してはいけないか」といった、判断基準として頻繁に必要になる内容である。

また、詳細な仕様は CLAUDE.md に直接書くのではなく、docs/ に配置したうえで、@path/to/file の形式で参照させる構成にすると整理しやすい。

この import 構文を利用することで、CLAUDE.md をエントリポイントとして、複数のドキュメントを参照する構造を作ることができる。

Claude Code のドキュメントでも、@path/to/file による import によって、構成をネットワーク状に分割できることが示されており、最大で数階層のネストが可能である（[https://code.claude.com/docs/ja/memory）](https://code.claude.com/docs/ja/memory%EF%BC%89)

この仕組みを前提に、仕様は docs 側に分離し、CLAUDE.md では参照関係のみを管理する構成とする方が、保守性と可読性の両面で扱いやすい。

さらに、仕様の中でも特に逸脱を避けたいものについては、rules として分離する。

例えば、API の互換性制約、ドメイン上の不変条件、識別子の命名規則、セキュリティ上の禁止事項などは、単なる説明ではなく、継続的に守らせたい制約として rules に置く方が整理しやすい。

このため、アプリケーション仕様は次のように分けて考えると分かりやすい。

* 仕様そのものは docs や設計書に置く
* 常時参照させたい前提は CLAUDE.md に置く
* 継続的に守らせたい制約は rules に置く

このように分離することで、CLAUDE.md に仕様を詰め込みすぎずに済み、必要な情報だけを適切な粒度で Claude に渡しやすくなる。

また、仕様を docs に置いたうえで、CLAUDE.md から参照先を明示しておく構成にすると、構造としても分かりやすい。

例えば、以下のような形で整理できる。

```
repo/
├─ .claude/
│ ├─ agents/
│ ├─ hooks/
│ ├─ rules/
│ ├─ skills/
│ ├─ CLAUDE.local.md
│ ├─ settings.json
│ └─ settings.local.json
├─ CLAUDE.md
├─ docs/
│ ├─ domain.md
│ ├─ api.md
│ ├─ requirements.md
```

## .claude 以下のファイルの読み込み

.claude 配下の構成要素は、ファイル順に上から順番に読み込まれるものではない。

一般的な設定ファイルのように「すべてを読み込んでから処理する」モデルではなく、Claude Code はタスクの内容や実行状況に応じて、必要な情報を選択的に参照する形で動作する。

このため、各ファイルは「読み込み順」ではなく「適用条件」によって挙動が決まると考える方が実態に近い。

まず、CLAUDE.md はセッション開始時に読み込まれるプロジェクトの前提として扱われる。ここに記載された内容は比較的参照されやすく、Claude の判断のベースとなるため、頻繁に必要となる前提や方針を記述する。

一方で、rules、skills、agents は、配置しただけで常時参照されるものではない。

rules はファイルパスやタスク内容に応じて条件的に適用される規約であり、常にすべてが読み込まれるわけではない

skills は再利用可能な手順として定義されるが、明示的に呼び出すか、Claude が必要と判断した場合にのみ適用される

agents も同様に、処理の分業が必要と判断された場合にのみ利用される補助的な実行単位である

これらはすべて「候補」として存在しており、文脈に応じて選択的に利用される前提で設計する必要がある。

一方で、hooks および settings.json は、Claude の推論とは独立して動作する実行制御レイヤーである。

hooks は SessionStart や UserPromptSubmit などのイベントをトリガーとして必ず実行される処理であり、設定された内容は確実に適用される。

settings.json も同様に、permissions や hooks の接続などを通じて、Claude の動作を外側から制御する。

この違いを整理すると、各構成要素は次のように位置付けられる。

* CLAUDE.md: セッション開始時に参照される前提（比較的強く効く）
* rules: 条件一致時に適用される規約
* skills: 必要に応じて呼び出される手順
* agents: 必要に応じて利用される役割分担
* hooks: イベントに応じて必ず実行される処理
* settings.json: 実行環境を制御する設定

このため、.claude 配下にファイルを配置するだけで、すべてが自動的に適用されるわけではない。

特に rules、skills、agents については、「配置すれば常に効く」という前提ではなく、「必要に応じて選択される」ことを前提に設計することが重要である。

また、確実に適用したい制約や処理については、可能な限り hooks や settings.json に寄せて制御する方が安定する。

まとめると、.claude は読み込み順によって動作する設定群ではなく、適用条件と実行タイミングによって挙動が決まる構造である。

この前提を踏まえて、各要素を適切なレイヤーに配置することが、Claude の挙動を安定させる上で重要となる。

## まとめ

.claude は設定ファイルの集合ではなく、Claude をどのように動かすかを設計するための領域として扱う。

モデル単体ではなく、rules、skills、agents、hooks、settings といった外部構造を含めて設計することで、実務で利用可能なエージェントとしての振る舞いを実現する。

そのためには、「前提」「規約」「手順」「分業」「強制」「制御」というレイヤーごとに役割を分離し、それぞれを適切な場所に配置することが重要である。

具体的には、前提は CLAUDE.md、条件付き規約は rules、手順は skills、役割分担は agents、強制処理は hooks、実行環境の制御は settings に置く。

これらは読み込み順ではなく、適用条件と実行タイミングによって動作するため、「どこに書けば自然に効くか」という観点で設計する必要がある。

また、rules、skills、agents は配置しただけで常に適用されるわけではなく、文脈に応じて選択される前提で設計することが重要である。

一方で、必ず守らせたい制約や確実に実行したい処理については、hooks や settings に寄せることで挙動の安定性を高めることができる。

要するに、.claude は単なる設定ではなく、Claude をシステムとして扱うための構造であり、各レイヤーの責務を意識して設計することが、安定した運用の鍵となる。
