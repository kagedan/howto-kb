## TL;DR

**AI にコードを書かせ始めたら、逆に考える量が爆増した。**

個人開発で Claude Code に実装を任せるようになってから、コードを書く時間は劇的に減った。代わりに、判断の頻度が跳ね上がった。どの順で作るか、どこまでをこの PR でやるか、いま見えている「やった方が良さそうなこと」をどう扱うか、AI が持ってきた成果物が本当に作りたかったものなのか——AI が止まるたび、こちらの判断待ちになる。

半年くらい試行錯誤して、自分が握るべきものは **「何を作るか (Spec)」と「やらないこと (Backlog)」の 2 つだけ** だと分かった。その 2 つの運用方法を、現状のスナップショットとして残しておきたい。

## 用語の補足

本文に出てくる略称・固有名詞を最初に並べておく (読み飛ばし可)。

| 用語 | 説明 |
| --- | --- |
| **c2** | 自分が Claude Code に付けたあだ名。私に絶対的な力を与えてくれている点では、某魔女と同じですし。 |
| **SoT** | Source of Truth、ただ一つの正本 |
| **Spec** | 設計ログ。Notion 1 ページ = 1 Spec |
| **Backlog** | 「やらないこと」リスト。ローカル md |
| **Plan** | 実装チェックリスト。AI 用の作業メモ |
| **magic word** | PR description に書く `closes <ID>` 等のキーワード。Notion / GitHub が自動で関連付け |
| **unique\_id** | Notion DB が自動採番する短い ID (例: `CC-10`） 以下私は `CC-N` と一般化して記載しています |
| **Gherkin** | `Given / When / Then` 形式の振る舞い記述 |
| **ULID** | 時系列順ソート可能な 26 文字 ID |

## 全体像

文章で書くと分かりにくいので、先に図を出しておく。

自分が触るのは **1 と 3 だけ**。残りは全部 AI が動かす。Spec と Backlog は二つの「拠り所」で、Plan はそれを動かすための作業メモ、というのが全体像。

時系列で見ると、各アーティファクト (Spec / Plan / PR / Backlog) はこういう順で関わってくる。

Spec は残り続け、Plan は merge と同時に archive、Backlog は溜まっていく、PR は流れていく——という非対称な寿命を持つ。

### まとめ

* 自分が触るのは **Spec (設計) と Backlog (やらない仕分け)** の 2 つだけ
* 実装計画 (Plan) は AI に書かせて、AI が自走するための一時ファイルとして使う
* Spec の SoT (Source of Truth) は Notion、Backlog の SoT はローカルの md ファイル
* GitHub PR と Notion Spec のリンクは「PR description の冒頭に `closes <unique_id>`」だけで自動でつながる (Notion 公式の機能)
* 自分が握るのは「何を作るか」と「やらない理由」、それ以外は全部 AI に任せる

## きっかけ

ここ最近、2 つの出来事があって、ようやく運用が回り出した。

1. **["The unreasonable effectiveness of HTML"](https://thariqs.github.io/html-effectiveness/)** — AI が生成する成果物は、**人間が読む** ときに HTML の方が markdown より理解しやすい (図表や CSS で UI 確認もしやすい) という主張。ここから得た刺激は「HTML がいい」そのものではなく、**Spec (当初は Plan と呼んでいた) をもっと見やすくする必要がある** という自覚だった。最初は素直に HTML で Spec を書かせていたが、直近で作った Spec を振り返ると Mermaid 程度の図で十分で、HTML の表現力までは要らない。じゃあ markdown を読みやすくレンダリングする先がほしい——以前同じ課題のために自作した [mdview](https://github.com/bon-clevique/mdview) もあったが、ちょうどそこに 2 番目のきっかけ (Notion Developer Platform) が来た。「そもそも Notion でいいのでは」と。
2. **[Notion Developer Platform 発表](https://www.notion.com/ja/blog/introducing-developer-platform)** (2026-05-13) — Claude Code / Cursor / Codex などのエージェントが Notion と統合できる「外部エージェント」機能、CLI ツール `ntn`、決定論的実行が可能な Worker。これで Spec の SoT を Notion に置く運用が現実的になった。MCP では辛かった「AI が Notion を書きに行く」が、ようやくまともに動く。

この 2 つが揃って、Spec を Notion に置き、Backlog をローカル md に置き、両者を `closes <unique_id>` で繋ぐ運用に踏み切れた。

## 補足: Anthropic 公式が言っていることと、ほぼ同じだった

この運用に辿り着いてから、Anthropic 公式の [Claude Code Best Practices for Large Codebases](https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start) を読んだ。

<https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start>

記事は大企業 / large codebase / multi-engineer org 向けに書かれていて、自分の前提 (個人開発 + ネイティブモバイル) とは規模軸が違う。

それでも、推奨されている方針の大半が、自分が試行錯誤で辿り着いた運用と一致していた。うれしかぁ〜

| 公式の推奨 | 自分の運用 | 一致度 |
| --- | --- | --- |
| **harness が性能を決める** (モデル単体ではない) | 規律を 3 層 (Structural / Operational / Norms) に分けて hook と skill と agent で構造化 | ◎ |
| **CLAUDE.md は pointer 中心で lean に保つ** | CLAUDE.md は index 化。規律本体は別ファイル (workflow-enforcement.md など) に分離 | ◎ |
| **hook で learning を capture** (Stop hook で CLAUDE.md 更新提案) | Stop hook で `activity.log` を自動記録、規律候補は ADR 昇格運用 | ◎ |
| **Skills で specialized workflow を offload** | `/spec` `/ship` `/backlog` など skill 化済 | ◎ |
| **承認 workflow を早期に確立** | Spec 承認は自分の明示発話のみ、c2 の autonomous dispatch を hook で enforce | ◎ |
| **noise を systematic に exclude** | `.gitignore` + `permissions.deny` + `autoMode.hard_deny` で多層化 | ◎ |
| **sub-agent で exploration と editing を分離** | Delivery Manager / Mini-DA / 3 観点 review agent で運用 | ◎ |
| **MCP server で internal tool 接続** | Notion は MCP ではなく REST API 直叩き (理由は本文の Spec の節を参照) | △ 方針同じ、実装手段が違う |

逆に、公式記事に書かれていなかったが自分が深掘りした部分もある:

* **Backlog Discipline (a)+(b) と (a) 現在の Goal と独立 ∧ (b) 今やらなくてもブロックしない、両方 yes なら Backlog 行きという自律判定ルール** — 公式記事は単一 codebase 想定で、横断 backlog は前提に入っていない
* **Plan File Lifecycle Protocol** (active / archived の自動切り替え) — 公式記事は plan が単発で済む想定
* **Spec → Plan → Delivery の 3 段階フローと state file による段階間の継承** — 公式記事は「approval workflow」と一括りに書いている部分を、自分は段階ごとに分離した

自分は何か新しいことを発明したわけではなく、**solo dev という制約に合わせて深掘りした** だけだった、というのが正直なところ。逆に言うと、Anthropic 公式の方針は個人開発でもそのまま効く。組織向けの「DRI 設置」や「cross-functional governance」みたいな部分だけ読み飛ばせば、ほぼ全部使える。

## 何が起きていたか

1 つの Spec を起点に PR が 2-3 本出ることもあれば、実装中に「これは別タスクだな」と気付くことも多い。気付いた瞬間に、AI が手を止めてこちらに投げてくる。

最初は ToDo アプリで管理しようとした。あるいは GitHub Issues、Notion のデータベース、ローカルの TODO.md。色々試したが、どれも **AI と一緒に使えなかった**。AI が読みに行けない、AI が書き込めない、AI の判断と自分の判断の境界が曖昧になる。

「AI に任せる」ためには、AI が判断するための **拠り所** が要る。「人間の頭の中にしかない要件」を AI に逐次伝え続けるのは、自分が逐次判断しているのと変わらない。

最終的に残ったのが、Spec と Backlog という 2 つのレイヤーだった。

## Spec — 「何を作るか」を Notion に置く

> **Spec は「自分が AI と合意した設計」を Notion 1 ページに固めたもの。コードは一行も書かない。**

Spec は設計ログで、Notion DB「Claude Code Specs」に置いている。1 Spec = 1 Notion ページ。

![](https://static.zenn.studio/user-upload/852675bff6e2-20260520.png)

各 Spec の中身は 10 セクションで構成されている。

```
§1 Context        — 問題・背景
§2 Glossary       — 主要用語の定義
§3 Goal           — 達成すべき最終状態
§4 Big Picture    — Goal 達成後の全体像 (sequenceDiagram)
§5 Non-Goal       — 明示的に対象外とすること
§6 Acceptance Scenarios — Gherkin 形式の検証可能条件
§7 Devil's Advocate — 7 観点のチェックリスト ※現在削減検討中、後述
§8 Design         — Phase ごとの sequenceDiagram
§9 Risks & Mitigations
§10 Out of Scope  — Backlog 候補
```

ポイントは 1 つだけ。**Spec はあとから読み返すためのもの**。AI のセッションは終わると忘れる。1 週間後に「あの設計、何だっけ」となったとき、コードを読んでも分からない。Spec を読めば分かる。

### Spec の作り方

> **`/spec` は 4 つの起動パターンを取る。引数で source を指定できるので、会話文脈以外 (自然言語要求 / ローカル md / Notion URL) からも Spec を起こせる。**

`/spec` skill は 0 個または 1 個の引数を取り、第 1 トークンで起動 Mode を分岐させる。

| Mode | 引数の形 | source |
| --- | --- | --- |
| **M1** | 引数なし | 起動以前の会話文脈 |
| **M2** | 自由文字列 (URL/path どちらでもない) | 引数を「ユーザー要求」として受領 |
| **M3** | `Read` 成功する file path | 該当ファイルの内容 |
| **M4** | `https://www.notion.so/...` または `https://<workspace>.notion.site/...` | `read_page.py` で取得した Notion page 内容 |

source が決まると、c2 はそこから Spec 全 section が埋められるか判定する。埋まらないところは `AskUserQuestion` で **最大 3 問まとめて** 投げる (Context / Goal の解像度 → Acceptance Scenarios の閾値 → Out of Scope の境界、の優先順)。source だけで全 section 確定する場合は無質問で進む。

その後の流れは Mode 共通。c2 が日本語タイトルを命名し、テンプレートから md を生成、各 section を埋め、Notion に upload する。upload 後の編集は同 Notion page を上書きし続け、user OK でローカル md を削除して **Notion がただ一つの SoT** になる。

```
# c2 が実行する流れ (ざっくり)
$ /spec [<引数なし | 自然言語要求 | local .md path | Notion page URL>]
# → c2 が引数を classify して Mode 判定 (M1〜M4)
# → source 取得 (Mode 別: 会話文脈 / 引数 / Read / read_page.py)
# → 不足 section があれば最大 3 問まとめて質問
# → source + 回答から日本語タイトルを命名 (例「Notion magic word連携でPR linkを簡素化」)
# → docs/specs/<title>.md にテンプレを展開
# → c2 が各 section を埋める
# → Notion に upload、page_id を state file に保存
# → 自分が Notion URL で内容を確認、OK なら承認
# → ローカル md を削除、Plan Mode へ
```

引数を取れることで何が変わったかというと、Spec の **起点を会話文脈以外にも置ける** ようになった。自分が Notion で書きかけたメモを `/spec <URL>` で食わせる、過去の調査 md を `/spec <path>` で起点にする、Slack や issue の文章を `/spec <貼り付けた要求文>` で渡す——どのパターンでも同じ Spec フォーマットに収束する。会話文脈に依存していた頃は「議論が終わった直後でないと Spec が起こしにくい」という制約があったが、それが解けた。

承認は自分の明示発話 (「OK」「進めて」「承認」など) だけが trigger になる。c2 が勝手に「承認されたものとみなす」ことは禁じてある (沈黙を肯定と解釈させない)。

ここは譲れない部分で、**設計判断の責任は自分が握る** ためのゲートとして残してある。AI に判断を任せる範囲を広げる中で、ここだけは手放さない、と決めた境界線。

### Spec と PR をつなぐ

> **PR description の冒頭に `closes CC-10` と 1 行書くだけで、Notion 側が自動でリレーションを張る。**

実装後の PR を Notion Spec ページに紐付けたい。これは Notion 公式の GitHub integration で実現できる。

PR description の冒頭に `closes CC-10` のような形で Spec の unique\_id を書くと、Notion-GitHub sync が自動で PR と Spec の双方向リレーションを張ってくれる。

```
closes CC-10

## Summary
- Spec CC-10 を実装
- ...
```

これに気づくまで丸 1 日溶かした。Notion AI が「relation property に URL を投げろ」「`pages.create` で external collection に書け」「`connections.notion.*` を使え」と 3 種類の嘘を続けて教えてきて、全部 public API でリジェクトされた。「これは Notion 側で確認しなければならなさそうです」と一旦諦めて Notion 公式 docs を見たら、3 行で「PR description に magic word 書くだけ」と書いてあった。

公式 docs を最初に読むべきだった、というだけの話なのだけれど、AI 出力の自信に流される弱さは記録しておきたい。

## Backlog — 「やらないこと」を仕分ける

> **「気になること」を全部やると PR が膨らむ。Backlog は「やらない」を明示的に格納する場所。**

実装中に「これも直した方が良いな」「あれも気になる」が必ず出る。そのまま手を出すと、PR が際限なく膨らむ。

Backlog はそれを **明示的に「やらないこと」リスト** に格納する仕組み。SoT はローカル `~/.claude/backlog/<project-slug>/` 配下の md ファイル。

```
~/.claude/backlog/
├── claude-config/
│   ├── 01KRV78HAB9J4S796GDRG195X8-magic-word-fixes-vs-closes.md
│   └── human-01KRV78HERQMS6HA77G5J04X98-claude-state.md
└── toukan/
    └── ...
```

frontmatter は 6 フィールド固定で、勝手に増やせない。

```
---
id: <26-char ULID>
slug: <kebab-case>
project: <slug or _global>
status: ready | doing | done
created: <ISO 8601 UTC>
updated: <ISO 8601 UTC>
---
```

body は 3 セクション。

```
## Why
<なぜこのタスクが queue に積まれたか — 作成時に必ず書く>

## How to resume
<再開時の具体的な手順 — file path / 行番号 / コマンド名。「TBD」禁止>

## Done when
<完了の判定基準 — /backlog done で inline log に使われる>
```

「Why」「How to resume」「Done when」が **placeholder (TBD / TODO / N/A など) だと書き込み時点で reject** される。「あとで書く」を構造的に禁止している。

### Backlog Discipline (a) + (b)

> **(a) 現在の Goal と独立 ∧ (b) 今やらなくてもブロックしない、両方 yes なら Backlog 行き。c2 が自律的に判定する。**

実装中に「これも気になる」が出たとき、c2 は次の 2 条件で判定する。

* **(a) 現在の Goal と独立か** — Spec §3 Goal に直接寄与しないなら、別タスク
* **(b) 今やらなくてもブロックしないか** — 今すぐ手を出さなくても進めるなら、後回し可

(a) と (b) が両方 yes なら、Backlog に振り分ける。どちらか no なら、現在の PR スコープ内で処理する。

この判定は c2 が自律的にやる。自分は確認しない。やってみて、後で「これは Backlog じゃなくて本 PR でやるべきだった」とぶつかったら、フィードバックして Spec or 規律に反映する。

### 自動登録

> **Plan 承認時点で「Backlog 候補」を事前登録 → PR merge 時に hook が読みに行って自動生成する。**

PR が merge されると、`plan-archive-on-merge.sh` という hook が Plan ファイルの `## 📋 Backlog 候補` セクションを読みに行って、対応する md ファイルを Backlog DB に自動生成する。

これも半年かけて辿り着いた形で、最初は「完了報告の table を parse する」という設計だったが、parse が 30% くらい失敗していて (AI が table の形式を微妙に崩す)、信頼できなかった。Plan 承認時点で「Backlog 候補」を事前登録する設計に変えたら、parse は要らなくなった。

## なぜ Plan ファイルは作るのか

> **Plan ファイルは AI が「いまどの Phase か」を覚えておくための作業メモ。自分は読まない。**

Spec があって Backlog があれば、間にある Plan ファイルは要らないんじゃないか、という議論を C2さんと何度もやった。実際、設計の途中で「Plan ファイルを廃止する」案を真剣に検討した。

結論としては Plan ファイルは残した。理由は 1 つだけ:

**AI が実装中に「いま自分は何の Phase をやっているか」を覚えておくため**。

Spec は静的な設計ドキュメントなので、Phase 進行のチェックボックスを書き込む場所がない。会話文脈に残しても、context compaction (会話の圧縮) で消える。harness が用意してくれている `~/.claude/plans/<slug>.md` という path に Plan ファイルを置き、PostToolUse hook で active/archived を自動切り替えする仕組みに乗っかった。

PR merge と同時に archive される。自分は Plan ファイルを直接読まない。AI のための作業メモ、という位置付け。

## 自分が握っているもの

> **自分が握るのは 3 つだけ: Spec の承認 / 境界の調整 / 「やらない」の決定権。**

ここまでで、自分が握っているのは 3 つだけ。

1. **Spec の承認** — 「OK」と書くまで実装は始まらない
2. **Backlog Discipline の境界調整** — 後でぶつかったら、AI に対する規律を更新する
3. **「やらない」の決定権** — Spec §5 Non-Goal と §10 Out of Scope は自分が決める

それ以外は全部 AI に任せている。Plan を書くのも、コードを書くのも、PR を作るのも、merge するのも。

「任せる」と言うと無責任に聞こえるかもしれないが、実態は **自分の判断が必要な点を 3 つに絞り込んだ** ということだと思っている。3 つに絞り込めば、それぞれをきちんと考えられる。

絞り込めない頃は、AI が走った後の「何かおかしい」を逐一拾って、その場で修正指示を出し続けていた。今思うと、自分が判断の境界を握れていなかった。境界が無いと、AI は判断しないでこちらに毎回投げてくる。境界が決まると、AI はその内側で自律的に動いてくれる。

## どこまで AI に任せるかは、規律の問題

> **「AI に任せる」は AI 性能の話ではなく、こちらの規律設計の話。AI が判断する拠り所を整えれば、AI は止まらない。**

「AI に任せる」と書くと、AI の性能の話に聞こえる。でも実際は AI 側の問題ではなくて、**こちらの規律設計の問題** だった。

任せる範囲を広げるには、AI が判断するときの拠り所が必要になる。

* Goal が明文化されているか (Spec §3)
* 検証条件が Gherkin で書けているか (Spec §6)
* やらないことが明文化されているか (Spec §5, §10)
* 実装中に出てきた候補をどう仕分けるか (Backlog Discipline (a)+(b))
* 承認は誰が出すか (自分の明示発話だけ)

これらが揃っていない状態で「AI に任せる」と、AI は判断のたびに止まる。多肢選択で投げてくる。「どうしますか?」と聞いてくる。任せたつもりが、結局自分が逐次判断している状態になる。

逆に、上記が揃うと、AI は止まらない。Spec から Plan を起こし、Plan に従って実装し、PR を作り、merge する。自分は Spec を承認した後、merge 完了の報告を見るだけ、ということが起きる。

## 残っているもの

> **§7 Devil's Advocate (42 チェック項目) は重すぎ / Backlog 50 件で読まなくなる兆候 / 月次の「捨てる」セッションが未整備。**

Spec の §7 Devil's Advocate は **7 観点 (Security / PdM / CEO / CTO / Test Architect / Ops / DX) × 6 sub-question = 42 個のチェックボックス** で、これは多すぎる気がしている。「N/A も理由必須」というルールで運用しているのだけれど、N/A の理由を AI に書かせると、N/A の境界が曖昧な観点 (たとえば solo dev なのに CEO 観点で「収益性」とか) が紛れ込む。これは多分、もう半年くらいかけて削っていく対象。

それから、Backlog のタスクが溜まりすぎると、自分が読み返さなくなる。今 50 件くらいあって、たぶん限界が近い。月次で「捨てる」セッションを入れた方がいいのだろうな、と思いつつ、まだ仕組み化できていない。

## まとめ

AI で開発するときに、自分が握るべきものは思っていたより少ない。Spec と Backlog があれば、あとは大体 AI が動いてくれる。
