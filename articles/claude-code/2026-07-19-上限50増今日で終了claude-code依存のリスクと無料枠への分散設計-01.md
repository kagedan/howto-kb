「Claude Codeの週次上限、今日50%増から標準に戻る」——これだけ聞くと他人事に思えるかもしれません。でも私にとっては、"単一ベンダーの上限カレンダーに生活を預けなくてよかった"と再確認する日です。

結論から言います。**Anthropicは2026年5月13日から、Claude Codeの週次利用上限を50%引き上げるプロモを実施し、当初7月13日締切だったものを7月19日23:59（米国太平洋時間）まで延長していました。今日がその期限です。** 終了後は自動的に標準上限へ戻ります（申請不要・課金プランに変更なし）。この記事では、①何が今日変わるのか、②課金コーディングエージェント1本だけに頼る設計のリスク、③私が実際に無料枠中心で運用している設計、の3つを比較して持ち帰れるようにします。

![](https://assets.st-note.com/img/1784394234-UDz8h3BriuQnkoSWesMqOjR5.png?width=1200)

## 今日、正確には何が起きるのか（前提条件）

まず前提を正確にそろえます。日付が命の記事なので、執筆時点（2026年7月19日）で確認できた一次情報だけを使います。

* 対象: Pro・Max・Team、レガシーシート型Enterpriseプランの週次利用上限が**50%増**（CLI・IDE拡張・デスクトップ・Web版すべてのClaude Codeが対象）
* 対象外: Freeプラン、従量課金型Enterpriseシート、Claude Code以外のClaude製品の上限
* 期限: **2026年7月19日23:59（米国太平洋時間）**。延長は今回で2回目（当初終了予定は7月13日）
* 終了後: 自動的に標準の週次上限へ戻る。プラン・請求には変更なし

出典は以下の一次・準一次情報です。

* Help Net Security「Claude Code users keep 50% higher limits until July 19」(2026-07-13)
* Anthropicサポート記事「Claude Code May–July 2026 weekly limits promotion」
* Anthropic公式ニュース「higher-limits-spacex」(2026-05-06、計算資源パートナーシップの発表)

同じ延長期限（7/19 23:59 PT）は複数の独立した情報源（Hacker Newsの議論スレッド、実務系ブログ複数、Anthropicサポートページの更新履歴）でも一致しており、単一ソースの誤報ではないことを確認済みです。なお、このプロモの土台になっているのは2026年5月6日にAnthropicが発表したSpaceXとの計算資源パートナーシップで、これによりClaude Codeの5時間レート制限が倍増し、ピーク時間帯の追加制限が撤廃されています。この記事はプロモの延長・終了そのものを扱っており、5月の発表とは別イベントとして書き分けています。

## 課金コーディングエージェント1本だけに賭けるリスク

ここが本題です。プロモが終わること自体は「元に戻るだけ」なので害はありません。

問題は別のところにあります。**多くの個人開発者が「上限が上がっている今の体感」を基準に、日々の作業量を設計してしまっている**ことです。今日を境に、同じ作業量でも上限に先に当たるようになります。

* 単一ベンダーのプロモカレンダーに、自分の生産性の前提を預けている
* 上限に当たった瞬間、作業が完全に止まる（代替経路がない）
* 「値上げ」ではなく「割引終了」なので、体感の変化に気づきにくい

![](https://assets.st-note.com/img/1784394235-4RNnyqZ1wIcOkX7Kt6Fg8AzW.png?width=1200)

## 私が実際にやっている設計: 無料の第2経路を常設する

私は個人開発者として、Hermesという常駐エージェント基盤をOCI（Oracle Cloudの無料枠サーバー）上で動かしています。設計の要点は「Claude Codeを主戦力にしつつ、日常の調査・監視・下書き生成はSuperGrok（xAIの定額枠）経由のGrok-4.5に渡す」という役割分担です。

* 朝のブリーフ、日次の情報収集、週次の記事下書きはすべてGrok-4.5のcronジョブが無料枠内で処理（例: 毎日16:15に自動起動する調査ジョブ1本だけで、その日のネタ収集が完結する）
* Claude Codeは「人間の判断が要る実装・デバッグ」だけに集中させ、消費を絞る
* どちらか一方が上限や障害に当たっても、もう一方が止まらない

![](https://assets.st-note.com/img/1784394235-ujJxsQ8bHGZlEhIP0wpRSKV1.png?width=1200)

今日のようなプロモ終了日は、単一ベンダー設計だと「体感の変化にやきもきする日」になります。私の場合は「Claude Codeの消費が今日から絞られる」という前提を先に織り込んでいるので、作業計画を変える必要がありません。

![](https://assets.st-note.com/img/1784394236-dtivcyAx4FQH5upIkNhC1wKV.png?width=1200)

## どちらの設計が向くか

私が選んでいるのは後者です。理由は単純で、**上限や値上げのニュースに一喜一憂する回数を減らしたい**から。今日のプロモ終了は、その設計が機能しているかを確認する良いタイミングでした。実際、今日は上限を気にせずいつも通り作業できています。

Claude Proを2アカウント契約していたんですが、昨日付で１つのアカウントは満期日を迎えていたので契約更新していません。  
Fable5 使えるうちにたくさん既存のローカル、OCI環境でのAIエージェント活用の仕組みを作りこんで、これはこれで楽しめました。

## まとめ

* 2026年7月19日23:59(PT)でClaude Codeの週次上限50%増プロモが終了し、標準上限に自動で戻る
* 対象はPro/Max/Team/レガシーEnterprise、Claude Code全環境。申請不要
* 課金エージェント1本への依存はプロモ終了のような「静かな変化」に弱い
* 無料枠（SuperGrok経由Grok-4.5等）へ定型作業を切り出す役割分担が、上限変動に強い設計になる

自分の作業のうち「判断が要らない定型作業」がどれくらいあるか、一度棚卸ししてみることをおすすめします。判定基準は単純です。**同じ手順を毎回同じ形式で繰り返しているか**（朝の情報収集、定期監視、下書き生成など）。当てはまる作業から無料枠へ切り出すのが第一候補です。私自身、この分担にしてからOCI（Oracle Cloudの無料枠サーバー）とSuperGrok（xAIの定額プラン）だけで、日次の調査・監視cronを課金なしで回せています。
