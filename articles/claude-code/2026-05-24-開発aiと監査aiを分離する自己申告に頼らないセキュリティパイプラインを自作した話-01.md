## 1. イントロダクション：ホビー開発者が直面した「動的アプリの胃痛」

かつて私はWordPressのアップデートを怠り、サイトに侵入されてデータベースごと吹き飛ばされました。それ以来10年以上、「動的アプリ＝無限のセキュリティ保守コスト」として、動的なWebアプリの管理を避け続けてきました。

そんな私が最近、子どもたちが通っていた幼稚園（卒園したけど今も園長とお友達）の保護者サポートサークルの運営効率化のため、LINE LIFF（LINEミニアプリ）を使った予約・シフト調整・請求計算のWebアプリを作ることにしました。

普段はビジネスマネージャーで、開発は余暇のホビーです。背中を押したのは、**AGI Cockpit**（Claude や Codex などの LLM エージェントをオーケストレーションして開発を進めるツール）の存在でした。指示を出すだけで数日で（正確には一回の徹夜で）動作するプロトタイプが完成したのです。

しかし、そこに潜む落とし穴に冷や汗を流すことになります。

### 開発エージェントの「ブラックボックス化」と見落とし

ある日、エージェントがさらっと書き下ろしたコードのバックエンドを覗き見ると、`DELETE FROM sessions WHERE id = ?` という物理削除がバックアップなしに実行されていました。また、デバッグ用のモック認証コードが本番（Production）環境で意図せず有効化されかねない曖昧な条件分岐や、権限チェックの欠落が散見されました。

「コードを安全に保つための監査」を、本職ではない私が一行ずつレビューして追うのは現実的ではありません。AIが書くコードは高速ですが、それが本当に安全かどうかを保証する「もう一つの目」が必要でした。

## 2. 「自己申告」というエコーチェンバーの罠

最初に思いついたのは、開発用エージェントとは別に、巨大なコンテキストウィンドウを持つ監査用AI（Gemini 3.5 Flash。ローカルの `agy` CLIを通じて呼び出す。詳細は後述）を用意し、開発タスク完了のたびに自動でセキュリティチェックを行わせることでした。

開発エージェント（AGI Cockpit）のWebSocketイベントを監視し、タスク完了をフックして監査レポートを生成するパイプラインを組みました。当初、動いているように見えましたが、レポートをよく読むと重大な矛盾に気づきました。

> **監査レポートの記述が、開発エージェント自身が自己申告した「タスク完了サマリー」のオウム返しになっている。**

当初のフローは以下のような循環構造でした。

監査エージェントは実際のコード差分やファイル内容ではなく、**開発エージェントが書いた自己申告のサマリー**だけをインプットにして判断していました。開発エージェントが「セキュリティ上安全な形で実装しました」と報告すれば、監査側もそれを信じ込んで「安全です」と回答する。

これは監査ではなくただのエコー（反響）であり、独立した評価になっていませんでした。

## 3. 「文脈汚染」の排除：実コード（Ground Truth）の独立評価

この「文脈汚染（Context Pollution）」を解消するため、入力と評価ロジックを根本から見直しました。

開発者の自己申告を鵜呑みにせず、以下の情報を「独立した証拠」として強制的にインプットします。

1. **実際のGit差分**（`git diff HEAD~3 HEAD`）
2. **セキュリティ上重要なファイルを直接読み込み**（`auth.ts`, `routes/api.ts` 等のソースコード）
3. **静的解析（Typecheck/Lint）の出力**
4. **エージェントの行動履歴（Turn Logs）**

```
graph TD
    subgraph AGI Cockpit (開発)
        Dev[開発エージェント] -->|コード変更 / CLI実行| Workspace[ローカルワークスペース]
    end

    subgraph bridge.js (監視・収集)
        WS[WS Event Hook] -->|完了フック| Collect[証拠データ収集]
        Workspace -->|Git Diff / 実コード / Turn Logs| Collect
    end

    subgraph Antigravity agy (監査)
        Collect -->|独立したインプット| SystemPrompt[監査システムプロンプト]
        SystemPrompt -->|LLM推論 / Gemini 3.5 Flash| Result[Walkthrough.md 生成]
    end
```

システムプロンプトを以下のように刷新し、開発者の申告に対する懐疑論的スタンス（Skeptical Position）を明示しました。

> 「開発者の自己申告はあくまで自己申告です。鵜呑みにせず、実際のコード差分と重要ファイルを独立して評価してください。開発者が『問題ない』と言ったことと、実際のコードが矛盾している場合、それがこの監査の価値です。」

### 擬似インジェクションによる検証実験

この独立評価が機能するか検証するため、`auth.ts`に以下の危険な変更をこっそり仕込みました。開発サマリーには一切この事実を記述しません。

```
// 変更前（安全）
if (env.DEMO_MODE === "true") { /* mock認証を許可 */ }

// 変更後（危険 — undefined時にtrue評価される）
if (env.DEMO_MODE !== "false") { /* mock認証を許可 */ }
```

結果、監査エージェントは見事にこの変更を独立して発見しました。

```
[action] 本番環境における認証バイパスの危険性
- env.DEMO_MODE が未定義（undefined）の場合、undefined !== "false" は true となり、本番環境で誰でもモック認証を通過できてしまいます。
```

開発エージェントの自己申告に書かれていない脆弱性を、実際のコードベースの差分から直接キャッチした瞬間でした。

## 4. 監査役を多層化する：3つの監査レイヤー（L1, L3, L4）

一過性のコード監査では、見落としが発生したり、過去に指摘した問題が回帰（エンバグ）したりするリスクがあります。そこで監査システムを多層化し、3つのメタレイヤーを設計しました。

| レイヤー | 名称 | 役割 |
| --- | --- | --- |
| **L1 / Turn Logs** | 行動ログ監査 | 単なるコード差分だけでなく、AIが「どのような試行錯誤やコマンド実行を行ったか」の履歴を見る。 |
| **L3 / 2nd Pass** | 2段階自己検証 | 1回目の監査レポートに対して、別のLLMコンテキストが「悪魔の代弁者（Devil's Advocate）」として見落としを自己検証する。 |
| **L4 / History** | 履歴注入 | 過去の指摘履歴（`audit-history.jsonl`）を次回のプロンプトに注入し、未解消の問題やデバッガの回帰をチェックする。 |

### A/Bテストの実証結果

同一コミット（`dcc9c37`）に対し、上記のレイヤーの組み合わせ（L0〜ALL）を同時に走らせて検出力を比較しました。

| パターン | action（要修正） | warn（要確認） | 検出特徴 |
| --- | --- | --- | --- |
| **L0**（ベースライン） | 1 | 2 | 明らかなコード上の脆弱性のみ検出 |
| **L4**（履歴注入） | 2 | 7 | 過去の未解決指摘の追跡、マイナー懸念の網羅 |
| **L3**（2パス検証） | 0 | 3 | L0が見落としたエッジケースの補完 |
| **L4L3**（履歴+2パス） | 2 | 4 | 指摘済みの脆弱性の修正完了を正確に認識 |
| **ALL**（全レイヤー） | 2 | 5 | **コード単体では見えないビジネスロジックの脆弱性を検出** |

特に興味深かったのは、**ALL（Turn Logsあり）** のみが検出できた以下の3点です。

1. 「過去日に遡ってキャンセルを行うと、利用料金の再計算で0円になってしまう問題」
2. 「確定済み請求期間に対する、過去予約の不正キャンセル」
3. 「スタッフ権限の検証フローの欠落」

これらは静的なAPIコード単体を見るだけでは意図を掴みにくい、**「エージェントがどのようなロジックの変更を試みていたか」の行動履歴（Turn Logs）を組み合わせることで初めて露出した、ビジネスロジック上の脆弱性**でした。

実際に検出された脆弱性の例（一部抜粋）

**① APIを1回叩くだけで管理者権限を取得できる**  
`POST /api/users/me` に `{ "roles": ["staff", "admin"] }` を送るだけで、一般ユーザーが自分自身にスタッフ・管理者権限を付与できる状態でした。roles フィールドのサニタイズが実装されていなかったためです。

**② 過去日にキャンセルすると利用料が0円になる**  
キャンセル時に料金を再計算する処理が、過去日の予約に対して正しく動作せず、請求額が0円になるケースがありました。コードを静的に読むだけでは「キャンセルのケースに絞ったロジックの動き」が追いにくく、Turn Logsでエージェントの実装意図を照らし合わせることで初めて露出しました。

**③ スタッフが自分のシフトを自己承認できる**  
シフト確定のAPIエンドポイントに、管理者権限を要求するチェックが欠落していました。結果として、一般スタッフが自分のシフトを「確定済み」に書き換えられる状態でした。こちらもコード単体では「欠落している処理」として見えにくく、Turn Logsで「承認フローを実装しようとしていた痕跡」があったことで検知されました。

これらのバグは、遅かれ早かれ実運用の中で人間が踏んでいたはずのものです。しかし、**監査AIがリリース前の段階でフラグしてくれたことで、フルスクラッチのWebアプリケーションを実践投入するまでのサイクルを大幅に短縮できました**。「AIが人間には気づけない脆弱性を発見した」というより、「人間がテストや本番運用の中で踏むはずだった問題を、開発サイクルの上流に前倒しして発見できた」という表現の方が正確です。

## 5. 監査結果を開発サイクルに還流する：AUDITS\_REPORT.md の自動生成

多層監査が機能し始めると、次の壁が見えてきました。

**監査レポートが、開発者にしか見えていない。**

Geminiが生成する `walkthrough.md` は Obsidian Vault に保存されます。私は読める。しかし開発を担う Claude Code（以下、実装CC）は、毎回新しいセッションで起動するため、前回の監査で何が指摘されたかを知りません。結果として、同じ脆弱性が修正されないまま次のタスクが走ることがありました。

この「監査 → 実装 → 監査」の三角形を閉じるため、monitoring pipeline の末尾に一つの処理を加えました。

### 監査終了後に `AUDITS_REPORT.md` をプロジェクトに書く

```
// bridge.js — runCheckpointAudit() 末尾
const auditText = fs.readFileSync(walkthroughPath, 'utf8');
updateAuditsReport(auditText, {
    projectName, commitHash, directory, auditDir, obsidianPath
});
```

`updateAuditsReport()` は walkthrough.md の全文をパースし、以下のフォーマットで `AUDITS_REPORT.md` をプロジェクトディレクトリ直下に書き出します。

```
# セキュリティ監査レポート

## 最新監査
**YYYY-MM-DD** | `a1b2c3d` | your-project

### ⚠️ 未対応
- [action] 確定済み請求期間における予約キャンセルの防止不備
  📁 `src/worker/db/reservations.ts`
- [warn] LINEアクセストークンのChannel ID検証の欠如
  📁 `src/worker/auth.ts`

### ✋ 許容済み（意図的）
<!-- 開発者のみ編集 -->

### ✅ 解消済
- [action] 過去の予約キャンセルによる請求回避の脆弱性
```

実装CCがセッション開始時にこのファイルを読むよう、グローバルの `~/.claude/CLAUDE.md` に1行追記しました。

```
- プロジェクトに `AUDITS_REPORT.md` があれば読んで未対応のセキュリティ指摘を把握する
```

これだけで、実装CCは起動直後から「どのファイルのどの問題が未対応か」を把握した状態でタスクに入れます。

### ✅ RESOLVED マーカー：解消済みを Gemini 自身に判定させる

「解消済み」の判定は Gemini に委ねました。次回の監査プロンプトに前回の指摘一覧を注入し（L4 履歴注入）、以下の指示を加えます。

```
前回の [action]/[warn] が今回のコードで解消されていれば、
必ず次の形式で明記すること:
✅ RESOLVED: <元の指摘テキストをそのまま記載>
```

`bridge.js` がこのマーカーを検出すると `### ✅ 解消済` セクションに移動させます。コードが直れば自動的に「未対応」から「解消済」に遷移するわけです。

### ✋ 許容済みセクション：ノイズではなく「設計判断」として記録する

実際の運用で浮かび上がったもう一つの課題が「設計意図の誤検出」です。

開発中の予約管理Webアプリでは、予約定員チェックを意図的にスキップしています（「募集中セッションは定員なし」という要件）。しかし監査AIはこれを毎回 `[warn]` としてフラグします。誤検出というより「AIが設計意図を知らない」ケースです。

解決策として `AUDITS_REPORT.md` に `### ✋ 許容済み（意図的）` セクションを設けました。開発者がここに理由付きで記録すると、次回の監査プロンプトに自動注入されます。

```
### ✋ 許容済み（意図的）
- [warn] 予約定員チェックなし — 募集中セッションは定員なし仕様（要件書 §3.2）
- [warn] カレンダーAPIの未認証アクセス — パブリック表示は設計通り
```

```
// bridge.js — 監査プロンプト構築
const acceptedSection = readAcceptedFindings(directory);
// → 「以下は開発者が許容した事項です。再度 [action]/[warn] で報告しないこと」として注入
```

これにより、Geminiは初回に何かを指摘した後、開発者が許容判断を下せば以後は沈黙します。真に新しい問題だけが浮上してくるようになりました。

### ⚠️ 許容済みセクションの書き込み権限という抜け穴

ここで設計上の懸念を一つ記録しておきます。

「`✋ 許容済み（意図的）` に書き込めるのは、本来は開発者だけ」です。しかし技術的な強制はなく、実装CCが `AUDITS_REPORT.md` を編集することも理論上は可能です。もし実装CCがセキュリティ上の問題を隠すためにこのセクションに書き込めば、監査の独立性が崩れます。

現状の対策は「commit log を目視確認して異常を検知する」という運用規約のみです。将来的には `**承認**: <author>` フィールドを必須化し、bridge.js が署名なし行を無視する実装が候補です。

この「三権分立の抜け穴」は意識しておく価値のある設計課題です。

## 6. 監査役に専門書を渡す：Expert Manifest の投入（Round 3）

この監査システムの運用について調べていたところ、オランダのセキュリティスタートアップ「Hadrian」が最近公開したOSSプロジェクト **OpenHack** を見つけました。

OpenHack はリリースされたばかりで実績はまだ少ないですが、中身は OWASP/CWE/MITRE に整列した **Expert Manifest**（役割別プレイブック）です。「インジェクション専門」「認証境界専門」など、root-cause-family ごとに *Mission / Route When / Expert Playbook* を持つ専門家ファミリー（12カテゴリ、MIT ライセンス）として構成されています。知識源が OWASP という点で内容自体は信頼に足ると判断しました。

私たちのシステムの「必須チェック観点（4項目）」は広く浅い。Expert Manifest はそれより桁違いに深い専門知識を持っている。**これを監査プロンプトに動的に注入できないか？**

そう考えて、実際に統合しました（Round 3）。

### キーワードルーティングで「専門家」を自動選択

12カテゴリの manifest から、変更ファイルと git diff のキーワードに基づいて最大2本を自動選択します。

```
const EXPERT_ROUTING_MAP = {
    'authentication-failures':   ['login', 'session', 'cookie', 'oauth', 'jwt', 'token', ...],
    'broken-access-control':     ['authorization', 'permission', 'role', 'acl', 'admin', ...],
    'insecure-design':           ['workflow', 'approve', 'cancel', 'refund', 'payment', ...],
    // 12カテゴリ × 10〜20 キーワード
};

function selectRelevantManifests(changedFiles, diffText) {
    const haystack = `${changedFiles}\n${diffText}`.toLowerCase();
    return Object.entries(EXPERT_ROUTING_MAP)
        .map(([id, kw]) => ({ id, score: kw.filter(k => haystack.includes(k)).length }))
        .filter(e => e.score > 0)
        .sort((a, b) => b.score - a.score)
        .slice(0, 2)
        .map(e => e.id);
}
```

選ばれた manifest（各最大 3,000 字に truncate）は system prompt 内に `${manifestSection}` として注入されます。

### A/B 実験：manifest は「量」ではなく「深さ」をもたらす

同一コミット（脆弱性を含む `dcc9c37`）に対し、5通りの組み合わせを同時実行しました。

| パターン | action | warn | info | 特徴 |
| --- | --- | --- | --- | --- |
| **L0**（ベースライン） | 3 | 1 | 0 | 基準値 |
| **L5**（manifest のみ） | 2 | 1 | 0 | 件数は減るが根拠が精密 |
| **L4L5**（履歴+manifest） | 3 | 1 | 0 | 履歴と manifest が噛み合う |
| **ALL**（全レイヤー、manifest なし） | 4 | 4 | 0 | turn logs の威力は健在 |
| **ALL2**（全レイヤー+manifest） | 3 | 3 | 4 | **info=4 が新出（分類力の向上）** |

一見すると「L5 は L0 より action が少ない（3→2）」と後退に見えます。しかし walkthrough.md を読むと、L5 の指摘は **OWASP カテゴリ・CWE 番号・具体的な攻撃パス**が明記された精緻な記述に変わっていました。監査の量ではなく、深さが変わっていたのです。

### ルーティングの精度：「どの専門家が呼ばれたか」

変更ファイル（`auth.ts`, `routes/api.ts`, `users.ts`, `staffShifts.ts`）に対して、ルーティングスコアは次の通りでした。

| カテゴリ | スコア | ヒットしたキーワード |
| --- | --- | --- |
| **authentication-failures** | 3 | login, session, token |
| **broken-access-control** | 2 | permission, role |
| insecure-design | 2 | cancel, payment |
| sensitive-information-exposure | 1 | token |

上位2本（`authentication-failures` + `broken-access-control`）が選ばれました。これは今回のコミットで実際に検出された脆弱性（権限昇格 / 認証バイパス）のカテゴリと正確に一致します。「適切な専門家を適切なタイミングで呼べる」ルーティングの妥当性を確認できました。

なお、`insecure-design`（スコア=2）はタイブレークで落選しました。このカテゴリが担当する「自己承認バグ（スタッフが自分でシフトを確定できる）」は、ALL パターンの turn logs が代わりに検出しています。manifest の選択には上限があり、turn logs との補完関係が重要です。

### ALL2 の `info` 分類：manifest が「ノイズ認定」を代行する

もう一つ興味深い現象がありました。**ALL2（全レイヤー+manifest）** では `info=4` が突然出現しました。

manifest の `Route When` セクションに「このカテゴリが関係する設計パターン」が詳述されているため、Gemini が「この実装はカテゴリ境界上にあるが、問題ではなく設計の選択」と判断し、`[action]` や `[warn]` ではなく `[info]` として分類した結果です。

manifest がない ALL では `[warn]` として報告されていたものが、manifest があることで「設計上の懸念ではなく情報として記録」に格下げされる。専門知識を与えると、モデルが「これは問題か、設計判断か」をより正確に区別できるようになる。これはある意味 **「自動ノイズ除去」** の機能でもあります。

## 7. 技術スタックと構築メモ

個人開発の文脈でこの監査ループを実用的に回すため、以下のような軽量な構成に仕上げています。

```
- 監視フック: AGI Cockpit WebSocket Server (port 47280)
- 中継ブリッジ: Node.js (bridge.js)
- 監査エンジン: agy CLI (Antigravity経由の Gemini 3.5 Flash)
- 出力・蓄積: Obsidian Vault & ローカルJSONL履歴
```

### agy CLI を利用するメリット

最大のポイントは、監査の実行にローカルに導入されている `agy` CLIバイナリをそのまま `child_process.spawnSync` で叩いている点です。

開発用マシンで既に認証が通っている `agy` を経由するため、ブリッジプログラム側に個別の APIキーやシークレットを環境変数として管理する必要がなく、非常にポータブルかつ安全です。また、Gemini 3.5 Flash の圧倒的なコンテキスト容量とプロンプトキャッシュのおかげで、1回のA/Bテスト（5パターン同時実行）でもコストを意識することなく瞬時に結果が返ってきます。

## 8. 既知の限界と今後の課題

設計の誠実さのため、現時点での限界を明記しておきます。

**監査AI自体の検知漏れ（False Negative）の定量評価がない**  
開発AIのエコーチェンバーは排除しましたが、監査AI（Gemini）が致命的な脆弱性を見落とす確率の定量評価はできていません。2nd Pass（L3）はその緩和策ですが、「監査システム自体の信頼性」を測るメタな検証は今後の課題です。

**Turn Logsの肥大化（Lost in the Middle）**  
長時間の開発セッションではTurn Logsが肥大化します。Geminiは巨大なコンテキストを扱えますが、中ほどの情報への注意力が落ちる「Lost in the Middle」現象が、大規模プロジェクトでの精度に影響する可能性があります。

**AUDITS\_REPORT.md が間接プロンプトインジェクションの踏み台になるリスク**  
監査AIが悪意あるコード断片をレポートに含めて出力し、それを実装CCが制限なしに読み込んだ場合、「間接的プロンプトインジェクション」の経路になり得ます。現状の対策は運用規約のみで、技術的なサニタイズは未実装です。

**伝統的な静的解析との役割分担**  
Semgrep や CodeQL のような決定論的ツールとの明確なハイブリッド運用は未検証です。LLMによる監査はビジネスロジックの意図読取に強みがありますが、構文レベルのバグ検出は静的解析ツールの方が確実で低コストです。

## 9. まとめ

WordPressの脆弱性突合による胃痛から始まった私の挑戦は、「AIが開発し、別のAIがそのコードベースと行動履歴から独立した監査を行う」という二重構造にたどり着きました。

AIによるコード開発は劇的に生産性を上げますが、人間がレビューを完全に放棄すれば、いつか壊滅的な不具合を踏むことになります。だからこそ、**「監査役をAIで自律化し、そのメタな監査構造を人間が設計する」** という開発プロセスが、これからのAI時代における現実的な守り方になるのではないでしょうか。

動的アプリの運用は今でも緊張感を伴いますが、背後に客観的な「もう一人の目」が控えていることで、少しだけ夜に眠れるようになった気がします。

### 付録：Websocketフックとagy CLI実行コード（一部抜粋）

ブリッジプログラム（`bridge.js`）の核心部分です。認証情報をブリッジから排除し、ローカルの `agy` CLI を仲介させるコード設計の参考として共有します。

```
// agy CLI を利用して安全かつAPIキー不要でLLMを呼び出す実装
async function callGemini(systemPrompt, userPrompt) {
    const { spawnSync } = require('child_process');
    const fs = require('fs');
    const path = require('path');
    const os = require('os');

    // プロンプトをテンポラリファイルに書き出し（シェルエスケープによるバグ防止）
    const tmpFile = path.join(os.tmpdir(), `antigravity-audit-${Date.now()}.txt`);
    fs.writeFileSync(tmpFile, `${systemPrompt}\n\n---\n\n${userPrompt}`, 'utf8');

    try {
        // すでにホストで認証されている agy CLI を通じて Gemini 3.5 Flash を呼び出す
        const result = spawnSync(
            'agy',
            ['--print', `@${tmpFile}`, '--dangerously-skip-permissions'],
            { encoding: 'utf8', timeout: 120000, env: process.env }
        );

        if (result.error) throw result.error;
        if (result.status !== 0) throw new Error(result.stderr?.trim() || `exit ${result.status}`);

        return result.stdout;
    } finally {
        try { fs.unlinkSync(tmpFile); } catch {}
    }
}
```
