## 1. 「ハーネス」という言葉の重さ — Messages APIからCMAまでの道のり

ここでは、Isabellaさんが冒頭で語った「ここまでの歴史」を入り口に、ハーネスという言葉がなぜ重要なのかを整理してみます。

そもそも「ハーネス(harness)」という用語を、私は普段から「モデルを実際に動かすときの足回り一式」=「エージェントループ、ツール実行、状態管理、リトライ、観測、スケーリング、サンドボックス」をまとめた呼び方として使っています。エージェント開発に踏み込んだことのある人なら、この「足回り」の重さに一度は頭を抱えたことがあるはずです。

Isabellaさんは、Claudeのプログラム的な利用が「Messages API → Agent SDK → Claude Managed Agents」という3段階で進化してきた、と整理していました。聞きながら頭の中で並べ直すと、これは **「開発者が自前で組まなければならない範囲が、段階的に削られていく」歴史** として読めると感じました。

セッション中、この3段階を一枚で並べたスライドが出てきました。各世代で「YOU MANAGE(自分でやる範囲)」と「ANTHROPIC PROVIDES(Anthropicが提供する範囲)」がどう変わっていくか、視覚的に追えるようになっています。

![Messages API / Agent SDK / Managed Agents を並べた進化図。世代を追うごとに「自分でやる範囲」が削れ、「Anthropicが提供する範囲」が広がっていく構造](https://static.zenn.studio/user-upload/deployed-images/421597577566135ec7ae0f76.png?sha=d9886cac871380f8be42c36d8f21bddef403c330)

**第1段階(2023年〜):Messages API。** いわゆる「トークンを投げてトークンを返してもらう」生のAPIです。プリミティブだから自由度はとても高い反面、エージェントループも、ツール実行も、コンテキスト管理も、コンパクションも、観測も、スケーリングも、足回りは全部自分で組まなければなりませんでした。この時期にエージェントらしきものを動かしていた人は、おそらく相当な量の「LLMの周りを支えるコード」を書いてきたはずです。

**第2段階:Agent SDK。** ここで視界が一段広がりました。Claude Codeをプログラム的に呼び出せるハーネスとして提供され、Claudeが計算機やファイルシステム上で実際に行動を取れる、強力なベースが手に入ります。第1段階で重荷だったエージェントループやツール実行の主要部分は、SDK側が引き受けてくれるようになった一方で、ホスティング・スケーリング・サンドボックス管理は、依然として開発者側の責任として残っていました。

**第3段階:Claude Managed Agents(CMA)。** そして今回の主役です。第2段階で残っていた「ホスティング・スケーリング・サンドボックス」までを含めて、エージェント実行の足回りをまるごとAnthropic側のマネージドインフラに寄せにきた、初のハーネスです。開発者は、ドメイン知識やカスタマイズ性が本当に必要な部分 — タスクの設計、エージェントの設定、カスタムツールのロジック — にフォーカスできます。

こうして並べてみると、 **「Messages APIの頃に自分で組んでいた『LLMの周りを支えるコード』が、世代を追うごとに少しずつハーネス側に吸収されてきた」** という構図がはっきりします。私はもともと、自分の業務の中でこの3段階を意識して使い分けていたわけではありませんでした。ただ、過去に「PoCで作ったエージェントの足回りを、半年後にまるごと作り変える羽目になる」というシナリオを何度か見てきた身としては、「自前ハーネスの維持コスト」という問題意識は強く持っていました。今回のセッションで一番刺さったのは、まさにこの問題意識に正面から向き合った一文です。

> "Harnesses should evolve alongside your agents."  
> — Isabella He
>
> 「ハーネスは、エージェントと共に進化しなければならない」

**ここで感じたこと:** この一文は、しばらく自分の机の前に貼っておきたいくらい刺さりました。モデルの世代が変わるだけで、過去に最適化したつもりのプロンプト・ツール設計・終了判定ロジックが、まるごと意味を失うことがあります。彼女自身、Sonnet 4.5の「コンテキスト不安(context anxiety)」(余裕があるのに早めにタスクを切り上げる挙動)に対応するための緩和策をハーネスに組み込んだら、続いて出てきたOpus 4.5でその挙動が消えていて、緩和策ごと不要になった、という生々しいエピソードを語っていました。これは内製ハーネスを抱えた組織なら誰でも遭遇しうる風景です。

### ここで全体構造を掴むための例え

このあたりで、私の頭の中で全体像を支えてくれた例えを差し出しておこうと思います。CMAを理解しようとするとき、私は **「個人で飲食店をゼロから立ち上げるか、フランチャイズに加盟するか」** という比較で考えると、しっくり来ました。

* **Messages API**:キッチンも厨房機器も什器も、自分で揃え、衛生管理・在庫管理・POS導入・スタッフ教育まで全部自前でやる個人飲食店。圧倒的な自由度の代わりに、店を開ける前にやることが膨大。
* **Agent SDK**:仕入れ網と一部レシピは本部から提供されるが、店舗運営(出店場所選び、内装、ホール運営)は自分で頑張る形態。だいぶ楽になるが、店舗運営の責任は自分。
* **CMA**:本部が「店舗運営フレーム」=POS・物流・冷蔵設備・清掃マニュアル・衛生監査・スタッフトレーニングまで一通り提供してくれるフルパッケージ。オーナーは「自店ならではの品揃え・接客方針・地域特性」に集中できる。

この比喩で省いていることもあります。フランチャイズには本部の意向に縛られるという制約があるのと同じで、CMAにもベンダー固有のサービスに依存するという面はあります。それでも、「足回りを誰が持つか」というレイヤの分け方を理解するには、この比喩は十分に効くと感じます。

Isabellaさんはもう一つ、印象的な数字を出していました。

![「The fastest way to build and ship transformative agentic products — 10–15× faster to production」と書かれたスライド](https://static.zenn.studio/user-upload/deployed-images/3007fc6d03372cf0054f2077.png?sha=4ed8f42dd705e0cf2acb907c744cb4519c427174)

> "We've seen people build 10 to 15 times faster to production with Claude Managed Agents by leveraging our purpose-built harness."  
> — Isabella He
>
> 「Claude Managed Agentsの専用ハーネスを使うことで、本番出荷までのスピードが10〜15倍になった事例を見てきました」

**ここで感じたこと:** この10〜15倍という数字は、もちろんAnthropic側の観測値であって、自分の現場でそのまま再現できる保証はありません。ただ、「本番化のラストワンマイル」(ホスティング・スケーリング・耐障害設計・セッション永続化)が抽象化されたときの出荷速度の変化は、肌感覚としてはとても理解できます。私自身、過去にエージェントの再開性を真面目に設計しようとして、イベントソーシング的なログ基盤を組み始めた瞬間に、プロジェクトが一気に「インフラ案件」化してしまった経験があります。あのときの痛みを思い出すと、ハーネス層をベンダーに寄せるという選択肢は、確かに数倍のスピード差を生む可能性がある、と素直に感じます。

ただし、これが自分の現場でそのまま通じるかは、まだ分からない部分もあります。マネージドの選択は、自由度・可逆性・データ境界の制約とのトレードオフでもあるからです。このあたりは、本記事の後半でもう一度触れたいと思います。

## 2. CMAを構成する3つのリソース — agents / environments / sessions

ここでは、CMAの上で何かを構築するときに使う、最も基本的な3つのリソースを整理します。

Isabellaさんの説明はこうでした。

![Three primary resources：agents(/v1/agents)、environments(/v1/environments)、sessions(/v1/sessions) の3つを横並びで比較したスライド](https://static.zenn.studio/user-upload/deployed-images/c10ccf488bb6a9de479a7cfa.png?sha=58a2348483dcb76eb6a6cc6faf3d194bf6200858)

1. **agents** — 「ペルソナと能力(persona and capabilities)」を定義する場所。使うモデル、システムプロンプト、使うツール、利用するMCPサーバー、スキルなど、エージェントがエージェントループの中で参照できる「頭脳側」の設計図。
2. **environments** — エージェントが実際に行動を取る「場所」。コンテナとネットワーク設定の組み合わせで、エージェントの「手」が動く物理的(=実行環境的)な空間。
3. **sessions** — agentsとenvironmentsを結びつける単位。「あるenvironment上の、あるagentインスタンスのセッション」が1つ立ち上がり、ユーザーとのやり取りを行う器になる。

ここで、先ほどの「フランチャイズの飲食店」の比喩を引き継ぐと、整理がしやすいです。

* **agents** = 「うちの店は何屋で、何を売りにするか」の店舗コンセプトとレシピ集
* **environments** = 「実際に料理を出す厨房と店内」
* **sessions** = 「お客さん一組ごとの、入店から会計までの一連のやり取りログ」

もう少し技術寄りに置き換えるなら、私の現場感覚ではKubernetesの世界の対応関係に近いと感じました。

* **agents** ≒ Deployment(設計図)
* **environments** ≒ Pod(実体)
* **sessions** ≒ そのPod上で起きたやり取りのログ・ストリーム

この対応で考えると、「同じagent設計から複数のenvironmentを立ち上げる」「同じenvironment上で複数のsessionが走る」といったスケーリング設計が、見通しよく整理できます。インフラエンジニア視点で言えば、これはマルチテナント設計の素材としてもかなり扱いやすい構造です。「ユーザーごとにenvironmentを分離してセキュリティ境界を引く」「sessionは使い捨てに設計してログだけ残す」など、すでに見慣れたパターンが自然に成立します。

Isabellaさんは、もう1つ重要なポイントを挟んでいました。アーキテクチャ上「エージェントループはサーバーサイドで動く」という、地味だが効いてくる前提です。

![Architecture: The agent loop runs server-side。「You send events. You stream events back. The loop runs reliably whether your client is connected or not.」というメッセージ](https://static.zenn.studio/user-upload/deployed-images/11345230001aec349213dee5.png?sha=f11c80fa8ca12823bd0359f2b458c4eba3e524cd)

> "A key thing here, as I alluded to briefly before, claude managed agent has the agent loop run server side. This means that a lot of the complexities that come with managing hosting and scaling are abstracted away."  
> — Isabella He
>
> 「重要な点として、Claude Managed Agentsではエージェントループがサーバーサイドで動きます。つまり、ホスティングとスケーリングに伴う多くの複雑さが抽象化されています」

**ここで感じたこと:** 「エージェントループがサーバーサイドで動く」という一言は、聞き流せるくらいシンプルですが、運用設計の前提を結構大きく動かす一文だと受け取りました。これまでは「ユーザーセッションをWebアプリ側でどう保持するか」を頑張って設計していたところに、もう一段「エージェント側のセッションの永続性」というレイヤが加わるからです。CMAはここを引き受けに来てくれた、と理解しました。

!

**💡章のまとめ:3つのリソースの役割分担**

* **agents** = 設計図(モデル・プロンプト・ツール)
* **environments** = 実行コンテナ(ネットワーク設定込み)
* **sessions** = agent × environment 上で進行する一回の対話=イベントログ
* エージェントループはサーバーサイドで動くため、ラップトップを閉じても状態が維持される

## 3. 「脳」と「手」を分離する — 設計判断の本質

ここでは、CMAのアーキテクチャの中核にある設計判断「エージェントループ(脳)とツール実行(手)の分離」について整理します。私にとっては、このセッションでもっとも考えさせられたパートでした。

彼女の説明をかみ砕くと、こうなります。

これまでの多くのエージェントハーネスでは、エージェントループ(脳)とツール実行(手)が **密結合(tightly coupled)** されていました。同じコンテナの中で、Claudeが思考もすればツールも叩く、というスタイルです。この設計は、Claude Codeのようにエージェントにファイルシステム上で自由に振る舞ってほしい場合には、今でも理にかなっています。

しかし、エージェントの種類によっては、「脳と手を切り離したい」シナリオが出てきます。たとえば、

* クレデンシャル(credentials)とセキュリティの観点。脳と手を切り離せば、エージェントは「暗号化された状態でしか実際のクレデンシャルにアクセスできない」設計にできる。
* サンドボックス境界を明確にしたい場合。脳側を分離することで、「副作用を持つ層」がきれいに局所化される。
* レイテンシ。同じコンテナ内に同居していた頃は、セッションを立ち上げるたびにコンテナを起動する必要があり、TTFT(time to first token)が悪化していた。

セッション中、この設計判断を表す「The brain left the box(脳が箱を出た)」というタイトルの図が示されていました。左の「BEFORE」では、エージェントループとツール実行が同じ箱に詰まっていて、セッションごとにそれを1つ立ち上げる構造です。右の「NOW」では、 **脳(Agent loop)はAnthropicがマネージしている1つのサービスで動き、手(Sandbox)はツールが本当に必要なときだけ立ち上がる** という分離になっています。

![BEFORE vs NOW の比較図。BEFOREは「ONE CONTAINER PER SESSION」で脳と手が同じ箱に入っていた。NOWは脳がAnthropic-managedの常駐サービスで、手(Sandbox)は必要なときだけプロビジョニングされる構造](https://static.zenn.studio/user-upload/deployed-images/a32d920e85bb0dabe51cbef8.png?sha=3d5c044f4886c87e858a439b8456c9d6828f5972)

そしてIsabellaさんはここで、印象的な数字を挙げていました。

> "Our teams actually saw reductions in time to first token along the lines of over 90% reduction in TTFT for our P95 metrics on latency."  
> — Isabella He
>
> 「私たちのチームでは、レイテンシのP95メトリクスにおいて、TTFTが90%以上削減されたケースを観測しています」

**ここで感じたこと:** この90%という数字も、文脈をきちんと読む必要があると感じました。あくまでAnthropic内部の特定条件下での観測値であり、自分の現場で同じ数字を期待するのは早計です。ただ一方で、「コールドスタートを避ける効果」については、サーバーレス設計でTTFTに苦しんだ経験のあるエンジニアなら、肌感覚で十分にうなずける話だと思います。脳と手を分けるだけで、手側のコンテナの起動コストをセッション立ち上げのクリティカルパスから外せる、というのは構造的に効く改善です。

そして、私にとってこのセッションでいちばん象徴的だったフレーズが、次の一文です。

> "You essentially want to be able to decouple the hands from the brains of the agents."  
> — Isabella He
>
> 「本質的にやりたいのは、エージェントの『手』を『脳』から切り離せるようにすることです」

**ここで感じたこと:** このフレーズは、これから先のエージェント設計の標語のように使われていく予感がしました。私はもともと「マイクロサービスの責務分離」を仕事の軸にしてきた人間なので、「LLM側の判断と、副作用を持つ実行系を疎結合にする」という考え方には、強い共感を覚えます。LLMがどんどん強くなる以上、副作用境界の設計はますます重要になっていく、というのが私の見立てです。

### 私の中で既存知識と橋渡しした整理

ここで一段、自分の知っているクラウド/Kubernetes界隈の概念と対応をとってみると、こんな整理になりました。

* 「脳」と「手」の分離 ≒ **コントロールプレーンとデータプレーンの分離**
* マイクロサービスにおける **「ステートを持つサービス」と「副作用を持つサービス」の責務分割**
* サーバーレス設計における、ロジック層とI/O層の分離

つまり、CMAが提示している分離アーキテクチャは、完全に新しい発想というよりは、分散システムの世界で広く受け入れられている設計思想を、エージェント領域に丁寧に持ち込んだものだと感じました。新しい言葉に身構える必要はあまりなく、むしろ「これは知っているやつだ」と既存の引き出しから道具を取り出せる、というのが、個人的にはありがたい発見です。

もちろん、これは私の受け取り方であって、別の整理の仕方もあると思います。たとえば「Claudeが思考しながら同時にツールを叩く時間が確保されたほうが、エージェントの賢さが活きる」という観点を重視する人にとっては、密結合のメリットを擁護する余地はまだあるはずです。このあたりは、ユースケースごとに是々非々で考えるしかないところです。

**💡章のまとめ:脳と手の分離が同時に効く3つの観点**

* **セキュリティ**:クレデンシャルやサンドボックス境界を脳側から切り離せる

## 4. セッションは「イベント」で語る — リクエスト/レスポンスからの脱却

ここでは、CMAがセッションをどう表現しているか、という話を整理します。これも、ハンズオンの中盤でIsabellaさんがさらっと触れていたのですが、後から効いてくる重要なポイントでした。

CMAのセッションは、「トークンが入ってトークンが出る」モデルではなく、**イベント(events)** を単位として動きます。具体的には、

* ユーザーからのメッセージ
* エージェントによるツール呼び出し
* エージェントからの応答
* (内部の)ステート遷移

といったものが、すべてイベントとしてセッションログに追記されていきます。

セッション中に示された「Sessions speak in events, not request/response」のスライドが、これを直感的に伝えてくれます。左に「自分が送るイベント(user.message、user.custom\_tool\_resultなど)」、右に「自分が受け取るイベント(agent.message、agent.tool\_use、session.status\_idleなど)」が並んでおり、エージェントとのやり取り全体が `{domain}.{action}` という命名規則のイベント列として表現されているのが分かります。

![「Sessions speak in events, not request/response」のスライド。左:「Events you send」(user.message、user.custom_tool_result、user.tool_confirmation、user.interrupt) / 右:「Events you receive」(agent.message、agent.tool_use、agent.custom_tool_use、agent.mcp_tool_use、session.status_idle、session.error)](https://static.zenn.studio/user-upload/deployed-images/ef4ccb1bdcd83a242b50650b.png?sha=be27c65fab9ae25478a69b41b4f8a2c0506101f3)

> "The key portion here is that when our Claude Managed Agents runs within a single session, instead of responding in tokens in and tokens out, it actually works in units of events."  
> — Isabella He
>
> 「ここでの重要な点は、Claude Managed Agentsが1つのセッションの中で動くとき、『トークンが入ってトークンが出る』のではなく、『イベント単位』で動作する、ということです」

**ここで感じたこと:** 「セッションはイベントで語る」という発想は、私の頭の中ではほぼ **イベントソーシング(Event Sourcing)** と重なりました。リクエスト/レスポンス型だと、「過去のあるタイミングの状態を再現する」「途中から再開する」「同じ症状を再現する」といったことがとても難しくなりますが、イベントログ型なら、過去のセッションを「再生」する形で再現できます。これは本番運用におけるデバッグ・監査・障害分析の解像度を、根本から底上げします。

ここで使える例えとしては、**「録画した試合のリプレイ」** が分かりやすいかもしれません。リクエスト/レスポンス型のエージェントは「試合中の選手にその場で質問する」ようなもので、終わってしまえば情報は霧散します。一方、イベント駆動のセッションは「試合全体を録画しておいて、後から何度でも巻き戻して見られる」状態に近いです。クリティカルなプレーがあれば、後から何度でも検証できる。エージェントの本番運用において、この差はかなり大きいと感じます。

### セッションが持つ「状態」

彼女は、セッションが以下の4つの代表的な状態を持つことにも触れていました。

* **idle**(待機中)
* **running**(実行中)
* **rescheduling**(再試行スケジューリング中)
* **terminated**(終了)

これは単なる状態名の列挙ではなく、 **状態機械(state machine)として明確に定義されている** という点が重要です。セッションは `idle` で始まり、`user.*` イベントが来ると `running` に遷移、一時的なエラーが起きれば `rescheduling` でリトライ、回復不能なら `terminated` で終わる — という遷移が、ハーネス側で完結しています。

![Four session statusesの状態遷移図。idle → user.*イベント → running → (transient errorで)rescheduling → auto-retryでrunningに戻る、または unrecoverable errorで terminated に遷移する流れ](https://static.zenn.studio/user-upload/deployed-images/782ab541b867229300d35608.png?sha=e5a74ae1f284c61434071d6c6641e95c41e4fab5)

これらの状態遷移をハーネスが面倒見てくれるおかげで、Webhookを使って外部イベントからセッションを再開させる、特定の状態のエージェントを起こす、といった連携が、追加の設計コストなしで実現できます。エージェントは「ユーザーが話しかけたから動く」だけでなく、「監視アラートや業務イベントを起点に自分で動き出す」ことができるようになる、ということです。

### 私の現場感での整理

私の現場感覚で、エージェントの本番運用でいちばん困るのは、次の3点でした。

1. 「いつ・何が・どんな順番で起きたのか」が分からない
2. 同じ症状を再現できない
3. ユーザーの主観報告(「変な答えが返ってきた」)と、実際の挙動を突き合わせられない

CMAがイベントログ駆動になっているということは、この3点に対する解像度が、構造的に上がる、ということを意味します。インフラエンジニアにとっては、「分散トレーシングや構造化ログでアプリの内部を可視化する」のと同じ感覚で、エージェントの内部を可視化できる、と理解しました。

さらに地味だが嬉しい副作用として、 **「会話がクラウドに残る」** という性質があります。Isabellaさんのスライドではこう示されていました — ハードリフレッシュしてもセッションは残っている、なぜなら自分でデータベースを書いていないから、と。`client.beta.sessions.list(agent_id=...)` で過去のセッション一覧が取れ、`client.beta.sessions.events.list(session_id)` でその完全なリプレイができる、というシンプルなAPIで、セッション永続化が表現されています。

![「The conversation lives in the cloud」のスライド。client.beta.sessions.list(agent_id=...) で全セッション、client.beta.sessions.events.list(session_id) でフルリプレイが取れることを示すコード例](https://static.zenn.studio/user-upload/deployed-images/8ce5211757805ea3484196d4.png?sha=bff5cfe28e960dcc8b312df8b217d4d82740480c)

!

**💡章のまとめ:セッション=イベントログの威力**

* ユーザーメッセージ、ツール呼び出し、エージェント応答すべてがイベント化される
* 状態(idle/running/rescheduling/terminated)はハーネスが自動管理
* Webhookと組み合わせれば、外部イベント駆動でエージェントを起動可能
* リジューム、再現、監査、障害分析の解像度が一段上がる

**ここで感じたこと:** この **「oversight」という言葉の選び方** に、私は強く意味を感じました。「automation」でも「replacement」でもなく「oversight」と表現しているのがポイントです。エージェントが何でも勝手にやる未来ではなく、「人間が見ている前で、エージェントが手を動かす」未来像が示されている、と私は受け取りました。

ここはちょうど、自動運転車の議論に似ているかもしれません。完全自動運転を一足飛びに目指すのではなく、まずは **「自動操縦 + 管制官」** のような、人間が空全体を見渡しながら、機械が個別の操作を担う構図に近づいていく、というイメージです。エージェントが手を動かし、人間は重大な判断と全体の方向性を握る。

私自身、現場ではエージェントを「完全代替」ではなく「強力な共同作業者」として位置付ける設計のほうが、運用上うまく回る印象を持っています。「どこまで自動化させるか」のラインを、業務リスクごとに段階的に引き上げていく — たとえば、

* まずは「原因特定+提案」までを自動化、修正は人間
* 次に「修正パッチの作成」まで自動化、マージは人間
* 一部の低リスクな修正(設定値の戻しなど)は自動マージまで許容
* インシデント宣言、関係者通知、ポストモーテム下書きまで自動化

といった具合に、業務ごとに違うラインを引いていくのが現実的だな、と感じます。

ただ、ここはまだ自分の中で言語化が追いついていない部分かもしれません。「監督役に回る」というのは口にするのは簡単でも、実際にどんな運用フロー、どんなUI、どんな承認基準で支えるかは、組織ごとに大きく違ってくるはずです。このあたりは、これから現場で試行錯誤しながら言葉を作っていくテーマだと感じます。

> — Isabella He
>
> 「クラウド側や自社インフラ側に残しておきたくないセッションがあるなら、削除をプロアクティブに管理できます。削除されたセッションは、あらゆるログ面からも取り除かれます」

**ここで感じたこと:** 「データを残せます」というアピールはどのベンダーもしますが、「データを消せます、しかも全ログ面から」というアピールを正面から打ち出すサービスは、案外少ない印象です。実運用ではむしろ「右から左に消せること」のほうが大事なケースが多く、エンタープライズ導入の文脈ではここがかなり重要になります。個人情報、機微情報、ユーザーからの開示請求、保持期間ポリシー — どれを取っても、「消せる」が前提でないと話が進まない領域があるからです。

ここでは、CMAの「最低限のプリミティブ」のさらに先にある、発展機能群について整理します。Isabellaさんは、後半でこれらを駆け足で紹介していました。

セッションでは、9つの発展機能(全部Betaラベル付き)が1スライドに整然と並んでいました。

![「Beyond the basics」のスライド。Subagents、Memory、Outcomes、Vaults、MCP servers、Webhooks、Permission policies、Interrupt、Console agent builder の9機能がアイコンと短い説明付きで並ぶ。全てBetaラベル付き](https://static.zenn.studio/user-upload/deployed-images/423fb5b8188a0d213608ceb3.png?sha=7e92f5cc0c3cd58850e71f85db4f35b345d42480)

* **Sub-agents(サブエージェント/マルチエージェント)** — オーケストレーターとなるエージェントが、別のエージェントのコンテキストを立ち上げてタスクを分担する仕組み。各サブエージェントが独自のコンテキストウィンドウを持つため、コンテキスト管理にも並列処理にも効く。
* **Memory + Dreaming(メモリと「夢を見る」サービス)** — Claudeが自分のメモリログを振り返り、「何を覚えておくべきか」を自分で判断する仕組み。ユーザー嗜好や訂正履歴のうち、本当に大事なものだけを長期記憶として保持できる。
* **Outcomes(アウトカム)** — エージェントのタスクを「ツールコールの集合」ではなく「達成すべき結果(rubric)」として定義する仕組み。エージェントは達成基準に向かって、どのツールをどう呼ぶかを自分で考える。
* **Vaults(ボールト/クレデンシャル管理)** — ユーザー単位、セッション単位で、クレデンシャルを暗号化された形で安全に管理する仕組み。「脳」から実際のクレデンシャルが見えない設計が、原理的に成立する。
* **Webhook / Fine-grained Permissions / MCP server controls / Console Agent Builder** — 外部イベント駆動、細粒度の権限ポリシー、新しいMCPサーバーコントロール、コンソール上のAgent Builderなど、本番運用に必要な周辺装備。

この中でも、私がとくに印象に残ったのは、Outcomesの設計思想でした。

> "With outcomes you can define a rubric of exactly what you want the agent to produce and it'll figure out along the way which tool calls and what it needs to do to execute towards that final result."  
> — Isabella He
>
> 「Outcomesでは、エージェントに何を作ってほしいか、その評価基準(ルブリック)を定義できます。エージェントはその最終結果に向けて、どのツールをどう呼ぶかを自分で考えて進めます」

**ここで感じたこと:** このアプローチは、私の中では **「コードで命令する」から「結果を契約する」への転換** に近いと感じました。SLAやSLOで結果を保証するクラウドサービスの考え方を、エージェント領域に持ち込んだもの、とも言えそうです。プロンプトに細かい手順を全部書くスタイルは、長期的には少しずつ廃れていく、というのが私の読みです(これはあくまで現時点での私の予測で、現場のリアリティは違うかもしれません)。

VaultsとWebhook + Fine-grained Permissionの組み合わせも、インフラエンジニアにとっては「これがあるとないとで運用設計が大きく変わるな」と感じる部分でした。Vaultsは、自前で「シークレット管理基盤(Vault、Secrets Manager、KMS、IAMロール…)」を組み立ててきた経験のある人なら、嬉しさが直感的に分かるはずです。エージェントが「クレデンシャルそのもの」ではなく「クレデンシャルを使った結果」だけにアクセスできる、という分離設計は、原理的には **ゼロトラストの考え方** と非常に近い形に着地しています。

これらの発展機能を全部自前で組もうとすると、それぞれが「そこそこ大きなプロジェクト」になります。これらが「箱の中」に揃っていることは、戦略的には大きな意味を持ちます。

## 7. 私が明日から試してみたいこと

ここまで動画を消化してきて、自分の現場に持ち帰りたいことはたくさんあります。Sub-agents、Outcomes、Vaults、Webhook駆動 — どれも魅力的に映ります。

ただ、全部を一度に試そうとすると、結局どれも中途半端に終わるのが目に見えています。なので、ここでは思い切って **1つに絞って** 書いてみたいと思います。

### 試してみたい1つ:既存のエージェント実装を「3層」に分けて図示してみる

具体的には、自分が普段触っているエージェント実装(または、お客様と一緒に設計しているエージェント案件)を、

* **agents層**(モデル・プロンプト・ツール定義 = 「脳」の設計)
* **environments層**(実行コンテナ・ネットワーク境界・クレデンシャル境界 = 「手」が動く場所)
* **sessions層**(イベントログ・状態遷移・再開ポリシー = やり取りの履歴)

の3つに分けて、ホワイトボード1枚に図示してみる、という小さな試みです。

これは派手なアウトプットではありませんが、自分の中で **「どこを自前で持っていて、どこをマネージドに寄せられるのか」** という責任分界点を、目に見える形で意識する効果があると感じています。やってみると、たぶん次のような問いが自然と出てくるはずです。

* 「sessions層を、本当にイベントログとして観測できる仕組みになっているだろうか?」
* 「environments層のクレデンシャル管理は、最小権限の原則に沿っているだろうか?」
* 「agents層のプロンプトに、本当はツール側に押し出すべき記述が残っていないだろうか?」

これらは、いきなり「CMAに乗り換える」「自前ハーネスを全部捨てる」といった大きな話ではなく、明日から1時間あれば始められる地味な棚卸しです。私は派手な意思決定よりも、こういう地味な棚卸しのほうが、結果的には自分の現場を変えていくと感じています。

そして、この棚卸しの結果次第で、「次に何を試すか」が見えてくるはずです。観測性が薄ければイベントログ基盤の整備が先になるでしょうし、クレデンシャル管理が散らかっていればVaults相当の仕組みを検討する流れになるかもしれません。優先順位は、自分の現場の現状から逆算するのが一番自然だと感じています。

ただし、こういう棚卸しが「やる側」の自己満足で終わらないようにするには、お客様やチームメンバーと一緒にやることが大事だな、とも思っています。自分の頭の中だけで完結させず、図を一緒に書きながら「この境界はどこに引きたい?」を会話の材料にできれば、棚卸しそのものが組織知になっていきます。

全部を一度に変える必要はないと思っているので、まずはこの「3層の図示」から手をつけてみるつもりです。

実は、この記事を書きながら、自分でも実際にホワイトボードに書き出してみました。

![エージェント実装を「3層」で棚卸しするホワイトボード。agents「脳」/ environments「手」/ sessions「履歴」の3層構造、その下にAgent loopがserver-sideで走る図、右側に「この図で見えてくること」「棚卸しで出てくる問い」のメモ](https://static.zenn.studio/user-upload/deployed-images/9430569beb78ad2fa60e719f.jpg?sha=d61b598a0602187b074e9a4021a8a2d67f502ecb)

書きながら気づいたのは、私の現場のエージェント実装は **sessions層の観測が一番薄い** ということでした。agents と environments は普段から触っているので脳内モデルがありましたが、sessions を「イベントログ＋状態遷移＋再開ポリシー」として独立した層に置く視点は、図示してはじめて手触りを持って入ってきました。

1 時間あれば、たぶん誰でも自分の現場に対応する図が書けます。読み手の中に同じ図を立ち上げる手前で、**自分自身の図** がまず必要だった ── そう感じています。

## おわりに

セッションの最後、Isabellaさんは「What we did today」というRecapスライドで、参加者と一緒に振り返りをしていました。 **(1) メンタルモデルを獲得した(2) 動くエージェントを出荷した(3) 次にどこへ行けばいいか分かった** という3点で締める構成です。

![「What we did today: Recap」のスライド。3つのチェックボックス「Got the mental model」「Shipped a working agent」「Know where to go next」が並ぶ](https://static.zenn.studio/user-upload/deployed-images/6d281900e69209c06d3b38fa.png?sha=7aa314945f74517cef12d969d197beaee4fe2471)

私もこのRecapに重ねて、自分の中で大きく整理できたことを言葉にしておくと、「これまで自前で抱えていた『エージェント運用基盤』というレイヤを、本気で外出しに来たサービスが現れた」という事実でした。

技術的に新しい概念を山ほど覚えなければならない、という話ではありません。むしろ、 **既存の分散システムの知識(イベントソーシング、コントロールプレーン/データプレーン分離、ゼロトラスト、SREプラクティス)が、エージェント領域にほぼそのまま翻訳できる** ということを、改めて確認できたセッションでした。これはインフラエンジニアやAI Solutions Architectとして、わりと心強い知らせだと感じています。

そして、もう一段抽象度を上げて自分の働き方に当てはめてみると、こんなことを考えていました。

Isabellaさんの "Harnesses should evolve alongside your agents." という一文は、技術的な話としてはハーネス層の話を指しています。ただ、これを自分の頭の中に置き換えてみると、**「自分の頭の中の整理(=自分のハーネス)も、エージェントの進化に合わせて更新し続けなければならない」** という話としても受け取れる気がしました。

PoCのときに作った「LLMの使い方」の頭の整理を、半年も使い続けると、現実とズレてきます。モデルが変わり、ハーネスが変わり、お客様の期待値が変わる。であれば、自分の頭の中の整理も、定期的に「アップデート対象」として扱う必要がある、ということです。

カンファレンスのセッションを夜の時間に少しずつ消化するのは、まさにそういう「頭の中のハーネス更新作業」だな、と思います。一本一本のセッションを丁寧に辿る経験は、次のセッションへの解像度を確実に上げてくれます。今回のIsabellaさんのセッションも、自分の中の「エージェント本番運用」という引き出しを、少し整理し直してくれた1本でした。

最後にもう一度、私が一番刺さった一文を引用しておきます。

> "Harnesses should evolve alongside your agents."  
> — Isabella He

これを「エージェントの足回りの話」としてだけ受け取るのではなく、「自分自身の頭の中の枠組みも、扱う対象と一緒に進化させ続ける」というメッセージとして、私は持ち帰ろうと思います。

このBlogが、同じように「エージェントの本番運用ってどう設計したらいいんだろう」と日々考えている誰かの、頭の整理の小さな助けになれば嬉しいです。
