## はじめに

Claude Code のサブエージェント機能を使うと、メインの Claude を「COO」に見立てて、その下に専門部署（サブエージェント）を並べた **「AI組織」** が作れます。X やブログでも「1人会社をAI社員で組織化する」という話をよく見かけるようになりました。

私もこれをやりました。CEO（自分）─ COO（メインClaude）の下に **16部署** を設計して、それぞれにエージェント定義ファイルを書いて、組織図まで CLAUDE.md に載せる本格仕様です。

結論から書くと、**数ヶ月運用して実際に稼働していたのは2部署だけ**でした。先日、残りの14部署を解体しました。

この記事は、その失敗の記録と「では何を残したのか」の話です。これから Claude Code で AI 組織を作ろうとしている人が、同じ回り道をしなくて済むように書きます。

## 何を作ったか：組織図から始めた16部署

やったことはシンプルで、「会社にありそうな部署」を先に考えて、それぞれをサブエージェントとして定義しました。

* CLAUDE.md に組織図（経営層・本部・部署の表）を記載
* `.claude/agents/` 配下に部署ごとの定義ファイル（役割・責務・報告フォーマット）を作成
* COO（メインClaude）が案件に応じて各部署へ委譲するルールを整備

定義ファイルを書いている間はとても楽しいです。「うちの会社にはコンテンツ部門もマーケ部門も監査部門もある」という全能感があります。**そして、これが罠でした。**

## 数ヶ月後の棚卸し：稼働2部署、幽霊14部署

あるとき、各部署の稼働実績をセッションログから棚卸ししました。結果がこれです。

| 分類 | 部署数 | 実態 |
| --- | --- | --- |
| 実稼働 | 2 | 記事の全件監査役と、出典裏取りのソースハンター |
| 未稼働（幽霊部署） | 14 | 一度も呼ばれない、または数回呼んで使わなくなった |

では、幽霊部署に割り当てたはずの仕事はどこへ行ったのか。消えたわけではなく、**3つの場所に吸収されていました。**

1. **スキル（手順書）**：定型ワークフローは、エージェントではなく手順を書いたスキルファイルの方が確実に再現できる
2. **自動化**：デプロイや定期チェックは CI/CD や cron に落ちた。そもそも LLM の仕事ではなかった
3. **COO直営**：メインの Claude が文脈を全部持ったまま直接やる方が、委譲のオーバーヘッドより速くて正確

つまり16部署のうち14は、**「サブエージェントに分離する構造的メリット」が最初から無かった**わけです。

## なぜ組織図先行は失敗するのか

解体してから理由を整理すると、3つに集約されました。

### 1. やったことのない業務は評価できない

一番大きいのはこれです。自分で一度もやったことのない業務をエージェントに任せると、返ってきた成果物が **「80点に見える40点」** でも見抜けません。合格基準を自分の言葉で書けない業務は、レビューもできない。レビューできない委譲は、品質の劣化を検出できないままアウトプットが積み上がります。

### 2. ボトルネックが見える前に分割した

サブエージェント運用の鉄則は「ボトルネックが見えてから分割する」です。組織図先行はこれの真逆で、ボトルネックがまだ存在しない業務に先に器だけ作ります。器があると「せっかく作ったから使おう」という逆転が起きて、委譲する必要のない仕事まで委譲し、遅くなります。

### 3. 組織を作ること自体が目的化する

定義ファイルを書いてる時間は、事業のアウトプットを1ミリも進めていません。組織図が立派になるほど満足感は上がりますが、成果物は増えない。「AI組織を作ること」がゴールにすり替わっていました。

## 解体後：残したのは「委譲の型」3つだけ

14部署を解体して、部署を「業務の種類」ではなく **「委譲に構造的メリットがある型」** で持ち直しました。

| 型 | 部署 | なぜ分離する価値があるか |
| --- | --- | --- |
| ① 大量読み込み | 調査部（記事監査・ソース裏取り） | 数百記事の全件スキャンはメインのコンテキストを食い潰す。分離必須 |
| ② 独立検証 | 監査部（新設） | 作った本人は自分の盲点を検証できない。独立した視点が構造的に必要 |
| ③ 並列バッチ | 汎用エージェントを都度並列 | 5件超の同型作業は並列化の意味がある。専任部署は不要 |

この型に当てはまらない仕事は、すべて COO 直営です。LP 制作でもセキュリティ対応でも、案件が来たらまず直営でこなし、**委譲の型が見えた時点で初めて部署化を検討**します。組織は先に作らない。

### 監査部だけは「失敗から」新設した

型②の監査部は、解体と同時に唯一新設した部署です。きっかけは実際の事故でした。

記事の FAQ 構造化データに「質問文が数字で始まると JSON-LD の質問名から先頭が欠落する」というバグがあり、**自分で書式チェックを通したにもかかわらず、公開後まで見逃しました**。表示は正常なので、作った本人の視点では発見できない種類のバグです。

「COO 直営は速いが、自己盲点だけは構造的に塞げない」——これが数ヶ月運用して見えた唯一のボトルネックで、だからこそ独立検証の部署には分離する価値がありました。**ボトルネックが先、部署が後**。順序が逆だった16部署との違いはここだけです。

## これから作る人へ：正しい順序

遠回りして分かった順序は、結局これだけです。

1. まず全役割を自分で経験する
2. 繰り返した業務の手順と合格基準を Markdown に書く
3. 最も「定型 × 評価可能 × 可逆」な1業務だけ委譲する
4. 結果をレビューして基準を更新する
5. ボトルネックが見えた業務から分割・追加する

判断基準は一つで、**「委譲する業務の合格基準を、自分の言葉で書けるか」**。書けないなら、まだ委譲のタイミングではありません。

この順序の詳しい方法論（委譲判断のマトリクスや、Microsoft の調査でAI活用の成果は個人要因32%より組織要因67%に依存するという話）は、別記事にまとめています。

## おわりに

16部署の組織図は、いま見ると「AI で会社を作る」という夢のスクリーンショットとしては良くできています。ただ、事業を前に進めたのは、地味に生き残った2部署と、手順書と、自分の手でした。
