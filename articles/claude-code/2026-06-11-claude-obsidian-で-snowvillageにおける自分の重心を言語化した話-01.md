# コミュニティ活動のなかでやりたいこと

じゃあ一体何をやりたいの？という点について。シンプルに言語化してみると、

* 日本のコミュニティの熱気が世界にも届いている
* 価値あるコンテンツが日本⇔世界で双方向にとどいている
* 頑張ってる個人やグループ（●●をやってる●●さん）が世界に知られている

のような環境をSnowVillageブランドのもとに構築できないかなと考えています。

### 日本のITコミュニティの強さは世界のモデルになり得る

今はAIの時代でIT環境の進化がはやく、世界中が横並びで実用に向けて走っている状態です。この現状において、技術面で日本が後追いになっているということは決してありません。（企業規模の違いでデカいことをやってるとかの違いはある。）

社内など特定の環境での取り組みは、会社、人や組織、成長ステージなどによる要素が大きく、個人の経験談にしかなりません。世界が横並びで前例がない今において、それが集まって世間の知識や経験とつながれるのがコミュニティの不変だけど改めて注目すべき価値になっています。  
東京一極集中（日本全国だとしても時差なく交流できる）で、世界から見たら非常に小さいエリアでアクティブな活動ができているSnowVillageは、今だからこその価値を発揮できているコミュニティの一つなんじゃないかと思います。  
AIなんて、皆はじめてだった所からたった数年ですが、SnowVillageのおかげで海外勢と言語の壁があってすら議論についていけるだけのことを学べているし、経験や視点を発信できるほどに身につけられている実感があります。

そんな恵まれた環境にある日本の取り組みを世界に伝えて、コミュニティのモデルを示すことが一つの責任なんじゃないか。そんなことを思ったのが昨年末に海外のData Superheroes勢と交流したときでした。  
もうちょっと日本勢にとって実利のあるところでは、普段から世界に届くコンテンツとして露出して、ブランドが育っていることで、年末のData Superheroes選定にあたっても日本勢のプレゼンスをもっと高めていけるんじゃないかと考えています。

###### AWARD応募のための作文をしたい（させたい！）

膨大な説明を書かないといけなくて、一度は「よし、自分で作文しよ」ってなり、30分ぐらい筆が止まって、Claudeに作文させるために前提知識をどうやって教え込むかを再び考えたのでした。

##### AWARD作文を機にメモ置き場を「資産」として持ち直したくなった

そもそも私は自分用のメモを書き残す習慣がありませんでした。  
ずっと <https://stackedit.io/> を会社の Google ドライブに繋いで、案件など最低限のメモ置き場にしていました。ブラウザで開けて、マークダウンが書けて、クラウドに同期される。それで十分だと思っていました。粒度ぐちゃぐちゃ、再利用性は低いので積みあがったファイルはほとんど案件終了後は見ることもなく・・・  
Claudeを日常的に使うようになって、見え方が変わりました。  
Slackやブラウザの音声入力の精度が上がったおかげで、ここ半年ぐらいは犬の散歩をしながら仕事やコミュニティのあらゆる考え事をメモに残し、PC内のフォルダに置いていました。Claudeが見やすいような文章をファイル名につけ、あとは1階層に全部置いてました。  
「このメモ、Claude から読み書きできたらどれだけ便利だろう」「そもそもこの知識、自分の資産として積み上がってないよな？」と考え始めると、stackedit + Google ドライブという構成は限界が来ていました。

メモは書いて終わりではなくて、後から AI に読ませて整理させたり、何かを書くときの素材として引っ張り出したりしてこそ価値が出ます。

* AI（Claude）から素直に読み書きできる自分のローカルにもつこと
* 知識同士のつながりを表現できてあとから利用できる「資産」として持てること
* ノウハウが出回っていて容易に構築できること

この3つが揃った場所にメモを移したい。そう思って乗り換え先を探し、たどり着いたのが [Obsidian](https://obsidian.md/) でした。

##### Obsidian ― ローカルにプライベートなナレッジグラフを持てる

Obsidianでは、ノート同士を `[[wikilink]]` で繋いでいくと、知識が自然と**グラフ**になっていきます。  
「この話は何の一部か」「具体例はどれか」を意識してリンクを張っていくと、点だったメモが、いつのまにか地図のように繋がっていく。Zettelkasten というらしいですが、1ノート＝1トピックで小さく書いて、リンクで構造を表現する書き方とも相性が良いです。

ローカルに、プライベートに、自分だけのナレッジグラフを持てることが今の課題感にフィットしました。コミュニティ活動みたいに「断片的な出来事や判断や複数の立場がたくさん積み上がっていくけど、全体像が頭の中だけにある」ものを扱うとき、外に出してグラフにできるだけで、頭の中の解像度がぜんぜん変わります。

![SnowVillageが好きすぎていつも中心にいる](https://static.zenn.studio/user-upload/27da5039d851-20260611.png)

### プライベートナレッジグラフの実装

Claude × Obsidian の話はインターネッツにたくさん出回っているので、 「運用規約に従って整理してくれる」仕組みはClaudeにお任せで以下のようにつくりました。  
作ったといってもClaude Codeにそうお願いしただけですが。

具体的な構成はこんな感じです。

###### Vault を4大テーマに大別して、各テーマ内は Zettelkasten 寄りに

まず Vault（Obsidian の保管庫）のトップを、3つの大テーマに分けました。

* **Community** … コミュニティ活動（SnowVillage / DSH / JPUG など）。今のところ大半がこれ
* **Life** … 完全な私生活
* **Exture** … 所属企業として、明確に企業案件と分かるもの

そして各テーマの中は Zettelkasten 寄りに、

* `notes/` … 確定した atomic note（1ノート＝1トピック）をフラットに溜める
* `index/` … MOC（Map of Content。テーマの目次・ハブとなる地図ノート）

という構成にして、**フォルダを深く切らずに、構造は wikilink とタグで表現する**方針にしました。フォルダ階層で分類しようとすると「これはどっちのフォルダ？」で手が止まるけど、タグとリンクなら多重に紐づけられて、後から地図（MOC）から辿れます。（探す人もAIなんですけどね。省力で辿れます。）

![フォルダ階層はこれだけ](https://static.zenn.studio/user-upload/8702879db6a4-20260611.png)  
notesの下に細切れテキストがたくさん

###### `temp/` という放り込み入口を作る

整理の起点として、`temp/` というグローバルな inbox を置きました。  
雑メモも、会話やしゃべったことの書き起こしも、とりあえず全部ここに放り込みます。あとで整理するときに、推敲 → テーマ振り分け → タグ付け → リンク付け → 各テーマの `notes/` へ昇格、という流れで確定ノートに育てていきます。（Claudeがやってくれる）

「とりあえず置く場所」と「確定した場所」を分けるだけで、書くときの心理的ハードルがぐっと下がります。  
![アイデア入力は音声で](https://static.zenn.studio/user-upload/27132d9eccd9-20260611.png)  
アイデア入力は音声でとにかく思う限りのことをしゃべりきる

###### 2つのエージェントに役割を分ける（整理する係／出力する係）

仕組みの肝はここになると思います。思い付きで作ってみたので長期的に上手くいくかは未知数ですが、Claude Code のエージェントを**役割で2つに分けました**。

* **`knowledge-organizer`（整理する係）** … temp の昇格、会話からのノート化、推敲、大きいノートの切り出し（atomization）、重複の統合、MOC の整備、用語集の保守。つまり**知識グラフ側の正本を育てる・整える**仕事。
* **`knowledge-output`（出力する係）** … 既存の知識ノートを**素材として**、応募文・登壇原稿・ブログ記事・自己紹介・活動まとめなどを組み上げて出力する仕事。**知識グラフは読むだけで、育てない**。

「知識を蓄える」と「知識を使って何かを書く」は、似ているようで全然性質が違う作業だからです。整理係に「ついでに応募文も書いといて」と頼むと、知識グラフを書き換えながら成果物も作る、みたいに責務が混ざって、何が正本なのか分からなくなります。逆に、出力係に知識グラフを勝手に育てさせると、文章を作るたびにグラフが汚れていく。

なので、**正本を育てるのは整理係だけ、文章を出力するのは出力係だけ**、と決め打ちしました。ちなみに今このブログを書いているのは出力係（`knowledge-output`）の方です。

###### タグは「登録制」にする

タグ作成をClaudeに気ままにやらせると `branding` と `ブランディング` と `brand` が乱立して、すぐ機能しなくなります。  
そこで、**「タグ登録簿」という正本の表を一箇所に持って、そこに載っているタグしか使わない**という登録制にしました。新しいタグを使いたくなったら、まず登録簿に追記してから使います。（そういうルールをclaude.mdに書いておく）。  
整理係（organizer）はこのルールでタグの語彙を育てるけれど、出力係（output）は**既存のタグしか使わない**（成果物を作るたびに語彙が散らからないように）。

地味なルールですが、これがあるおかげでタグがちゃんと「探すための軸」として機能し続けています。

###### リンクは「一方通行」にする

もう一つ大事にしたのが、**一方通行リンクの原則**です。

* ✅ 成果物（Output）から、根拠にした知識ノートへリンクを張るのは OK
* ❌ 知識ノート側から、成果物へ逆リンクを張るのは NG

理由はシンプルで、成果物（このブログ草稿みたいなもの）は**後から自分の手で気軽に消したり書き換えたりする前提**だからです。もし知識グラフ側が成果物に依存してリンクを張っていたら、成果物を消した瞬間にリンク切れが起きて、肝心の知識グラフ（＝正本）が壊れる。

「成果物が消えても知識グラフは無傷」を保証するために、参照は必ず素材の方向（成果物 → 知識）にだけ流します。

![タグやリンクの運用は完全claude任せ](https://static.zenn.studio/user-upload/67b1ea6c1e9c-20260611.png)  
タグやリンクが適切に運用されてこそのObsidianのナレッジグラフなんだけど、人力の運用はまず不可能ですね。自分のパーソナルな2時間程度で作ったものでもう全く管理しきれません。ここはルールを明確にしてClaudeにやってもらいましょう。

###### CLAUDE.md を「運用規約＝エージェントへの指示の正本」にする

ここまでのルール ― テーマ分け、temp の運用、エージェントの役割分担、タグ登録制、一方通行リンク ― を全部、`CLAUDE.md` という1枚の運用規約ファイルに書いて、Vault のルート（正確には作業用のワークスペース）に置いています。

Claude Code は作業を始めるとき、まずこの `CLAUDE.md` を読みます。  
そして「これは知識を整理するタスクか、それとも知識を使って成果物を出すタスクか」を見極めて、対応するエージェント（organizer か output か）にルーティングする。詳細フローは各エージェントの定義ファイルが正本で、`CLAUDE.md` は両者の共通土台＋振り分けルールという位置づけです。

# SnowVillage の現在地と自分の重心

この文章はリアルにObsidianのナレッジグラフを参照させてClaudeに作文させたそのままです。  
この仕組みを作って何が良かったかというと、**「自分が今コミュニティで何に重心を置いているのか」をちゃんと言語化・棚卸しできた** ことでした。SnowVillageの活動では様々な役割・立場を兼任しながら、複数の活動が同時進行しています。  
この活動はこっちの立場でこんな思惑で作ってて～なんて、MayorsでありDSHであり大きな数か月スパンのことを会話しているのに、明日のイチ運営タスクの雑用のほうが脳内を占めて焦っていたりするわけで。  
それを音声入力で2時間ぐらいかけてひたすら temp に放り込んで、整理係にノートに育てさせて、グラフで眺めると、自分でも「ああ、今こういうステージなんだな」と見えてきました。

### 今のステージ ― 創設者の退任から、ユーザー主導の Mayors 体制へ

SnowVillage は元々、Snowflake 社員でもあったKTを中心に集まった方々で始まり、KTが強力に引っ張ってきました。 2024 年に退任したとき、正直「Snowflake の元気なコミュニティ、この先は大丈夫か？」と心配されるくらいのインパクトがありました。

そこで私たちが選んだのは、特定の誰かのカリスマに依存し続けるのではなく、**ユーザー自身でセルフドライブしていく体制**へ移ることでした。それが Mayors 制であり、1年の運用を経てもう少し役割を明文化した部門制を開始しました。村（Village）の長＝ Mayors という位置づけで、

* Mayors イベント部門（体験を創り出す）
* Mayors ブランド部門（発信と場作り）
* グロース部門（あらゆるレイヤーでの成長）
* オペレーション部門

の4部門で運営しています。

### 正直な課題感

きれいごとだけ言うつもりはなくて、課題感もちゃんとあります。  
カリスマ的な前任者がいなくなった後、「参加者主導でなんでもやれる立場」にはなったものの、じゃあどう動くんだっけ、というのが描けずに、しばらく停滞した時期もありました。運営メンバーも、全員のモチベーションが一様に高いわけではなく、現実には時間を取れる少数のメンバーが活動を推進している状態です。

イベントの数やコンテンツは維持されていても、一時期は盛り上がり・結束が目に見えて落ちていた。発信者が偏り、新しいタレントが続いてこない。「1年後このコミュニティどうなってるんだろう」を誰も描けない時期があった。これは内側にいるからこそ感じ取れた、リアルな課題感です。

### その中で、私が注力していること

こういうステージで、私は Mayors イベント部門のリーダー兼、新設の Mayors ブランド部門を最も精力的に動かす一人として、いくつかのことに重心を置いています。

> ※ブログ化の際にお気持ち補足：  
> 課題感とか、そのために私個人がどんな活動をするかは別にMayorsの中で共通認識があるわけでもなく、各々が思うように善き行いをしようよ、ってうぐらいの合意のもので注力しているって感じです。

**1つめは、熱量と結束をもう一度作ること。**  
2025 年の Snowflake World Tour Tokyo では、コミュニティとして初めてブースを出す企画をリードしました。これは極端に言うと「落ちかけた結束をもう一度作る」ことを強く意図した活動でした。各ユーザーグループの中は運営しているんだけど他のグループの活動への関心が薄いとか、大きい企画を誰がリードするか不透明感があり、それを何としても打ち破りたかった。  
結果としてコミュニティに活気と結束は最高潮に持って行けたと思うし、新しいボランティアやタレントの発掘にもつながり、「これからもちゃんとセルフドライブしていく新生 SnowVillage」を皆に印象づけられたと思っています。

コミュニティはその後も成長を続けていて、SnowVillageのSlack参加者数はその後の1年で 1800人 → 2400人に伸びました。

**2つめは、ブランディング ― 良質なコンテンツに「コミュニティのお墨付き」を与えること。**  
情報発信の文化が強いコミュニティなので、ブログなどのコンテンツはたくさん生まれます。でもその多くは新機能やってみたのような一過性のものだったり、企業の色がついていたり。

> ※ブログ化の際にお気持ち補足：  
> 鮮度が命の情報はもちろん重要だし、企業としての取り組みがみんなに有益なものなら全然いいと思ってます。継続的にそれができる人は既にグローバルからみても稀有な発信者だし、たまたま単発で一過性の情報を出してもその人に長期的には焦点が当たらないし、企業の宣伝ルートとしてはイヤラシくない範囲で上手くやってよ、なので、ここでいうブランドで魅力を後押しする対象ではないかなって感じです。別の形として、企業としての動きだけどコミュニティのためのコンテンツを充実させていくみたいなお声がけをいただくことも最近あり、そういう企業名を出したご支援を受けるのも考えなきゃなとかは思ってます。

頑張っている人やそのコンテンツにもっと光があたる活動としてコミュニティとして取り上げたり、コミュニティそのものがブランド価値として育ち、そこで並ぶコンテンツをずっと支援する仕組みが、これまでありませんでした。

ブランド部門として、世に出すべきコンテンツをちゃんと選んで出し、ブランド価値を高めることが新しい発信の動機になり、その仕組みを介してコンテンツに箔がつき、さらに広まっていく ― そういう循環を作ろうとしています。  
ブランドが掲載可否を判定する査問機関になるという意図は全然なく、息の長い有益コンテンツを目立たせることでこれからも人が集まり、その先で鮮度優先のコンテンツももっと見てもらえる。または息の長い有益コンテンツを俺たちも作ろう！というモチベにはなって欲しいと思っています。

**3つめは、グローバル発信。**  
東京は IT 人口が一極集中している背景もあって、毎月のように百人規模のイベントを開けている。これは世界の Data Superhero のローカルコミュニティを見ても、正直かなり稀なことです。年1回・数ヶ月に1回が普通の中で、日本は高頻度・高集客で、最新機能が母国語で流通している。

それでも、言語の壁で日本の良質なコンテンツは世界に届きにくいし、世界の良質なコンテンツも日本に入ってきにくい。だから SnowVillage の活動を世界へ届け、こんな活発で有益なコミュニティが日本にはあるんだぜ！をグローバルから認知させ、世界のコンテンツも日本に紹介する口を作ることに取り組んでいます。SnowVillageサイトの英語版を整備したり、世界の Data Superhero が出した新機能まとめを日本向けにホスティング・翻訳して公開したり。

「ブログ化に際しお気持ち補足」というところだけ書き足しました。

Snowflake には、技術力・コミュニティ貢献・影響力をグローバルに称える仕組みがあります。  
私自身も Snowflake Data Superheroesとしての3年目を迎え、昨年末には世界の Data Superhero が集まって製品チームと議論する場（DSH Council）にも参加してきました。そういうグローバルな評価軸・コミュニティアワードの世界が、ちゃんと存在します。

そして去年は、日本の地方やテーマ別コミュニティで頑張ってくれている人を Data Superhero に推薦して、ちゃんと認定してもらえています。海外からの評価を日本のメンバーにちゃんと向けられた、嬉しさと責任感などを身に染みて感じ、これを継続的な活動にしたいと思い始めたのでした。

世界の基準で見てもこれだけ高頻度・高集客で、母国語で最新の知見が流通していて、セルフドライブして成長し続けている ― こんなコミュニティ、そうそうないんです。これを日本の中だけで完結させるのは、もったいなさすぎる。

なので、グローバルの評価軸に日本コミュニティのブランドそのものを乗せにいきます。そして、 **頑張っているみんなの推薦も書きまくりたいです！！！** 海外からの評価を、もっともっと日本のみんなに還元したい。

**日本のコミュニティの熱量、世界に見せてやろうぜ！！！**
