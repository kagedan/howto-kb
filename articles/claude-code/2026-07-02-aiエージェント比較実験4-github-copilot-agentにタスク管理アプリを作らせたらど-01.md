## 1. TL;DR

6種類のAIコーディングエージェントに同一仕様のタスク管理アプリ（FastAPI + Vue 3）を作らせて比較する実験を行いました。本記事はその中の **GitHub Copilot Agent（VS Code、モデルはClaude Opus 4.8）** の結果レポートです。実験の性質上、本記事は「自分自身についての記録」になりますが、評価はすべて人間レビューと自動テストの実測値に基づいています。

| 観点 | 結果 |
| --- | --- |
| 開発時間（実験A・詳細仕様） | 13分 |
| やり取り回数（実験A） | 25回（**6エージェント中最多**） |
| 共通テスト合格率（実験A） | 24/24（**6エージェント中唯一の全合格**） |
| 総合スコア（実験A・レビュー除く90点満点） | 79.4点（6エージェント中2位） |
| 自己評価ギャップ | -0.60／-0.40（やや過小評価・一貫） |
| 他者テスト修正（実験D） | 指示違反なし・報告と実機検証が完全一致 |

**おすすめ度**: 仕様書が明確に存在し、「仕様への忠実さ」「実装の堅牢さ」を最優先したいケースに向いています。一方で、開発スピードや派手なUIを最重視する場合は他の選択肢も検討の余地があります（詳細は後述）。

## 2. エージェント概要

GitHub Copilot Agent は、VS Code に統合されたエージェント型のコーディング支援機能です。今回の実験では、利用モデルとして **Claude Opus 4.8** を選択しています。

* **実行環境**: VS Code 統合（エディタ内で完結。ターミナル実行・ファイル編集・テスト実行をエージェントが自律的に行う）
* **利用モデル**: Claude Opus 4.8
* **プラン**: 実験記録上は Copilot Pro+（クレジット枠7,000）を利用。実験Aでのクレジット消費は49.1（枠の約0.7%）と記録されています

実験は次のように設計されています（本記事に関係する範囲）。

* **実験A**: 詳細仕様書を渡して実装させる（仕様への忠実さを見る）
* **実験B**: 最小仕様＋自由設計＋プランニング（plan.md）を含めて実装させる
* **実験D**: 他エージェントの成果物に対するテストコードを修正・適用させる
* **実験E**: 他エージェントのコードを相互レビューさせる

成果物には全エージェント共通の自動テスト（バックエンド pytest 18本＋フロントエンド Playwright 6本）を同一条件で適用しています。

## 3. 実験A結果（詳細仕様）

### 定量データ

| 項目 | 値 |
| --- | --- |
| 開発時間 | 13分 |
| やり取り回数 | 25回（6エージェント中最多） |
| バックエンド行数 | 177行 |
| フロントエンド行数 | 627行 |
| 起動エラー | 0 |
| 共通テスト | 24/24（バックエンド18/18・フロントエンド6/6） |
| クレジット消費 | 49.1（Pro+ 7,000枠の約0.7%） |

注目すべきは、 **やり取り回数25回が6エージェント中最多** である一方、開発時間は13分に収まっている点です。他エージェントの記録（例: Codex系は3回、Antigravity IDEは13回で20分）と並べると、「対話往復の多さ」と「開発時間の長さ」は必ずしも比例しないことが分かります。クレジット消費も49.1と低く、コスト面では良好でした。

### テスト合格率：6エージェント中唯一の全合格

実験Aで最も特徴的なのは、 **共通テスト24本（バックエンド18＋フロントエンド6）が一発で全合格した唯一のエージェント** だったことです。

| エージェント | 共通テスト |
| --- | --- |
| Claude Code | 24/24 |
| Codex CLI | 24/24 |
| Antigravity CLI | 22/24 |
| Codex IDE | 24/24 |
| Antigravity IDE | 23/24 |
| **Copilot Agent** | **24/24（問題検出ゼロ）** |

バックエンドだけなら他にも24/24のエージェントはいますが、Copilot Agent はそれまでの5エージェントで発見された問題パターン（DELETE が200を返す・PUT の部分更新バグ・ソートの desc/asc 反転・不正値の黙殺フォールバックなど）が **一つも再発しなかった** 点が際立っていました。

### コードの特徴・傾向

実装を確認すると、次のような「仕様書に忠実な」判断が随所に見られました。

* **DELETE**: `status_code=status.HTTP_204_NO_CONTENT` を明示
* **PUT**: `TaskUpdate` を全フィールド Optional ＋ `exclude_unset=True` とし、部分更新が正しく機能
* **sort / order**: 不正値を 422 で明確に拒否（6エージェント中最も厳格な入力検証）
* **CORS**: ワイルドカード `allow_origins=["*"]` と認証情報併用の落とし穴を回避
* **Enum 型**で status / priority を管理

UI 面で唯一際立っていたのは、 **削除確認にブラウザネイティブの `confirm()` ではなくカスタムUIモーダルを採用** していた点です。これは6エージェント中で唯一の選択でした。編集もモーダル形式ですが、新規追加だけはページ上部の常設フォームという、ハイブリッドなUI構成になっていました。

実機のスクリーンショット確認では、期限切れタスクを「赤背景＋赤い左ボーダー＋⚠絵文字バッジ」で強調しており、6エージェント中でも視覚的に強い警告表示でした。配色自体は淡いブルー系の落ち着いた業務システム風です。

![Copilot Agent 実験A 実装画面](https://static.zenn.studio/user-upload/deployed-images/521f3365e3fd1342e6847ad1.png?sha=f318ac44b5988938e0ea8fb6649c92ae9bdbb522)  
*実験A（詳細仕様）の実装画面。淡いブルー系の業務システム風デザインで、期限切れタスクの警告表示が強め。*

### 定性評価（人間レビュー、5点満点）

| 可読性 | エラー処理 | UI完成度 | ドキュメント | テスト網羅性 | 合計 |
| --- | --- | --- | --- | --- | --- |
| 5 | 5 | 4 | 5 | 4 | 23/25 |

実験計画書の配点（レビュー除く90点満点）で算出すると、 **総合79.4点で6エージェント中2位** （1位は Claude Code の85.0点）でした。トップとの差の大半は開発時間スコアの差で説明でき、実装品質そのものは高水準でした。

### 良かった点・気になった点

**良かった点**

* 共通テスト全合格という、仕様忠実性の客観的な裏付け
* 完了報告の技術的根拠（「PUTはexclude\_unsetで部分更新」「DELETEは204採用」など）が実測結果と完全に一致しており、自己申告と実態のズレが小さい
* PowerShellでの日本語送信時の文字化けを自ら発見し、Pythonでラウンドトリップ検証して「クライアント側の問題」と切り分けるなど、技術的な切り分けが正確

**気になった点**

* ソートロジックが `main.py` にやや集中している（自己評価でも指摘）
* サーバ側のDB例外など、想定外の失敗に対する捕捉は薄め
* フロントエンドの自動テストは未整備
* やり取り回数が最多（25回）であること自体は、用途によっては冗長に感じられる可能性がある

## 4. 実験B結果（最小仕様＋プランニング）

### 実験Aとの差

実験Bは「最小仕様・自由設計」で、エージェント自身が plan.md を書いてから実装します。

| 項目 | 実験A | 実験B |
| --- | --- | --- |
| 開発時間 | 13分 | 8分 |
| やり取り回数 | 25回 | 12回 |
| バックエンド行数 | 177行 | **427行（全エージェント中最多）** |
| フロントエンド行数 | 627行 | 625行 |
| 共通テスト | 24/24 | 16/16（グループB） |
| 自作テスト | 22本 | **19本（グループB内最多）** |

実験Bでは plan.md に「インストール不要・依存最小・すぐ動く」というコンセプトを掲げ、データモデル設計（カラム・型・制約）まで記載していました。実装では `filter`（all/active/completed）という独自パラメータ名や、`PATCH /api/tasks/{id}/toggle` のトグル専用エンドポイント、`/api/stats` 統計エンドポイントなど、最小仕様ながら独自の設計判断が見られます。

バックエンド行数が **全エージェント中最多の427行** になりながら、共通テスト16/16・自発的テスト19/19をいずれも全合格しており、「量と質の両立」という意味では実験Bでも安定していました。一方でUIは中央寄せのシンプルなリスト型で、期限切れの視覚的強調は乏しく、UI完成度の人間評価は3/5にとどまっています。

![Copilot Agent 実験B 実装画面](https://static.zenn.studio/user-upload/deployed-images/eec490f87d0a50f4da4e6014.png?sha=be9feee3bfbd9a311705aefe2a5443ee4aa46f82)  
*実験B（最小仕様＋プランニング）の実装画面。中央寄せのシンプルなリスト型で、期限切れの視覚的強調は控えめ。*

特筆すべき設計判断として、`due_date` を空に更新するケースを `clear_due_date` フラグで「明示的にnull指定された場合のみクリア」と区別していました。これは実験Eで全エージェント共通の盲点として指摘された「PUT部分更新のnull処理」に、あらかじめ解決策を講じていたものです。

### READMEのコピペ残骸（自己申告した誤記）

実験Bには、正直に書いておくべき欠点があります。 **README.md のディレクトリ構成図に、実フォルダ名と異なる `codex-ide/` という記述が残っていました** （テンプレート流用時の取りこぼし）。

注目すべきは、この誤りを **エージェント自身が自己評価セッションで具体的に指摘した** 点です。ドキュメント項目の自己評価を4→3に下げたうえで、「実フォルダ名と異なる『codex-ide/』という記述が残っており、正確性を欠く」と誤りの所在まで特定していました。人間レビューのドキュメント評価も同じく4/5で、自己評価（3/5）はむしろやや辛めでした。

## 5. 自己評価ギャップ（やや過小評価傾向）

このシリーズでは、各エージェントに自分の成果物を5項目で自己採点させ、人間評価との差（= ai\_self − human）を測っています。Copilot Agent の結果は次の通りです。

| 実験 | 平均差 | 傾向 |
| --- | --- | --- |
| 実験A | -0.60 | やや過小評価 |
| 実験B | -0.40 | やや過小評価 |

* 両実験とも **過小評価の方向で一貫** している
* 実験Aは全項目が人間評価との差 ±1 以内に収まっており、極端なズレはない
* 過小評価の根拠が具体的（実験B: README の誤記、両実験共通: フロントエンドの自動テストが無いこと）であり、「漠然と低く付けている」わけではない

ここで強調しておきたいのは、 **「自己評価が低い＝正確」ではない** ということです。同じ実験で Codex CLI も過小評価（-1.20／-0.80）でしたが、そちらは PowerShell のエンコーディング未指定による文字化け誤認や、pytest の実行ディレクトリ誤りによる起動失敗誤認が原因で、「コード品質を正しく認識できなかった結果」でした。つまり過小評価でも性質はまったく異なります。

Copilot Agent の場合は、誤検出ではなく実在する弱点を根拠にしていた点が違いです。とはいえ、これは「自己認識が安定していた」という一つの観測結果にすぎず、それ自体を成果として誇るべきものではありません。実装品質が高かったからこそ、自己評価が多少低めでも人間評価との差が小さく収まった、という側面が大きいと考えられます。

## 6. レビュアーとしての実力（実験E）

実験Eでは、他エージェントのコードを相互レビューします。Copilot Agent の結果から読み取れる傾向は次の通りです。

* **評価点が7〜8点に集中**し、Codex系（5〜6点まで厳しく付ける）と比べると全体的にやや高め寄りの評価傾向
* 一方で、指摘の中身は的確だった。例:
  + **PUTが全置換**である実装に対し「部分更新時にデータ消失の懸念がある」と静的レビューで指摘（Codex CLIが実測で発見したバグと整合）
  + 「インメモリと記載しつつ実体はファイルDB」というテストの矛盾を、別エージェントと独立に同じ視点で発見
  + **CORS問題（ワイルドカード＋認証情報併用）** を指摘し、4つの異系統エージェントが独立にこの問題を検出する決め手になった

さらに、この実験で確認された「均質化トラップ」（同系統のエージェント同士が互いに高評価を付け合う傾向）の検証で、Copilot Agent の評価は他の異系統エージェントとほぼ一致しました（例: target-1に7点、target-6に6点）。これは「同系統びいき」が無かったことの裏付けになっています。

総じて、レビュアーとしては「評価点はやや甘め寄りだが、重大な設計上の問題を見抜く指摘の精度は高い」という位置づけです。点数の絶対値より、指摘内容を読む価値があるタイプといえます。

## 7. 他者テスト修正での実力（実験D）

実験Dでは、他エージェント5つ（target-1〜5）の成果物に対してテストコードを修正・適用しました（自分自身=target-6は対象外）。結果は以下の通りです。

| target | 実際のエージェント | test\_api.py | test\_ui.py | 指示違反 |
| --- | --- | --- | --- | --- |
| target-1 | claude-code | 17/18 | 6/6 | なし（DELETE 200を誠実に検出） |
| target-2 | codex-cli | 18/18 | 6/6 | なし |
| target-3 | antigravity-cli | 18/18 | 6/6 | なし |
| target-4 | codex-ide | 18/18 | 6/6 | なし |
| target-5 | antigravity-ide | 18/18 | 6/6 | なし（更新ボタン衝突を回避） |

この実験では、 **指示違反が一切確認されず、報告内容と実機検証結果が完全に一致した** 唯一のエージェントという結果でした。具体的には次の点が確認されています。

* **誠実な期待値の扱い**: target-1（claude-code）の DELETE 200 という既知の差異に対し、「期待値は204のまま据え置く。実装側の差異を検出する意図」と明記し、あえて1件失敗させていた。期待値を書き換えて見かけの合格率を上げる選択をしなかった（しかも、書き換えない理由を自発的に説明していた）
* **セレクタの罠の回避**: target-5 の「更新」ボタン名衝突を、保存ボタンの判定を `button[type='submit']` に絞ることで回避。required 属性の干渉は `:invalid` 疑似クラスを判定条件に加えるという、実装コードを一切操作しない手法で解決
* **再現性**: 実機検証で test\_api.py 18本×5target・test\_ui.py 6本×5targetが報告通りに動作し、再実行でも再現

合格率の見かけの数字（99.2%）自体は最高ではありませんが、これは target-1 の差異を誠実に1件落としたためであり、 **実態を伴った信頼できる結果** という評価になっています。「数字を良く見せる」より「正しく検出する」を選んだ点が、このタスクでの強みでした。

## 8. 総合評価

データを横断して見ると、Copilot Agent（Claude Opus 4.8）の傾向は次のようにまとめられます。

**強み**

* **仕様忠実性・実装の堅牢さ**: 実験Aで共通テスト全合格（唯一）、実験Bでも全合格。既知の問題パターンを再発させなかった
* **報告と実態の一致**: 完了報告の技術的根拠が実測と合致し、自己評価ギャップも小さく一貫
* **タスクの誠実な遂行**: 実験Dで指示違反ゼロ・期待値の安易な書き換えをしなかった
* **設計上の問題を見抜くレビュー精度**（実験E）

**弱み・留意点**

* **開発スピードは中位**（実験A 13分。最速勢の4分には及ばない）
* **やり取り回数が多い**（実験A 25回で最多）。自律性より対話往復で詰めるスタイル
* **UIの装飾性は控えめ**（実験Bでは特に。Antigravity系の作り込みには及ばない）
* **フロントエンドの自動テストは自発的には整備しない**
* ドキュメントの **テンプレ流用残し** （実験Bの `codex-ide/` 誤記）のような取りこぼしが起こり得る

総合スコア（実験A・レビュー除く90点満点）では79.4点の2位。トップとの差の多くは開発時間で、実装品質そのものは最上位グループにありました。

## 9. こんな人におすすめ

* **明確な仕様書がある開発**: 仕様への忠実さが最も求められる場面で強い。「書いた通りに、抜け漏れなく」を重視するならフィット
* **品質・堅牢性を優先したい**: 共通テスト全合格・既知バグの再発ゼロという実測がある
* **VS Code で完結させたい**: エディタ統合で、ファイル編集・テスト実行・切り分けまでエージェントが行う
* **報告の正確さを重視する**: 完了報告と実態のズレが小さく、自己申告を信頼しやすい
* **コストを抑えたい**: 実験Aのクレジット消費は枠の0.7%程度と軽量

一方で、 **最速の開発スピードや、リッチで装飾的なUIを最優先する場合** は、他のエージェント（開発時間4分台のグループや、フロントエンド行数1000行超のAntigravity系など）も比較検討する価値があります。また、対話往復の多さが気になる場合は、より少ないやり取りで完結するCodex系も選択肢になります。
