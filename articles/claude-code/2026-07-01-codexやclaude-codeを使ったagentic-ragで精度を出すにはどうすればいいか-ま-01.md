大量のMarkdownファイルを集めたら、それをそのままAIのナレッジベースにできるのでしょうか。

最近のモデルはコンテキスト長が長く、エージェントもファイルを探索できます。そう考えると、チャンク分割や埋め込みをしなくても、プレーンテキストを大量に置いておけば、AIが勝手に読んでドメイン知識を身につけてくれそうに見えます。

しかし、実際に試してみると、結果は「それっぽい出力」止まりでした。

問題は、AIが「知らない知識」にうまく到達できないことです。探索はしてくれますが、すでにモデルが知っている一般知識を補強するような文書ばかり拾いやすく、ナレッジベースに入っている固有の知識を使い切れていませんでした。

この記事では、Markdownで構成したナレッジベースをAIに効率よく探索させるために、どのようなインデックス戦略が効くのかを検証します。

## 前提

### この記事でいうインデックス

先に、この記事で使う「インデックス」という言葉を定義しておきます。

ここでいうインデックスは、データベースの検索を高速化するための一般的なDBインデックスではありません。Markdownファイルの中に、参照すべきファイルの一覧、タイトル、要約、分類、読み方の手がかりを書いたものを指しています。

つまり、AIエージェントに渡すための目次や探索地図です。

例えば、次のような情報をMarkdownにまとめたものを、この記事ではインデックスと呼びます。

* ファイルパス
* 記事タイトル
* 記事要約
* 分類やカテゴリ
* 関連する文書
* その文書をいつ読むべきか

### 問い

今回の問いは、次のようなものです。

大量のMarkdownファイルを集めたとき、AIにその知識を使わせるには、どうすればいいのでしょうか。どうすれば「AIが知らない情報」にAI自身を到達させて回答を生成させられるでしょうか。

最初に考えた選択肢は、大きく分けて二つです。

ファインチューニングは一見スマートです。小さなモデルを専門分野ごとにファインチューニングし、サブエージェントのように使えばよさそうにも見えます。

ただ、専門分野ごとにモデルを分けるには、学習データを分類する必要があります。知識を更新するたびに再学習も必要です。結局、データを整理する手間はインデックスを作る場合と大きく変わらず、さらに学習と運用のコストも乗ってきます。

そこで今回は、再学習ではなくインデックス設計に絞って検証しました。

## 比較したインデックス戦略

比較した手法は、次の四つです。

| 手法 | 内容 |
| --- | --- |
| 手法1 | 記事のタイトルとファイルパスだけを並べる |
| 手法2 | タイトルとファイルパスに加えて、記事の要約を付ける |
| 手法3 | 知識構造に沿って分類し、タイトルと要約を持つ構造化インデックスを作る |
| 手法4 | ファイルやフォルダを分割し、物理的な構造として表現する |

使ったのは、約2000件のWebクリップをマークダウンファイルとしてローカルに保存したものです。このファイル群に対し、上の四つのインデックス戦略を順に試しました。

評価では、AIにYouTube動画の台本や解説記事を書かせました。出力された文章を、面白い記事になっているかという観点で採点しています。（採点にはChatGPT ５.5 Extra Highを使用）

採点項目は次の通りです。

| 項目 | 点数 |
| --- | --- |
| 冒頭の引き | /15 |
| 構成 | /15 |
| 情報価値 | /15 |
| ストーリー性 | /15 |
| 音声台本としての自然さ | /10 |

評価は定性的なものです。そのため、絶対的なベンチマークではありません。ただ、同じテーマに対してインデックス戦略だけを変えたとき、出力の差を見るには十分と判断しました。

## 考察

今回の結果から分かったことは、大量のMarkdownファイルをただ検索対象に置くだけでは、モデルにドメイン知識を十分に使わせることは難しいということです。

プレーンテキストを積み上げただけの状態では、モデルは「知らない知識」に到達するというより、すでに知っている一般知識を補強するような文書を拾いやすくなります。その結果、出力はそれっぽく見えても、ナレッジベース固有の情報が薄くなります。

一方で、インデックスの作り方を変えると、出力品質には差が出ました。

特に良かったのは、手法3の構造化インデックスです。タイトル、ファイルパス、要約をただ並べるのではなく、知識構造に沿って分類し、文書の位置づけを見えるようにすることで、モデルが「どの知識を使うべきか」を判断しやすくなります。

手法4のファイル/フォルダ分割も有効でした。ただし、物理的な構造だけでは、各文書の意味や使いどころまでは十分に伝わりません。実用上は、手法3を基本形にし、手法4のフォルダ構造やREADMEを補助的に使うのがよさそうです。

逆に、手法2は今回の結果では弱く出ました。要約を足すだけでは、情報量は増えても、知識同士の関係や優先順位が見えません。モデルが材料を見つけやすくなる一方で、何をどう使うべきかの判断はあまり助けられなかったのだと思います。

## 運用方針

今回の検証を踏まえると、運用方針は次のようになります。

* 基本は手法3を採用します。知識構造に沿ったインデックスに、タイトル、要約、位置づけを持たせます。
* 手法4を補助的に使います。フォルダ分割やREADMEで、探索の入口と大きな地図を作ります。
* 手法2は単独では使いません。要約を足すだけでは、知識の関係性が弱く、出力品質が伸びにくいためです。
* テーマ選定を重視します。分散分析ではテーマ差の影響が大きく、インデックス改善だけでは補えない差がありました。
* 再学習は最後の手段にします。知識更新と管理コストを考えると、まずは構造化インデックスを整える方が現実的です。

要するに、AIにナレッジベースを使わせるには、文書を大量に渡すだけでは足りません。必要なのは、AIが探索できる知識の地図を作ることです。

タイトルとファイルパスだけでは入口が粗すぎます。要約だけでは関係性が足りません。効いたのは、知識構造に沿って文書を整理し、モデルが「何を探すべきか」「どの文書を読むべきか」「その知識をどの文脈で使うべきか」を判断できる状態にすることでした。

## まとめ

長いコンテキストやエージェンティック探索があっても、知識の構造化は依然として重要です。

AIに専門知識を使わせるには、単に資料を増やすだけでは足りません。資料同士の関係、階層、用途、優先順位を明示する必要があります。

今後のナレッジベース構築では、チャンク分割や埋め込み以前に、まず人間にもAIにも分かる知識の地図を作ることが、出力品質を上げる最短ルートになると考えています。
