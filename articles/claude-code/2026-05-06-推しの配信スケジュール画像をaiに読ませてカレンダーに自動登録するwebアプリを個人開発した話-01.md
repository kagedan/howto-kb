## はじめに

この記事は、以前投稿した GAS 版の記事の続編（Web アプリ版）です。  
GAS 版の記事 → [1日32時間配信視聴VTuberオタクがスケジュール画像自動解析・カレンダー連携システムを作った話【Gemini×GAS】](https://zenn.dev/elettrocannone/articles/afc3433b2907cd)

推しが1人なら、その人のスケジュール画像1枚を見ればいい。

でも推しが増えると話は変わる。個人勢、企業勢、グループ所属——スケジュールの告知方法は人それぞれで様々だ。YouTube のコミュニティ投稿、Twitter（現：X）の画像投稿、公式サイトの更新。フォーマットも統一されていない。

それを1人ずつ確認して、画像を目で読んで、見逃したくないものはリマインド登録。

——「スケジュール画像をAIが読み取って、カレンダーに入れてくれればいいのに」

そう思って作られたのが **[推しスケ](https://oshi-sche-webapp.vercel.app/)** です。

<https://oshi-sche-webapp.vercel.app/>

![推しスケの画面イメージ](https://static.zenn.studio/user-upload/dd97c93ec815-20260507.png)

私は1日に約32時間分の配信を視聴している視聴者です（複数の配信を同時視聴し、寝るときも聴きながら寝ているため24時間を超過）。

自分のスケールで使って満足できるなら、他の人は何も気にせず扱えるだろう、という思想で設計しています。

**開発期間は約3週間**（バイトと並行で、空き時間に Claude Code を使って開発）。

## 何ができるか

1. 推しの配信スケジュール画像をアップロードする
2. **Gemini AI が画像を読み取り**、配信タイトル・日付・開始時刻を自動で抽出する
3. 抽出結果をプレビュー・**手動で編集**して「登録」を押す
4. **iCal 購読 URL が発行される**。購読するアプリは Google カレンダー・Apple カレンダー・Outlook、どれでもOK
5. 一度購読すれば、以降は登録するたびにカレンダーが自動で更新される

ログインして画像を投げるだけ。それだけ。

ログインなしで触れるデモ → <https://oshi-sche-webapp.vercel.app/demo>

### スケジュール解析だけじゃない

推しスケはスケジュール解析がメインだが、登録後のスケジュールを見やすく管理する機能も作り込んでいます。

**カレンダービュー（日別 / 週間 / 月間）**

ホーム画面にはタブ切り替えで3つのビューを用意した:

* **日別ビュー** — 今日の配信を一覧表示。推しごとの担当カラーで誰の配信か一目で分かる
* **週間ビュー** — 1週間の配信をまとめて確認。当日の予定は強調表示される
* **月間ビュー** — 月カレンダーから日付を選ぶと、その日のスケジュールを表示

**推し名簿**

推しのプロフィールを管理する機能。名前・所属だけでなく、以下の設定ができる:

* **推しカラー** — 推しごとにテーマカラーを設定可能。カレンダーのイベントカードや名前バッジに反映される。推しが多いほどカラーで識別できるのが便利
* **誕生日** — 設定すると日別ビューに🎂バナーが表示され、3日前からカウントダウンが始まる。週間ビューにも🎂バッジが出る
* **記念日**（デビュー日など） — 誕生日と同様にカレンダーに反映。iCal にも出力される
* **所属グループ** — 所属（事務所など）を設定すると、名簿がグループ別にまとまる。**所属グループごとのカレンダー**（`/group/[affiliation]`）で、同じグループの推しの配信をまとめて確認できる
* **個別推しカレンダー** — 推し名簿から推しをタップすると、その推し専用の週間カレンダーが開く。「この推しだけの予定を見たい」ときに使う

## UX へのこだわり — ゲーマーの設計思想

設計の根底にあるのは **「出来の良いゲームと同じレベルのレスポンスと体験の質」** を目指すということ。基底の部分はゲーマーなので、「待たされる」「カクつく」「勝手に決められる」が許せない。

具体的には:

* **処理落ちが発生しそうな実装はしない。** CSS の `backdrop-filter` や多重シャドウなどの重い表現は使わず、色・余白・タイポグラフィの精度でリッチさを出す。モバイルのスクロール性能を最優先
* **AI 解析のフォールバックも「待たせない」ために設計している。** Gemini が 503 を返したとき、指数バックオフで何秒も待たせるのではなく即座に代替モデルに切り替える（後述）
* **解析結果をそのままカレンダーに登録しない。** AI の出力には誤りがあり得るので、ユーザーが手動で編集・確認してから登録する設計にしている。「勝手にやっといたよ」は便利なようでいて、間違ったときに最悪の体験になる

## 設計判断 5選

ここからが本題。「なぜそうしたか」「他に何を検討して、なぜ捨てたか」を書く。

### 1. Google Calendar API を捨てて iCal 購読に一本化した

**最初は Google Calendar API で直接イベントを登録する方式で作っていた。** MVP から Phase 3 まではこの方式で動いていて、ちゃんと使えた。

でもいざ OAuth 審査を出そうとした段階で、**Google Calendar API を使うには `calendar.events` スコープが必要で、これが「機密（sensitive）」に分類される** ことを知ってしっかり狼狽しました。  
CASA Tier 2 相当のセキュリティ審査が求められる可能性があり、費用は数万〜数十万円。  
現在フリーターで時給1,150円なので1ヶ月分ぐらいの出費であり、このプロダクトでペイはしようと思っていないので断念。

そこで、Phase 4 で **全部捨てて iCal 購読に一本化しました**。  
場当たり的な判断ではあるが、結果的にこれが正解だった。  
遠回りこそが俺の最短の道でした。

**iCal に切り替えて得られたもの:**

* **OAuth スコープが `openid email profile` だけで済む**。
* GCC 審査は branding 審査のみ（=ロゴやアプリ名の確認）で通過できた
* **Google ユーザー以外も使える**。Apple カレンダーや Outlook でもそのまま購読できる

**iCal 購読の仕組み:**

```
ユーザーが画像をアップロード → Gemini が解析 → イベントを Redis に保存
                                                          ↓
カレンダーアプリが定期的に購読URLにアクセス ← .ics を生成して返す
```

トークン（UUIDv4）で認証された URL を1つ発行するだけ。  
以降はカレンダーアプリが勝手にポーリングしてくれる。ユーザーの手数が圧倒的に少ない。

**デメリット:** カレンダーアプリのポーリング間隔はアプリ側の実装に依存するため、更新の反映が即時ではない。  
Google カレンダーは数時間〜最大24時間おきにしかポーリングしないこともあり、「登録したのにカレンダーに反映されない」と感じるケースがある。Apple カレンダーも同様にまちまち。  
ここは iCal 購読方式の構造的な制約。

この制約はアプリの FAQ ページで「カレンダーへの反映には時間がかかる場合があります」と明示し、設定画面でも購読 URL の説明と合わせて案内している。  
構造的に解決できない以上、ユーザーの期待値をコントロールするしかない。

### 2. フリーミアム + BYOK のハイブリッド課金

AI API を使うサービスの最大の課題は **「開発者の API 課金が爆発するおそれがある」** こと。

推しスケでは **ソシャゲのスタミナ制** と同じ設計思想で対処することとした。

| 種別 | 週の上限 | API キー | 開発者コスト |
| --- | --- | --- | --- |
| 無料ユーザー | **10枚/週** | 不要（開発者の共有キー） | 現状0円（Gemini Unpaid Tier） |
| BYOK ユーザー | **無制限** | 自分で取得（Gemini は無料で発行可能） | **0円** |

Gemini の Unpaid Tier は無料枠がかなり太いので、現状はユーザーが相当増えない限り開発者側の API コストは発生していない。とはいえ Google の方針変更に備えて、課金が発生しても BYOK への誘導でコストを転嫁できる設計にしてある。

BYOK（Bring Your Own Key）は、ユーザーが自分の Gemini API キーをブラウザの localStorage に保存して、**ブラウザから直接 Gemini API を叩く** 方式。  
サーバーを経由しないので、開発者側のコストはゼロ。

**BYOK における API キーの安全性について:** キーは端末の localStorage にのみ保存され、サーバーには一切送信しない。  
XSS によるキー窃取リスクを下げるため、アプリ全体で `dangerouslySetInnerHTML` の使用を禁止し、ユーザー入力はサニタイズしている。

```
無料ユーザー:  ブラウザ → /api/analyze（サーバー） → Gemini API（開発者キー）
BYOKユーザー:  ブラウザ → Gemini API（ユーザー自身のキー）  ※サーバー経由しない
```

**無料枠の使用量カウントは Redis でアトミックに管理している:**

```
// INCR で先にカウントを確保（race condition 防止）
const newUsage = await redis.incr(usageKey);

if (newUsage > WEEKLY_LIMIT) {
  // 上限超過 → INCR で増えた分を戻す
  await redis.decr(usageKey).catch(() => {});
  return NextResponse.json(
    { error: "QUOTA_EXCEEDED" },
    { status: 429 }
  );
}

// Gemini 呼び出し...
// もし Gemini が失敗したら DECR で戻す（枠を消費しない）
```

ポイントは **Gemini の呼び出しが失敗したら `DECR` でカウントを戻す** こと。  
「解析に失敗したのに枠だけ消費された」はユーザー体験として最悪なので。

**ユーザー識別にはソルト付き SHA-256 ハッシュを使っている:**

```
export function hashUserId(sub: string): string {
  const salt = process.env.USAGE_HASH_SALT;
  return createHash("sha256")
    .update(sub + salt)
    .digest("hex")
    .slice(0, 16);
}
```

Google の `sub`（ユーザー固有ID）をそのまま Redis に保存するのではなく、ソルト付きハッシュにすることで **仮に Redis が漏洩しても個人を特定できない** 設計にしている。

### 3. Gemini 503 への即時フォールバック戦略

Gemini API は混雑時に 503（過負荷）を返すことがある。

よくあるリトライ戦略は「指数バックオフで数回リトライする」だが、**Vercel の Serverless Function には 30秒のタイムアウト制限がある**。Gemini 2.5 Flash は Thinking（思考）機能が内蔵されていて、1回の呼び出しで 10〜20秒かかることがある。リトライしている余裕がない。

前述の通り、**「待たされる」は UX を致命的に損ねる** ので、503 を検知したら待機なしで即座に `gemini-2.0-flash` にフォールバックする戦略を取った:

```
try {
  // まず gemini-2.5-flash で試行
  const result = await model.generateContent(content);
  rawText = result.response.text();
} catch (err) {
  if (isGeminiOverloadError(err)) {
    const elapsed = Date.now() - startedAt;

    if (elapsed > 20_000) {
      // Thinking に時間を取られすぎ → フォールバックの余裕なし
      return error503Response();
    }

    // 即座に gemini-2.0-flash へフォールバック（待機なし）
    const fallbackModel = genAI.getGenerativeModel({
      model: "gemini-2.0-flash",
    });
    const fallbackResult = await fallbackModel.generateContent(content);
    rawText = fallbackResult.response.text();
    fallbackUsed = true;
  }
}
```

一方、**BYOK（ブラウザ直接呼び出し）側にはタイムアウト制限がない** ので、こちらは従来通りの指数バックオフ + ジッター（待機時間にランダム要素を加えて同時リトライの集中を防ぐ手法）でリトライする。

**同じ「503 対策」でも、実行環境の制約で戦略が変わる。** Vercel のタイムアウトを意識した設計は、個人開発で Hobby プラン（無料）を使っている以上は避けて通れない。

### 4. localStorage + Redis の同期設計

推しのプロフィール（名前・所属・カラー）はどこに保存するか。

DB を立てるほどのデータ量ではない（1ユーザーあたり数KB）。かといって localStorage だけだと、端末を変えたりブラウザのデータを消したら全部消える。

**採用した設計: localStorage をプライマリ、Upstash Redis をバックアップとする同期方式。**

```
ログイン時 → pullOshiData()
  ├─ Redis なし、localStorage あり → Push（初回同期）
  ├─ Redis あり、localStorage なし → Restore（新端末 or データ消失）
  └─ 両方あり → updatedAt が新しい方を採用（Last Write Wins）

推し変更時 → pushOshiData()
  └─ localStorage → Redis に PUT（失敗時は指数バックオフで最大3回リトライ）
```

**なぜ Google Drive AppData を使わなかったか:**

最初は Google Drive の隠しフォルダ（`drive.appdata` スコープ）に JSON を保存する案を検討した。しかし:

* レイテンシが 200〜500ms。Redis は 5〜15ms
* OAuth スコープが増える（審査に影響）
* データがユーザーの Drive に散在して、開発者側で利用統計を取れない

Redis は既に使用量カウントで導入済みだったので、追加コストなし。

**コンフリクト解決は Last Write Wins（最終書き込み勝ち）。** 推しリストは少件数（上限100件）・低頻度更新なので、CRDT やマージロジックはオーバーエンジニアリングと判断した。

### 5. 退会処理の実装

「Google アカウントとの連携を解除すれば終わり」……ではない。

Redis にはハッシュ化済みとはいえユーザーデータが残る:

* 使用量カウント（`usage:{hash}:{week}`）
* iCal トークン・イベントデータ
* 推し同期データ

GDPR や個人情報保護法の観点から、ユーザーが自分でデータを消せる手段は必要。

```
POST /api/user/delete
  → Redis のユーザー関連キーを全削除
  → セッション破棄
  → ログアウト
```

設定画面に「アカウント削除」ボタンを置いて、確認ダイアログを出して、押したら全部消えるようにしました。

## Gemini のプロンプト設計

配信スケジュール画像は VTuber ごとにデザインがバラバラなので、必須の項目のみ書いた。

```
export function buildPrompt(): string {
  const now = new Date();
  return `この画像はVTuberの配信スケジュールです。
以下の情報を抽出してJSON形式で出力してください。

本日は${currentYear}年${currentMonth}月${currentDate}日（${currentDay}）です。

注意事項:
- 年が書かれていない場合は以下のルールで判定：
  - 基本は${currentYear}年
  - 12月に1月のスケジュールがあれば翌年
  - 1月に12月のスケジュールがあれば前年
- 「24:00」「25:00」等の深夜表記は翌日に変換
- 配信がない日（OFF）も含めて出力
- JSON以外の文字は出力しないでください`;
}
```

**ポイント:**

1. **「今日の日付」をプロンプトに含める。** 画像に年が書かれていないケースが大半なので、年を推定するための基準日が必要
2. **年末年始の跨ぎを明示的にルール化。** 12月に翌年1月のスケジュールが載っているケースへの対応
3. **深夜帯の変換ルールを指定。** VTuber 文化では「25時」「26時」表記が普通に使われる
4. **休みの日もデータとして出力させる。** カレンダーに「OFF」と表示できるし、曜日の連続性検証にも使える

イラスト制作ソフト（Photoshop や CLIP STUDIO PAINT 等）で作成されたスケジュール画像に対しては概ね上手くいっている。一方で、手書きのカレンダーや、「ポ」と「ボ」のような濁点の有無の判別には弱い部分がある。

## 開発スタイル — Claude Code との AI ペアプログラミング

推しスケの開発は **Claude Code等を全面的に活用した AI ペアプログラミング** で進めた。

### ドキュメント駆動開発

プロジェクトには以下のドキュメント群を整備してある:

| ファイル | 役割 |
| --- | --- |
| `CLAUDE.md` | AIのための開発ダッシュボード。 技術スタック・設計方針・禁止事項・画面構成を集約 |
| `devnote.md` | 人間が見るための開発ダッシュボード。TODO・インボックス・作業ログを1ファイルで管理 |
| `decisions.md` | 決定事項集（ADR 代わり）。「なぜそうしたか」を全て記録 |
| 提案書群 | 新機能は実装前に `t○○-xxx-proposal.md` で設計を書き、レビューしてから着手 |

この構成により、**AI にコンテキストを渡すだけで「前回の続き」から作業を再開できる**。セッションが途切れても、ドキュメントが記憶の代わりになる。

### 「提案 → 承認 → 実装」のワークフロー

新機能の追加や設計変更は、必ず以下の流れで進める

1. AI に提案書を書かせる（or 自分で書く）
2. 提案書をレビューして方針を決定する
3. 決定事項を `decisions.md` に記録する
4. 実装に入る

**「いきなりコードを書かない」** のが原則。特に設計判断は期間が空くと「なぜこうしたんだっけ」と忘れることもあるので、意思決定の経緯をテキストで残しておく価値がある。

### テスト

Vitest で 65 テストを通過させている。テスト方針は「仕様ベース」で、実装を通すだけのテストは書かない。AI が書くテストは仕様で正当性を担保する。

### プロジェクト管理はテキストファイルで完結

Jira も Notion も使っていない。`devnote.md` の TODO テーブルとインボックスで全タスクを管理し、完了したものはアーカイブ用 `.md` に移動する。1人開発にはこのぐらいが身軽でいい。

## 推しスケの理想形

推しスケの現状は「画像をアップロードして AI に読ませる」が起点になっている。でも本当に目指しているのはその先。

**ゴールは、複数の事務所や多数の個人と関連する組織が、推しスケのようなサービスをより高いクオリティで提供してくれること。**

GAS 版の記事では「スケジュールの自動収集」を展望として書いた。  
でもよく考えると、クローラーで勝手に取ってくるよりも、**何かしら本人の承認プロセスを経たうえで、配信者側からスケジュールデータを提供してもらう** 方がずっとスマート。

普段から忙しい配信者の手を煩わせたり、心理的な負荷になるようなものではなく、  
**出せる週だけ気軽に投函できる仕組み**だといいなと思う。

最悪、自分の推しスケでなくてもいい。配信者とオタクの双方にとって「スケジュール共有」がもっと自然で楽になればいいな、と思っている。

## まとめ

推しスケは「画像をカレンダーに変換する」というシンプルな課題を解くアプリだが、裏側では:

* OAuth スコープの最小化と審査戦略
* AI API のコスト管理とフォールバック設計
* Vercel のタイムアウト制約下でのリトライ戦略
* localStorage と Redis のハイブリッド同期
* ゲーマー目線の UX 品質基準
* AI ペアプログラミングを前提にしたドキュメント駆動の開発体制
* 退会処理・プライバシー対応

……といった、**個人開発でも「ちゃんとやる」と決めたら意外と奥が深い設計判断** が連続だった。非常に楽しかった。
