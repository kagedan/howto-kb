今朝のタイムラインは、面白いくらい両極に振れてた。

片方は「もうAIに作らせるだけじゃなくて、テストも評価もPDCAも全部任せちゃおう」っていう自走化の話。もう片方は「いやいや、その前にM365とかExcelの基礎、ちゃんと押さえてる？」っていう足場固めの話。一見すると正反対なんだけど、根っこは同じだなと思って見てた。

Xのスレッドでは5トピックをサクッとまとめたけど、投稿だと「で、自分は何から手をつければいいの？」ってところまで書ききれない。この記事では各トピックの背景と、明日から試せる具体的なアクションまで掘り下げていくよ。

## 1. URLから「デザインの意図」を抜き出すスキル「Taste」

任意のURLを解析して、AIエージェント用のデザイン文脈に変換するClaude Code / Gemini CLIスキル、それが「Taste」。

よくある抽出ツールって、色コードやフォント、角丸の数値をペッと吐き出すだけなんだよね。でもTasteが違うのは、「なぜこの白なのか」「なぜ1pxのインセットを入れたのか」っていう、**意思決定とトレードオフまで書き出す**ところ。Extract→Detect→Infer→Observe の4つのエージェントが順番に動いて、`{ドメイン名}.md` と `.json` を生成してくれる。

なんでこれが効くのか。AI生成のUIが、どれもこれも似たような「量産型」（Interフォント・グレー配色・均一な8px角丸）になりがちな理由って、結局「数値だけ真似して、理由を知らない」からなんだよね。トークン（色やサイズの値）だけ渡されたAIは、それを盲目的にコピーするしかない。でも「なぜそうしたか」まで渡すと、初見のページでもちゃんと判断が効くようになる。

**実践ポイント**: 提案資料やLP、社内ダッシュボードのトーンを「あの既存サイトっぽく」寄せたい時に、Tasteで抽出した1ページをCLAUDE.mdに仕込んでおく。「Stripeっぽくして」みたいな曖昧な指示の精度が一気に上がる。ただし抽出した"意図"はあくまで推論だし、ブランドの模倣には配慮がいる点だけ注意。

ちなみに、Codex系の「impeccable / taste」っていう別のデザインスキル（参考画像を1:1でコード化するやつ）とは名前が似てるだけで完全に別物。混同しないようにね。

## 2. AIの回答が薄いのは「聞き方」のせい、直す4つの型

「ChatGPTもGeminiもClaudeも、3ヶ月ずっと回答が薄かった。原因を探ったら、自分の聞き方が悪かっただけだった」――これ、めちゃくちゃ共感する人多いんじゃないかな。

挙げられてた4つの型を整理するとこう。

* **① 「〜して」の前に役割を1行足す**（例：あなたはSNSマーケターのプロです）。Anthropicも公式に「役割付与で専門性と一貫性が上がる」と書いてる。
* **② ゴールを「相手の状態」で伝える**。何を作るかじゃなく、読んだ相手がどうなってほしいか。
* **③ 制約と形式を最初に渡す**（500字以内・箇条書き3つ・結論から）。
* **④ 自己採点→改善版まで完結させる**。一度出させて、自己採点させて、直させる。

特に技術的に「なるほど」と思ったのが③。AIって長い入力の真ん中を見落としやすい「lost in the middle」っていう性質があるんだよね。だから本当に守ってほしい制約は、文の冒頭か末尾に置くのが理にかなってる。

**実践ポイント**: 4つ全部いきなりやらなくていい。まずは④の「叩き台を出させる→自己採点させる→直させる」を1セットで回すクセをつけるのがおすすめ。最初から完璧を狙うより、結局これが一番速い。あと、ツールの使い分けも触れられてて、長文や構造化はClaude（XMLタグで指示を区切れる）、会話やブレストはChatGPT、みたいな棲み分けは覚えておくと損しない。

## 3. SharePointリストをCopilotのナレッジにできるようになった

地味だけど、M365使ってる会社にとってはかなり効く話。Copilot StudioとAgent Builderで、**SharePointの「リスト」をそのままナレッジソースに追加できる**ようになってた。

何が嬉しいかというと、今まではExcelやPDFをアップロードして読ませる方式が中心だった。でもアップロードってその時点の「スナップショット」なんだよね。一方でリストを直接ナレッジにできると、在庫表やタスク一覧みたいな**常に更新される構造化データを、最新のまま参照させられる**。これは大きい。

実機で確認した人によると、「試してみる」だと最初うまく動かなかったけど、エージェントを正式に作成してから対話したらちゃんとリストから応答が返ってきたとのこと。正式には7月からパブリックプレビューの見込み。

**実践ポイント**: リストを追加する時は構造化データなので、コードジェネレーター（コードインタープリター）をオンにするのがコツ。ただし制限もあって、リストの「ビュー」は選べない／glossary・synonymsは非対応／デフォルトビューに12個を超えるlookup列があるリストはNG（12列以下に絞る）。まずは小さいリストから試すのが安全。情シスの承認の壁でClaude Codeみたいな外部ツールを入れにくい企業でも、M365の中で完結できる選択肢になるのが地味にありがたい。

## 4. AIに「作らせる」の次は「評価とPDCA」も任せるloop

これが今朝いちばん本質的だなと思ったトピック。

「ちょっと前までは、AIに作らせて、人間が手触りを試したり評価する側、みたいな話だった。でもloopの概念はそこをぶっ壊す。評価すらAIに託せば、人介在のボトルネックがなくなって、AIだけで全部回る」――つまり、ビジネス目線で言うと「PDCAを丸ごとAIで回すのが当たり前になる」って話。

これ、実は最近のClaude Codeの進化とぴったり重なってる。Claude Codeの作者ボリス・チェルニーも「俺はもうClaudeにプロンプトを与えてない。プロンプトを与えるシステム（ループ）を作ってる。俺の仕事はループを書くことだ」って言ってるんだよね。実際v2.1.139では `/goal` っていう機能が入って、**条件を満たすまでターンをまたいで自走し、別の高速モデルが毎ターン採点する**っていう仕組みになってる。投稿で言われてた「テストや評価をどうやらせるか」がキモ、っていう指摘とまさに一致する。

ただ、ここで大事なのは「いきなり全自動は危険」ってこと。自走化でよくある失敗が、永続記憶を用意してない／検証役を分離してない／停止条件がなくて課金が止まらない、あたり。

**実践ポイント**: 入口はめちゃくちゃ地味でいい。まず毎回打ってる定型指示をCLAUDE.mdに移して、「10分ごとにPRを確認」みたいな小さな定期チェックから自走させ始める。これだけで「あ、自分が待ちの主役になってたんだ」って気づけると思う。

## 5. Copilotで化ける人は、M365の「基礎」を押さえてる

トピック4が「先端」なら、これは「足元」の話。両方並べると今朝のトレンドの全体像が見えてくる。

自律で動くAIで「ファイルが全部消えた」「情報が漏れた」みたいな事故が増えて、情シスの承認や外部ツール禁止の壁から「だったらM365 Copilotで改善しよう」って流れが業種問わず増えてるらしい。でもここで指摘されてたのが、Copilotは結局M365の上で動くから、Copilot単体のスキルだけ追っても足りない、ってこと。

成果を出す人は土台が違う、と。具体的には――SharePointとOneDriveの違いが分かる／M365全体のアプリ機能を理解してる（Copilotを使う場面と使わない場面の判断ができる）／ExcelのXLOOKUPやFILTERみたいな基本関数を押さえてる／**データの置き方でCopilotの回答精度が変わると知ってる**。

**実践ポイント**: 新機能を追いかけるより、自分が毎日触ってるアプリを一段深く理解する。遠回りに見えて、これが一番の近道かもしれない。特に「データをどこにどう置くか」は、トピック3のSharePointリスト連携にも直結する話。土台がないままCopilotだけ覚えると、「思った答えが返らない」で止まっちゃうんだよね。

## 今朝のトレンドを俯瞰すると

並べてみると、今朝は「自走化の理想（loop/PDCAをAIに任せる）」と「土台固めの現実（M365の基礎、データの置き方）」が同時に伸びてた。一見すると真逆の方向なんだけど、根っこは完全に同じだなと思う。

それは「**AIの出力品質と再現性を、どう仕組み化するか**」っていう一点。

プロンプトの聞き方4型も、デザインスキルのTasteも、突き詰めると「曖昧な指示をやめて、意図と制約を構造的に渡す」っていう同じ話に収束する。loopだって「良いプロンプトを一回書く」から「AIが動く仕組みを設計する」への移行だし、M365の基礎の話も「データを構造的に置けば回答精度が上がる」っていう、結局は同じ原則。

だから、先端を追うか足場を固めるか、で迷う必要はないんだよね。やることは「指示と環境を構造化する」で一貫してる。先端を試したいなら `/goal` や loop、足場からなら CLAUDE.md やデータの置き方。入口が違うだけで、向かってる先は同じ。

## まとめ

今朝のポイントをざっくり3行で。

* AIの仕事は「作る」から「評価・PDCAまで自走させる」フェーズへ。入口は小さな定期チェックから
* 出力品質を上げるカギは「意図と制約を構造的に渡す」こと。プロンプトもデザインも同じ
* 先端を追う前に、M365やデータの置き方みたいな足場を固めると遠回りに見えて近道
