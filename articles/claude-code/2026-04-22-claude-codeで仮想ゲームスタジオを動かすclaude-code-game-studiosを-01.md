ゲームのアイデアがぽつんと浮かんだとき、最初の一手に迷いませんか。仕様書を書き始める、エンジンを選ぶ、とりあえずプロトタイプを動かす。やることは山ほどあるのに、手が止まる。筆者も何度か経験があります。

そんな時に見つけたのが[Donchitos/Claude-Code-Game-Studios](https://github.com/Donchitos/Claude-Code-Game-Studios)というリポジトリです。Claude Codeを仮想のゲーム開発スタジオとして動かすテンプレートで、49体のエージェントと72のスラッシュコマンドがセットになっています。

題材は「ネズミが主人公でマンションに住み着いて食料を持ってくるゲーム」というふわっとしたアイデア。ここからどう動かしていくのかを追ってみます。

## Claude-Code-Game-Studiosとは

Claude Codeを本物のゲームスタジオのように役割分担させて動かすテンプレートです。MITライセンスで公開されているので、cloneしてそのまま使えます。

普通のClaude Codeと違うのは、49体の専門エージェントが3階層で組織化されていて、企画から実装、QA、リリースまでを72のスラッシュコマンドがカバーしている点です。コミットやアセット変更のタイミングで自動の検証フックも走ります。エンジンはGodot 4、Unity、Unreal Engine 5の3つに対応していて、それぞれに専門エージェントが用意されています。

勝手にゲームを完成させてくれるわけではありません。エージェントは専門知識と構造を提供するだけで、最終判断は常にユーザーが下す設計です。自動運転ではなく、協調して進めるための土台と思っておくとしっくりきます。

## 3階層のエージェント構成

上位が下位に仕事を振り、同レベル同士で相談し合う、会社の組織図に近い動き方をします。

| 階層 | 役割 | エージェントの例 |
| --- | --- | --- |
| Tier 1 | ディレクター層 | クリエイティブディレクター、テクニカルディレクター、プロデューサー |
| Tier 2 | 部門リード | ゲームデザイナー、リードプログラマー、アートディレクター、QAリード |
| Tier 3 | スペシャリスト | ゲームプレイプログラマー、AIプログラマー、シェーダー専門家、UXデザイナー |

ソロ開発のしんどさって、結局は全部の役割を一人で背負うところにあります。プログラミングに熱が入ってナラティブが薄くなる、バランス調整を後回しにしてリリース直前で焦る。よくある話です。このテンプレートだとフェーズごとに別の視点を持つエージェントが口を挟んでくるので、盲点が放置されにくくなります。

## 主なスラッシュコマンド

72個あるうち、よく使いそうなものをカテゴリ別に並べるとこんな感じです。

```
# 企画・設計
/brainstorm              # アイデアの深掘り
/design-system           # コアループの言語化
/create-architecture     # 技術設計

# ストーリー管理
/create-epics            # 大きな機能をエピックに分解
/create-stories          # 実装単位に落とす
/dev-story               # ストーリーを実装する
/story-done              # 完了マークを付ける

# レビュー
/code-review             # コードレビュー
/design-review           # デザインレビュー
/balance-check           # ゲームバランスの確認

# QA・リリース
/qa-plan                 # テスト計画
/smoke-check             # 起動確認
/soak-test               # 長時間プレイの耐久テスト
/release-checklist
/launch-checklist
```

エピック単位の計画から耐久テスト、リリース前チェックまで、スタジオで普通にやる工程がひとまず揃っています。

## 導入手順

Claude Codeとgitは既に使える前提で話を進めます。やることは3つだけです。

### ステップ1 リポジトリをclone

```
git clone https://github.com/Donchitos/Claude-Code-Game-Studios.git mouse-game
cd mouse-game
```

プロジェクト名は好きに変えてください。筆者は今回ネズミゲームを作るつもりなので`mouse-game`にしました。後で愛着が湧きますね。

### ステップ2 Claude Codeを起動

`.claude/`ディレクトリ配下のエージェントやスキル、フックが自動で読み込まれます。

### ステップ3 `/start`を実行

最初はこれを必ず打ちます。現在地を聞かれるので、アイデアの熟し具合に応じて選びます。

* no idea まだ何も決まっていない
* vague concept ふわっとしたアイデアだけある
* clear design 設計は固まっている
* existing work 既存プロジェクトがある

ネズミが主人公でマンションに食料を持ってくる、くらいだとvague conceptが妥当です。ここを飛ばすとエージェントの動きがちぐはぐになるので、必ず通してください。

## ネズミゲームで動かしてみる

vague conceptから始めた場合の流れを、フェーズごとに追っていきます。

### Phase 1 コンセプト固め

クリエイティブディレクターが音頭を取り、ゲームデザイナーやナラティブディレクターを連れて質問を投げてきます。

* ジャンルはステルス、パズル、ローグライトのどれに寄せるか
* 視点は2D横スクロール、トップダウン、3Dか
* 住人との関係は敵か、共存か、観察される側か
* 食料の使い道は巣作りか、仲間に配るか、繁殖か
* エンディング条件は冬を越すか、家族を増やすか

答えていくと、たとえば2Dトップダウンのステルスパズル、住人に見つからず食料を巣に運ぶ、みたいに輪郭が見えてきます。

続いてコアループを言語化します。探索して食料を見つけ、住人を避けて巣まで運び、アップグレードに回す。この繰り返しをドキュメントに落とします。

### Phase 2 技術設計

テクニカルディレクターがエンジン選定と構造設計を引き受けます。ネズミの2Dトップダウンなら、軽量で個人開発と相性の良いGodot 4を薦められそうです。視界システムやAIステートマシン、インベントリまわりもここで骨組みが決まります。

生成物はGDDが`design/`に、スプリント計画が`production/`に配置されます。

### Phase 3 エピックとストーリーへの分解

```
/create-epics
/create-stories
```

プロデューサーが大きな機能をエピックに割り、さらに実装単位のストーリーへ落とします。ネズミゲームだとこんな粒度になります。

* Epic ネズミの移動システム
  + Story 8方向移動の実装
  + Story 壁沿い移動とスニーク状態
  + Story 家具の下に隠れるインタラクション
* Epic 住人AI
  + Story 巡回ルートの生成
  + Story 視界コーンと音の検知
  + Story ネズミを見つけた時の反応

### Phase 4 実装ループ

ストーリーを1つ選ぶと、必要な専門家が自動で呼び出されます。ゲームプレイプログラマーが移動を実装し、シェーダー専門家が毛並みの見せ方を考え、オーディオディレクターがカサカサ音のアイデアを出す、みたいな流れです。

```
/code-review
/balance-check
```

実装後はコードレビューとバランスチェック。住人の視界が広すぎて初心者には厳しい、みたいな調整がここで入ります。

完了マークを付けて次のストーリーへ。`/dev-story`から`/story-done`まで、このループを回し続けるのが実装フェーズの中心になります。

### Phase 5 QAとリリース

```
/qa-plan
/smoke-check
/soak-test
/release-checklist
/launch-checklist
```

テスト計画を立て、起動確認を通し、長時間プレイで落ちないか見て、リリース前のチェックリストを埋める。この順で回していく形です。

## ソロ開発で効く理由

筆者がこのテンプレートで一番おもしろいと感じたのは、ナラティブディレクターみたいなソロだと後回しになる役割が、普通に出張ってくるところです。

ネズミゲームだと、食料を盗むことへの罪悪感だとか、住人の家族構成が垣間見える演出だとか、そういう味付けを提案してくる可能性があります。プログラマー脳で進めていたら絶対に出てこない角度ですね。バランス調整やQAもフェーズに組み込まれているので、リリース直前の慌てもたぶん減ります。

## 向いているケースと向いていないケース

何にでも使えばいいわけでもなくて、合う相手と合わない相手があります。

合いそうなのはこんな人です。

* アイデアはあるが、設計と実装の規律を保ちたい人
* 中規模以上のゲームをソロで作る人
* 複数のエンジンを行き来する可能性がある人

逆に、ゲームジャムで48時間のうちに作り切るような超短期プロジェクトには重すぎます。エージェント連携のオーバーヘッドが効いてきて、プロトタイプを素早く回したい場面ではむしろ足を引っ張ります。

## コストの話

リポジトリ自体は無料ですが、動かすにはClaude Codeの利用料がかかります。49体のエージェントが連鎖して動く設計なので、トークンの消化はわりと早めです。Opusを使うと特に早いです。

ProプランとAPI従量課金のどちらでも動きますが、本格的に使うならMaxプラン以上が無難、というのが筆者の肌感です。お試しなら一度Proで雰囲気を掴んでから判断するのがよさそうです。

## おわりに

Claude-Code-Game-Studiosは、Claude Codeのサブエージェントとスラッシュコマンドを使い倒したテンプレートです。ソロだけどスタジオ並みの規律で作りたい、という人には刺さりそうだと感じました。

ネズミゲームくらいの、フックがはっきりした中規模プロジェクトとは相性が良いはずです。筆者も実際に`/start`まで進めてみて、続きを記事にしていく予定です。
