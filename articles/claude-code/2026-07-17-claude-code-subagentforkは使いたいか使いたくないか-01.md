## 背景: fork既定は本当にコスト最適か

自作のマルチエージェント・オーケストレーション基盤では、大きめのタスクを1つのサブオーケストレーターに丸ごと投げて「分解 → 並列実装 → 検証」を一サイクル回す仕組みを使っている。このサブオーケストレーターを呼び出す側には、実行全体を `fork`（呼び出し元のcontextとprompt cacheを丸ごと継承するサブエージェント起動方式）に包むか、呼び出し元のプロセス内で直接（inline）実行するかという選択がある。

これまでの経験則としては「forkはcontext rot対策として有効」という定性的な結論は出せていた。長時間の対話の中で試行錯誤したcontextを引き継がずに済むし、呼び出し元のcontextに重い実行ログが積み上がらずに済む。しかし「それが本当にコスト最適か」——fork自体のオーバーヘッド、prompt cache共有によるコスト削減効果、実行時間の差——は、まだ勘に頼った判断のままだった。

### forkがコスト・実行時間で有利だと考えられる理由

forkを既定にしたくなる理由は、context rot対策だけではない。むしろ実務上はコストと実行時間の面での期待の方が大きい。

* **prompt cacheの共有**: forkは親会話のcontextをそのまま継承するため、親側で既にキャッシュ済みのプレフィックス（システムプロンプト＋それまでの会話履歴）をそのままキャッシュヒットさせられる。キャッシュ読み出しは新規入力トークンより大幅に安いので、共有部分が大きいほどforkの実行コストは下がる。
* **手動キュレーションの手間が消える**: fork以外の起動方式では、呼び出し元が必要な情報を毎回grepしたり要約したりして手作業でプロンプトに埋め込む必要がある。この作業自体が呼び出し元のターンでトークンと時間を消費する。forkはこの手間そのものを丸ごと省略できる。
* **プレフィルの高速化**: キャッシュヒットした分だけ、実行開始からの応答までの待ち時間（prefill）も短縮される理論上の余地がある。新規起動のエージェントは毎回ゼロから長いプロンプトを処理する必要があるのに対し、forkは共有プレフィックスの再計算をスキップできる。

ただしこれはあくまで理論上の期待であって、無条件に成り立つ話ではない。forkは親の会話を丸ごと継承するので、cache読み出しの単価が安くても**継承するトークン総量そのものが大きい**可能性がある。会話が長く育ち、雑談やdead endまで積もっていれば、キャッシュ割引を差し引いてもforkの方が高くつく場面はありうる。逆に、タイトに手作業でキュレーションされた新規起動プロンプトは、キャッシュ割引が無くても総トークン数がそもそも小さく、結果的に安く速いことも十分考えられる。

つまり「forkは安くて速いはず」という予測にはそれなりの理屈があるが、それが**実際にどちらに転ぶか**は、親会話がどれだけ長く育っているか・タスクがどれだけタイトにキュレーション可能かに依存する。ここが定性的な理屈だけで決め切れない部分であり、今回計測基盤を足す動機に直結している。

同じ基盤には、モデル階層の選択を実績データ（実行ごとの成否・コスト・所要時間を溜める記録ストア）から決定論的に決める仕組みが既にある。delegation戦略（fork/inline）の選択も、いずれは同じ枠組みで実績から学習させたい。ただし今はまだ比較データが存在しない。そこで今回のタスクのスコープは**計測基盤の追加だけ**に絞ることにした。自動判定ロジックの実装はしない——これは「measure してから policy に反映する」という、このプロジェクト全体を貫く"build ≠ validate"の思想をそのまま踏襲した判断だ。

## なぜ既存の「モデル比較」機構をそのまま使えないか

このオーケストレーション基盤には、既に「同一タスクを別モデルのworkerで並行実行して比較する」機構がある。しかしこれは**サブオーケストレーターの内部**（同一runの中）で完結する比較であり、両方の分岐はオーケストレーター自身が把握・制御できる範囲にある。

一方「forkで包むか/inlineで実行するか」は、**サブオーケストレーター自身ではなく、それを呼び出す側（呼び出し元のロジックを実行しているLLM自身）の選択**だ。サブオーケストレーターの内部からは「自分がforkの中で動いているか、呼び出し元から直接呼ばれたか」を区別する手段が無い。したがって「1回の実行内で両分岐を並行実行して比較する」という既存機構の形はそもそも当てはまらない。既存の比較ロジックには一切手を入れず、別の経路で計測することにした。

## 提案した設計: 新しい実行機構は作らない

比較すべきは「等価なタスクを、fork経由で実行した場合とinlineで実行した場合の、コスト・所要時間・成否」。これは**サブオーケストレーターの外側**（呼び出し元のLLM）が手動で選択し、実行後に**手動で**既存の記録コマンドに書き足せばよい。新しい実行機構は不要で、既存の記録コマンドに1つオプション引数を足すだけで足りる、という結論になった。

具体的には次の4点だけを足す。

1. **既存の記録コマンドに、任意のラベル引数を追加する。** どちらの戦略で実行したかを示す文字列（例: `fork` / `inline`）を省略可能な形で受け付ける。値のバリデーションはしない——既存の分類ラベル引数と同じ緩さに揃える。
2. **記録先のレコード構造体に、対応する任意フィールドを追加する。** 既存の別フィールド（人手評価ラベルなど）で使っている後方互換パターン（デフォルト値・未設定時はシリアライズしない）をそのまま踏襲する。これにより、このフィールドが無い過去の記録行も問題なく読み込め、記録しない限り出力にキーが増えることもない。
3. **呼び出し元のガイドラインに2点を追記する。**
   * **既定バイアス**: サブオーケストレーターを起動する際は既定で`fork`に包む。例外は「タスクが極小・単発ファイルで、ユーザーが経過を対話的に見たいと明示した場合」のみ直接実行する。
   * **手動記録の呼びかけ**: 実行が完了したら（fork/inlineいずれでも）、観測できたコスト・所要時間を、既存の記録コマンドに戦略ラベルを添えて記録する。これは自動比較機構ではなく手動の実績記録であり、狙いは「等価なタスクのfork実行/inline実行の実績を、時間をかけて記録ストアに貯める」ことだけだ。
4. **仮説を1件、明確な検証基準つきで登録する。** 「fork経由での実行とinline実行とで、コスト・所要時間に有意な差があるか」という仮説を、具体的な検証基準（目安として実績10件以上、fork/inline両方最低3件以上）とともにopenのまま登録する。十分な実績が貯まった時点で、次の測定ステップがこれを拾い、validate/rejectする。

## 明示的にスコープ外にしたこと

* **既存のモデル選択ロジックへのdelegation軸の追加は実装しない。** データが無い状態でロジックだけ足しても検証しようがない。仮説がvalidateされてからの別タスクとする。
* **既存のモデル比較機構の内部ロジックの変更はしない。** 今回のdelegation計測とは無関係。
* **context使用率からfork/inlineを自動判定する仕組みも実装しない。** context可観測性の仕組み自体は既にあるが、それを使った自動判定はデータが無い今の時点では時期尚早。呼び出し元のガイドラインは固定の既定バイアス（常にfork、例外のみinline）に留める。

## レビューして見えた、計測ループ自体の詰まりどころ

この設計は、shadow-runとの切り分けや既存パターンの流用など、方針としては妥当だと思っている。一方で「この計測ループが実際に回るか」という観点で、後からレビューして気づいた懸念点が3つある。

1. **既定forkバイアスがinlineサンプルを枯渇させかねない。** 既定が常にforkで例外が稀なら、仮説が要求する「inline側も最低3件」に到達するまで非常に長くかかる可能性がある。データを集める設計のはずが、既定バイアス自体がデータの偏りを生む——という入れ子になっている。
2. **記録ステップが完全に手動でゲートが無い。** 記録コマンドを呼ぶかどうかは呼び出し元のLLMの「思い出し」任せで、これを強制する決定論的な仕組みが無い。「判断はLLM、決定論はバイナリ」という基盤全体の設計思想からすると、この計測ステップだけがLLM任せになっている。記録漏れが起きても検知できない。
3. **ラベルが無検証のfree-text。** 意図的な緩さではあるが、表記ゆれ（大文字小文字の違いなど）が後の閾値判定にそのまま混入しうる。

つまり「forkを使うかどうかを最終的にはデータで決めたい」という発想自体は正しいのに、そのデータを集める仕組みそのものが、判断をLLMの記憶に依存させたり既定バイアスで分布を歪めたりする、同じ種類の脆さを持ちうる——という点は、次にこの計測ループに手を入れるときに気に留めておきたい。

## まとめ

* forkの理論上の有利さは「prompt cache共有によるコスト削減」と「手動キュレーション省略による速度向上」に集約されるが、継承するcontext総量が大きくなるとこの優位性は薄れる可能性があり、理屈だけでは決め切れない
* fork/inlineのような「質的には妥当だが定量的に未検証」な既定判断は、いきなり自動化ロジックを書くのではなく、まず最小限の計測基盤だけを足すのが筋が良い
* 既存の record 系コマンドや構造体に、後方互換な任意フィールドを1つ足すだけで、新しい実行機構を作らずに済む
* ただし「手動記録に依存する計測ループ」自体が、既定バイアスによるサンプル偏りや記録漏れという新しいリスクを持ち込むことは忘れてはいけない。計測基盤を作ること自体もレビュー対象にする価値がある
