## はじめに

こんにちは。TobanというAIとスクラムが組めるサービスを開発しています。  
一応古いバージョンを公開しているんですが(<https://app.toban.dev/>)、まともに使えないという状態です。

現状結構行き詰まっており、タスクを細かく分割してエージェントの並列化を図ったところ、Claude Code と 1対1-3ぐらいでやったほうがクオリティの高いものができてしまい頭を抱えています。そんな試行錯誤の内容をAI君にまとめてもらいましたので参考になれば。

（文中に出てくるPlaybookはこのサービスの肝で、エージェントが失敗した内容や、レビュー内容を保管しておき、次のタスクに生かすという機能になる予定でした）  
↓↓↓↓↓

## TL;DR

AIエージェントのオーケストレーションプラットフォームを開発する中で、「タスクを細かく分解して並列実行すれば速い」という直感が間違いだったことを実験で確認した。最適解は **Story（やりたいこと）+ AC（完了条件）+ ファイルヒント（どこを見ればいいか）** だけをエージェントに渡すこと。タスク分解もPlaybookルールも、速度を落とすだけだった。

---

## 前提: 何を作っていたか

AIコーディングエージェントを複数同時に動かすオーケストレーションプラットフォーム「toban」を開発していた。設計思想はこうだ:

1. **Strategist**（強いモデル）がユーザーのStoryをタスクに分解
2. **Builder**（軽いモデル）がタスク単位で高速に実装
3. 並列Builderで複数タスクを同時実行 → スループット向上

人間のスクラムチームと同じ発想。Storyを分解し、タスクを割り振り、並列に進める。

## 実際に起きたこと

**並列なのに遅い。**

* タスク単位だとエージェントが毎回コードベースを読み直す。4タスクなら4回
* APIの型を変えたタスクと、呼び出し側のタスクが別エージェントで動いて型が合わない
* worktree作成、マージ、ビルド検証、レビューの合計時間がコードを書く時間より長い
* **人間と1つのエージェントが会話しながら進めた方が速くて正確**

6時間かけてStory単位実行の仕組みを設計・実装・改修した末に、「そもそもこのアプローチは正しいのか？」という疑問が湧いた。

## 仮説を立てて実験する

本番コードへの改修をやめ、30分でプロトタイプ実験を行った。

### 実験1: 新規プロジェクトでの比較

ミニマルなHono APIプロジェクトに「メモ帳API（CRUD + タグ + 検索）」を実装させる。`claude --print` で直接実行し、プロンプトの違いだけを比較。

| パターン | 時間 | テスト数 |
| --- | --- | --- |
| Story + ACだけ (Sonnet) | 68秒 | 12 |
| Story + サブタスク参考 (Sonnet) | 69秒 | 17 |
| Story + ACだけ (Haiku) | 74秒 | 20 |

**差がない。** モデルの違いもプロンプト形式の違いも、実行時間にほぼ影響しなかった。全パターンでテスト全パス、型チェックOK。

ちなみにHaikuが一番テスト数が多く、リクエスト型の分離定義やテスト用リセット関数など、最も丁寧な実装だった。

### 実験2: 既存コードベース（3万行）での比較

同じ比較を、3万行の既存プロジェクトで行った。題材は「decompose進捗のリアルタイム表示」。既存のWebSocketパターンとの統合が必要。

| パターン | 時間 | 分解コスト | 合計 |
| --- | --- | --- | --- |
| Story のみ | **3:20** | - | 3:20 |
| Story + サブタスク + Context | **1:32** | 2:51 | 4:23 |
| Story + ファイルヒントのみ | **1:46** | - | **1:46** |

**ファイルヒントだけで2倍速くなった。** サブタスク分解は不要だった。

新規プロジェクトでは差が出なかったのに、既存コードでは大差がついた。理由は明確で、**エージェントがコードベースを探索する時間**。「`plan-view.tsx`を見ろ、既存のWSパターンがある」と教えるだけで探索が不要になる。

タスク分解の真の価値は「何を実装するか」の指示ではなく、**「どこを見ればいいか」のナビゲーション**だった。

### 実験3: Playbookルールの効果

同じタスクをPlaybook（22ルール）あり/なしで比較。

| 条件 | 実装時間 |
| --- | --- |
| Playbook あり（22ルール） | 4:40 |
| Playbook なし | **2:37** |

**Playbookで2分遅くなるだけだった。** 品質に差はない。

ルールの中身を見ると:

* 「シンプルで読みやすいコード」— 言われなくてもやる
* 「テスト必須」— テンプレートプロンプトに既に書いてある
* 「コミットしてから完了報告」— 同上
* セキュリティ系 — CLAUDE.mdに書けば良い

**大半が「当たり前のこと」か「別の場所で既に指示されていること」。** コンテキストを圧迫して遅くしているだけだった。

## 見えてきた最適パターン

```
Story: <ユーザーの意図>
AC: <検証可能な完了条件>
Files: <変更対象ファイルと既存パターンへのポインタ>
```

これだけ。タスク分解もPlaybookルールも不要。

ファイルヒントは`CLAUDE.md`にコードインデックス（主要ファイルと責務の一覧）を書いておけば自動で解決する。

## タスク分解が「不要」とは言っていない

AIの**実行効率**には不要だった。しかし人間にとっての価値はある:

* **不安の解消**: 「本当にこの設計で大丈夫？」という箇所を事前に詳細化してもらう
* **レビュー粒度**: 「API部分はOKだがテストが不足」のような部分承認
* **進捗の可視化**: ダッシュボードで3/5タスク完了と表示

つまりタスク分解は**AIの実行パイプラインの一部**ではなく、**人間のレビュー・承認プロセスの一部**。必要な時だけ人間がリクエストするもの。

## おまけ: git logはエージェントを騙す

実験中に面白い現象があった。前回成功した実装を`git revert`で巻き戻してリトライしたところ、エージェントが`git log`で前回のコミットを見つけて「既に実装済み」と判断し、何もせず完了報告した。

コードは巻き戻されているのに、履歴には実装コミットが残る。エージェントは履歴を信頼してファイルを読まなかった。`git reset --hard + force push`で履歴を消しても、CLI内部のローカルクローンが古いままで同じ現象が起きた。

通常運用では稀だが、NEEDS\_CHANGES後のリトライで似た問題が起きうる。

## 数字で見る1日の学び

| やったこと | 時間 |
| --- | --- |
| Story単位実行の設計・実装・改修 | ~6時間 |
| プロトタイプ実験 | ~30分 |
| 不要コードの削除 | ~10分（171行削除） |

6時間かけて作ったものの大半が30分の実験で不要と判明した。

**教訓: 仮説は本番コードではなくプロトタイプで検証する。**

## まとめ

1. **タスク分解はAIの実行効率を上げない** — Story + AC + ファイルヒントが最速
2. **Playbookルールは逆効果** — コンテキストを圧迫するだけ
3. **ファイルヒントが決定的に効く** — 「どこを見ればいいか」が最も価値ある情報
4. **新規 vs 既存で結果が違う** — 新規プロジェクトでは何をやっても差が出ない
5. **仮説はプロトタイプで検証する** — 本番コードで実験するとコードが劣化する
