# はじめに

久々の投稿になります。直近だと、こんなことをしていました。

* セキュリティ関係の仕事をやっていた(そのまま正社員になった)
* AI関係の仕事も継続でやっている
* オフショアのマネジメントをやったり、共同研究やったり
* 車の免許を取った  
  などなど

本格的にアラサーになったのが原因か分かりませんが、とにかくドキュメントを見て理解することが増えました。特にAIの凄まじい進歩で、プログラムを書くよりもドキュメントを正しく理解する認知負荷の方が大きくなっている印象が日に日に強くなっています。

実体験を踏まえると、AIワークスロップが多く、中身を確認せずに半ばドキュメントを押し付けてくるケースが最近だと特に多い。過去の文書を含め、下記の印象が強く、ストレスが大きくなっていました。

* オフショアの報告書(というよりも、海外の方がよく書く報告書全般ですが、)がアップリケが多くてシンプルに読みづらく、日本語に戻すと余計に認知負荷が増大することが多かった
* 最近見る報告書も殆どAI依存だが、あまり中身を確認してないことが多いため、文章がとにかく冗長
* 第三者の過去の報告書がとにかく読みにくい時がある。淡白な文章で良いので、簡潔に書いて欲しい欲求が個人的に強まった
* 過去の経験だと、文章に「個性」が入っており、ドキュメントを見ると誰が書いたか分かったり癖があって人によって読みにくいケースがある→統一感を出したい
* 評論っぽい書き方の方が、なんだかんだで読みやすいしカッコ良いじゃん？(個人的感想)

そこで、上記の問題を解決するためのプラグインとしてmeisekiを作成した形になります。他にも素晴らしいプラグインがあったのですが、中身を簡潔に読み取りたいことに特化させるなら自作した方が良いかなと思い、作ってみた次第です。

成果物はこちらになります。  
<https://github.com/bamboo-nova/meiseki>

# 概要

AIに日本語の技術ドキュメントを書かせると、文法的にはどこも間違っていないのに、一度読んだだけでは意味が取れない文がよく出てくる。読み返して、頭の中で語順を組み替えて、ようやく分かる。原因を探っていくと、語彙の問題ではなく構文の問題だった。そこで、その構文だけを削ぎ落とす Claude Code プラグイン `meiseki`（明晰＝澄んで分かること）を作った。最初は一枚のスキルとして書き、最終的に textlint を束ねたプラグインへ育てた。この記事はその動機と作り方、効果の記録である。

## きっかけ：既存の「humanizer」が解いていない課題

AI文章を直すツールはすでにいくつもある。整理すると、源流は Wikipedia にある。

WikiProject AI Cleanup というボランティア集団が、AI生成記事の増加に対抗して、観測したパターンを [Wikipedia:Signs of AI writing](https://en.wikipedia.org/wiki/Wikipedia:Signs_of_AI_writing) というページにまとめている。約1万5千語の「フィールドガイド」で、ルールではなく観測記録だという立場をとる。挙げられているのは、たとえば次のようなパターンだ（参照：[WikiProject AI Cleanup](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_AI_Cleanup)）。

* 否定の並列（"not X, but Y"）
* 太字＋コロンの箇条書き
* 全角ダッシュの多用
* 意味を水増しする末尾の分詞句
* ありふれた事実を「重要」「画期的」と持ち上げる傾向

英語圏の humanizer 系ツールの多くは、このガイドを土台にしている。

日本語版もある。

これらを一通り読んで気づいたのは、ほぼ全部が「自然さ」「声」または「AI検出の回避」を目標にしているということだ。人間が書いたように聞こえること、書き手の主体性が見えること、検出器に引っかからないことを目指している。

でも、私が困っていたのはそこではなかった。困っていたのは、AIが書いたドキュメントが**読みにくい**ことだ。自然かどうかより前に、まず一度で理解できない。しかも、その読みにくさは「AIっぽい単語」を直しても消えなかった。

## 仮説：読みにくさは語彙ではなく構文に宿る

サンプルを集めて眺めると、引っかかる文には共通の構造があった。

* **二重否定**：「設定を変更しないと動作しないわけではありません」。打ち消しを2回、頭の中で処理させられる。
* **長い連体修飾**：名詞の前に説明の節を積み上げ、最後の名詞に着くまで主題が分からない。
* **「の」の連鎖**：「システムの可用性の向上のための施策の検討」。抽象名詞を数珠つなぎにして、係り受けが多義になる。
* **過剰な名詞化**：「改善の実現に寄与します」。動詞で言えば一発で分かるものを、わざわざ名詞に固めている。

どれも、ネイティブでも一度読んで意味が取れない。共通点は「読み手に脳内で組み替え作業をさせる」ことだ。逆に言えば、ここを機械的に直せば、語彙をいじらなくても読みやすくなるはずだった。

これが設計の軸になった。**評価軸を「自然さ」から「明晰さ」へ振り替える。** 自然に聞こえるかどうかは問わない。一度読んで理解できるかどうかだけを見る。

## 設計：何を借りて、何を変えたか

ゼロから作らず、既存の良いところを組み合わせた。

stop-slop の「**削る**」思想を借りた。読みにくさの正体は足りないものではなく余計な構文なので、加えるより削る方が筋がいい。humanizer-ja からは「**出力は本文のみ**」「**意味とトーンは守る**」という運用方針を踏襲した。診断レポートではなく、読める本文を返すのが目的だからだ。

その上で、3つを変えた。

1. **目的を明晰さに限定した。** 個性や語り口を足す処理（英語版や humanizer-ja が持つ「personality」の注入）は入れない。読みやすくすること以外はしない。
2. **二重否定と否定の入れ子を最優先に置いた。** ここが日本語ドキュメントで最も認知負荷が高く、かつAIが量産する構文だからだ。
3. **対象を技術記事・ドキュメントに広げた。** humanizer-ja が対象外にしている領域だが、私が一番直したいのはここだった。

## 声 vs 明晰さ

いちばん近いのは [stop-ai-slop-jp](https://github.com/iKora128/stop-ai-slop-jp) だ。ただ、向きは逆と言っていい。あちらは「AI臭の正体は書き手の不在」という立場から、一般論を「自分は」に書き換えて言い切らせ、リズムにムラを入れ、偏愛語を効かせる。つまり**声・主体性を足す**。

meiseki はその逆で、**声を足さない**。技術ドキュメントでは、書き手の個性より「誰が読んでも同じ意味に取れる中立さ」のほうが大事だからだ。仕様書に「自分はこう思う」と立場を立てられても困る。だから meiseki は、一般論を意見に書き換えたり、語り口を強めたりはしない。あくまで構文をほどいて読解負荷を下げるだけにとどめる。

棲み分けはこうなる。意見記事やエッセイで「自分の声」を立てたいなら stop-ai-slop-jp、仕様書・README・設計書を中立に読みやすくしたいなら meiseki。両方ほしいときは、先に meiseki で構文をほどき、あとから stop-ai-slop-jp で声を足す、と重ねればいい。

## 作り方：高負荷構文をカタログ化し、スキルにする

Claude Code のスキルは Markdown ファイルである。中核の `SKILL.md` に方針・手順・ガードレールを書き、詳しい変換規則は `references/patterns.md` に分けた。patterns.md は必要なときだけ読み込ませて、本体を軽く保つ。

`patterns.md` は、高負荷構文を優先度順に6カテゴリへ整理した。

* **A. 否定・条件の入れ子**（最優先）
* **B. 係り受け・修飾の距離**
* **C. 名詞化・漢語の重さ**
* **D. 冗長・空虚な表現**
* **E. 構造・体裁**
* **F. 情報の構造化（埋もれた列挙の箇条書き化）**

最後の F は、3つ以上の同格の項目が一文に埋もれているときだけ箇条書きに開く、という後から足したカテゴリだ。E（飾りだけの箇条書きは散文に戻す）と一見矛盾するが、統一原理は「構造を内容の論理形状に合わせる」ことで揃えている。埋もれた列挙は開き、飾りの箇条書きは畳む。ここで「太字＋コロン」の定型に寄せないことも明記した。それ自体がAIっぽさの指紋だからだ。

各項目に「問題 / Before / After / 注意」を付けた。重要なのは「注意」で、ここに**やりすぎを止める条件**を書いている。たとえば二重否定の項には「真意が肯定なら肯定形に畳む。ただし真偽の反転に最大限注意し、控えめな肯定のニュアンスを担うものは残す」と明記した。

ガードレールは強めに置いた。意味・事実・主張を変えない。技術用語・API名・数値・コードは触らない。易しくすると意味が変わる箇所は、語の置換ではなく文の分割で対処する。そして「元から読みやすい文は触らない」。これは作ってみて初めて必要だと分かった条件で、全文を均質に整えると、今度は無機質な「エディタが撫でた感じ」が出てしまうからだ。

## 効果：実際にかけてみる

「いかにもAIが書いた技術ドキュメント」を一本かけてみた。

**Before**

> 近年、マイクロサービスアーキテクチャの採用が進む中で、サービス間通信における信頼性の確保は重要な課題となっています。本ライブラリは、ネットワーク障害が発生した際にリクエストを自動的に再送する処理を担当するリトライ機構を提供するものですが、すべてのエラーに対して再送が行われるわけではなく、再送が適切でないとされるエラーについては再送が行われないように設定することができます。また、リトライ回数の上限の設定の検討が必要となる場合があり、これを適切に設定しないとシステムへの負荷の増大を招かないとは言えません。

**After**

> 本ライブラリは、ネットワーク障害が起きたときにリクエストを自動で再送するリトライ機構です。ただし、すべてのエラーを再送するわけではありません。再送が適切でないエラーは、再送しないように設定できます。
>
> また、リトライ回数には上限の設定が必要な場合もあります。これを適切に設定しないと、システムの負荷が増えることがあります。

冒頭の一般論、長い連体修飾、「の」の連鎖、「することができます」が消えている。

一番の見どころは最後の二重否定だ。「負荷の増大を招かないとは言えません」は、論理的には「招くことがある」という意味になる。ここを「負荷が増えない」と畳んでしまうと、ドキュメントが事実と逆のことを言ってしまう。スキルには「二重否定を肯定に直すときは真偽の反転を必ず検算する」というガードレールを入れてあり、結果として「負荷が増えることがあります」へ正しく畳めている。読みやすさを上げる処理が、意味を壊す事故と紙一重であることがよく分かる例だ。

## スキルからプラグインへ：textlint で「測れる明晰さ」にする

ここまでは、何が高負荷かをスキル（＝LLM）の判断に委ねていた。これだと判断が安定しないし、before/after を数字で示せない。

そこで気づいたのが、日本語にはすでに [textlint](https://github.com/textlint/textlint) という決定論的な校正リンターがあり、技術文書向けプリセット [preset-ja-technical-writing](https://github.com/textlint-ja/textlint-rule-preset-ja-technical-writing) が meiseki の狙う構文をほぼ持っていることだった。二重否定（`no-double-negative-ja`）、一文の長さ（`sentence-length`）、読点過多（`max-ten`）、連続漢字（`max-kanji-continuous-len`）、冗長表現「することができる」（`ja-no-redundant-expression`）。これらを、行・列つきの機械可読な結果で返してくれる。

Python で独自の検出器を常駐させる手もあったが、それは違うと判断した。外部APIも持たないテキスト解析に常駐プロセスはいらない。確立した textlint を Bash で呼んで結果を読むほうが軽いし、決定論性は textlint 側がすでに担保している。

そこで meiseki を「スキル単体」から「プラグイン」に育てることにした。役割分担はこうなる。

* 検出と採点は textlint（決定論層）。meiseki 用に絞った `.textlintrc` を同梱する。フルプリセットだと表記ゆれや感嘆符まで効いて「整えすぎ」になるので、読解負荷に効くルールだけを残す。
* リライトはスキル（LLM層）。textlint を実行して結果を読み、二重否定や長い一文を直し、もう一度 textlint をかけて before→after で検算する。
* textlint では取れない構文 ── 長い連体修飾、「の」連鎖の深さ、一般の名詞化、そして埋もれた列挙の箇条書き化 ── は、引き続きスキル側が担う。ここが「ただの textlint 設定」と meiseki の差になる部分だ。

図にすると、こうなる。meiseki スキル（LLM オーケストレーター）が入力ドキュメントを受け取り、決定論層の textlint と LLM 判断層のパターンカタログ A–F に振り分ける。最後に、明晰化した本文と読解負荷スコア（before→after）だけを返す。

読解負荷を一つの指標にまとめる。textlint の指摘をカテゴリ別に重みづけし、文数で正規化した「読解負荷スコア」を出して、リライト前後で下がったかを見る。二重否定は重みを最大にして原則ゼロを目標にする。これで「自然かどうか」という主観ではなく、「読解負荷が下がったか」を数字で追える。

ここでも気をつけたのは、声を足さないことだ。textlint には弱い表現（「〜と思います」）を叩く `ja-no-weak-phrase` もあるが、これは「曖昧で意味が取りにくいときだけ外す」用途に限る。stop-ai-slop-jp のように立場を言い切らせる方向には使わない。測るのはあくまで読解負荷であって、書き手の声ではない。

## おまけ：textlint を入れる以上、サプライチェーンも気にする

プラグインに textlint を入れた以上、`npm install` でサプライチェーン攻撃の経路を増やすことになる。侵害された npm パッケージは、公開直後に取り込まれて、数日のうちに発覚・unpublish される、というパターンが多い。そこで `meiseki` には cooldown を効かせた。同梱の `.npmrc` に一行だけ書いてある。

これで「公開から7日未満のバージョンはインストールしない」。npm 11.5 以降の標準機能で、内部的には `before = 現在 − 7日` に変換して依存を解決する。あえて「枯れた」バージョンだけを採用して、侵害が世に知れ渡るまでのリスク窓を避ける、という発想だ。最新をすぐ入れたいときは `npm install --min-release-age=0` で一時的に外せる。

明晰化そのものとは別の話だが、「人が読むための道具が、別の経路で人を危険にさらさない」ようにしておくのは、プラグインとして配る以上は最低限だと思う。

## 限界と、次にやること

正直に書くと、これで完成ではない。

`Signs of AI writing` のページ自体が興味深い注意を残している。「これらの兆候を直すべき問題そのものとして扱うな。それは検出を難しくするだけかもしれない」。表面のパターンを潰すことと、文章の中身を良くすることは別だ、という指摘である。meiseki は読みやすさを上げるが、内容の薄さや論理の飛躍までは直さない。そこは人間の仕事として残る。

読解負荷スコアという評価の枠組みは設計したが、重みの較正はこれからだ。次は、自分の実ドキュメントを束で通して「削りすぎ／削り足りない」を見ながら、`.textlintrc` のしきい値とスコアの重みを詰めていく。二重否定のニュアンスをどこまで肯定に畳むかは好みが割れるところなので、技術ドキュメントで許容する範囲を決めて、ルールを尖らせるつもりだ。

それでも、軸を「自然さ」から「明晰さ」に振り替えるだけで、AIが書いた日本語ドキュメントはかなり読めるものになる。自然に聞こえることより、一度で理解できることのほうが、ドキュメントには大事だと思う。
