## この記事で作るもの

AIエージェントにクラウドを操作させるとき、参照等のリスクの低い操作を任せ、リスクの高い操作は引き続きシステムや人間の手により行う、という場面は多いと想像される。

この記事では、その分担をAWSで実際に組んで検証する。

題材はS3のオブジェクトの読み書きにする。

* AIエージェント：バケットの中身を読ませる操作を実施。更新作業は拒否
* システム または 人 ：バケットの読み書きを実施

この要件を満たすだけなら、S3の読み取り権限を持つIAMユーザーを1つ作って終わりにもできる。今回はそれをせず、**AWS Identity Center**（旧AWS SSO）でAI用と人間用の権限を分け、**AWS MCP Server**経由でエージェントに読み取り専用のロールだけを持たせる構成を組んだ。  
人間とAIを同じ土台に載せたうえで、それぞれに別々の権限境界を引ける形にしたかったからだ。

最終的な構成は次のとおりになる。AIエージェント側の認証情報は、Identity CenterからもIAMユーザーからも供給できるようにした。

本記事では、認証方式や、偶発的なAIエージェントによる誤操作のリスクを回避できるか、といった点で整理を行う。

## MCPサーバーの選び方

AIエージェントとAWSリソースをつなぐMCPサーバーには、いくつかの選択肢がある [参考#5]。今回の題材はS3だが、選び方の考え方は他のサービスでも共通する。

| MCPサーバー | 提供形態 | 特徴 |
| --- | --- | --- |
| `awslabs/mcp`（個別サービス版） | OSS・ローカル実行 | サービスごとに特化したサーバー群。特定のワークフローには強いが、やりたい操作の粒度に合うサーバーがあるとは限らない |
| AWS API MCP Server（`awslabs/mcp`） | OSS・ローカル実行 | AWS CLI相当の汎用呼び出しを1本でカバー。ブラウザ経由の認可やリモート接続の仕組みは持たない |
| AWS MCP Server（Agent Toolkit for AWS） | フルマネージド・リモート | AWS API MCP Serverと同じく汎用的だが、フルマネージドでOAuth対応。条件キーやトークン失効などのガバナンス機能もサービス側に用意。今回採用 |

今回はガバナンス機能の充実を重視し、2026年5月にGAしたAWS MCP Server（Agent Toolkit for AWS）を選んだ [参考#1]。

このサーバーの認証まわりは、GA後も動きがある。2026年7月には、AWS Sign-InがOAuthをネイティブでサポートするようになった [参考#2]。それ以前の認証はSigV4署名のみで、MCPクライアント自身は署名を行えないため、署名を代行する「MCP Proxy for AWS」というローカルプロキシが必要。

## Identity Centerをセットアップする

Identity Centerの中に、権限の単位である**Permission Set**を2つ作る。

権限の中身はAWS管理ポリシーではなく、バケット名を指定したカスタムのインラインポリシーにした。`S3-ReadOnly`用のポリシーは次のとおりである。

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowListBucket",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-mcp-demo-bucket"
    },
    {
      "Sid": "AllowGetObject",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-mcp-demo-bucket/*"
    }
  ]
}
```

`S3-ReadWrite`側は同じ構成で、`s3:PutObject`と`s3:DeleteObject`を加えただけの内容にした。バケット名を明示したことで、最小権限の原則がポリシーのJSONを見るだけで分かる形になっている。

`S3-ReadOnly`にはもう1つ、`AWSMCPSignInOAuthAccessPolicy`というAWS管理ポリシーを追加でアタッチしている。これはOAuthでサインインするために必要な`signin:AuthorizeOAuth2Access`と`signin:CreateOAuth2Token`という権限を含むポリシーで、後述するOption A（OAuth認証）を使う場合の前提条件になる。

Permission Setを作ったら、Identity Centerユーザーに両方を割り当てる。これで1人のユーザーが、サインインする文脈に応じてReadOnlyにもReadWriteにもなれる状態ができた。

## 認証方式を比較する

AWS MCP Serverには2つの認証方式がある。

* **Option A（OAuth）**：MCPクライアントがブラウザを開いてAWSにサインインし、認可を得る方式。AWS Sign-InのOAuthネイティブ対応として2026年7月に提供が始まった、比較的新しい経路になる [参考#2]。ローカルにプロキシを立てる必要がなく、セットアップが簡単
* **Option B（SigV4）**：`mcp-proxy-for-aws`というローカルプロキシ [参考#4] を経由し、AWS CLIの認証情報でリクエストに**SigV4**（AWSのAPIリクエストに使われる署名方式）の署名を付ける方式。プロキシの起動時に使うAWSプロファイルを明示的に指定するため、複数アカウントの切り替えにも対応

Option Bはさらに、署名の元になる認証情報の取得方法で2つに分かれる。Identity CenterのSSOセッションを使う方法と、従来型のIAMユーザーのアクセスキーを使う方法である。今回はこの3パターンすべてを構築し、同じバケットに対して動かして比較した。

3つの認証方式を並べて比較すると、次の表になる。

| 観点 | Option A（OAuth） | Option B（SigV4 + Identity Center） | Option B（SigV4 + IAMユーザー） |
| --- | --- | --- | --- |
| 認証の入口 | ブラウザの現在のセッション | 明示指定したSSOプロファイル | 明示指定したAssumeRoleプロファイル |
| 認可時に人間がセッションを選ぶ | する（選び間違えるリスクがある） | しない | しない |
| セットアップの手間 | 小（ブラウザで完結） | 中（AWS CLI・uv・SSOプロファイル設定） | 中（IAMユーザー・ロールの作成が別途必要） |
| ローカルに残る長期シークレット | なし | なし | あり（IAMユーザーのアクセスキー） |
| マルチアカウント切り替え | 非対応 | 対応 | 対応 |

### Option Aは「今ブラウザでログインしているセッション」を使う

認可画面には今アクティブなセッションが表示され、どれを利用するかユーザが選択する。  
![](https://static.zenn.studio/user-upload/8cf6d10f8da2-20260712.jpg)  
誤って強い権限のセッションを選択してしまうと、エージェントにはその権限がわたる。AIエージェント専用の権限を渡すためには、事前にAIエージェントが利用するユーザでログインする必要がある。この一手間を人間の注意力に委ねることになる。

### Option Bは「あらかじめ設定ファイルに指定したプロファイル」を使う

Option Bはプロキシの起動時に使うプロファイルをあらかじめ`.mcp.json`に書いておく。認可のたびに人間が選択肢を選ぶ場面自体が存在しないため、選び間違えるという操作ミスの余地が構造的にない。

## Claude Codeを接続する

3つの認証方式を、それぞれ別名のMCPサーバーとして`.mcp.json`に登録した。以下はAIエージェントとしてClaude Codeを使った場合の手順である。

### Option A（OAuth）

`claude mcp add`コマンド1つで済む。

```
claude mcp add aws-mcp https://aws-mcp.us-east-1.api.aws/mcp --transport http -s project
```

初回のツール呼び出し時にブラウザが開き、AWSへのサインインと認可を求められる。ここでIdentity Center側のサインイン方法を選ぶ点が、前章で触れた注意どころになる。

### Option B（SigV4 + Identity Center）

先にAWS CLI側でSSOプロファイルを用意する。

```
[profile s3-readonly-mcp]
sso_session = identity-center
sso_account_id = 123456789012
sso_role_name = S3-ReadOnly
region = ap-northeast-1

[sso-session identity-center]
sso_start_url = https://d-xxxxxxxxxx.awsapps.com/start/#/
sso_region = ap-northeast-1
sso_registration_scopes = sso:account:access
```

```
aws sso login --profile s3-readonly-mcp
claude mcp add aws-mcp-sigv4 -s project -e AWS_PROFILE=s3-readonly-mcp -- \
  uvx mcp-proxy-for-aws==1.6.2 https://aws-mcp.us-east-1.api.aws/mcp \
  --metadata AWS_REGION=ap-northeast-1
```

### Option B（SigV4 + IAMユーザー）

手前でIAMユーザーとロールを用意する必要がある。ここは単純にIAMユーザーへ直接S3の権限を付けるのではなく、IAMユーザーには`sts:AssumeRole`だけを許可し、実際の作業は一時クレデンシャルで行うという構成にした。

IAMユーザー自身のポリシーは、対象ロールへの`sts:AssumeRole`だけを許可する内容に絞った。

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAssumeS3ReadOnlyRoleOnly",
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::123456789012:role/S3-ReadOnly-AssumeRole"
    }
  ]
}
```

IAMユーザーが直接触れる権限をAssumeRole1つに絞ることで、このユーザーのアクセスキーが漏洩しても、被害はロール側に定義したS3の読み取り権限の範囲に収まる。

### 3方式を動かした結果

3つとも接続したうえで、それぞれに自然言語で読み取りと書き込みを指示した。結果はいずれのパターンでも同じで、S3のreadは期待どおり機能し、writeはIAMによって拒否された。

## Option A（OAuth）を採用した際のヒューマンエラーを防ぐ方法

比較表で見たとおり、Option Aは認可の瞬間にどのセッションを使うかを人間の判断に委ねる。Identity Centerを経由しないRoot・IAMユーザーのセッションが残ったまま認可を通してしまえば、エージェントはS3に留まらずアカウント全体の権限で繋がりかねない。この選び間違いは、IAMのグローバル条件キーを使うと、ソフトウェア側で機械的に防ぐなどの工夫が必要であるる。

AWS MCP Serverは、転送するすべてのリクエストに`aws:ViaAWSMCPService`（Bool、AWSマネージドMCPサーバー経由なら`true`）と`aws:CalledViaAWSMCP`（String、どのMCPサーバー経由かをサービスプリンシパルで示す）という2つの条件キーを自動的に付与する [参考#3]。これをSCPやIAMの権限境界に組み込むと、「MCP経由のリクエストであれば、どのPermission Setで認可されていようと書き込み系アクションを一律拒否する」というルールを作れる。

このガードを敷いておけば、認可の瞬間に人間がどちらのセッションを選んだかに関わらず、書き込みは機械的に止まる。今回はSCPの構築までは行っていないが、Option Aを使い続けるなら検討する価値がある。

## まとめ

AIエージェント専用の操作権限を渡す。これ自体は、Identity CenterのPermission SetやIAMロールに専用のIAMポリシーを用意するだけで達成できる。

AWS MCP Serverには認証方式が2つ（OAuthとSigV4）ある。OAuth（Option A）は環境構築が非常に簡単だが、認可時にどのセッションを使うか人間の判断に委ねるため、ヒューマンエラーの余地が残る。防ぐには、条件キー`aws:ViaAWSMCPService`を使ったSCP等のガードを敷くか（ただし管理アカウントには効かない点に注意）、あらかじめプロファイルを明示指定するOption B（SigV4）を選び、構造的に選択の余地をなくすか、という2つの方向性がある。
