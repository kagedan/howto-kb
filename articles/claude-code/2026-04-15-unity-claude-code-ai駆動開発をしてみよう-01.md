## ◆**はじめに**

「バイブコーディング」という言葉が流行っています。  
AIに指示を出してコードを生成させ、開発を進めるというのは未来を感じますよね。

* 業務品質のゲーム開発でそんな開発できるの？
* それってスキル成長止まらないの？

さっそく結論ですが、「バイブコーディング」とまではいかなくとも「AI駆動開発」は可能です。  
コードへの責任のとり方はかわってくるかもしれません。

今回実践した経験をもとに、考え方・運用フロー・環境構築までを紹介します。

### **バイブコーディングとAI駆動開発**

AI任せ、どこまでいけるものかと疑問を持ちつつ開始し、  
積極的にAIに任せる方針で開始しました。  
実際、バイブコーディングまでは任せずという運用とっています。

|  | バイブコーディング | ← 実際はこの辺り → | 従来型開発 |
| --- | --- | --- | --- |
| **コードを書く** | AIが全部書く | AIが書く | 人間が書く |
| **コードを読む** | 読まない | ほぼ読まない（AIがレビュー） | 人間がレビュー |
| **設計** | AIに丸投げ | 方針だけ人間が決める | 人間が詳細設計 |
| **品質管理** | 動けばOK | AIテスト＋人間が手直し | 人間がテスト設計 |
| **人間の介入** | ほぼなし | 要所で介入 | 全工程に関与 |

**つまり「AIにほぼ任せるが、人間が要所で介入する」** というスタイルです。  
ほぼコードを見ていないものの、不具合を解消できるだろうと見込みを持っています。

### **AI駆動開発のリスク**

具体的なフローに入る前に想定していた（している）リスクです。

#### **⚠️ 「AIが書いたコードを理解していない」問題**

AIに任せる範囲が広がるほど、**自分が書いていないコードがプロジェクトに増えていく**という現実があります。これは超怖いです。

* 予想外の不具合が起きたとき、対応できるの？
* AIが「動く」って言ったら、それでいいの？
* そもそもそれって私のスキル？

#### **📈 それでも、AIでコーディングする時代は来ている**

とはいえ、AIでコーディングをする時代はもう来ています。 エンジニアに求められるスキルの重心が変わりつつあります。

| 相対的に価値が下がるスキル | 価値が高まるスキル |
| --- | --- |
| 定型的なコードを書く力 | **全体設計・アーキテクチャ設計** |
| 言語やフレームワークの文法知識 | **原因不明の不具合を解消する力** |
| 単純なデータ変換・整形 | **開発フロー全体の設計・最適化** |

### **使用ツール**

* **Claude Code** — AIコーディングエージェント（メイン）
* **Unity MCP** — UnityエディタとAIを接続するプラグイン
* **GitHub Copilot** — コードレビュー・補完（補助的に使用）

## ◆**AI駆動開発の運用フロー**

実際にどうやってClaude Codeを使って開発を進めているのか、1タスクあたりの流れを紹介します。

AI駆動開発の運用フロー

### **Step 1：☁️ クラウドで作業開始**

Claude Codeはクラウド上でも動作するため、**まずはクラウド環境で作業を開始**します。ロジックの実装やファイル生成など、Unityエディタが不要な作業はここで一気に進めます。

### **Step 2：🖥️ ローカルにテレポート**

Prefab作成やUnityエディタ固有の対応が必要になったタイミングで  
**会話セッションをキープしたまま** ローカル環境に切り替えます。

### **Step 3：🔨 Prefab作成 → 手直し**

Unity MCPを介して直接Prefabを作成させます。  
思ったよりやってくれますが、手直し必要です。  
他、不具合もAIに修正させます。

### **Step 4：🧹 リファクタ・テストコード・デバッグコード**

動くものができたら、品質を上げるフェーズです。

* **リファクタリング** — 「今回の対応をリファクタリングして」と指示  
  AIとやり取りして実装した部分も、まとめられたり、違うフローだったりを解消
* **テストコード** — 「今回の対応で必要なテストがあったら追加して」と指示  
  この工程はAIが活きますね。人間がやるより信頼できそうです。
* **デバッグコード** — 必要に応じてログ出力や検証用コードを追加

### **Step 5：📝 PR作成**

これもAIで対応します。  
（実際これを見てなるほどとなることも…）

### **Step 6：👀 レビュー → 対応 → マージ**

1. **GitHub Copilot**でPRをレビュー
2. Copilotのレビュー指摘をコピーし、**Claude Codeにプロンプトとして投げて対応させる**
3. Claude Codeがコードを修正し、必要ならPRコメントへのreplyも行う
4. 人間が**ざっと確認してマージ**

## ◆**どこが厳しい？Unity開発のAIとの「責任分界点」**

「UnityでどこまでAIに任せられるの？」という疑問に対する回答は、  
「**Prefab、Unity対応は人が責任とりましょう**」です。

ここまでの実践を通じてわかった、AI駆動開発の得意・不得意をまとめます。

### **✅ AIがほぼ完遂できる領域（得意）**

* **ピュアなC#ロジック**（計算、データクラスの自動生成など）
* **ルールが明確な変換作業**（フォーマット変換、データ整形）
* **大量ファイルの一括処理**（インポート設定の統一など）
* **リファクタリング・テストコード作成**（既存コードの品質向上）

### **❌ 人間が手を入れる前提の領域（不得意）**

* **複雑な階層を持つPrefabの構成**
* **外部パッケージの選択**
* **空間的・視覚的構造の理解** （アングル、レイアウト）

Prefabは結構作ってくれます。  
ボタンアニメーションも指示しておけば作ってくれます。  
それでも100%信用するのではなく、  
**70〜80%の完成度で受け取って人間が仕上げる**という心構えで取り組みましょう。

## ◆**環境構築：AIに「目」と「手」を与える**

前章の運用フローを回すために必要な環境構築を解説します。

先述の通りAIは「見た目」が苦手ですが、ある程度それを補うことができます。

### **Unity MCP の導入**

Unity MCPを導入することで、AIがUnityエディタと直接通信できるようになります。  
`com.unity.ai.assistant`  
βですが、MCP部分だけ使います。

* シーン情報の取得
* コンポーネントの操作
* コンソールログの確認

### **Game View Capture：AIに「目」を与えるMCPツール**

AIに画面をキャプチャさせましょう。  
Unity MCPのAttributeがあるのでMCPで拡張もいい感じです。

```
	[McpTool("Unity_GameView_Capture",
		"Captures the Game View during Play Mode and returns the image as base64 PNG. " +
		"Uses ScreenCapture to capture the full Game View output " +
		"including all cameras, UI overlays, and post-processing effects. " +
		"Only works when the Unity Editor is in Play Mode.",
		EnabledByDefault = true,
		Groups = new[] { "editor" })]
	public class GameViewCaptureTool : IUnityMcpTool<GameViewCaptureParams>
```

AIが自分の作業結果を視覚的に検証できることで、  
**ある程度は**自律的にUIや見た目の修正作業が可能になります。

### **スキルファイル（.claude/skills/）の活用**

AIに繰り返し行わせたい作業を「スキル」として定義しておくと、  
精度と効率が大幅に向上します。PRテンプレートやキャプチャ手順など、**チーム共有のナレッジベース**としても機能します。  
手順だけでなくて、ナレッジ、ガイドラインもスキルに流しましょう。

* デバッグ機能の作成手順
* UIPrefabのガイドライン
* プッシュ、PR作成作成の手順
* Unityのドメインリロード

push\_and\_pr.md

サンプルです。  
これもAIで作って、AIで育てていきましょう。  
（私も内容、初めて見ました。）

```
---
name: push_and_pr
description: 現在のブランチをpushしてPull Requestを作成する。担当者は作成者、Label/Milestoneは提案して確認を取る。
user_invocable: true
---

現在のブランチをpushしてPull Requestを作成してください。

## 手順

### 1. 現在の状態確認

```bash
# ブランチ名、未コミット変更、リモートとの差分を確認
git branch --show-current
git status --short
git log develop..HEAD --oneline
```

未コミット変更がある場合はユーザーに確認してからコミットする。

### 2. Push

```bash
git push origin {branch_name}
```

### 3. PR情報の準備

#### 担当者
```bash
# 現在のGitHubユーザー名を取得して担当者にする
gh api user --jq '.login'
```

#### Label/Milestone
利用可能なLabel/Milestoneを取得してユーザーに提案する：
```bash
gh label list --json name --jq '.[].name'
gh api repos/{owner}/{repo}/milestones --jq '.[] | "\(.number): \(.title)"'
```

提案例：
> Label: `バトル`、Milestone: `α版（292277026596/12/4）` でよいですか？

**ユーザーの確認を得てから次に進む。**

#### PRタイトルとbody
- `git log develop..HEAD --oneline` と `git diff develop --stat` から変更内容を把握する
- `.github/pull_request_template.md` を Read ツールで読み取り、テンプレートのセクション構造に従ってbodyを作成する
- テンプレート内のHTMLコメント（`<!-- ... -->`）はそのまま残す
- 各セクションの `- ` プレースホルダーを実際の内容で置き換える

### 4. PR作成

```bash
gh pr create \
  --title "{タイトル}" \
  --assignee {ユーザー名} \
  --milestone "{マイルストーン名}" \
  --label "{ラベル名}" \
  --body "$(cat <<'EOF'
{テンプレートに従ったbody}
EOF
)"
```

### 5. 結果報告

PR URLをユーザーに表示する。
Copilotレビューが必要な場合はGitHub UIから手動追加が必要な旨を伝える。
```

## ◆**まとめ**

バイブコーディングは「AIでコードを書く」という入口として優れていますが、  
業務品質のUnity開発では**その先の仕組み化**が重要です。

AI駆動開発の最大のメリットは**ノウハウが蓄積していく**ことです。

* **人間側：** AIとの協働を重ねるほど、指示の出し方や任せ方が洗練される
* **AI側：** スキルファイルやPRテンプレートが充実するほど、AIの出力品質が上がる

今日うまくいかなかったことが、スキルファイルを1つ追加するだけで明日はスムーズに回る。  
**人とAIの両方にナレッジが溜まっていく**ことで、開発体験は確実に良くなっていきます。  
**一連のフローが回り始めると、開発の生産性は加速します。**

### 💡 **コラム：それって後輩くん（若手）は成長するの？**

「定型コードを書く機会が減ったら、若手はどうやって育つのか？」  
確実なことは言えないですが、  
こういったサイクルの中でも成長は見込めるのではと感じました。

AIを利用すると、これらの対応に  
少し遅れて気づくことが多くなるはずです。

気づき、対応する機会があるなら、成長するチャンスはあるはずです。  
「正しく成長する」の重要性が高まりそうですね。  
逆に「アーキテクチャの良し悪し」を判断する機会は多くなるかもしれません。
