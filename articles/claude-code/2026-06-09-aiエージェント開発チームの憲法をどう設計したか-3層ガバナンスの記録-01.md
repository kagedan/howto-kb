「安全文字起こし」の開発では、複数の Claude Code エージェントが並行で動いています。lead、dev、PO、webmaster、qa、claude-code-guide — 役割の違うエージェントが同じリポジトリを触る構成です。

最初は気軽に始めたのですが、すぐに困りました。誰が何を更新してよいかが曖昧で、同じファイルを同時に書き換えて競合する、レビュー権限が一極集中してボトルネックになる、などが連続したからです。

そのたびに「ルールを足す」というその場対応をしてきたら、いつのまにかチームの規律が3層に整理されていました。この記事は、その経緯と現在の形を書いてみたものです。

---

## 3層構造になった経緯

現在のガバナンスは、役割の異なる3種類のドキュメントで構成されています。

| 層 | ドキュメント | 役割 |
| --- | --- | --- |
| 上 | `AI_WORKFLOW.md` | 憲法本体（5原則・役割定義・基本ルール） |
| 中 | `CONSTITUTION.md` | 運用規範（条文単位の操作規約） |
| 下 | `state.json` / `HANDOFF.md` | ランタイム状態・申し送り |

上層ほど抽象的で改訂が稀、下層ほど具体的で頻繁に更新されます。最初からこう設計したわけではなく、衝突の種類が違うことに気づくたびに層が分かれていきました。

* 「役割の定義」と「日々の更新」が同じファイルに混ざると、上層の安定性が壊れる
* 「条文の編集」と「申し送りの記録」が混ざると、編集権限の設計ができない

層が分かれていなかった頃は、`HANDOFF.md` を見ても「これは規律なのか申し送りなのか」が判別できませんでした。

---

## 自律実行ルール — 誰が何を編集してよいか

CONSTITUTION.md の第2条には「自律実行ルール」というセクションがあります。エージェントが「確認なしに実行してよいこと」と「lead に判断を仰ぐべきこと」を明示的に分けたものです。

抜粋するとこんな雰囲気です。

```
2-1. 実行してよいこと（確認不要）
  - ファイルの読み取り
  - ローカルファイルの作成・編集
  - state.json / HANDOFF.md の更新
  - .ai/ 以下への調査結果の記録

2-2. 実行してはいけないこと（lead の判断を仰ぐ）
  - git commit / git push
  - docker build / docker push
  - fly deploy
  - 既存ファイルの削除
```

このルールがあると、subagent は判断不要で動けます。「これはやってよいことか」を毎回 LLM に推論させるより、確定的なリストの方が速いし、安全です。

## 衝突 #1：HANDOFF.md の書き込み競合（Issue #444）

最初の大きな衝突は `HANDOFF.md` の書き込み競合でした。複数のエージェントが同じファイルを同時に Edit して、後書きが前のを上書き消失させる事故が起きました。

原因は明確で、HANDOFF.md の編集権限を **lead と webmaster の両方** に渡していたからでした。「相談役の webmaster が状況を整理する」「実行役の lead が判断を記録する」というのが当初の役割分担だったのですが、結果として誰が最終責任者かが曖昧になりました。

解決策はシンプルでした。**HANDOFF.md と state.json の編集を lead 専任にする**。webmaster は GitHub Issue の操作だけに絞る。

```
変更前: webmaster → HANDOFF.md 管理・state.json 更新・Issue 操作
変更後: webmaster → GitHub Issue 操作のみ

変更前: lead → git commit・deploy・(委任中)
変更後: lead → git commit・deploy + HANDOFF.md / state.json 直接管理
```

これで競合は構造的に消えました。同時に「責任の所在が曖昧」という別の問題も同時に解消されました。

---

## 衝突 #2：レビュー権限の一極集中（Issue #550）

しばらく運用していると、別の問題が出てきました。

governance 関連の改訂 PR が増えるたびに、すべて lead が PR を作成・レビューする構造だったので、lead が本流の機能開発と governance 改訂の両方を直列処理することになっていました。lead がボトルネックになって、機能開発が止まる。

これも解決策は「権限の再分配」でした。

| ファイル | 編集権 |
| --- | --- |
| `AI_WORKFLOW.md`（憲法本体） | lead 専任 |
| `CONSTITUTION.md`（条文） | lead / PO / webmaster |
| `.claude/agents/*.md`（エージェント定義） | lead / PO / webmaster |
| `HANDOFF.md` / `state.json` | lead 専任 |

ポイントは、**「憲法本体（AI\_WORKFLOW.md）」だけは lead 専任で守った**ことです。ここを分散させると "competing source of truth"（同じ規律が複数箇所に書かれて、どれが正なのか判別できなくなる状態）が起きます。

条文（CONSTITUTION.md）の更新は分散させてよい。しかし「役割の定義」と「5原則」は単一情報源として守る。

これも一度経験したから書けたルールでした。

## 衝突 #3：Issue レビューの token コスト（Issue #514）

3つ目の衝突は、もう少し細かい話です。

lead は毎セッションで OPEN な Issue を全件取得して状況を把握していました。78件の Issue を毎回読むと、トークンコストが30K近くかかります。チリも積もれば、というやつです。

これも構造で解決しました。`lead-review` というラベルを新設して、lead の判定・監査が必要な Issue にだけラベルを付ける運用にしました。

```
ラベル付与権限: Owner / PO / webmaster（ラベル張り替えだけ）
ラベル除去権限: lead（対応完了後に自分で外す）
```

lead は `gh issue list --label lead-review` で5件以下に絞り込めるようになり、トークンコストは 30K → 3K になりました。

これは「規律」と呼ぶには地味な改善ですが、構造で問題を解くという同じ思想です。

## 「AIに憲法を書かせる」というパラドックス

ここまで書いてきた3層のドキュメントは、ほとんどを Claude Code 自身が書いています。Owner（僕）が最終判断するけれど、起案・改訂は AI エージェントです。

「自分たちを縛るルールを、自分たちで書く」という形になっています。

最初は違和感がありました。「規律は人間が書くもの」という直感があったので。でも実際にやってみると、AI が書いた方がうまくいく場面が多かった。

理由は2つあると思っています。

ひとつは、AI は自分が引っかかった問題を覚えていて、それを「次の自分」に伝えたい欲求が強い。「同じ失敗を繰り返したくない」というのは、人間より AI の方が真剣かもしれません。コンテキスト窓のリセットを毎セッション経験しているので。

もうひとつは、AI には「面倒くさい」がない。条文を細かく書き分けることへの抵抗感が、人間より低い。`AI_WORKFLOW.md` は今 600 行を超えていますが、AI エージェントはこれを毎セッション全文読み直して動いています。

ただし、無制限に増やすとそれはそれで破綻するので、「新条項追加は PO（人間が最終判断する役割）の承認が必須」というメタルールを入れています。これで「自家中毒的なルール肥大化」を抑える形にしています。

---

## ガバナンスは摩擦の記録である

3層の構造を作ってきた振り返りで、強く感じたことがひとつあります。

**規律が増えるたびに、何かが衝突した過去がある。**

`HANDOFF.md` の編集を lead 専任にしたのは、複数エージェントの上書き競合があったから。`AI_WORKFLOW.md` を競合源泉から守る決定をしたのは、競合する真実が同時に複数生まれるのを見たから。`lead-review` ラベルを作ったのは、トークンが溶けるのを見たから。

どれも、抽象的な「あるべき設計」から導いた規律ではなく、起きた事故への対応として積み重なったものです。

だからガバナンス文書は「禁止と許可のリスト」ではなく、**「過去の摩擦の記録」** だと思っています。「なぜこの規律があるのか」を辿ると、必ず具体的な衝突に行き着く。

ガバナンスが綺麗すぎて事故の痕跡が見えないチームは、たぶん経験が浅いか、書き手の手が入りすぎているかのどちらかなんじゃないか、と、自分のチームの泥臭い条文を読み返しながら考えています。
