## 1. 💬 プロンプトの書き方 ― 成果の8割はここで決まる

Claude Code で最も重要なのは「何をどう伝えるか」です。同じタスクでも、プロンプトの質で結果が劇的に変わります。

### ❌ 悪い例 vs ✅ 良い例

### 良いプロンプトの3要素

| 要素 | 説明 | 例 |
| --- | --- | --- |
| **🎯 具体的なゴール** | 何を達成したいか明確に | 「ユーザー登録APIにバリデーションを追加」 |
| **📎 コンテキスト** | 関連ファイル・エラー・スクショ | `@src/api/users.ts` を参照して |
| **✅ 検証手段** | 成功をどう確認するか | 「テストを実行してすべてパスさせて」 |

### コンテキストの渡し方

> 💡 **ポイント**: `@` を使ってファイルを直接参照すると、Claude がファイルを読む時間を省略でき、より正確な回答が得られます。

## 2. 📄 CLAUDE.md ― プロジェクトの取扱説明書

CLAUDE.md は、Claude がすべてのセッション開始時に読む特別なファイルです。いわば「新しいチームメンバーへの引き継ぎ書」のようなものです。

### CLAUDE.md の読み込み階層

### CLAUDE.md に書くべきこと

```
# プロジェクト概要
TypeScript + React のSPAアプリケーション
 
# ビルド・テストコマンド
- ビルド: `npm run build`
- テスト: `npm run test`
- Lint: `npm run lint`
 
# コードスタイル
- ES Modules (import/export) を使用、CommonJS (require) は不可
- インポートは可能な限り分割代入
- コンポーネントは関数コンポーネントで記述
 
# やってはいけないこと
- any 型の使用禁止
- console.log をプロダクションコードに残さない
- deprecated な API を使わない
```

### CLAUDE.md を肥大化させないためのルール

> ⚠️ **重要**: CLAUDE.md は **200行以下** を目標に。肥大化すると Claude が本当に重要な指示を見落とす原因になります。

### `/init` で始めよう

最初の CLAUDE.md は `/init` コマンドで自動生成できます：

```
# プロジェクトディレクトリで実行
claude
> /init
```

Claude がコードベースを分析して、ビルドシステム、テストフレームワーク、コードパターンを検出し、適切な CLAUDE.md を生成してくれます。

## 3. 🧠 コンテキスト管理 ― Claude の「記憶」を最適に保つ

Claude Code には約 200K トークンのコンテキストウィンドウがあります。これは Claude の「短期記憶」のようなもので、使い方を誤ると精度が下がります。

### コンテキストの仕組み

### コンテキストを管理する3つのコマンド

> 💡 **ベストプラクティス**: 関連のないタスクの間では必ず `/clear` を実行しましょう。コンテキストの60%を超えると精度が低下し始めるという報告があります。

## 4. 🗂️ 探索 → 計画 → 実装 ― 三段階アプローチ

大きなタスクをいきなり実装させるのではなく、段階を踏むことで品質が大幅に向上します。

### 三段階のフロー

### Plan Mode の使い方

Plan Mode は Claude に「読み取り専用」で調査・計画だけさせるモードです。

**Plan Mode の起動方法:**

```
# 新しいセッションを Plan Mode で開始
claude --permission-mode plan
 
# ヘッドレスモードで計画だけ取得
claude --permission-mode plan -p "認証システムの改善案を分析して"
```

> 💡 **いつ Plan Mode を使う？**
>
> * 複雑な機能のマルチファイル実装前
> * コードベースの探索・調査
> * リファクタリングの影響範囲の把握

## 5. ⚙️ 環境設定 ― Claude の「仕事場」を整える

### パーミッション設定

### フック ― 決定論的な自動化

フックは「この操作の前後に必ずこのスクリプトを実行する」という仕組みです。CLAUDE.md の指示と違い、**100%実行が保証**されます。

**フックの設定例:**

```
# Claude にフックを書いてもらう
> すべてのファイル編集後に eslint を実行するフックを書いて
> migrations フォルダへの書き込みをブロックするフックを書いて
```

### MCP（外部サービス連携）

> ⚠️ **注意**: MCP サーバーごとにツール定義がコンテキストに追加されます。`/mcp` でサーバーごとのコストを確認し、**必要なものだけ接続**しましょう。

## 6. 🤖 サブエージェント ― タスクの委譲で効率アップ

サブエージェントは、メインの Claude とは独立した「コンテキストウィンドウ」を持つ子プロセスです。

### サブエージェントの仕組み

### なぜサブエージェントが有効なのか

### カスタムサブエージェントの作り方

`.claude/agents/` にマークダウンファイルを作成します：

```
# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---
You are a code reviewer. Analyze the code and provide
specific, actionable feedback on quality, security,
and best practices.
```

**呼び出し方:**

```
# 自然言語で呼び出す
> code-reviewer サブエージェントで最近の変更をレビューして
 
# @メンションで確実に呼び出す
> @code-reviewer この PR をレビューして
```

## 7. 🔄 ワークフロー統合 ― Git & CI/CD との連携

### Git ワークフロー

### カスタムコマンド（スラッシュコマンド）

チーム共通のワークフローをコマンド化できます：

```
# .claude/commands/fix-issue.md
---
name: fix-issue
description: GitHub Issue を修正して PR を作成
argument-hint: [issue-number]
disable-model-invocation: true
---
1. `gh issue view` で Issue の詳細を取得
2. 問題を理解する
3. コードベースで関連ファイルを検索
4. 必要な変更を実装
5. テストを書いて実行し、修正を検証
6. Lint と型チェックをパス
7. 説明的なコミットメッセージを作成
8. プッシュして PR を作成
```

## 🎯 すぐに始められるチェックリスト

```
□ claude を起動して /init で CLAUDE.md を生成する
□ CLAUDE.md にビルド・テストコマンドを追加する
□ コーディング規約を CLAUDE.md に記載する（200行以内）
□ タスク間で /clear を使う習慣をつける
□ プロンプトに「具体的なゴール + コンテキスト + 検証手段」を含める
□ よく使うコマンド（lint, test）をパーミッションホワイトリストに追加する
□ 大きなタスクは Plan Mode で計画してから実装する
□ 長いセッションでは /compact で手動圧縮する
□ チーム共通のワークフローをカスタムコマンド化する
□ 調査タスクはサブエージェントに委譲する
```

## おわりに

Claude Code のベストプラクティスは「固定のルール」ではなく「出発点」です。公式ドキュメントでも以下のように述べられています：

> 何が機能するかに注意を払いましょう。Claude が素晴らしい出力を生成するとき、あなたが何をしたかに注意してください。時間をかけて、ガイドが捉えることができない直感を開発してください。

自分のプロジェクトや開発スタイルに合わせて、これらのプラクティスをカスタマイズしていきましょう。
