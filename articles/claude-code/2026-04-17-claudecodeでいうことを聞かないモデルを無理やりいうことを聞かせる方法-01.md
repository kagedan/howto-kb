冒頭におそらく **hooks** だろうなと思った方は大正解です。学びがなさそうであれば読まなくても大丈夫です。

## 背景

普段は **GLM5.1** というモデルを ClaudeCode で利用しています。ただこのモデルは Claude Sonnet や Opus に比べてユーザーの要望を無視しやすい傾向があります（コンテキストの長さが短いのが原因かもしれません）。

特に以下の2点を守らせたいのに、何度言っても無視されることがありました。

* **main ブランチではコミットを積まない**
* **ビルドができない状態では完了報告をしてほしくない**

口頭の指示だけでは守ってもらえないので、**仕組みとしてブロックする**ことにしました。

## ClaudeCode の hooks とは

ClaudeCode には、ツール実行や応答のタイミングで任意のシェルスクリプトを実行できる **hooks** という機能があります。

<https://code.claude.com/docs/ja/hooks>

hooks の主なイベント:

| イベント | タイミング |
| --- | --- |
| `PreToolUse` | ツールを実行する直前 |
| `PostToolUse` | ツールを実行した直後 |
| `Stop` | Claude が応答を終了するとき |

`PreToolUse` でスクリプトが `exit 2` を返すと、そのツール実行をブロックして Claude にエラーメッセージをフィードバックできます。

## 設定方法

### 1. hooks の設定ファイルを作成する

`.claude/settings.local.json` を作成して以下の内容を記述します。

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/block-main-commit.sh"
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/verify-task-stop.sh"
          }
        ]
      }
    ]
  }
}
```

* **PreToolUse（Bash）**: Bash コマンドが実行される直前に `block-main-commit.sh` を実行
* **Stop**: Claude が応答を完了しようとするたびに `verify-task-stop.sh` を実行

### 2. hook スクリプトを作成する

#### block-main-commit.sh（main ブランチへのコミットをブロック）

`.claude/hooks/block-main-commit.sh` を作成します。

```
#!/bin/bash

# stdinからJSON入力を読み込む
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')
CWD=$(echo "$INPUT" | jq -r '.cwd')

# 作業ディレクトリに移動
cd "$CWD"

# 現在のブランチを取得
BRANCH=$(git branch --show-current)

# mainブランチかつ、コマンドに"git commit"が含まれる場合はブロックする
if [[ "$BRANCH" == "main" && "$COMMAND" == *"git commit"* ]]; then
  # stderrに書き込んだメッセージはClaudeにフィードバックされます
  echo "エラー: mainブランチへの直接のコミットは許可されていません。" >&2
  exit 2
fi

# 問題なければ終了コード0で許可して続行
exit 0
```

`PreToolUse` hook に渡される JSON の stdin から実行しようとしていたコマンドと作業ディレクトリを取得しています。main ブランチで `git commit` しようとした場合に `exit 2` を返すことでブロックし、stderr に書いたメッセージが Claude へのフィードバックとして伝わります。

#### verify-task-stop.sh（ビルド失敗時に完了をブロック）

`.claude/hooks/verify-task-stop.sh` を作成します。

```
#!/bin/bash

# stdinからJSON入力を読み込む
INPUT=$(cat)
CWD=$(echo "$INPUT" | jq -r '.cwd // empty')

# 作業ディレクトリが取得できない場合はスキップ
if [ -z "$CWD" ] || [ ! -d "$CWD" ]; then
  exit 0
fi

cd "$CWD"

# ビルドコマンドが存在するプロジェクトかチェック（例: Gradle プロジェクト）
if [ -f "./gradlew" ]; then
  # ビルドを実行して結果を確認
  ./gradlew build -q 2>/dev/null
  if [ $? -ne 0 ]; then
    echo "エラー: ビルドが失敗しています。ビルドエラーを修正してから完了してください。" >&2
    exit 2
  fi
fi

exit 0
```

`Stop` イベントは Claude が応答を終了しようとするたびに発火します。`exit 2` を返すと Claude に処理を続けさせることができます。

### 3. スクリプトに実行権限を付与する

```
chmod +x .claude/hooks/block-main-commit.sh
chmod +x .claude/hooks/verify-task-stop.sh
```

## 動作確認方法

ClaudeCode を起動したあと（**設定変更後は再起動が必要**）、以下を試してみてください。

**ケース①: main ブランチでコミット**

→ Claude が `git commit` を実行しようとした瞬間にブロックされ、「mainブランチへの直接のコミットは許可されていません」というエラーが Claude へフィードバックされます。

**ケース②: ビルドが壊れた状態で完了しようとする**

```
App.kt に存在しないクラスのインポートを追加してください
```

→ タスク完了を宣言しようとしたタイミングでビルドチェックが走り、ビルド失敗を検知すると Claude に修正を促します。

## つまづいたポイント

ドキュメントを見て最初は `TaskCompleted` イベントを使おうとしましたが、何度試してもうまく動作しませんでした。

<https://code.claude.com/docs/ja/hooks>

結局 `Stop` イベントを使うことで期待通りに動作しました。ドキュメントのイベント名と実際の挙動が一致しない場合があるので、動かないときは別のイベントを試してみると良いと思います。

## まとめ

ClaudeCode の hooks を使うことで、モデルが指示を無視しても**仕組みとして強制する**ことができます。

* `PreToolUse` + `exit 2` → ツール実行をブロックしてエラーを Claude にフィードバック
* `Stop` + `exit 2` → 完了を阻止して Claude に修正を促す

「プロンプトで何度言っても守ってくれない」という悩みは hooks で解決できるかもしれません。ぜひ試してみてください。
