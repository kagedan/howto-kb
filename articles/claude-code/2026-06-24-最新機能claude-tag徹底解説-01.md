Claude Tagは、SlackでClaudeを呼べる新機能として見ると危険です。

本当に決めるべきことは、AIがどのチャンネルを読み、どのツールを使い、どこまで仕事を任され、いくら使ってよいかです。

この記事では、2026年6月23日にAnthropicが発表したClaude Tagを、社内AI導入担当の判断材料へ落とし込みます。

読み終わると、Claude Tagを試す前に決めるべきチャンネル、権限、予算、ログ、PoC業務が分かります。

## Claude Tagで変わるのはSlackではなく権限設計

![](https://assets.st-note.com/img/1782260756-xOfgomp4zMA0Dr2kHaBTqItP.png?width=1200)

Claude Tagの本質は、SlackにAIを入れることではありません。

チームの会話、社内ツール、データ、コードベースへ、どの範囲でClaudeを参加させるかを決めることです。

AnthropicはClaude Tagを、チームがClaudeと働く新しい方法として発表しました。

開始地点はSlackです。

管理者が選んだチャンネル、ツール、データ、コードベースへClaudeを接続し、メンバーがチャンネル内で **`@Claude`** を呼んで仕事を委任します。

ここで重要なのは、導入の主語が個人ではなく組織になることです。

個人がチャットでClaudeへ相談する使い方とは、考えるべき項目が変わります。

## 目次

## Claude Tagでできること

![](https://assets.st-note.com/img/1782260778-r9iCaVKLXOUufekPJj5x6l2m.png?width=1200)

Claude Tagは、Slack上でClaudeへ仕事を委任する仕組みです。

Anthropicの説明では、Claudeは依頼を段階へ分解し、アクセス権のあるツールを使い、完了後にSlackスレッドへ結果を返します。

特徴は4つあります。

* **複数人が同じチャンネル内でClaudeとやり取りできる**
* **チャンネルの文脈をもとに、説明のやり直しを減らせる**
* **ambient behaviorを有効にすると、Claudeが必要そうな情報を自発的に知らせる**
* **非同期でタスクを進め、将来の作業を計画できる**

Claude Tagは、Claude CodeやCoworkを使ったことがある人には近い体験です。

ただし違いは、作業場所が個人のチャットや端末ではなく、チームのSlackチャンネルになる点です。

## 導入判断は便利さより先に対象部署を決める

![](https://assets.st-note.com/img/1782260792-Z6fVMJCYThytzb1S7Ev9Axjm.png?width=1200)

Claude Tagは、**Claude Enterprise / Team顧客向け**にbeta提供されています。

そのため、最初に見るべき点は「全員が使えるか」ではなく**「どの部署で試すか」**です。

Anthropicは、Claude TagがOpus 4.8で動作すると説明しています。

Anthropicは、社内ではプロダクトチームのコードの65%がClaude Tagの内部版で作成されていると説明しています。

同時に、用途は開発だけではありません。

プロダクト指標の確認、サポートチケット処理、難しいバグの原因調査にも使っていると説明されています。

ただし、この数字をそのまま自社へ当てはめるのは危険です。

Anthropic社内の利用実績であり、すべての会社で同じ成果が出る根拠ではありません。

自社で最初に見るべき候補は3つです。

* **依頼がSlackへ集まる部署**
* **ツールとデータの権限範囲を絞りやすい部署**
* **完了条件を人間が確認しやすい業務**

最初のPoCは、全社横断より1部署1チャンネルです。

便利そうだから広げるのではなく、説明できる範囲で始めます。

## 権限と記憶はチャンネル単位で考える

![](https://assets.st-note.com/img/1782260825-13xCsrmP9WwcBdR5KMiaYoJD.png?width=1200)

Claude Tagの導入で一番重要なのは、チャンネル単位の権限設計です。

Anthropicは、管理者がどのツールと情報へアクセスできるか、どのチャンネルで使うかを指定すると説明しています。

メモリもスコープが重要です。

Anthropicの説明では、用途ごとに別々のClaude identityを作る考え方が示されています。

たとえば営業用に設定されたClaudeのメモリが、エンジニアリング用へ引き継がれない設計です。

さらに、Claudeは許可された場合、他のSlackチャンネルやデータソースから学習できます。

ただし、Anthropicは、非公開チャンネルの内容を報告しないと説明しています。

導入担当が先に決めることは5つあります。

* **Claudeを入れるチャンネル**
* **読ませるデータソース**
* **使わせるツール**
* **メモリを分ける単位**
* **依頼できない仕事**

ここを決めないまま始めると、あとから説明が難しくなります。

AIの便利さではなく、アクセス境界から設計してください。

## 最初のテストは非公開チャンネルで始める

![](https://assets.st-note.com/img/1782260844-wRfO82B3AYzWC9UkPMhqFice.png?width=1200)

Anthropicは、Claude Tagの開始手順として、Slack workspaceとの接続、ツールアクセス付与、月次支出上限設定、非公開チャンネルでのテストを挙げています。

この順番は、そのまま社内PoCの順番として使えます。

最初のチャンネルは、いきなり本番部門へ入れない方が安全です。

まず、導入担当、業務担当、レビュー担当だけがいるテスト用チャンネルを作ります。

チャンネル名は分かりやすくします。

```
#ai-claude-tag-test
目的: Claude Tagの小規模検証
対象業務: 社内問い合わせ整理、Slackスレッド要約、調査依頼の下書き
利用者: 導入担当、対象部署リーダー、レビュー担当
禁止事項: 機密データ投入、顧客個人情報投入、外部送信
完了条件: 人間が結果を確認し、採用可否を記録する
```

最初の1週間は、便利さより記録を優先します。

何を頼んだか。  
どの出力が使えたか。  
人間の修正がどれだけ必要だったか。  
予算をどれだけ使ったか。

この記録がないと、次に広げる判断ができません。

## 任せる業務は3種類に絞る

![](https://assets.st-note.com/img/1782260859-7f5wi3Hnq6ICQbmuVL81RhlD.png?width=1200)

Claude Tagへ最初に任せる業務は、広げすぎない方がよいです。

Claude Tagは非同期で動き、ツールへ接続し、チャンネル文脈を使えるため、何でも任せたくなります。

ただし、PoCで見るべきは万能性ではありません。

自社の業務で再現できるかです。

最初は3種類で十分。

* **Slackスレッドの要約と未決事項整理**
* **社内ドキュメントやチャンネル文脈を使った調査**
* **GitHubやチケットに接続した軽い原因調査**

ポイントは、出力を人間が短時間で評価できることです。

営業メール自動送信、顧客対応の自動返信、本番コード変更などは、最初のPoCには重すぎます。

Claude Tagは「AIに任せる」機能ではありません。  
**AIに任せた結果を、人間が確認し、運用として残す仕組み**です。

## 失敗する導入は予算とログを後回しにする

![](https://assets.st-note.com/img/1782260881-s1c6EwznAqGyQURClmW4PM0k.png?width=1200)

Claude Tag導入で後回しにしてはいけないのは、予算とログです。

Anthropicは、管理者が組織全体と個別チャンネルのtoken spend上限を設定でき、Claudeが行ったことと依頼者のログを見られると説明しています。

ここは導入担当にとって一番重要です。

長時間実行型のAIは、便利なほど使われます。

使われるほど、コストと説明責任が増えます。

導入前に最低限決める項目は6つです。

* **組織全体の上限**
* **チャンネルごとの上限**
* **利用者ごとの運用ルール**
* **ログを見る担当者**
* **出力を採用する責任者**
* **問題発生時に止める条件**

Claude Tagは、既存のClaude in Slackアプリを置き換える位置づけです。

Anthropicは、管理者が30日以内に移行を選べると説明しています。

すでにClaude in Slackを使っている組織ほど、移行前にログ、予算、権限、チャンネルを棚卸ししてください。

「今まで使っていたから大丈夫」は、導入判断ではありません。

## 今日やるチェックリスト

![](https://assets.st-note.com/img/1782260898-eUAprPyKL8QwT6CERzb9duhf.png?width=1200)

Claude Tagは、発表を読んで終わるニュースではありません。

社内AI導入担当にとっては、Slack、権限、ツール、記憶、予算、ログをどう組み合わせるかの設計テーマです。

今日やることは、全社展開ではありません。  
テスト条件を1枚にすることです。

Claude Tagの価値は、SlackにClaudeを入れた瞬間には分かりません。

価値が分かるのは、どの仕事を任せ、どの権限で動かし、どの結果を人間が採用するかを決めた後です。

この小ささで始める方が、社内導入は続きます。
