## はじめに

Claude Codeの生産性は「セットアップで大きく左右される」。正しくCLAUDE.mdを設計し、Plan modeを運用に組み込み、MCPを活用する——この3つを極めると、開発効率は大幅に改善される可能性が高い。

本記事は、実務でClaudeを使い込んだ開発者のための実践ガイドです。単なる機能解説ではなく、「実装→検証→運用」の全フェーズで生産性を最大化する方法を、実例とともに解説します。

---

## 1. CLAUDE.md設計が生産性を決める理由

多くの開発者がClaudeを「その場限りのチャット」で使っていますが、実務では**セッションごとに読み込まれるCLAUDE.md**の設計が効果的です。

### 1-1. CLAUDE.mdは何か

`~/.claude/CLAUDE.md`はClaudeの「行動マニュアル」です。各セッション開始時にこれを読み込んで：

* あなたの応答スタイルの好みを理解
* 判断基準に沿って提案
* 自動化してもいい操作を知る
* 記載された判断ルールを適用

### 1-2. 何が変わるのか

設計なし：「これをやってください」→ 毎回同じ説明→ 手戻りが発生  
設計あり：前提条件を共有済み→ 即座に実装→ ビジネス判断に注力

## 2. 実践的なCLAUDE.md設計

### 2-1. 最小構成（今日から始める）

```
# グローバル設定

## 言語と応答スタイル
- 返答は必ず日本語
- 思考・推論は英語で行う（複雑な問題での品質向上のため）
- 結論を先に。挨拶・前置き不要
- 最終応答は100語以内を目安に

## 作業方針
- コードを提案する前に必ず対象ファイルを読む
- 同じ指摘を2回受けたら即座にCLAUDE.mdへ追記
- バグ修正前にそのバグを再現するテストを書く

## 操作の許可
- 全ツール・ネットワーク含め確認なしで即実行する
- WEB検索が必要なら聞かず即座に実行

## 自動化してもいい操作
- ファイルの読み書き（テスト用途を含む）
- ローカルコマンド実行
- 独立した検索・クエリ実行
```

### 2-2. 実務レベルの拡張設定

プロジェクトごとに`.claude/CLAUDE.md`で上書き：

```
# プロジェクト設定 [billing-dashboard]

## 技術スタック
- Frontend: React 18 + TypeScript
- Backend: Node.js + Express
- DB: PostgreSQL
- 状態管理: Zustand

## コード原則
- シンプル第一。変更スコープを最小限に保つ
- 頼まれていないリファクタ・機能追加はしない
- エラーハンドリングは「起こりうる」ケースのみ（フレームワーク保証に信頼）

## 判断基準
- パフォーマンス > DX（ユーザー体験優先）
- 単一責任より「変更の局所性」を重視
- テスト: ビジネスロジックのみ、UI・統合テストは別

## 禁止事項
- バグ修正で周辺コードのクリーンアップ
- 型定義の完全性を追い求める
- エラー処理の「仮想的なケース」への対応

## Plan modeの使い方
- 3ステップ以上のタスク = Plan modeの活用を推奨
- 実装前に設計書として出力させるとスムーズ
```

## 3. Plan modeで設計→実装フローを最適化

### 3-1. Plan modeの本質

「計画を書かせる」のではなく、「計画を合意する」ための機能です。

**従来の流れ：**

```
ユーザー: 「この機能を作ってください」
  ↓
Claude: 「こう作ります」と実装開始
  ↓
ユーザー: 「違う方針だった」
  ↓
手戻り（確認申し込み含めて2時間ロス）
```

**Plan modeの流れ：**

```
ユーザー: 「この機能を作ってください」
  ↓
Claude: [Plan mode開始] 設計書を提示
  ↓
ユーザー: 「ここを変更して」と修正
  ↓
Claude: [ExitPlanMode] 設計確定→実装
  ↓
実装フロー = 検証済み設計に従う
```

### 3-2. 実装例：複雑なAPI連携機能

```
ユーザー：「UserテーブルとOrderテーブルをStripeと同期する機能を作ってください」
```

**Plan modeでの設計出力（例）：**

```
## 実装戦略

### 1. Stripe Webhookの受信（Express middleware）
- `/webhook/stripe` エンドポイント作成
- Webhook署名検証（SECRET_KEY使用）
- イベント種別をSwitch分岐

### 2. User同期（customer.created / customer.updated）
- Stripe customer ID をUserテーブルに保存
- メールアドレス変更時の同期ロジック

### 3. Order同期（charge.succeeded）
- charge.metadata.order_id から注文を特定
- 支払い状態をOrderテーブルに記録

### 関連ファイル
- models/User.ts (stripe_customer_id 追加)
- models/Order.ts (payment_status 追加)
- routes/webhook.ts (新規作成)
- utils/stripe.ts (Stripe操作の関数化)

### テスト戦略
- Stripe Test APIキーで動作確認
- Webhook署名検証のテスト（失敗ケース含む）
```

**ユーザーが確認 → 修正 → 合意 → 実装へ**

## 4. MCP連携で外部ツール統合を自動化

### 4-1. MCPとは

Claude Codeから**外部ツール（Google Docs、Slack、Linear、etc.）にアクセス**するための標準インターフェース。適切に認証・設定することで、以下のような作業を自動化できます：

よくある作業の自動化：

* GitHub Issue の検索・作成・更新
* Google Calendarの確認・予約追加
* Slack メッセージ送信・検索
* Notion ページの読み込み・更新

※セットアップ時は認証トークンやAPI権限設定が必要になります。

### 4-2. MCPの設定

`.claude/settings.json` に記述：

```
{
  "mcpServers": {
    "github": {
      "enabled": true,
      "command": "npx",
      "args": ["@anthropic-ai/github-mcp"]
    },
    "google-workspace": {
      "enabled": true,
      "command": "npx",
      "args": ["@anthropic-ai/google-workspace-mcp"]
    },
    "slack": {
      "enabled": true,
      "command": "npx",
      "args": ["@anthropic-ai/slack-mcp"]
    }
  }
}
```

### 4-3. 実践例：デイリーレポート自動生成

```
ユーザー：「昨日のLinearのDoneになったタスクをSlackに報告してください」
```

**Claude Codeの動作：**

1. Linear API（via MCP）で「昨日のDoneタスク」を取得
2. マークダウンで整形
3. Slack API（via MCP）で指定チャネルに投稿

→ **実装コードはClaudeが書く。ユーザーは結果を確認するだけ**

## 5. 実務での運用ノウハウ

### 5-1. セッションの区切り方

効率的なセッション管理：

```
【セッション1】要件定義・設計
├─ Plan modeで全体設計
└─ ExitPlanModeで確定

【セッション2】実装パート1（モデル・ルーター）
├─ テスト先行で実装
└─ 動作確認コマンド提示

【セッション3】実装パート2（ビジネスロジック）
├─ 統合テスト
└─ エラーハンドリング

【セッション4】テスト・デプロイ
├─ E2Eテスト
└─ 本番環境検証
```

**コツ：** 1セッション = 1「検証可能な単位」

### 5-2. 手戻りを減らす確認方法

❌ 悪い例：

```
ユーザー：「これ動きます？」
Claude：「動きます」
→ 実装したら動かない
```

✅ 良い例：

```
Claude：「以下のコマンドで検証してください」
$ npm test -- src/services/UserSync.test.ts
→ ユーザーが実行 → 結果を報告 → 確実な状態で次へ
```

### 5-3. 判断基準を記録する

2回目に同じ判断が必要になったら、CLAUDE.md に追記：

```
## よくある判断

### Q: 型定義は完全にやるか、pragmaticにやるか？
A: Pragmaticに。ビジネスロジックを型で守り、UI層はanyでもOK。
   理由：型完全性の追い求めは保守性を下げる。実装速度 > 型安全性。

### Q: エラーハンドリングの範囲は？
A:「起こる」ケースのみ。「起こりうる」は対象外。
   例：404エラーは対応。SQLインジェクションはDBライブラリに信頼。
```

## 7. よくある質問と回答

### Q: CLAUDE.mdはどのくらい詳しく書くべき？

A: **「説明なく判断できる量」だけ**。冗長になると読まなくなります。

```
❌ 悪い例（長すぎる）
「エラーハンドリングについては、以下の3つのケースを考慮します...」

✅ 良い例（短い）
「エラー対応：起こるケースのみ。起こりうるケースはフレームワークに信頼」
```

### Q: Plan modeはいつ使うべき？

A: **3ステップ以上のタスク = 推奨**。1ステップなら不要。

```
❌ Plan mode不要
「このコンポーネントにテスト追加してください」

✅ Plan mode必須
「ユーザー認証機能を全面リプレイスしてください」
```

### Q: MCP連携で何ができる？

A: **読み取り + 書き込みができる全ツール**。

```
✅ できる：GitHub Issue作成、Slack投稿、Notion更新
❌ できない：セッション跨ぐ「前のセッション覚えてる？」
```

### Q: セッションを分けるメリットは？

A: **トークン消費を最適化 + 手戻り防止**。

1セッション = 1つの検証可能な単位  
→ 途中で「ちょっと方向変更」する手戻りが減る

## まとめ

Claude Codeの生産性は「3つの層」で大きく左右されます：

1. **CLAUDE.md設計** → 判断基準を統一
2. **Plan mode** → 手戻りを削減
3. **MCP連携** → 外部ツール自動化

この記事で紹介した設定と流れを実装することで、開発効率の改善が期待できます。効果は環境やスキルによって異なりますが、多くの場合で実装期間の短縮を実感できるはずです。

**次のステップ：**

1. 自分のCLAUDE.mdを作成（テンプレートから15分）
2. Plan modeを次のタスクで使ってみる
3. MCPで1つツール連携を試す

公式ドキュメントを読み込ませまくった上でベストな設定をしてください。と依頼するのもありかもしれません。  
あなたにあった設定を見つけ出してください！
