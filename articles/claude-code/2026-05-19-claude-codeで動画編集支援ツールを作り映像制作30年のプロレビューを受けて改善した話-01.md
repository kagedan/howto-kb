Claude Codeを使って、動画編集支援ツール CCT を作りました。

CCTは、Premiere Proで動画編集をしている人向けに、文字起こし・テキスト整形・字幕作業まわりの負担を減らすことを目的にしたAI支援ツールです。

最初は、Claude Codeを使えばどこまで実用的なツールを作れるのかを試すところから始まりました。

ただ、作ってみて分かったのは、AIでツールを実装することと、現場で使えるツールにすることはまったく別だということです。

初期版を作ったあと、映像制作歴30年のプロにレビューしてもらいました。

そこで見えた課題をもとに大幅に改善し、さらに既存ツールであるVrewとも比較しました。

この記事では、CCTの宣伝というより、以下のような開発・改善プロセスをまとめます。

レビュー後に改善した考え方  
Vrew比較で分かった評価軸  
個人開発AIツールを実用に近づけるために必要だったこと

背景：動画編集には「編集前後の地味な作業」が多い

動画編集の作業は、カット編集、テロップ、BGM、色調整、演出などが目立ちます。

ただ、実際にはその前後にある地味な作業もかなり重いです。

たとえば、

動画内容の把握  
音声の文字起こし  
話し言葉の整形  
字幕として読みやすい形への変換  
Premiere Proに持っていくための下準備  
修正しやすいテキスト形式への整理

などです。

特に、インタビュー動画、解説動画、講座動画、YouTube動画では、話し言葉をどう扱うかが重要になります。

単純な文字起こしだけでは、字幕としては読みにくい。

一方で、整形しすぎると話者のニュアンスが消える。

このあたりの調整は、編集者にとって意外と時間がかかる部分です。

CCTでは、まずこの「文字起こし後の処理」や「Premiere Pro編集前の下準備」をAIで支援できないかと考えました。

作ったもの：CCTの概要

CCTは、動画編集者向けのAI支援ツールです。

主な目的は、動画編集における以下のような作業を楽にすることです。

文字起こし結果の整理  
字幕向けのテキスト整形  
長尺動画の内容把握  
編集で使いやすい形式への変換  
Premiere Proでの作業につながる下準備

一般的な文字起こしツールというより、Premiere Proで編集する人のワークフローに寄せたツールを目指しています。

ここで意識したのは、単に「AIで文字起こしできます」というツールにしないことです。

既存ツールはすでに多くあります。

Vrewのような有名ツールもあります。

そこでCCTでは、より狭く、Premiere Pro編集者の作業フローに寄せることを意識しました。

Claude Codeで作って良かったこと

Claude Codeを使って良かった点は、アイデアから実装までの距離がかなり短くなったことです。

従来であれば、ちょっとしたツールを作る場合でも、設計、実装、UI、細かい修正にかなり時間がかかります。

Claude Codeを使うと、作りたいものの方向性を言語化しながら、機能を少しずつ形にできます。

特に良かったのは、以下のような点です。

小さな機能から作り始めやすい  
仕様変更を会話しながら反映しやすい  
UIや処理の改善を反復しやすい  
実装の細かい詰まりを相談しながら進められる  
個人開発でもプロトタイプを作る速度が上がる

CCTも、最初から完成形が見えていたわけではありません。

まずは「こういう処理ができると便利そう」という小さな機能から始めました。

そこから、実際の編集フローを想像しながら、機能や出力形式を調整していきました。

Claude Codeは、こうした試行錯誤と相性が良いと感じました。

ただし、AIに任せるだけでは良いツールにはならない

一方で、Claude Codeを使っていて強く感じたのは、AIに任せるだけでは良いツールにはならないということです。

AIはコードを書いてくれます。

UIも作ってくれます。

処理も提案してくれます。

でも、最終的に重要なのは、作る側がどれだけ具体的に課題を理解しているかです。

たとえば、

誰が使うのか  
どの作業を短縮したいのか  
どのタイミングで使うのか  
出力結果は次に何へ使われるのか  
ユーザーはどこで迷うのか  
どの形式なら編集作業に戻しやすいのか

このあたりを曖昧にしたままだと、ツールは「それっぽいけど使いにくいもの」になりやすいです。

特に動画編集支援ツールの場合、単体で完結する便利機能よりも、既存の編集ワークフローにどう入るかが重要です。

ここを考えないと、デモでは便利に見えても、現場では使われないツールになります。

初期版で見えてきた課題

初期版のCCTは、ある程度「できること」は形になりました。

ただ、自分で使っているだけでは、課題が見えにくい部分もありました。

個人開発では、作った本人が仕様を理解しすぎています。

そのため、初見ユーザーがどこで迷うのかに気づきにくいです。

また、自分では自然だと思っている操作も、他の人にとっては分かりにくいことがあります。

初期版で課題になりやすいと感じたのは、以下のような点です。

操作の流れが初見で分かりやすいか  
出力されたテキストをそのまま編集に使えるか  
修正がしやすいか  
余計なクリックや手戻りがないか  
Premiere Proでの作業につながる形になっているか  
既存ツールではなくCCTを使う理由があるか

特に最後の「CCTを使う理由」は重要でした。

既存ツールがある領域では、単に似た機能を作るだけでは弱いです。

なぜCCTを使うのか。

どの作業においてCCTの方が使いやすいのか。

ここを明確にする必要がありました。

プロレビューで見えたこと

そこで、映像制作歴30年のプロにCCTをレビューしてもらいました。

レビュー動画はこちらです。

▶ 映像制作歴30年のプロによるCCT本音レビュー

プロに見てもらって分かったのは、AIツールに必要なのは「機能の多さ」ではないということです。

むしろ大事なのは、現場で使うときの摩擦が少ないことでした。

たとえば、

使い始めるまでに迷わない  
出力結果がすぐ次の作業に使える  
修正や再調整がしやすい  
既存の編集フローを邪魔しない  
何のために使うツールなのかが明確

こうした点です。

開発者視点では、つい「何ができるか」に目が行きます。

しかし、実務で使う人にとっては、「それを使うことで作業が本当に楽になるか」の方が重要です。

レビューを通じて、CCTは「AIで何ができるか」から「編集者の作業をどう減らすか」へ考え方を変える必要があると感じました。

改善方針：機能追加より摩擦を減らす

レビュー後の改善で意識したのは、機能を増やすことではありません。

優先したのは、使うときの摩擦を減らすことです。

AIツールは、機能を追加しようと思えばいくらでも追加できます。

しかし、多機能になるほど分かりにくくなることもあります。

特に動画編集者向けのツールでは、ツールを触ること自体が目的ではありません。

目的は、編集作業を早く、楽に、分かりやすく進めることです。

そのため、改善では以下のような方向を意識しました。

出力を編集で使いやすい形に近づける  
文字起こし後の整形を扱いやすくする  
余計な手戻りを減らす  
初見でも流れが分かりやすいUIにする  
Premiere Proでの作業につながりやすくする  
CCTを使う理由が伝わるようにする

この改善を通して、初期版よりもCCTの方向性がかなり明確になりました。

Vrewと比較して分かった評価軸

改善後のCCTは、Vrewとも比較しました。

比較動画はこちらです。

▶ 改善後のCCTとVrewを比較した動画  
<https://youtu.be/7317p4AKC1g>

Vrewは、文字起こしや字幕作成の領域では非常に有名なツールです。

完成度も高く、多くのユーザーが使っています。

そのため、CCTを考えるうえでVrewとの比較は避けられません。

ただし、比較して感じたのは、単純に「どちらが上か」では語れないということです。

AI文字起こし・字幕支援ツールは、以下のような複数の軸で見る必要があります。

文字起こし精度  
整形のしやすさ  
字幕としての読みやすさ  
出力形式の扱いやすさ  
Premiere Proとの相性  
修正作業のしやすさ  
作業時間の短縮効果  
初見での分かりやすさ  
既存ワークフローへの入りやすさ

文字起こし精度だけが高くても、出力結果を編集に使いにくければ作業は楽になりません。

逆に、多少の修正が必要でも、編集フローにうまく入るなら実務では使いやすい場合があります。

この比較を通じて、CCTが目指すべき方向は「Vrewの完全な代替」ではなく、「Premiere Pro編集者向けに特化した選択肢」だと整理できました。

個人開発ツールが大きな既存ツールと戦うには

Vrewのような既存ツールと同じ土俵で、機能数や安定感だけで戦うのは簡単ではありません。

大きなツールには、ユーザー数、開発リソース、実績、完成度があります。

個人開発ツールが同じ方向で勝とうとすると、かなり厳しいです。

一方で、個人開発だからこそできることもあります。

それは、狭いターゲットに深く寄せることです。

CCTの場合は、Premiere Proを使う編集者の作業フローに寄せることを重視しています。

万人向けの字幕ツールではなく、Premiere Pro編集者が文字起こし・整形・字幕まわりの作業を減らすためのツールとして作る。

このように絞ることで、既存ツールとは違う価値を出せる可能性があります。

個人開発では、ターゲットを広げすぎるより、最初は狭く刺す方が良いと感じました。

Claude Codeで作るときに重要だったこと

今回の開発で、Claude Codeを使ううえで重要だと感じたのは、以下の点です。

1. 先にユーザーの作業フローを言語化する

AIに「便利なツールを作って」と頼んでも、便利なツールにはなりません。

まず必要なのは、ユーザーがどんな流れで作業しているのかを整理することです。

CCTの場合は、Premiere Pro編集者が文字起こしや字幕整形をどう扱うのかを考える必要がありました。

2. 出力結果の使われ方を決める

AIツールでは、出力が非常に重要です。

何を出すかだけでなく、その出力が次にどこで使われるのかを決める必要があります。

出力が良くても、次の作業に使いにくければ意味がありません。

3. 最初から多機能にしすぎない

Claude Codeを使うと、機能追加が比較的簡単です。

そのため、つい機能を増やしたくなります。

しかし、最初から多機能にすると、ツールの目的がぼやけやすいです。

まずは、誰のどの作業を短縮するのかに集中した方が良いと感じました。

4. 第三者レビューを早めに入れる

自分だけで使っていると、問題点に気づきにくいです。

特に、実務向けツールでは第三者レビューが重要です。

CCTの場合、プロレビューを入れたことで、改善すべき方向が一気に明確になりました。

5. 既存ツールとの比較で立ち位置を決める

既存ツールがある領域では、比較は避けられません。

ただし、比較の目的は勝ち負けを決めることではありません。

自分のツールがどこで価値を出すのかを明確にするために比較するべきだと感じました。

CCTが向いている人

現時点でCCTが向いているのは、以下のような人です。

Premiere Proを使って動画編集している人  
文字起こしや字幕整形に時間を取られている人  
インタビュー動画や解説動画をよく扱う人  
AIを編集作業に取り入れたい人  
Vrew以外の選択肢も試したい人  
自分の編集フローに合うツールを探している人

逆に、すでにVrewで作業が完結している人や、とにかく簡単に字幕付き動画を作れればよい人には、Vrewの方が合う場面もあると思います。

CCTは万能ツールではなく、Premiere Pro編集者向けに寄せて育てているツールです。

まとめ：AIで作るより、使える形に育てる方が難しい

Claude Codeを使うことで、個人でもかなり早くツールを作れるようになりました。

これは大きな変化だと思います。

ただ、今回CCTを作って感じたのは、ツールを作ることよりも、実際に使える形に育てることの方が難しいということです。

個人開発AIツールでは、最初から完璧を目指すよりも、まず作って、現場の声を入れて、改善していくことが重要だと感じています。

CCTもまだ完成形ではありません。
