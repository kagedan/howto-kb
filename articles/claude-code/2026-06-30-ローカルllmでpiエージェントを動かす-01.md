数年前、LLMがコードを書いてくれるようになった当初は、チャットインターフェースでAIとやり取りするのが当たり前でした。しかし、2025年を境にその常識は一変しました。今やチャット形式でのプログラミングは「非効率」とされ、AIエージェントが自律的にタスクをこなす時代になっています。

ただ、この「エージェント時代」には大きな悩みがあります。それは**課金**です。

### なぜ今、ローカルLLMエージェントなのか

現在、市場で最もシェアが高いと思われる「Claude Code」などの商用エージェントサービスは、サブスクリプション費用が月に200ドル（約3万円以上）かかることも珍しくありません。大学生や新社会人の個人開発において、この出費は無視できない重荷です。

もちろん、より安価なプランやAPI利用料を抑えたOpenモデルも存在しますが、クラウドモデルを使う以上、データが分析や学習に利用されるリスクはゼロではありません。**プライバシーと機密保持**の観点からも、「手元のマシンで完結する」ローカルLLMへの期待は高まる一方です。

> **余談ですが：**  
> エンジニアが毎月LLMの課金に10万円単位を投じる……という文化は、個人的には長くは続かないと思っています。エンジニアの本質は「コストを最適化しながら、最高のプロダクトを提供すること」にあるはず。ただエージェントに課金してボタンを押すだけなら、それは誰にでもできてしまいますからね。

今回は、そんな「課金なし・プライバシー重視」の設計思想に基づき、最近注目されている **「Pi（pi-coding-agent）」** をローカルLLMで駆動する環境を構築していきます。

## 選択したモデル：Qwen-3.6-35B (MoE)

今回メインで使用するのは、Unslothが公開している [unsloth/Qwen3.6-35B-A3B-GGUF](https://huggingface.co/unsloth/Qwen3.6-35B-A3B-GGUF) です。  
最近のQwenシリーズはコーディング能力が極めて高く、日本語性能に定評のあるGemma-4シリーズと比較しても、ツール呼び出し（Function Calling）の成功率においてQwenに軍配が上がることが多いと感じています。今回は英語メインのコードエージェントとして運用するため、より堅実なQwenを選択しました。

私のマシンは、お世辞にも最新とは言えない「少し前の世代」の構成です。

## 1. 推論サーバー（llama.cpp）のセットアップ

弱いGPUを限界まで使い倒すなら、選択肢は `llama.cpp` 一択です。VRAMを最大活用しつつ、溢れた分をメインメモリ（RAM）でカバーする運用をします。

ビルド手順はUnslothのドキュメントに準拠し、必要なバイナリのみをターゲットに指定してディスク容量を節約します。

```
apt-get update
apt-get install pciutils build-essential cmake curl libcurl4-openssl-dev -y
git clone https://github.com/ggml-org/llama.cpp
cmake llama.cpp -B llama.cpp/build \
    -DBUILD_SHARED_LIBS=OFF -DGGML_CUDA=ON
cmake --build llama.cpp/build --config Release -j --clean-first --target llama-cli llama-mtmd-cli llama-server llama-gguf-split
cp llama.cpp/build/bin/llama-* llama.cpp
```

サーバーの起動コマンドは、効率を重視した以下の設定を使用します。

```
./build/bin/llama-server \
  -m /quants/models/Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf \
  --port 8001 \
  --host 0.0.0.0 \
  -c 131072 \
  --fit on \
  -ctk q8_0 -ctv q8_0 \
  -np 1 \
  -ot "exps=CPU" \
  --no-mmap \
  --threads 8 \
  --cpu-range 0-7 \
  --cpu-strict 1 \
  --prio 2 \
  --temp 0.6 \
  --top-p 0.95 \
  --top-k 20 \
  --min-p 0.00 \
  --repeat-penalty 1.00 \
  --presence-penalty 0.00 \
  --chat-template-kwargs '{"preserve_thinking": true}' \
  --fit-target 512
```

## 2. Piエージェントのサンドボックス環境設計

AIエージェントは強力ですが、同時に「諸刃の剣」でもあります。ファイル編集やコマンド実行を許可するということは、一歩間違えればシステムを破壊しかねないということです。

どれほど洗練されたプロンプトを与えても、LLMが確率的にトークンを出力する以上、100%の信頼は禁物です。そこで、エージェントが暴走してもホストOSに影響を与えないよう、**Dockerによるサンドボックス化**を行います。

本来は `Docker Sandbox` を使うのがモダンですが、私の環境（Pop\_OS 22.04）が少し古いため、標準的なDockerコンテナで構築します。

### pi.Dockerfile の作成

```
FROM node:20-slim

# システム依存関係のインストール
RUN apt-get update && apt-get install -y \
    git python3 python3-pip curl build-essential ca-certificates \
    ripgrep fd-find unzip make \
    && rm -rf /var/lib/apt/lists/*

RUN ln -s /usr/bin/fdfind /usr/local/bin/fd

# Neovimのインストール（コンテナ内作業用）
RUN curl -LO https://github.com/neovim/neovim/releases/latest/download/nvim-linux-x86_64.tar.gz \
    && tar -C /usr/local --strip-components 1 -xzf nvim-linux-x86_64.tar.gz \
    && rm nvim-linux-x86_64.tar.gz

# フォーマッタ・リンターのインストール
RUN npm install -g prettier eslint
RUN pip3 install --break-system-packages ruff pyright

# Piエージェントをグローバルにインストール
RUN npm install -g @earendil-works/pi-coding-agent

# 隔離されたサンドボックスユーザー（UID: 5000）の作成
RUN groupadd -g 5000 pisandbox && \
    useradd -l -u 5000 -g pisandbox -m pisandbox

USER pisandbox
ENV PATH="/home/pisandbox/.local/bin:${PATH}"
WORKDIR /home/pisandbox/project

# uv（高速なPythonパッケージマネージャー）の導入
RUN curl -fsSL https://astral.sh/uv/install.sh | sh

RUN mkdir -p /home/pisandbox/.pi/agent

ENTRYPOINT ["pi"]
```

### このDockerfileのポイント

1. **エディタの同梱:** 私は昨年からNeovimユーザーなのですが、コンテナ内で直接ファイルを覗いたり、エージェントの修正を確認したりするために `nvim` を入れています。
2. **リンター/フォーマッタ:** エージェント自身に拡張機能を作ってもらい、`prettier` や `ruff` でコードを整えられるようにしています。
3. **権限管理:** セキュリティのため、rootではなくUID 5000の一般ユーザーで実行します。ホスト側とのファイル共有（Permission）については、後述するセクションで詳しく解説します。
4. **モダンなツールチェーン:** Python周りの管理を高速化するため、`uv` を採用しています。

## AIエージェントの「権限」をどう設計するか

ぶっちゃけた話、私はAIエージェントというものを100%は信用していません。コンテナの中で動かしている以上、ホストOSに致命的なダメージを与えることはないはずですが、それでも万が一のことがあれば復旧作業が面倒極まりないからです。

そこで、コンテナ内のエージェントには、ホストOSのユーザー（通常はUID:1000）とは異なる権限を与えます。具体的には、Dockerfileでも定義した通り**UID:5000**を設定します。これには明確なメリットが2つあります。

1. **プロセスの完全分離:**  
   万が一 `pi` が `rm -rf /home/user` のような暴走コマンドを実行したとしても、エージェントは自分自身の閉じた世界しか触ることができません。他のユーザーの領域には干渉できないのです。
2. **監査証跡（オーディット）:**  
   エージェントが作成したファイルはすべてUID:5000が所有者になります。これにより、「どれが自分が書いたコードで、どれがAIが生成したものか」を一目で見分けることができます。

> **エンジニアとしての独り言：**  
> エージェントが自分の代わりに仕事をしてくれるのは便利ですが、正直ちょっと怖いですよね。アプリ開発のコードを書くくらいならいいですが、自分のPC内のメールやSNSを自動操作させるなんて、恐ろしくて私にはできません。もし勝手に変な投稿をされたら、責任を取るのはAIではなく、その持ち主である「私」なのですから。

さて、権限を分けると「ファイルの共有」が問題になりますが、これはLinuxの**ACLs (Access Control Lists)** でスマートに解決しましょう。ホスト側のユーザー（1000）もエージェント（5000）も、お互いにストレスなく読み書き・実行ができるように設定します。

```
# プロジェクトフォルダと設定フォルダの所有者をエージェント(5000)にする
sudo chown -R 5000:5000 project pi_home

# ユーザー(1000)も既存のファイルを編集できるように権限を付与
sudo setfacl -R -m u:1000:rwx project pi_home

# 今後作成される新しいファイルも、ユーザー(1000)が編集できるようにデフォルト権限を設定
sudo setfacl -R -d -m u:1000:rwx project pi_home
```

### コンテナの「要塞化」

UIDを分離するだけでも大きな一歩ですが、実はDockerのデフォルト設定は、カーネルレベルで見ると意外と「ガバガバ」です。コンテナはホストOSのカーネルを共有しているため、デフォルトではシステム管理用の権限（Linux Capabilities）を一部保持したまま起動します。

もしエージェントがプロンプトインジェクションを受けたり、悪意のあるスクリプトを実行したりした場合、コンテナ内のシステムバイナリを書き換えられたり、カーネルレベルの命令を悪用してネットワーク内を探索されたりするリスクが残ります。

そこで、Docker Composeに数行書き加えるだけで実行できる **「コンテナの要塞化（Hardening）」** を施しましょう。

#### 1. カーネル権限の完全剥奪 (`cap_drop: [ALL]`)

Linuxには、root権限を細分化した「Capability」という仕組みがあります。通常、エージェントがコードを書いたりテストを実行したりするのに、システム時計を変更したり、カーネルモジュールを操作したりする権限は1ミリも必要ありません。  
`cap_drop: [ALL]` で全ての権限を一度剥奪します。これで、エージェントがカーネルレベルで悪さをすることは物理的に不可能になります。

#### 2. 権限昇格の禁止 (`no-new-privileges:true`)

これは「一度一般ユーザーとして走り出したら、二度とそれ以上の権限（sudoなど）は持たせない」という誓いです。`setuid` などを利用したローカル特権昇格の道を完全に封鎖します。

#### 3. ルートファイルシステムの読み取り専用化 (`read_only: true`)

個人的に一番気に入っているのがこれです。コンテナ内の `/bin` や `/usr` といったシステム領域をすべて「読み取り専用」にします。  
エージェントが勝手に新しいツールをインストールしたり、バックドアを仕込んだりしようとしても、OSから「Read-only file system」と一蹴されます。

ただし、これを設定すると `git` や `uv`、あるいはコンパイラなどが「一時ファイルが書けない！」と悲鳴をあげて動かなくなります。そこで、メモリ上のテンポラリ領域である **`tmpfs`** を scratchpad（書き込み用メモ帳）として割り当ててあげます。

もしエージェントが「ライブラリが足りないから `apt-get` したい」と言い出しても、それは人間の仕事。勝手にシステムを汚させないという制約が、逆に運用の安定性を生むんですよね。

勿論、これでコンテナが安定する代わりに **エージェントの自由度が少し下がります**。最初からネットに繋がっていませんからライブラリーのインストールなどは制限されています。でもこれはエージェントは環境を自分に適用させるではありません、エージェントはこの環境で頑張ってもらいます。

## ネットワークの「鎖」をかける

権限を絞っても、まだ大きな懸念が残ります。それは**インターネット接続**です。  
エージェントが勝手に外部から怪しいスクリプトをダウンロードしたり、こちらの機密データをどこかへ送信したりするリスクを排除しなければなりません。

基本戦略は **「ローカル・ネットワーク完結」** です。エージェントを家庭内LAN（またはDockerの内部ブリッジ）だけに閉じ込め、外部への出口を塞ぎます。もし新しいパッケージのインストールが必要になったら、それはエージェントにやらせるのではなく、人間が手動で行います。

例えば、`uv` を使って特定のパッケージを導入したい場合は、以下のように一時的にネットワークを許可したコンテナを立てて人間が実行します。

```
docker run --rm -v $(pwd)/project:/home/piuser/project \
  --network bridge \
  pi-sandbox-image uv pip install requests
```

この思想を反映させた、最終的な `docker-compose.yml` がこちらです。

```
networks:
  pi_internal:
    driver: bridge

services:
  pi-agent:
    build:
      context: .
      dockerfile: pi.Dockerfile
    container_name: pi_sandbox
    networks:
      - pi_internal
    volumes:
      - ./project:/home/pisandbox/project
      - ./pi_home:/home/pisandbox/.pi
      # Neovimの設定は読み取り専用でマウント
      - ~/.config/nvim:/home/pisandbox/.config/nvim:ro
      # Neovimの動作に必要なランタイムフォルダは書き込みを許可
      - ~/.local/share/nvim:/home/pisandbox/.local/share/nvim
      - ~/.local/state/nvim:/home/pisandbox/.local/state/nvim
      - ~/.cache/nvim:/home/pisandbox/.cache/nvim
    read_only: true
    tmpfs:
      - /tmp                        # 一般的な一時ファイル用
      - /run                        # システムロック、PIDファイル、ソケット用
      - /var/tmp                    # コンパイラ等のフォールバック用
      - /home/pisandbox/.cache      # uv, pip, ruff等のツールキャッシュ用
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    extra_hosts:
      - "host.docker.internal:host-gateway"
    stdin_open: true
    tty: true
```

起動はシンプルに：

```
docker compose up pi-agent
```

コンテナ内に入って作業を確認したい場合は：

```
docker-compose run --entrypoint bash pi-agent
```

### ファイアウォールの調整（Pop\_OS / Ubuntu）

もしコンテナからホスト側のLLM APIに繋がらない場合は、ファイアウォール（UFW）がブロックしている可能性があります。以下のコマンドで、Dockerネットワークからのリクエストを許可してあげましょう。

```
# 接続確認用の疎通テスト
ip route show | grep default
curl -v http://host.docker.internal:8001/v1/models

# 繋がらない場合はUFWでポート8001を開放
sudo ufw allow from 172.16.0.0/12 to any port 8001
```

## PiエージェントとローカルLLMを紐付ける

環境が整ったら、いよいよ `pi` エージェントに「どの脳（LLM）」を使うかを教えます。`pi_home` 内に、OpenAI互換APIとして立ち上げた `llama-server` の情報を記述した `models.json` を作成します。

```
mkdir -p ~/pi_home/agent

cat <<EOF > ~/pi_home/agent/models.json
{
  "providers": {
    "local-server": {
      "baseUrl": "http://host.docker.internal:8001/v1",
      "api": "openai-chat",
      "apiKey": "none",
      "models": [
        { "id": "qwen3.6-35b" }
      ]
    }
  },
  "defaultModel": "qwen3.6-35b"
}
EOF
```

これで、`pi` エージェントはローカルで動く Qwen 3.6 を自らの思考エンジンとして認識し、安全なサンドボックス内でコードを書き始める準備が整いました。

## ローカルLLMの弱点を「検索」で補う

ここまでで、安全な実行環境は整いました。しかし、ローカルLLM（特に35BクラスのMoEモデル）には構造的な弱点があります。それは、モデルサイズゆえの「知識量の限界」です。最新のライブラリ仕様や、特定のバグの解決策など、リアルタイムな情報を知らないことが多いのです。

これを解決するのが**ウェブ検索**です。

通常、検索APIといえばGoogleやBraveが思い浮かびますが、プライバシーを重視して構築している今回の環境で、外部APIにデータを投げるのは本末転倒です。そこで今回は、プライバシー特化型検索エンジン **SearXNG** と、HTMLをLLMが理解しやすいMarkdownに変換する **Crawl4AI** を組み合わせて、完全ローカルな検索基盤を構築します。

<https://github.com/searxng/searxng>  
<https://docs.crawl4ai.com/>

### 1. SearXNG：プライバシー至上主義の検索エンジン

SearXNGは、複数の検索エンジンを束ねるメタ検索エンジンです。これをDockerで動かします。

まずは設定から。

```
mkdir -p ~/searxng/searxng
cd ~/searxng
# シークレットキーの生成（別の端末で実行して控えておく）
openssl rand -hex 32
```

`searxng/settings.yml` を編集し、エージェントが解析しやすいように **JSON出力** を有効にするのが最大のポイントです。

```
use_default_settings: true

server:
  port: 8080
  bind_address: "0.0.0.0"
  secret_key: "YOUR_GENERATED_SECRET_KEY" # 生成したキーを貼り付け
  limiter: false                # ローカル利用なのでRedis制限はオフ
  image_proxy: true

search:
  formats:
    - html
    - json                      # エージェント連携に必須
```

`docker-compose.yml` で立ち上げます。メモリ制限を512M程度に絞っても軽快に動くのがSearXNGの良いところです。

```
services:
  searxng:
    image: docker.io/searxng/searxng:latest
    container_name: searxng
    restart: unless-stopped
    ports:
      - "8080:8080"
    volumes:
      - ./searxng:/etc/searxng:rw
    environment:
      - SEARXNG_BASE_URL=http://localhost:8080/
    deploy:
      resources:
        limits:
          memory: 512M
```

前述と同じくファイルウォールを無効化しておく。

# ### 通信の許可とリソース管理

設定が完了したら、まずは通信の通り道を確保しましょう。外部（あるいは他のコンテナ）からSearXNGにアクセスできるよう、ファイアウォールでポート8080を開放します。

これで準備万端です。いよいよサービスを起動しますが、ここで一つアドバイスを。  
SearXNGのような補助サービスを24時間立ち上げっぱなしにするのは、リソースの無駄遣いです。特に私のような「少し古めのマシン」を使っている環境では、限られたメモリやCPUをLLM本体に割り当てたいため、**「使う時だけ立ち上げ、終わったら即座に落とす」** という運用を徹底しています。

起動はSearXNGのディレクトリで行います：

```
# バックグラウンドで起動
docker compose up -d
```

作業が終わってエージェントを休ませる時は、忘れずにコンテナを停止しておきましょう：

```
# 資源を解放する
docker compose down
```

#### コンテナ間通信の要：ホストIPの特定

ここが少しハマりやすいポイントなのですが、Piエージェント（別のコンテナ）からSearXNGにリクエストを送る際、`localhost`では自分自身を指してしまい上手くいきません。確実な通信を行うために、ホストマシンのローカルIPアドレスを使用するのがベストプラクティスです。

自分のホストIPを確認するには、以下のコマンドを叩きます：

```
hostname -I | awk '{print $1}'
```

例えば、取得したIPが `192.168.1.100` だった場合、ウェブ検索のAPIエンドポイントは以下のようになります。  
`http://192.168.1.100:8080/search?q=<query>&format=json`

正しく動作しているか、`curl` でテストしてみましょう。

```
curl "http://192.168.1.100:8080/search?q=pi-agent&format=json"
```

JSON形式で検索結果がズラッと返ってくれば成功です。これで、エージェントが「外の世界」を覗くための窓口が完成しました。

### 2. Crawl4AI：HTMLのノイズを削ぎ落とす

検索結果からURLを取得できても、生のHTMLはLLMにとって「ノイズ」だらけです。そこで **Crawl4AI** を使います。これはURLを渡すだけで、不要なタグを除去し、LLMが最も好むMarkdown形式に変換してくれる神ツールです。

```
docker run -d \
  -p 11235:11235 \
  --name crawl4ai_server \
  -e CRAWL4AI_API_TOKEN="YOUR_SECRET_TOKEN" \
  unclecode/crawl4ai:latest
```

検証時、トークンなしだとリクエストが不安定になる挙動があったため、簡易的なトークンを設定しておくのが無難です。

```
curl -X POST http://127.0.0.1:11235/crawl \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_SECRET_TOKEN" \
  -d '{"urls": ["https://pi.dev/docs/latest/skills"]}'
```

コンテナを削除したい時は次のコマンドを使ってください。

```
docker stop crawl4ai_server && docker rm crawl4ai_server
```

## Piエージェントに「検索スキル」を実装する

さて、ここからが本番です。Piエージェントにこれらのツールを使わせるための「Skill」を作成します。

LLMは確率的な存在なので、複雑なAPIリクエストを直接書かせるより、決定的な**Pythonスクリプト**を介在させたほうが成功率は格段に上がります。

### ヘルパースクリプトの作成

`pi_home/agent/scripts/search_crawl.py` として保存します。

```
import sys
import json
import urllib.request
import urllib.parse

# 自身のホストIPに合わせて設定
SEARXNG_URL = "http://192.168.1.135:8080/search"
CRAWL4AI_URL = "http://host.docker.internal:11235/crawl"

def search_searxng(query):
    params = urllib.parse.urlencode({'q': query, 'format': 'json'})
    url = f"{SEARXNG_URL}?{params}"
    try:
        req = urllib.request.Request(url, headers={'User-Agent': 'Mozilla/5.0'})
        with urllib.request.urlopen(req) as response:
            data = json.loads(response.read().decode())
            results = data.get('results', [])[:5]  # Retrieve top 5 results
            formatted = []
            for r in results:
                formatted.append({
                    "title": r.get("title"),
                    "url": r.get("url"),
                    "snippet": r.get("content") or r.get("snippet")
                })
            return json.dumps(formatted, indent=2)
    except Exception as e:
        return json.dumps({"error": f"SearXNG search failed: {str(e)}"})

def crawl_url(target_url):
    payload = json.dumps({"urls": [target_url], "wait_after_scroll": 1})
    try:
        req = urllib.request.Request(
            CRAWL4AI_URL,
            data=payload.encode('utf-8'),
            headers={'Content-Type': 'application/json', 'User-Agent': 'Mozilla/5.0'},
            method='POST'
        )
        with urllib.request.urlopen(req) as response:
            res_data = json.loads(response.read().decode())
            # Parse response and extract markdown content safely
            results = res_data.get("results", [])
            if results and isinstance(results, list):
                markdown_content = results[0].get("markdown", "") or results[0].get("html", "")
                if markdown_content:
                    return markdown_content
            return json.dumps(res_data, indent=2)
    except Exception as e:
        return json.dumps({"error": f"Crawl4AI server request failed: {str(e)}"})

if __name__ == "__main__":
    if len(sys.argv) < 3:
        print("Usage:")
        print("  python3 search_crawl.py search \"query\"")
        print("  python3 search_crawl.py crawl \"https://example.com\"")
        sys.exit(1)

    action = sys.argv[1]
    param = sys.argv[2]

    if action == "search":
        print(search_searxng(param))
    elif action == "crawl":
        print(crawl_url(param))
    else:
        print("Unknown action. Choose 'search' or 'crawl'.")
```

スクリプトを実行でますように次のコマンドを打ち込みます。

```
chmod +x pi_home/agent/scripts/search_crawl.py
```

### 動作テスト：大阪のメイドカフェを検索させてみた

エージェントに「大阪でおすすめのメイドカフェを調べて」と投げると、自律的に検索を開始します。

1. `search` を実行し、複数のURLを取得。
2. JS重めのサイトで `crawl` が失敗しても、「このサイトはJSが多いな、別のソースを試そう」と判断して次のURLへ。
3. 最終的に複数のソースから情報を統合して回答。

この **「失敗しても自らリトライする」** 挙動こそ、チャットAIにはないエージェントの真骨頂です。

## 実際に運用してみた感想

正直、ここまで「使える」とは思っていませんでした。

以前、インターンで商用のエージェントを触ったことがありますが、今回のローカル構成（Qwen 3.6-35B）も、ツール利用の正確性に関しては全く引けを取りません。特にQwenの「指示追従性」は凄まじく、ツール実行の失敗を自分でリカバーする様子には感動すら覚えました。

**ただし、速度面では課題もあります。**  
私のGTX 1660 Super環境では、プロンプト処理（入力）が75t/s、生成（出力）が20t/s程度です。コンテキストが3万トークンを超えてくると目に見えて重くなるため、「爆速で開発」というよりは、「自分がメインの実装をしている間に、エージェントにサブ機能やドキュメント作成を任せておく」という**並行開発スタイル**が最も現実的だと感じました。
