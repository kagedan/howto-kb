## この記事が扱っているもの

今回の元記事は「Mastra が A2A をサポートしました」という機能紹介ではなく、**そもそも A2A とは何を解決するためのプロトコルなのか**を解きほぐす Foundations 記事です。

Mastra は 2026 年前半から、Channels、Harness、SDK Subagents、Agent Signals のように「1 個のエージェントを作る」より先の話を一気に進めています。焦点は、**複数のエージェントが、別々のランタイムや別々のチーム境界をまたいで協調できるか**です。

A2A はそのための土台です。ひとことで言えば、**あるエージェントが、別サーバー上の別エージェントへ仕事を委譲するための共通標準**です。

## まず押さえたいこと: A2A は「エージェント向けの HTTP」ではない

この話は「HTTP で別サービスを呼ぶだけでは？」と思うと見誤ります。A2A でやりたいのは、単なる API 呼び出しではありません。

普通のツール呼び出しなら、呼び出し先は単機能な関数に近い存在です。しかし A2A の相手は、プロンプト、メモリ、内部ツール、独自の判断を持った**独立したエージェント**です。しかもこちらは、その内部実装を知りません。

だから A2A では、次のことを最初から前提にしています。

* 相手のコードは見えない
* 相手のメモリもプロンプトも共有しない
* 仕事は数秒で終わるとは限らない
* ストリームは落ちるし Webhook は届かないかもしれない
* 認証や署名を含めて「本当にその相手か」を確認したい

この時点で、普通の tool call とは設計課題がかなり違います。

## MCP と A2A と ACP はどう切り分けると分かりやすいか

ここは一番混同されやすいところです。前回のワークショップ解説でも触れましたが、Mastra 周辺の接続口は少なくとも 3 つに分けて考えると整理しやすくなります。

* MCP は、エージェントが自分のツールや外部データソースにアクセスするための標準
* A2A は、エージェントが別のエージェントへ仕事を委譲するための標準
* ACP は、Claude Code や Gemini CLI のような既存のハーネスや CLI ベースの作業環境へ橋をかけるための標準

感覚的には、こうです。

* MCP は「道具を使う」
* A2A は「別の専門家に任せる」
* ACP は「既存の強い作業環境をひと仕事だけ借りる」

たとえば、社内ドキュメント検索は MCP サーバー経由で行い、その結果をもとに法務専用エージェントへレビューを依頼するなら、前半は MCP、後半は A2A になります。実運用では両方必要です。

そして、前回のワークショップ記事で整理したように、**委譲先が remote agent なのか、同一マシン寄りの coding harness なのか**で A2A と ACP の境界が見えてきます。今回の記事は、そのうち A2A 側を「なぜそこまで重い仕組みなのか」という観点から掘るものです。

## A2A の中核は 5 つのオブジェクト

元記事で強調されているのが、A2A のデータモデルです。仕様は Protocol Buffers を単一の定義源にし、その上で主に次の 5 つを定義しています。

* `Task`
* `Message`
* `AgentCard`
* `Part`
* `Artifact`

ここで重要なのは、A2A が「リクエスト」と「レスポンス」だけで済ませていないことです。

### `AgentCard`

相手のエージェントが自分を説明する名刺です。固定 URL に置かれ、1 回 GET するだけで次のことが分かります。

* 何をするエージェントか
* どこへ仕事を送るか
* streaming や push notification を話せるか
* 入出力フォーマットは何か
* 認証は何が必要か
* 対応しているプロトコルバージョンは何か

つまり、**相手の能力確認と接続条件の確認を、実行前に 1 回で済ませる**のが Agent Card です。

### `Task`

委譲された仕事そのものです。A2A の世界では、仕事は 1 回のレスポンスで終わらないことが前提です。だから `taskId` を持ち、サーバー側に状態を保持し、途中再開できる形で扱います。

### `Message`

会話の 1 ターンです。質問や補足、追加入力を運びます。`role` と `parts` を持ちます。

### `Part`

`Message` や `Artifact` の最小単位です。テキストだけでなく、構造化 JSON、バイナリ、ファイル URL も扱えます。A2A がモダリティ非依存だと言われるのはこのためです。

### `Artifact`

成果物です。重要なのは、**見失ってはいけない出力を Artifact 側へ置く**ことです。接続が落ちると途中メッセージは失われる可能性がありますが、Artifact はタスク状態に残ります。

## A2A が「長時間タスク前提」であることが大きい

この記事で個人的に一番重要だと思うのはここです。A2A は、エージェント委譲が**長時間・中断・再開を含む**ことを標準仕様の中心に置いています。

タスク状態は 8 つあり、ざっくり次の 3 グループに分かれます。

* Running: `submitted`, `working`
* Finished: `completed`, `failed`, `canceled`, `rejected`
* Paused: `input-required`, `auth-required`

この `Paused` があるのが大事です。

実際のエージェント委譲では、途中でこういうことが起きます。

* 追加情報が足りないので人間に聞きたい
* OAuth など追加認証が必要
* バックエンド処理が終わるまで待ちたい
* 別システムのジョブ完了を待ちたい

こういうとき、A2A は「失敗」にせず、**同じ `taskId` のまま止まって待つ**ことができます。数秒後でも数日後でも、follow-up のメッセージを送れば同じ仕事として再開できます。

---

## タスク監視を 3 通り持っているのも実務的

A2A では、進行中タスクの監視方法を 3 つ持ちます。

* Polling
* Streaming
* Webhook push

これも単なる親切設計ではありません。実際のシステムでは、同じ仕事を複数の立場から見たいからです。

* cron 的なバッチは polling で十分
* ダッシュボードは streaming が欲しい
* serverless な処理系は webhook push が便利

しかも、stream は切れますし webhook は未達があり得ます。だから A2A では、**最終的な source of truth はサーバー側の task 状態**だと割り切っています。何かおかしいと思ったら task を読み直す。ここがかなり現実的です。

## なぜ署名や version negotiation まで要るのか

ローカルツールを叩くときは、相手が何者かでそこまで悩みません。しかし A2A の相手は別サーバーです。しかも、エージェントという形でかなり大きな権限を委譲するかもしれない。

だから A2A では、仕事を送る前に次の確認を入れます。

* どの認証方式を受け付けるか
* Agent Card が本物で改ざんされていないか
* プロトコルバージョンが合っているか

特に Agent Card の署名は、「そのカードを誰が公開したか」「途中で書き換えられていないか」を確認するための仕組みです。記事でも触れられている通り、どの鍵を信頼するかまではプロトコルが決めません。そこは組織や製品の脅威モデルに依存するからです。

このあたりは TLS や API 署名の世界観に近いです。A2A は夢のような自律連携プロトコルではなく、**不完全なネットワークと不完全な信頼の上で成立する現実的な委譲プロトコル**として設計されています。

## Mastra は A2A のどこを吸収してくれるのか

元記事の後半では、Mastra が A2A をどう実装しているかが説明されます。ここは実用上かなり重要です。

前回のワークショップ記事では「Mastra は remote agent を subagent として扱えるので supervisor パターンと相性がよい」と書きました。今回の Foundations 記事は、その裏で Mastra が何を肩代わりしているかを、より protocol 寄りに説明しています。

### 1. Mastra エージェントをそのまま A2A 公開できる

Mastra Server に登録したエージェントは、そのまま A2A エンドポイントとして公開できます。Agent Card の生成、well-known パスでの配信、実行エンドポイント公開を Mastra 側が持ちます。

つまり、開発者から見ると「A2A 対応のための別サーバーを 1 本立てる」感覚ではなく、**Mastra のエージェントをそのまま外部委譲可能な形で出す**感覚です。

署名付き Agent Card も設定ブロックを足すだけで有効化できます。

### 2. 外部 A2A エージェントをサブエージェントとして組み込める

`A2AAgent` を使うと、リモートの A2A エージェントを Mastra のサブエージェントとして扱えます。ここがかなり強いです。

開発者は card URL と認証ヘッダーを渡すだけでよく、Mastra 側が裏で次を処理します。

* Agent Card の取得とキャッシュ
* 相手の capability の把握
* streaming 対応なら stream、非対応なら buffered result
* retry / backoff / timeout
* follow-up 入力時の再開

つまり、相手が Mastra 製でなくても、\*\*Mastra から見れば「外の専門エージェントを 1 個の部品として差し込む」\*\*形にかなり近づきます。

### 3. アプリケーションコードから直接 A2A を呼べる

`MastraClient.getA2A()` を使えば、親エージェント経由ではなく、普通のアプリケーションコードから A2A エージェントを呼べます。

これは UI から直接投げたいときや、バックエンド API の中で A2A エージェントを使いたいときに素直です。`resubscribeTask()` があるので、長時間ジョブの再接続もできます。

### 4. `verifyAgentCard` が実運用では効きそう

Mastra がいいのは、A2A を「つながるようにしました」で終わらせていない点です。`verifyAgentCard` フックで、たとえば provider の組織名や期待するドメインを検証し、怪しい相手なら**仕事を送る前に落とせる**ようになっています。

これは実運用ではかなり大事です。エージェント委譲は便利ですが、間違った相手に委譲すると一気に危険になるからです。

## どこで壊れるのかを最初から書いているのが良い

この記事の良さは、A2A をきれいな成功フローだけで語っていないことです。壊れ方が最初から整理されています。

* stream が落ちる
* status message が欠落する
* webhook が届かない
* card が偽造される
* version が合わない
* 受け取れる content type が合わない
* paused task が戻らない

そのうえで、「サーバー側 task を source of truth にする」「署名検証を先にやる」「version mismatch は loud fail にする」といった原則で設計している。ここはプロトコルとしてかなり筋が良いです。

## まとめ

A2A を理解するときに一番大事なのは、これを「AI エージェント版の API 呼び出し」くらいに軽く見ないことだと思います。

A2A が扱っているのは、次の 4 つです。

* 相手の中身が見えないこと
* 文脈を一緒に運ばなければならないこと
* ネットワークが壊れること
* 仕事が接続より長生きすること

だから Agent Card が要るし、Task が要るし、Artifact が要るし、署名と再購読が要るわけです。

発表解説記事とワークショップ解説を先に読んでいるなら、今回の記事で見えてくる追加ポイントは明確です。

* 発表解説では「A2A で何ができるようになったか」を見た
* ワークショップ解説では「A2A と ACP をどこで使い分けるか」を見た
* 今回は「A2A がその役割を果たすために、なぜここまで stateful で慎重な設計なのか」を見た

Mastra はこの A2A を、公開する側、委譲する側、アプリコードから呼ぶ側の全部で扱えるようにしています。今後 multi-agent 構成を本気でやるなら、MCP だけでなく A2A、そしてローカル寄りの作業委譲には ACP まで含めて、**どの境界にどの規格を当てるか**が設計論点になってきます。

まずは Mastra で 1 つエージェントを立てて、well-known の Agent Card を GET してみる。たぶんそこが、A2A を「分かったつもり」から「実感として分かった」へ変える最短ルートです。
