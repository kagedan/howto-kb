Claude Codeのベストプラクティスを調べると、大抵は「具体的な指示を出しましょう」「CLAUDE.mdを活用しましょう」という話になります。それ自体は正しいのですが、なぜそれが必要なのかを知らないと、どこまでやればいいか分からなくなりがちです。

公式ドキュメントを読んで気づくのは、ほぼすべての推奨事項が1つの制約から来ているということです。

**コンテキストウィンドウは想像より早く詰まる。**

それだけです。これを理解すれば、なぜCLAUDE.mdが必要なのか、なぜ探索→計画→実装のステップを踏むのか、なぜサブエージェントが有効なのか、全部つながります。この記事では実際に手を動かしながら、その流れを体験できるように整理しました。

## 環境を整える前に：パーミッション設定

最初に片付けておきたいのがパーミッションです。デフォルトのClaude Codeは、ファイル書き込みやBashコマンドのたびに確認を求めてきます。最初は安心感がありますが、10回クリックした頃には惰性でOKを押しているだけです。

`/permissions` を実行して、安全なコマンドをホワイトリストに入れておきます。

```
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git add *)",
      "Bash(git commit *)",
      "Bash(git status)",
      "Bash(npx eslint *)"
    ]
  }
}
```

もう少し踏み込むなら `--permission-mode auto` も使えます。背後で別の分類器モデルがコマンドをチェックして、危険そうなものだけブロックします。一般的な開発作業なら大体スルーされます。

## Step 1: CLAUDE.mdを設計する

プロジェクトルートで最初に打つコマンドはこれです。

コードベースを解析してCLAUDE.mdの雛形を生成してくれます。ただし、生成されたものをそのまま使うのは少し危険です。長くなるほどClaudeがルールを読み飛ばすようになるので、書き終えたら削ることが大事です。

判断基準は1つだけ：**「これを削除するとClaudeが間違いを犯すか？」** でNoならば削除です。

### 良いCLAUDE.mdの例

```
# CLAUDE.md

## ビルド・テスト
- テスト実行: `npm test -- --watch=false`
- 型チェック: `npm run typecheck`
- 単一テスト: `npm test -- --testPathPattern=<ファイル名>`

## コードスタイル
- モジュール形式: ESM (import/export)、CommonJS禁止
- コンポーネント: PascalCase
- カスタムフック: use プレフィックス必須

## 注意事項
- migrations/ への直接書き込み禁止、必ずスクリプト経由で
- 環境変数は .env.local に置く（.env は gitignore されていない）
```

### 削るべき内容

| 書きがちだけど不要 | 理由 |
| --- | --- |
| 「きれいなコードを書く」 | Claudeは当然知っている |
| TypeScriptの型定義ルール | コードから読み取れる |
| APIドキュメントの全文コピー | URLでリンクするだけで十分 |
| 「テストを書く」 | 具体的な実行コマンドだけ書けばいい |

CLAUDE.mdは複数箇所に置けて、それぞれスコープが違います。

| ファイルパス | 適用範囲 |
| --- | --- |
| `~/.claude/CLAUDE.md` | 全プロジェクト共通（個人設定） |
| `./CLAUDE.md` | プロジェクトルート（チームで共有） |
| `./src/api/CLAUDE.md` | そのディレクトリ内を操作するときのみ |

子ディレクトリのCLAUDE.mdはオンデマンドで読み込まれます。モノレポで特定サービスだけ別ルールを持たせたいときに便利です。

## Step 2: Explore → Plan → Code → Commitを実践する

Claude Codeでよくある使い方は「なんとなく依頼する」です。公式が推奨する4ステップのワークフローを1回体験すると、それとの差がはっきりします。

### Step 2-1: Explore（探索）

`Ctrl+Shift+P` でPlan Modeに入ります。この状態ではClaudeはファイルを読みますが、何も変更しません。

```
@src/auth/ を読んで、セッション管理とトークンリフレッシュのフローを把握してください。
既存のOAuthユーティリティがあるかどうかも確認してください。
```

探索させておいて、その間に自分は仕様のメモを整理する、という使い方が自然です。

### Step 2-2: Plan（計画）

Plan Modeのまま計画を立てさせます。

```
Google OAuthを追加したいです。
変更が必要なファイルは？セッションフローはどうなる？
実装計画をPLAN.mdに書き出してください。
```

`Ctrl+G` でPLAN.mdを開いて、**ここで必ず自分の目で確認します。** Claudeの計画は筋が通っていることが多いですが、要件のニュアンスはどうしても抜け落ちることがあります。気になる部分を直接編集してから次に進みます。

### Step 2-3: Code（実装）

`Ctrl+Shift+P` でNormal Modeに戻ります。

```
PLAN.mdに従ってOAuthフローを実装してください。
コールバックハンドラのテストを書いて、テストスイートを実行して失敗を修正してください。
```

### Step 2-4: Commit

```
変更内容を説明的なコミットメッセージでコミットして、PRを開いてください。
```

計画フェーズが重要になるのは「複数ファイルにまたがる変更」や「コードベースに不慣れな箇所」のときです。小さなtypo修正や変数名の変更なら、直接依頼して問題ありません。

## Step 3: 検証の仕組みを作る

公式ドキュメントが「最も高いレバレッジ」と表現しているのが、Claudeに自分の作業を検証させることです。

検証なしで依頼すると、もっともらしく見えるがエッジケースを処理しない実装が出てくることがあります。テストや期待出力を渡すと、Claudeが自分でフィードバックループを回せるようになります。

```
# 検証なし
validateEmail関数を実装してください。

# 検証あり（こちらを推奨）
validateEmail関数を書いてください。
期待する動作：
- user@example.com → true
- invalid → false
- user@.com → false
実装後に必ずテストを実行して、全パスを確認してください。
```

### Hooksで検証を自動化する

毎回「テストを実行してください」と書くのが面倒なら、Hooksに任せます。

```
// .claude/settings.json
{
  "hooks": {
    "postEditFile": [
      {
        "matcher": "*.{ts,tsx,js,jsx}",
        "command": "npx eslint $FILE --fix"
      }
    ],
    "postToolUse": [
      {
        "matcher": "Bash",
        "command": "npm run typecheck 2>&1 | tail -5"
      }
    ]
  }
}
```

HooksはCLAUDE.mdの指示と違って決定論的です。Claudeが気をきかせてスキップすることはありません。「必ずやること」はHooksに書く、「基本やること」はCLAUDE.mdに書く、という使い分けが明確になります。

Claude自身にHooksを書いてもらうこともできます：

```
すべてのファイル編集後にESLintを実行するHookを.claude/settings.jsonに書いてください。
```

## Step 4: サブエージェントで委譲する

コンテキストが積み重なってきたとき、最も効果的な対処がサブエージェントへの委譲です。

Claudeがコードベースを広く調査すると、大量のファイルを読み込んでコンテキストを消費します。サブエージェントは **別のコンテキスト** で動くため、調査結果の要約だけがメインの会話に戻ってきます。

```
サブエージェントを使って、認証システムのトークンリフレッシュの実装を調べてください。
既存のOAuthユーティリティがあれば再利用できるか確認してください。
```

### カスタムサブエージェントを定義する

`.claude/agents/` にエージェント定義を置くと、専門特化したアシスタントを作れます。

```
<!-- .claude/agents/security-reviewer.md -->
---
name: security-reviewer
description: コードのセキュリティ脆弱性を確認する
tools: Read, Grep, Glob, Bash
model: opus
---
シニアセキュリティエンジニアとして以下を確認する：
- SQLインジェクション、XSS、コマンドインジェクション
- 認証・認可の欠陥
- コード内の認証情報・シークレット
- 安全でないデータ処理

具体的な行番号と修正案を提示する。
```

実装後にこう呼び出します：

```
security-reviewerサブエージェントを使ってsrc/api/auth.ts をレビューしてください。
```

モデルに `opus` を指定しているのがポイントで、重要なセキュリティレビューだけOpusを使い、それ以外のメインの会話はデフォルトモデルで動かすという使い分けができます。

## Step 5: やりがちな失敗パターン4選

設定を整えても、使い方のパターンで損をしていることがあります。公式ドキュメントが挙げている失敗パターンを確認しておきます。

| パターン | どう見えるか | 対処法 |
| --- | --- | --- |
| キッチンシンクセッション | 1セッションで無関係なタスクを混ぜている | タスクが変わる前に `/clear` |
| 修正の繰り返し | 同じ修正を2回以上繰り返している | `/clear` してより具体的なプロンプトで再開 |
| 肥大化したCLAUDE.md | 長すぎてClaudeがルールを無視している | 「削除してもミスが増えないか？」で取捨選択 |
| 検証ギャップ | もっともらしい実装がエッジケースを落としている | テスト・スクリプトを必ず一緒に渡す |

「修正の繰り返し」は特に体感しやすいです。2回失敗した修正を続けてもコンテキストが失敗の記憶で汚染されているため改善しにくく、 `/clear` → より具体的な初期プロンプトで再開する方が結果として早いことが多いです。

巻き戻したいときは `Esc` を2回押すか `/rewind` を実行します。会話状態だけ戻す・コードだけ戻す・両方戻す、を選べます。

## 発展：ファンアウトパターンで大量処理する

もう少し踏み込んだ使い方として、複数ファイルへの一括適用があります。 `claude -p` で非対話型で呼び出し、ループで回します。

```
# ReactコンポーネントをVueに一括移行する例
for file in $(find src/components -name "*.jsx"); do
  claude -p "このファイルをReactからVueに移行してください: $file
移行が成功したらOK、失敗したらFAILだけ返してください。" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

`--allowedTools` でClaudeが使えるツールを制限するのが重要です。無人実行時の安全弁になります。最初は2〜3ファイルで試してプロンプトを調整してから、全量に流すのが安全です。

構造化出力が欲しい場合は `--output-format json` を追加できます：

```
claude -p "すべてのAPIエンドポイントをリストアップしてください" \
  --output-format json | jq '.endpoints[]'
```

## まとめ

Claude Codeのベストプラクティスは、突き詰めると「コンテキスト管理」と「検証の自動化」の2点に集約されます。CLAUDE.mdを適切な量に保ち、Explore→Plan→Code→Commitのリズムをつかんで、検証をHooksに任せる——この3つが揃えば、長いセッションでもパフォーマンスが落ちにくくなります。まずは `/init` と `/permissions` を打つところから始めてみてください。
