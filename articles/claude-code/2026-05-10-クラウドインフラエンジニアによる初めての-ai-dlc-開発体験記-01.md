## なぜこの記事を書いたか

私は普段、クラウドセキュリティ運用とインフラ寄りの仕事をしているエンジニアです。キャリアの前半は SIer で 15 年ほど業務システムの設計・開発に携わっていましたが、ここ数年はクラウドインフラ側に軸足を移しており、AWS のアーキテクチャ設計から運用、コスト最適化、セキュリティまで広く担当しています。アプリ開発とは少し違う筋肉を使う日々です。

そんな私が、AWS Summit Japan 2026 の **[AI-DLC ハッカソン](https://pages.awscloud.com/summit-japan-2026-hackathon-reg.html)** に応募することになりました。テーマは「人をダメにするサービス」、応募作品は日本酒ライフサイクル管理アプリ **SDLC: Sake Driven Life Cycle**。

この記事では、開発の中身そのものより、**「Kiro（実装エージェント）と Claude 系（コンセプト議論 / 差分レビュー）を、どう使い分けて開発を進めたか」** に絞って書きます。AI コーディングエージェントの選定に迷っているエンジニアの方、あるいは AI-DLC という方法論に興味がある方の参考になれば幸いです。

---

## AI-DLC とは何か

AI-DLC（AI-Driven Development Lifecycle）は、AWS が提唱している AI ネイティブな開発方法論です。ざっくり言うと：

* **Inception フェーズ**：要件分析 → ユーザーストーリー → アプリケーション設計 → Unit 分解
* **Construction フェーズ**：機能設計 → NFR 要件 → NFR 設計 → インフラ設計 → コード生成 → ビルドとテスト（NFR = Non-Functional Requirement / 非機能要件）
* **Operations フェーズ**：運用

の 12 ステージを、AI と人間が対話しながら段階的に進めます。各ステージごとに `aidlc-docs/` 配下にマークダウンが生成され、人間が承認したら次に進む、という流れです。

各ステージのアウトプットが、次ステージのインプットになる構造です。Construction フェーズは Unit ごとに 6 ステージ × N 回繰り返します。

具体的なワークフローは [awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows) として公開されており、Kiro や Claude Code などの AI コーディングエージェントから呼び出して使えます。

ここで簡単に登場ツールの紹介もしておきます。**[Kiro](https://kiro.dev/)** は AWS が提供する、Spec-Driven Development（仕様駆動開発）を中心に据えた**エージェント型 IDE**（VS Code ベース）+ **CLI** です。自然言語のプロンプトから要件・設計・実装タスクを段階的に生成し、AI-DLC のような構造化された開発フローと相性が良い設計になっています。一方、**[Claude Code](https://docs.claude.com/en/docs/claude-code/overview)** は Anthropic が提供する **ターミナル（CLI）ベース**の AI コーディングエージェントで、ローカルリポジトリのファイルや git の変更点を直接読み込んで対話的にレビュー・実装ができます。本記事ではこの 2 つを役割分担で組み合わせる、という構成を取ります。

---

## 役割分担の方針：Kiro が「実行」、Claude 系が「監督」

開発を始めるにあたって最初に決めたのは、**役割ごとに違う AI を使い分ける**という方針でした。

| AI | フェーズ | 役割 |
| --- | --- | --- |
| **Claude（claude.ai）** | 着手前（コンセプト議論） | テーマ解釈・社会的リスクの壁打ち相手。ピッチデッキ初稿のレビュー |
| **Kiro** | 開発全体 | aidlc-workflows の実行・ドキュメント生成・コード生成（実装エージェント） |
| **Claude Code（CLI）** | 各ステージのレビュー | ローカルリポジトリの変更点（git diff）を読み取り、横断的な整合性チェック・設計判断の妥当性検証・仕様の盲点指摘 |

なぜこの分担にしたか。理由は単純で、**1 つの AI に「実行」と「自己レビュー」を両方させると、自分の生成物のバイアスから抜け出せない**からです。Kiro が出してきた要件定義を、同じ Kiro に「レビューして」と頼んでも、構造的な見落としは見つかりません。別の AI に持っていって初めて「ここの数字、他のドキュメントと矛盾してませんか？」が指摘される。

これは人間のチーム開発でも同じで、書いた人と別の人がレビューするから品質が上がるわけです。AI を使った開発でも、この **「役割を分けてクロスチェックする」** という発想は有効でした。さらに、**「ファイルが揃う前のコンセプト議論」は対話型の Claude（claude.ai）、「ファイルが揃ってからの差分レビュー」は Claude Code（CLI）**、という形で、同じ Claude 系のサービスでも自然に使い分けが分かれました。

実際の Construction フェーズでのデータの流れはこんな具合です（コンセプト議論を終えた後の流れを示します）。

ポイントは、**Claude Code に見せる範囲は「Kiro が一区切りつけたタイミング」のステージ単位**にしたことです。ターミナルから `claude` を立ち上げて、「最新コミットでの変更点をレビューして」と依頼するだけで、Claude Code がローカルリポジトリ内の `aidlc-docs/` 配下のマークダウン群を直接読みに行ってくれる。コピペで貼り付ける必要がなく、関連ファイルへの横断参照も自然にできるので、レビューの粒度として現実的でした。

## Kiro での実装フロー

### Inception フェーズ：対話で要件を固める

最初は SDLC のピッチデッキ PDF を Kiro に渡すところから始めました。

```
こちらのコンセプトで Web アプリを開発したいです。（PDF添付）
```

Kiro は aidlc-workflows に従って、まず **11 問の要件確認質問**を生成してきます。「フロントエンドフレームワークは？」「認証方式は？」「PWA（Progressive Web App / ブラウザでネイティブアプリのように動く Web アプリ）対応の範囲は？」といった具合です。回答すると、追加で曖昧な点を 7 問追加質問してきました。これに答え終わると、`requirements.md` が出力されます。

ここで気づいたのは、**Kiro はかなり粘り強く曖昧さを潰しにくる**ことです。私が「両方対応で」と曖昧に答えると、「両方というのはテキスト入力と画像入力の両方ですか？それとも別の解釈ですか？」と確認してくる。インフラ専門でアプリ要件を詰めるのが苦手な私には、この性質がありがたかった。

Inception フェーズ全体（Workspace Detection から Units Generation まで）は、Kiro との対話を中心に **約 1 日**で完了しました。

### Construction フェーズ：Unit 分解された粒度で実装

Inception で 6 つの Unit に分解した後、Construction フェーズでは Unit ごとに「機能設計 → NFR 要件 → NFR 設計 → インフラ設計 → コード生成」を回します。

私が今回主に進めたのは **Unit 1 (Foundation)**：認証・共通基盤・DynamoDB スキーマ定義 の部分です。Kiro はここで Terraform モジュール、Lambda ハンドラー、React コンポーネント、テストコード（Vitest + fast-check）まで一気通貫で生成してくれました。

特筆すべきは **fast-check による Property-Based Testing** が標準で入ってくることです。「パスワードの長さ要件が NIST 準拠（米国国立標準技術研究所が定めるパスワードガイドライン SP 800-63B）であること」のような不変条件を、ランダム入力で検証するテストが自動生成される。これは自分で書こうとしたら相当時間がかかる箇所です。

## Claude Code でのレビューフロー

ここからが本記事の本題です。Kiro の生成物を、私はそのまま受け取らず、**毎ステージ Claude Code（CLI）に持ち込んでレビューしてもらいました**。具体的には、ターミナルで `claude` を起動し、「直前のコミットでの変更点をレビューして」と依頼するだけ。Claude Code はローカルリポジトリの差分と関連ファイルを直接読み取り、横断的な指摘を返してくれます。

### レビュー 1：ドキュメント間の整合性チェック

AI-DLC の成果物は 14 ファイル以上に分散します。`requirements.md`、`stories.md`、`personas.md`、`unit-of-work.md`、`components.md`、`audit.md`...。Kiro はステージごとに着実に生成しますが、**「ある変更が全ファイルに伝播しているか」までは見てくれない**ことがあります。

例えば、開発の途中で MFA（多要素認証）機能を追加した時のことです。Kiro は `business-rules.md` と `domain-entities.md` には MFA フィールドを追加しました。しかし Claude Code にレビューを依頼すると、こう返ってきました：

> MFA追加（US-02B）が以下の Inception 成果物に未反映です：
>
> * `unit-of-work.md`：Unit 1 関連ストーリーに US-02B が無い
> * `components.md`：FE-01 の責務に MFA が無い
> * `requirements.md`：FR-07 に MFA / NIST 記述が無い
> * `unit-of-work-story-map.md`：US-02B 行が無く、ストーリー数が 27 のまま

これは Kiro が悪いわけではなく、**「ある時点での生成タスク」に集中しているため、過去の成果物への波及反映が後回しになる**という構造的な性質です。Claude Code は git の変更点起点で **リポジトリ全体を俯瞰してドキュメント間の整合性を見にいける**のが得意で、ここの分担がきれいにハマりました。

### レビュー 2：技術的リスクと制約の指摘

これは特に印象に残ったケースです。Kiro が生成した要件定義書には「プッシュ通知に Amazon Pinpoint を使用」と書かれていました。素直に受け取りそうになったのですが、Claude Code にレビューを依頼すると：

> Amazon Pinpoint は **2026-10-30 にサポート終了**します。本サービスのリリースが 2026 Q3 とすると、リリース直後にサポート切れの状態になります。Web Push (VAPID) + Service Worker への切り替えを推奨します。

これは、Claude Code が AWS 側のアナウンスを把握していたから出てきた指摘です。Kiro は今ある AWS サービスから素直に選びますが、**「将来の EoS（End of Support / サポート終了）リスクを織り込む」までは生成時に見ない**。インフラ専門の人間ならピンとくる観点ですが、AI に任せきりだとスルーする可能性のある罠でした。

結局この指摘を受けて、8 ファイルを横断して Pinpoint → Web Push に置き換える作業を行いました。早期に気づけたのは大きかったです。

他にも、Claude Code のレビューでこんな指摘が入りました：

* **Cognito × NIST SP 800-63B のギャップ**：NIST 準拠のブロックリスト照合は Cognito ネイティブでは実現できない。Pre Sign-up Lambda Trigger を追加する設計が必要
* **Unit 並行開発の矛盾**：Unit 2 → Unit 3 の依存があるのに「並行開発可能」と書かれていた。共有インターフェースを Unit 1 で先行確定する三段階構成（Phase 0/1/2）に修正
* **Lambda 関数数の不整合**：`requirements.md` では 22、`components.md` の集計では 27 と矛盾

### レビュー 3：プロダクト設計の方向性チェック

ここだけは Claude Code ではなく **Claude（claude.ai）** に頼みました。Kiro が動き出す前の「そもそもこのコンセプトでいいのか？」という、リポジトリにまだ何も無い段階での壁打ちだったためです。

ハッカソンのテーマは「人をダメにする」ですが、最初の SDLC のコンセプトには「禁酒防止機能」という、AI が「飲まない」を否定する機能が含まれていました。ピッチデッキの初稿を Claude に共有してレビューしてもらうと：

> アルコール依存症の回復中の方や妊娠中・服薬中の方に対して、AI が「飲まない選択を修正する」ことを売りにしているように読めます。AWS という企業の公式イベントで、登壇者・審査員が「飲まない選択を否定する AI」を公に褒めるのは PR 上ほぼ不可能です。

ここから議論を重ねて、**「Skip Deploy」を一級機能として再設計**する方針が固まりました。「飲まない日も同じ UI の上で設計する」という、現在のコンセプトの核となる発想です。

整理すると、**「ファイルが揃う前のコンセプト議論は Claude（claude.ai）、ファイルが揃ってからの差分レビューは Claude Code（CLI）」** という形で、同じ Claude 系のサービスでも使い分けが自然に分かれました。Kiro は「決まった方針に従って実装する」のは得意ですが、「ハッカソンのテーマ解釈と社会的リスクのトレードオフを議論する」フェーズには、対話型のチャットインターフェースの方が向いています。

## 役割分担の効果と限界

### 効果

3 サイクルにわたるレビューで、最終的に以下のような是正ができました：

* **Cycle 1**：進捗管理・波及反映の是正（MFA 追加の伝播、数値整合）
* **Cycle 2**：技術的リスクの是正（Pinpoint EoS、Cognito × NIST 整合性、コスト戦略）
* **Cycle 3**：設計整合性・用語整理（Unit 並行開発の依存矛盾、トレーサビリティ強化）

これらは `audit.md` に時系列で全部記録されており、ハッカソンの書類審査でも「ドキュメントの品質」評価軸でアピールできる証跡になりました。

### 限界

この方法論にも限界はあります。

**1 つ目は、レビュー往復のオーバーヘッド**です。Kiro が出した成果物に対して Claude Code でレビュー依頼 → 指摘を Kiro に戻して修正 → 再度 Claude Code で確認、という往復が発生します。1 ステージあたり 30 分〜 1 時間ほど追加コストがかかりました。両方 CLI なので切り替え自体は速いものの、ステージごとの判断は積み重なります。

**2 つ目は、Claude Code のコンテキスト長の制約**です。`audit.md` だけで 50KB を超えてくるので、レビュー依頼時にどの範囲を読ませるかは取捨選択が必要でした。git diff 起点だと自然に「変更点 + 関連ファイル」に絞られるので、この問題は緩和されますが、横断レビューを依頼する時は意識が必要です。

**3 つ目は、両者の判断が衝突した時の調停は人間側に残る**ことです。Kiro と Claude Code が違う方針を示してきた時に、どちらを採用するか決めるのは結局自分。AI 同士で自動的に解決はしてくれません。設計判断の最終責任は、引き続き人間が持つ前提でした。

## クラウドインフラエンジニア視点での所感

最後に、普段インフラ寄りの仕事をしている自分から見ての所感を 3 点。

**1. AI-DLC は「インフラエンジニアでもアプリが作れる」を実現する方法論**

正直に言うと、私は React も TypeScript も普段は書きません。ですが今回、認証・PWA・状態管理まで含めた React アプリを Construction フェーズの最初のサイクルで生成できました。Kiro の生成能力もさることながら、**AI-DLC のステージ分割が適切なため、苦手領域でも段階的に詰められた**のが大きいです。「全部一気に作れ」ではなく、「機能設計 → NFR 要件 → NFR 設計 → インフラ設計 → コード生成」と段階を踏むので、各段階で人間が判断しやすい。

ここは個人的に追記しておくと、SIer 時代に経験した **要件定義 → 基本設計 → 詳細設計 → 実装 → テスト** という古典的な V 字モデル開発との対応が、AI-DLC では非常に自然です。AI が各ステージのアウトプットを生成してくれるので、レガシー側の設計手順をベースにしつつ、AI 駆動で時間を圧縮している感覚に近い。設計フェーズの感覚値（ここで何を決めるべきか、何を後回しにできるか）が AI-DLC の判断ポイントとそのままハマったので、苦手なはずのアプリ実装まで通せました。設計・開発の経験が多少でもある人にとって、AI-DLC は学習コストの低い方法論だと思います。

**2. Kiro は「優秀なジュニアエンジニア」、Claude 系は「経験豊富なシニア」**

たとえとして適切かわかりませんが、感覚的にはこんなイメージです。Kiro は与えられた指示を高速かつ正確に実装してくれますが、「過去の経験から潜在リスクを察知する」のは人間（とそれに近い対話型 AI）の役割です。コンセプト議論なら Claude（claude.ai）、差分レビューなら Claude Code（CLI）、と Claude 系の中でも面で使い分けることで、レビューの抜け漏れがかなり減りました。

**3. ドキュメント駆動の進化形**

私はインフラ運用の現場で「設計書と実装の乖離」を何度も見てきました。AI-DLC では設計書が AI 生成のソースになるため、**設計書こそが正本**になります。コードが先行して設計書が古くなる、という普段の悩みがそもそも発生しない構造です。これはインフラエンジニア的にとても気持ちの良い体験でした。

---

## まとめ：Kiro × Claude 系のプラクティスを試してみたい人へ

3 行で振り返りまとめると：

* **着手前は Claude（claude.ai）でコンセプト壁打ち、開発中は Kiro が実行、各ステージで Claude Code が監督**——という三役分担で、生成物の品質が体感で 1 段階上がった
* 特に効くのは **「ドキュメント間整合性」「技術リスクの先回り」「方向性の壁打ち」** の 3 領域。後者ほどコンセプト寄りなので、Claude（claude.ai）の対話インターフェースが向いている
* ただし往復コストはかかる。ステージごとに「これは Claude Code にも見せる」「これは Kiro 単独で OK」を判断する目利きが必要

AI-DLC ハッカソンの結果はこれから出るところですが、勝敗とは別に、この開発スタイル自体は今後も続けたいと感じました。1 つの AI に依存しすぎず、複数の AI を「同僚」として使い分ける時代の入り口にいるんだなと、GW の 1 週間で強く実感しました。
