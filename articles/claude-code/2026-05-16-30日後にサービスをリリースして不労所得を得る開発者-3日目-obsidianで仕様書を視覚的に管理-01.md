## はじめに ― 仕様書が増えるたびに「全部読むの大変」と言われる

[1 日目](https://zenn.dev/tempeso/articles/900ea8ade4c165) で土台を作り、[2 日目](https://zenn.dev/tempeso/articles/6e36f89a90af40) で CI を直し、ようやく機能開発に入ろうとした矢先のことです。

`docs/` 配下に **画面仕様書・DB 設計書・アーキ設計書** が積み上がっていきました。

```
EIGYOU/docs/
├── APP_OVERVIEW.md             プロダクト概要
├── ARCHITECTURE.md             4層アーキ・技術スタック（1,000行超）
├── DOMAIN_SKETCH.md            ドメイン用語・ER 関係
├── DB_DESIGN.md                DB 全体方針
├── DB_DESIGN_TABLES_FIRESTORE.md   Firestore 14 コレクション
├── DB_DESIGN_TABLES_DRIFT.md       Drift 16 テーブル
├── SCREENS.md                  画面一覧 32 画面
├── SCREENS_DETAIL.md           画面別最終仕様
└── mockups/<画面ID>/           HTML モックアップ
```

仕様書としては網羅的。…でも、こうなりました。

* 仕様書 / md ファイルが増えて、しかも **1 ファイル 1,000 行を超えるもの** まで出てきて、**もはや頭に入らない**
* スクロールしながら読んでいるうちに、**画面・テーブル・アーキの関係性という「全体像」が掴めない**
* **久しぶりに開いた時に「これ何だっけ？」** となり、結局また最初から読み直す
* AI（Claude）に渡すときも、**どのファイルをどこまで読ませるか** を毎回考える羽目になる

要するに、**書いた本人ですら、しばらく経つと自分の仕様書を読みこなせなくなる** ということです。

楽したい開発者として、これは早めに解決したい問題です。仕様書を読む時間も、AI に渡すコンテキストを毎回考える時間も、**「楽をしたい」が阻害される時間** だからです。

そこで 3 日目は、**Obsidian + 自作 skill** で仕様書を視覚化して、ドキュメント管理を「楽な仕事」に変える日にしました。

!

**この記事で得られること**

* 散らばった仕様書md を **リンクで束ねる** 仕様書 Wiki の設計思想
* Claude Code の **skill** に Wiki 生成を任せるレシピ（`SKILL.md` の公開）
* **Obsidian Canvas** で画面遷移図・ER 図・アーキ図を作る `.canvas` JSON の実物
* 運用ルール（複製禁止・1 行スタブ・Vault ルートの罠）

## 🗺 おさらい ― そもそも何を作ったのか

詳細は別記事 [**mdファイル、見にくくない？仕様書を Obsidian Canvas で「絵」にしたら開発が早くなった**](https://zenn.dev/tempeso/articles/36e556a0d78d5a) にまとめています。3 日目ではここを **画像 4 枚で超簡潔におさらい** してから、実装の中身に踏み込みます。

### 画面遷移図（screen-flow.canvas）

32 画面を **機能系統ごとの列に整列** させた Canvas。各ノードはダブルクリックで該当画面の markdown に飛びます。  
![](https://static.zenn.studio/user-upload/8cd30cc8d66c-20260516.png)

### Firestore ER 図（er-firestore.canvas）

サブコレクションの階層構造をグループで囲み、テーブル名・Path・フィールド表をノードに格納。

![](https://static.zenn.studio/user-upload/81db9710cce7-20260516.png)

### アーキ図（architecture.canvas）

4 層 Clean Architecture を縦に重ねて、各層に主要モジュールを box で配置。

![](https://static.zenn.studio/user-upload/044e72d2596e-20260516.png)

### 全体の入口になる `index.md`

ここまでの Canvas や各ハブページは、すべて **`index.md` という 1 枚のページ** から辿れるようにしてあります。**これさえ開けば、プロジェクトがどんな構造になっているか一目で把握できる** 入口です。

![](https://static.zenn.studio/user-upload/803475d6a115-20260516.png)

ここから先がこの記事の本題です。**この Wiki を Claude Code に自動生成させる方法** を、skill の中身ごと全部公開します 🚀

## 🤖 SKILL.md の中身を全公開する

`.claude/skills/eigyou-wiki/SKILL.md` がやっていることは、つきつめると次の 3 つだけです。

1. **いつ発動するか** を Claude に教える
2. **絶対に守らせたいルール** を最初に宣言する
3. **記法詳細は公式 skill に丸投げ** する

順に見ていきます。

### ① いつ発動するか ― frontmatter

.claude/skills/eigyou-wiki/SKILL.md

```
---
name: eigyou-wiki
description: docs/ 配下の仕様書を「リンク参照」で視覚的 Wiki に織り上げる。Wiki化・ER図/アーキ図/画面遷移の Canvas 化を指示された時に使用する。
---
```

`description` に **トリガーキーワード**（「Wiki化」「Canvas 化」など）を入れておくと、Claude が自動で発動するか判断してくれます。

「`SCREENS.md` を Wiki にして」みたいな曖昧な指示でも、Claude が勝手に `eigyou-wiki` を呼び出します。

### ② 絶対に守らせたいルール ― 設計の根本原則

```
## 設計の根本原則

- `docs/` が **真実の源**。Wiki 側に同じ内容を複製しない
- Wiki ページは **ハブ + リンク + 派生視覚化** だけを担う
- メンテナンスコストは docs/ 側に一本化する
```

SKILL.md の **冒頭ほど Claude は強く影響を受ける** ので、ここに「絶対やってはいけないこと（=複製）」を真っ先に書きます。

### ③ 専門 skill への委譲表

```
## 専門スキルへの委譲

このスキルは**「何をどこに置くか」**だけを担う。
**「どう書くか」は専門スキルに委譲**する。

| やりたいこと | 委譲先スキル |
| --- | --- |
| `.md` の frontmatter / wikilink / callout / embed / tag | `obsidian-markdown` |
| `.base` ファイル（views / filters / formulas） | `obsidian-bases` |
| `.canvas` ファイル（nodes / edges / groups） | `json-canvas` |
| Vault 内のノート操作・検索 | `obsidian-cli` |
```

これが今回の skill 設計の **最重要パート** です。

ここを書いておくと、Claude は Canvas を作るときに **自分から** `json-canvas` skill を呼びにいってくれます。Markdown のフォーマットも、Bases の views 定義も、全部それぞれの skill が責任を持つ。

`eigyou-wiki` 自身は記法を一切知りません。代わりに「Canvas を作るときは必ず `json-canvas` を呼べ」と命じるだけです。

#### 委譲先の Obsidian 公式 skill について

ここで委譲先に指定している 4 つの skill は、Obsidian の作者である Kepano さんが公開している **公式の Claude Code skill** です。

🔗 [kepano/obsidian-skills (GitHub)](https://github.com/kepano/obsidian-skills/tree/main/skills)

それぞれが「Obsidian の特定機能を AI に正しく書かせる」ことだけに特化していて、`eigyou-wiki` のようなプロジェクト固有の skill から呼び出して使うのに最適です。**自分で記法ガイドを書く必要が消えます**。

| skill | 何をしてくれるか | `eigyou-wiki` から呼ぶタイミング |
| --- | --- | --- |
| **[obsidian-markdown](https://github.com/kepano/obsidian-skills/tree/main/skills/obsidian-markdown)** | Obsidian Flavored Markdown の文法（`[[wikilink]]` / `![[embed]]` / `> [!note]` callout / properties / tags など）を厳密に書き分けるためのリファレンス | リンクハブや 1 行スタブを **新規作成・編集** するたび |
| **[obsidian-bases](https://github.com/kepano/obsidian-skills/tree/main/skills/obsidian-bases)** | `.base` ファイル（Obsidian Bases）の views / filters / formulas / summaries を構築 | 「画面を優先度や種別で絞り込める表ビューが欲しい」など、**動的なリスト・テーブルを作る** とき |
| **[json-canvas](https://github.com/kepano/obsidian-skills/tree/main/skills/json-canvas)** | JSON Canvas Spec 1.0 に準拠した `.canvas` ファイル（nodes / edges / groups / colors）を生成 | 画面遷移図・ER 図・アーキ図など、**ビジュアルマップを書く** とき |
| **[obsidian-cli](https://github.com/kepano/obsidian-skills/tree/main/skills/obsidian-cli)** | Obsidian CLI 経由で Vault のノートを開く・検索する・プロパティを更新する。プラグイン開発時の debug にも使える | 既存ノートの **一括検索・置換、孤立ページ検出** など、Vault 全体を機械的に走査するとき |

### ④ 3 つの動作モード

skill は次の 3 モードで動きます。Claude は指示文から **どのモードか自己判定** してくれます。

| モード | いつ動く | 何をする |
| --- | --- | --- |
| **Ingest** | 「Wiki 化して」 | docs/ への相対リンクと wikilink でハブを生成、Canvas を新規作成、index.md を更新 |
| **Query** | 「〇〇について教えて」 | `index.md` から関連ハブを辿り、必要な docs/ だけ Read して回答 |
| **Lint** | 「Wiki を点検して」 | リンク切れ・孤立ページ・docs/ にあるのに到達不可なファイルを検出 |

3 モードを明示しておくだけで、Claude が「今は Ingest？Query？Lint？」と勝手に切り替えてくれます。

## 🚀 実際の使い方 ― Claude Code に投げるコマンド集

ここまでの仕組みを仕込んでおくと、Claude Code で次のような指示が通るようになります。

### 全部 Wiki 化（初回構築）

```
> /eigyou-wiki docs を全部 Wiki 化して
```

Claude が裏側でこれをやってくれます:

1. `docs/` の主要ファイルを Read
2. 画面一覧・テーブル定義・アーキ層を抽出
3. リンクハブを生成
4. 画面スタブを 32 枚量産
5. Canvas 4 枚を JSON で生成（`json-canvas` skill 経由）
6. `index.md` を更新

### 部分取り込み

```
> /eigyou-wiki 画面 S-04 だけ取り込んで
```

スタブ 1 枚だけ作って screens/README.md の表に追記してくれます。

### DB 設計変更後の再生成

```
> /eigyou-wiki Firestore の ER 図だけ作り直して
```

スキーマ変更があったら ER Canvas を上書き再生成。docs/ の最新フィールド表を読み込んで、Canvas のテーブルノードを更新します。

### 健全性チェック

```
> /eigyou-wiki Wiki を点検して
```

孤立ページ、リンク切れ、docs/ にあるのに Wiki から到達不可なファイルを検出してくれます。

## ✨ 結果: 何が「楽になった」か

### Obsidian を開いて画面遷移が 1 秒で頭に入る

文章で「ホームから G-01 ナビ経由でカレンダー・顧客・設定に分岐」と書かれていても、読み返すたびに頭で組み立て直しが必要でした。Canvas を見ると **1 秒で全体像が入ります**。

### Claude に渡すコンテキストが二段構えになった

以前は「`docs/SCREENS_DETAIL.md` を全部読んで」と毎回フル読み込みでした。今は:

1. まず `eigyou_wiki/index.md` を読んでもらう
2. そこから関連ハブ → 必要な docs/ だけ深掘り

…の流れができたので、**トークン消費が大きく減りました**。

### 新しい仕様書を追加するときが楽

docs/ に新しい markdown を置いたら、Claude に `/eigyou-wiki 取り込んで` と投げるだけ。リンクハブの更新と Canvas への追記まで自動でやってくれます。

### 「楽したい」の達成度

* 仕様書を読み返す時間: **大幅減**（Canvas で全体像が一発）
* AI に渡すコンテキスト整備: **大幅減**（index.md 経由で必要な部分だけ）
* Wiki 自体のメンテ: **ほぼゼロ**（docs/ 更新時に skill が再生成）

不労所得 ─── まではまだ遠いですが、**「仕様書管理から解放されたい」** という目標には一歩近づきました 🛌

## 📝 まとめ

* 仕様書は **docs/ を真実の源**にして、Wiki 側は **リンクと派生視覚化だけ** を持つと、メンテコストが増えない
* Claude Code の **自作 skill** は「**何をどこに置くか**」だけ担い、**「どう書くか」は専門 skill に委譲** すると保守性が一気に上がる
* Canvas の `type: "file"` ノードは Vault ルートからの絶対パスで指定する（**Vault ルート判定に注意**）
* 画面遷移図・ER 図・アーキ図を **JSON で表現** できると、Claude が読み書き両方できる
* 運用ルールは **「複製禁止」「1 画面 1 スタブ」「入口は index.md だけ」** の 3 つで破綻を防ぐ

「楽したい」を起点に設計すると、ドキュメント管理は **「Claude にやらせる仕事」** に変わります。
