こんにちは、ブーストコンサルティングです！

Claude Codeに新モデル「Fable5」が登場しました。（ついに！！？）  
Anthropic社はこれを「Mythos（神話）」ティアと位置づけ、従来の最上位だったOpus4.8を一段上から見下ろす存在として投入してきました。

今回はそんな新モデル「Fable5」を使いこなしていくためのコツをお伝えします！（6/22までサブスクで利用可能とのこと）

※今後アップデートにより仕様が変わる可能性がありますので、最新情報は公式サイトでご確認ください。

気になるのはそのコストです。APIの利用コストはOpus 4.8のちょうど2倍。日々の開発で使うには、なかなかの重さです。

「全部 Fable5にすべき？」「いや、2倍はきつい……」

多くの開発者が、いま同じ悩みを抱えているのではないでしょうか。

## Fable5は本物か？数字で見る実力

Fable5の性能向上は「気のせい」ではなさそうです。

具体的には、次のような領域で差が出ると報告されています。

**アーキテクチャ判断：**CPythonのWASMサンドボックスをプロンプトから自律構築した事例

**セキュリティ分析：**  
「Mythosはセキュリティが桁違い」という評判。脆弱性の深掘りでOpusが見落とすパターンを拾う

**大規模リファクタリング：**  
5000万行のコードベース移行を1日でこなした報告

ただし、ここで一つ重要な注意点があります。

Fable5には安全フィルター（セーフガード）が組み込まれており、サイバーセキュリティや生物・化学に関連するクエリが検知された場合、自動的にOpus 4.8が代わりに応答します。Anthropic社の公式発表によれば、このフォールバックが発生するのはセッション全体の平均5%未満。つまり、95%以上のやり取りではFable5がそのまま動作します。

ただし、この5%が自分の作業領域と重なる場合、たとえばセキュリティ診断やインフラ構成のレビューを日常的に行う開発者、は注意が必要です。

コストはFable5の価格で発生しないとはいえ（フォールバック時はOpus 4.8の料金が適用されます）、「Fable5に頼んだつもりがOpus 4.8が答えていた」という状況は、品質の見積もりを狂わせる可能性があります。この「見えない切り替わり」は、使いこなしの重要な変数として頭に入れておく価値があります。

## なんとなく全部Fableが一番危ない！？

Fable5の登場で最も警戒すべきは、「FOMO（乗り遅れる恐怖）」による過  
剰消費です。

「みんな使ってるし」「最高峰なんだから」

そう思って日常的なコード生成やコミットメッセージ作成にまでFable5を使うと、コストだけが雪だるま式に増えていきます。1日あたり50〜150ドルのトークン費用が可視化されたとき、「この値段で続ける価値があるか」という問いに向き合うことになるでしょう。

裏を返せば、これはAnthropic社にとって理想的な構図でもあります。

ユーザーが自発的に「Fable5が必要なタスク」を探し、自発的に2倍のトークンを消費する。値上げを強制されたのではなく、自分で選んだ価値だと思えるからです。

ここで賢い開発者と、そうでない開発者の差がつくとみています。

前者は「どのタスクでFable5の2倍コストが元を取れるか」を冷静に測り、後者は「なんとなく良さそう」でモデルを選びます。

数ヶ月後、この判断の積み重ねが成果物の品質とコストの両面で明確な差になります。

## 最適解は「選択的併用」！？3つの領域に集中せよ！

では、具体的にどう使い分ければいいのか？

結論から言えば、**全タスクFable5でも、全タスクOpusでもない、「選択的併用」**が最も理にかなっていると思われます。

Fable5に振るべきは、次の3つの領域です。

**アーキテクチャ判断：**新規機能の設計、技術選定、システム構成の見直し。これらは「間違えたときの手戻りコスト」が極めて大きく、深い推論が長期的な手戻り防止に直結します。

**複雑なセキュリティ分析：**認証周り、APIの権限設計、脆弱性の発見。ここは「Mythos の本領」と評される領域であり、Opus 4.8では拾えなかった問題をFable5が検出する可能性が高い。

**大規模リファクタリングの設計：**影響範囲の広い変更の「見積もり」と「設計」。実装そのものではなく、「どこをどう変えるべきか」という判断をFable5に任せる。

一方、定型コード生成、ドキュメント作成、単純なバグ修正、コミットメッセー ジ作成といった日常作業は、Opus 4.8で十分です。これらのタスクでは2倍コストに見合う品質差が出にくく、Fable5は明らかに過剰です。

この使い分けの考え方は、タスクの「エラー時の損害」と「現在かけている品質管理の手厚さ」の2軸で整理するとわかりやすいでしょう。

## 本当の資産は「モデル」ではなく「自分のルール」だ

ここからが、この記事で最もお伝えしたいポイントです。

Fable5の登場は、個人開発者が蓄積してきた「ハーネス」──Claude Code  
のルール、スキル定義、レビュー観点、エージェント構成といった自動化レイヤー──の価値を改めて浮き彫りにしました。

なぜなら、**モデルは入れ替わるが、ハーネスは残る**からです。

Opus 4.8は Fable5の登場からわずか10日ちょっとで「旧世代」になりました。おそらく半年後には Fable 6が来るでしょう。そのたびに「どのモデルが最適か」という判断をゼロからやり直すのは消耗です。

そこで有効なのが、**ハーネスを「モデル非依存」に書き換える**という一手です。

具体的には、CLAUDE.mdやエージェント定義の中から「Opusでレビューする」のようなモデル名に依存した記述を探し、「推論深度が必要なタスクでは上位モデルでレビューする」のように抽象化します。

たとえばBefore → Afterで見てみましょう。

**Before（モデル依存）：**  
セキュリティレビューは Opus 4.8で3並列レビューを実行する。

**After（モデル非依存）：**  
セキュリティレビューは、その時点で最も推論深度の高いモデルで実行する。レビュー並列数はモデルの信頼性に応じて調整する。

この書き換えにかかる時間は数時間です。

しかし、近い将来Anthropic社がtier構造を再編したり、競合他社のモデルに乗り換えたりする局面で、移行コストを数日から数時間に圧縮できる保険になります。 さらに一歩進めるなら、Fable5自身を「ハーネス設計の相談役」として使うのも効果的です。「このルールは今でも最適か」「このエージェントの責務は適切 か」をFable5に問い、答えをハーネスに埋め込む。週に1回、30分で十分です。

Fable5の深い推論力を、目の前のコード生成に直接使うのではなく、自分の「開発インフラ」を強化する方に振り向ける。この逆転の発想が、長期的に最も 大きなリターンを生みます。

## すぐに始められる3つのアクション

ここまでの内容を、いますぐ動けるレベルに落としてみると次のようになります。

**アクション１：**費用対効果を2週間かけて測るセキュリティレビュー、アーキテクチャ判断、バグ修正、定型コード生成の4種類で、同じタスクをFable5とOpus 4.8の両方にやらせて結果を比較してください。「なんとなく良さそう」ではなく、自分の作業パターンでの実測値を持つことが、最適配分の出発点です。

**アクション２：**  
ハーネスの「モデル名依存」を棚卸しするCLAUDE.mdやエージェント定義、スキル定義を開き、「Opus」「Sonnet」といったモデル名が出てくる箇所をすべてリストアップしてください。それらを「上位モデル」「軽量モデル」のような抽象表現に置き換えます。所要時間は数時間。やればやるほど、将来の値上げやtier再編に振り回されにくくなります。

**アクション３：**  
週1回、競合モデルを触る習慣をつける Claude Codeをメインに据えたまま、GeminiやGPTでの開発体験を週に1タスクだけ試してください。「いま移行する」ためではなく、「移行が必要になったときに出遅れない」ための保険です。Anthropic社の値上げ速度が競合の能力向上速度を上回ったとき、この記録が「いつ動くか」の判断材料になります。

## Fable5は強力な道具になる

Fable5は間違いなく強力な道具です。

しかし、それ以上に重要なのは、その道具に振り回されない自分の「開発インフラ」を持っているかどうかです。モデルは入れ替わります。価格は変わります。tier構造は再編されます。

Anthropic社はIPO準備中とも報じられており、収益化圧力がこれから強まる可能性も十分にあります。万が一、半年後に Fable 6が「3倍コスト」で登場しても、驚くには値しません。 でも、あなたが蓄積したルール、スキル、判断基準は、どのモデルを使うことになってもあなたの資産として残り続けます。むしろ、モデルが高度化すればするほど、「どう指示を出すか」という設計力の差が成果を分けるようになるでしょう。

Fable5が来た今こそ、その資産を点検し、磨き上げるタイミングです。「どのモデルを使うか」より「どう使いこなすか」。その一歩を、今週から始めてみませんか。
