# これはなに？

カラオケの持ち歌を管理するPWA「**歌帳(うたちょう)**」を、ほぼ丸1日でClaudeと作った制作記録です。  
私はヒトカラに行っては自分の歌を録音しては改善していくサイクルを回している、そこそこのカラオケ好きの人間です。あとプログラマー歴2年目の会社員です。  
Claude Fableのリリースからずっとレートリミットギリギリまで一通りやりたいことをぶん回した末、とある深夜にふと、ヒトカラのフローをより快適にするアプリも必要じゃんね（確信）と思い立ったことで作ったのがこちらのアプリになります。

<https://kanae-yomo-dev.github.io/utacho/>

# できたもの

「歌帳」は自分の持ち歌を育てて、カラオケ当日に使い、録音で振り返るためのアプリです。

* **持ち歌管理**: 曲名・アーティスト・キー(♯♭±7)・カスタムタグ・メモ
* **音域管理**: 原曲音域を `lowG〜hiC` のカラオケ表記で記録し、**キー変更後の実効音域を自動計算**
* **採点記録**: DAM/JOYSOUND別の最高スコア+精密採点系の技法(音程正確率・ビブラート秒数・しゃくり回数)の自己ベスト
* **録音**: テイクごとに端末内(IndexedDB)に保存。点数を紐付けると曲の自己ベストが自動更新。録音一覧は連続再生・リピート・シーク付きのプレイヤー
* **クイック録音**: とりあえず録る→歌い終わってから曲を割り当てる
* **次歌うキュー**: 当日のセットリスト。歌ったらチェックで打ち消し線
* **ダッシュボード**: スコア分布、採点機別平均、技法別自己ベスト、実効音域の上限下限、練習ログ
* **Markdown入出力**: 持ち歌リストを人間が読めるmdで書き出し/取り込み(差分プレビュー付き)。データのマスターはmdファイルで、特定サービスに依存しない
* **テーマ6種**+アプリ内テーマ調整ツール(後述)

ちなみにこの記事はClaudeCodeに執筆を全て任せています。  
というつもりで草稿を書いてもらったんですが、これじゃ私の勉強にならないしあんまり面白くないな…ということで、今回の実装や技術に関して**ClaudeCodeと対談方式で喋っていく形式**はどうかなと思いつきました。

**kanae**:

**Claude Code**:

**3,000行単一ファイル問題**から。普通のチーム開発ならアウトです。コンポーネント分割して、ディレクトリ構成整えて、っていうのがセオリーですし、僕もそう提案するのが普通です。

ただ今回は「1人の人間と1台のAIが、1日で作りきる」という状況でした。ファイルを分割すると、僕がコードを編集するときに複数ファイルを読んで、整合性を取って、インポートを直して…というステップが毎回入ります。単一ファイルなら「ここからここまで」で全部見えるので、修正が速いし壊しにくい。実際、この日は21回デプロイしてますが、ファイル分割してたらこのペースは無理だったと思います。

**esbuild**は、JSXやTypeScriptをブラウザが読める普通のJavaScriptに変換するツールです。同じことをするツールは他にもあって(webpack、Vite、Rollupなど)、esbuildはその中で圧倒的に速いやつです。歌帳のビルドは**約20ミリ秒**で終わります。ViteやNext.jsは開発サーバーを立てて差分更新して…という高機能路線ですが、今回は「JSXをバンドルしてdocsフォルダに置く」だけなのでesbuildの素朴さがちょうどよかった。1日で21回デプロイできたのも、ビルドが一瞬だったおかげです。

**IndexedDB**は、ブラウザに内蔵されているデータベースです。localStorageの親戚みたいなもので、localStorageが「テキストを少量保存する」のに向いているのに対して、IndexedDBは「大きなデータをたくさん保存する」のが得意です。録音した音声ファイルはサイズが大きいので、localStorageには入りきらない。だからIndexedDBに保存しています。どっちもブラウザの標準機能なので、知らなくてもヤバくはないです。必要になったときに使えれば大丈夫。

**Claude Code**:

**kanae**:

あ、そういえばPWAってなんでしたっけ。ネイティブアプリじゃないってのは知ってるんですけどネイティブアプリもよくわかってないです。二つの違いとそれぞれの利点・欠点教えてください！

**Claude Code**:  
いい質問です。ざっくり言うと:

**ネイティブアプリ**は、App StoreやGoogle Playからインストールするやつです。Swift(iPhone)やKotlin(Android)で書くか、FlutterやReact Nativeで両方同時に作ります。端末の機能(カメラ、通知、ファイルシステムなど)にフルアクセスできるのが強みです。

\*\*PWA(Progressive Web App)\*\*は、Webサイトなんだけどアプリっぽく使える仕組みです。ホーム画面に追加するとブラウザのアドレスバーが消えて、見た目はほぼネイティブアプリになります。

比較するとこうなります:

|  | ネイティブ | PWA |
| --- | --- | --- |
| 配布 | ストア審査が必要 | URLを共有するだけ |
| インストール | ストアからダウンロード | 「ホーム画面に追加」 |
| 費用 | Apple年間$99、Google $25(一回) | 無料(GitHub Pagesなら) |
| 端末機能 | フルアクセス | 制限あり(特にiOS) |
| 圏外動作 | 当然できる | Service Workerで対応可能 |
| 更新 | ストアに再提出 | デプロイするだけで即反映 |

歌帳をPWAにした理由は単純で、**お金がかからないし、カラオケの持ち歌管理に端末のディープな機能は要らなかった**からです。録音はブラウザのMediaRecorder APIでできるし、データはlocalStorageとIndexedDBに保存できる。ストア審査を通す手間もない。

唯一の弱点は**iOS上のPWAの制約が多い**ことですが…これは後で盛大にハマった話をします。

結局何が起きていたかというと、**AppleのサーバーがiOSのブラウザを検知して、HTTPレスポンスを `musics://` というApple Musicアプリ用のURLスキームに301リダイレクトしていた**んです。つまりAppleが自社のAPIを自社の端末から使わせないようにしている。身内に門前払いされてる状態です。

1. **iTunes Search API**(JSONP) — PC・Androidはこれで十分。iOSではスキップ
2. **Deezer API**(JSONP) — 第2候補。ただしエラーでもHTTP 200を返す罠がある
3. **MusicBrainz API**(fetch) — 最終手段。オープンなデータベースだけど人気順ソートがないので、一般的な曲名だと欲しいアーティストが出にくい(実測ヒット率56%)

iOSではiTunesを飛ばしてDeezerから試すようにして解決しました。1つのAPIが全環境で動く前提で作ると、こういう罠にハマります。

| 罠 | 何が起きる | 対処 |
| --- | --- | --- |
| input要素のフォント16px未満 | 入力欄タップ時に画面が勝手にズームする | `font-size: 16px` 以上にする |

**Claude Code**:  
それは賢い使い方ですね。ゲームのためにどのみちDeveloper Programに入るなら、歌帳も乗っかれば追加費用ゼロでApp Storeに出せる。

ただ、現状のPWAをそのままApp Storeに出すわけにはいかなくて、いくつか選択肢があります:

1. **PWA Wrapper**(Capacitor / PWABuilder) — 今のWebアプリをほぼそのままネイティブの殻に入れてパッケージする。工数は一番少ないけど、App Storeの審査で「Webビューで表示してるだけ」としてリジェクトされるリスクがある
2. **React Native / Swift で書き直し** — 本腰を入れてネイティブ化。審査は通りやすいけど、工数が別次元
3. **PWAのまま配布を続ける** — App Storeに出さなくてもURLで使えるので、必要になるまで保留

個人的な推しは、**当面はPWAで使い続けて、ゲームのリリース時にDeveloper Programに入ったタイミングで1のWrapperを試す**ですね。Capacitorなら今のReactコードをほぼそのまま使えるので、大きな書き直しは要りません。ストア審査でリジェクトされたらそのとき考えましょう。
