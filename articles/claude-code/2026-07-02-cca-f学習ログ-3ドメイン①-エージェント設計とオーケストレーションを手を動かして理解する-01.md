## そもそも「エージェント」とは

### 単発呼び出し・ワークフロー・エージェントの違い

まず「エージェントとは何か」を整理します。Claude（LLM）の使い方は、大きく3段階に分けられます。

**段階1：単発呼び出し（Single Call）**

ユーザーが質問を送り、Claude が1回回答を返して終わり。最もシンプルな使い方です。

```
ユーザー → 「〇〇を教えて」 → Claude → 「〇〇です」 → 終了
```

**段階2：ワークフロー（Workflow）**

複数の Claude の呼び出しを人間が設計した順序でつなぐ仕組みです。処理の流れ（どのステップで何をするか）はプログラム側があらかじめ決めています。Claude は「各ステップでの応答を出力する」役割を担います。

```
ステップ1: テキストを要約 → ステップ2: 要約をもとにメール文を生成 → 完了
（次のステップに何を渡すかはプログラム側が決定）
```

**段階3：エージェント（Agent）**

「**モデル（Claude）自身が次に何をするかを決め、ツールを使いながら目標に向かって反復する**」仕組みです。プログラムが全ステップをあらかじめ決めるのではなく、Claude が「次はこのツールを使う」「もう少し情報が必要だから追加で検索する」といった判断を自分で行います。

エージェントのポイント：

* モデルが「次の手」を自分で選ぶ
* ツール（外部の関数・API）を呼び出せる
* 目標が達成されるまでループを繰り返す
* 事前に全ステップを人間が設計しない

![図1: 単発呼び出し・ワークフロー・エージェントの段階図](https://static.zenn.studio/user-upload/deployed-images/f0529caaefff683b64fb9232.png?sha=63fc16ec9c9266fc5d3c3b651b0b3450bd9ba58c)  
*図1: 左から「単発呼び出し」「ワークフロー」「エージェント」の3段階。右に行くほど Claude が自律的に判断する。「自分で次の手を決める」がエージェントの境界。*

### なぜエージェントが必要か

単発呼び出しで解決できる問題もたくさんありますが、次のような場面では不十分になります。

* **必要な情報が最初からわからない**：「この製品の最新価格を調べて比較して」→ 調べる前に何を調べるかが決まらない
* **ステップが動的に変わる**：途中の結果によって次の処理が変わる
* **外部システムとやり取りが必要**：データベースの検索・ファイル操作・API 呼び出し

こういった場合に「Claude に判断させる」エージェントのアプローチが力を発揮します。

## エージェントループの基礎

エージェントが目標に向かって「送信 → 結果を見て次の手を決める → また送信」を繰り返す仕組みを、ここでは**エージェントループ**と呼びます。まず材料となる `tool_use` を復習してから、ループ本体を組み立てます。

#2 で、構造化出力の手段として `tool_use` を使う例を示しました。`tool_choice={"type": "tool", "name": "..."}` で特定のツールを強制的に呼ばせ、JSON を取り出す使い方です。

今回はこれを「**行動**」として使います。Claude に「このツール（関数）を呼んで実際の作業をしてほしい」と頼む使い方です。

`tool_use` のブロック構造（#2 のおさらい）：

```
# レスポンスの content の中に tool_use ブロックが入ってくる
# block.type == "tool_use" のとき、以下のフィールドがある
# block.id      : ツール呼び出しの一意な ID（tool_result 返却時に使う）
# block.name    : 呼び出すツールの名前
# block.input   : ツールに渡す引数（dict 形式）
```

エージェントとして使う場合の違いは「tool\_result（実行結果）をモデルに返して続きの処理をさせる」点です。`tool_result` は `{"type": "tool_result", "tool_use_id": ..., "content": ...}` という形のブロックで、どの `tool_use` への結果かを `tool_use_id` で対応づけ、`user` ロールのメッセージに入れて返します（具体例は後述のコード）。

### エージェントループの仕組み

エージェントループは次の流れで動きます。

1. メッセージを Claude に送る
2. `stop_reason` を確認する
3. `stop_reason == "tool_use"` なら：ツールを実行して結果を返し、1に戻る
4. `stop_reason == "end_turn"` などの終了条件になったら：ループを終了する

![図2: エージェントループ図](https://static.zenn.studio/user-upload/deployed-images/033eb75b70e23ed5d79fe7e2.png?sha=9ec0a1c4fbc30856b38890992c717a18e64f3a86)  
*図2: メッセージ送信 → stop\_reason を確認 → tool\_use ならツール実行して tool\_result を返し再送 → end\_turn で完了。最大反復でループを打ち切る安全機構もあわせて示す。*

### 最小エージェントループのコード

実際にエージェントループを手書きで実装します。ここでは「現在の気温を調べる」ダミーツールを使ってループを回す例を示します。

```
import anthropic
import json

client = anthropic.Anthropic()

# -----------------------------------------------
# ① ツール定義
# -----------------------------------------------
tools = [
    {
        "name": "get_weather",
        # description は具体的に書く。Claude はここを読んで「このツールを使うべきか」を判断する。
        "description": "指定した都市名の現在の気温（摂氏）を返す。"
                       "リアルタイムの気象情報が必要なときに使う。",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "気温を調べる都市名（例: Tokyo, Osaka）"
                }
            },
            "required": ["city"]
        }
    }
]

# -----------------------------------------------
# ② ダミーのツール実行関数（実際はここで外部 API 等を呼ぶ）
# -----------------------------------------------
def run_tool(name: str, tool_input: dict) -> str:
    """ツール名と引数を受け取り、実行結果を文字列で返す。"""
    if name == "get_weather":
        city = tool_input.get("city", "")
        # 本来は外部 API を呼ぶ。ここではダミー値を返す。
        return f"{city} の現在の気温は 25°C です（ダミーデータ）。"
    return "該当するツールが見つかりませんでした。"

# -----------------------------------------------
# ③ エージェントループ
# -----------------------------------------------
MAX_ITERATIONS = 10  # 暴走防止：最大反復回数を設定する

messages = [
    {"role": "user", "content": "東京と大阪の今日の気温を教えてください。"}
]

# range(MAX_ITERATIONS) で反復回数に上限を設ける（while True より暴走しにくい）
for iteration in range(MAX_ITERATIONS):
    # Claude にリクエストを送る
    response = client.messages.create(
        model="claude-sonnet-4-6",  # 最新のモデル ID は公式ドキュメントで確認
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    # ---- stop_reason を確認する ----
    if response.stop_reason == "end_turn":
        # 通常の完了。ループを抜ける。
        print("=== 完了 ===")
        for block in response.content:
            if hasattr(block, "text"):
                print(block.text)
        break

    if response.stop_reason == "max_tokens":
        # 出力が途中で切れている。本番では別途ハンドリングが必要。
        print("警告: max_tokens に達しました。出力が不完全な可能性があります。")
        break

    if response.stop_reason != "tool_use":
        # 想定外の stop_reason（refusal, pause_turn, stop_sequence 等）
        # refusal は安全上の理由で応答が拒否された状態。ここでは打ち切る。
        print(f"想定外の stop_reason: {response.stop_reason}")
        break

    # ---- tool_use の処理 ----
    # まず今の response.content 全体を messages に追加する（assistant ターン）
    messages.append({"role": "assistant", "content": response.content})

    # response.content には複数の tool_use ブロックが含まれることがある（並列実行）。
    # 全てを処理して 1 つの user メッセージにまとめる。
    tool_results = []

    for block in response.content:
        if block.type != "tool_use":
            continue  # テキストブロック等はスキップ

        print(f"ツール実行: {block.name}({block.input})")
        result_text = run_tool(block.name, block.input)

        # tool_result ブロックを積む
        tool_results.append({
            "type": "tool_result",
            "tool_use_id": block.id,   # tool_use の id と対応させる（必須）
            "content": result_text
        })

    # 全 tool_result を 1 つの user メッセージに入れて返す（並列の場合も同様）
    messages.append({
        "role": "user",
        "content": tool_results
    })

else:
    # for ループが break されず MAX_ITERATIONS 回すべて回りきったときだけ実行される
    # （＝上限に達しても完了しなかった）
    print(f"警告: {MAX_ITERATIONS} 回の反復に達しましたが完了しませんでした。")
```

このコードのポイントをまとめます。

* **ループの終了は `stop_reason` で判定する**：「テキストに "完了" が含まれるかどうか」で判定するアンチパターンは使わない
* **`MAX_ITERATIONS` でループを打ち切る**：モデルが永遠にツールを呼び続ける「暴走」を防ぐ
* **tool\_result は必ず `tool_use_id` を対応させる**：どのツール呼び出しへの結果かをモデルが特定するために使う
* **並列ツールも全結果を 1 メッセージにまとめる**：後述の並列実行の節で詳しく説明する

## ツールの設計

### name / description / input\_schema

ツール定義の3つの要素を確認します。

| フィールド | 役割 |
| --- | --- |
| `name` | ツールの識別子。Python の関数名と同じルール（英数字・アンダースコア）が無難 |
| `description` | **モデルが「このツールを使うかどうか」を判断するための説明文**。具体的な使用場面・条件・何を返すかを書く |
| `input_schema` | ツールの引数の型・説明・必須フィールドを定義する。Python の辞書（dict）で「このキーは文字列」「このキーは必須」と書く記法（JSON Schema）を使う |

### 複数ツールの定義例

```
tools = [
    {
        "name": "search_web",
        "description": "指定したキーワードで Web 検索を行い、上位5件のタイトルとURLを返す。"
                       "最新情報や外部サイトの情報が必要なときに使う。",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "検索キーワード（例: 'Python 3.13 新機能'）"
                }
            },
            "required": ["query"]
        }
    },
    {
        "name": "read_file",
        "description": "指定したファイルパスのテキストファイルを読み込み、内容を返す。"
                       "ローカルファイルの内容を参照する必要があるときに使う。",
        "input_schema": {
            "type": "object",
            "properties": {
                "filepath": {
                    "type": "string",
                    "description": "読み込むファイルの絶対パス（例: '/tmp/data.txt'）"
                }
            },
            "required": ["filepath"]
        }
    },
    {
        "name": "calculate",
        "description": "算術式を受け取り、計算結果を数値で返す。"
                       "正確な数値計算が必要なときに使う（Claude 自身による暗算の代わり）。",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {
                    "type": "string",
                    "description": "計算する算術式（例: '(1234 * 5678) / 100'）"
                }
            },
            "required": ["expression"]
        }
    }
]
```

`tool_choice` で「どのツールをどう使わせるか」を制御できます。

| 値 | 意味 |
| --- | --- |
| `{"type": "auto"}` | Claude が必要かどうかを自分で判断する（デフォルト） |
| `{"type": "any"}` | 必ず何らかのツールを使わせる |
| `{"type": "tool", "name": "..."}` | 指定したツールを必ず使わせる（#2 の構造化出力と同じ手法） |
| `{"type": "none"}` | ツールを使わない（通常のテキスト応答のみ） |

通常のエージェントでは `"auto"` を使い、Claude が自分でツールの要否を判断するようにします。

## 並列ツール実行

Claude は 1 回の応答に**複数の `tool_use` ブロック**を返すことがあります。例えば「東京と大阪の気温を同時に調べて」というリクエストでは、2つの `tool_use` が同時に返ってくる場合があります。

**重要なルール**：複数の `tool_use` が返ってきた場合、**全てを実行し、1 つの `user` メッセージに全 `tool_result` をまとめて返す**必要があります。1 つずつ別々のメッセージで返すと API エラーになります。

![図4: 並列ツール実行図](https://static.zenn.studio/user-upload/deployed-images/ac4e9f10759557b64e436f3a.png?sha=e3e42a22193e3476b05c5bef39f9634634e43f36)  
*図4: 1 レスポンスに複数 tool\_use が入り、各ツールを実行後、全 tool\_result を 1 つの user メッセージにまとめて返す流れ。*

先ほどのエージェントループのコード（`③ エージェントループ`セクション）は既に並列ツール実行に対応しています。`response.content` の `tool_use` ブロックを全て処理して `tool_results` リストに積み、最後に 1 つの `user` メッセージとして送っています。

並列処理の実行順序について、実際に複数の `tool_use` が来た場合にどの順番で実行するかはアプリ側の判断です。独立したツール（結果が互いに依存しない）は `concurrent.futures` 等で本当に並列実行することもできます（`concurrent.futures` は Python 標準ライブラリで、API 呼び出しのような待ち時間の多い処理を並行して動かすのに向いています）。

```
import concurrent.futures

def execute_tool_calls(tool_use_blocks):
    """複数の tool_use ブロックを並列で実行する例。"""
    def execute_one(block):
        result_text = run_tool(block.name, block.input)
        return {
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": result_text
        }

    # ThreadPoolExecutor で tool_use ブロックを並列実行する
    with concurrent.futures.ThreadPoolExecutor() as executor:
        futures = [executor.submit(execute_one, block) for block in tool_use_blocks]
        results = [f.result() for f in futures]

    return results  # 全 tool_result のリスト（1 つの user メッセージに入れて返す）
```

## ツールのエラー処理

ツールの実行が失敗した場合、`tool_result` の `is_error` フラグを使ってモデルに失敗を伝えます。モデルはエラーを受け取ると、エラー内容を踏まえて別の手段を試みたり、ユーザーに状況を説明したりします。

```
def run_tool_with_error_handling(name: str, tool_input: dict, tool_use_id: str) -> dict:
    """ツールを実行し、結果（失敗時は is_error 付き）を tool_result ブロックで返す。"""
    try:
        result_text = run_tool(name, tool_input)
        return {
            "type": "tool_result",
            "tool_use_id": tool_use_id,  # どの tool_use への結果かを必ず対応づける
            "content": result_text
            # is_error を省略（または False）にすると「成功」として扱われる
        }
    except Exception as e:
        # エラー時は is_error: True と、人間が読める説明を content に入れる
        return {
            "type": "tool_result",
            "tool_use_id": tool_use_id,
            "is_error": True,
            "content": f"ツール実行でエラーが発生しました: {type(e).__name__}: {str(e)}"
        }

# エージェントループ内での使い方
tool_results = []
for block in response.content:
    if block.type != "tool_use":
        continue

    # block.id（tool_use の ID）をそのまま渡す
    result = run_tool_with_error_handling(block.name, block.input, block.id)
    tool_results.append(result)

messages.append({"role": "user", "content": tool_results})
```

**エラー処理の考え方**：

* `is_error: True` にしてもループは止まらず、Claude がエラーを見て次の判断をする
* エラーメッセージは短く・具体的に書く（「失敗」だけでは Claude が対応策を立てにくい）
* 同じツールが何度もエラーになる場合は、ループの上限（`MAX_ITERATIONS`）に頼るか、連続エラー回数を別途カウントして打ち切る

## オーケストレーション（複数エージェント）

1 つのエージェントで全部やろうとすると、コンテキストが膨らみすぎたり、複雑なタスクを1つのプロンプトに詰め込みすぎたりする問題が出てきます。そこで「**複数のエージェントにタスクを分けて連携させる**」オーケストレーションが必要になります。

オーケストレーション（orchestration）とは：音楽でいうオーケストラの指揮者（conductor）が各楽器（奏者）に役割を割り振るように、複数のエージェントやツールを組み合わせて1つの目標を達成する設計のことです。

### coordinator-subagent パターン

最も基本的なオーケストレーションの形は「**coordinator（コーディネーター）が複数の subagent（サブエージェント）にタスクを分解・委譲する**」パターンです。

**ポイント**：

* **coordinator**：全体の目標を把握し、タスクを分解して各 subagent に割り振る。結果を集約して最終的な回答を作る
* **subagent**：割り当てられた1つのサブタスクを実行する。coordinator の存在を知る必要はない
* **独立したコンテキスト**：各 subagent は独立したコンテキスト（ここでは会話履歴＝`messages` のこと）を持つ。coordinator の `messages` に subagent の作業詳細を全部ためこまずに済む

独立コンテキストのメリット：長いタスクでも coordinator のコンテキストが膨らまない。複数の subagent を並列に動かしやすい。

```
import anthropic
import concurrent.futures

client = anthropic.Anthropic()

def run_subagent(task_description: str) -> str:
    """
    サブエージェントとして独立したコンテキストで1つのサブタスクを実行する。
    coordinator から task_description（サブタスクの指示）を受け取り、結果を返す。
    """
    response = client.messages.create(
        model="claude-sonnet-4-6",  # 最新のモデル ID は公式で確認
        max_tokens=1024,
        messages=[
            {"role": "user", "content": task_description}
        ]
    )
    # この例はシンプルな1回呼び出し。実際には subagent もループを持てる。
    return response.content[0].text

def run_coordinator(user_goal: str) -> str:
    """
    コーディネーターとして動く。ユーザーゴールをサブタスクに分解し、
    各サブエージェントに並列委譲して結果を集約する。
    """
    # ステップ1: coordinator がサブタスクを洗い出す
    plan_response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="あなたはタスク分解の専門家です。ユーザーのゴールを、独立して実行できる"
               "2〜3個のサブタスクに分解してください。サブタスクを箇条書きで出力してください。",
        messages=[
            {"role": "user", "content": f"ゴール: {user_goal}"}
        ]
    )
    plan_text = plan_response.content[0].text

    # ステップ2: サブタスクを（簡易的に）行ごとに分割する
    # 注意: これはデモ用の簡易パース。Claude が箇条書き以外の形式（番号付きリスト等）
    #       で返すと subtasks が空になり得る。本番では構造化出力（#2 参照）で受け取るのが堅実。
    subtasks = [line.strip("- ").strip()
                for line in plan_text.split("\n")
                if line.strip().startswith("-")]

    print(f"サブタスク一覧:\n{chr(10).join(subtasks)}\n")

    # ステップ3: 各サブタスクを subagent に並列委譲する
    with concurrent.futures.ThreadPoolExecutor() as executor:
        futures = {executor.submit(run_subagent, task): task for task in subtasks}
        subagent_results = {}
        for future in concurrent.futures.as_completed(futures):
            task = futures[future]
            subagent_results[task] = future.result()

    # ステップ4: coordinator が結果を集約して最終回答を作る
    aggregated = "\n".join(
        f"サブタスク「{task}」の結果:\n{result}\n"
        for task, result in subagent_results.items()
    )
    final_response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="あなたは情報を集約するアシスタントです。複数のサブタスクの結果をもとに、"
               "ユーザーのゴールに対する最終回答をまとめてください。",
        messages=[
            {"role": "user", "content": f"ゴール: {user_goal}\n\n{aggregated}"}
        ]
    )
    return final_response.content[0].text

# 使い方
result = run_coordinator("Python と JavaScript の主な違いを比較してまとめて")
print(result)
```

### ワークフロー型パターンの地図

Anthropic の公開ガイダンス「Building Effective Agents」では、エージェント的な設計として **ワークフロー型（workflows）**と**自律エージェント型（agents）** を区別しています。

ワークフロー型はプログラム側が全体のフローを制御するため、予測可能で信頼性が高い反面、動的に対応できる範囲が限られます。自律エージェント型は Claude 自身が判断するため、動的な問題に対応できる反面、挙動の予測可能性が下がります。

以下の4パターンは[「Building Effective Agents」](https://www.anthropic.com/research/building-effective-agents)で紹介されているワークフロー型の代表的なパターンです。

| パターン | 概要 | 使い場面 |
| --- | --- | --- |
| **Chaining（チェーン型）** | 前のステップの出力を次のステップの入力にする。処理を順番につなぐ | 翻訳→校正→フォーマット変換のように、順序が決まっている処理 |
| **Routing（ルーティング型）** | 入力を分類して、適切な専門の処理フローに振り分ける | カスタマーサポートで「請求」「技術」「一般」に振り分けるなど |
| **Parallelization（並列化型）** | 独立したサブタスクを並列に実行して結果を合わせる | 複数ドキュメントの同時要約、複数視点での審査など |
| **Orchestrator-Workers（オーケストレーター-ワーカー型）** | coordinator-subagent と同じ構造。オーケストレーターが動的にタスクを分解してワーカーに委譲する | タスクが事前に確定できない、動的なサブタスク分解が必要な場合 |

**いつエージェント化すべきか**：

Anthropic の公開ガイダンスでは「シンプルで予測可能なフローには ワークフロー型を使い、動的な判断が必要な場面に限ってエージェント型を使う」という考え方を推奨しています。エージェントにするほど Claude に任せる判断が増え、エラーが積み重なるリスクも上がります。

![図6: ワークフロー型パターン地図](https://static.zenn.studio/user-upload/deployed-images/a11b913018d3033bbafceea0.png?sha=c0d0e3412320842e86de1e31568e8c73c75c7281)  
*図6: Chaining / Routing / Parallelization / Orchestrator-Workers の4パターンと「自律エージェント」の位置づけ。左に行くほどプログラム制御が強く、右に行くほど Claude の自律判断が増える。*

ここまで、エージェントループも複数エージェントの連携も手書きで実装してきました。手書きは仕組みが透明で学習に向きますが、実際のプロダクトでは SDK の高レベルの仕組みを使うほうが安全・簡潔になります。

Anthropic の Python SDK には `client.beta.messages.tool_runner`（正確な記法は公式で確認してください）という仕組みがあり、手書きの `for` / `while` ループを自動化できます。

概念としては「ツール定義とツール実行関数を渡すと、ループ管理・tool\_result の返却・終了判定を SDK が肩代わりする」ものです。

```
# 概念をつかむためのイメージ（最新の正確な記法は公式ドキュメントで確認してください）

from anthropic import Anthropic, beta_tool

client = Anthropic()

# ツール関数に @beta_tool デコレーターをつけると SDK が認識する
@beta_tool
def get_weather(city: str) -> str:
    """指定した都市名の現在の気温（摂氏）を返す。"""
    return f"{city} の現在の気温は 25°C です（ダミーデータ）。"

# tool_runner がループ（送信 → ツール実行 → 再送）を自動で回す
runner = client.beta.messages.tool_runner(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[get_weather],
    messages=[{"role": "user", "content": "東京の気温を教えて"}],
)
# 実行結果の取り出し方を含む正確な記法は、beta 仕様のため公式ドキュメントで確認してください。
```

### さらに上位：Claude Agent SDK / Managed Agents

Claude Agent SDK は、複数のエージェントの連携・タスク分解・サブエージェントへの委譲などを扱うための高レベルの開発キットです。手書きループや Tool Runner よりも多くの機能（コンテキスト管理・エージェント間通信・タスクキュー等）を提供します。

Managed Agents は、さらにその上位で、エージェントの実行環境そのものをマネージドサービスとして提供する仕組みです（2026年7月時点での提供状況と利用条件は、公式サイトで確認してください）。

**抽象度の3段階**：

1. **手書きループ**：`stop_reason` を自分で見て、`tool_result` を自分で返す。透明性が高く、学習向き
2. **SDK の Tool Runner**：ループ管理を SDK に任せる。コードが短くなる
3. **Claude Agent SDK / Managed Agents**：複数エージェントの連携・実行環境管理まで担う

最初は手書きループで仕組みを理解し、プロダクト開発では SDK や Managed Agents を検討する、という進め方が現実的です。

## ハマりどころ

エージェント実装でよく詰まるポイントをまとめます。

### ① ループ終了を自然言語で判定する（アンチパターン）

```
# BAD: 応答テキストに「完了」が含まれるかどうかでループを終わらせる
response_text = response.content[0].text
if "完了" in response_text or "終了" in response_text:
    break
```

この方法は Claude が「処理が完了しました」と書かない限りループが終わらないケースがあり、また意図せず「完了」という単語が出てきたときに途中で終了してしまいます。`stop_reason` を使うのが正しいアプローチです。

### ② `MAX_ITERATIONS` を設定しない

Claude が想定外のツール呼び出しを繰り返した場合に無限ループになります。必ず上限を設定してください。

```
# BAD: tool_result を 1 つずつ別のメッセージで返す
for block in tool_use_blocks:
    messages.append({"role": "user", "content": [{"type": "tool_result", ...}]})
    response = client.messages.create(...)  # ← 途中でリクエストするのは誤り
```

並列 `tool_use` が来た場合、全ての `tool_result` を 1 つの `user` メッセージにまとめてから次のリクエストを送ります。

### ④ コンテキストが膨らみすぎる

ループを重ねるごとに `messages` リストが長くなり、コンテキストウィンドウの上限に近づきます。長いエージェントタスクでは、古い会話履歴の圧縮・要約・削除が必要になります。コンテキスト管理の詳細は後続回（ドメイン⑤）で扱います。

### ⑤ ツールを冪等（べきとう）に設計していない

冪等（idempotent）とは「何度実行しても同じ結果になる」性質です。エージェントはツールを意図せず複数回呼ぶことがあります。例えば「ファイルを書き込む」ツールが冪等でない（実行ごとにデータが追記される）と、ループのエラーが実世界の副作用を起こします。副作用のあるツール（書き込み・送信・削除など）は特に慎重に設計してください。

### ⑥ stop\_reason の取りこぼし

`"end_turn"` だけをチェックして他の `stop_reason` を無視していると、`"max_tokens"` で出力が途中で切れたままループが終了するケースがあります。`stop_reason` の全パターンを考慮した分岐を書くことをお勧めします。

## まとめ

この記事で扱った内容を整理します。

**エージェントとは何か**

* 「単発呼び出し → ワークフロー → エージェント」の3段階で捉える
* エージェント = Claude 自身が次の手（ツール使用）を決め、目標に向かって反復する仕組み

**エージェントループの設計**

* ループ終了は `stop_reason` で判定する（自然言語判定はアンチパターン）
* `stop_reason == "tool_use"` でツール実行 → `tool_result` を返して再送
* `stop_reason` が `tool_use` 以外になったらループを抜ける（`max_tokens` など異常系は別途警告する）
* `MAX_ITERATIONS` による暴走防止は必須

**ツール設計**

* `name` / `description` / `input_schema` の3要素で定義する
* `description` は「何を返すか」「どんなときに使うか」「対象外は何か」を具体的に書く
* `tool_choice` で「必ず使わせる」「任せる」「使わない」を制御できる

**並列ツール実行とエラー処理**

* 複数 `tool_use` は全て実行して 1 つの `user` メッセージにまとめる
* 失敗した場合は `is_error: True` + 具体的なメッセージで Claude に伝える

**オーケストレーション**

* coordinator-subagent パターン：タスクを分解して並列委譲・結果を集約
* ワークフロー型4パターン（Chaining / Routing / Parallelization / Orchestrator-Workers）
* シンプルなフローにはワークフロー型を使い、必要な場面だけエージェント化する

**抽象度の選択肢**

* 手書きループ → SDK Tool Runner → Claude Agent SDK / Managed Agents の3段階
