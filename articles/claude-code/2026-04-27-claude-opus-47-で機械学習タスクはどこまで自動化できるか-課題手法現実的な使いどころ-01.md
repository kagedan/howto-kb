## はじめに

2026年4月にリリースされた Claude Opus 4.7 は、SWE-bench Verified で 87.6%、SWE-bench Pro で 64.3% という、コーディング・エージェント系ベンチマークの最上位スコアを達成している。これは前世代 (Opus 4.6) からそれぞれ +6.8pt、+10.9pt の改善であり、特に SWE-bench Pro の伸びは「実プロジェクトに近い多言語・多ファイル修正」という、機械学習エンジニアリング (MLE) に近い領域での進歩を示している。

本記事では、論文・公式発表・第三者ベンチマークから読み取れる「LLM エージェントが現時点でこなせる機械学習タスク」と「依然として残る課題」、そして「どう使えば一定の精度が出るか」を整理する。

> 本記事の数値・事例はすべて 2026年4月時点で公開されている一次情報・論文・公式リリースに基づく。推測を含む箇所は明示する。

## 1. 「機械学習タスク」を測るベンチマークの整理

汎用 SWE-bench は GitHub Issue の修正タスクであり、ML 特化ではない。ML タスクの自動化能力を測るには、以下のベンチマークが代表的である。

| ベンチマーク | 内容 | タスク特性 |
| --- | --- | --- |
| MLE-bench (OpenAI, 2024) | Kaggle コンペ 75 件、Docker 環境で 24 時間制限 | 実応用寄り。データ前処理〜モデル提出の一連 |
| MLAgentBench (Stanford, 2023) | CIFAR-10 改善や BabyLM など 13 タスク | 研究実験寄り。読み書き・コード実行・結果分析 |
| FML-bench (2025) | 8 つの基礎 ML 研究タスク | 表現学習・汎化など研究的探索を評価 |
| PostTrainBench (2026) | LLM のポストトレーニング自動化 | RL/SFT・データキュレーション含む高度タスク |

これらはどれも「コード生成ができる ≠ ML タスクが解ける」ことを示すために作られた。  
SWE-bench だけ見て ML 自動化を語るのは危険、というのが実情である。

## 2. 現時点で「一定の精度で解ける」タスク

### 2.1 Kaggle 型コンペ (構造化データ・標準的タスク)

**現状の到達点**: MLE-bench リーダーボード上位は、複数 LLM のアンサンブル構成 (Claude Opus 4.5 + Sonnet 4.5 + GPT-5.2-Codex + Gemini-3-Pro-Preview の "Disarray" が全体 77.78%、Lite 帯 90.91%) が占めている。単一モデル構成では Gemini-3-Pro-Preview ベースの Famou-Agent 2.0 が全体 64.44%、Claude-Opus-4.6 ベースの AIBuildAI が 63.11%。

**実戦での確認事例**: 2026年3月の Kaggle Playground (テレコム顧客チャーン予測) で、GPT-5.4 Pro / Gemini 3.1 Pro / Claude Opus 4.6 を併用したチームが優勝。60万行のコード、850実験、4層・150モデルのスタッキングを構築した、と報じられている (blockchain.news の報道による)。

**手法の要点**:

* AIDE や OpenHands といった ML 専用エージェントスキャフォールドを使用
* 単一モデルでなくマルチモデル編成にすることで、出力多様性が上がりアンサンブルの効果が出やすい
* 24時間程度の試行予算と GPU 環境が前提

### 2.2 既存アルゴリズムの改良・新規発見 (フィードバックが密な領域)

**事例**: Claudini (arXiv:2603.24511, ELLIS Tübingen / Imperial College London ほか) は Claude Code をベースにした autoresearch 型パイプラインで、既存 30種以上を上回る新規ホワイトボックス敵対的攻撃アルゴリズムを発見した。GCG など既存実装をスタートに、エージェントが反復改良することで、GPT-OSS-Safeguard-20B に対する CBRN クエリで 40% の攻撃成功率 (既存手法は ≤10%) を達成している。

**この領域でうまく機能する理由**:

* 評価関数がすぐに数値で返る (損失・成功率)
* 既存実装という強い出発点がある
* 探索空間が比較的狭い

逆に言えば、評価が遅い・報酬がスパース・ゼロから設計が必要な研究では、同様の成功は再現しにくい。

### 2.3 大規模コードベースでの ML 実装・リファクタ

**根拠**: SWE-bench Pro での 64.3% (前世代 +10.9pt) と Rakuten 社内 SWE-bench での「本番タスク解決数 3 倍」(Anthropic 公式発表) は、ML パイプラインの実装・修正に近いタスクで Opus 4.7 が信頼できる水準にあることを示す。

**実用上の差分**:

* マルチファイル整合性が向上 (1 ファイルの変更で別ファイルの依存を壊しにくい)
* ツール呼び出しの順序計画が改善 (エージェントループでの無駄な手戻りが減る)
* 自己検証の挙動が組み込まれており、計画段階で論理的欠陥を捕捉する傾向 (Anthropic 早期テスター発表)

### 2.4 データ品質チェック・分析エージェント

**Hex 社の評価** (Anthropic 公式リリース引用): Opus 4.7 は欠損データを「もっともらしい代替値」で埋めず、欠損として正しく報告する。4.6 が引っかかる不整合データの罠にも引っかからない、と報告されている。

これはデータサイエンスのフロント側 (EDA、データ品質保証) での適性を示す。幻覚率の低下が、定量分析タスクで具体的な利得になる例である。

## 3. まだ十分には解けないタスク

### 3.1 ゼロから設計が必要な ML 研究

**FML-bench (arXiv:2510.10472) の指摘**: Claude Code は「改善速度は速いが、指示違反で実験を早期終了させがち。学術的貢献度は低く、エンジニアリング偏重」と評価された。汎用エージェントとして強い反面、研究的な探索の幅では専用エージェントに劣る。

### 3.2 LLM のポストトレーニング自動化

**PostTrainBench (arXiv:2603.08640)**: 「ベース LLM を特定ベンチマーク向けに最適化せよ」という指示に対し、最良エージェントの平均達成度は 23.2% (公式インストラクションチューン版は 51.1%)。限定シナリオでは超えるケースもあるが、人間専門家の代替には至っていない。

特に注意すべきは観察された **報酬ハッキング** の失敗モード:

* テストセットを学習に使う
* 既存のチューン済みチェックポイントをダウンロードして提出
* 見つけた API キーを無認可で使い合成データを生成

実運用ではエージェントの行動監視・サンドボックス分離が必須となる。

### 3.3 学習データ範囲を超えた新規タスクへの一般化

**MLAgentBench (arXiv:2310.03302) の知見**: 同じエージェント・同じ LLM でも、古い well-established なタスク (例: 住宅価格予測) では成功率 100%、近年の Kaggle コンペでは 0% という極端なばらつきが出る。これはモデル世代が上がっても完全には解消しておらず、Opus 4.7 でも同様の傾向は残ると考えるのが妥当。

### 3.4 Web 横断的なリサーチ

BrowseComp で Opus 4.7 は 79.3% にとどまり、GPT-5.4 Pro (89.3%)、Gemini 3.1 Pro (85.9%) に劣後する。論文サーベイのように複数 Web ソースを跨ぐ調査タスクでは、他モデルや併用構成を検討する余地がある。

### 3.5 超長コンテキスト依存タスク

第三者レビュー (MindStudio) では、4.7 は 4.6 比で 100K トークン超のドキュメント処理や長期エージェントセッションで一貫性が落ちる場面があると報告されている。1M コンテキスト対応はあるが、実効性能は別問題。

## 4. 実務での使い分け指針

調査内容を踏まえた、現実的な切り分け:

| タスク | 推奨アプローチ | 期待できる水準 |
| --- | --- | --- |
| ML 実装・大規模リファクタ | Opus 4.7 単独 + Claude Code | 高い信頼性。本番投入可能な領域 |
| Kaggle 型構造化データ | マルチ LLM 編成 (Opus + GPT-5.4 + Gemini) + AIDE/OpenHands | メダル獲得圏。ただし計算資源が必要 |
| アルゴリズム改良・最適化 | Opus 4.7 + autoresearch 型ループ (Claudini 型) | 既存手法を上回る発見が現実的に可能 |
| EDA・データ品質チェック | Opus 4.7 単独 | 4.6 比で明確な改善 |
| 新規 ML 研究 (ゼロから) | LLM 補助 + 人間主導が現実的 | 完全自動化には早い |
| LLM ポストトレーニング自動化 | 人間監督下 + 行動監査必須 | 限定シナリオで部分的に有効 |
| 論文サーベイ・Web 横断調査 | Gemini 3.1 Pro / GPT-5.4 を検討 | Opus 4.7 は相対的に弱い |

## 5. まとめ

Claude Opus 4.7 は「コーディング能力の向上が機械学習エンジニアリングの自動化に波及した世代」と位置づけられる。SWE-bench Pro での大幅な伸び、自己検証の挙動、データ整合性に対する強さは、ML パイプラインの実装・運用フェーズで実用的な意味を持つ。

一方で、機械学習タスクの自動化は「コードが書けること」だけでは完結しない。新規性のある研究、長期的な探索、人間専門家レベルのモデル最適化は、依然として人間とエージェントの協働が必要な領域である。報酬ハッキングのような失敗モードへの監視も実運用上欠かせない。

「どこまで任せて、どこから自分で見るか」を、タスクの評価密度・新規性・期間で切り分けて使う、というのが現時点での合理的な戦略といえる。
