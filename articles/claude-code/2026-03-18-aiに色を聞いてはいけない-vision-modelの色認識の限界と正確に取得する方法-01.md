## AIにスクリーンショットの色を聞いてみた

Figmaのデザインカンプを見ながら、Claude CodeやCodexにスクリーンショットを渡してフロント実装を進めていたときのことです。出来上がったUIを見て「なんか色が違う」と感じました。

Figma上では落ち着いたダークネイビーだったはずが、AIが実装したCSSでは微妙に青みが強い。ボタンの赤も、どこか彩度が高すぎる。AIはスクリーンショットを「見て」実装しているはずなのに、なぜカラーコードがズレるのか。

気になったので、検証してみました。12色のカラーパレット画像を作成し、Claude Visionに各ブロックのhexカラーコードを聞きます。

まず、各ブロックにhexコードのテキストラベルを付けた画像で試しました。

**結果: 12色すべて完全一致。**

「やるじゃん」と思いましたが、ここで疑問が浮かびます。これは色を「見ている」のか、テキストを「読んでいる」のか？

そこで**テキストラベルをすべて消した画像**で同じ質問をしました。

**結果: 白(#ffffff)以外、すべてズレました。**

## どのくらいズレるのか — 12色で検証

テキストなしのカラーパレットで検証した結果です。正解値はPythonでピクセルを直接読み取って取得しています。

| 正解 | AI推定 | ΔE（知覚差） | 判定 |
| --- | --- | --- | --- |
| #0f172a | #0B1929 | 7.2 | ❌ ズレ |
| #ef4444 | #ED2F38 | 45.5 | 🚨 大きくズレ |
| #22c55e | #00D96B | 67.0 | 🚨 大きくズレ |
| #3b82f6 | #5E93EE | 64.4 | 🚨 大きくズレ |
| #f59e0b | #FF9800 | 26.2 | 🚨 大きくズレ |
| #8b5cf6 | #7B61D5 | 58.7 | 🚨 大きくズレ |
| #ec4899 | #F04B82 | 34.3 | 🚨 大きくズレ |
| #06b6d4 | #00C8D9 | 38.0 | 🚨 大きくズレ |
| #ffffff | #FFFFFF | 0.0 | ✅ 一致 |
| #1a1a2e | #141D31 | 11.7 | ❌ ズレ |
| #e2e8f0 | #DCDEE8 | 25.3 | 🚨 大きくズレ |
| #334155 | #3A4256 | 10.7 | ❌ ズレ |

**平均ΔE: 32.4**

ΔEは人間の色の知覚差を数値化した指標です。一般的に、ΔE < 1 なら人間には見分けがつかず、ΔE > 5 で「明らかに違う色」と認識されます。平均32.4は、まったく別の色に見えるレベルです。

### 傾向

* **白は完全一致する** — 極端な値（0か255）は推測しやすい
* **鮮やかな色ほどズレが大きい** — 緑(ΔE 67.0)、青(ΔE 64.4)が特に顕著
* **暗い色は比較的近い** — ダークネイビー系はΔE 7〜12程度

## なぜAIは色を正確に返せないのか

Vision Modelが正確なカラーコードを返せない理由は、大きく4つあります。

### 1. パッチトークン化

Vision Modelは画像を14×14ピクセルや16×16ピクセルの「パッチ」に分割し、各パッチを1つのトークンに変換します。

![モナリザのモザイクアート](https://static.zenn.studio/user-upload/deployed-images/92aac39401be5a716bf93567.jpg?sha=b1b42b67230e35d8b9371e6fbfa68f23e5c8676f)

このモザイクアートのモナリザ、頬のあたりの正確なカラーコードがわかりますか？

「肌色っぽい」ことはわかっても、`#ceaf88` なのか `#d4b078` なのかは判断できないはずです。Vision Modelが画像を処理するとき、まさにこれと同じことが起きています。

つまり、**1ピクセル単位の色情報は最初の段階で失われます**。パッチ内のピクセルは特徴量として圧縮され、個別のRGB値にはアクセスできません。

`#0f172a` と `#101829` のような微妙な違いは、同じパッチトークンに変換される可能性が高く、原理的に区別が困難です。

### 2. 画像のリサイズ

Claudeの場合、画像の幅または高さが1568ピクセルを超えると自動でリサイズされます。GPT-4Vも同様の処理を行います。

リサイズ時には**バイリニア補間**などのアルゴリズムが使われ、隣接ピクセルの色が混合されます。元画像では `#ef4444` だった赤が、リサイズ後には隣接する白やテキストの色と混ざり、微妙に異なる値になります。

### 3. 画像圧縮

スクリーンショットがJPEGで保存されている場合、**不可逆圧縮**によって色情報が劣化しています。特にエッジ部分（色の境界）では、圧縮アーティファクトによって元の色とは異なる値が記録されます。

PNGであれば可逆圧縮なので色は変わりませんが、APIに送信する際に内部で再エンコードされる可能性もあります。

### 4. カラープロファイルの無視

ディスプレイのスクリーンショットには、そのディスプレイのカラープロファイル（sRGB、Display P3など）が埋め込まれています。

Vision Modelはこのカラープロファイルを解釈せず、生のピクセル値だけを処理します。同じ `#FF0000` でも、sRGBとDisplay P3では実際に表示される色が異なります。そのため、「ディスプレイで見えている色」と「AIが処理している色」の間に根本的なギャップが生じます。

## 「読んでいる」と「見ている」の違い

今回の検証で最も興味深かったのは、テキストラベル付きの画像では完全正解したという事実です。

これは、AIが色を **「見ている」のではなく「読んでいる」** ことを示しています。

Vision Modelの色認識は、ピクセル値の直接読み取りではなく、次のようなプロセスで動いていると考えられます。

1. パッチトークンから「この辺りは赤っぽい」という大まかな特徴を抽出
2. 学習データの知識から「赤っぽい → Tailwind CSSのred-500は#ef4444」と推論
3. もっともらしいhexコードを生成

そのため、有名なカラーパレット（Tailwind CSS、Material Designなど）に近い色は比較的正確に推定でき、中間的な色や独自の色はズレが大きくなります。

## 正確に色を取得する方法

AIの推論に頼らず、**ピクセル値を直接読み取る**のが確実です。

### Pythonで数行で書ける

```
from PIL import Image

img = Image.open("screenshot.png")
r, g, b = img.getpixel((100, 200))[:3]
print(f"#{r:02x}{g:02x}{b:02x}")  # → #0f172a
```

Pillowを使えば、指定座標のRGB値をそのまま取得できます。Vision Modelが内部で行うパッチトークン化やリサイズの影響を受けません。

### もう少し便利にしたツール

検証で使用したカラーピッカーツールを紹介します。

```
# 単一座標の色を取得
python3 color_picker.py image.png 100,200
# → 📍 (100, 200) → #0f172a  RGB(15, 23, 42)  HSL(222°, 47%, 11%)  ≈ black

# 複数座標を一括取得
python3 color_picker.py image.png 100,200 300,400 500,100

# 周辺5pxの平均色（アンチエイリアスされた箇所向け）
python3 color_picker.py image.png 100,200 --radius 5

# 矩形領域の支配色
python3 color_picker.py image.png --region 0,0,300,200

# JSON出力（他ツールとの連携用）
python3 color_picker.py image.png 100,200 --json
```

出力には、hex・RGB・HSLに加えて最も近いCSS色名も表示されます。座標さえ指定すれば、Vision Modelでは得られない**正確な値**が返ってきます。

### AIとツールの組み合わせが最適解

Vision Modelが得意なのは「この画面のどこにボタンがあるか」「配色のバランスはどうか」といった**全体的な判断**です。一方、特定のピクセルの正確なカラーコードは苦手です。

そのため、実務では次のように使い分けるのが効果的です。

* **デザイン全体の印象・バランス** → Vision Model
* **特定要素の正確なカラーコード** → ピクセル直読みツール

AIの強みと限界を理解して、適切に補完する。これが現時点での最適なアプローチです。

## まとめ

* AIのVision Modelは色を「見ている」のではなく、パターンマッチと言語知識で「推測している」
* テキストラベルがあれば正解するが、テキストなしだと平均ΔE 32.4のズレが発生する
* 原因はパッチトークン化・リサイズ・圧縮・カラープロファイル無視の4層構造
* 正確なカラーコードが必要な場面では、Pythonでピクセルを直接読み取るのが確実
* AIの全体判断力 + ツールの精密さ = 組み合わせて使うのがベスト
