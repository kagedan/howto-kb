2026年1月に TOKIUM へ入社し、QA チームでテスト設計の自動化に取り組んでいる ikedan です。  
前回の記事で、私は Claude に「振り返り」と「試運転」という人間の習慣を持ち込んだ話を書きました。  
その結びで「次は Claude に何を覚えさせるかを書きます」と予告したので、本記事はその続きです。

当時、腑に落ちないことがありました。  
同じチームの同僚は、私が作った振り返りや試運転の仕組みを一つも入れていないのに、Claude の的外れな回答（ハルシネーション）が私より明らかに少なかったのです。  
同僚の Claude のほうが、私の Claude より賢く見えました。

違いは一つでした。  
同僚は知識を Obsidian のナレッジベースに整理し、Claude にはそこを引かせていた。  
私は仕組みを足すことに夢中で、知識の置き場所を放置していました。

この設計の元ネタは、元 Tesla の AI 責任者で OpenAI の共同創業者でもある Andrej Karpathy が示した[「LLM Knowledge Bases」という考え方](https://x.com/karpathy/status/2039805659525644595)です。  
生データと、LLM が読む整理済みの知識を分け、必要なときに引く。私に足りなかったのは、まさにこの整理でした。

この「記憶をどう扱うか」という問いには、社内でも以前からいくつもの発信が積み重なってきました。  
きっかけの一つは、弊社松原が X で示した「人間の記憶構造を Claude Code に重ねてみる」という見立て（[該当ポスト](https://x.com/tormatsux/status/2025811074726117535)）で、複数のメンバーがそれぞれの運用に取り入れていきました。  
その後、弊社まっさんが「[Claude Codeに「意識しなくてもできる」を実装した話──学習する記憶の3層構造](https://zenn.dev/tokium_dev/articles/claude-code-evolutionary-memory)」で、観察→記憶→進化という記憶の「動かし方」を書いています。  
本記事はその系譜の3番手として、同じ題材を「どこに置くか」という整理の角度から扱います。

## 「覚えている」と「使える」は別

Claude には毎ターン読み込まれるメモ（`MEMORY.md`。会話のたびに自動で文脈へ入る常駐メモ）があります。便利な反面、3つの落とし穴がありました。

1つ目は、メモが長くなると途中までしか読み込まれなくなることです。私のメモは255行まで膨らみ、切り捨ての警告が出ていました。Claude は「途中まで読んだ記憶」を全部だと思い込み、後半に書いた最新の訂正が抜け落ちます。

2つ目は、一時的な数値が事実として固定されることです。たとえば、ある時点で「テストケース49件・スコア90点」と書いたとします。1ヶ月後には件数もスコアも変わっているのに、「メモにあるから事実」として古い値を引用してしまう。

3つ目は、仕様をルール文で書くと類推で誤適用されることです。「この判定はD列（テストケースExcelのある列）を見る」のような知識を自然言語で書くと、似て非なるケースに当てはめて間違えます。

そして一番こたえたのは、これが AI だけの病ではないと気づいたことでした。  
フォルダの棚卸しで「これは使っている／使っていない」と私自身が判断した7件を、実際に grep（ファイルを横断して文字列を検索するコマンド）で参照元を確かめたところ、7件すべてが思い込みと逆でした（使っていると思ったものが未使用、その逆も）。  
AI のハルシネーションと、私の「ぼんやり覚えている」は、まったく同じ構造だったのです。

問題は「何を覚えるか」ではなく、「どこに置き、どう取り出すか」でした。  
そしてこの3つの落とし穴は、このあとの「4つの置き場所」と2つの規律（その都度引く／実証なきものを正本にしない）で和らげていきます。

## 知識を4つの場所に仕分ける

そこで知識を、常に手元に置くか、必要なときだけ引くか、という軸で4つの場所に分けました。

左にいくほど常に読まれるので薄く保ち、右にいくほど読まれないので量を恐れずに貯められる、という並びです。

意図して4つにしたというより、それぞれが解く問題が違ったので、自然とこの形になりました。あとから振り返ると、各層は既存の知識管理手法と響き合っていました。

| 置き場所 | 対応する手法（ひとことで言うと） | 解いている問題 |
| --- | --- | --- |
| 常時メモ（索引） | MemGPT の working memory（LLMが常時手元に置く少量の作業記憶） | 常時読み込みが膨らむ |
| 必要時メモ | Zettelkasten（1メモ1概念で小さく分け、必要なものだけ参照する手法） | 事例が積み上がる |
| 構造化メモ | Cline Memory Bank（AIが起動時に必ず読む固定ファイル群） | 起動時に読む量が膨らむ |
| 倉庫 | PARA の Archives（いま使わないものを隔離する置き場） | 休眠知識を隔離する |

これらは別々の文脈で生まれた手法ですが、共通して「常時読む量を絞る／必要なものだけ引く」という同じ問題を解いています。  
固有名詞を知らなくても、各層が何を防ぐかは右の列だけで追えます。

## 索引は薄く保つ

1つ目は、常に手元に置く索引です。常時メモは、索引だけにしました。中身は持たせません。  
255行あったメモを十数行まで削り、本体はすべて別の場所へのポインタに置き換えました。これが、長くなると後半が切り捨てられる落とし穴①への対策です。

具体的には、こんな置き換えです。

Before（中身を直接書いていた）:

```
- このプロジェクトの判定はD列を見る。E列は対象外。
- 並列生成したテスト観点の表記ゆれは後で正規化する。
```

After（索引＝ポインタだけ）:

```
- [判定列の仕様](./memory/judgment-column.md)
- [テスト観点の表記ゆれ](./memory/viewpoint-wording.md)
```

毎回読み込まれる設定ファイル（Claude に常時効かせる指示書＝`CLAUDE.md`）からも仕様の記述を抜き、階層によっては半分近く（実測で最大約48%）までスリムにできました。

ただし例外を一つ設けています。「事後に思い出しても手遅れな規律」だけは、索引に常駐させます。たとえば「ツール呼び出し（AIが外部のツールを呼ぶ操作）が失敗したら、同じ形で送り直さない」というルールは、必要になってから検索したのでは間に合いません。気づいたときには事故が起きている。こうした先回りの規律だけは、量を増やさない範囲で常時メモに残します。

## 必要時メモ ── 引くときだけ読む

2つ目は、必要なときだけ呼び出すメモです。いまは約65個あります。  
それぞれに短い説明文（description）を付けておくと、Claude がその説明文を見て「今の作業に関係しそうだ」と判断したときだけ自動で読み込んでくれます。

```
---
name: テスト観点の表記ゆれ
description: 並列生成したテスト観点は、用語の表記を後で正規化する
type: feedback
---
```

1メモ1概念で小さく分けるので、「D列を見る」のような仕様も1件ずつ独立させられます。  
似て非なるケースへ当てはめる類推誤適用（落とし穴③）が起きにくくなり、古くなった一時的な数値（落とし穴②）も、貯め込まず必要なときだけ引くことで紛れ込みにくくなります。  
常時は読まれないので、何個増えても索引は膨らみません。「読まれないこともある」を受け入れる代わりに、量を恐れなくてよくなる設計です。

## 構造化メモは Cline Memory Bank を借りた

3つ目は、体系的な背景知識です。ここは Cline（VSCode 用の AI エージェント）が使っている Memory Bank という設計を借りました。  
プロジェクトの目的・ドメイン知識・設計パターン・技術構成・進行中の状況・進捗、の6ファイルを「コア」として置きます。

Cline 自体は、この6つを必須コアとしつつ、必要なら追加ファイルも許容します。私はそこから「コアを固定する」発想だけを借り、新しい知見が出てもファイルを増やさず、既存の6ファイルを書き直して吸収する──増やさない──縛りを自分に課しました。コアが物理的に増えないので、起動時に読む量が膨らみません。「増やさない」という制約そのものが、肥大を構造的に防いでくれます。

## 倉庫 ── めったに使わないものは隔離する

4つ目は、倉庫です。特定のチケットでしか使わない調査ログや、旧版の成果物のように、めったに参照しないけれど捨てたくない知識をここへ隔離します。  
PARA の Archives と同じ発想で、索引からは見えなくしておく。常時メモや必要時メモを薄く保つために、「使わないものを退避させる先」を1つ用意しておく、という役割です。

## 「置けば安心」ではなかった

ここまでは「整理して置けばいい」という話でした。実際にやってみて一番ハマったのは、この先です。

当初、私は「知識はすべてナレッジベースに置けばいい」と考えていました。これは間違いでした。

業務の一次情報は、ソースコードや Excel など別の場所にあります。ナレッジベースには、それを人や AI が読みやすい形に写したページが混在します。だからページには2つの顔が生まれます。  
一つは、そこが正本（唯一の正しい出典、Single Source of Truth）であるページ。  
もう一つは、別の場所にある正本をコピーした、いわば写しのページです。後者は、正本を更新したのに反映を忘れると、古い値のまま残ります。  
そして Claude は、その古い写しを「ナレッジベースに書いてあるから正しい」と引用してしまう。整理したことが、かえって新しいハルシネーションの種になるのです。

対策は2つでした。  
一つは、どのページが正本でどれが写しかを、索引から機械的に判別できるようにすること。  
もう一つは、知識をベタ書きで抱え込まず、必要なときに引く形にして、常に新しい情報だけを文脈へ入れることです（落とし穴②と同じ「その都度引く」です）。  
「貯めて読む」から「その都度引く」に変えると、古い断片が紛れ込まなくなりました。

## どこに置くかは、この問いで決める

最終的に、知識の置き場所は次の問いで決めています。

* 常に必要で、事後では取り返しがつかないか → 索引（常時メモ）
* ときどき必要か → 必要時メモ
* 体系的な背景か → 構造化メモ
* 特定の作業でしか使わないか → 倉庫

そして置き場所と同じくらい効いたのが、「実証なき主張を正本にしない」というルールでした。  
「見た気がする」「使っているはず」を出典にしない。grep やファイルを開いて確かめたものだけを正本として扱う。  
これは Claude に課したルールであると同時に、冒頭で7件を取り違えた私自身への戒めでもありました。

## おわりに

前回は、振り返りと試運転という仕組みを足しました。今回学んだのは、足すより整理するほうが効く場面があったということです。

知識整理の答えは、覚えさせる量を増やすことではありませんでした。中身に応じて置き場所を選び、検索で鮮度を保ち、実証で守る。  
実際、常時読み込む索引は十数行まで、毎ターン効く指示ファイルも階層によっては半分近くまで軽くなり、固定費を小さく保てるようになりました。  
そして整理後にぶつかった「古い写しを正本と取り違える」問題も、写しを機械的に見分ける仕組みで抑えられています。  
貯めるのは誰でもできます。生かすのは、整理と裏取りのほうでした。

人と AI が一緒に品質を守る時代に、AI に何を渡すかと同じくらい、それをどこに置くかが効いてくる。  
次はこの整理を支える「先回りで止める仕組み」について、記事を書こうと思います。
