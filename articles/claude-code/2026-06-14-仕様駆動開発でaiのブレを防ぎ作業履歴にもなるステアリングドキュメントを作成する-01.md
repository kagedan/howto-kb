実際にコードを書く前に、AIとの認識のズレを防ぐため、作業単位のドキュメントである「ステアリングドキュメント」を整備していきます。

この記事では、実際にAIとやり取りしながら作成した要件定義・設計・タスクリストのドキュメント全文と、そこから得られた実務上の気づきをまとめました。

## この記事でわかること

* 実装フェーズでAIを迷子にさせないための「ステアリングドキュメント」の役割と全体像
* `requirements.md`、`design.md`、`tasklist.md` の3つの構成と具体的な中身
* AIが間違った手順を踏まないように人間が軌道修正するレビューのポイント
* なぜ「1ステップずつの実行とレビュー」がSDDにおいて極めて重要なのか

## なぜこのテーマが重要なのか

永続的ドキュメントを作成したあと、いきなりAIに「この仕様書通りにコードを全部書いて」とお願いするのは、手戻りを生む原因になりやすいです。

例えば、AIに一括で大きな実装を依頼すると、一見動くコードを出してくれますが、裏側で状態管理のロジックが矛盾していたり、既存のテスト環境を壊してしまったりします。  
そして、どこからエラーが出ているのか人間にさえ分からなくなるというカオスに陥りがちです。  
また、いつどのような意思決定がなされたか、経緯の履歴を追えなくなるリスクもあります。

だからこそ、開発作業単位ごとにAIと要件・設計・タスクリストをすり合わせる「ステアリングドキュメント」が必要不可欠なわけです。

## まず結論

実装に入る前に、`.steering/` ディレクトリ配下に以下の3つのドキュメントを作成してAIと合意を形成しておけば、**AI開発における手戻りやブレを劇的に減らすことができます。**

1. **`requirements.md`（要求定義）** ── この作業単位で「何を作るのか」（スコープと受け入れ条件）
2. **`design.md`（設計）** ── どう実装するのか（アプローチ・型定義・テスト方針）
3. **`tasklist.md`（タスクリスト）** ── どのような順番で作業を進めるか（進捗チェックリスト）

これらを準備してから1タスクずつ実装を進めることで、AIは常に正確なナビゲーションに従って安全にコードを書いてくれます。

## 目的

プロジェクトの開発環境を構築し、タイマー付きTodoアプリの MVP（最小限の動作する製品）を実装する。

## スコープ

### 含むもの（今回のスコープ）

1. **開発環境セットアップ**
   - Docker / Docker Compose による開発環境構築
   - Next.js + TypeScript + Tailwind CSS + Zustand のプロジェクト初期化
   - Vitest + React Testing Library のテスト環境構築

2. **タスク管理機能**
   - タスクの作成（タイトル必須、予定時間は任意）
   - タスクの編集・削除
   - タスクの完了マーク

3. **タイマー機能**
   - タスクごとのタイマー開始・停止
   - 同時に動かせるタイマーは1つのみ（他のタスク開始で自動停止）
   - ページリロード後のタイマー状態復元

4. **時間記録**
   - 作業セッションの保存（開始〜停止の1セッション = 1 WorkSession）
   - タスクごとの累積作業時間の表示

5. **日次サマリー**
   - 当日の作業タスクと合計時間の一覧表示

6. **完了済みタスク画面**
   - 完了タスクと累積作業時間の閲覧

### 含まないもの（スコープ外）

- ユーザー認証・クラウド同期
- PWA 対応・オフライン対応
- 過去日のサマリー参照（当日のみ）
- データのエクスポート機能

## 型定義

`src/types/index.ts`に全ドメイン型を定義する。

type Task = {
  id: string;
  title: string;
  estimatedMinutes: number | null;
  isCompleted: boolean;
  createdAt: string;       // ISO8601
  completedAt: string | null;
};

type WorkSession = {
  id: string;
  taskId: string;
  startedAt: string;       // ISO8601
  endedAt: string | null;
  durationSeconds: number | null;
};

type ActiveTimer = {
  taskId: string;
  sessionId: string;
  startedAt: string;       // ISO8601
};

## ストア設計

### taskStore

type TaskStore = {
  tasks: Task[];
  addTask: (title: string, estimatedMinutes: number | null) => void;
  updateTask: (id: string, patch: Partial<Pick<Task, 'title' | 'estimatedMinutes'>>) => void;
  completeTask: (id: string) => void;
  deleteTask: (id: string) => void;
};

### sessionStore

type SessionStore = {
  sessions: WorkSession[];
  startSession: (taskId: string, sessionId: string) => void;  // endedAt=null で追加
  endSession: (sessionId: string, endedAt: string) => void;   // endedAt・durationSeconds を確定
  getTotalSeconds: (taskId: string) => number;                 // 累積作業時間を返すセレクタ
  getTodaySessions: () => WorkSession[];                       // 当日セッションを返すセレクタ
};

### timerStore

type TimerStore = {
  activeTimer: ActiveTimer | null;
  startTimer: (taskId: string) => void;   // 既存タイマーを停止してから新規開始
  stopTimer: () => void;
  getElapsedSeconds: () => number;        // startedAt から現在時刻までの秒数
};

**ストア間の連携フロー（タイマー開始時）:**

timerStore.startTimer(taskId)
  → timerStore.stopTimer() で既存タイマーを停止
      → sessionStore.endSession() でセッションを終了・保存
  → 新規 sessionId を生成
  → sessionStore.startSession(taskId, sessionId)
  → activeTimer をセット・localStorage へ永続化

### localStorage キー

| キー | 対応ストア |
|---|---|
| `timer-task:tasks` | taskStore |
| `timer-task:sessions` | sessionStore |
| `timer-task:active-timer` | timerStore |

Zustand の `persist` ミドルウェアで自動同期する。

## コンポーネント設計

### 画面構成と状態

3つのビューをボトムナビで切り替える。ビューは URL ではなく React state で管理する（Next.js の SPA モードを活用）。

app/page.tsx（"use client"）
├── state: activeView = 'tasks' | 'summary' | 'completed'
├── <TaskList />        （activeView === 'tasks'）
├── <DailySummary />    （activeView === 'summary'）
├── <CompletedTaskList />（activeView === 'completed'）
└── <BottomNav />

### 各コンポーネントの責務

**`TaskList`**
- `taskStore` から未完了タスクを取得して一覧表示
- 「+タスクを追加」ボタンで `TaskFormModal`（新規）を開く

**`TaskItem`**
- タスク1行の表示（タイトル・予定時間・実績時間）
- タイマー開始/停止ボタン → `timerStore.startTimer` / `timerStore.stopTimer`
- 完了ボタン → `taskStore.completeTask`（タイマー動作中なら自動停止）
- 編集/削除メニュー → `TaskFormModal`（編集）または `taskStore.deleteTask`

**`ActiveTimerBadge`**
- `timerStore.activeTimer` が存在するタスク行に表示
- `setInterval(1000)` で `elapsedSeconds` を更新してリアルタイム表示
- アンマウント時に `clearInterval` する

**`TaskFormModal`**
- 新規・編集で同一コンポーネントを使用（`taskId` prop の有無で判別）
- タイトル（必須）・予定時間（任意・時間と分を別インプット）

**`DailySummary`**
- `sessionStore.getTodaySessions()` で当日セッションを取得
- タスクIDでグループ化して合計時間を算出
- アクティブタイマー中のタスクは現在時刻まで加算してリアルタイム表示

**`CompletedTaskList`**
- `taskStore` から `isCompleted === true` のタスクを一覧表示
- `sessionStore.getTotalSeconds(taskId)` で累積時間を表示

## タイマーリロード復元フロー

app/page.tsx の useEffect（初回のみ）
  → timerStore.activeTimer を確認
  → null でなければ、startedAt から現在時刻までの経過時間を算出
  → ActiveTimerBadge が即時リアルタイム表示を開始

`sessionStore` の該当セッション（`endedAt === null`）はそのまま継続中として扱う。

## 人間が軌道修正すべき3つのレビューポイント

成果物ごとにしっかりと人間が目を通してレビューし、軌道修正のプロンプトを投げることで、AIはより安全で一貫したコードを書いてくれるようになります。

今回の作業を通して学んだ、レビューのポイントをまとめます。

### スコープ（やること/やらないこと）の厳格化（要件のレビュー）

AIは良かれと思って、予定していない機能（例えば、将来的な認証機能のプレースホルダーや不要な設定ファイルなど）を勝手に盛り込む傾向があります。

余計なコードが増えると後からバグの温床になるため、今回の実装単位で「やらないこと」をスコープ外として定義させ、AIの暴走を防ぐガードレールを作ることが大切です。

### 制約とバージョンの整合性（設計のレビュー）

AIはインターネット上の多様な学習データをもとに回答するため、平気で少し古いバージョン（今回の Node 22 など）や、既存プロジェクトで禁止しているライブラリを提案してきます。

プロジェクト環境に適合しているか、必ず人間がチェックして「Node 24（LTS）を指定して」「状態管理は Jotai にして」と指示を追加する必要があります。

### タスクの粒度が大きすぎないか（タスクリストのレビュー）

AIが生成するタスクリストは、時に「ロジック層の実装」のように大雑把な1ステップになっていることがあります。

これだとAIが一度に何十行ものコードを一気に書き換えてしまい、バグが起きた際の調査が困難になります。

プロダクトの性質にもよりますが、「1関数」「1ストア」「1UIパーツ」単位でテストが通る極小のステップに分解されているか、人間の目でチェックしてあげるのが品質高く進めるコツかと思います。

## まとめ

今回の場合、実装開始前に作成する「ステアリングドキュメント」は、以下の3つで構成しました。

1. **`requirements.md`**：何を作り、何を作らないか（スコープと受け入れ条件）
2. **`design.md`**：どんな技術・構成・設計方針で実装するか（型、ストア、テスト）
3. **`tasklist.md`**：どのようなステップで開発・検証を進めるか（WBS）

これらをリポジトリの `.steering/[タスク名]/` 配下に配置し、AIと合意形成を行う。
