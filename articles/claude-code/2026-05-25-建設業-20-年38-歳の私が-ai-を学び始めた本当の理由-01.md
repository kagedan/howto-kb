## 1. キャリアの転換点

建設業の現場で 18 歳から 20 年やってきた。

クレーンの組立・解体、橋、トンネル、地中送電。  
親方の下で年季を積み、職人として、現場監督として、そして経営者として、業界 1 位の取引先の現場も任されてきた。

今年、AI に全振りした。  
38 歳になってからの判断だ。

きっかけは、Claude Code を使い始めた 3 週間目に訪れた。  
自分の業界の未来が、急に見えてしまった。

40 代がほぼ消えた建設業で、ベテランの頭の中だけにあるノウハウが、10 年で全部消える。  
誰かが残さないといけない。それを残せる立場にいるのは、現場と経営の両方を知っている自分だと気づいた。

この記事は、その「気づき」から最初の 90 日で動いたことの記録です。

## 2. 建設業が AI で変わる現実

建設業界の現実を数字で見ると、想像以上に厳しい。

* 60 代以上の職人が現場の大半
* 40 代がほぼ消えている（バブル崩壊後の世代）
* 20-30 代の新規入職率は下がり続けている

10 年で「ベテランの頭の中」のノウハウは消える。  
誰も残さなければ、業界全体が記憶喪失になる。

### なぜ建設業の AI 化は遅れているのか

「保守的だから」じゃない。本質はもっと深い。

製造業は工場でデータが取れる。同じ製品を同じラインで作る。再現性が高い。  
建設業は現場が毎回違う。同じ橋を二度作らない。同じ高さのビルを違う地盤で建てる。

「**再現性のないドメインの AI 化**」。  
これが本質の問いだ。

そしてこれは建設業だけの話じゃない。医療、農業、教育、小売 — 「現場が毎回違う業界」全てに当てはまる。

## 3. 最初の 90 日で動かした 3 プロダクト

抽象論で終わらせたくないので、具体に降ろす。

90 日で動かしたのは 3 つ。

1. **UKARU** — 特定技能試験対策アプリ（10 言語対応）
2. **書類自動化** — 安全書類・現場入場書類のタブレット入力 + 自動生成
3. **クレーン自動運転構想** — メーカーとの共同開発を視野に入れた設計フェーズ

このうち、UKARU を深掘りする。

### なぜ UKARU を最初に作ったか

うちの会社では、外国人労働者が現場で働いている。特定技能 1 号として入国した彼らは、技能評価試験を受ける必要がある。  
ところが市販の教材がほぼない。あっても日本人向けで、N4-N3 レベルの語彙では理解できない。

「これは AI で作れる」と直感した。  
プロンプト設計と問題テンプレート、人間レビューの組み合わせで、業界初の対策アプリが作れると確信した。

### Claude Code に渡したシステムプロンプトの一部

```
あなたは建設業の特定技能試験対策アプリの問題生成 AI です。

# 受験者プロファイル
- 日本語が母語ではない外国人労働者
- N4-N3 レベルの語彙範囲
- 建設業の現場経験あり / なし両方

# 出力ルール
1. 技能の本質を問う問題を作る（暗記ではなく理解）
2. N4-N3 を超える漢字には全てふりがな
3. 4 択、正解 1 つ、誤答は「もっともらしい誤答」
4. 解説は実務エピソード付き

# 出力フォーマット
JSON で { question, choices[], answer, explanation } を返す
```

このプロンプトに辿り着くまでに 2 週間かかった。  
最初は問題が硬すぎた。次は簡単すぎた。  
ふりがなも自動付与は精度 70%、人間レビューを最後に挟む構造に落ち着いた。  
UI も、最初は日本語で答えるだけだったのが、選択肢の言語切り替え機能を追加した。

3 つ作るたびに、3 つ失敗している。  
失敗を残せる立場にいることが、業界の中の人の特権だと最近思う。

## 4. Claude Code / Codex / Gemini の役割分担

3 プロダクトを並行で動かすには、AI ツールの使い分けが必要だった。

| 用途 | 使うツール | 理由 |
| --- | --- | --- |
| 長期プロジェクトのコード書き換え | Claude Code | コンテキスト保持、ファイル横断、CLAUDE.md で記憶 |
| 単発の関数 / スニペット | Codex | 速度、軽さ、IDE 統合 |
| 業界知識の調査 / 競合分析 | Gemini Deep Research | Web 横断、引用付き |
| 文章の最終整え / 公開直前レビュー | Claude Sonnet | 文体精度、トーン調整 |

実装フローはこうなった。

### ハマったポイント 3 つ

1. **AI に任せすぎたパターン**: 全部 LLM に投げて 3 日溶かした。最終的には自分で直す方が早かった
2. **役割分担を曖昧にしたパターン**: Claude Code と Codex が同じファイルを編集して衝突
3. **「動いた」で満足したパターン**: 検証を後回しにしたら現場で誤動作、信用を失いかけた

3 つとも自分のミス。AI のせいじゃない。  
便利な道具ではなく、反論してくる相棒として使うとき、AI は思考の鏡になる。

## 5. 業界の中の人が作るしかない

現場で AI の話をしても、職人は取り合ってくれない。

「お前はパソコンの人だろ」  
「現場はそんな簡単じゃない」  
「AI で重機が動かせるなら持ってこい」

18 歳から現場でやってきた自分が言っても、こうなる。  
業界の外から来た IT 屋が同じことを言えば、もっと門前払いだろう。

でも、これは建設業に限らない。  
医療、農業、教育、小売 — 既存業界全てで同じ風景が広がっている。

だからこそ命題が一つ残る。

> **業界の穴は、業界の中の人にしか見えない。**  
> そして、業界の中の人が AI を学ばないと、誰かが上から「正解」を持ち込んで失敗する。

業界の外からの AI 導入は、ほとんどが失敗する。  
失敗の理由は技術じゃない。文化・現場感・人間関係を読み違えるからだ。

中小建設会社の経営者の方へ。  
あなたの業界も、あなたが動かないと変わらない。  
誰かが上から正解を持ち込むのを待っていると、10 年後に業界そのものが消えている。
