Claude Code の会話ログを Obsidian に保存する運用や、タスク管理ツールなどの MCP でチケットを読む運用はすでにあります。ここで扱うのは、その次に困った「作業文脈をどう復元するか」です。

この記事は、チケット番号をキーに、設計判断・進捗・PR 粒度・引き継ぎメモを毎回同じ場所から復元する仕組みをローカルに作った話です。

## 何に困っていたか

ずいぶん前のことですが、設計まで Claude Code に任せたあとに `/compact` が走りました。そのまま実装を続けると、直前に決めた前提がまるっと抜け落ちたコードになっていました。

その時点では、設計方針などをファイルに残しておらず、`/compact` が走った時点で背景が抜け落ち、実装は compaction 後の文脈から推測する形になっていました。

* `CLAUDE.md` に設計判断を全部書くと肥大化して読まれなくなる
* Linear MCP でチケットを引いても概要しかなく、検討の経緯は載っていない
* タスク内の意思決定がどこにも残っていない

AI エージェントの記憶はあてにならないので、**毎回同じ場所を読ませる**仕組みを作るのが早い、という結論になりました。

handoff などのメモを残すだけでは足りません。再開時に AI が最初に読むファイルが一意に決まっていないと、結局どの前提を採用するかを AI に判断させることになります。

## 作ったもの

仕組みは大きく 3 つです。

* Obsidian Vault 内の `tasks/<TICKET>-<feature>/` ディレクトリ (チケットごとの雛形)
* グローバル `CLAUDE.md` の Auto-Reference Rule (チケット番号 -> 探索 -> 該当ディレクトリをロード)
* `/task-init` Skill (ディレクトリ初期化を一発でやる)

これに以下のルールを `_index.md` に書き込むことで、チケット番号から読むべき文脈と作業場所が決まるようにしています。

* 親チケット = ディレクトリ、サブチケット = ファイル
* 1 sub-ticket = 1 PR = 1 検証可能な成果物
* worktree も branch も `<TICKET>/<desc>` で揃える

会話ログを全部保存するのではなく、次のセッションで読むべき情報だけをチケット単位で残す、という割り切りです。

## ディレクトリ構成

```
projects/<product>/tasks/PROJ-100-user-import/
  _index.md
  PROJ-101_repository-split.md
  PROJ-102_import-handler.md
  design-notes.md
```

`_index.md` のテンプレートはこんな形です。

```
# PROJ-100 User Import

## Ticket Tree
- PROJ-100 (親)
  - PROJ-101 repository 層の分割
  - PROJ-102 import handler の実装

## Active
- PROJ-102_import-handler

## Current State
- Completed: PROJ-101
- In Progress: PROJ-102
- Not Started: -

## Implementation Order
1. PROJ-101 (前提: なし)
2. PROJ-102 (前提: PROJ-101)

## Files
- PROJ-101_repository-split.md
- PROJ-102_import-handler.md
- design-notes.md (非チケット、全体設計)
```

`## Active` は、作業中のサブチケットへのポインタです。

サブチケット側の plan ファイルには、完了条件と引き継ぎメモを必ず置いています。完了条件は人間向けのチェックリストではなく、PR 作成前に `/pr-create` が読む検証入力として扱っています。未達の項目があれば、PR を作る前に止めるか、引き継ぎメモに残します。

```
## 完了条件
- [ ] repository 経由で保存できる
- [ ] usecase の既存 test が通る
- [ ] handler_test.go に正常系・エラー系の新規ケースが追加され green

## 引き継ぎメモ
- 現状: handler までは実装、usecase の test が落ちている
- 次やること: mock_repository の戻り値を見直す
- ブロッカー: なし
- 変更ファイル: internal/usecase/user.go、mock_repository/...
```

再開時は、まず plan ファイルを読みます。更新は「コミット前」「PR 作成前」「セッション終了前」に寄せ、PR 作成時は `/pr-create` が次の処理を自動で行います。

PR 作成後は、だいたいこういう差分になります。

```
 # _index.md
 ## Active
-- PROJ-102_import-handler
+(none)

 ## Current State
-- Completed: PROJ-101
-- In Progress: PROJ-102
+- Completed: PROJ-101, PROJ-102 — #1234
+- In Progress: -
```

```
 # PROJ-102_import-handler.md
 ## 引き継ぎメモ
-- 現状: handler までは実装、usecase の test が落ちている
-- 次やること: mock_repository の戻り値を見直す
-- ブロッカー: なし
-- 変更ファイル: internal/usecase/user.go、mock_repository/...
+- 現状: 完了条件 3/3 達成、PR #1234 を draft で作成
+- 次やること: review 対応
+- ブロッカー: なし
+- 変更ファイル: internal/usecase/user.go、internal/handler/user.go、handler_test.go、mock_repository/...
```

## /task-init で初期化する

`/task-init PROJ-100 user-import` で Linear からチケット情報を取り、`_index.md` とサブチケット用の plan ファイルを生成できるようにしています。プロジェクト prefix で対象 product を自動判定 (例: `PROJ-*` -> product-a、`SRE-*` -> infra) しているので、複数プロダクトが混ざっていてもチケット番号だけで決まります。

## CLAUDE.md に書くルール

自動参照のキモは、グローバルの `~/.claude/CLAUDE.md` に書いたこの数行だけです。運用ルールは基本的に英語で書いています。

```
## Auto-Reference Rule

When a ticket number (e.g., PROJ-XXXX, SRE-XXXX) is mentioned:

1. Glob `projects/*/tasks/{TICKET}*/` for a directory name prefix match
2. If found: Read `_index.md` first, then check `## Active` to identify the current sub-ticket. Load that plan file before any others
3. If `## Active` is empty: load all `In Progress` items from `Current State`
4. If the ticket is a sub-ticket: also search file names across all task directories
5. If not found: suggest creating a new task directory with `/task-init`
```

使うときは、セッション冒頭にチケット番号を投げるだけです。

```
PROJ-100 の続き。まず task directory を読んで、現在地を確認して。
```

Claude が glob して該当ディレクトリをロードし、`## Active` で指された plan file の `完了条件` と `引き継ぎメモ` を読んでから作業に入ります。

## 運用して効いたところ

* 設計と実装のセッションが分かれても、`/compact` を挟んでも、`_index.md` と plan ファイルから前提を復元できます。体感では、前提を読み落としたまま実装に入ることがかなり減りました。
* 設計判断を確定したとき・test が green になったとき・別サブチケットへ移る前に `引き継ぎメモ` を更新するルールを CLAUDE.md に入れているので、`/compact` を踏んでも plan file から前提を取り戻せます。
* 設計判断がファイルに残るので、時間が経ってから別の人 (主に Claude や Codex) がコンテキストごと戻れます。
* worktree を組み合わせると、複数チケットを並列で進めても文脈が混ざりません。

## まだ人間が見るべきところ

サブチケット分割の粒度は、まだ人間が見た方がよさそうです。  
「1 PR で完結するか」「検証可能か」は、AI に任せると粒度が荒くなることがあります。ここだけは、最初に人間が決めておくほうが安定しました。

## まとめ

再開時に最初に読むファイルを一意にしておけば、Claude に覚えさせなくても前提は復元できます。

チケット番号をファイル名の prefix にして、`CLAUDE.md` に探索ルールを書きました。チケット粒度・PR 粒度・検証単位も、できるだけ揃えています。

Claude Code を複数チケットで回すなら、「記憶」より「どのファイルを読ませるか」の方が効きます。  
`CLAUDE.md` に全部書くのではなく、チケット番号で引ける作業メモに分けておくと、次のセッションを同じ前提から始めやすくなります。
