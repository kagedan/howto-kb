AIコーディングでいちばん誤解されているのは、「良いプロンプトを書ける人が強い」という話だと思う。

もちろん、雑な依頼よりは具体的な依頼のほうがいい。だがチーム開発で本当に効いてくるのは、チャット欄に毎回うまい文章を書く能力ではない。リポジトリ側に、AIが先に読むべき前提を置いておくことだ。

人間向けには`README.md`がある。セットアップ方法、開発コマンド、設計方針、ディレクトリ構成。新しく入った開発者は、まずそこを読んで、このプロジェクトで何を当然として扱うかを覚える。

では、AIエージェントは何を読むのか。

いま起きている変化は、ここにある。`AGENTS.md`、`CLAUDE.md`、`GEMINI.md`、`copilot-instructions.md`のようなファイルは、単なる補助メモではない。リポジトリに参加するAI向けのオンボーディング文書であり、場合によっては実行ルールそのものになる。

## プロンプトは、揮発する設計判断である

開発者がAIに毎回こう頼む。

「このプロジェクトではpnpmを使ってください」  
「このディレクトリは触らないでください」  
「UIコンポーネントは既存の設計に合わせてください」  
「テストを先に見てください」

これは一見まともだ。だが、毎回チャットに書いている時点で、その設計判断は揮発している。

同じリポジトリでも、Aさんは丁寧に制約を書く。Bさんは勢いで「直して」とだけ投げる。Cさんは昨日の会話を前提にしてしまう。結果として、AIの出力品質はモデル性能だけでなく、依頼した人の癖に引きずられる。

人間だけの開発なら、それでも会話で補正できた。AIエージェントがCLI、IDE、GitHub、モバイルアプリのような複数の面から作業に入ってくると、そうはいかない。入口が増えるほど、ルールは個人の記憶ではなく、リポジトリに置く必要がある。

つまり、プロンプトエンジニアリングの一部は、ドキュメント設計に吸収される。

## `AGENTS.md`はREADMEの置き換えではない

ここで大事なのは、AI向けREADMEは人間向けREADMEのコピーではない、という点だ。

人間向けREADMEは、だいたい「どう始めるか」を書く。インストール、起動、テスト、デプロイ、概要。人間は行間を読めるし、怪しいところで止まれる。わからなければSlackで聞く。

AI向けの文書は、もっと操作的でなければならない。読んで納得するためではなく、次の一手を間違えないためにあるからだ。

* どのパッケージマネージャを使うか
* 変更前にどのテストを見るか
* 触ってよいディレクトリと触ってはいけないディレクトリ
* 既存の抽象化に従うべき場所
* 追加してはいけない依存
* 生成物やビルド成果物を編集してよいか
* 失敗したときに、どこで止まるべきか

これは「親切な説明」ではなく、制約の列挙である。少し冷たいくらいでちょうどいい。

AIは、人間よりも素直に大きく動く。曖昧な依頼を受けると、存在しない設計意図を補ってしまう。便利なライブラリを勝手に足す。似ているファイルを見つけて、文脈が違うのにパターンを横展開する。テストを直すべきところで、テストを都合よく変えることもある。

だからAI向けREADMEには、「こうしてください」だけでなく「これはしないでください」をはっきり書く必要がある。

## コンテキストを増やすより、先に絞る

AIエージェントに大量のコンテキストを渡せば賢くなる、という考え方がある。半分は正しい。だが実務では、コンテキストは量ではなく形が効く。

リポジトリ全体を読ませれば、たしかに材料は増える。だが、材料が増えるほど、モデルは関係ないファイルにも引っ張られる。古い実装、今は使っていないパターン、例外的なコード、生成された成果物。全部が同じ重さで文脈に入ると、かえって判断が鈍る。

`AGENTS.md`のような設定ファイルの価値は、「もっと読ませる」ことではない。「最初に何を重く見るべきか」を指定することにある。

たとえば、次のような一文だけでも効く。

```
UI変更では、まず `app/components/ui` の既存コンポーネントを使う。新しいスタイル体系を追加しない。
```

これは小さな文だが、AIの探索範囲をかなり狭める。余計なCSSフレームワークを足さない。似ているが別系統の実装を作らない。レビュー側も「このルールに従っているか」を見ればよくなる。

AGENTS.mdの効率を測る研究が出てきたのも、自然な流れだと思う。AI向け設定は、精度だけの話ではない。実行時間、トークン消費、再試行の回数、レビュー負荷に影響する。つまりこれは、コスト設計でもある。

「AIに全部読ませればいい」は、かなり雑な設計である。普通の開発現場では、先に読むべきものを決めるほうが強い。

## エージェントがリポジトリに参加するなら、ルールもレビュー対象になる

GitHub Copilotのようなツールがエディタの補完を越えて、アプリやWeb、Issue、PRの近くで動くようになると、AIは「手元の便利機能」ではなくなる。作業セッションを受け取り、変更を作り、レビューに流す参加者になる。

そうなると、AIへの指示もコードレビューの対象にしたほうがいい。

`AGENTS.md`を誰かが変更したとする。

```
- 依存追加は事前に相談する
+ 必要なら依存を追加してよい
```

これは小さな差分に見える。だが、エージェントの行動範囲は大きく変わる。設定ファイルひとつで、AIが実装時に選ぶ解が変わる。トークン消費も変わる。レビューすべき差分の性質も変わる。

ならば、これはコードと同じように扱ったほうがいい。

* PRで変更する
* なぜそのルールが必要かを書く
* 古くなったルールを消す
* 実際の失敗例からルールを追加する
* チームの開発コマンドや設計変更に合わせて更新する

AI向け設定が放置されると、古いREADMEと同じ問題が起きる。誰も信じない。誰も読まない。AIだけが素直に読んで、古い手順を実行する。

それは地味に危ない。壊れるときも、だいたい地味に壊れる。

## 良いエージェント向けREADMEは、短い

ここで逆方向の失敗もある。AI向けREADMEを、社内規約の縮小版にしてしまうことだ。

長すぎる設定ファイルは読まれない。人間にも読まれないし、AIにも効きにくい。重要でないルールが増えるほど、本当に守ってほしい制約が薄まる。

良い`AGENTS.md`は、たぶん短い。少なくとも最初は短く始めたほうがいい。

書くべきなのは、プロジェクトごとに事故りやすい場所だ。

たとえばフロントエンドなら、デザインシステム、状態管理、APIクライアント、テスト方針。バックエンドなら、トランザクション境界、マイグレーション、認可、ログ、外部API。モノレポなら、パッケージ間の依存方向。生成物が多いプロジェクトなら、編集してよいファイルと生成元の区別。

AIが迷ったときに、レビューでいつも同じ指摘をしているなら、それは設定ファイルに落とす候補である。

逆に、「丁寧に作業してください」「品質を高くしてください」のような文は、ほとんど効かない。人間に言っても曖昧な言葉は、AIに言っても曖昧なままだ。

AI向けREADMEに必要なのは、精神論ではなく、作業可能な制約である。

## 2026年のリポジトリは、人間だけが読む場所ではない

これまでのリポジトリは、人間の共同作業場だった。READMEも、CONTRIBUTINGも、ADRも、コメントも、基本的には人間が読む前提で書かれていた。

だが、これからのリポジトリにはAIも入ってくる。しかも、読むだけではない。編集する。テストを走らせる。差分を作る。PRの説明を書く。場合によっては、複数のエージェントが別々の作業を並行して進める。もはや読者というより、作業者に近い。

そのとき、AIに対するオンボーディングが毎回チャット欄にしか存在しないのは、かなり弱い。

人間にREADMEを書かないチームは少ない。新しい開発者が来るたびに、口頭でセットアップ手順を説明し続けるのは無駄だからだ。同じことがAIにも起きている。毎回プロンプトに制約を書くのは、AI時代の口頭オンボーディングである。

だから、リポジトリにはエージェント向けREADMEがいる。

それはAIを過信するための文書ではない。むしろ逆だ。AIが勝手に賢く文脈を読んでくれる、という期待を捨てるための文書である。

良いチームは、AIにうまくお願いするだけでは終わらない。AIが読む前提をバージョン管理し、レビューし、更新する。プロンプトを個人技にせず、リポジトリの設計へ引き戻す。

2026年のREADMEは、人間だけに向けて書かれない。

AIエージェントを実務に入れるなら、最初に整えるべきなのはモデル選びではない。リポジトリが、そのエージェントに何を読ませ、何をさせず、どこで止めるのかを明文化しているかどうかだ。

プロンプトは会話で消える。設定ファイルは差分として残る。レビューできるし、責任の所在も追える。

この差は、思っているより大きい。
