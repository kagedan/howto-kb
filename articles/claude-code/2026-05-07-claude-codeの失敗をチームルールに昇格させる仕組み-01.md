## はじめに

こんにちは、クラシルのレシチャレを開発しているkiyokuroです。

これまで、ルール設計・Safety Hook・サブエージェント・3エージェントハーネスのブログを書いてきました。[前回の記事](https://zenn.dev/dely_jp/articles/a45bc3a9e69ab1)では3エージェントハーネスを取り入れて、実装のchunkを連鎖的に回すループまで辿り着きました。

ただし、回り始めたループから出てくる「失敗」をどう活かすかには触れていませんでした。この記事では、Evaluatorが拾った失敗を個人ローカルに蓄積し、3件以上たまればチームルールに昇格させ、さらにセッション全体を振り返る仕組みを紹介します。

Mitchell Hashimotoが[My AI Adoption Journey](https://mitchellh.com/writing/my-ai-adoption-journey)で紹介している「ハーネスエンジニアリング」は、エージェントがミスをした時に、同じミスが起きにくくなるように環境（Claude Codeで言えばCLAUDE.mdやrules、hookなど）を更新していく、という考え方として語られています。今回紹介するのは、この思想に回帰する仕組みです。

## Step 1: 失敗は蓄積させないと忘れる

エージェントハーネスで実装を行うと、実装内容の確認フェーズでNEEDS\_REVISION / GENERATOR\_FAILEDが出ます。人間が介入して修正を促した問題が、別セッションで同じパターンで再発することがあります。人間の記憶やgit logには痕跡が残っていますが、次のセッションを始めたGenerator / Evaluatorはそれを知りません。

失敗は「起きた瞬間に直す」だけでは不十分で、「次のセッションのエージェントが参照できる形」に残さないと、同じ失敗を繰り返します。

Mitchell Hashimotoの記事ではGhosttyプロジェクトの`AGENTS.md`（Claude Codeで言うCLAUDE.md / rulesに相当）の行のほとんどが、過去のエージェントの誤動作から生まれていると書かれています。

> （意訳）エージェントがミスをした時に、同じミスが起きにくくなるように環境を更新していく。

この考え方を自分たちのハーネスにも反映する必要があります。そのために用意した3つの仕組みが、`FAILURES.md`・`/failure-promote`・`/retro`です。

## Step 2: FAILURES.md — 失敗を個人ローカルに蓄積する

失敗を次のセッションに引き継ぐには、どこかに書き残す場所が必要です。ただし、いきなりチーム共有のルールに書き込むのは早すぎます。単発の失敗をルール化するとメンテナンスコストが膨らむので、まずは個人ローカルに蓄積し、繰り返すものだけを昇格させる段階設計にしました。

具体的には、Evaluator が NEEDS\_REVISION または GENERATOR\_FAILED を出した瞬間に、critique.jsonの`failed_checks`と`required_fixes`を`FAILURES.md`に追記します。追記はEvaluator skill末尾のpost処理として実行しています（hookではなくskill内に置いたのは、verdictとcritique構造を理解しているのがskill本体のためです）。skillがcritique.jsonを返却する直前に、`if verdict in ("NEEDS_REVISION", "GENERATOR_FAILED"): append_to_failures_md(critique)`相当のガードを1つ差し込むだけで、後段の`/failure-promote`が機能します。

.claude/workspace/FAILURES.md（例）

```
## 2026-04-20 14:32 UTC — feature/auto-fill / chunk 003_controller

- verdict: NEEDS_REVISION
- chunk kind: controller
- failed_checks:
  - grep_contains: "{feature}_params" が見つからない
- required_fixes:
  - "Params の許可漏れで Create が 422 を返している。{feature}_params メソッドを追加して strong_parameters でホワイトリスト化"
```

FAILURES.mdは`.claude/workspace/`配下に置き、`.gitignore`で除外しています。開発者ごとのローカルに持つ理由は2つあります。整理されていない試行錯誤を他のメンバーに見せないことと、単発の失敗をそのままチームルールにしないための緩衝材として使うためです。

FAILURES.mdに同じようなパターンが3件以上溜まった時点で、チームルールへの昇格候補として提案されます。3件という閾値は、単発の偶発ミスとパターン化した再発を分ける経験則として置いています（運用しながら調整中です）。判定条件は3つです。

* `verdict + failed_checks`の組み合わせが一致する
* `required_fixes`の要約が類似する（fix文を句点で分割して先頭の動詞句＋目的語を比較。例: 「{feature}\_paramsメソッドを追加」同士なら一致と見なす）
* chunkの`kind`（model / service / controller 等）が同じ

昇格候補が検出されると、`/failure-promote`skillが以下のような要約を提示します。

/failure-promoteの出力例

```
### パターン 1: Controller の strong_parameters 漏れ

- 検出回数: 5
- 最初の発生: 2026-04-10
- 最後の発生: 2026-04-13
- 典型的な required_fixes:
  - 「{feature}_params メソッドを追加して strong_parameters でホワイトリスト化」
  - 「Params の許可漏れで Create が 422 を返している」
- 昇格先候補: `.claude/rules/{project}/controllers.md`（既存ルールに追記）
```

昇格先はchunkの種類で分類しています。`.claude/rules/{project}/`配下には、Generator / Evaluatorがchunk種類ごとに参照するルール（例: `controllers.md`なら「`{feature}_params`が存在すること」のようなgrep条件）を置いています。

| chunk kind | 昇格先 |
| --- | --- |
| Controller | `.claude/rules/{project}/controllers.md` |
| Model | `.claude/rules/{project}/models.md` |
| Service | `.claude/rules/{project}/services.md` |
| Spec | `.claude/rules/{project}/spec.md` |
| Migration | `.claude/rules/{project}/migration.md` |
| 全般の設計原則 | `CLAUDE.md` |

人間が昇格を承認すると、該当ファイルに追記するDraft PRが作成されます。レビュー後にマージされると、以降のセッションで対応するchunk種のGenerator / Evaluatorがそのルールを参照するようになります。昇格済みパターンにはFAILURES.md側で`[PROMOTED]`マーカーを付け、同じパターンを再度昇格候補として検出しないようにしています。

## Step 4: /retro — セッション全体を振り返る

FAILURES.mdが拾うのはEvaluatorが出したcritiqueだけです。セッション中には、PRレビューのコメント・手修正・別のskill実行中に気づいた問題など、Evaluatorには届かない学びがたくさん出てきます。これらを取りこぼさないために、セッション終わりに`/retro`skillを走らせ、セッション全体を振り返ります。

`/retro`は、`git log`のコミット履歴、`gh api`で取得したPRレビューコメント、Generatorが付ける固定のcommit prefix以降の人間コミット（手修正の痕跡）を収集し、学びを5カテゴリに分類します。

| カテゴリ | フィードバック先 | 例 |
| --- | --- | --- |
| コードスタイル | Hook / Rubocop設定 | クォート・インデントの違反 |
| アーキテクチャ違反 | rules 更新 | Service層に書くべきロジックがcontrollerに漏れている |
| ドメイン知識不足 | memory 更新 | 社内APIの仕様、業務フロー |
| 外部連携ミス | skills 更新 | OpenAPI定義漏れ、DynamoDB設定漏れ |
| 危険操作 | Hook 追加 | 本番影響のあるコマンド実行 |

分類した項目はさらに3つの処理優先度で振り分けます。

* 即時: 個人学習。`~/.claude/projects/*/memory/`に追加。承認後すぐに適用
* PR: チーム学習。複数回発生した問題、または再発が確信できるもの。rules / HookにPRで反映
* TIL: 1回だけ発生した個人メモ。記録のみでルール化はしない

カテゴリと処理優先度は独立した2軸として扱います。「カテゴリ = どこに還元するか（rules / memory / Hook / skills）」、「処理優先度 = いつ・どの粒度で還元するか」を別軸で判断しています。例えば、同じ「アーキテクチャ違反」でも、複数回再発していればPR優先度でrules更新、1回きりならTIL優先度で個人メモに留める、という使い分けです。危険操作だけは例外で、1回でもPR優先度（Hook追加）に回します。

PRにするかの判断基準は1つだけです。

> この行を削除したらClaudeが間違えるか？

Yesなものだけルール化します。ルールは書けば書くほど効くわけではありません。書かないと忘れる行だけを残す。自分たちのプロジェクトでは、これをハーネスの健全性を保つ方針にしています。コードを2〜3ファイル読めば見つかる事柄や、Ruby/Railsの標準的なベストプラクティスは、ルール化せず省いています。

## Step 5: 失敗 → ルール → ハーネス更新の閉じたループ

3つの仕組みを組み合わせると、失敗がチームで共有できる形に落ちる経路が1本の線でつながります。

```
Evaluator が NEEDS_REVISION / GENERATOR_FAILED を出す
    ↓ Evaluator skill末尾で追記
FAILURES.md（個人ローカル・gitignored）
    ↓ 同一パターン3件以上
/failure-promote がチームルール昇格PRを作成
    ↓ レビュー・マージ
.claude/rules/{project}/（チーム共有・git管理）
    ↓ 次のセッションで
Generator / Evaluator が更新されたルールを参照
    ↓
同じパターンでつまずく頻度が下がる

＋ セッション単位では
/retro がコミット履歴 / PRレビュー / 手修正の学びを収集
    ↓ 5カテゴリ・3処理優先度に分類
memory（個人）/ rules・Hook（チーム）に還元
```

`/failure-promote`がEvaluatorで拾えた失敗を機械的に昇格させ、`/retro`が人間の観察も含めた学びを吸い上げる。役割が補完的なので、両方を運用しています。完全に再発をゼロにできるわけではなく、頻度を下げるのが目的です。

## まとめ

FAILURES.md・`/failure-promote`・`/retro`の3つで、エージェントが起こした失敗をチームルールに還元する経路を作りました。仕組みはシンプルで、個人ローカルに失敗を溜めて、繰り返したものだけチームルールに昇格させ、セッション全体は振り返りで別途拾う、というだけです。

本記事は当社のRailsプロジェクトでの運用をベースにした紹介で、前提としてSafety Hookとchunk分解の品質ゲートが先に整っていることを想定しています。この仕組みだけを切り出しても、拾う失敗の粒度が揃わないので効きが弱くなります。
