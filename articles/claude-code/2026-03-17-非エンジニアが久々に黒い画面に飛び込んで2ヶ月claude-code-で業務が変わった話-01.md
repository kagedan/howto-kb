## はじめに ── 「黒い画面」、怖くないですか？

「ターミナル」「CLI」「コマンドライン」。

エンジニアの皆さんにとっては当たり前の道具だと思います。でも、非エンジニアにとっては「何が起きているかわからない黒い画面」であり、「触ったらPCが壊れるんじゃないか」と感じている方も多いのではないでしょうか。

でも、ちょっと考えてみてください。あの黒い画面 ── コマンドラインインターフェース ── は、コンピューターの黎明期から存在しています。GUIが登場し、スマートフォンのタッチ操作が当たり前になった今でも、エンジニアやパワーユーザーほど好んで使い続けている。**半世紀以上、現役であり続けている道具**です。それだけ「便利」だから残っている、と考えるのが自然ではないでしょうか。

私はたまに触っていたので、そこまで強い抵抗感はありませんでしたが、周囲の話を聞くと「自分が触るものではない」と感じている方が多い。もったいない、と思うのです。

私は開発のマネジメントをしています。いわゆるPdMやPjMであり、プロダクトオーナーやスクラムマスターであり、ピープルマネージャーでもありますが、エンジニアではありません。コードは一切書けません。日常業務は競合調査、売上分析、会議運営、スクラムのプロセス改善、プロダクトの優先順位調整 ── いわゆる「マネジメント寄りの何でも屋」です。

そんな自分が、2026年1月末から Claude Code と Cursor を本格的に触り始めました。2ヶ月経った今、毎朝コマンドを1つ打つだけで7つのAIが並行して情報収集し、統合レポートとスライドが自動生成される環境を動かしています。

この記事は、非エンジニアの方々 ── PdM、PjM、編集、営業、ユーザーサポート、バックオフィスなど ── に向けて、「黒い画面は怖くない。むしろ、あなたの業務を最も変えてくれるのはここだ」ということを、自分の試行錯誤を通じてお伝えするものです。

## 自己紹介と前提

**立場**: 前述の通り開発部門のマネージャーです。IT・Webサービス業界で長年やってきましたが、キャリアの出発点はWebデザイナーとディレクターで、そこからPjM、PdM、スクラムマスター、プロダクトオーナーと変遷してきました。プロダクションコードを自分で書くことはありません。技術的な会話はある程度できますが、自分でプログラムを組んで何かを作った経験はほとんどないです。

**AIツール歴**: ChatGPT、Gemini、NotebookLM は業務で日常的に使っていました。Stable Diffusionなども少し触っています。過去にはChatGPTに複数のロールを演じさせてリーンキャンバスを書かせてみるなど、試行錯誤していました。

**きっかけ**: 2025年後半から、AIが「チャットで質問に答えてくれるもの」から「自律的にタスクを実行してくれるもの」に進化しつつあるのを肌で感じていました。特に Claude Code を触ったエンジニアたちが「これはヤバい」と口を揃えるのを聞いていて、年末に[Agent Skills](https://www.anthropic.com/news/skills)の発表を知り、さらに2月には[Claude Opus 4.6とAgent Teams](https://www.anthropic.com/news/claude-opus-4-6)がリリースされ、「マネジメント業務にも使えるのでは？」と思い始めたのが始まりです。

## 第1章 ── 最初の一歩：意外と簡単なインストール

### 黒い画面を開いた日

自宅のmini PCにUbuntuをインストールしてRoonを動かすなど、ちょっとマニアックなことはしたことがあります。とはいえ、UNIXコマンドなんて全然覚えておらず（ls、cd、mkdirくらい）、当時もひたすら検索して乗り越えました。

久しぶりにターミナルを開いて、Claude Codeをインストール。あっけなく終わる。インストール手順は[公式のセットアップガイド](https://docs.anthropic.com/en/docs/claude-code/setup)に詳しく書かれています。

もし全く触ったことがなければ、ここで躓くかもしれません。ターミナルに表示される文字列、`$` マーク、カーソルの点滅。「何を打てばいいんだ」「間違えたら取り返しがつかないのでは」という不安がつきまとうでしょう。Anthropicが公開している[ターミナルの基本ガイド](https://code.claude.com/docs/en/terminal-config)も参考になります。後述しますが、社内展開を進める中で、まさにそういう声を多く聞きました。

でも、当然ですが**PCは壊れません**。大丈夫です。

Claude Code の世界で最初にやることは、`CLAUDE.md` というテキストファイルを作ること。これはAIへの「自己紹介カード」のようなもので、「自分は誰で、どんな仕事をしていて、どう手伝ってほしいか」を書いておくファイルです。

```
# 私のワークスペース

## 役割
開発部門のマネージャー（非エンジニア）

## 主な業務
- 競合調査、売上分析
- 会議のアジェンダ作成・議事録
- スクラム運営の改善
- プロダクトの優先順位調整

## お願い
成果物は日本語で作ってください。
専門用語はわかりやすく説明してください。
```

これだけです。これを書いておくと、Claude Code は毎回「あなたが誰か」を理解した状態で対話を始めてくれます。毎回「私は〇〇の仕事をしていて...」と説明する必要がなくなる。Geminiの[カスタム指示](https://support.google.com/gemini/answer/16598625)やChatGPTの[Custom Instructions](https://help.openai.com/en/articles/8096356-chatgpt-custom-instructions)にも同様の設定があります。それとほとんど同じでしょう。

**非エンジニアにとってのポイント**: CLAUDE.md は「設定ファイル」ではなく「手紙」です。自然な日本語で、自分の仕事と困りごとを書くだけ。プログラミングの知識はゼロで大丈夫です。

### 最初にやったこと

最初の1週間は、手当たり次第に日常業務をClaude Codeに投げてみることから始めました。

* 「来週の定例会議のアジェンダを作って」
* 「この売上データのExcelを読み込んで、トレンドを教えて」
* 「競合A社の最近の動向をWebで調べてまとめて」

ChatGPTやGeminiでもできることです。でも決定的に違ったのは、**Claude Code はファイルを直接読み書きできる**ということでした。

チャットAIでは「ファイルをアップロードして」「結果をコピペして」という手作業が挟まります。Claude Code では「このフォルダのExcelを読んで分析して、結果をこのフォルダに保存して」と言えば、ファイルの読み込みから保存まで一気通貫でやってくれる。

使い始めて数日で「チャットAIには戻れない」と感じました。ファイルの読み書きが一気通貫でできる、というだけでそれだけの差がある。

## 第2章 ── 「スキル」という概念を知って世界が変わった

### 毎回同じ指示を出す無駄

最初の数週間、毎朝やっていたことがあります。

1. 「AI業界の最新ニュースを調べて」と打つ
2. 「漫画業界の動向も調べて」と打つ
3. 「昨日のNotionの更新をまとめて」と打つ
4. 「Slackの要約を見せて」と打つ

毎日4〜5回、似たような指示を打っていました。結果には満足していたのですが、「これ、毎日同じことやってるな...」というモヤモヤが積もっていきました。

### 「手順書」をAIに覚えさせる

Claude Code には **「スキル」** という仕組みがあります。

これは「業務手順書をAIが読める形式で書いたもの」です。たとえば、「AI業界の最新動向を調べてレポートにまとめる」という手順を `SKILL.md` というファイルに書いておくと、次からは「AI動向を調べて」の一言で、毎回同じ品質のリサーチとレポート生成が走ります。

```
# 手順
1. Google AI / Anthropic / OpenAI の公式ブログを確認
2. テックメディア（The Verge, TechCrunch等）の直近記事を検索
3. X（Twitter）での反応・議論をチェック
4. 重要度順に整理し、ビジネスインパクトを付記
5. reports/daily-report/ に日付付きMarkdownで保存
```

専門的に見えるかもしれませんが、やっていることは **「普段自分がブラウザで検索して情報を集めるときの思考プロセスを、手順として書き出した」** だけです。「まず公式ブログを見て、次にニュースサイトを確認して、SNSの反応もチェックする」── 頭の中で無意識にやっていることを言語化しただけ、とも言えます。それをAIが忠実に実行してくれる。

気がつくと、この「スキル」を2ヶ月間で22個作っていました。

そのスキルが最も威力を発揮したのが、朝の情報収集ルーティンでした。

## 第3章 ── 朝のルーティンが全自動になるまで

### Before: 毎朝の情報収集

開発部門のマネージャーとして、毎朝以下の情報をチェックする必要がありました。

| やること | 所要時間 |
| --- | --- |
| AI業界の最新動向チェック | 15分 |
| 漫画業界のニュース確認 | 10分 |
| X（Twitter）でのトレンド確認 | 10分 |
| Notionの昨日の更新確認 | 10分 |
| Slackの重要メッセージ確認 | 10分 |
| 自社プラットフォームの新規作品確認 | 5分 |

表の合計は**約60分**。これが毎朝、始業直後に発生していました。

### After: コマンド1つ、3分で完了

今はターミナルを開いて `/morning-routine` と打つだけです。

すると、**7つのAIエージェントが同時に起動**し、上記の情報収集を並行して実行します。

```
[Agent 1] AI業界動向をリサーチ中...
[Agent 2] 漫画トレンドを調査中...
[Agent 3] Notion更新を取得中...
[Agent 4] X/Twitterトレンドを調査中...
[Agent 5] Slack/GDriveサマリーを同期中...
[Agent 6] 自社プラットフォーム新着を確認中...
[Agent 7] AI活用Tipsをリサーチ中...
```

約2〜3分後、以下が自動生成されます:

* **統合ダイジェスト**（Markdown） ── 全カテゴリの情報を重要度順に統合した日次レポート
* **Marpスライド** ── そのままプレゼンに使える形式
* **編集可能なPowerPoint** ── 社内共有用にカスタマイズ可能
* **Notion同期** ── チームのNotionスペースに自動で反映

**60分 → 3分**。しかも、人が手作業でやるよりも網羅的だったりします。この間、別の作業もできます。

やったことは「毎朝の情報収集手順を、スキルとして書き出した」だけです。プログラミングは一行も書いていません。もちろん、ここに至るまでに試行錯誤はありましたが、技術的に特別なことは何もしていないはずです。

### 実際の統合レポートの中身（2026年3月17日の例）

ある日の統合ダイジェストの冒頭部分はこんな内容です:

> **Today's Top 5**
>
> 1. Nvidia GTC 2026開幕 ── 自律エージェント時代宣言
> 2. 社内プロジェクトの3月中リリースデッドライン
> 3. Claude Opus 4.6 / Sonnet 4.6 が100万トークンコンテキスト正式GA
> 4. 自社作品が米国Bookscan 2026年1月総合1位を獲得
> 5. クライアント案件の返答期限超過リスク（本日中確認必須）

AI業界の動きから自社ビジネスのリスクまで、「今日知っておくべきこと」が5項目に凝縮されています。これを見て1日の優先順位を決める ── という朝のリズムが出来上がりました。

## 第4章 ── 非エンジニアが作れるもの一覧

2ヶ月で私が Claude Code を使って構築したものの全体像です。

### 情報収集の自動化（7スキル）

| スキル | やること |
| --- | --- |
| AI動向ダイジェスト | OpenAI/Anthropic/Google等の最新情報を毎朝自動収集 |
| 漫画トレンドダイジェスト | 業界ニュース・新連載・アニメ化情報を収集 |
| Notion日次ダイジェスト | チームのNotion更新を自動サマリー |
| トレンドリサーチャー | X/テックメディア/公式ブログの3層調査 |
| 自社新着作品チェック | 自社プラットフォームの前日登録作品一覧を自動取得 |
| Slack/GDriveダイジェスト | Slack日次+GDrive週次サマリーを統合 |
| AI活用Tips | Google AI・Anthropicの実践Tipsを難易度ラベル付きで収集 |

### ドキュメント作成の効率化（6スキル）

| スキル | やること |
| --- | --- |
| PRD作成 | 8セクション構成のプロダクト要件定義書を対話形式で生成 |
| 議事録作成 | 会議の文字起こしから構造化された議事録を生成 |
| 社内コミュニケーション文書 | ステータスレポート・FAQ・インシデントレポートを定型生成 |
| 文章校正 | 文法・論理・流れのエラーを検出して修正提案 |
| ステークホルダーマップ | Power×Interestグリッドで関係者分析 |
| ドキュメント共同執筆 | 構造化ワークフローで提案書や仕様書を対話しながら作成 |

### ファイル操作（3スキル）

| スキル | やること |
| --- | --- |
| Excel処理 | CSV/Excelの読み書き・数式・フォーマット・分析 |
| PowerPoint生成 | プレゼン資料の新規作成・テンプレート適用・編集 |
| ブラウザ自動操作 | Chromeを操作してフォーム入力・データ取得を自動化 |

これらすべて、プログラミングのコードは一行も書いていません。日本語で手順を記述しただけです。正直なところ、自分でもここまで作れるとは思っていませんでした。

## 第5章 ── 「エージェント」という発想

### AIに「役割」を与える

スキルが「手順書」なら、エージェントは「専門の担当者」です。

先ほど触れたChatGPTに複数ロールを演じさせた実験 ── あのときの「事業責任者」「Webプロデューサー」「投資家」という役割分担の発想が、ここで本格的に形になった感覚があります。

Claude Code では、AIに特定の役割を与えた「サブエージェント」を定義できます。私のワークスペースには5つのエージェントがいます。

| エージェント | 担当領域 | 一言で言うと |
| --- | --- | --- |
| レポートリサーチャー | 競合調査・技術トレンド | 「調べてまとめる」専門家 |
| データアナリスト | Excel・CSV分析 | 「数字を読み解く」専門家 |
| ドキュメントコンポーザー | 議事録・提案書・メール | 「文書を整える」専門家 |
| スクラムコーチ | バックログ整理・スプリント計画 | 「開発プロセスを改善する」相談役 |
| コードレビュアー | コード品質チェック | 「コードの問題を見つける」目（読み取り専用） |

たとえば「この売上データを分析して」と言うと、Claude Code は自動的にデータアナリストを呼び出します。「来週の会議のアジェンダを作って」と言えばドキュメントコンポーザーが動く。指示する側は誰が動いているか意識する必要すらありません。

### 部署を1人で回している感覚

スキル（手順書）× エージェント（担当者）の組み合わせで、**1人で小さな部署を運営している感覚**に近いです。

朝の情報収集担当、資料作成担当、データ分析担当、プロセス改善アドバイザー。全員がAIで、指示は日本語、結果はファイルとして残る。人間の自分は「最終確認」と「判断」に集中できる。

## 第6章 ── 組織に広げる：Starter Kit という考え方

### 「自分だけ便利」では意味がない

個人の効率化は達成できました。次の課題は社内への展開です。開発チームではすでにClaude Codeを活用していますが、非エンジニアの社内メンバー ── 編集、営業、バックオフィスなど ── に広げるには、2ヶ月の構築コストを一人ひとりに繰り返させるわけにはいかない。

そこで作ったのが **Project Starter Kit** です。

### 2ステップで始められる仕組み

Starter Kit の使い方はシンプルです。

**ステップ1**: `docs/overview.md` にプロジェクトの概要を書く

```
## プロジェクトの目的
漫画編集部の業務効率化

## 主な業務
- 作家との連絡管理
- 原稿進捗の追跡
- 販促企画の立案

## 使いたいツール
Notion, Slack, Excel
```

**ステップ2**: `/project-init` と打つ

これだけで、AIが概要を読み取り、その業務に最適な:

* CLAUDE.md（AI設定ファイル）
* エージェント（専門担当者）
* スキル（業務手順書）
* コマンド（ショートカット）

を自動生成します。編集者なら編集業務向けの、営業なら営業業務向けの環境が、数分で出来上がる。

### 付属するスキル群

Starter Kit には、職種を問わず使えるスキルがプリセットされています（前章で紹介した自作スキルとは別に、汎用的なものが20種以上）:

* **Office系**: Excel処理、PowerPoint生成、Word文書操作、PDF解析
* **ドキュメント系**: PRD作成、議事録要約、文章校正、社内通知文書
* **分析系**: ステークホルダーマップ、センチメント分析、インタビュー要約
* **開発系**: ブラウザ自動操作、コードレビュー

これらは「日本語で指示するだけで動く」ものばかりです。

## 第7章 ── 社内展開の設計：心理的安全性を最優先に

### 「黒い画面」への抵抗感は想像以上に根深い

社内でAIツールの展開を計画する中で、最も重視したのは**心理的安全性**でした。

自分自身はターミナルに多少の経験があったので大きな抵抗感はありませんでしたが、周囲に話を聞くと温度差は歴然でした。「自分が触るものでは決してない」「エンジニアの領域でしょ」── 特に漫画の編集者、営業、バックオフィスのメンバーにとって、ターミナルを開くこと自体が強い心理的ハードルになっていました。

### セミナーに「安全ブリッジ」を設置

社内向けハンズオンセミナー（2時間）を設計する際、Claude Code のセッションの直前に **「心理的安全ブリッジ」** という2分間のセクションを設けました。

ここで伝えるのは3つだけ:

1. **「PCは壊れません」** ── ファイルを1つ作るだけの操作です
2. **「デモを見るだけでもOK」** ── 手を動かさなくても学びはあります
3. **「間違えても大丈夫」** ── Claude Code は確認してから実行します

さらに、セミナーの冒頭で私自身が実際に構築した morning-routine の画面を見せながら「エンジニアではありません。コードは書けません。でもこれを全部作りました」と実演することで、「自分にもできるかも」という感覚を持ってもらう設計にしました。

### 段階的なツール導入

一気に「全員Claude Code」は現実的ではありません。段階を設けました。

| ステップ | ツール | 対象 | 感覚 |
| --- | --- | --- | --- |
| 1 | Claude Desktop | 全員必須 | 「対話する」── ChatGPTの延長 |
| 2 | Claude Cowork | 全員推奨 | 「任せる」── 自律的に作業してもらう |
| 3 | Claude Code | 希望者 | 「作る」── 自分の業務ツールを作る |
| 4 | スキル・エージェント設計 | 意欲者 | 「仕組み化する」── チームの業務を変える |

ステップ3以降は強制しません。でも、ステップ1・2で「AIに任せる」体験をした人は、自然と「自分でも作ってみたい」と思うようになる ── という仮説で設計しています。

## 第8章 ── 2ヶ月間の学びと失敗

好奇心で突っ込んでいった分、失敗も多かったです。同じ轍を踏まないよう、具体的に共有します。

### 失敗1: エラーが出ると「自分のせい」と思い込む

Notion との連携がエラーになったとき、「自分の設定が間違っているに違いない」と何時間も格闘しました。結果的に、これはCursorのモデル選択が「Auto」の場合にMCPツールのパラメータが正しく渡されないという[既知のバグ](https://forum.cursor.com/t/cursor-auto-selected-model-stringifies-mcp-tool-parameters/145807)でした（[notion-mcp-server側のIssue](https://github.com/makenotion/notion-mcp-server/issues/176)にも報告されています）。

非エンジニアだと「エラー＝自分が壊した」と反射的に思いがちですが、ツール側やサービス側のバグであることも少なくありません。今後、非エンジニアがこうした環境を触るようになると、同じような場面に必ず遭遇するはずです。

**学び**: エラーが出たら、3つの層で切り分ける。(1) サービス全体の問題か、(2) 接続の中間層（MCPやツール）の問題か、(3) 自分側の問題か。自分を最初に疑わない。エラーメッセージをそのまま検索すると、同じ問題を報告している人が見つかることも多いです。

### 失敗2: 完璧を目指して手が止まる

「CLAUDE.md を完璧に書かないと」「スキルの手順を漏れなく書かないと」── 仕様書を書く癖が裏目に出て、着手できない時期がありました。

**学び**: 最初は3行でいい。使いながら育てる。CLAUDE.md もスキルも、「ver.1 は雑でいい」が正解でした。実際に動かすと「ここが足りない」「ここは余計」がすぐわかる。プロダクト開発でよく言われる「まずMVP（最小限の機能で動くものを早く出す）を出せ」は、自分のAI環境構築にも当てはまりました。

### 失敗3: 全部を1回のセッションでやろうとする

Claude Code のコンテキスト（会話の記憶容量）には上限があります。最初はそれを知らず、「ついでにこれも」「あとこれも」と欲張って巨大なタスクを一度に投げた結果、途中から品質が落ちるという現象に悩まされました。

**学び**: 大きなタスクは分割する。「調査」と「レポート作成」を別セッションにする。AIにも記憶の限界があることを理解し、「メモリ」や「スキル」という仕組みで知識を永続化する。人間のメモやマニュアルと同じ発想です。

### 失敗4: 気づいたら属人化していた

面白くてどんどん深掘りした結果、「自分しか使えない」環境が出来上がっていました。普段チームに対して「属人化を減らそう」と言っている人間が、自分で新たな属人化を作っていた。

**学び**: 早い段階から「他の人にも使ってもらう前提」で設計する。Starter Kit を作ったのは、この反省からでした。

## 第9章 ── 非エンジニアへのメッセージ

### 「コードが書けない」は意外と問題にならない

私自身、IT業界に長くいるので技術的な話は理解できますが、プロダクションコードを書く能力はありません。それでもここまで構築できた。Claude Code が求めているのは「コーディング能力」ではなく「業務の言語化能力」だからです。

AIエージェントの設計で最も重要なのは「業務を知っていること」です。どんなに優秀なエンジニアでも、あなたの業務のことはあなたほどには知りません。

* 編集者なら、作家とのコミュニケーションの勘所を知っている
* 営業なら、顧客の反応パターンを知っている
* バックオフィスなら、経理・法務の例外処理を知っている

その「業務知識」を自然な日本語でスキルに書き出せば、AIがそのとおりに動く。コードは一行もいりません。

### 始めるのに必要なもの

1. **Claude のアカウント**（Team プランを推奨 ── データが学習に使われない）
2. **テキストエディタ**（VS Code や Cursor ── 無料で使えます）
3. **好奇心と、30分の時間**

最初の30分で CLAUDE.md を書いて、1つだけ業務を頼んでみてください。「来週の会議のアジェンダを作って」でも「この資料を要約して」でも何でもいい。

「あ、動いた」。

この小さな体験が、次の一歩に繋がります。少なくとも自分はそうでした。

### 今日からできる3つのこと

**1. 自分の業務を書き出す**

箇条書きでいいので、毎日・毎週やっている業務を書き出してみてください。その中で「毎回同じことをやっている」ものが、自動化の最初の候補です。

**2. CLAUDE.md を書いてみる**

自分の役割、業務内容、困っていることを自然な日本語で書くだけ。完璧でなくていい。3行から始めて、使いながら育てる。

**3. 小さく試して、誰かに見せる**

最初の成功体験を誰かに共有すると、「自分もやってみたい」が広がります。組織全体のAI活用は、こうした小さな連鎖から始まります。

## おわりに ── 「業務を知っている人」が設計者になる時代

2ヶ月間やってみて、ひとつ確信に近いものがあります。

これまで「業務の言語化」は仕様書や手順書として行われてきましたが、それを読んで実行するのは人間でした。これからは、AIが読んで実行する。つまり、**手順書を書ける人が、そのままAIの設計者になれる**。その時代はもう始まっています。

そしてもうひとつ。**単体のAIに指示を出すだけでなく、複数のAIエージェントをどう組み合わせ、どう連携させるか ── 「オーケストレーション」が重要になってくる**ということです。7エージェントの並行実行も、スキル×エージェントで「小さな部署」を構成するのも、根本はオーケストレーションです。

これはプロジェクトマネジメントやチームビルディングの経験がある人ほど、感覚的に理解しやすい領域だと思います。人間のチームの業務設計と、AIエージェントのオーケストレーションは構造が同じです。

プロダクトマネージャーが書く PRD、プロジェクトマネージャーが書くWBS、編集者が持つ作品評価の目利き、営業が蓄積した顧客対応のノウハウ ── これらはすべて、AIスキルの「原材料」になり得ます。

半世紀以上現役であり続けた黒い画面は、AIとの組み合わせによって、非エンジニアにとっても有用な業務ツールに変わりつつあります。

まだ2ヶ月の試行錯誤の途中です。うまくいかないことも多いし、改善の余地はたくさんあります。それでも、「案外あっけなく動き出す」── その感覚を共有したくて、この記事を書きました。

何かの参考になれば幸いです。

## 補足: 技術的な詳細に興味がある方へ

この記事で触れた内容の技術的な背景を簡単に補足します。各項目にAnthropicの公式ドキュメントへのリンクを付記しているので、詳細はそちらを参照してください。また、これらのドキュメント自体をClaude（ChatやDesktop）に読ませて「この機能について教えて」と聞けば、わかりやすく説明してくれます。公式ドキュメントが英語で読みづらい場合は、まずAIに解説してもらう、というのも有効な使い方です。

### Claude Code とは

Anthropic が提供する CLI（コマンドラインインターフェース）ツール。ターミナル上で Claude と対話しながら、ファイルの読み書き、Web検索、外部サービス連携などを実行できます。チャットUIとの最大の違いは「ローカルファイルシステムへの直接アクセス」と「ツール実行の自律性」です。

→ [How Claude Code works（公式ドキュメント）](https://code.claude.com/docs/en/how-claude-code-works)

### CLAUDE.md

プロジェクトのルートに配置する Markdown ファイル。Claude Code がセッション開始時に自動読み込みし、プロジェクトのコンテキスト（誰が、何のために、どんなルールで作業するか）を理解します。いわば「AIへの永続的な指示書」。

→ [How Claude remembers your project（公式ドキュメント）](https://code.claude.com/docs/en/memory)

### スキル（SKILL.md）

`.claude/skills/` ディレクトリに配置する業務手順の定義ファイル。トリガーフレーズ（「議事録を作って」等）を定義しておくと、該当する発言をした際に自動的にそのスキルが起動します。

→ [Extend Claude with skills（公式ドキュメント）](https://code.claude.com/docs/en/skills)

### エージェント

`.claude/agents/` に定義する専門特化型のAIアシスタント。使用するモデル、アクセスできるツール、役割を指定することで、適材適所の処理が可能になります。

→ [Create custom subagents（公式ドキュメント）](https://code.claude.com/docs/en/sub-agents)

### MCP（Model Context Protocol）

Claude Code が Notion、Slack、Google Drive、自社データベースなどの外部サービスと接続するためのプロトコル。これにより「Notion の昨日の更新を取得する」「自社DBから新着作品を検索する」といった操作が可能になります。

→ [What is the Model Context Protocol?（公式ドキュメント）](https://docs.anthropic.com/en/docs/mcp)
