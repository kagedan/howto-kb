今では、日々当たり前のように聞くワード「Harness Engineering」。ただ、定義もバラバラでSNS上では多数の記事が飛び交っているので、ここで一度まとめてみることにしました。

AIを単なる「チャットボット」から「自律型エージェント」へと進化させるために必要なシステム設計、アーキテクチャ、制御メカニズム、および実装パターンという観点から読んでもらえるといいと思います。

## 1. Harness Engineeringとは何か？

2026年、ソフトウェアエンジニアリングの新たな柱として「Harness Engineering（ハーネスエンジニアリング）」が台頭しました。これは、プロンプトエンジニアリングやコンテキストエンジニアリングに続く概念であり、HashiCorpの共同創設者であるMitchell Hashimoto氏によって提唱され、広く支持を集めています。

その核心にあるのは **「馬と手綱（Horse and Reins）」のメタファー** です。  
強力だが方向性のないAIモデル（SOTAモデル）を「野生の馬」とするなら、Harness（ハーネス）は、その行動を制限し、導き、軌道修正するための「手綱（制御システム）」を意味します。

**基本方程式：**  
`AIエージェント = SOTAモデル (野生の馬) + Harness (制御システム) = エリートパフォーマー`

Harness Engineeringは、AIのDNA（モデル自体）を変えるのではなく、AIが確実に成果を出すために必要な環境とメカニズムを最適化し、生のAIの知能を信頼性が高く制御可能な生産力へと変換するためのエンジニアリング哲学とフレームワークです。

### 役割の進化（Traditional vs. Harness Era）

AIの進化に伴い、エンジニアの役割は「コード記述者」から「システム設計者」へとシフトしています。

* **従来のエンジニア:** 価値提案は「コード実装のスピードと品質」であり、コアスキルは手動コーディング、納品物はソースコードそのものでした。
* **Harness時代のエンジニア:** 価値提案は「システム設計とアーキテクチャ能力」になります。AIに対する制約、フィードバックループ、制御システムの設計がコアスキルとなり、エージェントのための信頼性の高いランタイム環境（ツール、抽象化、フィードバックループなどの支援構造）を構築することが主眼となります。我々はもはやレンガを一つずつ積む労働者ではなく、設計図を書き、ルールを定義する「アーキテクト」へと進化しています。

## 2. なぜHarness Engineeringが必要なのか？ (R.E.S.T)

AIに対する指示は「ソフトな制約」に過ぎず、品質や信頼性を保証するには不十分です。エージェントを本番環境で稼働させるには堅牢なフレームワークが必要です。その中核となるのが**R.E.S.Tフレームワーク**の4つの目標です。

1. **Reliability（信頼性）:** 予期せぬ入力や障害に直面しても、安定してタスクを完了する能力。
   * *要件:* 障害復旧（チェックポイントからの再開）、操作の冪等性（安全な再試行）、行動の一貫性。
2. **Efficiency（効率性）:** 計算、ストレージ、ネットワーク資源の効果的な使用。
   * *要件:* リソース制御（トークンやAPI呼び出しの予算管理）、低遅延な応答、高スループット。
3. **Security（セキュリティ）:** 権限のないアクセスや破壊からシステムを守る、妥協できない絶対条件。
   * *要件:* 最小権限の原則、サンドボックス実行（信頼できないコードの隔離）、I/Oフィルタリング（プロンプトインジェクションや機密データ漏洩の防止）。
4. **Traceability（トレーサビリティ）:** エージェントの意思決定や行動履歴を開発者が理解できるようにするためのデータ提供。
   * *要件:* エンドツーエンドのトレース、説明可能な意思決定、監査可能な状態（監査ログ）。

## 3. Harness Engineeringの構造（アーキテクチャとループ）

### 3.1 PPAFループ：エージェント知能のコアアーキテクチャ

自律型エージェントは、生の演算能力に決定論的な制約を課すことで、複雑なワークフローを実現します。その中核となるのが**PPAF（Perception, Planning, Action, Feedback/Reflection）ループ**です。

* **P - Perception（知覚）:** 外部環境や内部状態（ユーザープロンプト、APIペイロード、システムメトリクス、会話履歴など）からシグナルを収集します。この知覚の網羅性と正確性が、意思決定の質を直接的に定義します。
* **P - Planning（計画）:** LLMが「脳」として機能し、現在の状態と最終目標の差分を評価します。複雑な目標を実行可能なサブタスクに分解し、パスの選択、リソース割り当て、ツールの発見、リスク評価を行います。
* **A - Action（行動）:** エージェントが「エフェクター」を通じて世界と相互作用します。APIの呼び出し、コードの実行、ユーザーとの通信などを行い、ここでの精度と信頼性が成功の鍵となります。
* **F - Feedback/Reflection（フィードバック / 内省）:** 実行結果をフィードバックとして捉え、期待値とのズレを分析します。これにより、エラー修正や戦略の転換が引き起こされ、将来のサイクルのための「記憶」としてコミットされることで、継続的最適化の閉ループが形成されます。

### 3.2 Harnessの4つの次元

Harness Engineeringは、このPPAFループに対応する4つの次元に分類されます。

1. **Making the Reins（手綱を作る） - Perceptionに対応:** エージェントの知覚と行動のために、明確で安定した安全な境界とI/Oチャネル（インターフェースやプロトコル）を確立します。
2. **Riding the Horse（馬に乗る） - Planning & Actionに対応:** エージェントが自律的かつ効率的にタスクを完了できるように、コアとなる推論と実行ロジック（オーケストレーションや実行制御）を設計します。
3. **Evaluating the Horse（馬を評価する） - Reflectionに対応:** エージェントの能力、パフォーマンス、振る舞いを監視するための包括的な指標と評価フレームワークを構築します。
4. **Breeding the Horse（馬を育てる） - Reflection & Memoryに対応:** 経験から学習できるように、データフライホイール、ファインチューニング、知識の更新のための閉ループメカニズム（トレーニングや記憶システム）を設計します。

### 3.3 エージェントの進化マトリックス

エージェントの成熟度は、「認知ループ（受動的反応か、能動的な計画・内省か）」と「コンテキスト効率（手動・非効率か、自動化されたサンドボックス注入か）」の2軸で評価されます。

Harness Engineeringの真の価値は、エージェントを「非効率・受動的」な領域（基本的なQ&Aツールなど）から、高度に統合されたコンテキストを持つ「高効率・能動的」な領域（自律型エージェント）へと引き上げることにあります。

## 4. エンジニアリングシステムとしてのHarness実装

アーキテクチャレベルでは、Harnessは境界制御、ツールルーティング、決定論的フィードバックを備えた**管理されたREPL（Read-Eval-Print Loop）コンテナ**として機能します。LLMの「非決定論的な脳」を「決定論的なシェル」で包み込むイメージです。

### 4.1 コントロールプレーンとデータプレーンのアーキテクチャ

本番環境のHarnessは、論理的なコントロールプレーン（What）とデータプレーン（How）に分離され、4つのレイヤーで構成されます。

### 4.2 コアコンポーネントとその責務

エンジニアリングを支える主要なコンポーネントは以下の通りです。

* **Tool Gateway (計画段階):** ツールの登録、発見、スキーマのプロビジョニング、権限検証を一元管理します。RBAC（ロールベースアクセス制御）やスキーマの自動シリアライズなどを設計の要とします。
* **Call Interceptor (行動段階):** ツール実行の前後に、タイムアウト、クォータ、ロギングなどの横断的なロジックを注入します。AOP（アスペクト指向プログラミング）や再試行モジュールが活用されます。
* **Feedback Assembler (フィードバック段階):** 生の実行結果（Pythonの例外など）を、LLMが理解できる構造化された観察情報に変換します。エラースキーマの統一やスタックトレースの要約を行います。
* **Context State Manager (知覚/記憶段階):** トークン予算を管理し、コンテキストの保持・削除・アーカイブポリシーを統治します。情報価値に基づく優先度付けや、短期・中期・長期の階層ストレージを提供します。
* **Exception Handler (行動/フィードバック段階):** システム全体の例外を分類し、再試行やリカバリー戦略（サーキットブレーカーなど）をオーケストレーションします。

### 4.3 状態の分離とメモリインフラストラクチャ

Harness構築における最重要のアーキテクチャ決定は **「状態分離の原則」** です。  
LLMは純粋なステートレスな演算ユニット（CPU）として扱い、ユーザーセッションやタスクの進捗などの状態は、Harnessが制御する外部のContext State Managerや永続化エンジンにオフロードしなければなりません。これをプロンプトエンジニアリングで無理やり維持しようとすると、システムは予測不可能になります。

Memoryのインフラは3つの層に分類されます。

1. **L1 Sensory Memory (Context Window):** プロンプトやインコンテキスト情報。容量は限られ高コストですが、最も低遅延です。即時的な意思決定に必要なシグナルだけを残す厳密な予算管理が必要です。
2. **L2 Short-term / Working Memory:** RedisやセッションDBを使用。現在のタスクの中間状態や対話履歴を保存し、TTL（Time-To-Live）で管理されます。
3. **L3 Long-term Memory:** ベクトルデータベースやナレッジグラフ。大容量で低コストであり、RAG（検索拡張生成）などのオンデマンド検索を通じてアクセスされ、知識の蓄積と再利用に使われます。

これらのメモリを扱うためのマッピング戦略として、直近の会話を保持する「Sliding Window」、静的知識を参照する「RAG」、特定エンティティに言及した際にロードする「On-demand Loading」、複数の層を統合する「Hierarchical Memory」がシナリオに応じて使い分けられます。

### 4.4 トークン変換パイプライン

無限の世界の状態を有限のトークンに押し込むため、Harnessは呼び出しの前に**トークン変換パイプライン**を実行します。モデルに「何に注目すべきか」を委ねるのではなく、エンジニアリングによってコンテキストを能動的に構築します。

1. **収集 (Collection):** リクエストや記憶を集約。
2. **ランク付け (Ranking):** 最新性や意味的関連性に基づいてスコアリング。
3. **圧縮 (Compression):** 情報量が低くデータ量が多いコンテンツの要約。
4. **予算化 (Budgeting):** カテゴリごとのトークン制限の割り当て。
5. **組み立て (Assembly):** 構造化されたテンプレートを用いて最終的なプロンプトを作成。

### 4.5 実行環境とガバナンス（サンドボックス設計）

エージェントがシステムを破壊せずに「行動」できるよう、厳密に隔離された実行環境（サンドボックス）が必要です。

* **Level 1 (プロセスレベル):** chroot等。高速だがカーネルを共有。信頼された内部ツール向け。
* **Level 2 (コンテナ):** Docker等。業界標準。強化されたカーネルと読み取り専用ルートファイルシステムを組み合わせるのが基本戦略。
* **Level 3 (MicroVMs):** Firecracker等。独立した仮想カーネルを提供し、信頼できないコードの実行やマルチテナント環境に最適。
* **Level 4 (Full VMs):** KVM等。最高のセキュリティだが高コスト。

さらに、プランナーと実行レイヤーの間に**Policy Gateway**を配置し、RBACによる権限チェック、機密データのフィルタリング、インジェクション防御、監査ロギングを強制します。

## 5. 設計における制約と原則

システムを構築する際、モデルの不完全性（ハルシネーション等）、コンテキストウィンドウの制限、環境の予測不可能性という**3つの根本的な制約**に直面します。これらを乗り越えるための**6つのコア設計原則**が不可欠です。

1. **Design for Failure (失敗を前提とした設計):** 例外や失敗を異常ではなく「標準」として扱い、フォールトトレランスや再試行をサポートする。
2. **Contract-First (契約ファースト):** すべての相互作用を、機械が読み取れる明示的な契約（JSON Schema、API）を通じて定義する。
3. **Secure by Default (デフォルトでセキュア):** 最小権限、ゼロトラスト、多層防御を前提とする。
4. **Separation of Concerns (関心の分離):** 「何をすべきか（計画）」と「どう実行するか（実行）」を論理的・物理的に分離する。
5. **Everything is Measurable (すべてを測定可能に):** 行動やリソース消費を定量化し、最適化の道筋を作る。
6. **Data-Driven Evolution (データ駆動型の進化):** すべての実行を学習機会とし、データ収集とフィードバックの閉ループを構築する。

例えば、Function Callingにおいては、シリアライズから実行に至るまで失敗のリスクが伴います。JSONフォーマットエラー時の再生成要求や、実行失敗時の対話的な確認・再計画など、強固なフォールバックパス（Fallback Paths）の実装が不可欠です。

## 6. まとめ

モデルが同じ間違いを二度と繰り返さないようにするための仕組み（Ralph Loopによる検証や、エラーを引き金とする再計画など）を構築することが本質的な部分です。

AIがコードの記述を代替していく中で、エンジニアの役割は消滅するのではなく進化しています。信頼性の高いHarnessを設計することは、カオスと秩序のバランスを保つ作業であり、AIも人間と同様に完璧ではないことを前提として、不確実性を乗り越える回復力を備えたシステムを構築することこそが、真のエンジニアリングの知恵と言えるのではないでしょうか？
