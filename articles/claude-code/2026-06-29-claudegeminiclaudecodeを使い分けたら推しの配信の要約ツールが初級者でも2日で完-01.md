## 5. 詰まり①：推しのニュアンス問題

一番最初に詰まったのが「要約のニュアンス」問題でした。

AIに要約を作ってもらうのは簡単です。でも最初に出てきた要約を見て、「なんか違う…」 と思ったんですよね。

デフォルトのAIの要約が結構はっちゃけた表現で。「〇〇さんが楽しそうに語っていました！！」みたいな感じで、勝手に感情を解釈した表現が出てくるんです。「さすが〇〇さんですね！！」とかの主観的な評価とか。

これ、自分だけが見るツールならまだいいんですが、今回はXの公開アカウントに投稿しようと思っていたので、「誰が見ても不快にならないように、なるべく誤解が発生しないように、推しが伝えたいニュアンスを勝手にこちらが解釈しない」「推しの性格・バックグラウンドに沿った要約を作成する」というのを方針にしました。

そこでまずはGrokに推しのXの投稿を分析してもらって、キャラクターの解像度を上げることにしました。出てきた結果の概要がこちらです。  
（もっとプログラム内では詳細な情報を記載しています）

**・明るくポジティブでファンサービス精神が強い  
・天真爛漫で親しみやすいキャラクター  
・ファンとの距離が近く触れ合いを大切にする  
・責任感・正義感が強く誠実  
・エンターテイナー気質でコミカルな要素も多い**

この分析をもとに、プロンプトに落とし込んだルールがこちらです。

**「明るくポジティブな性格を継承する」**  
→ ふざけすぎず堅苦しくもない自然なトーン  
→「！」は全体で1〜2個程度に抑える（はっちゃけすぎ防止）

**「感情の勝手な解釈をしない」**  
→「楽しそうに」「嬉しそうに」など推測表現はなるべく禁止  
→ 実際に起きたこと・言ったことのみを事実として書く。

**「具体性を大切にする」**  
→「自身の考えを共有」「思いを語る」などの抽象表現のみは禁止  
→ 必ず「何を」話したかを併せて具体的に書く

**「キャラクターは自然に滲み出る程度でよい」**  
→ 推しのキャラクターを前面に押し出すのではなく、文章の雰囲気に自然に出る程度にする

ここは正直単なるこだわりですよね。  
でもこれをプログラムに入れるのと入れないとで文章のニュアンスが結構変わりました。  
推しへの解像度がそのままプロンプトの品質になる、という体験でした。

**▼ 有料部分で公開していること**  
実際にプログラムの中に書いた「配信者プロフィール」と「要約の文体ルール」をそのまま公開しています。「推しのキャラクターをどういう言葉でプロンプトに落とし込んだか」が具体的に見られます。自分の推しバージョンに書き換える際の参考にしてください。

## 6. 詰まり②：AIは文字数を守らない

次に詰まったのが「X投稿の文字数問題」です。

私はXで認証アカウントを利用していないので、1投稿140文字制限があります。なのでGeminiに「140文字で要約して」と指示したんですが、普通に140文字を超えてくるんですよね。

AIって文字数の指定が苦手なようです。トークン単位で考えているので、日本語の文字数を正確にカウントできないらしい。そこでClaudeとGeminiに相談したところ、「140文字と指定するより、少し厳しめに120文字と指定した方がいい」アドバイスをもらいました。

試してみたら確かに改善されました。でも**今も完璧には守ってくれないことがあります**。原因は正直不明です。

なので最終的な解決策はこうしました。

**プロンプトで120文字と指定する  
　　+  
GUIに「現在●文字」というカウンターを表示する  
　　+  
投稿前に目視でチェック＆修正する**

![](https://assets.st-note.com/img/1782663665-ZUMlum5EDeGBaXrzjTQxbW30.png?width=1200)

AIが作った要約では140文字をオーバーしている

![](https://assets.st-note.com/img/1782663674-Y06nHy4LUlhxKRgIW1rJqXSA.png?width=1200)

## 7. 詰まり③：長い配信でGroqが止まる

次に詰まったのが「文字起こしのレート制限問題」です。

このツールでは音声の文字起こしにGroqというサービスを使っています。無料で使えて精度も高いんですが、1時間あたりに処理できる音声の量に上限があります。

推しの配信って2時間を超えることがざらにあって。音声を15分ごとに分割して順番に文字起こしするんですが、9チャンク目（＝2時間15分あたり）で突然止まるんですよね。

```
チャンク 1/11 完了 ✅
チャンク 2/11 完了 ✅
　　・・・
チャンク 9/11 → レート制限エラー 🚫
```

しかも最初からやり直しになると、1〜8チャンク分の処理が全部無駄になってしまいます。

そこで3つの対策を実装しました。

**① 成功したチャンクをキャッシュ保存**  
→ 途中で止まっても成功済みの分はスキップして再開できる

**② レート制限エラー時に自動でリトライ**  
→ 処理しすぎると「429エラー（＝送りすぎだからちょっと待って、というサーバーからのメッセージ）」が返ってきます。このエラーメッセージの中に「XX秒後に再試行してください」という情報が含まれているので、その秒数を自動で読み取って待機してからリトライする仕組みを作りました

**③ 多様な時間表記に対応する**  
→ ②の「待機時間を自動で読み取る」処理を実装しようとしたとき、Geminiからこんな指摘をもらいました。「待機時間の表記って『290s』だけじゃなくて『4m33s』や『4.5s』など色々な形式がありますよ」と。言われてみれば確かに。この指摘がなければ特定の形式にしか対応できず、想定外のエラーメッセージが来たときに止まっていたと思います。Geminiのレビューが刺さった瞬間でした。

## 8. 詰まり④：投稿フォーマットの試行錯誤

最後に詰まったのが「X投稿のフォーマット設計」です。

技術的な問題というより、**何をどの順番で投稿するか**という設計の話です。これが意外と一番悩みました。

悩んだのがこれです。推しの配信って、毎回内容が違います。ライブやイベントの告知など「今後の予定」を話す回もあれば、全然触れない回もある。

最初は固定のフォーマットで作っていたんですが、「今後の予定がない回に無理やり枠を作っても意味がない、お金もかかるし。」と気づいて。

そこで**配信内容によって投稿数が動的に変わる**という設計にしました。

```
1投稿目（毎回固定）
〇〇さんがツイキャスを行いました！
開始時刻：xx時xx分
配信時間：xx分
内容はツリーをチェック👇

2投稿目（毎回固定・AI生成）
配信内容の要約

3投稿目（毎回固定・AI生成）
盛り上がりポイント

4投稿目（今後の予定がある回だけ追加・AI生成）
ライブ・イベントなどの告知情報

最後の投稿（毎回固定）
フルはコチラからチェックしてください👇
（アーカイブURL）
#ツイキャス #〇〇
```

「今後の予定があるかどうか」はAIが配信内容から自動で判断して、ある場合だけ投稿を追加する仕組みです。

毎回同じフォーマットで無理やり埋めるより、**その回の配信に合った投稿になる**方が読者にとっても自然だと思いました。

プログラムを自分ですると条件分岐とか頭こんがらがっちゃうタイプなんですが、AIだと簡単に柔軟な対応ができるのがとても嬉しいポイントでした。
