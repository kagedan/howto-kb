## きっかけ

Claude CodeでUI作ると、AIくさいUIが出てくる。動くし壊れてもいないけど、なんか違う。余白が全部同じ幅だったり、ボタンが何に効くか分かりにくかったり、見た目はきれいなのに情報が頭に入ってこなかったり。

UIは直感的に使えて、情報がちゃんと設計されていてほしい。その基準を毎回自分の言葉で伝えるのが面倒で、スキルにまとめた。

名前は「間（ma）」。軸は2つ。情報を人間が最も認識/操作しやすいUIにすること。装飾は情報のためのもので、デザインの全てに意味がなければならないこと。この2つだけ決めて、あとは具体的な検出パターンを5つの視点に散らした。

<https://github.com/yyyyyyy0/ma-skills>

## インストール

```
npx skills add yyyyyyy0/ma-skills -g --skill '*'
```

`ma` と `ma-review` は参照ファイルを含むのでディレクトリごと入る。手動なら：

```
git clone https://github.com/yyyyyyy0/ma-skills.git /tmp/ma-skills
cp -R /tmp/ma-skills/skills/* ~/.claude/skills/
```

## スキル一覧

| コマンド | 何をするか |
| --- | --- |
| `/ma` | UI新規設計。情報構造→認知順序→ビジュアル→実装方針 |
| `/ma-review` | 5軸で監査して SHIP / SHIP WITH NOTES / REVISE 判定 |
| `/ma-system` | トークン一貫性（角丸、余白、色のドリフト） |
| `/ma-legibility` | 読めるか（コントラスト、フォーカスリング、エラー文） |
| `/ma-mapping` | 何に効くか（Delete/Apply/Save の対象不明問題） |
| `/ma-flow` | 視線動線（CTA位置、読み順、余白の構造的役割） |
| `/ma-reduction` | 消せるものを消す（冗長コピー、不要確認、情報過多） |

`/ma` が設計、残り6つがレビュー。

## `/ma` で設計

```
/ma KPIカード3枚、折れ線チャート1つ、テーブル1つのダッシュボード
```

普通にUI作らせると「見た目の案」が返ってくる。`/ma` だと「なぜこの構成か」の根拠がセットで出る。認知順序、余白ルール、状態遷移、実装方針まで。

後から「チャート上にしたい」と言われたとき、「認知順序が崩れるけどいい？」と根拠つきで返せる。

## `/ma-review` で監査

素のClaude Codeにアカウント設定画面を作らせて、`/ma-review` にかけてみた。  
![AIが作ったアカウント設定画面](https://static.zenn.studio/user-upload/1096726f1cfe-20260323.png)

見た目はまとも。サイドバーナビ、カードレイアウト、トグルスイッチ。普通に使えそうに見える。結果：

```
ma-system   | セクション間余白     | 全セクション32px均一、危険ゾーンとの断絶がない | mid
ma-legibility | ファイル制限テキスト | text-gray-400、コントラスト比2.7:1（基準4.5:1）| high
ma-mapping  | 「変更を保存」ボタン  | 2箇所で同一ラベル、スコープが不明           | mid
ma-mapping  | 「変更」ボタン       | カード変更かプラン変更か曖昧               | mid
ma-mapping  | 「アカウントを削除」  | confirm()のみ、保護が弱い                | high
ma-flow     | パスワード変更フォーム | 3フィールド常時表示、変更意図なくても認知負荷  | mid
ma-reduction | プラン機能一覧       | 設定画面に常時表示する必要がない            | low

Verdict: REVISE（high 2件）
```

high 2件。ファイル制限テキストのコントラスト不足と、アカウント削除の保護の弱さ。どちらも見た目からは気づきにくい。

「変更を保存」が2箇所で同じラベルなのも、言われてみれば「プロフィールを保存」「通知設定を保存」のほうが親切。パスワード変更フォームが常時開いているのも、折りたたんでおけば認知負荷が減る。

判定は重大度だけ。low が何個あっても SHIP WITH NOTES だが、high が1個あれば REVISE。

指摘を全部取り込んだのがこれ：

![指摘反映後のアカウント設定画面](https://static.zenn.studio/user-upload/f657311f747b-20260323.png)  
ボタンのラベルにスコープがついて、パスワードフォームが折りたたまれて、危険ゾーンが他セクションから視覚的に断絶されている。派手な差ではないけど、使ったときの迷いが減る。

## 1軸だけ使う

全部回さなくていいときは直接叩く。

```
/ma-mapping このSaveボタン、何が保存されるか分からない
/ma-flow このLPのCTAが目に入ってこない
```

個人的に一番使うのは `/ma-mapping`。AIが作るUIはボタンの対象が曖昧なことが多い。「Delete」とだけ書いてあるけど行を消すのかカードごと消すのか分からない、みたいなやつ。

出力フォーマットは全スキル共通なので、後から `/ma-review` でまとめることもできる。

## 課題

テキストベースなので、CSSの描画崩れは検知できない。実際、修正版のトグルスイッチでON状態の白丸がトラックからはみ出していたが、DOMスナップショットでは `checkbox [checked]` としか見えないので `/ma-review` は素通りした。レイアウト情報も同様で、`/ma-flow` にはコードか口頭で構造を伝える必要がある。スクリーンショットを直接読めるようになったら変わると思う。

SKILL.md に入っているのは自分の設計判断なので、別チームがそのまま使うと合わないかもしれない。Markdown書き換えればいいだけだけど、チームで設計方針を合意するのはまた別の話。
