## はじめに

Claude Code の Skills（スキル）機能を触りながら、「スキルがいつ・何を読み込むのか」「分析スコープをどう選ぶべきか」を掘り下げていったところ、最終的に **"有限なコンテキストをどう配分するか"** という、より大きなテーマに行き着いた。

この記事は、その思考のプロセスと、**Claudeへの疑問の問いかけの回答からたどり着いた整理**（特に後半の「4層モデル」）をまとめたものである。Skills 単体の使い方にとどまらず、Claude Code 全体に通底する設計思想として読んでもらえると嬉しい。

## 前提：Skill は「段階的開示」で読み込まれる

Skills の挙動を理解する鍵は、**Progressive Disclosure（段階的開示）** という仕組みにある。スキルの中身を一度に全部読み込むのではなく、必要になった分だけ段階的に読み込む。具体的には3つの層に分かれる。

| 層 | 中身 | いつ読まれるか |
| --- | --- | --- |
| 層1 | 見出し（名前・説明文 / YAMLフロントマター） | **セッション開始時に常に**（ごく軽い） |
| 層2 | 本体（手順） | **依頼が関連したと判断されたとき** |
| 層3 | 詳細（参照ファイル・分析対象） | **その作業で実際に必要になったとき** |

本に例えるなら、常に手元にあるのは目次（層1）だけ。該当する章は読むときに開き（層2）、巻末の付録は本当に要るときだけめくる（層3）。これにより「深い知識を無制限に持てるのに、普段は軽い」という両立が可能になる。

### ここが最初の落とし穴

「本体（層2）が読み込まれるきっかけ」は何か？ 答えは **説明文（層1）と依頼のマッチング** である。スコープでも、ファイル数でもない。説明文が「どんなときに使うスキルか」を的確に表していないと、Claude が「関連する」と判断できず、本体が読み込まれない。

つまり **スキルが「いつ起動するか」は、説明文の書き方で設計者がコントロールする**。これがスキル作りの急所になる。

## 本題：分析スコープ3種類で「何を読みに行くか」

題材として「Vue コンポーネントを分析して最適化を提案するスキル」を考える。作成時に「分析スコープ」を選ぶ場面があり、ここが挙動を大きく左右する。

### スコープ1：Single Component（1ファイルだけ）

```
client/src/
├─ views/
│  ├─ Restocking.vue   ← これだけ読む
│  ├─ Dashboard.vue    （読まない）
│  └─ Orders.vue       （読まない）
├─ components/         （読まない）
└─ composables/        （読まない）
```

* **依頼のしかた**：対象を1つに絞って頼む（例：`Restocking.vue を分析して`）。対象が不明なら聞き返される。
* **性格**：1ファイルを深く・正確に見る。出力が軽い。ただし**ファイルをまたぐ問題は見えない**。

### スコープ2：Entire views/（views フォルダ全体）

```
client/src/
├─ views/
│  ├─ Restocking.vue   ← まとめて読む
│  ├─ Dashboard.vue    ← まとめて読む
│  └─ Orders.vue       ← まとめて読む
├─ components/         （読まない）
└─ composables/        （読まない）
```

* **依頼のしかた**：対象を指定せず、まとめて頼める（例：`主要画面を分析して`）。
* **性格**：画面どうしの整合性・重複に気づける。出力と消費は増える。

### スコープ3：Full client/src/（フロントエンド全体）

```
client/src/
├─ views/          ← 全部読む
├─ components/      ← 全部読む
└─ composables/     ← 全部読む
```

* **依頼のしかた**：アプリ全体をまとめて頼める（例：`アプリ全体を分析して最適化を提案して`）。
* **性格**：全体構造を踏まえた最適化ができる。最も重く、出力が膨大で、一つ一つの分析は浅くなりがち。

## 議論で生まれた疑問と、その答え

ここからが、実際に手を動かしながら詰めていった論点である。

### Q1. スコープが広いと、本体は呼び出されやすくなる？

**半分そのとおり**だった。

直接的には、本体が呼び出されるトリガーは「説明文と依頼のマッチング」であり、スコープではない。しかし、スコープが広いスキルは **説明文も広く書かれがち** で、広い説明文は多くの依頼にマッチしやすい。

```
スコープが広い → 説明文が広く書かれる → 多くの依頼にマッチ → 本体が呼び出されやすい
```

つまり「スコープが直接トリガーする」のではなく「**説明文を介して間接的に**」効く。トリガーを最終的に握るのは説明文だ、という核は変わらない。

### Q2. 似たスキルが複数あったら、両方読まれる？

**両方読まれることがある**。

Claude は依頼を受けると、各スキルの説明文を見比べて、関連すると判断したものの本体を読み込む。説明文が似ていると「どちらも同じくらい関連して見える」状態になり、両方読まれたり、選択が曖昧になったりする。

* **対策**：複数スキルを持つなら、それぞれの説明文で「いつ使うか」を**すみ分け**させる。
* **確実に1つだけ使いたい**なら、スキルを名指しで呼び出す（スラッシュコマンド等）。

### Q3. スコープ1だと、ファイルをまたぐ修正漏れが起きない？

**起きうる**。これは正しい懸念だった。

スコープ1は「1ファイル内で完結する問題」しか見ないので、別ファイルとの重複・不統一・全体構造の問題は**原理的に見えない**。「画面間の整合性」「重複の発見」といったまたぎの視点が欲しいなら、スコープ2や3を選ぶのが対策になる。

> スコープの広さ ＝ **見落としの少なさ** と **消費の重さ** のトレードオフ。

## 到達点：コンテキストを抑える「4層モデル」

「全体設計が必要でスコープ3を使いたい。でもコンテキストが重い」――この課題への対策を整理すると、効く場所ごとに4つの層に分けられた。

その前に、**スコープ3の重さは2種類ある**ことを押さえておく。

* **入力側の重さ**：大量のファイルを読み込むこと自体
* **出力側の重さ**：分析結果（レポート）が膨大になること

この2つの「どちらに効くか」で対策を整理する。

### 層1：そもそも一度に全部やらない（分割）

最も効く。「全部一気に」ではなく「まず views を、次に components を」と領域を分けて順に分析させる。各回が軽く済む。**「全体を見たい」と「一度に全部読む」は、必ずしもイコールではない**。

### 層2：入力を減らす（context fork / サブエージェント）

大量のファイル読み込みを別コンテキストに逃がす。**入力側の重さ**に効く。メインの会話には「結果」だけが返る。分析専門のサブエージェントに全体を見させて結果だけ受け取る形も同じ役割。

### 層3：出力を逃がす（Scratchpad）

分析結果を会話に溜めず、ファイル（例：`ANALYSIS.md`）に書き出す。**出力側の重さ**に効く。会話には要約だけ残し、詳細はファイルに逃がす。必要なときだけ読み直す。

### 層4：圧縮で受け止める（/compact）

それでも会話が膨らんだら `/compact` で「分析結果の要点は残して」と圧縮する。最後の受け皿。

### まとめ表

| 層 | 手段 | 効く場所 |
| --- | --- | --- |
| 層1 | 領域ごとに分割して順に実行 | 全体（最も効く） |
| 層2 | context fork / サブエージェント | 入力側（読み込み） |
| 層3 | Scratchpad（ファイルに書き出す） | 出力側（結果） |
| 層4 | `/compact` | 会話全体（最後の受け皿） |

実務的には **層1＋層3を基本**に、必要に応じて層2を足すのが現実的だと考えている。

## 一段上の視点：それはスキルでやるべきか

技術的な対策は上記のとおりだが、一段引くと「**そもそも全体分析をスキル1つで抱え込むべきか**」という問いもある。

アプリ全体を多角的に設計・監査するタスクは、**Agent Teams**（セキュリティ・パフォーマンス・UX を別エージェントに分担）が想定する領域でもある。各エージェントが別コンテキストを持つので、それ自体が層2（入力の分散）を体現している。

つまり「全体設計はスコープ3のスキルで抱え込む」より「**複数のサブエージェントに領域を分担させる**」ほうが、コンテキストの観点で自然なこともある。スコープの話は、サブエージェントやエージェントチームという上位の設計判断につながっていく。

## おわりに

Skills のスコープという小さな選択から出発したが、突き詰めると Claude Code のあらゆる機能――スキル、サブエージェント、フック、エージェントチーム、ワークツリー――が、**「有限なコンテキストをどう配分するか」** という一つの思想で貫かれていることが見えてきた。それぞれは「重い作業をどう分割・隔離・外部化するか」の異なる答えだ。

この視点を持っておくと、各機能を「バラバラの便利機能」ではなく、一貫した設計思想の現れとして捉えられる。スコープを選ぶときも、対策を考えるときも、「これは入力を減らす手か、出力を逃がす手か、そもそも分割する手か」と問えるようになる。
