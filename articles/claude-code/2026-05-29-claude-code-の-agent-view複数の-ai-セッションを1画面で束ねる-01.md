## はじめに

Claude Code を使っていると、こんな場面が増えてきます。

* バグ修正を Claude に任せている間、別の作業も進めたい
* 「PR レビュー」「フレーキーなテストの調査」「機能追加」を**同時並行**で走らせたい
* でもターミナルのタブを行ったり来たりするのは面倒

この「複数のセッションを並行で動かして、必要なものだけ手当てする」という使い方を、1画面でまとめて扱えるようにしたのが **Agent View** です。Anthropic が 2026年5月にリサーチプレビューとして公開しました。

この記事では、Agent View が何者で、どう開き、どう使うのかを、コマンドつきで整理します。

## Agent View とは

Agent View は、`claude agents` で開く「**バックグラウンドセッションの一覧画面**」です。

* いま何が動いているのか
* どのセッションが自分の入力を待っているのか
* どれが終わったのか

これらを、長いログをスクロールせずに一目で把握できます。新しいセッションをここから投入（ディスパッチ）し、手が必要になったものだけ開いて対応する、という流れになります。

ポイントは「**バックグラウンドセッション**」という考え方です。通常 `claude` で起動したセッションは、そのターミナルに紐づいていて、閉じると終わってしまいます。一方バックグラウンドセッションは、ターミナルに紐づかず動き続ける独立した Claude Code の会話です。後から開いて返事をして、また離れる、ということが自由にできます。

> 普通のターミナル＝「席に座っている間だけ作業してくれる助手」  
> バックグラウンドセッション＝「席を立っても作業を続け、用があれば呼べる助手」

## まずは開いてみる

シェルで次を実行します。

すると全画面に切り替わり、下部に入力欄、上部にセッションの一覧テーブルが表示されます。`Esc` でいつでもシェルに戻れますが、**戻ってもセッションは動き続け**、次に開いたときにまた一覧に出てきます。

`claude` の代わりに `claude agents` を起点にして、すべての作業をここからディスパッチする、という使い方もできます。

## セッションをディスパッチする（投入する）

新しいバックグラウンドセッションを始める方法は3つあります。

### 1. Agent View の入力欄から

下部の入力欄にタスクを書いて `Enter` を押すと、そのタスクで新しいセッションが始まります。

ここで重要なのは、**入力するたびに毎回「新しいセッション」が立ち上がる**こと。続けて別のプロンプトを打てば、1つ目への追記ではなく2つ目のセッションが並行で走ります。こうして何個も並列に動かせます。

### 2. 作業中のセッションから（`/bg`）

いま対話しているセッションをバックグラウンドに送りたいときは、セッション内で次を実行します。

エイリアスの `/bg` でも同じです。`/bg run the test suite and fix any failures` のように、送り際に追加の指示を渡すこともできます。

また、空のプロンプトで `←`（左矢印）を押すと、現在のセッションをバックグラウンド化して Agent View を開く、という操作が1ステップでできます。

### 3. シェルから（`--bg`）

ターミナルから直接バックグラウンド起動するには `--bg` を付けます。

```
claude --bg "investigate the flaky SettingsChangeDetector test"
```

特定のサブエージェントをメインとして走らせたいときは `--agent` と組み合わせます。

```
claude --agent code-reviewer --bg "address review comments on PR 1234"
```

`--name` で Agent View 上の表示名を付けられます。

```
claude --bg --name "flaky-test-fix" "investigate the flaky SettingsChangeDetector test"
```

バックグラウンド化すると、短い ID と管理用コマンドが表示されます。

```
backgrounded · 7c5dcf5d · flaky-test-fix
  claude agents             list sessions
  claude attach 7c5dcf5d    open in this terminal
  claude logs 7c5dcf5d      show recent output
  claude stop 7c5dcf5d      stop this session
```

## 一覧の見方

Agent View は、手当てが必要なものが上に来るようにセッションをグループ分けして表示します（`Ready for review` → `Needs input` → `Working` → `Completed`）。各行には、セッション名・いまの活動内容・最後に状態が変わってからの経過時間が並びます。

```
Needs input
  ✻ power-up design           needs input: double jump or wall climb?       1m

Working
  ✽ collision detection       Edit src/physics/CollisionSystem.ts           2m

Completed
  ✻ title screen              result: menu, options, and credits done       9m
```

### 状態アイコン（色）

行頭のアイコンの色と動きで状態がわかります。

| 状態 | 表示 | 意味 |
| --- | --- | --- |
| Working | アニメーション | ツール実行中・応答生成中 |
| Needs input | 黄色 | 質問や許可の判断を待っている |
| Idle | 暗い | やることがなく、次の指示を待っている |
| Completed | 緑 | タスクが正常に完了した |
| Failed | 赤 | エラーで終了した |
| Stopped | グレー | `Ctrl+X` や `claude stop` で停止された |

### プロセスの形（アイコンの形状）

色とは別に、アイコンの「形」が裏側のプロセスが生きているかを示します。

* `✻` / アニメーションの `✽`：プロセスが生きていて即応する
* `∙`：プロセスは終了済み。それでも peek・返信・attach は可能で、Claude は中断地点から再開する
* `✢`：`/loop` セッションが次の実行までスリープ中（実行回数とカウントダウンを表示）

### PR ラベル

セッションが Pull Request を開くと、行の右端に `PR #1234` のようなラベルが付きます。番号の色が PR の状態（チェック待ち＝黄、通過＝緑、マージ済み＝紫、ドラフト/クローズ＝グレー）を表すので、**緑になったらレビューしてマージ**、というのが多くのタスクのゴールになります。

### 行のサマリ生成について

各行の1行サマリは Haiku クラスのモデルで生成されています。作業中はおおむね15秒に1回（＋各ターン終了時）更新され、その都度ごく短いリクエストが通常どおり課金・処理されます。

## 様子を見る・返事する・乗り込む

| 操作 | キー | 内容 |
| --- | --- | --- |
| Peek（覗き見） | `Space` | 選択行の最新出力や待っている質問を、全文ログを開かずに表示。そのまま返信も可能 |
| Attach（乗り込む） | `Enter` または `→` | そのセッションの全対話に入る。`claude` を実行したのと同じ状態になる |
| Detach（離れる） | 空プロンプトで `←` | 一覧に戻る。**セッションは止まらず動き続ける** |

多くの場合、peek パネルだけで用が足ります。選択肢つきの質問なら数字キーで回答でき、`!` を先頭に付ければ Bash コマンドを送れます。

attach 中は通常のセッションとまったく同じで、すべてのコマンド・ショートカット・機能が使えます。なお `Ctrl+C` は attach 中も「実行中の応答を中断する」標準動作のまま（離脱したいときは空プロンプトで `Ctrl+C` を2回）。セッションを中から終了するには `/stop` を使います。

## よく使うキーボードショートカット

`?` を押すとその場で全ショートカットが見られます。代表的なものだけ抜粋します。

| ショートカット | 動作 |
| --- | --- |
| `↑` / `↓` | 行を移動 |
| `Enter` | 選択セッションに attach（入力欄に文字があればディスパッチ） |
| `Space` | peek パネルの開閉 |
| `Shift+Enter` | ディスパッチして即 attach |
| `Ctrl+S` | グルーピングを「状態」と「ディレクトリ」で切替 |
| `Ctrl+T` | セッションをピン留め（アイドルでもプロセスを生かす） |
| `Ctrl+R` | セッション名を変更 |
| `Ctrl+X` | 停止（2秒以内にもう一度で削除） |
| `Esc` | peek を閉じる・入力をクリア・終了 |

## シェルからセッションを管理する

Agent View を開かずに、シェルから直接操作することもできます。各セッションには短い ID が割り当てられています。

| コマンド | 用途 |
| --- | --- |
| `claude agents` | Agent View を開く |
| `claude agents --json` | 稼働中セッションを JSON で出力して終了（スクリプト向け） |
| `claude attach <id>` | このターミナルでセッションに乗り込む |
| `claude logs <id>` | 最近の出力を表示 |
| `claude stop <id>` | セッションを停止（`claude kill` も可） |
| `claude respawn <id>` | 会話を保ったまま再起動（更新後のバイナリに乗せ替える等） |
| `claude rm <id>` | 一覧から削除 |
| `claude daemon status` | 後述の supervisor の状態を表示 |

## ファイル編集はワークツリーで隔離される

並列でセッションを走らせると、「同じファイルを取り合って壊れないの？」という不安が出ます。Claude Code はここをうまく処理します。

バックグラウンドセッションは作業ディレクトリで始まりますが、**ファイルを編集する前に `.claude/worktrees/` 配下の独立した git ワークツリーに自分を移動**します。これにより、並列セッションは同じチェックアウトを読みつつ、書き込みはそれぞれ別の場所に行います。

git リポジトリでない場所では隔離されないため、同じファイルを編集する並列セッションは避けます。ワークツリー隔離が不都合なリポジトリでは、`.claude/settings.json` に次を書いて無効化できます（v2.1.143 以降）。

```
{
  "worktree": {
    "bgIsolation": "none"
  }
}
```

## 裏側：supervisor プロセス

バックグラウンドセッションは、ターミナルや Agent View とは別の「**ユーザーごとの supervisor プロセス**」がホストしています。

* 初めてセッションをバックグラウンド化するか Agent View を開いたときに自動で起動する
* ターミナルや Agent View を閉じても、投入済みの作業は動き続ける
* 認証は対話セッションと同じ資格情報を使い、モデル API 以外への追加接続はしない

完了して約1時間放置されたセッションは、リソース節約のためプロセスが停止されます（`Ctrl+T` でピン留めしたものは除外）。状態と会話履歴はディスクに残るので、次に attach・peek・返信したときに中断地点から新しいプロセスで再開します。

なお自動アップデータが新バイナリに置き換わると、supervisor はそれを検知して新バージョンに再起動します（ローカルのファイル監視であり、ネットワークチェックではありません）。

## モデルとパーミッションモード

* ディスパッチ時のデフォルトモデルは、ヘッダーに表示されているモデル（ユーザー設定の `model`）。`claude agents --model opus` のように起動時に上書きできます。
* セッションごとに別モデルにもできます（attach して `/model` で `s` を押すとそのセッションだけ切替）。
* パーミッションモードは起動方法に依存します。`/bg` や `←` でバックグラウンド化した場合は**現在のモードを引き継ぐ**ため、`acceptEdits` や `auto` にしていたセッションはそのまま。入力欄や `claude --bg` から新規ディスパッチした場合は、そのディレクトリの `defaultMode` が使われます。

`bypassPermissions` や `auto` のような「見ていなくても自動で動く」モードは、一度 `claude` を対話的に起動して受け入れるまで使えません。監視していないセッションが無断で動くのを防ぐためです。

## 使うときの注意点（リミテーション）

リサーチプレビューということもあり、いくつか前提があります。

* **使用量を消費する**：バックグラウンドセッションも対話セッションと同じくサブスクリプションのクォータを消費します。10個並列で動かせば、消費もおよそ10倍速くなります。
* **ローカル実行**：バックグラウンドセッションは自分のマシンで動きます。スリープは越えて保持されますが、**マシンをシャットダウンすると停止**します（次回 attach/peek/返信で再開可能）。
* **ワークツリー削除**：前述のとおり、Claude が作ったワークツリーはセッション削除で消えます。

Agent View をまるごと無効化したい場合は、設定 `disableAgentView` を `true` にするか、環境変数 `CLAUDE_CODE_DISABLE_AGENT_VIEW` を設定します。

## 他の並列実行手段との位置づけ

Claude Code で「並列に動かす」手段はいくつかあり、目的で使い分けます。

| 手段 | ざっくりした役割 |
| --- | --- |
| **Agent View** | 独立した複数のバックグラウンドセッションを1画面で監視・操作する |
| **サブエージェント** | 1つのセッションの中で、調査などの一部を別 Claude に任せる |
| **エージェントチーム** | 互いにメッセージを送り合う複数セッションを連携させる |
| **ワークツリー** | 同じリポジトリの別ブランチを並行で編集する土台 |

Agent View は、これらのうち「**人が見ていなくても進められる独立タスクを、複数まとめて回す**」ための入り口だと捉えるとわかりやすいです。

## まとめ

* Agent View（`claude agents`）は、バックグラウンドセッションを1画面で束ねるダッシュボード
* セッションは `--bg` / `/bg` / 入力欄から投入でき、入力ごとに新しいセッションが並列で立つ
* 行の色と形で状態が一目でわかり、peek で覗いて返信、attach で乗り込む
* ファイル編集はワークツリーで隔離され、supervisor がターミナルと切り離して走らせ続ける
* クォータ消費・ローカル実行・ワークツリー削除の3点に注意

「Claude に任せて、自分は別のことをする」を本格的に回したくなったら、Agent View が中心的な道具になります。まずは `claude agents` を開いて、軽いタスクを1つディスパッチしてみるところから始めるのがおすすめです。
