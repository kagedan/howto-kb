# はじめに

今回は、Claude Codeの**Agent Teams（以下、エージェントチーム）** 機能についてまとめてみました。

少し似たような機能に**サブエージェント**機能がありますが、その詳しい内容や**エージェントチームとの違い**についてはこちらの記事をご参照ください。

# エージェントチームとは

エージェントチームとは、単一のLLMではなく**複数のAIエージェントを協調させて開発・分析タスクを実行する仕組み**のことです。

#### 特徴

・1つのメインエージェント（チームリーダー）と複数のエージェント（チームメンバー）で構成される  
・各エージェントは役割（役職・専門領域）を持ち、分業してタスクを処理する  
・各エージェントは独自のコンテキストウィンドウを持ち、完全に独立している  
・各エージェントは互いに直接メッセージを送信し会話し合う  
・（注意）`/resume`、`/rewind`非対応  
・1セッション1チームのみ  
・チームリーダーの途中変更不可  
・（注意）CLAUDE.mdはチームメンバーにも読み込まれる  
・大規模開発・複雑な意思決定において特に効果を発揮する  
・まだ発展途上（実験的機能）であり、仕様や制限は頻繁に変化している

# なぜサブエージェントでは厳しくなるのか

それは、**サブエージェントは「1回の呼び出し前提」** だからです。（コンテキストを消費しないskillsのようなイメージ）  
サブエージェントは基本的に`入力 → (サブエージェントから)retrieval → 出力`で完結します。  
また、基本的に「stateless」のため、  
・何を検索したか  
・何が不足しているか  
・次に何を探すべきか  
などのリアルタイムの状態はメインエージェントに持たせる必要があり、コンテキスト負荷につながります。

これらを解決してくれるのがエージェントチームです。  
エージェントチームを使えば、一つのエージェントに管理者（Orchestrator）を設定することで以下のように  
状態管理、リトライ処理、分岐処理といった複雑なワークフローにも対応することができるようになります。

```
① 初回検索
↓
② 情報不足か判定
↓
③ 再検索（クエリ変更）
↓
④ 再評価
↓
⑤ 必要ならさらに取得
```

# こんなときに使いたい

（サブエージェントよりも）エージェントチームを使うべきケースをいくつかまとめてみました。

#### 1. 複雑な意思決定がある場合

理由：サブエージェントでは評価し合うことができないため  
複数案を比較して評価し合い、最適解を選ぶことができるようになります。

#### 2. ワークフローが状態遷移する場合

理由：リトライや分岐が発生した場合に、各ステップで状態を持つ必要があるため  
各ステップごとに担当のエージェントをアサインすることで、  
[要件定義 → 設計 → 実装 → テスト → 修正 → 再テスト]までを完遂させることがきます。

#### 3. 長時間・多段階タスク・並列処理したい場合

理由：サブエージェントではどうしてもメインエージェントにコンテキスト負荷が偏ってしまうため

#### 4. 評価指標を分離したい場合

理由：同一エージェント（またはメインエージェントに偏った場合）だと自己評価バイアスが出てしまうため  
→　エージェントチームは完全に独立したコンテキストで評価させ合うことができる

# 使用するうえでの注意点

まだまだ発展途上のエージェントチームですが、使用するうえで気にしておきたい注意点をいくつかまとめてみました。

## 1. コンテキスト共有による情報汚染・リーク

エージェントチームでは複数エージェントが同一タスク（Shared Task）に関与し、コンテキストを共有・引き継ぎます。  
このとき、あるエージェントの誤った推論や不要な情報が他エージェントに伝播し、「誤りの増幅（error propagation）」が発生することがあります。  
特に長いセッションでは、古い前提や不要なログが残り続けることで、意思決定の質が低下します。  
さらに、機密情報（APIキーや内部仕様）が他エージェントのプロンプトに混入し、**気づかないところでリークしてしまう**リスクもあります。

#### 対策

・明示的なコンテキスト削減を行う（要約・圧縮・フィルタリング）  
・機密情報のスコープを分離する  
・エージェントごとに「参照可能な情報」を制限する

## 2. ツール権限の過剰付与

エージェントチームでは各エージェントにツール（ファイル操作・シェル・GitHub・APIなど）が割り当てられます。  
これが過剰だと、以下のような問題が発生するリスクが高まります。

・誤ったファイルの削除や上書き  
・意図していないデプロイや外部通信  
・重大なセキュリティ侵害

#### 対策

・最小権限の原則に従った権限付与を意識する  
・読み取り専用エージェントと書き込みエージェントを分離する  
・本番環境への直接アクセス禁止（CI/CD経由に限定する、もしくは、手動承認プロセスをはさむ）

## 3. 非決定性と再現性の欠如

毎回同じエージェントチームを再利用したとしても、複数エージェントが並列（または順次）で処理する場合、以下のような要因で結果が変わることがあります。

・実行順序  
・コンテキストの微妙な差  
・LLMの確率的出力

これらの要因により、**同じ入力なのに異なる結果**が頻発します。  
これはCI/CDやレビュー工程で問題になる可能性があります。

#### 対策

・各ステップごとの成果物（artifact）を固定して管理する  
・エージェント間の入出力契約（schema）を定義する

## 4. 役割設計の不備（責務の曖昧さ）

エージェントチームの設計でよくある失敗は、各エージェントの役割が曖昧なことです。

例）  
「実装担当エージェント」と「レビュー担当エージェント」並列で同じ修正を行う  
→　誰が最終決定するか不明なため、無限ループ（修正→レビュー→再修正）に入ってしまったり、その結果、矛盾した変更が加えられるリスクがある

#### 対策

・明確に責務を分離する（例：計画担当 / 実装担当 / レビュー担当）  
・単一の意思決定エージェント（Orchestrator）を置き、すべての意思決定を1箇所に集約させることで、「誰が何をするか」はOrchestratorが決める（各エージェントに判断させない）  
→レビューの最大回数などを決めておくことも可能  
・フローを状態遷移として設計することで、「どこまで進んだか」「次に誰が動くか」を明確にする

## 5. コンテキスト肥大化とトークンコスト

複数エージェントが履歴を持つと、コンテキストが急激に膨張します。  
結果として、  
・レイテンシ増加  
・コスト増加  
・重要情報の欠落  
といった問題が発生してしまいます。

#### 対策

・ステップごとに要約・圧縮をおこなう  
・必要な情報だけ再取得させる（RAG: Retrieval-Augmented Generation）  
・不要なログは切り捨てる

## 6. 評価・監査の難しさ

単一エージェントと違い、どのエージェントがミスしたかが、どうしても特定しづらくなります。  
そのため、  
・デバッグが困難になりやすい  
・改善サイクルが停滞してしまう  
・セキュリティの不透明性  
といったリスクがどうしても発生しがちです。

#### 対策

・エージェントごとにログを分離する  
・入出力のトレースをおこなう（誰が何をしたか）  
・評価指標の分離（per-agent metrics）を行う  
　→エージェントごとに特定の評価基準を設け、数値で評価させることで、ボトルネックを特定する

# おわりに

エージェントチームは使いこなせるととても強力な機能ですが、その分「管理が難しい」「トークンの消費が大きい」といったデメリットもあります。  
そのため、サブエージェント機能でできる場合はそちらを優先することをおすすめします。
