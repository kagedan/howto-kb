## はじめに

生成 AI エージェントツール用の Deep Research スキルを作り、GitHub で公開しました。  
スキルの説明を兼ねた備忘録として記事を書いておくことにしました。

<https://github.com/h-daimatsu/deep-research>

このスキルは、「深く調べて」「ディープリサーチして」といったキーワードで起動します。  
起動するとエージェントがテーマに合わせてレポート構造を設計し、反復的なWeb調査を行い、構造化されたレポートを生成します。

動作確認は以下で行なっています。

モデル

* Claude Opus 4.7
* Claude Opus 4.8
* Claude Sonnet 5

Coding Agent

* Kiro CLI
* Kiro IDE
* Claude Code

## おことわり

このスキルは、私が最初にベースだけ作り、あとは AI と相談しながら形にしたものです。  
私自身はリサーチを専門にしているわけではないので、ここで書いた設計判断も、唯一の正解というより一つの作り方として読んでもらえればと思います。  
ちなみに、この記事も半分以上は AI に書かせています。たぶんそのほうが読みやすいので。

## このスキルの特徴

* **動的な構造設計**：テーマの調査計画に基づいて、章立てと分析要素、定量と定性の比率を毎回設計します
* **2段階構造設計**：調査前に仮構造を設計し、全調査の完了後に調査内容へ合わせて一度だけ改訂してから執筆します
* **取得多様性と源泉多様性の分離**：記事の本数ではなく、源泉まで辿った独立ソースの数で確からしさを数えます

先に構造を固定して調査結果をあてはめるのではなく、まず調査プランを立てたのちに Web 検索を行い、調べてわかったことに構造のほうを合わせ直します。

確からしさの数え方も同じ発想です。  
似た内容の記事が何本並んでも、源泉をたどって同じ一次情報に行き着くなら、独立したソースは一つと数えます。

## 生成されるレポートの例

実際にこのスキルで生成したサンプルレポートを、リポジトリに置いてあります。  
テーマの性質によって構造タイプが変わっているのが見て取れます。

どれも見出しの立て方が違います。  
この違いは、次に述べる構造設計の仕組みから生まれます。

## SKILL.md と 8 つの参照ファイル

スキルは、一つのエントリポイントと、役割ごとに分割した参照ドキュメントで構成しています。

```
deep-research-skills/
├── SKILL.md                        # 起動宣言、役割、思考ロジック、絶対ルール、参照マップ
└── references/
    ├── workflow.md                 # フェーズ1〜5の手順
    ├── structure-design-guide.md   # 章立て、接続、分析要素、定量と定性の比率の設計原則
    ├── common-skeleton.md          # エグゼクティブサマリー、結論、出典などの必須要素
    ├── citation-format.md          # 脚注記法 [^N] と確信度タグ
    ├── tone-rules.md               # 調査レポート文体の定義と、回避すべき7つの文の機能
    ├── conclusion-guide.md         # 結論セクションの5要素
    ├── output-calibration.md       # 展開深度の4層、打ち切りの禁止、複雑度への比例
    └── verification.md             # 最終突合の4検証
```

## 5 つのフェーズで動く

起動から成果物までの流れを、5フェーズで定義しています。

![](https://static.zenn.studio/user-upload/8ed6838832df-20260706.png)

成果物の置き場所として、調査開始時に次のフォルダを作ります。

```
{YYYYMMDD-HHmm}-deep-research-{テーマの短縮名}/
├── checkpoint.md      # 各ステップの調査結果を蓄積
├── structure.md       # 設計したレポート構造（仮設計から改訂までの履歴）
├── report.md          # 最終レポート
└── verification.md    # 最終突合の結果
```

`checkpoint.md` と `structure.md` を分けているのは意図的で、調査の記録と構造の記録を別に管理します。

### フェーズ1 調査計画と仮構造の設計

![](https://static.zenn.studio/user-upload/8cc577bd4fb0-20260706.png)

テーマを受け取っても、いきなり検索はしません。  
まずテーマの核心を特定し、複数の調査ステップに分解します。  
「現状の事実」「原因や背景」「影響と波及」「リスクや不確実性」「将来の展望」という5軸を意識しつつ、テーマに合わせて取捨選択します。

次に、その調査計画に基づいて仮のレポート構造を設計します。  
ここで選ぶのは、章と章をどうつなぐかという**論理パターン**です。  
章の見出しそのものではなく、章の並びと関係の型を、テーマに応じて選びます。

| 論理パターン | 適するテーマ |
| --- | --- |
| 因果連鎖 | 原因から結果、影響へと追うテーマ |
| 時系列 | 経緯や変遷を追うテーマ |
| 構造分解 | 複合的な対象を分解するテーマ |
| 比較軸 | 複数の対象を比較するテーマ |
| ステークホルダー別 | 影響が多方面に及ぶテーマ |
| スケール別 | ミクロからマクロへ展開するテーマ |

論理パターンが決めるのは、あくまで章のつなぎ方です。  
個々の見出しは、設計原則の一つ「各章は独立した問いに答える」に沿って、テーマごとに作ります。  
そのため、「関連情報」「その他の論点」のような、中身を特定しない見出しは弾かれます。  
問いを立てられない章は、統合か削除の対象になります。  
決め打ちのテンプレートと違い、見出しの名前も数も順序も、テーマごとに変わります。

計画と仮構造ができたら、両方をまとめてユーザーに提示し、承認を求めます。  
方向性を合意してから調査に入るので、数十分かけた末に的外れなレポートが出てくる事故を避けられます。

調査計画と仮構造は `structure.md` に保存します。

### フェーズ2 反復的調査

承認後、各調査ステップで「一次情報の探索、二次情報の補完、分析、飽和判定、チェックポイント記録」のループを回します。

分析では、収集した情報に対して3つの問いを立てます。

1. **なぜ**（因果関係）：この事実はなぜ起きているのか
2. **だから何**（波及効果）：他の領域にどう影響するか
3. **本当に**（検証）：他のソースでも裏付けられるか

この3問から次の検索キーワードを生成します。  
チェックリストを機械的に消化するのではなく、前ステップの発見が次ステップの方向を変えていきます。

調査の確からしさをめぐって、少し工夫した仕組みが入っています。  
**取得多様性と源泉多様性の分離**です。

複数の検索エンジンを併用しても、N本の記事が一致していても、それらの源泉を辿ると同じ一本の一次情報に行き着くなら、独立したソースは一つしかありません。  
記事本数の多さを確からしさと取り違えると、偽のコンセンサスに乗せられます。  
そこで、結論を支える柱の主張については、源泉まで畳んで独立ソース数を数え直します。

さらに、主張のタイプによって確信度の基準を切り替えます。

| 主張のタイプ | 例 | 確からしさの担保 | 一次照合 |
| --- | --- | --- | --- |
| 事実抽出系 | 財務数値、議決権比率、契約文言 | 原本そのもの | 必須 |
| 集合知系 | アナリスト予想、市場規模推計 | 集計それ自体が情報 | 不要 |
| 解釈系 | 「需給が支える」などの評価 | 複数論者の多様性 | 不要 |

事実抽出系は、原本に当たれなければ、二次記事が何本一致しても確信度は上がりません。  
一方、アナリストのコンセンサスや市場規模の推計に「一次原本がないから要確認」を機械適用するのは過剰な留保で、情報の価値を損ないます。  
確からしさを厳しく見る規則と、過剰な留保を避ける規則を、どちらも明文化しています。

各ステップの完了時には、`checkpoint.md` に必ず追記します。  
主要な発見、定量データ、因果の分析、波及効果、矛盾点、柱の主張の源泉、そして仮構造への示唆までを記録します。

### フェーズ3 構造の改訂

![](https://static.zenn.studio/user-upload/2dc2bf6a9e82-20260706.png)

このフェーズは、このスキルの特徴的な工程です。  
全調査が終わったら、まず `checkpoint.md` を全文読み直します。  
そのうえで、各ステップに記録した仮構造への示唆を集約し、仮構造 `structure.md` を改訂します。

改訂で検討するのは、章の追加、削除、分割、順序の変更、分析要素の変更、定量と定性の比率の調整、中心主題の微調整です。  
「調査してみたら予想外の重要な論点が出てきた」「この章は情報が薄いので統合しよう」といった判断を、ここで一度だけ行います。

改訂の理由も `structure.md` に必ず残します。

固定テンプレートなら不要な工程ですが、調べてわかったことに構造を合わせ直すという設計を実際に成立させているのは、このフェーズです。

### フェーズ4 レポート作成

改訂後の構造に従って `report.md` を書きます。  
動的に設計するのは本体セクションだけで、次の**共通骨格**はどのテーマでも必ず含めます。

```
**作成日**: YYYY-MM-DD JST
## エグゼクティブサマリー
  （本体セクション群 ← 動的に設計）
## 情報の矛盾・不確実な点
## 追加調査が必要な領域
## 結論
## 出典
```

二つの仕組みがあります。

一つ目は、**展開深度**による厚みの制御です（`output-calibration.md`）。  
制御の対象は分量ではなく展開の深さで、柱の発見は「事実、因果、波及、含意（だから何）」の4層まで展開します。  
事実の列挙や、表を置いたままの放置では止めません。  
文字数のノルマは設けず、深さの結果として分量が生まれる立て付けにしています。

二つ目は、**文体統制**です（`tone-rules.md`）。  
目指すのは、調査会社、投資銀行、コンサルの報告書の文体です。  
単語のブラックリストではなく、回避すべき7つの「文の機能」で定義しているのが特徴になります。

1. 読者への問いかけや反語
2. 著者の自己言及やメタ言及（「本レポートが示すのは」など）
3. 文学的な比喩による説明
4. 判断の完全な保留
5. 修辞的な繰り返しや対句
6. 誇張や感情的な強調
7. ベンダー的な装飾語の地の文への流用

加えて、LLMが量産しがちな空句（「重要なのは〜である」「多角的に」「掘り下げる」）や冗長も排除します。  
引用はマークダウンの脚注記法 `[^N]` で統一し、バンクーバー方式で通し番号を振ります。  
柱の主張の脚注には、`〔一次照合〕`、`〔二次・独立複数〕`、`〔要確認〕` の確信度タグを付けます。

### フェーズ5 最終突合

レポートの完成後、4つの検証を実施します。

1. 文体検証：7機能の逸脱を自己チェックと grep で検出し、地の文の逸脱がゼロになるまで修正します
2. 引用書式検証：脚注記法の禁止形、素のURL、孤立した参照、欠番、重複を機械スキャンします
3. チェックポイント突合と結論検証：`checkpoint.md` の発見がレポートに反映されているか、結論の5要素が揃っているか、`structure.md` の改訂が実現されているかを確認します
4. 源泉と確信度の検証：柱の主張に確信度タグが付いているか、偽のコンセンサスがないか、要確認の柱が本文でヘッジ語になっているかを確認します

検証の結果は `verification.md` に記録します。  
書いて終わりにせず、自己採点と修正までを工程に組み込んでいます。

## 使い方と注意点

起動のトリガーは、日本語なら「深く調べて」「徹底調査」「ディープリサーチ」「包括的に調査」、英語なら `deep research` や `in-depth analysis`、`deep dive` などです。  
入力はこのように渡します。

```
生成AIの電力需要がデータセンター立地と電力網に与える影響を、シナリオ別に徹底調査して
```

実行にあたっては、次の点に注意してください。

* 1回の調査に10〜30分程度かかります
* トークンの消費量が大きくなります
* 標準のWeb検索を使いますが、`tavily` や `brave` がある環境ではそちらも併用します
* フェーズ4以降は途中でコマンド承認を求められることが多いので、専用ワークスペースを用意し、自動承認を有効にしておくと快適に動きます

## おわりに

このスキルの要点は三つあります。  
テーマごとに設計し直す動的な構造設計、調査後に一度だけ構造を改訂する2段階構造設計、そして確からしさを源泉まで畳んで数える取得多様性と源泉多様性の分離です。  
あわせて、7つの文の機能で定義した文体統制と、4段階の最終突合で、出力の質を工程として担保しています。
