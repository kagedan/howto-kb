この記事は、Claude Code を日常的に使っているものの「うまく活用できている実感がない」と感じているエンジニア向けです。

特に、「CLAUDE.md は作ってみたけど、サブエージェント・スキル・フック・MCP などはまだ使いこなせていない」という方を対象にしています。

書籍[『実践 Claude Code 入門』](https://gihyo.jp/book/2026/978-4-297-15354-0)（技術評論社）を読んで得た気づきをベースにしているので、ぜひ本書も読んでみてください。

読者には以下の状態になってもらうことを目指しています。

* サブエージェント、スキル、フック、MCP がそれぞれ何のために存在するのかわかった
* 「何となく使っている」から「理解して使っている」状態へ
* 自分の環境で Claude Code をどう使うかのアイデアが湧いてくる

## Claude Code を使っていて感じていた課題

弊社では 2025 年 7 月から Claude Code を全エンジニア（と一部非エンジニア）に提供しています。

ありがたいことに比較的早い段階から Claude Code を使い倒すことができて開発の生産性はかなり上がりました。

ただ、導入から時間が経つにつれ、**Claude Code を何となく使ってしまっている**という漠然としたモヤモヤを感じるようになりました。

投げれば何か返ってくる。でも思い通りにならないことも多く、なぜそうなるのかもよくわからない。

そんな中、会社の書籍購入制度で[実践 Claude Code 入門](https://gihyo.jp/book/2026/978-4-297-15354-0)が入荷したので、この感覚の正体を突き止めるべく本書を読んでみました。

本書は Claude Code の各機能の紹介にとどまらず、**「AI をどう開発プロセスに組み込むか」という設計の視点**で書かれています。「機能の説明書」ではなく「活用のための思考法」を扱った本、という印象です。

## コーディングエージェントの 3 つの課題とその解決策

本書を読んで、「何となく使っている」状態の背景には**コーディングエージェントが抱える構造的な課題**があり、それぞれの課題の原因とその解決策を理解すれば、コーディングエージェントを使いこなせるという希望が見えました。

コーディングエージェントの課題は大きく 3 つに整理できます。

| 課題 | 原因 | 解決の方向性 |
| --- | --- | --- |
| **過去の指示を忘れてしまう** | コンテキストウィンドウの容量が有限 | コンテキストを分離する |
| **意図通りに動いてくれない** | 暗黙的な期待が明示されていない | 細かく指示する |
| **外部システムと連携できない** | Claude Code 単体だとできることに限界がある | 外部システムとつなげる |

---

原因に対する解決策が当たり前すぎますが、下記でもう少し詳しく見てみましょう。

### 「過去の指示を忘れてしまう」の解決策：コンテキストを分離する

コンテキストウィンドウの容量は有限なので、何も考えずに指示を出し続けると過去の指示を忘れてしまいます。

Claude Code では一定のしきい値（デフォルトでは 95%の容量）を超えると自動圧縮が実行されますが、重要な情報が抜け落ちてしまう可能性があります。

また、コンテキストが大きくなると LLM の性能そのものが落ちてしまうという問題もあります。

コンテキストが膨らんでしまうのであれば分割してしまいましょう。具体的には以下のような方法があります。

#### 解決策 1: 構造化ノート

構造化ノートとは、重要な情報をコンテキストウィンドウの外部に分離する戦略です。

やり方は色々ありますが、本書では以下の方法が紹介されていました。

1. `.steering/`という名前のディレクトリを作成する
2. 作業単位のディレクトリをこのディレクトリ内に作成する
3. `CLAUDE.md`にルールを定義し、具体的な作業内容、作業進捗、重要な決定事項などは全て作業単位のディレクトリ内で管理させる

```
# 本書で紹介されていたディレクトリ構造例

.steering/[YYYYMMDD]-[タスク名]
├── requirements.md  # 今回の作業の要求内容
├── design.md        # 設計
├── tasklist.md      # 実装タスク
├── blockers.md      # ブロッカー
└── decisions.md     # 重要な決定事項
```

作業単位でドキュメントを作成し、それを更新しながら作業を進めるように Claude Code に指示することで、コンテキストウィンドウを大幅に節約できます。

#### 解決策 2: サブエージェント

サブエージェントとは、親エージェント（メインの Claude Code）から独立して専門的な作業を行うエージェントです。マークダウンファイルで定義することで作成できます。

具体的な作成方法は調べてもらうとして、重要なことはサブエージェントはコンテキスト分離のために作成するということです。

成果物のレビュー、影響範囲分析、テストの実行、ログの分析など、大量の情報を読む作業を実施させることで効果を発揮します。

また制約事項として、エージェント間でコンテキストは共有されません。  
したがって、作業ログをファイルや GitHub Issue に書き出して受け渡すなど、レポートを介してエージェント同士にコミュニケーションを取らせる工夫をする必要があります。

### 「意図通りに動いてくれない」の解決策：細かく指示する

Claude Code に指示していると、「思ってたのと違う」となることがあります。

それは Claude Code が悪いのではなく、こちらの「普通そうするだろう」が指示に明確に落とし込まれていないことが問題なのです。

また、ドキュメント間で指示がコンフリクトしており、Claude Code がどちらを守れば良いのかわからない状態になってしまっている可能性もあります。

#### 解決策 1: スラッシュコマンド

※本書ではスラッシュコマンドとして紹介されていますが、現在は後述のスキルに統合されています。ただし、スラッシュコマンド自体は引き続き使うことができます。

スラッシュコマンドとは、よく使うプロンプトをマークダウンファイルに記述し、コマンドとしていつでも呼び出せるようにする機能です。

**「何を」「どの順番で実行するか」という手順**を明文化することで、指示に含まれる曖昧な部分を減らして Claude Code を意図通りに動かすことができます。

#### 解決策 2: スキル

スキルとは、**「どのツールを」「どのように使うか」という方針**を Claude Code に教える機能です。

方針を定義するという点では`CLAUDE.md`と似ていますが、`CLAUDE.md`は毎回コンテキストウィンドウに読み込まれるのに対し、スキルは必要に応じて読み込まれます。

したがって、全体的な開発方針は`CLAUDE.md`に記述し、「API 仕様書をどう作成するか」「セキュリティレビューをどう実施するか」といったより詳細な方針はスキル化してコンテキストを節約することをオススメします。

スキルを作成する際の観点として、本書では 3 つの切り口が紹介されています。参考にしてみてください。

1. 成果物単位：機能設計書、テスト計画書、運用手順書などの単位で分ける
2. タスク単位：フロントエンドの実装、テストの実行、セキュリティスキャンの実行などの単位で分ける
3. ドメイン単位：案件特有の規制要件、組織のブランドガイドライン、プロダクト固有のビジネスルールなどの単位で分ける

また、スキルの強みとして、単一のファイルだけでなくディレクトリ構造で複雑な情報を管理できるという点があります。

下記のように補助ドキュメント、スクリプト、テンプレートなどを含めることができます。

```
# 本書で紹介されていたディレクトリ構造例（一部修正）

.claude/skills/
├── frontend-patterns/
|   ├── SKILL.md              # スキル本体（必須）
|   ├── best-practices.md     # ベストプラクティス詳細（任意）
|   └── examples              # 実装例（任意）
|       ├── good-pattern.ts
|       └── bad-pattern.ts
├── backend-patterns/
...
```

#### 解決策 3: フック

フックとは、Claude Code のライフサイクルの特定のタイミングで、コマンドやスクリプトを実行する仕組みです。

スラッシュコマンドやスキルはプロンプトベースなので、どう動くかは LLM の推論に左右されてしまいます。

したがって、**絶対に守ってほしいルール**はフックを使って制御しましょう。

ツール実行前後、応答終了時など、Claude Code ではフックがトリガーされるタイミングが用意されています。どのタイミングで何を実行したいかを考えて、うまく設計しましょう。

例としては以下のようなものが挙げられます。

* ツール実行前に実行結果をログファイルに記録するよう指示する
* ツール実行後にファイルをフォーマットさせる
* Claude Code の作業が終わったらデスクトップに通知させる

### 「外部システムと連携できない」の解決策：外部システムとつなげる

実際の開発はエディタだけで完結するものではなく、GitHub でプルリクを作成したり、Figma でデザインを参照したり、ブラウザで動作確認をしたり、外部システムと連携しながら進めることが多いです。

したがって、Claude Code をフル活用するにはこのような外部システムとうまく連携する仕組みを作ることも必要になってきます。

#### 解決策 1: MCP（Model Context Protocol）

MCP とは、AI アプリケーションと外部システムの接続方法を標準化したプロトコルです。

Claude Code が扱えるツールを拡張し、Claude Code を通していろいろな操作ができるようにしてくれます。

できることは多岐にわたるので、何ができるのか、どうやって設定するのかは調べてみてください。

ただし、セキュリティリスクがあるので、使って問題ないかどうかは MCP サーバごとに慎重に検討してください。

弊社でもいくつかの MCP サーバは使用可能になっており、便利さを実感しています。

「Claude Code の能力を拡張する」という意味ではスキルと似ていますが、スキルはやり方（例：プルリクのタイトルにチケット番号を含めること）を定義するのに対し、MCP は操作自体（GitHub のプルリクを作成できる）を追加します。

#### 解決策 2: CLI ツール

とはいえ、全てのシステムが MCP サーバを提供しているわけではありません。

また、組織内の独自システムと連携したい場合もあります。

そんな時に便利なのが CLI ツールです。

GitHub CLI のような既存の CLI ツールや、自作のカスタム CLI ツールを活用することで、外部システムとの連携を柔軟にカスタマイズできます。

## Claude Code 周りのあれこれ整理

これまで出てきた Claude Code の機能を整理しておきましょう。一覧化すると以下のようになります。

各機能を使う目的をしっかり理解して、Claude Code を「理解して使っている」状態になりましょう！

| 機能 | 使う目的 | 向いている用途 |
| --- | --- | --- |
| **サブエージェント** | 親エージェントとコンテキストを分離する | 調査、レビュー、分析など、大量ファイルの読み取りや専門的なチェックを必要とする作業の実行者 |
| **スキル** | ツールの使い方や 方針を Claude Code に教えて、齟齬を減らす | コードレビューの観点、命名規則など「方針」を定義し、必要に応じて Claude Code に自動で読み込ませる |
| **フック** | 絶対に守ってほしいことをルール化して守らせる | ツール実行後のフォーマット、Claude Code の作業完了通知など、絶対に守ってほしいルールを定義する |
| **MCP** | Claude Code を外部システムと連携させて機能を拡張する | Figma でデザインを参照したり、ブラウザで動作確認をしたり、Claude Code 経由で様々な外部システムを利用する |
