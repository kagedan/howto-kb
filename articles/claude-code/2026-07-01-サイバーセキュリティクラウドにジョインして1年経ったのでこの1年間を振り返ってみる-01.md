こんにちは、CSC の [CloudFastener](https://cloud-fastener.com/) というプロダクトで TAM のポジションで働いている平木です！

2025年7月にサイバーセキュリティクラウド（CSC）に入社してから、今日でちょうど1年が経ちました。

振り返ってみると、個人のキャリア・AI の進化・サイバーセキュリティの情勢、いずれの面でも「あれからまだ1年しか経っていないのか」と時の流れの速さを実感させられる、密度の濃い日々でした。

この記事では、この1年を3つのテーマで振り返ります。

## サイバーセキュリティの一年

### 「あれからもう1年か」という感覚

あれから約1年、と実感させられる出来事が続いています。

入社して2ヶ月が経った2025年9月下旬、日本を代表する大手飲料メーカーがランサムウェア攻撃を受けました。  
VPN 機器を経由して侵入した攻撃者は10日間のうちに管理者権限を奪取し、生産・物流システムを広範囲にわたって停止させました。

ビールや清涼飲料の供給が全国規模で影響を受け、190万件超の個人情報漏洩とともに国内外で大きく報じられました。  
ロシア系ランサムウェアグループの関与が確認された、日本の大企業のサプライチェーンが文字通り止まった事件でした。

同年10月にも大手通販事業者がランサムウェア被害を受け、74万件の個人情報が漏洩しました。  
特に注目すべきは、初期侵入が被害発覚の4ヶ月以上前だったことやEDRなどを無効化した上で不正アクセスを行った点です。業務委託先のアカウントに多要素認証（MFA）が設定されていなかったことが根本原因とされており、「委託先のさらにその先までセキュリティの視野に入れられているか」という問いを日本の組織全体に突きつけました。

2026年6月には、国内大手通信事業者が ISP 向けメールインフラへの不正アクセスを公表しました。サードパーティ製ソフトウェアの脆弱性が悪用され、ニフティ・ビッグローブ・JCOM など提供先の ISP を通じた認証情報が最大1,422万件流出した可能性があります。解約済みを含む休眠アカウントの認証情報まで攻撃者の手に渡りうるという点で、影響は件数以上に深刻でした。

### オープンソースエコシステムへの攻撃が日常になった

攻撃者の標的が、コードそのものから「コードが依存するもの」全体へと広がった1年でした。  
使っているライブラリ、ビルドツール、CI/CD の配布経路、そしてライブラリ自体の脆弱性などあらゆる依存関係が攻撃対象になっています。

**2025年12月3日 — React2Shell（CVE-2025-55182）**

React Server Components における認証不要の RCE 脆弱性「React2Shell（CVE-2025-55182）」が公開されました。CVSS スコアは最大値の 10.0 で、HTTP リクエスト1つで任意コードを実行できるという深刻度です。

公開のわずか2日後、12月5日にはすでに暗号資産マイナー（XMRIG）の設置が確認され、その後も複数の攻撃者グループが並行して悪用を開始しました。  
Google の脅威インテリジェンスチームは、中国関連の APT グループ（UNC5454 / Earth Lamia、Jackpot Panda など）による SNOWLIGHT ダウンローダや HISONIC バックドアの展開を確認しています。

また JPCERT/CC が公表した事例では、1台のサーバーに複数の攻撃者が同時侵入するという異例の状況が発生しました。  
12月5〜7日の3日間で、コインマイナーの設置からバックドアの仕込み、そして最終的なサイト改ざんまでが連続して実行されています。「パッチ公開の翌々日には攻撃が始まっている」という事実が、対応の時間的余裕がいかに失われているかを示していました。

<https://cloud.google.com/blog/ja/topics/threat-intelligence/threat-actors-exploit-react2shell-cve-2025-55182>

<https://blogs.jpcert.or.jp/ja/2026/02/react2shell.html>

**2026年3月19日 — Trivy**

今年の3月からはサプライチェーン攻撃が活発になりました。

コンテナセキュリティの定番スキャナー「Trivy」が多段階のサプライチェーン攻撃を受けたのもまだ記憶に新しいと思います。  
攻撃者は事前に GitHub Actions ワークフローの設定ミスを突いて Personal Access Token を盗取し、Trivy のコアバイナリ・GitHub Action・Docker Hub イメージを侵害しました。

脆弱性を「見つける」側のセキュリティツールが、AWS・GCP・Azure の認証情報を盗み取る「武器」に変わった瞬間でした。  
守るためのツールが攻撃の入口になる。セキュリティを取り巻く「信頼の前提」が、気づかないうちに塗り替えられていると感じた出来事でした。

<https://thehackernews.com/2026/03/trivy-security-scanner-github-actions.html>

**2026年3月31日 — axios**

Trivy の対応がまだ続く中、 JavaScript の HTTP クライアントとして週間 1億ダウンロードを誇る「axios」が侵害されました。  
メンテナーの npm アカウントが侵害され、悪意あるバージョンが39分以内に連続公開されました。  
Microsoft Threat Intelligence は北朝鮮の国家支援グループ「Sapphire Sleet」による攻撃と特定しています。

削除まで約3時間でしたが、その間に `npm install` を実行したプロジェクトにはバックドアが仕込まれ、174,000 以上の依存パッケージへの波及が懸念されました。週間1億ダウンロードという数字は、もはや「安全の証明」にはならない。依存関係への信頼という前提そのものが揺らいだと感じた出来事でした。

<https://unit42.paloaltonetworks.com/ja/axios-supply-chain-attack/>

**2026年5月18日 — Megalodon キャンペーン**

セキュリティ企業 SafeDep が報告した「Megalodon」キャンペーンは、わずか6時間の間に 5,561 件の GitHub リポジトリへ 5,718 件の悪意あるコミットが自動的にプッシュされるという、かつてない規模のサプライチェーン攻撃でした。

攻撃者は GitHub Actions のワークフローを悪用して悪意ある bash スクリプトを注入。対象は AWS・Google Cloud・Azure の認証情報、SSH 秘密鍵、Docker 認証設定、Kubernetes 構成ファイルなど 30 種類以上のシークレット情報に及びました。オープンソースプロジェクト「Tiledesk」では複数のリポジトリにバックドアが仕込まれ、npm パッケージを通じてユーザーへ配布される事態にもなりました。

Hudson Rock の調査では、影響を受けたリポジトリに紐づくユーザー名の 33% が、インフォスティーラーに感染したコンピューターと直接一致していたことが判明。攻撃の起点は侵害された開発者の認証情報であり、使い捨ての GitHub アカウントを使って組織的に展開されたものでした。

<https://pc.watch.impress.co.jp/docs/news/2111821.html>

**2026年6月 — Mini Shai-Hulud / Mastra**

6月には、 Red Hat エンジニアの GitHub アカウントが侵害され、NPM の Trusted Publishing の仕組みを突破して AWS や GCP の認証情報を窃取する「Mini Shai-Hulud」ワームが仕込まれたパッケージが公開された出来事もありました。

6月17日には AI エージェント開発フレームワーク「Mastra」の npm パッケージが侵害され、AI 関連ライブラリさえも例外ではないことが明らかになりました。

<https://blog.flatt.tech/entry/mastra_compromise>

2026年の3月から6月というわずか4ヶ月の間だけでも、npm・PyPI・GitHub Actions・Docker Hub を含むあらゆる依存関係が恒常的に攻撃対象となりました。

サプライチェーンへの侵害も、ライブラリの脆弱性悪用も、根底にある問いは同じです。「自分たちが信頼しているものを、本当に信頼していいのか」——その前提を問い続けることが、これからのセキュリティの出発点になっていると感じます。脆弱性管理だけでなく依存関係の信頼性をどう担保するか、使用しているライブラリの配布経路に何が起きているかを継続的に監視することが、これからの重要な問いになっています。

## AI が変えた一年

### AI の使い方が変わった：コンテキストからループへ

「プロンプトエンジニアリング」という言葉が一般化して久しいですが、2025年6月25日、AI 研究者の Andrej Karpathy が X（旧 Twitter）でこんな言葉を投稿しました。

「プロンプトエンジニアリングという言葉は本質を見誤る。正しくはコンテキストエンジニアリングだ。コンテキストウィンドウ全体を最適な情報で埋める、繊細で複雑な技術を指す」

<https://x.com/karpathy/status/1937902205765607626>

今日でその発言から約1年が経ちます。  
これほど短期間に概念が更新され続けるのが、今の AI 分野の速度感です。  
Simon Willison も翌日に「この用語は定着するだろう」と同意見を表明し、「プロンプト」から「コンテキスト」へという認識の転換が広く共有されていきました。

<https://simonwillison.net/2025/jun/27/context-engineering/>

しかしそこから半年も経たないうちに、また次の概念が登場します。  
Anthropic の Claude Code 開発チーフである Boris Cherny は「Claude にプロンプトを送るべきではない。プロンプトを送るシステムを自分で構築すべき」と語り、エージェントが「観察 → 計画 → 実行 → 省察」を自律的に繰り返すループ設計の重要性を説きました。

<https://thenewstack.io/loop-engineering/>

「どう聞くか」から「どんな環境を設計するか」への問いの転換です。  
HashiCorp の Mitchell Hashimoto も「ハーネスエンジニアリング」として同様の概念を提唱し、フィードバックループ・終了条件・トークン予算の設計が新しいエンジニアのスキルとして浮上しました。

プロンプトエンジニアリング → コンテキストエンジニアリング → ハーネスエンジニアリング → ループエンジニアリング。  
この変遷がわずか1〜2年の間に起きたことが、今の AI 分野の加速度を象徴しています。  
「覚えた言葉が半年で古くなる。それでも追いかけることをやめたくない」という感覚がこの1年ずっと続いています。

自分自身もこの1年、Claude Code と MCP を組み合わせてワークフローを組む機会が増え、「AI を使う」から「AI と協働する」への変化を肌で感じてきました。

### 群雄割拠のモデル競争

パラダイムの変遷と並走するように、モデル性能の競争もこの1年で激烈になりました。

まず 2025年7月16日、Amazon Bedrock AgentCore がプレビュー公開されました。セッション分離・8時間連続実行・MCP 統合を備えたエンタープライズ向け Agent 基盤で、AWS に携わる者として「ついにここまで来たか」と感じた瞬間でした。

<https://aws.amazon.com/about-aws/whats-new/2025/07/amazon-bedrock-agentcore-preview/>

2025年8月には OpenAI の GPT-5 が登場し、コーディング・数学・マルチモーダルを含む複数のベンチマークで当時の最高水準を更新。同月 Anthropic も Claude Opus 4.1 をリリースしました。その後、Claude Sonnet 4.5（9月）・Claude Opus 4.5（11月）と更新が続き、Google も 11月に Gemini 3 Pro（Deep Think 機能搭載）、12月に Gemini 3 Flash を展開しました。

年が明けると各社の更新ペースはさらに加速します。  
Anthropic は Claude Opus 4.6 / Sonnet 4.6（2026年2月）・Opus 4.7（4月）・Opus 4.8（5月末）とほぼ月次ペースでモデルを更新。xAI は 4月末に Grok 4.3 を「現時点で最高の知能」と銘打ってリリース。Google は 5月に Gemini 3.5 Flash で推論コストの大幅低下を実現するなど、各社が同時多発的にモデルを投入し続ける状況が続きました。

そしてこの1年で最も衝撃を受けたのが、2026年4月7日に Anthropic が発表した Claude Mythos です。  
6月9日には Claude Fable 5 が登場しすぐに利用停止となりましたが高度な AI モデルが一般で使用できる未来が近づいていることにより一層震撼させました。  
フロンティアレベルの AI として発表された Mythos をはじめとした高度なモデルは、ゼロデイ脆弱性の自動発見やエクスプロイトコード生成の能力を持ち、セキュリティリスクの高さから限定公開となっています。

<https://www.anthropic.com/claude/mythos>

AI の進化の恩恵を日々受けながら、同じ技術が攻撃者の武器にもなる。セキュリティに携わる者として、技術進化の表と裏を同時に突きつけられた瞬間でした。防御側も AI を使いこなさなければ、もはや追いつかない時代になっています。

## キャリアを振り返って

ここからは個人的な話になりますが、この1年はキャリアの面でも大きな転換点となりました。

### プレイヤーからプレイングマネージャーへ

入社時には、TAM（Technical Account Manager）のロールとしてジョインしましたが、入社から半年でTAMとしてプレイングをしながらマネージャーとしての役割も担う立場へと変わりました。  
自分が技術を磨きながら顧客へ直接価値を届ける「プレイヤー」と合わせて、メンバーが成果を出せる環境を整える「マネージャー」への転換です。

最も大きく変わったのはマインドセットです。これまでは「自分がどう動くか」を考えていれば良かったのですが、マネージャーになってからは「チームがどう動けるか」を先に考えるようになりました。

メンバーの強みを活かしながら、チームとしての指針を作り、その実現に向けた環境を整える。それはプレイヤーとしての成果とは異なる難しさと面白さを持つ仕事です。

### プレイヤーとマネージャーのバランス

マネージャーになってから最も悩んだのが「技術力をどう維持・向上させるか」です。特にセキュリティという専門領域は、日々新しい脅威・ツール・規制が登場するため、インプットを止めると一瞬で陳腐化します。

自分が実践しているのは、手を動かすことを諦めないことです。Claude Code や MCP を使った自動化・ブログ執筆・イベントへの登壇を通じて、「知っている」だけでなく「体験している」状態を保つようにしています。マネージャーが技術の最前線にいることは、メンバーへの説得力にも、採用・育成の方針設定にも直結すると感じます。

プレイングマネージャーとしての葛藤は今も続いています。それでも「プレイヤーとしての自分」と「マネージャーとしての自分」の両方に投資し続けることが、この仕事の醍醐味だと思い始めています。

### AWS Top Engineers 選出

この1年のもう一つの大きなトピックが、AWS Top Engineers への選出です。AWS Top Engineers は、AWS コミュニティへの技術的な貢献を AWS Japan が認定するプログラムで、ブログ執筆・登壇・コミュニティ活動などが評価されます。

<https://aws.amazon.com/jp/blogs/psa/2026-japan-aws-top-engineers/>

個人の実績として認められることは素直に嬉しいのですが、それと同時に「チームや会社への信頼にも繋がるものだ」と実感しました。自分の発信が積み重なり、外部からの評価として返ってくる。個人としての研鑽が会社の信頼資産にもなるという経験は、マネージャーとしてメンバーの研鑽を支援する動機にもなっています。

## おわりに

入社1年を振り返ると、「技術・人・組織」の3つが同時に動き続けた1年でした。マネージャーとしてメンバーと向き合いながら、AI とサイバーセキュリティの激流に乗り続けるのは、決して楽ではありませんでした。  
それでも、CloudFastener というプロダクトを通じて顧客のセキュリティに貢献できる環境で、優秀なメンバーや仲間と一緒に走れたことは、大きな財産です。

次の1年は、チームをさらにスケールさせながら、AI 活用をプロダクトと自分自身の両面で深めていきたいと思っています。AWS Top Engineers の選出も、次のステップへの背中を押してくれる経験になりました。これからも情報発信を続けていきます。

2027年7月には世の中がどうなっているのか非常に楽しみです。
