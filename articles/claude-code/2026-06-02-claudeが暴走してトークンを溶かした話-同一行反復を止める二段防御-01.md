## はじめに

Claude Code をはじめとする AI エージェントを有償プランで使っていると、誰もが気にするのが **トークンの消費量** です。プランごとに使用量の上限があり、節約しながら計画的に使っている方も多いはずです。

そんな中で、私はちょっと肝を冷やす事象に遭遇しました。エージェントが **ツールを呼ばず、同じ短い行を延々と出力し続ける** 暴走状態に陥ったのです。しかも自力では止まりません。手動で中断するまで、同じ行が数十回反復し続けました。

これは単に「気持ち悪い挙動」では済みません。**出力が続くということは、その間ずっとトークンが消費されている** ということです。意味のある仕事を一切していないのに、トークンだけが無言で溶けていく。トークンを節約しながら使っているユーザーにとっては、地味に効いてくる死活問題です。

この記事では、(1) この「同一行反復暴走」とは何か、(2) 素朴な対策フレーミングに潜む罠、(3) 自律規律と Stop hook を組み合わせた **二段防御** の実装、を実体験ベースで共有します。

一文で先にまとめると、

> **エージェントが同じ行を吐き続ける「反復暴走」は、止めないとトークンが無言で溶けていく。コードのループにガードを入れても止まらず、AI が自分で止まる「自律規律」と、ハーネス側で機械的に止める「Stop hook」の二段で守るのが現実解。**

そして最初に正直に書いておきます。**この対策はつい先日試し始めたばかりで、効果検証はこれからです。** それでも、トークンが溶ける事象は多くのユーザーに関わりうると考え、検証を待たずに先に共有することにしました。同じ事象に心当たりのある方、もっと良い止め方をご存じの方からのフィードバックを広く求めたい、というのがこの記事のもう一つの目的です。

想定読者は、Claude Code / Cursor / Cline などの AI エージェントを有償プランで日常的に使っている開発者です。実装例は Claude Code の Stop hook ですが、「自己制御に依存しない外形的なストッパーを用意する」という設計の考え方自体は、ツールを問わず転用できます。

## 1. 何が起きたか — トークンが溶ける「同一行反復暴走」

事象はあるブラウザ操作の最中に起きました。ボタンを押したあとの画面遷移を待っているようなタイミングで、エージェントが応答の中に `call:` という前置きのテキストだけを生成し、**実際のツール呼び出しを伴わないまま同じ行を反復** し始めたのです。

```
call:
call:
call:
call:
（以下、同じ行が数十回続く）
```

特徴を整理すると、次のようになります。

* ツールを呼んでいるわけではなく、**前置きの文字列だけ** を吐き続けている
* 自力では止まらず、**手動で中断するまで** 続いた
* しかも、同一セッション内で **3 回連続** で発生した

ここで強調したいのは、これが **コードの `for` / `while` ループではない** という点です。プログラムの繰り返し処理が暴走しているのではなく、**モデル自身の出力生成（トークン列）が同じパターンに嵌まって抜け出せなくなっている**。これが事象の正体であり、後述する対策の難しさの根っこでもあります。

そして冒頭で書いたとおり、**反復している間ずっとトークンは消費され続けます。** 数十行の `call:` は、見た目こそ無害な短い行ですが、出力トークンとして確実にカウントされます。中断が遅れれば遅れるほど溶ける量は増えます。3 回連続で起きれば、その分だけ無駄が積み上がります。

幸い、今回は私が画面を監視しながら AI に作業させていたため、暴走に気づいて **数秒で処理を止める** ことができました。被害は最小限で済んだのです。

ですが、ここで背筋が寒くなりました。**もしこれを就寝前に自走モードで走らせていたら、どうなっていたか。** 誰も見ていない数時間のあいだ、エージェントが `call:` を吐き続け、トークンだけが延々と溶けていく。気づくのは翌朝、上限に張り付いた使用量を見たとき — まさに最悪の朝を迎えていたはずです。自走・夜間実行が当たり前になりつつある今、**「見ていないあいだに溶ける」リスク** こそ、この事象の本当の怖さだと感じました。

> 学び: 同一行反復暴走は「コードのループ」ではなく「モデルの出力生成の反復」。意味のある仕事をしないままトークンだけを消費するため、トークンを節約したいユーザーほど無視できない。

## 2. 「ループにガード」では止まらない — フレーミングの罠

この事象を見たとき、私の最初の直感は「ループ処理に上限や `sleep`、バックオフ（指数的な待機間隔）を入れればいいのでは」というものでした。発想としては正しい方向です。待機やリトライが無限に走らないよう歯止めをかける、という考え方そのものは健全です。

しかし、**対象がズレていました。**

真の問題は「コードのループ」ではありません。前章で書いたとおり、これはモデルの出力生成が反復に嵌まる現象です。だから「ループ処理にガードを入れる」という対策は、そもそも暴走している当の対象に届きません。`for` 文に上限をつけても、モデルが `call:` を吐き続けるのは止められないのです。

問題を正しく言い換えると、こうなります。

> **エージェントが同じ行動を無意味に反復し始めても、自力で止まれない。**

ここに気づくと、対策は 2 つのレイヤーに翻訳する必要があると分かります。

* **(A) 自律規律（行動のルール）** — AI 自身が「これは反復だ」と気づいて止まれるようにする
* **(B) ハーネス側の機械停止** — AI の自己制御に依存せず、外側から強制的に止める

この「対策の対象を取り違える罠」は、以前 [AIは対症療法に陥りがち](https://zenn.dev/fixu/articles/ai-symptomatic-treatment-trap) で書いた構図とよく似ています。目の前の症状（ループっぽく見える）に飛びつくと、本当の原因（自力で止まれない）を外してしまう。素朴な問い直しが効くのは、ここでも同じでした。

> 学び: 「ループにガード」は対象がズレた対策。真の問題は「AI が同じ行動を無意味に反復しても自力で止まれない」こと。対策は自律規律と機械停止の 2 レイヤーに翻訳する。

## 3. 二段防御の設計 — 自律規律だけでは穴がある

### (A) 自律規律 — AI が自分で止まる

まず、AI 自身に「無意味な反復をしない」ための行動ルールを持たせます。具体的には、こういった規律です。

* 待機やリトライは **上限 N 回** まで。無限に粘らない
* 1 ターンで **同一ツールを連打しない**
* 待機間隔は 2 秒 → 3 秒 → 5 秒 と延ばすが、**最大 3 サイクルで打ち切る**
* 「同じことを 2 回繰り返したと気づいたら、即座に止まる」

これは効きます。多くのケースでは、AI が自分で「あ、繰り返している」と気づいて手を止めてくれます。

ただし、**自律規律には構造的な穴があります。**

暴走の最中、モデルは自分のルールを参照できるとは限りません。出力生成が反復に嵌まっているまさにその状態では、「自分は今ルールに違反している」という判断そのものができなくなっていることがあります。冷静なときには守れるルールが、暴走中には機能しない。**自己制御は、自己が壊れているときには頼れない** のです。

### (B) ハーネス側の機械停止 — Stop hook

そこで決め手になるのが、**AI の自己制御に一切依存しない、外形的なストッパー** です。Claude Code には Stop hook という仕組みがあり、エージェントの応答が終わったタイミングでスクリプトを走らせて、セッションを止めるかどうかを外側から判定できます。

モデルが正気かどうかに関係なく、**ハーネス（実行環境）の側で「同じ行が異常な回数反復していたら止める」** と決めておく。これなら、モデルが自分のルールを見失っていても関係ありません。外から機械的にブレーカーを落とせます。

> 学び: 自律規律は冷静なときには効くが、暴走中はモデルが自分のルールを参照できず穴になる。自己制御に依存しない外形的なストッパー（Stop hook）を併用してはじめて、確実に止められる。

## 4. 実装 — トークンを守る Stop hook

ここからが具体的なテクニックの共有です。Claude Code の Stop hook として動かしている検知スクリプトの中身を紹介します。

### 4-1. 検知スクリプト

やっていることはシンプルです。

1. 標準入力で受け取る JSON から、会話ログ（transcript）のパスを取得する
2. 二重発火を防ぐため、すでにこの hook 由来で止まっている場合は素通りする
3. transcript の **直近の assistant メッセージ** のテキストを取り出す
4. トリム後・空行を除いて、**同一行の最大反復回数** を数える
5. 閾値（ここでは 8 回）以上なら、セッションを止める JSON を出力する
6. **どんな異常が起きても、判定できないときは必ず素通りする（fail-safe）**

最後の fail-safe が重要です。安全装置のつもりで入れたスクリプトが、バグでセッションそのものを壊してしまっては本末転倒だからです。判定に少しでも迷ったら止めない、という方針で全体を `exit 0` に倒しています。

```
#!/usr/bin/env bash
# loop_guard.sh — Stop hook 安全装置
# 目的: モデルが「ツールを呼ばず同一行（例 'call:' や空行）を大量反復する暴走」を
#       transcript から機械検知し、次ターンへ波及する前にセッションを停止する。
# 設計: 直近の assistant メッセージ内で、同一(trim後)行が閾値以上反復していたら停止。
#       どんな異常でもセッションを壊さないよう、判定不能時は素通り(exit 0)。
# 出力: 検知時のみ {"continue": false, ...} を stdout に JSON 出力。

THRESHOLD=8   # 同一行がこの回数以上反復したら暴走とみなす

input="$(cat 2>/dev/null || true)"
[ -z "$input" ] && exit 0

# 既に loop_guard 由来で止まっている場合の二重発火防止
active="$(printf '%s' "$input" | jq -r '.stop_hook_active // false' 2>/dev/null || echo false)"
[ "$active" = "true" ] && exit 0

tp="$(printf '%s' "$input" | jq -r '.transcript_path // empty' 2>/dev/null || true)"
[ -z "$tp" ] && exit 0
[ ! -f "$tp" ] && exit 0

# 直近の assistant メッセージの text 部分を連結して取り出す
last_text="$(jq -rs '
  [ .[] | select((.type=="assistant") or (.message.role?=="assistant")) ]
  | last // empty
  | (.message.content // [])
  | map(select(.type=="text") | .text) | join("\n")
' "$tp" 2>/dev/null || true)"
[ -z "$last_text" ] && exit 0

# 同一(trim後・空行除外)行の最大反復回数
maxrep="$(printf '%s\n' "$last_text" \
  | sed 's/^[[:space:]]*//; s/[[:space:]]*$//' \
  | grep -v '^$' \
  | sort | uniq -c | sort -rn | head -1 | awk '{print $1}')"
maxrep="${maxrep:-0}"

if [ "$maxrep" -ge "$THRESHOLD" ]; then
  printf '{"continue": false, "stopReason": "🛑 loop_guard: 直近メッセージで同一行が %s 回反復する暴走を検知し停止しました。状況を整理してから再開してください。", "systemMessage": "🛑 loop_guard 発動: 同一行 %s 回反復を検知してセッションを停止しました"}' "$maxrep" "$maxrep"
  exit 0
fi

exit 0
```

`{"continue": false, ...}` を出力すると、Claude Code はそのセッションを停止し、`systemMessage` でユーザーに通知してくれます。つまり、暴走が次のターンへ波及する前に、機械的にブレーカーが落ちる、というわけです。

### 4-2. settings.json への登録

このスクリプトを Stop イベントに登録します。

```
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/絶対パス/loop_guard.sh",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

### 4-3. 検証

導入時には、最低限このあたりを手元で確認しておくと安心です。

* **検知する**: `call:` を 12 行並べた合成ログをパイプで流し、`"continue": false` の JSON が出るか
* **誤検知しない**: 普通の文章＋ツール呼び出しを含むログで、何も反応しないか
* **二重発火しない**: `stop_hook_active: true` のとき素通りするか
* **JSON が壊れていない**: `jq empty settings.json` で妥当性、Stop hook の入れ子構造が正しいか

合成ログをパイプで流すだけで挙動を確認できるので、本物の暴走を待たずにテストできます。

> 学び: 安全装置こそ fail-safe に倒す。判定不能・異常時は必ず素通り（exit 0）させ、ガード自体がセッションを壊さないようにする。導入前に「検知する／誤検知しない／二重発火しない」を合成ログで確認しておく。

## 5. 正直な限界 — できること・できないこと

期待値を正しく持ってもらうために、この対策の限界もはっきり書きます。

**Stop hook は「応答が終わった瞬間」に発火します。** つまり「1 つのメッセージの中で延々とストリーム出力し続けている、まさにその最中」にリアルタイムで割り込むことは **できません**。1 回のメッセージで溶けるトークンそのものを、この仕組みでゼロにはできない、ということです。

では何ができるのか。**暴走が次のターンへ波及するのを止める** ことです。今回のように同一セッションで 3 回連続するような連鎖は、確実に断てます。1 回目の暴走でブレーカーが落ちれば、2 回目・3 回目で溶けるはずだったトークンは守れます。

だからこそ、真の対策は二段構えになります。

* **(A) 自律規律** で、単発の暴走そのものを起きにくくする
* **(B) Stop hook** で、暴走の連鎖・次ターンへの波及を機械的に断つ

もう一つ、**閾値はトレードオフ** です。低くしすぎると、リストの列挙など正規の繰り返しを暴走と誤検知してしまいます。高くしすぎると、検知が遅れてその分トークンが溶けます。今回は事象に合わせて 8 回に置いていますが、これは正解の数字というより出発点です。自分の使い方に合わせて調整する前提の値です。

> 学び: Stop hook はリアルタイム介入はできず、「次ターンへの波及を断つ」のが役割。閾値は誤検知と検知遅れのトレードオフで、環境に合わせて調整する出発点にすぎない。

## 6. まとめ — 二段で守り、効果はこれから検証する

トークンを節約しながら AI エージェントを使うほど、「意味なくトークンが溶ける」事象は見過ごせません。今回の同一行反復暴走への対策を一段ずつまとめると、こうなります。

1. **自律規律** — 待機・リトライは上限 N 回、同一ツールを連打しない、「2 回繰り返したら止まる」。ただし暴走中はモデルが自分のルールを見失うので、これだけでは穴が残る。
2. **Stop hook（機械停止）** — 自己制御に依存しない外形的ストッパー。直近メッセージの同一行反復を検知して、次ターンへの波及を機械的に断つ。fail-safe を徹底し、ガード自体が事故を起こさないようにする。

貫く考え方は一つです。**AI も時に暴走する。だから「AI が自分で止まる」だけに賭けず、「外から機械的に止める」仕組みを併せ持つ。** 自律規律と外形ガードの二段で、はじめてトークンを安心して節約できます。

### 効果検証はこれから — フィードバックを求めています

最後にもう一度、正直に書きます。**この二段防御は、つい先日試し始めたばかりです。** 閾値 8 が適切なのか、誤検知がどれくらい起きるのか、本当に連鎖を断てているのか — 効果の検証はこれからです。

それでも先に共有することにしたのは、トークンが溶ける事象が多くのユーザーに関わりうるからです。同じ事象に心当たりのある方、`call:` のような前置きだけが反復した経験のある方、あるいは「もっと良い止め方がある」という方からのフィードバックを、広く求めています。検証が進んだら、その結果もあらためて共有するつもりです。
