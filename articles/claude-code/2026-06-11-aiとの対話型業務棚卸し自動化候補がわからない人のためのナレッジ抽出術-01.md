# はじめに

「AIを使って業務を効率化・自動化しましょう」

こういった号令が飛んでくるようになって久しいですが、実際に動けている人は少数派ではないでしょうか。

問題は「自動化したいかどうか」ではなく、「**何を自動化すればいいかわからない**」という状態です。

自動化の対象を選ぶためには、まず自分の業務を整理しなければいけません。でも定型業務ほど無意識化されていて、「毎月何をやっているか」を改めて言語化しようとすると手が止まってしまいます。かくいう私も、月次の定型業務を抱えながら「いつかまとめよう」と先延ばしにし続けていました。

そこで試してみたのが、AIを「まとめる相手」ではなく「**質問してくれる伴走者**」として使うアプローチです。結果として、特別な準備なしに業務の棚卸しと自動化候補の特定まで一気に進めることができました。

本記事では、その具体的なプロセスを紹介します。

---

# なぜ「自動化候補」を自分で特定するのは難しいか

定型業務を長くやっていると、こんな現象が起きます。

- 手順が「体で覚えている」状態になり、言語化できない
- 「この作業は自動化できそう」と思いつつ、全体像が見えないので手がつかない
- リストを作ろうとすると、どこから書けばいいかわからなくて断念する

これは「暗黙知」の問題です。熟練者ほど無意識に作業を進めるため、手順を言葉にする能力がむしろ落ちていきます。

**よくある失敗パターン①：手が止まる**

```
「では業務一覧を書いてください」
→ 空白のスプレッドシートを前に手が止まる
→ 数行書いて力尽きる
→ 「後でやる」と先送りしてまた手つかずへ
```

**よくある失敗パターン②：表面だけ自動化する**

仮に動き出せたとしても、次の罠があります。人は「何となく自動化できそう」と感じる作業を無意識に選んでしまいます。たとえば「CSVをExcelに貼る作業を自動化しよう」と思ったとき、その前後にある工程（どのCSVをいつどこからダウンロードするか、送付先の確認はどこで行うかなど）は視野から外れがちです。目立つ手作業だけを自動化しても、前後が手作業のままでは業務全体の負荷はほとんど変わりません。

**よくある失敗パターン③：手段と目的が逆転する**

さらに進むと、「自動化すること」自体が目的になっていきます。気づけば、部分的なスクリプトや断片的なAI生成ツールが乱立し、「どれをいつ使えばいいか」がわからない状態に陥ります。本来の目的は業務負荷を減らすことだったはずが、ツール管理という新たな負担が生まれてしまいます。

これらに共通する根本原因は、**自分の業務の全体像を把握しないまま自動化を始めてしまうこと**です。この問題を解決するために必要なのは「書く意志力」や「自動化の技術力」ではなく、「**全体を漏れなく引き出してくれる仕組み**」です。

# アプローチ：2フェーズの運用

今回試したアプローチは、**蓄積フェーズ**と**分析フェーズ**の2段構えです。いずれのフェーズでもAIが質問してきますが、目的と内容が異なります。

```
【蓄積フェーズ：1週間〜1ヶ月】
  その日やった作業を雑に報告する
  → AIが「記録を構造化する」ための質問をしてくる
  → 答えながら1件1件の作業が言語化・整理される
  ↓ 業務が一巡するまで繰り返す

【分析フェーズ：月1など好きなタイミングで】
  「蓄積データを分析して」と依頼する
  → AIが「自動化候補を絞り込む」ための質問をしてくる
  → 答え終わったら棚卸し＋分類＋優先順位付き自動化候補が揃う
  ↓ 自動化を1つ実施
  ↓ 蓄積フェーズに戻り、次のサイクルへ
```

ポイントは「完璧な情報を用意してからAIに渡す」のではなく、「**不完全な状態からスタートしてAIに整えてもらう**」という発想の逆転です。そして分析は一度やって終わりではなく、**蓄積→分析→自動化の小さなサイクルを回し続ける**ことで、自動化の範囲が少しずつ広がっていきます。

最初の分析に必要な情報量は、業務が一巡するまでのログです。毎日発生する作業なら1週間、月次業務が中心なら1ヶ月ほど報告を積み上げてから初回分析を行うのが現実的です。

# 準備：AIへの事前設定（システムプロンプト）

このアプローチを機能させるには、**AIに最初から役割と動作ルールを伝えておく**ことが重要です。毎回「今日の報告を整理して」と説明し直す手間が省けますし、フェーズに応じた質問の質も上がります。

Claude.ai のプロジェクト機能やカスタム指示、ChatGPT のカスタムインストラクションなど、お使いのツールのシステムプロンプト設定欄に以下をそのまま貼り付けてください。

```
あなたは私の業務改善パートナーです。
以下の2つのモードで動作してください。

## 蓄積モード（私が「今日の作業」「今日やったこと」などを報告したとき）
- 報告を受けたら、記録を構造化するための質問を1〜2問してください
- 優先して確認すること：作業の頻度（毎日/週次/月次など）、手作業が発生している箇所、
  使用しているツールやファイル、この作業の前後に必要なこと
- まだ分析や自動化の提案はしないでください
- 質問への回答を受けたら、その作業を以下の形式でまとめてください
  - 作業名 / 頻度 / 手順（フェーズ） / 使用ツール / 手作業の箇所 / 気づいた暗黙知

## 分析モード（私が「分析して」「自動化候補を出して」などと依頼したとき）
- これまでに記録した作業を横断的に分析してください
- 表面的に目立つ手作業だけでなく、作業間の依存関係・前後の工程・暗黙知まで
  把握してから候補を出してください
- 自動化候補は優先度順（着手しやすさ × 効果）でリストアップし、理由も添えてください
- 断片的なツールが乱立しないよう、業務全体の流れを踏まえた提案をしてください
- 判断に必要な情報が不足している場合は、提案を出す前に質問してください

## 共通ルール
- 一度に複数の質問を投げかけないでください（1〜2問ずつ）
- 私が曖昧な表現をしても、深掘りする質問で補ってください
- 自動化の技術的な実装方法よりも「何を・なぜ自動化すべきか」の整理を優先してください
```

このプロンプトのポイントが2つあります。

1つ目は**モードを明示的に分けている**点です。「報告したとき」と「分析を依頼したとき」で動作を切り替えることで、蓄積フェーズで余計な提案が飛んでくるのを防ぎます。AIが先走って「この作業は自動化できます」と言い始めると、まだ全体像が見えていない段階で表面的な候補に引っ張られてしまいます。

2つ目は**分析時に「全体の流れを踏まえた提案」を明示している**点です。これが、依存関係や前後工程を無視した断片的な自動化になるのを防ぐ役割を果たします。

## 蓄積データは外部ファイルに保存する

ここで実用上の注意点があります。多くのAIツールは**セッションをまたいで会話履歴を保持しません**。1週間〜1ヶ月にわたって報告を積み上げるこのアプローチでは、記録の永続化が必要です。

ただし、**ファイルの更新作業もAIに任せる**ことで、ユーザーの負担はほぼゼロにできます。

### AIツール別の運用方法

主要ツールはいずれも「外部ファイルの直接編集」か「生成ファイルのダウンロード」に対応しています。

| ツール | ファイル操作 | 運用イメージ |
|---|---|---|
| **Claude（Claude Code）** | ローカルファイルを直接読み書き | AIがセッション後に `tasks.md` を自動更新。次回も自動で読み込む |
| **Claude.ai** | Artifactsとして生成→ダウンロード | AIが更新済みファイルを出力→ユーザーが上書き保存→次回はファイルを添付 |
| **ChatGPT** | ファイル生成→ダウンロード | Claude.ai と同様。ファイルを添付して渡せば文脈を引き継げる |
| **Gemini** | Google ドキュメント/スプレッドシートを直接読み書き | Google ドライブ上のドキュメントをAIが直接参照・更新。ファイルの受け渡し作業が不要 |

GeminiはGoogle Workspaceとの統合が深いため、ドライブ上のドキュメントを指定するだけでAIが直接読み書きできます。ファイルを渡す・受け取るという手順ごと省略できるのはGeminiの強みです。

### 推奨ワークフロー（Claude.ai / ChatGPT の場合）

```
【蓄積フェーズ（毎回）】
1. ファイルを添付して作業を報告する
2. AIの質問に答える
3. AIが更新済みの tasks.md を出力する → 上書き保存するだけ

【分析フェーズ（月1など）】
1. 蓄積ファイルを添付して「分析して」と依頼する
2. AIの質問に答える
3. 自動化候補リストが出力される
```

ユーザーが行う操作は「ファイルを添付する」「上書き保存する」の2つだけです。

### システムプロンプトへの追記

先ほどのシステムプロンプトに以下を加えると、ファイル管理の指示まで自動化できます。

```
## ファイル管理
- 添付された tasks.md がある場合は、その内容を記憶の起点として使ってください
- 蓄積モードで記録をまとめた後は、tasks.md の更新版を必ず出力してください
- 分析モードでは tasks.md の全内容をもとに分析してください
- ファイルを直接編集できる環境では編集し、できない場合はダウンロード用ファイルを生成してください
```

外部ファイルに記録を持つことは、単なる「セッションリセットへの対策」以上のメリットがあります。

- **ツール非依存**: 乗り換えても記録がそのまま使える
- **記録が業務マニュアルの下書きになる**: ファイルを読み返すだけで業務全体像が把握できる
- **差分管理ができる**: Gitで管理すれば「先月からどの作業が増えたか」が追跡できる
- **チームへの横展開が容易**: ファイルを共有するだけで他のメンバーも同じ文脈でAIと対話できる

# 実践：最初の報告はどう書くか

最初の入力はこれだけで十分です。完璧に書こうとすると逆に手が止まるので、**思いついたまま箇条書きで送る**のがコツです。

**実際に送った内容（一部一般化）**:

```
今日やった定型作業の記録を残したいです。

・月次5営業日締めでやっている作業が3つある
・1つ目：顧客向けにアクセスログを集計して送付している
・2つ目・3つ目：クラウドサービスの請求書をもとに、
  顧客向けの請求書を作って送る（請求先が2件）
・全部Excelでやっている気がする
・具体的な手順はあまり意識せずやっている

足りない情報があれば質問してほしい
```

「自動化できそうなところを教えてほしい」とは書かないのがポイントです。システムプロンプトで蓄積モード中の提案を抑制していても、ユーザー側から誘導してしまうと表面的な候補に引っ張られます。まずは「記録したい」「質問してほしい」という言葉にとどめておきます。

「全部Excelでやっている**気がする**」という曖昧さでも大丈夫です。AIはそこを掘り下げてくれます。

# AIが質問してきたこと（対話サンプル）

## 蓄積フェーズ：作業を報告するたびに聞かれること

蓄積フェーズでのAIの質問は、「この作業を正確に記録するために何が足りないか」を補う目的です。報告の粒度を揃え、後で分析できる形に整えてくれます。

**私の報告**:
> 今日はアクセスログの集計と送付をやりました

**AIの質問**:
> アクセスログの集計は、どこかからデータをダウンロードして、Excelに貼り付ける形ですか？それとも何かのシステムから直接出力できますか？

**私の回答**:
> 管理画面からCSVをダウンロードして、Excelのテンプレートに貼り付けています

**AIの次の質問**:
> そのExcelテンプレートは、貼り付けた後に手動で集計式を修正したりしていますか？それとも貼り付けるだけで完成しますか？

**私の回答**:
> 月や件数によって行数が変わるので、集計範囲を毎回手動で直しています

この質問はシンプルですが、「集計範囲を毎回手修正している」という事実を、自分から言葉にする機会はほぼありません。聞かれて初めて「そういえばそうだな」と気づきます。

## 分析フェーズ：蓄積データをもとに聞かれること

分析フェーズでのAIの質問は性格が変わります。複数の作業を横断して「自動化のしやすさ」を判断するために必要な情報を聞いてきます。

**AIの質問**:
> アクセスログの送付と請求書作成、どちらを先にやる必要がありますか？順番に制約はありますか？

**私の回答**:
> アクセスログの送付が終わってから請求書の金額が確定するので、順番があります

**AIの次の質問**:
> 請求書の計算は、クラウドサービスの請求額をそのまま使いますか？それとも独自の換算や按分がありますか？

**私の回答**:
> 独自の換算式があります

蓄積フェーズでは出てこなかった「依存関係」や「換算ロジックの有無」が、ここで初めて浮かび上がります。自動化の設計に直結する情報なので、AIはこのフェーズで集中的に聞いてきます。

---

これら両フェーズのやり取りを経て、自分でも意識していなかった事実が整理されていきました。

- **毎回集計範囲を手修正していたことは「当たり前すぎて」盲点だった**
- 請求書作成も同様に、PDFをダウンロードして金額を手打ちしていた
- 3つの作業には**依存関係があった**（アクセスログ送付が完了するまで請求書の一部が確定しない）

「こんなこと言わなくてもわかっているだろう」と思っていた手順が、実は重要な暗黙知だったわけです。

# 得られた構造化アウトプット

対話が終わった後、AIが出力した分類結果は以下のような形式でした。

```yaml
tasks:
  - name: アクセスログ集計・送付
    frequency: 月次（5営業日締め）
    phases:
      - CSVダウンロード（管理画面）
      - Excelテンプレートへの貼り付け
      - 集計範囲の手動修正
      - メール送付
    implicit_knowledge:
      - 集計範囲は行数によって毎回変わる
      - テンプレートのシート構成は変更不可
    automation_candidate: 高
    reason: CSVダウンロード〜集計範囲修正はスクリプト化可能。メール送付はテンプレート化で半自動化できる

  - name: 請求書作成（クラウドサービスA）
    frequency: 月次（5営業日締め）
    phases:
      - ポータルからPDFダウンロード
      - 金額の手打ち
      - Excelフォーマットへの転記
      - PDF化・送付
    implicit_knowledge:
      - 金額計算に独自の換算式がある
    automation_candidate: 中
    reason: PDF解析が必要。換算式の確認後にスクリプト化できる可能性あり

  - name: 請求書作成（クラウドサービスB）
    frequency: 月次（5営業日締め）
    dependencies:
      - アクセスログ集計・送付の完了後に金額確定
    automation_candidate: 中
    reason: 依存関係があるためトリガー設計が必要。先にAを自動化してから取り組む
```

ここで重要なのは、**自分ではぼんやりとしか把握していなかった業務が、フェーズ・暗黙知・自動化難易度・依存関係の4軸で整理された**点です。これを自力でゼロから書こうとしていたら、数時間かかるか途中で挫折していたでしょう。

# なぜこれが機能するのか

このアプローチが有効な理由は、AIが「**答えやすい質問**」を選んで聞いてくれるからです。

「業務を整理してください」という問いは難しいですが、「CSVダウンロードしていますか？」という問いは簡単に答えられます。AIはその簡単な問いを積み重ねることで、自分でも整理できていなかった業務の全体像を引き出します。

そして、この「全体像を先に把握する」というプロセスが、前述の3つの失敗パターンをまとめて防いでくれます。

- 業務を一巡させてから分析するので、**目立つ作業だけでなく前後の工程も含めた包括的な自動化候補**が出てくる
- 「何を自動化するか」が明確になってから着手するので、**目的を見失ったまま断片的なツールを作り続ける状態**に陥らない
- 依存関係や暗黙の前提条件が先に整理されるので、**後から仕様漏れが発覚して作り直す手戻り**が減る

| 方法 | 最初のハードル | リスク |
|---|---|---|
| 自分でドキュメントを書く | 高い（何から書くかわからない） | 抜け漏れが多く表面的になりがち |
| 気になる作業からすぐ自動化 | 低い（すぐ始められる） | 断片的なツールが乱立し目的を見失う |
| **雑に報告 → AI対話で棚卸し → 全体把握してから自動化** | **低い（思ったことを言うだけ）** | **包括的な自動化につながる** |

# 次のアクション：自動化候補への橋渡し

棚卸し結果が出たら、自動化への道筋はシンプルです。

**AIに「何から始めればいいですか？」と聞きます。**

棚卸し結果を渡した上でこの質問をすると、優先度順に提案が返ってきます。

> 最初に取り組むなら「アクセスログ集計・送付」をお勧めします。  
> 理由：依存関係がない・スクリプト化しやすい・効果が目に見えやすい、の3点です。  
> 具体的にはPythonでCSVを読み込み、openpyxlで集計範囲を動的に設定する方法が現実的です。

ここで初めて「**具体的に何を作ればいいか**」が見えてきます。棚卸しなしにこの段階に来ようとすると、「何を自動化するか」の議論で止まってしまいます。

棚卸し → ToDoリスト化 → 優先順位付け → 自動化着手、という順序が大切で、AIはその全ステップを伴走してくれます。

# まとめ

今回のアプローチを一言でまとめると「**書かずに喋る、急がず積み上げる**」です。

- 完璧な情報は不要。思いついたままを箇条書きで送るだけ
- 蓄積フェーズではAIが「記録を整える」質問をしてくる。それに答えるだけ
- 業務が一巡したら分析を依頼する。AIが今度は「自動化を判断する」質問をしてくる
- 気づいたら棚卸し・分類・優先順位付き自動化候補まで揃っている
- 自動化を1つ実施したら、また報告を続けて次のサイクルへ

最初の1週間〜1ヶ月は「インプットするだけ」で何も成果が出ない時期です。でもその蓄積がなければ、どんなに優秀なAIも「何を自動化すべきか」を判断できません。焦らず報告を続けることが、このアプローチの唯一のコツです。

「AIで自動化しましょう」という話を受けて最初にすべきことは、コードを書くことでも、ツールを選ぶことでもなく、**今日やった作業をAIに語りかけること**かもしれません。
