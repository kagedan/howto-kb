AIコーディングエージェントの使い方が、この1年で決定的に変わりました。

つい最近まで、私たちの仕事はこうでした。  
良いプロンプトを書く。返ってきたものを読む。次のプロンプトを書く。エージェントは道具であり、あなたはその道具を「ずっと手に持っている」状態です。

いま、その前提を覆す考え方が広がっています。**ループエンジニアリング（Loop Engineering）** です。ひとことで言えば、

> **「エージェントに指示を送る人」、すなわち、あなた自身をあなたが設計した仕組みに置き換える。**

Google の Addy Osmani が2026年6月にこの言葉を体系化し、一気に定着させました。ループとは「再帰的な目標（recursive goal）」のこと。目的を一度定義すれば、あとはAIが完了まで自分で回り続けます。あなたの役割は「プロンプトを打ち続ける係」から「仕組みを設計するエンジニア」へと移ります。

## はじめに ── この記事の位置づけ

先に、この記事の立ち位置を書いておきます。

これまで私は、企業向けの設計論や、評価器（evaluator）を組み込んだ自律的なループといった、やや専門的なテーマを中心に書いてきました。今回はあえて専門度を下げ、その土台にあたる「ループエンジニアリングとは何か」を一度きれいに整理し直します。すでに自分の他の記事を読んでいただいている方は気付かれているかもしれませんが、この数か月、私は自身の開発環境でこのループ理論を運用してきました。その中で強く感じているのは、ループエンジニアリングは単なる便利な自動化テクニックではなく、AIコーディングエージェントとの向き合い方そのものを変える、大きなパラダイムシフトだということです。

内容の多くは、第一線の実践者や公式ドキュメントで既に語られていることを、英語圏の一次情報からまとめ直したものです。新しい発見というより **「頭の整理」** として読んでもらえればと思います。

ただ、整理して終わりにはしません。ループエンジニアリングを一通り理解したあとに、必ず残る問いがあります。**「理屈はわかった。で、実際にはどう作るのか？」** ── この記事は、そこまでを一続きにするつもりで書いています。前半で理論を組み立て、最後に **それをそのまま形にした、すぐ試せる実装** を紹介します。理論と実装を、切り離さずにひとつの流れとして受け取ってもらえればと思います。

**この記事の流れ：**

1. ここに至るまでの潮流（プロンプト → コンテキスト → ハーネス → ループ）
2. ループエンジニアリングで、実際に何ができるのか
3. その裏にある理論
4. 理論を実装に落とす ── loop-kit

## ここに至るまで ── 積み上がってきた4つの階層

ループエンジニアリングは、ある日突然生まれたものではありません。ここ数年、工夫のしどころが「モデルそのもの」から少しずつ外側へ移ってきた結果です。順番に見ていきましょう。それぞれの層は前の層を包み込むもので、置き換えるものではありません。

### ① プロンプトエンジニアリング（〜2024年ごろ）

**「どう聞くか」** を工夫する段階です。ChatGPT登場以降の約2年間、成果を左右したのは一回の指示文の書き方でした。役割を与える、手本を見せる、順を追って考えさせる（chain-of-thought）── こうした技法が中心です。

限界もはっきりしていました。一回の指示は「次に何をするか」「何をもって完了とするか」までは決められません。

### ② コンテキストエンジニアリング（2025年半ば〜）

問いが **「どう聞くか」から「何を見せるか」** へ移りました。Andrej Karpathy や Shopify の Tobi Lütke が2025年6月ごろに「これこそ本当のスキルだ」と発信し、Anthropic も同年9月に公式記事「Effective context engineering for AI agents」で概念を整理しました。

Anthropic の定義を平たく言えば、**「モデルに渡すトークン（＝コンテキストの窓）全体を、望む挙動が出やすいように設計する」** こと。システム指示、検索してきた資料、使えるツール、過去のやり取り、記憶や状態 ── これらをどう組み合わせて窓に載せるかが、モデルの言い回しそのものより効いてくる、という気づきです。マルチターンで動くエージェントには不可欠の考え方でした。

### ③ ハーネスエンジニアリング（2026年初頭〜）

**「エージェントが動く環境そのものを作り込む」** 段階です。2026年初頭に広まった考え方で、Thoughtworks の Birgitta Böckeler が martinfowler.com で「Agent = Model + Harness（エージェント＝モデル＋ハーネス）」という明快なモデルとして整理し、Addy Osmani も「Agent Harness Engineering」で論じました。

ハーネス（harness）とは「モデル＋その周りに組み上げるすべて」── プロンプト、ツール、フック、サンドボックス、サブエージェント、フィードバック経路、復旧の仕組み。Osmani いわく、Claude Code、Cursor、Codex、Aider、Cline は、どれも中身は「ハーネス」です。同じモデルでも、体験を決めているのはハーネスの出来だ、というわけです。

この段階の核心は **「そこそこのモデル＋優れたハーネス」が「優れたモデル＋貧弱なハーネス」に勝つ**、という一点です。そしてもう一つ、**「一度締めたら緩めない」** という考え方。エージェントがミスをするたびに、それを一度きりの失敗として流すのではなく、「二度と起きないよう仕組みを締める合図」として扱う。規約を AGENTS.md に書き足す、危険なコマンドを止めるフックを足す、レビュー用のサブエージェントに検知させる。こうしてハーネスは失敗のたびに強くなります。

ただし、ハーネスが面倒を見るのは基本的に **「1回の実行（セッション）の内側」** です。その1回のなかで複数のサブエージェントを並列に走らせることは、むしろ普通にあります（Claude Code もそうします）。ハーネスが扱わないのは、その **実行と実行のあいだ** ── そこを設計するのが、次のループです。

### ④ ループエンジニアリング（2026年6月〜）

そしてハーネスの **「一つ上の階」** に立つのがループエンジニアリングです。Addy Osmani が2026年6月7日にこの語を打ち出し、同じ趣旨を先に語っていた二人の実践者の言葉を束ねました。

* **Peter Steinberger**（PSPDFKit の創業者。公開からまたたく間に数十万スターを集めたOSSエージェント「OpenClaw」を作り、現在は OpenAI 在籍）── もうエージェントにプロンプトを打つのはやめよう。エージェントにプロンプトを打つ「ループ」を設計すべきだ、と発信。
* **Boris Cherny**（Anthropic の Claude Code 責任者）── 自分はもう Claude に直接プロンプトを打っていない。Claude にプロンプトを打ち、次に何をするか判断する「ループ」を走らせている。自分の仕事はループを書くことだ、と語りました。

ハーネスが **「1回の実行の内側」** を作り込むのに対し、ループは **「実行と実行のあいだ」** を設計します。決まった時刻に起動し、必要なら複数のサブエージェントを走らせ、自分で次の一手を決め、自分に燃料をくべ続ける。仕事を見つけ、割り振り、確認し、済んだことを書き留め、次を決める ── その一連をあなたの代わりに回す小さなシステムを作り、あなたはエージェントを突く役から降りるのです。

面白いのは、これがもう「自前のツール自慢」ではなくなったこと。1年前ならbashスクリプトの山を自分で書いて永遠に保守する必要がありました。いまは同じ部品が Claude Code や Codex といった製品の中に標準で載っています。だからこそ「どのツールか」を争うより、**どのツールでも成り立つループを設計する** ほうが本質的になりました。

**まとめると流れはこうです：**

| 階層 | 焦点 | あなたの役割 |
| --- | --- | --- |
| プロンプト | どう言うか | 一手ずつ指示する |
| コンテキスト | 何を見せるか | 情報環境を整える |
| ハーネス | 実行環境を作り込む | 1回の実行を囲い込む |
| **ループ** | **実行と実行のあいだ** | **仕組みの設計者** |

大事な補足を一つ。ループの中で雑なプロンプトを使えば、雑な仕事が速く量産されるだけです。ループが足すのは、その全体を自律的に回すしくみであって、下の層の質までは肩代わりしません。

## ループエンジニアリングで、実際に何ができるのか

理論の前に、まず「何が嬉しいのか」を具体的に見てみましょう。Addy Osmani が挙げる典型例が、いちばん実感が湧きます。

**朝の仕分け（トリアージ）ループ。** 毎朝ひとりでに起動し、昨日のCI失敗・未対応のIssue・直近のコミットを読む。気づきをMarkdownファイル（あるいはLinearのボード）に書き出す。対処すべきものごとに独立した作業場所（worktree）を切り、サブエージェントに修正を書かせ、別のサブエージェントがその下書きをプロジェクトのスキルや既存テストに照らして点検する。ツール連携（コネクタ）経由でPRを開き、チケットを更新する。自分の手に負えないものは「仕分け用の受信箱」に入れて人間に回す。状態ファイルがあるおかげで、今日どこまで進んだかを覚えていて、明日の朝は続きから再開できる。── これは立派なエンジニアリングの流れ丸ごとです。しかも設計者は最初に一度組んだだけで、個々のステップにプロンプトを打っていません。

同じ発想の応用は、すでに現場で回っています。

* **PRのお守り（PR babysitter）**：自分のPRを定期的に見張り、レビュー指摘に応え、落ちたCIを直す
* **依存の掃除**：依存ライブラリの更新やCVE（脆弱性）対応。ただしメジャー変更は人間の承認を挟む
* **Issueの仕分け / チェンジログ下書き / マージ後の後片付け**
* **定型サマリ**：毎朝Slackの特定チャンネルを要約する、など

OpenAI も社内で、日々のIssueの仕分け、CI失敗の要約、コミットの要点まとめ、先週紛れ込んだバグ探しといった「退屈だが大事な雑務」にこの仕組みを使っている、と Osmani は書いています。

チーム視点では、こんな効き方もあります。

* **属人化の解消**：「このリポジトリではこう直す」という各自の暗黙知をスキル（後述）として書き出し、誰のループでも同じ品質で回るようにする
* **既存ツールとの連携**：Linear・Jira・Slackなどとつなぎ、閉じたループの中でチケットまで動かす

ここまではスケジュールで動くループが中心でした。もうひとつ、少し性格の違う使い方があります。**「一回きりだが、複雑な実装」をループに任せる** やり方です。

たとえば、ある機能をまるごと実装したいが、要件はまだ曖昧で、自分でも細部が詰まっていない ── そんなときに使えます。まずループに **要件そのものを整理させる**。何が不明で、どこに判断が必要かを洗い出し、あなたに確認を取り、仕様として固める。固まったら、その仕様に対するテストを書き、実装し、**テストが全部通る（緑になる）まで自分で試行錯誤を続ける**。あなたが最初に要件を渡して承認したあとは、完了まで手を離せます。毎朝動くわけでも、無人で回り続けるわけでもない。けれど「停止条件を満たすまでサイクルを回す」という点で、これも立派なループです。

要するにループが得意なのは、**「完了の条件がはっきりしていて、機械的に検証できる仕事」** です。それが毎朝繰り返される定型作業なら、あなたが寝ているあいだに片づけてくれる。一回きりの複雑な実装なら、あなたが別の仕事に集中しているあいだに、緑になるまで粘ってくれる。

## 仕組みを理解する ── ループの理論

「便利そう」の裏側にある考え方を、ここで整理します。ここが分かると、良いループと危ういループの差が見えてきます。

### そもそもループとは

Anthropic 公式（Claude Code チーム）の定義がいちばん明快です。**ループとは「停止条件を満たすまで、作業のサイクルを繰り返すエージェント」** のこと。

端的に言えば、**ループとは「チェック（検証）のついたタスク」** です。チェックのないタスクは、ただの願望にすぎません。この一点がすべての土台です。何かをやる → 結果を確認する → 続けるか止めるかを決める。技術のほとんどは「その確認を本物にすること」と「いつ止めるかを決めること」に宿ります。

### Anthropicが整理する4つのループ型

Anthropic の公式ドキュメントは、ループを起動と停止のしかたで4種類に分けています。入門としてこの分類は非常に見通しが良いので、そのまま紹介します（Claude Code の機能名を例に取っていますが、考え方はツール非依存です）。

1. **ターン型（turn-based）**：あなたのプロンプトで起動し、Claude が「できた」または「情報が足りない」と判断したら止まる。毎回の対話がこれ。いちばん基本のループです。
2. **ゴール型（`/goal`）**：あなたが「完了の条件」を定義する。各ターンのあと、**作業役とは別の小さな評価モデル**（既定では Haiku）が条件を満たしたか判定し、満たすまで（あるいは条件に書いた試行回数の上限まで）続けさせる。ここに一つ、正確に押さえておきたい点があります。**現在の仕様（2026年7月時点、Claude Code v2.1.139 以降）では、この評価モデルは自分でコマンドを実行したりファイルを読んだりしません**。会話（トランスクリプト）にすでに出力された内容だけを見て判定します。だから条件は「Claude自身の出力で示せる形」で書く必要があります。「テストが全部通る」が機能するのは、Claudeがテストを実際に走らせ、その結果が会話に残るからです。**機械的に判定できる証拠が会話に残る条件**ほど効果的、というわけです。
3. **時間型（`/loop`・`/schedule`）**：一定間隔で走る。繰り返しの定型作業や、外部システム（PRやCIなど）の変化に反応する用途に向く。
4. **プロアクティブ型（proactive）**：イベントやスケジュールで、リアルタイムの人間なしに起動する。各タスクは目標達成で終了し、ルーティン自体はあなたが止めるまで走り続けます。

タスクが単純なら、複雑なループは要りません。**いちばん単純なやり方から始め、必要なときだけ上の型に上げる** ── これが Anthropic の推奨です。

### ループを構成する部品：5つ＋「記憶」

Addy Osmani は、ツールに依存しないループの構成要素を **「5つの部品と、それとは別に1か所の記憶」** として挙げています（Claude Code も Codex も、いまはこの5つを全部持っています）。

1. **自動化（Automations）**：スケジュールで起動し、発見と仕分けを自分で行う。ループを一定間隔で起こす起点です
2. **ワークツリー（Worktrees）**：並列で走らせたときにエージェント同士がファイルを踏み合わないための隔離。Gitの独立した作業ディレクトリで、一方の編集がもう一方に触れられない
3. **スキル（Skills）**：エージェントが放っておけば推測で埋めてしまうプロジェクト知識を、外に書き出しておく（例：`SKILL.md`）。長い指示文を毎回貼るのではなく、名前で呼び出す
4. **プラグイン / コネクタ（Plugins and connectors）**：チケット更新・PR作成・Slack通知など、あなたが既に使っているツールにエージェントをつなぐ（MCPベース）
5. **サブエージェント（Sub-agents）**：一方がアイデアを出し、別の一方がそれを点検する（後述する最重要原則）

そして Osmani が「6つめのもの」と呼ぶのが **記憶（State / Memory）** です。Markdownファイルでも、Linearのボードでもいい。ひとつの会話の外で生き続け、「何が済んだか・次は何か」を保持するもの。モデルは実行と実行のあいだにすべてを忘れます。だから記憶は **コンテキスト（プロンプト）の中ではなく、ディスク上**（ファイルやgit履歴）に置く。**「エージェントは忘れるが、リポジトリは忘れない」**。これが長時間動くループが共通して依存する仕掛けです。

> 補足：この「毎回まっさらなコンテキストで起動し、記憶はファイルに書き出す」やり方は、Geoffrey Huntley が2026年初頭に「Ralph（ラルフ）テクニック」として広めたものが源流の一つです。素朴な `while` ループの中でエージェントを回し、書かれた仕様に対して毎回新しい文脈で一手ずつ進めます。

### いちばん大事な原則 ── 作る者と、採点する者を分ける

ループ理論の心臓部がこれです。**コードを書いたモデルは、自分の宿題を採点するには優しすぎる。**

生成するモデルには「書いたものを肯定したい」慣性が働きます。一方、別の指示（ときには別のモデル）で見るレビュー役は、最初のモデルが自分を言いくるめて通した箇所を捕まえられる。だから、コードを **書くエージェントと、それを点検するエージェントを別にする**（maker–checker 分離／generator–evaluator 分離）。

Claude Code の `/goal` が「止めてよいか」の判断を **作業役とは別のモデル** に委ねているのも、この発想の一形態です。ただし前に触れたとおり、その評価モデルは会話に書かれた内容を読むだけで、自分でテストを走らせ直すわけではありません。つまり **「判断の独立性」はあっても、「検証の独立性」まではない**。作業役が「テストは通った」と会話に書けば、評価役はそれを前提に判定します。この一段先 ── 点検役に検証コマンドを **自分の手で実行させる** ところまで踏み込むのが、後半で紹介する loop-kit の狙いです。

そして評価の順番は **「機械的で信頼できるものを先に」**。まず決定論的なチェック（テストを実際に走らせて緑になるか）、次にAIによるレビュー、人間は最後にエビデンス（証拠）だけを見る。**検証の質がループ全体の質を決める急所** になります。

### 止め方の設計 ── 「終わった」と「うまくいった」は別物

暴走を防ぐ設計も理論の一部です。ここでよくある落とし穴は、**停止条件と成功条件を混同すること**。仕分けループが何事もなく綺麗に終了したのに、実は何も直していなかった ── そんな事故は珍しくありません。「ループが止まった」は「うまくいった」を意味しないのです。だから停止のルールは、単に「終わったか」ではなく **「出力を採点する」** 必要があります。

実務では、次のような「ハードな歯止め」を明示的に置きます。

* **反復回数の上限**（例：5回試したら止める）
* **予算・トークンの上限**（青天井の燃焼を防ぐ）
* **停滞の検知**（同じ失敗を繰り返していないか、差分が出ているか）
* **人間ゲート**：取り返しのつかない操作や、要件そのものを変えるような判断は、必ず人間で止める

## ループが肩代わりしてくれないこと ── うまく回るほど鋭くなる3つの問題

ループは強力ですが、Addy Osmani は同じ熱量で危険も語っています。彼が強調するのは、**「ループが良くなるほど、簡単になるどころか、むしろ鋭くなっていく」3つの問題** です。

* **検証は、依然としてあなたの仕事（Verification is still on you）**：無人で走るループは、無人でミスを犯すループでもあります。「できた」は主張であって、証明ではない。あなたの仕事は、動くと確認できたコードを出荷することだ ── Osmani はそう繰り返します。
* **理解の負債（comprehension debt）**：あなたが書いていないコードをループが速く出荷するほど、「実際に存在するもの」と「あなたが把握しているもの」の差が開く。滑らかに回るループは、放っておけばこの差を速く広げます。だから、あなたが読まなければならない。
* **認知の明け渡し（cognitive surrender）**：ループが勝手に回っていると、意見を持つのをやめ、返ってきたものをそのまま受け取りたくなる。ループを設計する行為は、判断を持って行えば「薬」になり、考えないために行えば「毒」になる ── 同じ行動で、正反対の結果です。

Osmani の印象的な指摘に、**「同じループを二人が回しても、正反対の結果になりうる」** というものがあります。一方は浮いた時間で難しい問題の理解を深め、もう一方は理解を避けるために使う。ループはその違いを知りません。知っているのはあなただけです。だからこそ、ループ設計はプロンプト設計より **簡単ではなく、むしろ難しい**。

Cherny の言葉の要点も同じです。**楽になったのではなく、重要な作業が移ったのだ** ── プロンプトを書くことから、ループを設計することへ。

Osmani の結びを借りるなら：**ループを作れ。ただし「ボタンを押す人」ではなく、「エンジニアであり続けるつもりの人」として作れ。**

そしてもう一つ、忘れてはいけないバランス感覚。**直接プロンプトを打つやり方も、いまなお有効です。** すべてをループに委ねる必要はありません。要は使い分けです。

## 理論を実装に落とす ── 実装例としての loop-kit

ここまで並べてきた原則 ── **検証を中心に据える／作る者と採点する者を分ける／記憶を外に置く／ハードな歯止めで暴走を止める** ── は、実際のコードではどんな形になるのか。先ほどの三つの警告のうち、loop-kit が正面から引き受けるのは最初の **「検証は依然あなたの仕事」**。**「できた」という主張を、仕組みの側で証明に変える** ── それを **ひとつの実装例** として形にしたのが、私が公開している **loop-kit** です。

大事なのは、これが机上の試作ではないこと。loop-kit は **もともと私自身が構築し、日々の実運用で使っているしくみ** を、特定プロジェクトへの作り込みを削ぎ落として汎用化・簡素化したものです。実運用に耐えることを前提に設計・実証されたものを、**誰でも簡単にループエンジニアリングを試せる** 形へ整え直しました。ひとことで言えば **「Claude Code を、一回のチャットではなく『停止条件を満たすまで作業サイクルを回し続けるしくみ』として動かす、薄いシェル製のハーネスとスキル群」** です。サーバーもデーモンも要らず、シェルスクリプトとプロンプトだけで動きます。

loop-kit が実装しているのは、4つのループ型のうち **ゴール型** です ── 仕様をひとつ固定し、緑になるまで回し切る。さきほどの「一回きりだが複雑な実装をループに任せる」使い方が、まさにこれにあたります。なぜここから始めるのか。**検証を中心に据えたゴール型こそ、今後あらゆるハーネスの共通の基本単位になる** と考えているからです。ゴール型はそれ自体が小さく完結した部品であり、時間型やプロアクティブ型のような外側のしくみ ── 決まった時刻に起動する、イベントに反応する ── は、この部品を呼び出して組み合わせることで成り立ちます。私が取り組んでいる自己改善型のループも、同じ根の上に積み上がります。入門の実装として、まずここを押さえるのが理にかなっているわけです。

### 一度要件を渡すと、何が起きるか

あなたが要件を一度渡してから、完了まではおおよそ次の順に進みます。

1. **契約をつくる** ── Claude Code が対話でリポジトリを調べ、**自分では埋められない「未知」** を質問しながら、要件・非目標・受け入れ条件・**成功条件（＝いつ止めてよいか）** を「契約」として書き出す。
2. **契約を承認する** ── あなたが承認した瞬間の内容がハッシュで固定され、以後こっそり成功条件を書き換えられなくなる。
3. **タスクに分ける** ── 契約に沿って実装計画を立て、**依存があるものは順に、独立なものは並列に**（それぞれ隔離された作業場所で）進める。
4. **ループを回す** ── 各タスクで、契約・計画・進捗を読んで1ステップだけ **実装** し、モデルを介さないスクリプトがあなたの **検証コマンドを自分で再実行** して確かめ（**検証**）、まっさらな文脈のレビュアーが要件を先・差分を後に読んで仕様からのズレや後退を点検する（**レビュー**）。成功条件を満たせば止まり、満たさなければ **毎回まっさらな文脈でこの輪を繰り返す**。反復・時間の上限に達するか、人間の判断が要る場面（取り返しのつかない操作、要件そのものの変更）に当たれば、そこで止めてあなたに渡す。
5. **証拠を出す** ── 完了したら、差分の山ではなく **「何をどう検証したか」の証拠レポート** を出力する。並列で走った分は、マージした全体をもう一度、最初の契約に照らして最終レビューする。

心臓部は主に 4 つのループです：

> **実装 →（外部評価器で）検証 →（独立レビュアーで）レビュー → 満たせば停止 / 未達なら実装へ戻る**

ポイントは、**輪を止めてよいと決めるのが、作業役の自己申告ではない** こと。判定するのは、検証コマンドを機械的に再実行する外部評価器と、別文脈の独立レビュアーです。これが前半で触れた `/goal` との違い ──「判断の独立性」だけでなく **「検証の独立性」まで踏み込む** ── の正体です。

この流れの各所には、前半で挙げた原則がそのまま埋め込まれています ── **契約は固定・実装は自由／自己採点の禁止／記憶は外に置き毎回まっさら／ハードな歯止めで暴走を止める／人間は差分ではなく証拠を見る**。

loop-kit は、**最新のループエンジニアリングの理論に合わせて今後も改善していく可能性があります。** 具体的な使い方・設定・コマンド、そして現時点の実装の詳細は、つねに最新のリポジトリの README を参照してください。試す前に2点だけ。既定で **Opus** を使う前提なので、プランによっては軽いモデルへの振り替えが要ります。また要件によっては反復回数が増え、トークン消費も伸びるため、まずは小さく低リスクなタスクから回すのがおすすめです。

## おわりに

ループエンジニアリングは、2026年6月に一気に注目を集めた、AI時代のエンジニアリングの本質的な変化です。ただの「自動化」ではありません。**「どこまで判断を自分に残し、どこを仕組みに委ねるか」を設計する** 一段上のスキルです。

まだ黎明期で、慎重であるべき点も多い。トークンコストは油断すると膨らみ、理解の負債は静かに積み上がります。だからこそ、**低リスクなパターン（報告だけの仕分けなど）から一つ試し、一度自分の手で回して、露呈した弱点を締め直す** ── そこから始めるのがおすすめです。loop-kit も、その「まず一度回してみる」ための入口として使ってもらえればと思います。

道具に抵抗するのでも、無批判に明け渡すのでもなく。判断と、システムを見通す力を磨くこと。それが、この新しい階層でも「エンジニアであり続ける」ということだと思います。
