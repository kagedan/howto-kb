## はじめに

Claude Codeの公式ドキュメントに「Best Practices」ページが公開されています。

Anthropic社内チームや外部エンジニアの実践から得られた知見がまとめられたこのガイドは、Claude Codeを使いこなすための必読資料です。

本記事では、公式ガイドの内容を日本語で整理し、実務で役立つポイントを解説します。

---

## 大前提：全てはコンテキストウィンドウの管理に帰結する

公式ガイドは冒頭でこう述べています。

> Most best practices are based on one constraint: Claude's context window fills up fast, and performance degrades as it fills.  
> （ほとんどのベストプラクティスは1つの制約に基づく：コンテキストウィンドウはすぐ埋まり、埋まるほどパフォーマンスが落ちる）

ファイル読み込み、コマンド出力、会話履歴——全てがコンテキストを消費します。1回のデバッグセッションだけで数万トークンを消費することもあります。

**この「コンテキストは有限の最重要リソース」という認識が、以降の全てのプラクティスの根底にあります。**

---

## 鉄則1：検証手段を与える（最もレバレッジが高い）

公式ガイドが**最も重要**と位置づけているのがこれです。

> This is the single highest-leverage thing you can do.

Claudeに「自分の仕事を自分で検証させる」仕組みを渡すことで、出力品質が劇的に向上します。

### 具体例

**悪い例：**

```
メールバリデーション関数を実装して
```

**良い例：**

```
validateEmail関数を書いて。
テストケース: user@example.com → true、invalid → false、user@.com → false
実装後にテストを実行して確認して
```

**UIの場合：**

```
[スクリーンショット貼付]
このデザインを実装して。結果のスクショを撮って元画像と比較し、差分を修正して
```

テストケース、リンター、Bashコマンド——何でもいいので「検証が自動で回る仕組み」を渡しましょう。検証手段を決定論的な処理にする投資は必ず回収できます。

---

## 鉄則2：探索→計画→実装→コミットの4フェーズで進める

いきなりコーディングさせると、間違った問題を解いてしまうリスクがあります。

### 推奨フロー

**Phase 1 - 探索（Plan Mode）**

```
src/auth を読んで、セッション管理とログインの仕組みを理解して。
環境変数でシークレットをどう管理しているかも確認して。
```

また探索中のコンテキスト節約テクニックには以下があります。トークンに余裕がある場合は、agent teamsで探索を行わせるのもありでしょう。

* サブエージェントに調査を委任して、メインコンテキストを汚さない
* 探索結果を確認したら  
  /clearして実装フェーズへ（クリーンなコンテキストで再開）
* /btw で一時的な質問（履歴に残さない）
* ultrathink をプロンプトに含めると、その1回だけ深い推論を実行

**Phase 2 - 計画（Plan Mode）**

```
Google OAuthを追加したい。どのファイルを変更する必要がある？
セッションフローの計画を作成して。
```

Ctrl+Gで計画をエディタで直接編集できます。

**Phase 3 - 実装（Normal Mode）**

```
計画に沿ってOAuthフローを実装して。
コールバックハンドラのテストを書いて、テストスイートを実行して失敗を修正して。
```

**Phase 4 - コミット**

```
descriptiveなメッセージでコミットしてPRを作成して
```

### いつPlanを使うべきか？

タイポ修正やログ追加のような小タスクにはPlan不要です。  
計画が効くのは：

* アプローチが不確実な時
* 複数ファイルを横断する変更
* 触ったことのないコードを修正する時

> If you could describe the diff in one sentence, skip the plan.  
> （diffを一文で説明できるなら、計画はスキップして良い）

## 鉄則3：プロンプトに具体的なコンテキストを含める

曖昧なプロンプトは修正回数を増やし、コンテキストを無駄に消費します。

### 4つの戦略

| 戦略 | Before | After |  
|------|--------|-------|  
| **スコープを絞る** | 「foo.pyのテスト追加」 | 「foo.pyのユーザーがログアウトしている場合のエッジケーステストを書いて。モック不要」 |  
| **情報源を示す** | 「ExecutionFactoryのAPIなぜ変？」 | 「ExecutionFactoryのgit履歴を見てAPI変遷を要約して」 |  
| **既存パターンを参照** | 「カレンダーウィジェット追加」 | 「HotDogWidget.phpのパターンに従って新カレンダーウィジェットを実装して」 |  
| **症状を具体的に** | 「ログインバグを修正して」 | 「セッションタイムアウト後にログインが失敗する。src/auth/のトークンリフレッシュを確認して」 |

## 鉄則4：環境を整備する

`/init`で自動生成→継続的に改善が推奨フローです。

**含めるべきもの：**

* Claudeが推測できないBashコマンド
* デフォルトと異なるコードスタイル
* テスト方法とテストランナー
* ブランチ命名規則やPR規約
* プロジェクト固有のアーキテクチャ判断
* 環境の癖（必要な環境変数など）

**含めないもの：**

* コードを読めば分かること
* 標準的な言語規約
* 詳細なAPIドキュメント（リンクで十分）
* 頻繁に変わる情報
* 「きれいなコードを書け」のような自明なこと

> If Claude keeps doing something you don't want despite having a rule against it, the file is probably too long and the rule is getting lost.  
> （ルールを書いたのにClaudeが従わない場合、ファイルが長すぎてルールが埋もれている）

`@path/to/file`で他ファイルをインポート可能。配置場所は`~/.claude/CLAUDE.md`（グローバル）、`./CLAUDE.md`（チーム共有）、`./CLAUDE.local.md`（個人用）。

### 権限設定

3つのアプローチがあります：

* **Autoモード**：分類モデルが危険なコマンドだけブロック
* **Permission allowlist**：`/permissions`で安全なコマンドを登録
* **サンドボックス**：OS レベルのファイルシステム・ネットワーク分離

大規模な会社であれば、サンドボックス環境を作り、それをMDMで配布するというやり方もありでしょう。メルカリでそういった事例が紹介されていましたね。

### CLIツール活用

`gh`、`aws`、`gcloud`、`sentry-cli`等をClaude Codeに使わせましょう。未知のCLIでも`Use 'foo-cli --help' to learn about foo tool, then use it to...`で学習させられます。

### Hooks

CLAUDE.mdの指示はアドバイザリーですが、Hooksは**確定的に毎回実行**されます。例外なく実行すべきアクションはHookで設定しましょう。

### Skills

`.claude/skills/`に`SKILL.md`を配置。関連時に自動適用されるか、`/skill-name`で直接呼び出せます。CLAUDE.mdは毎セッション読み込まれるので肥大化しがちですが、Skillsは必要な時だけ読み込まれます。

### カスタムサブエージェント

`.claude/agents/`に定義ファイルを配置。独自のツール・モデルを指定でき、メインコンテキストを汚さずに専門タスクを処理できます。

## 鉄則5：効果的にコミュニケーションする

### コードベースの質問

シニアエンジニアに聞くように直接質問すればOKです：

### インタビュー手法（大きな機能向け）

大きな機能を作る時は、Claudeに**こちらをインタビューさせる**のが効果的です。

```
[簡単な説明]を作りたい。AskUserQuestionツールを使って詳しくインタビューして。

技術的実装、UI/UX、エッジケース、懸念事項、トレードオフについて聞いて。
明らかな質問ではなく、自分が考えていないかもしれない難しい部分を掘り下げて。

全てカバーしたらSPEC.mdに完全な仕様を書いて。
```

仕様完成後は**新しいセッション**で実装に入ります。クリーンなコンテキストで実装に集中できます。

## 鉄則6：セッションを積極的に管理する

### 軌道修正は早めに

> If you've corrected Claude more than twice on the same issue in one session, the context is cluttered with failed approaches. Run `/clear` and start fresh.  
> （同じ問題を2回以上修正したら、コンテキストは失敗したアプローチで汚染されている。`/clear`してやり直す）

### コンテキスト管理

### サブエージェントで調査を委任

調査タスクはサブエージェントに任せると、メインコンテキストを汚しません。

```
サブエージェントを使って、認証システムのトークンリフレッシュの仕組みと
既存のOAuthユーティリティがあるか調査して。
```

実装後のレビューにも使えます：

```
サブエージェントを使ってこのコードのエッジケースをレビューして
```

### セッション再開

```
claude --continue    # 最新セッション再開
claude --resume      # セッション一覧から選択
```

`/rename`で分かりやすい名前をつけておくと後から見つけやすくなります。

## 鉄則7：自動化してスケールさせる

### 非対話モード

```
claude -p "このプロジェクトが何をするか説明して"
claude -p "全APIエンドポイントをリストして" --output-format json
claude -p "このログファイルを分析して" --output-format stream-json
```

CI/CDパイプライン、pre-commitフック、スクリプトに組み込めます。

### 並列セッション（Writer/Reviewerパターン）

| セッションA（Writer） | セッションB（Reviewer） |  
|---|---|  
| `APIエンドポイントのレートリミッターを実装して` | |  
| | `src/middleware/rateLimiter.tsのレートリミッター実装をレビューして。エッジケース・競合状態・既存パターンとの一貫性を確認して` |  
| `レビューフィードバック：[Bの出力]。これらを修正して` | |

新鮮なコンテキストでレビューするため、自分が書いたコードへのバイアスがありません。

### ファンアウト（大量ファイル処理）

```
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

最初の2〜3ファイルで試して、プロンプトを洗練させてから全体に実行します。

## 鉄則8：よくある失敗パターンを避ける

### キッチンシンクセッション

1つのセッションに無関係なタスクを混ぜる。コンテキストが無関係な情報で埋まる。

→ **`/clear`でタスク間をリセット**

### 修正の繰り返し

修正→まだ違う→再修正...の繰り返しでコンテキストが汚染される。

→ **2回失敗したら`/clear`して、学んだことを含めた新プロンプトで再開**

### CLAUDE.mdの肥大化

長すぎてルールが無視される。重要な指示がノイズに埋もれる。

→ **ルールを定期的に削減。Claudeが既にできていることは削除、確定的に実行すべきものはHookに移行**

### 信頼→検証ギャップ

もっともらしい実装だがエッジケースが未対応。

→ **必ず検証手段を提供。検証できないなら出荷しない**

### 無限探索

スコープを絞らず「調査して」と指示。数百ファイルを読んでコンテキスト消費。

→ **範囲を狭めるか、サブエージェントに委任**

---

## 鉄則9：直感を育てる

公式ガイドの最後は、ルールに固執しすぎないことへの言及で締めくくられています。

> Pay attention to what works. When Claude produces great output, notice what you did: the prompt structure, the context you provided, the mode you were in.  
> （何がうまくいったかに注意を払う。良い出力が得られた時、自分が何をしたかに注目する：プロンプト構造、提供したコンテキスト、使っていたモード）

ルールは出発点であり、状況に応じた判断力を磨くことがゴールです。

## まとめ

Claude Code公式ベストプラクティスの9つの鉄則を整理しました。

| # | 鉄則 | 一言 |  
|---|------|------|  
| 1 | 検証手段を与える | テスト・スクショ・期待出力を渡す |  
| 2 | 探索→計画→実装 | いきなりコーディングさせない |  
| 3 | 具体的コンテキスト | ファイル・パターン・症状を明示 |  
| 4 | 環境を整備する | [CLAUDE.md](http://CLAUDE.md)・Hooks・Skills・サブエージェント |  
| 5 | 効果的なコミュニケーション | インタビュー手法で仕様を固める |  
| 6 | セッション管理 | `/clear`を頻繁に、サブエージェント活用 |  
| 7 | 自動化・スケール | 並列セッション・ファンアウト |  
| 8 | 失敗パターン回避 | 2回ミスしたら`/clear`してやり直す |  
| 9 | 直感を育てる | ルールは出発点、判断力を磨く |

全てに共通する根底の原則は**コンテキストウィンドウは最重要リソースであり、その管理がパフォーマンスを決める**ということです。
