こんにちは、クラシル社で iOS エンジニアをしている zhu tianren です。

> きっかけ：バージョンアップのたびに、手触りが変わった
>
> 最近、クラシルiOSアプリの大規模リファクタリングを実施している。10年間の運用で積み上がった技術的負債を清算し、混乱したアーキテクチャを整理することが目標だ。このプロセスで、Claude Code と Codex をおよそ 6:4 の割合で併用してきた。
>
> この問題に気づいたきっかけは、それぞれのツールが新バージョンにアップデートされたときの変化だった。Opus 4.6 から 4.7 へ、GPT-5.4 から 5.5 へのアップグレードを経て、何かが変わったと感じた。良くなった・悪くなったということではなく、それぞれのツールの「性格」がより鮮明になり、両者の違いが以前よりはっきりと感じられるようになった。
>
> 最初は Claude Code と Codex というツール自体の実装の違いだと思っていた。しかし [ARC-AGI-3 のベンチマーク分析](https://arcprize.org/blog/arc-agi-3-gpt-5-5-opus-4-7-analysis) を読んで、これはベースモデル自体の「思考の仕方」の違いかもしれないと考えるようになった。

# 一、DFS エキスパート vs BFS エキスパート

アルゴリズムで例えるのが、自分には一番しっくりくる。

* Claude Code + Opus 4.7 は、DFS エキスパートに近い。

一つの仮説を素早く見定め、そこに向かって強い実行力で突き進む。仮説が正しいときは圧倒的に強い——ほとんど手を添えなくても、高い品質でゴールまで走り切ってくれる。

問題は、仮説が間違っていたときも同じ実行力で突き進むことだ。10年動いてきた iOS コードベースをリファクタリングするとき、これが怖かった。過去のコンテキストを十分に理解しないまま、「もっともらしい」実行理論を素早く形成し、大量の書き換えを始めることがある。その前提が、往々にして正確ではない。

* Codex + GPT-5.5 は、BFS エキスパートに近い。

調査やアーキテクチャ議論の場面では圧倒的に優れている。エッジケース、設計上のトレードオフ、レガシーコードへの潜在的影響——こういったものを自然に広げてくれる。技術的なディスカッションは、上級エンジニアとレビューしているような感覚だ。

ただ、独特の失敗パターンがある。正しい方向性を言語化できても、それを実行プランに落とし込めないことがある。一度正しい方向を見つけても、また別の見慣れたフレームワークで問題を解釈し直そうとする——つまり、仮説空間を何度も「開き直す」。気づいたときには、いくつかの可能性の間を行き来したまま、実装は一向に進んでいないことがある。

# 二、ARC-AGI-3 の分析が、理論的な裏付けをくれた

正直に言うと、最初はこの手触りの違いをベースモデルの問題として捉えていなかった。

Claude Code と Codex は、ハーネスの実装レベルで大きく異なる——コンテキストの管理方法、ツール呼び出しの戦略、長いタスクの中間状態の扱い方……どの実装の差異も、最終的な結果に大きな影響を与えうる。だから最初は「これはツールのラッパー層の違いによる挙動の差であって、ベースモデル自体の問題ではない」と判断していた。

ARC-AGI-3 の分析レポートを読んで、その前提を見直すことになった。

レポートでは、二つのモデルが未知のロジックパズル環境を解く様子を観察し、まったく見たことのないルールの中でどのように仮説を立て、認識を更新するかを追っている。最終スコアは一つのデータポイントに過ぎず、より重要なのは**失敗の仕方が全く異なる**という点だ。

レポートの核心はこうだ：Opus 4.7 は「wrong compression（誤った圧縮）」、GPT-5.5 は「failure to compress（圧縮の失敗）」。

Opus は観察した情報を素早く実行理論に圧縮し、積極的に動き出す。問題はその理論が間違っていても止まらないことだ。GPT-5.5 は逆のパターンで、正しい方向を明確に言語化できても、それを実行可能なプランに圧縮できず、何度も解釈し直す。

これは自分がリファクタリングで感じた手触りと高度に一致する。そしてこの発見から、自分が観察していた挙動の差異は、Claude Code と Codex のツールレベルの差異だけでなく——ベースモデル間に、もともとこのような思考の違いがあるのかもしれないと気づいた。

一点補足しておくと、このレポートはどちらのモデルが優れているかを評価するものではない。両モデルの ARC-AGI-3 のスコアは極めて低く、このベンチマーク自体が「真に未知の環境における適応と推論」能力を測るために設計されたものだ。レポートの価値は、**二つのモデルの失敗の仕方の違いを可視化した点**にある。

# 三、開発者として、実際にどう使っているか

この違いを理解してから、二つのツールの使い方を意識的に変えた。

## Claude Code + Opus 4.7：序盤に投資するほど、後半が楽になる

現在は Claude Code をメインに使っているが、序盤に開発者側が求められるものは多い。

自分の実際のフローはこうだ：まず十分なコンテキストを渡し、段階的に問題を理解させる。提示されたプランをレビューし、各ステップが自分の意図と一致しているかを確認する。そこまで対話してはじめて、オートモードで走らせる。このプロセスを踏むと、最終的な結果と当初の期待がそれほど乖離しない。

それでも結果がズレた場合は、どのステップで方向がおかしくなったかを確認したうえで、**タスクのスコープを縮小する**。広い範囲で再計画させるより、短いパスで動かす方が成功率が上がる——パスが短いほど、誤った仮説が入り込む余地が減る。

この使い方の本質は、**開発者の判断を前倒しにすること**だ。序盤のレビューとガイドに余分にコストをかけることで、後半の実行フェーズでの介入を減らせる。

## Codex + GPT-5.5：調査フェーズは得意、実装フェーズは付き添いが必要

Codex の調査・議論フェーズでの価値は本物だ。複雑な問題のあらゆる側面を照らし出してくれる。これが自分が使う主な理由だ。

ただ実装フェーズに入ると、画面を張りつきで見ている時間が増える。各コード変更が、最初に合意した方向からズレていないかを逐一確認し、ズレが見つかれば止めて再ガイドする。この「止めて再ガイドする」頻度が、Claude Code を使うときより明らかに高い。

## 複雑なタスクでの組み合わせ

特に複雑なリファクタリングでは、二つのツールの強みを分けて使うことがある。まず Codex で調査し、BFS 的な探索力を活かして問題の全体像を把握する。次に自分がその結果を整理・圧縮して明確な方針を作る。そしてその方針を Claude Code に渡して実装させる。

これはすべての場面で使うわけではなく、タスクの複雑さがこの段階的な投資に見合うときだけ意味がある。ただ、背後にある考え方は明確だ——それぞれのツールが本当に得意なフェーズだけを担当させる。

# おわりに

このレポートはどちらが優れているかを論じるものではなく、この記事も同じだ。

伝えたいのは、二つのツールが開発者に求めるものは根本的に異なるということだ。Claude Code は序盤に判断力を集中させることを求め、Codex はプロセス全体を通じた継続的な関与を求める。どちらもコストは本物で、ただフェーズが違う。

そして、自分がこの差異に気づいたきっかけ自体が示唆的だ。同じシリーズのモデルでも、Opus 4.6 から 4.7、GPT-5.4 から 5.5 への更新だけで手触りが変わり、両者を並べて使うとその差はさらに顕著になった。モデルは継続的にアップデートされ、その「性格」も変わっていく。開発者として、この差異への感度を持ち続けることが必要だ——今日まとめた使い方が、次のバージョンでは見直しが必要になるかもしれない。
