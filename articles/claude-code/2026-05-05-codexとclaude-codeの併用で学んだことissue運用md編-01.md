Codex（GPT-5.5）とClaude Code（Ops4.6 / Sonnet4.6）を併用している中で、  
Issueとコンテキストファイル（CLAUDE.md / AGENTS.md）の使い方が重要だと感じたため、備忘録として記載します。

## 前提

AIコーディングエージェントは、基本的に「毎回コンテキストを渡さないと何も知らない状態」から始まります。

そのため、

* 毎回プロンプトで説明する
* コンテキストファイルに書いておく

のどちらかが必要になります。

このとき、以下の役割分担にすると安定します。

* Issue：今回やること
* AGENTS.md / CLAUDE.md：前提ルール
* プロンプト：その場の指示

## Issueは「指示書」として使う

Issueは単なるタスク管理ではなく、  
**AIに渡す前提の指示書**として書くと扱いやすくなります。

例えば以下のような構成にします。

* 背景
* 目的
* 要件
* スコープ（やること / やらないこと）

これを元に、まずCodexに設計させます。

## 基本フロー

1. Issueを作成する
2. Codexに「実装設計」を作らせる
3. Claude Codeに実装させる

この流れにすると、

* Codex：考える
* Claude Code：実装する

という役割分離ができます。

## AGENTS.md / CLAUDE.mdの役割

`AGENTS.md`と`CLAUDE.md`をプロジェクトのルートディレクトリに配置することで、  
AIエージェントは、リポジトリ内の特定ファイルを自動で読み込みます。

* Codex → `AGENTS.md`
* Claude Code → `CLAUDE.md`

どちらも「プロジェクト前提」を書くファイルです。

例えば以下のような内容を記載します。

* 技術スタック
* ディレクトリ構成
* コーディング規約
* 変更してよい範囲
* 禁止事項

これにより、毎回の指示を短くできます。

AGENTS.mdはクロスツールで使われる標準に近い形式で、  
複数のエージェントで共通利用できるのが特徴です。

## 運用のコツ

### 1. 情報を分離する

* Issue：今回の作業
* AGENTS.md / CLAUDE.md：不変ルール
* プロンプト：補足

これを混ぜないようにします。

### 2. 書きすぎない

* 無関係なルールが増える
* トークン消費が増える
* 判断がブレる

必要最低限に絞る方が安定します。

### 3. 1PR単位に分割する

Issueも実装も、1PRで完結する粒度にすると扱いやすくなります。

* コンテキストが小さくなる
* 修正範囲が限定される
* AIの迷いが減る

## まとめ

* Issueは「AIに渡す指示書」として書く
* 設計（Codex）と実装（Claude Code）を分離する
* AGENTS.md / CLAUDE.mdで前提を固定する

この3つを揃えると、

* 指示が短くなる
* 手戻りが減り、やり取りがスムーズになる
* 消費効率が安定する

という形で効いてきます。

## おまけ - CodexとClaude Codeへの指示文

### Codex用

```
以下のGitHub Issueを対応するための「実装設計」だけ作成してください。
まだコード修正はしないでください。

対象Issue:
#xxx

目的:
- Issueの要件を満たす
- 変更範囲を必要最小限にする
- Claude Codeに実装を引き継げる粒度に分解する

出力してほしい内容:
1. Issueの要約
2. 対応方針
3. 変更対象ファイル一覧
4. ファイルごとの修正内容
5. 実装順序
6. 動作確認方法
7. 注意点・壊れやすい箇所
8. Claude Codeに渡すための実装指示文

制約:
- 実装はしない
- 不明点は推測せず「要確認」と書く
- 変更対象を広げすぎない
- 1PRで対応できる範囲に収める
```

### Claude Code用

```
以下はCodexが作成した実装設計です。
この設計に従って実装してください。

重要:
- 指定された対象ファイル以外は変更しないでください
- 余計なリファクタリングはしないでください
- Issue対応に必要な最小差分にしてください
- 判断に迷う場合は実装せず、確認事項として止めてください
- 実装後、変更ファイルと確認方法を簡潔に報告してください

対象Issue:
#xxx

Codexの実装設計:
---
ここにCodexのアウトプットを貼る
---
```
