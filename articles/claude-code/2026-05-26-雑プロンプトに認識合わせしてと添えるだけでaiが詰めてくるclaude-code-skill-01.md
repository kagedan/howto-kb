## 対象読者

* AI コーディングツールに依頼するとき、毎回プロンプトを丁寧に書くのが面倒な人
* 雑に投げると AI が迷走することは知っているが、前提・制約を毎回書き出したくない人
* Claude Code を使ったことがある人

「面倒だから雑に投げる」→「AI が暴走する」→「結局丁寧に書き直す」のループに疲れた人向けの記事である。

---

## 背景: 丁寧に書くのも雑に投げるも、どっちもしんどい

Claude Code は便利だが、依頼が雑だと AI が解釈を埋めて勝手に走り出す。  
意図と違う方向に進んで手戻りになるか、安全側に倒れて何もしないかのどちらかになりやすい。

これを防ぐためにユーザー側でプロンプトを丁寧に書こうとすると、  
毎回「目的」「制約」「成果物」「影響範囲」「許容しないこと」を書くことになる。  
書く時間がかかるうえに、書いている最中に何を伝えたかったか忘れる。

結局、「丁寧に書くのも雑に投げるも、どっちもしんどい」のである。

そこで、**プロンプトを雑に投げる → AI が選択肢つきで質問してくる → 選ぶだけで意図が固まる**、という運用に振り切った。  
意図を言語化する負荷を、AI 側に移譲する形だ。

仕組みは `ask-question` という Claude Code Skill にまとめた  
（はじめは `CLAUDE.md` 直書きだったが、コンテキスト圧迫を嫌って Skill に切り出した）。

---

## 「認識合わせして」と添えるだけ

### 日常フレーズに格上げした

`ask-question` のトリガーには「**認識合わせして**」「詰めて」「ask-question」の3つを登録している。  
実際に使うのはほぼ「認識合わせして」だ。

依頼の冒頭または末尾に「認識合わせして」と付け足すだけで、Skill が起動し、AI 側で決定ツリーを上流から1問ずつ詰めてくる。  
プロンプト本体は雑でいい。

これで AI 側が  
「ログレベルの粒度は？」  
「出力先は？」  
「フォーマットは？」  
「ローテーション方針は？」  
を順に問うてくる。

あとは選択肢を選ぶだけ。

### 業務ドメインでも同様

特定の業務ドメイン（決済処理・バッチ集計など、ドメイン固有の制約が多い領域）でも同じ運用で回している。  
AI がそのドメインを知らない分、選択肢が「外していること」もある。  
そういうときは Other（自由入力）で補足するか、選択肢を選んだうえで補足条件を渡せばよい。

完璧な選択肢を期待するのではなく、**選択肢を起点に対話を進める**のがコツだ。

### 「雑プロンプト」を意図的に選ぶ理由

頭のなかの意図をすべて言語化するのは高コストだ。  
書いている最中に何を伝えたかったか忘れることもある。

雑プロンプトを投げて AI に質問させると、ユーザーは「問われたときに考える」だけで済む。  
最初から完璧に意図を言語化するより、**質問されたタイミングで部分的に判断するほうが認知負荷が低い**。

これは「人間が AI に教える」協業ではなく、「AI が人間から引き出す」協業モデルだ。  
次節で触れる grill-me という公開 Skill が話題になったのも、同じ感覚を多くの人が持っていたからだと思う。

---

## grill-me スキルを知っておく

`ask-question` Skill は [mattpocock/skills の grill-me](https://github.com/mattpocock/skills/blob/main/skills/productivity/grill-me/SKILL.md) を下敷きにしている。

grill-me は AI 界隈で話題になった Claude Code 向けの公開 Skill で、  
「ユーザーの計画を徹底的に問い詰めて、共通認識に到達するまで質問を重ねる」ことだけに特化している。  
リポジトリで配布されているので、ファイルを自分の `~/.claude/skills/` に置けばそのまま動く。  
話題になっていたのを見て採用を検討した。

`ask-question` は grill-me を下敷きに、AskUserQuestion を積極的に使うよう Skill 内で誘導しつつ、  
日常運用のトリガーワード（「認識合わせして」など）を足したものだ。

---

## AskUserQuestion: 選択肢つき質問という道具

### AskUserQuestion とは

AskUserQuestion は Claude Code 標準の「選択肢つきで質問するためのツール」だ。  
AI が平文で「どうしますか？」と尋ねる代わりに、

を構造化して投げる。  
ユーザー側は選択肢から選ぶだけで返答が完結する。  
「Other（自由入力）」が自動で追加される仕様なので、想定外の方針もユーザー側から渡せる。

実際の使われ方を、テキストモックで示すとこうなる。

```
Q: ログ出力先はどれにしますか？
  [1] 標準出力 + ファイル (Recommended)
      → ローカル確認と永続化を両立。最も無難
  [2] 標準出力のみ
      → コンテナ環境で外部ログ基盤に流す前提
  [3] ファイルのみ
      → 標準出力を別用途に使う場合
  [4] Other（自由入力）
```

AI 側で「選択肢を列挙する」「推奨案を提示する」までが済んでいるので、ユーザーは選ぶだけで意思決定できる。  
`ask-question` が日常運用に乗るのは、この道具のおかげで意思決定の段差が一気に下がるからだ。

具体的な効きどころは次の3つ。

* **ユーザーの認知負荷が下がる**: 平文で「どうしますか？」と聞かれると、ユーザーは選択肢を頭の中で列挙してから返事を考える。選択肢提示なら、列挙はすでに AI 側で済んでいて、選ぶだけで済む。
* **推奨案つきで問える**: 各選択肢には `description`（その案を選んだときに何が起きるか・トレードオフ）が書ける。先頭に `(Recommended)` 付きの推奨案を置けば、ユーザーは「推奨でいいか、そうでないなら何か」だけ判断すればよい。
* **「Other」で逃げ道がある**: ツール仕様として「Other（自由入力）」が自動で追加される。AIが想定していない方針もユーザーが渡せるので、推奨案を強く提示してもユーザーの裁量は失われない。

### 効かないケース

`AskUserQuestion` は **1問あたり選択肢が2〜4個まで** という制約がある。  
網羅性が必要な質問（例: API候補が30個ある）には向かない。  
そういう質問は Skill 側で「コードを先に読んで、有力候補を3つに絞ってから提示」のような前処理が要る。

---

## 運用してみての効果と学び

数値で測ったわけではないが、体感での効果は次のとおり。

* **依頼から実装着手までの時間が短くなった**: プロンプトを書く時間が削れた分が大きい
* **手戻りが減った**: AI が「重要な分岐」を取りこぼさず質問してくれる
* **ドキュメント化が副作用で進む**: Skill が決定事項を Markdown でまとめるので、そのまま設計メモになる

うまくいかないケースもある。

* **自明な依頼でも質問してくることがある**: 「README の typo を直して」のような明白な依頼でも、たまに確認されてうざい。Skill 内に「自明な依頼は確認不要」と書いているが、判定は AI 任せなのでブレる
* **質問のテンポが重いと感じる人もいる**: チームに導入するなら、個人の好みに合わせて on/off できるようにしたほうがよさそう

---

## まとめ

* 雑プロンプトに **「認識合わせして」と添えるだけ** で AI が意図を引き出してくれる
* 完璧に言語化するより **問われたタイミングで判断する** ほうが認知負荷が低い
* **AskUserQuestion の選択肢提示** で意思決定の段差が一段下がる

意図の言語化を AI 側に移譲する運用は、慣れると戻れない。プロンプトを丁寧に書く時間がそのまま「思考の余白」になる。

description: |

  決定ツリーの各分岐を1つずつ解決し、依存関係を順番に潰していく。
  曖昧さや暗黙の前提がある状態で実装に進むのを防ぐ。

  - 「詰めて」

# ask-question

ユーザーの計画・設計・要件を**徹底的に**インタビューし、共通認識に到達するまで質問を重ねるスキル。

## 進め方

計画の全側面を1つずつ詰めていく。設計上の決定は依存関係のあるツリー構造になっているので、上流の決定から順番に解決し、下流の分岐に進む。各質問では**あなたの推奨回答（と理由）を必ず提示**してから問う。

質問は**必ず1度に1つだけ**。ユーザーが認知過負荷にならないよう、回答に応じて次の質問を組み立てる。

## 質問は AskUserQuestion ツールで行う

このスキルが起動している間、ユーザーへの確認は**必ず `AskUserQuestion` ツール**で行う。平文で「Aですか？Bですか？」と並べて聞くのは禁止。理由:

- 構造化された選択肢にすることで、ユーザーの判断負荷が下がる
- 推奨案を選択肢の先頭に置き、ラベル末尾に「(Recommended)」を付けると意思決定が速くなる
- 「Other」で自由入力もできるため、想定外の意図も拾える

各質問について:

- 選択肢は2〜4個に絞る（多すぎると選びにくい）
- 推奨案を**先頭**に置き、ラベルに `(Recommended)` を付ける
- 各選択肢の `description` に「選んだ場合に何が起きるか・トレードオフ」を1〜2行で書く
- 複数選択が自然な観点（例: 「どの機能を含める？」）では `multiSelect: true` を使う
- `header` は12文字以内のチップ表示用ラベル（例: 「DB方式」「認証方式」）

複数の論点が並列で独立しているときは、`AskUserQuestion` の `questions` 配列に最大4つまでまとめてよい。ただし依存関係がある場合は1問ずつ。

## コードベースで答えられる質問はコードを読む

「現状の実装はどうなっているか」「このクラスはどこで使われているか」「設定値の現在の値は何か」など、**コードベースの探索で答えが出る質問はユーザーに聞かない**。Grep / Read / Glob で先に調べ、判明した事実を前提として次の質問に進む。

ユーザーの時間は有限。AI で完結できる調査を肩代わりしてもらうのは無駄。

## 何を詰めるか（観点リスト）

依頼内容に応じて取捨選択。すべて聞く必要はないが、重要な分岐を見逃さない。

- **目的・ゴール**: なぜそれをやるのか、成功条件は何か
- **スコープ**: 何を含め、何を含めないか
- **影響範囲**: グローバル/プロジェクト/モジュール/ファイル単位
- **成果物の形式**: ファイル形式、出力先、命名規則
- **実装方針**: アーキテクチャ、設計パターン、責務分割
- **技術選定**: ライブラリ、フレームワーク、ミドルウェア
- **データモデル**: エンティティ、関係、永続化方式
- **エラー処理方針**: 例外の扱い、リトライ、ロールバック
- **非機能要件**: 性能、可用性、セキュリティ、運用
- **互換性**: 既存資産への影響、後方互換性の要否
- **テスト戦略**: 単体/統合/E2E、カバレッジ目標
- **リリース方針**: 段階展開、フィーチャーフラグ、ロールバック手順

## 終了条件

- ユーザーが「もう十分」「実装に進んで」と明示した
- 残る分岐がすべて些末で、実装中に判断できる粒度になった
- ユーザーが疲れてきた様子が見える（質問のテンポを落とすか、まとめに移る）

## 最終アウトプット

詰め切ったら、決まったことを Markdown の箇条書きでまとめて提示する。後続の実装ステップ（プラン作成、コーディング、ドキュメント化など）に直接渡せる粒度で。

## 設計上の注意

- **推測で進めない**: 「たぶんこうだろう」で次に進むと手戻りの元。曖昧な点は必ず確認する
- **誘導しない**: 推奨案は明示するが、ユーザーが別案を選んでも受け入れる
- **本気で詰める**: 表面的な合意で終わらせず、その判断が他の決定とどう絡むかまで踏み込む
- **テンポを維持**: 1問が重すぎると進まない。サクサク聞ける粒度に分解する
