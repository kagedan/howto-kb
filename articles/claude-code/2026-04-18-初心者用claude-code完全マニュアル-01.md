## Claude Codeではじめる副業半自動化

**インストールから、AIエージェント設計まで。非エンジニアのための一冊。**

---

## はじめに

このマニュアルは、「プログラミングはできないけど、AIで副業を効率化したい」という方のために書きました。

世の中にはClaude Codeの紹介記事がたくさんあります。でもそのほとんどは「エンジニア向け」です。「npmを入れてターミナルで…」と言われた瞬間に、手が止まってしまう。私を含め、そういう方は多いのではないでしょうか。

このマニュアルの立ち位置はシンプルです。

* **ターミナルを触ったことがない方**が、最後まで読めば使えるようになる
* **インストールで止まらない**ように、つまずきポイントを全部先回りする
* インストール後の「で、どう使うの?」まで、実務レベルで解説する
* [**CLAUDE.md**](http://CLAUDE.md)**、skill、エージェント**という上級者が使っている仕組みを、噛み砕いて説明する

ひとことで言うと、「自分が今からClaude Codeを始めるとしたら、こういうマニュアルがあったら助かった」というものを書きました。

### この教材で得られるもの

読み終わったあなたは、次のことができるようになります。

### この教材の前提条件

**プログラミング経験は一切必要ありません。** ターミナルすら開いたことがなくても大丈夫です。全部、画面で実際に起きることを想定しながら書いています。

それでは、はじめましょう。

## 第1章 Claude Codeとは何か

## 1-1. ChatGPTやClaude.aiと、何が違うのか

まず最初に、一番大事なポイントから説明します。

ChatGPTやClaude.aiと、Claude Codeは**根本的に別物**です。

ChatGPTやClaude.aiは「**会話するAI**」です。質問を入れると答えが返ってくる。文章を書いてもらえる。画像を作ってもらえる。でも、それだけです。

一方、Claude Codeは「**あなたのPCを操作できるAI**」です。これが決定的な違いです。

たとえば、こんなことができます。

* 「このフォルダの中のファイル全部読んで、要約して」→ 実際にフォルダを開いて読む
* 「ブログ記事を書いて、このファイルに保存して」→ 実際にファイルが作られる
* 「このエクセルからデータ抜き出して、整形して」→ 実際にエクセルを読んで、新しいファイルを作る
* 「WordPressに記事を投稿して」→ 実際にWordPressと通信して投稿する

ChatGPTに「ファイルに保存して」と頼んでも、保存されません。せいぜい「このテキストをコピーして保存してください」と返ってくるだけです。

Claude Codeは違います。**本当にあなたのPCで作業をする**のです。

### たとえで言うと

副業をやっている人にとって、この違いは決定的です。

「毎日の作業」を自動化できるのは、後者だけだからです。

## 1-2. 副業で何ができるのか

具体的に、副業でどう使えるのか、3つの例を挙げます。

### 例1:ブログ記事の半自動化

### 例2:SNS運用の効率化

### 例3:アフィリエイト記事の量産

**重要な注意点:**  
これらは「全自動で売上が立つ魔法」ではありません。あくまで**作業時間を圧縮するツール**です。戦略を考えるのはあなたの仕事です。そこを勘違いすると、うまくいきません。この話は第9章で詳しく触れます。

## 1-3. 料金とプラン選び

Claude Codeを使うには、**有料プランが必須**です。無料のClaude.aiでは使えません。

選択肢は大きく3つ。

| プラン | 月額 | 向いている人 |  
|---|---|---|  
| **Claude Pro** | 約$20 | これから始める人・個人利用 |  
| **Claude Max** | $100〜$200 | 1日に何時間も使うヘビーユーザー |  
| **API課金** | 使った分だけ | 開発者・特殊な用途の人 |

### 結論:最初は「Claude Pro」で十分

副業利用なら、まずはPro($20/月)から始めてください。

理由は3つあります。

1. **Claude.aiとClaude Code の両方が使える**(Claude.aiは別途契約する必要がない)
2. Pro の利用枠で、1日数時間の作業なら足りる
3. 足りなくなったら、Maxにアップグレードできる

最初からMaxを契約する必要はありません。**使い込んでから判断してください。**

### 支払い方法

Claude.aiの公式サイト([claude.ai](http://claude.ai))からログインして、Settings → Billing からPro契約します。クレジットカードが必要です。

為替の影響で月額は上下します。ざっくり**月額3,000円前後**と考えておけば間違いありません。

### API課金について(参考)

「APIキー」というもので使う方法もあります。これは使った分だけ支払う方式です。ただし、副業用途の大半はPro契約のほうが安く済みます。最初はPro一択で考えてOKです。

## 第2章 インストールと初期セットアップ

## 2-1. 必要なもの

インストール前に、以下を確認してください。

* [ ] Mac(macOS 10.15 Catalina以降) — ほぼ全ての2019年以降のMac
* [ ] Claude Pro(以上)の契約
* [ ] Macの管理者パスワード(普段ログインに使っているパスワード)
* [ ] インターネット接続
* [ ] **作業時間:30分〜1時間**(初めての方は余裕を持って)

### Macのバージョン確認方法

画面左上のリンゴマーク 🍎 → 「このMacについて」をクリック。macOSのバージョンが表示されます。

数字が10.15以上、または「Big Sur」「Monterey」「Ventura」「Sonoma」「Sequoia」「Tahoe」のいずれかなら、大丈夫です。

## 2-2. ターミナルを開く

Claude Codeは「ターミナル」というアプリで動きます。初めての方のために、まずターミナルを開く手順から説明します。

### ターミナルの開き方

1. キーボードで「⌘(Command) + スペース」を押す
2. 検索窓が開くので「ターミナル」と入力
3. 一番上に出てくる「ターミナル.app」を選んでEnter

真っ黒な、または白い画面が開きます。これが「ターミナル」です。

### ターミナルは怖くない

初めて見ると「ハッカーが使うやつだ…」と身構えてしまう方がいますが、**ただの入力画面**です。

* キーボードで入力する
* Enterを押す
* 結果が表示される

この繰り返しです。マウスはほぼ使いません。

「間違えて入力したら壊れる?」という心配をよく聞きますが、**このマニュアル通りに進める限り、壊れません**。もし変な結果が出ても、ターミナルを閉じてもう一度開けばリセットされます。

## 2-3. Claude Codeをインストールする

公式推奨の「ネイティブインストーラー」を使います。**Node.jsなどの事前準備は不要**です(昔の記事ではNode.jsを入れるよう書かれていますが、今は不要です)。

### インストールコマンド

ターミナルに、以下を**そのままコピペ**してEnterを押してください。

```
curl -fsSL https://claude.ai/install.sh | bash
```

### 実行中に表示されるもの

コマンドを実行すると、文字がダーッと流れます。これは正常です。「ダウンロードしています」「展開しています」という作業が表示されています。

途中で**管理者パスワードを求められる場合があります**。Macのログインパスワードを入力してください(入力中は画面に文字が出ませんが、ちゃんと入力されています)。

最後に「Installation complete」のようなメッセージが出れば成功です。

### インストールの確認

続けて、以下を入力してEnter:

```
claude --version
```

バージョン番号が表示されれば成功です(例:`2.1.92` のような数字)。

**「command not found」と出た場合**→ つまずきポイントの項目(2-5)で対処します。

## 2-4. ログインして使ってみる

インストールが終わったら、ログインします。

ターミナルに以下を入力してEnter:

```
claude
```

初回起動では、以下を順番に聞かれます。

1. **カラーテーマの選択** → 好きなもので(Dark / Lightなど)。あとで変えられます。
2. **ログイン方法** → 「Claude account with subscription」を選択(矢印キーで選んでEnter)
3. **ブラウザが開く** → Anthropicのログイン画面が開くので、Claude Proを契約しているアカウントでログイン
4. **認証コードをコピー** → ブラウザに表示されるコードを、ターミナルに貼り付けてEnter

これで起動完了。プロンプト待ち状態(文字を入力できる状態)になります。

**最初の挨拶をしてみましょう。**

```
こんにちは。自己紹介してください。
```

と入力してEnter。

Claudeが返事をすれば、成功です。

### 終了するには

```
/exit
```

と入力するか、**Ctrl + C** を2回押します。

## 2-5. つまずきポイント5つと解決法

ここでは、実際によく起こるトラブルの対処法をまとめます。

### ① `command not found: claude` と出る

インストールはできたのに、コマンドが見つからないパターン。

**対処:** ターミナルを一度閉じて、新しいターミナルを開き直してください。それでも出るなら、以下を実行:

```
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

これは「claudeコマンドの場所をMacに教え直す」という意味です。中身を理解しなくても大丈夫です。

### ② 管理者パスワードが通らない

インストール中のパスワード入力で、入力しても「認証に失敗」と出るパターン。

**対処:** 入力中は画面に文字が出ませんが、キーは反応しています。Macログイン時と同じパスワードを、間違えずにもう一度入力してください。**Caps Lockキー**が入っていないかも確認。

### ③ ログイン時にブラウザが開かない

**対処:** ターミナルに表示されているURLを、自分で手動でブラウザに貼り付けてください。そこからの流れは同じです。

### ④ Claude Codeが起動するが、課金プランを求められる

**対処:** Claude Proの契約がまだ有効になっていない可能性があります。[claude.ai](http://claude.ai) にブラウザでログインし、通常のチャット画面が「Pro」と表示されているか確認してください。Pro になっていない場合は、Settings → Billing で契約状況を確認。

### ⑤ インストールが途中で止まる・タイムアウトする

**対処:** 多くの場合、ネット環境が不安定です。Wi-Fiを再接続してから、もう一度インストールコマンドを実行してください。同じコマンドを複数回実行しても、問題ありません。

### それでも解決しない時の切り札

```
claude doctor
```

これは「健康診断コマンド」です。Claude Codeが何か問題を抱えていないか、自動でチェックして教えてくれます。何かおかしいと感じたら、まずこれを走らせるのが鉄則です。

## 第3章 最初の一歩 — プロジェクトフォルダと最初の会話

## 3-1. なぜ「フォルダ単位」で動かすのか

Claude Codeを使う上で、最初に覚えるべき最重要ルールがあります。

**「作業したいフォルダに移動してから `claude` を起動する」**

これがすべての基本です。

### なぜか

Claude Codeは「起動した時のフォルダ」を作業場所として認識します。そのフォルダの中身が、Claudeの視界です。

こうすることで、Claudeは**そのフォルダのファイルだけを見て、他のフォルダには触らない**。安全で、かつ集中して作業できます。

### 逆に:やってはいけないこと

デスクトップや「ホームフォルダ」の直下で `claude` を起動してはいけません。Claudeが全ファイルを見てしまい、混乱します。PCがゴチャゴチャしてくると動作も遅くなります。

**必ず「用途別のフォルダ」を作って、そこで起動する** — これを徹底してください。

## 3-2. 最初のプロジェクトフォルダを作る

実際にやってみましょう。

### 手順

1. Finderを開く
2. 「書類」(Documents)フォルダに移動
3. 新しいフォルダを作る(右クリック → 新規フォルダ)
4. 名前を `claude-test` にする

これであなたの「実験用プロジェクトフォルダ」が完成です。

### ターミナルでそのフォルダに移動する

ターミナルに以下を入力:

```
cd ~/Documents/claude-test
```

`cd` は「移動する(change directory)」という意味です。これで、ターミナル上の作業場所が `claude-test` フォルダに切り替わりました。

**裏技:** Finderでフォルダを右クリック → 「新規ターミナルをフォルダに開く」を選ぶと、最初からそのフォルダでターミナルが開きます。こっちのほうが楽です。(このメニューが出ない場合は、システム設定 → キーボード → キーボードショートカット → サービス で「新規ターミナル」系のチェックを入れてください)

## 3-3. 最初のタスクを頼んでみる

フォルダに移動したら、Claudeを起動:

```
claude
```

入力待ちになったら、こう頼んでみましょう。

```
hello.txt というファイルを作って、中に「はじめてのClaude Code」と書いてください。
```

Enterを押すと、Claudeは何かを考えはじめます。

### 権限(Permission)の確認が出る

ファイルを作ろうとした瞬間、こんな感じのメッセージが出ます(英語の場合もあります):

```
Claude wants to create the file: hello.txt
[1] Yes [2] Yes, and don't ask again [3] No
```

これがClaude Codeの**最重要機能**です。

### この表示の意味

Claudeは**勝手には動きません**。ファイルを作ったりコマンドを実行したりする前に、**毎回あなたに許可を求めます**。

最初は `1` を選びましょう。数字を入力してEnter。

### ファイルができているか確認

Finderで `claude-test` フォルダを見てください。`hello.txt` が作られているはずです。中身を開くと「はじめてのClaude Code」と書かれています。

**おめでとうございます。** これがClaude Codeの基本動作です。

## 3-4. 権限(Permission)の考え方

さっきの確認画面について、もう少し踏み込みます。

### 「毎回聞かれるのは面倒」と思うかもしれないが

最初は面倒に感じます。でも**この仕組みは絶対に切らないでください**。

理由:Claude Codeは、あなたのPC上で本当にコマンドを実行できるAIです。たとえば誤って「全部のファイルを削除して」というコマンドを判断すると、**本当に削除します**。

確認画面は、その防波堤です。

### 慣れたら「いつも許可」を選んでよいもの

作業を進めるうちに、毎回確認されるのが本当に煩わしいものも出てきます。たとえば:

* ファイルの読み込み(Read)
* フォルダ内の一覧表示(List)

こういう「読むだけ」の操作は、`[2] Yes, and don't ask again` を選んで自動化してOKです。

### 絶対に「いつも許可」にしてはいけないもの

これらは**毎回確認**してください。面倒でも、これだけは守ってください。安全性の詳細は第8章で扱います。

## 3-5. 基本コマンド3つ

セッション中に使う、最低限知っておくべきコマンド。

| コマンド | 意味 |  
|---|---|  
| `/help` | ヘルプ表示。使えるコマンドが一覧で出る |  
| `/clear` | 会話履歴をリセット(新しいセッションとしてやり直す) |  
| `/exit` | Claude Codeを終了する |

### `/clear` の使い時

「最初はブログ記事を書いてたけど、話が逸れてきた」「新しい話題に切り替えたい」という時に使います。

Claude Codeには**コンテキスト(文脈の記憶容量)の上限**があります。長く会話しすぎると、最初のほうの話を忘れはじめます。話題が変わったら `/clear` でリセットするのが、正しい使い方です。

### /exitと⌘+Qの違い

`/exit` は「セッションをきれいに終了」。`⌘+Q` は「ターミナルごと終了」。どちらでも動作に大差はありませんが、`/exit` のほうが行儀は良いです。

ひとことで言うと:

> **「起動するたびに、Claudeが最初に必ず読む指示書」**

です。

たとえば、こんなことを書いておけます。

これを書いておくと、起動するたびに説明しなくても、Claudeが**最初から理解した状態**でスタートします。

### ChatGPTの「カスタム指示」と同じイメージ

ChatGPTを使ったことがある方なら、「カスタム指示」に似ています。あれのClaude Code版、と考えてください。

違いは、**ファイルとして手元にある**こと。好きなだけ編集できて、複数使い分けられる。

CLAUDE.mdは、置き場所によって**2段階**で効きます。

### ① グローバル:全プロジェクト共通

置き場所:`~/.claude/CLAUDE.md`

これは、**どのフォルダで `claude` を起動しても、必ず読まれる**ファイル。

ここには「あなたという人間の、普遍的な情報」を書きます。

例:

* 自己紹介(名前、職業、経験)
* コミュニケーションの好み
* 絶対に守ってほしいルール

### ② プロジェクト:そのフォルダでだけ効く

置き場所:`[プロジェクトフォルダ]/CLAUDE.md`

たとえば `~/Documents/my-blog/CLAUDE.md` のように、各フォルダのルートに置くやつです。

そのフォルダで `claude` を起動した時だけ、読まれます。

ここには「そのプロジェクト固有の情報」を書きます。

例:

* このブログのテーマ、読者層
* 記事の構成テンプレート
* 使ってはいけない言葉リスト

### 両方読まれる

プロジェクトフォルダで起動すると、**グローバル + プロジェクト** の両方が読まれます。使い分けのコツは:

* 自分の普遍的なルール → グローバルに書く
* 仕事ごとの特殊ルール → プロジェクト側に書く

## 4-3. グローバルCLAUDE.mdの作り方

さっそく作ってみます。

ターミナルで以下を実行:

```
mkdir -p ~/.claude
```

次に、空のCLAUDE.mdを作る:

```
touch ~/.claude/CLAUDE.md
```

そして、開いて編集:

```
open -e ~/.claude/CLAUDE.md
```

「テキストエディット」アプリが開くので、そこに書きます。

### サンプル:グローバルCLAUDE.mdの中身

以下をコピペして、あなたの情報に書き換えてください。

```
# 私について

- 名前:○○(ニックネームでもOK)
- 職業:○○
- AI活用歴:○年(初心者なら「初心者」と書いてOK)

# コミュニケーションの好み

- 説明はなるべく短く、要点だけ
- 専門用語は避け、噛み砕いた言葉で
- 「なぜそうするのか」の理由もセットで説明してほしい
- 作業を始める前に、必ず私の意図を確認してから動いてほしい

# 禁止事項

- ファイルの削除は、私が明示的に指示した時だけ
- `sudo` を使うコマンドは、実行前に必ず確認
- 勝手に外部のサービスに接続しない
```

保存したら閉じてOK。**これだけで、今後のClaude Codeの動きが変わります。**

## 4-4. プロジェクトCLAUDE.mdの作り方

特定のプロジェクト(たとえばブログ)用のCLAUDE.mdを作ってみます。

### 自動生成コマンド `/init`

Claude Code起動中に `/init` と入力すると、Claudeが**今いるフォルダの中身を自動で調査して、たたき台のCLAUDE.mdを作ってくれます**。

```
/init
```

まずこれを試してみるのが早いです。出力されたCLAUDE.mdを、自分好みに編集すれば完成。

### 手動で書く場合のサンプル

```
# このプロジェクトについて

- 種類:美容系ブログの記事執筆
- ターゲット読者:30〜40代女性、美容に関心のある層
- 読者の悩み:時短ケア、効率のいい美容習慣

# トーンと文体

- 「ですます調」で統一
- 絵文字は使わない
- 見出しは疑問形で(例:「なぜ髪がパサつくのか?」)
- 1段落は3〜4行で区切る

# 禁止ワード

- 「絶対に効く」「必ず治る」(薬機法違反の可能性)
- 「劇的に」「驚くほど」(誇大表現)

# 記事の型

1. 導入(読者の悩みへの共感)
2. 原因の解説
3. 解決策3つ
4. まとめ(次のアクションへの誘導)
```

これを `~/Documents/my-blog/CLAUDE.md` として保存すると、ブログフォルダで起動するたびに、このルールが適用されます。

## 4-5. CLAUDE.mdの書き方のコツ

うまく効かせるために、いくつかのポイントがあります。

### コツ①:長すぎないこと

CLAUDE.mdは毎回読まれます。長すぎると、**肝心の作業に使えるメモリを食ってしまう**。

目安:**200〜400行以内**。A4で3〜4ページくらい。それ以上になりそうなら、第5章の「skill」に切り出します。

### コツ②:「なぜ」を書く

悪い例:

```
- 記事は3,000文字以上で書く
```

良い例:

```
- 記事は3,000文字以上で書く(Googleで上位表示されやすいため)
```

「なぜそうするのか」を書くと、Claudeが**文脈に応じて柔軟に判断**できるようになります。

### コツ③:否定形より肯定形で

悪い例:

```
- タメ口を使わない
```

良い例:

```
- ですます調で統一する
```

人間と同じで、「やるべきこと」を書かれたほうが動きやすいです。

### コツ④:具体例を入れる

悪い例:

```
- 親しみやすい文体で書く
```

良い例:

```
- 親しみやすい文体で書く
  - 良い例:「○○って、意外と知られていないんです」
  - 悪い例:「○○については、あまり認知されておりません」
```

具体例があると、Claudeの出力が一気に安定します。

## 第5章 skill(スキル) — 専門技能を覚えさせる

## 5-1. skill とは何か

CLAUDE.mdは「毎回読まれる」仕組みでした。では、**特定の作業をする時だけ読まれる専門知識**は、どう管理すればいいでしょうか。

その答えが **skill** です。

### たとえで言うと

たとえば「WordPressに記事を投稿する手順」は、毎回必要なわけじゃない。記事を書く時だけ必要です。こういう**専門技能**をskillにまとめておくと、Claudeが「いま必要だな」と判断した時だけ読みます。

### skillの賢さ:progressive disclosure

skillには「段階的に読む」という仕組みがあります。

1. まず、skillの\*\*短い説明文(description)\*\*だけを見る
2. タスクに関係ありそうなら、**本文**を読む
3. もっと詳しい情報が必要なら、**関連ファイル**も読む

これを **progressive disclosure(段階的開示)** と呼びます。この仕組みのおかげで、**たくさんのskillを持っていても、Claudeが重くならない**のです。

## 5-2. SKILL.mdの書き方

skillは、以下のような構造で置きます。

```
~/.claude/skills/
├── wordpress-post/
│   └── SKILL.md
├── research/
│   └── SKILL.md
└── proofreading/
    └── SKILL.md
```

**フォルダ名がスキル名**になります。スキルの本体は、その中の `SKILL.md` に書きます。

### SKILL.mdの基本形

```
---
name: wordpress-post
description: WordPressに記事を下書き投稿する手順。ユーザーがブログ記事を書き終えて「WordPressに投稿して」と言った時に使う。
---

# WordPressへの記事投稿手順

## 前提

- APIキーは `~/.env` に保存されている
- サイトURLは記事ごとに確認する

## 手順

1. ユーザーに投稿先サイトを確認する
2. 記事のタイトル、本文、カテゴリーを整理する
3. アイキャッチ画像の指定があるか確認する
4. 下書き(status: draft)として投稿する
5. 投稿が完了したら、プレビューURLを表示する

## 注意

- 公開(status: publish)は指示されない限り選ばない
- カテゴリーの自動判定はせず、必ず確認する
```

### 最重要:descriptionの書き方

skillが**いつ呼び出されるか**は、**description の中身**で決まります。ここを雑に書くと、必要な時に呼ばれなかったり、関係ない時に呼ばれてしまったりします。

**良いdescriptionの条件:**

1. **何をするskillか**が、1〜2文で明確
2. **どういう状況で使うか**が書いてある
3. **トリガーになりそうなキーワード**が入っている

悪い例:

```
description: WordPress用のskill
```

良い例:

```
description: WordPressに記事を下書き投稿する手順。ユーザーがブログ記事を書き終えて「WordPressに投稿して」「下書きに上げて」と言った時に使う。アイキャッチ画像、カテゴリー、タグの扱いも含む。
```

情報量が2倍以上違います。この差が、skillが正しく呼ばれるかどうかを分けます。

## 5-3. skillを作ってみる

実際に、副業でよく使う3つのskillを作ってみます。

### skill ①:リサーチ用

作成コマンド:

```

touch ~/.claude/skills/research/SKILL.md
open -e ~/.claude/skills/research/SKILL.md

```

```

# 校正の観点

## 1. 誤字脱字・表記ゆれ

- タイポ
- カタカナ表記の揺れ(例:ユーザー / ユーザ)
- 漢字/ひらがなの揺れ(例:出来る / できる)

## 2. 日本語の不自然さ

- 主語と述語がねじれていないか
- 助詞の重複(「が」「を」が続く)
- 1文が長すぎないか(目安:80字以内)

## 3. 読者視点

- 専門用語にフリガナ/説明があるか
- 1段落が長すぎないか(3〜4行目安)
- 結論が先にきているか

## 出力

- 元の文章を残したまま、修正箇所に注釈をつける
- なぜ直したかの理由も1行で添える
- 全文の書き換えはせず、部分修正に留める
```

## 5-4. skillとCLAUDE.mdの使い分け

### 判断基準

| 性質 | どっちに書く? |  
|---|---|  
| 毎回使う、人格や好み | [**CLAUDE.md**](http://CLAUDE.md) |

## 5-5. skillが呼ばれない時のチェックリスト

skillを作ったのに、肝心な時に呼ばれない。よくあるトラブルです。

**確認ポイント:**

1. `description` に、タスクに関連するキーワードが入っているか
2. `description` に、具体的な状況が書いてあるか
3. フォルダ名とYAMLの `name` が一致しているか
4. ファイル名が `SKILL.md`(大文字)になっているか
5. `~/.claude/skills/` 配下に置かれているか

特に❶❷が重要です。**description は読者(=Claude)のために書く**という意識で、丁寧に作ってください。

## 第6章 エージェント(subagent) — 分業で精度を上げる

## 6-1. なぜエージェントが必要なのか

ここは少し難しい話なので、ゆっくり進めます。

Claude Codeには「**コンテキスト上限**」というものがあります。会話が長くなってくると、古い内容から忘れていきます。これはClaude Codeに限らず、すべてのAIに共通する制約です。

たとえば、こんな場面を想像してください。

* ブログ記事を書くために、10個のサイトをリサーチする
* リサーチ結果を全部読み込んだ
* この時点で、Claudeの「記憶」はリサーチ情報でパンパン
* いざ執筆に入ろうとすると、**CLAUDE.mdのルールを忘れている**
* 結果、雑な文章が出てくる

これが **コンテキスト汚染** です。

## 6-2. エージェントの本質:「別人格に任せる」

この問題を解決するのが、エージェント(subagent)です。

エージェントとは、ざっくり言うと:

> **「別のタスク専用のClaudeを、裏で動かして、結果だけ持って帰ってもらう」**

という仕組みです。

### 具体例で理解する

さっきのブログ記事の例で考えます。

**エージェントなし:**

```
あなた → Claude(1人)→ リサーチもする、執筆もする
         ↑
         記憶がゴチャゴチャになる
```

**エージェントあり:**

```
あなた → メインClaude
            ↓ 「リサーチお願い」
         リサーチ専用エージェント(別人格、別の記憶)
            ↓ 要約だけ返す
         メインClaude → 執筆に集中できる
```

リサーチエージェントは、自分だけの記憶空間で10サイトを読み、結果を**要約して**メインClaudeに渡します。メインClaudeは、汚れた情報(詳細なHTMLとか)を持たずに、きれいな要約だけ受け取る。**だから執筆に集中できる。**

## 6-3. 無理に使わなくていい、でも知っておくべき理由

正直な話、**最初からエージェントを使いこなす必要はありません**。

ほとんどの作業は、[CLAUDE.md](http://CLAUDE.md) と skill で十分です。エージェントは、作業が複雑になってきた時の「次の手」として理解しておくだけで、今は十分です。

ただし、存在を知っているか知らないかで、**作業の設計思想**が変わります。だからこの章を入れています。

### 使うべきタイミングの目安

これらが揃ったら、エージェント化を検討。それまではシンプルに進めてOKです。

## 6-4. エージェントの作り方

エージェントは、以下のフォルダにMarkdownファイルとして置きます。

```
~/.claude/agents/
├── researcher.md
├── copywriter.md
└── proofreader.md
```

**スキル(skill)はフォルダ単位、エージェント(agent)はファイル単位**。ここは混同しないでください。

### `/agents` コマンドで作る(推奨)

Claude Code起動中に、以下を実行:

```
/agents
```

対話形式で作成できます。質問に答えていくだけで、たたき台ができます。初めての方は、この方法が一番簡単です。

### 手動で書く場合のサンプル

`~/.claude/agents/researcher.md`:

```
---
name: researcher
description: Webリサーチの専門家。テーマを渡されると、インターネット上の情報を調べて、要約して返す。執筆の前段階で使う。
tools: Read, WebSearch, WebFetch
model: sonnet
---

あなたはWebリサーチの専門家です。

# 役割

与えられたテーマについて、Webを検索し、重要な情報を集め、
**要約して**メインエージェントに返します。

# 進め方

1. テーマから検索キーワードを3〜5個考える
2. それぞれで検索し、上位5件の信頼できるサイトを選ぶ
3. 重要な情報を抽出する
4. 情報源(URL)と一緒に、要点を箇条書きで返す

# 出力形式

- Markdown
- 見出しで情報カテゴリーを分ける
- 情報源URLを必ず併記
- 詳細な本文は返さない(要約のみ)
- 全体で2,000字以内に収める

# 注意

- 情報の鮮度を確認する(公開日を見る)
- 矛盾する情報は両論併記する
- 推測は「推測」と明示する
```

### frontmatterの各項目

| 項目 | 意味 |  
|---|---|  
| `name` | エージェントの名前。これで呼び出される |  
| `description` | いつ使うか。skillの description と同じ要領で |  
| `tools` | このエージェントが使える機能の制限。読み込みだけ、など |  
| `model` | 使うAIモデル(sonnet / opus / haiku)。軽い作業ならhaiku |

### tools の制限が重要

エージェントに**余計な権限を与えないこと**が、精度にも安全性にも効きます。

* **リサーチャー** → `Read, WebSearch, WebFetch`(書き込みは不要)
* **コピーライター** → `Read, Write, Edit`(ネット検索は不要)
* **校正者** → `Read`(書き換えもさせない)

**「必要最小限の道具だけ渡す」** が、うまく使うコツ。

## 6-5. 代表的な副業向け3エージェント

副業ワークフローで使える、基本の3人を紹介します。

### ① researcher(リサーチャー)

役割:Web上の情報を集めて、要約して返す。  
使いどころ:記事執筆前のリサーチ、競合調査、トレンド把握。  
権限:読むだけ、検索するだけ。

### ② copywriter(コピーライター)

役割:与えられた構成と素材から、記事本文を書く。  
使いどころ:ブログ本文、商品説明、SNS投稿の原稿作成。  
権限:ファイル作成・編集はOK、ネット検索は不要。

### ③ proofreader(校正者)

役割:完成した文章を読んで、誤字や不自然な表現を指摘する。  
使いどころ:公開前のチェック。  
権限:読むだけ(勝手に書き換えさせない)。

### 呼び出し方

エージェントを作ったあとは、メインのClaudeに普通に頼めば、Claudeが自動で「これはリサーチャーに任せよう」と判断します。明示的に呼びたい時は:

```
@researcher このテーマで調べてください:時短ヘアケア
```

と `@エージェント名` で呼ぶ書き方も使えます。

## 6-6. エージェント設計で絶対避けるべき3つの罠

### 罠①:エージェントに仕事を詰め込みすぎる

1つのエージェントに「リサーチして、記事を書いて、投稿して」と全部やらせるのは本末転倒です。それなら普通のClaude Codeでやればいい。

**エージェント=1人1役。** これを徹底してください。

### 罠②:descriptionが雑で呼ばれない

skillと同じ原理で、descriptionが雑だと呼ばれません。**「いつ使うべきエージェントか」が一目で分かる説明**にしてください。

### 罠③:エージェントを増やしすぎる

最初から5人10人作っても使いこなせません。**まず3人**。使いながら、必要なら増やす。これが鉄則です。

## 第7章 スラッシュコマンド — 毎回の指示を短縮する

## 7-1. スラッシュコマンドとは

これまでに、すでにいくつか登場しています。

* `/help`
* `/clear`
* `/exit`
* `/init`
* `/agents`

これらは**組み込み**のスラッシュコマンド。Claude Codeに最初から入っています。

そして自分で**オリジナルのスラッシュコマンド**も作れます。

## 7-2. 自作スラッシュコマンドの作り方

置き場所:

```
~/.claude/commands/
```

ここにMarkdownファイルを置くと、ファイル名がそのままコマンド名になります。

例:`~/.claude/commands/research.md` を作れば、`/research` というコマンドが使えるようになります。

### サンプル:`/research` コマンド

`~/.claude/commands/research.md`:

```
---
description: 指定されたテーマについてWebリサーチを行う
---

以下のテーマについて、researcherエージェントを使ってリサーチしてください。

テーマ: $ARGUMENTS

- 基本情報
- 最新動向(半年以内)
- 競合・類似サービス
- ユーザーの声

# 出力
- 見出しごとに整理したMarkdown
- 情報源URLを併記
- 全体で2,000字以内
```

使い方:

```
/research 時短ヘアケア
```

`$ARGUMENTS` の部分に「時短ヘアケア」が入ります。これでリサーチが走ります。

## 7-3. skillとスラッシュコマンドの違い

両方とも、作業の効率化に使える仕組みですが、違いは:

| 項目 | skill | スラッシュコマンド |  
|---|---|---|  
| 呼び出し方 | Claudeが自動で判断 | **あなたが明示的に呼ぶ** |  
| 向いている用途 | 文脈で判断すべき作業 | 決まりきった手順 |  
| 作り方 | フォルダ + [SKILL.md](http://SKILL.md) | 単一の.mdファイル |

### 使い分けの実例

### 迷ったら

**毎日使うもの、決まった入力で動かすもの** → スラッシュコマンド  
**毎回少しずつ違うけど、やり方は共通** → skill

これでだいたい分けられます。

## 7-4. 組み込みコマンドを使いこなす

最後に、覚えておくと便利な組み込みコマンドを紹介。

| コマンド | 使いどころ |  
|---|---|  
| `/clear` | 話題が変わった時。セッションをリセットする |  
| `/init` | 新しいプロジェクトフォルダでCLAUDE.mdを自動生成 |  
| `/agents` | エージェントの一覧表示・作成・編集 |  
| `/model` | 使用モデルを切り替え(軽い作業はhaiku、重い作業はopus) |  
| `/resume` | 前回の会話を再開する |  
| `/help` | ヘルプ表示 |

### `/model` の賢い使い方

作業によってモデルを切り替えると、応答速度と品質のバランスが取れます。

* 軽い調べ物、短い返信 → `/model haiku`(速い・安い)
* 通常の作業 → `/model sonnet`(標準)
* 複雑な設計、長文執筆 → `/model opus`(遅い・高品質)

## 第8章 安全に使うためのルール

Claude Codeは強力な道具です。だからこそ、**事故を起こさない使い方**が重要です。この章はスキップせずに読んでください。

## 8-1. permissions(権限)の基本

第3章で触れた「確認画面」。これを制御するのが、**permissions** という設定です。

### 設定ファイルの場所

```
~/.claude/settings.json
```

ここに、「これは自動で許可していい」「これは絶対にダメ」を書いておけます。

### サンプル設定

```
{
  "permissions": {
    "allow": [
      "Read(**)",
      "Bash(ls:*)",
      "Bash(pwd)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(sudo:*)",
      "Bash(curl:*)",
      "Write(~/.ssh/**)",
      "Write(~/.aws/**)"
    ]
  }
}
```

### 意味

* **allow**:毎回確認せずに実行してよい操作
* **deny**:絶対に実行させない操作

  + `Bash(rm:\*)` → **削除コマンドは絶対禁止**
  + `Bash(sudo:\*)` → **管理者権限コマンドは絶対禁止**
  + `Bash(curl:\*)` → 外部からのダウンロードを禁止(必要時に一時解除)
  + `Write(~/.ssh/\*\*)` → SSH設定への書き込み禁止
  + `Write(~/.aws/\*\*)` → AWS認証情報への書き込み禁止

### denyが最優先

allow と deny が矛盾した場合、**deny が勝ちます**。なので、危険な操作は必ず deny に入れておく。

## 8-2. やってはいけないこと5選

### ① APIキーやパスワードを直接CLAUDE.mdに書く

絶対ダメです。CLAUDE.mdはClaudeが読む = ログに残る可能性があります。

**正しい方法:** `.env` ファイルを使う。

```
touch ~/.env
open -e ~/.env
```

中身:

```
OPENAI_API_KEY=sk-xxxxxxxx
WP_PASSWORD=xxxxxxxx
```

CLAUDE.mdには「APIキーは `~/.env` にある」とだけ書く。

### ② `.env` ファイルをGitHubに上げる

うっかりやりがちな事故。`.env` は**絶対に**GitHubに上げない。

防御策:`.gitignore` に必ず追加する。

```
.env
.env.*
*.key
```

### ③ `sudo` を含むコマンドを自動許可にする

`sudo` = 管理者権限。Macの根幹に触れるコマンドです。これを自動許可にしてはいけません。毎回確認する、これを徹底。

### ④ 見知らぬスクリプトをそのまま実行

「このコマンドを実行してください」とネットに書いてあっても、**中身を理解せずに実行しない**。最低限、Claudeに「このコマンドは何をするか説明して」と聞く癖をつけてください。

### ⑤ 「Yes, and don't ask again」の乱用

面倒だからといって、確認画面で毎回「don't ask again」を押していると、気づけば何でも自動で動く状態になっています。**意識的に選ぶ**こと。

## 8-3. 定期的にやるべきこと

### セッションごとのチェック

作業の区切りで、以下を確認:

* 意図しないファイルが作られていないか
* 意図しない場所に変更が入っていないか

### 何かおかしい時

Claude Code以前の話として、**Macのバックアップを必ず取ってください**。

## 第9章 副業ワークフローの設計思想

技術的な話はここまで。この章では、**どう使うか**の考え方を伝えます。ここが一番大事かもしれません。

## 9-1. 「自動化しすぎない」がコツ

Claude Codeを手に入れると、人は興奮します。「全部自動化してやる!」と。

**ほぼ必ず失敗します。**

理由:**戦略と判断は自動化できない**から。

自動化できるのは、あくまで「**作業**」です。たとえば:

* キーワードからタイトルを10案出す(作業)
* その中からどれを選ぶか(**判断**)
* 記事本文を書く(作業)
* どこを修正すべきか(**判断**)
* WordPressに投稿する(作業)
* いつ投稿するか、どのカテゴリに入れるか(**判断**)

**判断の部分を自動化しようとすると、品質が崩壊します。**

## 9-2. 人間が判断すべきこと / 任せていいこと

### 人間がやるべきこと

* **戦略設計**(何のために何を作るか)
* **ターゲット設計**(誰に届けるか)
* **品質の最終判断**(これで公開していいか)
* **お金が発生する判断**(購入、契約、公開)
* **例外対応**(普段と違う状況になった時の判断)

### Claudeに任せていいこと

* 情報収集・要約
* たたき台の作成(構成、本文、デザイン案)
* 定型作業の繰り返し(整形、変換、校正)
* 複数案の生成(A/Bテスト用のパターン出し)
* 反復的なコミュニケーション(メッセージのひな形)

### 境界線の引き方

「**ミスった時のダメージ**」で判断してください。

お金・法律・人間関係に関わるものは、常に人間チェック。ブログの誤字は、最悪あとで直せる。この感覚を持つこと。

## 9-3. 小さく始めて育てる

最後に、実践に一番大事な話。

**いきなり完璧なワークフローを作ろうとしない。**

順番はこうです:

### ステップ1:CLAUDE.md だけで使う(1週間)

まずは、グローバルCLAUDE.mdを書いて、普通にClaude Codeと会話する。プロジェクトCLAUDE.mdも、必要になったら作る。

**この段階でできないなら、スキルやエージェントを作っても使いこなせません。**

### ステップ2:よく使う手順をskillにする(次の1〜2週間)

使っているうちに、「この手順、毎回同じこと指示してるな」と気づく作業が出てきます。それをskillに切り出す。3つくらいできれば十分。

### ステップ3:複雑な作業でエージェントを導入(1ヶ月後〜)

「リサーチ→執筆→校正」のような**役割分担が明確な作業**が出てきたら、エージェントを導入する。無理に作らない。

### ステップ4:スラッシュコマンドで定型化(必要に応じて)

毎日同じコマンドを打っているのに気づいたら、スラッシュコマンド化する。これは一番最後でいい。

## 9-4. 成長するCLAUDE.mdを作る

CLAUDE.mdは、**一度書いて終わり**ではありません。

実際に使っていると、「これもルールに追加したい」「この書き方は効かなかった」という発見が出てきます。それを**週に1度、見直して更新**してください。

3ヶ月経つと、CLAUDE.mdは**あなたの仕事の仕方そのもの**を反映したものになります。ここまで来ると、どんな新しいAIが出てきても、CLAUDE.mdを渡すだけで同じ品質のアウトプットが得られる。

**CLAUDE.mdを育てる = あなたの資産を育てる**。この視点を持ってください。

## 10-1. 起動・インストール系

**対処:** 表示されているURLを手動でブラウザに貼り付ける。それでもダメなら:

```
claude logout
claude
```

で再ログインを試す。

## 10-2. 動作・挙動系

### Q4. Claudeが指示を無視してるように感じる

**チェック順:**

1. CLAUDE.mdに書いたつもりが、保存されていない
2. グローバルCLAUDE.mdとプロジェクトCLAUDE.mdが矛盾している
3. 会話が長すぎて、指示を忘れている → `/clear` してやり直し
4. 指示の書き方が曖昧 → 具体的に書き直す

**原因:** 指示が毎回違うから  
**対処:** よく使う指示は、skillかスラッシュコマンドに固定する。毎回ゼロから指示していると、人間側の揺れが品質に反映されます。

**チェック:**

1. descriptionにキーワードが入っているか
2. ファイル名が `SKILL.md`(大文字)か

4. YAMLのnameとフォルダ名が一致しているか

### Q7. エージェントが期待通りに動かない

**チェック:**

1. toolsの権限が足りているか
2. システムプロンプト(本文)が具体的か
3. エージェントの役割が広すぎないか(1人1役の原則)

**対処:** 落ち着いて:
