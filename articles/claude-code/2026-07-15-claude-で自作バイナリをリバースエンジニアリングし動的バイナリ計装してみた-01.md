HENNGE株式会社でカスタマーサクセスエンジニアをしているオオマユです。

今回は、Claude で簡単な CLI ゲームを作成し、そのバイナリファイルを Claude の力を借りながら **リバースエンジニアリング** して内部の挙動を理解し、さらに **動的バイナリ計装** によってゲームの挙動を実行時に書き換える、という一連の流れを紹介します。

具体的には、「じゃんけんゲーム」のバイナリを解析して、最終的に **必ず勝てる状態** を作り出すところまでをゴールにします。

## このテーマを扱う背景

本記事を書いた狙いは、大きく次の 2 点です。

* マルウェア解析などで使われる技術に、まずは触れてみてほしい
* Claude をはじめとする生成 AI によって、リバースエンジニアリングや動的バイナリ計装の実施ハードルが大きく下がったことを知ってほしい

IT 業界に身を置いている方でも、実際にリバースエンジニアリングを行った経験のある方は、それほど多くないのではないでしょうか。これらは特に初学者にとってハードルが高く、低レイヤーの知識が求められる領域です。しかし、Claude などの生成 AI の登場によって、その入り口はぐっと入りやすくなったと感じています。

以下で紹介する内容は、実践的なマルウェア解析のレベルには及びませんが、リバースエンジニアリングや動的バイナリ計装を学び始める **最初の一歩** になれば幸いです。

> **⚠️ 注意事項**  
> 許可されていないソフトウェアに対するリバースエンジニアリングや動的バイナリ計装は、法令や利用規約で禁止されている場合があります。本記事では、あくまで **自分自身が作成したバイナリ** を対象にしています。第三者のソフトウェアを対象にする際は、必ず権利者の許可と適用される法令を確認してください。

## そもそもリバースエンジニアリングとは

リバースエンジニアリングとは、完成した製品やソフトウェアを分解・解析し、その構造や仕様を明らかにする手法の総称です。ソースコードが手元にない（＝バイナリしかない）状態から、逆アセンブル・デコンパイルによって処理内容を復元していく作業だと考えてください。目的はさまざまですが、セキュリティの分野では主に **マルウェア解析** や **脆弱性研究** で活用されます。

マルウェア解析者は、マルウェアをリバースエンジニアリングして内部仕様を解明することで、次のような成果につなげます。

* **検知シグネチャの作成**: 特徴的なコードや文字列から YARA ルールやアンチウイルスの検知パターンを作る
* **IOC（Indicator of Compromise）の抽出**: C2 サーバーのドメイン・IP、通信の暗号化方式、ファイル名やレジストリキーなどを洗い出す
* **攻撃全体像の把握**: 感染経路、永続化の手口、情報窃取・横展開のロジックを解明する

なお、実際のマルウェアは、こうした解析を妨げるために **パッキング（圧縮・暗号化して実体を隠す）** や **難読化** を施していることがほとんどです。そのため、静的なリバースエンジニアリングだけでは全容をつかみきれないケースも多く、次に述べる動的な解析手法と組み合わせることが重要になります。

## 動的バイナリ計装とは

動的バイナリ計装（Dynamic Binary Instrumentation, DBI）とは、プログラムを実行しながら、そのバイナリコードに解析用のコード（計装コード）を動的に挿入・実行させる技術です。プログラムを実際に動かした状態で、関数の引数・戻り値・メモリの中身などをリアルタイムに監視・分析でき、さらには挙動そのものを書き換えられるのが特徴です。代表的なツールとしては、本記事で使う **Frida** のほか、**Intel Pin**、**DynamoRIO**、**Valgrind** などがあります。

こちらもマルウェア解析でよく利用されます。リバースエンジニアリング（静的解析）と動的バイナリ計装（動的解析）は対立するものではなく、互いの弱点を補い合う関係にあります。

| 観点 | 静的解析（リバースエンジニアリング） | 動的解析（動的バイナリ計装など） |
| --- | --- | --- |
| プログラムの実行 | 実行しない | 実際に実行する |
| 安全性 | 高い（コードを読むだけ） | 低い（マルウェアが実際に動く。要・隔離環境） |
| 難読化・暗号化への強さ | 弱い（パッキングされると読めない） | 強い（実行時にメモリ上で復号された状態を観測できる） |
| 主な用途 | 全体構造・アルゴリズムの把握、シグネチャ作成 | 実際の通信・ファイル操作・復号後コードの観測、挙動の書き換え |

「マルウェアを実行しない状態でコードを読み解く」か「実際に実行し、挙動を観察しながら解析する」かの違いであり、どちらにもメリット・デメリットがあります。実際の解析では、この 2 つを行き来しながら全体像を明らかにしていくのが基本です。

> **⚠️ 実マルウェアを扱う際の大前提**  
> 動的解析はマルウェアを **実際に実行する** 行為です。本番端末や社内ネットワークにつながった環境で行うと感染を広げかねません。実務では、ネットワークを遮断（またはダミー化）し、スナップショットで復元できる **隔離された仮想環境** で行うのが鉄則です。本記事は無害な自作バイナリを対象にしているため、この限りではありません。

## STEP 1: Claude Code で CLI ゲームを作成する

まずは Claude Code を利用して、簡単な CLI のじゃんけんゲームを作成してみましょう。ここでは Claude Code のセットアップは済んでいる前提で進めます。

Claude Code に対して、次のシンプルなプロンプトを入力しました。

> シンプルなじゃんけんの CLI ゲームを作成してください。言語は、C 言語でお願いします。

出力されたソースコードは以下のとおりです。

```
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <time.h>

static const char *hands[] = {"グー", "チョキ", "パー"};

int main(void) {
    srand((unsigned int)time(NULL));
    char buf[64];
    int wins = 0, losses = 0, draws = 0;

    printf("じゃんけんゲームへようこそ!\n");
    printf("入力: 0=グー, 1=チョキ, 2=パー, q=終了\n\n");

    while (1) {
        printf("あなたの手 > ");
        if (!fgets(buf, sizeof(buf), stdin)) break;
        if (buf[0] == 'q' || buf[0] == 'Q') break;

        char *end;
        long player = strtol(buf, &end, 10);
        if (end == buf || player < 0 || player > 2) {
            printf("0, 1, 2 のいずれかを入力してください。\n\n");
            continue;
        }

        int cpu = rand() % 3;
        printf("あなた: %s / CPU: %s -> ", hands[player], hands[cpu]);

        int diff = (player - cpu + 3) % 3;
        if (diff == 0) {
            printf("あいこ\n\n");
            draws++;
        } else if (diff == 2) {
            printf("あなたの勝ち!\n\n");
            wins++;
        } else {
            printf("あなたの負け...\n\n");
            losses++;
        }
    }

    printf("\n=== 結果 ===\n");
    printf("勝ち: %d / 負け: %d / あいこ: %d\n", wins, losses, draws);
    return 0;
}
```

続いて、以下のコマンドでコンパイルします。

```
cc -Os -s -fno-ident -fno-asynchronous-unwind-tables -Wl,--build-id=none -o janken janken.c
```

各オプションの意味は次のとおりです。今回はより実際のバイナリ解析に近づけるため、シンボル情報などを削ぎ落とすオプションを付けています。

| オプション | 意味 |
| --- | --- |
| `-Os` | サイズ優先で最適化 |
| `-s` | シンボルテーブル・デバッグ情報を削除（`strip` 相当） |
| `-fno-ident` | `.comment` セクションにコンパイラ識別情報を残さない |
| `-fno-asynchronous-unwind-tables` | 非同期 unwind テーブルを生成しない |
| `-Wl,--build-id=none` | リンカに build-id を埋め込ませない |
| `-o janken` | 出力ファイル名を `janken` に指定 |

コンパイルしたファイルを実行してみます。

```
elliot@mint:~/simple_game$ ./janken
じゃんけんゲームへようこそ!
入力: 0=グー, 1=チョキ, 2=パー, q=終了

あなたの手 > 1
あなた: チョキ / CPU: パー -> あなたの勝ち!

あなたの手 > 2
あなた: パー / CPU: パー -> あいこ

あなたの手 > 0
あなた: グー / CPU: チョキ -> あなたの勝ち!

あなたの手 > 1
あなた: チョキ / CPU: グー -> あなたの負け...

あなたの手 > q

=== 結果 ===
勝ち: 2 / 負け: 1 / あいこ: 1
```

じゃんけんですから、当然、勝つか負けるかは運次第です。**これを、最終的に必ず勝てるように書き換えていきます。**

このレポートは、マルウェア解析の初動（トリアージ）で見る観点をある程度押さえています。

* **基本情報（ファイル種別・コンパイラ・保護機構）**: 対象がどのプラットフォーム向けで、どんなビルド環境で作られたかを把握する第一歩です。ここでは Stack Canary や PIE といった **エクスプロイト緩和機構** の有無も報告されています（今回は攻略対象ではありませんが、脆弱性研究では重要な情報です）。
* **インポート関数の一覧**: プログラムがどの API を呼んでいるかは、挙動を推測する最重要の手がかりです。実際のマルウェア解析でも、`socket`／`connect`（通信）、`CreateProcess`（プロセス生成）、暗号 API などのインポートを見て「何をするプログラムか」の当たりを付けます。今回は `rand`・`fgets`・`printf` 程度なので、素直な CLI アプリだと即座に判断できます。
* **隠しロジックの有無**: レポートが「イカサマや隠しロジックはない」と明言しているように、意図しない裏の挙動（バックドアやトリガー条件）がないかを確認するのも解析の目的の一つです。

そして今回の攻略にとって最大のポイントは、レポートにあるとおり **「CPU の手は `cpu = rand() % 3` で決まっている」** という点です。つまり、`rand()` の戻り値さえ操作できれば、CPU の手を完全にコントロールできる、ということが分かります。ある関数の戻り値を支配すれば下流のロジックを掌握できる、というこの考え方は、後述する API フックによる挙動改変の基本そのものです。

## STEP 3: Frida で動的バイナリ計装する

解析結果が手に入ったので、次はいよいよ動的バイナリ計装です。Claude に対して、次のように依頼しました。

> 解析結果をもとに、相手のじゃんけんの手をグーに固定する Frida のスクリプトを作ってください。

出力された Frida スクリプトは以下のとおりです。

```
/*
 * janken_always_rock.js
 * ---------------------------------------------------------------
 * じゃんけんゲーム (./janken) の CPU に「常にグー」を出させる Frida スクリプト。
 *
 * 解析結果より:
 *   - CPU の手は  cpu = rand() % 3  で決まる  (呼び出し元 0x00101204)
 *   - 手の対応:  0=グー, 1=チョキ, 2=パー
 *   - rand() はこの CPU の手を決める用途にしか使われていない
 *     (srand(time(NULL)) はシード設定のみ)
 *
 * よって libc の rand() をフックし、戻り値を 0 に固定すれば
 *   cpu = 0 % 3 = 0 = グー
 * となり、相手は毎回グーを出す。
 * (0 は 3 の倍数なので modulo の実装に依存せず必ず 0 になる)
 * ---------------------------------------------------------------
 */
'use strict';

// libc の rand を解決 (PLT 経由の呼び出しも最終的にこの実体を呼ぶ)
// Frida 17 で API が変わったため、複数の解決方法にフォールバックする
function resolveRand() {
    // Frida 17+: グローバルエクスポートから解決
    if (typeof Module.getGlobalExportByName === 'function') {
        try { return Module.getGlobalExportByName('rand'); } catch (e) {}
    }
    if (typeof Module.findGlobalExportByName === 'function') {
        const p = Module.findGlobalExportByName('rand');
        if (p !== null) return p;
    }
    // 各ロード済みモジュールを走査 (libc を含む)
    for (const m of Process.enumerateModules()) {
        const p = m.findExportByName ? m.findExportByName('rand') : null;
        if (p !== null) return p;
    }
    // 旧 API フォールバック
    if (typeof Module.getExportByName === 'function') {
        try { return Module.getExportByName(null, 'rand'); } catch (e) {}
    }
    return null;
}

const randPtr = resolveRand();
if (randPtr === null) {
    console.log('[!] rand() が見つかりませんでした');
} else {
    console.log('[*] rand() を ' + randPtr + ' でフックします -> 常に 0 (グー) を返させます');
    Interceptor.attach(randPtr, {
        onLeave: function (retval) {
            // 元の値をログに残しつつ 0 に書き換える
            // cpu = rand() % 3 = 0 % 3 = 0 = グー
            const original = retval.toInt32();
            retval.replace(0);
            console.log('[hook] rand(): ' + original + ' -> 0  (CPU = グー)');
        }
    });
}
```

### スクリプトの解説

短いスクリプトですが、動的バイナリ計装のエッセンスが詰まっています。何をやっているのかが分かると、実際のマルウェア解析での応用もイメージしやすくなるはずです。以下、順を追って解説します。

#### 攻略の考え方：なぜ「戻り値を 0 にする」だけでよいのか

STEP 2 の解析から、CPU の手を決めているのは次の 1 行だと分かっています。

```
cpu = rand() % 3;   // 0=グー, 1=チョキ, 2=パー
```

ここで `rand()` が **必ず 0 を返す** ようにできれば、

となり、CPU は毎回グーになります。ポイントは **「0 は 3 の倍数」** であることです。コンパイラが `% 3` をどんな命令列（除算を避けるためのマジックナンバー乗算など）に変換していても、入力が 0 であれば結果は必ず 0 になります。つまり剰余算の内部実装を気にせず、確実にグーへ固定できるというわけです。

> 3 や 6 など他の 3 の倍数でも結果は同じですが、最もシンプルで確実な 0 を採用しています。

#### Frida による書き換えの仕組み

Frida は、実行中のプロセスに JavaScript を注入し、任意の関数を **フック** できる動的計装ツールです。本スクリプトは libc の `rand()` をフックし、「関数から戻る直前」に戻り値を 0 へ書き換えます。

`janken` 本体は PLT（Procedure Linkage Table）経由で libc の `rand()` を呼びますが、最終的に実行されるのは libc 内にある `rand()` の実体です。したがって、そこを **1 か所フックするだけで、すべての `rand()` 呼び出しに効きます**。

#### コードを読む

**フックの登録 ── `Interceptor.attach`**

`Interceptor.attach(randPtr, { onLeave: ... })` で、`rand()` が **呼ばれて戻ってくる瞬間（`onLeave`）** に割り込みます。

| 要素 | 役割 |
| --- | --- |
| `retval` | `rand()` の戻り値を指すオブジェクト |
| `retval.toInt32()` | 本来返るはずだった乱数値（ログ表示用に控えておく） |
| `retval.replace(0)` | 呼び出し元へ返る値を 0 に差し替える → `cpu = 0 % 3 = 0 = グー` |

引数を扱う `onEnter`（呼び出し前）ではなく `onLeave`（戻る直前）を使うのは、書き換えたいのが **戻り値** だからです。

#### 実行方法

Frida には、対象を新規に起動する方法と、すでに動いているプロセスに後から割り込む方法があります。

| コマンド | 説明 |
| --- | --- |
| `frida -l janken_always_rock.js -f ./janken` | 対象を **spawn（起動）** してから注入する |
| `frida -l janken_always_rock.js janken` | すでに起動中のプロセスに **アタッチ** する |

* `-f` … 対象を spawn（起動）してから注入する
* `-l` … 注入する JavaScript ファイルを指定する

## まとめ

今回は、次のような流れを一通り体験しました。

1. Claude Code で CLI ゲーム（じゃんけん）のバイナリを作成する
2. Ghidra（＋ Mecha Ghidra）でリバースエンジニアリングし、内部仕様を把握する
3. 解析結果をもとに、Frida で特定の関数（`rand()`）をフックし、戻り値を書き換えて挙動を変える

このように、リバースエンジニアリングで内部仕様を確認したうえで動的バイナリ計装を行うと、特定の関数をフックして戻り値や引数の値を変えることで、**プログラムを実行しながらその挙動を動的に変更** できます。

今回はシンプルなバイナリで実施しましたが、実際のマルウェア解析はこう簡単にはいきません。マルウェアは解析されることを前提に、さまざまな **アンチ解析（anti-analysis）** の仕掛けを備えています。代表的なものだけでも、次のようなものがあります。

* **アンチデバッグ**: `IsDebuggerPresent` などでデバッガの存在を検知し、見つかると処理を止める・偽の挙動を見せる
* **アンチ VM／サンドボックス**: CPU 情報、MAC アドレス、特定のドライバやプロセス名などから「解析用の仮想環境」を見抜き、無害なフリをする
* **タイミング検知**: 命令の実行時間を計測し、デバッガや計装で不自然に遅くなっていないかを調べる
* **パッキング／難読化**: 実体を暗号化・圧縮し、実行時までコードを現さない

こうした防御に対しては、リバースエンジニアリングによる **静的解析** と、Frida などによる **動的解析** を組み合わせて対抗します。たとえば「仮想環境かどうかを判定している関数」をフックし、本記事と同じ要領で **戻り値を『物理マシンである』と偽装** すれば、マルウェアは検知をすり抜けたと勘違いして本来の挙動を見せてくれます。今回 `rand()` に対して行ったことの、そのまま実戦的な応用です。

ただし、これは終わりのない **いたちごっこ** でもあります。Frida のような計装ツール自体を検知して解析を妨害しようとするマルウェアも存在するため、解析者はさらにその検知を回避する……という攻防が続きます。この攻防の感覚をつかむうえでも、まずは自作バイナリで基礎を体験しておくことには大きな意味があります。

冒頭でもお伝えしたとおり、これらの技術はかつては初学者にとって非常にハードルの高いものでした。しかし、Claude のような生成 AI を組み合わせることで、その第一歩は驚くほど踏み出しやすくなっています。本記事が、リバースエンジニアリングや動的バイナリ計装に興味を持つきっかけになれば嬉しいです。
