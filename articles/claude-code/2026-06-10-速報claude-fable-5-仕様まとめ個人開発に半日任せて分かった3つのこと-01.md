# はじめに

2026年6月、Anthropicから新モデル **Claude Fable 5** がリリースされました。Opusの上に新設された最上位ティアのモデルです。

この記事では、公式情報ベースの仕様まとめと、**個人開発のWebアプリの改善サイクル（要件定義→実装→検証→デプロイ→修正）を半日Fable 5に任せて回してみた感想**を書きます。「スペックは分かったけど、実際どうなの？」が気になる方向けです。

# TL;DR

- Fable 5はOpusのさらに上の新ティア。1Mコンテキスト・128K出力・$10/$50 per 1Mトークン
- **2026年6月9日〜22日はPro/Max/Team/EnterpriseプランにFable 5が追加費用なしで含まれる**（無料プランは対象外。APIは通常の従量課金）
- 半日任せて分かったのは次の3つ
  1. コーディング能力より**「開発サイクルを完走する力」**が違う
  2. **症状だけのバグ報告**で複合原因まで辿り着く
  3. スクショ・制約・報告など、**人間側の「伝えるコスト」が下がる**
- 体験としては「指示を出す→結果を見る」から「方針を渡す→定期報告を受ける」への変化

| 項目 | 内容 |

## 期間限定の追加費用なし提供（6月9日〜22日）

[公式発表](https://www.anthropic.com/news/claude-fable-5-mythos-5)によると、**2026年6月9日〜6月22日の期間、Pro / Max / Team / シートベースEnterpriseプランにFable 5が追加費用なしで含まれます**。

> From today through June 22, Fable 5 is included on Pro, Max, Team, and seat-based Enterprise plans at no extra cost.

:::note warn
**無料プランは対象外**です。また、API（従量課金）は期間限定提供の対象外で、最初から通常のAPI料金（$10/$50）で利用できます。
:::

**6月23日以降はこれらのプランからFable 5が外れ、利用にはusageクレジットが別途必要**になります（容量が確保でき次第、標準プランへの復帰を目指すとのこと）。

## サブスクリプションでの使用枠の消費

Pro/Maxプラン等のサブスクリプション経由（claude.ai・Claude Code）では使用枠内で利用できますが、**Opus 4.8の約2倍の速さで使用枠を消費**します。期間中でも「無制限に使い放題」ではない点に注意してください。

## Claude Codeでの使い方

Claude Code v2.1.170以降でサポートされています。セッション内で以下のコマンドで切り替えられます。

```
/model fable
/model claude-fable-5
```

## APIの注意点（Opus 4.7/4.8からの変更点）

```python
# OK: adaptive thinkingのみ対応
response = client.messages.create(
    model="claude-fable-5",
    max_tokens=16000,
    thinking={"type": "adaptive"},
    messages=[...],
)
```

- `thinking` は `{type: "adaptive"}` のみ対応（`budget_tokens` は使用不可）
- `thinking: {type: "disabled"}` を明示すると**400エラー**になる → 思考をオフにしたい場合はパラメータ自体を省略する
- `temperature` / `top_p` / `top_k` は使用不可（400エラー）

## Claude Mythos 5 との違い

同じ基盤モデルですが、Mythos 5はサイバーセキュリティ・生物学研究向けの安全制限を解除したバージョンです。現在は米国政府のProject Glasswingパートナーおよび選ばれた生物学研究者のみ利用可能で、一般公開は未定です。

# やったこと

個人開発のWebアプリ（Next.js + Supabase + Vercel）の改善サイクルを、**半日ほどFable 5に任せて回しました**。

:::note info
**重要な前提**：自律ループ・メモリ・スケジューリング等は**Claude Code（ツール側）の機能**であり、Fable 5専用ではありません。Opus/Sonnet等でも同じ仕組みは使えます。この記事で評価しているのは「仕組み」ではなく、ループの中でFable 5が見せた**判断の質**（段取り・原因分析・検証・リカバリ）です。
:::

以下、一利用者として触って感じたことです。仕様・価格・ベンチマークは公式情報をあたってください。

# 分かったこと1: コーディング能力より「開発サイクルを完走する力」が違う

一番印象的だったのは、コードの賢さそのものより**段取り力**でした。「アプリをブラッシュアップして」という曖昧な依頼に対して、自分で改善項目を洗い出して優先順位を付け、1サイクルごとに

**実装 → lint/ビルド → デプロイ → 本番確認 → 次の計画**

まで完走します。「次に何をすべきか」の判断がほぼブレないので、半日回しても人間側の介入は方向修正くらいで済みました。

「完走できる」と感じた理由を分解すると、次の2つが効いています。

## 検証を「言われなくても」やるから、結果を信用できる

テストが整備されていないプロジェクトでしたが、毎回lintと本番ビルドを通してから次に進むのは徹底していました。パーサーのような複雑なロジックを書いたときは、その場で使い捨ての検証スクリプトを書いて実データと突き合わせ、入出力を往復させて集計値が一致するところまで確認してから「完了」と報告してきます。

報告も「デプロイしました」ではなく**「デプロイがReadyになり本番URLが200を返すのを確認した」という実測ベース**です。報告を信用してよいという感覚があるので、人間が毎回裏取りをする必要がなく、サイクルが止まりません。

## トラブルで止まらず、代替経路を探す

途中、GitHub→Vercelの自動デプロイが発火しないトラブルがありました。このとき「pushは届いているか」をまず確認してWebhook側の問題と切り分け、**CLIからの手動デプロイに切り替えて目的を達成**しました。

エラーで停止して人間に丸投げ、ではなく「目的に対して別の手段はないか」を探す。このリカバリの仕方は人間のシニアエンジニアっぽさがあり、自律サイクルが途中で空転しない大きな理由になっていました。

# 分かったこと2: 症状だけ伝えれば複合原因まで辿り着く

バグ報告で一番面倒なのは再現手順の整理ですが、Fable 5には「スマホでボタンが押しにくい」「アイコンが変わらない」「画面の切り替えが重い」のような**症状だけ**を投げれば足りました。原因の切り分けが正確で、しかも**原因が1つだと決めつけない**のがポイントです。

| 症状として伝えたこと | Fable 5が特定した原因 |
|:--|:--|
| スマホでボタンが押しにくい | iOSのホームインジケータとタップ領域の重なり（safe-area未対応）と特定して修正 |
| アイコンが変わらない | ①参照先の画像が実在しない ②認証ミドルウェアが画像リクエストまで弾いている、という**複合原因**を両方発見 |
| 画面の切り替えが重い | 認証チェックの二重実行・直列クエリ・ローディングUIの不在、と**多層の原因**に分解して全部に対処 |

特に「アイコンが変わらない」のケースは、片方だけ直しても症状が消えない複合原因でした。原因を1つ見つけた時点で満足せず、「これで全部か？」まで検証してから直してくるので、**直したはずなのに再発する、というやり取りがほぼ発生しません**。

再現手順を書かなくても伝わる・1往復で原因まで掘り切ってくれる、という2点で、バグ報告の心理的コストがかなり下がりました。

# 分かったこと3: 人間側の「伝えるコスト」が下がる

3つ目は、コミュニケーションの形が変わったことです。仕様書を書く・前提を繰り返す・報告を読み解く、という人間側のコストが目に見えて減りました。

## スクリーンショットが最強の要件定義書になる

参考にしたいアプリのスクショを数枚渡して「こんな感じにして」と言うだけで、画面ごとの機能を読み取って複数回分の開発計画に分解してくれました。ヘッダーの構成、グラフの形式、設定画面のトグルや並び替えUIみたいな細部まで拾ってきます。

**文章で仕様を書くよりスクショを投げる方が速くて正確**、というのは発見でした。逆方向もあって、自分で生成した画像を自分で開いて目視確認してから提出する、という動きもしていました。視覚情報を入出力の両方で使いこなしている印象です。

## 一度言った制約をずっと守る

途中で「この機能はコストがかかるから廃止して。今後も使わないで」と伝えたら、機能の削除・ライブラリのアンインストール・ドキュメントの掃除までやった上で、**以降の提案からその系統の機能が消えました**。

長いセッションでは「前に言ったのに忘れてる」が積み重なって信頼を削っていくものですが、それが無い。制約を一度だけ伝えればいい、というのは運用コストとして地味に大きい差です。

## 報告が読みやすいから、流し読みで追える

結論が先頭に来て、やったこと・確認したこと・残課題が分かれているので、毎回の報告を流し読みで追えます。専門用語の使い方も自然で、日本語で開発を進めることに全くストレスがありませんでした。

定期報告を受けながら回す働き方では「報告を読むコスト」が意外とボトルネックになるので、ここの品質はサイクル全体の速度に直結します。

# 気になった点・注意点（正直ベース）

- 自律ループは便利ですが、その分**トークン消費は積み上がります**。回す価値があるタスクかは見極めた方がいいです
- 自律実行はローカルセッション上で動くので、**PCを閉じたりスリープすると止まります**（ここはモデルというより実行環境の話）
- DBの権限操作や実機での体験確認など、構造的にAIが触れない部分は最後まで人間の仕事として残ります。むしろ役割分担が明確になりました
- 医療・統計系の参考数値などを実装に使う際は「概算・目安」と明示はしてくれますが、**正確性が必要な領域では人間の裏取りが必要**です
- 提案が積極的なので、放っておくとスコープが広がりがちです。バックログを共有ファイルにして人間が並び替えられるようにする運用が相性良かったです

# 総評

Fable 5は「優秀なコード生成器」ではなく**「開発サイクルの運転手」**として使えた、というのが半日使った率直な感想です。

人間の仕事は次の3つに収束していきました。

1. **方向を与える**（スクショ・制約・優先度）
2. **実機で触ってフィードバックする**
3. **権限的にAIが触れない操作**

「指示を出す→結果を見る」の往復から、**「方針を渡す→定期報告を受ける」という働き方に変わる**のが、このモデルの一番の体験変化だと思います。

# まとめ
