## はじめに

Claude Codeで実装が速くなると、「並行して走らせるプロジェクトの数を増やしたくなる」という誘惑が生まれます。筆者は技術スタックも目的も異なる3つのプロジェクトを並行開発する中で、\*\*品質を落とさず速度を維持する鍵は、AIの性能ではなく人間側の「タスクスイッチの作法」と「コンテキスト設計」\*\*にあると実感しました。

本記事では、この2点を軸に、Claude(claude.ai)のプロジェクト機能との連携も含めて、実際に効果があった手法・なかった手法を簡潔に整理します。Claude Codeの基本操作の解説ではなく、すでに使っている人が運用を一段洗練させるための実践記です。

## 並行開発で起きる典型的な問題

* **コンテキストの汚染**: 同じ感覚でディレクトリだけ`cd`して会話を続けると、前のプロジェクトの方針(例: 厳格な型安全志向)が別プロジェクト(例: スピード優先のMVP)に紛れ込む
* **設計判断のブレ**: 長時間セッションを続けると、初期に決めた方針(例: DB正規化)が機能追加のたびに崩れていく。プロジェクトを切り替えるほど「前回何を決めたか」の確認を省略しがちになり加速する
* **再開時の思い出すコスト**: 数日空けると、自分もAIも前回の意図を思い出すのにコストがかかる。3プロジェクト以上だと「Aで決めたことをBだと取り違える」事故も起きる

これらはすべて「コンテキストの管理コスト」に帰着します。以下、これを下げる具体策です。

## タスクスイッチの実践パターン

**1セッション=1プロジェクトを徹底し、切り替え時は必ずセッションを終了して新規に立て直す**のが最も効果が大きい原則でした。会話履歴を残したまま`cd`だけで移動すると、前述のコンテキスト汚染が起きます。

中断前には、AI自身に進捗をまとめさせ`PROGRESS.md`として残します。書く内容は3点だけに絞ります。

* 完了したこと
* 次にやるべきことと理由
* 保留にした判断（なぜその実装にしなかったか ― 最も早く消える情報）

再開時は、いきなり実装させず次の順序を踏みます。

1. `PROGRESS.md`を読ませて状況を要約させる
2. 要約が自分の認識と一致するか確認する
3. 今回やることをAIに宣言させてから着手する

特に3が重要で、誤解があれば実装前に気づけます。実装後に気づくと手戻りが大きくなります。

なお、**1日に切り替えるプロジェクト数は2つまでが損益分岐点**でした。3つ以上を回すとウォームアップのオーバーヘッドが実装時間を圧迫し、総スループットはむしろ落ちました。

## CLAUDE.mdによるコンテキスト設計

書ける情報は無限にありますが、詰め込みすぎると読み飛ばされます。効果が高かった順は次の通りです。

1. **検証コマンドの固定化**
2. **やってほしくないこと**
3. ディレクトリ構造・命名規則・技術選定の背景

1と2が圧倒的に効果が高く、3は「あれば助かる」程度でした。

**やってほしくないこと**は、実際にAIに「やられて困った」経験から逆算して1行ずつ追記する運用が現実的です。例:

* 既存マイグレーションを直接編集しない(新規追加する)
* 確認なしに依存関係を追加・削除しない
* テストが落ちた状態でコミットしない

**検証コマンドの固定化**は最も効果が大きい工夫でした。

```
## 実装完了の確認手順
1. `npm run lint`
2. `npm run typecheck`
3. 対応するテストを実行
4. すべて通った場合のみ「完了」と報告する
```

これを書く前は、型エラーが残ったまま「実装しました」と報告される事態がよく起きていましたが、書いた後は大きく減りました。

並行運用が進むと、**共通ルール(コミット書式、Git運用)はグローバル設定、個別の技術スタックやドメイン制約はプロジェクトごとのCLAUDE.md**という分離が必要になります。分けずに全プロジェクトのCLAUDE.mdに同じ内容をコピーしていた時期は、共通ルールの変更ごとに全プロジェクトを修正する手間が発生していました。

## claude.aiのプロジェクト機能との連携

Claude Codeだけで完結させず、claude.aiの「プロジェクト」機能を組み合わせると役割分担が明確になります。

**役割分割**: 設計や要件整理など「考える」作業はclaude.aiのプロジェクト内チャットで行い、固まった方針をCLAUDE.mdやIssueに落とし込んだ上でClaude Codeに実装させる、という流れにしています。Claude Codeのセッション内で延々と設計議論をすると、前述の「長時間セッションでの設計ブレ」が起きやすいため、議論と実装のチャネルを分けることがそのまま防止策になります。

**知識の蓄積**: プロジェクトごとにclaude.aiのプロジェクトを1つ用意し、競合調査・ペルソナ設計・過去の意思決定の経緯など、コードに落とし込む前の背景情報をプロジェクトナレッジとして蓄積しています。CLAUDE.mdには「今どう実装すべきか」という運用ルールを置き、claude.aiのプロジェクトには「なぜそうするのか」という背景を置く、という分担です。これにより、CLAUDE.mdが肥大化せずに済み、背景を聞かれたときはclaude.ai側のチャットを遡れば再現できます。

## 検証重視の運用フロー

AIの提案は提示された時点で「もっともらしく見える」ため、そのまま採用するのは危険です。実装直後に\*\*「このアプローチを選んだ理由と、検討した他の選択肢」を説明させる\*\*ワンステップを必ず挟みます。説明が曖昧、あるいは代替案が存在しない場合は再検討のシグナルです。

マイグレーションのような取り消しにくい操作は、AIに一括実行させず必ず差分を人間が確認します。確認するのは「対象が意図した範囲に限定されているか」「データを破壊する操作が含まれていないか」「ロールバック手順があるか」の3点です。

差分レビューの深さは、変更の種類で基準を分けています。

| 変更の種類 | レビューの深さ |
| --- | --- |
| UIの見た目調整 | 動作確認のみ |
| ビジネスロジック | 差分を全行読む |
| DBスキーマ・マイグレーション | 差分を全行読む+ロールバック確認 |
| 依存パッケージの追加 | 追加理由を確認した上で許可 |

基準を明文化しておくことで、「今回はどこまで見るべきか」を都度考える判断疲れがなくなります。

## 効果のあった工夫／なかった工夫

**効果が高かったもの**

* セッションのプロジェクト単位での完全分離
* CLAUDE.mdへの「やってほしくないこと」の追記運用
* 検証コマンドの固定化
* claude.aiプロジェクトとClaude Codeの役割分割(議論と実装の分離)

**期待したほど効果がなかったもの**

* CLAUDE.mdを最初から詳細に作り込むこと: 最小限から始めて運用しながら追記する方が結果的に質が上がりました
* 並行プロジェクト数を増やすこと自体: 3つ以上の同日切り替えはオーバーヘッドが勝り、総スループットは下がりました

## まとめ

並行プロジェクト開発のボトルネックは、AIの実装速度ではなく**人間側のタスクスイッチの作法とコンテキスト設計**でした。要点は次の4つです。

* セッションをプロジェクト単位で分離する
* 中断・再開のプロトコルを決めておく
* CLAUDE.mdは「やってほしくないこと」と「検証手順」を中心に育てる
* claude.aiのプロジェクトで設計議論と背景知識を蓄積し、Claude Codeには実装に専念させる

AI駆動開発は速さが強調されがちですが、並行運用のフェーズでは「どこまで型を決めて運用するか」が問われます。同じように複数の個人開発プロジェクトを抱えている方の参考になれば幸いです。
