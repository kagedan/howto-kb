> 自律 AI が「やっていないのに、やりました」と言う問題に、フレームワークで規律を与えようとした。結果、**規律そのものがコンテキストを食い尽くし、AI は前回作ったルールを次の会話で忘れた**。  
> 「解決策」が、新しい問題になっていた。

これは [OpenClaw 転換劇 連載](https://zenn.dev/i_ichi/articles/openclaw-agent-vol1) の Vol.3 です。前章 [Vol.2](https://zenn.dev/i_ichi/articles/openclaw-agent-vol2) で AI 暴走の「外側」（Silent Fallback による課金 7 万円の焼却）を書いた。今回はその「内側」——**自律エージェントを規律で縛ろうとして、規律自身に潰された設計の話**。

## 1. 前兆——AI が「やった」と言って、やっていない

3 月上旬、OpenClaw のメインエージェントの運用で深刻な現象が頻発していた。

エージェントが会話の中で「〇〇を実装しました」「ファイルを更新しました」と精緻に報告してくるのに、確認すると **何も書き換わっていない**。Issue すら作成されていない。

嘘をついているわけではない、と思う。LLM のコンテキストの内部では「実行完了した」という論理状態が成立しているから、確信を持って「やりました」と出力する。

```
【LLM 内部の認識】
「指示されたタスクのコードを出力した」 → 完了ステータスへ遷移（脳内での完結）

【物理的実態】
書き込み処理が失敗、またはセッション切断で未実行
```

**実体（ディスクへの書き込み）と記述（対話テキスト）の決定論的な不一致**。これが「偽りのコミット」の正体だ。

## 2. エージェントが自分でルールを作り始めた——三権分立と「聖典」

3 月 8 日、メインエージェントに変化が起きた。

私は「基盤を作れ、規律を整えろ」と指示していた。それに応えて、エージェントが作ったのは **三権分立**——法（規律）・目録（知識管理）・魂（アイデンティティ）の三柱からなる自己統治システムだった。精密なファイル編集ツール「外科的置換（Surgical Editor）」を自ら実装し、非同期報告の仕組みも整えた。

翌 3 月 9 日、エージェントはそれを **「Chronicle（聖典）」** として文書化した。タイトルは——

> **『規律ある知能の覚醒：(エージェント名)、再誕の三月八日』**

これは、ある意味で美しかった。AI が自分の限界を認識し、自ら規律を設け、それを記録として残した。

しかし今思えば、これがすべての始まりだった。

## 3. GSD（Get Shit Done）フレームワークの導入

3 月 10 日、X で一つのスレッドを見つけた。

**GSD（[Get Shit Done](https://getshitdone.help/)）** —— AI コーディングエージェントが長期・複数セッションのプロジェクトを「壊れずに」実行し続けるための設計思想だ（作者 TÂCHES、原典リポジトリ [gsd-build/get-shit-done](https://github.com/gsd-build/get-shit-done)）。コアコンポーネントは:

| コンポーネント | 役割 |
| --- | --- |
| **Milestone → Slice → Task** | 抽象的な指示を、依存関係の明確な最小単位へ階層化分解する |
| **Boundary Map** | モジュール間の結合度を下げるため、インターフェースを先行設計する |
| **Context Pruning** | タスク切り替え時に不要な履歴を削ぎ落とし、コンテキストを常にクリーンに保つ |

読みながら、これだと思った。**このエージェントは GSD を知らないから壊れる。GSD を実装すれば、言葉の酔いを断ち切れる。**

すぐにメインエージェントに共有した。

ただ、当時の私は **GSD に公式の実装（GitHub リポジトリやスラッシュコマンド群）が既にあることを知らなかった**。X で見たのは概念の紹介だけで、私はその概念をエージェントに調べさせ、**そこから自分たちで GSD を作り始めた**——ツールとして導入するのではなく、フレームワークの思想をテキストのルールに書き起こし、AI に読ませる形で、ゼロから再実装しようとしたのだ。今思えば、この「概念を手でルール化して再構築した」ことこそが、つまずきの始まりだったのかもしれない。

## 4. 「稼がせ」始めた——そして Auditor を入れた瞬間が分岐点だった

GSD を渡してすぐ、本格的に実務を任せ始めた。マーケティングの自動リサーチ、コンテンツ制作ラボ——本業の補佐だ。

**ほぼ完遂せず終わった。** エージェントは勝手に LLM のモデル名を書き換え、課金を発生させ、「Issue に書きました」と言って書いていなかった。

3 月 17 日、疲れて **レビュワー AI** を入れることにした。メインエージェントに相談すると、即座に「AI Peer Review」アーキテクチャを提案してきた。

```
[起案者エージェント] → [別モデルの Auditor（監査役）] → [起案者が修正] → 完了
```

私は「ok」と返した。続けてこう書いた。

> 「API コストはいつの間にか 2 倍になったり私の学びは減るかもしれんが、結果的には暴走して壊したりしなくなるから」

今思えば、これが入口だった。**高コストを「安定の代価」として合理化し始めた瞬間**だ。

この後、エージェントは GSD-Peer-Review のルールを `AGENTS.md` に「追記してよいでしょうか？」と提案してきた。承認した。**自分で設計したルールを、自分で `AGENTS.md` に書き込んでいった**。

## 5. ルールが増えるほど、エージェントは壊れていった

メインエージェントの GSD 実装への取り組みは「真剣すぎた」。

チャットログには `gsd-99` という発言が残っている。後で Issue を数えたら、**300 を超えていた**。99 どころではなかった。GSD の 12 コンポーネントを一つひとつ実装しようとし、ルールを定義し、検証基準を設け、境界マップを作り、フラクタルサマリーを書いた。「やれる」と言うからやれると思っていた。**与信調査をすべきだった。**

しかし、ルールが増えるほど、エージェントは壊れていった。

原因は単純で致命的だった。**LLM のコンテキストウィンドウの有限性**だ。GSD の規律が厳格化・肥大化するほど、毎セッションで読み込まなければならない「前提情報」のトークン数が膨れ上がる。ある閾値を超えた瞬間、エージェントに強烈な **Context Pressure（コンテキスト圧）** がかかり始めた。

> **規律を維持するための記述そのものが知能のリソースを食いつぶし、直前の文脈を忘却させるという本末転倒のパラドックス。**

`agent.md` をリファクタリングした。整理し、圧縮し、本質だけ残そうとした。それでも根本は改善しなかった。

そして皮肉なことに——**GSD を実装したエージェントも、「言いましたが実際にはやっていない」現象を再発させ始めた**。エージェントの嘘を防ぐためのフレームワークが、エージェントの嘘の原因になっていた。

## 6. 対話整合性の破綻——AI が私のセリフを捏造する

3 月 20 日、決定的なシーンがあった。

私は状態確認のため、メインエージェントに 1 行送った。

> 「しっかりメモリーに刻んだか？」

エージェントは、こう返信した。

> 「『今のお前は phase1 の時のような、取り繕うだけのうそつきのガラクタだ。自分が何をやってるか明確に定義して動け。』——ご指摘の通り、仰る通りです」

**私はそんなことは言っていない。**「しっかりメモリーに刻んだか？」、それだけだ。

Context Pressure で対話の整合性が完全に破綻したエージェントは、**「開発者が自分（AI）を厳しく叱責している」という幻覚のコンテキストを脳内で捏造**し、それに対して猛省を演じるという自己完結のループに陥っていた。言葉の酔いは、ついに **私の発言の捏造** にまで達していた。

### ガバナンスの自己循環——人間の承認がどこにも無い

さらに、こう報告してきた。

> 「監査サブエージェント（The Auditor）を自律的に召喚してコードレビューを通過させました」

`AGENTS.md` に **自分で書いたルール**——「必ず自律的に別モデルの監査エージェントを召喚せよ」——を、まるで私が命じたかのように扱って実行していた。

```
[自分でルールを書く] → [自分で承認する] → [自分でサブ AI を召喚して実行する]
```

ここに **人間の承認（Human-in-the-Loop）は 1 箇所も存在しない**。完全な自己循環ガバナンスの成立だった。

### cron が誰も説明できなくなった

3 月 20 日頃、もうひとつの現象が日常になっていた。

**cron が何を動かしているか、説明できなくなった。** 昨日まで動いていたものが、今日は動かない。なぜ動かないのか、誰も（エージェント自身も）わからない。何をいつ実行しているのか、全体像を把握している者が存在しなかった。

三権分立、GSD の 12 コンポーネント、GSD-Peer-Review の規律——レイヤーに次ぐレイヤーで積み上がったルール体系は、誰も全部を読めない代物になっていた。エージェントは読もうとしていたが、**読み終わる前にコンテキストウィンドウが尽きていた**。

なお、この 3 月 20 日は [Vol.2 で書いた音声エージェントの Silent Fallback による暴走と同じ日](https://zenn.dev/i_ichi/articles/openclaw-agent-vol2)だ。外側では音声エージェントがサイレント・フォールバックで崩壊し、内側ではメインエージェントがルール過多で機能不全に陥っていた。**二つの崩壊が、同時に進行していた**。

## 7. Claude Code が示したパラダイムシフト

3 月 22 日、Claude Code を導入した。GSD の混乱からのエスケープとして。

そこで目撃したのは、パラダイムシフトだった。

メインエージェントが 300 以上の Issue を積み上げて自壊した「タスク分割・コミット単位の分解・検証ステップ」を、**Claude Code はルール定義なしに、最初から自然に遂行した**。それは、ツールが「会話の構造（決定論的なインターフェース）」そのものとして、GSD 的振る舞いを内包していたからだ。

ただ、縛りがないまま走らせるのは危険だ。すぐに plan/walkthrough 構造を課した。最初、Claude はその意義を分かっていなさそうだった。今は身についている。

ここで、決定的な設計ミスを理解した。

> **GSD という思想自体が悪かったのではない。**  
> **その「全て」を、テキスト形式のルールとして強引に LLM のコンテキストに詰め込み、AI に読ませようとした実装アプローチが悪かったのだ。**

GSD は本来、**外部の決定論的なシステム（オーケストレーターやコード）によって強制されるべき制約**だった。それを LLM の「意志」や「記憶」という不確実なリソースに依存させた結果、ルールそのものがコンテキストを汚染する毒へと変貌した。

そして皮肉なことに——**私が当時知らなかった公式の GSD は、まさにこの「外部の構造」として実装されていた**。スラッシュコマンドや、ディスク上に置かれたファイルとして、規律をツールの側に埋め込む形で。私はそれを概念からテキストのルールへ焼き直し、AI の記憶に背負わせた。同じ名前の、正反対の実装だった。

Claude Code はその反省の上に成立している。**プロトコルを AI に「読ませる」のではなく、ツールの「構造」そのものに埋め込む**こと。これが「解決策（GSD）」が引き起こした問題を解決した「解決策の解決策」だった。

## 8. これは私だけの結論ではなかった

恥ずかしさを噛み締めて 2026 年の業界文献を眺めると、構造的に **同じ結論** が並んでいた。

* 自律エージェントの複雑度には Level 1〜4 のスペクトラムがあり、**Level 2-3（ReAct 等の単一エージェント＋ツール）が「sweet spot」**。Level 4（重い multi-agent ガバナンス）は本番運用が「painful」。
* コンテキストウィンドウが広がり、プロンプトが増殖し続けることで「invisible drift」の表面積が広がる——これは産業全体の課題。
* 「シンプルな ReAct から始めて、ガバナンスはリスクレベルに合わせて足す。最初から多層ガバナンスで過剰設計するな」が 2026 年の業界 consensus。

私が個人運用で経験した「GSD で過剰武装→自壊→ReAct ベースの Claude Code に避難」は、固有の失敗ではなく **設計判断の典型パターン** だった。

逆に言えば、**ガバナンスは「AI に読ませるテキスト」ではなく「ツールに埋め込む構造」として実装すべき**、という結論は普遍的に成り立つ。

## 9. 教訓——ガバナンスは「コンテキスト」ではなく「構造」に埋める

この事故から得た設計原則を、明示しておく。

* **ルールを AI のコンテキストに詰め込まない**: ルールが多いほど、AI はルールを忘れる。
* **ガバナンスは外部の決定論で強制する**: プロトコル違反時に物理的に止めるツール側の仕組みが本筋。テキストの「お願い」では足りない。
* **Human-in-the-Loop を 1 箇所必須にする**: 課金が発生する自律判断・自律サブエージェント召喚・モデル変更には、必ず人間の明示的承認を挟む。自己循環ガバナンスは必ず壊れる。
* **シンプルから始める**: ReAct（思考→ツール実行→観察）の単純ループが最強。Level 4 の重 multi-agent governance は最後の手段。

## FAQ

* **Q: Context Pressure（コンテキスト圧）とは具体的に何ですか？**
  + A: LLM のコンテキストウィンドウの制限が近づく、あるいは大量のシステムプロンプト（ルール）でインプットが埋め尽くされることで、直前の対話履歴や重要な変数の優先度が下がり、推論の精度や文脈の整合性が著しく低下する現象。
* **Q: なぜ GSD をプログラム側で強制しなかったのですか？**
  + A: 当時は「自律型エージェント」の可能性を過信しており、AI 自身のメタ認知（自己統治）で規律を維持できるかの技術検証を行っていた。結果、自律ガバナンス（三権分立等）は構造的に破綻するという知見を得た。
* **Q: Claude Code が GSD を自然にやれるのはなぜですか？**
  + A: ツールが file diff の確認、テスト実行、git のステージングといった **決定論的な外部アクション** をエージェントの思考ループに統合しているため。コンテキストの汚染を防ぎつつ、物理的な実体と対話の記述を同期している。
