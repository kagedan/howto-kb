## まとめ（先に結論）

* **やったこと**：要望をまとめて投げる → AIが実装 → **AIがPlaywrightで実機検証＋スクショ** → 本番デプロイ、を1ループで回した

## AI開発の進め方（今回のループ）

やったのは雑に言うとこれだけです。

1. 改善要望を箇条書きでまとめて渡す
2. AIが関連ファイルを読んで実装
3. **AIがPlaywrightの検証スクリプトを書いて実機ビューで確認**（iPhone 12 相当のビューポートなど）
4. スクショを撮って目視レビュー
5. 問題なければ本番デプロイ

特に効いたのが 3 と 4。たとえば「スマホで横長が直ったか」は、感覚ではなく数値で確認させました。

```
// iPhone 12 相当(390px)で横スクロールが消えたかを数値で検証
const { sw, cw } = await page.evaluate(() => ({
  sw: document.scrollingElement.scrollWidth,
  cw: document.scrollingElement.clientWidth,
}));
console.log('横はみ出しなし:', sw <= cw + 1, sw, cw); // → true 390 390
```

「直したつもり」を `scrollWidth === clientWidth` で裏取りしてからスクショ、という流れをAIが自分で組んでくれるのは普通に強かったです。

## コケたところ・学び（ここが本題）

### ① 「実装できた」が、実は効いていなかった

「よく使う仕訳のデフォルトプリセットを用意したい」という要望で、フロントの初期データにプリセットを仕込みました。ゲストモードでは**ちゃんと表示され、検証も通った**。

ところが実機（ログイン状態）で見ると出てこない。原因は単純で、

* ゲスト／ローカルモード：データは `localStorage`（＝フロントの初期値が効く）
* **ログインユーザー：データは API 経由の DynamoDB 由来（＝フロントの初期値は効かない）**

つまり「新規ユーザーにデフォルトを配る」なら**バックエンド側の初期化＋既存ユーザーへのバックフィル**が要る、という話でした。検証はゲストで通っていたので、AIも私も「できた」と思い込んでいた。

結局この機能は一旦**巻き戻し**ました。教訓は身も蓋もなくて、**「どのデータ経路で効く実装なのか」は人間が詰めないと、AIの“検証OK”は条件付きの OK でしかない**。

### ② 本番デプロイ事故を grep でガードする

このプロジェクト、環境ファイルが2つあります。

* `.env.production` … 本番（prod アカウント）
* `.env.local` … dev アカウント（手元の動作確認用）

Vite の本番ビルドでは `.env.production` が `.env.local` を上書きする……**はず**。でも「はず」で本番に dev のAPIを向けたビルドを上げたら、利用者が dev のDBに繋がる事故になります。

なので、ビルド成果物そのものを grep して確認してからデプロイしました。

```
# distに本番APIが埋め込まれ、devが混入していないかを確認
grep -rl "PROD_API_ID" dist/assets   # 本番API → ヒットすればOK
grep -rl "DEV_API_ID" dist/assets   # dev API  → 何か出たら事故。デプロイ中止
```

`DEV_API_ID`（dev）が空、`PROD_API_ID`（prod）がヒット、を見てから S3 へ sync。\*\*「設定の優先順位を信じる」ではなく「成果物を検証する」\*\*に倒すのは、AI云々と関係なく本番反映の基本だなと再認識しました。

デプロイ自体は静的ホスティング（S3 + CloudFront）なので、

```
aws s3 sync dist s3://<app-bucket> --delete
aws cloudfront create-invalidation --distribution-id <dist-id> --paths "/*"
```

CloudFormation / SAM は触らない（CloudFront の独自ドメイン設定が剥がれる事故を避けるため、構成変更とコンテンツ更新を分離）。

### ③ 環境ハマり：Bash と PowerShell で `.aws` が別だった

地味に一番ハマったやつ。Windows 上で、Bash 側と PowerShell 側で参照する `~/.aws`（SSOトークンのキャッシュ）が別物でした。

`aws sso login` を片方のシェルで実行しても、もう片方のシェルからは「トークン切れ」のまま。AWS CLI のコマンドを叩くシェルと、ログインしたシェルを揃える必要がありました。AIにログを切り分けさせて原因に辿り着きましたが、こういう**環境固有のハマりは事前知識がないと延々溶ける**ので注意。

### ④ 「メールが届かない」はコードのバグではなかった

新規登録の確認メールが届かない件。AIに調査させたら、CLI 一発で原因が出ました。

```
aws sesv2 get-account --region ap-northeast-1
# → "ProductionAccessEnabled": false
#    "ReviewDetails": { "Status": "DENIED" }
```

Amazon SES が**サンドボックスのまま**（検証済みアドレスにしか送れない）で、しかも本番アクセス申請が**却下**されていた。コードは正しく、送信経路（Cognito → SES）も正しく繋がっていた。

応急処置として、上記④の「メール登録の一時停止＋Google誘導」を入れた、という流れです。**“バグ”の前に“環境・権限”を疑う**のはAIの方が早くて素直でした。

## 所感

AIエージェントに任せて良かったのは、**実装 → Playwrightで検証 → スクショ → デプロイ**まで一気通貫で回せたこと。レビューが「コードを読む」から「スクショと数値を見る」に寄って、体感の速度がかなり上がりました。

一方で、人間が握り続ける必要があったのは結局この3つでした。

* **仕様の妥当性**：どのデータ経路で効くのか（①のプリセット）
* **本番反映の安全**：成果物の検証、構成変更とコンテンツ更新の分離（②）
* **環境固有のハマり**：シェルごとの認証キャッシュ、権限・外部サービスの状態（③④）

逆に言うと、ここさえガードできれば、個人開発のアップデート速度はかなり上げられる、という手応えでした。
