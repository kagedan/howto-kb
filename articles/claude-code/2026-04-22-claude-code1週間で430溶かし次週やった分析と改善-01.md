# はじめに

2026年4月7日〜13日の1週間、Claude Codeの利用料金が $430.85（約6.5万円）に達した。これは前週（$361.92）からさらに増加しており、通常の週（$60〜$150程度）と比較すると 3〜7倍 の異常値でした。

本記事では ccusage session のデータをもとに、何がコストを押し上げたのかを確認し、得られた教訓をまとめます。。

## 何がダメだったのか

### 1. サブエージェントのモデル指定漏れ

subagentセッションが全120セッション中117を占め、コスト$257.80。多くがHaikuで安く済んでいるが、一部のサブエージェントが Opusで起動されてしまい 、$83, $57,$34のセッションになっていた。  
cookスキルなどで自動的にTaskツールを使う際、model: "haiku"の指定が漏れると親セッションのモデル（Opus）を引き継いでしまうので、今後気をつける。

### 2. Figma MCPによるコンテキスト膨張

4/10のセッションでは、figma MCPを使ってfigma to codeの精度を確認したり、UIの改修面で何度も同じセッション内でfigmaの全体を読み込ませていました。

##### やってしまっていたこと。。

* 精度が低かったので、何度も繰り返しfiguma全体&コンポーネント単位で読ませていた
  + なんで精度が低いの？って聞いたら、「どのコンポーネントがどのfigmaの要素と一致するのかわかんないから、どれを修正したらいいかわからない」と言われました。
  + タグにdata-design属性をつけて、figmaのnode-idと紐づけるようにする。
* 同じセッションで上記を行っていた  
  結果として、セッションのコンテキストウィンドウには「figmaの全体情報 ×検証回数分」のデータが蓄積され、毎ターンのCache Readがめっちゃ増えた。これが1.25億トークンの Cache Readの要因の1つ。

#### 3. 複数プロジェクトの同時進行とコンフリクト解消の繰り返し

この週は3つのプロジェクトを並行して進めていた：

* dashboard: $236.87（55%）
* default-data: $173.05（40%）
* japan-cityos-sdk: $18.65（4%）

プロジェクト切り替えのたびに新しいセッションが立ち上がり、コンテキストの再構築コストが発生。さらにdevブランチとのコンフリクト解消が8回も繰り返され、merge→テスト→再修正→再マージのループで会話が長大化しました。  
github上のCIでのテストが長いので監視するコストも結構かかったと思います。。  
コース分けせずに、思いついた課題をissue化して実装するとコンフリクト起きやすいので気をつける。

#### 4. 会話を切らずに継続しすぎ

Cache Readが6.51億トークン（全体の95%）という数字が示すように、長い会話を切らずに続けた結果、毎ターンの読み込みコストが累積した。

**改善後の効果（4/14〜20の週）**

| 指標 | 爆発週 (4/7-13) | 改善後 (4/14-20) | 削減率 |
| --- | --- | --- | --- |
| コスト | $430.85 | $60.77 | -86% |
| トークン | 6.83億 | 1.99億 | -71% |
| セッション | 120 | 92 | -23% |

セッション数は23%しか減っていないのにコストは86%削減。  
この週は、figma to code を試していないから、モデル変更 ＋ data-design属性付与（figmaのnode-idとの紐付け）後に、再度比較してみる。

## 教訓まとめ

### やるべきだったこと

#### 1. Opusは必要な時だけ使う

調査・実装の大半はHaikuやSonnetで。Opusはアーキテクチャ判断や複雑なリファクタリングに限定する。実装時にHaikuを使うと精度が落ちた気がしたけど、/order はHaikuとか/cookはSonnetとかって決めておくようにする。

#### 2. サブエージェントには必ず model: "haiku" を指定する

指定漏れでOpusが走ると1セッション$50〜$100が一瞬で飛ぶので気をつける。。

#### 3. Figma MCPは計画的に使う

全体読み込みは必要な時だけ。精度が低い時は、「なんで？」「何がわかりづらかったの？」ってすぐ聞く。

#### 4. 長い会話は適度に /clear で切る

コンテキストが膨れるとCacheReadだけで数十ドルになることを体感。。  
実装がそのセッションで上手くいかなくても、何がわからなかったかclaudeに確認して、セッションを早めに閉じる。原因を潰してから別セッションで対応する。を徹底する。

#### 5. セッションごとのタスク数を制限する

コース分けして実装すると、どうしてもかなりトークンを消費するので、すぐ100%近くなる。  
この辺どうしようか考える。

#### 6. 1日のコスト上限を意識する

$30/日を超えたら使い方を見直すべき

#### 7. ccusageを毎日チェックする

異常値に早く気づけば4/10の$172は防げたはずなので、毎日今日使った分は把握する。

#### 8. 実装・調査スキルのフロー見直し

コストかかるフローや重複、不要なところや複雑になってしまってるところを確認するべきだった。

### やってはいけなかったこと

1. サブエージェントのモデル指定を省略する
2. MCPで同じリソース（figmaファイル全体など）を何度も読み込まない
3. 「まだ終わってないから」と会話を切らずに積み重ねる
4. コストを確認せずに週を通す
5. コンフリクトが起きやすい実装フロー
