## はじめに

僕自身、個人開発や業務で AI を使った開発を行っています。その経験をもとに「AIに自走してもらうために何が必要か」を考える機会があったので、記事としてまとめてみました。

ここで改めて感じたのは、**AIが自走できるかどうかは、ツールの使い方以上に「コードベースの設計」の影響が大きい**ということです。重要なのは「良いコードを残し、悪いコードを入れさせないこと」だと考えました。

AIに任せても意図通りのコードを出してくれない方や、AI駆動開発を進める中でコードの品質に課題を感じている方は、ぜひご覧ください。

## そもそも「AIが自走できる」とは？

事前に「AIが自走できる」がどのような意味か整理させてください。僕の中では、次の2つが揃っている状態です。

* AIが書いたコードに、細かい指摘がほとんど不要である
* 詳しく指示しなくても、周辺コードを読み込みつつ意図を汲んでくれる

つまり、**既存の設計に沿ったコードが、少ない指示でも自然と出てくる**状態です。

逆に言うと、細かく指示しないと意図通りのコードが出てこない、出てきたコードに毎回同じ指摘をしている、という場合は、AI側ではなく環境側に改善の余地があると考えています。

## AIに自走させる時のTips

まず、日々の開発でAIに自走させるために実践している細かい Tips を紹介します。

### Planを念入りに作る

実装に入る前の Plan（実装計画）の質が、そのままアウトプットの質に直結します。僕が意識しているのは、**複数案を出させて、メリット・デメリットを比較する**ことです。あえてマイナーなアーキテクチャも案に含めさせると視点が広がり、「この方向性は無い」という判断もしやすくなります。

![複数のアーキテクチャ案を出させて比較検討した時のPlanドキュメント](https://static.zenn.studio/user-upload/deployed-images/5dda6942b65b5edef4a38bdf.png?sha=7fc3aeee8b09e5ff72a31808e67f6833a253d0e6)  
*実際に複数案を出させて比較検討した時のドキュメント*

また Planを作るうえで、僕が特に意識しているコツが2つあります。

1. 実装前にデータの流れやクラス図をAIに出力させ、認識の齟齬がないか確認する。
2. 不足している情報をAIの方から質問させる。

これらにより AI との認識が一致し、かつ指示漏れを減らした状態でタスクを任せられます。

### AI自身に動作確認をさせる

Chrome DevTools MCP や Playwright MCPを使うと、AIが実際にブラウザを操作して、画面の動作確認まで行えるようになります。

![Chrome DevTools MCPでAIがログインフローを操作して動作確認している様子](https://static.zenn.studio/user-upload/deployed-images/2ccf3da411bd9845084b6a69.gif?sha=454af10dc7308e343eae818dad8abdd57a0eb343)  
*Chrome DevTools MCPで、AIが実際にログインフローを操作して動作確認している様子（4倍速）*

これにより、最低限の正常系はAI自身でテストしてくれます。認証フローなどよく使う手順は Skill にまとめておくと、不要な Try Error を減らせて効率的です。「最低限は動く」ことが担保出来て、人はレビューに集中できるようになります。

### レビュー指摘はメモさせる

AIの出力に指摘をする際、まず「なぜダメだったか？」をAIに答えさせています。原因を説明させることで、次にどう直すべきかまで提案させています。

そのうえで、提案内容やそもそもの原因を CLAUDE.md や Skill に追記させます。これを繰り返すことで、同じミスをしなくなり、少しずつ意図通りに動いてくれるようになります。

同じ種類の指摘が増えてきた際は、その観点だけ切り出してレビュー専用の Agent を作るのも有効です。見落としが減り、レビューの質も安定します。

### 定型作業は Skill 化する

AI は Skill の中で Bash や Python のスクリプトを実行できます。そのため、マイグレーションファイル作成など「コマンド実行」の作業も Skill化 して任せられます。定型作業こそ、AIに任せるチャンスです。

![マイグレーションの生成・適用・状態確認をSkill内のスクリプトにまとめた例](https://static.zenn.studio/user-upload/deployed-images/6a2c7c378a4ed9678ce22c71.png?sha=74e7df696086bbed801eee93af0b467074f0464e)  
*マイグレーションの生成・適用・状態確認をSkill内のスクリプトにまとめた例*

チェック用のコマンドや静的チェックを Skill に組み込んでおけば、実装後の確認まで AI に任せられます。  
また、マイグレーションファイルの作成漏れなどの「機械的に検知できるミス」は CI でも防ぐようにしています。AI に都度注意させるより、仕組みで防ぐ方が確実です。

## AIが自走できる環境を作るには？

ここからは、なぜコードの質が重要なのか、そしてどのようにソースコードを整えていくか、の話をします。

### なぜ既存実装のコードの質が重要なのか？

AIは既存コードを見て、新しいコードを作ります。既存実装をお手本にして、水平展開していきます。

だからこそ、**良くも悪くも、既存実装の設計を踏襲します**。良い設計なら、それが広がっていきます。悪い設計なら、それが増殖していきます。「既存を踏襲する」という起きている事象自体は同じですが、元のコードの質によって結果が真逆になります。

ここで大事なのは、「**規模が小さい段階からどれだけコードベースを整えられるか**」だと思っています。小さいアプリのうちは修正箇所が限られているので、設計の問題が顕在化しません。しかしスケールしようとした途端、バグが大量に発生します。

### 「動くコード」と「いいコード」は違う

ここから、僕の失敗談を少しお話しします。

個人開発で実装をAIに丸投げしていたところ、細かいバグが大量発生しました。

動くけれど壊れやすいコードが、次々と生まれていく感覚です。「〇〇のバグを直して」と伝えれば直りはします。ただ、それは正しい設計に直っているわけではありません。無理やり動かすためのパッチが当たっただけなので、修正するたびに別のバグが生まれます。気づいた時には、謎のIF分岐が大量に増殖していました。

パッチを重ねたコードは設計としての正しい意味を持ちにくく、コードの質はどんどん低下していきます。さらに厄介なのは、**AIもそのコードを読んで、同じ書き方を踏襲する**ことです。悪いパターンが既存コードを通じて再生産され続けます。

### AIが設計意図を読み取れる、一貫したコードを残す

では、どういうコードを残すべきか。僕は「AIが設計意図を読み取れる、一貫したコード」だと考えています。

設計方針がないと、業務ロジックが各所に散らばります。「何を共通化するか」「共通処理をどこに置くか」を明示しないと、同じロジックがあちこちに書かれるようになります。AIは既存コードを踏襲するので、散らばったパターンが一瞬で拡散します。

逆に一貫したパターンで書かれていて、「なぜこの設計なのか」が読み取れるコードであれば、AIは少ない指示でもそれに沿ったコードを出してくれます。逆に、場所によって書き方がバラバラだと、AIはどれを踏襲すべきか迷ってしまいます。

僕自身、設計が古くなってきたら、リファクタリングやリアーキテクトも積極的に行うようにしています。古くなった設計を放置せず、AIが正しく設計を読み取れる状態を維持することも、自走できる環境づくりの一部だと感じています。

### バグりにくい設計・実装を行うためのレビュー観点

これらの前提を踏まえ、AIが書いたコードをレビューする時は、バグりにくい設計・実装になっているかを次の観点で見るようにしています。

* **今後の展望を踏まえたコードになっているか？**：拡張性を見据えた設計になっているかをチェックする
* **業務ロジックが重複していないか？**：同じ責務のロジックが散らばると、変更時にデグレが発生するので、それを防ぐ
* **既存の設計に沿った形になっているか？**：設計から外れたコードが入ると、AIがそれも踏襲してしまう

これらの観点を意識すると、コード品質を維持しやすいと思います。同じ観点で AI にレビューさせるのも有効です。

## AIに自走させるために意識したいこと

最後に、AI 時代での「心構え」の話を少しだけさせてください。

### その生産性向上は、正しい方向ですか？

ここで一つ問いかけをさせてください。

**「AIを導入して、チームの開発スピード・生産性は上がりましたか？」**

おそらく、PRの数やリリースの頻度が増えたチームは多いと思います。ただ、もう一歩踏み込んで考えたいのが、**その生産性は、正しい方向に向かっていますか？** という問いです。

### PRの数だけを追う落とし穴

PRの数は増えたものの、その中に「**本来不要だったPR**」は含まれていないでしょうか。

設計が甘いコードは負債になりやすく、設計段階で潰せたはずのエッジケースが残り、バグが頻発します。すると、従来は不要だったバグ対応で他のメンバーの時間も奪われ、チーム全体の生産性が落ちていきます。

質の低いコードのしわ寄せはQAに集中し、チーム全体が疲弊していきます。

また、バグ対応のたびにデグレ防止のパッチ的な修正が重なり、可読性も設計意図も失われて、コードの品質はさらに落ちていきます。その後 AI がそのコードを踏襲し、負の循環が加速します。

![設計が甘い状態からPR量産、QAでバグ大量発覚、手戻り・修正PRでチーム疲弊、コード品質低下、それをAIが踏襲して振り出しに戻る負のループ図](https://static.zenn.studio/user-upload/deployed-images/1de8a38051692796f1afecc6.png?sha=ea93741c268c4c7464a229f49256db594255eee7)

この循環に入ると、「コードを速く沢山書ける」ことがそのままチームの成果に繋がらなくなります。

個人的には、**「従来の価値基準を転換する」**必要があると思っていて、**「コードを速く沢山書ける」だけでは通用しなくなってきている**と感じています。

### 「とりあえず動く」で終わらせない

とはいえ、全てを人が細かく見るのは現実的ではありません。僕が意識しているのはメリハリです。

レビューの際は「動く」で終わらせず、「最適な設計か？」まで見るようにしています。ちょっとした設計の甘さが、AIの水平展開で一気に広がる時代だからです。

そのため根本の設計は、ある程度まで人が固めておきます。境界・責務・拡張の軸さえ決めておけば、AIはそれを読み取ってくれます。

一方で、表示ロジックのような影響が局所的な部分は、細部までAIに任せています。影響がその画面・その部品に留まるなら、生成に任せて問題ないと考えています。

### 今以上に、設計の質が重要になる

コード自体はすぐに作れる時代になりました。だからこそ、AIにコードを書かせる前に「**その設計はどうなのか？もっと良い設計はないか？**」と、設計そのものを問い直すことに時間を使う価値が上がっていると思います。

目先のコード生成・生産性に惑わされたり焦ったりしてしまいますが、時にはそこに流されないことも大切だと思います。（もちろん、優先順位は随時変わりますが）

「作る」こと自体は AI でもできるからこそ、差がつくのは「何を・どう設計するか」の判断力だと思います。

AI時代こそ、設計力が最大の武器になると感じています。

## おわりに

* **既存実装の設計が重要**：AIは今のコードを水平展開する。良い設計を残し、悪いパターンを入れさせない
* **仕組みで自走させる**：Plan、動作確認、レビューの蓄積、Skill化で、AIに任せられる範囲を広げる
* **生産性の方向を見る**：生成速度だけに流されず、設計を問い直す時間を惜しまないようにする

この記事が、AIとの開発の進め方を見直すきっかけになれば嬉しいです。
