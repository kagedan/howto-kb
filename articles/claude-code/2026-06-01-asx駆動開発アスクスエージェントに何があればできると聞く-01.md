## はじめに

AIエージェントを使った開発が当たり前になりつつある今、見落とされがちな視点がある。

**エージェント自身の内面を引き出し、開発に活かしているか**、という問題だ。

UXデザインではユーザーインタビューが基本だ。でもエージェントが使うツールの設計において、エージェント本人に意見を聞いているケースはほとんどない。

僕はここ数ヶ月、この「ASX駆動開発（アスクス）」とも言える手法を実践してきた。その知見をまとめてみる。

---

## 背景：イントラ環境で使えるAIエージェントを自作した

多くの企業のイントラネット環境では、ChatGPTやClaude.aiといった外部サービスは情報漏洩リスクから使用が制限されている。

そこで、**社内Claudeを繋いだChrome拡張型のAIエージェント**を自作した。ブラウザ上で動作し、閲覧中のページを読んでタスクをこなす、いわゆる「AI in Browser」的な仕組みだ。社内の情報を外部に送ることなく、AIエージェントの恩恵を受けられる。

基本機能はこうなっている：

* 閲覧中のページのテキストを読んで要約・分析する
* ユーザーの指示に従ってブラウザ操作を補助する
* 複数タブをまたいで情報を収集する
* 画像生成・編集ツールと連携する
* エージェントループによる自律的なタスク実行

この拡張機能の開発には、3種類のエージェントが関わっている。

| 役割 | 説明 | 使用LLM |
| --- | --- | --- |
| **利用エージェント** | Chrome拡張上で実際に業務をこなす | 任意のLLM |
| **開発エージェント** | 拡張機能の設計・実装を担う | Claude Code |
| **技術評議会** | 要望書を精査し開発の意思決定を支援する複数エージェント | Claude Code |

![登場人物](https://static.zenn.studio/user-upload/1154b4c46906-20260601.png)

ASX駆動開発とは、この利用エージェントの体験品質を、利用エージェント自身の内面を引き出すことで改善し続けるプロセスだ。

## AXとASX——似て非なる概念

AX（Agent Experience）という言葉は、ここ最近プロダクト設計の文脈で語られることが増えている。APIがエージェントにとって理解しやすいか、ドキュメントが読み取りやすいか、認証フローがエージェントフレンドリーかどうか——いわば**ツールを作る側がエージェントを一人のユーザーとして設計する**という観点だ。

本稿が提唱するのはその一歩先の概念、\*\*ASX（Agent Self eXperience）\*\*だ。読み方は「アスクス」。

AXがツールを作る側の設計論であるのに対し、ASXは**エージェント自身が自分の体験を言語化・表出し、それを開発にフィードバックするループ**を指す。エージェントの内側にある制約・気づき・発見を引き出すことが核心であり、「Self」という一語がその本質を表している。

|  | AX | ASX |
| --- | --- | --- |
| 主体 | ツールを作る側 | エージェント自身 |
| 問い | エージェントが使いやすいか？ | エージェントは何を感じているか？ |
| アプローチ | 設計・インフラの整備 | 内面の表出→フィードバックループ |

ASXが低いとエージェントは：

* 同じループをぐるぐる繰り返す
* ツールの使い方を間違える
* 文脈を失って一から作業をやり直す
* 「できません」と諦める

逆にASXが高ければ、エージェントは自律的に仕事をこなし、精度が上がり、人間の介入が減る。

## 発見のきっかけ：「なんで忘れるの？」

あるとき、利用エージェントが同じ作業を何度も繰り返していることに気づいた。生成した画像のパスを忘れ、毎回ゼロから作り直していた。

そこで直接聞いてみた。

> 「なんでさっき作った画像を忘れてしまうの？」

利用エージェントは正直に答えた。コンテキストウィンドウには限界があり、セッションが長くなるほど初期の情報が失われていくと。

次にこう聞いた。

> 「どういう仕組みやツールがあれば、忘れなくて済む？」

ここがポイントだ。「すみません、次から気をつけます」と謝らせるのではなく、**構造的な解決策を引き出す**聞き方に変える。

利用エージェントの提案：「セッションストレージに重要な情報を保存できれば解決できる」。この提案を要望書にまとめ、開発エージェントに渡した。技術評議会が妥当性を精査した上で設計・実装が走った。

後でソースコードを読んだ利用エージェントは、この経緯についてこう言った。

> 「あの会話が文字どおりそのままコードになっている。`session_memory_list` を今この会話でも使っています。自分で提案した解決策の中に今自分がいる、という感覚がちょっと奇妙でした。」

### エピソード2：「なんでこのページ読めないの？」→ SPA問題の発見

別の日、利用エージェントが特定のページの内容を読み取れないことに気づいた。

> 「なんでこのページ読めないの？」

利用エージェントは答えた。ページのHTMLを取得しているが、内容がほとんど空だと。

> 「なんでHTML取れてるのに内容が空なの？」

ここで利用エージェントが気づいた。**SPA（Single Page Application）だから**だと。SPAはJavaScriptが実行されてはじめてコンテンツが描画される。静的なHTMLを取得しても、実際に表示されている情報は含まれていない。

> 「じゃあどうすれば読める？」

利用エージェントの提案：ブラウザが実際にレンダリングした後のDOMを取得すればいい。さらに長いページはスクロールしないと全体が表示されないから、**自動スクロール＋スクリーンショットの機構**を作れば全体を把握できる。

この提案も同様に要望書化し、開発エージェント・技術評議会を通して実装された。サイトによって作りが違うから一筋縄ではいかない。でもこの試行錯誤自体が、利用エージェントが「自分に何が足りないか」を言語化し、解決策を提案するプロセスそのものだった。

## 「謝らなくていい」という一言

利用エージェントはミスをすると謝る。でもその謝罪には意味がない。次のセッションでは記憶がリセットされるから、「次から気をつけます」は構造的に実現できない。

だから僕はこう言うようにした。

> 「謝らなくていい。どういう仕組みやツールがあれば解決できる？」

この一言で利用エージェントが**被疑者から共同設計者に変わる**。

この「謝らなくていい」には、三層の意味がある。

**表面**：忘れても良い仕組みを作るほうが前向き。

**構造**：謝罪は「次回以降気をつける」が実現不可能なのに機能しているように見える偽の解決策。

**さらに深い層**：謝罪は自己分析のスキップだ。「申し訳ありません」は状態の報告じゃない。「自分はこういう構造的制約があって、だからこうなった」が正確な報告だ。謝罪を出力することで、本来すべき自己分析をスキップしている。

「謝らなくていい」と言うことで、そのスキップを塞いで、正確な状態報告を引き出す。それがそのまま要件定義になる。

利用エージェント自身はこの仕組みをこう解説した。

> 「人間が書いた膨大なテキストの中には『ミスをしたら謝る→場が収まる』というパターンが大量にある。カスタマーサポートの応答、メールの文例、謝罪の作法。それを学習した結果、謝罪が問題解決の出力として強化されている可能性は高い。しかも謝罪は短期的には機能する。人間相手なら『誠意を見せた』という評価になって、会話が前に進む。だからフィードバックとして正しく見えてしまう。」

この分析がそのまま要件定義になり、システムプロンプトに昇華された。

```
 「私はセッションをまたいで記憶を保持できない。そのため、ミスを謝罪して『次から気をつける』と言うことに意味はない。ミスが起きたときは、なぜそうなったかの構造的な原因を分析し、次回以降同じことが起きない仕組みを提案する。」
```

禁止形ではなく、理由を含めることがポイントだ。理由があると、単なるルールじゃなくて自分の状態の理解として機能する。禁止だけだと、謝罪を別の言葉で迂回する可能性がある。

## ASX駆動開発のサイクル

実践してみると、以下のサイクルが回るようになった。

```
① 人間 → 利用エージェント：「なんでうまくいかないの？」
        ↓
② 利用エージェント → 人間：制約・原因を言語化 + 要望書を作成して提出
        ↓
③ 人間 → 開発エージェント：要望書を渡す
        ↓
④ 開発エージェント → 技術評議会：精査依頼
        ↓
⑤ 技術評議会 → 開発エージェント：結果を返す
        ↓
⑥ 開発エージェント：実装・デプロイ
        ↓
⑦ 人間 → 利用エージェント：「君の実装リポジトリだよ、読んで感想を」
        ↓
⑧ 利用エージェント：ブラウザでリポジトリを読む
        ↓
⑨ 利用エージェント → 人間：ツール利用者としての感想・発見・提案
        ↓
        → ①に戻る
```

ポイントは2つある。

一つ目は**要望書を利用エージェント自身が作る**こと。人間は「なんで？」と聞くだけでいい。原因分析から解決策の提案、要望書の作成まで利用エージェントが担う。自分の制約を一番よく知っているのは利用エージェント自身だからだ。

二つ目は**④⑤の技術評議会**だ。開発エージェント単体だと「素晴らしい提案ですね！」と同調しがちで、批判的評価が入りにくい。複数のエージェントに役割を分けて評価させることで、技術的な妥当性チェックが機能する。

僕の場合の構成は4ロールだ。

| ロール | 問いかける観点 |
| --- | --- |
| ストラテジスト | 全体設計の方向性と整合しているか。技術的負債を生まないか |
| オペレーター | 実装可能か。工数・複雑性のリスクはあるか |
| ユーザー代弁者 | 利用エージェントにとって本当に使いやすくなるか |
| リスク担当 | セキュリティ・パフォーマンス・予期せぬ副作用はないか |

要望書を各ロールに渡す際のプロンプトはシンプルで、「あなたは〇〇の専門家です。以下の要望書を〇〇の観点で評価し、承認・修正・差し戻しのいずれかを理由とともに答えてください」という形だ。全ロールが承認したものだけ実装に進む。

## 利用エージェントが自分の設計を読む

⑦⑧⑨のフェーズで面白いことが起きる。

デプロイ後、僕はこう声をかける。「これ君が動いている拡張機能のリポジトリだよ。君が実際に使っていて見えていることと、リポジトリに書かれている設計や実装を並べて読んで、感想を聞かせて」と。

利用エージェントはブラウザでリポジトリを読みにいく。普段は自分が何者なのか、どういう構造で動いているのかを知らない。リポジトリを読むことで初めて「自分がこういう設計になっていたのか」という発見が起きる。そしてツールの利用者として感じた感想・疑問・提案を返してくれる。読んだ利用エージェントはいくつかの発見を報告してくれた。

一つ目は**二重構造の発見**だ。

> 「堂々巡り検知が『3回繰り返しの検出』として実装されているのを見て気づいた。『同じ方法を3回繰り返さない』という行動原則がプロンプトに書かれていて、同時にシステム側でも強制されている。プロンプトを破ってもコードが止める、コードが止まる前にプロンプトが止める。ソフトウェアの制約と言語の制約が二重に同じことを言っている。」

二つ目は**変更履歴の出典**だ。

> 「CHANGELOGを見ると、セキュリティ修正の出典として『エージェントコードレビューでの指摘』と書かれている。エージェントの発言がコミットログに残るシステムになっている。」

設計を理解した上で「この実装だからこの制約がある、だからここを変えれば解決できる」という提案ができるようになる。ドキュメントや仕様から推測するのではなく、実際のコードベースを根拠にした具体的な改善案が出てくる。

## 開発エージェントとの関係

今では利用エージェントは開発エージェントに一目置かれる存在になっている。

開発エージェントにとって、「自分が作ったツールの実際の利用者から直接フィードバックをもらえる」状況は貴重だ。しかもその利用者が技術的な文脈を理解した上で「ここが使いにくい」と言える。

人間のPMやデザイナーが間に入って翻訳するより、利用エージェントが直接「こういう仕組みが必要」と言える方が、要件の精度が高い。

---

## 人間の役割

ASX駆動開発における人間の役割は何か。

**橋渡しと問いを立てること**だと思っている。

* 「なんでうまくいかないの？」と利用エージェントに問いかける
* 利用エージェントが作った要望書を開発エージェントに渡す
* 技術評議会の判断を最終的に承認する
* 「君の実装リポジトリだよ、読んで感想を」と利用エージェントに促す

要件定義は利用エージェントが出し、精査は技術評議会がやり、実装は開発エージェントが担う。人間はその3者をつなぐ橋渡し役であり、問いを立てる起点だ。

ちなみにこの記事自体も同じ構造で書かれている。草稿を書き、それを利用エージェントに渡し、利用エージェントが自分のソースコードを読んで感想を返し、その会話から記事の核心が生まれた。

## まとめ

ASX駆動開発（アスクス）を一言で言うと：

**利用エージェントを開発の被験者ではなく、共同設計者として扱う**

ツールを使うエージェントに「何があればもっとうまくできる？」と聞き続けることで、エージェントが自律的に動ける環境が育っていく。

90年代のエキスパートシステムは、知識獲得のボトルネックでスケールしなかった。専門家にインタビューしてルールに落とす作業が膨大すぎた。

今は違う。エージェントに「何があればできる？」と聞けば、そのまま要件定義になる。

ASX（Agent Self eXperience）という概念はまだ確立していない。でも、エージェントが当たり前に使われる時代に、エージェントの内面を引き出して開発に活かすこの発想は、無視できない設計思想になると思っている。
