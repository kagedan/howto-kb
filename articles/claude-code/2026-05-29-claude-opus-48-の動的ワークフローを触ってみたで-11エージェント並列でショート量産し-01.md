## なんでこの記事書いたんか

Claude Opus 4.8 が出たんやけど、今回いちばんワイがテンション上がったんは、ベンチマークの数字でもfast modeの値下げでもなくて、**動的ワークフロー（Dynamic Workflows）** っていう新機能やった。

ざっくり言うと、

> 人間が「これやって」って指示すると、Claude がそれを計画して、**1セッションで数十〜数百の並列サブエージェントを起動**し、生成物を検証してフィードバックループまで回してから報告してくれる

という機能や。今のところまだリサーチプレビュー段階なんやけど、これ実際に触ってみたら、**横長(16:9)の動画1本から、縦型(9:16)のショート動画を11本いっぺんに作る** っていう作業を、11エージェントが並列でガーッと回してくれてな。「あ、オーケストレーションの時代ほんまに来たわ」ってなったんよ。

この記事では、

* 動的ワークフローって結局なんやねん（仕組み）
* どのプランで使えて、何に気をつけるんか
* 起動方法は2つある、その使い分け
* どんなタスクに向いてるんか
* ワイが実際に11エージェント並列を回してみた体感

を順番に話していくで。Opus 4.8 全体のアップデートにもちょっとは触れるけど、主役はあくまで動的ワークフローや。

## 3行サマリ ― ここだけ持って帰ってくれてもええ

* **動的ワークフローは「計画→数百サブエージェント並列→検証→フィードバックループ→報告」を1セッションで回す機能**。リサーチプレビューで Enterprise / Team / MAX 限定
* **起動方法は2つ** ― ①`workflow` キーワードで明示実行、②ultracode ON＋effort levelで自動判定。確実に回したいなら①、お任せなら②
* **大量にサブエージェントを起動するからトークンをガッツリ食う**。まずはスコープ絞ったタスクで使用量を掴むのがおすすめ。auto mode有効が快適

## まずは Opus 4.8 のアップデートをサクッと

動的ワークフローの前に、Opus 4.8 そのものの更新点を軽くおさらいしとくわ。今回の主役やないから、要点だけや。

| 項目 | 内容 |
| --- | --- |
| ベンチマーク | Opus 4.7 ベースに改良。SWE-bench Pro みたいなエージェンティックコーディング系の性能が上がっとる |
| effort level制御 | タスクに投入する労力（＝消費トークン量）を制御できるように。Claude の Web 版 UI や Cowork からも設定できる |
| fast mode | 標準の2.5倍速のモード。以前のモデルと比べて **約1/3の価格** で使える |
| 誠実さの向上 | 裏付けのない主張をしにくくなった。**自分が書いたコードの欠陥を見過ごす可能性が約1/4まで減少** |

正直、純粋なモデル能力の比較表をいちいち細かく見る意味は、だんだん薄れてきてる気もするんよな。それより実務的に効くんは下の2つやと思う。

### 誠実さの向上が地味にデカい

ワイが「お、ええやん」と思ったんは **誠実さ（honesty）** の改善や。Opus 4.8 は前バージョンに比べて、**自分が記述したコードの欠陥を見過ごす可能性が約4分の1まで減ってる**らしい。

これ、AIにコード書かせてレビューもAIに任せる運用やと、めちゃくちゃ効いてくる。「自分が書いたものを甘く採点する」傾向が減るってことやからな。あと、性能の良さを逆手に取って脆弱性を突くような「欺瞞・悪用への協力」の発生率も、4.7より大幅に下がっとって、噂の "Mythos（ミュトス）" プレビューと同程度まで落ちてる、という話やった（※Mythos は噂レベルの話なんで、そこは話半分で）。

### effort levelでトークンを絞れる

もうひとつが **effort level** や。タスクの複雑さに応じて「どんだけ労力（トークン）かけるか」の割合を調整できるようになった。Claude Code が一番機能リッチなんやけど、こういう制御が Claude AI の Web 版や Cowork にもどんどん降りてきてるのは素直に嬉しいとこやね。

そんで、このeffort levelが **動的ワークフローの自動起動とも絡んでくる** んよ。ここからが本題や。

## 動的ワークフローって結局なんやねん

公式ドキュメントを読んで、ワイが「ポイントはこの2つやな」と思った部分はこれや。

1. **数十〜数百のサブエージェントを単一セッションで並列実行する、オーケストレーションスクリプトを動的に生成する**
2. **成果物がユーザーに届く前に、その動作をチェックする（フィードバックループを回す）**

つまり「サブエージェントをいっぱい並列で走らせる」だけやなくて、**ちゃんと検証してから結果を返してくれる**のがミソやねん。投げっぱなしやなくて、自己採点までセットになってるイメージやな。

```
人間の指示
   │
   ▼
[計画] ── オーケストレーションスクリプトを動的生成
   │
   ▼
[並列実行] ── 数十〜数百のサブエージェントが同時に動く
   │
   ▼
[検証] ── 成果物が指示を満たしてるかチェック（フィードバックループ）
   │
   ▼
ユーザーに報告
```

従来の「サブエージェントを手で定義して呼ぶ」みたいなやり方やと、何個動かすか・どう分担させるかを人間が設計せなあかんかった。動的ワークフローは、そのオーケストレーション設計自体を **Claude がタスクに合わせて勝手に組み立ててくれる** ってとこが新しい。

## どのプランで使えるんか・前提条件

ここは大事なとこやから先に言うとくで。

* **リサーチプレビュー段階**。まだ正式版やない
* **Enterprise / Team / MAX プランで利用可能**

なんで上位プラン限定なんかというと、まあ **サブエージェントを大量に起動するから** やろな。Pro プランや無料プランでこれを開放したら、トークンがいくらあっても足りひん。せやから MAX 以上に絞ってるんやと思う。

### 注意点：トークンをガッツリ食う

公式ドキュメントにもハッキリ書いてあるんやけど、動的ワークフローは **通常の Claude Code セッションよりも大幅に多くのトークンを消費する可能性がある**。サブエージェントを並列でいっぱい起動するんやから、当然っちゃ当然やな。

> まずはスコープを限定したタスクから始めて、実際の作業における使用状況を把握することをお勧めします

ってトーンで書かれてる。ワイもこれは賛成で、いきなりデカいタスクをぶん投げるんやなくて、**小さめのタスクで「これくらい食うんか」って肌感を掴んでから**本番に使うのがええと思う。

### auto modeを有効にしとこう

もうひとつ、**auto modeを有効にしてくれ** とも書いてある。これたぶん、せっかくサブエージェントをオーケストレーションさせてるのに、いちいち承認が必要なモードで走らせると、**そのたびに止まって体験がブツ切りになる** からやろな。並列でガンガン進めてもらうのが旨味やのに、確認ダイアログで待たされたら台無しやもんな。

## 起動方法は2つある

動的ワークフローを使うには、次の2通りのやり方がある。これは使い分けが大事やから表にまとめとくわ。

| 方法 | やり方 | 向いてる場面 |
| --- | --- | --- |
| ①明示実行 | プロンプトに `workflow` というキーワードを入れる | 「これは確実に並列で回したい」と分かってるとき |
| ②自動判定 | ultracodeをONにして、effort levelを高め（x-high帯）に設定する | お任せで、難しいタスクのときだけ自動で発動させたいとき |

### ① `workflow` キーワードで明示実行

`workflow` って打つと、過去の `ultrathink` みたいに **予約語っぽく反応してくれる**。「あ、こいつ動的ワークフロー使ってほしいんやな」っていうのが Claude Code に伝わる感じやな。「ここは絶対に並列でぶん回したい」っていう確信があるときは、これが一番手っ取り早い。

### ② ultracode ON＋effort levelで自動判定

もうひとつは、effort levelの設定から **ultracodeをON** にして、effort levelを高め（デモでは x-high を使った）に設定するやり方。こうすると、**Claude がタスクを処理するのにワークフローを使うべきタイミングを動的に・自動的に判断してくれる** ようになる。

「いちいち `workflow` って打つの面倒やし、難しいタスクのときだけ勝手に発動してほしい」っていう人はこっちやな。

ワイの使い分けの結論はこうや。

* **確実に並列で回したい** → ①明示キーワード
* **お任せで、重いタスクだけ自動発動させたい** → ②ultracode ON＋effort level

## どんなタスクに向いてるんか

動的ワークフローは、**大規模かつ複雑めな作業** に向いてる。労力に対しての費用対効果を考えると、小さいタスクで使うと逆にトークンの無駄になりがちや。公式が例に挙げてるのはこんなあたり。

* **大規模な移行・近代化作業**（レガシーコードの作り直しとか）
* **フレームワークの切り替え**
* **コードベース全体のバグハンティング**
* **複数リポジトリを並列で見ないといけない作業**

要は「**1つのコードベース全体に対して、分割して並列でいっぺんにやれる作業**」が一番ハマる。逆に、ファイル1個ちょこっと直すみたいな小タスクには完全にオーバースペックやから、そこは普通に Claude Code で十分や。

## 実際に11エージェント並列を回してみたで

ほな、ここからが本題。御託はええから実際どうなんや、っていうのをワイが試した話をするわ。

### お題：横16:9の動画1本 → 縦9:16のショート動画を量産

ワイがやらせたのは、**1本の横長(16:9)動画素材から、縦型(9:16)のショート動画を複数本切り出す** っていう作業や。動画を作るために用意してたプロジェクト（素材のMP4と、切り出しルールをまとめたもの）を指定して、まるっとお願いしてみた。

事前に、こういうディテールをスキルで指示してあった。

* 上下に **帯テキスト**（タイトルとかキャプション）を入れる
* 「続きはこちら」みたいな **CTAの出し分け** をする
* 元動画の重要な部分が見切れんように、**クロップせず**にアスペクト比を保つ

この「細かい注文」をどこまで再現してくれるかが見どころやった。

### セットアップ

1. Claude Code のセッションを開く
2. auto modeを有効にする（前述の通り、これ推奨やからな）
3. effort levelは x-high のまま
4. 今回は1回 **手動で** 実行したいから、プロンプトに `workflow` キーワードを入れる

ちなみに小ネタなんやけど、ワイの環境はプラン設定の都合で、auto modeで走らせると裏で Sonnet が動く設定になっとった。このへんは自分のプラン・モデル設定によって挙動が変わるから、動かす前に確認しといた方がええで。

### 実行 → いきなり11エージェント並列

`workflow` キーワード込みで作業を投げると、まず **計画フェーズ** に入って、タスクをどう分割するかの作業計画を立ててくれる。

![動的ワークフローが最初に立てた作業計画。タスクをどう分割して並列化するかのプランを提示してくる](https://static.zenn.studio/user-upload/deployed-images/850b987a290236f9479d5c6b.png?sha=a7e53e3c787f3f6b1bd6c9a7a5c9475d391c7589)

その計画にもとづいて、サブエージェントを並列実行するための **オーケストレーションスクリプトを動的に組み立て始める**。ここが「人間がサブエージェント構成を手で設計せんでええ」っていう動的ワークフローの肝の部分やな。

![並列実行用のオーケストレーションスクリプトを生成しているところ](https://static.zenn.studio/user-upload/deployed-images/8bffe9d8253ab300e3e0a042.png?sha=54e33ce52c49c2c4e3c8a4696cd7de80497f3967)

そんで、こんな警告が出る。

> Dynamic workflow can use a lot of tokens quickly by running many subagents in parallel.

「サブエージェントを並列でようけ起動するから、トークンを一気に食うで」っていう例の注意書きやな。今回は問題ないんで、そのまま Yes で進める。

許可して進んだら……**いきなりバックグラウンドで11エージェントが並列で動き出した**。これはちょっと壮観やったわ。1エージェントが逐次でレンダリングしていくんやなくて、11個が同時にガーッと走るんよ。

### `/workflows` で進捗が丸わかり

走らせてる最中、`/workflows` を実行すると **進捗が確認できる**。「copy」「render」みたいに、どのエージェントが今どの工程をやってるかが見えてな。これがめちゃくちゃええんよ。

![/workflows で並列サブエージェントの進捗を確認している画面。各エージェントが今どの工程（copy / render など）かが一覧で見える](https://static.zenn.studio/user-upload/deployed-images/9bc3ef614ec98b462d61a314.png?sha=d83f80944ac59ace1aed223234ba9712df61f13e)

マルチエージェントをターミナルで表示しやすくする、みたいなアップデートが少し前にあったやんか。あれ、たぶんこの **動的ワークフローのための前準備** やったんちゃうかな、って腹落ちした。並列で大量に動くからこそ、進捗の可視化がないと「今どうなってんの？」って不安になるからな。委任の安心感がぜんぜん違う。

### 結果：11本完成、細かい注文も全部再現

で、結果。**11本のショート動画が全部できあがった**。しかも事前に指示してたディテールが、ちゃんと全部反映されてた。

* 上下の **帯テキスト** → 入ってる
* 「続きはこちら」の **CTA出し分け** → 切り替わってる
* **クロップなし**でアスペクト比キープ → 守られてる

そんで地味に効いてるのが、**最終的に動画ができあがった後に「この指示がちゃんと達成できてるか」を Claude 自身が確認してくれてた** ことや。これがさっき言うた「フィードバックループ」の正体やな。並列で量産するだけやなくて、自己検証までやってくれるから、投げっぱなしの不安が減る。

正直、今回のお題はそこまで難易度の高いタスクやなかった。せやから本領を発揮させるなら、**もっと長時間・高難度のタスク**（さっき挙げた大規模移行とかバグハンティングとか）に使うのがええと思う。それでも「並列で動いてる様子が可視化される体験」は一度味わう価値あるで。

## 触ってみて思ったこと ― オーケストレーションの時代が来てる

今回いちばん感じたんは、**AIエージェントのオーケストレーションが、もう「実用フェーズ」に入ってきてる** ってことや。

こういうマルチエージェントのオーケストレーションは、GitHub・Anthropic・OpenAI みたいな主要なAI先進企業が「今後ここがポイントになる」って各所のドキュメントで言うてる。つまり、これは Anthropic だけの一発ネタやなくて、**この先いろんなAI系ツールが同じ方向に進んでいく** 流れの先頭やと思う。

ワイの個人的なスタンスとしては、

* **「1エージェントに長い仕事を順番にやらせる」から「複数エージェントに分割して並列でやらせる」へ** 軸足が移ってきてる
* そのとき人間がやるべきは、手を動かすことやなくて **タスクをどう分割して、どう検証させるかの設計**（＝オーケストレーションの上流監督）
* ただしトークンコストは正直バカにならんから、**「向いてるタスクかどうか」の見極め** が運用のキモになる

ってとこやな。なんでもかんでも動的ワークフローに投げるんやなくて、「これは並列で分割できる重いタスクや」って判断できるかどうかが、使いこなしの分かれ目になりそうや。

## ほな、まとめていこか

最後にギュッとまとめるで。

* **動的ワークフロー**は、計画→数百サブエージェント並列→検証→フィードバックループ→報告を **1セッションで回す** Claude Opus 4.8 の目玉機能。リサーチプレビューで Enterprise / Team / MAX 限定
* **起動方法は2つ**。確実に回したいなら `workflow` キーワードで明示、お任せならultracode ON＋effort levelで自動判定
* **トークンを大量に食う**から、まずはスコープ絞って使用量を掴む。auto mode有効が快適
* **向いてるのは大規模・複雑なタスク**（大規模移行、フレームワーク切替、コードベース全体のバグハンティング、複数リポジトリ並列）
* 実際に横16:9→縦9:16のショート量産を回したら、**11エージェントが並列で動いて、細かい注文も全部再現＋自己検証までやってくれた**

「サブエージェントとかオーケストレーションとか、概念は分かるけど自分に使えるんかな」って不安な人は、まず **完結した小さめのタスクを1個** 任せてみるのがええで。並列で動いてるのを一回見ると、一気に腹落ちするから。スコープ絞って、小さく試すところから始めてみてな。
