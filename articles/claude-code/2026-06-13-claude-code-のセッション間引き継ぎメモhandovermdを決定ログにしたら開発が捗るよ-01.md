## セッションからセッションへ、どうつなぐ？

Claude Code を使っていて、みんなが一度はぶつかる悩みが**セッションをまたぐと文脈が消える**問題ではないでしょうか。昨日「なぜこの設計にしたか」を散々議論しても、翌日の Claude はそれを知りません。同じ検討をもう一度させたり、せっかく潰した選択肢をまた提案されたり。「続きから」がうまくつながりません。

自分なりの答えとして作ったのが、引き継ぎメモ（`docs/handover.md`）の作成・更新を自動化する Claude Code skill です。

ちなみに最初は素朴に**ぜんぶを handover.md に詰め込んで**すぐ破綻しました。進捗もTODOも経緯も雑談も一緒くたで、あっという間に数百行。肥大化したメモは結局**読まれない文書**になったり、続きからはじめるのにはノイズになったりします。その反省を踏まえて落ち着いたのが今の形なので、それを共有します。

## handover.md は2層構造にする

肥大化の原因は「再開に必要な情報」と「あとで参照する情報」を混ぜていたことです。なので役割で2層に分割しました。

```
# Handover - [プロジェクト名]

## 現在のフェーズ
実装中

## 次にやること
（新セッションで即作業再開できる粒度で。次のタスク名・関連ファイルの絶対パス・直前の停止地点）

---
<!-- ↑ ここまでがセッション冒頭で必ず読むブロック。↓ は参照用 -->

## 決定事項ログ
## 既知の課題・保留
```

**冒頭ブロック（現在のフェーズ＋次にやること）をファイル先頭に固定**するのがポイントです。新セッションの Claude はまずここだけ読めば、即座に作業を再開できます。下の参照ブロックは必要なときに引きにいきます。

## 山場：効果的なのは「決定事項ログ」

毎日の開発で一番役に立っているのが、この**決定事項ログ**です。設計判断やスコープの境界を、**「決定」と並べて「不採用の案」と「なぜそう決めたか（理由）」まで残します**。

```
### 42 2026-06-08: [見出し]
- 種別: 設計判断
- 決定: 〜という方式を採用
- 理由: 〜だから（採らなかった代替案と、その却下理由も）
- 詳細: コミット `abc1234`（必要時のみ）
```

理由まで残すことの良いところ。

* **過去の判断を蒸し返さなくなる。** 「この方式、別案のほうがよくない？」とClaude（や未来の自分）が言い出しても、却下理由がログにあります。再検討のループに入りません。
* **同じレビュー指摘を再 litigate しない。** 「これは検討済みで、こういうトレードオフで見送った」と書いておけば、次のレビューで同じ所見が蒸し返されても即決着できます。
* **決定が覆っても歴史が残る。** 方針転換しても旧エントリは消さず、新エントリを追記します。「なぜ前の方式をやめたか」自体が次の判断材料になります。

決定が積み重なるほど、プロジェクトの「なぜ」が資産として溜まっていきます。これが handover.md の本体だと考えています。

## 肥大化を防ぐ3つの仕掛け

ただし放っておけばまた巨大化してしまいます。そこでskill 側にブレーキを3つ入れました。

1. **判定軸「`git log --oneline` で5秒で取れる情報は書かない」。** 何をやったかはコミットを見れば分かります。handover に書くのは「なぜ」と「次」だけです。
2. **1決定＝4〜5行に制限。** 書き切れない詳細はコミットメッセージへ逃がし、`詳細: コミット abc1234` で参照します。メモは索引、本文は git に置きます。
3. **古い決定は自動アーカイブ。** ログが3件を超えたら、古い順に `handover_archive.md` へ退避して本体は3件に保ちます。本体は常に「今ホットな決定」だけです。

## 脇役：trap.md

skill では handover 以外に、`trap.md` と `retrospective.md` も扱います。まず `trap.md` から。

ここには**環境固有の罠だけ**を書きます。「git log にも README にも出ないが、踏むと時間を溶かす事実」専用です。読み導線もはっきりしていて、該当領域のコードを触る前に grep する運用にしています（プロジェクトの CLAUDE.md に1行入れておく）。これは素直に役立っています。

問題は3本目の `retrospective.md` の方でした。

## もう一つの山場：retrospective.md を死蔵させない

`retrospective.md` は、進め方・設計判断の振り返りを書く場所です。「この段取りが効いた」「この判断は裏目に出た」を残します。

正直に言うと、**この記事を書き始めた時点では、これは死蔵していました**。書き溜める一方で活かす導線がなく、数百行に育ったまま。読み物としては最高に面白いClaudeCode日記帳。自分のプロジェクトが「どこでうまくいって、どこで判断を誤ったか」の生ログなので。けれど、面白いだけでは資産になりません。

この記事を書いていて、これが retrospective 最大の問題だと気づきました。**振り返りは、書くことより「還流させること」が大事**です。そこで skill に**還流ハーネス**を足しました。要点は3つ。

1. **定期蒸留で恒久ルールへ昇格させる。** フェーズが完了したら「蒸留しますか」と skill 側から提案します。溜まった振り返りを読み返し、繰り返している教訓を恒久ルール`~/.claude/rules/` やグローバル CLAUDE.md、プロジェクト固有なら auto memory、へ昇格させ、retrospective 本体は薄く再開します。死蔵させず、ルールに変換して回収するわけです。
2. **昇格には razor をかける。** 何でも昇格させればルールが増え、増えるほど一つひとつが守られなくなります（ルール枠は注意資源の予算）。だから昇格していいのは「**黙って壊れる・遅れて現れる失敗**」だけ。即座にエラーで気づけて、やり直しも安いミスの予防則は足しません。
3. **悪かったことは漏らさず、良かったことは選ぶ（非対称記録）。** 失点は選別せず1〜2行で必ず残します——「同じ失敗を何回くり返したか」を数える原資になるからです（追記前に grep して「再発3回目」と明記）。逆に「良かった」は再利用できる作法に昇華できるものだけ。ただの成果報告は書きません。

再発が3回に達したテーマは、蒸留を待たずその場で昇格を検討します。これで retrospective は「面白いだけの生ログ」から**ルールの供給源**に変わりました。冒頭で触れた肥大化も、整理ではなく\*\*蒸留（昇格させて archive へ切り離す）\*\*で解消します。

## auto memory との棲み分け

ここまで読んで、「それ、Claude Code の auto memory（永続メモリ）でよくない？」と思った人もいるはずです。実際、上の昇格先にも「プロジェクト固有の作業規範 → auto memory」が入っています。両者は競合しません。**情報の生存期間で棲み分けています。**

* **プロジェクトを通してずっと生き続ける教訓** → auto memory に書かせてよい。蒸留を通って「これは恒久ルールだ」と確定したものです。
* **一過性・この数セッションでだけ必要な情報**（今ホットな決定、直前の停止地点、まだ昇格に値しない振り返り）→ handover / retrospective に置く。

なぜ全部 auto memory に入れないか。**auto memory も放っておけば肥大化するから**です。何でも「覚えておいて」と渡すと、毎セッション読み込まれるメモリがノイズで膨らみ、結局 handover で陥ったのと同じ「読まれない文書」化が起きます。だから auto memory は**蒸留を通った恒久教訓だけ**の場所に保ち、流動的なものは handover 側で回す——という役割分担にしています。

## 導入

skill の全文は GitHub Gist に置きました（執筆時点のスナップショット）。コピーして `~/.claude/skills/my_handover_management/SKILL.md` に置けば使えます。

「今日はここまで」「handover 更新して」などのトリガーで起動します。

## まとめ

この skill の運用は、突き詰めると**肥大化との戦い**でした。handover も retrospective も auto memory も、全部書けば読まれず、書かなすぎれば再開できません。Claudeって、ほんとに長文書くの大好きなんですよね。ちょっと目を離すとすぐLooooong文章を書く。形になったのは、**情報を役割と生存期間で分け、それぞれに別のブレーキをかける**ということです。

* handover の決定事項ログは、「なぜ」と「次」だけを 4〜5行・3件で積む。
* retrospective は書きっぱなしにせず、蒸留して恒久ルールへ還流させる。
* 恒久的なものは auto memory、流動的なものは handover、と棲み分ける。

どれも、次のセッションの自分（と Claude）が**最小の文脈で続きから動ける**ことを狙っています。もちろん完成形ではありません。この記事を書いていて retrospective の死蔵に気づいたように、運用しながら境界を引き直していく。その「現状の解」をそのまま共有させていただきました。
