Claude Codeを「ただ動かすツール」として使っている間は、生産性の上限が低いまま終わる。

Claude Codeには、適切に設定すれば作業効率が体感で3〜5倍変わる仕組みが4つある。

【Claude.md】
プロジェクトルートに置くだけで、AIがコードベースの文脈・コーディング規約・禁止事項を自動で読み込む。毎回「このプロジェクトは〜」と説明する手間がゼロになる。

【Skills（スキル）】
繰り返し使う処理をスキルとして定義しておくと、1行の指示で複雑なタスクが走る。テスト生成・ドキュメント更新・リファクタリングを「コマンド1発」で完結させている開発者が増えている。

【Subagents（サブエージェント）】
大きなタスクを並列処理できる。フロント修正・バックエンド修正・テスト更新を同時進行させると、直列で動かすより作業時間が最大60%短縮できる。

【MCP（Model Context Protocol）】
GitHub・Slack・Notionなど外部ツールとClaude Codeを直接つなぐ仕組み。「Slackのスレッドを読んで、対応するコードをコミットして」が1コマンドで通る。

この4つを組み合わせると何が起きるか。

設計→実装→テスト→ドキュメント更新という一連のフローが、人間の確認待ちを除いてほぼ自動で回る。1日8時間の開発業務のうち、定型作業に使っていた2〜3時間が戻ってくる計算になる。

Claude Codeはデフォルト設定のまま使うと「少し賢いコード補完」で終わる。設定に1〜2時間投資するだけで、動き方が根本から変わる。

4つの設定の中で「Claude.md」が最優先になる理由がある。他の3つはタスク単位で効くが、Claude.mdはプロジェクト全体の精度に影響するため、ここが甘いとSkillsやSubagentsの出力品質も連動して落ちる。設定する順番があるとすれば、Claude.md→MCP→Skills→Subagentsの順になる。

「設定に時間をかけるより、そのまま使った方が早い」という反論はある。ただ、Claude.mdとSkillsの初期設定は合計で90分もかからない。一方、毎回コンテキストを口頭で補足する手間は1セッションあたり5〜10分ほど積み上がる。2週間もあれば設定コストは回収が終わる。

今日から動かすなら、まずプロジェクトルートに`CLAUDE.md`を作成し、「使用言語」「命名規則」「テストの書き方」「触ってはいけないファイル」の4項目だけ書いて置く。それだけで次のセッションからAIへの説明が不要になる。残り3つの設定はその後でいい。
