> このシリーズは「Claude Code を**使う**」記事ではなく、「Claude Code を中心に**開発環境ごと自作する**」記事です。

## TL;DR

* 既存の AI 統合 IDE のブラウザ由来のバグが引き金。だが本当の動機は「アプリの行き来が面倒」「ちょっとした機能（Markdown 目次など）のためにツールを乗り換えたくない」「マルチエージェントだと各エージェントの動きが見えない」という日々の不満。1 枚に統合し、かつ自分で機能を足せる土台がほしくて**自作**に振り切った
* 採用した割り切りは 4 つ:
  + **外部依存はターミナルライブラリ 1 個のみ**、残りは Apple 標準で代用
  + **App Sandbox を無効化**（`~/.claude/` の読み取りと PTY 起動のため）
  + **コードエディタ機能は作らない**（もう自分の手でコードを書かない＝実装は AI に任せる前提になったため）
  + **すべての設計判断を ADR で固定**して、後から自分で覆さないようにする
* 中核は「**人格別エージェント（Companion）× セッション**」のモデルと、Claude とアプリの**双方向ファイル通信**
* 本シリーズで一番読ませたいのは「ドキュメント設計」と「自律開発フロー」。序章はその全体像と導線です

## なぜ既存の IDE ではなく「自作」したのか

直接のきっかけは、ある Electron 製の AI コーディング IDE を実運用に乗せようとして詰まったことでした。`<webview>` 由来の制約で **OAuth ログインが壊れる・Geolocation が取れない・普段の Chrome プロファイルが使えない**——機能そのものは面白いのに、**ブラウザ由来のバグと永続的に付き合うコストが統合の便利さを上回る**と感じて、これは見送りました。

ただ、これは引き金にすぎません。本当の動機は、日々の AI 駆動開発でじわじわ溜まっていた小さな不満のほうでした。

### 1. アプリの行き来が面倒

ターミナル・ブラウザ・ファイラ・AI エージェント。1 つの作業で何度もアプリを切り替える。`Cmd+Tab` を叩き続ける生活は、地味だが確実に集中を削ってきます。「全部 1 枚のウィンドウで完結させたい」というのが最初の欲求でした。

### 2. 「ちょっとした機能」のためにツールを乗り換えたくない

たとえば **Markdown の目次表示**。昨今のエージェント開発では、仕様書・設計メモ・ADR など **Markdown を読む機会が圧倒的に増えました**。長い Markdown をスクロールで追うのは効率が悪く、目次がほしい。

でも、その「目次がほしいだけ」のためにエディタやビューアを丸ごと乗り換えるのは本末転倒です。**欲しい小さな機能を自分で足せる土台**があれば、こういう不満はその場で潰せる。これは既製ツールでは得にくい価値でした。

### 3. マルチエージェントだと「誰が何をしているか」が見えない

複数のエージェントを並行で走らせると、生産性は上がる一方で **それぞれが今どこで何をしているのかが途端に分からなくなります**。ターミナルのタブを行き来して目を凝らす、では追いつかない。

「人格別エージェントの状態を一望できる場」がほしい——これがアーキテクチャの中核（後述の Companion × Session モデル）に直結した動機です。

---

これらをまとめると、欲しかったのは「ターミナル・WebView・AI エージェント連携・ローカルファイル参照を **1 枚に統合し、かつ自分で機能を足せる**もの」でした。既製品には無い。だから作る、という結論です。

そのうえで私が立てた原則はシンプルです。

> **本来のブラウザの挙動と差異なく開発できることが最優先。**  
> 統合された UX は手段にすぎない。本物のブラウザで動くものが動かない・挙動が違う、ということに耐えてまで統合は求めない。

この原則から導いた答えが、macOS ネイティブ（Swift + SwiftUI）で組むことでした。`WKWebView` は本物の Safari エンジンなので、Geolocation も OAuth も **OS の権限システムに乗ってそのまま動く**。Electron で頑張って塞いでいた穴が、最初から無い。

もちろん「車輪の再発明では？」という声は自分の中にもありました。それに対する答えが、次の割り切りです。

---

## 設計を貫く 4 つの割り切り

### 1. 外部依存はターミナルライブラリ 1 個だけ

PTY（擬似端末）と端末 UI を担う [SwiftTerm](https://github.com/migueldeicaza/SwiftTerm)（MIT）**1 個のみ**に依存を絞り、残りはすべて Apple 標準フレームワークで賄います。

* UI → SwiftUI
* WebView → WebKit（`WKWebView`、`isInspectable = true`）
* ファイル監視 → CoreServices の `FSEventStream`
* 状態管理 → Observation の `@Observable`

個人プロジェクトで一番怖いのは「依存の腐敗」です。3 ヶ月触らない間にライブラリが破壊的更新を出して、ビルドが通らなくなる。依存を 1 個に絞れば、その事故が起きる面が最小化されます。

### 2. App Sandbox は無効

このアプリは Claude Code を中心に据えるので、`~/.claude/` 配下のリソース（スキルやエージェント定義）を読み、ターミナルから PTY を起動する必要があります。これらは App Sandbox とは相性が悪い。

そこで **Sandbox を無効化**し、代わりに「他人に配布しない・App Store に出さない・ユーザーは自分 1 人だけ」と割り切ることで成立させています。Developer Program（$99/年）も公証も不要で、Personal Team 署名で済みます。これは「個人用に振り切ったからこそ取れる」設計です。

### 3. コードエディタ機能は作らない

統合 IDE を名乗りながら、**コードエディタは持ちません**。

理由は単純で、**もう自分の手でコードを書かなくなったから**です。実装は AI エージェントに任せるのが前提になり、自分でエディタを開いてキーを叩く場面がほぼ消えました。そうなると、このアプリにエディタ機能を盛り込む意味がない。たまに手で直したいときは JetBrains や Neovim を開けば十分で、そこに自作で挑む理由はどこにもありません。

このアプリが受け持つのは「ターミナル・WebView・AI エージェント連携・ローカルファイル参照」を 1 枚に統合する部分だけ。**やらないことを先に決める**のが、個人開発を完走させるコツでした。

### 4. すべての判断を ADR で固定する

上の 3 つを含め、設計判断はすべて **ADR（Architecture Decision Record）** として 1 件 1 ファイルで記録しています。執筆時点で 30 件超。

```
docs/decisions/
├── 0001-swift-swiftui.md          # Electron ではなく Swift/SwiftUI を採用
├── 0002-no-code-editor.md         # コードエディタ機能を持たない
├── 0006-only-swiftterm-dependency.md  # 外部依存は SwiftTerm のみ
├── 0008-no-claude-autostart.md    # ターミナルでは claude を自動起動しない
├── 0013-session-as-first-class-object.md
└── ...（0031 まで）
```

個人開発で一番やりがちな失敗が「過去の自分の判断を忘れて、半年後に逆方向へ作り直す」こと。ADR があると、**「なぜそうしたか」を未来の自分に申し送れる**。AI エージェントに作業させるときも、ADR を読ませれば過去の決定を踏まえて動いてくれる——むしろ AI 駆動開発との相性のほうが本命でした。

## 全体アーキテクチャを 1 枚で

アプリの構造は「**含有関係の軸**」と「**通信の軸**」の 2 つで整理できます。

* **Window** はアプリ全体。**1 プロセス = 1 ウィンドウ = 1 リポジトリ**にして、複数リポジトリは別プロセスで同時に開きます。
* **Session** が状態を持つ実体。Claude を動かす各セッションには **Companion（人格別エージェント）** を割り当てていて、メイン担当・企画担当・秘書役…のように役割と口調を分けています。
* **Tool** は Session の機能種別（ターミナル / WebView / Claude / ファイラ / git diff / プレビュー等）。

### 通信軸: Backchannel と Frontchannel

構造軸とは独立した横断軸として、アプリと Claude の間に**双方向のファイルベース通信**を引いています。

* **Backchannel（Claude → アプリ）**: Claude が所定のディレクトリ（`.aidea/backchannels/<N>/`）にテキストを書き出すと、アプリが `FSEvents` で検知して反応する仕組み。たとえば音声合成エンジンで読み上げたり、別の Companion へタスクを渡したりします。
* **Frontchannel（アプリ → Claude）**: 逆方向は、アプリが PTY 経由で対話セッションに文字列を送り込む。

ポイントは、**この通信が「ファイル」と「PTY」という枯れた手段だけで成立している**こと。専用 IPC を作らず、`echo > file` や端末入力という最も単純なインターフェースに寄せたので、Claude 側は特別な SDK を一切必要としません。「ファイルが API」という思想は、第2章で詳しく扱います。

そのほか、ウィンドウ上部には時計などの **Widget** を並べていますが、これは付帯的な要素なので別章で。

## このシリーズで読ませたいもの

序章はあくまで地図です。本命は次の 2 章にあります。

| 章 | テーマ | なぜ読んでほしいか |
| --- | --- | --- |
| **第5章** | **ドキュメント設計** | `docs/` を `LAYOUT.md` を SSoT にして構造化し、frontmatter に `derived_from` / `syncs_with` / `impacts` を持たせて**依存グラフを機械可読**にしている。更新漏れを git hook で検出し、ADR の健全性も機械チェックする話 |
| **第6章** | **自律開発フロー** | issue 選定 → 仕様化 → 実装 → ドキュメント整合チェック → PR → CI 監視 → 自己修正 → マージ、を**人間の判断と検証だけ残して回す**話 |

「Claude Code を使ってみた」記事は世の中に大量にあります。差別化したいのは、**規律（ドキュメント設計）と自動化（自律開発フロー）まで作り込んだ実例**という点です。

### 章立て（予定）

* **序章**（この記事）— なぜ自作したか + 設計原則 + 全体像
* 第1章 — 一番やりたかったこと（コンパニオン ＝ 人格を持つ仲間として運用する）
* 第2章 — ファイルが API（Claude とアプリの双方向通信を「ファイル」と「PTY」だけで作る）
* 第3章 — 作った機能（音声入力・リマインド・統合スケジューラ・マルチインスタンス）
* 第4章 — Claude Skills / コマンド設計（段階的開示・失敗台帳・スコープ設計）
* **第5章 — ドキュメント設計** ⭐
* **第6章 — 自律開発フロー** 🚀
* 第7章 — 運用・課金・コスト
* 番外編 — iPhone から Tailscale で自宅のワークスペースに繋ぐ

各記事の冒頭・末尾からこの目次に戻れるようにして、回遊できる構成にします。

## 落とし穴: 「自作」は手段であって目的にしない

最後に、このアプローチを真似する人向けの注意を 1 つ。

**「自作した」こと自体に価値はありません。** 自作 IDE は、放っておくと「車輪の再発明を延々と磨く沼」になります。私がそれを回避できている（と思っている）のは、次の 3 点を最初に固定したからです。

1. **ユーザーを自分 1 人に限定した** — 配布・署名・設定 UI・後方互換、これら全部を「やらないこと」に倒せる
2. **やらないことを ADR で明記した** — 「コードエディタを作らない」と書面化しておくと、夜中に湧く「ついでにエディタも…」という誘惑を自分で却下できる
3. **成功基準を行動量で定義した** — 「1 年後、JetBrains を開く時間より Aidea を開く時間が長くなっていること」。LGTM 数でも機能数でもなく、自分の実利で測る

逆に言えば、これらを決めずに「統合 AI IDE を自作するぞ」と始めると、ほぼ確実に未完のまま放置されます。**割り切りの設計こそが、個人開発を完走させる本体**でした。
