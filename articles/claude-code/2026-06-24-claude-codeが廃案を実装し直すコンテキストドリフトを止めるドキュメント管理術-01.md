Claude Codeは、一度却下したはずの実装を、数セッション後に何事もなかったように書き直すことがある。  
半日かけて確定したはずの仕様や設計判断が、AIの出力のたびに過去へ巻き戻る。

この現象を **コンテキスト・ドリフト** と呼ぶ。  
「コンテキスト・ドリフト」は、次の4つの形で現れる。

* 廃案の復活（一度却下した案を、現行案として書き直す）
* 未決の既決化（まだ決めていないことを、決まったものとして実装する）
* 仕様と実装の取り違え（「こう動くべき」と「こう書いてある」を混同する）
* 古い前提の採用（すでに変わった前提を、まだ生きているものとして使う）

多くの開発者は、この問題に対して、プロジェクトのルールや経緯を「CLAUDE.md」や「AGENTS.md」へ詰め込むという対策をとる。  
この対策は、後述する実証データが示すとおり、むしろ性能を下げる。

本記事は、「コンテキスト・ドリフト」を構造的に抑えるドキュメント管理規格と、その最小の導入手順を示す。  
規格の名称は「コンテキスト・エンジニアリング・ブループリント」とする。  
コピー＆ペーストで再現できる設定ファイルと、その背後にあるLLMの認知上の弱点をセットで提示する。

## 詳細なコンテキストファイルが性能を下げる理由

詳細なコンテキストファイルをAIに与えるほど性能が上がるという前提は、実証実験で否定されている。  
ETH ZürichとLogicStar.aiの研究チームは、実在する12のPythonリポジトリから抽出した138件のバグ修正および機能追加タスクで、コーディングエージェントの性能を評価した（ベンチマーク「AGENTBENCH」、arXiv:2602.11988）。  
LLMが自動生成したリポジトリレベルのコンテキストファイルは、タスク成功率を平均3%低下させ、推論コストを20%以上増加させた。  
開発者が手動で作成したファイルであっても、成功率の改善は4%にとどまった。  
無秩序な情報の追加は、AIに過剰なファイル探索と無駄なテストを促し、本質的なタスクの解決を阻害する。

巨大なファイルが機能しない第一の理由は、LLMが入力の中盤を見落とす性質にある。  
スタンフォード大学などの研究が示した「Lost in the Middle」によれば、LLMは入力の先頭と末尾の情報には高い精度でアクセスするが、中間に配置された情報を見落としやすい（Liu et al., 2024）。  
入力の先頭にあるトークンは後続のすべてのトークンから参照されるのに対し、中間のトークンは全体からの参照量が相対的に低下する。  
この性能低下は、コンテキストウィンドウが128Kトークンに拡張されても解消されない。

第二の理由は、無関係な情報がモデルの注意を奪うことにある。  
関連する文脈から情報を抽出する注意ヘッドを「コンテキストヘッド」と呼ぶ（Zhu et al., 2025）。  
コンテキスト内にタスクと無関係な情報（ノイズや廃案のログ）が混入すると、「コンテキストヘッド」は関連情報へ十分な注意を割り当てられなくなる。  
無関係だが一貫した文脈を加えるだけで、主要モデルの推論性能は平均45%以上低下するという報告がある（Huang et al., arXiv:2502.01609）。  
原因はモデルの知識不足ではない。  
注意機構の構造的な偏りである。

これらの研究結果は、情報を一つのファイルに集約する手法が、LLMの構造的な弱点を意図せず突くことを示している。  
したがって、コンテキストは増やすのではなく、各情報に地位を与えて分離し、必要な最小限だけを渡す必要がある。

## 前提とする環境

本規格は、Claude Codeで複数セッションにまたがる開発を行う開発者を対象とする。  
読者には、Claude Codeの基本操作と、「フック（Hook）」および「スキル（Skill）」の概念に対する基礎的な理解を求める。

動作の前提は次のとおりである。  
Claude Codeは、フック機構とスキル機構に対応するバージョンを用いる。  
検証スクリプトはPython 3.8以降で動作し、標準ライブラリのみを使用する（外部パッケージのインストールは不要である）。  
対象環境はmacOS、Linux、およびWSL2である。

## 実装手順の全体像

本規格は、専用プラグインとして配布せず、設計書としてエージェントに渡す方式をとる。  
リポジトリは一つひとつ構成が異なるため、固定された梱包物よりも、各プロジェクトに合わせて構築される設計書のほうが適合しやすい。  
この方式は「スペック駆動開発（Spec-Driven Development）」に該当し、設計書そのものがAIへの指示として機能する。

設計書は次のリポジトリに置いた。

<https://github.com/Forest-Project-Lab/context-engineering-blueprint>

導入は、設計書をリポジトリに配置し、Claude Codeへ次のように指示するだけでよい。

```
この設計書をベースに、このリポジトリに合った最小構成を構築すること。
既存ファイルは変更せず、まず Level 1（最小5ファイルと薄いルーター）から開始すること。
```

本規格の核心は、確率的に判断するAIの「スキル」と、決定論的に制約を強制するシステムの「フック」を分離する点にある。  
役割分担を以下に示す。

| レイヤー | 性質 | 担当する責務 |
| --- | --- | --- |
| フック（Hook） | 決定論的。毎回必ず実行される | 不変条件の機械的な強制（リンタやCIに相当する） |
| スキル（Skill） | 確率的。文脈に応じてモデルが判断する | 手続きと判断の支援 |
| CLAUDE.md | 常時投入。毎ターン読み込まれる | 入口を示す薄いルーター |

以下の4点が、この規格の中核である。  
各手順は「目的」と「LLMのどの欠陥を補うか」をあわせて記述する。

## 手順1 全ドキュメントへのステータス付与

目的は、ドキュメントの「地位（Status）」をLLMが確定的に認識できるようにすることである。  
LLMにとって、リポジトリ内のすべてのテキストファイルは、均質な粒度の文字列に過ぎない。  
仕様書と廃案と未解決の課題が同じ場所にある場合、LLMには情報の地位を判定する能力がないため、古いログを現在の仕様として採用する。  
これが「コンテキスト・ドリフト」の発生源である。

第一歩として、すべてのドキュメント先頭のYAMLブロック（フロントマター）に、厳密に定義された7つのステータスのいずれかを強制的に付与する。  
自由記述のステータスは禁止する。  
統制された語彙だけを用いることで、現在の単一の真実（current）か、歴史的な証跡（archived）かを、AIに一意に認識させる。

| 値 | 意味 |
| --- | --- |
| `draft` | 草案。まだ現行ではない |
| `proposed` | 提案中。承認待ち |
| `current` | 現行。現在の真実 |
| `deprecated` | 廃案。撤回済み |
| `superseded` | 後継が存在し置換された |
| `archived` | 退役。証跡として保管 |
| `open` | 未決。決定していない |

各ドキュメントの先頭に置くフロントマターの最小形を以下に示す。  
必須項目を絞ることで、メタデータ管理の形骸化を抑える。

```
---
id: SPEC-014
title: 返金ポリシー
type: SPEC
status: current
owner: yamada          # チーム名ではなく個人名を記す。腐敗の責任の所在を示すため
updated: 2026-06-24
sources: [ADR-0009]    # 出所。人間由来かLLM由来か、どの根拠に基づくか
depends_on: [REQ-003]
---
```

## 手順2 保存時に走るドキュメントLinter

目的は、メタデータの品質をAI自身に自律的に維持させることである。  
メタデータ管理は、人間にとって退屈で形骸化しやすい作業である。  
運用が煩雑になると、開発者はメタデータの更新を怠り、規格全体が無用な装飾へと変わる。

これを回避するため、ファイルの変更時に実行されるドキュメントLinter（docs-linter.py）を「PostToolUse Hook」として組み込む。  
このLinterは、必須項目の欠落、不正なステータス、リンク切れを機械的に検知する。  
最も重要な設計は、Linterがエラーを検知した際の挙動にある。  
プロセスを停止して人間に介入を求めるのではなく、エラー出力をそのままClaudeのツール出力としてフィードバックする。  
これにより、AI自身が書式エラーを認識し、自律的に修正する自己修復ループが成立する。

Linterは標準ライブラリのみで実装する。  
外部パッケージへの依存をなくすことで、任意の環境で動作させる。

```
#!/usr/bin/env python3
# .claude/scripts/docs-linter.py  PostToolUse Hook が stdin で受け取る
import json, sys, re

STATUS = {"draft", "proposed", "current", "deprecated", "superseded", "archived", "open"}
STATUS_BY_TYPE = {"ADR": {"proposed", "accepted", "superseded", "deprecated"}}  # ADRのみ例外
REQUIRED = ["id", "title", "type", "status", "owner", "updated", "sources"]

def front(text):
    m = re.match(r"^---\s*\n(.*?)\n---", text, re.S)
    meta = {}
    if m:
        for line in m.group(1).splitlines():
            if ":" in line and not line.lstrip().startswith("#"):
                k, v = line.split(":", 1)
                meta[k.strip()] = v.strip().strip("[]")
    return meta

data = json.load(sys.stdin)
path = (data.get("tool_input") or {}).get("file_path", "")
if "/docs/" not in path or not path.endswith(".md") or "/99-archive/" in path:
    sys.exit(0)  # 対象外。アーカイブは不変なので検証もしない

meta = front(open(path, encoding="utf-8").read())
errs = [f"必須メタデータ欠落: {k}" for k in REQUIRED if not meta.get(k)]
allowed = STATUS_BY_TYPE.get(meta.get("type", ""), STATUS)
if meta.get("status") and meta["status"] not in allowed:
    errs.append(f"status '{meta['status']}' は型 {meta.get('type')} で不許可")

if errs:
    print(json.dumps({
        "decision": "block",
        "reason": "ドキュメントLinter違反:\n- " + "\n- ".join(errs),
        "continueOnBlock": True   # 停止せず、指摘をtool_resultとして返しAIに修正させる
    }))
sys.exit(0)
```

実運用では、参照先IDが実在するか（リンク切れ検出）、型とフォルダが整合するかの検査を追加する。  
このLinterを呼び出すフック設定を以下に示す。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "python3 \"${CLAUDE_PROJECT_DIR}/.claude/scripts/docs-linter.py\"",
            "continueOnBlock": true
          }
        ]
      }
    ]
  }
}
```

## 手順3 アーカイブと確定ADRの保護

目的は、AIによる過去の記録の改ざんを物理的に排除することである。  
アーキテクチャ決定記録（ADR）や、経緯を保存する「/99-archive」フォルダは、追記専用の記録として保護する。  
LLMは既存コードとの整合を取るために、過去の設計決定を現在の都合へ密かに書き換えるリスクを常に持つ。

これを防ぐため、「PreToolUse Hook」を用いて、保護されたパスへの書き込みや編集の要求を、システムレベルで拒否する。  
ADRは追記型として扱う。  
決定を覆す場合は、旧ADRを編集で消すのではなく、新しいADRを作成し、旧ADRに後継（superseded\_by）を記す。

```
#!/usr/bin/env python3
# .claude/scripts/archive-guard.py  PreToolUse Hook が stdin で受け取る
import json, sys, os, re

data = json.load(sys.stdin)
tool = data.get("tool_name", "")
path = (data.get("tool_input") or {}).get("file_path", "").replace("\\", "/")

protected = "/99-archive/" in path
adr_edit = bool(re.search(r"/40-decisions/ADR-.*\.md$", path)) and tool in {"Edit", "MultiEdit"} and os.path.exists(path)

if protected or adr_edit:
    reason = ("アーカイブ済み文書は不変です。現行は /20-spec で更新してください。" if protected
              else "ADRは追記型です。覆す場合は新しいADRを作成し、旧ADRに superseded_by を付けてください。")
    print(json.dumps({"hookSpecificOutput": {
        "hookEventName": "PreToolUse",
        "permissionDecision": "deny",
        "permissionDecisionReason": reason}}))
sys.exit(0)
```

物理的なアクセス制限を設けることで、過去の記憶の改ざんという挙動を排除する。

## 手順4 セッション開始時の最小コンテキスト注入

目的は、廃案を復活させる「燃料」をAIに渡さないことである。  
廃案となった提案の具体的な内容やコードを、セッション開始時のプロンプトに含めてはならない。  
前述の「コンテキストヘッド」の研究が示すとおり、LLMは与えられた文脈からもっともらしい続きを予測するため、廃案のロジックを読み込ませること自体が、その廃案を復活させる原因になる。

したがって「SessionStart Hook」では、ラベル、撤回日、非目標（Non-goals）、回帰監視リストといった、圧縮された事実関係の要約のみを渡す。  
廃案の具体的な中身は意図的に隠す。  
注入する内容は、本文全量ではなく、箇条書きの要点に限定する。

あわせて「CLAUDE.md」は、知識をすべて詰め込むデータベースではなく、必要なファイルの所在を示す薄いルーターとしてのみ機能させる。  
CLAUDE.mdの最小形を以下に示す。

```
# CLAUDE.md（薄いルーター）

このリポジトリの情報統治は /docs にある。ここに知識は集積しない。

## 常時参照する文書
- 確定済み事実: docs/90-llm/decided-facts.md
- 非目標（やらないこと）: docs/10-product/non-goals.md
- 戻してはならない事項: docs/70-test/regression-watchlist.md

## 規約
- 現行仕様は docs/20-spec/ にある。決定の経緯は docs/40-decisions/ にある（ADRは追記型）。
- docs/50-research/ と docs/99-archive/ の本文は現行ではない。現行として扱わない。
- 撤回済みの事項と未決の事項を、AIが独断で決定しない。未決は人間に返す。
```

タスク固有の詳細は、すべてリンク先に置く。  
常時投入を確定済みの事実に限定することで、CLAUDE.mdの肥大による性能低下とコスト増を抑える。

## 導入時のハマりどころ

導入の過程で想定されるつまずきと、その回避策を示す。

第一に、メタデータの付与漏れによるLinterの過検出である。  
必須項目を増やしすぎると、正当な編集のたびに警告が出て、運用が滞る。  
必須項目は7項目（id, title, type, status, owner, updated, sources）に絞り、依存関係などはレベルが上がった段階で段階的に必須化する。

第二に、フックの実行エラーである。  
スクリプトのパスが解決できない場合や、stdinのJSON解析に失敗する場合、フックは無言で停止する。  
パスは「${CLAUDE\_PROJECT\_DIR}」で解決し、`.md`以外のファイルは終了コード0で素通りさせる設計にする。

第三に、フックによる速度低下である。  
毎ターン実行されるLinterがリポジトリ全体を走査すると、エージェントが体感的に遅くなる。  
毎ターンのフックは触れた単一ファイルのみを検証し、全件のリンク切れ検査や陳腐化検査は、コマンドまたはCIへ分離する。

第四に、運用の形骸化である。  
メタデータが更新されず、回帰監視リストが保守されなくなった時点で、本規格は実態のない装飾へと変わる。  
ステータスと更新日の記入はAIに代行させ、人間は承認のみを担うことで、更新コストを下げる。

## まとめ

「コンテキスト・ドリフト」の原因は、プロンプトの文言ではない。  
現行の仕様も、廃案も、未決の課題も、同じ粒度で並び、情報に地位が与えられていないことである。

本規格は、この問題に対して4つの構造的な対策を提供する。  
すべての文書にステータスを付与し、保存時にLinterで自己修復させ、アーカイブと確定ADRをフックで保護し、セッション開始時に事実の要約のみを注入する。  
いずれも、LLMの注意機構の偏り（Lost in the MiddleとCDV）を構造の側から補う設計である。

導入は、設計書をリポジトリに置き、Claude Codeに構築を指示するだけでよい。  
設計書には、26種の文書型、変更時の影響分析フロー、各種テンプレートを同梱した。

<https://github.com/Forest-Project-Lab/context-engineering-blueprint>

まず Level 1 の最小構成から開始し、運用の痛みが生じた箇所だけを拡張することを推奨する。
