## はじめに

Lovable と Anthropic が公式の MCP (Model Context Protocol) ネイティブ統合を発表しました (公式公表: [lovable.dev/mcp](https://lovable.dev/mcp))。Claude のチャット・Claude Code・Claude Design から Lovable のワークスペースに直接接続できる設計です。

発表翌日に Nihonbashi AI Lab で運用している環境に対して接続して動かしてみたところ、「業務担当者が一人で複数の業務システムを設計・運用する」という状態に、これまで以上に近づきつつあると感じました。本記事では、この統合で何が実現できるようになるかを 3 つの角度で整理します。

経営判断・業務応用視点での整理は別途運営元の lab note にまとめています (canonical)。本稿は同じテーマを、現場の業務担当者から見た「日々のワークフローがどう変わるか」に振った内容です。

## 1. 「設計 → 実装 → 検証 → デプロイ」のループが一人で一気通貫になる

これまで、業務システムを内製化するときには「自然言語で要件を伝える」「実装させる」「ブラウザで動作確認する」「データベースの中身を SQL Editor で確認する」「本番にデプロイする」というステップが、それぞれ別ツールに分散していました。複雑な要件になるほど、ツール間を行き来する時間と認知負荷が膨らんでいきます。

公式 MCP 統合で、これらが Claude のチャットや Claude Code のターミナルから**一気通貫で扱える**ようになります。

具体的には、Claude Code 側から:

* プロジェクトの最新コミット・本番画面のスクリーンショット・過去の編集履歴を取得
* 本番データベースに SELECT を直接打って結果を見る (Lovable Cloud 配下のプロジェクトの場合)
* Lovable AI への custom instructions (Project Knowledge) を読み書き
* 必要に応じて Lovable 側にビルド・デプロイを依頼

これが一つのインターフェースから扱えるので、「自然言語で要件を伝える → 動かす → 確認する → 修正する」のループが、業務担当者一人の手元で完結する形に近づきます。「業務システムを作ったあと、運用や監視に使うツールが急に増える」という従来の悩みが軽くなります。

## 2. 一人で複数の業務システムを横断管理できる

業務担当者が業務システムを内製化していくと、現場の数だけシステムが増えていきます。営業案件管理 / 顧客問い合わせ / 社内申請承認 / 在庫発注 / 予約管理 — それぞれが小さく動くのは良いのですが、**全体を見る**のが負担になっていきます。

公式 MCP 統合は、複数プロジェクトをワークスペースとしてまとめて扱う設計になっています。Claude Code から:

* ワークスペース内の全プロジェクトを一覧
* 各プロジェクトの最新ステータスをまとめて取得
* 「あるプロジェクトに入っている運用ルールが、別のプロジェクトに展開されているか」のような**横断チェック**を、目視に頼らず実行

ができます。Nihonbashi AI Lab でも、複数の自社サービスを並行運用していますが、「各プロジェクトの設定や運用ルールが揃っているか」をこれまで人手で確認していた部分が、自動化される実感があります。

業務担当者がチーム単位で内製化を進めるとき、「各メンバーが独自ルールで業務システムを組んでバラバラになっていく」という、よく見るパターンを未然に防ぐ仕組みが、ようやくツール側に乗ったと感じます。

## 3. AI が「画面を作る AI」から「運用パートナー」に役割を広げる

これまで Lovable のような AI 駆動開発プラットフォームは、「画面を作る」「ボイラープレートを書く」というところに役割が集中していました。実装後の運用 — データの確認、エラー監視、改善提案 — は別ツールや人手に任せる前提でした。

公式 MCP 統合で、Claude (チャットや Claude Code) が「**運用パートナー**」として動ける幅が広がります。

* 業務担当者が「先月の問い合わせ件数を、業種別にまとめて」と話しかける → Claude が本番データベースを直接見て集計
* 「先週の Lovable の編集履歴を要約して」と頼む → 編集ログを読んで「どこを誰がいつ変えたか」をまとめる
* 「Project Knowledge に運用ルールを追記して」と頼む → そのまま Lovable AI への指示が更新される

「**画面を作る AI**」と「**運用を見る AI**」が分かれていた状態から、「一つの AI 駆動環境で要件定義から運用まで担当する」状態に近づきます。業務担当者から見ると、「日々の運用業務の一部を AI 駆動の運用パートナーに渡しながら、自分はより本質的な業務改善に集中する」という働き方が、もう少し現実的になってきました。

## ただし、「すべてが AI で完結する」わけではない

実際に運用して見えたのは、公式統合があっても **Lovable がホスティング・デプロイの中心であり続ける** という構造でした。本番反映のパイプライン、複雑なマイグレーション適用、シークレット管理、ビルド側で使うスキル — これらは Lovable Cloud がまだ強く担っており、Claude Code 側からは依頼する形になります。両者は補完関係です。

また、「業務設計 / データ構造・権限設計 / AI 活用設計」のような、上流の判断レイヤーは引き続き人間が担います。MCP 統合で運用導線の確認・実行が楽になったぶん、「**何を作るか・何を見るか**」を決める部分が、相対的に価値を増していると感じます。

## まとめ

Lovable × Claude 公式 MCP 統合で見えたのは、「業務システムを作る AI」と「運用を見る AI」が一つの環境に統合され、業務担当者一人が複数の業務システムを横断して回せる状態に近づきつつある、ということです。

経営判断としてこの統合をどう運用に組み込むか、Nihonbashi AI Lab が運用の中で見えた含意の整理は、運営元の lab note (canonical) に書きました。
