## MCPサーバー開発における「再起動問題」

MCPサーバーを開発していると、ある壁にぶつかる。

1. AIエージェント（Claude Codeなど）にMCPサーバーのコードを修正させる。
2. 修正が終わる。
3. 動作確認をする。

──というところまで来て、**AIは自分が接続しているMCPサーバーを再起動できない**ことに気づく。

MCPの仕組み上、クライアント（AIエージェント）とサーバーはstdioやSSEで接続されている。サーバーを再起動するということは、その接続自体を切断するということ。通常はAIの再起動が必要。

結果として、コードを修正するたびに人間が手動でMCPサーバーを再起動し、AIエージェントとの接続を復旧させる必要がある。面倒くさい。

**コード修正 → 再起動（人間） → 接続復旧（人間） → 動作確認 → 問題発見 → コード修正 → 再起動（人間）→ ……**

AIがせっかく高速にコードを書いても、このループの中で人間がボトルネックになる。PDCAが回らない。

## 発想の転換：MCPサーバーと実体を分離する

解決策は、**MCPサーバーにはインターフェースだけを置き、ロジックの実体は別のHTTPサーバーとして動かす**、というアプローチ。

MCPサーバー自体は起動したまま、実体のHTTPサーバーだけをAIが再起動する。MCP接続は切れない。PDCAが止まらない。

というわけで、作ったのが「実体サーバーの起動・停止・再起動」を管理する**Webserver4AI**。

## Webserver4AIが提供するもの

Webserver4AIはMCPサーバーとして動作し、AIエージェントに以下のツールを提供する。

| ツール | 役割 |
| --- | --- |
| `server_start` | サーバーを起動する |
| `server_stop` | サーバーを停止する |
| `server_restart` | サーバーを再起動する |
| `server_list` / `server_status` | 稼働中のサーバーを確認する |

### コマンドモード：MCPサーバー開発の本命

任意のコマンドをマネージドプロセスとして起動できる。MCPサーバー開発では、このモードが核になる。

```
server_start({
  name: "my-mcp-backend",
  directory: "/home/user/my-mcp-server",
  command: "python -m uvicorn app:app --port 8001",
  health_check: "http://localhost:8001/health"
})
```

ヘルスチェックURLを指定すれば、サーバーが実際にリクエストを受け付けられる状態になってから応答が返る。AIエージェントは起動完了を待ってすぐに動作確認に移れる。

コードを修正したら `server_restart` を呼ぶだけ。MCP接続を維持したまま、実体だけが再起動される。

### 静的ファイル配信モード

MCPサーバー開発に限らず、AIが生成したHTML/CSS/JSをすぐに確認したい場面でも使える。ライブリロード機能付き。Webサーバーなどをぶら下げた場合は、AIがWebサーバーを再起動した瞬間に、瞬時に表示が切り替わる。

## 具体的なワークフロー

MCPサーバー開発でのWebserver4AIの使い方を具体的に示す。

### 1. 構成

MCPクライアントの設定に**2つのMCPサーバー**を登録する。

```
{
  "mcpServers": {
    "webserver4ai": {
      "command": "npx",
      "args": ["-y", "webserver4ai"]
    },
    "my-mcp-server": {
      "command": "node",
      "args": ["./my-mcp-server/dist/index.js"]
    }
  }
}
```

* `webserver4ai`：サーバー管理用。実体サーバーの起動・停止・再起動を担当。
* `my-mcp-server`：開発中のMCPサーバー。インターフェース（ツール定義）だけを持ち、実際の処理は実体サーバーにHTTPで委譲する。

### 2. 開発ループ

AIエージェントに「MCPサーバーの○○機能を実装して」と指示すると、以下のループが**全自動**で回る。

1. AIが実体サーバーのコードを修正する
2. AIが `server_restart` で実体サーバーを再起動する
3. AIがMCPツール経由で動作確認する（実体サーバーにHTTPリクエストが飛ぶ）
4. 問題があればコードを修正し、2に戻る

人間の介入は不要。AIが自律的にPDCAを回し続ける。

### 3. MCPサーバー側の実装パターン

開発中のMCPサーバーは、ツールのハンドラ内で実体サーバーにHTTPリクエストを投げるだけの薄いラッパーになる。

```
// my-mcp-server/src/index.ts（イメージ）
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  // 実体サーバーにHTTPで委譲
  const response = await fetch("http://localhost:8001/api/do-something", {
    method: "POST",
    body: JSON.stringify(request.params),
  });
  return { content: [{ type: "text", text: await response.text() }] };
});
```

この構成なら、実体サーバー（port 8001）をWebserver4AIで何度再起動しても、MCPサーバー自体は動き続ける。

## なぜ専用ツールが必要なのか

「`server_restart`なんて、bashで `kill` して再起動すればいいのでは？」

実際にやらせてみるとわかるが、これが意外とうまくいかない。

* **サーバーが生きているか判別できない**：AIがサーバーを落とそうとするとき、まずそのプロセスが生きているかを確認する必要がある。`ps`や`lsof`で調べて、PIDを特定して、killして……という手順に時間とトークンを浪費する。Webserver4AIなら `server_status` 一発で状態がわかる。
* **落とせないケースがある**：プロセスが子プロセスを生んでいたり、シェル経由で起動されていたりすると、単純な `kill` では止まらないことがある。残ったゾンビプロセスがポートを握り続けて、再起動もできなくなる。
* **`kill -9`の乱発**：上記の問題にAIが対処しようとすると、`kill -9`を乱発し始める。これは非常に危険だ。関係ないプロセスを巻き込んだり、ファイルの書き込み途中で強制終了してデータを壊したりするリスクがある。

Webserver4AIは起動したプロセスを自分で追跡しているので、確実に・安全に・一発で停止・再起動できる。

## まとめ

Webserver4AIの存在意義は、**MCPサーバー開発においてAIエージェントのPDCAを止めないこと**にある。

MCPの構造上、AIエージェントは自分が接続しているサーバーを再起動できない。この制約を、「インターフェースと実体の分離」というアーキテクチャで回避し、実体サーバーの管理をWebserver4AIに委ねることで、AIによる完全自動の開発ループを実現する。

静的ファイル配信やライブリロードといった機能も備えているが、本質は**AIエージェントが自律的に「書いて、動かして、確認して、直す」を繰り返せるインフラ**だ。
