## はじめに：1 つのコンテキストの限界

Claude Code は長く複雑なタスクになるほど、3 つの構造的な問題にぶつかります。[Anthropic の公式ブログ](https://claude.com/blog/a-harness-for-every-task-dynamic-workflows-in-claude-code)が整理した失敗パターンです。

| 失敗パターン | 起きること | 例 |
| --- | --- | --- |
| **怠惰**（Agentic laziness） | 途中で止まり、部分的な進捗で完了宣言する | セキュリティレビュー 50 項目中 35 項目で「完了」 |
| **自己バイアス**（Self-preferential bias） | 自分の結果を自分で検証すると甘くなる | 自分が書いたコードのレビューで問題を見つけられない |
| **目標逸脱**（Goal drift） | ターンを重ねるうちに要件や制約が失われる | 「〇〇しないで」がコンパクション後に消える |

これらは Claude の能力の問題ではなく、**1 つのコンテキストで計画と実行を同時にやる構造の問題**です。**Dynamic Workflow** は計画をスクリプトに移し、各エージェントに独立したコンテキストを与えることで、これらを構造的に解決します。

## Dynamic Workflow とは何か

Dynamic Workflow は、Claude Code v2.1.154 で追加された機能です。ひとことで言えば、**Claude がタスクに合わせたオーケストレーション用スクリプトを自動生成し、数十〜数百の subagent をバックグラウンドで並列実行する仕組み**です。

### 通常セッションとの違い

通常の Claude Code セッションでは、**計画（次に何をやるか）も実行（ファイルを読む・書く・コマンドを走らせる）も、すべて 1 つのコンテキストウィンドウの中**で行われます。Claude がターンごとに「次は何をしよう」と考え、1 つずつ処理していきます。

Dynamic Workflow では、この構造が変わります。

1. **Claude がまず JavaScript のスクリプトを書きます**。このスクリプトには「どのタスクを、どの順番で、どう並列に回すか」が確定的にコード化されます
2. **ランタイムがそのスクリプトをバックグラウンドで実行**します。スクリプトに従って subagent が起動され、各 subagent は**独立したコンテキストウィンドウ**で作業します
3. **セッションは応答可能なまま**です。Dynamic Workflow が裏で動いている間も、別の作業ができます
4. **結果は最終的にセッションに返ります**。途中経過はスクリプト内の変数に留まり、Claude のコンテキストには最終結果だけが入ります

ユーザーがスクリプトを手書きする必要はありません。`ultracode` キーワードを付けてタスクを指示するだけで、Claude が適切なスクリプトを自動生成します。

### なぜこれで 3 つの問題が解決するのか

通常セッションの 3 つの失敗パターンは、いずれも「1 つのコンテキストに全部載せる」構造から生まれます。Dynamic Workflow はこの構造自体を変えることで、根本的に解決します。

| 問題 | 通常セッションで起きる理由 | Dynamic Workflow の解決方法 |
| --- | --- | --- |
| **怠惰** | 50 項目の作業を 1 つのコンテキストで処理すると、後半で集中が切れて「もう十分」と判断してしまう | 50 項目なら 50 エージェントが各 1 項目を担当する。各エージェントの目標が小さく明確なので、途中で止まらない |
| **自己バイアス** | 自分が出した結果を同じコンテキストで検証するため、無意識に甘い評価になる | 結果を出したエージェントとは**別のエージェント**が検証する。コンテキストが分離しているので、先入観が入らない |
| **目標逸脱** | 長いセッションでコンテキストが圧縮（コンパクション）されると、要件や「〇〇しないで」という制約が要約から落ちる | 各エージェントのコンテキストが独立かつ短命なので、コンパクション自体が発生しない |

実例として、JavaScript ランタイム **Bun** は Zig から Rust への全面書き換えに Dynamic Workflow を活用しています。タスクをコールサイト・テスト・モジュール単位に分割し、各修正を独立したエージェントが別 worktree で実行、さらに別のエージェントが修正をレビューする構成です。

## 他のマルチエージェント機能との使い分け

Claude Code には複数の並列化手段があります。違いは「誰が計画を持つか」です。

|  | subagent | skills | agent teams | Dynamic Workflow |
| --- | --- | --- | --- | --- |
| 計画の持ち主 | Claude（ターンごと判断） | Claude（手順に従う） | リーダーエージェント | スクリプト |
| 中間結果の置き場 | Claude のコンテキスト | Claude のコンテキスト | 共有タスクリスト | スクリプト変数 |
| スケール | 1 ターンに数件 | subagent と同等 | 数名の長期セッション | **数十〜数百エージェント** |
| 再現性 | ワーカー定義のみ | 手順のみ | チーム定義のみ | **オーケストレーション全体** |
| 中断時 | ターンからやり直し | ターンからやり直し | teammate は継続 | **同一セッション内で resume 可能** |

### Agent Teams との違い

Dynamic Workflow と agent teams はどちらも複数エージェントを使いますが、核心的な違いは**エージェント同士が会話するかどうか**です。

**Agent Teams** は teammate 同士がメッセージを送り合い、議論・反論・合意形成ができます。**Dynamic Workflow** は各エージェントが独立に結果を返すだけで、エージェント同士の通信はありません。スクリプトが結果を集約し、次のステップを確定的に制御します。

同じ「3 つの視点からレビューする」タスクでも動き方が異なります。

|  | Agent Teams | Dynamic Workflow |
| --- | --- | --- |
| 実行 | 3 人が独立にレビュー → 指摘を突き合わせて**議論** | 3 人が独立にレビュー → **スクリプトが結果を集約** |
| 対立時 | teammate 同士が反論し合い、立場を変えることもある | エージェント間の議論はない。スクリプトが判定する |
| 向く場面 | 対立する意見を**議論で深めたい** | 大量の結果を**構造的に処理したい** |
| スケール | 3〜5 名が現実的 | 数十〜数百でも動く |
| 再現性 | 毎回異なる議論になる | スクリプトを保存して同じ構成を再実行 |

判断基準はシンプルです。**エージェント同士が議論する必要があるか?** ——必要なら agent teams、独立した結果を集約すれば十分なら Dynamic Workflow です。

## どんなタスクに向くか

Dynamic Workflow が効くタスクは、突き詰めると **3 つの条件** に集約されます。

| パターン | 解決する問題 | 該当ユースケース |
| --- | --- | --- |
| **数が多い** — 1 コンテキストでは処理しきれない量がある | 怠惰 | マイグレーション、ファクトチェック、ソート、トリアージ |
| **独立した検証が要る** — 自分で自分を評価すると甘くなる | 自己バイアス | リサーチ、ルール準拠チェック |
| **多角的な探索が要る** — 1 つの視点では見落としが出る | 目標逸脱・確証バイアス | ルート原因調査、探索、評価、モデルルーティング |

以下、[公式ブログ](https://claude.com/blog/a-harness-for-every-task-dynamic-workflows-in-claude-code)の 10 ユースケースをこの 3 条件で分類して紹介します。

### パターン 1：数が多い — 分割して並列処理したい

1 つのコンテキストでは処理しきれない量のタスクを、エージェント単位に分割して並列実行します。怠惰（途中で止まる問題）の解決に直結します。

**マイグレーション・リファクタリング** — 大規模なコード変更を、コールサイト・テスト・モジュールなどの単位に分割し、各修正を独立エージェントが**別の worktree** で並列実行します。修正ごとに別のエージェントがレビューする構成も組めます。JavaScript ランタイム **Bun** が Zig から Rust への全面書き換えにこのパターンを使ったことが公式ブログで紹介されています。

```
ultracode: User モデルを Account にリネームして、全ファイルに反映せよ
```

**ファクトチェック・検証** — 文書中の技術的主張を 1 件ずつ抽出し、各主張を独立したエージェントがコードベースや公開ソースに照らして検証します。全主張を 1 つのコンテキストで処理すると後半で手を抜きがちですが、1 主張 1 エージェントなら目標が明確なので最後まで手を抜きません。

```
ultracode: このブログ記事のドラフトの技術的主張をすべてコードベースで検証せよ。間違いを出荷したくない
```

**ソート・ランク付け** — 1,000 件超のリストを定性的な基準でランク付けするタスクです。絶対スコア（「これは 8 点」）は信頼性が低いため、**ペアワイズ比較**（「A と B ではどちらが上か?」）をトーナメント形式で並列実行し、最終順位を決めます。

```
ultracode: この CLI ツールの名前を大量にブレインストームし、トーナメントでトップ 3 を選べ
```

**大規模トリアージ** — 大量のアイテム（履歴書、サポートチケット、インシデントレポート等）を自動分類し、重複を排除し、優先度付けします。信頼できないコンテンツ（外部入力）を読み取るエージェントと、アクションを実行するエージェントを分離する**隔離パターン**が推奨されており、プロンプトインジェクションのリスクを構造的に軽減できます。

```
ultracode: 80 件の履歴書をバックエンド職でランク付けし、上位 10 件をダブルチェックせよ。ルーブリックは AskUserQuestion で聞け
```

### パターン 2：独立した検証が要る — 自分で自分を評価すると甘くなる

結果を出したエージェントとは別のエージェントが検証する構成を作り、自己バイアスを構造的に排除します。

**深いリサーチ** — 質問を複数の角度に分解し、並列に Web 検索をかけ、見つけたソースを相互検証してから引用付きレポートにまとめます。組み込みの `/deep-research` がこのパターンの代表例です。単一セッションだと検索結果を自分で評価することになりますが、Dynamic Workflow では検索エージェントと検証エージェントが分離しています。

```
/deep-research Node.js の Permission Model は v20 から v22 で何が変わったか？
```

**ルール準拠チェック** — プロジェクトのルール（CLAUDE.md、コーディング規約等）に対する違反を検出する専用 verifier エージェントを配置します。逆方向の使い方もあり、過去セッションの修正履歴をマイニングして「繰り返し起きている修正パターン」を発見し、新しいルールとして提案させることもできます。

```
ultracode: 直近 50 セッションを調べ、繰り返している修正を CLAUDE.md のルールにせよ
```

### パターン 3：多角的な探索が要る — 1 つの視点では見落とす

独立したエージェントに異なる角度から並列に取り組ませ、確証バイアスや視野の狭さを防ぎます。

**ルート原因調査** — バグやインシデントに対して、複数の独立した仮説を**互いの仮説を見ずに**並列生成し、それぞれが証拠を集めます。その後、仮説同士を突き合わせて生き残ったものを真因とします。1 つのセッションで順番に仮説を立てると最初の仮説に引っ張られますが、独立エージェントならそれを構造的に排除できます。コードだけでなく、売上データの分析やデータパイプラインの障害調査にも使えます。

```
ultracode: このテストは 50 回に 1 回失敗する。再現させて、競合する仮説を立てろ。1 つの仮説が証拠に耐えるまで止めるな
```

**探索と定性判断** — 設計案、ネーミング、ビジネスプランなど、正解がない定性的な判断を複数視点で並列に評価します。各エージェントに異なるペルソナ（投資家・顧客・競合）を割り当て、それぞれの視点からルーブリックに基づいて評価させます。

```
ultracode: このビジネスプランを投資家・顧客・競合の視点からそれぞれ批判せよ
```

**評価（Evals）** — スキルやプロンプトの出力品質を、ルーブリックに照らして採点する軽量な評価フレームワークとして使えます。同じ入力に対して複数回実行し、出力のばらつきや品質をエージェントが自動評価することで、スキル開発の反復的な改善サイクルを回せます。

**モデル・知能ルーティング** — タスクの複雑度（対象ファイル数、コードベースの構造、予想されるツール呼び出し数など）をエージェントが判定し、Sonnet と Opus を自動的に振り分けます。単純なタスクにはコストの低い Sonnet、複雑な推論が要るタスクには Opus を割り当てることで、品質を維持しながらコストを最適化できます。

## 実測：ルール準拠チェックを Dynamic Workflow で回してみた

ユースケース「パターン 2：ルール準拠チェック」を実際に試しました。直近 50 セッションのトランスクリプトから、繰り返し起きている修正パターンを抽出し、CLAUDE.md のルールとして整理するタスクです。

```
ultracode: 直近 50 セッションを調べ、繰り返している修正を CLAUDE.md のルールにせよ
```

Claude はまずセッション格納先を確認し、505 件の修正・フィードバックを抽出。その後 Dynamic Workflow を起動し、6 エージェントがチャンクを並列分析、最後に統合エージェントがパターンを整理しました。

| 項目 | 値 |
| --- | --- |
| エージェント数 | **7**（分析 6 + 統合 1） |
| 抽出した修正件数 | 505 件 |
| 発見したパターン数 | 51 → 上位に絞り込み |
| 所要時間（API） | 約 14 分 |
| コスト | **$10.99** |

**実測**: コストの大半は Opus のキャッシュ読み書き（約 800 万トークン）です。505 件の修正を 1 セッションで処理すると agentic laziness で後半の分析が雑になりますが、6 エージェントに分割したことで各チャンクの分析品質が均一に保たれました。最頻出パターンは「質問への直接回答の欠如」（11 回）で、実際にセッション中の修正指示と一致していました。

## 全体像：使い方マップ

Dynamic Workflow には複数の使い方があります。まず全体像を押さえてから、それぞれの詳細に進みます。

```
┌─────────────────────────────────────────────────────┐
│              Dynamic Workflow の使い方               │
├──────────────┬──────────────────────────────────────┤
│  自分のタスク  │  ① ultracode キーワードで 1 回だけ起動  │
│  を実行したい  │  ② /effort ultracode でセッション全体   │
│              │  ③ 保存済みを /<名前> で再実行          │
├──────────────┼──────────────────────────────────────┤
│  まず試したい  │  ④ /deep-research（組み込み済み）       │
├──────────────┼──────────────────────────────────────┤
│  調整したい   │  ⑤ 生成されたスクリプトを編集 → 再実行   │
└──────────────┴──────────────────────────────────────┘
```

| # | 方法 | 一言で |
| --- | --- | --- |
| ① | `ultracode: 〇〇せよ` | 1 回限りで Claude にスクリプトを書かせる |
| ② | `/effort ultracode` | 全タスクで自動的に Dynamic Workflow を検討させる |
| ③ | `/<名前>` | 保存済みスクリプトをコマンドとして再実行 |
| ④ | `/deep-research <質問>` | 組み込み済みのリサーチ用 Dynamic Workflow |
| ⑤ | スクリプト編集 → 再実行 | Claude が書いたスクリプトを微調整して再実行 |

どの方法でも、**スクリプトは Claude が書きます**。ユーザーが JavaScript を手書きする必要はありません。

## セットアップ

### Step 1: バージョンを確認する

Dynamic Workflow には **v2.1.154 以降**が必要です。

### Step 2: 有効化する

| 方法 | 手順 | 持続 |
| --- | --- | --- |
| `/config` | Dynamic workflows の行をオンにする | セッション超えて持続 |
| `settings.json` | `"disableWorkflows": false` を確認（デフォルト有効） | 持続 |
| `/effort ultracode` | ultracode モードを有効化（Dynamic Workflow を自動適用） | セッション中のみ |

Pro プランでは `/config` の Dynamic workflows 行からオンにします。それ以外の有料プランではデフォルトで有効です。

## 使い方

### Step 1: Dynamic Workflow を起動する

Dynamic Workflow を起動するには 3 つの方法があります。

**方法 A: `ultracode` キーワード（1 回限り）**

プロンプトに `ultracode` と書くと、Claude がそのタスク用の Dynamic Workflow スクリプトを自動生成します。セッション設定は変わらず、そのプロンプトだけに効きます。

```
ultracode: src/routes/ 配下の全 API エンドポイントで認証チェック漏れを検出せよ
```

Claude Code がキーワードをハイライト表示します。意図しない場合は macOS で `Option+W`、Windows/Linux で `Alt+W` でハイライトを解除できます。キーワード自体を無効にしたいときは `/config` の Ultracode keyword trigger をオフにします。

自然言語で「Dynamic Workflow を使って」「ワークフローで実行して」と書いても同じ効果です。

**方法 B: `/effort ultracode`（セッション全体）**

`/effort ultracode` は、**最高の推論努力**（xhigh）と **Dynamic Workflow の自動適用**を組み合わせたモードです。設定すると、キーワードなしでも Claude が**すべてのタスクで Dynamic Workflow を自動検討**します。

1 つのリクエストが複数の Dynamic Workflow に分かれることもあります。たとえば「コードを理解する → 変更を加える → 検証する」のように連鎖します。そのぶんトークン消費は大きくなります。

通常作業に戻すときは `/effort high` で解除します。セッション中のみ有効で、新しいセッションではリセットされます。

| effort レベル | 推論努力 | Dynamic Workflow |
| --- | --- | --- |
| low〜high | 段階的 | 手動リクエスト時のみ |
| xhigh | 最高 | 手動リクエスト時のみ |
| **ultracode** | 最高 | **全タスクで自動適用** |

**方法 C: 保存済み Dynamic Workflow をコマンドで実行**

保存した Dynamic Workflow は `/<名前>` で直接起動できます。`ultracode` キーワードも effort 設定も不要です。

### Step 2: 承認する

Claude がスクリプトを生成すると承認プロンプトが表示されます。

| 選択肢 | 動作 |
| --- | --- |
| Yes, run it | 実行開始 |
| Yes, and don't ask again | このプロジェクトでこの Dynamic Workflow の承認を省略 |
| View raw script | スクリプトを確認してから判断 |
| No | キャンセル |

`Ctrl+G` でエディタにスクリプトを開いて編集することもできます。

### Step 3: 進捗を監視する

実行中は `/workflows` で進捗をリアルタイムに確認できます。

| キー | 操作 |
| --- | --- |
| `↑` / `↓` | フェーズ・エージェントを選択 |
| `Enter` / `→` | 詳細にドリルダウン |
| `Esc` | 1 階層戻る |
| `p` | 一時停止 / 再開 |
| `x` | 選択中のエージェントまたは Dynamic Workflow 全体を停止 |
| `s` | スクリプトをコマンドとして保存 |

### Step 4: 保存して再利用する

同じオーケストレーションを繰り返したいときは、`/workflows` で `s` を押して保存します。

| 保存先 | スコープ |
| --- | --- |
| `.claude/workflows/` | プロジェクト共有（Git 管理向き） |
| `~/.claude/workflows/` | 個人用（全プロジェクトで利用可能） |

保存した Dynamic Workflow は `/<名前>` で起動できます。`args` パラメータで入力を渡すことも可能です。

## `/deep-research`：組み込み済み Dynamic Workflow

Claude Code には `/deep-research` という**最初から組み込まれた Dynamic Workflow** が同梱されています。`ultracode` キーワードも effort 設定も不要で、コマンドを打つだけで動きます。

```
/deep-research Node.js の Permission Model は v20 から v22 で何が変わったか？
```

裏では以下のオーケストレーションが実行されます。

1. 質問を複数の角度に分解し、**Web 検索を並列にファンアウト**
2. 見つけたソースを**フェッチして相互検証**（クロスチェック）
3. 各クレーム（主張）に対して**投票**し、検証を生き残らなかったものを除外
4. **引用付きレポート**として合成して返す

単なる並列検索ではなく信頼性の高いレポートを返せるのは、裏で**エージェント同士の相互検証**が組み込まれているからです（[Steel 社の実装解析](https://steel.dev/blog/claude-code-deep-research-autopsy)に詳しい）。`ultracode` で自分のタスクを実行するときも、Claude はタスクの性質に応じてこうした検証の仕組みをスクリプトに自動で組み込みます。

## 仕組み：スクリプトの構造

Dynamic Workflow のスクリプトは通常 **Claude が自動生成**します。ユーザーがゼロから書く必要はありません。ただし、生成されたスクリプトを確認・編集・再実行する場面は頻繁にあるので、構造を理解しておくと調整がしやすくなります。

### ファイル形式

| 項目 | 内容 |
| --- | --- |
| 言語 | **素の JavaScript**（TypeScript 構文は不可） |
| 拡張子 | `.js` |
| 実行環境 | async コンテキスト（トップレベルで `await` が使える） |

### スクリプトの配置

| 場所 | いつ作られるか | 用途 |
| --- | --- | --- |
| `~/.claude/projects/.../<session>/` | Claude がスクリプトを生成するたびに自動保存 | 実行ログ。編集して再実行も可能 |
| `~/.claude/workflows/<name>.js` | `/workflows` で `s` を押して保存（個人用） | 全プロジェクトで `/<name>` として再利用 |
| `.claude/workflows/<name>.js` | `/workflows` で `s` を押して保存（プロジェクト用） | Git 管理でチーム共有 |

プロジェクト用と個人用で同じ名前がある場合、**プロジェクト用が優先**されます。

### スクリプトの構成

スクリプトは **meta ブロック**（メタデータ）と**本体**（オーケストレーション）の 2 段構成です。

```
// meta ブロック — 必ずファイル先頭に書く
export const meta = {
  name: 'find-auth-bugs',
  description: '全 API エンドポイントの認証漏れを検出する',
  phases: [
    { title: 'スキャン', detail: 'エンドポイント一覧を収集' },
    { title: '検証', detail: '各エンドポイントの認証チェック' },
  ],
}

// 本体 — agent / parallel / pipeline / phase / log が使える
phase('スキャン')
const endpoints = await agent('src/routes/ 配下の全 API エンドポイントを列挙せよ', {
  schema: ENDPOINTS_SCHEMA,
})

phase('検証')
const results = await parallel(
  endpoints.items.map(ep => () =>
    agent(`${ep.path} に認証チェックがあるか検証せよ`, {
      label: `check:${ep.path}`,
      schema: VERDICT_SCHEMA,
    })
  )
)
```

### 主な API

スクリプトの中身を読むために、主要な関数の役割だけ押さえておきます。

| 関数 | ひとことで言うと |
| --- | --- |
| `agent(prompt)` | subagent を 1 つ起動して結果を返す |
| `parallel([...])` | 複数の処理を並行実行し、**全部終わるまで待つ** |
| `pipeline(items, ステージ1, ステージ2, ...)` | 各アイテムをステージに順番に通す。**ステージ間で待たない**のが parallel との違い |
| `phase('名前')` | `/workflows` の進捗表示でフェーズを切り替える |
| `log('メッセージ')` | ユーザーに進捗メッセージを表示する |

### スクリプトの編集と再実行

Claude が生成したスクリプトを編集して再実行すると、**変更されていないエージェントはキャッシュから即座に返り**、変更部分だけが新たに実行されます（resume）。

1. `ultracode` でタスクを実行 → スクリプトが自動生成される
2. 結果を確認し、プロンプトの調整が必要と判断
3. `Ctrl+G` またはファイルパスを聞いてエディタで編集
4. Claude に再実行を依頼 → 変更部分のみ実行

### ランタイムの制約

| 制約 | 値 |
| --- | --- |
| 同時実行エージェント数 | min(16, CPU コア数 - 2) |
| 1 ラン当たりの総エージェント上限 | 1,000 |
| 実行中のユーザー入力 | 不可（権限承認のみ例外） |

## コストと注意点

Dynamic Workflow は多数のエージェントを起動するため、**通常のセッションより大幅にトークンを消費します**。公式ブログも「通常のコーディングタスクには不要」と明言しており、使う前に\*\*「本当にそれだけの計算資源が必要か?」\*\*と自問することを勧めています。

| 判断基準 | 使うべきか |
| --- | --- |
| 1 つのコンテキストでは途中で止まる・精度が落ちるタスク | **Yes** |
| 大規模（50 ファイル超のマイグレーション、数十件のレビュー等） | **Yes** |
| 通常のコーディング、バグ修正、小規模リファクタリング | **No** — 通常セッションか subagent で十分 |

### 実践 Tips

* **小さく試す**: 50 ファイルの移行なら、まず 1 ディレクトリだけで実行。`/workflows` で各エージェントのトークン消費がリアルタイムで見える
* **resume を活用する**: 途中で停止しても、同一セッション内なら完了済みエージェントのキャッシュが効く
* **モデルを使い分ける**: プロンプトで「探索フェーズは Sonnet、最終判定は Opus で」と指示するとコストを抑えられる
* **保存して再利用する**: うまくいった Dynamic Workflow は `/workflows` で `s` を押して保存。次回から `/<名前>` で再実行

### 制約

| 状況 | 対処 |
| --- | --- |
| 実行中にユーザー入力できない | 段階的な承認が必要なら、フェーズごとに別の Dynamic Workflow に分ける |
| セッション終了で実行が失われる | resume は同一セッション内のみ。次回はスクリプトを再実行する |
| 権限プロンプトで中断される | よく使うツールを事前に allowlist に追加しておく |

## まとめ

Dynamic Workflow は、1 つのコンテキストウィンドウで起きる 3 つの構造的問題——**怠惰**（途中で止まる）、**自己バイアス**（自分に甘い検証）、**目標逸脱**（制約の喪失）——を、計画をスクリプトに移し各エージェントに独立したコンテキストを与えることで解決します。

判断基準はシンプルです。**1 つのコンテキストでは破綻するタスクか?** ——大規模マイグレーション、網羅的なレビュー、クロスチェック付きリサーチなど、Yes なら Dynamic Workflow。通常のコーディングや小規模タスクなら不要です。まずは `/deep-research` か、小さなタスクに `ultracode` キーワードを付けて 1 つ回してみてください。
