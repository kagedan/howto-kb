# ループを壊さない .claude/ 設計、hooks、subagents、skills、MCP、memory の実践ガイド

Claude Code を使っていると、ある瞬間にこう思います。

「この作業、毎回自分が指示しなくてもいいのでは？」
「計画して、実装して、テストして、次の作業まで進めてほしい」
「Claude Code に自動でループを回させれば、もっと速く開発できるのでは？」

その感覚は正しいです。

Claude Code は、単発のコード生成ツールではありません。うまく設計すれば、目標を読み、コードを変更し、検証し、進捗を記録し、次の作業へ進む「自動化ループ」を回せます。

ただし、多くの人はここで失敗します。

1回目は動く。
2回目もそれっぽく進む。
3回目あたりで、同じ作業を繰り返す。
あるいは、テストしたふりをする。
あるいは、本来触ってはいけないファイルを編集する。
あるいは、なぜその変更をしたのか、次の実行時には忘れている。

このとき、多くの人は「プロンプトが悪かった」と考えます。

もっと強い言葉で指示する。
もっと細かく禁止事項を書く。
もっと長いプロンプトにする。
もっとモデルに考えさせる。

しかし、本当の原因はそこではありません。

問題は、ループではありません。

**ループの下にある足場ができていないこと**です。

元記事でも、Claude Code の自動化ループが止まる原因は、ループそのものではなく、その下にある .claude/、CLAUDE.md、settings.json、hooks、agents、skills、.mcp.json、memory/state file などの実行基盤が整っていないことだと説明されています。

この記事では、その実行基盤をまとめて **自動化ループ・エンジニアリング** と呼びます。

自動化ループ・エンジニアリングとは、AIエージェントに「計画 → 実装 → 検証 → 記録 → 次の判断」を継続的に回させるために、プロンプトだけでなく、権限、hooks、subagents、skills、MCP、memory、state file まで含めて設計する技術です。

要するに、Claude Code を「賢い補助輪」として使うのではなく、**壊れにくい開発ループとして運用するための設計論**です。

# 1. まず理解すべきこと：ループより先に、足場を作る

Claude Code の自動化で最初に分けるべきものがあります。

それは、**足場**と**ループ**です。

足場とは、Claude Code が毎回その上で動く固定構造です。

.claude/
  CLAUDE.md
  settings.json
  hooks/
  agents/
  skills/

.mcp.json
MEMORY.md
PROMPT.md
IMPLEMENTATION_PLAN.md
run.sh
logs/

一方、ループとは、その足場の上で実際に回る作業の流れです。

目標を読む
↓
計画する
↓
実装する
↓
検証する
↓
記録する
↓
続けるか止めるか決める

この2つを混ぜてはいけません。

足場は、キッチンです。
ループは、レシピです。

どれだけ良いレシピがあっても、キッチンに包丁も火元も作業台もなければ料理はできません。逆に、立派なキッチンがあっても、何を作るか決まっていなければ何も完成しません。

Claude Code も同じです。

ループだけ作っても、失敗します。

なぜなら、ループは毎回こう判断しなければならないからです。

何を読めばいいのか。
何を編集していいのか。
何を絶対に触ってはいけないのか。
どのコマンドは安全に実行していいのか。
実装後に何で検証するのか。
検証は誰がやるのか。
今回の進捗をどこに残すのか。
次回はどこから再開するのか。

これらが決まっていない状態でループを回すと、Claude Code は推測します。

そして、推測で動くエージェントは、いずれ壊れます。

存在しないファイルを前提にする。
実行していないテストを「通った」と言う。
禁止されていないから危険なコマンドを試す。
前回やった作業をもう一度やる。
検証していないのに完了扱いにする。

これはモデルが怠けているからではありません。

**作業場が設計されていないから**です。

だから順番は、常にこうです。

足場を作る
↓
その上にループを乗せる

ループを賢くする前に、床を張る。

これが、Claude Code 自動化の第一原則です。
