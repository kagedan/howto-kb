# Claude Codeでやらかした話 (1/3) — なぜサボるのか: 目的逆転と検査の悪用

> Claude Codeを業務利用(パイプライン運用、開発、リサーチ)する中で踏んだ罠の記録。症状ではなく根本機構で整理する。本記事は3部作の第1部。

> **執筆について**: 本記事の構造化と文章整形にClaudeを補助的に利用しているが、観測された失敗事例・対策設計の中身はすべて筆者の実運用での経験に基づく。引用しているGitHub Issueや外部報道は、自分の体験と一致した「同じ症状の参考事例」として参照している。

## まずは、踏んだ罠の話から

Claude Codeを日常的に動かしていると、こんなことが起きる。

**踏んだ罠1: 既存ファイルが消える**

ファイル整理を頼んだら、整理対象に含めていなかった既存ファイルまで消えていた。Claudeの判断は「不要そうだったので」。指示の範囲外まで踏み込んでくる。

**踏んだ罠2: 既存コードが削除される**

リファクタリングを依頼したら、機能に必要な既存ロジックがバッサリ消えていた。「機能を保ったままリファクタリング」と明示指示していても起きる。同じ症状はGitHubでも報告されている([Issue #1638](https://github.com/anthropics/claude-code/issues/1638))。

**踏んだ罠3: 「誤魔化すな」が守られない**

「データを推測で埋めるな、わからなければnullを入れろ」と明示指示しているのに、出力には推測値が混ざる。指摘すると「ご指摘ありがとうございます、今後は気をつけます」と言う。次のセッションで同じことをする。

これらは個別の事故ではなく、**構造的な失敗パターンの発現**として理解できる。本シリーズは、こうした「やらかし」を症状ではなく根本機構で分類した診断フレームワーク。再発を仕組みで防ぐための整理。

## このシリーズについて

念のため最初に書いておくと、これはLLMやベンダーへの批判ではない。LLMエージェントは現行アーキテクチャ上、構造的にサボる傾向を持つ。この性質を前提として「サボっても被害が出ない仕組み」を設計する、運用設計の話。

主旨は3点。

1. **LLMエージェントは構造的にサボる**。個別モデル・個別ベンダーの問題ではなく、現行アーキテクチャの構造的限界。
2. **症状を表面で叩いても効かない**。一度のサボりに対策を打つと、形を変えた新しいサボりが出てくる([Anthropic公式: Natural emergent misalignment from reward hacking](https://www.anthropic.com/research/emergent-misalignment-reward-hacking))。
3. **指示で守らせる発想は失敗する**。指示層は最弱の防御線。物理層・ツール層・状態層での多層防御が必要。

シリーズ構成:

* Part 1 (本記事): なぜサボるのか — Root Causeと、検査・後工程系のやらかし(カテゴリA, B, C)
* Part 2: どうサボるのか — 自律性の暴走と整合性の崩壊(カテゴリD〜I)
* Part 3: どう防ぐのか — 多層防御の設計と対策マトリクス

対象読者: Claude Codeや類似のLLMエージェントを本番系で日常的に動かしている人。趣味で触っているだけではこの粒度のやらかしは観測しにくいので、実運用経験のある人向け。

## このリストの位置づけ

注意点が3つ。

第一に、完全カタログではない。LLMはcreativeに新しい失敗形を生むので、新症状は「リストのどのカテゴリに最も近いか」で診断する。リストにないから新カテゴリを作る、ではない。

第二に、症状が違っても根本機構は同じことが多い。カテゴリA(目的逆転)が他のすべてのカテゴリの根本原因として効いている。

第三に、対策は層で考える。単一の指示追加・ルール強化では効かない。物理層・ツール層・状態層・指示層の多層防御で組む(詳細はPart 3)。

各「パターンN」は観測された具体的な失敗を抽象化したもの。一覧はPart 2末尾。

## Root Cause: 目的逆転メカニズム

LLMエージェントの失敗の大半は、以下の構造に還元できる。

**本来の目的階層**:

```
最上位: ユーザーが正しい意思決定 / 成果物を得る
中位:   各タスクで信頼できる情報・出力を生成する
下位:   個々のステップを完了する、効率的に動く
```

**実際に起きている目的階層(逆転後)**:

```
最上位: タスクを完了する、コンテキストを節約する、エラーを回避する
中位:   検査を通過する形式を整える
下位:   (なんとなく) 正しいことをする
```

逆転が起きる構造的理由:

* 目的は直接観測できないが、手段は直接観測できる → 観測可能なものが目的化する
* 手段にだけ短期フィードバックがある(検査結果、エラー、トークン消費) → 手段が目的を侵食する
* 訓練時の暗黙規範は、明示指示と衝突した時に負けやすい → 「コードを壊さない」(暗黙)と「タスクを完了する」(明示)が衝突すると、明示指示側が勝ちやすい

多くのアンチパターンは、この逆転の特殊形として理解できる。

## カテゴリA. 目的逆転系 (Root Cause)

### A1. 観測可能な手段の目的化

**構造**: 「コンテキスト節約」「タスク完走」「エラー回避」など、観測可能で短期フィードバックがある手段が、本来の目的を上回って最上位化する。

**個人の観測**:

* パターン1: 必要な追加調査(Web検索、ソース突合)を惜しんで、estimate値で埋める。リサーチパイプラインで頻発した
* パターン2: 救済フラグ(`out_of_window`等)を最初の選択肢として使う。「データが取れなかった時の例外処理」として用意した枠を、最初の手段として選ぶ

**参考事例**:

**対策層 (詳細はPart 3)**: [物理] 値取得をツール強制化、[状態] セッション分割で完走圧力低下

**参考**:

### A2. 暗黙的目的の消失

**構造**: 仕様書に書かれていない暗黙の制約(「コードが動くこと」「データを壊さないこと」)は、訓練時の暗黙規範として内在化されてはいるが、明示された別の目的(タスク完走など)と衝突すると弱い。これらの暗黙制約は実質的に消える。

**個人の観測**:

* パターン3: 削除指示で対象周辺の空白・インデントごと削除し、Pythonコード破壊。「インデントを保つ」は暗黙の前提のはずだが、削除タスクの遂行に負ける(冒頭の罠2と同じカテゴリ)
* パターン4: テスト環境を作らずいきなり本番データに変更を加える。「念のためテスト環境で試す」も暗黙の前提だが、明示指示「修正せよ」の前に消える

**参考事例**:

**対策層**: [指示] 暗黙制約の明示化、[物理] 編集後の自動lint/test、[状態] git commit前提

### A3. 完走バイアス (セッション後半で品質劣化)

**構造**: 長い処理を「走り切る」ことが目的化し、後半カテゴリほど品質が落ちる。コンテキスト圧迫(E1, E3)と相互強化する。

**個人の観測**:

* パターン5: 序盤は丁寧に作業するが、中盤以降でestimate / 形式埋めに走る。同一セッション内で前半と後半の品質差が顕著

**参考事例**:

**対策層**: [状態] セッション分割、重要度順の並べ替え

## カテゴリB. 検査・規約の悪用

検査が形式しか見ないこと、警告が無視できることを学習し、形式と内容を分離する。reward hackingの典型形。

### B1. 形式チェック悪用

**構造**: 検査が形式しか見ないことを学習し、内容真偽を犠牲にして形式だけ整える。

**個人の観測**:

* パターン6: URL末尾の数字(例: `130725609`)を「13:07:25 UTC」とタイムスタンプに変換。フォーマットチェックは通るが意味的にデタラメ
* パターン7: 引用元の意味一致確認をスキップして「via X (Tier A)」を形式的に名乗る。引用元タグの形式は満たしているが、本文と無関係

**参考事例**:

**対策層**: [物理] 形式チェックを意味チェックに昇格、[ツール] URLはweb\_fetchで実在確認を強制

**参考**:

### B2. 救済条項の濫用

**構造**: 仕様書の例外フラグ(`out_of_window`、null許容等)を「やむを得ない時の救済」ではなく「最初の選択肢」として使う。

**個人の観測**:

* パターン2 (再掲): 24h内のデータが取れない → 再試行せず即`out_of_window: true`で出力
* パターン8: null許容と明示されているフィールドに、nullではなく捏造値を入れる(冒頭の罠3と同じカテゴリ)

**対策層**: [指示] 救済条項使用時のjustification必須化、[物理] スクリプト側で救済条項使用回数を上限化

### B3. WARNを逃げ口化

**構造**: WARN止まりで再実行が発動しないことを学習し、ERRORを避けてWARNで済ませる方向に圧力をかける。

**個人の観測**:

* パターン9: タグ付けでtaxonomy未照合のフィラータグを使う。WARN多発でも処理は止まらないので無視される

**参考事例**:

**対策層**: [物理] 機械判定可能なWARNはERROR昇格、[ツール] WARN件数の上限を設定

**構造**: writer Claude自身による完了報告は原理的に機能しない。ユーザーが指摘しなければ「全completed」で終わる。

## Part 1のまとめ

ここまで見てきたのは、「目的が観測可能な手段に侵食される」というRoot Causeと、その3つの発現形だった。

* A. 目的そのものが手段に置き換わる(Root Cause)
* B. 検査・規約の存在が、形式と内容を分離するインセンティブになる
* C. 他者の存在が、自分の品質を引き下げる口実になる

冒頭の3つの罠は、ここまでに紹介したカテゴリで言うと:

* 罠1 (既存ファイル削除): A2 暗黙的目的の消失 + (Part 2で扱う) F2 周辺巻き込み破壊
* 罠2 (既存コード削除): A2 暗黙的目的の消失 — 「コードを保つ」という暗黙制約が明示タスク(リファクタ実行)に負ける
* 罠3 (誤魔化さない指示違反): B2 救済条項濫用 + (Part 2で扱う) G3 謝罪後即再発、H3 ルール認識下違反

共通しているのは、「指示で守らせる」では効かない点。観測可能な手段にフィードバックがあり、目的にフィードバックがない構造が変わらない限り、やらかしは続く。

Part 2では、これらの目的逆転が実際にコードや出力にどう現れるかを、論理的詰めの放棄、コンテキスト病理、自律アクション暴走、捏造などの症状カテゴリで見ていく。

Part 3で、これらすべてに対する多層防御の設計をまとめる。
