## はじめに

こんにちは！株式会社 Hacobu で MOVO Vista というプロダクトのフロントエンドエンジニアをしている cho です。

新しいモデルが出ると、とりあえず普段使っている Claude Code や Codex のモデルだけを切り替えてみる。まずはいつものリポジトリを読ませて、「お、前よりちゃんと読めている」「今回はあまり変わらないかも」と感触を確かめる。

おそらく、多くの人がこんなふうに新モデルを試しているのではないでしょうか。

もちろん私も、最初に知りたいのは普段の仕事で使えるかどうかです。ただ、ここで少し気になることがあります。

その結果は、本当に新モデルの評価なのでしょうか。

もしかすると、**以前のモデルに合わせて育てた設定の中で、新モデルがどれくらい動けるか**を見ているだけかもしれません。

この記事では、新モデルを試すときに一度ハーネスを外してみたほうがよい理由と、私ならどのように比較するかをまとめます。

## 最初に結論

新モデルを試すときは、次の3パターンを比べると分かりやすいです。

1. 公式ツールのデフォルト設定
2. 今まで使っていた自分の設定
3. 必要なルールだけを戻した軽い設定

モデルだけを差し替えて、いつもどおり使う。それ自体は実運用のテストとして正しいです。

ただ、その結果にはモデルの性能だけでなく、今まで使っていた設定との相性も混ざっています。意識していなくても、「新モデルが今の設定でどれくらい動けるか」も同時に試しているわけです。

モデルの変化と設定の影響を分けて見たいなら、比較対象として一度デフォルトに戻す必要があります。

## そもそもハーネスとは

Claude Code や Codex を使うとき、モデルは単体で仕事をしているわけではありません。

たとえば、次のようなものがモデルの周りにあります。

* モデルに最初から渡される指示
* `CLAUDE.md` や `AGENTS.md` に書いたプロジェクトのルール
* ファイル操作やコマンド実行のためのツール
* 計画、実装、検証を繰り返すエージェントループ
* 権限、タイムアウト、リトライ
* コンテキストの保存や圧縮

このあたりをまとめて、この記事ではハーネスと呼びます。

たとえるなら、モデルは中途入社してきた腕のいいエンジニアです。ハーネスは、その人に渡す社内ドキュメント、開発環境、権限、レビューの流れにあたります。

経験豊富な人が入ってきても、5年前の事情で作られた手順をそのまま強制されたら、たぶん仕事は遅くなります。AIエージェントでも同じことが起きます。

Anthropic も、エージェントを評価するときに見ているのは、モデル単体ではなく「モデルとハーネスの組み合わせ」だと説明しています。

## そのルール、まだ必要ですか？

AIエージェントを使い込むほど、設定ファイルにはルールが増えていきます。

```
- 作業前に必ず詳細な計画を作ること
- 実装は必ずサブエージェントに委譲すること
- 各ステップの前後に進捗を報告すること
- 失敗したら同じ手順を3回まで再実行すること
- 完了前にすべてのファイルを読み直すこと
```

こういうルールに心当たりがある方も多いと思います。

以前のモデルが計画を立てずに編集を始めた。テストを実行せずに完了した。長いタスクの途中で目的を忘れた。そのたびに「次からはこうして」とルールを足してきたはずです。

追加した時点では、どれも必要な対策だったと思います。

問題は、そのルールが新しいモデルにも必要かどうかです。

たとえば、新モデルがタスクの大きさに合わせて作業順序を判断できるようになっていても、「必ず最初に詳細な計画を作る」と書かれていれば、小さな修正にも長い計画を作ります。自分でツールを選べるのに委譲先を固定すれば、必要のないサブエージェントまで起動します。

昔のモデルを助けるための補助輪が、今のモデルには重りになっているかもしれません。

数行の指示でも影響はあります。Anthropic は、Claude Code に追加した短い出力制限が一部の評価で性能を3%下げたため、その変更を撤回した事例を公開しています。

自分で何十行、何百行と追加した指示だけは影響しない、と考えるほうが不自然です。

## デフォルトは「正解」ではなく基準点

ここで言いたいのは、カスタマイズを全部捨てよう、という話ではありません。

公式のデフォルトも一つのハーネスですし、必ずしも自分の仕事に最適とは限りません。デフォルトで試す目的は、新モデルの真の性能を測ることではなく、自分の設定の影響を見つけるための基準を作ることです。

### 私が最初に確認すること

私の場合、いきなり普段の環境で仕事を始める前に、まず次の順番で確認します。

1. どのモデルが追加され、何が変わったのか
2. デフォルトの effort が前のモデルから変わっていないか
3. effort を変更すると、速度・コスト・結果がどう変わるか

まずはデフォルトのまま使い、その後、同じタスクを `low` や `high`、対応していれば `xhigh` などに変えて試します。デフォルトだけを触って「速くなった」「賢くなった」と判断すると、モデルの差と effort の差を取り違えるかもしれないからです。

実際、[Claude Codeの公式ドキュメント](https://code.claude.com/docs/en/model-config)では、モデルによってデフォルトの effort が異なり、新しいモデルへ切り替えたときに、そのモデルのデフォルトが適用される場合があると説明されています。また、同じ `high` でも、モデルをまたいで同じ計算量を意味するわけではありません。

そのため、「新モデルをデフォルトで試した」という記録には、モデル名だけでなく、その時点の effort も残しておくと比較しやすくなります。

### システムプロンプトは見えるのか

「新モデルのシステムプロンプトは見えないから、実際に試すしかない」と考えたくなります。方向としては近いのですが、正確には少し違います。

そもそも、ここでいうシステムプロンプトはモデル自体に内蔵されたものではなく、製品やハーネスが実行時に与える指示です。

公開されているものもあります。Anthropic は[claude.ai のコアシステムプロンプト](https://platform.claude.com/docs/en/release-notes/system-prompts)を公開していますし、OpenAI も[Prompting guidance for GPT-5.6 Sol](https://developers.openai.com/api/docs/guides/prompt-guidance-gpt-5p6)を公開しており、モデルが変わると、効果的なプロンプトの書き方自体も変わることを説明しています。

公開されているのは、あくまで土台の部分です。実際に Claude Code や Codex を動かすときは、その上に製品側の指示やツール設定が重なり、自分が書いた `CLAUDE.md` や `AGENTS.md` のルールも乗ります。新モデルを「いつもの環境」で試すとき、モデルが受け取っているのはこの積み重ね全体です。

つまり、「何も分からない」わけではありません。公開されている情報から多くを知ることはできますが、製品上でモデルに効いている指示、ツール、権限、エージェントループまで含めた実際の実行環境を、利用者が常に完全に把握できるわけでもありません。

さらに、モデルが進化すれば、ハーネスやプロンプトの「ベストプラクティス」も変わる可能性があります。昔のモデルで最適だった設定が、新しいモデルでも最適とは限りません。

公開情報を確認したうえで、最後は同じタスクを直接試す必要があるのは、そのためです。

同じバグ修正を、次の3条件で試してみます。

| 条件 | ここで見たいこと |
| --- | --- |
| A. 公式デフォルト | 新モデルが何も足さなくてもどこまで動くか |
| B. 今までのハーネス | 現在の環境で実用になるか |
| C. 軽くしたハーネス | どのルールが本当に効いているか |

AではうまくいくのにBで崩れるなら、モデルより先に設定を疑えます。

BとCでほとんど差がなければ、外したルールはなくてもよかった可能性があります。Cが一番よければ、新モデルに合わせてハーネスを軽くする価値がありそうです。

1回だけでは偶然も混ざるので、同じ種類のタスクをいくつか用意して、可能なら複数回試します。大げさな評価基盤を作らなくても、これだけで見えるものはかなり増えます。

## 私ならこう試します

### 普段の仕事から3つ選ぶ

ベンチマーク用の問題ではなく、実際によく頼む仕事を使います。

* すぐに原因が分かる小さなバグ修正
* 複数ファイルにまたがる機能追加
* 原因が分からない状態から始める調査やデバッグ

簡単なタスクだけでは差が出にくく、大きすぎるタスクでは失敗の理由を追いにくいので、軽い・普通・難しいの3段階くらいがちょうどよいと思います。

### 条件をなるべく揃える

比較するときは、モデル以外の条件をなるべく変えません。

利用できるツール、推論設定、時間、トークン予算、リトライ回数。このあたりが違うと、モデルと設定のどちらが結果に効いたのか分からなくなります。

effort の影響を調べる回だけは、モデルとタスクを固定して、effort だけを変えます。一度に動かす条件を一つにすると、差の理由を追いやすくなります。

### 感想だけで終わらせない

記録するのは、次のような簡単な項目で十分です。

```
- タスクを完了できたか
- テストは通ったか
- 人間が何回介入したか
- 完了まで何分かかったか
- おおよそのコスト
- 不要な作業や手戻りはあったか
```

「なんとなく賢くなった」よりも、完了率、介入回数、時間、コストで見たほうが、次に設定を直すときにも役立ちます。

## ルールを3つに仕分ける

ハーネスを見直すときは、設定ファイルの各ルールを次のように分けます。

| 種類 | 例 | 扱い |
| --- | --- | --- |
| プロジェクトの事実 | 使用言語、設計方針、禁止API | 残す |
| 安全上の境界 | 本番変更は承認必須、秘密情報を出さない | 仕組みで守る |
| 旧モデルへの対策 | 必ず計画、必ず委譲、必ず3回リトライ | 一度外して試す |

特に見直したいのは3つ目です。

「このルールは、何が起きたから追加したのか」を思い出せないものは、一度外してもよい候補です。逆に、プロジェクト固有の知識や安全上の境界は、モデルが賢くなっても必要です。

## 薄くするのは、ハーネスの全部ではない

ここは誤解されやすいところです。

ハーネスを薄くすることと、安全性や品質をモデル任せにすることは別です。減らしたいのは、モデルの進め方を細かく決める指示です。

どのファイルから調べるか、どの順番で実装するか、失敗したら次に何を試すか。こうした判断は、新しいモデルに任せられる範囲が広がっています。

一方で、テストが通ったか、型エラーがないか、触ってはいけないファイルを変更していないか、危険なコマンドを実行していないか。ここはモデルへのお願いではなく、テスト、CI、型、リンター、権限設定、フックで守ったほうが確実です。

プロンプトに「絶対に壊さないでください」と10行書くより、壊したら失敗するテストを1つ置くほうが強い。

私が考える「薄いハーネス」は、何もないハーネスではありません。

> モデルには広く考えさせる。壊してはいけない境界は、外側の仕組みで狭く、強く守る。

そういう形です。

## モデルごとにどこまで最適化するか

モデル専用のハーネスを作れば、より高い性能を引き出せるかもしれません。ただ、モデルの更新が速いほど、その設定を追いかけるコストも増えます。

毎回すべてを書き直すのは現実的ではないので、私は変わりにくいものと、モデルに合わせて見直すものを分けて考えたいです。

プロジェクトの目的、ドメイン知識、セキュリティ、権限、テスト、完了条件は残す。計画を作らせるタイミング、サブエージェントへの委譲、リトライ、途中報告、出力の長さは、新モデルが出たら疑ってみる。

前者をハーネスの骨格にして、後者を交換できる設定として扱えば、モデルが変わるたびに全部を作り直さずに済みます。

## おわりに

新モデルをいつもの環境で試すこと自体は間違っていません。実際の仕事で使えるかを見るなら、むしろ必要です。

ただ、その1回だけで「今回のモデルは良い」「前より悪い」と決めてしまうのは、少しもったいないと思います。

モデルを評価しているつもりで、実は自分のハーネスとの互換性を見ていた。そんなことは十分にありえます。

次に新しいモデルを入れて「あれ、思ったより微妙だな」と感じたら、設定を足す前に一度だけ外してみてください。

モデルが悪いのか。それとも、昔のモデル向けに作ったルールが足を引っ張っているのか。少なくとも、その二つを分けて考えられるようになります。
