* `/code-review` がローカルで見る観点は「正しさのバグ」と「再利用・単純化・効率化の整理」に固定されていて、`effort` や `--comment` では足し引きできない。
* `effort` は指摘の広さと確信度を、`--comment` と `--fix` は出力先を変えるだけ。`ultra` でも観点は増えず、検証が深まるだけだ。
* ローカルで観点を変えるレバーは自作の `skill` 側にあり、`CLAUDE.md` や `REVIEW.md` は GitHub 連携の自動レビュー (Code Review) 側の調整つまみで、どちらもコマンドのヘルプ表面からは見えにくい。

---

私は以前、`/code-review` にコメントや引数で注文を足せば、レビューの観点が増えて堅牢になると思い込んでいました。  
同じ勘違いをしている人は、ほかにもいるはずです。  
でも、これは思い込みでした。  
ローカルの `/code-review` が見るのは「正しさのバグ」と「再利用・単純化・効率化の整理」だけです。  
それが `code.claude.com > Commands` に挙がる観点で、引数やフラグで足し引きする入口はありません。  
`effort` を上げても観点は変わらず、変わるのは指摘の広さと確信度です。  
低い `effort` は数を絞って確信度の高い指摘を返し、`high` から `max` は対象を広げて確信度の低い指摘も含みます。  
`--comment` と `--fix` も同じで、これは出力先を選ぶフラグです。  
`--comment` は指摘を PR のコメントとして投稿し、`--fix` はレビュー後に指摘を手元のファイル (作業ツリー) へ適用します。  
どちらも「何を見るか」は動かしません。

![effort や --comment が変えるものと、固定された観点の対応](https://static.zenn.studio/user-upload/2b12b7a3f5e0-20260608.png)

## 観点を変えるレバーは、コマンドの外の別ファイルと自作 skill にある

引数で観点を足せそうだと感じる理由は、LLM のコマンドは自然言語で何でも頼める、という期待がそのまま乗るからです。  
普段は文章で自由に注文できるので、`/code-review` にも「ここはセキュリティ重視で」と引数やコメントで足せるはずだと感じてしまいます。

`ultra` (`/code-review ultra`) を持ち出しても、観点は増えません。  
`ultra` はクラウド上で多数のエージェントが並列に変更を調べ、各指摘を独立に再現・検証して本物のバグに絞り込む、深さと網羅のための機能です。  
増えるのは検証の深さと網羅で、新しい観点が足されるわけではありません。

観点を変える本当のレバーは、コマンドの外側にあります。  
ここではっきりさせておきますが、`/code-review` そのものに観点を足す方法はありません。  
見る範囲は固定で、引数でもフラグでも、新しい観点を差し込む入口はありません。  
security や性能、設計といった別の観点が欲しいときは、`/code-review` を拡張しません。  
代わりに、自作の `skill` で別のレビューを用意します。  
これが観点を増やすときの正規ルートです。  
指示を書いた `SKILL.md` を置くと Claude の道具に加わり、`description` が「いつ使うか」の判断に使われます。  
これは `code.claude.com > Extend Claude with skills` に書かれています。  
具体的な作り方は別の記事に譲ります。  
ここでは「`/code-review` は拡張できず、別の観点は自作 skill が担う」という役割分担だけ押さえてください。

ここでもう一つ、よくある勘違いについて言及しておきます。  
レビューする内容や報告する内容そのものを変えたいなら `CLAUDE.md` や `REVIEW.md` を使えばいい、という思い込みです。  
たしかにこれらは「何をどの深刻度で報告するか」を変えますが、その相手はローカルの `/code-review` ではありません。  
これらは、GitHub 連携で Pull Request を自動レビューする別機能 (Code Review) 側の調整つまみです。  
とくに `REVIEW.md` はそのレビューパイプラインの全エージェントへ最優先で渡されますが、手元の `/code-review` には何も効きません。  
だから `/code-review` の引数のつもりで `CLAUDE.md` や `REVIEW.md` をいじっても、観点は変わりません。  
これは `code.claude.com > Code Review` に書いてあります。

これらのレバーは、どれもコマンドの引数やヘルプ表面ではなく、別のファイルや別のコマンドの側にあります。  
だから引数だけを眺めていても気づけず、誤解が解けないままになります。  
レビュー系コマンドそのものの使い分けは別の記事にまとめたので、ここでは「設定で観点を多角化できる」という誤解の構造だけを扱いました。
