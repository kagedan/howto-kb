# Claude Codeのルールを「育てる資産」として運用して — 効いたこと、まだ解けていないこと

こんにちは、km111です。Nishikaでアプリケーションエンジニアをしています。

私たちは SecureMemo という製品を、フロントエンド（Flutter デスクトップアプリ）とバックエンド（Python の文字起こしサービス基盤）の2つのプロジェクトで開発しています。スタックも設計思想も違いますが、Claude Code の運用については**1つの共通した考え方**で回しています。

それは、**Claude を「設計に従う実装者」として扱う**という考え方です。

設計判断やレビューは人間が握る。一方で、「決められた設計に沿ってコードを書く」「機械的に検証できる品質を担保する」といった作業は、ルール・スキル・フックという形で Claude に委譲する。そして重要なのは、これらを**一度書いて終わりにせず、資産として育てていく**ことです。

この記事では、

1. ルール／スキル／フックで何をどう委譲しているか（実物つき）
2. それらを「育てる資産」として運用する考え方
3. それでもまだ解けていない問題

の順で書きます。

## 1. Claude が読むルールの階層

Claude Code はセッション開始時に設定ファイルを自動で読み込みます。私たちは役割を3つの階層に分けています。

| ファイル | 役割 |
| --- | --- |
| `CLAUDE.md` | プロジェクト全体の指針・コマンド・構造・**禁止事項** |
| `.claude/rules/*.md` | レイヤー／コンテキストごとの詳細規約・レビュー観点 |
| `CLAUDE.local.md` | 個人ローカル設定（日本語返信、プラン保存先など） |

`CLAUDE.md` には「やってほしいこと」だけでなく、**「やってはいけないこと」を明示的に書く**のが効きます。たとえば FE の `CLAUDE.md` の禁止事項はこうです。

```
## 禁止事項

- `print()` を本番コードで使わない — `logger` ユーティリティを使用する
- `dynamic` 型を避ける — `strict-casts: true` が有効
- `as` によるキャストを安易に使わない — パターンマッチまたは型チェックを優先する
- 未使用の import / 変数を残さない
- `setState()` をできる限り使わない — Riverpod + Hooks で状態管理する
```

「こう書け」だけだと裁量の余地が残りますが、「これをやるな」を添えると、生成物のブレが目に見えて減ります。

同じことは BE（Python）側でもやっています。スタックは違っても「禁止を明示する」思想は共通で、中身だけがそのスタック固有になります。

```
## 禁止事項（BE / 抜粋）

- ビジネスロジックでの生SQL `text()` を書かない — ORM + repository を使う
- 素の `HTTPException` や `Exception` を投げない — 共通例外は `platform/http/errors.py`
- context 境界違反をしない（import-linter が `uv run lint-imports` で強制）
- コメント・docstring の括弧は半角 `( )` を使う — ruff の RUF002/003 が全角括弧を弾く
```

最後の「全角括弧は半角で」あたりは些末に見えますが、日本語話者が書くと全角丸括弧が紛れ込みがちで、実際にレビューで何度も指摘していたものです。機械（ruff）が弾くルールを言葉でも明示しておくと、そもそも生成段階で混入しなくなります。

なお、禁止事項は `CLAUDE.md` だけに書くものではありません。実際には `.claude/rules/` の各層ルールにも「禁止パターン」を明記しています。たとえば 例外処理のルール（`error-handling.md`）でも、「やってはいけない書き方」を並べています。

```
// NG: throw e（スタックトレースが失われる → rethrow を使う）
} on Exception catch (e) {
  throw e;
}

// NG: 型指定なしの catch（on Exception catch / on 具体型 catch を使う）
try {
  await repository.fetchItems();
} catch (e) {}

// NG: finally 内で例外を throw する（元の例外が消える）
```

「これをやるな」を、CLAUDE.md（全体）とレイヤールール（局所）の両方に、その層で実際に踏んだ地雷とセットで書く——これが生成物のブレを減らす一番効いた運用です。

### 層ごとに、適用範囲を絞ったルールを置く

`CLAUDE.md` は「できれば200行以内」を目安にしているので、レイヤーごとの細かい規約は `.claude/rules/` に逃がします。FE では現在13本のルールファイルを運用しています。

```
.claude/rules/
├── data-repositories.md         # Repository 層の責務と禁止事項
├── data-services.md             # Service 層（API/Storage/WebSocket）
├── domain-models.md             # Freezed モデル定義
├── ui-core.md / ui-widgets.md / ui-view-models.md  # UI 各層
├── error-handling.md / testing.md / enums.md ...
```

ポイントは、各ルールファイルの先頭に **「どのディレクトリのコードに適用されるか」** を Front Matter で宣言していることです。たとえば ViewModel 層のルールはこうなっています。

```
---
paths:
  - "lib/ui/**/view_models/**/*.dart"
---
```

これで、`lib/ui/.../view_models/` 配下を触るときだけ ViewModel のルールが効く、という形になります。ルールが多くても、関係ない層の規約が混ざってノイズになることを防げます。

ルールの中身は、単なる規約の羅列ではなく**判断フロー**として書くと Claude が従いやすくなります。ViewModel ルールから、Controller のロジックを Helper と UseCase のどちらに切り出すかの判断フローを抜粋します。

```
## Helper vs UseCase の使い分け

Controller からロジックを切り出したい
  ├─ ドメインビジネスロジック？（発話の削除/結合、話者更新 等）
  │   → UseCase（lib/domain/usecases/）
  ├─ Repository / Service アクセスが必要？
  │   → UseCase（Ref 依存 UseCase）
  ├─ 他の Controller からも使う？
  │   → UseCase（ドメイン層に置いて共有）
  ├─ Stateful な UI 制御？（ScrollController, Timer 等）
  │   → Helper（Stateful Helper）
  └─ 特定 Controller だけで使う純粋計算/変換？
      → Helper（Static Helper）
```

これは本来、レビューのたびに人間が口頭で説明していた「配置の勘どころ」です。それをルールに昇格させると、以後は Claude が最初からこの分岐に沿って実装してくれます。

## 2. 機械に委ねる二重のガードレール

ルールは「お願い」であって、Claude が必ず守るとは限りません。だから、**機械的に検証できる品質は、人間が見ずに自動で止める**仕組みを二段構えで置いています。

### 編集ごとに走るフック

FE では、Claude がファイルを編集するたびに `dart format` を自動実行します（`.claude/settings.json`）。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "bash .claude/hooks/dart-format.sh" }
        ]
      }
    ]
  }
}
```

フック本体は、自動生成ファイル（`.freezed.dart` / `.g.dart`）を除外したうえで整形だけ行うシンプルなものです。

```
# .claude/hooks/dart-format.sh（抜粋）
[[ "$FILE_NORM" != *.dart ]] && exit 0
[[ "$FILE_NORM" == *.freezed.dart ]] && exit 0
[[ "$FILE_NORM" == *.g.dart ]] && exit 0

dart format -l 120 "$FILE_NORM"
```

### コミット前に走るゲート

`git commit` の直前には、変更された `.dart` ファイルだけに `dart analyze` をかけるフックを挟んでいます。

```
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      {
        "type": "command",
        "command": "bash .claude/hooks/analyze-changed.sh",
        "if": "Bash(git commit*)"
      }
    ]
  }
]
```

BE 側も同じ思想で、`pre-commit` に `ruff format` / `ruff check` / `ty`（型チェック）の3点を入れて、フォーマット・静的解析・型を機械でゲートしています。

こうしておくと、完了報告が「たぶん通っているはず」ではなく **「ゲートの出力が客観的な証拠」** になります。整形・静的解析・型のような主観の入らない品質は、人間がレビューで見る必要がなくなります。

### 壊すと取り返しがつかないものは、チェックリストで縛る

ガードレールが特に効くのは、**ミスると不可逆な被害が出る**領域です。FE はデスクトップアプリなので、DB スキーマのマイグレーションを誤ると**過去バージョンのユーザーの DB が壊れます**。ここは専用ルール（`data-database-migration.md`）で、アンチパターンとチェックリストを徹底的に縛っています。

```
## 絶対に守るルール（アンチパターン）

1. `schemaVersion` を上げずにテーブル定義を変更しない
   → 既存ユーザーで onUpgrade が走らず起動時に SQL エラーになる
2. onUpgrade の既存 `if (from < N)` ブロックを編集しない
   → 新しい if を「追加」する
3. `else if` チェーンで書かない（必ず if の連鎖）
   → 複数バージョンを跨ぐ起動でステップが抜けるのを防ぐ
6. onUpgrade 内で dropAll + createAll でリセットしない
   → ユーザーデータが全消滅する
```

```
## スキーマ変更時の必須チェックリスト

- [ ] schemaVersion を +1（飛ばさない）
- [ ] onUpgrade に if (from < N) ブロックを追加
- [ ] drift schema dump で drift_schema_v{N}.json を生成・コミット
- [ ] migration_v{N-1}_to_v{N}_test.dart を追加
```

加えて、BE ではマイグレーションファイルの手書きを禁止し、`alembic revision --autogenerate` で生成させる運用にしています。**人間が設計の是非を判断し、生成は機械に任せ、生成物を改変させない** ——「破壊的・外部公開・不可逆な操作は Claude に握らせない」という線引きです。

## 3. ルール・スキル・フックを「育てる資産」として運用する

ここからが本題です。

AI への指示には2種類あります。**使い捨ての指示**（その場のチャットで完結し、履歴に消える）と、**資産化された指示**（`.claude/` に書かれ、全員・毎回効く）。

同じ指摘を毎回チャットで言うのは、**毎回コストを払っている**ことに他なりません。一度ルール化すれば、以後はゼロコストで全員に強制されます。Claude への指示は「複利で効く資産」になり得る。資産に積むか、毎回支払うか —— ここが分かれ道です。

私たちは、3つの資産の使い分けの目安をこう置いています（これは Claude 自身が言語化してくれたものです）。

> **2回同じ指摘をしたらルール。機械で測れるならフック。3回やった手順はスキル。**

* **ルール**（`rules/`）に積むもの：設計判断・規約・禁止事項
* **フック**に積むもの：客観的に測れる品質ゲート
* **スキル**に積むもの：繰り返す作業手順

### スキルの例：レビュー観点を畳む

レビューはまさに「毎回同じことをやる作業」なので、スキルに落としています。FE では `/rv-full` という統合レビュースキルが、5観点（Lint / ルール準拠 / パフォーマンス / セキュリティ / UIデザイン）を**並列のサブエージェント**で実行します。

```
---
name: rv-full
description: Lint・ルール準拠(3回統合)・パフォーマンス・セキュリティ・UIデザインの
  5観点レビューを並列実行し、結果を統合する。
disable-model-invocation: true
---
```

注目してほしいのは、**ルール準拠レビューを「3回」独立で回している**ことです（`rv-rules-x3`）。LLM のレビューにはガチャ要素があり、1回では見落とします。そこで3回走らせ、何回検出されたかを信頼度として扱います。

```
### 指摘の信頼度サマリー
| 指摘内容 | レビュアー1 | レビュアー2 | レビュアー3 | 検出数 |
|----------|:---------:|:---------:|:---------:|:-----:|
| {指摘の要約} |    ○     |    ○     |    -     |  2/3  |

3人中2人以上が指摘 → 重要度を1段階上げることを検討
（Suggestion → Should Fix、Should Fix → Must Fix）
```

AI が書くコードは量が多く、人間が全部は見切れません。だからレビューも AI にやらせる。すると「結局 AI 同士がやり取りしているだけでは？」という感覚に陥るのですが——これについては後述します。

### フライホイール：気づいたら、その場で還元する

運用の肝は、**レビューで一度した指摘を、その場で `.claude/` に還元する**ことです。「なぜこの指摘が出たのか」を考え、ルールとして書き戻す。これを続けると、ルールを整備する前と比べて、**指摘項目が明らかに減り、最初から意図した形でプルリクが上がってくる**ようになりました（感覚値ですが、確かに効いています）。

チーム運用としては、レビュー時に「毎回同じ指摘があるな」と気づいた人が起点になります。設計担当が発信してルールを更新する場合もあれば、メンバーから「これもルールに入れたほうがいいのでは」とプルリクをもらう場合もある。各自のローカルルールで先に試して、良ければチームのルールに昇格させる、という流れも自然に回っています。

`.claude/` 配下は全員に配布しているので、一度ルール化すればチーム全体に効きます。**効果の指標は「人が同じ説明を繰り返していないか」**。同じ指摘をチャットで繰り返していたら、それは資産化のサインを見逃しているということです。

## 4. それでも、まだ解けていない問題

ここまでは「効いたこと」です。でも、正直に書くと、**うまくいっていないこと・まだ答えが出ていないこと**もたくさんあります。むしろこちらのほうが本質的だと思っています。

### (1) ルールが本当に効いているか、検証しきれていない

あるルールが有効かどうかを厳密に確かめるには、本来は **そのルールを消した状態で同じプロンプトを流し、失敗するかを見る**——アブレーション（条件を変えた比較）が必要です。でも、ルール1本ごとにそれをやるのは現実的に面倒で、**正直やれていません**。「このルール、たぶん効いてるよね」という感覚で運用している部分が、確実に残っています。

「ルールを書きすぎなのでは」という疑いは常にあるのに、削る判断の客観的な根拠を作れていない。ここは未解決です。

### (2) モデルが変わったときのルール棚卸しが、堂々巡りになる

LLM は賢くなるので、古いモデル向けに書いた過剰なルールは削りたい。実際、モデルが上がったタイミングで削減を試みました。やり方は「新旧のモデルにお互いのルールをレビューさせて、削れるものを出させる」。

……お気づきの通り、これは **モデル自身にモデル向けのルールの要否を判断させている**わけで、(1) の検証問題を回避できていません。結局「効いていそうなものを、それっぽく残す」に落ち着きがちです。

### (3) AI 同士のレビューの危うさ

AI が書いたコードを AI がレビューする。これは省力化として効いている一方で、「人間の目が一度も入らないループ」を作りかねません。私たちは今のところ、**変更ファイルが多いもの・仕様が複雑なものは人間が必ず動作確認する**という線で運用していますが、「どこまで AI 同士に任せ、どこから人が見るか」の最適点は、まだ手探りです。

### (4) 強すぎるルールは、回避が常態化する

最初に作ったルールが強制力過剰で、正当な作業まで止めてしまうことがあります。すると現場は回避策を常用し始め、ルールが形骸化する。**ルールは書くだけでなく、消す・直すまで含めて運用しないと負債になる** ——これはアンチパターンとして意識しているものの、棚卸しを定期実行できているとは言い難いのが実情です。

## まとめ

私たちの運用を一言でまとめると、**「Claude は設計に従う実装者、人間は設計判断とレビューに集中する」** です。

* ルールは明文化し、機械で測れるものはフック／pre-commit で強制する
* 配置・テスト・マイグレーション生成は規約に委譲する
* 破壊的・不可逆・外部仕様に関わる操作は人間が握る
* 気づきはその場で `.claude/` に還元し、複利で効く資産として育てる

そして同じくらい大事なのは、**この仕組み自体がまだ完成していないと認めること**です。ルールの有効性検証、モデル移行時の棚卸し、AI 同士レビューの境界——どれも答えは出ていません。

「使うほど強くなる仕組みを、チームで積み上げる」。その積み上げ方そのものを、まだ試行錯誤しています。こういう「正解のない運用設計」を一緒に面白がれる人とは、きっと良い仕事ができると思っています。
