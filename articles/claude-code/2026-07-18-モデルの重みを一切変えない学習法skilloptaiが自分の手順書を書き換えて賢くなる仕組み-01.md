「AIを賢くする」と聞くと、多くの人が**大量のGPUを回して、モデルの中の数字（重み）を微調整する**光景を思い浮かべるのではないでしょうか。

でも、もし「モデルの中身には一切触れず、AIに渡す"指示書"を書き換えるだけで性能を上げる」方法があるとしたら、どうでしょう？

Microsoftが公開した **SkillOpt** は、まさにそれをやってのけるフレームワークです。しかも面白いのは、その仕組みが**従来の深層学習（ディープラーニング）とそっくり同じ構造**をしていること。ただし、動かしているのは「数学」ではなく「言語」です。

この記事では、SkillOptの仕組みを、深層学習でおなじみの用語（重み・勾配・学習率…）と一つずつ対比させながら、やさしく解き明かしていきます。約8分で読めます。

---

## なぜ今「重みを変えない最適化」なのか

これまでAIエージェントに持たせる「スキル（手順書・プロンプト）」は、人間が手作業で書くか、AIに一度だけ生成させて終わり、というのが普通でした。

料理でたとえるなら、**レシピを一度書いたら、あとは味見もせず使い続けている**ような状態です。何度作っても改善されません。

一方、深層学習の世界には「作る→採点する→少し直す」を延々と繰り返して精度を上げる、という規律あるループがあります。SkillOptがやったのは、この**深層学習の"回し方"を、テキスト（手順書）の世界にそっくり持ち込むこと**でした。

しかもモデルの重みには触りません。SkillOptは対象を「凍結された（frozen）LLM」と表現します。つまり中身は固定されたまま。書き換えるのは、あくまでモデルに読ませる**外側の指示書**だけ。だからGPUで巨大な再学習をする必要がありません。

この「近年注目されているテキスト最適化（Textual Optimization）」というアプローチが、具体的にどう深層学習と対応するのか。ここからが本題です。

## 基礎構成要素：モデル重み ⇄ スキル文書

まず、最適化の**主役**が入れ替わります。

深層学習で書き換えていくのは「重み（Weight）」──ニューラルネットワークのノード同士をつなぐ、無数の数値パラメータです。この数字が変わることで、モデルの振る舞いが変わります。

SkillOptで書き換えていくのは「**スキル文書（Skill document）**」──LLMに渡すテキストベースの手順書（プロンプト）です。

深層学習 → SkillOpt   
モデルの重み（数値） → スキル文書（テキスト）   
 `.pth` などのバイナリ → 人間も読めるMarkdown   
 微分・数値計算 → 言語での書き換え 

ひとことでまとめると、**「調整するダイヤルが、数字のツマミから"日本語（自然言語）の文章"に変わった」**ということです。モデルそのものをいじるのではなく、モデルに読ませるカンペを賢く推敲していく、と考えるとしっくりきます。

## 学習ループを翻訳する：SkillOptの6ステップ

ここが記事の心臓部です。深層学習の学習ループは、ざっくり「予測する → 誤差を測る → 直す方向を求めて更新する」の繰り返し。SkillOptはこれを、より丁寧に **6つのステージ**へ分解して回します（各エポックごとに繰り返されます）。

一つずつ、深層学習の言葉に翻訳しながら見ていきましょう。

### ① Rollout（実行）⇄ Forward Pass

深層学習の **Forward Pass** は、入力をネットワークに通して予測を得るプロセス。

SkillOptの **Rollout** では、今のスキル文書を持たせたエージェントに実際のタスクをやり切らせ、その**実行の軌跡（trajectory）と点数**を集めます。\*\*「今のカンペで本番を何回かやってみて、記録を残す」\*\*イメージです。

### ② Reflect（内省）⇄ 勾配のもとになる誤差分析

集めた軌跡を、**別の"最適化役"のLLM**が分析します。どこでつまずいたのかを特定し、「ここをこう直すべき」という\*\*編集パッチ（edit patch）\*\*を提案するステップです。

深層学習が数式で「誤差の原因」をたどるのに対し、SkillOptは**LLMの読解力で失敗を振り返る**。ここが最大の発想転換です。

### ③ Aggregate（統合）⇄ バッチ勾配の平均

複数のRolloutから出てきた編集パッチを、**ひとつにまとめます**。重複や似た指摘を統合し、矛盾を整理する工程です。

深層学習で「バッチ内の勾配を平均して安定させる」のと同じ役割。**個々の失敗に振り回されず、全体傾向として直すべき方向**を導きます。

### ④ Select（選定）⇄ 勾配クリッピング

まとめた編集案を**ランク付けし、上位だけに絞り込みます**。この絞り込みの上限を決めるのが、後述する **`learning\_rate`（1エポックあたりの最大編集数）**。

これは深層学習の **勾配クリッピング**にあたります。**一度に大量に書き換えて手順書を破壊しないためのブレーキ**です。

### ⑤ Update（更新）⇄ SGDステップ

選ばれた編集を、実際にスキル文書へ **追加・削除・置換（add / delete / replace）** の形で適用します。

これはまさに、重みを一歩動かす **SGD（確率的勾配降下）のステップ**に相当。ただし動くのは数値ではなく**文章**です。

### ⑥ Gate（検証ゲート）⇄ Validationによる採用判定

最後に、更新した文書を**学習に使っていない検証用データ（held-out split）**で評価します。**スコアが上がったときだけ、その編集を正式採用**する。逆に悪化すれば却下します。

これが深層学習でいう **Validation** の役割で、特定タスクへの過学習でAI全体が劣化するのを防ぐ「門番」です。

> **ポイント**  
> ①→⑥をエポック単位で回し続ける、という骨格が深層学習と完全に一致しています。違うのは、勾配を「微分」ではなく「LLMの内省」で作っている点だけなのです。

## 暴走を防ぐ制御機構：学習率・スケジューラ・Optimizer状態

「回すだけ」だと手順書は暴走します。書き換えすぎて壊れたり、特定タスクに過剰適応したり。深層学習が積み上げてきた**安定化の工夫**も、SkillOptはしっかり翻訳しています。

* **学習率（Learning Rate）⇄ 1エポックの最大編集数**  
  深層学習の学習率が「一回の更新でどれだけ大胆に動かすか」であるように、SkillOptの `learning\_rate` は **「1エポックで手順書を何箇所まで書き換えてよいか」**の上限。大きすぎれば発散、小さすぎれば進まない、という性質までそっくりです。
* **学習率スケジューラ（lr\_scheduler）⇄ 同名の仕組み**  
  なんと **`cosine` / `linear` / `constant`** といったスケジューラまで用意されています。学習の序盤は大きく、終盤は小さく編集量を減らしていく、といった調整ができ、深層学習の作法がそのまま持ち込まれています。
* **Optimizer状態 ⇄ 却下バッファ / 緩やかなメタ更新**  
  AdamなどのOptimizerが過去の勾配を覚えて滑らかに更新するように、SkillOptも **「過去にこの修正をしたら失敗した」という履歴（Rejected-edit Buffer）** を記憶。さらにエポックの境目で **slow update / meta skill** という緩やかな記憶更新を行い、急激な変更を避けながら少しずつ洗練させます。
* **エポック・バッチ ⇄ そのままエポック・バッチ**  
  `epochs` や `batch size` という概念もそのまま採用。\*\*「深層学習の設定ファイルを、テキスト最適化用に読み替えた」\*\*と言えるほど、対応関係が徹底されています。

## チェックポイント = `best\_skill.md`

深層学習では、学習の途中で最も成績が良かった瞬間の重みを「ベストチェックポイント」として `.pth` や `.ckpt` に保存します。

SkillOptにおけるチェックポイントが、最終成果物である **`best\_skill.md`** です。検証タスクで最高性能を発揮したスキル文書がここに保存されます。

ここが従来と決定的に違うところ。中身は**バイナリではなく、人間もAIもいつでも読めるMarkdownテキスト**です。SkillOptによれば、そのサイズはおおよそ **300〜2,000トークン**程度のコンパクトな文書で、デプロイ時の推論オーバーヘッドはゼロ（普段のプロンプトに差し込むだけ）。「学習結果」が、目で読んで中身を確認できる一枚のカンペになっている、というのは新鮮な感覚ではないでしょうか。

## 軽く触れる：実際に試すには＆どこまで来ているか

SkillOptはMicrosoftがOSSとして公開しており、導入はシンプルです。

```
pip install skillopt
```

モニタリング用のダッシュボード（Gradio製WebUI）も用意されています。

```
pip install -e ".[webui]"
python -m skillopt_webui.app
```

対応バックエンドは **OpenAI / Azure / Claude / Qwen / MiniMax / OpenAI互換 / Codex CLI / Claude Code CLI** と幅広く、独自のバックエンドやベンチマークも追加できる拡張構造になっています。

公開されている評価では、**6つのベンチマーク × 7モデル × 3実行環境＝52通りすべてで最高または同率トップ**。GPT-5.5では、直接チャットで **+23.5ポイント**、Codexエージェント内で **+24.8ポイント**、Claude Code内で **+19.1ポイント**の精度向上が報告されています。

さらに v0.2.0 では **SkillOpt-Sleep** という機能が加わりました。これは**夜間にオフラインで自己進化する仕組み**（コーディング作業の履歴を「収穫→採掘→再生→統合」し、検証ゲートを通して定着させる）で、人間のレビューを挟んでスキルやメモリを更新していく発想です。詳しい使い方や最新仕様は、必ず[公式リポジトリ](https://github.com/microsoft/SkillOpt)のドキュメントを確認してください。

## まとめ：数学の最適化を"言語の最適化"へ翻訳したフレームワーク

今回は、Microsoftの **SkillOpt** を深層学習の用語と対比しながら解説しました。

ポイントをおさらいすると：

* **最適化の対象**が「モデルの重み（数値）」から「スキル文書（テキスト）」に変わった
* **学習ループ**は Rollout → Reflect → Aggregate → Select → Update → Gate の6段階で、深層学習の「実行→勾配→クリッピング→更新→検証」に一対一で対応する
* **学習率・スケジューラ・Optimizer状態・エポック**まで、安定化の工夫がテキスト版として移植されている
* 成果物は人間も読める `best\_skill.md`──\*\*"学習結果"が読めるカンペになった\*\*

SkillOptがすごいのは、深層学習が「微分と数学」で回していた最適化ループを、**LLMの推論力を使って「言語と分析」に丸ごと翻訳**してみせたこと。GPUで重みを更新する代わりに、AI自身が失敗を振り返り、手順書を推敲していくのです。

「モデルを鍛える」時代から、「モデルに渡す言葉を鍛える」時代へ。この発想の転換は、これからのAI活用の景色を静かに変えていくかもしれません。

まずは公式リポジトリを眺めて、`best\_skill.md` が実際にどんなテキストになるのか、覗いてみることから始めてみてください。
