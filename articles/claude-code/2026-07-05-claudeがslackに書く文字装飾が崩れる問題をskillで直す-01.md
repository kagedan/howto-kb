Claude CodeでSlackの下書きを作るようになってから、`**重要**` と書いたつもりの太字が、Slackの画面ではアスタリスクが見えたまま斜体になってしまう、という崩れに悩まされていました。

原因は次の二つでした。

1. SlackがMarkdownではなく **mrkdwn** という独自記法を使っていること
2. 装飾の描画に **空白の境界** を必要とすること

この記事では崩れの仕組みを順に整理し、最後に正規化スクリプトをSkill化して自動で直す方法を紹介します。

## なぜ崩れるのか

### 1. SlackはMarkdownではなくmrkdwn

そもそもSlackのテキストはmrkdwnという独自記法です。Markdownに着想を得てはいますが、ルールは別物で、装飾は **一重の記号** で書きます。

| 記法 | 意味 |
| --- | --- |
| `_italic_` | イタリック |
| `*bold*` | 太字 |
| `~strike~` | 取り消し線 |

基本の装飾はこの3つだけで、**下線はありません**。mrkdwnに下線の記法はなく、Slackの入力ツールバーにも下線は出てきません。

### 2. MCPが標準Markdownをmrkdwnに変換する

ところがClaude CodeのSlack MCP（`slack_send_message_draft` と `slack_send_message`）は、渡した本文をいったん **標準のMarkdown** として受け取り、内部でmrkdwnへ変換してから送ります。ここでずれが生じます。

| 入力（標準Markdown） | 変換後（mrkdwn） | Slackでの表示 |
| --- | --- | --- |
| `**x**` | `*x*` | 太字 |
| `*x*` | `_x_` | イタリック |
| `~~x~~` | `~x~` | 取り消し線 |
| `` `x` `` | `` `x` `` | コード |

書いたときの見た目と、実際にSlackへ届く記号が食い違っています。`**x**` は太字のルートに乗りますが、一重の `*x*` はイタリック扱いに化けるため、太字のつもりで書いた語が斜体になります。

### 3. 描画には「空白の境界」が必要

化けるだけならまだ直しようがありますが、やっかいな点がもう一つあります。

変換後の `*x*` や `_x_`、`~x~` をSlackが装飾として描くのは、**開き記号の直前と閉じ記号の直後がどちらも空白か行の境界のときだけ** という条件があります。

英語なら単語の前後に自然と半角スペースが入るので滅多に踏みませんが、日本語は句読点や助詞が記号にぴったりくっつきます。

### 4. 日本語では境界が埋まって崩れる

全角文字が記号に触れると、装飾が無効になります。

| 書いたもの | 送信後 | 結果 |
| --- | --- | --- |
| `・**太字**になる` | `・*太字*に` | 記号の外側に全角が密着 → 描画されない |
| `そこで、**重要**な点` | `、*重要*な` | `、` と `な` が密着 → 太字にならない |
| `取り消しは~~これ~~と` | `は~これ~と` | `は` と `と` が密着 → 線が引かれない |

これは太字もイタリックも取り消し線も同じでした。日本語特有の詰まり方が、変換のずれと重なって崩れを増やしていたわけです。

### 5. インラインコードだけは安全

唯一安定していたのがインラインコードです。バッククォートで囲んだ中身は **他の装飾を一切受け付けない** と、Slackのドキュメントにも明記されています。ですから `user_id` のような識別子はバッククォートで守れば崩れません。

## 対策の方針

やることは2つです。

1. 太字・イタリック・取り消し線を、それぞれ **一つの記法に揃える**
2. 各スパンの外側が非空白なら **半角スペースを差し込む**

記法の統一は次のように寄せます。

* **太字**: `**x**` に寄せる。表記ゆれの `__x__` もここへ統一
* **イタリック**: `_x_` のまま残し、一重アスタリスクの `*x*` だけ `_x_` に揃える
* **取り消し線**: 一重チルダ `~x~` を `~~x~~` に直す

崩れの原因は装飾の種類ではなく **境界の空白が無いこと** なので、種類は変えずに境界だけ作ります。コードブロックとインラインコードの中身は触りません。

## 実装

毎回手で直すのは面倒なので、この正規化をスクリプトにしてSkillにしました。下書きを作るときにSkillが発火し、本文をスクリプトに通してから `message` に渡します。

### 記号を揃える

記号を揃える処理の中心はこの2行です。

```
# *x* -> _x_（一重アスタリスクのイタリックを _ に統一。** には触らない）
text = re.sub(r"(?<!\*)\*(?!\*)([^*\n]+?)\*(?!\*)", r"_\1_", text)
# ~x~ -> ~~x~~（一重チルダを取り消し線へ。既存の ~~ は触らない）
text = re.sub(r"(?<!~)~(?!~)([^~\n]+?)~(?!~)", r"~~\1~~", text)
```

### 境界スペースを挿入する

太字・イタリック・取り消し線のスパンを拾い、前後が非空白なら半角スペースを足します。イタリックだけは `user_id` のような識別子を巻き込まないよう、両端が識別子文字のときは触りません。

```
def _pad(text):
    pat = re.compile(r"\*\*[^*\n]+?\*\*|~~[^~\n]+?~~|_[^_\n]+?_")
    def repl(m):
        s, e = m.span()
        tok = m.group(0)
        before = text[s - 1] if s > 0 else ""
        after  = text[e]     if e < len(text) else ""
        if tok[0] == "_" and before and after and IDENT.match(before) and IDENT.match(after):
            return tok
        left  = " " if (before and not before.isspace()) else ""
        right = " " if (after  and not after.isspace())  else ""
        return left + tok + right
    return pat.sub(repl, text)
```

`IDENT` は識別子文字（英数字など）にマッチする正規表現です。`snake_case` や `user_id` のような識別子は、アンダースコア対の外側が両方とも英数字なら強調とみなさず素通しします。コードブロックとインラインコードはいったん退避して、変換が終わってから元へ戻します。`like_this` がうっかり装飾になる事故はこれで防げます。

## 動作確認

手元で通した結果がこちらです。

| 入力 | 出力 | 説明 |
| --- | --- | --- |
| `*ポイント*` | `_ポイント_` | イタリックは残す |
| `これは_強調_です` | `これは _強調_ です` | 前後に半角スペースを挿入 |
| `__二重__` | `**二重**` | 二重アンダースコアは太字 |
| `・**太字**になる` | `・ **太字** になる` | 境界に半角スペース |
| `そこで、**重要**な点` | `そこで、 **重要** な点` | 境界に半角スペース |
| `取り消しは~これ~と` | `取り消しは ~~これ~~ と` | 一重チルダを取り消し線へ |
| `user_id と auth_id` | `user_id と auth_id` | 識別子は維持 |
| `` `a_b_c` はそのまま `` | `` `a_b_c` はそのまま `` | インラインコード保護 |
| `10~20件` | `10~20件` | 対になっていないチルダは放置 |

狙いどおり、イタリックはイタリックのまま残り、全角に密着していた箇所には半角スペースが入り、識別子とコードは無傷で残りました。

## 制限事項

下記の制限があります

* 境界用の半角スペースは外側が非空白なら一律で入るので、丸括弧やコロンの前後でも挿入され、`( **x** )` のような並びになることがあります。日本語のチャットなら気になりませんが、整えたいなら出力を目で直す必要があります。
* 平文中の `__main__` が太字化されることもあります。壊したくないコード片はバッククォートで囲っておくのが無難です。
