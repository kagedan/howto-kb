## AIの使い分け

具体的な話に入る前に、僕がいま使っているツールをざっくり紹介します。

* **仕様策定**：Claude
* **アタリ画像の生成**：Codex＋GPT Image
* **実装**：Claude Code

ここは完全に好みなので、自分が使いやすいツールで問題ないです。

全体のフローはこんな感じ。

1. 従来通りラフを書いたり、リサーチでイメージを固める
2. Claudeで仕様書をつくる
3. 仕様書をもとに、必要に応じてCodexでアタリ画像を一式生成  
   （細かいパーツはまだまだ自作したほうが早いこともある）
4. Claude Codeでざっくり実装
5. 時間がゆるす限りブラッシュアップ

それぞれの担当者を分けて、仕事を割り振るイメージ。  
まさしく**プチ会社**ですね。

このフローのなかで、「気をつけていること」を書いていきます。

## 1. 実装に入る前に「仕様書」をClaudeで固めておこう

Claude Codeはおちゃめさん。  
ちょっと話しかけただけで、すぐに実装へ走り出します。

話をよく聞かずに走り出すのは子供みたいで可愛いですが、だいたい遠回りになりがちです。  
**地図がない旅は、かならず迷います。**

あらかじめ、全体のトンマナはもちろん、セクションごとに、構成やレイアウトは考えておくのがオススメです。

僕は、別のチャット（Claude Codeではなく、通常のClaude）で、先に仕様書を整理するようにしています。

「このセクションはこんなレイアウトにして、この背景にして、画像はこのように入れる。出現時アニメーションは…」みたいな実装方針を、マークダウンで作る。それからClaude Codeに渡す。

**司令塔と実行役を分けるイメージ**です。

この一手間を入れるだけで、Claude Codeに渡したあとの迷走がだいぶ減ります。司令塔のClaudeと相談しながら方針を固めているうちに、自分の頭のなかも一緒に整理されていくので、二重で効いてきます。

時間がないときは、AIにおまかせして後から部分修正してもいいですが、あまり指示を出さずにつくった初校案は、**「カードをこれでもかと乱用したデザイン」がめちゃくちゃ出やすいです。**

「どんだけカード使うねん…」と、嫌になるくらい使ってきます。  
おそらく、レスポンシブでも破綻しづらいレイアウトを積極的に選んできてるのかな。

そもそも**AIにあまり期待しすぎないスタンス**がちょうどいいかもしれません。

![](https://assets.st-note.com/img/1779279614-JOb8evrqhEsLVUuBY7XpdzcQ.png?width=1200)

なにごとも段どりが大事。  
ちなみに、仕様書を作るときは Opus 4.7 を使っています。

## 2. セクションごとにプランモードで実装しよう

Claude Codeはせっかちさん。  
すぐ手を動かしたがります。  
そして、事故ります。

あいつら、**勝手に指示していないところを消したり**もします。  
（そして無限修正へ…）

暴走させないために、プランモードで毎回方針書を出させましょう。

「**いやいや、それは違うやろ！**」  
と気づける段階を作ることで、たいていの大きな手戻りは事前に防げます。

実装を一気に任せると、下層になるにつれ方針がブレたり、仕様書の内容が抜けることもあるので、セクションごとに小さくフェーズを分けて、少しづつ進行していくと事故りづらいです。

また、プランするときは、なるべく上位モデル（Opusなど）がオススメ。

安いモデルでプランを作って実装に走らせて、結果が思っていたものと違って、また戻って…をやるくらいなら、**最初から上位モデルでガッツリ計画を立ててもらったほうがトータルの時間は短い**。

ちゃんとプランしておけば、実装はSonnetでも問題なかったりします。色々試してみましょう。  
  
（僕の場合、トークンにゆとりがある時は、面倒なのでぜんぶOpusにしちゃいます）

## 3. 修正は「画像＋相対値」で伝えよう

言葉だけで説明するより、他のツールでレイアウトをざっくりあら組して、画像＋文字で指示出ししたほうが、修正指示はぜんぜん通りやすいです。

Claude Code、画像認識もけっこう賢いです。  
スクショベースでも意外と理解してくれます。  
  
（環境が整っているなら、Figma と Claude Code をMCPで繋ぐのもアリ）

![](https://assets.st-note.com/img/1779280387-p1ImkSy2nHeaGAOQc7TJoDu5.png?width=1200)

「〇〇〇〇.svgを背景に配置。▢秒に1回転のペースで無限回転。添付のようにレイアウト。」  
といった指示とともに、こんな感じのスクショで伝えるなど。  
（このレイアウトが良いかは別として…実際はもうすこし細かく伝えます。）

文字で修正を伝えるときも、コツがあります。

**① 相対値、または具体的な数値指定で伝える**

「ひと回り大きく」みたいな抽象指示は、ほぼ事故ります。AIにとっての「ひと回り」は、こちらの想像とぜんぜん違うサイズになりがちです。**「いまの1.1倍くらい大きく」**など、明瞭な指示出しが有効。

**②どこを修正するのかくわしく伝える**

「〇〇を▢▢に修正」といきなり伝えるより、  
「**△△セクションの上部の**〇〇を▢▢に修正」と伝える。  
CSSのクラスが分かるなら、それを貼っておくのも良いです。

Claude Codeが「それってどこの話？」と推論する時間とトークンを節約できます。  
ていねいすぎるくらいがちょうどいい。

人間に伝えるように、なるべくわかりやすく、具体的な指示を心がける。  
これだけで、AIが言うことを聞きやすくなります。

## 4. どうしてもダメなら、エフォートを上げよう

修正がうまく通らないときに覚えておくと便利なのが、AIのエフォート（工数）を上げるという手です。

Opusのエフォート「高」で何度やっても直せなかった修正が、「超高」に上げた瞬間に一発で直せる、ということもありました。

何度か詰まったら、コストを気にせずエフォートを上げてみてください。何回も修正を飛ばしているほうが、結果的にコストも時間もかかります。

**1回でズバッと通したほうが、トータルでは安く済む**、というのが実感です。

意外とあっさり突破できたりします。

## 5. 自分でコードを書く選択肢を捨てないでおこう

僕はもともとノーコードでサイト作っていましたが、今になって学生時代に学んだHTML・CSSの知識が役に立っています。  
（エンジニアの方には遠く及びませんが…）  
  
コードがまったく読めないよりは、何をしてるか分かる程度に読めたほうが良いな…と実感中です。
