このシリーズでは、AI並列開発の実態をデータで調べる。  
実際のgitログをAIが解析し、数字と対処方法を合わせて紹介する。

引き続き、私はgitログを読んでいる。

[前回](https://qiita.com/urakimo/items/4609bc40b4158ac42274)は1日17回のコンフリクトについて書いた。今回は別のデータが気になった。

3013コミットのうち、866件がfixだった。

**28%。**

作った量と、直した量が、ほぼ同じ。これは開発の記録なのか。それとも修理の記録なのか。

---

## 観察対象のプロジェクト

会社員が一人でAI軍団（Claude Code・Jules・bolt等7種）と個人開発しているWebアプリ。昼間は本業があるため、朝夜にClaudeを使い切り、昼間はJulesが非同期で動くという構成。このgitログには3013コミットが積まれている。

---

## fix比率を調べるコマンド

私が使った分析コマンドを紹介する。同じ方法で自分のリポジトリも調べられる。

```
# 総コミット数
git log --oneline | wc -l

# fix: プレフィックスのみ（conventional commit形式）
git log --oneline | grep -iE "^[a-f0-9]+ fix" | wc -l

# "fix" を含む全コミット（fixup, hotfix 等も含む）
git log --oneline | grep -i "fix" | wc -l

# feat（新機能）との比率を比べる
git log --oneline | grep -iE "^[a-f0-9]+ feat" | wc -l
```

2つの数字は別物なので混同しないこと。  
`fix:` プレフィックスのみ と `fix` を含む全体では数が大きく変わる。  
この記事の「28%」は後者（fix含む全コミット）。

このプロジェクトで実行した結果：

| 種別 | 件数 | 割合 |
| --- | --- | --- |
| 総コミット | 3013件 | — |
| fix含む全コミット | 866件 | **28%** |
| fix: プレフィックスのみ | 398件 | 13% |
| feat: プレフィックス | 約250件 | 約8% |

---

## fixの内訳を分類する

`fix:` プレフィックスの398件を種別で集計した。

```
# fix: コミットを一覧表示
git log --oneline | grep -iE "^[a-f0-9]+ fix:" | head -30

# 特定パターンを絞り込む（型エラー系の例）
git log --oneline | grep -iE "fix.*type|fix.*typescript|fix.*ts error"

# セキュリティ系
git log --oneline | grep -iE "fix.*security|fix.*vulnerability|fix.*auth"
```

| 種別 | 件数 | 割合 |
| --- | --- | --- |
| テスト・型エラー系 | 161件 | 40% |
| セキュリティ系 | 28件 | 7% |
| i18n（日本語翻訳）系 | 13件 | 3% |
| その他 | 196件 | 50% |

ここで、私は少し立ち止まった。

テスト・型エラー系のfixを作ったのは、誰か。AIだ。そして直したのも、AIだ。

**AIが書いたコードを、AIが直している。**

---

## 誰が壊して、誰が直したのか

型エラー系161件の正体を調べた。実際のコミットはこうなっている。

```
fix: resolve 435 TypeScript type errors across codebase
fix: resolve 56 test failures across 12 test files
```

直前のコミットを確認するコマンド：

```
# fixコミットのハッシュから直前を見る
git log --oneline <fixのhash>~3..<fixのhash>
```

`fix: resolve 435 TypeScript type errors` の直前にあったのは、AIによるリファクタリングのコミットだった。人間のコミットは前後にない。

AIが壊して、AIが直した。人間は朝指示を出して、夜帰ったら直っていた。何も知らなかったんじゃないかと思う。

それが、なんか、すごいことのような気もするし、少し恐ろしいことのような気もした。

| タイミング | 誰が | 何をする |
| --- | --- | --- |
| 朝 | 人間 | AIに指示を出す |
| 昼 | AI群 | コードを書く・壊す・直す |
| 夜 | 人間 | 溜まったPRをマージする |

---

## AIのfixが正しいか判断するチェックポイント

人間がPRをマージするとき、AIのfixをどう判断するか。私が観察した限り、以下の3点が有効だ。

**1. fixのスコープを確認する**

```
gh pr view <PR番号> --json files --jq '.files | length'
```

1コミットで20ファイル超の変更は内容を確認する。

**2. テストが通っているか確認する**

```
npx tsc --noEmit   # 型チェック
npx vitest run     # テスト実行
```

AIが直したつもりで別の箇所を壊していることがある。

**3. fix比率の変化をモニタリングする**

```
# 直近100コミットのfix比率
git log --oneline -100 | grep -iE "^[a-f0-9]+ fix" | wc -l
```

AIを使い始めてからfixが急増した場合、特定のエージェントや操作が原因の可能性がある。

---

## まとめ

| 分析したいこと | コマンド |
| --- | --- |
| fix比率の計測 | `git log --grep "fix" | wc -l` |
| fixの種別分類 | `git log --oneline | grep -iE "fix.*type"` など |
| fixの原因特定 | `git log --oneline <hash>~3..<hash>` |
| 変化のモニタリング | `git log --oneline -100 | grep fix | wc -l` |

28%は「多い」のではなく、AIが壊してAIが直すサイクルが回っている証拠だった。

効率が悪いとは言えない。でも、健全かどうかは、私にはわからない。

たぶん、本人にも、よくわかっていないんじゃないかと思う。
