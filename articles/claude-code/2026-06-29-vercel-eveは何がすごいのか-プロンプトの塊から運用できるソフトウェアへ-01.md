AIエージェントのデモは作りやすくなりました。

プロンプトを書く。ツールを数個つなぐ。SlackやWeb UIから呼び出せるようにする。小さな用途なら、ここまでで十分です。

業務やプロダクトに入れる段階では、別の種類の問題が出ます。

* このエージェントは、どのツールを呼べるのか
* Slackから来た入力とWebから来た入力で、同じ処理が走るのか
* 途中で止まった実行を再開できるのか
* 人間の承認待ちをどう扱うのか
* モデルにファイル操作や外部操作を許すとき、境界をどこに置くのか
* レビュー時に、どのファイルを見れば能力とリスクを確認できるのか

ここでは、こうした問題をまとめて「運用」と呼びます。運用には、エージェントの能力をレビューできること、実行状態を追えること、途中停止と再開を扱えることが含まれます。副作用のある操作を承認に回すことや、安全境界を設定することも含みます。

Vercelの `eve` は、この運用上の問題を扱うためのフレームワークです。

<https://github.com/vercel/eve>

eveの強みは、「エージェントをプロンプトの塊ではなく、レビューできるソフトウェアの形に置き直す」ところにあります。能力をファイルとして分ける。buildで発見する。infoで確認する。sessionとstreamで実行を追う。こうした構造によって、チームはレビュー、承認、再開、失敗調査の対象を分けられます。

この記事では、eveを `agent/`、build、info、tool discovery、session、streamという運用契約を持つフレームワークとして見ます。モデル性能は評価対象から外します。

公式docsを読んだうえで、`npx eve@latest init`、`eve build`、`eve info`、HTTP session API、tool追加、sandbox設定を試しました。本文には、そこで得たコマンド出力と観察結果だけを載せます。

図では、この記事で見ている変化をこう整理しています。eveはエージェントの中身を、ファイル、検査、実行、境界という4つの面に分けます。

![eveがエージェントを運用単位へ整える流れ](https://static.zenn.studio/user-upload/a1fb6d1772b5-20260629.png)

この図では、eveの価値をモデル性能ではなく、エージェントを検査・追跡・レビューできる構造へ寄せる点として示しています。

## 誰向けの記事か

この記事は、AIエージェントを業務やプロダクトに組み込みたいテックリード、プラットフォームエンジニア、バックエンドエンジニア向けです。

LLMを呼ぶコードは書ける。けれど、エージェントとして運用する構造をまだ決められていない。そのような読者を想定しています。

単発のチャットボットや、1回だけLLMを呼ぶ小さなスクリプトなら、eveは重く見えるかもしれません。eveは複雑になりやすいエージェントを整理するための道具です。短いLLM呼び出しを最小コードで書く用途とは少し違います。

## eveとは何か

公式docsは、eveを「ordinary files in a TypeScript project」でdurable agentsを作るフレームワークだと説明しています。GitHub READMEも、eveを「filesystem-first framework for durable AI agents」と呼んでいます。

`eve` では、エージェントを `agent/` ディレクトリとして書きます。

基本形はこうです。

```
agent/
  agent.ts
  instructions.md
  tools/
  skills/
  channels/
  sandbox/
  subagents/
  schedules/
```

それぞれの役割は次の通りです。

* `instructions.md`: エージェントに常時渡す指示
* `agent.ts`: モデルやruntime設定
* `tools/`: モデルが呼べるTypeScript関数
* `skills/`: 必要なときに読む手順書
* `channels/`: HTTP、Slack、Discordなどの入口
* `sandbox/`: ファイル操作やコマンド実行のための作業場設定
* `subagents/`: 親エージェントから委譲する専門エージェント
* `schedules/`: 定期実行

この構造を見ると、eveの設計方針が分かります。eveはエージェントを巨大な設定オブジェクトに詰め込みません。ファイルシステム上の役割分担として扱います。

## eveはどこから出てきたのか

eveには、Vercelのエージェント開発の蓄積があります。

Vercelの発表記事では、eveを作った背景として、同社が何年もエージェントを作ってきたこと、v0を含むエージェント開発を通じて、どのエージェントにも似た構造が現れたことを挙げています。

最初は、生産性の高い小さなエージェントが増えていきます。けれど数が増えると、各チームが同じような土台を何度も作ることになります。状態をどう持つか。外部サービスの認証をどう扱うか。承認待ちをどう表すか。ログやtraceをどう残すか。Slack、Web、HTTPへ入口を広げるとき、同じagent loopをどう使い回すか。

Vercelはこの状況を、Ship 2026で「Agent Stack」として整理しました。AI SDK、AI Gateway、Workflow SDK、Vercel Sandbox、Vercel Connectなどのprimitiveを並べ、その上でeveを「agentを1つのdirectoryとして扱えるframework」として紹介しています。Vercelのchangelogでは、eveは2026年6月17日にpublic previewとして公開されています。

この経緯を見ると、eveの設計意図が分かります。Vercelは、社内で何度も作ってきたproduction agentの共通形を、filesystem-firstなframeworkとして外に出しました。

公開後のchangelogにも、その性格が出ています。approval resume、continuation token、channel、sandbox、eval、health checkなど、変更の多くは運用時に効く部分です。eveは、エージェントを止めずに動かし、観測し、承認し、保守する部分を重視しています。

## scaffoldした時点で、エージェントの置き場所が決まる

まず、scaffoldを試しました。

```
npx -y eve@latest init my-agent
```

作成後、主要なファイルはこうなりました。

```
agent/agent.ts
agent/channels/eve.ts
agent/instructions.md
package.json
tsconfig.json
```

`package.json` には、次のscriptsが入りました。

```
{
  "build": "eve build",
  "dev": "eve dev",
  "start": "eve start",
  "typecheck": "tsc"
}
```

`agent/agent.ts` は `defineAgent` を使っていました。

```
import { defineAgent } from "eve";

export default defineAgent({
  model: "anthropic/claude-sonnet-4.6",
});
```

このscaffold直後の設定は、Vercel AI Gateway経由のmodel idです。後続のモデル応答、tool call、approvalの検証では、OpenAI keyを直接使うために `@ai-sdk/openai` を追加し、次のようにdirect provider設定へ変えました。

```
import { openai } from "@ai-sdk/openai";
import { defineAgent } from "eve";

export default defineAgent({
  model: openai("gpt-4o-mini"),
});
```

scaffold直後から `agent/` が中心に置かれます。プロンプト、チャネル、ツール、sandbox、subagentに、それぞれ置き場所があります。

チームで触るエージェントでは、この置き場所が役に立ちます。置き場所が決まると、変更差分の意味を読みやすくなります。tool追加は `tools/`、入口追加は `channels/`、常時指示の変更は `instructions.md` としてレビューできます。

レビューする人は、まず `agent/` を見ます。新しいツールは `agent/tools/` に置く。入口は `agent/channels/` に置く。この単純さによって、「どこを見ればよいか」が分かります。

## buildとinfoで、エージェントの構造を検査できる

次に、buildとinfoを確認しました。

```
npm run build
npx eve info
```

`npm run build` は成功しました。

`eve info` は、次の情報を出しました。

```
Layout        nested
Compile       ready
Diagnostics   0 errors, 0 warnings
Instructions  instructions.md
Compiled Manifest   .eve/compile/compiled-agent-manifest.json
Discovery Manifest  .eve/discovery/agent-discovery-manifest.json
```

eveは `agent/` 配下を発見し、`.eve/` 以下に discovery / compiled manifest を作ります。

チームは、どのinstructionsとtoolをruntimeが認識したかをレビュー時に確認できます。

レビュー担当者は、次を確認できます。

* eveはどのinstructionsを読んだか
* どのtoolを発見したか
* どのmanifestを出したか
* diagnosticsにエラーや警告があるか

`eve info` は、その入口になります。プロンプトを眺めて感覚で判断するより、確認できる対象が増えます。

次に、tool discoveryを試しました。

`agent/tools/get_weather.ts` を追加しました。

```
import { defineTool } from "eve/tools";
import { z } from "zod";

export default defineTool({
  description: "Return mock weather data for a city.",
  inputSchema: z.object({ city: z.string().min(1) }),
  async execute({ city }) {
    return { city, condition: "Sunny", temperatureF: 72 };
  },
});
```

build後、compiled manifestには次のように入りました。

```
logicalPath: tools/get_weather.ts
name: get_weather
sourceId: tools/get_weather.ts
```

`tools/get_weather.ts` が tool `get_weather` になりました。

公式docsも、eveではpathがidentityを持つと説明しています。ファイルの場所とtool名が結びつきます。

レビュー担当者は、`agent/tools/` と `eve info` を見て「このエージェントが呼べるtoolは何か」を確認できます。プロンプトの中に暗黙の能力が埋まっている状態より、扱いやすいです。

## sessionとstreamの契約がある

`eve info` は、HTTP messaging contract として次のrouteを表示しました。

```
Create       POST /eve/v1/session
Continue     POST /eve/v1/session/:sessionId
Stream       GET /eve/v1/session/:sessionId/stream
```

built serverを起動し、`POST /eve/v1/session` を呼びました。

レスポンスは `202` でした。bodyには `sessionId` と `continuationToken` が入っていました。

```
{
  "continuationToken": "eve:...",
  "ok": true,
  "sessionId": "wrun_..."
}
```

`sessionId` はstreamや検査のためのhandleです。`continuationToken` は会話を続けるためのhandleです。eve docsも、この2つを分けています。

この分離は、長く動くエージェントで意味を持ちます。承認待ち、再接続、後続入力を扱うなら、「実行を観測するhandle」と「会話を続けるhandle」を分けておく方が扱いやすくなります。

続いて、stream endpointを呼びました。

```
GET /eve/v1/session/<sessionId>/stream
```

NDJSONで次のeventが返りました。

```
session.started
turn.started
message.received
step.started
message.appended
message.completed
step.completed
turn.completed
session.waiting
```

短い応答を求めるpromptでは、`message.appended` が増分として流れ、最後に `message.completed` が返りました。実際の最終messageは次でした。

この時点で、session作成、model call、streaming response、turn completion、次の入力待ちまで確認しました。

ローカル検証では、eveが「1回のHTTPレスポンスで終わるチャット」よりも、sessionとevent streamを持つruntimeに近いことを確認しました。

## 成功も失敗もeventとして見える

AIエージェントを運用する人は、成功時の返答だけを見ても足りません。

失敗時には、どこで止まったかを見たくなります。

* モデル認証がない
* ツールが失敗した
* 承認待ちで止まった
* subagentが失敗した
* sessionが終端した

最初の検証では、モデル認証の不備によって `MODEL_CALL_FAILED` が出ました。その失敗は、HTTPの裏側で消えず、stream eventとして確認しました。

その後、OpenAI direct provider設定に変えて再実行すると、成功時には `message.appended`、`message.completed`、`step.completed`、`turn.completed` が流れました。成功と失敗の両方が同じevent stream上で扱われます。

監視やログ収集を設計するとき、成功系と失敗系を同じevent契約上で扱える可能性があります。

## approvalは停止と再開まで確認した

副作用を模したtoolとして、`refund_mock` を追加しました。

```
import { defineTool } from "eve/tools";
import { always } from "eve/tools/approval";
import { z } from "zod";

export default defineTool({
  description: "Mock a refund request. Does not contact any external payment system.",
  inputSchema: z.object({ chargeId: z.string().min(1), amountCents: z.number().int().positive() }),
  approval: always(),
  async execute({ chargeId, amountCents }) {
    return { ok: true, chargeId, amountCents, externalSideEffect: false };
  },
});
```

このtoolはbuildでき、compiled manifestにも `refund_mock` として入りました。

モデルに `refund_mock` を使うよう依頼すると、streamには次の流れが出ました。

```
actions.requested
step.completed
input.requested
turn.completed
session.waiting
```

`input.requested` には、`Approve tool call: refund_mock` というpromptと、`approve` / `deny` の選択肢が含まれていました。ここでsessionは `session.waiting` になり、人間の入力を待ちます。

続けて、同じsessionへ `continuationToken` と `approve` を送ると、次のeventが流れました。

```
turn.started
message.received
action.result
message.completed
step.completed
turn.completed
session.waiting
```

`action.result` には、mock toolの結果が入りました。

```
{
  "ok": true,
  "chargeId": "ch_test_456",
  "amountCents": 700,
  "externalSideEffect": false
}
```

ローカル検証では、approval-gated toolが人間の承認待ちで止まり、`approve` 後にtool resultへ進むことを確認しました。

sandboxについては、設定の発見と、組み込みの `bash` / `write_file` / `read_file` の実行を確認しました。

`agent/sandbox.ts` を追加しました。

```
import { defineSandbox } from "eve/sandbox";

export default defineSandbox({
  async onSession({ use }) {
    await use({ networkPolicy: "deny-all" });
  },
});
```

build後、discovery manifestとcompiled manifestの両方にsandboxが入りました。

```
logicalPath: sandbox.ts
sourceId: sandbox.ts
sourceKind: module
```

その後、modelにsandbox内でfileを書き、読み戻すよう依頼しました。streamでは `write_file` と `read_file` の `action.result` が返り、`/workspace/hello.txt` の内容として次を確認しました。

さらに、sandbox内で `bash` を実行するよう依頼し、次のstdoutを確認しました。

ローカル検証では、sandbox設定がauthored slotとして認識されるだけでなく、sandbox toolが `/workspace` を対象にread/write/bashを実行するところまで確認しました。default network egressなどの安全設定については、公式docsに従って利用者が設計する必要があります。

## 今回の検証で言えること

今回の検証から、eveについて強く言えることは次です。

* `npx eve@latest init` でfilesystem-firstなagent projectが作られる
* `agent/` 配下のファイルがauthored surfaceになる
* `eve build` と `eve info` でdiscovery / compiled artifactを確認できる
* toolのpathからtool名が導かれる
* HTTP session APIは `sessionId` と `continuationToken` を分けて返す
* stream endpointはNDJSON eventを返す
* 成功時には message / step / turn completion がeventとして観測できる
* 認証失敗時には session / turn / step failure がeventとして観測できる
* approval-gated tool は `input.requested` で承認待ちになり、`approve` 後に `action.result` へ進む
* sandbox設定はmanifestに現れる
* sandbox tool は `/workspace` に対して `write_file` / `read_file` / `bash` を実行できる

今回の検証範囲では、eveはエージェントを運用可能な単位へ分け、検査できる形にしています。

ファイルとして読める。buildできる。infoで確認できる。sessionとstreamの契約がある。成功も失敗もeventになる。approvalで止められる。sandboxでfile操作とbash実行ができる。

今回確認した範囲では、この「確認できる面」にeveの価値があります。AIエージェントをチームやプロダクトに入れる人は、モデルの賢さだけでなく、能力、入口、承認、安全境界、失敗をどう扱うかを見なければいけません。eveは、それぞれに名前と置き場所を与えます。

## 採用を検討しやすい場面

今回の検証から直接言える、検討しやすい場面は次です。

* ツール呼び出しをファイル構造として整理したい
* セッションを途中で止めたり再開したりする設計が必要
* エージェントの能力をレビューできる形にしたい
* build / infoでagent構造を確認したい
* approvalやsandboxを含めて、バックエンドAIエージェントとして運用したい

公式docs上の設計から見ると、次の場面も検討対象になります。

* SlackやWeb UIなど、複数の入口から同じエージェントを使いたい
* subagent委譲を、ファイル構造として整理したい
* scheduleを含む定期実行のエージェントを扱いたい

次の用途なら、まず軽い実装から始める選択肢があります。

* 1回だけLLMを呼ぶ小さな処理を書きたい
* session / stream / approval / sandbox が不要
* preview段階の仕様変更を受け入れにくい
* Vercel AI GatewayやOIDCまわりの前提をまだ整えたくない

## この記事で採用する判断

この記事では、モデル実行の性能ではなく、エージェントを運用可能な構造にするためのフレームワークとしてeveを評価します。

一方で、今回の結論には次の話を含めません。

* モデル応答の品質
* production scale での信頼性
* 採用実績や利用者数

速度については、5回のローカルsmoke測定だけ行いました。`message.completed` までの中央値は約1.12秒、`session.waiting` までの中央値は約1.16秒でした。ただし、これはローカル環境、OpenAI direct provider、短いpromptでの参考値であり、性能評価としては扱いません。

ここは、production運用を含む別の検証で扱う範囲です。

今回の証拠で支えられるのは、もっと手前の話です。eveは、エージェントをどこに置き、どう発見し、どうbuildし、どう観測するかを整えます。この記事では、その点をeveの強みとして扱います。

## まとめ

`eve`は、エージェントの置き場所と実行契約を整える道具です。AIを急に賢くする道具としては扱いません。

今回確認できた範囲では、eveはAIエージェントを次の形に近づけます。

* ファイルシステムで能力が見える
* buildとinfoで発見結果を確認できる
* toolのidentityがpathから決まる
* sessionとstreamの契約を持つ
* 成功と失敗をeventとして観測できる
* approvalで停止し、承認後に再開できる
* sandboxでbash/read/writeを実行できる

エージェントをプロンプトの塊として扱うと、レビュー、再開、承認、安全境界が見えにくくなります。eveは、それらをファイルとruntime contractに寄せます。

まだpreview/betaなので、性能や本番実績まで大きく語る段階には見えません。この記事の結論は、`agent/`、build、info、tool discovery、session、stream、approval、sandbox toolという運用契約を確認できた、という点に置きます。
