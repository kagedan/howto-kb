# はじめに

こんにちは！TOKIUMでQAチームに所属している富田です。

この度、弊社QAチームとして**新しいテスト自動化の取り組み**を行ったのでその紹介をさせていただきたく思います。

TOKIUMでは、E2Eテストの自動化は以前から進めていました。しかし、**新機能開発に対するシステムテストの自動化**には手をつけられていませんでした。

ここでいう「システムテスト」とは、**E2Eテストや受入テストの前段階で行われる、開発から渡された新機能が要件どおりに動くかを確認するテスト**を指します。リグレッションテストのように既存機能を広く通すのではなく、新たに実装された機能に対して、テストケース単位で細かく検証するものです。

これまでQA界隈では、E2Eテストのように既存機能を横断的に通す自動化は着実に進んできました。しかし、**新たに開発された機能に対する粒度の細かいテスト**は、毎回テスト対象の画面構造が異なり、カスタムコンポーネントの操作やユーザー権限の切り替えなど「人間が画面を見て判断する」部分が多いため、**自動化の実現が難しい領域**とされてきました。

今回、Claude Code + Playwrightを使って**テストケース（テスト仕様書）を渡すだけで、テストの自動実行・結果記録・スクリーンショット取得・動画録画まで一気通貫で行う仕組み**を構築し、この領域で実現の一歩を踏み出すことができました。

本記事では、構築の過程で得られた「AIにテストを任せるための工夫」を紹介します。

# 今回の自動化対象

まず、弊社での新規開発の全体フローと、今回どこを自動化したのかを示します。

![](https://static.zenn.studio/user-upload/0d23f15c7795-20260407.png)

開発チームが要件定義→実装→単体・結合テスト→テスト環境デプロイを経て、QAチームにテスト依頼が来ます。QAチームは要件をもとにテスト設計を行い、テスト環境にデプロイされた新機能に対して **システムテスト（機能要件をみたしているかの確認）** を実施します。その後、E2Eテストを通してリリースへ進みます。

今回、このシステムテスト部分をClaude Code + Playwrightで自動化しました。

# テスト自動実行の全体像

今回作成した自動実行ツールは、以下のような流れで利用します。  
![](https://static.zenn.studio/user-upload/a42cb0bbec2c-20260407.png)

QA(人間側)は、テストケースを用意さえすれば、あとはClaudeにテスト実行命令を出すだけで、動いてくれる仕様となります。

# ポイントになる箇所

### テストアカウント/データは人間が事前準備する

![](https://static.zenn.studio/user-upload/317c68187ecd-20260407.png)

テスト環境のアカウントはTOKIUM SSOで管理されているため、Claude Codeの中からアカウントを作成することが難しいです（不可能ではないが、データミスに繋がりやすい）。そこで、**必要なアカウント/データは人間が事前に作成し、`.env.staging`にロール別で定義しておく**方式にしました。

```
# .env.staging
ADMIN_EMAIL=XXXXXXXXX@tokium.jp
APPLICANT_EMAIL=●●●●●●●●●@tokium.jp
GENERAL_EMAIL=△△△△△△△@tokium.jp
```

テストスクリプトからは `loginAs(page, "admin")` のように呼び出すだけ。ロールの追加も `.env.staging` に1行追加するだけで済みます。

### テストケースはGherkin(ガーキン)記法で記述

![](https://static.zenn.studio/user-upload/cd69afbb5a1b-20260407.png)

テストケース（テスト仕様書）は、Gherkin記法（Given/When/Then）で記述しています。

| 列 | 内容 | 例 |
| --- | --- | --- |
| Given（前提条件） | テスト開始時の状態 | 「一般ユーザーでサインインしている」 |
| When（操作手順） | 実行する操作 | 「申請メニューから購買申請を作成する」 |
| Then（期待結果） | 確認すべき結果 | 「タイムラインに『承認されました』が表示」 |

この記法により、テストケースが人間にとって読みやすいだけでなく、AIがテストコードに変換する際にも**Given→ログイン・画面遷移のセットアップ、When→UI操作、Then→アサーション**と明確にマッピングできます。

### 「そもそも自動化できるか」を判定する

![](https://static.zenn.studio/user-upload/3e8a542f363c-20260407.png)

テスト実行に入る前に、全テストケースを以下の4カテゴリに分類します。

| 判定 | 基準 | 例 |
| --- | --- | --- |
| ◎ 自動化可能 | UIの表示確認、ボタン操作、画面遷移 | 「メニューが表示されること」 |
| ○ 条件付き | 事前データ操作が必要 | 「特定の権限のユーザーで確認」 |
| △ 手動推奨 | 目視確認、ブラウザ互換性 | 「レイアウトが崩れないこと」 |
| ✗ 自動化不可 | 特殊環境が必要 | 「パフォーマンス計測」 |

**◎と○のテストケースのみ自動実行の対象**とし、△と✗は手動テスト対象としてスキップします。この事前判定により、自動化に向かないテストケースに時間を浪費することを防ぎます。

### テストスクリプト作成前に、画面のHTML構造を調査する

![](https://static.zenn.studio/user-upload/f3e8ec2cc668-20260407.png)

初回の実行で最も多かった失敗原因は「**UIの構造を知らないままテストコードを書いた**」ことでした。

* 確認ダイアログの存在を知らずに、申請ボタンのクリック後に何も起きない
* 共有モーダルが標準的なselectではなく、カスタムMultiSelectコンポーネントだった
* 「申請」ボタンがヘッダーとフォーム内の2箇所に存在していた

そこで、**spec作成前にフロントエンドのソースコードを読んで、対象画面のコンポーネント構造を調査するステップ**を追加しました。

```
# CONVENTIONS.md より

## 9. spec作成前のUI調査
- 対象画面のフロントエンドソースを読み、カスタムコンポーネントの有無を確認
- 確認ダイアログ、モーダル内のARIA属性を事前に把握
- setup.mjs実行時に各画面のスクリーンショットを保存し、セレクタ設計の材料にする
```

これにより、初回実行でのセレクタ起因の失敗が大幅に減りました。

### 新人テスターを育てるように、ナレッジを蓄積する

![](https://static.zenn.studio/user-upload/9e3077c88b60-20260407.png)

ここが一番重要なポイントです。

Claude Codeは会話が終わるとコンテキストがリセットされます。つまり、**前回の会話で学んだことを次回の会話に引き継げない**。これは新人テスターが毎日記憶を失うようなものです。

そこで、**2つのナレッジファイル**を用意しました。  
（**これらのファイルをどう育てるかが、非常に重要となります！**）

```
runtest/
  CONVENTIONS.md           ← 「どうやるか」（技術ルール）
  TEST-JUDGEMENT-GUIDE.md  ← 「どう判断するか」（判断基準）
```

#### CONVENTIONS.md = 技術ルール

セレクタの書き方、待機の仕方、セッション管理など、テスト自動化の技術的なルールを蓄積します。

```
## 10. セレクタ設計ルール
- リンクセレクタは対象ページ固有のパスで絞り込む
  - ✗ a[href*="/request/"]（サイドメニューにもマッチ）
  - ✓ a[href*="/shared_requests/"]（リスト内リンクのみ）
- 同名ボタンが複数存在する場合、.first() / .last() を明示する

## 11. テストアカウントの選択
- テスト仕様が「○○権限のユーザー」と指定した場合、
  その権限だけを持つアカウントを使う
```

#### TEST-JUDGEMENT-GUIDE.md = 判断基準

PASS/NGの判定で間違えた事例と、正しい判断を記録します。**PASS事例だけでなく、正しくNGを出せた事例も記録**します。

```
## 原則
1. テスト仕様に書かれた操作対象が見つからなければNG
2. NGの備考には「なぜ実行できなかったか」を明記
3. テスト対象の権限以外の上位権限を持つアカウントを使わない
4. フォールバック処理は判定を甘くする方向に使わない

## NG事例集
### NG-001: 対象フィールドが見つからない場合に別の要素で代替してPASS判定
- 発生: 〇〇機能テストA テストケースID-4
- テスト仕様: 「契約金額総額の金額を変更する」
- 何が起きたか: フィールドが見つからず、コメント欄のtextareaに値を書き込みPASS判定
- 教訓: フォールバックで「何かしら編集できた」はPASSの根拠にならない

### NG-002: テスト仕様が指定するロールと異なる権限のアカウントを使用
- 発生: △△機能テストB テストケースID-7
- テスト仕様: 「承認者権限をもつユーザー」
- 何が起きたか: 管理者兼承認者アカウントを使用し、管理者権限でPASS
- 教訓: 「承認者」のテストには承認者だけのアカウントを使う

## OK事例集
### OK-001: 画面に存在しないボタンに対して正しくNG判定
- 発生: XX機能テストC テストケースID-3
- 共有された申請画面に「取り下げ」ボタンがない → 正しくNG
```

**この仕組みにより、テスト実行で指摘を受けるたびにナレッジが成長し、同じミスを繰り返さなくなります。**

### わざとNGになるテストケースを作って、判定力を鍛える

前述のナレッジの蓄積の話と繋がりますが、  
AIは「何かしら操作ができた」ことをPASSと判断しがちです。実際に起きた偽PASSの例：

> テストの期待値: 「契約金額総額フォームの金額を変更できること」
>
> テスト自動実行の結果: テスト対象の画面に、該当の金額フォームが表示されていないバグが発生していた → にもかかわらず、別の**入力欄のtextareaを「編集可能なフィールド」として誤検出** → その入力欄に金額を書き込み → PASS判定してしまった

これは明らかに偽PASS（false positive）です。テスト仕様が指定した要素を操作できたかどうかだけが判定基準であり、別の要素で代替してはいけません。

この問題を検出するため、**わざと実態と異なる期待値を書いたテストケースを混ぜて**実行しました。例えば「共有された申請の詳細画面から取り下げボタンを押下する」というテストケース。実際には共有先ユーザーに取り下げ権限はなく、ボタンは表示されません。

結果、正しくNGを出せたことで、判定ロジックの信頼性を確認できるような仕組みにしました。  
もしここで正しくNGを出せなかった場合、次の「新人テスターを育てるように、ナレッジを蓄積する」の指示をおこないます。

# 実際の検証結果

弊社が提供する、「社内の申請・承認をオンラインで完結させるワークフローシステム」を対象に今回この自動テストの精度検証を行いました。

**■テスト対象**  
上記ワークフローシステム内に実装した「**すべての申請**」メニューをテスト対象とします。

**■プロダクト仕様**  
・この「すべての申請」メニューはアカウント権限（管理者・承認者・一般ユーザー等）に応じてメニューの表示/非表示が切り替わる　※ **"管理者"** 権限をもつアカウントだけメニューが表示  
・「すべての申請」メニュー内の申請書では、 **承認・差し戻し・コメント** 等の操作を行える  
　※**申請内容（申請書の中身）は編集できない**

**■テストデータ（事前に用意しているアカウント）**  
自動テスト専用に用意しているアカウントです。  
（今回の「すべての申請メニュー」のテスト用に準備したわけではなく、様々な自動テスト全般に汎用的に使っているアカウントです）  
毎回テストの自動実行をClaudeに依頼する際、**Claudeはこれらのアカウントの中から最適なものを選び**、各テストケースの実行に入ってもらいます。

| --- | --- | --- |
| テストユーザーA | 申請者 | (権限なし) |
| テストユーザーB | 承認者 | "承認者" |

| テストユーザーD | 集計者 | "集計者" |

**結果(1回目)**：  
自動テスト実行時、画面要素がうまく取得できない箇所があり、大半のテストが落ちました。  
この学びを活かし、テストスクリプト作成前に「**画面のHTML構造を調査する**」 STEPを追加しました。  
その上で、同じテストケースに対し2回目のテスト実施を行いました。

**結果(2回目)**：  
2回目では大半のテストが成功しました。  
しかし、**#3**については、想定では❌**NG** になるべき内容でしたが、✅**OK**となっておりました。

# まとめ

今回の自動化で、特に工夫したことは以下です。

1. **Gherkin記法でテストケースを構造化する** — Given/When/Thenの明確な分離がAIの変換精度を上げる
2. **自動化可否を事前判定する** — ◎○△✗の分類で、自動化に向かないケースを事前に除外する
3. **事前調査を仕組みに組み込む** — UI構造を知らずにコードを書かせない
4. **わざとNGになるケースで判定力を確認する** — 偽PASSを許さない
5. **ミスのたびにナレッジを蓄積する** — 新人テスターが経験を積むように、AIも成長させる

特に5つ目の「ナレッジ蓄積」が最も重要です。CONVENTIONS.md（技術ルール）とTEST-JUDGEMENT-GUIDE.md（判断基準）の2ファイルに事例を書き溜めていくことで、テストを重ねるほど精度が上がっていきます。

今回このようなテスト自動化ツールの作成を通じ感じたことは、

「E2Eテスト」の自動化は"一度書いたら同じテストをやり続ける"ものなので、「**"テスト"を育てる**」ものですが、  
「新機能に対するシステムテスト」の自動化は、人間のテスターのように"毎回テスト内容が異なるケースを臨機応変に考えてテスト" してもらうものなので「**"テスター"を育てる**」感覚にとても近いと思いました。

これまで自動化が難しいとされてきたこの領域で、今回ようやく実現の一歩を踏み出すことができました。  
今後、このテスター（AI）をどんどん成長させ、やがて一人前にしていくことを目指していきます。  
また進捗があれば記事に書きます。
