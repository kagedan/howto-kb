## はじめに

VS CodeのCopilot Chatで `.agent.md` を使ったカスタムエージェント、便利ですよね。

特に `runSubagent` を使い、複数の専門エージェントをオーケストレーションする構成は、  
VS Code公式ドキュメントでも推奨ユースケースとして紹介されています。

一方で、実際に高度な構成を組んでみると、  
「**サブエージェント側（`.agent.md`）で指定したモデルが無視され、親エージェントのモデルが強制的に使われる**」  
という挙動に直面します。

つい昨日（2026年3月25日）リリースされた最新のVS Code 1.113.0でも検証しましたが、残念ながらこの挙動は変わっていません。

この記事では、現在発生しているこの事象の事実関係と、コミュニティでの推測、そして公式リポジトリでの最新動向をまとめました。

## 発生している事象

たとえば、静的型付けの厳密なレビューを任せるためのサブエージェント `type-checker.agent.md` を作ったとします。

軽いルーチン処理はメインのGPT-4o mini等に任せつつ、型の複雑な推論が必要なこのエージェントだけは、賢いモデル（Claude Sonnet 3.6など）を使いたいというケースです。

```
---
name: type-checker
description: 静的型付けの厳密なコードレビューを行うエージェント
tools: ["read", "edit"]
model:
  - "Claude Sonnet 3.6 (copilot)"
  - "GPT-5 (copilot)"
---
```

このようにフロントマターの `model` に指定しても、実際に呼び出されると**親（メインチャット）で選ばれているモデルがそのまま引き継がれます。**

### 公式ドキュメントとの矛盾

GitHub Copilot公式ドキュメントでは、`.agent.md` のフロントマターに `model` を指定できることが明記されています。

にもかかわらず、VS Code + Copilot Chat の **subagent 実行経路ではモデル切り替えが反映されません**。

## 確認されている関連Issue・議論

この挙動と直接・間接的に関係するIssueや議論は、以下で確認できます。

### 🔗 モデル／エージェント情報がリクエストから落ちる不具合

このIssueでは、サブエージェントやskills情報が  
**モデルリクエスト生成時に欠落するケース**が報告されています。

> custom agents が存在していても、model request に反映されないタイミングがある

という点は、モデル指定無視問題と構造的に近いです。

---

### 🔗 プレミアムモデルの「課金・計測境界」に関するIssue

また、この問題を整理した技術記事も公開されています。

この記事では、

* モデル実行
* 課金・リクエスト消費
* サブエージェント呼び出し

の **境界が一致していない** ことが、明確に指摘されています。

## なぜモデル指定が無視されるのか？

Microsoft / GitHubから  
**「意図的な仕様です」と明言された一次情報は存在しません。**

しかし、以下の事実が確認できます。

これらを踏まえると、

> **サブエージェントで任意にモデルを切り替えられると、  
> 課金・レート制限・利用枠の制御が崩壊する**

という判断がシステム側にある可能性は高いと言えます。

実際に「モデルは切り替わらない」「フォールバックが効かずループする」  
といった報告は複数上がっています。

## 現状の現実的な運用策

VS Code 1.113.0時点でもこの問題は未解決であり、モデルの使い分けによるコストやパフォーマンスの最適化は不可能です。

そのため、オーケストレーションを安定稼働させるには「**メインエージェント側に、タスク全体の最大負荷に耐えられる強力なプレミアムモデルを最初から固定で選択しておく**」というアプローチを取るしかありません。

今後のアップデートで、正当にリクエスト枠を消費する形での「モデルの独立指定」がサポートされることを期待しつつ、今はプロンプトの洗練やコンテキスト（渡すファイル等）の絞り込みに注力するのが良さそうです。
