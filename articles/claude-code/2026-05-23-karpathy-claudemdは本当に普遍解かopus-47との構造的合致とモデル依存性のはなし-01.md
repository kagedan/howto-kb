## TL;DR

* Karpathy CLAUDE.mdが急速に広まったのは偶然ではなく、Opus 4.7の言葉通りに解釈する挙動＋agentic stamina向上と構造的に噛み合ったから
* ただし他のモデルに渡すと効き方が変わる。特に原則4（Goal-Driven Execution）はモデル依存度が高い。
* CLAUDE.mdは「コード」であり、モデルはランタイム。同じファイルでも実行環境が変われば挙動が変わる。
* 組織でAI活用する場合、CLAUDE.mdを「育てていくドキュメント」として扱う運用が要る

## はじめに

GitHubで爆発的に伸びている「Karpathy CLAUDE.md」というファイルがある。  
実際の著者はForrest Chang氏で、Andrej KarpathyがLLMコーディングの失敗パターンについて公開した観察を、4つの原則に蒸留したもの。  
Karpathy本人が書いたわけではないが、彼が自分のチャンネルで紹介していて事実上のお墨付きがある。

中身は65行・4原則のシンプルなmarkdown：

* Think Before Coding
* Simplicity First
* Surgical Changes
* Goal-Driven Execution

一時的にバズっていたようでがこれをCLAUDE.mdの土台に採用している人が多い。  
このファイルが急速に広がったタイミングはOpus 4.7のリリース前後だった。

本記事では次の3つを順に整理する：

* なぜKarpathy CLAUDE.mdはOpus 4.7と構造的に噛み合うのか
* 他のモデルに渡しても同じ効き方をするのか
* ここから組織のAI運用設計に何が言えるのか

## Opus 4.7との構造的合致

Karpathy CLAUDE.mdが4.7と相性がいいのは、4.7の2つの変化に直接結びついていた。

### 変化1：4.7は「ギャップを埋めなくなった」

4.6では書かれていない部分を「たぶんこういう意味だろう」と推測で埋めてくれることが多かった。4.7はこれをやめて、書かれた通りに動く言葉通り寄りに振れている。Anthropic公式発表も4.7について明確に「Opus 4.7 takes the instructions literally」と書いている。  
便利な場面も多い反面、「言外の好み・チームのコンテキスト」を渡す手間が増えた。

あるブログ記事の言い方を借りれば  
**「CLAUDE.mdはこれまで以上に重要になった」**  
要はモデルがもう推測してくれない以上、デフォルトを明示的に書いておく場所が必要になったから。

ここでKarpathy CLAUDE.mdの **原則1（Think Before Coding）と原則3（Surgical Changes）** が活きる。

* 原則1（曖昧なら止まって聞け、推測するな）── 4.7はそもそも推測したくない。原則1はその性質を補強する
* 原則3（隣接コードを"ついでに改善"するな）── 4.6が気を利かせてやりがちだったことを、4.7はやらなくなっているが、明示しておけばより確実

### 変化2：4.7はagentic stamina（多段階実行力）が上がった

複数のコーディングベンチで10〜15%の向上。  
（Anthropic公式発表より：93タスクのコーディング評価で+13%、Cursor内部評価で70%対58%、Factory Droidsで10〜15%向上など）

長尺のagenticトレース中に進捗報告を自然に出すようになり、さらに**自分の出力を検証する仕組みを自ら考案する**ようになった。  
（公式表現："devises ways to verify its own outputs before reporting back"）  
「人間が逐次介入しなくても、ゴールに向かって走り続ける」能力の向上。

ここで **原則4（Goal-Driven Execution）** が効いてくる。  
「成功条件を渡せばモデルが自走でループする」というこの原則は、agentic staminaを前提にしている。4.7の長尺実行力と原則4の設計思想は、構造的にハマる。

つまり 4.7 + Karpathy CLAUDE.mdは、

* 推測しないモデルに、必要なデフォルトを明示で渡す
* 多段階実行が得意になったモデルに、検証可能なゴールを渡す

という構造になっていて、4.7の性質を欠点ではなく特徴として活かす設計になっている。  
急速に広まったのは偶然ではなく、4.7のリリースタイミングと噛み合ったから、と考えるのが自然だと思う。

## 他のモデルだとどう効くか

実際、X上でこんな観察が共有されていた。

そして補足のツイート：

これは結構興味深い実例👀  
Karpathy CLAUDE.mdが万人にとって最適解ではなく、**使い方やモデルとの相性によって"逆効果"にもなりうる**ことを示している。「Opusポンコツ化」と感じた原因が、評判のいいスキルファイル自体だったかもしれない。

では、何が「合う／合わない」を決めるのか。Karpathy CLAUDE.mdの4原則を、モデル別の効き方で整理してみた。

### 原則1〜3：ほぼ全モデルで有効（普遍的な歯止め）

オーバーエンジニアリング、隠れた仮定、隣接コード破壊。  
これらはLLM共通の失敗モード。Opus 4.7だろうがSonnet 4.6だろうが、放っておけば「気を利かせて」やりがち。

なので原則1（Think Before Coding）、原則2（Simplicity First）、原則3（Surgical Changes）は**モデルを選ばず効く普遍的な歯止め**として機能する。

### 原則4：モデル依存度が高い

問題は原則4（Goal-Driven Execution）

「成功条件を渡せばモデルが自走でループする」というこの原則は、agentic stamina（多段階実行を最後まで完遂する能力）を前提にしている。

モデル別にざっくり整理すると：

| モデル | 原則4の機能性 |
| --- | --- |
| Opus 4.7 | 本領発揮（言葉通りの指示理解＋長尺実行力） |
| Opus 4.6 | 機能するが、推測でショートカット取られて意図とズレることがある |
| Sonnet 4.6 | 短いループは回るが、複雑なゴールだと途中で迷子になる確率が上がる |
| Haiku 4.5 | ほぼ機能しない。ペアプロ前提のほうが現実的 |

ここで注意したいのは、**原則4が機能しきらないモデルにそのまま渡すと、むしろノイズになる可能性がある**ということ。「自走しろ」と書かれているのに自走力が足りないモデルは、無理にループしようとして変な振る舞いをすることがある。

### 「ポンコツ化」の正体は適材適所のズレかも

冒頭のツイートに戻ると、ツイート主の具体的な使い方は分からないので推測になるけど、構造的にはこういう仮説が立つ

* そのCLAUDE.mdは特定の使い方（おそらく長尺agenticタスク）を前提に設計されていた
* でも実際の使い方は、もっと短いやり取り・対話的なものが中心だった
* 結果、ファイルが要求するスタイルと実際の使い方がミスマッチを起こした

つまり「ファイルが悪い」のでも「モデルが悪い」のでもなく、**ファイルとモデルと使い方の組み合わせの問題**。「バズってるからとりあえず入れる」という採用パターンが、このミスマッチを生みやすい構造になっている。

## CLAUDE.mdも「コード」だよねって話

ここまでの話を一段抽象化すると、こう言える。

**CLAUDE.mdは「コード」であり、モデルがランタイムである。**

普通のコードは、ランタイム依存性を当たり前のように意識する。Vue 2からVue 3への移行で苦労した経験は多くのフロントエンドエンジニアが持っているし、`package.json`でライブラリのバージョンを固定し、`engines`フィールドでNodeのバージョンまで揃える習慣がある。

ところがCLAUDE.mdに対しては、この感覚が薄いのではないかなと感じている。

* 多くのCLAUDE.mdは「どのモデルを前提にしているか」を書いていない
* 他のリポジトリのCLAUDE.mdをコピペするとき、自分のチームが使っているモデルとの相性を検証することは少ない
* モデルが新バージョンになっても、CLAUDE.mdの再評価はほぼ行われない

しかし前のセクションで見た通り、**同じCLAUDE.mdでもモデルによって効き方が変わる**。これはコードのランタイム依存性とまったく同じ構造だ。

別の言い方をすると、CLAUDE.mdは「自然言語で書かれたコード」であり、**モデルバージョンという気付かれない依存性を持つ**。気付かれないのは、書き換えなくても動いてしまうから、依存性そのものに意識が向きにくいからなのでは。

Anthropic公式ドキュメントも、CLAUDE.mdの性質についてこう書いている。

> CLAUDE.mdの内容はsystem promptの後にuser messageとして渡される。Claudeはそれを読んで従おうとするが、strict complianceは保証されない。特に曖昧または矛盾する指示の場合は。

つまりCLAUDE.mdは**強制ルールというより文脈**であり、その解釈はモデルに委ねられている。同じ文脈でもモデルによって解釈が変わるのは、構造上避けられない。

「ファイルが古くなったけど一応動いてる」状態のCLAUDE.mdは、たぶん多くのチームのリポジトリに存在する。そして次のモデル世代が出たときに、また「なんか調子悪い」が再現される。

では、組織でこれをどう扱えばいいか。

## 組織への含意──「モデルを意識したCLAUDE.md」という成熟度

個人で使っている分には、「あれ、最近Claudeの調子悪いな」と気付いた人が自分でCLAUDE.mdを見直せばいい。手間ではあるが、フィードバックループは短い。

組織に持ち込むと話が変わる。

* 複数人がCLAUDE.mdを共有している
* 誰かが書いたファイルを、別の誰かが使っている
* モデル更新の影響が、チーム全体に一斉に出る
* 「最近Claudeが調子悪い」がチーム内で散発的に発生して、原因がCLAUDE.mdだと気付くまでに時間がかかる

冒頭で引用したツイートのような「ポンコツ化」体験が、個人レベルでは数日で解決するのに対し、組織レベルでは数週間〜数ヶ月放置されることがある。これは普通に運用コストとしてバカにならない。

### harness-maturityの軸として整理する

組織のAI活用成熟度を測る軸はいろいろある（プロンプト品質、ツール選定、レビュー体制、など）。ここに「CLAUDE.mdがモデルを意識しているか」という軸を加えてみる。

| レベル | 状態 | 説明 |
| --- | --- | --- |
| Lv1 | コピペ運用 | 評判のいいCLAUDE.mdをそのまま流用。モデルの想定なし |
| Lv2 | プロジェクト固有化 | 自社のコード規約・スタック情報を追記している |
| Lv3 | モデルを意識した記述 | 使用モデルを前提に注記している（"原則4はOpus 4.7前提"など） |
| Lv4 | バージョン管理運用 | モデル更新時にCLAUDE.mdをレビューする運用がある |

世の中の多くのチームはおそらくLv1かLv2で止まっているのでは？  
Lv3・Lv4への移行は地味な作業だが、長期的に見ると「気付かれない劣化」を防ぐ効果がある。

### 具体的に何ができるか

* CLAUDE.mdの冒頭に「想定モデル: Claude Opus 4.7」のような見出しや記述を入れる
* モデル更新の際にCLAUDE.mdを見直すチェックリストを持つ
* 「このCLAUDE.mdはどのモデルで検証したか」をPR時に明記する
* 社内で複数モデルを使い分けている場合、モデル別のセクションを持つ

どれも特別なツールは要らなくて、「意識して書く」と「定期的に見直す」だけで成立する仕組みが出来そう。

### 環境を設計する側から見ると

AIネイティブな組織は、「個別の出力をレビューする」から「出力を生む環境を設計する」へと役割が移行していく。CLAUDE.mdはまさにその「環境」の一部になる。

ただ、その環境がモデル依存性を持っていることを意識しないと、せっかくの環境設計がモデル更新によって静かに無効化される。  
**環境を設計するというのは、環境の依存性も含めて設計することなんだろうな** みつを。

CLAUDE.mdを書いて終わり、ではなく、**CLAUDE.mdを「育てていくドキュメント」として扱う運用**。これが、agentic codingが本格的に組織展開していく時代のharness-maturityの一つの軸になっていくはず。

## というポエムでした

私は今月からAI活用文化を組織に根付かせるための仕事をしているが、CLAUDE.mdを巡るこの一連の観察は、結構象徴的な事例だなーと思ってる。  
良いプラクティスが多くのチームに届くこと自体はとっても良いこと。ただ、それが「コピペで導入」だけで止まると、そのプラクティスが持っているはずの効果が、モデルや使い方とのミスマッチで打ち消されてしまう。

今のCLAUDE.mdは、どのモデルを前提に書かれているか、育てていくドキュメントとして意識していきたい。
