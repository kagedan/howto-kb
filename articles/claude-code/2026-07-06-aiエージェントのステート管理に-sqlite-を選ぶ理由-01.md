最近、以前なら迷わず Postgres を選んでいた場面で、SQLite を検討することが増えてきました。

きっかけは小さなサービスでした。そこから、「マルチテナント SaaS を SQLite で運用できるのか」「AI エージェントのステートは、中央 DB ではなくローカル DB に置くほうが自然ではないか」という問いに広がっていきました。

いまのところ、この方向性で最もしっくりきているのは Turso です。特に Cloudflare と組み合わせたときの相性がよいと感じています。一方で、Cloudflare D1 に Turso の embedded replica 相当の仕組みがほしいですし、AWS にも RDS for Postgres のような SQLite / libSQL 向けマネージドサービスがほしいと思っています。

もちろん、これは「Postgres はもう不要」という話ではありません。私自身、いまでも多くの場面で Postgres を使っています。この記事で書きたいのは、Postgres を置き換える話ではなく、**ワークステートの置き場所として SQLite を先に検討する場面が増えている**、という話です。

現時点での整理はこうです。

* SQLite は Postgres の代替ではない
* ただし、作業中の一時的なステートには向いている
* AI エージェントは、その傾向を強める
* 重要なのは「すべてローカル」ではなく、ローカルの作業台と中央の台帳を分けること

## これまで中央 DB が自然だった理由

長いあいだ、アプリケーションのデータはサーバー上の Postgres に置くのが自然でした。共有 Postgres に `tenant_id` を付け、API の背後で管理する形です。

これは優れたアーキテクチャだったからというより、最も運用しやすく、安く、枯れていたからです。

SQLite は昔からあらゆる場所にありました。しかし、SaaS の基盤として使うには周辺機能が足りませんでした。ネットワーク越しのアクセス、レプリケーション、バックアップ、多数の小さな DB を管理する仕組みなどです。

そのため、中央に集約した Postgres が最も抵抗の少ない選択肢でした。

変わったのは SQLite 本体ではありません。SQLite のまわりのエコシステムです。特に、Turso / libSQL、Cloudflare D1、Durable Objects、Litestream / LiteFS のような仕組みによって、これまで足りなかった運用レイヤーが少しずつ補われてきました。

## SQLite の制約は消えたわけではない

SQLite には今も明確な制約があります。

* ネットワーク DB ではなく、プロセスに埋め込むライブラリである
* 基本は単一ファイルであり、標準ではレプリケーションを持たない
* WAL によって多数のリーダーは扱えるが、ライターは基本的に 1 つである

これらの制約がなくなったわけではありません。むしろ、制約を理解したうえで、周辺の仕組みで補う形になっています。

たとえば Turso / libSQL は、SQLite 互換の体験にマネージドなサーバーレイヤー、リモートアクセス、embedded replica、暗号化、ベクトル検索を加えています。Cloudflare D1 は、Cloudflare Workers との組み合わせで使いやすいサーバーレス SQLite です。Durable Objects は、コードと同じ場所に SQLite ベースのストレージを持てます。Litestream / LiteFS は、WAL を使ってバックアップやレプリケーションを扱いやすくします。

つまり、「SQLite はネットワーク化できない」「共有インフラでは使えない」という前提は、少なくとも一部のワークロードでは以前ほど正しくなくなっています。

## ステートは、よく読み書きする場所に寄る

私が最近よく使っている考え方があります。

**ステートは、それを最も頻繁に読み書きする場所へ寄っていく**、というものです。

ライターとステートが離れているほど、操作のたびにコストが発生します。レイテンシ、ネットワーク転送、コネクション管理、整合性の調整などです。逆に、両者を近づければ、その多くは小さくなります。

これまでの Web アプリでは、ライターは主に人間でした。クリックやフォーム送信は頻度が低く、多少のネットワーク往復も許容できました。そのため、中央 DB にステートを置く設計が自然でした。

AI エージェントでは、この前提が変わります。

実用的なエージェントは、作業環境の内側で動きます。ファイルシステム、ターミナル、IDE、ブラウザ、認証済みセッションに近い場所です。Codex CLI や Claude Code のようなツールは、ローカルのコードベースを読み、ファイルを編集し、コマンドを実行します。

エージェントが扱うステートも多くなります。

* タスク計画
* ファイルインデックス
* 差分
* ツール呼び出し履歴
* 失敗とリトライ
* 承認状態
* 検索インデックス
* embeddings のキャッシュ
* 同期カーソル
* undo / redo ログ

これらは頻繁に書き換わります。そして多くの場合、ローカルに閉じています。他のユーザーに共有する価値も、中央の台帳に保存する必要もありません。

こうしたステートをすべて中央 Postgres に往復させると、レイテンシ、コスト、プライバシーの面で不利になります。ライターがローカルにいるなら、ワークステートもローカルに置くほうが自然です。

## ローカルは近いだけでなく、十分に大きい

この話が特に当てはまるのは、ユーザーのマシン上で動く常駐エージェントです。

現代のノート PC は、ユーザー 1 人に対してかなり大きなリソースを持っています。複数の CPU コア、十分な RAM、高速な NVMe SSD、ローカルファイルシステム、場合によっては GPU / NPU もあります。

一方、クラウドのマルチテナント実行環境が 1 ユーザーに割り当てるリソースは、その一部です。さらに、サーバー側で安全にツールを動かすには、サンドボックス、VM、ネットワーク制御、権限管理、監査ログ、キューイングも必要になります。

モデルの推論はクラウドでよいとしても、読み取り、書き込み、差分計算、インデックス化、検索、キャッシュは、データと計算資源がある場所で行うほうが安く済みます。

ただし、この議論はサーバー側で動くマルチテナント SaaS エージェントにはそのまま当てはまりません。その場合、ユーザーのノート PC はループの中にありません。SQLite を選ぶ理由は、ローカルリソースではなく、テナント単位の隔離、コスト構造、障害範囲の小ささにあります。

## database-per-tenant という選択肢

SaaS でも、SQLite を使うなら **テナントごとに 1 つの SQLite データベースを持つ** という設計が有力になります。ユーザーごと、エージェントごとでも同じです。

共有 Postgres に `tenant_id` を持たせるのではなく、物理的に DB を分けます。

SQLite はデータベースがファイルなので、この設計と相性がよくなります。

* 10,000 番目のテナントも、もう 1 つのファイルにすぎない
* コストは DB 数ではなく、実際の使用量に寄せやすい
* `WHERE tenant_id = ...` の書き忘れではなく、物理的に隔離できる
* バックアップ、エクスポート、削除、復元をテナント単位で考えやすい
* 失敗したマイグレーションの影響範囲を限定しやすい

一方で、代償もあります。

* クロステナントのクエリは難しくなる
* マイグレーションは全 DB に扇状に広がる
* DB ごとの単一ライター制約は残る
* D1 のように、1 DB あたりの容量や実行モデルの制約があるサービスもある

特にクロステナント分析は、実行時に多数の SQLite ファイルを直接 JOIN する発想ではなく、テナントごとの outbox からイベントストリームへ流し、中央のウェアハウスで扱う設計になります。

つまり、共有 Postgres の簡潔さを一部手放す代わりに、物理的な隔離、テナント単位の運用、限定された障害範囲を得る設計です。

## 読み取りはローカル、書き込みは中央

SQLite 系の構成では、読み取りと書き込みを分けて考えることが重要です。

```
PRIMARY
  │ WAL frames
  ▼
EDGE REPLICAS
  │
  ▼
EMBEDDED REPLICA
  └── writes ──▶ PRIMARY
```

リージョナルなリードレプリカは、データをユーザーに近づけます。ただし、それでもネットワーク読み取りです。

一方、Turso / libSQL の embedded replica は、SQLite ファイルをプロセス内に同期します。読み取りがマシンの外に出ません。エージェントが自分のステートを頻繁に読むなら、この差は大きくなります。

ただし、書き込みは最終的にプライマリへ送られます。単一ライターの制約は消えません。読み取りのローカリティを取り、書き込みの中央集約を受け入れる設計です。

これは「ローカルの作業台」と「中央の事実」を分ける考え方と相性がよいです。

## エージェントの DB は作業台であり、記憶でもある

エージェントにとって SQLite が向いているのは、単なる軽量 DB だからではありません。

エージェントの作業には、構造化データ、検索インデックス、履歴、キャッシュ、メモリが混在します。これらを同じ DB に置けると扱いやすくなります。

たとえば Turso は、リレーショナルなデータとベクトル検索を同じ場所で扱えます。エージェントのメモリや RAG 用の embeddings を、別のベクトル DB に分けずに済みます。

Durable Objects のような仕組みでは、特定のエージェントやワークスペースに対して、計算と SQLite ストレージを同じ場所に置けます。「このエージェントがこのワークスペースを所有する」という境界を、そのまま実装に落とし込めます。

## それでも中央に置くべきもの

すべてを SQLite に寄せるべきではありません。むしろ、中央に残すべきものを明確に分けることが重要です。

中央の厳格なストアに残すべきなのは、主に次の領域です。

* コラボレーションされるオブジェクト
* 権限、監査、操作履歴
* お金、請求、契約、在庫のようなグローバルな真実

ローカル SQLite は、同期、コンフリクト解決、権限管理、監査を自動的に解決してくれるわけではありません。

誰が何を承認したのか、どのモデルがどの権限で操作したのか、何が最終的に変更されたのか。これらはローカルマシンだけに置くべきではありません。

そのため、全体像は次のようになります。

```
SYSTEM OF WORK
  ローカル SQLite
  embedded replica
  tenant-per-database
        │
        ▼
  event log
        │
        ▼
SYSTEM OF RECORD
  Postgres
  warehouse
  search index
```

ローカルは速く、頻繁に変わる作業台です。中央は遅くてもよい代わりに、厳格で共有可能な台帳です。

## 1 つの変更をどう扱うか

たとえば、ローカルで動くエージェントが顧客向けのレコードを編集したとします。

すべてを中央に置けば、監査や権限管理はきれいにできます。ただし、作業中の細かなステートまでネットワーク越しに送ることになります。

すべてをローカルに置けば、作業は速くなります。しかし、チームから見える監査や権限の置き場所がありません。

現実的なのは中間です。

作業中のスクラッチなステートはローカル SQLite に置く。重要な変更だけをイベントログに流し、中央の台帳で認可・監査・共有する。この分担が、AI エージェントのステート管理には合っていると感じています。

## 2026 年時点で足りないもの

Turso / libSQL は、現時点でこの方向性を最もよく実装している選択肢だと感じています。マネージド DB、embedded replica、ベクトル検索、テナント単位の経済性、オープンソースのコアを持っています。

Cloudflare D1 は、Cloudflare 上で完結するなら非常に扱いやすい選択肢です。Workers、Durable Objects、R2 と組み合わせやすく、運用負荷も小さいです。一方で、Turso の embedded replica のように、プロセス内のローカルファイルから読むモデルではありません。

Durable Objects は、エンティティごとに計算と SQLite ストレージを同じ場所へ置けるプリミティブです。エージェントやワークスペース単位のステート管理と相性がよいと見ています。

一方、AWS にはまだ RDS for Postgres に相当する第一級の SQLite / libSQL マネージド層がありません。EFS、S3、Litestream、EC2 上の rqlite などを組み合わせることはできますが、標準的なマネージドサービスとしてはまだ弱いです。AWS 標準のチームにとっては、ここが採用の大きなハードルになります。

## まとめ

SQLite は Postgres を置き換えるものではありません。

ただし、AI エージェントのワークステートを考えると、SQLite はかなり自然な選択肢になります。エージェントのステートは頻繁に変わり、ローカルに閉じ、プライベートで、寿命が短いことが多いからです。

Postgres は、中央の台帳として残ります。お金、請求、契約、監査、共有すべき事実を扱う場所です。

SQLite は、テナントごと、ユーザーごと、エージェントごとの作業台になります。イベントログが、その差分を中央へ運びます。

私が最近感じているのは、この役割分担のほうが、多くの AI エージェント向けワークロードに自然に見えるということです。習慣で中央 DB を選ぶ前に、ステートを最もよく読み書きする場所はどこかを考える。その結果として、SQLite に手が伸びる場面が増えています。
