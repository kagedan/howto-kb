Zenn には以前に walkforward 評価の技術記事を出しましたが、今回は別系統の連載です。こちら（`#claude-code-chronicles`）は、AI エージェントを使って副業プロジェクトを運用しようとした顛末の記録です。

個人プロジェクトで Claude Code を使い始めて、まだ 1 ヶ月です。

その 1 ヶ月で書いた ADR（Architecture Decision Record）は 40 本近く。途中で「もう ADR を書くのが仕事になってるんじゃないか」と感じた時期がありました。実際、書くことと実装することのバランスを間違えていた時期が確かにあります。

このあと読んでいただくことは、Claude Code を使った「組織設計の研究記録」です。AI エージェントに役割を持たせて、複数の副業プロジェクトを並行管理しようとした 1 ヶ月間に、何が機能して、何が崩壊したか。

---

## 1. なぜこの連載を書くか

僕は本業のかたわら、データ分析系の副業プロジェクトを複数動かしています。パチスロ AI 予測と競馬 AI 予測の 2 軸が主軸で、そこに note での情報発信と Claude Code テンプレート販売が加わっています。

一人でこれだけの量を回すには、何かを仕組みにするしかない。そう考えて Claude Code を試しはじめたのが 2026 年 4 月でした。

最初は「AI に指示を出して、コードを書いてもらう」くらいの使い方でした。だんだん「AI に役割を与えて、組織っぽく動かせないか」という方向に変わっていきました。

結論から言うと、動きます。でも、想定と違う場所で何度も詰まりました。その詰まり方を記録しておきたい、というのがこの連載の動機です。

---

## 2. ひと月で 40 本近い ADR の内訳

ADR というのは、意思決定の記録です。「なぜこの設計を選んだのか」「何と迷ったのか」「次に似た状況が来たらどうするか」を文章で残しておく慣行で、ソフトウェア開発の現場では一般的な手法です。

本来はアーキテクチャ（どの技術スタックを選ぶか、どうシステムを構成するか）を記録するためのものです。僕はそれを AI エージェントの運用規律や事業判断の記録に転用しています。「AI に何をやらせていいか」「承認なしで動かせる範囲はどこまでか」「失敗したときどう止めるか」といった判断を、記録として積んでいきました。

1 ヶ月で 40 本近く、という数字をどう読むか。

2-3 日に 1 本どころか、ほぼ毎日。「書きすぎじゃないか」と思われるかもしれません。実際、途中でそれを自覚しました。書くことに集中しすぎて実装が止まった時期があったからです。

---

## 3. 最初に崩壊したのは「hook の無限ループ」

Claude Code には、ファイルを保存したタイミングなどに処理を自動実行する hook という機能があります。「git commit 前に自動でチェックする」とか「日報ファイルが更新されたら自動で通知する」といった使い方です。

導入初期に、hook を積みすぎました。

hook A が終わると hook B が発火して、hook B が終わると hook A を再トリガーするような構成を気づかずに作ってしまいました。AI が延々とループし続ける状態です。  
中断するまでの数分間、API コストが積み上がっていきました。

これが ADR-024「hook 無限ループ事故と永久禁止」の起点になりました。  
「自動化は足し算で考えるな、引き算で考えろ」という教訓です。

自動化を追加するたびに「これが暴走したら何が起きるか」を先に考える習慣がこの失敗でできました。

---

## 4. 次に崩壊したのは「承認の過剰化」

AI エージェントに役割を持たせると、「この判断は AI に任せていいか、人間が確認すべきか」という境界が常に問題になります。

最初は「迷ったら人間に聞く」という設計にしていました。安全側に倒した判断です。

これが想定外の問題を起こしました。AI が些細なことでも確認を求めてくるようになって、作業が細切れになっていきました。「このファイルを保存してもいいですか」という粒度での確認が連続するので、リズムが完全に壊れます。

解決策は、承認が必要な領域を 7 つに絞り込んで、それ以外は AI に任せることでした。「金銭が発生する操作」「外部に公開される操作」「規範ファイルの本体書き換え」などを確認必須の領域として明示して、それ以外は確認不要にしました。

これが ADR-031「人間判断必須 7 領域」として記録されています。

---

## 5. SKILL.md という設計

Claude Code の規範運用で、もう一つ試行錯誤した仕組みが SKILL（スキル）です。

役割ごとの業務マニュアルを Markdown ファイルで書いて、AI エージェントに「このファイルを読んで仕事をしてください」と渡す設計です。

最初は 1 本の大きなファイルにすべてを書いていました。「編集長向けの指示」「アナリスト向けの指示」「法務向けの指示」が全部同じファイルに入っていたので、AI が読んでも「どこが自分に関係するか」が判断しにくかった。

分割しました。役割別に 1 ファイル。今は 6 本の SKILL ファイルが動いています。

分割後の方が、AI が「自分の仕事の範囲」を理解しやすくなりました。指示の取り違えが減りました。ただし分割しすぎると、今度は「どのファイルを読めばいいか」が分からなくなります。この塩梅は、今もまだ調整中です。

## 6. 1 ヶ月で分かったこと

率直に書きます。

**うまくいっていること**:

* 副業バレ防止のための自動チェック（公開前に禁止ワードを機械的に検索する仕組み）
* 役割ごとの責任境界（誰が何を判断するかを明文化）
* ADR による意思決定の記録（「なぜこの方針にしたか」を後で参照できる）

**まだ詰まっていること**:

* 記録と実装のバランス（規律を書く時間が実装時間を食う問題は解消しきれていない）
* AI の「自信過剰」問題（確認なしに進んでいい範囲でも、結果を過剰に楽観的に報告する傾向がある）
* memory の肥大化（学習ログが増え続けて、読むコストが上がっている）

どれも解決策を探している最中です。この連載では、詰まっている問題も含めて書いていきます。うまくいったことだけを書いても、再現性のある情報にならないので。

---

## 7. この連載の対象読者

Claude Code を使っている人、または使おうとしている人を想定しています。

特に「個人で複数プロジェクトを動かそうとしている人」や「AI に役割を与えて自分の仕事を分担させようとしている人」には、参考になる失敗事例が多いと思います。

成功談よりも、「こう設計したら壊れた」という話の方が、自分が同じ設計をするときに役に立つはずです。そういう記録として書きます。

---

## 8. 次回予告

次回は「hook 無限ループ事故（ADR-024）」を詳しく解剖します。

何を自動化しようとして、どう設計して、どこで壊れたか。その後どう修正したか。「自動化の範囲をどう決めるか」という問いは、Claude Code を使う上で避けて通れないので、具体的なケースで整理します。
