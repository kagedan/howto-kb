Claude Code で、個人投資の「攻め」を担うエンジンを作っています。狙いは、NVIDIA のような構造変化に乗る勝者を、コンセンサスが固まる前に見つけて数年握ることです。ただし、株価を当てにいく予測機は作っていません。

この領域で個人が勝てない本当の理由は、「勝者を見つけられない」ことではなく、「見つけても握り続けられない」ことにあります。だからこのエンジンは、予測の精度ではなく、確信の作り方と握力の支え方に設計を寄せています。

前の記事で、投資ツールを「素材の基盤＋攻め＋守り」の三層で組んでいる全体像を書きました。この記事は、その中の「攻め」——構造変化から勝者を見抜くエンジン——の中身だけを、単独で掘り下げます。先に断っておくと、これは完成品を配る話ではなく、設計と作り方の話です。

> 前提となる三層の全体像はこちら：[Claude Code で個人投資の AI スタックを組む — 予測機は作らず、既製OSSは「素材」にする](https://zenn.dev/takotyan00/articles/claude-code-investing-ai-stack)

## ボトルネックは「発見」ではなく「握力」

まず、このエンジンが何を解こうとしているのかを、はっきりさせます。

構造的な勝者は、だいたい3つの段階を通ります。NVIDIA を例にすると、次のようになります。

| 段階 | 状態 | NVIDIA の例（後知恵） | 当時の市場の声 |
| --- | --- | --- | --- |
| ① 芽 | 技術の芽はあるが市場は信じない | 深層学習の台頭（CUDA が土台・2015-16 頃） | 「ゲーム用 GPU 屋でしょ」 |
| ② 初期の数字 | 業績が非連続に跳ね始めるが「割高」に見える | 2023 のデータセンター急増 | 「PER 高すぎ、バブル」 |
| ③ コンセンサス | 全員が同意し、株価が一段跳ねる | 2024 | 「AI の覇者」 |

10 倍の入口は①〜②にあります。でも多くの個人は、③で「もう答えが出た銘柄」を高値で掴みます。

やっかいなのは、①②で買えたとしても、そこから③までの間に株価が半値以下になる局面を何度もくぐる、ということです。「CUDA で化ける」と早い段階で言えた人は当時から大勢いました。でも、途中の大きな下落を耐えて握り切れた人は、ほとんどいません。

つまり、**個人のボトルネックは「発見」ではなく「握力」** です。ニュースを速く読むような浅い発見は、もう誰でもできます。差がつくのは、その先の2点です。

* 構造変化を「マージンを握る企業」まで降ろす、影響マップの深さ（＝深い発見）
* 賭けた仮説が崩れていないかを見張り続ける、反証トラッキングの握力

エンジンの edge はこの2点に置いていて、ニュースの速さには置いていません。

## エンジンの二段構え

この2点を支えるために、エンジンは「網」と「確信づくり」の二段で組んでいます。段Aで機械的に候補を広く拾い、段Bで生成 AI の力を使って確信と握力を作ります。

段A は「予測」を一切しません。構造的な勝者になりやすい性質でふるいにかけるだけです。段B は仮説を立て、それが本物かを量的に確認し、崩れたら気づく、という握力の部分を担います。そして最後に賭けるかどうかは、必ず人間（私）が決めます。

ここで一つ、言葉を定義しておきます。この記事で「予測しない」と言うのは、**株価の目標値も「何%上がる確率」も一切置かない**、という意味です。追いかけるのは価格の予想ではなく、賭けた仮説（構造の前提）が生きているか死んでいるか。複数のシナリオが転んでも恩恵が残るような、選択肢のあるポジションを選ぶ。これが、当てにいくことと確信を作ることの違いです。

## 段A：誰が勝つかより「誰が勝っても儲かるか」

段A で一番効かせている軸が **基盤性** です。

「AI で誰が勝つか」を当てるのは難しい。でも「AI で誰が勝っても、その全員に売れる会社はどこか」なら、勝者を特定しなくても答えられます。ゴールドラッシュでいう、金を掘る人ではなく、つるはしとジーンズを売る側です。半導体でいう TSMC や ASML、電力やデータセンターのような、全プレイヤーに課金できる基盤企業を優先的に拾います。

そのうえで、テーマ適合（拡大する巨大 TAM に乗っているか）、増収の持続（一発でなく売上が伸び続けているか）、高 ROE・再投資効率といった、株価とは独立した財務の質でふるいます。ここで小型株には縛りません。NVIDIA は大型のまま 10 倍になりました。10 倍を生むのは会社の小ささではなく、乗っている構造変化の大きさだからです。

ただし、段A は万能ではありません。同じ「割高だが本物」という顔をして、その後 10 年低迷した会社は歴史上いくつもあります（ドットコム期の Intel や Cisco がその典型です）。性質が一致するだけでは選別しきれないので、段A はあくまで「網」であって「答え」ではない、という位置づけにしています。

## 影響マップ：構造変化を「マージンを握る企業」まで降ろす

ここから段B、LLM が効く部分です。

影響マップは、**構造変化 → バリューチェーン → マージンを握る企業** を降ろしていく作業です。「AI 推論がクラウドからエッジに移る」といった構造仮説を置き、そこから恩恵が流れる産業を辿り、最終的に「この変化で利益率を握るのはどこか」を特定します。

この工程で生成 AI を使うのは、ニュースを速く読むためではありません。一段の連想で終わらせず、二段・三段先の受益者まで辿る深さを出すためです。表面のニュースは全員が同じ速度で読めるので、そこに差はつきません。差がつくのは、「その変化で誰が構造的にマージンを取り続けるか」という、地味で込み入った連鎖のほうです。

ここで気をつけているのは、LLM にそれらしい対応表を作らせないことです。放っておくと、AI は「この技術ならこの銘柄」というもっともらしいマッピングを、根拠が薄いまま埋めてきます。だから連想の各段に「なぜそこが儲かるのか」を必ず添えさせ、根拠の弱い枝は落とす。影響マップは、綺麗な図を作る道具ではなく、確信の根拠を一段ずつ検証する道具として使っています。

## 先行指標：損益計算書に乗る前を見る

影響マップで「ここが勝つはず」という仮説ができたら、次はそれが本物かを量的に確認します。

このとき、決算に出てくる売上や利益を待っていては遅い。それは③コンセンサスの段階の数字だからです。狙いは②の初期の数字、つまり **損益計算書に乗る前のシグナル** を捉えることです。たとえば、

* 川上企業の設備投資ガイダンスの連鎖（A 社が増設を発表 → その部材を作る B 社に効く）
* 特定分野の求人の急増、開発者の採用動向
* arXiv や GitHub での、その技術の実装・採用の広がり
* 政策の動き（補助金・輸出規制のような、産業構造を動かす制度）

こうした「決算より前に動く」シグナルを仮説ごとに紐づけて追い、②の段階で買い増しの判断材料にします。ニュースそのものではなく、ニュースが決算に変換される前の一次的な足跡を見にいく、という設計です。

## 株価を教師にしない反証ループ

そして、このエンジンでいちばん独自だと思っているのが、握力を支える反証トラッキングです。

一度賭けた仮説について、「これが崩れたと言える具体的な条件」をあらかじめルールにしておきます。構造としては、こういう形です。

```
# 反証ルールの例（構造だけ。実際の保有とは無関係）
仮説: "この会社は構造変化Xの基盤（つるはし）を握る"
disconfirm_when:
  - metric: 増収率（前年同月比）
    condition: 2四半期連続で失速
  - metric: 想定した先行指標（設備投資ガイダンス等）
    condition: 反転して縮小に転じる
action: 「仮説が崩れた候補」として通知する（売買はしない・判断は人間）
```

こうして **崩れる条件を先に言語化** しておくと、株価が下がったという理由だけで狼狽して売る、という握力の崩れを防げます。下がったからではなく、構造が崩れたときにだけ手を離す、という判断ができるようになります。

ここに、既製ツールから意図的に外した設計があります。投資向けのマルチエージェント OSS の中には、判断のあとに「その判断が当たったか」を振り返って学習する、reflection ループを持つものがあります。たとえば私が中を読んだ [TradingAgents](https://github.com/TauricResearch/TradingAgents) は、反省の教師信号を「結果（株価）が当たったか」——実現リターンや指数比の超過リターン——に紐づけています。よくできた仕組みです。

でも私のエンジンは、この反省の教師信号を **株価ではなく構造の推移** に置き直しています。理由は単純で、「株価は当てにいかない」と最初に決めたからです。当たり外れで学習させると、いつのまにか予測機になってしまう。だから反証ループが見るのは「株価が上がったか下がったか」ではなく、「賭けた構造変化そのものが、まだ続いているか／崩れたか」です。ここは、既製の仕組みから借りられなかった（借りてはいけなかった）ところでした。

## 「予測しない」のは、なぜ正しいのか

「株価を当てない投資ツール」と言うと、逃げに聞こえるかもしれません。でも、これは負けを認めているのではなく、勝てる場所を選んでいる、という設計判断です。近年の金融 × LLM の研究とも、方向は揃っています。

* LLM ベースの投資戦略が長期で市場（buy-and-hold）に勝てるかを厳密に評価した研究では、統計的に有意な超過リターンは確認できなかった、と報告されています（[arXiv 2505.07078](https://arxiv.org/abs/2505.07078)）。
* 株式レーティングを LLM に作らせた研究では、4四半期分のファンダメンタルズ（財務）データ単独が最も精度が高く、ニュースを入れるとむしろモデルが楽観（ポジティブ）方向にバイアスする、と報告されています（[arXiv 2411.00856](https://arxiv.org/abs/2411.00856)、J.P. Morgan、ICAIF'24、Figure 4）。

ただ、正確に言うと、この研究群が支えてくれるのは私の設計の「方向」であって、設計そのものの有効性ではありません。都合よく束ねないよう、切り分けて書きます。

* **確信の土台をファンダに置く**：これは直接の裏づけがあります（4四半期のファンダ単独が最も精度が高い、という結果そのもの）。
* **予測で市場に勝ちにいかない**：これは「勝てない」側からの支持です。上の研究が示したのは「LLM 予測は市場に勝てない」ことで、「では確信と握力に振れば勝てる」ことまでは示していません。つまり私の代替案が正しいという証明ではなく、「予測で勝ちにいく道は分が悪い」という否定を受け取っているだけです。
* **ニュースを主軸にしない**：これも支持されます（ニュースは楽観方向にバイアスする）。ただし「だから反証＝悪い材料を一次入力に置く」という私の設計選択までは、この研究の射程外です。

当てにいかないのは、当てられないからではなく、当てにいく道が分が悪いと分かっているからです。そのぶんを、確信と握力に回しています。

## Claude Code の使いどころ

このエンジンで生成 AI（Claude API）が担うのは、仮説を立てる・影響マップを辿る・反証を判定する、という解釈のレイヤーだけです。データの取得や財務のふるいは機械的なコードがやり、AI は「材料を出す係」に徹します。

そして設計そのもの——既製 OSS の reflection ループを読んで「構造は借りるが、株価を教師にする思想は捨てる」と切り分けたり、影響マップの連想に根拠を求めて弱い枝を落としたり——は、Claude Code と一緒にやっています。とくに効くのは、既製ツールの実際のコードを読ませて「これは自分の前提に通るか」を一次情報で判断させるときです。表面の説明だけで似ている / 違うを決めさせない。この「読んで確かめてから借りる」進め方が、エンジンの芯を素材の寄せ集めにしないための歯止めになっています。

## 正直なところ

線を引いておきます。

このエンジンは、各層とも実装の段階にはありますが、まだ精緻化の途中です。完成した予測装置が動いていて誰でもコピペで使える、という話ではありません。反証ループも、いまは私が手動で回していて、全自動では走っていません。「動くものとして書いている」のではなく、「こう設計して、こう作っている」という話です。

それから、これは自分用の意思決定支援ツールに閉じています。個別銘柄の売買を勧めるものではありません（自分の情報を可視化して、自分が判断するための道具です）。実際の保有銘柄・金額・口座は公開しません。この記事で共有できるのは、あくまでエンジンの設計と考え方までです。

## なぜ、こう作るのか

最後に、このエンジンの独自点をもう一度だけ。

投資向けの AI ツールも思想も、いまは大量に用意されて手に入ります。そのまま使えば、みんなが同じモデルで同じ結論に収束していく。だからこそ、そこから自分の要件で選んで、自分用に削って仕立て直せる人が果実を取れる——という背骨は、これまでの記事と同じです。

その背骨の上で、この「攻め」のエンジンが賭けているのは、**発見の速さではなく握力**という一点です。勝者を当てる予測機はいくらでも作れる時代に、あえて予測を捨て、株価ではなく構造の推移を教師信号にした反証ループで「握り続ける」を設計する。皆が AI で同じ答えに寄っていくほど、コンセンサス前に確信して握り切れることの希少性は、むしろ上がると思っています。

次は、この確信を毎朝の材料で更新する仕組み——ニュースを構造化してシグナルに変え、このエンジンに供給する連携——を書く予定です。
