はじめまして。セーフィー株式会社の池淵峻一です。2025 年 3 月より AI Studio チームで、バックエンドの開発を担当しています。

[Safie AI Studio](https://safie.jp/ai-studio/) は、セーフィーのカメラ映像に対して AI 解析を導入・開発できるプラットフォームです。人検知や車番認識といった映像 AI ソリューションを、35 万台以上のカメラに展開できる基盤を提供しています。私はこのプラットフォームのバックエンド——API サーバー、データベース、非同期ワークフローなどの開発に携わっています。

この記事では、MLOps 基盤を作る上で AI コーディングアシスタント Claude Code を導入・活用しようとした際に直面した課題と、それを解決するために構築したナレッジベース「Odin」について紹介します。

## Claude Code を導入したら、壁にぶつかった

### セーフィーのバックエンド開発

セーフィーはクラウド録画型の映像プラットフォームを提供している会社です。建設現場の安全管理、小売店の防犯、オフィスの入退室管理など、さまざまな現場にカメラを届けています。

その裏側を支えるバックエンドシステムは、認証、映像配信、AI 解析、外部 API など複数のサービスに分かれており、それぞれ異なるチームが開発しています。

### Claude Code への期待と現実

2025 年、バックエンドチームに AI コーディングアシスタント Claude Code（Anthropic 社）を導入しました。コードの生成、レビュー支援、設計の相談——開発のあらゆる場面で AI が力を貸してくれるツールです。

導入の期待は大きく、実際にコード生成の速度は上がりました。しかし、使い込むうちに**ある壁**にぶつかりました。

**AI は社内の文脈を知らない。**

AI は一般的なプログラミングの知識は豊富です。しかし「うちの認証サービスでは OAuth2.0 のスコープをこう定義している」「エラーレスポンスはこのフォーマットで返す」「データベースの変更はこの順序で行う」——こうした社内固有の決まりごとは知りません。

結果として、Claude Code の提案はどうしても「教科書的な一般論」にとどまります。正しいが、**うちのやり方ではない**。このギャップを埋めるには、毎回チャットで文脈を説明し直す必要がありました。

### なぜ AI に文脈を渡せなかったのか

では、社内のドキュメントを AI に渡せばいいのでは？　そう考えて振り返ると、**そもそも「渡せるドキュメント」が存在しなかった**のです。

ソフトウェア開発チームが大きくなると、知識が人の頭の中に閉じ込められていきます。「この API の仕様はどこに書いてある？」——Notion の中にあるかもしれない。Slack の過去スレッドかもしれない。あるいは、誰かの頭の中にだけあるかもしれない。

セーフィーのバックエンドは複数のサービスで構成されており、それぞれオーナーとなるチームが異なります。サービスごとにドキュメントの書き方も置き場所も違い、**ドキュメント全体のオーナーシップが不明確**な状態でした。あるサービスの設計書は Notion に丁寧に書かれているが、別のサービスは Slack の過去ログを辿るしかない。

さらに厄介なのが、**サービスをまたぐ情報**です。たとえば「新しい API 権限を追加する」には、認証サービスで権限を定義し、API サーバーでその権限を検証し、外部開発者向けドキュメントを更新する必要があります。3 つのサービス、3 つのチーム。この調整は Slack で「この名前で合ってますか？」とやりとりし、返答を待ち、認識のずれを修正し……と、実装よりも調整に時間がかかることも珍しくありません。

この状態では、AI に渡せる「まとまった文脈」がどこにもない。Claude Code を本当に活用するには、まず**AI に渡すための基盤**を作る必要がありました。

### 発想の転換 —— 「人が読む」から「AI も読む」へ

ここで考え方を変えました。

これからのドキュメントの読者は人間だけではない。**AI もドキュメントの消費者である**。この前提に立つと、ドキュメントに求められる性質が変わります。

* **構造化されたメタデータ** — AI が文書の関連性を判断できるように
* **コードとの双方向リンク** — 「このドキュメントはどのコードに対応するか」を明示
* **一貫したフォーマット** — 機械的な解析を可能に

こうして生まれたのが **Odin**（北欧神話の知恵の神の名前）です。単なるドキュメント置き場ではなく、**Claude Code に社内の文脈を渡すための基盤**として設計しました。

## AI に文脈を渡すために何を作ったか

### コンテキストマネジメントという考え方

Odin を「ドキュメント管理ツール」と呼ぶのは正確ではありません。Odin が管理しているのは**コンテキスト**（文脈）です。

AI がコードを書くとき、本当に必要なのはプログラミング言語の文法知識ではありません。「このプロジェクトではエラーをどう返すか」「認証はどのサービスが担当するか」「データベースの変更はどの順序で行うか」——こうした**プロジェクト固有の文脈**がなければ、AI は正しい判断ができません。

人間のエンジニアは、この文脈をチーム内の会話やコードレビューの経験から暗黙的に蓄積しています。しかし AI にはそれができない。だから**文脈を明示的に構造化し、AI が参照できる形で管理する仕組み**が必要でした。

Odin はその仕組みです。設計判断の根拠、サービス間の取り決め、運用手順、命名規則——開発に必要な文脈を一箇所に集め、人間にも AI にも読める形で提供します。

### 全体像

Odin は MkDocs（Markdown で書かれた文書から Web サイトを生成するツール）をベースにしたモノレポ（複数のプロジェクトを一つのリポジトリで管理する方式）です。

含まれているもの:

* **技術ドキュメント**: 1,300 以上の Markdown ファイル（日本語・英語の両方）
* **テスト自動化ツール**: ステージング環境での動作確認を自動化
* **ローカルテスト環境**: 開発者の手元で動く検証環境
* **エラーコード管理**: 全サービス共通のエラーコード体系
* **AI 向け検索システム**: ドキュメントを AI が検索できる仕組み

### AI が理解できるドキュメント設計

AI にドキュメントを読ませるには、人間が読みやすい文章を書くだけでは足りません。AI が「この文書は何について書かれていて、どのコードと関係があり、どの粒度の情報なのか」を判断できる構造が必要です。

Odin ではすべてのドキュメントに対して、3 つの層で AI 向けの構造を設計しています。

**第 1 層: メタデータ**

文書の先頭に frontmatter（文書の冒頭に埋め込むメタデータ）を付けています。カテゴリ、タグ、ステータス、担当チーム、抽象度（high / medium / low）などの情報です。AI はこのメタデータを索引として使い、「この文書が今の質問に関係あるか」を全文を読む前に判断できます。

さらに `code_references` フィールドで、ドキュメントとソースコードを双方向にリンクしています:

```
code_references:
  - repository: "example/api-server"
    path: "src/handlers/resource.py"
    lines: "45-120"
    element: "create_resource"
    type: "function"
```

これにより AI は「この API ドキュメントは、あるリポジトリの特定ファイル・特定行の関数に対応する」と正確に理解できます。

**第 2 層: テンプレート**

ドキュメントの種類ごとに、AI が構造を理解しやすいテンプレートを用意しています。サービスドキュメントは「なぜ必要か（Why）→ 何か（What）→ どう使うか（How）」の順で書く。冒頭に TL;DR（一言要約・対象読者・主要機能）を置く。「このドキュメントを読む前に何を読むべきか」を learning\_path として定義する。

テンプレートがあるからこそ、AI も人間も同じ品質でドキュメントを生成できます。1,300 以上のドキュメントが一定の品質を保てているのは、この仕組みのおかげです。

**第 3 層: 日英バイリンガル**

すべてのドキュメントを日本語版と英語版のペアで管理しています。日本語版はチームメンバーが読むため背景説明を含む詳細な記述、英語版は AI が読むためトークン効率（AI が処理する情報量の効率）を重視した簡潔な記述。同じ内容でも読者に合わせて最適化しています。

### サービスをまたぐ仕様の管理と検索

各サービスのリポジトリにはコードがあります。しかし**サービス間の取り決め（API の仕様、認証の流れ、データの受け渡し方）は、どのリポジトリにも属さない情報**です。Odin はこの「どこにも置けなかった情報」の置き場所になっています。

たとえば「認証フロー」のドキュメントには、認証サービスのトークン発行処理と、API サーバーの権限検証処理の両方が code\_references で紐付いています。一つのドキュメントから、関連する複数サービスのコードを参照できる構造です。

この構造の上に、2 つのツールを構築しています。

**Odin-RAG**: AI がドキュメントを検索して回答に活用する仕組み（RAG: Retrieval-Augmented Generation）を、MCP（AI ツールが外部データにアクセスするための標準規格）として実装しました。Claude Code から「エラーハンドリングのパターンは？」と聞くだけで、社内の設計ガイドラインと実際のコード実装が返ってきます。

**Bifrost（ビフレスト）**: ドキュメントとコードの整合性を自動チェックするツールです。Claude Code から「認証まわりのドキュメントと実際のコードにずれはないか？」と聞くだけで、仕様と実装を突き合わせてくれます。

### 品質を保つ仕組み

ドキュメントをリポジトリに保存する前に、6 種類の自動チェックが走ります（Pre-commit フック: コード保存前に自動実行されるチェック機構）。文書構文、日本語の表記ゆれ、英語の文体、図の構文、メタデータの形式、日英ペアの存在を自動で検証します。

89 の専門用語を一元管理し、日本語・英語の両方のチェックに自動反映する仕組みも構築しています。「OAuth2.0」と「OAuth 2.0」のような表記ゆれを、人間が意識しなくても自動で統一できます。これは AI が概念を誤認しないための仕組みでもあります。

## 基盤を整えたら、開発がこう変わった

### AI の出力品質を決めるのは、モデルではなくコンテキスト

Odin を作って運用する中で確信したことがあります。

**AI の出力品質は、AI モデルの性能ではなく、与えるコンテキスト（文脈情報）の質で決まる。**

同じ Claude Code に「API エンドポイントを作って」と指示しても、何も文脈を与えなければ教科書的なコードが返ってきます。しかし Odin を通じて社内の API 設計ガイドライン、エラーレスポンスの仕様、既存の類似実装を渡すと、社内のコードベースと一貫性のある実装が返ってきます。

この差は、AI モデルをより高性能なものに切り替えても埋まりません。社内固有の文脈は、自分たちで構造化して AI に渡すしかないのです。

### Claude Code がチームの知識を持つようになった

基盤を整えた結果、Claude Code の振る舞いが変わりました:

* 「認証フローに問題があるのでは？」→ 社内の認証仕様を参照して回答
* 「新しい API を作りたい」→ 既存の API 設計ガイドラインに沿った提案
* 「このエラー処理は正しい？」→ 社内のエラーレスポンス仕様と照合

AI が「一般論」ではなく **「うちの会社のやり方」** で支援してくれる。導入当初にぶつかった壁を、コンテキスト基盤で乗り越えた形です。

### サービスをまたぐ変更を Claude Code が支援する

冒頭で述べた「サービス間の調整コスト」の問題に対して、Claude Code が直接効果を発揮する場面があります。

たとえば「データ API に新しいフィルタ機能を追加したい」と Claude Code に指示すると、まず Odin の検索システムで既存の API 設計ガイドラインと関連ドキュメントを探します。次に Bifrost で関連するサービスのコードとドキュメントの整合性を確認します。

その上で、「API サーバーにフィルタパラメータを追加し、データベースにインデックスを追加し、内部 API を更新する」という**サービスをまたいだ変更計画**を、既存のコードパターンに沿った形で提示してくれます。

従来なら各チームに Slack で確認して回っていた「影響範囲の特定」を、Claude Code が Odin の知識を使って一度に洗い出してくれる。最終判断は人間が行いますが、調査と整理を AI に任せられるようになりました。

### 新メンバーのオンボーディングが変わる

従来、新しくチームに加わったメンバーは「誰に何を聞けばいいかわからない」状態からスタートしていました。Odin と Claude Code の組み合わせにより、「AI Platform のアーキテクチャを教えて」「認証の仕組みはどうなっている？」と Claude Code に聞くだけで、Odin から関連するドキュメントとコードの参照先が返ってきます。先輩エンジニアの時間を使わずに、自分のペースで文脈をキャッチアップできる環境が整いつつあります。

### 設計書を Claude Code が自律生成する

Odin には Claude Code 用のカスタムコマンドが 12 個実装されています。中でも `/architect` コマンドは、機能の説明を入力するだけで包括的な設計書（blueprint）を自律的に生成します。

たとえば `/architect "データセット一覧にサムネイルを表示"` と入力すると、Claude Code は以下を自動で行います:

1. Odin のドキュメントと関連リポジトリのコードを探索
2. 既存の API 設計パターンや過去の設計判断を参照
3. 要件分析（機能要件・非機能要件・制約）を整理
4. 代替案を比較し、選定理由を明記した ADR（設計判断記録）を作成
5. API 設計、データモデル、シーケンス図を生成
6. テスト戦略（ユニット・統合・E2E）を策定
7. タスクを依存関係付きで分解し、実装順序を提示

生成された blueprint は `.architect/` ディレクトリに蓄積されていきます。これが次の設計書を作るときの参照元になる。設計判断が記録として残り、「なぜこの設計にしたのか」をあとから追えるようになります。

人間の仕事は、この設計書をレビューして承認すること。ゼロから設計書を書く作業から、AI が作った設計書の妥当性を判断する作業に変わります。

## AI First 開発へ向けて

### ロードマップ

Odin は完成したわけではありません。改善の余地がまだまだたくさんあります。以下は、これから取り組んでいく方向性です。

**Phase 1: チーム開発基盤** — CI（継続的インテグレーション）での自動チェック、変更リクエストのレビュールール、AI によるレビュー。加えて、コードとドキュメントの乖離（ドリフト）を週次で自動検知し、陳腐化を早期に発見する仕組みを導入する。

**Phase 2: コンテキストエンジニアリング強化** — 文書間の関係を「構造・メタデータ・意味」の 3 層で定義し、AI がより正確に文脈を理解できる構造を構築する。ドキュメント間の依存関係をグラフとして可視化し、コンテキストの鮮度と粒度を自動管理する。

**Phase 3: 開発パイプライン + インサイト自動収集** — 課題チケットの起票から実装・レビュー・統合まで、各段階に品質ゲートを設置し自動化する。さらに、コードレビューのコメントから設計判断の根拠を抽出し、課題の解決過程からトラブルシュートの知見を収集する——業務を遂行する過程で得られたインサイトを自動的に蓄積する仕組みをつくる。

**Phase 4: ナレッジアクセス** — エンジニア以外（プロダクトマネージャー、営業、カスタマーサポート）も、専門知識なしにドキュメントにアクセスできる環境を構築する。

### 基盤は一日にしてならず

米国を中心に、ソフトウェアエンジニアのレイオフが続いています。この動きの背景には、AI がコーディングの生産性を大きく変えるという各社の判断があります。AI First なソフトウェア開発は、もはや一部の先進企業の実験ではなく、世界的なトレンドになりつつあります。

多くの企業が、来るべき AI First 時代に備えて動き始めているでしょう。AI コーディングツールを導入し、開発プロセスへの組み込みを模索している。

しかし、私たちが Odin を作る過程で学んだのは、**AI ツールを入れるだけでは AI First にはならない**ということです。AI に渡せるコンテキスト——社内の設計判断、サービス間の取り決め、運用手順——が構造化されていなければ、AI は一般論しか返せません。

そして、この基盤は一朝一夕では作れません。ドキュメントを整理し、メタデータを付与し、コードとの紐付けを作り、品質チェックの仕組みを整え、用語を統一する。地味で、終わりのない作業です。

AI モデルは日々進化しています。来年にはさらに賢いモデルが登場するでしょう。しかし、「うちの会社の認証フローはこうなっている」「このサービス間のデータの流れはこう設計した」——こうした社内固有の文脈は、どれだけモデルが進化しても、自分たちで構造化して AI に渡すしかありません。

AI 時代に重要になるのは、**ナレッジとコンテキストの民主化**だと考えています。技術的な文脈がエンジニアの頭の中だけに閉じている状態では、AI を活用できるのもエンジニアだけです。基盤が育てば、プロダクトマネージャーや営業、カスタマーサポートも、専門知識なしに技術情報にアクセスできるようになる。AI が「翻訳者」として、技術と非技術の間の橋渡しをしてくれる。そのためにも、AI が参照できる形で文脈を整えておくことが前提条件になります。

Odin は「ドキュメントの置き場所」ではなく、**AI と人間が同じ文脈を共有するための基盤**です。基盤が脆弱なままでは、AI First どころではない。だからこそ、地道に、一歩ずつ積み上げていく。それが私たちの選んだ道です。
