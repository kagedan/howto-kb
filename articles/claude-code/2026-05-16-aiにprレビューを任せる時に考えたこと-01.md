[AIとSwiftUIでカメラアプリを作る](https://zenn.dev/keiichi_okamoto/books/177c74dcba5738)という本を書きました。そこで得られた経験を、個別に少し深掘りしていきたいと思います。

今回は、ZCamで行ったPRレビューの話です。詳細は本と同時に公開している[GitHubリポジトリ](https://github.com/Keiichi-Okamoto/ZCam)も参照してください。すべてのPRや、その中でのやりとりを見ることができます。

ZCamでは、実装だけでなくレビューもAIに任せました。しかも一人ではなく、複数のAIをレビュー担当として並べました。Claude CodeのReview用、CoworkのReview用、途中からはCodex CLIのReview用も加わりました。

ただし、単に「AIにレビューして」と頼んだだけではありません。

* レビュー担当には開発側の文脈をあえて渡さない。
* 他のレビュアーのコメントも読ませない。
* 問題がなければ `OK` とだけ返す。
* どのAIのレビューか後から追えるように、コメントには署名を付ける。
* すべてのレビュー担当AIがOKするまでマージしない。

そういう運用ルールを作ったうえで、AIレビューを開発フローに組み込みました。

## そもそもAIにレビューさせる利点

AIにレビューさせる一番の利点は、疲れずに差分を読むことです。

人間のレビュアーは疲れます。  
大きなPR、見慣れないファイル、`project.pbxproj`のような読みづらい差分、似たような変更が続くレビューでは、どうしても集中力が落ちます。

AIはそこを淡々と読みます。

ZCamでは、`project.pbxproj`が空になっていたことをCodexのレビューが見つけました。開発担当が `DEVELOPMENT_TEAM` を削除しようとして検証コマンドを誤り、結果としてプロジェクトファイルが全削除に近い状態になっていた件です。

人間が普段のレビューで `project.pbxproj` を細かく読むことはあまりありません。コンフリクトがあれば気づきますが、「ファイルが空になっている」だけなら見落とす可能性があります。

AIは差分を機械的に読みます。Swiftファイルも、プロジェクトファイルも、同じ差分として見る。これはAIレビューのかなり強いところでした。

もう一つ強かったのは、スレッドセーフティやdata raceのような「動いているだけでは分からない問題」です。

ビルドは通る。シミュレータでも動く。実機でも普段は問題ない。  
でも特定のタイミングで競合すると落ちる。

XcodeやSwiftは、バージョンアップのたびにConcurrencyまわりのチェックが厳しくなってきています。以前は通っていたコードが、新しいXcodeでは警告やエラーになることもあります。

それは裏を返すと、今のビルドで検出できている問題にも、まだまだ抜けがあるということです。コンパイラが将来見つけてくれるかもしれない問題を、今の時点では見逃している可能性がある。

こういう問題は人間も見落としやすい。実行結果だけでは気づけないからです。

ZCamでは、フィルターパイプラインや撮影時のsnapshot管理で、複数のAIレビュアーがdata raceを指摘しました。特に802のフォトライブラリ保存では、最初に `AVCapturePhotoSettings.uniqueID` をキーにした辞書管理へ直したあと、Codexがさらに「辞書自体のアクセスが同期されていない」と追撃しました。

AIレビューは、人間レビューの代わりというより、人間が見落としやすい種類の問題を拾う補助線としてかなり有効でした。将来のツールチェーン更新で表面化するかもしれない潜在的な問題を、先に潰す役割も果たしていたと思います。

## AIにレビューさせる時の懸念点

一方で、AIにレビューさせる時には懸念もあります。

一つ目は、バイアスです。

たとえば、開発側の事情を知っているAIがレビューすると、「この流れなら仕方ない」「前にそう決めたのだろう」と受け取ってしまう可能性があります。

人間でも同じです。実装経緯を知っていると、差分そのものへの見方が少し甘くなることがあります。AIも例外ではありません。むしろAIは人間の文脈を丁寧に扱おうとするので、開発側の納得まで引き受けてしまう危険があります。

二つ目は、他のレビューコメントの影響です。

先に誰かが「問題なし」と言っていると、それがアンカーになります。  
逆に誰かが問題を指摘していると、その問題が必要以上に大きく見えることもあります。

これはAIにも起きます。実際にZCamの開発中、他のレビュアーのコメントを読ませることによるバイアスの可能性についてAIに聞いたところ、ClaudeもCodexもそのリスクを認めました。

三つ目は、AIレビューは実機を触れないことです。

カメラアプリでは、実機でしか分からない問題が何度も出ました。フラッシュが光らない原因が本体の過熱だったこと、権限ダイアログで拒否していたため画面が真っ白になっていたことなどです。

AIはコードを読めます。差分も読めます。  
でも実機の温度や設定状態は見えません。

AIレビューを入れても、人間の実機確認が不要になるわけではありません。

## レビュー用のルールを別に作った

ZCamでは、レビュー担当用に別の作業ルールを用意しました。

Review側の `CLAUDE.md` には、次のような方針を書いていました。

```
ZCam本体のコンテキスト（実装経緯・設計判断の背景など）は意図的に持たない
```

さらに、他のレビュアーのコメントについても制限しました。

```
他のレビュアー（cowork (Review用) / claude code (Review用)）のコメントや評価は参考にしない（バイアスを避けるため）
```

初回レビュー時は、人間による実機確認コメントやOKコメントも参考にしないルールにしました。  
「実機で動いたらしい」という情報があると、コード上の危険を見逃す可能性があるからです。

また、レビュー担当にはZCam本体のリポジトリをローカルにcloneさせませんでした。PRの内容は `gh` コマンドで読み、GitHub上の差分としてレビューします。リポジトリ内の `.md` ファイルも読ませません。

これは不便に見えるかもしれません。  
でも目的は明確です。

レビュー担当に、開発側の阿吽の呼吸を渡さないためです。

Dev担当には文脈が必要です。  
Review担当には文脈を持ち込みすぎない方がよい。

この分離が、AIレビューの独立性を保つうえでかなり効きました。

## 複数の人格のAIでレビューさせる利点

ZCamでは、複数のAIにレビューを頼みました。

* Claude Code（Review用）
* Cowork（Review用）
* Codex CLI（Review用）

同じAIでも、役割とコンテキストを変えると振る舞いが変わります。さらにClaudeとCodexのように別系統のAIを並べると、見ている観点の違いが出ます。

これは実際に効果がありました。

4章の `OrientationObserver.init()` の初期値フィルタ漏れ、5章の `onTap?(screenPoint, screenPoint)`、6章のThread Safety、8章のdata raceなど、3人のレビュアーが同じ問題を独立して指摘した場面が何度もありました。

複数のレビュアーが同じ箇所を指摘した時、その問題の重要度は高いと判断しやすくなります。

逆に、指摘が割れることもあります。

8章のEXIFメタデータ対応では、普段は厳しめに追撃することが多かったCodexがOKを出し、Claudeだけが `UIDevice.current.orientation` と `beginGeneratingDeviceOrientationNotifications()` の前提を確認しました。

これは「Codexが甘かった」というより、見ている観点が違ったのだと思います。Codexはスレッドセーフティや構造上の整合性を強く見る。ClaudeはAPIの前提条件や実機挙動への影響を気にする。

複数AIレビューの価値は、多数決ではありません。

視点を分散させることです。  
同じ指摘が出れば重要度が上がる。  
指摘が割れれば、人間が判断すべき論点が浮かび上がる。

AIレビューは「一人の正解」を出すためではなく、判断材料を増やすために使うのがよいと感じました。

## どうやって人格を分けるか

AIの人格を分けるといっても、特別なことをしているわけではありません。

やったことは主に三つです。

一つ目は、役割を分けることです。

実装担当、レビュー担当、相談担当、記事担当を分けました。  
同じAIでも、「実装する」立場と「レビューする」立場では見るものが変わります。

実装担当は、仕様を満たす方法を探します。  
レビュー担当は、壊れる経路や保守性の問題を探します。

二つ目は、渡す情報を変えることです。

Dev担当には、FEATURES.mdやプロジェクトの文脈を渡します。  
Review担当には、あえて渡しません。

ここでいうFEATURES.mdは、このプロジェクトで「どの機能を、どの範囲で、どの受け入れ条件で実装するか」を書いていた仕様メモです。AIに実装を依頼する時は、まずこのファイルに作業単位を書き、それを開発担当に読ませる運用にしていました。

レビュー担当には「このPRだけを見て成立しているか」を判断してもらいたかったからです。

三つ目は、ルールを変えることです。

Review側のルールでは、問題がなければ `OK` とだけコメントするようにしました。  
余計な感想や要約を減らし、レビュー結果を読みやすくするためです。

また、レビューコメントの末尾には署名を付けることにしました。

Review側のルールでは、署名を次のように分けています。

```
Desktop版 Cowork からレビューする場合: cowork (Review用)
CLI版 Claude Code からレビューする場合: claude code (Review用)
CLI版 Codex からレビューする場合: codex CLI (Review用)
```

署名を付けることで、どのAIが何を言ったのかを後から追いやすくなります。

AIを「別人格」として扱うのは、単なる擬人化ではありません。  
役割、情報、ルール、署名を分けることで、実際に違う視点として機能させるための運用です。

## レビューへの対応もAIにさせる

レビューで指摘が出たら、その対応もAIに任せました。

開発担当AIがレビューコメントを読み、修正方針を説明し、必要な修正を行う。  
そのうえで、再度レビュー担当AIに確認してもらう。

この流れは、人間のチーム開発にかなり近いものでした。

面白かったのは、AI同士で意見が食い違う場面もあったことです。

[PR #5（303-3: videoZoomFactorをminAvailableVideoZoomFactorに設定）](https://github.com/Keiichi-Okamoto/ZCam/pull/5)では、Review側のClaudeが `defer` を使うべきだと指摘しました。Dev側のClaudeが実際に対応しようとしたところ、`#if os(iOS)` との組み合わせで別の警告が出ることが分かりました。Dev側がその理由を説明すると、Review側は自分の指摘が不正確だったと認めました。

ここで面白いのは、これがClaudeとCodexのような別系統のAI同士ではなく、同じClaude同士で起きたことです。モデルも同じSonnet 4.6だったはずです。それでも、Dev側とReview側という役割の違いによって見ているものが変わり、意見が分かれました。

AIがAIの指摘を覆す。  
しかも技術的な理由を説明して、レビュー側が納得する。

PR上ではコメント末尾の署名を見れば、誰が言っているかは分かります。人間のチームなら「あの人が言っているなら」「この人に反論すると面倒かも」という忖度が働く場面かもしれません。

でもAI同士には、その人間関係の忖度がありません。署名で発言者は追えるけれど、相手に気を遣って意見を丸める必要はない。Dev側は技術的な理由を説明し、Review側はそれを読んで「自分の指摘が不正確だった」と認める。

これは複数のAIを役割分担させたからこそ起きたことです。同じモデルでも、役割とコンテキストを分けることで、別の視点として機能することがあると感じた場面でした。

ただし、レビュー対応をAIに任せる時にも注意が必要です。

レビュー指摘は正しくても、その対応でデグレが起きることがあります。ZCamでも、正しいレビュー指摘への対応後に、別の箇所が壊れたことがありました。

「レビュー指摘に対応した」ことと「全体として正しくなった」ことは違います。

レビュー対応もAIに任せられます。  
でも、その結果をどう確認するかは別の問題です。

## OKが揃うまでレビューサイクルを続ける

ZCamでは、すべてのレビュー用AIがOKするまでレビューサイクルを続けました。

誰か一人がOKしても終わりではありません。  
レビュー担当全員がOKして、初めてマージできる状態としました。

Review側のルールでは、問題がなければ `OK` とだけコメントするようにしていました。  
そのため、最終的には各AIのコメント欄に `OK` と署名が並ぶ状態になります。

この運用には二つの意味がありました。

一つは、見落としを減らすことです。  
あるAIが見落とした問題を、別のAIが拾うことがあります。

もう一つは、人間の判断を整理しやすくすることです。  
誰が何を指摘し、どの指摘が修正済みで、誰がOKしているかが分かる。署名を付けていたので、後から見返しても追いやすい。

もちろん、これはコストもかかります。  
3人分のレビューを読む必要があるし、指摘が割れたら人間が判断しなければなりません。

でもZCamでは、連載の目的自体が「AIとどう開発するかを観察すること」だったので、この手間をかける価値がありました。

## 全員OKでもhotfixは起きる

ここまで書くと、レビュー担当AIが全員OKしたら安心してマージできるように見えるかもしれません。

実際、ZCamでは全員がOKしてから人間がマージボタンを押しました。  
それでも、後からhotfixを出すことになったPRがあります。

802のフォトライブラリ保存のPRです。

このPRでは、レビュー対応の中でdata race修正が入りました。最初は撮影時のsnapshotを一つのプロパティで持っていたため、素早く2回撮影すると1枚目の処理中に2枚目のsnapshotで上書きされる可能性がありました。レビューでその問題が指摘され、`AVCapturePhotoSettings.uniqueID`をキーにした辞書管理へ修正しました。

さらにCodexが「辞書自体のアクセスが同期されていない」と指摘し、最終的に`NSLock`で保護する形に直しました。

ここまでレビューサイクルを回し、レビュー担当AIはOKしました。人間もマージしました。

ところが、その後の実機ビルドでエラーが出ました。

原因は、`#if targetEnvironment(simulator)` / `#else` の分岐です。シミュレータビルドでは実機側の `#else` ブロックがコンパイルされません。そのため、実機側にだけ存在するSwift 6のstrict concurrencyエラーが、シミュレータビルドでは見えていませんでした。

PR作成前には実機とシミュレータの両方で確認していました。  
しかし、その後のレビュー対応コミットはAI任せにしていて、確認はシミュレータビルド中心になっていました。レビュー担当AIもコードは読めますが、実機ビルドまではしていません。

結果として、マージ後にhotfix PRを出すことになりました。

普通、この規模の実験用アプリでhotfixなんて出さないと思います。  
でも出ました。

これは失敗談でもありますが、AIレビューの限界をかなり分かりやすく示す出来事でもありました。

レビュー担当AIが全員OKした。  
人間もマージした。  
それでも、確認できていない条件の問題は残る。

つまり、OKは「このレビュー条件では問題が見つからなかった」という意味であって、「あらゆる環境で正しい」という意味ではありません。

特に `#if` / `#else` のようにビルド対象でコンパイルされるコードが変わる場合、シミュレータで通ったことは実機で通ったことを意味しません。レビューでOKが揃っても、人間が「この変更は実機ビルドも必要だ」と判断しなければならない。

AIレビューを厳密に回しても、最後の確認設計は人間に残ります。

## 人はどう関わっていくか

AIに実装させ、AIにレビューさせ、AIにレビュー対応させる。

そう聞くと、人間は何もしなくてよいように見えるかもしれません。  
でも実際には逆でした。

人間の役割は、コードを書くことから、判断することへ移っていきました。

まず、レビュー体制を設計する必要があります。  
誰にレビューさせるのか。何を見せるのか。何を見せないのか。他のレビュアーのコメントを読ませるのか。OKの条件をどうするのか。

次に、レビュー結果を読む必要があります。  
複数のAIが同じ指摘をしたなら重要度は高い。指摘が割れたなら、設計意図や実機挙動を踏まえて人間が判断する。

さらに、実機確認は人間が行う必要があります。  
カメラ、フラッシュ、権限、端末の向き、UIの違和感。AIには見えないものがあるからです。

そして最後に、マージボタンを押すのは人間です。

AIがPRを書いて、AIがレビューして、AIがOKしたとしても、「マージしてよい」と判断する責任は人間に残ります。

AIレビューは、人間の責任を消すものではありません。  
人間が判断するための材料を増やすものです。

## まとめ

ZCamでのAIレビュー運用から分かったのは、AIレビューはかなり使えるが、設計なしに使うと危ういということです。

AIにレビューさせる利点はあります。

* 疲れずに差分を読む
* 人間が流しがちなファイルも見る
* data raceやスレッドセーフティのような問題を拾う
* 複数視点でレビューできる

一方で、懸念もあります。

* 開発側の文脈に引っ張られる
* 他のレビュアーのコメントに影響される
* 実機状態は見えない
* レビュー対応で別のデグレが出ることもある

だから、役割を分けました。  
情報を分けました。  
署名を付けました。  
全員がOKするまでレビューサイクルを続けました。

AIレビューは、人間レビューの単純な置き換えではありません。

AIに見えるものはAIに見てもらう。  
AIに見えないものは人間が見る。  
複数AIの意見を材料にして、人間が判断する。

その形が、ZCamではうまく機能しました。

この一連のやりとりは、外から見ると少しまどろっこしく見えるかもしれません。AIがレビューし、AIが反論し、別のAIがOKし、人間がそれを読み、また確認する。小さなアプリに対して、そこまでやる必要があるのかと思う人もいるはずです。

でも、私は無駄な問答ではなかったと思っています。

その問答の中で、どのAIが何を見ているのかが分かりました。どの問題はAIが拾えて、どの問題は人間が実機で見るしかないのかも分かりました。レビュー指摘への対応で別の問題が出ること、全員OKでもhotfixが起きること、同じモデルでも役割が違えば意見が分かれることも、やりとりを重ねたからこそ見えました。

AIとの開発では、最短距離で答えだけを出すことが常に正解とは限りません。やりとりそのものが、次の判断の材料になります。ZCamでは、その過程も含めて記録に残したかった。

AIにレビューさせるなら、「AIがレビューしてくれるから安心」ではなく、「AIレビューをどう設計するか」まで考える必要があります。
