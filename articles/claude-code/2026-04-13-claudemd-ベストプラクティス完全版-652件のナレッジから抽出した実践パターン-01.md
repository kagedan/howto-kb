## はじめに

Claude Codeの出力品質は**設定ファイルで9割決まる**。

これは誇張ではなく、Zenn・Qiita・GitHub・YouTube・note・はてなの652記事を収集・分析した[CCIndex](https://ccindex.unravel-workspace.com/)のデータから見えてきた結論です。

本記事では、CLAUDE.mdに関する**29記事のナレッジを統合**し、個人開発からチーム運用まで使えるベストプラクティスを体系化しました。各パターンの引用元を明記していますので、詳細は元記事を参照してください。

## 1. CLAUDE.mdの配置と優先順位

CLAUDE.mdは3階層で配置でき、**下位が上位を上書き**します[2][7]。

| 階層 | パス | 用途 |
| --- | --- | --- |
| グローバル | `~/.claude/CLAUDE.md` | 個人の全プロジェクト共通ルール |
| プロジェクト | `./CLAUDE.md` | プロジェクト固有の設計方針 |
| サブディレクトリ | `./frontend/CLAUDE.md` | モジュール固有の制約 |

**実践パターン：** グローバルには「コーディング規約」「言語設定（日本語で応答）」「禁止事項」を。プロジェクトにはスタック情報・アーキテクチャ・テスト方針を。サブディレクトリにはコンポーネント規約やAPI仕様を配置[2][4]。

## 2. 必須セクション — 最小構成テンプレート

効果の高いセクションを分析した記事群[4][6][7]から抽出しました。

```
# プロジェクト概要
- 名前: [プロジェクト名]
- スタック: FastAPI + SQLite + Vite/React
- 目的: [1行で]

# コーディング規約
- Python: ruff準拠、型ヒント必須
- フロントエンド: TypeScript strict、Tailwind CSS
- テスト: pytest、カバレッジ80%以上

# 禁止事項
- console.logをコミットしない
- anyの使用禁止
- 既存のテストを削除しない

# セッション開始時の動作
1. CLAUDE.mdを読む
2. 直近のgit logを確認する
3. 変更前にテストを実行する
```

**ポイント：** 「セッション開始時の動作」を明記すると、Claudeが毎回コンテキストを拾い上げる精度が上がります[6]。

## 3. 品質を劇的に変える5つのテクニック

### 3-1. 禁止パターンの明文化

「やってほしいこと」より「**やらないでほしいこと**」の方が遵守率が高い。DESIGN.mdとCLAUDE.mdを組み合わせた事例[1]では、Tailwindクラス指定と禁止パターン76項目を列挙することでUI生成の一貫性が大幅に向上しています。

```
# 絶対にやらないこと
- styled-componentsを使わない（Tailwindのみ）
- divのネストを3階層以上にしない
- インラインスタイルを書かない
- !importantを使わない
```

### 3-2. DESIGN.mdとの併用

Googleが提唱するDESIGN.md標準[1a]をCLAUDE.mdと組み合わせることでUI生成の品質が安定します。

```
# デザインシステム
→ DESIGN.md を参照すること

# カラーパレット
- Primary: #FF6154
- Background: #ffffff / #0f172a (dark)
- Text: #21293C / #f1f5f9 (dark)
```

### 3-3. メモリ3階層設計

セッション間で知識を蓄積する仕組みが紹介されています[3]：

1. **CLAUDE.md** — 静的なルール（変更頻度：月1）
2. **CLAUDE.md内のメモリセクション** — Claude Code自身が更新する動的メモ
3. **DOCSフォルダ** — 詳細仕様書・ADR・API定義

### 3-4. Hooks + Skillsとの連携

CLAUDE.md単体では「見るが従わない」問題がGitHub 30件超で報告されています[15]。これはLLMの構造的限界で、Hooksで機械的に強制するのが有効です[8]。

```
# Hooks連携
- PostToolUse: ruff check + pytest実行
- PreToolUse: openapi-check（API変更時）
→ .claude/hooks/ 参照
```

### 3-5. repomixによる初期生成

初見のリポジトリでCLAUDE.mdを作る際、repomixでコードベースをコンテキスト化してから生成させると品質が上がるという検証結果があります[11]。

```
npx repomix . --output context.md
# → このcontext.mdをClaudeに渡してCLAUDE.mdを生成させる
```

## 4. チーム運用 — 「チーム憲法」としてのCLAUDE.md

5人チームで3ヶ月運用した事例[5]と、PR数4倍でも破綻しなかった事例[12]から、チーム利用時のポイントが見えています。

**個人メモ → チーム憲法への昇華：**

* CLAUDE.mdをPRレビュー対象にする[5]
* 階層分離：共通ルール（ルート）+ チーム固有（サブディレクトリ）
* コスト可視化の仕組みを組み込む[5]
* PR数が4倍に増えても破綻しなかったのは、CLAUDE.mdの標準化が効いた[12]

```
# チームルール
- PRは必ず人間がレビューする
- 自動生成コードには [AI] プレフィックスをコミットメッセージに付ける
- CLAUDE.mdの変更はチームレビュー必須
```

## 5. 非エンジニアのためのCLAUDE.md

プログラミング経験がなくてもCLAUDE.mdは使えます。投資方針の記憶[17]、エラー恐怖症の克服[13]、「開発チーム」の構築[9]など、非エンジニア向けの活用事例が増えています。

```
# プロジェクト情報
- 私はプログラミング初心者です
- エラーが出たら、まず「何が起きたか」を平易に説明してください
- コードを変更する前に、何をするか日本語で説明してください

# 段階的サポート [13]
- 最初は「安心モード」：エラーの説明を丁寧に
- 慣れたら「ガイドモード」：選択肢を提示
- 最終的に「自立モード」：最小限のヒントのみ
```

## まとめ

CLAUDE.mdのベストプラクティスを一言でまとめると：

> **短く、具体的に、検証可能に。**

* 35行以下のコアルール + 階層分離
* 「やること」より「やらないこと」
* Hooks/Skillsで機械的に強制
* チームでは「憲法」として運用
