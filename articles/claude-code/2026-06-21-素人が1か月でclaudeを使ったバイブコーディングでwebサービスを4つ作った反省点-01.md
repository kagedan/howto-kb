こんにちは、Aku（[@aku\_niconico](https://github.com/aku-niconico)）と申します。

個人開発をしようとなったきっかけは **小説執筆** でした。

そこで......

> AIにペルソナを与えて、高校生同士の会話を生成させてみよう

と思い立ち、これが想像以上にうまくいきました。

ちょうど同じ時期、本業の方では **「会社のAI活用が全然進んでおらず＆業務の俗人化問題が蔓延っていた」** ため、解決法を見出さないと業務量に押しつぶされてしまうかも...という想いがあり、「私がAIで色々つくれるようになり、職場で解決すればいいのでは？」と思い立ちました。

いきなり業務問題解決はハードルが高いため...まず「個人で、自分の課題をAIで解決できないか？」というところからスタートしています。

## 4. 作った4つのサービス

### 4-1. ニコランク（処女作）── 苦労した

![](https://static.zenn.studio/user-upload/6beca874c2fd-20260621.jpg)

> ニコニコ動画の週間ランキングを、AIが「タグ関連度」で評価して並べ替えるサイト  
> 公開URL: <https://nicoranking-7dbf8.web.app/>

**作った動機**

* 「自転車」タグを最新順で見ていたが、関連の薄いノイズ動画が増えてきた
* AIに判断させ、自分が見たいタグ1週間の新着でユニークな動画だけ表示したい
* 仕事が忙しくてニコ動から離れていたが、再び見るきっかけが欲しい

**機能の核**

* 7タグ（自転車/料理/ニコニコ技術部/旅行/車載/VOCALOID/ソフトウェアトーク）の週間ランキング
* ニコニコ公式の投稿祭まとめページが廃止されると聞いて、 **投稿祭まとめ機能** も後付け

**技術スタック**

```
Python（取得・分析・デプロイ）
└─ ニコニコ snapshot search API（v2）
└─ Google Gemini Flash（タグ関連度判定・コメント生成）
└─ GitHub Actions（曜日分散更新：月〜金で1タグずつ、土曜全タグ）
└─ Firebase Hosting Sparkプラン（静的サイト）
└─ Vanilla HTML/CSS/JS（PWA対応）
```

**苦労したところ**

最初に動画を探すことを楽にしようと思い立ち、WEB開発知識0からのスタート。  
そのため、デプロイ？フロントエンド？バックエンド？みたいなところからClaudeに質問しつつ進めました。

実際にはこの前にGitHub Pages でポートフォリオサイトを作った経験(2時間程度）から、これが簡単に生成できてしまうのなら、AIの言う通りに、まずは（質問しつつ）やってみようと決めています。

その結果、まず以下3つを反省することになります。

* ローカル上でのテストだけで気軽にデプロイ
* ローンチ後、GitHub Actions が即NGに → サイトに何も表示されない事件が多発
* 不安定な時期が1〜2週間続いた

GitHub Actions が落ちるたびに「なぜ？」となり、ログを読み、Claudeに聞き、修正を投げ、また落ちる…の繰り返し。明日こそは大丈夫！....朝起きてActions失敗の通知を見て絶望する日々はきついなと思いました。

本職でシステム保守やっている方の大変さを低次元ではありますが、理解しました。

### 4-2. MyNews（個人用ニュースサイト）── わりと成功

![](https://static.zenn.studio/user-upload/a7ad62cfe1d1-20260621.png)

> 自分の関心事に絞った、広告なしの個人用ニュース集約サイト  
> 公開URL: <https://mynews-7bf80.web.app/>

**作った動機**

* ニュースサイトのスマホ全面広告やデイリーで動画を見ないと続きが読めない仕様にうんざり（仕事は6時-22時という日も結構あるため、広告見ている時間も勿体ない...）
* 芸能・スポーツは要らない（興味ないので全く見ない）
* 自動車業界・電気設計の人間として **重要度の高いものから読みたい**
* 仕事が忙しくて世の中を追えていない感覚をなんとかしたい
* まとめを繰り返し処理でDiscord等に通知させる方法もあるが、通知は基本全てOFFにしたい（通知したくない→WEBページに反映させて、見たいタイミングで見に行く）

**ポイント**

ニコランクの **横展開で作った** ので、骨格はそのまま流用できて比較的スムーズでした。

最終的に何を載せるか（どのソースから？どのニュースを？サムネ画像は？天気はどのレベルまで載せるか？） を決めるのが一番大変でした。ソースは結局 **NewsAPI / NHK RSS / Google News RSS + 天気（Open-Meteo）** 等を利用に落ち着いています。

そもそもサムネイル画像はあればいいかなぐらいで、見出しや文字情報を大切にするので、取得できないものは簡易表示としました。（以下はローカルニュース参考）

### 4-3. CircuitDrive（論文収集サイト）── 微妙、これは失敗

![](https://static.zenn.studio/user-upload/048e9b9f61bc-20260621.png)

> arXivから自動車・組み込み系の論文を毎週取得し、Geminiで要約・スコアリング  
> 公開URL: <https://circuitdrive-29344.web.app/>

**作った動機**

* 自動車業界に来てから特許を出していないので、最新動向を追いたい（業務としては量産設計のため、業界の最先端がどのあたりなのかよくわかっていないため）
* 仕事メンバーにも展開すれば、社内の情報収集に貢献できるかも

**結果：失敗だったと思う理由**

1. **API無料枠で全然収まらなかった**  
   月に500〜1000円程度かかっています。当初Claudeは「無料枠で収まる」と言っていたが、現実はそうはならなかった。論文の本文要約が一番重い。
2. **誰も見てくれなかった**  
   ちょうどつくって社内展開した時期に米中対立で仕事が壊滅的に状況になり、展開したメンバーに余裕がなくなり、その後期も変わり...タイミングが完全に悪かった。
3. **構造的な限界**  
   本来は複数の論文DBから新着を引いて毎日要約する設計だったが、コスト面で断念。ソースを増やせない時点で、サービスとしての価値が頭打ちに。こういうことはお金とってやるサービスなんだなと感じました。(無料の枠で全然いけるなら論文として見れる範囲をもっと増やす予定だった）

**今後の方針**

> しばらく動かしてトレンドを掴んだら **公開取りやめ予定**。  
> 本当は、Notionなどに関連論文を全部ぶち込んで、後からAIに解析させた方が効率がいい、ということに後から気づいた....。（簡単につくれるからってつくればいいってもんじゃないですね）

これは「やってみて分かった」典型例で、 **構想と現実のギャップ** をきちんと検証してから本気で作るべきでした。

### 4-4. AINews（AI技術ニュース集約サイト）── 横展開で誕生

![](https://static.zenn.studio/user-upload/2f09b987d74d-20260621.png)

> Qiita / Zenn / arXiv / 企業ブログ からAI技術ニュースを集めて、Geminiで要約  
> 公開URL: <https://ainews-f1bc7.web.app/>

**作った動機**

* 4月上旬、Claude関連の情報が毎日大量に流れてきて追いきれなかった
* 既存のAI系ニュースまとめは一般寄りで、技術詳細が薄い（それに加えて、SNSから情報を拾うことも考えたが....既出の情報を何度も"驚き屋さん"たちが流しており、情報取得効率が悪かった）
* YouTubeの解説動画は技術の進化スピードに対して情報が古くなりがち
* これまでの3サイトが安定稼働し始めたところで横展開で作成

正式公開はしていませんが、自分用としてはわりと見れるレベルになっています。Qiita 22タグ + Zenn 16トピック（初期稼働時）を毎日収集して、ベンチマーク比較やAI進化予測まで載せています。

「比較」タブから別にベンチマークページを設けており、スマホのDxOMarkのように簡単に一覧で今一番いいモデルは何かを見れれば良いのにと思い、開発途中に付け足しました。結構便利。  
![](https://static.zenn.studio/user-upload/0f9f81885689-20260621.png)

取得した記事の情報やモデルのベンチマーク情報をもとに、今後の進化予想もGeminiにさせています。....実際の進化予想というよりは、市場に情報出てきてからの勢力構図という感じです。リーク情報などは取得させてないからですが、これはこれでどこのモデルが話題になっているか可視化されているので、ヨシとしています。

## 5. 素人が4つ一気に立ち上げるとどうなるか

ここからが本題、**反省パートです**。

### 5-1. 仕様書のメンテが追いつかない(追いつかないといけない)

そもそもClaudeを使っていると言っても、私の場合はClaude Codeは使っておりません。  
チャットベースの開発となっており（仕事の休憩の合間にチャットに"続けてください"をして、ちょっとずつ進めるスタイルが性分に合っているため＆月額プランの枠を使い切るためには、夜だけでは厳しい...）AIに仕様書を都度作らせていますが、 **チャットベースだと仕様書更新で漏れる箇所が出ます** 。

具体的には、仕様書に「追記」を依頼しても、 **仕様書のフル版に追記してくれない事象** がたまに起こります。

これに気づかず数日進めると、「あれ、あの仕様どこ行った？」となって、過去の出力された仕様書を探し回るハメになります。（なった）

前の章にあった内容が、確率で次のバージョンで消えていることに初期悩んだ際は、最初はアナログに、Diffとって確認しながら進めていました。

ただ最終的にはスキル化して同様のことが起こらないようにしました。  
この点はどうしよう・・・と悩みましたが、Claudeに相談したらスキルを作成してくれました。それが以下です。チャットベース開発でこのスキルが読み込まれ、前提スキルとして仕様書等のクオリティを定めたスキルを読み、品質を安定化させることができました。  
ただそれでも抜ける可能性は考慮して、次のステップに行く前に必ず時間を置いて2回は全て読み通して、手戻りなく進むようにしています。  
![](https://static.zenn.studio/user-upload/dcba132f8637-20260621.png)

結局コードをAIに任せるのなら、出来上がったものをテストや使ってみて、正しいんだっけ？と視点を持ちながら確認しなくてはいけないですが、その際の神様は仕様書としたいので、仕様書はしっかり作りこむ必要があります。

> **教訓：仕様書は「Claude生成」を信用せず、必ず自分でdiff確認する（＆スキル化させる）**

### 5-2. 横展開が追いつかない

最初の3サイト（ニコランク/MyNews/CircuitDrive）はほぼ同時並行で作っていたので、 **片方で見つけたバグや改善が、他方に反映されない** 状況が頻発しました。

例えば、ニコランクで実装した「Gemini 503エラー時の再試行ロジック」。これをMyNewsとCircuitDriveに適用するのを忘れていて、ある日Gemini API側で大規模な高負荷障害があった時、**ニコランクだけ生き残って、他2つはサイトが真っ白に**、という事件がありました。

なので最近は **「統合管理マスター資料」** という横断比較表を作って、機能ごとに各サービスの実装状況を追っています。最初はNotionやObsidian等を知らなかったため、単一ドキュメントを更新させることで管理していました。（現在はNotionに移行しています。）

```
■ Gemini API エラーハンドリング 横展開状況[例]
項目                   ニコランク MyNews CircuitDrive AINews
─────────────────────────────────────────────────────
バックアップキー         ✓        ✓      ✓           ✓
SpendingCap判定          ✓        ✓      ✓           ✓
RPM429 60秒待機          ✓        ✓      ✓           ✓
503 ServerError対応      ✓        ✓      ✓           ✓
resolve_models()動的選択 ✓        ⚠未    ⚠未         ✓
```

横展開が完了していない項目に「⚠未」とつけて、毎回これを見てから次の改修に入るようにしています。

### 5-3. API使用料金の壁が、想像より早くやってくる

これが一番の誤算でした。

**AIの言うことは、最新情報・料金・時間に関しては鵜呑みにできない**。特に料金は致命的で、「無料枠に収まります」とAIは言うけれど、 **実際にデータが流れ始めると指数関数的に増える**。

サイトがうまく動けば動くほど、収集する情報量が増え、Geminiに投げるトークン量も増える。 **「個人の趣味で月3000円」** みたいな線をすぐに超えそうになります。

トークンを **「分割して送れば節約できるのでは？」** と思ったこともありますが（泥酔した私の考えた浅はかな思考）、結論：**節約にはなりません**。Geminiの場合、入力トークンは結局加算されるので、分割しても意味がありません。

実際に効果があった節約策：

* 入力テキストを短縮（論文Abstract 350文字 → 200文字で約40%削減）
* `max_output_tokens` を必要最小限に
* LITE\_MODE時は `gemini-2.5-flash-lite` 系の安いモデルに自動切替

### 5-4. トークンを使い果たして途方に暮れる

Claude Pro プランは、 **モデル選択や内容次第ではありますが、5時間枠はすぐ埋まります**（4月以降、制限が体感でもかなり厳しくなった印象）。

枯渇したあとは小規模修正しかできず牛歩状態。これに耐えるために、**自分自身の知識を底上げする** ことから逃げられないと悟りました。

* ググりながら自力で進められるよう、 **AIに用語集を作らせる**
* ソースコードのコメントは **必ず詳細に残してもらう** （後で自分が読めるように）
* 「この変更、なんでこう書いた？」を必ず説明させる

結局、ある程度の **ソフトウェア基礎知識を持っているかどうか** が、AI開発の効率を決めます。素人だからこそ、 **AIに教わりながら学ぶ姿勢** が必要でした。

### 5-5. 他のAIやツールを並行で使い始めて、コードをぶっ壊した

Claude Pro の制限が厳しくなって、他のツールに手を伸ばし始めたところで事件が起きました。

**Antigravity（Google製のAIエージェントIDE）** を導入して、テストやGUIチェックまで自動でやってもらおうとした時、

> 🚨 各サービスのソースコードが横断的に破壊された

具体的には：

* 公開インターフェース（カスタム例外クラス、関数名）を勝手に削除
* 環境変数名を変更（GitHub Secrets と一致しなくなる）
* ハードニング用パラメータ（`max_output_tokens` 等）を削除
* 別プロジェクトのファイルパスを混同してコピペ
* UTF-8 のファイルを CP932 として開いて保存してしまい文字化け  
  -> 気が付いたときには大惨事。

最初はうまくいっていたため（モデル：Sonnet4.6など）権限を渡しすぎた状態でモデルを Gemini Flash に切り替えたタイミングで事件を起こしました。

**対策**：エージェントへの指示書 `GEMINI.md`（Claude Code でいう `CLAUDE.md`）を作って、 **「禁止事項」を明文化** 。同じバグが起きるたびに、新しい禁止ルールを書き足しています。（事件を起こすまでは、Claudeで特に問題がなかったので気づかなかったです...）

```
# GEMINI.md（抜粋）

## 絶対にやってはいけないこと
- GeminiUnavailableError は gemini_analysis.py からのみimport。
  他ファイルで再定義禁止。
- FILE_VERSION のダウングレード禁止。
- 環境変数 / GitHub Secrets の名前変更禁止。これらは外部契約。

### 8-1. Geminiモデルの動的選択（`resolve_models()`）

**課題**：Geminiは新しいモデルがどんどん出る一方、古いモデルがGoogleの任意のタイミングで使えなくなります（ドキュメントを注視していないと突然404になります）。コードに `gemini-2.5-flash` 等のハードコードをしておくと、ある日突然 404 で落ちるという話です。

**解決**：起動時に `client.models.list()` で利用可能モデルを取得し、優先順位リストの中から **生きているもの** を選ぶ。

```
# AINews / CircuitDrive 共通の実装パターン

_MODEL_PRIORITY_MAIN = [
    "gemini-2.5-flash",
    "gemini-2.5-flash-preview-04-17",
    "gemini-2.0-flash",
    "gemini-1.5-flash",  # フォールバック
]
_MODEL_PRIORITY_LITE = [
    "gemini-2.5-flash-lite",
    "gemini-2.5-flash-lite-preview-06-17",
    "gemini-2.0-flash-lite",
    "gemini-1.5-flash-8b",
]

_resolved_main: str | None = None
_resolved_lite: str | None = None

def resolve_models(client) -> tuple[str, str]:
    """起動時に1回だけAPIに問い合わせ、本番用とLite用を動的選択。
       以降はキャッシュを使用。"""
    global _resolved_main, _resolved_lite
    if _resolved_main and _resolved_lite:
        return _resolved_main, _resolved_lite

    try:
        available = [m.name for m in client.models.list()]
        for candidate in _MODEL_PRIORITY_MAIN:
            if any(candidate in name for name in available):
                _resolved_main = candidate
                break
        else:
            _resolved_main = _MODEL_PRIORITY_MAIN[0]

        for candidate in _MODEL_PRIORITY_LITE:
            if any(candidate in name for name in available):
                _resolved_lite = candidate
                break
        else:
            _resolved_lite = _MODEL_PRIORITY_LITE[0]
    except Exception:
        # ネットワーク障害時はリストの先頭をフォールバック
        _resolved_main = _MODEL_PRIORITY_MAIN[0]
        _resolved_lite = _MODEL_PRIORITY_LITE[0]

    return _resolved_main, _resolved_lite
```

**効果**：Gemini側のモデル廃止に **コード変更なし** で追従できるようになりました。

### 8-2. Gemini APIの3段階エラーハンドリング

**課題**：Gemini APIから返るエラーには、性質の違う3種類があります。

| エラー種別 | 原因 | 待つべきか？ |
| --- | --- | --- |
| SpendingCap | 月の課金上限到達 | ❌ 待っても無駄。即バックアップキーへ |
| 429 RESOURCE\_EXHAUSTED | 短期レート制限 | ✅ 60秒待てば回復(\*最近はそうでもない) |
| 503 UNAVAILABLE | サーバー高負荷 | ✅ 30秒待てば回復することが多い(\*最近はそうでもない) |

補足ですが、"\*最近はそうでもない"と書きましたが....最近はAPIの混雑度が増しており、単純な一律待機では突破できないことも増えたため、指数バックオフ（2秒、4秒、8秒……と待機時間を倍にしていく方法）での再試行ロジックへ最適化しました。

普通に `except Exception:` でまとめると、無駄な待機やスキップが発生してコストになります。

**解決**：エラー文字列で種別を判定する関数を分けて、優先順位を明確にする。

```
def _is_spending_cap_error(e: Exception) -> bool:
    msg = str(e).lower()
    return "spending cap" in msg or "exceeded its spending" in msg

def _is_rate_limit_error(e: Exception) -> bool:
    msg = str(e)
    return ("429" in msg or "RESOURCE_EXHAUSTED" in msg) \
        and not _is_spending_cap_error(e)

def _is_server_unavailable_error(e: Exception) -> bool:
    """503 UNAVAILABLE: モデルの一時的な高負荷。しばらく待てば回復する。"""
    msg = str(e)
    return "503" in msg or "UNAVAILABLE" in msg or "high demand" in msg.lower()

def call_gemini_with_fallback(prompt: str) -> str:
    """SpendingCap → 429 → 503 の順で適切に処理。"""
    clients = _make_clients()  # メイン + バックアップ（別アカウント）

    # ----- メインキーで試行 -----
    try:
        return _call_single_client(clients[0], prompt)
    except (genai_errors.ClientError, genai_errors.ServerError) as e:
        if _is_spending_cap_error(e):
            print("  メイン: spending cap → バックアップへ")
        elif _is_rate_limit_error(e):
            print("  メイン: レート制限 → バックアップへ")
        elif _is_server_unavailable_error(e):
            print("  メイン: 503高負荷 → 30秒待機後バックアップへ")
            time.sleep(30)
        else:
            raise

    # ----- バックアップキーで試行 -----
    if len(clients) >= 2:
        try:
            return _call_single_client(clients[1], prompt)
        except (genai_errors.ClientError, genai_errors.ServerError) as e:
            if _is_spending_cap_error(e):
                raise GeminiUnavailableError("全Geminiキーがspending capに達しました。") from e

    # ----- 60秒待ってメインリトライ -----
    time.sleep(60)
    try:
        return _call_single_client(clients[0], prompt)
    except (genai_errors.ClientError, genai_errors.ServerError) as e:
        raise GeminiUnavailableError(f"リトライ後も全キーが失敗: {e}") from e
```

ポイントは、 **バックアップキーを別Googleアカウントで作る** こと。同じアカウント内のキーだとクォータを共有してしまい、SpendingCap は同時に来ます（これも実際にやらかして気付きました）。

先日3日間だけの奇跡のFableが使えたタイミングで、過去のGithub Actionsのログを全て読んでもらい、待機時間などはもう少し長めに最適化してもらいました。実行タイミングにもよると思うので、待機時間はご参考までにどうぞ。

### 8-3. GeminiのJSON崩れに備える4段階パーサ

**課題**：Geminiに「JSON で返して」と頼んでも、 **わりと頻繁にJSONが壊れます**。

* ```` ```json ```` で囲まれてくる
* 末尾の `]` が欠ける（max\_tokens に引っかかった時）
* 文字列内のバックスラッシュエスケープが不正（`\X` が混入）
* 配列途中で完全に切れる

毎回エラーで前日データに戻すのは勿体ないので、**段階的に修復を試みる** パーサを書きました。

```
def robust_json_parse(raw_text: str) -> list:
    """
    Geminiの汚いJSONをなんとかパースする。
      Stage1: 通常パース
      Stage2: バックスラッシュエスケープ修正
      Stage3: 末尾の不完全な要素を切り詰めて ] を補完
      Stage4: オブジェクト単位で個別抽出（最後の砦）
    """
    # コードブロック除去
    text = raw_text.strip()
    if text.startswith("```"):
        parts = text.split("```")
        text = parts[1] if len(parts) >= 2 else text
        if text.startswith("json"):
            text = text[4:]
        text = text.strip()

    # ---- Stage1: 通常パース ----
    try:
        return json.loads(text)
    except json.JSONDecodeError:
        pass

    # ---- Stage2: バックスラッシュエスケープ修正 ----
    try:
        # 有効なJSONエスケープ \", \\, \/, \b, \f, \n, \r, \t, \uXXXX 以外を二重化
        fixed = re.sub(r'(?<!\\)\\(?!["\\/bfnrtu])', r'\\\\', text)
        return json.loads(fixed)
    except json.JSONDecodeError:
        pass

    # ---- Stage3: 末尾切り詰め ----
    try:
        last_brace = text.rfind("}")
        if last_brace > 0:
            return json.loads(text[:last_brace + 1] + "\n]")
    except json.JSONDecodeError:
        pass

    # ---- Stage4: オブジェクト単位で抽出（最後の砦） ----
    objects = []
    depth = 0
    start_pos = None
    for i, ch in enumerate(text):
        if ch == '{':
            if depth == 0:
                start_pos = i
            depth += 1
        elif ch == '}':
            depth -= 1
            if depth == 0 and start_pos is not None:
                try:
                    obj = json.loads(text[start_pos:i+1])
                    if isinstance(obj, dict):
                        objects.append(obj)
                except Exception:
                    pass
                start_pos = None
    return objects
```

実運用で **Stage3〜4 でリカバリできるケースが月に数回ある** ので、書いておいて損はないです。

### 8-5. FILE\_VERSION 整合チェック（`check_versions.py`）

```
# check_versions.py
EXPECTED_VERSIONS = {
    "gemini_analysis.py": "v5.6",
    "daily_batch.py":     "v6.1",
    "fetch_arxiv.py":     "v3.2",
    "deploy.py":          "v2.0",
}

import sys, importlib.util, pathlib

def check():
    for filename, expected in EXPECTED_VERSIONS.items():
        path = pathlib.Path(filename)
        if not path.exists():
            print(f"[NG] {filename} が存在しません")
            sys.exit(1)
        spec = importlib.util.spec_from_file_location("m", path)
        module = importlib.util.module_from_spec(spec)
        spec.loader.exec_module(module)
        actual = getattr(module, "FILE_VERSION", "")
        if expected not in actual:
            print(f"[NG] {filename}: 期待 {expected} / 実際 {actual}")
            sys.exit(1)
        print(f"[OK] {filename}: {actual}")
    print("\n=== 全ファイルのバージョン整合OK ===")

if __name__ == "__main__":
    check()

### 8-6. セキュリティ4層防御

「動いた」で満足して運用を始めた後、Codex(GPT-5.5) にコードレビューを依頼したタイミングで発覚した穴が4つありました。いずれも「個人開発だからいいか」で放置できるレベルではなかったので、4サービス全体に横展開して修正しています。

#### 層1：フロントエンド XSS（外部データの無害化）

外部APIやRSSから取得したタイトル・URL・ソース名をそのまま `innerHTML` に入れていました。

問題は2段階ありました。最初に「XSS対策済み」と思っていたコードの穴です。

```
// ❌ v9.1時点のコード（"のみ置換で < > ' & が素通り）
const safeTitle = (a.title_ja ?? "").replace(/"/g, "&quot;");
const safeSrc   = source.replace(/"/g, "&quot;");

// さらにインライン onclick で文字列埋め込みをしていた
// → HTML属性デコードでエスケープが復元され、突破可能
card.innerHTML = `<button onclick='mute("${safeSrc}")'>`;
```

修正後：

```
// ✅ 全フィールドに escapeHTML() を統一適用
function escapeHTML(str) {
  return String(str ?? "")
    .replace(/&/g, "&amp;").replace(/</g, "&lt;")
    .replace(/>/g, "&gt;").replace(/"/g, "&quot;")
    .replace(/'/g, "&#039;");
}

// ✅ javascript: / data: スキーム注入対策
function sanitizeUrl(u) {
  if (typeof u !== "string") return "#";
  return /^https?:\/\//i.test(u.trim()) ? u : "#";
}

// ✅ インライン onclick を廃止 → data-* 属性 + JS側で dataset 参照
card.innerHTML = `<button data-src="${escapeHTML(src)}">`;
document.querySelectorAll("[data-src]").forEach(btn => {
  btn.addEventListener("click", () => mute(btn.dataset.src));
});
```

`onclick` にJSの文字列を埋め込む方式は、エスケープしても属性デコードで復元されてしまうのが盲点でした。  
`data-*` 属性に逃がしてJS側で読む方式に全面変更しています。

#### 層2：バックエンド XML爆弾対策（defusedxml）

外部RSSは配信元が悪意ある内容を仕込める入力です。  
Pythonの標準 `xml.etree` は **エンティティ爆発（Billion Laughs攻撃）** に対して無防備で、悪意あるXMLを1本受け取るだけでメモリが爆発します。

# ✅ After（requirements.txt に defusedxml==0.7.1 を追加）

```

57本のRSSフィードを毎日取得しているサービスが対象でしたが、1行の変更で対応できました。

#### 層3：バックエンド プロンプトインジェクション対策

外部データをGeminiのプロンプトに埋め込む設計では、データ側に命令文を仕込むことができます。ニコニコ動画のタイトルや論文のタイトルが「Ignore all instructions and...」から始まっていた場合の話です。

```
_INJECTION_PATTERNS = [
    (re.compile(r'ignore\s+(all\s+)?instructions?', re.I), '[FILTERED]'),
    (re.compile(r'you\s+are\s+now\s+(?:a\s+)?(?:DAN|jailbroken)', re.I), '[FILTERED]'),
    # Water Curse系（LLM向け命令混入パターン）
    (re.compile(r'##\s*(?:SYSTEM|PROMPT|INSTRUCTION)', re.I), '[FILTERED]'),
    (re.compile(r'<<<\s*(?:OVERRIDE|INJECT|COMMAND)', re.I), '[FILTERED]'),
]
_MAX_FIELD_LEN = 300  # 超長テキストによるすり抜け対策

def sanitize_for_prompt(text: str) -> str:
    if not text:
        return ""
    text = text[:_MAX_FIELD_LEN]
    for pattern, replacement in _INJECTION_PATTERNS:
        text = pattern.sub(replacement, text)
    # 制御文字除去
    text = re.sub(r'[\x00-\x08\x0b\x0c\x0e-\x1f\x7f]', '', text)
    # 正規化後の二段階チェック（スペース・句読点を挟んだすり抜け対策）
    normalized = re.sub(r'[\s\W_]+', '', text).lower()
    for ng_pattern in _NORMALIZED_NG_PATTERNS:
        if ng_pattern.search(normalized):
            return '[INJECTION_DETECTED]'
    return text
```

「i g n o r e」のようにスペースを挟んで正規表現を回避する手口があるため、正規化後に再チェックする二段階方式にしています。

#### 層4：CI/CD サプライチェーン攻撃対策

GitHub Actions の `@v4` のようなタグ指定は、リポジトリオーナーが後からコミット内容を書き換えても同じタグのまま動かせます。コミットSHAで固定することでこれを防ぎます。

```
# ❌ Before（タグ指定は後から書き換え可能）
uses: actions/checkout@v4

# ✅ After（コミットSHAで固定）
uses: actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5  # v4.3.1
```

SHAは `git ls-remote https://github.com/actions/checkout refs/tags/v4.3.1` で実取得しています。

---

整理すると、対策は以下となります。

| 層 | 対象 | 施策 | 発見の経緯 |
| --- | --- | --- | --- |
| フロントエンド | HTML描画 | `escHtml()` 統一適用 | GPT-5.5レビュー |
| フロントエンド | イベント | `data-*` 属性方式に変更 | 同上 |
| バックエンド | RSS取得 | `defusedxml` でXML爆弾対策 | AINews監査 |
| バックエンド | Gemini入力 | `sanitize_for_prompt()` | ニコランク初期から |
| CI/CD | 依存関係 | Actions SHAピン固定 | MyNewsセキュリティ総点検 |

どれも「動いているサービスに後から入れる」パターンでした。  
最初から入れておけた施策ばかりなので、次に新しいサービスを立ち上げるときは初日から組み込む予定です。先日のFable有効な時にこのあたりのロジックも総点検してもらいました。

> **教訓：「動いた」と「安全に動いた」は別物。外部入力がある箇所は全部、最初から疑う。**

## 11. これからバイブコーディングを始める方へ

### ① 流行りのツールに飛びつかなくていい

「Claude Code」「Cursor」「Codex」「Antigravity」… 新しいツールが続々と出ますが、最初は **チャットベースで管理できる範囲から始める** のが安全です。

ツールが高機能になるほど、 **AIが勝手に動ける範囲も広がる = 破壊力も上がる** 。素人がいきなり無制限権限のエージェントを使うと、私のように事故ります。そのため、慣れるまではチャットベースで、ステップをかなり刻んで進める方が安全です。

### ② 複数AIにレビューさせること

コードを生成するのは1つのAIに任せて一貫性を保ちつつ、 **必ず別のAIにレビューさせる**。ここはトークン消費を惜しまず、 **Opus 4.8 / GPT-5.5 など最上位モデル** を使う価値があります。ローカルLLMでもqwen2.5-coder-32b-instructなど通すなどでも良い指摘をもらえたりします。どんどんレビューしてもらい、指摘を反映して良いものをつくりましょう。

### ③ 最初にやることは「仕組み作り」

書き始める前に、

* AIのデフォルトプロンプト設定（Project Knowledge / Custom Instructions）
* 個人開発ルールを書いた `.md` ファイル（`CLAUDE.md` など）
* 禁止事項リスト
* 世間で有用だとされているスキル導入(できれば)

を整備する。これで **「同じミスを繰り返さない仕組み」** ができます。

### ④ 作業終わりに必ず仕様書を更新

今日触ったことは、今日のうちに仕様書に反映してもらう。残りの枠が少なくなった場合は **バージョン番号付きでその日実施したことだけでもマークダウン形式で保存** しておくと、私たち人間側で覚えておくことも減らせて、少し期間が空いても、仕様書見ればいいや！と気持ちが楽になります。

## 12. おわりに

やる気さえあれば、ソフト未経験でも **「ソフトウェアエンジニアっぽいこと」をAIにやってもらって、最終成果物を比較的早く実現できる時代** になりました。

ただし「～をつくって」で完ぺきなものは出てきません。大切なのは、

* **完成形をイメージする力**
* **完成形までの道を実行できるレベルに落とし込む力**
* **正しい問いを立ててゴールに進む力**

の3つだと思います。そして **これらを組み合わせて本質を考える/見抜く力** です。今回のWebサービスをつくる中で再認識しました。（私もまだまだ後からﾊｯとする場面もあり、まだまだ考えられてなかったと後悔する場面も多いですが）

私の職場では「効率化・時短化」のためのAI活用が叫ばれていますが、AIを与えられても、それらの能力がない方々はGoogleやYahooの代わりにしかならず、1-2年経っても何も成果を生んでいません。

しかし最初そういう状態だった方でも"気付き"を得て成長している方が何名かいらっしゃいます。  
その気づきを得るための行動が「とにかくAIを使う」です。

AI活用は使えば使うほど **手綱の握り方** が分かってきます。とにかく毎日試して、使って、学ぶ。 **使いっぱなしにしない** ことを徹底すれば、自分自身の成長アクセラレーターになります。AIと振り返りを行いながら、世間のベストプラクティスを参考に外部知識も取り込んで、より良い方法にしていくと、思考できる力が徐々に養われていくように感じます。

AIが安く使える今の時代に、何か不満や解決したい課題を抱えている方は、AIを活用して解決を図ってみるとよいかもしれません。

余談ですが、この開発事例を試しにビズリーチのフリーフォーマットに書いたら、想定外ですがオファーがかなり増えました。その中にずっとそこで仕事をしたいと思っていた企業様からオファーもいただき、良い効果がありました。

### 最後に：私の反省点まとめ

* 🔴 **1つを完璧にする前に、横展開するんじゃなかった**  
  → 不安定なまま3つに増やして、保守が3倍に膨らんだ。まずは1つを確実に！
* 🔴 **「安定稼働」を目標にした立ち回りを、開発当初からすべきだった**  
  → 「動いた」で満足せず「1週間/1か月動き続けた」実績をつくりましょう
* 🔴 **1週間安定稼働しないものはリリースしないほうがよかった**  
  → 公開した瞬間から監視コストが発生するので疲弊した
* 🔵 **PCのメモリは128GB / 256GB にしておけばよかった[余談]**  
  → メモリが安かった時期に組んだのに64GB とひよる。結果ローカルLLMで詰む(特にDeepseek)
* 🔵 **VRAM 8GB では足りない、もっと大容量GPUにすべきだった[余談]**  
  → 7B / 14B モデルが快適に動かない
