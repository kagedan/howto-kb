## 何が起きたか

Googleの検索結果に表示される「AI Overviews」(AI要約機能)が、**事実と異なる内容を生成し、それが個人や企業の名誉・信用を傷つけたとして訴訟に発展**した。ドイツの裁判所はこの件について、AI Overviewsが生成した誤った記述について**Googleが法的責任を負うとの判断を下した。**

この判決で特に実務上重要なのは、**Googleの防御ロジックが通らなかった点**だ。GoogleはAI Overviews上に「自動生成されるサマリー機能には内容に誤りが含まれる可能性があるため、利用者自身が内容を確認すべき」という趣旨の免責事項(disclaimer)を表示しており、これによって自社の法的責任は免除されると主張した。

しかし裁判所はこの主張を退けた。「免責事項を貼っておけば、生成物の内容に対する責任を丸ごと利用者に転嫁できる」という理屈は認められなかったということだ。これはAIプロダクトを展開する事業者にとって重要な意味を持つ。**UIの片隅に注意書きを置くだけでは、出力内容そのものの品質・正確性に対する責任からは逃れられない、という判断を示した異例のケース**だからだ(ただし本件はミュンヘン地裁の仮処分であり、Googleは控訴の意向を示している。確定判決ではない点には留保が必要)。

## 「AIエージェント=企業の代理人」論

この判決の法的ロジックの核心は、**AIエージェントを企業が雇用する「人間の執筆者」と同列に扱っている点**にある。

企業がライターを雇い、そのライターが誤った記事を書いて公開した場合、企業は「ライターが書いたことなので当社は関知しない」とは主張できない。雇用主として、成果物の内容について一定の管理・監督責任を負うのが通常の理解だ。

今回の判決は、この構図をそのままAIエージェントに当てはめた。AIが生成した出力は、それを開発・提供する企業が代理人(エージェント)として送り出したコンテンツであり、**免責事項の有無にかかわらず、企業自身の発言・記述と同等に扱われる**、という考え方だ。

言い換えると、**「AIが自動生成したので当社の見解ではない」という切り離しのロジックそのものが、法的に成立しなくなりつつある**ということになる。これはRAGシステムやエージェントを組み込んだプロダクトを提供する開発者にとって、無視できない前提の変化だ。

## 誤りの技術的発生源

なぜAIはこうした誤りを生成するのか。技術的には主に2つの経路がある。

1つ目は**RAG(Retrieval-Augmented Generation)における情報源の問題**だ。AI Overviewsのような機能は、検索結果からドキュメントを取得し、それを要約・統合して回答を生成する。この取得(Retrieval)段階で、以下のような要因が誤情報の混入を招く。

2つ目は**Agentic Loop(自律的な推論・行動ループ)における暴走リスク**だ。エージェントが「検索→評価→生成」のループを繰り返す設計になっている場合、**初期段階で誤った前提や不正確な中間結論が生まれると、後続のループでその誤りが訂正されずに増幅されるケース**がある。人間のレビューが介在しないループ構造では、誤りが自己強化的に固定化されてしまう。

要するに、単発のプロンプト誤答ではなく、情報取得パイプラインとループ構造そのものに誤り発生の余地が組み込まれている、という点が今回の事案の技術的背景にある。

## 開発者への実務インパクト

この判決がRAGシステムやAIエージェントを実装・運用する開発者に与える実務上の影響は大きく3つに整理できる。

### 1. 出力検証の実質的な義務化

**「免責事項を出せば良い」という運用は法的な防御として機能しなくなった**。生成物の正確性そのものを担保する仕組みを、システム側に組み込む必要性が高まる。

日本の「AI事業者ガイドライン」も同方向の規定を置く。RAGによるハルシネーション抑制・出力根拠の透明性向上を明記した上で、出力品質は運用後に変動しうる特性を踏まえた事後検証記録の保存と品質説明義務をAI提供者に課している。第1.2版(2026年3月)では自律型AIエージェント誤処理時の検知・停止・再発防止手順の事前策定も要求 — 「免責事項だけでは済まない」という本判決の帰結と方向性が一致する。

### 2. 監査ログ・トレーサビリティの要求

ある出力がなぜ生成されたのかを事後的に説明できる必要が出てくる。どの情報源を参照し、**どのような推論ステップを経て最終的な回答に至ったのかを追跡できるログ設計が、コンプライアンス上の要件になっていく可能性が高い**。

こちらについても「AI事業者ガイドライン」に同様の要求を置く。共通指針「アカウンタビリティ」でトレーサビリティ確保を求め、開発過程・入出力・推論過程・判断根拠等のログ記録・保存を事業者に課す。第1.2版(2026年3月)ではリスクレベルに応じた人間関与の目安も示されている — 高リスクは事前承認、中リスクは事後ログ、低リスクはサンプリング確認、という区分だ(この3区分は経産省一次資料の明文ではなく解説記事による整理である点は留意)。同ガイドラインは非拘束のソフトローで法的罰則はなく、裁判所が責任を認定する今回の判決とは性質が異なる。

### 3. 人間によるレビューポイントの再設計

完全自動化されたAgentic Loopに、どこで人間のチェックポイントを挟むかという設計判断が、単なるUX上の話ではなく法的リスク管理の話になる。

この設計判断を人力運用だけで担保するのは非現実的で、承認ゲート・kill switchをプロトコル/ライブラリ層に落とし込む動きが実務では先行している(詳細は後述の対策アプローチを参照)。

## 対策アプローチ

**技術的な対策として現実的なのは、生成物を「出す前」に検証する層を挟む**ことだ。

1つは**LLM-as-a-Judgeによる自己検証**だ。生成した回答を別のLLM(あるいは同一モデルの別セッション)に「この回答は与えられたソースと矛盾していないか」を判定させ、矛盾が検出された場合は出力を差し止める、あるいは確度の低さを明示するフラグを立てる。

もう1つは**MCP Server経由での出力検証プロトコルの整備**だ。エージェントが外部ツールやデータソースにアクセスする際、MCPのレイヤーで「どの情報源から何を取得したか」を構造化して記録し、生成結果とソースの対応関係を機械的に検証できるようにする。これにより、Retrieval段階の誤りとGeneration段階の誤りを切り分けて追跡できる。

verify\_output.py

```
import json

def verify_with_judge(answer: str, sources: list[str], judge_llm, max_retries: int = 2) -> dict:
    prompt = f"""
    以下の回答が、与えられたソースの内容と矛盾していないか判定してください。
    矛盾がある場合はcontradictionをtrueにし、該当箇所を指摘してください。

    回答: {answer}
    ソース: {sources}
    """

    for attempt in range(max_retries + 1):
        raw = judge_llm.generate(prompt, response_format="json")
        try:
            result = json.loads(raw)
            return {
                "contradiction": result["contradiction"],
                "flagged_span": result.get("flagged_span"),
                "confidence": result.get("confidence"),
            }
        except (json.JSONDecodeError, KeyError):
            if attempt == max_retries:
                # パース不能時は安全側(要人間レビュー)に倒す。silent failで検証をすり抜けさせない
                return {"contradiction": True, "flagged_span": None, "confidence": 0.0}
```

3つ目は、**これらの機構をゼロから作らず既存OSSに乗る選択肢**だ。

* **承認ゲート**: ByteDance発「DeerFlow」(OSS、76k+ stars)。  
  LangGraphのcheckpoint機構を土台にhuman-in-the-loopを一級機能として組み込み、あらかじめ定義したポイントで実行を一時停止し中間結果を人間の確認に回せる。「承認待ちだけでは自律ワークフローが立ち行かない」という開発コミュニティの指摘(GitHub Issue #1213)を受け、policy-drivenなtool call認可を行う「Guardrails」層も追加、タイムアウト・手動によるkill switchも備える。v2.0で2026年2月GitHub Trending 1位を獲得
* **呼び出しレベルでの暴走制御**: 「Portkey AI Gateway」(OSS、12k+ stars)。  
  プロセス単位でエージェントを一時停止させる専用ツールは、調査した範囲ではまだ実績の伴うものが見当たらなかった。代わりに実務で使われているのが、LLM呼び出し自体をゲートウェイ層で制御するアプローチだ。Portkeyのcircuit breakerはP99レイテンシやエラー率が閾値を超えると該当ルートを一時的に遮断し、プローブリクエストで復旧を確認してからトラフィックを再開する。加えてingress時点でリクエスト数・トークン数・コストの上限を強制する「Usage Policies」も備え、エージェントが暴走してAPIを叩き続ける事態を呼び出し元で機械的に止められる。circuit breakerやbudget limitは元々SaaS限定機能だったが、Gateway 2.0でOSS版に統合された
* **監査ログ基盤**: 「Langfuse」(OSS、30k+ stars)。trace/observation/session/scoreという構造でエージェントの多段実行を記録し、ネストしたtool call・sub-agentの呼び出し関係まで追跡できる点が単純なリクエスト/レスポンスロガーとの違い。ClickHouse+PostgreSQLでセルフホスト可能なため、社内に監査ログを持ち出さずに保持したい場合の選択肢になる

「Kill Switch(完全停止)」と「Circuit Breaker(特定アクションの頻度制限)」を区別し、高コストアクション別・目的別にbreakerを設計するという考え方が、業界標準になりつつある。

## 展望

今回の判決は**単発の事例に留まらず、今後の法整備の方向性を示すシグナルとして読むべき**だ。EU域内ではAI Actをはじめとする規制の枠組みが整備されつつあり、「AIが生成したから」という理由で企業の責任が軽減されるという発想自体が、各国の司法判断において後退していく可能性が高い。

開発現場としては、規制が明文化されるのを待つのではなく、以下を今から準備しておく価値がある。

* **生成物とソースの対応関係を保存する仕組み(トレーサビリティ)**
* **出力前検証層(LLM-as-a-Judge等)の組み込み**
* **免責事項に依存しない、実質的な品質保証プロセスの設計**

「エージェントの自律性を高める」ことと「エージェントの出力に責任を持てる体制を作る」ことは、今後表裏一体の要件になっていくと筆者は考える。  
生成AIは便利なツールではあるが、使い方を誤れば法規に触れてしまう可能性もゼロではないという事を認識した上で、最大限活用していけるよう日々勉強、取り組んでいく事が肝要だ。  
それでは良きAIライフを。
