Claude Codeの性能を上げたい人は必見。
Anthropicが公開した「Claudeのハーネス設計術」が勉強になる。

世の中に色んなClaude CodeのTipsがありますが、公式の解説が一番為になるので、マストで読みましょう。

・ハーネスとは何か？
・性能が上がる設計術

などが学べます。

重要なのでスレッドで解説👇

(2/n)
そもそも「ハーネス」とは、モデルの周囲にある構造のこと。
UX・コスト・セキュリティを強制するために開発者が組む足場です。

Anthropic共同創設者Chris Olahの指摘:

「生成AIは"作られる"より"育つ"」

だから、ハーネスに埋め込んだ制約はモデルが賢くなる度に陳腐化する。
核心の問いはたった一つ:

"What can I stop doing?"（何をやめられるか？）

実例:
Sonnet 4.5はコンテキスト逼迫で早期終了する癖があり、多くの開発者が「リセット機能」を足した。
→ Opus 4.5でその挙動は消えた。リセット機能は足を引っ張るだけに。

ハーネスは"足す"より"剥がす"発想が、Claudeの伸びしろを解放する。

(3/n)
パターン①: Claudeが既に熟知している汎用ツールを使う。

独自ツールを大量に設計するのではなく、bashとtext editorのような"Claudeが長年学習で触ってきたツール"を渡すのが最強。

証拠:
Claude 3.5 Sonnetは bash + text editor だけでSWE-bench Verifiedを49%達成。

Claude Codeもこの2つが基盤になっている。
さらに、programmatic tool calling / Agent Skills / memory tool も、結局はbash+text editorの組み合わせから派生している。

→ 特殊ツールを増やす前に、汎用ツールの使い倒しを先に考えるべき。

(4/n)
パターン②-A: ツール結果をClaude自身にフィルタさせる。

従来のハーネス:
「ツール結果は全部コンテキストに流し込む」

問題:
大きなテーブルから1列だけ欲しい時も全行が流れ込み、不要なトークンでattentionを食い潰す。

解決策:
Claudeにコード実行環境(bash/REPL)を渡し、「何をコンテキストに通すか」をClaude自身にコードで書かせる。

結果:
BrowseCompベンチマーク、Opus 4.6で 45.3% → 61.6%（フィルタ機能付与で）。

ハーネスが判断するのではなく、Claudeに判断させる方が速いし賢い。

(5/n)
パターン②-B: システムプロンプトを膨らませる代わりに、Claude自身にコンテキストを管理させる。

従来:
全タスク固有の指示をシステムプロンプトに詰め込む
→ 拡張不可＋attention budgetを浪費

3つの武器:

Skills: YAML frontmatterに短い説明だけプリロード。詳細は必要時に進行的開示。
Context editing: 古い思考ブロックや結果を選択的に削除。
Subagents: 別コンテキストにフォークして並列探索 → BrowseCompで +2.8%（Opus 4.6）。

「全部見せる」ではなく「Claudeが必要な時に取りに行く」設計に切り替える。

(6/n)
パターン②-C: Claude自身にノートを書かせて長期タスクを継続させる。

Compaction（会話の自己要約）:
・Sonnet 4.5: 43%で頭打ち
・Opus 4.5: 68%
・Opus 4.6: 84%

モデルが賢くなるほど"自分の過去"を上手くまとめられる。

Memory folder（ファイルにノート書き出し）:
Sonnet 4.5のBrowseComp-Plus: 60.4% → 67.2%

わかりやすいポケモンの実例:
・Sonnet 3.5 は14,000ステップで31ファイル(重複多め)作って2つ目の街どまり
・Opus 4.6 は同じステップ数で10ファイルに戦術ノート、ジム3つ制覇

外部の検索インフラ無しで、Claude自身がメモを残すだけで成績が上がる。

(7/n)
パターン③: 剥がす一方ではない。キャッシュとセキュリティの境界だけは丁寧に。

キャッシュヒット率の最大化（キャッシュトークン = 基本の10%コスト）:
・静的→動的の順に並べる（システムプロンプト→ツール→メッセージ）
・プロンプトを書き換えず <system-reminder> を追加する形で更新
・モデル切り替え禁止（キャッシュはモデル固
