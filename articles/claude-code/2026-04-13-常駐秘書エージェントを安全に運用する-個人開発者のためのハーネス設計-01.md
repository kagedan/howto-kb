## はじめに

Claude Codeを使って常駐型の秘書エージェントを構築しました。Gmail・Googleカレンダー・Obsidianと連携し、朝のブリーフィング生成やメール要約、スケジュール管理などを自律的に行うエージェントです。今後はSlackとのMCP連携も予定しており、エージェントの操作範囲はさらに広がっていく見込みです。

2026年2月、OpenAIが「Harness engineering」という記事を公開し、AIエージェントの環境設計を指す「ハーネスエンジニアリング」という概念が一気に広まりました。コーディングエージェントの文脈で語られることが多いですが、常駐する秘書エージェントにも同じ考え方が必要です。連携先が増えるほど「意図しない操作を実行してしまうリスク」は高まり、制約・行動ルール・フィードバックの仕組みを設計しないまま運用するのは危険です。

本記事では、秘書エージェントのハーネスをどう設計・実装したかを整理します。Slack連携時のハーネス更新についても最後に触れています。

## ハーネスとは

ハーネスとは、AIエージェントが正しく動ける環境を設計する仕組みのことです。馬具のハーネスが馬の力を正しい方向に導くように、エージェントの行動を制約し、フィードバックで改善していく構造を指します。

本記事では、秘書エージェントのハーネスを「多層防御」と「フィードバックループ」の2軸で設計しました。

### 多層防御：3つの制約層

| 層 | 内容 | 強制力 |
| --- | --- | --- |
| サンドボックス（OS保護） | ファイルアクセス・ネットワークをOS側で制限 | 強い（カーネルレベル） |
| settings.json（ツール権限） | MCP経由の操作を許可/確認/拒否で管理 | 強い（Claude Code側で強制） |
| CLAUDE.md（行動ルール） | エージェントへの指示ベースのルール | 弱い（従おうとするが技術的強制力なし） |

上から順に強制力が強く、下の層が破られても上の層で被害を限定できる構造です。

### フィードバックループ：Feedback.mdによる自己改善

制約だけではハーネスとして不十分です。エージェントがミスをしたとき、同じ失敗を繰り返さない仕組みが必要です。

本エージェントでは`Feedback.md`というファイルをこの目的で運用しています。セッション終了時にエージェント自身がその日の反省・気づきを日付付きで追記し、翌朝のセッション開始時に読み込んで行動に反映します。

```
セッション開始（おはよう）
  → Feedback.md を読み込む
  → 過去の反省を踏まえて行動

セッション終了（おやすみ）
  → Feedback.md に反省・気づきを追記
```

これはHashiCorp共同創業者のMitchell Hashimoto氏が提唱した「エージェントがミスをするたびに、そのミスが二度と起きないような仕組みを設計する」というハーネスエンジニアリングの原則そのものです。CLAUDE.mdが静的なルール集であるのに対し、Feedback.mdは運用を通じて成長する動的なルール集として機能します。

## 前提：エージェントの構成

* 常時稼働のMacBook Pro上でClaude Codeを実行
* tmuxセッション内で常駐
* MCP経由でGmail・Googleカレンダーに接続（今後Slackも追加予定）
* Obsidian Vault（ローカルのMarkdownフォルダ）にブリーフィングやログを出力
* Claude Code Remote Control経由でモバイルからリモートで指示を送信

## 第1層：サンドボックス（/sandbox）

### なぜ標準ユーザー分離ではなく/sandboxか

当初、macOSで管理者権限のない専用ユーザーアカウントを作成する案を検討しました。しかし、エージェントの作業ファイルやMCPサーバーがすべてメインアカウントの環境に依存しており、ユーザーを分けるとクラウド同期やアプリ間連携が破綻するため断念しました。

Claude Codeには`/sandbox`というビルトインのサンドボックス機能があり、今の環境構成を変えずにOS保護を得られます。

### 有効化手順

Claude Codeのプロンプトで以下を入力します。

モード選択メニューが表示されるので「Sandbox BashTool, with auto-allow」を選択します。

* **auto-allow**：サンドボックス範囲内のコマンドは自動実行。範囲外は確認を求められる
* **regular permissions**：すべてのコマンドで毎回確認が出る

常駐エージェントには**auto-allow**が適しています。毎回確認が出ると、人間が承認するまで作業が止まり、自律的な秘書として機能しません。

### macOSではすぐ使える

macOSにはSeatbeltというサンドボックスフレームワークが組み込まれており、追加インストールなしで動作します。

### 書き込み許可パスの追加

サンドボックスのデフォルトでは、Claude Codeの起動ディレクトリとそのサブディレクトリのみ書き込み可能です。エージェントの起動ディレクトリとObsidian Vaultが別の場所にある場合、`settings.local.json`で明示的に許可します。

```
{
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": true,
    "filesystem": {
      "allowWrite": [
        "/Users/yourname/Obsidian/Vault"
      ]
    }
  }
}
```

これにより：

* エージェント起動ディレクトリ → 書き込みOK
* Obsidian Vault → 書き込みOK
* それ以外（~/.ssh、システムファイル等） → OSがブロック

### サンドボックスの限界

`/sandbox`は万能ではありません。

* **Bashツールのみに適用**：Claude CodeのRead/Editツールには別途permissionルールが必要
* **MCP経由の操作には効かない**：GmailやカレンダーへのアクセスはMCPサーバー経由のため、サンドボックスの管轄外
* **osascript（AppleScript）はブロックされる**：アプリ間通信（XPC）がサンドボックスにより遮断される

後述のsettings.jsonとCLAUDE.mdで補完する必要があります。

## 第2層：settings.json（ツール権限）

Claude Codeの`settings.local.json`で、各ツールの自動許可/確認/拒否を設定します。

### 設計方針

**読み取り系は自動許可、書き込み・送信系は確認を要求**

```
自動許可（allow）:
  - カレンダー予定の取得
  - Gmail一覧・詳細の取得
  - アカウント一覧
  - カレンダー予定の作成

確認が必要（未登録 = デフォルトでask）:
  - Gmail送信
  - カレンダー予定の変更・削除
  - Slackメッセージ送信（連携後に追加予定）
  - 未許可ドメインへのWebFetch
```

この設定により、エージェントが勝手にメールを送信したり予定を削除したりすることを防げます。

## 第3層：CLAUDE.md（行動ルール）

CLAUDE.mdにセキュリティハーネスのセクションを追記します。技術的な強制力はありませんが、Claude Codeが従おうとするルールとして機能します。

```
## セキュリティハーネス

### システム操作の制限
- sudo の使用：禁止
- 外部スクリプトのダウンロード・実行：禁止
- ~/.ssh, ~/.config 等の読み書き：禁止
- パッケージのインストール（brew, npm install, pip install等）：禁止
- osascript の使用：禁止
- launchdへの新規登録：ユーザーの許可を得てから
- アクセス可能なディレクトリ：作業用Vault配下のみ

### 外部サービス操作の制限
#### Gmail
- 読み取り・要約：自由に行ってよい
- 下書き作成：可
- 送信：禁止。下書きを作成しユーザーの確認を待つ

#### Googleカレンダー
- 読み取り：自由に行ってよい
- 予定の作成：可（ブリーフィングで報告）
- 予定の変更・削除：ユーザーの確認を得てから実行

### ファイル操作の制限
- 作業用Vault外のファイル変更：禁止
- 既存ファイルの削除：原則禁止（アーカイブ運用で対応）

### 判断に迷う場合
- 実行せず、ユーザーに確認を求める
```

## 検証で得た知見

### osascriptはサンドボックスが遮断する

スマートフォンのリマインダーアプリとの連携を検討しました。macOSのリマインダーにAppleScript経由で書き込み、iCloud同期でスマートフォンに反映させる構想です。

```
osascript -e 'tell application "Reminders" to get name of every list'
```

ユーザーのターミナルでは正常に動作しましたが、サンドボックス内では**アプリ間通信（XPC）がブロック**されてエラーになりました。リマインダーアプリを事前に起動しておいても同様です。

専用シェルスクリプトを作成して間接的に呼び出す方法も試しましたが、スクリプト内のosascriptも同様にブロックされます。

osascriptをサンドボックスから除外する設定（`sandbox.excludedCommands`）もありますが、**osascriptはリマインダー以外のアプリも自由に操作でき、ターミナル経由で任意コマンドを実行できてしまう**ため、セキュリティハーネスの目的と完全に矛盾します。

代替案としてはIFTTTなどのWebhookサービス経由での連携や、Googleカレンダーでの代替が考えられます。

### 環境構築はユーザー、運用はエージェント

パッケージのインストールを禁止してもエージェントの秘書業務には支障がありません。秘書の仕事（カレンダー確認、メール要約、ファイル整理）はすべてClaude Code本体とMCPサーバーで完結します。

将来的にSlack Bot構築などの開発が必要になった場合は、ユーザーが環境構築を行い、完成した仕組みをエージェントに使わせる、という役割分担にします。

### 起動ディレクトリ = サンドボックスの境界

サンドボックスの書き込み範囲は「Claude Codeの起動ディレクトリ」が基準です。エージェントの起動ディレクトリと実際の作業ディレクトリが異なる場合は、`sandbox.filesystem.allowWrite`で明示的に許可する必要があります。

## 今後の展望：Slack連携とハーネスの拡張

現在、Slack MCPサーバーとの連携を予定しています。連携先が増えるたびに、ハーネスも合わせて更新する必要があります。Gmail・Googleカレンダーと同じ原則を適用し、CLAUDE.mdとsettings.local.jsonにSlack用のルールを追加します。

```
#### Slack
- チャンネル・スレッドの読み取り：自由に行ってよい
- メッセージ送信：禁止。下書きを作成しユーザーの確認を待つ
- チャンネルの作成・アーカイブ：ユーザーの許可を得てから
```

### ハーネス拡張の原則

新しい連携先が増えた際の判断基準は一貫しています。

* **読み取り** → 自動許可（情報収集は秘書の基本業務）
* **作成・下書き** → 自動許可または確認（取り消し可能な操作）
* **送信・削除・変更** → 確認必須（取り消しが難しい操作）

この原則に沿って設定すれば、連携先が増えてもハーネスの一貫性を保てます。

## まとめ

常駐エージェントのハーネスは、「制約」と「フィードバック」の両輪で設計するのが重要です。

```
制約（何をさせないか）
  サンドボックス（OS強制） ← 最後の砦。技術的にブロック
        ↑
  settings.json（ツール権限） ← 操作単位で許可/確認/拒否
        ↑
  CLAUDE.md（行動ルール） ← エージェントが従おうとする指示

フィードバック（どう改善するか）
  Feedback.md ← 失敗を蓄積し、同じミスを繰り返さない
```

制約だけでは「今あるリスク」にしか対応できません。Feedback.mdによるフィードバックループを回すことで、運用しながらハーネス自体が成長していきます。完璧なセキュリティは存在しませんが、制約で被害を限定し、フィードバックで継続的に改善するアプローチが、個人開発者にとって現実的な落としどころです。
