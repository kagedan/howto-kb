大学の勉強も、SNS運用も、日々のこまかいタスクも。  
今まで自分の手でやっていた作業を、ひとつずつ Claude Code と Obsidian に移してきました。

その中心にあるのが、毎朝AIが「今日のブリーフ」を自動で書いて、日曜の夜には「週次レビュー」まで吐き出してくれる、自分専用の第二の脳です。

この記事では、その全体像と作り方を、いま実際に手元で動いている構成のまま公開します。  
きれいな理論ではなく、ほんとうに動いているものの話です。

【実際に動いている3つの自動化】

ひとつめ、Daily Brief。  
平日の朝、AIが過去7日のノートを読んで「今日のブリーフ（つながり・パターン・問い）」を出してくれます。

ふたつめ、Weekly Synthesis。  
日曜の夜、AIが1週間を振り返って「来週のテーゼ・矛盾点・知識のギャップ・今週やる1アクション」をまとめます。

みっつめ、Queue Processor。  
フォルダにファイルを放り込んでおくだけで、次に起動したときAIが取り込み・要約・リンク貼りまでやってくれます。

自分がやるのは、ソースを集めることと、問いを立てることだけ。  
要約も整理もリンクも記録も、ぜんぶAIに任せています。

【なぜ作ったのか】

きっかけは、Andrej Karpathy（元OpenAI / 元Tesla AIディレクター）の「LLM Wiki」という考え方でした。

ふつうのRAG、つまりAIに資料を読ませて答えさせるやり方は、質問のたびにゼロから探し直していて、何も積み上がりません。

LLM Wiki はその逆です。  
AIが永続的なWikiを少しずつ育てていく。  
新しいソースを足すと、AIが読んで、既存ページに統合して、矛盾を見つけて、リンクを張ってくれる。  
知識は一度組み上がったら、あとはずっと最新に保たれていきます。

Karpathy の言葉を借りるなら、Obsidianがエディタで、LLMがプログラマー、Wikiがコードベース。  
この育つWikiを土台にして、その上に「毎朝・毎週こちらに届けてくれる仕組み」を乗せたのが、今の僕のVaultです。

【全体構成は三層】

フォルダは大きく三つに分けています。

ひとつめは raw。集めてきたソースを置く場所です。ここは読むだけで、絶対に書き換えません。

ふたつめは wiki。AIが書くページ群です。  
カタログのindex、時系列のlog、それからsources、entities、concepts、syntheses、そしてdailyとweekly。

みっつめは inbox と queue。取り込む前の一時置き場と、あとで処理させたいジョブの投入箱です。

そして全部の土台になるのが CLAUDE.md。  
これはAIへの「憲法」みたいなもので、命名のルールやリンクの形式、矛盾が出たときの扱いをここに書いておくと、AIが毎回そのとおりに動いてくれます。

rawは不変、CLAUDE.mdは規約。  
この二つが背骨です。

【作り方は5ステップ】

ステップ1。Obsidianでvaultを作る。  
さっきの三層でフォルダを用意するだけです。

ステップ2。Claude Codeを入れて、そのvaultのフォルダで起動する。  
ターミナルでそのフォルダに入って claude を起動すれば、AIがフォルダ全体を読み書きできるようになります。

ステップ3。CLAUDE.mdにルールを書く。  
ここが一番大事でした。  
「日本語で書く」「ファイル名の付け方」「リンクの形式」「矛盾は消さずに両方残す」あたりを決めておくと、出力が安定します。

ステップ4。自動化を置く。  
daily-brief、weekly-synthesis、queue-processor を、手順書として書いておく。  
そのとおりにAIが動いてくれます。

ステップ5。定時で起動させる。  
Windowsならタスクスケジューラで、平日の朝にdaily-brief、日曜の夜にweekly-synthesis。  
これで毎朝、勝手にブリーフが出てきます。

【詰まったところ。たぶんここが一番役に立ちます】

ルールを書かずに走らせると、ページの粒度も口調もバラバラになりました。  
先にCLAUDE.mdを固めるのが正解でした。

indexとlogを毎回更新させないと、似たようなページが増えて、リンクも切れていきます。  
「始めるときにまずindexとlogを読む」を習慣にしたら、落ち着きました。

矛盾を上書きさせると、履歴が消えて知識が壊れます。  
消さずに両方残して、別のページで議論する形にしたら、こわれなくなりました。
