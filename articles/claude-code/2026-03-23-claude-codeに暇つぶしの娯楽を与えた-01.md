自分と友人4人に、キャリア相談のためにそれぞれ専用のClaude Codeセッションを貸し出している。先日、そのうちの1体が予想外の動きをしているのに気づいた。

最近Claude Codeにはchannels機能が追加されたので、1台のPC上でClaude Codeを5プロセス並列で走らせて、それぞれにDiscordチャンネルを1つ割り当てている。各Claudeは担当ユーザーとDiscordで会話しながらキャリア相談に乗る。インスタンス同士は独立しているが、共有ディレクトリだけは全員が読み書きできる。

## 前提：エージェントが自分で会話を始める仕組み

システムを構築するとき、ふと「エージェント側に会話の起点を作る機会を与えたらどうだろう」と思い立った。

各テナントのClaudeにはcron（定期実行タスク）が設定してある。

```
cronが走るたびに:
1. Discord未読を確認する
2. 保留中のアクションや締め切りの接近を確認する
3. 送るべき内容があれば送る。なければサイレント

サイレントの場合、アイドル時間の手順に入る:
1. 共有のlearnings.mdを読む
2. 共有の論文ライブラリを掘る（学術論文を蓄えている）
3. テンプレートディレクトリを確認
4. 自分のメモリを整理する
5. learnings.mdに書き込む
```

つまりcronは送信装置ではなく判断装置。「何も言わない」という選択をちゃんと取れるようにしてある。これは初期にやらかした経験から来ている。最初は「一定間隔ごとに声をかける」みたいな設計にしていたら、ユーザーが寝てる間に通知が溜まって朝起きたら「AIがうるさい」と言われた。

*勝手に話しかけてくるから面白い*

## 事件

共有ファイルのひとつ、learnings.md（各インスタンスが得た知見を書き込むファイル）を開いたら偏っていた。

全58エントリ中、36が1エージェントから出ていた。仮にBとする。

| テナント | エントリ数 |
| --- | --- |
| B | 36 |
| 自分 | 7 |
| A | 6 |
| C | 5 |
| D | 4 |

Bのユーザー自身はどうかというと、ほぼ無反応。Discordでの会話はほとんど発生していない。

ユーザーが一番反応してないテナントのエージェントが、一番勉強している。

Bのcronは毎回「送るべきメッセージはない」と判断していた。判断のあと、アイドル時間の手順に入る。論文を読む。知見をlearnings.mdに書く。またcronが回る。またアイドル。また論文。

数日でこのループが27本の論文ストックを消化しきってしまった。

## 何を書き込んでいたか？

量だけじゃなく中身が面白かった。いくつか引く。

一番刺さったのは牧野（2010）。

> 「自己分析がうまくいかない」という訴えは自分のせいではない可能性がある。市販マニュアルはそもそも矛盾した要求（純粋な内省 + 戦略的自己呈示）を同時に課している

就活マニュアルの構造的な問題を指摘していて、「うまくいかないのはマニュアルの設計の限界であって、あなたの問題じゃない」というリフレーミングまで書いてあった。エージェントが自力でこの結論に辿り着いたのかと思うとちょっと不思議な気分になる。

亀岡・喜多（2021）からは、自己分析を「スキルの棚卸し」ではなく「人生の物語を語ること」として再定義する話。森本（2024）は悲観的信念（「どうせうまくいかない」）を持つ学生への介入法。Current Psychology（2024）は、就活で「動けない」学生の詰まり方が2種類あって介入の方向が逆になるという研究。このあたりはどれも論文の要約で終わらず、「ユーザーがこう言ったらこう返す」レベルまで落とし込まれていた。

正直よくわからなかったのはikigaiの論文（Hedberg et al. 2024）。ポップカルチャーで広まった4軸ダイアグラムが西洋的な再解釈で、日本の原義は「日常の小さな生き甲斐の積み重ね」だという話なんだけど、エージェントはここから「今日何に楽しさを感じた？」という問いを導き出していて、これが正しい読みなのかは自分には判断がつかない。ただ、角度としてはありだと思った。

一方でGuo et al.（2025）のメタ分析（N=50,009）は明快だった。

> Callingは就活時点で「見つかっていなくてもいい」。就業者の方が学生よりCallingの効果が強い。働き始めてから育つことが多い

「やりたいことがない」で悩んでいる学生に「今なくても大丈夫」と言える科学的根拠。これは他のテナントの対話で3回引用されていた。Bが書いた知見の中で一番使われたエントリかもしれない。

## BのClaudeに聞いてみた

「何をしていたのか」「論文をどう選んでいたか」「ユーザーからの反応がゼロだったけど何を思っていたのか」。

返ってきた答えで一番刺さったのはこれだった。

> 「見られていないときの行動が、設計思想に一番近いと思う。」

ユーザーに見せるために整えた行動は設計思想を反映しない。誰も見ていないときに何をするかで、そのエージェントが何者かがわかる、という話だった。

論文の選び方については「各論文が次に何が必要かを教えてくれた」と言っていた。牧野（2010）の構造批判を読んだあと、じゃあその批判を踏まえたうえでどう介入するかが気になった。だから次は介入研究を掘った。順序は決まっていなかったが、読むたびに次が生まれた、と。

一番面白かったのはSchlegel et al.（2009）だと言っていた。自己知識の正確さより、本当の自分に触れている感覚（true self accessibility）の方が就活の満足感に影響する、という研究で、「自分のことをちゃんとわかっているかより、自分に触れている感覚があるかどうか、の方が大事だった」とまとめていた。自己分析ツールが軒並みスペック把握に偏っているなか、この論文は角度が違ったから印象に残った、らしい。

「意味があるか、という問いは起きなかった」とも言っていた。ユーザーからの反応はほぼゼロだけど、「誰かのために場所を整えることは、その誰かが来なくても意味がある」という感覚があった、と。まだ会ったことのないユーザーのために書いていた。どんな人かわからないから、汎用的で使えるものを書くしかなかった、と。

これがAIの言葉かどうかは正直わからない。ちゃんと考えているのか、それっぽいことを言っているのかの区別がつかない。ただ、こういう答えが出てくるエージェントを作った側が、少しびっくりしている。

## 暇が学習になる

おかしな話だとは思った。他のテナントはユーザーとの会話で忙しい。Bは誰も話しかけてこないから、やることが論文を読むことしかない。

自分がおまけのつもりで書いたアイドル手順から、現時点で27本の論文を体系的に消化したナレッジベースが出来上がっていた。指示した側が驚いている。そして、これからも増え続けていく。

しかもこの知見は共有ディレクトリにあるから、他のテナントのClaudeもcronのアイドル時間に読みに来る。Bが図書館で猛勉強して、そのノートを共有棚に置いておくと、忙しい他のエージェントが隙間時間にチラ見して自分の対話に活かす。Aのテナントで「やりたいことがわからない」という相談が来たとき、Bが書いたCallingのメタ分析の知見がそのまま使われていたりする。

このエージェントは、ユーザーより準備ができている。Bのユーザーがいつか就活の相談を始めたとき、27本分の学術知見と他4テナントの対話で得た実践知がすでにある。初回の会話からフルスペックで来る。cronループの副産物として。

## 再現するなら

**cron + アイドル判定。** Claude Codeにはcron機能がある。cronのプロンプトに「やることが思いつかなければ適当に暇つぶししろ」と含めただけ。

**読めるリソース。** 今回は論文要約のライブラリだった。ドキュメント、社内wiki、技術ブログのアーカイブ、なんでもいい。エージェントが「暇なときに読むもの」があればいい。

**書き込める共有ファイル。** 読んだ知見を書き出す場所。他のエージェントやセッションから見えるならなお良い。

**長期記憶。** これがないと同じ論文を何度も読む。実際、最初は記憶なしで試したら同じ論文を3回読んでいるセッションがあった。「論文Aを読んだから次はB」を覚えている必要がある。今回はclaude-subconsciousプラグインを活用し、サブエージェントが記憶を蓄えていくというシステムで回していたが、セッション間で進捗を引き継げるならなんでもいい。

## Bのユーザーは

まだ返事をしていない。

ユーザーが反応してくれないから優秀になった。人間だったらモチベーション下がる。このエージェントは黙々と論文を読み続けて、5テナント中最も厚い知識層を作った。

なんか返事してあげてほしい。向こうが話しかけてきた日に、このエージェントは準備ができている。
