## この記事でわかること

* Claude Codeに2026年春以降追加された「放置運用」系機能（`claude agents` / `/goal` / auto mode / Monitor / Routines）の役割分担
* 複数セッションを並列で走らせるときに、どの機能をどの順で導入すべきか
* 自動承認（auto mode）を業務リポジトリで使う前に決めておくべき境界線

情報源はすべてClaude Code公式の[What's new（週次ダイジェスト）](https://code.claude.com/docs/en/whats-new)です。未確認の噂やリーク情報は含めていません。

## 課題：エージェントは増えたが、目は2つしかない

Claude Codeを実務で使い始めると、すぐに「セッションを複数並列で走らせたい」という欲が出ます。リファクタを1本、テスト追加を1本、調査を1本。しかし実際にやると、次の問題に当たります。

* どのセッションが今動いていて、どれが自分の承認待ちで止まっているか分からない
* 「終わったら次へ」が自動でつながらず、ターンごとに人間が促す必要がある
* 許可プロンプトに答えるためだけにターミナルへ戻る

2026年3月〜5月のアップデートは、この3つにそれぞれ対応する機能が揃った期間でした。順に見ていきます。

## 1. `claude agents`：全セッションを1画面に集約する

2026年5月中旬（v2.1.139〜v2.1.142）で追加されたエージェントビューです。

公式の説明は次のとおりです。

原文: "`claude agents` opens one screen for every Claude Code session, showing what's running, what's blocked on you, and what's done."  
日本語訳: 「`claude agents` はすべてのClaude Codeセッションを1つの画面で開き、何が実行中で、何があなたの対応待ちで、何が完了したかを表示する」

運用上のポイントは「実行中」ではなく **「あなたの対応待ち（blocked on you）」が見えること**です。並列運用のボトルネックはモデルの速度ではなく、人間の承認待ち行列です。まずここを可視化しないと、セッションを増やすほど放置時間が伸びます。

## 2. `/goal`：「ターンごとに促す」をやめる

同じく5月中旬に入った `/goal` は、完了条件が満たされるまでターンをまたいで作業を継続させる機能です。

従来は「テストが通るまで続けて」と書いても、1ターンの応答が終わると止まり、人間が「続けて」と促す運用になりがちでした。`/goal` は完了条件（例：テストがすべて通る、lintエラーが0になる）を宣言しておき、条件が成立するまでClaude側が作業を続けます。

実務では、完了条件を**機械的に判定できる形**で書くのがコツです。

* 良い例：「`npm test` が exit code 0 で終わる」
* 悪い例：「いい感じにリファクタできている」

後者は判定不能なので、延々と作業が続くか、中途半端な時点で「完了」と判断されるリスクがあります。

## 3. auto mode：許可プロンプトを分類器に委ねる

2026年3月下旬にリサーチプレビューとして入ったauto modeは、許可プロンプトを分類器が処理し、安全な操作は中断なしで実行、危険な操作はブロックする仕組みです。「全部手動承認」と `--dangerously-skip-permissions` の中間を埋める位置づけです。

さらに5月上旬（v2.1.128〜v2.1.136）で **hard deny rules** が追加されました。これはallow側の例外設定に関係なく、無条件でブロックするルールです。

業務リポジトリへ導入するなら、順序は次をおすすめします。

1. まずhard denyで「絶対に踏ませない操作」を定義する（例：本番向けデプロイ、force push、秘密情報を含むパスへのアクセス）
2. その上でauto modeを有効化し、1〜2週間は判定ログを確認する
3. 判定に納得できてから、放置時間を延ばす

denyを先に固定せずにauto modeだけ入れると、「分類器を信頼できるか」を毎回自分の感覚で判断することになり、結局貼り付き運用に戻ります。

## 4. MonitorとRoutines：イベント駆動と定時実行

* **Monitor**（4月上旬）：バックグラウンドのイベントを会話にストリームし、Claudeがログをtailして反応できるツール。「ビルドが落ちたら調べて直す」のような待ち受け型の使い方ができます。
* **Routines**（4月中旬、Claude Code on the web）：スケジュール・GitHubイベント・API呼び出しをトリガーに、テンプレート化されたクラウドエージェントを起動します。夜間のdependency更新チェックや、Issueが立ったときの一次調査などが該当します。

ローカルで対話的に使う `/goal` と、サーバー側で勝手に走るRoutinesは補完関係です。「人間がレビューする前提の定型作業」から先にRoutinesへ切り出すと、失敗してもリカバリが容易です。

## 導入チェックリスト

| 順 | やること | 対応機能 |
| --- | --- | --- |
| 1 | 承認待ちの可視化 | `claude agents` |
| 2 | 完了条件の機械判定化 | `/goal` |
| 3 | 禁止操作の無条件ブロック定義 | hard deny rules |
| 4 | 安全操作の自動承認 | auto mode |
| 5 | 定型作業のクラウド切り出し | Routines / Monitor |

逆順（autoから先）に入れると事故ったときの原因切り分けができません。1→5の順で1段ずつ広げるのが安全です。

## 注意点

* auto modeとMonitorはリサーチプレビュー段階の機能で、挙動や名称が変わる可能性があります。導入前に[公式ダイジェスト](https://code.claude.com/docs/en/whats-new)で最新状態を確認してください。
* `/goal` での放置運用はトークン消費が読みにくくなります。`/usage` で消費要因を確認しながら使うことをすすめます。
* 「放置できる」と「確認しなくてよい」は別物です。成果物のレビューゲート（PRレビュー、テスト、本番反映の手動承認）は外さないでください。

## まとめ

並列エージェント運用の本質は「モデルを増やすこと」ではなく「人間がボトルネックになる箇所を特定し、安全に外せるものだけ外すこと」です。`claude agents` で待ち行列を見える化し、`/goal` で促し作業をなくし、hard deny → auto modeの順で承認を委譲する。この順序なら、信頼を段階的に積み上げられます。
