## サマリー

* Claude Code（モデルはFable 5）を利用してAWSのコスト分析〜削減候補の特定〜削除スクリプトの生成までを行い、**半日の作業でクラウド利用費の削減**を実現できました
* AIには読み取り専用の権限のみを渡すことで安全性を高めています
* 一連の手順はClaude CodeのSkillsとしてまとめており、`/aws-cost-report` と実行するだけで毎回同じ手順・観点のレポートが得られます
* この記事をClaude Codeに読み込ませれば、同様のSkillsを作成できます

## はじめに

皆様のAWS環境に、検証やPoCで作ったまま残り続けているリソースはないでしょうか？

弊社では生成AIを活用した検証やPoCを数多く行っているのですが、その過程で作成したリソースが残り続け、気づけばAWSの月額費用が増加傾向にありました。我々のような少人数の組織ではコストの棚卸しはどうしても後回しになりがちですし、複数のアカウントをコンソールで確認して回るのもなかなか大変な作業です。

そこで今回は、この棚卸し〜削減の作業をClaude Code（モデルはFable 5）に任せるとどこまでできるのかを検証してみました。結果として半日で費用を大幅に削減することができたため、その仕組みと気をつけるポイントをまとめます。同じ課題をお持ちの方の参考になれば幸いです。

## 前提

* 今回の検証はAWSを前提にします
  + AWS Organizationsで複数アカウントを運用している環境です
* ただ、AzureやGoogle Cloudでも同様の考え方で実現可能です。
  + コマンドを `az` / `gcloud` に置き換え、課金データの取得方法（Google CloudはBigQueryへの請求エクスポート等）を各クラウドの流儀に合わせれば、同じ構成を組めます
* エージェントにはClaude Code（モデルはFable 5）をローカルのMacで利用しています
* AWS CLIが利用可能な環境を想定します

## 解決したい課題

弊社ではプロジェクトや検証の用途、メンバーの技術検証用環境など、用途ごとにAWSアカウントを分けて運用しています。理想的には、検証の完了時やプロジェクトの終了時などのタイミングで不要なリソースは削除されるべきです。ただ、実情としてはリソースの消し忘れやいつか使うかもしれないから残しておこうと思って忘れていたスナップショットなどが残っている状態となっていました。

そして一番厄介なのが、「これは消していいのか？」の判断がリソースを作成した本人以外は誰も答えることができないことです。結果として、正体のよく分からないリソースだけが積み上がっていくことになります。

## 解決するための仕組みの構築 / 試したこと

大きくは以下の5つのステップで進めました。

1. コスト分析専用の読み取り専用IAMユーザーを作成し、AIエージェントに利用させる
2. Cost Explorer APIで実額を多角的に取得する（月次×サービス別 / 日次 / アカウント別 / USAGE\_TYPE別）
3. 全アカウント×課金実績のあるリージョンのリソースをリアルタイムに確認するスクリプトをAIに生成させ、人間が実行する
4. AIが課金データと実測を突き合わせて、削減候補のリストを証拠つきで作成する
5. 削除スクリプトをAIが生成し、人間が実行し、AIが結果を検証する

AIに削除そのものを任せるのではなく、**AIは分析とスクリプトの生成まで、実行は人間が行う**という線引きをしている点がポイントです。また、分析部分（ステップ2〜4）はClaude CodeのSkillsとして固めているため、2回目以降は `/aws-cost-report` と実行するだけで前回レポートとの差分つきで再現できます（詳細は後述します）。

## 各ステップで実際にやっていること

**1. コスト分析専用の読み取り専用IAMユーザーを作成し、AIエージェントに利用させる**

AWSリソースのRead権限を持つ新しいIAMユーザーを作成します。ベースにはAWS管理ポリシーのViewOnlyAccessを利用します。ViewOnlyAccessはリソースの一覧・メタデータの参照が中心で、S3オブジェクトやDynamoDBレコードといった**データの中身を読むActionを含みません**。ReadOnlyAccess（`s3:Get*` や `dynamodb:Scan` を含む）と比べて、AIエージェントに渡す情報を必要最小限に絞ることができます。

```
# 1. ユーザー作成
aws iam create-user --user-name cost-report-readonly

# 2. ベース: 全サービス横断のメタデータ読み取り（AWS管理ポリシー）
aws iam attach-user-policy \
  --user-name cost-report-readonly \
  --policy-arn arn:aws:iam::aws:policy/job-function/ViewOnlyAccess

# 3. 補完: ViewOnlyAccessに含まれない必須分
aws iam put-user-policy \
  --user-name cost-report-readonly \
  --policy-name cost-analysis-supplement \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Action": [
        "ce:Get*",
        "ce:List*",
        "apprunner:List*",
        "apprunner:Describe*",
        "secretsmanager:ListSecrets",
        "tag:GetResources"
      ],
      "Resource": "*"
    }]
  }'

# 4. アクセスキー発行
aws iam create-access-key --user-name cost-report-readonly
```

発行されたアクセスキーは `~/.aws/credentials` に専用プロファイルとして登録します。

```
[cost-report]
aws_access_key_id     = AKIAXXXXXXXXXXXXXXXX
aws_secret_access_key = xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

ここで大事なのは、**アクセスキーの値そのものをAIとのチャットに貼らない**ことです。credentialsファイルに置いてしまえば、AIには「`cost-report` というプロファイルを使ってください」とプロファイル名だけ伝えれば済みます（Claude Codeの場合は`CLAUDE.md`や後述の`SKILL.md`に書いておきます）。なお、ローカルで動くエージェントは設定次第で `~/.aws/credentials` そのものを読めてしまうため、Claude Codeではpermission設定でこのファイルへのアクセスを拒否（deny）しておくとより安全です。

最後に動作確認を行います。3つ目のコマンドが**AccessDeniedになること（=書き込めないこと）まで確認するのがポイント**です。「書けないこと」がこの仕組みの安全性の根拠だからです。

```
# 認証確認（AccountとArnが返ればOK）
aws sts get-caller-identity --profile cost-report

# Cost Explorer APIの疎通確認（$0.01かかります）
aws ce get-cost-and-usage \
  --time-period Start=2026-07-01,End=2026-07-07 \
  --granularity DAILY --metrics UnblendedCost \
  --profile cost-report

# 読み取り専用であることの確認（AccessDeniedになれば正しい）
aws ec2 create-key-pair --key-name should-fail --profile cost-report
```

**2. Cost Explorer APIで実額を多角的に取得する（月次×サービス別 / 日次 / アカウント別 / USAGE\_TYPE別）**

既に権限を付与したため、このIAMユーザーを利用して実際に情報を取得します。この処理はシェルスクリプトを利用しています。出力内容としては、以下の4つのJSONファイルと2つの補助ファイルが作成されます。

**① cost\_monthly\_service\_cost-report.json — 月次×サービス別**

直近3ヶ月+当月の実額を、サービス名ごとに月単位で集計したものです。「何にいくら払っているか」「前月比で急増したサービスはどれか」を確認するためのファイルです。

**② cost\_daily\_cost-report.json — 日次合計**

今月分の実額を1日単位で集計したものです（Cost ExplorerのTimePeriodはEnd日を含まないため、前日分までが対象です）。フィルタやグループ化をかけていないため、**Organization全体の日次合算**が日付ごとに並びます。「急に増えた日はいつか」を特定するための急増検知用のデータで、コストが跳ねた日とデプロイや検証作業の日付を突き合わせることで原因を推定できます。

**③ cost\_monthly\_account\_cost-report.json — 月次×アカウント別**

直近3ヶ月+当月の実額を、連結アカウント（12桁のアカウントID）ごとに集計したものです。マネジメントアカウントのCost Explorerには組織全体の請求データが含まれるため、**メンバーアカウントに個別にログインすることなく、このプロファイル1つで全アカウントの費用配分が取得できます。**

**④ cost\_monthly\_usagetype\_cost-report.json — 月次×USAGE\_TYPE別**

直近3ヶ月+当月の実額を、課金項目（USAGE\_TYPE）単位で集計したものです。「固定費は満額なのに従量課金がゼロ」という判定（ステップ4）は、この粒度のデータを使って行います。

**⑤ aws\_accounts.txt — プロファイルと認証状態の記録（補助）**

実行に使用したAWSプロファイルと、その接続先アカウントID・認証状態（OK / BROKEN）を記録したものです。BROKENの場合はアクセスキーの失効が疑われるため、AIはレポートにその旨を明記してキーの再発行を依頼する運用にしています。なお、これは「ローカルのプロファイル一覧」であり組織内のアカウント一覧ではありません（全アカウントの費用は③に含まれます）。

**⑥ aws\_errors.log — エラーの集約先（補助）**

スクリプト実行中に発生したエラー（権限不足など）をすべて集約するログです。AIはレポート生成時に必ずこのログを確認し、取得できなかった情報があれば「確認できなかったこと」として明記します。エラーを握りつぶして「取得できたところだけのレポート」ができてしまうことを防ぐためのファイルです。

■costs.sh

```
#!/bin/bash
# Cost Explorer APIから実額コストを取得する（$0.01/リクエスト × 4回/プロファイル）
# 使い方: costs.sh <出力ディレクトリ> [プロファイル ...]（省略時はcost-report）
# 制限: CEのNextPageToken（ページネーション）は未対応。月次粒度では稀だが、USAGE_TYPE別は件数次第で分割され得る
set -u

OUT_DIR="${1:?出力ディレクトリを指定してください}"
shift
PROFILES=("${@:-cost-report}")

command -v aws >/dev/null || { echo "ERROR: aws CLIが見つかりません" >&2; exit 1; }
command -v jq  >/dev/null || { echo "ERROR: jqが見つかりません" >&2; exit 1; }

mkdir -p "$OUT_DIR"
ERR="$OUT_DIR/aws_errors.log"
ACCOUNTS="$OUT_DIR/aws_accounts.txt"
: > "$ERR"
: > "$ACCOUNTS"   # 再実行時の重複を防ぐため毎回初期化
FAILED=0

START_3M=$(date -v-3m +%Y-%m-01 2>/dev/null || date -d "3 months ago" +%Y-%m-01)  # macOS/Linux両対応
TODAY=$(date +%Y-%m-%d)
MONTH_START=$(date +%Y-%m-01)

# CE呼び出しの共通処理: 実行 → レスポンスの中身まで検証 → 成否を表示
fetch_ce() {
  local prof="$1" outfile="$2"; shift 2
  if aws --profile "$prof" ce get-cost-and-usage "$@" --output json > "$outfile" 2>>"$ERR" \
     && jq -e '.ResultsByTime | length > 0' "$outfile" >/dev/null 2>>"$ERR"; then
    echo "OK  ${outfile##*/}"
  else
    echo "NG  ${outfile##*/}  → 詳細は $ERR"
    FAILED=1
  fi
}

for prof in "${PROFILES[@]}"; do
  id=$(aws --profile "$prof" sts get-caller-identity --query Account --output text 2>>"$ERR")
  if [ -z "${id:-}" ]; then
    echo "$prof BROKEN" >> "$ACCOUNTS"
    echo "NG  プロファイル $prof の認証に失敗 → 詳細は $ERR"
    FAILED=1
    continue
  fi
  echo "$prof $id OK" >> "$ACCOUNTS"

  # 月次×サービス別（直近3ヶ月）
  fetch_ce "$prof" "$OUT_DIR/cost_monthly_service_$prof.json" \
    --time-period "Start=$START_3M,End=$TODAY" \
    --granularity MONTHLY --metrics UnblendedCost \
    --group-by Type=DIMENSION,Key=SERVICE

  # 日次合計（今月・急増検知用）
  fetch_ce "$prof" "$OUT_DIR/cost_daily_$prof.json" \
    --time-period "Start=$MONTH_START,End=$TODAY" \
    --granularity DAILY --metrics UnblendedCost

  # 連結アカウント別（Organizations利用時のみ意味を持つ）
  fetch_ce "$prof" "$OUT_DIR/cost_monthly_account_$prof.json" \
    --time-period "Start=$START_3M,End=$TODAY" \
    --granularity MONTHLY --metrics UnblendedCost \
    --group-by Type=DIMENSION,Key=LINKED_ACCOUNT

  # USAGE_TYPE別（固定費vs従量の突き合わせ用）
  fetch_ce "$prof" "$OUT_DIR/cost_monthly_usagetype_$prof.json" \
    --time-period "Start=$START_3M,End=$TODAY" \
    --granularity MONTHLY --metrics UnblendedCost \
    --group-by Type=DIMENSION,Key=USAGE_TYPE
done

if [ "$FAILED" -eq 0 ]; then
  echo "COSTS DONE"
else
  echo "COSTS DONE (一部失敗あり — aws_errors.log を確認)"
  exit 1
fi
```

**3. 全アカウント×課金実績のあるリージョンのリソースをリアルタイムに確認するスクリプトをAIに生成させ、人間が実行する**

ステップ2の課金データには2つの弱点があります。①反映が実態より24〜48時間遅れること、②集計値のため削除コマンドに必要なリソースIDが取れないことです。そこで「今この瞬間に何が存在しているか」をDescribe / List系のAPIで直接列挙する、リアルタイムなリソース確認を行います。

単一アカウントの環境であれば、手順1で作成した読み取り専用プロファイルを使ってAIエージェント自身がそのまま実行できます。一方、AWS Organizationsで複数アカウントを運用している場合は、メンバーアカウントの中を見るために管理者ロールへのスイッチ（assume-role）が必要になります。読み取り専用ユーザーにはその権限を持たせていないため、\*\*このステップだけは「AIがスクリプトを生成し、管理者権限を持つ人間が実行する」\*\*という分担にしています。

■`aws_inventory.sh`

```
#!/bin/bash
# 全AWSアカウント×課金実績のあるリージョンのリソースをリアルタイムに確認する（参照系コマンドのみ）
# 前提: 管理アカウントの管理者権限プロファイル（MGMT_PROFILE）が設定済みであること
set -u
TS=$(date +%Y%m%d-%H%M)
OUT="$HOME/aws-cost-reports/inventory-$TS.txt"
mkdir -p "$(dirname "$OUT")"
command -v jq >/dev/null || { echo "ERROR: jqが見つかりません" >&2; exit 1; }
REGIONS="ap-northeast-1 us-east-1"    # 課金実績のあるリージョンに絞る
MGMT_PROFILE="admin"                  # 管理アカウントの管理者プロファイル名
MGMT_ACCOUNT_ID="111111111111"        # 管理アカウントのID
exec > >(tee "$OUT") 2>&1             # 以降の全出力を画面とファイルの両方へ

# ②認証の切り替え: mgmt=管理アカウントのプロファイル / member=一時クレデンシャル
AWSC_MODE="mgmt"
a() {
  if [ "$AWSC_MODE" = "mgmt" ]; then
    aws --profile "$MGMT_PROFILE" "$@"
  else
    aws "$@"   # assume-roleでexportした環境変数の一時クレデンシャルで実行
  fi
}

# ③1アカウント分のスキャン（課金に効くリソースを網羅）
scan() {
  local label="$1"
  echo "== アカウント: $label"
  for r in $REGIONS; do
    echo "---- [$label | $r] ----"
    echo "-- EC2インスタンス"
    a ec2 describe-instances --region "$r" \
      --query "Reservations[].Instances[].[InstanceId,InstanceType,State.Name,Tags[?Key=='Name']|[0].Value]" --output text
    echo "-- EBSボリューム"
    a ec2 describe-volumes --region "$r" --query "Volumes[].[VolumeId,Size,State]" --output text
    echo "-- EBSスナップショット (自己所有: 件数/合計GB)"
    a ec2 describe-snapshots --owner-ids self --region "$r" \
      --query "Snapshots[].VolumeSize" --output text | tr '\t' '\n' \
      | awk 'BEGIN{n=0;s=0}{n+=1;s+=$1}END{print n"件 / "s"GB"}'
    echo "-- Elastic IP"
    a ec2 describe-addresses --region "$r" --query "Addresses[].[PublicIp,AssociationId]" --output text
    echo "-- 使用中Public IPv4 (ENI単位)"
    a ec2 describe-network-interfaces --region "$r" \
      --query "NetworkInterfaces[?Association.PublicIp!=null].[Association.PublicIp,Description]" --output text
    echo "-- NAT Gateway"
    a ec2 describe-nat-gateways --region "$r" --query "NatGateways[].[NatGatewayId,State]" --output text
    echo "-- ロードバランサー"
    a elbv2 describe-load-balancers --region "$r" \
      --query "LoadBalancers[].[LoadBalancerName,Type,CreatedTime]" --output text
    echo "-- ECSクラスター/サービス (desired/running)"
    for c in $(a ecs list-clusters --region "$r" --query "clusterArns[]" --output text); do
      echo "  cluster: ${c##*/}"
      svcs=$(a ecs list-services --cluster "$c" --region "$r" --query "serviceArns[]" --output text)
      if [ -n "$svcs" ]; then
        a ecs describe-services --cluster "$c" --services $svcs --region "$r" \
          --query "services[].[serviceName,desiredCount,runningCount,launchType]" --output text
      fi
    done
    echo "-- App Runner"
    a apprunner list-services --region "$r" \
      --query "ServiceSummaryList[].[ServiceName,Status]" --output text
    echo "-- RDS / Aurora"
    a rds describe-db-instances --region "$r" \
      --query "DBInstances[].[DBInstanceIdentifier,DBInstanceClass,DBInstanceStatus]" --output text
    a rds describe-db-clusters --region "$r" \
      --query "DBClusters[].[DBClusterIdentifier,Status]" --output text
    echo "-- Lambda関数"
    a lambda list-functions --region "$r" --query "Functions[].[FunctionName,Runtime]" --output text
    echo "-- WAF WebACL (REGIONAL)"
    a wafv2 list-web-acls --scope REGIONAL --region "$r" --query "WebACLs[].Name" --output text
    echo "-- Secrets Manager (件数)"
    a secretsmanager list-secrets --region "$r" --query "length(SecretList)" --output text
    echo "-- CloudFormationスタック (削除の単位に使える)"
    a cloudformation describe-stacks --region "$r" \
      --query "Stacks[].[StackName,StackStatus]" --output text
  done
  echo "---- [$label | グローバル] ----"
  a wafv2 list-web-acls --scope CLOUDFRONT --region us-east-1 --query "WebACLs[].Name" --output text
  a route53 list-hosted-zones --query "HostedZones[].[Name,ResourceRecordSetCount]" --output text
  a s3api list-buckets --query "Buckets[].Name" --output text
}

# ④アカウント一覧をOrganizations APIで動的取得（ハードコードしない=棚卸し漏れを防ぐ）
ACCOUNTS=$(a organizations list-accounts \
  --query "Accounts[?Status=='ACTIVE'].[Id,Name]" --output text)

# 管理アカウントは直接スキャン
AWSC_MODE="mgmt"
scan "管理アカウント($MGMT_ACCOUNT_ID)"

# メンバーアカウントはOrganizationAccountAccessRoleへスイッチしながらスキャン
while IFS=$'\t' read -r id name; do
  [ -z "$id" ] && continue
  [ "$id" = "$MGMT_ACCOUNT_ID" ] && continue
  unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN 2>/dev/null
  AWSC_MODE="mgmt"
  CRED=$(a sts assume-role \
    --role-arn "arn:aws:iam::$id:role/OrganizationAccountAccessRole" \
    --role-session-name inventory --duration-seconds 3600 --output json)
  if [ -z "$CRED" ]; then
    echo "!! $name($id): assume-role失敗 → スキップ（別手段での確認が必要）"
    continue
  fi
  export AWS_ACCESS_KEY_ID=$(echo "$CRED" | jq -r .Credentials.AccessKeyId)
  export AWS_SECRET_ACCESS_KEY=$(echo "$CRED" | jq -r .Credentials.SecretAccessKey)
  export AWS_SESSION_TOKEN=$(echo "$CRED" | jq -r .Credentials.SessionToken)
  AWSC_MODE="member"
  scan "$name($id)"
done <<< "$ACCOUNTS"

echo "== 完了: $OUT =="
```

いくつか補足します。

* `OrganizationAccountAccessRole` は、Organizationsでアカウントを作成すると自動で用意される管理ロールです。管理アカウントからassume-roleすることでメンバーアカウントの中を参照できます（招待で参加した既存アカウントには自動作成されないため、その場合は別途同名ロールの作成が必要です）
* スクリプトが実行するのはDescribe / List系とassume-roleのみで、**リソースを変更する操作は含みません**。assume-roleに失敗したアカウントは黙って飛ばさず「!!」付きで明記します
* CloudFormationスタックの一覧を含めているのは、削除をリソース単位ではなく**スタック単位**で行えるようにするためです。

実行すると `inventory-<日時>.txt` に全アカウント×対象リージョンのスナップショットが1ファイルで残ります。ステップ2のコストデータに見覚えのないリージョンの課金が現れた場合は、`REGIONS` にそのリージョンを追加して再実行してください。このファイルをAIに読み込ませてステップ2の課金データと突き合わせると、「このALBは背後のEC2が停止済みのため、実質的に使われていない」といった指摘がリソースID付きで返ってきます。

**4. AIが課金データと実測を突き合わせて、削減候補のリストを証拠つきで作成する**

ここからがAIの本領です。ステップ2の課金データ（JSON）とステップ3のリソース一覧（テキスト）をAIに読み込ませ、突き合わせを依頼します。人間がやることは以下のような依頼を1つ投げるだけです。

```
costs.shの出力（cost_*.json）と inventory-*.txt を読み込んで、
削減候補を月額の大きい順に一覧化してください。
候補ごとに「使われていないと判断した根拠」と「削除する場合のコマンド」を添えてください。
断定できないものは「要確認」として、確認方法を提示してください。
```

なお、この依頼文の内容は後述の`SKILL.md`に分析観点として織り込んであるため、Skills化後はこの依頼すら不要になり、`/aws-cost-report` と入力するだけで同じ形の削減候補リストが得られます。

AIが行う突き合わせは、大きく3パターンです。

| パターン | 検知するもの | 例 |
| --- | --- | --- |
| 金額の異常 | 前月比で急増したサービス、コストが跳ねた日 | 「今月からApp Runnerの課金が始まっている」 |
| 帳簿と現物のズレ | 課金はあるのに対応する実体が見えない（またはその逆） | 「EBSは2台分課金されているのに稼働中のEC2は1台」→停止中インスタンスの発見 |
| 存在するが使われていない | 固定費は満額発生しているのに、利用量連動の課金がほぼゼロ | 下記参照 |

多くのAWSサービスは「固定費」と「従量課金」の課金体系になっているため、リクエストの関連する従量課金の情報を確認することで利用状態を把握します。例えばApp Runnerの場合は以下のとおりです。

| 課金項目 | 実測 | 意味 |
| --- | --- | --- |
| プロビジョニング分のメモリ課金（Provisioned-GB-hours） | 720時間分（フル稼働） | 常時プロビジョニングで固定費が発生中 |
| リクエスト処理時のvCPU課金 | ほぼゼロ | リクエストがほぼ来ていない ✅ |

最終的にAIからは、次のような形の削減候補リストが返ってきます。

| 優先 | 対象 | 月額 | 根拠 | 削除手段 | 可逆性 |
| --- | --- | --- | --- | --- | --- |
| 1 | App Runner `hoge-api` | $13.7 | プロビジョニング720hに対しリクエスト処理ほぼゼロ | サービスの一時停止 or 削除 | 一時停止（PAUSED）なら再開可能 |
| 2 | 停止中EC2のEBS 30GB | $2.4 | インスタンスは3ヶ月前から停止したまま | スナップショット取得→削除 | スナップショットから復元可 |

ポイントは、AIに「消してよいか」を決めさせないことです。AIの仕事は**判断材料（金額・証拠・可逆性）を揃えるところまで**で、リストを見て削除を決めるのは人間です。そもそもステップ1で読み取り専用の権限しか渡していないため、AIには消せない構造になっています。

**5. 削除スクリプトをAIが生成し、人間が実行し、AIが結果を検証する**

削減候補リストから「消す」と決めたものについて、AIに削除スクリプトを生成させます。実行するのは管理者権限を持つ人間です（AIの読み取り専用プロファイルではそもそも実行できません）。

スクリプトを生成させる際は、以下の安全装置を必ず入れるようAIに指示しています。

* **削除前の最終確認**: 対象の現在の状態を表示してから削除に進む
* **可逆な操作を優先**: EC2は削除ではなくまず停止、EBSはスナップショット取得後に削除、Secrets Managerは復旧猶予つき削除（デフォルト30日、`restore-secret`で復元可）を選ぶ
* **削除前のバックアップ**: 復元手段のないもの（Route 53のレコードなど）は削除前にJSONで保存する
* **失敗時は黙って先に進まない**: 各ステップで失敗したら理由を表示してスキップする
* **実行後の自動検証**: 削除後に残存リソースを再列挙して結果を表示する

例としてApp Runnerを削除する場合、AIが生成するスクリプトは以下のような形になります。

■`delete_apprunner.sh`

```
#!/bin/bash
# App Runnerサービスの削除（確認→削除→検証）
# 実行者: 管理者権限を持つ人間（プロファイルは環境に合わせて変更）
set -u
PROFILE="admin"
REGION="ap-northeast-1"
SERVICE_ARN="arn:aws:apprunner:ap-northeast-1:111111111111:service/hoge-api/xxxxxxxx"

# 1) 削除前の最終確認: 対象の名前・状態・URLを目視で確認
aws --profile "$PROFILE" apprunner describe-service \
  --service-arn "$SERVICE_ARN" --region "$REGION" \
  --query "Service.[ServiceName,Status,ServiceUrl]" --output table
printf "このサービスを削除しますか? [y/N]: "
read -r ans
[ "$ans" != "y" ] && { echo "中止しました"; exit 0; }

# 2) 削除（非同期で進行する）
aws --profile "$PROFILE" apprunner delete-service \
  --service-arn "$SERVICE_ARN" --region "$REGION" \
  --query "OperationId" --output text || { echo "!! 削除失敗"; exit 1; }

# 3) 検証: 一覧から消えたことを確認（完了まで数分かかる）
sleep 60
aws --profile "$PROFILE" apprunner list-services --region "$REGION" \
  --query "ServiceSummaryList[].[ServiceName,Status]" --output text
echo "↑に対象サービスが表示されなければ削除完了"
```

## Skillsにまとめて再利用可能な状態にする

ここまでの手順のうち、分析部分（ステップ2〜4）は毎回同じ流れになります。そこでClaude Codeの\*\*Skills（スラッシュコマンド）\*\*として固めておきます。一度Skills化してしまえば、2回目以降は `/aws-cost-report` と入力するだけで、データ取得→分析→レポート生成→前回との差分報告までが毎回同じ手順で実行されます。

ディレクトリ構成はシンプルで、手順書（`SKILL.md`）とスクリプト群を置くだけです。

```
~/.claude/skills/aws-cost-report/
├── SKILL.md          # スキル定義（いつ発動するか・何をどの順でやるか・注意事項）
└── scripts/
    ├── costs.sh      # ステップ2: Cost Explorer APIから実額を取得
    └── waste_scan.sh # 補助: 未使用EIP・未接続EBS・停止中インスタンスなど定番の残骸をスキャン
```

`SKILL.md`には「いつ発動するか（トリガー）」「何をどの順で実行するか」「何に気をつけるか」を書いておきます。実際に使っているものが以下です。

```
---
name: aws-cost-report
description: AWSの利用金額・利用状況の確認やコスト削減の検討を求められたとき、
  または定期実行タスクとしてAWSコストレポートの生成を指示されたときに使う。
  トリガー: "aws-cost-report", "AWSコスト", "AWSの請求", "Cost Explorer"
---

# AWS Cost Report

全プロファイル・全リージョンのAWS利用金額（Cost Explorer実額）と利用状況を集計し、
削減案を提示するMarkdownレポートを生成する。

## 役割分担（最重要・この原則を破らない）
- AIが自分で実行してよいのは、読み取り専用プロファイル cost-report を使った
  参照系（Cost Explorer / Describe / List）のみ
- リソースの削除・停止・変更、assume-roleを伴う操作は、管理者権限の
  プロファイルが利用可能に見えても、理由を問わずAIは直接実行しない
- 管理者権限が必要な作業は、実行用スクリプトを ~/aws-cost-reports/pending/ に
  生成してパスを提示し、人間に実行を依頼する
- 人間の実行後、AIが結果を検証する（即時=Describe系で残存確認 /
  翌日以降=Cost Explorer日次で課金低下を確認）

## 前提
- 認証は専用プロファイル cost-report（読み取り専用IAMユーザー）を使う
- 認証エラーになった場合はキー失効の可能性。レポートに明記してユーザーに
  再発行を依頼する（認証情報の手動入力・表示は絶対にしない）
- Cost Explorer APIは$0.01/リクエスト（1回の実行で数円。脚注に記載する）

## 実行手順
1. costs.sh で実額コストを取得（月次×サービス別 / 日次 / アカウント別 / USAGE_TYPE別）
2. waste_scan.sh で削減余地をスキャン。完了後にエラーログを確認し、
   権限・認証エラーはレポートに明記する（握りつぶさない）
3. 分析: 急増検知（日次の前週比・サービス別の前月比+50%超）/
   常時稼働（LaunchTimeが数ヶ月前のまま=点けっぱなし候補）/
   残骸（停止中インスタンス・未接続EBS・未使用EIP）/
   未使用の証明（固定費が満額なのに従量課金がほぼゼロの構図をUSAGE_TYPE別に確認）/
   実測突き合わせ（最新の inventory-*.txt があれば課金データと突き合わせる）
4. ~/aws-cost-reports/YYYY-MM-DD.md に保存し、要約をチャットにも提示する
   定期実行時は前回レポートと比較し、差分を冒頭で報告する

## 注意事項
- 事実（Cost Explorer実額・リソース実測）と推定（削減見込み額）を
  区別して書く
- シークレット値や環境変数の中身はレポートに転記しない
- 指摘済みの削減候補が未対応のまま残っている場合は「繰り越し」として再掲する
```

ポイントは、分析の観点（急増検知のしきい値や点けっぱなしの判定基準）や運用ルールを書いておくだけで、AIが毎回その観点・ルールに沿って動きます。チームメンバーが実行しても同じ観点のレポートが出るため、コスト棚卸しの属人化も解消できます。

さらに定期実行（cronやClaude Codeのスケジュール実行）と組み合わせれば、週次でコストレポートが自動で届くようになり、今回のような「気づいたら費用が増えていた」を早期に検知できるようになります。

### この記事を読み込ませれば環境を再現できます

最後に、この仕組みの導入方法についてです。この記事の情報を渡せば再現可能かと思います。Claude Code（もしくはお使いのコーディングエージェント）にこの記事のURLを渡して、以下のように依頼してみてください。

```
この記事を読んで、同じ仕組みを私の環境に構築してください。
まずSkills（SKILL.mdとscripts）を生成し、IAMユーザー作成など
人間の作業が必要な部分は手順を提示してください。
```

この記事には主要スクリプトの全文、`SKILL.md`の内容、IAMポリシー、安全装置の設計方針まで掲載しているため（未掲載の`waste_scan.sh`は、`SKILL.md`の記述からAIが生成できます）、AIはこれを仕様書としてSkills一式を生成できます。人間が手を動かすのは、手順1のIAMユーザー作成とアクセスキーの登録（ここだけは必ずAIを介さずに行います）、そして生成されたスクリプトのレビューだけです。「AIに任せる仕組みそのものを、AIに作らせる」ところまで含めて、是非試してみてください。

## まとめ

この記事では、Fable 5にAWSコストの分析〜削減候補の特定〜削除スクリプトの生成を任せる検証を行い、半日の作業で月額費用を大幅に削減できることを確認しました。

リソースの一覧を眺めるだけでは削除の意思決定はできませんが、課金実績（USAGE\_TYPE別にみた固定費と従量課金の内訳）とリアルタイムなリソース確認の結果を突き合わせて「使われていないことの証拠」を揃えるところまでAIに任せられるのは大きな発見でした。分析部分はSkillsとして固めてあるため、実運用ではcronなどから週次で定期実行する形にすると、コストの急増にも早く気づけるようになりそうです。

これからクラウドコストの棚卸しをされる方の参考になれば幸いです。
