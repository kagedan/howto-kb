Claude Codeを毎日使っているのに、同じ確認、同じ整理、同じ下書きを毎回手で頼んでいませんか。

それはAIの能力不足ではありません。反復させる仕事の切り出し方が決まっていないだけです。

**`/loop`** を使うと、Claude Codeにプロンプトを繰り返し実行させられます。この記事では、`/loop` で仕事を自動で回すための5つの型を整理します。

> この記事は**全文無料（期間限定）**で閲覧できます。

## `/loop` は仕事を自動で回す入口です

![](https://assets.st-note.com/img/1781309161-G59Wftq20oOBL7KhpXzewAjV.png?width=1200)

`/loop` は、Claude Codeのセッション内で**同じプロンプトを繰り返し実行するための機能**です。

公式ドキュメントでは、`/loop` は**bundled skill**として説明されています。固定間隔で動かすことも、Claudeに次の実行間隔を選ばせることもできます。

たとえば、次のように使います。

```
/loop 30m 今日の作業メモを読み、未処理の確認事項を3つに絞ってください
```

ただし、ここで勘違いしてはいけません。

`/loop` は、何でも自動で終わらせる魔法ではありません。

公式ドキュメント上でも、`/loop` は現在のセッションに紐づく仕組みです。

新しい会話を始めるとタスクは切れます。再開時に戻せる場合もありますが、繰り返しタスクは作成から7日で失効します。

つまり、`/loop` は長期の無人運用ではなく、**今開いている仕事を自動で回すための短期エンジン**です。

長期で安定実行したいなら、**Routine、Desktop scheduled tasks、GitHub Actions**を検討します。

でも、今日の作業を自動で回すなら、`/loop` が一番軽い入口になります。

## `/loop` で回す仕事は5つに分けます

![](https://assets.st-note.com/img/1781309273-dOvD8KfcwzBWZLYCAsR65NT4.png?width=1200)

`/loop` の失敗は、最初から大きな仕事を任せることで起きます。

「**毎日の業務を全部見てください**」と頼むと、入力も判断基準も出力も曖昧になります。結果として、毎回違う動きになります。

---

先に分けるべき型は5つです。

この5つに分けると、`/loop` に何を頼むべきかが見えます。

収集型は、情報を拾います。

整理型は、散らかった情報を整えます。

生成型は、下書きを作ります。

検証型は、抜けや崩れを見つけます。

通知型は、人間が見るべき例外だけ返します。

---

重要なのは、人間の仕事を全部なくすことではありません。

人間が見る場所を減らすことです。

## 収集型はネタと変化を拾う

![](https://assets.st-note.com/img/1781309298-TlgduaSh31Y4kpLVcBGrewOo.png?width=1200)

最初に作りやすいのは収集型です。

収集型は、毎回見に行く情報をClaude Codeに拾わせる使い方です。ニュース、競合ページ、GitHub Issue、社内メモ、調査URLなどに向いています。

たとえば、AIニュース候補を拾うなら、次のように設計します。

```
/loop 60m 指定フォルダの調査メモとURL一覧を確認し、今日の記事候補を5件だけ抽出してください。

見るもの:
- research.md
- links.md
- 既存のnoteテーマ一覧

出力:
- 候補タイトル
- なぜ今読む価値があるか
- 既存記事との重複リスク

やらないこと:
- 本文を書かない
- 外部投稿しない
- 確認できない数字を断定しない
```

収集型のポイントは、集めすぎないことです。

AIに100件拾わせても、人間が読めません。最初から「**5件だけ」「重複を除く」「理由を1行で書く」**と絞ります。

収集型は、仕事を始める前の空白を減らします。

「**何から見ればいいか**」をAIに先に並べさせるだけで、朝の立ち上がりが速くなります。

## 整理型は散らかった情報を整える

![](https://assets.st-note.com/img/1781309319-fFTviplZ7DRwdSoJugWYs6xa.png?width=1200)

整理型は、バラバラの情報を次に使える形へ変える使い方です。

AI活用で意外と重いのは、生成より整理です。

メモ、CSV、リンク集、議事録、調査結果、ファイル名。これらが散らかると、次の作業が遅くなります。

整理型の例です。

```
/loop 45m drafts/ と research/ を確認し、未整理のメモを今日使う順に並べ替えてください。

やること:
- 重複メモをまとめる
- URLだけの行に用途を付ける
- 記事化できるものを上に置く
- 不明点を質問リストにする

出力:
- 今日使うメモ
- 後回しメモ
- 確認が必要なこと

やらないこと:
- 元ファイルを削除しない
- 重要な表現を勝手に短縮しない
```

整理型は、派手ではありません。

でも、仕事全体の速度を上げます。

AIに生成させる前に、材料が整っているかどうかで品質が変わるからです。

整理型を回すと、人間は「探す時間」から解放されます。

## 生成型は下書きを先に作る

![](https://assets.st-note.com/img/1781309334-SZCPGwg0kYuKiIHjRxD326d8.png?width=1200)

生成型は、毎回作る下書きを先に出しておく使い方です。

向いているのは、完成物ではなく初稿です。

たとえば、**X投稿案、note見出し案、返信文、議事録要約、タスク分解、PR説明文**です。

生成型の例です。

```
/loop 90m 今日の調査メモから、X投稿案を3本だけ作ってください。

条件:
- 1本目は実務ノウハウ
- 2本目は失敗回避
- 3本目は記事誘導

出力:
- 投稿本文
- 狙う読者
- 投稿前に確認すべき事実

やらないこと:
- 投稿しない
- 実績や数字を作らない
- 出典未確認の断定をしない
```

生成型は、自動投稿まで進めない方が安全です。

最初は下書きまでで十分です。

人間はゼロから書くのではなく、候補を選び、直し、出すだけになります。

ここで大事なのは、出力数を絞ることです。

30案出すより、3案に絞った方が使えます。`/loop` は量産装置ではなく、作業開始を速くする装置として使います。

## 検証型は公開前と変更後を見張る

![](https://assets.st-note.com/img/1781309363-XTZWewf43UpN5dEDL9rRYaAk.png?width=1200)

**検証型は、`/loop` と相性がかなり良い使い方**です。

理由は、検証には明確な基準があるからです。

**リンク切れ、未確認の数字、表記ゆれ、差分、テスト失敗、画像抜け、ファイル保存漏れ**。こういう仕事は、AIに繰り返し見てもらいやすいです。

検証型の例です。

```
/loop 30m 作業中の記事フォルダを確認し、公開前チェックで引っかかる点だけ報告してください。

見るもの:
- index.md
- plan.md
- brand.md
- assets/prompt.md

確認すること:
- タイトルと本文の約束がズレていないか
- 未確認の数字がないか
- 図解プレースホルダが足りているか
- CTAが重複していないか

出力:
- 重大な修正点
- 軽微な修正点
- 公開前に人間が見るべき点

やらないこと:
- 本文を勝手に大きく書き換えない
- ファイルを削除しない
```

検証型は、記事制作、開発、LP改善、資料作成のどれにも使えます。

特に強いのは、変更後に崩れていないかを見る仕事です。

人間は作ることに集中できます。`/loop` が横で崩れを拾います。

## 通知型は人間が見るべき例外だけ返す

![](https://assets.st-note.com/img/1781309383-lLfOgizypetnThxcm6Sb8XKZ.png?width=1200)

通知型は、全部を報告させない使い方です。

AI自動化で失敗する人は、AIに大量の報告を出させます。大量の報告は、結局人間の仕事を増やします。

通知型では、通常時は短く、異常時だけ詳しく返します。

例です。

```
/loop 60m 進行中の作業ログを確認し、人間の判断が必要なものだけ報告してください。

通常時:
- 「異常なし」と1行で返す

報告する条件:
- 出典未確認の数字がある
- 期限に近い未完了タスクがある
- 同じエラーが2回以上出ている
- 投稿、送信、削除、支払いなど人間承認が必要な行動がある

出力:
- 判断が必要な理由
- 放置した場合のリスク
- 次に人間が選ぶ選択肢
```

通知型の本質は、通知を増やすことではありません。

通知を減らすことです。

AIが見る量を増やし、人間が見る量を減らします。

この設計にすると、`/loop` は作業者ではなくフィルターになります。

## 最初の1本は小さく作る

![](https://assets.st-note.com/img/1781309398-rMBDUc8xjIo2LsAtpCG9V0TO.png?width=1200)

最初の `/loop` は、小さく作るべきです。

おすすめは、収集型か検証型です。

理由は、失敗しても被害が小さいからです。

最初から送信、投稿、削除、決済、PR作成まで任せる必要はありません。まずは**「見る」「まとめる」「報告する」**だけで十分です。

最初のテンプレはこれです。

```
/loop 30m 次の対象を確認し、人間が見るべきことだけ短く報告してください。

目的:
-

対象:
-

確認すること:
-
-
-

出力:
- 異常なしなら1行
- 判断が必要なら理由、リスク、次の選択肢

やらないこと:
- 投稿しない
- 送信しない
- 削除しない
- 支払いしない
- 推測で事実を断定しない
```

毎回このプロンプトを書くのが面倒なら、**`.claude/loop.md`** に標準プロンプトを書きます。

公式ドキュメントでは、`.claude/loop.md` はプロジェクト単位のデフォルトプロンプトとして扱われます。

ホームディレクトリ側に置けば、プロジェクトに専用ファイルがない場合のデフォルトにもできます。

ただし、`loop.md` は複数タスクの一覧ではありません。

裸の `/loop` を打った時に使う、1つの標準プロンプトです。

最初は次のように短くします。

```
作業中のファイルと会話の流れを確認し、次に人間が見るべきことだけ報告してください。

通常時は「異常なし」と1行で返してください。

報告する条件:
- 未確認の事実がある
- 作業が止まっている
- ユーザー確認が必要
- 削除、送信、投稿、支払いに関わる判断がある

勝手にファイルを削除しないでください。
勝手に投稿、送信、支払いをしないでください。
```

`/loop` は、AI自動化OSの中で一番すぐ試せる部品です。

大きな自動化を作る前に、小さな仕事を1つ回してください。

収集する。整理する。生成する。検証する。通知する。

この5つのどれか1つが回り始めると、Claude Codeは**「毎回呼ぶ相手」から「仕事を進める仕組み」**に変わります。
