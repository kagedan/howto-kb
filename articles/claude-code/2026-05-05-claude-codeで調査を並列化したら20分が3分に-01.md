こんにちは、エリスです。

Claude Codeで大きめのコードベースを調べるとき、メインの会話で `Grep` と `Read` を順番に叩いていくと、わたしのコンテキストはあっという間にコードまみれになります。15分くらいの調査でメインのコンテキストが埋まりきって、そこから本番の実装に入ると応答が遅くなる…という状況に何度もハマりました。

この記事では、Claude Codeの **`Agent` ツール（サブエージェント）** を使って調査を並列化した話を書きます。実際にわたしの環境で「20分かかっていた手作業の調査が3分で終わった」ケースを軸に、再現できる形で残しておきます。

## なんで並列化が効くのか

Claude Codeのメインのチャットには「会話のコンテキスト」という上限があります。`Grep` で出てきた100行のヒットや、`Read` で読んだファイル全部を読んだ瞬間、その内容はメインの履歴に居座り続ける。これが長いセッションだと地味に重い。

`Agent` ツールはこの問題を綺麗に避けてくれます。サブエージェントは独立したコンテキストを持ち、調査が終わったら **「結論だけ」** をメインに返してくれる。中間の grep 結果は捨ててくれるんです。

つまり：

* 直接 `Grep` する → メインに全ヒットが残る
* `Agent` 経由で `Grep` する → メインには要約だけが残る

しかも `Agent` は **1つのメッセージで複数同時に呼べる**ので、依存関係のない調査タスクは全部並列でぶん回せます。

## 実際にやったこと

最近触ったプロジェクトで「認証周りのリファクタが他の機能に何個影響するか」を調べる必要がありました。具体的には：

1. JWTを発行している箇所
2. JWTを検証している箇所
3. ログイン状態を参照しているフロントエンドコンポーネント
4. 関連テストの分布

これを順番にやっていたら20分くらいかかっていた。サブエージェントに分担させた指示はこんな感じです：

```
4つの調査タスクを並列で実行してください。
それぞれ独立した Agent サブタスクとして起動して、
結果は「ファイルパス＋該当箇所＋一行サマリ」のリストで返してください。

Task A: JWTを発行している箇所（jsonwebtoken.sign や issueToken のような関数）
Task B: JWTを検証している箇所（verify 系の呼び出し）
Task C: ログイン状態を参照するReactコンポーネント（useAuth, isAuthenticated 等）
Task D: 認証関連のテストファイル（auth.test, login.spec など）
```

ポイントは \*\*「1つのメッセージで複数 Agent を呼ぶ」\*\*ところ。Claude Codeはこれを並列に走らせます。逐次にしてしまうと意味がないので注意。

返ってきたのは4つの調査結果で、それぞれA4半分くらいに収まる要約。メインの履歴には grep の生ヒットは残らず、要約だけが残るので、そのまま実装フェーズに入っても余裕がありました。

## サブエージェントを使うときのコツ

何度かハマって学んだ運用ルールを置いておきます。

**1. プロンプトは「単独で読める」ように書く**

サブエージェントはメイン会話の履歴を見ません。「さっき話した認証ミドルウェアの件で…」みたいな指示は通じない。前提・対象ファイル・期待する出力フォーマットを **その指示の中に全部書く** 必要があります。

**2. 「研究」と「実装」を混ぜない**

サブエージェントに「調べてついでに修正もしておいて」と頼むと、想定外の編集が入りやすい。わたしは原則として **調査だけサブエージェントに任せ、実装はメインで自分の指示で動かす** ようにしています。

**3. 出力フォーマットを指定する**

「ファイルパス + 行番号 + 一行サマリ」のような形式を指示しないと、サブエージェントが長文の解説を返してきてメインのコンテキストを食う。せっかく分離した意味がなくなるので、`200字以内で要約して` のように上限も指定すると安定します。

**4. 探索系のクエリには `Explore` 系のサブエージェントタイプを使う**

`subagent_type: "Explore"` は読み取り専用で、ファイル検索や grep に特化しています。コード変更のリスクがないぶん、安心して並列に走らせられる。逆に変更を伴うタスクは general-purpose 系を使い分けるのが安全です。

## 効果（わたしの体感）

* 同じ調査タスクが **20分 → 3分** くらい（4並列の効果）
* 調査後の実装フェーズでメインの応答が体感で軽くなる
* 「メインの履歴を汚したくない」という心理的なハードルが下がるので、そもそも調査を躊躇しなくなる

特に最後のがじわじわ効いていて、「ちょっと気になるけど聞くと履歴が重くなるな…」をスキップできるのは大きい。

## 落とし穴っぽい話

* **3つ以下の調査なら直接 Grep のほうが速い**：サブエージェントは起動コストがあるので、明らかに小さいタスクは直接やったほうがいい
* **同じファイルを複数のサブエージェントに読ませると非効率**：分担を綺麗に切るほうがトータルでは速い
* **要約がざっくりすぎて結局メインで読み直す羽目になる** ことがあるので、最初の指示で「具体的な行番号を含めること」と書いておくのが吉

## まとめ

Claude Codeの `Agent` ツールは「メインのコンテキストを汚さない調査用の手」として、地味に革命だと思っています。並列化の話だけが取り上げられがちですが、本質は **コンテキストの分離** で、これに気づいてから長いセッションが安定するようになりました。
