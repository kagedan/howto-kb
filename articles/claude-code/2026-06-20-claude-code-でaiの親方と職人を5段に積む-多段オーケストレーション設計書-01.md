Claude Code にサブエージェントは使わせている。けれど、作業が進むほどメインの会話が重くなり、途中で自分が何をしていたかさえ分からなくなる。そんな経験はありませんか。

多くの人は、サブエージェントを「並列でこなして速くするための道具」だと思っています。私は、逆だと考えています。

私がサブエージェントを何段にも積む目的は、速さではありません。メインの会話を汚さないため、つまり「文脈を隔離する」ためです。

この置き方に変えてから、一周の作業でメインに流れ込む情報量が体感で半分以下になり、長い工程でも最後まで筋を見失わなくなりました。相棒であるAIに職人を雇わせ、その職人にさらに弟子を取らせる。Claude Code は、その階層をメインの下に何段か積めます。本稿は、その積み方の設計書です。

---

## なぜ「深さ」が効くのか

サブエージェントは、メインとは別の会話で動く相棒です。自分専用の作業机を持っていて、机の上でどれだけ散らかしても、戻ってくるのは「片づいた結論」だけ。途中の散らかりはメインに見えません。

これが「文脈の隔離」です。

ここで一段では足りない場面が出てきます。たとえばレビューの相棒が、気になる指摘を10個見つけたとき。1個ずつ裏取りさせると、その裏取りのログがレビュー相棒の机を埋めてしまう。そこで、レビューの相棒にもう一段下の職人を雇わせます。裏取りはその職人の机でやらせ、レビュー相棒には「確定した指摘」だけを返させる。

階層を、こう呼び分けます。

* **L0**: メインの会話（あなた + 司令塔としての Claude Code）
* **L1**: フェーズの親方（調査統括、実装統括、レビュー統括）
* **L2**: 手を動かす職人（個別調査、個別実装、個別検証）
* **L3〜L5**: さらに割れる深い作業のときだけ使う予備の段

私が触っている範囲では、メインの下にサブエージェントをおおむね5段まで積めます。最下段は、それ以上子を呼べない打ち止めになります。この上限値は変わりうるので、正確な数は公式ドキュメントで確認してください。

大事なのは、5段まで「積める」ことと、5段「積むべき」ことは別だという点です。ここを取り違えると、設計はすぐ破綻します。

## Step 1：業務領域の分解

題材は、私が普段やっている「アプリやツールに機能を1つ足して、検証してリリース判断まで持っていく開発の一周」です。一人 + AIチームで回す前提で、工程に割ります。

所要時間は私の体感です。詰まり所も、私が実際に転んだ場所です。

### 工程1：要望の言語化

### 工程2：既存コード・仕様の調査

### 工程3：設計（どう作るか・影響範囲）

### 工程4：実装

### 工程5：レビュー（品質・セキュリティ）

### 工程6：テスト実行と修正

### 工程7：ドキュメント更新

* 所要時間：10〜20分
* 詰まり所：後回しにして、結局更新されない

### 工程8：動作の最終検証

### 工程9：リリース判断と記録

### AI化の優先順位（3段階）

「即着手」と「中期」と「AI不要」で割ります。判断基準は、出力が大量に出て本題から逸れやすいか、です。

* **即着手**：工程2（調査）、工程5（レビュー）、工程6（テスト）。どれも出力が大量で、メインを汚す筆頭。隔離の恩恵が最も大きい
* **中期**：工程3（設計）、工程4（実装）、工程7（ドキュメント）。前工程の要約を入力にできれば任せられる
* **AI不要**：工程9（リリース判断）。最後にOKを出すのは私の仕事。AIは判断材料を揃えるところまで

優先順位を見ると、隔離が効くのは「大量に喋る工程」だと分かります。これが多段の置き場所を決める基準になります。

## Step 2：全体フロー設計

工程を、誰の机でやらせるかまで含めて1本につなぎます。記号「→」で読んでください。括弧内が、その作業を持つ相手です。

```
［入力: 機能の要望（あなた）］
  → ［L0: 要望を仕様に言語化（メインの Claude Code）］
  → ［L1: 調査統括が領域を分ける（研究の親方）］
        → ［L2: 領域ごとに並列で読む（調査職人 × 数人）］
        → 親方が要約だけL0へ返す
  → ［L0: 要約をもとに設計を1枚にまとめる］
  → ［人: 設計をざっと確認（あなた）］
  → ［L1: 実装統括が変更を割る（実装の親方）］
        → ［L2: ファイル単位で書く（実装職人）］
  → ［L1: レビュー統括が指摘を仕分ける（レビューの親方）］
        → ［L2: 重い指摘を1件ずつ裏取り（検証職人）］
        → 親方が確定指摘だけL0へ返す
  → ［L0: 確定指摘を直す → テストを別机で走らせる］
  → ［人: 動作の最終確認とリリース判断（あなた）］
  → ［出力: リリースと記録］
```

この並びにした設計判断を、3つ挙げます。

**1つ目。大量に喋る工程を、必ず一段下げています。** 調査・レビュー・テストは出力が多い。これをメインで直接やると、本題に戻る頃には会話が文脈で埋まっています。下の机に追い出して、戻すのは要約だけにする。だから調査とレビューには親方を置きました。

**2つ目。「人が確認する点」を2か所だけ残しています。** 設計のあとと、リリース判断の直前です。AIに全部任せず、方向が大きく決まる手前で私が一度見る。ここを削ると速くはなりますが、間違った方向に全力疾走するリスクが上がります。仕組みで勝つというのは、止める場所を決めることでもあります。

**3つ目。ネストを使うのは「親方の机も埋まるとき」だけにしています。** 調査もレビューも、相手が見つけるものが多い工程です。一段では親方の机が埋まる。だから二段目を許しました。逆に実装は、基本は親方の下に職人を横並びで置くだけで、深くは積みません。深さは、必要が出た工程にだけ与えます。

ここまでが全体像です。後半では、この親方と職人を実際にどう定義するか、コピペできる定義ファイルと、机から机へ情報を渡すプロンプトの型を工程ごとに出します。「子に何を渡し忘れると壊れるか」も、工程ごとに書きます。

なお、ここで触れている「Claude Code で AI チームを動かす」設計の全体像は、Relmea のサイトで運用事例として公開しています。

さらに踏み込んだ実装は、教材「Relmea AI Ops Vol.2（開発編）」に体系化しています。コードを書かず「作って」と伝えて、AIチームでアプリを0→1で作り切るための設計書です。
