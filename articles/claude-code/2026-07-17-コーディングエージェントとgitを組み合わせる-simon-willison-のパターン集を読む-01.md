この記事では、Simon Willison 氏のガイド [Using Git with coding agents](https://simonwillison.net/guides/agentic-engineering-patterns/using-git-with-coding-agents/) を日本語で要約し、コーディングエージェントとGitを組み合わせて使うための考え方を整理します。Claude Code のようなエージェントを普段の開発に使っていて、Gitまわりをもっと任せたいと感じている方を対象としています。

## このガイドの主張

Simon Willison 氏は、AIエージェントを活用したソフトウェア開発（agentic engineering）のパターンを継続的にまとめています。その中の一本がこのガイドです。

主張はシンプルです。**コーディングエージェントはGitの用語や操作に精通しているので、あなたがコマンドの構文を覚えていなくてもGitの機能をフルに引き出せる**、というものです。つまり `git reset --soft HEAD~1` のような呪文を暗記する代わりに、「直前のコミットを取り消して」と伝えればよいという発想です。

エージェントに任せるのは操作の実行で、あなたが握るのは意図と最終的な判断です。この役割分担を前提に、ガイドではいくつかのパターンが紹介されています。以下では、それらをテーマごとにまとめ直します。

## 基本の操作を言葉で頼む

もっとも基本的なのは、日々のGit操作を自然な言葉でエージェントに頼むパターンです。

* リポジトリの初期化を「このプロジェクトをGitで管理して」と頼み、`git init` を実行してもらう
* 作業の区切りで「この変更をコミットして」と伝え、内容に合ったメッセージ付きで記録してもらう
* 「`username/repo` をGitHubのリモートに追加して」と頼み、リモート設定を任せる

構文を思い出す手間がなくなり、やりたいことだけに集中できます。

## git log を使ってコンテキストを読み込ませる

このガイドでとくに実用的なのが、**変更履歴をエージェントのコンテキストとして読み込ませる**パターンです。

エージェントは新しいセッションを始めるたびに、前回までのやり取りを覚えていません。そこで、次のように直近の履歴を確認してもらいます。

* 「今日の変更を確認して」
* 「直近3つのコミットを見て」

こう伝えると、エージェントは `git log` を実行し、これまでに何がどう変わってきたのかを一気にコンテキストへ取り込みます。原文でも、これは新しいセッションを始めるときの有効な手立てとして紹介されています。履歴を読み込ませたあとは、その流れを踏まえて「次はこう直したい」「このコードはなぜこうなっているのか」といった会話を、より的確に進められます。

## 面倒な状況を任せる

Gitで手が止まりがちな場面こそ、エージェントに任せる価値があります。

* マージとリベースのどちらで取り込むか迷ったとき、「main の変更を取り込む方法を相談したい」と投げて選択肢を出してもらう
* コンフリクトやステージングの取り違えで混乱したとき、「このGitの混乱を解消して」と頼む
* 消してしまったように見えるコードを、「〜をするコードを探して復元して」と頼み、reflog や別ブランチから見つけてもらう

「自分では手に負えないGitの状態」を言葉で丸ごと渡せるのは、エージェントならではの使い方です。

## デバッグに使う

原因の切り分けにもGitの機能が役立ちます。ガイドでは `git bisect` を例に挙げています。

「`git bisect` を使って、このバグがいつ入ったのか特定して」と頼めば、bisect の構文を覚えていなくても、問題のコミットを二分探索で効率よく突き止められます。

## 履歴を整える

コミット履歴を後から整える操作も、言葉で頼めます。

* 「直前のコミットを取り消して」（`git reset --soft HEAD~1` を覚えなくてよい）
* 「さっきのコミットから `uv.lock` を外して」
* 「直近3つのコミットを、より良いメッセージでまとめて」
* コードの一部を別リポジトリに切り出しつつ、「作者やコミット日時を引き継いで、同じような履歴を再現して」

履歴はプロジェクトの記録であり、あとから読む人へのメッセージでもあります。エージェントに整えてもらいつつ、どう残すかはあなたが決める、という関係が保てます。

## 読んでみての所感

私がこのガイドに行き着いたのは、`git status` のリネーム検出について書いていたときでした。ファイルを移動しただけの変更が「削除」と「追加」に分かれて表示されると読みにくい、という話を掘っていくうちに、**読みやすい履歴は人間だけでなくエージェントにとっても価値がある**のではないか、と考えるようになったのがきっかけです。

その視点で読むと、いちばん刺さったのは「`git log` でコンテキストを読み込ませる」パターンでした。エージェントに履歴を読ませるのなら、その履歴自体が読みやすいほど効くはずです。リネームがリネームとして残っているか、コミットメッセージに理由が書いてあるか。そういう地味な整え方が、そのままエージェントの理解の質につながる。自分が普段書いている記事の題材と、このガイドの話がきれいにつながった感覚がありました。

もうひとつ気に入ったのは、ここで紹介されているパターンが**新しいツールを入れなくても今日から試せる**ことです。特別なフレームワークではなく、いつものGitとエージェントの組み合わせでできる。私はふだん記事を書くときに、まず手元で実際に動かして確かめるのですが、これらのパターンも同じ感覚で、自分のリポジトリですぐ試せるのがうれしいところでした。

一方で、コンフリクト解消のように履歴を書き換える操作を丸ごと任せるのは、正直まだ少し身構えます。だからこそ、原文が「操作は任せても、どう残すかの判断は自分が握る」と釘を刺しているのは、そのとおりだと感じました。エージェントに任せる範囲と、自分で確認する範囲の線引きは、これから使いながら見つけていきたいところです。

## まとめ

このガイドの要点を整理します。

* コーディングエージェントはGitの用語と操作に精通しているので、構文を暗記せず意図を伝えるだけで使える
* 日々のコミットやリモート設定は、自然な言葉で頼める
* 「直近のコミットを見て」と頼むと `git log` でコンテキストを読み込め、新しいセッションの出発点になる
* コンフリクト解消やコード復元、`git bisect` でのバグ調査など、詰まりやすい場面ほど任せる価値がある
* 履歴の整理も言葉で頼め、どう残すかの判断は自分が握る

構文を覚えることから、意図を伝えることへ。Gitとの付き合い方が少し変わるガイドです。詳しくは原文を読んでみましょう。
