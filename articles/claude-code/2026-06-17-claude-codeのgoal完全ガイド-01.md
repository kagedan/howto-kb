Claude Codeに大きな作業を頼んだものの、途中で止まるたびに「続けて」「残りも実装して」「テストが通るまで修正して」と**何度も入力した経験**はないだろうか。

Claude Codeはもともと、1回の指示の中でファイルを調査し、コードを書き、コマンドを実行し、テストするところまで自律的に進められる。しかし、Claude自身が「今回のターンはここまで」と判断すると、作業が完全には終わっていなくてもユーザーへ制御を返すことがある。

この問題を解決する公式機能が、/goalである。/goalを使うと、単に作業内容を指示するのではなく、**「何が確認できたら終了してよいか」という完了条件**をClaude Codeに設定できる。Claudeはその条件が満たされるまで、複数ターンにわたって作業を続ける。

本記事では、/goalの仕組み、基本操作、失敗しない条件の書き方、Auto Modeとの違い、安全な運用方法、実用テンプレートまで整理する。

## まず結論

/goalを理解するうえで最も重要なのは、次の一文である。

> /goalは、Claudeに「やること」を渡す機能ではなく、「いつ終わってよいか」を渡す機能である。

たとえば、通常の指示ではこう書く。

```
認証機能のバグを修正してください。
```

これを/goal向けにすると、次のようになる。

```
/goal 認証機能のバグが修正され、認証関連テストがすべて成功し、
npm run typecheck と npm run lint がexit 0で完了している
```

前者は「修正する」という作業を依頼しているだけだが、後者には「テスト、型チェック、lintがすべて成功したら終了」という判定基準がある。

/goalに向いているのは、テストがすべて通るまでバグを修正する、古いAPIの呼び出しをすべて新APIへ移行する、型エラーを0件にする、指定されたIssueをすべて処理する、ファイルを一定サイズ以下に分割する、ドキュメントと実装の不一致を解消する、JSONやデータを指定スキーマに準拠させる、といった仕事である。

一方で、「良いデザインになるまで」「面白い記事になるまで」「医学的に完璧になるまで」のように、合否が主観に依存する仕事には向いていない。

## /goalの内部では何が起きているのか

通常のClaude Codeは、1回のユーザー入力に対して複数の操作を行い、最後に回答を返す。

```
ユーザーが指示
    ↓
Claudeが調査・編集・テスト
    ↓
Claudeがターン終了を判断
    ↓
ユーザーの次の入力を待つ
```

/goalを設定すると、Claudeがターンを終了しようとした時点で、別の小型モデルによる完了判定が入る。

```
ユーザーが /goal を設定
    ↓
Claudeが調査・編集・テスト
    ↓
Claudeが1ターンを終了
    ↓
小型の評価モデルが完了条件を確認
    ├─ 未達成 → 理由をClaudeへ返し、次のターンを開始
    └─ 達成済み → Goal achievedとして終了
```

評価には、通常はHaikuのような小型で高速なモデルが使われる。作業を担当するSonnetやOpusとは別のモデルが、「本当に完了条件を満たしたか」を確認する構造である。これにより、作業を行ったClaude自身が「たぶん終わった」と判断するだけではなく、別の評価器によるチェックが入る。

ただし、ここには重要な制約がある。**評価モデルは、ファイルを直接読んだり、テストを実行したり、Webサイトを確認したりできない。**

評価モデルが見られるのは、Claudeが会話ログに残した情報だけである。そのため、Claude自身がテストを実行し、結果を会話上に示している必要がある。

たとえば、次の条件は評価しやすい。

```
npm test がexit 0で完了している
```

Claudeが実際にnpm testを実行すれば、その結果がログに残るからだ。一方、次の条件は不安定である。

```
コードが十分に高品質になっている
```

「十分に高品質」の定義も、確認方法も、会話ログ上の証拠も曖昧だからである。

## /goalを使うための条件

/goalにはClaude Code v2.1.139以降が必要である。CLIを使っている場合は、次のコマンドでバージョンを確認できる。

```
claude --version
```

古い場合は、次のコマンドで最新版へ更新する。

```
claude update
```

また、/goalはClaude CodeのHooks機構を利用している。そのため、Workspaceの信頼確認を承認している必要があり、設定でHooks全体が無効化されている環境では利用できない。

Claude Code Desktopでも、通常の入力欄に/goalと入力して使用できる。

## 最も基本的な使い方

基本構文は非常に単純である。

```
/goal [完了条件]
```

たとえば、認証関連のテストをすべて通したい場合はこうなる。

```
/goal test/auth配下のテストがすべて成功し、lintエラーが0件になっている
```

/goalを入力すると、その条件自体が作業指示として扱われ、すぐに作業が始まる。別途「この作業を開始してください」と送る必要はない。

1つのセッションで有効にできるGoalは1つだけである。Goalが有効な状態で新しい/goalを設定すると、以前のGoalは新しいものに置き換えられる。

### 現在の状態を確認する

引数なしで/goalを実行する。

```
/goal
```

現在の条件、経過時間、評価されたターン数、使用トークン、直近の評価理由などを確認できる。

### 途中で停止する

```
/goal clear
```

次の表現も同じ意味で使える。

```
/goal stop
/goal cancel
/goal off
/goal reset
/goal none
```

### セッション終了後に再開する

Goalが未達成の状態でセッションを終了した場合、--resumeまたは--continueで再開するとGoalも復元される。

```
claude --continue
```

```
claude --resume
```

ただし、再開時にはターン数、経過時間、トークン計測の基準がリセットされる。すでに達成または解除したGoalは復元されない。

### 非対話モードで実行する

ターミナルから1回のコマンドとして実行することもできる。

```
claude -p "/goal すべてのテストが成功し、型エラーが0件になっている"
```

途中で停止したい場合はCtrl+Cを使う。

## 条件の書き方で結果の大部分が決まる

優れたGoal条件には、基本的に次の3要素がある。

1. 最終的にどうなっていればよいか
2. 何を実行して確認するか
3. 何を変更してはいけないか

たとえば、次のGoalは不十分である。

```
/goal ログイン機能を直す
```

「直った」の定義がなく、テスト方法も、変更してよい範囲も分からない。

改善すると次のようになる。

```
/goal ログイン失敗時に500エラーではなく401が返るよう修正され、
認証関連テスト、npm run typecheck、npm run lintがすべてexit 0で完了している。
既存テストの削除、スキップ、期待値の弱体化は行わない
```

この条件には、終了状態、確認方法、禁止事項が含まれている。

良い条件の例は次の通りである。

```
/goal TypeScriptの型エラーが0件になり、
npm run typecheckとnpm testがexit 0で完了している
```

```
/goal src/api内に残っている旧APIの呼び出しが0件になり、
新APIへの移行後もビルドと関連テストが成功している
```

```
/goal 未処理のmigration-readyラベル付きIssueが0件になり、
各Issueに対応するテストと変更内容が記録されている
```

一方、次のような条件は避けた方がよい。

```
/goal コードをきれいにする
```

```
/goal 良い感じに完成させる
```

```
/goal 品質が十分高くなるまで改善する
```

```
/goal バズる記事を書く
```

主観的な条件では、評価器が早すぎる段階で「達成」と判断したり、反対に永遠に未達成と判断したりする。

## 実用的なGoalテンプレート

複雑な仕事では、次の構造にすると安定しやすい。

```
/goal [最終的に達成したい状態]

context:
- [プロジェクトや作業の背景]
- [使用技術や現在の問題]

done when:
- [測定可能な完了条件1]
- [測定可能な完了条件2]
- [実行すべき検証コマンド]

do not:
- [変更してはいけないファイル]
- [禁止する操作]
- [避けるべき実装方法]

report:
- 変更内容
- 検証結果
- 残っている制約
をfinal-report.mdに記録する

or stop after 12 turns
```

context:やdone when:はClaude Codeの特別な構文ではない。人間とClaudeが条件を読みやすくするための整理方法である。

Goal条件は最大4,000文字まで指定できるが、長ければよいわけではない。関係のない背景説明を大量に入れるより、必要な終了条件を明確にする方が重要である。

## 実行時間に上限を付ける

Goalが終わらない事態を避けるため、公式ドキュメントではターン数や時間を条件に加える方法が案内されている。

```
/goal すべてのテストが成功する、または10ターン経過したら停止する
```

```
/goal 移行作業を完了する、または30分経過したら停止する
```

英語なら次のようにも書ける。

```
or stop after 10 turns
```

ただし、これはOSレベルの強制終了ではない。ターン数や時間も会話ログをもとに評価モデルが判定する条件である。

絶対に30分でプロセスを終了させたい場合は、CIのタイムアウトや外部スクリプトなど、Claude Codeの外側にも制限を設ける必要がある。

## /goalとAuto Modeの違い

/goalとAuto Modeは、似ているようで解決する問題が異なる。

機能省略するもの/goalターン終了後に人間が「続けて」と入力する作業Auto Modeファイル編集やコマンド実行ごとの承認両方を使用ターン間の入力と、途中の承認の両方

/goalだけを使っても、権限確認が必要な操作ではClaudeが承認待ちになる。反対にAuto Modeだけを使っても、Claude自身がターンを終了すればそこで止まる。したがって、長時間の自律実行では両者を組み合わせると効果が大きい。

CLIでAuto Modeから起動する場合は次のコマンドを使う。

```
claude --permission-mode auto
```

Claude Code Desktopでは、送信ボタン付近にあるPermission Modeの選択欄からAuto Modeを選ぶ。

古い解説記事などで見かける次のコマンドは、現行の公式コマンドではない。

```
claude --approval-mode full-auto
```

Auto Modeは、単純にすべての操作を無条件で許可するモードではない。別の安全判定モデルが操作を確認し、依頼範囲を逸脱する処理、本番デプロイ、force push、mainへの直接push、大量削除、外部への機密情報送信などを止める。

ただし、Auto ModeはResearch Previewであり、安全を保証するものではない。機密情報、課金、本番環境、データベース破壊、インフラ変更が関係する仕事では、人間による確認を残すべきである。

Auto Modeより慎重に進めたい場合は、acceptEditsモードも選択肢になる。

```
claude --permission-mode acceptEdits
```

これは作業フォルダ内のファイル編集などを自動承認するが、それ以外のコマンドでは確認を残す。個人開発では、いきなりAuto Modeを使うより、まずacceptEditsと/goalの組み合わせから始める方が安全である。

## 最も安全で再現性が高い使い方

実用上は、次の順序が安定する。

### 1. 先にPlan Modeで方針を作る

```
/plan 認証処理の不具合原因を調査し、修正方針と影響範囲を提案してください
```

Plan Modeでは、Claudeはコードを調査するが、原則として実装を変更しない。

大きな作業をいきなり/goalへ渡すと、誤った方針のまま高速に進む可能性がある。まず計画を確認し、その方針が妥当だと判断してからGoalを設定する。

### 2. 専用ブランチまたはWorktreeを使う

複数の作業を並列化するときは、同じフォルダを複数セッションで同時編集してはいけない。ファイルの上書きや競合が起きる。

CLIではWorktreeを使える。

```
claude --worktree fix-auth
```

別の作業は別Worktreeで起動する。

```
claude --worktree update-docs
```

Claude Code Desktopでは、新規セッションを独立したWorktreeで扱える仕組みが用意されている。

### 3. Goalを設定する

```
/goal 認証バグが修正され、関連テスト、型チェック、lintがすべて成功している。
既存テストを削除・スキップ・弱体化しない。
本番環境には接続しない。
または12ターンで停止する
```

### 4. 最後は人間が差分を見る

```
git diff
```

テストが通ることと、正しい実装であることは同じではない。過剰な変更、不要な依存関係、テストの形骸化、セキュリティ上の問題がないかを確認する。

### 5. 問題なければコミットする

/goalは作業を継続させる機能であり、レビューを不要にする機能ではない。

## CLAUDE.mdとの組み合わせ

プロジェクトルートのCLAUDE.mdには、毎回説明したくないプロジェクト共通ルールを書ける。

```
# Project rules

## Commands
- Type check: npm run typecheck
- Lint: npm run lint
- Tests: npm test

## Coding rules
- TypeScriptのany型は禁止
- APIエラーをそのままクライアントへ返さない
- 既存テストを削除またはskipしない

## Git rules
- mainへ直接pushしない
- ユーザーの許可なくcommitしない
- 本番デプロイを実行しない
```

CLAUDE.mdは、Claudeがセッション開始時に読むプロジェクト指示書である。/goalを使う際にも有益だが、重要な注意点がある。

**CLAUDE.mdはClaudeへの指示であり、技術的な強制ルールではない。**

「本番へデプロイしない」と書けばClaudeは従おうとするが、絶対に実行できなくなるわけではない。確実に禁止したい操作は、Claude CodeのPermission denyルールやHooksでブロックする必要がある。

CLAUDE.mdには何でも詰め込まず、プロジェクト全体に常に必要な情報だけを書く。公式には、簡潔で具体的にし、目安として200行未満に保つことが推奨されている。

## progress.mdは必要か

短い作業では不要である。数十分程度のバグ修正なら、Goal条件とGit差分だけで十分なことが多い。毎ターンprogress.mdを書かせると、その更新や読み直しに余計なトークンを使う。

一方、複数時間、複数機能、コンテキスト圧縮をまたぐ作業では、外部ファイルへの進捗記録が有効になる。

```
# Progress

## Completed
- 認証処理の原因調査
- APIレスポンス修正
- 単体テスト追加

## Remaining
- E2Eテスト
- エラーメッセージ統一
- ドキュメント更新

## Decisions
- 認証失敗は401へ統一
- DBスキーマは変更しない
```

より大きな仕事では、次のように分けてもよい。

```
plan.md       実装方針
progress.md   完了済み・残作業
decisions.md  選択した設計と理由
```

これらは/goal専用の公式ファイルではない。長時間エージェントの一貫性を保つための実践的な運用方法である。

## 具体的な利用例

### バグ修正

```
/goal ログイン失敗時に500ではなく401が返るよう修正され、
認証関連テスト、npm run typecheck、npm run lintがすべて成功している。
既存テストは削除・skip・弱体化しない。
本番環境には接続しない。
または10ターンで停止する
```

### API移行

```
/goal src配下に残る旧API client.auth.getUser()の呼び出しが0件になり、
新APIへの置換後もビルド、型チェック、テストがすべて成功している。
移行と無関係なリファクタリングは行わない。
または15ターンで停止する
```

### テストカバレッジ改善

```
/goal src/services/authの分岐カバレッジが80%以上になり、
全テストが成功している。
本番コードをテストのためだけに不自然に変更しない。
テスト対象を除外設定へ追加しない。
または12ターンで停止する
```

### 大きなファイルの分割

```
/goal src/api/quiz.tsを責務ごとのモジュールへ分割し、
各ファイルを300行以下にする。
公開APIと既存動作を維持し、型チェックと全関連テストが成功している。
または15ターンで停止する
```

### JSONデータの修正

```
/goal data/questions配下の全JSONがschema/question.schema.jsonに準拠し、
ID重複、選択肢不足、正答不在が0件になっている。
元の問題文の意味を必要なく変更しない。
検証スクリプトがexit 0で完了する。
または10ターンで停止する
```

### 依存関係監査

```
/goal package.json内の全依存関係について、
現在のバージョン、最新安定版、既知の脆弱性、推奨対応を
dependency-audit.mdにまとめている。
実際のパッケージ更新やpackage.jsonの変更は行わない。
対象漏れがないことを確認する。
または8ターンで停止する
```

### ドキュメント更新

```
/goal 現在のAPI実装とdocs/api.mdの不一致が0件になり、
全エンドポイントの入力、出力、認証要否、エラー形式が記載されている。
実装コードは変更しない。
または8ターンで停止する
```

## コーディング以外にも使えるか

/goalはコーディング専用ではない。Claude Codeがアクセスできるファイルやツールを使い、終了状態を確認できる仕事なら利用できる。ただし、コーディング以外では「何をもって完成とするか」をより慎重に定義する必要がある。

### リサーチ

```
/goal 指定テーマについて一次情報と公式資料を中心に調査し、
発表日、発表主体、主要主張、限界、反対意見を含む
research-report.mdを作成している。
すべての主要な事実に出典名と公開日が付いている。
出典不明の断定が0件になっている。
または10ターンで停止する
```

ただし、「20件集めた」ことは確認できても、その20件だけで論点を網羅している保証はない。重要な調査では人間による出典確認が必要である。

### 記事制作

```
/goal 指定テーマの記事が完成し、
導入、結論、仕組み、具体例、限界、実践手順の章を含んでいる。
同じ説明の重複を削減し、禁止表現が0件になっている。
事実に関する主要主張には出典情報が付いている。
または8ターンで停止する
```

記事制作では、「章がそろっている」「禁止表現がない」「文字数内」といった形式条件には向いている。

一方で、「面白い」「独自性が高い」「バズる」「文章が美しい」は主観的である。最終品質の判定を/goalの評価器だけに任せるべきではない。

### 医療・法律・金融など高リスク分野

JSON形式が正しい、必須項目がすべてある、引用が付いている、日付や単位が統一されている、といった構造チェックには使える。

しかし、「医学的に正しい」「法的に問題ない」「投資判断として妥当」といった専門的正確性を、小型評価モデルの完了判定に任せてはいけない。

/goalで形式と機械的整合性を整えた後、専門家または別の高性能モデルによる批判的レビューが必要である。

## /goal、/loop、Stop Hook、Scheduleの違い

機能次の実行が始まる条件主な用途/goal前のターンが終わり、条件未達と判定されたとき完了条件まで作業する/loop指定した時間間隔状態を定期確認するStop Hookターン終了後に独自スクリプトや評価を実行厳密な終了判定Scheduled Task / Routine指定日時や定期スケジュールセッション外の定期処理

/loopは、5分ごとにデプロイ状態を確認する、長時間テストの終了を定期確認する、PRのCI結果を一定間隔で見る、といった用途に向いている。

Stop Hookは、実際にテストスクリプトを実行してexit codeで厳密に判定する、特定ファイルの存在や内容を機械的に検証する、全プロジェクトで同じ終了判定を使う、といった用途に向いている。

/goalの評価器はファイルやコマンドを直接確認できない。より厳密で決定論的な判定が必要なら、スクリプト型Stop Hookの方が適している。

Scheduleは、毎朝Issueを整理する、毎週依存関係を監査する、夜間にテストを実行する、定期的にレポートを生成する、といった用途に向いている。/goalは現在のセッションを継続する機能であり、毎日決まった時刻に新しい仕事を始める機能ではない。

## 複数のGoalを並列で動かす

1セッションにつき有効なGoalは1つだが、複数セッションを立ち上げれば並列実行できる。

たとえば、認証バグの修正、ドキュメント更新、依存関係監査の3つを同時に進められる。

現在のClaude Codeには、複数のバックグラウンドセッションを一覧管理するAgent Viewもある。

```
claude agents
```

ただし、並列実行には制約がある。各セッションが利用枠やトークンを消費し、CPU、メモリ、ディスクを同時に使用する。同じファイルを編集すると競合し、同じDB、ポート、開発サーバーを使うと衝突する。外部APIのレート制限に達する可能性もある。

同じリポジトリで複数作業を実行する場合は、必ずWorktreeや別ブランチで分離する。

## トークン消費はどうなるか

評価モデルに使われるトークンは、通常、メインのSonnetやOpusの作業量と比較すると小さい。

しかし、Goal未達成と判定されるたびに、メインモデルによる新しいターンが始まる。この作業ターンは通常どおり利用枠やAPI料金を消費する。

つまり、曖昧なGoalは次の悪循環を起こす。

```
完了条件が曖昧
    ↓
評価器が未達成と判定
    ↓
Claudeが追加作業
    ↓
まだ条件を確認できない
    ↓
さらに追加作業
```

/goalはトークン節約機能ではない。**人間の監視時間を減らす代わりに、必要に応じて複数ターンを自動実行する機能**である。

## /goalが失敗しやすいパターン

### 完了条件が主観的

```
/goal UXが良くなるまで改善する
```

改善例は次のようになる。

```
/goal フォームの全入力項目にlabelがあり、
キーボードのみで送信でき、
既存E2Eテストとアクセシビリティテストが成功している
```

### 達成不能な条件

存在しないAPI、アクセスできない本番環境、提供されていない認証情報が必要な条件では終わらない。Goalを設定する前に、Claudeが必要なファイル、コマンド、サービスへアクセスできるか確認する。

### 証拠がログに残らない

ファイル上は完了していても、Claudeが確認コマンドを実行していなければ、評価器は完了を判断できない。

条件には確認方法を書く。

```
npm testがexit 0
grepで旧API呼び出しが0件
検証スクリプトがexit 0
```

### テストを通すこと自体が目的化する

Claudeが本質的な修正ではなく、テストを削除、skip、期待値変更によって合格させる可能性がある。

次の制約を加える。

```
既存テストを削除、skip、弱体化しない
テスト対象を除外設定へ追加しない
本番コードの仕様を勝手に変更しない
```

### 作業範囲が大きすぎる

```
/goal アプリ全体を完成させる
```

このようなGoalは、設計変更、認証、決済、UI、テスト、デプロイなど多数の判断を含みすぎる。

大きな仕事は、DBスキーマ、API、UI、テスト、ドキュメントなどに分割する。/goalは巨大な曖昧タスクより、明確な中規模タスクで最も安定する。

## 安全装置として知っておくべきこと

### 「しないで」と書くだけでは不十分

会話やCLAUDE.mdに禁止事項を書くことは有効だが、絶対的な強制ではない。

重要な禁止事項はPermission denyルールで技術的に止める。たとえば、mainへのpush、本番DBへの接続、terraform apply、rm -rf、secretsファイルの外部送信などである。

### Gitを使う

Claude Codeには/rewindによるチェックポイント機能があるが、すべての変更を戻せるわけではない。

Claudeの編集ツールによる変更は追跡されるが、Bashで実行したrm、mv、cpなどによるファイル変更は完全には追跡されないことがある。

Gitのブランチ、コミット、Worktreeは引き続き必要である。

### 本番環境では使わない

Auto Modeと/goalを組み合わせた長時間実行は、ローカル開発環境、検証環境、隔離されたWorktreeで使う。

本番データ、本番DB、クラウドインフラ、外部への公開操作を含む仕事は、各段階で人間の承認を残すべきである。

## トラブルシューティング

### /goalが表示されない

次を確認する。

```
claude --version
```

v2.1.139未満なら更新する。

```
claude update
```

それでも使えない場合は、Workspaceの信頼確認、Hooksの無効化設定、組織の管理ポリシーを確認する。

### Auto Modeが選べない

Auto Modeには対応モデルとアカウント条件がある。また、Team・Enterpriseでは管理者が無効化している場合がある。

Claude Code Desktopでは、設定画面でAuto Modeを有効にした後、送信ボタン付近のモード選択から切り替える。

### Goalが完了しているのに止まらない

Claudeが確認結果を会話に示していない可能性がある。

次のような確認を条件に含める。

```
最後にnpm test、npm run typecheck、npm run lintを再実行し、
終了コードと結果を最終回答に記載する
```

### すぐに達成扱いになる

条件が曖昧か、Claudeの自己申告だけで満たせる内容になっている可能性がある。

「実装が完了した」ではなく、「どのコマンドがどの結果になれば完了か」を指定する。

### 同じことを繰り返す

条件が達成不能、必要な情報へアクセスできない、または評価器が確認できる証拠がない可能性がある。

一度/goal clearで止め、原因を確認してGoalを書き直す。

## まず試すなら、このGoalがおすすめ

既存プロジェクトで、小さなテスト失敗を直すところから始める。

```
/goal 現在失敗しているテストの原因を特定して修正し、
全テスト、型チェック、lintがexit 0で完了している。
既存テストの削除、skip、期待値の弱体化は行わない。
変更したファイルと検証結果を最後に一覧化する。
本番環境への接続、commit、push、deployは行わない。
または8ターンで停止する
```

最初から「アプリを全部完成させる」「一晩でサービスを作る」といった巨大Goalを設定するより、小さく成功させてから対象を広げる方がよい。

## クイックリファレンス

```
# Goalを設定
/goal [完了条件]

# 状態確認
/goal

# 途中停止
/goal clear

# 別の停止表現
/goal stop
/goal cancel
/goal off

# 前回セッションを再開
claude --continue

# 指定セッションを再開
claude --resume

# 非対話モード
claude -p "/goal [完了条件]"

# Auto Modeで起動
claude --permission-mode auto

# ファイル編集のみ自動承認
claude --permission-mode acceptEdits

# Worktreeで隔離
claude --worktree task-name

# 複数セッションを管理
claude agents

# バージョン確認
claude --version

# アップデート
claude update
```

## まとめ

/goalの本質は、Claude Codeを魔法のような完全自律AIに変えることではない。Claudeに対して、作業内容だけでなく、**検証可能な終了条件を渡すこと**にある。

成功させる原則は次の通りである。

1. 最終状態を測定可能にする
2. 確認コマンドや証拠を指定する
3. 変更してはいけないことを書く
4. 大きな仕事はPlan Modeで方針を確認する
5. ブランチやWorktreeで作業を隔離する
6. Auto Modeを安全保証だと考えない
7. 最後は人間が差分と結果をレビューする

正しいGoalを設定できれば、「続けて」と何度も入力する時間を減らし、Claude Codeに中規模の作業をまとまった単位で任せられる。

一方で、間違った完了条件を設定すれば、Claudeは間違った方向へ効率よく進んでしまう。

/goalで最も重要な能力は、プロンプトを長く書くことではない。

> **何を確認できたら、この仕事は本当に終わったと言えるのか。**

それを明確に定義する能力である。
