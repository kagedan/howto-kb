## はじめに

この文書は <https://github.com/yukineko/claude-harnesses> で開発してるharness/pluginの機能のちctxrotの仕様と実装について記述してます

Claude Codeでエージェント運用を長く続けていると、ある時点から明らかに応答の質が落ちる瞬間がある。序盤に伝えた決定事項をモデルが忘れる、巨大なログを1回読み込んだだけでコンテキストウィンドウが埋まる、`/compact`のたびに大事な仕様が「要約で消えている」——いわゆる **context rot（コンテキスト腐敗）** だ。

自分はこの問題に対して、場当たり的にプロンプトを工夫するのではなく、Claude Codeの各種フック（`PreToolUse` / `PostToolUse` / `UserPromptSubmit` / `PreCompact` / `SessionStart` / `Stop`）に配線するRust製プラグインを何本か書いて機械的に対処している。この記事では、その設計の中で特に効いていると感じている4つの観点——

1. **バイナリ処理をしない**という前提の置き方
2. \*\*lossy圧縮（要約）\*\*をどう隔離して扱うか
3. **sub-agent**にどう仕事を逃がすか
4. **コンテキストサイズ**そのものをどう制御するか

について、実際に動いている実装をベースに書く。そのうえで、「じゃあ素の`/compact`運用のままで良かったのでは？」という問いにも正直に答える。

---

## 前提: LLMが扱えるのは常にテキストだけである

最初に置いた設計原則はシンプルで、「**バイナリ処理はしない**」というものだ。これは「バイナリファイルを避ける」という運用ルールではなく、もっと根本的な話をしている。

エージェント同士（main loopとsub-agent）がやり取りする手段は、突き詰めるとすべて**テキスト**でしかない。

* エージェントへの指示 → テキスト（プロンプト）
* エージェントからの返答 → テキスト（JSONもテキストとして返ってくる）
* セッションをまたぐ状態の保存 → テキスト（JSONLファイル）
* worker/verifier間でのコードの受け渡し → テキスト（git worktree上のファイル）

つまり「コンテキストを最適化する」という作業の実体は、**どのテキストをどのエージェントに渡すか（あるいは渡さないか）を制御すること**に還元される。ここでバイナリな中間表現（独自の圧縮フォーマットやベクトルキャッシュのようなもの）を持ち込むと、途端に「何が失われたか」を人間が検証できなくなる。だから状態はすべて生のJSONL・Markdownとしてファイルシステムに置き、フックはstdinからJSONを読んでstdoutにJSON（`additionalContext`など）を返すだけ、というシンプルなI/Oに統一している。

この制約が、後述する「lossy圧縮の隔離」と「sub-agentへの分離」の両方の土台になっている。

## 1. lossy対策 — 「圧縮していい情報」と「圧縮してはいけない情報」を型で分ける

Claude Codeの組み込みcompaction（`/compact`）は、長い会話をLLMに要約させる。これは必要な機能だが、素のまま使うと2つの失敗モードを踏む。

* **規範が消える**: contractや不変条件、命名規則のような「毎ターン効いていてほしい」情報が、要約でparaphraseされたり脱落したりする
* **逐語情報が壊れる**: IDやコマンド、仕様の原文のような「一字一句そのまま」であるべき情報が要約で改変される

ここで重要なのは、**「圧縮するかどうか」を都度LLMや人間の判断に委ねない**ことだ。自分のプラグイン(`context-governor`)では、コンテキストに乗るアイテムを起動時に3つのレーンに分類する。

| レーン | 意味 |
| --- | --- |
| `Pinned` | 最終コンテキストに常駐させる。規範・契約・不変条件 |
| `Verbatim` | lossy圧縮禁止。compaction前にバッキングストアへスナップショットする |
| `Evictable` | 刈り込み・退避・あとで想起可能。圧縮ハンドラが触ってよいのはここだけ |

そして、圧縮を担う唯一のハンドラ（`ToolResultGroomer`）は\*\*`Evictable`のトークンしか受け付けないシグネチャ**にしてある。`Pinned`や`Verbatim`を渡そうとするコードはコンパイルが通らない。つまり「verbatimな情報は圧縮されない」という不変条件を、実行時のチェックではなく**型システムで表現不能な状態にしてしまう\*\*。「うっかり規範を圧縮対象に混ぜてしまった」というバグがそもそも書けない。

### 決定論的な退避 と LLMによる要約を分離する

もう一つの分離軸が、**「決定論的にファイルへ書き出す」ことと「LLMに意味を要約させる」ことを別工程にする**という点だ。

* **rescue**（`PreCompact`フックで発火）: compaction直前にtranscriptを走査し、「決定事項・残課題・触ったファイル・直近の生ターン」をルールベースで抽出してMarkdownに書き出す。LLMは一切使わない。だから`PreCompact`の厳しいタイムアウト内でも確実に完走し、失敗してもexit 0でターンを壊さない。
* **distill**（非同期の`claude -p`呼び出し）: rescueとは別に、切り離したプロセスでLLMにtranscriptを意味的に要約させる。こちらは元の会話とは別の文章になる、正真正銘の**lossy圧縮**である。compactionをブロックしない形でfire-and-forgetする。

決定論的な抽出（rescue）を「安全網」として必ず先に確保し、質の高い要約（distill）は「あれば嬉しいがブロッキングしない付加機能」として後ろに置く。この順序を逆にする——LLM要約が失敗したら何も残らない設計——にはしない、というのがここでの学びだった。

蒸留ノートには契約も課している。restoreが読む見出し（決定事項/残課題）を欠いたノートは、書き込み時点でexit 1で拒否する。「スキーマがずれて次セッションで空のcarryoverしか復元されない」という事故を、黙って起きるのではなく書き込み側で大声で落とすようにしている。

### 効いているかを実測する

サイズ軸は「主張」ではなく「計測」が必要だ、というのがもう一つの学びだった。groom/inject/snapshotの3アクションは実行のたびに1行のJSONLをledgerに追記し、`saved_tokens`（実際に回収したウィンドウ占有）と`resident_tokens`（アクション後の占有）を記録する。これを集計する読み取り専用コマンドを別に用意しておくと、「ピン留めすればウィンドウが小さくなるはず」のような直感の誤り（実際にはピン留めは常駐コストを増やす）を、後から実数で否定できる。

## 2. sub-agentの活用 — 「重い実行の中身」をmain loopに戻さない

もう一つの大きな柱が、作業そのものを新規起動のsub-agentに逃がすことだ。

自分の実装(`condukt`)では、タスクの分解を担うinterpreter、実装を担うworker、検証を担うverifierを、それぞれ独立したsub-agentとして起動する。ここでの前提は一つだけ:

> **sub-agentはこの会話を一切見ていない。**

interpreterやworkerが知ることができるのは、main loopがプロンプトとして文字列で渡したものだけである。sub-agentが終了すると、その実行の中身（ビルドログ・diff全文・grep結果）はmain loopのコンテキストに戻ってこない。**戻ってくるのは結果だけ**だ。これにより「重い作業の出力がmain loopに積み上がらない」という効果が生まれる。

### 渡す情報は変換せず、切り出す

ここで大事なのは、sub-agentに渡す情報を「要約して渡す」のではなく「**必要な部分だけ切り出して渡す**」ことだ。

| フィールド | やっていること |
| --- | --- |
| `interface_context` | ファイル全体ではなく、grepで関数シグネチャの行だけを抜いて渡す |
| `knowledge_context` | 規約テキストをそのまま渡す（要約しない） |
| `peer_tasks` | 隣接タスクの`title`と`touched_files`だけを渡し、`done_criteria`や差分は渡さない |

「どの部分がこのエージェントに必要か」をmain loop側が判断し、該当箇所だけをテキストとして手組みしてプロンプトに埋め込む。worker自身にgrepさせるとworkerのコンテキストが汚れるので、main loopが事前に刈り取って渡す、という設計になっている。

### 会話を共有できないなら、ファイルシステムを共有メモリにする

sub-agent同士が会話を共有できないなら、「workerが書いたコードをverifierがどう読むか」という問題が残る。答えはシンプルで、**ファイルシステムを共有メモリとして使う**。workerはgit worktreeにコードを書いてcommitし、"done"とだけ返す。verifierにはmain loopが「このworktreeパスを見ろ」とテキストで渡し、verifierはそのパスを自分でReadする。実行状態も同様に、JSONLの状態ファイルに書いて、どのセッション・どのsub-agentからも読めるようにしている。

この設計は、前段の3軸モデルでいう**Cost軸**（main loopのプレフィルを安く保つ）と**Size軸**（重い出力をmain contextに積まない）の両方に効く。

## 3. context sizeへの対策 — 「入口で止める」と「着弾後に削る」の二段構え

ここまでは「質(correctness)」と「実行の隔離」の話だったが、最後にウィンドウそのものの占有量（size）をどう抑えているかを書く。

フックがトークンを減らせるタイミングは、実は限られている。**すでに窓に載ってしまったトークンを後から追い出すことはできない**。だからコントロールは「入口で止める・絞る」ことと「compactionのような再構成のタイミングで実効化する」ことの2点に集約される。

### 入口で止める: サイズゲート

`PreToolUse`のタイミングで、上限なしの巨大な`Read`（1MB以上）を理由付きで事前にdenyする。優先順位は「deny glob（絶対拒否）→ limit指定 → allow glob（迂回) → サイズ判定」で、通常のソースコード読み込みには影響しないよう狭く設計してある。ここで止めることで、「降ろせないトークン」がそもそもウィンドウに乗る前に防ぐ。

サイズゲートをすり抜ける50KB〜1MB程度の中間帯の出力に対しては、`PostToolUse`で「次回は同じ読み方をせず、sub-agent経由で結論だけ持ち帰るように」と誘導する。これは**ブロックしない**。すでにコンテキストに入ってしまった後なので、正確性を守るためではなく、**次ターン以降の占有を減らす**ことが目的になる。

さらに`ToolResultGroomer`が、ウィンドウ圧力を見ながらグルーム予算を動的に調整し、`Evictable`なツール結果のhead/tailだけを残してトリムする。冒頭（import・構造）と末尾（エラー行など）は情報価値が高い、という経験則に基づく。出力は常に入力より小さいことをproperty testで保証している。

### 使用率そのものを可視化する

もう一つ効いているのが、常時表示のステータスライン(`ctxrot 52% ▮▮▯▯ band1 ~104k/200k`)で、使用率の帯(50/75/90%)ごとに色を変えて出す。帯をまたいだ瞬間に一度だけ助言を注入し、75%以上ではcompaction前に先回りしてrescueノートを書いておく。「助言そのものがcontext rotの新たな発生源にならないよう、1帯につき1回だけ」というのは地味だが効果が大きいルールで、毎ターン注意喚起を挟むと逆にノイズでウィンドウを圧迫してしまう。

## 3つの軸を混同しないというメタ原則

ここまでの施策はバラバラに見えるが、根底には共通の失敗の避け方がある。それは、コンテキスト最適化が触れる **Size / Cost / Correctness という3つの独立した軸を混同しないこと** だ。

| 軸 | 何を変えるか | 効く手段 |
| --- | --- | --- |
| **Size** | ウィンドウ占有を実際に縮める | ツール結果のトリム・参照本体の外部化 |
| **Cost** | プレフィルを安く保つ | プロンプトキャッシュを温かく保つ＝安定したプレフィックスの維持 |
| **Correctness** | 規範が要約で消えるのを防ぐ | lossy圧縮を隔離し、何が残るかだけを管理する |

「ピン留めすればウィンドウが小さくなるはず」「compactionの閾値を下げれば省サイズになるはず」というのは、この3軸を取り違えた典型的な誤りだ。実際にはピン留めは常駐コストを**増やす**し、閾値を下げるのはCorrectness側の話であってSizeを減らす手段ではない。この混同に気づかないまま「効いているはず」で施策を積むと、後から実測して初めて何も効いていなかったと分かる。だからこそ、圧縮そのものは自作せず組み込みのcompactionに委譲し、自分たちのプラグインは「何を常駐させ・何を退避し・何を後で呼び戻すか」という**規律の層**に徹する、という役割分担にしている。

## 素の`/compact`運用と比べて、実際どれだけ得なのか

ここまで書いた施策は、当然ながら「複雑さの追加」でもある。フックが増え、コードが増え、設定項目が増える。ここで一度立ち止まって、「結局、これは何もせず素の`/compact`（と手動での`/clear`）だけに任せる運用と比べて優位なのか」を正直に評価しておく。

### 素の`/compact`運用がやっていること・やっていないこと

Claude Code組み込みのcompactionだけに任せた場合、実態はこうなる。

* ウィンドウが埋まってから（あるいはユーザーが手動で打ってから）初めてLLMが会話全体を要約する
* 要約にはPinned/Verbatimのような区別がなく、規範も逐語情報も同じ扱いで圧縮対象になる
* 巨大ファイルの`Read`を止める仕組みはないので、1回の操作でウィンドウの大半を持っていかれることがある
* セッションをまたいだ引き継ぎは、次のセッションが同じ会話を継続する場合を除き基本的に無い
* 「今どれくらい危ないか」「何が実際に縮んだか」を見る手段がない

つまり素の運用は、**劣化が起きてから、劣化への唯一の対処(要約)を、劣化の程度を知らないまま実行する**という構造になっている。

### 明確に優位だと言える点

1. **タイミングが根本的に違う**: compactは「ウィンドウが埋まった後」にしか効かないが、preguardやサイズゲートは**埋まる前に止める**。lost-in-the-middleは compact する時点ですでに一定進行しているので、「埋まってから直す」より「そもそも埋めない」方が上流で効く。
2. **規範・逐語情報の生存が要約品質に依存しなくなる**: 素のcompactでは「今回のLLM要約がたまたま良い出来か」に規範の生存が懸けられている。Pinned/Verbatimレーンは、要約の巧拙に関係なく機械的に保護するので、再現性がある。
3. **セッションをまたいだ引き継ぎが構造として保証される**: 素の運用では、セッションを`/clear`したり新規に開いたりした瞬間に文脈は原則として失われる。rescue/restoreは「LLMの要約が良かったから」ではなく「ルールベースの抽出をファイルに書いたから」引き継げる、という決定論的な保証を持つ。
4. **測定できる**: 素の運用は「compactしたら軽くなった気がする」以上の情報を持たない。ledgerによる実測は、施策の効果を後から検証・反証できる。

### 優位とは言えない、あるいはコストになる点

正直に書くと、以下は素の運用に対する明確な勝ちではない。

1. **compact自体は置き換えていない**: `Evictable`に分類された情報や、そもそも分類の設定をしていない情報は、結局は組み込みcompactionのLLM要約に委ねられる。つまり**分類・設定を正しくやって初めて効果が出る**ものであり、何もしなければ素の運用と同じリスクをそのまま抱える。
2. **複雑さと障害点の増加**: フックの数だけ「バグって誤動作する」余地が増える。fail-soft（エラーでも exit 0）に徹してターンを壊さない設計にはしているが、それでも「preguardが誤ってdenyする」「toolguardが的外れなタイミングで誘導する」といった誤検知コストはゼロではない。
3. **レイテンシとオーバーヘッド**: `PostToolUse`/`PreToolUse`/`UserPromptSubmit`という高頻度イベントにフックを常時配線するので、1回1回は軽くても積算コストは素の運用より確実に大きい。
4. **過信のリスク**: 「ゲートと計測があるから安心」という感覚は、実際にはmain loopが必要な情報を正しく`interface_context`などに切り出せているか、SpecClassifierが情報を正しいレーンに分類できているか、という**運用の質**に依存する。分類ミス（本来Verbatimであるべき情報をEvictableに入れてしまう等）は、何もしない素の運用より始末が悪い——「圧縮されないはず」という誤った確信のもとで実際には圧縮されてしまうからだ。
5. **セットアップコストがゼロではない**: 素の運用は何も設定しなくても動く。この仕組みは、効果を出すために事前の分類・設定・しきい値調整を要求する。

### 結論

短時間・単発のセッションや、規範的な制約が少ないカジュアルな作業であれば、素の`/compact`運用で十分であり、この規模の機構を持ち込むのは明らかにオーバーエンジニアリングだ。

一方で、**長時間・複数セッションにまたがる、かつ「特定の契約や不変条件を最後まで守り切る必要がある」エージェント運用**においては、素の運用に対する優位性は構造的なものだと考えている。理由は単純で、素の運用は「劣化が起きた後にLLMの要約品質に賭ける」設計であるのに対し、この仕組みは「劣化が起きる前に止める」「要約品質に依存しない生存保証を作る」という、そもそも競合するレイヤーに手を入れているからだ。ただしその優位性は**無条件には発動しない**。分類・設定・監視という運用コストを払って初めて、素の運用に対する差が実際の数字として現れる。「導入しただけで自動的に安全になる」ものではない、というのが正直な評価になる。

## まとめ

長時間のClaude Codeセッションでcontext rotに対処するために自分が積んだ施策は、突き詰めると次の4つに集約される。

* **バイナリ処理をしない**: エージェント間の通信は常にテキストに統一し、独自の中間表現を持ち込まない。これが後続すべての施策の前提になる
* **lossy圧縮の隔離**: 圧縮していい情報(`Evictable`)と圧縮してはいけない情報(`Pinned`/`Verbatim`)を型で分離し、決定論的な抽出(rescue)とLLMによる要約(distill)を別工程にする
* **sub-agentの活用**: 重い実行の中身を新規起動のsub-agentに任せ、結果だけをmain loopに戻す。渡す情報は要約せず、必要な部分だけを切り出す
* **サイズそのものの制御**: 入口(サイズゲート)で巨大な読み込みを止め、着弾後(groomer/toolguard)は減らすだけに徹し、使用率を常時可視化する

そしてこれらすべてを貫くのが、Size / Cost / Correctnessという3軸を混同しない、というメタ原則だった。素の`/compact`運用と比べた優位性は無条件ではなく、正しい分類・設定・監視という運用コストと引き換えに得られるものだ。「効いているはず」で終わらせず、実測して初めて「効いている」と言えるようにする——地味だが、長時間運用するエージェントシステムを作る上で一番効いた学びはここだったと思う。
