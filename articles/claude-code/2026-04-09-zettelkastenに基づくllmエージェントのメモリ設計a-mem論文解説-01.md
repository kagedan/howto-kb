# はじめに：LLMエージェントはどう記憶するか

Claude CodeのようなAIコーディングエージェントは、過去のことをどのように記憶しているのだろうか。まず基本となるのはコンテキストウィンドウである。ユーザーとの対話履歴やコードの内容は、コンテキストとしてモデルに渡される。モデルはこのコンテキストの範囲内であれば、過去のやり取りを参照して一貫した応答を返すことができる。しかし、コンテキストウィンドウには上限がある。対話が長くなれば古い情報は押し出されていくし、そもそもセッションが終われば対話履歴はリセットされる。コンテキストだけでは、長期的な記憶を維持することはできない。

そこで、セッションをまたいで情報を保持する外部メモリの仕組みが重要になる。たとえばClaude Codeでは、プロジェクトルートの`CLAUDE.md`に規約を書いておけばセッションをまたいで参照されるし、`~/.claude/projects/`配下にはプロジェクトごとの`memory/`ディレクトリが作られ、エージェントが自律的に学んだ情報が蓄積されていく。さらに興味深いのは、メモリへの保存がユーザーの明示的な指示だけでなく、エージェントの自律的な判断でも行われる点である。たとえば文体に関するフィードバックを一度伝えると、エージェントはそれを「今後も適用すべき指示」と判断し、自発的にメモリへ保存する場合がある。次回以降のセッションでも同じ文体が維持されるため、「前にも同じことを説明したのに」というストレスはかなり軽減される。メモリの有無はエージェントのユーザー体験を大きく左右する要素であり、それゆえにLLMエージェントのメモリシステムは活発な研究テーマでもある。

本記事では、その中でもZettelkastenという知識管理手法の原則を応用し、**メモリ自身が自律的にリンクを形成し進化する**仕組みを提案した論文「[A-Mem: Agentic Memory for LLM Agents](https://arxiv.org/abs/2502.12110)」（Xu et al., NeurIPS 2025）を紹介する。

# 既存のメモリシステムの課題

研究の世界では、LLMエージェントのメモリをより賢くしようと、独自のメモリアーキテクチャの設計が進められている。OSの仮想メモリに着想を得た[MemGPT](https://arxiv.org/abs/2310.08560)、密ベクトル検索と忘却曲線に基づく更新を組み合わせた[MemoryBank](https://arxiv.org/abs/2305.10250)、グラフデータベースでメモリ間の関係を構造化する[Mem0](https://arxiv.org/abs/2504.19413)などがその代表例である。

A-Memの論文は、これらの既存メモリシステムに共通する課題として以下の3点を指摘している。

1. **操作の固定性：** メモリの保存・検索のワークフローが開発者によって事前に設計されており、エージェントが状況に応じて柔軟に操作を変えることが難しい
2. **接続の欠如：** メモリ同士の意味的な関連を発見して接続を形成したり、新しい組織パターンを自ら作り出したりすることができない
3. **構造の静的さ：** 一度保存されたメモリの内容は固定され、新しい知識が追加されても既存メモリの記述が更新・深化することがない

これらは研究レベルのメモリシステムに対する指摘であるが、「はじめに」で触れたClaude Codeのメモリにも当てはまる部分がある。Claude Codeの`memory/`は保存タイミングをエージェントが自律的に判断するため、操作の固定性（課題1）はかなり解消されている。しかし、メモリファイル間の意味的なリンクを自動生成する仕組みや、新しいメモリの追加をトリガーとして関連する既存メモリの記述が更新されるような仕組みは、公式ドキュメントには記載されていない。接続の欠如（課題2）は実用ツールにおいても未解決の課題といえる。構造の静的さ（課題3）については、Claude Codeには蓄積されたメモリを定期的に整理・統合するAuto-dreamと呼ばれるメモリ整理機能が存在する。ただし、これは矛盾の解消や重複の統合といったクリーンアップが中心であり、A-Memのように新しいメモリの追加をトリガーとして既存メモリの文脈的理解を書き換えるものとは性質が異なる。

A-Memは、メモリ間のリンク生成と既存メモリの進化という欠けている操作を追加することで、 **メモリを「独立したファイルの集合」から「相互に接続され進化する知識のネットワーク」** に変える。その設計原則として採用しているのがZettelkastenである。

# Zettelkastenとは何か

Zettelkastenは、ドイツの社会学者ニクラス・ルーマンが開発した、メモ同士を関連付けて知識のネットワークを構築する知識管理手法である。基本原則は以下の3つである。

* **原子性：** 1枚のカードに1つのアイデアだけを書く。自己完結した知識単位を作る
* **リンキング：** カード同士を意味的な関連に基づいてリンクする。カテゴリによる分類ではなく、文脈に基づく接続を重視する
* **進化：** 新しいカードを追加する際に既存のカードとの関連を考え、必要に応じて既存カードの記述を更新する。知識ネットワークは時間とともに成長・深化する

このZettelkastenの思想は、現代ではObsidianのようなナレッジ管理ツールに受け継がれている。Obsidianではノート間を `[[内部リンク]]` で接続でき、バックリンクやグラフビューによってリンク構造を可視化できる。フォルダによる階層分類だけでなく、リンクによるネットワーク構造で知識を組織化できるという設計思想は、Zettelkastenの原則と通じるものがある。

重要なのは、Zettelkastenが単なる「メモの整理術」ではなく、**知識が有機的にネットワークを形成し、そのネットワーク自体が新しい発見を生む**という思想を持っている点である。A-Memはこの3つの原則を、人間の手動操作ではなくLLMの自律的な判断によって実現する。原子性はノート構築（Note Construction）に、リンキングはリンク生成（Link Generation）に、進化はメモリ進化（Memory Evolution）に、それぞれ対応している。

# A-Memの設計：メモリの自律的な組織化

従来のメモリシステムでは、LLMエージェントがメモリに対してRead/Writeを行い、メモリ側は受動的なストレージとして振る舞う。A-Memの核心は、メモリ側にもLLMを持たせ、**書き込みをトリガーにメモリ内部で知識の組織化が自律的に動く**という点にある。

![従来のメモリシステムとA-Memのアプローチの対比](https://static.zenn.studio/user-upload/deployed-images/b4c9f234d92f4d45cbe6ec46.png?sha=5d0a111245ce9d157509d0726b267eec975e2d2d)

具体的には、Note Construction（ノート構築）、Link Generation（リンク生成）、Memory Evolution（メモリ進化）、Memory Retrieval（メモリ検索）の4つのコンポーネントで構成される（Figure 2）。

## ノート構築：対話や行動の記録から知識を抽出する

エージェントが環境とインタラクションすると、A-Memはその内容を**構造化されたノート**として保存する。各ノートは以下の属性を持つ。

| 属性 | 内容 | 生成方法 |
| --- | --- | --- |
| Content (c\_i) | 元のインタラクション内容 | そのまま保存 |
| Timestamp (t\_i) | 時刻 | 自動記録 |
| Keywords (K\_i) | 核心概念を捉えるキーワード | LLMが生成 |
| Tags (G\_i) | カテゴリ分類用のタグ | LLMが生成 |
| Context (X\_i) | 文脈的記述（意味理解を提供） | LLMが生成 |
| Embedding (e\_i) | 密ベクトル表現 | テキストエンコーダ |
| Links (L\_i) | リンクされたメモリの集合 | リンク生成プロセス |

ここで注目すべきは、Keywords、Tags、Contextの3つがLLMによって自律的に生成される点である。対話や行動の記録をそのまま保存するのではなく、LLMに「このやり取りの核心は何か」「どのような文脈で理解すべきか」を分析させることで、暗黙的な知識を明示的な構造に変換している。

Zettelkastenの原子性原則に従い、各ノートは1つの自己完結した知識単位として設計される。また、テキストエンコーダ（all-MiniLM-l6-v2）による密ベクトル表現も同時に生成され、後述するリンク生成と検索に使用される。

## リンク生成：メモリ同士の接続を自動的に発見する

A-Memの核心的な仕組みの1つが、新しいメモリが追加されるたびに既存メモリとのリンクを自動的に生成するプロセスである。

具体的には、2段階のアプローチを取る。

**第1段階：埋め込みベースの候補選択**

新しいノートの埋め込みベクトルと、既存の全ノートの埋め込みベクトルとのコサイン類似度を計算し、top-k個の候補を選出する。全メモリに対してLLMで関連性を判断するとコストが大きいため、まずベクトル類似度で候補を絞り込み、次の段階でLLMによる精緻な判断を行うという設計である。

**第2段階：LLMによる関連性分析**

選出された候補に対して、LLMが「このメモリとこのメモリの間に意味的な接続があるか」を判断する。ここがA-Memの巧みなところで、埋め込みベースの類似度だけでは捉えられない関連性、例えば因果関係、対比関係、概念的な上位・下位関係をLLMの言語理解能力で識別できる。

Zettelkastenにおいて、カード同士のリンクが単なるカテゴリ分類ではなく文脈に基づく意味的な接続であるのと同様に、A-Memのリンクも表面的な類似性を超えた関係性を捉える設計になっている。

## メモリ進化：新しい経験が既存の知識を更新する

A-Memで最も興味深い仕組みが、メモリ進化（Memory Evolution）である。新しいメモリが追加され、既存メモリとのリンクが生成された後、A-Memはリンク先の既存メモリの内容自体を更新する。具体的には、既存メモリの文脈記述（Context）、キーワード、タグが、新しいメモリとの関係性を踏まえて書き換えられる。

これは、人間の学習プロセスに対する洞察に基づいている。我々が新しい知識を得ると、既存の知識の理解も変化する。ある概念を最初に学んだときの理解と、関連する複数の概念を学んだ後の理解では、同じ概念に対する記述が変わるのは自然なことである。時間とともに、個々のメモリは孤立した情報の断片ではなく、ネットワークの中で文脈化された知識へと成長していく。

## メモリ検索：リンクを辿って関連知識を引き出す

検索時は、クエリテキストの埋め込みベクトルとのコサイン類似度でtop-k個のメモリを取得する。ここまでは一般的なベクトル検索と同じだが、A-Memでは**検索されたメモリと同じ「ボックス」に属するメモリも自動的に取得される**。

ボックスとは、リンクで相互に接続されたメモリのグループであり、Zettelkastenで関連するカードが同じ箱にまとめられることに由来する。1つのメモリは複数のボックスに同時に属することができる。直接的にはクエリに類似していなくても、文脈的に関連する情報がリンクを通じて芋づる式に引き出される。この特性がどのようなタスクで効果を発揮するかは、次の実験セクションで詳しく見ていく。

# A-Memの評価：既存手法との比較

A-Memは2つの長期対話データセット上で、6つの基盤モデル（GPT-4o-mini、GPT-4o、Qwen2.5-1.5B/3B、Llama 3.2-1B/3B）を用いて評価されている。

* **LoCoMoデータセット**：平均9Kトークン、最大35セッションにおよぶ長期対話データセット。7,512の質問-回答ペアを含み、Single-Hop、Multi-Hop、Temporal、Open Domain、Adversarialの5カテゴリで構成される
* **DialSimデータセット**：人気TVシリーズ（Friends、The Big Bang Theory、The Office）に基づく長期マルチパーティ対話データセット。1,300セッション、約350,000トークンを含む

比較対象のベースラインには、前述のMemGPTやMemoryBankに加え、LoCoMoデータセットと同名のLoCoMo手法、およびReadAgentが使用されている。前節で紹介したMem0は、論文のRelated Workでは課題の分析対象として取り上げられているが、実験の比較対象には含まれていない。

## マルチホップ推論での顕著な改善

マルチホップ推論とは、1つの情報源だけでは回答できず、複数の情報を「ホップ」して辿る必要がある推論のことである。たとえば「Aliceの同僚が好きな映画は？」という質問に答えるには、まず「Aliceの同僚はBob」という情報を見つけ、次に「Bobが好きな映画はX」という情報を見つける必要がある。

![LoCoMoデータセットでの実験結果](https://static.zenn.studio/user-upload/deployed-images/c66846baa89335996493bd67.png?sha=478b6b92fa842c319c9bec24ff20291b2029eff5)  
*出典：Xu et al. (2025) Table 1 — LoCoMoデータセットでの5カテゴリ×6モデルの実験結果（F1/BLEU-1スコア）*

Table 1によると、A-Memが最も大きな効果を発揮するのはMulti-Hopカテゴリである。GPT-4oでのMulti-Hop F1スコアを見ると、A-Memが32.86で最高値を記録し、他のベースライン手法を上回っている。オープンソースモデルではこの差がさらに顕著になる。

これはリンク構造の効果を直接反映している。マルチホップ推論では、散在する複数の情報を結びつける必要があるが、A-Memのリンクにより関連メモリが自動的に連鎖して取得されるため、この種のタスクで特に有利に働く。

## リンク生成とメモリ進化の相補性

Ablation Study（Table 3）では、リンク生成（LG）とメモリ進化（ME）の各モジュールの貢献が検証されている。両方を除去すると特にMulti-HopとOpen Domainで顕著な性能劣化が発生する。LGのみを残した場合は中間的な性能を維持し、リンク生成がメモリ組織化の基盤として不可欠であることを示している。MEはその上にさらなる洗練を加え、特に複雑な推論を要するタスクで効果を発揮する。両モジュールは相補的な関係にある。

# 実装から見るA-Memの現在地

ここまでA-Memの設計思想と実験結果を見てきた。理論的には魅力的だが、では実際に日頃からClaude Codeなどを使っている一般ユーザーが、この仕組みを自分のメモリ管理に導入できるのだろうか。論文著者らが公開している[プロダクション向け実装](https://github.com/WujiangXu/A-mem-sys)のコードを読み、現時点での実用性を評価した。

## 現時点の実装状況

この実装はPythonライブラリとして提供されており、想定される使い方は**自分でLLMエージェントをPythonで構築し、そのメモリ層としてA-Memを組み込む**というものである。

```
from agentic_memory.memory_system import AgenticMemorySystem

# エージェントの初期化時にA-Memをメモリ層として構成
memory = AgenticMemorySystem(
    llm_backend="openai",
    llm_model="gpt-4o-mini"
)

# エージェントのメインループ内で直接呼び出す
memory.add_note("ユーザーはPythonでデータ分析を行っている")
memory.add_note("ユーザーはpandasよりpolarsを好む")

# 検索時はリンクを辿って関連メモリも取得される
results = memory.search_agentic("ユーザーの技術的な好み", k=5)
```

つまりA-Memは、Claude CodeやLangChainのような既存フレームワークに後付けするプラグインではなく、エージェントをゼロから構築する研究者がメモリ層として使うライブラリとして設計されている。そのため、既存フレームワークのユーザーがA-Memを利用するには、フレームワークとA-MemのPython APIの間を橋渡しするアダプタ層を自分で開発する必要がある。たとえばA-Memの`add_note`や`search`をフレームワーク側のツール呼び出しとして公開する中間サーバーなどが考えられる。

たとえばClaude Codeとの統合を考えると、A-MemのPython APIをMCP（Model Context Protocol）サーバーとしてラップし、`save_memory`や`search_memory`といったツールを公開する方法が自然である。その場合のワークフローは以下のようになる。

```
【保存フロー】
ユーザーとの対話
  ↓
Claude Code（LLM）が「このやり取りはメモリに残すべき」と判断
  ↓
MCPツール save_memory(content="ユーザーはpandasよりpolarsを好む") を呼び出す
  ↓
MCPサーバー内部で A-Mem の add_note() が実行される
  ├→ analyze_content(): キーワード・タグ・コンテキストを生成（LLM呼び出し1回目）
  └→ process_memory(): 近傍メモリとのリンク生成・既存メモリの進化（LLM呼び出し2回目）

【検索フロー】
ユーザーが質問する
  ↓
Claude Code（LLM）が「過去のメモリを参照すべき」と判断
  ↓
MCPツール search_memory(query="データ分析の好み") を呼び出す
  ↓
MCPサーバー内部で A-Mem の search_agentic() が実行される
  └→ ベクトル検索 + リンクを辿って関連メモリを芋づる式に取得
  ↓
取得されたメモリがClaude Codeのコンテキストに注入され、応答に活用される
```

ポイントは、「いつ・何を保存/検索するか」はClaude Code側が判断し、「保存された内容をどう組織化するか」「検索時にどうリンクを辿るか」はA-Mem側が自律的に決めるという責務の分離である。この分離はMCPのツール呼び出しモデルとよく合致する。

また、現時点ではメモリの永続化が未実装（インメモリのみで、プロセス終了時に消失する）、LLMバックエンドにAnthropic APIがないといった制約もある。ただし、これらはエンジニアリング上の未実装であり、追加開発で対応可能な範囲である。A-Memの設計自体がこれらを妨げているわけではない。

## 本質的な制約：書き込みコスト

より根本的な課題は、**メモリを1件追加するたびにLLMを最低2回呼び出す**というアーキテクチャ上のコストである。1回目はキーワード・タグ・コンテキストの生成（`analyze_content`）、2回目は近傍メモリとの関連分析に基づくリンク生成と進化判断（`process_memory`）である。

論文が示した「操作あたり約1,200トークン、$0.0003未満」という数値は、選択的top-k検索メカニズムによる効率化を反映した平均値である。しかし実装を見ると、書き込み時にはこの平均に加えて、メタデータ生成と進化プロセスのLLM呼び出しが発生する。さらに、メモリ件数が増えるにつれて近傍検索の候補が増え、進化プロンプトに含まれる既存メモリの情報量も増加する。

これはリンク生成とメモリ進化をLLMの判断で自動化するというA-Memの設計思想に内在するトレードオフである。メモリを「書いて終わり」ではなく「書くたびにネットワーク全体を賢くする」という設計は、必然的に書き込み側のコストを引き上げる。日常的に頻繁にメモリを書き込むユースケースでは、レイテンシとAPIコストの両面で負担になりうる。メモリの永続化やMCP対応と異なり、この制約はA-Memのアーキテクチャそのものに由来するため、単純な追加実装では解消できない。

ただし、前述のようにA-MemをMCPサーバーとして実装すれば、エージェント本体とは独立したプロセスになるため、異なるLLMを使うことができる。たとえばエージェント本体はフロンティアモデルで動かしつつ、A-Mem内部のメタデータ生成・進化判断にはOllamaでローカルLLMを使う構成にすれば、書き込みのたびに発生するAPI料金をゼロにできる。この場合、残る課題はレイテンシに限定される。一方で、論文の実験結果が示すように、GPT系のフロンティアモデルからローカルで動作するオープンソースモデルに変えると、マルチホップ推論などのタスク精度が低下する傾向がある。タスク精度を取るか、コストとレイテンシを取るか。この「品質 vs コスト」のトレードオフは、A-Memを実運用に載せる際に避けて通れない判断になる。

# A-Memから得られる示唆

A-Memが示しているのは、**同じモデルを使っていても、メモリの構造や更新の設計次第でエージェントの振る舞いは大きく変わる**ということである。論文の実験結果が端的にそれを示している。同一のGPT-4oでも、メモリシステムをA-Memに変えるだけでマルチホップ推論の性能が向上する。モデルは同じでも、メモリの組織化が変われば引き出せる知識が変わるのである。

では、どのようなメモリ構造が有効なのか。情報をMarkdownなどのテキストデータとして並べておくだけでは、AIエージェントにとって扱いやすいとは言いがたい。エージェントがAgentic Searchの仕組みで情報を検索・取得することを考えると、散在する情報を結びつける能力が重要になる。A-Memの実験でマルチホップ推論の性能が特に向上したのは、Zettelkastenのリンキング原則に基づいてメモリ間に文脈的な接続を持たせたためである。カテゴリ分類ではなく文脈に基づく接続を重視するというZettelkastenの原則が、LLMのメモリにおいても有効であることを、A-Memは実験的に示した。こうした知識のネットワーク化をAIエージェントのメモリに活用する動きは研究にとどまらず、実践レベルでも進んでいる。たとえば、MCP経由でObsidianをClaude Codeの外部メモリとして接続し、セマンティック検索やwikilinksによるナレッジグラフを活用する[取り組み](https://x.com/intheworldofai/status/2039255561280057794)も登場している。

# おわりに

A-Memの設計は、LLMエージェントのメモリシステムにおける重要な方向転換を示している。従来の「保存して検索する」パラダイムから、**メモリ自身が自律的に組織化・進化する**パラダイムへの移行である。Zettelkastenの原子性・リンキング・進化という3つの原則がシステム設計に具体的に反映され、特にマルチホップ推論のような複雑なタスクで既存手法を上回る成果を示した。

一方で、実装を読むと現時点では研究プロトタイプの段階であり、書き込みのたびにLLMを呼び出すコストはアーキテクチャに内在するトレードオフとして残る。メモリの組織化品質が基盤LLMの能力に依存するという制約や、テキスト以外のモーダリティへの対応も今後の課題である。それでも、同じモデルでもメモリの構造次第でエージェントの振る舞いが変わるというA-Memの知見は、エージェントのメモリ設計を考えるうえで重要な示唆を与えるものだと考えている。
