## カスタマイズ疲れ

GitHub Copilot のカスタマイズ機能（Custom Instructions、Prompt Files、Custom Agents、Agent Skills、MCP、Hooks、Agent Plugins）は、急速に増えました。把握すべき概念が多すぎて、何から始めればいいのかわからない。そういう人は少なくないはずです。

また、ある程度トレンドを追っていて、カスタマイズの方法はしっているという人でも、「どのカスタマイズ方法をいつ使えばいいのか？」や「自分の目的にはどのプリミティブが最適なのか？」といった具体的な判断に迷うことがあると思います。

こうした問題を一気に解決するのが、VS Code の Copilot Chat 拡張機能に最初から同梱されている `agent-customization` スキルです。このスキルを使えば、今のベストプラクティスに従ったカスタマイズを、GitHub Copilot 自身に相談しながら進めることができます。

## Copilot Chat 拡張の 3 つの組み込みスキル

GitHub Copilot Chat 拡張機能をインストールすると、いくつかの設定が注入されます。`> Chat: Open Customization` コマンドから設定を見ると、Built-in として 3 つのスキルが入っていることがわかります。

![](https://static.zenn.studio/user-upload/ec78acbb912e-20260402.png)

1. `get-search-view-results`: VS Code の検索ビュー（`Ctrl+Shift+F`）の結果をエージェントに渡すスキルです。  
   「検索ビューに表示されている結果を分析して」のように依頼すると、内部で VS Code のコマンドを実行し、検索結果をエージェントのコンテキストに取り込みます。地味ですが、エージェントと「同じ画面を見ながら話す」体験を実現するスキルです。
2. `troubleshoot`: 「なぜスキルが読み込まれなかったのか」「なぜリクエストに時間がかかったのか」「なぜツールがスキップされたのか」。こういう疑問を、Copilot 自身のデバッグログ（JSONL）を読んで調査してくれます。
3. `agent-customization`: カスタマイズファイルの作成・更新・レビュー・デバッグを行うワークフロースキルです。

通常のスキル呼び出しと同じですが、これらのスキルを利用するには以下の２パターンがあります。

* `/<skill-name>` で明示的にスラッシュコマンドとして実行する
* スキルが必要になりそうな指示や文脈を与えて、自動的に読み込ませる

## agent-customization スキルを深掘りする

3 つの中で、今回取り上げるのは agent-customization です。実際の定義は手元の VS Code で確認（上述の設定パネル参照）していただければと思いますが、このスキルを使うメリットは大きく２つあります。

### ① プリミティブの使い分けを覚えなくていい

このスキルの SKILL.md には、Workspace Instructions / File Instructions / Prompts / Skills / Custom Agents / MCP / Hooks という 7 つのカスタマイズプリミティブの使い分け判断表（Decision Flow）が入っています。

ここで重要なのは、この判断表を自分で暗記する必要はないということです。スキルがすでに知っているので、「○○したいんだけど、どのカスタマイズを使えばいい？」と聞けばいい。次のような判断も任せられます。

* 「こういう指示は Instructions と Skill、どっちにすべき？」
* 「やっぱり Prompt がいいんじゃない？だめなの？なんで？」

この手の使い分けは公式ドキュメントを行き来しても結論が出ないことがあるので、スキルに丸投げするのが手っ取り早いです。

### ② アンチパターン（Common Pitfalls）を避けられる

個人的にこのスキルで最も価値があると感じるのが、Common Pitfalls のセクションです。これはやってはいけないアンチパターン・留意すべき点を教えてくれるセクションです。

たとえば次のようなことが書いてあります。

* スキルにとって `description` が発見の鍵だということ。`"Use when: ..."` パターンで具体的なキーワードを入れるのが定石。
* YAML フロントマターのサイレント失敗。値にエスケープされていないコロンが含まれている、タブがスペースの代わりに使われている、`name` がフォルダ名と一致しないなどのよくある構文エラー。
* インストラクションの `applyTo: "**"` はコンテキストを無駄に消費する。`**/*.py` や `src/api/**` のように具体的な glob を使うべきです。

## どんな風に使えるか

特別な設定は不要です。Copilot Chat にカスタマイズの相談を投げるだけ。よくあるシナリオごとに試し方を見ていきます。

### シナリオ 1: カスタマイズをゼロから始めたい

プロジェクトにまだカスタマイズファイルが1つもない状態。何から手をつけるべきかわからないとき。

「このプロジェクトのコーディング規約を Copilot に覚えさせたい」「Python の型ヒントを必ず付けるようにしたい」のように聞くと、スキルが Decision Flow に沿って「まず `copilot-instructions.md` を作りましょう」と提案し、プロジェクトのコードを分析してルールを生成してくれます。プリミティブの選定を自分でやる必要がない。

### シナリオ 2: カスタマイズが効かない原因を知りたい

`.instructions.md` を書いたのに反映されない、`.agent.md` がドロップダウンに表示されない。こういうときです。

「この `.instructions.md` がロードされない原因を調べて」「カスタムエージェントが表示されないのはなぜ？」のように聞いてみてください。

個人的な体験ですが、自分で `.agent.md` を書いたとき、チャットのドロップダウンにエージェントが表示されない問題が起きました。agent-customization スキルに聞いたところ、YAML フロントマターの構文エラー（description のコロンがエスケープされていない）を指摘してくれた。さらに troubleshoot スキルと組み合わせて discovery ログを確認すると、`name` フィールドとフォルダ名の不一致も見つかった。こういったことも可能です。

### シナリオ 3: 既存のカスタマイズを整理したい

1 つのファイルに全部詰め込んでしまって、何が何に効いているのかわからなくなった。そういう状態です。

「この copilot-instructions.md が肥大化したので整理したい」「ペルソナ定義と記事フォーマットのルールを分離したい」と伝えると、プリミティブの分離方針を提案してくれます。

ブログ執筆エージェントを作る際にまさにこの状態に陥りました。`.instructions.md` にペルソナ定義、記事フォーマット、執筆手順を全部書いたら、何が何に効いているのかわからなくなった。agent-customization スキルに相談したところ、ペルソナ定義は `.agent.md` に分離し、記事フォーマットのルールは `applyTo: "drafts/**"` の `.instructions.md` にするよう提案されました。理由も「ペルソナはエージェントモードの切り替えに紐づく情報だから `.agent.md` に属する」「フォーマットルールは特定ディレクトリにだけ適用すればいいから `applyTo` 付きの Instructions が適切」と、判断基準ごと返してくれる。

### シナリオ 4: 自分でスキルやエージェントを作りたい

組み込みスキルでは足りなくなったとき、自分のドメインに特化したスキルやエージェントを作る段階。

「テスト実行とデバッグのスキルを作りたい」「レビュー専用のカスタムエージェントを作りたい。ファイル編集ツールは使わせたくない」のように伝えると、Skill vs Prompt や Agent vs Skill の判断を整理したうえで、テンプレートを生成してくれます。`references/` 配下の詳細リファレンスを参照しながらファイルまで作ってくれるので、ゼロからの手探りとは効率が段違いです。

## まとめ: カスタマイズの相談相手はすでにいる

GitHub Copilot のカスタマイズ設定は柔軟である一方で複雑です。そうした複雑さを軽減するために、ぜひ VS Code の Copilot Chat で `agent-customization` スキルを使ってみてください。
