## システム構成

全体は3層です｡

```
┌──────────────────────────────────────────────┐
│ チャンネル層                                  │
│  daily_news / metal_report / その他           │
│  （各チャンネル = 独立したパイプライン）      │
└──────────────────────────────────────────────┘
                     ↓ 共通処理を呼ぶ
┌──────────────────────────────────────────────┐
│ 共通処理層                                    │
│  news_fetcher / tts_* / subtitle_* / video /  │
│  thumbnail / upload / cost_calculator ...     │
└──────────────────────────────────────────────┘
                     ↓ 外部APIを叩く
┌──────────────────────────────────────────────┐
│ 外部API層                                     │
│  Claude / Gemini TTS / VOICEVOX / ElevenLabs  │
│  STT / GNews / yfinance / YouTube Data API    │
└──────────────────────────────────────────────┘
```

### パイプラインの実行方式

肝は`pipeline_base.py`で､**各ステップを別プロセス（subprocess）として起動し､結果をディスク上のJSONでバケツリレー**していく設計です｡

src/common/pipeline\_base.py（概念）

```
# stepNN_*.py を subprocess.Popen で起動し、
# start_new_session=True でプロセスグループごとタイムアウトkillできるようにする。
# ステップ間の受け渡しは output/<channel>/<run_id>/*.json 経由。
```

なぜモノリシックな1プロセスにしなかったかというと､

* ステップ単位でタイムアウト・リトライ・強制killを効かせたい（TTSが固まっても他を巻き込まない）
* ステップ単体を`--run-id`指定で個別再実行してデバッグしたい
* メモリのリークやライブラリの相性事故を1ステップに閉じ込めたい

実運用ではAPIが平気で無応答になったり暴走したりするので､この「プロセス分離＋JSONバケツリレー」がかなり効きました｡

## 処理の流れ（大きく12ステップ）

メインの【多読ラジオ】（海外ニュース×日本経済）を例に取ると､1本の動画はこんな流れで作られます｡

1. **ニュース収集** — GNews APIから記事を集め､Claudeで重要度を判定して選定
2. **関連ニュース収集** — 選定テーマの関連記事・本文URLを追加取得
3. **市場影響分析** — Claudeで「このニュースが日本市場に与える影響」を構造化
4. **台本生成** — Claudeで2人の対話形式（専門家＋聞き手）の台本を生成
5. **TTS音声合成** — 話者ごとにGemini TTSで音声化 → イントロ/間を含めて結合
6. **字幕生成** — 生成音声をElevenLabs STTで書き起こし､台本と突き合わせてタイムスタンプ字幕を作成
7. **動画合成** — FFmpegで背景・字幕PNG・BGMを合成
8. **メタデータ生成** — Claudeでタイトル・説明・タグを生成
9. **コメント生成** — Claudeで固定コメントを生成
10. **サムネ生成** — Pillowでサムネ画像を描画
11. **アップロード** — YouTube Data APIで投稿
12. **完了処理** — コスト集計・一時ファイル削除・Discord通知

依存関係のないステップは並列で回しています｡まず**5・8・9（音声・メタ・コメント）を並列**で生成し､音声ができたら**それを使う6（字幕）と10（サムネ）を並列**で作る､という2段構えです｡台本を対話形式にしているのは､聞き手役の質問が視聴者の疑問を代弁してくれて理解しやすくなる／声が変わってリズムが出る､という狙いです｡

ここまでが動画生成の背骨です｡ここからが本題､**動かし続けながらの試行錯誤**の話｡

## 試行錯誤の日々

### LLMプロバイダの乗り換え遍歴

最初はニュース分析をGeminiでやっていました｡が､2025年10月時点のGemini無料枠がとにかく不安定で､ある日こうなりました｡

```
500 Internal error encountered.
google.api_core.exceptions.InternalServerError: 500 Internal error encountered.
```

1日で500 / 429 / 503が頻発してパイプラインが止まる｡無料枠は**サーバ側のエラー（500/503）が出やすく不安定**なうえ､**レート/クォータ上限（429）に当たればそこで生成が打ち止め**になります｡「使い切ると不安定」というより「そもそも安定して量産できる状態ではない」という感覚でした｡そこで分析はClaudeに移して､指数バックオフ（2s→4s→8s）のリトライを入れました｡

その後もLLMまわりは落ち着かず､

* Claude側が高負荷だと`529 overloaded`を返してくるので､GPTへのフォールバック経路を用意
* モデル廃止の追いかけっこ：`gemini-2.0-flash`系が2026-03-31で廃止予告 → 2.5へ｡`gpt-4o-mini`も引退でリネーム対応
* そのリネームで`gpt-5.1-mini`と書いたら**そんなモデルは存在しない**と怒られ､`gpt-5-mini`に即修正（コミット1本ぶんの恥ずかしいミス）

一時期は台本生成を含む大半を`claude-haiku-4-5`に寄せていました｡台本生成は1万字超の出力で全体コストの大半を占めるので､ここをHaikuにするだけでAPIコストが\*\*約66%\*\*下がったからです｡品質も対話台本なら十分でした｡その後､旧モデルの廃止リスク回避とチャンネル間の品質統一のため､**最終的には`claude-sonnet-4-6`に一本化**しています｡

教訓としては､**モデル名とエンドポイントはコード直書きにせず設定ファイルに逃がす**｡この領域はモデルが四半期で入れ替わるので､`settings.yaml`から差し替えられるようにしておかないと廃止のたびにコードを触るはめになります｡

### Gemini TTSのundocumentedな挙動（最大の沼）

一番時間を溶かしたのがGemini TTSです｡プレビュー版特有の､**ドキュメントに書いていない挙動**を片っ端から踏みました｡

**① 1日100リクエスト（RPD）の壁**

無料枠のTTSは1日あたりのリクエスト数上限が厳しく､最初は「100文字ごとにチャンク」していたので10分動画で約101リクエスト＝**1本で上限到達**でした｡そこで文字数ベースをやめて\*\*行数ベース（同じ話者の連続10行を1ブロック）\*\*に変えたところ､リクエスト数が約1/10になりました｡

```
旧: 100文字ごと → 101リクエスト
新: 10行ごと   → 約10リクエスト
削減率: 90%
```

話者が変わる箇所では必ずブロックを割る制約も入れています｡（その後さらに作り直し､いまは④の音質劣化とのバランスで**500文字前後を目安に台詞の切れ目で区切る**方式です）

**② `systemInstruction`非対応で、話し方の指示が読み上げられる**

「TVニュースキャスターとして落ち着いて話して」みたいなstyle promptを本文にprependしていたら､**その指示文そのものが音声として読み上げられる**事故が非決定的に起きました｡

原因は明快で､**TTSモデルが`systemInstruction`に非対応**だったこと｡本文に混ぜる以外に渡す方法がなく､混ぜれば読み上げリスクが残る｡最終的にstyle promptの使用自体をやめました｡

さらに保険として､生成音声の**冒頭5秒をSTTで書き起こして指示文の混入を検出**する仕組みも入れています｡

src/common/tts\_speaker\_unified.py

```
# TTS音声の冒頭を検証するための検出キーワード
_TTS_CONTAMINATION_KEYWORDS = [
    "ニュースキャスター", "キャスター", "専門家",
    "明瞭で", "温かみ", "落ち着いた", "権威のある",
    "話し方をして", "話してください",
    "あなたは",
]
```

**③ 特定の本文で400、しかも再送しても直らない**

句読点が過剰に連続していたり疑問符が密集した本文を投げると､400でこう返ってくることがありました｡

```
should only be used for TTS ...
（"tried to generate text" 系のメッセージ）
```

Geminiが入力を「音声化ではなくテキスト生成」と誤判定するらしく､**同じ本文を再送しても決定論的に同じ400**になります｡つまりリトライは無意味で､解消するには台本そのものを作り直すしかない｡そこで専用の例外を切って上位に投げ､**Step4（台本生成）からやり直す**フローにしました（最大2回）｡

src/common/tts\_gemini.py

```
# 本文起因の400（音声化拒否）は同一本文の再送では解消しないため、
# リトライせず例外を送出して上位（台本再生成）に委ねる
if response.status_code == 400 and _is_tts_content_rejection(response.text):
    logger.error("❌ Gemini TTSが本文の音声化を拒否（コンテキスト起因の400）。再送では解消しないため中断します。")
    raise TtsContentRejectedError((response.text or "")[:300])
```

**④ 長文を一度に渡すと音声がガビガビに劣化する**

文字数課金＋リクエスト数制限なので､コストを下げようと1リクエストに長文を詰めたくなります｡ところが**長文を渡すと生成音声がガビガビ（音割れ）になる**――これもundocumentedでした｡いまは**話者を問わず500文字前後を目安に台詞の切れ目でチャンク分割**（途中では切らない）して安定させています｡「長文でまとめたい（コスト）」と「長文だと崩れる（音質）」のバランスで目安文字数を決めている､というわけです｡

![視聴者コメント：反響する声をやめてほしいという指摘](https://static.zenn.studio/user-upload/deployed-images/56ec77a48487b5574bac747f.png?sha=01da55685cadda0e79e6fae3342bf9199c3a29ad)

!

**現在（2026年7月時点）はどうなっているか**

ここで挙げた沼の多くは「プレビュー版ゆえ」のものでしたが､TTS系モデルは今も**Preview扱い**のままです｡`gemini-2.5-flash-preview-tts` / `-pro-preview-tts` に加えて新しい `gemini-3.1-flash-tts-preview` が増えましたが､いずれもPreview｡モデル名に`-preview`が付く点も当時のままです｡

話し方の指定も依然として**専用パラメータ（`systemInstruction`）ではなくテキスト埋め込み方式**で､②の根っこは残っています｡ただし当時と違い､`[whispers]` `[excited]` のような**インラインの音声タグ**が公式に用意され､より構造化した指定はできるようになりました｡

レート制限（①）は全体的に緩和が進んでいますが､TTSの無料枠は今も案内が薄く､量産用途では有料ティア前提と考えたほうが安全です｡（参考: [公式ドキュメント Speech generation](https://ai.google.dev/gemini-api/docs/speech-generation)）

### TTSエンジンそのものの選定

TTSは「品質・コスト・言語対応」のトレードオフで､チャンネルごとに使い分けています｡

メインの多読ラジオは､わりと早い段階で**Gemini 2.5のTTS**に決めました｡自然で表現力のある日本語ナレーションが出せて､対話形式のニュース解説に向いていたからです｡

TTS選定では､性質の違う2つの問題を別々に潰しました｡

**① 無料枠のレート制限 → 有料ティアへ**  
まず引っかかったのが無料枠のレート制限です｡**Gemini無料枠は3 RPM**と厳しく､10分台本（13セグメント）がまともに通りません｡初期テストでは**13セグメント中1本しか完成せず（約8%）**､完了まで30分以上かかる見込みでした｡メインの多読ラジオは､ここで**有料ティアに切り替えてレート制限を解消**しています｡

**② 文字数課金でコストが積む → VOICEVOXも併用**  
もう一つはコストです｡Gemini TTSは**文字数に応じた課金**なので､毎日何本も回すチャンネルで使い続けると地味に積みます｡そこで**コストを抑えたいチャンネルにはVOICEVOX**（ローカルDockerで無料）を使う構成を増やしました｡週刊メタルレポートはこちらで､レート制限もAPI課金もかからず安定します｡

参考までに､初期に無料枠のGeminiとローカルのVOICEVOXを比べたときの数字がこれです｡

| 指標 | Gemini TTS（無料枠） | VOICEVOX |
| --- | --- | --- |
| 成功率 | 8%（1/13） | 100%（13/13） |
| 生成時間 | 30分超（見込み） | 55秒 |
| レート制限 | 多発（3 RPM） | 0 |
| コスト | $0（だが詰まる） | $0 |

**ただしVOICEVOXにも弱点があります｡無料で高品質なのはいいのですが､基本的に日本語専用**です｡韓国語・繁体字といった多言語チャンネルではそもそも使えず､そこはGemini/ElevenLabs側に頼ることになります｡

さらに日本語チャンネルでも､**英語の固有名詞・銘柄名・専門用語をそのまま正しく読んでくれません**｡横文字だらけの金属／暗号資産チャンネルだと特に顕著です｡そこで**読み替え辞書**（`config/pronunciation_ja.yaml`）を用意して読みを補正しています｡

config/pronunciation\_ja.yaml

```
# 固有名詞や特殊な読み方をする単語の読み方を定義
孫正義: そんまさよし
ChatGPT: チャットジーピーティー
Vitalik: ヴィタリック
```

この辞書は運用しながら「またこの単語を誤読した」と気づくたびに追記していて､地味にメンテが効くところです｡

### 声色の統一（結局あきらめた話）

対話形式のニュース解説は､A（聞き手）とB（専門家）が交互に喋ります｡ところがGemini TTSは**リクエストごとに声が微妙に揺れる**ので､台詞を1つずつ投げると**同じ話者なのに声色・トーン・話速がブレて**､聞くと「毎回ちょっと別人」のような違和感が出ます｡

当初は2話者を1リクエストで喋らせる**マルチスピーカー方式**（`multiSpeakerVoiceConfig`）を使っていましたが､台本が長くてチャンクをまたぐと､やはり同じ話者の声色がブレました｡

いま採用しているのが**話者統一方式**です｡考え方はシンプルで：

1. **話者ごとに台詞をまとめて生成する** — Aの台詞はできるだけ1リクエストに､Bも同様に｡同じリクエストで生成したぶんは声色がそろいやすくなります｡ただし長い話者は④のとおり台詞の切れ目で500文字前後に分割するので､**そのチャンクごとに別リクエスト＝生成タイミングが変わると結局ズレます**（あくまで"マシになる"程度）
2. ただしこれだと「Aの音声ブロック」「Bの音声ブロック」が別々にできるだけで､**会話の順番には並んでいません**
3. そこで各話者の音声を**STTで書き起こしてタイムスタンプを取り､台詞間の無音区間で元の1台詞ずつに切り分け､対話の順番に再構築**します

この「長い単一話者音声を､元の台詞単位に正確に切り戻す」ところが一番の難所で､次の**字幕タイミング**（無音区間検出＋DP最適分割）とまったく同じ仕組みを使っています｡

ただ正直に言うと､**これで声色のブレが完全に消えるわけではありません**｡話者ごとにまとめて生成することでマシにはなりますが､最終的にどんな声色・トーンで返ってくるかは**そのときのGeminiの"機嫌"次第**で､こちらから完全にコントロールする手段がない｡なので**完璧な統一はいったん諦めていて**､TTSモデル側が進化して安定するのを待っている､というのが実状です｡

声のブレ具合について､視聴者からこんな指摘が…｡申し訳ない｡

### 字幕とセリフのズレ

字幕は「TTSで作った音声をSTTで書き起こし直して､台本テキストと突き合わせてタイムスタンプを取る」という方式にしています｡最初にElevenLabs STTを入れた効果は劇的で､**音声と字幕のズレが124秒 → 2秒**まで縮みました｡

ただ､STTのテキストマッチにも限界があり（誤認識で対応が崩れる）､最終的には**無音区間検出による動的計画法（DP）の最適分割**を主軸にして､STTは分割が範囲外になったときの**フォールバック**へ降格させました｡いまは「無音DP分割＋STTフォールバック＋タイミング整合チェック」の三段構えです｡

### 環境・ビルドまわりの罠

APIだけでなく､ローカル環境やビルド構成にも刺されました｡

**`No such filter: 'drawtext'`**

エンディングのCTAテキストをffmpegの`drawtext`で描いていたら､ある端末で突然これが出ました｡

```
No such filter: 'drawtext'
```

原因は端末のffmpeg（8.1.1）が**libfreetype無しでビルド**されていたこと｡`drawtext`はfreetype依存なので､ビルドフラグ次第で存在しません｡そこでテキストは**Pillowで透明PNGに描画してoverlay合成**する方式に統一し､ffmpegのビルド構成に依存しないようにしました（字幕もこの方式）｡

そのほかにも､

* スリープでAPI接続が切れる → `caffeinate -i`で実行中はスリープ抑止
* Apple Siliconでx86\_64パッケージが混ざって`incompatible architecture` → venvでクリーンに
* パイプラインが7時間走り続ける事故 → ステップ単位のタイムアウト（既定10〜20分）に加え、一部チャンネルは全体2時間キャップ

といった「動かし続けて初めて出る」系を潰していきました｡

### 運用・プラットフォーム側の罠

**YouTube Data APIのquota**

多言語展開を狙ったとき､動画1本のアップロードが1,600 quota｡8言語なら`1,600 × 8 = 12,800`で**デフォルトの10,000/日を超過**します｡quota増加申請は審査に数週間かかるので､**言語（チャンネル）ごとに独立したGCPプロジェクトを作って10,000 quotaずつ確保**する構成にしました｡

**ニュース取得ソースの変遷**

* NewsAPI → 無料枠だと日本語記事が弱く､`/v2/top-headlines`の`country=jp`が0件
* Google News RSS → 無料・無制限で日本語が豊富｡いったんここに移行｡ただし**RSSが返すのはGoogleNews経由のリダイレクトURLで元記事の本文が読めず**､本文なしでは肝心のニュース分析ができないので断念
* GNews API（有料） → **本文まで取れる**ので最終的にここへ

**ポリシー違反でBan**

多言語版のうち1チャンネルが､YouTubeのスパム・欺瞞・詐欺ポリシー違反で削除されました｡厄介なのは**ポリシー名しか提示されず具体的な該当箇所が分からない**こと｡断定的な予測をサムネに書いていたのがクリックベイト扱いされた疑いはありますが確証はなく､予防的にメタ生成プロンプトへ「将来予測はサムネ/タイトルに断定で書かない」を追加しました｡多言語化の仕組み自体は残しつつ､そのチャンネルのコード・認証は撤去しています｡

## 堅牢化と自動運用のしくみ（軽め）

全自動で毎日回す以上､「壊れても勝手に立ち直る／気づける」仕掛けが必要でした｡

* **コスト集計**: ステップごとにClaudeのトークン数・Gemini TTSの文字数・ElevenLabsの秒数を集計してJPY換算（`cost_calculator.py`）
* **自動クリーンアップ**: 7日より古い`.mp4`/`.mp3`を削除（台本・メタ・サムネは残す）｡ディスクを大きく節約
* **外出先からの再実行**: 以前はGitHub Issueをポーリングする自作watcherで失敗チャンネルを再実行していましたが､いまは**Claudeのディスパッチ機能で外出先から直接指示**しています（自作の仕掛けが不要になった）
* **Discord通知**: チャンネルごとに成功／失敗を通知
* **評価サブシステム**: 生成した台本・メタデータ・ニュース選定をLLMで採点して週次レポート化

実際の通知はこんな感じで飛んできます｡失敗はエラー詳細つき､成功は実行時間とAIコストつき｡

## 自己ドキュメント文化

このプロジェクトは`Claude Code`でのいわゆるバイブコーディングで育てたのですが､途中から**意思決定と失敗を必ずログに残す**ようにしました｡

* `DECISION_LOG.md` — 「なぜGeminiをやめてClaudeにしたか」などの技術判断とエラー実物
* `IMPROVEMENT_LOG.md` — 「仮説 → 観察指標 → チェックポイント → 結果」の閉ループで施策を検証

この記事の失敗談の多くは､このログとコミットメッセージから拾い直したものです｡ちなみにコミットの`Co-Authored-By`を辿ると`Claude` → `Opus 4.5` → `4.6` → `4.7` → `Opus 4.8`と並んでいて､**使ったモデルの世代交代の記録**にもなっていました｡自動化を作るときこそ､判断の理由を書き残しておくと後の自分がめちゃくちゃ助かります｡

## まとめ

「AIで動画を全自動量産」と言うと簡単に聞こえますが､実体は**当時の尖ったAPIと延々と格闘するエンジニアリング**でした｡最後にハマりどころを一覧にしておきます｡なお､ここに挙げたのは代表的なものだけで､実際は他にも数え切れないほど踏んでいます｡挙げるとキリがないので､この辺で｡

| ハマった罠 | 症状 | 回避策 |
| --- | --- | --- |
| Gemini無料枠が量産に不向き | 500/503頻発＋429で打ち止め | 分析をClaudeへ＋指数バックオフ |
| Gemini TTSのRPD | 1本で100req/日に到達 | 文字数→行数ベースで90%削減 |
| systemInstruction非対応 | 指示文が読み上げられる | style prompt停止＋冒頭STT検証 |
| 本文起因の400 | 再送しても直らない | 台本ごと再生成 |
| 長文で音質劣化 | 長文を渡すとガビガビ音割れ | 台詞の切れ目で分割（〜500字目安） |
| Gemini TTS無料枠のレート制限 | 3 RPMで成功率8% | 有料ティアへ |
| Gemini TTSの文字数課金 | 毎日回すとコストが積む | 一部チャンネルをVOICEVOXへ |
| 字幕ズレ | 音声と124秒ズレ | STT＋無音DP分割の多段構え |
| ffmpeg drawtext | `No such filter` | Pillowオーバーレイへ |
| YouTube quota | 8言語で上限超過 | 言語別GCPプロジェクト |

ここまでが「作り方」と「ハマった罠」の技術的なリキャップです｡最後に､運用してみての正直な感想を書いて締めます｡

## で､これは割に合うのか？（運用のリアル）

･･･正直､微妙です｡

ここまで「全自動で動画が量産できる」と説明してきましたが､**実態はそんなに甘くありません**｡

* **毎日十数本**を人手ゼロで生成・投稿できるのは､確かに気持ちいい｡毎朝起きると勝手に動画が上がっています｡
* でも､**ほぼ毎日どこかを改修しています**｡APIの仕様変更､モデルの廃止､たまたま踏んだ本文パターンの400…｡この記事の失敗談は"過去形"で書いていますが､新しい罠は今も増え続けていて､**完全放置で回り続けるわけではない**のが実態です｡
* 極めつけは**YouTubeのBan祭り**（前述の「運用・プラットフォーム側の罠」）｡ポリシー名しか提示されず理由の詳細は分からず､対策も手探り｡コツコツ積み上げたチャンネルが一発で消えるリスクは常に隣にあります｡

トータルで見ると､**正直「割に合う」とは思えません(笑)**｡それでも､"毎朝勝手に動画が上がっている"という状態そのものが面白くて続けています｡

なお､**収益・視聴指標・チャンネル運営まわりの話は技術とは別軸なので､別記事に書く予定**です｡この記事はあくまで「作り方と､ハマった罠」に絞りました｡

同じように「AIで動画自動生成やってみたい」という方の地図になれば幸いです｡
