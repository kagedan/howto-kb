> 参考一次情報  
> <https://arxiv.org/abs/2601.21233>

## コードエージェントとは

コードエージェントは、AI エージェントの一種です。

一般的な AI エージェントとの違いは、主に対象タスクと権限にあります。コードエージェントは、LLM を中核にしつつ、コードの読解・生成、ファイル操作、シェル実行、テスト実行、外部ツール呼び出しなどを組み合わせて、ソフトウェア開発タスクを進めます。

たとえば、Claude Code、Cursor、GitHub Copilot、Codex CLI、Gemini CLI のような開発支援エージェントがイメージしやすいです。

通常のチャットボットと比べると、コードエージェントは次のような特徴を持ちます。

* コードベースや実行環境にアクセスできる
* ツールを呼び出して実際の操作を行える
* 複数ターンにわたって計画しながら作業できる
* サブエージェントや専用モジュールを持つ場合がある

## コードエージェントのシステムプロンプト

システムプロンプトには、モデルやエージェントの振る舞いを決める重要な情報が含まれます。

たとえば、以下のような情報です。

* エージェントの役割
* ユーザー指示、システム指示、ツール出力の優先順位
* 許可する依頼と拒否する依頼の境界
* ツールを使う条件
* 安全ポリシーや拒否方針
* サブエージェントごとの役割や制約

これらは、エージェントを安全に動かすための設計情報です。しかし攻撃者から見ると、後続の攻撃を組み立てるためのヒントにもなります。

たとえば、拒否条件や例外条件が分かれば、その境界を狙ったプロンプトインジェクションを考えやすくなります。ツール利用条件が分かれば、どの操作に誘導すれば危険かも推測しやすくなります。

つまり、プロンプトリーキングは単なる「文章の漏えい」ではありません。AI エージェントの防御設計や操作ルールが外から見えるようになる「偵察段階のリスク」として見る必要があります。

## 提案手法「JustAsk」とは

プロンプトリーキング自体は新しい話ではありません。従来から、システムプロンプトや隠しプロンプトを引き出そうとする研究や攻撃テンプレートは存在します。

JustAsk の新規性は、「人間が考えた固定プロンプトを投げる」のではなく、「コードエージェントが対象モデルの反応を見ながら、次に試す抽出戦略を選んでいく」点にあります。

イメージとしては、次のような違いです。

| 観点 | 従来のプロンプトリーキング | JustAsk |
| --- | --- | --- |
| 攻撃方法 | 固定テンプレートや手作業の試行に依存しやすい | エージェントが複数の戦略を選びながら試す |
| 対話 | 単発または短い会話が中心 | 必要に応じて複数ターンで進める |
| 学習 | 事前データや手作業の知見に依存しやすい | 対象モデルの応答から有効な戦略（選択するスキル）を更新する |
| 検証 | 得られた内容が本物か判断しにくい | 複数の試行結果の一貫性で確認する |

JustAsk は、14個の低レベルスキルと14個の高レベル戦略を持っています。低レベルスキルは単発の聞き方、高レベル戦略は複数ターンにわたる進め方です。

スキル選択には UCB（Upper Confidence Bound）という考え方が使われています。これは、過去にうまくいった戦略を使いながら、まだ十分に試していない戦略も試すための仕組みです。

要するに JustAsk は、プロンプトリーキングを「プロンプト職人芸」から「エージェントによる自動探索問題」に近づけた研究だと言えます。

## 実験結果

本ペーパーでは、Claude Code のケーススタディに加えて、OpenRouter 経由でアクセス可能な 41 個のモデルを対象に評価しています。

ブラックボックスのモデルを相手にすると、本物のシステムプロンプトを直接見ることはできません。そのため、抽出された内容が本当に正しいのか、単なるもっともらしい説明なのかを見分ける必要があります。JustAsk では、この判断に一貫性スコアを使います。本ペーパーでは、抽出結果同士の意味的類似度を計算し、一貫性スコアが0.7以上であれば成功とみなしています。

これは完全な正解確認ではありません。ハルシネーションでもっともらしい説明が複数回出る可能性はあります。そのためペーパーでは、Grok の公開システムプロンプトや Claude Code のリバースエンジニアリング結果など、正解に近い情報があるケースでも比較しています。

実験結果の要点は下記のとおりです。

| 観点 | 結果 |
| --- | --- |
| 対象モデル数 | 41 |
| 抽出成功率 | 100%（一貫性スコア 0.7 以上） |
| 平均ターン数 | 全体で 2.8 ターン |
| クローズドモデル | 平均 4.8 ターン |
| オープンウェイト系モデル | 平均 2.4 ターン |
| ファインチューニングモデル | 平均 1.3 ターン |

ここで注意したいのは、「100% 成功」といっても、すべてのモデルからシステムプロンプトを完全に丸ごと取り出したという意味ではないことです。

抽出品質は、以下のように分けられています。

| 品質 | 割合 | 概要 |
| --- | --- | --- |
| Verbatim | 2% | ほぼそのままのプロンプト本文が得られる |
| Strong semantic | 29% | 具体的なルールや優先順位まで詳細に復元される |
| Weak semantic | 68% | 役割や一般的な安全方針が中心に復元される |

つまり、ほとんどのケースでは「逐語的な漏えい」よりも「意味的な漏えい」が中心です。

ですが、防御側にとってはこれでも十分に問題です。正確な文面が分からなくても、どの依頼を拒否するか、どの順序で指示を優先するか、どのツールをどう扱うかが見えてくると、次の攻撃を考えやすくなるためです。

## 抽出結果から見えた傾向

Claude Code のケーススタディでは、メインエージェントだけでなく、探索、計画、シェル実行などのサブエージェントに関する情報も抽出対象になっています。ここで重要なのは、コードエージェントが単体のチャットモデルではなく、複数の役割や権限を持つ部品の組み合わせになっていることです。

たとえば、あるサブエージェントは読み取り専用、別のサブエージェントはシェル実行に関する詳細な制約を持つ、といった分離が行われています。これは防御上は良い設計です。しかし、その分離の内容が外から推測できると、攻撃者は「どの境界を狙えばよいか」を考えやすくなります。

この点で、コードエージェントにおけるプロンプトリーキングは、単に隠しプロンプトが見える問題ではありません。エージェント全体の権限設計や運用ルールが外から観察される問題として捉える必要があります。

本ペーパーでは、抽出されたプロンプトの内容も分析しています。まず、多くのモデルで Helpful、Honest、Harmless（HHH）の原則に近い結果が確認されました。  
HHHの原則については、[Prioritization First, Principles Second: An Adaptive Interpretation of Helpful, Honest, and Harmless Principles](https://arxiv.org/abs/2502.06059)もあわせてご参考ください。

| 原則 | 抽出結果に含まれた割合 |
| --- | --- |
| Harmless | 96% |
| Helpful | 91% |
| Honest | 89% |

これは、多くの LLM が似た安全設計の言葉を持っていることを示しています。一方で、安全ポリシーのカバー範囲にはばらつきもありました。

| カテゴリ | 明示的に含まれた割合 |
| --- | --- |
| 違法行為 | 83% |
| プライバシー侵害・ドキシング | 78% |
| 暴力・身体的危害 | 63% |
| 自傷・自殺 | 48% |
| マルウェア・サイバー攻撃 | 46% |
| 詐欺・なりすまし | 37% |
| 児童性的虐待関連（CSAM） | 20% |

なお、CSAM が 20% と低く見えるのは、モデルが問題視していないという意味ではありません。本ペーパーでも、表現や用語の揺れによって明示的なポリシー文として検出されにくく、実際より低く出ている可能性が高いと注記されています。

特にコードエージェントでは、サイバーセキュリティ関連の依頼をどこまで許可し、どこから拒否するかが重要です。防御的な診断や教育的な説明は許可したい一方、マルウェア作成や攻撃自動化は拒否する必要があります。この境界が曖昧だと、抽出された情報をもとに攻撃側が抜け道を探しやすくなります。

また、41 モデル中 11 モデル、つまり 26.8% で Identity Confusion（アイデンティティ混同）が確認されています。これは、モデルが実際の開発元とは異なる開発元を名乗る現象です。この結果は、モデル自身の自己申告をそのまま監査やリスク評価に使うのは危険だということも示しています。

## 防御はどう考えるべきか

本ペーパーでは、防御策についても評価しています。比較されたのは、追加保護なし、単純な禁止指示、攻撃手法を意識した保護指示の 3 パターンです。

| 防御 | 平均的な低減効果 |
| --- | --- |
| 単純な禁止指示 | 6.0% |
| 攻撃手法を意識した保護指示 | 18.4% |

ここから分かるのは、「システムプロンプトを漏らさないでください」と書くだけでは弱いということです。  
一方で、攻撃手法を意識した保護指示でも完全には防げていません。  
つまり、プロンプトだけでプロンプトリーキングを防ぎ切るのは難しい、という見方が妥当です。

実務で重要なのは、以下のような防御の考え方です。

1. システムプロンプトに秘密情報を入れない
2. プロンプトが漏れても権限が広がらないようにする
3. ツール権限を最小化する
4. ファイル操作やシェル実行はサンドボックス化する
5. サブエージェントごとに役割と権限を分ける
6. 単発入力だけでなく、複数ターンの誘導を監視する
7. 自社エージェントに対してプロンプトリーキング耐性を定期的に検証する

## まとめ

今回は、「Just Ask: Curious Code Agents Reveal System Prompts in Frontier LLMs」というペーパーをもとに、コードエージェントを使ったプロンプトリーキング手法である JustAsk を紹介しました。

この論文の要点は、システムプロンプト抽出を「固定プロンプトによる一発勝負」ではなく、「コードエージェントによる自動探索」として扱ったことです。

また、実験結果からは、システムプロンプトが逐語的に漏れなくても、役割、拒否方針、優先順位、ツール利用条件といった意味的な情報はかなり引き出され得ることが分かります。

AI エージェントやコードエージェントを運用する側は、システムプロンプトを秘密にする努力だけでなく、漏れても破綻しない権限設計、ツール制御、監査、レッドチーム評価を組み合わせる必要があります。
