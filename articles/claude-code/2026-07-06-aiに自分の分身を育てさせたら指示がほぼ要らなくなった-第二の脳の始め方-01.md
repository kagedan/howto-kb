一人でWeb開発会社を経営しています。PMとして複数の案件を並行で回しながら、自分でも設計や実装をします。

AIをコーディングに使っている人は多いと思いますが、私はここ最近、**AIに「自分の分身」を作らせる**ことに一番時間対効果を感じています。この記事では、その仕組み — 私は「第二の脳（second brain）」と呼んでいます — の作り方を、実際に使っているファイル構成とプロンプト付きで紹介します。

## AIは優秀だが、毎回記憶喪失

AIアシスタント（私はClaude Codeを使っています）は、コードは書けるし文章もうまい。でも新しいセッションを開くたびに、こちらのことを何も知らない状態から始まります。

* 「うちは検証を省かない方針で」と毎回言う
* 「顧客向けの文章はゼロ回答しない書き方で」と毎回直す
* 「それは前も却下した案なんだよな」と毎回却下する

つまり、**AIの能力ではなく、AIに渡す「自分の情報」がボトルネック**なんです。ここに気づいてから、判断基準・仕事の進め方・文章のスタイルをぜんぶファイルに書き出して、AIに読ませるようにしました。

## 仕組み: 判断基準をMarkdownのリポジトリにする

大げさなツールは使いません。**ただのMarkdownファイルをgitリポジトリに入れるだけ**です。私の構成はこうなっています。

```
second-brain/
├── README.md            # このリポジトリの読み方
├── identity/            # 人物そのもの
│   ├── core.md          # 最重要。AIへの「起動時指示」
│   ├── thinking-principles.md  # 思考の原則とその理由
│   └── work-style.md    # 仕事の進め方・時間の使い方
├── playbooks/           # 場面別の判断基準
│   ├── pm/              # 案件判断・リスク察知・レビュー観点
│   ├── engineering/     # コードレビューの観点
│   └── writing/         # 顧客向け文章の型
└── projects/            # 案件ごとの背景・経緯（変化が速い情報）
```

ポイントは3層に分けていることです。

1. **identity** — 長期間変わらない「自分の性質」。判断の原則、好み、強み弱み
2. **playbooks** — 場面ごとの「こういうときはこうする」。レビュー観点、文章の型
3. **projects** — 変化の速い案件情報。ここだけ頻繁に更新する

AIには冒頭で `identity/core.md` を必ず読ませて、あとはタスクに応じて必要なファイルだけ開かせます。全部読ませるとコンテキストの無駄なので、**README に「どのタスクのときにどのファイルを読むか」の手順を書いておく**のがコツです。

## core.md は「人物紹介」ではなく「設定ファイル」として書く

最初、core.md を自己紹介のように書いたら、AIの動きはあまり変わりませんでした。効いたのは、**AIへの指示として書き直した**ことです。

ビフォー（人物説明）:

> 私は慎重な性格で、品質を重視します。

アフター（振る舞いの指示）:

> **取り返しがつくことはスピード優先。取り返しがつかないことは品質優先。**  
> あなた（AI）の実装方法:
>
> 1. 提案には「これは取り返しがつくか」を添える
> 2. 可逆なタスクは確認を待たず進めてよい。不可逆な一歩の手前では必ず止まる
> 3. 検証・テストの省略は絶対に提案しない

この形式にしてから、AIの提案の質が目に見えて変わりました。「こう動いてほしい」を、そのまま動作仕様として書く。**人物の描写ではなく、ブートローダー（起動時に読み込まれる設定）だと思って書く**のがいちばん伝わります。

## 作り方: AIに自分をインタビューさせる

「自分の判断基準を書き出せ」と言われても、自分では意外と書けません。私はAIにインタビュアーをやらせました。実際に使ったプロンプトの骨子です。

```
あなたは私の「第二の脳」を作るためのインタビュアーです。
目的: 私の判断基準・仕事の進め方を引き出し、AIが再現できる形で文書化すること。

ルール:
- 質問は1回に1つ。抽象的な質問ではなく、具体的なエピソードを聞く
  （例:「品質とスピードで迷った最近の場面を教えてください。どう決めましたか？」）
- 私の回答から「原則」を仮説として言語化し、合っているか確認してから記録する
- 迷いや矛盾があったら、それ自体を記録する（無理にきれいにまとめない）
- 1回30分程度で区切り、毎回ファイルへの追記案を提示する
```

これを数回やると、自分でも言語化できていなかった行動原理が出てきます。私の場合は「判断の軸は結局ぜんぶ、取り返しがつくかどうかだった」という発見があって、これが core.md の中心になりました。

## 運用: 書いて終わりにしない仕組み

作って満足して腐らせたら意味がないので、運用ルールを3つだけ決めています。

1. **セッション終了時に「収穫」する** — その日AIとやり取りした中で、新しい判断・訂正・良かった文面があれば、該当ファイルへの追記案をAIに出させる。採用するかは自分が決める
2. **identityとplaybooksはAIに勝手に書き換えさせない** — 追記案の提示までがAIの仕事。反映は人間の承認制にする。ここを自動にすると、いつの間にか「AIが作った自分」になっていく
3. **鮮度をfrontmatterに書く** — 各ファイルに `updated: 日付` を入れ、古いファイルは「古い可能性がある」とAI側に明示させる

あと実務上大事なのが、**顧客名や案件の機密は `_confidential/` に分離して、AIに読ませる範囲を制御する**ことです。分身づくりと情報管理はセットです。

## 効果: 何が変わったか

運用を始めてまだ日は浅いですが、すでに体感が大きいのは次の3つです。

* **指示が短くなった** — 「例の方針でレビューして」で、自分のレビュー観点のレビューが返ってくる
* **アウトプットの手戻りが減った** — 顧客向け文章のドラフトが、最初から「自分が書きそうな文章」で出てくる
* **自分の思考の癖が見えた** — 文書化の過程そのものが、いちばんの自己分析になった

逆に、まだできていないのは「文章スタイルの完全再現」です。判断基準より文体のほうが暗黙知が多く、実例を集めて検証中です。このあたりは続きの記事で書きます。

## まず今日やるなら

1. リポジトリを作って `identity/core.md` を1ファイルだけ置く
2. AIに上のインタビュープロンプトを渡して、30分だけ話す
3. 出てきた追記案を自分で編集して確定する

最初の1ファイルができた時点で、もうAIの動きは変わり始めます。完璧な構成を目指すより、1原則だけでも「ブートローダー形式」で書いてみてください。
