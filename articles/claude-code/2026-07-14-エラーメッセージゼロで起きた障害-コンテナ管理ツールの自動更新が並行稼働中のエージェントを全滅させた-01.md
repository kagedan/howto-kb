## 何が起きたか(先に結論)

複数プロジェクトを並行運用するために Docker コンテナ上で AI エージェントを何本も走らせていたところ、ある晩、稼働中のセッションが3つとも同時に無反応になりました。原因は OS のパッケージ管理システムによる**Docker 本体のバックグラウンド自動更新**で、更新の過程で Docker デーモンが再起動し、`--rm`付きで起動していた使い捨てコンテナが全部道連れで破棄されていました。

この障害でいちばん厄介だったのは、実行に失敗していないのに失敗しているように見える、という話でも、逆に問題なく動いているように見えて実は死んでいる、という話でもありません。**エラーが一切出ないまま、外形的には「正常」と「異常」の区別がつかない状態になる**ことでした。この記事では、実際のタイムラインと気づけなかった理由、その後の対策を具体的に共有します。プロジェクト固有の情報は一般化していますが、構造と数字はそのままにしています。

## 前提構成(障害を理解するために必要な分だけ)

* 複数のプロジェクトそれぞれに、tmux セッション + Docker コンテナ1つを割り当てて並行稼働させている
* コンテナは `docker run -it --rm ...` で起動する使い捨て方式(壊れたら作り直す前提)
* tmux セッション側は `... ; exec bash` という起動になっていて、コンテナのプロセスが終了しても tmux セッション自体は生き残ってホストのシェルに戻る仕様
* Docker の実体は Linux ディストリビューションのパッケージ管理(snap)経由でインストールされている
* 定期巡回で各セッションの生死とタスクの進捗を確認する運用(数十分〜1時間おき)

この「コンテナが死んでも tmux セッションは生き残る」という仕様が、後述する検知の遅れに直結しています。

## タイムライン

詳細タイムラインを見る

| 時刻(相対) | 出来事 |
| --- | --- |
| T+0:00 | 3プロジェクトそれぞれのコンテナが正常稼働中。各セッションでタスクを実行中 |
| T+0:XX | OS のパッケージ管理システムが定期実行のバックグラウンド更新を開始。対象に Docker 本体が含まれていた |
| T+0:XX | 更新処理の一環で Docker デーモンが再起動。稼働中の全コンテナが停止 |
| T+0:XX | `--rm`指定により、停止と同時に3コンテナすべてが破棄される |
| T+0:XX〜 | 各 tmux セッションは`exec bash`によりホストのシェルへフォールバックした状態のまま放置。画面上はプロンプトが表示されているだけで、エラーも通知も一切発生しない |
| 検知まで数十分〜1時間程度 | 定期巡回で `docker ps` 相当のステータス確認コマンドを実行した際、稼働中のはずのコンテナ名がすべて一覧から消えていることで発覚 |
| 検知後 数分 | 各プロジェクトの状態管理ファイル(タスク一覧・作業ログ)を確認し、直前にどこまで進んでいたかを特定 |
| 検知後 数分〜10分程度 | 3プロジェクトそれぞれでコンテナを再起動し、状態ファイルの内容をもとに直前の作業指示を再送。作業を再開 |

体感として、**障害の発生から検知まで**が最も長く、**検知から復旧まで**は数分で終わっています。この非対称性がこの障害の本質だと考えています。

## なぜ気づくのが遅れたか

原因は3つ重なっています。

1. **エラーが一切出ない**: コンテナの異常終了は、アプリケーションのエラーとは違って、tmux の画面上に何かメッセージを出すものではありません。単にプロンプトが変わる(コンテナ内のプロンプトからホストのプロンプトに戻る)だけで、パッと見の画面は「今ちょうど何も出力していないだけ」と見分けがつきません
2. **`exec bash` によるフォールバックがノイズを隠した**: コンテナプロセスが終了した瞬間に tmux セッションごと終了する設計であれば、`tmux ls` で異常が一覧に出た可能性が高いです。しかし「セッション自体は残す」設計にしていたため、外形的な生存確認(セッションが存在するか)は「生きている」判定のままになっていました
3. **通知の仕組みがこの種の異常終了を想定していなかった**: 定期巡回の通知トリガーは「タスクが完了した」「エラーで停止した」といった、エージェント側が能動的に発するシグナルを前提に設計していました。今回のようにエージェントごとインフラごと消える障害は、エージェント自身が何かを発する前に無音化するため、既存の通知設計の対象外だったのです

裏を返すと、**「セッションが存在する」ことと「セッションが仕事をしている」ことは別の確認項目だ**ということを、この障害で初めて明確に意識しました。

## 復旧が数分で済んだ理由

これは幸運ではなく、障害以前からの設計の副産物でした。各プロジェクトで「今何を実行中か」「直前まで何をしていたか」を、会話の記憶ではなく専用のファイル(タスク一覧・作業ログ)に書き出す運用を徹底していたため、コンテナとセッションの中身が消えても、**何を再開すべきかを判断する材料自体は失われていませんでした**。

もしこの情報が会話履歴だけに存在していたら、コンテナの破棄と同時に「どこまで進んでいたか」という情報そのものが消え、復旧は「最初から状況を推測し直す」作業になっていたはずです。実際にかかったのは、コンテナを立て直し、直前の状態ファイルを渡して続きを指示するだけの数分でした。

## 対策

**実施したもの**

* 定期巡回のチェック項目に「稼働しているはずの全コンテナが実際に一覧に存在するか」を明示的に追加しました。これまでは「セッションが存在するか」しか見ていなかったのを、「コンテナ実体が存在するか」まで踏み込んで確認するように変更しています
* 併せて「各プロジェクトの状態ファイルの最終更新時刻が、想定される作業ペースに対して不自然に古くないか」も確認項目に加えました。コンテナは生きているが実際には応答していない、という今回とは別種の異常(後述)も同じチェックで拾えるようにするためです
* この2点の確認自体は、人間を待たずに監督役が定期的に自動でチェックし、異常を検知した場合はコンテナの再起動までを人間の承認なしに実行してよい範囲としました。一方で、その先の恒久対策(次項)の適用は人間の承認を経る運用のままにしています

**保留したもの**

## 教訓

* **「プロセスが生きているか」と「セッションが生きているか」は別の確認軸**です。前者を後者で代用していると、今回のような中間状態(セッションはあるが中身は死んでいる)を見逃します
* **エラーを出さない障害は、エラー駆動の通知設計ではそもそも拾えません**。定期的な生存確認(ポーリング)は、通知に頼れない障害クラスに対する保険として別枠で必要になります
* **状態を外部ファイルに整理しておく設計は、平常時よりも障害発生時にこそ効いてきます**。今回の障害は復旧手順そのものよりも、復旧に必要な情報が壊れなかったことの方が価値が大きかったです

この記事で扱ったのは、数ヶ月の運用の中で起きた障害のうち1件だけです。同じ体制で、これとは性質の異なる障害(認証情報の失効によって、コンテナは動いているのに応答だけが止まる、というより発見しにくいケース)も経験しています。1件の事故から得た教訓は再現性が低く、「たまたま」なのか「構造的な弱点」なのかの見分けがつきにくいのが実情です。複数の障害を横断して見えてきた共通パターン(検知の遅れ方の型、復旧を人間の承認なしに任せてよい範囲の線引き、保留にした対策とその理由)は、「Claude Code で作る自律マルチエージェント開発体制」の障害対応の章でまとめて扱っています。
