## Claude Codeのコンテキスト管理、雑にやってたら出力がガタ落ちした話

先週、夜中の2時にClaude Codeでちょっと大きめのリファクタをやっていたときの話。コーヒー片手に20ファイルくらい読み込ませて、テスト修正して、さあ仕上げだってところで、Claudeがさっき直したファイルをもう一度直し始めた。「それ30分前にやったよ？」って言ったらちゃんと対応してくれたんだけど、明らかに前半と比べて受け答えの精度が落ちてる。

原因はコンテキストウィンドウの圧迫だった。Claude Codeはファイル読み込み・ツール実行・会話履歴をすべてトークンとして抱えていて、これが溜まると新しい情報を正しく処理できなくなる。使用率が60%を超えたあたりから体感で質が変わり始めて、80〜90%まで行くと自動compactが走る。ただ、その自動要約の精度がいまひとつで、大事な前提条件が消えたりする。

いくつか試して、自分なりに落ち着いた運用を書いておく。

## /compact は自動任せにしない

自動compactは80〜90%で発動する。この時点だとコンテキストがいっぱいで、要約するにも材料が多すぎるのか、重要な指示や前提がごっそり抜けることがあった。

60〜70%の段階で手動 `/compact` を叩くようにしたら、セッション後半の精度がかなり変わった。余裕がある状態で要約されるぶん、必要な情報の取捨選択がうまくいくんだと思う。

```
# 自分が目安にしてる基準
# ~60%   : まだ余裕。でもそろそろ意識する
# 60-70% : 手動 /compact のタイミング
# 80-90% : 自動compactが走る（要約の精度△）
# 95%~   : 会話が怪しくなってくる
```

ステータスバーにトークン使用量が出るので、コードの区切りがいいところでちらっと確認する癖をつけた。これだけで「さっきの話忘れてない？」ってなる頻度が体感3分の1くらいに減った。

## /clear と /compact は全然別物だった

最初の頃、この2つの違いがよくわかってなくて、なんでも `/compact` で済ませていた。

`/compact` は今の会話を圧縮して続ける。認証周りの修正をしていて次にそのテストを書く、みたいな一連の流れなら `/compact` でいい。前の文脈が要約として残るから、「さっき直したところのテスト書いて」が通る。

`/clear` は会話をまるごとリセットする。フロントのCSS調整が終わって、次はバックエンドのバッチ処理、みたいに完全に別の作業に移るときはこっち。前のCSSの情報が残っていると、Claudeが「さっきの変数定義と整合性をとったほうがいいですか？」みたいな余計な気遣いを始めることがあって、それが地味に邪魔になる。

自分は1セッションの中で5回に1回くらいの頻度で `/clear` を挟んでる。1つのセッションにやることを詰め込みすぎないほうが結果的に早い。

## ワークツリーで物理的にコンテキストを分ける

Claude Codeの `--worktree`（`-w`）オプションが想像以上に便利だった。gitのworktreeを使って完全に独立したセッションが起動する。ブランチも別、コンテキストも別。

自分はメインのターミナルでfeatureブランチの実装を進めつつ、隣のターミナルで `-w` 起動してドキュメント整理やちょっとしたバグ修正を並行でやってる。コンテキストが完全に独立してるので、片方で大量にファイルを読み込んでも、もう片方には影響しない。

ただ、調子に乗って3つ同時に起動したらPCのメモリがきつくなった。16GBのマシンだと2セッション同時が現実的なライン。32GBあれば3つはいけると思うけど、4つ以上はそもそも人間側のコンテキストスイッチがしんどい。

## 雑感

コンテキスト管理って地味な話なんだけど、「Claudeが微妙な回答してくる」の原因がだいたいここにあった。最新のOpus 4.6は1Mトークンまで対応しているとはいえ、枠があるから雑に使っていいわけじゃない。「少なめに保つ」を意識するだけでセッション全体の体験がだいぶ違う。
