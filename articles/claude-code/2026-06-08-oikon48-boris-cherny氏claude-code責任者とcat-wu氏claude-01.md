Boris Cherny氏（Claude Code責任者）とCat Wu氏（Claude Code プロダクト責任者）が、Claude Codeの最初の1年を振り返る動画を公開

--

1年前、Claude Codeを一般提供（GA）開始した。社内プロジェクトとして始まったものは、いまや世界中の開発者や組織で使われている。

リアクションが2件しかつかなかったSlackでのデモから、エンジニアリングチームがコードベース全体に展開するまでの道のりを語る。

1：Claude Codeの起源と進化

最初にリリースした時はSlackのデモにリアクションが2件しか付かず、簡単なタスクはこなせる程度だった。発表から1年が経ち、今では何千ものエージェントがTree構造で動くような使い方をしている。
重要な気づきは、Claudeがミスをした時に「違うやり方をして」と指示するのではなく、CLAUDE.mdやスキルに書き込んで再発を防ぐこと。そうすればClaudeは延々と動き続けられる。また「検証（verification）」の重要性も後から分かった。

2：Claudeを検証に強くする方法

多くの人は検証をユニットテストやlint、型チェックだと誤解しているが、それらはすでに自動化済み。
エージェントにとっての検証とは「エージェント自身が実際に動かせるか」であり、その方法を見つけるのが難しい。
Opus 4の頃にClaudeが自分でCLIを開いて自作機能をテストしたのは衝撃だった。今ではiOS/Androidシミュレータやデスクトップでもループが回っている。デスクトップアプリ開発では、スキルを使ってClaudeがローカルアプリを起動しコンピュータ操作で検証・デバッグし、問題があればスキル自体を更新させている。

3：役割の融合。エンジニアを超えるClaude Code

Anthropicのチームは全員がコードを書く。Claudeがコードを書いてくれるので、重要なのは「どんなアイデアを持つか」になり、プロダクト/ビジネス文脈を持つ人ほど良いアイデアを出せる。
デザイナーがPRを出すのも今や普通。企業でもまずエンジニアが導入し、隣接職種が真似して広がる。デザイナーがプロトタイプを直接作り、PMがアプリを変更し、財務チームが予測を立て、データサイエンティストも使うなど、職種を問わず万能に使われている。

４：CI・コードレビューなどへのルーティン活用

ルーティン（routines）に注目している。あるエンジニアは、voiceモードに関する全チケットやバグ報告を自動で拾い、修正PRを出してメンションするルーティンを構築。さらに「5時間以上未対応のバグ報告を拾って修正し、検証しやすいものはマージする」ルーティンも作った。
今では「別の人のClaudeがすでに直していた」ということが頻繁に起きる。Agent SDKから始まり、ルーティンが最初の明確な応用例となった。コードレビュー対応、CI修正、リベースなどはもうやらなくなった。

5：Borisの定番機能 — autoモード

以前はplanモードを使っていたが、今は使わずautoモード一択。新しいモデル（4.6、4.7以降）はもう計画ステップを必要としない。autoモードならClaudeを起動して放置し、すぐ次のエージェントに移れる。
従来はツール実行ごとに許可プロンプトでyes/noを聞いていたが、別モデルにルーティングしてセキュリティチェックさせる方がはるかに優れている。怪しいコマンドはモデルが拒否し、後から許可も可能。1年半前には実現できなかった大きな進化。

6：autoモードのセキュリティ確保 — レッドチーミングと評価

99%をyesと答えていると人は流し読みしてしまうため、autoモードの方が安全。重要なものだけに注目できるから。セキュリティは語るのと実装するのは別物で、常にレッドチーミング・ペンテスト・脅威モデル構築が必要。
リリース前に数千件のエージェント軌跡＋許可プロンプトを集めてautoモードに安全性を分類させ、レッドチームや社内チームにプロンプトインジェクションを試させて、それらを評価（eval）化し全て防げるよう改善した。
実環境の脆弱性だけでなく、最も高度な攻撃にも対応している。新しいモデル上での開発は従来の常識を捨てる必要があり、autoモードもその一例。

7：なぜ「ループ」が次の飛躍なのか

1年半前、「エンジニアが触れるべき対象はソースコードではなくエージェントだ」という飛躍があった。
今起きているのは次の飛躍。もうエージェントとも話さず、ループやルーティンと話し、それがClaudeにプロンプトを出す。1年半で2つの大きな飛躍が起きた。
