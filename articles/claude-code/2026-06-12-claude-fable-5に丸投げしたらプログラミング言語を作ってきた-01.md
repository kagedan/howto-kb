## TL;DR

寝る前にClaude Code（Fable 5）へ「**何を作るかも含めて全部任せる。朝までに完成させて**」とだけ投げて寝た。

朝起きたら、**プログラミング言語が1つ生まれていた。**

* 言語名：**夜（Yoru）** — 「あなたが眠っている夜の間に生まれた言語だから」らしい
* Hindley-Milner型推論つきの静的型付き関数型言語
* 字句解析器 → 構文解析器 → 型推論 → バイトコードコンパイラ → VM → REPL → **ブラウザPlayground** まで全部入り
* テスト98本、全部グリーン
* 標準ライブラリは**その言語自身で書かれている**
* 実行時依存ライブラリ：**ゼロ**（TypeScriptのみ）

冗談みたいな話ですが、全部本当です。この記事では、投げたプロンプト・できたもの・途中でAIが自力で倒したバグ・触ってみた感想をまとめます。

## 「ちゃんとした言語」である証拠

「どうせハリボテでしょ？」と思った方へ。私も思いました。検証した結果がこちらです。

### 1. 型推論が本物

型注釈ゼロで、REPLが型を導出してきます。

```
夜> :type map
  : (a -> b) -> List a -> List b
夜> :type fn f x -> f (f x)
  : (a -> a) -> a -> a
```

しかも教科書的な「let多相の罠」を正しく区別します。

```
let f = fn x -> x in (f 1, f "s")   -- OK（letで多相化される）
fn f -> (f 1, f "s")                -- 型エラー（ラムダ引数は単相）
fn x -> x x                         -- 型エラー（occurs checkが無限型を検出）
```

これ、HM型推論を「雰囲気で」実装すると確実に踏み抜くポイントです。テストにもこのケースが入っていました。

### 2. 末尾呼び出し最適化が効いている

```
let rec count acc n = if n == 0 then acc else count (acc + 1) (n - 1)
count 0 1000000    -- => 1000000（スタック消費は一定）
```

末尾位置の呼び出しは `TAILCALL` 命令にコンパイルされ、フレームを再利用する設計。100万回ループのテストが入っています。逆に**末尾位置でない**深い再帰は、JSのクラッシュではなく言語側のエラー「スタックオーバーフロー（再帰が深すぎます）」として捕捉されます。芸が細かい。

### 3. 標準ライブラリがセルフホスト（の一歩手前）

`map` / `filter` / `foldl` / `sort` / `zip` / `Option` 型などの標準関数約30個は、TypeScriptではなく**Yoru自身で書かれて**いて、起動時に自分のコンパイラでコンパイルされます。

prelude（抜粋）

```
let rec foldl f acc xs = match xs with
  | [] -> acc
  | x :: rest -> foldl f (f acc x) rest

let map f xs = reverse (foldl (fn acc x -> f x :: acc) [] xs)
```

### 4. 極めつけ：その言語の上で動く電卓インタプリタ

サンプルの中に `05_calculator.yoru` というのがあって、開いて笑いました。**昨夜生まれた言語の上で、もう1つの言語処理系（電卓）が動いている**。字句解析→再帰下降構文解析→評価まで全部Yoruで書かれています。

```
1 + 2 * 3 = 7
2 * (3 + 4) * (5 - 1) = 56
```

## AIが夜中に自力で倒したバグが面白い

作業ログを読み返すと、一番面白かったのはここです。

最初のスモークテストで、こんなエラーが出ています。

```
型エラー: 無限型になります（occurs check に失敗）
1 | let rec fib n = ... fib (n - 1) + fib (n - 2)
  |                                   ^^^^^^^^^^^
```

fibの定義は正しいのに、なぜ？

原因は**型推論ではなくパーサ**でした。ML系構文は改行を無視するので、

```
let rec fib n = ... fib (n - 2)
print (fib 20)
```

の2行目 `print` が、**前の行の関数適用の続き**として貪欲にパースされていた（`fib (n-2) print (fib 20)` という適用に化けて、fibに引数を取りすぎ→無限型）。

これに対してAIは、ログ上で「個別の最小ケースでは再現しない→2行あると再現する→**パースの問題だ**」と切り分けたうえで、Elm/F#と同じ**オフサイドルール**（宣言の開始カラムより深くインデントされた行だけを式の継続とみなす）をパーサに実装して解決しています。

エラーメッセージは型エラーなのに、真因はパーサにあると見抜く。このデバッグの「跳び方」は、正直ちょっとゾッとしました。言語設計あるあるの罠を、設計判断ごと正しく潰しています。

## 感想：「丸投げ」の質が変わった

これまでのAIコーディングは「仕様を渡して実装してもらう」でした。今回やったのは「**仕様すら渡さない**」です。

* テーマ選定（「能力が最も誠実に試されるのは言語処理系」という判断）
