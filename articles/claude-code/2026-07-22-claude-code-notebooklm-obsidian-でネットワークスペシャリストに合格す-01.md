## はじめに

一人法人を経営していると、「学習」は完全に自己投資であり、同時に経営リソースの配分そのものです。時間を割いた分だけ、事業に使える時間が減る。だからこそ、資格学習のような長期戦こそ**個人の気合いではなく仕組みで回したい**、というのが今回の出発点です。

2026年11月、ネットワークスペシャリスト試験（NW）を受験します。午前I免除なし、つまり午前Iから午後IIまでフルで戦う条件です。

ただ漫然と過去問を解くだけでは、経営者としての時間の使い方として合格率に見合いません。そこで、**Claude Code をオーケストレーターに、Gemini の NotebookLM と Obsidian を組み合わせて、学習プロセスそのものを「運用できる資産」に変える**仕組みを作りました。この連載では、その構築過程と実際の運用ログを公開していきます。

この連載の狙いは3つです。

* **合格する**（当然ながら最優先の成果指標）
* 学習で得た知識を、その場限りで終わらせず**検索可能な知的資産**として蓄積する
* 進捗・正答率・弱点を可視化し、**次に何に時間を投資すべきかを毎朝判断できる状態**を作る

これは単なる「勉強法の紹介」ではなく、**一人法人の代表が自分の学習という個人プロジェクトを、事業運営と同じ感覚でマネジメントする試み**でもあります。うまくいけば、この後受験する情報処理安全確保支援士の学習にもそのまま横展開できます。

初回となる本記事は、使っているツールと全体構成、そして「なぜこの3点セットで仕組み化したのか」を整理する **環境構築編** です。

## 全体構成 — 学習を「業務」として設計する

事業を回すときと同じように、まず**役割分担**を明確にしました。属人的な気合いに頼らず、それぞれのツールに明確な責任範囲を持たせています。

| ツール | 役割 | 経営で言うなら |
| --- | --- | --- |
| **Claude Code** | 全体のオーケストレーション。各ツールを横断して進捗を集約し、毎朝ブリーフィングを出す | 秘書・マネジメントレイヤー |
| **学習エージェント（LINE Bot）** | 問題の出題・採点・弱点タグ付け。学習記録をデータベースに蓄積 | 現場のオペレーション・実行部隊 |
| **NotebookLM** | 過去問・解説サイトを取り込み、根拠付きで質問応答できる「相談できる過去問集」 | 専門顧問・ナレッジソース |
| **Obsidian** | 得た知識を概念ノートとして蓄積する恒久ナレッジベース。試験範囲を MOC で俯瞰 | 会社に残るドキュメント資産 |

**「解く」を現場エージェント、「調べる・深掘りする」を顧問（NotebookLM）、「貯める」を資産管理（Obsidian）、「経営判断につなげる」を秘書レイヤー（Claude Code）が担当する**、という構造です。事業のオペレーションをそのまま学習に持ち込んだ形になります。

## なぜこの3点セットなのか

### 1. 学習エージェント（出題・採点）— 「解く」を仕組みで習慣化する

資格学習で一番のコストは、実は知識量ではなく**継続できないこと**です。意志力に頼る運用は経営として脆弱なので、LINE で毎日問題が飛んでくる自作の学習ボットに「解く」を委任しています。回答すると自動採点され、間違えた問題には弱点タグ（例: `DNS`, `VPN`, `PKI`）が付いて記録される仕組みです。

ここで重要なのは、**正答率と弱点タグが「データ」として蓄積される**ことです。感覚で「そろそろ大丈夫だろう」と判断するのではなく、後述する Claude Code 側で「直近7日の正答率」「重点強化すべき分野」として毎朝レポートされる。経営の意思決定と同じで、勘ではなく数字で次の投資先を決める運用にしています。

### 2. NotebookLM — 「相談できる顧問」を安く雇う

NW は過去問演習が中心になりますが、解説を読んでも「なぜそう判断するのか」の勘所が腹落ちしないことがあります。そこで過去問・解説記事を NotebookLM に取り込み、**出典付きで質問できる状態**を作っています。

「この設問、なぜ OSPF ではなく BGP を選ぶのか」と聞けば、取り込んだソースを根拠に答えてくれる。ソースを限定することでハルシネーションのリスクを下げつつ、疑問が出た瞬間にすぐ相談できる。これは実務でいう「その場で聞ける専門顧問」を、学習という個人プロジェクトのために低コストで用意しているようなものです。

### 3. Obsidian — 知識を「その場で消費して終わらせない」

学習エージェントで解いて、NotebookLM で深掘りして、それで終わりだと知識は流れて消えていきます。事業でいうナレッジマネジメントの発想をそのまま持ち込み、理解した概念は Obsidian の概念ノートに書き出し、`nw` タグと双方向リンクで相互接続しています。

試験範囲全体は MOC（Map of Content）ノートで俯瞰できるようにしていて、「まだノート化していない範囲＝手薄な範囲」が一目でわかる。これは合格後も六創堂の技術資産として残り続けるという点で、単発の暗記より投資対効果が高いと考えています。

### 4. Claude Code — 全体を経営レイヤーで束ねる

上記3つはそれぞれ別のデータストアを持っていて、放っておくとバラバラのまま結局見なくなります。そこで Claude Code に横断して集約させ、毎朝こういうブリーフィングを出させています。

```
## 📚 学習進捗
- 直近7日正答率: 65%（目標: 60%以上）
- 直近30日セッション数: 12回
- NW 平均正答率（30日）: 68%
- 重点強化分野（7日）: DNS, VPN, PKI
```

さらに「NotebookLM のまとめを Obsidian に整理して」と一言指示すれば、過去問の相談内容から知識を抽出して概念ノートに合流させる、というワークフローも組んでいます。**学習を回すほど、勝手にナレッジベースが育つ**のが狙いです。毎朝この数字を見るだけで、「今日どこに時間を割くべきか」の意思決定が完了する状態を目指しています。

## 学習スケジュール — 事業計画と同じ粒度で逆算する

試験日（2026-11-14）から逆算した大枠は次の通りです。フェーズごとにKPIを置き、事業のマイルストーン管理と同じ感覚で進捗を追っています。

| フェーズ | 期間 | KPI |
| --- | --- | --- |
| 知識インプット（午前中心） | 〜7月中旬 | 7月末に NW 30日正答率 60%以上 |
| 午後I 記述対策 | 7月中旬〜8月末 | 記述の型を固める |
| 午後II・実戦演習 | 9月〜10月中旬 | 9月末に 70%以上 |
| 直前総仕上げ | 10月下旬〜前日 | 10月中旬に 75〜80% |

この連載では、各フェーズの終わりに**実際の正答率データと、その時期に効いた学習方法**を振り返っていきます。うまくいかなかったことも含めて、経営の振り返りと同じように正直に書くつもりです。

## 次回予告

次回は、**学習エージェント（出題・採点ボット）の中身と、弱点タグがどう Claude Code のブリーフィングに繋がるか**を具体的に掘り下げます。データで弱点を潰す学習ループの実装編です。

資格学習に限らず、「継続が必要な個人プロジェクトを仕組み化したい」という方の参考になれば嬉しいです。合否を含め、リアルな記録を残していきます。
