# Claude Opus 4.8は本当にnerfされたのか：Fable/Mythos停止後の品質劣化感を検証する

2026年6月中旬以降、Claude Opus 4.8の出力品質が明らかに落ちたように感じている。特に、Claude Codeでの指示解釈、日本語での語用論的な読み、ユーザの意図の復元、過剰な先回り、不要な確認要求が増えた。  
考慮すべきは、Claude Codeにおけるeffortが、2026-06-15時点ではHigh(デフォルト)→Extra→Max→Ultraだったのに対し、2026-06-18現在、High→Maxに変更された点で、明らかな推論コストの抑制の兆候だ。さらに、4.6の劣化であるとして不評だった Opus 4.7 は、Mythos公開と同時期であったことを考えると、Mythos側の変化とClaude (特にOpus)の性能の悪化が連動している可能性が高まる。  
今回は、Opus 4.8 の性能が、Mythos/Fable 5 停止により、低下している可能性を検証する。

結論：  
事実としては、Opus 4.8の劣化感は、Fable/Mythos停止後のトラフィック変化、Opus 4.8の障害頻発、effort設定・Claude Code側system promptの変更、安全化による挙動変化が重なって生じた可能性が高い。  
また、これらを考慮すると、少なくとも何らかのハイパーパラメータに修正が加えられ、性能が大きく劣化した可能性は否めない。

この記事では、確定事実と推定を分ける。

# 1. まず確定していること

## 1.1 Opus 4.8は5月28日に公開された

Anthropicは2026年5月28日にClaude Opus 4.8を公開した。公式発表では、Opus 4.8はOpus 4.7を土台に、ベンチマーク、エージェントタスク、協働性を改善したモデルとされている。価格は通常利用ではOpus 4.7と同じで、fast modeは旧Opusより安価になった。

同時に重要なのは、Opus 4.8では「effort control」が導入・強調されている点である。AnthropicのAPIドキュメントでは、Opus 4.8のeffortデフォルトはClaude APIおよびClaude Codeを含めて `high` と明記されている。 これは後述する「品質劣化感」が、モデル重みだけでなく、推論量、UI、Claude Code側の送信設定によって変わることを意味する。

## 1.2 Fable 5とMythos 5は6月9日に公開され、6月12日に停止された

Anthropicは2026年6月9日にClaude Fable 5とClaude Mythos 5を発表した。Fable 5は「Mythos-class」の一般提供モデル、Mythos 5はサイバー防御・生命科学などの限定アクセスモデルとして位置づけられた。

しかし6月12日、米国政府の輸出管理指令を受け、AnthropicはFable 5とMythos 5を全ユーザ向けに無効化した。Anthropicの説明では、指令は外国籍者によるFable 5/Mythos 5へのアクセス停止を求めるものだったが、外国籍者だけを即時に分離することが困難であるため、全顧客向けに両モデルを無効化したという。

ここで注意すべきなのは、Anthropicは同声明で「他のAnthropicモデルへのアクセスは影響を受けない」と述べている点である。 したがって、「Fable/Mythos停止によりOpus 4.8が公式に劣化した」とは読めない。影響がなかったと明示されたのはアクセス可否であり、サービング品質・混雑・レイテンシ・エラー率への間接影響までは否定されていない、という程度に留めるべきである。

## 1.3 Fable 5にはOpus 4.8へのフォールバック設計があった

Fable 5は、危険領域と判定された一部クエリをOpus 4.8へ回す設計を持っていた。AnthropicのFable 5発表では、サイバーセキュリティ、 biology/chemistry、distillation関連のリクエストが分類器に検出された場合、Fable 5ではなくOpus 4.8が応答すると説明されている。

これは重要である。Fable 5は最初から「Fableが答えない領域をOpus 4.8へ逃がす」設計だった。したがって、Fable 5公開時点でOpus 4.8には、従来のOpus 4.8利用に加えて、Fable 5からのフォールバック負荷が乗る構造があった。

ただし、ここから「Fable/Mythos停止後、Fableの全トラフィックがOpus 4.8に流れた」とまでは言えない。公式に確認できるのは、Fable 5が一部クエリをOpus 4.8へフォールバックしていたこと、Fable/Mythosが6月12日に停止されたこと、Opus 4.8が停止後の代替候補になりやすいことまでである。全量流入は推定である。

## 1.4 6月中旬にOpus 4.8の障害は実際に多発している

Claude公式ステータスには、6月13日、15日、16日、17日にOpus 4.8関連のエラーが複数記録されている。特に6月16日は、Opus 4.8単独または複数モデル横断のエラーが複数回発生しており、17日にも複数回のOpus 4.8エラーが記録されている。

これは「体感品質の悪化」が単なる錯覚ではない可能性を強める。もっとも、ステータスページが示すのは主にエラー率やサービス障害であって、同一モデルが正常応答しているときの語用論的品質低下までは直接測っていない。したがって、障害情報は「インフラ側が不安定だった」根拠であり、「モデルが賢くなくなった」直接証拠ではない。

# 2. 「nerf」という語をそのまま使うと何が危険か

「nerf」はユーザ体感を表す言葉としては便利だが、技術的には曖昧である。少なくとも次の4つが混ざる。

| 層 | 具体例 | ユーザからの見え方 |
| --- | --- | --- |
| モデル重み | RLHF、SFT、蒸留、量子化、チェックポイント差し替え | 本当に賢さが変わる |
| 推論時設定 | effort、thinking、temperature相当、推論量、ルーティング | 深く考えなくなったように見える |
| 製品層 | Claude Codeのsystem prompt、短文化指示、コンテキスト圧縮、thinking履歴処理 | 指示理解、長期文脈、道具使用が劣化する |
| インフラ層 | レート制限、混雑、エラー、タイムアウト、フォールバック | 途中停止、浅い回答、失敗、再試行増加として見える |

ユーザが「nerfされた」と感じる場合、実際には第1層のモデル重みではなく、第2〜第4層の変更である可能性が高い。

この点について、Anthropic自身が2026年4月23日に重要な投稿をしている。Claude Codeの品質劣化報告について、Anthropicは「モデルを意図的に劣化させたわけではない」としつつ、実際に品質問題が発生したことは認めた。原因は、Claude Codeのデフォルトreasoning effortを `high` から `medium` に変えたこと、thinking履歴削除のバグ、短文化system promptの副作用だった。

つまり、ユーザから見える「賢さ」は、モデルそのものだけでは決まらない。Claude Codeの場合、製品層の小さな変更でも、ユーザには広範な能力低下として見える。

# 3. Opus 4.8で起きている可能性が高いこと

## 3.1 可能性A: コアモデルのnerf

最も強い仮説は、「Opus 4.8のモデル自体がリリース後に安全化され、能力が落ちた」というものである。

しかし、現時点でこの仮説を直接支持する一次情報は弱い。AnthropicがOpus 4.8の重み変更、RLHF追加、量子化変更、ルーティング変更を公式に発表しているわけではない。また、公開ベンチマークで日次にOpus 4.8の同一タスク性能が落ちたことを示す十分な独立データもまだ少ない。

したがって、この仮説は、直ちに正しいと断定することはできない。

## 3.2 可能性B: effortまたは推論量の問題

こちらはかなり現実的である。AnthropicはOpus 4.8のeffortデフォルトを `high` としているが、ユーザが実際に使うClaude.ai、Claude Code、API、fast mode、混雑時ルーティングで、どの程度の推論量が一貫して確保されているかは外部から見えない。

4月のClaude Codeポストモーテムでは、effortを `high` から `medium` に落としただけで、ユーザは「Claude Codeが賢くなくなった」と感じたことが説明されている。 したがって、Opus 4.8でも、明示的なモデル変更がなくても、推論量やeffortの実効値が変われば同じ現象が起こりうる。

## 3.3 可能性C: Claude Code側のsystem prompt・コンテキスト処理の副作用

Claude Codeは単なるモデル呼び出しではない。system prompt、ツール権限、コンテキスト圧縮、thinking履歴、ファイル読み取り、実行計画、サブエージェントなどを含む製品である。

4月の公式ポストモーテムでは、短文化を促すsystem promptがコード品質に3%の低下をもたらしたと説明されている。 これは非常に示唆的である。モデルの能力が同じでも、「道具の前後では25語以内」「最終回答は100語以内」といった一見小さな指示が、調査深度、設計判断、説明責任、エラー検出を弱める。

最近のOpus 4.8で観察される「早く止まる」「雑に確認してくる」「勝手にタスクを狭める」「言外の制約を読まない」は、コアモデルの劣化ではなく、製品層の最適化や安全化で説明できる可能性がある。

## 3.4 可能性D: Fable/Mythos停止による負荷・代替利用の増加

Fable 5は、6月9日時点で「需要が非常に高く、予測が難しい」とAnthropic自身が書いていた。 そのモデルが6月12日に突然停止された以上、ユーザや開発者が代替としてOpus 4.8へ戻ることは自然である。

さらに、Fable 5は安全分類により一部リクエストをOpus 4.8へフォールバックしていた。 停止前からOpus 4.8はFableの安全弁として使われていた。停止後にOpus 4.8の利用が増えた可能性は高い。

ただし、これは状況証拠である。Anthropicが「Fable停止によりOpus 4.8の負荷が増え、その結果としてサービング品質が落ちた」と認めたわけではない。つまり、「因果関係は未確認だが、時系列と設計上の経路は存在する」、ということしか言えない。

# 4. ベンチマークが良いのに「賢くない」と感じる理由

Opus 4.8は公式には、コーディング、エージェント能力、知識作業、誠実性の改善が強調されている。 それでも、ユーザが「言語理解が落ちた」と感じることは矛盾ではない。

理由は、ベンチマークが主にformal reasoningを測っており、pragmatic competenceを十分に測っていないためである。

formal reasoningとは、明示された問題に対して論理的手順で正答を出す能力である。SWE-bench、GPQA、数学ベンチマークなどはこの方向に近い。

pragmatic competenceとは、発話意図、含意、前提、文脈、宣言と命令の区別、言われていない制約の復元を扱う能力である。たとえば、「このスレッドをLaTeX全般スレッドにします」は、作業命令ではなく運用上の宣言である。しかしモデルがこれを「LaTeX全般ハブを作れ」という命令として扱えば、formal reasoningは動いていても、語用論的理解は破綻している。実際、クリエイティブ系では主に Opus 4.6 が推奨されている。

この領域は近年研究が進んでいる。PUB benchmarkは、implicature、presupposition、reference、deixisを含む28kデータポイントでLLMの語用能力を評価し、人間とモデルの間にギャップが残ることを示している。 CEI benchmarkも、皮肉、遠回しな拒否、戦略的丁寧さ、受動攻撃性など、文脈依存の発話解釈がLLMにとって難しいことを扱っている。

したがって、「ベンチマークでは上がっているが、実務では馬鹿になったように見える」という現象は、かなり自然に起こる。ベンチマークが測っている能力と、ユーザが日常業務で要求している能力が違う。

| 命題 | 判定 |

| Opus 4.8は5月28日に公開された | 確定 |
| Fable 5/Mythos 5は6月12日に停止された | 確定 |
| Fable 5は一部クエリをOpus 4.8へフォールバックする設計だった | 確定 |
| 6月13〜17日にOpus 4.8のエラーが複数回発生した | 確定 |

| Opus 4.8のモデル重みがnerfされた | 未確認 |

しかし、少なくとも2026-06-15時点でのClaude Codeのeffortの設定が、High(デフォルト)→Extra→Max→Ultraだったのに対し、2026-06-18現在、High→Maxに変更された点は、明らかな推論コストの削減を図っている。これは、性能の悪化と大きく関係する。明示的に確認できる仕様をここまで変更しているのであれば、それ以外のハイパーパラメータがかなり弄られているという可能性を疑うのは必然だろう。

# 6. ひとまずの対策

Claude CodeやClaude.aiでOpus 4.8を使い続ける場合、次のように切り分ける。

1. effortの確認  
   これは Opus 4.7 で行われていた対策だが、Claude Codeでは `/effort` を確認し、重い作業では `high` 以上、可能なら `Max` を使う。Opus 4.8は公式にはhighがデフォルトだが、実際のセッションで明示確認した方がよい。
2. 同一プロンプトでA/Bテストする  
   同じ入力、同じファイル、同じ時間帯、同じeffortで、Opus 4.6、Opus 4.8、Fable復帰後のFableを比較する。主観評価だけでなく、誤読数、不要作業数、確認要求数、引用の正確性、ファイル読み取り回数を記録する。
3. 語用論的タスクをベンチ化する  
   自分の業務では「コードが通るか」だけでは足りない。宣言を命令と誤読しないか、推定を確定情報として書かないか、明示されていない前提を勝手に作らないかを評価項目にする。
4. 障害の可能性を考慮する  
   Claude はかなり頻繁に障害が発生する。公式情報のほか、[downdetector](https://downdetector.jp/en/status/claude-ai/)などのリアルタイムの障害報告サイトも参考になる。![参考：2026-06-18 (17時頃)の障害報告](https://static.zenn.studio/user-upload/39275adf7f8d-20260618.png)

# 7. 結論

Opus 4.8は、少なくとも公式発表上はOpus 4.7からの改善モデルである。誠実性、エージェント能力、ツール利用、長期作業での安定性が強調されている。しかし、モデルの性能は常に一定ではない。同じモデルでも、数日でnerfされたように感じることもある。そしてこれは、必ずしも「体感」だけではなく、他のユーザも同様に感じている可能性もあるため、適宜発信を行うことが重要だ。

* Opus 4.7は、Mythosと登場時期を同じくしていた。→ 安全性能の過剰強化
* Fable/Mythos停止により、Opus 4.8が代替先として重要になった → 万全を期して発表されたFable 5 だが、Amazonの指摘により停止を余儀なくされたために、Opus 4.8の役割が変わった。
* Fable 5はもともと分類器によりOpus 4.8へフォールバックする設計だった
* 6月13〜18日にOpus 4.8の障害が複数回発生した
* Claude Codeでは過去にeffort・thinking・system prompt変更が品質劣化として現れた → 今回の設定変更にも明らかな変化が見られる。
* ベンチマークが測るformal reasoningと、実務で必要なpragmatic competenceは一致しない

今後必要なのは、どのタスクでどのように劣化したのかを、ユーザが少しずつ報告していくことだ。  
LLMの品質劣化を論じるには、「モデルが悪くなった」ではなく、「どの層が、どの条件で、どの能力を落としたのか」を記録する必要があるためだ。
