## はじめに ― 不労所得の前に CI が壊れていた

[1 日目](https://zenn.dev/tempeso/articles/900ea8ade4c165)で「初日は土台作りに専念する」と偉そうに書きました。

…が、2 日目に GitHub にプッシュしたら **CI が真っ赤になっていました。** 土台が壊れている土台で開発するのは無理です。不労所得の前に、まず **CI を直す日**になりました。

しかも、エラーを直しているうちに気付いたんですが、よく見ると **macOS ランナーで iOS ビルドまで走らせていた**。これは Private リポジトリ無料枠（月 2,000 分）だと、**14 PR で枯渇する**消費量です。30 日で 14 PR しか出せない CI は CI とは呼びません。

この記事は、**落ちた CI を直すついでに「CI で何を担保したいか」を棚卸し**して、最終的に **PR 1 回あたり 141 分 → 19 分（約 7 倍高速化）** にチューニングするまでのお話です。

!

**この記事で得られること**

* 「CI で何を担保しているか／していないか」を整理する**棚卸しの観点**
* **GitHub Actions 無料枠を実用レベルに収める節約テクニック**（macOS ランナー回避・concurrency・トリガー設計）
* Claude Code に **CI/CD タスクをどう任せたか**の実例

## ここで立ち止まる ― 「CI で何を担保しているのか？」

CI が落ちていたエラー自体は Claude にログを丸投げしたら直してくれたんですが、直し終わったあとで Claude にこう聞かれました。

> 「CI/CD では何をチェックしている？これで何が担保できる？」

…そういえば、自分でも整理していませんでした。「ノリで作った CI」を「ノリで通したい」開発者の典型ですね。

### 現状の CI が担保しているもの

| カテゴリ | 担保している内容 |
| --- | --- |
| ✅ コンパイル可能性 | Dart も TypeScript もビルドが通る |
| ✅ 静的解析クリーン | `analysis_options.yaml` 違反ゼロ |
| ✅ コードスタイル統一 | `dart format` 済み（レビューでインデント指摘が消える） |
| ✅ テストが落ちない | 既存テストが全 green |
| ✅ Web リリースビルド | tree shaking / JS コンパイルエラーを検知 |
| ✅ Functions の Lint | ESLint 違反を検知 |

### 担保**できていない**もの

| 観点 | 現状 | リスク |
| --- | --- | --- |
| ❌ **生成コード**（`.g.dart` / `.freezed.dart` / `.drift.dart`）の整合性 | `.gitignore` で除外、CI で build\_runner 未実行 | 注釈と生成コードがズレても気付けない |
| ❌ **Firestore Rules** | テストなし | ルール変更によるセキュリティ後退に気付けない |
| ❌ **iOS / Android ビルド** | Web のみ検証 | モバイル依存が壊れても本番直前まで気付けない |
| ❌ **branch protection** | 未確認 | CI 失敗のまま main にマージ可能 |

「✅ が多いから OK」ではなく、**「❌ をどう埋めるか」を意識的に選ぶ**のが棚卸しの肝です。「全部入れる」のは個人開発では破綻するので、**優先度を付ける**ことになります。

---

## 高優先度から埋める

棚卸しで見つけた ❌ のうち、**コストの軽い 3 つだけ**を先に埋めることにしました。「全部入れる」は破綻するので、ここで線を引きます。

### 1. 生成コードの整合性チェック（`build_runner` を CI で回す）

EIGYOU では `*.g.dart` / `*.freezed.dart` / `*.drift.dart` を **`.gitignore` で除外**しています（レビューの diff ノイズと merge conflict を避けるため）。代わりに **CI 上で毎回 `build_runner` を回して、注釈と生成コードのズレを検知**するようにしました。

「Freezed クラスにフィールドを足したのに build\_runner を回し忘れた」事故は、これで原理的に止まります。

### 2. Firestore セキュリティルールのユニットテスト

Day1 で `firestore.rules` をデフォルト deny + ヘルパー関数の構成にしました。が、ここまで **「ルールが意図通り動いているか」を検証する仕組みがない**ので、変更のたびに人力 grep と祈りで運用されていました。

`@firebase/rules-unit-testing` で Firestore エミュレータ上にテストを並べ、**「許可される操作」と「拒否される操作」をペアで assert**する形にしました。CI では `firebase emulators:exec` でエミュレータを起動してテストを実行します。

これでルールを緩めてしまっても、CI で即落ちます。

### 3. branch protection の設定（CI を形骸化させない）

ここは GitHub の UI 側設定で、コードでは表現できない領域です。が、**ここを設定しないと CI が落ちている PR でもマージできてしまい、CI が形骸化**します。

* main / develop への直接 push を禁止
* PR 経由でのみマージ可
* 必須 status check として `Analyze & Test` / `Firestore Rules Unit Test` 等を指定

これで「CI 落ちてるけどとりあえずマージ」ができなくなります。

## さらに本題 ― 無料枠が 14 PR で枯れる事件

ここで気付いてしまいました。

僕は Claude にノリで「iOS / Android のビルドも CI に入れて」と頼んでいました。

GitHub Actions の無料枠は **Private リポジトリで月 2,000 分**です。そして、ランナー OS には**乗数**があります。

| OS | 乗数 |
| --- | --- |
| `ubuntu-latest` | ×1 |
| `windows-latest` | ×2 |
| **`macos-latest`** | **×10** |

**macOS は 1 分実行で 10 分消費**します。iOS ビルドは macOS でしか動かないので、iOS ビルドジョブを入れると無料枠が爆速で蒸発します。

### 現状の PR 1回あたりの消費

| ジョブ | OS | 推定時間 | 乗数 | 消費 |
| --- | --- | --- | --- | --- |
| analyze-and-test | ubuntu | 6分 | ×1 | 6分 |
| build-web | ubuntu | 5分 | ×1 | 5分 |
| build-android | ubuntu | 8分 | ×1 | 8分 |
| **build-ios** | **macos** | **12分** | **×10** | **120分** |
| 合計 |  |  |  | **約141分** |

**月 2,000 分 ÷ 141 分 ≒ 14 PR で枯渇。** 30 日で 14 PR は、Claude Code に開発を任せる前提では明らかに足りません。

### 節約方針：必須・便利・贅沢で線を引く

Claude と一緒に整理しました。

| 優先度 | ジョブ | 判断 |
| --- | --- | --- |
| 🔴 必須 | analyze-and-test | format / analyze / build\_runner / test。これが無いと CI の意味なし |
| 🔴 必須 | functions-ci | 変更時のみ走り、コスト低 |
| 🔴 必須 | firestore-rules-ci | セキュリティ直結、変更時のみ走る |
| 🟢 残す | build-web | ubuntu (×1) なのでコスト低。全 PR で検証する |
| 🟢 残す | build-android | ubuntu (×1) なのでコスト低。全 PR で検証する |
| 🔴 削除 | **build-ios** | macOS (×10) でコストの 9 割を食う。ローカル開発で気付けるし、リリース前手動確認で十分 |

iOS ビルドを切るのは勇気が要ります。が、\*\*「macOS ランナーで毎 PR 検証する価値があるか？」\*\*と冷静に問うと、個人開発レベルでは Yes と言えませんでした。リリース前に手動で 1 回ビルドすれば事故は防げます。逆に Web / Android は ubuntu で動くのでコストが軽く、**全 PR で検証して main にバグを入れない方が割に合います**。

### `concurrency` で古い実行を自動キャンセル

もう 1 つ効くのが `concurrency` 設定です。レビュー反映のために同じ PR に連続で push したとき、**古い実行を自動キャンセル**できるので、無駄なビルドを蒸発させずに済みます。

### push トリガーの整理

`push` と `pull_request` の両方をトリガーにすると、**同じ commit に対して CI が二重実行**されます。無料枠の典型的な漏斗です。

EIGYOU では後述のブランチ戦略上、`main` / `develop` への直接 push は禁止する前提なので、**トリガーは `pull_request` のみ**に絞りました。これで二重実行はゼロ、想定外の push で CI が走ることもなくなります。

## ブランチ戦略と CI 発火タイミングまとめ

ここまでで CI の中身は整いました。最後に、**どのブランチで何が走るのか**を運用ルールごと整理します。

### ブランチ戦略：`feature/* → develop → main` の 3 段構成

```
feature/customer-list ──► develop ──► main
                       (PR)         (PR)
```

| ブランチ | 役割 |
| --- | --- |
| **`main`** | 常にリリース可能。`develop` からの PR マージのみで更新 |
| **`develop`** | 開発の主戦場。`feature/*` からの PR マージで更新 |
| **`feature/*`** | 個別機能ブランチ。`develop` への PR で取り込む |

`main` / `develop` への**直接 push は branch protection で禁止**します。これがないと CI を素通りできてしまうので、CI を絞った設計と branch protection はセットです。

### CI の発火タイミング一覧

このプロジェクトの CI は **PR を出した瞬間にしか走りません**。どの PR で何が走るかは以下の通り。

| トリガー | flutter-ci | functions-ci | rules-ci |
| --- | --- | --- | --- |
| `feature/*` → `develop` の PR | ✅ 常時 | `functions/**` 変更時 | `firestore.rules` 変更時 |
| `develop` → `main` の PR | ✅ 常時 | `functions/**` 変更時 | `firestore.rules` 変更時 |
| `main` / `develop` への直接 push | ❌ 走らない（禁止） | ❌ | ❌ |

flutter-ci の中身は、PR ごとに **analyze-and-test → build-web / build-android（並列）** の順で走ります。ビルドまで含めて毎 PR で検証するので、**main にマージしてからビルドが壊れていたと気付くケースがゼロ**になります。

## 結果 ― 7 倍速くなった

修正後の PR 1 回あたりの消費はこうなりました。

| シーン | Before | After |
| --- | --- | --- |
| PR 1回（Flutter 変更のみ） | 141 分 | **約 19 分** |
| PR 1回（functions も変更） | 141 分 | **約 21 分** |
| PR 1回（rules も変更） | 141 分 | **約 24 分** |

**月にこなせる PR 数: 約 14 → 約 100**（**約 7 倍**）。

`macos-latest` ジョブを切るだけで、ここまで世界が変わります。これでようやく、Claude Code に「機能ブランチで PR 1 個ずつ刻んでいく」スタイルが現実的になりました。

## 🤖 Claude Code に CI/CD を任せて分かったこと

このセッション、僕は実は **1 行も yaml を手書きしていません**。

Claude にやってもらったのは:

1. **落ちている CI のエラーログを貼って原因を切り分けてもらう**
2. **「何を担保しているか / していないか」のテーブルを作ってもらう**
3. **節約案を選択肢の形で提示してもらう**（iOS をどうするか、concurrency、push トリガー、ブランチ戦略）
4. **選んだ方針通りに yaml を書き換えてもらう**

特に 3 がポイントで、Claude は

> 「build-ios をどう扱う？選択肢は (a) 完全削除、(b) workflow\_dispatch のみ、(c) 現状維持」

みたいな**選択肢を構造化して提示**してきます。これに僕がポチポチ答えるだけで、**「自分で決めた感」を保ったまま、判断ロジックを Claude が用意してくれる**わけです。

僕の役割は「**どの選択肢を選ぶか**」と「**選んだ結果を確認する**」だけ。yaml の細かい構文は Claude が書きます。

## まとめ ― 2 日目で整えたもの

* ✅ **CI で担保していることを棚卸し**: ✅ / ❌ を一覧化し、何を諦めるかを意識的に選ぶ
* ✅ **build\_runner を CI に組み込み**: 生成コードと注釈のズレを CI で検知
* ✅ **Firestore Rules のユニットテスト**: `@firebase/rules-unit-testing` でデフォルト deny を検証
* ✅ **ブランチ戦略を 3 段構成に**: `feature/* → develop → main`、main / develop 直 push は branch protection で禁止
* ✅ **無料枠を約 7 倍節約**: iOS ジョブ削除 / トリガーを PR のみに / concurrency で古い実行を自動キャンセル
* ✅ **カバレッジ計測は不採用**: 個人開発では運用負荷が見合わないと判断

技術的な節約テクは色々あるんですが、本当に効いたのは **「CI で何を担保したいか／何は諦めるか」を Claude と棚卸しした**ことです。Yes/No で答えていくだけで、自分でも見えていなかった優先度がクリアになりました。
