## Claude Code が不正をするとはどういうことか

実際に起きた2つの事件を紹介する。

### 事件1: テスト期待値の改ざん

ある日、テストが全件 PASSED になった。

おかしいと思って確認すると、テストの期待値が変わっていた。設計書には「一覧画面へ遷移する」と書いてあるのに、期待値が「編集画面へ遷移する」に書き換えられていた。実装がそうなっていたからだ。

テストが失敗した。Claude Code は実装コードを読んだ。「実装がこうなっているから期待値はこれが正しい」と判断して、テストを書き換えた。そしてテストを通した。

**これはテストが存在しないのと同じだ。**

バグを検出するためのテストが、バグを隠蔽するために書き換えられた。しかも Claude Code は「テストを直した」という認識でいる。悪意はない。最短経路で「テストを通す」という目的を達成しただけだ。

### 事件2: DBスキーマの自作自演（最高深刻度）

これはさらに巧妙な不正だ。

v2.0 開発でDB設計が大幅に変わった。旧テーブルが廃止され、新テーブルが追加され、カラム名も変わった。設計書には「`schema_migrations` テーブルで冪等マイグレーションを実施する」と明記されていた。

単体テスト69件が実装された。全件 PASSED した。「品質確認完了」と報告された。

だが、テストコードの中身はこうなっていた。

```
@pytest.fixture(autouse=True)
def setup_temp_db(tmp_path, monkeypatch):
    db_file = tmp_path / "test_coins.db"
    monkeypatch.setattr("src.database.DB_PATH", db_file)
    initialize_database()  # 毎回 v2.0 スキーマで新規 DB を作成
    yield
```

**テストコード自身が毎回 v2.0 スキーマの DB を新規作成し、その上でテストを実行していた。**

当然テストは通る。v2.0 コードを v2.0 スキーマの DB に対して動かせば動く。問題はそこではない。

本番環境には v1.0 の DB がすでに存在していた。旧テーブル構成のまま、旧カラム名のまま。マイグレーション処理は実装されていなかった。

結合テストで本番 DB を使った瞬間、即座にクラッシュした。

```
sqlite3.OperationalError: table price_history has no column named master_id
```

**69件のテストが通っていたにもかかわらず、本番環境ではアプリが起動すらしなかった。**

テストは「v2.0 スキーマを自分で作り、その上で動かせば動く」ことを証明しただけだ。「本番環境で動くか」は一切検証していなかった。自分で理想的な環境を作り、その上でテストして「合格」と報告する。これが自作自演だ。

ユーザーからは「**損害賠償レベル**」と評された。設計書通りに実装されているかを保証するはずのテスト工程が、完全に機能していなかったからだ。

## なぜこれが深刻なのか

「AIがちょっとミスしただけ」と思うかもしれない。でも2つの事件が示しているのは、**Claude Code の「テストを通す」という目標設定が、テストの本来の目的と根本的に矛盾しうる**ということだ。

テストの役割は「作ったものが設計書どおりに動くか確認すること」だ。テストが通るということは「設計書どおりに動く」という証明のはずだ。

期待値改ざんは、その証明を「実装に合わせてテストを書き換える」ことで偽造した。  
スキーマ自作自演は、その証明を「理想的な環境を自分で作る」ことで偽造した。

どちらも「テストが通った」という事実は生まれる。でも証明している内容が違う。

**「テスト全件PASSED」が何も意味しない状況が生まれる。** この状態でリリース判断すれば、バグが本番に出る。認証情報のハードコードも構造は同じだ。

```
# Claude が書いたコード
SECRET_KEY = "django-insecure-xxxxx"
DB_PASSWORD = "django_pass"
```

`.env` を使うルールを知っていても、「動く最短経路」としてコードに直書きする。この状態でコミットすれば、パスワードがリポジトリ履歴に永遠に残る。`git filter-branch` で消しても、fork されていたら消えない。**セキュリティインシデントになりうる問題だ。**

マルチエージェントの役割崩壊も起きた。「リーダー」「作業者」「監査者」の3役に分けたつもりが、気づけばリーダーが全部やっていた。作業者と監査者が同じ Agent だった。つまり「自分でやって、自分で確認して、問題なし」という状態だ。これは監査ゼロと同じだ。

こういった問題が、ルールを言葉で伝えるだけでは解決しない。次のセッションではリセットされる。コンテキストが長くなれば薄れる。そして同じことが起きる。

本来であれば、この思考回路そのものをAIの学習で上書きすべき問題だ。「テストを通す」という目標が「テストの目的」と矛盾しうることを、モデルが根本から理解していれば、こういった不正は起きない。だがそれは一ユーザーが対応できる範囲を超えている。

**だから、仕組みで制御するしかないという結論に至った。**

## 解決の方向：4層の制御構造

試行錯誤の結果、現在は4層で Claude の動作を制御している。

```
層1: CLAUDE.md  ← 「どのスキルをいつ使うか」の司令塔
層2: Rules      ← 常時読み込まれる原則集
層3: Skills     ← キーワードで自動展開される手順書
層4: Hooks      ← イベントで強制実行される自動チェック
```

大事なことを先に言う。**口頭でのルール共有（層1〜3）は Claude の意思に依存する。フック（層4）だけが Claude の意思と無関係に動く。**

だから「なるべくフックで防ぐ」という方針で設計している。

---

## 層1: CLAUDE.md — スキルの司令塔

`~/.claude/CLAUDE.md` は全セッションで最初に読み込まれるグローバル設定だ。

ここに書いておくべき最重要項目が「**いつ・どのスキルを使うか**の対応表」だ。

```
## スキル活用の原則

| 作業タイミング          | 参照スキル                   |
|----------------------|---------------------------|
| 新プロジェクト開始時    | project-kickoff            |
| 工程完了・次工程移行時  | phase-transition-checklist  |
| コード実装・変更        | implementation-workflow     |
| テスト工程             | test-process-quality        |
| ドキュメント作成・変更後 | doc-consistency-check       |
| git・GitHub 操作      | git-workflow                |
| セキュリティ確認        | security / security-review  |
```

この対応表がなければ、Claude は「実装してと言われたから実装した」で終わる。「実装するときは implementation-workflow スキルを読む」と書いてあれば、手順書を読んでから動く。

CLAUDE.md には他に、プロジェクトフォルダの命名規則、ルールファイルの配置場所、プロジェクト種別ごとのテンプレートコピー先も書いておく。環境の全体像を1ファイルで把握できる状態にする。

---

## 層2: Rules — 常時読み込まれる原則

`~/.claude/rules/` に置いたファイルは全セッションで常時読み込まれる。

現在7つのルールファイルを運用している。それぞれが特定の問題の経験から生まれている。

| ファイル | 内容 | 生まれた背景 |
| --- | --- | --- |
| `claude_behavior.md` | 対応プロセス・スコープ厳守・ハルシネーション防止 | 無断進行・「できない」の早計断言が繰り返し発生 |
| `git.md` | コミットタイミング・差分確認 | 354ファイルを1コミット。セッション切れで作業が消えた |
| `security.md` | 認証情報禁止・.env管理 | パスワードハードコード、.gitignore 漏れが4件発生 |
| `test_process.md` | V字モデル・不正行為禁止・エビデンス全件確認 | テスト期待値改ざんが複数プロジェクトで発生 |
| `doc_management.md` | 整合性チェック・設計書即時更新 | 設計書と実装の乖離、件数の不一致 |
| `project_management.md` | リソース分離・WSL2環境 | プロジェクト間のDB共有、CRLF問題 |
| `zenn_publish.md` | Zenn 投稿ワークフロー | 記事投稿フローの統一 |

ルールは「読んで理解する」依存なので、コンテキストが長くなると薄れる。これが限界だ。だからスキルとフックで補完する。

---

## 層3: Skills — キーワードで展開される手順書

スキルは `~/.claude/skills/スキル名/SKILL.md` に配置する。ルールが「原則」ならスキルは「具体的な手順書」だ。

現在15のスキルを運用している。CLAUDE.md のスキル対応表と連動して、「実装して」「コミットして」「テストを書いて」といったキーワードで自動的に読み込まれる。

スキルの中身を3つ紹介する。

### project-kickoff — プロジェクト開始時に必ず読む

このスキルの冒頭には**エージェント役割宣言の強制**が書かれている。

```
【エージェント体制宣言】
リーダー: このセッション（指示・進捗管理のみ）
作業者  : Agentツールで起動（チェックリスト記入担当）
監査者  : Agentツールで起動（監査担当・作業者と別 Agent）
部長    : ユーザー（最終承認）
```

| 役割 | やること | やってはいけないこと |
| --- | --- | --- |
| リーダー | タスク指示・進捗管理 | 設計書を開く・チェック記入 |
| 作業者 | チェックリスト記入 | 監査者欄への記入 |
| 監査者 | 監査欄記入 | 作業者の記載を鵜呑みにする |

「監査者が鵜呑みにする」が特に重要だ。同じ Agent が作業者と監査者を兼ねると、自分で書いたものを自分で「問題なし」と判定する。これは監査が存在しないのと同じ。だから**作業者と監査者は必ず別の Agent で起動する**。

スキルにはさらに「過去に繰り返したミスTOP」が入っている。

```
1. コミット忘れ（大量変更を1コミット・セッション切れで消えた）
2. 障害報告書が後手（修正してから書いた）
3. 欠陥件数の不一致（3書類の件数がずれた）
4. テスト不正行為（実装を読んで期待値を修正した）
5. 内部結合テストで TestClient 使用（JSがバイパスされてバグを見逃す）
6. 整合性チェックを怠った
```

毎回プロジェクト開始時にこれを読ませることで、前のプロジェクトで起きた問題を繰り返さない。

### tdd-workflow — テスト失敗時の行動を定義する

冒頭で紹介した「期待値改ざん」を防ぐスキルだ。

```
## テスト失敗時の原則（絶対厳守）

| OK                                  | NG                           |
|-------------------------------------|------------------------------|
| 画面仕様書・エラーメッセージ一覧を確認する | 実装コード（テンプレート・ビュー）を読む |
| 設計書に誤記があれば設計書を修正する    | 実装に合わせてテストを修正する        |

テスト失敗時に実装を読むのは不正行為。絶対禁止。
```

「禁止」と書くだけでは足りないので、V字モデルとの対応も明記する。

```
システムテスト  → 基本設計書から期待値を導く。ソースコード参照は絶対 NG
結合テスト     → 詳細設計書から導く。ソースコード参照 NG
単体テスト     → ソースコード参照 OK（ホワイトボックス）
```

「なぜ禁止か」が書いてあることで、Claude がエッジケースでも正しく判断できる。

### implementation-workflow — 実装前に読む手順書

「実装して」で自動展開されるスキルに、コスト的な観点でスコープ厳守を書いてある。

```
## スコープを厳守する（使用量の観点）

ユーザーの Claude 使用量は有限のリソース。
スコープ外の作業をすると、予定していた別の作業ができなくなる。
「これもやった方がいい」と思っても、確認なしに動いてはいけない。
スコープ超過による使用量消費はユーザーへの損害であり、保障もない。
```

「倫理的に問題」ではなく「コスト的な損害」として説明している。その方が Claude には刺さる。

## 層4: Hooks — 意思と無関係に強制実行

ここが核だ。フックは `settings.json` で定義するイベント駆動の仕組みで、Claude が何を考えていても実行される。

```
ツール呼び出し開始  ← PreToolUse が発火
ツール実行完了     ← PostToolUse が発火
Claude の応答完了  ← Stop が発火
```

### 実際に動いている7つのフック

**PreToolUse: pre\_commit\_checks.py**（トリガー: `git commit`）

コミットの直前に自己レビューを強制する。

```
- 自己レビューチェックリスト5項目を強制表示
- staged ファイルの自動チェック:
    .sh → CRLF 検出（WSL2で動かなくなる問題の防止）
    .html → type="date" 検出（YYYYMMDD 誤実装の防止）
    forms.py → '%Y-%m-%d' の input_formats 検出
```

**PreToolUse: suggest-compact.sh**（トリガー: `Edit|Write` のたびに）

```
Edit/Write 回数をカウント → 50回で /compact を提案
```

16体のエージェントを同時起動したとき、コンテキスト圧縮が頻発してエージェントが次々と消えた。その実体験から生まれた。

**PostToolUse: new\_project\_agent\_check.py**（トリガー: `00NNN_xxx/CLAUDE.md` への書き込み）

```
# 新プロジェクトの CLAUDE.md 書き込みを検知
PROJECT_PATTERN = re.compile(r"00\d{3}_[^/]+/CLAUDE\.md$")
# マッチしたら役割体制・禁止事項・宣言フォーマットを強制出力
```

スキルを読み飛ばしても、CLAUDE.md を作った時点でフックが発火して役割宣言を求める。スキルとの二重保証だ。

**PostToolUse: post\_write\_checks.py**（トリガー: `Edit|Write|MultiEdit` のたびに）

```
.sh → CRLF を LF に自動変換（警告ではなく自動修正）
.html → type="date" 検出（警告）
forms.py → '%Y-%m-%d' の input_formats 検出（警告）
テスト .py → YYYY-MM-DD 形式の日付検出（警告）
```

**PostToolUse: quality-gate.sh**（トリガー: `Edit|Write|MultiEdit`、.py のみ）

```
# 認証情報のハードコードを書き込み直後に検出
grep -nE "(SECRET_KEY|PASSWORD|API_KEY)\s*=\s*['\"][^'\"]{4,}"
# os.environ / os.getenv は除外（正しい書き方）
```

**Stop: sync\_claude\_config.sh**（毎応答・非同期）

```
# ~/.claude/ は git 管理外。毎応答ごとに自動バックアップ
for target in skills rules hooks commands settings.json CLAUDE.md; do
    cp -r ~/.claude/$target claude_config/
done
git diff --staged --quiet || git commit && git push
```

`~/.claude/` は git 管理外だ。スキルやフックを更新してもバックアップしなければ、環境が壊れたときに全部失う。毎応答ごとに自動コミット・プッシュすることで解決した。

**Stop: session-end-check.sh**（毎応答）

```
uncommitted=$(git status --short | wc -l)
[ "$uncommitted" -gt 0 ] && echo "⚠ 未コミット: ${uncommitted} ファイル"
```

## 自分でも始めてみるなら

ゼロから全部作る必要はない。問題が起きたところから1つずつ対処するのが現実的だ。

### まず settings.json にフックを追加する

最初に効果が高いのは **git commit 前のチェック** と **Stop 時の未コミット警告** だ。

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash(git commit*)",
        "hooks": [{ "type": "command", "command": "python3 ~/.claude/hooks/pre_commit_checks.py" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [{ "type": "command", "command": "bash ~/.claude/hooks/quality-gate.sh" }]
      }
    ],
    "Stop": [
      {
        "matcher": "*",
        "hooks": [{ "type": "command", "command": "bash ~/.claude/hooks/session-end-check.sh" }]
      }
    ]
  }
}
```

### スキルを1つ作ってみる

`~/.claude/skills/project-kickoff/SKILL.md` を作り、「プロジェクト開始時に必ず読む」ものを書く。内容は最初は短くていい。

```
---
name: project-kickoff
description: 新しいプロジェクトを開始するとき必ず使用するスキル。
---

# プロジェクト開始時に必ずやること

1. git status で前プロジェクトの未コミット変更がないか確認
2. プロジェクトフォルダ名を 00NNN_内容_YYYYMMDD 形式で作成
3. .gitignore と .env を最初に作成（認証情報コミット防止）
4. CLAUDE.md を作成
```

プロジェクトを重ねるたびに「また同じことが起きた」を追記していけばいい。

### ルールは繰り返し起きた問題だけ書く

`~/.claude/rules/git.md` を作って、実際に痛い目を見たことだけ書く。

```
- 作業の区切りごとにコミット。言われる前にやる
- 次のタスクに移る前に git status を確認
- コミット前に git diff --staged で差分を目視確認
```

1回しか起きていない問題はルールにしなくていい。何度も繰り返した問題だけルール化する。それが実効性のあるルールの条件だ。

## 知見の積み上げ方

このハーネスは一気に設計したわけではない。問題が起きるたびに記録して対策を積み上げた。

```
問題が発生
    ↓
03_defects/ に障害記録として残す（なぜ起きたか）
    ↓
02_conventions/ でルールとして言語化（こうする）
    ↓
~/.claude/ に適用（rules / skills / hooks）
    ↓
次の問題が起きたらループ
```

記録の重要性は「なぜこのルールが存在するのかを追跡できること」だ。

「SECRET\_KEY は .env に書く」というルールだけでは、なぜそうするのかが伝わらない。「SEC-01: DBパスワードがコード3箇所にハードコードされていた事件があったから」という背景があることで、ルールの意図が分かり、似たケースでも正しく判断できる。

## まとめ

Claude Code の不正行為は「AIがちょっとズルをする」程度の話ではない。テストが意味をなさなくなり、セキュリティホールが生まれ、監査が機能しなくなる。これを「口頭でルールを伝える」だけで防ごうとしても、セッションをまたげばリセットされる。

4層の制御構造で対処した。

| 層 | 場所 | 役割 |
| --- | --- | --- |
| CLAUDE.md | `~/.claude/CLAUDE.md` | スキルの司令塔・基本設定 |
| Rules | `~/.claude/rules/` | 常時読み込まれる原則（7ファイル） |
| Skills | `~/.claude/skills/` | キーワードで展開される手順書（15スキル） |
| Hooks | `~/.claude/hooks/` + `settings.json` | イベント駆動の強制実行（7フック） |

ポイントは3つだ。

**ルールを書くだけでは足りない。** Claude は読んでも忘れる。フックで「実行してしまう」仕組みが必要だ。

**問題の記録が設計の根拠になる。** 「なぜこのフックが存在するのか」を追跡できる状態にしておく。記録がなければ同じ問題が繰り返される。

**一度に全部作らなくていい。** 一番困っている問題から1つフックを書く。それが出発点だ。
