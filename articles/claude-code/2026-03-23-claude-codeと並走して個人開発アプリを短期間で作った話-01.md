# はじめに

バイブコーディングと初めて聞いたとき、ノリノリのウキウキウェーイでコーディングすることなのかと思っていた私です。  
どうやら Gemini 曰く、自然言語（普段の言葉）で AI に「こういう機能が欲しい」と伝えるだけでアプリや Web サービスの開発・修正を AI が自動で行う新しい手法のこと、なのだそうです。

これまで AI といえば主に ChatGPT ぐらいしか使っていなかったのですが、昨年下旬あたりからようやく重い腰を上げて ClaudeCode を本格的に使い始めまして。  
そして迎えた年末年始。長期休暇。ノリと雰囲気で何か作ってくれる ClaudeCode。何も起きないはずがなく…。  
ということで冬休みのテンションに身を任せてバイブコーディングでアプリを作ってみた話をしようと思います。

# アプリの概要

作ったのは植物の育成を記録するアプリのミドリトです。  
[iOS](https://apps.apple.com/jp/app/%E3%83%9F%E3%83%89%E3%83%AA%E3%83%88/id6757359432)  
[Android](https://play.google.com/store/apps/details?id=com.astrsk.midorito&hl=ja)  
![](https://static.zenn.studio/user-upload/cf8ad8a55ef5-20260304.png)

* 育成記録（写真＋メモ）
* お世話管理（水やりなど）
* 育成メモ（Tips）
* リマインダー

植物種ごとに育成メモをつけていくことで「育てた経験が次の育成に活きる」ことを大事にしています。  
育成記録を公開する機能もあり、公開された他の人の植物観察記録を眺めたり自分の育てている観察記録を公開して「みてみてうちの子こんなに大きくなったの！」と自慢することもできます。

![](https://static.zenn.studio/user-upload/152c4edecf01-20260304.png)

# 作ろうと思った理由

実は作りたいと思ったのは2024年の初夏頃のことでした。  
その辺の時期から家庭菜園だの観葉植物だのに目覚めた私は水やり記録を行うためのアプリが欲しいなぁといろいろなアプリを試していました。  
が、やたら課金圧が強かったり機能過多で良くわからなかったりシンプルだけど UI がイマイチ合わなかったりと良き出会いには恵まれず…。  
なら自分で作れば良いのでは？と思い立ったのが始まりです。

# 一人で頑張りタイム

作るぞ！と思い立ってからはまず手書きで大体どんな UI にするか、必要と思われる画面を書き出してみたり名前を決めてテンション上げてみたりしてました。  
ちなみにここで使用したのは[コンセプトアプリ](https://concepts.app/ja/)というアプリです。  
無限に広がるスケッチブックに思うがまま頭の中を書き出せるので長年愛用していますいつもありがとうございます。  
![](https://static.zenn.studio/user-upload/c2f0a35873e2-20260304.png)  
幸い DB ちょっとだけわかる民だったので殴り書きのデザインもどきを元に ER 図の作成もします。  
アプリ開発はそんなに詳しくないものの Flutter で作ろうということは既に決めていて、端末で SQLite 扱えることは知っていたのでそれでいいかーなんとかなるかーみたいな軽いノリです。大体ノリで生きてます。

ざっくり諸々決めたところで Flutter プロジェクトを立ち上げます。  
好みの関係でクリーンアーキテクチャっぽい構成にディレクトリ切って最初はモックでええじゃろととりあえず画面からちまちま作り始めます。  
そして簡素な UI が出来てきた辺りでやたら忙しくなりそこからしばらく寝かしつける事に…おやすみグンナイ…。

# 放置、からの再燃（本題）

お待たせしました。ここからが本題です。

2025年末、年末年始休暇中に Claude を使えば放置プレイの末に忘れ去られかけていたあの可哀想なアプリを救済できるのでは…？と思い立ち再びプロジェクトを起動しました。  
ですが最後に作業したのは1年以上前…。バージョンも古く荒れ果てたプロジェクトを前にいけるのか？これいけるのか？とドキドキしながら Claude にライブラリ等のバージョンアップを依頼。  
サクッとやってのける相棒。スタンディングオベーション。

スキーマ定義も「おらよ！これが ER 図じゃ！」と渡すだけでなんかいい感じに作ってくれる（作っててよかった ER 図）。  
画面や機能を追加する際もなるべく既存の構成に合わせて書いてくれるのでアーキテクチャー無視なども起きませんでした。

そこからは Claude に実装の大部分を任せる形に。  
この機能作ってーこのボタン上の方にやってーみたいな作業依頼から、こういうの作りたいんだけどこの設計でいけるかなー？などの相談も行いました。  
特に設計面では自分でも最適解が浮かんでいない場合も多いので、その手があったのかと思わず膝を打ってしまうような返答がもらえることもしばしば。

# そうだ、AI にやらせよう

そんな中どうせなら Notion に書いてた ER 図をプロジェクトの docs に置いておいたらドキュメント更新も一緒にやってくれるのでは…？と思い立ち早速移動。  
スキーマ定義に更新がある度「あ、ドキュメントも更新してね☆」と一言添えるだけで全て綺麗に整えてくれる。素敵。

認証周りなど複雑な仕様の箇所もドキュメント作っといてもらえば自分でも確認ができるし詳しい認証の仕様についてはこのファイルを読んでね、と AI に投げられるので一石二鳥です。

ドキュメント作りや更新作業ってめちゃくちゃめんどくさいけどあった方が楽なのも確かなのでその辺を全任せできるのはとてもありがたかったし、逐一仕様の確認のためにコードを捜索させる必要もなくなるのでこれは今後もぜひ使っていこうと思います。  
![](https://static.zenn.studio/user-upload/68738f40573c-20260304.png)

# 曖昧からは曖昧しか生まれない

例えば、育成記録一覧を作成して、とざっくり指示を投げれば AI は一般的に必要と思われそうな要素（ラベル、画像、育成開始日など）をよしなに並べて作ってくれます。一覧て大体こんなんだよね、みたいな思考が透けて見えますね。  
とはいえ一覧であればまぁこれでも十分です。いい感じに一覧が見れれば目標達成です。ちょろいぜバイブコーディング！この勢いでガンガン実装進めようぜ相棒！

…なんて浮かれていたのも束の間。気付けば強引な実装が各所に蔓延っていたり謎のカラムが追加されていたり拡張性が無いために追加したい機能がうまく進められなくなっていたりと、私のかわいいミドリトちゃんはやさぐれヤンキープロジェクトに成長していました。ちょっとそこに正座しようか相棒。  
結局自分自身でコードを読み解き、ここはこうだよね？AカラムがあるからBカラムは必要ないよね？なんてやり取りを繰り返してなんとか軌道修正…。正直このやりとりが一番時間使った気がします。

もしかしたらデザインがあれば多少進め方は違ってくるのかもしれませんが、悲しいことに今回頼れるのは自分の言語化能力だけです。  
自分の能力の限界を見極めた上で適切な粒度で指示を出していくことの大切さを教えてもらった気がしました。  
というか普通にざっくり曖昧な指示出されたら私だって困る。雰囲気でよろぴこてへぺろなんて言われたら殴りたくもなります。  
そう考えるとちょっぴり親近感が湧いてくる不思議。お前も苦労してるんだな Claude…。

# Claude と仲良くする方法

あわや方向性の違いで何度も解散の危機を迎えそうになりながらも試行錯誤を続け、最終的には以下のフローに落ち着きました。

* 機能と画面フローを説明する
* 実装プランを立ててもらう
* 見た目は気にせず実装させる
* そこから修正指示を出す

Claude には Plan mode があるので、まずはプランを立ててもらってから自分の想像している実装と大きく乖離していないかを確認&修正、問題なければ実装、細かな修正は後から、みたいな流れです。  
この時点で私の想像していたバイブコーディングとはちょっと違うぞと気付き始めます。  
もっとノリと雰囲気だけでいけるもんだと思ってた…。

# 並走の意味を知る

開発始めた当初はなるべくコードに関与せずにどこまでいけるかやってみよう、という気持ちだったのですが、進めていくうちに先述の通り段々と雲行きが怪しくなってきてしまいました。  
結局大まかにどういう実装をしていくか、みたいなところは全任せするよりも自分で受け持った方が手戻り少ないし結果として良いコードができるのかもしれません。  
コーディング速度は言うまでもなく速いので多少の修正が生じても大まかな方向性さえ間違っていなければどうにでもなります。爆速で。

今後どこまで AI が進化するのかは分かりませんが、現段階では「自分でもできることを代わりに爆速でこなし、尚且つ相談にも乗ってくれる良き相棒」というのがしっくりきています。  
人によるとは思いますが、これが私に合った並走の仕方なのかもしれません。

# 驚きの速さ

そんなこんなでとりあえずオフライン完結で一通り機能を揃えたところで大体3〜4日ぐらいです。  
土台があったとはいえ1年以上も放置されていたとは思えないほど急速な発展を遂げることとなりました。  
ここらで一旦リリースもありかなぁと思いつつも、あまりにも早く実装が終わってしまったので不完全燃焼です。  
**こんなんじゃ終われねぇよな Claude！！ まだまだいけるよな！！？**  
ということでまたもや謎のテンションに身を任せて育成記録の公開機能を作るためバックエンドを作り始めることになるのですが、その辺はまた機会と気力があればお話しできるといいなと思っています。

# 諸々終わってから知ったこと

今回の開発では毎回その場で、

* プランを出してもらう
* 設計案を比較する
* ドキュメントを更新してもらう

みたいなやりとりをしていました。

恐らく読んでいる方の中には疑問に思っている方もいるかもしれません。  
**それ、CLAUDE.md に書いとけばよかったのでは？**

実装前のプラン提示、コンポーネント分割、/docs の更新などなど、予めルール宣言しておけば指示が楽になったし、もしかしたらもう少し開発体験も変わったのかもしれません。  
とはいえ開発開始当初はそんなものの存在すら知らなかったので、これは今後に活かしていこうと思います。  
ClaudeCode、まだまだ奥が深いぜ…！
