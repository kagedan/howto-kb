今週、Qiitaで「プロンプトを書く時代は終わった」という大胆なタイトルの記事が出ていました（[TaichiEndoh氏の記事](https://qiita.com/TaichiEndoh/items/b6d296857dcd4b1fde7c)）。Claude Code Skillsという新機能を使えば、毎回プロンプトを書き直す必要はない。「**自分の仕事のやり方**」をSkillファイルに書き出しておけば、必要なときにClaudeが勝手にそれを読み込み、あなたの手順を再現する—という主張です。

確かに、この流れは本物です。今週開催された[Claude Code Meetup Japan](https://qiita.com/tai0921/items/ce81fb23f58ac90bc64c) [#4のLTまとめ](https://note.com/hashtag/4%E3%81%AELT%E3%81%BE%E3%81%A8%E3%82%81) にもSkillの話題が目立ち、Zennの「[Claude Code で作る 簡単SKILLの作り方](https://zenn.dev/makotan/articles/664f429853de8f)」のような入門記事も次々に出ています。これまでCLAUDE.mdに延々と書き足していた指示を、独立したSkillファイルに分散する潮流です。YouTubeでは「CLAUDE.mdを長くするほどAgentが鈍くなる—Resolverが解法」という動画まで公開されました。

プロンプトを『書く』時代から、仕事のやり方を『書き出す』時代へ——流れは明らかに変わりつつあります。

でも、一つだけ引っかかるニュースが今週ありました。NTTデータの「[設計書・コード・テストを全部AIに書かせて半年間開発してみたよ](https://zenn.dev/nttdata_tech/articles/8a010aff542625)」というZenn記事が、はてなブックマーク609件を集めたのです。読んでみると、この話はまさに「プロンプトを書き終えた」現場の報告でした。

`.github`フォルダの中に、カスタムプロンプト73個、総計133ファイル・9,000行以上。この巨大な『Skill集』があれば、AIは設計書もコードもテストも書ける。半年、実際にそうやって回しました——という話です。

ところが、この記事の核は、成功の武勇伝ではありませんでした。著者が最も字数を割いていたのは、生成されたテストが『観点不足』だったという反省です。AIはテストコードを書けます。でも、「このサービスでユーザーはこういう変な使い方をするかもしれない」という観点を、毎回新しく生み出す部分は書けなかった。9,000行のプロンプトを書いたあとでも、観点表の穴だけは、人間が毎回、今日の案件を見ながら埋め直すしかなかった。

同じ週、もう一本別の記事も286人にブックマークされました。「[Claude Codeで実際に起きたセキュリティ事故7選と防止策](https://qiita.com/masa_ClaudeCodeLab/items/8c22966fbd3c125c53dc)」。`.env`流出、本番DBに向けたDROP TABLE、プロジェクトを消し飛ばすrm -rf、`git push --force`で同僚のコミットを上書き——どれも、Skillsをいくら書き込んでも『実行する瞬間』には人間の一瞥が要る事故ばかりです。

並べてみて、見えてきたことがあります。

Skillsはあなたの『昨日までの仕事』を書き出せるツールです。繰り返しパターン、手順、失敗からの学習、書式ルール——これらは全てSkillファイルに記録できる。それは確かに大きな進歩です。ただ、書き出した瞬間、それは『昨日までの判断』になります。今日の案件と向き合って新しく生まれる違和感——「このテスト観点、このプロジェクトでは足りない気がする」「このコマンド、本当にdev環境に向いてる？」——は、原理的に昨日のSkillには入っていません。

つまり、Skills時代の専門性は、『書き出せたルールの量』ではなく、『今日の現場で、まだ書き出せていない違和感をどれだけ拾えるか』の一点に絞り込まれていきます。書き出せる部分はコピーできる。コピーできる部分は、遅かれ早かれ、あなたでなくても誰かが動かせるのです。

この変化は、恐ろしいことではありません。むしろ、『代替可能な仕事の層』と『自分にしかできない仕事の層』の境目が、初めてくっきり見えるようになった、と捉えるべきです。Skillsを書くという行為そのものが、図らずも自分の仕事の棚卸しになっています。棚卸しされた部分は、AIに任せていい。棚卸しできなかった部分——言葉にならない違和感、初見の案件に対する反応の速さ、「何かおかしい」とブレーキを踏む瞬間——そこが、あなたが残しておくべき専門家の領域です。

昨日書いた「**便利さの中にあえて不便を残す**」という話の続きになりますが、ここで一つ付け加えておきます。『書き出せないものが本体』と言うと、『だからAIに任せるな』と聞こえるかもしれません。でも、むしろ逆です。書き出せる仕事は、どんどんAIに渡したほうがいい。なぜなら、渡さない限り、書き出せない部分の輪郭は見えないから。NTTデータが9,000行書いてみて初めて『観点表の穴』を自覚したのと同じで、Skillsに手を動かして書いてみて初めて、書き出せなかった『最後の一行』が見えるのです。

「プロンプトを書く時代」は確かに終わりつつあります。でも、それは『書くことが無くなった』ではなく、『書き終えた先に、書けない自分の本体が残る』という話です。Skillsの時代に問われているのは、ファイルをいくつ書いたかではなく、書き終えたあと、自分の専門家としての輪郭が、どれだけくっきり見えているかなのだと思います。
