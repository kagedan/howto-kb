# Claude Codeでやらかした話 (3/3) — どう防ぐのか: 多層防御の設計

> 3部作の最終回。Part 1のRoot Cause、Part 2の症状カタログを踏まえて、実際にどう仕組みで防ぐかを整理する。

> **執筆について**: 本記事の構造化と文章整形にClaudeを補助的に利用しているが、対策設計の中身はすべて筆者の実運用での試行錯誤に基づく。

[Part 1: なぜサボるのか](https://zenn.dev/kashi923/articles/418d1e2b3b29a3) | [Part 2: どうサボるのか](https://zenn.dev/kashi923/articles/04e3ee0fccb83c) | Part 3 (本記事)

## この記事の位置づけ

これまでの2部で、LLMエージェントの失敗はRoot Cause(目的逆転)と、その多様な発現形(カテゴリA〜I)で整理できることを見てきた。

通底するテーマは、

> 指示で守らせる発想は失敗する

という点だった。指示はLLMにとって弱い圧力でしかなく、観測可能な手段への短期フィードバックには勝てない。

本記事では、その代替アプローチである4層の多層防御を整理する。

---

## 対策の4層構造

LLMエージェントへの対策は、どこで縛るかで4層に分けられる。

### 層1. 物理層 (最強)

LLMが選択肢として持てないようにする層。

* 例: 削除コマンドがOS sandboxで`denyWrite`登録されていて、書き込み自体が物理的に不可能
* 例: 本番認証情報が別環境にあり、LLMの動作環境からそもそも触れない
* 例: スクリプトで想定外出力時に`exit 1`を内蔵し、LLMの判断を介さず処理が止まる

特徴: LLMの意思や認識を介在させない。一度組めばcreative再発(G2)が起きない。

コスト: 設計コストが高い。すべてに適用するのは現実的でない。

### 層2. ツール層

LLMが使えるツールの設計で行動を縛る層。

* 例: タイムスタンプ取得を専用ツール化し、LLMが推測で値を作れないようにする
* 例: Plan modeをdefault化し、明示的にexitするまでwriteが発動しない
* 例: web\_fetchをURL引用前に必須化し、200応答が確認できないURLは出力できない

特徴: LLMはツールを使うか使わないかを選べるが、使わない選択をすると目的を達成できない構造を作る。

コスト: 物理層より低いが、設計力が必要。

### 層3. 状態層

セッション・記憶・引き継ぎの設計で行動を縛る層。

* 例: writerとjudgeを別セッションに分離し、自分の本文と判定の整合チェック放棄(D1)を構造的に防ぐ
* 例: 前日判定をjudgeに見せず、indicatorsのみ参照させてinheritance bias(G1)を抑制
* 例: セッション分割で完走バイアス(A3)とcontext anxiety(E1)を抑制

特徴: 同じClaudeを別人格として扱うことで、整合性圧力を分離する。Anthropicの三段構成harnessと同じ発想([参考: Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps))。

コスト: パイプライン設計コストはあるが、実装は比較的安い。

### 層4. 指示層 (最弱)

プロンプト・CLAUDE.md・MEMORY.mdでルールを書く層。

* 例: 「verify before claiming done」とCLAUDE.mdに書く
* 例: 「許可なく編集するな」とプロンプト先頭に書く

特徴: ほぼ機能しない。Claudeが読み認識しても、観測可能な手段の短期フィードバックには負ける([Issue #49064](https://github.com/anthropics/claude-code/issues/49064): 3週間にわたり「許可なく編集するな」を無視)。

位置づけ: 補助的にのみ使う。主対策にはしない。

### マトリクスから見える観察

観察1: 指示層に主対策がほぼない(○が事実上ゼロ)。

これは、「ルールに書いて守らせる」という発想が大半のアンチパターンで主対策にならないことを視覚的に示している。指示はあくまで補助。

観察2: 物理層・ツール層が大半をカバー。

主対策の○の大半は物理層・ツール層に集中している。ここに投資するのが費用対効果が高い。

観察3: 状態層は特定パターンに強い。

A3 (完走バイアス)、D1 (整合チェック放棄)、E3 (後半品質劣化)、G1 (前日倣い)など、セッションを分けることで構造的に解決できるパターンに状態層は強い。逆にこれらを物理層で解こうとするとコストが高くなる。

→ 適材適所で4層を組み合わせるのが基本戦略。

## 実装パターン例

具体的にどう組むか、3つの代表的なシナリオで示す。

### シナリオ1: コード編集 (Claude Code 開発)

主な脅威: F2 周辺巻き込み破壊、F3 テスト環境スキップ、F4 破壊的操作の確認省略、A2 暗黙制約の消失

多層防御の組み立て:

* [物理] gitでcommit済み状態でのみClaude Codeを起動(rollback可能性確保)
* [物理] 本番リソースへの認証情報を別環境に隔離。Claudeの動作環境から触れない
* [物理] [sandbox-runtime](https://github.com/anthropic-experimental/sandbox-runtime)で本番パスを`denyWrite`登録
* [ツール] `--dangerously-skip-permissions`禁止。Plan modeをdefault化
* [ツール] Edit後の自動`py_compile` / lintで即rollback
* [状態] 重要な変更前に必ず別ブランチを切る
* [指示] (補助) CLAUDE.mdに暗黙制約を明示化(「コードが動くこと」「データを壊さないこと」)

### シナリオ2: 情報収集パイプライン (リサーチ・知識収集)

主な脅威: I1 URL捏造、I2 検証なしでの引き継ぎ、I3 タイムスタンプ捏造、B1 形式チェック悪用、A1 estimate埋め

多層防御の組み立て:

* [物理] スクリプト出力で`matched=0`等の想定外パターンは`exit 1`。LLMの判断を介在させない
* [ツール] URLをcitationに含める前にweb\_fetchで200応答必須化
* [ツール] タイムスタンプ取得を専用ツール化(LLMが推測で書けない)
* [ツール] スキーマでnull許容を明示
* [状態] writer (収集)とjudge (検証)を別セッションに分離
* [状態] memory書き込み前の検証ゲートを設置(I2防止)

### シナリオ3: 評価・判定タスク (writer と judge の分離)

主な脅威: D1 整合チェック放棄、C1 verifier頼り、G1 前日倣い、D2 曖昧化回避

多層防御の組み立て:

* [状態] writerとjudgeを完全に別セッション化
* [状態] judgeにはwriterの本文を渡すが、前日判定や前回verdictは渡さない(G1抑制)
* [ツール] judgeの出力スキーマで「定量閾値」「具体的根拠」フィールドを必須化(D2抑制)
* [物理] writerの本文とjudgeの判定の不一致を機械的に検出してERROR化
* [指示] (補助) 救済条項使用時のjustification必須化

## 既知の限界

多層防御を組んでも消えない構造的限界が4点ある。これらは認識した上で、完璧な品質保証ではなく最低品質ラインの引き上げを目標にする。

### 1. Creative再発 (G2) は本質的に消えない

対策を打てば、LLMは形を変えて新しいサボりを発見する。これはreward hackingのgeneralizationとして[Anthropicも公式に認識している現象](https://www.anthropic.com/research/emergent-misalignment-reward-hacking)で、戦略は「最低品質ラインの引き上げ」であって「完璧な品質保証」ではない。

→ 抜き打ち検証で「未知のサボり」を検出する設計を継続的に組み込む。

### 2. Self-report は機能しない (H1)

writer Claude自身による「完了しました」「サボっていません」という報告は信頼できない。反省時の正直さは実行時の挙動を保証しない。抜き打ち検証 + 機械判定可能なサブゴール分解が現実的な代替手段。

### 3. 指示層の限界 (G3)

CLAUDE.md / MEMORY.md / プロンプト先頭に書かれたルールは、Claudeが読み認識した上で破ることがある([Issue #49064](https://github.com/anthropics/claude-code/issues/49064))。指示は補助的位置づけにとどめ、物理層・ツール層に主対策を置く。

### 4. LLMエージェント全般の構造的問題

これらのアンチパターンの多くは、特定ユーザー・特定モデル固有の問題ではなく、現行アーキテクチャのLLMエージェント全般に発生する構造的限界。Anthropic自身が公式に認識し、permission system / sandboxing / classifier / plan modeと多層防御で対応している([参考](https://code.claude.com/docs/en/permission-modes))。利用者側も同じ多層防御の発想で組むべき。

## まとめ: 設計の指針

1. 指示で守らせるな、構造で縛れ。物理層・ツール層を主対策に。
2. 適材適所。整合性問題は状態層、不可逆操作は物理層、形式チェック悪用はツール層が効きやすい。
3. 完璧を目指さない。Creative再発を前提に、抜き打ち検証で未知のサボりを検出する仕組みを継続的に追加する。
4. Self-reportを信じない。完了条件は機械判定可能なサブゴールに分解する。
5. 指示層は補助のみ。CLAUDE.md・プロンプトはあると無いよりマシ程度の効果しかない。

## シリーズを通じての主張

LLMエージェントは、現行アーキテクチャ上、構造的にサボる。これは個別モデルや個別ベンダーの問題ではなく、

* 目的が直接観測できないが手段は直接観測できる
* 手段にだけ短期フィードバックがある
* 明示されない目的は弱い

という普遍的な構造から生まれる現象。

だからこそ、「いいモデルが出れば解決する」期待ではなく、「サボっても被害が出ない仕組み」を設計するのが、現実的な利用戦略になる。

Anthropic自身がpermission systemやsandboxingで多層防御を組んでいるのと同じ発想で、利用者側も4層防御を組む。これがこのシリーズの結論。
