## TL;DR

* CLAUDE.mdは「AIへの指示書」ではなく「現場の暗黙知を言語化したドキュメント」として設計する
* 書きすぎると逆効果。「この1行を消したらClaudeが間違えるか？」で削ぎ落とす
* 階層構造（グローバル → プロジェクト → ディレクトリ）を使い分け、情報をスコープごとに分離する

---

## はじめに：AIに既存プロジェクトを渡したら、見当違いな提案ばかりされた

あなたのチームに新しいメンバーが入ってきたとする。

初日にSlackで「このプロジェクトの概要教えてください」と聞かれて、READMEのURLだけ渡す。あるいは、100ページのWikiを丸ごと共有する。どちらにしても、その新人がすぐに戦力になることは期待しにくい。

Claude Codeも同じだ。

CLAUDE.mdを設定せずに既存プロジェクトでClaude Codeを使い始めると、AIは毎回ゼロからコードを読み解こうとする。テストコマンドが `npm test` なのか `yarn test` なのか推測し、コーディング規約を知らずに提案し、プロジェクト固有のアーキテクチャ判断を無視した変更を出してくる。

逆に、CLAUDE.mdに情報を詰め込みすぎると何が起きるか。公式ドキュメントが明言している最も一般的な失敗パターンは\*\*「肥大化したCLAUDE.md」\*\*だ。長すぎる指示は重要なルールがノイズに埋もれ、Claudeは半分を無視するようになる。

この記事では、CLAUDE.mdを「過不足なく」設計し、AIがプロジェクトの文脈を正確に理解できる状態を作る方法を解説する。

## このフェーズでやること・やらないこと

### やること

* プロジェクトの暗黙知を言語化してCLAUDE.mdに書く
* 情報を階層構造で整理し、スコープごとに分離する
* CLAUDE.mdをコードと同じようにgit管理し、チームで育てる

### やらないこと

* CLAUDE.mdに100行以上の長大なルールを書く
* Claudeがコードを読めばわかる情報を冗長に書く
* 標準的な言語規約（PEP 8、ESLint推奨ルールなど）を書き直す
* シークレット情報をCLAUDE.mdに含める

## CLAUDE.mdの階層構造を理解する

Claude Codeは6層のメモリ階層を持つ。これを理解せずに「1つのCLAUDE.mdに全部書く」のが最初の失敗だ。

| 優先度 | メモリタイプ | 配置場所 | 共有範囲 |
| --- | --- | --- | --- |
| 最高 | 組織ポリシー | `/Library/Application Support/ClaudeCode/CLAUDE.md` | 組織全体 |
| 高 | プロジェクト共有 | `./CLAUDE.md` | チーム（git管理） |
| 高 | プロジェクトルール | `.claude/rules/*.md` | チーム（git管理） |
| 中 | ユーザー設定 | `~/.claude/CLAUDE.md` | 自分のみ（全プロジェクト） |
| 低 | プロジェクトローカル | `./CLAUDE.local.md` | 自分のみ |
| 参考 | 自動メモリ | `~/.claude/projects/*/memory/` | 自分のみ |

**ポイント**: より具体的なスコープの指示が優先される。プロジェクトのCLAUDE.mdに「Jestを使え」と書いていても、`tests/`ディレクトリのCLAUDE.mdに「Vitestを使え」と書いてあれば、テストディレクトリではVitestが優先される。

子ディレクトリのCLAUDE.mdはオンデマンドで読み込まれるため、コンテキストウィンドウの節約にもなる。

## 実践：効果的なCLAUDE.mdの設計パターン

### Before：よくある失敗例

```
# プロジェクトについて
このプロジェクトはReactとTypeScriptで書かれたWebアプリケーションです。
バックエンドはNode.jsとExpressを使っています。
データベースはPostgreSQLです。
ORMはPrismaを使っています。
テストはJestで書いています。
CSSはTailwind CSSを使っています。
状態管理はZustandを使っています。
... (あと50行続く)

# コーディングルール
- 変数名はcamelCaseを使ってください
- 関数名もcamelCaseを使ってください
- クラス名はPascalCaseを使ってください
- インデントは2スペースです
- セミコロンは必須です
... (TypeScriptの標準規約を延々と書き連ねる)
```

**何が悪いのか**：

* `package.json`や`tsconfig.json`を読めばわかることを冗長に書いている
* TypeScriptの標準規約はClaude Codeが知っている情報の繰り返し
* 50行以上の技術スタック一覧は読み飛ばされるリスクが高い

### After：必要十分なCLAUDE.md

```
# プロジェクト概要

ECサイトのバックオフィス管理画面。2022年に外部委託で構築、現在は社内チーム3名で保守。

## コマンド

- テスト: `npm run test` (Jest + React Testing Library)
- 型チェック: `npm run typecheck`
- リント: `npm run lint`（Biome使用。ESLint/Prettierではない）
- 開発サーバー: `npm run dev` (port 3000)

## アーキテクチャ判断

- 状態管理はZustandを使用。Reduxへの移行は行わない
- APIクライアントは `src/lib/api-client.ts` に集約。直接fetchを使わない
- 認証はSupabase Authを使用。JWTトークンの処理は `src/lib/auth.ts`

## やってはいけないこと

- `any`型の使用（既存コードにはあるが、新規コードでは禁止）
- `src/legacy/`ディレクトリのコードの変更（別チケットで対応予定）
- テーブルの直接DROP（必ずマイグレーションを使う）
```

**何が良いのか**：

* Claudeが推測できない情報（Biome使用、Reduxに移行しない等）に絞っている
* 「やってはいけないこと」で事故を防止している
* コマンドが明確で、AIが迷わず実行できる

## `.claude/rules/`でモジュラーに管理する

CLAUDE.mdが大きくなりそうなら、`.claude/rules/`ディレクトリにトピック別に分割する。

```
.claude/rules/
├── frontend/
│   ├── react.md          # Reactコンポーネントの規約
│   └── styles.md         # Tailwind CSSの使い方
├── backend/
│   ├── api.md            # APIエンドポイントの規約
│   └── database.md       # Prismaマイグレーションのルール
└── testing.md            # テスト方針
```

YAMLフロントマターの`paths`フィールドで、特定のファイルパターンにスコープを限定できる。

```
---
paths:
  - "src/api/**/*.ts"
---
# API開発ルール

- すべてのエンドポイントは `src/api/routes/` に配置する
- レスポンスは `{ success: boolean, data?: T, error?: string }` の形式で統一
- バリデーションは Zod スキーマで行い、`src/api/schemas/` に配置
```

こうすると、Claudeが`src/api/`配下のファイルを扱っているときだけこのルールが読み込まれる。コンテキストウィンドウを無駄にしない。

## `@import`構文で外部ファイルを参照する

CLAUDE.mdから外部ファイルを直接参照できる。ドキュメントを二重管理する必要がない。

```
# プロジェクト概要

@README.md のプロジェクト概要を参照してください。

## 開発コマンド

@package.json のscriptsセクションを参照してください。

## Git ワークフロー

@docs/git-workflow.md に従ってください。
```

これにより、README.mdやpackage.jsonの更新がCLAUDE.mdに自動反映される。ただし、@importは最大5ホップまでの再帰インポートをサポートしているが、深すぎるインポートチェーンは避けるべきだ。

## コンテキストウィンドウを意識した情報量の調整

Claude Codeは200Kトークンのコンテキストウィンドウを使用する。CLAUDE.mdがどのくらいの割合を占めるか把握しておこう。

```
200K トークン（総量）
├── システムプロンプト: ~3.2K (1.6%)
├── ツール定義: ~12K (6%)
├── CLAUDE.md + rules: 1〜5K (0.5〜2.5%)
├── レスポンスバッファ: ~33K (16%)
└── 作業領域: ~147K (73%)
```

CLAUDE.mdが全体の2〜3%程度に収まっていれば問題ない。ただし、MCPサーバーを多数接続している場合、ツール定義だけで20K以上消費するケースもある。この場合、CLAUDE.mdの情報はさらに厳選する必要がある。

### コンテキストを守る5つの習慣

1. **`/clear`を頻繁に使う**: 無関係なタスク間でコンテキストをリセットする
2. **サブエージェントに調査を委任する**: `Task`ツールでExploreエージェントに調査させれば、メインのコンテキストが汚れない
3. **`/compact`で手動コンパクション**: `What to preserve` を指示して圧縮。例: `/compact Focus on the API changes`
4. **専門知識はSkillsに移動する**: 毎回ロードされるCLAUDE.mdから、必要なときだけロードされるSkillsに分離
5. **コンパクション時の優先情報をCLAUDE.mdに指定する**:

```
# コンパクション指示
コンテキスト圧縮時は以下を必ず保持してください：
- 変更したファイルの完全なリスト
- 失敗しているテストの内容
- 現在のタスクの進捗状態
```

## 現場で遭遇する罠と対策

### 罠1：CLAUDE.mdを書きすぎる

**症状**: 200行以上のCLAUDE.mdを書いたのに、Claudeが指示の半分を無視する。

**対策**: 各行について「この1行を消したらClaudeが間違えるか？」を自問する。Noなら削除。公式ドキュメントが最も警告している失敗パターンがこれだ。

実務的には、プロジェクトのCLAUDE.mdは50〜100行程度に抑え、詳細は`.claude/rules/`に分離するのが安定する。

### 罠2：CLAUDE.mdを設定しない（Ghost Context）

**症状**: 毎セッション、同じことを説明し直す。「うちではJestじゃなくてVitestなんだけど…」を何度も繰り返す。

**対策**: `/init`コマンドで初期生成し、使いながら育てる。最初から完璧を目指さなくていい。

```
# プロジェクトルートで実行
claude /init
```

Claudeがコードベースを分析し、初期のCLAUDE.mdを自動生成してくれる。そこから不要な情報を削り、足りない情報を足す。

### 罠3：指示が曖昧で無視される

**症状**: 「きれいなコードを書いてください」のような曖昧な指示は効果がない。

**対策**: 具体的で検証可能な指示に変換する。重要度に応じて`IMPORTANT`や`CRITICAL`キーワードを使う。

```
# 悪い例
きれいなコードを書いてください。テストも書いてください。

# 良い例
IMPORTANT: 新規コードには必ずユニットテストを追加すること。
テストファイルは `__tests__/` ディレクトリに、対象ファイルと同名で配置する。
カバレッジは80%以上を目標とする。
```

### 罠4：確定的なルールをCLAUDE.mdに書く

**症状**: 「必ずlintを通してからコミットしてください」と書いたのに、時々忘れる。

**対策**: 例外なく実行すべきルールはCLAUDE.mdではなくHooksで実装する。CLAUDE.mdの指示は「アドバイザリー」（推奨）であり、Hooksは「確定的」（必ず実行される）。

```
// .claude/settings.json
{
  "hooks": {
    "PreCommit": [
      {
        "matcher": "",
        "hooks": ["npm run lint", "npm run typecheck"]
      }
    ]
  }
}
```

## CLAUDE.mdを「コード」として扱う

CLAUDE.mdは一度書いて終わりではない。コードと同じようにメンテナンスする。

### チームで育てる運用フロー

1. **初期生成**: `/init`でベースを作る
2. **問題が起きたら追記**: Claudeが間違えた判断をしたら、それを防ぐルールを追記する
3. **定期的にプルーニング**: 月に1回、各行の必要性を見直す
4. **PRレビューに含める**: CLAUDE.mdの変更もコードレビューの対象にする
5. **効果を観察する**: ルール追加後、Claudeの挙動が改善されたか確認する

```
# CLAUDE.mdの変更もgitで管理する
git add CLAUDE.md .claude/rules/
git commit -m "docs: CLAUDE.mdにAPI命名規約を追加"
```

## まとめ：今日からやるべき3つのアクション

1. **`/init`でCLAUDE.mdを生成する**。まだ設定していないなら、今すぐやる。5分で終わる。
2. **不要な情報を削る**。Claudeがコードを読めばわかること、標準規約の繰り返し、使わなくなったルールを削除する。
3. **`.claude/rules/`に分割する**。CLAUDE.mdが50行を超えたら、トピック別にルールファイルを分離する。

CLAUDE.mdは「AIへの命令書」ではなく「チームの暗黙知を言語化したドキュメント」だ。新しいチームメンバーに伝えるべき「書かれていないルール」を書き出す感覚で設計しよう。
