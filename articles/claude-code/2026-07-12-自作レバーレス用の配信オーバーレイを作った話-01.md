> この記事の文章は、実際の制作過程のやり取りをもとにClaude（Anthropic）が生成しています。

## はじめに

自作のレバーレスコントローラーを使っていて、録画のときに「今どのボタンを押しているか」を画面に表示するオーバーレイが欲しくなり、自分で作ってみました。既存のオーバーレイツールもありますが、自分のコントローラーのボタン配置にぴったり合わせたものが欲しかったのが動機です。

先に断っておきますが、この記事では以下の2点を守って書いています。

1. **作ったもの自体（アプリ・コード）は配布しません**
2. **コードの中身の詳細には踏み込みません**（個人利用前提で作ったもので、他の方の環境でそのまま使われても責任が取れないためです）

あくまで「どんなツールをどう組み合わせて、どういう流れで作ったか」という制作記録として読んでもらえればと思います。

![Overlay](https://static.zenn.studio/user-upload/f26c6899e756-20260712.png)

## 全体の流れ

1. KiCadでコントローラーの基板データからSVGを書き出す
2. Inkscapeで整形（ボタンごとに図形を整理）
3. ClaudeでSVGを加工（id・classの付与、不要なメタデータの削除）
4. HTMLに組み込み、見た目を作る
5. Visual Studio 2026 + C#で、コントローラーの入力を受け取ってブラウザに反映するアプリを作成

順番に振り返ります。

## 1. KiCad → SVG書き出し

コントローラー自体はKiCadで基板を設計しているので、その基板データをそのままオーバーレイの土台として流用しました。実物のボタン配置と完全に一致するSVGを最初から手に入れられるのが大きなメリットです。  
![](https://static.zenn.studio/user-upload/87e51e205417-20260712.png)

## 2. Inkscapeで調整

Inkscapeでは、不要な線の削除、パスの端点の結合、ボタンごとの図形のグループ化まで手作業で整えました。

![](https://static.zenn.studio/user-upload/a0e9c9d8f8ec-20260712.png)

## 3. Claudeで加工

グループ化した図形ひとつひとつに、後でプログラムから参照できるような`id`やクラスを付けていく作業が地味に大変だったので、Claudeに依頼して機械的に付与してもらいました。あわせて、Inkscapeが書き出す際に残る制作ソフト由来のメタデータも、公開・配布する予定はないものの念のため削除しています。

![](https://static.zenn.studio/user-upload/90f50ee48977-20260712.png)

## 4. HTMLへ組み込み

加工したSVGをHTMLに埋め込み、「対応するidのボタンが押されたら見た目を変える」というシンプルな仕組みを用意しました。これはOBSのブラウザソースとして読み込ませる前提で作っています。

## 5. Visual Studio 2026 + C#で入力受信アプリを作成

一番時間をかけたのがここです。Windows標準のコントローラー入力の仕組みを使ってボタンの押下状態を取得し、ブラウザ側にリアルタイムで反映する、という構成にしました。

ざっくりとした構成要素だけ挙げると

* ローカルにWebサーバーを立てる（.NET / ASP.NET Core）
* コントローラーの状態を一定間隔で読み取る
* 変化があったボタンだけをリアルタイム通信でブラウザ側に送る
* ブラウザ側のオーバーレイがそれを受け取って見た目に反映する

という、よくあるWeb技術の組み合わせです。具体的な実装や設定値はここでは割愛します。

## つまずいたポイント

### そもそもなぜサーバーを立てる必要があるのかピンとこなかった

OBSのブラウザソースは「指定したURLを表示するだけの小さなブラウザ」です。ボタンの押下をリアルタイムに反映し続けるには、そのページを配信し続ける何か（＝サーバー）が同じPC上で常に動いている必要がある、と理解するまで少し時間がかかりました。

### AIに任せた座標調整がブラックボックス化した

ボタンの画像サイズ・配置調整をすべてClaudeに任せたため、後から自分で細かく数値を触ろうとしてもどこをどう直せば見た目が変わるのか把握しづらく、少しのズレを直したいだけでも毎回AIに頼ることになってしまいました。「任せて楽になった部分」がそのまま「自分でメンテできない部分」になったのは想定外でした。

### コンソールウィンドウを非表示にした

起動時に表示されたままだと配信画面に映り込むリスクがあるため、非表示にしました。

### 起動・停止をわかりやすくした

URL表示とコピー・停止ボタンだけを備えた、小さなウィンドウを用意しました。

### ポート競合とアプリ終了まわりでしばらくハマった

誤ってアプリを2重に起動してしまったとき、詳細不明なエラー画面がそのまま出て固まる、という不具合がありました。原因を辿ると、もう1つ起動していた自分自身のプロセスにポートを使われていただけだったので、2重起動時にも分かりやすいメッセージを出して静かに終了するように直しました。もう1つ、「停止して終了」ボタンを押したのに、ブラウザ側をリロードせず開いたままにしていると、なぜかアプリがしばらく反応し続ける、という現象にも遭遇しました。原因は、ブラウザとの接続が生きている間はサーバー側が行儀よく終了処理を待ち続けてしまう仕様のためで、この待ち時間を短くすることで解消しました。

### HTTPS化は不要と判断した

ローカルの2台のPCで完結する使い方なので、暗号化までは必要ないと判断しました。

このあたりは、実際に手を動かしながらClaude Codeと一緒にひとつずつ潰していきました。

## 使ってみた感想

自分のコントローラーの見た目そのままで入力が表示されるので、既存のオーバーレイツールにはない愛着があります。基板設計からソフトウェアまで一貫して自分の手で作れたのも面白い経験でした。

## おわりに

今回は個人利用が前提で、配布や汎用化は考えていません。同じようなことをやってみたい方は、「Vortice.XInput」「SignalR（またはWebSocket）」「OBS ブラウザソース」あたりのキーワードで調べてみると、近い情報が見つかると思います。
