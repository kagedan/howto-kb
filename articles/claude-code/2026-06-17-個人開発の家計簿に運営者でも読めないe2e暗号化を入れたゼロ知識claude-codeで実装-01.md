複式簿記の家計簿アプリ **黒福簿（kurofukubo）** に、\*\*端末側のエンドツーエンド暗号化（ゼロ知識）**を入れました。  
狙いは一点、**「運営者（＝個人開発者の私）でも、預けられた家計データを読めない」\*\*状態にすることです。実装はほぼ Claude Code（AIエージェント）と進めました。

この記事は「ゼロ知識E2Eを実アプリに後付けすると、どこが原理的に難しく、UX/移行で何を割り切るか」の実録です。

## まとめ（先に結論）

* 家計簿は「運営者に中身を見られるのでは」という不安が最大の障壁。\*\*“見ない約束”より“見られない設計”\*\*にした。
* **保管時暗号化（AWS KMS等）≠ 運営者が読めない**。運営者が鍵を持つ＝復号できる。**claimは正確に**。
* ゼロ知識にするには**鍵をユーザー側だけが持つ**必要がある＝**パス/パスキー等の秘密は原理的に不可避**。「手間ゼロ＋運営者も読めない」は両立しない。
* 経路・サーバーメモリにも平文を出さないなら、**暗号化はクライアントで**やるしかない（サーバーが復号する設計は経路もメモリも平文になる）。
* 既存の平文をあとから暗号化しても、**バックアップ(PITR)には保持期間ぶん平文が残る**。完全に消したいなら**最初から有効化**。
* UXの肝：**鍵を端末のIndexedDBに保持して“毎回パス入力”を回避**。ただし**リカバリキーは必須**（忘れたら復元不能）。

## なぜやるか：「保管時暗号化」では足りない

初期の想定は普通の構成でした：HTTPS＋DynamoDBの保管時暗号化＋テナント分離＋「運営者は見ない」方針。  
でもこれ、**運営者が普通にDBをクエリすれば平文が返ります**（保管時暗号化はディスク/インフラ対策で、透過的に復号される）。家計データのセンシティブさを考えると、**“見ない約束”では不安が消えない**。

そこで「運営者が**その気でも**読めない」＝**ゼロ知識E2E**を目指しました。

## 設計：クライアント側エンベロープ暗号

鍵は2段（エンベロープ暗号）。**鍵はサーバーに送らない**のが肝。

```
パスフレーズ ──KDF(salt)──▶ KEK ──wrap──▶ [DEK] ──AES-GCM──▶ データを暗号化
リカバリキー ─KDF(salt2)──▶ RKEK ─wrap──▶ [DEK]   （復旧用に別ラップ）
```

* **DEK**（データ鍵, ランダム256bit）でデータを暗号化。
* **DEK を「パスフレーズ由来鍵」と「リカバリキー由来鍵」の2つでラップ**して保存。サーバーには**暗号文とラップ済み鍵だけ**＝平文鍵は無い。
* パス変更は**DEKの再ラップだけ**（全データ再暗号化は不要）。

暗号は Web Crypto（依存ゼロ）。KDFは PBKDF2-SHA256(60万回)、暗号は AES-256-GCM。

```
// 封緘/開封（DEKでJSONを暗号化/復号）
export async function seal(dek, obj) {
  const nonce = rand(12);
  const ct = new Uint8Array(await subtle.encrypt({ name: 'AES-GCM', iv: nonce }, dek, te.encode(JSON.stringify(obj))));
  return toB64(concat(nonce, ct));
}
// パスフレーズ + salt → KEK（DEKラップ用）
async function deriveKEK(passphrase, salt) {
  const base = await subtle.importKey('raw', te.encode(passphrase), 'PBKDF2', false, ['deriveKey']);
  return subtle.deriveKey({ name: 'PBKDF2', salt, iterations: 600000, hash: 'SHA-256' },
    base, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']);
}
```

サーバーは暗号文ブロブ＋鍵バンドルを置くだけ：

```
[クライアント] 暗号化 → 送信(暗号文) → [サーバー] 暗号文を保管/返却 → [クライアント] 復号
```

→ **経路もサーバーのメモリもDBも、サーバー側はずっと暗号文**。鍵が無いので運営者は読めません。

## ハマり・学び（ここが本題）

### ① 「保管時暗号化」を「運営者が読めない」と書いてはいけない

AWSのKMS暗号化は運営者が復号できます。LPの安全性ページでは\*\*「保管時暗号化」とは書くが「運営者が見られない」とは書かない\*\*よう、表現を厳密に分けました（ゼロ知識は別途の機能として明記）。

### ② 「手間ゼロ＋運営者も読めない」は原理的に無理

「ユーザーに紐づく鍵をサーバーが保管してログイン時に渡せば手間ゼロでは？」と考えましたが、**サーバーが“使える鍵”を渡せる＝運営者も持てる＝読める**。ゼロ知識を保つには**ユーザーだけが持つ秘密が必須**です。  
唯一の折衷は「**鍵をユーザーのログインパスワードでラップして保管**」（サーバーは開けない／SRPでパスワードも知らない）。これならメール勢は別パス不要に近づきますが、Googleログインや**パスワード再設定＝鍵が変わりデータ復旧不可**という宿題が残り、結局**リカバリキー**が要ります。

### ③ 経路・メモリも守るならクライアント暗号化一択

「サーバー管理鍵で保管時だけ暗号化」案は、**復号がサーバーで起きる＝経路もサーバーメモリも平文**になります。ダンプ流出対策にはなっても“運営者対策”にはならない。経路もメモリも平文を出さないなら、暗号化は**クライアントで**やるしかありませんでした。

### ④ 既存の平文はあとから消しても“バックアップに残る”

有効化時に**サーバー上の平文（per-type items）を削除**して暗号文に置換します。が、DynamoDBの **PITR（継続バックアップ）が有効だと、復元すれば約35日は過去の平文が読めます**（運営者の意図的な復元＋監査ログ付き）。**完全ゼロ知識を保証したいなら「最初から有効化」**（平文を一度もサーバーに書かない）が確実、と正直に整理しました。

### ⑤ UX：毎回パスは無理。鍵を端末に持たせて自動解錠

ゼロ知識だと「リロードのたびにパス入力」になりがち。そこで**解錠済みのDEKを端末のIndexedDBに保持**し、\*\*同じ端末なら自動解錠（パス不要）\*\*に。別端末・データ消去・鍵不一致のときだけ解錠画面に戻します。鍵はサーバーに送らないのでゼロ知識は維持。

![](https://static.zenn.studio/user-upload/3fc5cebb9b3f-20260617.png)

### ⑥ リカバリキーは必須（ゼロ知識の宿命）

運営者がリセットできない＝**忘れたら誰も復元できない**。なので**リカバリキーで第2のラップ**を作り、パス紛失/再設定時はこれで復元→新パスで再ラップ。自動生成＋ファイルDL＋「後で」＋設定から再発行、にして摩擦を最小化。未保存時は**アイコン＋ホバー**でそっと警告（常時の文字警告は出さない）。

## AI（Claude Code）での進め方

家計データの暗号は**バグると全データ復号不能**なので、**層ごとにボトムアップで実装＋都度テスト**しました。

1. 暗号コア（seal/open・KDF・エンベロープ・リカバリ）→ Nodeでラウンドトリップ検証
2. localStorage入出力層 → モックで検証
3. DataContext統合（useLocal＋API両対応、暗号化時は状態のみ＋ブロブ保存）
4. UI（解錠ゲート・設定パネル・リカバリ）→ Playwrightで「有効化→暗号文化→リロードで自動解錠→鍵消去でロック→復元」をE2E検証

各層をAIに実装させ、**検証スクリプトもAIに書かせて数値で裏取り**するのが効きました。  
ハマりどころは環境系で、**WindowsでチャットのBashログインがPowerShell側の `aws` キャッシュに反映されない**（SSOトークン別）など、AIにログを切り分けさせて解決しました。

## 透明性も同時に

「安全とデータの扱い」ページを用意し、第三者提供なし・銀行連携なし・データ所有権（エクスポート/削除）・端末のみ利用（ゲスト）を平易に明記。**過剰主張を避け**、ゼロ知識E2Eは“追加の選択肢/今後の標準化”として位置づけました。

## 所感

* ゼロ知識は「鍵をサーバーに持たせない」一点に尽き、**便利さ（パス無し・運営者リセット）とは構造的にトレードオフ**。きれいな“いいとこ取り”は無い、と理解するのが出発点でした。
* 一方で**端末側の鍵保持（自動解錠）＋リカバリキー**で、日常の手間はかなりゼロに寄せられます。
* 個人開発でこの規模の暗号を入れられたのは、**層ごとに実装→検証→デプロイ**をAIと高速に回せたから。家計のような“壊せない”データほど、検証の自動化が効きます。
