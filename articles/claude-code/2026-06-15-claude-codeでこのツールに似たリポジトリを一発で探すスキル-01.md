> 「いま使っているこのライブラリ、**同じことをする別の実装**や、**もっと新しい代替**って何があるんだろう？」

開発中にこう思って、GitHub やキーワード検索をさまよった経験はありませんか？キーワード検索は「形態素解析」のような**トピック**には強い一方で、「**この `mecab` というツールにどれだけ近いか**」という軸では並べてくれません。表現が少し違うだけの近いツールは取りこぼし、「mecab への近さ順」に並べ替えることもできない。1,200 件を超えるリポジトリを手でカテゴリごとに見比べるのも、なかなか骨が折れます。

そこで、**起点となるツール名を 1 つ渡すと、同じ・関連する処理をするリポジトリを「収録リスト」と「Web」の両方から探し、ランク付けして並べてくれる** Claude Code スキル `similar-resources` を作りました。`mecab` と打てば fugashi や kuromoji が、`oseti に似たツール` と打てば asari や pymlask が返ってきます（mecab は文を単語に区切る**形態素解析**、oseti は文の感情を判定する**感情分析**のツールです）。

![awesome-japanese-nlp-resources メイン画像](https://static.zenn.studio/user-upload/3b525472b183-20260616.png)

これは、[awesome-japanese-nlp-resources](https://github.com/taishi-i/awesome-japanese-nlp-resources) に収録された **1,200 件以上の日本語 NLP リソース**を Claude Code から横断検索できる **Plugin** に追加した、5 つ目のスキルです。前回はその Plugin 自体を紹介しました。

ただ、この記事で一番お伝えしたいのは「便利なスキルができました」という話ではありません。**自分のデータ（社内ナレッジでも、自前の awesome リストでも）に同じ仕組みを応用する方法**です。ポイントは 2 つです。

* **重い計算は Bash + Python に逃がし、意味的な判断は Claude に任せる**という役割分担
* スキルの追加は **「フォルダ 1 つ ＋ `SKILL.md` 1 枚」** で済むという身軽さ

日本語 NLP を使わない方でも、この 2 点はそのまま応用できます。

!

**この記事でわかること**

* 「あるツールに似たリポジトリ」を、収録リスト＋Web から一発で並べる使い方
* `search`（キーワード検索）と `similar-resources`（類似検索）の**使い分け**
* 安く絞り込み（Bash + Python）→ 意味で決める（Claude）という、**自分のデータにも応用できる設計パターン**
* 自分のリストに Claude Code スキルを 1 つ足す手順（コピペできるチェックリスト付き）

## こんな感じで動きます

インストール（後述）が済んでいれば、Claude Code 上でスラッシュコマンドを打つだけです。まずは英語クエリから。

```
/awesome-japanese-nlp-resources:similar-resources mecab
```

## インストール

`similar-resources` を含む 5 つのスキルは、Claude Code 上で次の 3 行を実行すればインストールできます（Claude Code が使える環境が前提です）。

```
/plugin marketplace add taishi-i/awesome-japanese-nlp-resources
/plugin install awesome-japanese-nlp-resources@awesome-japanese-nlp-resources
/reload-plugins
```

1 行目でこのリポジトリをマーケットプレイスとして追加し、2 行目でプラグインをインストール、3 行目で再読み込みします（環境によっては再読み込みは不要な場合もあります）。完了すると、次の 5 つのスキルが使えるようになります（すべてクエリの言語を自動判定し、英語／日本語で応答します）。

| スキル | 用途 |
| --- | --- |
| `/awesome-japanese-nlp-resources:search <query>` | 同梱の 1,200+ リソースを検索 |
| `/awesome-japanese-nlp-resources:similar-resources <repo>` | リポジトリ/ツールを起点に、似たものを発見 |
| `/awesome-japanese-nlp-resources:find-new-resources <topic>` | リスト未収録の GitHub リポジトリを発見（貢献支援） |
| `/awesome-japanese-nlp-resources:research-trends <topic>` | データセット＋Web 調査でトレンドレポート |
| `/awesome-japanese-nlp-resources:research-issues <topic>` | あるトピックの課題・制限・解決策を調査 |

詳しい仕様は [Plugin の README](https://github.com/taishi-i/awesome-japanese-nlp-resources/blob/main/plugins/awesome-japanese-nlp-resources/README.md) をご覧ください。

## `search` と `similar-resources`、どっちを使う？

スキルが増えると、次に気になるのは「どっちを使えばいいのか」です。どちらも「リソースを探す」スキルですが、**答える問いが違います**。

* **`search`** は「**トピック → リソース**」。「日本語の形態素解析ライブラリは？」のような**キーワード検索**です。まず広く探したいとき。
* **`similar-resources`** は「**この特定のツール → 似たもの**」。「いま mecab を使っているけど、代替・後継は？」のような**起点（seed）ありきの検索**です。乗り換え先を絞り込みたいとき。

|  | `search` | `similar-resources`（今回） |
| --- | --- | --- |
| 入力 | トピック・キーワード | 特定のリポジトリ／ツール（起点） |
| 問い | 「○○なリソースは？」 | 「この**ツールに似たもの**は？」 |
| 並べ替えの軸 | クエリとの関連度 ＋ 人気度 | **起点への内容類似度** ＋ 機能の近さ |
| 主な用途 | まず広く探す | 乗り換え先・代替・後継を絞り込む |

キーワード検索だと「mecab への近さ」という軸で順位を付けられず、表現が違うだけで意味的に近いリソースも取りこぼしがちです。その隙間を埋めるのが `similar-resources` です。

## 仕組み：安く絞り込み → Claude が意味で決める

`similar-resources` の設計の方針は、ひとつだけです。

> **重い計算は Bash + Python に逃がし、意味的な判断は Claude が直接行う。**

具体的には、まず Python が同梱データ **1,170 件**（本記事執筆時点のスナップショット。リストの更新に合わせて増えます）を機械的に採点し、**上位約 15 件に絞り込みます**。そのうえで「fugashi は汎用ツールキットより mecab に近い」といった**意味の判断**を Claude が担います。

1,170 件すべてを Claude に読ませる必要はなく、かといって最終順位を数値だけで決めることもしない。この役割分担が肝で、**そのまま自分のデータにも応用できる**ところです。

全体は「**データセット内の類似度**と**Web 調査**を並走させ、最後にマージする」構成になっています。

1. **同梱データセット内の類似度スコアリング**：バンドルした `resources.json`（1,170 件）を、seed への内容類似度で採点
2. **Web での補完**：WebSearch / WebFetch で、**まだ収録していない**関連リポジトリを探す
3. **マージして再ランク**：2 つの結果を統合（収録済み／未収録で重複しないので、上下の表に分かれます）

### 類似度スコアの考え方

データの各項目は、カテゴリ `c` と、カンマ区切りの**意味ラベル** `s` を持っています。たとえば形態素解析系なら `Syntactic Text Processing, Text Segmentation, Tagging, Morphology` のような具合です。

データ 1 件は、こんな素朴な JSON です（`oseti` の例）。

```
{
  "n": "oseti",
  "u": "https://github.com/ikegami-yukino/oseti",
  "c": "Python library",
  "s": "Sentiment Analysis, Vocabulary, Dictionary, and Language Input Method",
  "d": "Dictionary based Sentiment Analysis for Japanese",
  "st": 93,
  "ns": 5.21
}
```

* `n` / `u`：名前と GitHub URL
* `c`：カテゴリ、`s`：カンマ区切りの意味ラベル、`d`：英語の説明（**類似度は主にこの 3 つ、加えて名前 `n` も素材に決まります**）
* `st`：GitHub スター数（Hugging Face リソースは代わりに `dl` ＝ ダウンロード数）、`ns`：人気度を正規化した値

**自分のリストでも、この形の配列を 1 つ用意すれば同じスキルが動きます。** 重いのはデータ作りのほうで、スキル側（`SKILL.md`）は薄いまま使い回せます。

スコアはおおまかに次の合算で決まります。

* **同じカテゴリ**なら +10、**同じ系統**（family ＝ カテゴリ名の `(` より前の部分。たとえば `Python library (Morphology)` なら `Python library`）なら +5
* **共有する意味ラベル**の重み（IDF 加重）
* **共有する説明文・名前の単語**の重み（IDF 加重）

肝は **IDF 加重**（＝逆文書頻度。ありふれたラベルほど軽く、珍しいラベルほど強く効かせる重み付け）です。`Dictionary` のようにどこにでも付くラベルは効きを弱く、`Sentiment Analysis` のような稀少で識別力の高いラベルは効きを強くします。これで「ただカテゴリが同じ」よりも「**やっていることが本当に近い**」リソースが上に来ます。

しきい値（スコア 8 以上）を超えた候補は、最後に人気度をほんの少し加味して（おおよそ `スコア + 0.5 × 人気度`）並べます。内容の近さを主、人気度を従、というバランスです。

こうして残った候補を、**Claude が「機能的な近さ」で並べ替え、「類似する理由」も書きます**。たとえば MeCab ラッパーの fugashi は、汎用 NLP ツールキットより mecab に「近い」と言えます。こうした判断は数値だけでは付けきれないので、ここは Claude にまかせています。

### Web 探索で気をつけていること

Web 側は WebSearch と WebFetch だけを使い、見つけた `github.com/owner/repo` を集めたあと、

* `resources.json` に**すでにある**ものは `grep` で落とす（＝未収録の表には常に「リストに無いもの」だけが並ぶ）
* archived なリポジトリや、放置された fork は除外する

という後処理を入れています。seed 自体がリストに無い場合は、Web でその素性を調べたうえで最寄りの収録ツールを返し、「これは収録候補なので `find-new-resources` をどうぞ」と案内します。

## 自分の awesome リスト／データにスキルを足す方法

ここからは、冒頭でふれた**再利用できる部分**の具体的な手順です。「自分のリストでも同じことをやりたい」という方向けに、**スキルを 1 つ追加する流れ**を、今回の `similar-resources` を例にまとめます。

### 1. Plugin の最小構成を知る

Claude Code Plugin のディレクトリは、ざっくり次の形です。

```
plugins/awesome-japanese-nlp-resources/
├── .claude-plugin/
│   └── plugin.json          # プラグインのメタ情報
├── data/
│   └── resources.json       # 同梱データ
└── skills/
    ├── search/
    │   └── SKILL.md
    ├── similar-resources/   # ← 今回追加したフォルダ
    │   └── SKILL.md
    ├── find-new-resources/
    │   └── SKILL.md
    ├── research-trends/
    │   └── SKILL.md
    └── research-issues/
        └── SKILL.md
```

ここで一番大事なポイントは、

> **スキル = `skills/` の下のフォルダ 1 つ ＋ その中の `SKILL.md` 1 枚**

ということです。スキルを増やすのに、どこかへコードを登録したり配列に追記したりする必要はありません。**フォルダを足すだけ**で Claude Code が自動で見つけてくれます。今回も `skills/similar-resources/SKILL.md` を 1 枚足しただけです。

### 2. frontmatter を書く

`SKILL.md` の先頭には YAML の frontmatter を置きます。今回の `similar-resources` の実際の値はこうです。

```
---
description: Given a Japanese NLP GitHub repo (URL / owner/repo / tool name), find repositories that do the same or related processing. Mines the bundled dataset for content-similar items, then expands via web research, then merges and re-ranks.
when_to_use: "Use when the user names a SPECIFIC repository or tool and wants ones like it — alternatives, equivalents, or related tools. Trigger phrases include 'mecabに似たツール', 'fugashiの代替', 'ginzaと同じことができるライブラリ', '〜の類似リポジトリ', '〜の代わり', 'alternatives to fugashi', 'repos like manga-ocr'. For a topic/keyword search use the search skill; to find repos NOT yet in the list use find-new-resources."
argument-hint: [github-url | owner/repo | tool-name]
allowed-tools: Bash WebSearch WebFetch
---
```

各フィールドの役割は次のとおりです。

* **`description`**：このスキルが何をするか。Claude がどのスキルを呼ぶか判断するときの材料になります。
* **`when_to_use`**：**いつ呼ぶか**。ここが一番重要です。`mecabに似たツール` `fugashiの代替` `〜の代わり` `alternatives to fugashi` といった**トリガー語句**を厚めに書いておくと、ユーザーが自然文で書いても適切に発火します。さらに「トピック検索は `search`、未収録探しは `find-new-resources`」と**兄弟スキルとの棲み分け**まで書いておくと、似た用途のスキルどうしで誤発火しにくくなります。
* **`argument-hint`**：引数のヒント。`[github-url | owner/repo | tool-name]` のように受け取れる形を示します。
* **`allowed-tools`**：スキルが使ってよいツール。`similar-resources` は `Bash WebSearch WebFetch` を許可しています。

> 細かい点ですが、`when_to_use` はアンダースコア区切り、`argument-hint` / `allowed-tools` はハイフン区切りです。フィールドによって区切り文字が違うので、コピーするときはこのまま書き写してください。

### 3. 本文は「Claude への手順書」

frontmatter の下の本文は、そのまま **Claude に渡す手順書（プロンプト）** です。`similar-resources` の骨子はこんな流れになっています。

1. **入力検証**：引数が空なら使い方を表示して停止
2. **データの場所を解決**：`${CLAUDE_PLUGIN_ROOT}/data/resources.json` を参照
3. **スコアリング**：1 回の Bash 呼び出しで Python を実行し、seed への類似度で採点
4. **Web 調査**：未収録の関連リポジトリを WebSearch / WebFetch で補完
5. **再ランク**：Claude が機能的な近さで並べ替え
6. **出力整形**：日本語／英語のテンプレートで表を生成

実際に書いてみて効いた、**他のスキルにも使い回せるイディオム**が 4 つあります。

```
# データの場所はこれ 1 行で解決（インストール場所に依存しない）
RESOURCES_PATH="${CLAUDE_PLUGIN_ROOT}/data/resources.json"
[ -f "$RESOURCES_PATH" ] || RESOURCES_PATH="$(find "${HOME}/.claude/plugins" -name resources.json | grep awesome-japanese-nlp-resources | head -1)"
```

* **`${CLAUDE_PLUGIN_ROOT}`**：同梱データへのパスはこれで解決します（Claude Code がインストール先に置き換えてくれる）。
* **巨大ファイルに Read ツールを使わない**：`resources.json` は大きいので、Read で読むとトークンを浪費します。`SKILL.md` 内に「Do NOT use the Read tool」と明記し、Bash + Python で処理させます。
* **言語自動判定ルール**：「引数に日本語が含まれていれば日本語、そうでなければ英語」と本文に書いておくだけで、出力言語が切り替わります。
* **年号をハードコードしない**：本文の冒頭で `date` を 1 回だけ実行し、その値を出力全体に差し込みます。

```
# 実行時の date から「今年」を取り、出力中の年号はすべてこれを使う
echo "YEAR_NOW=$(date +%Y)"
```

年号を直書きすると「最終更新 2024」のような古い表記が紛れ込みますが、これを使えば年をまたいでも出力が古びません。Web 調査を伴うスキルで特に効きます。

### 4. 反映してテストする

スキルを追加したら、再読み込みして呼び出すだけです。

```
/reload-plugins
/awesome-japanese-nlp-resources:similar-resources mecab
```

### 5. コピーして使えるチェックリスト

最後に、自分のデータでスキルを 1 つ足すときの最小チェックリストをまとめておきます。

データ（の質）さえ用意できれば、awesome リストでも社内ナレッジでも、「データ + `SKILL.md` 1 枚」で実用的なスキルが足せます。

## 限界と、向いていないケース

万能ではありません。使う前に知っておきたい前提を、正直に書いておきます。

* **データの質に依存します。** スコアは各リソースのカテゴリ・ラベル・説明文から計算するので、ラベル付けが薄いデータでは精度が落ちます（だからこそ「データ作りが本体」です）。
* **Web の結果は実行ごとに揺れます。** 未収録の表は検索のたびに少し変わり得ます。再現性が要る用途では、収録済みの表を基準にしてください。
* **意味ラベルは英語中心です。** ラベルが英語で整備されているため、日本語の説明だけに頼るクエリは弱くなることがあります。
* **「似ている」≠「おすすめ」「高品質」。** あくまで機能的な近さの提示です。採用前には star 数・更新日・ライセンスを自分で確認してください。
* **同梱データはスナップショット**です。最新の収録状況はリスト本体（README）で確認できます。

## まとめ

* awesome-japanese-nlp-resources の Claude Code Plugin に、5 つ目のスキル **`similar-resources`** を追加しました。起点リポジトリを 1 つ渡すと、**収録済みの近いリソース**と **Web で見つけた未収録リポジトリ**を、内容類似度 + Claude の意味判断で並べて返します。
* 設計の肝は **「安く絞り込み（Bash + Python）→ 意味で決める（Claude）」** の分業。スキルの追加は **「フォルダ 1 つ ＋ `SKILL.md` 1 枚」**。この 2 つは、あなたのデータ（社内ナレッジでも自前リストでも）にそのまま応用できます。

まずは Claude Code 上で 3 行を実行し、**あなたの推しツール名**を `similar-resources` に渡してみてください。そして気が向いたら、上のチェックリストを参考に、**自分のデータ用のスキル**を 1 つ作ってみてください。awesome-japanese-nlp-resources は CC0 で公開しており、リソースの追加 PR も歓迎しています。

それでは、最後までお読みいただきありがとうございました！
