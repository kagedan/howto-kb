Claude CLIとMCPサーバーを使って、AI経由で簡単な業務アプリを操作できるかPoCしていました。

次に考える必要があったのが、MCPサーバーを呼び出す側、いわゆるMCPホストをどう選ぶかです。  
今回はローカルファイルやローカルのviewerアプリを操作する可能性があったため、機能の多さだけでなく、PoCからの移行しやすさ、コスト、将来クラウド側に寄せられるかを見ながら選びました。

この記事は、そのときの技術選定メモです。

## 前提

* Claude CLIでPoCしてきた
* 会社でClaudeの有料プランを使っている
* ローカルファイルやローカルアプリを操作する可能性がある
* PoCの挙動を大きく変えずに実装へ進めたい

## 結論

現時点では、第一候補を **Claude Agent SDK** にしました。

理由は、Claude CLIで作ってきたPoCに一番近い形で進められそうだったからです。  
Claude Agent SDKは、Claude Codeを支えているエージェントループやツール実行、MCP連携をアプリケーション側から使えるSDKです。

ローカルファイルを読む、コマンドを実行する、MCPサーバーにつなぐ、といった用途に近いため、今回の前提とは相性がよさそうでした。

一方で、**Strands Agents** は第二候補、**Amazon Bedrock AgentCore** は今回は見送りにしました。

## Claude Agent SDKを選んだ理由

一番大きいのは、今回の前提に対してバランスがよさそうだったことです。

別のエージェントSDKに移すと、プロンプトの扱い、ツール選択、権限確認、実行ループなどが変わる可能性があります。  
PoCで見ていた挙動が変わると、何が原因で変わったのか追いにくくなります。

その点、Claude Agent SDKはClaude CLI / Claude Codeに近い流れで進められそうです。  
ローカル操作との相性、Claudeの有料プランでの検証しやすさ、Bedrock経由で将来のモデル選択肢が広がりそうな点も含めて、まずここから始めるのが自然だと考えました。

## OpenAI対応について

最初は「Claude Agent SDKならBedrockもOpenAIも使える」と考えていました。  
ここは少し分けて考えたほうがよさそうです。

まず、**OpenAIモデルはAmazon Bedrockで使えるようになっています**。  
OpenAIは2026年6月1日に、OpenAI frontier modelsとCodexがAWSで利用できるようになったと発表しています。  
AWS側にも、OpenAIモデルをAmazon Bedrock上で提供するページがあります。

一方で、Claude Agent SDKのBedrock連携は、Claude Code / Claude Agent SDKをAWS基盤で使うための設定です。  
そのため、**BedrockでOpenAIが使えること**と、**Claude Agent SDKのモデルをOpenAIにそのまま切り替えられること**は、現時点では分けて考えたほうがよさそうです。

ただ、Bedrock側でOpenAI対応が進んでいるので、将来的な選択肢としては楽観的に見ています。  
今すぐ使うとは限りませんが、将来モデルごとの挙動差を比較したくなった場合にも、AWS側に寄せておくと調べやすそうです。

## 料金まわり

Claudeの有料プランで検証できるかも確認しました。  
公式ヘルプでは、2026年6月15日からPro、Max、Team、Enterprise向けにAgent SDK用の月額クレジットが用意されると説明されています。

ただし、クレジットの扱いや商用利用の条件は契約・プランによって変わる可能性があります。  
社内PoCとして試す分には進めやすそうですが、社外向けサービスや本番利用に使う場合は、APIキー利用や契約条件を確認したほうがよさそうです。

また、トークンを多く使いそうな動作検証は、引き続きClaude CLI経由で行う方式がよさそうです。  
Agent SDKで実装に近い形を確認しつつ、試行錯誤が多い検証はCLI側に寄せると、コストや利用枠を見ながら進めやすいと考えました。

## Strands Agentsは第二候補

Strands Agentsは、AWSが公開しているオープンソースのエージェントSDKです。  
Apache 2.0ライセンスで、Amazon Bedrock、OpenAI、Anthropicなど複数のモデルプロバイダーに対応しています。

AWS中心で作りたい場合や、将来的にAgentCoreまで含めてAWS側に寄せたい場合は有力な候補だと思います。

ただ、今回は一旦採用しませんでした。  
理由は、PoCから挙動が変わる可能性があるためです。

Strands Agentsは「model-driven」な設計で、モデルがツール利用や次の行動を動的に判断することが特徴です。  
これは柔軟に動けるメリットですが、今回のように「PoCで見た挙動をまず再現したい」段階では、指示した手順と少し違う進め方を選ぶ可能性もあります。

そのため、採用するならツール権限、ガードレール、評価、ログ確認をきちんと設計する必要がありそうです。

## AgentCoreは今回は見送り

Amazon Bedrock AgentCoreは、エージェントやMCPサーバーをAWS上で動かすための基盤としては便利そうです。  
Runtime、Gateway、Memory、Identity、Observabilityなどがまとまっていて、AWS上で本番運用したい場合には魅力があります。

ただ、今回はローカルファイルやローカルアプリを操作する可能性があります。  
AgentCore RuntimeはMCPサーバーをAWS上にデプロイして動かす文脈が強いため、ローカルPC上のアプリを直接操作したいPoCとは相性がよくありません。

クラウドにホストを置くこと自体が無理というわけではありませんが、ローカル操作が前提に残っている段階では、まずローカル側で挙動を安定させるほうがよいと考えました。  
クラウド上で完結できると分かった段階で、改めて検討するのがよさそうです。

## 比較

| 候補 | よさそうな点 | 今回の懸念 | 判断 |
| --- | --- | --- | --- |
| Claude Agent SDK | Claude CLIのPoCに近い。MCPやローカル操作と相性がよい。Bedrock経由で将来のモデル選択肢も広がりそう | Claude Agent SDKからそのままOpenAIモデルに切り替える前提は現時点では要確認。利用条件も確認が必要 | 第一候補 |
| Strands Agents | AWS寄り。Apache 2.0。複数モデル対応。モデル主導で柔軟に動ける | PoCから実行ループや挙動が変わる可能性がある。手順の再現性を重視するなら評価やガードレール設計が必要 | 第二候補 |
| Amazon Bedrock AgentCore | AWS上で本番運用しやすい。MCPサーバーのデプロイ基盤として使える | ローカルアプリ操作には向きにくい | 今回は見送り |

## まとめ

今回の選定では、機能の多さだけでなく、ローカル操作のしやすさ、PoCからの移行しやすさ、コスト、将来のクラウド移行をまとめて見ました。

その前提では、まずClaude Agent SDKから始めるのが自然そうです。  
コストがかかりそうな動作検証はClaude CLI経由で進め、クラウド完結できると分かったらStrands AgentsやAgentCoreを再検討する、という流れがよさそうだと思いました。
