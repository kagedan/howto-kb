## はじめに

最近 X で「AI の使い方は **直列か並列かで出力品質が全然違う**」という話を読んで、自分の運用と照らして妙に納得した。

並列のほうが時間効率がいいのは間違いない。何個も Claude を同時に走らせれば、単純計算で n 倍速い。だが、実装してみると別の問題が現れる。**並列で走らせた n 本のうち 1 本でも壊れた瞬間に、依存していた後続が全部やり直しになる**。期待 retry 回数が膨らんで、トータルの壁時計時間も推論コストも、結局直列で組んだほうが安かった、という現象を何度か踏んだ。

順序にすべきか、並列に開いていいか。これは AI ワークフローを組むときに毎回出てくる判断で、自分の場合は「PRD を受け取ってタスクグラフを作る」段階で大半が決まる。本稿はその設計の中身を、実コードを引きながら共有する。

前回(zenn-007)は Purple Codens の deterministic step machine を全体像として書いた。今回はその一段下、**PRD → タスクグラフの parse** と、**ノード間のエッジが順序か並列かを何で決めているか**、というレイヤーに絞る。

## 全体像: PRD はその場ではタスクにならない

まず Purple での流れを軽くおさらいする。

Green Codens(PRD 管理) で PRD のステータスが特定の値(たとえば "approved")に遷移すると、Purple は polling でそれを拾う。ここで重要なのは、**この時点ではまだ PurpleTask を作らない**ということ。`pending_analyses` という中継テーブルに「まだ分析されてない PRD」として積むだけ。

```
# infrastructure/tasks/green_sync.py 抜粋
matching_prds = [p for p in prd_list if p.status == target_status]
# ...
pending = PendingAnalysis(
    purple_project_id=pp.purple_project_id,
    source="green_prd",
    external_id=prd_id,
    payload=payload,
    status="queued",
)
created = await pa_repo.create(pending)
```

なぜワンクッション挟むかというと、**PRD のままではタスクグラフに分解できない**から。「OAuth ログインを実装して」と書かれたチケットを見て、ドメインエンティティ・リポジトリ・ユースケース・API ルート・フロントエンドのどれをどの順で触るかは、リポジトリの構造を読まないと判断できない。

そこで `process_pending_analyses` という Celery beat が 30 秒おきに `queued` のものを取り出し、VPS で read-only の Claude Code セッションを立てる。CLAUDE.md とディレクトリツリーをまず読ませて、それから「この PRD を実装するとどういう構造のサブタスク群になるか」を JSON で吐かせる。

```
# use_cases/task/analyze_import_use_case.py の prompt 抜粋
"""
## Phase 1: Repository Analysis (complete BEFORE writing specs)

1. Read CLAUDE.md at each repo root
2. Explore directory structure
3. Check test configuration
4. Grep for relevant existing code
5. Read a sample test file

## Phase 2: Generate Task Specifications

For each item, produce ONE parent task with 2-5 sub-tasks.
Order sub-tasks by implementation dependency:
  - Domain entities / DB models first
  - Repository / infrastructure second
  - Use case / service logic third
  - API layer / routes fourth
  - Frontend components last

Use `dependency_indices` to encode these ordering constraints
within each parent task's sub_task list.
"""
```

「分析しないとタスクに落ちない」という前提があるから、PRD 受領 → 即タスク生成にしていない。AI はリポジトリを読んだ上でグラフ構造を提案する役割で、その提案結果が `TaskSpec` というデータクラスに落ちる。

## TaskSpec の構造

返ってくる JSON はこういう形をしている。

```
# use_cases/task/analyze_import_use_case.py
@dataclass
class SubTaskSpec:
    title: str
    goal: str
    priority: int = 5
    workflow_id: Optional[str] = None
    repository_id: Optional[str] = None
    dependency_indices: List[int] = field(default_factory=list)

@dataclass
class TaskSpec:
    title: str
    goal: str
    source: str
    external_id: Optional[str] = None
    priority: int = 5
    dependency_ids: List[str] = field(default_factory=list)
    sub_tasks: List[SubTaskSpec] = field(default_factory=list)
```

ポイントは `dependency_indices` で、「この sub\_task は同じ親の何番目のサブタスクが終わってから走らせろ」という形で **ノード間のエッジを宣言する**。空リストなら依存なし、つまり「親が unblock されたら他の兄弟と並列に走っていい」を意味する。

importer 側では、その index を実際の `task_id` に変換しながら DB に書き込む。

```
# api/v1/routes/purple_tasks.py 抜粋
created_sub_tasks: list[UUID] = []  # 作成済みサブタスクのIDを順に保持

for sub_spec in spec.sub_tasks:
    # インデックス指定の依存を実タスクIDに変換
    sub_dep_ids: list[UUID] = []
    for idx in sub_spec.dependency_indices:
        if 0 <= idx < len(created_sub_tasks):
            sub_dep_ids.append(created_sub_tasks[idx])

    sub_task = await use_case.create_task(
        purple_project_id=purple_project_id,
        title=sub_spec.title,
        goal=sub_spec.goal,
        parent_task_id=parent_task.task_id,
        dependency_ids=sub_dep_ids,
        # ...
    )
    created_sub_tasks.append(sub_task.task_id)
```

ここで作られた `dependency_ids` がそのままタスクグラフの順序エッジになる。エッジが張られた依存先タスクが終了状態(`done` か `cancelled`)に入るまで、後続タスクは `blocked` のまま動かない。

## 順序エッジを張るかどうかの判断基準

「この 2 タスクを順序にすべきか並列にしていいか」は、自分の場合次の 4 つで判定している。プロンプトで AI に判断を委ねる部分もあるが、最終的には DB レイヤで防御している。

### 1. データ依存(後段が前段の出力を参照する)

これは絶対に順序エッジ。たとえば「ドメインエンティティを作る」サブタスクと「そのエンティティを使うリポジトリ実装」サブタスクは、後者が前者のクラス定義を import する以上、並列にしたら後者が import error で必ず壊れる。

`analyze_import_use_case.py` のプロンプトで Phase 2 に明示している順序

```
Domain entities / DB models first
Repository / infrastructure second
Use case / service logic third
API layer / routes fourth
Frontend components last
```

これはまさにデータ依存(コンパイル時依存)を AI に教えているところ。並列にしてもコスト的に得しない領域。

### 2. 副作用が排他(同じブランチ、同じテーブル、同じチケットに書き込む)

並列で走らせた 2 本が同じ Git ブランチに push しようとすると、後発のほうがコンフリクトで落ちる。同じ Notion ページのステータスを 2 本が同時に書き換えると、片方が上書きで消える。

Purple のワークフローエンジンは、各ステップが書き込みを行う「対象」を context にトラッキングしている。たとえば `branch_name` / `feature_branch` / `current_step_type` などの key 経由で、後続ステップが安全に再開できるようになっている。

```
# infrastructure/workflow/engine.py 抜粋
run.context["current_step_type"] = step.type
# ...
created_branches_str = run.context.get("created_branches") or ""
branch_name = run.context.get("branch_name") or run.context.get("feature_branch", "")
changed_repos_str = run.context.get("changed_repos") or ""
changed_repos: set[str] = {r.strip() for r in changed_repos_str.split(",") if r.strip()}
```

副作用が同じリソースに当たる以上、並列化の選択肢自体を消す。グラフ生成の段階で「同じリポジトリに対する書き込みは順序」と決めておく。

### 3. ロールバックコストが高い

これは順序か並列かというより「並列の安全マージン」に関わる。並列実行では「失敗した 1 本を巻き戻すコスト」が壁時計時間と推論コストの両方を食う。具体的には:

* ロールバックが軽い(branch を消すだけ、DB 書き込みなし) → 並列にしてよい
* ロールバックが重い(マージ済みコミットを revert する、外部 API call の補償が必要) → 順序にして失敗を早く検知する

Purple では PR を merge する直前まではロールバックが軽いので、開発フェーズは並列で開く余地がある。一方、`merge_pr` ステップ以降はロールバックコストが跳ね上がるので、ここから先は完全に直列。エンジン側でも `merge_pr` の outcome に応じて分岐し、コンフリクトがあれば `check_conflicts` に戻る、という非対称な設計になっている。

```
{
  "id": "merge_to_develop",
  "type": "merge_pr",
  "config": {"merge_method": "squash"},
  "on_success": "check_deploy_develop",
  "on_failure": "check_conflicts_develop",
  "on_conflict": "check_conflicts_develop"
}
```

### 4. 推論コスト × 平均失敗率

ここが一番見落としやすい。

並列で n 本走らせるとき、それぞれの 1 回あたりの成功確率を p とすると、全部成功する確率は p^n。p=0.7 で n=4 なら全成功率は 0.24 しかなく、「どれか 1 本は壊れて再実行」が常態化する。再実行した本は再び p=0.7 で、最悪のケースでは何度も繰り返す。

つまり、**平均失敗率の高いステップを横に広げると、期待 retry 回数が指数的に膨らむ**。具体的に何が「失敗率が高い」かというと、コードベース内の文脈をたくさん必要とするステップ。たとえばドメインモデルの新規設計や API 設計のような「先に決めたほうが後段が楽になる」タイプの作業。これらは 1 本ずつ直列に通して、出力を context として後段に渡したほうが、トータルの推論回数が減る。

逆に「ローカルテストを各リポジトリで走らせる」みたいな失敗率の低い独立な作業は、並列で開いて損がない。Purple では `run_tests` 系は基本独立なので、複数リポジトリに対して並列ワーカーで動かす設計にしている。

## グラフが動き始める瞬間

タスクグラフ自体は静的に DB に積まれるだけで、これを「動かす」のは別の Celery beat。30 秒ごとに `auto_start_pending_tasks` が走り、依存関係が全部解決済みで pending のタスクを拾って起動する。

```
# infrastructure/tasks/auto_start_pending_tasks.py 抜粋
async def _auto_start() -> Dict[str, Any]:
    async with managed_session() as session:
        run_repo = WorkflowRunRepository(session)
        task_repo = PurpleTaskRepository(session)

        active_runs = await run_repo.list_active()
        active_count = len(active_runs)

        max_concurrent = await _get_max_concurrent(session)

        if active_count >= max_concurrent:
            return {"started": 0, "skipped_capacity": 0, "at_limit": True}

        budget = max_concurrent - active_count
        candidates = await task_repo.list_pending_auto_start(limit=budget)

    for task in candidates:
        await _start_one_task(task.task_id)
```

ここで `budget` が並列度の上限。単純な FIFO ではなく、**capacity に空きがある分だけ並列に始動する**。

そして `list_pending_auto_start` の中身が興味深くて、SQL レベルで「未完了の依存先があるタスクは絶対に拾わない」という防御が入っている。

```
# infrastructure/repositories/purple_task_repository.py 抜粋
dep_task_alias = aliased(PurpleTaskModel)
unmet_dep_exists = (
    select(PurpleTaskDependencyModel.task_id)
    .join(
        dep_task_alias,
        PurpleTaskDependencyModel.depends_on_task_id == dep_task_alias.task_id,
    )
    .where(
        PurpleTaskDependencyModel.task_id == PurpleTaskModel.task_id,
        dep_task_alias.status.not_in(
            [PurpleTask.STATUS_DONE, PurpleTask.STATUS_CANCELLED]
        ),
    )
    .correlate(PurpleTaskModel)
    .exists()
)
stmt = (
    select(PurpleTaskModel)
    .where(
        PurpleTaskModel.status == PurpleTask.STATUS_PENDING,
        PurpleTaskModel.auto_start.is_(True),
        PurpleTaskModel.workflow_id.is_not(None),
        not_(unmet_dep_exists),
    )
    .order_by(PurpleTaskModel.priority.asc(), PurpleTaskModel.created_at.asc())
    .limit(limit)
)
```

これが効いているのは、**アプリケーション層の整合性チェックが race window を取りこぼした場合の最後の壁** になっているから。タスクが unblock される瞬間と auto\_start beat が回る瞬間の間に何かおかしなことが起きても、SQL の `EXISTS` 句で「依存先が done でも cancelled でもない」タスクは弾かれる。

## 失敗の伝播はもう一段下

逆に、依存先が `failed` した場合は何が起きるか。これも順序設計の重要な一部。

```
# use_cases/purple_task/purple_task_use_case.py 抜粋
async def fail_task(self, task_id: UUID) -> Optional[PurpleTask]:
    task = await self._repository.get_by_id(task_id)
    task.fail()
    failed = await self._repository.update(task)
    # Cascade failure to blocked tasks that depended on this
    await self._cascade_failure_to_dependents(task_id)
    return failed

async def _cascade_failure_to_dependents(self, failed_task_id: UUID) -> None:
    blocked_tasks = await self._repository.list_blocked_tasks_depending_on(failed_task_id)
    for blocked_task in blocked_tasks:
        blocked_task.fail()
        await self._repository.update(blocked_task)
```

依存先が落ちた時点で blocked のままだった後続は、自動で `failed` にされる。これは「並列に開いていたほうがリトライ無駄が増える」を実装で受け止める部分で、**順序エッジは「待つ」だけでなく「失敗を伝播する」セマンティクスも持つ**。並列ブランチではこの伝播がない代わりに、片方が落ちても他方が走り続ける(独立性の代償として)。

順序エッジを張った瞬間、片方が壊れたら無条件で後段全部やり直しになる。並列にしていれば壊れた本だけ再実行できる。この対称性を意識すると、「依存があると思って順序で組んでいたが、実は出力を参照してないから並列でよかった」みたいな間違いが減る。

## ハマりどころ: Celery group / chord / chain は使わない選択

Celery には `group` / `chord` / `chain` という並列・順序合成のプリミティブがあって、最初は使うつもりだった。

ただ、実装を進めるうちに気づいたのは、**workflow の状態が Celery 側の構造体に閉じ込められると、外部から「いま何処まで進んだ」が見えにくい**ということ。

* `chord` の中で 1 本失敗したらどうリカバリするか、構造体外側の状態(DB の WorkflowRun)と整合を取るのが面倒
* `chain` の途中ステップを human approval で停めて再開する、というのは Celery の責務外
* そもそも step graph の定義は JSON プリセットで Web UI から編集したい(workflow\_registry にロードする)

結局、**Celery は「次の step を 1 つ enqueue する」だけのキュー** として使い、順序や並列の制御はすべて DB の状態(`PurpleTask.status`, `WorkflowRun.current_step_id`, `dependency_ids`)で表現する設計に落ち着いた。

```
# infrastructure/workflow/engine.py 抜粋
# Pending Celery dispatches collected during step execution.
# Dispatched by the caller (workflow_executor) AFTER session.commit()
# to prevent a race where the next task reads from DB before the
# context update is committed (READ COMMITTED isolation).
self._pending_dispatches: list[tuple] = []
```

ステップ実行中に「次に何を enqueue すべきか」を集めておいて、DB commit が成功した後で Celery に投げる。これで「Celery は走り出したけど DB はまだ前の状態」という race を消している。

ここを `chord` や `chain` でやろうとすると、Celery の broker レイヤと DB の transaction レイヤを跨いだ整合をどう保つかという問題になって、自分には扱いきれなかった。

## 並列にできる箇所は思ったより少ない、という体感

実際にこの設計で動かしてみての体感だが、**1 つの PRD から作られたサブタスク群を見ると、並列にしていいエッジは思ったより少ない**。

PRD ベースの開発タスクは、ほぼ必ずどこかで「先に作ったエンティティを使う」「先に書いたエンドポイントを叩く」という構造を持つ。AI に「並列にできる sub\_task は dependency\_indices を空にしていい」と伝えても、実際に空のまま返ってくるのは全体の 2-3 割という印象。残りはチェーン状のグラフになる。

逆に、Codens の他サービス間で見ると並列度がはっきり違う。Green → Purple → Blue は順序設計(PRD が先、実装、QA の順、各段が前段の出力を読む)で、これを並列化する旨味はない。一方、Red(エラー検知 → 自動修正)と Yellow(活動集計)は前段に依存しないので、Purple の本流とは別ワーカーで並列に動かしている。

「いつ並列にしていいか」をワークフロー全体で 1 つのルールに統一しようとすると破綻する。**サービスのレイヤごとに違う基準で線を引いたほうが安定する**、というのが運用してみての感触。

## まとめ

PRD をタスクグラフに parse する設計と、ノード間のエッジを順序にするか並列にするかの判断軸を書いた。判断基準は (1) データ依存があるか (2) 副作用が同じリソースに当たるか (3) ロールバックコスト (4) 失敗率 × 期待 retry 回数、の 4 つで、迷ったら順序を選ぶ。並列は「失敗確率が低くて、互いに独立で、ロールバックが軽い」3 条件が揃って初めて選ぶ。逆に言うと、その条件を満たすステップは少ない。直列で組んでおいて、ボトルネックが見えてから個別に並列化していくほうが、結果として早かった。
