## 「AIコーディングの本質」という評価は妥当？

かなり妥当です。ただし、正確には\*\*“AIの思考を覗く”というより、“実装中に発生した判断を成果物として記録させる”テクニック\*\*です。

Anthropic の Claude Code ベストプラクティスでは、Claude Code はファイルを読み、コマンドを実行し、変更を加えながら自律的に作業するエージェント型コーディング環境だと説明されています。そのうえで、効果的な使い方として「探索 → 計画 → 実装」を分けること、曖昧な依頼ではなく具体的な文脈や制約を与えること、検証方法を与えることが推奨されています。([Anthropic](https://www.anthropic.com/engineering/claude-code-best-practices?_bhlid=6d35c25e04616eabb4469b78a6e8848b61863ba0))

この implementation-notes プロンプトは、まさにその延長です。仕様書に書かれていない判断、仕様からの逸脱、トレードオフ、未確認事項をファイルに残させることで、あとから人間がレビューしやすくなります。

## 公式ドキュメントとの整合性

Anthropic のドキュメントでは、CLAUDE.md のようなファイルを使って、プロジェクトのルール、設計方針、よくある注意点などを Claude に継続的に渡すことが推奨されています。CLAUDE.md はセッション開始時に読み込まれる「永続的な文脈」として扱われます。([Claude API Docs](https://docs.anthropic.com/en/docs/claude-code/memory))

また、Claude Code のベストプラクティスでは、CLAUDE.md に「アーキテクチャ上の判断」「非自明な挙動」「テスト方法」「ワークフロー規則」などを書くのがよいとされています。([Anthropic](https://www.anthropic.com/engineering/claude-code-best-practices?_bhlid=6d35c25e04616eabb4469b78a6e8848b61863ba0))

つまり今回の implementation-notes は、CLAUDE.md が「事前に渡す記憶」だとすると、**実装中に生まれた“差分の記憶”を残すもの**です。これはかなり筋がいいです。

## ただし注意点

SNS投稿の「AIが作業中に何を考えたかまで記録させる」という言い方は、少し危ういです。LLMの説明や推論ログは、必ずしも本当の内部思考を忠実に表しているとは限りません。研究でも、Chain-of-Thought 形式の説明が実際の判断理由を正確に反映しない場合があると指摘されています。([arXiv](https://arxiv.org/abs/2305.04388?utm_source=chatgpt.com))

なので、このプロンプトで残せるのは「AIの本当の内心」ではなく、**実装時に外部化された判断メモ**と見るのが安全です。ここ大事。魔法の水晶玉ではなく、作業日誌です。
