## このアプリ、私はコードを 1 行も書いていません

先日 [写経Programming](https://lp.shakyo-programming.dev) という Web サービスをリリースしました。私にとって **初めての Web アプリリリース** です。そして、**コードは 1 行も自分で書いていません**。

* アプリケーション(Lambda の Go コード)
* Terraform で組んだ AWS インフラ一式
* React + TypeScript のフロントエンド
* テストコード

ぜんぶ Claude Code が書きました。

**人間がやったのは、成果物の確認だけ**です。動かしてみて、違和感があれば「ここおかしい」「ここ動かない」と Claude に投げて、直してもらって、また動かす。その繰り返しでサービスが立ち上がりました。エディタを開いて自分で 1 文字でもコードを書く、ということを今回はやっていません。

技術選定とアーキテクチャの判断だけは自分で握っています。「Lambda を責務ごとに分ける」「マッチングは SQS FIFO」「レーティングは Glicko-2」。このあたりの方針決めですね。**設計の意思決定は人間、実装はぜんぶ Claude、検収もまた人間** ― これが今回の分業です。

Claude Code を契約していなければ、このサービスは間違いなく私のローカルで死んでいました。**Claude が書いてくれなかったら、私はこの記事を書けていません**。それくらい依存して作りました。

「AI で全部作れました」という綺麗な話ではないです。**人間 1 人 + Claude Code の二人三脚**で、何をどこまで分担したら個人開発が成立するか。その実験の記録だと思って読んでもらえると、書く側として一番嬉しいです。

リリースしたばかりで、いまはまだ静かなサービスです。読んでくれた **あなたが、最初の一人** になってくれるかもしれません。気が向いたら、ぜひ一度のぞいてみてください。**お待ちしています**。

## Claude Code に書かせた軌跡

今回やったのは、**設計の地図を引くこと**と、**Claude が間違った道に入ったとき方向を直すこと**。それだけです。何が動いて、何が動かなかったか、順に書きます。

### 全体図

ざっくりこんな感じ。

フロントは React + TypeScript の SPA(PWA)、バックエンドは **Go × 5 つの Lambda** です。EC2 も ECS も使っていません。「個人開発で固定費を月いくら払うか」が常に頭にあったので、極力サーバレスに寄せました。

**この構成図だけは自分で組みました**。そして、**ここから先のコードは、すべて Claude Code が書いています**。「Lambda を責務ごとに分ける」「マッチングは SQS FIFO」「レーティングは Glicko-2」。こういう設計の地図を人間が引いて、地図の上のコードを書く作業は Claude に丸投げ。

そんな分業の中身を、ここから具体的に書きます。取り上げるのは、**一番厄介だったマッチング** と **数式の山だった Glicko-2** の 2 つです。

### マッチングを SQS FIFO で書いた話(いちばん書きたかったやつ)

これがこの記事で**一番書きたかった**ところです。

要件は単純そうに見えて、書いてみると地味に厄介でした。

* 同じくらいのレート帯の人を**先着順**でペアリングしたい
* 同じ人を**二度マッチさせたくない**
* 大量に人が並んでも壊れないでほしい

最初は DynamoDB にキューを持って、cron で回す Lambda が 2 人 pop してマッチさせる、という素朴版を書いてもらいました。**ここの初版も Claude Code に丸投げ**です。プロンプトは「マッチング待機列を DynamoDB で表現して、cron Lambda で 2 人ずつ pop してマッチングさせて」くらいのざっくりさ。数分後には動くコードが返ってきました。ローカルでは普通に動きます。

問題はそこから先でした。**並列で複数の Lambda が同じキューを舐めると、同じ人を 2 つの部屋にマッチさせてしまう**バグが速攻で出たんです。条件付き書き込みで耐える方向の修正も、Claude に投げて何往復かやり取りしました。確かに動くようにはなる。けどコードがどんどん重くなる。ここで **「いや、これ設計を変えた方が早いな」** と判断して、Claude に違う方針を伝え直しました。

そこで諦めて **SQS FIFO** に乗り換えました。鍵は `MessageGroupId` に**レート帯の bucket(例: `rb-1500`)を入れる**こと。

* 同じ bucket の中では順序保証 + 重複なし
* 違う bucket 同士は別グループとして並列処理される

これで「先着順」「重複しない」「レート帯ごとに並列」が AWS 側に丸投げできます。マッチング処理 Lambda の本体は驚くほど薄くて、ハンドラはこれだけ。

```
func handle(ctx context.Context, ev events.SQSEvent) error {
    for _, r := range ev.Records {
        if err := matcher.ProcessBucket(ctx, r.Body); err != nil {
            log.Printf("process bucket %s: %v", r.Body, err)
            return err
        }
    }
    return nil
}
```

`ProcessBucket` の中で、その bucket のキュー(DynamoDB)から待っている 2 人を pop してルームを作って、両者の WebSocket コネクションに `match.found` を `apigatewaymanagementapi` 経由で push します。

**設計が決まってから先のコードは、また Claude に書かせました**。これがびっくりするほど素直でした。上の handler も `ProcessBucket` も、出てきたコードがそのまま本番に乗っています。設計が間違っていると Claude をどれだけ回しても抜けられず、設計が正しいと一発で書いてくれる。**この章は、その対比を一番強く感じた場面**でした。

書き終わってみて、**「分散の難しいやつは、できるだけ AWS のマネージドの特性に押し付ける」が正解**だなと改めて思いました。FIFO の制約(MessageGroupId 単位の順序保証)を「不便」と捉えるか「ちょうど欲しい挙動」と捉えるかで、設計の見え方がガラッと変わります。

### Glicko-2 を Claude に書かせた話

Glicko-2、Go で手頃なライブラリが見つかりませんでした。なら自分で書くか、と一瞬思ったんですが、**数式の山**を Go に翻訳する作業が、想像しただけでつらい。

ここはためらわず Claude Code に投げました。Glickman の論文 PDF([*Example of the Glicko-2 system*](https://www.glicko.net/glicko/glicko2.pdf))をそのまま渡して、

> この論文の Glicko-2 アルゴリズムを Go で実装して。step 1 から step 8 まで関数を分けて。テストも書いて。

くらいのプロンプト。**初版は数分で上がってきました**。

ただ、ここで素直に喜べないのが Glicko-2 の怖いところです。**Claude が書いてきたコードが本当に論文と一致しているかは、Claude 自身は保証してくれません**。動きはするんですよ。けど数値がほんのちょっとズレていても、私には気づけない。

救いは Glickman 本人が論文に**例題と期待値**を載せてくれていたこと。

> Example calculation:
>
> `1500/200` vs `1400/30 (W)`, `1550/100 (L)`, `1700/300 (L)`  
> → `1464.06 / 151.52 / 0.05999`
>
> *― Glickman, M. E. [*Example of the Glicko-2 system*](https://www.glicko.net/glicko/glicko2.pdf) より引用*

この例題値を Claude に渡して、「**この期待値どおりに動くか確認するテストも一緒に書いて**」と頼みました。テストもまた Claude が書きました。ここでやったのは、**「何を検証すべきか」を Claude に伝えること**。実装も検証コードも Claude、人間は『正しさの基準』を指定する側、という分業です。今回の中で一番健全な形だったと思います。

**「数式が複雑なやつほど、論文の例題で検証コードを 1 本でも持っておく」**。AI に書かせる時代でも(むしろ時代だからこそ)効くプラクティスだと思っています。

## 対戦中に相手が回線を切ったときの挙動

これがいちばん泥臭くて、いちばん体験に効きます。

「ブラウザを閉じた」「電車に乗ってトンネルに入った」「ブラウザバックした」あたりを愚直に拾わないと、相手側で永遠に待たされるか、勝敗が確定せずに部屋が宙ぶらりんになります。今は接続断を検知したらオーバーレイで明示的に出して、退室時にサーバ側でルームを片付けるところまで作っています。

ここはまだ細かい改善が残っていて、独立した記事にしたいくらいネタがあります。たぶん書きます。
