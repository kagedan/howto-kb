## はじめに

こんにちは。クラシルでレシチャレのAndroidエンジニアをしているkenzoです。  
自分たちのチームではプロジェクトの設計パターンを体現した「お手本画面」をデバッグメニュー内に作り、AIに参照させるようにしました。体感として手直しが減ったのはもちろん、小規模な比較検証でも、確認した10観点のうち7つで生成コードの品質向上が見られました。

![](https://static.zenn.studio/user-upload/231c71783af4-20260406.png)

きっかけは、Android開発にClaude Codeを導入したものの、出てきたコードへの手直しが多かったことです。たとえばViewModelのエラーハンドリングです。プロジェクトの命名規則に従わず独自の名前でハンドラーを作り、エラーメッセージを日本語でハードコードしてきました。Repositoryでは不必要にコルーチンをラップした実装が出てきたこともありました。どちらも「動くけど、うちのプロジェクトではそう書かない」というタイプの問題です。

Claude Codeではプロジェクトルートに置いたCLAUDE.mdや`.claude/rules/`配下のルールファイルを読み込ませることで、AIにコーディング規約を伝えられます。自分たちもそこに設計方針を言語化していましたが、細かい命名や共通処理の使い分けまでテキストで伝えきるのは難しく、「それらしいが微妙に違う」実装がどうしても出てきました。そこで試したのが「お手本画面」でした。

## お手本画面とは

お手本画面（BestPractice画面、以下BP）は、架空の商品詳細画面を題材にしたプロジェクトの模範実装です。

なぜ既存のプロダクトコードではなく、専用のお手本を作ったのか。

* **新メンバーのオンボーディングにも使える。** AIだけでなく、新たに参画したメンバーもここを見れば「このプロジェクトではこう書く」がわかります
* **プロジェクトに複数の書き方が混在している。** 方針が変わった経緯で古い書き方と新しい書き方が共存していると、AIも新メンバーも何が正解かわかりません。お手本を1つ決めることで迷いをなくせます
* **都合のいいリファレンスがあるとは限らない。** プロジェクトの様々な要素がバランスよく入った画面は、そもそも存在しないことが多いです。あったとしても独自の事情やノイズが混ざっています
* **デバッグ画面なら安全に更新できる。** 新しい書き方を導入する際、プロダクト画面をわざわざ書き換えるのはリスクがあります。デバッグ画面であればたとえ壊れても影響は小さいです

API定義・Repository・UseCase・レスポンス変換・DI構成といったdata層から、ViewModel・Screen・Mapperなどのui層、そしてドメインモデルまで、一つの機能に必要な要素を一通りカバーしています。

デバッグメニューの中に配置しており、Androidのビルドバリアント機能を使ってreleaseビルドでは何もしない実装（no-op）に差し替えています。そのため本番の動作には影響しません。モックを差し込むために専用のRetrofitインスタンスを作るなど、デバッグ用に割り切っている箇所もあります。

## 何を盛り込んだか

お手本に盛り込んだのは、プロジェクトで共通して使っていて、開発中によく出てくる書き方です。

たとえば以下のようなパターンを含めています。

* **ドメインモデルからUIモデルへの変換**: ドメインモデルをそのままUIに渡さず、Mapperを通して表示用のモデルに変換する。価格のフォーマットやラベルの解決はMapper側で行う
* **エラーハンドリングの使い分け**: 致命的なエラーは全画面エラー、非致命的なエラーはSnackbarで通知、というように重大度で表示方法を切り替える
* **一度きりのイベント処理**: 画面遷移やSnackbar表示のように一度だけ実行したいイベント（状態として保持すべきでないもの）の定義パターン
* **Previewのバリエーション**: ハッピーパスだけでなく、状態の異なるPreviewを複数作成する

こうしたパターンを実際のコードで示しています。

## どう参照させているか

実装時にBPを参照するよう、CLAUDE.mdと`.claude/rules/`配下のレイヤー別ルールファイルの両方に指示を入れています。

CLAUDE.mdにはBPの場所と「実装時は参照すること」という全体的な指示を書いています。加えて、`data-layer.md` や `ui-layer.md` といったレイヤー別ルールファイルにも、該当レイヤーの実装タスクが始まる前にBPを参照するよう指示を入れています。これにより、タスクの種類に応じて必要なBPのファイルを読み込ませる運用にしています。

## 検証: BestPracticeあり vs なしの比較

BPを導入してしばらく運用する中で、プロジェクトの書き方に沿ったコードを書いてくれるようになった体感がありました。実際にどの程度効果があるのか、同一条件で比較検証を行いました。

### 検証方法

架空の「ショップ詳細画面」（詳細表示 + お気に入り登録/解除、API 3本、全レイヤー新規作成）を題材に、以下の2パターンで実装を生成しました。

* **パターンA（BPなし）**: CLAUDE.mdと`.claude/rules/`からBestPractice参照を除去した状態
* **パターンB（BPあり）**: プロジェクト標準の設定のまま

同一の仕様書・プロンプトを使用し、生成結果を複数の観点で比較しました。

### 結果

| 観点 | BPなし | BPあり | 結果 |
| --- | --- | --- | --- |
| レスポンス変換 | 必須フィールドの変換処理に誤り | 正しいnullチェックで処理 | **効果あり** |
| Repository | 不要なコルーチンラッパーの誤用 | プロジェクト標準の共通処理で統一 | **効果あり** |
| ViewModel | 独自命名・ハードコード文字列 | BP命名準拠・共通エラー処理を活用 | **効果あり** |
| ワンショットイベント | `sealed class`・共通パターン未活用 | `sealed interface`・共通エラー表示に委譲 | **効果あり** |
| Screen | フラット展開・ハードコード | UiState分離・stringResource | **効果あり** |
| Preview | 1バリエーションのみ | 3バリエーション | **効果あり** |
| エラーハンドリング | 基本分離のみ | APIエラーメッセージの抽出・冪等性担保 | **効果あり** |
| Responseモデル | 規約通り | 規約通り | 差異なし |
| DI構成 | 正しい構成 | 正しい構成 | 差異なし |
| UseCase | 共通処理の部分使用 | 設計整合性あり | 微差 |

比較した観点のうち、7つで品質向上が見られました。題材やAIへの指示の単位次第で結果は変わる可能性があります。

### 特に効果が大きかった領域

**UI層の設計品質**がもっとも差が出ました。UiState分離、stringResource活用、Previewの充実は、ルールファイルだけでは「具体的にどう書くか」が伝わりにくい領域です。お手本があることで、実装の構造がプロジェクトの規約に沿った形になっていました。

**プロジェクト固有のユーティリティ活用**も顕著でした。たとえばViewModelのエラーハンドリングでは、以下のような差が出ています。

BPなしの場合、独自命名のハンドラーにハードコード文字列が使われています。

```
// BPなし: 独自命名、エラーメッセージをハードコード
private val favoriteExceptionHandler = CoroutineExceptionHandler { _, _ ->
    viewModelScope.launch {
        sideEffect.send(SideEffect.ShowSnackbar("エラーが発生しました"))
    }
}
```

BPありの場合、プロジェクト共通の命名規則に従い、APIエラーからメッセージを抽出しています。

```
// BPあり: 共通の命名規則、APIエラーメッセージを活用
private val snackbarExceptionHandler = CoroutineExceptionHandler { _, throwable ->
    val message = (throwable as? ApiException)?.errorMessage
        ?: getString(R.string.common_error_message)
    viewModelScope.launch {
        sideEffect.send(SideEffect.ShowSnackbar(message))
    }
}
```

※実際のコードを簡略化した例です。

コードベースに既にある共通パターンが使われるようになったのが大きな変化です。

**Repository/UseCaseの設計整合性**では、BPなしで発生した不要なコルーチンラッパーのような「それらしいが誤った実装」が防止されました。

一方で、差異が出なかったResponseモデルやDI構成は、`.claude/rules/`のルールファイルだけで十分にガイドできる領域でした。

### コストはどうか

トークン消費量はBPありの方が約24%増加しました。主因はBPのファイル読み込みです。

ただし、BPなしのコードを後からBPありレベルに修正すると15ファイル・+328行/-268行の変更が必要でした。この1タスクに関しては、BPありの24%のトークン増加分よりも、後から規約に合わせて修正する手間やそれにかかるトークンのほうが大きいと判断しています。最初からプロジェクト規約に沿った形で生成させる方がトータルで有利でした。

## 逆にBestPracticeが裏目に出たケース

BPありが常に優れていたわけではありません。BPと微妙に異なる要件では、お手本に引っ張られて誤った実装が生成されたケースもありました。

* BPの詳細画面が`Column + verticalScroll`だったため、`LazyColumn`を使うべき場面でも`Column`が採用された
* `ImmutableList`を使うルールがあるのに、`List`で生成された
* BPでは本文中にボタンを配置していたため、仕様上`bottomBar`に固定すべきボタンもスクロール領域内に置かれた

いずれもBPの「商品詳細画面」と、今回の「ショップ詳細画面」でUI要件が微妙に異なることが原因です。これらのケースでは、お手本がないぶん、BPなしの方がルールファイルと仕様書に素直に従った判断になっていたと考えています。

つまり、BPは「ルールに書きにくい暗黙知」を伝えるのに強いですが、お手本と微妙に異なる要件では逆に足を引っ張ることもあります。現時点ではまだ特定の機能をカバーしているだけなので、今後パターンや要素を増やしてお手本と仕様の差異を減らしていく予定です。

## まとめ

お手本を作ってAIに参照させるのは、「うちのプロジェクトではこう書く」を伝える手段として有効でした。特にテキストのルールだけでは伝わりにくいUI層の設計パターンで効果を実感しています。
