Claude Code で、毎朝のニュースを構造化して、個人投資の「攻め」のエンジンに供給するパイプラインを作っています。ただ、この連携には順番があるので、そこを先に正直に書いておきます。

もともと私は、投資とは関係なく、自分用のニュースアプリを先に作っていました。世の中のニュースは、自分の欲しい情報を毎回自分で選り分けないと読めません。その手間から解放されたくて、自分が追いたいテーマだけを毎朝集め、欲しいインサイトの形にして届く仕組みを、自分のために作りました。とくに育休中は、それまで会社の同僚から自然に入ってきていた最新の技術動向が、入らなくなります。置いていかれたくない、という動機が大きかったです。

そのあとに、投資のツールを作りました。そこで、「毎朝ニュースを構造化して集める仕組みがもうあるなら、投資の判断にも活かせるのでは」と思い、この2つをパイプラインでつないだ——というのが、この記事で書く連携です。だから、投資のために作ったニュースアプリではありません。もとからあった自分用のニュース処理層に、あとから投資エンジンを「消費先」としてつないだ、という順番です。

そのうえで、いちばん引っかかりそうなところを潰しておきます。前の記事で私は、「個人のボトルネックは発見ではなく握力で、エンジンの edge はニュースの速さには置いていない」と書きました。さらに、ニュースを入れるとモデルの判断が楽観方向にバイアスする、という研究（[arXiv 2411.00856](https://arxiv.org/abs/2411.00856)）も引きました。それなのに、なぜ毎朝ニュースを処理する仕組みをわざわざ作るのか。

答えを先に言うと、このパイプラインは**新しい銘柄を速く見つけるためではなく、すでに賭けた仮説が「まだ生きているか」を毎朝見張るため**にあります。ニュースを買う理由として使うのではなく、仮説が崩れる兆しを拾う一次入力として使う。同じニュースでも、向ける方向が正反対です。

> 供給先である「攻め」のエンジンの設計はこちら：[Claude Code で個人投資の「攻め」エンジンを作る — 構造変化の勝者を、予測せず握る](https://zenn.dev/takotyan00/articles/claude-code-structural-change-engine)

## なぜ「速く読む」ためではないのか

前の記事で、構造的な勝者は「芽 → 初期の数字 → コンセンサス」の3段階を通り、個人が負けるのは「見つけられない」からではなく「見つけても握り続けられない」からだ、と書きました。その握力を支えるのが、賭けた仮説が崩れていないかを見張り続ける反証トラッキングです。

問題は、この反証トラッキングも、その手前の先行指標の監視も、**放っておくと止まる**ということです。一度「この会社は構造変化 X の基盤を握る」という仮説を立てたら、その前提が生きているかどうかは、毎日入ってくる出来事の中に少しずつ現れます。設備投資のガイダンスが反転した、産業を動かす政策が変わった、想定していた採用の波が引いた——こうした兆しは、決算を待たずにニュースの中を流れていきます。

だから、毎朝ニュースを処理する目的は「誰も知らない銘柄を先に知る」ことではありません。それは前の記事で書いたとおり、もう誰にでもできる浅い発見です。目的は、すでに握っている仮説の前提が、今日も生きているかを確かめ続けること。ここでも効かせているのは速さではなく、継続のほうです。

| ニュースの読み方 | 何を探すか | 何に使うか |
| --- | --- | --- |
| 速報で買う（やらない） | 上がりそうな材料 | 新規の買い判断 |
| 反証で握る（やる） | 仮説が崩れる兆し | 保有の握力の維持 |

上の表の「やらない」ほうを捨てて、「やる」ほうにパイプライン全体を寄せています。これが、ニュースを扱うのに「速報性」を売りにしない理由です。

## パイプラインの全体像

具体的な流れは、こうなっています。前半（①〜③）が汎用のニュース処理層、後半（④以降）が投資エンジン側です。

ポイントは、**ニュースを処理する層と、それを投資判断に使う層を、はっきり分けている**ことです。前半はニュースを構造化するだけで、どの銘柄を握っているかも、どの仮説を賭けているかも知りません。後半のエンジンだけが、自分の保有仮説と突き合わせて意味を判定します。この分け方の理由は、あとで書きます。

## 一次要約は安いモデル、解釈は Claude

ニュースを畳んで構造化するところと、その先で仮説と突き合わせて判定するところとで、あえてモデルを分けています。

大量のニュースをざっと畳んで構造的な事実だけを残す前半（②〜③）は、Gemini Flash 級の安い軽量モデルに任せます。ここは処理量が多く、賢さより安さとスループットが要る領域です。一方で、その先でエンジンが下す「これは崩れる兆しか」という判定だけを、Claude に渡します。

これは、[投資スタックの三層設計の記事](https://zenn.dev/takotyan00/articles/claude-code-investing-ai-stack)で「守り」について書いた、**どこに知能を効かせて、どこをあえて機械的に固めるか**という発想と同じです。収集や保存はコードがやり、大量処理は安いモデルがやり、判断に効く一手だけ高いモデルを使う。全部を一番賢いモデルに通すのは、コストの面でも精度の面でも得ではありませんでした。

## 「構造化シグナル」とは何か

③の抽出が、このパイプラインの地味な要です。

ニュースは散文です。「A 社が新工場の建設を発表」という文章のままでは、機械が仮説と突き合わせられません。そこで、こういう構造に落とします。

* **どの構造変化テーマに属すか**（例：AI 向けの電力・データセンター増強）
* **影響の向き**（追い風か、逆風か）
* **どの先行指標に効くか**（設備投資ガイダンス、政策、採用、arXiv・GitHub での採用の広がり…）
* **出所**（あとで人間が一次ソースに戻れるように）

たとえば1件のニュースからは、こんな形のシグナルが1つ落ちます。

```
# 構造化シグナルの例（1 件のニュースから抽出した形）
テーマ: "AI データセンターの電力逼迫"
影響の向き: 追い風          # 追い風 / 逆風 の2値だけ
効く先行指標: 設備投資ガイダンス
出所: "<一次ソースの URL>"
# 「株価が上がる／下がる」は、ここにわざと入れない
```

ここで意識して外しているのが、「この会社の株は上がるか／下がるか」という判定です。抽出の段では、それは一切やらせません。抽出がやるのは、**株価に触れない構造的な事実を、型のあるシグナルに変えることだけ**です。株価にとって良い/悪いの解釈は、そのあとエンジン側が、自分の仮説マップに照らして初めて下します。

なぜ抽出の段で株価判定をさせないのか。ここが、冒頭で触れた「ニュースはモデルを楽観方向にバイアスする」への対処です。ニュースを見せて「良い材料か？」と聞くと、モデルはポジティブに寄せがちになる。だから、そもそも良し悪しを聞かない。事実だけを構造化させ、良し悪しの判定を後段の別ロジックに分離することで、楽観バイアスが混じる入口を1つ塞いでいます。

## ニュースを「反証の一次入力」にする

前の記事でいちばん独自だと書いたのが、反省の教師信号を株価ではなく構造の推移に置いた反証ループでした。このパイプラインは、その反証ループに毎朝の燃料をくべる役です。

やっていることは、構造化シグナルを、保有中の仮説の「崩れる条件」に紐づけることです。エンジン側には、仮説ごとにこういうルールを先に書いてあります。

```
# 反証ルールの例（構造だけ。実際の保有とは無関係）
仮説: "この会社は構造変化 X の基盤（つるはし）を握る"
disconfirm_when:
  - 先行指標: 想定した設備投資ガイダンスの連鎖
    condition: 反転して縮小に転じる
  - 先行指標: 政策・制度（補助金・規制）
    condition: この構造変化に逆風の方向へ動く
action: 「仮説が崩れた候補」として人間に通知する（売買はしない）
```

毎朝のニュースから抽出したシグナルを、この `disconfirm_when` に照合します。当たったものだけを「仮説が崩れたかもしれない候補」として人間に上げる。逆に、追い風のニュースは、あってもわざわざ通知しません。良い知らせで買い増したくなるのは、たいてい高値掴みの入口だからです。

もちろん、これで捨てているものもあります。追い風のニュースには、仮説を正しく補強する（買い増しや確信を上げる）材料も混じっている。それを承知のうえで、握力の維持に振り切って、あえて拾わないことにしています。

これは、ニュースを主軸にしないという前の記事の方針と矛盾しません。仮説を作る主軸はあくまでファンダ（財務）で、ニュースはその仮説の生死を見張る一次入力に回している。むしろ、モデルが楽観に振れやすいニュースを、逆張りに——悪い兆しを探す側に——使っているところが肝です。

ただ、正直に穴も書いておきます。楽観バイアスの入口を塞いだといっても、**塞げたのは抽出の段だけです**。「崩れる兆しか、ただの雑音か」を判定する一手は、まだニュースを読むモデルに委ねている。同じ楽観バイアスがここで効けば、「崩れた」を見落とす方向に転びます。買い材料を過大評価するより、反証の見張りにとっては、こちらのほうがずっと危ないのです。

だからこの判定は、自由な良し悪しではなく、**あらかじめ機械的に書いた「崩れる条件」への照合に絞り、問いを「崩れたか」という否定方向だけに向けています**。それでも完全には信用しきれないので、最終判断は人間に残す。あとの「正直なところ」で反証ループをまだ手で回していると書くのは、この本丸の入口を、モデルに開けっ放しにしたくないからです。

## なぜ「毎朝」なのか

速さに edge を置かないと言いながら「毎朝」処理するのは、一見ちぐはぐに見えるかもしれません。でも、狙いが違います。

秒単位の速報を追うのは、コンセンサスが動く瞬間に反応する短期のゲームで、これは最初に降りると決めた土俵です。一方で「毎朝1回」は、数年握る構造仮説にとってちょうどいい粒度です。設備投資のガイダンスも政策も採用動向も、日をまたいで少しずつ動く。**リアルタイムである必要はないが、途切れさせてはいけない**——この監視の連続性を担保するのが日次のリズムです。

では、なぜ週次でなく毎朝なのか。ここは要件というより実務です。すでにある朝のニュースまとめの仕組みに相乗りできるので、日次にしても手間はほとんど増えません。逆に間隔を空けるほど、兆しを取りこぼす窓が開きます。日次は、監視を切らさずに手間も増やさない下限として選んでいます。

言い換えると、毎朝処理するのは「誰より速く知る」ためではなく、「握っている間、前提の崩れを見落とさない」ためです。速報性ではなく、監視が止まらないことに価値を置いています。

## なぜ二層に「疎結合」で分けたのか

さいごに、ニュース処理層とエンジンを分けた理由です。ここは、投資スタックの三層設計の記事と地続きです。

その三層設計の記事で、株価や財務の生データを取ってくる中立の市場データ基盤を1つ置き、その上に思想の違う「攻め」と「守り」を載せた、と書きました。ニュース処理層は、それの「ニュース版」です。ニュース処理層は、どの仮説を賭けているかを持ちません。ただ世の中の出来事を構造化して配るだけの、思想を持たない中立の供給源にしてあります。

| 層 | 役割 | 持つもの / 持たないもの |
| --- | --- | --- |
| ニュース処理層 | 出来事を構造化シグナルに変えて配る | 事実の構造化を持つ。仮説・保有は持たない |
| 攻めエンジン | シグナルを自分の仮説に照らして判定 | 仮説マップ・反証条件を持つ |

私がここで効かせているのは、**疎結合**という古い設計の原則です。関数でもモジュールでも、部品どうしは互いの中身に依存しすぎないほうがいい、と昔から言われてきました。それを、1つのアプリの中ではなく、自分が作ったアプリどうしの間でやっている、というだけの話です。ニュース処理層とエンジンは、片方が相手の内部を知らないまま、決まった形のシグナルだけをやり取りします。ちょうど API 越しに、あるいは MCP のように、疎につながっているわけです。

疎結合にしておく効きめは、3つあります。

1つは、**関心の置き場所が一箇所に決まる**こと。「何を追いたいか（＝どの仮説を握っているか）」はエンジン側だけが持ち、ニュース処理層には持たせない。両方に散らばると、更新するたびに二重管理になります。

もう1つは、**同じ処理層を別の消費先にも回せる**こと。じっさい、この処理層はもともと投資のためではなく、自分の関心を追う日々の情報整理のために作ったものでした。投資エンジンは、あとから足した2つ目の消費先にすぎません。中立の供給源にしておいたから、思想の違う消費先を後付けできた——これは机上の利点ではなく、実際にそうやって育った順番です。作るのは1つ、仕立てるのは各消費先、という形です。

そしてもう1つが、**片方が止まっても、もう片方が生き残る**こと。エンジンはニュース処理層を read-only で参照するだけで、そこに強くは依存していません。だからニュース処理層が落ちても、エンジンの芯——株価とは独立した財務の質でふるう部分——はそのまま動きます。ニュースは握力を支える補助の入力であって、無ければ何も回らない前提ではない。疎結合にしておくと、こういう「一部が壊れても全部は倒れない」余裕が生まれます。

個人開発のアプリを、ただ独立して並べるだけでなく、必要なときに API のようにゆるくつなぐ。そうやって、自分が作ってきたものが少しずつ有機的につながっていく——このニュースの連携も、その一つの実例です。

## Claude Code の使いどころ

このパイプラインで Claude Code と一緒にやっているのは、主に設計のほうです。

構造化シグナルのスキーマ（何を型として抜くか）を決めたり、「これは崩れる兆しか、ただの雑音か」の線引きを詰めたり。とくに気をつけているのは、前の記事の影響マップと同じで、**抽出モデルにそれらしい対応表を作らせない**ことです。放っておくと、AI は「このニュースならこの銘柄に逆風」というもっともらしい紐づけを、根拠が薄いまま埋めてきます。だから、紐づけの各段に「なぜそう言えるか」を要求し、根拠の弱いものは通さない。この歯止めの設計を、実際のコードと出力を読ませながら Claude Code と詰めています。

## 正直なところ

線を引いておきます。

まず、供給のパイプ自体は端から端まで通っていることを確認済みですが、**シグナルの精度と運用の詰めは、まだ検証の途中**です。とくに、ただの雑音を「崩れる兆し」と拾ってしまう誤検知の率や、同じ出来事の重複をどう均すかは、これから実データで詰めるところです。「毎朝きれいなシグナルが自動で流れて、握力が勝手に維持される」という完成品ではありません。反証の見直しループも、いまは私が手で回していて、全自動では走っていません。「こう設計して、こう作っている」という話です。

それから、これは自分用の意思決定支援に閉じています。個別銘柄の売買を勧めるものではありません。実際の保有銘柄・金額・口座は公開しません。この記事で共有できるのは、ニュースを構造化して反証に使う、という設計と考え方までです。

## なぜ、こう作るのか

最後に、この連携の背骨を。

いまは、ニュースを要約する AI も、投資判断を出す AI も、大量に用意されて手に入る時代です。だからこそ、みんなが同じニュースを同じ速度で読み、同じ結論に寄っていく。速く読むこと自体は、もう差になりません。

差がつくのは、その出来事を**どう構造化して、自分の仮説のどこに接続するか**です。中立の材料供給を1つ置き、各消費先が自分の要件で仕立てる——という形は、三層設計の記事で書いた市場データ基盤とまったく同じ骨格でした。そして、皆が速報で同じ方向へ群れるほど、「上がる理由」ではなく「崩れる兆し」を毎朝探しにいく側の目は、むしろ少なくなっていきます。私がこのパイプラインで賭けているのは、その少ないほうです。
