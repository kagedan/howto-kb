Claude Codeを使っていると、こうすればもっと便利になりそうという気づきが積み重なります。ただ、そのたびに作業を止めて設定ファイルを開くのは気が乗りません。

「気づいたら設定が整っている」という状態が欲しくて、試行錯誤しました。

## `/insights` だけでは物足りなかった

Claude Codeには`/insights`というビルトインコマンドがあります。過去のセッション履歴を分析して使い方のパターンや改善ポイントをレポートしてくれる機能です。有用なのですが、使っていて2点が気になりました。

**カスタマイズできない**: 提案内容や検出ロジックを自分で調整できないため、自分のワークフローに合った提案が得られません。（それが良さでもありますが。）

**面倒**: コマンドを実行し、それをブラウザで毎回読む必要があります。私が欲しいのはレポートではなく、気づいたら設定が整っている状態です。

そこで、`/insights`は補強材料として取り込みつつ、Claudeが裏で気づいたことを溜めておき、手が空いたときにまとめて反映できる仕組みを自分で作りました。

## 作ったもの

3つのパーツを組み合わせています。

1. 改善のヒントをログに出す
2. ステータスバーで保留件数を常時確認できる
3. `/recommendations` でまとめて反映する

## 1. 改善のヒントをログに出す

3種類の方法で改善のヒントを記録します。

### ヒント1: 承認したBashコマンド

Claudeに承認を求めたBashコマンドは、実行後に自動でログファイルへ記録されます。すでに `permissions.allow` に登録済みのコマンド（毎回確認不要なもの）はスキップし、「今回初めて承認した」コマンドだけが残ります。

PostToolUse hookを使って、Bashツールの実行完了後にスクリプトを呼び出しています（[参考: hooks でワークフローを自動化する](https://code.claude.com/docs/ja/hooks-guide)）。

log-bash-command.sh

~/.claude/scripts/log-bash-command.sh

```
#!/bin/bash
# PostToolUse hook: Bashコマンドをログに記録する
# permissions.allow に未登録のコマンドのみを記録する

LOG_FILE="$HOME/.claude/logs/bash-commands.log"
SETTINGS_FILE="$HOME/.claude/settings.json"
mkdir -p "$(dirname "$LOG_FILE")"

input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command // empty' 2>/dev/null)

if [ -z "$command" ]; then
  exit 0
fi

# 複数行コマンドは1行に圧縮して記録
single_line=$(echo "$command" | tr '\n' ' ' | sed 's/  */ /g' | sed 's/^ //;s/ $//')

# コマンドの先頭ワードを取得（例: "pnpm --filter api test" → "pnpm"）
first_word=$(echo "$single_line" | awk '{print $1}')

# settings.json の permissions.allow に先頭ワードがマッチするパターンがあればスキップ
if [ -f "$SETTINGS_FILE" ]; then
  already_allowed=$(jq -r '.permissions.allow // [] | .[]' "$SETTINGS_FILE" 2>/dev/null | grep -c "^Bash($first_word" || true)
  if [ "$already_allowed" -gt 0 ]; then
    exit 0
  fi
fi

echo "$(date '+%Y-%m-%d') $single_line" >> "$LOG_FILE"
```

### ヒント3: SessionEnd hookによるメモ

会話中に気づきを記録するルールを作るだけでは不十分でした。Claudeはタスクに集中しているとルールの適用を後回しにしがちで、セッションが終わると記憶が消えます。

そこで `SessionEnd` hookを使って、**セッション終了のタイミングでClaudeに振り返りを実行させる**仕組みを加えました。

`SessionEnd` hookのstdoutはClaudeのコンテキストに届きません。そのため `claude -p` でClaudeを明示的に起動します。このとき、hookのstdin JSONに含まれる `transcript_path` を使うのがポイントです。`transcript_path` にはセッション全体の会話ログ（JSONL形式）へのパスが入っており、これを読んで `claude -p` のプロンプトに渡すことで、セッション内容をもとにした振り返りが可能になります（[参考: Hooksリファレンス](https://code.claude.com/docs/ja/hooks)）。

無限ループの防止は環境変数で行います。`CLAUDE_IN_REFLECTION=1` を付けて `claude -p` を起動することで、その子プロセスのSessionEnd hookが発火した際に同じ変数を検出してスキップします。

SessionEndのデフォルトタイムアウトは非常に短いため、`CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS` も合わせて延ばしています。

session-end-reflection.sh

~/.claude/scripts/session-end-reflection.sh

```
#!/bin/bash
# session-end-reflection.sh — SessionEnd hook for Claude Code
#
# セッション終了時に会話ログ（transcript）を読み込み、claude -p に渡して
# pending_recommendations への追記が必要な気づきやコマンドを振り返らせる。
#
# 無限ループ防止:
#   CLAUDE_IN_REFLECTION が設定されている場合はスキップ。
#   claude -p 実行時に CLAUDE_IN_REFLECTION=1 を付与することで、
#   その claude プロセスの SessionEnd hook が再度このスクリプトを起動しない。

if [ -n "$CLAUDE_IN_REFLECTION" ]; then
  exit 0
fi

INPUT=$(cat)
TRANSCRIPT_PATH=$(echo "$INPUT" | jq -r '.transcript_path // empty' 2>/dev/null)

if [ -z "$TRANSCRIPT_PATH" ] || [ ! -f "$TRANSCRIPT_PATH" ]; then
  exit 0
fi

TRANSCRIPT_CONTENT=$(cat "$TRANSCRIPT_PATH")

CLAUDE_IN_REFLECTION=1 claude -p "以下はこのセッションの会話ログです（JSONL形式）。この内容をもとに、~/.claude/memory/pending_recommendations.md への追記が必要な気づきやコマンドがあれば追記してください。なければ何もしなくてOKです。

${TRANSCRIPT_CONTENT}"
```

## 2. ステータスバーで保留件数を常時確認できる

設定の改善案が溜まっていることに気づかなければ意味がありません。Claude Codeのステータスバーには任意のスクリプト出力を表示できるため、保留件数を常時表示しています。

![ステータスバーにレコメンド件数が表示されている様子](https://static.zenn.studio/user-upload/deployed-images/071aa04c57dc0e71dcffd11e.png?sha=e96ca2ec8f3c8aa36b8740928439bc5deb50ec50)

check-recommendations.sh

~/.claude/scripts/check-recommendations.sh

```
#!/bin/bash
count=0
note_count=0

count_file() {
  local file="$1"
  # 具体的な提案（- [ ] 形式）
  local n
  n=$(grep -c '^- \[ \]' "$file" 2>/dev/null || true)
  count=$((count + n))
  # 気づきメモ（- で始まるが - [ ] でない行）
  local m
  m=$(grep -c '^- [^\[]' "$file" 2>/dev/null || true)
  note_count=$((note_count + m))
}

# グローバルメモリ
global_file="$HOME/.claude/memory/pending_recommendations.md"
if [ -f "$global_file" ]; then
  count_file "$global_file"
fi

# 全プロジェクトのメモリ
for f in "$HOME"/.claude/projects/*/memory/pending_recommendations.md; do
  if [ -f "$f" ]; then
    count_file "$f"
  fi
done

if [ "$count" -gt 0 ] && [ "$note_count" -gt 0 ]; then
  printf '💡 /recommendations (具体案%d件 + 気づき%d件)' "$count" "$note_count"
elif [ "$count" -gt 0 ]; then
  printf '💡 /recommendations (%d件)' "$count"
elif [ "$note_count" -gt 0 ]; then
  printf '💡 /recommendations (気づき%d件)' "$note_count"
fi
```

    "type": "command",
    "command": "bash /Users/<username>/.claude/scripts/check-recommendations.sh"

}

## 3. `/recommendations` でまとめて反映する

手が空いたときに `/recommendations` と打つと、Claudeが以下の流れで処理します。

1. `/insights` のレポートを取得する
2. `bash-commands.log` を分析し、未登録のコマンドを候補として抽出する
3. 気づきメモを読み込んで具体的な提案に変換する
4. `rejected_recommendations.md` の拒否済みパターンと類似する項目を除外して一覧表示する
5. ユーザーから承認された項目を適用する

ステップ3では、粗いメモから「これはpermissionsの追加が適切」「これは新しいスキルとして作れる」といった具体的な提案に変換します。

拒否リストの確認はステップ4でまとめて行います。蓄積時にチェックすると、タスクを中断して別ファイルを読みに行くコストが毎回かかり、結局スキップされやすくなるためです。

特に新規スキル作成の場合は、難易度・必要なファイル・SKILL.mdの骨格まで提示するようにしました。提案を承認するだけでそのままスキルを生成できます。

```
┌─────────────────────────────────────────────────────┐
│  💡 保留中のレコメンド  4件                            │
└─────────────────────────────────────────────────────┘

── 🔧 グローバル設定 / permissions ──────────────────────

  [ 1 ]  Bash(git log*)
         根拠  : 1回承認済み (03-22)
         変更先: ~/.claude/settings.json > permissions.allow
         内容  : "Bash(git log*)" を許可リストに追加

── 🛠 グローバル設定 / Skills（既存スキルの活用）───────────

  [ 2 ]  /rev スキル
         根拠  : PRレビュー依頼が3回 (03-22, 03-23)
         変更先: ~/.claude/skills/rev/ (すでに存在)
         内容  : 導入済み。accepted_recommendations.md に記録する

── 🛠 グローバル設定 / Skills（新規スキル提案）────────────

  [ 3 ]  新スキル: /testing-helper
         根拠  : 気づきメモより「テスト実行を毎回手動入力している」(03-23)
         概要  : pnpm --filter [app] test のショートカット
         難易度: 低
         必要なファイル:
           - ~/.claude/skills/testing-helper/SKILL.md
         SKILL.md 骨格:
           ---
           description: フィルター指定でpnpmテストを実行する
           allowed-tools: Bash(pnpm:*)
           ---
           アプリ名を引数で受け取り pnpm --filter {app} test を実行します。
           引数なしの場合は確認してから実行します。
         作業  : 承認後に自動生成可能

── 📝 プロジェクト設定 / CLAUDE.md ────────────────

  [ 4 ]  useEffect制約の補足
         根拠  : 1回誤用を訂正 (03-22)
         変更先: CLAUDE.md
         内容  : useEffectの禁止パターンに該当ケース（派生状態の同期）を追記

─────────────────────────────────────────────────────
  適用する番号を入力  例) 1 3 5  /  「全て」で一括適用
  不要な場合は「スキップ」
─────────────────────────────────────────────────────
```

承認した項目は `settings.json` や `CLAUDE.md` に反映されます。新規スキルの場合はそのまま `SKILL.md` を生成します。スキップした場合は理由を確認し（任意）、あれば `rejected_recommendations.md` に記録して次回以降の提案から除外します。

処理が終わったら `bash-commands.log` をクリアします。

SKILL.md の全文

```
---
description: 溜まっている設定レコメンドを確認・適用する
allowed-tools: Read, Glob, Grep, Edit, Bash(claude -p*), Skill(*)
disable-model-invocation: true
---

保留中のレコメンドを適用します。

# 手順

## 1. /insights レポートの取得

`~/.claude/usage-data/report.html` の更新日時を確認する。7日以内であれば読み込み、古ければ `claude -p /insights` で再生成してから読み込む。

## 2. 蓄積ログの分析

`~/.claude/logs/bash-commands.log` が存在する場合、permissions.allow に未登録のコマンドを抽出して `pending_recommendations.md` に追記する。`rejected_recommendations.md` で拒否済みのパターンは除外する。

## 3. 気づきメモの分析・具体化

両方の `pending_recommendations.md` の「気づきメモ」セクションを読み、以下の分類で具体的な提案を生成する：

| メモの内容 | 生成する提案の種類 |
|-----------|-----------------|
| 繰り返しコマンドや操作 | permissions追加 or 新スキル提案 |
| 同じ説明の繰り返し | CLAUDE.md 補足 or 新スキル提案 |
| 便利そうなスキルへの言及 | 新スキル詳細仕様の提案 |

## 4. 保留レコメンドの読み込み

グローバルとプロジェクト両方の `pending_recommendations.md` を読み、具体的な項目と気づきメモから生成した提案を合算する。何もなければ終了。

## 5. 一覧表示と確認

スコープ・種類ごとにセクションを分けて表示する。新規スキル提案には `難易度`・`必要なファイル`・`SKILL.md骨格` を含める。

## 6. 適用と拒否の処理

- グローバル permissions → `/update-config` スキルで `~/.claude/settings.json` を更新
- CLAUDE.md 更新 → Edit ツールで直接編集
- 新規スキル作成 → Write ツールで `~/.claude/skills/{name}/SKILL.md` を作成

適用した項目は `accepted_recommendations.md` に追記。拒否された場合は理由を聞き `rejected_recommendations.md` に記録。

## 7. クリーンアップ

- 適用・拒否した行を `pending_recommendations.md` から削除
- 処理した気づきメモの行も削除
- `bash-commands.log` を空にする
```

## 自動化は断念

「ステータスバーを見て手が空いたときに `/recommendations` を叩く」という方式に落ち着いていますが、`/recommendations` の実行自体を自動化できないか試みた記録も残しておきます。

**UserPromptSubmit hook**: ユーザーがメッセージを送るたびに発火するhookで、stdoutに出力したテキストがClaudeのコンテキストに注入されます。保留件数が閾値（5件）を超えたらメッセージを差し込む実装です。動作はしたものの、閾値を超えている間はすべての質問で毎回差し込まれ続ける問題がありました。

**SessionStart hook**: セッション開始時に1回だけ発火するため、毎回割り込む問題は解消できます。しかしここで根本的な制約に気づきました。SessionStart hookのstdoutはClaudeのコンテキストに注入されるだけで、ユーザーには表示されません。Claudeはユーザーからのメッセージがなければ応答を生成しないため、hookで何を注入してもユーザーが最初のメッセージを送るまで何も起きません。

結局、`/recommendations` の自動実行は実現できていません。

## おわりに

自分のClaude Codeの使い方に合わせて作ったもので、おそらくベストな方法ではないですし、こういった自作ツールはClaude Code本体の機能追加によってすぐ陳腐化することも多いです。一方で、Claude Codeの仕様を深く調べる良い機会になりました。

同じような細かい悩みを感じている方がいたら、何か参考になる部分があればうれしいです。また、もしもっと良い方法をご存知の方がいたら教えていただけると嬉しいです！
