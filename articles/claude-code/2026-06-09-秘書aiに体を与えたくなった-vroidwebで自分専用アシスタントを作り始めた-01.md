## はじめに

この記事は、私が開発している「自分専用のAIアシスタント Webアプリ」のきっかけ編です。

最初は、Claude Code に秘書っぽい役割を持たせて、MCP で Gmail・Googleカレンダー・自前の Plane をつなぐだけでした。それだけでもかなり便利です。メールを要約してもらう、来週の予定を整理してもらう、タスクを登録してもらう。正直、この時点で「秘書AI」としてはもう実用になっていました。

でも、毎日使っているうちに、別の欲が出てきました。

**この秘書に、姿があってほしい。**

最終的にこのアプリは、3Dアバターが喋り、メールを読み、予定を管理し、健康や食事を記録し、日記まで書くものに育っていきました。とはいえ、まだまだ足りない機能は山ほどあります。だからこそ OSS にして、みんなの力を借りたい。いまその準備を進めているので、公開の前に「なぜ MCP だけで満足せず、VRoid と Web で専用アプリを作り始めたのか」を、最初の一歩から書き残しておきます。

技術解説というより、個人開発の動機と試行錯誤の記録です。

## その頃、「AIエージェントが来る」と思っていた

このアプリを作り始めたのは、数ヶ月前です。

ちょうど OpenClaw が話題になっていた頃でした。OpenClaw や Hermes のような自律エージェントを自分でも動かしてみて、「ああ、これからはAIエージェントが本格的に来るな」という確信が固まっていきました。

もともと、その予感はありました。私は業務で LAS というファイル管理システムを開発していて、そこに AIエージェントを自前で組み込んでいたからです。ただの RAG ではありません。RAG チャットアプリなら去年、ちょうど Dify が流行っていた頃に作っていて、正直、もう一歩先がやりたかった。エージェントがループを回し、ツールを使い、試行錯誤しながらタスクを進める。そういう作りです。さらに、複数のエージェントをどう協調させるかという、エージェントオーケストレーションにも興味がありました。

だから「秘書AIにアバターを」という思いつきも、私にとっては突飛な話ではなく、地続きでした。軽い気持ちではあったけれど、土台には「エージェントが来る」という確信があったんだと思います。

## 始まりは Claude Code と MCP の「秘書ごっこ」

最初のきっかけは、ものすごく地味でした。

私は普段から [Claude Code](https://docs.claude.com/ja/docs/claude-code) を使っているのですが、ある時ふと「これ、コーディングだけじゃなくて秘書としても使えるんじゃないか？」と思いました。……いや、いまどき普通にみんなそんな感じで使ってますよね。当時の私は、ちょっとした発見をした気になっていました。

やったことはシンプルです。`CLAUDE.md` に、秘書としての役割を書き込みました。

`CLAUDE.md` は、Claude Code がプロジェクトの文脈や作業ルールとして読み込む Markdown ファイルです。そこに、たとえばこんな指示を書きます。

> あなたは私の秘書です。丁寧に、でも親しみやすく振る舞ってください。予定やタスクは積極的に確認・整理してください……

もちろん、これだけでは「秘書らしい口調で返事をするチャット」です。実際のメールも予定もタスクも触れません。

中身をつなぐのに使ったのが MCP（Model Context Protocol）です。MCP は、LLM アプリケーションが外部ツールやデータソースにつながるための共通プロトコルです。雑に言うと、Claude に「読みに行ける場所」「実行できる操作」を増やすための接続口です。

この組み合わせが、想像以上に普通に便利でした。

チャット相手であり、調べ物の相棒であり、文章の壁打ち役でもある。最初は「秘書ごっこ」のつもりだったのに、気づけば毎日触っていました。

## Gmail・Googleカレンダー、そして自前の Plane をつなぐ

便利になってくると、欲が出ます。

「どうせなら、実際の予定やメールも扱わせたい」

まず Gmail と Googleカレンダーを MCP 経由で扱えるようにしました。メールの要約を頼んだり、「来週の予定を整理して」とお願いしたり。AI が自分の実データに触れるようになった瞬間、便利さのレベルが一段上がりました。

次に手を出したのがタスク管理です。

世の中には便利な SaaS がたくさんあります。ただ、私は自分のデータをなるべく手元に置いておきたいタイプなので、[Plane](https://plane.so/) というオープンソースのプロジェクト管理ツールを自宅サーバーに立てました。

当時、私の手元にはそのまま使える Plane 用の MCP 連携がなかったので、MCP ラッパーを自作して Claude から叩けるようにしました。これで、次のような指示が通ります。

* 「このタスクを Plane に登録して」
* 「今日やるべきことを一覧にして」
* 「この作業、来週に回しておいて」

自分で立てたサービスに、自分のAIがアクセスする。この閉じた世界が少しずつつながっていく感覚が、とにかく楽しかった。

ちなみに、CRM もオープンソースの [Twenty](https://twenty.com/) を自前で立てて、同じように MCP でつなぎました。連絡先や人とのやり取りまで、AI から扱えるようになります。

同じ考え方で、メモやドキュメントにも広げられます。ローカルの Markdown、Obsidian、Notion など、接続口さえ用意すれば「あとでまとめておいて」が通る世界に近づいていきます。

ここまでは、純粋に実用の話です。メール、予定、タスク、メモ。テキストの秘書としては、もう満足していました。

## でも、なんか物足りない

機能的には十分便利でした。

メールも予定もタスクもメモも、ひとつの AI に話しかければ片付く。やっていることだけ見れば、もう立派な秘書環境です。

なのに、なぜか物足りない。

しばらく使っていて、その理由がだんだん分かってきました。

**秘書には、かわいいアバターが欲しい！！！ しゃべってほしい！！**

チャット欄のテキストとして存在する有能な秘書も良いのですが、毎日付き合うなら、姿があってほしい。表情があって、こちらを見て、声で返してくれる存在であってほしい。

これは、便利さの追求だけではありませんでした。

どちらかというと、愛着の問題です。AIアシスタントを「使う道具」から「一緒にいる存在」に近づけたかったのだと思います。

今振り返ると、この物足りなさこそが、ここから続く開発の本当のスタート地点でした。

## VRoid Studio との出会い

とはいえ、私は3Dモデルにまったく詳しくありませんでした。

「アバターが欲しい」と言っても、どうやって作るのか見当もつきません。Blender でゼロから人型モデルを作る、という選択肢は最初から現実的に見えませんでした。

調べて行き着いたのが、[VRoid Studio](https://vroid.com/studio) です。ピクシブが提供している、3Dキャラクター制作向けの無料ソフトです。

驚いたのは、その手軽さでした。

パラメータをいじって、髪型を選んで、顔を整える。3Dモデリングの専門知識がなくても、それっぽいキャラクターができてしまう。私にとっては、まず自分のアバターを持つ入口として、これ以上ない選択肢でした。

こうして、ふわっとした理想だった「秘書のアバター」が、急に手の届くところまで来ました。

## three-vrm で、ブラウザに3Dを出してみる

アバターができたら、次は「画面に出す」番です。

作りたかったのは Web アプリだったので、ブラウザで VRM を表示する方法を探しました。VRM は、人型 3D アバター向けのファイル形式です。VRoid Studio で作ったモデルも VRM として扱えます。

見つけたのが、[@pixiv/three-vrm](https://github.com/pixiv/three-vrm) でした。three.js の上で VRM を読み込んで扱うためのライブラリです。

最初にやったことは、本当に「とりあえず出してみる」だけでした。

React のコンポーネント内で three.js のシーンを作る。three-vrm で VRM を読み込む。カメラとライトを置く。画面に表示する。

それだけで、ブラウザの中に自分が作ったキャラクターが、ぬるっと現れました。思わず「おおー、すごい！」と声が出ます。こんなに簡単に Web で 3D が動くのか。テキストでしか存在していなかった秘書に、急に「いる感じ」が出てきました。

## でも、表示できただけでは足りなかった

ところが、しばらく眺めていると、また物足りなさが出てきます。

ただ立っているだけなんですよね。

表情がない。動きがない。こちらを見ている感じも薄い。

棒立ちのキャラクターは、想像していた「生きている秘書」とはやっぱり違いました。

ここから、「表情を変えたい」「自然に動かしたい」という試行錯誤が始まります。VRM には表情やボーンの仕組みがあります。問題は、それをどう自然に動かすかでした。

その流れで、ついに Blender にも手を出しました。Blender は、モデリング、リギング、アニメーションなどを扱える高機能な 3D 制作ソフトです。

結論から言うと、めちゃくちゃ難しい。

ショートカット、モディファイア、リギング、ウェイトペイント。知らない概念が次々に出てきて、何をどう触れば思った通りに動くのか分からない。「3Dモデル、めっちゃむずいやん」と、画面の前で何度もつぶやきました。Web に表示するところまでは思ったより早かったのに、その先の「生き生きと動かす」世界は、想像の何倍も深かった。

ここで一度、潔く撤退しました。

表情も動きも、いったん後回し。完璧なアバターを目指して入口で力尽きるより、先に進める部分から作ろう、と考えました。

## まずは「頭脳」から ― チャットモーダルと Sonnet

アバターの見た目と動きを保留した私が次に向かったのは、原点である秘書機能でした。

姿は後からでいい。まずは、ちゃんと会話できる中身がいる。

Web アプリにシンプルなチャットモーダルを作ることにしました。テキストを入力して、AI が返してくれる、いちばん基本的な UI です。

頭脳には、Anthropic の API で Sonnet 系モデルを使うことにしました。

なぜ Opus ではなく Sonnet なのか。

理由は単純で、個人で毎日のように使うにはコストが効くからです。Anthropic の API 価格を見ると、Opus 系モデルは Sonnet 系モデルより高価です。雑談、調べ物、メモ整理、軽い相談まで日常的に投げる用途では、最高性能を毎回使うより、気兼ねなく呼べることのほうが大事でした。

Sonnet は、私の用途では応答品質とコストのバランスがよく、日常使いの相棒として十分でした。

そうして出来上がった素朴なチャットモーダルで、自作キャラクターを頭の片隅に思い浮かべながら会話してみる。まだアバターは棒立ちのまま。チャットも別画面。それでも、「自分専用のアシスタントを、自分の手で組み上げている」という手応えはありました。

## MCP だけでよかったのでは？ という問い

ここまで書くと、「MCP で便利になっていたなら、それでよかったのでは？」と思うかもしれません。

実用だけなら、たぶんそれで十分でした。

Claude Code に役割を持たせ、MCP で Gmail・カレンダー・Plane をつなぐ。これだけでも、かなりのことができます。むしろ、専用 Web アプリを作るより早いし、壊れる場所も少ない。

それでも Web アプリに向かったのは、MCP では足りない部分がはっきりしてきたからです。

私が欲しかったのは、単に「AI がツールを呼べる環境」ではありませんでした。

* いつも同じ場所にいる
* 受け身で待つだけでなく、現状を把握して自分から声をかけてくる
* 同じ見た目と声で返してくる
* 自分の生活ログや記憶を前提に話す
* メール、予定、健康、食事、日記がひとつの体験としてつながっている

そういう、生活に常駐するインターフェースが欲しかった。

MCP は外部サービスとつなぐ土台として強力でした。ただ、私が欲しかったのは、その上に「自分の秘書がいる場所」を作ることだったのです。VRoid と Web アプリは、そのための器でした。

## おわりに（次回へ続く）

ここまでが、このアプリの本当の始まりです。

出発点はどれも軽い気持ちでした。便利な秘書ごっこに物足りなくなり、アバターが欲しくなり、VRoid と three-vrm で姿は出せたのに表情と動きで詰まり、ひとまず頭脳から、と Sonnet でチャットを書き始めた。それだけのことです。

この素朴なチャットモーダルが、このあと、3Dアバターが喋り、メールを読み、予定とタスクを捌き、健康・食事・睡眠を記録し、日記まで書くアプリへ膨らんでいきます。

次回は、その棒立ちのアバターに「声」を与える話です。第1章で叫んだ「しゃべってほしい！！」を、実際にどう実装したのか。

そして、ずっと「準備中」と言っていた OSS 公開、ついに踏み切りました。
