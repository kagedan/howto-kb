#### AIエージェントの定義

**AIエージェント**とは、LLM(大規模言語モデル)を「頭脳」として、外部世界を**観察し、計画を立て、ツールを使って行動し、その結果を見て次の行動を調整する、`自律的な`ソフトウェアシステム**です。つまり、制御フローを**開発者がコードで決める**のではなく、**AIが動的に決めます。**

#### マルチエージェント

**マルチエージェント**は「複数エージェントが協働する仕組み」の総称で、以下はその中の代表的な設計パターンです。

|  | 構造 | 意思決定 | 向いているケース | 弱点 |
| --- | --- | --- | --- | --- |
| **オーケストレータ型** | 明確な階層(司令塔→部下) | オーケストレータが一元管理 | 手順が複雑で、状況に応じて次の担当を選び分けたいとき | 司令塔がボトルネック・単一障害点になる |
| **並列型(Parallel)** | フラット。同時に独立実行し、後で集約 | 分散(各自が自分の担当のみ判断)+ 集約役 | 分割可能で相互依存の少ない作業(複数資料の同時要約など) | 結果の統合と重複・矛盾の解消が必要 |
| **対等型(Peer-to-peer)** | 上下関係なし。エージェント同士が直接やり取り | 完全に分散。合意形成や交渉で決まる | 議論・レビュー・多視点での検討(討論させて結論を出す等) | 収束しない、コスト増、挙動が読みにくい |
| **パイプライン型** | 一直線の流れ作業(A→B→C) | 事実上、設計時に固定済み | 工程が決まっている定型処理(抽出→整形→検証など) | 途中の誤りが下流に伝播、分岐に弱い |

##### 使い分けの目安

* **工程が固定** → パイプライン型
* **工程は固定だが動的な振り分けが要る** → オーケストレータ型
* **独立した大量処理を速くしたい** → 並列型
* **答えが一つに決まらず、質を議論で高めたい** → 対等型

実務では単独で使うより、「オーケストレータの配下で並列実行し、その結果をパイプラインで整形する」といった**組み合わせ**になることが多いです。

#### サブエージェント

**サブエージェント**は、メインエージェントの会話履歴を圧迫せずに特定のタスク(コードレビュー、デバッグ、データ分析など)を専門に行う独立したエージェントで、メインエージェント実行から生成されるバックグラウンドエージェントで実行される。各サブエージェントは独自のセッションで実行され、 完了すると、その結果を依頼元へ通知します。

### README.md

README.md

```
# software_company

## プロジェクト名

software company

## 概要

LLMを用いて、ソフトウェア開発の上流から下流まで、AIエージェントが役割分担を行いながら実行できる会社を作る。

## アーキテクチャ概要

`main.py` は単一の進行役セッション（Claude Agent SDK の `query()`）を起動し、PM・アーキテクト・開発者・QAの4役を `AgentDefinition` によるサブエージェントとして登録する。

**どのサブエージェントをいつ呼ぶか、QAでバグが出た場合に開発者へ差し戻すかどうかは、進行役のLLM自身がプロンプトの指示に従って判断する。** Pythonコード側に決定木やループ制御は存在せず、`build_orchestrator_prompt()` が指示するパイプライン手順（PM→アーキテクト→開発者→QA、バグがあれば開発者に差し戻して再QA、最大3回）をLLMが自律的に実行する。

## エージェント構成

| エージェント | 役割 | 主な責務 | モデル |
| --- | --- | --- | --- |
| pm（プロジェクトマネージャー） | 上流管理 | ユーザー要件を要件定義書に整理 | claude-opus-4-8 |
| architect（アーキテクト） | 設計 | 要件定義書をもとにシステム設計書を作成 | claude-opus-4-8 |
| developer（開発者） | 実装 | 設計書、またはQAのバグ報告をもとにコードを実装・修正 | claude-sonnet-4-6 |
| qa（QA） | 品質保証 | 実装コードをレビューし、バグの有無を報告 | claude-sonnet-4-6 |

各エージェントのシステムプロンプトは `prompts/{pm,architect,developer,qa}.txt` に定義されている。進行役自身は `claude-opus-4-8` で動作する。

### 連携方式：進行役LLMによる自律制御

```text
ユーザー要件
    ↓
[進行役] pm に委譲
    │  出力：requirements.md（要件定義書）
    ↓
[進行役] architect に委譲
    │  出力：design.md（設計書）
    ↓
[進行役] developer に委譲
    │  出力：code_v{n}.md（実装コード）
    ↓
[進行役] qa に委譲
    │  出力：qa_report_{n}.md（テストレポート）
    │
    ├─ レポート末尾が [BUG_FOUND] → developer に差し戻して再実装 → qa で再レビュー（最大3回）
    │
    └─ レポート末尾が [PASS]、または3回目のレビュー後
         ↓
        成果物一式を outputs/<実行日時>/ に保存
```

QAループの停止条件（最大3回、`[PASS]`/`[BUG_FOUND]` マーカーの解釈）は Python の `for` ループではなく、進行役への指示文（[main.py:75-89](main.py#L75-L89)）としてプロンプトに埋め込まれている。

### main.py

main.py

```
import argparse
import asyncio
from datetime import datetime
from pathlib import Path

from claude_agent_sdk import (
    query,
    ClaudeAgentOptions,
    AgentDefinition,
    AssistantMessage,
    UserMessage,
    ResultMessage,
    TextBlock,
    ToolUseBlock,
    ToolResultBlock,
)

MAX_QA_LOOPS = 3
PROMPTS_DIR = Path("prompts")

# サブエージェント名 -> 保存ファイル名（developer/qa はループ回数をファイル名に付与する）
SAVE_NAME = {"pm": "requirements", "architect": "design"}

def load_prompt(filename: str) -> str:
    return (PROMPTS_DIR / filename).read_text(encoding="utf-8")

AGENTS = {
    "pm": AgentDefinition(
        description="ユーザーの要件を要件定義書に整理する。パイプラインの最初に使う。",
        prompt=load_prompt("pm.txt"),
        model="claude-opus-4-8",
        tools=[],
    ),
    "architect": AgentDefinition(
        description="要件定義書をもとにシステム設計書を作成する。pm エージェントの後に使う。",
        prompt=load_prompt("architect.txt"),
        model="claude-opus-4-8",
        tools=[],
    ),
    "developer": AgentDefinition(
        description="設計書、またはQAのバグ報告をもとにコードを実装・修正する。",
        prompt=load_prompt("developer.txt"),
        model="claude-sonnet-4-6",
        tools=[],
    ),
    "qa": AgentDefinition(
        description="実装されたコードをレビューし、バグの有無を報告する。developer エージェントの後に使う。",
        prompt=load_prompt("qa.txt"),
        model="claude-sonnet-4-6",
        tools=[],
    ),
}

def save(name: str, content: str, run_dir: Path) -> None:
    path = run_dir / f"{name}.md"
    path.write_text(content, encoding="utf-8")
    print(f"  → {path}")

def extract_text(content) -> str:
    if isinstance(content, str):
        return content
    parts = []
    for item in content or []:
        if isinstance(item, dict) and item.get("type") == "text":
            parts.append(item.get("text", ""))
        elif hasattr(item, "text"):
            parts.append(item.text)
    return "\n".join(parts)

def build_orchestrator_prompt(task: str) -> str:
    return (
        "以下の手順でソフトウェア開発パイプラインを実行してください。\n\n"
        "1. pm エージェントを使って、ユーザーの要件を要件定義書にまとめる。\n"
        "2. architect エージェントを使って、要件定義書を設計書にする。\n"
        "3. developer エージェントを使って、設計書をもとにコードを実装する。\n"
        "4. qa エージェントを使って、実装したコードをレビューする。\n"
        f"   - QAレポートの末尾に `[BUG_FOUND]` がある場合、そのレポートを developer エージェントに渡して"
        f"修正させ、再度 qa エージェントでレビューする。この developer→QA のループは最大{MAX_QA_LOOPS}回まで。\n"
        f"   - {MAX_QA_LOOPS}回目のレビューでもバグが残る場合は、そこで終了し最後のコードとQAレポートを"
        "そのまま報告する。\n"
        "   - QAレポートの末尾に `[PASS]` がある場合はそこで終了する。\n\n"
        "各エージェントには、前の工程の成果物をそのまま入力として渡してください。\n\n"
        f"# ユーザーの要件\n{task}\n"
    )

async def run_pipeline(task: str) -> None:
    run_dir = Path("outputs") / datetime.now().strftime("%Y%m%d_%H%M%S")
    run_dir.mkdir(parents=True, exist_ok=True)

    pending: dict[str, str] = {}  # tool_use_id -> エージェント名
    counts: dict[str, int] = {}

    options = ClaudeAgentOptions(
        agents=AGENTS,
        tools=["Agent"],
        allowed_tools=["Agent"],
        max_turns=40,
        model="claude-opus-4-8",
    )

    async for message in query(prompt=build_orchestrator_prompt(task), options=options):
        if getattr(message, "parent_tool_use_id", None):
            continue  # サブエージェント内部のメッセージはスキップ

        if isinstance(message, AssistantMessage):
            for block in message.content:
                if isinstance(block, TextBlock) and block.text.strip():
                    print(f"[進行役] {block.text.strip()}")
                elif isinstance(block, ToolUseBlock) and block.name in ("Agent", "Task"):
                    agent_name = block.input.get("subagent_type")
                    if agent_name in AGENTS:
                        pending[block.id] = agent_name
                        print(f"  → '{agent_name}' エージェントに委譲")

        elif isinstance(message, UserMessage):
            for block in message.content or []:
                if not isinstance(block, ToolResultBlock):
                    continue
                agent_name = pending.pop(getattr(block, "tool_use_id", None), None)
                if agent_name is None:
                    continue
                text = extract_text(block.content)
                counts[agent_name] = counts.get(agent_name, 0) + 1
                n = counts[agent_name]

                if agent_name in SAVE_NAME:
                    save(SAVE_NAME[agent_name], text, run_dir)
                elif agent_name == "developer":
                    save(f"code_v{n}", text, run_dir)
                elif agent_name == "qa":
                    save(f"qa_report_{n}", text, run_dir)

        elif isinstance(message, ResultMessage):
            print("\n=== 完了 ===")
            if message.result:
                print(message.result)

if __name__ == "__main__":
    parser = argparse.ArgumentParser(description="AIエージェントによるソフトウェア開発パイプライン")
    parser.add_argument("--task", required=True, help="作りたいソフトウェアの要件")
    args = parser.parse_args()
    asyncio.run(run_pipeline(args.task))
```

## パイプライン処理フロー

`main.py` は、AIエージェントによるソフトウェア開発パイプラインを実行するスクリプトです。  
以下に2つの観点からフローチャートを示します。

1. **開発パイプライン全体の流れ**（オーケストレータが各サブエージェントに委譲する業務ロジック）
2. **run\_pipeline関数の詳細な処理フロー**（メッセージストリームの処理ロジック）

### 1. 開発パイプライン全体の流れ

`build_orchestrator_prompt()` で構築される指示に基づき、オーケストレータ（進行役エージェント）が`pm → architect → developer → qa` の順にサブエージェントへ処理を委譲します。

### 2. run\_pipeline関数の詳細な処理フロー

実際のPythonコードは、`query()` から流れてくるメッセージをリアルタイムに監視し、サブエージェントへの委譲（ToolUseBlock）とその結果ToolResultBlock）を対応付けて成果物をファイルに保存します。これは両方の性質を併せ持つ「**エージェント指向のパイプライン**」と言えます。

# ToDoリスト CLIツール システム設計書

## 1. システム概要

本ツールは、ターミナル上でタスク（ToDo）を管理する軽量なコマンドラインアプリケーションである。サブコマンド方式（`add` / `list` / `done` / `edit` / `delete` / `help`）を採用し、タスク情報をローカルのJSONファイルに永続化する。常駐プロセスを持たず、コマンド実行のたびにプロセスが起動・終了するステートレスな設計とする。

### 1.1 設計上の主要な決定事項
| 項目 | 決定内容 | 根拠（要件） |
|------|----------|--------------|
| 実装言語 | Python 3.8+（標準ライブラリのみ） | 追加ランタイム不要、マルチOS対応（非機能要件: 可搬性） |
| 引数解析 | `argparse`（サブパーサ方式） | 標準ライブラリ、`--help` 自動生成 |
| データ形式 | JSON（UTF-8, `ensure_ascii=False`） | 国際化（日本語対応） |
| 保存場所 | ユーザーホーム配下 `~/.todo_cli/tasks.json` | 制約: 設定ディレクトリ既定 |
| 書き込み方式 | 一時ファイル + `os.replace()` による原子的置換 | 信頼性: データ破損最小化 |
| 外部通信 | なし | セキュリティ: 外部送信なし |

## 2. アーキテクチャ方針

### 2.1 レイヤー構成
責務ごとにモジュールを分離した3層構造とする。各層は下位層のみに依存し、上位層への逆依存を持たない。

```
┌─────────────────────────────────────────────┐
│  CLI層 (cli.py)                              │
│  引数解析・コマンドディスパッチ・入出力       │
└───────────────┬─────────────────────────────┘
                │ 呼び出し
┌───────────────▼─────────────────────────────┐
│  サービス層 (service.py)                     │
│  ビジネスロジック（採番・状態遷移・検索）     │
└───────────────┬─────────────────────────────┘
                │ 呼び出し
┌───────────────▼─────────────────────────────┐
│  永続化層 (storage.py)                       │
│  JSONの読み書き・原子的書き込み               │
└─────────────────────────────────────────────┘

  横断的関心事:
  - models.py    : データ構造の定義・変換
  - errors.py    : 例外定義
  - formatter.py : 一覧表示の整形
```

### 2.2 依存方向の原則
- CLI層はサービス層を呼び出し、結果を整形して標準出力へ書き出す。
- サービス層は永続化層を通じてデータをロード・保存する。ドメインルール（ID採番、状態遷移、フィルタ、ソート）を担う。
- 永続化層はファイルI/Oのみを担い、ビジネスロジックを持たない。
- `models` / `errors` / `formatter` は全層から参照可能な共通部品とする。

## 5. データモデル

### 5.1 列挙型定義（models.py）

```python
class Status(str, Enum):
    PENDING = "pending"   # 未完了
    DONE = "done"         # 完了

class Priority(str, Enum):
    LOW = "low"
    MEDIUM = "medium"
    HIGH = "high"
```

- `str` を継承させることで、JSONシリアライズ時に自動的に文字列化される。
- 優先度は未指定時 `medium` を既定とする。

## 7. データフロー

### 7.1 全コマンド共通の基本フロー
```
[ユーザー] → コマンド実行
   ↓
cli.main: argparse で解析
   ↓
cli.cmd_*: 引数を検証・変換（Priority/Due）
   ↓
TaskService: storage.load() でデータ取得（1回）
   ↓
TaskService: メモリ上で操作（採番/変更/削除/検索）
   ↓
TaskService: storage.save() で原子的書き込み（読み取り系のlistでは省略）
   ↓
cli.cmd_*: 結果を formatter で整形し stdout へ出力
   ↓
main: 終了コード 0 を返す
```

### 7.2 コマンド別処理フロー

#### add（AC-1）
1. `title` の空文字チェック（`ValidationError`）。
2. `--due` 指定時、`validate_due` で形式検証・正規化。
3. `service.add()`: `load()` → `data["next_id"]` をIDに採番 → `next_id += 1` → `Task(status=PENDING, created_at=now_iso())` を生成し `tasks` へ追加 → `save()`。
4. `Added task #<id>: "<title>"` を出力。

#### list（AC-2, AC-3）
1. 排他オプションから `status_filter` を決定（`--pending`→PENDING, `--done`→DONE, 既定→None）。
2. `service.list(status_filter, sort_by)`: `load()` → フィルタ適用 → ソート（既定はID昇順）。
3. `formatter.format_task_table()` で整形し出力。**保存は行わない。**

#### done（AC-4）
1. `service.complete(id)`: `load()` → ID一致タスクを検索（無ければ `TaskNotFoundError`）→ `status=DONE`, `completed_at=now_iso()` → `save()`。
2. `Completed task #<id>: "<title>"` を出力。

#### edit（AC-5）
1. 指定オプションを収集。全項目未指定なら `ValidationError`（「更新項目がありません」）。
2. `--due` 指定時は `validate_due` で検証。
3. `service.edit(id, ...)`: 対象を検索（無ければ `TaskNotFoundError`）→ `None` でない項目のみ上書き → `save()`。
4. `Updated task #<id>` を出力。

#### delete（AC-6）
1. `service.delete(id)`: `load()` → 対象を検索（無ければ `TaskNotFoundError`）→ `tasks` から除去 → `save()`。`next_id` は減算しない。
2. `Deleted task #<id>: "<title>"` を出力。

#### help（AC-9）
1. 引数なし / `--help` / 未知サブコマンド → argparse がヘルプを表示。
2. 引数なしの場合は `parser.print_help()` を明示呼び出しし、終了コード 0 を返す。

## 8. エラーハンドリング方針

### 8.1 基本原則
- 想定内エラーはすべて `TodoError` サブクラスで表現し、CLI層 `main` の単一の `try/except` で捕捉する（AC-7）。
- 例外を最下層で握り潰さず、意味のある例外へ変換して上位へ伝播させる。
- いかなる入力でもトレースバックをそのまま露出させず、`Error: <message>` として stderr へ出力する。

### 8.2 例外と終了コードの対応
| 例外 | 発生条件 | 終了コード | stderr メッセージ例 |
|------|----------|-----------|--------------------|
| `ValidationError` | 空タイトル、不正な日付、更新項目なし | 2 | `Error: due date must be YYYY-MM-DD` |
| `TaskNotFoundError`| 存在しないID指定（done/edit/delete）| 3 | `Error: task #99 not found` |
| `StorageError` | JSON破損、書き込み失敗 | 4 | `Error: failed to read data file (corrupted JSON)` |
| その他 `Exception`| 予期せぬ内部エラー | 1 | `Error: unexpected error: <詳細>` |
| 正常終了 | — | 0 | — |

### 8.3 main の捕捉パターン（擬似コード）
```python
def main(argv=None) -> int:
    try:
        ... # 解析・ディスパッチ
    except TodoError as e:
        print(f"Error: {e}", file=sys.stderr)
        return e.exit_code
    except Exception as e:
        print(f"Error: unexpected error: {e}", file=sys.stderr)
        return 1
```
`argparse` の引数エラー（未知オプション等）は argparse 自身が終了コード 2 で処理するため、そのまま委譲する。

### 8.4 データ破損への耐性
- `load()` で JSON パース失敗時は `StorageError` を送出し、破損ファイルを上書きしない（データ保護優先）。
- 復旧手段として、破損検知時に破損ファイルを `tasks.json.corrupted-<timestamp>` へ退避する処理を任意で実装可（実装工程で判断）。

## 9. 永続化（原子的書き込み）の実装方針

書き込み途中の異常終了でも既存データを破損させないため、「別の一時ファイルへ完全に書き込んでから、既存ファイルへ原子的に置換」する方式を採る。`os.replace()` は同一ファイルシステム上で原子的（POSIX / Windows いずれも rename がアトミック）である。

### 9.1 save() の実装手順
```python
import json, os, tempfile
from pathlib import Path

def save(self, data: dict) -> None:
    path = self.path
    path.parent.mkdir(parents=True, exist_ok=True)  # 保存先ディレクトリ確保

    # 1. 同一ディレクトリ内に一時ファイルを作成（os.replaceのアトミック性の前提）
    fd, tmp_path = tempfile.mkstemp(dir=str(path.parent),
                                    prefix=".tmp_tasks_", suffix=".json")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            json.dump(data, f, ensure_ascii=False, indent=2)
            f.flush()
            os.fsync(f.fileno())          # 2. ディスクへ確実に書き出し
        os.replace(tmp_path, path)        # 3. 原子的に置換（rename）
    except Exception as e:
        # 失敗時は一時ファイルを掃除し、既存ファイルは無傷のまま
        if os.path.exists(tmp_path):
            os.unlink(tmp_path)
        raise StorageError(f"failed to write data file: {e}") from e
```

### 9.2 要点
- **一時ファイルは保存先と同一ディレクトリに作成**する。別ドライブ/テンポラリを跨ぐと `os.replace()` が原子的にならず、`cross-device link` エラーの原因にもなるため。
- `os.fsync()` で OS バッファをディスクへ確実に反映してから置換する。
- 置換前に異常終了しても、既存 `tasks.json` は変更されない（最悪でも一時ファイルが残るのみ）。
- ファイルは常に UTF-8 / `ensure_ascii=False` で書き込み、日本語をそのまま保存する（AC-10）。`indent=2` で可読性を確保。

## 10. 実装上の注意点

1. **文字コード**: 読み書きともに `encoding="utf-8"` を明示。`ensure_ascii=False` を徹底し日本語の Unicode エスケープを防ぐ（AC-10）。
2. **ID一意性**: 採番は必ず `next_id` を用い、削除後も減算しない。既存タスク最大ID+1 ではなく `next_id` を単調増加させることで、削除→追加での衝突を防ぐ（AC-1）。
3. **read/writeの単位**: 各コマンドで `load()` は1回、書き込み系は末尾で `save()` を1回のみ。list は読み取り専用で `save()` を呼ばない。
4. **日時形式**: `created_at` / `completed_at` は UTC・ISO 8601（末尾 `Z`）で統一。表示時のローカル変換は初期スコープ外とする。
5. **ソート仕様**: `due` ソートでは `null`（期限なし）を末尾に配置。`priority` ソートは `high > medium > low` の降順を既定とする。安定ソートで同順位はID昇順を維持。
6. **終了コード**: `main` は必ず int を返し、`__main__.py` / `console_scripts` から `sys.exit(main())` を呼ぶ。CI やスクリプト連携で結果判定できるようにする（8章の対応表）。
7. **argparse のヘルプ**: 各サブパーサに `help` と `description` を設定し、`prog="todo"` を明示。引数なし実行時は `parser.print_help()` を呼び終了コード 0。
8. **テスト容易性**: `config.get_data_file()` を環境変数 `TODO_CLI_HOME` で切替可能にし、テストで一時ディレクトリを注入する。`TaskService` は `JsonStorage` を DI で受け取り、モック差し替えを可能にする。
9. **パフォーマンス**: 全件メモリ読み込み方式。1,000件規模では線形処理で十分1秒以内を満たす（非機能要件）。インデックス等の最適化は不要。
10. **後方互換**: `load()` 時に `version` を確認。将来スキーマ変更時はここでマイグレーション分岐を追加する拡張点として残す。
