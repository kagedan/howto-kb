## はじめに

Downloadsフォルダが、いつの間にか散らかってます。

請求書のPDF、スクショ、Excel、よく分からない `functions_dev.txt`。「あとで整理しよう」と思ったまま、放り込まれたファイルが積もっていく。たぶん多くの人が同じ状態だと思います。

これを自動で片付けたいんですが、拡張子やファイル名のルールで分けるのには限界があります。`請求書_最終版(2).pdf` ならまだしも、`IMG_2084.png` や `スクリーンショット 2026-06-16 8.54.01.png` は、名前を見ても中身が何なのか分かりません。

で、どうせやるなら **AIに中身を読ませて判断させればいいんじゃないか** と思って作りました。

とはいえ、ただ動けばいいわけではなくて、こう動いてほしいというのが3つあります。

* **置いた瞬間に動く**（常駐して自動で）
* **APIキー不要・追加課金なしで始められる**
* **安全**（勝手に消さない、間違えても戻せる）

できたのがこれです。

<https://github.com/knktkc/copper-golem>

![桜の下で、木のロボットが箱からファイルを取り出して仕分けているイラスト](https://static.zenn.studio/user-upload/deployed-images/01507cf4993d3c2855d1e68a.png?sha=4708303eddc4b174d8230a780a96a2dd32c715a7)

名前は、ゲーム「Minecraft」に出てくる、落ちている物を拾って「同じ物が入った箱」へ仕分けてくれる銅（copper）製のロボット（golem）から取りました。その働きをファイル整理に翻訳した、というだけのツールです（ゲームを知らなくても使い方は変わりません）。

## どんなツールか

使い方はシンプルで、整理したいフォルダ（散らかってる Downloads そのものでいいです）の中に、**「請求書」「写真」「資料」** みたいな入れ先フォルダを自分で作っておきます。あとはファイルを置くだけ。

中身を読んで、いちばん合う入れ先へ勝手に移してくれます。ターミナルにはこんなログが出ます。

```
[move] blog_0619.png      ->  ブログ関連/   (conf=0.98)
[move] audio-align.pdf    ->  勉強会/       (conf=0.65)
[keep] メモ.txt           （どこにも当てはまらず、そのまま）
```

ポイントは、ファイル名ではなく **中身で判断する** ところ。`blog_0619.png` という素っ気ない名前でも、中を見て「ブログ用の画像だな」と分かればブログ関連へ入ります。

どの入れ先にも当てはまらないファイルは、無理に動かしません。そのうえで、監視フォルダの中に `_未分類レポート.md` という一覧を作って、**「なぜ分けられなかったか」を理由付きで書き出して** くれます。

```
## メモ.txt
- 確信度: 0.05
- 理由: 個人的なメモで、写真や請求書のいずれにも当てはまらない
```

入れ先フォルダには「何を入れる場所か」を書いた `.golem.md` という小さなメモを置けて、これがあると振り分けが正確になります。手で書いてもいいんですが、フォルダの中身から自動生成するコマンドも用意しました。

```
$ python3 golem.py describe --apply
[write] 請求書/.golem.md
```

ここまでが、ユーザーから見たプロダクトの全体像です。

## 設計の話：AIを「エージェント」ではなく「純粋関数」として使う

ここからが、個人的にいちばん面白かったところです。

最近のAI活用は **エージェント** が主流です。AIにツール（ファイル操作やコマンド実行）を持たせて、自律的にあれこれやらせる。実際それは強力で、このツールも実装自体はAIエージェント（Claude Code）にやってもらいました。

でも、**できあがったツールの中でのAIの役割は、その逆** にしています。

このツールでAIがやるのは、たった一つだけです。

> 「ファイルの中身」と「候補フォルダの説明」を渡されたら、`{"folder": "請求書", "confidence": 0.95, "reason": "…"}` というJSONを返す。

入力を受け取って、出力を返す。それだけです。**副作用のない、ただの関数**。中身の抽出も、フォルダへの移動も、全部こちら側のコードがやって、AIには「読んで、選ぶ」ことしかさせません。

言い換えると、**ファイルを動かすのはAIではなく、こちらのコード**。AIには判断だけさせて、操作の権限は渡さない。これがこのツールの設計の芯です。

### なぜ「純粋関数」だと小さく・安全に作れるのか

「AIに権限を渡してエージェントにした方が楽なのでは？」と思うかもしれませんが、タスクが「分類」のように閉じている場合、純粋関数に閉じ込めた方が圧倒的に小さく・安全になります。理由を分解すると、こうです。

**1. 失敗しても被害が限定される。**  
AIが判断を間違えても、起こるのは「分類が外れる」ことだけです。ファイルが消えたり壊れたりはしません。移動するのはこちらのコードで、やることは「別フォルダへ移す」だけ。しかも全移動を記録していて、まとめて元に戻すコマンド（`undo`）も用意しました。もしAIにファイル操作の権限を渡していたら、最悪のケースの幅がぐっと広がります。

**2. プロンプトインジェクションが効かない。**  
ファイルの中身をAIに読ませる以上、悪意ある文章が紛れ込む可能性はあります。でもAIはツールを持っていないので、誘導されてもできるのは「変なフォルダ名を返す」くらい。そしてその名前が候補フォルダに無ければ、こちらのコードが弾いて「該当なし」にします。**攻撃して動かせる対象がそもそも存在しない**。

**3. テストできる。**  
AIを含む常駐ツールって、非決定的でテストしづらそうに見えます。でも純粋関数として切り出してあるので、**AI呼び出しの一箇所をモックに差し替える** だけで、移動・キャッシュ・undo・衝突回避といったロジックは全部きれいにユニットテストできます。実際このツールには83件のテストがありますが、その間Claudeは一度も呼ばれません。

**4. 依存が小さい。**  
エージェント基盤も、ツール定義も、実行ループも、MCPも要りません。やるのは `claude -p` を1回呼ぶだけ。だからコードも状態も小さく保てます（外部ライブラリ依存ゼロ・Python標準ライブラリだけで動きます）。

**5. 権限設計が要らない。**  
AIに権限を渡さないので、「どこまで許可するか」「実行前に確認を挟むか」みたいなサンドボックスや承認フローの設計が、まるごと不要になります。

要するに、**エージェントは強力だけど、タスクが閉じているなら純粋関数として閉じ込めた方が、小さく・安全・速い** ということです。なんでもエージェントにする流れの中で、ここは逆張りして良かったと思っています。

ちなみに、APIキーが要らないのもこの設計のおかげです。`claude -p`（Claude Code のヘッドレス実行）をそのまま分類器として呼ぶので、普段のサブスク認証のまま動きます。モデルも、分類は一回のLLM呼び出しで済むので、軽くて速いHaikuで十分でした。

## 処理の流れ

ファイルが1つ置かれてから片付くまでの流れは、こんな感じです。

`claude -p` が返すのは点線の1本、JSONだけ。ファイルを動かしているのは全部 `golem.py` 側、というのがこの図からも分かると思います。

## 使い続けるために工夫したこと

機能を盛るというより、「常駐させてもストレスなく使い続けられるか」を意識して手を入れた部分がいくつかあります。

### うるさくしない

常駐させると、変化を検知するたびにフォルダ全体をスキャンします。素朴に作ると、毎回「どこにも当てはまらないファイル」を何度も判定して、そのたびに通知が飛んできて、正直うるさい。

なので、一度「該当なし」と判断したファイルは記憶しておいて、次からはスキップするようにしました。再判定するのは、**そのファイルの中身が変わったとき** か、**新しい入れ先フォルダが増えたとき**（その子の新しい行き先になるかもしれないので）だけ。これで、同じ通知が何度も出ることも、同じファイルを何度もAIに投げることも無くなりました。

### フォルダに説明を持たせる（.golem.md）

入れ先フォルダに `.golem.md` を置くと、それが分類の基準になります。中身は「入れるもの」「入れないもの」「手がかり」の3つ。

```
## 入れるもの
取引先に発行した請求書。
## 入れないもの
領収書、見積書
```

面白いのは、これをフォルダの現状から自動生成できること。`describe` コマンドが、フォルダにすでに入っているファイルを読んで「ここは何を入れる場所か」をAIに書かせて、`.golem.md` として保存します。**フォルダの中身から説明を作り、その説明で次の分類をする** という循環になっていて、特に「入れないもの」に一言足すだけで精度がぐっと上がります。

### 分けられなかった理由を残す

これが地味に効きました。どこにも当てはまらなかったファイルは、監視フォルダの中に `_未分類レポート.md` として、理由付きで一覧化されます。

「なんで片付かないんだろう」が一目で分かるので、「じゃあ `資料/` フォルダを作るか」とか「`.golem.md` の除外条件を直すか」という次の一手につながります。ファイルが全部片付けば、このレポートは自動で消えます。

## ハマったところ

ここからは、作っている途中で踏んだ落とし穴です。macOSで常駐ツールを書く人には、わりと刺さるんじゃないかと思います。

### `claude` がlaunchdから見つからない

最初の大物がこれでした。手元では普通に動くのに、launchd（常駐）から起動すると `claude: command not found` で全部失敗する。

調べたら、`claude` は実体のコマンドじゃなくて **シェル関数** でした。さらに launchd は、ログインシェルのPATHを引き継ぎません。なので「`claude` が無い」となる。実体のバイナリ（npm/mise配下）を自前で探して、しかもそれが内部で使う `node` も見つかるようにPATHを補ってあげる、という対応をしました。

### 通知が出ない

「振り分けました」のデスクトップ通知を、`osascript`（AppleScript）で出していたんですが、これが新しめのmacOSだと **launchd から呼ぶと弾かれる**。手で実行したときは出るのに、常駐からだと無音。

加えて、plistの `ProcessType` を `Background` にしていたのも通知を抑制する要因でした。最終的に `terminal-notifier` という小さなツール経由に切り替えて、ようやく常駐からも通知が届くようになりました。

### 監視フォルダにレポートを書いたら無限ループ

未分類レポートを監視フォルダの中に書き出す設計にしたんですが、よく考えるとこれ、**フォルダに変化が起きる → launchdが再発火 → またレポートを書く → 変化が起きる →** …で無限ループになります。

対策として、**レポートの内容が前回と変わったときだけ書き込む** ようにしました。中身が同じなら何もしない（＝フォルダに変化を起こさない）ので、定常状態ではループが止まります。タイムスタンプを入れると毎回変わってしまうので、あえて入れていません。

### 日本語フォルダに入らない（NFD / NFC）

これは気づくまで一番ハマりました。「ブログ関連」フォルダに入るべき画像が、なぜか毎回「該当なし」になる。AIは正しく `"ブログ関連"` と答えているのに、コードが「そんなフォルダは無い」と判定している。

原因は **Unicodeの正規化** でした。macOSはファイル名を **NFD（濁点が分離した形）** で保存するのに対し、Claudeは **NFC（合成した形）** で返してきます。見た目は同じ「ブログ関連」でも、`グ` や `ブ` の濁点の扱いが違って、単純な文字列比較で一致しない。フォルダ名を比較するときに両方をNFCに正規化することで直りました。日本語のフォルダ名を扱うなら、これは踏みやすい罠だと思います。

## プライバシーと料金について

正直に書いておくと、ファイルの中身を判断するのはAI（Claude）なので、**判定するファイルの内容の一部はClaudeに送られます**。普段Claude Codeに資料を読ませるのと同じ扱いです。気になる場合は、そもそも **Claudeの設定でデータを学習に使わせない（オプトアウト）** にしておくのがおすすめです。機密性の高いフォルダをわざわざ監視対象にしない、という運用上の割り切りも併用するとよいと思います。

料金についても一言。このツール自体は無料（MIT）ですし、いまは普段使っているClaude Codeの範囲で動きます。ただ、利用条件を決めるのは提供元のAnthropicなので、**いまは無料で使えても、将来この使い方が課金の対象になる可能性はあります**。そのあたりはAnthropicの情報を気にかけておくのがよさそうです。

## 今後の展開

いまはmacOS（launchd）専用ですが、Linux（systemd / inotify）やWindowsにも広げられたら、と思っています。あとは確信度のしきい値の自動調整や、`.golem.md` をテンプレートとして共有できる仕組みなど。とはいえ、自分の用途では今ので十分片付いているので、しばらくはこのまま使ってみるつもりです。

## まとめ

エージェントが何でもやってくれる時代ですが、「分類」みたいに役割がはっきり閉じているタスクは、**AIを純粋関数として閉じ込める** と、小さく・安全・テストしやすく作れる、という話でした。AIには「読んで選ぶ」だけをさせて、ファイルを操作する権限は渡さない。それだけで、設計がずいぶん楽になります。

ちなみにこのツールも、方針を決めて躓きを直していくところは人間がやりましたが、実装のほとんどはClaude Codeと一緒に進めました。「散らかったDownloadsをAIで片付けたい」から始まって、ここまで形になるのは、便利な時代になったなぁと思います。
