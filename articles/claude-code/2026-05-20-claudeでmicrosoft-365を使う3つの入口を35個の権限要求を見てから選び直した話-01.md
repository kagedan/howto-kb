## はじめに：同意画面で立ち止まった

「ClaudeとMicrosoft 365を繋げば、メールもカレンダーもAIから一発で操れる」。  
そう聞いて、Softeria製のMS365 MCPサーバーをClaude Codeに追加した。

ところが同意画面を開いて、固まった。

権限要求が35個ズラっと並んでいる。一部抜粋すると、

* Read your mail
* Read mail you can access
* Read all users' full profiles
* Read your mailbox settings
* Read your calendars
* Read your files
* Read your OneNote notebooks
* Read your tasks and task lists
* Read directory data
* Read all groups
* Read your chat messages
* Read user chat messages
* Read your channel messages
* Read items in all site collections
* Read all transcripts of online meetings
* Read all recordings of online meetings
* Read presence information of all users in your organization
* Maintain access to data you have given it access to

全部「Read」だから書き込み事故はない。  
しかし、要求範囲が広すぎる。

しかも画面下部には「組織の代理として同意する」というチェックボックス。  
組織に属する全てのユーザーに対して、この35個の権限を一括で与えるかどうかを、画面を見ている一人が決められる仕組みだ。

弊社は私が全管理しているため、判断して同意した。  
ただ、これが「情シスのある企業」だったら、確実に止められるレベルの権限要求だ。

そして翌日、別の選択肢があることを知った。  
Anthropicが2026年5月11日に Claude for Microsoft 365 を一般提供開始していた。  
こちらは経路も権限モデルもそもそも違う。

私は一応セキュリティ業界の人間として、こうした「アプリへの権限委譲」は仕事でも何度も見てきた。せっかくなので、ClaudeでMS365を使う3つの入口を整理し、それぞれで「どこに何が送られるか」を比較してみる。機密データを扱う業務の人にとって、入口の選び方は無視できない論点だと思う。

## ClaudeでMicrosoft 365を使う3つの入口

| 入口 | 仕組み | 経由するAPI | 主にできること | 権限の渡し方 |
| --- | --- | --- | --- | --- |
| 1. MCP連携 | Claude が外部MCPサーバー経由でGraph APIを叩く | Microsoft Graph API | メール・カレンダー・OneDrive・Excel範囲・Teams議事録の読み取り、データの横断処理 | OAuth同意で30個前後の権限を一括付与 |
| 2. Claude for Microsoft 365 | Office アプリ内にClaudeアドインを統合 | Office JavaScript API（PowerPoint・Excel・Word向け） | Excel・PowerPoint・Word内でネイティブ編集・テンプレ準拠ライティング | Officeアプリ内に閉じる |
| 3. Webアプリ（claude.ai） | claude.aiにファイルを直接アップロード | なし（手動アップロード） | 単発のファイル処理・要約・編集案 | 都度アップロードする情報のみ |

3つは全く別の技術スタック・別の権限モデルで動く。  
「ClaudeがMicrosoft 365にアクセスできる」と一言で言っても、入口によって渡す情報の範囲と経路が大きく違う。

## MCP連携の権限要求35個を読み解く

サードパーティMCPサーバーは複数存在するが、今回触ったのはSofteria製の MS 365 MCP Server。  
特に選定したわけではなく、MS365向けMCP実装として最も広く知られていたものを採用した。  
他にはコミュニティ実装のMCPサーバーが GitHub上に複数あり、企業導入時は自社でMCPサーバーを構築して権限スコープを業務範囲に絞るのが本来の最適解になる。サードパーティ実装を使う場合は権限要求を1つずつ精査する必要がある。

同意画面で確認できる権限を、影響範囲別に分類するとこうなる。

### 個人スコープ（自分のデータのみ）

* メール（Read your mail / Read your mailbox settings）
* カレンダー（Read your calendars）
* ファイル（Read your files）
* OneNote（Read your OneNote notebooks）
* タスク（Read your tasks and task lists）
* 連絡先（Read your contacts）
* プロファイル（Sign you in and read your profile）
* チャット（Read your chat messages）
* チャネル投稿（Read your channel messages）
* オンライン会議（Read your online meetings）
* プレゼンス（Read your presence information）
* 関連する人物リスト（Read your relevant people list）

ここまでは「使う本人のデータを読みに行く」範囲で、まあ理解できる。

### 組織横断スコープ（影響範囲が広がる）

注意して見るべきはこちらだ。

* Read all users' full profiles … 組織内の全ユーザーのプロファイル
* Read directory data … ディレクトリ情報全般
* Read all groups … 組織内の全グループ
* Read group memberships … グループメンバーシップ
* Read user chat messages … 他ユーザーのチャットメッセージ
* Read items in all site collections … 全SharePointサイトのアイテム
* Read all transcripts of online meetings … 全オンライン会議の文字起こし
* Read all recordings of online meetings … 全オンライン会議の録画
* Read presence information of all users in your organization … 組織内全員のオンライン状況
* Read all company places … 会社の全拠点情報

組織にユーザーが100人いる中堅企業で、この同意が通ったとしたら何が起きるか。  
1人が誤ってMCPサーバーに何らかの脆弱性を作り込めば、組織全体のチャット・会議録画・SharePointまでアクセス可能な状態になる。

### 「組織の代理として同意する」チェックボックスの意味

これにチェックを入れると、組織内の全ユーザー分まとめて同意したことになる。  
一度の操作で組織全体に対する権限委譲が完了する。

逆に、チェックなしなら同意した個人のデータ範囲だけに限定される。  
all 系の権限も、自分のテナント内で実際にアクセスできる範囲は契約内容と RBAC によって絞られる。

私は会社規模が小さく、影響範囲が限定的なのでチェックなしで同意した。  
中規模以上の組織で同じことをやろうとしたら、まず情シスの審査を通す必要があるレベルだ。

## 5月11日に一般提供開始 Claude for Microsoft 365

Anthropicが2026年2月にPowerPoint向けアドインのbeta、4月にWord、そして5月11日にExcel・PowerPoint・Word全てが一般提供開始（GA）。  
これは MCPとは全く別ルートで Microsoft 365 にアクセスする仕組みだ。

### 仕組み

Microsoft AppSourceから提供される Office アドイン。  
PowerPointの「ホーム > アドイン」から「Claude」を検索して追加。インストールするとPowerPoint内サイドバーにClaudeが表示される。

技術的には Office JavaScript API を経由する。Microsoft 公式の Add-in 配信基盤を通って、Officeアプリ内のオブジェクト（スライド、シート、文書）を直接操作する仕組み。

### できること

* スライドマスター・レイアウト・フォント・カラーを読み取って、テンプレートに準拠したスライド生成
* 既存スライドの選択編集（「ここをもっと簡潔に」のような指示で対応）
* 画像ではなくネイティブPowerPoint要素として出力（編集可能）
* Excel・PowerPoint・Word間でクロスapp context（4月のWord追加で3点セット完成）
* Skills（再利用可能なワークフロー）と Connectors（外部データ連携）

### MCPと何が違うか

* アクセス範囲：Officeアプリ内に閉じる。メール・カレンダー・SharePointには触れない
* 権限委譲：Office Add-in としての標準権限（Microsoft AppSource審査済み）
* 利用条件：Microsoft 365サブスク + Claude Pro/Max/Team/Enterprise

「全Teams会議の文字起こし」のような大きな権限を渡さなくても、文書編集の用途だけならアドイン経由で完結する。

## 用途別の選び方

判断軸はシンプルで、「何をClaudeにやらせたいか」だけ。

文書編集（スライド作成、Excel操作、Word文書）が主目的なら Claude for Microsoft 365 アドイン一択。MCPの大きな権限を渡す必要はない。

データ横断や自動化（メール一括処理、カレンダー操作、Teams議事録の自動取得）が必要なら MCP連携が必要になる。ただし権限要求の範囲は精査が要る。

両方やりたいなら、アドインとMCPの併用。権限が重複する範囲もあるので、用途ごとに使い分ける設計が要る。

単発で使う、たまにファイルを処理するだけなら Webアプリで都度アップロードが最も権限ミニマム。

## どれを使うと何がどこに送られるか

便利さの裏で「データがどこに送られるか」を整理しておく。3つの入口それぞれで送信経路が違う。

### 共通：いずれもAnthropicには送られる

3つともClaudeのバックエンドはAnthropicが運営している。  
だから「Claudeに何かを処理させた瞬間、その情報はAnthropicのサーバーに送られる」のは共通。  
「ローカルで完結する選択肢はない」と理解しておく。

これは Microsoft Copilot を使う場合の Microsoft、ChatGPT を使う場合の OpenAI と同じ構造。AI を使う以上、どこかのベンダーに情報を渡している。

### MCP連携の経路

Claude（Anthropic） → MCP Server（Softeria） → Microsoft Graph API、という3段階の経路を通る。

Claudeが「メール読みたい」と判断すると、まずMCPサーバーがGraph APIを叩く。結果をClaudeに返す。この間、データの経路にMCPサーバー運営会社(今回はSofteria)が中継として入る。

MCPサーバーがOSSであっても、ホスト先で改変されていないかは確認できない。

リスクとしては、MCPサーバー側に意図しない処理(ログ取得・別ホスト送信)があった場合、35個の権限分のデータが流出する可能性。

対策は3つ。信頼できるMCPサーバーを選ぶ、自社内でMCPサーバーをセルフホストする、ログ監査の仕組みを別途持つ。

### Claude for Microsoft 365 の経路

PowerPointアドインから Anthropic Server へ直接通信。AnthropicとPowerPointの間にサードパーティは入らない（Microsoft AppSourceは配信のためのストアであって、データの通り道ではない）。

送信されるデータは、Officeアプリ内で操作している文書の内容。スライド本文、セル範囲、Word本文等がそのままAnthropicに渡る。

リスクは、機密データを含むスライドを開いた状態でClaudeに「要約して」と頼んだら、その機密データはAnthropicに送信される点。

対策は、開く前にマスキング処理する、秘匿性が高い資料ではアドインを起動しない、組織の生成AI利用ポリシーに沿って判断する。

### Webアプリ(claude.ai)の経路

ユーザーが自分でファイルを選んでアップロードする。経路はシンプルで、ブラウザから Anthropic Server へ直送。

送信するデータの範囲を都度コントロールできる。他のルートと違って「権限委譲」の概念がない代わりに、ヒューマンエラー（間違ったファイルをアップロード）が直接的な原因になる。

対策は、アップロード前の目視確認、マスキング済みファイルを使うこと。

### リスク比較表

| 入口 | データ送信先 | 経由する第三者 | 一度の同意で渡る範囲 | 機密データの扱いやすさ |
| --- | --- | --- | --- | --- |
| MCP連携 | Anthropic | MCPサーバー運営会社 | 35個程度の権限(同意時に確定) | 注意：権限を渡した範囲は常時アクセス可能 |
| Claude for M365 | Anthropic | なし(Microsoft AppSourceは配信のみ) | アドインの動作範囲(Officeアプリ内) | 中：開いた文書がそのまま送信される |
| Webアプリ | Anthropic | なし | アップロードしたファイルのみ | 最も制御しやすい |

## 採用業務・士業など機密情報を扱う仕事での判断

私は採用コンサルが本業で、候補者の職務経歴書、面接記録、契約書、提案書といった機密データを日々扱う。  
こうした業務でClaudeを使う場合、入口選びは「便利さ」だけでは決められない。

### マスクツールとClaude for M365 の組み合わせ

機密情報を含む候補者情報や提案書をAIで扱うとき、最初にやることはマスキング。  
個人名・所属企業・連絡先・年収などを伏字化してから、AIに渡す。

このとき、マスク後のデータをClaude for Microsoft 365 で編集するなら、Anthropicに送られるのはマスク済みデータだけ。  
逆にMCP経由で「全ファイル横断検索」のような使い方をすると、マスクされていない原本データが渡る可能性がある。

つまり、業務の前段にマスクツールがあれば、Claude for M365 はかなり安全に使える組み合わせになる。  
弊社では自社開発のマスクツールを使っているが、市販のマスキングサービスでも同じ思想で設計できる。

### MCP連携でデータ横断処理する場合

メール処理やTeams議事録抽出などをMCPで自動化する場合は、マスクツールを通せない経路になる。  
ここは「組織内のデータが Anthropic と MCPサーバー運営会社に渡る前提」で許容できるかの組織判断になる。

弊社のような小規模な体制なら判断はシンプル。  
中規模以上の組織なら、情シスとセキュリティ担当の合意が必要なレベル。

### セキュアにAIを使いこなす

弊社ではAIのプロエンジニアと組んで「セキュアにAIを使いこなす」をテーマにしたサービス検討を進めている。  
そこで議論しているのは、まさにこの「入口の選択」だ。

* データを外に出さないでいい業務は、ローカルLLMで完結させる
* マスクすれば外に出して問題ない業務は、マスクツール経由でクラウドLLMへ
* マスクしなくても良い業務は、クラウドLLMそのまま

ClaudeでMS365を使う場合も同じで、用途と機密度で入口を切り分ける設計が要る。  
「便利だから入れる」では事故る。

## Claude Code(CLI)併用パターン

最後に、私が日常的にやっているハイブリッド運用を紹介する。

### これまでの運用：python-pptxスクリプト + 手動修正

PPTX資料の作成は、Claude Code(CLI)に python-pptx でスクリプトを書かせて、構造を一気に生成。  
その後、細かい文言・レイアウトはPowerPointで手動修正する流れだった。

問題は、修正後に「もう少しテキスト変更したい」と頼むと、Claude Codeが再生成して手動修正分を上書きしてしまう事故が頻発したこと。恥ずかしながら何度もやらかした。

### 新運用：Claude Code + Claude for PowerPoint

これからはこう棲み分ける。

構造作成・大規模変更はClaude Code(CLI)で。20ページのPPTXを一気に生成、全スライドの表形式を統一、複数ファイル横断(gitとの連携も含む)といった大量編集はこっちが得意。

細部の編集はClaude for PowerPoint(アドイン)で。「このスライドのここを簡潔に」「テンプレートに合わせてレイアウト整え直して」のような会話ベースの修正は、PowerPoint内でClaudeと話しながら直す方が速い。

両方をハブとして使う運用なら、python-pptxスクリプトの巻き戻し事故も起きない。  
ただし、Claude Code側の生成スクリプトと PowerPoint側で手動編集した内容を整合させる運用ルールは別途必要。私は何度かやらかして、最終的に「ユーザーが直したらClaude側で取り込む」というフローに落ち着いた。

## まとめ

ClaudeとMicrosoft 365を繋ぐルートは3つある。それぞれ送るデータの経路と権限モデルが違う。

| 用途 | 推奨ルート |
| --- | --- |
| 文書編集中心 | Claude for Microsoft 365 アドイン |
| データ横断・自動化 | MCP連携(ただし権限要求を精査) |
| 単発・たまに使う | Webアプリで都度アップロード |

「便利だから入れる」ではなく、「どの入口から何を渡すか」を設計判断する。  
特に機密情報を扱う業務では、マスクツールやローカルLLMとの組み合わせを含めて、用途別に入口を切り分ける設計が要る。

AIエージェントの導入が増えるほど、こうした権限設計を判断できる人材が必要になる。  
これは攻撃を防ぐ系のセキュリティとは別の役割で、AI時代に新しく求められるスキルセットだと思う。  
採用業界にいる人間から見ても、「AI導入の権限設計を語れるセキュリティ人材」は明らかに不足している。今後の採用ニーズの中心がここに来ると見ている。
