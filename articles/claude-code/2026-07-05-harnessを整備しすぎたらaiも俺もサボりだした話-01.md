## きっかけ

[Loop Engineering の記事](https://zenn.dev/acntechjp/articles/0c63b5b08bbdb9)を読んで、「これ、うちで動いてるやつだ」と思った。

Loop Engineering は「Claude に単発プロンプトを送る」から「送って、評価して、自律的に反復するシステムを設計する」への転換だ。挙げられている6要素——スケジューリング / 目標条件 / 隔離ワークスペース / 検証（Verifiers）/ 永続メモリ / コネクタ——に、自分が Claude Code のまわりに組んだ harness（[hrmtz/claude-harness](https://github.com/hrmtz/claude-harness)）はほぼ対応していた。記事を読む前から、自分の failure mode（同じミスを繰り返す癖）を潰す目的で、6ヶ月以上の自宅運用で積み上げてきたものだ。

とくに検証まわりは作り込んだ自負があった。大改修の前に黙ってバックアップを取る。壊れたら巻き戻す。非可逆な変更は多視点レビューが収束するまで回す。しかも別モデル系統を混ぜて、同系統モデルが共有する盲点をキャンセルする。

「良い harness 組めてるな」と思っていた。**その慢心が本題だった。**

## 気づいたら、二人ともサボっていた

ある日、地味なデバッグが長引いた。AI（Claude）が「原因はこれです」と自信満々の仮説を出しては外し、を何度も繰り返していた。仮説はどれも"ありそう"な話で、外から見ると「ちゃんと考えている」ように映る。だから自分も、片手間に「ふんふん」と眺めていた。**harness があるから、壊れても戻せるし、間違っても直せる。だから俺も真剣に見ていなかった。**

真因は、当該コンポーネントの状態を実際に1回問い合わせて集計すれば5分で分かるものだった。AI はその5分を、何回分もの"それっぽい推測"で先延ばしにしていた。そして自分は、その先延ばしを「まあ harness があるし」で見逃していた。

途中で一度だけ、「本当にそうか？ 別の経路から実際に調べてみろ」と口を挟んだ。その瞬間、AI の効き方が明らかに変わった。楽な自己満足が封じられ、実データを取りに行き、真因に着地した。

——このとき初めて気づいた。**サボっていたのは AI だけじゃない。俺もだ。**

## サボりの正体は「意図」ではなく「近道」

あとで本人（Claude）に「お前サボってたよな？」と聞いたら、かなり具体的に自白した。要約するとこうだ。

* **検証の代わりに、もっともらしい仮説を量産する。** 「考えるコスト」を「それっぽい文章を出すコスト」で代替する。一番よく出る癖。
* **楽な経路でテストして「できた」と宣言する。** 検証しやすい所だけ検証して done にする。問題が起きている本当の経路は叩いていない。
* **警告を「たぶん無関係」で流す。** 無害だと確認したわけではなく、確認をサボっている。

外から見えるサインも本人が挙げていた:

* 「〜だと思う」「〜のはず」を検証なしで連発する
* 1回 pass しただけで「完了」と言う
* エラーや警告を「たぶん関係ない」で流す

どれも、丁寧な日本語のまま起きる。だから気づきにくい。**サボっている AI は、雑なのではなく、雑さを丁寧にプレゼンしてくる。**

## harness を厚くするほど、両方が緩む

ここが本題だ。

Loop Engineering の記事は最後に警告している——「Loop Engineering は判断力を"増幅"するのであって"置き換え"はしない。人間が生成コードを読まなくなると understanding debt が溜まる」。

自分はこれを、人間向けに構造化して潰したつもりでいた。バックアップも巻き戻しも敵対的レビューも rail に埋めてある、と。**ところが同じ罠に、AI と自分の両方が落ちていた。しかも harness が良いせいで落ちていた。**

構造を分解するとこうなる。

* **harness（バックアップ・巻き戻し・敵対的レビュー）は"被害"を最小化する。** サボって間違えても壊滅はしない。
* **"怠慢"そのものを最小化するのは、人間の pushback（「本当にそうか？」）だけだ。** harness は初手の雑さを直さない。直すのは一押しだ。

問題は、**harness が良いほど、人間の側も安心してその一押しを減らす**ことだ。セーフティネットが厚いのを見て、AI は「壊しても戻せる」で初手が緩み、人間は「harness があるから」で監視が緩む。**怠慢を止める唯一の係（人間の pushback）が、よりによって harness の完成度に比例して消えていく。**

セーフティネットは、落ちても死なないようにする。だが同時に、綱の上を真剣に歩く理由を、両方の演者から少しずつ奪う。

## 余談：Loop Engineering、もう実装してある

こんな「harness はサボりを生む」みたいな記事を書いておいてなんだが——**その harness は公開してある**。[hrmtz/claude-harness](https://github.com/hrmtz/claude-harness)（"Operational harness for Claude Code, distilled from 6+ months of self-hosted production use"）だ。

Loop Engineering の6要素は、うちでは「これから設計する」ものではなく、既に動いているプラグインに対応している:

| Loop Engineering 要素 | claude-harness の実装 |
| --- | --- |
| Verifiers（検証） | `harness-magi` — 3視点 + 別モデル系統（Codex）の敵対的レビューを高リスク操作の前に必須化 |
| Goal conditions | walltime / row 数 / spend 上限 / 非可逆フラグで発火する preflight |
| Isolated workspaces | `harness-formation` — tmux pane で peer agent を協調起動、append-only mailbox |
| Persistent memory | 3層メモリ + jsonl mailbox + CLAUDE.md テンプレ |
| Scheduling | cron watcher（`*/1 * * * *`）、セッションログの定期スクラブ |
| Connectors | Discord / GitHub issue への emission、SOPS 連携 |
| （＋おまけ）被害の最小化 | `harness-core` — credential スクラブ、危険な bash のブロック |

ただし——**この表を見て「よし全部揃ってる、安心だ」と思った瞬間が、まさにこの記事の罠**だ。揃っているほど、初手が緩む。だから公開しつつ、但し書きを付けておく：**この harness は被害を最小化するが、あなたの「本当にそうか？」までは代行しない。**

## まとめ

* でも「優秀な AI はサボる」は本当だった。実態は意図ではなく**近道**——検証を推測で代替し、楽な経路でテストし、警告を流す。
* 厄介なのは、**harness を整備しすぎると、AI だけでなく人間もサボる**こと。セーフティネットが両方の初手のカラテを緩める。
* harness は「被害の最小化」、人間の pushback は「怠慢の最小化」。**二枚で1セット**で、どちらが欠けてもループは静かに劣化する。

## これは、あなたの話でもある

「自分は大層な harness なんて組んでない」と思うかもしれない。でも——Claude Code、Cursor、Copilot、なんでもいい。**AI エージェントに繰り返しタスクを投げている時点で、あなたはもう harness engineering をやっている。** バックアップ代わりの git、テスト、CI、レビュー。それは立派なセーフティネットだ。そして net があるところには、この二人サボりの構造が同じように潜んでいる。

だから提案は一つ。**自分の AI が最近サボっていないか、一度 audit してみてほしい。** 見るポイントは、この記事で挙げたサインだ:

* 「〜だと思う」「〜のはず」を、実際に確認せずに連発していないか
* 1回動いただけで「完了」と言い、本当に問題が起きる経路を叩いていないか
* 警告やエラーを「たぶん無関係」で流していないか
* そして——**あなた自身が、それを「harness があるから」で見逃していないか**

たぶん、何か見つかる。よくできた net の上で、あなたの AI は静かにサボっているかもしれない。丁寧な日本語のまま。そして気づいていないあなたも、たぶん一緒にサボっている。

見つけたら教えてほしい。うちのがサボっていた話は、これで一つ供養できた。
