本記事では、ハーネスエンジニアリングの定義から、注目されるようになった経緯、エンジニアが現場で活用できる実践的TIPSまで体系的にまとめていきます。

## ハーネスエンジニアリングとは？

ハーネスエンジニアリングとは、 **モデルの外側にある環境・制約・フィードバックを設計し、エージェントの動作を安定させる技術体系** のことです。

2026年2月に、HashiCorp 共同創業者（現在は独立）の Mitchell Hashimoto が自身のブログ記事で提唱し、その後OpenAIのCodexチームの記事でも取り上げられたことで広く知られるようになりました。

<https://mitchellh.com/writing/my-ai-adoption-journey>

<https://openai.com/index/harness-engineering/>

### ハーネスが持つ4つの役割

OpenAIのエンジニア Ryan Lopopolo は、ハーネスの役割を次の4つに整理しています。

| 役割 | 内容 |
| --- | --- |
| **制約（Constrain）** | エージェントができることを制限する |
| **情報提供（Inform）** | エージェントに何をすべきかを知らせる |
| **検証（Verify）** | エージェントが正しく実行したかを確認する |
| **修正（Correct）** | エージェントが間違えたときに自己修正させる |

この4つはプロンプトでも実現できそうに見えますが、プロンプトはあくまでもモデルへの「お願い」であり、エージェントが従うとは限りません。  
一方、ハーネスは **仕組みとしての強制** であり、設定した通りに必ず実行されます。

### フレームワークとハーネスの違い

LangChainなどのフレームワークとハーネスは混同されることがありますが、根本的に異なります。

| 比較軸 | フレームワーク（LangChain等） | ハーネス（Claude Code等） |
| --- | --- | --- |
| **組み立て** | 人間が部品を配線して組み立てる | 組み立て不要・動作する状態で完結 |
| **設計の主体** | 人間アーキテクト | エージェント自身 |
| **提供物** | 抽象化レイヤー（State・Chain・Memory等） | 動作するエージェント |
| **使い方** | 人間がエージェントを組み立てるために使う | エージェントがタスクを実行するために使う |

フレームワークは「人間がエージェントを組み立てるツール」です。  
ハーネスは「エージェントがタスクを実行する場」として提供される、完成した動作単位です。

### ハーネスエンジニアリングの効果

ハーネスエンジニアリングの効果はデータでも裏付けられています。

LangChainのコーディングエージェントは、モデルを変えずにハーネス改善のみで、[Terminal Bench 2.0](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-benchmark)のスコアを約14ポイント向上させることに成功しました。

| 変更内容 | スコアへの影響 |
| --- | --- |
| ハーネスのみ改善（モデル変更なし） | 52.8% → 66.5% |

<https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering>

また、[SWE-bench](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-benchmark#%E3%82%B3%E3%83%BC%E3%83%87%E3%82%A3%E3%83%B3%E3%82%B0)でも同様の傾向が見られています。  
モデルのアップグレードによる改善が数ポイントに留まる一方で、環境設計（ハーネス）の変更では20ポイント以上の向上が報告されています。

| 変更内容 | スコアへの影響 |
| --- | --- |
| モデルをアップグレード | ≈ 数ポイント |
| 環境設計（ハーネス）を変更 | 20ポイント以上 |

最高クラスの複数モデルを比較しても、スコア差はごくわずかに収束しています。モデルはすでにコモディティ化（どれを使っても同水準になる均質化）しており、**差別化要因は環境設計** に移っていると言えるでしょう。

<https://particula.tech/blog/agent-scaffolding-beats-model-upgrades-swe-bench>

## なぜ今ハーネスエンジニアリングが注目されているのか？

エンジニア・技術者の方であれば、ご存知の内容も含まれるかもしれませんが、ハーネスエンジニアリングの背景を理解するうえで重要な文脈ですので、AI駆動開発の変遷を一緒に振り返っておきましょう。

### AI駆動開発の変遷

[AI駆動開発](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-ai-driven-development)が本格化した2024年以降、エンジニアが注力すべき領域は段階的に変化してきました。

**2024年：[プロンプトエンジニアリング](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-prompt-engineering)**

この頃は、GPT-4やClaude 3が開発現場に普及し始めたばかりで、エンジニアにとってAIはまだ「どう使いこなすか」を模索する段階にありました。そのため、関心は **「いかに良いプロンプトを書いて、モデルをうまく動かすか」** に集まり、[CoT](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-chain-of-thought)・[Few-shot](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-few-shot-prompting)・ReAct といった手法が次々と標準化されていきました。

**2025年：コンテキストエンジニアリング**

2025年に入ると、複雑なタスクへの需要が高まり、「1回のやり取りをうまくする」だけでは対応しきれない場面が増えてきます。こうした背景から、OpenAI共同創業者・Eureka Labs創業者の Andrej Karpathy が **コンテキスト設計** の重要性を提唱し、注目を集めます。[RAG](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-rag) の標準化や[MCP](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-mcp) の登場とも重なり、メモリ管理やシステムプロンプト設計が中心テーマになっていきました。

**2026年：ハーネスエンジニアリング**

さらに2026年に入ると、エージェントが複数ステップ・長時間にわたって動き続けるユースケースが広がりました。こうしたタスクでは、プロンプトやコンテキストをいくら整えても追いつかない問題が顕在化してきます。そこで注目されたのが、 **「モデルへの指示」** ではなく **「モデルが動く環境そのもの」** を設計するという視点です。OpenAI・Anthropic・Martin Fowlerなど複数の組織が独立してこの概念に辿り着いたことから、ハーネスエンジニアリングは特定の誰かが発明したというよりも、 **時代の必然** として生まれた考え方であることが分かります。

![AI駆動開発の変遷](https://res.cloudinary.com/zenn/image/fetch/s--x4YZGx-D--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_002.png?_a=BACAGSGT)  
*AI駆動開発の変遷 — プロンプト設計（2024年）からコンテキスト設計（2025年）、ハーネス設計（2026年）へと、エンジニアが注力すべき領域が段階的に変化してきた流れ*

### 3つのアプローチの違い

プロンプト、コンテキスト、ハーネスの関係性について、もう少し詳しく整理します。

|  | プロンプト設計 | コンテキスト設計 | ハーネス設計 |
| --- | --- | --- | --- |
| **アプローチ** | うまく書かせる | 適切な情報を与える | 間違えられない状態を作る |
| **制御の場所** | モデルへの入力 | モデルが参照する情報 | 実行環境・フィードバック |
| **安定性** | セッション依存 | コンテキスト依存 | 仕組みとして担保 |
| **失敗への対応** | プロンプトを修正 | 情報を補強 | 環境に制約を追加 |

ここで強調しておきたいのは、3つのアプローチは互いを **否定** するものでも、 **置き換える** ものでもないということです。  
下図に示すように、内側からプロンプト → コンテキスト → ハーネスの順に、入れ子の構造になっており、外側の層ほど、モデルの挙動に左右されない安定した制御が実現できます。

![プロンプト設計・コンテキスト設計・ハーネス設計の入れ子構造を示した同心円図](https://res.cloudinary.com/zenn/image/fetch/s--gO5AfoDi--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_001.png?_a=BACAGSGT)  
*図：プロンプト・コンテキスト・ハーネスの入れ子構造*

「プロンプト設計をやめてハーネスに移行する」といった **択一的** な考え方ではなく、プロンプトの上にコンテキストを、コンテキストの上にハーネスを重ねていくといった **積層的** な考え方をするのが重要です。

ここまで「なぜハーネスエンジニアリングが必要になったか」の背景を整理しました。次は、その具体的な仕組みを見ていきます。

## ハーネスエンジニアリングの基本構造

ハーネスの仕組みは、大きく分けると、事前制御と事後制御という2種類に分類されます。

**事前制御** は、エージェントが行動する前に「何をすべきか・何をしてはいけないか」を定義する仕組みです。ルールファイル・型システム・権限設定などが該当します。

**事後制御** は、エージェントが行動した後に「正しかったか」を検証し、問題があれば自己修正を促す仕組みです。テスト・リンター・フックによる自動検証などが該当します。

事後制御はさらに2種類に分かれます。

| 種類 | 特徴 | 具体例 |
| --- | --- | --- |
| **計算型** | 決定論的・高速（CPU実行） | テスト、リンター、型チェック |
| **推論型** | 意味的・AI判定（GPU実行） | AIコードレビュー、[LLM](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-llm)による品質判定 |

コスト・速度・信頼性の観点から、まず計算型の事後制御を整備し、計算型で対応できない意味的な判断にのみ推論型を使うのが基本戦略です。

![ハーネスの基本構造：事前制御と事後制御](https://res.cloudinary.com/zenn/image/fetch/s--gZyCd6JI--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_003.png?_a=BACAGSGT)  
*ハーネスの基本構造：事前制御と事後制御 — エージェントの行動前にルールで制約する「事前制御」と、行動後にテスト・AI判定で検証する「事後制御」の2層構造*

## 実践ガイド

仕組みを理解したところで、実際に構築する手順を見ていきます。ハーネス設計は次の **4ステップ** で進めます。

| ステップ | 内容 |
| --- | --- |
| **STEP1** | ルールを文書化する（AGENTS.md） |
| **STEP2** | 自動検証を仕組みに埋め込む（フック） |
| **STEP3** | コンテキストを分離する（サブエージェント） |
| **STEP4** | 失敗を制約に変え続ける（継続改善） |

### STEP1：ルールを文書化する

最初のステップは、エージェントへの指示をルールとして文書化することです。

その役割を担うのが **AGENTS.md**（Claude Code では CLAUDE.md）です。エージェントへの指示を記述するファイル形式として広く使われています。

設計の原則は「短く、実際の動作から書き起こす」ことです。60行以内を目安に、 **エージェントが実際に繰り返した悪いパターン** を列挙します。机上で考えた理想ルールより、実際の失敗から書き起こしたルールの方が効果的と言われています。

```
# AGENTS.md の記述例

## 禁止事項
- mainブランチへの直接pushは禁止
- APIキーをコードにハードコードしない
- ファイル削除前にユーザーへの確認を取る

## 必須事項
- コミット前にlintを実行する
- テストが通ることを確認してからPRを作成する
- コミットメッセージは日本語で記述する
```

### STEP2：自動検証を仕組みに埋め込む

次のステップは、検証の仕組みをコードに埋め込み、自動で回るようにすることです。

プロンプトで「毎回テストして」と指示する代わりに、コードの変更を検知して自動的に検証が走る仕組みを設定します。これを実現するのが **[フック（Hook）](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-hooks)** です。

```
コードを書く
  → フックが自動起動
    → テスト・リントを実行
      → 失敗した場合、エラー詳細をエージェントに返す
        → エージェントが自己修正する
```

フックの本質は、フィードバックループの自動化です。エージェントが「自分でミスに気づいて直せる」状態を、仕組みとして確立します。

Claude Code の場合、`.claude/settings.json` に以下のように記述するとファイル編集のたびにlintとテストが自動実行され、結果がエージェントに返されます。

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint && npm test 2>&1 | tail -20",
            "timeout": 60
          }
        ]
      }
    ]
  }
}
```

### STEP3：コンテキストを分離する

3つ目のステップは、複雑なタスクをサブエージェントに分割し、コンテキストを分離することです。

1つのエージェントにすべてを任せると、コンテキストが肥大化して精度が落ちます。 **[サブエージェント](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-multi-agent)** を活用することで、各タスクを独立した[コンテキストウィンドウ](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-context-window)に切り出せます。

サブエージェント活用の主な利点は2点あります。

1. 親エージェントのコンテキストを汚染せず、長期的な一貫性を保てる
2. 親には高性能モデルを使いつつ、サブエージェントには低コストモデルを割り当てるコスト最適化ができる

Anthropic SDK を使うと、フェーズごとに独立したコンテキストでサブエージェントを呼び出せます。

```
import anthropic

client = anthropic.Anthropic()

def run_subagent(system_prompt: str, task: str) -> str:
    """独立したコンテキストウィンドウでサブエージェントを実行"""
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=8096,
        system=system_prompt,
        messages=[{"role": "user", "content": task}]
    )
    return response.content[0].text

# フェーズごとに独立したサブエージェントに委譲
review = run_subagent(
    system_prompt="コードレビューの専門家として問題点を指摘してください。",
    task=source_code
)
fix = run_subagent(
    system_prompt="コードの修正案を作成してください。",
    task=f"レビュー結果：\n{review}\n\n対象コード：\n{source_code}"
)
```

各 `run_subagent` の呼び出しは独立したコンテキストウィンドウで実行されるため、前フェーズのやり取りが次フェーズに持ち込まれません。

![サブエージェントによるコンテキスト分離](https://res.cloudinary.com/zenn/image/fetch/s--o3loNs9Q--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_005.png?_a=BACAGSGT)  
*サブエージェントによるコンテキスト分離 — 親エージェントがタスクを委譲し、各サブエージェントが独立したコンテキストウィンドウで動作することで、親のコンテキスト肥大化を防ぐ構造*

【Anthropicの事例】長時間エージェントの状態管理

複数のコンテキストウィンドウをまたいで作業する場合、状態の引き継ぎが課題になります。Anthropicは次の二層構造を推奨しています。

* **初期化エージェント**：最初のセッションで環境をセットアップ（スクリプト・進捗ファイル・初期コミット）
* **コーディングエージェント**：以降のセッションで段階的に進捗を推進

進捗ファイルとGit履歴を組み合わせることで、新しいコンテキストで起動した際も前の状態を素早く把握できます。「状態の引き継ぎ」をプロンプトではなくファイル設計で解決する——これも典型的なハーネスエンジニアリングです。

<https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents>

### STEP4：失敗を制約に変え続ける

最後のステップは、ハーネスを一度作って終わりにせず、失敗のたびに制約を追加し続けることです。

エージェントが失敗するたびに制約を追加し、同じ失敗が起きない環境に変えていく——この繰り返しがハーネスを成熟させます。[TDD（テスト駆動開発）](https://zenn.dev/umi_mori/books/generative-ai-glossary/viewer/what-is-tdd)がバグを見つけるたびにテストを追加してスイートを育てるのと、まったく同じ構造です。

また、AGENTS.mdやフック設定はすべてコードと同様に扱うことができるため、バージョン管理やPRレビューの対象として管理することで、チーム全体でハーネスを育てることができます。

## 実践的TIPS〜ハーネスエンジニアリングの効果を引き出すためにやるべきこと〜

4ステップでハーネスの骨格を作ったら、次は効果をさらに引き出す **7つのTIPS** です。コンテキスト管理・タスク分解・自動化・スロップ排除など、現場で今日から使える観点を体系的にまとめています。

*ハーネスエンジニアリングの実践的TIPS — コンテキスト管理・タスク分解・自動化・スロップ排除など、現場で即活用できる7つのTIPSのサマリー*

### TIPS 1：コンテキストウィンドウの「スマートゾーン」を守る

コンテキストウィンドウには **「スマートゾーン」** と **「ダムゾーン」** があります。  
使用率が約 **40%** を超えると精度が落ちやすくなるため、この境界を意識することが重要です。

コンテキストが増えると、モデルは大量のトークンに均等に注意を向けなければならず、古い情報や重要な指示が埋もれやすくなります。MCPツールがJSONやUUIDを大量に流し込む場合は特に注意が必要で、ほぼすべての作業がダムゾーンで行われることになります。

Claude Codeでは80〜90%でコンパクション（会話履歴の圧縮）が自動実行されますが、それを待たずに約40%を目安として能動的に圧縮することで、高い精度を維持できます。

![コンテキストウィンドウのスマートゾーンとダムゾーン](https://res.cloudinary.com/zenn/image/fetch/s--LyM-PWcx--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_004.png?_a=BACAGSGT)  
*コンテキストウィンドウのスマートゾーンとダムゾーン — 使用率約40%を境に精度が低下する「ダムゾーン」に突入するため、Claude Codeの自動コンパクション（80〜90%）を待たず能動的に圧縮するのが推奨*

### TIPS 2：「continue」を言わせないコンテキスト設計をする

エージェントに対して `continue` や「次に進んで」と入力しなければならない状況は、 **コンテキスト設計の失敗** のサインです。エージェントが次に何をすべきかを自律的に判断できていないのは、コンテキストに必要な情報が不足しているからです。

AGENTS.md に **「テストがグリーンになるまで実装を続ける」** のような完了条件を明記することで、エージェントが中断せず自律的に動き続けられるようになります。

### TIPS 3：RPI（リサーチ・プラン・インプリメント）で複雑タスクを分解する

複雑な既存コードベースでの開発では、いきなり実装を依頼するのではなく、3フェーズに分けて進めると品質が安定します。

| フェーズ | 目的 | 出力物 |
| --- | --- | --- |
| **リサーチ** | システムを客観的に調査し、関連ファイル・行番号を特定する | ファイル名・行番号・コードスニペットを含む圧縮ドキュメント |
| **プラン** | 正確な手順・テスト手順を含む計画を作成する | 計画ファイル |
| **インプリメント** | 計画に沿って実装し、コンテキストを低く保つ | 実装コード |

タスクの複雑さに応じてRPIの深さを調整しましょう。ボタンの色変更なら直接指示で十分ですが、複数リポジトリにまたがる機能なら1回のリサーチ＋プランが必要になります。

重要な原則は **「思考をアウトソーシングしない」** ことです。悪いリサーチは何百行もの悪いコードに直結します。リサーチドキュメントには必ずファイル名・行番号・コードスニペットを含め、密度の高い圧縮情報にしてください。

### TIPS 4：システムプロンプトのキャッシュを保護する

**プレフィックスキャッシュ** とは、同一のトークン列で始まるAPIリクエストを効率化する仕組みです。システムプロンプトの先頭部分が変わらない限り、Anthropicのインフラがそのキャッシュを再利用してAPIコストを削減します。

そのため、ルールやコアな指示（**静的コンテンツ**）は先頭に固定し、タスク固有の情報（**動的コンテンツ**）は末尾に追加するのが基本です。動的な部分を先頭に置いてしまうと、変更のたびにキャッシュ全体が破壊され、コストが増加します。

### TIPS 5：スロップを体系的に排除する

**「スロップ（Slop）」** とは、エージェントが繰り返す不良動作パターンです。「コメントを書きすぎる」「テストをスキップする」「デバッグコードを残す」といった行動がその典型例です。個別にプロンプトを修正するのではなく、体系的に排除することでハーネスが成熟していきます。

```
スロップを観察・分類する
  → ESLintルール・テスト・エラーメッセージとして固める
    → AGENTS.md に文書化する
      → 全エージェントが自己修正できる状態になる
```

このループを継続的に回すための実践として、 **週1回の「ガベージコレクションデー」** を設けることをお勧めします。その週に観察したスロップをカテゴリ分類して体系的に排除する日を設けることで、ハーネスが自然と成熟していきます。

![スロップ排除のサイクル（TIPS 5）](https://res.cloudinary.com/zenn/image/fetch/s--ET8BzhVI--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-contents/ai_harness_engineering_harness_engineering_006.png?_a=BACAGSGT)  
*スロップ排除のサイクル — 不良動作を①観察→②分類→③文書化（AGENTS.md）→④自己修正の順で体系的に排除し、ループを継続することでハーネスが成熟していく*

### TIPS 6：コードベースを均質化する

コードベースの統一性を高めると、エージェントが任意のファイルを参照しても **転移可能なコンテキスト** が得られます。「このファイルはどんな書き方をされているか」を毎回学習し直す必要がなくなるため、精度と速度の両面で効果があります。

具体的には、同一言語・同一ヘルパー・同一CI（継続的インテグレーション）スクリプトに統一することが基本です。 **ファイルは350行以内** に収めることでコンテキスト効率が高まり、ビジネスロジックとスタックレイヤーでパッケージを分離することで、小さなユーティリティとして再利用できるようになります。

### TIPS 7：レビューエージェントをCIに組み込む

セキュリティ・信頼性・QA（品質保証）のペルソナを持つレビューエージェントをPushのたびに起動し、一定以上の問題をブロックする仕組みを作ります。

| ペルソナ | チェック観点の例 |
| --- | --- |
| **セキュリティ** | タイムアウト・リトライ・安全なインターフェース設計 |
| **信頼性** | エラーハンドリング・冪等性・障害時の挙動 |
| **QA** | テストカバレッジ・エッジケース・ドキュメント整合性 |

ポイントは、各ペルソナの判断基準を **リポジトリに文書化** しておくことです。「良いQAプランとは何か」を一度書き下ろすことで、すべてのエージェントが同じ基準でレビューを実施できます。プランファイルをPRに添付することで、チームの「メンタルアライメント（認識一致）」も維持しやすくなります。

## エージェント導入の判断基準

エージェントを導入すべきかどうか迷ったとき、以下の4軸チェックリストが判断の助けになります。

| チェック軸 | エージェントが適している | ワークフローで十分 |
| --- | --- | --- |
| **タスクの複雑性** | 曖昧で複雑な問題空間 | 決定木でマッピングできる |
| **タスクの価値** | トークンコストに見合う高い価値がある | バジェットが数セント程度の大量処理 |
| **ボトルネックの有無** | クリティカルな能力が確認済み | ボトルネックが多くスコープを絞る必要がある |
| **エラーのコストと発見容易性** | エラーが発見しやすい（単体テスト・CI等） | エラーが高リスクかつ発見困難 |

4軸すべてが「エージェントが適している」に当てはまる場合はエージェント採用を検討しましょう。「ワークフローで十分」が2つ以上あれば、まずシンプルな自動化から始めるのがお勧めです。

## まとめ

ハーネスエンジニアリングの本質は、「モデルを良くする」のではなく「エージェントが動く環境を良くする」という発想の転換にあります。

AIは、置かれた環境をそのまま反映する、いわば増幅器のような存在です。良い制約・良いフィードバックを積み上げれば品質が増幅され、曖昧な環境では曖昧な結果が増幅されるのです。

これからのAI駆動開発においては、「どのモデルを使うか」よりも「どんな環境に置くか」という問いが、より重要になっていきます。
