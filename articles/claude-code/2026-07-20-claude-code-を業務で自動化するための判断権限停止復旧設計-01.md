Claude Code に同じ種類の修正を繰り返し依頼していると、依頼文は次第に長くなる。対象ファイル、互換性、変更禁止範囲、必須試験、報告形式を毎回書き足すためである。

しかし、長い依頼文を保存して使い回しても、継続的な作業環境にはならない。規則を複数の依頼文へ複製すると、更新済みの版と古い版が混在する。反対に、「この不具合を直して」のように短くするだけでは、対象、維持条件、完了条件を Claude Code の推測に委ねることになる。

短い指示が成立するのは、判断を省略したときではない。恒久条件を設定へ移し、現在の作業で変わる差分だけを対話で指定できたときである。本稿では、Claude Code の機能を順番に紹介するのではなく、どの条件を満たしたら自動化、並列化、外部連携へ進めるのかを整理する。

## 1. 長い依頼文を複製しても作業条件は固定されない

POSIX shell のリポジトリを例にする。修正のたびに必要になる条件は、少なくとも次のようなものである。

* `/bin/sh` で動作する構文を使う
* `which` ではなく `command -v` を使う
* Bash 固有の配列や `[[ ... ]]` を追加しない
* 既存のコマンドライン引数を維持する
* 変更後に `./run_tests.sh` を実行する
* 必須試験を実行していない場合は成功と報告しない

これらを毎回の依頼文へ書けば、その場の指示は具体的になる。ただし、依頼文 A に「外部依存を追加しない」を追記し、依頼文 B へ反映し忘れれば、同じリポジトリに対して異なる規則が使われる。依頼文を長くすることは、規則の保管場所を増やすことにもなる。

指示の長さと明確さは、別の軸で考える必要がある。

| 指示の状態 | 恒久条件 | 作業固有の指定 | 判断の所在 |
| --- | --- | --- | --- |
| 長く具体的 | 対話欄へ毎回記載する | 対象、目的、制約、検証方法をまとめる | その場では明確だが、再入力と更新漏れが発生する |
| 短く具体的 | `CLAUDE.md`、rules、設定ファイルへ保存する | 今回の対象、目的、維持条件、停止点だけを指定する | 恒久条件と個別判断の境界が明確になる |
| 短く曖昧 | 必要な条件が保存されていない | 対象や完了条件を限定しない | 不足した条件を Claude Code が推測する |

同じ作業を継続するために必要なのは、長文の定型プロンプトではない。作業に必要な状態が、適切な場所に一つずつ存在する構成である。恒久条件を対話欄から外し、作業固有の差分だけを入力する構造が、短い指示と曖昧な指示を分ける。

## 2. `CLAUDE.md` と rules に判断条件を固定する

### 2.1 `CLAUDE.md` には全作業で必要な条件を書く

`CLAUDE.md` は、長い依頼文をそのまま移す場所ではない。プロジェクトについての事実、常に維持する技術的制約、すべての変更に共通する検証条件を置く。

CLAUDE.md

```
# Project overview

This repository contains POSIX shell scripts for Debian systems.

# Implementation rules

- Use /bin/sh syntax.
- Use command -v instead of which.
- Do not introduce Bash-specific syntax.
- Preserve existing command-line interfaces unless explicitly requested.
- Do not add external dependencies without explicit approval.

# Validation

- Run ./run_tests.sh after changing scripts.
- Run shellcheck when it is available.
- Report changed files, executed checks, and test results.
- Do not report success when a required check was skipped.

# File operations

- Do not delete or move files without explicit approval.
- Do not modify files outside this repository.
- Do not edit generated files when their source file can be changed instead.
```

この内容は、生成時の参考情報であると同時に、成果物を検品する基準にもなる。Bash の配列を使った差分は実装規則に反し、`./run_tests.sh` を省略した完了報告は検証規則を満たさない。規則を「守るべき雰囲気」ではなく、作業後に照合できる条件として書くことが要点である。

一方、「今回の不具合について 3 案を比較する」「変更前に承認を待つ」といった進行手順は、作業ごとに変わる。これを `CLAUDE.md` へ蓄積すると、無関係な作業にも同じ工程が持ち込まれる。

### 2.2 対象限定の規則は `.claude/rules/` へ分ける

シェルスクリプトだけに適用する規則は、パス限定 rules に分離できる。

.claude/rules/shell-scripts.md

```
---
paths:
  - "**/*.sh"
---

# Shell script rules

- Use POSIX /bin/sh syntax.
- Quote variable expansions unless word splitting is intentional.
- Use command -v instead of which.
- Do not use arrays or [[ ... ]].
```

`paths` を持つ rules は、Claude Code が指定したパターンに一致するファイルを扱うときに適用される。`paths` がない rules は無条件に読み込まれる。ファイルを分けただけでは適用範囲は限定されないため、配置と対象条件を同時に確認する必要がある。

情報の種類ごとに、配置先を次のように分けられる。

| 情報 | 配置先 |
| --- | --- |
| プロジェクト全体の事実と常設規則 | `CLAUDE.md` |
| 現在の端末だけで使う個人条件 | `CLAUDE.local.md` またはローカル設定 |
| 特定のファイル種別にだけ必要な規則 | `.claude/rules/*.md` |
| 必要時に呼び出す複数工程 | `.claude/skills/<name>/SKILL.md` |
| 今回だけの目的、対象、完了条件 | 現在の依頼 |
| 機械的に禁止または確認する操作 | Permissions または Hooks |

### 2.3 読み込まれた状態まで確認する

設定ファイルを作成したことと、現在のセッションへ読み込まれたことは同じではない。上位ディレクトリと現在の作業ディレクトリにある `CLAUDE.md`、各階層の `CLAUDE.local.md` は、単純な上書きではなく順番に連結される。下位ディレクトリの指示は、その配下のファイルを扱う時点で追加される。

設定後は、Claude Code 内で `/memory` を実行し、プロジェクトと利用者の指示ファイルを確認する。

パス限定 rules は、対象パターンに一致するファイルを扱う時点で読み込まれる。編集前に対象ファイルを読み、適用すべき制約を列挙させれば、glob の誤りや規則の競合を実装前に発見しやすい。

規則が期待どおりに読まれていなければ、内容を追加する前に配置と glob パターンを直す。複数の場所へ同じ規則を書き足すと、適用対象、更新時期、優先関係が見えなくなる。

ただし、`CLAUDE.md` に「削除しない」と書いても、削除コマンドが技術的に実行不能になるわけではない。指示としての規則と、実行経路を遮断する設定は分ける必要がある。

## 3. Plan mode、Permissions、Hooks は異なる強制力を持つ

### 3.1 Plan mode は変更前の承認点である

影響範囲をまだ説明できない作業は、編集から始めない。Claude Code を Plan mode で起動すると、ファイルの読み取りや読み取り専用コマンドによる調査は行うが、ソースファイルを編集しない。

```
claude --permission-mode plan
```

対話中は `Shift+Tab` または `/plan` で Plan mode へ移れる。ここで確認するのは、計画が丁寧に書かれているかではない。

* 原因候補と、その根拠
* 変更対象と変更しない範囲
* 互換性への影響
* 実行する試験
* 失敗時に戻す単位
* 実装前に人間が確定すべき選択肢

Plan mode は、提案と実行の間に承認点を置く。実装案の比較が終わっていない段階で編集権限を与えないための境界である。

### 3.2 Permissions は道具とコマンドの到達範囲を制限する

プロジェクト単位の権限は `.claude/settings.json` に置ける。

.claude/settings.json

```
{
  "permissions": {
    "allow": [
      "Bash(./run_tests.sh)",
      "Bash(shellcheck *)"
    ],
    "ask": [
      "Bash(git commit *)"
    ],
    "deny": [
      "Bash(git push *)",
      "Bash(rm *)"
    ]
  }
}
```

この設定では、試験は確認なしで実行できる。コミットは毎回確認し、`git push` と `rm` は拒否する。

権限ルールは `deny`、`ask`、`allow` の順で評価され、最初に一致した結果が採用される。より具体的な `allow` があっても、広い `deny` に一致すれば拒否される。たとえば `Bash(git *)` を拒否した後に `Bash(git status)` を許可して例外化する構成は成立しない。

Bash のパターンでは空白も意味を持つ。

* `Bash(ls *)` は `ls -la` に一致するが、`lsof` には一致しない
* `Bash(ls*)` は `ls -la` と `lsof` の両方に一致し得る
* `Bash(git push *)` は `git push` で始まるコマンドに一致する

規則を追加した後は `/permissions` で、実際に読み込まれたルールと設定元を確認する。設定ファイルの見た目だけでなく、対象コマンドがどの規則へ一致するかを試験する必要がある。

### 3.3 Hooks は実行時の入力と状態を検査する

Permissions はコマンド文字列や道具の種類による静的な制御に向いている。現在のブランチ、対象パス、引数の組み合わせなどを見て判断する場合は `PreToolUse` Hook を使う。

次の例では、Bash 呼び出しの直前に検査スクリプトを実行する。

.claude/settings.json

```
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/block-destructive.sh",
            "args": []
          }
        ]
      }
    ]
  }
}
```

フックスクリプトは標準入力から JSON を受け取り、`.tool_input.command` を検査できる。

.claude/hooks/block-destructive.sh

```
#!/bin/sh

set -eu

if ! command -v jq >/dev/null 2>&1; then
  printf '%s\n' 'Blocked: jq is required to validate the tool input.' >&2
  exit 2
fi

if ! command_value=$(jq -er '.tool_input.command | strings' </dev/stdin); then
  printf '%s\n' 'Blocked: Bash command could not be read from hook input.' >&2
  exit 2
fi

if [ -z "$command_value" ]; then
  printf '%s\n' 'Blocked: Bash command is empty.' >&2
  exit 2
fi

case "$command_value" in
  rm\ *|rm)
    printf '%s\n' 'Blocked: direct rm commands require an approved workflow.' >&2
    exit 2
    ;;
esac

exit 0
```

`PreToolUse` は、Claude がツールの引数を作成した後、実際の呼び出しを処理する前に動く。終了コード `2` はツール呼び出しを遮断し、標準エラー出力を Claude へ返す。通常の Unix コマンドで失敗を示す `1` は、多くの Hook イベントでは非遮断エラーとして扱われ、処理が継続する。入力解析に失敗した場合も `2` で止め、検査不能な状態を許可へ倒さない構成にする。

この最小例が検出するのは、コマンド先頭にある直接的な `rm` だけである。`sh -c`、コマンド置換、別名の実行ファイルまで判定するシェル構文解析器ではない。広い禁止は Permissions へ置き、Hook では対象ディレクトリ、現在のブランチ、外部環境など、用途を限定した条件を検査するほうが保守しやすい。

ここまでの役割は次のように分かれる。

| 機構 | 制御する対象 |
| --- | --- |
| `CLAUDE.md` と rules | Claude Code が何を選ぶべきか |
| Plan mode | 編集へ進む前に何を承認するか |
| Permissions | どの道具とコマンドを使用できるか |
| Hooks | 実行時の引数や状態をどう判定するか |

判断を導く指示と、実行を止める機構を同じものとして扱うと、「書いてあるから防げる」という誤認が生じる。

## 4. 自動化する前に復旧境界を決める

### 4.1 チェックポイントが戻せる範囲を確認する

Claude Code は、ファイル編集ツールによる変更をユーザープロンプトごとのチェックポイントとして追跡する。巻き戻し画面は次のコマンドで開ける。

空の入力欄で `Esc` を 2 回押しても同じ画面を開ける。執筆時点の公式文書では、セッション内の最新 100 個のチェックポイントが保持され、セッションとともに標準で 30 日後に削除される。

ただし、チェックポイントはファイルシステム全体のスナップショットではない。次のような Bash コマンドによる変更は追跡されない。

```
rm file.txt
mv old.txt new.txt
cp source.txt dest.txt
```

そのため、Permissions で Bash を広く許可し、「失敗したら `/rewind` すればよい」と考えるのは危険である。Claude Code の編集ツール、Git、バックアップ、外部サービスの取消処理は、それぞれ異なる復旧範囲を持つ。

### 4.2 復旧単位を三つに分ける

| 復旧対象 | 主な手段 | 戻せない可能性があるもの |
| --- | --- | --- |
| 現在の対話と直接編集 | Claude Code のチェックポイント | Bash、別セッション、外部プロセスによる変更 |
| リポジトリの永続状態 | Git のブランチ、コミット、worktree | 未記録の外部サービス操作 |
| 外部サービス | 下書き、取消 API、補償処理、監査記録 | 送信済み通知、公開済み情報、不可逆操作 |

自動化できるかどうかは、一度成功したかでは決めない。失敗したときの状態を説明できることが必要である。

* どのエラーで処理を止めるか
* 途中まで変更した場合に何を戻すか
* 再実行で二重処理が起きないか
* 必須試験の未実行を検出できるか
* 外部書き込みが部分成功した場合にどう補償するか

成功時の手順しか書けない工程は、まだ対話から切り離せない。

## 5. 入力、停止、報告を定義できた工程だけを Skill にする

### 5.1 `CLAUDE.md` から反復手順を分離する

同じチェックリストや複数工程を何度も貼り付ける状態になったら、Skill へ分離する。

```
.claude/
└── skills/
    └── review-changes/
        ├── SKILL.md
        ├── checklist.md
        └── scripts/
            └── validate.sh
```

各 Skill は `SKILL.md` を入口とする。Claude Code は関連すると判断したときに Skill を使えるほか、`/review-changes` のように直接呼び出せる。Skill の本文は使用時に読み込まれるため、常時参照する事実を置く `CLAUDE.md` と、必要時だけ使う手順を分けられる。

### 5.2 Skill に正常系だけを書かない

次は、シェルスクリプトの変更を検品する Skill の骨格である。

.claude/skills/review-changes/SKILL.md

```
---
name: review-changes
description: Review changed POSIX shell scripts and run required checks.
argument-hint: "[target-path]"
---

Review the specified change.

1. Confirm that the target exists and is inside the repository.
2. Read CLAUDE.md and applicable path-scoped rules.
3. Inspect the diff and identify compatibility risks.
4. Run ./run_tests.sh.
5. Run shellcheck for changed shell scripts when available.
6. Stop if an unexpected file is modified or a required check fails.
7. Report:
   - changed files
   - detected risks
   - commands executed
   - test results
   - skipped checks and reasons
```

これだけでも手順は再利用できるが、非対話実行へ進めるには終了状態をさらに分ける必要がある。

* 変更不要
* 修正済みで全試験成功
* 修正済みだが試験失敗
* 原因候補を特定したが承認待ち
* 入力不足で未着手
* 想定外の変更を検出して停止

「調査して修正し、テストする」だけでは、どの状態を成功として後段へ渡すか決まらない。Skill は成功手順の保存ではなく、入力、工程、検証、停止、報告を一つの契約へまとめる単位である。

### 5.3 対話中の補正に依存していないか確認する

Skill を何度か実行し、次を観察する。

* 対象ファイルがないときに推測で別ファイルを選ばないか
* 試験が失敗した後に、成功した部分だけを根拠に完了と報告しないか
* `shellcheck` がない場合に、未実行を明示するか
* 想定外の差分を見つけたときに処理を止めるか
* 同じ入力で終了状態が大きく揺れないか

人間が会話の途中で毎回補正しているなら、その補正条件はまだ Skill に入っていない。

## 6. `claude -p` は自動化の開始点ではなく判定境界である

### 6.1 非対話実行では結果を機械判定できる形にする

Claude Code は `-p` または `--print` で非対話実行できる。

```
claude -p \
  "Run the review-changes skill for src/example.sh" \
  --allowedTools "Read,Edit,Bash" \
  --output-format json
```

`--allowedTools` は指定した道具を確認なしで使えるようにし、`--output-format json` は結果、セッション識別子、利用量などを JSON で返す。特定の形式が必要なら `--json-schema` を併用できる。

たとえば、後段の処理が終了状態を判定するなら、自然言語の報告だけではなく、明示した列挙値を返す。

```
claude -p \
  "Review src/example.sh and return the requested result." \
  --allowedTools "Read,Edit,Bash" \
  --output-format json \
  --json-schema '{
    "type": "object",
    "properties": {
      "status": {
        "type": "string",
        "enum": [
          "unchanged",
          "passed",
          "failed",
          "approval_required",
          "input_error",
          "stopped"
        ]
      },
      "tests_run": {
        "type": "array",
        "items": { "type": "string" }
      },
      "summary": { "type": "string" }
    },
    "required": ["status", "tests_run", "summary"]
  }'
```

構造化出力にすれば、`status == "passed"` のときだけ後段へ進める。文章中に「一部のテストは失敗したが、おおむね問題ない」と書かれた場合でも、曖昧な成功判定を避けられる。

### 6.2 `--bare` で暗黙の読み込みを減らす

通常の `claude -p` は、対話セッションと同様に `CLAUDE.md`、Hooks、Skills、プラグイン、MCP サーバー、自動メモリなどを読み込む。実行端末ごとの差を減らしたい CI や単機能スクリプトでは、`--bare` を使い、必要な設定だけを明示的に渡す方法がある。

```
claude --bare -p \
  "Summarize the supplied diff and report compatibility risks." \
  --allowedTools "Read" \
  --output-format json
```

`--bare` は安全機能を自動的に強化する指定ではない。暗黙の設定を読み込まなくなるため、必要なシステムプロンプト、`--settings`、`--mcp-config`、`--agents` などを明示しなければならない。再現性を上げる代わりに、設定の供給責任が呼び出し側へ移る。

### 6.3 対話から非対話へ進める条件

| 判定対象 | 進めてよい状態 | 進めてはいけない状態 |
| --- | --- | --- |
| 入力 | 必須値を機械的に検査できる | 不足値を会話中の推測で補っている |
| 権限 | 必要な道具とコマンドを列挙できる | Bash や MCP 全体を無条件に許可する |
| 検証 | 終了コードや構造化出力で成功を判定できる | 「問題ありません」という文章に依存する |
| 停止 | 異常時に後続処理へ進まない | 警告を出しながら処理を継続する |
| 復旧 | 変更単位と復旧手段が決まっている | 途中状態から手作業で推測して戻す |
| 再実行 | 二重処理を避けられる | 同じ通知や登録を重複して作る |

対話では、人間が途中の異常に気づいて追加指示を出せる。非対話実行では、その補正が消える。自動化の基準は平均的な成功率ではなく、誤った入力を拒否し、失敗を成功として出力せず、停止後の状態を説明できることである。

## 7. サブエージェントと MCP は前段の判断を増幅する

### 7.1 役割の分離と作業場所の分離は別である

サブエージェントは、調査、実装、検品などの役割を独立した文脈へ分けられる。

.claude/agents/reviewer.md

```
---
name: reviewer
description: Review a proposed shell-script change without modifying files.
tools: Read, Glob, Grep, Bash
model: sonnet
---

Review the proposed change against CLAUDE.md and applicable rules.

Check POSIX compatibility, command-line compatibility, error propagation,
test coverage, and unexpected file changes.

Do not modify files. Report concrete findings with file paths and line numbers.
```

役割を分けても、サブエージェントは通常、メイン会話と同じ作業ディレクトリから開始する。複数のエージェントに編集させる場合は、同じファイルを同時に変更する競合が残る。

独立したリポジトリコピーが必要なら、エージェント定義へ次を加える。

`isolation: worktree` を指定したサブエージェントは、分離された Git worktree 内で Bash とファイル操作を行う。役割の分離は文脈の分離であり、作業場所の分離は別の設定である。

並列化する前に、比較可能な条件を固定する必要がある。

* 同じ起点コミット
* 同じ入力資料
* 同じ変更禁止範囲
* 同じ必須試験
* 同じ性能と互換性の評価軸
* 結果を統合する責任者

共通の評価軸がなければ、成果物の数は増えても、選択根拠は強くならない。

### 7.2 MCP はローカルな候補を共有状態の変更へ変える

MCP を使うと、Claude Code は外部サービスが公開する情報や操作へ接続できる。プロジェクト単位の HTTP サーバーは、たとえば次のように登録する。

```
claude mcp add --transport http \
  issue-tracker --scope project \
  https://example.invalid/mcp

claude mcp list
claude mcp get issue-tracker
```

`--scope project` で登録した接続定義は、プロジェクト直下の `.mcp.json` へ保存され、バージョン管理によって共有できる。接続後は `claude mcp list`、`claude mcp get`、Claude Code 内の `/mcp` で、接続先、適用範囲、承認状態、公開される道具を確認する。

MCP の導入で変わるのは、参照できる情報量だけではない。ローカルで作った候補を、課題、予定、文書、データベースなどの共有状態へ確定できる経路が増える。

| 操作 | 権限方針 |
| --- | --- |
| 課題の検索、状態確認 | `allow` の候補 |
| コメントや変更案の下書き | `allow` または `ask` |
| 課題更新、担当変更、ラベル変更 | `ask` |
| 公開、送信、削除、統合 | `deny` または個別の明示承認 |

サーバーへ認証できることと、サーバーが提供するすべての道具を自動実行してよいことは同義ではない。読み取り、下書き、書き込み、不可逆操作を分けて権限を設定する。

### 7.3 高度な機能は正しさを補正しない

各機能が増やすものを並べると、導入順序の意味が見える。

* Skills は、同じ判断を再利用する
* `claude -p` は、同じ判断を無人で実行する
* 予定実行は、同じ判断を時間方向へ反復する
* サブエージェントは、同じ判断を並列化する
* worktree は、作業場所を分離する
* MCP は、判断結果を外部サービスへ反映する

これらの機能は、前段の判断条件が正しいかを自動的に検証しない。明確な規則も、曖昧な規則も、そのまま反復回数と影響範囲を増やす。

曖昧な手順を Skill にすれば、曖昧な処理が再利用される。終了条件がないまま `claude -p` へ移せば、失敗を人間が補正する機会が消える。共通評価軸なしに並列化すれば、比較不能な成果物が増える。MCP の書き込み権限まで許可すれば、ローカルな誤りが共有された業務状態へ残る。

## 8. Claude Code を業務環境へ変えるのは境界設計である

Claude Code を継続的な業務へ組み込む順序は、機能の便利さではなく、前段の条件が後段の影響に耐えられるかで決める。

1. `CLAUDE.md` と rules で恒久条件を固定する
2. 作業ごとの目的、対象、維持条件、完了条件を対話で指定する
3. Plan mode で変更前の因果関係と影響範囲を確認する
4. Permissions と Hooks で実行経路を限定する
5. チェックポイント、Git、バックアップで復旧単位を作る
6. 入力、検証、停止、報告を定義できた工程だけを Skill にする
7. 異常終了を機械判定できる工程だけを `claude -p` へ移す
8. 共通評価軸を持つ作業だけを並列化する
9. 読み取りと書き込みを分けたうえで MCP へ接続する

この構成は、六つの境界として整理できる。

| 境界 | 決める内容 |
| --- | --- |
| 入力境界 | 何を事実と規則として読み込むか |
| 判断境界 | Claude Code が提案する範囲と、人間が確定する範囲 |
| 実行境界 | 使用可能な道具、コマンド、ファイル、外部サービスの操作 |
| 時間境界 | 単発、反復、再試行、終了条件 |
| 復旧境界 | どの単位で戻せるか、何が不可逆か |
| 責任境界 | 誰が差分、外部操作、公開結果を採用したか |

Claude Code が単発の対話支援から継続的な業務環境へ変わる境界は、自動実行できる機能の数では決まらない。何を読ませ、何を許可し、どこで止め、どの単位で戻し、誰が結果を採用したかを、利用者の注意力ではなく環境の構成として説明できるかで決まる。
