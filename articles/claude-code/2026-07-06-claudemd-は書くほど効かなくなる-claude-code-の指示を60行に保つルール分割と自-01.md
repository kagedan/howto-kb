Claude Code に同じ間違いを繰り返させたくなくて、気づくたびに `CLAUDE.md` へ注意書きを足していました。ところがある時期から、前に書いたはずのルールをすり抜ける場面が増えます。新しく足したルールは効くのに、古いものが忘れられていく。

原因は「書き方」ではなく「量」でした。`CLAUDE.md` はセッションのたびに丸ごと読み込まれ、その中身は目の前のタスクと同じ土俵でコンテキストの枠を奪い合っています。指示は足せば効くのではなく、増やすほど一つひとつの重みが薄まる。全部を毎回読ませると、どれも中途半端にしか効かない状態へ近づいていく。

だからこの記事で書くのは「何を `CLAUDE.md` に書くか」ではなく、**何を常時読ませ、何を必要なときだけ引くか**の設計です。常時読ませる面は薄く保ったまま、知識そのものは減らさない。そのために使っている3つの仕掛け——60行上限とルールの切り出し、二層のメモリ、同じ失敗を繰り返させない自己更新ループ——を順に書きます。この一式は、Claude Code を1つのリポジトリで運用する仕組みの一部です（[全体像はこちら](https://zenn.dev/takotyan00/articles/claude-code-life-repository)）。

## 「常時読ませる」には値段がある

`CLAUDE.md` の中身は、毎セッション、何もしなくてもコンテキストに載ります。これは便利な反面、載せた分だけ「今から解く問題」に使える枠を先に食っている、ということでもあります。

外部ツール（MCP）を有効化しすぎると使えるコンテキストがごっそり減る、という話は知られていますが、同じことが指示ファイルでも起きます。しかもタチが悪いことに、量が増えると容量を食うだけでなく、**個々の指示に向く注意が薄まる**。10個の中の1行と、50個の中の1行は、同じ文でも効き方が違う。私のグローバル設定も、気づけばルールが積み上がって、肝心の一行が埋もれていました。

|  | 太った CLAUDE.md | 薄い CLAUDE.md |
| --- | --- | --- |
| 毎回のコスト | 全ルールを常時ロード＝枠を圧迫 | 目次と最重要だけ＝軽い |
| 個々の指示の効き | 埋もれて薄まる | 立って効く |
| 知識の総量 | 多いが死蔵しがち | 同じ量を別の層に逃がす |

正解は、知識を捨てることではありません。**「常時」から「必要時」へ動かす**ことです。以降の3つは、置き場所を変えるだけで総量は減らさない、という同じ発想の変奏です。

## 60行に区切って、詳細は `.claude/rules/` へ出す

まず `CLAUDE.md` 自身に、たとえばこんな趣旨の一行を入れています。

> このファイルは60行以内に保つ。超えたら詳細は `.claude/rules/` へ分割し、反復手順は Skill へ切り出すこと。

ファイルが自分の肥大化を監視する、という趣向です。60という数字自体に根拠はなくて、「常時ロードして負担にならない上限」を1つ決めておくのが目的です。超えたら中身を消すのではなく、外へ出す。

出し先が `.claude/rules/` です。ここに置いたルールは常時ロードされず、Claude が「今このルールが要る」と判断したときだけ開きます。常時面に残すのは、本体ではなく1行のポインタだけ。

```
.claude/
├── CLAUDE.md            # 常時読む・60行以内。目次と最重要ルールだけ
└── rules/               # 必要時だけ開く詳細ルール
    ├── ml-environment.md     # 機械学習まわりの環境制約
    ├── design-anti-slop.md   # UI 実装時に避けるパターン
    ├── code-health.md        # コード品質を保つ運用ルール
    └── ...
```

ここに落とし穴が1つあります。ただ切り出しただけだと、Claude はその `rules` が存在することに気づけません。分割と引き換えに、`CLAUDE.md` 側へ「いつ読むか」のトリガーを1行残す必要があります。たとえば「UI を実装するときは `design-anti-slop.md` を参照」のように、条件と参照先をセットにしておく。トリガーの無い分割は、ただの行方不明です。

これで、常時コンテキストは薄いまま、詳細は該当する場面でだけ厚くなります。ルールの総数は増やしていい。増えて困るのは、常時読ませる面の厚みだけだからです。

この `.claude/rules/` に実際どんなルールを置くかは、[AI に私を読み違えさせない補正ルール](https://zenn.dev/takotyan00/articles/claude-code-correction-rules)の記事で一例を掘り下げています。

## 覚えておきたい事実は「索引付き」で貯める

ルールとは別に、セッションを跨いで覚えておいてほしい事実があります。好み、過去に下した判断、繰り返し出てくる文脈。これらを `CLAUDE.md` に書けば太るし、書かなければ毎回説明し直しになる。どちらも避けたい。

そこでメモリを二層に分けています。

* **索引層**：常に読む。厳選した事実だけを1行ずつ並べた目次（`MEMORY.md`）。
* **本体層**：必要時だけ引く。1事実1ファイルで貯めるローカルストア。**索引には載せず**、下記のフックが関連するときだけ引く。数が増えても常時コストはゼロ。

本体層を「必要時だけ」引く仕掛けが、プロンプト送信のたびに走るフック（`UserPromptSubmit`）です。送信された文と、各メモリファイルの `tags` / `description` をキーワード照合し、関連する上位だけをその場でコンテキストに差し込む。無関係な話題のときは、何も差し込まない。

中身は素朴で、tags と説明語がプロンプトに何個重なるかを数えているだけです。

```
# プロンプトと各メモリの tags/description を突き合わせ、
# 関連する上位だけを注入する（無関係なら何も出さない）
for name in os.listdir(mem_dir):
    fm, body = parse(read(name))              # frontmatter と本文
    score = 0
    for term in fm["tags"] + words(fm["description"]):
        weight = 2 if term in fm["tags"] else 1   # tags を厚く見る
        if match(term, prompt):                   # CJK は部分一致 / 英字は語境界一致
            score += weight
    if score >= 1:
        hits.append((score, fm, body))

for fm, body in top(hits, n=5):               # 上位5件だけ・本文は頭だけ
    inject(fm["date"], fm["description"], body[:600])
```

だから、貯めるときは `tags` が命になります。呼び出しはこの tags 照合が入口なので、ここが雑だと、せっかく書いた事実は二度と引っかからない。

```
---
date: 2026-07-06
tags: [料理, 段取り, 好み]
description: 平日夜の献立は「品数より一皿の満足度」を優先する
---

（本文：やり取りの要点。tags と description が呼び出しの鍵）
```

素朴な照合なので、日本語は部分一致、英字は語境界一致にして誤爆を抑えています（「くま」が「あくまで」に当たる類のノイズを避ける）。そして差し込まれた内容は、あくまで**参考情報**として扱う建て付けにしています。過去の蓄積であって、その場の確定事実ではない。断定の根拠にする前には、必要なら本人に確かめる。覚えていてほしいけれど、毎回は読ませたくない——この二層は、その我儘をそのまま形にしたものです。

## 「同じ失敗を繰り返すな」を仕組みにする

ここまでの2つは、知識を薄い層と厚い層に振り分ける話でした。最後は、その振り分けを**手作業の善意に頼らない**話です。

グローバル設定には、こういうルールを置いています。「修正を受けるたびに `CLAUDE.md` へ追記して、同じミスを繰り返さないこと」。ただし、これを素直にやると `CLAUDE.md` が太る。冒頭の問題にまっすぐ逆戻りします。追記の圧力と、薄く保つ要請が、正面からぶつかる。

解いた方法は2つあります。追記を「気づいたら書く」から仕組みへ移すこと。そして、書く先を `CLAUDE.md` に固定しないこと。3つの部品でできています。

1. **検知**：セッションが終わるとき、その回に訂正やルール化の合図があったかをフックが拾い、印を残す。
2. **催促**：次にセッションを始めるとき、別のフックが「未処理の学びが N 件あります」と促す。
3. **処理**：`/reflect` というコマンドが、その回の訂正・新しく決めた運用・繰り返しそうなミスを抜き出し、**正しい保存先へ振り分けて**差分で見せ、承認したら反映する。

肝は3番の「振り分け」です。学びを何でも `CLAUDE.md` に足すのではなく、性質ごとに行き先を変える。

| 学びの性質 | 行き先 |
| --- | --- |
| どのプロジェクトにも効く自分の癖 | 全体の個人メモリ |
| このリポでの振る舞い（軽量・普遍） | `CLAUDE.md`（60行厳守・溢れたら rules へ） |
| 詳細なルール・判断基準・反復手順 | `.claude/rules/` |
| 生の対話ログ | ローカルメモリ |

これで、学び続けても `CLAUDE.md` だけが太る、という事故が起きにくくなります。追記の行き先が最初から分散しているからです。そしてもう一つ、この処理は**勝手に書き換えません**。必ず差分を出して、承認を挟む。設定ファイルが自分の意図しない方向へ育っていくのは、指示が薄まるのと同じくらい怖いので、そこは人間の一手を残しています。

## 効くかどうかは、書いた量ではなかった

3つの仕掛けは、見た目は別々でも、やっていることは1つです。常時読ませる面を薄く保ち、知識は「どこに置き、いつ引くか」で設計する。指示は書けば効くのではなく、限られた注意をどう配るかの**予算配分**でした。

* ルールは、常時の1行ポインタと、必要時に開く本体に分ける。
* 事実は、常時の索引と、tags で引く本体に分ける。
* 学びは、性質ごとに4つの層へ振り分け、`CLAUDE.md` を太らせない。

`CLAUDE.md` を育てるのをやめて、置き場所のほうを設計しはじめてから、Claude が指示どおりに動いてくれる場面が増えました。効くかどうかは、書いた分量ではなく、常時と必要時をどう切り分けたかで決まっていた。

もちろん完成形ではありません。tags の付け方次第でメモリは引っかからないし、細かく分割しすぎれば今度は Claude がどの `rules` を開くべきか迷います。薄さと網羅はいつでもトレードオフで、その境目を手で調整し続ける前提の仕組みです。それでも、「全部を常に読ませる」よりは、ずっと素直に効くようになりました。
