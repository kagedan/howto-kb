## 今日のゴール

日常的に使うスラッシュコマンドとPermission Modeの仕組みを理解し、「コンテキスト」という考え方を使って会話を上手にコントロールできるようになる。

## 座学

### コンテキストとは何か

Claude Codeとの会話は、やり取りを重ねるごとに「これまでの会話内容」がすべて次のやり取りに引き継がれていきます。この引き継がれる情報の集まりを「コンテキスト」と呼びます。  
コンテキストには上限(コンテキストウィンドウ)があり、会話が長くなるほど、あるいは大きなファイルを読み込むほど、この上限に近づいていきます。

コンテキストが上限近くまで埋まってくると、Claude Codeは古い指示を忘れたり、同じ間違いを繰り返したりするなど、応答の精度が落ちていく傾向があります。Anthropic公式のベストプラクティスでも以下のように述べられています。

> ほとんどのベストプラクティスは、ある一つの制約に基づいています。それは、Claudeのコンテキストウィンドウがすぐにいっぱいになり、いっぱいになるにつれてパフォーマンスが低下するということです。

<https://code.claude.com/docs/ja/best-practices>

つまり、便利なコマンドを覚えることは、実質的に「コンテキストをどう手入れするか」を覚えることに直結します。

### 会話をコントロールするスラッシュコマンド

Claude Codeでは、`/`から始まる特別な指示(スラッシュコマンド)を使って、会話そのものを操作できます。まず覚えておきたい4つを紹介します。

| コマンド | 用途 |
| --- | --- |
| `/help` | 利用できるコマンドの一覧を表示する |
| `/clear` | 会話をリセットし、新しい会話を始める(作業フォルダの設定などは引き継がれる) |
| `/compact` | ここまでの会話を要約し、コンテキストの空き容量を作る |
| `/context` | 現在コンテキストがどれくらい埋まっているかを可視化する |

`/context` を時々実行してコンテキストの残量を確認し、埋まってきたと感じたら `/compact` で要約する、あるいは話題が変わるタイミングで `/clear` して仕切り直す、という運用が基本になります。

`/compact` は、続きの作業に関係する部分だけを残して要約したい場合、`/compact 直近のファイル整理の作業に絞って要約して` のように、何を残したいかを指示に添えることもできます。

### 「脇道の質問」を汚さずに聞く

作業の途中で、本筋とは関係ない小さな疑問が浮かぶことはよくあります。そうした質問をそのまま聞いてしまうと、コンテキストに本題と関係のないやり取りが溜まっていきます。`/btw` コマンドを使うと、その場限りの質問として処理され、通常の会話履歴には残りません。

```
/btw さっき出てきたPermission Modeって、Plan Modeと何が違うんでしたっけ
```

「ついでに聞きたいこと」はいったんこちらで済ませ、本筋のコンテキストをきれいに保つ、という使い分けを意識しましょう。

### 画面を見せて相談する — 画像貼り付けと`@`ファイル参照

Claude Codeへの入力は文章だけではありません。管理コンソールのエラー画面やダッシュボードのスクリーンショットをそのまま貼り付けて、「この画面のエラーの原因を調べてください」と相談することができます。文章で状況を説明し直す手間が省け、特に見た目の確認が必要な作業では効果的です。

また、`@` に続けてファイル名を入力すると、フォルダ内のファイルを指定して読み込ませることができます(例: `@report.md の内容を要約して`)。「このフォルダ全体を見て」ではなく、必要なファイルだけを`@`で指定することも、コンテキストを無駄に消費しない工夫の一つです。

### Permission Mode — Claude Codeにどこまで任せるか

Claude Codeは、ファイルの変更やコマンドの実行といった「実際に何かを行う」操作を行う前に、どこまで確認を求めるかを選べます。この設定をPermission Modeと呼びます。公式ドキュメントでは、次の4つのモードが紹介されています。

<https://code.claude.com/docs/ja/how-claude-code-works>

| モード | 動作 |
| --- | --- |
| Default | ファイルの変更やコマンド実行のたびに確認を求める。最も慎重なモード |
| Auto-accept edits | ファイルの編集や `mkdir`・`mv` など基本的なコマンドは確認なしで自動的に実行する。それ以外のコマンドは確認を求める |
| Plan mode | 実際の変更は一切行わず、まず「何をするつもりか」という計画だけを提示する |
| Auto mode | バックグラウンドの安全性チェックを踏まえてほぼすべての操作を自動実行する、リサーチプレビュー段階のモード |

モードの切り替えは、キーボードの `Shift+Tab` で行います。押すたびに Default → Auto-accept edits → Plan mode → Auto mode の順に切り替わっていきます。第1回のハンズオンで「ファイルを保存してよいか確認された」場面があったと思いますが、あれは Default モードの挙動です。Auto modeはまだ実験段階の機能のため、業務での常用は避け、まずはDefault・Auto-accept edits・Plan modeの3つを使い分けられるようになることを目指しましょう。

### Plan Modeの価値

Plan modeは、特に影響範囲の大きい作業や、初めて触るフォルダで作業する際に有効です。Claude Codeがまず「これから何をするか」を文章で提示し、それに対して承認するかどうかを判断できるため、意図しない変更を未然に防げます。セキュリティ診断のようにミスが許されない業務との相性がよいモードです。

### 「壊しても戻せる」を知っておく — チェックポイントと`/rewind`

Permission Modeについて学ぶと、「間違った変更を許可してしまったらどうしよう」という不安が出てくるかもしれません。実はClaude Codeは、ファイルを編集する前に必ずその時点の内容を自動的にスナップショット(チェックポイント)として保存しています。想定外の変更が行われてしまった場合は、`Esc`キーを2回押すか、`/rewind`(別名`/checkpoint`)コマンドで、直前の状態まで巻き戻すことができます。

<https://code.claude.com/docs/ja/how-claude-code-works#stay-safe-with-checkpoints-and-permissions>

この「いつでも元に戻せる」という安心感は、Permission Modeの理解と同じくらい重要です。

最初のうちは、確認を求められた変更を許可することに慎重になりすぎず、「もし違ったら`/rewind`すればいい」という前提で、積極的に試してみることをお勧めします。

ただし、チェックポイントが巻き戻せるのはファイルの変更のみで、外部システム(データベースやAPIなど)に影響するコマンドの実行結果までは戻せない点には注意してください。

## ミニハンズオン

### 手順1: コンテキストの残量を見てみる

前回作成した `~/claude-work` フォルダでClaude Codeを起動し、いくつかやり取りをしたあとに次を実行します。

```
# 前回のフォルダがない人向け
mkdir ~/claude-work
cd ~/claude-work
claude

# コンテキストの使用状況を確認
/context
```

コンテキストの使用状況が表示されます。会話を始めた直後と、いくつかファイルを読み込ませたあとで、どれくらい変化するかを見比べてみましょう。

### 手順2: Plan Modeを試す

`Shift+Tab` を2回押して `plan` モードに切り替えます。その状態で、次のように複数ファイルに関わる依頼をしてみます。

```
# 1個目の指示
中に適当なコメントを残したコメントとランダムな日付を付与したメモをサンプルのファイルに生成してください

# planモードへの切り替え

# 2個目の指示
このフォルダの中のメモをすべて日付ごとのファイルに整理し直してください
```

Claude Codeが実際に手を動かす前に、どのような手順で進めるつもりかという計画を提示することを確認してください。計画に問題がなければ承認し、実行に移ります。

### 手順3: /compactで要約してみる

ある程度やり取りが進んだところで、次を実行します。

```
/compact ここまでのファイル整理の内容だけ残して要約して
```

要約後に `/context` をもう一度実行し、コンテキストの空き容量が回復していることを確認しましょう。

### 手順4: チェックポイントで元に戻してみる

`Shift+Tab` で Auto-accept edits モードに切り替え、適当なメモファイルの内容を書き換えてもらいます。

```
このメモの見出しをすべて英語に翻訳して書き換えてください
```

変更が反映されたら、`Esc` キーを2回押すか `/rewind` と入力し、変更前の状態に戻せることを確認しましょう。「試して、気に入らなければ戻す」という進め方に慣れておくと、以降の回で扱う操作にも安心して挑戦できるようになります。

## まとめ+チーム資産化アクション

* コンテキストは有限であり、Claude Codeの応答品質は「今どれだけコンテキストが埋まっているか」に左右される
* `/context` で残量を確認し、`/compact` で要約、話題が変わったら `/clear` で仕切り直すのが基本の運用
* Permission Modeを `Shift+Tab` で切り替え、影響が大きい作業は Plan mode で計画を確認してから進める
* ファイルの変更は`/rewind`(`Esc`を2回)でいつでも元に戻せるため、過度に慎重にならず試行できる

**チームへの持ち帰りアクション**: 自分がよく使う作業パターンについて、「このタイミングでは Plan mode を使う」というマイルールを1つ決め、次回までに実践してみてください。次回はこのフォルダに置く `CLAUDE.md` を使って、こうしたルールをチームで共有できる形にしていきます。

## ベストプラクティスのひとこと

公式のベストプラクティスでは、同じ問題に対する修正指示を2回試してもうまくいかない場合、3回目を試すのではなく `/clear` していったん仕切り直し、より具体的な指示を出し直すことが推奨されています。これは、うまくいかない修正のやり取り自体がコンテキストに残り続け、かえって次の試行の精度を下げてしまうためです。

| うまくいかないパターン | 推奨されるパターン |
| --- | --- |
| 同じ会話の中で3回目、4回目の修正を試み続ける | 2回試してだめなら `/clear` し、状況を整理した具体的な指示で仕切り直す |
