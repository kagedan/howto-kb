## はじめに

嫁さんが店舗のシフトを手で作成してると聞き興味を持ったが放置していた数理最適化。  
全体像を把握するために、LLM（Claude）に手伝ってもらいながら複雑な工場のシフトのシミュレーションデータを作成し、最適化に挑戦してみました。  
**この検証の仕方であってるかはわかりません。**

やったこと:

* 合成データで工場シフト問題を3難易度で作成
* ランダム、貪欲法、ソルバー、LLM直接、ハイブリッド、ヒューリスティック進化、メタヒューリスティクスなど**25手法**を実装・比較
* 最も難しいHard問題（40名/28日/10制約）に**14手法**がfeasible達成

先に結論を書いておきます。

### 結論

**LLMの最大の価値は「解くこと」ではなく「設計すること」にある。**

LLMが直接シフト表を作ると大規模問題は解けない。しかし目的関数の設計（+15%品質向上）、ヒューリスティック戦略の進化、自然言語からのコード自動生成で、ソルバー単体を大幅に上回った。

最高品質はLLMにソルバーの目的関数を設計させるハイブリッド手法で、ベースラインのソルバー単体比\*\*+27%\*\*を達成した。一方、単純な貪欲法は900回試して1回もHard問題を解けなかったが、SWO+IFS（バックトラック）の組合せではソルバーなしでも解けた。

## 問題の説明

### 工場シフト最適化とは

以下の要素を組み合わせて、最適なシフト表を作る問題。

```
【作業者】         【シフト】        【工程】
40名              朝勤 06:00-14:00   組立（Assembly）
├─ スキル 3〜5種   昼勤 14:00-22:00   溶接（Welding）
├─ 習熟度 0〜5     夜勤 22:00-06:00   検査（Inspection）
├─ 得意工程                          機械操作（Machine Op.）
├─ 苦手工程                          梱包（Packaging）
└─ 希望シフト                        フォークリフト（Forklift）
                                     品質管理（QC）
```

**習熟度の6段階**:

| レベル | 意味 | 例 |
| --- | --- | --- |
| 0 | 不可 | その工程を行えない |
| 1 | 初心者（要指導） | 先輩の監督が必要 |
| 2 | 基本（低速） | 遅いが一人で作業可能 |
| 3 | 中級（通常速度） | 標準的な作業速度 |
| 4 | 上級（高速・高品質） | 安全監督者になれる |
| 5 | 熟練（指導可能） | 他者の指導が可能 |

### 評価指標

各手法の良し悪しを以下の指標で評価した。

| 指標 | 意味 | 良い方向 |
| --- | --- | --- |
| **Feasibility** | 全ハード制約を満たしているか（1=OK, 0=NG） | 1が必須 |
| **Hard Violations** | ハード制約の違反数 | 0が必須 |
| **Soft Score** | ソフト制約の重み付き総合スコア | 高いほど良い |
| **Proficiency** | 全配置の習熟度レベルの合計 | 高い=適材適所 |
| **Total Cost** | 総人件費（円） | 低い=コスト効率 |
| **WLStd** | 作業者間の労働時間の標準偏差 | 低い=公平 |
| **Pref%** | 希望シフトの充足率 | 高い=満足度 |

**Feasibility（実行可能性）が最優先**。ハード制約を1つでも破るシフト表は使えない。Feasibility=1を達成した上で、Soft Scoreが高い解が良い解となる。

Soft Scoreはソフト制約SC1〜SC8の重み付き合計で計算される。たとえばSC1（習熟度最大化、重み10）の寄与は `配置された全作業者の習熟度合計 × 10`、SC2（コスト最小化、重み8）の寄与は `-総コスト/10000 × 8` のように、各制約を統一スコアに変換して合算する。

## 従来手法の解説

シフト最適化はどう解かれてきたか。初学者向けに簡単にまとめる。

### 厳密解法（Exact Methods）

**混合整数計画法（MIP）/ 制約プログラミング（CP）**

問題を数式で定式化し、ソルバー（Gurobi、OR-Tools等）で厳密に解く。最適解が保証されるが、定式化に専門知識が要る。

```
最小化: Σ（コスト × 割当変数）
制約:   各シフトの人数 ≥ 必要人数
        各作業者の週時間 ≤ 上限 ...
```

**列生成法（Column Generation）**

作業者の「勤務パターン」を生成し、最良の組合せを選ぶ。大規模問題（100名以上）に強い。看護師ロスタリングで多くの成功例。

### メタヒューリスティクス

最適性は保証しないが、大規模問題に実用的な近似解を得る。

| 手法 | 仕組み |
| --- | --- |
| **遺伝的アルゴリズム（GA）** | 複数の解を交叉・突然変異で進化させる |
| **焼きなまし法（SA）** | 悪化も確率的に受理し局所最適を脱出 |
| **タブー探索** | タブーリストで循環を防ぎ系統的に探索 |
| **大近傍探索（LNS）** | 解の一部を壊して再構築を繰り返す |

### 構成的ヒューリスティクス

| 手法 | 仕組み |
| --- | --- |
| **貪欲法** | その時点で最良の選択を繰り返す。シンプルだが「地平線効果」で大問題は解けない |
| **SWO** | 違反箇所を検知→優先度を上げて再構築を繰り返す反復手法 |
| **Ejection Chain** | 連鎖的な再配置で、1回のスワップでは解けない局面を解消 |

### LLMベースの新手法（2023年〜）

| 手法 | 仕組み |
| --- | --- |
| **LLM直接生成** | LLMにデータを渡して直接シフト表を出力。小規模なら動くが大規模は困難 |
| **LLM+ソルバー ハイブリッド** | LLMが目的関数を設計、ソルバーが解く。OptiGuide (Microsoft) が先駆 |
| **LLMヒューリスティック進化** | LLMがアルゴリズム自体を進化。FunSearch (Nature, 2023)、EoH、ReEvo、RoCo |
| **NL→ソルバーコード** | 自然言語からソルバーコードを自動生成。OptiMUS (ICML, 2024) |

### 手法の位置づけ

```
                    解の品質保証
                    ↑
         厳密解法    │    LLMハイブリッド
        (MIP, CP)   │    (OptiGuide, OptiMUS)
                    │
    ─────────────────┼─────────────────→ スケーラビリティ
                    │
   メタヒューリスティクス    LLMヒューリスティック進化
   (SA, GA, Tabu, LNS)    (FunSearch, EoH, ReEvo)
                    │
                    │    構成的ヒューリスティクス
                    │    (Greedy, SWO, Ejection Chain)
```

## 実験結果

### 結果の読み方

結果テーブルの見方を先に説明する。

* **Feas**: Feasibility。全ハード制約を満たしているか。`o`=OK（違反0）、`x`=NG（1つ以上違反）
* **Soft Score**: ソフト制約の重み付き総合スコア。高いほど良い。**Feas=oの解の中で比較する**のが正しい見方
* **WLStd**: 作業者間の労働時間の標準偏差。低いほど公平なシフト
* **Pref%**: 作業者の希望シフトがどれだけ反映されたか

### Easy難易度（10名/7日/3HC/2SC）

制約が少なく余裕がある問題。**全手法がfeasible**を達成した。差が出るのはSoft Scoreの品質。

## わかったこと — 7つの発見

### 発見1: LLMの最大の価値は「解く」ことではなく「設計する」こと

LLMが直接シフト表を作るとHardは解けない。しかし:

* **目的関数の設計**で+15%の品質向上（expG13 vs expC00）
* **ヒューリスティック戦略の進化**（EoH, FunSearch）でMedium難易度の高品質解
* **自然言語での問題理解**（OptiMUS）で公平性WLStd=9.03を達成

### 発見2: 目的関数と評価関数の一致が決定的に重要

expG13/G14の成功要因は「評価関数を正確にソルバー目的関数にマッピング」したこと。

```
従来: ソルバー目的関数 ≈ 評価関数（近似）→ soft=19,777
改善: ソルバー目的関数 = 評価関数（精密一致）→ soft=22,811（+15%）
```

「賢い重み付け」より「正確な再現」が+15%の差を生んだ。さらにソルバーは「過剰配置は得」という人間が思いつかない戦略を自動発見した。

### 発見3: 単純な貪欲法ではHardは解けない（証明済み）

expG10で900通りの順序・シードで貪欲法を実行。**900回中0回がfeasible**。

理由は「地平線効果」: 早い段階の割当が後のスロットの容量を食い潰す。グローバルな推論なしにはHard問題は原理的に解けない。

### 発見4: ソルバーなしでもHardは解ける

expG15が**ソルバーを一切使わずにfeasibleを達成**（soft=18,908）。

必要だったのは:

* **SWO**: 問題箇所を検知して優先度を上げる反復フィードバック
* **IFS**: 割当の巻き戻しと再試行（バックトラック）
* **ドメイン知識**: 訓練ペアリング（HC9）を最優先で処理

### 発見5: 問題の分解の仕方で結果が激変

| 分解方法 | 手法 | 結果 | 理由 |
| --- | --- | --- | --- |
| 時間分割（週単位） | expG03 | 成功 | 制約の構造に沿った分解 |
| ロスター分解 | expG09 | 大失敗（69違反） | 「いつ」と「何を」の分離がスキル制約を破壊 |

**教訓**: 分解は制約の構造に沿って行う必要がある。

### 発見6: 各手法に得意分野がある

| 目的 | 最適手法 | 根拠 |
| --- | --- | --- |
| 最高品質 | expG13 | soft=22,811（全実験中1位） |
| 公平性 | expF03 OptiMUS | WLStd=9.03（Medium） |
| 希望充足 | expF03 OptiMUS | Pref=61.3%（Medium） |
| ソルバー不要 | expG15 SWO+IFS | Python標準ライブラリのみ |
| 費用対効果 | expC00 Hybrid | 全難易度feasible + 高品質 |

### 発見7: Feasibilityの境界線

```
         【feasible達成の共通点】
         グローバル推論能力がある
         ├─ ソルバー（制約伝播+バックトラッキング）
         ├─ SWO + IFS（反復フィードバック+巻き戻し）
         └─ マルチエージェント協調

         【infeasibleの共通点】
         局所的操作のみ
         ├─ 単純貪欲法（900回全敗）
         ├─ GA/SA（局所探索のみの場合）
         ├─ FunSearch（スコア関数改善のみ）
         └─ LLM反復修復（5違反まで減らすが最後が解消不可）
```

## おわりに

数理最適化の初学者でも、LLMの力を借りれば25手法を実装・比較できた。最も重要な学びは「LLMは問題を解くのではなく、問題の解き方を設計するのが得意」ということ。ソルバーに「何を最適化すべきか」を正確に伝えるだけで+27%の品質向上が得られた。

一方で、単純な貪欲法では900回試しても解けない問題が、SWOとバックトラックの組合せならソルバーなしでも解けることがわかった。手法の選び方次第で結果は劇的に変わる。
