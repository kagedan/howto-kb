# 思いつきは突然でした。
Qiitaをご覧の方は皆様お忙しいと思いますので、この記事に興味がありましたらお気に入りのAIに”要約して”とお伝えくださいm(_ _)m

長々とたいした事ない文章を書いているだけです^^;

まずは、私の背景から・・・
地方の小さな会社で今で言うDXというのを導入しようとしたら、導入まで長くなりますのでお時間のある方はお付き合い下さいm(_ _)m

もう30年前か？^^;
高校生の偏差値は50に届くかどうかという状況で、とある地方大学の工学部になんとか滑り込んで学んだ後、社会人になってから紆余曲折あって15年ほど前に今の会社に入ってからはなんとなく過ごしていました。
過去の記事後ご覧になられた方はなんとなく察しておられるかもしれませんが、勤め先は地方の小さなEC事業を行っている会社です。

入社した当初は在庫管理が手書きのノートという、さすがの私も"昭和の会社ですか？"と思うほどの状況だったのですが、アレコレあって受注管理システム(Order Management System = OMS：ネクストエンジン)を導入する運びになりました。
私が入社してから4年後の2014年頃だったかと思います。

手書きの在庫帳がありえないと思っていた私にとって、OMSの導入は”そらそうだろ”と思うところがあって、導入担当者になってなんやかんややりながら何とか導入にこぎつけたように思います。
(なにせ10年以上前なので大変だった記憶はありますが詳細を覚えていません(-_-;)

WMSの導入もやりましたし、倉庫建設に携わって重量ラックの導入や、フォークリフトの選定を行ったりもしましたし、運送会社の導入もやりました。
ってなことをAIに職務経歴書を書かせようと思いやり取りしていたところ

情シス(情報システム部門) → Sler(システムインテグレーター) → DX推進 → ビジネスアーキテクト

と言われたのはおそらく過大評価でしょう^^;

所属は在庫管理の部署で一応長をやっておりますので、OMSを導入出来たら販売可能数の共有を何らかの形で行いたくなるのは必然かと思います。

というものの、私が入社前の社内で在庫情報の共有は手書きの在庫帳というレベルでしたが、お金がない会社でしたので情報の共有を行うために無料でGoogleドライブを導入することになったのもさもありなん・・・

じゃあ、OMSから在庫情報を手動でダウンロードして、スプレッドシートにアップロードしたら、販売可能数(フリー在庫数)を共有できるでしょ？
と思い、スプレッドシートの作成を行いました。

当初は、人力で1日3回の更新を行っておりました。

ですが、めんどくさい。
モールのセールがあったら忙しくなって時間通りに更新できない・・・

じゃあ、人力で更新しない方法を探さないといけない・・・

お金がある会社であれば色々あったであろうし、今どきであればハイパーオートメーションツールやEAIといったものがありますが、そういうのにお金を払いたくない会社なので無料若しくは小コストでなんとかするしかなかったのが当時の現状でした。

## なんとなく自動化
手動での更新の手間と更新忘れを解決すべく行ったのが、OMSであるネクストエンジンの機能にあるカスタムデータのアップロードスケジュールでした。

有料ではあるものの、希望の情報(商品コード、商品名、在庫数・・・)という情報(フィールド)を指定して、指定したSFTPサーバーへ定期的にアップロードできる仕組みとなります。

しかし、うまく動作するかわからない状態で、固定グローバルIPアドレスを取得できるお金があるわけでもなく・・・
倉庫で使っていたルーターを導入したのは私でして、幸いにしてTP-Linkであったことから、グローバルIPアドレスが変わっても、NO-IPというサービスを使用することでSFTPサーバを立ち上げられる事がわかりました。
立ち上げと言っても、無料のアプリをどうさせてポート番号などを決めるだけでしたので、そんなにハードルが高いわけではありませんでした。

人力で行っていた在庫情報の社内共有が自動化出来た(^^)
コレで数年使っていました。

## リアルタイム性の追求
SFTPサーバーを使った在庫情報の共有が自動化なのか？と言われると疑問があります(-_-;)
ECをやっていて、1日3回しか更新しないのか・・・・(-_-;)

なので、更新頻度を上げるべく私で出来ることを探した結果、Tepsというアプリを使用することになりました。
こちらはノーコードツールに分類され、コーディングをすること無く、任意の動作をクラウドで行ってくれるというアプリでした。
Tepsのサービスが始まった直後から使用していたことで、色々良くしてくれていましたので、在庫情報の更新頻度を1日3回から1時間に1回まで高めたように思います。
また、Tepsだけでは取得した在庫情報の商品コードがわからないという事で、SFTPサーバーを利用した在庫情報のアップロードは継続して行ったり、Google Apps Script = GAS を利用した在庫情報の書き換えを補助として使っていたりしました。

## コスト上昇による退化
そのような状況がいつまでも続くわけでもなく、2013年頃に値上げの話が出てきたのですが、お金がないウチの会社ではその値上げは許容できるはずもなく、無料プランへと退化することになりました。
まぁ、ボランティアでやってるわけでないですからね・・・

# じゃあAIにコーディングさせればいいじゃんか！
OMSであるネクストエンジンと社内の情報共有を行うGoogleスプレッドシートを接続したいという思いはOMS導入当時からあったのですが、そんな技術は私にあろうはずもなく、出来る範囲でアレコレやっておりました。

というものの、なんとかしたいという思いは持ち続けていましたので、その要望が高まったり静まったりしながら、お金がない中何年も経過していたのですが・・・

忘れもしない2025年8月17日に突然
**令和の今どきならAIがコーディングできるやろ！**
と思い、始めたのがバイブコーディングでした。

当時はAIにコーディングをさせるバイブコーディングという言葉すら知らなかったのですが、思いつく半年前である2025年2月にコンピューター科学者Andrej Karpathyが提唱していたそうですね。

## バイブコーディングの壁
AIはそういう言語との親和性が高いはずなので、ネクストエンジンAPIの認証の壁なんて簡単に突破できるでしょ・・・

と思っていましたが、2週間の間は前に進みませんでした。

当時はAIのモデルもそれほど精度が高くなかったと思いますし、私のプロンプトエンジニアリングも上手くなかったと思います。

なにより、そういうのに得意なAIに出会っていなかったというのが大きかったと思います。

### 使用したAI
どのAIをどの順番で使用したかは詳しく覚えていませんが、

* GPT
* Gemini
* Github Copilot

を使用したと思いますが、ネクストエンジンAPIの認証の壁を突破することは出来ませんでした。

そこで2025年9月1日にGeminiにコーディングの得意なAIを聞いたところ、Claudeを紹介されたのが転機となりました。

### Claudeのチャットでのコーディング
お金があったらClaude Codeが使えたのでしょうがね(-_-;)

AIに払うお金もない会社ですので、Claudeの無料枠でのチャットでコーディングを行うことにしました。
2025年9月1日時点のモデル比較ですが、Claudeは他のモデルよりあからさまに秀でていました。

他のAIとやり取りして上手く行かなかった経緯を伝えると、ネクストエンジンAPIの仕様を確認すると回答があり、画像を認識したうえでコーディングが始まりました。

いきなり上手く言ったわけではありませんでしたが、他のAIでは2週間ほど前に進まなかった認証が、Claudeではわずか2時間で突破したように思います。

それからは、手動でネクストエンジンからCSVダウンロードしていたものを、ネクストエンジンAPIからGASで取得し、スプレッドシートに転機するというのが一気に進みだしたように思います。

その後、初めてのプロジェクトである在庫情報の取得は、3000行超えの情報を20秒弱でスプレッドシートへ転機出来るようになりました。

### Claudeチャットでのトークン消費
というものの、Claudeも無料で使っていると万能ではなく、指示してコーディングに入ってもらったとしてトークンを使い果たして、途中まで作成してくれたコードが消失するということを何度か経験しました。

その経緯から考えに至ったのが
**スモールステップ**
でした。

開発を小さなフェーズに分けて少量のコードを生成させながらその動作をその都度確認していくことで、AIにより作らえたコードではなく、動作するコードを我が手にすることが出来るのではないか？と思ってそのように行ってまいりました。

## エージェント型エンジニアリングへの移行
バイブコーディングからエージェント型エンジニアリングへ移行したのは2026年初頭だったように思います。

スモールステップで進めていた開発でしたが、Claudeのトークンが一瞬で食いつぶされるという症状が何度も続きました。

AIと言えどもボランティアではないので、その返答内容について特にコーディングという重い回答の場合は、その会社の方針やモデルに大きな影響を受けると感じます。

ビンボー無料のチャット枠で使用している私の体感ですが^^;

Claudeは何かが変わらないと難しいかもしれない・・・
と思い使い出したのが
**Google Antigravity**
でして、バージョンが2.0へ移行してからは
**エージェント型エンジニアリングツール**
と呼ばれているとか・・・

確かに、1.0から2.0への移行は衝撃的でした。

1.0はVisual Studio Codeをベースに開発がされたそうですが、
2.0になるとそもそもコードの表示がされなくなり、
エージェントとチャットで指示を出すというスタイルに変わりました。

これは(・・;)

それが全世界に受け入れられたわけでもなく、2.0へ移行してから少しすると1.0のようなIDE型アプリを使用することも出来るようになりました。

### 道具が変わることへの拒否感
開発を始めた当初はAIからの回答をGASの編集画面に貼り付けて実行させていましたが、履歴管理の重要性に気がついてGitを使うべく、VSCodeを使用するようになりました。

そんな中、2026年1月からはGoogle Antigravityも使用することになりました。

コーディングを初めてから半年以内で道具が色々変わってきましたが、それに対しての拒否感は驚くほどなかったように思います。

確かに、Antigravityが1.0から2.0へ移行したのは相当な衝撃でしたが、コードを見たい場合はVSCodeを開けば良いという認識でしたし、AntigravityのIDE版が出てきてからはそちらも使用しております。

なんとなく直感で、その道具とは仲良くなれそうにないと思ってしまうと使うことは出来ませんが、昨年から使用してるVSCode、Antigravity IDE、Antigravity 2.0についてはうまく使ってみたいという欲求のほうが高くて、使用することに対しての拒否感は感じません。

# 自身の価値を知りたい
作りたいものを作れば良いという思いもありますが、やはり自身の成果に対して正当に評価されたいというのは人として自然な欲求かと思います。

何があった！？
1か月でいきなり59まで上昇しました(・・;)
偏差値が上がりやすい方法を試したのは事実です。

関数のヘッダーコメントを充実させたり、コミットメッセージで詳細な変更内容を書いたり、プルリクエストを行うようになったり・・・
と言った内容をAIに提案させるようになったのは事実です。

また、何かの開発を行おうとした際に、最近ではClaudeのチャットで開発方針を定めてからマークダウン形式の開発指示書を作成してもらって、Antigravity 2.0へ開発依頼を行うのですが、出てきたコードをそのまま保存するのではなく、関数と説明文事に保存＆コミットメッセージを添えたコミットを行ってまいりました。

さらに、README.mdファイルの内容充実も行うようにしました。

また、楽天APIからランキングの取得を行えるようにしましたし、スプレッドシートでの情報蓄積に限界が存在するということで、初めてのクラウド型RDBであるSupabaseの導入も行いだしました。
Supabaseは在庫情報の蓄積を行えるようになった後、在庫情報の違いのあるもののみ保存しタイムスタンプを記録するという仕様にバージョンアップし、ネクストエンジンで非表示になった商品をis_activeフラグをFALSEにすることで取得対象外になるようにバージョンアップし、在庫情報をタイムスタンプを元にして差分取得＆配布するようにしましたし、差分取得の際にGASの実行時間制限を鑑みてMapを使った更新にもバージョンアップさせました。

エンジニアとしてAIだろうがなんだろうが、眼の前にあるコードの大まかな動作の流れがわからないのは許せないので、AIにコードを渡して動作を聞きながら説明文を追加して行ったように思います。

その関数がどういう原理で作られているのか？そのプロジェクトが動作する概要はどうなっているのか？
もっと突き詰めると、リトライさせる待機時間の妥当性は？トリガーを作成させるための待機時間は何故その遅延時間になっているのか？そのテスト関数は何を目的に作られているのか？

AIが短時間で作ってくれたコードでも気になって調べようとすればいくらでも調べられます。
その調べ方は当然ですが、AIにコードを渡して気になる点を聞くという方法ですけどね。

また、反復して仕様を確認することもよくあります。
コードを作成してもらったのですが、仕様の確認を行いたいと思いますと言った質問を投げかけて、README.mdの更新を行うこともあります。

この偏差値は何らかの間違いであったかもしれませんが、少なくともGASの開発を行いだしてから10か月ほどの私では、AI無しにこの偏差値が出てくるのはあり得なかったように思います。

来月には下がってたりして^^;

ワンチャン、60台に乗ったら求人の引き合いがあったりして(笑)

# 重要なこと
人が作成したコードであっても、バイブコーディングであっても、エージェント型エンジニアリングであっても、本質は正しく動作していることが確認できるコードではないかと思います。

AIが作成したコードのプルリクエストを人が評価しなければならないとか、AIが作成したコードが・・・・とか言う記事を見ますが、本質は望んだ動作を安定して行うことの出来るコードではないかと思います。

エージェント型エンジニアリングの世の中で、私のようなスモールステップであったり、お金がないからClaude Codeを使っていないというのが正しいとは口が避けても言えませんし、どちらかと言えば底辺のエンジニアの言葉でしょうが、自身の製品なので正常に動作し続けられるというのは気になって見続けてしまいますよね。

ログでエラーが出ているのに対してAIに原因を探させるというのは私も行っていますし、今の世の中であれば当たり前の話だと思いますが、AIと言えどもとても詳しい赤の他人なのでその回答の信憑性はやはり人が批評すべきだと思います。

AI無しの開発は私にはありえませんが、AIを完全に信頼しているというのもありえません。

近くにいてくれるような気がする全くの赤の他人

そういう認識でAIをうまく利用していけば良いのではないかと思います。
