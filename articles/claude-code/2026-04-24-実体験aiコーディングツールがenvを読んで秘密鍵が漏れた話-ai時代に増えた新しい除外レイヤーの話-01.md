## はじめに

AIコーディングツール（Cursor、Claude Code、Codeium、GitHub Copilot Workspace等）は、もはやエンジニアの日常に深く浸透しました。コード補完、リファクタ、デバッグ、ドキュメント生成。便利さは疑う余地がありません。

しかし先日、私は自分の開発環境で冷や汗をかく事件を起こしました。**AIツールがプロジェクト内の `.env` ファイルを自動で読み込み、APIキーや秘密鍵がLLMコンテキストに含まれていた**のです。

この記事では、何が起きたか・なぜ起きたか・どう対策したかを共有します。AIツール導入済の方は、**この記事を読んだ後に即、手元の除外設定を確認することを強く推奨します**。

## 事件の概要

### 環境

* エディタ: VS Code + AIコーディング拡張機能
* プロジェクト: Node.js / TypeScript製の自動化ツール
* `.env` に含まれていたもの
  + `OPENAI_API_KEY`
  + `DATABASE_URL`（本番DBの接続文字列）
  + `ENCRYPTION_SECRET`
  + Firestoreサービスアカウントの一部情報

### 起きたこと

1. AIツールに「このプロジェクトのエラーを調べて」と指示
2. AIがコンテキストとしてプロジェクトファイル群を読み込む
3. **読み込まれたファイルリストに `.env` が含まれていた**
4. `.env` の中身がLLMへの入力として送信された
5. 生成されたサジェストに一部の値が含まれて発覚

## なぜ起きたか：従来の `.gitignore` だけでは足りない

### 従来の防御層

```
[コードベース]
   ↓ コミット
[Gitリポジトリ] ← .gitignore で .env を除外
   ↓ push
[GitHub等の共有リポジトリ]
```

`.gitignore` に `.env` を書いておけば、コミットも push もされないので「リポジトリ経由の漏洩」は防げます。これは10年以上続いてきたベストプラクティス。

### AIツールが加わった後の防御層

```
[コードベース]
   ├→ Gitリポジトリ（.gitignore で除外）
   │
   └→ AIツールのコンテキスト（独自の除外設定が必要）
         ↓
      [LLM API（外部サービス）]
```

**AIツールは `.gitignore` を見ない**（一部見るものもあるが、デフォルトではない）。独自の除外設定ファイル、もしくはIDE/ツール側の設定が必要です。

言い換えると、**「リポジトリ経由」と「AIコンテキスト経由」の2つの漏洩経路が存在する時代**になりました。

## 各AIツールの除外設定方法

主要ツールの `.env` 等を除外する設定を整理しておきます。

### Cursor

`.cursorignore` ファイルをプロジェクトルートに配置。書式は `.gitignore` と同じ。

```
# Secrets
.env
.env.*
*.key
*.pem
credentials.json
service-account*.json

# Logs
*.log
logs/

# Caches
node_modules/
.next/
.cache/
```

### Claude Code

`.claudeignore` またはプロジェクトの `settings.json` で制御。

```
// .claude/settings.json
{
  "permissions": {
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Read(*.key)",
      "Read(credentials/*)"
    ]
  }
}
```

Claude Codeは `.gitignore` を尊重するが、明示的に `deny` を設定するとより確実。

### GitHub Copilot

Repository settings → Copilot → "Content exclusion" で除外パターンを登録（Business/Enterpriseプランのみ）。

```
# Exclusions
.env
.env.local
secrets/**
```

### Codeium / その他

各ツールの `ignore` 設定を確認。ドキュメント検索時は `"<ツール名> ignore files"` で検索。

## 対策：2段階で実施した内容

### 対策1: 除外設定の標準化

チーム全体で同じ除外ルールを適用するため、以下をプロジェクトテンプレートに入れました。

```
# .cursorignore（Cursor用、プロジェクトルート）
.env
.env.*
*.pem
*.key
secrets/
credentials/
service-account*.json
.aws/
.ssh/
```

`.gitignore` と `.cursorignore` と `.claudeignore` を**3点セット**で配布するのが現時点のベストプラクティスだと考えています。

### 対策2: 過去のGit履歴清掃 + キーローテーション

念のため、過去のgit履歴に `.env` の痕跡が残っていないかも確認しました。

```
# 履歴内の .env を検索
git log --all --full-history -- '.env'

# 履歴に残っていた場合、BFG Repo-Cleaner で完全削除
bfg --delete-files .env
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# force push（他メンバーに事前連絡必須）
git push --force-with-lease --all
```

**さらに重要なのはキーローテーション**。漏れた可能性のあるキーは即座に再発行：

```
✅ OPENAI_API_KEY → OpenAI Dashboard で再発行
✅ DATABASE_URL → DB側でパスワード変更
✅ ENCRYPTION_SECRET → 再生成＋復号データの再暗号化（データ量多いと地獄）
✅ Firestore Service Account → 新規SAキー発行、旧キー無効化
```

## 一般化：「何を読ませないか」の設計

今回の事件から得た一番の学びは、**新機能導入時に「何を使えるか」だけでなく「何を読ませないか」を先に決める**必要があるということです。

### 導入チェックリスト

AIツール導入時、以下を必ず確認する運用にしました。

```
□ 除外設定ファイルがプロジェクトに配置されている
□ .env / *.key / credentials / *.pem が除外されている
□ チームメンバー全員がツールのバージョンを揃えている
□ ツールのアップデート時に除外設定が維持されるか確認
□ オンプレLLMとクラウドLLMの切替ポリシーが明文化されている
□ 事故発覚時のローテーション手順が文書化されている
```

### 「便利さ」と「安全設計」の同居

```
❌ NG: とりあえず導入 → 便利 → いつの間にか .env 読み込まれている
✅ OK: 導入前に除外設計 → 動作確認 → 便利に使う
```

導入初日の「5分の除外設定」を省くと、後で数時間〜数日のローテーション祭りが発生します。ROIとしては絶対に初日側で払うべきコストです。

## 発覚時の初動

もし「AIツールが秘密情報を読んだかも？」と気付いた瞬間にやるべきこと。

1. **即、キーローテーション開始**（並行作業で最優先）
2. AIツールのログ・履歴を確認（LLM提供者側にログが残っている可能性を認識）
3. 外部サービスのアクセスログで不正利用がなかったか確認
4. `.env` を他の場所（`.env.local` や secret manager）に移す
5. 除外設定を再点検

「後でやろう」は絶対禁止です。漏れた可能性がある時点で、数時間以内に対処が基本。

## これから導入する人へのおすすめ設計

### 開発機では本番キーを持たない

最強の対策は **「開発環境には本番の秘密情報を置かない」** こと。

```
開発環境:
  .env に入れるのは → ダミー値 or ステージング用の限定権限キー

本番環境:
  AWS Secrets Manager / GCP Secret Manager / HashiCorp Vault 等から取得
```

開発機の `.env` を見られても、漏れるのはステージング用キーだけ、という構造を作る。

### シークレットマネージャーを使う

```
// 旧: .env から読む
const apiKey = process.env.OPENAI_API_KEY;

// 新: Secret Manager から取得
import { SecretManagerServiceClient } from '@google-cloud/secret-manager';

async function getApiKey(): Promise<string> {
  const client = new SecretManagerServiceClient();
  const [version] = await client.accessSecretVersion({
    name: 'projects/my-project/secrets/openai-api-key/versions/latest',
  });
  return version.payload?.data?.toString() || '';
}
```

これなら、そもそも `.env` に秘密情報を書かないので、読まれても無害。

## まとめ

* **AIツール導入で、秘密情報漏洩の経路が2つ（Git経由 + AIコンテキスト経由）になった**
* `.gitignore` だけでは不十分。AIツール固有の除外設定が必要
* Cursor / Claude Code / Copilot 等、各ツールで異なる設定を把握
* 導入前に「何を読ませないか」を設計
* 漏れた場合は即ローテーション（発覚時点で数時間以内）
* そもそも開発機に本番キーを置かない設計が理想

AIツールを導入済みの方は、**今すぐ `.env` / `*.key` 等が除外されているか確認してください**。設定ファイルが存在しないなら、それが存在している想定でLLMが動いてしまいます。
