## はじめに

初めまして。  
Contreaという医療系スタートアップでテックリードをしている河野です。

ここ最近、AIの進化は目まぐるしく、開発やデリバリーもグッと速くなりました。  
その一方で、ライブラリやツールのセキュリティ事故が一気に増えています。これはもう SRE や DevOps だけの話ではなく、いちエンジニアの私たちが向き合う問題になってきました。

例えばこのような代表的なインシデントが起こっています。

> * **axiosのマルウェア**  
>   npm上の特定バージョンにマルウェアが仕込まれており、`npm install`をした時点で遠隔操作ツールが入る供給網攻撃が起きた
> * **Nx Console（VSCode拡張）のトロイの木馬**  
>   トロイの木馬版に差し替えられ、1Password や Claude Code の設定、GitHub・AWS の認証情報が盗まれた
> * **MCPのツールポイズニング**  
>   ツールの説明文に隠れた指示を AI が読んで、そのまま実行されてしまう

こういうニュースを見ると、当事者であればもちろん原因調査しなければいけませんし、「自分の環境やプロダクトは大丈夫か」「他にも同じような脆弱性はないかな」とヒヤッとしてしまいます。

最近の攻撃で狙われているのは、アプリのコードよりも「**取り込んでいる環境**」です。

依存パッケージ、拡張機能、AIのツール、それらが動くCI/CD……開発をより良く、プロダクトをより良くするために導入したものが、そのまま侵入口になっています。

ものづくりをしている全ての人にとって、プロダクトではその影響が最終的に利用者にまで届くからこそ、「何を信頼し、何を検証するか」を線引きしておく必要があります。  
とはいえ、全部を疑うのは現実的ではなく、どこかは必ず信頼して使うしかありません。

そこで事例とともに、どう信頼を担保していくか、そしてどう自衛をしていくべきかを簡単にご紹介できたらと思います。

さて、直近1年（2025年6月〜2026年6月）から、代表的な事故についてまとめてみました。

【用語集】

| 用語 | 説明 |
| --- | --- |
| **CVE** | 公に知られた脆弱性に振る一意の管理番号と基本記録。MITRE が運営し、GitHub や各ベンダーなどが採番している。 |
| **CVSS** | 脆弱性の深刻度を 0〜10 で表すスコア。9 以上が Critical |

| **RAT**（Remote Access Trojan） | 遠隔操作型のマルウェア。入れられると外部から端末を操作される |
| **postinstall / prepare スクリプト** | パッケージのインストール時に自動実行されるスクリプト。マルウェアの起動点に使われやすい |
| **wiper** | データの破壊そのものを目的とするマルウェア |
| **OIDC**（OpenID Connect） | 長命の鍵を置かず、その場限りの短命トークンで認証する仕組み |
| **MCP**（Model Context Protocol） | AI エージェントと外部ツールをつなぐ規格 |
| **provenance**（来歴） | パッケージがどのソース・どの CI で作られたかの証明 |

### 代表的なインシデント 11選

ビルドシステム - s1ngularity / Nx（2025年8月）

* **どんなものか**：JS/TS 向けのモノレポ用ビルドシステム。多くの開発現場が `npm install` で取り込む
* **週ダウンロード**：数百万規模
* **CVE / 公表時期**：2025年8月26〜27日
* **侵入経路**：悪意ある Nx バージョンが npm に公開され、インストール時のスクリプトで実行された
* **挙動**：ローカルを走査して GitHub トークン・SSH 鍵・npm 認証情報・暗号通貨ウォレットを収集。さらに被害者の AI CLI（Claude / Gemini / Q）を呼び出し、秘密情報を探させた（初の事例）
* **侵害時の影響**：認証情報が流出され、盗んだ鍵で非公開リポジトリが公開状態にされた

<https://www.wiz.io/blog/s1ngularity-supply-chain-attack>

* **侵害時の影響**：暗号資産の窃取。ただし公開は約2時間で気づかれ、実被害は限定的

<https://www.paloaltonetworks.com/blog/cloud-security/npm-supply-chain-attack/>

## 攻撃の正体 ─ どこで止められるか

さて、事例を並べてきましたが、**主要なライブラリが狙われてきている** のもそうですが、**攻撃の種類が多くなってきている** ということが分かると思います。

悪意のある指示をAIに実行させるツールポイズニング、認証や流出ができなかった場合は侵入環境のファイルを全部削除するなど、AI時代だからこそ多種多様な攻撃バリエーションが増えていっています。

ただ手口はいろいろですが、難しく見えるインシデントも「**環境にある大事な情報が、外に出ていく**」か、「**悪意のあるプログラムを実行される**」に大きく分類されると思っています。

侵入経路を塞ぎ、たとえ侵入されても、出ていく経路を封じる。そして無闇に実行させない。

完璧に防ぎ切るのは難しいですが、予防することはできます。この意識を常にしておくだけでも、環境起因での被害はずいぶん小さくできます。

## 今日から始めるPC環境の予防

簡単ではありますが、予防ということで、  
より個別性が低く、自身の環境にも適用できそうな設定の一例を紹介したいと思います。

他にもこういうのがあるよというものがあったら、意見 & コメントお願いします！

### 1. アカウントを乗っ取られないように2FAを有効にする

当たり前ですが、まずは自身の**Googleアカウント、GitHubアカウント、Microsoftのアカウント**などには、必ず**2FA**を有効にしてください。

そしてワンタイムコード（TOTP）ではなく、**パスキー認証** を使いましょう。

先ほど紹介した chalk / debug・Nx・GhostAction の事例は、どれもメンテナンスユーザーのアカウント乗っ取り（多くはフィッシング）が起点でした。

ここを破られるのが一番やばいです。  
基本ですが、パスワードも容易なものではなくパスワードマネージャー管理にして強力なパスワードにもしておきましょう。

### 2. インストールやAIエージェントを隔離する（サンドボックス）

`npm install` も AI エージェントも、中身は「他人の書いたコードを自分の権限で実行する」ことです。  
なので **隔離された使い捨て環境で走らせる** のが、いちばんの根治になります。

それが「**サンドボックス（ユーザーが通常利用する領域から隔離した、保護された空間のこと）**」になります

Claude Code の設定（`~/.claude/settings.json`）

設定は `~/.claude/settings.json` に書きます。

`permissions.deny`（アプリレベルで Claude のツールを止める）と `sandbox`（OSレベルで bash の子プロセスを止める）を**両方**書くことです。

```
// ~/.claude/settings.json
{
  "permissions": {
    "deny": [
      "Read(**/.env*)",
      "Read(**/secrets/**)",
      "Read(~/.ssh/**)",
    ]
  },
  "sandbox": {
    "enabled": true,
    "allowUnsandboxedCommands": false,
    "filesystem": { "denyRead": ["~/.ssh","~/.gnupg", "~/.config/gcloud"] },
    "network": { "allowedDomains": ["github.com", "*.npmjs.org"] }
  }
}
```

ネットワークが開いていると SSH 鍵などを外に出されてしまうため、`network.allowedDomains` で外向き通信をレジストリなどに絞ります。

もうひとつ、`sandbox.filesystem.denyRead` は bash サブプロセスのファイルアクセスを OS レベルで止めますが、**Claude の Read ツール**は止めません。  
ですので `permissions.deny` の `Read(...)` ルールも併記します。

また、`Read(./.env)` の deny は Read ツールを塞ぎますが `cat .env` は塞がないので、こちらも防ぐようにしておきましょう

Codex の設定（`~/.codex/config.toml`）

既定の `workspace-write` モードでは、書き込みがワークスペースと一時ディレクトリに限定され、ネットワークは遮断されます。設定は `config.toml` に書きます。

```
# ~/.codex/config.toml
approval_policy = "on-request"
sandbox_mode   = "workspace-write"  # 既定。書き込みはワークスペース＋tmpに限定し、ネットワークは遮断
# read-only          … 読み取りのみ
# danger-full-access … 隔離なし。基本は使わない
```

隔離は OS のカーネルレベルで効きます。macOS は Seatbelt、Linux は bwrap + seccomp + Landlock、WSL2 は Linux 実装、ネイティブ Windows は AppContainer の制限付きトークンで外向き通信を遮断します。`codex sandbox macos` や `codex sandbox landlock` で、コマンドが実際にどう制限されるかを手元で試せます。組織で運用するなら `requirements.toml` で `danger-full-access` を禁止する、といった強制もかけられます。

なお Docker は共有カーネルなので、これ単体では「サンドボックス」とは言い切れません。コンテナ単体に頼らず、microVM（軽量VM）や gVisor（サンドボックス・ランタイム）のような、より強い隔離を検討してください。

### 3. 盗めるものを置かない

PCに侵入されたとしても、**子プロセスに認証情報を渡さなければ流出しない** です。  
AIからの流出を抑えるために、Claude Code は専用のスクラブ変数、Codex は環境変数ポリシーで制御できます。

Claude Code の設定

Claude Code には、プロセスの環境変数から機密性の高い認証情報を自動的に除去（スクラブ）する変数が用意されています。（ただし、とても限定的です。詳細は[こちら](https://dev.classmethod.jp/articles/claude-code-env-scrub-secret-leak-defense-in-depth/)）

ターミナルで下記を実行するだけで有効になります。

```
export CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1
```

Codexの設定

Codex は、`config.toml` で設定することができます。 `[shell_environment_policy]` で、シェルの子プロセスに渡す環境変数を制御できます。

```
# ~/.codex/config.toml
[shell_environment_policy]
inherit = "none"                 # 既定の環境変数の継承を止める
include_only = ["PATH", "HOME"]  # 通すものだけを明示する
# 継承したまま落とす場合は exclude を使う
# inherit = "all"
# exclude = ["AWS_*", "AZURE_*", "*_TOKEN", "*_KEY"]
```

### 4. 更新したばかりのライブラリバージョンを採用しない

乗っ取り版は、公開から数時間で消されることが多いです。chalk / debug は約2時間、axios は約4〜5時間で消されました。

つまり **公開直後を避けるだけ** で、人気パッケージの乗っ取りはかなり踏まずに済みます。  
パッケージマネージャに「最小リリース経過時間」というものを設定できるので、こちらで手動の install も CI も両方守れます。

```
npm（v11.10.0〜）：`.npmrc` に `min-release-age=7`。npm は日単位
pnpm（v10.16〜）：`minimumReleaseAge: 10080`。pnpm は分単位で 10080 が7日。信頼するパッケージの除外リストがあり、推移依存にも効く
yarn（v4.10.0〜）：`.yarnrc.yml` に `npmMinimalAgeGate: "7d"`。yarn は期間文字列（`"7d"` など）で指定
bun（v1.3〜）：`bunfig.toml` に `minimumReleaseAge = 604800`。bun は秒単位で 604800 が7日
```

### 5. Takumi Guard を噛ませる

Takumi Guard は、npm クライアントと公開 npm レジストリの間に入るプロキシです。  
`npm install` のたびに各パッケージをブロックリストと照合し、マルウェアやタイポスクワット、侵害済みパッケージを **tarball のダウンロード前に止めます**。

パッケージのブロック機能は個人・法人とも無料です。

すぐ入れられるものなのでぜひ試してみてください。  
<https://flatt.tech/takumi/features/guard>

### 6. エディターの拡張機能を絞る

エディタの拡張機能は、そもそも入れた時点でエディタと同じ権限の任意コードになるため、これだけは決定的な根治はありません。

それでも、**使う拡張を本当に要るものだけに絞る**、**Workspace Trust を有効にする** といった仕組みで、攻撃を減らせます。

* 使う拡張機能は本当に必要なもののみ。使っていないものは消す
* 第三者拡張を初めて入れるとき、VS Code が出す発行元の信頼ダイアログで確認する
  + コマンドライン（`code --install-extension`）でのインストールは発行元が自動で信頼されないのでやめましょう
* Workspace Trust を有効にして、信頼しないフォルダでは拡張のコード実行を止める

```
// settings.json
{ "security.workspace.trust.enabled": true }
```

※Workspace Trust は「プロジェクト内の悪意あるコード」を制限するもので、すでにグローバルに入っている拡張機能には効かないです。

### 7. CI/CD と GitHub を固める

CI/CD はデプロイ鍵やクラウド認証を持つ「鍵の宝庫」です。

**ワークフローの権限は最小**、**クラウド認証は OIDC の短命トークン**、**Action は commit SHA で固定**、**fork からの PR に注意**、**runner の外向き通信を制限** の5つをそれぞれ固めていくことで、より頑健になります。

```
# .github/workflows/deploy.yml
permissions: read-all  # トップレベルは読み取り専用にしておく
 
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write  # OIDC を使うジョブにだけ書き込み権限を付与
    steps:
      # Action はタグではなく commit SHA で固定し、コメントに版を残す
      - uses: actions/checkout@<commit-sha>  # v4.2.2
        with:
          role-to-assume: arn:aws:iam::<account-id>:role/<deploy-role>
          aws-region: ap-northeast-1
          # 長命の アクセスキーは置かず、OIDC で短命トークンを assume する
```

`<commit-sha>` への固定は `pinact` で自動化でき、`zizmor` で権限過多や `pull_request_target` の危険な使い方を静的にチェックできます。

* **ワークフローの権限を最小にする**。トップレベルで読み取り専用（`permissions: read-all`）にし、書き込みが必要なときにはジョブにだけ付与します
* **クラウド認証は OIDC の短命トークンにする**。長命のクラウド環境の秘密鍵を GitHub Secrets に置かないように、OIDCにしましょう
* **使う Action は commit SHA で固定する**。タグは後から悪意あるコミットに書き換えられます（tj-actions がこれです）。実は第三者 Action をすべて SHA まで固定しているリポジトリは全体の約4%しかありません。`pinact` で自動固定、`zizmor` で設定ミスの静的チェックなどを行いましょう。
* **fork からの PR に注意する**。`pull_request_target` は Shai-Hulud 2.0 や TanStack の侵入口になりました。使うなら読み取り専用権限にし、PR の head は checkout しない。外部からの PR は実行前にdevelopブランチであれ、承認を必須にしましょう。
* **runner の外向き通信を監視・制限する**（Harden-Runner など）。盗んだ鍵を外へ送る経路を塞ぎます
