## AI YouTuber の動画を見ていますか

ここ1〜2年、AI系のYouTuberが急速に増えました。

新しいモデルのリリース速報、AIエージェントのデモ、プロンプトの技法。キャッチアップの手段としては非常に有効です。自分も時間を見つけては見ています。

ただ、正直に言います。

あの動画群は、主にWeb系のエンジニアやスタートアップ界隈に向けて作られています。「Next.jsでアプリを作りました」「SaaSのプロトタイプが3日でできました」。それはそれですごいことですが、自分たちの現場とは遠い話です。

EtherCATのスレーブデバイスを設計している人間にとって、Reactのチュートリアルは直接的には役に立ちません。マイコンのレジスタマップと格闘している人間が、Tailwind CSSの話を聞いてもピンとこないのは当然です。

**だから見なくていい、という話ではありません。**

問題は、「自分の領域とは関係ない」と判断して、AIそのものから距離を取ってしまうことです。

## 自分の話をします

経歴はハードウェア設計が主です。ASICとFPGAの設計を40年近くやってきました。

ただ、純粋なHW設計者ではありません。検証用のテストベンチ、装置制御のファームウェア、計測データの処理ツール、校正アルゴリズムの実装。設計の現場で必要になるソフトウェアは、C、C++、C#で自分で書いてきました。

ハードウェアエンジニアにはこういう人が少なくないはずです。管理職にはならず、専門職として現場に残り、設計もするしコードも書く。器用貧乏と言われることもあるかもしれません。

ただ、この「設計もできてコードも書ける」という立ち位置が、AIエージェントとの協働において想像以上に効いたのです。

## JS未経験から1ヶ月で2本作った事実

JavaScript / TypeScript はほぼ未経験でした。Webシステムの構築は未知の領域でした。

それでも、AIエージェント（Claude Code / Codex）を使って、約1ヶ月で本番稼働レベルのシステムを2本構築しました。

**AILBA（AI Log Brain Agent）**: メールを受信してAIが解析・要約し、チケットを発行するシステム。Claude Desktop + Claude Codeで開発。Python / FastAPI / React / TypeScript / PostgreSQL / Redis / MinIO。

**MSH（Manufacturing Service Hub）**: 製造トレーサビリティ管理システム。ChatGPT + Codexで開発。React + Ant Design / FastAPI / PostgreSQL の3コンテナ構成。

自分が書いたコードは実質ゼロです。

## なぜ作れたのか：コーディング力ではない

ここを誤解してほしくないので、はっきり書きます。

AIが優秀だから作れた、のではありません。**自分がAIに渡す仕様の精度が高かったから作れた**のです。

AIエージェントに「メール管理システムを作って」と投げれば、若手でも何かそれらしいものは得られます。ただ、以下のような仕様を定義できるかどうかで、アウトプットの質はまったく変わります。

* メール1通に複数の課題が混在している場合、どう分割するか
* 添付の仕様書と本文で内容が食い違う場合、どちらを優先するか
* Gmail APIのポーリング間隔と、業務上の許容遅延のバランスをどこに置くか

この種の判断は、40年近くメールベースの業務で設計をやってきた人間だからこそ出せるものです。

前回の記事で「AIエージェントはシリコンに似ている」と書きました。投入する仕様の質がそのままアウトプットの質になる。この構造は、ASIC設計の経験がある人間にとっては直感的に理解できるはずです。

## シニアの「現場力」がAIに対して優位な3つの場面

### 入出力を先に定義する

AIエージェントに何を渡して、何が返ってくるべきか。ここが曖昧だと、生成されるコードも曖昧になり、手戻りが増えます。

HW設計ではインターフェース仕様を先に固めるのが基本です。この習慣がある人は、AIエージェントへのプロンプト作成で圧倒的に有利です。

### 例外パスを嗅ぎ取る

「この仕様だと、このケースで破綻する」。この嗅覚は、何度も痛い目に遭った人間にしかありません。

AIは正常系のコード生成は得意です。しかし、現場で起こるイレギュラーを自分から想定してはくれません。「30年前のあのトラブルと同じ構造だ」と気づける人間が、AIの出力をレビューすることで、実用レベルのシステムになります。

### 「どこまでやれば動く」を判断する

AIはコードを無限に書けます。機能も無限に追加できます。

でも、「この機能は後回し」「これがないと業務で使えない」「この品質なら出荷できる」。こうした判断は、実際に製品を設計し、出荷し、顧客に使ってもらってきた経験がなければ下せません。

MSHのプロトタイプが1週間で動いたのは、AIの速度だけではなく、「製造トレーサビリティとして最低限何が必要か」を即座に判断できたからです。

## 問題は「見ているだけ」で止まっていること

ここからが本題です。

AI系YouTuberの動画を見て「すごいな」と思う。SNSで若手がAIでプロダクトを作っている投稿を見る。でも、自分では触らない。

理由は何でしょうか。

「Webは自分の領域ではない」。それはわかります。自分もそうでした。

「今の業務が忙しい」。それもわかります。エンジニアの日常は割り込みで埋まっています。

「もう少し情報を集めてから」。それが一番危険です。

AIエージェントの進化速度は、半年前の情報がもう古くなるペースです。2025年2月に自分が体験したClaude CodeとCodexの挙動は、今とはもう変わっています。完璧なタイミングは来ません。

若手はすでに走っています。彼らは経験値こそ少ないですが、AIとの対話を通じて猛烈な速度で学んでいます。仕様を定義する力も、やがて追いついてきます。

**シニアのアドバンテージには賞味期限があります。**

## 始め方は難しくない

大げさなことをする必要はありません。

**まず、今抱えている課題をAIに話してみてください。**

Claude（claude.ai）かChatGPTを開いて、「こういう問題を解決するシステムが欲しい」と自然言語で伝える。AIが質問を返してきます。それに答えるだけです。

答えているうちに、自分の中で課題の構造が整理されていきます。これが「AIと対話して仕様を固める」プロセスの入口です。

次に、対話で見えてきた構造を「入力」「出力」「例外処理」の形式で書き出してみてください。HW設計者なら馴染みのある形式のはずです。これがそのまま、AIエージェントに投入するプロンプトの骨格になります。

Webシステムを作る必要はありません。自分の領域の課題から始めてください。テストベンチの自動生成でも、仕様書のレビューでも、計測データの処理でも。AIは言語を選びません。

## この先に見据えていること

今回のWeb開発は、あくまで試金石です。

自分が見据えているのは、ハードウェアの深い理解がなければ仕様を定義することすらできない領域です。Webアプリとは違い、ハードウェアの知識と設計経験が仕様策定の前提条件になる世界。

AI YouTuberのデモではカバーされない領域です。そこにこそ、現場で手を動かし続けてきたシニアエンジニアの本当の出番があると考えています。

Webで確立したワークフローを、その世界に持ち込んだとき何が起きるか。追って報告します。
