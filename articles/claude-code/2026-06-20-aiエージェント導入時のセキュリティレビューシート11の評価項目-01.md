## はじめに

組織環境へAIエージェントを導入する際、多くの議論は「どのLLMを利用するか」に集中しがちです。

しかし実際のセキュリティレビューでは、

* 誰の権限で動作するのか
* 何のデータにアクセスできるのか
* 外部へ送信できるのか
* 誤操作や攻撃に耐えられるのか

を体系的に評価する必要があります。

本記事では、組織環境へのAIエージェント導入を想定して整理した11項目のレビュー観点を紹介します。

これら11項目はすべて最終的に  
「侵害された場合にどのような事業影響が発生するか」  
を評価するために存在します。

### 11のレビュー項目

レビューは以下の順序で実施します。  
前半の5項目を「アクセス権レビュー」、後半を「リスクレビュー」と考えると理解しやすくなります。

#### アクセス権レビュー

1. Identity
2. Authentication(OIDC)
3. Authorization Model(OAuth)
4. Authorization Scope
5. Resource Boundary

#### リスクレビュー

6. Data Classification
7. Trust Boundary
8. Tool Security
9. Human Approval
10. Monitoring & Audit
11. Business Impact

## なぜこの11項目なのか

AIエージェントの設計レビューでは、個別のセキュリティ機能だけを見ても全体のリスクは判断できません。

例えば、

* MFAを導入している
* Prompt Injection対策を実装している
* 監査ログを取得している

といった対策があっても、

* 誰の権限で動作するのか
* 何のデータにアクセスできるのか
* どの範囲まで到達できるのか

が不明であれば、本当のリスクは評価できません。

AIエージェントのリスクを整理する際、私はまず以下の流れで考えます。

Identity  
↓  
Authorization  
↓  
Resource  
↓  
Business Impact

まず「誰のIdentityで動作するのか」を確認します。  
次に「どのような権限を持っているのか」を確認します。  
そして「その権限によってどのリソースへ到達できるのか」を確認します。  
最後に「侵害された場合、どのような事業影響が発生するのか」を評価します。

例えば Files.Read.All という権限だけを見ても危険度は判断できません。  
その権限でアクセスできる先が、

* 公開文書サイトなのか
* 役員専用SharePointなのか

によってリスクは大きく変わります。

本記事で紹介する11項目は、この考え方をより詳細にレビューするために整理したものです。

前半の「アクセス権レビュー」では、

* 誰が
* どのように認証され
* どの権限を持ち
* どこまで到達できるのか

を確認します。

後半の「リスクレビュー」では、

* どのようなデータを扱うのか
* 外部へ持ち出せるのか
* Tool Abuseに耐えられるのか
* 監査できるのか

を確認し、最終的なBusiness Impactを評価します。

## 1. Identity

### 確認事項

* アプリのユーザーIDは人か？人以外か？
  + Human Identity / Non-Human Identity（Service Principal / Managed Identity）

### レビュー観点

* 誰のIdentityか？
* Human / NHI(Non-Human Identity) が明確か？
* IDの作成・変更・削除プロセスは管理されているか？
* 認証情報は何か？  
  例：Client Secret / Certificate / Managed Identity / API Key
* 認証情報は適切に保護されているか？

## 2. Authentication (OIDC)

### 確認事項

### レビュー観点

* 組織の認証基盤で認証されるか？
* MFAは必須か？
* ID Tokenは適切に検証されるか？
* 認証セッションの有効期限は適切か？

## 3. Authorization Model(OAuth)

### 確認事項

* Agentは誰の権限で動作するか？
  + Delegated Permission / Application Permission
  + RBAC / ABAC / ReBAC

### レビュー観点

* Agentはユーザー権限か？アプリ権限か？
* 利用者の権限に応じてアクセス制御されるか？
* 権限昇格の可能性はないか？

## 4. Authorization Scope

### 確認事項

* 最小権限の原則適用
  + Files.Read / Files.Read.All / Sites.Read.All / Mail.Send / Delete / Update

### レビュー観点

* 不要な操作権限を持っていないか？（最小権限の原則）

## 5. Resource Boundary

### 確認事項

* どのリソース範囲に到達できるか？
  + SharePoint / OneDrive / Database / CRM / GitHub Repository

### レビュー観点

* 到達可能なResource範囲は 全社か？ 部門限定か？ 個人限定か？  
  例
  + 全社SharePoint
  + 人事部SharePoint
  + 役員専用SharePoint
  + 個人OneDrive
  + 特定のDatabase

## 6. Data Classification

### 確認事項

* アクセス先リソースの機密度
  + Public / Internal / Confidential / Restricted

### レビュー観点

* 機密データへ到達可能か？
* PII・顧客情報・財務情報・知財は含まれるか？

## 7. Trust Boundary

### 確認事項

### レビュー観点

* メール送信・インターネット検索・外部連携などがあるか？
* Internal → External へのデータ持ち出しが可能か？  
  例：Email / Slack / GitHub / Internet Search / MCP Server

### 確認事項

* LLMのツール誤選択・インジェクション対策
  + MCP認証 / Tool認可 / Tool引数検証 / File Validation など

### レビュー観点

* Tool Abuse や Injection により意図しないTool実行が可能か？

## 9. Human Approval

### 確認事項

* 人間による承認有無
  + HITL(Human-In-The-Loop)あるか？ 承認フローあるか？

### レビュー観点

* 高リスク操作に人間承認が必要か？  
  例：Delete / Update / External Send / Financial Transaction

## 10. Monitoring & Audit

### 確認事項

* 監査ログ
  + Prompt Log / Tool Log / Access Log / Audit Log

### レビュー観点

* 誰が・いつ・何に・何をしたか？ が記録され追跡調査が可能か？

## 11. Business Impact

### 確認事項

### レビュー観点

* 機密性・完全性・可用性・事業影響の観点で最大被害を評価する。  
  例：
  + 機密性→情報漏えい
  + 完全性→DB改ざん
  + 可用性→業務停止
  + 事業影響→法令違反、信用失墜、金銭損失

## おわりに

AIエージェントのリスクは、単にLLMの性能だけでは決まりません。

* 誰のIdentityで動作し、
* どのように認証され、
* どの権限を持ち、
* どのデータへ到達でき、
* どの境界を越え、
* 最終的にどのような事業影響を与えるか。

この流れを体系的に確認することが、AIエージェントの設計レビューでは重要です。

本記事で紹介したレビュー観点が、AIエージェント導入時のセキュリティ評価の参考になれば幸いです。
