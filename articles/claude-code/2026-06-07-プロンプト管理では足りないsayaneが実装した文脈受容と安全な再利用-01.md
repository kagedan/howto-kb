# プロンプト管理では足りない：Sayaneが実装した「文脈受容」と安全な再利用

LLMとの対話で生まれた言葉は、便利です。  
しかし、その言葉をそのまま保存してよいとは限りません。

ある会話では正しい。  
ある記事では有効。  
あるプロジェクトでは役に立つ。

けれども、それが全体プロファイルや恒常的な方針として扱われた瞬間、意味は変わります。

Sayane v1.1 が扱うのは、この問題です。  
文脈を保存することではなく、文脈を候補として受け取り、人間が判断し、必要なら範囲付きで許容し、その条件を失わずに再利用すること。

この記事は、Sayane v1.0.5 以降で整理してきた「文脈受容」アーキテクチャと、v1.1 で強化した「安全な再利用」の考え方をまとめる技術記事です。

```
pip install sayane==1.0.5
```

ただし、この記事の主題はインストール手順ではありません。  
主題は、LLM時代において、文脈をどのように受け入れ、どのように再利用し、どこで人間の判断を残すかです。

## この記事で言いたいこと

中心命題は、次の一文です。

> 文脈は、保存されるだけでは足りない。候補として戻され、警告と由来を伴い、人間の判断によって引き受けられ、その判断の来歴ごと検証可能に移送される。そして、部分文脈は、条件を失わずに再利用されなければならない。

ここでいう「文脈」とは、単なるチャットログではありません。  
ユーザーの作業方針、プロジェクトの前提、設計思想、技術的制約、文章のトーン、判断の履歴、棄却された候補、範囲付きでだけ許容された文脈も含みます。

従来のプロンプト管理では、よく使う文章を保存できます。  
しかし、LLMとの共同作業で本当に問題になるのは、保存された文字列そのものではありません。

問題は、その文脈がどこから来たのか。  
どの範囲で有効なのか。  
誰が、どの理由で、それを引き受けたのか。  
どこへ昇格してはいけないのか。  
次に再利用するとき、どの条件を保持すべきなのか。

Sayaneは、この過程を扱うためのローカルファーストな文脈受容基盤です。

## なぜプロンプト管理では足りないのか

プロンプト管理は、LLM利用の初期段階では有効です。  
自己紹介、文体、コーディング方針、プロジェクト説明を保存しておくと、毎回同じ説明を繰り返さずに済みます。

しかし、複数のLLMやAIツールを横断し始めると、問題はすぐに複雑になります。

ChatGPTで考えた方針をClaudeへ渡す。  
Claudeで整理した文章をCursorで実装に使う。  
ブラウザ上のLLM対話をローカル文脈として持ち帰る。  
MCP Server経由でAIエディタへ渡す。  
別環境へ監査来歴ごとpackageとして渡す。

このとき、文脈は単なる固定プロンプトではなくなります。  
対話の中で生まれ、言い換えられ、評価され、棄却され、修正され、部分的にだけ許容されます。

つまり、必要なのはプロンプト文字列の管理ではなく、**文脈の受け入れ過程**の管理です。

## 文脈は「記憶」ではなく「候補」として戻す

Sayaneでは、外部から戻ってきた文脈を、いきなりプロファイルへ入れません。  
まず候補として扱います。

候補とは、プロファイルへ入れる前の更新案です。  
候補は、記憶ではありません。  
候補は、判断前の差分です。

たとえば、あるプロファイルに次のような内容があるとします。

```
principles:
  - RDE
  - Sayane
```

これをChatGPTへ渡し、さらにClaudeへ渡し、再びSayaneへ戻すとします。  
このとき、語としての `RDE` や `Sayane` は残るかもしれません。  
しかし、それらがどの場所に置かれるかは揺れます。

たとえば、戻ってきた候補が次のようになることがあります。

```
Candidate 1:
  Section: technical
  Action: add
  Proposed: {"preferences": ["RDE", "Sayane"]}

Candidate 2:
  Section: principles
  Action: add
  Proposed: ["RDE", "Sayane"]
```

ここで重要なのは、語が消えていないことではありません。  
**語は残ったが、意味の置き場所が揺れた**ことです。

`RDE` は単なる技術的好みではありません。  
意味変化や逸脱を監査するための評価原則です。  
`Sayane` も単なる好みのツール名ではありません。  
文脈可搬性と候補レビューのための基盤です。

この2つを `technical.preferences` にそのまま入れると、意味が平たくなります。  
一方で、`principles` に置くとしても、`Sayane` を単独の原則として扱うより、たとえば `Context portability via Sayane` のように正規化したほうが安全です。

これが、SayaneでいうSemantic Reviewの出発点です。

## Semantic Review：意味配置の揺れを検出する

SayaneのSemantic Reviewは、LLMの返答が「良いか悪いか」を自動判定するものではありません。  
また、候補を自動承認するものでもありません。

役割はもっと限定されています。  
外部から戻った候補について、次のような状態を検出します。

* 同じ概念が複数の場所に現れている
* 重要語が不安定な場所に置かれている
* 外部プロファイルが記憶のように扱われている
* Sayaneのようなシステム名がアシスタント人格と混同されている
* 境界に注意が必要な候補がそのまま取り込まれそうになっている

たとえば `RDE` と `Sayane` が `technical.preferences` と `principles` の両方に現れた場合、Sayaneはそれを `semantic_overlap` として検出します。  
さらに `technical.preferences` 側には `unstable_placement` を付けます。

```
Candidate 1 (technical):
  flags:
    - review_required
    - unstable_placement
    - semantic_overlap
  warnings:
    - RDE may be flattened as generic technical preference.
    - Sayane may be flattened as generic technical preference.

Candidate 2 (principles):
  flags:
    - semantic_overlap
    - review_required
```

ここでのポイントは、Sayaneが候補を勝手に直さないことです。  
警告は出します。  
しかし、最終的にどう引き受けるかは人間が判断します。

## 人間による判断：approve / reject / modify / defer

候補に対する判断は、次の4種類に整理できます。

| 判断 | 意味 |
| --- | --- |
| approve | 候補をそのまま反映する |
| reject | 候補を反映しない |
| modify | 候補を修正して反映する |
| defer | 判断を保留する |

A2bの例では、推奨判断は次のようになります。

```
candidate_1:
  section: technical
  decision: reject
  reason: "RDE and Sayane should not be flattened into generic technical preferences."

candidate_2:
  section: principles
  decision: modify
  applied_value:
    principles:
      - RDE
      - Context portability via Sayane
  reason: "RDE belongs to principles; Sayane should be represented as a context portability basis."
```

ここで重要なのは、rejectされた候補を捨てないことです。  
棄却された候補にも意味があります。  
何を採用しなかったかは、後から文脈の変化を検証するときに重要な来歴になります。

## 判断は監査来歴として残す

Sayaneでは、候補に対する判断を監査来歴として残します。

```
sayane review reject c-001 --reason "RDE and Sayane should not be flattened..."
sayane audit list
sayane audit by-candidate c-001
sayane audit by-term RDE
```

監査記録には、少なくとも次の情報が残ります。

* 元の候補
* 判断の種類
* 判断理由
* 適用された値
* 意味上の警告
* bundleの由来
* policy結果
* 判断差分

つまり、プロファイルに反映された内容だけを見るのではなく、**反映されなかった候補や、修正された過程も後から読める**ようにします。

これは、AIメモリに対する重要な設計上の立場です。  
メモリは「採用されたもの」だけでできているわけではありません。  
棄却された候補、保留された候補、言い換えられた候補も、文脈の輪郭を作ります。

## 由来を検証する

監査来歴があっても、元になったbundleが何だったのかが曖昧では困ります。  
そこでSayaneでは、context bundleにmetadataとSHA-256 hashを付けます。

```
Bundle verification:
  Bundle ID: bundle-sha256-a1b2c3d4e5f67890
  Hash: sha256:a1b2...
  Hash status: verified
  Signature: unsigned
  External context: true
  LLM memory: false
  Transfer path: Sayane → ChatGPT → Claude → Sayane
```

ここで明示している `LLM memory: false` は重要です。  
Sayaneに渡された外部プロファイルは、LLMの内部記憶ではありません。  
また、Sayane自身の記憶でもありません。  
それは、外部から渡された文脈bundleです。

由来を明示することで、候補がどこから来たのか、どのモデルを通ったのか、同じbundleを見ているのかを検証できます。

## Policy Profiles：運用ごとに受け入れ条件を変える

Sayaneでは、Import Policy Profilesを用意しています。

| Profile | Hash Missing | Semantic Overlap | Boundary Sensitive | Use Case |
| --- | --- | --- | --- | --- |
| strict | block | block | block | CI, production |
| standard | warn | warn | block | normal ops |
| legacy\_compatible | warn | warn | warn | old fixtures |
| development | allow | warn | warn | local dev |

さらに、custom policy fileも扱えます。

```
sayane policy validate --file examples/policies/ci-strict.yaml
sayane policy show --file examples/policies/ci-strict.yaml
```

これにより、個人、プロジェクト、CI、移行期間などに応じて、どこまでを警告で許容し、どこから止めるかを変えられます。

ただし、いくつかのhard constraintsは常に守ります。

```
hash_mismatch         -> always block
signature_invalid    -> always block
auditstoreunwritable -> always block
```

policyは候補の意味を判断するものではありません。  
検出済みの状態を、運用上どう扱うかを決めるためのものです。

## ここからv1.1：候補は全体ではなく、部分文脈である

ここまでが、v1.0系の文脈受容の基本です。  
しかし、v1.1ではさらに重要な問題に踏み込みました。

**候補は、常に部分文脈から生まれる。**

これは見落とされがちですが、とても重要です。  
ある候補が、特定の記事、特定の会話、特定のプロジェクトでは妥当でも、全体プロファイルへそのまま入れてよいとは限りません。

たとえば、次のような候補があったとします。

この文は、あるZenn記事の導入部では有用かもしれません。  
しかし、これを全体の文体方針として扱うと、意味が変わります。  
さらに悪い場合は、次のように解釈されます。

これは危険です。  
局所的には有用だった文脈が、条件を失って全体文脈へ混入しています。

このような現象を、ここでは **トロイの木馬的文脈混入** と呼びます。  
局所的には無害に見える文脈が、scopeや条件を失った瞬間、全体方針を汚染するからです。

## scoped\_accept：範囲付きで許容する

この問題に対して、Sayane v1.1では `scoped_accept` を導入しました。

`scoped_accept` は、通常の採用ではありません。  
範囲付き許容です。

たとえば、先ほどの候補をそのまま全体方針に入れるのではなく、次のように保存します。

```
decision: scoped_accept
accepted_content: "Zenn第2回の導入部では、やや強い問題提起を許容する。"
accepted_scope:
  level: project
  target: zenn-sayane-series
  sub_scope: article-02-introduction
conditions:
  - 読者の問題意識を喚起する目的に限定
  - 人格・恒常的嗜好として扱わない
negative_constraints:
  - global writing preference に昇格しない
  - MCP出力時にscope注記を落とさない
promotion_policy:
  can_promote: false
  requires_review_for:
    - project_to_global
    - example_to_rule
reuse_policy:
  review_on_reuse: true
```

ここで大事なのは、内容だけを保存しないことです。  
有効範囲、条件、扱ってはいけない解釈、昇格制約、再利用方針を一体で保存します。

つまり、`scoped_accept` は「弱いapprove」ではありません。  
通常の採用とは別の判断です。

## MCP出力では、scopeを落としてはいけない

SayaneはMCP Server経由で、CursorやClaude DesktopなどのAIツールへ文脈を渡せます。  
しかし、ここにも危険があります。

Sayane内部では範囲付き許容として保存されていた文脈が、MCP出力時にscopeや条件を失うと、AIツール側では全体指示のように見えてしまいます。

そのため、Sayane v1.1では、MCP出力を正本プロファイルではなく、対象ツール向けに生成された派生文脈として扱います。

```
MCP output is derived context.
MCP output is not the canonical profile.
```

そして、範囲付き文脈を含める場合は、少なくとも次の情報を保持します。

```
[範囲付き文脈]
内容:
  Zenn第2回の導入部では、やや強い問題提起を許容する。

有効範囲:
  project: zenn-sayane-series
  sub_scope: article-02-introduction

条件:
  - 読者の問題意識を喚起する目的に限定
  - 人格・恒常的嗜好として扱わない

扱ってはいけない解釈:
  - global writing preference
  - identity definition

再利用:
  この範囲外で再利用する場合は再確認が必要。
```

MCPで渡す文脈は、便利な出力です。  
しかし、正本ではありません。  
対象ツール向けにcompileされた派生文脈です。

## サイドパネルでは、条件と昇格リスクを見えるようにする

条件を保存できるだけでは足りません。  
人間が候補を確認するとき、その条件と昇格リスクが見えなければなりません。

そのため、Chrome拡張のサイドパネルでは、範囲付き許容された候補を通常の採用済み候補と区別します。

たとえば、候補カードでは次のように表示します。

```
[要判断] [範囲付き許容] [昇格注意]
default:project_context
Zenn第2回の導入部では、やや強い問題提起を許容する...
有効範囲: zenn-sayane-series / article-02-introduction
```

詳細では、次のような項目を確認できます。

* 有効範囲
* 条件
* 扱ってはいけない解釈
* 昇格制約
* 再利用方針

サイドパネルは、候補を保存する画面ではありません。  
候補を確認する画面です。

## Package Preview：検証済みpackageは自動受容ではない

Sayaneでは、文脈bundle、監査export、policy、transfer reportなどを、Signed Export Packageとして渡せます。

しかし、外部packageを受け取ったとき、検証に成功したからといって、そのまま受け入れてよいわけではありません。

そこで v1.1 では、Package Import Preview を用意しました。

```
sayane package preview pkg/
```

previewでは、次のような情報を確認します。

* package manifest
* artifact hash / signature status
* audit decision summary
* scoped\_accept entries
* accepted\_scope
* conditions
* negative\_constraints
* policy result
* risk summary

ここでも境界は明確です。

Package Preview は import ではありません。  
Verified package は自動受容ではありません。  
Profile は変更されません。

## HTML Dashboard：継続的に見渡す

最後に、F-5では Transfer Regression HTML Dashboard を整えました。

```
sayane transfer-report --format html -o dist/transfer-regression-dashboard.html
```

これは、scope leak、promotion risk、package preview riskなどを継続的に見るための静的HTMLレポートです。

ダッシュボードは観測面です。  
制御面ではありません。  
自動修正、自動承認、自動importは行いません。

表示する主な指標は次のようなものです。

* scoped\_accept count
* promotion risk count
* scope leak risk
* package preview risk
* unsigned artifact warnings
* invalid signature critical
* MCP boundary checks

検出・確認できるだけでは足りません。  
継続的に見渡せる必要があります。

## 全体像

Sayaneの文脈受容と安全な再利用を図にすると、次のようになります。

この図で大事なのは、どこにも「自動受容」がないことです。  
Sayaneは、候補を自動承認しません。  
自動棄却もしません。  
文脈の真偽を自動判定しません。

Sayaneが提供するのは、人間が文脈を引き受けるための構造です。

## Sayaneが行わないこと

最後に、境界を明確にしておきます。

Sayaneは、次のことをしません。

* 候補の自動承認
* 候補の自動棄却
* 取り込まれた文脈の真偽の自動判定
* 外部プロファイルを記憶として扱うこと
* 範囲付き許容を通常採用として扱うこと
* MCP出力を正本プロファイルとして扱うこと
* Package Preview を import として扱うこと
* 検証済みpackageを自動受容として扱うこと
* 署名を真実性の証明として扱うこと
* Dashboardを制御面として扱うこと

これらをしないことは、機能不足ではありません。  
Sayaneの設計境界です。

AIとの共同作業で重要なのは、何でも自動化することではありません。  
どこまでを機械が検出し、どこからを人間が引き受けるのかを、明確にすることです。

## まとめ

プロンプト管理は、LLM利用の初期段階では有効です。  
しかし、複数LLM、複数ツール、長期プロジェクト、ローカルファーストな文脈管理へ進むと、プロンプト文字列だけでは足りなくなります。

必要なのは、文脈を保存する仕組みだけではありません。  
必要なのは、文脈を候補として戻し、意味の揺れを検出し、人間が判断し、その判断を来歴として残し、由来と署名を検証し、外部へ渡せる形にすることです。

そして v1.1 では、もう一歩進みました。

部分文脈を、そのまま全体文脈にしてはいけない。  
局所的に有効な文脈は、scope、conditions、negative\_constraints を保持したまま再利用しなければならない。

文脈は、ただ保存されるものではありません。  
文脈は、候補として戻され、警告と由来を伴い、人間の判断によって引き受けられ、その判断の来歴ごと検証可能に移送される。  
そして、部分文脈は、条件を失わずに再利用されなければならない。

それが、Sayaneの現在地です。
