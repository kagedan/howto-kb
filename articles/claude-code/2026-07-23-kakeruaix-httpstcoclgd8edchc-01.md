僕もテンプレを集めてた側だったので、わかる。あれが最前線だった時期は確かにあった。でも海外の議論を追うと、プロンプトの書き方の話はもうほとんど出てこない。

# ① プロンプト——良い指示文を書く

最初の時代は、とにかく「聞き方」の勝負でした。

同じことを頼んでも、聞き方で答えの質が全然変わる。だから「あなたはプロの編集者です」と役割を与える。見本を先に見せる。手順を番号で区切る。そういうテクニックが山ほど共有されてた。

たしかに、指示が上手いと返ってくる答えは良くなる。

ただ、この時代の限界もはっきりしてた。毎回、人間が書く。10回頼むなら10回書くし、チャットを閉じたら文脈は消えて、明日また同じ説明を最初からやる。しかも上手く書けても、良くなるのは「その1回の答え」だけ。指示のうまさは、どれだけ磨いても資産にならない。

だから話は「書き方」から「読ませ方」に進んだ。

# ② コンテキスト——AIに何を読ませるか

やってみるとわかるんですが、AIの答えがズレる原因の大半は、指示が下手だからじゃない。**AIがこっちの事情を何も知らない**からです。

何も知らないAIに「集客どうしよう」と聞けば、一般論が返ってくる。当たり前で、向こうはこっちの商品も、客層も、今まで何を試したかも知らない。

そこで、指示の前に「読ませるもの」を整える方向に進んだ。自分が何者で、何をやっていて、どんなルールで動いているか。前提・資料・過去の経緯を、頼む前からAIが読める状態にしておく。

僕も、仕事を頼む前に前提を1枚のシートにして渡すようにした。それだけで毎回の説明し直しが消えて、返ってくる答えが「世間の話」から「うちの話」に変わった。

Anthropicも「コンテキストエンジニアリングはプロンプトエンジニアリングの自然な進化」と公式に書いてます。

ただ、ここまで来ても、AIは聞かれたら答えるだけ。読ませた上で「次これやって」と動かすのは、相変わらず毎回人間でした。

# ③ ハーネス——AIが働ける環境を組む

チャットのAIは、答えを「言う」ことしかできません。「このファイルを直して」と頼んでも、直し方を説明して終わり。実際に手を動かすのは自分。

そこで、AIに道具と作業場を渡す段階が来た。ファイルを読み書きできる。コマンドを実行できる。ブラウザを触れる。こうなるとAIは「言う」から「やる」に変わります。「資料をまとめて」と頼めば、実際にファイルを開いて、書き換えて、保存までやる。

同時に、どこまで触っていいか（権限）と、失敗したらどう止めるか（安全装置）も先に決めておく。手を持たせるなら、事故らないための線引きが要るからです。この道具・作業場・権限の一式が「ハーネス」。

身近な例がClaude Codeそのもので、あれはClaudeという頭脳に道具一式が付いたものです。今月、海外最大級のAIカンファレンスでこの層の話だけで丸ごと1トラックが立ったくらい、いま議論のど真ん中にある。

環境はできた。ただ、その環境の中で何をどの順でやるかは、まだ人が都度指示してた。

# ④ ループ——手順ごと任せて回す

次は、1回ずつ頼むのをやめる段階です。

仕事は、だいたい同じ流れの繰り返しでできてる。調べる→作る→確認する→記録する。この流れを丸ごとAIに渡して、自動で何周も回してもらう。

このとき人間が決めるのは「どうなったら終わりか」の停止条件だけ。「テストが全部通ったら終わり」「下書きが5本できたら終わり」。先に決めておかないと、どこまでも回り続けるか、中途半端なところで勝手に満足して止まるかになる。

僕のこのアカウントも、毎朝AIがXを巡回してネタを集め、下書きを書き、僕のダメ出しを記録する、というループで回ってます。僕がやるのは、上がってきたものへの✅かダメ出しだけ。1周ごとにダメ出しが貯まるので、来週のループは今週より少し賢い。

ここにも限界がある。ループは一本道。決めた流れは回せるけど、状況次第でやることが変わる仕事や、複数の流れが絡む仕事は、1本のループに収まらない。
