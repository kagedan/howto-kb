Claude Codeを導入する会社は増えてきたけど、ちゃんとした導入方法に関する議論はまだまだ足りてないように思う。

うちでは業務進行やタスク管理なども含めて、Claude Codeを基盤になるように改革(を現在進行形で進めてる)。

せっかくなので、どういうふうに会社でClaude Codeを導入しているか書く。

・事故は起きるものとして設計する
・従業員のPCにはインストールさせない
・GitHubで情報共有
・本物のAPIキーは従業員に渡さない
・Claude Code導入前に決めること

1.前提として事故は起きるものとして設計する

従業員が悪くなくても何もしなくても事故は起きる。うっかりなのか、知識不足なのかは関係なく事故は起きる。だから性善説でも性悪説でもなく、事故が起きる前提で設計することが重要

2.従業員のPCにはインストールさせない

Claude CodeをPCにインストールすると、どうしても情報漏洩のリスクや漏洩時の被害が拡大するため、Claude Codeの直接インストールはNG。

実行環境はブラウザから、会社が用意したテンプレと設定で丸ごと立ち上げることにした(従業員の変更不可)。

Claude Codeの実行環境とローカルファイルが切り離されるから、PCからの情報漏洩は起きないし、全員が同じ構成と設定に統一できるのが大きなメリット。

あと何かあったら環境ごと使い捨てにできるのも、あまりITリテラシーが高くない従業員に使わせるには都合がいい。

3.GitHubで情報共有

Claude Codeで業務をやっていると成果物も過程も含めてGitHubにアップしてくれる。

GitHub上になんでもあるし、誰が何を持っているかを気にすることもない。Claude Codeに聞けば、何がどこにあるかが一発でわかる。最高だ。

4.本物のAPIキーは従業員に渡さない

APIキーの扱いは非エンジニアには難しすぎると判断した結果、APIキーは使わせないことにした。それでも業務をやっていると、どうしてもAPIキーを使いたい場面が出てくる。

その場合は、会社の中継サーバーを経由させて使わせる。要するに本物のAPIキーは中継サーバー側にあって、従業員の手元にはない(+あっても使えない)。つまりは従業員からAPIキーが漏洩することはありえない。

従業員が持っているのは会社で構築した中継サーバー用のキーだけ。それ自体は会社側でいつでも無効化できる。さらに使用量もモデルも全て会社がコントロールできる。

漏洩したら困るものは本人に持たせないことで、会社側も従業員側も守られる仕組みにしている。パスキーと似た発想。

 5.Claude Code導入前に決めること

Claude Codeの導入で最初にやるべきなのは、使い方を覚えることよりも、何をどこまでClaude Codeに任せるか決めること。

ぶっちゃけ全ての権限と情報をClaude Codeに扱わせるのが最も生産性を上げることになるが、その何十倍もリスクは大きくなる。

そこをどう考えるのかは、会社判断で正解があるものではないが、少なくともこの大AI時代において従来と同じセキュリティ要件を求めるのは無理があると思っている。

これからの仕事ではAIを使うのは前提として、どのように使うかが重要で、会社経営でも最重要課題になる。そのためにもどうやってAIを会社に導入するかという議論はもっと加速させていかないといけない。経営者層の方は、ぜひ情報交換しましょう。

くれぐれも「AIなんかすごいから、あとは現場でよろしく」と経営者がならないように。結局AI導入がうまくいっている会社は例外なくトップがAIをめちゃくちゃ触ってるし、理解しているところだと思う。
