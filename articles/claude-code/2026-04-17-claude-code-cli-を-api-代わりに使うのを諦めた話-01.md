AutoCAD 上で動作する AI エージェントを個人で開発しています。  
このエージェントは LLM モデルを選べる設計にしていて、OpenAI の ChatGPT、Google の Gemini、ローカルの LLM、そして Anthropic の Claude の4系統に対応する計画です。

通常、これらの API を使うと従量課金になります。使えば使うほどコストがかさむため、できれば避けたい。ユーザー目線でも API キーを取得するよりサブスクアカウントでログインする方が導入の壁が低くなります。

そこで目を付けたのが Claude Code CLI です。ちょうど自分が Claude の Max プランを契約していたこともあり、アカウントログインだけで利用可能な Claude Code CLI を経由すれば、従量課金の API を使わずにサブスク契約の範囲内で Claude が利用できるのではないか、というところから始まりました。

![Claude Code CLI 起動画面](https://static.zenn.studio/user-upload/df7fa943196e-20260417.png)

つまり Claude Code CLI を C# アプリから呼び出して、API 課金を回避する構成です。

## やろうとした構成

C# アプリから Claude Code CLI をサブプロセスとして起動して、プロンプトを渡して応答を受け取る、というシンプルな構成です。

実装の詳細は本題から外れるので省きますが、ユーザーの PC 上で動いている Claude Code を呼び出すことで、API キー契約なしに Claude の応答を得られる構成です。

![AutoCAD 上で動作する AI エージェント](https://static.zenn.studio/user-upload/1c22d0bdb5a6-20260417.png)

結論から言うと、この方式には2つの問題がありました。

## 問題1: レイテンシが大きい

まず気づいたのがレイテンシの問題です。

Claude Code CLI は Node.js ベースのツールで、毎回の起動にそれなりの時間がかかります。加えて CLI 自体の初期化ロジックが毎回走るため、単純な API 呼び出しと比較して応答までに時間がかかります。

自分のシステムは Observe / Plan / Execute の三層構造(エージェント設計でよくあるパターン)になっていて、1回のユーザー入力に対して複数回の LLM 呼び出しが発生します。各呼び出しにオーバーヘッドが乗ると、体感ですぐに待ち時間が積み上がります。

AutoCAD で「線を引いて」のような軽いタスクをお願いしたときに、何十秒も待たされる UX はさすがに許容できません。

…ただし、これは副次的な問題でした。より深刻な問題が後から見えてきます。

## 問題2: Claude Code 側のシステムプロンプトが干渉する(こちらが本題)

より本質的だったのは、応答品質への影響です。

Claude Code CLI は「対話的なコーディングエージェント」として設計されており、その用途に最適化されたシステムプロンプトが内部で適用されているようです(具体的な内容は公開されていないため推測です)。

このプロンプトは Claude Code の本来用途では必要不可欠ですが、別アプリから呼び出す場合にはノイズとして作用します。

こちらが書いたシステムプロンプトと Claude Code の内蔵プロンプトが競合することで、応答がこちらの意図する形式から外れやすくなる現象が発生しました。

結果として:

* **トークン消費が増える(内蔵プロンプト分が毎回加算される) ← これが一番痛かった**
* 応答精度が下がる(意図しない方向に引っ張られる)
* 期待した応答形式にならず、リトライが発生する

本来、自分のアプリは LLM からの応答をユーザー指示に応じて柔軟に整形できる設計になっています:

![ブロック定義情報のテーブル表示](https://static.zenn.studio/user-upload/1d3b39af2aca-20260417.png)

たとえば上の画像はユーザーが「テーブルで出して」と指示したときの応答例ですが、こういった指示通りの応答を返すには、LLM がこちらのシステムプロンプトを素直に受け取ってくれる必要があります。

Claude Code の内蔵プロンプトが混入すると、こちらの指示が弱まって応答形式が崩れやすくなり、形式崩れやリトライが頻発する結果になりました。

また、Claude の利用量バーを見ていると、数回のやり取りだけでぐんぐん消費量が増えていくのが分かります。これはおそらく、こちらで設計したシステムプロンプトと Claude Code 側で設定されている(であろう)システムプロンプトが重なり、総トークン数が膨らんでしまった結果だと思います。

Max プランではある程度耐えられましたが、下位プランだとすぐに上限到達するレベルの消費でした。

本来 API 課金を避けるためにサブスク経由を選んだのに、トークン消費が膨らむ構造は本末転倒です。**「自分のシステムが Claude Code の設計思想に引っ張られている」** という、構造的な問題でした。

## 規約上の懸念もあった

技術的な問題に加えて、規約上のグレーゾーンも気になりました。

Claude Code はサブスクリプション製品で、対話的なコーディング用途を想定して設計されています。これを別アプリから API 代替として使う行為が、利用規約上どう解釈されるかは微妙なところです。

個人利用の範囲なら問題ないかもしれませんが、プロダクトとして配布する構成に組み込むのは安全性の観点で不安が残ります。

技術的な行き詰まりと合わせて、この構成は取りやめる判断をしました。

## Anthropic API 直接呼び出しに切り替え予定

Claude への接続は、Anthropic API を直接叩く方式に切り替える予定です。

HTTP で `/v1/messages` エンドポイントを呼び出す、シンプルな実装になる想定。API キーはユーザーに各自契約してもらう前提に変更します。

こうすることで:

* レイテンシは API のレスポンス時間のみ
* 自分のシステムプロンプトが 100% 効く
* 期待した形式で応答が返ってくる
* 規約上も明確にセーフ

サブスク活用でコストを抑えるアイデアは諦めることになりましたが、プロダクトとして健全な構成になるはずです。

## 学んだこと

この経験から得た教訓は、Claude Code 固有のものではなく、LLM エージェント製品全般に当てはまる一般原則だと思います。

**AI エージェント製品は、その用途に特化したシステムプロンプトを内蔵している。別の用途で組み込もうとすると、このプロンプトがノイズとして作用する。**

Cursor や Windsurf、Claude Code のような AI コーディングツールは、コーディング支援に最適化されています。ChatGPT Desktop も、汎用対話に最適化されたものです。

これらを「素の LLM API の代わり」として使おうとすると、各製品の設計思想が自分のシステムと衝突します。

結局、自分のアプリに組み込むなら、素の API を直接叩くのが最も素直でした。サブスクでコストを浮かせたい誘惑はありますが、設計思想の違いは技術的に越えられない壁になります。

もし AI サブスクを API 代わりに使いたくなっている人がいたら、この壁の存在を知った上で判断した方がいいかもしれません。

## 補足: Gemini CLI について

なお、Gemini CLI は同じく内蔵システムプロンプトを持ちますが、オープンソースでシステムプロンプトのオーバーライド機能も公式に提供されています。つまり、プロンプト干渉問題を回避できる可能性があります(未検証)。

AI エージェント製品を API 代わりに使いたい場合、「オープンソースかどうか」「プロンプトをカスタマイズできるか」を確認するのが一つの判断基準になりそうです。

同じような落とし穴にハマる人の参考になれば幸いです。
