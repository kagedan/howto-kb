# FinOps の誤解：Claude Code 活用で目指す「セキュリティと堅牢性を犠牲にしない」AI 時代のコスト最適化

クラウドコストの最適化は、2026 年現在のクラウドエンジニアにとって慢性的な課題です。経営層は月次請求の数字でコスト削減を要請し、現場では Reserved Instance、Savings Plans、Spot Instances、Right-Sizing といった既存の手段を組み合わせて対応してきました。

ここ半年でその選択肢に Claude Code が加わりました。IaC コードを読ませて PullReq 上で月額差分を試算する、コスト超過を hooks で弾く、Well-Architected 6 本柱の観点で自動レビューをかける――こうした事例が増え、コスト最適化のレパートリーは確実に広がっています。

ただ、こうした取り組みが「コストを下げること」自体を目的化していないか、は別途整理が必要なテーマです。タイトルの「FinOps の誤解」が指しているのは、FinOps を独立した目的として扱った結果、セキュリティや堅牢性とのトレードオフを見失う状態のことです。Claude Code を題材に、FinOps を「コスト起点」ではなく「設計起点」で捉え直す、というのが本記事の立て付けです。

## 1. 事例が出揃った先に考えるべきこと

Claude Code とクラウドコスト最適化を組み合わせる事例について、IaC コードを読み込ませて PullReq 上で月額差分を試算する、Well-Architected 6 本柱で自動レビューをかける、Infracost と組み合わせて閾値超過をブロックする、Terraform のプラン時にコスト見積もりを差し込む etc... 実装の素材としては十分なものが揃ってきています。

これらに共通しているのは、いずれも「コストを起点に Claude Code をどう使うか」という観点で組み立てられている点です。実装パターンの蓄積は現場が動くための前提条件であり、それ自体は否定すべきものではありません。問題があるとすれば、個別の手段をいくら積み上げてもその先にある設計判断は出てこない、という点です。

例えば、PullReq 上で月額差分を出せるようにしても、その差分を見てどう判断するか――何 % 増なら許容するか、どこからレビューを止めるか、何を犠牲にしてその金額を削るか――は別途設計されている必要があります。これを欠いたまま手段だけを揃えると、「数字を見せられたら下げる」という反射的な運用に陥りやすくなります。コストが下がっているように見えても、削ってはいけない箇所まで削ってしまう、という事故はここから起こります。

事例が出揃った段階で問われるのは、**「どう試算するか」ではなく「何を基準に判断するか」** です。以降では、その基準を「コスト」ではなく「設計」に置き直したときに Claude Code をどう使うべきか、という観点で整理していきます。

## 2. コストは設計の副産物

クラウドアーキテクチャの設計を進めるとき、まずトラフィック特性や SLO、運用体制、セキュリティ要件、データの分類や越境制約、障害時の復旧目標といった条件があり、それらを満たす構成を選んでいきます。その結果として、月額コストは特定のレンジに落ちます。設計が決まれば、コストはおのずと決まる。これが本来の順序です。

「コスト最適化」という言葉が独り歩きすると、この順序が逆転します。月額を下げることを先に決め、その範囲に収まる構成を後から探す。一見合理的に見えますが、満たすべき要件が後から評価される構造になるため、削ってはいけない箇所が削れてしまうリスクが入り込みます。

具体的な失敗パターンはいくつもあります。スポット中心の構成にしたら起動失敗時の復旧時間が伸び、結果として SLO 違反でビジネス側の信頼を失う。NAT Gateway のコストを抑えるために VPC エンドポイントの整備を後回しにし、外部 IP を直書きで穴を開けて監査で指摘される。単価の安いリージョンに動かしたらデータ越境の制約に引っかかり、コンプライアンス対応で追加工数が発生する。いずれも「コストを起点に判断した」結果として発生しています。

本来の FinOps は、こうした副作用を含めて総コストを最小化するための営みです。**月次請求の数字だけを見ているとこの視点が抜け落ち、「下がっているから良い」という単純な指標に置き換わってしまいます。** Claude Code をはじめとした AI エージェントを取り込むときも、この順序を見失わない姿勢が前提になります。設計が先で、コストは後についてくる。これを実装ワークフローに反映させることが、本記事で扱いたい議題です。

## 3. 設計段階で Claude Code が本当にやるべきこと

Claude Code の活用が「実装後の PullReq レビュー」中心で語られがちなのは、CI/CD ワークフローへの組み込みがしやすいからです。PullReq の hooks で Infracost を回し、差分を Claude に説明させる。この構成は分かりやすく、効果も測りやすい。ただし、ここに止まると Claude Code は「実装後の整合性チェッカー」にしかなりません。

設計起点で考えたとき、Claude Code をより効果的に使えるのは設計フェーズ、すなわち IaC コードが書かれる前の段階です。要件の整理、想定トラフィックの粒度、SLO の数値、データの分類、想定障害シナリオを与えたうえで、Lambda・Fargate・ECS on EC2・自前 EKS といった選択肢ごとの構成案を出させ、それぞれの月額レンジと、許容するトレードオフを並べてもらう。ここで重要なのは、最安構成を一案だけ出させるのではなく、複数案を並列で出させることです。

複数案を並べてもらうと、コストだけでは測れない比較軸が浮き上がります。例えば、トラフィックが日中だけスパイクするワークロードに対して、Lambda は確かに月額が安くなりやすい構成ですが、コールドスタートの許容範囲、VPC 内通信の前提、メモリ上限と実装制約、そして観測性の組み立て直しという別軸のコストが発生します。Fargate は中庸ですが、起動時間とイメージ管理運用が前提となる。EKS は固定費が乗りますが、組織にすでに Kubernetes の運用体制があるなら追加負荷が小さい。これらを一覧で並べたとき、初めて「自分たちにとって何が最適か」という議論ができるようになります。

Claude Code はこの「並列の複数案出し」と「軸ごとの比較整理」を、人間が手で書くより圧倒的に高速にこなせます。設計者の仕事は、出された候補から自社の要件に合うものを選び、選ばなかった理由を言語化することに移ります。コストはこの選択の結果として算出される数字であり、選択そのものではありません。

## 4. 過剰設計と過小設計の両方を見抜かせる

エンジニアが設計する IaC コードには、コスト要素を度外視したオーバースペックが入り込みやすいという傾向があります。例えば、t3.medium で十分なところに m6i.large を置く、プロビジョンド IOPS が不要なところに付けてしまう、Multi-AZ の Aurora を要件外の環境で立てる。これらに対して本人に悪意や特定の意図もなく、不確実性に対するバッファを取った結果ですが、積み上がると月額の数十パーセントが過剰分で占められることも珍しくありません。

ここで CloudWatch からエクスポートしたメトリクスや実際の負荷データを Claude Code に読み込ませると、過剰設計の検出は劇的に楽になります。「直近 90 日の CPU 使用率は平均 6%、p99 で 18% に収まっているので、インスタンスタイプを 2 段下げても余裕がある」「IOPS のピークは規定値の 15% で、プロビジョンド IOPS は外せる」といった具体的な提案を、実測ベースで出してくれます。設計レビューの粒度が、勘・経験ベースから実測ベースに変わります。

ただし、ここで止めると逆方向の事故が起きます。実測に過剰に依存して下げすぎると、年に数回しかないキャンペーンや決算期のピークで詰まり、機会損失や顧客影響を出してしまう。AI エージェントが見ているのは「過去の実測」であり、「今後発生しうるピーク」や「事業計画上の負荷増」は与えなければ分かりません。CloudWatch には現れていない既知の高負荷イベント、新規顧客のオンボーディング予定、季節性、これらは人間が業務知識として持ち込む必要があります。

実用的な使い方としては、Claude Code に出させた「下げ提案」を、業務知識を持ったエンジニアが「ピーク想定とのつき合わせ」で評価する、というワークフローが安定します。実測で過剰を炙り出し、業務知識で過小を防ぐ。両方が揃ったときに、初めて「適正設計」と呼べる状態になります。コスト削減はここでも、結果として現れる数字でしかありません。

## 5. コスト SLO とガードレール

機能要件の SLO、例えば「月次可用性 99.9% 以上」は、多くの組織でレビューの観点として定着しています。一方で、月額コストはレビュー対象として SLO と同じ重みでは扱われていないことが多いです。設計段階で「このシステムの月額上限は X 万円」というラインを引き、それを超える構成を素通りさせないルールは、まだ整備の余地があります。

ここに Claude Code を組み込むと、コストを「設計時のガードレール」として運用しやすくなります。月額上限と、その内訳ごとの目安をリポジトリ内に宣言しておき、IaC コードに変更が入った時点で Claude Code が想定影響を試算する。閾値を超えそうな構成変更には、変更者に「この変更で月額が X 円増えます。許容しますか、別案にしますか」と問いを返す仕組みを挟み込めます。コストを下げるための仕組みではなく、設計判断を遅らせない、漏らさないための仕組みです。

同時に重要なのが、コストの可視化基盤そのものを守るガードレールです。FinOps の運用は、リソースに付与されたタグや命名規則によるコスト配賦（コストの割り当て）が前提になっていますが、エージェントが生成するリソースはこの規約を抜けやすい性質があります。プロンプトに書き忘れる、テンプレートから漏れる、新しいリソース種別で規約が未整備、といった理由でタグが付かないと、コストはそのまま「不明」カテゴリに流れます。設計時の hooks でタグの有無・命名規則違反・必須属性の欠落を弾く仕組みを最初から組み込んでおくと、可視化の前提が崩れません。

これらは「コストを下げる仕組み」ではなく「FinOps が機能し続けるための土台を守る仕組み」です。Claude Code は、この土台維持を機械的に肩代わりさせる相手として相性が良く、組織・チームのレビュー負荷を増やさずに、配賦の正確性と設計時の制約遵守を維持できます。

## 6. 人間が判断する境界

ここまで Claude Code に任せられる領域を整理してきましたが、同じくらい重要なのは「任せてはいけない領域」を明確にしておくことです。AI エージェントの提案は確率的に正しいだけで、組織固有の事情や、ビジネス上の優先度、過去の障害から学んだ判断軸は内蔵されていません。

例えば、SLO を一時的に下げてでもコストを優先するフェーズかどうか、コンプライアンスの解釈が分かれる構成でどこまでが許容範囲か。こうした判断は、過去の意思決定の経緯や社内のステークホルダ間の力学、業界規制の解釈実務といった、コードや IaC やメトリクスには現れない情報を踏まえて下すものです。

**Claude Code を実装に組み込むときは、これらの判断ポイントを「自動化されない」状態で残す設計が重要になります。** 閾値超過時に必ず人間のレビューを挟む、特定タグの付いたリソースは変更時に承認フローを通す、リージョン間のデータ移動を伴う変更は別レーンで扱う、といった分岐を、ワークフロー自体に組み込みます。**「全部自動化する」ではなく「自動化すべきところと、判断のために残すところを切り分ける」のが設計の本質です。**

ここはクラウドエンジニアの職能の中心が移っていく領域でもあります。手を動かして IaC コードを書く部分は、Claude Code との分業で確実に省力化が進みます。一方で、「何を自動化し、何を残すか」「組織にとって正しい設計判断とは何か」を定義する役割は、引き続き人間に残り続けます。むしろ、エージェントの判断を組織に対して説明可能にする力、設計上の選択を社内外のステークホルダに筋道立てて伝える力が、これまで以上に重要になります。

## 7. おわりに： 真の FinOps とは何か

本記事を通じて整理したかったのは、Claude Code を「コストを下げる道具」として位置づけることの限界です。**コストは設計の結果として現れる数字であり、独立して下げるべき対象ではありません。設計のバランスを取る過程で Claude Code を使うと、結果としてコストも適正なレンジに収まる、というのが本来の順序です。**

真の FinOps は、コスト削減運動ではなく設計規律です。要件・SLO・セキュリティ・堅牢性・運用負荷、これらを総合した最適点を選び続けること。月額の数字はその最適点の射影に過ぎません。Claude Code は、この最適点を探すための比較・整理・検出を高速化する道具として価値を発揮します。

エージェントが普及した時代におけるクラウドエンジニアの仕事は、「いかにコストを下げるか」ではなく「いかにバランスの取れた設計を組織の標準として定着させるか」に重心が移っていきます。コストはそのバランスから自然に落ちる副産物であり、追いかけるべき対象ではない。この順序を意識しておくだけで、目の前の手段の選び方も、Claude Code の使い方も、自然と変わってきます。
