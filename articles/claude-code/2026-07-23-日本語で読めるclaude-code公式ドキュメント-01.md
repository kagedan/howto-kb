Claude Codeを学ぼうとして、つい英語の動画や海外の記事から探し始めていないでしょうか。実は公式サイトに日本語のドキュメントがそろっています。インストール手順からエージェントの仕組み、メモリやMCPまで、無料で読めます。

とはいえ、目次を上から順に全部読む必要はありません。まず最初の変更を自分の手で体験し、その仕組みを理解し、仕事の進め方を整えたうえで拡張機能へ——この流れを5日間に分けて進めると、機能名を暗記するだけで終わらずに済みます。

## 10秒で分かるまとめ

## 1日目：クイックスタートで、まず一度変更してみる

最初に読むのは「概要」です。Claude Codeはターミナルだけでなく、Web、デスクトップ、VS Codeなどからも使えます。全体像をつかんだら「クイックスタート」へ進みましょう。

クイックスタートは8段階構成です。インストールとログインに始まり、最初の質問、コードの変更、Gitの利用を経て、バグ修正まで進みます。ここは読むだけで終わらせず、壊しても元に戻せる小さなプロジェクトをひとつ用意してください。最初の指示は「このプロジェクトの構成を説明して。まだ変更はしないで」で十分です。

次に、文言や変数名など、差分を目で確認できる変更をひとつだけ頼んでみます。変更の前後を`git diff`で確認できれば、初日の成果は完成です。大きな機能追加を狙うより、「頼む → 差分を見る → 元に戻せる」という一周を先に体験しておきましょう。

ひとつ注意点があります。ドキュメントの閲覧は無料ですが、操作を試すには利用環境が必要です。対象となるのは、Pro・Max・Team・EnterpriseのClaudeサブスクリプション、Claude Console、または対応するクラウド提供元です。なお日本語ページでも、インストール部分など一部に英語が残っています。

## 2日目：3段階のループを理解する

「Claude Codeの仕組み」では、作業を3つの段階で説明しています。コンテキストを集め、行動し、結果を検証する——この流れです。質問するだけなら最初の段階で完結し、バグ修正なら3段階を何度も繰り返します。

このページを読む目的は、長いプロンプトを書けるようになることではありません。Claude Codeが、ファイル検索・編集・テストを自分で選んで進める、という前提を理解するためです。利用者の側も途中で作業を止め、追加の情報を渡し、別のやり方へ向け直すことができます。

1日目の作業履歴を、この3段階に当てはめてみましょう。「どのファイルを読んだか」「何を変更したか」「何をもって合格を確かめたか」を3行に書き出すだけで十分です。検証の欄が空のままなら、次の依頼ではテストや期待する出力を先に渡すようにします。

## 3日目：ベストプラクティスを、ひとつだけ使ってみる

「Claude Codeのベストプラクティス」は情報量が多いので、一度に全部を取り入れようとしないでください。最初に試すのは「Claudeに、自分の作業を検証する手段を与える」です。公式ドキュメントは、合否を判定できる確認方法を渡すよう勧めています。たとえばテスト、ビルド、比較用のスクリーンショットなどです。

具体的には、「エラーを直して」で終わらせず、「修正後にテストを実行して、失敗が0件になったことを確認して」と付け加えます。確認方法がなければ、たとえ誤りが残っていても、Claude Code自身にはそれを判定できません。テストを実行できる環境と権限があれば、結果を読みながら修正を繰り返しやすくなります。

続いて「一般的なワークフロー」からは、まず「編集前の計画」だけを試します。調査・計画・実装をまとめて頼まず、最初は調査と計画で一度止めておきます。変更対象と検証方法に納得してから実装に進めば、差分が広がりすぎる失敗を減らせます。

## 4日目・5日目：必要な拡張だけを選ぶ

4日目は「Claudeがあなたのプロジェクトを記憶する方法」を読みます。Claude Codeのメモリには、人が書く`CLAUDE.md`と、Claudeが自動で書くメモリの2種類があります。コーディング規則や毎回使う確認コマンドは`CLAUDE.md`に置き、作業中に見つかった癖や気づきは自動メモリに任せる、という使い分けです。

5日目は、自分の困りごとに合わせてひとつだけ選びます。同じ手順を何度も貼り付けているなら「スキルでClaudeを拡張する」。外部の課題管理やデータに接続したいなら「MCP」。調査結果やログで会話が埋まってしまうなら「カスタムサブエージェントの作成」です。

ここで、Skill・MCP・サブエージェントを同時に導入する必要はありません。まずはひとつだけ追加し、導入前と比べて「説明の回数」「会話に貼り付ける量」「確認の手間」のどれが減ったかを見ます。もし何も減らないなら、設定を増やす前に、1日目から3日目で身につけた「依頼の仕方」と「検証方法」に立ち返りましょう。

日本語ドキュメントの価値は、すべての機能を覚えられることではありません。公式の説明を読みながら、小さな変更と3行の作業記録を残し、さらに検証方法と恒久ルールをひとつずつ決めていける——この点にあります。5日後にこの4つがそろえば、次に読むべきページを、自分の困りごとから選べるようになります。
