松尾研究所のからあげ（[@karaage0703](https://twitter.com/karaage0703)）です。

松尾研究所では 2026年2月ごろから、Slack上でOpenClawライクな社内AIアシスタントソフトを、一部のプロジェクトで試験運用してきました。この記事では、AIアシスタントソフトの概要・活用例・実運用アンケート結果についてまとめたものの一部を紹介します。

社内ツールの紹介ではありますが、「どのように業務でAIを活用していくか」という同じ悩みを持つ方の参考になれば幸いです。

## AIを活用して業務の負荷軽減をしたい

松尾研究所では、共同開発案件の数が増えるにつれてPMまわりの「情報検索」「議事録・週報の作成」「PRレビューの一次対応」といった業務負荷が無視できなくなってきました。

また、長期プロジェクトは、メンバーの入れ替わりにより「なぜこの方針になったのか」「過去にどんな議論があったのか」といった情報がSlack、Notion、GitHub等に散逸してしまい調査や把握が難しくなるという問題がしばしばありました。結果として、新しく入った人が毎回同じ質問をし、同じ調査を繰り返すといったことも発生していました。

**プロジェクトのことを何でも知っている「長老」** みたいな存在がいると助かるのですが、そういう役割の人がいないPJTもあるのが実情でした。

これらの問題を解決するため、PJTの重要な情報をすぐ引き出せる形でメモリとして蓄積した上で、AIがSlack・Notion・GitHubなどに分散している情報にいつでもアクセス出来るようにする仕組みが必要です。単発の質問応答AIではなく、**プロジェクトの集合知を人格化した存在**としてAIを育てていきたい。そう考えたときに、OpenClawのようなパーソナルなAIアシスタントをPJTで活用することでこの仕組みが実現できるのではないかという仮説を持ち、小規模に試験運用をすることにしました。

## AIアシスタントソフトの開発

「AIアシスタントが業務に役立つのか？」という仮説をクイックに実証するため、既存のOSSを活用する前提でOpenClawを中心に調査をしました。簡単にまとめた結果は、[OpenClawライクなソフトをまとめてみた](https://zenn.dev/karaage0703/articles/0e2e3484169db2)を参照ください。

現在の2026年4月末時点では、こういったAIアシスタントソフトは、元祖となるOpenClawとそれに次いでHermes Agentが人気です。これらについての詳細は弊社ブログの[OpenClawとHermesの違いを思想から理解する](https://zenn.dev/mkj/articles/9431e342db202f)という記事を参照ください。

これらのソフトの元祖であるOpenClawの活用も検討しましたが、非常に多機能で多くのプラットフォームに対応しているものの、社内Slackだけで使いたい場合を考えると、機能過多であること、バージョンアップが多く追従のコストが大きいこと、当時セキュリティリスクについての情報も多かったこともあり選択肢から外しました。他のソフトも比較検討し、最終的には当時OpenClaw系ソフトとして人気が高く（スター数が多く）、最小限の機能を持ちシンプルな[nanobot](https://github.com/HKUDS/nanobot)というOSSをベースにすることにしました（注：Hermes Agentは開発検討していた2月にはまだ公開されていませんでした）。

nanobotは、当時最小限の機能しかなかったため、松尾研究所の社内要件（プロジェクト単位の分離、GitHubプライベートリポジトリアクセス、Slack対応、Notion接続など）最低限の機能を盛り込み、mbotという名称で開発しました（コードは非公開となります）。

## mbot の概要

mbotはSlack の各プロジェクトチャンネルに常駐し、メンションされたスレッドでClaude Code/Codex CLI/Local LLMのセッションを動かすPython製のAIアシスタントです。

LLMのAPIでなく、Claude CodeやCodex CLIの機能（Agent Skills等）をそのまま活用できる構成にすることで、開発のコストを減らしました。

### 主な機能

* Slack対応：SlackのメンションでAIが応答。ファイルの授受、スレッド単位での継続会話が可能。チャンネルごとにトピックをシステムプロンプトとして注入可能
* 複数のバックエンドに対応：Claude Code/Codex CLI/Local LLMに対応することで、PJTの要件によって様々な選択が可能
* GitHub Apps認証対応：AIがアクセスできるリポジトリと権限をコントロール可能
* スケジュール機能：特定のタイミングでAIが自律的に動ける機能
* コンテナ対応：コンテナ上の隔離した環境でAIが作業できるようにする機能

2月時点のnanobotは、Slack対応始めとして、ほとんどの機能がなかったため、必要となる機能を最低限追加した形です。

また、PJTで必要となる個別の機能（Notionへのアクセス、スライドの作成、GitHubのPull Requestレビューなど）は、基本的にはAgent Skillsで拡張していく方針としました。

### アーキテクチャ

構成の概略としては以下となります。PJTごとにワークスペースを持ち、SkillやMCPでWeb・Notion・GitHubとAIを接続します。

### ワークスペース

ワークスペースは、AIツールが作業を行う場所で一般的に`AGENTS.md`などのシステムプロンプトやSkillファイルなどがワークスペース以下におかれます。mbotのワークスペースは、Claude Code/Codex CLIのAGENTS.md（CLAUDE.md）やAgent Skillsの仕組みをそのまま使うことで、蓄積したデータを極力特定のAIツールに依存しない形にしています。

AIツールによって、システムプロンプトのファイル名やスキルの格納場所が少しずつ違ったりもするのですが、そういった違いは、シンボリックリンクで吸収しています。また、独自のファイルなどはなるべくおかず、分かりやすいシンプルな構成にしています。以下はワークスペースの例です。

```
workspace/
├── AGENTS.md # システムプロンプト（実体）
├── CLAUDE.md # → AGENTS.md へのシンボリックリンク（Codex CLI用）
├── skills/ # スキル（機能の追加先）
├── .claude/skills # → skillsへのシンボリックリンク（Claude Code用）
├── .agents/skills # → skills へのシンボリックリンク（Codex CLI用）
├── memory/ # メモリ（日々の記憶の保存先）
├── MEMORY.md # メモリ（重要な記憶の保存先）
└── README.md # PJT情報のまとめ
```

## 試験運用

### 試験運用の条件

運用としては、クラウド上の隔離したインスタンスでbotを動かしていました。

botを使用する人はPJT関係者に制限し、botがアクセスできるNotionページはNotionのコネクトの設定、GitHubリポジトリはGitHub Appsの設定で、PJTのメンバーがアクセスできる範囲と同等に制限しました。

### botの活用方法

具体的な活用方法です。前提として、botにはあらかじめ、プロジェクトのNotionや資料をもとに、プロジェクト概要をREADMEとしてまとめた上で、システムプロンプトに注入しています。

以下の例は、PJT情報や個人情報が含まれるため、一部画像加工していることをご了承ください。

#### 週報の記載

Notionの情報をもとに週報を書いてくれます。

試行した3PJTは、開始時期が大きく異なることもあり、タスク管理の方法がNotion、GitHub Projectsなど使用するプラットフォームや管理方法もバラバラだったのですが、初回スキルをチューニングすると、2回目からは、自律的に情報を探して週報をまとめてくれるようになりました。

![](https://static.zenn.studio/user-upload/187a33179ebd-20260428.png)

#### 論文やテックニュースの自動配信

botにはスケジュール機能もつけているので、例えば毎朝の決まった時間にPJTに関連するニュースや論文を配信することが可能です。

![](https://static.zenn.studio/user-upload/fd6448f30610-20260427.png)

ポイントとしてbotがPJT情報を持っているため、ニュースがPJTでどのように活用できるかまでパーソナライズドしてアドバイスしてくれます。また、深堀りしたい場合はリプライで聞くと、botが更に詳しい情報を教えてくれます。

#### コードや資料のレビュー

コードのPull Request(PR)のURLや資料をSlackで添付すると、手軽にレビューしてもらうことができます。

![](https://static.zenn.studio/user-upload/5d0434c2b9b9-20260428.png)  
*Pull Requestのレビュー*

![](https://static.zenn.studio/user-upload/98844bcef610-20260427.png)  
*資料のレビュー*

コードレビューに関しては、弊社テックブログの[Claude Code Orchestra: Claude Code × Codex CLI × Gemini CLIの最適解を探る](https://zenn.dev/mkj/articles/claude-code-orchestra_20260120)記事を参考に、マルチエージェントのスキルを構築しています。これにより「みんなでレビューして」と一言付け加えるだけで、Claude Code/Codex CLI/Gemini CLIを用いたマルチエージェント的なレビューを実現できます。

#### その他の活用

上記以外も、PJTごとに様々な活用をしています。基本的には個別のSkillを作っています。Skill自体の作成も、botに依頼して、bot自身に作成してもらっています。気づいたら、いつの間にか便利なSkillが増えていることもしばしばです。

以下はその他の活用方法の例です。

* Notionの調査
* スライドの作成
* アイデア出し
* etc...

## 試験運用の利用者アンケート

2026年2〜4月にかけて、3つの社内プロジェクトで試験運用した、利用者アンケートデータの一部を公開します。

試験運用に参加している3PJTのメンバーにアンケートを取りました。使用しているのは、10名以上ですが、そのうち8名の回答となります。

#### 何に使っているか

![主な使用用途](https://static.zenn.studio/user-upload/70cf843f8713-20260426.png)

コードレビュー（85.7%） と 資料検索・情報収集（85.7%） が突出しています。自由記述でも「PRのコードレビューを、他のメンバーにレビューしてもらう前に見れる」「Notion の検索や GitHub の大量のコードベースを横断して人間より早く回答できる」が気に入っている機能として挙がりました。

#### 業務効率化への貢献度

![業務効率化への貢献度](https://static.zenn.studio/user-upload/b4e727885410-20260426.png)

10 段階評価で **7 以上が 100%**、9 以上が 62.5%。使っている人にとってはしっかり効いている印象です。

#### 時間短縮効果

![週あたりの時間短縮](https://static.zenn.studio/user-upload/d1bc055b86ae-20260426.png)

週あたり30分以上の短縮が 75%、1 時間以上も25%。「週に数回」利用のユーザーが多いことを踏まえると、1回あたり 10〜30分クラスの作業（コードレビュー、情報検索）をAIに任せてその分を別の仕事に回せているイメージです。

#### 全社導入すべきか

![全社導入推奨度](https://static.zenn.studio/user-upload/c69f8f33b5bc-20260426.png)

「強く思う（5）」が 50%、4 以上で 62.5%。少なくとも試用した人にとっては「会社全体で展開してよい」という手応えになっています。

### 使用者のコメント

自由記述で出てきた指摘も重要なので紹介しておきます。

* **重いタスクのタイムアウト**：Notion の過去議事録を全件精査するような長時間タスクでタイムアウトに引っかかり、逐次ログを残して呼び起こす工夫が必要だった。
* **短答以外への拡張**：コーディングして PR 作成、実験管理など、メンバーが普段やっているタスクも移譲できる余地がある（「人間が仕事をサボって雑にならない前提で」とのコメント付き 😅）。
* **出典へのリンク**：回答中に参照した資料へのリンクをもっと明示してほしい。
* **データソースの可視化**：「bot がどこのデータソースを参照しているのかわからず、使わなかった一因」というコメントも。**UI 以上に「何ができるかの説明」のチューニングが必要**だと分かった。

### 課題

実際に使用していて、特に今後の展開にあたり課題になると感じた点です。

#### セットアップ

セットアップは、クラウド環境、Slack Botの設定、Notionの接続設定、GitHub Appsの設定等、地味な作業が多岐にわたります。難易度が高い項目はないものの、知らない人が全て1から実施するのは正直かなり大変です。

広く社内展開していくためには、このセットアップを大きく簡易化・自動化する必要があると感じました。

#### コスト

気になるコストです。コストの内訳と条件はPJTによって異なります。本記事では詳細は非公開とさせていただきますが、今のところは、おおよそ1PJTあたり1〜2万円といった価格感になります。

ただ、当然コストは利用頻度と利用者が増えるほど基本的に増加していくので、コストの低減・費用対効果の可視化が重要になっていくと感じました。

#### セキュリティ

AIのセキュリティも、人間とは異なる新しい課題となります。こちらに関しては、社内の情報セキュリティを担当するチームと密に情報を共有し進めています。

試験運用では、対象となる人数を制限した上で、NotionやGitHubの権限を制限する形で、バックアップをとりつつ運用していました。使い方によっては、botがデータを破壊してしまう可能性もありましたが、幸いにも試験運用では特段大きな問題は発生しませんでした。

AIにできることを制限すると、それだけAIにできることが減り、AI導入の効果が減ってしまうため、安易にAIの権限を減らさずに、安心して使用できる仕組みを継続して検討しています。

## まとめ

社内向けAIアシスタントを3か月間という短い間ですが、試験運用した結果を簡単に共有しました。

アンケート結果や実際に自分自身やメンバーが使っている様子をみた所感になりますが、PJTでAIアシスタントを活用することは、非常に大きな価値と可能性があると感じています。

当初に期待していた効率化はもちろんですが、どちらかというと、PJTのコードや資料をチェックして「過去の議事録のこの指摘が反映されてない」とか「この論文の内容がPJTで使えそう」といった、PJTのクオリティを高める、まさにPJTの長老的なポジションを担ってくれる体感があります。

特に長く続いているPJTでは「そこは後でbotで確認して」や「botでまずレビューして」といったフレーズが頻繁にでるようになってきました。一方で、導入しただけであまり使われずに終わってしまったPJTもあったり、アンケート結果の効果からも、まだ活用が十分に進み、大きな効果が出ているとはいえないのも正直なところです。

ただ、社内では今回の結果に興味を示してくれたメンバーも多く、試験導入を追加で希望する声も出てきている状況です。またこのような盛り上がりから、導入と並行して、社内独自のAIアシスタントを開発するという動きも起きています。

松尾研究所では、このような社内基盤を含めて、AIと一緒に働くのが当たり前になることを目指して  
AIの社会実装を進めています。興味のある方はぜひ [採用ページ](https://matsuo-institute.com/recruit/) もご覧ください。
