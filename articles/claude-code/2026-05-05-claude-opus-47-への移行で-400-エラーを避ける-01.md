## サマリ

Claude Opus 4.7（2026/4/16 リリース）は、過去バージョンとの API 互換性を壊す変更を含んでいる。特に以下の 4 点は「Opus 4.6 で動いていたコードがそのままだと 400 エラーになる」直接の原因になる。

1. **`thinking.type: "enabled"` + `budget_tokens` は 400 エラー**。Adaptive Thinking（`thinking.type: "adaptive"`）のみサポート
2. **サンプリングパラメータ `temperature` / `top_p` / `top_k` が廃止**。Bedrock ではリクエストが拒否される
3. **`thinking.display` のデフォルトが `"summarized"` → `"omitted"`**。UI で思考過程を表示している場合、明示指定しないと空ブロックが返る
4. **思考深度の制御は `output_config.effort`**（5 段階の soft guidance）。配置は `thinking` 配下ではなく **リクエスト root 直下の `output_config` 配下**

併せて、`output_config.effort` が **Bedrock 経由でも 3 ルート（`InvokeModel` / `Converse` / `bedrock-mantle`）すべてで透過的に通る** ことを実機検証した。配置ミスでの典型エラーも合わせて載せる。

Claude Code / Kiro CLI は Anthropic 公式 SDK が自動吸収してくれるが、MCP サーバー・カスタム Bedrock クライアント・LangChain 系フレームワーク経由の呼び出しではこれらの変更が直接表に出るため、事前に潰しておきたい。

## はじめに

本記事は以下の読者を想定する。

* Bedrock / Anthropic API 経由で Claude を呼んでいる開発者
* Opus 4.6 / Sonnet 4.6 から Opus 4.7 への移行を検討している
* Claude Code / Kiro CLI を使っているが、カスタム設定・MCP で手を入れている
* 「移行時の落とし穴」を事前に把握したい

構成は以下のとおり。

1. Opus 4.7 で何が変わったか（4 つの破壊的変更）
2. 新しい制御手段 — Adaptive Thinking と `output_config.effort`
3. Bedrock 経由の実機検証（3 ルートすべてで effort が通ることの確認）
4. 移行チェックリスト
5. 400 エラーの発生源とデバッグパス

## 1. Opus 4.7 で何が変わったか

Anthropic 公式ドキュメント（Extended Thinking / Adaptive Thinking / Effort ページ）と AWS Bedrock の Opus 4.7 モデルカードから、2026/4/16 リリースで導入された互換性影響をまとめる。

### 1.1 `thinking.type: "enabled"` + `budget_tokens` は 400 エラー

過去モデル（Sonnet 3.7 / Opus 4 系 / Sonnet 4 系 / Haiku 4.5 等）では、Extended Thinking の有効化は以下だった。

```
{
  "thinking": {
    "type": "enabled",
    "budget_tokens": 10000
  }
}
```

Opus 4.7 ではこの指定が **400 エラーで拒否される**。公式 docs より:

> Manual extended thinking (`thinking: {type: "enabled", budget_tokens: N}`) is no longer supported on Claude Opus 4.7 or later models and returns a 400 error.

代わりに `thinking.type: "adaptive"` を使う。

```
{
  "thinking": {
    "type": "adaptive"
  }
}
```

`budget_tokens` による上限指定はできない。思考量の制御は `output_config.effort`（§2 で詳解）に置き換わる。

Opus 4.6 / Sonnet 4.6 では `type: "enabled"` は deprecated だが **まだ動作する**。「移行期の互換性パス」として残っているが、将来のモデルリリースで削除予定。

### 1.2 サンプリングパラメータ廃止: `temperature` / `top_p` / `top_k`

Bedrock の Opus 4.7 モデルカードに明記されている:

> Starting with Claude Opus 4.7, `temperature`, `top_p`, and `top_k` parameters are no longer supported. The recommended migration path is to omit these parameters entirely from your requests and to use prompting to guide the model's behavior. If you previously relied on temperature = 0 for deterministic outputs, note that this setting did not guarantee identical responses across invocations.

Bedrock ではこれらを含むリクエストは拒否される。Anthropic API 直接呼び出しの詳細挙動は公式に明示されていないため、いずれにせよリクエストから除去するのが安全。

実務的な含意:

* `temperature = 0` で決定論的出力を期待していたコードは意図どおりに動かなくなる（そもそも過去モデルでも厳密な決定論は保証されていなかったが、慣習として使われていた）
* 制御はプロンプトエンジニアリングへシフトする
* `top_p = 0.95` のような微調整は不可

### 1.3 `thinking.display` デフォルト変更: `"summarized"` → `"omitted"`

過去 Claude 4 モデル系では `thinking.display` 未指定時のデフォルトは `"summarized"` で、API レスポンスの thinking block に要約済みの思考過程が入っていた。

Opus 4.7（および Claude Mythos Preview）は **`"omitted"` がデフォルト** に変わった。この場合、thinking block は以下のように返る:

```
{
  "type": "thinking",
  "thinking": "",
  "signature": "EosnCkYICxIMMb3LzNrMu..."
}
```

`thinking` フィールドは空文字列、`signature` のみ含まれる（multi-turn 継続時の reasoning 復元に使う）。UI で思考過程を表示している実装では、**明示的に `display: "summarized"` を指定** しないと「思考過程が見えなくなった」という現象が起きる。

```
{
  "thinking": {
    "type": "adaptive",
    "display": "summarized"
  }
}
```

課金は `"summarized"` / `"omitted"` どちらも thinking トークン全量に対して発生する（削減されるのはレイテンシであってコストではない）点に注意。

### 1.4 Interleaved Thinking 自動有効 / Thinking Block Preservation

Opus 4.7 では adaptive thinking が唯一の thinking モードで、**Interleaved Thinking（ツール呼び出し間での思考）が自動的に有効** になる。過去モデルで必要だった `interleaved-thinking-2025-05-14` beta header は不要（Opus 4.6 / Sonnet 4.6 ではまだ使えるが deprecated）。

加えて Opus 4.5+ / Sonnet 4.6+ から、過去ターンの thinking block は **デフォルトで保持される**（それ以前の Opus/Sonnet 系および Haiku 全モデルは削除）。Opus 4.7 もこの挙動を引き継ぐ。

利点:

* Multi-turn 会話で推論文脈が継続する
* Prompt caching との相性が良い（thinking block がキャッシュヒット対象）
* モデル性能への悪影響なし

注意点:

* 長い会話では thinking block が累積し、コンテキストウィンドウを消費する
* 過去の thinking block もインプットトークン課金対象

## 2. 新しい制御手段 — Adaptive Thinking と `output_config.effort`

### 2.1 対応モデルと基本形

Adaptive Thinking は Opus 4.6 / Sonnet 4.6 で推奨モードとして導入され、Opus 4.7 で **唯一の thinking モード** になった。モデル自身がタスクの複雑性に応じて思考深度を動的に決める。対応モデルは Claude Mythos Preview、Opus 4.7、Opus 4.6、Sonnet 4.6。

```
{
  "thinking": {
    "type": "adaptive"
  },
  "output_config": {
    "effort": "medium"
  }
}
```

### 2.2 `output_config.effort` の配置に注意

直感的に `thinking.effort` に置きたくなるが、**正しい配置はリクエスト root 直下の `output_config` 配下** である。`thinking` 配下やリクエスト root 直下の `effort` は 400 エラーになる（§3 で実機確認）。

### 2.3 値は 5 段階（soft guidance）

| Effort level | 挙動 |
| --- | --- |
| `max` | 制約なく深く思考する |
| `xhigh` | 深い探索を伴って常に思考する（**Opus 4.7 のみ**） |
| `high`（デフォルト） | 常に思考する。複雑なタスクで深い推論 |
| `medium` | 中程度。単純なクエリでは思考をスキップすることがある |
| `low` | 思考を最小化。単純なタスクはスキップ |

`effort` は hard limit ではなく **soft guidance** であり、モデルがタスクごとに実際の思考トークン数を調整する。`budget_tokens`（hard limit）と直接の 1 対 1 対応はない。コスト上限が必要な場合は `max_tokens`（thinking + 本文の合計上限）を併用する。

### 2.4 Opus 4.7 での推奨

公式 docs の Opus 4.7 向けガイダンス:

* **`xhigh`**: コーディング・エージェント用途の推奨スタート地点
* **`high`**（デフォルト）: 知能と token 効率のバランス点
* **`medium`**: コスト優先のドロップイン
* **`low`**: スコープが小さい短いタスク向け。複数セクションのタスクは明示的なチェックリストと併用
* **`max`**: フロンティア問題のみ。多くの workload ではコスト増に見合わず、構造化出力では overthinking の傾向も

`xhigh` / `max` 使用時は `max_tokens` を大きめ（公式推奨のスタートは 64k）にしておかないと `stop_reason: "max_tokens"` で途切れる可能性が上がる。

## 3. Bedrock 経由の実機検証

`output_config.effort` は Anthropic Messages API のフィールドだが、Bedrock 経由でどの呼び出しパスでも使えるのか、公式モデルカードには明示がない。実機で検証した結果、**3 ルートすべてで透過的に通る** ことを確認した。

検証環境:

* Model ID: `us.anthropic.claude-opus-4-7`（Geo Inference ID）
* Region: `us-west-2`（mantle のみ `us-east-1`）
* Libraries: `boto3 1.42.88`、`anthropic 0.98.1`
* 検証日: 2026/5

### 3.1 `bedrock-runtime` + `InvokeModel`

body 直下に `output_config` オブジェクトを入れる。

```
import json
import boto3

client = boto3.client("bedrock-runtime", region_name="us-west-2")
body = {
    "anthropic_version": "bedrock-2023-05-31",
    "messages": [{"role": "user", "content": "What is 2+2?"}],
    "max_tokens": 4096,
    "thinking": {"type": "adaptive"},
    "output_config": {"effort": "medium"},
}
resp = client.invoke_model(
    modelId="us.anthropic.claude-opus-4-7",
    body=json.dumps(body),
)
```

結果: ✅ 正常応答。`stop_reason: end_turn`、text block 正常。

### 3.2 `bedrock-runtime` + `Converse`

`additionalModelRequestFields` に Messages API 互換フィールドを載せる。

```
client.converse(
    modelId="us.anthropic.claude-opus-4-7",
    messages=[{"role": "user", "content": [{"text": "What is 2+2?"}]}],
    inferenceConfig={"maxTokens": 4096},
    additionalModelRequestFields={
        "thinking": {"type": "adaptive"},
        "output_config": {"effort": "medium"},
    },
)
```

結果: ✅ 正常応答。`stopReason: end_turn`。

### 3.3 `bedrock-mantle` + `AnthropicBedrockMantle` SDK

Anthropic API とまったく同じ書き方。

```
from anthropic import AnthropicBedrockMantle

client = AnthropicBedrockMantle(aws_region="us-east-1")
resp = client.messages.create(
    model="anthropic.claude-opus-4-7",
    max_tokens=4096,
    thinking={"type": "adaptive"},
    output_config={"effort": "medium"},
    messages=[{"role": "user", "content": "What is 2+2?"}],
)
```

結果: ✅ 正常応答。

### 3.4 配置ミス時の典型エラー（こちらも実機確認）

| 誤った配置 | 返るエラー |
| --- | --- |
| `body.effort` | `ValidationException: effort: Extra inputs are not permitted` |
| `body.thinking.effort` | `ValidationException: thinking.adaptive.effort: Extra inputs are not permitted` |
| `additionalModelRequestFields.effort`（Converse） | `ValidationException: effort: Extra inputs are not permitted` |

「`budget_tokens` がダメなら `thinking` 配下に `effort` を入れればいいだろう」とやると落ちる典型パターン。`thinking` 配下の `adaptive` に対しても拒否される。

### 3.5 エンドポイント選択の指針

| 要件 | 推奨エンドポイント |
| --- | --- |
| Guardrails / Knowledge Base / Agents / Flows 等の Bedrock 機能統合が必須 | `bedrock-runtime` |
| 規制業界のデータレジデンシー要件下で JP In-Region を使う | `bedrock-runtime` |
| Anthropic SDK の最新機能をそのまま使いたい | `bedrock-mantle` |
| 既存の `InvokeModel` / `Converse` コードベース | `bedrock-runtime`（移行コストゼロ） |

デフォルトクォータは `bedrock-runtime` 15M TPM、`bedrock-mantle` 10M TPM。

## 4. 移行チェックリスト

Opus 4.6 / Sonnet 4.6 から Opus 4.7 への移行時に、コードベースで機械的に拾える修正点。

### 4.1 thinking パラメータの書き換え

```
 {
-  "thinking": {
-    "type": "enabled",
-    "budget_tokens": 10000
-  }
+  "thinking": {
+    "type": "adaptive"
+  },
+  "output_config": {
+    "effort": "medium"
+  }
 }
```

`effort` は `thinking` 配下ではなく `output_config` 配下に置く（§2.2 / §3.4）。

### 4.2 サンプリングパラメータの除去

```
 {
-  "temperature": 0.7,
-  "top_p": 0.95,
-  "top_k": 40,
   "messages": [...]
 }
```

### 4.3 `display` の明示指定（UI に思考過程を出す場合のみ）

```
{
  "thinking": {
    "type": "adaptive",
    "display": "summarized"
  }
}
```

バックエンド推論のみの場合は省略でよい（Opus 4.7 デフォルト `"omitted"` のままでストリーミングの time-to-first-text-token が速くなる）。

```
 {
-  "anthropic-beta": "interleaved-thinking-2025-05-14",
   ...
 }
```

Opus 4.7 では adaptive thinking が唯一のモードで interleaved thinking は自動有効のため不要。

### 4.5 `budget_tokens` 依存の除去

思考量を細かくコントロールしていたコードは `effort`（`low` / `medium` / `high` / `xhigh` / `max`）へマッピング。soft guidance のため 1 対 1 対応にはならない。コスト上限が必要なら `max_tokens`（thinking + 本文合計）を併用。品質差が気になる場合は A/B テストで確認する。

## 5. 400 エラーの発生源とデバッグパス

### 5.1 Claude Code / Kiro CLI 経由なら大半は透過的

`anthropic` Python / TypeScript SDK の最新版は Opus 4.7 を認識し、`temperature` などのサンプリングパラメータを送信前にフィルタする動作が期待される。公式 SDK のみを使っている場合、アプリ側の修正なしで Opus 4.7 に切り替えられる。Claude Code（`claude-code` CLI）は内部で Anthropic SDK を使っているため、通常のユースケースでは透過的に Opus 4.7 を呼べる。

### 5.2 落とし穴が残っている層

一方、以下の層では 400 エラーが顕在化する:

* カスタム Bedrock クライアントで `temperature` を明示送信している
* MCP サーバーが旧い設定ファイルで `temperature` / `top_p` を固定値にしている
* LangChain / LlamaIndex のデフォルトサンプリング設定がそのまま流れ込んでいる
* 社内ライブラリが Bedrock を直接呼んでおり、Opus 4.6 の前提で書かれている
* 自作フレームワークが「未知のモデルには enabled thinking を暗黙付与」や「モデルごとの adaptive 対応 whitelist」を持っており、Opus 4.7 対応が漏れている

対応の基本:

1. リクエストビルダー / MCP 設定 / フレームワーク層すべてで `temperature` / `top_p` / `top_k` を除去
2. `thinking.type: "enabled"` / `budget_tokens` が残っていないか grep
3. Anthropic SDK / `anthropic[bedrock]` / `boto3` を最新版に更新
4. 自作フレームワークの adaptive thinking whitelist に Opus 4.7 を追加

### 5.3 デバッグパス

Opus 4.7 で 400 エラーが出た場合のチェック順序:

1. **リクエストボディをダンプして確認** — `temperature` / `top_p` / `top_k` / `thinking.type: "enabled"` / `budget_tokens` が含まれていないか。加えて `effort` が `output_config` 配下に正しく置かれているか
2. **フレームワーク層を追う** — どの層が sampling パラメータを注入しているか特定。LangChain / LlamaIndex などは `model_kwargs` や `llm_kwargs` で既定値が入ることがある
3. **MCP サーバー設定の確認** — `~/.mcp.json` / プロジェクト固有 mcp 設定の `env` / `args` に古い設定が残っていないか
4. **Bedrock エンドポイントを意識** — `InvokeModel` / `Converse` / `bedrock-mantle` で渡す場所が違う（§3）
5. **SDK バージョン** — Anthropic / AWS 双方を最新化

## まとめ

Opus 4.7 への移行で押さえる 5 点:

1. **`thinking.type: "enabled"` は 400 エラー** → `thinking.type: "adaptive"` + `output_config.effort` に書き換える
2. **`temperature` / `top_p` / `top_k` は廃止** → 全層から除去。挙動制御はプロンプトへ
3. **`display` デフォルトが `"omitted"`** → UI に思考過程を出すなら `"summarized"` を明示
4. **`output_config.effort` は Bedrock 経由（`InvokeModel` / `Converse` / `bedrock-mantle`）でも透過的に通る** → 配置は `thinking` 配下ではなく `output_config` 配下、誤配置は `ValidationException` で 400
5. **Claude Code / Kiro CLI は SDK 自動吸収** → 影響が出るのは MCP サーバー・カスタム Bedrock クライアント・フレームワーク経由

Opus 4.7 は 1M context、JP In-Region、JP CRIS、Adaptive Thinking、Interleaved Thinking 自動有効、`xhigh` effort など実務上のメリットが多い。API 変更は数としては小さく、チェックリストに沿って対応すれば透過的に切り替えられる。既存コードベースを抱える組織では、移行前に grep ベースでチェックしておくことで本番インシデントを予防できる。
