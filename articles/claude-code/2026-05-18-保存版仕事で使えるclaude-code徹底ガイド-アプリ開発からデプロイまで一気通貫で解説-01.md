「Claude Codeって便利らしいけど、結局どう仕事に組み込めばいいの?」 そんな疑問に答える記事です。本記事では、アメリカで現場のエンジニアが公開している実践レポートやワークフロー解説をもとに、**アイデアを形にして本番環境にデプロイするまでの全工程**を、すぐに業務で再現できるレベルでまとめました。

「コード生成ツール」として浅く使うのではなく、**ジュニアエンジニアをペアで動かす感覚**で運用するための具体的な手順とノウハウを、順番に見ていきます。

---

## なぜ今、Claude Codeなのか

まず押さえておきたいのは、Claude Codeが従来のコード補完ツールとは根本的に異なるという点です。GitHub Copilotのような「賢いオートコンプリート」を期待して使うと、その実力を引き出せません。

Claude Codeはファイルを読み、コードベース全体を横断して推論し、編集案を提示し、シェルコマンドを実行し、その結果を踏まえて自分の動きを調整します。これは単なる予測ではなく「実行」です。

つまり、プロンプトを投げて返答を待つ受け身のツールではなく、自律的にタスクをこなす**エージェント**として動きます。だからこそ、「何を作ってほしいか」を伝える設計と、「どう動かすか」のワークフロー設計が成果を大きく左右します。

ある現場では、4ヶ月にわたって本格運用した結果、生産性が約40%向上したという報告も出ています(参考記事は末尾)。一方で、無計画に使えば期待外れで終わるのも事実です。違いを生むのは「使い方の構造」です。

> Anthropicの社内テストでは、無計画なまま実行に移したタスクの成功率は約33%だった――という報告もあります。高アウトプットのエンジニアと他の人との差は、プロンプトの質ではなく、Claudeに実行させる前にどれだけ構造を作り込むかにあるとされています。

ここがすべての出発点です。

## 全体像 ― 仕事で使う5ステップ

実務でClaude Codeを使う流れは、おおまかに次の5段階に分かれます。

1. **環境構築とプロジェクトの文脈づくり**(CLAUDE.md)
2. **Plan Mode で計画を作り、レビューする**
3. **実装サイクル(TDDや小刻みなコミット)**
4. **GitHub Actions による自動レビュー・CI**
5. **Vercelなどへの自動デプロイ**

それぞれ順番に見ていきます。

## ステップ1:環境構築と CLAUDE.md ― 「Claudeの脳」を作る

### インストール

Claude Codeはターミナルで動くCLIツールです。Node.jsが入った環境でnpmからインストールし、プロジェクトのルートディレクトリでclaudeコマンドを起動します。詳細な手順や前提条件は公式ドキュメント(https://docs.claude.com/en/docs/claude-code/overview)を参照してください。

### CLAUDE.md がすべての出発点

仕事で使うなら**真っ先に作るべきファイル**が CLAUDE.md です。これはプロジェクトのルートに置くマークダウンファイルで、Claudeが起動時に自動で読み込みます。

CLAUDE.mdはAIにとってプロジェクト固有の「脳」のようなものです。コーディング規約、重要なファイルの場所、テスト方法、よく使うコマンドなど、リポジトリの本質的なコンテキストを書き留めておくシンプルなマークダウンファイルです。このファイルがClaudeにベースラインの理解を与え、提案や動作がチームの慣習と一致するようにします。

仕事で使うCLAUDE.mdの例(抜粋):

```
# プロジェクト概要
- Next.js 14 + TypeScript + Tailwind CSSのWebアプリ
- ユーザー認証はNextAuthを使用

# コーディング規約
- 関数コンポーネントのみ使用、クラスコンポーネント禁止
- 状態管理は Zustand
- 型は any 禁止、unknown を使う
- インポートは絶対パス(@/components/...)

# テスト
- 単体テストは Vitest、E2E は Playwright
- カバレッジ80%以上を維持

# ディレクトリ構成
- src/app : ルーティング
- src/components : 共通UI
- src/lib : ユーティリティとAPIクライアント

# 重要事項
- mainブランチへの直push禁止、必ずPR経由
- シークレットは .env.local のみ、コミット禁止
- できる限りシンプルな実装を選ぶこと(過剰な抽象化を避ける)
```

ポイントは「**Claudeを過剰に賢くしない**」こと。Claudeは指示しない限り、余計な抽象化や時期尚早なリファクタリングを加えがちです。「最もシンプルなアプローチを使ってください」という一文をCLAUDE.mdに入れるだけで大きく改善します。

### MCPサーバーで文脈を広げる

GitHub・Linear・Sentry などをMCP(Model Context Protocol)経由で接続すると、Claudeが直接Issueを読んだりエラーログを参照したりできます。あるアメリカのソロ開発者は、Linear MCP Serverをワークフローの中核に据え、Claudeから直接Issueを作成したり実装タスクを委譲したりしている、Sentry MCP もエラートラッキング用に組み合わせていると紹介しています。

## ステップ2:Plan Mode ― 「コードを書く前に必ず計画する」

仕事で使う上で**最大の鬼門**がここです。Claudeにいきなり実装させると、20個の判断ポイントで80%の正答率しか出ないと仮定すると、すべて当てる確率は 0.8の20乗 ≒ 1% にまで落ちます。

Plan Mode はこの問題を解決します。20個の曖昧な判断を、レビュー済みの仕様書として1つにまとめてしまえば、各判断は事実上100%確定するからです(あなたがすでに決めたから)。

### 実際の手順

ターミナルで Shift+Tab を2回押すと Plan Mode に入ります。この状態ではClaudeはファイルを読むだけで一切編集しません。

実際にプロダクションで使われている定番の流れがこちらです:

> Claudeに「実装はまだしないで、計画だけ作って」と依頼する 計画をエディタで開き、Claudeが間違えた箇所に注釈を入れる 「すべての注釈に対応してほしい。実装はまだしないで」と送り返す すべての判断が解決するまで繰り返す ようやく実装を依頼する

「**実装はまだしないで**(don't implement yet)」というガード文がポイント。これを抜くと、Claudeは修正をスキップしていきなりコードを書き始めます。

### 効果の実例

あるエンジニアは、12ステップの仕様書に2時間かけた結果、実装時間を6〜10時間節約できたと報告しています。大きな機能になればなるほど、この事前投資のリターンは大きくなります。

これは「急がば回れ」の典型例で、業務で品質を担保するなら絶対に省略すべきではない工程です。

## ステップ3:実装サイクル ― TDDとサブエージェントを活用する

計画が固まったら、いよいよ実装に入ります。仕事で使うときの肝は3つです。

### 1. テスト駆動開発(TDD)を回す

あるエンジニアは、Linear IssueからPRまでを自動化するスキルを作り、その中でテスト駆動開発、並列コードレビュー、体系的なバリデーションによる品質保証を、Issue分析からPR作成までのライフサイクル全体で回す運用を確立しています。

「テストを書く → 実装 → テストを通す」のループを Claude にやらせると、デグレに気づきやすくなり、品質が安定します。

### 2. サブエージェントでコンテキストを節約する

巨大なコードベースを調査させると、Claudeのコンテキスト(200Kトークン)がすぐ埋まります。200Kトークンの上限に達するとClaudeは自動で要約して圧縮しますが、その際に重要な詳細が失われやすく、タスクが忘れられたり性能が低下したりすることがあるのが現実です。

対策はシンプルで、**重い探索はサブエージェントに任せる**こと。サブエージェントは独立したコンテキストで動き、結果だけを要約して親に返してくれます。

```
サブエージェントを使って、認証システムがトークンリフレッシュをどう扱っているか調べて
```

これだけで、メインの会話を汚さずに調査結果だけを受け取れます。

### 3. 「勝利宣言」に騙されない

Claudeはタスクが完了したと**先に宣言してしまう癖**があります。Claudeは時折、タスクの一部しか対応していないのに完了と宣言します。受け入れる前に、自分でテストを走らせてください。

業務で使うなら、必ず**人間がローカルでテスト実行とビルドを確認**するルールにしましょう。

### 4. README と CHANGELOG を最初に作る

セッションをまたぐと、Claudeは前回の文脈を忘れます。最初にClaudeにREADMEを作らせて、それを更新し続けることを推奨します。加えて、各ステップで何が変わったかをまとめるchangelogを維持します。この2つのファイルは人間のためだけではなく、Claudeが後から文脈を取り戻すのにも役立ちます。これがないと、モデルは何が起きているかを理解するためにコードベース全体を探索する必要があり、時間とコンテキスト予算を浪費します。

## ステップ4:GitHub Actions で CI とコードレビューを自動化

実装が動くようになったら、次は CI/CD です。Claude Code は GitHub Actions と直接統合できます。

### 導入手順(最速)

ターミナルでClaude Codeを開いて、/install-github-app を実行するだけ。Installボタンをクリックし、リポジトリを選び、指示に従うと自動的にPRが作成されます。マージすると .github/workflows フォルダが新しく作られます。

### 何ができるようになるか

ある記事ではこの統合を次のように説明しています:Claude と GitHub Actions の統合は別のパラダイムを表しています。これは開発者を行単位でアシストするものではなく、ワークフローのまるごと一段階を自動化するものです。GitHub Copilotが「IDE内のペアプログラマー」だとすれば、Claude + GitHub Actions は「CI/CDパイプライン内で自律的に動作し、PRの送信後に分析・要約・レビュー・修正までこなすエージェント」です。

### ワークフローを書かせる際のコツ

CI設定ファイルは YAML の細かい構文や依存関係でハマりがちですが、CLAUDE.mdに**自分のプロジェクトのCI/CD規約**を書いておけば、Claudeが一発で生成してくれます。

CI/CD規約の例:

* mainブランチへの直接push禁止、必ずPR経由
* すべてのPRはCI通過必須(ci-required ステータスチェック)
* ブランチフロー:feature/\* → develop → main

パイプライン段階(順序厳守):

1. lint ― ESLint + Prettier チェック
2. test ― Jest(最低80%のラインカバレッジ必須)
3. build ― tsc + vite build
4. deploy ― mainマージ時のみ、本番へBlue-Green

このルールをCLAUDE.mdに書いて、「Node.js + TypeScript の GitHub Actions ワークフローを作って」と依頼すれば、汎用テンプレートではなく**プロジェクト固有の制約を反映したパイプライン**が生成されます。

### 重要な権限設定

ワークフロー内でPR作成や変更push を許可するには、YAMLのpermissionsスコープに pull-requests: write や contents: write を明示的に追加する必要があります。Claude Code はセキュリティのために、デフォルトでは読み取り専用かつ非破壊的なツールセットで動作します。

権限は**必要最小限**で。これも仕事で使う上での鉄則です。

## ステップ5:Vercel への自動デプロイ

ここまで来れば、本番投入は目前です。Next.js などのフロントエンドアプリなら、Vercel との組み合わせが最速ルートです。

### 流れ

1. Claude Code でローカルにプロジェクトを生成
2. git init → GitHubリポジトリにpush
3. Vercel ダッシュボードで「Import Project」 → GitHubリポジトリを選択
4. 環境変数を Vercel 側で設定
5. デプロイ完了 ― 公開URLが発行される

GitHubと接続が済めば、CI/CDは自動です。pushするたびに自動でデプロイされ、手動の手順は一切不要です。プレビューデプロイは無料で即時生成されます。マージ前にClaudeと新機能を試したいときに使いましょう。環境変数はリポジトリではなく、Vercelのダッシュボードに置きます。最初のデプロイ前にそこで設定してください。

### バックエンドが必要な場合の注意

純粋なフロントエンドサイトはこの流れで簡単にデプロイできますが、バックエンドを伴う機能には Claude Design ワークフローには現実的な限界があります。フロントエンドのみの境界内にプロジェクトが収まるなら、このガイドの手順で効率的にライブURLまで辿り着けます。一度通せば、2回目以降はデザインからデプロイまでの距離がぐっと短くなります。

データベースや認証、サーバーレス関数を絡めるなら、Supabase、PlanetScale、Vercel Postgres などをCLAUDE.mdに明記し、Claudeに統合させる流れがスムーズです。

## 仕事で使うときに必ず守りたい5つの原則

ここまでの流れを踏まえて、現場の知見から導き出された「やってはいけないこと」をまとめます。

### 1. Claudeを「魔法のコード生成器」として扱わない

Claude Code は魔法のコード生成器として扱うとうまく動きません。「速く書けるけれども、構造と監督と明確な境界を必要とするジュニアエンジニア」として扱うと動きます。これを内面化した瞬間に、ワークフロー全体が変わります。

### 2. 「成功基準が不明確なタスク」は任せない

「完了」がどう見えるかをモデルが検証できる形で記述できないなら、ワークフローは限界に達するか、あなたが止めるまで反復し続けます。セキュリティや決済に触れるコードパスは、誰が(あるいは何が)変更を生成したかに関わらず、人間のレビューが必要です。

### 3. コンテキスト圧縮の前に /clear する

200Kトークン上限に近づいたら、自動圧縮を待たずに一度クリアして仕切り直す。これだけで品質が安定します。

### 4. 1ファイル50〜100行ルール

CLAUDE.mdやスキルファイルは長く書きすぎない。処方箋的なプロンプトエンジニアリングのルールには従わないこと。Claudeは進化するからです。代わりに原則に従いましょう:ドキュメントは50〜100行に保つ、最低限から始める、実際のタスクでテストする、結果に応じて反復する。

### 5. セッションが迷走したらリセットする

セッションが横道にそれたら、Anthropic自身のアドバイスはこうです:状態を保存して、Claudeに新鮮なセッションで再挑戦させて、結果を受け入れるか最初からやり直す。劣化したセッションで修正をめぐって格闘するほうが、リセットするよりずっと時間を浪費します。

## まとめ ― Claude Code は「実行システム」である

ここまで読んでいただければ、Claude Codeが**単なるコード補助ツールではなく、自律的に実行するエージェントシステム**だと理解できたはずです。

オートコンプリートからエージェント実行へのシフトは、すでに進行中です。Claude Codeはより賢い提案エンジンではなく、読み・行動し・観察し・調整する実行システムです。この区別が「信頼性」の意味を変えます。きちんと運用された Claude Code ワークフローは、機械的なリファクタに費やす時間の削減、より速いテストカバレッジ、そして誰かのバックログで止まっていた移行タスクの完了――こうした現実の成果をもたらします。

仕事で本気で使うなら、今日からやるべきことはシンプルです:

1. **CLAUDE.md を作る**(プロジェクトの脳になる)
2. **Plan Mode を必ず使う**(「実装はまだしないで」を口癖に)
3. **GitHub Actions と連携する**(/install-github-app 一発)
4. **Vercelなどに自動デプロイ**(pushしたら本番)
5. **人間がレビューする最後の一線は譲らない**

この5つを守れば、Claude Codeは「面白いガジェット」から「日々の生産性を底上げする業務インフラ」に変わります。
