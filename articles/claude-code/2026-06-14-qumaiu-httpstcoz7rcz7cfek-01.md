## **「何を作ったか」ではなく「なぜ作ったか」が記録される世界に。**

2月、元GitHub CEOのThomas Dohmkeが率いるEntireHQが、$60Mのシードラウンドとともに、**エージェントのコンテキストをGitコミットに紐づけて保存する**「Checkpoints」をOSSで公開しました。

そして先日6月11日、コードエディタZedが、**AIへの指示まで履歴として保存する新バージョン管理システム「DeltaDB」**を発表。

![](https://pbs.twimg.com/media/HKxTrVaasAAMGOF.jpg)

海外では「Intent Engineering」という言葉が急速に広がり始めています。しかし、日本ではまだこの概念を体系的に紹介した記事はほとんど見当たりません。

上記は別々のタイミング、別々のプロダクトです。しかし、向かっている方向は同じです。**「何を変えたか（コード差分）」だけでなく「なぜ変えたか（意図）」を記録し、永続化する。**

これは、僕がずっと言い続けてきた「コンテキストの質が全てを決める」という話のインフラ層が動き始めたということです。

## Prompt → Context → Intent：進化の3段階

AI駆動開発の方法論は、この2年で急速に進化してきました。

**第1段階：プロンプトエンジニアリング**。
AIへの指示の書き方を最適化すること。「こう聞けばいい答えが返ってくる」という単発のやり取りの改善です。

**第2段階：コンテキストエンジニアリング。
**AIに渡す情報そのものを最適化すること。CLAUDE.mdやADR、用語集を整備し、AIが参照するコンテキストの質を上げる。僕が「コンテキストもウォーターフォールで進む」と表現した領域です。

**第3段階：インテントエンジニアリング。
**AIに「何を達成すべきか」を構造化して伝えること。目的、成功基準、制約条件、中止条件まで含めた「意図」を設計する。

プロンプトエンジニアリングが「言い方」の最適化、コンテキストエンジニアリングが「情報」の最適化だとすれば、**インテントエンジニアリングは「目的」の最適化**です。

そして今、DeltaDBやEntireHQが取り組んでいるのは、この第3段階を支えるインフラの構築です。

## 予兆はすでにあった

実は、この流れの予兆はすでに多くの開発者の目の前にありました。

2025年にGitHubがリリースした「Spec Kit」。コード生成の前に仕様（Spec）を定義し、それを起点にAIエージェントを動かすワークフローです。AWSがKiro IDEに統合したSpec-firstワークフロー。JetBrainsやCursorが導入したプランニングモード。

これらは当時、「AIの出力を安定させるためのベストプラクティス」として受け止められていました。しかし、今振り返ると、**これらはすべてインテントエンジニアリングの予兆だった**と言えます。

「コードを書く前に、意図を構造化するSpec KitもKiroも、本質的にはこのメッセージを発していたのです。ただ、当時はそれを「インテントエンジニアリング」という枠組みで捉えている人はほとんどいませんでした。

## そして今、インフラ層が動き始めた

Spec KitやKiroが「意図を先に定義する」というプラクティスを広めた段階だとすれば、DeltaDBとEntireHQが取り組んでいるのは**その次のレイヤー**です。

定義した意図を、揮発させず、永続化し、追跡可能にする。コードと意図を一体として管理するインフラを構築する。

DeltaDBは、Zedが発表した新しいバージョン管理システムです。

従来のGitはコミット単位で「ファイルの状態」を記録します。しかし、AIエージェントとの開発では「エラー処理を追加して」「テストも直して」と短い指示と編集が何度も繰り返されます。**最終的な差分だけ見ても、「どの指示でどの変更が生まれたのか」がわからなくなります**。

DeltaDBは、作業中のファイルと開発中の会話をまとめて扱います。変更の差分を「デルタ」として連続的に記録し、AIエージェントが特定の行を書き換えた直後の状態も、さらに修正が加わった後の状態も、後から参照できるようにします。

つまり、**「最新版のコード」だけでなく「エージェントが最初に書いた時点のコード」まで追跡が可能**になるのです。
