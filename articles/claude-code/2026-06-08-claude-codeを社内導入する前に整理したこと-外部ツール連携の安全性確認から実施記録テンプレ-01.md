## はじめに

エンジニア3名のチームでClaude Codeを社内導入するにあたり、
**外部ツール（GitHub・Slack・Box・Figma・Backlog）との連携が安全にできるか**を調査・整理した記録です。

「とりあえず使ってみる」前に、**どのツールが安全に連携できるか・何を守るべきか**を先に整理しておく必要があると判断し、この一連の作業を行いました。同じように社内導入を検討しているチームの参考になれば幸いです。

---

## 前提：MCPとは何か

外部ツールとの連携は **MCP（Model Context Protocol）** という仕組みを使います。

```
Claude Code ←→ MCPサーバー ←→ GitHub / Slack / Backlog / Box / Figma など
```

重要なのは**「誰が運営しているMCPサーバーか」**で、安全性が大きく変わります。

| 種類 | 内容 | 安全性 |
|------|------|--------|
| ツール公式MCP | そのツールの開発元が作成 | 高い |
| Anthropic公式MCP | Anthropicが作成・維持管理 | 高い |
| サードパーティMCP | 第三者企業が作成（例：Composio） | 企業による |
| コミュニティMCP | 個人開発者がGitHubで公開 | 不明確 |
| 自社構築MCP | 公式APIを使って自社で作成 | 最も高い |

データが「第三者サーバーを経由するか否か」が最大のリスク判断基準になります。

## 各ツールの連携調査結果

### GitHub ─ ✅ OK

Anthropic公式サポートがあり、接続範囲は**セッションを開始したリポジトリのみ**に限定されます。

**必須の安全対策：**
- 顧客情報が含まれるリポジトリは接続しない
- 接続リポジトリは事前にリーダーへ確認
- `.env`ファイルがリポジトリに含まれていないことを確認

> 参考：[Claude Code and new admin controls for business plans（Anthropic公式）](https://www.anthropic.com/news/claude-code-on-team-and-enterprise)
> 該当箇所：「MCP server configurations」の管理設定に関する記述

---

### Slack ─ ⚠️ 条件付きOK

**Slack社（Salesforce）が直接提供する公式MCPサーバー**（`mcp.slack.com`）があります。

注意点として、過去にAnthropicのリファレンスSlack MCPサーバーに **CVE-2025-34072（CVSS 9.3 Critical）** というリンク展開経由のデータ漏洩脆弱性がありました。このサーバーは**パッチなしでアーカイブ**されています（修正は行われていない）。現在使用するSlack公式MCPは別サーバーのため直接の影響はありませんが、**必ずSlack公式（`mcp.slack.com`）のみを使用する**ことが重要です。

**必須の安全対策：**
- Slack公式MCPサーバーのみ使用
- 顧客情報を含むチャンネルにClaudeを招待しない
- Bot権限を読み取り専用に最小化する
- DMにBotを招待しない

---

### Box ─ ✅ 条件付きOK

**Box社が直接提供する公式MCPサーバー**があります（2025年8月にGA正式リリース）。
既存のBoxセキュリティポリシー・アクセス権限がそのままAIにも適用されるため、
**既に権限管理が整っているチームは導入しやすい**です。

**必須の安全対策：**
- 顧客情報を含むフォルダへのアクセス権限を最小化する
- Claudeにアクセスさせるフォルダの範囲を事前に定義する

---

### Figma ─ ✅ OK

Figmaは**Claude Codeの公式パートナー**です。Anthropic公式プラグインとして提供されており、インストールが簡単です。

Remote版（一般ユーザー推奨）とDesktop版（Enterprise向け）の2種類があり、2026年2月以降は**コードからFigmaキャンバスへの双方向連携**にも対応しました。

```bash
# インストールコマンド
claude plugin install figma@claude-plugins-official
```

---

### Backlog ─ ✅ 条件付きOK（Claude Code CLIに限る）

**Backlog開発元であるNulab社が公式MCPサーバーを提供**しています（`nulab/backlog-mcp-server`）。

データの流れが以下のようにローカル完結なのが安全性の根拠です：

```
Claude Code（ローカル）
    ↕ stdio（ローカル通信・第三者経由なし）
backlog-mcp-server（ローカルプロセス）
    ↕ HTTPS
Backlog API（Nulab社サーバー）
```

ただし**stdioトランスポートのみ対応**のため、ブラウザ版claude.aiやモバイルからは接続できません。

| 接続方法 | 可否 |
|---------|------|
| Claude Code CLI | ✅ 可能 |
| Claude Desktop | ✅ 可能 |
| ブラウザ版 claude.ai | ❌ 技術的に不可 |
| Claude Mobile | ❌ 技術的に不可 |

**設定コマンド：**

```bash
claude mcp add --scope user backlog \
  -e BACKLOG_DOMAIN=yourspace.backlog.jp \
  -e BACKLOG_API_KEY=YOUR_API_KEY \
  -- npx -y backlog-mcp-server
```

⚠️ APIキーは設定ファイルに平文で保存されます。Gitにコミットしたり共有ストレージに保存しないこと。

> 参考：[BacklogのMCPサーバーを公開しました（Nulab公式ブログ）](https://nulab.com/ja/blog/backlog/released-backlog-mcp-server/)
> 該当箇所：「このたびGitHub上にて、BacklogのMCP（Model Context Protocol）サーバーの実装を公開しました」

---

### Gmail・Google Calendar ─ ⛔ 現時点では禁止

Anthropic公式のGmail / Google Calendar MCPサーバーは**存在しません**。サードパーティMCP経由では、メール・カレンダーデータが第三者サーバーを経由するリスクがあります。

弊チームでは**Google公式APIを使った自社MCPサーバーの構築**を選択しました（エンジニア工数：約1〜2日）。自社構築完了まではサードパーティMCP経由での連携を禁止しています。

## ProプランからTeamプランへ移行する場合

弊チームはProプランで使用中ですが、Teamプランへの移行を検討しています。
外部ツール連携の観点での差分を整理しました。

### MCP接続設定はそのまま引き継がれる

MCP接続設定ファイル（`~/.claude.json` / `.mcp.json`）は**ローカルPCに保存されるファイル**のため、プラン移行後もそのまま引き継がれます。設定のやり直しは不要です。

> 参考：[Claude Code の設定（Claude Code公式ドキュメント）](https://code.claude.com/docs/ja/settings)
> 該当箇所：「MCP servers：ユーザースコープは `~/.claude.json`、プロジェクトスコープは `.mcp.json`」

ただし、移行後は念のため `/mcp` コマンドで接続状態を確認することを推奨します。

### データ保護のレベルが向上する

最も重要な変化はここです。

| | Proプラン | Teamプラン |
|--|----------|-----------|
| 学習への利用 | 個人設定でOFFにしないと学習に使われ得る | **契約上、自動でOFF（学習に使われない）** |
| MCPで取得したデータ | 契約上の保護保証なし | **契約上の保護が保証される** |

> 参考：[Updates to Consumer Terms and Privacy Policy（Anthropic公式）](https://www.anthropic.com/news/updates-to-our-consumer-terms)
> 該当箇所：「These updates do not apply to services under our Commercial Terms, including: Claude for Work, which includes our Team and Enterprise plans」

外部ツールを連携してデータを扱う場合、Proプランは「個人の設定に依存」する状態ですが、Teamプランは「契約上の保証」になる点が決定的な違いです。

## 実施記録テンプレートを用意した理由

調査だけで終わると「どのツールをいつ誰が試して、何を確認したか」が残りません。そこで各ツールごとに以下の記録テンプレートを用意しました。

```
実施日：
担当者：
接続したプロジェクト・リポジトリ：
確認した機能：
動作確認結果（OK / NG）：
発生した問題：
業務上のメリット（実感したもの）：
```

これをBacklogドキュメントに置いておくことで、チームメンバーが試したときに記録が蓄積されていく形にしています。

## 業務上で注意すべきリスク（技術設定だけでは守れない部分）

技術的な設定だけでなく、業務・法的観点でのリスクも整理しておく必要がありました。

### 個人情報保護法

クライアント顧客の個人情報をAIに入力することは、日本の個人情報保護法上「第三者提供」に該当する可能性があります。また、Anthropicは米国企業のため「国外移転」にも該当し得ます。

### NDA（秘密保持契約）

締結済みのNDAに「AIツールへの入力を禁止する」条項が含まれているケースがあります。明記がなくても「第三者に提供しない」という条項でAI利用が違反と解釈される可能性があります。

### 対策

- 顧客情報はダミーデータに置き換えてからプロンプトに入力する
- クライアントへのAIツール使用の開示・同意取得の方針を決める
- 情報漏洩が発生した場合の対応フローと責任者を事前に決める

| ツール | 判断 | ポイント |
|--------|------|---------|
| GitHub | ✅ OK | セッション単位でのアクセス範囲限定 |

| Figma | ✅ OK | Claude Code公式パートナー |

「安全性確認なしにいきなり連携」は情報漏洩リスクがあります。ツールごとに**誰が運営するMCPか・どのデータが経由するか・契約上の保証があるか**を確認することが出発点です。
