## 🚀 はじめに

2026年6月にAddy Osmani氏が提唱したループエンジニアリング（Loop Engineering）は、「人間がプロンプトを打つ」のをやめ、AIエージェントが自走するループそのものを設計する手法として急速に広まりました。

一方で、実際にループを回し始めたチームから聞こえてくるのは成功談ばかりではありません。**ループは設計者の判断を増幅する装置です。良い判断も増幅しますが、判断の欠陥も等倍で増幅します。** 本記事では、実際に報告されている失敗パターンを6つに整理し、それぞれの対策をBad/Good形式で紹介します。

対象読者は、Claude CodeやCodex等のコーディングエージェントでループを組み始めた（組もうとしている）エンジニアです。

## 🔄 前提：ループの基本構造

失敗例に入る前に、ループの最小構成を確認します。

```
起動（トリガー）
  → タスク読込（外部メモリ）
  → 実装（Worktree上で隔離実行）
  → 検証（Verifier = 独立したサブエージェント）
  → 状態保存（外部メモリへ書き戻し）
  → 次周へ / 停止判定
```

以降の失敗例は、ほぼすべてこの構造のどこかの欠落・手抜きに起因します。

## 🎭 失敗例1：Verifier Theater —— 検証しているフリ

**症状：** Verifierが承認したのにCIが落ちる。「LGTM」を量産するだけの検証エージェントが完成する。

最も多く、最も根が深い失敗です。原因はVerifierのプロンプトが曖昧なこと、そしてテストやlintの実行を省略して「コードを読んだ感想」だけで判定させていることです。

LLMには出力を肯定しやすいバイアスがあるため、「問題ないか確認して」という指示は実質的に「承認して」と同義になります。

```
# Bad: 曖昧な検証指示
verifier:
  prompt: "この変更に問題がないか確認してください"
```

```
# Good: 拒否理由を探すフレーミング + 機械的検証の必須化
verifier:
  prompt: |
    あなたの役割はこの変更を「拒否する理由」を探すことです。
    以下をすべて実行し、1つでも失敗したら拒否してください。
  required_inputs:
    - test_output      # テスト実行結果を必須入力に
    - lint_output
  model: "implementer と異なるモデルファミリー"  # 自己採点バイアスの排除
```

実装者と検証者を同一モデル（同一コンテキスト）にするのは論外です。自分の答案を自分で採点させれば、採点基準の方が答案に寄っていきます。

## ♾️ 失敗例2：Infinite Fix Loop —— 収束しない自動修正

**症状：** 同一のPRやチケットに対して自動修正が5回、10回と適用され続け、朝起きたらトークンだけ消えている。

原因はリトライ上限（attempt cap）の未設定です。Verifierの誤承認や誤った根本原因診断が修正を再トリガーし続け、同じ場所を掘り続けます。

```
# Bad: 停止条件がない
loop:
  trigger: ci_failure
  action: auto_fix

# Good: attempt cap + エスカレーション + kill switch
loop:
  trigger: ci_failure
  action: auto_fix
  max_attempts: 3                    # 上限到達で人間のインボックスへ
  escalate_to: human_inbox
  kill_on_consecutive_failures: 2    # 同一エラー2連続で即停止
```

重要なのは、**「進捗がない」ことの検出**です。単なる回数制限ではなく、「同じエラーを繰り返している＝ループが停滞している」を検知して止める設計が必要です。

## 💸 失敗例3：コスト爆発 —— 24時間稼働の請求書

**症状：** 月末にAPI利用料の請求を見て青ざめる。

ループは失敗しても課金されます。無限リトライは無限請求です。典型的な誘発要因は3つあります。予算上限の未設定、サブ分単位の過剰なcadence（実行間隔）、そして「処理すべきアイテムがなくても毎回フルパスを回す」構造です。

```
# Good: 予算とcadenceの設計例
budget:
  token_daily_limit: 2_000_000
  pause_at_pct: 80          # 80%到達で自動一時停止

cadence: "*/10 * * * *"     # 分単位ではなく最低5〜10分間隔

pipeline:
  - triage:                  # 安価なモデルで仕分けだけ実行
      model: cheap
      early_exit_if_empty: true   # アイテムゼロなら即終了（最大の節約点）
  - action:                  # 実処理は必要な時だけ強いモデルで
      model: strong
```

「安いモデルでtriage → 必要な時だけ強いモデルでaction」の2段構造と、「空なら即終了」の早期リターンだけで、コストは桁が変わります。

## 🗑️ 失敗例4：State Rot —— 腐っていく外部メモリ

**症状：** ループの状態ファイル（`STATE.md`等）にクローズ済みタスクが増殖し、エージェントが古い情報を読んで誤動作する。

ループは外部メモリ（MarkdownファイルやLinearチケット）に状態を書き出して「記憶」を継続します。ここにpruning（剪定）ステップがないと、状態ファイルは書き込まれる一方で肥大化し、やがてコンテキストを圧迫してエージェントの判断品質を劣化させます。複数のループが同一ファイルに書き込む構成では、競合でさらに加速します。

対策は2つです。**実行ごとにクローズ済みアイテムを削除するステップをループ自体に組み込む**こと、そして**ループごとに状態ファイルを分離する**ことです。ステートフルなシステムにGC（ガベージコレクション）が必要なのは、AIループでも変わりません。

## 📉 失敗例5：Comprehension Debt Spiral —— 誰も読んでいないコードの山

**症状：** ある日、チームの誰も変更の意図を説明できないコードが本番で動いていることに気づく。

ループが速く成果を出すほど、「人間が読んでいないコード」が積み上がります。auto-mergeの範囲を少しずつ広げ、週次レビューを「今週は忙しいから」と省略する。この2つが揃うと、理解負債は複利で増えます。

技術的負債はリファクタリングで返せますが、理解負債は**障害が起きた瞬間に一括返済を迫られます**。誰もデバッグできないシステムの障害対応ほど高くつくものはありません。

対策は運用ルールとして強制することです。週次ダイジェスト（ループが何をしたかの要約）の生成と読み合わせを義務化し、リスクが中程度以上の変更はauto-mergeから人手ゲートに戻します。「読む時間も含めてループの運用コスト」と見積もるべきです。

## 🚫 失敗例6：検証不能タスクのループ化 —— そもそも回してはいけないもの

**症状：** 「UIをもっと使いやすく改善し続けて」のようなループが、もっともらしいが誰も望んでいない変更を量産する。

これは設計ミスというより**タスク選定のミス**です。ループが機能するのは、成果を機械的に検証できるタスクに限られます。導入前に次の4条件でフィルタすべきです。

1. **反復性：** そのタスクは繰り返し発生するか
2. **低検証コスト：** 機械的なテスト・判定が可能か
3. **狭い影響範囲：** 失敗しても致命傷にならないか
4. **24時間稼働の価値：** APIコストを払う価値があるか

「セキュアであること」「使いやすいこと」のような検証器を書きにくい品質特性は、ループの外側で人間がレビューするしかありません。テストが通ることと仕様を満たすことは同値ではない（Goodhartの法則）、という当たり前の事実が、ループでは増幅されて返ってきます。

## 🛑 まとめ：停止条件をFirst-Classに設計する

6つの失敗例を貫く共通原則は1つです。**ループを作るより先に、ループをどう止めるかを設計する。**

* Verifierには「拒否する理由を探させ」、実装者と分離する
* attempt capと「進捗なし検出」で収束を保証する
* 予算上限・triage 2段構造・早期終了でコストを抑える
* 外部メモリにはGCを組み込む
* 週次レビューを省略しない。理解負債は障害時に一括請求される
* 検証不能なタスクはループに入れない

ループエンジニアリングは「シルクハットを被ったCronジョブ」と揶揄されることもありますが、cronと決定的に違うのは、中で動くのが非決定的なエージェントだという点です。非決定的なコンポーネントを無監視で回すことのリスクは、インフラ運用をやってきた人ほど直感的に理解できるはずです。

停止条件なしのループを本番で回してはならない。これを不変のルールとして、安全にレバレッジを効かせていきましょう。
