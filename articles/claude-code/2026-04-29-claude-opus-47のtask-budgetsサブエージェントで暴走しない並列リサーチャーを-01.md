## 概要

2026年4月16日に出た **Claude Opus 4.7** 、エージェント開発者にとってはいくつか地味やけど重要な新機能が入っとる。中でも **「Task Budgets」** っちゅう、 **エージェントループ全体の トークン消費目安をモデルに伝える** 機能が個人的に大ヒットや。

「サブエージェントが暴走してトークン爆食いした……」っちゅう悲しい事故、誰でも一度はやったことあるやろ？今回は **Task Budgets × サブエージェント並列実行** で **「暴走しないリサーチエージェント」** を作る方法を、実装付きで紹介するで。

---

## 背景・課題：なぜ「Task Budgets」が必要やったか

### サブエージェントの「コスト見積もりムズい問題」

Claude Agent SDKでサブエージェントを使うと、こんな構成になる。

```
親エージェント（オーケストレーター）
  ├── サブエージェントA（リサーチ）
  ├── サブエージェントB（コード解析）
  └── サブエージェントC（要約）
```

サブエージェントは **親と独立した context window** を持つから、 **親のコンテキストを汚さんと並列でガンガン仕事させられる** っちゅう超強力な仕組みや。

ただし問題は **コスト見積もり** や。たとえばリサーチサブエージェントに「調べて」って投げたら、 **何回ツール呼び出すか、何トークン使うか** が事前にわからん。「ちょっと調査」が **数十万トークン** になることもザラやった。

### 従来の対策とその限界

これまでの対策はだいたいこの3つやった。

* **max\_turns で ターン数を制限** → ターン浅すぎると未完了で終わる
* **max\_thinking\_tokens で 思考量を絞る** → 思考だけ絞ってもツール呼び出しは止まらん
* **プロンプトで「短くまとめて」とお願い** → ガン無視されることが多い（笑）

要するに **「全体の総量」を直接コントロールする手段がなかった** んや。

### Task Budgetsとは

Opus 4.7で導入された **Task Budgets** は、 **「このエージェントループ全体で だいたいこのくらいのトークンを目安にしてや」** ってモデルに伝える機能や。

公式ドキュメントによると、この目安は **思考・ツール呼び出し・ツール結果・最終出力** を全部ひっくるめた値で、 **モデル側が自律的に予算配分を考えて動く** ようになる。

ハードリミットやなくて **「目安」** やからソフトコントロールやけど、これがめちゃくちゃ効く。

## 実装・手順：暴走しない並列リサーチャーを作る

### 1. 環境構築

```
pip install claude-agent-sdk anthropic
export ANTHROPIC_API_KEY=sk-ant-...
```

`claude-agent-sdk` は0.4系以降がOpus 4.7に対応しとる。古いバージョンやとTask Budgetsのパラメータが通らんから注意な。

### 2. サブエージェント定義（リサーチ用）

サブエージェントは **「専門化＋ツール制限＋予算」** の3点セットで作るのがコツや。

```
from claude_agent_sdk import Agent, Tool

researcher = Agent(
    name="researcher",
    model="claude-opus-4-7",
    description="特定トピックを深掘りリサーチして要約を返すエージェント",
    system_prompt="""あなたは熟練のリサーチャーです。
与えられたトピックについてWebSearchとWebFetchを駆使して情報を集め、
最終的に300-500字の要約を返してください。
冗長な調査は避け、重要なソース3〜5件に絞ること。""",
    allowed_tools=["WebSearch", "WebFetch"],  # ファイル操作などは禁止
    task_budget=15000,  # ★ これが今回の主役
)
```

ポイントは3つ。

* **`allowed\_tools` でツール制限** ：リサーチ役にファイル書き込みは要らんから外す
* **`task\_budget=15000`** ：このサブエージェント1回の実行で **目安1万5千トークン**
* **system\_promptで「3〜5件に絞れ」と明示** ：予算と整合する指示を出す

### 3. オーケストレーター（親エージェント）

```
from claude_agent_sdk import Agent

orchestrator = Agent(
    name="orchestrator",
    model="claude-opus-4-7",
    description="複数トピックを並列リサーチして統合レポートを作る",
    system_prompt="""あなたは編集長です。
与えられたテーマを 3つのサブトピックに分解し、
researcherサブエージェントに 並列で投げてください。
返ってきた要約を統合して、最終レポートを作成します。""",
    subagents=[researcher],
    task_budget=80000,  # 全体予算
)

result = orchestrator.run(
    "2026年4月のAI業界トレンドを 調査してレポートしてください"
)
print(result.final_message)
```

### 4. 並列実行はモデルに任せる

ここが大事なポイントやけど、 **「並列に走らせるかどうか」はモデルが決める** ねん。Agent SDKの設計思想は **「能力（capability）は開発者が定義、スケジューリングはモデルが判断」** や。

つまり開発者がやることは、

これだけ。あとはモデルが「あ、このリサーチは独立してるから3並列で投げよう」って勝手に判断してくれる。

### 5. 実行ログで予算消費を確認

実行後に各エージェントのトークン消費を見てみよう。

```
for sub_run in result.subagent_runs:
    print(f"{sub_run.name}: {sub_run.tokens_used} / budget {sub_run.budget}")
```

出力例：

```
researcher (topic=GPT-5.5): 12450 / budget 15000
researcher (topic=Claude Opus 4.7): 14200 / budget 15000
researcher (topic=Google投資): 11800 / budget 15000
orchestrator: 38600 / budget 80000
```

予算の **80〜95%程度** に収まるんが理想や。常に超えてるならbudget増やす、常に余ってるなら絞る、で調整していく。

## なぜこの設計か：3つの意図

### 意図1：サブエージェントは「ツール」として設計する

`description`は **親エージェントに見せる説明文** やと割り切る。 **「いつ呼ぶべきか」が一発でわかる名前と説明** にすると、モデルが正しく使い分けてくれる。 **長文プロンプトで人格を演じさせるより、ジョブディスクリプションを明確に書くほうが10倍ええ** 。

### 意図2：task\_budgetは「契約」として使う

予算は **見積もり書** やと思え。「リサーチ1回 = 15000トークン」って契約しといたら、モデルは **その範囲で 仕事を完結させようと最適化** する。これがあるかないかで、 **コスト予測の精度が一気に上がる** で。

### 意図3：親が context を持ちすぎない

サブエージェントの中間結果を **親が全部受け取ろうとすると、親のcontextがすぐ埋まる** 。Agent SDKは **「サブエージェントの最終メッセージだけを親に返す」** 設計やから、これに乗っかるんが正解。 **親は司令塔、サブはブラックボックス** で割り切ろう。

## ハマりどころ

### ハマり①：Task Budgetsはハードリミットやない

「budget=10000で 設定したのに 12000使われた！」みたいなことが普通に起きる。あくまで **目安・ソフトコントロール** やから、 **「絶対超えたらアカン」用途には max\_tokensを 別途設定** する必要があるで。

### ハマり②：サブエージェントから別のサブエージェントは呼べん

これ仕様としてハッキリしてる： **サブエージェントは Task（Agent）ツールを持てへん** 。多階層のエージェント構造を組みたくなる気持ちはわかるけど、 **2階層までで設計しなおす** ほうが早いし安定する。

### ハマり③：並列実行を強制できへん

「絶対並列で走らせたい」っちゅう要件でも、モデルが **「シーケンシャルでええかな」って判断したら シーケンシャルになる** 。プロンプトで強く促すか、それでもダメなら **自前で `asyncio.gather` で 投げる** しかない。

### ハマり④：tokenizerが変わってる

Opus 4.7は **新しいトークナイザー** に切り替わってるから、同じテキストでも **過去モデルの 1.0〜1.35倍のトークン** になる場合があるで。budget設定を **過去の感覚で やると足らん** ことがあるから、最初は気持ち多めにしとくのが安全や。

## まとめ

Claude Opus 4.7の **Task Budgets** は、 **エージェントの「暴走コスト問題」を ソフトに 解決する強力な仕組み** や。

おさらいすると、

* サブエージェントは **専門化＋ツール制限＋予算の3点セット**
* 並列化判断は **モデルに委ねる** （capabilityだけ定義）
* task\_budgetは **契約書** として使うとコスト予測が安定する
* ハードリミットやないから、致命的な制限は **max\_tokensで 別途**
