> 「どの AI エージェントが読みに来てもいいように、日報を4箇所に残しておく」
> ——という運用を、`/report` 1コマンドで実装した話。

**こんな人向け**
- 日報・週報を複数プラットフォームに残す運用をしている / したい
- Claude Code の `/command` を実用レベルで使い倒したい
- AI エージェントから読み取りやすい「データの置き場所」を設計したい

---

## まず、これだけ見てほしい

```
✅ 日報を記録しました！
  📁 Vault:  Reports/2026/04/2026-04-30_日報_[名前].md
  🐙 GitHub: Issue #[番号]（updated）
  📘 Notion: created
  💬 Slack:  posted（Bot のスレに返信）

🎉 全員の日報が揃ったので Issue #[番号] を自動クローズしました！
```

`/report` を1回打つだけで、これが全部終わる。

- Obsidian Vault に Markdown 保存
- GitHub Issue にチーム集約
- Notion DB にレコード登録
- Slack の Bot スレに返信
- git に push して永続化

**所要時間：約2分。** 今まで15〜20分かかってた作業が。

---

## なぜ「4箇所に残す」のか — これが本記事のキモ

**結論から言うと、AI エージェントネイティブな環境を作りたいから。**

最近、Claude Code、Codex、その他いろんな AI エージェントを業務に組み込んでる。
で、どのエージェントを使うときも、**「過去の自分たちが何をやってきたか」をエージェントに読み取らせたい**。

ただ、エージェントごとに得意な「読み取り口」が違う。

| 場所 | 得意な読み取り | 用途 |
|---|---|---|
| 📁 Obsidian Vault | git grep / ファイル検索 | コードと並列で読める。CLI から速い |
| 🐙 GitHub Issue | gh api / GitHub MCP | チーム単位で集約・レビュー |
| 📘 Notion DB | Notion MCP / プロパティ集計 | 構造化クエリ・ダッシュボード |
| 💬 Slack | Slack MCP | 速報・スレッドでの議論 |

それぞれ**役割が違う**。1個には絞れない。

「Notion だけにしよう」「Slack だけにしよう」と何度も思ったけど、
**どの AI エージェントが読み取りに来てもいいように4箇所に残しておくのが、結局いちばん柔軟**だった。

イメージとしては、こういう構造になってる。

```
       ┌───────────────────────────────────────────┐
       │   Claude Code  /  Codex  /  他エージェント   │
       └────────┬───────┬───────┬───────┬──────────┘
                │       │       │       │           AIから
                ▼       ▼       ▼       ▼          読み取る
            📁 Vault  🐙 Issue  📘 Notion  💬 Slack
                ▲       ▲       ▲       ▲
                └───────┴───┬───┴───────┘
                            │                       /report で
                       /report (2分)               一括書き込み
                            │
                            👤  人間
```

「**人間は1箇所に書く、AI はどこからでも読める**」という形を作りたい。

問題は、**人間が4箇所に書くのは現実的じゃない**ということ。
振り返りをしたいのに、思い出すのに時間がかかって業務時間を使うのは、本末転倒すぎる。

なので Claude Code に `/report` を仕込んで、**「1コマンドで4箇所同時に残す」** をやらせることにした。

| やってること | 結果 |

所要時間 **15〜20分 → 2分**。

そして、**4箇所すべてに同じ内容が残るので、どの AI エージェントから読みに来ても答えが返せる**。

## `/report` を打つとどうなるか

ターミナルで叩く。

```
/report
```

Claude が動き出す。

```
名前: [名前]（git user.name から自動判定）
対象日: 2026-04-30 でいい？
```

**……名前、勝手に判定された。**

`git config user.name` を見て判定テーブルにマッチすれば、ユーザー確認はスキップ。

体調を聞かれる。

```
体調/気分を教えて（絵文字+一言）
> 🔋(90%) 集中できた
```

ここから先がいい。**Ship（今日やったこと）の叩き台が勝手に出てくる。**

```
📦 Ship自動収集結果（YYYY-MM-DD）

🔧 Git（GitHub全体）
- [リポジトリ名]: [コミットメッセージ1行目]
- [リポジトリ名]: [コミットメッセージ1行目]

📅 Googleカレンダー
- [HH:MM] [予定タイトル]
- [HH:MM] [予定タイトル]

💬 Slack（チームチャンネル）
- [本人の投稿 先頭100字まで]

📝 Notion（検索）
- [本人が触ったページ]

✅ Notion（タスクDB — 実施予定日）
- [タスク名]（進捗: ○○ / 優先度: ○○）
```

5ソースを並列で叩いて「今日のあなたの足跡」を全部拾ってくる。

- GitHub 全リポジトリ横断のコミット
- Google Calendar の予定
- Slack の自分の投稿
- Notion の検索（自分が触ったページ）
- Notion DB タスク（実施予定日が今日）

ユーザーは「OK / 修正・追加」を返すだけ。**ゼロから書かない**。

これがいちばん効く。「今日何やったっけ」を思い出す時間が消滅する。

残り（Block / Next / 感謝）は1メッセージでまとめて聞かれて終わり。**2分。**

## ここがキモ（5つの設計）

### ① `git config user.name` から名前を自動判定

毎回「あなたは誰？」を聞かれない。判定テーブル1個持つだけ。

```markdown
| git user.name に含まれる文字列 | メンバー |
|---|---|
| [文字列1]   | A |
| [文字列2]   | B |
| [文字列3]   | C |
```

ヒットしなければ初回だけ確認。これだけで体感が変わる。

### ② Ship の叩き台を「全足跡から」自動生成

ここが本記事でいちばん効いてる部分。

`/report` は名前と対象日が決まった瞬間に、**5ソースを並列で叩いて自分の今日の足跡を全部拾ってくる**。

| ソース | 取りに行くもの |
|---|---|
| GitHub | 全リポジトリ横断のコミット（`gh api search/commits`） |
| Google Calendar | 対象日の予定一覧 |
| Slack | チームチャンネルでの本人の投稿 |
| Notion（検索） | 本人が触ったページ |
| Notion（タスクDB） | 実施予定日が今日のタスク |

ユーザーは出てきた叩き台を「OK / 修正・追加」するだけ。**ゼロから書かない**。

これで「**振り返りに時間を使いすぎる問題**」の核が解ける。
振り返りをしたいのに、思い出すフェーズに業務時間を溶かすのは、本当にもったいない。

### ③ GitHub Issue は「1日1Issue × 全員集約」

これが運用としていちばん気に入ってる。

- タイトル: `YYYY-MM-DD 日報`
- ラベル: `daily-report`
- body の中に `## A` `## B` `## C` の3セクション

各人が `/report` を打つと、**自分の `## 名前` セクションだけが書き換わる**。

そして、

**3人揃ったら Issue が自動でクローズされる。**

```
🎉 全員の日報が揃ったので Issue #N を自動クローズしました！
```

揃ってなければ：

```
📝 日報未提出: [名前]
```

これだけで「誰がまだ書いてないか」が GitHub Issue 一覧で**一発で見える**。
軽いリマインダーで足りる。Issue が open か closed かだけで進捗がわかる。

### ④ Notion の体調プロパティは絵文字に自動マッピング

入力は「🔋(90%) 集中できた」みたいな自由記述。Notion DB プロパティは ☀️ / ☁️ / ☔️ の3択。

```
70%以上    → ☀️
40〜69%   → ☁️
39%以下    → ☔️
絵文字入力 → 該当する天気アイコンに寄せる
```

入力 UX は自由、データはきれいに集計される。両取り。

Notion 側でダッシュボード化するときに効く。

### ⑤ Slack は**デフォルトで「Bot のスレに返信」**

ここ、設計判断としてしっかり決めた点。

`/report` は Slack に投稿するとき、**チャンネルに直接ポストしない**。
デフォルトの挙動は、

> 夕方に Bot が流している日報テンプレ投稿を探して、**そのスレッドに返信する**

```
【YYYY年MM月DD日日報】— Bot からのテンプレ投稿（親）
　└─ 🧵 A のスレ返信（/report で投げる）
　└─ 🧵 B のスレ返信
　└─ 🧵 C のスレ返信
```

理由は2つ。

- 🟢 **タイムラインを日報で埋めたくない**
- 🟢 **全員の日報を1スレに集約したい**（スレを開けばその日全員分が読める）

フォールバック動作：もし朝の Bot テンプレが見つからなかった日は、Slack だけスキップする。Vault / GitHub / Notion の3つはちゃんと保存される。

「投稿先がない日にチャンネル直書きで救済」みたいなことは**わざとやらない**。
集約のためのデザインを崩したくない。

## 再編集モード（同じ日に書き直したいとき）

`/report` を再度叩くと、既存ファイルがあれば**自動で再編集モード**。

```
📝 既存の日報が見つかりました（YYYY-MM-DD [名前]）

体調/気分: 🔋(90%) 集中できた
Ship: ...

編集したい項目を選んでください（カンマ区切りで複数可）:
  1. 体調/気分
  2. Ship
  3. Block
  4. Next
  5. 感謝・気づき
  6. 全部
```

選んだ項目だけ更新。**未選択の項目は触らない**。

差分は4箇所すべてに反映される：

- Vault: ファイル更新 ＋ `## 編集履歴` を追記
- GitHub Issue: 該当サブセクションだけ置換
- Notion: 該当プロパティだけ更新
- Slack: 同じスレに「🔄 日報再編集」で追加投稿（変更項目だけ）

「Block 1個だけ追加したい」みたいなとき、**体験が壊れない**。

---

## 最後は git commit & push

ファイル保存だけでは意味ない。最後に勝手に push する。

```bash
git add "Reports/YYYY/MM/YYYY-MM-DD_日報_[名前].md"
git commit -m "日報（YYYY-MM-DD [名前]）"
git push
```

push に失敗したら `git pull --rebase && git push` で1回だけリトライ。

これで Vault が**チームの永続ログ**になる。

**日報を git で grep できる世界。**
「先月のあのプロジェクトで誰が何やったっけ」が `git log -- Reports/` で全部出る。

そして、**Claude Code でも Codex でも、Vault のディレクトリを `Read` するだけで、過去の日報がそのまま読める**。
これが AI エージェントネイティブ運用の本体。

## まとめ：4箇所に残すのは「AI のため」

- 過去：日報をSlack　Botない返信。15〜20分かかる。月2回くらい漏れも出る
- 現在：`/report` 1コマンドで Vault / GitHub / Notion / Slack / git push まで **2分** で終わる
- 設計のキモは **「入力は自由、出力は構造化」** と **「チーム集約は GitHub Issue 1本」**
- そして本質は、**「どの AI エージェントが読み取りに来てもいいように、4箇所すべてに残しておく」**

データの置き場所を1個に絞らないというのが、結局いちばん柔軟だった。
人間が4箇所に書くのは無理だが、**Claude Code 経由なら2分で済む。**

「人間は1箇所に書く、AI はどこからでも読める」——
この形を作るのに、Claude Code の slash command は本当に強い武器になる。

- 📄 `commands/report.md` — `/report` コマンド本体（処理フロー・モード分岐・テンプレ）
