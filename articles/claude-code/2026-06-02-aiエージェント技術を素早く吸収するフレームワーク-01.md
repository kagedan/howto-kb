この記事では、色んなサービスややり方が毎日のように出てくるAIエージェント関連情報を、素早く吸収するためのフレームワークをご紹介いたします。  
このフレームワークを基に、後続の記事ではより詳しい情報やサービス、最新情報をまとめていく予定ですので、もし良いやり方だと感じましたらいいねお願いします！

## 新しい技術を素早く吸収するための工夫

AI関連の情報は毎日のように新しい情報が出てきます。私が思う最も理解しやすい情報の吸収のやり方は、「これは何のために作られたか？」で理解することです。つまり、目標と課題ですね。  
このAI・IT業界で20年以上色んな技術を学んできましたが、目標と課題はそこまで多くないですし、そんなに変わらないことに気づきました。領域が違ったとしても、人が持っている目標や悩みはそこまで大きく変わらないものですよね。  
AIエージェントも同じです。色んな技術や単語が次から次へと出てきますが、目標と課題は限られていて、それを基に情報を整理すると、新しい情報の整理と比較が可能になります。

AIエージェントが達成したい目標は何で、それを達成するための課題は何かを理解していきましょう。

### AIエージェントの目標は何か？

AIエージェントは基本的にLLMを使います。(最近は色んな生成AIを使う場合が多いですが、、説明を簡単にするために。)  
LLMはテキストを入力すると、そのテキストに対する出力が返ってくるだけで、記憶はないものです(stateless)。つまり、前回の会話も覚えていないし、周りの状況も分かってないです。  
AIエージェントの目標は、そんなstatelessなLLMを使って「仕事を任せる」ことだと思います。

### LLMに仕事を任せるためにはどんな課題があるのか？

仕事を任せるために必要なことを整理すると、大きく4つあると考えられます。

1. 仕事の状況をちゃんと理解できていること
2. 思考レベルが高いこと
3. 実際に作業物が作れること
4. 任せても大丈夫なこと

実際にAIエージェント関連技術は、この４つの課題を解決するために作られることが多いと思います。（開発・運用の課題もありますが、それは別記事で、、）

この四つの軸を図で表すとこのようになります。

課題ベースではなく、機能要素ベースでまとめるのも効果的ですね。Lilian Weng（OpenAI）による広く知られた整理では、AIエージェントは**LLMを中心に、Planning・Memory・Toolsの３つのコンポーネント**で構成されます。

![](https://static.zenn.studio/user-upload/6f0251b0ab1f-20260602.png)

*出典: Lilian Weng, "LLM Powered Autonomous Agents" (2023)*  
<https://lilianweng.github.io/posts/2023-06-23-agent/>

上のフレームワークと対応させると、**Memory → 1. 状況を理解させる**、**Planning → 2. 思考レベルを上げる**、**Tools・Action → 3. 行動させる**に当たります。一方で **4. 任せても大丈夫にする（安全性・品質）** はこの図には含まれていませんが、仕事を任せる時は外せない重要な項目なので含めました。（皆様も追加が必要な軸があると感じましたら、自分だけの軸をどんどん追加していきましょう！）

3年前(2023)の整理を見ても、今と課題やゴールそのものは実際にそこまで変わっていません。  
目標と課題の4つの軸で、AIエージェントに関連する技術の全体図を整理すると、新しい技術がどんどん追加されても分かりやすく整理が出来ます。「どんな課題があり、それをどう解決しようとしているのか」を見ていきましょう。

## 代表的なAIエージェントの技術をまとめてみる

ここでは、AIエージェントの代表的な技術を４つの軸でまとめていきます。  
各課題に対して、どんなアプローチがあるのか、そして、そのアプローチはどんな課題をどう解決しようとするのかの視点で整理します。

### 1. 仕事の状況をちゃんと理解できていること（状況理解）

新入社員に仕事を振る時と同じで、状況や背景を理解させないと正しく動けません。

**アプローチ①：直接指示する（System Prompt, Few-shot）**

* **【仕事の要件】** 仕事を任せるには、まず「あなたの役割は何か」「どういうやり方で進めるか」を教える必要があります。
* **【LLMの限界】** しかし、LLMは自分が何者で、どんな状況で呼び出されたのか全く知らない真っ白な状態です。
* **【どう解決するか】** だから、**System Prompt**で「あなたはソフトウェアエンジニアで、この仕事をお願いします」とテキストで直接役割を与えたり、**Few-shot**で「例えばこういう風にやるんだよ」と具体例を直接教え込みます。

**アプローチ②：情報を動的に取得させる（RAG, Memory等）**

* **【仕事の要件】** 実際の仕事では、社内マニュアルや過去の顧客とのやり取りなど、様々な情報を参照しながら進める必要があります。
* **【LLMの限界】** だけど、情報量が多すぎると一回にLLMに渡せる上限（Context Window）に引っかかりますし、LLMは過去の会話をすぐに忘れてしまいます。
* **【どう解決するか】** だから、必要な情報を「動的に」取得して渡す仕組みを作ります。
  + **RAG / Vector Store**: 質問に関連する社内文書だけを検索し、ピンポイントで渡します。
  + **Short/Long-term Memory**: 忘れてしまう過去のやり取りやユーザーの好みをシステム側で記憶しておき、毎回必要な分だけ渡します。
  + **Entity Memory**: LLMは詳細な情報を抜け落とす傾向がありますが、仕事だと困ります。だから特定の人物や物に関する詳細設定（Entity）だけを構造化して記憶させ、参照させます。

**アプローチ③：情報量を管理する（Trimming, Summarizing）**

* **【仕事の要件】** 仕事を行うための全ての情報を渡したいです。会話や作業履歴も記憶させたいです。
* **【LLMの限界】** しかし、何度も言うようにLLMが一度に読める情報には限界があり、限界を超えるとエラーになります。
* **【どう解決するか】** だから、溢れないように情報を圧縮します。古い会話を切り捨てたり（**Trimming**）、会話を短く要約したり（**Summarizing**）して情報量をコントロールします。

### 2. 思考レベルが高いこと（推論・構造化）

状況や情報が揃っていても、適切な手順やロジックに従って推論できなければ、結果の品質は上がりません。

**アプローチ①：プロンプトで思考を誘導する（CoT, ReAct等）**

* **【仕事の要件】** ある仕事を遂行するために必要な考え方などを伝える必要があります。
* **【LLMの限界】** しかしLLMにはそのようなやり方を持ってないです。
* **【どう解決するか】** だから、プロンプトの工夫で「賢い思考」を教えます。
  + **Chain of Thought (CoT)**: 「ステップバイステップで考えて」と指示し、推論プロセスを書き出させます。
  + **ReAct**: 「考えて(Thought) → 行動し(Action) → 観察する(Observation)」という試行錯誤のループを回させます。
  + **Reflexion**: 失敗した時に「なぜ間違えたか」を自分で振り返らせてから再試行させます。

**アプローチ②：設計パターンで思考を構造化する（ワークフロー構築）**

* **【仕事の要件】** 実際の業務は「リサーチして、書いて、レビューして、修正する」といった複数の工程から成り立っています。
* **【LLMの限界】** LLMでは1回の呼び出し（1回のプロンプト）でこれら全てを完璧にこなすことは難しいです。
* **【どう解決するか】** だから、作業を分割し、構造化します。
  + **Task Decomposition**: 大きな仕事を、実行可能な小さなタスクに分解して順に処理させます。
  + **Evaluator-optimizer**: 「作る人」と「チェックする人」を分け、修正ループを組んで品質を上げます。
  + **Supervisor / Swarm**: 複数の専門エージェントでチームを作り、リーダー（Supervisor）が指示を出したり、エージェント同士で協調（Swarm）して複雑な問題を解かせます。

### 3. 実際に作業物が作れること（行動）

考えて終わりではなく、実際に手を動かしてもらわないと仕事は完了しません。

**アプローチ：外部操作ができる手足をつける（Tool calling等）**

* **【仕事の要件】** 仕事を完結させるには、スケジュールをカレンダーに入れたり、Excelを作ったり、最新のニュースを調べたりと「手足を動かす」必要があります。
* **【LLMの限界】** しかしLLMは単なるテキスト生成器なので、箱の中に閉じこもっており、自力では外部システムを一切操作できません。
* **【どう解決するか】** だから、LLMに「手足（ツール）」を与えます。
  + **Tool calling**: LLMに「今はこのツールを使え」と判断させ、ツールを使うための指示を出力させます。
  + **色んなツールを与える**: tool callingで API連携 / Web Search / Code実行 / File操作など、色んな操作を可能にします。

### 4. 任せても大丈夫なこと（安全性・品質）

何かを間違っても大丈夫な作業環境を作ります。

**アプローチ①：暴走を防ぐ（Guardrails等）**

* **【仕事の要件】** 業務を任せる以上、顧客に不適切な発言をしたり、会社の予算を無駄遣いしたりしないという「安全性」が絶対に必要です。
* **【LLMの限界】** しかし自律的に動くエージェントは、時として予測不能な行動をとったり、無限ループに入ってしまうリスクがあります。
* **【どう解決するか】** だから、行動を監視し、制限をかける仕組みを導入します。
  + **Guardrails（ガードレール）**: 発言内容や行動がルールから外れていないか監視し、ブロックします。
  + **Cost-limiter / Error Handling**: APIのコストに上限を設けたり、エラー時の安全な停止方法を定めておきます。

**アプローチ②：品質を測る（Evals等）**

* **【仕事の要件】** 「任せる」ためには、アウトプットの品質が一定水準を満たしているか、定量的に判断できなければなりません。
* **【LLMの限界】** エージェントの出力は毎回揺らぎがあるため、「たまたま上手くいった」のか「実用レベルに達している」のか、目視だけでは判断しきれません。
* **【どう解決するか】** だから、品質を自動的・定量的に測る仕組みを作ります。
  + **評価 (Evals) / Benchmarking**: テストデータを使って、タスクの達成率や正確性をスコア化します。
  + **A/B Shadow Testing**: 本番の裏側でエージェントを動かし、人間の結果と比較してリリース基準を満たしているか確かめます。

## まとめ

AIエージェントの技術スタックを追っていると、「LangChainだ」「RAGだ」「ReActだ」と横文字が飛び交い、混乱してしまうかもしれません。  
しかし、「この技術は、LLMのどんな限界を乗り越えて、どんな仕事を任せるために作られたのか？」という視点を持つと、似た技術の洗い出しや比較ができるようになり、より効率的に理解することが可能になります。

* **状況を理解させる**（LLMは文脈を知らない、限界があるから）
* **思考レベルを上げる**（LLMは1回で複雑な推論は性能が落ちるから）
* **行動させる**（LLMは手足がないから）
* **任せても大丈夫にする**（LLMは予測不能で揺らぐから）

例えば、**Claude Codeのhooks** とかはどう理解できるのでしょうか？  
Hooksは、Claude Codeが特定のファイルやタスクなどの行動を行った時に、自動的に決まったshell コマンドを実行する機能です。  
個人的にはこのように技術を説明してもらっても少し理解が難しいと感じます。なので、この記事のように課題や目的から考えてみましょう。

【仕事の要件】仕事では決まったプロセスが存在し、特定の作業が終わった後は必ず行うべき作業があります。例えば、コード編集後はcommitするなどですね。  
【LLMの限界】しかし、LLMにそれを任せると、忘れてしまう場合があります。  
【どう解決するか】なので、LLMにお任せするのではなく、特定のファイルが編集されたことや、タスク終了後に自動的にshellを実行することで、安心してAIエージェントに仕事を頼めるようになります。

つまり、Hooksは、 **4. 任せても大丈夫にする** に当てはまることですね。人によっては、3番がより自然かも知れません。  
4番の場合、アプローチ①：暴走を防ぐにも、アプローチ②の品質を測るにも当てはまらないので、アプローチ③：ルールを確実に守るなどを追加するのもありだと思います。

今後新しいツールや論文が出てきたときも、この「4つの箱」のどこに入るのかを意識してみてください。日々のキャッチアップがずっと楽になるはずです。  
もしこの軸が足りなくなったら、それに合わせて追加＆変更することで、より皆様の独特な軸を作っていきましょう。

### 全体図

| 軸 | 課題 | 関連技術 |
| --- | --- | --- |
| 1. 状況を理解させる | 直接指示する | System Prompt, Few-shot |
|  | 必要な情報を取得させる | RAG, Vector Store, Short-term Memory, Long-term Memory, Entity Memory |
|  | 情報量を管理する | Trimming, Summarizing, Context Window管理 |
| 2. 思考レベルを上げる | プロンプトで思考を誘導 | CoT, Tree of Thought, ReAct, Reflexion |
|  | 設計パターンで構造化 | Sequential, Parallel, Routing, Prompt Chaining, Coordinator, Supervisor, Hierarchical, Swarm, Human-in-the-loop 等 |
| 3. 行動させる | Tool calling | API連携, Web Search, File操作, Code実行 等 |
|  | 出力形式の制約 | Structured Output |
| 4. 任せても大丈夫にする | 暴走を防ぐ | Guardrails, Cost-limiter, Error Handling |
|  | 品質を測る | 評価/Benchmarking, A/B Shadow Testing, Bias/Fairness, Self-reflection |

次の記事では、それぞれの課題に対して、実際にどのような技術があるかを一つずつ深めていきたいと思います。  
その後には、AIエージェントを実際の現場に導入するための課題とそれを解決するための工夫を整理します。MCPなどの工夫ですね。  
そして最後には世の中のフレームワークや製品などを見て、これはどのようなビジネス課題を解決しようとするのかを見ていきましょう。
