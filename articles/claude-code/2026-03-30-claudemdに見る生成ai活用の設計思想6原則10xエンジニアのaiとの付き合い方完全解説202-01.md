「AIコーディングツールを毎日使ってるのに、なぜか生産性が上がった実感がない」

正直に言うと、これは多くのエンジニアが今まさに抱えている、かなり深刻な問題です。

[Stack Overflow 2025 Developer Survey](https://survey.stackoverflow.co/2025/ai)によると、プロの開発者の**51%**がAIツールを**毎日使用**しています。2024年の76%から、使用中または使用予定のエンジニアは**84%**にまで増加しました。

でも、ここからが衝撃的なデータです。

同じ調査で、AIツールへの信頼度はむしろ**低下**しています。「高く信頼する」と答えたのはわずか**3.1%**。**45.7%**が精度に不信感を示し、ポジティブな感情は2023-2024年の**70%超**から2025年には**60%**に低下しました。

つまり、こういうことです。

「みんな使ってる。でも、みんな不満を感じてる」

もっと言えば、[METR（AI安全性研究機関）が実施したランダム化比較試験](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)では、経験豊富なオープンソース開発者**16名**・**246タスク**を対象にした結果、AI使用時に実際には完了時間が**19%遅くなった**という結果が出ています。

ここだけの話、さらに面白いのが——開発者自身は事前に「**24%速くなる**」と予測し、事後も「**20%速くなった**」と信じていたということです。

認知バイアスと実際の生産性の乖離。これ、他人事じゃありません。

あなたも心当たりがありませんか？

Claude CodeやGitHub Copilotを導入して、なんとなく「速くなった気がする」。でも、ふと振り返ると、AIの出力をチェックして直す時間、プロンプトを試行錯誤する時間、コンテキストがズレて最初からやり直す時間——トータルで見ると、本当に速くなったのか怪しい。

この記事は、まさにその問題に正面から切り込みます。

[![画像](https://assets.st-note.com/img/1774818263-AW4fGsikrYtID19mBaLbMZwC.png?width=1200)](https://assets.st-note.com/img/1774818263-AW4fGsikrYtID19mBaLbMZwC.png)

「AIを使いこなせない状態」から「AIに自走させる状態」への変革イメージ

この記事で得られるもの——

* Claude Code開発者Boris Cherny本人が実践する**6つの設計原則**の完全解説
* 「AIが同じミスを繰り返す問題」を根本解決する**自己改善ループ**の具体的な構築方法
* **10〜15セッション並列稼働**で圧倒的な生産性を実現するワークフロー設計

* 1週間・1ヶ月・3ヶ月の**段階的導入ロードマップ**

ただのツール紹介記事ではありません。「AIとの協働をどう設計するか」という**思想レベル**の話です。

読み終わった後、あなたのClaude Codeとの付き合い方は根本から変わります。

## この記事を書くに至った背景——「自己流の限界」が見えた瞬間

2025年から2026年にかけて、生成AIのコーディング支援は爆発的に普及しました。

GitHub Copilot、Cursor、Claude Code——選択肢はどんどん増え、「AIを使わないエンジニアは時代遅れ」とまで言われるようになった。実際、[Stack Overflow調査](https://survey.stackoverflow.co/2025/ai)によれば、開発者の**84%**がAIツールを開発プロセスで使用中または使用予定です。

でも、McKinseyの調査が示す現実はもう少し複雑です。

McKinseyによると、生成AIツールを使用する開発者は、使用しない開発者と比較して「**フロー状態に入った**」と感じる割合が**39%高く**、AI使用者は**幸福感・充実感を報告する割合が2倍**だったとされています。

「気持ちよく働けている」——これ自体は素晴らしいことです。

しかし同じMcKinseyの調査では、**複雑なタスクの開発時間短縮は12%**、中程度のタスクで**10%**にとどまり、「コード品質を維持するには、品質の構成要素を理解し適切にプロンプトする必要がある」と指摘されています。

つまり、「気分はいいが、実際の成果は思ったほど出ていない」というのが多くの現場の実態です。

ぶっちゃけ、これはツールの問題ではなく、**使い方の設計**の問題です。

ここで登場するのが、Claude Code開発者のBoris Cherny氏です。

[Boris Cherny氏（@bcherny）](https://x.com/bcherny/status/2007179832300581177)は、Anthropic社内でClaude Codeを開発しているエンジニアで、常時**10〜15のClaude Codeセッション**を並行稼働させています。ターミナル**5つ**＋ブラウザ**5〜10**＋モバイル。**3〜5個のgit worktree**を同時に立ち上げ、それぞれ独立したClaudeセッションを並列実行する手法を「**最大の生産性向上策**」としてチーム内で共有しています。

しかも、彼がサブエージェントを「スラッシュコマンドのように扱う」ことで、コード簡略化・E2Eテスト・セキュリティレビュー等を自動化しているという事実。

ここで重要なのは、Boris Cherny氏が単に「たくさんAIを使っている」のではなく、**AIとの協働を体系的に設計している**という点です。

その設計思想が凝縮されたのが、CLAUDE.mdというたった1つのファイルでした。

[Argenis Leon氏がGitHub Gistに構造化して公開](https://gist.github.com/argenisleon/a477ab4d8e5595d639132e6bcac015f2)したこのCLAUDE.mdには、Boris Cherny氏が日々の開発で磨き上げた**6つの設計原則**が記されています。

これはただの設定ファイルではありません。

「AIに何をさせ、何をさせないか」「AIが間違えたとき、どうやってシステムとして学習させるか」「人間のコンテキストスイッチをゼロにするにはどうすればいいか」——こうした問いに対する、実戦で鍛え抜かれた**回答集**です。

## この記事の全体像——6原則を「3層構造」で理解する

この記事では、Boris Cherny氏のCLAUDE.mdに記された6つの設計原則を、独自の「**AVI（Architect-Verify-Iterate）フレームワーク**」で体系化して解説します。

なぜ独自のフレームワークで整理するのか。

Boris Cherny氏のCLAUDE.mdは実践者向けの簡潔なルール集であり、それぞれの原則がなぜ重要で、どう連携するかについては明示的に書かれていません。しかし、6つの原則を俯瞰すると、明確な構造が浮かび上がります。

**AVIフレームワーク——3つの層**

**第1層: Architect（設計する）**

この層は「AIに仕事をさせる前に、どう設計するか」。多くのエンジニアが陥る「とりあえずAIに投げる」を防ぐ層です。

**第2層: Verify（検証する）**

この層は「AIの出力をどう品質担保するか」。AIが生成したコードを盲目的に受け入れない仕組みです。

**第3層: Iterate（反復改善する）**

この層は「AIをどう育てるか」。使えば使うほどAIが賢くなる、複利的な仕組みです。

この3層は独立しているのではなく、**循環する**構造になっています。

設計（Architect）→ 検証（Verify）→ 反復改善（Iterate）→ 次の設計にフィードバック

この循環こそが、Boris Cherny氏が「**compounding system（複利システム）**」と呼ぶものの正体です。

## CLAUDE.mdの基礎知識——「設定ファイル」ではなく「設計書」である

ここで基礎を整理しておきます。

CLAUDE.mdとは何か。

CLAUDE.mdは、Claude Codeがすべての会話の開始時に自動的に読み込む特別なMarkdownファイルです。プロジェクトのルートディレクトリに配置するだけで、Claudeに永続的なコンテキストを提供できます。

[Anthropic公式のClaude Codeベストプラクティスドキュメント](https://code.claude.com/docs/en/best-practices)には、こう明記されています。

> 「ほとんどのベストプラクティスは一つの制約に基づいている：**Claudeのコンテキストウィンドウはすぐに埋まり、埋まるとパフォーマンスが低下する**」

これ、目から鱗でした。

多くのエンジニアが「AIに詳しく指示すればするほど良い」と考えています。でも実際は逆で、コンテキストウィンドウが埋まるほど、AIの出力品質は下がるのです。

だからこそ、CLAUDE.mdは「必要最小限の、しかし決定的に重要な情報だけを書く」ファイルでなければならない。

**よくある誤解を3つ正します。**

**誤解1: 「CLAUDE.mdはプロンプトの保存場所」**

違います。CLAUDE.mdはプロンプトではなく、**行動原則の定義書**です。「こう聞かれたらこう答えろ」ではなく、「あらゆる状況でこの原則に従え」という設計思想を書く場所です。

Boris Cherny氏が[XDA Developersの記事](https://www.xda-developers.com/set-up-claude-code-like-boris-cherny/)で明かしているように、Anthropic社内のClaude Codeチームでは、チーム全員で**単一のCLAUDE.md**ファイルをgitにチェックインして共有し、**週に複数回更新**しています。

もしこれが単なるプロンプトの保存場所なら、共有する意味はありません。設計思想だからこそ、チーム全員で共有し、全員で磨き上げるのです。

**誤解2: 「CLAUDE.mdは長ければ長いほど良い」**

正反対です。[Anthropic公式ドキュメント](https://code.claude.com/docs/en/best-practices)によるCLAUDE.mdの設計指針には、こう書かれています。

* 簡潔に保ち、各行について「**これを削除したらClaudeがミスするか？**」と自問する
* Claudeがコードから推測できることは書かない
* `@path/to/import`構文で別ファイルをインポートし**段階的開示**する
* **CLAUDE.mdが長すぎるとルールが無視される**ため定期的に剪定する
* 強調には「**IMPORTANT**」や「**YOU MUST**」を使う

実際、多くの開発者が**500行、800行、1,000行超**のCLAUDE.mdを書いていますが、Boris Cherny氏のアプローチはおよそ**100行前後**。短いからこそ、すべての行が「生きたルール」として機能します。

**誤解3: 「CLAUDE.mdは一度書いたら終わり」**

ここが最も重要な誤解です。CLAUDE.mdは**生き物**です。

Boris Cherny氏は、Claudeが誤った動作をするたびにCLAUDE.mdにルールを追加し、同じミスを防止する「**自己改善ループ**」を実践しています。PRコードレビュー時に`@.claude`タグで同僚のPRにもCLAUDE.md更新を組み込んでいるほどです。

Cherny氏曰く——「**Claudeは自分自身のためのルールを書くのが驚くほど上手い**」。

[![画像](https://assets.st-note.com/img/1774818267-3QA4ycEm1zrOj8HTvMk2nJl5.png?width=1200)](https://assets.st-note.com/img/1774818267-3QA4ycEm1zrOj8HTvMk2nJl5.png)

CLAUDE.mdの自己改善ループ: 使うほどミス率が下がる複利効果

もう一つ、前提知識として押さえておくべきこと。

CLAUDE.mdには**3つの配置レベル**があります。

* **プロジェクトルート**: `./CLAUDE.md` → そのプロジェクト固有のルール
* **ユーザーグローバル**: `~/.claude/CLAUDE.md` → 全プロジェクト共通の個人設定
* **サブディレクトリ**: `./src/CLAUDE.md` → 特定ディレクトリのルール（段階的開示）

この階層構造を使いこなすことで、「チーム共通のルール」「個人の好み」「特定モジュール固有の注意事項」を分離できます。

さて、基礎知識はここまでです。

ここからが本題——Boris Cherny氏の6つの設計原則を、実際のテンプレートやコード例とともに、徹底的に解説していきます。

## 有料パートで得られるもの——あなたのAI開発が「仕組み」に変わる

ここまでの無料パートで、以下のことが分かったはずです。

* AIコーディングツールの**生産性は「使い方」で決まる**（ツール自体の問題ではない）
* CLAUDE.mdは「設定ファイル」ではなく「**設計書**」である
* Boris Cherny氏の6原則は**AVIフレームワーク**（設計→検証→反復）の循環構造を持つ
* CLAUDE.mdは**短く、生き物のように育てる**ことが重要

有料パートでは、この理解をもとに**実装レベル**まで落とし込みます。

具体的にあなたが得られるもの——

**即使えるテンプレート5点セット**

* CLAUDE.md基本テンプレート（ソロ開発者向け）
* CLAUDE.mdチーム版テンプレート（3〜10名チーム向け）
* lessons.md自己改善テンプレート
* tasks/todo.md タスク管理テンプレート
* 1週間セットアップチェックリスト

**6原則の完全実装ガイド**

**実例に基づくケーススタディ2本**

**1週間・1ヶ月・3ヶ月の導入ロードマップ**

[Anthropic公式](https://code.claude.com/docs/en/best-practices)も推奨しているように、「**Claudeに自身の作業を検証する手段を与えることが、最もレバレッジの高い施策**」です。この有料パートを読み終えた後、あなたはその「レバレッジの高い施策」を具体的に実装できるようになります。

AIを「なんとなく使う」から、「**仕組みとして運用する**」へ。

その転換点が、この先にあります。

## 実践セクション1: CLAUDE.md導入の準備編——環境構築からファイル設計まで

ここからが本番です。

まずはCLAUDE.mdを導入するための環境を整えましょう。「とりあえずファイルを作ればいい」と思うかもしれませんが、準備段階での設計ミスが後々の運用を大きく左右します。

**ステップ1: Claude Codeの導入確認**

Claude Codeがインストール済みでない場合は、まず公式サイトからインストールしてください。ターミナル、デスクトップアプリ、VS Code拡張、JetBrains拡張のいずれかで利用できます。

確認コマンド:

```
claude --version
```

**ステップ2: プロジェクトのディレクトリ構造を決める**

Boris Cherny氏の実践に基づく推奨ディレクトリ構造は以下です。

```
your-project/
├── CLAUDE.md              ← プロジェクトルートの設計書
├── tasks/
│   ├── todo.md            ← タスク管理ファイル
│   └── lessons.md         ← 自己改善ループの記録
├── src/
│   └── CLAUDE.md          ← （任意）サブディレクトリ固有ルール
└── .claude/
    └── settings.json      ← Claude Code設定
```

ここで重要なのが`tasks/`ディレクトリです。Boris Cherny氏のCLAUDE.mdでは、タスク管理について以下のフローが定義されています。

1. **Plan First**: `tasks/todo.md`にチェック可能な項目を書く
2. **Verify Plan**: 実装開始前にプランを確認する
3. **Track Progress**: 進捗に応じて項目を完了にする
4. **Explain Changes**: 各ステップで変更の概要を説明する
5. **Document Results**: `tasks/todo.md`にレビューセクションを追加する
6. **Capture Lessons**: 修正後に`tasks/lessons.md`を更新する

これは「ファイルを作る」話ではなく、**AIとの協働プロトコルを定義する**話です。

**ステップ3: CLAUDE.mdの初期ファイルを作成する**

ここで、私が「**スケルトン・ファースト原則**」と呼んでいるアプローチを紹介します。

多くの人は、CLAUDE.mdを最初から完璧に書こうとします。しかしBoris Cherny氏のアプローチは逆です。**最小限のスケルトン（骨格）から始めて、実際のミスに応じて育てていく**。

初期のCLAUDE.mdは**20〜30行**で十分です。

```
## Core Principles
- Simplicity First: 最小限のコード変更で目的を達成する
- No Laziness: 根本原因を見つける。一時的な修正は禁止

## Workflow
- 3ステップ以上のタスクは必ずplan modeに入る
- タスクの計画はtasks/todo.mdに書く
- 修正を受けたらtasks/lessons.mdを更新する

## Verification
- タスク完了前に必ず動作確認する
- テストを実行し、ログを確認し、正しさを証明する
```

この20行が「種」です。ここから、日々の開発でClaudeがミスするたびにルールが追加され、1ヶ月後には50〜100行の洗練されたCLAUDE.mdに育ちます。

**ステップ4: gitにチェックインする**

これを忘れる人が多いのですが、**CLAUDE.mdは必ずgitで管理する**べきです。

理由は3つ。

第一に、チームメンバーと共有できる。Boris Cherny氏のチームでは、全員が同じCLAUDE.mdを参照し、週に複数回更新しています。

第二に、変更履歴が残る。「このルールはいつ、なぜ追加されたのか」をgit logで追跡できます。

第三に、ルールの剪定がしやすい。「このルールは3ヶ月前に追加されたけど、もう不要では？」という判断ができます。

```
git add CLAUDE.md tasks/
git commit -m "feat: add CLAUDE.md and task management structure"
```

CLAUDE.md導入の4ステップ: 環境構築からgit管理まで

**ステップ5: ユーザーグローバルCLAUDE.mdを設定する（任意）**

プロジェクト固有のルールはプロジェクトルートのCLAUDE.mdに書きますが、全プロジェクト共通の個人設定は`~/.claude/CLAUDE.md`に書けます。

例えば、以下のような内容です。

