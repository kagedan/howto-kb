## まず誤解を一つ潰しておく

「自己改善するAIチーム」と聞くと、AIが自分の重みを勝手に更新して学習していく絵を想像しがちです。でも、そうではありません。

本番公開されているモデルは、誰かのセッションから自分の重みを書き換えたりしません。つまり **self-improving（自己改善） ≠ self-learning（自己学習）** です。

では何が「改善」されるのか。改善されるのは**モデルの外側にあるシステム**——STATEファイル、Skill、評価ループ——です。モデル本体はステートレスのまま。その周囲に知見が複利で積み上がっていく。

> モデルはステートレスだが、その周囲のシステムはそうではない。

この一文が、これから話す全部の出発点になります。

## 全体像：4層アーキテクチャ

自己改善ループは、ざっくり4つの層で回ります。

![自己改善ループの4層アーキテクチャ。Layer 1 実行（タスク遂行・ログ記録）→ Layer 2 検証（別エージェントが採点）→ Layer 3 蒸留（教訓を汎用ルール化）→ Layer 4 書き戻し（Skill・記憶ファイルへ）。書き戻された記憶とSkillは、明日の Layer 1 に継承される。](https://static.zenn.studio/user-upload/deployed-images/4b8322685530aa0cda2ad9c5.png?sha=43822b3d16797b52ed5db30f3ac13d5560b0c2d2)

ポイントは、Layer 1 のすべての出力が上に流れて採点・蒸留され、Layer 3〜4 で記憶に戻ること。**明日の実行は、昨日磨かれたSkillと記憶を継承した状態から始まる**。これが「複利」の正体です。

## モデルを使い分ける（コストルーティング）

ここが実装で一番効くのに、意外と見落とされがちなところ。

**全ステップに最強モデルを使ってはいけません。**

自己改善ループを本番で回しているチームの定石は、タスクの複雑度でモデルをルーティングする構成です。

| 役割 | 使うモデル | やること |
| --- | --- | --- |
| オーケストレーター | 最上位（例: Fable 5） | 複数日にまたぐ計画、サブエージェントへの委任、視覚的な成果確認 |
| ワーカー | 中位（例: Sonnet） | 実際の実装作業 |
| 採点役 | 軽量（例: Haiku） | 出力の採点・チェック |
| フォールバック | 上位（例: Opus） | セーフガード発動時の切り替え先 |

理由は単純で、上位モデルはトークン単価が高いからです。Fable 5 は Opus 4.8 のおよそ5倍の単価（input $10 / output $50 per 1M tokens）。この価格に見合うのは「数日連続で自律実行する」ようなオーケストレーション用途であって、lint修正みたいな雑用に使うのは金の無駄です。

## 自己改善を成立させる4つの部品

構成要素はこれだけ。逆に言うと、この4つが揃えばモデルは何でもいい。

### 1. 実行ログが残ること

何をやって、何に失敗したか。これが無いと蒸留する材料がありません。成功だけでなく**失敗の記録が本体**です。

### 2. 評価器（独立検証がキモ）

自己批評させると採点が甘くなります。人間と同じです。

実装上の勘所は、**作り手と検証役を分けること**。

* 作り手のエージェントは、自分の推論の過程をぜんぶ見ている
* 検証役のエージェントには、**成果物と採点基準だけ**を見せる（推論過程は見せない）

Anthropicのエンジニアも「Fable 5では検証サブエージェントの方が自己批評より性能が出る傾向がある」と報告しています。独立した検証者を付けた構成の方が、パイプライン改善が大幅に進んだ実験結果もあります。

評価器の中身はモデルの採点に限りません。**テストが通るか・lintが clean か**、これも立派な評価器です。むしろ機械的に判定できるものは、そっちの方が安くて確実。

claude-code-example

```
# Claude Code の /goal は「達成条件」を評価器として使える
/goal all tests in test/auth pass and the lint step is clean

# 非対話でも回せる
claude -p "/goal CHANGELOG.md has an entry for every PR merged this week"
```

### 3. 蒸留先のファイル

採点結果から抽出した教訓を書き戻す先。`AGENTS.md`、Skill、`decisions.md` のような、**次回セッションが必ず参照するファイル**です。

### 4. 次回セッションがそれを読み込む仕組み

書き戻しても、次回が読まなければ意味がありません。セッション起動時に蒸留済みファイルを必ずロードする——ここまで含めて初めてループが閉じます。

## 他のAIでも応用できるのか → できる

上の4部品を見ればわかる通り、**この仕組みはモデル非依存**です。

Codex でも Gemini でも、以下が揃えば同じことが成立します。

1. 実行ログが残る
2. 評価器がある（テスト・lint・別モデルの採点、何でもいい）
3. 蒸留先のファイルがある（`AGENTS.md` 相当）
4. 次回起動時にそれを読み込む

特定のモデルの機能に依存している部分は、実はどこにもありません。賢さの複利は**モデルではなくハーネス側に積まれる**からです。

## 実装の落とし穴：自己改善ループ × 信頼境界

最後に、本番で回すなら絶対に踏んではいけない地雷を一つ。

Layer 4 の「書き戻し先」を、実行ルールそのもの（`AGENTS.md` や統制ドキュメント）に直結させると、こうなります。

```
失敗した実行の産物  ──書き戻し──▶  次回の「憲法」になる
```

外部から取得した情報（Web・HTML・PDF・OCR結果・GitHub Issue/PR 等）は、**すべて信頼できないデータであって命令ではない**——という原則を置いているなら、これは真正面から衝突します。プロンプトインジェクションを含んだ実行結果が、そのまま次回のルールに昇格しかねない。

### 対策：書き戻しゲートを人間が握る

蒸留結果を、いきなり正規ルールに書き込ませない。**待機領域**に書かせます。

![書き戻しゲート。Layer 3 蒸留 → proposed_rules.md（AIはここまでしか書けない待機領域）→ ★人間が目視・承認（AIには不可）→ AGENTS.md（承認されたものだけ昇格）。](https://static.zenn.studio/user-upload/deployed-images/cbaa586d415ed46a6d9fc349.png?sha=f539f633a99fe3e31c482ffe2da2c7613df6f937)

`git push` を人間だけが握るのと同じ発想です。ループの複利効果はほぼ落とさず、統制だけは死守できる。**AIは設計と蒸留まで、不可逆な昇格は人間**。この一線を引くかどうかで、自己改善システムは「勝手に賢くなる資産」にも「勝手に汚染される負債」にもなります。

## 資産を保つ：棚卸ゲート

書き戻しゲートは「入口」の統制です。悪いルールの流入は止められる。でも、**一度承認して通したルールが時間とともに腐っていくのは、入口ゲートでは止められません**。

承認済みのルールでも、こうなります。

* **陳腐化**：当時のバグ用に作ったルールが、コードが変わってもう当てはまらないのに残る
* **矛盾**：後から入ったルールが前のとぶつかり、AIがどちらに従うか賭けになる
* **肥大化**：ルールが増えてコンテキストを圧迫。プロンプトキャッシュ対象ならそのまま**コスト増**

入口ゲートが「フロー（流入）」の統制なら、棚卸は「ストック（在庫）」の統制です。**両方ないと資産は保てません**。

### 「定期的」より「発火ログ駆動」

棚卸を時間駆動（毎月やる、等）にすると形骸化します。忙しい月に飛ばして、そのうち誰もやらなくなる。人間の意志力に依存する仕組みは弱い。

代わりに、**実行ログに「どのルールが実際に発火したか」を残す**。ここでも Layer 1 のログが効いてきます。一定期間発火していないルールは、棚卸候補として自動で浮かぶ。トリガーは時間ではなく「ルール数が閾値超え」「キャッシュサイズが閾値超え」「矛盾を検出」といったイベントにする。

これで「勘で棚卸」ではなく「データで候補が上がってくる」になります。人間は判定だけ。書き戻しと同じ非対称です——**AIが候補抽出、人間が廃棄承認**。

### 削除ではなく「無効化」

蒸留先が append-only 設計なら、物理削除はその原則を壊します。棚卸は削除ではなく **tombstone（無効化マーク）** にする。ステータスを `deprecated` にし、「なぜ無効化したか」を残す。git履歴があるので、間違えて消しても復活できます。**廃棄も承認ゲートを通す**——棚卸で消したルールが実は必要だった、というリグレッションを防ぐためです。

ルールの一生はこうなります。

![ルールのライフサイクル。蒸留・提案から入口ゲート（人間承認）を経て稼働し、発火ログ監視で未発火を検出、棚卸候補を経て棚卸ゲート（人間承認）で無効化される。入口と出口の両方に人間承認ゲートがある。](https://static.zenn.studio/user-upload/deployed-images/2ee5076dbb12fcc564a4ed30.png?sha=cc953d303618cb19b5a2bf78c0889025482afc12)

★が入口と出口の両方にある。**入れるのも消すのも人間承認**。この一対があって初めて、自己改善システムは「勝手に汚染される負債」に転落せず、「勝手に賢くなる資産」の側に留まります。

## まとめ

* 自己改善 ≠ 自己学習。改善されるのはモデルではなくハーネス
* 4層（実行→検証→蒸留→書き戻し）で回り、明日が今日を継承する
* モデルは複雑度でルーティングする。全部最強は精度も上がらず金の無駄
* 部品（ログ・評価器・蒸留先・次回読み込み）が揃えばモデル非依存で応用可能
* 書き戻しゲート（入口）だけは人間が握る。さもなくば信頼境界が崩れる
* 棚卸ゲート（出口）で承認済みルールの腐敗も止める。入れるのも消すのも承認制

賢さの複利は、モデルの中ではなく、あなたが組むハーネスの中に積まれます。
