Claude Codeに「テスト直して」と頼んだら、ファイルを何個も開いてコマンドまで実行して、気づいたら修正が終わっている。便利なんだけど、裏で何が起きているのかよくわからない——そんな状態で使っている方は少なくないはずです。

実はこの「裏側」を把握しておくだけで、指示の出し方がけっこう変わります。この記事では、Claude Codeの中核であるエージェントループの仕組みから、ツール、コンテキスト管理、安全機構まで、基礎から解説していきます。

## エージェントループ——Claude Codeの心臓部

Claude Codeに何か頼むと、内部では **エージェントループ** と呼ばれるサイクルが回り始めます。やっていることは意外とシンプルで、3つのフェーズの繰り返しです。

| フェーズ | やること | 具体例 |
| --- | --- | --- |
| **情報収集** | ファイルを読む、検索する、コマンドで状態を確認 | `grep` でエラー箇所を探す |
| **アクション** | コード編集、ファイル作成、コマンド実行 | 該当ファイルを修正する |
| **検証** | テスト実行、ビルド確認、差分チェック | `npm test` で修正を検証 |

ポイントは、この3フェーズが厳密に順番通り進むわけではないということです。情報収集しながら編集に入ることもあるし、検証結果を見てまた情報収集に戻ることもあります。

たとえば「テストが落ちてるから直して」と頼んだ場合、Claude Codeは以下のように動きます:

1. テストスイートを実行して失敗箇所を確認
2. エラー出力を読む
3. 関連するソースファイルを検索
4. ファイルを読んでコードの意図を理解
5. 修正を加える
6. テストを再実行して検証

ステップ2の結果がステップ3の検索クエリを決め、ステップ6の結果次第でステップ3に戻る——という具合に、各ステップが次の判断材料になっています。

そして見落としがちなのが、**このループの途中でいつでも割り込める** ことです。明らかに違う方向に進んでいるなと思ったら、そのまま修正指示を入力すれば Claude は方針を切り替えてくれます。完了を待つ必要はありません。

### ループを回す2つの部品

エージェントループは **モデル**（推論）と **ツール**（実行）の組み合わせで動いています。

```
あなたのプロンプト
    ↓
モデル（Claude）が推論: 「何をすべきか」を考える
    ↓
ツールを呼び出す: ファイル読み取り、コマンド実行、編集...
    ↓
結果をモデルにフィードバック
    ↓
モデルが次のアクションを判断
    ↓
（タスク完了まで繰り返し）
```

モデルだけだとテキストを返すことしかできません。ツールがあるからファイルを読んだり編集したりコマンドを叩いたりできるわけで、この「推論 + 実行」のループがClaude Codeをただのチャットボットと分けている部分です。

## ツール——エージェントの手足

組み込みツールは5カテゴリに分かれています。

| カテゴリ | できること | 例 |
| --- | --- | --- |
| **ファイル操作** | 読み取り、編集、新規作成、リネーム | `Read`, `Edit`, `Write` |
| **検索** | ファイル名パターンマッチ、正規表現で内容検索 | `Glob`, `Grep` |
| **実行** | シェルコマンド、テスト、ビルド、git操作 | `Bash` |
| **Web** | Web検索、ドキュメント取得 | `WebFetch`, `WebSearch` |
| **コードインテリジェンス** | 型エラー検出、定義ジャンプ、参照検索 | `LSP` |

これに加えてサブエージェントの起動やユーザーへの質問など、オーケストレーション系のツールもあります。

### ツールは拡張できる

組み込みだけで足りない場合の拡張手段が3つあります。使い分けはシンプルです。

| やりたいこと | 使う仕組み | 例 |
| --- | --- | --- |
| 外部サービスと接続 | **MCP** | Slack通知、Jiraチケット更新 |
| 繰り返すワークフロー | **Skills** | `/review-pr`, `/deploy-staging` |
| ツール実行前後に自動処理 | **Hooks** | 編集後に自動フォーマット |

## Claude Codeが「見えている」もの

`claude` コマンドを叩いた瞬間から、Claude Codeがアクセスできる情報は以下の通りです。

| 情報ソース | 内容 |
| --- | --- |
| **プロジェクトファイル** | カレントディレクトリ以下のファイル全体 |
| **ターミナル** | ビルドツール、git、パッケージマネージャなどCLI全般 |
| **git状態** | 現在のブランチ、未コミットの変更、コミット履歴 |
| **CLAUDE.md** | プロジェクト固有の指示・コーディング規約 |
| **auto memory** | 過去セッションで学んだパターンや設定 |
| **拡張機能** | MCP、Skills、サブエージェント等 |

「プロジェクト全体が見える」のがインラインのコード補完ツールとの決定的な違いです。認証のバグ修正を頼めば、関連ファイルを横断検索して複数ファイルにまたがる修正を一度にやってくれます。1ファイルしか見えないツールではこうはいきません。

CLAUDE.mdはセッション開始時に自動で読み込まれます。「テストは `pytest` で実行」「コミットメッセージは英語で」といったルールをここに書いておけば、毎回伝える手間が省けます。

CLAUDE.mdは複数の場所に置けます。全部読み込まれますが、より具体的な場所の指示が優先されます。

```
~/.claude/CLAUDE.md              # 全プロジェクト共通の個人設定
プロジェクトルート/CLAUDE.md      # プロジェクト固有（git管理推奨）
サブディレクトリ/CLAUDE.md       # ディレクトリ固有の追加指示
```

## コンテキストウィンドウ——知らないと詰む有限メモリ

ここが一番「知っておくべきなのに見落とされがち」な部分です。

Claude Codeのコンテキストウィンドウには、会話履歴・ファイル内容・コマンド出力・CLAUDE.md・auto memory・スキル定義・システム指示——全部が詰め込まれています。いわば **容量に上限のあるワーキングメモリ** です。

### 溢れたらどうなるか

コンテキストが上限に近づくと、Claude Codeは自動で **コンパクション** を実行します。

```
1. 古いツール出力を削除
       ↓ まだ足りない場合
2. 会話履歴を要約
       ↓ 単一の巨大出力が原因の場合
3. エラー表示（自動回復不能）
```

ユーザーのリクエストや重要なコードは保持されますが、**会話の初期に伝えた細かい指示は失われる可能性があります**。繰り返し伝えたいルールをCLAUDE.mdに書くべき理由はここにあります。

巨大なログファイルを丸ごと貼ると、コンテキストが一瞬で埋まります。`tail -100` や `grep` でフィルタしてから渡す癖をつけましょう。

### 今のコンテキスト使用状況を確認する

これで何がどれだけコンテキストを消費しているか一目でわかります。MCPのツール定義が想像以上に大きいこともあるので、`/mcp` でサーバーごとのコストもチェックしておくと安心です。

### サブエージェントでコンテキストを節約する

サブエージェントは **独立したコンテキストウィンドウ** を持ちます。メインの会話とは完全に別の空間で動くので、調査結果の要約だけがメインに返ってきます。

```
メイン会話: [指示, 会話履歴, コード, ...] ← ここが逼迫
    ↓ 調査タスクを委譲
サブエージェント: [独立した新しいコンテキスト] ← 別枠
    ↓ 完了
メイン会話に要約だけ返す ← コンテキスト消費が少ない
```

長いセッションでコンテキストが厳しくなってきたら、「この調査はサブエージェントに任せて」と指示するだけで自動的に委譲されます。

## 安全機構——チェックポイントと権限モード

### チェックポイント: ファイル編集は全て巻き戻せる

Claude Codeがファイルを編集する前に、変更前の状態が自動でスナップショットされます。問題があれば `Esc` を2回押すだけで過去の状態に巻き戻せます。

チェックポイントはgitとは独立した仕組みです。コミットしていなくても巻き戻せます。ただし、外部システムへの操作（DBへの書き込み、APIコール等）はスナップショットの対象外です。だからClaude Codeは外部への副作用がある操作の前に確認を求めてくるのです。

### 権限モード: `Shift+Tab` で切り替え

| モード | ファイル編集 | コマンド実行 | こんなときに |
| --- | --- | --- | --- |
| **Default** | 確認あり | 確認あり | 慎重に作業したいとき |
| **Auto-accept edits** | 自動 | 一部自動 | 日常的なコーディング |
| **Plan mode** | 読み取り専用 | 不可 | 計画だけ先に見たいとき |
| **Auto mode** | 自動 | 自動 | 信頼できるリポジトリ |

よく使うコマンドは `.claude/settings.json` に登録しておくと、毎回の確認をスキップできます。

```
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run lint)",
      "Bash(git status)",
      "Bash(git diff)"
    ]
  }
}
```

## セッション管理: 会話の中断と再開

会話は `~/.claude/projects/` にJSONL形式で自動保存されています。再開・分岐用のコマンドは以下の通りです。

```
# 直前のセッションを再開
claude --continue

# 過去のセッション一覧から選んで再開
claude --resume

# セッションを分岐（元のセッションはそのまま残る）
claude --continue --fork-session
```

同じセッションを複数ターミナルで `--continue` するとメッセージが混在します。並行作業には `--fork-session` か git worktree を使いましょう。

## まとめ

Claude Codeの中核は「情報収集 → アクション → 検証」を繰り返すエージェントループです。コンテキストウィンドウが有限であることを意識して、CLAUDE.mdに指示を集約し、必要に応じてサブエージェントに委譲する。これだけで、Claude Codeの挙動がかなり予測しやすくなるはずです。
