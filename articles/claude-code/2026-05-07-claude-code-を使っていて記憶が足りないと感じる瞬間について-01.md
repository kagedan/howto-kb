## はじめに

ターミナル上で直接コードを読み書きし、タスクを自律的に進めてくれる [Claude Code](https://code.claude.com/docs/en/overview) は、開発のワークフローに革新をもたらしつつあります。特に局所的なリファクタリングや、新規スクリプトの作成、エラーのトラブルシューティングなどにおいては、もはや手放せないツールになりつつあると感じている開発者も多いでしょう。

しかし、数日、数週間と同じプロジェクトで Claude Code と「協業」していると、ある種の明確なストレス、あるいは「隐性的な断絶感（見えない壁）」を感じる瞬間がないでしょうか。

それはモデルの推論能力が低いからでも、コンテキストウィンドウが狭いからでもありません。「継続的に記憶する」という能力が決定的に欠けているからです。

本記事では、一人の開発者の視点から、AI エージェントにおける「記憶不足」が引き起こす具体的なペインを整理し、それが単なるプロンプトエンジニアリングの問題ではなく、システムアーキテクチャにおける「[長期記憶層](https://ai.meta.com/research/publications/memory-layers-at-scale/)（[Memory Layer](https://ai.meta.com/research/publications/memory-layers-at-scale/)）」の欠如であることを考察します。

## Claude Code を使っていて「記憶が足りない」と感じる瞬間

日常的な開発の中で、AI の「健忘症」が開発体験を損なう典型的な瞬間は以下の通りです。

### 1. 新しいセッションを立ち上げるたびに「前提」を説明する徒労感

Claude Code のセッションを新しく切った瞬間、AI は「初日を迎えた優秀な新人」にリセットされます。プロジェクトの全体的なアーキテクチャ、現在採用している技術スタックの微妙なバージョン制約、あるいは「なぜこのディレクトリ構成になっているのか」といったコンテキストを、毎回何らかの形でロードし直す必要があります。

### 2. 「昨日合意したコーディング規約」が今日には忘れられている

「エラーハンドリングは例外を投げるのではなく、Result型を返すようにして」と以前のセッションで指示し、AI もそれを完璧にこなしていたとします。しかし翌日、別の機能追加を依頼すると、平然と `throw new Error()` を書いたコードを提案してきます。過去のセッションで形成された「共識」は、セッションを跨いだ瞬間に揮発してしまうのです。

### 3. 長期タスクを分割した際のコンテキストの欠落

巨大な機能開発を複数のPRに分割して進める場合、前半のPRで議論した設計上のトレードオフや、特定のモジュールに対する制約を、後半のPRの作業時には AI が意識できなくなります。「すでに見たはずのコード」や「すでに解決したはずの懸念」が自然に継承されないため、手戻りや人間側からの指摘（プロンプトによる修正）が増加します。

### 4. ユーザー個人の「好み」や「思考の癖」が定着しない

「常にテストコードを先に書いてほしい」「依存関係は最小限にしたい」「コミットメッセージのプレフィックスはこうしてほしい」など、開発者固有のワークスタイルや好みは、何度 AI と対話しても「AI 自身に定着」することはありません。毎回 `.cursorrules` のような静的ファイルに書き込むか、カスタムプロンプトに詰め込むしかなく、動的な暗黙知としては蓄積されません。

## それは本当に「文脈（コンテキスト）不足」の問題なのか

こうした問題に直面したとき、多くの開発者は「プロンプトに過去のやり取りを含める」「もっと大量のファイルをコンテキストとして読み込ませる」といったワークアラウンドに走ります。

しかし、ここでいくつかの概念を明確に区別する必要があります。

* **Context Window（コンテキストウィンドウ）:** 1回のプロンプトで処理できる情報量。「記憶力」ではなく「短期的な作業メモリ（RAM）」に過ぎません。
* **Chat History（会話履歴）:** 過去の文字列の羅列。AI が後から構造的に情報を引き出したり、一般化して適用したりするための「知識」にはなっていません。
* **Project Knowledge（プロジェクト知識）:** README やルールファイル。静的であり、日々の対話から動的に更新・進化していくものではありません。
* **Persistent Memory（永続的記憶）:** 過去の経験から学習し、セッションやタスクを超えて動的に保持され、適切なタイミングで引き出される構造化されたコンテキスト。

現状の Claude Code は、数十万トークンという巨大な Context Window を持っていますが、それは「一度に分厚いマニュアルを読める」というだけであり、「昨日教えたことを今日覚えている」という Persistent Memory を持っているわけではありません。

すべての履歴やルールを毎回プロンプトに詰め込むアプローチ（Stateless なモデルに対する力技）は、トークンコストの増大やレスポンスの遅延を招くだけでなく、「Lost in the middle（中間情報の喪失）」を引き起こし、AI の推論精度自体を低下させる原因にもなります。

## Claude Code が得意なことと、苦手になりやすいこと

ここで Claude Code（および背後にある Claude等のモデル）の特性を公平に整理します。

**得意なこと（推論と局所的実行）:**

* 目の前に提示されたコードの意図を正確に読み取る
* 指定された要件に基づく複雑なロジックの実装やリファクタリング
* 「現在のセッション内」での文脈を踏まえた、高度な推論とステップ実行

**苦手なこと（継続性と状態保持）:**

* セッションを跨いだ文脈の維持
* 過去の失敗やフィードバックからの「教訓」の自律的な蓄積と適用
* プロジェクトのライフサイクルに寄り添った、動的な知識の更新

つまり、LLM は「考える（Reasoning）」ことは極めて得意ですが、「長期的に覚え続ける（Remembering）」ことは、そもそもモデル単体に期待すべき機能ではなく、別の設計課題であると言えます。

## 長期協業には「推論」だけでなく「独立した記憶の層」が必要

AI コーディングアシスタントとの関係を「単発のタスク依頼」から「長期的なペアプログラミング」へと昇華させるためには、アーキテクチャの観点から根本的な見直しが必要です。

賢い推論エンジン（LLM）に対して、毎回ゼロから背景を説明するのではなく、**システム全体として「継続的・持続的・移行可能な Memory Layer（記憶層）」を独立したインフラとして設けること**が、論理的な解決策となります。

単なる RAG（検索拡張生成）やベクターデータベースの導入ではありません。単に過去のテキストをベクトル化して検索するだけでは、「古い情報と新しい情報のコンフリクト解決」や「開発者の意図の抽象化」ができないからです。必要なのは、AI が人間と同じように「情報を整理し、信念を更新し、必要な時に自然に引き出せる」ような、より高度でステートフルな記憶の仕組みです。

## 記憶をインフラとして切り出す「MemoryLake」という選択肢

この「AI エージェントにおける長期記憶の欠如」というアーキテクチャ上の課題に対して、より根本的なアプローチをとっているのが **[Memorylake](https://www.memorylake.ai/)** のような記憶インフラストラクチャです。

もし現在の Claude Code の運用において「毎回プロンプトで前提を埋め込む限界」を感じているのであれば、MemoryLake が提供する以下のような概念は非常に理にかなっています。

### 1. Persistent AI Memory Layer（永続的な記憶層）

MemoryLake は、単なるチャット履歴の保存庫や RAG 用のベクターDB ではありません。AI エージェントが読み書きできる「独立した記憶のレイヤー」として機能します。AI との対話から得られた「ユーザーの好み」「プロジェクトの規約」「過去の決定事項」を構造化し、永続的に保持・更新し続けます。

### 2. Cross-Session / Cross-Agent の連続性

最大の強みは、セッションやツールを跨いだ連続性（Continuity）の担保です。今日 Claude Code で合意したアーキテクチャの決定事項が MemoryLake に記憶されれば、明日別のセッションを開いたときや、あるいは別の AI ツール（例えばブラウザ上の Claude や別の Agent フレームワーク）を使ったときでも、その記憶をシームレスに引き継ぐことができます。

### 3. Portable & User-Owned（記憶のパスポート）

特定のツールやプロバイダーにロックインされた履歴ではなく、開発者自身（あるいはチーム）がポータブルな形で所有・管理（Governance）できる「Second Brain」として機能します。AI が何を記憶しているかを可視化し、必要に応じて人間が介入して修正・整理することも可能な設計思想を持っています。

## どんな人にこの視点が刺さるか

* **Claude Code や [Cursor](https://cursor.com/) などの AI ツールを日常的に使い込んでいる開発者**
* 複数のプロジェクトやマイクロサービスを並行して担当しており、AI に「今どの文脈にいるか」を切り替えさせるのに疲れている人
* 巨大な `rules` ファイルのメンテナンスや、コピペ用のプロンプトテンプレートの管理に限界を感じている人
* AI を「使い捨ての優秀な作業員」ではなく、「共に成長する長期的なパートナー」として育てたいと考えている人

## まとめ

Claude Code は間違いなく強力な推論エンジンであり、日々のコーディングタスクを劇的に加速させます。しかし、私たちが日常的に感じる「あれ、この話昨日もしたよね？」という隐性的なストレスは、コンテキストウィンドウの拡張だけでは根本的には解決しません。

LLM が担うべきは「現在の推論（Reasoning）」であり、過去の文脈やプロジェクトの歴史、個人の好みを保持するのは「記憶インフラ（Memory Infrastructure）」の役割です。この2つを分離して考えることで、AI との協業は次の次元へと進むはずです。

もしあなたが今、Claude Code の「短期的な賢さ」には満足しつつも、長期的なコラボレーションにおける「記憶の足りなさ」に直面しているなら、次に取り組むべきはプロンプトの工夫ではありません。**AI に対して、いかにして持続可能な Memory Layer を提供するか**というアーキテクチャのアップデートです。

その観点において、[Memorylake](https://www.memorylake.ai/) のような Cross-session, Portable な記憶インフラの導入は、今後の AI 開発ワークフローにおいて非常に有望かつ、検討に値するアプローチだと言えるでしょう。
