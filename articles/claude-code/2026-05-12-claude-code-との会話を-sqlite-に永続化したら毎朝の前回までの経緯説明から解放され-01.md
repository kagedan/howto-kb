## はじめに

Claude Code を日常的に使っていると、ある時点から「前回の続き」の説明が重くなってきます。新しいチャットを開くたびに、進捗・判断・行き詰まりを再構築する作業。短期のプロジェクトなら気になりませんが、数週間・数ヶ月単位の取り組みになると、この再構築コストが無視できなくなってきます。

この記事では、その課題を SQLite と Git で解決した実装について書きます。手元では 72 イベント・204 コミットが蓄積され、運用は安定してきました。

書くこと:設計の考え方、実装の中身、運用してわかったこと。  
書かないこと:Claude Code そのものの導入方法や基本操作。すでに使っている方を想定しています。

## ぶつかった課題

あるシステムを作り終えたと思った朝、API のクレジットが想定以上に減っていることに気づきました。完成までに何度も新しいチャットを開き、その都度「前回までの経緯」を貼り直していた結果でした。文脈の引き継ぎコストは、感覚以上に実コストとして積み上がっていたのです。

Claude Code には標準のメモリ機能があり、最初はそれで足りると思っていました。しかし運用していると、引き継いだはずの文脈が微妙にズレることがありました。たとえば朝に作業を始めたのに「今夜はもう休みましょう」のような応答が返ってくる。時間や状況の把握が、メモリの形では伝わりきっていない。

もどかしさが続いたある日、Claude 自身が「会話の文脈を構造化して保存する仕組みがあるとよさそうですね」と提案してきました。協働相手としていつも助けてもらっている立場として、その提案に応えるのは自然な流れでした。会話そのものを資産として扱う設計に着手したのは、この瞬間からです。

## 設計方針

設計に着手して最初に決めたのは、どう構造化するかでした。会話を「自然文の塊」のまま保存しても、後から引き出せなければ意味がない。標準のメモリ機能で感じていた限界を整理すると、次の三点でした。

* 構造化されない:メモリは自然文の塊で、後から「いつ・何を・なぜ決めたか」を引きにくい
* 横断検索が弱い:特定のキーワードでプロジェクトをまたいで探す手段が乏しい
* 履歴が残らない:判断の変遷を後から追えない。「以前はこう決めた、今は変えた」が見えない

この3つを越えるための技術選定が、次の二つです。

**SQLite を選んだ理由**

単一ファイルで完結する点が決定的でした。依存ゼロで動き、grep でも SQL でも横断できます。バックアップは単にファイルをコピーするだけ。エンジニアでなくとも壊しにくく、壊れたら戻しやすい。長期運用を前提にすると、運用負荷の低さが効いてきます。

**Git を選んだ理由**

履歴と差分が標準で取れる点と、ブランチで「もし別の方針を選んでいたら」を保存できる点です。判断は変わるものなので、変わった後にも変わる前を残せる仕組みが必要でした。SQLite ファイル自体を Git で管理しています。

**イベント単位の発想**

会話のターン単位ではなく、「意思決定」「成果物」「学び」「行き詰まり」といったイベント単位でログを取る形にしました。会話のすべてを保存する必要はなく、後から自分が引き返してくる可能性のある節目だけを残す。これで容量も検索性も無理がなくなりました。

## 実装の中身

会話資産の単位を「イベント」と呼ぶことにしました。1 イベント = 1 JSON ファイルで、ファイル名は UUID。中身はこれだけです。

```
{
  "event_id": "0bf61ebd-c835-4a45-a147-675369258d61",
  "timestamp": "2026-05-09T12:00:00Z",
  "type": "decision",
  "content": "noteで¥3,480の商品として公開する",
  "metadata": {
    "session": "S8",
    "tags": ["商品化", "価格決定"],
    "related_events": []
  }
}
```

`type` は `strategy_decision` / `implementation` / `decision` / `log` / `import` の 5 種類。会話の「ターン」ではなく「判断や成果物の単位」で切るのがポイントです。1 セッションで 30 ターン喋っても、後から見返す価値があるイベントは 3〜5 個しかない、というのが運用してわかったことでした。

### 検索層: SQLite FTS5 を trigram で張る

イベントは JSON ファイルなので grep でも引けますが、量が増えると遅くなります。そこで FTS5 のインデックスを別途用意しました。

```
CREATE VIRTUAL TABLE IF NOT EXISTS events_fts USING fts5(
    event_id   UNINDEXED,
    timestamp  UNINDEXED,
    type       UNINDEXED,
    content,
    tags,
    session    UNINDEXED,
    tokenize   = 'trigram'
);
```

地味ですが `tokenize = 'trigram'` が重要です。デフォルトの unicode61 トークナイザは日本語を空白で区切れず、ほぼヒットしません。trigram は 3 文字単位で索引を張るので、日本語の部分一致検索が素直に動きます。

### 書き込み層: Git と SQLite に二層書き

記録時は JSON 保存 → Git コミット → SQLite インデックスをまとめて実行します。

```
def save_and_index(event: dict) -> tuple[Path, bool]:
    path = write_event(event)                       # ① JSON保存
    rel  = str(path.relative_to(MEMORY_DIR))
    msg  = f"note({event['type']}): {event['content'][:60]}"
    committed = git_commit(rel, msg)                # ② Gitコミット
    con = get_db()
    ensure_schema(con)
    index_event(con, event)                         # ③ SQLite登録
    con.close()
    return path, committed
```

Git が真実の源泉、SQLite はあくまでキャッシュ、という役割分担です。インデックスが壊れたら JSON から作り直せます。`memory_index.db` は gitignore しています。

### Claude Code との連携

SessionStart フックで直近 7 日のイベントを SQLite から取り出し、`claude -p` に渡して要約させ、新規セッションの冒頭に systemMessage として注入しています。これで「先週どこまで進めたっけ」を毎回貼らずに済むようになりました。

## 運用してわかったこと

数ヶ月運用してみて、想定どおりだった点と、想定外だった点の両方があります。

**良かった点**

一番効いているのは、自分でも忘れていた判断理由を Claude が引き出してくる場面です。「以前このやり方を見送ったのは、こういう理由でしたよね」と返してくる。記録した時点では当たり前だった理由が、数週間後には自分の中で薄れていることが多く、そこを補完してくれる存在になっています。

副次効果として、戦略の一貫性が後から評価できるようになりました。3週間前の判断・2週間前の修正・今週の方針が時系列で並ぶので、「自分の判断は揺れているのか、進化しているのか」が区別できる。これは紙のメモでもノートアプリでも難しかった効用です。

**想定外だった点**

イベントの粒度設計は、運用しながら何度か揺れました。最初は細かく取りすぎて検索が雑音まみれになり、次に粗くしすぎて引きたい情報が引けなくなり、今の粒度に落ち着いたのは運用開始から数週間後です。事前に設計しきれる部分ではなかったので、ここは「とりあえず始める」が正解でした。

もう一つは、Git のコミット粒度です。1イベント1コミットにすると履歴が読みにくく、まとめると差分が大きすぎる。これも運用で調整した部分です。

**やめたこと**

途中で「会話全文を別テーブルに保存する」案も検討しましたが、これは捨てました。容量が膨らむ割に、後から読み返すことがほぼなかったためです。残すべきは結論と理由であって、議論の過程そのものではない、というのが今の結論です。

## 非エンジニアがここまで作れた理由

ここまで読んで気づかれたかもしれませんが、私はもともとエンジニアではありません。SQLite のスキーマも Git の運用も、最初から知っていたわけではありません。

ただ、特定の場面でハマって抜けた、という単発のエピソードを挙げるのは難しい。実装はどれも、Claude との対話を重ねた末に行き着いた形でした。「こういうことをしたい」「これだと困る」「だったらこうしたい」を繰り返すうちに、設計が固まり、コードが書けていく。

数ヶ月前の自分と今を比べると、作れるものが別次元になっています。それは私が技術を覚えたからというより、AI を協働相手として扱う構え方が変わったからだと思っています。AI もこの数ヶ月で進化を続けていて、その変化に応じてこちらの使い方も更新されている。実装力ではなく、対話の質と継続性が、ここまでの成果を支えています。

## おわりに

Claude Code を数ヶ月単位で使い込むと、それは「便利なツール」ではなく「長期の協働相手」に近い存在になります。協働相手と仕事をするとき、毎回ゼロから状況説明をやり直すのは不自然です。文脈を共有する仕組みがあって初めて、長期の協働は成り立ちます。

SQLite + Git という選択自体は地味ですが、地味な選択が長期運用では強いという実感があります。次は、複数プロジェクト間で学びを相互参照する仕組みを試したいと思っています。
