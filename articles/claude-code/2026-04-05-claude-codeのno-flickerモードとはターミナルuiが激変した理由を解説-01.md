この記事を読めば、Claude CodeのNO\_FLICKERモードが**なぜ話題になっているのか**、そして**ターミナルUIの内部で何が起きているのか**がわかります。

先日、Anthropicエンジニアの@trq212がXに投稿した一件のツイートが6.8万いいね・120万ビューを叩き出しました。内容は「Claude CodeにNO\_FLICKERモードを追加した」という技術的な発表でした。なぜこれがこれほど話題になったのか？個人開発者として気になったので深掘りしてみました。

---

## そもそもターミナルが「ちらつく」のはなぜ？

普通のCLIツールがターミナルに何かを描画するとき、「前の出力を消す → 新しい出力を書く」という操作を繰り返します。この消去と再描画の間に一瞬のギャップが生まれるため、高頻度で更新が起きるとちらつき（フリッカー）として見えてしまいます。

Claude Codeのような対話型AIツールは、レスポンスをリアルタイムにストリーミング表示します。つまり、更新頻度が非常に高い。その結果、ターミナル画面がビシビシと点滅して見えることがあったわけです。

---

## NO\_FLICKERモードで何が変わるのか？

NO\_FLICKERモードの核心は、**仮想ビューポートレンダラー**の導入です。

`vim` や `less` を使ったことがある人はわかると思いますが、あれらは「代替スクリーンバッファ（alternate screen buffer）」という仕組みを使っています。ターミナルをフルスクリーン的に占有し、差分だけを描き換えることで高速かつちらつきのない描画を実現しています。

NO\_FLICKERモードはこれと同様のアプローチで、Claude Codeのターミナル描画エンジンを根本から書き直したものです。具体的には以下の3つが変わりました：

### 1. 差分ベースの描画（Diff Rendering）

画面全体を毎回再描画するのではなく、「前回と変わった部分だけ」を更新します。これにより描画オーバーヘッドが大幅に削減され、フリッカーが消えます。

### 2. マウス対応（SGRマウスプロトコル）

ターミナル上でマウス操作が使えるようになりました。テキスト選択やスクロールがより直感的になります。コード補完候補の選択などでも活用が期待されます。

### 3. プロンプトの下部固定（Sticky Prompt）

入力プロンプトが常にターミナルの下端に固定されます。長いレスポンスが流れても、入力欄が画面外に消えてしまうことがなくなりました。これは体感的に大きな改善です。

## 個人開発者にとって何が嬉しいのか？

正直に言うと、「Claude Codeを長時間使う人ほど恩恵を受ける」変更です。

僕はClaude Codeを毎日6〜8時間使って個人開発をしているのですが、ターミナルのちらつきって地味にストレスなんですよね。特にコード生成のストリーミングが速いとき、画面が激しく点滅して「目が疲れる…」と思っていました。

NO\_FLICKERモードはそのストレスをほぼゼロにしてくれます。マウス操作対応も、ターミナルとエディターを行き来する回数が減るという意味で生産性向上につながります。

また、Claude Code 2.1.90でこのモードが正式統合される予定との情報もあります（@ClaudeCodeLog報告）。近いうちにすべてのユーザーがデフォルトで恩恵を受けられるようになりそうです。

---

## まとめ

Claude CodeのNO\_FLICKERモードは、ターミナルUIの描画エンジンを根本から書き直した改善です。

* **フリッカー解消**: 仮想ビューポートレンダラーによる差分描画
* **マウス対応**: ターミナル内でのポインタ操作が可能に
* **プロンプト固定**: 入力欄が常に画面下部に留まる

「たかがUIの変更」と思うかもしれませんが、毎日何時間も使うツールのUX改善は積み重なると大きな生産性の差になります。Claude Code 2.1.90でデフォルト化されたら、ぜひ試してみてください。
