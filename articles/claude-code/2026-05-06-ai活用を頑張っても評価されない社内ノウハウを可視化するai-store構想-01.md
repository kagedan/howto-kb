## はじめに

あなたが作った便利なプロンプト、チームに広まっていますか？

そしてもう一つ。AI活用を頑張ってきた結果が、評価面談で**数字として語れますか？**

多くの企業でいま、こんなことが起きています。

* ChatGPTやCopilotを導入した。でも「使い方は各自で」で終わっている
* 個人で便利なプロンプトを作っているが、Slackに流れて消えてしまう
* 隣のチームが同じようなプロンプトを別々に作っている
* AIを使って業務効率化のWebアプリやツールを作ったが、自分の部署だけで使って終わっている
* AI活用ノウハウは個人の頭の中にあるだけで、組織の資産になっていない
* AI活用で業務効率化しても、評価面談でどう説明すればいいか分からない

ツールを導入することと、組織がAIを使いこなすことは別の話です。

この記事では、この課題を解決するために考えた「**社内AI Store**」という構想を紹介します。

まだ構想段階ですが、同じ課題を感じている方と議論したくて書きました。

この記事で伝えたいのは、AI活用を広げるには「ツール導入」だけでなく、**ノウハウが流通し、貢献が可視化される仕組み**が必要だということです。

---

## 社内AI活用が「個人芸」で終わる理由

生成AIの普及によって、社員一人ひとりが**プロンプト・AIエージェント・簡易Webアプリ・業務自動化ツール**を作れる時代になりました。

以前なら開発者にしか作れなかった社内ツールが、今やノーコード・AIコーディングツールを使えば誰でも作れます。議事録自動生成、スライド生成アプリ、Excel集計ツール——こういったものが社内のあちこちで生まれています。

でも同時に、こんな問題が起きています。

### ノウハウが個人に閉じてしまう

便利なプロンプトを作っても、それを共有する場がない。SlackやTeamsで「このプロンプト便利でした！」と投稿しても、数日で流れてしまいます。結果として、**良いノウハウは作った人の手元にしか残らない**。  
![](https://static.zenn.studio/user-upload/46c2d2e05f5f-20260507.png)

### 発信力がある人のアイデアしか広がらない

社内でAI活用が広まるかどうかが、**個人の発信力**に依存しています。積極的に発信する人のプロンプトは使われ、地道にツールを作り続ける人の成果は埋もれる。これはフェアではありません。

### 重複が生まれ続ける

部署をまたいで確認する手段がないため、似たようなプロンプトやツールが何度も重複して作られています。せっかくの労力が無駄になっています。

---

## もう一つの問題：貢献が「定性的」にしか評価されない

AI活用を推進した人が、きちんと評価されているでしょうか。

「議事録作成の時間を半分に削減した」「設計レビューのプロンプトをチームに展開した」——こういった貢献は、現状では**定性的な言葉でしか語れない**ことが多いです。

評価する側も困っています。

* どのくらいの人が使ったのか分からない
* 本当に業務効率化につながったか数字で判断できない
* AI活用への取り組みをどう評価すればいいか基準がない

その結果、**AI活用を頑張った人が正当に評価されにくい構造**が生まれてしまいます。これでは、AI活用を積極的に推進しようというモチベーションが育ちません。

---

## 解決策：社内AI Storeとは

![](https://static.zenn.studio/user-upload/66bd84e426d6-20260507.png)

この2つの課題——「個人に閉じてしまう」と「貢献が評価されない」——を同時に解決するために考えたのが「**社内AI Store**」です。

一言で言うと、**社内で作られたAI活用資産を、一元管理・共有・評価できるプラットフォーム**です。

### 対象となるコンテンツ

エージェントだけが対象ではありません。社内で作られたAI活用資産をすべてカバーします。

| カテゴリ | 具体例 |
| --- | --- |
| **プロンプト・指示文** | 議事録要約、メール作成、設計レビュー、コードレビュー、障害調査 |
| **AIエージェント・Skills** | Copilot Agent、Gemini Gem、Claude Code Skills、Codex Skills、Cursorルール |
| **社内効率化アプリ** | スライド生成、Excel集計自動化、仕様書自動生成、ナレッジ検索 |
| **GASツール・Webアプリ** | 社内独自ツール全般 |

---

## 主な機能

### 登録・検索・カテゴリ

プロンプト・AIエージェント・業務効率化ツール・社内Webアプリ・AI活用ナレッジなど、社員が作ったAI活用資産を何でも登録・検索・利用できます。

**登録時の入力項目**

* タイトル・概要・使い方
* 対象業務・対象部署
* 利用ツール（ChatGPT / Copilot / Claude / Gemini など）
* 注意事項・公開範囲

**検索条件の例**

* キーワード・カテゴリ・業務領域
* 人気順・新着順・評価順

カテゴリは「開発」「議事録」「コードレビュー」「調査」「営業支援」など業務ドメインごとに整理します。

---

### コピー数・利用回数の計測（ここが重要）

プロンプトや指示文がコピーされた回数、アプリへのアクセス数・利用回数を記録します。

**「なんとなく使われた」を「数字で説明できる」に変えることが、この仕組みの核心です。**

---

### 作成者ページ：貢献を可視化する

作成者ごとのダッシュボードで、以下が確認できます。

| 指標 | 意味 |
| --- | --- |
| コピー数 | プロンプトがどれだけ使われたか |
| 利用回数 | ツール・アプリがどれだけ使われたか |
| いいね数 | 利用者からの評価 |
| レビュー数 | 実際の活用フィードバック |
| 利用部署数 | 横展開の規模 |
| 想定削減時間 | 業務効率化への貢献度 |

これにより、「自分が作ったものが、どの部署で、何回使われたか」が一目でわかります。**評価面談で貢献を説明する材料になる数字**です。

---

## 既存サービスとの違い（補足）

「MicrosoftのAgent StoreやGoogleのAgent Galleryと何が違うの？」という疑問があると思います。

| 比較項目 | Microsoft Copilot Studio | Google Gemini Enterprise | 社内AI Store構想 |
| --- | --- | --- | --- |
| 主な対象 | Copilot Agent | Gemini Agent | プロンプト・Agent・Skills・GASアプリ全般 |
| 複数ツール横断 | 弱い | 弱い | **強み** |
| プロンプト共有 | 中心機能ではない | 中心機能ではない | **中心機能** |
| 作成者の貢献可視化 | 標準では弱い | 標準では弱い | **作成者別ダッシュボード** |
| 評価制度との接続 | 別途設計が必要 | 別途設計が必要 | **最初から設計に含める** |

MicrosoftやGoogleのサービスは「AIを作る・公開する・管理する」基盤として優れています。

社内AI Storeが担うのは別のレイヤー——**ChatGPT・Copilot・Claude・Gemini・GAS・独自アプリなど、複数ツールに分散したAI活用資産を横断的に集約し、使われ方を可視化して、作成者の貢献を評価につなげる基盤**です。

競合ではなく、補完関係です。

---

## ただし、自由に置けるだけでは危ない

AI Storeは、社内のAI活用資産を共有・再利用するための仕組みです。

しかし、何でも自由に登録できる場所にしてしまうと、セキュリティや品質面のリスクが生まれます。

例えば、以下のような観点は必ず考える必要があります。

* 機密情報を含むプロンプトを登録しない
* 外部AIに送ってはいけない情報を明確にする
* 公開前レビューや承認済みラベルを設ける
* 利用対象部署や公開範囲を設定する
* 作成者・管理者・更新日を明示する
* メンテナンスされていないツールを放置しない
* 利用ログや評価指標を過度に人事評価へ直結させない

AI Storeは「何でも自由に置ける場所」ではなく、社内で安全に再利用するためのガバナンスを含めた仕組みとして設計する必要があります。

特に、AI活用を評価につなげる場合は、コピー数や利用回数だけで判断するのではなく、レビューコメント、利用部署数、実際の業務改善効果などを組み合わせて見ることが重要です。

## MVPと段階的な展開

最初から大きく作る必要はありません。段階的に進めます。

### Phase 1：プロンプト共有から始める

まずはプロンプト・指示文の共有に特化します。

* プロンプト登録・検索・カテゴリ
* コピー機能・コピー数計測
* いいね・作成者表示・ランキング

**この段階でも、「誰のプロンプトが何回コピーされたか」が可視化できます。**

### Phase 2：AIエージェント・Skillsに広げる

* Copilot Agent、Gemini Gem、Claude Code Skills、Codex Skills、Cursorルールの登録
* セットアップ手順・バージョン管理
* 承認済みラベル

### Phase 3：社内Webアプリ・GASのカタログ化

* WebアプリURL・GASツールの登録
* アクセス数・利用部署数の計測
* 改善要望・問い合わせ導線

MVPはNotionから始めるのもあり

Phase 1はNotionのデータベースで代替できます。プロパティで「コピー数（手動更新）」「いいね数」「作成者」「カテゴリ」を管理し、まず運用を試してみるのが現実的かもしれません。本格的なWebアプリは、需要が確認できてから作る方がリスクが低いです。

## この仕組みがもたらす「三方良し」

冒頭で挙げた2つの問題——「ノウハウが広がらない」「貢献が評価されない」——は、実は同じ根っこから来ています。**AI活用を個人の努力に依存させている**という構造的な問題です。

社内AI Storeがうまく機能すると、その構造が変わります。3者それぞれにメリットが生まれ、良い循環が回り始めます。

**共有する人（作成者）**  
作ったプロンプトやツールの利用数が蓄積され、AI活用への貢献を数字で説明できます。「頑張りました」ではなく、「このプロンプトが社内で500回コピーされました」と言える。

**使う人（利用者）**  
優れたプロンプトやツールをゼロから作らなくても、検索して即使えます。重複作業がなくなり、質の高いノウハウが全社に届きます。

**会社（組織）**  
AI活用の「ツール導入して終わり」問題が解消されます。個人のノウハウが組織の資産になり、AIネイティブ化が自然と進む仕組みが生まれます。

---

## まとめ

AIネイティブ化は「ツールを入れれば終わり」ではありません。個人の知見が組織に広がり、貢献した人が正当に評価される仕組みがあってこそ、本当の意味で組織のAI活用が進むと考えています。

社内AI Storeで実現したいのは、この4つです。

* **広がらない問題**：便利なプロンプト・ツール・Webアプリを社内で検索・再利用できる場を作る
* **評価されない問題**：コピー数・利用回数・いいね数で貢献を定量化する
* **重複作業の問題**：社内で何が作られているかを一元管理する
* **ツール導入で終わり問題**：ノウハウを流通させる仕組みでAIネイティブ化を加速する

まだ構想段階で、実装はこれからです。
