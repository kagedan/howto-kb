## はじめに

Finatextグループの社内AIコンテストで、Claude Codeのセッション履歴から、社内共有候補となるユースケース文書を生成するSkillを試作しました。

実際に1人分、3週間の履歴へ適用し、9件の文書をGitHubのPull Requestとして生成しました。しかし内容を確認すると、その多くは「他の社員が再利用できる知見」というより、個人の作業を読みやすく整理した記録に留まっていました。

本記事では、試作した仕組みと実際の生成物を紹介したうえで、セッション履歴を組織のナレッジへ変換する際に、文章生成だけでは解決できなかった問題を整理します。

## AI活用の知見が組織に残りにくい理由

Claude CodeやCursorを日々の業務で利用すると、AIへの依頼方法、調査の進め方、人間が判断を挟む箇所など、さまざまな工夫が生まれます。

しかし、これらはそのまま組織の知識になるわけではありません。

第一に、AIの利用方法は、対象となる業務、リポジトリの構成、既存の設計、利用者が持つ前提知識などに強く依存します。本人にとっては有効だった使い方でも、どの部分を切り出せば他の人にも役立つのかは自明ではありません。

第二に、共有するには、本来の業務とは別に言語化する作業が必要です。セッションを振り返り、固有のファイル名や事情を取り除き、目的、進め方、注意点を他人が理解できる形へ整理しなければなりません。作業を完了した本人にとって、この追加作業の優先度は高くなりにくいものです。

第三に、AIツールやモデルは短期間で変化します。具体的な操作方法やプロンプトだけを共有しても、ツールの更新によって短期間で価値を失う可能性があります。そのため、単なる手順ではなく、どの条件で有効だったのか、どのような判断を行ったのかまで残す必要があります。

こうした理由から、各社員が個別に得た工夫や失敗が本人のセッション内に残り、別の社員が類似した探索や試行錯誤を繰り返す状態が起こり得ます。

問題は、AIの利用履歴が存在しないことではありません。履歴は存在していても、他人が利用できる知識の形になっていないことです。

## 今回検証した仮説

そこで、次の仮説を置きました。

> セッション履歴から共有候補となる文書の下書きを生成できれば、AI活用事例をゼロから書き起こす負荷を減らせるのではないか。

目指したのは、生のセッションログをそのまま共有することでも、生成した文書を人間の確認なしに公開することでもありません。

生ログには、ファイルパス、社内システム名、取引先名、個人名などの固有情報が含まれます。また、試行錯誤の経緯は残っていても、その作業が他人にとって有用なのか、どの条件で再利用できるのかまでは明示されていない場合があります。

そのため今回の試作では、次の範囲を対象としました。

1. セッション履歴から共有候補となるタスクを選ぶ
2. 固有の情報を一般化し、投稿用の文書へ変換する
3. 機密情報と思われる記述を検出・置換する
4. 人間が確認できるGitHubのPull Requestとして出力する

まずは完成した組織ナレッジを自動生成するのではなく、人間が確認・修正するための下書きをどこまで作れるかを検証しました。

## 試作した仕組み

今回試作したSkillは、おおむね次の流れで動作します。

```
collect_sessions.py
セッション履歴の抽出・ノイズ除去
        ↓
usecase-collector
共有候補となるタスクの選定
        ↓
analyze_session.py
選定した履歴の構造化
        ↓
usecase-analyzer
ユースケース文書の生成・一般化
        ↓
confidential-review
機密情報と思われる記述の検出・置換
        ↓
create_usecase_pr.sh
ブランチ作成・コミット・Pull Request作成
```

履歴のパース、ファイルパスの正規化、Git操作など、手順を機械的に決められる処理はコードで実装しました。一方、共有候補とするか、複数セッションを同じタスクとまとめるか、固有情報をどこまで一般化するか、機密情報が含まれるか、といった意味的な判断はLLMへ任せています。

特に、共有候補かどうかの判定は、以下のように指示しています。

```
価値があるユースケースの基準:
- 再利用可能
- 他人が真似できる
- 業務速度を改善
- 失敗回避に役立つ
- 高頻度業務
- 高難易度業務
- 特定ドメイン知識を含む

価値が低いものの例:
- 単発調査
- 雑な質問
- 一時的なデバッグ
- 個人メモ
- AIへの愚痴
- 思考整理
- 顧客固有すぎる内容
```

処理全体を1つのエージェントへ任せるのではなく、候補選定・文書化・機密確認の段階に分け、中間生成物を段階ごとに確認できるようにしています。

## 実際に生成されたもの

今回の試作では、3週間の1人分のセッション履歴から9件のユースケース候補が生成されました。

![9件のユースケース候補を追加したGitHub Pull Request](https://static.zenn.studio/user-upload/308e8cee5361-20260622.png)

各文書には「実行目的」「主要な成果物」「ユーザー入力例」「大まかな進め方」「使用したSkillやSub Agent」が含まれます。

例えば「LLM抽出ロジックの完全置換方式への設計変更」では、次のような文書が生成されました。

```
# LLM抽出ロジックの完全置換方式への設計変更

## 実行目的

LLMを用いた情報抽出処理において、過去データとの差分更新方式を
採用していたが、配列の特定要素追加やオブジェクトの一部更新に
対応できない問題があった。

設計方針の議論を経て、LLMに常に完成形を出力させ、
全件を完全置換する方式へ移行する。
```

これは作成者が確認した範囲では、単なる作業手順ではなく設計上の選択を含んでおり、9件の中では比較的共有候補になり得る内容でした。

一方、「PydanticのDiscriminated Unionへの設計変更」では次のような内容でした。

```
1. 既存のデータ型定義と利用箇所を調査する
2. Discriminated Union方式でモデルを設計する
3. 既存JSONとテストファイルを更新する
4. テストを実行する
```

読みやすいものの、内容は一般的な移行手順に留まっており、この程度であればセッション履歴を参照しなくても作成できます。なぜ通常のUnionでは不十分だったのか、移行時に何が問題になったのかといった、実際の経験から得られる情報は残っていませんでした。

| 比較的共有価値があった例 | 作業記録に留まった例 |
| --- | --- |
| LLM出力の差分更新と完全置換の設計判断 | Discriminated Unionへ移行する一般的な手順 |
| 複数案の選択が含まれる | 調査、実装、テストという一般工程のみ |
| 他のLLMシステムにも転用可能性がある | セッション履歴由来の固有知見が残っていない |

このように、生成された文書には共有候補になり得るものと、一般化された作業記録に留まるものが混在していました。

## 試作を通じて見えた問題

### Claude Codeで行ったタスクが、すべて再利用可能な知識になるわけではない

初期実装では、高難易度や高頻度といったタスク側の特徴を、共有価値の手がかりとしていました。しかし今回の生成物では、他人が再利用できる情報が十分に残らない事例がありました。

再利用価値があるのは、規模が小さくても次のような情報を含む事例です。

* ユースケースが有効な条件
* AIへの依頼順序の工夫、人間が判断した箇所
* 再現可能な検証方法、別の環境では成立しない条件

### セッション、タスク、共有知識はそれぞれ別の単位だった

今回はClaude Codeのセッションを、タイムスタンプではなく「ユーザーが依頼したタスクのゴール」を基準にまとめることでユースケースとしました。

しかしタスクの単位でまとめるだけでも不十分でした。例えばシード設計、実装、テスト、Terraformの修正は同じ開発の流れに含まれますが、共有する知識としては別テーマです。反対に、別々のセッションでも「既存実装を調査し、設計判断の根拠として使う」という同じパターンの場合もあります。

つまり、次の3つは別々に考える必要があります。

1. コーディングエージェントのセッション単位
2. 利用者が行ったタスクの単位
3. 他人が再利用する知識の単位

### セッション履歴だけでは、共有価値を判断する情報が足りない

生成された文書に判断理由や適用条件が残らなかった原因の一部は、文書生成時の指示だけではなく、そもそも元のセッション履歴に必要な情報が含まれていないことでした。

セッション履歴では、次のような情報は明示されないことがあります。

* 削減できた時間数
* ユースケースが有効である条件
* 作成者が得た注意点や反省

これらは完成したユースケース文書を履歴だけから自動生成するのではなく、まず下書きを生成し、不足している情報だけを投稿者へ質問して補う方が現実的だと考えています。

## 現在の限界

今回の試作を通じて様々な限界が見えてきました。以下はその一例です。

* **生成物の価値を第三者が評価していない**: 作成者が妥当に見えることと、第三者が再利用できることは別です。
* **手書きより負荷が小さいか確認していない**: 生成後の確認・機密修正・表現調整まで含めて速いかは計測していません。
* **機密検出の精度を評価していない**: 意図的に人名・URL・案件名・認証情報を含めたテストデータでの評価は未実施です。
* **出力形式に揺れがある**: カテゴリ名の日英混在、ID表記の不統一、タグ粒度の差などがあり、スキーマ検証や正規化処理が必要です。
* **収集できない環境がある**: devcontainerなど履歴がローカルに残らない環境では収集できません。

## まとめ

今回のコンテストでは、Claude Codeのセッション履歴から共有候補となる文書を生成し、GitHubのPull Requestとして出力するSkillを試作しました。1人分、3週間の履歴から9件を生成できましたが、その多くは再利用可能な知見ではなく、一般化された作業記録に留まりました。

試作を通じて分かったのは、履歴から読みやすい文章を作ることと、組織で再利用できるナレッジを作ることは別の問題だということです。

実用化には、共有価値のある内容を選別し、知識の単位へ整理したうえで、履歴にない判断理由や適用条件を投稿者に補ってもらう必要があります。今回の試作は、AI活用の知見を自動で集める仕組みそのものよりも、どこまでを自動化し、どこを人間に残すべきかを考えるきっかけになりました。
