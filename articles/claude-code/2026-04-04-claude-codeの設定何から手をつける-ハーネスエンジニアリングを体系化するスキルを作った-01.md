## 「ハーネスエンジニアリング」って、何から始めればいいの？

Claude Codeを使い込んでいくと、ある時点で**ハーネスエンジニアリング**という概念に出会います。CLAUDE.md、hooks、permissions、memory、subagents——Claudeの動作を制御する設定群を総称して「ハーネス」と呼び、それを体系的に設計・運用することをハーネスエンジニアリングと言います。

### なぜハーネスエンジニアリングが必要なのか

Claude Codeは素の状態でも十分に賢いです。でも、使い続けていると気づきます——**同じミスを何度も繰り返す**。フォーマットが崩れる、触ってほしくないファイルを編集する、lintの設定を勝手に緩める、`--no-verify`でgit hookをスキップする。

毎回「それやめて」と言えば直りますが、次のセッションではまた同じことが起きます。CLAUDE.mdに書いても、指示が増えるほど守られなくなる。これは「助言」の限界です。

ハーネスエンジニアリングは、この\*\*「毎回言い聞かせる」を「仕組みで防ぐ」に変える\*\*アプローチです。hookで確定的に制御し、permissionで危険な操作をブロックし、memoryで学習を永続化する。プロンプトの工夫ではなく、環境の設計でエージェントの品質を担保します。

### でも、壁がある

「やらなきゃ」とは思う。でも、こんな壁にぶつかりませんか？

* **何から手をつければいいのか分からない。** CLAUDE.mdを書くのは分かった。でもhooksは？permissionsは？memoryは？全部やるの？
* **今の自分のプロジェクトが正しい状態なのか分からない。** 設定はいくつか入れた。でもそれが十分なのか、何か抜け漏れがあるのか、判断基準がない。
* **「ベストプラクティス」が散在していて全体像が見えない。** 公式ドキュメント、コミュニティの記事、Xのポスト——情報はあるが、体系化されていない。

この記事では、これらの悩みを解決するために作ったスキル **[harness-audit](https://github.com/kosk-t/harness-audit)** を紹介します。Claude Code環境を8つの観点から診断・スコアリングし、「次に何をすべきか」を優先度付きで教えてくれるスキルです。

**対象読者**: Claude Codeを日常的に使っていて、設定をもっと整えたい・最適化したいと感じている開発者

## harness-auditの概要

```
audit my setup → 8次元の診断 → スコアリング → 優先度付きの改善提案 → 対話的に修正
```

「audit my setup」「improve my config」「harness」などのフレーズで自動的にトリガーされ、現在のプロジェクトの設定状態を評価します。

## 設計を支える7つの原則

スキルの診断基準とレコメンデーションは、公式ドキュメント、コミュニティの知見（HumanLayer、revfactory/harness、everything-claude-code等）、そして実体験から導き出した7つの原則に基づいています。

### P1: Instruction Budget — コンテキストは有限

CLAUDE.mdは**200行以内**に抑えるべきです。ETH Zurichの研究では、指示を詰め込みすぎるとトークンコストが20%以上増加する一方、タスク解決率は改善しないことが示されています。

深い情報が必要なら `.claude/rules/` や `@import` で段階的に読み込ませます。

### P2: Silent Success, Verbose Failure — 成功は黙って、失敗だけ喋る

Hookやテストの出力は**成功時は抑制し、エラー時のみ表示**すべきです。フルテストスイートの出力がコンテキストに流れ込むと、エージェントが迷走する原因になります。

### P3: Deterministic Over Advisory — 確定的な制御を優先

CLAUDE.mdの指示は「助言」であり、モデルが無視する可能性があります。一方、Hooksは**100%発火する確定的な制御**です。フォーマット、危険コマンドのブロック、ファイル保護など「絶対に守らせたいルール」はHooksで実装します。

### P4: Grow From Failure — 理論ではなく失敗から育てる

セキュリティのベースライン（deny rules、基本的なフォーマット）以外は、**実際にエージェントがミスを犯してから設定を追加**します。「念のため」で大量のhooksやskillsを入れるのはP1違反です。

### P5: Context Isolation Over Role Specialization

サブエージェントは「フロントエンドエンジニア」「バックエンドエンジニア」のような**役割分担ではなく、コンテキスト隔離**のために使います。親のコンテキストウィンドウを汚さないことが価値です。

### P6: Why Over ALWAYS/NEVER — 理由を書く

「ALWAYS use X」「NEVER do Y」のような絶対ルールは脆弱です。**理由を説明する**（「pdfplumberを使う。PyPDF2は行列構造を保持しないため」）ことで、LLMがエッジケースに対応できます。

### P7: Deny-First Security — まず拒否

Permission deny rulesは**常に勝ちます**。bypassPermissionsモードでも、hookが"allow"を返しても、denyが優先されます。安全なデフォルトは「危険な操作を明示的に拒否し、安全な操作だけ許可する」です。

## 8次元の診断

harness-auditは以下の8つの観点からClaude Code環境を0〜3点で評価し、合計24点満点でスコアリングします。

| # | 次元 | 何を見るか |
| --- | --- | --- |
| 1 | **Project Instructions** | CLAUDE.md の存在・行数・具体性、`.claude/rules/`、`@import`、`CLAUDE.local.md` |
| 2 | **Automation (Hooks)** | PostToolUseの自動フォーマット、PreToolUseのガード、secret scanner、`--no-verify`ブロック、Stopフックのループガード |
| 3 | **Workflow** | カスタムコマンド、スキル、`allowed-tools`、`disable-model-invocation`、MCP統合 |
| 4 | **Context Persistence** | Memory の 4 タイプ（user, feedback, project, reference）のカバー率と品質 |
| 5 | **Agent Design** | サブエージェントの目的（コンテキスト隔離か？）、tool制限、maxTurns、モデルルーティング |
| 6 | **Access Control** | deny rules、危険なワイルドカード（`rm:*`, `curl:*`等）、チームと個人の設定分離 |
| 7 | **External Integration** | MCP、git hooks、CI/CD統合 |
| 8 | **Documentation** | `@import`の活用、README vs CLAUDE.md の役割分離、オンボーディング完全性 |

### スコアの意味

各次元の0〜3は一貫した基準です。

* **0**: 設定なし
* **1**: 存在するが不十分（曖昧、制限なし等）
* **2**: 実用的に機能している（具体的な設定、ガードあり）
* **3**: 最適化されている（モジュラー設計、least-privilege、継続改善の仕組み）

### 成熟度レベル

合計スコアは5段階の成熟度にマッピングされます。

| レベル | スコア | 状態 |
| --- | --- | --- |
| L0 | 0-3 | **Raw** — ハーネス設定なし |
| L1 | 4-8 | **Foundation** — 基本的な指示が存在 |
| L2 | 9-14 | **Structured** — 自動化とワークフローが出現 |
| L3 | 15-19 | **Mature** — 包括的な品質ゲート |
| L4 | 20-24 | **Optimized** — 完全にエンジニアリングされ、継続的に改善 |

## 影響度の評価フレームワーク

単に「設定がない」と指摘するだけでは不十分です。harness-auditは各発見事項を**3つの軸**で評価し、優先度を決定します。

| 軸 | 質問 | スケール |
| --- | --- | --- |
| **頻度** | どのくらいの頻度で影響するか？ | every-session / weekly / rare |
| **深刻度** | 結果はどの程度深刻か？ | data-loss / quality-degradation / inconvenience |
| **修正コスト** | 修正にどれだけの労力がかかるか？ | minutes / hours / days |

これにより、例えば「毎セッション発生 × データ損失リスク × 数分で修正可能」はP0（今すぐ修正）、「まれに発生 × 不便レベル × 時間がかかる」はP3（余裕があれば）と分類されます。

## ハーネストリアージ表 — ミスをどこで防ぐか

診断結果をもとに改善する際、**どのハーネス層に対策を仕込むか**を判断する基準表も組み込んでいます。

| ミスの性質 | 仕込む先 | 理由 |
| --- | --- | --- |
| 毎回やらかす（フォーマット、import順） | Hook (PostToolUse) | 100%発火、思考不要で矯正 |
| やったら取り返しつかない（secret, rm -rf） | Hook (PreToolUse) + Permission deny | 実行前ブロック |
| 判断を間違える（設計方針、命名） | CLAUDE.md / `.claude/rules/` | 方針を明示。Why で汎化 |
| 毎回同じ指摘をしている | Memory (feedback) | 次セッションから自動適用 |
| 特定ファイル群でだけ起きる | `.claude/rules/` with `paths:` | 対象ファイルにだけルール適用 |
| 設定を弱める方向に逃げる | Hook (PreToolUse) でブロック | エージェントは設定を弱める傾向がある |
| git hooks をスキップする | Hook + Permission deny | 品質ゲートの迂回を防ぐ |
| コンテキスト汚染で迷走 | Subagent | 調査を分離、メインを汚さない |

この表は、Claude Codeの設定体系全体を「ミスの性質からの逆引き」として整理したもので、harness-auditの改善提案の根拠になっています。

## 実際の使い方

### 導入方法

Claude Codeのプラグインとしてインストールできます。

**Claude Code上でインストール：**

```
/plugin marketplace add kosk-t/harness-audit
/plugin install harness-audit@kosk-t-harness-audit
```

**または `.claude/settings.json` に直接追加（チームで共有可能）：**

```
{
  "extraKnownMarketplaces": {
    "kosk-harness": {
      "source": {
        "source": "github",
        "repo": "kosk-t/harness-audit"
      }
    }
  },
  "enabledPlugins": {
    "harness-audit@kosk-harness": true
  }
}
```

### 基本的な使い方

これだけで、現在のプロジェクトのClaude Code環境を全8次元で診断し、レポートを出力します。

### 特定の領域に絞る

```
> harness hooks
> check my CLAUDE.md
> fix permissions
```

特定の次元を指定すると、フル診断をスキップしてその領域を深掘りします。

### 出力されるレポートの例

```
## Harness Maturity Report

**Overall: L1 — Foundation** (7/24)

| Dimension          | Score | Findings                    | Impact               |
|--------------------|-------|-----------------------------|----------------------|
| Project Instructions | 2/3  | CLAUDE.md 145行、具体的     | —                    |
| Automation          | 0/3  | Hooks未設定                 | every-session × quality = P0 |
| Workflow            | 1/3  | コマンド2つ、frontmatter不足 | weekly × inconvenience = P2 |
| Access Control      | 0/3  | deny rules なし             | every-session × data-loss = P0 |
| ...                 | ...  | ...                         | ...                  |

### P0 — Fix now
- **Permission deny rules を追加** — `rm -rf`、`curl`、`--no-verify` が無制限。
  Fix: settings.json に deny rules を追加。

### P1 — Fix soon
- **PostToolUse フォーマッターを設定** — 編集のたびにフォーマットが崩れる。
  Fix: prettier の PostToolUse hook を追加。
```

レポート出力後は「どの領域から改善しますか？」と対話的に進めます。選んだ領域の設定変更を提案し、確認を得てから実装します。

## 組み込みのHookレシピ

harness-auditには、すぐにコピー&ペーストで使えるHookレシピが組み込まれています。

### 編集後の自動フォーマット

```
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write 2>/dev/null; exit 0"
      }]
    }]
  }
}
```

### 保護ファイルの編集ブロック

`.env`、lockfile、linter設定などへの編集を防ぎます。

```
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "if": "Edit(.env*)|Write(.env*)|Edit(*.lock)|Write(*.lock)|Edit(.eslintrc*)|Write(.eslintrc*)",
        "command": "echo 'Blocked: protected file' >&2; exit 2"
      }]
    }]
  }
}
```

### Secretスキャナー

書き込み内容にAWSキー、GitHubトークン、秘密鍵などが含まれていないかチェックします。

```
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "INPUT=$(cat); echo \"$INPUT\" | jq -r '.tool_input.content // .tool_input.new_string // empty' | grep -qEi '(AKIA[0-9A-Z]{16}|ghp_[a-zA-Z0-9]{36}|-----BEGIN (RSA |EC )?PRIVATE KEY|sk-[a-zA-Z0-9]{20,})' && { echo 'Blocked: potential secret detected' >&2; exit 2; }; exit 0"
      }]
    }]
  }
}
```

### Permissionの安全なデフォルト

```
{
  "permissions": {
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(curl:*)",
      "Bash(wget:*)",
      "Bash(kill -9:*)",
      "Bash(*--no-verify*)"
    ],
    "allow": [
      "Bash(npm run *)",
      "Bash(npm test)",
      "Bash(git:*)",
      "Bash(ls:*)"
    ]
  }
}
```

## スキルの設計で工夫したこと

### 原則ベースの診断基準

スコアリング基準をチェックリストではなく**原則ベース**にしています。「Hookがあるか？」だけでなく、「P3（確定的制御の優先）に基づいて、CLAUDE.mdの助言ではなくHookで制御すべきものがHookになっているか？」と評価します。

これにより、形式的に設定が存在するだけでは高スコアにならず、実質的に機能している設定が評価されます。

### 影響度ベースの優先順位

「設定がない＝悪い」ではなく、**頻度 × 深刻度 × 修正コスト**で優先度を決めます。毎セッション発生するフォーマット崩れ（P0）は、滅多に起きないCI/CD連携の不足（P3）より先に修正すべきです。

### 対話的な改善フロー

レポートを出して終わりではなく、「どこから改善しますか？」と聞いて、選ばれた領域を実際に修正するところまでがスキルの範囲です。確認してから書き込むので、意図しない変更は起きません。

### アンチパターンの明示

各次元でアンチパターンを明記しています。例えば：

* `Bash(rm:*)` を allow に入れている → `rm -rf /` が実行可能になる
* Stop hookに `stop_hook_active` チェックがない → 無限ループ
* PostToolUseで「やり直し」を試みる → 遅すぎる、PreToolUseでブロックすべき

「なぜダメなのか」と「具体的にどうなるか」をセットで記載することで、スキル自身がP6（Why Over ALWAYS/NEVER）を体現しています。

## まとめ

harness-auditは、Claude Code環境の設定を「なんとなく」ではなく「体系的に」整えるためのスキルです。

* **7つの原則**に基づく一貫した診断基準
* **8次元 × 0-3点**の定量的スコアリング
* **頻度 × 深刻度 × 修正コスト**による優先度付け
* **対話的な改善フロー**で実際に修正まで完了

「とりあえずCLAUDE.mdは書いたけど、他に何をすればいいか分からない」という段階から、Claude Code環境を段階的に成熟させていくためのガイドとして活用できます。
