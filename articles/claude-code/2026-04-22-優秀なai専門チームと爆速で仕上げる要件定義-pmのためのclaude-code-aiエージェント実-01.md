## 1. はじめに

前回の記事では、Claude CodeでPM業務がどう変わったかを **3つの仕事** を軸に紹介しました。

> [PMが本来やるべき仕事に時間を取り戻す — Claude Codeで変わった3つのPM業務](https://zenn.dev/medirom_tech/articles/4a61491a6b5059)

前回は3つの仕事の全体像を俯瞰する内容でした。3つどれも重要ですが、今回はまず1つ目の **要件定義・仕様化** にフォーカスして、詳しく解説していきます。

業界でよく言われる経験則として「品質は要件定義で大きく決まる」と語られるほど、ここはPMの一丁目一番地。サボると後工程すべてが歪みます。  
だからこそ時間をかけたい。でも現実には、**書く作業と調べる作業に時間を奪われて、肝心の「判断」に集中できない** ——そんなPMの方に向けた記事です。

この記事のゴールは2つです：

* **ワークフローの考え方**：AIとPMの役割分担をどう設計するか
* **自社で実践するための道筋**：明日から何を始めればいいか

> **補足：本記事での「Design Doc」について**  
> 弊社では従来型の要件定義書ではなく、**Design Doc（デザインドック）** を作成しています。背景・課題・目的・解決策・技術的な考慮点などを1つの文書にまとめるフォーマットで、関係者間の認識合わせと意思決定の記録を兼ねるものです。本記事ではDesign Docを例に説明しますが、考え方は要件定義書やPRDなど他のフォーマットにも応用できます。
>
> **Design Doc そのものについて詳しく知りたい方へ**: [プロダクトマネージャーの必須スキル：デザインドックの書き方（note / kosuke mori）](https://note.com/kosukemori/n/n968cd16c53eb)

## 2. 要件定義の「本当のつらさ」はどこにあるか

PMなら誰でも覚えがあるはずです。

* **白紙から書き始めるのがつらい**。カーソルが点滅する画面を前に、何から書けばいいか分からない
* **書いたと思ったら漏れだらけ**。レビューで「この観点は？」と指摘され、また書き直し
* **用語がブレて関係者と認識ずれ**。同じ機能を違う名前で呼んでいて、会議が空転する
* **エビデンスを探すのに時間が溶ける**。「この方式は本当に有効なのか」を裏付ける論文や業界レポートを探し回る

でも、よく考えてみてください。  
このつらさの正体は **「判断」そのもの** ではなく、**「判断の前後にある作業」** です。

テンプレートに沿って構造化する。漏れがないかチェックする。関連情報を集めて整理する。用語を統一する——これらは **PMの判断力がなくても、手順さえあればできる作業** です。

**ここをAIエージェントに任せる。** それが本記事のテーマです。

## 3. PMは「Why」を伝えるだけ — Claudeが専門チームを編成する

前回の記事ではSkill・Sub Agent・Pluginsという3つの機能を紹介しました。  
今回は要件定義に直接関わる **SkillとSub Agent** に絞って、もう少し詳しく見ていきます。

### SkillとSub Agentの違い

この土台を踏まえて、2つの違いを押さえておきましょう。

| 機能 | 会社にたとえると | 要件定義での役割 |
| --- | --- | --- |
| **Skill** | 業務マニュアル＋専門知識集 | Claudeの能力を拡張する再利用可能な指示セット。手順の型化だけでなく、自社の規約やドメイン知識を覚えさせることもできる。`/コマンド名`で直接呼び出すことも可能 |
| **Sub Agent** | 専門職の部下 | 特定の専門タスクを独立して実行する。調査、レビュー、文書作成など |

Skillが"どう進めるか"と"何を知っているか"を拡張し、Sub Agentが"専門作業"を担当する。  
PMは「何を作りたいか（Why）」を伝えるだけです。

### 要件定義で活躍するメンバーたち

私が要件定義用に **自作した** SkillとSub Agentの組み合わせはこうなっています（これらはClaude Codeに最初から入っているものではなく、自分で定義ファイルを作成したものです）：

| 種類 | 具体例 | 何をしてくれるか |
| --- | --- | --- |
| Skill | **Design Doc作成スキル** | ヒアリング→テンプレ適用→セルフレビューの一連の手順をClaudeに指示 |
| Sub Agent | **research-analyst** | 論文・業界レポートの下調べと要約 |
| Sub Agent | **critical-reviewer** | 要件の漏れ・矛盾・曖昧な記述の自動チェック |
| Sub Agent | **document-writer** | 要件定義書・Design Docの構造化された文書を作成 |
| Sub Agent | **customer-perspective-analyst** | 想定顧客になりきり、要件をユーザー目線で評価・フィードバック |

たとえば「この機能のDesign Docを書いて」と頼むと、Claudeがこんな流れを組み立ててくれます（各Sub Agentの説明文（description）を適切に書いておくことで、Claudeが自動で判断します）：

1. Design Doc作成スキルの手順に従ってヒアリング開始
2. Claudeがresearch-analystを呼び出し、論文調査を依頼
3. Claudeがdocument-writerを呼び出し、構造化された文書を作成させる
4. Claudeがcustomer-perspective-analystを呼び出し、「ユーザーから見てどうか？」を確認
5. Claudeがcritical-reviewerを呼び出し、漏れ・矛盾をレビューさせる

※ Sub Agent同士が直接やりとりするのではなく、Claudeが司令塔として各エージェントを順に呼び出す形です。

**PMの感覚としては、「優秀な専門チームに要件定義を依頼したら、叩き台とレビュー結果が返ってきた」** というイメージです。

具体的なSkillやSub Agentの作成方法は、**セクション7「まず試してみる」** でステップごとに解説します。

## 4. Claude Codeで変わる要件定義ワークフロー

では、実際に要件定義はどう変わるのか。まずBefore/Afterで全体像を見てから、3つのステップで具体的に解説します。

### Before / After

| 観点 | Before | After |
| --- | --- | --- |
| 初稿作成 | 白紙から書く（半日〜1日） | テンプレ＋AI生成（1〜2時間） |
| 網羅性チェック | 自分で漏れを確認 | critical-reviewerが自動チェック |
| エビデンス調査 | 論文・規格を自分で当たる | research-analystが要約 |
| 用語の一貫性 | 手動で統一 | セルフレビューで自動検出 |
| ユーザー視点の検証 | 想像で補完 or 後回し | customer-perspective-analystが即座にフィードバック |

**チームのテンプレートと前提情報が揃っていれば** ここまで縮みますが、ゼロから整える場合は当然もっとかかります。

### 3つのステップ

具体的なワークフローは、大きく3ステップです。

**Step 1：ヒアリング — AIが聞き、PMが答える**

Claudeが背景・課題・目的・スコープ・非機能要件を対話的に確認してきます。  
PMは質問に答えるだけ。「白紙から何を書くか考える」苦しみがなくなり、**対話の中で自分の考えが整理されていく** 感覚です。

ただし、この対話型ヒアリングを実現するにはひと工夫必要です。方法は2つあります：

* **自作Skillに「まず質問してから進めること」と指示を書く** — 「いきなり書き始めず、まず以下の観点を1つずつ確認してから叩き台を作れ」とSkillに定義しておく
* **[superpowers プラグイン](https://github.com/obra/superpowers)を導入する** — Claude Code公式マーケットプレイスに採用されているプラグインで、brainstorming（ブレスト）スキルが含まれています。1つずつ質問しながらアイデアを固め、設計に落とし込む対話フローが最初から組み込まれているので、自作しなくてもヒアリング→設計の流れを体験できます。superpowersはコード生成の文脈で紹介されることが多いですが、brainstormingやplanning（計画立案）のスキルはドキュメント作成にもそのまま使えます

まずはsuperpowersプラグインで対話型のフローを体験し、慣れてきたら自社のテンプレートに合わせたSkillに発展させていく——という進め方がおすすめです。

**superpowersプラグインについて詳しく知りたい方へ**:

*PMが判断すること：何を作るのか（Why）、誰のどんな課題を解決するのか、やらないことは何か*

**Step 2：叩き台生成 — 8割完成した状態からスタート**

ヒアリング結果をもとに、自社テンプレートに沿ったDesign Docが生成されます。  
必要に応じてresearch-analystが論文や業界レポートを下調べし、document-writerが構造化された文書に仕上げてくれます。

「白紙から書く」のと「8割完成した叩き台を編集する」のでは、負担がまるで違います。

*PMが判断すること：生成された内容の取捨選択、優先度の決定、プロジェクト固有の制約の追記*

**Step 3：セルフレビュー — 漏れと矛盾を自動検出**

critical-reviewerが要件の漏れ・矛盾・曖昧な記述を指摘します。  
さらにcustomer-perspective-analystが想定顧客の目線で「この要件、ユーザーにとって本当に価値があるか？」を評価してくれます。

PMは指摘を見て「直すか、このままでいいか」を判断するだけです。

*PMが判断すること：指摘への対応方針、エビデンスの一次情報の裏取り、最終的な品質判断*

## 5. 「情報の集約」と「自社Skill」が品質を決める

ワークフローを回すだけでは、**どのプロジェクトでも通用しそうな一般論** しか出てきません。  
AIの出力品質を実用レベルに引き上げるには、2つの仕込みが必要です。

### 5-1. プロジェクトフォルダに情報を集約する

Claude Codeはローカルフォルダのファイルを読めます。  
つまり、同じプロジェクトフォルダに関連情報を置いておくだけで、AIが参照できるコンテキストになるのです。

置くべきものの例：

* 過去の要件定義書・Design Doc — 前回の要件との整合性を自動で確認できる
* システムの仕様書・アーキテクチャ図 — 技術的制約を踏まえた要件が書ける
* 用語集・ドメイン知識 — 用語の一貫性が保たれる
* 過去の議事録・決定事項 — 「なぜこの仕様になったか」の経緯を反映できる

フォルダにコンテキストが増えるほど、AIの出力がプロジェクト固有の文脈を踏まえたものになります。  
PMの感覚で言えば、記憶力のある優秀なチームに、過去の経緯をすべて共有してある状態です。新しいメンバーに一から説明する手間がない。それだけで、アウトプットの質が変わります。

### 5-2. 自社フォーマットのSkillを作るべき理由

汎用のAIに「要件定義書を書いて」と頼んでも、出てくるのは教科書的なテンプレートです。  
自社の要件定義書やDesign Docのフォーマットに合わせたSkillを作ることで、初めて実用レベルになります。

Skillに込めるべきもの：

* 自社テンプレートの構造 — セクション構成、必須項目、記載粒度
* 過去の失敗から学んだチェック観点 — 「この観点が漏れると後工程で必ず手戻りになる」
* プロジェクト固有の用語・制約 — 業界特有の概念、社内用語の定義

PMの暗黙知が形式知に変換され、組織の知見がツールに蓄積されていく。これが自社Skillの最大の価値です。

会社ごとに要件定義書のフォーマットは違います。チェック観点も違います。だからこそ、**自社のやり方に合わせたSkillを自分たちで作る**ことが重要なのです。具体的な作り方はセクション7で解説します。

## 6. 失敗から学んだ、AIとの要件定義で気をつけること

正直に言うと、最初からうまくいったわけではありません。

**最初のAI生成Design Docは、形だけ整って中身スカスカでした。**  
セクションは綺麗に並んでいるのに、中身はどのプロジェクトでも通用しそうな一般論ばかり。プロジェクト固有の制約も、過去のステークホルダー合意も反映されていませんでした。

さらに危なかったのが、**「もっともらしい数字」の罠** です。引用された数値が裏取りなしの"それっぽいエビデンス"だった、という場面がありました。気づかずにそのまま通していたらと思うとぞっとします。

もう1つ。**PMがレビューをサボると、AIの叩き台がそのまま最終成果物になってしまう。** 後工程で「この要件、何を根拠に決めたの？」と聞かれて答えられない——そんな手戻りも経験しました。

これらの失敗から学んだのが、次の3つです。

### 1. 「Why」の解像度がAIの出力品質を決める

誰のどんな課題か。成功基準は何か。やらないことは何か。  
PMの意思決定の質 ＝ AIの成果物の質です。曖昧な指示からは曖昧な要件しか出てきません。

### 2. エビデンスの一次情報は必ずPMが裏取りする

AIは下調べを爆速にしてくれますが、出てきた情報をそのまま信じるのは危険です。  
research-analystが出してきた論文や数値は、必ず一次情報に当たって確認する。これは鉄則です。

### 3. Skillは一度作って終わりではない

レビューで指摘された観点をテンプレに追加する。失敗した経験をチェック項目に変える。  
使うたびにSkillを磨き続けることで、組織の要件定義の質が底上げされていきます。

## 7. まず試してみる — 今日できる最初の一手

### Step 1：まず触ってみる

1. **Claude Codeをインストールする**（[公式クイックスタート](https://code.claude.com/docs/ja/quickstart)）
2. **プロジェクトフォルダで起動して、過去の要件定義書について質問してみる** — 「この要件定義書の課題を整理して」だけで、フォルダ内のファイルを読んで回答してくれます

「フォルダ内のファイルを読んで回答してくれる」体験が、最初の「おお」になるはずです。

### 知っておきたい：Claude Codeの「設定の土台」

SkillやSub Agentの作り方の話に入る前に、Claude Codeの設定がどこに置かれるかを簡単に説明します。

Claude Codeをプロジェクトフォルダで起動すると、そのフォルダ直下に **`.claude/`** というディレクトリが作られます。SkillやSub Agentの定義ファイルは、このディレクトリの中に配置します。

![](https://static.zenn.studio/user-upload/67ebe813bba1-20260415.png)

**`CLAUDE.md`** はプロジェクトのルートに置く特別なファイルで、「このプロジェクトではこういうルールで進める」「この用語はこう定義する」といった**プロジェクト全体の指示**をClaudeに伝えるものです。会社にたとえるなら「組織の憲法」や「プロジェクトの基本方針書」にあたります。

つまり全体像はこうなります：

| ファイル/ディレクトリ | 役割 | 会社にたとえると |
| --- | --- | --- |
| `CLAUDE.md` | プロジェクト全体のルール・方針 | 組織の憲法 |
| `.claude/skills/` | Skillの定義ファイル群 | 業務マニュアル棚 |
| `.claude/agents/` | Sub Agentの定義ファイル群 | 専門職の名簿と業務指示書 |

SkillもSub Agentも、実態は**Markdownファイル**です。特別なプログラミング知識は要りません。そして、これらのファイルも自分でゼロから書く必要はありません。**Claude Codeに「要件定義用のSkillを作って」「レビュー担当のSub Agentを作って」と頼めば、Claude自身が定義ファイルを生成してくれます。** PMは生成されたファイルを確認・修正するだけです。

### Step 2：自社用のSkillを作ってみる

慣れてきたら、自社テンプレートに合わせたSkillを作りましょう。自分でゼロから書く必要はありません。Claude Codeに頼めば一緒に作ってくれます。

たとえば、こう指示するだけです：

```
うちのDesign Docテンプレートに沿って要件定義を進めるSkillを作って。
テンプレートはこのフォルダの「テンプレート_DesignDoc.md」を参考にして。
```

Claudeが `.claude/skills/` 配下にSKILL.mdファイルを生成してくれます。生成された内容を確認して、必要に応じて修正すればOKです。

### Step 3：専門エージェントを追加する

Skillが動き始めたら、専門的な作業を担当するSub Agentを追加していきます。  
ここでもClaude Codeに相談するのが一番早いです。まず必要なエージェントの設計から一緒に考えてもらいましょう：

```
Design Docを作成するのに必要なSub Agentの案を考えて。
どんな専門家がいればDesign Docの品質が上がるか提案して。
```

Claudeが「調査担当」「レビュー担当」「顧客視点の評価担当」といったエージェント案を提案してくれます。納得したら、そのまま作成を依頼します：

```
提案してくれたSub Agentを全部作成して。
うちの過去の手戻り事例（レビュー指摘事項.md）も参考にして。
```

Claudeが `.claude/agents/` 配下に各エージェントの定義ファイルを生成してくれます。descriptionを適切に書いてあれば、次回以降Claudeが必要に応じて自動で呼び出してくれます。

このように、**Skill → Sub Agent と段階的にチームを育てていくイメージ**です。最初から完璧を目指す必要はありません。使いながら磨いていけば大丈夫です。

## 8. おわりに — 要件定義が変わると、チームが変わる

> **要件定義の「白紙からつらい」は、もう終わりにできます。**

要件定義の質が上がると、起きる変化は「PMが楽になる」だけではありません。

関係者との合意形成のスタート地点が変わります。叩き台の完成度が高いから、レビューの議論が「ここが足りない」ではなく「ここをどう判断するか」に変わる。後工程の手戻りが減り、チーム全体のスピードと信頼が積み上がっていきます。

PMの仕事は変わりません。「Why」を決め、判断し、責任を持つ。  
変わったのは、その前後にある作業を任せられるAIエージェントという「優秀な専門チーム」が手に入ったことです。
