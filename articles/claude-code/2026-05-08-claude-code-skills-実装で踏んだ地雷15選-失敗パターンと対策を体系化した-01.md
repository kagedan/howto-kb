## はじめに

「Claude Codeを使いこなせれば生産性10倍」という話を聞いて始めてみたら、最初の2週間は地雷原でした。この記事は、その地雷の場所をすべてマッピングした記録です。

Claude Code のカスタムスラッシュコマンド（Skills）は、一度うまく設計できれば非常に強力なツールです。しかし、公式ドキュメントには書かれていない「やってみると必ず引っかかる」パターンが存在します。本記事では、実際の実装経験から得た15の失敗パターンと、それぞれの対策を体系的にまとめた内容になる。

### この記事で扱う「失敗」の3分類

すべての失敗は次の3種類に分類できます。

| カテゴリ | 意味 | 典型症状 |
| --- | --- | --- |
| 動かない | 実行エラーが発生する | エラーメッセージ・Skillが呼び出されない |
| 意図と違う | 動くが期待した結果にならない | 誤った操作・的外れな出力 |
| 壊れやすい | 今は動くが設計に問題がある | 特定条件で失敗・スケールしない |

## カテゴリ1：プロンプト・指示設計の失敗（地雷 #1〜#5）

### 地雷 #1 ： Skill の説明文が曖昧すぎてClaudeが使ってくれない

**失敗例**

```
---
description: "ファイルを操作する"
---
```

**原因**

Claude は Skill の `description` を読んで「このSkillをいつ使うべきか」を判断します。説明が抽象的すぎると、呼び出しトリガーとなるシチュエーションが特定できず、Skillが無視されます。

**対策：「いつ使うべきか」を明示するテンプレート**

```
---
description: |
  ユーザーが「〇〇して」「〇〇を作成して」と依頼した場合に使用する。
  具体的には以下のケースで起動する:
  - ケース1: ○○のとき
  - ケース2: △△のとき
  このSkillは××には使わない（代わりに /○○-skill を使う）。
---
```

「使うべき状況」と「使わない状況」の両方を書くと、Claudeの判断精度が大幅に上がります。

---

### 地雷 #2 ： パラメータの型定義が緩すぎて誤動作する

**失敗例**

```
parameters:
  target_path:
    type: string
    description: "対象のパス"
```

`type: string` だけ定義して、あとはClaudeに任せると、相対パス・絶対パス・URLが混在して誤動作します。

**対策：`pattern` と `enum` で制約を追加する**

```
parameters:
  target_path:
    type: string
    description: "操作対象のファイルパス（絶対パス必須）"
    pattern: "^/"
  environment:
    type: string
    enum: ["development", "staging", "production"]
    description: "デプロイ先環境"
  required:
    - target_path
    - environment
```

バリデーション責務をClaudeに丸投げしない設計が基本です。

### 地雷 #3 ： System Prompt と Skill 定義が矛盾する

**失敗例**

`CLAUDE.md` (System Prompt)に「本番環境への直接操作禁止」と書いたのに、Skillのプロンプト内に「本番DBを直接更新する」手順が含まれている状態。

**原因**

設定が `CLAUDE.md` / `~/.claude/skills/` / `.claude/settings.json` に分散すると、整合性チェックが抜け落ちます。

**対策：Skill追加時の整合性チェックリスト**

新しいSkillを追加するたびに以下を確認します。

```
□ CLAUDE.md の「禁止事項」リストと矛盾していないか？
□ allowedTools の設定が CLAUDE.md の権限方針と一致するか？
□ Skill内の手順が既存Skillと重複・矛盾していないか？
□ 環境変数への依存が settings.json に宣言されているか？
```

### 地雷 #4 ： 「必ずこのSkillを使え」という強制が逆効果

**失敗例**

```
# CLAUDE.md

## 必須ルール
- ファイル操作は必ず /file-manager を使うこと（直接操作禁止）
```

**原因**

Claudeは自律的に最適なツールを選択しようとします。強制指示があると、その解釈に余計なトークンを消費し、かえって判断がぶれることがあります。

**対策：「推奨」と「強制」を使い分ける**

```
## Skill利用ガイドライン

### 推奨（状況に応じて判断）
- 複数ファイルの一括操作 → /file-manager を推奨
- セッション開始時 → /session-start を推奨

### 必須（例外なく実行）
- 本番環境への操作 → 必ず /deploy-safe を経由する（セーフガードが含まれるため）
```

「必須」にする理由を添えると、Claudeが意図を理解しやすくなります。

### 地雷 #5 ： マルチステップSkillで中間状態を考慮しない

**失敗例**

```
## 手順
1. リポジトリをクローンする
2. テストを実行する
3. ビルドする
4. デプロイする
5. Slackに通知する
```

ステップ2でテストが失敗しても、ステップ3以降を続行してしまいます。

**対策：チェックポイント設計パターン**

```
## 手順

### ステップ1: リポジトリのクローン
- 実行: `git clone <URL>`
- 成功確認: ディレクトリが存在するか確認
- ** 失敗時: 以降の手順を中断してエラーを報告する **

### ステップ2: テスト実行
- 実行: `npm test`
- 成功確認: exit code 0
- ** 失敗時: テスト結果を表示して中断。「デプロイしますか？」を確認しない **

### ステップ3: ビルド（ステップ2成功時のみ）
...
```

「失敗時の挙動」を各ステップに明記するのがポイントです。

## カテゴリ2：コンテキスト・メモリ管理の失敗（地雷 #6〜#9）

### 地雷 #6 ： 長大なコンテキストでSkillの説明が押し出される

**失敗例**

1時間の作業セッション後半で、序盤に使えていたSkillが「そのようなスキルは存在しない」と言われる。

**原因**

Claude Code のContext Windowには上限があります。会話が長くなるとTruncation（切り捨て）が発生し、最初に読み込んだSkillの定義情報が消えることがあります。

**対策**

1. **定期的なコンテキストリセット** ：`/clear` を活用して、長い作業は複数セッションに分割する
2. **CLAUDE.md の配置を後方寄りに** ：頻繁に参照する定義は会話の後半でも生きやすい位置に書く
3. **Skillの呼び出し頻度を記録** ：使用頻度の高いSkillはシンプルに保ち、コンテキスト消費を抑える

```
# コンテキスト消費量の目安（目安値）
CLAUDE.md全文    : 約 2,000〜5,000 トークン
各Skill定義      : 約 500〜2,000 トークン
コード1ファイル  : 約 1,000〜10,000 トークン（行数依存）
```

### 地雷 #7 ： Skill実行結果をコンテキストに残しすぎる

**失敗例**

「このリポジトリのすべてのファイルを分析して改善点を教えて」という処理を1 Skillで実行し、全ファイル内容をコンテキストに詰め込む設計。

**原因**

大量データを一度に渡すとToken消費が爆発し、後続の処理精度が下がります。

**対策：サマリー返却パターン**

```
## ファイル分析Skillの出力フォーマット

以下の形式でサマリーのみ返却する（ファイル全体をコンテキストに残さない）:

```json
{
  "analyzed_files": 42,
  "critical_issues": [
    {"file": "src/auth.ts", "issue": "SQLインジェクション脆弱性", "line": 128}
  ],
  "warnings": 7,
  "summary": "全体的に型定義が不足。特に src/api/ 以下が要対応"
}
```

詳細が必要な場合はファイル名を指定して追加で確認する。

```
「全部渡す」ではなく「必要な情報だけ渡す」設計に切り替えるだけでパフォーマンスが劇的に改善します。

### 地雷 #8 ： セッションをまたいだ状態管理を想定していない

**失敗例**
```

ユーザー: 「昨日作ったマイグレーションファイルを本番に適用して」  
Claude: 「どのファイルのことでしょうか？」

```
**原因**

Claude Code はステートレスです。新しいセッションでは前の会話内容を覚えていません。「前回の続き」を前提としたSkillは必ず失敗する。

**対策：外部状態ストア設計**

状態をファイルシステムやメモリファイルに外部化します。

```markdown
# ~/.claude/memory/session-state.md（自動生成）

## 最終セッション: 2026-04-06

### 作業中のタスク
- [ ] マイグレーション: 20260406_add_users_table.sql（未適用）

### 前回の作業パス
- `/Users/naoya/dev/projects/self/myapp/db/migrations/`

### 次のアクション
- 本番DBへのマイグレーション適用（`/deploy-migration` Skillを使用）
```

セッション終了時に状態を書き出し、次のセッション開始時に読み込む `session-start` / `session-end` Skillを実装すると、ステートレス問題を回避できます。

```
# ~/.claude/skills/session-start/SKILL.md

## 手順
1. `~/.claude/memory/session-state.md` を読み込む
2. 「前回の未完了タスク」と「次のアクション」を報告する
3. 今日の作業方針をユーザーに確認する
```

### 地雷 #9 ： 並列Skill実行での競合状態

**失敗例**

複数のSkillが同一ファイルに並行して書き込みを行い、内容が中途半端な状態で混在する。

```
# 問題が発生するパターン
Skill A: report.md の第1章を書き込み中...
Skill B: 同時に report.md の第2章を書き込み中...
# → 結果: ファイルが破損、または一方の内容が上書きされる
```

**対策：ロック機構と逐次実行の明示**

```
## 並列実行に関するルール

以下のSkillは同時実行を禁止する:
- /write-report と /update-report（同一ファイルへの書き込み）
- /deploy-staging と /deploy-production（環境の整合性）

逐次実行が必要な場合は明示的に記述する:
1. /generate-content を実行し、完了を確認する
2. 確認後、/publish-content を実行する
（2つのSkillを同時に起動しないこと）
```

また、一時ファイルを使ったロック機構を組み込む方法もあります。

```
# Bash Tool内でのロック実装例
LOCK_FILE="/tmp/skill-write.lock"

if [ -f "$LOCK_FILE" ]; then
  echo "ERROR: 別のSkillが書き込み中です。完了を待ってから再実行してください。"
  exit 1
fi

touch "$LOCK_FILE"
# ... 書き込み処理 ...
rm -f "$LOCK_FILE"
```

## カテゴリ3：環境・設定周りの失敗（地雷 #10〜#12）

### 地雷 #10 ： 環境変数が引き継がれない

**失敗例**

```
// Skillから呼び出したスクリプト内
const apiKey = process.env.API_KEY; // → undefined になる
```

**原因**

Claude Code のBash Toolはサブシェルで実行されます。ターミナルで設定した環境変数が引き継がれないケースがあります。

**対策：`.claude/settings.json` で明示的に宣言する**

```
{
  "env": {
    "API_KEY": "${API_KEY}",
    "DATABASE_URL": "${DATABASE_URL}",
    "NODE_ENV": "development"
  }
}
```

プロジェクトルートの `.claude/settings.json` に `env` セクションを追加することで、Skill実行時に確実に環境変数が引き継がれます。センシティブな値は直書きせず、シェル変数展開 (`${VAR_NAME}`) を使います。

### 地雷 #11 ： パス解決が実行環境によって変わる

**失敗例**

```
# Skillプロンプト内の指示
cd ./scripts && node deploy.js
```

ローカルでは動くのに、CI環境（GitHub Actions等）やリモートセッションで失敗する。

**原因**

相対パスは `cwd`（カレントディレクトリ）に依存します。Claude Codeのセッション開始位置が環境によって異なると、`./scripts` が指す場所が変わります。

**対策：絶対パスを基準にする**

```
## パス解決ルール

Skill内でファイルパスを指定する場合は以下の優先順位で解決する:

1. ** 環境変数から取得 **:
   ```bash
   PROJECT_ROOT="${PROJECT_ROOT:-/Users/naoya/dev/projects/self/myapp}"
   node "${PROJECT_ROOT}/scripts/deploy.js"
```

2. **git root を基準にする**:

   ```
   GIT_ROOT=$(git rev-parse --show-toplevel)
   node "${GIT_ROOT}/scripts/deploy.js"
   ```
3. **相対パスは原則禁止** （`./` 始まりのパス指定はSkillで使わない）

```

### 地雷 #12 ： 権限設定（allowedTools）を緩くしすぎる

**失敗例**

```json
{
  "allowedTools": ["*"]
}
```

「とりあえず全部許可」でセットアップしたら、意図しないファイル削除やネットワークアクセスが発生した。

**対策：ユースケース別の最小権限設計**

```
// コードレビュー専用設定
{
  "allowedTools": [
    "Read",
    "Grep",
    "Glob",
    "Bash(git diff:*)",
    "Bash(git log:*)"
  ],
  "deniedTools": [
    "Write",
    "Edit",
    "Bash(rm:*)",
    "Bash(curl:*)"
  ]
}
```

```
// デプロイSkill専用設定
{
  "allowedTools": [
    "Bash(git push:*)",
    "Bash(npm run build:*)",
    "Bash(vercel deploy:*)"
  ]
}
```

最小権限の原則は、人間の権限設計と同じ考え方をAIエージェントにも適用するということです。

## カテゴリ4：設計・アーキテクチャの失敗（地雷 #13〜#15）

### 地雷 #13 ： 1つのSkillに責務を詰め込みすぎる

**失敗例**

```
# /mega-skill
## このSkillができること
1. 記事のリサーチ
2. 記事の執筆
3. 画像の生成
4. SNSへの投稿
5. アクセス解析レポートの生成
6. 次の記事テーマの提案
```

このSkillはデバッグが悪夢です。どのステップで問題が起きたか特定が難しく、一部だけ再実行することもできません。

**対策：単一責任原則でSkillを分割する**

```
/content-research   ← リサーチのみ
/content-write      ← 執筆のみ（リサーチ結果を引数で受け取る）
/content-publish    ← 公開のみ（記事ファイルパスを引数で受け取る）
/content-analyze    ← 分析のみ
```

分割の判断基準は「このSkillの目的を一文で説明できるか？」です。説明に「そして」が入ったら分割を検討します。

### 地雷 #14 ： テスト戦略を持たずに本番投入する

**失敗例**

「Claude の出力は非決定的だからテストできない」という誤解のもと、手動確認だけで複雑なSkillを本番運用する。

**現実** ：テストは可能です。

**対策：Skillのテスト3層**

**Layer 1：プロンプトのスモークテスト**

```
# 期待する出力に必要なキーワードが含まれているか検証
OUTPUT=$(claude -p "$(cat ~/.claude/skills/da-review/SKILL.md)" "test input")
echo "$OUTPUT" | grep -q "セキュリティ" && echo "PASS" || echo "FAIL"
```

**Layer 2：サンプル入力でのスナップショットテスト**

```
tests/skills/
├── da-review/
│   ├── input-1.md          ← テスト入力
│   ├── expected-1.md       ← 期待する出力の構造（キーワード・形式）
│   └── run.sh              ← テスト実行スクリプト
```

**Layer 3：ステージング環境での実運用テスト**

本番投入前に、同一構成のステージング環境でSkillを実行し、副作用（ファイル変更・API呼び出し等）を確認します。

### 地雷 #15 ： エラーメッセージがClaudeに伝わらない設計

**失敗例**

```
#!/bin/bash
# Skill から呼び出されるスクリプト

deploy() {
  # ... 処理 ...
  if [ $? -ne 0 ]; then
    exit 1  # ← エラー理由を何も返さない
  fi
}
```

Claudeはexit code 1を受け取っても「何が失敗したか」を理解できず、同じ操作を繰り返してループします。

**対策：エラーレスポンスの標準フォーマット**

```
#!/bin/bash

deploy() {
  BUILD_OUTPUT=$(npm run build 2>&1)
  if [ $? -ne 0 ]; then
    cat <<EOF
ERROR: ビルドが失敗しました

【失敗したコマンド】
npm run build

【エラー出力】
${BUILD_OUTPUT}

【推奨アクション】
1. 上記のエラーメッセージを確認する
2. TypeScriptの型エラーの場合は型定義を修正する
3. 依存関係エラーの場合は npm install を実行する

【再実行前に確認すること】
- src/ ディレクトリにビルドエラーがないか確認
- package.json の scripts.build が正しいか確認
EOF
    exit 1
  fi
}
```

「何が失敗したか」「なぜ失敗したか」「次に何をすべきか」の3点をエラーメッセージに含めると、Claudeが自律的に問題解決できる確率が上がります。

## 失敗を防ぐための設計チェックリスト

### Skill作成前チェックリスト

```
□ このSkillの目的を一文で説明できるか？
□ 既存のSkillと責務が重複していないか？
□ CLAUDE.md の制約と矛盾しないか？
□ テスト入力例を3つ以上用意できるか？
□ 失敗時の期待する挙動を定義したか？
```

### Skill実装中チェックリスト

```
□ description に「いつ使うべきか」と「いつ使わないか」を書いたか？
□ パラメータに enum / pattern / required を設定したか？
□ 各ステップに「成功確認」と「失敗時の挙動」を書いたか？
□ 環境変数への依存を settings.json に宣言したか？
□ パスの参照を絶対パス（または git root 相対）にしたか？
□ allowedTools を最小権限に絞ったか？
□ エラーレスポンスに「何が・なぜ・次に何を」を含めたか？
```

### Skill本番投入前チェックリスト

```
□ ステージング環境で実行テストを完了したか？
□ 並列実行時の競合リスクを評価したか？
□ コンテキスト消費量を見積もったか？（長時間セッションでも動くか）
□ セッションをまたいだ状態管理が必要な場合、外部ストアを設計したか？
□ 意図しない副作用（ファイル削除・外部API呼び出し等）がないか確認したか？
```

地雷を踏む前にこのリストを参照していれば、と思う失敗がいくつもありました。特に **地雷 #15（エラー設計）** と **地雷 #8（ステートレス問題）** は、発見するまでに多くの時間を使いました。

Claude Code Skills の強みは「一度うまく設計できれば、定型作業が自律的に動く」点にあります。その「うまく設計できれば」の部分を支えるのが、この15のパターンへの対策です。

地雷を知ってから設計することで、同じ失敗を繰り返さずに済みます。
