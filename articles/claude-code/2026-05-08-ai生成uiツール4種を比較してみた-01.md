## はじめに

AIでUIやWebサイトのプロトタイプを生成できるツールが増えてきました。

Claude Design、Claude Code + Frontend Design、Figma Make、v0 などは、いずれも自然言語からUI案を生成できるツールです。

一方で、AIにUIを作らせると、どこかで見たようなSaaS風UI、綺麗だけど無難なダッシュボード、グラデーションと角丸カードだらけの画面になりがちではないか、という感覚もあります。

そこで今回は、自分が運営している neonmap.world のリニューアル案を題材に、4つのAIツールに同条件で入力し、生成されたUIビジュアルを比較してみることにしました。

フランチャイズはもちろんのことですが、今までクリエイティブにエッジの効いていた分野である、ホテル、カフェやバーですらFacadeやinteriorの均質化が進んでいます。

もちろん、それ自体が悪いわけではないのですが、街の違い、店の違い、土地の違い、作り手の癖のようなものが、少しずつ見えにくくなっているような。旅先で知らない文化に出会いに行っているはずが、気づけば世界中に複製された「安心できるおしゃれ」の中で休憩している。それとも自分が歳とってただノスタルジーに浸ってるだけなのか・・・。

## ガラスネオンは、街の癖を記録するメディアだったのかもしれない

私はかれこれ10年以上、世界中のガラスネオンを見て、写真を撮ってきました。

現在はアーティストとして自分でガラスを曲げて製作も行っています。

🌩️🌩️🌩️[アート](https://instagram.com/neonxray)🌩️🌩️🌩️  
⚡⚡⚡[看板](https://www.instagram.com/hyakkiyako_neon/)⚡⚡⚡

アジア、アメリカ、ヨーロッパ、いろいろな街でネオンを見てきましたが、ガラスネオンには、その土地ごとの癖が強く残っていたように思います。

もちろん、ガラスネオンという素材自体は世界中にありましたし、ここで「ガラスネオンは均質化の反対側にある」と言ってる訳ではないです。ただ自分が慣れ親しんだ素材なので例として出しています。

発光しているガラス管、店名をかたどる文字、夜の街に浮かぶサイン。  
形式だけ見れば、どこの国にも似たようなネオンはあります。

でも、実際には、文字の形、typeface、ガラス管の曲げ方、色の選び方、トランスの位置、取り付け方、建物との距離、ネオンと他部分の組み合わせ、看板の劣化の仕方。そういう細部に、国や地域や店の個性が出ていた気がします。

同じ「OPEN」という文字でも、アメリカのロードサイドのネオンと、香港の路地裏のネオンと、ヨーロッパの古いバーのネオンでは、まったく違う顔をしている。その違いは、単なるデザインの違いというより、その街の速度や湿度、商売の仕方、人の欲望の出方の違いに近いものだったのかもしれない。

一方で、LEDサインの時代になると、どうしても見た目が均質化しているような。  
グローバルに共有された「Hipさ」のフォーマットが強くなっていっているこの時代に勃興したメディウムだからということも相まってか尚更そう見えてしまう。

もちろんLEDにも優れた表現はありますが、多くの場合、既製のモジュール、既製のフォント、既製の発光パターン、既製の施工方法に寄っていく気がします。

結果として、どこの国でも、どこの店でも、似たような光になっていく。  
（※あくまで個人の感想です）

ガラスネオンは、効率の悪い素材です。  
曲げる人の手癖が出るし、施工も面倒だし、壊れるし、個体差もあります。

でも、その非効率さ（手仕事）の中に、街ごとの違いが残っていたのではないかと思っています。

## neonmap.worldとは

[neonmap.world](https://neonmap.world) は、世界中の本物のガラスネオンを地図上に記録していくWebサービスです。

ユーザーは、街で見つけたネオンサインの写真を投稿し、場所情報とともに地図上に残すことができます。LEDが投稿されたら私が消します、スマセンw

単なる地図アプリというより、都市の夜、古い看板、街の記憶、消えつつある光の文化を記録するアーカイブとして育てていきたいと考えています。

大袈裟に言いましたが、旅行行った時あのネオンの前でtiktok撮りたいんだよねギャルからロケ地を探している映画監督、はたまた看板フェチのご老人まで老若男女適当に使って頂けるようになれば嬉しいです。

リリースしてから６年以上放置していますがそこから全くメンテしておらず、ずっと赤字を垂れ流しています。プロモーションも一切してこなかったのでユーザーも限られてますし、なんなら認証が壊れていて今FBログインすらできないんですが、まずはデザインからリニューアルしようと思いこれを書くに至ってます。

と、まあ前置きは以上にして今回のUIツール比較の本題に入っていこうと思います。

## なぜこの4つなのか

今回比較するのは以下の4つです。

| ツール | 今回の位置づけ |
| --- | --- |
| Claude Design | ビジュアル制作・プロトタイプ生成寄り |
| Claude Code + Frontend Design | コード生成前提だが、デザイン品質も重視するUI生成 |
| Figma Make | Figma上でのAIプロトタイプ生成 |
| v0 | Generative UI / Web UI生成 |

この4つは厳密には同じカテゴリのツールではありませんが、今回はこの4つを比較対象とします。

Lovable や Bolt などもUIを生成してくれますが、どちらかというとアプリケーション全体を生成するツールとしての性格が強いと感じたため、今回は対象から外しました。

今回はあくまで、「同じ条件を入力したときに、どのようなUIビジュアルが生成されるのか」を比較します。なのでコード品質、実装のしやすさ、保守性などは評価対象に含めないものとします。

## 実際に使用したプロンプト

現行の[neonmap.world](https://neonmap.world)にはすでにページ構成と基本機能があるため、今回はそのルーティング構成をベースにしつつ、UI / UX / 情報設計 / 世界観を再設計する方向でプロンプトを書きました。

実際に使用したプロンプト全文

```
Redesign the UI concept for neonmap.world.

Primary requirement:
The redesign must be fully responsive.
The current UI is not sufficiently optimized for mobile, so responsive design should be treated as one of the most important goals.
Create a design system that works across desktop, and mobile.
Do not design only for a large desktop screen.
Each page should have a clear mobile layout strategy, especially for browsing photos, navigating countries/cities, viewing neon detail records, and submitting a neon record.

neonmap.world is a global map archive of real glass neon signs.
Users can browse neon signs around the world, view photos, discover countries and cities, view individual neon records, and contribute their own findings.

This redesign should be based on the current product structure and routing.
You may significantly improve the visual design, layout, hierarchy, responsive behavior, and atmosphere, but keep the core page structure and core functions recognizable.

Current route structure to preserve conceptually:

- Homepage
  /

- Country Page
  /neons/<CountryName>
  e.g. /neons/Japan

- City / Region Page
  /neons/<CountryName>/<CityName>
  e.g. /neons/Thailand/Bangkok

- Neon Detail Page
  /np/<neon uuid>

- User Page
  /users/<uuid>

Important:
This is not a generic map app, social media app, or travel booking app.
It should feel like a cultural archive, a night-walking map, and a photographic field notebook for real glass neon signs.

Do not interpret “neon” as cyberpunk.
This project is about real glass neon signs in actual cities, not futuristic sci-fi aesthetics.

The visual world can be reimagined freely.
Do not feel bound to the current UI style.
However, the redesign should still be grounded in the existing information architecture and page roles.

Core concept:
- A global archive of real glass neon signs
- A map for walking through cities at night
- A record of disappearing urban light
- A field notebook connecting photography, place, typography, and street culture
- A service that makes people want to discover real neon signs in real locations

Important visual attitude:
- Nocturnal, but not cyberpunk
- Archival, but not academic
- Photographic, but not just a gallery
- Raw and tactile, but still usable
- Quiet, strange, memorable, and culturally grounded
- More like an urban visual archive than a startup product

Avoid:
- generic SaaS dashboard aesthetics
- blue-purple neon gradients
- glassmorphism
- cyberpunk clichés
- rounded cards everywhere
- generic map pins
- clean startup landing page style
- overly futuristic UI
- fake hologram aesthetics
- corporate travel app aesthetics

Main UX problems in the current UI that should be improved:
- The current UI is not sufficiently optimized for mobile and needs a fully responsive redesign
- On large desktop monitors, the layout feels too left-aligned and underuses screen space
- Information hierarchy is weak and pages feel visually empty
- Country and city pages are hard to scan and could better connect image browsing with navigation
- The product does not yet communicate a strong cultural or archival identity
- The action for opening Google Maps from a neon detail record is unclear and too hidden
- User contribution flow exists, but the posting / submission UI feels weak and not very intentional

Design goals:
- Improve clarity, hierarchy, and usability
- Make the UI work well across desktop, tablet, and mobile
- Better use wide desktop layouts without leaving huge empty areas
- Create a stronger sense of editorial direction and visual identity
- Make browsing feel like discovering real neon signs in the world
- Make location-related actions much clearer
- Keep the product feeling usable, not just atmospheric

Required pages:

1. Homepage
URL: /

The homepage should introduce neonmap.world as a global archive of real glass neon signs.
It should immediately communicate that this is not a generic map app.
It should feel like entering a nocturnal archive of city lights.

The homepage should include:
- a strong visual introduction to the project
- a clear explanation of what neonmap.world is
- a preview of global browsing
- entry points into countries, cities, and featured neon records
- featured photos or featured cities
- search
- a clear but not overly commercial call-to-action for contributing a neon record

What it should communicate:
- this is a living archive of real glass neon signs
- this is about cities, culture, memory, and place
- users can browse, discover, and contribute

Responsive behavior:
- On desktop, use the wide screen effectively with a strong visual layout
- On mobile, make the introduction, search, featured records, and contribution CTA easy to access
- Avoid hiding the core browsing experience behind too many menus

2. Country Page
URL pattern: /neons/<CountryName>

This page should show all neon signs recorded in a specific country.
It should feel like opening a country-level archive, not just a list.

The page should include:
- country title
- total number of recorded neon signs
- neon photo entries
- navigation to cities / regions within that country
- filtering or browsing support
- strong visual browsing of photos
- a layout that works well on large desktop screens
- a layout that works well on mobile screens

What it should communicate:
- this country has its own neon landscape
- there is local variation inside the country
- users can move deeper into cities or specific records

Responsive behavior:
- On desktop, balance photo browsing, country metadata, and city navigation
- On mobile, prioritize the country title, neon count, search/filter access, and scrollable photo entries
- Make country/city navigation easy to use without tiny text links

3. City / Region Page
URL pattern: /neons/<CountryName>/<CityName>

This page should focus on a city or region such as Bangkok or California.
It should feel like walking through a specific place at night and discovering neon signs one by one.

The page should include:
- city / region title
- neon entries in that location
- strong photo browsing
- location-aware structure
- nearby or related browsing if useful
- better balance between list / archive / navigation
- optional local map presence if it helps the concept

What it should communicate:
- this is a more local slice of the archive
- the user is exploring a place, not just scrolling content
- neon signs belong to neighborhoods, streets, and venues

Responsive behavior:
- On desktop, create a strong relationship between location, photos, and navigation
- On mobile, make it feel like a compact city guide or walking archive
- Make it easy to jump from a city page to individual neon records

4. Neon Detail Page
URL pattern: /np/<neon uuid>

This page is extremely important.
Each neon sign should be treated like an observed cultural object, not just an uploaded image.

The page should include:
- large photo display
- sign name or venue name
- city, country, and location information
- contributor / photographer credit
- description or notes
- visual hierarchy around metadata
- clear location section
- a very clear action for opening the place in Google Maps
- ideally a map preview, location card, or clearly labeled external map action
- nearby related neon records if useful

Important UX fix:
The current way of opening Google Maps from the neon detail page is too unclear.
In the redesign, location should be much easier to understand.
Do not hide it behind a vague tiny icon.
Use a clearly labeled action such as “Open in Google Maps” or a similarly obvious location CTA.

What it should communicate:
- this neon sign exists in a real place
- this is both a visual object and a documented location
- the page is useful both for inspiration and for actually visiting the place

Responsive behavior:
- On desktop, give the photo enough presence while making metadata and location actions clear
- On mobile, prioritize the image, title, location, and “Open in Google Maps” action
- The Google Maps action should be obvious and thumb-friendly on mobile

5. User Page
URL pattern: /users/<uuid>

This page should show a contributor’s archive.
It should feel less like a generic social profile and more like a personal record of sightings and contributions.

The page should include:
- contributor name / profile
- list or grid of submitted neon records
- contribution count
- optional map of places they documented
- contribution-related actions
- a clearer “Add Map” / “Register Neon” / upload flow

The posting flow currently exists as a modal.
You may redesign that flow, but keep the idea that users can submit a neon record with:
- location
- image
- description / notes

What it should communicate:
- contributors are helping build a collective archive
- contribution is meaningful, not just casual posting
- user pages are part of the documentary nature of the platform

Responsive behavior:
- On desktop, make the user’s archive feel browsable and substantial
- On mobile, make uploaded records easy to scroll and the contribution action easy to find
- The submission flow should be simple and usable on mobile

Design requirements:
- Create a high-fidelity responsive web UI concept
- Show both desktop and mobile layout ideas if possible
- Prioritize visual direction, responsive structure, and UX clarity over implementation details
- Use real-feeling photography placeholders
- Make the design feel specific to neonmap.world
- Preserve the current product structure conceptually, while rethinking layout and presentation
- Improve usability on both large monitors and mobile screens
- Focus on UI/UX and information design, not code or backend implementation
```

## 総評

### 🧠 Claude系はコンテキスト理解が強かった

Claude系統の2種の生成方法は、どちらもコンテキストを深く読み取ってくれていると感じました。

メトリクスや数字の生成を指定していると、よくあるダッシュボードの提案がされがちでしたが、今回はそういった方向には寄りませんでした。

そして、AIに限らず今までのUI生成サービスとは違った点として、**フォントの提案がかなり改善されていること**に驚きました。

---

### ✍️ Figma Make はコンテキスト量がボトルネックかもしれない

Figma Make はコンテキスト量、つまり入力できる文字数がそもそも限られていました。  
この辺は無料プランだったのでオプションがあるのか検証できていないですが。  
もしかするとその為に出力がやや一般的なプロダクトUIに寄った可能性があるかもと感じました。

面白かったのは画像がきちんとneon signになっていたところでした。そこは他のツールと顕著な違いが見れました。もしかするとその点のチューニングがFigma Makeは上手に設定できているのかもしれないです。

そして、claude系では正直UIの情報量などが多く、そこからどう減らしていくかがポイントになりそうだなと思いましたが、Figma Makeは4つの中で一番バランスよく、そのままUIとして使えそうなちょうど良い量でした。

---

### ⚛️ v0 の強みは、今回の比較軸では見えにくかった

v0 の売りは、React + shadcn + Tailwind で React コードをすぐ触れるという点だったと思います。

ただ、今回のように「生成された見た目だけ」を比較する場合、その強みはそこまで大きな差別化要素としては感じにくかったです。

実装に進む段階では便利だと思いますが、初回のUIビジュアル比較という意味では、Claude系の方が強く感じました。

---

### 🧩 Figma Make / v0 はワークフロー統合込みで見るべきかもしれない

Figma Make にしろ、Vercel社の v0 にしろ、自分たちでモデルを開発している訳ではないはずなので、その点でも少し難しいかもしれないと感じました。

正直、**モデルのコンテキスト理解力や表現力の差**がワークフローでの優位性より大きいと感じました。

ただ、UIに特に拘らず、簡単な業務アプリを作る場合であれば、金額の優位性で使うことはあるかもしれないです。

特に Figma Make は、チームで触れたり、Figma上で共有したりする可能性もあるので、その点は個人で使うのとは勝手が違うかもしれません。

---

### 🛠️ エンジニアなら Claude Code、方向性探りなら Claude Design

コードを書けて理解できるエンジニアであれば、**Claude Code + frontend-design skill** がかなり良いのではないかと思いました。

一方で、エンジニアではない方、またはエンジニアだけどコードを書く前にデザインになんとなく当たりをつけたい場合は、**Claude Design** が良いのではないでしょうか。

ただどちらもトークンをかなり食うと思うのでその点は注意が要りますし、なんならclaude designはpro以上の契約じゃないと使えなさそうです。

---

### 🔍 深いラリーは未検証

ただし、今回はまだ初回UI生成後、AIと深くラリーした訳ではありません。

正直、この極めてベーシックなUIから、どこまで自分好みのUIに持っていけるのかは未検証です。

また、そもそもデザインする上でトークンをかなり食ってしまうという制約もあります。

そのあたりは、もう少し試してみようと思います。

以上、拝読ありがとうございました。
