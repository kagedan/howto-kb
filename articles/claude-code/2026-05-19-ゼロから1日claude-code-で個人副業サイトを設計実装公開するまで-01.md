## はじめに

副業で受託案件を取りに行くために、自分のランディングページが必要だった。とはいえデザインに何時間もかけている余裕はないし、Figma で凝ったモックアップを起こす気力もない。

そこで「**Claude Code を相方にして、丸1日で公開状態まで持っていく**」というルールを自分に課して、実際にやってみた記録です。

成果物はこちら:

5ページ構成（Home / About / Services / Case Studies / Contact）、レスポンシブ、OGP画像自動生成、お問い合わせフォーム + Discord 通知、ランニングコスト **¥0/月**。

この記事は、**「AI駆動で個人サイトを作りたい副業エンジニア」** が再現できるように、決めごと・つまずいた点・コード断片・運用判断まで全部書き残します。Claude Code を業務で使い始めた人にも、**「個人サイト規模でどこまでやれるか」** の参考になるはず。

## ゴール設定と制約

最初に「何を作って何を作らないか」を言語化しました。

### ゴール

* ✅ クライアントに名刺代わりに見せられる **個人副業 LP**
* ✅ 5ページ程度の構成（Home / About / Services / Case Studies / Contact）
* ✅ **レスポンシブ**（PC・タブレット・スマホで読める）
* ✅ お問い合わせフォームが **動く**（メールではなく Discord に通知）
* ✅ **OGP 画像が出る**（SNS シェア時の見栄え）
* ✅ **ランニングコスト ¥0**

### 制約

* 🚫 顔出し不可（プライバシー優先）
* 🚫 デザイン凝りすぎない（モノクロベースでミニマル）
* 🚫 1日で完成しないなら設計を捨てる（タイムボックス優先）

「**何をやらないか**」を最初に決めることで、Claude Code に渡す指示がブレなくなります。これは AI 駆動開発全般で効く考え方。

## 技術スタック

### 採用

| 領域 | 選定 | 理由 |
| --- | --- | --- |
| フレームワーク | **Next.js 16 (App Router)** | 最新版を試したかった。SSG が効くページ構成なので無駄なクライアント JS が乗らない |
| スタイル | **Tailwind CSS v4** | PostCSS プラグイン形式に変わって設定が劇的にシンプル。`@theme inline` でカスタムカラー定義がスッキリ |
| ホスティング | **Vercel（Hobby）** | Next.js とシームレス。サブドメイン `*.vercel.app` で無料運用 |
| フォーム通知 | **Discord Webhook** | Slack より分離しやすく、副業用ワークスペースを作るのが楽 |
| バージョン管理 | **GitHub（Private）** | 副業活動用に既存とは別アカウントで運用 |
| ドメイン | **Vercel サブドメイン** | 初期コスト ¥0、収益化してから独自ドメインに移行する戦略 |

### 採用しなかったもの

* **Supabase / DB**: Contact フォームを Webhook 通知に倒したので不要
* **CMS（Sanity, Contentful 等）**: ページ数少ない & 更新頻度低いので TSX 直書き
* **Figma 等のデザインツール**: Claude Code に直接デザイン指示を出す方が速かった
* **独自ドメイン**: 後回し。月1,500円も収益化前から払いたくない

## プロジェクト初期化

ここからログ風に時系列で書きます。

### Next.js プロジェクト作成

```
mkdir s-nagaya
cd s-nagaya
npx create-next-app@latest . --typescript --tailwind --app --no-eslint
```

対話で `App Router: Yes` / `Turbopack: Yes` / `src/ directory: No` を選択。

### 動作確認

`http://localhost:3000` で Next.js のデフォルトページが表示されれば OK。

### Git 初期化（つまずきポイント①）

`create-next-app` が新しい Node 環境だと **Git 初期化をスキップする** ことがあります。私の環境ではスキップされました。

```
git status
# fatal: not a git repository (or any of the parent directories): .git
```

慌てずに手動初期化:

```
git init
git add .
git config --global user.email "副業用Gmail"
git config --global user.name "副業用GitHubユーザー名"
git commit -m "Initial commit: Next.js setup"
```

### GitHub リポジトリ作成

`gh` CLI を使うと一発:

```
gh auth login    # ブラウザ認証
gh repo create s-nagaya --private --source=. --push
```

⚠️ **アカウント切り替えの罠**: 本業の GitHub アカウントと副業用が異なる場合、`gh auth status` で必ず確認してから作成すること。間違えて本業アカウントに副業リポジトリを作ると後で移すのが面倒です。

## Claude Code への最初の指示

ここからが本番。Claude Code を起動して、一気にスケルトン構築を依頼します。

私が最初に渡したプロンプト（要約版）:

```
このリポジトリで個人ランディングページサイトを構築してください。
副業エンジニアのポートフォリオサイトです。

【方針】
- Next.js 16 (App Router) + Tailwind v4 の既存構成を活かす
- ミニマル・モノクロデザイン（Anthropic 寄り）
- Home + Contact の MVP を最優先で作って動かす
- その後 About / Services / Case Studies を追加
- 各ステップで git commit
- localhost で確認しながら段階的に進める

【デザインシステム】
カラー:
- Primary text:   #0A0A0A
- Secondary text: #525252
- Background:     #FAFAFA
- Surface:        #FFFFFF
- Border:         #E5E5E5
- アクセントカラーなし（完全モノクロ）

タイポグラフィ:
- 英語: Geist Sans
- 日本語: Noto Sans JP
- Hero見出し: clamp(2.5rem, 6vw, 4.5rem)
- Section見出し: clamp(2rem, 4vw, 3rem)
- 本文: 16px / 1.7行間

レイアウト:
- 最大幅: 1200px
- セクション間: clamp(4rem, 10vw, 7.5rem)
- 左揃え基本（Hero のみ中央揃え）
- アニメーション最小限（fade-in 程度）

【サイト構成】
- / (Home): Hero + Stats + Services + About 抜粋 + CTA
- /about: 経歴タイムライン + 専門領域 + 副業アプローチ
- /services: 3 サービスの詳細
- /case-studies: 実績ハイライト
- /contact: フォーム + 補足情報
- /api/contact: POST → Slack/Discord 通知

【共通要素】
- ヘッダー: テキストロゴ + ナビゲーション
- フッター: コピーライト + SNSリンク
- レスポンシブ対応

【進め方】
1. globals.css のカラー変数とタイポグラフィを整備
2. layout.tsx にヘッダー・フッター統合
3. Home ページの実装
4. Contact ページ + API ルート
5. About / Services / Case Studies を追加
6. ビルド確認 → commit & push
```

ポイントは「**設計指示書として最初に与え切る**」こと。途中で「やっぱりこの色じゃない」と言うと巻き戻しが発生して時間を食います。

### 渡した内容のうち、特に効いたこと

1. **「アクセントカラーなし、完全モノクロ」と明示**

   * これを書かないと、AI は親切心で青や緑を差し込んできます
   * モノクロを徹底するだけで、整っているように見える効果は大きい
2. **タイポグラフィに `clamp()` を指示**

   * レスポンシブ対応がほぼ「勝手に」できる
   * ブレイクポイントごとにメディアクエリを書く工数が消える
3. **「Home + Contact の MVP を最優先」と段階指示**

   * 一気に5ページ作らせると Claude のコンテキストが切れがち
   * 動くものを早く作って、ビルド・デプロイを通したい

## OGP 画像の動的生成

SNS シェア時の見栄えのため、`/opengraph-image` を作ります。

Next.js 13+ には `ImageResponse` という Edge Runtime ベースの画像生成 API があり、これを **`app/opengraph-image.tsx`** に置くだけで自動的に `og:image` メタタグに反映されます。

```
// app/opengraph-image.tsx
import { ImageResponse } from "next/og";

export const runtime = "edge";
export const alt = "S. Nagaya — AI-Driven Development × QA";
export const size = { width: 1200, height: 630 };
export const contentType = "image/png";

export default async function Image() {
  return new ImageResponse(
    (
      <div style={{
        height: "100%", width: "100%",
        display: "flex", flexDirection: "column",
        justifyContent: "space-between",
        padding: "72px 80px",
        background: "#fafafa", color: "#0a0a0a",
        fontFamily: "sans-serif",
      }}>
        <div style={{ fontSize: 32, fontWeight: 600 }}>
          s.nagaya
        </div>
        <div style={{ display: "flex", flexDirection: "column", gap: "32px" }}>
          <div style={{ fontSize: 18, letterSpacing: "0.18em", textTransform: "uppercase", color: "#a3a3a3" }}>
            AI-Driven Development × Quality Assurance
          </div>
          <div style={{ fontSize: 96, fontWeight: 700, lineHeight: 1.05, letterSpacing: "-0.04em" }}>
            <div>Build with AI.</div>
            <div>Ship with rigor.</div>
          </div>
        </div>
        <div style={{
          display: "flex", justifyContent: "space-between",
          paddingTop: "32px",
          borderTop: "1px solid #e5e5e5",
          fontSize: 20, color: "#525252",
        }}>
          <span>S. Nagaya — Side Job Portfolio</span>
          <span style={{ fontFamily: "monospace" }}>s-nagaya.vercel.app</span>
        </div>
      </div>
    ),
    { ...size }
  );
}
```

最初、`app/twitter-image.tsx` を以下のように書いて DRY にしようとしました:

```
// ❌ これは動かない
export { default } from "./opengraph-image";
export { alt, size, contentType, runtime } from "./opengraph-image";
```

ビルドエラー:

```
Next.js can't recognize the exported `runtime` field in route. 
It mustn't be reexported.
```

Next.js の Route Segment Config は **コンパイル時に静的解析できる形** でないとダメ。再エクスポートでは認識されません。

対処は素直に **`twitter-image.tsx` を独立した実装としてコピー** すること。`runtime = "edge"` を直接書き直す。DRY を諦める判断です。

### 確認方法

ビルド後 `https://s-nagaya.vercel.app/opengraph-image` を直接開けば、生成された PNG が見られます。SNS シェアの実検証は Discord に URL を貼って Enter するのが手軽。

## Discord Webhook 連携

問い合わせフォームの送信通知は Slack の代わりに Discord を選びました。本業の Slack と混ざらないため。

### Discord 側のセットアップ

1. Discord で副業用サーバーを作成（自分1人だけでOK）
2. `#contact-form` チャンネルを作成
3. チャンネル設定 → 連携サービス → **「ウェブフックを作成」**
4. 生成された URL をコピー

### 環境変数の設定

ローカルでは `.env.local`、本番は **Vercel の Project Settings → Environments** で `DISCORD_WEBHOOK_URL` を追加。

**注意**: Vercel の UI が 2025 年に変わって、`Environment Variables` という独立タブが消えて `Environments` 配下に統合されました。**「Environments を開いて Production の中の `Environment Variables` セクション」** から追加します。

### API ルート実装

```
// app/api/contact/route.ts
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const { name, email, company, message } = await request.json();
  
  // バリデーション省略

  const discordWebhookUrl = process.env.DISCORD_WEBHOOK_URL;
  
  if (discordWebhookUrl) {
    await fetch(discordWebhookUrl, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        username: "s.nagaya site",
        embeds: [{
          title: "新規お問い合わせ",
          color: 0x0a0a0a,
          fields: [
            { name: "Name", value: name, inline: true },
            { name: "Email", value: email, inline: true },
            { name: "Company", value: company || "—", inline: false },
            { name: "Message", value: message, inline: false },
          ],
          timestamp: new Date().toISOString(),
        }],
      }),
    });
  } else {
    console.log("[Contact Form]", { name, email, company, message });
  }
  
  return NextResponse.json({ ok: true });
}
```

Embed 形式で送ると見た目が綺麗。Slack の Block Kit より圧倒的にラクです。

通知が来ない場合は **Vercel ダッシュボード → Logs → Functions** で `/api/contact` のログを直接確認できます。

## つまずいた点 TOP3

### 1. GitHub アカウント切り替え

副業用アカウントでログインしているつもりが、`gh auth login` で誤って本業アカウントにいた。リポジトリ作成時に「すでに同名リポジトリがあります」エラーで気づきました。

対処:

```
gh auth status        # 現在ログイン中のアカウント確認
gh auth logout        # 一旦ログアウト
gh auth login         # ブラウザで副業用アカウントを選び直し
```

Git の commit user も忘れず副業用に切り替える:

```
git config --global user.email "副業用@gmail.com"
git config --global user.name "副業用ユーザー名"
git commit --amend --reset-author --no-edit
```

### 2. Windows での CRLF 警告祭り

```
warning: in the working copy of '.gitignore', LF will be replaced by CRLF...
```

ファイル数だけ出ますが **完全に無視で OK**。本番動作には影響なし。気になる場合は `.gitattributes` で改行コード方針を固定できますが、個人サイト規模ではやらなくていい。

### 3. Vercel の UI 更新で環境変数の場所が分かりにくい

「Environment Variables」というメニュー項目が見当たらず迷子。実際は **「Environments」**（複数形）に統合されました。Production を選んで詳細を開くと、**`Environment Variables` セクション** があります。

## 学んだこと

### Claude Code を「設計指示で動かす」のが効く

「コードを書いて」ではなく **「設計方針 → 制約 → 進め方の順序」** を渡すと、Claude Code は段階的に動いて、各ステップで commit を切ってくれます。後から「あれ、ここで何が変わった？」を追いやすい。

### 「やらないこと」を最初に決める力

* ❌ 凝ったアニメーション
* ❌ ダークモード切り替え
* ❌ 多言語対応
* ❌ CMS 統合

これら全部「**今回は作らない**」と決めた瞬間、Claude Code に渡す指示も短くなり、ビルドも軽くなりました。**「やらないことリスト」は AI 駆動開発の生産性に直結します**。

### 個人サイトは vercel.app で十分

副業初期の段階で独自ドメインに ¥1,500/年を払うより、その分を Connects（Upwork の応募トークン）や Resend（メール送信）に回した方が、目に見える ROI が大きい。**ブランド資産は売上が立ってから整える** で OK。

### Discord は副業ワークスペースに最適

Slack だと本業との混在リスクが常にある。Discord なら個人用サーバーを作ってチャンネル分離が完全。Webhook の作りやすさも同等。**副業用 Slack を新規ワークスペースで作るくらいなら、Discord にした方が後悔しない**。

## 次にやること

サイトを公開しただけでは仕事は来ないので、副業マーケットへの登録・営業活動が本番です。私の場合の次のステップ:

1. ビザスクexpert に登録、サイト URL を「その他」欄に貼る
2. Offers のプロフィールに URL を貼る
3. Upwork のポートフォリオ欄に URL を貼る（Overview 内には貼れない仕様）
4. X / LinkedIn のプロフィールにも反映

サイトはあくまで **「信頼補強の役割」**。集客は別のプラットフォーム経由で、そこから「詳しくはこちら」で誘導する設計です。
