## はじめに

Web制作のフリーランスを2年やってきて、フロントエンドエンジニアへの転向を目指して学習中です。  
学習スタイルは動画講座メイン。  
ただ、見ているうちに知識が右から左に流れていく感覚に、ずっとモヤモヤしていました。

その流れを止めたくて試行錯誤した結果、**学んだ内容を自分の言葉に変えて、Claude Code に声で残していく**というやり方に落ち着いたので、観察ベースで書き残しておきます。  
同じように学習中の方にとって、何かしらの参考になれば嬉しいです。

## 動画を見るだけだと、知識は右から左に流れていく

動画講座は分かりやすくて、見ているあいだは「理解できている気」になります。  
でも一区切り終わって振り返ると、輪郭がぼんやりしている。手を動かしてみると、思った以上に身についていない。

「分かった気になる」感覚は、自分には一番厄介でした。  
アウトプットが大事と言われますが、ノートにきれいにまとめようとすると今度は手が止まります。動画が進まなくなって、結局その日の学習がしぼむ。  
ワンセッション終わる頃には、まとめる前に頭から抜けている──これを何度か繰り返しました。

## 試した方法と、それぞれの限界

これまで試してきた方法と、自分が感じた限界は次の通りです。

* **ノートに書く**：書いている途中で動画を止める頻度が増えて、進まなくなる。完璧にまとめようとするほどハードルが上がる
* **後でまとめる**：そのまま結局やらない。やるときにはもう熱量が落ちている
* **ブログにまとめる**：書くこと自体が大変で続かない。文章として整える工程で止まる
* **SNSに毎日投稿する**：とりあえず続けてみたけど、定型文や日記のような中身の薄い投稿が並ぶアカウントになってしまった

どの方法も「**学習とアウトプットを別工程で回そうとしていた**」点が共通していたと、いま振り返ると感じます。  
別工程にした瞬間、片方が後回しになっていく感覚がありました。

## 環境：物理4画面・機能4役

いまの環境を一度整理します。

物理的には4画面、機能としては4つの役割を割り当てています。

```
[27インチ WQHD]         [27インチ WQHD]
       ↑                       ↑
     役割A                   役割B
 VS Code で実装       Udemy／Web検索（インプット）

[MacBook Pro 14"]       [モバイルモニター 14"]
       ↓                       ↓
     役割C                   役割D
作業補助（Finder/Slack 等）  Claude Code（声で言語化）
```

| 役割 | 画面 | 用途 |
| --- | --- | --- |
| A | 左上 27インチ WQHD | VS Code で実装 |
| B | 右上 27インチ WQHD | Udemy 動画講座・Web検索（インプット） |
| C | MacBook Pro 14インチ | Finder・Slack など作業を補助する画面 |
| D | モバイルモニター 14インチ | Claude Code に学びを声で言語化して残す |

物理画面が4枚あるおかげで、視線を移すだけで A／B／D を切り替えられます（C は補助なので常時参照）。  
この「視線の切り替えだけで完結する」のが地味に効いていて、画面を切り替えるためのコマンドや、ウィンドウを探す数秒のロスがゼロになります。  
学習の集中はこういう小さな摩擦の積み重ねで途切れるので、配置に投資するのは思った以上にリターンがありました。

## やり方：学んだことを自分の言葉に変えて声に残す

具体的な動きはとてもシンプルです。

1. 役割B（Udemy）で動画を流す
2. 役割A（VS Code）で講師と一緒に手を動かす
3. 役割D（Claude Code）に、 **いま聞いた内容を自分の言葉に変えて声で残していく**

ここでのポイントは、**動画の音声をそのまま流すのではなく、いったん自分の頭で受け止めてから、自分の言葉に置き換えて口にする**ことです。  
「いまの説明、〜ってこと？」「これ自分の案件で使えそう」「なんで `useEffect` のここでコールバックを返してるんだろう」みたいな、頭に浮かんだまんまの粗い言葉で構いません。  
整える必要はありません。

音声入力は Mac 標準の音声入力と、無料の **Handy**（Whisper ベースのオープンソース・オフライン動作）を使い分けています。  
Mac 標準は短い気づき向き、Handy は長めに話したいときに便利、という雑な使い分けです。

Claude Code 側は、日常業務全般をサポートしてくれる専用ディレクトリ（自分は秘書モードと呼んでいます）で動かしています。  
このディレクトリには `CLAUDE.md` でルールが書かれていて、セッション開始時に自動で秘書モードに切り替わるようにしています。  
学習ログ・タスク管理・発信ネタの捕獲がこの1つのコンテキストの中で回るので、わざわざ「いまから学習ログを取ります」と宣言しなくてもよい構成です。

声にした内容は、その場で AI が整形してログに残してくれます。  
分からない用語があれば、そのまま「これってどういう意味？」と聞けば答えが返ってきます。Web検索で寄り道する代わりに、文脈に絞った答えがその場で返ってくるイメージです。

## 「自分の言葉に変える」その瞬間に定着が起きている

このやり方をしばらく続けて気づいたのは、**記録は副産物で、本命は「自分の言葉に変えて口にする」その瞬間にある**ということです。

聞いた内容を自分の言葉で言い直そうとすると、頭の中で一度内容を咀嚼することになります。  
咀嚼の過程で、「なんとなく分かった気になっていた部分」が浮かび上がってきます。  
言葉にできない＝まだ理解できていない、と即座に気づけるわけです。

声に出すという物理的な行為そのものが、頭の中の曖昧な理解を、輪郭のあるアウトプットに変える工程になっています。  
動画をただ見ているだけのときと比べて、定着の手触りが明らかに違うと感じています。  
ファインマン・テクニックのような「自分の言葉で説明できることが、本当に理解している証拠」という考え方とも、感覚的には近い気がします。

## なぜネットサーフィンの沼に落ちにくいか

学習中に Web 検索すると、関連記事に飛んで、次の記事に飛んで……というふうに発散しがちです。  
気づくと最初の動画から1時間離れている、という経験は何度もしました。

Claude Code に直接聞くと、いま見ている動画の文脈に絞った答えが返ってくるので、自然と元の話に戻りやすくなります。  
回答画面が「動画の続きに戻るための入口」のままなので、心理的にロックがかかる感覚があります。  
結果として、学習セッションが脱線しにくいというメリットがありました。

## インプット・思考整理・アウトプット種が同時に回る

このワークフローで特に効いていると実感しているのは、**3つの工程が並列で回る**ことです。

通常は次のように直列で動きます。

このやり方だと、3つを並列で回せます。

声に出して言語化すること自体が思考整理になり、その整理結果がそのまま発信ネタの原型になります。  
「発信のために改めて書く時間」が、ほぼゼロになりました。

## アウトプットが勝手に資産になる

整形済みのログから、Zenn 記事・ブログ記事・Twitter 下書きが派生していきます。  
じつは、いまあなたが読んでいるこの記事も、まさにそのワークフローから生まれた1本です。

学習ログを「自分のためのノート」で終わらせず、最初から発信できる形のまま積み上がっていく感覚があります。

* 過去の自分が振り返るときの材料になる
* 同じ学習領域に立っている人にとって、追体験できるログになる
* 未来の自分が「あの時こう考えていた」を確認できる

学習と発信を別工程にしないと決めただけで、資産が積み上がるスピードが変わったと実感しています。

## 「学習途中の発信は大事」と言われるけど、時間が惜しい問題への解

「学習中の発信のほうが価値がある」とよく言われます。  
言いたいことは分かる。でも実際、学習中に発信に時間を割くのは大変です。学習の手も止まる。

このワークフローは、その時間コストを限りなくゼロに寄せられました。

* 自分の学習ペースは落とさない
* 同時に発信ネタが積み上がる
* 結果として、勉強や本業の仕事に割ける時間が増える

以前は毎日投稿を続けているうちに、内容より継続を優先してしまっていた時期がありました。いまは「**作業ログから抽出された、中身のある発信**」が自然に残るようになっています。

## 注意点・限界

押しつけにならないように、自分が感じている限界も書いておきます。

* 音声入力に慣れるまで、最初は恥ずかしいし、うまく言語化できない
* 静かな環境がほぼ必須（共用スペースだと使いにくい）
* AI が完璧にまとめてくれるわけではない。最後に人の目を通す工程は残る
* チュートリアル系の動画では、まず手を動かす方を優先する。声での言語化は補助に回したほうが定着する
* 物理画面が3〜4枚必要というハードウェア要件はある。最初はメイン+サブの2画面でも始められると思います

## おわりに

「動画を見る」「ノートに書く」「ブログにまとめる」を別工程として回していたときは、どこかが必ず止まっていました。  
**学習工程と発信工程を分離しない**という発想にしたら、止まるポイントがほぼ消えました。

AI を「学習ノート係」として常駐させる感覚が、自分には合っていると感じています。  
同じように動画講座で勉強している方が、自分の環境に合わせて取り入れてくれたら嬉しいです。

このやり方を続けながら、半年後のフロントエンドエンジニア転向に向けて学習ログを積み上げていきます。
