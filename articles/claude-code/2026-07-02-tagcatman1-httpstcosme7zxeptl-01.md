2025年秋頃から、業務で基本的にコードを1行も書いていません。

フロントエンドの仕事を10年、フリーランスとして9年。いまは受託の実案件を、Claude Codeを中心とした仕組みで回しています。コードを書かなくなっても、納品物の品質のコントロールは手放していない——検証できる実績としては、Claude Codeを使って個人開発したFigmaプラグインを4本公開しています（Vertical Text JP ほか、合計ユーザー数は執筆時点で5,000超。Figma Communityで誰でも実数を確認できます）。

この記事は、その仕組みにたどり着く前に私が食らった「3つの破綻」の話です。他人の失敗談ではなく、自分が通った道の記録です。

# 最初の1週間は、たしかに魔法だった

Claude Codeを初めて実務に使ったときの感覚は、いまでも覚えています。「このセクションのレイアウトを組んで」と伝えると、数十秒で動くコードが返ってくる。単発のタスクでは、ほとんど無敵に見えました。

この初期体験は本物です。問題は、ここから「実案件もこの調子でいける」と推論してしまうことです。単発タスクと実案件には決定的な違いがあります。実案件は長く続き、何往復もし、最終的な品質の責任を自分が負う、という点です。

思いついたことをその場でチャットに打ち込み、資料をその都度貼り、出てきたものを使う——私はこの運用を「場当たりプロンプト」と呼んでいます。単発タスクでは最速の方法です。しかし案件が進むにつれて、3つの破綻が構造的に——つまり、あなたの腕前やプロンプトの上手さと無関係に——やってきます。

# 破綻その1 — 「AI流のコード」が増えるほど、自分が関与できなくなる

何も指定せずに任せると、出てくるのは「AI流」のコードです。AI流のコードは下手ではありません。動きますし、命名も整っています。だから最初は問題に見えない。

問題が出るのは、書かせたときではなく「直すとき」です。実案件では修正・追加・クライアント対応が何往復も続きます。その場面で、AI流のコードは「他人のコード」として目の前に現れる。どこを直せばいいかを毎回読み解くところから始まり、案件が進むほどそのコストは膨らみ、やがて「修正箇所をAIに聞かないと自分のプロジェクトを触れない」状態に行き着きます。

納品物に最終責任を負うのは自分なのに、中身を自分の言葉で説明できない。私はこれを「コントロールの喪失」と呼んでいます。品質の問題である以前に、職業上の危機です。

# 破綻その2 — セッションが切れるたび、プロジェクトの記憶が消える

Claudeの記憶は、セッションの外に持ち出せません。昨日3時間かけて伝えた案件の背景、一緒に決めた設計方針——翌朝の新しいセッションのClaudeは、その一切を覚えていません。

私の開発記録には、こんな一行が残っています。「前セッションの途中段階では正常だった。そのときのコードは保存なし」。一度は動いていたのに、セッションが変わった時点で「動いていたとき何をしていたか」を誰も正確に覚えていない。結果、解決済みだったはずの地点まで試行錯誤をまるごとやり直しました。

セッションは必ず切れます。これは工夫で回避する類の問題ではなく、前提として設計に織り込むべき制約です。

# 破綻その3 — 口頭ルールは蓄積しない。逸脱だけが蓄積する

案件には決まりごとが増えていきます。「ブレークポイントは1024px」「imgタグに直接スタイルをあてない」。チャットで伝えれば、そのセッションの間は守られます。しかしセッションが変わればルールも消える。チャットでの指示は、蓄積しないのです。

すると、セッションごとに「伝え忘れたルール」が少しずつ破られます。火曜のコードと木曜のコードで書き方が違う。一つひとつは小さな逸脱ですが、納品物の上では一貫性の崩壊として蓄積し、後から入る修正がその逸脱を「既存の書き方」として踏襲して、崩れが崩れを呼びます。

# 3つの破綻は、同じ構造をしている

並べると共通点は明白です。渡すべき情報を、毎回その場で、揮発する場所（チャット）に渡している。AIの性能の問題でも、プロンプトの巧拙の問題でもありません。

だから解決の方向も「もっと上手いプロンプトを書く」ではありません。毎回伝えるべきことを、AIが自動で読む場所にファイルとして固定する——仕組み化です。自分のコードの癖はルールファイルに。プロジェクトの記憶はAI自身に書き残させて次セッションに読ませる。運用ルールは起動時に自動で読み込まれる場所に。
