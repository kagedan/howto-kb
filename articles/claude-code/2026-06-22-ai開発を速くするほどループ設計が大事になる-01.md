最近個人でCodexを試してますが、直近のClaude Codeに比べるとレスポンスが体感で倍以上は速く、開発が捗ります。

デスクトップアプリでセッション管理からレビューまで完結する点も気に入ってます。（Claude Codeもできますが）処理の定期実行ができるので、事前整理したIssueを `/goal` スキルで複数渡し、調査→実装→検証→PRまで作ってくれるようになってきました。  
**コーディングが高速化することで、これまで以上に次のような場所が気になり始めました。**

* Codexのセッションを複数Issueで使い回すと、適切にワークツリーを分離してくれないことがある
* デプロイ先や検証環境が競合する
* 実装は終わっているのに、動作確認やレビュー待ちで止まる
* 失敗したときに、どこまでAIに任せてよいか分からなくなる

つまり、本格的にボトルネックが「AIがコードを書けるか」ではなく、 **AIが安全に何度も開発ループを回せるか** に移りつつあります。

これをどう解決しているか、チームに適用するには追加で何を考えるべきか、今の考えをまとめておきたいと思います。

## 対象読者

* **すでにClaude CodeやCodexなどのAIエージェントを使って開発している人**
* 個人・チーム開発でAI活用をもう一段進めたい人／複数のAIを並列で動かすことに関心がある人
* 「AIにコードを書かせる」段階は一度通った前提で、その次の **「どう安全に何度も開発を回すか」** に関心がある人

各ツールの導入手順やプロンプトの基礎には触れません。Issue/PRベースの開発フローや仕様駆動開発（SDD）の概要を知っていると、より読みやすいはずです。

その前に、本記事で何度も出てくる「ループを回す」という言葉の意味を、先に一言で定義しておきます。

!

**本記事で「ループを回す」とは**

人間が毎回プロンプトを入れ直さなくても、あらかじめ決めた契約（入力・出力・検証・停止条件・人間確認・還元。詳しくは後述）に沿って AI が次の Issue・次のステップへ自律的に進み、**停止条件か人間確認の境界に当たるまで反復が続く状態** を指します。

逆に「回っていない」のは、1ステップごとに人間が指示を入れ直している状態（＝単発依頼の連続）です。

プログラミングのforループのように、 **契約という枠の中で、止まるべきところまで AI を自走させる** という意味で使っています。

## 実装速度の次のボトルネック

AIエージェントが1体だけなら、人間がだいたい流れを見ています。

「このIssueをやって」「PRを作って」「レビューして」

このくらいなら、多少失敗しても人間が見て直せます。

でも、複数のIssueを並行で進めたり、Goalを渡して調査・実装・検証まで進めたりすると、そうもいきません。たとえば、次のようなことが起きます。

1. あるAIセッションがステージング環境へデプロイしている途中で、別のAIセッションも同じ環境へデプロイする
2. あるPRの動作確認をしているはずなのに、環境には別PRの変更が入っている
3. ステージング環境でのテストが落ちたが、原因が実装なのか環境競合なのか分からない
4. デプロイやE2Eが高コストで、AIが何度も回すと時間もお金も溶ける

これはAI特有ではなく、CI/CDやテスト環境を共有している開発チームで起きえる問題です。ただ、実装の高速化によって、たった数人のチームでもこうした問題の発生頻度が高くなっている実感があります。

人間が1日に数回しかデプロイしない前提なら雑に運用できていた環境も、AIが何度もPRを作り、何度も検証しようとするとすぐ限界が見えます。手で書くよりAIに任せる時間が長くなるほど、環境やレビューの雑さがそのまま開発速度の上限になります。

ここで必要になるのが、単なる自動化ではなく、 **開発ループそのものの設計** です。

## 理想とする状態

ここでのゴールは、人間を完全に外すことではありません。むしろ人間が見るべきところを明確にするために、それ以外の繰り返し作業をループ化したいという思想です。

**AIに任せたいのは判断そのものではなく、判断できる状態を作るところ** です。

この「契約に沿って止まるところまで自走する」という前提が、後段の環境ロックやコスト・停止条件の話まで一本でつながっていきます。

## Loop Engineeringがチーム運用の高速化の鍵

直接のきっかけは、Acrosstudioテックブログの [もうプロンプトを書くな──「Loop Engineering」という新しいパラダイムの正体](https://zenn.dev/acrosstudioblog/articles/38509c0473683a) でした。

元々、自分の中にも「AIに単発で依頼するより、Goal、Automation、スキル、サブエージェント、コネクタを組み合わせて継続的に回す方が重要になる」という構想はありました。  
ただ、この記事ではPrompt EngineeringからContext Engineering、Harness Engineering、Loop Engineeringへと徐々に効きどころが移っている、という流れをかなり具体的に整理してくれています。

特に、ループを構成するものとしてAutomation、ワークツリー、スキル、サブエージェント、コネクタ、Memoryを並べる見方は、自分が[lifeリポジトリで試している](https://zenn.dev/monokaai/articles/zenn-life-repo-with-claude-code)こととかなり重なりました。

これらの元になっている海外文脈として、Addy Osmaniの [Loop Engineering](https://addyosmani.com/blog/loop-engineering/) があります。Addyの記事では、Loop Engineeringを **「人間がエージェントへプロンプトする役割を、代わりにプロンプトするシステム設計へ置き換えること」** と説明しています。また、Claude Codeの作者であるBoris Cherny氏が、複数セッションや多数のサブエージェント、`/loops`、Routinesを使って継続的に作業を回している、という海外記事も出ています。

Claude Code公式ドキュメントでも、Claudeが文脈を集め、行動し、結果を検証して繰り返す [agentic loop](https://code.claude.com/docs/en/how-claude-code-works) が説明されています。これをAgentic loopと呼ぶそうです。

上記の整備が大事な理由として、 **個人の実装速度が上がってもレビュー・検証・デプロイ・ルール整備が追いつかないとチーム全体の速度は思ったほど上がりません。** このあたりは [Claude Code活用を社内で1ヶ月推進した際の備忘録](https://zenn.dev/spectee/articles/claude-code-internal-promotion-1month) にも書きました。

## Agentic loopを運用に広げて考える

先の記事を踏まえて、この記事では自分の運用に引き寄せたLoop Engineeringを次のように整理します。

> AIへの単発依頼を、入力、出力、停止条件、検証証拠、人間確認の境界を持つ反復運用に変えること

たとえば、1つの開発ループには次のような要素があります。

| 要素 | 問い |
| --- | --- |
| 入力 | 何をAIに渡すのか。Issueか、Goalか、仕様書か |
| 出力 | 何ができたら完了なのか。PRか、設計メモか、検証ログか |
| 検証 | 成功を何で証明するのか。lintやテストか、レビューか |
| 停止条件 | どこでAIを止めるのか。権限、失敗回数、コスト、判断不能か |
| 人間確認 | 何は人間が見るのか。公開判断、マージ、権限拡大、価値判断か |
| 還元 | 得た知見をどこへ戻すのか。docs、スキル、Issue、memoryか |

この形にしておくと、AIに任せる範囲を少しずつ広げやすくなります。

これらを契約と呼び、そのままCodexへ渡すGoalやプロンプトに落とし込みます。たとえば、次のような契約を先に渡します。

ループを回すプロンプト例

```
ready ラベルの付いた open Issue を優先度順に1件ずつ処理して。各 Issue は以下の契約で進めること。

- 入力: アサイン済みの open Issue（ラベル: ready）
- 出力: 実装とテストを含む PR（1 Issue = 1 PR、目安500行以内）
- 検証: lint とテストが通ること。落ちたらログを読んで自分で直す
- 停止条件: テストが3回連続で失敗、または破壊的・不可逆な操作が必要になったら止めて報告する
- 人間確認: PR 作成までが担当範囲。マージと公開判断は人間が行う
- 還元: 詰まった点や再発しそうな知見は docs か memory に残す

1件終わったら次の Issue へ進んで。
```

入力・出力・検証・停止条件・人間確認・還元という6つの要素を、そのままプロンプトの箇条書きに対応させているのがポイントです。この「契約」を最初に渡しておくことで、AIが暴走せず、人間が見るべき境界（マージ・公開判断）だけが手元に残ります。

最初は読み取り専用の調査だけ。次にMarkdownの下書き。次にPR作成。次にテスト実行。さらに進めるならデプロイと動作確認。

**いきなり全自動にするのではなく、ループごとにするたびに任せる境界を決めて広げていく** イメージで改善するのが良さそうです。

## 環境もループの一部として扱う

チーム開発で重要になるのが、検証環境です。

AIが実装してPRを作るだけなら、まだローカルやCIの範囲で閉じられます。しかし、アプリやAPIの動作確認まで任せたいなら、どこかの環境へデプロイする必要があります。

そこで環境を利用したいループ同士での競合問題が起きます。今対処法として考えているのは、デプロイ先を「ただの環境」ではなく **ループが一時的に占有する共有資源** として扱い、利用中はロックすることです。

この「環境ロック」により、 **環境を排他制御の対象として複数のAIエージェントに扱わせ、デプロイを順番待ちさせることが可能** になります。

あるループが環境を使っている間は、他のループはその環境を前提にした検証を進めない。使い終わったら、検証結果とともに次へ渡せる状態にする。

これにより、複数のAIセッションが動いても「どの変更をどの環境で確認したのか」を追いやすくなります。

逆に、ここを曖昧にしたままAIにデプロイやE2Eまで任せると、失敗したときに原因が追えません。AIの実装が悪いのか、環境が壊れているのか、別のAIが上書きしたのか分からなくなるからです。

そして、 **環境をループに含めると、待ち時間や再実行もループの一部になります。** だからこそ、次に効いてくるのがコストと停止条件です。

## コストも停止条件に入れる

AI開発ループでは、コストも設計対象になります。ここでいうコストは、単に月額いくらという話だけではありません。

* AIの利用料
* CI/CDの実行時間
* デプロイやE2Eの待ち時間
* AWSなどクラウド環境の利用料
* **失敗時に人間が調査する時間**
* **レビュー待ちでPRが滞留する時間**

こうしたものが全部コストです。AIは疲れないので、放っておくと同じ検証を何度も回せます。  
これは便利な一方で、 **失敗条件や上限がないと高速で環境を破壊する装置になる可能性があり、危険です。**

たとえば、次のような制御が必要になります。

* テストは何回まで再実行するか
* デプロイ失敗時に自動リトライするか
* E2Eは毎回走らせるのか、差分に応じて絞るのか
* 何分を超えたら人間に戻すのか
* どの失敗はAIが直してよく、どの失敗は止めるのか

このあたりも、ループ契約として先に書いておくべきだと感じています。

## 最終的に目指したい姿

最終的には、`/goal` スキルに目的を渡すだけで、次のような流れが回る状態にしたいです。

このときも、各ステップには「どこまで進めるか」「何回まで試すか」「どこで人間へ戻すか」という上限を持たせます。速く回すためには、止まり方も設計しておく必要があります。

ループが回ると、人間が確認するのは次のような点に絞られます。

* 実装できたか
* テストで要件が正しく整理・確認されているか
* デプロイして実際に動作確認できたか
* PR本文に検証証拠が残っているか
* 自動マージ候補なのか、人間確認必須なのか、ブロック中なのか
* ドキュメントは適切に更新されているか（変更内容の追記修正、意思決定履歴の記載、リリース後のロールバック手順など）

ここまで整理されていれば、人間は「この変更を受け入れるか」「公開してよいか」「次のループで権限を広げてよいか」に集中できます。

## 導入結果

**個人開発の場合はリスクの許容度が高いので、私の場合は7割の精度を目指して即日ループを導入し、少しずつ気になったところを改善するアプローチ**を取りました。

これにより、Issueの解決速度がそれ以前のAI伴走における2,3個/日→30個/日以上まで成長しつつあります。  
直近1日では50個以上の解決も記録しています（画像右端のJun 10あたりから急増）。

## おわりに

AI開発は、コード生成の話だけでは終わらなくなってきました。むしろ、コードを書く部分はかなり当たり前になりつつあります。

その次に効いてくるのが、Issueの切り方、検証環境、レビュー、マージ判断、学びの蓄積です。

だからこそ、これからは「AIに何を頼むか」だけでなく、 **AIが何度も安全に回れるループをどう設計するか** が大事になると思っています。

個人では、この考え方をLoop Engineeringとして育てていくつもりですし、チーム開発にも活かしていくつもりです。
