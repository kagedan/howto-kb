フロントエンドエンジニアの usako です。

皆さんは Cording Agent を使うにあたって、「〇〇の方が性能がなんとなく悪い気がする…」  
ハッキリとした根拠がなくモヤっとした気持ちのまま使っていることがありませんか？

今回は実際のところどうなのか？という自らの疑問を少しでも解消できるように、各 Cording Agent の性能差について調べてみました。

本記事では弊社が利用を認めていたツールに限定して取り上げています。  
そのため、OpenAI の Codex など、現時点で社内利用の承認を得ていないものは対象外としています。

## 特徴

まずは各 Cording Agent の特徴を簡単にまとめてみます。

### GitHub Copilot

GitHub が提供する AI コーディングアシスタントです。  
VS Code や JetBrains などの IDE に統合され、コード補完や自動生成を行います。  
GitHub リポジトリに基づいたインデックスを活用し、コードベース全体の理解をサポートします。

参考：[GitHub Copilot · あなたの AI ペア プログラマー](https://github.com/features/copilot?locale=ja)

### Claude Code

Anthropic が提供する AI コーディングアシスタントです。  
大規模なコードベースの理解に特化しており、複雑なコードの解析や修正を支援します。  
独自のエージェント駆動型検索を採用し、リアルタイムでコードを探索できます。

参考：[Claude Code by Anthropic | AI Coding Agent, Terminal, IDE](https://claude.com/ja/product/claude-code)

### Cursor

Anysphere 社が開発・提供する、VS Code をフォークして作られた AI 特化型のコードエディター（IDE）です。  
エディター自体に AI が深く統合されており、プロジェクト全体のコードベースを自動でインデックス化することで、文脈を正確に理解した高精度な提案を行います。  
とくに強力なのがマルチファイル編集に対応した **Composer 機能** で、単一のファイルに留まらず、アプリ全体の機能追加や複数ファイルにまたがる修正・リファクタリングを、AI が一括で自律的に実行できます。

## インデックス

各 Cording Agent のワークスペースのインデックスの構築方法を見ていきます。  
インデックスはモデルがコードベースを理解して関連する情報を提供するための基盤となります。

### GitHub Copilot

Copilot はすべてのワークスペースに対して自動でインデックスを構築・維持します。  
インデックスの作成元となるソースを2つにピックアップします。  
それぞれ利用可能になるまでの時間が異なります。

現在のインデックス作成状況は、VS Code のステータスバーにある「Copilot ステータスダッシュボード」から確認できます。

参考：[How Copilot understands your workspace](https://code.visualstudio.com/docs/copilot/reference/workspace-context)

#### GitHub リポジトリ

GitHub.com または GitHub Enterprise Cloud でホストされているリポジトリは、GitHub 側が自動でインデックス化します（GitHub アカウントでのサインインが必要）。リポジトリごとに1度構築すればよいため多くの場合で即座に利用可能になりますが、数十万ファイルに及ぶ大規模リポジトリでは時間がかかる場合があります。

#### その他のワークスペース（ローカルフォルダーなど）

GitHub に紐づいていないローカルフォルダーなどは、Copilot が自動でインデックスを構築します。最初の構築には数分かかりますが、その後はバックグラウンドで常に最新の状態に保たれます。

#### インデックスの対象となるファイル・除外されるファイル

セマンティック検索の対象になるのは、現在のプロジェクトに含まれる「関連性の高いテキストファイル」すべてです。特定のプログラミング言語に限定されません。

ただし、検索のノイズを減らすために、以下のファイルはインデックスから自動的に除外される仕組みになっています。

自動除外される一般的なファイル：.tmp や .out など、開発に直接関連しないファイル  
設定による除外ファイル：VS Code の files.exclude 設定や、.gitignore ファイルで除外されているファイル  
バイナリファイル：画像や PDF など

### Claude Code

従来の AI コーディングツールの多くは、コードベース全体を事前にインデックスを構築する方式（RAG）を採用しています。ですが、Claude Code はこのアプローチをあえて採用していません。

参考：[How Claude Code works in large codebases: Best practices and where to start | Claude](https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start)

#### エージェント駆動のリアルタイム探索

Claude Code は重いインデックス作成やサーバーへのアップロードを一切必要としません。その代わりに、「自律的なエージェント駆動型検索（Agentic Search）」を採用しています。

Claude Code は人間のエンジニアとまったく同じように、プロジェクトのディレクトリ構造をその場で走査し、grep などの内蔵コマンドを自ら実行して必要なファイルをリアルタイムに探し出します。これにより、インデックスの更新漏れによる「古いコードへの参照」を防ぎ、常に最新のローカルにあるコードをベースにした正確なコンテキスト把握を可能にしています。

#### 「Harness（ハーネス）」によるパフォーマンス向上

インデックスを持たない一方、大規模なコードベースで探索のスピードと精度を極限まで高めているのが「Harness」と呼ばれる概念です。

AI が手探りでコードを探すのは効率的ではありません。  
開発者がプロジェクト固有の「Harness」をあらかじめ用意しておくことで、Claude Code のパフォーマンスは爆発的に向上します。  
ハーネスは、CLAUDE.md ファイル、フック、スキル、プラグイン、MCP サーバーという5つの拡張ポイントから構成されています。

Claude Code は探索を開始する際、まずこれらの「Harness」をヒントとして読み込みます。  
プロジェクトの構造や重要なエンドポイントが明文化されているため、エージェントは迷うことなく、数秒で目的のコード（修正すべきバグや依存関係）へと自律的にたどり着くことができます。

### Cursor

セマンティック検索を実現するため、Cursor はプロジェクトを開いた際に、コードベースの検索可能なインデックスを構築します。小規模なプロジェクトであれば、この処理はほぼ瞬時に完了します。

数万ファイル規模の大規模なリポジトリでは、処理に数時間かかる場合があります。Cursor では、すべてを再処理するムダを省くために、以下のような技術を採用しています。

ゼロからインデックスを作成するタイプの AI エージェントとの差別化ポイントは、**「大規模リポジトリでも数秒でインデックスが終わる速さ」** と、**「他人のコードが混ざらない安全性（セキュリティ）」** を両立している点にあります。

参考：[大規模コードベースを安全にインデックス化する · Cursor](https://cursor.com/ja/blog/secure-codebase-indexing)

#### Merkle tree（メルクルツリー）

* 各ファイルの暗号学的ハッシュと、それをもとに計算されるフォルダーのハッシュを使って、コードベースの構造をツリー化します。
* 開発者がコードを編集した際、Cursor は「変更されたファイル」と「ルートまでの親ディレクトリ」のハッシュだけをサーバー側と比較します。これにより、2つのツリーの分岐点を瞬時に特定し、ハッシュが異なる部分（差分）だけを同期することが可能です。
* この仕組みにより、5万ファイル（ファイル名と SHA-256 ハッシュだけで約3.2MB）あるようなワークスペースであっても、毎回すべてのデータを転送するムダを排除しています。

#### 構文単位のチャンク分割と埋め込み（Embedding）のキャッシュ

* ファイルが変更されると、コードを構文単位（チャンク）に分割し、セマンティック検索用の「埋め込み」へと変換します。
* この変換はコストが高いため、バックグラウンドで非同期に実行されます。また、チャンクの内容ごとに埋め込みをキャッシュしているため、変更のない大部分のコードはキャッシュにヒットし、余計な計算コストや時間を削減できます。

#### 安全なインデックス再利用（アクセス証明）

* 新しいメンバーがチームに参加した際、他のメンバーが作成したインデックスを再利用することで、わずか「数秒」で検索可能にする仕組みです。
* これによって、「自分が持っていないコード（他人の未プッシュのコードなど）の検索結果が見えてはいけない」というセキュリティ上の制約もクリアしています。

## ツール

それでは、各 Cording Agent が提供するツールについて見ていきます。

### GitHub Copilot

GitHub Copilot のエージェントは、コードベースを理解するために自動で最適な検索ツールを選択・組み合わせ、開発者のように繰り返し検索（探索）を行います。  
プロンプト内で `#ツール名` を入力することで、特定のツールを明示的に指定することも可能です。

参考：[How Copilot understands your workspace](https://code.visualstudio.com/docs/copilot/reference/workspace-context#_semantic-index-sources)

#### ワークスペース内の検索・読み取りツール

* **Semantic search (`#codebase`)**: キーワードの完全一致だけでなく、質問の「意味・意図」に合致するコードを検索（※要インデックス）。
* **Text search**: ファイル内から特定のキーワードに一致するテキストを検索。
* **Grep**: インデックスなしで、ファイル全体からテキストや正規表現パターンを検索。
* **File search**: ファイル名やグロブパターンからファイルを検索。
* **Usages**: 定義への移動や参照検索を組み合わせ、シンボルがどう使われているかを追跡。
* **List directory**: ディレクトリ構造を一覧表示し、プロジェクト構成を把握。
* **Read file**: 特定のファイルの中身を詳しく読み込む。

#### 外部（GitHub）リポジトリの検索ツール

ワークスペース外のコード（上流ライブラリのAPIなど）を調べたい時に使用します。

* **Semantic Search GitHub Repository (`#githubRepo`)**: 指定したGitHubリポジトリ（owner/repo）を意味的に検索。
* **GitHub Text Search (`#githubTextSearch`)**: 指定したリポジトリや組織内からキーワードやコードパターンをテキスト検索。

#### 検索精度の向上（ノイズの除外）

ログや自動生成ファイルなどによるコンテキストのムダ遣いを防ぐため、以下の設定を使ってエージェントの検索対象から除外（ exclusion ）できます。

* **`.gitignore`**: テキスト検索、Grep、セマンティックインデックスから除外。
* **`files.exclude`**: エクスプローラーで非表示にし、検索・インデックスからも除外。
* **`search.exclude`**: エクスプローラーには表示しつつ、テキスト検索とGrepから除外。

### Claude Code

コードの理解や修正をサポートする、便利な組み込みツールが標準で用意されています。  
ツール名は、権限ルール、subagent のツールリスト、およびフックマッチャーを指定する際の識別子（文字列）として使用します。

独自にカスタムツールを追加したい場合は、MCP サーバーを接続。  
また、何度も使うプロンプトをワークフロー化して Claude の機能を拡張したい場合は、新しく「skill（スキル）」を作成します。  
なお、これは新しいツール自体を追加するのではなく、既存の Skill ツールを介して実行される仕組みです。

ツールが多すぎるため、各ツールの説明を省略します。  
詳細は公式ドキュメントを参照してください。

参考：[ツール リファレンス - Claude Code Docs](https://code.claude.com/docs/ja/tools-reference)

### Cursor

コードベースを理解するために自動で最適な検索ツールを選択・組み合わせ、開発者のように繰り返し検索（探索）を行います。

参考：

#### 1. Terminal（ターミナル）

* **概要**: AI エージェント（Cursor Agent）がユーザーに代わってローカル環境のターミナルコマンドを実行できる機能。
* **主な役割**: コード変更後のビルドやテストの実行、パッケージのインストールなどを AI が自動で行い、エラーが出ればその場で修正します（安全のため、一部の破壊的なコマンドは自動実行されずユーザーの承認が必要です）。

#### 2. Browser（ブラウザ）

* **概要**: AI エージェントがWebブラウザを使ってインターネット上の情報を閲覧・取得できる機能。
* **主な役割**: 最新の公式ドキュメント、API 仕様、エラーの解決策、あるいは開発中のローカルサーバー上のWeb画面などを AI が直接読み込み、正確な情報をもとにコードを作成・修正します。

#### 3. Semantic & Agentic Search（セマンティック＆エージェンティック検索）

* **概要**: 単なるキーワード一致ではなく、コードの意味や文脈を理解して行う高度なコード検索機能。
* **主な役割**: AI が自律的（Agentic）にプロジェクト内を探索し、「この処理に関連する関数はどこにあるか」「影響の出るファイルはどれか」を自然言語の指示から正確に見つけ出します。

#### 4. Canvases（キャンバス）

* **概要**: コード編集の全体像や、複数のファイルにまたがる修正・アイデアの計画を、視覚的または構造的に整理して提示・検討できる機能。
* **主な役割**: 複雑なリファクタリングや新機能追加の際、AIがどのファイルをどう変更するかという「設計図（キャンバス）」を作成し、ユーザーと対話しながら段階的に適用します。

#### 5. Worktrees（ワークツリー）

* **概要**: Git の「worktree」仕組みを利用し、現在の作業ディレクトリとは別に、エージェント専用の独立した作業フォルダー（ブランチ）を自動作成する機能。
* **主な役割**: 「手元で別の実装をしている最中（WIP）だけど、AIに別のバグ修正やリファクターを裏で並行してやらせたい」という場合に、現在の未コミットのコードを汚したり退避（stash）させたりすることなく、安全にAIに作業を任せられます。

## Context Window

モデルが一度に考慮または「記憶」できるトークン単位のテキスト量。コンテキスト・ウィンドウを大きくすると、 AI モデルはより長いインプットを処理し、各アウトプットに大量の情報を組み込むことができます。  
コンテキストウィンドウの容量が限界になると、モデルは新しいインプットを処理できなくなり、古い情報を忘れてしまいます。

参考：[コンテキスト・ウィンドウとは？ | IBM](https://www.ibm.com/jp-ja/think/topics/context-window)

各ツール別に Context Window の最大値を調べてみました。  
利用頻度の高い Claude Sonnet 4.6 ベースで言及してみます。

* Claude Code
* Cursor

### Claude Code

**1,000,000（1M）トークン**

Anthropic が提供する CLI ツールである Claude Code は  
もっともモデル本来の性能を引き出せる設定になっています。

Claude Pro / Team / Enterprise プランのユーザーであれば、Sonnet 4.6 のフルコンテキストを利用可能です。ただし、デフォルトではトークン消費を抑えるために制限がかかっている場合があり、必要に応じて「extra usage」を有効化（または課金設定の調整）することで 1M まで解放されます。

参考：[How large is the context window on paid Claude plans? | Claude Help Center](https://support.claude.com/en/articles/8606394-how-large-is-the-context-window-on-paid-claude-plans#h_9172002f0a)

### Cursor

**1,000,000 (1M)トークン**

Max Mode を使用すると  
モデルがサポートする最大 context size まで拡張できます。

モデルの具体的な最大値はモデル表に示されています。  
確認する場合は、Cursor Settings または Models & Pricing ページのモデル表から、モデルの最大 context window を確認できます。

参考：[モデルと料金 | Cursor Docs](https://cursor.com/ja/docs/models-and-pricing)
