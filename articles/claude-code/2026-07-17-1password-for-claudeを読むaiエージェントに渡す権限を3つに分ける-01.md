実は、エージェントにパスワードを見せないだけでは足りない。

必要なのは、誰が、どの操作のために、どこへ権限を通したかを分けて扱うことだ。7月16日に公開された 1Password for Claude は、その分け方をブラウザ側まで持ち込んだ。Claude がログインを必要とすると、1Password が要求されたログイン項目と理由を表示し、利用者の承認後に拡張機能がページへ直接入力する。秘密は Claudeのコンテキストやメモリに入らない。[発表](https://1password.com/blog/1password-for-claude) と [設定ガイド](https://support.1password.com/1password-claude/) に詳細がある。

## 1Password for Claude が渡しているのは文字列ではない

従来の自動化は、環境変数か `.env` に秘密を置き、エージェントに読ませて使わせる形になりやすい。ここでは秘密がデータとして流れる。ログ、会話履歴、ツール出力のどこかへ残る経路も増える。

1Password for Claude のブラウザ経路は違う。Claude は「このサイトでログインが必要」という状態までは知るが、資格情報そのものは受け取らない。承認後、拡張機能が対象ページへ直接入力し、Claude はログイン済みのブラウザを操作する。必要なのはパスワードの値ではなく、認証済みセッションという効果だからだ。

これは地味だけど大きい。モデルのコンテキストは、いったん秘密が入ると境界として弱い。プロンプトインジェクション、デバッグ出力、後続タスクへの持ち越しまで、秘密を取り出す経路が増える。一方で直接入力なら、モデルが見ているのは「ログインできた」という結果だけになる。

ただし、秘密が見えないことと、操作が安全なことは別問題。たとえば注文の確定や請求先の変更まで任せるなら、ログイン承認だけでは粗い。どのサイトで、何をしようとしていて、どこまで戻せる操作なのか。ここは資格情報の保護とは別に判定したい。

## 権限を3つに切る

自分なら、エージェントのアクセスを次の3層で考える。1つの「許可」ボタンに全部を押し込まない。

| 層 | 何を決めるか | 失敗したときの被害 |
| --- | --- | --- |
| 操作権限 | エージェントが閲覧・変更・送信のどこまでできるか | 意図しない更新や購入 |
| 秘密の保管 | パスワード、TOTP、APIキーを誰が読めるか | 秘密の流出と再利用 |
| 実行時の受け渡し | どのページ、どのプロセスだけが秘密を使えるか | 別サイトや別プロセスへの横流し |

![AIエージェントへの権限を3層に分ける概念図（操作権限・秘密の保管・実行時の受け渡し）](https://static.zenn.studio/user-upload/cffb59711a0b-20260717.png)

1Password for Claude は、少なくとも後ろ2つをはっきり分離している。公式ドキュメントによると、対応するのはユーザー名、パスワード、ワンタイムパスワードで、ソーシャルログインとパスキーは現時点で対象外だ。Mac 上の 1Password と Claude デスクトップアプリ、Claude のブラウザ拡張という前提もある。万能なブラウザ権限ではなく、境界を狭くした最初の実装として読むのが正確だと思う。

## MCP 側では「実行するプロセス」へ渡す

この話はブラウザだけでは終わらない。1Password の MCP Server は、Codex などの MCP クライアントに変数名の一覧や Environment の管理をさせつつ、秘密の値を返さない設計を取っている。[開発者向けドキュメント](https://www.1password.dev/environments/mcp-server) では、承認済みプロセスに対してだけ、実行時にメモリ上の FIFO 経由で変数を渡すと説明されている。

ブラウザでは「ページへ直接入力」、開発環境では「プロセスへ直接注入」。運び方は違うが、モデルはオーケストレーターであり、秘密の保管庫ではないという線は同じだ。

変数名まで隠そうとして、結局 `.env` を丸ごと渡す設計はよくない。`DATABASE_URL` が必要だと分かれば、エージェントは構成や起動コマンドを組み立てられる。実際に値を読むのは、そのアプリのプロセスだけでいい。

小さな脱線だけど、ローカル開発ではこの線引きがいちばん崩れやすい。失敗したときに中身を `printenv` で見たくなる。そこを一度許すと、せっかくモデルの外へ出した秘密がターミナル出力として戻ってくる。開発用でも、値を見ないで接続テストの成否だけ返すラッパーのほうが後で効く。

## 承認画面は「何のための権限か」まで示したい

今回の仕組みは、エージェントの認可を人間のログイン認可から切り離す一歩になる。次に重要なのは承認の粒度だと思う。資格情報の名前だけを見せるのではなく、対象ドメイン、要求元のエージェント、実行予定の操作、失効時刻を同じ画面で示す。送信や購入のように戻せない操作なら、秘密の使用承認とは別に最終確認を置く。

この境界があれば、モデルが誘導されても被害は「許可されたページで、許可された時間に使える認証」までに閉じる。逆に境界がなければ、秘密を見せない設計でも、ログイン済みブラウザを使った危険な操作は残る。

エージェントに必要なのは、資格情報の所有権ではなく、限定された能力だ。ブラウザの直接入力とプロセスへの実行時注入は、その能力をモデルの外で発行する2つの実装になっている。これからエージェント連携を足すときは、「秘密を隠せたか」だけで終わらせず、操作権限、秘密の保管、受け渡し先の3層を別々にレビューしたい。そうすると、承認画面に何を出すべきかも、ログに何を残すべきかもかなり具体的になる。
