## はじめに

株式会社GENDA 開発部エンジニアの奥山です。

先日、[Claude Fable 5 公開をきっかけにサブエージェント活用ルールを見直した話](https://zenn.dev/genda_jp/articles/b6045575e2e13d)を書きました。  
メインセッションは上位モデルのまま、実装や調査は `.claude/agents/` 配下のサブエージェント経由で Opus / Sonnet / Haiku に委譲する、という役割ベースのルールです。

<https://zenn.dev/genda_jp/articles/b6045575e2e13d>

その記事を公開したのとほぼ同時期、Claude Code v2.1.172 でサブエージェントの **ネスト機能**(サブエージェントから更にサブエージェントを呼び出せる挙動)が入りました。  
公式 CHANGELOG の記述は次の一行です。

> Sub-agents can now spawn their own sub-agents (up to 5 levels deep)
>
> — [Claude Code CHANGELOG v2.1.172](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md)

公式ドキュメントにも同じ趣旨の記述があります。

> As of Claude Code v2.1.172, a subagent can spawn its own subagents.
>
> — [Create custom subagents — Spawn nested subagents](https://code.claude.com/docs/en/sub-agents#spawn-nested-subagents)

前回記事で整えた `code-explore` / `implementer` / `heavy-implementer` / `test-runner` の運用をそのまま続けていたところ、サブエージェントが**孫として自前のサブエージェントを呼んでくれている**ことに気づきました。  
こちらからネストを意識的に使い始めたわけではなく、既存のサブエージェント定義に対してネスト機能が後から効いてきた、という順序です。  
「サブエージェントを用意しておくこと自体の価値」が想定以上に大きいと感じたので、その気付きを共有します。

## 前提: 役割ベースのモデル使い分け

前回記事で整えた役割ベースのモデル使い分けを簡単におさらいします。  
タスクの役割ごとに担当モデルを固定し、それぞれを `.claude/agents/` 配下のサブエージェントとして定義しています。

メインセッションは計画・設計判断・レビューを Opus(Fable の代替)で担当し、それ以外の作業を以下のサブエージェントに委譲します。

```
~/.claude/
├── instructions/
│   └── role-based-model-selection.md   ← CLAUDE.md から @ 参照する判断基準
└── agents/
    ├── code-explore.md       ← 広域なコード調査・参照追跡(opus)
    ├── implementer.md        ← 仕様が明確な実装・機械的編集(sonnet)
    ├── heavy-implementer.md  ← 複数ファイル実装・デバッグ(opus)
    └── test-runner.md        ← テスト実行と合否の要約(haiku)
```

賢さが結果を左右する仕事はメインセッションに残し、手数が多いだけの仕事はサブエージェントに逃がす、という分け方です。  
本記事ではこのうち、サブエージェント(`code-explore` / `implementer` / `heavy-implementer` / `test-runner`)が孫として連鎖する場面に焦点を当てます。

## 観察: 実際にネストが発生している様子

実際に気づいたのは、`heavy-implementer` に作業を委譲した 1 回のセッションでの挙動です。  
同じセッションを時系列に沿って 3 つのスナップショットで見ていきます。

### `heavy-implementer` → `code-explore`

メインセッションから作業委譲のためのサブエージェント呼び出しが実行されます。

```
● 実装を heavy-implementer に委譲します。

● heavy-implementer(ある関数の呼び出し箇所を一通り直す)
  ⎿ ● code-explore(関数の呼び出し箇所を列挙)
    ⎿ Bash(grep -rn "parseDate" src/)
       Read(src/payment/charge.ts)
       Read(src/api/orders/handler.ts)
       Running...
```

### `heavy-implementer` → `test-runner`

`code-explore` の調査後に実装をしそれが終わったタイミングで自発的に別のサブエージェントを呼び出し、作業委譲が発生しています。

```
● heavy-implementer(ある関数の呼び出し箇所を一通り直す)
  ⎿  ● test-runner(pnpm test を実行して結果を要約)
     ⎿ Bash(pnpm exec test)
        Running...
```

### メインセッション

最初に作業委譲したサブエージェントの作業がすべて完了するとメインセッションにその旨が伝わります。  
内部でテストも実行していたことがわかります。

```
● heavy-implementer(ある関数の呼び出し箇所を一通り直す)
Done (100 tool uses · 100.0k tokens · 10m 00s)

● heavy-implementer の作業が完了しました。pnpm exec test によるテストも全 PASS しているようです。変更内容を確認します。
  ...
```

---

自分はメインセッションで `heavy-implementer` を一度呼んだだけでした。  
にもかかわらず、`heavy-implementer` の中で必要に応じて `code-explore` と `test-runner` が呼ばれています。  
**メインセッションから明示的に指示していないのに**、それぞれの役割を持ったサブエージェントが孫として連鎖したわけです。

## 以前と今: v2.1.172 で何が変わったか

v2.1.172 までは「サブエージェントは更にサブエージェントを呼べない」というのが Claude Code の明示的な制約でした。  
そのため、これまでは次の図のような **2層構造** に限定されていました。  
この制約のもとでは、1 つのサブエージェントが複数の役割の作業を抱え込んだり、メインセッションが複数のサブエージェントの中継役を担ったりする必要があります。  
結果として担当する層の文脈が膨らみやすく、作業を並行に進める難易度も上がる懸念がありました。

v2.1.172 以降は、次の図のようにネストが可能になりました。  
サブエージェントが「これは別ロールの仕事だ」と判断したら、孫として別エージェントへ更に委譲できます。

呼び出し階層が深くなっても、各層は自分の担当に集中できます。  
そして重要なのは、**孫として呼ばれる側は何も変えなくていい**ということです。  
既存のサブエージェントがそのまま再利用されます。

自前のサブエージェント定義の価値が **単発の委譲先** ではなく **再帰的に呼ばれる共通部品** にあるのだと実感しました。

## なぜ共通部品として機能するのか

サブエージェント定義が「再帰的に呼ばれる共通部品」として効くのは、**全階層から同じ構造化された選択肢として見える**からだと考えています。

`.claude/agents/*.md` のサブエージェント定義は、`name` / `description` / `model` / システムプロンプトを含む宣言として Agent ツールに登録され、`subagent_type` の選択肢として上位の Claude に提示されます。  
メインセッションだけでなく、孫を呼びうる子エージェントの位置からも同じ選択肢として見えるため、どの階層からでも同じ役割を呼び分けられます。  
モデルやツール権限は定義の `model:` / `tools:` で構造的に固定されているため、孫を呼ぶ層が変わっても挙動は揃います。

代替手段として CLAUDE.md / rules / Skill のいずれかに同じガイドラインを置く方法もありますが、ネスト階層を跨いで「役割固定 + モデル固定 + 別 context での実行」を素直に満たせるのはサブエージェント定義だけです。

| 方式 | 形態 | モデル固定 | 別 context で実行 |
| --- | --- | --- | --- |
| CLAUDE.md / rules | 会話履歴中のテキスト指示 | × | × |
| Skill | Skill ツールの選択肢として常時提示(本体は on-demand) | × | △(`context: fork` 指定時) |
| **サブエージェント定義** | **Agent ツールの構造化された宣言** | **○**(`model:`) | **○** |

CLAUDE.md / rules はテキスト指示として渡され、[公式ドキュメント](https://code.claude.com/docs/en/memory#claude-isnt-following-my-claude-md)に "no guarantee of strict compliance" とあるとおり強制力はなく、長いセッションでは他のコンテキストに埋もれて相対的なウェイトも下がります。  
Skill は description が常時ツール的に提示される利点がある一方、モデルは呼び出し元から継承されるため、「Sonnet で実装、Haiku でテスト」のような役割ごとのモデル固定はできません(`context: fork + agent:` を使えば実現できますが、それは結局サブエージェント定義の上に乗る形になります)。

前回記事では「frontmatter の `model:` でモデル選択を構造的に固定する」という話をしました。  
ネスト機能が入ると、これに「**全階層から再利用される共通部品として、構造的に存在を保証する**」という別の意味が加わります。

## 判断軸の更新: 「単発の委譲先」から「共通部品」へ

前回記事の時点では、エージェントを用意するかどうかの判断軸は次のようなものでした。

* 同じパターンの委譲が**メインセッションから**繰り返し発生する → 定義しておく
* 特定の1回限りの作業 → Agent ツールの `model` パラメータで都度指定で十分

ネスト機能を踏まえると、これに加えて次の軸が増えました。

* **サブエージェントから孫として呼びたくなりそうな役割** → 早めに定義しておく

例えば「テストを実行して合否の要約だけ返す」`test-runner` は、メインセッションから直接呼ぶよりも、`implementer` や `heavy-implementer` が自分の変更を検証するフェーズで孫として呼ぶケースの方が体感では多くなりました。  
長いテスト出力を呼び出し元のサブエージェントの文脈に取り込まずに済むため、二段重ねの「文脈隔離」が効いている形です。

`code-explore` も同様で、`heavy-implementer` がデバッグ中に「広域な参照追跡が必要」と判断したときの孫呼び出し先として、想像以上に利用頻度が高くなっています。

逆に、ここまで考えた上で「自分しか使わない・他のエージェントは絶対に呼ばない」ような特殊なエージェントは、無理に定義せず都度指定でも問題ないと考えています。  
**汎用性のあるロールほど定義する価値が増した**、という言い方が近いかもしれません。

## おわりに

サブエージェントのネスト機能を機に、サブエージェントの位置付けが「便利な委譲先」から「全階層で再利用される共通部品」に変わったと感じています。  
ルールを文章で書くより、サブエージェント定義として置いておく方が確実に効く場面が増えました。

エージェントを用意することの初期コストは大したものではない一方、ネストによって呼ばれる回数は確実に増えていきます。  
「いつか使うかも」程度のロールでも、早めにエージェント化しておくと、思わぬ階層から呼ばれてくれることがあります。

本記事が Claude Code のサブエージェント設計の一助になれば幸いです。
