## 1. はじめに

最近、**Loop Engineering**という言葉を見かけるようになりました。少し前まで Context Engineering の話題で持ちきりだったのが、もうエージェントを「どう回し続けるか」へ関心が移っています。用語の移り変わりは速く、追いかけるだけでも忙しいと感じている方は多いと思います。

ただ、用語が増えるほど気になることがあります。Prompt をどう書くか、Context に何を見せるか、Loop をどう回すか──手段の議論は精緻になっていきます。一方で、AIエージェントを自律的にループで動かそうとするほど、「**何を目指すのか**」を先に決めておくことも重要ではないのか、と感じるようになりました。

Context も Loop も、「何を目指すか（Intent）」が決まって初めて機能します。この記事では、より自律的な AIエージェントをこれから実装していくために必要になる、もう一つの観点として **Intent Engineering** を紹介します。

注）対象として想定しているのは、企業で AIエージェントを設計・導入する技術者の方です。Prompt Engineering・Context Engineering はすでにご存じの前提で進めます。基礎用語の説明は最小限にします。

## 2. なぜ用語が増えたのか ─ Prompt → Context → Loop を並べてみる

まず、いま使われている3つの言葉を、設計対象の違いとして並べてみます。

* **Prompt Engineering（何を言うか）**: モデルへの指示の与え方。最初に注目された軸です。
* **Context Engineering（何を見せるか）**: モデルが推論する瞬間に、どの情報を渡すか。Andrej Karpathy や Tobi Lütke が広め、Anthropic が **"curating the optimal set of tokens during inference"**(推論時に最適なトークン集合を選び抜くこと)として定式化しました〔Anthropic, "Effective context engineering for AI agents", 2025-09〕。この記事は、これを prompt engineering の自然な発展と位置づけています。
* **Loop Engineering（どう回し続けるか）**: エージェントを単発で終わらせず、フィードバックを受けて反復させる設計。Claude Code を作った Boris Cherny や OpenClaw の Peter Steinberger が言及し、**Addy Osmani が "Loop Engineering" として命名・整理**しました（2026-06）。別系統では Swyx の "loopcraft" を経て LangChain が "loop engineering" を使っています（2026-06-16）。

この3つは、設計対象の粒度が少しずつ大きくなっていると見ることができます。Prompt は一回の指示、Context は推論時に渡す情報、Loop はエージェントが反復し続ける仕組みを扱います。つまり、関心は「どう言うか」から「何を見せるか」、そして「どう自律して動き続けるか」へ広がってきました。

ただ、手段の自動化がどれだけ上に伸びても、**「何を目指すのか」だけは誰かが決めなければなりません**。エージェントはサブタスクや中間目標を立てることはできます。しかし、その上位にある「何を良しとするか」「どこまでやるべきか」という基準は、設計者や利用者が与えなければなりません。  
この「何を目指すか」を、いったん Intent と呼ぶことにします。具体的には、目標、制約、完了条件、確認すべき未確定事項のように、エージェントが動く前に決めておくべき前提のことです。

そして、この「何を目指すか」をめぐる問いは、はっきり名前がつくより前から意識されてきたように思います。たとえば私は2025年12月に、OpenAI の GPT-5 系プロンプトガイドを紹介する記事を書きました（[GPT-5 / GPT-5.1 / GPT-5.2 の変更点](https://zenn.dev/dxclab/articles/9fd51274dea848)）。そこには、エージェントの**停止条件**を明確に定義する、要求された範囲**だけ**を実装してスコープを広げない、曖昧な点は推測せず**確認を求める**——といった勧めが並んでいました。いま Intent と呼んでいる完了条件・制約・確認は、用語が整理されるより前から、プロンプト設計ガイドとして語られていたわけです。Intent は「次の段階」ではなく、**3つの手段の下に最初からあった土台**だと考えています。

注）Prompt Engineering や Context Engineering は、主要ベンダーや研究者・実務家の間で比較的参照しやすい定義や実践例が増えています。一方で、Loop Engineering はまだ提唱者ごとに呼び方が揺れていて、統一された一次定義があるとは言いにくい状況です。

## 3. 足りないのは「何を目指すか」の定義

Context を厚くし、Loop を上手に回す。それでも解けない問題があります。**「何が正解か」が定義されていなければ、エージェントは速く・長く・間違い続ける**ということです。

理屈はシンプルです。ループとは、ある**目標に対して反復し、採点する仕組み**です。目標がなければ、採点する基準もありません。採点できなければ、反復は「同じところを速く回るだけ」になります。Context をいくら厚くしても、向かう先が決まっていなければ、それは情報を多く抱えた状態で迷っているのと変わりません。

つまり、Context と Loop は手段であって、その手段に意味を与えるのは「目標が定義されていること」です。**手段を磨く前に、目標を設計対象として扱う**。これが必要であると感じています。

自分自身、似たつまずきを今もよく経験しています。よく練ったプロンプトを用意し、必要な資料も十分にコンテキストへ載せた。それでもエージェントが期待通りに動かないことがあります。原因をたどると、プロンプトの出来でもコンテキストの量でもなく、**「どこまでやれば完了なのか」「どこから先は人間が確認・承認すべきなのか」が曖昧だった**ことでした。エージェントは、止まるべき場所で止まらないこともあれば、逆に、期待した水準に届かないうちに「終わりました」と完了してしまうこともあります。指示の賢さではなく、完了の基準と目標の定義が足りていなかった、という反省です。

## 4. Intent Engineering ─ 要件定義と同じ構造で設計する

Intent Engineering という言葉に最初に出会ったのは、[2026年4月の MCP Dev Summit のセッション](https://mcpdevsummitna26.sched.com/event/2Hbir/intent-engineering-the-death-of-the-mono-directional-prompt-rizel-scarlett-block-inc)でした。MCP でツールを呼び出す前に、ユーザーの意図を確認してから実行する、という内容で、強く共感したのを覚えています。

ただ、聞きながら考えたのは、これは MCP に限った話ではない、ということです。ツールを呼ぶ前に意図を確かめることも、エージェントを動かす前に目標を定めることも、根は同じ「何を目指すか」の問題です。そこで MCP の文脈を離れて、エージェント全体に通じる要件項目として、自分なりに整理し直しました。

ここから先は私の整理です。業界のコンセンサスではなく、個人のフレーミングとして読んでください。

### Intent は Context と Loop を「含む」のではなく、補い合う

Intent は Context と Loop を含む上位概念ではありません。三者は役割の違う仲間で、**Intent が目標を与え、Context と Loop がそれを支える**関係だと捉えています。

* Context は、Intent の**ために**情報を選びます。
* Loop は、Intent **に向かって**採点し、反復します。
* Intent は、その両方に**目標を与えます**。

目標が決まるからこそ、見せるべき情報も、採点の基準も決まります。同じ抽象度で3つを並べると、こうなります（これも私の整理です）。

| 層 | 問い | 構成要素の例 |
| --- | --- | --- |
| Context Engineering | 何を見せるか | Memory / RAG / Tool Context / Session State |
| Loop Engineering | どう回し続けるか | Scheduling / Verification / Persistence / Retry / Evaluator |
| **Intent Engineering** | **何を目指すか** | Goal / Success Criteria / 非機能要件 / 制約 / 完了条件 / 承認境界 / 評価設計 |

### 新しい技法ではなく、要件定義の再利用

ここで言いたいのは、これを「要件定義」と呼び直そう、ということではありません。**機能要件＋非機能要件＋受け入れ基準という古典的な要件定義と構造が同じだから、その確立した規律をそのままエージェント設計に持ち込めばいい**、という話です。

* 「何ができればよいか」＝機能要件
* 「どれだけ速く・安全に・追跡可能にできればよいか」＝非機能要件
* 「何をもって完了とみなすか」＝受け入れ基準

重なるのは項目だけではありません。要件定義とは本来、**ユーザーの曖昧な要望を解釈し、確認を重ねて、具体的な目標・制約・受け入れ基準へ構造化していく営み**でした。エージェントに Intent を与える作業も、これと同じです。「いい感じにして」をそのまま渡すのではなく、何を達成すれば良いか、どこまでが範囲か、何をもって完了とするかを解きほぐして言葉にする。やっていることは、昔から要件定義と呼ばれてきた作業そのものだと感じています。

エージェントになって変わったのは、要件が**実行時に柔軟に指定できる、またはユーザーと合意できる**ようになったことです。従来は人間が要件を読んで実装しました。エージェントでは、完了条件や承認境界の定義が、そのまま実行時のループの停止条件や人間の介在ポイントになります。要件定義という規律は変わらず、その**適用先が実行ループの中に移った**、と捉えられるのではないかと思います。だとすれば、新しい技法をゼロから編み出すより、長く積み上げられてきた要件定義の作法を持ち込むほうが、ずっと近道です。

### 要件項目として整理する8要素

エージェントへの指示を要件として分解すると、私の整理では次の8項目になります。MCP Dev Summit で受け取った「実行前に意図を固める」という発想を、エージェント全般に当てはまる形へ広げたものです。古典的な要件定義の語彙を、エージェント向けに具体化しています。

| 要素 | 要件定義での対応 | エージェントで効く理由 |
| --- | --- | --- |
| Goal | 機能要件（目的） | 何に向かって反復するかの基準 |
| Priority | 要件の優先順位 | トレードオフ時の判断軸 |
| Constraints | 制約条件 | 使ってよい手段・禁止事項の境界 |
| Done | 受け入れ基準 | ループの完了・停止条件になる |
| Missing Info | 前提・未確定事項 | 勝手な仮定で暴走させないための確認点 |
| Tool rules | 非機能要件（連携） | どのツールをどう使ってよいか |
| Approval Gate | 運用要件（統制） | 人間が承認すべき境界 |
| Output Contract | インターフェース仕様 | 出力の形式・契約 |

こうした整理は、言葉を知ってから始めたわけではありません。私は2026年2月に、調査用のエージェントに対して「一次情報を最優先する」「正確性・最新性・実用性を網羅性より優先する」といったゴールと優先順位を、あらかじめ「カスタムインストラクション」として定義してアウトプットを改善した例を記事にしています（[IBM Bob と MCP で自分専用の Research モードをつくる](https://zenn.dev/dxclab/articles/9d7d0228bd8457)）。当時は要件定義という意識はありませんでしたが、振り返れば Goal と Priority、Output Contract を実行前に固定していたことになります。同じことを、実案件のプロンプト設計でも続けてきました。

Intent Engineering は、まだブログや個別の登壇で語られている段階の言葉で、確立した規律でもベンダー公認の用語でもありません。この呼び名が定着するかも分かりません。  
ただ、エージェントに何を任せるか、どこまでやれば完了か、どこで人間が承認すべきかを決めることは、呼び名に関係なく必要だと考えています。

## 5. 企業システムで何が変わるか

PoCの後、本番導入で必ず問われるのは安全性と運用性です。ここでは、Intent を「要件」として明確にしておくことで設計しやすくなる場面を、一般的な設計パターンとして整理します。

* **評価ファースト設計**: Success Criteria と評価設計を先に置きます。何をもって良しとするかが決まっていないと、改善のしようがありません。テスト駆動に近い発想を、エージェントの目標定義に持ち込む形です。
* **完了条件・停止条件**: 暴走と未収束を防ぐ要です。Google ADK の `LoopAgent` のように、サブエージェントを終了条件まで反復させる仕組みでも、**停止条件そのものは設計者が定義する必要があります**。Loop の作り込みではなく、Intent 側の Done をどう書くかの問題です。
* **承認境界・権限制御**: どこで人間が介在すべきかを決めます。なぜ人間の介在が要るのか。**人間が、AIの知らない文脈（組織の事情・過去の経緯・言外の制約）を持っているから**です。だから、その文脈が判断を左右する場面を承認境界として定義します。これは非機能要件（統制）の一部であって、要件定義に内包される話だと捉えています。
* **出典追跡・非機能要件**: 何を根拠にその出力に至ったかを追えること。本番運用では、正しさと同じくらい「説明できること」が要件になります。

これらはセキュリティやガバナンスの話に見えますが、新しい特別な領域というより、**非機能要件・承認境界・権限制御という、要件定義に元から含まれる項目**として自然に出てくるものだと考えています。

### 迷ったらこう選ぶ ─ うまく動かないエージェントの切り分け

エージェントの調子が悪いとき、どこに手を入れるべきか。次の順で見分けるようにしています。

1. **そもそも目標・完了条件が曖昧ではないか（Intent）**: まずここを疑います。「何が正解か」「正解のない業務なら、何が理想形か」を人間が言えないなら、Context も Loop も有効に働きません。完了条件・承認境界を言語化することが重要です。
2. **必要な情報が渡っていないか（Context）**: 目標は明確なのに、エージェントが的を外すなら、見せている情報が足りない・古い・余計なものは無いかを疑います。
3. **反復や検証の仕組みが弱くないか（Loop）**: 目標も情報も揃っているのに収束しないなら、採点・再試行・停止の作り込みに進みます。

多くの場合、最初に手を入れたくなるのは Context や Loop ですが、**実際に詰まっているのは Intent（目標が曖昧なこと）なこともある**、というのが私の実感です。

## 6. まとめ

要点を再掲します。

* Prompt / Context / Loop は、どれか一つが前のものを置き換える関係ではありません。それぞれ、AIエージェントの別の側面を設計するための観点です。
* その全部に目標を与えるのが Intent で、その設計は要件定義と同じ構造を持ちます。Intent は3軸を含むのではなく、**補い合うなかで目標を与える**層です。
* Context は Intent の**ために**情報を選び、Loop は Intent **に向かって**採点・反復します。
* 企業導入で重要なのは、賢いプロンプトだけでなく、**目標・完了条件・承認境界の設計**だと考えています。

用語は今後も増えると思います。そのたびに「これは何を目指すための手段なのか」と問い直す癖をつけておくと、流行に振り回されずに済むと感じています。次々に増える言葉を追いかけるのに少し疲れている方の、頭の整理に役立てば嬉しいです。
