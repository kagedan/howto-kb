## AWS連携の壁を取り除く、Bedrock対話式セットアップウィザード

**BedrockやAWS経由でClaude Codeを使いたい方、あるいはこれから導入を検討している方に特に嬉しい変更です。**

AWS Bedrockを利用してClaude Codeを組織に導入しようとしていた方向けの改善です。

### 何が変わったか

ログイン画面で「3rd-party platform」を選ぶと、新たに**Bedrockセットアップウィザード**が起動するようになりました。ウィザードでは以下のステップを対話形式で案内してくれます。

1. AWS認証情報の設定
2. リージョンの選択・設定
3. クレデンシャルの検証
4. 使用モデルの固定（model pinning）

### なぜ嬉しいか

これまでBedrockを使うには、環境変数や設定ファイルを手動で書き換える必要があり、「どこに何を書けばいいかわからない」というつまずきポイントになりがちでした。ウィザードによって**コマンドラインを離れることなく、画面の指示に従うだけでAWS連携が完結**するようになったため、導入ハードルが大幅に下がります。

```
# ログイン画面で「3rd-party platform」を選択するだけで
# ウィザードが自動的に起動します
```

ウィザードを使うにはAWS側でBedrockのアクセス権が事前に設定されている必要があります。IAMの権限設定は別途確認してください。

## `/cost`がさらに詳細に、モデル別・キャッシュ別のコスト内訳

**コスト管理を真剣に行いたいサブスクリプションユーザー向けの改善です。**

Claude Codeのサブスクリプションプランを利用していて、トークン消費の内訳を把握したい方向けの機能です。

### 何が変わったか

`/cost`コマンドで表示されるコスト情報に、**モデルごとの内訳**と**キャッシュヒット・ミスの内訳**が加わりました。

たとえばメインモデルとサブエージェントで使われる小型モデルが別々に集計され、さらに「どれだけプロンプトキャッシュが効いていたか」も確認できます。

### なぜ嬉しいか

これまで`/cost`は合計値しか見られず、「なぜこんなにトークンを使っているのか」の原因が掴みにくい状態でした。内訳が見えることで、\*\*「キャッシュが効いていないのでプロンプト設計を見直そう」「サブエージェントが思ったよりトークンを消費しているな」\*\*といった具体的なアクションにつなげやすくなります。

> キャッシュヒット率を意識することで、長時間セッションのコスト最適化が現実的になります。

## リモートセッション名にホスト名が自動付与、複数マシン管理が快適に

**複数の開発マシンやリモートサーバーでClaude Codeを使い分けている方に嬉しい変更です。**

リモートコントロール機能を使って複数環境からClaude Codeを操作している方向けの改善です。

### 何が変わったか

Remote Controlのセッション名が、デフォルトで**ホスト名をプレフィックスとした名前**になりました。

```
# 変更前（例）
graceful-unicorn

# 変更後（例）
myhost-graceful-unicorn
```

プレフィックスは`--remote-control-session-name-prefix`オプションで任意の文字列に変更することもできます。

```
# カスタムプレフィックスを指定する例
claude --remote-control-session-name-prefix myproject
```

### なぜ嬉しいか

以前はランダムな名前だけが表示されていたため、どのマシンのセッションかを一目で判別できませんでした。ホスト名が付くことで、**セッション一覧から目的のマシンのセッションを素早く特定**できるようになります。
