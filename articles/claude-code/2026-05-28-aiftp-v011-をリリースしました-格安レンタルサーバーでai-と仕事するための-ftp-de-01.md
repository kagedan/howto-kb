## なぜ作ったか

AI 開発ができるようになった現在、非エンジニアでも開発ができるようになりました。

複雑な開発はまだ人間のエンジニアの力が必要ですが、Word / Excel / PowerPoint のような Office 系のソフトを使った単純仕事は AI に置き換わってきています。

Web 制作においても、素晴らしいデザインを AI が提案してくれる時代になりました。我が社の Web サイトも AI が主で作ったものに全て置き換わりました。要した時間は各サイト半日程度の稼働です。

ただし、現段階においては、1 回のプロンプトで思ったデザインの Web サイトが完成ということは難しく、何度か AI エージェントと修正ラリーを繰り返す必要があります。

その都度、AI が作り上げた HTML ファイル等を手動で Web サーバー上にアップロードしなければならない作業が面倒臭く感じました。これが aiftp を作りたいと思ったきっかけです。

エンジニアであればローカル PC 上に環境を作って出来を確認しながら進めることもできると思いますが、非エンジニアにとってはこれも面倒ですし、どのみち本番用の Web サーバーへのアップロードはやらなければなりません。

また非エンジニアの Web デザイナーが使う Web サーバは SSH などの遠隔操作ができない格安サーバを使っていることも多く、その都度 FTP ソフトなどで HTML 等をアップしている人も多いことを、仕事上知っていたのでニーズもあると思いました。

### 技術的にどういう壁があるのか

日本のレンタルサーバー (Lolipop / Sakura / Xserver / StarServer など) は AWS / Vercel / Cloudflare Pages とは違う世界です。

* **SSH が使えないプランが約半数**: `rsync` や `git pull` の自動化が成立しない
* **WP-CLI も使えない**: WordPress 公式 MCP (Abilities API) も WP-CLI 経由なので、レンタルサーバーでは到達できない
* **FTP/FTPS/SFTP のみが選択肢**: でも AI agent から FTP を直接叩くと、`.env` や `wp-config.php` の誤デプロイ事故が起きる

aiftp はこの「**SSH/WP-CLI/GUI に頼らずに、AI と安全に deploy する**」というニッチに振り切って設計しています。

## v0.11 の新機能 (4 つの柱)

### 1. WordPress 特化テンプレ 7 種

![](https://static.zenn.studio/user-upload/defa4be6dbb7-20260528.png)  
`aiftp init --template wordpress-swell` のように 1 コマンドで `.aiftp.toml` を生成。

```
$ aiftp init --template list
wordpress-swell    - WordPress with SWELL theme
wordpress-lightning - WordPress with LIGHTNING (Vektor) theme
wordpress-cocoon   - WordPress with Cocoon theme
wordpress-standard - WordPress (theme-agnostic baseline)
static             - Static site / Jamstack build output
laravel            - Laravel on shared hosting
php-simple         - Standalone PHP scripts (contact form, mini-API)
```

各テンプレが「テーマ固有のキャッシュ / バックアップディレクトリ / debug.log / `wp-rocket/cache/**`」など **hard-exclude すべきパターン** を持っているので、AI agent が知らずに本番に配信してしまう事故を防ぎます。

MCP tool としても `aiftp_init_template_list` を提供しているので、Claude Code から直接テンプレ一覧を取得できます。

### 2. SFTP 対応

`protocol = "sftp"` + `ssh_key_path = "~/.ssh/id_ed25519"` で SFTP デプロイが可能になりました。

* SSH 鍵 permission check (`0o600` / `0o400`) を connect 前に実行 — `ssh(1)` 準拠
* symlink 拒否で TOCTOU 攻撃を緩和
* tilde 展開を `assertSafeRemotePath` で path traversal 防御
* `aiftp doctor` に SFTP 専用 4 check (`ssh-port-reachable` / `ssh-key-permissions` / `sftp-handshake` / `sftp-remote-root`)

**既知の limitation**: v0.11 では known\_hosts / TOFU pinning を実装していません。経路 MITM のリスクがあるため、本番運用時は host key fingerprint を hosting provider の文書で確認してください。v0.12 で `.aiftp/known_hosts` TOFU を実装予定。

### 3. PromptFlow — 三重防御 init UX

`aiftp init` を新フレームワーク `PromptFlow` に移行しました。三重防御:

* **A. 入力ヒント表示**: 各フィールドの直前に「何を入れるべきか」「例」を表示
* **B. `:back` 戻りナビ**: 任意のフィールドで `:back` と入力すると前のフィールドに戻れる
* **C. summary review**: 全フィールド入力後に番号付きサマリーを表示、`1-10` で個別フィールド編集 (v0.10.4 から)

レンタルサーバー固有の quirks (StarServer の TLS hostname mismatch、Lolipop の MLSD 非対応など) は init で `server_kind` を選ぶだけで自動設定されます。

### 4. smoke CI

`.github/workflows/smoke.yml` で 3 OS (macOS / Ubuntu / Windows) × 2 Node (22 / 24) の matrix が release publish / 毎週月曜 09:00 JST / 手動 dispatch で起動します。

* `npm install -g @aiftp-tools/cli@latest` で実際の公開パッケージを入れ直す (source-tree テストでは検出できない install 経路 regression を catch)
* MCP JSON-RPC stdio probe で `initialize` → `tools/list` の round-trip を確認
* `aiftp init` の非 TTY 拒否を gate (パイプ経由で誤って profile が作られないことを保証)

## 設計の柱: AI agent との関係

aiftp は MCP server を「副産物」ではなく「第一級市民」として設計しています。

**prepare → confirm の 2 段ゲート**: 破壊的操作 (push / backup restore / rollback / config migrate / FileZilla import) は必ず 2 段階。

1. `aiftp_*_prepare` で plan を作る → `plan_id` / `diff_hash` / `confirm_token` を返す
2. `aiftp_*_confirm` で同じ token を echo back → 実行

これにより、AI agent が「うっかり 1 tool call で本番を上書き」するシナリオを構造的に防ぎます。Plan TTL (5 分) も付いているので、レビュー中に状況が変わったら自動的に invalidate されます。

**Read-only と write の明確な分離**: `aiftp_status` / `aiftp_profile_list` / `aiftp_init_template_list` などの read-only tool は token なしで呼べる一方、write tool は必ず prepare/confirm ペアです。

## セキュリティの姿勢

v0.11 の release では Codex に 3 ラウンドのレビューを依頼しました。

1. **Pillar β review** (テンプレ系) — NO-GO → HIGH 2 件 fix → GO
2. **Pillar γ+δ review** (SFTP + smoke CI) — NO-GO → CRITICAL 1 + HIGH 3 件 fix → GO
3. **Security 専用 review** (OWASP/CWE 観点) — NO-GO → HIGH 3 + MEDIUM 2 件 fix → GO

特に security review では「**schema 上は ssh\_key\_path が定義されているのに、実 deploy 経路では password fallback している**」という致命的なバグを発見しました。Codex に独立した目で見てもらわなければ v0.11 が「動いてるように見えて鍵を使わない」状態で出荷されていた可能性が高いです。

### クロスチェックを取り入れた理由

私自身は非エンジニアですが PM の経験があり、システムの品質を維持するためにはテストが重要だということを知っていました。AI 開発をするにあたって感じたことは、**AI は自分自身について甘く判断することが多い**ということです。モデルが Opus 4.7 や GPT-5.5 になっても、サブエージェントを使ってもそれは変わりませんでした。

そこで Claude が GPT を、GPT が Claude をチェックするクロスチェックをするようにしたら、競合だからかチェックが厳しく、多くのバグの発見や品質の向上に繋がりました。

今回の release でも、Claude が実装 → Codex が独立 review → NO-GO → 修正 → 再 review という流れを 3 ラウンド回しました。「気の毒」と感じるくらい厳しい指摘が並ぶこともありましたが、結果として「**schema は通るのに実 deploy で鍵を使わない**」という致命的バグを release 前に潰せたのは大きな価値でした。

修正したセキュリティ blocker (今 release で fix 済み):

* **CWE-22/94**: profile 名の TOML key injection → schema + init parse + FileZilla importer の 3 層で `isValidProfileName` 強制
* **CWE-22**: `remote_root` / `<RemoteDir>` の path traversal → `assertSafeRemotePath`
* **CWE-22/367**: `ssh_key_path` traversal + symlink TOCTOU → `safeExpandLocalPath` + `lstatSync`
* **CWE-200/732**: MCP confirm tempfile mode → `{ mode: 0o600 }` 明示

非交渉ルール (今後も on で固定):

* 暗号化 default ON (`AES-256-GCM` for backup)
* `wp-config.php` / `.env*` / `db.php` などは hard-exclude (user pattern では override 不可)
* パスワードは Keychain のみ (CLI 引数で受けない)
* `.gitignore` への `.aiftp/` 追加を init で強制

## 使い方 (30 秒 onboarding)

```
# 1. インストール (Node 22+ 必須)
npm install -g @aiftp-tools/cli

# 2. プロジェクトディレクトリで init
cd /path/to/your-wp-site
aiftp init --template wordpress-swell
# → 対話プロンプトで host / user / password などを入力
# → .aiftp.toml + .gitignore に .aiftp/ が追加される
# → password は Keychain に保存

# 3. 接続確認
aiftp doctor

# 4. dry-run push (実際にはアップしない)
aiftp push --dry-run

# 5. 本番 push (prepare → confirm の 2 段ゲート)
aiftp push
```

**Claude Code / Cursor / Codex から使う場合**: `.mcp.json` (または `~/.claude/settings.json`) に登録:

```
{
  "mcpServers": {
    "aiftp": {
      "command": "aiftp",
      "args": ["mcp"]
    }
  }
}
```

## 競合との関係

詳細は [競合比較ドキュメント](https://github.com/aiftp-tools/aiftp/blob/main/docs/competitive-comparison.md) にまとめましたが、ざっくり:

* **alxspiker/mcp-server-ftp**: 同カテゴリ最大の競合。aiftp は「AI に安全に deploy させる」レイヤーで、認証情報 Keychain + prepare→confirm + hard-exclude + 日本のレンタルサーバー quirks が差分
* **Computer Use / Operator / Claude in Chrome**: GUI 自動操作専用 (WP 管理画面)。ファイル deploy の守備範囲外
* **WordPress 公式 MCP (Abilities API)**: WP コア機能 API。aiftp はその「外」のファイル配置担当。**併用が最強**
* **git-ftp / SamKirkland 系**: ターゲットがそもそも違う (「git push と同じノリで FTP」 vs 「AI が自律的に deploy する」)

## 既知 limitation と次のロードマップ

### v0.11 で明示する limitation

* **SFTP host key TOFU 未実装**: 経路 MITM 防御のため、本番では fingerprint を out-of-band で確認してください
* **SFTP upload が atomic rename でない**: 接続断で partial file が残る可能性
* **FileZilla XML `processEntities: true`**: DoS 理論的耐性が低い (fast-xml-parser@5.8.0 に既知 CVE なし)

### v0.12 計画

* `.aiftp/known_hosts` TOFU pinning (CWE-295 解消)
* 他 30 箇所の prompt を PromptFlow に移行 (`aiftp auth` / `aiftp config edit` など)
* doctor の `errorKind` 分解 (passphrase mismatch / known\_hosts / permission を区別)

### v0.11.1 計画

* SFTP atomic rename (CWE-664)
* FileZilla XML `processEntities: false` + 入力サイズ上限 (CWE-400)
* WordPress 4 templates 共通の `debug.log` exclude
* MCP `aiftp_init_template_show` tool
