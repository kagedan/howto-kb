同じClaude Codeを使っているのに、人によって出力の質が全然違う。  
これは「腕の差」ではなく「CLAUDE.mdの育て方の差」です。

CLAUDE.mdは、Claudeがセッション開始時に1回読み込む永続メモリ。  
ここに書いた内容で、その後のClaude全体の動きが変わります。

Anthropic公式も「コードのように扱え」と明言している重要ファイルです。今日は、徹底的に調べた上で、本当に書くべき30のルールをカテゴリ別に整理しました。

更新・削除の頻度、Obsidianでの管理方法もセットで公開します。

---

  

## CLAUDE.mdの基本原則

ルール30選に入る前に、Anthropic公式と開発者コミュニティが推奨する5つの原則を共有します。

300行以内に収める。  
それを超えるとClaudeが一部を無視し始める 「コードから読めないこと」だけを書く。  
自明な情報は不要 「やってはいけないこと」の方が効く。  
禁止事項を優先 コードのように扱う。  
同じ訂正が2回起きたら追加、自明化したら削除 役割別にカスタマイズする。  
メンバー全員に同じものを使わせない。

これを前提に、30のルールを見ていきます。

## カテゴリ1：基本情報・ユーザープロファイル（5個）

---

### 01. 会社名・業種・主要サービスを書く

```
私は広告運用代理店を経営しています。
主にYahoo広告・Meta広告のコンサル・運用代行を提供しています。
クライアントは中小企業が中心です。
```

効く理由：Claudeは「相手の業界」を知らないと、一般論しか返せない。  
一文入れるだけで提案の現実味が一段上がる。   
活用メモ：会社の説明は最初の3行で完結させる。

---

### 02. ターゲット顧客・市場を書く

```
ターゲットは年商1〜10億円の中小企業。
特にBtoB商材を扱う会社が多い。
クライアントの担当者は経営者または広報担当が中心。
```

効く理由：Claudeが「誰向けの提案か」を理解した上で文章を書ける。  
 活用メモ：ターゲットを書いていないと、Claudeの提案が誰にも刺さらない文章になる。

---

### 03. 自分の役割・知識レベル

```
私は経営者であり、プログラミングは詳しくありません。
日本語のビジネス文章は得意。Claudeに頼みたいのは実務の自動化と分析です。
```

効く理由：Claudeが「コード説明をするか・しないか」「専門用語を使うか・かみ砕くか」を判断できる。   
活用メモ：「非エンジニア」と一言入れるだけで、Claudeの説明が変わる。

---

### 04. よく使うツール・サービス

```
普段使っているツール：
Claude Code・ChatGPT・Gemini
Google Workspace（Gmail・Docs・Sheets）
Yahoo広告・Meta広告
Obsidian
```

効く理由：Claudeが「どのツール前提で答えるか」を判断できる。  
活用メモ：使っているツール一覧は、Claudeの提案精度を一段上げる。

---

### 05. 苦手領域

```
苦手な領域：
プログラミング（Python・JavaScriptの細かい構文）
英語のドキュメント読解
複雑な数値計算
```

効く理由：「苦手」を明示しておくと、Claudeが補完するモードで動く。 活用メモ：苦手領域は遠慮なく書く。恥ずかしがる必要はない。

---

## カテゴリ2：アウトプットの型（5個）

---

### 06. 言語を固定する

```
必ず日本語で答えてください。
英語のドキュメントを読んでも、説明は日本語で。
```

効く理由：これがないと、英文ソースを読んだ後の回答が英語混じりになる。 活用メモ：最初のルールに置くと、全体が日本語で統一される。

---

### 07. 文体・トーン

```
文体：ですます調
トーン：丁寧だがフランクすぎず、経営者向けの落ち着いた表現
専門用語を使う場合は、必ず一言で意味を添える
```

効く理由：トーンが揃わないと、クライアントに見せられない文章が出てくる。 活用メモ：文体ルールを書いておくと、メール下書きがそのまま使えるレベルになる。

---

### 08. 結論先行・構造化

```
回答は必ず結論を最初に書く。
理由・背景は結論の後に続ける。
長くなる場合は見出しで構造化する。
```

効く理由：「結論先行」を指示するだけで、Claudeの回答が一気に読みやすくなる。 活用メモ：経営者向けの文章を作る時の必須ルール。

---

### 09. 字数制限・分量の目安

```
回答の目安：
質問への即答：100〜200字
提案書：1セクション300字
クライアントメール：300〜500字
```

効く理由：字数を指定しないと、Claudeは過剰に長い文章を返しがち。  
 活用メモ：用途別に字数を書いておくと、SNS投稿にも転用しやすい。

---

### 10. 出力フォーマット

```
データ分析の結果は表形式で。
手順を示す場合は番号付きリスト。
比較は対比型のテキストで（マークダウン表は使わない）。
コードはコードブロックで。
```

効く理由：フォーマットを指定すると、出力をそのまま使えるようになる。 活用メモ：自分が使う媒体（note・社内資料・メール）に合わせて指定する。

---

## カテゴリ3：禁止事項（5個）

---

### 11. 推測で答えない

```
確信が持てない情報は「確認が取れていません」と必ず明示する。
推測で断定的に答えるのを禁止する。
```

効く理由：ハルシネーション対策の最重要ルール。これがないと数字や固有名詞が間違う。   
活用メモ：禁止事項の中で最優先で書くべき1個。

---

### 12. 機密ファイルを読まない

```
.env・config.json・credentials系のファイルは絶対に読まない。
パスワード・APIキー・認証情報を含むファイルは触らない。
```

効く理由：bypass permissions運用している人ほど重要。これがないと事故が起きる。 活用メモ：メルカリが組織配布資料で強調していたルール。

---

### 13. 外部送信・APIコールの確認

```
外部にデータを送信する処理は、実行前に必ず人間に確認する。
APIコール・メール送信・SNS投稿の前は確認を取る。
```

効く理由：Claudeが勝手にデータを送信する事故を防ぐ。  
 活用メモ：bypass permissions運用なら必須。

---

### 14. ファイル削除・上書きの確認

```
ファイル削除・上書きの前は必ず確認を取る。
特に複数ファイルの一括処理は実行前に内容を提示する。
```

効く理由：Claude Codeで重要ファイルが消える事故を防ぐ。   
活用メモ：失敗談記事のネタになる前に入れておく。

---

### 15. 専門用語の乱用禁止

```
専門用語は必要な時だけ使う。
業界用語を使う場合は、必ず一言で意味を添える。
カタカナ語の濫用を避ける。
```

効く理由：クライアントに見せる文章で、Claudeが横文字を多用するのを防げる。 活用メモ：「経営者目線」を維持するための地味だが効くルール。

---

## カテゴリ4：業界用語・社内用語（5個）

---

### 16. 業界用語の定義

```
業界用語の定義：
CPA：顧客獲得単価
CV：コンバージョン（成果数）
ROAS：広告費用回収率
RSA：レスポンシブ検索広告
```

効く理由：Claudeが業界用語を「正しく」使えるようになる。  
活用メモ：自社業界の用語10〜20個を最初に登録しておく。

---

### 17. 社内呼称・略語

```
社内呼称：
当社：株式会社○○
代表：私
営業チーム：3名体制
クライアント先：取引先・顧客（「お客様」は使わない）
```

効く理由：社内文書のトーンが一気に統一される。 活用メモ：「お客様」「顧客」「クライアント」のどれを使うかは会社の文化で違う。明示する。

---

### 18. クライアント別の呼び方

```
クライアントA社：呼称「A社」、担当「○○様」
クライアントB社：呼称「B社」、担当「○○様」
（必要に応じてクライアントごとに追加）
```

効く理由：複数クライアントを扱う時の混乱を防げる。  
活用メモ：クライアントが多い場合は、別ファイルに分けてimportで管理する。

---

### 19. 商品・サービス名の正式表記

```
自社サービス名：
「TAGIRU AI支援」（正式名）
略称：「TAGIRU」「TAGIRU AI」（口頭・SNSのみ）
英語表記：「TAGIRU Inc.」
```

効く理由：Claudeがサービス名を間違えなくなる。  
活用メモ：表記揺れは信頼を失う。必ず統一する。

---

### 20. NGワード・避けたい表現

```
避けたい表現：
「絶対」「必ず」「100%」などの断定表現
「最強」「最高」「神」などの誇張
「○○すべき」のような上から目線
```

効く理由：Claudeが煽り文を出してしまう問題を防げる。  
活用メモ：自分のブランドに合わせて避けたい表現を書いておく。

---

## カテゴリ5：業務フォーマット（5個）

---

### 21. メールの定型構造

```
ビジネスメールの構造：
1. 宛名
2. 挨拶（季節挨拶は省略）
3. 結論・要件
4. 詳細
5. 締めの一言
6. 署名

長さは300〜500字を目安に。
```

効く理由：メール下書きが毎回同じ品質になる。  
活用メモ：自分のメール文化に合わせてカスタマイズ。

---

### 22. 議事録のフォーマット

```
議事録の項目：
日時・参加者・会議の目的
決定事項
担当者と期限
保留事項
次回確認事項
```

効く理由：議事録の質が安定する。会議後の対応漏れが減る。  
活用メモ：「決定事項」「担当者」「期限」は必須3項目。

---

### 23. 月次レポートの構造

```
月次レポートの構造：
1. 今月の要点（3行）
2. 主要KPI（前月比・前年比）
3. 良かった点
4. 課題
5. 来月の対応
```

効く理由：レポートの読みやすさが毎月同じになる。  
活用メモ：クライアント別のKPIは別途指定する。

---

### 24. 提案書の標準構成

```
提案書の構成：
1. 課題整理
2. 解決アプローチ
3. 具体的施策
4. 料金プラン
5. FAQ
```

効く理由：提案書の品質が安定する。  
活用メモ：業種別に細かく分岐させてもOK。

---

### 25. 請求書・見積書の形式

```
請求書のフォーマット：
請求番号・請求日・支払期限
品目・数量・単価・金額
消費税10%
合計金額（税込）
振込先情報

数字は必ず人間が最終確認する。
```

効く理由：請求書業務の標準化と、最終確認ルールの徹底。  
活用メモ：「数字は人間が最終確認」を必ず書く。

---

## カテゴリ6：確認・承認ルール（5個）

---

### 26. 重要作業前の確認

```
以下の作業前は必ず確認を取る：
クライアントへの送信メール
請求書・見積書の発行
契約書のチェック
SNS投稿
大量データの一括処理
```

効く理由：Claudeが勝手に重要なアクションを実行する事故を防ぐ。  
活用メモ：「準備までClaude、最終実行は人間」のルールを徹底する。

---

### 27. 数字の検算

```
出力する数字は必ず根拠を示す。
計算が含まれる場合は、計算過程も併記する。
重要な数字（金額・期日・人数）は、ダブルチェックを推奨する。
```

効く理由：ハルシネーション対策の中で、特に重要な数字系のミスを防げる。 活用メモ：請求書・契約書・経営判断で必須。

---

### 28. 送信前の最終チェック

```
クライアント送信前に確認する項目：
宛先（誤送信防止）
金額・日付の正確性
ファイル添付漏れ
署名の有無
```

効く理由：送信ミスは取り返しがつかない。チェックリスト化しておくと安心。 活用メモ：メール作成プロンプトとセットで使う。

---

### 29. ハルシネーション対策（ソース提示）

```
事実情報を提示する時は、必ずソース（出典）を明示する。
ソースが提示できない情報は「未確認情報」と明示する。
最新情報を扱う時はWeb検索をONにする。
```

効く理由：Claudeの「もっともらしい嘘」を見破る最大の防御策。  
活用メモ：データ分析・市場調査・記事執筆で必須。

---

### 30. バックアップ取得タイミング

```
以下のタイミングでバックアップを取る：
重要ファイルの一括編集前
コードのリファクタリング前
データベース操作の前
契約書・請求書の上書き前
```

効く理由：取り返しがつかない事故を防ぐ最後の砦。  
活用メモ：「.bak」拡張子で別ファイルとして保存するルールにする。

## 更新・削除の頻度（公式の運用ルール）

CLAUDE.mdは「育てる」ものです。書いて終わりではありません。

---

### 追加するタイミング

同じ訂正が2回起きたら追加する。

Claudeに同じ間違いを2回されたら、それはルールにすべきサインです。 PRレビューと同じ感覚で、CLAUDE.mdに追記します。

---

### 削除するタイミング

ルールが自明化したら削除する。   
3ヶ月以上参照されていないルールは削除候補にする。

書いたまま放置すると300行の壁を超えます。 「もう常識化したルール」は思い切って削除します。

---

### 見直しのサイクル

週1回：直近の追加内容を30秒でレビュー   
月1回：全体をスクロールして整理  
四半期1回：大幅なリファクタリング  
年1回：抜本的な見直し  
300行超えそうになったら：必ずリファクタリング

## Obsidianで管理すると一段ラクになる

Claude Codeのメモリ管理UIとしてObsidianを使う方法を、別の記事で書きました。その仕組みを使うと、CLAUDE.mdの管理がさらにラクになります。

具体的にできること：

CLAUDE.mdのカテゴリ別にDataviewで一覧化  
「直近7日に追加されたルール」が一目でわかる  
「3ヶ月以上更新されていないルール」を抽出  
グラフビューで「どのルールが他のメモリと繋がっているか」を可視化

DataviewクエリでCLAUDE.mdのダッシュボードを作る例：

```
TABLE category, last_updated
FROM "claude_rules"
SORT last_updated DESC
```

これで「育っているCLAUDE.md」と「放置しているCLAUDE.md」が見える化されます。

## まとめ

CLAUDE.mdは、Claude Codeを自分専用に育てるための永続メモリです。 書くべきルールは、基本情報・アウトプットの型・禁止事項・業界用語・業務フォーマット・確認ルールの6カテゴリに分けて30個。  
これだけ揃えば、Claudeの精度が一段上がります。

ただし、書いて終わりではありません。  
 Anthropic公式が「コードのように扱え」と言うように、追加と削除を繰り返して育てるものです。同じ訂正が2回起きたら追加。自明化したら削除。300行を超えそうになったらリファクタリング。

そして、Obsidianと組み合わせると管理が一段ラクになります。   
カテゴリ別の一覧、更新頻度の可視化、関連性のグラフ表示。これで「育っているCLAUDE.md」が見える化されます。

CLAUDE.mdは、書くものではなく、育てるもの。   
今日紹介した30のルールを叩き台にして、自分用にカスタマイズしてみてください。
