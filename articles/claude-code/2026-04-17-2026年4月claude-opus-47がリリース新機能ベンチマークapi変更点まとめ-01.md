## はじめに

2026年4月16日、Anthropicから **Claude Opus 4.7** が一般提供開始されました。2026年2月にリリースされた Opus 4.6 から約2ヶ月、ナンバリング上はマイナーアップデートながら、**コーディング・エージェント・ビジョンの3領域で世代交代級の性能向上**を叩き出した、実務利用者にとって見逃せないリリースです。

特筆すべきは以下の3点です。

* **価格据え置き**（入力 $5 / 出力 $25 per 1M tokens）で Opus 4.6 から大幅な性能向上
* **`xhigh` effort level** の新設と **2,576px 高解像度ビジョン** という、エージェント運用を直接的に改善する2つの目玉機能
* **新トークナイザー導入**による実質コスト増の可能性という、移行時に必ず押さえたい注意点

本記事はClaude API・Claude Code・Amazon Bedrock経由でOpusを利用している開発者向けに、Opus 4.6 → 4.7 移行判断に必要な情報をまとめます。

## 主要な新機能

### 1. `xhigh` Effort Level（新規）

従来の `low / medium / high / max` に加え、**`xhigh`**（エクストラハイ）が追加され、`high` と `max` の間で推論深度とレイテンシのトレードオフを細かく制御できるようになりました。Claude Code では **デフォルトが `xhigh` に設定**されており、Anthropic 公式もコーディング・エージェント用途では `high` か `xhigh` を推奨しています。

```
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=4096,
    effort="xhigh",  # 新設された推論深度レベル
    messages=[
        {"role": "user", "content": "このコードベースのリファクタ方針を提案してください"}
    ],
)
```

### 2. 高解像度ビジョン対応

画像入力の最大解像度が **長辺1,568px（約1.15MP）から2,576px（約3.75MP）へ約3倍以上に拡大**。密集したドキュメント・チャート・UIスクリーンショットの認識精度が劇的に向上しました。XBOW視覚敏捷性ベンチマークで **54.5% → 98.5%** という過去最大の跳躍を記録しています。

### 3. エージェンティックコーディングの大幅強化

* **SWE-bench Pro: 64.3%**（Opus 4.6 比 +10.9pt）
* **Rakuten-SWE-Bench**：本番タスク解決率が **3倍**
* 複雑なマルチステップタスクで **ツールエラーが約1/3**、トークン使用量も削減
* 不要なラッパー関数・フォールバック実装を書かなくなる傾向（コード品質面の定性的改善）

### 4. Claude Code の新機能

* **`/ultrareview` コマンド**：複数パスでのバグ検出・設計問題レビューを実施する新規コマンド。全ユーザーに無料レビュー枠が付与されます。
* **Auto Mode（Max ユーザー向け）**：ファイル検索・テスト実行・変更検証のタイミングをClaude側が自動判断し、長時間タスクの中断を減らします。

### 5. Task Budgets（Public Beta）

タスク単位・会話単位で**最大トークン消費の上限を設定**できる機能。エージェントの暴走コストを事前に封じ込められるため、本番エージェント運用の必須機能になりそうです。

### 6. 新トークナイザー導入 ⚠️

Opus 4.7 は**新しいトークナイザー**を採用しており、同一テキストでも**コンテンツ種別により最大1.35倍のトークン数**になるケースがあります。価格は据え置きですが、**実コストは増える可能性**があるため、既存プロンプトでの再計測を強く推奨します。

## API移行時の注意点（Breaking Changes）

### ⚠️ 新トークナイザーで実質コストが最大1.35倍に

Opus 4.6 と同一テキストでも、トークン化結果が変わるため入力・出力トークン数が増える可能性があります。**プロンプトキャッシュのキーも変わるため、キャッシュヒット率が一時的にゼロになる**点にも注意。移行前に必ず本番プロンプトで `count_tokens` API による再計測と月次コスト試算の更新を行ってください。

### ⚠️ 指示解釈が厳密化

Opus 4.7 はプロンプトの指示を **より文字通りに解釈**します。Opus 4.6 で暗黙に意図を汲んでもらえていた曖昧なプロンプトが別挙動になる可能性があり、**移行前にゴールデンパス＋主要エッジケースのA/B比較**を推奨します。

### ✅ 価格据え置き・モデルID形式も同一

```
# Before（Opus 4.6）
response = client.messages.create(
    model="claude-opus-4-6",
    effort="high",
    ...
)

# After（Opus 4.7）
response = client.messages.create(
    model="claude-opus-4-7",
    effort="xhigh",  # 新設レベルを活用推奨
    ...
)
```

モデルIDを差し替え、`effort` を `xhigh` に上げる程度で既存コードはそのまま動作します。

### ✅ Claude Code / Bedrock / Vertex AI / Foundry で同時GA

Anthropic API だけでなく **Amazon Bedrock（東京リージョンを含む4リージョン）・Google Cloud Vertex AI・Microsoft Foundry・Claude Code の全チャネルで即日GA**。マルチクラウド構成でも移行のタイミング差が発生しません。

## まとめ

ナンバリングはマイナーアップですが、**コーディング・エージェント・ビジョンの実効値は世代交代級**です。Opus 4.6 を本番運用しているチームは、以下の観点で移行検討すべきタイミングに来ています。

* **コスト再測定**：新トークナイザーで実質コストが 1.0〜1.35倍。本番プロンプトで事前に `count_tokens` 実施
* **プロンプト再調整**：厳密化された指示解釈に合わせ、曖昧表現を減らす
* **`xhigh` effort の活用**：コード生成・エージェント用途ではデフォルトを `xhigh` に引き上げる価値あり
* **ビジョン案件の再評価**：3.75MP対応＆視覚敏捷性 98.5% で、帳票・図表・UI系タスクの精度が一段上に
* **Breaking Changes対応**：キャッシュ再構築とA/Bテストを移行フローに組み込む

特にコード特化・エージェント運用のワークロードでは、**即検討価値あり**です。
