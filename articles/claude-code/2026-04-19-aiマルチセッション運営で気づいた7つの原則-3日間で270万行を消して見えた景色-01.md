## はじめに

3日間で、**9つのマイクロサービスリポジトリ**を横断して、**14,337 ファイル**に手を入れ、**約72本の PR** を作成・マージし、**約 2,706,951 行**を削除しました。主役は Claude Code の親セッション1つとサブセッション9つ、それに GitHub Copilot のレビューと、画面の前にいる人間1人です。

数字だけ並べると派手ですが、実際には「AI の処理能力をフル活用できた」という達成感と、「並列度を欲張りすぎて中盤は認知が追いつかなかった」という反省が、同じ重さで残りました。

この記事は、その3日間で起きたことを7つの原則に蒸留したものです。頭の中だけで作った体系論ではなく、実際にサブセッションから飛び込んでくるコピペを捌きながら気づいたことを、あとから言語化したものです。AIマルチセッション運営に今まさに取り組んでいる方の参考になれば幸いです。

---

## 前提：3日間で何をやっていたか

前提として、筆者のチームは Laravel モノリスを9つのマイクロサービスに分割するプロジェクトを進めています。分割自体は別の期間で済ませており、今回の3日間は「分割後に各リポジトリに残っている旧モノリス由来の重複コードや dead code を hard fork 的に一気に整理する」フェーズでした。

体制はこうです。

* **人間（筆者）1人**: 経営判断・スコープ決定・サブセッション間のコピペ同期
* **親セッション（Claude Code デスクトップ版）1つ**: 全体進行管理・サブセッション間調整・Notion 集約
* **サブセッション（Claude Code Terminal 版）9つ**: 各リポジトリ専任で実作業
* **GitHub Copilot**: 各 PR の自動レビュー

成果と想定外は、同時に出ました。

* 成果: 9リポジトリ横断で **約270万行** を削除、**14,337 ファイル** に変更、約**72本の PR** 完遂
* 想定外: ピーク時に8-9並列を走らせた結果、親セッション誤指示・横断課題の手戻り・Copilot レビューの遅延が連鎖的に発生

この「成功も混乱も両方起きた」経験から抽出した原則が、以下の7つです。

---

## 原則1: 段取り80%の法則

従来の開発は、ざっくり「人間の段取り50% + 人間の実行50%」の比率で回っていました。AIエージェントを実行役に据えると、実行のコストが桁で下がります。比率は「**人間の段取り80% + AIの実行20%**」に再分配される、というのが3日間でもっとも強く感じたことです。

これは「段取りが大事」という古くからの教訓の焼き直しに見えますが、AIネイティブ時代における意味合いは少し違います。

* 従来: 段取りが雑でも、実行中に人間が軌道修正できる
* 現在: 段取りが雑だと、AI は雑なまま高速に走り抜けて戻れない距離まで行ってしまう

AI の実行速度が上がるほど、事前段取りの**相対的な価値**が上がります。段取りが完璧でも10%速くなる程度ですが、段取りが雑だと事後整理で2-3倍の工数を払うハメになる、という非対称性が発生します。

救いは、「段取り自体も AI にやらせられる」ことです。横断の依存関係調査・共通パターンの grep・プロンプトテンプレートの整備などは、AI が得意とする作業の典型です。段取りを AI に手伝わせ、統合判断だけを人間がメタ認知で取りまとめる。この役割分担が、3日間で最後まで機能しました。

---

## 原則2: 並列度 ≠ 速さ

並列度を上げれば速くなる、というのは直感的ですが、3日間の実データでははっきりと反証されました。

ピーク時には8〜9並列でサブセッションを走らせていましたが、その時間帯は以下がすべて同時に悪化しました。

* **親セッションの誤指示頻度**: 各サブセッションの状況を親が正確に把握できなくなり、スコープの食い違いが発生
* **人間の認知負荷**: 「いま誰が何をやっているか」を追えなくなり、コピペ同期で混線
* **API のレートリミット**: 並列数に比例して頭打ちの頻度が上がり、待ち時間で結局シーケンシャル化
* **Copilot レビューの質**: 同時多発の大規模 PR に対する指摘が断片化

結果として、**並列度を3〜4に絞った時間帯の方が、単位時間あたりの確定アウトプットは多かった**のです。

### 並列度の上限は「人間の同期処理能力」で決まる

見落としがちなのが、並列度の上限を決めるのは AI のスループットではなく、**人間がコピペやレビューで同期できる能力**だ、という点です。

| 並列度 | AIスループット | 人間の同期能力 | 結果 |
| --- | --- | --- | --- |
| 1〜2 | 人間側にスラックが残る | 余裕 | 人間が追加の判断を入れやすい |
| **3〜4（適正）** | **ほぼ理論値** | **ギリギリ追える** | **確定アウトプットが最大** |
| 5〜7 | AIは走るが指示が劣化 | 追えない箇所が出る | 見落とし・手戻り増 |
| 8〜9 | レートリミットで頭打ち | 完全にオーバーフロー | 誤指示と混線が連鎖 |

「AI を何並列で走らせられるか」ではなく、「自分が何並列までコピペと判断を捌けるか」が設計変数です。ここを見誤ると、並列度を上げれば上げるほど遅くなります。

---

## 原則3: 横断課題は事前に洗い出す

9リポジトリに手を入れ始めてから気づいたのですが、リポジトリ間で「同じパターンの課題」が山のように潜んでいました。

* 古い認証情報の残骸（複数リポジトリで別々に発見）
* Terraform の整形違反（多数のリポジトリで同日に CI が赤化）
* 環境変数の命名不整合（consumer 側とのズレ）
* 特定のユーティリティクラスへの cross-namespace 依存
* Secrets Manager の運用差異

個別に踏みながら直していった結果、**同じ構造の問題を何度も踏み直す**ことになりました。もしこれを着手前の事前調査で洗い出せていれば、**作業量を30〜50%は削減できた**という感触です。

事前調査の項目は、経験上、以下の6つをテンプレにしておくとほぼ取りこぼしません。

| # | 調査項目 | 手段 | 所要時間 |
| --- | --- | --- | --- |
| 1 | 依存関係マッピング | Python BFS やAIによる cross-repo 解析 | 30分-半日 |
| 2 | 認証情報の横断スキャン | `gitleaks` + `grep` | 30分 |
| 3 | インフラ設定の整形チェック | `terraform fmt -check` 等を全 repo で | 15分 |
| 4 | 主要な環境変数の実態確認 | 各実行環境へのダンプ | 30分 |
| 5 | Secrets の構成差異確認 | AWS CLI で列挙 + 差分 | 30分 |
| 6 | 前回フェーズ残タスクの設計意図再確認 | ドキュメントと挙動の照合 | 30分 |

**合計2〜3時間の投資**です。それで3日分の作業から1日分が消える計算なので、非常にペイのよい投資になります。

## 原則4: 「一気に vs 段階的」の判断フレーム

複数リポジトリに同じパターンの整理を入れるとき、大きく2つの戦略があります。

* **一気に hard fork**: 全 N リポジトリに並列で同じ整理を走らせる
* **段階的分割**: 共通レイヤーで先に解決 → 下位に展開していく

筆者は前者（一気 hard fork）を選びました。結果は前述のとおり「できたけど共通課題の重複解決が発生した」です。あとから段階的分割のシミュレーションをしてみると、無視できない差がありました。

```
fixU 分割ツリー（振り返りで描き直したもの）
│
Level 0: モノリス そのまま
Level 1: web / batch （2 repo）
Level 2: web → public / user / admin ＋ batch （4 repo）
Level 3: user → browser / api に分けて （5 repo）
Level 4: 最終的な （9 repo）← 今回の採択
```

段階的分割の本質的な利点は、\*\*「どこまで分割するかを途中で決められる」\*\*ことです。Level 2 で止めて4 repo 運用に落ち着ける選択肢も、Level 3 で止める選択肢もあります。一気方式だと、途中打ち切りの余地がありません。

共通課題で見ると、差は明瞭です。

| 共通課題 | 一気 hard fork | 段階的分割 |
| --- | --- | --- |
| 認証情報の一掃 | 9 repo × 個別対応 | 上流で1回 |
| Terraform 整形の整備 | 9 repo で同日違反 | 上流で解決、下流は継承 |
| cross-namespace なユーティリティの整理 | 下流7 repo で再発 | 上流で1回 |
| 環境変数設計 | 4 repo で個別検討 | 上流で1回 |
| Secrets 運用設計 | 9 repo 個別 | 上流で共通層 |
| 共通プロンプト/テンプレート整備 | 作業中に試行錯誤 | 上流で標準化 |

一気方式は「独立性確立の速さ」と引き換えに「共通課題の N 回重複解決」を受け入れる戦略です。N が小さければ許容できますが、N=9 ともなると重複コストが効いてきます。

次回、類似の判断ポイントに立ったときに問うべきは、「**共通課題の重横断性スコア**」と「**途中打ち切りの価値**」の2軸でしょう。重横断性が高く、中間層で止める価値があるなら段階的、そうでなければ一気、という使い分けが効きそうです。

## 原則5: 手戻りバイアスにメタ認知で勝つ

原則4を書いておいて何ですが、実は筆者自身、作業の途中で段階的分割案を思いついていました。それでも採用しなかった理由は、**手戻りのバイアス**です。

> 作業途中で段階的分割案を思いついたが、手戻りが惜しく、確信も持てなかったので、そのまま一気に走りきった

これは人間の経営判断に普遍的に発生する bias で、とくに **AI が既に大量の work-in-progress を抱えている状態**では強力に作用します。AI が走っている進捗を止めて再設計する、という心理的コストが、客観的なコストよりも大きく感じられるからです。

学びは単純で、**作業途中で「こうすべきだった」と感じたら、まず手戻りバイアスを疑う**、という一段深いメタ認知を常備する、ということです。

* 「手戻りしない場合のトータルコスト」と「手戻りする場合のトータルコスト」を紙に書く
* 「手戻りしたくない」という感情と「手戻りすべきでない」という論理を分離する
* 判断を AI に投げてセカンドオピニオンを取る（AI は手戻りバイアスを持たない）

AI 時代に人間が担う最大の価値は「方針転換を決断できる」ことだと最近よく思います。そのためにも、自分のバイアスを自分で検出する仕組みが要ります。

## 原則6: 前提誤りは必ず混入する

サブセッションへのプロンプトに書き込んだ「前提」が、実環境の実態と乖離していた、という事象が3日間で複数回発生しました。

* 「このリポジトリのスケジューラは既に停止している」と前提 → 実は動いていた
* 「この機能には対応データがまだ登録されていない」と前提 → 実は登録済みだった

どちらも、親セッション側の観測時点でのメモを鵜呑みにしていたのが原因です。**大規模並列作業では、観測時点の情報が作業後には古くなる**、という当たり前の現象を甘く見ていました。

対策はシンプルで、**プロンプトの最初のタスクに「前提確認ステップ」を必ず入れる**ことです。

```
(サブセッションへのプロンプト先頭に必ず入れる)

Step 0: 実測で前提を検証する
  - [前提A] は本当か → 実行環境で確認し、乖離していたら報告せよ
  - [前提B] は本当か → ログ/メトリクスで確認し、乖離していたら報告せよ
  - 乖離が見つかった場合は、作業計画の再立案まで戻ること
```

「前提は人間が書いて AI が疑う」という役割分担にすると、前提誤りは実行前に吸収できます。前提を疑うコストは AI にとって非常に低いので、ここは全面的に AI 側に寄せるのが効率的です。

## 原則7: 人間コピペは摩擦ではなく同期ポイント

最後の原則は、もっとも構造的で、もっとも過去の経験が効いたものです。

以前、筆者は [Dispatch（Claude Agent SDK ベースのサブセッション指揮役）で複数セッションを指揮しようとして破綻した経験](https://zenn.dev/fixu/articles/dispatch-multi-session-pm-failure)があります。Dispatch の設計意図は「人間を loop から外して自動化する」ことでしたが、実際には通信の不安定さから「人間が進捗確認 polling 要員として loop 内に引き戻される」という逆転現象が発生していました。

今回の3日間は、その反省から **親セッション(Code) + サブセッション(Terminal) + 人間のコピペ同期** という体制に切り替えました。一見すると「人間にコピペ作業を課している」ように見えます。実際、コピペ自体は地味な肉体作業です。しかし、この設計は機能しました。

| 観点 | Dispatch（破綻） | Code + Terminal（成功） |
| --- | --- | --- |
| セッション間通信 | 不安定 → 状態確認 polling が頻発 | 人間経由で安定（ローカル I/O） |
| 人間に期待した役割 | loop から外して自動化 | loop 内で意図的に同期 |
| **実態** | **人間が「進捗確認 polling 要員」に強制転換** | **コピペで自然に同期 + 認知キャッチアップ** |
| 人間負荷の性質 | **低価値な polling と待機** | **高価値な判断とスコープ決定** |
| スケーラビリティ | 通信品質に依存（脆弱） | 人間のコピペ処理能力に依存（確実） |

Dispatch の破綻は「人間を抜こうとして抜けなかった」結果でした。今回の成功は「人間を入れる前提で、人間が入る役割を高価値側に寄せた」結果です。

### 学びの定式化

> **AIマルチセッション運営では「人間負荷を減らす」のは誤り。「負荷の質を高価値側に寄せる」のが正しい。**

人間がサブセッション間で内容をコピペするのは、表面的には摩擦です。ですが実際には、コピペをしている最中に人間は内容を読み、認知キャッチアップし、ズレを検出しています。この「**判断と同期を同時に行う副次効果**」が、Dispatch 構成では得られませんでした。

コピペは摩擦ではなく、**強制同期ポイント**です。これを設計として受け入れるか、「自動化すべきノイズ」と見なすかで、構成の成否がひっくり返ります。

## まとめ: 7原則の一行サマリ

| # | 原則 | 一行で |
| --- | --- | --- |
| 1 | 段取り80%の法則 | AI実行が速いほど、事前段取りの相対価値が上がる |
| 2 | 並列度 ≠ 速さ | 適正並列度は3-4。上限は人間の同期能力で決まる |
| 3 | 横断課題の事前特定 | 2-3時間の事前調査で作業量30-50%削減 |
| 4 | 「一気 vs 段階的」判断フレーム | 共通課題の重横断性と途中打ち切り価値で選ぶ |
| 5 | 手戻りバイアスへのメタ認知 | 途中で違和感を抱いたら、まず自分のバイアスを疑う |
| 6 | 前提誤りの必然性 | 前提確認ステップをプロンプト先頭に必ず置く |
| 7 | 人間コピペは同期ポイント | 人間負荷を減らすのではなく、高価値側に寄せる |

この3日間で学んだ最大の構造的教訓は、次の一行に収まります。

> **「AI に並列でやらせる速さ」ではなく、「AI が事前段取りを支援する速さ」を最大化する。**

AI のスループットが上がるほど、勝負は実行フェーズではなく、その前後の設計と同期に移ります。270万行を消せたという事実よりも、次の同規模プロジェクトではこの7原則で3日を2日に縮められそうだ、という感触の方が、筆者にとっては大きな収穫でした。

同じようなマルチセッション運営に取り組んでいる方の、次の一歩のヒントになれば嬉しいです。
