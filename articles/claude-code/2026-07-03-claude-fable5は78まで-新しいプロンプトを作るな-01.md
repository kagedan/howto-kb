Fable 5がサブスクで使えるのは、日本時間7/8の16時ごろまでです。

残りわずか。新しいプロンプトを試したくなりますよね。

その使い方、実は少し損をします。

正確には、米太平洋時間で7/7の終日まで。日本時間だと7/8の16時ごろです。それを過ぎると、Fable 5は追加クレジットでの従量課金に切り替わります。

しかも、サブスク内で使えるのは週間利用枠の最大50%まで。Fable 5は他のモデルより枠の消費が速いので、実質的に無駄打ちできないモデルです。

新しいプロンプトの開発は、書いて、試して、直して、また試す作業です。消費の速いモデルで試行錯誤をやると、枠があっという間に溶けます。

だから僕は、残り時間でFable 5に新しいプロンプトを作らせません。

代わりに頼むのは、Claudeの中に溜まった「仕事の型」の総点検です。

この記事を読むと、期限が切れる前にFable 5へ頼むべき棚卸しの手順と、そのままコピペで使えるプロンプト2本が手に入ります。後半のプロンプトは、Claude Code、Codex、Obsidianまで使っている人向けです。

新しいプロンプトより先に、見るものがある

最上位モデルが使えると聞くと、まず新しいことをやらせたくなります。

その気持ちはわかります。僕も最初はそうでした。

でも、Claudeを日常的に使っている人ほど、先に見るべきものがあります。

すでに持っているものを、整えることです。

Claudeの中には、過去のProject指示、Knowledge、何度も使う文章の型、作りかけのプロンプト、古い作業ルールが残っています。

これらはうまく整っていれば資産です。

逆に、古い情報や矛盾した指示が混ざっていると、Claudeの回答をズラす原因になります。

AIの答えが古いのは、モデルのせいではなく、渡している前提のせいです。

そして、この点検作業こそ、Fable 5に向いています。

ProjectもKnowledgeも横断して、矛盾や古さを見つける作業は、読ませる文脈が広く、判断が重い。こういうタスクほど、モデルの実力差が出ます。しかも基本は一撃で終わるので、試行錯誤型の作業より枠の消費が読みやすいです。

Claude内の仕事資産とは

ここでいう仕事資産は、単なるプロンプト集ではありません。

たとえば、こういうものです。

よく使うプロンプト  
Claudeのプロジェクト機能（以下、Project）に入れている指示  
Knowledgeとして入れた資料  
何度も使う文章の型  
自分用の判断基準  
過去につくったSkill  
作業手順をまとめたMarkdown  
クライアント別、案件別の運用メモ

これらは、毎回ゼロから考えなくてよくするための資産です。

ただし、放置すると資産ではなくノイズになります。

古い料金表が残っている。  
昔のサービス説明が残っている。  
今は使っていないルールが残っている。  
似たプロンプトが増えすぎて、どれを使うべきかわからない。

この状態で新しいプロンプトを増やしても、運用はさらに散らかります。

僕のClaudeにも、複数のProject、記事制作用のプロンプト、Knowledgeに入れた資料が溜まっています。毎日使っているぶん、量だけは立派です。

まず見直すべき7項目

Fable 5に見てもらうなら、見るべきポイントはこの7つです。

古い情報  
価格、仕様、UI手順、日付、サービス説明が今も正しいか。Knowledgeに去年の料金表が残っている、のようなケースです。

矛盾した指示  
Project、Knowledge、Skill、Markdownの間で、言っていることがズレていないか。Projectでは、です・ます調を指定しているのに、別のメモでは常体を指定している、のようなケースです。

重複したプロンプト  
似た目的のプロンプトが増えすぎていないか。目的がほぼ同じ記事作成プロンプトが、3本も4本もある状態です。

長すぎるルール  
今のモデルなら不要な細かい指示や、過剰なMUSTが残っていないか。旧モデル時代の禁止事項が、延々と続いているやつです。

機密情報  
クライアント名、内部URL、APIキー、公開NGの情報が混ざっていないか。検証のときに貼った内部情報が、そのまま残っていることがあります。

もう使っていない手順  
昔の作業フローや、今は使っていないツール前提の指示が残っていないか。乗り換えたはずのツールの手順書が、まだ生きているケースです。

まだ型化できていない作業  
何度も繰り返しているのに、まだテンプレやSkillになっていない作業がないか。毎週手で書いている定型連絡などが典型です。

プロンプト1: Claude内仕事資産の棚卸し

まずは、Claudeだけで使える棚卸しプロンプトです。

```
# タスク
Claudeの中に溜まっている仕事資産を棚卸ししてください。
目的は、新しいプロンプトを増やすことではなく、すでにある指示・知識・型を、今後も使える状態に整えることです。

# 対象
このチャット、Project、Knowledge、利用できる範囲のSkill、過去に貼ったプロンプト、作業用Markdownなど、確認できる範囲を対象にしてください。

# 見る観点
1. 古い情報
価格、仕様、UI手順、日付、サービス説明が古くなっていないか。

2. 矛盾
Project、Knowledge、Skill、プロンプトの間で、指示が食い違っていないか。

3. 重複
似た目的のプロンプトやルールが増えすぎていないか。

4. 肥大化
今のモデルなら不要な細かすぎる指示、長すぎる前提、過剰なMUSTが残っていないか。

5. 危険
クライアント名、内部URL、APIキー、公開NG情報、機密情報が混ざっていないか。

6. 死蔵
今の仕事では使っていない古い手順やプロンプトが残っていないか。

7. 欠落
何度も繰り返しているのに、まだテンプレ化・Skill化されていない作業はないか。

# 制約
- ファイル編集はしないでください。
- 削除もしないでください。
- 推測で断定しないでください。
- 実物を読めていないものは「要確認」と書いてください。
- まずは棚卸し結果だけ出してください。

# 出力
1. 今すぐ直すべきもの
2. 残すが軽くするもの
3. 統合した方がいいもの
4. 捨ててもよさそうなもの
5. 新しくテンプレ化・Skill化した方がいいもの
6. 最初にやるべき3アクション
```

このプロンプトで大事なのは、いきなり直させないことです。

まずは「何が古いのか」「何が重複しているのか」「何が危ないのか」を見える化します。

AIにいきなり編集させると、必要なルールまで削られることがあります。棚卸しと修正は分けた方が安全です。

結果が出たら、全部直さない

棚卸し結果が出ると、全部きれいにしたくなります。

でも、最初から全部直す必要はありません。

まず見るのは3つだけです。

公開や納品に影響するもの  
機密情報や古い情報が混ざっているもの  
毎回使っているのに、今の仕事とズレているもの

この3つだけ先に直します。

残りは、残す、軽くする、統合する、捨てる、保留に分ければ十分です。

棚卸しの目的は、完璧な整理ではありません。

明日からの出力をズラす原因を減らすことです。

棚卸しが終わったClaudeはどうなるか

一度この整理をやると、Claudeを開いた瞬間に迷わなくなります。

Projectには現役の指示だけ。  
Knowledgeには今の前提だけ。

同じ依頼を投げても、返ってくる答えが今日の仕事に合っています。

この状態を先に作っておくと、新しいプロンプトを作ったときの効きも変わります。土台が整っている場所に足すのと、散らかった場所に足すのとでは、結果が別物になるからです。

応用編: Claude Code、Codex、Obsidianまで使っている人へ

Claudeだけなら、プロンプトとProjectを見直すだけでも効果があります。

ただ、Claude Code、Codex、Obsidianまで使っている人は、もう一段深く見た方がいいです。

理由は、AIの役割が分かれているからです。

Claudeは考える場所。  
Claude Codeは作業する場所。  
Codexは判断や設計を整理する場所。  
Obsidianは記録を残す場所。

この分担が崩れると、同じ話を何度も説明したり、古いルールを参照したり、確認待ちのメモを正式な判断として使ってしまったりします。

だから、濃く使っている人ほど、見るべきなのは「プロンプト単体」ではありません。

AI運用全体の構造です。

プロンプト2: AI運用全体の総点検

Claude Code、Codex、Obsidianまで使っている人は、

```
# タスク
Claude、Claude Code、Codex、Obsidianに分散している仕事資産の運用構造をレビューしてください。
目的は、ファイルを編集することではなく、運用上の改善点を見つけることです。

# 対象
確認できる範囲で、以下を見てください。

- Claude側のSkills、Projects、Knowledge、CLAUDE.md
- Claude Code側の作業ルール、コマンド、作業用Markdown
- Codex側のAGENTS.md、運用Markdown、設計ルール
- Obsidian側のフォルダ構造と記憶運用
- 確認待ちメモ、正式記憶、日次・週次・月次の棚卸しファイル

# 見る観点
1. 確認待ちと正式記憶が混ざっていないか
2. 古い価格、仕様、UI手順、公開方針が残っていないか
3. AGENTS.md、CLAUDE.md、Skills、Obsidian記憶で矛盾する指示がないか
4. 同じ役割のファイルやルールが重複していないか
5. 今は使っていない古い運用ルールが残っていないか
6. 機密情報、クライアント名、APIキー、内部URLが公開向け文面に混ざるリスクがないか
7. 日次、週次、月次の棚卸しで拾うべき情報が整理されているか
8. 考える場所、作業する場所、記録する場所の役割分担が崩れていないか

# 制約
- ファイル編集はしないでください。
- 削除もしないでください。
- 推測で断定しないでください。
- 実物を読めていないものは「要確認」と書いてください。
- 改善案は「今すぐ直すもの」「保留でよいもの」「本人確認が必要なもの」に分けてください。

# 出力
1. 全体評価
2. 最も危ない運用上の問題
3. 矛盾している可能性があるルール
4. 古くなっていそうな情報
5. 重複・肥大化している領域
6. 改善した方がいいフォルダ構造または命名ルール
7. 今すぐやる3つ
8. 本人に確認すべきこと
```

この2つ目のプロンプトは、万人向けではありません。

でも、AIを仕事の中に深く入れている人にはかなり効きます。

AI活用は、ツールを増やすほど強くなるわけではありません。

考える場所、作業する場所、記録する場所が分かれているほど、運用は安定します。

まとめ

残り時間でFable 5に頼むべきは、新しいプロンプト作りではありません。

Claudeの中に溜まった仕事の型を、点検してもらうことです。

古い指示を消す。  
重複したプロンプトをまとめる。  
危ない情報を分ける。  
何度も使う作業を型にする。

新しいプロンプトは、7/9以降でも作れます。他のモデルでも作れます。

でも、Fable 5に自分の環境をまるごと点検してもらえるのは、サブスクの範囲では7/8までです。

この2本のプロンプトは、Fable 5専用ではありません。次のモデルが来たときも、同じ手順でそのまま使えます。今すぐやらない人も、保存だけしておいてください。

ProjectやKnowledge、Claude CodeやObsidianまで広げると、どれが現役でどれがノイズか、自分では線を引きづらくなります。AIに渡す前提を整理して、外部脳として育てる運用は、僕が実務でやっている領域です。自分の環境ごと見直したい方は、DMでご相談ください。
