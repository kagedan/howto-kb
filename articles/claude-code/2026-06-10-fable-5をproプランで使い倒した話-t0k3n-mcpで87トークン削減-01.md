2026 6/9 Claude Fable 5 が出た。触ってみたら普通にやばくて、リリース直後に「Rustで3Dゲームエンジン作って」って頼んだら物理エンジン・GUIエディタ・スクリプトエディタまで全部ワンセッションで仕上がった。

MaxプランもMax 2プランも使ってない。Proプラン（$20/月）だ。

なんでそれができたかって話をする。

---

## Fable 5、トークン食いすぎ問題

Fable 5 はコーディング性能が本物なんだけど、料金が Opus 4.8 の2倍（Input $10/M・Output $50/M tokens）。つまりそのまま使うとトークンがみるみる減る。

大きめのプロジェクトで作業してると、コンテキストが溢れる→リセット→また最初からファイル読み込む、みたいなループに入って地獄になるやつ。Fable 5 だとそれが倍速で来る。

自分はそれを避けるために **T0K3N-MCP** という MCP サーバーを自分で作って使ってる。

---

## T0K3N-MCP って何

AIコーディングツールのトークン消費を最大87%削減する MCP サーバー。Rustで書いた。

Claude Code や Cursor の標準的なファイル読み込みは、ファイルをまるごとコンテキストに突っ込む。

```
read_file("server/mod.rs")  →  4,997 トークン消費
                                ↑ 大半は今の質問と無関係
```

T0K3N-MCP は「まずシグネチャだけ取って、必要な関数だけ本文を取る」設計にしてある。

```
read_code_skeleton("server/mod.rs")  →  1,162 トークン（シグネチャのみ）
read_code_body(["function:54-67"])   →    150 トークン（その関数だけ）
                                          ─────────────────────────────
合計                                        1,312 トークン  ← 74% 削減
```

### 実測数字

ベンチマークは Anthropic 公式のトークンカウント API（`claude-haiku-4-5-20251001`）で計測した。Fable 5 で計測したわけじゃないけど、削減率はモデルによらず構造的に決まる話なので同じ効果が出る。

Rust・TypeScript・Python・Go の4言語で計測済み。

| コードベース | 言語 | ファイル平均削減率 | プロジェクト全体 |
| --- | --- | --- | --- |
| T0K3N-MCP 本体 | Rust | **87.3%** | ~90% |
| vercel/commerce | TypeScript | **75.5%** | **86.0%** |
| pallets/flask | Python | **78.8%** | **90.5%** |
| gin-gonic/gin | Go | **70.4%** | **85.6%** |
| **4言語平均** | — | **78.0%** | **88.0%** |

Goが一番低い（70%）のには理由がある。Goはインターフェースをメソッドの羅列で定義するスタイルが多くて、`func (c *Context) GetString(key any) string` みたいなシグネチャがずらっと並ぶ。スケルトンに残る情報が他の言語より多い分、削減幅が小さくなる。それでも70%削減なので普通に使える。逆にPythonは78.8%でRustに次いで高い。

200K のコンテキストウィンドウが実質4.5〜8倍になる計算。

## で、何を作ったか

Fable 5 リリース直後に「Rustで3D対応のゲームエンジンをフルスクラッチで作って」って頼んだ。

作ったもの：

* 3Dレンダリングパイプライン
* ECS（Entity-Component-System）
* 物理エンジン（フル実装）
* GUIエディタ
* スクリプトエディタ

これが全部ワンセッションで終わった。

「ワンセッション」って言ったとき、時間の話をしてるわけじゃない。**トークン予算を使い切らずに完走できた**、ってことが本質。Fable 5 はトークン消費が重いから、普通にやったらどこかでコンテキストが溢れてリセットがかかる。T0K3N-MCP の87%削減があったから、最後まで溢れなかった。

---

## 途中でやってた工夫

T0K3N-MCP のツール群に加えて、`/clear` をこまめに打つのが地味に効いた。

会話履歴が積み上がるとそれ自体がトークンを食うので、作業のフェーズが変わるタイミング（設計が固まったとか、あるモジュールが完成したとか）で `/clear` してコンテキストをリセットする。

ただリセットすると状態が消えるので、クリア前に `session_snapshot` で作業状態を保存しておいて、再開時に `session_restore` で戻す。これで「リセットしたら忘れた」問題がなくなる。

あとは T0K3N-MCP に任せる。`check_budget` が残トークンを見て読み込み戦略を自動で決めてくれるし、`batch_read` で複数ファイルのリードをまとめたり、`memory_save` でセッションをまたいだ情報を持ち越せたりする。

## 計算してみると

Fable 5 は Opus 4.8 の2倍コストだけど、T0K3N-MCP の87%削減を掛けると：

```
Fable 5 素のコスト   =  Opus 4.8 × 2.0
T0K3N-MCP 適用後     =  Fable 5 × 0.13
                     =  Opus 4.8 × 2.0 × 0.13
                     =  Opus 4.8 × 0.26
```

Mythosクラスのモデルが Opus の4分の1以下のトークン消費で動く。

MaxプランやMax 2プランを使ってる人も多いと思うけど、T0K3N-MCP 入れればProプランで十分戦えると思ってる。
