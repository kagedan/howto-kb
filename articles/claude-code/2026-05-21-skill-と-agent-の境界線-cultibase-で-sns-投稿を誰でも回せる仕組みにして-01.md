CULTIBASE では、毎日の SNS 投稿を Skills と Agents の組み合わせで運用しています。

元々は特定のメンバーひとりに頼っていた SNS 運用を、  
チームの誰でもルール通りに回せる状態に少しずつ変えてきました。

その過程でいちばん時間をかけて考えたのが、  
「**Skill と Agent の境界をどこに引くか**」だったかなと思っています。

この記事では、

* 元々の運用がどんな課題を抱えていたか
* それをどう分けて整理したか
* Skill と Agent の境界をどう引いたか
* workflow.md を単一正本にした

について書いていきます。

SNS 運用に限らず、  
社内の業務を「1人に閉じない」かたちで仕組み化したい人の参考になればうれしいです。

---

## かつての運用と、ぶつかった壁

CULTIBASE の SNS 運用は、もともと 1 人の担当者がほぼ手作業で回してくれていました。

* どの記事を投稿するかの選定
* 投稿文の作成（トーンの調整も含めて）
* 各 SNS への予約・配信
* 過去の投稿との重複管理

これらが、その担当者の頭の中にすべて入っている状態だったんです。

やがて運用負荷が問題視され、担当者自身が個人で Skill 化・Agent 化を進めてくれていました。

これは大きな前進だったんですが、  
改善も「その 1 人に閉じている」状態だったかなと思っています。

> 何をどう判断して投稿しているのか、周りからは見えない。  
> 助けたくても、入り口がない。

属人化の本当のつらさって、

「1 人に負荷が集中する」ことだけじゃなくて、

「**周りが改善に参加できない**」ことなんだなと、このときよく感じました。

---

## 4つのレイヤーに分けて整理してみた

そこで、個人で作られていた Skill / Agent を一度ほどいて、チームで共有できる形に再設計してみました。中心にあるのは、次の 4 つのレイヤーです。

それぞれの役割をざっくり言うと、こんな感じです。

| レイヤー | 役割 | 例 |
| --- | --- | --- |
| **Skill** | AI への「指示書」。何をどう作らせるか・出力形式・禁則 | 投稿文を 1 本生成する／コンテンツデータベースから 1 記事を選ぶ |
| **Tool** | 実 I/O を担当するスクリプト | コンテンツデータベースの読み込み、X・Facebook API への投稿 |
| **Agent** | 「○○して」と話しかける相手。業務一連を完遂する | 即時で投稿する Agent／日次で複数枠を予約する Agent |
| **Workflow** | Agent が読む手順書（`workflow.md`） | 複数枠を順次予約する手順 |

## Skill と Agent の境界をどう引いたか

この設計でいちばん時間をかけたのがここかも。

最初は「全部 Skill でいいんじゃないかな」と思ってたんですよね。

でも、実際に動かしてみると Skill だけじゃ収まらないシーンが出てきました。

### 判断は 3 つの問いで済ませる

僕が境界を引くときに使ってるルールはシンプルで、こんな感じです。

1. **これは 1 つの単発タスクか？** → Yes なら **Skill** 候補
2. **これは状況によって順序や条件が変わる業務か？** → Yes なら **Agent / Workflow** 候補
3. **これは実コードでしかできないか？** → Yes なら **Tool** として切り出し

具体例で見るとわかりやすいかなと思います。

### Skill にしたもの（毎回ほぼ同じ判断で動くもの）

* **投稿文の生成**：トーンも字数制限も毎回同じ
* **記事の選定（1 件）**：蓄積されたコンテンツのデータベースから、選定ロジック（伸びの度合い順・重複回避ルール）にもとづいて 1 件を選ぶ
* **投稿の伸び具合の更新**：各種 SNS のインプレッションから機械的に算出

このあたりは「入力と出力が明確で、毎回ぶれない」ので Skill にしました。Skill の定義ファイル（SKILL.md）に "判断ルール" と "出力フォーマット" を書いておくと、誰が呼んでも同じ品質で動いてくれる感覚があります。

### Agent にしたもの（複数の Skill を組み合わせる業務）

代表例は、毎日 3 つの時間帯にまとめて予約投稿してくれる Agent です。

* 09:00 = 蓄積されたコンテンツからの再配信
* 19:00 = 新規のコンテンツを優先
* 20:00 = よく伸びている投稿を優先

を順番に処理します。

これは Skill にはしませんでした。理由としては、

* 時間帯ごとに**選定方針が違う**
* 時間帯のあいだの**順序が大事**（前のスロットの結果をコンテンツデータベースの履歴に反映してから次へ）
* 完了後に**3 枠まとめての報告**が必要

つまり「業務の手順」を持つ単位だからです。

ここを Skill 1 つに詰め込んじゃうと定義ファイルがふくれあがって、

AI にとっても人間にとっても読みにくくなるなと思って、分けることにしました。

* コンテンツデータベースとのやり取りを担うクライアント
* X / Facebook API への投稿スクリプト
* インプレッション集計

このあたりは「AI に判断させる必要はなくて、実コードで叩くだけ」のものです。

Skill 内から呼び出すだけにしておくと、Skill の責務がぐっと小さくなって、見通しがよくなった気がしています。

### 境界が引けると、増やしやすくなる

この境界が引けたことで、副次的にうれしかったのが、  
**新しい Skill や Agent を追加しやすくなった**ことです。

新しい運用ニーズが来たときに、

* 単発タスクっぽい → 既存の Skill を真似て 1 つ足す
* 業務フローっぽい → 既存の Skill を組み合わせて Agent を 1 つ足す

みたいに、シンプルに判断できるようになりました。「どこに何を書くか」で迷う時間が減ったのが、地味ですがけっこう大きい変化だったかなと思っています。

## workflow.md を「単一正本」にした

社内では Claude Code と Gemini CLI の両方が使われています。

SNS 運用の Agent も、どちらからでも呼び出せるようにしたかったんですが、同じ手順書を `CLAUDE.md` と `GEMINI.md` に二重で書くと、片方だけ更新されてズレていくのは、そらそうですよね。

なので、こんな構成にしてみました。

```
.
├── agents/                              # Agent 定義の "正本"
│   ├── example-agent/
│   │   ├── workflow.md                  # ← 業務フローの正本（中身はここだけ）
│   │   ├── CLAUDE.md                    # Claude Code 用アダプター（frontmatter のみ）
│   │   └── GEMINI.md                    # Gemini CLI 用アダプター（frontmatter のみ）
│   └── another-agent/
│       ├── workflow.md
│       ├── CLAUDE.md
│       └── GEMINI.md
├── .claude/                             # Claude Code から見える置き場（symlink）
│   └── agents/
│       ├── example-agent.md             →  agents/example-agent/CLAUDE.md
│       └── another-agent.md             →  agents/another-agent/CLAUDE.md
└── .gemini/                             # Gemini CLI から見える置き場（symlink）
    └── agents/
        ├── example-agent.md             →  agents/example-agent/GEMINI.md
        └── another-agent.md             →  agents/another-agent/GEMINI.md
```

ポイントは、

* `agents/<name>/workflow.md` を**正本**にして、フローの定義はここだけに書く
* `CLAUDE.md` / `GEMINI.md` は frontmatter だけ持って、本文は `workflow.md` を参照する
* `.claude/agents/` `.gemini/agents/` からは symlink で参照する

ところです。

### 各ファイルの中身のイメージ

正本になる `workflow.md` には、業務フローのすべてが書かれています。たとえばこんな感じ。

```
# Example Agent

<この Agent が何をするか>

## 使用スキル

- skill-a
- skill-b

## トリガーフレーズ

- 「...」
- 「...」

## 実行フロー

### Step 1: ...

### Step 2: ...

## 実行にあたっての注意

- ...
```

`CLAUDE.md` と `GEMINI.md` のほうは中身を持たず、frontmatter で自己紹介だけして、本文は `workflow.md` に "丸投げ" する形にしています。

`agents/example-agent/CLAUDE.md`:

```
---
name: example-agent
description: <Agent の説明>
---

業務フローは [workflow.md](./workflow.md) を参照してください。
```

`agents/example-agent/GEMINI.md`:

```
---
name: example-agent
description: <Agent の説明>
---

業務フローは [workflow.md](./workflow.md) を参照してください。
```

frontmatter の項目は Claude Code と Gemini CLI でそれぞれの仕様に合わせていますが、**本文（フローの定義）は両方とも `workflow.md` を見にいくだけ**、というかたちにしています。

### symlink で「両方の世界」から見えるようにする

Claude Code は `.claude/agents/` を、Gemini CLI は `.gemini/agents/` を、それぞれ Agent の置き場として見にきます。

なので、それぞれの置き場から本体（`agents/example-agent/CLAUDE.md` や `GEMINI.md`）に symlink を張っておきます。

```
.claude/agents/example-agent.md   →  agents/example-agent/CLAUDE.md
.gemini/agents/example-agent.md   →  agents/example-agent/GEMINI.md
```

こうしておくと、それぞれの CLI からは「いつもの場所に Agent がある」ように見えるし、編集するときは本体（`agents/` 配下）だけ触ればよくなります。

### 全体の関係はこんな感じ

地味な工夫ですが、

* フローを更新したいときは `workflow.md` だけ直せばいい
* Claude Code / Gemini CLI どちらで動かしても同じ挙動になる
* どちらかだけ更新してズレる、みたいな事故が起きない

というメリットがあって、「正本は 1 つ」というだけで改善時の手戻りやドキュメントの食い違いがほぼなくなった感覚があります。

ちなみに、Windows で clone するときは symlink の取り扱いに少し注意が必要なんですが、そのへんは README 側で案内する形にしていて、記事の本筋からは外れるのでここでは触れません。

## で、何が変わったか

運用や設計を整えた結果、いちばんうれしい変化はこれかなと思っています。

> **チームの誰でも、SNS 運用のルールに沿った投稿ができるようになった。**

* 「今日の 3 枠予約して」と Agent に話しかければ、ルール通りに動いてくれる
* 選定ロジックは Skill にあるので、トーンや基準を変えたいときは Skill だけ直せばいい
* 改善のアイデアを誰かが持ったとき、「どこを触ればいいか」がはっきりしている

属人化してた業務が、**仕組みのかたちでチームに開いた**感覚があって、これは個人的にすごくよかったなと思っています。

---

## 境界を引くと、関わり方が広がる

最後に、この経験から得たいちばんの学びを書いておこうかなと。

Skill / Tool / Agent / Workflow を分ける本質って、結局のところ

「**何が変わりやすくて、何が変わりにくいか**」で分けることだったなと思います。

* トーンや選定基準は変わりやすい → **Skill** に集約
* API の叩き方は変わりにくい → **Tool** に固定
* 業務フローはたまに変わる → **Agent / Workflow** に置く

この分け方ができてると、改善するときに「ここを触ればいい」が一意に決まる。  
それが結果として、属人化しない・誰でも触れる仕組みになる、というのが今回いちばん腹落ちしたポイントでした。

SNS 運用に限らず、社内のツールを「1 人に閉じない」かたちで育てたい人の参考になればうれしいです。
